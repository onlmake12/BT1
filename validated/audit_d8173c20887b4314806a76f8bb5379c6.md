### Title
Multi-Token Fee Validation Missing Spot Price Conversion — (`target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

### Summary

The Osmosis-specific `is_fee_sufficient` function in the Pyth CosmWasm contract accepts alternative fee denominations but performs no spot-price conversion before comparing amounts. The code explicitly acknowledges this as a placeholder ("this logic will be updated to use spot price for different valid tokens in future"), meaning any Osmosis-allowed token is accepted at a 1:1 nominal ratio with `uosmo`. A user can underpay fees by submitting a cheaper alternative token.

### Finding Description

The Osmosis build of the Pyth CosmWasm contract supports multi-token fee payment. `is_fee_sufficient` (Osmosis variant) iterates over the caller's sent coins, validates each denom via `is_allowed_tx_fees_denom`, sums their raw amounts, and then checks:

```rust
Ok(base_denom_fee.amount.u128() <= total_amount)
``` [1](#0-0) 

The comment in the code is explicit:

> "NOTE: the base fee denom right now is = denom: 'uosmo', amount: 1, which is almost negligible. It's not important to convert the price right now. For now we are keeping the base fee amount same for each valid denom -> 1 but this logic will be updated to use spot price for different valid tokens in future" [2](#0-1) 

The same missing conversion exists in `get_update_fee_for_denom`:

```rust
// this will be change later on to add custom logic using spot price for valid tokens
Ok(coin(get_update_fee_amount(deps, vaas)?, denom))
``` [3](#0-2) 

The intended design — convert each alternative token to its `uosmo` equivalent using spot price before summing — is described in the comment block but never implemented:

> "how to change this in future: for given coins verify they are allowed in txfee module, convert each of them to the base token that is 'uosmo', combine all the converted token, check with `has_coins`" [4](#0-3) 

This is structurally identical to the Mantra M-04 pattern: a custom conversion component (`ConvertToDenom` / spot-price lookup) is defined and intended to be used, but is not wired into the fee-sufficiency check, so the protocol falls back to a trivially incorrect comparison (raw amounts treated as equal across all denoms).

### Impact Explanation

Any caller of `update_price_feeds` or `parse_price_feed_updates` on the Osmosis deployment can pay fees in an Osmosis-allowed alternative token (e.g., `uion`, IBC tokens) at a 1:1 nominal ratio with `uosmo`. If the alternative token is worth less than `uosmo`, the caller underpays the required fee. As the fee amount is governance-configurable and could be raised, the gap between nominal and real value can grow. The protocol's intended fee revenue from multi-token support is bypassed. [5](#0-4) 

### Likelihood Explanation

The entry path is fully unprivileged: any transaction sender calling `update_price_feeds` on the Osmosis deployment with an alternative token triggers this path. The `is_allowed_tx_fees_denom` check only validates that the token is listed in Osmosis's txfee module — it does not enforce any value equivalence. The Osmosis feature is a production build flag, not a test-only path. [6](#0-5) 

### Recommendation

Implement the spot-price conversion that the code already describes as the intended future behavior. Before summing `total_amount`, query the Osmosis spot price for each non-base denom and convert its amount to `uosmo` equivalents. Only then compare against `base_denom_fee.amount`. The `get_update_fee_for_denom` function should similarly apply the inverse conversion when returning a fee quoted in an alternative denom.

### Proof of Concept

1. Deploy the Osmosis build of the Pyth CosmWasm contract with `fee = Coin::new(1000, "uosmo")`.
2. Call `update_price_feeds` sending `1000 uion` (where `uion` trades at, say, 0.01 uosmo).
3. `is_allowed_tx_fees_denom` returns `true` for `uion`.
4. `total_amount = 1000`, `base_denom_fee.amount = 1000`.
5. The check `1000 <= 1000` passes — the caller paid ~10 uosmo worth of value instead of 1000 uosmo. [7](#0-6)

fn is_allowed_tx_fees_denom(deps: &Deps, denom: &String) -> bool {
    // TxFeesQuerier uses stargate queries which we can't mock as of now.
    // The capability has not been implemented in `cosmwasm-std` yet.
    // Hence, we are hacking it with a feature flag to be able to write tests.
    // FIXME
    #[cfg(test)]
    if denom == "uion"
        || denom == "ibc/FF3065989E34457F342D4EFB8692406D49D4E2B5C70F725F127862E22CE6BDCD"
    {
        return true;
    }

    let querier = TxfeesQuerier::new(&deps.querier);
    match querier.denom_pool_id(denom.to_string()) {
        Ok(_) => true,
        Err(_) => false,
    }
}
