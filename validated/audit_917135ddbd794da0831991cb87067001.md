Audit Report

## Title
Self-Referential Hole Punching Messages Allow Resource Exhaustion via Unbounded Background Task Spawning — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary

`ConnectionRequestProcess::execute()` lacks a guard rejecting messages where `content.from == self_peer_id`, allowing an attacker to poison `pending_delivered` with attacker-controlled addresses keyed under the victim's own peer ID. A subsequent stream of `ConnectionSync(from=victim_id, to=victim_id, route=[])` messages — each passing the 1/second rate limit — causes the victim to repeatedly spawn background tasks that run up to 24 concurrent `try_nat_traversal` futures per message, each retrying TCP connections for 30 seconds. Because `pending_delivered` is read with `get()` rather than `remove()`, the poisoned entry is never consumed, enabling indefinite re-triggering.

## Finding Description

**Step 1 — Poison `pending_delivered`**

In `ConnectionRequestProcess::execute()`, the guards before `respond_delivered` are:

- Line 128: `content.route.contains(self_peer_id)` — passes with an empty route
- Lines 132–143: `forward_rate_limiter.check_key(&(content.from, content.to, self.msg_item_id))` — allows 1/second for the key `(victim_id, victim_id, item_id)`
- Line 145: `self_peer_id == &content.to` — true when `to = victim_id`

There is no check that `content.from != self_peer_id`. When the attacker sends `from = victim_id, to = victim_id`, the condition at line 145 is satisfied and `respond_delivered(from_peer_id = victim_id, listen_addrs = [attacker_ip:port])` is called.

Inside `respond_delivered`:
- Lines 161–167: 2-minute cooldown check — passes on first call
- Lines 196–215: TCP/IPv4/IPv6 address filter — passes for a valid `/ip4/.../tcp/...` address
- Lines 217–219: Returns early if `remote_listens.is_empty()` — passes if attacker provides a valid address
- Lines 226–232: Sends `ConnectionRequestDelivered` back to the attacker's session — succeeds if attacker is connected
- Lines 234–237: `self.protocol.pending_delivered.insert(from_peer_id, (remote_listens, now))` — stores `victim_id → [attacker_ip:port]`

**Step 2 — Trigger background tasks via `ConnectionSync`**

In `ConnectionSyncProcess::execute()`, when `route` is empty and `self_peer_id == content.to` (both true when `from = to = victim_id`), lines 111–115 do:

```rust
let listens_info = self.protocol.pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
```

`get()` is used, not `remove()`, so the poisoned entry persists across all subsequent `ConnectionSync` messages. Lines 119–124 create one `try_nat_traversal` future per address (up to `ADDRS_COUNT_LIMIT = 24`), and lines 143–162 spawn a single background task running all futures concurrently via `select_ok`. The spawn is gated only on `listen_addresses.first()` being `Some` (line 140), which is true for any listening node.

**`try_nat_traversal` behavior** (`component/mod.rs` lines 49–115): Each future retries TCP connections to the target address for up to 30 seconds with ~200ms intervals (~150 attempts). If the attacker refuses connections, all 24 futures run for the full 30 seconds.

**Rate limiting does not prevent the attack:**

The `forward_rate_limiter` is keyed by `(from, to, item_id)` at 1/second. With `from = to = victim_id`, the attacker sends 1 `ConnectionSync` per second. Each spawns 1 background task containing up to 24 concurrent futures. After 30 seconds: **30 background tasks × 24 concurrent futures = 720 concurrent TCP connection attempts**, each holding a socket for up to 30 seconds.

The 2-minute cooldown in `respond_delivered` (lines 161–167) only prevents re-poisoning `pending_delivered`, not re-triggering via `ConnectionSync`. Within a single 2-minute poisoning window, the attacker can send 120 `ConnectionSync` messages, spawning 120 tasks × 24 futures = 2,880 concurrent TCP connections.

## Impact Explanation

The unbounded accumulation of background tasks and concurrent TCP sockets exhausts file descriptors and CPU on the victim node, causing it to crash or become unresponsive. This matches **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

Any peer connected to the victim over the P2P network can execute this attack. No special privileges, keys, or majority hashpower are required. The `from` and `to` fields in `ConnectionRequest` and `ConnectionSync` are unauthenticated byte fields — any peer can set them to any value, including the victim's own `local_peer_id()`. The attack requires only two crafted P2P message types and is locally testable. The 1/second rate limit on `ConnectionSync` does not prevent the attack; it only paces the task accumulation.

## Recommendation

Add an explicit guard in `ConnectionRequestProcess::execute()` rejecting messages where `content.from == self_peer_id` or `content.from == content.to`, placed before the `self_peer_id == &content.to` branch:

```rust
if &content.from == self_peer_id || content.from == content.to {
    return StatusCode::Ignore.with_context("self-referential request rejected");
}
```

Add a symmetric guard in `ConnectionSyncProcess::execute()` rejecting `content.from == self_peer_id` when the node is the `to` target. Additionally, replace `get()` with `remove()` in `ConnectionSyncProcess` at lines 111–115 so that each `pending_delivered` entry can only trigger one round of NAT traversal tasks.

## Proof of Concept

```rust
// Pseudocode unit test
let victim_peer_id = node.local_peer_id().clone();

// Step 1: Send ConnectionRequest(from=victim, to=victim, listen_addrs=[attacker:1337])
// Precondition: attacker is connected to victim
let req = ConnectionRequest {
    from: victim_peer_id.clone(),
    to: victim_peer_id.clone(),
    listen_addrs: vec!["/ip4/1.2.3.4/tcp/1337".parse().unwrap()],
    route: vec![],  // empty route bypasses route check
    max_hops: 6,
};
attacker_session.send(req);
// Assert: pending_delivered[victim_peer_id] == [/ip4/1.2.3.4/tcp/1337/p2p/<victim>]

// Step 2: Send 30 ConnectionSync(from=victim, to=victim, route=[]) at 1/second
for _ in 0..30 {
    let sync = ConnectionSync {
        from: victim_peer_id.clone(),
        to: victim_peer_id.clone(),
        route: vec![],
    };
    attacker_session.send(sync);
    sleep(Duration::from_secs(1));
}
// Assert: 30 background tasks active, each running up to 24 concurrent try_nat_traversal futures
// Assert: TCP SYN packets observed toward 1.2.3.4:1337
// Assert: node file descriptor count approaches system limit
```
