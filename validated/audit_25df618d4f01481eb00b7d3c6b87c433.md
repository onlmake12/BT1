### Title
Ignored `async_send_message_to` Return Value Stalls Compact Block Reconstruction — (File: `sync/src/relayer/block_transactions_process.rs`)

### Summary
In `BlockTransactionsProcess::execute()`, when a node needs to re-request missing transactions for compact block reconstruction, the result of the network send call is silently discarded with `let _ignore = ...`. Immediately after, the node unconditionally updates its pending compact block state as if the request was successfully delivered. If the send fails, the node's state records that a `GetBlockTransactions` request is in-flight when none was actually sent, stalling compact block reconstruction for that block from that peer indefinitely.

### Finding Description
In `sync/src/relayer/block_transactions_process.rs`, inside `BlockTransactionsProcess::execute()`, after a `ReconstructionResult::Missing` or `ReconstructionResult::Collided` outcome, the node builds a `GetBlockTransactions` message and attempts to send it:

```rust
let _ignore = async_send_message_to(&self.nc, self.peer, &message).await;

let _ignore_prev_value =
    mem::replace(expected_transaction_indexes, missing_transactions);
let _ignore_prev_value = mem::replace(expected_uncle_indexes, missing_uncles);
``` [1](#0-0) 

The `async_send_message_to` call returns a `Result` indicating whether the message was actually queued/sent to the peer. By binding it to `_ignore`, any send failure (e.g., peer buffer full, connection dropped mid-flight) is silently swallowed. The two `mem::replace` calls then overwrite `expected_transaction_indexes` and `expected_uncle_indexes` in the `pending_compact_blocks` map unconditionally, recording the new missing-index lists as "requested." [2](#0-1) 

The `pending_compact_blocks` state is a shared async-locked map keyed by block hash. After the failed-but-ignored send, the entry for this block hash now contains the updated missing-index lists, and the node waits for a `BlockTransactions` reply that the peer will never send because it never received the request. [3](#0-2) 

A parallel ignored-result pattern exists in `get_headers_process.rs` inside `send_in_ibd`, where the result of sending an `InIBD` message is also discarded:

```rust
let _ignore =
    async_send_message(SupportProtocols::Sync.protocol_id(), &nc, peer, &message)
        .await;
``` [4](#0-3) 

And in `headers_process.rs`, the result of `async_disconnect` is discarded, meaning a peer that should be disconnected during IBD may remain connected:

```rust
let _ignore = nc
    .async_disconnect(self.peer, "useless outbound peer in IBD")
    .await;
``` [5](#0-4) 

The most protocol-impactful of these is the `block_transactions_process.rs` case because it corrupts in-flight reconstruction state.

### Impact Explanation
When `async_send_message_to` fails silently and state is updated as if it succeeded:

- The `pending_compact_blocks` entry for the block hash retains the updated `expected_transaction_indexes` / `expected_uncle_indexes` lists, marking those indexes as "already requested."
- No retry is triggered because the node believes the request is in-flight.
- The compact block for that hash from that peer cannot be reconstructed until the entry is evicted by a timeout or the peer disconnects.
- Block propagation latency increases; in a targeted scenario where an attacker controls the timing of partial `BlockTransactions` responses to repeatedly trigger this path, the node's compact block reconstruction pipeline for specific blocks can be repeatedly stalled. [6](#0-5) 

### Likelihood Explanation
The attacker-controlled entry path is straightforward for any connected peer:

1. Attacker peer sends a valid compact block whose short IDs are not all in the node's tx-pool, triggering a `GetBlockTransactions` request.
2. Attacker responds with a `BlockTransactions` message that satisfies verification but still leaves some transactions missing (e.g., by providing only a subset of the requested transactions).
3. The node enters the `ReconstructionResult::Missing` branch and attempts to send a follow-up `GetBlockTransactions`.
4. If the attacker has filled the peer's send buffer or the connection degrades at this moment, `async_send_message_to` returns an error that is silently discarded.
5. The node's state is updated as if the follow-up request was sent; reconstruction stalls. [7](#0-6) 

Network send failures are realistic under normal congestion or adversarial buffer-filling. No privileged access, key material, or majority hashpower is required.

### Recommendation
Check the return value of `async_send_message_to` and handle failure explicitly. If the send fails, do **not** update `expected_transaction_indexes` / `expected_uncle_indexes`, so the node retains the previous state and can retry or fall back to requesting the full block from another peer:

```rust
if async_send_message_to(&self.nc, self.peer, &message).await.is_ok() {
    *expected_transaction_indexes = missing_transactions;
    *expected_uncle_indexes = missing_uncles;
} else {
    // log and return an error status so the caller can handle the failure
    return StatusCode::Network.with_context("failed to send GetBlockTransactions");
}
```

Apply the same pattern to `send_in_ibd` in `get_headers_process.rs` and the `async_disconnect` call in `headers_process.rs`.

### Proof of Concept

1. Connect an attacker-controlled peer to a CKB node.
2. Send a compact block whose short IDs are not in the node's mempool, causing the node to issue `GetBlockTransactions`.
3. Reply with a `BlockTransactions` message that passes `BlockTransactionsVerifier::verify` but leaves at least one transaction index still missing, triggering `ReconstructionResult::Missing`.
4. At the moment the node calls `async_send_message_to` for the follow-up `GetBlockTransactions`, exhaust the peer's send buffer (e.g., flood the connection with large messages) so the send returns an error.
5. Observe that the node's `pending_compact_blocks` entry for the block hash is updated with the new missing-index lists but no actual request was sent. The node waits indefinitely for a `BlockTransactions` reply that never arrives, stalling reconstruction of that block from that peer. [8](#0-7) [9](#0-8)
