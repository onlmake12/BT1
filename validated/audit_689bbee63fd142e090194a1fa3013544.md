### Title
Inconsistent Script Group Attribution in `complete()` vs `resumable_verify()` / `resume_from_state()` - (File: `script/src/verify.rs`)

### Summary

The `complete()` function in `TransactionScriptsVerifier` uses a stale `current_group` reference (the snapshot group) instead of the current loop iteration's `group` when accumulating cycles for subsequent script groups. This is inconsistent with the parallel `resumable_verify()` and `resume_from_state()` functions, which correctly use the current `group`. The result is that if `wrapping_cycles_add` returns an error (cycle overflow) for any script group processed after the snapshot group, the error is attributed to the wrong script group — the one that was suspended, not the one that actually caused the overflow.

### Finding Description

In `script/src/verify.rs`, three functions implement resumable script verification: `resumable_verify`, `resume_from_state`, and `complete`. All three iterate over script groups and accumulate cycles using `wrapping_cycles_add(cycles, used_cycles, group)`, where the third argument is the script group used for error source attribution.

In `resumable_verify` (line 245), the correct `group` is used:

```rust
cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
```

In `resume_from_state` (line 324–325), the correct `group` is also used:

```rust
current_used = wrapping_cycles_add(current_used, consumed_cycles, group)?;
cycles = wrapping_cycles_add(cycles, consumed_cycles, group)?;
```

However, in `complete` (line 395), the loop over subsequent groups (after the snapshot group) incorrectly uses `current_group` — the snapshot group — instead of the current iteration's `group`:

```rust
for (_hash, group) in self.groups().skip(snap.current + 1) {
    ...
    match self.verify_group_with_chunk(group, remain_cycles, &None) {
        Ok(ChunkState::Completed(used_cycles, _consumed_cycles)) => {
            cycles = wrapping_cycles_add(cycles, used_cycles, current_group)?; // BUG: should be `group`
        }

### Impact Explanation

When `wrapping_cycles_add` returns an error (cycle overflow) for a script group processed after the snapshot group in `complete()`, the error is attributed to `current_group` (the suspended group) rather than the actual failing `group`. This produces a `TransactionScriptError` with the wrong source, which propagates up through `ContextualTransactionVerifier::complete()`: [4](#0-3) 

The misattributed error then reaches the chain verification path and the tx-pool's `Reject::Verification` classification. The `is_malformed_from_verification` check formats the error as a string and checks for specific content: [5](#0-4) 

A cycle-overflow error attributed to the wrong script group produces a different error string than expected, which can cause incorrect malformed-tx classification and incorrect peer reputation decisions (e.g., whether a relaying peer is penalized for submitting a transaction that fails in a subsequent group).

### Likelihood Explanation

The `complete()` code path is exercised when a suspended script verification is finalized — used in the tx-pool's chunked verification flow and in block verification. Any transaction sender can submit a transaction with multiple script groups (e.g., multiple inputs with different lock scripts, or inputs with both lock and type scripts). If the cumulative cycles across groups approach `u64::MAX` (or the configured `max_block_cycles`), the overflow path in `wrapping_cycles_add` is triggered. While extreme cycle values are uncommon in practice, the inconsistency is structurally identical to the reported pattern and is reachable by any unprivileged transaction sender or block relayer.

### Recommendation

In the `complete()` function's loop over subsequent groups, replace `current_group` with `group` in the `wrapping_cycles_add` call to match the behavior of `resumable_verify` and `resume_from_state`:

```rust
// Before (line 395):
cycles = wrapping_cycles_add(cycles, used_cycles, current_group)?;

// After:
cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
``` [6](#0-5) 

### Proof of Concept

1. Craft a transaction with at least two script groups (e.g., two inputs with different lock scripts).
2. Ensure the first group's script suspends mid-execution (triggering the `complete()` path).
3. Ensure the second group's script execution causes `wrapping_cycles_add` to return an error (e.g., by crafting cycles near overflow).
4. Submit the transaction via RPC (`send_transaction`) or relay it via the P2P relay protocol.
5. Observe that the resulting `TransactionScriptError` attributes the failure to the first (snapshot) script group rather than the second (actual failing) group, producing a misattributed error that can cause incorrect malformed-tx classification downstream. [7](#0-6)

    pub fn complete(&self, snap: &TransactionState, max_cycles: Cycle) -> Result<Cycle, Error> {
        let mut cycles = snap.current_cycles;

        let (_hash, current_group) = self.groups().nth(snap.current).ok_or_else(|| {
            ScriptError::Other(format!("snapshot group missing {:?}", snap.current))
                .unknown_source()
        })?;

        if max_cycles < cycles {
            return Err(ScriptError::ExceededMaximumCycles(max_cycles)
                .source(current_group)
                .into());
        }

        // continue snapshot current script
        // max_cycles - cycles checked
        match self.verify_group_with_chunk(current_group, max_cycles - cycles, &snap.state) {
            Ok(ChunkState::Completed(used_cycles, _consumed_cycles)) => {
                cycles = wrapping_cycles_add(cycles, used_cycles, current_group)?;
            }
            Ok(ChunkState::Suspended(_)) => {
                return Err(ScriptError::ExceededMaximumCycles(max_cycles)
                    .source(current_group)
                    .into());
            }
            Err(e) => {
                #[cfg(feature = "logging")]
                logging::on_script_error(_hash, &self.hash(), &e);
                return Err(e.source(current_group).into());
            }
        }

        for (_hash, group) in self.groups().skip(snap.current + 1) {
            let remain_cycles = max_cycles.checked_sub(cycles).ok_or_else(|| {
                ScriptError::Other(format!("expect invalid cycles {max_cycles} {cycles}"))
                    .source(group)
            })?;

            match self.verify_group_with_chunk(group, remain_cycles, &None) {
                Ok(ChunkState::Completed(used_cycles, _consumed_cycles)) => {
                    cycles = wrapping_cycles_add(cycles, used_cycles, current_group)?;
                }
                Ok(ChunkState::Suspended(_)) => {
                    return Err(ScriptError::ExceededMaximumCycles(max_cycles)
                        .source(group)
                        .into());
                }
                Err(e) => {
                    #[cfg(feature = "logging")]
                    logging::on_script_error(_hash, &self.hash(), &e);
                    return Err(e.source(group).into());
                }
            }
        }

        Ok(cycles)
    }

fn is_malformed_from_verification(error: &Error) -> bool {

            .expect("error kind checked")

                .kind()

    }
