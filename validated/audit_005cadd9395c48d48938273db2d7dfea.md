### Title
TOCTOU Race in `resumeble_process_tx` Allows Duplicate Transactions Into Verify Queue — (`tx-pool/src/process.rs`)

---

### Summary

`TxPoolService::resumeble_process_tx` performs duplicate-existence checks (orphan pool, verify queue) under separate, short-lived read locks, then enqueues the transaction under a separate write lock. Because each `.await` point is a yield point in Tokio, two concurrent async tasks carrying the same transaction can both pass all checks and both call `enqueue_verify_queue`, inserting the same transaction into the verify queue twice. This is the direct Rust/async analog of a missing reentrancy guard: a check-then-act sequence that is not atomic.

---

### Finding Description

`resumeble_process_tx` (called by every remote relay path via `submit_remote_tx` and every local notify path via `notify_tx`) performs three sequential async operations, each acquiring and releasing its own lock:

```
resumeble_process_tx (tx-pool/src/process.rs:335-353)
  1. orphan_contains(&tx).await          // read-locks orphan, releases
  2. verify_queue_contains(&tx).await    // read-locks verify_queue, releases
  3. enqueue_verify_queue(tx, ...).await // write-locks verify_queue, inserts
``` [1](#0-0) 

Each of `orphan_contains` and `verify_queue_contains` acquires a Tokio `RwLock` read guard, performs the check, and drops the guard before returning: [2](#0-1) 

Because Tokio is cooperative and every `.await` is a scheduling point, the following interleaving is possible when two relay handlers (Task A and Task B) carry the same transaction `T`:

```
Task A: orphan_contains(T)       → false  (lock released)
Task B: orphan_contains(T)       → false  (lock released)
Task A: verify_queue_contains(T) → false  (lock released)
Task B: verify_queue_contains(T) → false  (lock released)
Task A: enqueue_verify_queue(T)  → Ok     (T inserted)
Task B: enqueue_verify_queue(T)  → Ok     (T inserted again)
```

No single lock is held across all three steps, so there is no mutual exclusion protecting the check-then-act sequence. This is structurally identical to a Solidity function that reads state, releases the reentrancy guard, and then writes state — the pattern the original report flags.

The same pattern appears in `process_tx` (line 409) and `test_accept_tx` (line 390): [3](#0-2) [4](#0-3) 

---

### Impact Explanation

When the same transaction is enqueued twice, the verify worker (`verify_mgr.rs`) dequeues and processes it twice via `_process_tx`: [5](#0-4) 

Each `_process_tx` call runs `pre_check` (read lock, cheap) and then `verify_rtx` — the full CKB-VM script execution, which is the most CPU-expensive operation in the node. The second execution is entirely redundant. The `check_txid_collision` inside `pre_check` will eventually reject the second entry at the pool-write stage, but only **after** the full script verification has already been paid for: [6](#0-5) 

An attacker who controls multiple peers can submit the same high-cycle transaction simultaneously from N peers, causing N redundant full script verifications. Because `max_block_cycles` can be up to the consensus limit (tens of millions of cycles), each redundant verification is a significant CPU expenditure. This is a realistic, externally-reachable resource-exhaustion (DoS) path requiring no privileged access.

---

### Likelihood Explanation

- **Entry path**: Any unprivileged P2P peer can relay a transaction via the standard relay protocol. Multiple peers can send the same transaction simultaneously — this is a normal network condition, not an exotic scenario.
- **Trigger**: The race window exists at every `.await` yield between the two read-lock checks and the write-lock enqueue. Under Tokio's cooperative scheduler, concurrent relay handlers for the same transaction will routinely interleave at these points.
- **No special conditions**: No admin keys, no 51% hashpower, no Sybil attack required. A single attacker with two peer connections is sufficient.

---

### Recommendation

Collapse the three-step check-then-enqueue into a single critical section that holds the write lock for the entire duration:

```rust
pub(crate) async fn resumeble_process_tx(...) -> Result<bool, Reject> {
    self.non_contextual_verify(&tx, remote).await?;

    // Hold write lock across check AND enqueue — no yield between them
    let mut queue = self.verify_queue.write().await;
    if self.orphan.read().await.contains_key(&tx.proposal_short_id()) {
        return Err(Reject::Duplicated(tx.hash()));
    }
    if queue.contains_key(&tx.proposal_short_id()) {
        return Err(Reject::Duplicated(tx.hash()));
    }
    queue.push(/* entry */);
    Ok(true)
}
```

Alternatively, make `enqueue_verify_queue` itself idempotent by returning a distinct error if the entry already exists, and treat that as a non-fatal duplicate rejection at the call site — ensuring the second enqueue is a no-op regardless of the race outcome.

---

### Proof of Concept

1. Attacker controls peers P1 and P2, both connected to the victim node.
2. Attacker crafts transaction `T` with a script that consumes close to `max_block_cycles` cycles.
3. P1 and P2 simultaneously send a `RelayTransactionHashes` / `RelayTransactions` message for `T`.
4. The node spawns two concurrent async relay handlers, both calling `submit_remote_tx(T, ...)`.
5. Both handlers pass `orphan_contains` and `verify_queue_contains` (both return `false`) before either has called `enqueue_verify_queue`.
6. Both handlers call `enqueue_verify_queue`, inserting `T` twice into the verify queue.
7. The verify worker dequeues `T` twice and runs full CKB-VM script verification twice.
8. The second run is rejected at `submit_entry` → `check_txid_collision`, but only after the full CPU cost of script execution has been paid.
9. Repeating at high frequency from multiple peers causes sustained CPU exhaustion on the victim node. [7](#0-6) [2](#0-1) [8](#0-7)
