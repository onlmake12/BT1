### Title
Unprotected Direct Callback in Legacy `revealWithCallback` Path Allows Requester to Grief Provider/Relayer - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains two distinct execution paths. The newer path (for requests with an explicit `gasLimit10k != 0`) uses `excessivelySafeCall` to safely catch callback reverts. However, the legacy `else` branch — entered for all requests created via `requestWithCallback` (which stores `gasLimit10k = 0`) — calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly with no revert protection. A malicious requester can cause this call to revert, making the entire `revealWithCallback` transaction revert and causing the provider/relayer to lose gas without compensation.

---

### Finding Description

`requestWithCallback` passes `gasLimit = 0` to `requestV2`, which stores `gasLimit10k = 0` in the request struct: [1](#0-0) 

When `revealWithCallback` is called for such a request, the condition at line 574 evaluates to `false` (because `req.gasLimit10k == 0`), so execution falls into the `else` branch: [2](#0-1) 

Inside the `else` branch, after `clearRequest` is called, the callback is invoked directly with no try/catch or safe call: [3](#0-2) 

If `_entropyCallback` reverts, the entire transaction reverts — including the `clearRequest` call — leaving the request in its original pending state. The provider/relayer loses all gas spent on the transaction and receives no fee.

By contrast, the protected path uses `excessivelySafeCall` which catches reverts and transitions the request to `CALLBACK_FAILED` state: [4](#0-3) 

---

### Impact Explanation

A malicious requester can permanently grief any provider/relayer attempting to fulfill their legacy `requestWithCallback` request. The provider loses gas on every fulfillment attempt. Because the request is never cleared (the revert undoes `clearRequest`), the request remains stuck and the provider cannot earn their fee. If the requester's contract is designed as a honeypot (succeeds in simulation, reverts on-chain), the provider has no reliable way to detect this before submitting the transaction.

---

### Likelihood Explanation

The attack is straightforward: a requester deploys a contract whose `_entropyCallback` can be toggled between success and revert (e.g., via a state variable). The requester calls `requestWithCallback` normally, then front-runs the provider's `revealWithCallback` transaction to flip the toggle. This is a realistic front-running scenario on any public mempool chain where Pyth Entropy is deployed. The `requestWithCallback` function is the primary legacy entry point and is widely used.

---

### Recommendation

Wrap the direct callback invocation in the `else` branch with a try/catch (or use `excessivelySafeCall`) so that a reverting callback does not propagate to the caller:

```solidity
// Instead of:
IEntropyConsumer(callAddress)._entropyCallback(
    sequenceNumber,
    provider,
    randomNumber
);

// Use:
try IEntropyConsumer(callAddress)._entropyCallback(
    sequenceNumber,
    provider,
    randomNumber
) {} catch {
    emit CallbackFailed(...);
}
```

This mirrors the protection already present in the `gasLimit10k != 0` path.

---

### Proof of Concept

1. Attacker deploys `MaliciousConsumer` with a `bool public shouldRevert` flag. `_entropyCallback` reverts when `shouldRevert == true`.
2. Attacker calls `requestWithCallback(provider, userContribution)` — `shouldRevert` is `false`, so off-chain simulation succeeds.
3. Provider observes the request and submits `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`.
4. Attacker front-runs the provider's transaction and sets `shouldRevert = true`.
5. Provider's `revealWithCallback` executes the `else` branch, calls `_entropyCallback`, which reverts.
6. The entire transaction reverts. Provider loses gas. Request remains pending.
7. Attacker resets `shouldRevert = false` to allow simulation to pass again, and repeats indefinitely. [5](#0-4)
