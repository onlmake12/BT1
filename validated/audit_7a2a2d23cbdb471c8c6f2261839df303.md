### Title
Unbounded `firstUnfulfilledSeq` Scan in `executeCallback` Enables Griefing via Request Inflation — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

Any unprivileged caller can inflate `_state.currentSequenceNumber` by submitting many `requestPriceUpdatesWithCallback()` calls. After those requests are fulfilled out of order (all except the oldest), the unbounded `while` loop inside `executeCallback()` that advances `_state.firstUnfulfilledSeq` must scan through every already-cleared sequence number. With enough requests, this loop consumes gas exceeding the block gas limit, permanently preventing fulfillment of the oldest pending request and locking its deposited funds.

---

### Finding Description

`Echo.sol` maintains a monotonically increasing `_state.currentSequenceNumber` and a `_state.firstUnfulfilledSeq` pointer. After each successful callback execution, `executeCallback()` advances `firstUnfulfilledSeq` past all already-cleared requests via an unbounded `while` loop:

```solidity
// Echo.sol lines 169-174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

The loop calls `findRequest()` on every iteration, which performs one or two cold `SLOAD` operations (2,100 gas each) against the fixed-size `requests[32]` array and the `requestsOverflow` mapping:

```solidity
// Echo.sol lines 310-321
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];
    }
}
``` [2](#0-1) 

There is no cap on the number of iterations. The developers themselves flagged this in a TODO comment:

> "TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. a better solution would be a doubly-linked list of active requests." [3](#0-2) 

`requestPriceUpdatesWithCallback()` is callable by any address and increments `currentSequenceNumber` on every call:

```solidity
// Echo.sol line 73
requestSequenceNumber = _state.currentSequenceNumber++;
``` [4](#0-3) 

The only input constraint is `publishTime <= block.timestamp + 60` and a fee payment; there is no per-address rate limit or cap on total outstanding requests. [5](#0-4) 

---

### Impact Explanation

**Attack scenario:**

1. A legitimate user submits request #1 (sequence number `S`).
2. An attacker submits N requests (#2 … #N+1), each with `callbackGasLimit = 0` to minimize cost.
3. The provider (or the attacker themselves after the exclusivity period) fulfills requests #2 … #N+1. Each of these `executeCallback` calls is cheap because `firstUnfulfilledSeq` is still stuck at `S` (request #1 is still active), so the `while` loop exits immediately.
4. The provider now attempts to fulfill request #1. `clearRequest(S)` marks it inactive, then the `while` loop starts at `S` and must scan through all N already-cleared sequence numbers before stopping.
5. Each iteration costs ≈ 2,200–4,350 gas (cold `SLOAD` for `requestsOverflow` + warm `SSTORE` for `firstUnfulfilledSeq++`). With N ≈ 7,000–13,000 requests, the loop alone consumes the entire block gas limit (~30 M gas on Ethereum mainnet; lower on many L2s).
6. `executeCallback` for request #1 reverts out-of-gas. The funds locked in request #1 (`req.fee`) are permanently unrecoverable — there is no cancellation or refund path.

The `EchoState` confirms there is no upper bound on `currentSequenceNumber` and no mechanism to cancel or refund a stuck request: [6](#0-5) 

---

### Likelihood Explanation

- **Entry path is fully unprivileged.** `requestPriceUpdatesWithCallback()` requires only a fee payment; no role or whitelist check exists.
- **Cost is bounded and predictable.** With `callbackGasLimit = 0` and 1 price feed, the per-request fee is `pythFeeInWei + baseFeeInWei + feePerFeedInWei`. On chains where these fees are low (e.g., L2s), the total cost to create enough requests to exceed the block gas limit is modest.
- **The exclusivity period is irrelevant.** After `exclusivityPeriodSeconds` elapses, the attacker can fulfill their own requests themselves, controlling the order of fulfillment precisely.
- **The TODO comment confirms the developers are aware of the unbounded scan** but have not yet mitigated it.
- **Lower block gas limits on L2s** (e.g., Arbitrum, Base, BNB Chain) reduce the number of requests needed, lowering the attack cost further.

---

### Recommendation

1. **Cap the `while` loop** to a fixed maximum number of iterations per `executeCallback` call (e.g., 64). Accept that `firstUnfulfilledSeq` may lag and catch up over multiple fulfillments.
2. **Replace the linear scan with a doubly-linked list** of active requests (as the existing TODO suggests), so advancing the pointer is O(1).
3. **Add a per-address or global cap** on the number of outstanding (unfulfilled) requests to bound `currentSequenceNumber - firstUnfulfilledSeq`.

### Proof of Concept

```solidity
// Pseudocode — attacker contract
function attack(IEcho echo, address provider, bytes32[] calldata priceIds) external payable {
    uint32 gasLimit = 0; // minimize fee
    uint96 fee = echo.getFee(provider, gasLimit, priceIds);

    // Step 1: victim creates request #1 (seq = S) — already in-flight

    // Step 2: attacker creates N requests (seq = S+1 … S+N)
    uint N = 10_000;
    for (uint i = 0; i < N; i++) {
        echo.requestPriceUpdatesWithCallback{value: fee}(
            provider,
            block.timestamp,
            priceIds,
            gasLimit
        );
    }

    // Step 3: after exclusivity period, attacker fulfills requests S+1 … S+N
    // (each executeCallback is cheap; firstUnfulfilledSeq stays at S)
    for (uint64 seq = S + 1; seq <= S + N; seq++) {
        echo.executeCallback{value: pythFee}(provider, seq, updateData, priceIds);
    }

    // Step 4: provider attempts to fulfill request S
    // The while loop runs N times → OOG → request S funds permanently locked
}
```

The `while` loop at lines 169–174 of `Echo.sol` has no iteration cap, and `findRequest()` at lines 310–321 performs cold storage reads on every iteration, making the gas cost grow linearly with the number of pre-fulfilled requests between `firstUnfulfilledSeq` and the sequence number being fulfilled.
