### Title
`SizeVerifier` Allows Transactions Up to `max_block_bytes` That Can Never Fit in a Valid Block — (`verification/src/transaction_verifier.rs`)

---

### Summary

`SizeVerifier`, used inside `NonContextualTransactionVerifier`, checks that a single transaction's serialized size does not exceed `consensus.max_block_bytes()`. However, `BlockBytesVerifier` checks that the **entire block's** serialized size (header + cellbase + all transactions + uncle headers) does not exceed the same `max_block_bytes`. Because a block always carries overhead beyond the transaction payload, a transaction that passes `SizeVerifier` at the maximum allowed size can never be included in a valid block. The two checks use the same threshold but measure fundamentally different quantities, creating a direct contradiction analogous to the reported H-5 bug.

---

### Finding Description

**Check 1 — `SizeVerifier` (per-transaction):**

`NonContextualTransactionVerifier` is constructed with `SizeVerifier::new(tx, consensus.max_block_bytes())`:

```rust
// verification/src/transaction_verifier.rs
size: SizeVerifier::new(tx, consensus.max_block_bytes()),
```

`SizeVerifier::verify()` passes a transaction if:

```rust
let size = self.transaction.data().serialized_size_in_block() as u64;
if size <= self.block_bytes_limit { Ok(()) }
```

So a single transaction is allowed to be up to `max_block_bytes` = `TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT` = 597 × 1,000 = **597,000 bytes**.

**Check 2 — `BlockBytesVerifier` (per-block):**

```rust
// verification/src/block_verifier.rs
let block_bytes = block.data().serialized_size_without_uncle_proposals() as u64;
if block_bytes <= self.block_bytes_limit { Ok(()) }
```

This measures the **total block size**: block header (~208 bytes), cellbase transaction, all committed transactions, uncle block headers, and Molecule framing overhead. The limit is the same `max_block_bytes` = 597,000 bytes.

**The contradiction:**

A transaction of size exactly `max_block_bytes` passes `SizeVerifier`. But when placed in a block, the block's total serialized size is `max_block_bytes + block_header_size + cellbase_size + framing_overhead`, which always exceeds `max_block_bytes`. `BlockBytesVerifier` then rejects the block. The transaction that was declared valid by `SizeVerifier` can never be committed in any valid block.

The tx-pool adds a secondary guard (`TRANSACTION_SIZE_LIMIT = 512,000` bytes) in `non_contextual_verify`, but this is a local policy constant, not a consensus rule, and it is applied *after* `SizeVerifier`. `SizeVerifier` itself is also invoked in `contextual_block_verifier.rs` during block verification, where the `TRANSACTION_SIZE_LIMIT` guard is absent.

---

### Impact Explanation

- A transaction with serialized size in the range `(TRANSACTION_SIZE_LIMIT, max_block_bytes]` — i.e., between 512 KB and 597 KB — passes `SizeVerifier` in block verification but causes `BlockBytesVerifier` to reject the containing block.
- The `SizeVerifier` check is therefore too permissive: it certifies transactions as size-valid that are structurally impossible to commit on-chain.
- Any block relayed by a peer that contains such a transaction will be rejected at `BlockBytesVerifier`, but the per-transaction `SizeVerifier` will have already returned `Ok`, creating an inconsistent validation signal.
- The tx-pool's `TRANSACTION_SIZE_LIMIT` guard partially masks the issue for locally submitted transactions, but does not apply to blocks received over the P2P relay path.

---

### Likelihood Explanation

The contradiction is always present whenever a transaction's serialized size is in the range `(TRANSACTION_SIZE_LIMIT, max_block_bytes]`. Any transaction sender or block relayer can craft such a transaction. The block verifier will ultimately reject the block, but the per-transaction check incorrectly signals validity. This is reachable by any unprivileged transaction sender or block relayer.

---

### Recommendation

`SizeVerifier` should use a limit that accounts for the minimum block overhead (header, cellbase, Molecule framing), not the raw `max_block_bytes`. Concretely:

```rust
// Subtract minimum block overhead before using as per-tx limit
size: SizeVerifier::new(tx, consensus.max_block_bytes() - MIN_BLOCK_OVERHEAD),
```

Alternatively, align `SizeVerifier`'s limit with the existing `TRANSACTION_SIZE_LIMIT` constant (512,000 bytes), which already accounts for this overhead, and promote it to a consensus-level parameter rather than a local policy constant.

---

### Proof of Concept

1. `MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT = 597 × 1,000 = 597,000`. [1](#0-0) 

2. `SizeVerifier` is constructed with `consensus.max_block_bytes()` as the per-transaction limit. [2](#0-1) 

3. `SizeVerifier::verify()` passes any transaction with `size <= block_bytes_limit` (i.e., up to 597,000 bytes). [3](#0-2) 

4. `BlockBytesVerifier::verify()` measures the **total block** serialized size against the same 597,000-byte limit. [4](#0-3) 

5. A block containing a single transaction of size 597,000 bytes has total size ≥ 597,000 + header (~208 B) + cellbase overhead + Molecule framing, which always exceeds 597,000. `BlockBytesVerifier` rejects it while `SizeVerifier` approved the transaction.

6. The tx-pool's secondary guard (`TRANSACTION_SIZE_LIMIT = 512,000`) is a local policy applied only in `non_contextual_verify`, not in the block verification path. [5](#0-4) [6](#0-5)
