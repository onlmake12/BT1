Audit Report

## Title
`add_entry` overwrites `total_tx_size`/`total_tx_cycles` with stale pre-computed values after ancestor-eviction decrements them - (File: tx-pool/src/component/pool_map.rs)

## Summary
In `PoolMap::add_entry`, size/cycle totals are snapshotted into locals before `check_and_record_ancestors` runs. When that function evicts entries via `remove_entry_and_descendants` ‚Üí `update_stat_for_remove_tx`, it decrements `self.total_tx_size` and `self.total_tx_cycles` in place. The stale pre-eviction locals are then unconditionally written back, silently discarding those decrements and permanently overcounting pool size and cycle usage by the sum of all evicted transactions' sizes/cycles. The inflated totals then cause `limit_size` to evict additional legitimate transactions to compensate for phantom capacity.

## Finding Description
**Root cause ‚Äî stale snapshot written after in-place mutation:**

`add_entry` (lines 200‚Äì221): [1](#0-0) 

- Lines 210‚Äì211: `updated_stat_for_add_tx` snapshots `self.total_tx_size + entry.size` into a local variable. At this point no eviction has occurred.
- Line 213: `check_and_record_ancestors` may call `remove_entry_and_descendants` (line 618) in a loop, which internally calls `update_stat_for_remove_tx`.
- Lines 733‚Äì740: `update_stat_for_remove_tx` directly mutates `self.total_tx_size` and `self.total_tx_cycles` in place via `checked_sub`. [2](#0-1) 
- Lines 218‚Äì219: The stale pre-eviction locals are written back unconditionally, overwriting the already-decremented live fields. The correct post-eviction value `original + entry.size ‚àí Œ£(evicted_sizes)` is replaced with `original + entry.size`.

**Eviction trigger path ‚Äî `check_and_record_ancestors` (lines 598‚Äì625):** [3](#0-2) 

The eviction branch fires when `ancestors_count > max_ancestors_count` (line 598 guard fails) AND `ancestors_count ‚àí cell_ref_parents.len() ‚â§ max_ancestors_count` (line 603 guard passes). `cell_ref_parents` are existing pool transactions that reference as a cell dep an output the new transaction spends as an input. When both conditions hold, `remove_entry_and_descendants` is called in a loop until the ancestor count drops below the limit.

**Why existing guards are insufficient:**
`updated_stat_for_add_tx` (lines 711‚Äì729) only checks for arithmetic overflow; it does not account for subsequent in-place mutations to the same fields. [4](#0-3) 
There is no re-synchronisation of the local snapshot after `check_and_record_ancestors` returns. The underflow recovery path in `update_stat_for_remove_tx` (lines 742‚Äì756) only fires when subtraction underflows, not when a later overwrite discards a valid decrement.

## Impact Explanation
The overcounted `total_tx_size` causes `limit_size` to read an inflated value and evict legitimate pending transactions that should remain in the pool. Each successful trigger of the eviction path in `check_and_record_ancestors` permanently inflates the accounting until an underflow recomputation accidentally corrects it. This is an important correctness degradation of the mempool, mapping to **Low (501‚Äì2000 points): Any other important performance improvements for CKB**, as it silently drops valid user transactions and corrupts pool accounting.

## Likelihood Explanation
An unprivileged user can trigger this by: (a) placing transactions in the pool that reference a specific unspent output as a cell dep, and (b) submitting a new transaction that spends that output as an input while also having a deep ancestor chain exceeding `max_ancestors_count` (default 25) but satisfying the `cell_ref_parents` reduction condition. Both conditions are achievable with normal transaction submission APIs and modest on-chain resources. The scenario is narrow but fully within reach of a motivated attacker and is repeatable.

## Recommendation
Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` returns, so the snapshot is taken from the already-decremented live values:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals AFTER all eviction mutations are complete
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, after `check_and_record_ancestors` returns, re-derive the final values directly from the live fields rather than from the pre-eviction snapshot.

## Proof of Concept
1. Create a `PoolMap` with `max_ancestors_count = 3`.
2. Insert a chain tx_A ‚Üí tx_B ‚Üí tx_C where tx_C produces output O.
3. Insert tx_D that uses output O as a **cell dep** (making tx_C a `cell_ref_parent` of any future tx spending O as an input).
4. Build a chain tx_E ‚Üí tx_F ‚Üí tx_G ‚Üí tx_H where tx_H spends output O as an **input** (ancestor count = 4 > 3, but `cell_ref_parents = {tx_C}` so `4 ‚àí 1 = 3 ‚â§ 3`, triggering the eviction branch).
5. Call `pool_map.add_entry(tx_H_entry, Status::Pending)`.
6. Assert: `pool_map.total_tx_size == Œ£ sizes of all remaining entries`. With the bug, the value will be larger by `size(tx_C) + size(tx_D)` (the evicted entries whose decrement was discarded).
7. Confirm `limit_size` then evicts an additional legitimate entry to compensate for the phantom size inflation.

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {

            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }
