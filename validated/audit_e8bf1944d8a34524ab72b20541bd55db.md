### Title
SetGovernanceDataSource `initial_sequence` Overwrites Sequence Counter Without Monotonicity Check, Re-Opening Governance Replay Window — (`target_chains/sui/contracts/sources/governance/set_governance_data_source.move`)

### Summary
`set_governance_data_source::execute` unconditionally overwrites `last_executed_governance_sequence` with the attacker-controlled `initial_sequence` field from the VAA payload. Because this write happens **after** the sequence guard in `execute_governance_instruction` has already passed, a relayer can submit two legitimately-signed SetGovernanceDataSource VAAs in sequence where the second carries a lower `initial_sequence`, rolling the counter backward and re-opening a replay window for governance VAAs in the vacated range.

### Finding Description
`execute_governance_instruction` enforces `VAA.sequence > last_executed_governance_sequence` and then sets `last_executed_governance_sequence = VAA.sequence`. It then dispatches to `set_governance_data_source::execute`, which calls `state::set_last_executed_governance_sequence(latest_only, pyth_state, initial_sequence)` — a second, unchecked write that can decrease the counter to any value encoded in the payload.

Given:
- VAA1: Wormhole sequence S1, `initial_sequence = X` → counter ends at X
- VAA2: Wormhole sequence S2 > X, `initial_sequence = Y < X` → counter ends at Y

After VAA2, any governance VAA from the newly-set source with Wormhole sequence in `(Y, X)` satisfies `sequence > Y` and will be accepted, even if those VAAs were previously considered expired.

### Impact Explanation
An attacker who controls the relay order of two legitimately-signed governance VAAs can roll back the governance sequence counter. This re-opens a replay window for governance actions (SetDataSources, SetUpdateFee, SetStalePriceThreshold, etc.) in the vacated sequence range, allowing previously-superseded governance instructions to be re-executed. The scoped impact matches "smart contract unable to operate" / griefing: an adversary can force the contract back to stale fee, data-source, or threshold configurations.

### Likelihood Explanation
Governance VAAs are publicly broadcast once signed by Wormhole guardians. Any unprivileged relayer can observe and selectively submit them. The only precondition is the existence of two legitimately-signed SetGovernanceDataSource VAAs where the second carries a lower `initial_sequence` — a realistic scenario during governance source rotation or rollback procedures.

### Recommendation
Add a monotonicity check inside `set_governance_data_source::execute` before calling `set_last_executed_governance_sequence`:

```move
let current = state::get_last_executed_governance_sequence(pyth_state);
assert!(initial_sequence >= current, E_INITIAL_SEQUENCE_MUST_NOT_DECREASE);
state::set_last_executed_governance_sequence(latest_only, pyth_state, initial_sequence);
```

Alternatively, remove the second `set_last_executed_governance_sequence` call from `set_governance_data_source::execute` entirely, since `execute_governance_instruction` already updates the counter to the VAA's Wormhole sequence number before dispatching.

    public fun execute_governance_instruction(
        pyth_state : &mut State,
        receipt: WormholeVAAVerificationReceipt,
    ) {
        // This capability ensures that the current build version is used.
        let latest_only = state::assert_latest_only(pyth_state);

        // Get the sequence number of the governance VAA that was used to
        // generate the receipt.
        let sequence = receipt.sequence;

        // Require that new sequence number is greater than last executed sequence number.
        assert!(sequence > state::get_last_executed_governance_sequence(pyth_state),
            E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);

        // Update latest executed sequence number to current one.
        state::set_last_executed_governance_sequence(&latest_only, pyth_state, sequence);

        let payload = receipt.payload;

        destroy(receipt);

        let instruction = governance_instruction::from_byte_vec(payload);

        // Get the governance action.
        let action = governance_instruction::get_action(&instruction);

        // Dispatch the instruction to the appropriate handler.
        if (action == governance_action::new_contract_upgrade()) {
            abort(E_MUST_USE_CONTRACT_UPGRADE_MODULE_TO_DO_UPGRADES)
        } else if (action == governance_action::new_set_governance_data_source()) {
            set_governance_data_source::execute(&latest_only, pyth_state, governance_instruction::destroy(instruction));
