### Title
`PythLazer.verifyUpdate()` Does Not Validate Payload Timestamp Against `block.timestamp`, Enabling Indefinite Replay of Stale Signed Price Updates — (`File: lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` verifies only that the ECDSA signature is valid and that the signing key has not expired. It never checks whether the payload's embedded `timestamp` field is recent relative to `block.timestamp`. Because the same signed update bytes can be submitted repeatedly with no replay protection, any attacker can capture a past signed Lazer price update and replay it indefinitely to any consumer contract that relies on `verifyUpdate()` for price authenticity.

---

### Finding Description

`PythLazer.verifyUpdate()` performs three checks:

1. The EVM format magic is correct.
2. The ECDSA signature over `keccak256(payload)` recovers to a non-zero address.
3. That address maps to a signer whose `expiresAt > block.timestamp`. [1](#0-0) 

The payload itself is a structured binary blob whose first 12 bytes (after a 4-byte magic) encode a `uint64 timestamp` representing when the price data was produced: [2](#0-1) 

`verifyUpdate()` never reads or validates this `timestamp` field. It returns the raw `payload` bytes to the caller without any staleness assertion. There is also no nonce, sequence number, or "used update" set that would prevent the same `update` bytes from being submitted a second (or thousandth) time.

The `isValidSigner()` guard only checks whether the **signing key** is still within its authorized window — it says nothing about the **age of the data** the key signed: [3](#0-2) 

---

### Impact Explanation

Any consumer contract that calls `verifyUpdate()` and then uses the returned `payload` without independently checking the embedded timestamp will accept arbitrarily old price data as "verified." An attacker who captured a signed update when a price was at a favorable level can replay that update at any future time (while the signer key remains valid) to:

- Trigger under-collateralized borrows or avoid liquidations by presenting a stale high collateral price.
- Execute trades against a stale favorable price in a perp/spot protocol.
- Manipulate any on-chain state that is gated on a Lazer-verified price.

The Pyth documentation example does show a `_timestamp > timestamp` guard in consumer code, but this is advisory only — `verifyUpdate()` itself enforces no such constraint, and consumers that omit the check receive no protocol-level protection. [2](#0-1) 

---

### Likelihood Explanation

- The entry path is fully unprivileged: any EOA can call `verifyUpdate{value: fee}(oldUpdate)`.
- Signed Lazer updates are broadcast over a public WebSocket stream and appear in on-chain transaction calldata, so capturing historical updates requires no special access.
- Lazer is designed for high-frequency (sub-second) price feeds; a price from even a few seconds ago can differ materially from the current price.
- The signer key validity window is set far in the future (e.g., `3000000000000000` in the test), so the replay window is effectively unbounded. [4](#0-3) 

---

### Recommendation

Add a staleness check inside `verifyUpdate()` that reads the `timestamp` from the payload and reverts if it is older than a configurable `maxStalenessSeconds`:

```solidity
// After recovering payload:
uint64 payloadTimestamp = uint64(bytes8(payload[4:12])); // skip 4-byte magic
require(
    block.timestamp <= uint256(payloadTimestamp) + maxStalenessSeconds,
    "stale price update"
);
```

Alternatively, track a mapping of `keccak256(update) => bool` to prevent replay of the exact same bytes. Both mitigations together provide the strongest protection.

---

### Proof of Concept

1. At time `T`, observe a signed Lazer update `U` (e.g., from a past transaction) where BTC/USD = $100,000.
2. At time `T + 3600` (one hour later), BTC/USD = $90,000.
3. Call `pythLazer.verifyUpdate{value: fee}(U)`.
4. `verifyUpdate()` recovers the signer, confirms `isValidSigner()` returns `true`, and returns `payload` — no staleness check occurs.
5. The consumer contract parses `payload`, reads price = $100,000, and acts on it (e.g., allows a borrow that would be under-collateralized at the real price of $90,000).

The same `U` can be submitted again at `T + 7200`, `T + 86400`, etc., as long as the signer key has not expired. [5](#0-4)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
