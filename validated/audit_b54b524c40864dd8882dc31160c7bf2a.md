Audit Report

## Title
Off-by-One in `check_purge` Group Eviction Threshold Allows Peer Store Freeze via Crafted Discovery Messages — (File: `network/src/peer_store/peer_store_impl.rs`)

## Summary

The group-based eviction path in `check_purge` uses `> 4` at line 378 when selecting eviction candidates from network-segment groups. An attacker can send crafted `Nodes` discovery messages advertising exactly 4 public IP addresses across 4096 distinct `/16` subnets, filling all 16384 `ADDR_COUNT_LIMIT` slots. Once full, `check_purge` produces zero eviction candidates and permanently returns `PeerStoreError::EvictionFailed`, freezing the peer store and blocking all new peer address discovery.

## Finding Description

**Root cause — line 378:**
The comment at line 338 documents the intent as "more than 4 peer," and the eviction logic at line 378 uses strict `> 4`. A group of exactly 4 entries returns `None` from `flat_map`, contributing nothing to `candidate_peers`. With 4096 groups each of size exactly 4, the entire second eviction stage produces an empty candidate list.

**Step 1 (non-connectable eviction) is bypassed:**
Fresh entries inserted by `add_addr` have `last_connected_at_ms = 0` and `attempts_count = 0`. With `attempts_count = 0 < ADDR_MAX_RETRIES (3)`, the first non-connectable guard in `is_connectable` does not fire. With `attempts_count = 0 < ADDR_MAX_FAILURES (10)`, the second guard does not fire either. All fresh entries return `true` from `is_connectable`, so step 1 evicts nothing.

**`AddrManager::add` resets `attempts_count` on re-advertisement:**
When the attacker re-sends the same addresses, the new `AddrInfo` has `last_connected_at_ms = 0`. Since `0 >= 0` is true at `addr_manager.rs:29`, the existing entry is overwritten with `attempts_count = 0`, preventing natural expiry via failed dial accumulation.

**`EvictionFailed` is silently swallowed:**
The error is logged only at `debug!` level in `discovery/mod.rs`. No rate-limiting, no session penalty, no operator alert.

**Entry point — P2P Discovery protocol:**
With `MAX_ADDR_TO_SEND = 1000` items and `MAX_ADDRS = 3` per item, one non-announce `Nodes` message delivers up to 3000 addresses. Filling 16384 slots requires approximately 6 separate connections.

**`is_valid_addr` does not prevent the attack:**
Only private/loopback IPs are filtered via `is_reachable`. Any public IP across 4096 distinct `/16` subnets passes. The attacker does not need to own or control these IPs.

**Network group is `/16` subnet:**
`Group::IP4([bits[0], bits[1]])` in `network_group.rs` confirms grouping by first two octets of IPv4. IPv4 space provides 65536 possible `/16` subnets; only 4096 are needed.

## Impact Explanation

Once the store is filled with 4096 groups of exactly 4 connectable addresses, every subsequent `add_addr` call returns `Err(PeerStoreError::EvictionFailed)`. The node's peer store is frozen: no legitimate peer addresses discovered via the network can be recorded. Peer discovery degrades to zero new entries until the node is restarted or the peer store file is manually deleted. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation

The attacker requires only one accepted inbound or outbound P2P connection per ~3000 addresses injected. No proof-of-work, no key material, and no privileged access are required. The crafted addresses are arbitrary public IPs; the attacker does not need to control them. Approximately 6 connections are sufficient to fill the store. The attack is repeatable and maintainable indefinitely by periodically re-advertising the same addresses to reset `attempts_count` via the `AddrManager::add` overwrite path.

## Recommendation

Change the strict inequality to `>=` at line 378 of `network/src/peer_store/peer_store_impl.rs`:

```rust
// Before
if addrs.len() > 4 {
// After
if addrs.len() >= 4 {
```

This ensures any group at or above the per-subnet cap of 4 is eligible for eviction, closing the off-by-one gap and aligning the code with the documented intent at line 338.
