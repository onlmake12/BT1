### Title
`MAX_DEPOSIT_LIMIT` Bypass in `addFunds` for Permanent Subscriptions — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `addFunds` function in `Scheduler.sol` validates only the incoming `msg.value` against `MAX_DEPOSIT_LIMIT`, not the cumulative subscription balance. Because permanent subscriptions prohibit fund withdrawal, an attacker can call `addFunds` repeatedly with amounts just below the limit to permanently lock an arbitrarily large amount of ETH in a single subscription, bypassing the 100 ETH cap entirely.

---

### Finding Description

`MAX_DEPOSIT_LIMIT` (100 ETH) is intended to cap how much ETH can be deposited into a permanent subscription. The check is applied in both `createSubscription` and `addFunds`:

**`createSubscription` (line 48):**
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**`addFunds` (line 613):**
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

Both checks validate only the *new deposit* (`msg.value`), not the *resulting cumulative balance* (`status.balanceInWei + msg.value`). This is the same class of error as the reported `extend_expiration` bug: the validation uses the wrong reference point (the delta, not the absolute post-operation value).

Since `addFunds` has no `onlyManager` modifier — confirmed by the `testAnyoneCanAddFunds` test — **any caller** can add funds to any permanent subscription. A caller can invoke `addFunds` with `MAX_DEPOSIT_LIMIT` (100 ETH) per call, accumulating a balance of `N × 100 ETH` with `N` calls.

`withdrawFunds` unconditionally reverts for permanent subscriptions:
```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

So all excess ETH is permanently locked.

The existing test `testPermanentSubscriptionDepositLimit` only tests that a *single* deposit exceeding `MAX_DEPOSIT_LIMIT` reverts. It does not test multiple sub-limit deposits that cumulatively exceed the cap, leaving the bypass undetected.

---

### Impact Explanation

- The `MAX_DEPOSIT_LIMIT` invariant for permanent subscriptions is completely bypassed.
- An attacker (or the subscription manager) can permanently lock an arbitrarily large amount of ETH in a single permanent subscription.
- Because permanent subscriptions cannot be updated or have funds withdrawn, the excess ETH is irrecoverably locked in the contract until consumed by keeper fees — which, for a 100× over-funded subscription, could take an extremely long time.
- This violates the protocol's intended economic controls and can be used to grief the contract by permanently locking large sums.

---

### Likelihood Explanation

- The entry path is `addFunds(subscriptionId)`, which is `external payable` with no access control.
- Any unprivileged caller with ETH can trigger this on any existing permanent subscription.
- The attack requires no special permissions, no leaked keys, and no governance majority.
- The only prerequisite is the existence of a permanent subscription, which any user can create.

---

### Recommendation

Change the `addFunds` check to validate the **post-deposit cumulative balance**, not just the incoming `msg.value`:

```solidity
// Before (incorrect — checks only the new deposit):
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// After (correct — checks the resulting total balance):
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Apply the same fix to `createSubscription` for consistency (though the initial deposit check is functionally correct since `balanceInWei` starts at 0, making it robust to future refactors is still advisable).

---

### Proof of Concept

```solidity
function testBypassMaxDepositLimit() public {
    // Create a permanent subscription at exactly MAX_DEPOSIT_LIMIT
    SchedulerStructs.SubscriptionParams memory params =
        createDefaultSubscriptionParams(2, address(reader));
    params.isPermanent = true;

    uint256 maxLimit = scheduler.MAX_DEPOSIT_LIMIT(); // 100 ETH
    vm.deal(address(this), maxLimit * 3);

    uint256 subId = scheduler.createSubscription{value: maxLimit}(params);

    // Each addFunds call with maxLimit passes the check (msg.value == MAX_DEPOSIT_LIMIT)
    scheduler.addFunds{value: maxLimit}(subId);
    scheduler.addFunds{value: maxLimit}(subId);

    (, SchedulerStructs.SubscriptionStatus memory status) =
        scheduler.getSubscription(subId);

    // Balance is now 3x MAX_DEPOSIT_LIMIT — limit bypassed
    assertEq(status.balanceInWei, maxLimit * 3);

    // Withdrawal is impossible — funds are permanently locked
    vm.expectRevert(SchedulerErrors.CannotUpdatePermanentSubscription.selector);
    scheduler.withdrawFunds(subId, 1);
}
```
