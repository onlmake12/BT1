### Title
Missing Parameter Validation in `set_parameters` Allows Invalid `m`/`z` Values, Breaking OIS Pool Cap Accounting - (File: `pythnet/stake_caps_parameters/programs/stake_caps_parameters/src/lib.rs`)

---

### Summary

The `set_parameters` instruction in the `stake_caps_parameters` Solana program accepts the critical OIS pool cap parameters `m` and `z` without any sanity or threshold checks. Setting `z = 0` causes division by zero in the pool cap formula `C_p = M · Σ(1 / max(n_s, Z))`, and setting `m = 0` collapses all pool caps to zero. Additionally, setting `current_authority = Pubkey::default()` (all zeros) silently opens the parameter account to writes by any signer, since the access guard explicitly permits the default pubkey as a valid authority.

---

### Finding Description

The sole instruction in the `stake_caps_parameters` program is:

```rust
pub fn set_parameters(ctx: Context<SetParameters>, parameters: Parameters) -> Result<()> {
    let stored_parameters = &mut ctx.accounts.parameters;
    require!(
        ctx.accounts.signer.key() == stored_parameters.current_authority
            || stored_parameters.current_authority == Pubkey::default(),
        ErrorCode::WrongAuthority
    );
    **stored_parameters = parameters;
    Ok(())
}
```

The `Parameters` struct contains three fields:

```rust
pub struct Parameters {
    pub current_authority: Pubkey,
    pub m: u64,
    pub z: u64,
}
```

After the authority check, the entire struct is written verbatim with no validation on `m` or `z`. [1](#0-0) 

These two parameters directly govern the OIS pool cap formula documented as:

```
C_p = M · Σ_{s ∈ Symbols_p} ( 1 / max(n_s, Z) )
```

where `M` maps to `m` and `Z` maps to `z`. [2](#0-1) 

**Issue 1 — `z = 0` causes division by zero.** The denominator `max(n_s, Z)` becomes `max(n_s, 0)`. For any symbol with zero active publishers (`n_s = 0`), the denominator is zero, producing a divide-by-zero in the off-chain publisher caps computation. The resulting `PublisherStakeCapsMessage` either fails to be produced or carries corrupted cap values that are then submitted on-chain and consumed by the integrity pool.

**Issue 2 — `m = 0` zeroes all pool caps.** With `M = 0`, every publisher's pool cap collapses to zero regardless of how many symbols they publish. Any staker who entered a pool based on a non-zero cap is now fully "above cap" and exposed to slashing with no reward offset.

**Issue 3 — `current_authority = Pubkey::default()` opens writes to any signer.** The access guard contains the clause `|| stored_parameters.current_authority == Pubkey::default()`. If the authority ever writes `current_authority = Pubkey::default()` (the zero pubkey, which is the Rust default for `Pubkey`), the account becomes writable by any transaction sender on the next call. An unprivileged attacker can then immediately call `set_parameters` with `m = 0` or `z = 0`. [3](#0-2) 

The staking frontend reads `m` and `z` directly from this account and uses them to display pool capacities and APY estimates to users making staking decisions. [4](#0-3) 

---

### Impact Explanation

- **`z = 0`**: The off-chain publisher caps service cannot compute valid caps; it either crashes or emits zero/infinite caps for all publishers. The on-chain `PublisherStakeCapsMessage` becomes invalid, breaking the entire OIS reward and slashing accounting for the epoch.
- **`m = 0`**: All pool caps drop to zero. Stakers who delegated based on a positive cap are now fully above-cap, meaning their stake is at slashing risk with no reward eligibility. This is a direct economic loss for delegators.
- **`current_authority = Pubkey::default()`**: Escalates to an unprivileged write path — any attacker can set `m = 0` or `z = 0` without needing the authority key.

---

### Likelihood Explanation

The authority is a single keypair operated via the CLI tool at `pythnet/stake_caps_parameters/cli/src/main.rs`. [5](#0-4)  The CLI accepts `m`, `z`, and `authority` as raw command-line arguments with no client-side validation. [6](#0-5)  A typo (e.g., `--z 0` instead of a valid value, or omitting `--authority` which defaults to the zero pubkey) is a realistic operational mistake. The `Pubkey::default()` authority scenario is particularly likely because Rust's default for `Pubkey` is all zeros, and a misconfigured CLI invocation could silently pass it.

---

### Recommendation

1. Add explicit guards in `set_parameters`:
   ```rust
   require!(parameters.z > 0, ErrorCode::InvalidZ);
   require!(parameters.m > 0, ErrorCode::InvalidM);
   require!(
       parameters.current_authority != Pubkey::default(),
       ErrorCode::InvalidAuthority
   );
   ```
2. Remove the `|| stored_parameters.current_authority == Pubkey::default()` bypass from the authority check, or restrict it strictly to the first-ever initialization (e.g., check that the account was just created via `init_if_needed`).
3. Add client-side validation in the CLI before submitting the transaction.

### Proof of Concept

**Step 1 — Authority accidentally sets `current_authority = Pubkey::default()`:**
```bash
./stake-caps-parameters-cli \
  --m 1000000 \
  --z 5 \
  --authority 11111111111111111111111111111111   # Pubkey::default()
```

**Step 2 — Any unprivileged attacker now calls `set_parameters` with `m = 0`:**
```rust
let ix = Instruction {
    program_id: stake_caps_parameters::id(),
    accounts: stake_caps_parameters::accounts::SetParameters {
        signer: attacker.pubkey(),          // any keypair
        parameters: PARAMETERS_ADDRESS,
        system_program: system_program::id(),
    }.to_account_metas(None),
    data: stake_caps_parameters::instruction::SetParameters {
        parameters: Parameters {
            current_authority: attacker.pubkey(),
            m: 0,   // collapses all pool caps to zero
            z: 0,   // causes division by zero in cap formula
        },
    }.data(),
};
```

The `require!` check passes because `stored_parameters.current_authority == Pubkey::default()` is true. All OIS pool caps are now zero, and the cap computation service panics on division by zero for any symbol with `n_s = 0`. Stakers who entered pools based on prior non-zero caps are now fully above-cap and exposed to slashing with no reward. [1](#0-0)

        keypair,

        m,
        z,

        signer: keypair.pubkey(),

    let rpc_client = RpcClient::new_with_commitment(rpc_url, CommitmentConfig::confirmed());

    let instruction = Instruction {
        program_id: stake_caps_parameters::id(),

                m,
                z,
                current_authority: authority,
            },
        }
        .data(),
    };

    let mut transaction = Transaction::new_with_payer(&[instruction], Some(&keypair.pubkey()));
    transaction.sign(&[&keypair], rpc_client.get_latest_blockhash().unwrap());

        .send_and_confirm_transaction_with_spinner(&transaction)
        .unwrap();
