Audit Report

## Title
Unbounded Block Iteration via `difficulty_boundary=0` + `difficulties=[]` Bypasses Size Guard — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The `GetLastStateProofProcess::execute` handler contains a size guard that counts only request-parameter entries (`difficulties.len() + last_n_blocks * 2`), not the actual number of blocks that will be fetched. An unauthenticated peer can send a crafted `GetLastStateProof` message with `difficulties=[]`, `difficulty_boundary=U256::zero()`, and `last_n_blocks=0` to force the server to iterate, fetch, and serialize every block in the chain with no secondary cap, exhausting CPU, memory, and I/O.

## Finding Description

**Size guard is bypassed.**
The upfront limit at L201–205 evaluates to `0 + 0*2 = 0`, which is not `> 1000`. The check passes unconditionally. [1](#0-0) 

**All per-field difficulty validation checks pass on empty input.**
- `difficulties.windows(2).any(...)` — empty slice, `any()` returns `false`.
- `difficulties.last().map(...).unwrap_or(false)` — `last()` is `None`, `unwrap_or(false)` is `false`.
- `difficulties.first()` — `None`, so the `if let Some(start_difficulty)` block is skipped entirely. [2](#0-1) 

**Zero boundary causes `difficulty_boundary_block_number = start_block_number`.**
Since `last_block_number - start_block_number > 0 = last_n_blocks`, the `else` branch is entered at L298. `get_first_block_total_difficulty_is_not_less_than` is called with `min_total_difficulty = U256::zero()`. At L30–32, `start_total_difficulty >= U256::zero()` is always true for an unsigned type, so the function returns `(start_block_number, ...)` immediately. [3](#0-2) [4](#0-3) 

**Adjustment guard never fires.**
The guard `last_block_number - difficulty_boundary_block_number < last_n_blocks` becomes `last_block_number - 0 < 0`, which is always false for `u64`. No adjustment occurs. [5](#0-4) 

**`last_n_numbers` covers the entire chain.**
With `difficulty_boundary_block_number = 0` and `start_block_number = 0`, the condition `difficulty_boundary_block_number > 0` at L321 is false, so the `else` branch at L345 returns `(Vec::new(), last_n_numbers)` where `last_n_numbers = (0..last_block_number)` — every block in the chain. [6](#0-5) [7](#0-6) 

**`complete_headers` iterates over all blocks with no cap.**
For each block number in `block_numbers`, the server performs three expensive operations: `get_ancestor` (O(N) chain traversal), `get_block` (full DB fetch), and `chain_root_mmr(*number - 1).get_root()` (MMR root computation). There is no secondary size check before or inside this loop. [8](#0-7) [9](#0-8) 

## Impact Explanation

A single unauthenticated peer can force the light-client server to perform O(N) database lookups, O(N) MMR root computations, and O(N) memory allocation for every block in the chain. On a chain with millions of blocks, this exhausts CPU, memory, and I/O on the server process. Multiple concurrent requests from different peer connections amplify the effect. This matches the **High** impact category: *Vulnerabilities which could easily crash a CKB node* and *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation

The attack requires no credentials, no proof-of-work, and no prior state. The attacker only needs the current tip hash, which is publicly observable from any chain explorer or peer. The crafted message is minimal (empty difficulties array, zero boundary scalar, zero `last_n_blocks`). The attack is repeatable from any number of peer connections simultaneously. No rate-limiting or per-peer request throttling is present in this code path.

## Recommendation

1. Add a secondary size check on the computed `block_numbers` before calling `complete_headers`, capping it at `GET_LAST_STATE_PROOF_LIMIT`:
   ```rust
   if block_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
       return StatusCode::InvalidRequest.with_context("too many blocks in response");
   }
   ```
2. Reject `difficulty_boundary == U256::zero()` explicitly when entering the `else` branch (i.e., when `last_block_number - start_block_number > last_n_blocks`), since a zero boundary is semantically meaningless and enables this bypass.
3. Consider capping `reorg_last_n_numbers` independently, as it is also derived from `last_n_blocks` without a direct size guard.

## Proof of Concept

```
Setup: CKB node with light-client server enabled, chain height N (e.g., 10,000 blocks).

Craft GetLastStateProof message:
  last_hash           = current tip hash (publicly known)
  start_hash          = genesis hash
  start_number        = 0
  last_n_blocks       = 0
  difficulty_boundary = U256::zero() (32 zero bytes)
  difficulties        = [] (empty)

Send from any unauthenticated peer connection.

Trace through execute():
  1. Size guard: 0 + 0*2 = 0 <= 1000 → passes.
  2. Difficulty validations: all skipped (empty slice).
  3. reorg_last_n_numbers: start_block_number == 0 → empty.
  4. last_block_number - 0 > 0 → enters else branch.
  5. get_first_block_total_difficulty_is_not_less_than(0, N, 0):
       start_total_difficulty >= 0 always true → returns (0, ...).
  6. difficulty_boundary_block_number = 0.
  7. Adjustment guard: N - 0 < 0 → false (u64), no adjustment.
  8. difficulty_boundary_block_number > 0 → false → else branch.
  9. last_n_numbers = (0..N) → N entries.
  10. block_numbers = N entries, no cap check.
  11. complete_headers: N × (get_ancestor + get_block + MMR root).

Expected: server rejects with InvalidRequest or returns at most last_n_blocks headers.
Actual: server processes all N blocks, exhausting CPU/memory/I/O.

Repeat from multiple connections to amplify resource exhaustion.
```
