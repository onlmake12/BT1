Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Attacker-Controlled `(from, to)` PeerIds — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
The `HolePunching` protocol handler maintains a `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` backed by `governor::state::keyed::HashMapStateStore`. The `(from, to)` key components are fully attacker-controlled fields from the message body. The `retain_recent()` cleanup is called only in `disconnected()`, never in `notify()` or `received()`. An attacker who maintains a persistent session and sends messages with unique `(from, to)` pairs causes the internal HashMap to grow without bound, exhausting heap memory and crashing the node.

## Finding Description
The `RateLimiter<T>` type alias at lines 31–35 of `mod.rs` uses `HashMapStateStore<T>`, and `forward_rate_limiter` is keyed by `(PeerId, PeerId, u32)` (line 46). [1](#0-0) 

In `connection_request.rs` lines 132–143, the `forward_rate_limiter.check_key()` call uses `content.from` and `content.to` — both parsed directly from the attacker-supplied molecule message body. Every call with a new `(from, to)` pair inserts a fresh entry into the HashMap. [2](#0-1) 

The `retain_recent()` call exists only in `disconnected()` (lines 67–68), which fires only when a peer disconnects. [3](#0-2) 

The `notify()` handler (lines 169–175), which fires every 5 minutes via `CHECK_INTERVAL`, cleans only `pending_delivered` and `inflight_requests` — it never calls `retain_recent()` on either rate limiter. [4](#0-3) 

The outer rate limiter (lines 95–107) limits to 30 messages/sec per `(session_id, item_id)`. Since `item_id` is a fixed enum discriminant (0 for `ConnectionRequest`), this merely sets the insertion rate to 30 entries/sec — it does not bound the total map size. [5](#0-4) 

The `route` self-check at lines 127–130 is bypassed by sending an empty `route` field, which is valid per the molecule schema. [6](#0-5) 

The `forward_rate_limiter` quota is 1/sec per key (lines 254–257). The first `check_key()` for any new `(from, to, item_id)` triple always succeeds (no prior state), inserting a new entry. Since the attacker never reuses a pair, every message passes and inserts. [7](#0-6) 

## Impact Explanation
This is a **High** severity finding: **Vulnerabilities which could easily crash a CKB node**. A single persistent attacker session sending 30 unique-keyed `ConnectionRequest` messages per second causes unbounded heap growth (~90–100 bytes per entry). After hours of sustained attack, the node process OOMs and crashes. With multiple concurrent sessions (up to the node's max peer count), the growth rate scales linearly, reducing the time to crash proportionally.

## Likelihood Explanation
The attacker requires only a standard P2P connection — no privileges, no proof-of-work, no keys. The `HolePunching` protocol is a standard registered protocol. The attack requires only that the attacker not disconnect, which is trivially maintained. The outer rate limiter does not prevent the attack; it merely sets the growth rate. The `route` self-check is trivially bypassed with an empty route field.

## Recommendation
Add `self.forward_rate_limiter.retain_recent()` and `self.rate_limiter.retain_recent()` calls inside the `notify()` handler, which already fires on the 5-minute `CHECK_INTERVAL`. This bounds the map size to entries active within the last rate-limit window, regardless of whether peers disconnect. [4](#0-3) 

## Proof of Concept
1. Connect to a target CKB node with `HolePunching` enabled.
2. In a loop at 30 messages/sec (staying under the outer limiter):
   - Generate a fresh random `PeerId` for `from` and a fresh random `PeerId` for `to`.
   - Construct a valid `ConnectionRequest` molecule message with `route = []` (empty, bypassing the self-peer-id check) and at least one valid `listen_addrs` entry.
   - Send the message over the `HolePunching` protocol stream.
3. Never disconnect.
4. After `T` seconds, the `forward_rate_limiter` `HashMapStateStore` contains `30*T` entries. Assert map size grows linearly with `T` and is not bounded by the number of active sessions.
5. After sufficient time (hours to days depending on available RAM), the node process OOMs and crashes.
