Audit Report

## Title
Unbounded OrphanBlockPool Growth via PoW-Free Block Flooding — (`chain/src/utils/orphan_block_pool.rs`, `chain/src/chain_service.rs`)

## Summary
An unprivileged remote peer can flood a CKB node with syntactically valid `SendBlock` messages whose blocks carry random parent hashes and no Eaglesong PoW work. These blocks bypass `non_contextual_verify` (which performs no PoW check), are written to RocksDB, and are inserted into `OrphanBlockPool` which enforces no hard size cap. Sustained flooding causes unbounded heap and disk growth, ultimately crashing the node via OOM kill or disk exhaustion.

## Finding Description

**Root cause 1 — `non_contextual_verify` omits PoW.**

`ChainService::non_contextual_verify` invokes only `BlockVerifier` and `NonContextualBlockTxsVerifier`: [1](#0-0) 

`BlockVerifier::verify` runs only structural checks — proposals limit, byte size, cellbase, duplicates, and merkle root — with no `PowVerifier` call: [2](#0-1) 

`PowVerifier` is invoked exclusively inside `HeaderVerifier`, which is a *contextual* verifier requiring the parent to be known: [3](#0-2) 

The `SendBlock` sync path (`BlockProcess::execute`) performs no PoW check — it calls `new_block_received` for deduplication only, then immediately forwards to `asynchronous_process_remote_block`: [4](#0-3) 

**Root cause 2 — `OrphanBlockPool::insert` has no hard size limit.**

`ORPHAN_BLOCK_SIZE` is set to `BLOCK_DOWNLOAD_WINDOW = 8192` and passed to `HashMap::with_capacity`, which is an advisory pre-allocation hint, not a cap: [5](#0-4) 

`InnerPool::insert` performs zero size checks before inserting into `blocks`, `parents`, and `leaders`: [6](#0-5) 

**Root cause 3 — Expiry cleanup is ineffective against a live attacker.**

`clean_expired_blocks` only removes blocks where `epoch_number + EXPIRED_EPOCH (6) < tip_epoch`. Blocks crafted with the current epoch number survive approximately 6 epochs (~24 hours on mainnet) before expiry: [7](#0-6) 

**Root cause 4 — `insert_block` (DB write) precedes orphan pool insertion.**

`asynchronous_process_block` writes the block to RocksDB via `insert_block` *before* calling `orphan_broker.process_lonely_block`, meaning disk exhaustion occurs in parallel with heap growth: [8](#0-7) 

**Attack path:**
```
Peer sends SendBlock (P2P sync protocol)
  → Synchronizer::try_process → BlockProcess::execute
  → new_block_received (dedup only, no PoW check)
  → asynchronous_process_remote_block
  → process_block_rx channel (bounded(24), throttles rate but does not cap total)
  → ChainService::asynchronous_process_block
  → non_contextual_verify  ← passes without PoW
  → insert_block           ← writes to RocksDB (disk exhaustion side-effect)
  → orphan_broker.process_lonely_block
  → orphan_blocks_broker.insert()  ← no size limit, unbounded growth
``` [9](#0-8) 

## Impact Explanation

Each orphan insertion allocates entries in three `HashMap`/`HashSet` structures (`blocks`, `parents`, `leaders`) with no eviction policy and no hard cap, causing unbounded RSS growth and ultimately an OOM kill of the node process. Concurrently, `insert_block` writes each crafted block to RocksDB before orphan insertion, exhausting disk space in parallel. This matches the **High (10001–15000 points)** impact: *"Vulnerabilities which could easily crash a CKB node."*

## Likelihood Explanation

The attack requires no PoW computation, no privileged access, and no Sybil capability. A single peer connection over the standard sync protocol is sufficient. Crafting a structurally valid block (correct cellbase `since` field equal to block number, valid merkle root, within byte limit) is computationally trivial. The `process_block_rx` channel (bounded 24) throttles throughput to the chain service's processing rate but does not bound total orphan pool size. The attack is repeatable and deterministic. [10](#0-9) 

## Recommendation

1. **Enforce a hard size limit in `InnerPool::insert`**: reject or evict the oldest leader's subtree when `parents.len() >= capacity`.
2. **Add `PowVerifier` to `non_contextual_verify`** (or to `BlockVerifier`) so blocks without valid PoW are rejected before reaching the orphan pool or the database.
3. **Do not call `insert_block`** (DB write) before the block's parent is known to be reachable.
4. **Add per-peer rate limiting** on `SendBlock` messages in the `Synchronizer`, analogous to any existing rate limiting in the `Relayer`.

## Proof of Concept

```rust
// Connect as a normal peer via the Sync protocol
// Craft N blocks each with a unique random parent hash, valid cellbase, and correct merkle root
for _ in 0..N {
    let random_parent = Byte32::from(rand::random::<[u8; 32]>());
    let block = BlockBuilder::default()
        .parent_hash(random_parent)
        .number(1u64.pack())
        .epoch(EpochNumberWithFraction::new(current_epoch, 0, 1000).pack())
        .timestamp(unix_time_as_millis().pack())
        // valid cellbase (since == block number), correct merkle root, no PoW
        .build_unchecked();
    let content = packed::SendBlock::new_builder().block(block.data()).build();
    let message = packed::SyncMessage::new_builder().set(content).build();
    peer.send(SupportProtocols::Sync, message.as_bytes());
}
// Assert: orphan_blocks_broker.len() grows to N with no eviction
// Assert: node RSS grows proportionally; OOM kill at large N
// Assert: RocksDB data directory grows proportionally
```

Submitting `BLOCK_DOWNLOAD_WINDOW * 10 = 81920` blocks is sufficient to demonstrate growth well past the nominal `with_capacity(8192)` advisory hint. [11](#0-10)

    fn insert(&mut self, lonely_block: LonelyBlockHash) {
        let hash = lonely_block.hash();
        let parent_hash = lonely_block.parent_hash();
        self.blocks
            .entry(parent_hash.clone())
            .or_default()
            .insert(hash.clone(), lonely_block);
        // Out-of-order insertion needs to be deduplicated
        self.leaders.remove(&hash);
        // It is a possible optimization to make the judgment in advance,
        // because the parent of the block must not be equal to its own hash,
        // so we can judge first, which may reduce one arc clone
        if !self.parents.contains_key(&parent_hash) {
            // Block referenced by `parent_hash` is not in the pool,
            // and it has at least one child, the new inserted block, so add it to leaders.
            self.leaders.insert(parent_hash.clone());
        }
        self.parents.insert(hash, parent_hash);
