Audit Report

## Title
Random Eviction with No Per-Peer Quota Allows Any Relay Peer to Exhaust the OrphanPool — (`tx-pool/src/component/orphan.rs`)

## Summary
`OrphanPool` is hard-capped at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100`. When full, `limit_size()` evicts via `self.entries.keys().next()` — arbitrary HashMap iteration order with no fee-rate ordering and no per-peer accounting. Any connected relay peer can fill all 100 slots with zero-cost orphan transactions, causing legitimate orphan transactions to be evicted and never auto-promoted when their parent arrives.

## Finding Description

The pool cap and expiry are confirmed at: [1](#0-0) 

The `Entry` struct records the submitting peer, but `limit_size()` never consults it: [2](#0-1) 

The eviction loop uses `self.entries.keys().next()` — effectively arbitrary for a `HashMap`: [3](#0-2) 

`add_orphan_tx` inserts with no per-peer quota check before calling `limit_size()`: [4](#0-3) 

The relay path that feeds the orphan pool: `after_process()` calls `add_orphan()` when `is_missing_input(reject)` is true: [5](#0-4) 

`add_orphan()` passes directly to `add_orphan_tx` with no per-peer guard: [6](#0-5) 

**Existing guards are insufficient:**

The `add_ask_for_txs` per-peer soft limit of `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767` only activates after the global `unknown_tx_hashes` queue reaches 50,000 entries or `peers * 32767` entries: [7](#0-6) 

The attacker only needs to fill 100 orphan pool slots — far below the threshold where the per-peer hash limit activates. The `TransactionsProcess` filter does require `requesting_peer() == Some(self.peer)`, but this is satisfied naturally: the attacker announces 100 hashes, the node issues `GetRelayTransactions`, and the attacker responds with orphan transactions whose parents are absent from the chain: [8](#0-7) 

**Attack flow:**
1. Attacker connects as a relay peer and sends `RelayTransactionHashes` with 100 unknown hashes.
2. Victim node adds them to `unknown_tx_hashes` and issues `GetRelayTransactions`.
3. Attacker responds with 100 transactions whose parent cells are absent — each lands in `OrphanPool` via `add_orphan()`.
4. Pool is now at capacity (100 entries). A legitimate user's orphan transaction triggers `limit_size()`, which evicts an arbitrary entry.
5. Attacker continuously re-announces and re-fills, keeping the pool saturated. The legitimate orphan is never present when its parent arrives, so `process_orphan_tx` never promotes it.

## Impact Explanation

This is a **bad design that can cause CKB transaction processing denial of service with trivially low cost**, matching the High impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* A single attacker peer with no funds and no privileged access can persistently prevent legitimate child transactions from being auto-promoted on any targeted node. Applied simultaneously to multiple nodes — which is trivially cheap since no funds are required — this degrades the network's ability to process dependent transaction chains, forcing repeated user resubmissions with no guarantee of success while the attacker maintains the attack.

## Likelihood Explanation

Any connected relay peer can execute this. No keys, no funds, and no special privileges are required — only a single persistent peer connection. The pool size of 100 is trivially exhausted with 100 crafted orphan transactions. The `ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL` means attacker entries persist for a very long time without refresh: [9](#0-8) 

The attack is indefinitely repeatable because evicted entries are simply dropped with no record preventing re-insertion.

## Recommendation

1. **Per-peer eviction priority**: Track per-peer orphan counts. When `limit_size()` must evict, select the victim from the peer with the most entries (analogous to Bitcoin Core's orphan pool strategy). The `peer` field already exists in `Entry` — it just needs to be consulted during eviction.
2. **Per-peer submission cap**: Enforce a hard limit on how many orphan entries a single peer may hold in the pool at any time, analogous to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` used in the relay hash tracking logic.
3. **Fee-rate ordering**: Evict the lowest-fee-rate orphan rather than an arbitrary one, so cheap attacker transactions are preferentially removed.

## Proof of Concept

**Unit test** (against `tx-pool/src/component/orphan.rs`):

```rust
#[test]
fn test_orphan_pool_exhaustion_evicts_legitimate_tx() {
    let mut pool = OrphanPool::new();
    let attacker_peer: PeerIndex = 1.into();
    let victim_peer: PeerIndex = 2.into();

    // Fill pool with 100 attacker orphans
    for i in 0..100u32 {
        let fake_parent = Byte32::from_slice(&[i as u8; 32]).unwrap();
        let tx = build_tx(vec![(&fake_parent, 0)], 1);
        pool.add_orphan_tx(tx, attacker_peer, 0);
    }
    assert_eq!(pool.len(), 100);

    // Legitimate user submits orphan — triggers eviction
    let legitimate_parent = Byte32::from_slice(&[0xffu8; 32]).unwrap();
    let legitimate_tx = build_tx(vec![(&legitimate_parent, 0)], 1);
    let legitimate_id = legitimate_tx.proposal_short_id();
    pool.add_orphan_tx(legitimate_tx, victim_peer, 0);

    // Pool stays at 100; legitimate tx may have been evicted
    assert_eq!(pool.len(), 100);
    // With attacker continuously refilling, legitimate_id is not reliably present:
    assert!(!pool.contains_key(&legitimate_id),
        "legitimate orphan was evicted by random eviction with no per-peer protection");
}
```

**Integration path**: Connect a test relay peer, announce 100 unknown hashes, respond with 100 orphan transactions, then submit a legitimate child transaction via a second peer and verify it is absent from the orphan pool when its parent is later submitted.

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
