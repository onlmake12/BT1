### Title
Softfork Activation Locks In Below Configured Threshold Due to Floor Division in `threshold_number` — (`File: spec/src/versionbits/mod.rs`)

---

### Summary

The `threshold_number` helper in CKB's versionbits softfork activation logic uses integer floor division to compute the minimum number of signaling blocks required for lock-in. Because epoch lengths are variable, `total * numer` is frequently not divisible by `denom`, causing the computed threshold to be strictly lower than the configured ratio. A miner coalition that controls fewer blocks than the true threshold can still lock in a softfork, directly analogous to the Audius "ties are approved" bug where a tie (equal stake) incorrectly triggered approval.

---

### Finding Description

In `spec/src/versionbits/mod.rs`, the function `threshold_number` computes the minimum signaling block count as:

```rust
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    length
        .checked_mul(threshold.numer())
        .and_then(|ret| ret.checked_div(threshold.denom()))
}
```

This is `floor(length × numer / denom)`.

The result is used at line 343:

```rust
let threshold_number = threshold_number(total, self.threshold())?;
if count >= threshold_number {
    next_state = ThresholdState::LockedIn;
}
```

The semantic intent of `threshold` (documented as "minimum ratio of blocks per period") requires that `count / total >= numer / denom`, which is equivalent to `count >= ceil(total × numer / denom)`. The code uses `floor` instead of `ceil`. Whenever `total × numer` is not evenly divisible by `denom`, `floor < ceil`, so the effective activation threshold is lower than the configured ratio.

**Concrete numeric examples:**

| Threshold | total | floor result | ceil (correct) | Effective % | Configured % |
|-----------|-------|-------------|----------------|-------------|--------------|
| 3/4 (75%) | 5     | 3           | 4              | 60%         | 75%          |
| 8/10 (80%)| 11    | 8           | 9              | 72.7%       | 80%          |
| 3/4 (75%) | 9     | 6           | 7              | 66.7%       | 75%          |

The mainnet threshold is `LC_MAINNET_ACTIVATION_THRESHOLD = Ratio::new(8, 10)` and testnet is `TESTNET_ACTIVATION_THRESHOLD = Ratio::new(3, 4)`. [1](#0-0) 

The `total` variable is the sum of block counts across `period` epochs, and because CKB epoch lengths adjust dynamically based on orphan rate, `total` is rarely a clean multiple of `denom`. [2](#0-1) 

---

### Impact Explanation

The softfork activation threshold is a consensus security parameter. It is set to ensure a genuine supermajority of miners support a protocol change before it locks in. Floor division systematically underestimates the required count by up to `denom − 1` blocks per evaluation window. For the mainnet threshold of 8/10 with large epoch sizes (~1800 blocks, `total` ≈ 3600 per `period=2`), the discrepancy is at most 1 block, which is negligible in practice. However, for testnet (threshold 3/4, smaller epoch sizes) or any custom chain with small epoch lengths, the discrepancy can be several percentage points — enough for a miner coalition that falls short of the true threshold to still achieve lock-in. Once locked in, the softfork transitions to `Active` and its new consensus rules are enforced on all nodes. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

The condition is triggered whenever `total × numer mod denom ≠ 0`, which is the common case for variable-length epochs. Any miner or coordinated pool that can produce blocks in the signaling window and controls a fraction of hashrate between `floor(total × numer / denom) / total` and the true threshold `numer / denom` can exploit this. No special access, leaked keys, or majority hashpower is required — only the ability to include a version bit in produced blocks, which is a standard miner capability. The entry point is the block relay and chain verification path triggered by any block producer. [5](#0-4) 

---

### Recommendation

Replace floor division with ceiling division in `threshold_number`, or eliminate the helper entirely and use a cross-multiplication comparison that avoids rounding:

```rust
// Option A: ceiling division
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    length
        .checked_mul(threshold.numer())
        .map(|ret| (ret + threshold.denom() - 1) / threshold.denom())
}

// Option B: avoid division entirely in the caller
// Replace: count >= threshold_number
// With:    count * self.threshold().denom() >= total * self.threshold().numer()
```

Option B is preferable because it is exact (no rounding at all) and directly expresses the ratio comparison `count/total >= numer/denom`. [5](#0-4) 

---

### Proof of Concept

Given `threshold = Ratio::new(3, 4)` (75%) and a signaling window where `total = 5` blocks:

1. `threshold_number(5, Ratio::new(3, 4))` computes `floor(5 × 3 / 4) = floor(3.75) = 3`.
2. The check `count >= 3` passes with only 3 out of 5 blocks signaling (60%).
3. The correct ceiling check `ceil(3.75) = 4` would require 4 out of 5 blocks (80%).
4. A miner controlling 60% of blocks in that window — well below the 75% threshold — successfully locks in the softfork.

This is reachable on testnet today (threshold 3/4, epoch lengths as small as 4 blocks in early chain or test environments) and on any custom chain. On mainnet with large epoch sizes the gap shrinks to ≤1 block, but the root cause is present in all configurations. [6](#0-5) [5](#0-4)
