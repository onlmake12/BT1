### Title
Deprecated `parse_and_verify_le_ecdsa_update` Aborts on Valid Channel=4 (FixedRate1000ms) Updates, Freezing Consumer Protocol Fund Operations — (`lazer/contracts/sui/sources/channel.move`, `update.move`, `pyth_lazer.move`)

### Summary
A validly-signed Lazer update carrying channel byte `4` (FixedRate1000ms) — a value the Lazer server legitimately produces — causes an unrecoverable `EInvalidChannel` abort when processed through the deprecated `parse_and_verify_le_ecdsa_update` (v1) entry point. Any consumer protocol whose price-gated fund operations depend exclusively on this entry point will have those operations permanently blocked once the Lazer server rotates to the 1000ms channel.

### Finding Description

**Step 1 — `channel_v2` accepts value 4.**
`channel_v2::from_u8` validates `value >= 1 && value <= 4`, so channel byte `4` is a fully legal, parseable value. [1](#0-0) 

**Step 2 — `channel::from_v2` only handles values 1–3.**
The deprecated `channel::from_v2` converts a `channel_v2::Channel` back to the old `channel::Channel` enum. It handles `is_real_time()` (1), `is_fixed_rate_50ms()` (2), and `is_fixed_rate_200ms()` (3), and falls through to `abort EInvalidChannel` for any other value — including `4`. [2](#0-1) 

**Step 3 — `update_v1::from_v2` calls `channel::from_v2`.**
The deprecated `update::from_v2` unconditionally calls `channel::from_v2(update.channel())`, propagating the abort. [3](#0-2) 

**Step 4 — `parse_and_verify_le_ecdsa_update` (v1 entry point) calls `update_v1::from_v2`.**
The deprecated public entry point first calls `parse_and_verify_le_ecdsa_update_v2` (which succeeds for channel=4), then immediately calls `update_v1::from_v2`, triggering the abort. [4](#0-3) 

**Step 5 — The Pyth test suite explicitly confirms this failure.**
`test_parse_and_verify_le_ecdsa_incompatible_update` uses `TEST_LAZER_UPDATE` (a real, signed 1000ms channel update) against `parse_and_verify_le_ecdsa_update` and asserts `expected_failure(abort_code = EInvalidChannel)`. The companion test `test_parse_and_verify_le_ecdsa_update_v2` uses the identical payload and succeeds, confirming the v2 path is unaffected. [5](#0-4) [6](#0-5) 

### Impact Explanation
Any consumer contract that:
1. calls `parse_and_verify_le_ecdsa_update` (the deprecated v1 path) as its sole price-update entry point, and
2. gates fund operations (withdrawals, liquidations, settlements) on the returned `update_v1::Update`,

will have every such transaction abort with `EInvalidChannel` the moment the Lazer server begins publishing on the 1000ms channel. If the consumer contract is immutable or upgrade-gated, this constitutes a **permanent freeze** of all price-dependent fund flows. Even for upgradeable contracts, the window between server rotation and contract upgrade is a period of **temporary fund freeze**.

### Likelihood Explanation
- The Lazer server already supports and publishes FixedRate1000ms (channel=4); the test data is a real signed payload from a live subscription.
- Any relayer (unprivileged) can submit this update to any consumer using the v1 path.
- No key compromise, governance attack, or privileged access is required.
- The deprecated v1 function remains `public` and callable on-chain. [7](#0-6) 

### Recommendation
1. **In `channel::from_v2`**: add a branch for `channel.is_fixed_rate_1000ms()` mapping to a new `Channel::FixedRate1000ms` variant, or return a sentinel/error type instead of aborting, so the v1 path degrades gracefully.
2. **Alternatively**, gate `parse_and_verify_le_ecdsa_update` at the entry point to reject channel=4 with a clear, documented error rather than propagating an opaque abort from deep in the conversion chain.
3. **Communicate urgently** to all known consumers of the deprecated v1 path that they must migrate to `parse_and_verify_le_ecdsa_update_v2` before the Lazer server rotates to the 1000ms channel in production.
