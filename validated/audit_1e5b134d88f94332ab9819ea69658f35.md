### Title
Missing `BLOCK_INVALID` Early-Exit in `HeaderAcceptor::accept()` Enables Repeated PoW Re-Verification DoS — (File: `sync/src/synchronizer/headers_process.rs`)

---

### Summary

The `HeaderAcceptor::accept()` function in the sync layer checks whether a header has already been validated (`HEADER_VALID`) but **never checks whether it has already been rejected** (`BLOCK_INVALID`). A developer `FIXME` comment at the exact location explicitly acknowledges this gap. Because `BLOCK_INVALID` does not overlap with `HEADER_VALID` in the bitflag layout, a previously-rejected header falls through to full re-verification — including expensive PoW re-verification — every time it is received from a peer.

---

### Finding Description

`BlockStatus` is defined as a bitflag type:

```
UNKNOWN       = 0
HEADER_VALID  = 1
BLOCK_RECEIVED= 1 | (HEADER_VALID << 1)  = 3
BLOCK_STORED  = 1 | (BLOCK_RECEIVED << 1)= 7
BLOCK_VALID   = 1 | (BLOCK_STORED << 1)  = 15
BLOCK_INVALID = 1 << 12                  = 4096
``` [1](#0-0) 

`BLOCK_INVALID` (4096) shares **no bits** with `HEADER_VALID` (1). Therefore `BlockStatus::BLOCK_INVALID.contains(BlockStatus::HEADER_VALID)` is `false`.

In `HeaderAcceptor::accept()`, the only early-exit guard is:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... return early
    return result;
}
``` [2](#0-1) 

When `status == BLOCK_INVALID`, this guard is **not taken**. The function then proceeds to:

1. `prev_block_check` — checks parent status (cheap)
2. `non_contextual_check` — calls `self.verifier.verify(self.header)` via `HeaderVerifier`, which includes **PoW verification** (expensive)
3. `version_check` — checks header version [3](#0-2) 

A header that passes PoW but fails `version_check` (e.g., `version != 0`) will be marked `BLOCK_INVALID`. On every subsequent receipt of the same header, the node re-runs the full pipeline — including PoW verification — before re-marking it `BLOCK_INVALID`. The attacker's cost is one-time (mine one valid-PoW, wrong-version header); the victim's cost is unbounded (re-verify PoW on every replay).

The `HeadersProcess` handles `SendHeaders` P2P messages from any unprivileged peer: [4](#0-3) 

---

### Impact Explanation

An unprivileged P2P peer can repeatedly send a `SendHeaders` message containing a single crafted header (valid PoW, invalid version or invalid parent chain). Each delivery forces the victim node to re-execute `HeaderVerifier::verify()` — including PoW hash verification — before the header is re-rejected. With no rate limit on `SendHeaders` in the synchronizer path and no early-exit for `BLOCK_INVALID`, the attacker can sustain a CPU-exhaustion DoS against any reachable CKB node.

---

### Likelihood Explanation

The attacker must mine one header with valid PoW that fails for a non-PoW reason (e.g., `version = 1`). This is a one-time cost. Once obtained, the header can be replayed to any number of peers indefinitely. The `FIXME` comment in the production code confirms the developers are aware of the missing guard but have not resolved it.

---

### Recommendation

Add an explicit `BLOCK_INVALID` early-exit at the top of `HeaderAcceptor::accept()`, before any verification work is performed:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early-exit
    return result;
}
```

This mirrors the pattern already used in `contextual_check` for compact blocks: [5](#0-4) 

---

### Proof of Concept

1. Attacker mines a block header with valid PoW but `version = 1` (non-zero, rejected by `version_check`).
2. Attacker connects to a victim CKB node as a P2P peer.
3. Attacker sends a `SendHeaders` message containing this header. The node runs `prev_block_check` (pass), `non_contextual_check` including PoW (pass), `version_check` (fail) → marks `BLOCK_INVALID`.
4. Attacker immediately re-sends the same `SendHeaders` message.
5. `get_block_status` returns `BLOCK_INVALID`. `BLOCK_INVALID.contains(HEADER_VALID)` is `false`. The node re-runs the full pipeline including PoW verification.
6. Steps 4–5 repeat indefinitely, consuming victim CPU proportional to PoW verification cost per message.

The `FIXME` comment at `sync/src/synchronizer/headers_process.rs:301` is the authoritative confirmation that this guard is intentionally absent pending a resolution. [6](#0-5)
