Audit Report

## Title
RBF Extra-Fee Computed on Serialized Size Instead of Transaction Weight, Allowing Cycle-Heavy Replacements to Underpay — (`File: tx-pool/src/pool.rs`)

## Summary

`calculate_min_replace_fee` computes the mandatory RBF extra fee using only the raw serialized byte size of the incoming transaction, while the rest of the tx-pool uses transaction weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). For a cycle-heavy replacement transaction, the weight can be orders of magnitude larger than the serialized size, causing the required extra fee to be severely underestimated. An attacker can repeatedly replace mempool transactions at a fraction of the intended cost, enabling near-zero-cost mempool churn and network bandwidth exhaustion.

## Finding Description

`get_transaction_weight` in `util/types/src/core/tx_pool.rs` defines the canonical weight metric: [1](#0-0) 

`TxEntry::fee_rate()` correctly uses this weight: [2](#0-1) 

`FeeRate::fee()` multiplies the stored rate by whatever `weight` argument is passed: [3](#0-2) 

However, `calculate_min_replace_fee` passes only `size` (raw serialized bytes) to `fee()`, ignoring cycles entirely: [4](#0-3) 

The enforcement gate in `check_rbf` calls it with only `entry.size`: [5](#0-4) 

The secondary call site `min_replace_fee` has the same defect: [6](#0-5) 

`TxEntry` carries both `size` and `cycles` fields, so `entry.cycles` is available at both call sites but is never passed: [7](#0-6) 

For a transaction with 100-byte body and 70,000,000 cycles:
- `weight = max(100, 70_000_000 × 0.000_170_571_4) ≈ 11,940`
- `extra_rbf_fee (actual) = min_rbf_rate × 100 / 1000`
- `extra_rbf_fee (correct) = min_rbf_rate × 11,940 / 1000` (~119× larger)

The replacement is admitted after paying only ~0.8% of the intended extra fee. No existing guard compensates for this: all other RBF rules (Rules #1–#5) check structural properties, not fee-rate adequacy.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker can craft cycle-heavy replacement transactions that satisfy the RBF fee check while paying ~0.8% of the intended cost. Each accepted replacement propagates across the P2P network, consuming miner and relay bandwidth. Because the marginal cost per replacement is ~119× lower than the node operator configured, an attacker can sustain a high-frequency replacement loop at negligible expense, exhausting mempool processing capacity and network bandwidth across all nodes that have RBF enabled.

## Likelihood Explanation

Any unprivileged caller with access to `send_transaction` (RPC or P2P) can trigger this. Producing a transaction with near-maximum cycle consumption while keeping the serialized body small is straightforward for any script author. No special privilege, leaked key, or victim mistake is required. The CKB VM cycle limit is enforced after admission, so the declared cycle count is trusted for fee-rate purposes at the point the RBF check runs. The attack is repeatable indefinitely.

## Recommendation

Change `calculate_min_replace_fee` to accept `cycles: Cycle` and compute weight before calling `fee()`:

```rust
fn calculate_min_replace_fee(
    &self,
    conflicts: &[&PoolEntry],
    size: usize,
    cycles: Cycle,
) -> Option<Capacity> {
    let weight = get_transaction_weight(size, cycles);
    let extra_rbf_fee = self.config.min_rbf_rate.fee(weight);
    // ... rest unchanged
}
```

Update both call sites to pass `entry.cycles` / `tx.cycles` alongside `entry.size` / `tx.size`:
- `check_rbf` line 665: `self.calculate_min_replace_fee(&all_conflicted, entry.size, entry.cycles)`
- `min_replace_fee` line 98: `self.calculate_min_replace_fee(&conflicts, tx.size, tx.cycles)`

## Proof of Concept

1. Configure node: `min_rbf_rate = 1500` shannons/KW, `max_tx_verify_cycles = 70_000_000`.
2. Submit T1: 100-byte body, fee = 10,000 shannons → accepted.
3. Craft T2: same inputs, 100-byte body, 70,000,000 cycles, fee = 10,150 shannons.
   - Node computes: `extra_rbf_fee = 1500 × 100 / 1000 = 150` shannons → check passes.
   - Correct value: `1500 × 11,940 / 1000 = 17,910` shannons → T2 should be rejected.
4. T2 is accepted. Repeat from step 2 with T3 replacing T2, and so on, sustaining indefinite mempool churn at 150 shannons per cycle instead of the required 17,910 shannons.
