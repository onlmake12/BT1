### Title
Missing Request Expiry/Deadline in `executeCallback()` Allows Stale Price Delivery to Consumer Contracts — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol::executeCallback()` has no deadline or expiry check on when a pending price-update request must be fulfilled. A provider can delay execution indefinitely and then deliver a price pinned to the original `publishTime` — which may be arbitrarily stale by the time the callback fires — to a consumer contract that has no way to distinguish a timely fulfillment from a delayed one.

---

### Finding Description

When a user calls `requestPriceUpdatesWithCallback()`, the contract stores a `Request` with a `publishTime` (capped at `block.timestamp + 60`). The request remains active in storage with no expiry field and no cancellation path. [1](#0-0) 

When `executeCallback()` is later called, the only time-based check is the `exclusivityPeriodSeconds` window, which controls *which* provider may fulfill — not *when* the request must be fulfilled: [2](#0-1) 

The Pyth `parsePriceFeedUpdates` call inside `executeCallback()` pins the price to exactly `req.publishTime` (used as both `minPublishTime` and `maxPublishTime`): [3](#0-2) 

This means the consumer's `echoCallback` always receives a price from the original `publishTime`, regardless of how much wall-clock time has elapsed since the request was made. There is no on-chain mechanism to expire or cancel a request, and no deadline after which the requester can reclaim their fee. [4](#0-3) 

The `IEcho` interface documents no expiry semantics, and the `IEchoConsumer` callback receives only `(sequenceNumber, priceFeeds)` — no indication of how stale the fulfillment is relative to the current block: [5](#0-4) 

---

### Impact Explanation

A consumer protocol that uses Echo to obtain a price for a time-sensitive financial operation (e.g., settling a derivative, triggering a liquidation, executing a limit order) may receive a price from `publishTime` that is hours or days old by the time the callback fires. Because the consumer has no on-chain way to know when the callback will execute, it cannot reliably guard against this without adding its own staleness check on `priceFeeds[i].price.publishTime` — a check that is not enforced or documented as required by the Echo interface.

Additionally, because there is no cancellation path, a requester whose provider goes offline or acts maliciously has their fee permanently locked in the contract.

---

### Likelihood Explanation

After the `exclusivityPeriodSeconds` window expires, **any** registered provider can call `executeCallback()`. A malicious actor can:

1. Monitor pending Echo requests.
2. Wait for the exclusivity period to elapse.
3. Observe that the price has moved significantly from `publishTime`.
4. Call `executeCallback()` with historical update data from `publishTime`, delivering a stale price to the consumer at a moment favorable to the attacker.

This requires only a registered provider account (permissionless via `registerProvider()`) and the ability to call a public function — no privileged access is needed. [6](#0-5) 

---

### Recommendation

1. **Add a `deadline` or `expiryTime` field to `EchoState.Request`** — set at request time (e.g., `publishTime + maxFulfillmentWindow`). In `executeCallback()`, revert if `block.timestamp > req.expiryTime`.
2. **Add a `cancelRequest()` function** that allows the original requester to reclaim their fee after the deadline has passed.
3. **Document in `IEchoConsumer`** that `echoCallback` implementations must validate `priceFeeds[i].price.publishTime` against `block.timestamp` before using prices for financial decisions.

---

### Proof of Concept

```
T=0:   Consumer calls requestPriceUpdatesWithCallback(provider, publishTime=T, ...)
       → Request stored with publishTime=T, no expiry.

T=0 to T+exclusivityPeriod:
       Assigned provider is offline / withholds fulfillment.

T+exclusivityPeriod+1:
       Attacker (registered as a second provider) observes that the asset price
       has moved 20% since T. Calls executeCallback() with valid Pyth update data
       for timestamp T.

       parsePriceFeedUpdates(updateData, priceIds, T, T) succeeds — price from T
       is delivered to consumer's echoCallback.

       Consumer's protocol settles a trade at the T-price, which is now 20% stale,
       allowing the attacker to profit from the discrepancy.
```

The `testExecuteCallbackAfterExclusivity` test in the test suite confirms that any registered provider can call `executeCallback` after the exclusivity period with no time upper bound enforced: [7](#0-6)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
