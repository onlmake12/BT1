### Title
User Fees Permanently Locked Due to Missing Request Cancellation Mechanism — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo` contract accepts ETH fees from users in `requestPriceUpdatesWithCallback` but provides no mechanism to cancel a request and recover those fees. If `executeCallback` becomes permanently unfulfillable — for example, because Pyth price data for the exact `publishTime` is no longer available — the user's ETH fee is locked in the contract forever. The developers themselves acknowledge this risk in TODO comments within the code.

---

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, the contract immediately splits the `msg.value` into two parts:

1. The Pyth protocol fee, credited immediately to `_state.accruedFeesInWei`
2. The provider fee, stored in `req.fee = msg.value - pythFeeInWei` [1](#0-0) 

The provider fee (`req.fee`) is only credited to the provider when `executeCallback` is successfully called. Inside `executeCallback`, the contract calls `parsePriceFeedUpdates` with **exact timestamp matching** — both `minPublishTime` and `maxPublishTime` are set to `req.publishTime`: [2](#0-1) 

If `parsePriceFeedUpdates` reverts (e.g., because the Pyth data for that exact timestamp is no longer available), the entire `executeCallback` transaction reverts. The request remains active indefinitely. The fee stored in `req.fee` is never credited to any provider and is never returned to the user.

There is **no `cancelRequest` function** anywhere in the `IEcho` interface or `Echo.sol` implementation: [3](#0-2) 

The `withdrawFees` function only allows the admin to withdraw `_state.accruedFeesInWei` (the Pyth protocol portion), not the per-request `req.fee` amounts locked in individual requests: [4](#0-3) 

The developers explicitly acknowledge this risk in two TODO comments: [5](#0-4) 

The `firstUnfulfilledSeq` tracker compounds the issue: a stuck request at a low sequence number causes `getFirstActiveRequests` to always surface it first, causing the provider's keeper bot to repeatedly waste gas attempting to fulfill an unfulfillable request, while newer requests may be deprioritized. [6](#0-5) 

---

### Impact Explanation

Any ETH paid as `req.fee` for a request that can never be fulfilled is permanently locked in the `Echo` contract. There is no admin escape hatch, no user cancellation path, and no timeout-based refund. The `withdrawFees` admin function only covers `_state.accruedFeesInWei`, not per-request fees. At scale, this results in a growing pool of irrecoverable user ETH.

---

### Likelihood Explanation

The scenario is realistic and requires no privileged access:

- `requestPriceUpdatesWithCallback` only enforces `publishTime <= block.timestamp + 60`, placing no lower bound on `publishTime`. A user can request data for a timestamp already in the past.
- Pyth price data has a finite availability window. If the assigned provider is slow, offline, or experiences high load, the data for `req.publishTime` may expire before `executeCallback` is called.
- The exclusivity period (`exclusivityPeriodSeconds`) delays fulfillment by non-assigned providers, widening the window during which data can expire.
- No keeper bot is guaranteed to fulfill every request; the TODO comment at line 157 explicitly notes the absence of a penalty mechanism for non-fulfillment. [7](#0-6) 

---

### Recommendation

1. **Add a `cancelRequest(uint64 sequenceNumber)` function** that allows the original requester to cancel their request and recover `req.fee` after a configurable timeout (e.g., after the exclusivity period has elapsed plus a grace period).
2. **Add a lower bound on `publishTime`** to reject requests for timestamps whose Pyth data is already likely unavailable.
3. **Widen the timestamp range** in `parsePriceFeedUpdates` (e.g., `[publishTime - delta, publishTime + delta]`) to tolerate minor timestamp drift and reduce the chance of permanent lock.
4. **Add an admin recovery path** for `req.fee` amounts in requests that have been active beyond a maximum age.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback(provider, block.timestamp, priceIds, gasLimit)` with `msg.value = fee`. Contract stores `req.fee = msg.value - pythFeeInWei` and emits `PriceUpdateRequested`.
2. The assigned provider's keeper bot is offline for several minutes.
3. The Pyth price data for the exact `block.timestamp` captured at step 1 is no longer available from the Pyth network.
4. Any call to `executeCallback(..., sequenceNumber, updateData, priceIds)` now reverts inside `parsePriceFeedUpdates` because no update satisfies `minPublishTime == maxPublishTime == req.publishTime`.
5. The request remains permanently active. `req.fee` is locked in the contract. The user has no function to call to recover their ETH.
6. `getFirstActiveRequests` continues to return this stuck request, causing keeper bots to waste gas on it indefinitely. [8](#0-7) [9](#0-8)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);
