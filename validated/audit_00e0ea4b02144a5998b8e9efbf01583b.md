Audit Report

## Title
Min-Fee-Rate Gate Uses Raw `tx_size` Instead of Transaction Weight, Allowing Sub-Minimum-Fee-Rate Transactions Into the Pool — (File: `tx-pool/src/util.rs`)

## Summary
`check_tx_fee` enforces the minimum fee rate using only the serialized transaction size (`tx_size`), while the true transaction weight — `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)` — is used for sorting and eviction. After `verify_rtx` returns the actual cycle count, no second weight-based fee check is performed. A transaction with small serialized size but high script cycles can pass the size-only gate with a fee far below `min_fee_rate × weight`, entering the pool and being relayed network-wide.

## Finding Description
In `tx-pool/src/util.rs` lines 42–52, `check_tx_fee` computes the minimum fee using only `tx_size`, with a comment explicitly acknowledging the mismatch:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
```

The admission pipeline in `_process_tx` (`tx-pool/src/process.rs` lines 705–777):
1. `pre_check` calls `check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)` at line 289 — size-only gate, before script execution.
2. `verify_rtx` executes scripts and returns `verified.cycles` (lines 724–734).
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is constructed at line 751 and submitted — **no second fee check against weight**.

The true weight function in `util/types/src/core/tx_pool.rs` lines 298–303:
```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```
is only used in `TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs` lines 114–118) for sorting/eviction keys, never for admission gating.

For a transaction with `tx_size = 200` bytes and `cycles = 5_000_000`, the weight is `max(200, 5_000_000 × 0.000_170_571_4) ≈ 853`. The size-only gate requires fee ≥ `min_fee_rate × 200`, but the actual weight-based fee rate is `fee / 853` — roughly 4× below `min_fee_rate`.

## Impact Explanation
This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**. An attacker can repeatedly submit transactions that pass the size-based gate but carry a weight-based fee rate well below `min_fee_rate`. These transactions are admitted to the local pool and relayed to peers (which have the same code path), propagating sub-minimum-fee-rate transactions across the network. The eviction mechanism uses weight-based fee rate, so these entries are evicted first when the pool fills — but they are admitted and relayed before eviction occurs, and the attacker can continuously resubmit to maintain pool pressure. This degrades the anti-spam guarantee of `min_fee_rate` at low cost.

## Likelihood Explanation
Any unprivileged user with RPC access to `send_transaction` can trigger this. Crafting a high-cycles, small-size transaction requires only writing a CKB script that executes a tight loop — trivial for any script author. The `max_tx_verify_cycles` limit bounds cycles per transaction but does not close the mismatch; within that bound the gap between size and weight is always exploitable when cycles dominate. No special privileges, leaked keys, or victim mistakes are required.

## Recommendation
After `verify_rtx` returns the actual cycle count in `_process_tx`, perform a second fee-rate check using the true weight before constructing `TxEntry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(
        tx_pool_config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64()
    )), snapshot));
}
```

Alternatively, at the pre-execution stage in `check_tx_fee`, apply a conservative upper-bound estimate using `max_tx_verify_cycles` so the size-only check is never more lenient than the weight-based check.

## Proof of Concept
1. Write a CKB lock script that executes a tight loop consuming ~5,000,000 cycles. Keep the serialized transaction size small (≈200 bytes).
2. Set output capacity so `fee = min_fee_rate × tx_size` (e.g., 200 shannons at 1000 shannons/KW default).
3. Submit via `send_transaction` RPC.
4. Observe: `check_tx_fee` passes (`fee=200 >= min_fee=200`); `verify_rtx` succeeds; transaction enters the pool.
5. Query `get_raw_tx_pool` — the entry shows `cycles ≈ 5_000_000`, `size ≈ 200`. Compute actual fee rate: `200 / max(200, 5_000_000 × 0.000_170_571_4) ≈ 200 / 853 ≈ 234 shannons/KW` — well below the 1000 shannons/KW `min_fee_rate`.
6. Repeat to fill the pool with sub-minimum-fee-rate entries, crowding out legitimate transactions and degrading the anti-spam guarantee network-wide.
