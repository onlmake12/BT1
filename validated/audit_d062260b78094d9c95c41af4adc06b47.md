### Title
Unvalidated Provider URI Stored On-Chain Enables SSRF in Off-Chain Tooling — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract stores a provider-supplied `uri` field as raw bytes with zero format or content validation. Off-chain tooling reads this URI directly from on-chain state and uses it verbatim to construct and execute HTTP requests. Any registered Entropy provider — a role open to any unprivileged address — can set an arbitrary URI, causing off-chain systems that follow the documented SDK pattern to make HTTP requests to attacker-controlled endpoints (SSRF).

---

### Finding Description

**Root cause — on-chain, no validation:**

`Entropy.sol::register()` stores the caller-supplied `uri` parameter directly into `ProviderInfo.uri` with no format, scheme, length, or content check: [1](#0-0) 

`Entropy.sol::setProviderUri()` similarly stores the caller-supplied `newUri` with only a registration check (`sequenceNumber != 0`), no URI validation: [2](#0-1) 

The `ProviderInfo` struct documents the `uri` field as the endpoint "Client SDKs can use … to automatically determine how to retrieve random values for each provider": [3](#0-2) 

**Root cause — off-chain, unvalidated use:**

`contract_manager/scripts/entropy_debug_reveal.ts` reads `providerInfo.uri` from the contract and directly concatenates it into a URL, then issues an unauthenticated `fetch()` to that URL: [4](#0-3) 

`contract_manager/scripts/latency_entropy.ts` performs the identical pattern: [5](#0-4) 

No URL scheme validation, hostname allowlist, or sanitization is applied at any point in either script before the HTTP request is issued.

**Attacker-controlled entry path:**

1. Attacker calls `Entropy.register(fee, commitment, metadata, chainLength, maliciousUri)` — no privilege required, any address may register.
2. `maliciousUri` is stored verbatim in `_state.providers[msg.sender].uri`.
3. Any off-chain system that calls `getProviderInfo(attacker)` and follows the documented pattern of constructing a fetch URL from `providerInfo.uri` will issue an HTTP request to the attacker-controlled endpoint.

---

### Impact Explanation

**SSRF (Server-Side Request Forgery):** A malicious provider can set `uri` to an internal network address (e.g., `http://169.254.169.254/latest/meta-data/` on AWS, `http://internal-service/`, `file:///etc/passwd` on runtimes that support it). Any Pyth operator running `entropy_debug_reveal.ts` or `latency_entropy.ts` against a chain where the malicious provider is the default provider will have their machine issue HTTP requests to the attacker-specified endpoint. This can:

- Expose cloud instance metadata credentials (AWS/GCP/Azure IMDS).
- Probe and interact with internal network services not reachable from the public internet.
- Exfiltrate data from the operator's internal environment.

Additionally, any third-party integrator following the documented SDK pattern of reading `providerInfo.uri` to fetch revelations is equally exposed.

---

### Likelihood Explanation

- **No privilege required**: Any Ethereum address can call `register()` and become an Entropy provider.
- **Persistent**: The malicious URI remains on-chain until the provider updates it.
- **Documented attack surface**: The `uri` field is explicitly documented as the mechanism for clients to discover the revelation endpoint, making it a natural integration point that developers will use.
- **Existing scripts use it directly**: `entropy_debug_reveal.ts` and `latency_entropy.ts` are production operational scripts that already implement the vulnerable pattern.

---

### Recommendation

1. **On-chain (preferred):** Add URI format validation in `register()` and `setProviderUri()` — at minimum enforce that the URI begins with `https://` and does not exceed a reasonable length (e.g., 256 bytes).
2. **Off-chain (required regardless):** In `entropy_debug_reveal.ts`, `latency_entropy.ts`, and any SDK code that reads `providerInfo.uri`, validate the URI against an allowlist of expected schemes (`https`) and optionally a hostname allowlist before issuing any HTTP request.
3. **SDK documentation:** Warn integrators that `providerInfo.uri` is provider-controlled and must be validated before use.

---

### Proof of Concept

```solidity
// Attacker registers as an Entropy provider with a malicious URI
IEntropy entropy = IEntropy(ENTROPY_CONTRACT);
entropy.register{value: 0}(
    0,                          // feeInWei
    bytes32(0),                 // commitment (any valid value)
    bytes(""),                  // commitmentMetadata
    1,                          // chainLength >= 1
    bytes("http://169.254.169.254/latest/meta-data/iam/security-credentials/")
);
```

When a Pyth operator then runs:

```bash
npx ts-node contract_manager/scripts/entropy_debug_reveal.ts \
  --chain <chain-id> \
  --private-key <key> \
  --sequence-number 0
```

The script executes:

```typescript
// entropy_debug_reveal.ts line 85-86
const revealUrl = providerInfo.uri + `/revelations/${sequenceNumber}`;
// → "http://169.254.169.254/latest/meta-data/iam/security-credentials//revelations/0"
const fortunaResponse = await fetch(revealUrl);  // SSRF issued from operator's machine
```

The operator's machine issues an HTTP request to the AWS IMDS endpoint, returning IAM credentials to the attacker if the response is logged or the attacker controls a redirect chain.

    // Set provider uri. It will revert if provider is not registered.
    function setProviderUri(bytes calldata newUri) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        bytes memory oldUri = provider.uri;
        provider.uri = newUri;
        emit ProviderUriUpdated(msg.sender, oldUri, newUri);
        emit EntropyEventsV2.ProviderUriUpdated(
            msg.sender,
            oldUri,
            newUri,
            bytes("")
        );
    }
