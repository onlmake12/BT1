Audit Report

## Title
Missing ban enforcement for `StatusCode::TooManyRequests` allows indefinite relay message flooding — (`sync/src/relayer/mod.rs`, `sync/src/status.rs`)

## Summary
`StatusCode::TooManyRequests` is assigned value `110`, placing it in the 1xx informational range. `should_ban()` only acts on 4xx codes and `should_warn()` only on 5xx codes, so every rate-limit violation silently falls through to a debug log with no peer ban or disconnect. A connected peer can flood relay messages above the 30 req/sec cap indefinitely with zero enforcement consequence.

## Finding Description
`Relayer::new` configures a per-`(PeerIndex, message_type)` token-bucket rate limiter at 30 req/sec. [1](#0-0) 

When the bucket is exhausted, `try_process` returns `StatusCode::TooManyRequests.with_context(item_name)` immediately, before any expensive processing occurs. [2](#0-1) 

`process()` then evaluates `should_ban()` and `should_warn()` on that status. `should_ban()` gates on the 4xx range: [3](#0-2) 

`should_warn()` gates on ≥ 500: [4](#0-3) 

`TooManyRequests = 110` satisfies neither condition, so `process()` emits only a debug log and returns — `nc.ban_peer()` is never called. [5](#0-4) 

Critically, message deserialization (`from_compatible_slice`) occurs in `received()` **before** `try_process` is even invoked, meaning the node pays deserialization cost for every inbound packet regardless of how far above the cap the sender is. [6](#0-5) 

## Impact Explanation
A single connected peer can sustain an arbitrarily high send rate of relay messages (e.g., `RelayTransactions`, `GetRelayTransactions`). The node must receive, deserialize, and rate-check every packet with no mechanism to shed load by disconnecting the offending peer. At sufficient volume this can exhaust CPU and network I/O on the targeted node. This matches **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points). It also fits **High — bad designs which could cause CKB network congestion with few costs**, since a targeted full node that becomes unresponsive degrades relay propagation for the broader network.

## Likelihood Explanation
Any peer with a standard P2P connection can trigger this. No special privilege, key, or hashpower is required. The attacker needs only to open a connection and send relay protocol messages at high rate, which is trivially achievable with a modified CKB client or raw socket. The attack is repeatable and persistent.

## Recommendation
The simplest fix is to reclassify `TooManyRequests` as a 4xx code (e.g., `TooManyRequests = 419`) so it is automatically caught by the existing `should_ban()` range check. Alternatively, add an explicit match arm in `should_ban()`:

```rust
pub fn should_ban(&self) -> Option<Duration> {
    match self.code {
        StatusCode::TooManyRequests => Some(SYNC_USELESS_BAN_TIME),
        _ if (400..500).contains(&(self.code as u16)) => { /* existing logic */ }
        _ => return None,
    }
}
```

A complementary approach is a per-peer repeated-violation counter that calls `nc.ban_peer()` after N consecutive rate-limit hits within a sliding window, without requiring a code reclassification.

## Proof of Concept
1. Connect a peer to a CKB node (standard P2P handshake).
2. Send `GetRelayTransactions` messages at 100/sec for 60 seconds from that peer.
3. Observe: the node emits only `debug`-level log lines (`receive GetRelayTransactions from <peer>, TooManyRequests(110): ...`); the peer is never banned or disconnected.
4. Confirm: `nc.ban_peer` is never called for this peer during the entire window (instrument the call site at `sync/src/relayer/mod.rs:204`).
5. Scale to multiple message types simultaneously (each has its own token bucket) to maximize throughput while remaining below per-type limits.
6. Expected fixed behavior: after the first (or Nth) rate-limit violation, the peer is banned for at least `SYNC_USELESS_BAN_TIME`.
