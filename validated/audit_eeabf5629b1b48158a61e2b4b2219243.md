### Title
No Maximum Fee Parameter in `requestV2()` and Excess `msg.value` Not Refunded — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The `requestV2()` family of functions in Pyth's `Entropy.sol` contract does not accept a `maxFee` parameter. Provider fees can change at any time via `setProviderFee()`, and the Fortuna keeper service (`sync_fee`) updates fees automatically. Critically, any `msg.value` sent above the required fee is **permanently captured** by the Pyth protocol fee pool and never refunded to the caller. This is the direct analog to the Renzo M-07 finding: users have no slippage-equivalent protection against fee changes between the time they query `getFeeV2()` and the time their transaction executes.

### Finding Description

The `requestHelper()` internal function in `Entropy.sol` checks that `msg.value >= requiredFee`, but then credits **all** excess value to `_state.accruedPythFeesInWei` rather than refunding it:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The `IEntropyV2` interface explicitly documents this behavior but provides no mitigation:

> "Further note that excess value is *not* refunded to the caller." [2](#0-1) 

Provider fees are mutable at any time via `setProviderFee()`: [3](#0-2) 

The Fortuna off-chain service (`sync_fee`) automatically updates provider fees based on current gas prices, making fee changes a routine, non-malicious occurrence: [4](#0-3) 

The total fee is the sum of the provider fee and the Pyth protocol fee: [5](#0-4) 

### Impact Explanation

**Financial loss for callers who send excess `msg.value`:**

A caller (user or integrating contract) who sends `msg.value` slightly above the quoted `getFeeV2()` amount — a common defensive pattern to avoid reverts — permanently loses the excess to the Pyth protocol fee pool. There is no mechanism to cap the fee paid at a user-specified maximum.

**Concrete scenario:**
1. User calls `getFeeV2()` off-chain, gets `fee = 100 wei`.
2. User submits `requestV2{value: 150}()` (with a 50 wei buffer to avoid revert).
3. Before execution, Fortuna's `sync_fee` increases the provider fee to 150 wei.
4. Transaction succeeds, but the user pays 150 wei instead of 100 wei. The 50 wei excess is permanently captured by the Pyth fee pool.
5. The user has no recourse and no way to have prevented this with the current API.

**Revert DoS scenario:**
1. User sends exactly `msg.value = getFeeV2()`.
2. Fee increases before execution.
3. Transaction reverts with `InsufficientFee`.
4. User must retry, potentially in a loop if fees are volatile.

### Likelihood Explanation

**Moderate.** The Fortuna keeper service (`sync_fee`) is designed to update provider fees automatically and periodically to track gas price changes. Fee changes between a user's `getFeeV2()` call and their `requestV2()` execution are a normal, expected occurrence — not a hypothetical edge case. Any integrating contract that queries the fee in a prior transaction or off-chain and then submits the request in a separate transaction is exposed. The no-refund behavior is documented but there is no API-level protection.

### Recommendation

1. Add a `maxFee` parameter to `requestV2()` variants, reverting if `msg.value > maxFee` or if `requiredFee > maxFee`.
2. Alternatively, refund excess `msg.value` to `msg.sender` after deducting the required fee, rather than crediting it to `accruedPythFeesInWei`.
3. At minimum, document a recommended pattern for integrating contracts (e.g., always re-query `getFeeV2()` atomically in the same transaction immediately before calling `requestV2()`).

### Proof of Concept

```solidity
// Attacker is the Fortuna provider (or any registered provider/fee manager).
// Step 1: User queries fee off-chain.
uint128 fee = entropy.getFeeV2(); // returns 100 wei

// Step 2: User submits tx with buffer to avoid revert.
// msg.value = 150 wei

// Step 3: Before user's tx lands, provider calls setProviderFee(150 wei).

// Step 4: User's tx executes.
// requestHelper: requiredFee = 150 wei, msg.value = 150 wei → passes check
// providerInfo.accruedFeesInWei += 150 wei (provider fee)
// _state.accruedPythFeesInWei += (150 - 150) = 0 (pyth fee)
// User paid 150 wei instead of 100 wei. 50 wei lost permanently.

// If user had sent exactly 100 wei:
// requestHelper: requiredFee = 150 wei, msg.value = 100 wei → revert InsufficientFee
// User's tx reverts. No refund needed, but user is DoS'd.

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
