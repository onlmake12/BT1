### Title
Unbounded Blocking Wait in `blocking_process_block_internal` Allows RPC Thread Exhaustion via Concurrent `submit_block` Calls ‚Äî (`chain/src/chain_controller.rs`)

---

### Summary

The `submit_block` RPC handler calls `blocking_process_block_internal`, which blocks the calling thread indefinitely ‚Äî with no timeout ‚Äî waiting for the chain service to finish verifying a block. Because the chain service processes blocks sequentially, any number of concurrent `submit_block` RPC calls each hold a thread forever until the queue drains. An unprivileged RPC caller (including any miner or node operator with RPC access) can flood the node with concurrent `submit_block` requests, exhausting the thread pool and rendering the node unresponsive.

---

### Finding Description

`blocking_process_block_internal` in `chain/src/chain_controller.rs` implements a synchronous request-response pattern: it enqueues a block for processing and then blocks on a one-shot channel receiver with no timeout:

```rust
self.asynchronous_process_lonely_block(lonely_block);
verify_result_rx.recv().unwrap_or_else(|err| { ... })
``` [1](#0-0) 

`verify_result_rx.recv()` is a blocking, timeout-free call. It returns only when the chain service sends back a result via `verify_result_tx`. The chain service processes blocks one at a time. If block verification is slow (e.g., a block with many transactions requiring script execution), every concurrent caller of `blocking_process_block_internal` is blocked for the full duration.

This function is called directly from the `submit_block` RPC handler:

```rust
let is_new = self
    .chain
    .blocking_process_block(Arc::clone(&block))
    .map_err(|err| handle_submit_error(&work_id, &err))?;
``` [2](#0-1) 

The `submit_block` RPC is publicly declared and reachable by any RPC caller: [3](#0-2) 

The same pattern exists in the tx-pool service via the `send_message!` macro, which calls `block_in_place(|| response.recv())` ‚Äî also a timeout-free blocking wait ‚Äî used by `send_transaction`, `test_tx_pool_accept`, `get_block_template`, and others: [4](#0-3) 

However, the tx-pool path has a bounded channel (`DEFAULT_CHANNEL_SIZE = 32`) with `try_send`, which provides backpressure and fails fast when full. The chain path has no such guard: `asynchronous_process_lonely_block` uses `Request::call_without_response` which enqueues without limit, and the caller then blocks indefinitely. [5](#0-4) 

---

### Impact Explanation

Each concurrent `submit_block` RPC call holds one OS thread indefinitely (no timeout). The CKB RPC server uses a thread pool. An attacker who sends N concurrent `submit_block` requests with syntactically valid but computationally expensive blocks (e.g., blocks containing transactions with complex lock scripts) will hold N threads blocked in `verify_result_rx.recv()`. Once the thread pool is saturated, the node cannot serve any further RPC requests ‚Äî including `send_transaction`, `get_block_template`, `sync_state`, and all other endpoints ‚Äî causing a full RPC denial of service. P2P message handling that shares the same async runtime may also degrade.

---

### Likelihood Explanation

The `submit_block` RPC endpoint is reachable by any configured RPC caller. In many deployments, the RPC port is exposed to miners or internal tooling. The attacker only needs to send concurrent HTTP POST requests to the RPC endpoint ‚Äî no special privileges, no cryptographic material, and no majority hashpower. Constructing a block that passes `HeaderVerifier` but is slow to fully verify (e.g., by including transactions with expensive scripts) is feasible. Even without slow blocks, sending many concurrent `submit_block` calls for the same or different blocks will queue them all in the chain service and hold all calling threads until each is processed.

---

### Recommendation

1. Add a timeout to `verify_result_rx.recv()` in `blocking_process_block_internal`, analogous to the 5-minute timeout in the reported `ProposeAndWait`. Return an error if verification does not complete within the deadline.
2. Prefer the existing asynchronous path (`asynchronous_process_remote_block`) for `submit_block` and return the block hash immediately, allowing callers to poll for confirmation ‚Äî consistent with how remote blocks from P2P are handled.
3. Apply rate-limiting or concurrency limits on the `submit_block` RPC endpoint to bound the number of simultaneous in-flight block submissions.

---

### Proof of Concept

```
# Send 200 concurrent submit_block RPC calls with a valid-header block
# Each call blocks a thread in verify_result_rx.recv() with no timeout
for i in $(seq 1 200); do
  curl -s -X POST http://127.0.0.1:8114/ \
    -H 'Content-Type: application/json' \
    -d '{"id":1,"jsonrpc":"2.0","method":"submit_block","params":["0", <VALID_BLOCK_JSON>]}' &
done

# Node RPC is now unresponsive ‚Äî all threads are blocked
curl -s -X POST http://127.0.0.1:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"get_tip_block_number","params":[]}'
# ‚Üí hangs or times out
```

    fn blocking_process_block_internal(
        &self,
        block: Arc<BlockView>,
        switch: Option<Switch>,
    ) -> VerifyResult {
        let (verify_result_tx, verify_result_rx) = ckb_channel::oneshot::channel::<VerifyResult>();

        let verify_callback = {
            move |result: VerifyResult| {
                if let Err(err) = verify_result_tx.send(result) {
                    error!(
                        "blocking send verify_result failed: {}, this shouldn't happen",
                        err
                    )
                }
            }
        };

        let lonely_block = LonelyBlock {
            block,
            switch,
            verify_callback: Some(Box::new(verify_callback)),
        };

        self.asynchronous_process_lonely_block(lonely_block);
        verify_result_rx.recv().unwrap_or_else(|err| {
            Err(InternalErrorKind::System
                .other(format!("blocking recv verify_result failed: {}", err))
                .into())
        })
    }
