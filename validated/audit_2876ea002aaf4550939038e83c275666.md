### Title
`dao_type_hash` Can Silently Be All-Zeros in `ConsensusBuilder::build()`, Spoofing `CapacityVerifier::valid_dao_withdraw_transaction()` and Bypassing the `OutputsSumOverflow` Check — (`spec/src/consensus.rs`, `verification/src/transaction_verifier.rs`)

---

### Summary

`ConsensusBuilder::build()` derives `dao_type_hash` from the genesis block's DAO output via `unwrap_or_default()`, silently leaving it as `Byte32::default()` (32 zero bytes) when the genesis DAO cell carries no type script. No non-zero assertion guards this field, unlike every other critical consensus parameter. `CapacityVerifier::valid_dao_withdraw_transaction()` then uses this zero hash to identify DAO cells; any input cell whose type script has `code_hash = 0x000…000` and `hash_type = Type` satisfies the check, causing the `OutputsSumOverflow` guard to be skipped entirely.

---

**Root cause — `ConsensusBuilder::build()` overwrites it with `unwrap_or_default()`, silently keeping all-zeros when the genesis DAO output has no type script:** [2](#0-1) 

`get_type_hash` returns `None` when `output.type_().to_opt()` is absent: [3](#0-2) 

**Missing guard — every other critical consensus parameter has a `debug_assert!` for non-zero, but `dao_type_hash` has none:** [4](#0-3) 

**Downstream sink — `CapacityVerifier::verify()` skips `OutputsSumOverflow` when `valid_dao_withdraw_transaction()` is true:** [5](#0-4) 

**`valid_dao_withdraw_transaction()` delegates to `cell_uses_dao_type_script` with the potentially-zero hash:** [6](#0-5) 

**`cell_uses_dao_type_script` matches any cell whose type script has `code_hash == dao_type_hash` and `hash_type == Type`; when `dao_type_hash` is all-zeros, this matches any cell with a zero-code_hash type script:** [7](#0-6) 

**Trigger condition — a chain spec where the DAO genesis cell is built without `create_type_id = true` causes `get_type_hash(OUTPUT_INDEX_DAO)` to return `None`:** [3](#0-2) 

The `check_block` genesis validator also explicitly skips lock scripts with zero `code_hash`, confirming zero-hash cells are a recognized and reachable state in the codebase: [8](#0-7) 

**Exploit flow:**

1. Node operator initializes a chain spec where the DAO system cell is configured without a type_id (e.g., `create_type_id = false`). `ConsensusBuilder::build()` silently sets `dao_type_hash = Byte32::default()`.
2. Attacker (RPC/tx-pool submitter) creates a live cell with `type_script { code_hash: 0x000…000, hash_type: Type }`.
3. Attacker submits a transaction spending that cell with outputs exceeding inputs.
4. `CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` (zero matches zero), the `OutputsSumOverflow` check is skipped, and the transaction passes capacity verification.
5. In the `skip_script_verify = true` code path (present in `verification/src/transaction_verifier.rs` and `verification/contextual/src/contextual_block_verifier.rs`), script execution is also skipped, making the bypass complete.

---

### Impact Explanation

When `dao_type_hash` is all-zeros, the `OutputsSumOverflow` invariant — the fundamental CKB rule that transaction outputs cannot exceed inputs — is bypassed for any transaction whose input carries a zero-code_hash type script. In contexts where `skip_script_verify = true` (block re-processing, IBD fast-sync, or re-org paths), this allows a transaction sender to create CKB capacity out of thin air, permanently corrupting the ledger's capacity accounting. Even without `skip_script_verify`, the `CapacityVerifier` is the first and only capacity gate; its silent bypass is a critical invariant violation.

---

### Likelihood Explanation

The trigger requires a chain spec where the DAO genesis cell lacks a type_id. The default bundled specs (`dev.toml`, `testnet.toml`) do include `create_type_id = true` for the DAO cell, so mainnet and standard testnet nodes are not affected by default. However, custom dev chains — an explicitly supported and documented use case — can omit this, and the code provides no warning, no `debug_assert!`, and no error when `dao_type_hash` resolves to zero. The inconsistency with other guarded parameters (`initial_primary_epoch_reward`, `epoch_duration_target`, genesis difficulty) makes this a latent defect that is easy to trigger silently.

---

### Recommendation

1. Add a `debug_assert!` (or a hard `panic!` / `Result::Err`) in `ConsensusBuilder::build()` asserting `dao_type_hash != Byte32::default()`, consistent with the existing guards for other critical parameters.
2. Change `unwrap_or_default()` to an explicit `expect("Genesis DAO cell must have a type_id script")` or return an error from `build()`.
3. Document in the chain spec format that `create_type_id = true` is mandatory for the DAO system cell.
