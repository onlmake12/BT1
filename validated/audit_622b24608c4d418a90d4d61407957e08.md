Audit Report

## Title
Missing Duplicate Index Check in `GetBlockTransactionsProcess::execute()` Enables Bandwidth Amplification — (File: `sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute()` validates only the *count* of requested transaction indexes against `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) but never checks for duplicates within the list. Any peer that has completed the P2P handshake can craft a `GetBlockTransactions` message with the same index repeated up to 32,767 times, causing the node to serialize and transmit the same transaction data 32,767 times in a single `BlockTransactions` reply, constituting a targeted outbound bandwidth amplification attack.

## Finding Description
In `sync/src/relayer/get_block_transactions_process.rs`, `execute()` performs two length-only guards: [1](#0-0) 

After these pass, the handler iterates the raw `indexes` list with no deduplication: [2](#0-1) 

Every element — including duplicates — produces one entry in the response `Vec`, which is then serialized and sent back. The sibling handler `GetTransactionsProcess` for `GetRelayTransactions` already guards against this by building a `HashSet` and comparing its length to the original list length, returning `StatusCode::RequestDuplicate` on mismatch: [3](#0-2) 

That protection is entirely absent in `GetBlockTransactionsProcess`. The rate limiter in `mod.rs` is keyed by `(peer, message_type)` at 30 req/s, which still permits 30 × 32,767 = ~983,010 duplicate transaction serializations per second per peer connection. [4](#0-3) 

## Impact Explanation
An attacker sends a single small `GetBlockTransactions` message with `indexes = [0; 32767]` (all pointing to the always-present coinbase transaction). The node responds with a `BlockTransactions` message containing 32,767 serialized copies of the coinbase transaction. The amplification ratio is `MAX_RELAY_TXS_NUM_PER_BATCH × sizeof(targeted_tx)`. Repeated across multiple simultaneous connections this constitutes a targeted bandwidth-exhaustion vector against a specific node. This matches the allowed **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The entry path requires only a standard P2P connection and completion of the identify/sync handshake — no special privilege, key, or hash-power is needed. The crafted message is syntactically valid and passes all existing guards (list length ≤ `MAX_RELAY_TXS_NUM_PER_BATCH`). The attack is trivially reproducible with a single crafted packet and is repeatable from multiple simultaneous connections.

## Recommendation
Add a duplicate-index check immediately after the existing length guards, mirroring the pattern already used in `GetTransactionsProcess`:

```rust
use std::collections::HashSet;

let indexes_set: HashSet<_> = get_block_transactions.indexes().iter().collect();
if indexes_set.len() != get_block_transactions.indexes().len() {
    return StatusCode::ProtocolMessageIsMalformed
        .with_context("Duplicate index in GetBlockTransactions");
}
let uncle_indexes_set: HashSet<_> = get_block_transactions.uncle_indexes().iter().collect();
if uncle_indexes_set.len() != get_block_transactions.uncle_indexes().len() {
    return StatusCode::ProtocolMessageIsMalformed
        .with_context("Duplicate uncle index in GetBlockTransactions");
}
```

## Proof of Concept
1. Connect to a CKB node as a peer and complete the identify/sync handshake.
2. Obtain any block hash `H` the node has stored.
3. Craft a `GetBlockTransactions` message: `block_hash = H`, `indexes = [0; 32767]` (all pointing to the always-present coinbase transaction).
4. Send the message over the relay protocol.
5. Observe the node replies with a `BlockTransactions` message containing 32,767 serialized copies of the coinbase transaction. No deduplication occurs before the response is built or transmitted. [5](#0-4) [6](#0-5)
