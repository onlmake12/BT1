### Title
Echo Protocol Fee Never Credited to `_state.accruedFeesInWei` — `withdrawFees()` Always Reverts — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` deducts `pythFee` from the amount credited to the provider but never increments `_state.accruedFeesInWei`. Because `withdrawFees()` checks and decrements only `_state.accruedFeesInWei`, the admin can never withdraw any Pyth protocol fees: the call always reverts with `"Insufficient balance"`. Every wei of Pyth protocol fee paid by users is permanently locked in the contract.

---

### Finding Description

`Echo.executeCallback()` computes the provider's credit as:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [1](#0-0) 

The `pythFee` portion is subtracted from the provider's credit — implying it belongs to the Pyth protocol — but it is never added to `_state.accruedFeesInWei`. No other code path in the contract increments `_state.accruedFeesInWei`.

`withdrawFees()` enforces:

```solidity
require(_state.accruedFeesInWei >= amount, "Insufficient balance");
_state.accruedFeesInWei -= amount;
``` [2](#0-1) 

Because `_state.accruedFeesInWei` is never incremented, it remains `0` regardless of how many fees users have paid. Every call to `withdrawFees()` reverts.

`getAccruedPythFees()` exposes this same field:

```solidity
accruedFeesInWei = _state.accruedFeesInWei;
``` [3](#0-2) 

The state variable is declared in `EchoState` (via `_state`) and is the sole source of truth for the admin withdrawal path. [4](#0-3) 

Compare with the correct pattern in `Entropy.sol`, where `_state.accruedPythFeesInWei` is explicitly incremented during `requestHelper`:

```solidity
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [5](#0-4) 

Echo lacks this credit step entirely.

---

### Impact Explanation

All Pyth protocol fees collected by the Echo contract are permanently locked. The admin's `withdrawFees()` call always reverts with `"Insufficient balance"` because `_state.accruedFeesInWei` is always `0`. The ETH is held by the contract but is irrecoverable through the intended withdrawal path. This is a complete and permanent loss of protocol revenue for every Echo deployment.

---

### Likelihood Explanation

The bug is triggered by the normal, intended usage of the protocol. Any user calling `requestPriceUpdatesWithCallback` followed by a provider calling `executeCallback` exercises the broken path. No special privileges, no adversarial input, and no external dependency failure is required. The admin discovering the issue when attempting to withdraw fees is the only trigger needed to confirm the loss.

---

### Recommendation

In `executeCallback`, after computing `pythFee`, add the missing credit step before or alongside the provider credit:

```solidity
_state.accruedFeesInWei += SafeCast.toUint128(pythFee);
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

This mirrors the correct pattern used in `Entropy.sol`'s `requestHelper`. [6](#0-5) 

---

### Proof of Concept

1. Deploy `Echo` with `pythFeeInWei = 1000 wei`.
2. Register a provider.
3. User calls `requestPriceUpdatesWithCallback{value: getFee(...)}(...)`.
4. Provider calls `executeCallback(...)` — provider's `accruedFeesInWei` increases; `_state.accruedFeesInWei` stays `0`.
5. Admin calls `withdrawFees(1)` → reverts: `"Insufficient balance"`.
6. Call `getAccruedPythFees()` → returns `0`, confirming the fee was never credited despite ETH being held by the contract. [1](#0-0) [2](#0-1)
