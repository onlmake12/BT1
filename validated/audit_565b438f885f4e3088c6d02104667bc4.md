### Title
Stale Price Returned Without Recency Check in `latestRoundData()` / `latestAnswer()` — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatibility adapter promoted for protocol migration, implements every price-reading function — including the non-deprecated `latestRoundData()` — on top of `getPriceUnsafe()`, which performs no staleness check. Additionally, `latestRoundData()` sets `answeredInRound = roundId = uint80(price.publishTime)`, which always satisfies the standard Chainlink staleness guard (`require(answeredInRound >= roundId)`), silently bypassing it in any migrated protocol.

### Finding Description

`PythAggregatorV3` is the official Pyth SDK contract for migrating Chainlink-dependent protocols to Pyth price feeds. Every price-reading function in the contract calls `getPriceUnsafe()`:

- `latestAnswer()` — line 54 [1](#0-0) 
- `latestTimestamp()` — line 59 [2](#0-1) 
- `latestRound()` — delegates to `latestTimestamp()` [3](#0-2) 
- `getRoundData()` — line 89 [4](#0-3) 
- `latestRoundData()` — line 110 [5](#0-4) 

`getPriceUnsafe()` is explicitly documented as returning a price from arbitrarily far in the past with no recency guarantee: [6](#0-5) 

The compounding issue is in `latestRoundData()`:

```solidity
roundId = uint80(price.publishTime);   // line 111
return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
```

`answeredInRound` is set equal to `roundId` unconditionally. [5](#0-4) 

The canonical Chainlink staleness guard used by virtually every migrating protocol is:

```solidity
(uint80 roundId, int256 answer, , uint256 updatedAt, uint80 answeredInRound) = feed.latestRoundData();
require(answeredInRound >= roundId, "Stale price");
require(block.timestamp - updatedAt < MAX_DELAY, "Stale price");
```

Because `answeredInRound == roundId` always, the first check always passes regardless of how old the price is. Protocols that rely solely on the first check (a common pattern) receive no staleness protection at all.

### Impact Explanation

Any DeFi protocol that:
1. Migrates from Chainlink to Pyth using `PythAggregatorV3` (as documented and promoted in the official migration guide), [7](#0-6) 
2. Uses the standard Chainlink staleness check `require(answeredInRound >= roundId)`,

will silently accept arbitrarily stale prices. An attacker can exploit this by:
- Allowing the on-chain price to go stale (no update submitted)
- Interacting with the protocol at a favorable stale price (e.g., borrowing against an inflated collateral value, or liquidating at a deflated price)

This is a price validation bypass leading to direct fund loss in any lending, derivatives, or collateral-management protocol using this adapter.

### Likelihood Explanation

The `PythAggregatorV3` contract is the officially recommended migration path from Chainlink. The migration guide explicitly instructs protocols to deploy it as a drop-in replacement. Protocols that follow the guide and retain their existing Chainlink staleness check pattern are silently unprotected. The likelihood is high because the pattern is standard and the failure is invisible — no revert occurs, no warning is emitted.

### Recommendation

1. Replace `getPriceUnsafe()` in `latestRoundData()` and `latestAnswer()` with `getPriceNoOlderThan(priceId, maxAge)` where `maxAge` is a configurable constructor parameter.
2. If `getPriceUnsafe()` must be retained for backward compatibility, add a `require(block.timestamp - price.publishTime <= maxAge, "Stale price")` guard inside `latestRoundData()`.
3. Document clearly that `answeredInRound >= roundId` is not a meaningful staleness check for this adapter, and that callers must check `updatedAt` against `block.timestamp`.
4. Consider separating the deprecated legacy functions (`latestAnswer`, `latestTimestamp`, `latestRound`) from the AggregatorV3 functions to avoid confusion about which are safe to use.

### Proof of Concept

```solidity
// Scenario: Protocol uses PythAggregatorV3 with standard Chainlink staleness check

PythAggregatorV3 adapter = new PythAggregatorV3(pythAddress, priceId);

// Price was last updated 24 hours ago (stale), no one has called updateFeeds()

(uint80 roundId, int256 answer, , uint256 updatedAt, uint80 answeredInRound) =
    adapter.latestRoundData();

// answeredInRound == roundId ALWAYS (both = uint80(price.publishTime))
// So this check PASSES even though price is 24 hours old:
require(answeredInRound >= roundId, "Stale price"); // passes silently

// Protocol uses 24-hour-old price for collateral valuation / liquidation
// Attacker profits from the price discrepancy
```
