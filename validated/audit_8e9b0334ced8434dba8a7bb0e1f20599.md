Audit Report

## Title
Expired Alert Replay via P2P â€” Missing `notice_until` Validation Enables Signature Replay and Network-Wide Flooding â€” (`util/network-alert/src/alert_relayer.rs`)

## Summary
`AlertRelayer::received()` performs no expiry check on incoming P2P alerts. Once `clear_expired_alerts()` removes an expired alert from `received_alerts` on peer connection, `has_received()` returns `false` for that alert ID, allowing any P2P peer to replay the original signed bytes verbatim. The signatures remain cryptographically valid, the alert is re-accepted and re-broadcast to all connected peers, and the cycle can be repeated indefinitely at negligible cost.

## Finding Description
The RPC handler `send_alert` in `rpc/src/module/alert.rs` (L104â€“110) explicitly rejects alerts whose `notice_until` is in the past. The P2P handler `AlertRelayer::received()` in `util/network-alert/src/alert_relayer.rs` (L98â€“179) performs no such check â€” its only guards are UTF-8 validation, a deduplication check via `has_received()`, and a signature check.

`has_received()` in `notifier.rs` (L147â€“149) returns `true` only if the alert ID is in `received_alerts` (a `HashMap`) or `cancel_filter` (an LRU cache of *cancelled* alerts). `clear_expired_alerts()` (L135â€“144) removes expired entries from `received_alerts` and `noticed_alerts` using `retain`, but **never inserts them into `cancel_filter`**. This means after expiry, `has_received(id)` returns `false` for the expired alert ID.

`clear_expired_alerts()` is called every time a new peer connects (L87 in `connected()`), giving the attacker a trivially forceable trigger.

**Exploit chain:**
1. A legitimate alert (ID=X, `notice_until`=T) is broadcast and accepted by all nodes.
2. Time passes; T is now in the past.
3. Attacker connects a new peer â†’ `connected()` â†’ `clear_expired_alerts()` â†’ alert ID=X removed from `received_alerts`. `has_received(X)` â†’ `false`.
4. Attacker sends the original signed alert bytes for ID=X over the Alert protocol.
5. `has_received(X)` â†’ `false` (not in `received_alerts`, not in `cancel_filter`) â€” dedup guard bypassed.
6. `verify_signatures()` â†’ `Ok(())` â€” ECDSA signatures over the alert hash are still valid; the attacker replays the exact same bytes.
7. `notifier.add(&alert)` is called: alert re-enters `received_alerts` and `noticed_alerts`; `notify_network_alert` fires.
8. Alert is re-broadcast to all connected peers, propagating network-wide.
9. Attacker disconnects and reconnects to force another `clear_expired_alerts()`, then replays again â€” indefinitely.

## Impact Explanation
**High â€” Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker with only a P2P connection (no keys, no privilege) can repeatedly flood the entire CKB P2P network with replayed alert messages. Each replay propagates to every reachable node, which re-broadcasts to all its peers. The attacker sustains this by cycling connections to force `clear_expired_alerts()` on victim nodes. Additionally, `noticed_alerts` (returned by `get_blockchain_info`) is polluted with expired alerts, causing false alarms for operators and monitoring systems.

## Likelihood Explanation
Preconditions are trivially met: the attacker needs only a P2P connection and the raw bytes of any previously-broadcast alert (observable on the public P2P network; hardcoded in the test suite). The trigger (`clear_expired_alerts()` on peer connection) is a routine network event the attacker can force by connecting and disconnecting. CKB has issued real network alerts with permanently valid signatures. The attack is repeatable indefinitely with no rate limiting.

## Recommendation
Add an expiry check in `AlertRelayer::received()` immediately after parsing the alert, mirroring the check already present in the RPC handler:

```rust
// In alert_relayer.rs, after line 143 (after parsing the alert):
let now_ms = ckb_systemtime::unix_time_as_millis();
let notice_until: u64 = alert.as_reader().raw().notice_until().into();
if notice_until < now_ms {
    nc.ban_peer(peer_index, BAD_MESSAGE_BAN_TIME,
        String::from("send us an expired alert"));
    return;
}
```

Additionally, in `clear_expired_alerts()`, insert expired alert IDs into `cancel_filter` before removing them from `received_alerts`, so `has_received()` continues to return `true` for them even after expiry:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until <= now {
            self.cancel_filter.put(*id, ());  // persist replay barrier
            false
        } else {
            true
        }
    });
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

## Proof of Concept
1. Capture raw Molecule-encoded bytes of a previously-broadcast alert (e.g., from P2P traffic or from the hardcoded test signatures in `util/network-alert/src/tests/`).
2. Wait for the alert's `notice_until` to pass.
3. Connect a new peer to the victim node. This triggers `AlertRelayer::connected()` â†’ `clear_expired_alerts()` â†’ the expired alert ID is removed from `received_alerts`. Confirm `has_received(id)` returns `false`.
4. Send the captured alert bytes to the victim node over the Alert protocol.
5. Observe: `AlertRelayer::received()` accepts the alert (no expiry check), `verify_signatures()` passes, `notifier.add()` fires `notify_network_alert`, and the alert is re-broadcast to all connected peers.
6. Confirm via `get_blockchain_info` RPC that the expired alert reappears in `alerts`.
7. Disconnect and reconnect to repeat the cycle â€” demonstrating indefinite repeatability.
