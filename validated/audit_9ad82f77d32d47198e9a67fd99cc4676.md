### Title
`RecentReject::put` never increments `self.total_keys_num`, causing unbounded disk growth — (`tx-pool/src/component/recent_reject.rs`)

### Summary
`RecentReject::put` computes a local `total_keys_num = self.total_keys_num + 1` but never writes it back to `self.total_keys_num` in the non-shrink branch. As a result, `self.total_keys_num` is permanently stuck at its initial startup estimate, `shrink()` is never triggered, and the RocksDB-backed `DBWithTTL` grows unboundedly as rejected transactions accumulate.

### Finding Description
In `put()` ( [4](#0-3) ), the pattern is:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // BUG: missing `self.total_keys_num = total_keys_num;`
}
```

`self.total_keys_num` is set once in `build()` from `estimate_num_keys_cf` and is only ever updated inside `shrink()`. Because the write-back is absent, every subsequent call to `put()` re-evaluates the same stale value. On a freshly started node (estimate ≈ 0), `0 + 1 > count_limit` is false for any `count_limit ≥ 1`, so `shrink()` is never reached. The `self.db.put()` call on line 60 still executes unconditionally, writing every rejected tx to disk.

### Impact Explanation
An unprivileged peer or RPC caller can submit an unlimited number of transactions that are rejected for any reason (invalid signature, capacity error, full pool, etc.). Each rejected tx is stored in `RecentReject` via `put()`. Because `shrink()` never fires, the `count_limit` invariant is permanently violated and disk usage grows without bound, leading to disk exhaustion and node crash/unavailability.

### Likelihood Explanation
The attacker path requires only the ability to submit transactions — a standard, unauthenticated operation available over both P2P and RPC. No special privileges, keys, or hashpower are needed. The transactions do not need to be valid; they only need to be distinct (different hashes) to avoid the `Reject::Duplicated` early-exit path. This is trivially achievable.

### Recommendation
Add the missing write-back in the non-shrink branch of `put()`:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    } else {
        self.total_keys_num = total_keys_num;  // ← fix
    }
} else {
    self.shrink()?;
}
```

### Proof of Concept
With `shard_num=2`, `count_limit=10`, starting from an empty DB (`total_keys_num=0`):

1. Insert 10,000 distinct rejected tx hashes via `put()`.
2. Each call evaluates `0 + 1 = 1 > 10` → false → no shrink, no write-back.
3. After 10,000 inserts, `self.total_keys_num` is still `0`; the DB contains 10,000 entries.
4. Assert `self.total_keys_num == 0` and `db.estimate_num_keys_cf(...)` ≈ 10,000 — confirming the invariant is broken and disk usage is unbounded.
