Audit Report

## Title
Out-of-Bounds Index in `remove_unconfirmed_tx` Causes Node Panic — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary

`TxConfirmStat::remove_unconfirmed_tx` guards the `block_unconfirmed_txs` decrement with a bounds check when `tx_age >= 1000`, but the subsequent `confirm_blocks_to_failed_txs[tx_age - 1]` write at line 215 has no equivalent guard. When a tracked transaction has been in the pool for more than 1000 blocks and is then rejected, `tx_age - 1 >= 1000` exceeds the array's valid range (0–999), causing a Rust index-out-of-bounds panic that crashes the node process.

## Finding Description

`MAX_CONFIRM_BLOCKS` is set to 1000 and both `confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` are allocated with exactly 1000 rows (valid indices 0–999): [1](#0-0) [2](#0-1) 

In `remove_unconfirmed_tx`, the guard at line 208 correctly handles `block_unconfirmed_txs` for `tx_age >= 1000` (routing to `old_unconfirmed_txs`), but the `count_failure` write at line 215 is placed unconditionally outside that branch with no bounds check: [3](#0-2) 

When `tx_age = 1001` (i.e., `tip_height - entry_height = 1001`), `tx_age - 1 = 1000` is past the last valid index (999). Rust panics with an index-out-of-bounds, crashing the node.

The full call chain is confirmed in the code:

- `reject_tx` (line 475–477) calls `drop_tx`
- `drop_tx` (line 428–430) calls `drop_tx_inner(tx_hash, true)`
- `drop_tx_inner` (line 416–424) calls `remove_unconfirmed_tx(..., count_failure=true)` [4](#0-3) [5](#0-4) 

This is wired into the tx-pool rejection path: [6](#0-5) 

## Impact Explanation

A Rust index-out-of-bounds panic is an unrecoverable abort. Any CKB node running the `ConfirmationFraction` fee estimator that evicts or rejects a transaction tracked for more than 1000 blocks will crash. This matches the allowed bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**, triggered by an unprivileged external user.

## Likelihood Explanation

At CKB's ~8-second block time, 1000 blocks ≈ 2.2 hours. The default transaction expiry is 72 hours, meaning transactions routinely remain in the pool far longer than 1000 blocks. An attacker submits a low-fee transaction (below the mining threshold) that will never be confirmed. After 1000+ blocks, the node's own eviction logic (expiry, RBF, or pool-size eviction) calls `reject_tx`, triggering the panic. No special privilege is required — only the ability to submit a transaction via P2P relay or the `send_transaction` RPC.

## Recommendation

Add a bounds check before the `count_failure` write, mirroring the guard already used for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

## Proof of Concept

1. Configure the node with the `ConfirmationFraction` fee estimator.
2. Submit a transaction with a fee rate below the mining threshold at block height `H`. The estimator records it via `track_tx`.
3. Allow the chain to advance past block `H + 1000` (approximately 2.2 hours at 8s/block).
4. Trigger eviction (wait for expiry, submit an RBF replacement, or fill the pool). The node calls `reject_tx(&tx_hash)`.
5. Execution reaches `remove_unconfirmed_tx(H, H+1001, bucket_index, true)`.
6. `tx_age = 1001`; `confirm_blocks_to_failed_txs[1000]` is accessed — index out of bounds — **node panics and crashes**.

A unit test can reproduce this directly against `TxConfirmStat`:

```rust
let mut stat = TxConfirmStat::new(buckets, 1000, decay);
stat.add_unconfirmed_tx(0, fee_rate);
// Simulate tx_age = 1001
stat.remove_unconfirmed_tx(0, 1001, bucket_index, true); // panics
```
