### Title
Missing WebSocket Message-Size Limit Allows Memory Exhaustion DoS — (`File: apps/quorum/src/ws.rs`)

### Summary
The `apps/quorum` WebSocket server upgrades connections without setting any maximum inbound message size, unlike the Hermes WebSocket server which explicitly caps client messages at 1 MiB. Any client that can reach the quorum endpoint can send an arbitrarily large WebSocket frame (up to the tungstenite library default of 64 MiB per message), causing the server to buffer the entire payload in memory before the application layer receives it. Repeated connections each sending maximum-size frames can exhaust available memory and crash the process.

### Finding Description
`apps/hermes/server/src/api/ws.rs` explicitly sets a 1 MiB inbound message cap before upgrading the connection:

```rust
const MAX_CLIENT_MESSAGE_SIZE: usize = 1025 * 1024; // 1 MiB
...
let mut ws = ws.max_message_size(MAX_CLIENT_MESSAGE_SIZE);
``` [1](#0-0) [2](#0-1) 

The `apps/quorum` WebSocket route handler performs no equivalent call:

```rust
pub async fn ws_route_handler(
    ws: WebSocketUpgrade,
    state: axum::extract::State<State>,
) -> impl IntoResponse {
    ws.on_upgrade(move |socket| websocket_handler(state, socket))
}
``` [3](#0-2) 

No `max_message_size`, `max_frame_size`, or equivalent guard is applied anywhere in the quorum WebSocket path. The `handle_client_message` function accepts `Text` and `Binary` frames silently (returns `Ok(())`), meaning the server will fully buffer any size frame sent by a client before discarding it. [4](#0-3) 

The tungstenite library default maximum message size is 64 MiB. With no limit set, each connected client can force the server to allocate up to 64 MiB per message. With many concurrent connections each sending maximum-size frames, the process heap grows without bound until OOM.

### Impact Explanation
An attacker who can reach the quorum server's listening port opens N concurrent WebSocket connections and sends a 64 MiB frame on each. The server allocates N × 64 MiB of heap memory before any application-level code runs. This exhausts available RAM, triggering an OOM kill of the quorum process. The quorum server relays VAA data to Hermes nodes; its unavailability degrades or halts price-feed propagation for the duration of the attack.

### Likelihood Explanation
The quorum server is a standalone Rust application (`apps/quorum`) that binds a TCP port and accepts unauthenticated WebSocket connections — no API key, no IP allowlist, and no authentication is visible in the handler code. [5](#0-4) 

If the port is reachable from the public internet (or from any network segment an attacker controls), exploitation requires only a standard WebSocket client. The attack is low-skill, requires no credentials, and can be repeated immediately after a process restart. Likelihood is **medium** because it depends on whether the quorum port is exposed beyond trusted infrastructure.

### Recommendation
Apply the same guard used in the Hermes server:

```rust
const MAX_CLIENT_MESSAGE_SIZE: usize = 1025 * 1024; // 1 MiB

pub async fn ws_route_handler(
    ws: WebSocketUpgrade,
    state: axum::extract::State<State>,
) -> impl IntoResponse {
    ws.max_message_size(MAX_CLIENT_MESSAGE_SIZE)
      .on_upgrade(move |socket| websocket_handler(state, socket))
}
```

Additionally, restrict the quorum port to known trusted IP ranges at the network/firewall level so that unauthenticated WebSocket access is not reachable from arbitrary clients.

### Proof of Concept
1. Identify the quorum server's listening address (configured via `--listen-addr`).
2. Open a WebSocket connection to `ws://<host>:<port>`.
3. Send a single binary frame of 64 MiB (tungstenite default limit):
   ```python
   import websocket, os
   ws = websocket.create_connection("ws://<host>:<port>")
   ws.send_binary(os.urandom(64 * 1024 * 1024))
   ```
4. Repeat step 2–3 from multiple concurrent clients.
5. Observe the quorum process RSS growing by 64 MiB per connection until OOM termination, confirmed by `dmesg` or process monitoring.

The root cause is the absence of `ws.max_message_size(...)` in `apps/quorum/src/ws.rs` line 43, in direct contrast to the explicit 1 MiB cap applied in `apps/hermes/server/src/api/ws.rs` line 229.

    let subscriber_id = state.ws.subscriber_counter.fetch_add(1, Ordering::SeqCst);
    let (sender, receiver) = stream.split();
    let new_receiver = state.ws.broadcast_receiver.resubscribe();
    let mut subscriber = Subscriber::new(subscriber_id, new_receiver, receiver, sender);
    subscriber.run().await;
}

    async fn handle_client_message(&mut self, message: Message) -> Result<()> {
        let start = std::time::Instant::now();
        let message_type;

        let result: anyhow::Result<()> = match message {
            Message::Close(_) => {
                // Closing the connection. We don't remove it from the subscribers
                // list, instead when the Subscriber struct is dropped the channel
                // to subscribers list will be closed and it will eventually get
                // removed.
                // Send the close message to gracefully shut down the connection
                // Otherwise the client might get an abnormal Websocket closure
                // error.
                message_type = "close";
                self.sender.close().await?;
                self.closed = true;
                Ok(())
            }
            Message::Text(_) => {
                message_type = "text";
                Ok(())
            }
            Message::Binary(_) => {
                message_type = "binary";
                Ok(())
            }
            Message::Ping(_) => {
                message_type = "ping";
                Ok(())
            }
            Message::Pong(_) => {
                message_type = "pong";
