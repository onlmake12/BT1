### Title
Hardcoded `AVG_BLOCK_INTERVAL` in Fee Estimator Causes Miscalibrated Fee Priority Tiers When Actual Block Times Diverge - (File: `util/fee-estimator/src/constants.rs`)

### Summary

The CKB fee estimator defines all time-based fee priority targets (NoPriority ≈ 1 hour, LowPriority ≈ 30 min, MediumPriority ≈ 10 min) using a compile-time constant `AVG_BLOCK_INTERVAL` derived from the hardcoded `MAX_BLOCK_INTERVAL` and `MIN_BLOCK_INTERVAL` bounds. Because the actual CKB block time is dynamically adjusted by the difficulty algorithm and can differ substantially from this hardcoded average, the block-count targets that represent each priority tier become permanently miscalibrated relative to real wall-clock time. This is a direct analog to the Basin oracle's immutable `BLOCK_TIME` causing economic flaws when actual block times change.

### Finding Description

In `util/fee-estimator/src/constants.rs`, the fee estimator computes all priority-tier block-count targets from a single compile-time constant:

```rust
pub(crate) const AVG_BLOCK_INTERVAL: u64 = (MAX_BLOCK_INTERVAL + MIN_BLOCK_INTERVAL) / 2;
// = (48 + 8) / 2 = 28 seconds

pub(crate) const MAX_TARGET: BlockNumber = (60 * 60) / AVG_BLOCK_INTERVAL;
// = 3600 / 28 = 128 blocks  (labeled "about 1 hour")
pub const LOW_TARGET:    BlockNumber = DEFAULT_TARGET / 2;   // 64 blocks, "about 30 minutes"
pub const MEDIUM_TARGET: BlockNumber = LOW_TARGET / 3;       // 42 blocks, "about 10 minutes"
pub const HIGH_TARGET:   BlockNumber = MIN_TARGET;           // 5 blocks
``` [1](#0-0) 

`MAX_BLOCK_INTERVAL` (48 s) and `MIN_BLOCK_INTERVAL` (8 s) are themselves compile-time constants in `spec/src/consensus.rs`: [2](#0-1) 

These block-count targets are then used directly in `FeeEstimator::target_blocks_for_estimate_mode` to map each `EstimateMode` variant to a fixed block count: [3](#0-2) 

The `WeightUnitsFlow` algorithm uses `MAX_TARGET` to expire historical data and to compute `removed_weight` per block: [4](#0-3) [5](#0-4) 

The `ConfirmationFraction` algorithm similarly uses `target_blocks` derived from these constants to estimate confirmation probability windows.

**The actual CKB block time is not 28 seconds.** The difficulty adjustment algorithm targets `epoch_duration_target = 14,400 seconds` over `genesis_epoch_length = 1,000` blocks, yielding an expected block time of ~14.4 seconds — roughly half of `AVG_BLOCK_INTERVAL`. The epoch length itself adjusts dynamically between `MIN_EPOCH_LENGTH = 300` and `MAX_EPOCH_LENGTH = 1,800` blocks, meaning actual block times can range widely: [6](#0-5) 

None of these runtime-adjusted values feed back into the fee estimator's compile-time constants. The constants are fixed at build time and cannot be updated without a full code change and node redeployment.

A secondary instance of the same pattern appears in `sync/src/types/mod.rs`, where `tip_synced()` computes a re-sync delay using the same hardcoded average:

```rust
let avg_interval = (MAX_BLOCK_INTERVAL + MIN_BLOCK_INTERVAL) / 2;
self.headers_sync_state = HeadersSyncState::TipSynced(now + avg_interval * 1000);
``` [7](#0-6) 

And `POW_INTERVAL = 10` (seconds) is used in the headers-sync timeout calculation to estimate expected sync progress, another hardcoded block-time assumption: [8](#0-7) [9](#0-8) 

### Impact Explanation

**Too-conservative case (actual block time < `AVG_BLOCK_INTERVAL`):** At the actual ~14.4 s block time, `MAX_TARGET = 128 blocks` represents only ~30 minutes of real time, not 1 hour. Users calling `estimate_fee_rate(NoPriority)` via RPC receive a fee rate calibrated for 128 blocks but their transaction is confirmed in ~30 minutes. They systematically overpay fees relative to their actual confirmation-time requirement.

**Too-aggressive case (actual block time > `AVG_BLOCK_INTERVAL`):** If a protocol upgrade or sustained hash-rate drop causes actual block times to exceed 28 seconds, `MAX_TARGET = 128 blocks` represents more than 1 hour of real time. Users relying on `HighPriority` (5 blocks) for urgent transactions would wait far longer than the labeled "next block" expectation. The `WeightUnitsFlow` algorithm's `removed_weight` calculation (`MAX_BLOCK_BYTES * 85/100 * target_blocks`) would underestimate how much weight is cleared per real-time unit, causing the estimator to recommend fee rates that are too low, leaving transactions stuck in the mempool.

The `tip_synced` re-sync delay and `POW_INTERVAL`-based sync-speed threshold are similarly miscalibrated: if actual block times are shorter than the hardcoded values, the node re-syncs headers less frequently than optimal; if longer, it may incorrectly flag honest peers as too slow and disconnect them.

### Likelihood Explanation

CKB's difficulty adjustment algorithm actively changes the epoch length (and thus effective block time) every epoch in response to hash rate. The actual block time has historically differed from `AVG_BLOCK_INTERVAL`. Any future protocol upgrade that changes `epoch_duration_target` or the epoch-length bounds would immediately invalidate the hardcoded fee-estimator constants without any corresponding update mechanism. Any RPC caller using `get_fee_rate_statics` or `estimate_fee_rate` is affected.

### Recommendation

1. Replace the compile-time `AVG_BLOCK_INTERVAL` constant with a runtime value derived from the live `Consensus` object's `epoch_duration_target` and the current epoch's actual length, so that block-count targets track real wall-clock time dynamically.
2. Pass the `Consensus` reference into the fee estimator at construction time and recompute `MAX_TARGET`, `LOW_TARGET`, `MEDIUM_TARGET` from `consensus.epoch_duration_target() / consensus.genesis_epoch_ext().length()` (or a rolling average of recent epoch block times).
3. Similarly, replace the hardcoded `POW_INTERVAL` in the sync timeout calculation with a value derived from the consensus epoch duration target.

### Proof of Concept

At the current CKB mainnet parameters:
- `epoch_duration_target = 14,400 s`, `genesis_epoch_length = 1,000` → expected block time ≈ 14.4 s
- `AVG_BLOCK_INTERVAL = 28 s` (hardcoded)
- `MAX_TARGET = 3600 / 28 = 128 blocks`
- Actual wall-clock time for 128 blocks at 14.4 s/block = **1,843 s ≈ 30.7 minutes**, not 1 hour

An RPC caller invoking `estimate_fee_rate` with `EstimateMode::NoPriority` receives a fee rate calibrated for a 128-block (labeled "1 hour") window, but the transaction is confirmed in ~30 minutes. Conversely, if a future upgrade doubles the epoch duration target to 8 hours (block time ~28.8 s), `MAX_TARGET = 128 blocks` would represent ~61 minutes — coincidentally correct — but `MEDIUM_TARGET = 42 blocks` would represent ~20 minutes instead of the labeled 10 minutes, causing users to underpay for medium-priority transactions and have them stuck in the mempool.
