Audit Report

## Title
Missing `BLOCK_INVALID` Status Guard in `HeaderAcceptor::accept()` Enables Repeated PoW Re-verification and Sync State Corruption - (File: sync/src/synchronizer/headers_process.rs)

## Summary

`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` with no guard for `BLOCK_INVALID`. Because `BLOCK_INVALID` (`1 << 12`) is disjoint from `HEADER_VALID` (`1`), a header whose block was marked invalid falls through all three sub-checks and reaches `insert_valid_header()`. Critically, `insert_valid_header()` only writes to `header_map` — not `block_status_map` — so `get_block_status()` continues returning `BLOCK_INVALID` on every subsequent call, making the PoW re-verification path permanently reachable for that header on every repeated `SendHeaders` message.

## Finding Description

In `sync/src/synchronizer/headers_process.rs` at lines 301–303, the developer explicitly acknowledges the missing guard:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
```

The only early-return check is `status.contains(BlockStatus::HEADER_VALID)` (line 304). From `shared/src/block_status.rs` lines 11 and 16, `HEADER_VALID = 1` and `BLOCK_INVALID = 1 << 12` are completely disjoint bits, so a `BLOCK_INVALID` status never satisfies this check.

Execution then falls through to:
1. `prev_block_check` (line 324) — checks the **parent** block's status, not the current block's. If the parent is valid, this passes.
2. `non_contextual_check` (line 334) — calls `HeaderVerifier::verify()`, which includes PoW verification. A block with a valid header but invalid body (e.g., script failure) has a structurally valid header that passes this check.
3. `version_check` (line 346) — passes for version 0.
4. `sync_shared.insert_valid_header(self.peer, self.header)` (line 356) is called.

The critical detail that makes this **permanently repeatable** is in `insert_valid_header()` at `sync/src/types/mod.rs` lines 1094–1141: it only inserts into `header_map` (line 1129) and updates peer state (line 1132). It does **not** write to `block_status_map`.

`get_block_status()` in `shared/src/shared.rs` lines 425–444 checks `block_status_map` first:
```rust
match self.block_status_map().get(block_hash) {
    Some(status_ref) => *status_ref.value(),  // returns BLOCK_INVALID every time
    None => { if self.header_map().contains_key(...) { HEADER_VALID } ... }
}
```

Since `BLOCK_INVALID` was written to `block_status_map` by the chain verifier and `insert_valid_header` never clears or overwrites it, every subsequent call to `get_block_status` for that hash still returns `BLOCK_INVALID`. The `HEADER_VALID` check at line 304 fails again, and the full `non_contextual_check` (PoW) runs again — indefinitely.

By contrast, `compact_block_process.rs` lines 259–261 correctly short-circuits:
```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
```

## Impact Explanation

**High — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs"**

An attacker with a single `SendHeaders` message containing one known-invalid header can force the victim node to execute `HeaderVerifier::verify()` (including PoW verification) on every message receipt, with no caching protection. The attacker's cost is negligible (one TCP message per iteration); the victim's cost is repeated PoW verification. Additionally, `best_known_header` for the attacking peer is updated to point to the invalid block's header, potentially misdirecting the block fetcher into repeatedly requesting and attempting to verify a block that will always fail.

## Likelihood Explanation

The exploit requires only a standard P2P connection — no privileges, no special network position. A block with a valid header but invalid body (invalid scripts, invalid DAO fields) is trivially constructable on testnet or observable on mainnet after any rejected block. The `FIXME` comment at line 301 is the developers' own acknowledgment that the gap exists. The attack is repeatable indefinitely with a single such header.

## Recommendation

Add an early-return guard for `BLOCK_INVALID` immediately after reading the status, before the `HEADER_VALID` check, mirroring the pattern in `compact_block_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a new BlockInvalid variant
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early return
}
```

A dedicated `ValidationError::BlockInvalid` variant would be cleaner than reusing `InvalidParent`, resolving the `FIXME`'s open question about which error type to return.

## Proof of Concept

1. Construct block B with a valid header (valid PoW, valid parent, version 0) but invalid body (e.g., a script that always fails).
2. Submit B to node N via the normal block submission path. N fully verifies B, script execution fails, and `BLOCK_INVALID` is written to `block_status_map` for B's hash.
3. Connect attacker peer P to N. P sends a `SendHeaders` message containing only B's header.
4. `HeaderAcceptor::accept()` is called. `get_block_status` returns `BLOCK_INVALID` from `block_status_map`. The `HEADER_VALID` check at line 304 fails (disjoint bit). No `BLOCK_INVALID` guard exists.
5. `prev_block_check` passes (B's parent is valid). `non_contextual_check` runs full `HeaderVerifier::verify()` including PoW — passes. `version_check` passes.
6. `insert_valid_header(peer, &B_header)` is called. This writes to `header_map` only; `block_status_map` still holds `BLOCK_INVALID`.
7. Repeat step 3 in a tight loop. Each iteration forces a full PoW check. `get_block_status` returns `BLOCK_INVALID` every time (from `block_status_map`), the `HEADER_VALID` check fails every time, and `non_contextual_check` runs every time — with no termination condition.
