### Title
Unbounded O(N) Fisher-Yates Shuffle Per Inbound GetNodes Message with No Reconnect Rate Limit — (`network/src/peer_store/addr_manager.rs`, `network/src/protocols/discovery/mod.rs`)

### Summary

An unprivileged remote peer can repeatedly connect, send one `GetNodes` message, disconnect, and reconnect to force the victim node to execute a full O(N) Fisher-Yates shuffle over `AddrManager::random_ids` on every cycle. The per-session `received_get_nodes` guard is reset to `false` on each new `SessionState::new()`, so the one-per-session deduplication provides no protection across reconnects.

### Finding Description

**Step 1 — `received_get_nodes` resets on every new session.**

`SessionState::new()` always initializes `received_get_nodes: false`: [1](#0-0) 

The guard in `DiscoveryProtocol::received` only disconnects a peer that sends a *second* `GetNodes` within the *same* session: [2](#0-1) 

Disconnect → reconnect → fresh `SessionState` → `received_get_nodes = false` again. The guard is fully bypassed.

**Step 2 — Every accepted GetNodes triggers `get_random(2500, flags)`.** [3](#0-2) 

This calls `DiscoveryAddressManager::get_random` → `peer_store.fetch_random_addrs(n, flags)`: [4](#0-3) 

Which calls `addr_manager.fetch_random(count, filter)`: [5](#0-4) 

**Step 3 — `AddrManager::fetch_random` performs a full O(N) Fisher-Yates shuffle.**

The loop iterates over every entry in `random_ids`, performing a random swap and two `HashMap` lookups per iteration. It only exits early if `count` results pass the filter: [6](#0-5) 

The `fetch_random_addrs` filter requires `peer_addr.connected(|t| t > addr_expired_ms)` (connected within 7 days). If the peer store is populated with addresses that have `last_connected_at_ms = 0` (discovered but never connected), **zero entries pass the filter** and the loop runs through all N entries without early termination. This is the worst-case amplification scenario.

**Step 4 — No per-IP reconnect rate limiting in the discovery layer.**

`DiscoveryAddressManager::register` only checks `is_anchor`; there is no per-IP connection frequency limit or cooldown at the discovery protocol level: [7](#0-6) 

The network layer's total inbound connection cap limits *simultaneous* connections but does not prevent rapid sequential reconnects from the same peer.

### Impact Explanation

Each reconnect cycle forces one O(N) shuffle over `random_ids`. With `ADDR_COUNT_LIMIT` entries (the question cites N=16,384), each call performs ~16,384 iterations of: `rng.gen_range`, two `HashMap` lookups with `get_mut`, and an `AddrInfo` clone. An attacker cycling connections at even modest rates (tens per second) can sustain continuous CPU pressure on the victim's network thread, degrading block/transaction relay and sync throughput, potentially causing the node to fall behind the chain tip.

### Likelihood Explanation

The attack requires only an unprivileged TCP connection to the victim's P2P port. No PoW, no keys, no special privileges. The code path is direct and requires no race conditions. The worst-case filter-miss scenario (peer store full of never-connected addresses) is realistic on a node that has received many discovered addresses but has not yet connected to them.

### Recommendation

1. **Cache or rate-limit `fetch_random` results**: Cache the shuffled result for a short TTL (e.g., 30 seconds) so repeated calls within the window reuse the cached output.
2. **Add per-IP GetNodes rate limiting**: Track the last GetNodes response time per remote IP and refuse to re-serve within a cooldown window (e.g., 60 seconds), independent of session identity.
3. **Early-exit optimization**: If the filter is known to be highly selective, consider maintaining a pre-filtered index rather than scanning all N entries on every call.
4. **Connection rate limiting per IP**: Implement a per-source-IP reconnect cooldown at the network layer to bound the reconnect cycle rate.

### Proof of Concept

```
1. Populate victim's peer store with N=16384 addresses (last_connected_at_ms=0).
2. Connect to victim's P2P port (inbound session created, received_get_nodes=false).
3. Send a valid GetNodes message.
   → Victim calls fetch_random_addrs(2500, flags) → full O(N) shuffle, 0 results pass filter.
4. Disconnect.
5. Repeat steps 2–4 in a tight loop.
6. Measure: victim CPU on network thread grows linearly with reconnect count.
   Assert: no rate limiting prevents step 5 from running at connection-setup speed.
```

    fn get_random(&mut self, n: usize, flags: Flags) -> Vec<(Multiaddr, Flags)> {
        let fetch_random_addrs = self
            .network_state
            .with_peer_store_mut(|peer_store| peer_store.fetch_random_addrs(n, flags));
        let addrs = fetch_random_addrs
            .into_iter()
            .filter_map(|paddr| {
                if !self.is_valid_addr(&paddr.addr) {
                    return None;
                }
                let f = Flags::from_bits_truncate(paddr.flags);
                Some((paddr.addr, f))
            })
            .collect();
        trace!("Discovered random addrs: {:?}", addrs);
        addrs
    }

    pub fn fetch_random_addrs(&mut self, count: usize, required_flags: Flags) -> Vec<AddrInfo> {
        // Get info:
        // 1. Connected within 7 days

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let addr_expired_ms = now_ms.saturating_sub(ADDR_TIMEOUT_MS);

        let filter = |peer_addr: &AddrInfo| {
            required_flags_filter(required_flags, Flags::from_bits_truncate(peer_addr.flags))
                && peer_addr.connected(|t| t > addr_expired_ms)
        };

        // get success connected addrs.
        self.addr_manager.fetch_random(count, filter)
    }
