Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Spoofed `from` PeerId — (`network/src/protocols/hole_punching/mod.rs`, `network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter` accepts attacker-controlled `content.from` and `content.to` fields as HashMap keys with no validation that `content.from` matches the actual sending peer. The outer rate limiter guarantees exactly 30 new unique keys per second per session can be inserted. Because `retain_recent()` is called only in `disconnected()` and never in `notify()`, entries accumulate without bound for the duration of any connection, enabling OOM-based node crash.

## Finding Description
**Root cause — no sender validation:** `content.from` is parsed directly from message bytes at [1](#0-0)  with no check that it matches the actual sending session's peer ID. The `ConnectionRequestProcess` receives the real sender as `peer: PeerIndex` but never compares it against `content.from`.

**Inner rate limiter keyed by attacker-controlled tuple:** The `forward_rate_limiter` is checked at [2](#0-1)  using `(content.from, content.to, self.msg_item_id)`. Each unique `from` value creates a new `HashMapStateStore` entry. Since `msg_item_id` is fixed per message type and `to` can also be varied, the attacker controls the key space entirely.

**Outer rate limiter is the attack enabler, not a defense:** The outer limiter is keyed by `(session_id, msg.item_id())` at [3](#0-2)  with a quota of 30/sec (set at [4](#0-3) ). This guarantees exactly 30 `ConnectionRequest` messages per second per session reach `execute()`, each with a distinct spoofed `from`, creating 30 new HashMap entries per second.

**No periodic cleanup:** `retain_recent()` is called only in `disconnected()` at [5](#0-4) . The `notify()` callback fires every `CHECK_INTERVAL = 5 minutes` but only prunes `pending_delivered` and `inflight_requests` at [6](#0-5)  — it never calls `retain_recent()` on either rate limiter. A grep across the entire codebase confirms `retain_recent` appears only twice, both in `disconnected()`.

**Shared state across all sessions:** `forward_rate_limiter` is a single field on the `HolePunching` struct at [7](#0-6) , shared across all concurrent sessions. Multiple attacker connections each contribute 30 entries/sec to the same HashMap.

## Impact Explanation
This is a **High** severity vulnerability matching: **"Vulnerabilities which could easily crash a CKB node."**

Each `(PeerId, PeerId, u32)` entry in `HashMapStateStore` occupies approximately 150–200 bytes. With 50 concurrent attacker sessions at 30 entries/sec each: ~90 MB in 5 minutes, ~1 GB in 1 hour, ~8.6 GB in 8 hours — sufficient to exhaust typical node memory and trigger OOM crash. The attack requires no special privileges, only a standard inbound P2P connection.

## Likelihood Explanation
Any unprivileged peer on the public internet can open a connection to a CKB node. Generating 30 syntactically valid `PeerId` values per second is trivial (`PeerId::from_bytes` accepts any valid multihash). The outer rate limiter is not a defense — it is precisely what guarantees 30 new `forward_rate_limiter` entries per second per session. The attack scales linearly with concurrent sessions and is repeatable indefinitely without disconnect.

## Recommendation
1. **Validate `from` against the actual sender:** In `ConnectionRequestProcess::execute()`, reject messages where `content.from` does not match the peer ID of the sending session (looked up via `peer: PeerIndex`). This eliminates the ability to create arbitrary new keys.
2. **Call `retain_recent()` periodically:** Inside the `notify()` callback, call `retain_recent()` on both `rate_limiter` and `forward_rate_limiter`, not only on disconnect.
3. **Cap the `forward_rate_limiter` store size:** Add a maximum entry count and evict oldest entries when the cap is reached as defense-in-depth.

## Proof of Concept
```rust
// Setup: connect one attacker session
let mut protocol = HolePunching::new(network_state);
let session_id = PeerIndex::new(1);

// Attack: send 30 ConnectionRequest/sec for 300 seconds, each with unique `from`
for second in 0..300 {
    for i in 0..30 {
        let from = generate_unique_peer_id(second * 30 + i); // distinct each time
        let to   = generate_peer_id(0);
        let msg  = build_connection_request(from, to, max_hops=1);
        protocol.received(ctx_for(session_id), msg).await;
    }
    advance_clock_by(1_second);
    // notify() fires every 5 min but does NOT call retain_recent()
}

// After 300 seconds with no disconnect, retain_recent() has never been called.
// forward_rate_limiter contains ~9,000 entries.
// With 50 concurrent attacker sessions: ~450,000 entries (~90 MB) in 5 minutes.
assert!(
    forward_rate_limiter_len(&protocol) <= 30, // FAILS: actual ~9000
    "invariant violated: forward_rate_limiter grew unboundedly"
);
```
