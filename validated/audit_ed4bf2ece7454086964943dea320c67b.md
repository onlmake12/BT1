### Title
`block.number`-Based `blockhash` Window Expires Near-Instantly on Fast-Block Networks, Permanently Locking User Fees â€” (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The legacy `request()` function in `Entropy.sol` allows callers to set `useBlockHash = true`, which causes the contract to record `block.number` at request time and later retrieve `blockhash(req.blockNumber)` during reveal. On EVM-compatible networks where each transaction produces its own block (e.g., Optimism), the EVM's 256-block `blockhash` availability window expires in seconds. Any `useBlockhash = true` request on such a network becomes permanently irrevealable, and the user's paid fee is locked in the contract with no refund path.

---

### Finding Description

In `requestHelper`, the current block number is stored into the request struct:

```solidity
req.blockNumber = SafeCast.toUint64(block.number);
req.useBlockhash = useBlockhash;
``` [1](#0-0) 

During `revealHelper`, if `req.useBlockhash` is true, the contract calls:

```solidity
bytes32 _blockHash = blockhash(req.blockNumber);
if (_blockHash == bytes32(uint256(0)))
    revert EntropyErrors.BlockhashUnavailable();
``` [2](#0-1) 

The EVM `blockhash` opcode returns zero for any block older than 256 blocks. On mainnet Ethereum (~12s/block), 256 blocks â‰ˆ 51 minutes â€” a reasonable window. On Optimism, where each transaction is placed in its own block and blocks are not produced at a constant rate, 256 blocks can pass in seconds (as few as 256 transactions by other users). Once the window expires, `blockhash(req.blockNumber)` returns `bytes32(0)` and the reveal permanently reverts.

The `Request` struct stores `blockNumber` as a `uint64`: [3](#0-2) 

The legacy `request()` function â€” still live and callable by any unprivileged user â€” accepts `useBlockHash` as a caller-controlled boolean: [4](#0-3) 

There is no cancel or refund function in the contract. Once fees are paid into `providerInfo.accruedFeesInWei` and `_state.accruedPythFeesInWei` inside `requestHelper`, they are irrecoverable by the user: [5](#0-4) 

Note: `requestWithCallback` and `requestV2` both hardcode `useBlockhash = false` and are not affected. Only the legacy `request()` path is vulnerable. [6](#0-5)

### Impact Explanation

A user who calls `request(provider, userCommitment, true)` on Optimism (or any fast-block EVM chain) pays a fee in `msg.value`. Within seconds â€” as soon as 256 subsequent transactions are mined â€” the blockhash for their request block becomes unavailable. Every subsequent call to `reveal()` reverts with `BlockhashUnavailable`. The user's fee is permanently locked in the contract, split between the provider and Pyth protocol balances. The user receives no random number and no refund.

---

### Likelihood Explanation

The `request()` function with `useBlockHash = true` is a live, permissionless entry point. Pyth Entropy is deployed on Optimism and other fast-block L2s. On Optimism, 256 blocks can pass in under a minute of normal network activity. Any user or integrating contract that calls the legacy `request()` with `useBlockHash = true` on such a network will reliably trigger this condition. The likelihood is **medium**: it requires use of the legacy function path, but that path is publicly documented and callable by anyone.

---

### Recommendation

1. **Remove or gate `useBlockhash = true` on fast-block networks**: Add a check that reverts if `useBlockhash == true` on chains where the 256-block window is unreliable, or deprecate the parameter entirely (it is already `false` in all modern entry points).
2. **Add a request cancellation / refund path**: Allow the original requester to cancel a stuck request and recover their fee after a timeout, using `block.timestamp` rather than `block.number` to define the expiry window.
3. **Replace `block.number` with `block.timestamp`**: If a time-based expiry is needed, store `block.timestamp` in the request and compare against it during reveal, consistent with the analog fix described in the referenced report.

---

### Proof of Concept

1. Deploy or interact with the Pyth Entropy contract on Optimism.
2. Call `request(provider, userCommitment, true)` with sufficient fee. Note the returned `sequenceNumber` and the current `block.number` (call it `N`).
3. Wait for 256 Optimism transactions to be mined (seconds to low minutes under normal load).
4. Call `reveal(provider, sequenceNumber, userRandomness, providerRandomness)`.
5. Observe the transaction reverts with `BlockhashUnavailable` because `blockhash(N)` now returns `bytes32(0)`.
6. Confirm there is no `cancel` or `refund` function available to recover the fee.
