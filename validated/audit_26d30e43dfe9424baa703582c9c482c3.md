### Title
Unhandled `unwrap()` on User-Controlled Deserialization Input Causes Panic in NEAR Receiver — (`target_chains/near/receiver/src/lib.rs`)

---

### Summary

The NEAR Pyth receiver contract's public price-update entry point and its downstream callbacks call `.unwrap()` directly on user-supplied byte buffers during deserialization. In NEAR Protocol, a panic in a contract function causes the transaction to revert and all state changes to be rolled back. While the attached deposit is refunded, the panic bypasses the contract's own `Result`-based error-handling path, silently discarding the structured error and preventing any meaningful on-chain error reporting. More critically, panics inside the two-step promise callbacks (`verify_wormhole_batch_callback`, `verify_wormhole_merkle_callback`) abort price-feed storage after Wormhole verification gas has already been consumed, and the `refund_vaa` safety net only refunds the deposit — it does not restore the gas cost of the Wormhole cross-contract call.

---

### Finding Description

`update_price_feeds` is a `#[payable]` public function callable by any NEAR account. It decodes a hex-encoded byte buffer supplied by the caller and immediately calls `.unwrap()` on two fallible operations before any cross-contract call is made: [1](#0-0) 

If the decoded buffer is fewer than 4 bytes, line 171 panics. If the buffer begins with `PYTHNET_ACCUMULATOR_UPDATE_MAGIC` but the remainder is not a valid `AccumulatorUpdateData`, line 176 panics.

For the non-accumulator (legacy batch) path, `verify_encoded_vaa_source` is called, which also uses `.unwrap()` on the raw user bytes: [2](#0-1) 

Any byte sequence that is not parseable by `serde_wormhole::from_slice` causes a panic here, even though the function signature returns `Result<(), Error>` and the caller uses `?` — the panic escapes the `Result` entirely.

Inside the Wormhole-verified callback `verify_wormhole_batch_callback`, four additional `.unwrap()` calls operate on the same user-supplied string re-passed as a callback argument: [3](#0-2) 

Notably, lines 254 and 263 parse the *inner payload* of the VAA. Wormhole only verifies the outer VAA signature; it does not validate the payload structure. A VAA with a valid signature but a payload that has a correct `P2W_MAGIC` header followed by malformed `BatchPriceAttestation` bytes will pass Wormhole verification and reach line 263, where `.unwrap()` panics.

Similarly, `verify_wormhole_merkle_callback` re-deserializes the full accumulator blob and the embedded VAA with `.unwrap()`: [4](#0-3) 

The governance callback `authorize_gov_source_transfer` uses `.expect()` (equivalent to `.unwrap()`) on both the outer VAA and the governance instruction payload after Wormhole verification: [5](#0-4) 

---

### Impact Explanation

In NEAR Protocol, a panic in a contract function reverts all state changes and refunds the attached deposit to the caller. This differs from the Dfinity/ICP case in the reference report (where a panic crashes the process), but the impact is still concrete:

1. **Structured error handling is bypassed.** The contract's `Result<(), Error>` return type and `#[handle_result]` attribute are designed to surface errors to callers and downstream promise chains. A panic escapes this path entirely, making the failure opaque.

2. **Gas is not refunded.** In the callback cases (`verify_wormhole_batch_callback`, `verify_wormhole_merkle_callback`), the Wormhole cross-contract call has already executed and consumed gas before the panic occurs. The `refund_vaa` safety net only returns the attached NEAR deposit — it does not compensate for the gas burned on the Wormhole call.

3. **Price feed updates are silently dropped.** A panic in either callback means no price feeds are stored, with no on-chain error code distinguishing this from a Wormhole verification failure.

4. **Governance liveness risk.** The `.expect()` calls in `authorize_gov_source_transfer` mean that a governance VAA whose outer signature is valid but whose payload is not parseable by `serde_wormhole` or `GovernanceInstruction::deserialize` will permanently fail to execute, with no recoverable error path.

---

### Likelihood Explanation

- **`update_price_feeds` (lines 171, 176, 626):** Any unprivileged NEAR account can call this function with arbitrary hex-encoded bytes. Triggering the panic at line 171 requires only a hex string shorter than 8 characters (4 decoded bytes). Triggering line 176 requires a string whose first 4 decoded bytes equal `PYTHNET_ACCUMULATOR_UPDATE_MAGIC` followed by malformed data. Both are trivially constructable. Likelihood: **High** for the pre-call panics.

- **Callback panics (lines 248–263, 330–336):** Requires a VAA that passes Wormhole signature verification but has a malformed inner payload. This requires either a compromised or colluding Wormhole guardian set, or a format mismatch between what Wormhole accepts and what `serde_wormhole`/`BatchPriceAttestation` expects. Likelihood: **Low-Medium**.

- **Governance callback (lines 410, 417):** Requires a governance VAA signed by the Pyth governance multisig with a malformed payload. Likelihood: **Low** under normal operation, but the `.expect()` creates a latent liveness risk if the payload format ever diverges.

---

### Recommendation

Replace every `.unwrap()` and `.expect()` on user-controlled or externally-sourced data with proper `Result`-propagating alternatives. Since the functions already return `Result<(), Error>`, use `.map_err(|_| Error::InvalidVaa)?` or equivalent:

```rust
// Before (panics):
cursor.clone().read_exact(&mut header).unwrap();

// After (propagates error):
cursor.clone().read_exact(&mut header).map_err(|_| Error::InvalidVaa)?;
```

```rust
// Before (panics):
let update_data = AccumulatorUpdateData::try_from_slice(&cursor.clone().into_inner()).unwrap();

// After:
let update_data = AccumulatorUpdateData::try_from_slice(&cursor.clone().into_inner())
    .map_err(|_| Error::InvalidAccumulatorMessage)?;
```

```rust
// Before (panics):
let vaa: wormhole_sdk::Vaa<&RawMessage> = serde_wormhole::from_slice(vaa).unwrap();

// After:
let vaa: wormhole_sdk::Vaa<&RawMessage> =
    serde_wormhole::from_slice(vaa).map_err(|_| Error::InvalidVaa)?;
```

Apply the same pattern to all `.unwrap()` / `.expect()` calls in `verify_wormhole_batch_callback`, `verify_wormhole_merkle_callback`, and `authorize_gov_source_transfer`.

### Proof of Concept

**Triggering the panic in `update_price_feeds` (line 171):**

```
# Encode 3 bytes (fewer than the 4-byte header read)
data = hex::encode([0x01, 0x02, 0x03])  # "010203"

# Call update_price_feeds with attached deposit
near call <pyth_contract> update_price_feeds '{"data": "010203"}' \
    --deposit 0.01 --accountId attacker.near
```

The `read_exact` at line 171 attempts to fill a `[0u8; 4]` buffer from 3 bytes, returns `Err(UnexpectedEof)`, and `.unwrap()` panics. The transaction reverts; the deposit is refunded; gas is consumed.

**Triggering the panic in `update_price_feeds` (line 176):**

```
# First 4 bytes = PYTHNET_ACCUMULATOR_UPDATE_MAGIC (0x504e4155), rest is garbage
magic = b"\x50\x4e\x41\x55"
garbage = b"\xff\xff\xff\xff\xff"
data = hex::encode(magic + garbage)

near call <pyth_contract> update_price_feeds '{"data": "<hex>"}' \
    --deposit 0.01 --accountId attacker.near
```

The header check passes, `AccumulatorUpdateData::try_from_slice` fails on the malformed remainder, and `.unwrap()` panics. [6](#0-5) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
