### Title
Unbounded `while` Loop in `executeCallback` Enables Gas Exhaustion DoS — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` function contains an unbounded `while` loop that linearly scans storage after every fulfilled request to advance the `firstUnfulfilledSeq` pointer. An unprivileged requester can force this loop to iterate over an arbitrarily large range of sequence numbers by creating many requests and having them fulfilled out of order, causing the final `executeCallback` to hit the block gas limit and revert — permanently blocking fulfillment of the earliest pending request.

---

### Finding Description

After a successful callback, `executeCallback` runs the following loop:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
// After successful callback, update firstUnfulfilledSeq if needed
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest(_state.firstUnfulfilledSeq)` (a storage read via ring-buffer index) and `isActive(...)` (another storage read). The number of iterations equals the number of consecutive already-fulfilled requests starting from `firstUnfulfilledSeq`. This is not bounded by any constant — it grows with the total number of requests created and fulfilled out of order.

The code itself carries a developer TODO acknowledging the gas problem and suggesting a doubly-linked list as the fix.

The ring-buffer size `NUM_REQUESTS` bounds how many requests can be *active simultaneously*, but does **not** bound the while-loop iteration count, because `currentSequenceNumber` is a monotonically increasing counter that is never reset. Once `firstUnfulfilledSeq` falls far behind `currentSequenceNumber` (due to out-of-order fulfillment), the loop must scan the entire gap.

The gas benchmark test explicitly documents the worst-case scenario:

```
// The last fulfillment will be the most expensive since it needs
// to linearly scan through all the fulfilled requests in storage
// in order to update _state.lastUnfulfilledReq
``` [2](#0-1) 

---

### Impact Explanation

- If the while loop exhausts the block gas limit, `executeCallback` reverts for the earliest unfulfilled request.
- The request's fee is held in the contract but the request cannot be fulfilled (the provider cannot skip the loop — it runs unconditionally after every callback).
- This constitutes a **permanent DoS** on the oldest pending request: the provider is forced to pay ever-increasing gas costs to fulfill it, and if gas exceeds the block limit, the request is stuck indefinitely.
- Fees paid by the requester are effectively locked.

---

### Likelihood Explanation

- Any unprivileged user can call `requestPriceUpdatesWithCallback` to create requests.
- Out-of-order fulfillment is a natural operational pattern (providers may fulfill by `publishTime` rather than `sequenceNumber`).
- No special privilege is required; the attacker only needs to pay the request fee.
- The `EchoGasBenchmark.t.sol` test already demonstrates the cost grows linearly with the number of out-of-order fulfilled requests. [3](#0-2) 

---

### Recommendation

Replace the linear scan with a doubly-linked list of active requests (as the TODO comment already suggests), or maintain a min-heap / sorted set of active sequence numbers so that `firstUnfulfilledSeq` can be advanced in O(1) per callback. Alternatively, cap the number of iterations in the while loop per transaction and allow the pointer to be advanced lazily in a separate call.

---

### Proof of Concept

1. Deploy `Echo.sol` with `NUM_REQUESTS = 128`.
2. Call `requestPriceUpdatesWithCallback` 128 times from an unprivileged address, creating requests with sequence numbers 1–128.
3. Provider calls `executeCallback` for requests 2–128 (in any order), leaving request 1 unfulfilled. After each of these calls, `firstUnfulfilledSeq` stays at 1 because request 1 is still active.
4. Provider calls `executeCallback` for request 1. After the callback, the while loop must iterate from `firstUnfulfilledSeq = 1` through `currentSequenceNumber = 129`, performing 127 storage reads (`findRequest` + `isActive`) to advance the pointer.
5. Each storage read costs ~2,100 gas (cold SLOAD). With 127 iterations × 2 SLOADs each ≈ 533,400 gas just for the loop — on top of the callback gas. Scaling to thousands of requests (which is realistic over time) pushes this toward the 30M block gas limit.
6. The `testMultipleRequestsOutOfOrderFulfillment` test in `EchoGasBenchmark.t.sol` already reproduces this exact scenario and measures the gas cost differential. [4](#0-3) [5](#0-4)
