Audit Report

## Title
Any Connected Peer Can Freeze the Sync Fallback Download Gate for 2 Seconds per Block via Stale `pending_compact_blocks` Timestamp — (File: `sync/src/relayer/compact_block_process.rs`)

## Summary
When a compact block cannot be fully reconstructed, `missing_or_collided_post_process` inserts the current timestamp into `pending_compact_blocks` using `or_insert_with`, freezing it at the moment the **first** peer triggers the path. Because `contextual_check` only rejects a message if the **same** peer is already in the map, any other connected peer can race to be first and freeze the timestamp at an attacker-controlled time. `BlockFetcher` then refuses to schedule a `GetBlocks` download for that block hash for the full 2-second window, delaying the sync fallback path for every new block the attacker targets.

## Finding Description
**Root cause — timestamp is set by the first peer, not the local node.**

In `sync/src/relayer/compact_block_process.rs` lines 354–361:
```rust
shared
    .state()
    .pending_compact_blocks()
    .await
    .entry(block_hash.clone())
    .or_insert_with(|| (compact_block, HashMap::default(), unix_time_as_millis()))
    .1
    .insert(peer, (missing_transactions.clone(), missing_uncles.clone()));
```
`or_insert_with` fires only on the first insertion. Any peer whose compact block fails reconstruction triggers this path and freezes the timestamp.

**Guard is insufficient — only the same peer is rejected.**

In `contextual_check` (lines 284–291):
```rust
if pending_compact_blocks
    .get(&block_hash)
    .map(|(_, peers_map, _)| peers_map.contains_key(&peer))
    .unwrap_or(false)
{
    return StatusCode::CompactBlockIsAlreadyPending.with_context(block_hash);
}
```
A different peer (the attacker) is not rejected and proceeds to `missing_or_collided_post_process`. The timestamp is already frozen; subsequent peers only add themselves to `peers_map`.

**Gate enforced in `BlockFetcher` for non-IBD nodes.**

In `sync/src/types/mod.rs` lines 1362–1370:
```rust
pub fn compare_with_pending_compact(&self, hash: &Byte32, now: u64) -> bool {
    let pending = self.pending_compact_blocks.blocking_lock();
    pending.is_empty()
        || pending
            .get(hash)
            .map(|(_, _, time)| now > time + 2000)
            .unwrap_or(true)
}
```
Returns `false` (blocking sync) while `now <= time + 2000`.

In `sync/src/synchronizer/block_fetcher.rs` lines 271–275:
```rust
} else if (matches!(self.ibd, IBDState::In)
    || state.compare_with_pending_compact(&hash, now))
    && state
        .write_inflight_blocks()
        .insert(self.peer, (header.number(), hash).into())
```
Non-IBD nodes skip the download entirely while the gate is closed.

## Impact Explanation
A connected peer can delay a victim non-IBD node's sync fallback path by up to 2 seconds per targeted block. If the attacker also ensures compact block relay fails (e.g., by using different short-ID nonces and ignoring `GetBlockTransactions`), the node cannot obtain the block through any path for the full 2-second window. Repeated across successive blocks, this constitutes a sustained propagation delay against a targeted node. This matches **Low (501–2000 points): Any other important performance improvements for CKB**, as the impact is bounded (2 s per block, node eventually syncs) and does not crash the node, cause consensus deviation, or damage the economy.

## Likelihood Explanation
The attacker requires only a standard TCP peer connection — no cryptographic material, operator keys, or hashpower. The attacker must learn the block hash before the victim's legitimate compact block relay completes, which is achievable by being better-connected to the mining pool or racing on the same network segment. The attack is repeatable for every new block and requires no victim interaction or mistake.

## Recommendation
Decouple the 2-second gate timestamp from peer-driven compact block arrival:
- Record the insertion timestamp only when the **local node** initiates the compact block request (i.e., after sending `GetBlockTransactions`), not when any arbitrary peer delivers a `CompactBlock`.
- Alternatively, replace the `pending_compact_blocks` gate in `BlockFetcher` with a direct check of `inflight_blocks`, which is only written by the local node's own download scheduler, eliminating the externally controllable timestamp entirely.

## Proof of Concept
1. Victim node V (non-IBD) is connected to attacker peer A and legitimate peer L.
2. Block H is mined; L prepares to relay it to V.
3. A races L: sends `CompactBlock(H)` to V with all short IDs claimed missing.
4. V calls `missing_or_collided_post_process`: `or_insert_with` fires, timestamp `T` is recorded.
5. V's `BlockFetcher` runs: `compare_with_pending_compact(H, now)` returns `false` (`now ≤ T + 2000`). No `GetBlocks` is issued.
6. L's `CompactBlock(H)` arrives; `contextual_check` passes (L not in `peers_map`). `or_insert_with` does **not** update the timestamp. L is added to `peers_map`.
7. A ignores V's `GetBlockTransactions`. If A used different nonces, reconstruction from L's response against A's stored compact block fails.
8. For the full 2-second window, V cannot fall back to sync for block H.
9. Attacker repeats for each new block, sustaining ≥2-second propagation delay against V.
