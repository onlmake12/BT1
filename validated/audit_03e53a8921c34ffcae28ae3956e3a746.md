### Title
SSRF via Unguarded HTTP Redirect Following in BlockAssembler Notify Client - (File: `tx-pool/src/block_assembler/mod.rs`)

### Summary
The `BlockAssembler` uses a `hyper_util::client::legacy::Client<HttpConnector>` to POST block-template notifications to operator-configured URLs. This client is a direct port of hyper 0.14's `Client`, which follows HTTP redirects by default (up to 10 hops). No redirect policy is configured on the `poster` client. If an attacker compromises a server at a configured notify URL, they can respond with a redirect to a local address (e.g., the CKB RPC at `http://localhost:8114/`), causing the node to issue an unintended HTTP request to a local service.

### Finding Description
In `BlockAssembler::new()`, the outbound HTTP client is constructed as:

```rust
Client::builder(hyper_util::rt::TokioExecutor::new())
    .build::<_, Full<bytes::Bytes>>(HttpConnector::new()),
``` [1](#0-0) 

No redirect policy (e.g., `redirect::Policy::none()`) is set. The `hyper_util::client::legacy` module is an explicit port of hyper 0.14's `Client`, which followed redirects by default. This `poster` client is then used in `BlockAssembler::notify()` to POST the current block template JSON to every URL in `config.notify`:

```rust
for url in &self.config.notify {
    if let Ok(req) = Request::builder()
        .method(Method::POST)
        .uri(url.as_ref())
        ...
    {
        let client = Arc::clone(&self.poster);
        tokio::spawn(async move {
            let _resp = timeout(notify_timeout, client.request(req)).await ...;
        });
    }
}
``` [2](#0-1) 

The `notify` field is a `Vec<Url>` in `BlockAssemblerConfig`, populated from the node's TOML config: [3](#0-2) 

### Impact Explanation
If an attacker compromises the server at a configured notify URL and responds with an HTTP 307 redirect (which preserves the POST method) to `http://localhost:8114/<rpc_method>`, the `poster` client will follow the redirect and issue a POST to the CKB node's own unauthenticated RPC. The CKB RPC has no authentication by default and exposes sensitive methods such as `clear_tx_pool` and `remove_transaction`. This allows the attacker to manipulate the local tx-pool or trigger other RPC side-effects without any credentials.

### Likelihood Explanation
Likelihood is **low-to-medium**. The notify feature must be explicitly configured by the operator, and the attacker must compromise the third-party server at the configured notify URL. However, operators who use external mining-pool notification endpoints or webhook aggregators are exposed. The CKB RPC is unauthenticated by default and binds to `127.0.0.1:8114`, making it reachable from the same host. The `HttpConnector` used by `poster` supports plain HTTP, so `http://localhost:8114/` is a valid redirect target.

### Recommendation
Construct the `poster` client with an explicit no-redirect policy:

```rust
Client::builder(hyper_util::rt::TokioExecutor::new())
    .redirect(hyper_util::client::legacy::redirect::Policy::none())
    .build::<_, Full<bytes::Bytes>>(HttpConnector::new()),
```

Additionally, validate that each URL in `config.notify` does not resolve to a loopback or link-local address before issuing the request.

### Proof of Concept
1. Operator configures `ckb.toml`:
   ```toml
   [block_assembler]
   notify = ["http://attacker-controlled.example.com/notify"]
   ```
2. Attacker controls `attacker-controlled.example.com` and serves:
   ```
   HTTP/1.1 307 Temporary Redirect
   Location: http://localhost:8114/
   Content-Length: 0
   ```
3. `BlockAssembler::notify()` fires on every new block template update.
4. The `poster` client follows the 307 redirect and POSTs the block-template JSON body to `http://localhost:8114/`.
5. The CKB RPC parses the body as a JSON-RPC call. With a crafted redirect target such as `http://localhost:8114/` and a body shaped as `{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}`, the attacker clears the node's mempool. [4](#0-3) [5](#0-4)
