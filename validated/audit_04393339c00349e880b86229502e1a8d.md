Audit Report

## Title
`script_exists_in_output` Re-Checks `row_lock` Instead of `row_type`, Silently Skipping the Type-Script Existence Check on PostgreSQL During Block Rollback — (File: util/rich-indexer/src/indexer/remove.rs)

## Summary
In `script_exists_in_output` (lines 204–257 of `util/rich-indexer/src/indexer/remove.rs`), the final `match` block at line 252 reads `row_lock.try_get::<bool, _>(0)` a second time instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, this causes the type-script query result to be fetched but never evaluated, returning the stale lock-script boolean for both checks. During `rollback_block`, scripts referenced exclusively as `type_script_id` in surviving outputs are incorrectly reported as absent and permanently deleted from the `script` table.

## Finding Description
`script_exists_in_output` executes two `SELECT EXISTS` queries — one against `lock_script_id` (line 208), one against `type_script_id` (line 237). The first `match` on `row_lock` (line 223) correctly early-returns `Ok(true)` when the lock query finds a row. After fetching `row_type` (lines 237–249), the second `match` at line 252 mistakenly re-reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, the column type is `BOOLEAN`, so `try_get::<bool, _>` succeeds and the `Ok(r)` branch returns the stale lock-script boolean — `row_type` is fetched but its value is never read. On SQLite, the column type is `BIGINT`, `try_get::<bool, _>` fails, the `Err(_)` branch correctly uses `row_type.get::<i64, _>(0)`, so SQLite is unaffected. The caller in `rollback_block` (lines 28–39) pushes any script ID for which `script_exists_in_output` returns `false` into `script_id_list_to_remove`, then deletes all of them from the `script` table via `remove_batch_by_blobs`.

## Impact Explanation
This is a concrete corruption of the CKB rich indexer's state storage mechanism, matching the **Medium (2001–10000 points)** bounty class: *Suboptimal implementation of CKB state storage mechanism*. After any rollback involving outputs with type scripts on a PostgreSQL-backed node, the `script` table loses rows still actively referenced by surviving outputs. Subsequent RPC calls (`get_cells`, `get_transactions`, `get_cells_capacity`) filtering by those type scripts return empty or incorrect results. The corruption is permanent until a full re-index is performed.

## Likelihood Explanation
The trigger is a chain reorganization — a routine and expected event in CKB requiring no attacker capability. Any valid competing chain with more accumulated work causes the node to call `indexer.rollback()` (line 170 of `util/indexer-sync/src/lib.rs`). Type-scripted outputs (DAO deposits, UDTs, NFTs) are extremely common, so virtually every reorg on a PostgreSQL-backed rich indexer node triggers the bug.

## Recommendation
Replace `row_lock` with `row_type` in the final `match` block of `script_exists_in_output` at line 252:

```rust
// Before (buggy) — util/rich-indexer/src/indexer/remove.rs line 252
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}

// After (correct)
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Add a PostgreSQL-backed integration test for `rollback_block` asserting that a script used exclusively as a `type_script_id` in surviving outputs is not deleted after rollback.

## Proof of Concept
1. Start a rich indexer with a PostgreSQL backend.
2. Append block 1 containing an output with `lock_script_id = A`, `type_script_id = B` (script B appears nowhere as a lock script).
3. Append block 2 spending that output and creating a new output with `lock_script_id = C`, `type_script_id = B`.
4. Trigger rollback of block 2 (simulate a reorg).
5. **Expected:** `script_exists_in_output(B)` returns `true` (B is still referenced as `type_script_id` in the block-1 output); script B is not deleted.
6. **Actual (PostgreSQL):** The second `match` evaluates `row_lock.try_get::<bool, _>(0)` → `Ok(false)` (lock query found nothing for B); function returns `false`; script B is added to `script_id_list_to_remove` and deleted. Subsequent `get_cells` filtered by script B returns empty results despite live outputs referencing it.
