Audit Report

## Title
`check_tx_fee` Admits Below-Minimum-Weight-Fee-Rate Transactions via Size-Only Gate — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` gates mempool admission using only serialized transaction size as the fee-rate denominator. After `verify_rtx` returns the actual cycle count in `_process_tx`, the true weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`) is never checked against `min_fee_rate`. A transaction with small serialized size but near-maximum cycle consumption passes the size-based gate with a fee orders of magnitude below the true weight-based minimum, enabling sustained low-cost mempool flooding.

## Finding Description

`check_tx_fee` (`tx-pool/src/util.rs` lines 42–52) explicitly uses `tx_size` as the sole denominator, with a comment acknowledging this is intentionally a "cheap check":

```rust
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The canonical weight function `get_transaction_weight` (`util/types/src/core/tx_pool.rs` lines 298–303) computes `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)` and is used by every post-admission decision path.

In `_process_tx` (`tx-pool/src/process.rs` lines 715–753), the flow is:
1. `pre_check` → `check_tx_fee` (size-based gate, cycles unknown) — line 715
2. `verify_rtx` → returns actual `verified.cycles` — line 734
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry constructed with actual cycles — line 751
4. `submit_entry` — no weight-based fee-rate check performed — line 753

Inspection of `submit_entry` (`tx-pool/src/process.rs` lines 96–170) confirms it only performs RBF checks, time-relative verification on tip change, and size-limit eviction. No `min_fee_rate` check against the true weight occurs anywhere after step 2.

`TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs` lines 115–118) correctly uses `get_transaction_weight(self.size, self.cycles)` for all post-admission decisions (block template selection, eviction ordering via `EvictKey`), creating the gap: the admission gate uses size, but all downstream ranking uses weight.

The eviction key (`tx-pool/src/component/sort_key.rs` lines 80–103) orders by weight-based `fee_rate` ascending, so attacker transactions (with very low weight-based fee rates) are evicted first when the pool is full. However, since evicted transactions are never mined, their UTXOs remain unspent and the attacker can immediately re-submit the same transactions, maintaining perpetual pool pressure at zero marginal fee cost per cycle.

## Impact Explanation

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` and `max_tx_verify_cycles = 70_000_000`, a 200-byte transaction consuming 70 M cycles has:
- Size-based min fee (checked): `1000 × 200 / 1000 = 200 shannons` → passes
- Actual weight: `max(200, 11940) = 11940`
- Actual weight-based fee rate: `200 / 11940 × 1000 ≈ 16.7 shannons/KW` — ~60× below `min_fee_rate`

Such transactions are never selected for block templates. The attacker can continuously re-submit evicted transactions (UTXOs remain unspent since transactions are never mined), keeping the pool perpetually at capacity. Every legitimate transaction admission triggers an eviction cycle. The fee estimator (`util/fee-estimator/src/estimator/weight_units_flow.rs` lines 153–161) calls `accept_tx` on pool admission, computing weight-based fee rates; attacker transactions with ~16.7 shannons/KW pollute fee estimation, causing legitimate users to underpay and have transactions stuck.

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

Any unprivileged user with RPC or P2P relay access can trigger this. Writing a CKB-VM script that loops to consume ~70 M cycles is straightforward. Since evicted transactions are never mined, their UTXOs are never spent — the attacker re-submits the same transactions at zero marginal fee cost. The only capital requirement is locking CKB in UTXOs (recoverable), not spending it. With the default 180 MB pool and 200-byte transactions (~900,000 slots), a well-funded attacker can maintain full pool saturation. Even a partial attack (thousands of UTXOs) causes measurable pool churn and fee estimation skew. The 12-hour expiry window does not limit re-submission of the same unconfirmed transactions.

## Recommendation

After `verify_rtx` returns the actual cycle count and before `TxEntry::new` is constructed in `_process_tx`, add a second fee-rate check using the true weight:

```rust
// After: let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_fee_rate = FeeRate::calculate(fee, actual_weight);
if actual_fee_rate < tx_pool_config.min_fee_rate {
    return Some((
        Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate,
            tx_pool_config.min_fee_rate.fee(actual_weight).as_u64(),
            fee.as_u64())),
        snapshot,
    ));
}
```

The existing size-based check in `check_tx_fee` can remain as a cheap early-exit for obviously under-fee transactions, but must not be the sole admission gate.

## Proof of Concept

1. Deploy a CKB lock script that loops to consume exactly `max_tx_verify_cycles - 1` cycles (≈70 M cycles).
2. Create a UTXO locked by this script.
3. Build a transaction spending that UTXO. Serialized size ≈ 200 bytes.
4. Set output capacity so `fee = ceil(min_fee_rate × tx_size / 1000) = 200 shannons`.
5. Submit via `send_transaction` RPC.
6. Observe: transaction accepted into mempool (`check_tx_fee` passes: `fee(200) ≥ min_fee(200)`).
7. Observe: `entry.fee_rate()` ≈ 16.7 shannons/KW — far below `min_fee_rate = 1000`.
8. Observe: transaction never appears in block templates.
9. When evicted from pool (pool full or 12-hour expiry), re-submit the same transaction immediately (UTXO is still unspent). Repeat to maintain sustained pool pressure and fee estimation pollution.
