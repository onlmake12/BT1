Audit Report

## Title
Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` When `tx_age` Exceeds Circular Buffer Size — (File: `util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary
`TxConfirmStat` initializes `confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` to length `MAX_CONFIRM_BLOCKS = 1000`. In `remove_unconfirmed_tx`, a bounds guard protects the `block_unconfirmed_txs` ring-buffer access, but `confirm_blocks_to_failed_txs[tx_age - 1]` is indexed unconditionally. When a tracked transaction is rejected or evicted after more than 1000 blocks, `tx_age - 1 >= 1000` causes a Rust index-out-of-bounds panic, crashing the node.

## Finding Description
Both arrays are initialized to length 1000: [1](#0-0) 

In `remove_unconfirmed_tx` (L197–217), the guard at L208 routes `block_unconfirmed_txs` safely when `tx_age >= self.block_unconfirmed_txs.len()`, but L214–216 has no corresponding check: [2](#0-1) 

When `tx_age > 1000`, `tx_age - 1 >= 1000` is out of bounds for a `Vec` of length 1000. Rust panics unconditionally in both debug and release builds.

The call chain that sets `count_failure = true`:

- `reject_tx` (L475–477) calls `drop_tx` (L428–430): [3](#0-2) 
- `drop_tx` calls `drop_tx_inner(tx_hash, true)` (L416–425): [4](#0-3) 
- `remove_expired` in `pool.rs` calls `callbacks.call_reject`, which invokes `fee_estimator.reject_tx`: [5](#0-4) 

## Impact Explanation
When `ConfirmationFraction` is enabled and a tracked transaction remains unconfirmed for more than 1000 blocks before being rejected or evicted, the node panics and crashes. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**. The crash is deterministic (Rust index-out-of-bounds panic, not undefined behavior) and reproducible.

## Likelihood Explanation
`ConfirmationFraction` is opt-in via `fee_estimator.algorithm = "ConfirmationFraction"` in `ckb.toml`. Once enabled, the trigger requires no special privileges: submit a low-fee transaction that miners ignore, wait ~2.8 hours (1001 blocks at 10 s/block), then wait for the default 12-hour expiry timer (`remove_expired`) or fill the pool to trigger `limit_size` eviction. Any unprivileged user with RPC access to `send_transaction` can trigger this condition.

## Recommendation
Add a bounds guard before indexing `confirm_blocks_to_failed_txs`, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This ensures failure counts are only recorded for transactions whose age falls within the tracked window, consistent with the design intent of `MAX_CONFIRM_BLOCKS`.

## Proof of Concept
1. Configure the node: `fee_estimator.algorithm = "ConfirmationFraction"` in `ckb.toml`.
2. Submit a transaction with a fee rate just above `min_fee_rate` but low enough that miners do not include it. The tx is accepted and tracked at height `H` with `best_height = H`.
3. Allow 1001+ blocks to be produced. The tx remains unconfirmed; `tx_age` grows beyond 1000.
4. Either wait for the 12-hour expiry timer (`remove_expired` → `callbacks.call_reject` → `fee_estimator.reject_tx`) or fill the pool to trigger `limit_size` eviction.
5. `remove_unconfirmed_tx` is called with `tx_age > 1000` and `count_failure = true`.
6. `self.confirm_blocks_to_failed_txs[tx_age - 1]` panics: **thread panicked at 'index out of bounds: the len is 1000 but the index is N'**, crashing the node.

Unit test to confirm: construct a `TxConfirmStat` with `max_confirm_blocks = 5`, call `add_unconfirmed_tx` at height 0, then call `remove_unconfirmed_tx(entry_height=0, tip_height=6, bucket_index, count_failure=true)` — this produces `tx_age = 6`, which panics at `confirm_blocks_to_failed_txs[5]` on a `Vec` of length 5.
