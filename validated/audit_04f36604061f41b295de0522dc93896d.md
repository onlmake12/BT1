### Title
Provider Fee Parameters (`setProviderFee`, `setDefaultGasLimit`) Can Be Changed Immediately During Pending Requests, Causing User Transactions to Fail or Overpay — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract allows any registered provider (or their designated fee manager) to change `feeInWei` and `defaultGasLimit` at any time with immediate, global effect. This is a direct analog to the auction vulnerability: just as auction parameters could be changed mid-auction causing bidder transactions to fail, Entropy provider parameters can be changed between a user's fee query and their request submission, causing the request to revert with `InsufficientFee`. Additionally, because excess `msg.value` is never refunded, a fee decrease between query and submission causes users to silently overpay, with the surplus permanently credited to the Pyth protocol treasury.

---

### Finding Description

`setProviderFee` and `setProviderFeeAsFeeManager` write the new fee directly to `provider.feeInWei` with no delay or pending-request guard: [1](#0-0) 

`setDefaultGasLimit` similarly writes `provider.defaultGasLimit` immediately: [2](#0-1) 

Both parameters feed directly into `getProviderFee`, which is called inside `requestHelper` to validate `msg.value` at request time: [3](#0-2) 

The fee formula scales with `defaultGasLimit`: decreasing `defaultGasLimit` while keeping `feeInWei` constant raises the per-unit-gas price, increasing the required fee for any request whose `gasLimit > new defaultGasLimit`: [4](#0-3) 

Excess `msg.value` beyond `providerFee` is credited to `accruedPythFeesInWei` and is never returned to the caller: [5](#0-4) 

The interface documentation acknowledges fees can change but places the entire burden on callers to re-query before every invocation: [6](#0-5) 

---

### Impact Explanation

**Transaction failure (DoS of individual requests):** A provider raises `feeInWei` or lowers `defaultGasLimit` between a user's `getFeeV2()` call and their `requestV2{value: queriedFee}()` submission. The request reverts with `InsufficientFee`. The user loses gas and must retry.

**Silent overpayment (loss of funds):** A provider lowers `feeInWei` between a user's fee query and submission. The user sends the old (higher) fee; the excess is permanently credited to `accruedPythFeesInWei` (Pyth treasury), not refunded. The user suffers a direct, unrecoverable loss of native tokens.

**Griefing loop:** A malicious provider can repeatedly toggle fees up/down to consistently invalidate in-flight user transactions, effectively denying service to users of that provider without any on-chain cost beyond gas.

---

### Likelihood Explanation

The Entropy provider role is explicitly listed as a valid attacker role in the Immunefi scope. Any registered provider can call `setProviderFee` permissionlessly. The fee manager role (set via `setFeeManager`) can also trigger this. The Fortuna keeper already calls `set_provider_fee_as_fee_manager` automatically in response to gas price changes: [7](#0-6) 

This means fee changes happen in production regularly, making the race condition between `getFeeV2()` and `requestV2()` a realistic occurrence even without malicious intent. A malicious provider can exploit this deliberately via front-running.

---

### Recommendation

1. **Snapshot the fee at request time and store it in the `Request` struct.** This ensures the fee a user paid is the fee that was in effect when they submitted, and any future fee changes do not retroactively affect pending requests.
2. **Refund excess `msg.value`** beyond the actual required fee back to `msg.sender` in `requestHelper`, rather than silently crediting it to the Pyth treasury.
3. **Add a minimum delay (timelock) for fee increases**, so users have time to react before a higher fee takes effect. Fee decreases can remain immediate.

---

### Proof of Concept

```
1. Provider registers with feeInWei = 100 wei, defaultGasLimit = 100_000.
2. User calls getFeeV2(provider, 0) → returns 100 + pythFee = 101 wei.
3. Provider calls setProviderFee(10_000) (fee raised 100x).
4. User submits requestV2{value: 101}() → reverts InsufficientFee.
   User loses gas; request never created.

--- Overpayment variant ---
1. Provider registers with feeInWei = 10_000 wei.
2. User calls getFeeV2(provider, 0) → returns 10_001 wei.
3. Provider calls setProviderFee(1) (fee dropped to 1 wei).
4. User submits requestV2{value: 10_001}() → succeeds.
   providerFee = 1 wei; accruedPythFeesInWei += 10_000 wei.
   User overpays 9_999 wei with no refund path.
``` [3](#0-2) [1](#0-0) [8](#0-7)

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();

            provider,
            oldFeeInWei,

            bytes("")

    function setDefaultGasLimit(uint32 gasLimit) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        // Check that we can round the gas limit into the 10k gas. This reverts
        // if the provided value exceeds the max.
        roundTo10kGas(gasLimit);

        uint32 oldGasLimit = provider.defaultGasLimit;
        provider.defaultGasLimit = gasLimit;
        emit ProviderDefaultGasLimitUpdated(msg.sender, oldGasLimit, gasLimit);
        emit EntropyEventsV2.ProviderDefaultGasLimitUpdated(
            msg.sender,
            oldGasLimit,
            gasLimit,
            bytes("")
        );
    }
