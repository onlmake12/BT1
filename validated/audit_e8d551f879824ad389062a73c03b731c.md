Let me check the governor crate's `retain_recent` behavior and the rate limiter key types more carefully.

### Title
Unbounded `forward_rate_limiter` HashMap Growth via Unique `(PeerId, PeerId)` Pairs in HolePunching Protocol — (`network/src/protocols/hole_punching/mod.rs`)

### Summary

The `forward_rate_limiter` in `HolePunching` uses a `HashMapStateStore<(PeerId, PeerId, u32)>` that grows without bound during a long-lived session. `retain_recent()` is called only in `disconnected()`, never in `notify()`. An attacker holding a persistent session can send `ConnectionRequest` messages with unique `(from, to)` PeerId pairs at up to 30/second, inserting a new HashMap entry per unique key, causing unbounded memory growth.

---

### Finding Description

`HolePunching` declares two rate limiters:

```rust
rate_limiter: RateLimiter<(PeerIndex, u32)>,
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,
``` [1](#0-0) 

Both are backed by `governor::state::keyed::HashMapStateStore<T>`, which accumulates one entry per unique key and must be periodically pruned via `retain_recent()`.

`retain_recent()` is called in exactly one place:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    ...
}
``` [2](#0-1) 

The `notify()` handler (fired every 5 minutes via `CHECK_INTERVAL`) cleans up `pending_delivered` and `inflight_requests` but **never calls `retain_recent()`** on either rate limiter:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    ...
    self.pending_delivered.retain(|_, (_, t)| (now - *t) < TIMEOUT);
    self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);
    // forward_rate_limiter.retain_recent() is ABSENT
}
``` [3](#0-2) 

In `received()`, a per-session rate limiter check gates entry at 30 req/sec per `(session_id, msg_type)`: [4](#0-3) 

But this only limits the **rate**, not the number of unique keys. Each `ConnectionRequest` with a distinct `(from, to)` pair then calls:

```rust
self.protocol.forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
``` [5](#0-4) 

inserting a new `HashMapStateStore` entry per unique `(from, to, item_id)` triple. The same pattern exists in `ConnectionRequestDelivered` and `ConnectionSync` processing. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

At 30 unique `(from, to)` pairs per second, each entry consuming roughly ~200 bytes (two `PeerId`s of ~39 bytes each + `u32` + HashMap overhead + governor state), the HashMap grows at ~6 KB/sec. Over a 24-hour session this is ~500 MB per attacker session. Multiple concurrent attacker sessions multiply this linearly. Sustained growth causes process-level OOM, which terminates the node and its embedded RPC API.

---

### Likelihood Explanation

The attacker only needs: (1) a long-lived P2P session with HolePunching enabled, and (2) the ability to craft `ConnectionRequest` messages with arbitrary `from`/`to` PeerId bytes. Both are standard P2P capabilities requiring no privilege. `PeerId` bytes are validated for format but not for any registry membership, so arbitrary valid PeerId values are accepted. [8](#0-7) 

---

### Recommendation

Call `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside `notify()`, alongside the existing `pending_delivered` and `inflight_requests` cleanup. This ensures the HashMap is pruned every `CHECK_INTERVAL` (5 minutes) regardless of whether any peer disconnects. [3](#0-2) 

---

### Proof of Concept

```rust
// Pseudocode unit test
let mut hp = HolePunching::new(network_state);
for i in 0..100_000u64 {
    let from = PeerId::random();
    let to   = PeerId::random();
    let msg  = build_connection_request(from, to);
    hp.received(session_ctx, msg).await;
}
// Assert: forward_rate_limiter internal map size == 100_000
// (retain_recent() was never called; no disconnect occurred)
// Memory consumed: ~20 MB and growing with no upper bound
```
