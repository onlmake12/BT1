### Title
Missing Per-Update Replay Protection in `PythLazer.verifyUpdate()` Allows Stale Price Injection - (File: lazer/contracts/evm/src/PythLazer.sol)

---

### Summary

`PythLazer.verifyUpdate()` performs signature and signer-expiry checks but maintains **no record of previously processed updates**. Any valid signed Lazer payload can be replayed an unlimited number of times by any unprivileged caller. Consumer contracts that rely solely on `verifyUpdate()` for security — without independently validating the payload timestamp — will silently accept arbitrarily old price data.

---

### Finding Description

`PythLazer.verifyUpdate()` is the sole on-chain verification gate for EVM Lazer price updates. Its logic is:

1. Collect fee.
2. Check EVM format magic bytes.
3. Recover the ECDSA signer from the signature over `keccak256(payload)`.
4. Assert the recovered signer is in the trusted-signer list and not expired.
5. Return `(payload, signer)`. [1](#0-0) 

There is no mapping of used payload hashes, no nonce field, and no timestamp bound enforced inside the contract. The payload does carry a `uint64` timestamp (parseable via `PythLazerLib.parsePayloadHeader`), but `PythLazer.sol` never reads or validates it. [2](#0-1) 

The analog to the reported bug is direct: the reported system tracked nonces in a single global set instead of per-address, so one address's nonce collided with another's. Here, Pyth tracks **nothing at all** — the entire replay-protection responsibility is silently delegated to each consumer contract, with no documentation warning in the contract itself.

---

### Impact Explanation

Any DeFi protocol that:
- calls `verifyUpdate()` to obtain a verified `payload`, and
- does not independently enforce `timestamp > lastAcceptedTimestamp`

can be fed an arbitrarily old Lazer price by any unprivileged attacker who observed a prior valid update on-chain. Consequences include:

- **Stale liquidation prices**: an attacker replays a price from a period of high volatility to trigger or prevent liquidations.
- **Stale oracle reads for swaps/lending**: incorrect collateral valuations.
- **Fee drain**: each replay call costs only `verification_fee` (currently 1 wei), making mass replay economically trivial. [3](#0-2) 

---

### Likelihood Explanation

- All historical Lazer updates are visible on-chain (emitted in calldata).
- The attacker needs no special role — only the ability to send a transaction.
- The fee is 1 wei, making replay essentially free.
- Consumer contracts that follow the documented integration pattern (call `verifyUpdate`, parse payload, use price) without adding their own staleness guard are directly exploitable. [4](#0-3) 

---

### Recommendation

Add replay protection inside `verifyUpdate()` itself so that the contract is safe by default regardless of consumer implementation. Two complementary approaches:

1. **Hash-based deduplication**: maintain `mapping(bytes32 => bool) usedPayloadHashes` and revert if `usedPayloadHashes[keccak256(update[0:update.length])]` is already set.
2. **Timestamp bound**: parse the `uint64 timestamp` from the payload header and revert if `timestamp < block.timestamp - MAX_AGE`, where `MAX_AGE` is a governance-settable parameter.

Approach 2 is storage-efficient and mirrors how Pyth's classic price-feed contracts handle staleness. Approach 1 gives absolute replay prevention at the cost of one storage slot per update.

### Proof of Concept

```solidity
// Attacker contract
contract LazerReplayer {
    PythLazer lazer;
    IConsumer consumer;

    constructor(address _lazer, address _consumer) {
        lazer = PythLazer(_lazer);
        consumer = IConsumer(_consumer);
    }

    // Step 1: capture `oldUpdate` from a past tx (calldata is public)
    // Step 2: call this function to replay it
    function replay(bytes calldata oldUpdate) external payable {
        uint256 fee = lazer.verification_fee();
        // verifyUpdate succeeds — no replay check exists
        (bytes calldata payload, ) = lazer.verifyUpdate{value: fee}(oldUpdate);
        // consumer accepts the stale payload because verifyUpdate returned success
        consumer.updatePrice(payload);
    }
}
```

`verifyUpdate` will succeed on `oldUpdate` because:
- The magic bytes are still valid.
- The signature is still valid.
- The trusted signer has not expired.
- **No used-update record exists to reject it.** [1](#0-0)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
