Audit Report

## Title
Unbounded Response Allocation via Duplicate Indexes and Missing Byte-Size Cap in `GetBlockTransactionsProcess::execute` ظ¤ (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` lacks both index deduplication and a `MAX_RELAY_TXS_BYTES_PER_BATCH` byte-size guard that every analogous handler enforces. An unprivileged peer can send a `GetBlockTransactions` message with 32767 identical indexes pointing at the same transaction, forcing the node to clone and serialize that transaction 32767 times into a single unbounded `BlockTransactions` response, enabling a memory exhaustion crash.

## Finding Description

**Defect 1 ظ¤ Off-by-one in count guard (line 37)**

The guard uses strict `>`:
```rust
if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
```
`MAX_RELAY_TXS_NUM_PER_BATCH = 32767`, so a message with exactly 32767 entries passes (`32767 > 32767 == false`). Note: `GetTransactionsProcess` uses the same `>` operator, so this is consistent behavior rather than a unique defect here, but it sets the upper bound for the attack payload.

**Defect 2 ظ¤ No deduplication of indexes (lines 61ظô71)**

The collection loop iterates the peer-supplied index list with no deduplication:
```rust
let transactions = self
    .message
    .indexes()
    .iter()
    .filter_map(|i| block.transactions().get(Into::<u32>::into(i) as usize).cloned())
    .collect::<Vec<_>>();
```
Sending `indexes = [0u32; 32767]` causes 32767 `.cloned()` copies of `block.transactions()[0]` (the coinbase) to be collected into a single `Vec`. By contrast, `GetTransactionsProcess` explicitly rejects duplicates:
```rust
if message_len != tx_hashes_set.len() {
    return StatusCode::RequestDuplicate.with_context("Request duplicate transaction");
}
```
And `GetBlockProposalProcess` deduplicates via `HashSet` before processing, also rejecting if `proposals.len() != message_len`.

**Defect 3 ظ¤ No `MAX_RELAY_TXS_BYTES_PER_BATCH` cap (lines 80ظô97)**

The entire `transactions` vec is packed into one message and sent with no byte-size check:
```rust
let content = packed::BlockTransactions::new_builder()
    .block_hash(block_hash)
    .transactions(transactions.into_iter().map(|tx| tx.data()).collect::<Vec<_>>())
    ...
    .build();
let message = packed::RelayMessage::new_builder().set(content).build();
return async_send_message_to(&self.nc, self.peer, &message).await;
```
Both `GetTransactionsProcess` (lines 87ظô101) and `GetBlockProposalProcess` (lines 79ظô95) split responses when `relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MiB). `GetBlockTransactionsProcess` imports only `MAX_RELAY_TXS_NUM_PER_BATCH` and never imports or applies `MAX_RELAY_TXS_BYTES_PER_BATCH`.

**Combined exploit path:**
1. Attacker obtains any stored block hash (trivially from headers or a block explorer).
2. Attacker sends `GetBlockTransactions { block_hash, indexes: [0u32; 32767], uncle_indexes: [] }`.
3. The count guard passes (32767 ظëج 32767).
4. No deduplication: 32767 clones of the coinbase are collected.
5. No byte-size cap: all 32767 copies are serialized into one message.
6. Peak allocation = `32767 ├ù coinbase_tx_size` before any send occurs.

## Impact Explanation

**High ظ¤ Vulnerabilities which could easily crash a CKB node.**

With a typical mainnet coinbase (~500 bytes): `32767 ├ù 500 B ظëê 16 MB` per request. The rate limiter allows 30 requests/second per peer per message type, yielding ~480 MB/s from a single peer. With multiple concurrent peers, this drives the node to OOM. In the worst case (a block whose coinbase approaches `max_block_bytes` ظëê 597 KB ظ¤ valid since a block containing only a coinbase is consensus-legal): `32767 ├ù 597 KB ظëê 19.5 GB` allocated in a single async task, causing an immediate OOM crash. The rate limiter does not bound per-request allocation; it only limits request frequency.

## Likelihood Explanation

The attack requires only a TCP connection to the relay protocol ظ¤ no proof-of-work, no key, no privileged role. The attacker needs a stored block hash, obtainable from any block explorer or by syncing headers. The message is structurally valid and passes all existing checks. The attack is repeatable at 30 req/s per peer and can be parallelized across multiple peers. The 16 MB/request baseline (typical coinbase) is sufficient to exhaust memory over time even without a pathologically large coinbase.

## Recommendation

1. **Deduplicate indexes before lookup**, matching the pattern in `GetTransactionsProcess` and `GetBlockProposalProcess`:
   ```rust
   let indexes: HashSet<u32> = self.message.indexes().iter().map(Into::into).collect();
   if indexes.len() != self.message.indexes().len() {
       return StatusCode::RequestDuplicate.with_context("Request duplicate index");
   }
   ```
2. **Add a `MAX_RELAY_TXS_BYTES_PER_BATCH` accumulator** and truncate or split the response, matching the pattern in `GetTransactionsProcess::execute` and `GetBlockProposalProcess::execute`.
3. **Optionally change the count guard to `>=`** for strict inclusive enforcement, though this is a minor point given the above fixes.

## Proof of Concept

```rust
// Unit test pseudocode
let stored_block = /* any block stored by the node, e.g. genesis */;
let block_hash = stored_block.hash();
let coinbase_size = stored_block.transactions()[0].data().total_size();

// Craft malicious message: 32767 copies of index 0
let indexes = vec![0u32; 32767];
let msg = GetBlockTransactions { block_hash, indexes, uncle_indexes: vec![] };

// execute() will:
//   1. Pass count guard (32767 > 32767 == false)
//   2. Clone coinbase 32767 times (no dedup)
//   3. Build single response of size 32767 * coinbase_size (no byte cap)
// Expected: response serialized size == 32767 * coinbase_size
// With coinbase_size = 500 bytes: ~16 MB allocated per call
// With coinbase_size = 597 KB (max consensus): ~19.5 GB ظْ OOM crash
assert_eq!(response_size, 32767 * coinbase_size);
```

Fuzz test: generate random `indexes` arrays with high repetition rates targeting index 0, verify that `execute()` never allocates more than `MAX_RELAY_TXS_BYTES_PER_BATCH` bytes in a single response.
