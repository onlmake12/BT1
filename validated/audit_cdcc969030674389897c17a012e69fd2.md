### Title
Unbounded `updateData` Array in `Scheduler.updatePriceFeeds` Allows Any Keeper to Drain a Subscription's Balance via Inflated Gas Reimbursement ŌĆö (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `updatePriceFeeds` function in `Scheduler.sol` accepts an unbounded `bytes[] calldata updateData` array from any permissionless caller. The function measures actual gas consumed and reimburses the caller from the subscription's balance. Because there is no cap on `updateData.length`, a malicious keeper can supply a large array of valid Wormhole accumulator updates (freely obtained from the public Pyth price service), artificially inflating gas consumption and extracting the subscription's balance far beyond what a legitimate update would cost.

---

### Finding Description

`updatePriceFeeds` records `startGas = gasleft()` at entry, then passes the full attacker-supplied `updateData` array to the Pyth contract for parsing. After all processing completes, `_processFeesAndPayKeeper` computes:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
```

and transfers `gasCost + keeperSpecificFee` to `msg.sender` from `status.balanceInWei`. [1](#0-0) [2](#0-1) 

There is no bound on `updateData.length` anywhere in `updatePriceFeeds` or in `_validateSubscriptionParams` (which only validates `priceIds` and `readerWhitelist`). [3](#0-2) 

The Pyth contract's `parsePriceFeedUpdatesWithConfig` iterates over every element of `updateData`, parsing each Wormhole Merkle proof and every price message it contains. A single accumulator blob can embed up to 255 price messages; the Pyth fee is charged per message. An attacker who supplies N large but valid accumulator blobs forces the Pyth contract to do O(N ├Ś messages_per_blob) work, all of which is measured by `startGas - gasleft()` and reimbursed from the subscription balance. [4](#0-3) [5](#0-4) 

The `GAS_OVERHEAD` constant (30 000 gas) is added on top of actual consumption, so the attacker is always reimbursed slightly more than they spend, making repeated calls profitable. [6](#0-5) 

---

### Impact Explanation

A malicious keeper can repeatedly call `updatePriceFeeds` on any active subscription, each time supplying a maximally large `updateData` array. Each successful call:

1. Deducts the inflated Pyth fee from `status.balanceInWei`.
2. Deducts the inflated gas reimbursement from `status.balanceInWei`.
3. Transfers the full reimbursement to the attacker.

The subscription balance is drained far faster than the subscription manager anticipated. Once `balanceInWei` falls below `getMinimumBalance`, legitimate keepers can no longer be paid and the subscription effectively stops receiving price updates. Any on-chain protocol that reads prices from this subscription will stall on stale data. [7](#0-6) 

---

### Likelihood Explanation

- `updatePriceFeeds` is permissionless ŌĆö no registration or stake is required.
- Valid Wormhole Merkle accumulator blobs are freely available from the public Pyth Hermes price service endpoint; the attacker does not need any privileged key.
- The attacker profits on every call (gas reimbursement > gas spent, due to `GAS_OVERHEAD`), so there is a direct economic incentive.
- The only prerequisite is that the target subscription has a non-zero balance, which is guaranteed for any active subscription (the contract enforces a minimum balance on creation). [8](#0-7) [9](#0-8) 

---

### Recommendation

1. **Cap `updateData.length`**: Add a check at the top of `updatePriceFeeds` that reverts if `updateData.length` exceeds a small constant (e.g., `MAX_UPDATE_DATA_LENGTH = 5`). A legitimate keeper only needs one accumulator blob per update cycle.

```solidity
if (updateData.length > MAX_UPDATE_DATA_LENGTH) {
    revert SchedulerErrors.TooManyUpdateDataElements();
}
```

2. **Cap gas reimbursement**: Alternatively or additionally, cap the reimbursable gas at a protocol-defined maximum (e.g., `MAX_REIMBURSABLE_GAS`) so that even if a large array is supplied, the subscription balance cannot be drained beyond a predictable bound.

3. **Validate `updateData` element count against `params.priceIds.length`**: Since each subscription has a fixed, bounded set of price IDs (max 255, enforced by `MAX_PRICE_IDS_PER_SUBSCRIPTION`), a single well-formed accumulator blob is sufficient. Reject calls where `updateData.length > params.priceIds.length`. [10](#0-9)

        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
