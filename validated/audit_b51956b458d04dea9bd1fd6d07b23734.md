### Title
Lazer Price Update Signatures Lack Chain-Specific Binding, Enabling Cross-Chain Replay — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` computes the verification hash as `keccak256(payload)` where `payload` contains only price data (timestamp, channel, feed prices). No `chainId`, contract address, or EIP-712 domain separator is included in the signed message. Since `PythLazer` is deployed at the same address on 20+ EVM chains with the same trusted signer keys, a valid price update captured from any one chain passes signature verification on every other chain.

---

### Finding Description

In `verifyUpdate()`, the signature is verified over the raw payload hash:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` structure (magic, timestamp, channel, feed data) contains no chain-specific context. There is no `chainId`, no contract address, and no EIP-712 domain separator bound into the signed digest. [2](#0-1) 

The contract is deployed at the identical address `0xACeA761c27A909d4D3895128EBe6370FDE2dF481` on Ethereum, Base, Arbitrum, Polygon, and 15+ other chains, all sharing the same trusted signer set: [3](#0-2) 

`verifyUpdate()` performs no timestamp staleness check itself — that responsibility is delegated entirely to consumer contracts: [4](#0-3) 

The payload magic constant `PAYLOAD_MAGIC = 2479346549` is shared across all chains and all Lazer contract variants (EVM, Sui, Aptos), providing no chain discrimination: [5](#0-4) 

---

### Impact Explanation

An unprivileged attacker who captures a valid Lazer price update at time T (price = P) can submit that exact update to any other EVM chain at time T+Δ. `verifyUpdate()` will accept it — the signature is valid, the trusted signer is valid, and no chain binding is checked. Consumer contracts that do not independently enforce tight staleness windows will consume the replayed price P instead of the current price. In a chain-fork scenario, all historical Lazer updates from the original chain remain permanently valid on the fork, since the trusted signer keys and contract address are identical.

---

### Likelihood Explanation

The attack requires no privileged access. Lazer price updates are publicly streamed via the Lazer WebSocket. The attacker only needs to:
1. Subscribe to the Lazer feed and capture a valid signed update.
2. Wait for the price to move.
3. Submit the captured update to a target chain.

The `PythLazer` contract is live on 20+ chains simultaneously with the same signer set, making cross-chain submission trivially easy. Consumer contracts that rely solely on `verifyUpdate()` for freshness guarantees are directly exposed.

---

### Recommendation

Include `block.chainid` and `address(this)` in the signed digest, or adopt EIP-712 domain separation:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    payload
));
```

This ensures a signature produced for chain A is cryptographically invalid on chain B, and a signature produced for one contract deployment cannot be replayed against a redeployed contract at a different address.

---

### Proof of Concept

1. Subscribe to the Lazer WebSocket and capture a valid EVM-format update `U` at time `T` (price = `P_old`).
2. Wait for the price to move to `P_new` (e.g., a 2% move within the staleness window).
3. Call `PythLazer.verifyUpdate{value: fee}(U)` on a different EVM chain (e.g., submit an Ethereum-captured update to Base).
4. `verifyUpdate()` recovers the signer from `keccak256(payload)`, finds it in `trustedSignerToExpiresAtMapping`, and returns `(payload, signer)` — no revert.
5. Any consumer contract that calls `verifyUpdate()` and does not independently enforce a tight timestamp window now holds `P_old` as the current price, enabling price-manipulation exploits (e.g., borrowing against an inflated collateral price or liquidating at a stale price). [2](#0-1)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
