### Title
Keeper Compensation in `_processFeesAndPayKeeper` Omits L1 Data Fee on L2 Chains - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler._processFeesAndPayKeeper` compensates keepers using only `(gasUsed + GAS_OVERHEAD) * tx.gasprice`, which captures only the L2 execution fee. On OP-stack chains (Base, Optimism) and Arbitrum, every transaction also incurs an L1 data fee charged by the sequencer for posting calldata to L1. This fee is invisible to `tx.gasprice` and `gasleft()` arithmetic, so keepers suffer a systematic, unrecoverable loss on every `updatePriceFeeds` call.

---

### Finding Description

`_processFeesAndPayKeeper` in `Scheduler.sol` calculates the keeper reimbursement as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
``` [1](#0-0) 

`GAS_OVERHEAD` is a fixed constant of `30000` gas units: [2](#0-1) 

This formula is purely a function of L2 EVM gas consumption and `tx.gasprice`. On OP-stack chains (Base, Optimism) and Arbitrum, every transaction additionally incurs an **L1 data fee** — the cost of posting the transaction's calldata to Ethereum L1. This fee is:

- Charged directly from the sender's wallet by the sequencer at the protocol level.
- **Not** reflected in `tx.gasprice`, `gasleft()`, or any EVM opcode accessible inside the contract.
- Proportional to the compressed calldata size of the transaction, not to EVM gas consumed.

For `updatePriceFeeds`, the calldata includes Pyth price update blobs (`bytes[] calldata updateData`), which can be several kilobytes. On Base/Optimism, the L1 data fee for such a transaction can easily exceed the L2 execution fee by a factor of 2–10x depending on L1 gas prices and blob/calldata pricing.

The contract has no mechanism to include or estimate this cost. The `singleUpdateKeeperFeeInWei` flat per-feed bonus is a governance-set constant that does not dynamically track L1 data costs: [3](#0-2) 

---

### Impact Explanation

Every keeper that calls `updatePriceFeeds` on a Scheduler deployment on an L2 chain (Base, Optimism, Arbitrum) pays the L1 data fee out of pocket but is only reimbursed for the L2 execution portion. The shortfall is permanent — there is no recovery path. Over time, rational keepers will stop submitting updates, degrading liveness of all subscriptions. This directly impacts the economic viability of the keeper role and the reliability of price feed delivery for all Scheduler subscribers.

---

### Likelihood Explanation

The Scheduler is explicitly designed for L2 deployment. Base and Optimism are primary target chains for Pyth. L1 data fees on these chains are non-negligible and well-documented. Any keeper operating on these chains will observe the loss immediately on their first transaction. The vulnerability is triggered by the normal, intended operation of calling `updatePriceFeeds` — no special conditions are required.

---

### Recommendation

Incorporate the L1 data fee into keeper compensation. On OP-stack chains, the L1 data fee for the current transaction is available via the `OVM_GasPriceOracle` precompile at `0x420000000000000000000000000000000000000F`, specifically the `getL1Fee(bytes)` function. On Arbitrum, the `ArbGasInfo` precompile exposes per-calldata-byte L1 pricing. The contract should:

1. Query the appropriate precompile for the L1 data fee of the current transaction.
2. Add this amount to `gasCost` before paying the keeper.
3. Alternatively, increase `singleUpdateKeeperFeeInWei` to a governance-managed value that accounts for typical L1 data costs, with a mechanism to update it as L1 gas prices change.

---

### Proof of Concept

1. Deploy `Scheduler` on Base (OP-stack L2).
2. Create a subscription with several price feeds and fund it.
3. Record keeper wallet ETH balance before calling `updatePriceFeeds`.
4. Call `updatePriceFeeds` with a realistic `updateData` payload (several KB of Pyth VAA data).
5. Observe: the keeper receives `(gasUsed + 30000) * tx.gasprice` wei from the contract.
6. Observe: the keeper's actual ETH balance decreased by more than that amount — the difference is the L1 data fee charged by the Base sequencer.
7. The keeper's net balance after the call is lower than before, confirming a loss on every transaction. [4](#0-3)

```text
    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
```
