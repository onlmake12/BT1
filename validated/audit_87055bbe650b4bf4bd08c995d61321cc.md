### Title
Unbounded `while` Loop in `executeCallback` Enables Out-of-Gas DoS on Callback Execution — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that linearly scans fulfilled sequence numbers to advance `_state.firstUnfulfilledSeq`. An unprivileged attacker can create many requests and have them fulfilled out of order, forcing the loop to iterate over an arbitrarily large range of sequence numbers when the oldest unfulfilled request is finally executed. This causes the `executeCallback` transaction to run out of gas, permanently blocking callback delivery for that request.

---

### Finding Description

Inside `executeCallback`, after clearing the current request, the following loop runs unconditionally before the user callback is invoked:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
// After successful callback, update firstUnfulfilledSeq if needed
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest(_state.firstUnfulfilledSeq)`, which performs at least one cold `SLOAD` (2,100 gas on EIP-2929 chains). The loop has no iteration cap.

The `firstUnfulfilledSeq` pointer only advances past a gap of fulfilled requests when the *oldest* still-active request is finally fulfilled. Until that moment, every `executeCallback` call for a newer sequence number leaves `firstUnfulfilledSeq` unchanged. When the oldest request is eventually fulfilled, the loop must scan the entire gap in a single transaction.

`executeCallback` has no access control — any address may call it:

```solidity
function executeCallback(
    address providerToCredit,
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override {
``` [2](#0-1) 

The developers themselves flagged this in a TODO comment, confirming awareness of the gas concern but leaving it unmitigated.

---

### Impact Explanation

If the `while` loop consumes more gas than the block gas limit (~30 M gas on Ethereum mainnet), `executeCallback` for the oldest request can **never succeed**. Because the loop executes before the `try/catch` callback block and after `clearRequest`, a revert rolls back `clearRequest` as well, leaving the request permanently stuck. The user's pre-paid fee is locked in the contract and the callback is never delivered.

Even below the block gas limit, an attacker can force the keeper to supply an unpredictably large gas budget, making reliable operation economically infeasible.

---

### Likelihood Explanation

`executeCallback` is permissionless. Any address can submit valid Pyth price-feed update data (freely obtainable from Hermes) to fulfill any active request. The attacker's cost is proportional to the number of requests created (each requires paying `getFee()`). To saturate the 30 M gas block limit with cold SLOADs at 2,100 gas each, roughly **~14,000 requests** are needed. This is a realistic griefing budget for a motivated attacker targeting a high-value subscription. The attack is therefore medium likelihood.

---

### Recommendation

Replace the linear scan with a data structure that does not require iteration:

1. **Doubly-linked list** of active sequence numbers (the TODO comment already suggests this): O(1) removal when a request is fulfilled.
2. **Counter-only approach**: track only the count of outstanding requests; drop `firstUnfulfilledSeq` entirely if it is not required for external consumers.
3. **Hard iteration cap**: add `uint256 maxIter = 100; while (...) { if (--maxIter == 0) break; ... }` as a short-term mitigation, accepting that `firstUnfulfilledSeq` may lag.

---

### Proof of Concept

1. Attacker calls `requestPriceUpdatesWithCallback` N times (sequence numbers 1 … N), paying the required fee each time. `_state.firstUnfulfilledSeq = 1`, `_state.currentSequenceNumber = N+1`.

2. Attacker (or any third party) calls `executeCallback` for sequence numbers 2, 3, …, N, supplying valid Hermes update data. Each call checks: *is request 1 active?* → yes → loop stops immediately. `firstUnfulfilledSeq` remains 1.

3. When `executeCallback` is finally called for sequence number 1:
   - `clearRequest(1)` executes.
   - The `while` loop starts at `firstUnfulfilledSeq = 1`.
   - Requests 1 … N are all inactive → loop iterates **N times**, each doing a cold `SLOAD` via `findRequest`.
   - Gas consumed ≈ N × 2,100 gas.
   - For N ≥ ~14,000 on Ethereum, gas exceeds the block limit → transaction reverts → request 1 is never fulfilled and its fee is locked. [3](#0-2)
