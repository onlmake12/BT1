### Title
Predictable Time-Based Ping Nonce Enables Forged Pong Responses and Zombie Connection Maintenance — (File: `network/src/protocols/ping.rs`)

---

### Summary

The CKB P2P ping protocol derives its challenge nonce from elapsed wall-clock seconds since node startup rather than from a cryptographically random source. Because the nonce is fully deterministic and identical across all connected peers at any given moment, an unprivileged remote peer can predict or reconstruct the nonce and forge a valid pong response without genuinely processing the ping, allowing it to avoid disconnection and maintain a persistent zombie connection.

---

### Finding Description

The `nonce()` helper in `network/src/protocols/ping.rs` computes the challenge value sent to every peer as:

```rust
fn nonce(t: &Instant, start_time: Instant) -> u32 {
    t.saturating_duration_since(start_time).as_secs() as u32
}
``` [1](#0-0) 

This value is the number of whole seconds the node has been running — a 32-bit integer with at most second-level granularity and zero randomness. It is computed once per ping round and assigned identically to every peer session simultaneously:

```rust
let send_nonce = nonce(&now, self.start_time);
// ...
ps.nonce = send_nonce;   // same value for every peer
``` [2](#0-1) 

The pong handler then uses this nonce as the sole correctness check before marking a peer alive:

```rust
if let Some(status) = self.connected_session_ids.get_mut(&session.id)
    && (true, nonce) == (status.processing, status.nonce())
{
    status.processing = false;
    // ...
    return;
}
// if nonce is incorrect or can't find ping info → disconnect
``` [3](#0-2) 

The nonce is the **only** value checked to validate a pong. There is no MAC, no per-session secret, and no randomness.

---

### Impact Explanation

An attacker who connects to a CKB node can:

1. **Recover the node's uptime from the first ping it receives.** The nonce value directly encodes `elapsed_seconds_since_start`, so a single observed ping message reveals the node's uptime with one-second precision.
2. **Predict every future nonce.** Because the nonce is monotonically increasing by one per second, the attacker knows the exact nonce for every future ping round.
3. **Forge pong responses.** The attacker can send a `Pong(predicted_nonce)` immediately after `processing` is set to `true` (i.e., after a ping is dispatched), without having genuinely processed the ping. The node accepts this as a valid liveness signal.
4. **Exploit the shared nonce across all sessions.** Because all peers receive the same nonce at the same time, an attacker with multiple simultaneous connections can observe the nonce on one session and immediately forge pongs on all other sessions before the ping messages even arrive on those sessions.

The practical consequence is that a malicious peer can maintain persistent connections to a CKB node while being entirely unresponsive to real protocol traffic. This is a prerequisite step for eclipse-style attacks: an attacker fills the node's inbound connection slots with zombie peers that never time out, starving the node of honest peers.

---

### Likelihood Explanation

**Medium.** The attacker requires only a standard TCP connection to the target node — no credentials, no privileged role. The node's uptime is recoverable from the first ping message (or can be estimated from public network monitoring). The nonce space is a 32-bit counter with second granularity, making brute-force unnecessary. The only constraint is that `processing` must be `true` (i.e., a ping round must be in flight), which occurs on a fixed, configurable interval.

---

### Recommendation

Replace the time-derived nonce with a cryptographically random value generated per ping round using `rand::thread_rng()` (already used elsewhere in the codebase for key generation):

```rust
fn nonce() -> u32 {
    rand::thread_rng().gen()
}
```

Optionally, generate a distinct random nonce per session rather than a shared one, to eliminate the cross-session forgery vector entirely.

---

### Proof of Concept

1. Attacker opens two TCP connections (`S1`, `S2`) to a CKB node.
2. Node sends `Ping(nonce=N)` on both sessions (same `N` for both, since `send_nonce` is shared).
3. Attacker reads `N` from `S1`'s ping message.
4. Attacker immediately sends `Pong(N)` on `S2` — before the ping on `S2` is even processed by the attacker's own stack.
5. Node receives `Pong(N)` on `S2`, checks `processing==true && nonce==N` → passes, marks `S2` as alive, resets `processing=false`.
6. Attacker never sends a real pong on `S1`, causing `S1` to time out and disconnect — but `S2` remains alive indefinitely by repeating step 3–5 each ping interval.
7. Attacker scales this to fill the node's inbound connection slots with zombie sessions, each kept alive by forged pongs derived from the predictable nonce.
