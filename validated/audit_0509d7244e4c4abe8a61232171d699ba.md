Audit Report

## Title
Orphan Pool Permanently Saturated via Silent Drop of Exhausted Parent-Request Entries — (`sync/src/types/mod.rs`, `tx-pool/src/component/orphan.rs`)

## Summary
In `pop_ask_for_txs`, when `next_request_peer()` returns `None` (single announcing peer already asked once), the `unknown_tx_hashes` entry is silently discarded with no corresponding orphan eviction. Because `OrphanPool` has no per-peer slot limit and no recovery path once the parent-request entry is dropped, a single attacker peer can permanently saturate the 100-slot orphan pool for up to `100 * MAX_BLOCK_INTERVAL` seconds per wave, causing random eviction of legitimate orphan transactions.

## Finding Description
**Root cause — silent drop in `pop_ask_for_txs`:**
`pop_ask_for_txs` pops entries from `unknown_tx_hashes` and calls `next_request_peer()`. When `next_request_peer()` returns `Some(peer)`, the entry is re-pushed. When it returns `None`, the entry is simply not re-pushed and is permanently discarded (`sync/src/types/mod.rs` L1466–1474). `next_request_peer()` returns `None` exactly when `requested == true` and `peers.len() <= 1` (`sync/src/types/mod.rs` L1277–1284). With `RETRY_ASK_TX_TIMEOUT_INCREASE = 30s`, the entry is dropped approximately 30 seconds after the first request.

**Orphan admission path:**
When a relayed transaction fails with a missing-input error, it is added to `OrphanPool` and the announcing peer is queued in `unknown_tx_hashes` via `add_ask_for_txs` (`tx-pool/src/process.rs` L507–512, `sync/src/relayer/mod.rs` L676–685).

**Orphan expiry and eviction:**
`ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL` (~80 minutes). When the pool exceeds `DEFAULT_MAX_ORPHAN_TRANSACTIONS` (100), a random entry is evicted (`tx-pool/src/component/orphan.rs` L119–125). There is no mechanism to evict an orphan when its corresponding `unknown_tx_hashes` entry is dropped.

**No per-peer orphan slot accounting:**
`OrphanPool` tracks entries only by `ProposalShortId` with no per-peer accounting (`tx-pool/src/component/orphan.rs` L42–45), so a single peer can occupy all 100 slots.

**Why existing checks are insufficient:**
- `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` limits entries in `unknown_tx_hashes` per peer, but once those entries are dropped after the single retry, the orphan pool slots remain occupied with no recovery path.
- `limit_size()` only evicts expired entries or entries beyond the 100-slot cap — it does not evict orphans whose parent requests have been abandoned.
- There is no mechanism to re-queue a dropped `unknown_tx_hashes` entry when a new peer later announces the same parent hash, because the orphan is already in the pool and the parent hash is not re-added to `unknown_tx_hashes` from the orphan side.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**
A single attacker peer can saturate the 100-slot orphan pool of any reachable CKB node for up to ~80 minutes per wave. During this window, every legitimate orphan transaction submitted to that node is subject to random eviction, breaking dependent transaction chains for honest users. Applied to multiple nodes simultaneously, this degrades network-wide transaction relay with negligible cost to the attacker.

## Likelihood Explanation
- Any peer opening the `RelayV3` protocol can send `RelayTransactions` messages — no special role required.
- Constructing orphan transactions requires only referencing non-existent `OutPoint`s; the node cannot verify the lock script before the missing-input error fires, so no valid signatures are needed.
- Only 100 such transactions are needed to saturate the pool.
- The attacker's only required action is to ignore `GetRelayTransactions` responses.
- The attack is fully repeatable with no cooldown beyond the ~80-minute expiry window, which can itself be bypassed by continuously injecting new orphan transactions.

## Recommendation
1. **Evict orphan on request abandonment**: When a `unknown_tx_hashes` entry is permanently dropped (all peers exhausted, `next_request_peer()` returns `None`), proactively remove the corresponding orphan transaction(s) from `OrphanPool` rather than letting them occupy slots until expiry.
2. **Re-queue on exhausted peers**: In `pop_ask_for_txs`, when `next_request_peer()` returns `None`, re-insert the entry into `unknown_tx_hashes` with a significantly longer timeout (e.g., 5 minutes) rather than silently dropping it, so that a future peer announcement can still resolve it.
3. **Per-peer orphan slot accounting**: Limit the number of orphan pool slots attributable to a single peer to prevent one peer from monopolizing the 100-slot pool.

## Proof of Concept
```
1. Attacker connects to victim node, opens RelayV3 protocol.
2. Attacker generates 100 transactions each spending a distinct non-existent OutPoint
   (no valid signatures required — missing-input fires before lock script verification).
3. Attacker sends RelayTransactions for all 100 to the victim.
4. Victim: adds all 100 to OrphanPool (pool now 100/100);
   calls add_ask_for_txs(attacker_peer, parent_hashes) for each.
5. At t≈0s: pop_ask_for_txs fires; next_request_peer() returns attacker_peer
   (requested=false → sets requested=true); entries re-queued.
6. Attacker ignores all GetRelayTransactions messages.
7. At t≈30s: pop_ask_for_txs fires again; next_request_peer() returns None
   (requested=true, peers.len()==1); all 100 entries are silently dropped.
8. OrphanPool remains at 100/100 for ~80 minutes with no recovery path.
9. Any legitimate orphan tx submitted during this window triggers random eviction
   (limit_size() evicts a random entry), breaking dependent tx chains.
10. Attacker sustains the attack indefinitely by injecting new orphan txs
    as old ones expire, keeping the pool permanently saturated.
```
