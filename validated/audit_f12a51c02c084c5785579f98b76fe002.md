### Title
`ScriptHashTypeVerifier` Omits Type Script `hash_type` Validation, Allowing Transactions with Unsupported Type Script Hash Types to Bypass Non-Contextual Verification — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` consensus allowlist only against each output's **lock** script. The **type** script's `hash_type` field is never checked against that allowlist. Any unprivileged transaction sender can craft a transaction whose output carries a type script with a future/unsupported `hash_type` value (e.g., `Data3 = 6`), bypass the non-contextual gate entirely, and force the node to perform full contextual script verification before the transaction is finally rejected.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` defines the four currently permitted values:

```
0 (Data), 1 (Type), 2 (Data1), 4 (Data2)
``` [1](#0-0) 

`ScriptHashTypeVerifier::verify()` iterates over every output and validates only `output.lock().hash_type()`:

```rust
for output in self.transaction.outputs() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { … }
    } else { … }
}
``` [2](#0-1) 

`output.type_()` is never touched. The struct comment even documents this gap explicitly: *"Check whether output **lock** hash type within enabled range."* [3](#0-2) 

This verifier is the sole `hash_type` allowlist gate inside `NonContextualTransactionVerifier`, which is called for every incoming transaction in both the tx-pool path and the block-tx path: [4](#0-3) [5](#0-4) 

The lower-level `check_data` function does validate both lock and type scripts, but only for **structural** validity (even byte or `1`), not for the consensus allowlist: [6](#0-5) 

`ScriptHashType::verify_value` accepts any even byte or `1`, so `Data3 = 6`, `Data4 = 8`, … all pass `check_data` but are absent from `ENABLED_SCRIPT_HASH_TYPE`: [7](#0-6) 

The two-path inconsistency is therefore:

| Path | Lock `hash_type` | Type `hash_type` |
|---|---|---|
| `check_data` (P2P gate) | structural only | structural only |
| `ScriptHashTypeVerifier` (consensus gate) | allowlist checked | **not checked** |

---

### Impact Explanation

A transaction whose output has a type script with `hash_type = Data3` (byte value `6`) passes both `check_data` and `ScriptHashTypeVerifier`, enters the tx-pool, and is only rejected later inside `select_version()` during full contextual script execution: [8](#0-7) 

Consequences:
- **Tx-pool resource exhaustion**: an attacker can flood the pool with such transactions; each one occupies a pool slot and triggers expensive contextual verification (cycle counting, VM setup) before rejection.
- **Inconsistent enforcement**: the consensus rule "only `{0,1,2,4}` are permitted" is enforced for lock scripts at the cheap non-contextual stage but deferred to the expensive contextual stage for type scripts, creating an asymmetric cost for the node.

---

### Likelihood Explanation

The entry path is fully unprivileged. Any RPC caller (`send_transaction`) or P2P relay peer can submit a transaction with an output whose type script carries `hash_type = 6`. No special key, miner role, or network position is required. The malformed field is a single byte in the serialized transaction.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` when present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check …
        check_hash_type(output.lock().hash_type())?;

        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

This mirrors the symmetry already present in `check_data` and eliminates the inconsistency between the two validation paths.

---

### Proof of Concept

1. Build a transaction with one output whose type script has `hash_type = 6` (`Data3`):
   ```
   CellOutput {
     lock: Script { hash_type: 0 (Data), … },   // passes ScriptHashTypeVerifier
     type: Script { hash_type: 6 (Data3), … },  // NOT checked by ScriptHashTypeVerifier
   }
   ```
2. Submit via `send_transaction` RPC.
3. `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()` iterates outputs, checks only `lock.hash_type() = 0` → passes.
4. Transaction enters the tx-pool.
5. `ContextualTransactionVerifier` → `ScriptVerifier` → `select_version()` hits the catch-all arm for `Data3` → returns `Err(ScriptError::InvalidScriptHashType(...))` → transaction is rejected.
6. Repeat at high rate to exhaust tx-pool admission bandwidth. [9](#0-8) [10](#0-9)

// Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}
