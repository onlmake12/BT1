### Title
Legacy-Path Callback Revert Permanently Blocks Provider Keeper — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
In `Entropy.sol`'s `revealWithCallback`, when a provider has `defaultGasLimit == 0` (the legacy path), the requester's `_entropyCallback` is invoked with a bare, unchecked call. A malicious requester whose contract always reverts in `_entropyCallback` causes every `revealWithCallback` attempt for that sequence number to revert entirely, permanently blocking the provider's keeper from fulfilling the request and wasting keeper gas indefinitely.

### Finding Description
`revealWithCallback` branches on `req.gasLimit10k`:

- **New path (`gasLimit10k != 0`):** Uses `excessivelySafeCall`, catches reverts, emits `CallbackFailed`, and sets `CALLBACK_FAILED` status — the transaction succeeds regardless of callback behavior.
- **Legacy path (`gasLimit10k == 0`):** Calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly with no error catching. [1](#0-0) 

The legacy path is entered whenever `providerInfo.defaultGasLimit == 0` at request time: [2](#0-1) 

In the legacy branch, `clearRequest` is called before the callback (checks-effects-interactions), but if the callback reverts the entire transaction reverts, undoing `clearRequest`: [3](#0-2) 

The result: the request remains permanently active and every subsequent `revealWithCallback` attempt for that sequence number also reverts.

This is confirmed by the existing test `testRequestWithCallbackAndRevealWithCallbackFailing`, which asserts that `revealWithCallback` reverts when the callback reverts and the provider has `defaultGasLimit == 0`: [4](#0-3) 

### Impact Explanation
- The provider's automated keeper bot (`fortuna`) calls `revealWithCallback` for every pending request. For the stuck request it will loop indefinitely, burning gas on every attempt.
- The provider's `currentCommitmentSequenceNumber` cannot advance past the stuck sequence number without a manual `advanceProviderCommitment` call. If `maxNumHashes` is configured, requests whose sequence numbers exceed `currentCommitmentSequenceNumber + maxNumHashes` will revert with `LastRevealedTooOld`, blocking all future users of that provider.
- The attacker's paid fee is already credited to the provider at request time, so the attacker sacrifices only the request fee to impose unbounded keeper gas costs.

### Likelihood Explanation
Any provider that has not called `setDefaultGasLimit` retains `defaultGasLimit == 0`. Providers registered before the V2 gas-limit feature was introduced are in this state. An unprivileged user can permissionlessly call `requestWithCallback` against any such provider, deploying a trivial reverting contract as the requester. No special access is required.

### Recommendation
1. **Short-term:** In the legacy path, wrap the callback in a low-level `call` and check the return value rather than propagating the revert. Alternatively, gate `requestWithCallback` so it is only accepted from providers with `defaultGasLimit != 0`.
2. **Long-term:** Deprecate the legacy path entirely; require all providers to set a non-zero `defaultGasLimit` before accepting new callback requests.
3. **Provider guidance:** All active providers should call `setDefaultGasLimit` with a non-zero value to opt into the safe callback failure state flow.

### Proof of Concept
1. Deploy a contract `MaliciousRequester` that implements `IEntropyConsumer` and always reverts in `_entropyCallback`.
2. Identify a provider with `defaultGasLimit == 0` (any legacy provider).
3. Call `requestWithCallback(provider, userRandomNumber)` from `MaliciousRequester`, paying the required fee.
4. The provider's keeper calls `revealWithCallback(provider, sequenceNumber, ...)`.
5. `_entropyCallback` reverts → entire `revealWithCallback` transaction reverts → request stays active.
6. Repeat step 4 indefinitely: every attempt reverts, wasting keeper gas. The provider must manually call `advanceProviderCommitment` to skip the stuck sequence number. [5](#0-4)
