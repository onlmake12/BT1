Audit Report

## Title
Weak Single-Child Epoch Check in `need_clean` Enables Orphan Pool Memory Exhaustion — (File: `chain/src/utils/orphan_block_pool.rs`)

## Summary
`InnerPool::need_clean` samples only the first child returned by a non-deterministic `HashMap` iterator to decide whether to evict an entire orphan subtree. An attacker can insert two sibling orphan blocks under the same parent — one with a recent epoch, one with an expired epoch — so that the recent-epoch child is iterated first, permanently blocking eviction of the expired child. Because `OrphanBlockPool::insert` has no size cap and epoch numbers are not validated before orphan pool insertion, this allows unbounded memory and disk growth that can crash a CKB node.

## Finding Description

**Root cause — `need_clean` samples one arbitrary child:**

`InnerPool::need_clean` at lines 113–122 of `chain/src/utils/orphan_block_pool.rs`:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← arbitrary first child
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
```

`self.blocks` maps each parent hash to a `HashMap<packed::Byte32, LonelyBlockHash>` of its children. `map.iter().next()` returns an **arbitrary** child because Rust `HashMap` iteration order is randomized per process (SipHash with a random seed). The eviction decision for the entire subtree — including all siblings — is based on whichever child happens to be iterated first.

`clean_expired_blocks` calls `need_clean` per leader and removes the whole subtree only if it returns `true`. If two siblings exist under parent `P` — one with `epoch_number = 19` (recent) and one with `epoch_number = 9` (expired), `tip_epoch = 20` — and the recent child is sampled: `19 + 6 = 25 < 20` → false → neither block is evicted. The expired block is permanently retained.

**No size cap on insertion:**

`OrphanBlockPool::insert` has no size guard. The comment at line 125 explicitly forbids `LruCache` to prevent implicit pruning, meaning there is no fallback eviction path.

**Epoch numbers are not validated before orphan pool insertion:**

`asynchronous_process_block` in `chain_service.rs` calls `non_contextual_verify` before routing to the orphan broker. `BlockVerifier::verify` (the non-contextual verifier) only checks proposals limit, block bytes, cellbase structure, duplicates, and merkle root — **no PoW check, no epoch number check**. PoW and epoch validation live in `HeaderVerifier`, which requires the parent header and is therefore not reachable for orphan blocks whose parent is unknown.

**Blocks are also written to disk before orphan pool insertion:**

`asynchronous_process_block` calls `self.insert_block(&lonely_block)` (which commits the block to the DB via `db_txn.insert_block`) before `self.orphan_broker.process_lonely_block(lonely_block.into())`. Blocks that are never cleaned from the orphan pool are also never deleted from the DB, causing unbounded disk growth in addition to memory exhaustion.

**Exploit flow:**

1. Attacker connects to a CKB node via P2P.
2. Attacker crafts block **A**: `parent_hash = P` (unknown to node), `epoch_number = tip_epoch − 1` (recent). Passes `BlockVerifier` (no epoch check).
3. Attacker crafts block **B**: `parent_hash = P`, `epoch_number = tip_epoch − 10` (expired). Passes `BlockVerifier`.
4. Both are inserted under `blocks[P]` in the orphan pool and committed to the DB.
5. When `clean_expired_blocks(tip_epoch)` runs (every 60 seconds), `need_clean(P, tip_epoch)` samples one child. If A is sampled: `(tip_epoch − 1) + 6 < tip_epoch` → false → neither A nor B is evicted.
6. Attacker repeats with new parent hashes P1, P2, … Pn. Pool and DB grow by 2 entries per round with no upper bound.

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Repeated insertion of sibling orphan blocks with mixed epoch numbers causes the orphan pool to grow without bound in both memory and disk. Because the pool is protected by an `RwLock` and consulted on every block arrival and every 60-second cleanup tick, unbounded growth leads to memory exhaustion and eventual node crash or severe unresponsiveness. Disk exhaustion from the DB writes is an additional crash vector. The attack targets a single node, placing it in the High severity band.

## Likelihood Explanation

Any peer on the CKB P2P network can relay block messages. Non-contextual verification does not check PoW or epoch numbers, so crafting the required blocks requires no mining capability — only the ability to construct well-formed block structures (valid cellbase, correct merkle root, valid byte size). The attack is cheap, stateless, and indefinitely repeatable. The cleanup timer fires only once per minute, giving the attacker ample time to grow the pool between cleanup cycles. With many parent hashes, statistically ~50% of pairs will evade eviction per cleanup cycle, and the attacker can continuously insert new pairs to maintain unbounded growth.

## Recommendation

1. **Fix the weak condition** — replace the single-child sample with a check over all children; retain the subtree only if at least one child is still within the expiry window:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH <= tip_epoch)
        })
        .unwrap_or_default()
}
```

2. **Fix the off-by-one** — change `<` to `<=` so blocks at exactly `tip_epoch − EXPIRED_EPOCH` are cleaned as documented.

3. **Enforce a pool size cap** — add a maximum entry count to `OrphanBlockPool::insert` and evict the oldest/lowest-epoch entries when the cap is reached, independent of the epoch-based cleanup path.

## Proof of Concept

```
Constants: EXPIRED_EPOCH = 6, tip_epoch = 20, cleanup_interval = 60s

Setup:
  Craft Block A: parent_hash = P, epoch_number = 19 (recent)
  Craft Block B: parent_hash = P, epoch_number = 9  (expired)
  Both pass BlockVerifier (no PoW/epoch check in non-contextual path)

Attack loop (repeat with P1, P2, … Pn):
  Send A → inserted as blocks[P][hash_A], committed to DB
  Send B → inserted as blocks[P][hash_B], committed to DB

  clean_expired_blocks(20):
    need_clean(P, 20):
      map.iter().next() → (hash_A, A)   [non-deterministic; A likely first ~50% of runs]
      19 + 6 = 25 < 20 → false
      → subtree not evicted, DB entries not deleted

Result: 2 blocks permanently retained per parent P in both memory and disk.
        After n rounds: 2n blocks in pool and DB, growing without bound.
        Node OOM or disk exhaustion crash after sufficient iterations.
```

Minimal unit test: construct an `InnerPool` directly, insert two `LonelyBlockHash` entries with the same parent and epoch numbers 19 and 9 respectively, call `clean_expired_blocks(20)`, and assert that `parents.len() == 2` (neither was evicted) — demonstrating the bug without any network setup.
