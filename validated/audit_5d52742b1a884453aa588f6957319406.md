### Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` from Multiple Peers — (`sync/src/types/mod.rs`)

---

### Summary

`push_peer()` appends to `UnknownTxHashPriority.peers` without deduplication or a per-entry cap. The overflow guard in `add_ask_for_txs()` only checks the number of **distinct hash keys** in `unknown_tx_hashes`, not the total number of peer references stored across all entries' `peers` Vecs. Up to 128 peers can each announce the same 32 767 hashes, causing every entry's `peers` Vec to grow to 128 elements with no mitigation.

---

### Finding Description

**`push_peer` — no deduplication, no cap** [1](#0-0) 

Every call unconditionally appends `peer_index`; the same peer can be added multiple times, and there is no maximum length enforced.

**`add_ask_for_txs` — Occupied path calls `push_peer` freely** [2](#0-1) 

When a hash is already present (Occupied), the incoming peer is pushed into the existing entry's `peers` Vec with no guard.

**Overflow check measures key count, not peer-reference count** [3](#0-2) 

The condition `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50 000) or `>= peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` counts **distinct hash keys**, not the sum of all `peers` Vec lengths. With 32 767 identical hashes announced by 128 peers, the key count stays at 32 767 — well below 50 000 — so the guard never fires and the per-peer counter check at line 1524 is never reached.

**Constants that bound the worst case** [4](#0-3) [5](#0-4) 

`MAX_RELAY_TXS_NUM_PER_BATCH` = 32 767, `MAX_RELAY_PEERS` = 128, `MAX_UNKNOWN_TX_HASHES_SIZE` = 50 000.

**Entry point — per-message hash count is checked, but per-peer accumulation is not** [6](#0-5) 

The only guard here rejects a single message that exceeds 32 767 hashes; it does not prevent the same hashes from being re-announced by many peers.

---

### Impact Explanation

Worst-case allocation from the `peers` Vecs alone:

```
128 peers × 32 767 hashes × 8 bytes (PeerIndex/SessionId) ≈ 33.5 MB
```

Beyond raw memory, `pop_ask_for_txs()` iterates every entry's `peers` Vec on each 100 ms timer tick, and the overflow-check loop at line 1517 iterates all peer references when the key-count threshold is eventually crossed. Both become O(keys × peers) = O(4.2 M) operations, degrading relay throughput.

---

### Likelihood Explanation

The attack requires only 128 inbound TCP connections (the default `MAX_RELAY_PEERS`) and one `RelayTransactionHashes` message per peer containing 32 767 fabricated-but-syntactically-valid hashes. No PoW, no keys, no privileged access. The rate limiter (30 req/s per peer per message type) does not prevent a single message per peer.

---

### Recommendation

1. **Deduplicate in `push_peer`**: check `self.peers.contains(&peer_index)` before pushing, or use a `HashSet`.
2. **Cap the `peers` Vec**: enforce a maximum length (e.g., `MAX_RELAY_PEERS`) inside `push_peer`.
3. **Fix the overflow guard**: track total peer-reference count (sum of all `peers` Vec lengths) in addition to the key count, and enforce a global ceiling.
