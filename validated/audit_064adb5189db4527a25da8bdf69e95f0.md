### Title
`PythLazer.sol` `verifyUpdate` Does Not Validate Payload Timestamp Against `block.timestamp`, Enabling Stale Price Submission After L2 Sequencer Downtime — (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

The `verifyUpdate` function in `PythLazer.sol` only validates the ECDSA signature and signer expiry. It does **not** compare the `uint64 timestamp` embedded in the Lazer payload against `block.timestamp`. On L2 chains where the sequencer can go offline, an attacker can submit a Lazer price update signed before the downtime after the sequencer resumes. The contract accepts the stale update as fully valid, and any consuming protocol receives an outdated price that may diverge significantly from the current market price.

---

### Finding Description

`PythLazer.sol` exposes a single on-chain verification entry point, `verifyUpdate`, which performs the following checks:

1. Fee payment
2. EVM format magic (`uint32 EVM_FORMAT_MAGIC = 706910618`)
3. ECDSA signature recovery
4. Signer validity via `isValidSigner`, which checks `block.timestamp < trustedSignerToExpiresAtMapping[signer]`

```solidity
// lazer/contracts/evm/src/PythLazer.sol
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    ...
    payload = update[71:71 + payload_len];
    bytes32 hash = keccak256(payload);
    (signer, , ) = ECDSA.tryRecover(hash, ...);
    if (signer == address(0)) { revert("invalid signature"); }
    if (!isValidSigner(signer)) { revert("invalid signer"); }
    // ← NO timestamp check against block.timestamp
}
```

The Lazer payload format, as confirmed by `PythLazerLib.parsePayloadHeader` and `PythLazerStructs.Update`, contains a `uint64 timestamp` field at bytes 4–12 of the payload:

```solidity
// lazer/contracts/evm/src/PythLazerLib.sol  (lines 110–136)
function parsePayloadHeader(bytes memory update)
    public pure returns (uint64 timestamp, ...) {
    ...
    timestamp = _readBytes8(update, pos);  // bytes 4–12
    ...
}
```

```solidity
// lazer/contracts/evm/src/PythLazerStructs.sol  (lines 74–78)
struct Update {
    uint64 timestamp;
    Channel channel;
    Feed[] feeds;
}
```

`verifyUpdate` never reads or validates this timestamp. It returns the raw `payload` bytes to the caller, leaving freshness enforcement entirely to the consuming contract — with no documentation or on-chain guard requiring it.

**L2 sequencer downtime attack path:**

1. PythLazer is deployed on an L2 (Arbitrum, Optimism, Base, etc.).
2. The L2 sequencer goes offline. `block.timestamp` on L2 continues to advance (derived from L1), but no user transactions are processed.
3. During downtime, the market price of an asset moves significantly (e.g., ETH drops 10%).
4. Lazer continues generating signed price updates off-chain at the pre-downtime price.
5. When the sequencer resumes, an attacker submits a Lazer update signed at the pre-downtime price.
6. `verifyUpdate` accepts it: the signer has not expired, and the signature is valid.
7. The consuming protocol (e.g., a perp DEX or lending protocol) receives the stale price and executes trades or valuations at the pre-downtime level.

---

### Impact Explanation

Consuming protocols on L2s that rely on `verifyUpdate` as their sole price-freshness gate will accept stale Lazer prices after sequencer downtime. Concrete consequences include:

- **Lending protocols**: collateral valued at a stale (inflated) price, enabling undercollateralized borrowing or blocking valid liquidations.
- **Derivative/perp protocols**: positions opened or closed at a price that diverges from the true market price, causing direct financial loss to the protocol or its counterparties.
- **Any protocol using Lazer for executable prices**: the stale price can be exploited by any user who holds the old signed update, with no special privilege required.

The severity is proportional to the magnitude of price movement during the downtime period and the notional value of positions affected.

---

### Likelihood Explanation

- L2 sequencer downtime is a documented, recurring real-world event (Arbitrum and Optimism have both experienced multi-hour outages).
- Lazer is explicitly designed for high-frequency, latency-sensitive trading on L2s — the exact environment where sequencer downtime risk is highest.
- The attack requires no privileged access: any user who retained a signed Lazer update from before the downtime can submit it.
- The attacker has a clear economic incentive when prices have moved during the downtime window.
- No existing on-chain guard in `verifyUpdate` prevents this.

---

### Recommendation

Add a `maxAge` parameter (or a hardcoded staleness bound) to `verifyUpdate` that parses the payload timestamp and rejects updates older than the threshold:

```solidity
uint32 constant PAYLOAD_FORMAT_MAGIC = 2479346549;
uint64 constant MAX_UPDATE_AGE_SECONDS = 60; // configurable

function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    // ... existing checks ...

    payload = update[71:71 + payload_len];

    // Validate payload timestamp freshness
    uint32 magic = uint32(bytes4(payload[0:4]));
    require(magic == PAYLOAD_FORMAT_MAGIC, "invalid payload magic");
    uint64 payloadTimestamp = uint64(bytes8(payload[4:12]));
    require(
        block.timestamp <= payloadTimestamp + MAX_UPDATE_AGE_SECONDS,
        "Lazer update too old"
    );

    // ... signature and signer checks ...
}
```

Additionally, for L2 deployments, consider integrating a Chainlink L2 sequencer uptime feed to invalidate any update whose timestamp falls within a known sequencer downtime window.
