Audit Report

## Title
Concurrent `HeaderMapKernel::get()` Race on Sled Backend Causes Node Crash via `unwrap_or_else(panic)` — (`shared/src/types/header_map/kernel_lru.rs`)

## Summary
`HeaderMapKernel::get()` performs a non-atomic check-remove-insert sequence across the sled backend with no outer lock protecting the full operation. Two concurrent sync threads processing the same `HEADER_VALID` header can both miss the memory map, race on `backend.remove()`, and one thread receives `None`. This `None` propagates through `get_header_index_view()` to an `unwrap_or_else(|| panic!(...))` in `HeaderAcceptor::accept()`, crashing the node process.

## Finding Description
**Root cause — `HeaderMapKernel::get()` (`shared/src/types/header_map/kernel_lru.rs` L109–142):**

The function performs three distinct, unlocked steps:
1. `self.memory.get_refresh(hash)` — acquires and releases the `RwLock` write lock; returns `None` if the entry is only in sled.
2. `self.backend.remove(hash)` — sled's key-level atomic remove; no outer lock is held.
3. `self.memory.insert(view.clone())` — acquires and releases the `RwLock` write lock.

Two threads (Thread A and Thread B) can both complete step 1 with `None` before either reaches step 2. Sled's `remove` is atomic at the key level, so exactly one thread wins `Some(view)` and the other gets `None`. The losing thread returns `None` from `get()`.

**Propagation — `get_header_index_view()` (`sync/src/types/mod.rs` L1158–1166):**

With `store_first = false` (i.e., `BLOCK_STORED` is not set), the function calls `header_map().get(hash)` first. If that returns `None` and the block is not yet in the chain store, the entire function returns `None`.

**Panic — `HeaderAcceptor::accept()` (`sync/src/synchronizer/headers_process.rs` L304–316):**

Under the `HEADER_VALID` branch, the result of `get_header_index_view()` is immediately unwrapped with no recovery path.

**Why `HEADER_VALID` is set while the entry is in sled:**

`get_block_status()` returns `HEADER_VALID` when `header_map().contains_key()` is true. `contains_key()` checks both the memory map and the sled backend, so a header evicted to sled still reports `HEADER_VALID`.

**Complete exploit sequence:**
1. Attacker connects two peers (A and B) to the target node during IBD.
2. Peer A sends a valid header H; Thread 0 processes it, calls `insert_valid_header()`, inserts H into the memory map.
3. The background `limit_memory()` task (fires every 5 seconds) evicts H to the sled backend and removes it from memory.
4. Peers A and B simultaneously send header H again.
5. Thread A and Thread B both enter `HeaderAcceptor::accept()`, both call `get_block_status(H)` → `HEADER_VALID` (H is in sled, `contains_key()` returns `true`).
6. Both call `get_header_index_view(H, false)` → `header_map().get(H)` → `HeaderMapKernel::get(H)`.
7. Both call `memory.get_refresh(H)` → `None` (H is only in sled).
8. Both call `backend.remove(H)`: Thread A wins `Some(view)`, Thread B gets `None`.
9. Thread B returns `None` from `get()`, `get_header_index_view()` returns `None`, `unwrap_or_else(|| panic!(...))` fires.
10. Node process terminates.

## Impact Explanation
The node process terminates via `panic!`. This matches the **High** impact class: *"Vulnerabilities which could easily crash a CKB node."* Any external peer can trigger this by connecting two sockets and sending the same syntactically valid header from both simultaneously while the node is in IBD with headers evicted to sled — a normal operational condition requiring no special privileges.

## Likelihood Explanation
During IBD the memory limit is routinely exceeded, causing the background task to push headers to sled every 5 seconds. The race window spans a sled disk I/O operation (potentially milliseconds to tens of milliseconds under load), making it non-trivially wide. An attacker with two peer connections can attempt the race repeatedly at negligible cost. Reliable exploitation requires timing precision but is achievable under IBD load when sled I/O latency widens the window.

## Recommendation
Replace the destructive `backend.remove()` promotion in `HeaderMapKernel::get()` with a non-destructive `backend.get()` followed by `memory.insert()`, and perform deferred eviction from the backend separately. This eliminates the possibility of a concurrent caller receiving `None` for a key that exists. Alternatively, protect the entire check-remove-insert sequence with a per-key or global mutex so that only one thread can promote a given key at a time.
