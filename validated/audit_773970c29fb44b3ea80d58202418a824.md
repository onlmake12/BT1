### Title
Subscription Manager Can Drain Third-Party Funds via Permissionless `addFunds` + Unilateral Deactivation — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.addFunds()` is intentionally permissionless — any address can deposit ETH into any subscription. However, `withdrawFunds()` is gated exclusively to the subscription manager via `onlyManager`. When the manager deactivates the subscription, the minimum-balance floor is lifted and the manager can withdraw the entire balance, including ETH contributed by third parties who had no withdrawal rights. This is a direct structural analog of the Rigor M-18 finding: a single party with limited trust can drain pooled funds that originated from others, bypassing the service-delivery escrow.

---

### Finding Description

`addFunds` carries no caller restriction:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    ...
    status.balanceInWei += msg.value;
    ...
}
``` [1](#0-0) 

The protocol test `testAnyoneCanAddFunds` explicitly validates this design intent — a completely different address successfully funds a subscription it does not own.

`withdrawFunds` is restricted to the manager and enforces a minimum balance **only while the subscription is active**:

```solidity
function withdrawFunds(...) external override onlyManager(subscriptionId) {
    ...
    if (params.isActive) {
        // minimum balance check
    }
    status.balanceInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    ...
}
``` [2](#0-1) 

The manager can deactivate the subscription unilaterally via `updateSubscription` with `newParams.isActive = false`: [3](#0-2) 

After deactivation, `params.isActive` is `false`, so the minimum-balance guard in `withdrawFunds` is skipped entirely, and the manager can withdraw `status.balanceInWei` in full.

The `onlyManager` modifier confirms only the original creator controls withdrawal: [4](#0-3) 

---

### Impact Explanation

Any ETH deposited by third parties via `addFunds` is permanently at the mercy of the subscription manager. The manager can:

1. Accept third-party deposits while the subscription is active.
2. Deactivate the subscription (single unilateral transaction).
3. Withdraw the entire balance — including all third-party ETH — to their own address.

Third-party funders have no recourse: there is no per-funder accounting, no refund path, and no multi-party consent requirement. The subscription-based service-delivery escrow is completely bypassed. The impact is direct, permanent loss of user ETH.

---

### Likelihood Explanation

- `addFunds` is explicitly designed to be permissionless (confirmed by the `testAnyoneCanAddFunds` test).
- The attack requires only two sequential transactions from the manager: `updateSubscription(isActive=false)` then `withdrawFunds(fullBalance)`.
- No leaked keys, no governance majority, no oracle manipulation — only the manager's own wallet is needed.
- The attack is invisible to third-party funders until after the funds are gone.

---

### Recommendation

Two complementary mitigations:

**A) Track per-funder contributions.** Record how much each address deposited and restrict `withdrawFunds` to only allow the manager to withdraw their own share. Third-party funders should be able to reclaim their own deposits independently.

**B) Restrict `addFunds` to the manager only.** If the intended use case is self-funding, remove the permissionless property. If community funding is desired, implement the per-funder accounting above.

Either option eliminates the asymmetry where anyone can deposit but only the manager can withdraw everything.

---

### Proof of Concept

```solidity
// 1. Manager creates subscription with minimum balance
uint256 subId = scheduler.createSubscription{value: minimumBalance}(params);

// 2. Third party adds funds (permissionless — no access control)
vm.prank(thirdParty);
scheduler.addFunds{value: 10 ether}(subId);
// thirdParty has no way to withdraw; only manager can

// 3. Manager deactivates subscription (single unilateral tx)
params.isActive = false;
scheduler.updateSubscription(subId, params);
// subscription is now inactive; minimum-balance floor is lifted

// 4. Manager drains entire balance including third-party ETH
uint256 totalBalance = scheduler.getSubscription(subId).status.balanceInWei;
scheduler.withdrawFunds(subId, totalBalance);
// manager receives minimumBalance + 10 ether; thirdParty loses 10 ether
```

The root cause is the combination of:
- `addFunds` at [1](#0-0)  (no access control)
- `withdrawFunds` at [2](#0-1)  (manager-only, no floor when inactive)
- `updateSubscription` deactivation path at [3](#0-2)  (unilateral, no consent from funders)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {

                uint8(params.priceIds.length)
            );

                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
