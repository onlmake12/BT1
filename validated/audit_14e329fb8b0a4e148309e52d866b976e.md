Audit Report

## Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` Below Guard Threshold â€” (`sync/src/types/mod.rs`)

## Summary
`push_peer()` appends to the `peers` Vec with no length cap. The size guard in `add_ask_for_txs` checks only `unknown_tx_hashes.len()` (count of distinct hash keys), which never increases when the same hashes are re-announced. A single peer sending K < 32767 fabricated hashes repeatedly causes unbounded `peers` Vec growth, bypassing the guard entirely and eventually crashing the node with OOM.

## Finding Description

`push_peer()` appends unconditionally with no bound check: [1](#0-0) 

In `add_ask_for_txs`, the `Occupied` branch calls `push_peer` without inspecting the current `peers` Vec length: [2](#0-1) 

The size guard fires only when `unknown_tx_hashes.len()` (distinct keys) exceeds the threshold â€” it never checks the length of any individual `peers` Vec: [3](#0-2) 

With one connected peer, the threshold is `1 Ã— 32767 = 32767`. Sending K = 32766 hashes fills `unknown_tx_hashes` to 32766 distinct keys. On every subsequent re-announcement, all 32766 entries hit the `Occupied` branch, `push_peer` is called 32766 times, but `unknown_tx_hashes.len()` stays at 32766 â€” always below 32767 â€” so the guard never fires and the function returns `Status::ok()`.

The `tx_filter` does not block this path. Hashes enter `tx_filter` only via `mark_as_known_txs`, which is called only after a transaction is resolved: [4](#0-3) 

Hashes sitting in `unknown_tx_hashes` are invisible to `tx_filter`, so the filter check in `execute()` passes on every re-announcement: [5](#0-4) 

The existing integration test only tests the K = 32767 boundary (exactly at threshold), which triggers `TooManyUnknownTransactions` and a ban on the first call â€” it does not cover the repeated re-announcement attack with K < threshold: [6](#0-5) 

## Impact Explanation

Memory grows as `K Ã— M Ã— sizeof(PeerIndex)` where K = 32766 and M = number of repeated messages. At M = 10,000: `32766 Ã— 10,000 Ã— 8 bytes â‰ˆ 2.6 GB`. The node runs out of memory and crashes or becomes unresponsive. This matches the allowed impact: **"Vulnerabilities which could easily crash a CKB node" (High, 10001â€“15000 points)**.

## Likelihood Explanation

The attack requires only a standard P2P connection. No PoW, no stake, no cryptographic material, and no coordination with other peers is needed. The hashes do not need to correspond to real transactions. The attacker simply sends the same `RelayTransactionHashes` message with 32766 fabricated hashes in a tight loop. The existing guard is trivially bypassed by staying one hash below the per-peer limit. `MAX_RELAY_TXS_NUM_PER_BATCH` = `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = 32767: [7](#0-6) 

## Recommendation

Cap the `peers` Vec length inside `push_peer` or before calling it in the `Occupied` branch of `add_ask_for_txs`. A reasonable bound is the total number of connected peers (`self.peers.state.len()`). Additionally, deduplicate peer entries before appending â€” skip the push if the peer is already present in the Vec â€” to prevent a single peer from inflating the Vec at all. The `tx_filter` should also mark hashes as seen when they first enter `unknown_tx_hashes` to prevent the same peer from re-announcing the same hash repeatedly.
