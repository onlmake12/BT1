### Title
Static Echo Pyth-Fee Estimate Diverges from Dynamic Execution-Time Pyth Fee, Locking User Funds — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.sol` stores a static `_state.pythFeeInWei` at request time but pays the live `pyth.getUpdateFee(updateData)` at execution time. When the two values diverge — because Pyth governance raised `singleUpdateFeeInWei` or because the executor submits a blob with more price-feed updates than the Echo admin anticipated — the arithmetic in `executeCallback` underflows and reverts, permanently locking the requester's ETH inside the contract.

### Finding Description

**Request phase** (`requestPriceUpdatesWithCallback`, line 84):

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```

The stored `req.fee` is the user's payment minus the Echo-level Pyth-fee estimate (`_state.pythFeeInWei`). The Echo protocol fee is immediately accrued:

```solidity
_state.accruedFeesInWei += _state.pythFeeInWei;   // line 99
```

**Execution phase** (`executeCallback`, lines 145 and 161–162):

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);   // live, dynamic
...
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - pythFee);
```

The actual Pyth fee paid is `pyth.getUpdateFee(updateData)`, which scales with the number of price-feed messages packed in the VAA blob — not with `_state.pythFeeInWei`. If `pythFee > req.fee + msg.value`, the subtraction underflows (Solidity 0.8 checked arithmetic), `executeCallback` reverts, and the user's ETH is permanently locked because `clearRequest` is never reached.

Two independent triggers exist:
1. **Pyth governance raises `singleUpdateFeeInWei`** after the request is submitted. The Echo admin must separately call a setter to update `_state.pythFeeInWei`; until they do, every pending request is under-funded.
2. **The executor submits a VAA blob containing more price-feed messages than the user requested.** `getUpdateFee` charges per message in the blob, so a blob with N feeds costs N × `singleUpdateFeeInWei`, regardless of how many `priceIds` the user asked for. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

When `pythFee > req.fee + msg.value`:

- `executeCallback` reverts with an arithmetic underflow.
- `clearRequest` is never executed, so the request remains "active" but can never be fulfilled.
- The user's ETH (`req.fee + _state.pythFeeInWei`) is permanently locked in the Echo contract with no withdrawal path for the requester.

When `pythFee < _state.pythFeeInWei`:

- The provider is over-credited by `_state.pythFeeInWei − pythFee`.
- `_state.accruedFeesInWei` already holds `_state.pythFeeInWei` for the Pyth protocol, but only `pythFee` was actually paid to Pyth. The difference is double-counted, making the contract insolvent by that amount per request. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

- Pyth governance routinely adjusts `singleUpdateFeeInWei` across chains. Any upward adjustment after a request is submitted but before `executeCallback` is called triggers the mismatch.
- The `executeCallback` function is `payable`, so an executor *could* top up the shortfall, but there is no on-chain mechanism that forces or signals this; most executor implementations will call with `msg.value = 0`.
- The window between request and execution is bounded only by the 60-second `publishTime` limit on the *request* side, but execution can happen at any time after the publish time passes — a window that can span many blocks during which a governance fee change could land. [6](#0-5) [7](#0-6) 

### Recommendation

Replace the static `_state.pythFeeInWei` subtraction with the actual live Pyth fee at request time, or redesign the accounting so that the Pyth fee is computed and deducted dynamically at execution time from a separately tracked escrow rather than from `req.fee`. Concretely:

1. At request time, call `pyth.getUpdateFee` with a representative blob (or store the number of `priceIds` and compute the fee from `singleUpdateFeeInWei`) to determine the exact Pyth fee to escrow.
2. Store the escrowed Pyth fee separately from the provider fee in the `Request` struct.
3. In `executeCallback`, deduct the escrowed Pyth fee from the Pyth-fee escrow, not from `req.fee + msg.value`, so provider earnings are never reduced by Pyth fee fluctuations. [8](#0-7) [9](#0-8) 

### Proof of Concept

```solidity
// Assume: _state.pythFeeInWei = 1000 wei (set by Echo admin)
//         singleUpdateFeeInWei on Pyth = 1000 wei at request time
//         Provider fees = 5000 wei
//
// User calls requestPriceUpdatesWithCallback{value: 6000}(...)
//   req.fee = 6000 - 1000 = 5000
//   _state.accruedFeesInWei += 1000
//
// Pyth governance raises singleUpdateFeeInWei to 6000 wei.
//
// Executor calls executeCallback{value: 0}(...)
//   pythFee = pyth.getUpdateFee(updateData) = 6000
//   (req.fee + msg.value) - pythFee = (5000 + 0) - 6000 → UNDERFLOW → REVERT
//
// User's 6000 wei is permanently locked. No refund path exists.
``` [10](#0-9) [11](#0-10)

    function getUpdateFee(
        bytes[] calldata updateData
    ) public view override returns (uint feeAmount) {
        uint totalNumUpdates = 0;
        for (uint i = 0; i < updateData.length; i++) {
            if (
                updateData[i].length > 4 &&
                UnsafeCalldataBytesLib.toUint32(updateData[i], 0) ==
                ACCUMULATOR_MAGIC
            ) {
                (
                    uint offset,
                    UpdateType updateType
                ) = extractUpdateTypeFromAccumulatorHeader(updateData[i]);
                if (updateType != UpdateType.WormholeMerkle) {
                    revert PythErrors.InvalidUpdateData();
                }
                totalNumUpdates += parseWormholeMerkleHeaderNumUpdates(
                    updateData[i],
                    offset
                );
            } else {
                revert PythErrors.InvalidUpdateData();
            }
        }
        return getTotalFee(totalNumUpdates);
    }
