### Title
Write-Before-Read on Overflow Request Storage Causes Callback Delivery Failure — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the `executeCallback` function clears the active request from storage via `clearRequest(sequenceNumber)` **before** reading `req.requester` and `req.callbackGasLimit` to dispatch the consumer callback. For requests stored in the overflow mapping (`_state.requestsOverflow`), `clearRequest` issues a Solidity `delete` on the mapping entry, zeroing all struct fields. Because `req` is a `storage` pointer to that same slot, subsequent reads of `req.requester` and `req.callbackGasLimit` return `address(0)` and `0` respectively. The callback is therefore dispatched to the zero address with zero gas, permanently silencing the consumer's callback while the provider still collects the fee.

---

### Finding Description

`requestPriceUpdatesWithCallback` stores each new request via `allocRequest`. When two sequence numbers hash to the same short key (8-bit bucket), the incumbent request is evicted to `_state.requestsOverflow[reqKey]` and the new request occupies the primary slot. [1](#0-0) 

When `executeCallback` is called for an overflow request, `findActiveRequest` → `findRequest` returns a `storage` pointer directly into `_state.requestsOverflow[key]`: [2](#0-1) 

Inside `executeCallback`, the sequence of operations is:

```solidity
// 1. Fee credit — reads req.fee BEFORE clear (correct)
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

// 2. Clears the request — for overflow entries this is `delete _state.requestsOverflow[key]`
//    which ZEROS every field of the struct that `req` points to
clearRequest(sequenceNumber);

// 3. Reads req.requester and req.callbackGasLimit — now both 0 for overflow requests
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
``` [3](#0-2) 

`clearRequest` for the overflow path:

```solidity
} else {
    delete _state.requestsOverflow[key];   // zeros all struct fields
}
``` [4](#0-3) 

After `delete`, `req.requester == address(0)` and `req.callbackGasLimit == 0`. The `try` call sends 0 gas to the zero address. The call silently succeeds (no code at address(0)), the `catch` branches are never taken, and the consumer contract never receives its price-feed callback.

---

### Impact Explanation

- **Consumer funds lost with no service rendered.** The user paid `getFee(provider, callbackGasLimit, priceIds)` at request time. The fee is fully credited to the provider (`req.fee` is read before `clearRequest`), but the consumer's `_echoCallback` is never invoked.
- **Permanent, non-recoverable.** The request is deleted; there is no retry path.
- **Silent failure.** The `try/catch` block catches only reverts. A call to address(0) with 0 gas does not revert, so no `PriceUpdateCallbackFailed` event is emitted either.

---

### Likelihood Explanation

Overflow occurs whenever two in-flight requests share the same 8-bit bucket (`shortKey = uint8(keccak256(sequenceNumber)[0] & NUM_REQUESTS_MASK)`). With a 256-slot ring buffer, the birthday bound means overflow begins with ~16–20 concurrent open requests — a realistic load for any active Echo deployment. Any unprivileged user can deliberately create colliding requests to force a victim request into the overflow map, then call `executeCallback` on it to trigger the bug.

---

### Recommendation

Read all fields needed after the callback **before** calling `clearRequest`, or restructure so that `clearRequest` is called only after the callback dispatch:

```solidity
// Cache fields before clearing
address requester = req.requester;
uint32 gasLimit = req.callbackGasLimit;

_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

clearRequest(sequenceNumber);

// Use cached values
try IEchoConsumer(requester)._echoCallback{gas: gasLimit}(sequenceNumber, priceFeeds) {
    ...
}
```

---

### Proof of Concept

1. Deploy `EchoUpgradable` with `NUM_REQUESTS = 256`.
2. Submit 17 requests from `userA` such that requests `i` and `j` share the same `shortKey` (trivially achievable by brute-forcing sequence numbers or by controlling timing). Request `j` displaces request `i` into `_state.requestsOverflow`.
3. Call `executeCallback(provider, seqNum_i, updateData, priceIds)`.
4. Observe: `PriceUpdateCallbackFailed` is **not** emitted; `IEchoConsumer(userA)._echoCallback` is **never** called; provider's `accruedFeesInWei` is incremented normally; `userA` receives no price data despite having paid the full fee.

        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];

        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        if (req.sequenceNumber == sequenceNumber) {
