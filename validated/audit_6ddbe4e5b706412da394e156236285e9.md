### Title
Delegators Permanently Stuck When Publisher Is Removed from `pool_data` - (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

In Pyth's Oracle Integrity Staking (OIS) system, the `undelegate` instruction requires the target publisher to be present in `pool_data`. If a publisher is removed from `pool_data` (e.g., after being dropped from `publisher_caps` via the `advance` instruction), delegators can no longer call `advance_delegation_record` or `undelegate` for that publisher. This creates a deadlock where delegator tokens are permanently stuck in a position targeting a non-existent publisher, with no escape path.

---

### Finding Description

The integrity pool program enforces two sequential requirements for undelegation:

1. **`advance_delegation_record` must be called first** — the `OutdatedDelegatorAccounting` error (code 6008) is thrown by `undelegate` if the delegator's accounting is not up to date.
2. **Both `advance_delegation_record` and `undelegate` require the publisher to be present in `pool_data`** — the `PublisherNotFound` error (code 6000) is thrown if the publisher key is not found in `pool_data`. [1](#0-0) 

The `undelegate` instruction explicitly documents: `"CHECK : The publisher will be checked against data in the pool_data"`. [2](#0-1) 

The `advance_delegation_record` instruction carries the same publisher-in-`pool_data` requirement. [3](#0-2) 

The `advance` instruction updates `pool_data` from `publisher_caps`. When a publisher stops submitting prices and is dropped from `publisher_caps`, the next `advance` call removes them from `pool_data`. After that point:

- `advance_delegation_record(publisher)` → fails with `PublisherNotFound`
- `undelegate(publisher)` → fails with `OutdatedDelegatorAccounting` (because accounting is stale) **and** would also fail with `PublisherNotFound` even if accounting were current

The SDK's `unstakeFromPublisher` and `unstakeFromAllPublishers` both route through `integrityPoolProgram.methods.undelegate(...)`, which hits this on-chain check. [4](#0-3) [5](#0-4) 

There is no alternative on-chain instruction that allows a delegator to close a position targeting a publisher that is absent from `pool_data`.

---

### Impact Explanation

Delegators whose stake is assigned to a publisher subsequently removed from `pool_data` cannot:
- Call `advance_delegation_record` to settle accounting
- Call `undelegate` to initiate cooldown
- Redirect their stake to a functioning publisher

Their PYTH tokens remain locked in a `LOCKED` or `LOCKING` position indefinitely, earning no rewards (the removed publisher generates no reward flow) and unable to be withdrawn or reassigned. The tokens are effectively frozen at the protocol level with no user-accessible escape path.

---

### Likelihood Explanation

Publishers are real-world entities that can stop operating (infrastructure failure, business decision, key loss). The `advance` instruction is permissionless and will update `pool_data` to reflect the new `publisher_caps` on the next epoch boundary. Any delegator who has not undelegated before the publisher is dropped from caps will be caught in the deadlock. This is a realistic, non-adversarial scenario requiring no attacker — a publisher simply ceasing operations is sufficient.

---

### Recommendation

Add a fallback `force_undelegate` instruction (or modify `undelegate`) that allows a delegator to close a position targeting a publisher that is **absent** from `pool_data`, bypassing the `PublisherNotFound` check and settling accounting at zero reward for the missing publisher. This mirrors the SKALE recommendation: allow token holders to exit early when the counterparty (publisher/validator) is no longer functional.

---

### Proof of Concept

1. Alice delegates 1000 PYTH to publisher `P` via `delegate`. Position enters `LOCKING` state, then `LOCKED` next epoch.
2. Publisher `P` stops submitting prices. In the next epoch, `advance` is called with new `publisher_caps` that omit `P`. `pool_data` no longer contains `P`.
3. Alice calls `advance_delegation_record` for `P` → transaction reverts with `PublisherNotFound` (6000).
4. Alice calls `undelegate` for `P` → transaction reverts with `OutdatedDelegatorAccounting` (6008) because step 3 failed, and would also revert with `PublisherNotFound` (6000) even if accounting were current.
5. Alice's 1000 PYTH remain locked in a position targeting a non-existent publisher with no on-chain path to recover them. [1](#0-0) [6](#0-5) [7](#0-6)

  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }
