Audit Report

## Title
Unbounded `forward_rate_limiter` HashMap Growth via Persistent Connection with Unique PeerId Pairs — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

`HolePunching.forward_rate_limiter` is a `governor::RateLimiter` backed by `HashMapStateStore<(PeerId, PeerId, u32)>`. Its only cleanup call, `retain_recent()`, is placed exclusively in the `disconnected()` handler. The periodic `notify()` handler (every 5 minutes) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on `forward_rate_limiter`. An attacker maintaining a persistent connection and sending `ConnectionRequest` messages with unique attacker-controlled `(from, to)` PeerId pairs causes the HashMap to grow without bound, eventually exhausting heap memory and crashing the node.

## Finding Description

**Root cause — type declaration:**
`forward_rate_limiter` is declared as `RateLimiter<(PeerId, PeerId, u32)>` backed by `HashMapStateStore`, which grows with each unique key and requires explicit `retain_recent()` calls to evict stale entries. [1](#0-0) 

**Cleanup only in `disconnected()`:**
`retain_recent()` is called on `forward_rate_limiter` only when a peer disconnects. If the attacker never disconnects, this code never runs. [2](#0-1) 

**`notify()` omits `retain_recent()`:**
The periodic timer handler (every 5 minutes) cleans `pending_delivered` and `inflight_requests` but makes no call to `retain_recent()` on either rate limiter, leaving stale entries to accumulate indefinitely. [3](#0-2) 

**Per-session rate limiter does not bound unique keys:**
The outer `rate_limiter` is keyed by `(session_id, msg.item_id())` — a fixed key per session per message type — capping throughput at 30 messages/second but placing no limit on the number of distinct `(from, to)` pairs that reach `forward_rate_limiter`. [4](#0-3) 

**Each unique pair inserts a new HashMap entry:**
Every `ConnectionRequest` with a novel `(from, to)` pair calls `check_key` on `forward_rate_limiter`, inserting a new entry into the `HashMapStateStore` that is never evicted while the session is live. [5](#0-4) 

**`from` and `to` are fully attacker-controlled:**
Both fields are parsed directly from raw bytes in the message payload with no validation against any connected peer registry. [6](#0-5) 

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Growth rate is 30 entries/second per persistent connection (bounded only by the per-session rate limiter). Each entry stores two `PeerId` values (~39 bytes each), a `u32`, governor internal state, and HashMap overhead — approximately 150–200 bytes per entry. After 24 hours on a single connection: ~2.6 million entries ≈ ~400 MB of heap growth. Multiple simultaneous attacker connections multiply the rate linearly. No automatic eviction occurs until the attacker disconnects; the node has no mechanism to force cleanup of `forward_rate_limiter` while the session is live. Result: heap exhaustion → OOM → node crash.

## Likelihood Explanation

Requires only one persistent inbound P2P connection — no special privileges, no proof-of-work, no cryptographic keys. PeerId values in `from`/`to` fields are not validated against any connected peer registry; arbitrary bytes are accepted as long as they parse as valid multihash PeerIds. The attack is slow (hours to days for a single connection) but reliable and amplifiable with multiple simultaneous connections. No existing guard prevents it: the per-session rate limiter bounds rate but not total unique keys; `notify()` does not call `retain_recent()`.

## Recommendation

Add `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside the `notify()` handler in `network/src/protocols/hole_punching/mod.rs`, immediately after the existing `retain` calls on `pending_delivered` and `inflight_requests` at lines 173–175. This ensures periodic eviction of stale entries regardless of whether peers disconnect, bounding the HashMap to entries active within the last rate-limit window. [3](#0-2) 

## Proof of Concept

1. Connect one persistent TCP session to a victim CKB node's HolePunching protocol endpoint.
2. In a loop, construct `ConnectionRequest` messages where `from = PeerId::random()` and `to = PeerId::random()` (unique per iteration), with valid `listen_addrs` and `max_hops > 0`.
3. Send at most 30 messages/second (to stay under the per-session rate limiter keyed by `(session_id, msg_item_id)`).
4. Never close the connection.
5. After N seconds, observe that `forward_rate_limiter`'s internal `HashMapStateStore` contains N unique entries and that RSS of the `ckb` process grows proportionally.
6. Assert: after 10^6 messages (~9.3 hours), the HashMap holds ~10^6 entries consuming ~150–200 MB; `retain_recent()` has never been called because `disconnected()` was never triggered.
