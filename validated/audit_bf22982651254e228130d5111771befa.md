### Title
Missing Ancestor-Height Check in `GetBlocksProofProcess::execute` Allows Unbanned Repeated `InternalError` Triggering — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

---

### Summary

An unprivileged remote peer can craft a `GetBlocksProof` message where `block_hashes` contains hashes of main-chain blocks whose heights exceed `last_hash`'s height. The server accepts these as "found" blocks, computes their MMR leaf positions, then calls `gen_proof` against an MMR sized only to `last_block.number()-1`. Because the positions exceed the MMR size, `gen_proof` returns an error, causing `reply_proof` to return `StatusCode::InternalError`. Since `InternalError` (500) is outside the 400–499 ban range, the peer is never banned and can repeat this indefinitely.

### Finding Description

**Step 1 — Attacker-controlled entrypoint.**

Any peer on the P2P network can send a `GetBlocksProof` message. The handler is registered in `try_process`: [1](#0-0) 

**Step 2 — Missing height guard in `execute()`.**

The function validates that `last_hash` is on the main chain and that each hash in `block_hashes` is on the main chain, but it never checks that those blocks' heights are ≤ `last_block.number()`: [2](#0-1) [3](#0-2) 

A block at height 50 that is on the main chain passes `is_main_chain()` and lands in `found`. Its MMR leaf position `leaf_index_to_pos(50)` is pushed into `positions`.

**Step 3 — MMR is sized to `last_block.number()-1`.**

`reply_proof` constructs the MMR anchored at `last_block`, which covers only leaf indices 0 through `last_block.number()-1`: [4](#0-3) 

When `last_block.number() = 10` and `positions` contains `leaf_index_to_pos(50)`, `gen_proof` is called with a position that is outside the MMR's range. The MMR library returns an error, and the function returns `StatusCode::InternalError`.

**Step 4 — `InternalError` (500) does not trigger a ban.**

`should_ban()` only returns `Some(ban_time)` for codes in the 400–499 range: [5](#0-4) 

`InternalError = 500` falls in the 500–599 range, which only triggers a warning log: [6](#0-5) 

The peer is never disconnected or banned.

### Impact Explanation

An attacker can send an unlimited stream of `GetBlocksProof` messages with `last_hash` pointing to a low-height main-chain block and `block_hashes` containing hashes of higher-height main-chain blocks. Each message causes the server to:
1. Look up the `last_block` header.
2. Fetch headers and uncle data for each "found" block.
3. Open the MMR store and compute the root (`get_root()` succeeds).
4. Attempt `gen_proof` (fails, returns `InternalError`).

No response is sent to the attacker, but the server wastes I/O and CPU on every request. Because no ban is applied, the attacker can sustain this at the maximum rate the network allows, degrading proof-service throughput for legitimate light clients.

---

### Likelihood Explanation

The exploit requires only a valid P2P connection and knowledge of two main-chain block hashes at different heights — both trivially obtainable from any public node or block explorer. No PoW, no key material, no privileged role. The attack is free to repeat.

---

### Recommendation

In `execute()`, after resolving `last_block`, reject any block hash in `block_hashes` whose header number exceeds `last_block.number()`. Blocks at heights strictly greater than `last_hash`'s height cannot be ancestors of `last_hash` and should be treated as a malformed request (returning `StatusCode::MalformedProtocolMessage`, which is in the 400–499 range and **does** trigger a ban):

```rust
// After the `found` partition loop, before building `positions`:
for block_hash in &found {
    let header = snapshot.get_block_header(block_hash).expect("...");
    if header.number() > last_block.number() {
        return StatusCode::MalformedProtocolMessage
            .with_context("block hash is not an ancestor of last_hash");
    }
    // ... existing position/header/uncle logic
}
```

This converts the error from a server-side `InternalError` (no ban) into a client-side `MalformedProtocolMessage` (ban), closing the free-repeat loop.
