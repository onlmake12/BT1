### Title
Starknet Pyth Contract `MessageType` Enum Only Supports `PriceFeed` (Type 0), Blocking All `TwapPriceFeed` (Type 1) Accumulator Updates — (File: `target_chains/starknet/contracts/src/pyth/price_update.cairo`)

---

### Summary

The Starknet Pyth contract's `MessageType` enum defines only `PriceFeed` (type `0`). The `U8TryIntoMessageType` implementation returns `None` for any other value, causing `read_and_verify_message` to panic when it encounters a `TwapPriceFeed` message (type `1`). This prevents any accumulator update containing TWAP messages from being processed on Starknet, while the EVM contract fully supports both message types.

---

### Finding Description

In `price_update.cairo`, the `MessageType` enum has only one variant:

```cairo
enum MessageType {
    PriceFeed,
}

impl U8TryIntoMessageType of TryInto<u8, MessageType> {
    fn try_into(self: u8) -> Option<MessageType> {
        if self == 0 {
            Option::Some(MessageType::PriceFeed)
        } else {
            Option::None   // <-- TwapPriceFeed (type 1) falls here
        }
    }
}
``` [1](#0-0) 

In `read_and_verify_message`, the message type byte is read and converted with `.expect()`:

```cairo
let message_type: MessageType = message_reader
    .read_u8()
    .try_into()
    .expect(UpdatePriceFeedsError::InvalidUpdateData.into());

match message_type {
    MessageType::PriceFeed => {},
}
``` [2](#0-1) 

If the message type byte is `1` (`TwapPriceFeed`), `try_into()` returns `None`, and `.expect()` panics with `InvalidUpdateData`, reverting the entire update.

This function is called inside `update_price_feeds_internal` for every message in the accumulator update:

```cairo
while i < num_updates {
    let message = read_and_verify_message(ref reader, root_digest);
    ...
}
``` [3](#0-2) 

By contrast, the EVM contract (`PythAccumulator.sol`) defines both variants and accepts both in `extractAndValidateEncodedMessage`:

```solidity
enum MessageType {
    PriceFeed,
    TwapPriceFeed
}
...
if (
    messageType != MessageType.PriceFeed &&
    messageType != MessageType.TwapPriceFeed
) {
    revert PythErrors.InvalidUpdateData();
}
``` [4](#0-3) [5](#0-4) 

`TwapPriceFeed` is already a live, deployed feature on EVM chains, exposed via `parseTwapPriceFeedUpdates` in `IPyth.sol`: [6](#0-5) 

---

### Impact Explanation

Any accumulator update containing a `TwapPriceFeed` message (type byte `0x01`) submitted to the Starknet Pyth contract will unconditionally panic and revert. Because the panic occurs inside the per-message loop, even a mixed update containing valid `PriceFeed` messages alongside a single `TwapPriceFeed` message will fail entirely — no price feeds are updated. Starknet users and applications that depend on TWAP price data are permanently blocked from this feature, and any keeper or relayer attempting to submit such an update loses gas with no state change.

---

### Likelihood Explanation

`TwapPriceFeed` (message type `1`) is already a production feature on EVM chains. As Pythnet begins publishing TWAP messages in accumulator updates destined for Starknet, any transaction sender or keeper submitting those updates will trigger this revert. The likelihood is high and grows with TWAP adoption across chains.

---

### Recommendation

Add `TwapPriceFeed` to the `MessageType` enum in `price_update.cairo` and update `U8TryIntoMessageType` to map value `1` to `MessageType::TwapPriceFeed`. Update `read_and_verify_message` to either parse the TWAP message into a new return type or skip it gracefully (analogous to how the EVM contract routes by type). This mirrors the fix pattern recommended for EigenPod: add a new handler for the new type rather than hard-failing on it.

```cairo
enum MessageType {
    PriceFeed,
    TwapPriceFeed,  // add this
}

impl U8TryIntoMessageType of TryInto<u8, MessageType> {
    fn try_into(self: u8) -> Option<MessageType> {
        if self == 0 {
            Option::Some(MessageType::PriceFeed)
        } else if self == 1 {
            Option::Some(MessageType::TwapPriceFeed)  // add this
        } else {
            Option::None
        }
    }
}
```

### Proof of Concept

1. Obtain a valid Wormhole-signed accumulator update from Pythnet containing a `TwapPriceFeed` message (message type byte = `0x01`).
2. Submit this update to the Starknet Pyth contract via `update_price_feeds` or `parse_price_feed_updates`.
3. `update_price_feeds_internal` calls `read_and_verify_message` for each message in the update.
4. `read_and_verify_message` reads the message type byte `0x01` and calls `try_into()`.
5. `U8TryIntoMessageType::try_into(1)` returns `None`.
6. `.expect(UpdatePriceFeedsError::InvalidUpdateData.into())` panics.
7. The entire transaction reverts; no price feeds are updated on Starknet. [7](#0-6)

pub fn read_and_verify_message(ref reader: Reader, root_digest: u256) -> PriceFeedMessage {
    let message_size = reader.read_u16();
    let message = reader.read_byte_array(message_size.into());
    read_and_verify_proof(root_digest, @message, ref reader);

    let mut message_reader = ReaderImpl::new(message);
    let message_type: MessageType = message_reader
        .read_u8()
        .try_into()
        .expect(UpdatePriceFeedsError::InvalidUpdateData.into());

    match message_type {
        MessageType::PriceFeed => {},
    }

    let price_id = message_reader.read_u256();
    let price = u64_as_i64(message_reader.read_u64());
    let conf = message_reader.read_u64();
    let expo = u32_as_i32(message_reader.read_u32());
    let publish_time = message_reader.read_u64();
    let prev_publish_time = message_reader.read_u64();
    let ema_price = u64_as_i64(message_reader.read_u64());
    let ema_conf = message_reader.read_u64();

    PriceFeedMessage {
        price_id, price, conf, expo, publish_time, prev_publish_time, ema_price, ema_conf,
    }
}
