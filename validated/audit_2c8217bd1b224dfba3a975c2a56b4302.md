### Title
`HermesClient.httpRequest` Retries Unconditionally on HTTP 429 Without Respecting `Retry-After`, Causing Rate-Limit Amplification and Stale On-Chain Prices - (File: `apps/hermes/client/js/src/hermes-client.ts`)

---

### Summary

`HermesClient.httpRequest` retries on every non-`AbortError` failure, including HTTP 429 (Too Many Requests), using a short exponential backoff starting at 100–200 ms. It never reads the `Retry-After` header that Hermes sends with a 429. This causes the `price_pusher` (and any other `HermesClient` consumer) to fire up to 4 requests in ~1 second when rate-limited, all of which receive 429, potentially extending the 60-second Hermes ban and causing on-chain prices to go stale.

---

### Finding Description

`HermesClient` is the canonical Pyth SDK for fetching price updates from Hermes. It is used by `price_pusher` across all supported chains (EVM, Aptos, Solana, Injective, NEAR, Fuel, TON) to call `getLatestPriceUpdates` on every push cycle.

The private `httpRequest` method handles all HTTP responses:

```typescript
if (!response.ok) {
  const errorBody = await response.text();
  throw new Error(
    `HTTP error! status: ${response.status.toString()}${...}`,
  );
}
```

A 429 response throws a plain `Error` (not an `AbortError`). The catch block then retries unconditionally:

```typescript
} catch (error) {
  if (
    retries > 0 &&
    !(error instanceof Error && error.name === "AbortError")
  ) {
    await new Promise((resolve) => setTimeout(resolve, backoff));
    return this.httpRequest(url, schema, options, retries - 1, backoff * 2);
  }
  throw error;
}
```

With `DEFAULT_HTTP_RETRIES = 3` and initial backoff of 100–200 ms, this fires 4 total requests in approximately 1 050 ms. The `Retry-After` header (which Hermes sets to 60 seconds on a 429) is never read or respected. [1](#0-0) 

The Hermes public endpoint enforces 10 requests per 10 seconds per IP, and a 429 triggers a 60-second ban window. [2](#0-1) 

`price_pusher` calls `hermesClient.getLatestPriceUpdates` on every push cycle for all configured price feeds: [3](#0-2) 

Additionally, `PythPriceListener.startListening` reconnects the SSE stream after any error with a fixed 5-second delay, without distinguishing a 429 from other errors:

```typescript
eventSource.onerror = async (error: Event) => {
  eventSource.close();
  await sleep(5000);
  void this.startListening(); // Attempt to restart the listener
};
``` [4](#0-3) 

During a 60-second Hermes ban, a pusher with a 10-second `pushing-frequency` makes 6 push cycles × 4 requests = 24 requests, all receiving 429, continuously refreshing the ban window.

By contrast, the legacy `PriceServiceConnection` uses `axiosRetry` with the default `isNetworkOrIdempotentRequestError` condition, which does **not** retry on 429. The new `HermesClient` regressed this behavior. [5](#0-4) 

---

### Impact Explanation

When a `price_pusher` operator hits the Hermes public rate limit (e.g., multiple pusher instances sharing an IP, or a short `pushing-frequency`), the pusher enters a self-reinforcing ban cycle. It cannot fetch fresh VAAs from Hermes, so `updatePriceFeed` is never called, and on-chain Pyth prices go stale. DeFi protocols relying on the price pusher for regular updates will read stale prices. The `PythPriceListener` health check throws after 30 seconds of no updates, crashing the process, which then restarts and immediately hammers Hermes again. [6](#0-5) 

---

### Likelihood Explanation

Medium. Operators running `price_pusher` against the public Hermes endpoint without an API key are subject to the 10 req/10s limit. Multiple pusher instances behind a shared NAT, or a `pushing-frequency` below 1 second, can trigger the limit. The SSE reconnect loop (5-second interval) can independently trigger it during network instability. The issue is latent in every deployment using the public endpoint without authentication.

---

### Recommendation

1. In `httpRequest`, parse the HTTP status before throwing. If status is 429, read the `Retry-After` header and either delay by that amount or throw a typed error that the retry loop can inspect. Do not retry a 429 with the standard short backoff.
2. In `PythPriceListener.onerror`, inspect the error type. If it is a 429/rate-limit error, back off for at least 60 seconds before reconnecting.
3. Align `HermesClient` retry behavior with the `apps/mcp/src/clients/retry.ts` pattern, which already correctly parses `Retry-After` and caps delay at 30 seconds. [7](#0-6) 

---

### Proof of Concept

1. Run `price_pusher` for EVM against the public Hermes endpoint with no `--hermes-access-token` and `--pushing-frequency 1`.
2. After a few seconds, Hermes returns HTTP 429.
3. `httpRequest` retries 3 times with backoffs of ~150 ms, ~300 ms, ~600 ms — 4 total requests in ~1 050 ms, all receiving 429.
4. The push cycle fails; `updatePriceFeed` is never called.
5. On the next cycle (1 second later), the same 4-request burst repeats, refreshing the 60-second ban.
6. After 30 seconds with no successful SSE message, the health check at line 51 of `pyth-price-listener.ts` throws `"Hermes Price feeds are not updating."`, crashing the process.
7. On restart, the process immediately reconnects and repeats the burst, extending the ban indefinitely. [8](#0-7) [9](#0-8)

    try {

      const response = await fetch(url, {
        ...options,

        headers: { ...authHeaders, ...this.headers, ...options?.headers },
        signal: AbortSignal.any([
          ...(options?.signal ? [options.signal] : []),
          AbortSignal.timeout(this.timeout),
        ]),
      });
      if (!response.ok) {
        const errorBody = await response.text();
        throw new Error(
          `HTTP error! status: ${response.status.toString()}${
            errorBody ? `, body: ${errorBody}` : ""
          }`,
        );

      const data = await response.json();
      return schema.parse(data);
    } catch (error) {

        return this.httpRequest(url, schema, options, retries - 1, backoff * 2); // Exponential backoff
      }

    }
  }

function getDelay(err: unknown): number {
  if (err instanceof HttpError && err.retryAfter != null) {
    return Math.min(err.retryAfter * 1000, MAX_RETRY_DELAY_MS);
  }
  return RETRY_BASE_DELAY_MS + Math.random() * JITTER_MAX_MS;

export function parseRetryAfter(res: Response): number | undefined {
  const header = res.headers.get("Retry-After");
  if (!header) return undefined;
  const seconds = Number(header);
  if (Number.isFinite(seconds)) return seconds;
  const dateMs = Date.parse(header);
  if (Number.isNaN(dateMs)) return undefined;
  return Math.max(0, (dateMs - Date.now()) / 1000);
}
