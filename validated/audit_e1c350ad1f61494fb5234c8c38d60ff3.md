Audit Report

## Title
Memory Amplification via Duplicate Indexes and Missing Byte-Size Cap in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` accepts up to 32,767 peer-supplied transaction indexes with no deduplication and no byte-size cap on the assembled response. An unprivileged peer can send all 32,767 indexes pointing at the same transaction, forcing the node to clone and buffer that transaction 32,767 times before sending. Both analogous handlers (`GetTransactionsProcess`, `GetBlockProposalProcess`) explicitly reject duplicate entries and enforce a 1 MiB `MAX_RELAY_TXS_BYTES_PER_BATCH` cap; this handler does neither.

## Finding Description

**Root cause 1 — no deduplication:**

The handler iterates raw peer-supplied indexes with no `HashSet` deduplication step:

```rust
let transactions = self
    .message
    .indexes()
    .iter()
    .filter_map(|i| {
        block
            .transactions()
            .get(Into::<u32>::into(i) as usize)
            .cloned()   // clones on every iteration, including duplicates
    })
    .collect::<Vec<_>>();
``` [1](#0-0) 

Sending `indexes = [0u32; 32767]` produces a `Vec` of 32,767 clones of `block.transactions()[0]`. The count guard at line 37 uses `>` (not `>=`), so exactly 32,767 entries pass through: [2](#0-1) 

By contrast, `GetTransactionsProcess` builds a `HashSet` and immediately returns `StatusCode::RequestDuplicate` if duplicates are detected: [3](#0-2) 

And `GetBlockProposalProcess` does the same: [4](#0-3) 

**Root cause 2 — no byte-size cap:**

The entire `transactions` vec is packed into one message and sent unconditionally with no `MAX_RELAY_TXS_BYTES_PER_BATCH` accumulator: [5](#0-4) 

Both `GetTransactionsProcess` and `GetBlockProposalProcess` accumulate `relay_bytes` and flush when `relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH`: [6](#0-5) [7](#0-6) 

`MAX_RELAY_TXS_BYTES_PER_BATCH` is defined as 1 MiB: [8](#0-7) 

**Exploit flow:**
1. Attacker establishes a TCP connection to the relay protocol (no credentials required).
2. Attacker obtains any block hash stored on the target node (trivially from a block explorer or header sync).
3. Attacker sends a `GetBlockTransactions` message with `block_hash = <stored_hash>` and `indexes = [0u32; 32767]`.
4. The count guard passes (`32767 > 32767` is false).
5. The handler clones `block.transactions()[0]` 32,767 times into a `Vec`.
6. The entire vec is packed into a single `BlockTransactions` message and sent — no byte cap applied.

**Why existing guards fail:**
- The count guard (`> MAX_RELAY_TXS_NUM_PER_BATCH`) only limits the number of index entries, not their uniqueness or the resulting byte size.
- The rate limiter (30 req/s per peer) does not bound per-request allocation; 30 × 16 MB = 480 MB/s with a typical coinbase. [9](#0-8) 

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Peak allocation per request = `32,767 × coinbase_tx_size`. With a typical mainnet coinbase of ~500 bytes, a single request allocates ~16 MB. At the rate-limiter ceiling of 30 req/s per peer, one peer alone can drive ~480 MB/s of allocation. Multiple simultaneous peers exhaust memory rapidly. In the worst case (coinbase approaching CKB's `max_block_bytes` of ~597 KB), a single request allocates ~19.5 GB — a guaranteed OOM crash on any normal server. The allocation occurs entirely before any send, so network backpressure provides no relief.

## Likelihood Explanation

The attack requires only a TCP connection to the relay protocol — no proof-of-work, no key, no privileged role. The required block hash is publicly available from any block explorer or by syncing headers. The message is structurally valid and passes all existing guards. The attack is repeatable, parallelizable across many peers, and deterministic.

## Recommendation

1. **Deduplicate indexes before lookup**: collect `indexes` into a `HashSet<u32>`, return `StatusCode::RequestDuplicate` if `set.len() != message_len`, then iterate the deduplicated set — matching the pattern in `GetTransactionsProcess` and `GetBlockProposalProcess`.
2. **Add a `MAX_RELAY_TXS_BYTES_PER_BATCH` accumulator** over the assembled transactions and either truncate or split the response when the limit is exceeded, matching the batching loop in `GetTransactionsProcess::execute` and `GetBlockProposalProcess::execute`.
3. Optionally change the count guard from `>` to `>=` for semantic consistency with the constant's meaning, though this is a minor hardening step.

## Proof of Concept

```rust
// Minimal unit test sketch
let indexes = vec![0u32; 32767]; // all pointing at coinbase
let msg = GetBlockTransactions {
    block_hash: stored_block_hash, // any block stored on the target node
    indexes,
    uncle_indexes: vec![],
};
// Node executes GetBlockTransactionsProcess::execute():
//   - count guard passes (32767 > 32767 == false)
//   - no dedup: 32767 clones of block.transactions()[0] collected
//   - no byte cap: entire vec packed into one BlockTransactions message
// With coinbase_size = 500 B  → ~16 MB allocated per request
// With coinbase_size = 597 KB → ~19.5 GB allocated per request → OOM crash
```

A fuzz test targeting `GetBlockTransactionsProcess::execute` with a stored block containing a large coinbase and `indexes` filled with `[0u32; 32767]` will reproduce the allocation spike deterministically.
