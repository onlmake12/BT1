### Title
Attacker Can DoS Entropy Randomness Requests by Flooding Unrevealed Requests Past `maxNumHashes` - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

An unprivileged user can call `requestV2` exactly `maxNumHashes` times without ever revealing those requests. Because `currentCommitmentSequenceNumber` only advances on reveal, the next legitimate request's computed `numHashes` exceeds `maxNumHashes`, causing it to revert with `LastRevealedTooOld`. This blocks all new Entropy requests until the Fortuna keeper calls `advanceProviderCommitment`, which polls only every 30 seconds — a window the attacker can exploit repeatedly.

---

### Finding Description

In `requestHelper`, the contract computes:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber - providerInfo.currentCommitmentSequenceNumber
);
if (providerInfo.maxNumHashes != 0 && req.numHashes > providerInfo.maxNumHashes) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [1](#0-0) 

`currentCommitmentSequenceNumber` only advances when `revealHelper` is called (i.e., a request is actually revealed):

```solidity
if (providerInfo.currentCommitmentSequenceNumber < req.sequenceNumber) {
    providerInfo.currentCommitmentSequenceNumber = req.sequenceNumber;
    providerInfo.currentCommitment = providerContribution;
}
``` [2](#0-1) 

If an attacker makes exactly `maxNumHashes` requests and never reveals any of them, `currentCommitmentSequenceNumber` stays at its initial value. The very next request by any user gets `numHashes = maxNumHashes + 1`, which exceeds the limit and reverts.

The Fortuna keeper (`commitment.rs`) is the only mitigation — it monitors `outstanding_requests` and calls `advanceProviderCommitment` when the threshold is crossed:

```rust
const UPDATE_COMMITMENTS_INTERVAL: Duration = Duration::from_secs(30);
const UPDATE_COMMITMENTS_THRESHOLD_FACTOR: f64 = 0.95;
...
if outstanding_requests > threshold {
    tracing::warn!("Update commitments threshold reached -- possible outage or DDOS attack...");
    // calls advanceProviderCommitment
}
``` [3](#0-2) 

The keeper polls every 30 seconds. After it calls `advanceProviderCommitment`, the attacker can immediately re-flood with another `maxNumHashes` requests, re-triggering the DoS. The cycle repeats indefinitely.

The test suite explicitly confirms this attack path:

```solidity
function testLastRevealedTooOld() public {
    for (uint256 i = 0; i < provider1MaxNumHashes; i++) {
        request(user1, provider1, 42, false);
    }
    assertRequestReverts(..., EntropyErrors.LastRevealedTooOld.selector);
}
``` [4](#0-3) 

---

### Impact Explanation

Any DeFi protocol, game, or application relying on Pyth Entropy for randomness is blocked from creating new requests during the attack window. The `requestV2` / `requestWithCallback` entry points revert for all callers — not just the attacker — until the Fortuna keeper responds. Since the attacker can re-flood immediately after each keeper response, the service can be kept in a near-continuous degraded state. Affected users lose the fees they paid for failed transactions.

---

### Likelihood Explanation

The attack requires paying `maxNumHashes` × `feeInWei` per cycle. For the default Pyth provider, fees are non-trivial but the attack is economically rational for a motivated adversary (e.g., a competitor disrupting a protocol that depends on Entropy). The attacker does not need any privileged role — only the ability to call `requestV2` with sufficient ETH. The 30-second keeper polling interval means each attack cycle causes a guaranteed outage window. The Fortuna keeper's own log message ("possible outage or DDOS attack") confirms this is a recognized, realistic threat. [5](#0-4) 

---

### Recommendation

1. **Rate-limit requests per address per block** in `requestHelper` to raise the cost of flooding.
2. **Reduce the keeper polling interval** (`UPDATE_COMMITMENTS_INTERVAL`) significantly (e.g., to 1–2 seconds) or trigger `advanceProviderCommitment` reactively on each new request when `outstanding_requests` approaches the threshold.
3. **Decouple `numHashes` from `currentCommitmentSequenceNumber`** for the purpose of the `maxNumHashes` check — for example, only count hashes for requests that have been pending longer than a grace period, so fresh unrevealed requests do not inflate the counter for new requests.
4. Consider requiring a minimum deposit or bond for requests that is slashable if the request is never revealed within N blocks, disincentivizing the "request and abandon" pattern.

---

### Proof of Concept

```
Setup:
  - Provider registers with maxNumHashes = M (e.g., 100)
  - Provider fee = F wei per request

Attack:
  1. Attacker calls requestV2() M times, paying M * F wei total.
     Each call succeeds; attacker never calls revealWithCallback.
     currentCommitmentSequenceNumber remains at its initial value (e.g., 0).
     providerInfo.sequenceNumber is now M+1.

  2. Legitimate user calls requestV2():
     assignedSequenceNumber = M+1
     numHashes = (M+1) - 0 = M+1 > maxNumHashes (M)
     → REVERT: LastRevealedTooOld

  3. All new requests revert for up to 30 seconds until Fortuna keeper
     calls advanceProviderCommitment(provider, M, proof[M]).

  4. Attacker immediately calls requestV2() M more times.
     Cycle repeats from step 2.

Cost per cycle: M * F wei (lost by attacker, accrued to provider/Pyth).
DoS window per cycle: up to 30 seconds (keeper polling interval).
``` [6](#0-5) [7](#0-6) [8](#0-7)

const UPDATE_COMMITMENTS_INTERVAL: Duration = Duration::from_secs(30);
const UPDATE_COMMITMENTS_THRESHOLD_FACTOR: f64 = 0.95;

#[tracing::instrument(name = "update_commitments", skip_all)]
pub async fn update_commitments_loop(
    contract: Arc<InstrumentedSignablePythContract>,
    chain_state: BlockchainState,
) {
    loop {
        if let Err(e) = update_commitments_if_necessary(contract.clone(), &chain_state)
            .in_current_span()
            .await
        {
            tracing::error!("Update commitments. error: {:?}", e);
        }
        time::sleep(UPDATE_COMMITMENTS_INTERVAL).await;
    }
}

pub async fn update_commitments_if_necessary(
    contract: Arc<InstrumentedSignablePythContract>,
    chain_state: &BlockchainState,
) -> Result<()> {
    //TODO: we can reuse the result from the last call from the watch_blocks thread to reduce RPCs
    let latest_safe_block = get_latest_safe_block(chain_state).in_current_span().await;
    let provider_address = chain_state.provider_address;
    let provider_info = contract
        .get_provider_info_v2(provider_address)
        .block(latest_safe_block) // To ensure we are not revealing sooner than we should
        .call()
        .await
        .map_err(|e| {
            anyhow!(
                "Error while getting provider info at block {}. error: {:?}",
                latest_safe_block,
                e
            )
        })?;
    if provider_info.max_num_hashes == 0 {
        return Ok(());
    }
    let threshold =
        ((provider_info.max_num_hashes as f64) * UPDATE_COMMITMENTS_THRESHOLD_FACTOR) as u64;
    let outstanding_requests =
        provider_info.sequence_number - provider_info.current_commitment_sequence_number;
    if outstanding_requests > threshold {
        // NOTE: This log message triggers a grafana alert. If you want to change the text, please change the alert also.
        tracing::warn!("Update commitments threshold reached -- possible outage or DDOS attack. Number of outstanding requests: {:?} Threshold: {:?}", outstanding_requests, threshold);
        let seq_number = provider_info.sequence_number - 1;
        let provider_revelation = chain_state
            .state
            .reveal(seq_number)
            .map_err(|e| anyhow!("Error revealing: {:?}", e))?;
        let contract_call =
            contract.advance_provider_commitment(provider_address, seq_number, provider_revelation);
        send_and_confirm(contract_call).await?;
    }
    Ok(())
