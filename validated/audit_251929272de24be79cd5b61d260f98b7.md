### Title
Missing `publishSlot` Equality Guard Causes Division-by-Zero in `calculateTwap` — (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary
`validateTwapPriceInfo` rejects `start > end` but permits `start == end`. When `publishSlot` values are equal, `slotDiff = 0` flows into three unchecked integer divisions in `calculateTwap`, causing an EVM panic revert after full Wormhole VAA verification gas has been consumed for both blobs.

### Finding Description
`parseTwapPriceFeedUpdates` accepts two VAA blobs, verifies them via Wormhole (expensive), then calls `validateTwapPriceInfo` followed by `calculateTwap`.

The validation at line 604 is: [1](#0-0) 

The condition is `>` not `>=`, so equal slots pass. `calculateTwap` then performs: [2](#0-1) [3](#0-2) 

All three divide by `slotDiff` with no zero-guard. Solidity 0.8+ panics on integer division by zero, reverting the entire transaction.

A relayer can trivially trigger this by submitting the **same valid VAA** as both `updateData[0]` and `updateData[1]` — both blobs are independently Merkle/Wormhole-verified, there is no deduplication check between start and end inputs. [4](#0-3) 

### Impact Explanation
Any caller of `parseTwapPriceFeedUpdates` (a public, payable external function) who relies on an untrusted relayer for VAA data can have their transaction forced into a revert after paying full Wormhole verification gas for two blobs. In keeper/relayer-driven architectures (common for on-chain TWAP consumers), a single malicious relayer can grief every TWAP query indefinitely at near-zero cost to themselves (they only need one valid VAA, reused twice). The contract remains permanently unusable for any price feed whose data is sourced through that relayer path.

### Likelihood Explanation
The precondition is trivially satisfiable: any single valid TWAP VAA can be duplicated. No privileged access, no key compromise, no governance majority is required. The attacker pays only the Pyth fee and their own gas; the victim pays full Wormhole verification gas for two VAAs before the revert.

### Recommendation
Change the strict inequality to `>=` in `validateTwapPriceInfo`:

```solidity
// Before (line 604):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After:
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
```

This ensures `slotDiff >= 1` before `calculateTwap` is ever reached, eliminating all three division-by-zero sites without any other code change. [1](#0-0)
