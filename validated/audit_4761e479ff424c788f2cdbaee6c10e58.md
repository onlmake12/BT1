Audit Report

## Title
Rich-Indexer `insert_transaction` input loop aborts on first filtered-out cell, leaving subsequent indexed cells permanently marked `is_spent=0` — (`util/rich-indexer/src/indexer/mod.rs`)

## Summary
In `AsyncRichIndexer::insert_transaction`, the input-processing loop issues a `break` whenever `spend_cell` returns `false`. `spend_cell` returns `false` when the referenced output is absent from the DB (filtered out by `cell_filter`). Any inputs after the first filtered-out input are never visited, so their `is_spent` column is never updated to `1`, leaving stale live-cell records in the indexer's `output` table permanently.

## Finding Description
`spend_cell` (`insert.rs` L403–427) executes `UPDATE output SET is_spent = 1 WHERE ...` and returns `Ok(rows_affected > 0)`. When the referenced output was excluded by `cell_filter` and is absent from the DB, `rows_affected == 0`, so it returns `Ok(false)`.

The input loop in `mod.rs` L229–248:
```rust
for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
    let out_point = input.previous_output();
    if !spend_cell(&out_point, tx).await? {
        break;   // exits the entire loop
    }
    // ... build_input_rows for indexed inputs ...
}
```
When input `i` references a filtered-out cell, `spend_cell` returns `false` and `break` exits the loop. Inputs `i+1, i+2, …` are never processed. Their DB rows retain `is_spent = 0` indefinitely, surviving node restarts.

## Impact Explanation
This is an incorrect implementation of CKB's state storage mechanism (rich-indexer), matching **Medium (2001–10000 points)**. The `output` table permanently misrepresents spent cells as live. Any RPC that queries live cells (e.g., `get_cells`) returns already-spent outputs as unspent. The corruption is persistent and affects all callers of that node's indexer.

## Likelihood Explanation
Triggerable by any unprivileged user who submits a standard valid CKB transaction with two or more inputs where the first input's cell is excluded by `cell_filter` (e.g., a lock-only cell) and at least one later input's cell is indexed (e.g., a type-script cell). No miner access, no special network position, and no victim mistake is required. This is a routine transaction pattern.

## Recommendation
Replace `break` with `continue` so that filtered-out inputs are skipped without aborting the loop:
```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // was: break
}
```

## Proof of Concept
1. Configure rich-indexer with a `cell_filter` that only indexes cells with a type script.
2. Create Cell A (lock-only, no type script — filtered out) and Cell B (has type script — indexed).
3. Submit a transaction: `inputs = [Cell A, Cell B]`, spending both.
4. Wait for the block to be indexed.
5. Query the `output` table for Cell B: `is_spent = 0`.
6. Call `get_cells` for Cell B's script: it is returned as a live cell despite being spent on-chain.

```rust
pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
```
