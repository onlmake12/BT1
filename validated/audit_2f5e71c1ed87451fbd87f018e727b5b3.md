### Title
Unbounded `firstUnfulfilledSeq` Scan Loop in `executeCallback` Enables Gas-Exhaustion DoS, Permanently Locking Victim Funds — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the `executeCallback` function contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` by scanning forward through all cleared (inactive) sequence numbers. An unprivileged attacker can submit a large number of requests after a victim's request, fulfill all of them (making them inactive), and leave `firstUnfulfilledSeq` pinned at the victim's sequence number. When the provider subsequently attempts to fulfill the victim's request, the while loop must scan through every one of the attacker's inactive sequence numbers, consuming enough gas to cause an out-of-gas revert. Because the entire transaction reverts, the victim's request remains permanently unfulfillable and their prepaid fee is permanently locked in the contract.

---

### Finding Description

**Root cause — unbounded loop in `executeCallback`:**

After clearing a fulfilled request, `executeCallback` runs:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

There is no cap on the number of iterations. Each iteration calls `findRequest`, which performs a `keccak256` and at least one cold `SLOAD` (~2,100 gas). The developers themselves acknowledge the problem in a TODO comment immediately above the loop:

> "I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. a better solution would be a doubly-linked list of active requests." [2](#0-1) 

**Attack flow:**

1. **Victim** calls `requestPriceUpdatesWithCallback`, receiving sequence number `S`. `firstUnfulfilledSeq` is at `S-1` (or 0 initially).
2. **Attacker** calls `requestPriceUpdatesWithCallback` K times, receiving sequence numbers `S+1` through `S+K`. Any registered provider is accepted; the attacker pays the required fee per request. [3](#0-2) 
3. **Attacker** calls `executeCallback` for each of their K requests (Pyth price update data is publicly available). Each call clears the attacker's request. The while loop tries to advance `firstUnfulfilledSeq` but stops immediately at `S` because the victim's request is still active. After all K fulfillments, `firstUnfulfilledSeq = S` and sequence numbers `S+1` through `S+K` are all inactive.
4. **Provider** calls `executeCallback` for the victim's request at sequence number `S`:
   - `clearRequest(S)` marks it inactive.
   - The while loop starts at `firstUnfulfilledSeq = S`. Sequence `S` is now inactive → advance. Sequence `S+1` is inactive → advance. … Sequence `S+K` is inactive → advance. This is **K iterations**, each costing ~2,100 gas (cold SLOAD).
   - With K ≈ 14,000, the loop alone consumes ~29.4M gas, exceeding the 30M block gas limit.
   - The entire transaction reverts. `clearRequest(S)` is undone; the victim's request is restored as active.
5. Every subsequent attempt to fulfill the victim's request hits the same loop and reverts. The victim's prepaid fee is permanently locked. [4](#0-3) 

**Why there is no escape:** There is no user-callable cancellation function. The victim cannot reclaim their fee. The `requestsOverflow` mapping is used for hash collisions, not for this scenario. [5](#0-4) 

The developers explicitly flag the fund-lock risk in a TODO comment inside `executeCallback`:

> "if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract." [6](#0-5) 

---

### Impact Explanation

- **Permanent fund lock**: The victim's prepaid fee (stored in `req.fee`) can never be recovered because `executeCallback` always reverts with out-of-gas and there is no cancellation path.
- **Callback denial**: The victim's consumer contract never receives its price update callback, breaking any downstream protocol logic that depends on it.
- **Severity**: High — direct, irreversible loss of user funds triggered by an unprivileged attacker paying only the Echo request fees.

---

### Likelihood Explanation

- `requestPriceUpdatesWithCallback` is fully permissionless; any address can submit requests for any registered provider. [7](#0-6) 
- `executeCallback` is also permissionless after the exclusivity period; the attacker can fulfill their own requests without any special role. [8](#0-7) 
- Pyth price update data is publicly available, so the attacker can always supply valid `updateData`.
- The only cost is K × (minimum Echo fee). With a low provider fee configuration, this is economically feasible for a motivated attacker targeting a high-value victim request.
- The `publishTime <= block.timestamp + 60` guard does not prevent this attack; the attacker simply uses `publishTime = block.timestamp`. [9](#0-8) 

---

### Recommendation

Cap the number of iterations in the `firstUnfulfilledSeq` advancement loop, or replace the linear scan with a data structure (e.g., a doubly-linked list of active requests, as the TODO comment already suggests) that allows O(1) advancement. A simple fix is to add a maximum step count:

```solidity
uint256 maxSteps = 1000; // tunable
while (
    maxSteps-- > 0 &&
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

Additionally, add a user-callable `cancelRequest` function so that users can reclaim their fee if their request becomes permanently stuck.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

// Assume `echo` is a deployed Echo contract, `provider` is a registered provider,
// `victim` and `attacker` are EOAs, and `validUpdateData`/`priceIds` are valid Pyth data.

// Step 1: Victim submits request (seq = 1)
uint64 victimSeq = echo.requestPriceUpdatesWithCallback{value: fee}(
    provider, block.timestamp, priceIds, callbackGasLimit
);
// firstUnfulfilledSeq = 0, victimSeq = 1

// Step 2: Attacker submits K = 15000 requests (seq = 2 .. 15001)
for (uint i = 0; i < 15000; i++) {
    echo.requestPriceUpdatesWithCallback{value: minFee}(
        provider, block.timestamp, priceIds, 0
    );
}

// Step 3: Attacker fulfills all their own requests
// Each call: loop stops at seq=1 (victim still active), firstUnfulfilledSeq stays at 1
for (uint64 seq = 2; seq <= 15001; seq++) {
    echo.executeCallback(provider, seq, validUpdateData, priceIds);
}
// Now: firstUnfulfilledSeq = 1, seq 2..15001 all inactive

// Step 4: Provider attempts to fulfill victim's request — OUT OF GAS
// The while loop scans 15000 inactive slots → ~31.5M gas → reverts
echo.executeCallback{gas: 30_000_000}(provider, victimSeq, validUpdateData, priceIds);
// Reverts. Victim's fee is permanently locked. Request remains active but unfulfillable.
```
