### Title
Stale Price Feed Masking via MAX Timestamp in `getPricesNoOlderThan` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `Scheduler` contract's `getPricesNoOlderThan()` function performs its staleness check against a single `priceLastUpdatedAt` value that is set to the **maximum** `publishTime` across all feeds in a subscription batch. When a subscription contains feeds from both active markets (e.g., crypto) and closed markets (e.g., equities), the MAX timestamp from the fresh feed masks the staleness of the closed-market feed. A consuming protocol calling `getPricesNoOlderThan()` receives a false freshness guarantee: the check passes, but individual prices in the returned array may be hours old.

### Finding Description

In `_validateShouldUpdatePrices()`, the Scheduler selects the **maximum** `publishTime` across all feeds as the canonical `updateTimestamp`:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [1](#0-0) 

This `updateTimestamp` is then stored as `status.priceLastUpdatedAt`:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

The `getPricesNoOlderThan()` function checks staleness against this single MAX value:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [3](#0-2) 

The design rationale is documented in the code comments:

> "Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period." [4](#0-3) 

The `parsePriceFeedUpdatesWithConfig` call uses `minValidTime = 0`, explicitly allowing arbitrarily old per-feed timestamps through: [5](#0-4) 

The test `testUpdatePriceFeedsSucceedsWithStaleFeedIfLatestIsValid` explicitly confirms this behavior — an update batch containing one feed with a timestamp 1 hour 5 minutes old and one feed 30 minutes old is accepted, and `priceLastUpdatedAt` is set to the 30-minute timestamp: [6](#0-5) 

### Impact Explanation

A protocol consuming a multi-asset subscription (e.g., ETH/USD + AAPL/USD) calls `getPricesNoOlderThan(subscriptionId, [], 60)` expecting all returned prices to be no older than 60 seconds. If ETH/USD was updated 10 seconds ago and AAPL/USD was last updated 4 hours ago (equity market closed), the staleness check passes (`distance(now, 10s) = 10 < 60`), but the returned array contains AAPL/USD at a 4-hour-old price. The protocol has no way to detect this from the API alone — the function's name and signature imply a per-feed freshness guarantee that is not enforced.

This directly mirrors the `ProductApi3ReaderProxyV1` pattern: a derived/aggregated timestamp (`MAX` instead of `block.timestamp`) masks the true age of underlying data.

### Likelihood Explanation

Mixed-market subscriptions (crypto + equities, or crypto + FX during weekends) are a realistic and documented use case for the Scheduler. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is 1 hour, so a closed-market feed can be up to 1 hour stale before the MAX-timestamp check would reject the batch — but only if the freshest feed in the batch is also older than 1 hour. In practice, a crypto feed will always be fresh, allowing equity feeds to be arbitrarily stale (up to days) while the batch is accepted. An unprivileged keeper can submit such a batch at any time. [7](#0-6) 

### Recommendation

`getPricesNoOlderThan()` should check staleness per-feed rather than against the subscription-level MAX timestamp. The function should iterate over the returned `PriceFeed` array and verify that each individual `price.publishTime` satisfies the `age_seconds` constraint before returning. Alternatively, the function's documentation should explicitly warn that the staleness guarantee applies only to the most recently updated feed in the subscription, not to all feeds.

### Proof of Concept

1. Create a Scheduler subscription with two price IDs: `ETH/USD` (crypto, always active) and `AAPL/USD` (equity, market closed on weekends).
2. On a Saturday, a keeper calls `updatePriceFeeds()` with a batch where:
   - `ETH/USD.publishTime = block.timestamp - 5` (5 seconds old, fresh)
   - `AAPL/USD.publishTime = block.timestamp - 14400` (4 hours old, last Friday close)
   - Both feeds share the same Pythnet slot (slot check passes)
3. `_validateShouldUpdatePrices` computes `updateTimestamp = MAX(now-5, now-14400) = now-5`. The `minAllowedTimestamp` check passes (5 < 3600). `priceLastUpdatedAt` is set to `now-5`.
4. A whitelisted reader calls `getPricesNoOlderThan(subscriptionId, [], 60)`.
5. The check `distance(block.timestamp, now-5) = 5 < 60` passes.
6. The function returns both prices, including `AAPL/USD` at a 4-hour-old price.
7. The consuming protocol believes both prices are no older than 60 seconds and proceeds with the stale AAPL/USD price.
