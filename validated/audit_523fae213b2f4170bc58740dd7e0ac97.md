Looking at the actual code to verify the binary search behavior, the `start_block_number` advancement, the constant value, and whether any rate limiting exists.

### Title
O(N · log H) DB Read Amplification via `GetLastStateProof` Difficulties Binary Search — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

An unprivileged remote peer can send a single `GetLastStateProof` message with up to 999 `difficulties` entries (with `last_n_blocks = 0`) and cause the server to perform O(N · log H) RocksDB reads — approximately 20,000 reads against a 1 M-block chain — instead of the O(N) reads the protocol limit is designed to bound.

### Finding Description

**Limit check** (`execute`, line 201–205):

```
difficulties.len() + last_n_blocks * 2 <= GET_LAST_STATE_PROOF_LIMIT (= 1000)
```

With `last_n_blocks = 0` the attacker may supply up to 1000 difficulty values. [1](#0-0) [2](#0-1) 

**Per-difficulty binary search** (`get_block_numbers_via_difficulties`, lines 73–103):

For each difficulty value the loop calls `get_first_block_total_difficulty_is_not_less_than(start_block_number, end_block_number, difficulty)`, which performs a full binary search over `[start_block_number, end_block_number)`. [3](#0-2) 

**The `start_block_number = num - 1` optimisation does not prevent O(N · log H)**:

After finding block `num` for difficulty `d[i]`, the loop sets `start_block_number = num - 1` (line 91), not `num`. The next search therefore covers `[num-1, end_block_number)`. [4](#0-3) 

Worst-case construction: set `start_block_number = 0`, `difficulty_boundary` just above the chain tip, `last_n_blocks = 0`, and supply difficulties `[1, 2, 3, …, 999]` against a chain where block `k` has total difficulty `k`. Then:

- difficulty 1 → search `[0, H)`, finds block 1, `start = 0`
- difficulty 2 → search `[0, H)`, finds block 2, `start = 1`
- difficulty 3 → search `[1, H)`, finds block 3, `start = 2`
- …
- difficulty k → search `[k-2, H)`, range ≈ H

Each search costs O(log H) calls to `get_block_total_difficulty`, which itself issues two RocksDB reads (`get_block_hash` + `get_block_ext`). [5](#0-4) [6](#0-5) 

Total DB reads per request ≈ 999 × log₂(1 000 000) × 2 ≈ **40 000 RocksDB reads**.

**No rate limiting** exists for `GetLastStateProof` in the light-client protocol server; the `LightClientProtocol` handler contains no `RateLimiter` whatsoever. [7](#0-6) 

The `difficulties` filter at lines 325–328 (`take_while(|d| *d <= total_difficulty)`) only removes difficulties that exceed the total difficulty at the boundary block; it does not reduce the binary-search cost for the remaining valid difficulties. [8](#0-7) 

### Impact Explanation

A single TCP connection can flood the server with `GetLastStateProof` messages. Each message forces ~40 000 RocksDB reads (2 reads × ~20 binary-search steps × 999 difficulties). With multiple concurrent connections the I/O load scales linearly with connection count, starving the node of disk bandwidth and degrading or halting block processing and sync.

### Likelihood Explanation

The attack requires only a valid P2P connection to a node that has the light-client protocol enabled. No PoW, no key, no privileged role. The crafted message passes all existing validation checks (monotonically increasing difficulties, boundary constraint, start-difficulty constraint). The amplification factor (~40×) is modest per request but unlimited in request rate.

### Recommendation

1. **Bound total binary-search work**: track the cumulative search range across all difficulties and cap it (e.g., at `limit × log₂(chain_height)`), or replace the per-difficulty binary search with a single linear scan that advances a cursor monotonically through the block range.
2. **Add per-peer rate limiting** for `GetLastStateProof` analogous to the rate limiters used in other protocol handlers.
3. **Advance `start_block_number` to `num`, not `num - 1`**: the off-by-one causes the next search to re-examine block `num - 1` unnecessarily and widens every subsequent search range by one.

### Proof of Concept

```
1. Connect to a CKB full node with light-client protocol enabled.
2. Obtain a valid tip hash (last_hash) and start_hash = genesis hash.
3. Set last_n_blocks = 0, start_number = 0.
4. Set difficulty_boundary = chain_tip_total_difficulty + 1.
5. Set difficulties = [1, 2, 3, …, 999]  (all < difficulty_boundary, monotonically increasing,
   all reachable on a chain where block k has cumulative difficulty k).
6. Send GetLastStateProof.
7. Instrument get_block_total_difficulty call count on the server.
8. Assert call count ≈ 999 × log₂(chain_height) >> 999.
   For a 1 M-block chain: ~20 000 calls, each issuing 2 DB reads → ~40 000 RocksDB ops.
9. Repeat in a tight loop from one or more connections to exhaust server I/O.
```

impl CKBProtocolHandler for LightClientProtocol {
    async fn init(&mut self, _nc: Arc<dyn CKBProtocolContext + Sync>) {}

    async fn connected(
        &mut self,
        _nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        version: &str,
    ) {
        info!("LightClient({}).connected peer={}", version, peer);
    }

    async fn disconnected(&mut self, _nc: Arc<dyn CKBProtocolContext + Sync>, peer: PeerIndex) {
        info!("LightClient.disconnected peer={}", peer);
    }

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
}
