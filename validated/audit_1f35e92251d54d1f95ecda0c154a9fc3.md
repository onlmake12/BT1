### Title
Unprivileged Relayer Can Permanently Skip Governance VAAs via Non-Contiguous Sequence Execution — (`governance/remote_executor/programs/remote-executor/src/lib.rs`)

---

### Summary

The `execute_posted_vaa` instruction in the `remote-executor` program enforces only a strictly-greater-than sequence check (`posted_vaa.sequence > claim_record.sequence`) with no contiguity requirement and no access control on the caller. Any unprivileged account can call `execute_posted_vaa` with a higher-sequence governance VAA, advancing `ClaimRecord.sequence` past lower-sequence VAAs and permanently rendering those lower-sequence governance actions unexecutable.

---

### Finding Description

The `ExecutePostedVaa` account constraint in `lib.rs` is:

```
constraint = posted_vaa.sequence > claim_record.sequence
    @ ExecutorError::NonIncreasingSequence
``` [1](#0-0) 

After passing this check, the program immediately writes the executed sequence back:

```rust
claim_record.sequence = posted_vaa.sequence;
``` [2](#0-1) 

The `ClaimRecord` is a single PDA per emitter, seeded by `[CLAIM_RECORD_SEED, emitter_address]`: [3](#0-2) 

The `payer` field is only required to be a `Signer` — there is no authority check, no allowlist, and no restriction on who may submit a VAA: [4](#0-3) 

The `posted_vaa` account must be owned by the Wormhole bridge program (enforced via the `Owner` trait), so the attacker cannot forge VAAs — but all legitimate governance VAAs are already publicly posted to Wormhole and available to anyone: [5](#0-4) 

The existing test suite explicitly demonstrates and accepts this non-contiguous behavior: sequence 3 is executed after sequence 1 (skipping 2), and then sequence 2 is permanently blocked with `NonIncreasingSequence`: [6](#0-5) 

The legitimate relayer (`crank_pythnet_relayer`) processes VAAs strictly in ascending order starting from `lastSequenceNumber + 1`: [7](#0-6) 

An adversary who races ahead of this relayer by submitting a higher-sequence VAA first will permanently advance `ClaimRecord.sequence`, making all lower-sequence VAAs fail with `NonIncreasingSequence` forever.

---

### Impact Explanation

Any governance action encoded in a skipped VAA — data source updates, fee changes, contract upgrades, or any other `ExecutorPayload` instruction — is permanently unexecutable on Pythnet. The governance multisig voted on and signed those actions, but they can never take effect. This directly satisfies the scope criterion of "governance voting result manipulation."

---

### Likelihood Explanation

The attack requires only:
1. Monitoring Wormhole for governance VAAs from the Pyth emitter (public data).
2. Calling `execute_posted_vaa` with a higher-sequence VAA before the legitimate relayer does (a simple race condition requiring only SOL for fees).

No privileged access, no key compromise, and no Wormhole guardian collusion is needed. The attacker only needs to be faster than the `crank_pythnet_relayer` for a single transaction.

---

### Recommendation

Replace the strictly-greater-than check with a strict equality check requiring contiguous execution:

```rust
constraint = posted_vaa.sequence == claim_record.sequence + 1
    @ ExecutorError::NonContiguousSequence
```

If non-contiguous execution is intentionally required (e.g., to skip non-Pythnet VAAs), then restrict the caller to a privileged, trusted relayer address via an additional `constraint` or a stored authority in the `ClaimRecord`. This ensures only the authorized relayer can decide which sequences to skip.

---

### Proof of Concept

State-based test (extends the existing test harness):

1. Create three governance VAAs for the same emitter at sequences 1, 2, 3.
2. Call `execute_posted_vaa` with sequence 3 first (skipping 1 and 2). Assert success and `claim_record.sequence == 3`.
3. Attempt `execute_posted_vaa` with sequence 1. Assert it fails with `NonIncreasingSequence`.
4. Attempt `execute_posted_vaa` with sequence 2. Assert it fails with `NonIncreasingSequence`.
5. Confirm the instructions encoded in VAAs 1 and 2 were never executed (e.g., target accounts remain unmodified).

This is directly analogous to the existing test at lines 99–134 of `test_basic_instructions.rs`, which already proves the permanent-blocking behavior. [6](#0-5)
