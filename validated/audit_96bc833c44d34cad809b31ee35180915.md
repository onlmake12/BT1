### Title
Silent `u128 → u64` Truncation in `calculate_maximum_withdraw` Corrupts DAO Withdrawal Amount — (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the DAO interest-bearing withdrawal amount using `u128` arithmetic to avoid intermediate overflow, but then silently truncates the final result back to `u64` with an unchecked `as u64` cast. Every other analogous `u128 → u64` narrowing in the same file uses the safe `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern. The inconsistency means that if the product `counted_capacity × withdrawing_ar / deposit_ar` exceeds `u64::MAX`, the returned capacity is silently wrong — a DAO depositor receives a drastically smaller amount than they are owed.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the withdrawable capacity as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← unsafe truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The three other `u128 → u64` narrowings in the same file all use the safe checked conversion:

```rust
// secondary_block_reward
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch – miner issuance
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)

// dao_field_with_current_epoch – ar_increase
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

The `ar` (accumulate rate) field is a `u64` stored in the DAO header field, starting at `DEFAULT_GENESIS_ACCUMULATE_RATE = 10_000_000_000_000_000` (10¹⁶) and growing monotonically. [3](#0-2) 

The overflow condition is:

```
counted_capacity  ×  withdrawing_ar  /  deposit_ar  >  u64::MAX
```

- `counted_capacity` ≤ `u64::MAX` ≈ 1.8 × 10¹⁹ shannons  
- `deposit_ar` ≥ 10¹⁶ (genesis floor)  
- `withdrawing_ar` ≤ `u64::MAX` ≈ 1.8 × 10¹⁹  
- Maximum ratio: `withdrawing_ar / deposit_ar` ≤ ~1 800  
- Maximum product before division: `u64::MAX × u64::MAX` ≈ 3.24 × 10³⁸ — fits in `u128`  
- After division: up to ~1 800 × `u64::MAX` ≈ 3.24 × 10²² — **exceeds `u64::MAX`**

When the product exceeds `u64::MAX`, `withdraw_counted_capacity as u64` silently wraps, producing a value far smaller than the true result. The depositor's `withdraw_capacity` is then computed from this corrupted value, causing them to receive far less than they deposited plus interest.

The `calculate_maximum_withdraw` function is called both during DAO withdrawal verification in `transaction_maximum_withdraw` and directly from the RPC/SDK layer for any user querying their maximum withdrawal. [4](#0-3) 

---

### Impact Explanation

A NervosDAO depositor who withdraws a cell whose `counted_capacity × (withdrawing_ar / deposit_ar)` exceeds `u64::MAX` receives a silently truncated (much smaller) capacity. The depositor loses the difference — potentially their entire principal plus interest — with no error raised. Because `Capacity::shannons` accepts the truncated value without complaint and `safe_add` succeeds normally, the chain accepts the malformed withdrawal as valid.

---

### Likelihood Explanation

For `ar` to grow from 10¹⁶ to a value that causes overflow with a maximum-capacity cell, it must increase by a factor of ~1 800. The observed growth rate is extremely slow (block 5 892 shows `ar ≈ 10¹⁶ + 6 × 10¹¹`, a ~0.006% increase over ~5 900 blocks). [5](#0-4) 

In practice this overflow is not reachable on the current mainnet within any foreseeable timeframe. The finding is valid as a **latent code defect** — the inconsistency with every other narrowing in the same file is a clear oversight — but the realistic likelihood of exploitation is very low.

---

### Recommendation

Replace the silent cast with the same checked pattern used everywhere else in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64)

// After (safe, consistent with the rest of the file):
Capacity::shannons(
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
)
``` [6](#0-5) 

---

### Proof of Concept

The overflow path is reached whenever a DAO cell is withdrawn and the following holds:

```
counted_capacity * withdrawing_ar / deposit_ar > u64::MAX
```

Concretely, with:
- `counted_capacity = u64::MAX` (maximum cell capacity minus occupied bytes)
- `deposit_ar = 10^16` (genesis accumulate rate)
- `withdrawing_ar = 1_800 * 10^16 + 1` (hypothetical far-future value)

```
withdraw_counted_capacity (u128) = u64::MAX * (1_800 * 10^16 + 1) / 10^16
                                 ≈ 1_800 * u64::MAX
                                 ≈ 3.24 × 10^22   (fits in u128)

withdraw_counted_capacity as u64 ≈ 3.24 × 10^22 mod 2^64
                                 ≈ some small value << u64::MAX
```

The depositor's `withdraw_capacity` is computed from this small truncated value, causing a massive loss of funds with no error returned to the caller.
