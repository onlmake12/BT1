### Title
Zero-Length Epoch in `since` Field Causes Panic via `to_rational()` — (`File: util/types/src/core/extras.rs`, `verification/src/transaction_verifier.rs`)

### Summary
An attacker-controlled `since` field in a transaction input can encode an `EpochNumberWithFraction` with `length = 0` and `number > 0`. When the `SinceVerifier` calls `to_rational()` on this value via `from_full_value_unchecked`, it reaches `RationalU256::new(index, 0)`, which unconditionally panics. This is a direct analog to the "zero price used before initialization" class: a zero-valued field (epoch length) is consumed in arithmetic before any guard is applied.

### Finding Description

`EpochNumberWithFraction::to_rational()` contains the following logic: [1](#0-0) 

The guard `if self.0 == 0` only handles the all-zero genesis sentinel. Any value where `number > 0` but `length = 0` passes the guard and reaches:

```rust
RationalU256::new(self.index().into(), self.length().into())
```

`RationalU256::new` unconditionally panics when `denom` is zero: [2](#0-1) 

The codebase itself documents this risk in the `EpochNumberWithFraction` type comment: [3](#0-2) 

And in `from_full_value_unchecked`: [4](#0-3) 

The safe variant `from_full_value` calls `normalize()` to fix zero-length epochs, but `from_full_value_unchecked` is used directly inside `verification/src/transaction_verifier.rs`: [5](#0-4) 

The `since` field in a transaction input is a raw `u64` fully controlled by the transaction sender. The epoch encoding packs `length` into bits 40–55, `index` into bits 24–39, and `number` into bits 0–23: [6](#0-5) 

An attacker sets `length = 0`, `number = 1`, `index = 0` → packed value `= (1 << 0) = 1`, which is non-zero, bypasses the genesis guard, and triggers the panic.

The `is_well_formed` check exists but is not enforced before `to_rational()` is called in the verifier path: [7](#0-6) 

### Impact Explanation

A Rust integer-division-by-zero or `RationalU256::new` panic with `denom = 0` unwinds the verifying thread. If the panic is not caught with `catch_unwind` at the tx-pool or sync boundary, the node process crashes. Even if caught, the panic propagates an unhandled error path that can corrupt in-flight verification state. Any unprivileged transaction sender or relay peer can trigger this by submitting a single crafted transaction with a malformed `since` epoch field, causing a remote denial-of-service against any node that attempts to verify it.

### Likelihood Explanation

The `since` field is a raw u64 accepted from any RPC caller (`send_transaction`) or relay peer. No wire-level or pre-verification format check enforces `length > 0` on the epoch sub-field before `to_rational()` is reached. The crafted value is trivially constructable (set bits 0–23 to any non-zero epoch number, leave bits 40–55 at zero). Likelihood is high for any node exposed to untrusted transactions.

### Recommendation

Add a `is_well_formed()` guard before calling `to_rational()` on any `EpochNumberWithFraction` derived from external input (e.g., the `since` field). Alternatively, replace `from_full_value_unchecked` with `from_full_value` (which calls `normalize()`) in all verifier paths that process attacker-supplied data. A minimal fix:

```rust
// Before calling to_rational() on a since-derived epoch:
if !epoch.is_well_formed() {
    return Err(TransactionError::InvalidSince { index }.into());
}
```

### Proof of Concept

1. Construct a transaction input with `since` encoding: epoch `number=1, index=0, length=0`.
   - Packed value: `(0 << 40) | (0 << 24) | 1 = 0x0000000000000001`
   - Set the since type flag bits to indicate epoch-based since.
2. Submit via `send_transaction` RPC or relay to a peer.
3. The node's `SinceVerifier` calls `from_full_value_unchecked(since_value)` → `EpochNumberWithFraction(1)`.
4. `to_rational()` is called: `self.0 = 1 ≠ 0`, so the genesis guard is skipped.
5. `RationalU256::new(U256::from(0u64), U256::from(0u64))` → **panic: "denominator == 0"**. [8](#0-7) [1](#0-0)
