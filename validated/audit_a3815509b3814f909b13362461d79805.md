### Title
Silent `u64`-to-`i64` Truncation in Rich-Indexer Cell Capacity Storage Corrupts Capacity Range Queries — (`util/rich-indexer/src/indexer/insert.rs`)

---

### Summary

The rich-indexer stores cell capacity (a `u64` CKB type) into a signed `i64` SQL column using a bare `as i64` cast with no bounds check. Any cell whose capacity exceeds `i64::MAX` shannons (~92.2 billion CKB) is silently stored as a negative integer. SQL range comparisons against that column then produce wrong results — cells are either invisible to capacity-range filters or appear in the wrong range — without any error being raised. The pattern is the direct Rust analog of the Solidity `intToUint` issue: a total function that silently produces a semantically wrong value instead of rejecting the input.

---

### Finding Description

In `build_output_cell_rows`, the cell capacity is extracted as `u64` and immediately cast to `i64` with no overflow guard:

```rust
// util/rich-indexer/src/indexer/insert.rs  line 543-546
let cell_capacity: u64 = cell.capacity().into();
let cell_row = (
    output_index as i32,
    cell_capacity as i64,   // ← silent wrapping cast; negative if > i64::MAX
``` [1](#0-0) 

The resulting `i64` value is stored in the `output.capacity` column of the SQL database.

When cells are later retrieved in `build_indexer_cell`, the stored `i64` is cast back to `u64`:

```rust
// get_cells.rs  line 271
.capacity(row.get::<i64, _>("capacity") as u64)
``` [2](#0-1) 

In Rust, `as` casts between same-width integer types are bitwise reinterpretations, so the round-trip `u64 → i64 → u64` is lossless for the returned value. **However**, the SQL engine sees a signed `BIGINT` column. Any cell with capacity > `i64::MAX` is stored as a negative number, and SQL comparisons (`>=`, `<`) treat it as such. This breaks capacity-range filtering.

The `build_cell_filter` function inlines the filter bounds directly into the SQL `WHERE` clause:

```rust
// mod.rs  lines 214-216
if let Some(capacity_range) = &filter.output_capacity_range {
    query_builder.and_where_ge("output.capacity", capacity_range.start());
    query_builder.and_where_lt("output.capacity", capacity_range.end());
}
``` [3](#0-2) 

A cell stored as a negative `i64` will never satisfy `capacity >= 0`, so it disappears from all capacity-range queries. The `convert_max_values_in_search_filter` helper only clamps `u64::MAX` sentinel values to `i64::MAX`; it does not handle the general case of any filter bound or stored value exceeding `i64::MAX`:

```rust
// mod.rs  lines 307-316
let start = if (i64::MAX as u64) < u64::from(r.start()) {
    i64::MAX as u64
} else {
    r.start().into()
};
``` [4](#0-3) 

The same unchecked `as i64` cast also appears for `block_number` and `timestamp` in `block_view_to_field_values`:

```rust
// insert.rs  lines 728, 732
(block_view.number() as i64).into(),
...
(block_view.timestamp() as i64).into(),
``` [5](#0-4) 

---

### Impact Explanation

An RPC caller using `get_cells`, `get_transactions`, or `get_cells_capacity` with an `output_capacity_range` filter would silently receive incomplete or incorrect results for any cell whose capacity exceeds `i64::MAX` shannons. The `get_cells_capacity` aggregate sum would also be wrong because the SQL `SUM()` operates on the stored signed values. No error is returned; the caller has no indication that data is missing or wrong.

---

### Likelihood Explanation

**Low.** The current total CKB issuance is approximately 33.6 billion CKB at genesis plus secondary issuance, totalling well under `i64::MAX` shannons (~92.2 billion CKB). Because CKB consensus enforces that output capacity ≤ input capacity, no individual cell can hold more than the total circulating supply. Under current supply parameters the threshold is unreachable. The risk is latent: the pattern is error-prone and the absence of a guard means any future supply change or miscalculation could silently corrupt indexer query results without any protocol-level rejection.

---

### Recommendation

**Short term:** Replace the bare `as i64` cast with a checked conversion that returns an error if the value exceeds `i64::MAX`:

```rust
let cell_capacity: u64 = cell.capacity().into();
let cell_capacity_i64 = i64::try_from(cell_capacity)
    .map_err(|_| Error::DB("cell capacity exceeds i64::MAX".to_string()))?;
```

Apply the same fix to `block_number` and `timestamp` casts in `block_view_to_field_values`.

**Long term:** Consider storing capacity as a `NUMERIC`/`DECIMAL` or as a big-endian byte blob to avoid the signed/unsigned impedance mismatch between Rust's `u64` and SQL's `BIGINT`, or document and enforce an invariant that capacity will never exceed `i64::MAX` with an explicit assertion at the insertion site.

---

### Proof of Concept

1. Construct a `CellOutput` with `capacity = i64::MAX as u64 + 1` (= `9_223_372_036_854_775_808` shannons).
2. Call `build_output_cell_rows` on it; observe `cell_capacity as i64` yields `i64::MIN` (`-9_223_372_036_854_775_808`).
3. Insert this row into the rich-indexer database.
4. Issue a `get_cells` RPC call with `output_capacity_range: ["0x0", "0x8000000000000001"]` (i.e., `[0, i64::MAX+1]`).
5. The SQL condition `output.capacity >= 0 AND output.capacity < 9223372036854775808` evaluates against the stored `-9223372036854775808`, which fails both predicates — the cell is silently absent from the result, with no error returned to the caller.
