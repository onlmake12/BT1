### Title
Osmosis Fee Sufficiency Check Assumes 1:1 Token Value Equivalence Without Price Conversion - (File: `target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

### Summary
The Osmosis-specific `is_fee_sufficient` function in the Pyth CosmWasm contract accepts any Osmosis-allowed fee token at a 1:1 raw unit ratio against the base denom (`uosmo`), without converting to equivalent value via a spot price oracle. This mirrors the exact vulnerability class of the reported issue: a hardcoded 1:1 price assumption between tokens of different market values.

### Finding Description
In `target_chains/cosmwasm/contracts/pyth/src/contract.rs`, the `#[cfg(feature = "osmosis")]` build of `is_fee_sufficient` (lines 163–197) sums raw token amounts across all submitted coins — regardless of denom — and compares the raw sum against the required base-denom fee amount:

```rust
let mut total_amount = 0u128;
for coin in &info.funds {
    if coin.denom != state.fee.denom && !is_allowed_tx_fees_denom(deps, &coin.denom) {
        return Err(PythContractError::InvalidFeeDenom { ... })?;
    }
    total_amount = total_amount.checked_add(coin.amount.u128())...;
}
let base_denom_fee = get_update_fee(deps, data)?;
Ok(base_denom_fee.amount.u128() <= total_amount)
```

The code itself acknowledges the missing conversion with an explicit TODO comment:

> *"NOTE: the base fee denom right now is = denom: 'uosmo', amount: 1, which is almost negligible. It's not important to convert the price right now. For now we are keeping the base fee amount same for each valid denom -> 1 but this logic will be updated to use spot price for different valid tokens in future"*

The same assumption is present in `get_update_fee_for_denom` (lines 717–735):

> *"for other valid denoms too we are using the base amount as 1 … this will be change later on to add custom logic using spot price for valid tokens"* [1](#0-0) [2](#0-1) 

### Impact Explanation
Any caller of `update_price_feeds` on the Osmosis mainnet deployment can pay the required fee using an Osmosis-allowed alternative token (e.g., `uion`) at a 1:1 raw unit ratio against `uosmo`. If the alternative token's market price is lower than `uosmo`, the caller pays less USD value than the protocol intends to collect. The protocol receives less economic value than configured. If governance ever raises the fee to a non-negligible amount (currently 1 uosmo ≈ $0.000001), the underpayment gap scales proportionally with the price divergence between `uosmo` and the alternative token.

**Impact: Medium** — fee accounting is incorrect; protocol collects less value than intended when alternative tokens are used.

### Likelihood Explanation
The Osmosis chain is a live mainnet deployment (`"mainnet": true` in `contract_manager/src/store/chains/CosmWasmChains.json`). Any unprivileged user submitting a price update VAA can exploit this by simply sending an allowed alternative token instead of `uosmo`. No special access or key is required. The entry path is the standard `update_price_feeds` execute message. [3](#0-2) 

**Likelihood: Medium** — trivially reachable by any transaction sender; exploitability scales with fee size and token price divergence.

### Recommendation
Replace the raw-amount comparison with a spot-price-converted comparison. The code already outlines the correct approach in its own TODO comment: query the Osmosis `TxFees` module (or an on-chain TWAP/spot price source) to convert each submitted coin's amount into `uosmo`-equivalent units before summing and comparing against `base_denom_fee`. Until that conversion is implemented, the contract should restrict fee payment to the base denom only on Osmosis, removing the false equivalence.

### Proof of Concept
1. Governance sets the Pyth update fee to `N uosmo` (any non-zero value).
2. Attacker calls `update_price_feeds` on the Osmosis Pyth contract, sending `N uion` (ION token) as `msg.value` instead of `N uosmo`.
3. `is_allowed_tx_fees_denom` returns `true` for `uion` (it has a valid Osmosis pool ID).
4. `total_amount` accumulates `N` raw units of `uion`.
5. `base_denom_fee.amount.u128()` equals `N`.
6. The check `N <= N` passes — fee is accepted.
7. If `uion` trades at a lower USD price than `uosmo`, the attacker has paid less USD value than required, and the protocol has under-collected fees. [4](#0-3)
