### Title
Cache Key Mismatch in `readd_detached_tx` Causes Guaranteed Full Re-verification of All Detached Transactions on Every Reorg — (`tx-pool/src/process.rs`)

### Summary

`fetch_txs_verify_cache` builds a `HashMap` keyed by `witness_hash`, but `readd_detached_tx` looks up that map using `tx_hash`. Since `tx_hash ≠ witness_hash` for any transaction carrying witnesses, the lookup always returns `None`. This forces `verify_rtx` to call `block_in_place(ContextualTransactionVerifier::verify)` for every detached transaction while holding the `tx_pool.write()` lock, blocking all tx-pool operations for the full duration of reorg processing.

### Finding Description

**Step 1 — Cache is populated with `witness_hash` keys.**

`fetch_txs_verify_cache` (called at line 828 of `process.rs`) builds the returned `HashMap<Byte32, CacheEntry>` keyed by `wtx_hash = tx.witness_hash()`: [1](#0-0) 

The underlying `TxVerificationCache` is also keyed by `witness_hash` — confirmed by how `_process_tx` writes to it (`guard.put(wtx_hash, verified)` at line 763) and how `fetch_tx_verify_cache` reads it (`guard.peek(&tx.witness_hash())` at line 78). [2](#0-1) 

**Step 2 — `readd_detached_tx` looks up the map with `tx_hash`.**

At line 887, `tx_hash = tx.hash()` is obtained (the hash of `RawTransaction`, without witnesses). At line 891, the lookup is `fetched_cache.get(&tx_hash)`: [3](#0-2) 

`tx_hash` and `witness_hash` are computed over different data: [4](#0-3) 

For any transaction with non-empty witnesses (all real CKB transactions use witnesses for lock script authorization), `tx_hash ≠ witness_hash`. The lookup always returns `None`.

**Step 3 — `None` cache entry forces `block_in_place(ContextualTransactionVerifier::verify)`.**

`verify_rtx` is called with `command_rx = None` and `cache_entry = &None`. It falls into the `else` branch: [5](#0-4) 

`block_in_place` blocks the current tokio worker thread for the full duration of script execution, up to `max_tx_verify_cycles` per transaction.

**Step 4 — The `tx_pool.write()` lock is held for the entire loop.**

`readd_detached_tx` is called while `tx_pool.write()` is held (lines 836–851). The sequential loop over all detached transactions means the write lock is held for `N_txs × verify_time`: [6](#0-5) 

During this window, all tx-pool operations — RPC (`get_transaction`, `send_transaction`), block template generation, and new transaction admission — are blocked.

### Impact Explanation

- Every reorg (including natural 1-block reorgs) triggers full re-verification of all detached transactions from scratch, with zero cache benefit.
- An attacker who mines even a single block containing transactions at `max_tx_verify_cycles` (default `TWO_IN_TWO_OUT_CYCLES × 1000 ≈ 3.5 × 10⁹` cycles per block) forces honest nodes to spend the maximum possible CPU time re-verifying those transactions during the reorg, while holding the tx-pool write lock.
- P2P relay, RPC, and block template generation are all starved for the duration. [7](#0-6) 

### Likelihood Explanation

The bug is triggered on **every** reorg, not only attacker-induced ones. Any miner — even one with a small fraction of hashpower — can mine a block with high-cycle transactions and release it to cause a 1-block reorg. No majority hashpower is required. The key mismatch is unconditional: there is no code path in `readd_detached_tx` that uses `witness_hash` for the lookup.

### Recommendation

Change line 891 of `tx-pool/src/process.rs` to look up by `witness_hash` instead of `tx_hash`:

```rust
// Before (wrong key):
let verify_cache = fetched_cache.get(&tx_hash).cloned();

// After (correct key):
let verify_cache = fetched_cache.get(&tx.witness_hash()).cloned();
```

The `tx` variable is still in scope at that point (it was moved into `resolve_tx` but the `TransactionView` is `Clone`-able, or `witness_hash` can be computed before the move). Alternatively, compute and store `witness_hash` alongside `tx_hash` at the top of the loop body.

Additionally, the comment at line 848 ("readd_detached_tx don't update cache") should be revisited — after the fix, the cache should also be updated post-verification to avoid re-verification on subsequent reorgs.

    async fn readd_detached_tx(
        &self,
        tx_pool: &mut TxPool,
        txs: Vec<TransactionView>,
        fetched_cache: HashMap<Byte32, CacheEntry>,
    ) {
        let max_cycles = self.tx_pool_config.max_tx_verify_cycles;
        for tx in txs {
            let tx_size = tx.data().serialized_size_in_block();
            let tx_hash = tx.hash();
            if let Ok((rtx, status)) = resolve_tx(tx_pool, tx_pool.snapshot(), tx, false)
                && let Ok(fee) = check_tx_fee(tx_pool, tx_pool.snapshot(), &rtx, tx_size)
            {
                let verify_cache = fetched_cache.get(&tx_hash).cloned();
                let snapshot = tx_pool.cloned_snapshot();
                let tip_header = snapshot.tip_header();
                let tx_env = Arc::new(status.with_env(tip_header));
                if let Ok(verified) = verify_rtx(
                    snapshot,
                    Arc::clone(&rtx),
                    tx_env,
                    &verify_cache,
                    max_cycles,
                    None,
                )
                .await
                {
                    let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
                    if let Err(e) = _submit_entry(tx_pool, status, entry, &self.callbacks) {
                        error!("readd_detached_tx submit_entry {} error {}", tx_hash, e);
                    } else {
                        debug!("readd_detached_tx submit_entry {}", tx_hash);
                    }
                }
            }
        }
    }
