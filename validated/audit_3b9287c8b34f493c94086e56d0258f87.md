### Title
Missing Timeout on Miner RPC HTTP Client Causes Indefinite Blocking of Mining Loop - (File: miner/src/client.rs)

### Summary

The `Rpc` HTTP client in the CKB miner is constructed without any request timeout. When the CKB node's RPC endpoint becomes unresponsive (e.g., due to node overload, a bug, or network stall), the miner's HTTP requests hang indefinitely. Because `blocking_fetch_block_template()` is called synchronously on the miner's main loop thread after every solved block, a single hung request permanently freezes the mining loop, preventing block submission and stopping all mining activity.

### Finding Description

In `miner/src/client.rs`, `Rpc::new` constructs a `hyper_util` legacy HTTP client with no timeout configured:

```rust
let https = hyper_tls::HttpsConnector::new();
let client = HttpClient::builder(TokioExecutor::new()).build::<_, Full<Bytes>>(https);
``` [1](#0-0) 

Each RPC call spawns a task that awaits `client.request(req)` and then `BodyExt::collect(body)` with no `tokio::time::timeout` wrapper:

```rust
loop_handle.spawn(async move {
    let request = match client
        .request(req)
        .await
        .map(|res| res.into_body())
    {
        Ok(body) => BodyExt::collect(body).await.map_err(RpcError::Http).map(|t| t.to_bytes()),
        Err(err) => Err(RpcError::Client(err)),
    };
    if sender.send(request).is_err() { ... }
});
``` [2](#0-1) 

The `Rpc::request` future then awaits the `oneshot::Receiver` `rev` indefinitely:

```rust
rev.map_err(|_| RpcError::Canceled)
    .await?
    .and_then(|chunk| serde_json::from_slice(&chunk).map_err(RpcError::Json))
``` [3](#0-2) 

If the HTTP request stalls (TCP connection established but no response), the spawned task never drops its `oneshot::Sender`, so `rev.await` never resolves.

The critical call site is in `miner/src/miner.rs`, inside `submit_nonce`, which calls `blocking_fetch_block_template()` synchronously on the miner's main thread after every found block:

```rust
self.client.blocking_fetch_block_template();
self.notify_workers(WorkerMessage::Start);
``` [4](#0-3) 

`blocking_fetch_block_template` calls `self.handle.block_on(self.fetch_block_template())`: [5](#0-4) 

`block_on` blocks the calling OS thread until the future resolves. With no timeout, this thread is permanently blocked.

By contrast, the block assembler's outbound HTTP notify calls are correctly wrapped with `tokio::time::timeout`: [6](#0-5) 

The miner's `Rpc` client has no equivalent protection.

### Impact Explanation

When the CKB node RPC becomes unresponsive (due to node overload, a bug, or a network stall between the miner and node), the miner's main loop thread blocks indefinitely at `blocking_fetch_block_template()`. Consequences:
- The miner cannot process new work items from `work_rx`.
- The miner cannot respond to the `stop_rx` shutdown signal.
- Workers are never restarted (`notify_workers(WorkerMessage::Start)` is never reached).
- The miner process must be killed externally to recover.
- Any valid block found during the stall is lost.

The `block_on_submit` path in `submit_block` has the same defect: [7](#0-6) 

### Likelihood Explanation

The CKB node RPC server applies a 30-second `TimeoutLayer` to incoming requests: [8](#0-7) 

However, this only protects the server from slow clients. It does not protect the miner client from a slow or stalled server. A TCP connection that is established but then stalls (e.g., due to a kernel buffer fill, a half-open connection, or a node that accepts the connection but never sends a response) will not be caught by the server-side timeout. This scenario is realistic under node overload or network partition conditions.

### Recommendation

Wrap `client.request(req).await` and `BodyExt::collect(body).await` with `tokio::time::timeout` in the spawned task inside `Rpc::new`, using a configurable or sensible default (e.g., 30 seconds). Alternatively, use `hyper-timeout` or configure a connection/read timeout on the `HttpClient` builder. The `MinerClientConfig` could expose a `rpc_timeout_millis` field analogous to `notify_timeout_millis` in `BlockAssemblerConfig`.

### Proof of Concept

1. Start `ckb-miner` pointed at a CKB node.
2. Configure the CKB node (or a proxy in front of it) to accept TCP connections on the RPC port but never send any response bytes (e.g., using `iptables` to drop outbound packets after the TCP handshake, or a simple netcat listener).
3. The miner finds a nonce, calls `submit_block` (which may or may not hang depending on `block_on_submit`), then calls `blocking_fetch_block_template()`.
4. `blocking_fetch_block_template()` → `block_on(fetch_block_template())` → `get_block_template()` → `Rpc::request()` → `client.request(req).await` stalls indefinitely.
5. The miner's main thread is permanently blocked. No new work is dispatched, no shutdown signal is processed. The miner is effectively dead until killed. [9](#0-8) [10](#0-9)

    pub fn new(url: Uri, handle: Handle) -> Rpc {
        let (sender, mut receiver) = mpsc::channel(65_535);
        let stop_rx: CancellationToken = new_tokio_exit_rx();

        let https = hyper_tls::HttpsConnector::new();
        let client = HttpClient::builder(TokioExecutor::new()).build::<_, Full<Bytes>>(https);
        let loop_handle = handle.clone();
        handle.spawn(async move {
            loop {
                tokio::select! {
                    Some(item) = receiver.recv() => {
                        let (sender, call): RpcRequest = item;
                        let req_url = url.clone();
                        let request_json = serde_json::to_vec(&call).expect("valid rpc call");

                        let mut req = Request::builder().uri(req_url).method("POST").header(CONTENT_TYPE, "application/json");

                        if let Some(value) = parse_authorization(&url) {
                            req = req
                                .header(hyper::header::AUTHORIZATION, value);
                        }
                        let req = req.body(Full::new(Bytes::from(request_json))).unwrap();
                        let client = client.clone();
                        loop_handle.spawn(async move {
                            let request = match client
                                .request(req)
                                .await
                                .map(|res|res.into_body())
                            {
                                Ok(body) => BodyExt::collect(body).await.map_err(RpcError::Http).map(|t| t.to_bytes()),

                            };

                            }

                        break

                }

        });
