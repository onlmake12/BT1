Audit Report

## Title
`consecutive_failed` Never Reset on Successful Inclusion Causes Premature Block-Template Loop Exit — (`File: tx-pool/src/component/tx_selector.rs`)

## Summary
In `TxSelector::txs_to_commit`, the `consecutive_failed` counter is incremented on two failure paths (capacity overflow and non-proposed ancestor) but is never reset to zero after a successful transaction package inclusion. An attacker with the ability to get transactions into the proposed pool with permanently non-proposed ancestors can accumulate 4,001 failures and trigger the `MAX_CONSECUTIVE_FAILURES` early exit, causing the block assembler to silently omit valid high-fee proposed transactions from the block template.

## Finding Description
`txs_to_commit` in `tx-pool/src/component/tx_selector.rs` initializes `consecutive_failed = 0` at line 104. Two paths increment it:

- **Capacity path** (lines 152–162): `next_cycles > cycles_limit || next_size > size_limit`
- **Non-proposed ancestor path** (lines 176–189): `ancestors_ids.iter().any(|id| !self.pool_map.has_proposed(id))`

Both paths check `if consecutive_failed > MAX_CONSECUTIVE_FAILURES { break; }`. The success path (lines 207–220) — where ancestors are packaged and `update_modified_entries` is called — contains no `consecutive_failed = 0` reset. The variable is named `consecutive_failed`, and the comment at lines 47–50 explicitly describes it as a heuristic for when the block is "close to full," implying the intent is to count *consecutive* failures. The omission of the reset means any mix of successes and failures accumulates monotonically toward the 4,000 threshold.

The non-proposed ancestor path is the critical attack vector: it fires regardless of remaining block capacity. A child transaction whose parent was never proposed will permanently fail this check on every template refresh, since `TxSelector` is constructed fresh per `package_txs` call (line 542 of `pool.rs`) and `failed_txs` is not persisted across calls. The `skip_proposed_entry` fast-path (lines 235–239) only skips entries already in `fetched_txs`, `modified_entries`, or `failed_txs` — none of which cover the main-iterator path for non-modified entries — so the same orphaned children are re-evaluated every refresh.

## Impact Explanation
When `consecutive_failed` exceeds 4,000, the loop breaks unconditionally. If the block is not yet full, all remaining proposed transactions sorted after the 4,001st failure are excluded from the template without any error or log warning. Miners receive a suboptimal template with reduced fee revenue, and legitimate users experience delayed transaction confirmation. This matches **Low (501–2000 points): Any other important performance improvements for CKB**, as it degrades block-packing efficiency and transaction throughput without crashing nodes or causing consensus deviation.

## Likelihood Explanation
The attack requires getting 4,001 child transactions into `Status::Proposed` while their parents remain non-proposed. In CKB's two-phase commit, proposal-zone inclusion is controlled by miners. An attacker must either operate mining infrastructure or pay fees to a cooperating miner to include proposals. This is a non-trivial cost. Additionally, proposed transactions that are not committed within the proposal window revert to pending, so the attacker must continuously re-propose to sustain the attack across template refreshes. The attack is repeatable but resource-intensive, limiting it to well-funded or miner-class adversaries.

## Recommendation
Reset `consecutive_failed` to `0` after each successful package inclusion, immediately after `self.update_modified_entries(&ancestors)` at line 220:

```rust
self.update_modified_entries(&ancestors);
consecutive_failed = 0; // reset streak after successful inclusion
```

Additionally, consider whether the non-proposed ancestor failure path should contribute to `consecutive_failed` at all, since it is not a capacity signal. A pre-filter on `sorted_proposed_iter` that excludes entries with non-proposed ancestors, or a separate counter for that path, would eliminate the attack surface entirely.

## Proof of Concept
1. Attacker submits 4,001 child transactions `C_1…C_4001`, each spending an output of a distinct parent `P_i` that is never submitted to the pool.
2. Attacker (as miner or via colluding miner) includes `C_1…C_4001` in proposal zones across consecutive blocks; each `C_i` transitions to `Status::Proposed`.
3. On the next `get_block_template` → `package_txs` → `txs_to_commit` call, the iterator yields `C_1, C_2, …` sorted by fee rate. Each `C_i` fails `has_proposed(P_i)` at line 178, incrementing `consecutive_failed`.
4. At `C_4001`, `consecutive_failed > 4000` triggers `break` at line 185.
5. Any legitimate proposed transaction `T` sorted after `C_4001` is excluded from the template even if the block has ample remaining cycles and bytes. The template is returned silently as if complete, with no error logged.
