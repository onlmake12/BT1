### Title
`total_tx_size` / `total_tx_cycles` Inflated After Ancestor-Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new aggregate pool-size statistics (`total_tx_size`, `total_tx_cycles`) are computed **before** any ancestor evictions occur, then unconditionally written back **after** those evictions. Because each eviction already decrements the live counters via `update_stat_for_remove_tx`, the final write overwrites the correct post-eviction values with stale pre-eviction values, permanently inflating both counters by the total size/cycles of every evicted transaction.

---

### Finding Description

`add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```
// Step 1 – snapshot pre-eviction totals + new tx
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211

// Step 2 – may evict N transactions; each calls update_stat_for_remove_tx,
//           which decrements self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

// Step 3 – unconditionally overwrite with the stale snapshot
self.total_tx_size  = total_tx_size;                            // line 218
self.total_tx_cycles = total_tx_cycles;                         // line 219
``` [1](#0-0) 

`updated_stat_for_add_tx` reads `self.total_tx_size` and `self.total_tx_cycles` at the moment of the call and adds the new transaction's size/cycles to produce local variables. [2](#0-1) 

`check_and_record_ancestors` enters the eviction branch when the incoming transaction's ancestor count exceeds `max_ancestors_count` but some ancestors are "cell-ref parents" (transactions that reference the same cell as a dep). It calls `remove_entry_and_descendants` for each evicted candidate. [3](#0-2) 

`remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` decrements `self.total_tx_size` and `self.total_tx_cycles` for every removed transaction. [4](#0-3) [5](#0-4) 

After `check_and_record_ancestors` returns, the live counters correctly reflect the post-eviction state. But lines 218-219 then overwrite them with the stale snapshot, re-adding the sizes/cycles of every evicted transaction back into the totals. The evicted transactions are gone from the pool; their sizes/cycles are not.

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict further transactions:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee entries
}
``` [6](#0-5) 

An inflated `total_tx_size` causes `limit_size` to believe the pool is over capacity and to evict additional legitimate, high-fee transactions that should have been retained. This also causes `tx_pool_info` RPC to report incorrect `total_tx_size` and `total_tx_cycles` values, misleading operators and fee-estimation logic. [7](#0-6) 

The inflation is **permanent** until the pool is cleared or the node restarts: every subsequent `add_entry` call that triggers evictions compounds the error further.

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is reachable by any unprivileged transaction sender via the `send_transaction` RPC or P2P relay. The attacker needs to submit a transaction whose inputs or cell-deps overlap with existing pool transactions such that the ancestor count exceeds `max_ancestors_count` (default 1,000) but the excess is attributable to cell-ref parents. This is a crafted but realistic scenario: the attacker pre-populates the pool with a chain of transactions that share a cell dep, then submits a transaction that references that cell dep, triggering the eviction branch. No special privilege, key, or majority hashpower is required.

---

### Recommendation

Move the final assignment of `self.total_tx_size` and `self.total_tx_cycles` to **after** `check_and_record_ancestors` returns, so the snapshot is taken from the already-updated (post-eviction) counters before adding the new transaction's contribution:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// Re-read counters after evictions, then add new tx
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, move the `updated_stat_for_add_tx` call (and the final assignment) to after `check_and_record_ancestors`, so the snapshot is always taken from the correct post-eviction state.

---

### Proof of Concept

1. Fill the pool with transactions `T1…TN` that all reference the same cell dep `C` (making them all "cell-ref parents" of each other in the ancestor graph). Each has size `S` bytes.
2. Submit a new transaction `TX` whose inputs consume a cell produced by `T1`, and whose cell dep is also `C`. This makes `TX`'s ancestor count = N+1, exceeding `max_ancestors_count`.
3. Because the excess ancestors are all cell-ref parents, `check_and_record_ancestors` enters the eviction branch and removes `K` of them. Each removal calls `update_stat_for_remove_tx`, decrementing `self.total_tx_size` by `K*S`.
4. Lines 218-219 then overwrite `self.total_tx_size` with `(pre-eviction total) + size(TX)`, which is `K*S` bytes higher than the true value.
5. `limit_size` now sees `total_tx_size > max_tx_pool_size` and begins evicting additional legitimate transactions, even though the pool is actually under capacity.
6. Repeating steps 1-5 compounds the inflation with each iteration. [8](#0-7) [9](#0-8)

        mut entry: TxEntry,
        status: Status,
    ) -> Result<(bool, HashSet<TxEntry>), Reject> {

        let mut evicts = Default::default();

        self.record_entry_descendants(&entry);
        self.track_entry_statics(None, Some(status));
        self.total_tx_size = total_tx_size;
        self.total_tx_cycles = total_tx_cycles;
        Ok((true, evicts))

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {

            self.remove_entry_links(id);

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
