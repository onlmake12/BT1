### Title
Registered Echo providers cannot be deregistered, leaving compromised or offline providers permanently active - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary
`Echo.sol` exposes a permissionless `registerProvider()` function that permanently sets `isRegistered = true` for any caller. No corresponding admin deregistration function exists. Once registered, a provider can never be removed or disabled by the admin, mirroring the external report's irreversible-configuration class. If a registered provider goes offline or becomes malicious, the admin has no on-chain mechanism to prevent users from routing requests to it, and user funds paid for those requests are locked for the duration of the exclusivity period with no refund path.

---

### Finding Description

`registerProvider()` in `Echo.sol` writes `isRegistered = true` into `_state.providers[msg.sender]` and never allows this flag to be cleared:

```solidity
// Echo.sol L381-393
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    provider.feePerFeedInWei = feePerFeedInWei;
    provider.feePerGasInWei = feePerGasInWei;
    provider.isRegistered = true;
    emit ProviderRegistered(msg.sender, feePerGasInWei);
}
``` [1](#0-0) 

The entire admin surface for provider management is limited to `setDefaultProvider()` (which only changes which registered provider is the default) and `setProviderFee()` (which is restricted to the provider or their fee manager, not the admin):

```solidity
// Echo.sol L395-408
function setProviderFee(...) external override {
    require(_state.providers[provider].isRegistered, "Provider not registered");
    require(
        msg.sender == provider ||
            msg.sender == _state.providers[provider].feeManager,
        "Only provider or fee manager can invoke this method"
    );
``` [2](#0-1) 

There is no `unregisterProvider()`, no admin flag to disable a provider, and no `cancelRequest()` or refund path for users. The contract's own inline TODO acknowledges the fund-locking risk:

```solidity
// Echo.sol L155-158
// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
// If executeCallback can revert, then funds can be permanently locked in the contract.
// TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
``` [3](#0-2) 

During the exclusivity window, only the originally assigned provider may call `executeCallback`:

```solidity
// Echo.sol L114-121
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
``` [4](#0-3) 

`exclusivityPeriodSeconds` is an admin-configurable `uint32`, meaning the window can be arbitrarily long. If the assigned provider is offline or malicious and refuses to call `executeCallback`, user funds (the full `msg.value` paid at request time) are locked in the contract for the entire exclusivity period with no recourse. [5](#0-4) 

---

### Impact Explanation

1. **Irreversible provider registration**: Any address that calls `registerProvider()` is permanently active. The admin cannot set `isRegistered = false` for any provider, regardless of circumstances.
2. **User fund locking**: A user who calls `requestPriceUpdatesWithCallback` targeting a registered-but-offline provider pays the full fee upfront. During the exclusivity period no other party can execute the callback, so the user's ETH is locked in the contract with no refund mechanism.
3. **No admin circuit-breaker**: Unlike `setDefaultProvider()` (which only affects the default), there is no function that prevents users from explicitly routing new requests to a deprecated or compromised provider address.

---

### Likelihood Explanation

- `registerProvider()` is permissionless — any EOA or contract can call it.
- Providers are expected to be long-running off-chain services; operational failures (key loss, infrastructure outage, deprecation) are realistic over a contract's lifetime.
- Users or integrating contracts that hard-code a specific provider address (e.g., the initial default provider) will continue routing requests to it even after it goes offline, because the contract provides no on-chain signal that the provider is inactive.
- The exclusivity period is configurable and could be set to hours or days, amplifying the fund-locking window.

---

### Recommendation

1. Add an admin-only `deregisterProvider(address provider)` function that sets `isRegistered = false` and emits a `ProviderDeregistered` event.
2. Add a `cancelRequest(uint64 sequenceNumber)` function that allows the original requester to reclaim their fee after the exclusivity period has elapsed without fulfillment.
3. Consider emitting a `ProviderDeregistered` event so off-chain tooling and users can detect deprecated providers.

---

### Proof of Concept

```
1. Attacker (or any address) calls registerProvider(1, 1, 1) → isRegistered = true permanently.
2. User calls requestPriceUpdatesWithCallback(attackerAddress, publishTime, priceIds, gasLimit)
   with msg.value = getFee(attackerAddress, gasLimit, priceIds).
   → req.provider = attackerAddress, req.fee = msg.value - pythFeeInWei stored in contract.
3. Attacker does NOT call executeCallback during the exclusivity window
   (block.timestamp < publishTime + exclusivityPeriodSeconds).
4. No other address can call executeCallback during this window (enforced by the require check).
5. User's ETH is locked in the contract for the full exclusivity period.
6. Admin calls setDefaultProvider(someOtherProvider) — this has no effect on the locked request
   and does not deregister the attacker's provider.
7. After exclusivity expires, a third party CAN execute the callback — but if no one does,
   funds remain locked indefinitely with no refund path.
```
