### Title
Subscription with Insufficient Keeper Fee Balance Causes `updatePriceFeeds` to Revert After Pyth Fee Is Already Paid, Griefing Keepers â€” (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol::updatePriceFeeds()`, the Pyth oracle fee is deducted from the subscription balance and forwarded to the Pyth contract **before** the keeper fee sufficiency check in `_processFeesAndPayKeeper()`. If the subscription balance is sufficient for the Pyth fee but insufficient for the keeper fee (which includes actual gas consumed), the entire transaction reverts. While the Pyth fee ETH is refunded by the revert, the keeper loses all gas spent executing the update. A subscription owner can deliberately maintain their balance in this "sweet spot" to grief keepers indefinitely, or it can occur naturally as a subscription's balance depletes.

---

### Finding Description

`updatePriceFeeds()` executes in this order:

1. **Early check** â€” reverts only if `balance < pythFee` [1](#0-0) 
2. **Pyth fee deducted** from `status.balanceInWei` and `status.totalSpent` [2](#0-1) 
3. **Pyth fee paid** via `parsePriceFeedUpdatesWithConfig{value: pythFee}(...)` â€” expensive external call [3](#0-2) 
4. Slot validation, timestamp validation, price deviation checks â€” all can revert [4](#0-3) 
5. Price feeds stored, status updated [5](#0-4) 
6. **`_processFeesAndPayKeeper()` called** â€” only here is the keeper fee checked against the remaining balance [6](#0-5) 

Inside `_processFeesAndPayKeeper()`, the keeper fee is computed as actual gas consumed (`startGas - gasleft()`) plus a fixed per-feed fee. If `status.balanceInWei < totalKeeperFee`, it reverts with `InsufficientBalance`: [7](#0-6) 

Because the revert unwinds all state changes including the ETH transfer to Pyth, the subscription owner loses nothing. However, the keeper has already consumed gas executing steps 1â€“5 (parsing VAAs, validating slots, storing prices) and receives no compensation.

The keeper fee is **non-deterministic** at call time because it depends on actual gas used. The early check at line 295 only guards against `balance < pythFee`, not against `balance < pythFee + keeperFee`. A subscription owner can deliberately fund their subscription to exactly `pythFee â‰¤ balance < pythFee + keeperFee` to ensure every keeper update attempt reverts after doing all the work. This is confirmed by the existing test `testUpdatePriceFeedsRevertsInsufficientBalanceForKeeperFee`, which explicitly demonstrates this revert path. [8](#0-7) 

---

### Impact Explanation

- **Keeper gas griefing**: Any keeper calling `updatePriceFeeds` on a subscription in the griefing state wastes all gas for the full execution (Pyth VAA parsing, slot validation, price storage) with zero compensation.
- **Stale price feeds**: The subscription's prices are never updated while in this state. Protocols consuming these feeds via `IScheduler` receive stale data, which can lead to incorrect liquidations, mispriced options, or other DeFi accounting errors.
- **Persistent DoS**: The subscription owner can maintain this state indefinitely by never topping up above the threshold, effectively disabling keeper service for their subscription at the cost of zero funds.

---

### Likelihood Explanation

- **Natural occurrence**: As a subscription's balance depletes through normal updates, it will naturally pass through the griefing range (`pythFee â‰¤ balance < pythFee + keeperFee`) before reaching `balance < pythFee`. During this window, all keeper update attempts revert.
- **Deliberate attack**: Any subscription owner (unprivileged user) can create a subscription and deliberately fund it to stay in the griefing range. No special access is required â€” `createSubscription` and `addFunds` are permissionless. [9](#0-8) 
- The `getMinimumBalance` enforcement at subscription creation/funding time does not prevent the balance from naturally depleting into the griefing range after updates.

---

### Recommendation

Move the keeper fee sufficiency check **before** the expensive Pyth fee payment. Pre-estimate the expected keeper fee using a gas estimate (e.g., using `gasleft()` at entry and a conservative overhead constant) and verify `balance >= pythFee + estimatedKeeperFee` before deducting the Pyth fee or calling the Pyth contract. Alternatively, refund the Pyth fee to the subscription balance on revert by catching the failure in `_processFeesAndPayKeeper` and restoring `status.balanceInWei` rather than reverting the entire transaction.

---

### Proof of Concept

1. Subscription owner calls `createSubscription` with a balance just above `pythFee` (e.g., `pythFee + 1 wei`), passing the minimum balance check.
2. After a few updates, `status.balanceInWei` naturally depletes to a value in `[pythFee, pythFee + keeperFee)`.
3. Keeper calls `updatePriceFeeds(subscriptionId, updateData)`:
   - Line 295: `balance >= pythFee` â†’ passes early check.
   - Line 305: `status.balanceInWei -= pythFee` â†’ balance now `< keeperFee`.
   - Line 311: Pyth fee forwarded to Pyth contract (expensive external call).
   - Lines 321â€“343: Slot validation, timestamp validation, price storage all succeed.
   - Line 345 â†’ `_processFeesAndPayKeeper`: `status.balanceInWei < totalKeeperFee` â†’ **reverts with `InsufficientBalance`**.
4. Entire transaction reverts. Keeper loses gas (~100kâ€“300k gas depending on number of feeds). Subscription prices remain stale. Subscription owner loses nothing.
5. This repeats on every subsequent keeper attempt until the subscription owner adds funds â€” which they can choose never to do.

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
