### Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Unique (from, to) PeerId Pairs During HolePunching Session — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

`HolePunching::forward_rate_limiter` uses `governor::HashMapStateStore<(PeerId, PeerId, u32)>` as its backing store. Every call to `check_key` with a previously-unseen key inserts a new entry into the internal `HashMap`. `retain_recent()` — the only mechanism that evicts stale entries — is called exclusively in `disconnected()`, never during `received()` or `notify()`. An unprivileged remote peer can therefore drive unbounded heap growth by sending messages with unique attacker-controlled `(from, to)` PeerId pairs for the duration of the session.

---

### Finding Description

`forward_rate_limiter` is declared as:

```rust
type RateLimiter<T> = governor::RateLimiter<
    T,
    governor::state::keyed::HashMapStateStore<T>,
    governor::clock::DefaultClock,
>;
// ...
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,
``` [1](#0-0) 

The third tuple element (`msg_item_id`) is the molecule union discriminant — always `0`, `1`, or `2` — derived from `msg.item_id()`: [2](#0-1) 

The first two elements (`content.from`, `content.to`) are **attacker-controlled PeerIds** parsed directly from the message body: [3](#0-2) 

Each `ConnectionRequest` (or `ConnectionRequestDelivered`, `ConnectionSync`) calls:

```rust
self.protocol
    .forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
``` [4](#0-3) 

`governor::HashMapStateStore::check_key` inserts a new entry for every previously-unseen key. The only eviction call is:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    ...
}
``` [5](#0-4) 

The `notify()` handler (fired every 5 minutes) cleans `pending_delivered` and `inflight_requests` but **never** calls `retain_recent()` on either rate limiter: [6](#0-5) 

---

### Impact Explanation

The outer `rate_limiter` (keyed by `(session_id, item_id)`) caps throughput at 30 messages/sec per session per message type: [7](#0-6) 

This limits the **rate** of insertion to ~30 new `(PeerId, PeerId, u32)` entries/sec, but does not bound the **total** size over a long-lived session. Over a 24-hour session: `30 × 86400 ≈ 2.6 million` entries. Each entry holds two `PeerId` values (~39 bytes each) plus a `u32` plus governor state plus `HashMap` overhead — roughly 150–200 bytes per entry — yielding ~400–500 MB of heap growth per persistent attacker session, with no in-session reclamation.

---

### Likelihood Explanation

Any unprivileged peer that can establish a P2P connection and speak the HolePunching protocol can execute this. No PoW, no key material, no privileged role is required. The attacker simply generates fresh random `PeerId` byte strings for `from`/`to` fields in each message. The outer rate limiter does not prevent this; it only slows the growth rate.

---

### Recommendation

Call `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside `notify()`, which already fires on a 5-minute `CHECK_INTERVAL`. This ensures stale entries (whose 1-second rate-limit window has long expired) are evicted periodically rather than only at disconnect. [8](#0-7) 

---

### Proof of Concept

```rust
// Pseudocode unit test
let mut protocol = HolePunching::new(network_state);
for i in 0..10_000u32 {
    let from = PeerId::random();
    let to   = PeerId::random();
    let msg  = build_connection_request(from, to, /* valid fields */);
    protocol.received(mock_ctx(), msg).await;
}
// Assert: forward_rate_limiter internal map has ~10_000 entries
// (no retain_recent() was called; disconnected() was never triggered)
assert!(forward_rate_limiter_len(&protocol) >= 10_000);
// After calling disconnected():
protocol.disconnected(mock_ctx()).await;
// All stale entries evicted — map is now near-empty
assert!(forward_rate_limiter_len(&protocol) < 10);
```

The outer rate limiter (30 req/sec) throttles the loop in real deployment, but the map still grows without bound for the session lifetime.
