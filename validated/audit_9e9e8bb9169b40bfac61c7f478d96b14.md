Audit Report

## Title
Out-of-Bounds Panic in `remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`File: util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary

`TxConfirmStat::remove_unconfirmed_tx` computes an unbounded `tx_age` and correctly guards the `block_unconfirmed_txs` array access for `tx_age >= 1000`, but unconditionally indexes `confirm_blocks_to_failed_txs[tx_age - 1]` without an equivalent bounds check. When `tx_age >= 1001`, the index `tx_age - 1 >= 1000` exceeds the array's length of 1000, causing a Rust index-out-of-bounds panic that crashes the node.

## Finding Description

`MAX_CONFIRM_BLOCKS = 1000` is defined at line 22. In `TxConfirmStat::new`, all three parallel arrays are allocated with the same length `max_confirm_blocks` (1000): [1](#0-0) 

In `remove_unconfirmed_tx` (lines 197–217), `tx_age` is unbounded: [2](#0-1) 

The `block_unconfirmed_txs` access is correctly guarded: [3](#0-2) 

But the `confirm_blocks_to_failed_txs` access immediately after has **no bounds check**: [4](#0-3) 

When `tx_age = 1001`, `tx_age - 1 = 1000` is out of bounds on a length-1000 array → **panic**.

The full call chain with `count_failure = true`:

- `shared_builder.rs:600` calls `fee_estimator.reject_tx(&tx_hash)` [5](#0-4) 
- `mod.rs:87` dispatches to `algo.write().reject_tx(tx_hash)` [6](#0-5) 
- `Algorithm::reject_tx` (line 475) calls `self.drop_tx(tx_hash)` [7](#0-6) 
- `drop_tx` (line 428) calls `drop_tx_inner(tx_hash, true)` [8](#0-7) 
- `drop_tx_inner` calls `remove_unconfirmed_tx(..., count_failure=true)` [9](#0-8) 

## Impact Explanation

A Rust index-out-of-bounds panic terminates the node process. This matches the **High** impact class: "Vulnerabilities which could easily crash a CKB node." The fee estimator runs inside the tx-pool service; a panic there brings down the node. No funds are at risk, but node availability is completely lost until manually restarted.

## Likelihood Explanation

No privileged access is required. Any user who can submit a transaction to the RPC or P2P relay can trigger this. The attacker submits a valid transaction with a very low fee rate so it is not confirmed, waits for the chain to advance 1001+ blocks (~7.8 hours at 28-second block intervals), then causes the transaction to be rejected — either by submitting a conflicting double-spend, waiting for pool eviction, or triggering a reorg that invalidates the transaction's inputs. The `ConfirmationFraction` estimator must be configured (it is not the default), but this is a documented configuration option.

## Recommendation

Add a bounds check before indexing `confirm_blocks_to_failed_txs`, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This ensures that transactions older than `MAX_CONFIRM_BLOCKS` are silently dropped from the failure-count statistics rather than causing a panic, consistent with how `block_unconfirmed_txs` already handles the same case.

## Proof of Concept

1. Configure the node with the `ConfirmationFraction` fee estimator.
2. Submit transaction `T` with a very low fee rate at block height `H`. The estimator records `TxRecord { height: H, bucket_index, fee_rate }` via `track_tx`.
3. Allow the chain to advance to block `H + 1001` without `T` being confirmed (`best_height = H + 1001`).
4. Cause `T` to be rejected (e.g., submit a conflicting transaction that spends the same input and gets confirmed). The tx-pool fires `fee_estimator.reject_tx(&T.hash())`.
5. Inside `drop_tx_inner(T.hash(), count_failure=true)`, `remove_unconfirmed_tx` is called with `entry_height = H`, `tip_height = H + 1001`, `count_failure = true`.
6. `tx_age = 1001`. The guard at line 208 fires (`1001 >= 1000`), so `block_unconfirmed_txs` is handled safely.
7. Line 215: `self.confirm_blocks_to_failed_txs[1001 - 1]` = `confirm_blocks_to_failed_txs[1000]` — **index out of bounds on a length-1000 array → panic → node crash**.

A unit test can reproduce this directly by constructing a `TxConfirmStat` with `max_confirm_blocks = 1000`, calling `add_unconfirmed_tx(0, fee_rate)`, then calling `remove_unconfirmed_tx(0, 1001, bucket_index, true)` and observing the panic.
