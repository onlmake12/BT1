### Title
Unbounded `forward_rate_limiter` HashMap growth via unique (from, to) PeerId pairs in long-lived sessions — (`network/src/protocols/hole_punching/mod.rs`)

### Summary
The `HolePunching` protocol's `forward_rate_limiter` (`HashMapStateStore<(PeerId, PeerId, u32)>`) accumulates stale entries indefinitely during a session because `retain_recent()` is only invoked in `disconnected()` and never in `received()` or `notify()`. An unprivileged remote peer can exploit this by sending `ConnectionSync` messages with unique attacker-controlled `(from, to)` PeerId pairs at the maximum permitted rate (30/sec), causing unbounded memory growth proportional to session duration.

### Finding Description
In `network/src/protocols/hole_punching/mod.rs`, the `HolePunching` struct holds two rate limiters:
- `rate_limiter: RateLimiter<(PeerIndex, u32)>` — keyed by session ID and message type
- `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` — keyed by attacker-controlled `(from, to)` PeerIds and a fixed message-type discriminant

The `disconnected()` handler calls `retain_recent()` on both, which removes stale entries from the underlying `HashMapStateStore`. However, `notify()` (which fires every 5 minutes via `CHECK_INTERVAL`) prunes `pending_delivered` and `inflight_requests` but **never** calls `retain_recent()` on either rate limiter.

The outer `rate_limiter` caps the total message rate at 30/sec per `(session_id, msg_item_id)`. This limits the insertion rate into `forward_rate_limiter` to 30 entries/sec, but does not bound the total number of entries. Since `from` and `to` are arbitrary bytes from the message body (parsed via `PeerId::from_bytes`), the attacker can supply a fresh unique pair in every message. Each unique pair inserts a new entry into the `HashMapStateStore` that is never evicted until the session disconnects.

### Impact Explanation
- At 30 unique pairs/sec, after 1 hour: **108,000 entries**
- Key size: `(PeerId, PeerId, u32)` ≈ 39 + 39 + 4 = 82 bytes; plus `governor`'s per-key `AtomicU64` state and `HashMap` overhead ≈ ~200 bytes/entry
- Memory consumed: ~108,000 × 200 bytes ≈ **~21 MB per session per hour**
- With multiple concurrent long-lived sessions (CKB nodes maintain many peers), memory pressure compounds
- The node cannot reclaim this memory until the attacker disconnects, enabling a slow-burn denial-of-service against long-running nodes

### Likelihood Explanation
The attack requires only a single valid P2P session with the HolePunching protocol enabled. No PoW, no privileged role, no Sybil attack. The attacker simply sends well-formed `ConnectionSync` messages with unique `from`/`to` byte sequences. The outer rate limiter (30/sec) is the only throttle, and it does not prevent the growth — it merely sets the rate. The `notify()` timer fires every 5 minutes but provides no relief. This is concretely exploitable by any peer.

### Recommendation
Call `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside `notify()` so that stale entries are periodically evicted regardless of whether a session disconnects. This mirrors the pattern already used in `disconnected()` and would bound the HashMap size to entries active within the last rate-limit window.

### Proof of Concept
```rust
// Pseudocode: attacker loop for one session
for i in 0..108_000 {
    let from = PeerId::random(); // unique per message
    let to   = PeerId::random(); // unique per message
    send_connection_sync(session, from, to);
    sleep(Duration::from_millis(33)); // ~30/sec, within outer rate limit
}
// After 1 hour: forward_rate_limiter HashMap has ~108,000 entries, ~21 MB
// retain_recent() has never been called; all entries are stale but resident
assert!(forward_rate_limiter_size() > 100_000); // no bound enforced
```

The fix is one line in `notify()`:
```rust

self.forward_rate_limiter.retain_recent();
self.rate_limiter.retain_recent();
