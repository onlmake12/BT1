### Title
Missing Per-Peer Rate Limiting in Light Client Protocol Server Allows Sustained Near-Maximum-Cost Request Flooding — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

---

### Summary

`GetBlocksProofProcess::execute` performs up to ~4000 database reads plus an MMR `gen_proof` over up to 999 positions per request. The `LightClientProtocol` handler has **no rate limiter** of any kind, and a structurally valid request (≤1000 unique hashes, valid `last_hash`) always returns `StatusCode::OK` (200), which never triggers a ban. A single unprivileged peer can loop this request indefinitely at full cost with zero consequence.

---

### Finding Description

**Entry point** — `LightClientProtocol::received` → `try_process` → `GetBlocksProofProcess::execute`.

The `LightClientProtocol` struct contains only a `shared: Shared` field; there is no `rate_limiter` field anywhere in the crate: [1](#0-0) 

`try_process` dispatches directly to the handler with no rate-limit check: [2](#0-1) 

Contrast with `Relayer`, which gates every non-PoW message through a per-peer per-message-type `RateLimiter` at 30 req/s before any processing: [3](#0-2) 

**Work performed per valid request with 999 found + 1 missing hash:**

1. 1000 × `snapshot.is_main_chain()` (partition step): [4](#0-3) 

2. For each of 999 found blocks — `get_block_header`, `get_block_uncles`, `get_block_extension`, `calc_uncles_hash` (~4 DB reads/hash): [5](#0-4) 

3. `mmr.gen_proof(999 positions)` — O(N log N) MMR traversal: [6](#0-5) 

**Why no ban fires:** `should_ban()` only returns `Some` for 4xx status codes. A structurally valid request returns `StatusCode::OK` (200): [7](#0-6) [8](#0-7) 

The limit constant is 1000: [9](#0-8) 

---

### Impact Explanation

A single peer can sustain near-maximum server CPU and RocksDB I/O load indefinitely. Each request forces ~4000 synchronous DB reads and a 999-leaf MMR proof generation. Because the light client protocol handler runs in the same async runtime as the rest of the node, sustained flooding degrades block relay, sync, and transaction processing for all peers — causing effective CKB network participation degradation from a single low-cost attacker.

---

### Likelihood Explanation

The attack requires only a valid P2P connection to a node with the light client protocol enabled, knowledge of 999 main-chain block hashes (trivially obtained from any block explorer or prior sync), and a loop. No PoW, no stake, no privileged access. The `GET_BLOCKS_PROOF_LIMIT = 1000` constant was intended as a size cap, not a rate cap, and the missing rate limiter is a concrete, reproducible gap relative to the `Relayer` protocol which has the same limit but also has rate limiting.

---

### Recommendation

Add a per-peer per-message-type `RateLimiter` to `LightClientProtocol`, mirroring the pattern in `Relayer`: [10](#0-9) 

The check should be inserted at the top of `try_process` before any handler dispatch, returning a non-ban status (e.g., `StatusCode::TooManyRequests` if added, or silently dropping) so the peer is throttled but not permanently banned for legitimate bursts.

---

### Proof of Concept

```
1. Connect to a CKB full node with light client protocol enabled.
2. Obtain 999 valid main-chain block hashes (e.g., blocks 1–999) and the current tip hash.
3. Append one random 32-byte hash as the 1000th entry.
4. Serialize a GetBlocksProof{last_hash: tip, block_hashes: [h1..h999, random]} message.
5. Send in a tight loop.
6. Observe: server responds to every message with SendBlocksProofV1 (999 proved + 1 missing).
7. Observe: peer is never banned (status is always OK/200).
8. Measure: server CPU and RocksDB read IOPS remain pegged at near-maximum for the duration.
```

impl Relayer {
    /// Init relay protocol handle
    ///
    /// This is a runtime relay protocol shared state, and any relay messages will be processed and forwarded by it
    pub fn new(chain: ChainController, shared: Arc<SyncShared>) -> Self {
        // setup a rate limiter keyed by peer and message type that lets through 30 requests per second
        // current max rps is 10 (ASK_FOR_TXS_TOKEN / TX_PROPOSAL_TOKEN), 30 is a flexible hard cap with buffer
        let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
        let rate_limiter = RateLimiter::hashmap(quota);

        Relayer {
            chain,
            shared,
            rate_limiter,
        }
    }

    /// Get shared state
    pub fn shared(&self) -> &Arc<SyncShared> {
        &self.shared
    }

    async fn try_process(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        message: packed::RelayMessageUnionReader<'_>,
    ) -> Status {
        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =
            !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));

        if should_check_rate
            && self
                .rate_limiter
                .check_key(&(peer, message.item_id()))
                .is_err()
        {
            return StatusCode::TooManyRequests.with_context(message.item_name());
        }
