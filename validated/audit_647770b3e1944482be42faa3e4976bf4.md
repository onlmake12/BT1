### Title
Network Alert Signature Digest Excludes Chain Identifier, Enabling Cross-Network Replay — (File: `util/network-alert/src/verifier.rs`)

### Summary
The `RawAlert` message digest used to verify network alert signatures does not include any chain-specific identifier (genesis hash, chain name, or network ID). A validly signed alert from one CKB network (e.g., testnet) can be replayed verbatim on a different CKB network (e.g., mainnet) and will pass signature verification, causing the alert to propagate and be displayed to all nodes on the target network.

### Finding Description
The `RawAlert` molecule struct is defined with the following fields only:

```
table RawAlert {
    notice_until:   Uint64,
    id:             Uint32,
    cancel:         Uint32,
    priority:       Uint32,
    message:        Bytes,
    min_version:    BytesOpt,
    max_version:    BytesOpt,
}
``` [1](#0-0) 

No chain identifier — genesis block hash, chain spec name, or network ID — is present in this struct.

The `calc_alert_hash()` function computes the signing digest by hashing only the raw bytes of `RawAlert`:

```rust
impl<'r> packed::RawAlertReader<'r> {
    pub fn calc_alert_hash(&self) -> packed::Byte32 {
        self.calc_hash()   // blake2b_256(self.as_slice())
    }
}
``` [2](#0-1) 

The signature verifier in `verify_signatures()` uses this hash directly as the ECDSA message:

```rust
let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
``` [3](#0-2) 

Because the digest is computed solely from `RawAlert` content — which contains no chain-binding field — the same `(RawAlert, signatures)` pair produces an identical digest on every CKB network that shares the same Nervos Foundation signing keys.

### Impact Explanation
The network alert system is the primary mechanism for broadcasting critical security warnings to all CKB nodes. A successfully verified alert is stored, propagated peer-to-peer, and surfaced to node operators via the RPC `get_blockchain_info` response. [4](#0-3) 

An attacker who captures a legitimately signed alert from testnet can relay it to mainnet peers. Because `verify_signatures()` passes (the digest and keys are identical across networks), the alert propagates through the entire mainnet P2P network. This can:

- Cause mainnet node operators to take unnecessary emergency actions (halt operations, avoid transactions) in response to a testnet-only drill or test alert.
- Conversely, suppress or confuse response to a real mainnet alert if operators assume it is a replay.
- Undermine trust in the alert mechanism as a reliable security channel.

### Likelihood Explanation
The Nervos Foundation uses a single set of multi-sig keys (`NetworkAlertConfig.public_keys`) to sign alerts. [5](#0-4) 

The same key set is operationally expected to be used across mainnet and testnet (there is no per-network key rotation enforced by the protocol). Any unprivileged peer connected to both networks can observe a signed alert on one network and relay it to the other — no special access, leaked key, or majority hashpower is required. The attack is passive observation followed by a single P2P message relay.

### Recommendation
Include a chain-specific binding value in the `RawAlert` digest. The natural choice in CKB is the genesis block hash, which is already used as the network identity discriminator (`identify_name()` in `spec/src/consensus.rs`). [6](#0-5) 

Concretely, either:
1. Add a `genesis_hash: Byte32` field to `RawAlert` and require signers to populate it, **or**
2. In `verify_signatures()`, compute the digest as `blake2b(genesis_hash || raw_alert_bytes)` so the chain context is mixed in at verification time without a schema change.

Option 2 is backward-compatible with existing alert wire format but requires the verifier to receive the genesis hash at construction time.

### Proof of Concept

1. Operator signs a testnet alert with the Nervos Foundation keys:
   ```
   RawAlert { id: 42, message: "testnet drill", notice_until: T, ... }
   digest = blake2b(raw_alert.as_slice())   // no chain binding
   sig = secp256k1_sign(privkey, digest)
   ```
2. Attacker observes the signed `Alert` message on the testnet P2P network.
3. Attacker connects to a mainnet peer and sends the identical `Alert` molecule bytes.
4. Mainnet node calls `verify_signatures()`:
   ```rust
   let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
   // digest is identical to testnet — passes
   verify_m_of_n(&message, threshold, &signatures, &pubkeys)  // OK
   ``` [7](#0-6) 
5. Alert is accepted, stored, and relayed to all mainnet peers; it appears in every node's `get_blockchain_info` response as a live alert.

    pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
        trace!("Verifying alert {:?}", alert);
        let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
        let signatures: Vec<Signature> = alert
            .signatures()
            .into_iter()
            .filter_map(
                |sig_data| match Signature::from_slice(sig_data.as_reader().raw_data()) {
                    Ok(sig) => {
                        if sig.is_valid() {
                            Some(sig)
                        } else {
                            debug!("invalid signature: {:?}", sig);
                            None
                        }
                    }
                    Err(err) => {
                        debug!("signature error: {}", err);
                        None
                    }
                },
            )
            .collect();
        verify_m_of_n(
            &message,
            self.config.signatures_threshold,
            &signatures,
            &self.pubkeys,
        )
        .map_err(|err| err.kind())?;
        Ok(())
```
