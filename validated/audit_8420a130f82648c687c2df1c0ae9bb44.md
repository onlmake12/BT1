### Title
Stale `descendants_*` Stats on Ancestor Entries After `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`remove_entry_and_descendants` pre-removes all link graph entries before calling `remove_entry` on each removed transaction. Because `update_ancestors_index_key` relies on the live link graph to find which remaining entries need their `descendants_*` stats decremented, pre-removing the links causes it to return an empty set — leaving every ancestor of the removed subtree with permanently inflated `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee`. This is the direct CKB analog of the Fenwick tree incomplete-reset bug: a partial structural teardown silently corrupts the cumulative accounting of entries that survive the removal.

---

### Finding Description

**Root cause — `remove_entry_and_descendants`:**

```
pool_map.rs:252-265
``` [1](#0-0) 

The function first collects the root id and all its descendants, then calls `remove_entry_links` for **every** id in that set before calling `remove_entry` on any of them:

```rust
// update links state for remove, so that we won't update_descendants_index_key in remove_entry
for id in &removed_ids {
    self.remove_entry_links(id);   // ← strips the root from its parents' children lists
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry_links` for the root entry removes the root from its parents' `children` sets and then deletes the root's own link entry: [2](#0-1) 

When `remove_entry` is subsequently called for the root, it invokes `update_ancestors_index_key`: [3](#0-2) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`. Because the root's link entry was already deleted, `calc_ancestors` returns an **empty set**: [4](#0-3) 

Consequently, `sub_descendant_weight` is **never called** on any ancestor of the removed root. Every entry that was a parent (or grandparent, etc.) of the removed root retains stale `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` that still count the removed transactions.

**The `TxEntry` fields that become stale:** [5](#0-4) 

**Accumulation on re-insertion:** When a new child of the surviving ancestor is later added, `record_entry_descendants` calls `update_ancestors_index_key(new_child, Add)`, which calls `add_descendant_weight` on the ancestor — stacking new values on top of the already-inflated ones: [6](#0-5) 

**The existing test does not catch this:** `test_remove_entry_and_descendants` only asserts that `calc_descendants` (the live link graph) is correct after removal. It never checks whether tx1's `descendants_count` / `descendants_fee` / etc. were decremented: [7](#0-6) 

---

### Impact Explanation

The `descendants_*` fields feed directly into `EvictKey`, which governs which transaction is selected for eviction when the pool is full: [8](#0-7) 

An ancestor with inflated `descendants_fee` / `descendants_size` will have an artificially high `descendants_feerate`, making it appear more valuable than it is. `limit_size` uses `next_evict_entry` (ordered by `EvictKey`) to decide what to drop: [9](#0-8) 

Concrete consequences:
1. **Eviction bypass:** A low-fee root transaction is never evicted because its `EvictKey` is inflated by the fees of already-removed descendants.
2. **Unbounded inflation:** Each cycle of "add high-fee child → trigger removal via RBF/conflict → add another child" further inflates the ancestor's stats without bound.
3. **Legitimate transaction displacement:** Correctly-accounted transactions with genuinely high fee rates are evicted in preference to the attacker's inflated-stat transaction.
4. **Incorrect `check_and_record_ancestors` eviction:** The same `iter_by_evict_key` is used to select which `cell_ref_parents` to evict when the ancestor limit is exceeded; stale stats corrupt that selection too. [10](#0-9) 

---

### Likelihood Explanation

`remove_entry_and_descendants` is called from **six distinct code paths**, all reachable by an unprivileged submitter or relayer:

| Caller | Trigger |
|---|---|
| `process_rbf` (process.rs:205) | RBF replacement — standard user action |
| `resolve_conflict` (pool_map.rs:310,321) | Block committed consuming a pool input |
| `resolve_conflict_header_dep` (pool_map.rs:285) | Block invalidates a header dep |
| `remove_by_detached_proposal` (pool.rs:343) | Proposal window rolls |
| `limit_size` (pool.rs:307) | Pool size limit reached |
| `check_and_record_ancestors` (pool_map.rs:618) | Ancestor count limit eviction | [11](#0-10) [12](#0-11) 

The bug fires whenever the removed entry has at least one ancestor that remains in the pool — a routine condition in any chain of dependent transactions. RBF alone is sufficient: submit tx1→tx2, replace tx2 via RBF, and tx1's stats are permanently corrupted.

---

### Recommendation

Before pre-removing links, capture the set of **external ancestors** (ancestors of the root that are not themselves in the removed set) and call `sub_descendant_weight` on each of them for every entry being removed. Concretely, for the root entry `id`, compute `calc_ancestors(id)` while the link graph is still intact, then decrement each surviving ancestor's descendant stats for every entry in `removed_ids`. Alternatively, restructure `remove_entry_and_descendants` to call `update_ancestors_index_key` for the root entry **before** stripping its links, so the normal `remove_entry` path can propagate the decrement correctly to surviving ancestors.

---

### Proof of Concept

```
State: tx1 (fee=10, size=100) → tx2 (fee=1000, size=100) → tx3 (fee=1000, size=100)
All three are in the pool.

After add_proposed(tx1, tx2, tx3):
  tx1.descendants_count  = 3
  tx1.descendants_fee    = 2010
  tx1.descendants_size   = 300

Call remove_entry_and_descendants(tx2_id):
  Step 1: remove_entry_links(tx2_id) → tx1's children no longer contains tx2;
          tx2's link entry deleted.
          remove_entry_links(tx3_id) → tx3's link entry deleted.
  Step 2: remove_entry(tx2_id):
          update_ancestors_index_key(tx2, Remove):
            calc_ancestors(tx2_id) → {} (link entry gone) → no-op
          → tx1.descendants_* NOT decremented
  Step 3: remove_entry(tx3_id): same — no-op

After removal:
  tx2, tx3 are gone from the pool.
  tx1.descendants_count  = 3   ← should be 1
  tx1.descendants_fee    = 2010 ← should be 10
  tx1.descendants_size   = 300  ← should be 100

tx1's EvictKey now reflects descendants_feerate ≈ 2010/weight(300,…)
instead of feerate = 10/weight(100,…).
tx1 is treated as a high-value transaction and is never evicted.

Repeat: submit tx4 (fee=1000) as child of tx1 → add_descendant_weight adds
1000 to already-inflated tx1.descendants_fee = 3010. Remove tx4 via RBF.
tx1.descendants_fee = 3010 (should be 10). Each cycle inflates further.
```

The existing test `test_remove_entry_and_descendants` confirms the link graph is cleaned up but never asserts `tx1.descendants_count == 1` or `tx1.descendants_fee == tx1.fee` after the call, leaving the stale-stat corruption undetected. [1](#0-0) [3](#0-2) [13](#0-12) [9](#0-8)

    fn record_entry_descendants(&mut self, entry: &TxEntry) {
        let tx_short_id: ProposalShortId = entry.proposal_short_id();
        let outputs = entry.transaction().output_pts();
        let mut children = HashSet::new();

        // collect children
        for o in outputs {
            if let Some(ids) = self.edges.get_deps_ref(&o).cloned() {
                children.extend(ids);
            }
            if let Some(id) = self.edges.get_input_ref(&o).cloned() {
                children.insert(id);
            }
        }
        // update children
        if !children.is_empty() {
            for child in &children {
                self.links.add_parent(child, tx_short_id.clone());
            }
            if let Some(links) = self.links.inner.get_mut(&tx_short_id) {
                links.children.extend(children);
            }
            self.update_descendants_index_key(entry, EntryOp::Add);
        }
        // update ancestor's index key for adding new entry
        self.update_ancestors_index_key(entry, EntryOp::Add);

            }
        }
