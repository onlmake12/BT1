### Title
Unhandled Callback Revert in Legacy `revealWithCallback` Path Permanently Locks User Fees - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
In `Entropy.sol`, the `revealWithCallback` function has two execution paths. When a provider has not set a `defaultGasLimit` (i.e., `providerInfo.defaultGasLimit == 0`), requests are stored with `gasLimit10k = 0`. During fulfillment, these requests take the `else` branch which calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly — with no `try/catch` or `excessivelySafeCall` wrapper. If the consumer's callback reverts for any reason, the entire `revealWithCallback` transaction reverts, the request is never cleared, and the user's paid fee is permanently locked with no refund path.

### Finding Description

In `requestHelper`, when `providerInfo.defaultGasLimit == 0`, the request is stored with `req.gasLimit10k = 0`:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;
} else { ... }
```

In `revealWithCallback`, the branching logic is:

```solidity
if (
    req.gasLimit10k != 0 &&
    req.callbackStatus == EntropyStatusConstants.CALLBACK_NOT_STARTED
) {
    // SAFE PATH: uses excessivelySafeCall, catches reverts, emits CallbackFailed
    (success, ret) = req.requester.excessivelySafeCall(...);
    ...
} else {
    // LEGACY PATH: no error handling
    address callAddress = req.requester;
    clearRequest(provider, sequenceNumber);
    if (len != 0) {
        IEntropyConsumer(callAddress)._entropyCallback(  // <-- bare call, no try/catch
            sequenceNumber, provider, randomNumber
        );
    }
}
```

In the `else` branch, `clearRequest` is called before the callback. If `_entropyCallback` reverts, the entire transaction reverts — including `clearRequest` — leaving the request permanently active but unfulfillable. There is no `cancelRequest` or fee-refund function anywhere in the contract.

### Impact Explanation

1. **Permanent fee loss**: A user whose callback reverts (due to a bug, state change between request and reveal, reentrancy guard, or any other condition) loses their paid fee with no recovery path.
2. **Keeper griefing**: The Fortuna keeper service will repeatedly attempt `revealWithCallback` for stuck requests, wasting gas indefinitely.
3. **Request permanently stuck**: The sequence number slot is consumed and the request can never be cleared, as every fulfillment attempt reverts.

### Likelihood Explanation

Any user interacting with a provider that has `defaultGasLimit == 0` (the legacy configuration) is exposed. Callbacks can legitimately revert due to: state changes between request and reveal (e.g., a `require` on a game state that has since changed), reentrancy guards, contract upgrades, or simple bugs. The Fortuna keeper is the one calling `revealWithCallback`, so the user cannot even choose to skip the callback. This is a realistic scenario for any production consumer contract with non-trivial callback logic.

### Recommendation

Apply `excessivelySafeCall` (already imported and used in the safe path) to the legacy `else` branch as well, or add a `try/catch` wrapper around the direct `_entropyCallback` invocation. Additionally, implement a `cancelRequest` or fee-refund function so users can recover fees from permanently stuck requests.

### Proof of Concept

1. Provider registers with `defaultGasLimit == 0` (legacy mode, never calls `setDefaultGasLimit`).
2. User deploys a consumer contract whose `entropyCallback` reverts when a certain state variable is set (e.g., `require(!paused)`).
3. User calls `requestWithCallback` — request stored with `gasLimit10k = 0`.
4. User's contract is paused between request and reveal.
5. Fortuna keeper calls `revealWithCallback` → `_entropyCallback` reverts → entire tx reverts → `clearRequest` is rolled back → request remains active.
6. Every subsequent `revealWithCallback` attempt reverts. User's fee is permanently locked.
