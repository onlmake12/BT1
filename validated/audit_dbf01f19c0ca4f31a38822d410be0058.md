Let me look at the `update_single_price_feed` logic more carefully, specifically the iteration and the `is_fresh_update` check.

### Title
Relayer-Controlled VAA Ordering in `create_price_infos_hot_potato` Allows Stale Price to Overwrite Fresher Price — (`target_chains/sui/contracts/sources/pyth.move`)

---

### Summary

An unprivileged relayer can supply two valid VAAs from two different configured data sources, both attesting to the same `price_identifier` but with different timestamps, and by controlling the order of the VAA vector passed to `create_price_infos_hot_potato`, deterministically place the **older** price at index 0 of the resulting `HotPotatoVector`. Because `update_single_price_feed` breaks on the **first** matching entry and `is_fresh_update` only compares against the currently-cached price (not against other entries in the same batch), the older price is written to the oracle while the newer price is silently ignored.

---

### Finding Description

**Step 1 — VAA ingestion in `create_price_infos_hot_potato`**

The function consumes the input `verified_vaas` vector with `vector::pop_back` (LIFO), and for each VAA it also drains the inner `price_infos` vector with `vector::pop_back` (LIFO), appending each result to `price_updates` with `vector::push_back`. [1](#0-0) 

Consequence: the VAA that appears **last** in the input vector is processed **first**, and its price entries land at the **lowest indices** of `price_updates`.

**Step 2 — First-match-wins in `update_single_price_feed`**

The function iterates `price_updates` from index 0 upward, and the moment it finds a `PriceInfo` whose `price_identifier` matches the target `PriceInfoObject`, it calls `update_cache` and **breaks**. [2](#0-1) 

It never inspects any later entry for the same `price_identifier`, even if that entry carries a higher timestamp.

**Step 3 — `is_fresh_update` only guards against the cached baseline**

`update_cache` applies the update only when `update_timestamp > cached_timestamp`, where `cached_timestamp` is the timestamp already stored in the `PriceInfoObject`. [3](#0-2) 

There is no comparison between the two in-batch entries. As long as the chosen (older) entry is still fresher than the on-chain cache, it passes the guard.

**Exploit construction**

| Symbol | Meaning |
|--------|---------|
| T0 | timestamp currently cached on-chain |
| T1 | timestamp in VAA_A (older, T0 < T1 < T2) |
| T2 | timestamp in VAA_B (newer) |

Pass `[VAA_B, VAA_A]` to `create_price_infos_hot_potato`:
- `pop_back` → VAA_A processed first → its `PriceInfo` (T1) lands at index 0
- `pop_back` → VAA_B processed next → its `PriceInfo` (T2) lands at index 1

`update_single_price_feed` hits index 0 first, checks T1 > T0 (true), writes T1, breaks. The T2 entry is never reached.

Pass `[VAA_A, VAA_B]` instead and T2 lands at index 0 — the newer price wins.

The relayer fully controls which price is written by choosing the VAA ordering.

---

### Impact Explanation

The oracle can be made to store a price that is older than the best available attested price, by a gap equal to the time difference between the two VAAs. Any DeFi protocol (lending, perpetuals, options) that reads the oracle immediately after such an update will act on a stale price. The attacker does not need to forge any signature or corrupt any guardian; they only need to relay two legitimately signed VAAs in a chosen order. The impact matches the stated scope: attacker-controlled price ordering determines which price is written to the oracle.

---

### Likelihood Explanation

The preconditions are realistic:

1. **Multiple valid data sources** — `data_sources_for_test_vaa()` already shows three configured sources; production deployments commonly include a primary (Pythnet) and one or more backup emitters. [4](#0-3) 

2. **Same `price_identifier` in both VAAs** — redundant data sources are expected to attest to the same symbols.

3. **Unprivileged relayer** — `create_price_infos_hot_potato` is a `public fun` with no signer restriction; any account can call it. [5](#0-4) 

4. **Hot-potato atomicity** — the relayer constructs a single Sui PTB that calls `create_price_infos_hot_potato`, then `update_single_price_feed`, then `hot_potato_vector::destroy`, so no cooperation from the victim is required.

---

### Recommendation

Replace the first-match-break pattern in `update_single_price_feed` with a full scan that selects the entry with the **highest timestamp** among all entries matching the target `price_identifier`, then applies only that one. Alternatively, deduplicate entries for the same `price_identifier` inside `create_price_infos_hot_potato` itself, keeping only the highest-timestamp entry before constructing the `HotPotatoVector`.

---

### Proof of Concept

```
// Pseudocode differential test
let vaa_A = sign_price(price_id=X, price=100, timestamp=T1)  // T0 < T1 < T2
let vaa_B = sign_price(price_id=X, price=200, timestamp=T2)

// Attack ordering: VAA_B last → processed first → T2 at index 1, T1 at index 0
let potato_attack = create_price_infos_hot_potato(state, [vaa_A, vaa_B], clock)
update_single_price_feed(state, potato_attack, price_info_obj_X, fee, clock)
assert(get_price(price_info_obj_X) == 100)  // T1 price written, NOT T2

// Correct ordering: VAA_A last → processed first → T1 at index 1, T2 at index 0
let potato_correct = create_price_infos_hot_potato(state, [vaa_B, vaa_A], clock)
update_single_price_feed(state, potato_correct, price_info_obj_X, fee, clock)
assert(get_price(price_info_obj_X) == 200)  // T2 price written correctly
```

The two runs produce different on-chain prices despite identical VAA content, proving the invariant is broken by input ordering alone.
