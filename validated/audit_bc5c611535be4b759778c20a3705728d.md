### Title
Unbounded `while` Loop in `executeCallback` Enables Requester-Triggered DoS That Locks Funds — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that linearly scans fulfilled sequence numbers to advance `_state.firstUnfulfilledSeq`. Any unprivileged user can call `requestPriceUpdatesWithCallback` to grow the sequence space. If enough requests accumulate and are fulfilled out of order (oldest last), the while loop in the final `executeCallback` call will exhaust the block gas limit, causing the transaction to revert and permanently preventing fulfillment of the oldest request — locking the requester's fee in the contract.

### Finding Description

In `Echo.sol`, `executeCallback` runs the following loop after clearing the fulfilled request:

```solidity
clearRequest(sequenceNumber);

while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration of the loop calls `findRequest`, which performs at least one cold `SLOAD` (~2100 gas) against `_state.requests[shortKey]`, and potentially a second `SLOAD` against `_state.requestsOverflow[key]`. It also writes `_state.firstUnfulfilledSeq++` (warm `SSTORE`, ~5000 gas). The per-iteration cost is approximately 7,000–9,000 gas. [2](#0-1) 

The loop is bounded only by `_state.currentSequenceNumber`, which is incremented by every call to `requestPriceUpdatesWithCallback` — a function open to any caller: [3](#0-2) 

The `State` struct stores `firstUnfulfilledSeq` and `currentSequenceNumber` as `uint64` fields with no cap: [4](#0-3) 

The developers themselves acknowledge this in a TODO comment directly above the loop:

> "I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. a better solution would be a doubly-linked list of active requests." [5](#0-4) 

The gas benchmark test `testMultipleRequestsOutOfOrderFulfillment` explicitly documents the worst-case scenario (10 requests fulfilled in reverse order), noting that the last fulfillment is the most expensive: [6](#0-5) 

### Impact Explanation

If the while loop exhausts the block gas limit, the entire `executeCallback` transaction reverts. Because `clearRequest` and the fee-credit (`accruedFeesInWei +=`) both occur before the loop, a revert undoes them: [7](#0-6) 

Consequences:
- The request at `firstUnfulfilledSeq` cannot be fulfilled — it is permanently stuck.
- The requester's fee (`req.fee`) remains locked in the contract with no withdrawal path for the requester.
- The provider cannot earn the fee for that request.
- Every subsequent `executeCallback` call for any sequence number will also hit the same loop and revert, because `firstUnfulfilledSeq` never advances past the stuck request.

This effectively bricks the entire Echo contract for all users once the attack threshold is reached.

### Likelihood Explanation

- `requestPriceUpdatesWithCallback` is permissionless and requires only paying the provider fee. An attacker can create thousands of requests at the minimum fee.
- The 60-second `publishTime` cap does not limit the number of requests — only how far in the future the timestamp can be. An attacker can submit many requests with `publishTime = block.timestamp`.
- At ~7,000–9,000 gas per loop iteration and a 30M block gas limit, approximately 3,300–4,300 requests fulfilled out of order are sufficient to make `executeCallback` revert.
- Providers naturally fulfill requests in FIFO or reverse order depending on their implementation. If a provider processes newer requests first (e.g., due to queue ordering), the oldest request is always fulfilled last, triggering the worst-case loop.
- The attack cost is bounded by `N × (provider_fee + pyth_fee)`. At low fee settings, this is economically viable.

### Recommendation

Replace the unbounded `while` loop with a bounded scan (e.g., cap the number of iterations per call), or replace the linear scan entirely with a doubly-linked list of active requests (as the TODO comment already suggests). Alternatively, remove the `firstUnfulfilledSeq` advancement from `executeCallback` and expose a separate, paginated admin/keeper function to advance it in batches.

### Proof of Concept

1. Attacker calls `requestPriceUpdatesWithCallback` N times (e.g., N = 4000), paying the required fee each time. `_state.currentSequenceNumber` advances to N+1.
2. The provider fulfills requests N, N-1, N-2, ..., 2 (in reverse order). Each `executeCallback` call succeeds because `firstUnfulfilledSeq` is still 1 (request 1 is still active), so the while loop exits immediately on each call.
3. The provider attempts to fulfill request 1 (the oldest). After `clearRequest(1)`, the while loop starts at `firstUnfulfilledSeq = 1`:
   - Iteration 1: request 1 is now inactive → advance to 2
   - Iteration 2: request 2 is inactive (already fulfilled) → advance to 3
   - ...
   - Iteration N: request N is inactive → advance to N+1 → stop
4. At ~8,000 gas/iteration × 4,000 iterations = 32,000,000 gas, the transaction exceeds the block gas limit and reverts.
5. Request 1 is restored (revert undoes `clearRequest`). The provider retries and hits the same out-of-gas error every time. Request 1 is permanently unfulfillable, and the requester's fee is permanently locked.
