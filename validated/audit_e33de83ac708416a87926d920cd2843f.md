Audit Report

## Title
Stale `evict_key` on Ancestor Entries After `remove_entry_and_descendants` Causes Incorrect Tx-Pool Eviction Order — (File: tx-pool/src/component/pool_map.rs)

## Summary
`PoolMap::remove_entry_and_descendants` clears all link records for every node in the removed subtree before calling `remove_entry` on each. Because `update_ancestors_index_key` relies on `links.calc_ancestors` to find which in-pool ancestors to update, and those links are already gone, ancestors that remain in the pool never have their `descendants_fee`/`descendants_size`/`descendants_cycles`/`descendants_count` decremented and their `evict_key` is never recomputed. The result is an inflated `evict_key` on surviving ancestors, causing the pool to skip them during eviction and instead evict legitimate higher-fee transactions.

## Finding Description
`remove_entry_and_descendants` (L252–265) first iterates all nodes in the subtree and calls `remove_entry_links(id)` for each (L257–259). [1](#0-0) 

`remove_entry_links` (L418–430) removes `id` from its parents' children sets, removes all parents from `id`'s parent set, and then calls `self.links.remove(id)`, deleting the link record entirely. [2](#0-1) 

Only after all links are cleared does the function call `remove_entry(id)` for each node (L261–264). Inside `remove_entry` (L235–250), `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` is called at L242. [3](#0-2) 

`update_ancestors_index_key` (L432–445) computes `self.links.calc_ancestors(&child.proposal_short_id())` to find which in-pool ancestors to update. Because `id`'s link record was already deleted by `remove_entry_links`, `calc_ancestors` returns an empty set. No ancestor has `sub_descendant_weight` called on it, and no ancestor's `evict_key` is recomputed via `e.evict_key = e.inner.as_evict_key()` (L442). [4](#0-3) 

The `EvictKey` (entry.rs L234–247) is derived from `descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight)`. An ancestor P whose high-fee child C was removed via `remove_entry_and_descendants` retains C's contribution in `P.descendants_fee`, `P.descendants_size`, `P.descendants_cycles`, and `P.descendants_count`, yielding an inflated `evict_key`. [5](#0-4) 

`next_evict_entry` (L380–385) iterates `iter_by_evict_key()` ascending to find the lowest-priority entry to drop. P's stale high `evict_key` causes it to be skipped, and legitimate higher-fee transactions are evicted instead. [6](#0-5) 

The triggering call sites are `resolve_conflict` (L305–332) and `resolve_conflict_header_dep` (L267–292), both of which call `remove_entry_and_descendants` on subtrees whose roots may have in-pool parents. [7](#0-6) 

## Impact Explanation
An attacker can fill the pool with low-fee parent transactions whose `evict_key` is artificially inflated by now-removed high-fee children. `next_evict_entry` skips these stale-key parents and evicts legitimate higher-fee transactions instead. This constitutes a low-cost, repeatable attack that degrades miner block-template revenue and causes the pool to reject or evict legitimate high-fee transactions. Impact: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The trigger requires only the ability to submit transactions to the pool, which any unprivileged network participant can do. The attacker submits a low-fee parent P, then high-fee children to inflate P's `evict_key`, then a double-spend of one child to trigger `resolve_conflict` → `remove_entry_and_descendants`. This is a standard, frequent pool operation. The attack is repeatable at low cost and requires no key material, majority hashpower, or victim mistakes.

## Recommendation
In `remove_entry_and_descendants`, update the ancestors of the subtree root **before** clearing any links, so `update_ancestors_index_key` can still traverse them:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Update ancestors of the root BEFORE clearing links
    if let Some(root_entry) = self.get(id).cloned() {
        self.update_ancestors_index_key(&root_entry, EntryOp::Remove);
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

Additionally, suppress the redundant `update_ancestors_index_key` call inside `remove_entry` when invoked from this path (e.g., by checking whether the link record still exists before traversing ancestors), to avoid a no-op traversal on an already-cleared link.

## Proof of Concept
1. Insert parent P (fee=10, size=100) into the pool.
2. Insert child B (fee=500, size=100) spending P's output. P's `descendants_fee` becomes 510, `evict_key` is high.
3. Insert child C (fee=500, size=100) spending another of P's outputs. P's `descendants_fee` becomes 1010.
4. Submit B' (double-spend of B's input). `resolve_conflict` calls `remove_entry_and_descendants(B)`:
   - `remove_entry_links(B)` deletes B's link record (severs P→B).
   - `remove_entry(B)` calls `update_ancestors_index_key(B, Remove)` → `calc_ancestors(B)` returns `{}` → P not updated.
5. Pool state: P.`descendants_fee` = 1010 (stale, should be 510); P.`evict_key` remains inflated.
6. Fill the pool. `next_evict_entry()` iterates ascending by `evict_key`, skips P (stale high key), evicts a legitimate higher-fee transaction D (fee=50) instead.

A unit test can assert that after step 4, `pool_map.get(&P_id).unwrap().descendants_fee == 510` and the `evict_key` equals the correctly recomputed value. With the current code, both assertions fail.

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
