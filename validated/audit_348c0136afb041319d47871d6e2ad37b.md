Audit Report

## Title
`WellKnownScriptsOnly` Args Validation Bypass via `starts_with(&[])` Always-True — (`rpc/src/module/pool.rs`)

## Summary

`is_well_known_script` validates a submitted script's args using Rust's `starts_with` against the well-known script's args. All well-known scripts are hardcoded with `"args": "0x"` (empty bytes), making `well_known_script.args().as_slice()` always `&[]`. In Rust, `any_slice.starts_with(&[])` is unconditionally `true`, so the args field is never validated. Any script sharing a well-known `code_hash` and `hash_type` passes the `WellKnownScriptsOnly` validator regardless of its args content.

## Finding Description

`is_well_known_script` at lines 912–919 performs three checks: `hash_type` equality, `code_hash` equality, and an args prefix check using `starts_with`: [1](#0-0) 

All well-known lock scripts (anyone-can-pay, cheque) for both mainnet and testnet are hardcoded with `"args": "0x"`: [2](#0-1) 

All well-known type scripts (Simple UDT) are likewise hardcoded with `"args": "0x"`: [3](#0-2) 

Because `well_known_script.args().as_slice()` is always `&[]`, the third condition reduces to `script.args().as_slice().starts_with(&[])`, which is `true` for every possible byte slice in Rust. The args field is never validated.

The full exploit path is reachable by any unprivileged RPC caller:

1. Caller invokes `send_transaction` with `outputs_validator: WellKnownScriptsOnly` and a transaction whose output lock script uses a well-known `code_hash`+`hash_type` with arbitrary args.
2. `check_output_validator` dispatches to `WellKnownScriptsOnlyValidator::validate`: [4](#0-3) 
3. `validate_lock_script` falls through to `validate_well_known_lock_scripts` after the two secp256k1 checks fail (different `code_hash`): [5](#0-4) 
4. `validate_well_known_lock_scripts` calls `is_well_known_script`, which returns `true` due to the empty-prefix bypass: [6](#0-5) 

## Impact Explanation

The `WellKnownScriptsOnly` validator is the safety mechanism wallets and exchanges rely on when calling `send_transaction` to ensure outputs only use scripts with known-safe semantics. Because the args check is a no-op, any output whose lock script carries the anyone-can-pay or cheque `code_hash`+`hash_type` with arbitrary args passes the validator. A wallet or exchange relying on this validator would accept and broadcast such a transaction, potentially sending funds to an unspendable address (malformed args that no valid unlock witness can satisfy, permanently destroying CKB) or an attacker-controlled address (args crafted to be spendable by the attacker under the script's logic). This matches the allowed impact: **Critical — Vulnerabilities which could easily damage CKB economy**.

## Likelihood Explanation

The anyone-can-pay and cheque scripts are widely deployed on mainnet and testnet. Any attacker can craft a payment address using a well-known `code_hash` with non-standard args (e.g., 100 arbitrary bytes instead of the expected 20-byte lock hash). No special privileges are required — the attacker only needs to supply such an address to a wallet or exchange that uses `WellKnownScriptsOnly`. The validator will accept it without further scrutiny. The attack is repeatable and requires no on-chain state or prior access.

## Recommendation

Replace the `starts_with` comparison with an exact equality check on args:

```rust
fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script.args() == well_known_script.args()
}
```

If prefix-matching is intentionally desired for future well-known scripts that carry non-empty args, add a guard `!well_known_script.args().is_empty()` before the `starts_with` call to prevent the empty-prefix bypass.

## Proof of Concept

```rust
use ckb_types::{packed, prelude::*};

fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script
            .args()
            .as_slice()
            .starts_with(well_known_script.args().as_slice())
}

#[test]
fn prove_args_bypass() {
    // well-known anyone-can-pay script with empty args, as hardcoded in build_well_known_lock_scripts
    let well_known = packed::Script::new_builder()
        .code_hash([0xd3u8; 32].pack())
        .hash_type(1u8.into())
        .args(packed::Bytes::default()) // "args": "0x"
        .build();

    // attacker script: same code_hash/hash_type, 100 arbitrary bytes as args
    let attacker_script = packed::Script::new_builder()
        .code_hash([0xd3u8; 32].pack())
        .hash_type(1u8.into())
        .args(vec![0xffu8; 100].pack())
        .build();

    // Should be false. Actual result: true — bypass confirmed.
    assert!(
        is_well_known_script(&attacker_script, &well_known),
        "BUG: starts_with(&[]) is always true; args are never validated"
    );
}
```

Running this unit test against the code at lines 912–919 confirms the invariant violation: `starts_with(&[])` returns `true` unconditionally, so the args field is never validated.
