### Title
Unchecked `u64` Multiplication in `total_fee` Enables Fee Bypass in Fuel Price-Feed Updates — (`File: target_chains/fuel/contracts/pyth-interface/src/utils.sw`)

---

### Summary

The `total_fee` helper in the Fuel Pyth contract multiplies `total_number_of_updates` by `single_update_fee` using the bare Sway `*` operator on two `u64` values. The FuelVM's `MUL` opcode silently wraps the result to the lower 64 bits on overflow; the Sway compiler adds no overflow check. An unprivileged caller who submits enough update-data blobs can make the computed fee wrap to zero (or a small value), pass the `require(msg_amount() >= required_fee)` guard, and have all price-feed state changes committed without paying the correct fee.

---

### Finding Description

**Root cause — `total_fee`:**

```sway
// target_chains/fuel/contracts/pyth-interface/src/utils.sw  line 16
pub fn total_fee(
    total_number_of_updates: u64,
    single_update_fee: StorageKey<u64>,
) -> u64 {
    total_number_of_updates * single_update_fee.read()   // ← unchecked u64 × u64
}
``` [1](#0-0) 

The FuelVM `MUL` opcode stores the upper 64 bits of the product in the overflow register `$of` and the lower 64 bits in the destination register. Sway's `*` operator emits this opcode directly and never inspects `$of`, so the returned value silently wraps.

**Call site 1 — `update_fee` (read-only fee query):**

```sway
// main.sw  line 374
total_fee(total_number_of_updates, storage.single_update_fee)
``` [2](#0-1) 

**Call site 2 — `update_price_feeds` (state-mutating, critical):**

```sway
// main.sw  lines 377-429
fn update_price_feeds(update_data: Vec<Bytes>) {
    // 1. All price-feed state changes happen here (storage writes)
    total_number_of_updates += number_of_updates;

    // 2. Fee computed AFTER state is already written
    let required_fee = total_fee(total_number_of_updates, storage.single_update_fee);

    // 3. Guard checked against potentially-wrapped value
    require(msg_amount() >= required_fee, PythError::InsufficientFee);
}
``` [3](#0-2) 

The `number_of_updates` per accumulator blob is a single byte (max 255) read from the wire:

```sway
// accumulator_update.sw  line 85-92
let number_of_updates = encoded_slice.get(offset);   // u8 → u64
``` [4](#0-3) 

An attacker submits many blobs so that `total_number_of_updates` accumulates to a value `N` where `N × single_update_fee mod 2^64` is small (or zero).

---

### Impact Explanation

- **Fee bypass**: The attacker pays a negligible (or zero) fee while having all price-feed storage writes committed. In Fuel, a transaction revert rolls back state; but if the wrapped fee value is ≥ `msg_amount()`, the `require` passes and the transaction succeeds with state committed.
- **Protocol revenue loss**: Every `update_price_feeds` call that exploits the wrap pays far less than the configured `single_update_fee × N`.
- **Incorrect `update_fee` return value**: The read-only `update_fee` ABI function also calls `total_fee`, so callers who query the fee before submitting will receive a wrapped (incorrect) value and may under-pay or be misled.

---

### Likelihood Explanation

- The entry point (`update_price_feeds`) is permissionless — any transaction sender qualifies.
- The attacker controls `update_data` length and content, directly controlling `total_number_of_updates`.
- The current `single_update_fee` is 1 (minimum), making overflow require an astronomically large `N`. However, governance can raise the fee at any time; at `single_update_fee = 2^32`, only `2^32` updates are needed to wrap to zero. The structural bug is present regardless of the current fee value.
- No privileged access, leaked key, or oracle manipulation is required.

---

### Recommendation

Replace the bare `*` with a checked or saturating multiply and revert on overflow:

```sway
// utils.sw
pub fn total_fee(
    total_number_of_updates: u64,
    single_update_fee: StorageKey<u64>,
) -> u64 {
    let fee = single_update_fee.read();
    // Use checked_mul equivalent; revert if overflow
    require(
        total_number_of_updates == 0 || fee <= u64::max() / total_number_of_updates,
        PythError::FeeOverflow,
    );
    total_number_of_updates * fee
}
```

Additionally, move the fee check in `update_price_feeds` to **before** any storage writes (as already done correctly in `parse_price_feed_updates`), so that an overflow-induced revert cannot be triggered after state has been mutated.

---

### Proof of Concept

Assume `single_update_fee = 2^32` (governance-settable).

1. Attacker constructs `2^32` valid accumulator update blobs (each containing 1 update), so `total_number_of_updates = 2^32`.
2. `total_fee` computes `2^32 × 2^32 = 2^64 ≡ 0 (mod 2^64)` → `required_fee = 0`.
3. Attacker calls `update_price_feeds` with `msg_amount = 0`.
4. `require(0 >= 0)` passes.
5. All `2^32` price-feed storage writes are committed; attacker paid zero fee.

At the current fee of 1, the wrap point is `N = 2^64`, which is impractical. The risk materialises immediately if governance raises the fee to any value ≥ 2 and the attacker can submit enough blobs within Fuel's transaction-size limits.
