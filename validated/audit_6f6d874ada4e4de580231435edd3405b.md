Audit Report

## Title
RBF Replacement Permanently DoS-able via Descendant Flooding — (File: tx-pool/src/pool.rs)

## Summary
`TxPool::check_rbf` enforces a hard cap of 100 replacement candidates by summing all descendants of every conflicting transaction, but `PoolMap` has no `max_descendants_count` admission guard — only `max_ancestors_count`. An attacker can pre-flood the pool with ≥100 descendants of a victim's pending transaction, causing every subsequent RBF replacement attempt to be permanently rejected with `"Tx conflict with too many txs"`.

## Finding Description
`MAX_REPLACEMENT_CANDIDATES` is fixed at 100: [1](#0-0) 

`check_rbf` counts all descendants of each conflicting transaction and rejects if the total exceeds 100: [2](#0-1) 

`PoolMap` exposes `max_ancestors_count` but has no corresponding `max_descendants_count` field: [3](#0-2) 

`check_and_record_ancestors` enforces the ancestor limit on admission, but there is no symmetric descendant enforcement anywhere in the codebase (confirmed by zero matches for `max_descendants_count`): [4](#0-3) 

`check_rbf` is invoked inside the write lock in `submit_entry`, so the descendant count it observes is the live pool state at the moment of the replacement attempt: [5](#0-4) 

**Attack flow:**
1. Victim submits `T1` (with ≥2 outputs) to the pool.
2. Attacker observes `T1` via `get_raw_tx_pool` RPC or P2P relay.
3. Attacker builds a 7-level binary tree of descendants rooted at `T1`: 2+4+8+16+32+64+128 = 254 descendants. Each descendant has ≤7 ancestors — well within the default `max_ancestors_count` of 25 — so every descendant is accepted by the pool.
4. Victim submits `T2` (same inputs as `T1`, higher fee) to RBF-replace `T1`.
5. `check_rbf` counts 254 descendants of `T1` → `replace_count = 255 > 100` → returns `Reject::RBFRejected("Tx conflict with too many txs …")`.
6. The victim's replacement is rejected indefinitely as long as the attacker maintains ≥100 descendants.

## Impact Explanation
This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** An attacker can flood the pool with minimum-fee descendant transactions at negligible cost (~100 txs) to permanently block legitimate RBF replacements. The attack can be applied simultaneously to multiple victims' transactions, degrading the RBF mechanism across the network. Time-sensitive transactions (e.g., those using `since` locks near expiry) cannot be fee-bumped and may expire from the pool without confirmation.

## Likelihood Explanation
Medium. RBF must be enabled (`min_rbf_rate > min_fee_rate`), which is opt-in but documented and used. The attacker requires only ~100 valid transactions with minimum fees, no privileged access, and no cryptographic material beyond their own keys. The victim's pending transaction is publicly observable via RPC or P2P relay. The attacker bears no loss if `T1` is eventually confirmed (their descendants confirm too). If the pool evicts some descendants, the attacker can cheaply resubmit them.

## Recommendation
Introduce a `max_descendants_count` field in `PoolMap` (symmetric to `max_ancestors_count`) and enforce it on admission in `add_entry`. When a new transaction would push any of its ancestors' descendant counts above the limit, reject it with `Reject::ExceededMaximumDescendantsCount`. This mirrors Bitcoin Core's `limitdescendantcount` policy and makes the descendant-flooding attack infeasible. Additionally, document `MAX_REPLACEMENT_CANDIDATES` as a DoS surface if no descendant cap is enforced.

## Proof of Concept
```
1. Node configured with min_rbf_rate > min_fee_rate (RBF enabled).

2. Victim submits T1 with 2 outputs (out[0], out[1]) to the pool.

3. Attacker builds a 7-level binary tree rooted at T1:
   - Level 1: C1_0 spends T1.out[0], C1_1 spends T1.out[1]          (2 txs)
   - Level 2: C2_00 spends C1_0.out[0], C2_01 spends C1_0.out[1],
              C2_10 spends C1_1.out[0], C2_11 spends C1_1.out[1]    (4 txs)
   - ...
   - Level 7: 128 txs
   Total descendants: 254. Each has ≤7 ancestors → accepted by pool
   (max_ancestors_count default = 25).

4. Victim submits T2 (same inputs as T1, fee > T1.fee + min_rbf_fee):
   check_rbf:
     conflict_ids = {T1}
     descendants of T1 = 254
     replace_count = 254 + 1 = 255 > MAX_REPLACEMENT_CANDIDATES (100)
     → Err(RBFRejected("Tx conflict with too many txs, conflict txs count: 255, expect <= 100"))

5. T2 is rejected. Victim cannot replace T1.
   Attacker resubmits any evicted descendants to maintain the block.
   Attack is sustained indefinitely at minimum-fee cost.
```
