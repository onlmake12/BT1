### Title
Excess ETH Not Refunded to User in `Entropy::requestHelper()` — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, all public randomness request entry points (`request`, `requestWithCallback`, `requestV2`) route through `requestHelper`, which accepts `msg.value >= requiredFee` but silently absorbs any excess ETH into `_state.accruedPythFeesInWei` instead of refunding it to the caller. The user permanently loses the overpaid amount.

---

### Finding Description

`requestHelper` performs the following fee accounting:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The check on line 235 only requires `msg.value >= requiredFee`. When `msg.value > requiredFee`, the entire surplus above `providerFee` is credited to `_state.accruedPythFeesInWei` — the Pyth protocol treasury — with no refund to `msg.sender`. This is explicitly acknowledged in the NatSpec of the public interface:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

All three public entry points are affected:

- `request(address provider, bytes32 userCommitment, bool useBlockHash)` [4](#0-3) 
- `requestWithCallback(address provider, bytes32 userContribution)` [5](#0-4) 
- `requestV2(address provider, bytes32 userContribution, uint32 gasLimit)` [6](#0-5) 

---

### Impact Explanation

Any user who sends `msg.value` exceeding the exact required fee permanently loses the surplus ETH to the Pyth protocol fee pool (`accruedPythFeesInWei`). There is no mechanism for the user to recover this ETH. The Pyth admin can later withdraw it via governance, but the user has no recourse.

Realistic overpayment scenarios:
1. **Fee decrease between `getFee()` call and tx inclusion**: A user queries `getFee(provider)` and sends that amount, but the provider lowers their fee before the tx is mined — the user overpays.
2. **Integrating contracts that forward `msg.value` directly**: Wrapper contracts that pass through the full `msg.value` without computing the exact fee will silently donate the surplus to Pyth.
3. **Manual/scripted callers rounding up**: Users or bots who add a buffer to ensure the tx doesn't revert lose the buffer permanently.

---

### Likelihood Explanation

Medium. The behavior is documented in the NatSpec, so sophisticated integrators are warned. However, naive callers, wrapper contracts, and users experiencing fee changes between query and execution will routinely overpay. The Entropy contract is a live production contract with real ETH at stake on every randomness request.

---

### Recommendation

Either:

1. **Refund excess ETH** to `msg.sender` after fee accounting:
   ```solidity
   if (msg.value > requiredFee) {
       payable(msg.sender).transfer(msg.value - requiredFee);
   }
   ```
   And then use `requiredFee` (not `msg.value`) in the Pyth fee accounting line:
   ```solidity
   _state.accruedPythFeesInWei += (requiredFee - providerFee);
   ```

2. **Revert if `msg.value != requiredFee`**, enforcing exact payment.

Option 1 is preferred as it improves UX without breaking callers that slightly overpay.

---

### Proof of Concept

1. Provider registers with a fee of 1000 wei. User calls `getFee(provider)` → returns 1000 wei.
2. Provider updates their fee to 500 wei before the user's tx is mined.
3. User's tx executes with `msg.value = 1000 wei` and `requiredFee = 500 wei`.
4. Line 235: `1000 >= 500` → no revert.
5. `providerFee = 500`, `providerInfo.accruedFeesInWei += 500`.
6. `_state.accruedPythFeesInWei += (1000 - 500) = 500` — the 500 wei surplus goes to Pyth, not the user.
7. User loses 500 wei with no recourse. [7](#0-6)

        address provider,
        bytes32 userContribution,
        uint32 gasLimit
    ) public payable override returns (uint64) {
        EntropyStructsV2.Request storage req = requestHelper(
            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.
            // If we remove the blockHash from this, the provider would have no choice but to provide its committed
            // random number. Hence, useBlockHash is set to false.
            false,
            true,
            gasLimit
        );

        emit RequestedWithCallback(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            EntropyStructConverter.toV1Request(req)
        );
        emit EntropyEventsV2.Requested(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            uint32(req.gasLimit10k) * TEN_THOUSAND,
            bytes("")
        );
        return req.sequenceNumber;
    }
