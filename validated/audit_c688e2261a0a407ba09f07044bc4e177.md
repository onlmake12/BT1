### Title
PostgreSQL-only copy-paste bug in `script_exists_in_output` silently deletes live type-script entries during reorg — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

`script_exists_in_output` contains a copy-paste error on line 252: the final `match` branch reads `row_lock` instead of `row_type`. On PostgreSQL this causes the type-script existence check to be silently discarded, so any script referenced **only** as a `type_script_id` in surviving outputs is incorrectly deleted during a rollback. SQLite is unaffected because its type mismatch forces the correct fallback path.

---

### Finding Description

The function fetches two rows:

- `row_lock` — `SELECT EXISTS(… WHERE lock_script_id = $1)`
- `row_type` — `SELECT EXISTS(… WHERE type_script_id = $1)`

After the lock check fails to return early, the final decision should read `row_type`. Instead it re-reads `row_lock`:

```
// line 252 — BUG: row_lock used instead of row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),          // always Ok(false) on PG → script deleted
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),  // SQLite: correct
}
``` [1](#0-0) 

**PostgreSQL path:** `row_lock.try_get::<bool, _>(0)` succeeds (PG returns `BOOLEAN`). Since the lock check already returned `false`, this call returns `Ok(false)`. The `row_type` result is never consulted. The function returns `Ok(false)` → script is pushed onto `script_id_list_to_remove` → deleted.

**SQLite path:** `row_lock.try_get::<bool, _>(0)` fails (SQLite returns `BIGINT`). The `Err(_)` arm executes `row_type.get::<i64, _>(0) == 1`, which correctly reads the type-script existence result. [2](#0-1) 

The deletion is then executed unconditionally: [3](#0-2) 

---

### Impact Explanation

After `rollback_block` completes on PostgreSQL:

1. The `script` row for script S is deleted.
2. Surviving outputs in block B1 still hold `type_script_id = S.id` — a dangling foreign-key reference (or silent NULL depending on FK enforcement).
3. All rich-indexer RPC calls that join on `script` for that `type_script_id` return empty results.
4. The corruption is permanent until a full re-sync; no self-healing path exists.

---

### Likelihood Explanation

- Reorgs are a normal, unprivileged network event. Any block relayer can cause one by propagating a competing chain tip.
- The bug fires on every reorg where a script appears **only** as a `type_script_id` (no `lock_script_id` usage) in the rolled-back block while still referenced in a surviving block.
- The PostgreSQL backend is a supported production deployment option for the rich indexer.
- The differential is deterministic and locally reproducible.

---

### Recommendation

Change line 252 from `row_lock` to `row_type`:

```rust
// correct
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

---

### Proof of Concept

```
1. Start rich-indexer with PostgreSQL backend.
2. Append block B1 containing tx with output: lock=L, type=S.
3. Append block B2 (tip) containing tx with output: lock=L2, type=S  (same S, different lock).
4. Call rollback_block() to remove B2.
5. Query: SELECT COUNT(*) FROM script WHERE id = <S.id>
   → PostgreSQL returns 0  (BUG: S deleted despite B1 still referencing it)
   → SQLite returns 1      (correct)
6. Query get_cells for script S on PostgreSQL → empty result set.
```
