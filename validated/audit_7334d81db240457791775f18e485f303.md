### Title
`MAX_DEPOSIT_LIMIT` Invariant Broken for Permanent Subscriptions via Cumulative `addFunds` Calls — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `addFunds` function in `Scheduler.sol` checks only whether the single incoming deposit (`msg.value`) exceeds `MAX_DEPOSIT_LIMIT`, not whether the **post-deposit cumulative balance** (`status.balanceInWei + msg.value`) would exceed it. This allows anyone to repeatedly top up a permanent subscription in increments ≤ `MAX_DEPOSIT_LIMIT`, accumulating a balance far beyond the intended cap and permanently locking an unbounded amount of ETH in the contract.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
``` [1](#0-0) 

`Scheduler.sol`'s `addFunds` enforces this limit as:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

status.balanceInWei += msg.value;
``` [2](#0-1) 

The guard compares only `msg.value` against `MAX_DEPOSIT_LIMIT`. It does **not** check `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`. Because `addFunds` has no access control (any address may call it), an attacker can call it N times with `msg.value = MAX_DEPOSIT_LIMIT` each time, passing the check on every call while the stored balance grows to `N × 100 ether`.

For comparison, `createSubscription` has the same per-call check at line 48, but there the balance starts at zero so `msg.value` equals the total balance — the check is accidentally correct only at creation time. [3](#0-2) 

---

### Impact Explanation

Permanent subscriptions are designed to be irrevocable: `withdrawFunds` unconditionally reverts for them, and `updateSubscription` also reverts. Any ETH deposited into a permanent subscription is **permanently locked** in the contract. [4](#0-3) 

`MAX_DEPOSIT_LIMIT` is the sole protocol safeguard against unbounded permanent ETH lock-up. By bypassing it, an attacker (or a subscription manager acting maliciously) can lock an arbitrarily large amount of ETH permanently, breaking the intended invariant that no single permanent subscription holds more than 100 ETH.

---

### Likelihood Explanation

- `addFunds` is a public, payable function with **no access control** — any EOA or contract can call it for any active subscription ID.
- The exploit requires only repeated calls with `msg.value ≤ MAX_DEPOSIT_LIMIT`; no privileged role, leaked key, or external oracle is needed.
- The subscription manager themselves may also exploit this inadvertently or deliberately.

---

### Recommendation

Replace the per-call check with a post-deposit cumulative check:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent) {
    if (status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }
}

status.balanceInWei += msg.value;
```

Apply the same fix to `createSubscription` for consistency (though it is not currently exploitable there since the balance starts at zero).

---

### Proof of Concept

```solidity
// Assume MAX_DEPOSIT_LIMIT = 100 ether
// 1. Create a permanent subscription with exactly MAX_DEPOSIT_LIMIT
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);
// status.balanceInWei == 100 ether — at the cap

// 2. Call addFunds repeatedly; each call passes the check (msg.value == MAX_DEPOSIT_LIMIT, not >)
scheduler.addFunds{value: 100 ether}(subId); // balance = 200 ether — cap bypassed
scheduler.addFunds{value: 100 ether}(subId); // balance = 300 ether
// ... repeat N times → balance = (N+1) × 100 ether, all permanently locked
```

The check `msg.value > MAX_DEPOSIT_LIMIT` passes for `msg.value == MAX_DEPOSIT_LIMIT` (no revert), so the cap is bypassed on every subsequent call. Since `withdrawFunds` reverts for permanent subscriptions, all accumulated ETH is irrecoverably locked. [5](#0-4)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
