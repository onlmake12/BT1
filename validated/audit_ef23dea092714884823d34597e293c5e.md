### Title
Unbounded `tx.gasprice` in `_processFeesAndPayKeeper()` Allows Keeper to Drain Subscription Balance - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler._processFeesAndPayKeeper()` computes the keeper reimbursement as `(gasUsed + GAS_OVERHEAD) * tx.gasprice` with no upper bound on `tx.gasprice`. Because `updatePriceFeeds()` is permissionless, any caller can submit the transaction with an arbitrarily inflated gas price, extracting the subscription's entire deposited balance in a single call.

---

### Finding Description

`updatePriceFeeds()` is callable by any address acting as a keeper. At the end of the call it invokes `_processFeesAndPayKeeper()`:

```solidity
// Scheduler.sol line 846
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;

if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}

status.balanceInWei -= totalKeeperFee;
// ...
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
```

`tx.gasprice` is a property of the submitted transaction and is entirely under the caller's control. There is no configured maximum gas price, no cap constant in `SchedulerConstants.sol`, and no comparison against a reference market gas price. The only guard is the balance check, which passes as long as the subscription has funds — meaning the attacker simply sets `tx.gasprice` high enough to consume the full balance in one shot.

The entry path is:
1. Attacker observes an active subscription with a meaningful balance.
2. Attacker waits for (or triggers) a valid update condition (heartbeat or deviation threshold).
3. Attacker calls `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice` set to an extreme value (e.g., `balance / (gasUsed + GAS_OVERHEAD)`).
4. `_processFeesAndPayKeeper` computes `totalKeeperFee ≈ balance`, passes the balance check, deducts the full balance, and transfers it to the attacker.

---

### Impact Explanation

A subscription owner deposits ETH to fund automated price updates. A malicious keeper can drain the entire subscription balance in a single transaction by inflating `tx.gasprice`. The subscription owner loses all deposited funds. For permanent subscriptions (which cannot be withdrawn from by the owner), the loss is total and irrecoverable.

---

### Likelihood Explanation

`updatePriceFeeds()` has no access control — any EOA or contract can call it. The attacker only needs to satisfy the update trigger condition (heartbeat or price deviation), which is publicly observable on-chain. The attack requires no special privileges, no leaked keys, and no governance majority. It is executable by any unprivileged actor on any chain where the Scheduler is deployed.

---

### Recommendation

Introduce a maximum gas price cap. Either:
- Add a `maxGasPriceInWei` parameter to `SchedulerConstants` or as an admin-configurable state variable, and revert in `_processFeesAndPayKeeper` if `tx.gasprice > maxGasPriceInWei`.
- Alternatively, cap the reimbursed gas price at a multiple of the block base fee (e.g., `min(tx.gasprice, block.basefee * 2)`), which is manipulation-resistant on EIP-1559 chains.

---

contract DrainSubscription {

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
