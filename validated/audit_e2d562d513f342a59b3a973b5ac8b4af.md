### Title
Missing `ScriptHashType` Validation for Output Type Scripts in `ScriptHashTypeVerifier` — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` consensus constraint only on the **lock script** of each transaction output. The **type script** of each output is never checked. An unprivileged transaction sender can craft an output whose type script carries a disallowed hash type (e.g., `Data3 = 6`), bypass all non-contextual verification, and force the node to admit the transaction into the tx pool before it fails during expensive contextual script execution.

---

### Finding Description

`ScriptHashTypeVerifier` is part of `NonContextualTransactionVerifier`, the cheap, context-free gate that runs before any script execution or chain-state lookup.

`ENABLED_SCRIPT_HASH_TYPE` is defined as:

```rust
// util/constant/src/consensus.rs
pub static ENABLED_SCRIPT_HASH_TYPE: Set<u8> = phf_set! {
    0u8, // Data
    1u8, // Type
    2u8, // Data1
    4u8, // Data2
};
```

Values `6` (`Data3`), `8` (`Data4`), … are **not** in this set. [1](#0-0) 

`ScriptHashType::verify_value` — used by `check_data()` — accepts any even number or `1`, so `Data3 = 6` passes the molecule-level sanity check:

```rust
pub fn verify_value(v: u8) -> bool {
    v.is_multiple_of(2) || v == 1
}
``` [2](#0-1) 

`check_data()` on a `CellOutput` checks both lock and type scripts for structural validity, but only for known variants — not for whether the variant is in `ENABLED_SCRIPT_HASH_TYPE`:

```rust
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()
    }
}
``` [3](#0-2) 

`ScriptHashTypeVerifier::verify()` then iterates over outputs and checks **only** `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...ScriptHashTypeNotPermitted...);
            }
        } else {
            return Err(...InvalidScriptHashType...);
        }
    }
    Ok(())
}
```

`output.type_()` is never inspected. [4](#0-3) 

The `NonContextualTransactionVerifier` doc comment even acknowledges the gap: it says "Check whether output **lock** hash type within enabled range" — the type script is absent by design or omission. [5](#0-4) 

When the transaction later reaches contextual verification, `extract_script_and_dep_index` in the script engine hits the catch-all arm and returns `ScriptError::InvalidScriptHashType`:

```rust
hash_type => {
    return Err(ScriptError::InvalidScriptHashType(format!(
        "The ScriptHashType/{:?} has not been activated...",
        hash_type
    )));
}
``` [6](#0-5) 

---

### Impact Explanation

A transaction with an output whose type script uses `Data3` (value `6`) or any other non-enabled hash type:

1. Passes `check_data()` (structurally valid variant).
2. Passes `ScriptHashTypeVerifier` (only lock is checked).
3. Passes all remaining non-contextual checks (version, size, empty, duplicate deps, outputs-data length).
4. Is admitted to the tx pool after fee payment.
5. Fails during contextual script execution — wasting CPU and pool slots.

An attacker paying the minimum fee rate can continuously inject such transactions, filling the tx pool with entries that will never confirm, displacing legitimate transactions and forcing repeated expensive script-execution failures on every node that relays or mines the transaction.

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction via RPC (`send_transaction`) or P2P relay — no privileged access, no key material, no majority hashpower. The only cost is the minimum fee rate per transaction. The `ScriptHashType` enum exposes many non-enabled variants (`Data3`–`Data127`) that are trivially encodable in a crafted output. The gap is structural and reproducible on every node running this code.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type for each output, mirroring the existing lock script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        let lock_ht = output.lock().hash_type();
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(lock_ht) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType { hash_type: lock_ht }.into());
        }

        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_ht = type_script.hash_type();
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_ht) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType { hash_type: type_ht }.into());
            }
        }
    }
    Ok(())
}
```

The `TransactionError` variants `ScriptHashTypeNotPermitted` and `InvalidScriptHashType` already exist and are appropriate for this case. [7](#0-6) 

---

### Proof of Concept

1. Build a transaction with one output whose **type script** has `hash_type = 6` (`Data3`) and whose lock script has `hash_type = 0` (`Data`).
2. Submit via `send_transaction` RPC.
3. Observe: `NonContextualTransactionVerifier` (including `ScriptHashTypeVerifier`) returns `Ok(())`.
4. The transaction enters the tx pool.
5. During contextual verification, `TransactionScriptsVerifier` returns `ScriptError::InvalidScriptHashType("Data3 has not been activated")`.
6. Repeat at minimum fee rate to exhaust pool capacity.

The existing test `test_not_enabled_hash_type_output_lock` confirms the lock path is guarded; no analogous test exists for the type script path, confirming the gap. [8](#0-7)

/// Context-independent verification checks for transaction
///
/// Basic checks that don't depend on any context
/// Contains:
/// - Check for version
/// - Check for size
/// - Check inputs and output empty
/// - Check for duplicate deps
/// - Check for whether outputs match data
/// - Check whether output lock hash type within enabled range
pub struct NonContextualTransactionVerifier<'a> {
    pub(crate) version: VersionVerifier<'a>,
    pub(crate) size: SizeVerifier<'a>,
    pub(crate) empty: EmptyVerifier<'a>,
    pub(crate) duplicate_deps: DuplicateDepsVerifier<'a>,
    pub(crate) outputs_data_verifier: OutputsDataVerifier<'a>,
    pub(crate) script_hash_type: ScriptHashTypeVerifier<'a>,
}

impl<'a> NonContextualTransactionVerifier<'a> {
    /// Creates a new NonContextualTransactionVerifier
    pub fn new(tx: &'a TransactionView, consensus: &'a Consensus) -> Self {
        NonContextualTransactionVerifier {
            version: VersionVerifier::new(tx, consensus.tx_version()),
            size: SizeVerifier::new(tx, consensus.max_block_bytes()),
            empty: EmptyVerifier::new(tx),
            duplicate_deps: DuplicateDepsVerifier::new(tx),
            outputs_data_verifier: OutputsDataVerifier::new(tx),
            script_hash_type: ScriptHashTypeVerifier::new(tx),
        }
    }

    /// Perform context-independent verification
    pub fn verify(&self) -> Result<(), Error> {
        self.version.verify()?;
        self.size.verify()?;
        self.empty.verify()?;
        self.duplicate_deps.verify()?;
        self.outputs_data_verifier.verify()?;
        self.script_hash_type.verify()?;
        Ok(())
    }
}
