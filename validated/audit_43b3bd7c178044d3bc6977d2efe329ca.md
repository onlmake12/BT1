Audit Report

## Title
Pool Size Limit (`max_tx_pool_size`) Not Enforced After Reorg Re-insertion — (File: `tx-pool/src/process.rs`)

## Summary
`readd_detached_tx` re-inserts detached transactions via `_submit_entry` during chain reorganization without calling `limit_size`, while the normal `submit_entry` path unconditionally calls `limit_size` after `_submit_entry`. This allows `total_tx_size` to exceed the operator-configured `max_tx_pool_size` after any reorg, with the violation persisting until the next externally submitted transaction triggers eviction.

## Finding Description
The divergence is confirmed in the actual code:

**Normal path** (`submit_entry`, `tx-pool/src/process.rs` lines 137–152): calls `_submit_entry` at line 137, then unconditionally calls `tx_pool.limit_size(...)` at line 151. [1](#0-0) 

**Reorg path** (`readd_detached_tx`, `tx-pool/src/process.rs` lines 878–914): calls `_submit_entry` at line 906 inside a `for` loop with no `limit_size` call anywhere in the function body. [2](#0-1) 

`update_tx_pool_for_reorg` (lines 802–858) calls `readd_detached_tx` at line 849 and also contains no `limit_size` call. [3](#0-2) 

A `grep` across all `tx-pool/src/*.rs` files confirms `limit_size` appears only twice in `process.rs`, both within the normal `submit_entry` closure. `limit_size` in `tx-pool/src/pool.rs` (lines 292–329) is the sole enforcement point, looping while `total_tx_size > max_tx_pool_size` and evicting lowest-fee-rate entries — it is never reached from the reorg path. [4](#0-3) 

`check_tx_fee` (called at line 889) validates only the minimum fee rate of the individual transaction; it does not consult or enforce the aggregate pool size. Each `_submit_entry` call increments `pool_map.total_tx_size` with no eviction, so the pool's memory footprint grows unbounded relative to `max_tx_pool_size` for the duration of the reorg re-insertion.

**Exploit flow:**
1. Pool is at or near `max_tx_pool_size`.
2. A competing block at the same height arrives (natural one-block reorg).
3. `update_tx_pool_for_reorg` → `readd_detached_tx` re-inserts all transactions from the detached block not present in the new tip.
4. Each `_submit_entry` call increments `total_tx_size` with no eviction.
5. `total_tx_size` now exceeds `max_tx_pool_size` by up to the aggregate size of the detached block's transactions.
6. Violation persists until the next externally submitted transaction reaches `submit_entry` and triggers `limit_size`.

## Impact Explanation
This is a concrete violation of the tx-pool's size-bounding invariant. The tx-pool is the node's primary in-memory transaction state store, and `max_tx_pool_size` is its key resource constraint. The pool's memory footprint can exceed the operator-configured bound after every reorg, with the excess bounded by the detached block's transaction payload. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation
One-block reorgs are routine in any PoW network and require no attacker involvement — they occur naturally when two miners find blocks at the same height near-simultaneously. Any peer acting as a block relayer can deliver a valid competing block to trigger the path. No special privileges, majority hashpower, or victim mistakes are required. The condition is repeatable on every reorg event.

## Recommendation
After the `for` loop in `readd_detached_tx` completes, call `limit_size` to restore the invariant:

```rust
// tx-pool/src/process.rs, after the for-loop in readd_detached_tx
tx_pool.limit_size(&self.callbacks, None);
```

Alternatively, refactor `readd_detached_tx` to call the full `submit_entry` method (which already includes the `limit_size` call) instead of the bare `_submit_entry` free function, consistent with the normal submission path.

## Proof of Concept
1. Configure a `TxPool` with a small `max_tx_pool_size` (e.g., 10 KB).
2. Fill the pool to exactly `max_tx_pool_size` with pending transactions.
3. Simulate a one-block reorg: call `update_tx_pool_for_reorg` with a `detached_blocks` containing transactions totaling >1 KB and an `attached_blocks` with different transactions.
4. After the call returns, assert `tx_pool.pool_map.total_tx_size <= max_tx_pool_size`.
5. Without the fix, the assertion fails — `total_tx_size` exceeds the configured limit by the size of the re-added detached transactions.

    async fn readd_detached_tx(
        &self,
        tx_pool: &mut TxPool,
        txs: Vec<TransactionView>,
        fetched_cache: HashMap<Byte32, CacheEntry>,
    ) {
        let max_cycles = self.tx_pool_config.max_tx_verify_cycles;
        for tx in txs {
            let tx_size = tx.data().serialized_size_in_block();
            let tx_hash = tx.hash();
            if let Ok((rtx, status)) = resolve_tx(tx_pool, tx_pool.snapshot(), tx, false)
                && let Ok(fee) = check_tx_fee(tx_pool, tx_pool.snapshot(), &rtx, tx_size)
            {
                let verify_cache = fetched_cache.get(&tx_hash).cloned();
                let snapshot = tx_pool.cloned_snapshot();
                let tip_header = snapshot.tip_header();
                let tx_env = Arc::new(status.with_env(tip_header));
                if let Ok(verified) = verify_rtx(
                    snapshot,
                    Arc::clone(&rtx),
                    tx_env,
                    &verify_cache,
                    max_cycles,
                    None,
                )
                .await
                {
                    let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
                    if let Err(e) = _submit_entry(tx_pool, status, entry, &self.callbacks) {
                        error!("readd_detached_tx submit_entry {} error {}", tx_hash, e);
                    } else {
                        debug!("readd_detached_tx submit_entry {}", tx_hash);
                    }
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
