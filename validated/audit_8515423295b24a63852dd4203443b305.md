Audit Report

## Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` With Pre-Eviction Snapshot, Permanently Inflating Pool Accounting — (File: tx-pool/src/component/pool_map.rs)

## Summary
In `add_entry`, `updated_stat_for_add_tx` computes a snapshot of the new totals before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts cell-ref-parent transactions, it decrements `self.total_tx_size` and `self.total_tx_cycles` in-place via `update_stat_for_remove_tx`. The stale pre-eviction snapshot is then unconditionally written back at lines 218–219, permanently re-inflating the counters by the sizes and cycles of every evicted transaction. This inflation is cumulative and can be driven arbitrarily high by any unprivileged RPC caller, eventually causing the pool to reject all incoming transactions while nearly empty.

## Finding Description
In `add_entry` (L200–221), the ordering is:

```
// (A) snapshot = old_total + new_entry — local variable, not yet written to self
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // L210–211

// (B) may evict cell-ref parents → remove_entry_and_descendants
//     → update_stat_for_remove_tx → self.total_tx_size -= evicted_size  (in-place, L738–740)
evicts = self.check_and_record_ancestors(&mut entry)?;          // L213

// (C) OVERWRITES self.total_tx_size with the stale pre-eviction snapshot
self.total_tx_size = total_tx_size;                             // L218
self.total_tx_cycles = total_tx_cycles;                         // L219
```

`updated_stat_for_add_tx` (L711–728) is a pure read: it returns `self.total_tx_size + tx_size` as a local value without modifying `self`. `update_stat_for_remove_tx` (L733–757) is a write: it directly assigns `self.total_tx_size = self.total_tx_size - evicted_size`. Because step (C) unconditionally overwrites `self.total_tx_size` with the value from step (A), every decrement performed in step (B) is silently discarded.

The eviction branch in `check_and_record_ancestors` (L603–625) is entered when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count` but `ancestors_count > self.max_ancestors_count`. It calls `remove_entry_and_descendants` for each cell-ref parent, which triggers `update_stat_for_remove_tx`. No existing guard prevents the overwrite at L218–219 from discarding those decrements.

## Impact Explanation
`total_tx_size` is the sole gate for two critical pool-admission decisions. `limit_size` (L298) evicts transactions while `total_tx_size > max_tx_pool_size`; an inflated counter causes legitimate fee-paying transactions to be evicted unnecessarily. `updated_stat_for_add_tx` rejects new submissions with `Reject::Full` when the pool appears full; an inflated counter causes valid transactions to be rejected even when the pool has real capacity. Repeated triggering drives `total_tx_size` arbitrarily high, eventually making the pool reject all new submissions while nearly empty. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
The eviction branch is reachable by any unprivileged RPC caller via `send_transaction`. No key material, privileged access, or majority hashpower is required. The attacker submits: (1) Tx-A into the pool; (2) Tx-B referencing Tx-A's output as a cell dep, making Tx-A a `cell_ref_parent`; (3) Tx-C whose `ancestors_count > max_ancestors_count` but `ancestors_count - 1 <= max_ancestors_count`. Each iteration inflates `total_tx_size` by Tx-A's size. The attack is repeatable at low cost with no special network position.

## Recommendation
Compute the new totals **after** `check_and_record_ancestors` completes so that in-place decrements from evictions are already reflected:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals AFTER all evictions have updated self.total_tx_size
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, replace the snapshot pattern entirely with direct in-place addition (`self.total_tx_size += entry.size`) after all evictions and insertions are complete, or use `recompute_total_stat` as a post-insertion consistency check.

## Proof of Concept
**Setup:** `max_ancestors_count = 2`, `max_tx_pool_size = 10_000`
- Tx-A (size=5000) is in the pool.
- Tx-B (size=100) uses Tx-A's output as a cell dep (`cell_ref_parent = {Tx-A}`). `total_tx_size = 5100`.

**Attack:** Submit Tx-C (size=200) with `ancestors_count = 3 > 2`, but `3 - 1 = 2 <= 2`.

**Execution in `add_entry` for Tx-C:**
1. `updated_stat_for_add_tx(200, ...)` → snapshot `total_tx_size = 5100 + 200 = 5300` (local var).
2. `check_and_record_ancestors` evicts Tx-A: `update_stat_for_remove_tx(5000, ...)` → `self.total_tx_size = 5100 - 5000 = 100`.
3. L218: `self.total_tx_size = 5300` — overwrites; Tx-A's 5000 bytes are re-counted.

**Result:** Pool contains Tx-B (100) + Tx-C (200) = 300 bytes real data, but `total_tx_size = 5300`. A unit test can confirm this by asserting `pool_map.total_tx_size == pool_map.recompute_total_stat().unwrap().0` after triggering the eviction path; the assertion will fail on the unfixed code and pass after the fix.
