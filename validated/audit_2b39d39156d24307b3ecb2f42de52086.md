### Title
`next_request_peer` Always Selects the First-Announcing Peer for Unknown Tx Requests, Enabling Silent Tx-Hash Eviction — (`sync/src/types/mod.rs`)

---

### Summary

`UnknownTxHashPriority::next_request_peer()` always selects `peers[0]` — the first peer to announce an unknown transaction hash — for the initial fetch request, with no randomisation. When that peer is the sole announcer and fails to respond, the retry path returns `None`, and `pop_ask_for_txs` silently drops the tx hash from `unknown_tx_hashes` without re-queuing it. The node never requests that transaction again, regardless of how many other peers are connected.

---

### Finding Description

In `sync/src/types/mod.rs`, `UnknownTxHashPriority` tracks which peers have announced a given unknown tx hash:

```rust
pub fn next_request_peer(&mut self) -> Option<PeerIndex> {
    if self.requested {
        if self.peers.len() > 1 {
            self.request_time = Instant::now();
            self.peers.swap_remove(0);
            self.peers.first().cloned()   // rotate only on retry
        } else {
            None                          // single-peer case: returns None
        }
    } else {
        self.requested = true;
        self.peers.first().cloned()       // always picks peers[0] on first call
    }
}
```

`peers` is populated by `push_peer`, which appends peers in announcement-arrival order. The first peer to announce a tx hash is always placed at index 0 and always receives the initial fetch request — there is no randomisation or round-robin.

The caller, `pop_ask_for_txs`, does not re-queue the entry when `next_request_peer` returns `None`:

```rust
while let Some((tx_hash, mut priority)) = unknown_tx_hashes.pop() {
    if priority.should_request(now) {
        if let Some(peer_index) = priority.next_request_peer() {
            result.entry(peer_index)...;
            unknown_tx_hashes.push(tx_hash, priority); // re-queued only on Some
        }
        // None branch: tx_hash is silently dropped
    } else { ... }
}
```

When `peers.len() == 1` and the sole peer does not deliver the transaction within `RETRY_ASK_TX_TIMEOUT_INCREASE`, the retry fires, `next_request_peer` returns `None`, and the tx hash is permanently evicted from `unknown_tx_hashes`. The node will not request that transaction again unless another peer independently re-announces it.

---

### Impact Explanation

An unprivileged relay peer can exploit this to censor specific transactions at a victim node:

1. The attacker connects as a peer and is the first (or only) peer to send a `RelayTransactionHashes` message containing target tx hash `H`.
2. The victim calls `add_ask_for_txs`, inserting the attacker as `peers[0]` for `H`.
3. `pop_ask_for_txs` calls `next_request_peer()`, which returns the attacker's peer index.
4. The attacker ignores the `GetRelayTransactions` request (or disconnects immediately after announcing).
5. After `RETRY_ASK_TX_TIMEOUT_INCREASE`, the retry fires. Because `peers.len() == 1`, `next_request_peer()` returns `None`.
6. The tx hash is silently dropped. The victim node never fetches transaction `H` and will not relay it to its own peers.

The victim's tx-pool and relay graph are effectively blind to `H` until another peer re-announces it — which may never happen if the attacker is the only propagation path for that transaction (e.g., the transaction originator's only connected node).

---

### Likelihood Explanation

The attacker role is an ordinary unprivileged P2P peer — no keys, no stake, no operator access required. The attack is cheap: the attacker only needs to be the first to relay a `RelayTransactionHashes` message for the target tx hash, which is trivially achievable by a well-connected or low-latency peer. The victim has no way to distinguish a legitimate slow peer from a malicious withholder until after `RETRY_ASK_TX_TIMEOUT_INCREASE` has elapsed and the hash has already been dropped.

---

### Recommendation

1. **Randomise initial peer selection**: instead of always picking `peers[0]`, shuffle the `peers` list on construction or select a random index for the first request.
2. **Re-queue on exhaustion**: when `next_request_peer` returns `None` (all peers exhausted), do not silently drop the entry; instead keep it in `unknown_tx_hashes` so that newly connecting peers that later announce the same hash can be tried.
3. **Broadcast to all announcers**: for the initial request, consider sending `GetRelayTransactions` to all peers that announced the hash simultaneously (bounded by a small fan-out limit), analogous to a round-robin or parallel fetch strategy.
