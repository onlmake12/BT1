### Title
DAO Withdrawal Header-Dep Index Truncation Mismatch Between Rust Node and C VM — (`util/dao/src/tests.rs`, `util/dao/src/lib.rs`)

### Summary

The Rust `DaoCalculator` reads the DAO withdrawal witness `input_type` field as a full `u64` to select the header-dep index, while the on-chain C DAO type script resolves the same field using only its lowest byte (treating it as a `u8`). When a transaction carries a witness index whose value exceeds 255 (e.g., 257, lowest byte = 1), the two implementations resolve different header-dep entries. The Rust node rejects the transaction as invalid while the C VM would accept it, producing a consensus split or a permanent DoS on valid DAO withdrawal transactions.

### Finding Description

The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents the discrepancy:

```
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

The test constructs a transaction where:
- `header_deps[1]` = deposit block (block 100)
- `header_deps[257]` = withdraw block (block 200)
- `witness.input_type` = `257u64` (little-endian bytes)

The C VM DAO script reads the witness index as a `u8`, so `257 & 0xFF = 1`, resolving to the deposit block (block 100). The deposit block number matches the cell data (100), so the C VM **accepts** the transaction.

The Rust `DaoCalculator::transaction_fee` reads the same witness field as a full `u64` (257), resolving to the withdraw block (block 200). The withdraw block number (200) does not match the deposit block number stored in cell data (100), so the Rust node **rejects** the transaction with an error.

The test asserts `result.is_err()`, confirming the Rust node rejects what the C VM would accept. [1](#0-0) 

The production fee-check path in `tx-pool/src/util.rs` calls `DaoCalculator::transaction_fee` during tx-pool admission: [2](#0-1) 

The `FeeCalculator` in `verification/src/transaction_verifier.rs` also calls `DaoCalculator::transaction_fee`, which is invoked during contextual transaction verification — a step that runs during both tx-pool admission and block validation: [3](#0-2) 

### Impact Explanation

**Scenario 1 — DoS on valid DAO withdrawals (tx-pool path):** Any user who constructs a DAO withdrawal with a witness `input_type` index whose value exceeds 255 (a valid u64 that the C VM truncates to a valid deposit-block index) will have their transaction permanently rejected by every honest Rust node's tx-pool. The transaction is valid on-chain but unreachable through the normal submission path.

**Scenario 2 — Consensus split (block validation path):** If `DaoCalculator` is invoked during block validation (via `ContextualTransactionVerifier`), a miner who directly assembles a block containing such a transaction would produce a block that the C VM accepts but that Rust nodes reject. This causes a chain split between miners and validating nodes.

Both impacts are reachable by an unprivileged transaction sender with no special keys or access.

### Likelihood Explanation

Constructing a transaction with 258+ header-deps is unusual but entirely valid per the CKB protocol. A motivated attacker or a user with a large number of header-deps can trigger this path. The discrepancy is already documented in the test suite, confirming the root cause exists in production code.

### Recommendation

The Rust `DaoCalculator` should read the witness `input_type` index using the same truncation the C VM applies — mask the u64 value to its lowest byte before using it as a `header_deps` index:

```rust
let index = (raw_index as u8) as usize; // mirror C VM behavior
```

Alternatively, enforce at the transaction-validation layer that the witness index must fit in a `u8`, rejecting transactions with index > 255 with a clear error before they reach the fee calculator.

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` is a direct proof of concept:

1. Build a transaction with 258 `header_deps`: position 1 = deposit block (number 100), position 257 = withdraw block (number 200).
2. Set `witness.input_type = 257u64` (little-endian).
3. Call `DaoCalculator::transaction_fee` on the Rust node → returns `Err` (uses index 257 → withdraw block → number mismatch).
4. The same transaction submitted to the C VM DAO script → accepted (uses index `257 & 0xFF = 1` → deposit block → number matches cell data).

The test already asserts `result.is_err()`, confirming the Rust node rejects what the on-chain script accepts. [4](#0-3)

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)

pub struct FeeCalculator<DL> {
    transaction: Arc<ResolvedTransaction>,
    consensus: Arc<Consensus>,
    data_loader: DL,
}

impl<DL: CellDataProvider + HeaderProvider + ExtensionProvider + EpochProvider> FeeCalculator<DL> {
    fn new(
        transaction: Arc<ResolvedTransaction>,
        consensus: Arc<Consensus>,
        data_loader: DL,
    ) -> Self {
        Self {
            transaction,
            consensus,
            data_loader,
        }
    }

    fn transaction_fee(&self) -> Result<Capacity, DaoError> {
        // skip tx fee calculation for cellbase
        if self.transaction.is_cellbase() {
            Ok(Capacity::zero())
        } else {
            DaoCalculator::new(self.consensus.as_ref(), &self.data_loader)
                .transaction_fee(&self.transaction)
        }
    }
