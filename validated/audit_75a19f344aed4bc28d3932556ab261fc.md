### Title
`check_tx_fee` Uses Size-Only Metric for Admission While All Downstream Accounting Uses `get_transaction_weight(size, cycles)`, Enabling Cycle-Inflated Transactions to Bypass the Minimum Fee Rate Guard - (File: `tx-pool/src/util.rs`)

### Summary

The tx-pool admission gate in `check_tx_fee` enforces the minimum fee rate using **serialized byte size alone** as the weight denominator. Every other subsystem that measures fee rate — pool scoring, eviction, and both fee-estimation algorithms — uses `get_transaction_weight(size, cycles)` = `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. A transaction whose cycle count dominates its size can pass the size-only gate while carrying an actual fee rate orders of magnitude below `min_fee_rate`, polluting the mempool, corrupting fee estimates, and wasting verifier CPU at negligible cost to the attacker.

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee as:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The comment acknowledges the inconsistency but treats it as harmless. The actual weight function used everywhere else is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

with `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

Every downstream consumer of fee rate uses this weight:

- **Pool scoring / eviction** (`TxEntry::fee_rate`, `AncestorsScoreSortKey`, `EvictKey`): [4](#0-3) 
- **`weight_units_flow` fee estimator**: [5](#0-4) 
- **`confirmation_fraction` fee estimator**: [6](#0-5) 
- **Historical fee-rate statistics** (`FeeRateCollector`): [7](#0-6) 

The admission check and all downstream accounting therefore use **two different metrics** for the same `min_fee_rate` threshold — a direct structural analog to the oracle-drift issue in the reference report.

### Impact Explanation

**Concrete numbers** (default config: `min_fee_rate = 1000 shannons/KW`, `max_tx_verify_cycles = 70_000_000`):

| Parameter | Value |
|---|---|
| Tx serialized size | 200 bytes |
| Tx cycles | 70,000,000 |
| Cycle-equivalent weight | `70,000,000 × 0.000_170_571_4 ≈ 11,940` bytes |
| Actual weight | `max(200, 11940) = 11,940` |
| Admission check min fee | `1000 × 200 / 1000 = 200 shannons` |
| Actual fee rate at 200 shannons | `200 × 1000 / 11940 ≈ 16.7 shannons/KW` |
| Ratio below `min_fee_rate` | **~60×** |

An attacker paying 200 shannons per transaction can:

1. **Flood the mempool** with transactions that pass the admission gate but carry an actual fee rate ~60× below the stated minimum. The pool's 180 MB size limit (`max_tx_pool_size`) is the only backstop, and these transactions are cheap enough to fill it. [8](#0-7) 

2. **Exhaust verifier CPU** — each admitted transaction triggers full CKB-VM execution up to 70 M cycles before the fee check is applied, burning node CPU at negligible fee cost.

3. **Corrupt fee estimation** — both `weight_units_flow` and `confirmation_fraction` record admitted transactions using their actual weight-based fee rate. Injecting many transactions with actual fee rate ≈ 17 shannons/KW skews the estimator's historical data downward, causing `estimate_fee_rate` to return artificially low values and causing legitimate users to underpay and have transactions stuck. [9](#0-8) 

4. **Displace legitimate transactions** — when the pool is full, eviction uses the actual weight-based fee rate. Legitimate transactions with moderate fees but small cycle counts may be evicted in favour of the attacker's cycle-heavy transactions that scored higher on the size-only admission check. [10](#0-9) 

### Likelihood Explanation

The attack requires only the ability to submit transactions via the public `send_transaction` RPC or P2P relay — no privileged access, no key material, no majority hashpower. The cost per attack transaction is ~200 shannons (≈ negligible CKB). The attacker needs only a valid UTXO to spend and a script that consumes close to `max_tx_verify_cycles` cycles. This is trivially constructable by any script author. The gap between the size-only admission metric and the weight-based actual metric is structural and always present; it does not depend on any timing or race condition.

### Recommendation

Replace the size-only denominator in `check_tx_fee` with `get_transaction_weight`. Because cycles are not yet known at the point `check_tx_fee` is called (they are determined by script execution), the check should be deferred until after `verify_rtx` returns the actual cycle count, or a conservative upper-bound weight should be used (e.g., `get_transaction_weight(tx_size, max_tx_verify_cycles)`). The latter is a safe over-approximation that closes the gap without restructuring the verification pipeline.

```rust
// After verify_rtx returns actual cycles:
let weight = get_transaction_weight(tx_size, actual_cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

### Proof of Concept

1. Obtain a live cell with ≥ 201 shannons capacity.
2. Deploy a lock script that loops for ~70 M cycles (e.g., a tight RISC-V loop).
3. Construct a transaction spending that cell, outputting `capacity - 200` shannons (fee = 200 shannons), serialized size ≈ 200 bytes.
4. Submit via `send_transaction` RPC.

**Expected result**: The node accepts the transaction because `200 shannons ≥ min_fee_rate.fee(200) = 200 shannons`. The actual fee rate stored in the pool entry is `200 × 1000 / 11940 ≈ 16.7 shannons/KW`, far below `min_fee_rate = 1000 shannons/KW`. Repeating this with many UTXOs fills the mempool with cycle-heavy, effectively zero-fee-rate transactions, degrades fee estimation, and wastes verifier CPU — all for a cost of 200 shannons per transaction.
