Audit Report

## Title
Missing rate limiting for `GetHeaders` in `Synchronizer` allows CPU/IO amplification by any connected peer — (`sync/src/synchronizer/get_headers_process.rs`)

## Summary
`GetHeadersProcess::execute` performs up to 2,101 RocksDB reads per message (101 in `locate_latest_common_block` + 2,000 in `get_locator_response`) with no per-peer frequency cap. Unlike `Relayer`, the `Synchronizer` has no `rate_limiter` field and `getheaders_received` is a confirmed no-op stub. Any connected peer can flood `GetHeaders` at an unbounded rate post-IBD, degrading block sync throughput for all legitimate peers on the targeted node.

## Finding Description
`GetHeadersProcess::execute` (`sync/src/synchronizer/get_headers_process.rs` L36–99) performs two expensive DB operations per message: `locate_latest_common_block` (L69, iterates up to `MAX_LOCATOR_SIZE = 101` entries calling `snapshot.get_block_number` per entry, `sync/src/types/mod.rs` L1872–1877) and `get_locator_response` (L79, reads up to `MAX_HEADERS_LEN = 2,000` block headers via `store().get_block_header`, `sync/src/types/mod.rs` L1914–1920). The call to `self.synchronizer.peers().getheaders_received(self.peer)` at L77 is a confirmed no-op (`sync/src/types/mod.rs` L897–899: body is `// TODO:`). The `Synchronizer` struct (`sync/src/synchronizer/mod.rs` L357–362) has only `chain`, `shared`, and `fetch_channel` fields — no `rate_limiter`. `try_process` (L381–422) dispatches `GetHeadersProcess` with no rate check. By contrast, `Relayer` (`sync/src/relayer/mod.rs` L78–82) holds a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field and `Relayer::try_process` (L116–123) enforces 30 req/s per `(peer, message_item_id)` before any dispatch. The `GetHeadersMissCommonAncestors` ban path (L94–96) is bypassed because `locate_latest_common_block` (`sync/src/types/mod.rs` L1866–1868) requires the last locator entry to be the genesis hash, and if it is, returns `Some(0)` (L1879) rather than `None`, proceeding to `get_locator_response(0, ...)` which reads up to 2,000 headers. The only existing guards — `MAX_LOCATOR_SIZE` check (L47–51) and IBD guard (L53–66) — limit message size and IBD state but impose no frequency constraint.

## Impact Explanation
Sustained flooding saturates the RocksDB I/O path and the `tokio::task::block_in_place` thread pool used for sync message processing (`sync/src/synchronizer/mod.rs` L398), degrading block download throughput for all legitimate peers on the targeted node. Impact: **Low (501–2000 points) — measurable sync performance degradation**, matching the allowed CKB bounty impact "Any other important performance improvements for CKB."

## Likelihood Explanation
Any connected peer (inbound or outbound, no special flag, no PoW, no key) can execute this. The only required knowledge is the genesis hash, which is public. The attacker needs a single TCP connection and a tight send loop. The attack is repeatable and persistent with no self-limiting mechanism.

## Recommendation
Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to the `Synchronizer` struct mirroring the one in `Relayer` (30 req/s keyed by `(PeerIndex, message_item_id)`), and check it at the top of `try_process` before dispatching any message. Additionally, implement `getheaders_received` to track per-peer `GetHeaders` budget and enforce a hard cap.

## Proof of Concept
1. Connect a peer to a non-IBD CKB node (tip age < 24 hours).
2. In a tight loop, send `GetHeaders` with `block_locator_hashes = [<unknown hash not in target chain>, <genesis hash>]` (2 entries, well within `MAX_LOCATOR_SIZE = 101`).
3. Each message passes the locator size check, bypasses the IBD guard, causes `locate_latest_common_block` to return `Some(0)` (genesis found at index 1, `latest_common == Some(0)` short-circuits at L1879), then calls `get_locator_response(0, ...)` (up to 2,000 DB reads), and spawns an async send task — returning `Status::ok()` with no ban.
4. Benchmark legitimate peer sync throughput (blocks/s) with and without the flood to assert measurable degradation.
