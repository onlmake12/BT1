Audit Report

## Title
Unbounded CPU Exhaustion via Repeated `test_tx_pool_accept` with Max-Cycle Script — (`tx-pool/src/process.rs`, `rpc/src/module/pool.rs`)

## Summary
`_test_accept_tx` runs `verify_rtx` with `max_block_cycles()` and `command_rx = None`, making each invocation uninterruptible. The service loop spawns every `TestAcceptTx` message as an independent tokio task with no concurrency cap, and the dedup guard never fires because the transaction is never inserted into the pool. An unprivileged attacker can flood `test_tx_pool_accept` with a max-cycle transaction to saturate all tokio worker threads, starving block template generation and relay processing.

## Finding Description
`_test_accept_tx` (`tx-pool/src/process.rs` L787) hardcodes `max_cycles = self.consensus.max_block_cycles()` and passes `command_rx = None` to `verify_rtx` (L791-799), so CKB-VM runs to completion with no yield point and no way for the runtime to interrupt it on block arrival. By contrast, `_process_tx` (`tx-pool/src/process.rs` L724-731) passes a real `command_rx` so block arrival can interrupt in-progress verification. The service loop (`tx-pool/src/service.rs` L619-621) spawns each `Message::TestAcceptTx` as a separate concurrent tokio task with no concurrency cap. The deduplication check in `test_accept_tx` (`tx-pool/src/process.rs` L390-397) only rejects transactions already in `verify_queue` or `orphan_pool`; since `_test_accept_tx` never inserts the transaction into either structure, the same max-cycle transaction can be resubmitted immediately after each response. No rate limiting or per-IP throttle exists at the RPC layer (`rpc/src/module/pool.rs` L637-660).

## Impact Explanation
Sustained CPU saturation across all tokio worker threads. `get_block_template` requests are spawned as tasks but cannot be scheduled while worker threads are pinned by uninterruptible `verify_rtx` calls. Relay and reorg processing share the same tokio runtime and are similarly starved. Block template generation stalls, causing mining nodes to miss block intervals. This matches the High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
Any node with the Pool RPC module enabled and the RPC endpoint reachable is exploitable — common for mining pool backends and public RPC nodes. The attacker needs only one valid UTXO to construct the transaction. The same transaction can be reused across all calls indefinitely because it is never inserted into the pool. No hashpower, no key material, and no privileged access are required.

## Recommendation
1. Pass a real `command_rx` to `verify_rtx` inside `_test_accept_tx` so block arrival can interrupt verification, consistent with `_process_tx`.
2. Apply a per-IP or global concurrency limit on `test_tx_pool_accept` at the RPC layer.
3. Cap `max_cycles` for `_test_accept_tx` to a fraction of `max_block_cycles()` (e.g., `max_tx_verify_cycles`).

## Proof of Concept
1. Deploy a CKB devnet node with `[rpc] modules = ["Pool"]` and RPC exposed on a reachable interface.
2. Deploy a lock script that loops for `max_block_cycles` cycles.
3. Create a transaction spending a cell locked by that script.
4. Spawn 16 threads each calling `test_tx_pool_accept` with that transaction in a tight loop.
5. Concurrently call `get_block_template` and measure response latency.
6. Assert that `get_block_template` latency exceeds one block interval (8 seconds on mainnet), confirming block template generation stall.
