### Title
Unprivileged Caller Can Drain Subscription Balance via High `tx.gasprice` in `updatePriceFeeds` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.updatePriceFeeds()` is open to any caller and pays the caller a keeper fee calculated as `(gasUsed + GAS_OVERHEAD) * tx.gasprice + singleUpdateKeeperFeeInWei * numPriceIds`. Because `tx.gasprice` is fully attacker-controlled and there is no cap, an attacker can set an arbitrarily high gas price to drain a subscription's entire balance in a single valid update call.

---

### Finding Description

`updatePriceFeeds` has no access control — any address may call it. [1](#0-0) 

After verifying the price update, the contract calls `_processFeesAndPayKeeper`, which computes the keeper fee as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [2](#0-1) 

The only guard is that `totalKeeperFee` must not exceed the remaining subscription balance:

```solidity
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [3](#0-2) 

The full keeper fee is then sent to `msg.sender`:

```solidity
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [4](#0-3) 

There is no maximum gas price check anywhere in the contract. An attacker who sets `tx.gasprice = P_max` where `P_max = (balance - pythFee - singleUpdateKeeperFeeInWei * numPriceIds) / (gasUsed + GAS_OVERHEAD)` will cause `totalKeeperFee` to equal the entire remaining subscription balance, draining it completely. The attacker's net profit is `GAS_OVERHEAD * P_max + singleUpdateKeeperFeeInWei * numPriceIds`, which is a positive fraction of the subscription balance (proportional to `GAS_OVERHEAD / (gasUsed + GAS_OVERHEAD)`).

The analog to the external report is exact: the "summoner" role (here: any keeper/attacker) can redirect user-deposited funds (subscription balance) to themselves, with no enforcement that the funds are used legitimately. The subscription owner deposited ETH expecting it to be consumed at market gas prices for genuine price updates; instead, an attacker can extract it at an inflated gas price.

---

### Impact Explanation

- **Direct theft of user funds**: A subscription owner's entire deposited balance can be extracted by an unprivileged attacker in a single transaction.
- **Permanent loss**: Once drained, the subscription becomes underfunded and deactivated; the owner cannot recover the stolen ETH.
- **Scales with subscription size**: Larger subscriptions (e.g., permanent subscriptions with `MAX_DEPOSIT_LIMIT`) yield proportionally larger attacker profit.

---

### Likelihood Explanation

- `updatePriceFeeds` is intentionally permissionless (any keeper can call it). [1](#0-0) 
- Valid price update data is freely available from the public Hermes API; the attacker only needs to wait for the heartbeat period or find a subscription with deviation-based criteria.
- Setting a high `tx.gasprice` is a standard EVM transaction parameter requiring no special access.
- On chains without EIP-1559 base fee enforcement (or where the base fee is low), the attacker can set `tx.gasprice` to an arbitrarily large value.
- The attack is profitable as long as the subscription balance exceeds the attacker's gas cost, which is guaranteed by the `GAS_OVERHEAD` term.

---

### Recommendation

1. **Cap `tx.gasprice`**: Introduce a maximum gas price parameter (e.g., a governance-settable `maxGasPriceInWei`) and clamp the fee calculation: `effectiveGasPrice = min(tx.gasprice, maxGasPriceInWei)`.
2. **Alternatively, use a fixed keeper fee**: Replace the gas-cost-based reimbursement with a purely fixed per-update fee (`singleUpdateKeeperFeeInWei * numPriceIds`) that is set by governance to approximate real costs, eliminating the `tx.gasprice` dependency entirely.
3. **Whitelist keepers**: Restrict `updatePriceFeeds` to a set of trusted keeper addresses, analogous to the "pull pattern" fix in the external report — only authorized parties can trigger fee-bearing updates.

---

### Proof of Concept

```solidity
// Attacker contract
contract DrainSubscription {
    IScheduler scheduler;
    
    constructor(address _scheduler) { scheduler = IScheduler(_scheduler); }

    // Called with a very high tx.gasprice
    function drain(uint256 subscriptionId, bytes[] calldata updateData) external {
        // tx.gasprice is set by the caller to P_max
        // After this call, msg.sender receives the entire subscription balance as keeper fee
        scheduler.updatePriceFeeds(subscriptionId, updateData);
    }
}
```

**Steps:**
1. Identify a subscription with a large `balanceInWei` via `getActiveSubscriptions()`. [5](#0-4) 
2. Fetch valid price update data from Hermes satisfying the subscription's update criteria.
3. Compute `P_max = (balance - pythFee - singleUpdateKeeperFeeInWei * numPriceIds) / (estimatedGasUsed + GAS_OVERHEAD)`.
4. Submit `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = P_max`.
5. The contract pays `totalKeeperFee ≈ balance` to `msg.sender`, draining the subscription. [6](#0-5)
