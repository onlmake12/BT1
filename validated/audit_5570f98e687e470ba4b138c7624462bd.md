### Title
PostgreSQL-Only `row_lock`/`row_type` Copy-Paste Bug in `script_exists_in_output` Causes Silent Script Deletion on Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

`script_exists_in_output` contains a copy-paste error at line 252: the final `match` reads `row_lock` a second time instead of `row_type`. On PostgreSQL this silently returns the wrong boolean, causing scripts that are referenced **only** as `type_script_id` in surviving outputs to be incorrectly deleted during any block rollback. SQLite is unaffected because the same expression fails type-coercion and falls through to the correct branch.

---

### Finding Description

`script_exists_in_output` is called from `rollback_block` to decide whether a script referenced by a rolled-back output is still referenced by any surviving output, and therefore must not be deleted. [1](#0-0) 

The function issues two queries:

- `row_lock`: `SELECT EXISTS (... WHERE lock_script_id = $1)` — checks lock usage
- `row_type`: `SELECT EXISTS (... WHERE type_script_id = $1)` — checks type usage

After the lock check fails to produce an early-return, the function fetches `row_type` but then evaluates the **wrong** row at line 252:

```rust
// line 237-249: row_type is fetched correctly
let row_type = sqlx::query(...)
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await ...;

// line 252: BUG — reads row_lock again, not row_type
match row_lock.try_get::<bool, _>(0) {   // ← should be row_type
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [2](#0-1) 

On **PostgreSQL**, `EXISTS` returns a native `BOOLEAN`, so `row_lock.try_get::<bool, _>(0)` always succeeds (`Ok`). The `Ok(r)` arm returns the value from `row_lock` — the lock-column check — not from `row_type`. If the script is used only as `type_script_id`, `row_lock` is `false`, so the function returns `false` regardless of what `row_type` contains.

On **SQLite**, `EXISTS` returns `BIGINT`, so `row_lock.try_get::<bool, _>(0)` always fails (`Err`). The `Err(_)` arm correctly reads `row_type.get::<i64, _>(0) == 1`. SQLite is unaffected.

---

### Impact Explanation

During `rollback_block`, after the rolled-back outputs are deleted from the `output` table: [3](#0-2) 

For every `(_, lock_script_id, type_script_id)` tuple from the rolled-back block, `script_exists_in_output` is called for both IDs. For a script `S2` that is **never** a `lock_script_id` in any surviving output but **is** a `type_script_id` in a surviving output:

1. `row_lock` query → `false` (correct; S2 is not a lock script anywhere)
2. No early return
3. `row_type` query → `true` (correct; S2 is still referenced by a surviving output)
4. **Bug**: `match row_lock.try_get::<bool, _>(0)` → `Ok(false)` — returns `false`
5. S2 is pushed into `script_id_list_to_remove` and deleted

The surviving output's `type_script_id` column now holds a dangling foreign-key value pointing to a deleted script row. The `get_cells` query joins `output` to `script` on `type_script_id`: [4](#0-3) 

Because the script row no longer exists, the join returns no rows. The live cell is invisible to all type-script queries. The corruption is permanent until a full re-index.

---

### Likelihood Explanation

- Chain reorgs are a normal, routine event on CKB mainnet. Any reorg that rolls back a block containing an output whose `type_script_id` is not also used as a `lock_script_id` in that same block triggers the bug.
- No attacker action is required beyond participating in normal block relay (an explicitly listed entry point). A natural reorg suffices.
- The condition (script used only as `type_script_id`) is extremely common — every token/UDT cell uses a type script that is distinct from any lock script.
- PostgreSQL is a supported production backend for the rich-indexer.

---

### Recommendation

Change line 252 from `row_lock` to `row_type`:

```rust
// BEFORE (buggy):
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}

// AFTER (correct):
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [2](#0-1)

    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }

        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,

    .fetch_one(tx.as_mut())
