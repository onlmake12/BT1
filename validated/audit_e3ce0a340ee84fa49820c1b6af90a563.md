### Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Same-Hash Multi-Peer Relay — (`sync/src/types/mod.rs`)

### Summary

The guard in `SyncState::add_ask_for_txs` checks only the number of **distinct** hashes in `unknown_tx_hashes`, not the total size of all `peers` Vecs inside each `UnknownTxHashPriority` entry. When N peers announce the same set of hashes, `push_peer` appends each peer's `PeerIndex` to the existing entry's `peers` Vec without any cap, growing total in-memory state to O(N × hashes) while the guard remains permanently silent.

---

### Finding Description

**Entry point:** Any unauthenticated P2P peer sends a `RelayTransactionHashes` message.

**Step 1 — per-message cap (passes):**
`TransactionHashesProcess::execute` rejects messages with more than `MAX_RELAY_TXS_NUM_PER_BATCH` (32767) hashes. [1](#0-0) 

**Step 2 — `add_ask_for_txs` insertion (unbounded):**
For each hash already present in `unknown_tx_hashes` (Occupied branch), `priority.push_peer(peer_index)` is called unconditionally. `push_peer` is a bare `Vec::push` with no length check. [2](#0-1) [3](#0-2) 

**Step 3 — the guard (broken for same-hash scenario):**
The post-insertion check fires only when:
```
unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE (50000)
OR
unknown_tx_hashes.len() >= peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER
``` [4](#0-3) 

When N peers all announce the **same** 32767 hashes, `unknown_tx_hashes.len()` stays fixed at 32767 (no new keys are inserted). The conditions evaluate to:
- `32767 >= 50000` → **FALSE**
- `32767 >= N × 32767` → **TRUE only when N = 1**, **FALSE for N ≥ 2**

The guard is permanently bypassed for every peer after the first. Each subsequent peer silently appends its `PeerIndex` to every entry's `peers` Vec, and `add_ask_for_txs` returns `Status::ok()`.

---

### Impact Explanation

With `MAX_RELAY_PEERS` = 128 peers each announcing 32767 identical hashes:

| Component | Size |
|---|---|
| PeerIndex entries (128 × 32767 × 8 B) | ~33.5 MB |
| Vec struct overhead (32767 × 24 B) | ~786 KB |
| Total | ~34 MB |

34 MB alone is not OOM-inducing. However:

1. **Reconnection amplification:** When a peer disconnects and reconnects it receives a new `PeerIndex`. There is no cleanup of stale `PeerIndex` values from existing `peers` Vecs on disconnect. A single attacker cycling through k connections accumulates k × 32767 `PeerIndex` entries in the queue, growing without bound.
2. **Mutex contention:** The entire insertion loop runs under `unknown_tx_hashes.lock()`. With a large peers Vec, `pop_ask_for_txs` (which iterates all entries) and the per-peer counter scan in the guard (lines 1517–1523) both become O(N × hashes) under the same Mutex, causing lock starvation.

The "OOM crash" framing in the question is overstated for a static 128-peer scenario (~34 MB). The realistic impact is **memory amplification + Mutex starvation** that worsens with reconnections or high peer churn.

---

### Likelihood Explanation

The path requires only unauthenticated P2P connections — no PoW, no keys, no privileged role. Any peer can send `RelayTransactionHashes`. The same-hash strategy is trivially constructable (announce 32767 fixed hashes from each connection). The guard's logical flaw (checking key count instead of total Vec size) is a concrete, locally testable code defect.

---

### Recommendation

1. **Cap `peers` Vec in `push_peer`:** Reject the push if `self.peers.len()` already equals some constant (e.g., `MAX_RELAY_PEERS`). [3](#0-2) 

2. **Fix the guard metric:** Track total `peers` Vec entries across all `UnknownTxHashPriority` values (not just `unknown_tx_hashes.len()`), or maintain a per-peer counter outside the queue so the O(N) scan is not needed. [4](#0-3) 

3. **Clean up stale peer entries on disconnect:** When a peer disconnects, remove its `PeerIndex` from all `peers` Vecs in `unknown_tx_hashes`.

---

### Proof of Concept

```
1. Connect 128 peers to a CKB node (within MAX_RELAY_PEERS).
2. Each peer sends RelayTransactionHashes with the same 32767 fixed hashes.
3. Peer 1: 32767 Vacant entries created, peers Vec length = 1 each.
4. Peers 2–128: each triggers 32767 push_peer calls (Occupied branch).
   Guard check: 32767 >= 128*32767 → FALSE → returns Status::ok() silently.
5. Assert: for any entry in unknown_tx_hashes, entry.peers.len() == 128.
   Total PeerIndex allocations = 128 * 32767 = 4,194,176.
6. Repeat with reconnecting peers to demonstrate unbounded growth beyond 128.
``` [5](#0-4) [6](#0-5) [7](#0-6)
