### Title
Uninitialized `current_tip = 0` in `confirmation_fraction::Algorithm` Causes Out-of-Bounds Panic via Tx Tracked at Height 0 — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

The `ConfirmationFraction` fee-estimator `Algorithm` initializes `current_tip` and `best_height` to `0`. A tx submitted to the pool before the first `commit_block` call (the brief window after IBD ends) is tracked with `entry_height = 0`. When that tx is later dropped from the pool (not confirmed) after the chain has advanced ≥ 1001 blocks, `remove_unconfirmed_tx` computes `tx_age = best_height − 0 ≥ 1001` and performs an unchecked index `confirm_blocks_to_failed_txs[tx_age − 1]` into a 1000-element array, causing an out-of-bounds panic. This is the direct CKB analog of the uninitialized `lastRewardTimestamp = 0` pattern: a zero-initialized counter is used in a subtraction before it is properly set, producing an inflated value that corrupts downstream accounting.

---

### Finding Description

`Algorithm::new()` zero-initializes both `current_tip` and `best_height`:

```rust
// util/fee-estimator/src/estimator/confirmation_fraction.rs  lines 352-360
pub fn new() -> Self {
    Self {
        best_height: 0,
        start_height: 0,
        tx_confirm_stat: Default::default(),
        tracked_txs: Default::default(),
        current_tip: 0,       // ← uninitialized sentinel
        is_ready: false,
    }
}
```

When IBD ends, `update_ibd_state(false)` calls `clear()`, which resets both fields back to `0` and sets `is_ready = true`:

```rust
// lines 455-461
fn clear(&mut self) {
    self.best_height = 0;
    self.start_height = 0;
    self.tx_confirm_stat = Default::default();
    self.tracked_txs.clear();
    self.current_tip = 0;
}
```

`accept_tx` immediately passes `self.current_tip` (still `0`) to `track_tx`:

```rust
// lines 469-473
pub fn accept_tx(&mut self, tx_hash: Byte32, info: TxEntryInfo) {
    let weight = get_transaction_weight(info.size as usize, info.cycles);
    let fee_rate = FeeRate::calculate(info.fee, weight);
    self.track_tx(tx_hash, fee_rate, self.current_tip)  // height = 0
}
```

Inside `track_tx`, the guard `if height != self.best_height { return; }` passes because both are `0`, so the tx is inserted into `tracked_txs` with `height: 0` and into `block_unconfirmed_txs[0 % 1000][bucket_index]`.

Later, when the tx is evicted from the pool (low fee, pool-size pressure, replacement), `reject_tx` → `drop_tx` → `drop_tx_inner(hash, count_failure=true)` → `remove_unconfirmed_tx(entry_height=0, tip_height=best_height, …, count_failure=true)` is called:

```rust
// lines 197-217
fn remove_unconfirmed_tx(
    &mut self,
    entry_height: u64,
    tip_height: u64,
    bucket_index: usize,
    count_failure: bool,
) {
    let tx_age = tip_height.saturating_sub(entry_height) as usize;  // = best_height − 0
    if tx_age < 1 { return; }
    if tx_age >= self.block_unconfirmed_txs.len() {          // len = 1000
        self.bucket_stats[bucket_index].old_unconfirmed_txs -= 1;  // underflow
    } else { … }
    if count_failure {
        // confirm_blocks_to_failed_txs has len = MAX_CONFIRM_BLOCKS = 1000
        self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
        // ^^^ OUT-OF-BOUNDS when tx_age >= 1001 (i.e. best_height >= 1001)
    }
}
```

`confirm_blocks_to_failed_txs` is allocated with exactly `MAX_CONFIRM_BLOCKS = 1000` rows. When `tx_age ≥ 1001`, `tx_age − 1 ≥ 1000` is past the end of the vector. Rust panics on this out-of-bounds index access.

Note that the `WeightUnitsFlow` algorithm explicitly guards against this with `if self.current_tip == 0 { return; }` in its own `accept_tx`, but `ConfirmationFraction` has no such guard.

---

### Impact Explanation

A panic inside the fee-estimator's `reject_tx` path propagates through the tx-pool service task. Depending on the async runtime's panic handling, this can crash the tx-pool worker, rendering the node unable to accept or relay transactions until restarted. Even in release builds (where integer underflow wraps rather than panics), `old_unconfirmed_txs` wraps to `usize::MAX`, permanently corrupting the bucket statistics and causing the fee estimator to produce wildly incorrect fee-rate recommendations for all subsequent callers of `estimate_fee_rate` via RPC.

---

### Likelihood Explanation

After IBD completes, there is a brief but real window between `update_ibd_state(false)` setting `is_ready = true` and the first `commit_block` call updating `current_tip`. Any tx submitted to the pool during this window is tracked at height 0. An attacker monitoring the node's sync progress (e.g., via the `get_tip_block_number` RPC) can time a low-fee-rate tx submission to land in this window. The tx will naturally be evicted once the pool fills or a higher-fee replacement arrives, triggering the panic after ≥ 1001 blocks.

---

### Recommendation

Mirror the guard already present in `WeightUnitsFlow::accept_tx`:

```rust
pub fn accept_tx(&mut self, tx_hash: Byte32, info: TxEntryInfo) {
    if self.current_tip == 0 {
        return;  // not yet initialized; drop silently
    }
    …
}
```

Additionally, add a bounds check in `remove_unconfirmed_tx` before indexing `confirm_blocks_to_failed_txs`:

```rust
if count_failure && tx_age >= 1 && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

---

### Proof of Concept

1. Run a CKB node configured with the `ConfirmationFraction` fee estimator until it is near the end of IBD.
2. Poll `get_tip_block_number` until the node exits IBD (`update_ibd_state(false)` fires, `is_ready = true`, `current_tip` reset to `0`).
3. Immediately submit a tx with a very low fee rate via `send_transaction` RPC. The tx is accepted into the pool and tracked at `entry_height = 0` because `current_tip == best_height == 0`.
4. Allow the node to sync 1001+ additional blocks. The pool fills with higher-fee txs, evicting the attacker's low-fee tx via `reject_tx`.
5. `remove_unconfirmed_tx(0, best_height ≥ 1001, bucket_index, true)` is called; `tx_age = best_height ≥ 1001`; `confirm_blocks_to_failed_txs[tx_age − 1]` is accessed out of bounds → **panic / node crash**.

    fn track_tx(&mut self, tx_hash: Byte32, fee_rate: FeeRate, height: u64) {
        if self.tracked_txs.contains_key(&tx_hash) {
            // already in track
            return;
        }
        if height != self.best_height {
            // ignore wrong height txs
            return;
        }
        if let Some(bucket_index) = self.tx_confirm_stat.add_unconfirmed_tx(height, fee_rate) {
            self.tracked_txs.insert(
                tx_hash,
                TxRecord {
                    height,
                    bucket_index,
                    fee_rate,
                },
            );
        }
    }
