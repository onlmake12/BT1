### Title
PostgreSQL-Only `row_lock`/`row_type` Copy-Paste Bug in `script_exists_in_output` Permanently Deletes Live Type-Script Rows on Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

`script_exists_in_output` contains a copy-paste error at line 252: the final `match` reads `row_lock` a second time instead of `row_type`. On PostgreSQL, `row_lock.try_get::<bool, _>(0)` always succeeds (PostgreSQL returns `BOOLEAN` for `EXISTS`), so the `row_type` query result is silently discarded. Any script that is referenced **only** as a `type_script_id` in surviving outputs is therefore always evaluated as non-existent and unconditionally deleted during every rollback. On SQLite the `Err(_)` branch is taken and `row_type` is read correctly, so SQLite is unaffected.

### Finding Description

**The bug — lines 237–256:**

```rust
// row_type is fetched correctly …
let row_type = sqlx::query("SELECT EXISTS (SELECT 1 FROM output WHERE type_script_id = $1)")
    .bind(script_id)
    .fetch_one(tx.as_mut()).await …;

// … but the final match reads row_lock again, not row_type
match row_lock.try_get::<bool, _>(0) {   // ← line 252: should be row_type
    Ok(r) => Ok(r),                       // PostgreSQL always takes this arm
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),  // SQLite takes this arm (correct)
}
``` [1](#0-0) 

On PostgreSQL, by the time execution reaches line 252, `row_lock` is already known to be `false` (the early-return at lines 223–227 would have fired otherwise). So `Ok(r)` always returns `Ok(false)`, making the function lie: it says the script does not exist in any output even when it does exist as a `type_script_id`. [2](#0-1) 

**The caller — `rollback_block` lines 28–39:**

```rust

Because `script_exists_in_output` always returns `false` for pure type-scripts on PostgreSQL, every such script from the rolled-back block is added to `script_id_list_to_remove` and hard-deleted, even when surviving outputs still reference it.

**Trigger path (no privileges required):**

The sync loop in `IndexerSyncService::try_loop_sync` calls `indexer.rollback()` whenever the next block's `parent_hash` does not match the current tip — i.e., on any natural 1-block reorg, which is a routine network event requiring no attacker. [4](#0-3) 

`RichIndexer::rollback` delegates directly to `rollback_block`. [5](#0-4) 

### Impact Explanation

After the script row for S2 is deleted, the `output` row for the surviving cell still holds `type_script_id = <deleted id>`. Every `get_cells` query that joins `output` to `script` on `type_script_id` will find no matching script row and return zero results for that live cell. The cell is permanently invisible to type-script queries until a full re-index. Applications that track UDT balances, NFT ownership, or any type-script-gated asset via the rich-indexer will silently report incorrect state. [6](#0-5) 

### Likelihood Explanation

- Affects **PostgreSQL deployments only** (SQLite takes the correct `Err` branch).
- Triggered by **any 1-block reorg** where the rolled-back block and a surviving block share a script used only as a type script. Short reorgs are a normal occurrence on any live CKB network.
- No attacker action is required; the bug fires on ordinary chain reorganisation.
- All existing rollback tests run against SQLite (`connect_sqlite(MEMORY_DB)`), so the bug is invisible to the current test suite. [7](#0-6) 

### Recommendation

Change line 252 from `row_lock` to `row_type`:

```rust
// Before (buggy):
match row_lock.try_get::<bool, _>(0) {

// After (correct):
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [8](#0-7) 

Add a PostgreSQL integration test that appends two blocks both containing an output with the same type-only script, rolls back the second block, and asserts the script row still exists.

### Proof of Concept

1. Start a PostgreSQL-backed rich-indexer.
2. Append **block0** containing a transaction with one output: `lock = L, type = S2`.
3. Append **block1** containing a transaction with one output: `lock = L2, type = S2` (S2 reused as type only).
4. Call `rollback()` to remove block1.
5. Query `SELECT id FROM script WHERE …` for S2.
6. **Expected**: S2 still exists (it is referenced by block0's surviving output).
7. **Actual on PostgreSQL**: S2 is deleted. Subsequent `get_cells` filtered by S2 as type script returns an empty list even though the cell from block0 is live. [9](#0-8)

    )

    .map_err(|err| Error::DB(err.to_string()))?;
