Audit Report

## Title
Unbounded `runtime::spawn` NAT Traversal Tasks via Spoofed `from` Field in `ConnectionRequest`/`ConnectionSync` — (`network/src/protocols/hole_punching/component/connection_sync.rs`, `connection_request.rs`)

## Summary
`ConnectionRequestProcess::execute` stores attacker-controlled addresses into `pending_delivered` keyed by an unauthenticated `content.from` peer ID. A single connected peer can poison this map with up to 30 distinct entries per second by cycling through different spoofed `from` values. Each subsequent `ConnectionSync` message with a matching `from` unconditionally calls `runtime::spawn` with up to 24 concurrent `try_nat_traversal` futures, each retrying TCP connections for 30 seconds with no global concurrency cap, exhausting file descriptors and ephemeral ports until the node can no longer accept or initiate connections.

## Finding Description

**Root cause — unauthenticated `from` field:**

`ConnectionRequestProcess::execute` checks `self_peer_id == &content.to` to decide whether to call `respond_delivered`, but never verifies that `content.from` matches the actual session's peer ID (`self.peer`). [1](#0-0) 

Inside `respond_delivered`, the only per-`from` guard is the `HOLE_PUNCHING_INTERVAL` (2 minutes) check, which is trivially bypassed by using a different `from` value for each message: [2](#0-1) 

The attacker-supplied addresses are then stored unconditionally: [3](#0-2) 

**Rate limiter bypass:**

The outer `rate_limiter` allows 30 messages/sec per `(session_id, msg.item_id())`: [4](#0-3) 

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`: [5](#0-4) 

Since `content.from` is attacker-controlled and different for each message, each spoofed `from` gets its own independent 1/sec bucket. The outer 30/sec cap is the only real constraint, allowing 30 distinct poisoned entries per second.

**`pending_delivered` has no size cap:**

`pending_delivered` is a plain `HashMap` with no maximum size. The cleanup in `notify` only runs every 5 minutes and only removes entries older than 5 minutes (`TIMEOUT`): [6](#0-5) 

**Unbounded `runtime::spawn` in `ConnectionSyncProcess::execute`:**

When `content.to == local_peer_id`, the code looks up `pending_delivered[content.from]` and, if found, unconditionally calls `runtime::spawn`: [7](#0-6) [8](#0-7) 

The same `forward_rate_limiter` bypass applies here — different `from` values each get their own bucket. The `pending_delivered` entry is **not removed** after being consumed, so the same entry can trigger repeated spawns.

**Resource exhaustion inside `try_nat_traversal`:**

Each spawned task runs up to 24 concurrent `try_nat_traversal` futures. Each future loops for 30 seconds, creating a new `TcpSocket` and attempting `connect()` every ~200ms with no semaphore or global concurrency limit: [9](#0-8) 

## Impact Explanation

At 30 poisoned entries/second and 24 addresses/entry, after 30 seconds the attacker maintains 900 active spawned tasks each holding 24 concurrent socket-creating loops — up to 21,600 concurrent TCP connection attempts. This exhausts the OS file descriptor table (default Linux limit: 1024–65536) and TCP ephemeral port space, causing the node to fail to accept or initiate any legitimate P2P connections. This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires only a single standard P2P connection to the victim. No privileged role, no proof-of-work, no key material, and no Sybil capability are needed. The `ConnectionRequest` and `ConnectionSync` messages are standard production protocol messages. The spoofed `from` field is never authenticated against the session. The attack is fully repeatable: after each 30-second `try_nat_traversal` cycle expires, the attacker re-sends `ConnectionSync` messages to re-trigger spawns (the `pending_delivered` entries persist for 5 minutes).

## Recommendation

1. **Authenticate `from`**: In `ConnectionRequestProcess::execute`, verify that `content.from` matches the peer ID of the actual session. Reject messages where `from` does not match the sender's authenticated peer ID.
2. **Cap `pending_delivered`**: Enforce a maximum size on the `pending_delivered` HashMap (e.g., 64 entries), evicting oldest entries when the cap is reached.
3. **Remove entry after consumption**: In `ConnectionSyncProcess::execute`, remove the `pending_delivered` entry after it is consumed to prevent re-triggering spawns from the same poisoned entry.
4. **Global concurrency limit**: Use a `tokio::sync::Semaphore` to bound the total number of concurrent `try_nat_traversal` tasks (e.g., max 8–16 globally).

## Proof of Concept

```
1. Attacker establishes one TCP session to the victim node.

2. For i in 0..30 (one batch per second, limited by rate_limiter):
     Send ConnectionRequest {
       from: random_peer_id_i,   // spoofed, different for each message
       to: victim_peer_id,
       listen_addrs: [24 × "1.2.3.4:PORT_i_j"],  // valid TCP/IP4 addresses
       max_hops: 6,
       route: [],
     }
   → victim stores pending_delivered[random_peer_id_i] = ([24 addrs], now)
   → forward_rate_limiter allows each because (from_i, to, item_id) is unique

3. For i in 0..30:
     Send ConnectionSync {
       from: random_peer_id_i,
       to: victim_peer_id,
       route: [],
     }
   → victim finds pending_delivered[random_peer_id_i], calls runtime::spawn
     with select_ok([24 × try_nat_traversal(...)])

4. Each try_nat_traversal runs for 30s, creating a TcpSocket every ~200ms.
   30 spawns × 24 futures × ~150 socket attempts = ~108,000 TCP connect()
   calls over 30 seconds.

5. Repeat step 3 every 30s (entries persist for 5 minutes) to maintain pressure.

6. Observable: victim's /proc/self/fd count approaches ulimit -n;
   legitimate connections are refused with EMFILE or ECONNREFUSED.
```
