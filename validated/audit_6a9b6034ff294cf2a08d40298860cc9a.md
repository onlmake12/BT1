### Title
Missing `service_started` Guard in RPC Tx-Pool Handlers Allows Premature Calls Before Service Initialization — (`tx-pool/src/service.rs`, `rpc/src/module/pool.rs`, `rpc/src/module/miner.rs`)

---

### Summary

`TxPoolController` exposes a `service_started()` readiness flag backed by an `AtomicBool` that is set to `true` only after all async processing loops are spawned inside `TxPoolServiceBuilder::start()`. The chain-verification path correctly gates its calls behind this flag. However, the RPC handlers for `send_transaction`, `test_tx_pool_accept`, `remove_transaction`, `tx_pool_info`, `clear_tx_pool`, `clear_tx_verify_queue`, `get_raw_tx_pool`, and `get_block_template` send messages to the tx-pool channel **without checking `service_started()` first**. An RPC caller who reaches these endpoints before the service is ready will cause the calling thread to block indefinitely inside `block_in_place(|| response.recv())`, hanging the RPC worker.

---

### Finding Description

`TxPoolController` carries a shared `started: Arc<AtomicBool>` initialized to `false`: [1](#0-0) 

It is set to `true` synchronously at the very end of `TxPoolServiceBuilder::start()`, after all `handle.spawn(...)` calls: [2](#0-1) 

The readiness predicate is exposed as: [3](#0-2) 

The chain-verification code correctly guards every tx-pool call behind this flag: [4](#0-3) 

However, the `send_message!` macro used by every RPC-facing controller method performs no such check: [5](#0-4) 

The macro calls `try_send` (which succeeds as long as the 512-slot channel has capacity) and then blocks the calling thread on `block_in_place(|| response.recv())`. If the async processing loop has not yet started, no one will ever dequeue the message and send a response, so the thread hangs indefinitely.

The affected RPC handlers — none of which check `service_started()` — are:

- `send_transaction` / `submit_local_tx`: [6](#0-5) 

- `test_tx_pool_accept` / `test_accept_tx`: [7](#0-6) 

- `tx_pool_info`, `clear_tx_pool`, `clear_tx_verify_queue`, `get_raw_tx_pool`: [8](#0-7) 

- `get_block_template` (miner RPC): [9](#0-8) 

The only RPC that correctly exposes the readiness state is `tx_pool_ready`, but it is advisory — callers are not forced to consult it, and the other handlers do not enforce it: [10](#0-9) 

---

### Impact Explanation

An RPC caller who sends `send_transaction`, `get_block_template`, or any of the other unguarded pool RPCs during the node startup window (before `TxPoolServiceBuilder::start()` completes and the async loop begins draining the channel) will cause the RPC worker thread to block indefinitely on `response.recv()`. Because the tokio `block_in_place` call occupies a thread-pool thread, repeated such calls can exhaust the RPC thread pool, making the node's entire JSON-RPC interface unresponsive — a local or remote denial-of-service against the node operator's RPC surface.

Additionally, because `started` is set to `true` synchronously before the spawned async tasks are actually scheduled by the tokio executor, there is a race window where `service_started()` returns `true` but the processing loop has not yet polled the channel. Any message sent in this window will queue silently and may be processed out of order relative to the persisted-data load that follows: [2](#0-1) 

---

### Likelihood Explanation

The RPC server is started after `TxPoolServiceBuilder::start()` in the normal production path, so the window is narrow. However:

1. The startup ordering is not enforced by the type system or any runtime assertion — it is a convention.
2. Any RPC client that connects immediately after the TCP listener opens (before the async tasks are scheduled) can trigger the hang.
3. The `tx_pool_ready` RPC exists precisely because this race is known, yet the other handlers do not enforce the guard, leaving the protection entirely to the caller.

---

### Recommendation

Add a `service_started()` guard at the top of every `TxPoolController` method that sends a message to the pool, mirroring the pattern already used in `chain/src/verify.rs`:

```rust
pub fn submit_local_tx(&self, tx: TransactionView) -> Result<SubmitTxResult, AnyError> {
    if !self.service_started() {
        return Err(ckb_error::OtherError::new("tx-pool service not started").into());
    }
    send_message!(self, SubmitLocalTx, tx)
}
```

Apply the same guard to `get_block_template`, `test_accept_tx`, `remove_local_tx`, `get_tx_pool_info`, `clear_pool`, `clear_verify_queue`, `get_all_entry_info`, `get_all_ids`, and `submit_remote_tx`. Alternatively, centralise the check inside the `send_message!` and `send_notify!` macros so it is impossible to forget.

---

### Proof of Concept

1. Start a CKB node and connect an RPC client immediately after the TCP listener opens.
2. Call `send_transaction` with any valid transaction before `tx_pool_ready` returns `true`.
3. The `try_send` into the 512-slot channel succeeds; `block_in_place(|| response.recv())` blocks the RPC thread indefinitely because the async processing loop has not yet started draining the channel.
4. Repeat with additional RPC calls to exhaust the RPC thread pool, rendering the node's JSON-RPC interface unresponsive. [5](#0-4) [11](#0-10)
