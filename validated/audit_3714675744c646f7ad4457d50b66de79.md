Audit Report

## Title
Missing Per-Peer Rate Limiting in `LightClientProtocol` Enables DB I/O Exhaustion via `GetBlocksProof` Flood — (`util/light-client-protocol-server/src/lib.rs`)

## Summary

`LightClientProtocol` carries no rate-limiter state and performs no throttle check before dispatching inbound messages. A single unprivileged peer can send `GetBlocksProof` messages at maximum network speed, each triggering up to ~3000 synchronous RocksDB reads and an MMR `gen_proof` computation, saturating the node's DB I/O and degrading block validation and relay for all other peers.

## Finding Description

`LightClientProtocol` is defined with only a `shared: Shared` field — no rate-limiter state exists: [1](#0-0) 

Its `received` handler parses the message and immediately calls `try_process` with no throttle check: [2](#0-1) 

`try_process` dispatches directly to all four handler processes — `GetLastState`, `GetLastStateProof`, `GetBlocksProof`, `GetTransactionsProof` — with no rate check at any point: [3](#0-2) 

`GetBlocksProofProcess::execute` performs, per message, up to 1000 `is_main_chain` DB lookups in the partition step, then for each found block: `get_block_header`, `get_block_uncles`, and `get_block_extension` — totaling up to ~3000 synchronous RocksDB point-reads: [4](#0-3) 

This is followed by an MMR `gen_proof` computation in `reply_proof`: [5](#0-4) 

The only guard is a per-message size cap of 1000 hashes (`GET_BLOCKS_PROOF_LIMIT`), which bounds work per message but places no bound on message rate: [6](#0-5) [7](#0-6) 

A grep across the entire `util/light-client-protocol-server/` directory confirms zero occurrences of any rate-limiting construct. By contrast, `Relayer` carries an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` field: [8](#0-7) 

And checks it at the top of `try_process` before dispatching to any handler: [9](#0-8) 

The same gap applies to `GetTransactionsProof`, which performs up to 1000 `get_transaction_info` + `is_main_chain` lookups, full block reads, and CBMT proof generation per message: [10](#0-9) 

## Impact Explanation

Each `GetBlocksProof` message with 1000 hashes causes up to 1000 `is_main_chain` DB lookups plus up to 3 additional reads per found block (`get_block_header`, `get_block_uncles`, `get_block_extension`) plus an MMR `gen_proof` computation. A single attacker peer sending these messages in a tight loop can saturate RocksDB read bandwidth, stalling block validation and relay for all peers on the node. This matches the allowed **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* A sustained flood from even one peer degrades the node's ability to process and relay blocks, affecting the broader network.

## Likelihood Explanation

The attack requires only a persistent TCP connection to the node's P2P port — no credentials, no PoW, no stake. Any node with the light-client protocol enabled is exposed. The attacker controls message rate and can sustain the flood indefinitely. The 1000-hash cap per message is the attacker's ally: it maximizes work per message while staying within the allowed limit.

## Recommendation

Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`, mirroring the `Relayer` pattern: [11](#0-10) 

Check it at the top of `try_process` before dispatching to any component, keyed by `(peer_index, message.item_id())`. A quota of 1–5 requests/second per peer per message type is sufficient for legitimate light-client usage while eliminating the flood vector.

## Proof of Concept

```
1. Connect to a CKB full node with the LightClient protocol enabled.
2. Obtain any 1000 valid main-chain block hashes (h1..h1000) and the current tip hash.
3. In a tight loop, send:
     GetBlocksProof { block_hashes: [h1..h1000], last_hash: tip }
4. Observe: each message triggers up to ~3000 RocksDB reads + MMR gen_proof.
   With no rate limit, the node's RocksDB read throughput is fully consumed
   by the single attacker peer, degrading block relay and validation for all
   other peers.
5. Repeat with GetTransactionsProof { tx_hashes: [t1..t1000], last_hash: tip }
   for the same effect via the transaction proof path.
```
