### Title
Missing Signer-Publisher Authorization in `set_publisher_stake_account` Allows Any User to Modify Any Publisher's Stake Account - (File: governance/pyth_staking_sdk/src/idl/integrity-pool.json)

### Summary
The `set_publisher_stake_account` instruction in the Pyth integrity pool program separates the `signer` (required transaction signer) from the `publisher` (the account whose stake is being modified). The `publisher` account is **not** a signer and is only annotated as "checked against data in pool_data" — meaning only its existence in the pool is verified, not that the `signer` is authorized to act on behalf of that publisher. Any unprivileged user can invoke this instruction for any registered publisher.

### Finding Description
The IDL for `set_publisher_stake_account` exposes the following account layout:

```json
{
  "name": "signer",
  "signer": true          // any arbitrary account
},
{
  "docs": ["CHECK : The publisher will be checked against data in the pool_data"],
  "name": "publisher"     // NOT a signer — only existence-checked
},
{
  "name": "pool_data",
  "writable": true
},
{
  "name": "new_stake_account_positions_option",
  "optional": true
},
{
  "name": "current_stake_account_positions_option",
  "optional": true
}
```

The `signer` is a generic required signer — it is **not** constrained to equal `publisher`, nor is it constrained to be a governance authority. The `publisher` account is passed as an unchecked account whose only documented validation is "checked against data in pool_data" (i.e., the publisher exists in the pool). No check enforces `signer == publisher` or `signer == pool_config.authority`.

This is confirmed by the SDK client, which passes `this.wallet` (any connected wallet) as the implicit signer while accepting an arbitrary `publisher` public key:

```typescript
// governance/pyth_staking_sdk/src/pyth-staking-client.ts
async setPublisherStakeAccount(
  publisher: PublicKey,
  stakeAccountPositions: PublicKey,
  newStakeAccountPositions: PublicKey | undefined,
) {
  const instruction = await this.integrityPoolProgram.methods
    .setPublisherStakeAccount()
    .accounts({
      currentStakeAccountPositionsOption: stakeAccountPositions,
      newStakeAccountPositionsOption: newStakeAccountPositions ?? null,
      publisher,   // ← arbitrary publisher, not verified against wallet
    })
    .instruction();
  await sendTransaction([instruction], this.connection, this.wallet);
```

The xc_admin CLI also confirms the signer and publisher are decoupled — it passes the multisig vault PDA as `signer` and an arbitrary `publisher` pubkey:

```typescript
// governance/xc_admin/packages/xc_admin_cli/src/index.ts
.accounts({
  publisher: new PublicKey(options.publisher),
  signer: await vault.getVaultAuthorityPDA(targetCluster),  // ← signer ≠ publisher
})
```

### Impact Explanation
An attacker can call `set_publisher_stake_account` for **any** registered publisher:

1. **Remove a publisher's stake account** (pass `newStakeAccountPositionsOption = None`): The publisher's staking position is unlinked from the pool. They stop accruing integrity staking rewards and their delegators' reward accounting is disrupted.
2. **Substitute a different stake account**: The attacker could point the publisher's entry to a stake account they control, potentially redirecting future reward distributions.

Both outcomes cause direct financial harm to publishers and their delegators without requiring any privileged access.

### Likelihood Explanation
The entry point is fully permissionless — any Solana wallet can submit this transaction. The only inputs required (publisher address, current stake account pubkey) are public on-chain data. No secret knowledge, governance key, or special role is needed.

### Recommendation
Add an explicit authorization check at the start of the `set_publisher_stake_account` handler. The signer must either be the publisher themselves or a recognized governance authority:

```rust
// Option A: publisher must sign for themselves
require!(ctx.accounts.signer.key() == ctx.accounts.publisher.key(), ErrorCode::Unauthorized);

// Option B: signer must be the pool_config authority (governance)
require!(ctx.accounts.signer.key() == ctx.accounts.pool_config.authority, ErrorCode::Unauthorized);
```

### Proof of Concept
1. Attacker observes on-chain that `publisher P` has `stake_account_positions = S`.
2. Attacker constructs a `set_publisher_stake_account` transaction:
   - `signer` = attacker's wallet (signs the tx)
   - `publisher` = P
   - `currentStakeAccountPositionsOption` = S
   - `newStakeAccountPositionsOption` = None (or attacker-controlled account)
3. Transaction succeeds because the program only verifies P exists in `pool_data`, not that the signer is P or governance.
4. Publisher P's stake account is removed or replaced; P loses staking rewards and delegators are affected. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
