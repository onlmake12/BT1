Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Attacker-Controlled `(from, to)` Keys — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter` uses `governor::state::keyed::HashMapStateStore<(PeerId, PeerId, u32)>`, which accumulates entries indefinitely until `retain_recent()` is called. That call exists only in `disconnected()`, never in the periodic `notify()` handler. An attacker holding a single live P2P session can insert up to 30 unique `(from, to)` keys per second by sending messages with arbitrary wire-supplied PeerIds, causing unbounded heap growth and eventual OOM crash of the victim node.

## Finding Description
`forward_rate_limiter` is declared as `RateLimiter<(PeerId, PeerId, u32)>` backed by `HashMapStateStore`: [1](#0-0) 

`retain_recent()` is called only inside `disconnected()`, which fires only when the session ends: [2](#0-1) 

The `notify()` handler (fired every 5 minutes via `CHECK_INTERVAL`) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter: [3](#0-2) 

In `ConnectionSyncProcess::execute()`, `from` and `to` are parsed directly from the wire with no check that they correspond to known or connected peers — only byte-validity is enforced: [4](#0-3) 

These attacker-controlled values are then used as the key inserted into `forward_rate_limiter`: [5](#0-4) 

The same pattern is present in `ConnectionRequestProcess::execute()`: [6](#0-5) 

And in `ConnectionRequestDeliveredProcess::execute()`: [7](#0-6) 

The outer `rate_limiter` is keyed by `(session_id, msg.item_id())` and limits to 30 req/s per session per message type: [8](#0-7) 

This throttles throughput but does not bound the number of distinct `(from, to)` keys inserted into `forward_rate_limiter` — each of the 30 allowed messages per second can carry a fresh unique `(from, to)` pair, inserting a new `HashMapStateStore` entry that is never evicted for the lifetime of the session.

## Impact Explanation
At 30 messages/second over a 24-hour session, an attacker inserts ~2.6 million unique `(PeerId, PeerId, u32)` keys. Each entry holds two `PeerId` values (~39 bytes each), governor atomic state, and `HashMap` overhead — approximately 150–250 bytes per entry — yielding ~400–650 MB of heap growth from a single attacker session. Multiple concurrent sessions multiply the effect linearly. Memory exhaustion crashes the node process.

This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attacker requires only a standard P2P connection — no special privileges, no PoW, no cryptographic key material. Maintaining a long-lived TCP session and sending 30 crafted messages per second is trivially achievable with any standard networking library. The `from`/`to` PeerIds need only pass `PeerId::from_bytes()` byte-validity; no signature, proof of ownership, or peer-registry membership is checked before the key is inserted into the rate-limiter store. The attack is repeatable and can be parallelized across multiple sessions.

## Recommendation
1. Call `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside the existing `notify()` handler alongside the existing `pending_delivered`/`inflight_requests` cleanup — this is the minimal fix.
2. Validate that `content.from` equals the sending session's peer ID before calling `check_key()`, eliminating the ability to inject arbitrary keys entirely.
3. Consider switching to a fixed-capacity LRU-based rate-limiter store to provide a hard memory ceiling regardless of eviction timing.

## Proof of Concept
```
1. Connect to a victim CKB node as a normal P2P peer.
2. In a loop at 30 msg/s, send ConnectionSync messages where each message has:
     from = random_peer_id_i   (unique per message, valid byte-length encoding)
     to   = random_peer_id_j   (unique per message)
     route = []                (empty, passes the MAX_HOPS check)
3. Maintain the session for several hours without disconnecting.
4. Observe victim node RSS growing proportionally to messages sent (~150-250 bytes per message).
5. Node OOMs and crashes; no disconnect occurs until the attacker chooses.

Unit test plan:
- Instantiate HolePunching with a mock NetworkState.
- Simulate N ConnectionSync messages with distinct (from, to) pairs via received().
- Assert that forward_rate_limiter internal map size equals N (no eviction occurred).
- Call notify() and assert map size is still N (confirming the missing retain_recent()).
- Call disconnected() and assert map size drops to 0 (confirming retain_recent() works when called).
```
