Audit Report

## Title
Unbounded Dead-Cell `NOT IN` Clause Exceeds SQLite Bind-Parameter Limit in `get_cells` — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

## Summary
When `RichIndexer` is enabled with `index_tx_pool = true` and the SQLite backend is used, `get_cells` and `get_cells_capacity` dynamically construct a `NOT IN ($1, $2, …, $N)` clause with one bind parameter per dead cell in the pool. SQLite's default `SQLITE_MAX_VARIABLE_NUMBER` is 999. If the pool contains ≥ 1000 distinct dead cells, every call to these RPC endpoints returns a `DB`-level error until the pool drains.

## Finding Description
In `get_cells`, all dead cells are collected from the pool without any count bound:

```rust
dead_cells = pool.dead_cells()
    .map(|out_point| { … })
    .collect::<Vec<(_, u32)>>()
```

One `$N` placeholder is emitted per entry and one `.bind(tx_hash)` call is made per entry. When `dead_cells.len() > 999`, SQLite rejects the query with `SQLITE_RANGE` (too many bind variables). The error propagates as `Error::DB(…)` to the RPC caller. The identical pattern exists in `get_cells_capacity`. The codebase already recognises this limit in `insert.rs` (`BATCH_SIZE_THRESHOLD = 1_000` with an explicit comment), but the same guard was never applied to the query path.

## Impact Explanation
The `get_cells` and `get_cells_capacity` RPC endpoints become non-functional (returning DB errors to all callers) for the duration the pool holds ≥ 1000 dead cells. The node itself does not crash. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation
Requires: (1) operator has explicitly enabled the `RichIndexer` module and `index_tx_pool = true` (non-default but a documented production configuration); (2) an attacker controls ≥ 1000 live cells on-chain and submits transactions spending them. The 180 MB pool size limit imposes no transaction-count ceiling that would prevent accumulating 1000 distinct inputs. On testnet the cost is negligible; on mainnet it requires real CKB but is not prohibitive. Once triggered, the condition persists until the pool drains (transactions confirmed or rejected).

## Recommendation
Before constructing the `NOT IN` clause, chunk `dead_cells` into batches of at most 999 (matching `BATCH_SIZE_THRESHOLD`) and either issue multiple queries (intersecting results) or use a temporary table / CTE approach to avoid the bind-parameter limit. Apply the same fix to `get_cells_capacity`. Alternatively, skip the SQLite bind-parameter path entirely by inserting dead-cell hashes into a temporary table and joining against it.

## Proof of Concept
Populate the indexer `Pool` with 1000 synthetic `OutPoint` entries via `pool.new_transaction(...)`, then call `get_cells` against a SQLite-backed `AsyncRichIndexerHandle`. Assert the call returns `Err(Error::DB(…))` rather than a successful result. The existing `BATCH_SIZE_THRESHOLD` constant and its comment in `insert.rs` confirm the developers are aware of the SQLite bind limit; the omission in the query path is the root cause.
