### Title
Hardcoded Well-Known Script Lists in `WellKnownScriptsOnly` Validator Limits Interoperability for Non-Mainnet/Testnet Chains — (File: `rpc/src/module/pool.rs`)

---

### Summary

The `build_well_known_lock_scripts` and `build_well_known_type_scripts` functions in `rpc/src/module/pool.rs` hardcode specific script code hashes for only `mainnet` and `testnet`. Any other officially supported chain (staging, preview, dev, or custom) receives an empty list, causing the `WellKnownScriptsOnly` output validator to silently reject all transactions that use anyone_can_pay or SUDT type scripts on those chains — even though those scripts are perfectly valid. New chains cannot be added to the well-known list without modifying source code.

---

### Finding Description

`build_well_known_lock_scripts` and `build_well_known_type_scripts` use a `match` on the chain spec name string, returning hardcoded JSON script arrays for `mainnet` and `testnet`, and returning `"[]"` (empty) for every other chain: [1](#0-0) [2](#0-1) 

These lists are consumed at `PoolRpcImpl::new` and stored as `well_known_lock_scripts` / `well_known_type_scripts`: [3](#0-2) 

They are then used by `WellKnownScriptsOnlyValidator` when an RPC caller invokes `send_transaction` or `test_tx_pool_accept` with `outputs_validator = "well_known_scripts_only"`: [4](#0-3) 

The validator's `validate_lock_script` falls through to `validate_well_known_lock_scripts` after failing the secp256k1 checks, and `validate_well_known_type_scripts` is called for type scripts: [5](#0-4) [6](#0-5) [7](#0-6) 

Because the well-known lists are empty for non-mainnet/testnet chains, `validate_well_known_lock_scripts` always returns `Err(NotWellKnownLockScript)` and `validate_well_known_type_scripts` always returns `Err(NotWellKnownTypeScript)` on those chains. The officially supported chain list includes `staging`, `preview`, and `dev`: [8](#0-7) 

---

### Impact Explanation

On any officially supported chain other than mainnet or testnet (e.g., staging, preview), an RPC caller using `outputs_validator = "well_known_scripts_only"` will have every transaction that uses an anyone_can_pay lock script or a SUDT type script rejected with `PoolRejectedTransactionByOutputsValidator`. The validator is supposed to be a safety net that allows well-known scripts through; instead it becomes a blanket rejection filter for all non-secp256k1/DAO scripts on those chains. Adding a new chain (e.g., a future public testnet) requires modifying the hardcoded match arms in source code — the design is not extensible.

**Impact: Medium** — limits the functionality of the `WellKnownScriptsOnly` validator on all non-mainnet/testnet chains; transactions are incorrectly rejected even when using valid, well-known scripts.

---

### Likelihood Explanation

**Likelihood: Medium** — The `staging` and `preview` chains are officially bundled and supported. Any operator or RPC caller on those chains who uses `well_known_scripts_only` (the documented safe default) will encounter this. Future public networks would also be affected without a code change.

---

### Recommendation

Replace the hardcoded `match chain_spec_name` blocks with a configurable or chain-spec-driven mechanism. Well-known scripts should be declared in the chain spec file (e.g., `specs/staging.toml`) and loaded at startup, rather than being embedded as string literals in Rust source. Alternatively, expose a configuration option in `ckb.toml` that allows operators to declare additional well-known lock and type scripts per chain, analogous to the existing `extra_well_known_lock_scripts` / `extra_well_known_type_scripts` parameters already accepted by `PoolRpcImpl::new`: [9](#0-8) 

---

### Proof of Concept

1. Start a CKB node with `--chain staging` or `--chain preview`.
2. Construct a transaction whose output uses the anyone_can_pay lock script (code hash `0xd369...` on mainnet, but any valid ACP hash on staging).
3. Submit via `send_transaction` with `"outputs_validator": "well_known_scripts_only"`.
4. The node returns `PoolRejectedTransactionByOutputsValidator` because `build_well_known_lock_scripts("ckb_staging")` returns `[]`, so `validate_well_known_lock_scripts` finds no match and returns `Err(NotWellKnownLockScript)`.
5. The same transaction submitted with `"outputs_validator": "passthrough"` succeeds, confirming the script itself is valid and the rejection is solely due to the hardcoded empty list. [1](#0-0) [10](#0-9)

fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script
            .args()
            .as_slice()
            .starts_with(well_known_script.args().as_slice())
}
