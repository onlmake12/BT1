Audit Report

## Title
Unbounded CPU/IO Amplification via Duplicate `tx_hashes` in `GetTransactionsProof` — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary
The `GetTransactionsProofProcess::execute()` handler accepts up to 1000 `tx_hashes` with no deduplication. Sending 1000 copies of the same confirmed transaction hash triggers 2000 DB reads (1000 `get_transaction_info` + 1000 `get_transaction_with_info`) and a `CBMT::build_merkle_proof` call with 1000 duplicate indices per request. No per-peer rate limiting exists in `LightClientProtocol`, and valid requests return `Status::ok()` (200), which never triggers a ban. This allows any P2P peer to repeatedly saturate the light client server's I/O and CPU at minimal attacker cost.

## Finding Description
The `execute()` function enforces only a count bound: [1](#0-0) 

`GET_TRANSACTIONS_PROOF_LIMIT` is 1000: [2](#0-1) 

After this check, the `partition` call at lines 54–64 invokes `get_transaction_info` for every hash — 1000 DB reads for 1000 duplicate hashes: [3](#0-2) 

The subsequent loop calls `get_transaction_with_info` for every entry in `found` — another 1000 DB reads — and pushes all 1000 `(tx, index)` pairs (with the same index) into the same block's Vec in `txs_in_blocks`: [4](#0-3) 

`CBMT::build_merkle_proof` is then called with 1000 duplicate indices, doing O(N log N) work, and `.expect()` panics if the implementation returns `None` for duplicate indices: [5](#0-4) 

`LightClientProtocol` has no rate limiter field and `try_process` performs no rate-limit check — unlike `Relayer`, which has an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` checked before dispatch: [6](#0-5) [7](#0-6) 

A successful request returns `Status::ok()` (200), which `should_ban()` never acts on — bans only fire for 4xx codes: [8](#0-7) 

## Impact Explanation
**Low (501–2000 points): Any other important performance improvements for CKB.**

The work per request is bounded (O(N log N), N ≤ 1000), so a single request cannot crash the node. However, the 2000× DB-read amplification per request combined with zero rate limiting and zero ban-on-valid-request means a single attacker connection can sustain continuous maximum-cost requests indefinitely, degrading the light client server's I/O throughput and CPU availability. The evidence does not prove a full node crash or main-chain network congestion, placing this firmly in the performance-improvement tier rather than High.

## Likelihood Explanation
- Precondition: any confirmed transaction on the main chain (always satisfied on a running node).
- Attacker capability: any unprivileged P2P peer that can open a connection to a light-client-enabled node.
- No PoW, no key, no privileged role required.
- Repeatable indefinitely with a single connection; no cooldown or ban is applied.

## Recommendation
1. **Deduplicate `tx_hashes`** before processing: convert to a `HashSet` immediately after the length check, before the `partition` call.
2. **Add per-peer rate limiting** to `LightClientProtocol::try_process`, mirroring the `RateLimiter<(PeerIndex, u32)>` pattern already used in `Relayer`.
3. Consider reducing `GET_TRANSACTIONS_PROOF_LIMIT` or adding a per-request cost budget proportional to unique hashes.

## Proof of Concept
```
1. Start a CKB node with light client protocol enabled.
2. Find any confirmed tx_hash H on the main chain.
3. Construct: GetTransactionsProof { last_hash: <valid tip hash>, tx_hashes: [H; 1000] }
4. Send the message repeatedly over a single P2P connection.
5. Observe: each message triggers 2000 DB reads + O(1000 log 1000) CBMT work;
   no ban is applied; requests are processed indefinitely.
```
