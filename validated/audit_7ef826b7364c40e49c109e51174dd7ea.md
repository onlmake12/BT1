The vulnerability is real. Here is the analysis:

---

### Title
Off-by-one epoch gate in `BlockTxsVerifier` allows RFC0044 `DaoScriptSizeVerifier` bypass in the first block of the activation epoch — (`verification/contextual/src/contextual_block_verifier.rs`)

### Summary

`BlockTxsVerifier::verify` gates `DaoScriptSizeVerifier` using the **parent block's** epoch number instead of the **current block's** epoch number. Because `rfc0044_active` uses a strict `>=` comparison, the first block of epoch `RFC0044_ACTIVE_EPOCH` is evaluated against epoch `RFC0044_ACTIVE_EPOCH - 1`, which returns `false`, causing the verifier to be silently skipped for exactly that one block.

### Finding Description

In `BlockTxsVerifier::verify`, the guard is:

```rust
if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {
    DaoScriptSizeVerifier::new(...).verify()?;
}
``` [1](#0-0) 

`rfc0044_active` is defined as:

```rust
pub fn rfc0044_active(&self, target: EpochNumber) -> bool {
    ...
    target >= rfc0044_active_epoch
}
``` [2](#0-1) 

When the block being verified is the **first block of epoch `RFC0044_ACTIVE_EPOCH`**, `self.parent` is the last block of epoch `RFC0044_ACTIVE_EPOCH - 1`. So `self.parent.epoch().number()` equals `RFC0044_ACTIVE_EPOCH - 1`, and `rfc0044_active(RFC0044_ACTIVE_EPOCH - 1)` evaluates to `false`. The `DaoScriptSizeVerifier` is never invoked.

The `DaoScriptSizeVerifier::verify` itself has no epoch-based gate — its only internal guard is a block-number check on the **deposit cell's** commit height (`starting_block_limiting_dao_withdrawing_lock`), not on the current block's epoch. [3](#0-2) 

So the only epoch gate is the one in `BlockTxsVerifier`, and it is off by one.

### Impact Explanation

A miner who produces the first block of epoch `RFC0044_ACTIVE_EPOCH` can include a DAO withdrawal transaction where the withdrawal cell's lock script has a **different serialized size** than the corresponding deposit cell's lock script. This transaction passes all block-level verification (the `DaoScriptSizeVerifier` is not called), and the block is accepted into the canonical chain. The RFC0044 invariant — that deposit and withdrawal cells must use equally-sized lock scripts — is violated for exactly this one block.

### Likelihood Explanation

Any miner with valid PoW for the first block of epoch `RFC0044_ACTIVE_EPOCH` can exploit this. No special privilege is required beyond mining that block. The attacker needs to:
1. Pre-create a DAO deposit cell with a known lock script size.
2. Prepare a withdrawal transaction with a different-sized lock script.
3. Mine the first block of epoch `RFC0044_ACTIVE_EPOCH` and include the transaction.

The window is exactly one block, but it is deterministic and predictable.

### Recommendation

Change the gate to use the **current block's** epoch number, not the parent's:

```rust
// Use self.header.epoch().number() instead of self.parent.epoch().number()
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
    DaoScriptSizeVerifier::new(...).verify()?;
}
``` [1](#0-0) 

Note that `BlockExtensionVerifier` has the same pattern (using `self.parent.epoch().number()` for the `rfc0044_active` check), which may warrant a parallel fix. [4](#0-3) 

### Proof of Concept

Differential test:
- Set `RFC0044_ACTIVE_EPOCH = N`.
- Create a DAO deposit cell committed at or after `starting_block_limiting_dao_withdrawing_lock`.
- Construct a DAO withdrawal transaction where the withdrawal cell's lock script size differs from the deposit cell's.
- Place this transaction in a block whose parent is the **last block of epoch N-1** (i.e., the first block of epoch N).
- Run `BlockTxsVerifier::verify` — it returns `Ok(...)` because `rfc0044_active(N-1)` is `false`.
- Place the same transaction in a block whose parent is the **first block of epoch N** (i.e., the second block of epoch N).
- Run `BlockTxsVerifier::verify` — it returns `Err(DaoLockSizeMismatch)` because `rfc0044_active(N)` is `true`.

The first block of epoch N incorrectly accepts the mismatched withdrawal; the second block correctly rejects it.
