### Title
Echo Protocol Fee Double-Counted: `_state.accruedFeesInWei` Credited at Request Time and Again Inside Provider Balance at Callback Time — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the Echo protocol fee (`_state.pythFeeInWei`) is credited to `_state.accruedFeesInWei` at request time. However, at callback time, the full `req.fee` — which already includes `_state.pythFeeInWei` — is also credited entirely to the provider's `accruedFeesInWei`. This creates a double-counting of the Echo protocol fee: it is recorded in both the admin's accounting bucket and the provider's accounting bucket, but the contract only holds the ETH once. The total claimable amount exceeds the actual ETH held by the contract by `_state.pythFeeInWei` per fulfilled request.

---

### Finding Description

**Request path** (`requestPriceUpdatesWithCallback`):

```solidity
_state.accruedFeesInWei += _state.pythFeeInWei;
```

The Echo protocol fee is credited to the admin's withdrawal bucket at request time. The full `msg.value` (= `pythFeeInWei + providerFees`) is stored as `req.fee`. [1](#0-0) 

**Callback path** (`executeCallback`):

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

Here `pythFee` is `pyth.getUpdateFee(updateData)` — the fee paid to the Pyth price-feed contract for the price update data — **not** the Echo protocol fee (`_state.pythFeeInWei`). Therefore `req.fee` is credited in full to the provider, including the `_state.pythFeeInWei` portion that was already credited to `_state.accruedFeesInWei`. [2](#0-1) 

**Fee structure** (`getFee`):

```solidity
feeAmount = baseFee + providerBaseFee + providerFeedFee + SafeCast.toUint96(gasFee);
```

`baseFee = _state.pythFeeInWei`, confirming that `req.fee` (= user's `msg.value`) includes the Echo protocol fee. [3](#0-2) 

**Withdrawal paths** that decrement the two separate buckets:

```solidity
// Admin withdrawal
_state.accruedFeesInWei -= amount;

// Provider withdrawal
_state.providers[provider].accruedFeesInWei -= amount;
``` [4](#0-3) [5](#0-4) 

**Accounting invariant violation per request:**

| Item | Value |
|---|---|
| ETH received by contract | `req.fee + msg.value_callback` |
| ETH paid out (Pyth price feed) | `pythFee_price_update` |
| Net ETH in contract | `req.fee + msg.value_callback − pythFee_price_update` |
| `_state.accruedFeesInWei` (admin) | `+= pythFeeInWei` |
| `provider.accruedFeesInWei` | `+= req.fee + msg.value_callback − pythFee_price_update` |
| **Total claimable** | `pythFeeInWei + req.fee + msg.value_callback − pythFee_price_update` |
| **Over-commitment per request** | **`pythFeeInWei`** |

---

### Impact Explanation

The total claimable amount across both withdrawal paths exceeds the actual ETH held by the contract by `_state.pythFeeInWei` per fulfilled request. This means:

- If providers withdraw their `accruedFeesInWei` first (the normal operational path, since Fortuna keepers call `executeCallback` and then withdraw), the admin's `withdrawFees` call will revert with `"Insufficient balance"` or drain ETH that belongs to other providers.
- If the admin withdraws first, providers will be unable to withdraw the full amount credited to them.

In either case, one party permanently loses funds equal to `_state.pythFeeInWei × numberOfFulfilledRequests`. The Echo protocol fees are effectively unrecoverable.

---

### Likelihood Explanation

This is triggered by the normal operational flow: any user calling `requestPriceUpdatesWithCallback` followed by any relayer/keeper calling `executeCallback`. No special privileges are required. The discrepancy accumulates with every fulfilled request. The Fortuna keeper service (`apps/fortuna`) is designed to call `executeCallback` automatically, meaning this path is exercised continuously in production. [6](#0-5) 

---

### Recommendation

At callback time, subtract the Echo protocol fee from the amount credited to the provider:

```solidity
// Before (buggy):
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

// After (fixed):
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee - _state.pythFeeInWei + msg.value) - pythFee);
```

This ensures the Echo protocol fee (`_state.pythFeeInWei`) remains exclusively in `_state.accruedFeesInWei` and is not also credited to the provider.

---

### Proof of Concept

1. Admin sets `_state.pythFeeInWei = 1000 wei`.
2. Provider registers with `baseFeeInWei = 500 wei`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`.
3. User calls `requestPriceUpdatesWithCallback{value: 1500}(provider, ...)`.
   - `_state.accruedFeesInWei += 1000` → admin bucket = 1000
   - `req.fee = 1500` stored
   - Contract holds 1500 wei
4. Keeper calls `executeCallback{value: 0}(provider, seq, updateData, priceIds)`.
   - `pythFee = pyth.getUpdateFee(updateData)` = 0 (assume zero for simplicity)
   - `provider.accruedFeesInWei += 1500 + 0 − 0 = 1500`
   - Contract holds 1500 wei
5. Provider calls `withdraw(1500)` → succeeds, contract holds 0 wei.
6. Admin calls `withdrawFees(1000)` → **reverts: "Insufficient balance"** (contract has 0 wei).

The Echo protocol fee of 1000 wei is permanently lost to the admin. The bug is analogous to `Lock.extendLock` not updating `totalLocked[_asset]`: one accounting path increments a tracker that is later decremented, but a parallel path already consumed the underlying funds.

/// Withdraws accumulated fees in the contract as needed to maintain the balance of the keeper wallet.
pub async fn withdraw_fees_if_necessary(
    contract_as_fee_manager: Arc<InstrumentedSignablePythContract>,
    provider_address: Address,
    keeper_address: Address,
    other_keeper_addresses: Vec<Address>,
    min_balance: U256,
) -> Result<()> {
    let provider = contract_as_fee_manager.provider();
    let fee_manager_wallet = contract_as_fee_manager.wallet();

    let keeper_balance = provider
        .get_balance(keeper_address, None)
        .await
        .map_err(|e| anyhow!("Error while getting balance. error: {:?}", e))?;

    // Only withdraw if our balance is below the minimum threshold
    if keeper_balance >= min_balance {
        return Ok(());
    }

    let provider_info = contract_as_fee_manager
        .get_provider_info_v2(provider_address)
        .call()
        .await
        .map_err(|e| anyhow!("Error while getting provider info. error: {:?}", e))?;

    let available_fees = U256::from(provider_info.accrued_fees_in_wei);

    // Determine how much we can fairly withdraw from the contract
    let withdrawal_amount = calculate_fair_fee_withdrawal_amount(
        Arc::new(provider.clone()),
        keeper_address,
        &other_keeper_addresses,
        available_fees,
    )
    .await?;

    // Only withdraw as long as we are at least doubling our keeper balance (avoids repeated withdrawals of tiny amounts)
    let min_withdrawal_amount = keeper_balance;
    if withdrawal_amount < min_withdrawal_amount {
        // We don't have enough to meaningfully top up the balance.
        // NOTE: This log message triggers a grafana alert. If you want to change the text, please change the alert also.
        tracing::warn!("Keeper balance {:?} is too low (< {:?}) but provider fees are not sufficient to top-up. (withdrawal_amount={:?} < min_withdrawal_amount={:?})", keeper_balance, min_balance, withdrawal_amount, min_withdrawal_amount);
        return Ok(());
    }

    tracing::info!(
        "Keeper balance {:?} below minimum {:?}, claiming {:?} out of available {:?}",
        keeper_balance,
        min_balance,
        withdrawal_amount,
        available_fees
    );

    // Proceed with withdrawal
    let contract_call = contract_as_fee_manager
        .withdraw_as_fee_manager(provider_address, withdrawal_amount.as_u128());
    send_and_confirm(contract_call).await?;
