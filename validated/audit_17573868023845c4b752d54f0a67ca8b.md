## Tracing the Logic

**`ADDR_COUNT_LIMIT = 16384`** is defined in `mod.rs`. [1](#0-0) 

**`add_addr`** calls `check_purge()` before inserting, and propagates any error via `?`. [2](#0-1) 

**`check_purge` step 1** collects all addresses where `is_connectable(now_ms)` returns `false`. [3](#0-2) 

**`is_connectable`** returns `false` only if:
- `last_connected_at_ms == 0 && attempts_count >= ADDR_MAX_RETRIES (3)`, OR
- `now_ms - last_connected_at_ms > ADDR_TIMEOUT_MS && attempts_count >= ADDR_MAX_FAILURES (10)` [4](#0-3) 

Addresses inserted via `add_addr` are created with `last_connected_at_ms = 0` and `attempts_count = 0`. With `attempts_count = 0`, neither non-connectable condition is satisfied, so **all freshly-added addresses are connectable** and step 1 evicts nothing. [5](#0-4) 

**`check_purge` step 2** groups addresses by `Group` (i.e., `/16` for IPv4), sorts groups by descending size, takes the top `len/2` groups, and for each group evicts 2 peers **only if `addrs.len() > 4`** (strictly greater than 4). [6](#0-5) 

The `Group::IP4([bits[0], bits[1]])` grouping confirms the `/16` boundary. [7](#0-6) 

---

## The Flaw

The eviction threshold is `> 4`, not `>= 4`. If every `/16` group contains **exactly 4** addresses, the condition `addrs.len() > 4` is `false` for every group, `candidate_peers` remains empty, and `check_purge` returns `Err(PeerStoreError::EvictionFailed)`. [8](#0-7) 

---

## Is the Precondition Reachable?

An unprivileged peer connected via P2P can send `Addr`/discovery messages advertising arbitrary IP addresses. The attacker does not need to own those IPs. With `16384 / 4 = 4096` distinct `/16` groups available (IPv4 has 65536 possible `/16` blocks), the attacker simply advertises exactly 4 addresses per group across 4096 groups. This fills the store to `ADDR_COUNT_LIMIT` with all entries connectable and all groups at size 4.

### Title
Peer Store Eviction Permanently Blocked via Crafted Address Distribution — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
`check_purge` uses a strict `> 4` threshold in its network-group eviction step. An attacker who fills the peer store with exactly 4 connectable addresses per `/16` group causes both eviction steps to produce empty candidate sets, returning `EvictionFailed` on every subsequent `add_addr` call.

### Finding Description
`check_purge` has two eviction stages:

1. Remove non-connectable addresses. Addresses added via `add_addr` have `last_connected_at_ms = 0` and `attempts_count = 0`, which satisfies neither non-connectable condition in `is_connectable`, so step 1 removes nothing.
2. Group by `/16`, sort by group size, take the top half, and evict 2 from each group **only if `addrs.len() > 4`**. If every group has exactly 4 entries, this condition is never true, so step 2 also removes nothing.

Both steps fail → `Err(PeerStoreError::EvictionFailed)` → `add_addr` returns an error → the node cannot record any new peer address.

### Impact Explanation
Once the store is locked, the node cannot learn about new peers via discovery. Its peer set becomes permanently stale. This enables eclipse-like conditions: the node only retains the attacker-seeded addresses, degrading block/transaction propagation and potentially isolating the node from the honest network.

### Likelihood Explanation
The attack requires only a single connected peer sending crafted `Addr` messages — a standard, unprivileged P2P operation. The attacker controls which IP addresses they advertise and can trivially distribute them as 4 per `/16` group. No PoW, no key material, no operator access is needed.

### Recommendation
Change the eviction threshold from strictly greater than 4 to greater than or equal to 4 (i.e., `addrs.len() >= 4`), or alternatively `addrs.len() > 1`, so that any non-singleton group is eligible for eviction when the store is full. Additionally, consider adding a fallback that evicts the oldest/lowest-scored address unconditionally when both steps fail, to guarantee `check_purge` always succeeds when the store is at capacity.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
