### Title
Unvalidated `providerToCredit` Parameter in `Echo.executeCallback` Enables Fee Theft After Exclusivity Period - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
The `Echo.executeCallback` function accepts a caller-supplied `providerToCredit` address and credits the user's pre-paid provider fee to it without validating that the address is the legitimate assigned provider. After the exclusivity period expires, any attacker who registers as a provider can call `executeCallback` with their own address as `providerToCredit`, stealing the fee the user paid for the original provider.

### Finding Description
In `Echo.sol`, `requestPriceUpdatesWithCallback` stores `req.fee = msg.value - pythFeeInWei` — the user's provider fee — against the request at the time of submission. [1](#0-0) 

`executeCallback` enforces that only `req.provider` can be credited during the exclusivity window, but once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, the check is skipped entirely: [2](#0-1) 

After that window, the fee is credited unconditionally to the caller-supplied `providerToCredit`: [3](#0-2) 

There is no check that `providerToCredit` is a registered provider, nor that it matches `req.provider`. An attacker who has registered as a provider and set themselves as fee manager can supply their own address and later drain the credited balance via `withdrawAsFeeManager`: [4](#0-3) 

The `setFeeManager` path that enables this is gated only on `isRegistered`, which any permissionless caller can satisfy by calling `registerProvider`: [5](#0-4) 

### Impact Explanation
An attacker can steal the provider fee paid by any user whose request has passed the exclusivity period without being fulfilled. The attacker pays only the Pyth price-update fee (a small operational cost) and receives `req.fee` — the full provider fee the user paid. This is a direct theft of user funds with no trust assumptions required beyond the attacker registering as a provider, which is permissionless.

### Likelihood Explanation
The exclusivity period is a configurable value (`exclusivityPeriodSeconds`). Any request that the assigned provider fails to fulfill within that window becomes exploitable. The attacker needs only to: (1) register as a provider (permissionless), (2) set themselves as fee manager (permissionless), (3) obtain valid Pyth price update data for the requested `publishTime` (publicly available from Hermes), and (4) call `executeCallback`. This is a realistic, low-barrier attack requiring no privileged access.

### Recommendation
After the exclusivity period, restrict `providerToCredit` to registered providers only, and/or require that `providerToCredit` is the `msg.sender` (i.e., the caller must be the provider they are crediting). This mirrors the pattern used in `setProviderFee` and `withdraw`, where `msg.sender` is the implicit provider identity:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit must be a registered provider"
);
require(
    msg.sender == providerToCredit,
    "Caller must be the provider to credit"
);
```

### Proof of Concept
1. Alice calls `requestPriceUpdatesWithCallback(providerP, publishTime, priceIds, gasLimit)` paying `req.fee` for provider P.
2. Provider P fails to call `executeCallback` within `exclusivityPeriodSeconds`.
3. Attacker calls `registerProvider(baseFee, feePerFeed, feePerGas)` — permissionless.
4. Attacker calls `setFeeManager(attackerAddress)` — permissionless for a registered provider.
5. After `block.timestamp >= publishTime + exclusivityPeriodSeconds`, attacker calls `executeCallback(attackerAddress, sequenceNumber, validUpdateData, priceIds)` with valid Pyth price data obtained from Hermes.
6. `_state.providers[attackerAddress].accruedFeesInWei` is incremented by `req.fee + msg.value - pythFee`.
7. Attacker calls `withdrawAsFeeManager(attackerAddress, stolenAmount)` and receives Alice's provider fee.
