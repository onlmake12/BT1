Audit Report

## Title
Stale Pre-Computed Pool Counters Overwrite Eviction-Adjusted Totals, Permanently Inflating `total_tx_size`/`total_tx_cycles` — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `total_tx_size` and `total_tx_cycles` are snapshotted before `check_and_record_ancestors` runs. That function can evict existing entries via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements the counters in place. However, `add_entry` then unconditionally overwrites those correctly-decremented counters with the stale pre-eviction snapshot, permanently inflating them by the aggregate size/cycles of every evicted entry. Repeated exploitation inflates the counters past pool capacity limits, causing the node's tx-pool to permanently reject all new transaction submissions.

## Finding Description
The exact sequence in `add_entry` is confirmed by the code:

**Step 1** — snapshot taken before any evictions: [1](#0-0) 

**Step 2** — `check_and_record_ancestors` may evict via `remove_entry_and_descendants`, which calls `remove_entry` → `update_stat_for_remove_tx`, decrementing `self.total_tx_size`/`self.total_tx_cycles` in place: [2](#0-1) [3](#0-2) 

**Step 3** — stale snapshot unconditionally overwrites the decremented counters: [4](#0-3) 

The eviction path inside `check_and_record_ancestors` is triggered when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count`: [5](#0-4) 

`updated_stat_for_add_tx` computes `old_total + entry.size` at snapshot time. If evictions remove entries with aggregate size `S_evicted`, `update_stat_for_remove_tx` correctly sets `self.total_tx_size = old_total - S_evicted`. Step 3 then writes `old_total + entry.size`, discarding the `- S_evicted` adjustment. The final counter is inflated by exactly `S_evicted`. [6](#0-5) [7](#0-6) 

The developers acknowledge this eviction scenario is real via a FIXME comment noting rollback is not handled: [8](#0-7) 

The `recompute_total_stat` underflow recovery path in `update_stat_for_remove_tx` does not help here — it only fires on underflow during removal, not on the inflation written by `add_entry`. [9](#0-8) 

## Impact Explanation
`total_tx_size` and `total_tx_cycles` are the sole gate for pool admission via `updated_stat_for_add_tx`. Once inflated past the configured pool capacity limits, every subsequent `add_entry` call returns a pool-full rejection, permanently blocking all new transaction submissions to the node's tx-pool. The node continues to sync blocks but cannot participate in transaction relay or block assembly. This constitutes a **High** impact: a targeted, low-cost attack that can permanently disable a CKB node's transaction ingestion, and if applied to multiple nodes simultaneously, causes CKB network congestion — matching the allowed impact "Vulnerabilities or bad designs which could cause CKB network congestion with few costs." [10](#0-9) 

## Likelihood Explanation
The exploit requires only unprivileged access to `send_transaction` RPC or P2P relay. The attacker pays only minimum fees per transaction. The required conditions — a cell-dep ancestor chain of length ≥ `max_ancestors_count` (default 25) with a new transaction referencing one of those ancestors as a cell dep — are straightforwardly constructable. The exploit is repeatable across multiple `add_entry` calls to accumulate inflation until the pool is permanently closed. No special privileges, leaked keys, or victim mistakes are required. [11](#0-10) 

## Recommendation
Remove the pre-computation before `check_and_record_ancestors`. After all evictions complete and the new entry is inserted, apply the size/cycle delta directly on the post-eviction baseline:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Apply delta on post-eviction baseline, not a pre-eviction snapshot:
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

Pool capacity validation (against `max_tx_pool_size` / `max_block_cycles`) should also be moved to after evictions complete, so the check reflects the actual post-eviction pool state. [12](#0-11) 

## Proof of Concept
1. Configure a node with `max_ancestors_count = 25` (default).
2. Submit 25 transactions forming a linear chain `tx_0 → tx_1 → … → tx_24` (each spends an output of the previous). All 25 are accepted.
3. Submit `tx_A` that spends an output of `tx_0` **and** uses `tx_12` as a cell dep. `tx_A`'s ancestor set is `tx_0`…`tx_24` (26 > 25). The cell-dep path satisfies `26 - 1 <= 25`, triggering eviction of `tx_12` and its descendants `tx_13`…`tx_24` (13 entries, aggregate size `S_evicted`).
4. `check_and_record_ancestors` calls `remove_entry_and_descendants(&tx_12_id)`, decrementing `self.total_tx_size` by `S_evicted`.
5. `add_entry` writes `self.total_tx_size = old_total + tx_A.size` (the pre-eviction snapshot), restoring the inflated value. `total_tx_size` is now inflated by `S_evicted`.
6. Repeat steps 2–5 with fresh transactions. Each iteration inflates `total_tx_size` by another `S_evicted`.
7. Once `total_tx_size` exceeds the configured pool size limit, all subsequent `send_transaction` RPC calls return a pool-full rejection even though the pool contains far fewer bytes than the limit.

A unit test can verify this by asserting `pool_map.total_tx_size == pool_map.recompute_total_stat().unwrap().0` after each `add_entry` call that triggers the eviction path — the assertion will fail on the buggy code. [13](#0-12)

                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
