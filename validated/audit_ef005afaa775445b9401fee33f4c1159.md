Audit Report

## Title
Stale Pre-Eviction `total_tx_size`/`total_tx_cycles` Overwrite Inflates Pool Accounting After Ancestor-Eviction — (File: tx-pool/src/component/pool_map.rs)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` computes a snapshot of `total_tx_size` and `total_tx_cycles` before `check_and_record_ancestors` runs. When that function evicts transactions via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, the live fields are correctly decremented. However, lines 218–219 unconditionally overwrite those fields with the stale pre-eviction snapshot, permanently inflating both counters by the sum of all evicted transactions' sizes and cycles. This causes valid transactions to be incorrectly rejected as `Reject::Full` and triggers unnecessary eviction of fee-paying transactions already in the pool.

## Finding Description
The exact sequence in `add_entry` (lines 200–221):

- **Lines 210–211**: `updated_stat_for_add_tx(entry.size, entry.cycles)` reads `self.total_tx_size` and `self.total_tx_cycles` and returns `(self.total_tx_size + entry.size, self.total_tx_cycles + entry.cycles)` as a local snapshot. No evictions have occurred yet.
- **Line 213**: `check_and_record_ancestors(&mut entry)` is called. When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (line 603), it enters the eviction loop (lines 615–625), calling `remove_entry_and_descendants` for each candidate. That calls `remove_entry` (line 263), which calls `update_stat_for_remove_tx` (line 247), correctly decrementing `self.total_tx_size` and `self.total_tx_cycles` for each evicted transaction.
- **Lines 218–219**: `self.total_tx_size = total_tx_size` and `self.total_tx_cycles = total_tx_cycles` unconditionally restore the stale pre-eviction snapshot, discarding all decrements performed during eviction.

Net result after one insertion that evicts N bytes of transactions:
```
correct value:  original_total - evicted_bytes + entry.size
stored value:   original_total               + entry.size   ← inflated by evicted_bytes
```

Every subsequent insertion that triggers the same eviction path compounds the inflation monotonically. The `recompute_total_stat` fallback at line 743 is only invoked on underflow during removal, not after `add_entry`, so it provides no protection here.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The inflated `total_tx_size` directly drives two enforcement paths:
1. `updated_stat_for_add_tx` (line 716) rejects new submissions with `Reject::Full` when `self.total_tx_size + tx_size` overflows or exceeds configured limits. A pool genuinely well below capacity will appear full, causing valid transactions to be permanently rejected.
2. External callers of `limit_size` loop while `total_tx_size > max_tx_pool_size` and evict additional valid, fee-paying transactions that should have remained. Each eviction further degrades pool quality.

An attacker can repeatedly trigger this path at negligible cost, continuously inflating the counter and degrading the mempool's ability to accept legitimate transactions — a sustained, low-cost denial-of-service against the tx-pool.

## Likelihood Explanation
The eviction branch in `check_and_record_ancestors` (lines 603–625) is reachable by any unprivileged submitter. The required precondition — a transaction chain at or near `max_ancestors_count` (default: 25) with at least one cell-dep-referencing parent — is achievable on a live network by any user who can submit transactions. No special privilege, leaked keys, or majority hash power is needed. The attacker can repeat the trigger indefinitely, compounding the inflation with each iteration.

## Recommendation
Move `updated_stat_for_add_tx` to execute **after** `check_and_record_ancestors` completes, so it reads the already-decremented live fields:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Compute AFTER evictions have already updated self.total_tx_size/cycles
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size  = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, call `recompute_total_stat()` after `check_and_record_ancestors` to rebuild counters from scratch before applying the new entry's contribution.

## Proof of Concept
Minimal manual steps:

1. Set `max_ancestors_count = 2` and `max_tx_pool_size = 1200` in config.
2. Submit `tx_A` (size=500) and `tx_B` (size=500, cell-dep on `tx_A`). Pool: `total_tx_size = 1000`.
3. Submit `tx_C` (size=100, cell-dep on `tx_A`). Ancestor count = 3 > 2; `cell_ref_parents = {tx_B}`; eviction branch fires.
   - `updated_stat_for_add_tx`: snapshot = 1000 + 100 = **1100**
   - `check_and_record_ancestors` evicts `tx_B` (500): `self.total_tx_size = 500`
   - Lines 218–219 write back **1100**
4. Actual pool: `tx_A` (500) + `tx_C` (100) = 600 bytes. Reported `total_tx_size` = **1100**.
5. Submit `tx_D` (size=200): `updated_stat_for_add_tx` sees 1100 + 200 = 1300 > 1200 → `Reject::Full`, even though actual occupancy is 800/1200.

A unit test asserting `pool_map.total_tx_size == actual_sum_of_entry_sizes` after this sequence will fail on the unpatched code and pass after the fix.
