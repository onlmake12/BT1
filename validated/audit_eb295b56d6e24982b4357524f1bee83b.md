Audit Report

## Title
Stale Aggregate Counter Overwrite After Ancestor-Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` snapshots `self.total_tx_size + entry.size` into a local variable at lines 210–211, before `check_and_record_ancestors` (line 213) may evict transactions via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`. The stale snapshot is then unconditionally written back to `self.total_tx_size` / `self.total_tx_cycles` at lines 218–219, silently undoing every decrement performed during eviction. This permanently inflates the pool's size and cycle counters, causing spurious downstream evictions and false `Reject::Full` responses.

## Finding Description
The exact sequence in `add_entry` (lines 200–221):

```rust
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // snapshot: old_total + new_size
// ...
evicts = self.check_and_record_ancestors(&mut entry)?;          // may evict, decrementing self.total_tx_size
// ...
self.total_tx_size = total_tx_size;   // OVERWRITES with stale snapshot
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) reads `self.total_tx_size` and returns `self.total_tx_size + tx_size` — a pure snapshot, not a mutation. It does not modify `self.total_tx_size`.

`check_and_record_ancestors` (lines 590–639) enters the eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count.saturating_sub(cell_ref_parents.len()) <= max_ancestors_count` (line 603). Inside that branch, `remove_entry_and_descendants` is called in a loop (line 618), which chains to `remove_entry` (line 235), which calls `update_stat_for_remove_tx` (line 247), which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in place (lines 738–741).

After `check_and_record_ancestors` returns, lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot. The result is:

```
self.total_tx_size = pre_eviction_total + new_entry_size
```

instead of the correct:

```
self.total_tx_size = post_eviction_total + new_entry_size
```

The inflation equals the sum of all evicted transaction sizes. No existing guard corrects this: `recompute_total_stat` (lines 699–708) is only called on underflow in `update_stat_for_remove_tx`, never after `add_entry`.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

1. **Spurious eviction loop**: `limit_size` (pool.rs lines 298–307) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated counter causes the loop to evict valid, fee-paying transactions that should remain in the pool.
2. **False `Reject::Full`**: `updated_stat_for_add_tx` rejects new submissions when the inflated `total_tx_size` appears to exceed capacity, even though the real pool is well within limits.
3. **Repeatability**: Each triggering submission inflates the counters further. An attacker can drive the node into a state where it rejects all new transactions with `Reject::Full` despite the pool being nearly empty. Targeting multiple nodes causes network-wide congestion.

## Likelihood Explanation
Any unprivileged user with access to `send_transaction` RPC or P2P relay can trigger this. The required setup — N > `max_ancestors_count` pooled transactions sharing a common `cell_dep` output (cell-ref parents), followed by a transaction consuming that output as an input — is a standard transaction pattern. The eviction branch at line 603 is explicitly exercised by the existing integration test `TxPoolLimitAncestorCount`. No special privilege, key material, or majority hashpower is required. Attack cost is only the transaction fees for the crafted chain, and the inflation is permanent until node restart.

## Recommendation
Move the snapshot computation to **after** `check_and_record_ancestors` completes, so it reads the already-correct post-eviction `self.total_tx_size`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

This ensures the final written values are always `post_eviction_total + new_entry_contribution`.

## Proof of Concept
1. Configure a node with default `max_ancestors_count` (e.g., 125).
2. Submit N > `max_ancestors_count` transactions, each spending an independent input but all referencing the same `cell_dep` output. All are accepted into the pool.
3. Submit a new transaction whose **input** consumes that same cell output. Its ancestor count = N + 1 > `max_ancestors_count`, but `(N+1) - N = 1 <= max_ancestors_count`, so the eviction branch at line 603 fires.
4. `check_and_record_ancestors` evicts `(N + 1 - max_ancestors_count)` cell-ref transactions, each correctly decrementing `self.total_tx_size` via `update_stat_for_remove_tx`.
5. Lines 218–219 overwrite `self.total_tx_size` with the stale snapshot = `old_total + new_entry_size`, which does not reflect the evictions.
6. Query `tx_pool_info` RPC: `total_tx_size` is inflated by `sum(evicted_tx_sizes)`.
7. Repeat step 3 with additional transactions to drive `total_tx_size` above `max_tx_pool_size`; observe that `limit_size` begins evicting valid transactions and new submissions receive `Reject::Full` despite the pool being underfull.

Adding an assertion `assert_eq!(pool.tx_pool_info().total_tx_size, actual_sum_of_pooled_tx_sizes)` after the eviction step in `TxPoolLimitAncestorCount` would reproduce the invariant violation.
