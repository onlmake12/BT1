Audit Report

## Title
Missing `BLOCK_INVALID` Status Check in `HeaderAcceptor::accept()` Enables Repeated Re-Validation of Rejected Headers - (File: sync/src/synchronizer/headers_process.rs)

## Summary
`HeaderAcceptor::accept()` fetches block status but only short-circuits on `HEADER_VALID`, not on `BLOCK_INVALID`. A developer `FIXME` comment at line 301 explicitly acknowledges this gap. An unprivileged P2P peer can repeatedly deliver headers already marked `BLOCK_INVALID` (specifically those that failed `non_contextual_check`, e.g. invalid PoW with a valid parent), forcing the node to re-execute PoW verification via `HeaderVerifier::verify` on every delivery with no short-circuit.

## Finding Description
In `HeaderAcceptor::accept()`, the status check at lines 301–322 reads:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer state and return early
    return result;
}
```

`BLOCK_INVALID` is defined as `1 << 12` in `shared/src/block_status.rs`, a bit entirely separate from `HEADER_VALID = 1`. When `status` is `BLOCK_INVALID`, `status.contains(BlockStatus::HEADER_VALID)` is false, so execution falls through to the full validation pipeline.

The critical re-execution path is through `non_contextual_check` (lines 334–344), which calls `self.verifier.verify(self.header)`. `HeaderVerifier::verify` in `verification/src/header_verifier.rs` runs **PoW verification first** (line 34: `PowVerifier::new(header, self.consensus.pow_engine().as_ref()).verify()?`), invoking the Eaglesong hash computation. A `PowError::InvalidNonce` is not a `BlockError::UnknownParent` and not a `TooNew` `HeaderError`, so `non_contextual_check` returns `Err(true)` (is_invalid), causing `shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID)` at line 341 — the same status that was already stored.

The correct attack vector: an attacker sends a header with a **valid, known parent** (so `prev_block_check` passes) but an **invalid PoW nonce**. This causes `BLOCK_INVALID` to be stored. On every subsequent re-delivery of the same header, `prev_block_check` passes again and `non_contextual_check` re-runs the full Eaglesong hash computation before re-storing the same `BLOCK_INVALID` status.

The analogous guard is correctly present in `sync/src/relayer/compact_block_process.rs` lines 259–261:
```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
```
No equivalent guard exists in `HeaderAcceptor::accept()`.

## Impact Explanation
This maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. An attacker with multiple standard P2P connections floods target nodes with `SendHeaders` messages containing headers already marked `BLOCK_INVALID` (valid parent, invalid PoW nonce). Each message triggers Eaglesong hash computation for up to `MAX_HEADERS_LEN` headers. At scale — multiple attacker peers each sending maximum-size `SendHeaders` messages in a tight loop — the sync thread's CPU budget is consumed re-verifying permanently-rejected headers, degrading or halting legitimate header synchronization. If enough nodes are targeted simultaneously, the network's ability to propagate new blocks is impaired.

## Likelihood Explanation
The attack requires only a standard P2P connection — no keys, stake, or privileged role. Crafting a header with a valid parent hash (any known chain tip) and an arbitrary (invalid) nonce is trivial. The `FIXME` comment at line 301 confirms developer awareness that the check is missing. The attack is fully repeatable, stateless on the attacker side, and amplifiable by opening multiple peer connections or packing `MAX_HEADERS_LEN` distinct `BLOCK_INVALID` headers per message.

## Recommendation
Add an early-return guard for `BLOCK_INVALID` immediately after fetching block status in `HeaderAcceptor::accept()`, mirroring the pattern in `compact_block_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // existing early-return logic
}
```

This eliminates all downstream validation work (including PoW re-verification) for permanently-rejected headers and resolves the acknowledged `FIXME`.

## Proof of Concept
1. Attacker connects to a victim CKB node as a standard P2P peer.
2. Attacker crafts a `SendHeaders` message containing a header `H` where: (a) `H.parent_hash` points to a known, valid block (so `prev_block_check` passes), and (b) `H.nonce` is arbitrary/invalid (so PoW verification fails).
3. Node processes `H`: `prev_block_check` passes, `non_contextual_check` calls `HeaderVerifier::verify` → `PowVerifier::verify` → Eaglesong hash computed → `PowError::InvalidNonce` → `is_invalid = true` → `shared.insert_block_status(H.hash(), BlockStatus::BLOCK_INVALID)`.
4. Attacker enters a tight loop sending `SendHeaders` messages containing `H` (now `BLOCK_INVALID`).
5. On each delivery: `accept()` fetches status (`BLOCK_INVALID`), `HEADER_VALID` is not set, falls through to `prev_block_check` (passes), then `non_contextual_check` re-runs the full Eaglesong computation before re-storing `BLOCK_INVALID`.
6. To amplify: pack `MAX_HEADERS_LEN` distinct `BLOCK_INVALID` headers (each with a different invalid nonce but valid parent) per message; open multiple peer connections to multiply throughput.
