### Title
Lazer Signed Price Update Lacks Chain Binding, Enabling Cross-Chain Replay — (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` verifies a Lazer price update by recovering the ECDSA signer from `keccak256(payload)`. The `payload` contains only price data (magic, timestamp, channel, feed values) with no chain ID or contract address. Because the same trusted signer is registered on every EVM deployment, a valid signed update from one chain (e.g., a testnet) passes verification on any other chain (e.g., mainnet), enabling cross-chain replay.

---

### Finding Description

`verifyUpdate` in `PythLazer.sol` performs the following steps:

1. Slices `payload = update[71:71+payload_len]`
2. Computes `hash = keccak256(payload)`
3. Recovers the signer via ECDSA
4. Checks `isValidSigner(signer)` [1](#0-0) 

The payload format parsed by `parsePayloadHeader` contains only: a 4-byte magic, an 8-byte timestamp, a 1-byte channel, a 1-byte feed count, and feed price data. There is no `chainId`, no contract address, and no nonce. [2](#0-1) 

The same contract address `0xACeA761c27A909d4D3895128EBe6370FDE2dF481` is deployed on both testnets (sepolia, base_sepolia, arbitrum_sepolia, bsc_testnet, etc.) and mainnets (base, arbitrum, polygon, bsc, etc.): [3](#0-2) 

Because the same governance-managed trusted signer is registered across all these deployments, a signed payload that passes `verifyUpdate` on one chain passes on every other chain.

Additionally, `verifyUpdate` is entirely stateless — it stores no record of processed updates. The test suite confirms the same `update` bytes can be submitted multiple times and both calls succeed: [4](#0-3) 

---

### Impact Explanation

An attacker who observes a valid signed Lazer price update on a testnet (e.g., Sepolia) can submit the identical `update` bytes to `verifyUpdate` on any mainnet deployment (e.g., Base, Arbitrum, Polygon). The call succeeds and returns the payload and signer as if the update were legitimately produced for that chain. Consumer contracts that rely solely on `verifyUpdate` for price integrity — without independently verifying chain context — will accept the replayed price data. This can result in stale or environment-mismatched prices being consumed on mainnet, potentially enabling price manipulation in DeFi protocols built on top of Lazer.

---

### Likelihood Explanation

The same contract address is deployed on 20+ chains including both testnets and mainnets. The trusted signer set is managed by a single governance process and is expected to be consistent across deployments. An attacker only needs to observe any valid signed Lazer update (publicly available from the Lazer WebSocket stream) and submit it to a different chain's `verifyUpdate`. No privileged access is required.

---

### Recommendation

Bind the signed payload to the target chain and contract by including `block.chainid` and `address(this)` in the signed data. For example, the signer should sign `keccak256(abi.encodePacked(chainId, contractAddress, payload))` rather than `keccak256(payload)`. This ensures a signature produced for one deployment is cryptographically invalid on any other.

Alternatively, maintain a per-chain nonce or sequence number in the signed payload and track consumed nonces on-chain to prevent both cross-chain and same-chain replay.

---

### Proof of Concept

1. Subscribe to the Lazer WebSocket stream on Sepolia and capture a valid `update` blob (the `evm` field from the stream response).
2. Call `verifyUpdate{value: 1 wei}(update)` on the Sepolia `PythLazer` at `0xACeA761c27A909d4D3895128EBe6370FDE2dF481` — it succeeds.
3. Submit the **identical** `update` bytes to `verifyUpdate{value: 1 wei}(update)` on the Base mainnet `PythLazer` at `0xACeA761c27A909d4D3895128EBe6370FDE2dF481`.
4. The call succeeds and returns the same `(payload, signer)` — the testnet price data is now accepted as valid on Base mainnet.
