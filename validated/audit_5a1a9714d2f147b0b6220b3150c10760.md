### Title
`check_tx_fee` Uses Serialized Size Instead of Actual Weight, Allowing High-Cycle Transactions to Bypass `min_fee_rate` — (`tx-pool/src/util.rs`)

---

### Summary

The `check_tx_fee` function computes the minimum required fee using only the transaction's serialized byte size, ignoring the cycles component of the transaction weight. This is the same class of omission as the Mantra bug: a fee-adequacy check that misses one of the cost dimensions. A transaction sender can craft a script that consumes many VM cycles but has a small serialized size, pay a fee that satisfies the size-only check, and have the transaction admitted to the pool with an effective fee rate far below `min_fee_rate`.

---

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum acceptable fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The code comment immediately above this line explicitly acknowledges the approximation:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [2](#0-1) 

However, the actual transaction weight used everywhere else in the system — for fee-rate computation, pool scoring, block assembly, and relay — is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

`TxEntry::fee_rate()` uses this weight-based formula:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [4](#0-3) 

The process flow is:
1. `check_tx_fee` runs **before** script execution — cycles are unknown, so size is used as a proxy.
2. `verify_rtx` runs scripts and determines actual cycles.
3. The transaction is added to the pool as a `TxEntry` with actual cycles — but **no post-verification fee rate check** is performed using the real weight. [5](#0-4) 

This means a transaction that passes the pre-verification size-based check can have an actual fee rate (computed with weight) far below `min_fee_rate`.

---

### Impact Explanation

An attacker submits a transaction with:
- Small serialized size (e.g., 200 bytes)
- High cycles (e.g., 50,000,000 — within the default `max_tx_verify_cycles` of 70,000,000)
- Fee = `min_fee_rate × size / 1000` = 200 shannons (at 1,000 shannons/KW)

`check_tx_fee` passes: `200 >= 1000 × 200 / 1000`.

Actual weight: `max(200, 50,000,000 × 0.000_170_571_4)` = `max(200, 8,528)` = **8,528**.

Actual fee rate: `200 × 1000 / 8,528` ≈ **23 shannons/KW** — 97 % below `min_fee_rate`. [6](#0-5) 

The transaction is admitted to the pool, bypassing the node operator's fee rate policy. Because the relay protocol also calls `check_tx_fee` on received transactions, such transactions can propagate across the network. The pool eviction policy (lowest fee rate evicted first) means these transactions are evicted under pressure, but until then they occupy pool space and consume relay bandwidth at a fraction of the intended cost.

---

### Likelihood Explanation

The entry path is the standard `send_transaction` RPC, reachable by any unprivileged transaction sender. Crafting a high-cycle, low-size script requires only writing a loop in a RISC-V program — a straightforward task for any script author. The attacker still pays some fee, but at a fraction of the required rate. No special privileges, keys, or majority hash power are needed. [7](#0-6) 

---

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee rate check using the real weight:

```rust
// Post-verification check with actual weight
let weight = get_transaction_weight(tx_size, completed.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

This mirrors the fix applied in the Mantra case: include the previously omitted cost dimension in the adequacy check.

---

### Proof of Concept

```
Transaction:
  serialized_size_in_block = 200 bytes
  script: loop { /* nop */ } × 50_000_000 iterations  → cycles ≈ 50_000_000
  fee = 200 shannons

check_tx_fee:
  min_fee = min_fee_rate.fee(200) = 1000 * 200 / 1000 = 200 shannons
  fee (200) >= min_fee (200)  → PASS

Actual weight:
  max(200, 50_000_000 * 0.000_170_571_4) = max(200, 8528) = 8528

Actual fee_rate:
  200 * 1000 / 8528 ≈ 23 shannons/KW   (min_fee_rate = 1000 shannons/KW)

Result: transaction admitted to pool with fee rate 97% below node policy.
``` [5](#0-4) [3](#0-2)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
