### Title
Hardcoded `MAX_BLOCK_BYTES` Constant in `WeightUnitsFlow` Fee Estimator Produces Incorrect Fee Rate Estimates for Non-Default Chain Configurations — (`File: util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

The `WeightUnitsFlow` fee estimator's `do_estimate` function uses the compile-time constant `MAX_BLOCK_BYTES` (imported from `ckb_chain_spec::consensus`) to compute `removed_weight` — the amount of transaction weight that can be cleared per block. This constant is the **default** mainnet value and is not read from the running chain's actual consensus parameters. Any chain (testnet, devnet, or custom) configured with a different `max_block_bytes` will receive systematically incorrect fee rate estimates from the `estimate_fee_rate` RPC.

---

### Finding Description

In `util/fee-estimator/src/estimator/weight_units_flow.rs`, the `WeightUnitsFlow` algorithm imports and uses the compile-time constant `MAX_BLOCK_BYTES`:

```rust
use ckb_chain_spec::consensus::MAX_BLOCK_BYTES;
``` [1](#0-0) 

This constant is defined as:

```rust
pub const MAX_BLOCK_BYTES: u64 = TWO_IN_TWO_OUT_BYTES * TWO_IN_TWO_OUT_COUNT;
// = 597 * 1_000 = 597_000
``` [2](#0-1) 

Inside `do_estimate`, this constant is used directly to compute how much weight each mined block removes from the mempool:

```rust
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
``` [3](#0-2) 

The `Consensus` struct exposes the actual per-chain `max_block_bytes` as a runtime value via `consensus.max_block_bytes()`: [4](#0-3) 

The fallback fee estimator in `tx-pool/src/pool.rs` correctly reads this runtime value:

```rust
self.snapshot.consensus().max_block_bytes() as usize,
``` [5](#0-4) 

But the primary `WeightUnitsFlow` algorithm does not receive a `Consensus` reference and instead relies on the hardcoded constant, creating an inconsistency between the two estimators.

The `estimate_fee_rate` RPC call path is:

`ExperimentRpcImpl::estimate_fee_rate` → `TxPoolController::estimate_fee_rate` → `TxPoolProcessor::estimate_fee_rate` → `FeeEstimator::WeightUnitsFlow::estimate_fee_rate` → `do_estimate` [6](#0-5) [7](#0-6) 

---

### Impact Explanation

- **Overestimated fee rate** (actual `max_block_bytes` > constant): The algorithm underestimates `removed_weight`, concludes that blocks cannot drain the mempool fast enough, and recommends a higher fee rate than necessary. Users overpay.
- **Underestimated fee rate** (actual `max_block_bytes` < constant): The algorithm overestimates `removed_weight`, concludes that blocks drain the mempool faster than they actually do, and recommends a fee rate that is too low. Transactions submitted using this estimate may not be confirmed within the expected number of blocks.

On mainnet the constant matches the actual value, so there is no impact. On testnet, devnet, or any custom chain with a non-default `max_block_bytes`, every call to `estimate_fee_rate` with the `WeightUnitsFlow` estimator returns a systematically wrong result.

---

### Likelihood Explanation

The `estimate_fee_rate` RPC endpoint is publicly accessible to any RPC caller. The `WeightUnitsFlow` estimator is the primary (non-fallback) algorithm. Any operator running a non-mainnet node with a custom `max_block_bytes` (which is a supported chain spec parameter) will silently receive incorrect estimates. The miscalculation is deterministic and persistent — it does not require any special timing or race condition.

---

### Recommendation

Pass the actual consensus `max_block_bytes` value into the `WeightUnitsFlow::Algorithm::estimate_fee_rate` and `do_estimate` functions as a parameter, rather than importing the compile-time constant. This mirrors the correct approach already used in the fallback estimator:

```rust
// In do_estimate, replace:
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
// With:
let removed_weight = (actual_max_block_bytes * 85 / 100) * target_blocks;
```

The `actual_max_block_bytes` should be sourced from `Consensus::max_block_bytes()` and threaded through from the call site in `TxPoolProcessor::estimate_fee_rate`, which already has access to the snapshot and consensus.

---

### Proof of Concept

1. Start a CKB node with a custom chain spec setting `max_block_bytes` to a value different from `597_000` (e.g., `1_000_000`).
2. Submit enough transactions to fill the mempool.
3. Call `estimate_fee_rate` via the Experiment RPC.
4. The `WeightUnitsFlow` estimator computes `removed_weight` using `597_000 * 85 / 100 = 507_450` per block instead of the actual `1_000_000 * 85 / 100 = 850_000`.
5. The returned fee rate estimate is higher than necessary — the algorithm incorrectly concludes the mempool cannot be drained at the actual block capacity.
