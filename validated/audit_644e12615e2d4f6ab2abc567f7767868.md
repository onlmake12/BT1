Audit Report

## Title
`ScriptHashTypeVerifier` Skips Type Script Hash Type Validation, Allowing Disallowed Hash Types Into Tx-Pool — (`File: verification/src/transaction_verifier.rs`)

## Summary
`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates `hash_type` only for each output's lock script against `ENABLED_SCRIPT_HASH_TYPE`, never inspecting the type script. The `ScriptHashType` enum already defines `Data3` through `Data127` (values 6, 8, …, 254) as valid variants today via `seq!(N in 3..=127 {...})`, all of which pass `check_data` but are absent from `ENABLED_SCRIPT_HASH_TYPE`. A transaction whose output carries a type script with `hash_type = 6` (Data3) bypasses this non-contextual gate, enters the tx-pool, and is relayed to peers before being rejected at the more expensive contextual verification stage.

## Finding Description
`ScriptHashTypeVerifier::verify()` at `verification/src/transaction_verifier.rs` lines 796–814 loops over outputs and calls `TryInto::<ScriptHashType>::try_into(output.lock().hash_type())`, checking the result against `ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}`. The type script (`output.type_()`) is never examined.

The `ScriptHashType` enum in `util/gen-types/src/core.rs` is generated with `seq!(N in 3..=127 { Data~N = N << 1 })`, producing `Data3 = 6`, `Data4 = 8`, …, `Data127 = 254` as valid Rust enum variants today. `ScriptHashType::verify_value(v)` returns `true` for any even `v` or `v == 1`, so `verify_value(6)` is `true`. Consequently `check_data` passes for a type script with `hash_type = 6`, and `TryInto::<ScriptHashType>::try_into(6u8)` succeeds with `Ok(Data3)`.

Exploit path:
1. Craft a `TransactionView` with one output: lock script `hash_type = 0` (Data, enabled), type script `hash_type = 6` (Data3, valid enum variant, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via RPC `send_transaction` or P2P relay.
3. `check_data` passes (molecule-level validity).
4. `ScriptHashTypeVerifier::verify()` checks only the lock script (hash_type 0 ∈ `ENABLED_SCRIPT_HASH_TYPE`) and returns `Ok(())`.
5. Transaction enters the tx-pool and is relayed to peers.
6. Contextual verification (script execution) eventually rejects it via the `hash_type => Err(ScriptError::InvalidScriptHashType(...))` catch-all in `select_version` (`script/src/types.rs` lines 930–935), but only after expensive processing.

The claim that `CellbaseVerifier` has the same gap is incorrect: lines 126–133 of `verification/src/block_verifier.rs` already explicitly forbid any type script on cellbase outputs (`output.type_().is_some()` → `InvalidTypeScript`), making the hash-type loop at lines 135–144 irrelevant for type scripts there.

The claimed "consensus split" impact is not supported: contextual block verification runs `select_version` for every script and rejects blocks containing transactions with unenabled hash types, so no chain divergence occurs in the current implementation.

## Impact Explanation
The missing check allows an attacker to inject transactions with disallowed type-script hash types through the cheap non-contextual gate into the tx-pool and across the P2P relay layer. Each such transaction is propagated to all peers, each of which must run the more expensive contextual verification before discarding it. This constitutes a resource-amplification vector: low cost to the attacker (transaction creation), higher cost to the network (contextual verification on every relaying node). This matches the allowed impact: **Low (501–2000 points) — any other important performance improvements for CKB**, specifically the elimination of unnecessary resource expenditure caused by a missing cheap admission check.

## Likelihood Explanation
The exploit is triggerable today without any future protocol changes. `Data3 = 6` is already a valid `ScriptHashType` enum variant, passes `check_data`, and is absent from `ENABLED_SCRIPT_HASH_TYPE`. Any unprivileged user with RPC access or P2P connectivity can submit such a transaction. No special privileges, victim interaction, or external conditions are required.

## Recommendation
In `ScriptHashTypeVerifier::verify()` (`verification/src/transaction_verifier.rs`), after the existing lock-script check, add an equivalent check for the optional type script:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check (unchanged)
        check_hash_type(output.lock().hash_type())?;

        // add: type script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

The `CellbaseVerifier` in `verification/src/block_verifier.rs` does not require the same fix because type scripts on cellbase outputs are already unconditionally rejected at lines 126–133.

## Proof of Concept
```rust
// hash_type = 6 is Data3: valid enum variant, passes check_data, NOT in ENABLED_SCRIPT_HASH_TYPE
let type_script = Script::new_builder()
    .hash_type(6u8.into())   // Data3
    .build();
let lock_script = Script::new_builder()
    .hash_type(0u8.into())   // Data, enabled
    .build();
let output = CellOutput::new_builder()
    .lock(lock_script)
    .type_(Some(type_script).pack())
    .build();
let tx = TransactionBuilder::default()
    .output(output)
    .build();

// Passes check_data: verify_value(6) == true
// Passes ScriptHashTypeVerifier: only lock (hash_type=0) is checked
let result = ScriptHashTypeVerifier::new(&tx).verify();
assert!(result.is_ok()); // disallowed type-script hash type not detected
```

The transaction then enters the tx-pool and is relayed to peers, where each node must run contextual verification before discarding it.
