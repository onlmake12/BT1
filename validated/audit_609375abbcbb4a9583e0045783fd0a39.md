Audit Report

## Title
`check_tx_fee` Enforces Minimum Fee Rate Using Serialized Size Instead of Weight, Allowing Below-Minimum Fee Rate Transactions Into the Pool — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, while the rest of the system uses `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` returns actual cycles, no weight-based fee-rate gate is applied before `submit_entry`. An attacker can craft a transaction with high VM cycles and small serialized size that passes the size-only admission check but carries an actual fee rate far below `min_fee_rate`, and the transaction is then relayed to all peers who apply the same flawed check.

## Finding Description

`check_tx_fee` at `tx-pool/src/util.rs:45` computes:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code itself acknowledges the mismatch with an inline comment at lines 42–44: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly."*

The weight-based formula used everywhere else (`TxEntry::fee_rate`, scoring, eviction, block assembly) is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

In `_process_tx` (`tx-pool/src/process.rs:705–777`), the flow is:

1. `pre_check` → calls `check_tx_fee` with size only (L715, L289)
2. `verify_rtx` → executes scripts, returns actual `verified.cycles` (L724–734)
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is constructed (L751)
4. `submit_entry` is called immediately (L753)

There is **no post-execution fee-rate check** using `get_transaction_weight(tx_size, verified.cycles)` between steps 2 and 4. The entry enters the pool with cycles that may make its true weight-based fee rate far below `min_fee_rate`.

`TxEntry::fee_rate` at `tx-pool/src/component/entry.rs:115–117` uses the weight formula for scoring and eviction, confirming the divergence from the admission check.

## Impact Explanation

**Concrete bypass with defaults (`min_fee_rate = 1000 shannons/KW`, `max_tx_verify_cycles = 70,000,000`):**

| Parameter | Value |
|---|---|
| Serialized size | 200 bytes |
| Cycles | 70,000,000 |
| Weight | max(200, 11,940) = **11,940** |
| Admission min_fee (size-only) | 200 shannons |
| Fee paid | 200 shannons (passes) |
| Actual fee rate | ~16 shannons/KW (~60× below floor) |

Each such transaction causes every receiving node to execute 70M cycles of script verification, then relay the transaction to all its peers, who repeat the same process. The attacker pays only 200 shannons per transaction while imposing O(peers × 70M cycles) CPU cost across the network. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

## Likelihood Explanation

The exploit requires only an unprivileged `send_transaction` RPC call or a P2P relay message. The attacker must control a RISC-V lock script that executes a tight loop — straightforward with any compute-heavy script. No privileged access, key material, or majority hashpower is required. The attack is repeatable with many transactions, each costing only the size-based minimum fee.

## Recommendation

Add a post-execution fee-rate gate in `_process_tx` (`tx-pool/src/process.rs`) after `verify_rtx` returns actual cycles, before constructing `TxEntry` and calling `submit_entry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let actual_fee_rate = FeeRate::calculate(fee, weight);
if actual_fee_rate < tx_pool_config.min_fee_rate {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Alternatively, move the entire `min_fee_rate` enforcement into `submit_entry` using the weight-based formula after cycles are known. The existing size-only check in `check_tx_fee` can remain as a cheap pre-filter but must not be the sole gate.

## Proof of Concept

1. Deploy a RISC-V lock script that executes a tight loop consuming ~70,000,000 cycles but compiles to a small binary (~200 serialized bytes in the transaction).
2. Construct a transaction spending a cell locked by that script, with `fee = 200 shannons` (matching the size-only minimum).
3. Submit via `send_transaction` RPC.
4. **Expected (broken):** `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`; fee equals `min_fee`; transaction is admitted.
5. **Actual:** After script execution, `cycles ≈ 70,000,000`, `weight = 11,940`, `actual_fee_rate ≈ 16 shannons/KW` — ~60× below `min_fee_rate`.
6. The transaction is in the pool and relayed to all peers, each of which admits it via the same flawed check after executing 70M cycles.
