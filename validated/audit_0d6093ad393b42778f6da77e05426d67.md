### Title
`getPricesNoOlderThan` Age Limit Not Enforced Per-Feed тАФ Stale Closed-Market Prices Bypass Staleness Guard - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pulse `Scheduler` contract's `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions check staleness against a single subscription-level timestamp (`priceLastUpdatedAt`), which is set to the **maximum** `publishTime` across all feeds in a batch. For subscriptions containing mixed active and closed-market feeds, individual closed-market feeds can have `publishTime` values days in the past while the staleness guard passes, because the guard only checks the most-recent feed's timestamp. The user-specified `age_seconds` limit is therefore not enforced at the individual feed level.

---

### Finding Description

In `_validateShouldUpdatePrices`, the contract computes `updateTimestamp` as the maximum `publishTime` across all feeds in the update batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [1](#0-0) 

This maximum is then stored as the subscription-level `priceLastUpdatedAt`:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

The design is intentional for closed markets тАФ the code comment explicitly states:

> "We don't want to reject update data if it contains a price from a market that closed a few days ago, since it will contain a timestamp from the last trading period." [3](#0-2) 

However, `getPricesNoOlderThan` then checks staleness only against this subscription-level maximum:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [4](#0-3) 

The same pattern applies to `getEmaPricesNoOlderThan`: [5](#0-4) 

**Concrete scenario:**

A subscription contains two feeds:
- Feed A (BTC/USD, active market): `publishTime = now`
- Feed B (WHEAT/USD, closed market): `publishTime = now - 3 days`

After a keeper push, `priceLastUpdatedAt = now` (the max). A reader calls `getPricesNoOlderThan(subscriptionId, [feedB_id], 60)`. The check `distance(now, now) = 0 тЙд 60` passes. Feed B's price тАФ 3 days stale тАФ is returned without any revert.

The `IScheduler` interface documents `getPricesNoOlderThan` as:

> "Returns the price that is no older than `age` seconds of the current time. Reverts if the price wasn't updated sufficiently recently." [6](#0-5) 

This guarantee is not upheld at the individual feed level.

---

### Impact Explanation

Protocols integrating Pulse via `getPricesNoOlderThan` rely on it as a safety guard to ensure price freshness before executing financial logic (liquidations, collateral valuation, trade settlement). When a subscription contains any active-market feed alongside closed-market feeds, the staleness guard is bypassed for the closed-market feeds. A protocol that requests only the closed-market feed's price with a tight age limit (e.g., 60 seconds) will receive a price that is days old without any revert, potentially leading to:

- Incorrect collateral valuations
- Missed or incorrect liquidations
- Incorrect trade execution at stale prices

The README explicitly recommends `getPricesNoOlderThan` as the safe consumption path:

> "Readers are recommended to use the SDK's functions `get(Ema)PricesNoOlderThan`, which wrap the contract's `get(Ema)PricesUnsafe` functions and validate that the price is recent." [7](#0-6) 

---

### Likelihood Explanation

- Mixed active/closed-market subscriptions are an explicitly supported use case (the code comment at line 299тАУ304 confirms this).
- Any whitelisted reader тАФ or any reader when `whitelistEnabled = false` тАФ can call `getPricesNoOlderThan`. No privileged role is required.
- The `SubscriptionParams` struct allows arbitrary combinations of price feed IDs, making mixed subscriptions common in practice.
- The vulnerability is triggered passively: no attacker action is needed beyond calling the documented API with a reasonable age limit. [8](#0-7) 

---

### Recommendation

Replace the subscription-level staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed check against each individual feed's `publishTime`:

```solidity
function getPricesNoOlderThan(...) {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

Alternatively, document clearly that `getPricesNoOlderThan` only guarantees that the subscription's most-recent feed was updated within `age_seconds`, and that individual closed-market feeds may be older. Callers should check `publishTime` on each returned price independently.

---

### Proof of Concept

1. Create a subscription with two feeds: an active-market feed (e.g., BTC/USD) and a closed-market feed (e.g., a commodity).
2. A keeper calls `updatePriceFeeds`. The active feed has `publishTime = T_now`, the closed feed has `publishTime = T_now - 3 days`. Both share the same Pythnet slot. `_validateShouldUpdatePrices` accepts the update because `updateTimestamp = max(T_now, T_now - 3 days) = T_now`, which is within `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`. `status.priceLastUpdatedAt = T_now`.
3. A reader calls `getPricesNoOlderThan(subscriptionId, [closedMarketFeedId], 60)`.
4. The check `distance(block.timestamp, T_now) = 0 тЙд 60` passes.
5. The closed-market feed price with `publishTime = T_now - 3 days` is returned without revert, violating the function's documented guarantee. [9](#0-8) [10](#0-9)

    function _validateShouldUpdatePrices(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams storage params,
        SchedulerStructs.SubscriptionStatus storage status,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal view returns (uint256) {
        // Use the most recent timestamp, as some asset markets may be closed.
        // Closed markets will have a publishTime from their last trading period.
        // Since we verify all updates share the same Pythnet slot, we still ensure
        // that all price feeds are synchronized from the same update cycle.
        uint256 updateTimestamp = 0;
        for (uint8 i = 0; i < priceFeeds.length; i++) {
            if (priceFeeds[i].price.publishTime > updateTimestamp) {
                updateTimestamp = priceFeeds[i].price.publishTime;
            }
        }

        // Calculate the minimum acceptable timestamp (clamped at 0)
        // The maximum acceptable timestamp is enforced by the parsePriceFeedUpdatesWithSlots call
        uint256 minAllowedTimestamp = (block.timestamp >
            PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            : 0;

        // Validate that the update timestamp is not too old
        if (updateTimestamp < minAllowedTimestamp) {
            revert SchedulerErrors.TimestampTooOld(
                updateTimestamp,
                block.timestamp
            );
        }

        // Reject updates if they're older than the latest stored ones
        if (
            status.priceLastUpdatedAt > 0 &&
            updateTimestamp <= status.priceLastUpdatedAt
        ) {
            revert SchedulerErrors.TimestampOlderThanLastUpdate(
                updateTimestamp,
                status.priceLastUpdatedAt
            );
        }

        // If updateOnHeartbeat is enabled and the heartbeat interval has passed, trigger update
        if (params.updateCriteria.updateOnHeartbeat) {
            uint256 lastUpdateTime = status.priceLastUpdatedAt;

            if (
                lastUpdateTime == 0 ||
                updateTimestamp >=
                lastUpdateTime + params.updateCriteria.heartbeatSeconds
            ) {
                return updateTimestamp;
            }
        }
