### Title
Missing Peer Ban and Rate Limiting for Unsolicited Filter Response Messages — (`sync/src/filter/mod.rs`)

### Summary

The `BlockFilter` protocol handler accepts unsolicited `BlockFilters`, `BlockFilterHashes`, and `BlockFilterCheckPoints` response messages from any connected peer, returns `Status::ignored()` (code 101), and never bans or rate-limits the sender. A single unprivileged peer can flood the node with large, well-formed filter response messages indefinitely, consuming receive bandwidth and async handler CPU with no application-layer defense.

---

### Finding Description

In `try_process`, the three response-direction message variants are handled by a single arm that logs a warning and returns `Status::ignored()`: [1](#0-0) 

`Status::ignored()` constructs a `Status` with `StatusCode::Ignored = 101`: [2](#0-1) 

The `should_ban()` predicate only fires for codes in the range `400..500`: [3](#0-2) 

Because `101` is outside that range, the `process()` caller never invokes `nc.ban_peer()`: [4](#0-3) 

Additionally, the `BlockFilter` struct carries **no rate limiter at all**: [5](#0-4) 

This contrasts directly with the `Relayer`, which has a `governor`-based rate limiter keyed by `(PeerIndex, message_type)` capped at 30 req/s: [6](#0-5) 

and applies it before dispatching any message: [7](#0-6) 

The `HolePunching` protocol similarly has its own rate limiter: [8](#0-7) 

The `BlockFilter` handler has neither guard.

---

### Impact Explanation

A `BlockFilters` response message is bounded at ~1.8 MB per message by the server-side sender: [9](#0-8) 

An attacker sending well-formed `BlockFilters` messages at line rate can therefore:

- Saturate the node's inbound network bandwidth (each message up to ~1.8 MB, no cap on rate).
- Force repeated async handler invocations (molecule deserialization + match dispatch) on the Tokio runtime with no backpressure.
- Do so indefinitely from a **single connection** because the peer is never banned and no rate limit is enforced.

---

### Likelihood Explanation

The attack requires only a standard TCP connection to the Filter protocol port and the ability to send valid molecule-encoded `BlockFilterMessage` frames. No PoW, no key material, no privileged role. The TODO comment in the source code explicitly acknowledges the missing ban: [10](#0-9) 

---

### Recommendation

1. **Return a ban-triggering status** instead of `Status::ignored()` for unsolicited response messages. Use a `4xx` `StatusCode` (e.g., `StatusCode::ProtocolMessageIsMalformed`) so the existing `process()` logic calls `nc.ban_peer()` automatically.
2. **Add a per-peer rate limiter** to `BlockFilter` (mirroring the `Relayer`'s `governor::RateLimiter<(PeerIndex, u32)>`) to cap message processing rate before the match dispatch.

---

### Proof of Concept

```
1. Connect to a CKB node with SupportProtocols::Filter.
2. In a tight loop, send molecule-encoded BlockFilterMessage{BlockFilters{...}} frames
   (valid encoding, arbitrary start_number, empty block_hashes/filters vectors).
3. Observe: peer is never disconnected or banned; node logs only a WARN per message.
4. Measure: inbound bandwidth and Tokio async task CPU rise proportionally to send rate,
   with no application-layer throttle applied.
```

    pub fn should_ban(&self) -> Option<Duration> {
        if !(400..500).contains(&(self.code as u16)) {
            return None;
        }
        if let Some(context) = &self.context {
            // TODO: it might be worthwhile to formalize all error texts
            // that won't be banned.
            if context.contains(ARGV_TOO_LONG_TEXT) {
                return None;
            }
        }
        match self.code {
            StatusCode::GetHeadersMissCommonAncestors => Some(SYNC_USELESS_BAN_TIME),
            _ => Some(BAD_MESSAGE_BAN_TIME),
        }
    }
