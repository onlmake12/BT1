### Title
Silent Refund Failure Permanently Locks Excess ETH in `PythAggregatorV3` - (File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol)

### Summary
`PythAggregatorV3.updateFeeds()` accepts ETH as payment, pays the Pyth fee, then attempts to refund the remaining balance to `msg.sender`. The `success` return value of the refund call is explicitly ignored. When `msg.sender` is a contract that lacks a `receive()` or `fallback()` function, the refund silently fails and all excess ETH is permanently locked in `PythAggregatorV3`, which itself has no withdrawal mechanism.

### Finding Description
`PythAggregatorV3` is a production SDK contract that wraps Pyth price feeds behind a Chainlink-compatible interface. Its `updateFeeds()` function is `public payable` and is designed to accept ETH, pay the exact Pyth update fee, and refund the remainder to the caller.

The refund is implemented as:

```solidity
(bool success, ) = payable(msg.sender).call{
    value: address(this).balance
}("");
```

The `success` variable is immediately discarded (the `// solhint-disable-next-line no-unused-vars` comment confirms this is intentional). If the low-level call fails — which happens whenever `msg.sender` is a contract without a `receive()` or `fallback()` function — the ETH remains in `PythAggregatorV3`. The contract has no owner, no `receive()` function, no `fallback()`, and no withdrawal function, so the locked ETH is irrecoverable. [1](#0-0) 

This is the direct analog to the Infrared finding: Infrared could not *receive* native currency; here, `PythAggregatorV3` cannot *deliver* native currency back to contract callers that lack a receiver, and the failure is silently swallowed.

### Impact Explanation
**Medium.** Any excess ETH sent to `updateFeeds()` by a contract caller that does not implement `receive()` or `fallback()` is permanently locked. `PythAggregatorV3` has no owner, no admin, and no ETH withdrawal path. The locked funds are unrecoverable. Protocols that integrate `PythAggregatorV3` and call `updateFeeds()` with a buffer (a common pattern to avoid fee-estimation races) are directly affected. [2](#0-1) 

### Likelihood Explanation
**Medium.** `PythAggregatorV3` is explicitly positioned as a drop-in Chainlink replacement for DeFi protocols. DeFi contracts routinely call oracle update functions with a small ETH buffer to absorb fee fluctuations. Many such caller contracts are not designed to receive ETH and therefore lack `receive()`. The pattern of ignoring the refund `success` flag means every such call silently loses the overpayment.

### Recommendation
Check the `success` return value and revert if the refund fails, or use a pull-based refund pattern:

```solidity
function updateFeeds(bytes[] calldata priceUpdateData) public payable {
    uint fee = pyth.getUpdateFee(priceUpdateData);
    pyth.updatePriceFeeds{value: fee}(priceUpdateData);

    uint256 refund = address(this).balance;
    if (refund > 0) {
        (bool success, ) = payable(msg.sender).call{value: refund}("");
        require(success, "ETH refund failed");
    }
}
```

Alternatively, add a `receive()` function so the contract can at least hold ETH without reverting on direct sends, and expose a separate `withdrawDust()` function callable by the original payer.

### Proof of Concept
1. Deploy `PythAggregatorV3` pointing at a live Pyth contract.
2. Deploy an attacker/integrator contract `CallerContract` that has **no** `receive()` or `fallback()` function.
3. From `CallerContract`, call `pythAggregatorV3.updateFeeds{value: 1 ether}(priceUpdateData)` where the actual Pyth fee is, say, 1 wei.
4. Observe: `updateFeeds` succeeds (no revert), the Pyth fee is paid, but the refund call to `CallerContract` silently fails because `CallerContract` cannot receive ETH.
5. Check `address(pythAggregatorV3).balance` — it holds `1 ether - 1 wei` with no way to recover it. [3](#0-2)
