### Title
Softfork Activation Threshold Computed with Rounding-Down Integer Division Allows Lock-In Below Intended Signal Ratio — (`spec/src/versionbits/mod.rs`)

---

### Summary

The `threshold_number` helper in CKB's versionbits softfork state machine uses integer division (which truncates toward zero) to compute the minimum number of signaling blocks required to transition a deployment from `Started` to `LockedIn`. Because the result is rounded **down**, the actual required signal ratio can be strictly less than the configured `numer/denom` threshold, allowing a softfork to lock in with fewer miner signals than the protocol intends.

---

### Finding Description

`threshold_number` is defined as:

```rust
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    length
        .checked_mul(threshold.numer())
        .and_then(|ret| ret.checked_div(threshold.denom()))
}
``` [1](#0-0) 

`checked_div` performs truncating (floor) integer division. The result is then used in the lock-in check:

```rust
let threshold_number = threshold_number(total, self.threshold())?;
if count >= threshold_number {
    next_state = ThresholdState::LockedIn;
``` [2](#0-1) 

`total` is the sum of block counts across `period` epochs:

```rust
for _ in 0..period {
    let current_epoch_length = current_epoch_ext.length();
    total += current_epoch_length;
``` [3](#0-2) 

Because CKB epoch lengths are variable (between `MIN_EPOCH_LENGTH = 300` and `MAX_EPOCH_LENGTH = 1800`), `total * numer` is frequently not divisible by `denom`, causing the floor to produce a threshold strictly below the intended ratio.

**Concrete example — testnet threshold `TESTNET_ACTIVATION_THRESHOLD = Ratio::new(3, 4)` (75%):** [4](#0-3) 

| `total` blocks | `threshold_number` (floor) | Actual required ratio | Shortfall |
|---|---|---|---|
| 5 | `5*3/4 = 3` | 3/5 = **60.0%** | −15.0 pp |
| 7 | `7*3/4 = 5` | 5/7 ≈ **71.4%** | −3.6 pp |
| 9 | `9*3/4 = 6` | 6/9 ≈ **66.7%** | −8.3 pp |
| 301 | `301*3/4 = 225` | 225/301 ≈ **74.75%** | −0.25 pp |

**Concrete example — mainnet threshold `LC_MAINNET_ACTIVATION_THRESHOLD = Ratio::new(8, 10)` (80%):** [5](#0-4) 

| `total` blocks | `threshold_number` (floor) | Actual required ratio | Shortfall |
|---|---|---|---|
| 3 | `3*8/10 = 2` | 2/3 ≈ **66.7%** | −13.3 pp |
| 7 | `7*8/10 = 5` | 5/7 ≈ **71.4%** | −8.6 pp |
| 11 | `11*8/10 = 8` | 8/11 ≈ **72.7%** | −7.3 pp |

The maximum possible shortfall per period is bounded by `(denom − 1) / total`, which is small for large epoch counts but can be significant for short periods or small epoch lengths.

---

### Impact Explanation

**Impact: Medium**

A softfork deployment (e.g., `LightClient`, `Testdummy`) can transition to `LockedIn` — and subsequently to `Active` — with a miner signaling ratio strictly below the configured threshold. Once `Active`, the softfork rules are enforced by all nodes running that version. If the true hashpower fraction supporting the softfork is below the intended threshold, the activation is premature and may cause a chain split between upgraded and non-upgraded nodes, or activate consensus rule changes without the intended level of miner consensus. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: Low**

- Requires that `total * numer` is not divisible by `denom` (common in practice due to variable epoch lengths).
- Requires a coordinated group of miners who want to activate a softfork with slightly below-threshold hashpower.
- The shortfall is small when epoch lengths are large (the typical case on mainnet), but can be material for short periods or small epoch lengths.
- No privileged access is required — any miner participating in block production can signal via the `version` field of their cellbase witness. [7](#0-6) 

---

### Recommendation

Round **up** the threshold number instead of truncating down. Replace the floor division with a ceiling division:

```rust
// Before (floor division):
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    length
        .checked_mul(threshold.numer())
        .and_then(|ret| ret.checked_div(threshold.denom()))
}

// After (ceiling division):
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    length
        .checked_mul(threshold.numer())
        .and_then(|ret| {
            let denom = threshold.denom();
            ret.checked_add(denom - 1)?.checked_div(denom)
        })
}
```

This ensures `count >= threshold_number` is only satisfied when the true signal ratio is at or above `numer/denom`. [1](#0-0) 

---

### Proof of Concept

1. Deploy a CKB devnet with `period = 1`, epoch length forced to `5` blocks, and `threshold = Ratio::new(3, 4)` (75%).
2. `threshold_number(5, 3/4) = 5 * 3 / 4 = 3` (floor of 3.75).
3. Mine 5 blocks in the epoch; have only 3 of them signal the softfork bit (60% signal rate).
4. Observe that `count (3) >= threshold_number (3)` evaluates to `true`.
5. The deployment transitions to `LockedIn` despite only 60% of blocks signaling — well below the intended 75%. [8](#0-7)
