### Title
Hardcoded Fixed-Size `trustedSigners` Array Prevents Signer Rotation Beyond Capacity — (`lazer/contracts/evm/src/PythLazer.sol` and Solana Lazer IDL)

### Summary
The Pyth Lazer contracts hardcode the maximum number of trusted signers to a fixed-size array: **100 slots** in the EVM contract (`PythLazer.sol`) and **5 slots** in the Solana Lazer contract (as reflected in the on-chain IDL). When all slots are occupied, the `updateTrustedSigner` function reverts with `"no space for new signer"`, making it impossible to add new signers or perform key rotations. This directly mirrors the Curve bug class: a hardcoded fixed count that silently limits protocol functionality beyond a certain scale.

---

### Finding Description

**EVM (`lazer/contracts/evm/src/PythLazer.sol`):**

The `trustedSigners` storage variable is declared as a fixed-size array of exactly 100 elements:

```solidity
TrustedSignerInfo[100] internal trustedSigners;
```

The `updateTrustedSigner` function iterates over this array to find an empty slot when adding a new signer. If all 100 slots are occupied (by active or stale-but-not-yet-removed signers), the function exhausts the loop and reverts:

```solidity
revert("no space for new signer");
```

**Solana Lazer Contract (IDL):**

The on-chain account layout for the Solana Lazer `Storage` account, as defined in the IDL, hardcodes the `trustedSigners` field as a fixed array of exactly **5** elements:

```json
{
  "name": "trustedSigners",
  "type": {
    "array": [{ "defined": "TrustedSignerInfo" }, 5]
  }
}
```

A `numTrustedSigners: u8` counter tracks how many are active, but the underlying account allocation is fixed at 5 entries. The `extraSpace: [u8; 100]` field was added as a forward-compatibility buffer, but it does not expand the `trustedSigners` array itself.

---

### Impact Explanation

**Impact: Low** — No user funds are at risk. However:

1. Once all slots are filled, the `topAuthority` cannot add new trusted signers or perform key rotations without first removing an existing one.
2. If a signer key is compromised and all slots are occupied, the protocol cannot atomically remove the compromised key and add a replacement — creating a window where either the compromised key remains active or no valid signer exists.
3. For the Solana contract, the 5-signer cap is extremely tight. Pyth Lazer's operational model (multiple redundant signers with rolling expiry) can easily exhaust this limit.
4. When the limit is hit, any Lazer price update signed by a new/rotated key will fail verification for all users, breaking Lazer price feed consumption until a contract upgrade or redeployment is performed.

---

### Likelihood Explanation

**Likelihood: High** — The Solana contract's cap of 5 is operationally very small for a production oracle system that rotates keys and maintains redundancy. The EVM cap of 100 is more generous but still finite and fixed. As Lazer scales and key rotation cadence increases, hitting the limit is a realistic operational scenario. The `extraSpace` field in the Solana IDL itself signals that the developers anticipated needing to expand the struct, confirming this is a known growth concern.

---

### Recommendation

1. **EVM (`PythLazer.sol`):** Replace `TrustedSignerInfo[100]` with a dynamic array (`TrustedSignerInfo[] internal trustedSigners`) and a corresponding mapping for O(1) lookup. The `trustedSignerToExpiresAtMapping` already exists for fast validity checks; the fixed array is only used for enumeration and slot management.

2. **Solana Lazer Contract:** Redesign the `Storage` account to use a dynamic list (e.g., a separate PDA per signer, or a resizable account via `realloc`) rather than a fixed-size array of 5. The `extraSpace` buffer can be repurposed to store additional signer entries as an interim measure, but a proper dynamic structure is the correct long-term fix.

3. Add an explicit on-chain check that emits a warning event (or enforces a minimum free-slot invariant) before the array is completely full, to give operators advance notice.

---

### Proof of Concept

**EVM — Trigger "no space for new signer":**

1. Deploy `PythLazer.sol`.
2. Call `updateTrustedSigner(addr_i, expiresAt)` 100 times with 100 distinct non-zero addresses.
3. Call `updateTrustedSigner(addr_101, expiresAt)` — transaction reverts with `"no space for new signer"`.
4. The owner is now unable to add any new trusted signer. Any Lazer update signed by `addr_101` will fail `isValidSigner()` and revert in `verifyUpdate()`.

**Solana — Fixed 5-slot cap:**

The IDL account layout allocates exactly 5 `TrustedSignerInfo` entries. After 5 `update` instructions are executed by `topAuthority`, the 6th call will fail due to the fixed account size. A new signer's updates will be rejected by `verifyMessage`, breaking Lazer price feed verification for all downstream consumers until a contract upgrade is deployed.
