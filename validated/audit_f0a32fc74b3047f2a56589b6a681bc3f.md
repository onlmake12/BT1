Audit Report

## Title
Stale `total_tx_size`/`total_tx_cycles` Counter After Ancestor Eviction Inflates Pool Accounting — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the pool's `total_tx_size` and `total_tx_cycles` counters are snapshotted into local variables before `check_and_record_ancestors` runs. When that function evicts transactions via `remove_entry_and_descendants`, it correctly decrements `self.total_tx_size` in place via `update_stat_for_remove_tx`. However, `add_entry` then unconditionally overwrites `self.total_tx_size` with the pre-eviction snapshot, permanently inflating the counter by the total size of all evicted transactions. The inflated counter causes `limit_size` to evict legitimate pending transactions that should remain in the pool.

## Finding Description
`add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

- **Lines 210–211**: `updated_stat_for_add_tx` is a `&self` method (confirmed at line 711 — signature `fn updated_stat_for_add_tx(&self, ...)`) that reads `self.total_tx_size` and returns `self.total_tx_size + entry.size` into a local variable. `self.total_tx_size` is **not modified**. [1](#0-0) 

- **Line 213**: `check_and_record_ancestors` enters the eviction branch at line 603 when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count`. It calls `remove_entry_and_descendants` (line 618) for each evicted tx, which internally calls `update_stat_for_remove_tx` (lines 733–758), correctly decrementing `self.total_tx_size` and `self.total_tx_cycles` in place. [2](#0-1) [3](#0-2) 

- **Lines 218–219**: The stale pre-eviction local variables are unconditionally written back to `self.total_tx_size` and `self.total_tx_cycles`, overwriting the correctly-decremented values produced by the eviction loop. [4](#0-3) 

Net effect: `self.total_tx_size` ends up as `(initial + entry.size)` instead of `(initial + entry.size - sum_of_evicted_sizes)`. The counter is inflated by exactly the total size of all evicted transactions.

`limit_size` in `pool.rs` loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting legitimate pending transactions with `Reject::Full` until the inflated counter drops below the limit. [5](#0-4) 

## Impact Explanation
An unprivileged submitter can permanently inflate `total_tx_size` by triggering the ancestor-eviction path. After inflation, every subsequent call to `limit_size` (which runs after every new submission) evicts legitimate pending transactions with `Reject::Full`. The attack is repeatable: each trigger inflates the counter further. This constitutes a targeted, low-cost denial-of-service against the transaction pool of any node, causing honest users' transactions to be silently dropped even when the pool has ample real capacity. This matches the **High** impact: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation
The eviction path is reachable by any unprivileged submitter with no special keys or hash power. The existing integration test `TxPoolLimitAncestorCount` in `test/src/specs/tx_pool/limit.rs` (lines 70–116) documents and exercises exactly this scenario: submit 2000 cell-dep-referencing transactions, then submit one transaction that consumes the cell dep as an input, triggering eviction of 1002 cell-ref transactions. The attack requires only standard RPC access and a small amount of CKB to cover transaction fees. It is repeatable indefinitely.

## Recommendation
Move the counter snapshot **after** all evictions complete, so it reflects the post-eviction state of `self.total_tx_size`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// Compute totals AFTER evictions have already updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, subtract the total size/cycles of all entries in `evicts` from the pre-computed `total_tx_size`/`total_tx_cycles` before the final assignment.

## Proof of Concept
The existing integration test `TxPoolLimitAncestorCount` (`test/src/specs/tx_pool/limit.rs`, lines 70–116) already exercises the exact trigger path. To confirm the accounting bug:

1. Record `pool_map.total_tx_size` before submitting the consuming transaction.
2. After submission succeeds, assert `pool_map.total_tx_size == (pre_value + last_tx.size - sum_of_evicted_sizes)`.
3. Without the fix, the assertion fails: `total_tx_size` equals `pre_value + last_tx.size`, ignoring the 1002 evicted transactions' sizes.
4. Confirm that `limit_size` subsequently evicts additional legitimate transactions that were not part of the eviction set.

                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
