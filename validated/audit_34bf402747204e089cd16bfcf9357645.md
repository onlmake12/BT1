Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Stale Overwrite After Ancestor-Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, the new totals are computed via `updated_stat_for_add_tx` before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts `cell_ref_parent` entries, it correctly decrements `self.total_tx_size`/`self.total_tx_cycles` via `update_stat_for_remove_tx`. The stale pre-eviction snapshot is then unconditionally written back at lines 218–219, permanently inflating both accumulators by the size/cycles of every evicted entry. Repeated triggering causes `limit_size` to evict legitimate pool transactions even when the real pool byte count is within limits.

## Finding Description

In `add_entry` (lines 200–221 of `tx-pool/src/component/pool_map.rs`):

- **Lines 210–211**: `updated_stat_for_add_tx` snapshots `self.total_tx_size + entry.size` and `self.total_tx_cycles + entry.cycles` into locals before any eviction occurs. [1](#0-0) 

- **Line 213**: `check_and_record_ancestors` is called. When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (lines 603–625), it calls `remove_entry_and_descendants` in a loop for each `cell_ref_parent` candidate. [2](#0-1) 

- `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in-place. [3](#0-2) [4](#0-3) 

- **Lines 218–219**: The stale pre-eviction snapshot is unconditionally written back, overwriting the decrements made by `update_stat_for_remove_tx`. [5](#0-4) 

Net inflation per trigger: `sum(size of evicted cell_ref_parent entries)`. The inflation is permanent — `recompute_total_stat` is only invoked on underflow, never on this overwrite path.

`limit_size` drives eviction purely from the inflated accumulator: [6](#0-5) 

Once the phantom `total_tx_size` exceeds `max_tx_pool_size`, `limit_size` continuously evicts real pending/proposed transactions even though the actual pool byte count is within limits.

## Impact Explanation

Repeated triggering inflates `total_tx_size` without bound. Once the phantom total crosses `max_tx_pool_size`, `limit_size` evicts all real transactions from the pool and prevents re-admission of honest users' transactions. This constitutes a tx-pool denial-of-service against a targeted CKB node, matching the allowed **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

**Severity: High (10001–15000 points)**

## Likelihood Explanation

The trigger requires:
1. A chain of ≥ `max_ancestors_count − 1` (default: 24) transactions in the pool.
2. At least one in-pool transaction using a chain output as a **cell dep** (`cell_ref_parent`).
3. A new transaction consuming that same output as an **input**, pushing `ancestors_count` over the limit.

All three conditions are constructible with normal CKB transactions via the unprivileged `send_transaction` RPC. The attacker needs CKB to fund ~26 transactions per cycle; the attack is repeatable with fresh outputs. No privileged access, no victim mistakes, and no external dependencies are required.

**Likelihood: Medium**

## Recommendation

Move the stat computation to after `check_and_record_ancestors` completes, so it bases the addition on the post-eviction `self.total_tx_size`:

```rust
// In add_entry:
evicts = self.check_and_record_ancestors(&mut entry)?;  // evictions happen first
// Now self.total_tx_size reflects evictions; compute the addition on the current value:
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, add a debug-build invariant assertion after every `add_entry` that `total_tx_size == entries.iter().map(|e| e.inner.size).sum()`.

## Proof of Concept

**Configuration:** `max_ancestors_count = 25`, `max_tx_pool_size = 20 MB`

**Steps:**
1. Submit `tx_0` creating `cell_0`.
2. Submit `tx_1 … tx_24`, each spending the previous output (chain of 24 descendants of `tx_0`).
3. Submit `tx_dep` using `tx_24`'s output as a **cell dep** — it becomes a `cell_ref_parent` for any future tx spending that output.
4. Submit `tx_25` spending `tx_24`'s output as an **input**:
   - `get_tx_ancenstors` returns 25 ancestors + `tx_dep` as `cell_ref_parent` → `ancestors_count = 26 > 25`.
   - `check_and_record_ancestors` evicts `tx_dep` via `remove_entry_and_descendants` → `self.total_tx_size -= size(tx_dep)`.
   - `add_entry` writes back `total_tx_size = old_total + size(tx_25)` (stale snapshot, ignoring the subtraction).
   - **Net inflation: `+size(tx_dep)`**.
5. Repeat steps 3–4 with fresh outputs. Each iteration inflates `total_tx_size` by `size(tx_dep)`.
6. Once the phantom `total_tx_size` exceeds `max_tx_pool_size`, `limit_size` evicts all real transactions from the pool, even though the actual pool byte count is well within limits.
