### Title
`Indexer::prune` Skips Header and TxHash Pruning When No ConsumedOutPoint Entries Exist — (`util/indexer/src/indexer.rs`)

### Summary

`Indexer::prune` uses `min_block_number` (initialized to `BlockNumber::MAX`) as the seek key for the Header pruning iterator. This value is only updated inside the `ConsumedOutPoint` deletion loop. When no `ConsumedOutPoint` entries exist in the prunable range — which is the normal state for any chain segment containing only coinbase transactions — `min_block_number` stays at `BlockNumber::MAX`, the Header iterator seeks past all stored entries, and neither `Header` nor `TxHash` records are ever deleted.

### Finding Description

The root cause is in `Indexer::prune`:

```
let mut min_block_number = BlockNumber::MAX;          // line 765
for (block_number, key) in iter
    ...
    .take_while(|(block_number, _key)| prune_to_block.gt(block_number))
{
    if min_block_number == BlockNumber::MAX {
        min_block_number = block_number;              // line 778 — only set if loop runs
    }
    batch.delete(key)?;
}

// Header pruning
let mut key_prefix_header = vec![KeyPrefix::Header as u8];
key_prefix_header.extend_from_slice(&min_block_number.to_be_bytes()); // line 785
``` [1](#0-0) 

When the `ConsumedOutPoint` loop body never executes, `min_block_number` remains `u64::MAX`. The seek key built at line 785 becomes `[0xE0, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF]`. Since all stored `Header` keys encode real block numbers far below `u64::MAX`, the forward iterator finds nothing satisfying `block_number <= prune_to_block`, so the entire second loop is a no-op. [2](#0-1) 

`ConsumedOutPoint` entries are only written when a non-cellbase transaction spends a live cell:

```rust
// skip cellbase
if tx_index > 0 {
    ...
    batch.put_kv(Key::ConsumedOutPoint(block_number, &out_point), stored_live_cell)?;
``` [3](#0-2) 

Blocks containing only a coinbase transaction produce zero `ConsumedOutPoint` entries. This is valid, consensus-legal block content.

### Impact Explanation

Every appended block writes a `Header` entry (prefix `0xE0`) and one `TxHash` entry (prefix `0xC0`) per matched transaction. With pruning permanently broken for coinbase-only windows, these entries accumulate without bound. Over a long-running chain with low non-coinbase activity, the indexer's RocksDB store grows monotonically, degrading scan performance and eventually exhausting disk space.

### Likelihood Explanation

The trigger requires only that the prunable window (`tip - keep_num - 1` blocks) contains no non-cellbase spends. This is:

- **Normal at chain start** — early blocks frequently have only coinbase transactions.
- **Achievable by any miner** — a miner producing valid blocks with no user transactions (empty mempool, or deliberate exclusion) satisfies the condition without any special privilege or PoW majority.
- **Persistent** — once the prunable window is coinbase-only, every subsequent `prune()` call repeats the same no-op, so the backlog grows with each new block.

The existing `prune_bound` test (lines 1534–1600) exercises exactly this scenario — 21 coinbase-only blocks, `keep_num = 10` — but only asserts that rollback still works. It never asserts that old `Header` entries were deleted, so the bug is untested and undetected. [4](#0-3) 

### Recommendation

Replace the `min_block_number`-based seek with an unconditional start from block `0` for the Header pruning pass. The `ConsumedOutPoint` loop already limits deletions to `block_number < prune_to_block`; the Header loop has its own identical guard (`<= prune_to_block`) and does not need to be gated on whether any `ConsumedOutPoint` entries existed:

```rust
// Always start Header pruning from block 0, not from min_block_number
let mut key_prefix_header = vec![KeyPrefix::Header as u8];
key_prefix_header.extend_from_slice(&0u64.to_be_bytes());
```

```rust
#[test]
fn prune_coinbase_only_headers_are_deleted() {
    let indexer = new_indexer::<RocksdbStore>("prune_coinbase_only");
    let lock_script = ScriptBuilder::default()
        .code_hash(H256(rand::random()))
        .hash_type(ScriptHashType::Data)
        .build();

    // Append 20 coinbase-only blocks (no non-cellbase inputs)
    let mut pre_block = {
        let cb = TransactionBuilder::default()
            .input(CellInput::new_cellbase_input(0))
            .output(CellOutputBuilder::default()
                .capacity(capacity_bytes!(1000))
                .lock(lock_script.clone()).build())
            .output_data(Bytes::default()).build();
        let b = BlockBuilder::default()
            .transaction(cb)
            .header(HeaderBuilder::default().number(0).build())
            .build();
        indexer.append(&b).unwrap();
        b
    };
    for i in 1u64..=20 {
        let cb = TransactionBuilder::default()
            .input(CellInput::new_cellbase_input(i))
            .output(CellOutputBuilder::default()
                .capacity(capacity_bytes!(1000))
                .lock(lock_script.clone()).build())
            .output_data(Bytes::default()).build();
        pre_block = BlockBuilder::default()
            .transaction(cb)
            .header(HeaderBuilder::default()
                .number(i).parent_hash(pre_block.hash()).build())
            .build();
        indexer.append(&pre_block).unwrap();
    }

    // keep_num=10, tip=20 → prune_to_block=9
    // Header entries for blocks 0..=9 must be gone after prune
    indexer.prune().unwrap();

    for block_num in 0u64..=9 {
        let mut key = vec![KeyPrefix::Header as u8];
        key.extend_from_slice(&block_num.to_be_bytes());
        let found = indexer.store.iter(&key, IteratorDirection::Forward)
            .unwrap()
            .next()
            .map(|(k, _)| k.starts_with(&key))
            .unwrap_or(false);
        // FAILS: headers are never pruned because min_block_number stays at MAX
        assert!(!found, "Header for block {} was not pruned", block_num);
    }
}
