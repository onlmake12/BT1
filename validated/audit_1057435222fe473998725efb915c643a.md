Audit Report

## Title
Epoch-unaware `version_check` in `HeaderAcceptor` permanently marks `version=1` headers `BLOCK_INVALID` while the chain/relay path applies no version check, enabling a deterministic consensus split — (`sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::version_check` unconditionally rejects any header with `version != 0` and permanently marks it `BLOCK_INVALID`, with no epoch guard and no consultation of the hardfork switch. The chain verification path (`HeaderVerifier`, `BlockVerifier`, `ContextualBlockVerifier`) performs no version check at all. A single `version=1` block with valid PoW will be permanently accepted by nodes that receive it via the compact-block/`SendBlock` relay path and permanently rejected by nodes that first see it via `SendHeaders`, producing an irrecoverable consensus split.

## Finding Description
`HeaderAcceptor::version_check` at `sync/src/synchronizer/headers_process.rs:286–293` rejects any header whose `version() != 0` unconditionally:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    if self.header.version() != 0 {
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
``` [1](#0-0) 

On failure, `accept()` at line 346–354 calls `shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID)`, permanently poisoning that block hash on the receiving node: [2](#0-1) 

The `HeaderVerifier::verify` used in the compact-block relay path (`sync/src/relayer/compact_block_process.rs:324`) checks only PoW, parent existence, block number, epoch, and timestamp — no version field: [3](#0-2) 

`BlockVerifier::verify` in the chain service's `non_contextual_verify` path runs only `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier` — again no version check: [4](#0-3) 

`ContextualBlockVerifier::verify` similarly has no version check: [5](#0-4) 

`is_remove_header_version_reservation_rule_enabled` is defined via macro in `util/types/src/core/hardfork/ckb2023.rs` but a repository-wide grep confirms it is referenced **only** in that definition — it is never called in any production verifier: [6](#0-5) 

The divergence exists at **all** epochs: the sync path always rejects `version != 0`; the chain/relay path never checks version at all.

## Impact Explanation
Two nodes with identical chain state will permanently diverge if one receives a `version=1` block via `SendHeaders` and the other receives it via compact-block or `SendBlock` relay. The `BLOCK_INVALID` flag written at line 352 is permanent; the affected node refuses the block and all descendants forever, even if the full block is later delivered via `SendBlock`. This is a **Critical — consensus deviation** impact (15001–25000 points): a single crafted block can split the network into two irreconcilable views of the canonical chain. [2](#0-1) 

## Likelihood Explanation
The attacker must mine exactly one block with `header.version = 1` and valid Eaglesong PoW. No majority hashpower is required — any miner or pool participant can produce such a block. The attacker then propagates the header to a subset of peers via `SendHeaders` and the full block to others via the normal relay path. No privileged access, key material, or social engineering is required. The attack is repeatable and deterministic at any epoch. [1](#0-0) 

## Recommendation
`HeaderAcceptor::version_check` must be made epoch-aware by consulting `consensus.hardfork_switch.ckb2023.is_remove_header_version_reservation_rule_enabled(header.epoch().number())` before rejecting. Pre-activation it should reject `version != 0`; post-activation it should permit it (or be removed). Ideally, the version check should be unified into `HeaderVerifier::verify` (which already holds the `Consensus` reference) so both the sync and chain paths share a single authoritative rule and future divergence is structurally prevented. [7](#0-6) 

## Proof of Concept
1. Run two CKB nodes (A and B) on any chain (the bug is epoch-independent).
2. Mine a block with `header.version = 1` and valid Eaglesong PoW.
3. Send the full block to node A via the normal compact-block / `SendBlock` relay path.
4. Send only the header to node B via `SendHeaders`.
5. Observe: node A accepts the block (no version check in `HeaderVerifier` or `BlockVerifier`) and advances its tip; node B hits `version_check` at `headers_process.rs:287`, marks the block hash `BLOCK_INVALID` at line 352, and never advances past it.
6. Confirm node B's `BLOCK_INVALID` flag is permanent — subsequently sending the full block via `SendBlock` will be rejected at the `BLOCK_INVALID` status check in `compact_block_process.rs:259`. [8](#0-7)

    pub fn verify(
        &'a self,
        resolved: &'a [Arc<ResolvedTransaction>],
        block: &'a BlockView,
    ) -> Result<(Cycle, Vec<Completed>), Error> {
        let parent_hash = block.data().header().raw().parent_hash();
        let header = block.header();
        let parent = self
            .context
            .store
            .get_block_header(&parent_hash)
            .ok_or_else(|| UnknownParentError {
                parent_hash: parent_hash.clone(),
            })?;

        let epoch_ext = if block.is_genesis() {
            self.context.consensus.genesis_epoch_ext().to_owned()
        } else {
            self.context
                .consensus
                .next_epoch_ext(&parent, &self.context.store.borrow_as_data_loader())
                .ok_or_else(|| UnknownParentError {
                    parent_hash: parent.hash(),
                })?
                .epoch()
        };

        if !self.switch.disable_epoch() {
            EpochVerifier::new(&epoch_ext, block).verify()?;
        }

        if !self.switch.disable_uncles() {
            let uncle_verifier_context = UncleVerifierContext::new(&self.context, &epoch_ext);
            UnclesVerifier::new(uncle_verifier_context, block).verify()?;
        }

        if !self.switch.disable_two_phase_commit() {
            TwoPhaseCommitVerifier::new(&self.context, block).verify()?;
        }

        if !self.switch.disable_daoheader() {
            DaoHeaderVerifier::new(&self.context, resolved, &parent, &block.header()).verify()?;
        }

        if !self.switch.disable_reward() {
            RewardVerifier::new(&self.context, resolved, &parent).verify()?;
        }

        if !self.switch.disable_extension() {
            BlockExtensionVerifier::new(&self.context, self.chain_root_mmr, &parent)

        }

        let ret = BlockTxsVerifier::new(
            self.context.clone(),
            header,
            self.handle,
            &self.txs_verify_cache,
            &parent,
        )
        .verify(resolved, self.switch.disable_script())?;
        Ok(ret)
    }
