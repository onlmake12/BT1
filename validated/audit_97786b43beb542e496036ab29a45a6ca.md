Audit Report

## Title
Copy-paste error in `script_exists_in_output` reads `row_lock` instead of `row_type` in final branch, silently deleting referenced type scripts on PostgreSQL during rollback — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary
In `script_exists_in_output`, the final return branch at line 252 mistakenly calls `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, `row_lock` decodes successfully as `BOOLEAN` and returns `false` (since the early-return guard already confirmed it is false), so `row_type` is never consulted. The function always returns `false` for any script referenced only as a `type_script_id`, causing `rollback_block` to unconditionally enqueue those scripts for deletion even when surviving outputs still reference them.

## Finding Description
`script_exists_in_output` fetches two rows: `row_lock` (EXISTS for `lock_script_id`) and `row_type` (EXISTS for `type_script_id`). The early-return path (lines 222–235) is correct: if `row_lock` is true, return `true` immediately. After that guard, `row_lock` is known to be `false`. The function then fetches `row_type` (lines 237–249) but at line 252 reads `row_lock` again:

```rust
// line 252 — BUG: should be row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),          // PostgreSQL: succeeds, returns false (row_lock is already false)
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),  // SQLite: correct path
}
``` [1](#0-0) 

On PostgreSQL, `try_get::<bool, _>(0)` succeeds and returns the already-known-false value of `row_lock`, so `row_type` is never read. On SQLite, `try_get::<bool, _>(0)` fails (SQLite stores `EXISTS` as `BIGINT`), so the `Err` arm correctly reads `row_type`.

In `rollback_block`, outputs are deleted first (line 25), then `script_exists_in_output` is called to decide whether to delete each script (lines 30–37). [2](#0-1) 

Because the function always returns `false` for type-script-only scripts on PostgreSQL, any script that appears as `type_script_id` in the rolled-back block's outputs and is also referenced by surviving outputs in earlier blocks will be incorrectly added to `script_id_list_to_remove` and deleted. The surviving outputs retain a foreign-key reference to the now-deleted script row, permanently corrupting the indexer state until a full rebuild.

## Impact Explanation
This is a confirmed incorrect implementation of the CKB rich-indexer state storage mechanism. The bug permanently corrupts the `script` table on PostgreSQL after any chain reorg involving shared type scripts, causing all indexer RPC queries that join on `script` (e.g., `get_cells`, `get_transactions` filtered by type script) to return empty or incorrect results for affected cells. This matches the allowed impact: **Medium (2001–10000 points) — Suboptimal/incorrect implementation of CKB state storage mechanism.**

## Likelihood Explanation
Three conditions are required, all of which are routine in production:
1. The node operator uses PostgreSQL (a supported, documented deployment option).
2. A chain reorg occurs (rollback of at least one block) — a normal network event requiring no attacker capability.
3. The rolled-back block contains an output whose `type_script_id` is also referenced by an output in an earlier surviving block — common for any token/UDT type script used across multiple blocks.

No privileged access or attacker action is needed; the bug is triggered by normal CKB P2P network operation.

## Recommendation
Fix line 252 to use `row_type` instead of `row_lock`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // was: row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Additionally, deduplicate `script_id_list_to_remove` before calling `remove_batch_by_blobs` to avoid unnecessarily large SQL `IN` clauses when multiple outputs share a script.

## Proof of Concept
1. Start a PostgreSQL-backed rich indexer.
2. Index a block containing outputs O1 and O2, both with `type_script_id = S` (distinct lock scripts).
3. Index a second block containing output O3 also with `type_script_id = S`.
4. Trigger a rollback of block 2 (e.g., via a simulated reorg).
5. Query `SELECT id FROM script WHERE id = S`:
   - **SQLite**: script S still exists (correct — O1 and O2 still reference it).
   - **PostgreSQL**: script S is deleted (bug).
6. Call the indexer RPC `get_cells` filtered by type script S — returns empty on PostgreSQL despite O1 and O2 existing in block 1.
