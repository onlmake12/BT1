### Title
Unprivileged Front-Run of `Initialize` Permanently Locks Config Authority - (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary
The `initialize()` instruction has no deployer-only access control. Any signer can call it before the legitimate deployer, set an attacker-controlled `authority` in the config PDA, and permanently prevent any legitimate `InitializePublisher` from succeeding.

### Finding Description

`validate_payer()` only checks that the first account is a signer and writable — it does not verify the caller is the program upgrade authority or any specific key. [1](#0-0) 

The `authority` field written into the config PDA is taken directly from caller-supplied `InitializeArgs`, with no constraint on its value. [2](#0-1) 

The config PDA is derived from `CONFIG_SEED` alone, making it fully deterministic and predictable. [3](#0-2) 

Once the PDA is created, `accounts::config::create()` rejects any subsequent call with `AlreadyInitialized` because `data.format != 0`. [4](#0-3) 

### Impact Explanation

After the attacker's `Initialize` succeeds, `validate_authority()` in `initialize_publisher` enforces that the signer's key matches `config.authority`. [5](#0-4) 

Since the stored authority is the attacker's key, no legitimate operator can ever call `InitializePublisher`. There is no `UpdateAuthority` or re-initialization instruction in the program, making this permanent. The entire price-store program becomes inoperable for Pyth publishers.

### Likelihood Explanation

On Solana, "front-running" does not require a traditional mempool — any unprivileged account can simply call `Initialize` at any point between program deployment and the deployer's own initialization transaction. The attacker needs no special access, no leaked keys, and no privileged role. The cost is a single transaction fee.

### Recommendation

Restrict `initialize()` to the program's upgrade authority. Retrieve the upgrade authority from the program's `ProgramData` account (at `BPFLoaderUpgradeable`'s derived address) and assert that `payer.key` matches it before proceeding.

### Proof of Concept

1. Deploy the program.
2. Before the deployer calls `Initialize`, submit a transaction calling `Initialize` with `authority = attacker_pubkey`.
3. Config PDA is created with `config.authority = attacker_pubkey`.
4. Deployer's `Initialize` fails with `AlreadyInitialized`.
5. Call `InitializePublisher` with any legitimate authority — it fails `MissingRequiredSignature` because `authority.key != config.authority`.
6. The program is permanently locked.

This can be reproduced with `solana-program-test` by submitting two `Initialize` transactions with different authorities and asserting the second fails, then asserting `InitializePublisher` with the legitimate authority also fails. [6](#0-5)

pub fn validate_payer<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let payer = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, payer.is_signer);
    ensure!(ProgramError::InvalidArgument, payer.is_writable);
    Ok(payer)
}

pub fn initialize(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    args: &InitializeArgs,
) -> ProgramResult {
    let mut accounts = accounts.iter();
    let payer = validate_payer(accounts.next())?;
    let config = validate_config(accounts.next(), args.config_bump, program_id, true)?;
    let system = validate_system(accounts.next())?;

    let lamports = (Rent::get()?).minimum_balance(accounts::config::SIZE);

    invoke_signed(
        &system_instruction::create_account(
            payer.key,
            config.key,
            lamports,
            accounts::config::SIZE
                .try_into()
                .expect("unexpected overflow"),
            program_id,
        ),
        &[payer.clone(), config.clone(), system.clone()],
        &[&[CONFIG_SEED.as_bytes(), &[args.config_bump]]],
    )?;

    accounts::config::create(*config.data.borrow_mut(), args.authority)?;

    Ok(())
}
