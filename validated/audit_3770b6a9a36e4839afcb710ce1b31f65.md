### Title
Deviation Trigger Ignores Price Exponent, Producing Incorrect Basis-Point Calculation - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `_validateShouldUpdatePrices` function in `Scheduler.sol` computes price deviation by comparing raw `int64` mantissa values directly, without normalizing for the Pyth price exponent (`expo`). Because a Pyth price is defined as `actual_price = mantissa × 10^expo`, two mantissa values are only directly comparable when their exponents are identical. If the exponent differs between the stored previous price and the incoming current price, the deviation calculation produces a result that is off by a factor of `10^(expo_diff)`, exactly mirroring the decimal-precision mismatch class in the reference report.

---

### Finding Description

In `_validateShouldUpdatePrices`, the deviation check reads only the raw mantissa fields:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;   // mantissa only
int64 previousPrice = previousFeed.price.price;     // mantissa only

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
```

Neither `priceFeeds[i].price.expo` nor `previousFeed.price.expo` is ever read in this function (confirmed by grep: zero occurrences of `.expo` in `Scheduler.sol`). The Pyth price struct carries both fields:

```
actual_price = price × 10^expo
```

A correct deviation calculation must normalize both prices to a common exponent before subtracting. The current code implicitly assumes `currentExpo == previousExpo` at all times.

**Concrete example of incorrect behavior:**

| | Mantissa | Expo | Actual price |
|---|---|---|---|
| Previous | 100 | -2 | 1.00 |
| Current | 10000 | -4 | 1.0000 |

True deviation: **0 bps**

Code computes:
- `numerator = |10000 − 100| = 9900`
- `denominator = 100`
- `deviationBps = 9900 × 10000 / 100 = 990 000 bps (9900%)`

A threshold of, say, 100 bps would be triggered, causing a spurious update and draining the subscription's ETH balance.

The inverse case (exponent increases) suppresses a real deviation:

| | Mantissa | Expo | Actual price |
|---|---|---|---|
| Previous | 10000 | -4 | 1.0000 |
| Current | 200 | -2 | 2.00 |

True deviation: **100%** (10 000 bps)

Code computes:
- `numerator = |200 − 10000| = 9800`
- `denominator = 10000`
- `deviationBps = 9800 × 10000 / 10000 = 9800 bps`

This happens to be close to correct here, but for smaller actual deviations the error is severe.

---

### Impact Explanation

1. **Spurious updates / balance drain**: A keeper submits update data whose exponent has shifted downward (more negative). The mantissa is proportionally larger for the same real price. The deviation formula sees a huge mantissa difference and fires an update even though the real price is unchanged. The subscription's ETH balance is charged for the keeper fee and Pyth fee on every such spurious update, eventually exhausting the balance and deactivating the subscription.

2. **Missed legitimate updates**: A keeper submits update data whose exponent has shifted upward (less negative). The mantissa is proportionally smaller. A real price move that exceeds the configured threshold is not detected, so time-sensitive consumers (e.g., DeFi protocols relying on deviation-triggered freshness) receive stale prices.

Both outcomes are reachable by any unprivileged keeper calling `updatePriceFeeds()`.

---

### Likelihood Explanation

Pyth price feed exponents are **fixed per feed in practice** and have not changed on live feeds. However:

- The Pyth on-chain contract and the `parsePriceFeedUpdates` API impose **no invariant** that the exponent is constant across updates.
- A future Pyth network upgrade or a new feed configuration could change an exponent.
- A malicious or misconfigured publisher set could, in theory, cause an exponent shift.
- The code contains no assertion or check that `currentExpo == previousExpo`.

Likelihood is **low** under current Pyth network behavior, but the code is structurally incorrect and the risk is non-zero.

---

### Recommendation

Normalize both prices to a common exponent before computing the deviation. A safe approach:

```solidity
int32 currentExpo  = priceFeeds[i].price.expo;
int32 previousExpo = previousFeed.price.expo;

// Normalize to the more-negative (higher-precision) exponent
int32 targetExpo = currentExpo < previousExpo ? currentExpo : previousExpo;

// Scale each mantissa: multiply by 10^(ownExpo - targetExpo)  (always >= 0)
uint256 curNorm  = uint256(uint64(currentPrice))  * (10 ** uint32(int32(currentExpo)  - targetExpo));
uint256 prevNorm = uint256(uint64(previousPrice)) * (10 ** uint32(int32(previousExpo) - targetExpo));

uint256 numerator    = curNorm > prevNorm ? curNorm - prevNorm : prevNorm - curNorm;
uint256 deviationBps = Math.mulDiv(numerator, 10_000, prevNorm);
```

Alternatively, add an explicit revert if the exponents differ, to surface the assumption clearly.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` and create a subscription with `updateOnDeviation = true`, `deviationThresholdBps = 100` (1%).
2. Submit a first `updatePriceFeeds` call with `price.price = 100`, `price.expo = -2` (actual = 1.00). This sets the stored previous price.
3. Submit a second `updatePriceFeeds` call with `price.price = 10000`, `price.expo = -4` (actual = 1.0000 — identical real price, 0% deviation).
4. Observe that `_validateShouldUpdatePrices` computes `deviationBps = 990 000`, which exceeds the 100 bps threshold, and the update is accepted and the keeper is paid — despite zero real price movement.

    function _validateShouldUpdatePrices(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams storage params,
        SchedulerStructs.SubscriptionStatus storage status,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal view returns (uint256) {
        // Use the most recent timestamp, as some asset markets may be closed.
        // Closed markets will have a publishTime from their last trading period.
        // Since we verify all updates share the same Pythnet slot, we still ensure
        // that all price feeds are synchronized from the same update cycle.
        uint256 updateTimestamp = 0;
        for (uint8 i = 0; i < priceFeeds.length; i++) {
            if (priceFeeds[i].price.publishTime > updateTimestamp) {
                updateTimestamp = priceFeeds[i].price.publishTime;
            }
        }

        // Calculate the minimum acceptable timestamp (clamped at 0)
        // The maximum acceptable timestamp is enforced by the parsePriceFeedUpdatesWithSlots call
        uint256 minAllowedTimestamp = (block.timestamp >
            PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            : 0;

        // Validate that the update timestamp is not too old
        if (updateTimestamp < minAllowedTimestamp) {
            revert SchedulerErrors.TimestampTooOld(
                updateTimestamp,
                block.timestamp
            );
        }

        // Reject updates if they're older than the latest stored ones
        if (
            status.priceLastUpdatedAt > 0 &&
            updateTimestamp <= status.priceLastUpdatedAt
        ) {
            revert SchedulerErrors.TimestampOlderThanLastUpdate(
                updateTimestamp,
                status.priceLastUpdatedAt
            );
        }

        // If updateOnHeartbeat is enabled and the heartbeat interval has passed, trigger update
        if (params.updateCriteria.updateOnHeartbeat) {
            uint256 lastUpdateTime = status.priceLastUpdatedAt;

            if (
                lastUpdateTime == 0 ||
                updateTimestamp >=
                lastUpdateTime + params.updateCriteria.heartbeatSeconds
            ) {
                return updateTimestamp;
            }
        }

        // If updateOnDeviation is enabled, check if any price has deviated enough
        if (params.updateCriteria.updateOnDeviation) {
            for (uint8 i = 0; i < priceFeeds.length; i++) {
                // Get the previous price feed for this price ID using subscriptionId
                PythStructs.PriceFeed storage previousFeed = _state
                    .priceUpdates[subscriptionId][priceFeeds[i].id];

                // If there's no previous price, this is the first update
                if (previousFeed.id == bytes32(0)) {
                    return updateTimestamp;
                }

                // Calculate the deviation percentage
                int64 currentPrice = priceFeeds[i].price.price;
                int64 previousPrice = previousFeed.price.price;

                // Skip if either price is zero to avoid division by zero
                if (previousPrice == 0 || currentPrice == 0) {
                    continue;
                }

                // Calculate absolute deviation basis points (scaled by 1e4)
                uint256 numerator = SignedMath.abs(
                    currentPrice - previousPrice
                );
                uint256 denominator = SignedMath.abs(previousPrice);
                uint256 deviationBps = Math.mulDiv(
                    numerator,
                    10_000,
                    denominator
                );

                // If deviation exceeds threshold, trigger update
                if (
                    deviationBps >= params.updateCriteria.deviationThresholdBps
                ) {
                    return updateTimestamp;
                }
            }
        }

        revert SchedulerErrors.UpdateConditionsNotMet();
    }
