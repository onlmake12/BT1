### Title
`PythAggregatorV3.latestRoundData()` / `latestAnswer()` Return Stale Prices Without Staleness Check - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3.sol` is Pyth's official Chainlink `AggregatorV3Interface`-compatible adapter, explicitly recommended as the migration path for protocols moving from Chainlink to Pyth. Every price-reading function in this contract — `latestAnswer()`, `latestRoundData()`, `getRoundData()`, and `decimals()` — calls `pyth.getPriceUnsafe(priceId)` with no staleness check. A protocol that deploys this adapter and calls `latestRoundData()` will receive a price that may be arbitrarily old, with no revert and no signal that the data is stale.

---

### Finding Description

`PythAggregatorV3.sol` implements the Chainlink `AggregatorV3Interface` by wrapping a single Pyth price feed. All four price-reading functions delegate to `getPriceUnsafe`:

```solidity
// latestAnswer — no staleness check
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// latestRoundData — no staleness check
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

`getPriceUnsafe` is explicitly documented in `IPyth.sol` as returning "the most recent price update in this contract **without any recency checks**" and that "the returned price update may be arbitrarily far in the past." [1](#0-0) [2](#0-1) [3](#0-2) 

The `getPriceNoOlderThan` function exists precisely to enforce a staleness bound and reverts with `StalePrice` if the price is too old:

```solidity
function getPriceNoOlderThan(bytes32 id, uint age) public view virtual override
    returns (PythStructs.Price memory price) {
    price = getPriceUnsafe(id);
    if (diff(block.timestamp, price.publishTime) > age)
        revert PythErrors.StalePrice();
    return price;
}
``` [4](#0-3) 

`PythAggregatorV3` never calls this checked variant. The `updatedAt` field returned by `latestRoundData()` is set to `price.publishTime`, which may be hours or days old, and the function will not revert. [5](#0-4) 

---

### Impact Explanation

Protocols that use `PythAggregatorV3` as a drop-in Chainlink replacement — the documented migration path — will call `latestRoundData()` and receive a stale price with no revert. If the Pyth feed has not been updated recently (e.g., during network congestion, a Hermes outage, or a market closure), the adapter silently returns an outdated price. Downstream protocols using this price for liquidations, collateral valuation, or trade execution will operate on incorrect data, enabling:

- Under-liquidation of insolvent positions (stale high price protects a borrower who should be liquidated)
- Over-liquidation of solvent positions (stale low price triggers unwarranted liquidation)
- Arbitrage against the stale price at the expense of the protocol's liquidity providers [6](#0-5) 

---

### Likelihood Explanation

The migration guide explicitly instructs Chainlink users to deploy `PythAggregatorV3` as a replacement for their existing `AggregatorV3Interface` feeds. Protocols that follow this guide will inherit the missing staleness check. The Pyth pull-oracle model means the on-chain price is only as fresh as the last `updatePriceFeeds` call; if no keeper or user has updated the feed recently, the price returned by `latestRoundData()` can be arbitrarily stale. This is a realistic scenario during network congestion or when a protocol's update keeper fails. [7](#0-6) 

---

### Recommendation

Replace `getPriceUnsafe` with `getPriceNoOlderThan` in all price-reading functions, using a configurable `maxAge` parameter set at construction time:

```solidity
uint256 public maxAge; // e.g., 60 seconds

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    return int256(price.price);
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

This mirrors the pattern already used correctly in `AbstractPyth.sol` and matches the staleness-check guidance in the Pyth best-practices documentation. [4](#0-3) 

---

### Proof of Concept

1. A protocol deploys `PythAggregatorV3` for ETH/USD and integrates it as a Chainlink feed for collateral pricing.
2. The Pyth ETH/USD feed on-chain was last updated 2 hours ago (e.g., the keeper is down). The real ETH price has dropped 15%.
3. A user calls the protocol's liquidation function. The protocol calls `PythAggregatorV3.latestRoundData()`.
4. `latestRoundData()` calls `getPriceUnsafe`, which returns the 2-hour-old price with no revert.
5. The protocol sees a healthy collateral ratio (based on the stale high price) and does not liquidate the insolvent position.
6. The protocol accumulates bad debt.

No privileged access is required. Any transaction sender can trigger this path by interacting with the protocol while the feed is stale. [2](#0-1) [8](#0-7)
