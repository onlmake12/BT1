### Title
Deviation-Only Subscriptions Allow Unlimited `updatePriceFeeds` Calls Per Ethereum Block, Draining Subscription Balance — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol`'s `updatePriceFeeds` enforces no per-Ethereum-block rate limit. The only temporal guard uses the Pythnet publish timestamp (`updateTimestamp`), not `block.timestamp`. For deviation-only subscriptions, a keeper can bundle multiple calls in a single Ethereum block — each backed by a genuine Pythnet slot with a strictly greater timestamp and a price that meets the deviation threshold — draining the subscription balance at a rate far exceeding what the subscription owner intended.

---

### Finding Description

`_validateShouldUpdatePrices` enforces two conditions before accepting an update:

1. The Pythnet `updateTimestamp` must be strictly greater than `status.priceLastUpdatedAt`.
2. At least one update criterion (heartbeat or deviation) must be satisfied.

```solidity
// Reject updates if they're older than the latest stored ones
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(...);
}
``` [1](#0-0) 

The guard uses the **Pythnet publish time**, not `block.timestamp`. Pythnet produces a new slot approximately every 400 ms. In a single 12-second Ethereum block, roughly 30 Pythnet slots are produced. A keeper can collect all 30 genuine signed Pythnet updates and submit them in a single flashbot bundle, each with a strictly increasing Pythnet timestamp.

For a **deviation-only** subscription (`updateOnHeartbeat = false`, `updateOnDeviation = true`), the heartbeat path is skipped entirely:

```solidity
if (params.updateCriteria.updateOnHeartbeat) {
    ...
    if (lastUpdateTime == 0 ||
        updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds) {
        return updateTimestamp;
    }
}
// Falls through to deviation check
if (params.updateCriteria.updateOnDeviation) {
    ...
    if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
        return updateTimestamp;
    }
}
``` [2](#0-1) 

If the price oscillates across the deviation threshold across consecutive Pythnet slots (common during volatile markets), each of the 30 bundled calls passes validation. Each successful call pays the keeper:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
...
status.balanceInWei -= totalKeeperFee;
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [3](#0-2) 

`_validateSubscriptionParams` correctly rejects `heartbeatSeconds == 0` when `updateOnHeartbeat` is true, but imposes **no analogous minimum interval** for deviation-only subscriptions:

```solidity
if (
    params.updateCriteria.updateOnHeartbeat &&
    params.updateCriteria.heartbeatSeconds == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [4](#0-3) 

There is no equivalent guard preventing a deviation-only subscription from being updated arbitrarily many times within a single Ethereum block.

---

### Impact Explanation

Each `updatePriceFeeds` call deducts a Pyth protocol fee plus a keeper fee from `status.balanceInWei`. A keeper bundling N calls in one Ethereum block drains N × (gasCost + keeperFee) from the subscription in a single block. The subscription owner funded the contract expecting updates to occur at a rate bounded by market deviation events, not at the rate of Pythnet slot production (~30/block). The owner's balance is consumed orders of magnitude faster than anticipated, potentially exhausting the subscription balance before the owner can react.

Additionally, a manager wishing to withdraw excess funds faces a race: a keeper can frontrun the withdrawal transaction by draining the balance down to `minimumBalance` in the same block, leaving the manager unable to withdraw the surplus until they first deactivate the subscription. [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged address can call `updatePriceFeeds`. Pythnet price data is publicly available. During volatile market conditions (e.g., a flash crash or rapid price oscillation), a keeper can trivially collect multiple Pythnet updates within a single Ethereum block where consecutive prices differ by more than `deviationThresholdBps`. Flashbot bundles make atomic multi-call execution straightforward. The keeper is economically incentivized because each call pays them a fee. No privileged access, leaked key, or oracle manipulation is required.

---

### Recommendation

1. **Add a per-block rate limit**: Record `block.number` of the last update in `SubscriptionStatus` and revert if `block.number == lastUpdateBlock` for deviation-only subscriptions (or make this opt-in).
2. **Alternatively, enforce a minimum update interval**: Require a configurable `minimumUpdateIntervalSeconds > 0` even for deviation-only subscriptions, analogous to how `heartbeatSeconds == 0` is already rejected.
3. **Document the risk**: If multiple updates per block are intentional for deviation-only subscriptions, document the transaction-ordering risk for subscription managers and recommend they set conservative deviation thresholds.

---

### Proof of Concept

```
Block N (single Ethereum block, ~12 seconds):
  Pythnet slots available: T1, T2, ..., T30 (each ~400ms apart)
  Price at T1: 100.00, T2: 101.01 (+1.01%), T3: 99.99 (-1.01%), T4: 101.01 (+1.01%), ...

Keeper constructs flashbot bundle:
  tx1: updatePriceFeeds(subId, [pythnet_update_T1])  → passes (first update, no prior)
  tx2: updatePriceFeeds(subId, [pythnet_update_T2])  → passes (T2 > T1, deviation ≥ 100bps)
  tx3: updatePriceFeeds(subId, [pythnet_update_T3])  → passes (T3 > T2, deviation ≥ 100bps)
  ...
  tx15: updatePriceFeeds(subId, [pythnet_update_T15]) → passes

Result: 15 keeper fees extracted from subscription balance in one Ethereum block.
Owner expected at most 1 update per block for a 100bps deviation threshold.
```

The root cause is in `_validateShouldUpdatePrices` at `Scheduler.sol` lines 388–397 and 399–453: the temporal guard uses Pythnet publish time rather than `block.timestamp`, and no per-block call limit exists for deviation-only subscriptions. [6](#0-5)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

                PythStructs.PriceFeed storage previousFeed = _state
                    .priceUpdates[subscriptionId][priceFeeds[i].id];

                // If there's no previous price, this is the first update
                if (previousFeed.id == bytes32(0)) {
                    return updateTimestamp;
                }

                // Calculate the deviation percentage
                int64 currentPrice = priceFeeds[i].price.price;
                int64 previousPrice = previousFeed.price.price;

                // Skip if either price is zero to avoid division by zero
                if (previousPrice == 0 || currentPrice == 0) {
                    continue;

                uint256 numerator = SignedMath.abs(
                    currentPrice - previousPrice
