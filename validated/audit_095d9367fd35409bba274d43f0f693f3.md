### Title
`blockhash`-Based Reveal Window Expires Disproportionately Fast on Short-Block-Time Chains, Permanently Locking User Funds — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract stores `block.number` at request time and uses the EVM `blockhash()` opcode at reveal time to incorporate the block hash into the random number. The `blockhash()` opcode only returns a non-zero value for the **256 most recent blocks**. On Ethereum (~12 s/block) this window is ~51 minutes, but on BSC (~3 s/block) it shrinks to ~12.8 minutes and on Avalanche (~1 s/block) to ~4.3 minutes. If the provider fails to call `reveal()` within this window, the request permanently reverts with `BlockhashUnavailable` and there is no refund path, causing irreversible loss of the user's paid fee.

---

### Finding Description

In `requestHelper`, the current block number is recorded unconditionally:

```solidity
req.blockNumber = SafeCast.toUint64(block.number);
req.useBlockhash = useBlockhash;
``` [1](#0-0) 

When a user calls the legacy `request()` function with `useBlockHash = true`, `revealHelper` later calls:

```solidity
bytes32 _blockHash = blockhash(req.blockNumber);
// returns zero if not within the 256 most recent blocks
if (_blockHash == bytes32(uint256(0)))
    revert EntropyErrors.BlockhashUnavailable();
``` [2](#0-1) 

The `BlockhashUnavailable` error is defined as a hard revert with no recovery path: [3](#0-2) 

The `Request` struct stores `blockNumber` as a `uint64`: [4](#0-3) 

The legacy `request()` function is still publicly callable and accepts `useBlockHash = true` from any caller: [5](#0-4) 

There is no `cancel()`, `refund()`, or timeout-based recovery function anywhere in the contract. Fees are credited to the provider and Pyth at request time: [6](#0-5) 

---

### Impact Explanation

On chains with short block times the 256-block `blockhash` availability window expires in wall-clock time far faster than on Ethereum mainnet:

| Chain | Block time | 256-block window |
|---|---|---|
| Ethereum | ~12 s | ~51 min |
| BSC | ~3 s | ~12.8 min |
| Avalanche | ~1 s | ~4.3 min |

Any provider operational hiccup (network delay, restart, congestion) lasting longer than this window on a fast chain causes the request to become permanently unresolvable. The user's fee is already distributed to the provider and Pyth at request time and cannot be recovered. The request slot remains active (non-zero `sequenceNumber`) but every `reveal()` call will revert with `BlockhashUnavailable`, permanently locking the user's funds.

---

### Likelihood Explanation

Pyth Entropy is explicitly deployed on multiple chains including BSC and Avalanche. The legacy `request()` entry point is still part of the public interface and callable by any unprivileged user. Provider off-chain infrastructure experiencing even a brief (~5 minute) outage on Avalanche is sufficient to trigger the condition. No special access or collusion is required.

---

### Recommendation

1. **Primary fix**: Remove or gate the `useBlockhash` path entirely on non-Ethereum chains, or replace the 256-block hard limit with a `block.timestamp`-based deadline (e.g., store `block.timestamp` at request time and require reveal within a fixed number of seconds).
2. **Secondary fix**: Add a `cancelRequest()` function that allows the requester to reclaim their fee after a configurable timeout (e.g., `block.timestamp > req.requestTimestamp + MAX_REVEAL_DELAY`), preventing permanent fund lock.
3. The new `requestV2` path already correctly sets `useBlockhash = false`; the legacy `request()` path should be deprecated and removed.

---

### Proof of Concept

1. Deploy Entropy on Avalanche (1 s/block).
2. User calls `request(provider, userCommitment, true)` paying the required fee. Contract stores `req.blockNumber = block.number` and credits fees to provider/Pyth.
3. Provider's off-chain service is delayed by 5 minutes (~300 blocks on Avalanche, which exceeds 256).
4. Provider calls `reveal(provider, sequenceNumber, userContribution, providerContribution)`.
5. `revealHelper` executes `blockhash(req.blockNumber)` → returns `bytes32(0)` because the block is older than 256 blocks.
6. Transaction reverts with `BlockhashUnavailable`.
7. No refund function exists. User's fee is permanently lost. Request slot is permanently stuck.
