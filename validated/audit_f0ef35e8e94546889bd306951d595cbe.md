### Title
Missing Removal of Block Hash from `is_pending_verify` on Channel Send Failure — (`File: chain/src/orphan_broker.rs`)

### Summary
In `OrphanBroker::process_descendant`, a block hash is inserted into the shared `is_pending_verify` set before the block is sent to the preload channel. If the channel send fails (receiver closed), the hash is never removed. Subsequent orphan-routing decisions in `search_orphan_leader` and `process_lonely_block` continue to treat the block as "pending verification," causing orphan descendants to be incorrectly dispatched into a dead pipeline and permanently lost from the orphan pool without ever being verified.

### Finding Description

`process_descendant` unconditionally inserts the block hash into `is_pending_verify` and then calls `send_unverified_block`: [1](#0-0) 

Inside `send_unverified_block`, if the channel send fails, the function returns early without removing the hash from `is_pending_verify`: [2](#0-1) 

The only place the hash is removed is inside `ConsumeUnverifiedBlockProcessor::consume_unverified_blocks` (and its panic guard): [3](#0-2) [4](#0-3) 

If the block never reaches `ConsumeUnverifiedBlockProcessor` because the send failed, the removal never executes. The `is_pending_verify` set is a shared `Arc<DashSet<Byte32>>` between `OrphanBroker` and `ConsumeUnverifiedBlockProcessor`: [5](#0-4) [6](#0-5) 

The stale hash is then consulted in two routing decisions:

1. `process_lonely_block` — decides whether to process a new block's parent directly or queue it as an orphan: [7](#0-6) 

2. `search_orphan_leader` — decides whether to drain orphan descendants of a leader block: [8](#0-7) 

### Impact Explanation

When the `preload_unverified_block` thread exits (e.g., because the downstream `verify_blocks` thread exited via stop signal or unhandled error), the `preload_unverified_tx` channel's receiver is dropped. Any subsequent call to `process_descendant` from the still-running `ChainService` main thread will:

1. Insert the block hash into `is_pending_verify`.
2. Fail the channel send and return early — leaving the hash permanently in `is_pending_verify`.

All orphan blocks whose parent hash matches a stale `is_pending_verify` entry will be routed through `process_descendant` again, inserting more stale hashes and draining the orphan pool. These blocks are never verified, never committed to the chain, and never returned to the orphan pool. The node's unverified-tip pointer and chain state diverge from what was actually verified, and valid chain extensions are silently discarded.

### Likelihood Explanation

The `verify_blocks` thread uses `catch_unwind` only around `consume_unverified_blocks`: [9](#0-8) 

A panic outside that scope, or a normal exit via the stop signal, drops the `unverified_block_rx` receiver, which causes the preload thread to exit, which drops `preload_unverified_rx`, which makes all subsequent `preload_unverified_tx.send()` calls fail. The `ChainService` main thread (which drives `OrphanBroker`) is a separate thread and may continue running after the verify/preload threads exit, processing new blocks from the network during the window before full shutdown. A remote peer that triggers a panic in the verify pipeline (e.g., via a crafted block that causes an unhandled error outside the `catch_unwind` scope) could exploit this window.

### Recommendation

In `send_unverified_block`, on the `Err` branch, remove the block hash from `is_pending_verify` before returning:

```rust
Err(_) => {
    info!("send unverified_block_tx failed, the receiver has been closed");
    // Remove the hash that was inserted in process_descendant to avoid
    // stale is_pending_verify entries that corrupt orphan routing.
    self.is_pending_verify.remove(&block_hash);
    return;
}
```

Alternatively, restructure `process_descendant` to only insert into `is_pending_verify` after a successful send.

### Proof of Concept

1. Node starts; `verify_blocks` thread and `preload_unverified_block` thread are running.
2. An event causes the `verify_blocks` thread to exit (stop signal or panic outside `catch_unwind`). The `unverified_block_rx` receiver is dropped.
3. The `preload_unverified_block` thread detects the closed output channel and exits. The `preload_unverified_rx` receiver is dropped.
4. A remote peer sends block B whose parent is already stored. `ChainService` calls `OrphanBroker::process_lonely_block(B)`.
5. `process_lonely_block` calls `process_descendant(B)`, which inserts `B.hash` into `is_pending_verify` at [10](#0-9) .
6. `send_unverified_block` calls `preload_unverified_tx.send(B)`, which returns `Err` because the receiver is dropped. The function returns at line 177 without removing `B.hash` from `is_pending_verify`.
7. A remote peer sends block C whose parent is B. `process_lonely_block(C)` checks `is_pending_verify.contains(&B.hash)` — returns `true` — and calls `process_descendant(C)`. C is also lost.
8. The orphan pool is drained of all descendants of B, none are ever verified, and the node's chain state is inconsistent with what was actually committed.
