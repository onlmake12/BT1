### Title
Fortuna Keeper's `fulfilled_requests_cache` Permanently Abandons Entropy Requests After Chain Reorganization - (`File: apps/fortuna/src/keeper/block.rs`)

### Summary

The Fortuna keeper service processes Entropy `Requested` events from non-finalized blocks (default `BlockStatus::Latest`) and inserts their sequence numbers into an in-memory `fulfilled_requests_cache` (`HashSet<u64>`). This cache is never evicted within a session. If a chain reorganization removes the block containing the original request, the sequence number remains in the cache, and Fortuna permanently skips re-processing the request even when it reappears in the canonical chain. The user's fee is paid but the callback is never delivered.

### Finding Description

**Root cause 1 — Default block status is `Latest` (non-finalized):**

`BlockStatus` defaults to `Latest` in both `apps/fortuna/src/chain/reader.rs` and `apps/argus/src/adapters/ethereum.rs`. The `EthereumConfig.confirmed_block_status` field is annotated `#[serde(default)]`, meaning any chain that omits this field in its config (including the sample config `lightlink_pegasus`) silently uses `Latest`. [1](#0-0) [2](#0-1) 

**Root cause 2 — `fulfilled_requests_cache` is never evicted:**

In `process_single_block_batch`, when an event is found, its sequence number is inserted into `fulfilled_requests_cache`. The `if newly_inserted` guard means any subsequent scan of the same block range (including the `block_delays` retry passes) will silently skip the event. [3](#0-2) 

**Root cause 3 — No reorg detection in `watch_blocks`:**

`watch_blocks` only tracks the monotonically increasing `last_safe_block_processed`. It never compares block hashes to detect whether a previously processed block has been replaced by a different block at the same height. [4](#0-3) 

**Root cause 4 — `process_event_with_backoff` permanently gives up on `NoSuchRequest`:**

After 5 retries receiving `NoSuchRequest` (which is exactly what happens when the request's block is reorganized away), the task returns `Permanent` error. The sequence number stays in the cache. When the request reappears in the canonical chain, the `block_delays` retry scan finds the event but `newly_inserted` is `false`, so no new task is spawned. [5](#0-4) 

### Impact Explanation

A user who submits an Entropy request on a chain configured with `confirmed_block_status = Latest` (the default) and whose request block is subsequently reorganized away will:

1. Have paid the Entropy fee (accrued to the provider in contract storage).
2. Never receive the `entropyCallback` from Fortuna within the current session.
3. Have no mechanism to force Fortuna to retry — the sequence number is permanently blocked in the in-memory cache.

The user's fee is effectively lost until a Fortuna operator manually restarts the service, at which point the backlog scan may recover the request. However, there is no guarantee of restart, and the `backlog_range` window may have already passed. [6](#0-5) 

### Likelihood Explanation

- `BlockStatus::Latest` is the Rust `#[default]` for the enum, so any chain config that omits `confirmed_block_status` is affected. The sample config (`lightlink_pegasus`) omits it.
- Chain reorganizations occur on all PoW chains and on PoS chains during network instability. Many chains Fortuna supports (Avalanche, BNB Chain, Polygon, etc.) have documented reorg histories.
- The attack requires no privileged access: any user who submits an Entropy request during a reorg window is affected.
- The `reveal_delay_blocks: 0` setting in the sample config means Fortuna processes requests immediately from the latest block, maximizing exposure to reorgs. [7](#0-6) 

### Recommendation

1. **Short term:** Change the `BlockStatus` default from `Latest` to `Safe` or `Finalized`. This is the single highest-impact fix.

```rust
#[derive(Copy, Clone, Debug, Default, ...)]
pub enum BlockStatus {
    Latest,
    Finalized,
    #[default]  // change default here
    Safe,
}
```

2. **Short term:** Add reorg detection in `watch_blocks` by fetching and comparing the block hash of `last_safe_block_processed` on each poll. If the hash differs, evict the affected sequence numbers from `fulfilled_requests_cache` and re-queue the block range.

3. **Long term:** Replace the unbounded `fulfilled_requests_cache` with a bounded, block-height-keyed structure that can be invalidated when a reorg is detected. [8](#0-7) [9](#0-8) 

### Proof of Concept

1. Deploy Fortuna against a chain with `confirmed_block_status` omitted (defaults to `Latest`) and `reveal_delay_blocks: 0`.
2. Submit an Entropy `requestWithCallback` transaction. It lands in block N. Fortuna observes the `Requested` event and inserts sequence number S into `fulfilled_requests_cache`. It spawns `process_event_with_backoff(S)`.
3. Before Fortuna's `reveal_with_callback` transaction is mined, simulate a chain reorganization that removes block N (e.g., via a competing fork on a testnet).
4. `process_event_with_backoff(S)` calls `reveal_with_callback` → contract reverts `NoSuchRequest` (request no longer exists). After 5 retries, the task exits permanently.
5. The request reappears in the canonical chain (same sequence number S, new block N'). Fortuna's `block_delays` retry pass scans the block range, finds the event, but `fulfilled_requests_cache.insert(S)` returns `false` → no task is spawned.
6. The user's fee is locked in the contract. The callback is never delivered. [10](#0-9) [11](#0-10)

pub async fn watch_blocks(
    chain_state: BlockchainState,
    last_safe_block_processed: &mut BlockNumber,
    tx: mpsc::Sender<BlockRange>,
) -> Result<()> {
    tracing::info!("Watching blocks to handle new events");

    loop {
        time::sleep(POLL_INTERVAL).await;

        let latest_safe_block = get_latest_safe_block(&chain_state).in_current_span().await;
        if latest_safe_block > *last_safe_block_processed {
            let mut from = latest_safe_block.saturating_sub(RETRY_PREVIOUS_BLOCKS);

            // In normal situation, the difference between latest and last safe block should not be more than 2-3 (for arbitrum it can be 10)
            // TODO: add a metric for this in separate PR. We need alerts
            // But in extreme situation, where we were unable to send the block range multiple times, the difference between latest_safe_block and
            // last_safe_block_processed can grow. It is fine to not have the retry mechanisms for those earliest blocks as we expect the rpc
            // to be in consistency after this much time.
            if from > *last_safe_block_processed {
                from = *last_safe_block_processed;
            }
            match tx
                .send(BlockRange {
                    from,
                    to: latest_safe_block,
                })
                .await
            {
                Ok(_) => {
                    tracing::info!(
                        from_block = from,
                        to_block = &latest_safe_block,
                        "Block range sent to handle events",
                    );
                    *last_safe_block_processed = latest_safe_block;
                }
                Err(e) => {
                    tracing::error!(
                        from_block = from,
                        to_block = &latest_safe_block,
                        "Error while sending block range to handle events. These will be handled in next call. error: {:?}",
                        e
                    );
                }
            };
        }
    }
