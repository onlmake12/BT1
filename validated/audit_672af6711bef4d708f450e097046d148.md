### Title
`Echo.executeCallback` Permanently Loses User Price Updates When Requester Does Not Implement `IEchoConsumer` — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.executeCallback` credits provider fees and clears the request **before** attempting the callback to `IEchoConsumer._echoCallback`. If the requester contract does not inherit from `IEchoConsumer` (or its `_echoCallback` reverts for any reason), the callback silently fails with no retry mechanism. The user permanently loses their price update while the provider is fully compensated.

### Finding Description

In `Echo.executeCallback`, the execution order is:

1. Provider fees are credited unconditionally.
2. The request is cleared unconditionally.
3. The callback is attempted via `try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)`.
4. On failure, only a `PriceUpdateCallbackFailed` event is emitted — no retry, no refund. [1](#0-0) [2](#0-1) 

The `IEchoConsumer._echoCallback` function is an `external` function on an **abstract contract** that internally calls `getEcho()` (an `internal virtual` function) and enforces `msg.sender == echo`: [3](#0-2) 

This means the requester **must** inherit from `IEchoConsumer` and implement `getEcho()`. Any contract that:
- Does not inherit `IEchoConsumer`, or
- Implements `getEcho()` returning `address(0)` (triggering "Echo address not set"), or
- Implements `getEcho()` returning a wrong address (triggering "Only Echo can call this function")

…will silently fail to receive the price update. The `try/catch` swallows the revert and emits `PriceUpdateCallbackFailed`. The request slot is already gone and the provider is already paid.

The developers themselves flagged this ordering risk in a TODO comment: [4](#0-3) 

Unlike Pyth Entropy, which introduces a `CALLBACK_FAILED` state allowing the callback to be retried: [5](#0-4) 

Echo has no equivalent recovery path. Once `clearRequest` is called, the sequence number is gone.

### Impact Explanation

Any user who calls `requestPriceUpdatesWithCallback` from a contract that does not correctly inherit `IEchoConsumer` will:
- Pay the full fee (Pyth protocol fee + provider base fee + per-feed fee + gas fee).
- Have their request permanently cleared.
- Never receive the price update callback.
- Have no mechanism to retry or recover funds.

The provider is fully compensated regardless of callback outcome. This breaks composability: existing contracts (e.g., existing DeFi vaults, multisigs, or proxy patterns) cannot use Echo without being redeployed to inherit `IEchoConsumer`.

### Likelihood Explanation

Any developer who:
- Deploys a contract that calls `requestPriceUpdatesWithCallback` but forgets to inherit `IEchoConsumer`,
- Misconfigures `getEcho()` to return `address(0)` or a stale Echo address after an upgrade,
- Uses a proxy pattern where the implementation inherits `IEchoConsumer` but the proxy itself does not expose `_echoCallback`,

…will trigger this silent loss. Given that Echo is a new protocol requiring a non-standard abstract base contract, misconfiguration is a realistic and likely developer error.

### Recommendation

1. **Mirror Entropy's retry pattern**: Introduce a `CALLBACK_FAILED` state in `EchoState.Request` so that a failed callback can be retried by anyone (as Entropy does via `revealWithCallback` on a failed request).
2. **Validate interface at request time**: Check that `req.requester` exposes `_echoCallback` (e.g., via ERC-165 or a try-call at `requestPriceUpdatesWithCallback` time) and revert early rather than silently failing after fees are collected.
3. **Decouple fee settlement from callback success**: Only credit provider fees after a successful callback, or implement a partial-refund path for the user when the callback permanently fails.

### Proof of Concept

1. Deploy a contract `MyConsumer` that calls `echo.requestPriceUpdatesWithCallback{value: fee}(...)` but does **not** inherit `IEchoConsumer` (no `_echoCallback` function).
2. A provider calls `echo.executeCallback(provider, sequenceNumber, updateData, priceIds)`.
3. Execution reaches line 161–164: provider fees are credited, request is cleared.
4. The `try IEchoConsumer(address(MyConsumer))._echoCallback(...)` call fails because `MyConsumer` has no such function.
5. The `catch` block emits `PriceUpdateCallbackFailed` and returns normally.
6. `MyConsumer` never receives the price update. The provider has been paid. The sequence number no longer exists. No retry is possible. [6](#0-5)
