### Title
Unbounded `forward_rate_limiter` HashMap Growth via Persistent Sessions with Unique `(from, to)` PeerIds — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

The `HolePunching` protocol maintains a `forward_rate_limiter` keyed by `(PeerId, PeerId, u32)` using an unbounded `HashMapStateStore`. Its only cleanup path is `retain_recent()`, called exclusively inside `disconnected()`. There is no periodic cleanup. An unprivileged remote peer that holds persistent sessions and sends `ConnectionRequest` messages with unique, attacker-controlled `(from, to)` PeerId pairs causes the HashMap to grow without bound, exhausting node memory.

---

### Finding Description

`HolePunching` declares two rate limiters: [1](#0-0) 

- `rate_limiter: RateLimiter<(PeerIndex, u32)>` — keyed per session, bounds per-session message rate.
- `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` — keyed by message content (`from`, `to`, `item_id`), shared across all sessions.

Both use `governor::state::keyed::HashMapStateStore`, which is an unbounded `HashMap`. [2](#0-1) 

The only place `retain_recent()` is called on `forward_rate_limiter` is in `disconnected()`: [3](#0-2) 

The `notify()` callback (fired every `CHECK_INTERVAL = 5 minutes`) cleans up `pending_delivered` and `inflight_requests` but **never** calls `retain_recent()` on either rate limiter: [4](#0-3) 

Inside `ConnectionRequestProcess::execute()`, the `forward_rate_limiter` is checked with a key derived entirely from the message payload: [5](#0-4) 

The `from` and `to` fields are parsed from the raw message bytes with no validation that they correspond to any actually-connected peer: [6](#0-5) 

Every unique `(from, to)` pair that passes the outer `rate_limiter` check inserts a new entry into the `forward_rate_limiter` HashMap. The outer limiter allows 30 `ConnectionRequest` messages per second per session: [7](#0-6) 

The `forward_rate_limiter` quota is 1 req/sec per key: [8](#0-7) 

Because each unique `(from, to)` pair is a distinct key, the outer 30 req/sec cap per session translates directly into 30 new HashMap entries per second per session. With N persistent sessions, the map grows at N×30 entries/second with no upper bound and no periodic eviction.

---

### Impact Explanation

Each `HashMapStateStore` entry for `(PeerId, PeerId, u32)` occupies approximately 150–200 bytes (two ~39-byte multihash PeerIds, a u32, governor's internal `AtomicU64` state, and HashMap bucket overhead). With a typical inbound connection limit of ~125 peers:

- **Rate of growth:** 125 × 30 = 3,750 entries/sec ≈ ~562 KB/sec
- **After 10 minutes:** ~2.25M entries ≈ ~337 MB
- **After 1 hour:** ~13.5M entries ≈ ~2 GB

This causes OOM on typical node hardware, crashing the process. The attack requires no cryptographic work, no PoW, and no privileged access.

---

### Likelihood Explanation

The attack is straightforward: open as many inbound connections as the node accepts, send `ConnectionRequest` messages with randomly generated fake `PeerId` bytes in the `from`/`to` fields at the maximum allowed rate, and never disconnect. The `from`/`to` fields are only checked for valid multihash format, not for peer registry membership. The exploit is locally reproducible with a simple script.

---

### Recommendation

1. **Add periodic cleanup:** Call `self.rate_limiter.retain_recent()` and `self.forward_rate_limiter.retain_recent()` inside `notify()`, which already fires every 5 minutes.
2. **Cap the `forward_rate_limiter` map size:** Switch to a bounded LRU-backed keyed store, or enforce a hard cap on the number of tracked keys (e.g., reject new keys once the map exceeds `MAX_SESSIONS × SOME_CONSTANT`).
3. **Validate `from`/`to` against connected peers:** Reject `ConnectionRequest` messages where `from` does not correspond to a known connected peer, eliminating the ability to inject arbitrary keys.

---

### Proof of Concept

```
1. Start a CKB node with default config.
2. Open 100 TCP sessions to the node's P2P port, each negotiating the HolePunching protocol.
3. From each session, send 30 ConnectionRequest messages/sec, each with a freshly generated random PeerId in both `from` and `to` fields (valid multihash format, but not real peers).
4. Do NOT disconnect any session.
5. After 60 seconds: forward_rate_limiter map contains 100 × 30 × 60 = 180,000 entries.
6. After 10 minutes: ~2.25M entries, ~337 MB consumed by the map alone.
7. Continue until OOM kill.
```

The `notify()` timer fires every 5 minutes but does not call `retain_recent()`, so no cleanup occurs. The map grows monotonically until the process is killed by the OS.
