### Title
Unauthenticated Remote Peer Can Force Victim Node to TCP-Connect to Arbitrary Third-Party Hosts via Hole-Punching Protocol — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

---

### Summary

An unprivileged connected peer can inject arbitrary TCP addresses into the victim node's `pending_delivered` map via a crafted `ConnectionRequest`, then immediately trigger outbound TCP connection attempts to those addresses by sending a crafted `ConnectionSync`. No authentication, PoW, or privileged role is required.

---

### Finding Description

**Step 1 — Injecting addresses into `pending_delivered`**

When the victim node is the `to` target of a `ConnectionRequest`, `respond_delivered` is called: [1](#0-0) 

The only dedup guard is keyed on `from_peer_id`. With a fresh `PeerId` per request, this check is trivially bypassed: [1](#0-0) 

The `listen_addrs` from the message are filtered only for transport type (TCP + IPv4/IPv6). There is no check that the IP belongs to the sender, is a known peer, or is otherwise trusted: [2](#0-1) 

The attacker-controlled addresses are then stored verbatim: [3](#0-2) 

**Step 2 — Triggering TCP connections via `ConnectionSync`**

When a `ConnectionSync` arrives with `route` empty and `to == self_peer_id`, the node looks up `pending_delivered` by the `from` field of the message — a field the attacker fully controls — and immediately spawns `try_nat_traversal` tasks against every stored address: [4](#0-3) 

There is no check that the session sending the `ConnectionSync` is actually the peer identified by `content.from`. The attacker sends `ConnectionRequest` with `from=FAKE_ID`, stores the third-party address, then sends `ConnectionSync` with `from=FAKE_ID` — both from the same direct connection.

**Step 3 — What `try_nat_traversal` does**

It retries TCP `connect()` to the injected address for up to 30 seconds (~150 SYNs per address at 200 ms intervals): [5](#0-4) 

Up to 24 addresses per request (`ADDRS_COUNT_LIMIT`), and the per-session rate limiter allows 30 messages/second: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

The victim node becomes a TCP amplification vector. A single attacker session can cause the victim to send thousands of TCP SYNs per second to arbitrary third-party IP:port targets, enabling:
- TCP port scanning of internal or external hosts from the victim's IP
- Connection flooding / resource exhaustion against third-party services
- Abuse of the victim's IP reputation

---

### Likelihood Explanation

Any peer that can establish a P2P connection to the victim can trigger this. No special role, leaked key, or majority hashpower is needed. The attack requires only two crafted messages and is repeatable at the rate-limiter ceiling (30 req/s per session, fresh PeerIds bypass the 2-minute dedup).

---

### Recommendation

1. **Bind `pending_delivered` to the actual session**: verify that the session sending `ConnectionSync` matches the session that originally sent the `ConnectionRequest` with the same `from` PeerId.
2. **Validate `listen_addrs` against the sender's observed address**: reject addresses whose IP does not match the remote IP of the sending session.
3. **Rate-limit `pending_delivered` inserts globally**, not just per `(from, to)` pair, to prevent map flooding with fresh PeerIds.

---

### Proof of Concept

```
1. Attacker connects to victim (becomes session S).
2. Attacker sends ConnectionRequest:
     from  = FRESH_PEER_ID_N   (new random PeerId each iteration)
     to    = VICTIM_PEER_ID
     listen_addrs = [/ip4/TARGET_IP/tcp/TARGET_PORT]
3. Victim executes respond_delivered():
     - dedup check passes (FRESH_PEER_ID_N not in pending_delivered)
     - address passes TCP/IPv4 filter
     - pending_delivered.insert(FRESH_PEER_ID_N, ([/ip4/TARGET_IP/tcp/TARGET_PORT/p2p/FRESH_PEER_ID_N], now))
4. Attacker sends ConnectionSync:
     from  = FRESH_PEER_ID_N
     to    = VICTIM_PEER_ID
     route = []
5. Victim executes ConnectionSyncProcess::execute():
     - route is empty, self == to
     - listens_info = pending_delivered.get(FRESH_PEER_ID_N) => Some([TARGET_IP:PORT])
     - spawns try_nat_traversal(bind_addr, /ip4/TARGET_IP/tcp/TARGET_PORT/...)
6. Victim sends ~150 TCP SYNs to TARGET_IP:TARGET_PORT over 30 seconds.
7. Repeat from step 2 with FRESH_PEER_ID_(N+1) for each new target.
```
