### Title
Entropy Requests Have No Expiration or Cancellation Mechanism, Permanently Locking User Fees — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract credits provider fees immediately upon request creation and stores requests with no expiration field and no user-facing cancellation function. If a provider never fulfills a request, the user's fee is permanently locked with no recourse. Because provider registration is permissionless, a malicious actor can register as a provider, collect fees, and never reveal, directly stealing user funds.

---

### Finding Description

In `requestHelper`, fees are split and credited to the provider and Pyth the moment a request is created — before any randomness is delivered:

```solidity
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The request struct stores only `blockNumber` — there is no deadline or expiration timestamp. The `isActive` predicate is purely `req.sequenceNumber != 0`:

```solidity
function isActive(
    EntropyStructsV2.Request storage req
) internal view returns (bool) {
    return req.sequenceNumber != 0;
}
``` [2](#0-1) 

The `allocRequest` function itself documents the design gap:

> "There is a chance that some requests never get revealed and remain active forever." [3](#0-2) 

There is no `cancel`, `refund`, or `expire` function anywhere in `Entropy.sol` or `EntropyGovernance.sol`. The only way a request is cleared is via `clearRequest`, which is called exclusively inside `reveal` / `revealWithCallback` — both of which require the provider to supply a valid hash-chain preimage. [4](#0-3) 

Provider registration is permissionless — any address can call `register()` and immediately begin accepting paid requests. [5](#0-4) 

---

### Impact Explanation

A malicious provider registers, attracts users, collects fees via `requestV2` / `request`, and never calls `revealWithCallback`. The provider's `accruedFeesInWei` balance grows and can be withdrawn via `withdraw`. The user's ETH is gone with no on-chain path to recover it. This is a direct loss of user funds (native gas tokens) with no upper bound on the amount stolen.

---

### Likelihood Explanation

Provider registration is open to any address with no vetting, bond, or slashing. The attack requires only deploying a provider contract, advertising a competitive fee, and ignoring reveal transactions. The barrier is low and the profit is immediate. Likelihood is **medium-high**: the attack is cheap to execute and the contract provides no deterrent beyond reputation.

---

### Recommendation

1. **Add a request expiration field** (e.g., `expiresAt = block.number + MAX_BLOCKS`) to `EntropyStructsV2.Request`.
2. **Add a `cancelRequest` function** callable by `req.requester` after expiration, which clears the request and refunds the user's fee from the provider's `accruedFeesInWei` balance.
3. Alternatively, **hold provider fees in escrow** and only release them to the provider upon successful fulfillment, so an unresponsive provider cannot profit from unfulfilled requests.

---

### Proof of Concept

1. Attacker calls `register(...)` to become a permissionless provider.
2. Victim calls `requestV2{value: fee}(attackerProvider, ...)` — fee is immediately split: `providerInfo.accruedFeesInWei += providerFee`.
3. Attacker never calls `revealWithCallback`. The request remains `isActive` forever.
4. Attacker calls `withdraw(attackerProvider, providerFee)` to drain their accrued balance.
5. Victim has no `cancel` or `refund` path; their ETH is permanently lost.
