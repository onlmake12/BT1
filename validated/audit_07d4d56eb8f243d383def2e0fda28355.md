### Title
`MAX_DEPOSIT_LIMIT` Protection for Permanent Subscriptions Can Be Gamed via Multiple `addFunds` Calls - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `MAX_DEPOSIT_LIMIT` check in the Pulse `Scheduler` contract is applied per-call on `msg.value`, not on the cumulative `status.balanceInWei`. An unprivileged user can bypass the intended 100 ETH cap on permanent subscriptions by calling `addFunds` repeatedly, each time depositing up to 100 ETH, accumulating an unbounded balance. Since permanent subscriptions prohibit withdrawals, this ETH is permanently locked in the contract beyond the protocol's intended invariant.

---

### Finding Description

`SchedulerConstants` declares:

```solidity
/// Maximum deposit limit for permanent subscriptions in wei
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
```

The comment and naming imply this is a **total balance cap** for permanent subscriptions. However, both enforcement sites check only the per-call `msg.value`:

**`createSubscription`** (line 48):
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**`addFunds`** (line 613):
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Neither check compares `status.balanceInWei + msg.value` against `MAX_DEPOSIT_LIMIT`. The cumulative balance is never bounded. A caller can invoke `addFunds` N times with `msg.value = 100 ether` each time, accumulating `N × 100 ETH` in a single permanent subscription.

Permanent subscriptions additionally enforce:
- No `withdrawFunds` (reverts with `CannotUpdatePermanentSubscription`)
- No `updateSubscription` (same revert)
- No deactivation

So any ETH deposited beyond the intended 100 ETH cap is permanently and irrecoverably locked in the contract.

---

### Impact Explanation

**Impact: Medium**

1. **Protocol invariant broken**: The constant `MAX_DEPOSIT_LIMIT` is documented as the maximum deposit for permanent subscriptions, but the invariant `balanceInWei ≤ MAX_DEPOSIT_LIMIT` is never enforced on the cumulative balance.
2. **False sense of security**: Integrators and auditors reading the code believe permanent subscriptions are capped at 100 ETH. They are not.
3. **Irreversible ETH lock**: Any ETH deposited above the intended cap is permanently locked. A user who is tricked or mistaken about the cap (e.g., a script that calls `addFunds` in a loop) suffers permanent, unrecoverable loss of funds.
4. **No admin recovery path**: There is no emergency withdrawal or admin override for permanent subscriptions.

---

### Likelihood Explanation

**Likelihood: Medium**

- The bypass requires no special privilege — any address can call `addFunds` on any active subscription.
- The attack is trivially scriptable: a loop calling `addFunds{value: 100 ether}(subscriptionId)` N times.
- A user who intends to fund a long-running permanent subscription may naturally call `addFunds` multiple times over the subscription's lifetime, inadvertently bypassing the cap.
- The check gives operators a false belief that the cap is enforced, so no off-chain monitoring is likely to catch it.

---

### Recommendation

Replace the per-call `msg.value` check with a cumulative balance check in both `createSubscription` and `addFunds`:

```solidity
// In createSubscription:
if (subscriptionParams.isPermanent &&
    msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// Should be:
if (subscriptionParams.isPermanent &&
    msg.value > MAX_DEPOSIT_LIMIT) {   // initial deposit check is fine
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// In addFunds — replace:
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {

// With:
if (params.isPermanent &&
    status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

This ensures the **total balance** of a permanent subscription never exceeds `MAX_DEPOSIT_LIMIT`, matching the documented invariant.

---

### Proof of Concept

```solidity
// Attacker creates a permanent subscription with 100 ETH (passes the check)
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);

// Attacker calls addFunds 9 more times — each call passes msg.value <= MAX_DEPOSIT_LIMIT
for (uint i = 0; i < 9; i++) {
    scheduler.addFunds{value: 100 ether}(subId);
}

// Result: permanent subscription now holds 1000 ETH
// MAX_DEPOSIT_LIMIT (100 ETH) invariant is violated
// All 1000 ETH is permanently locked — no withdrawal path exists
(, SchedulerStructs.SubscriptionStatus memory status) = scheduler.getSubscription(subId);
assert(status.balanceInWei == 1000 ether); // 10× the intended cap
```
