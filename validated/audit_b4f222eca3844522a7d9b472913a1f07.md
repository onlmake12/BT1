Audit Report

## Title
`remove_expired` Uses Bare `remove_entry`, Orphaning Descendant Transactions in Proposed Pool — (File: `tx-pool/src/pool.rs`)

## Summary
`remove_expired` in `tx-pool/src/pool.rs` calls `pool_map.remove_entry` for each expired transaction instead of `remove_entry_and_descendants`. This severs the parent link from the descendant's perspective but leaves the descendant in `entries` with `Status::Proposed`. Because `calc_ancestors` on the orphaned descendant returns an empty set, `TxSelector::txs_to_commit` passes it into every subsequent block template, causing miners to repeatedly produce chain-invalid blocks until the descendant individually expires.

## Finding Description

**Root cause — `remove_expired` (pool.rs L271–288):**

`remove_expired` collects all entries satisfying `expiry + timestamp < now_ms` and calls `pool_map.remove_entry` for each:

```rust
// tx-pool/src/pool.rs L284
self.pool_map.remove_entry(&entry.proposal_short_id());
```

`remove_entry` (pool_map.rs L235–250) calls `remove_entry_links(id)` (pool_map.rs L418–430), which iterates the expired transaction's children and calls `links.remove_parent(&child, id)` for each, severing the parent pointer from `tx_B`'s record. It also calls `remove_entry_edges`, which removes the expired transaction's own `edges.inputs` entries. However, `tx_B` itself is never removed from `entries` or `edges`.

The resulting split state after `tx_A` expires:
- `links`: `tx_B` no longer lists `tx_A` as a parent → `calc_ancestors(&tx_B)` returns `∅`
- `edges.inputs`: `O1 → tx_B` remains intact
- `entries`: `tx_B` remains with `Status::Proposed`

This is directly confirmed by the existing unit test `test_remove_entry` (score_key.rs L157–167), which explicitly asserts that after `remove_entry(&tx1_id)`, both `tx2` and `tx3` remain in the pool, and that `calc_ancestors(&tx3_id)` returns only `{tx2_id}` — meaning `tx2`'s own ancestor set is now empty.

**Exploit path through `TxSelector` (tx_selector.rs L175–189):**

```rust
let ancestors_ids = self.pool_map.calc_ancestors(&short_id);
if ancestors_ids
    .iter()
    .any(|id| !self.pool_map.has_proposed(id))
{ continue; }
```

Because `calc_ancestors(&tx_B)` returns `∅`, the guard is vacuously false and `tx_B` passes into the block template. The mined block fails chain validation because `O1` does not exist on-chain (`tx_A` was never committed). The pool is not updated on block rejection, so `tx_B` is selected again in every subsequent template.

Note: the entire descendant subtree is affected. `tx_C` (grandchild of `tx_A`) has `tx_B` as its only remaining ancestor; since `tx_B` is still `Status::Proposed`, `tx_C` also passes the guard and is included alongside `tx_B`.

**All other removal paths use `remove_entry_and_descendants`:**
- `limit_size` → `remove_entry_and_descendants` (pool.rs L307)
- `remove_by_detached_proposal` → `remove_entry_and_descendants` (pool.rs L343)
- `resolve_conflict` → `remove_entry_and_descendants` (pool_map.rs L310, L321)
- `resolve_conflict_header_dep` → `remove_entry_and_descendants` (pool_map.rs L285)

`remove_expired` is the sole path using bare `remove_entry`.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Any miner whose pool contains an orphaned proposed descendant will include it in every block template. Each mined block is invalid and discarded by the network. The miner's effective hash rate contribution drops to zero for the duration of the attack window (up to `expiry_hours`, default 12 h, after the parent expires). An attacker who broadcasts `tx_A`/`tx_B` pairs to all reachable mining nodes simultaneously can suppress a significant fraction of the network's block production at the cost of two transaction fees, for up to 12 hours per attack cycle.

## Likelihood Explanation

Reachable by any unprivileged user with no special access:
1. Submit `tx_A` (spends any live cell, creates output `O1`) via standard RPC or P2P relay.
2. Submit `tx_B` (spends `O1`) some time later, ensuring a later timestamp so `tx_B` expires after `tx_A`.
3. Both transactions are proposed (appear in a block's proposal zone) within the two-epoch proposal window.
4. After `expiry_hours` (default 12 h from `tx_A`'s submission), `remove_expired` fires, removes `tx_A`, and leaves `tx_B` in `Status::Proposed`.
5. Every block template from that point includes `tx_B`; every mined block is invalid.

No majority hashpower, no privileged RPC access, and no social engineering are required. The attack is repeatable indefinitely.

## Recommendation

Replace the bare `remove_entry` call in `remove_expired` with `remove_entry_and_descendants`, consistent with all other removal paths:

```rust
// tx-pool/src/pool.rs — remove_expired
for entry in removed {
    let tx_hash = entry.transaction().hash();
    debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
    let evicted = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
    for e in evicted {
        let reject = Reject::Expiry(e.timestamp);
        callbacks.call_reject(self, &e, reject);
    }
}
```

This mirrors `remove_by_detached_proposal` and `limit_size` and ensures both `entries` and `edges`/`links` remain consistent when a transaction expires.

## Proof of Concept

The existing unit test `test_remove_entry` (score_key.rs L157–167) already demonstrates steps 1–5 of the orphaning:

```rust
map.remove_entry(&tx1_id);
assert!(!map.contains_key(&tx1_id));
assert!(map.contains_key(&tx2_id)); // orphan confirmed
assert!(map.contains_key(&tx3_id)); // orphan confirmed
assert_eq!(
    map.calc_ancestors(&tx3_id),
    vec![tx2_id].into_iter().collect()
    // tx2's own ancestor set is now empty — fools TxSelector
);
```

**Minimal extension to prove block template inclusion:**

1. After the state above, construct a `TxSelector` over the pool and call `txs_to_commit`.
2. Assert `tx_B` appears in the result (the ancestor guard at tx_selector.rs L176–179 is vacuously false for `tx_B` since `calc_ancestors(&tx_B)` returns `∅`).
3. Assert `tx_C` also appears (its only remaining ancestor `tx_B` is `Status::Proposed`).

**Manual integration steps:**

1. Run a local CKB node with `expiry_hours = 1`.
2. Submit `tx_A` and `tx_B` (child of `tx_A`) via `send_transaction` RPC.
3. Mine blocks to advance the proposal window so both are `Status::Proposed`.
4. Wait 1 hour for `tx_A` to expire.
5. Call `get_block_template`; observe `tx_B` in the template's `transactions` field.
6. Mine the block; observe rejection with an `OutPoint` resolution error on `O1`.
7. Call `get_block_template` again; observe `tx_B` is still present.
