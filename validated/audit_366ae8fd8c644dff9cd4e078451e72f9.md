### Title
Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` â€” (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

When the `ConfirmationFraction` fee estimator is enabled, a transaction that remains in the tx-pool for more than `MAX_CONFIRM_BLOCKS` (1000) blocks and is then evicted triggers an index-out-of-bounds panic in `remove_unconfirmed_tx`. The guard that handles the `block_unconfirmed_txs` overflow case does **not** extend to the `confirm_blocks_to_failed_txs` access on line 215, leaving a reachable panic path.

---

### Finding Description

`TxConfirmStat::remove_unconfirmed_tx` computes `tx_age = tip_height - entry_height`. It correctly guards the `block_unconfirmed_txs` access:

```rust
if tx_age >= self.block_unconfirmed_txs.len() {   // len = 1000
    self.bucket_stats[bucket_index].old_unconfirmed_txs -= 1;
} else { ... }
```

But immediately after, with no additional guard, it unconditionally indexes into `confirm_blocks_to_failed_txs` (also length 1000):

```rust
if count_failure {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;  // line 215
}
```

When `tx_age = 1001`, `tx_age - 1 = 1000` â€” one past the last valid index of a 1000-element `Vec` â€” causing a Rust index-out-of-bounds panic. [1](#0-0) 

Both `confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` are initialized to the same length (`max_confirm_blocks = 1000`): [2](#0-1) 

---

### Impact Explanation

The panic crashes the CKB node process. The scope is **local node crash** (0â€“500 points). No remote code execution or fund loss is possible, but a node operator running the `ConfirmationFraction` estimator will have their node killed.

---

### Likelihood Explanation

**The default tx-pool expiry is 12 hours** (`DEFAULT_EXPIRY_HOURS = 12`): [3](#0-2) 

The expiry is time-based (milliseconds): [4](#0-3) 

At CKB's ~10-second block time, 12 hours â‰ˆ 4,320 blocks â€” more than 4Ã— the 1000-block threshold. Any low-fee-rate transaction that is not confirmed within 1001 blocks and is later evicted by `remove_expired` will trigger the panic.

The eviction path is:

1. `remove_expired` â†’ `callbacks.call_reject` â†’ `FeeEstimator::reject_tx`
2. `reject_tx` (only wired for `ConfirmationFraction`, not `Dummy`/`WeightUnitsFlow`): [5](#0-4) 

3. â†’ `drop_tx` â†’ `drop_tx_inner(tx_hash, true)` â†’ `remove_unconfirmed_tx(entry_height, self.best_height, bucket_index, true)` [6](#0-5) 

The `ConfirmationFraction` estimator is a supported (non-default) configuration option: [7](#0-6) 

---

### Recommendation

Add a bounds check before the `confirm_blocks_to_failed_txs` access, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This silently drops the failure sample for txs older than the tracking window, which is consistent with how `old_unconfirmed_txs` handles the same overflow case.

### Proof of Concept

    let stat = TxConfirmStat::default(); // MAX_CONFIRM_BLOCKS = 1000
    let fee_rate = FeeRate::from_u64(1000);
    let mut stat = stat;
    // Add tx at height 0
    let bucket_index = stat.add_unconfirmed_tx(0, fee_rate).unwrap();
    // Simulate 1001 blocks passing, then evict with count_failure=true
    // tx_age = 1001 - 0 = 1001; confirm_blocks_to_failed_txs[1000] => OOB on len=1000
    stat.remove_unconfirmed_tx(0, 1001, bucket_index, true); // panics
}
