Audit Report

## Title
Unbounded CKB-VM Cycle Execution in `CyclesEstimator::run` via `estimate_cycles` RPC — (File: `rpc/src/module/chain.rs`)

## Summary
`CyclesEstimator::run` passes `consensus.max_block_cycles` (~3.5 billion cycles) directly to `ScriptVerifier::verify` with no per-query cap, no fee requirement, and no rate limiting. Any unauthenticated caller can submit a transaction referencing a computationally expensive on-chain script and cause the RPC handler to execute CKB-VM for several seconds per call. Flooding the node with concurrent requests saturates RPC worker threads, blocking critical RPC operations.

## Finding Description
At `rpc/src/module/chain.rs` L2380, `CyclesEstimator::run` sets `let max_cycles = consensus.max_block_cycles;` and passes it directly to `ScriptVerifier::new(...).verify(max_cycles)` at L2389. [1](#0-0) 

`ScriptVerifier::verify` at `script/src/verify.rs` L197–214 iterates every script group and calls `verify_script_group(group, max_cycles - cycles)`, running the CKB-VM scheduler until the script exits or the full cycle budget is consumed. [2](#0-1) 

The deprecated `dry_run_transaction` in `rpc/src/module/experiment.rs` L230–233 delegates to the same `CyclesEstimator::new(&self.shared).run(tx)` path, exposing the same surface. [3](#0-2) 

The execution is synchronous on the RPC handler thread. There is no intermediate cap, no authentication, and no rate limiting applied before `ScriptVerifier::verify` is invoked with the full block-level cycle budget. A single crafted call can occupy one worker thread for several seconds; a modest number of concurrent calls exhausts the entire RPC thread pool. [4](#0-3) 

## Impact Explanation
Maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** An attacker deploys one computationally expensive script (a one-time on-chain cost) and issues repeated unauthenticated HTTP POST requests to `estimate_cycles` across multiple full nodes. Each call consumes up to 3.5 billion cycles of CPU time. With enough concurrent connections, all RPC worker threads are occupied, blocking `send_transaction`, `get_block_template`, and other critical RPC calls. Applied simultaneously to many nodes, this degrades the network's ability to propagate transactions and blocks.

## Likelihood Explanation
- `estimate_cycles` is part of the default `Chain` RPC module, enabled on every full node with no authentication.
- No fee or stake is required beyond the one-time cost of deploying the expensive script (or reusing an existing expensive script already on-chain).
- The attack is a simple loop of HTTP POST requests requiring no special privilege or protocol knowledge.
- It is cheap, repeatable, and can be directed at any number of nodes simultaneously.

## Recommendation
1. Introduce a dedicated per-query cycle cap (e.g., `max_estimate_cycles` in the `[rpc]` config section, defaulting to `max_tx_verify_cycles` or a similarly conservative value) and apply it inside `CyclesEstimator::run` instead of `consensus.max_block_cycles`.
2. Execute `estimate_cycles` / `dry_run_transaction` in a separate, bounded thread pool so that long-running script executions cannot starve the main RPC worker pool.
3. Consider optional RPC-level rate limiting for script-executing endpoints.

## Proof of Concept
1. Write a RISC-V script that spins in a tight loop for ~3,000,000,000 cycles and deploy it to a CKB testnet cell.
2. Construct a transaction whose lock script references that cell.
3. Send the following request with 20+ concurrent connections:
```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "estimate_cycles",
  "params": [{ "cell_deps": [...], "inputs": [...], "outputs": [...], "outputs_data": ["0x"], "version": "0x0", "witnesses": [] }]
}
```
4. Observe that all RPC threads are occupied executing the script; subsequent `get_tip_block_number` or `send_transaction` calls time out or queue indefinitely, demonstrating node-level RPC denial of service. The same attack repeated against multiple nodes demonstrates network-wide congestion.

    pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();
                let tx_env = TxVerifyEnv::new_submit(tip_header);
                match ScriptVerifier::new(
                    Arc::new(resolved),
                    snapshot.as_data_loader(),
                    consensus,
                    Arc::new(tx_env),
                )
                .verify(max_cycles)
                {
                    Ok(cycles) => Ok(EstimateCycles {
                        cycles: cycles.into(),
                    }),
                    Err(err) => Err(RPCError::custom_with_error(
                        RPCError::TransactionFailedToVerify,
                        err,
                    )),
                }
            }
            Err(err) => Err(RPCError::custom_with_error(
                RPCError::TransactionFailedToResolve,
                err,
            )),
        }
    }

    pub fn verify(&self, max_cycles: Cycle) -> Result<Cycle, Error> {
        let mut cycles: Cycle = 0;

        // Now run each script group
        for (_hash, group) in self.groups() {
            // max_cycles must reduce by each group exec
            let used_cycles = self
                .verify_script_group(group, max_cycles - cycles)
                .map_err(|e| {
                    #[cfg(feature = "logging")]
                    logging::on_script_error(_hash, &self.hash(), &e);
                    e.source(group)
                })?;

            cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
        }
        Ok(cycles)
    }
