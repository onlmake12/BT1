### Title
Unbounded `while` Loop in `executeCallback` Advances `firstUnfulfilledSeq` via Repeated Storage Reads, Enabling Out-of-Gas DoS — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` by iterating one-by-one through cleared sequence numbers, performing a cold `SLOAD` on every iteration. An unprivileged attacker can manufacture a large gap between `firstUnfulfilledSeq` and `currentSequenceNumber` by creating many requests and fulfilling them out of order, causing the final `executeCallback` call to exhaust the block gas limit and permanently revert.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs:

```solidity
// Echo.sol lines 169-174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

`findRequest` reads from either the fixed-size `_state.requests` array or the `_state.requestsOverflow` mapping — both are storage reads. Each iteration therefore costs at minimum one cold `SLOAD` (2 100 gas post-EIP-2929). The loop runs until it finds the first sequence number whose slot is still active, or until it reaches `currentSequenceNumber`.

The developers themselves flagged this in the inline comment immediately above the loop:

> *"TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number."*

`firstUnfulfilledSeq` is a `uint64` field in `EchoState.State` with no upper bound on the gap it must traverse.

---

### Impact Explanation

If the loop must traverse **N** cleared slots, gas cost ≈ `N × 2 100` (SLOADs alone). At N = 15 000 the loop consumes ~31.5 M gas, exceeding Ethereum's 30 M block gas limit. The `executeCallback` transaction reverts, meaning:

- The request is never cleared from storage.
- The consumer's `_echoCallback` is never invoked.
- The provider's accrued fee credit (`_state.providers[providerToCredit].accruedFeesInWei`) is never written.
- Funds paid by the requester are permanently locked in the contract (no refund path exists once a request is stored).

This constitutes a **permanent, irreversible DoS** of the Echo callback mechanism for the targeted request.

---

### Likelihood Explanation

Both entry points are fully permissionless:

- `requestPriceUpdatesWithCallback` — any address, any number of calls, no rate limit.
- `executeCallback` — any address can call it after the exclusivity period; the attacker can be the caller.

Attack steps:

1. Attacker calls `requestPriceUpdatesWithCallback` N+1 times, obtaining sequence numbers `S`, `S+1`, …, `S+N`.
2. Attacker calls `executeCallback` for sequences `S+1`, `S+2`, …, `S+N` (in any order). Each call advances `firstUnfulfilledSeq` only up to `S` (the still-active slot), so the gap accumulates.
3. Attacker (or victim) calls `executeCallback` for sequence `S`. The while loop now iterates from `S` through `S+N`, performing N+1 SLOADs, exhausting gas.

The cost to the attacker is the sum of fees for N+1 requests. Because `getFee` is configurable by the provider and can be set to a minimal value, the attack cost can be kept low relative to the damage caused.

---

### Recommendation

Replace the unbounded sequential scan with a direct mathematical jump, analogous to the formula recommended in the original report. Since sequence numbers are dense integers, the next active slot can be found without iterating every cleared slot. The simplest safe fix is to bound the loop to a constant maximum number of steps per call (e.g., 100), accepting that `firstUnfulfilledSeq` may lag temporarily but will never cause a single transaction to run out of gas:

```solidity
uint256 maxSteps = 100;
while (
    maxSteps > 0 &&
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
    maxSteps--;
}
```

A more robust long-term fix is the doubly-linked list the developers already noted in the TODO comment, which would make the advancement O(1).

### Proof of Concept

```
Precondition: provider registered with minimal fees.

1. for i in 1..N+1:
       call requestPriceUpdatesWithCallback(provider, now, [priceId], gasLimit)
       → sequence numbers S, S+1, ..., S+N stored

2. for i in 1..N:
       call executeCallback(provider, S+i, updateData, priceIds)
       → each clears slot S+i; while loop stops at S (still active)
       → firstUnfulfilledSeq remains S

3. call executeCallback(provider, S, updateData, priceIds)
       → clearRequest(S) marks slot S inactive
       → while loop: iterates S → S+1 → ... → S+N (N+1 SLOADs)
       → at N ≈ 14 000, gas consumed > 30 M → transaction reverts
       → request S is never cleared, consumer never receives callback,
          funds locked permanently
