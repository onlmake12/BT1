### Title
Entropy.requestV2 – Users May Overpay or Be Griefed Due to Mutable Provider Fee With No Maximum Fee Guard - (File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol)

---

### Summary

`Entropy.requestHelper()` consumes the entire `msg.value` without refunding any excess, and providers can change their `feeInWei` at any time via `setProviderFee()`. There is no `maxFee` parameter in any `requestV2` variant. A provider can front-run a user's `requestV2` transaction with a fee increase, causing the user's transaction to either revert (if they sent exactly the quoted fee) or silently overpay (if they sent a buffer above the quoted fee), with the excess permanently captured by the contract.

---

### Finding Description

`requestHelper()` computes the required fee at execution time and checks `msg.value >= requiredFee`, but distributes **all** of `msg.value` — `providerFee` to the provider and the remainder to `_state.accruedPythFeesInWei` — with no refund path: [1](#0-0) 

The interface documentation explicitly acknowledges this: "excess value is *not* refunded to the caller." [2](#0-1) 

Any registered provider can change their fee at any time with no delay, cap, or timelock: [3](#0-2) 

None of the `requestV2` overloads accept a `maxFee` parameter: [4](#0-3) 

---

### Impact Explanation

**Case A – Griefing/DoS**: A user calls `getFeeV2(provider, gasLimit)` off-chain, receives fee `X`, and submits `requestV2{value: X}(provider, gasLimit)`. The provider front-runs with `setProviderFee(X + 1)`. The user's transaction reverts with `InsufficientFee`. The provider can repeat this indefinitely, permanently blocking any specific user or contract from obtaining randomness from that provider while wasting the user's gas.

**Case B – Silent Overpayment**: Integrating contracts commonly send `msg.value = getFeeV2() + buffer` to absorb gas price fluctuations. If the provider front-runs with `setProviderFee(originalFee + delta)` where `delta ≤ buffer`, the user's transaction succeeds but the excess ETH is permanently split between the provider (`delta`) and Pyth (`buffer - delta`). The user has no recourse and no way to specify a maximum acceptable fee. [5](#0-4) 

---

### Likelihood Explanation

The provider is an unprivileged but permissionlessly registered participant who controls their own `feeInWei`. The keeper infrastructure (`adjust_fee_if_necessary`) already adjusts fees dynamically based on gas prices, meaning fee changes are a normal operational event — not a hypothetical one. A malicious provider can exploit the no-refund property with zero cost beyond gas for the `setProviderFee` call. Integrating contracts that add a buffer (a standard defensive pattern) are the most exposed. [6](#0-5) 

---

### Recommendation

1. Add a `maxFee` parameter to `requestV2` variants and revert if `requiredFee > maxFee`, mirroring the fix applied in the referenced Pool.buy report.
2. Refund `msg.value - requiredFee` to `msg.sender` at the end of `requestHelper()` when `msg.value > requiredFee`.

---

### Proof of Concept

```
// 1. User queries fee off-chain
uint128 fee = entropy.getFeeV2(provider, gasLimit); // returns X

// 2. Provider front-runs:
//    entropy.setProviderFee(X + 1 wei)   ← provider tx lands first

// 3. User's tx executes:
entropy.requestV2{value: fee}(provider, gasLimit);
// → reverts: InsufficientFee  (Case A)

// --- OR ---

// 2b. User sends buffer:
entropy.requestV2{value: fee + 1 ether}(provider, gasLimit);
// Provider front-runs: setProviderFee(X + 0.5 ether)
// → tx succeeds; user pays X + 1 ether; provider captures 0.5 ether extra;
//   remaining 0.5 ether goes to accruedPythFeesInWei — never refunded (Case B)
``` [7](#0-6)

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
