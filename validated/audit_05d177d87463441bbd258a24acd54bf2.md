3. **Line 753** — `submit_entry(...)`: calls `add_pending` → `pool_map.add_entry` → `check_and_record_ancestors`.

### Does the bug cause the claimed amplification?

**With correct limit = 25:**
- Tx₁–Tx₂₅: scripts executed, accepted into pool.
- Tx₂₆: scripts executed (3M cycles wasted), then **rejected** by `check_and_record_ancestors`.
- Tx₂₇ onward: **cannot be submitted** — Tx₂₆ is not in the pool, so Tx₂₇'s input (Tx₂₆'s output) doesn't exist. `pre_check` / resolution fails before `verify_rtx` is reached.
- Total CKB-VM cost: ~26 × 3M cycles.

**With the bug (limit = 1000):**
- Tx₁–Tx₁₀₀₀: scripts executed, **all accepted** into pool.
- Total CKB-VM cost: 1000 × 3M = 3 billion cycles.

The amplification is real and concrete: ~38× more CKB-VM execution than the operator's policy intended.

### Title
Silent `cmp::max` floor overrides operator `max_ancestors_count`, enabling 40× CKB-VM script execution amplification via legacy config — (`util/app-config/src/legacy/tx_pool.rs`)

### Summary
A `cmp::max(DEFAULT_MAX_ANCESTORS_COUNT, max_ancestors_count)` expression in the legacy config conversion path unconditionally floors `max_ancestors_count` to 1000, silently discarding any operator-configured value below 1000. Because CKB-VM script execution occurs before the ancestor count check in `_process_tx`, an attacker can submit a 1000-deep transaction chain with heavy lock scripts, causing the node to execute up to 40× more CKB-VM cycles than the operator's policy permits.

### Finding Description
In `util/app-config/src/legacy/tx_pool.rs`, `From<TxPoolConfig> for crate::TxPoolConfig` at line 129:

```rust
max_ancestors_count: cmp::max(DEFAULT_MAX_ANCESTORS_COUNT, max_ancestors_count),
```

`DEFAULT_MAX_ANCESTORS_COUNT` is 1000. Any operator-configured value below 1000 (e.g., 25) is silently replaced with 1000. No warning is emitted.

In `tx-pool/src/process.rs`, `_process_tx` (lines 705–777) executes in this order:
1. `pre_check` — no ancestor count check.
2. `verify_rtx` — CKB-VM runs all lock/type scripts.
3. `submit_entry` → `pool_map.add_entry` → `check_and_record_ancestors` — ancestor limit enforced.

Because Tx₂₆ through Tx₁₀₀₀ are each accepted into the pool (limit is 1000, not 25), each subsequent transaction in the chain can be submitted, resolved, and script-verified. With a correct limit of 25, Tx₂₆ would be rejected and Tx₂₇ would fail resolution (Tx₂₆ not in pool), capping the chain at 26 script executions.

### Impact Explanation
An attacker submitting a 1000-deep chain where each transaction's lock script consumes 3,000,000 cycles causes the node to execute 3,000,000,000 total CKB-VM cycles — approximately 38× more than the operator's intended 25-ancestor policy would allow (~78,000,000 cycles). This enables sustained, low-cost CPU exhaustion and pool memory bloat (1000 entries vs. 25), degrading block template generation and peer relay performance.

### Likelihood Explanation
Any node operator who migrated from a legacy TOML config with a conservative `max_ancestors_count` (a common hardening practice) is silently running with limit 1000. The attack requires only standard transaction submission (RPC `send_transaction` or P2P relay), no privileges, and no special knowledge beyond the node's public behavior.

### Recommendation
Replace the `cmp::max` floor with the operator's value directly:

```rust
// Before (buggy):
max_ancestors_count: cmp::max(DEFAULT_MAX_ANCESTORS_COUNT, max_ancestors_count),

// After:
max_ancestors_count,
```

If a minimum floor is intentional for protocol correctness, document it explicitly and emit a warning when the operator's value is overridden. Additionally, consider moving the ancestor count check into `pre_check` (before `verify_rtx`) so that over-limit transactions never reach CKB-VM execution.

### Proof of Concept
1. Configure a node with a legacy TOML containing `max_ancestors_count = 25`.
2. Start the node; observe via RPC `tx_pool_info` that the effective limit is 1000 (or instrument the config loading path).
3. Submit Tx₁ (spends a confirmed UTXO, lock script burns 3,000,000 cycles).
4. Submit Tx₂ spending Tx₁'s output, ..., Tx₁₀₀₀ spending Tx₉₉₉'s output.
5. All 1000 transactions are accepted. Measure total CKB-VM cycles via metrics or logs: ~3,000,000,000 cycles consumed.
6. Repeat with a correctly-configured node (`max_ancestors_count = 25`): only Tx₁–Tx₂₅ are accepted; Tx₂₆ is rejected after script execution; Tx₂₇ fails resolution. Total: ~78,000,000 cycles.
7. Ratio confirms ~38× amplification attributable solely to the `cmp::max` bug.

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

                    break;
                }
