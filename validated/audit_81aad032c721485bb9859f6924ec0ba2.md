### Title
`PythAggregatorV3.latestAnswer()` and `latestRoundData()` Return Stale Prices Without Validation — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink-compatible adapter, explicitly promoted as a migration path for protocols moving from Chainlink to Pyth. Every price-returning function in the contract (`latestAnswer`, `latestRoundData`, `getRoundData`, `getAnswer`) calls `pyth.getPriceUnsafe()`, which by design makes **no staleness guarantees**. No `maxAge` parameter or `publishTime` check is enforced anywhere in the contract. Additionally, `latestRoundData()` sets `answeredInRound == roundId` unconditionally, which silently defeats the standard Chainlink staleness guard (`require(answeredInRound >= roundId)`), giving downstream protocols a false sense of safety.

---

### Finding Description

`PythAggregatorV3.sol` implements the Chainlink `AggregatorV3Interface` on top of Pyth price feeds. All four price-reading functions delegate to `pyth.getPriceUnsafe(priceId)`:

```solidity
// line 53-56
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// line 99-119
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
    //                                                                            ^^^^^^^ answeredInRound == roundId always
}
```

`IPyth.getPriceUnsafe` is explicitly documented as returning a price "from arbitrarily far in the past" with no recency guarantee:

> "This function returns the most recent price update in this contract without any recency checks. This function is unsafe as the returned price update may be arbitrarily far in the past."

There is no configurable `maxAge`, no `publishTime` comparison against `block.timestamp`, and no revert path for stale data anywhere in `PythAggregatorV3`.

A compounding issue: `latestRoundData()` returns `answeredInRound = uint80(price.publishTime)` and `roundId = uint80(price.publishTime)`, making them always equal. Protocols that apply the standard Chainlink staleness guard `require(answeredInRound >= roundId, "stale")` will **always pass** this check regardless of how old the price is.

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a drop-in Chainlink replacement — the exact use case the contract is designed and documented for — will:

1. Receive prices from arbitrarily far in the past with no on-chain revert.
2. Have their standard Chainlink staleness guard (`answeredInRound >= roundId`) silently bypassed.
3. Compute collateral values, borrow rates, liquidation thresholds, or other financial logic on stale data.

This is the same root impact as the reference report: wrong price calculations leading to incorrect protocol behavior (bad liquidations, mispriced collateral, exploitable arbitrage windows).

---

### Likelihood Explanation

The contract is actively promoted in Pyth's official documentation as the migration path from Chainlink. The migration guide (`apps/developer-hub/content/docs/price-feeds/core/migrate-an-app-to-pyth/chainlink.mdx`) shows deploying `PythAggregatorV3` and passing it directly to a `ChainlinkApp`. Any such protocol that does not independently add a `publishTime` staleness check is exposed. Because Pyth is a pull oracle, price updates are not guaranteed to be pushed on a schedule, making stale prices a realistic and recurring condition.

---

### Recommendation

Add a configurable `maxAge` parameter to the contract and enforce it in all price-reading functions using `getPriceNoOlderThan` instead of `getPriceUnsafe`:

```solidity
uint256 public maxAge; // configurable, e.g. 60 seconds

function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    return int256(price.price);
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

This mirrors the fix applied in the reference report (IndexCoop PR #142): switch from the unchecked call to the checked variant and add a configurable `maxPriceAge`.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a Pyth contract and a price feed ID.
2. Allow the on-chain Pyth price to go stale (no `updatePriceFeeds` call for, say, 10 minutes).
3. Call `latestAnswer()` — it returns the 10-minute-old price with no revert.
4. Call `latestRoundData()` — it returns the stale price; `answeredInRound == roundId` so `require(answeredInRound >= roundId)` passes silently.
5. A downstream lending protocol using this adapter computes collateral value on the stale price, enabling under-collateralized borrows or incorrect liquidations.
