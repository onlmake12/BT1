Audit Report

## Title
`check_purge` Permanently Returns `EvictionFailed` When Peer Store Is Filled With Exactly 4 Peers Per /16 Network Group — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

`check_purge` contains two compounding logic flaws: a strict `> 4` guard (instead of `>= 4`) and a `take(len/2)` that examines only half the network groups. An attacker who fills the peer store with exactly 16,384 addresses distributed across 4,096 distinct network groups (4 per group) causes `check_purge` to permanently return `PeerStoreError::EvictionFailed` on every subsequent `add_addr` call, permanently blocking peer discovery for the targeted node.

## Finding Description

**Root cause — two independent flaws in `check_purge`:**

**Step 1** collects non-connectable peers for eviction. `add_addr` always inserts entries via `AddrInfo::new(addr, 0, score, flags.bits())`, setting `last_connected_at_ms=0` and `attempts_count=0`. [1](#0-0) 

`is_connectable` only returns `false` for `last_connected_at_ms == 0` when `attempts_count >= ADDR_MAX_RETRIES` (i.e., `>= 3`). With `attempts_count=0`, every adversarially-inserted address is connectable, so Step 1 yields an empty `candidate_peers` and removes nothing. [2](#0-1) 

**Step 2** groups peers by network segment, sorts by group size descending, then applies two filters: [3](#0-2) 

- `.take(len / 2)`: only the top half of groups (by size) are examined.
- `if addrs.len() > 4`: only groups with **more than** 4 peers are eligible for eviction.

With 4,096 groups of exactly 4 peers each:
- `len = 4096`, `take(2048)` — only 2,048 groups examined (irrelevant here since all groups are identical in size)
- For every group: `4 > 4` → **false** → `None` returned
- `candidate_peers` remains empty
- Line 399–400 fires unconditionally: `return Err(PeerStoreError::EvictionFailed.into())` [4](#0-3) 

Even without the `take(len/2)` issue, the `> 4` off-by-one alone is sufficient to block eviction for all groups of size ≤ 4. The fill condition `4096 × 4 = 16384 = ADDR_COUNT_LIMIT` is exact and deterministic. [5](#0-4) 

## Impact Explanation

Once the peer store is saturated in this configuration, every call to `add_addr` — the sole entry point for all P2P-discovered peer addresses (used by the discovery protocol, DNS seeding, and identify protocol) — returns `Err(EvictionFailed)`. The node can no longer learn new peer addresses from the network. The poisoned state persists across restarts because `addr_manager.db` reloads the same entries. This constitutes peer discovery starvation: honest nodes cannot learn new peers, degrading connectivity and potentially enabling follow-on eclipse or chain-tip isolation attacks. Applied at scale to many nodes simultaneously, this degrades the CKB network's peer discovery infrastructure with very low attacker cost. This matches **High: Vulnerabilities or bad designs which could cause CKB network degradation with few costs**.

## Likelihood Explanation

The attack is reachable via the standard CKB discovery protocol (`GetNodes`/`Nodes` messages). A single adversarial peer accepted as a P2P connection can advertise up to 16,384 addresses across 4,096 /16 subnets. No special privilege, PoW, or key material is required. The fill condition is exact and deterministic. The attacker needs only one accepted P2P connection to a target node. The resulting state persists across restarts via `addr_manager.db`, making remediation require manual database deletion. [6](#0-5) 

## Recommendation

1. **Fix the off-by-one**: Change `addrs.len() > 4` to `addrs.len() >= 4` at line 378 so groups of exactly 4 are eligible for eviction.
2. **Remove or justify `take(len/2)`**: There is no documented reason to examine only half the groups under eviction pressure. Consider removing the `take` or replacing it with a full iteration.
3. **Add a guaranteed fallback**: If `candidate_peers` is still empty after both steps, fall back to evicting the oldest/least-recently-seen entries regardless of group size, ensuring the store can always accept new addresses. [7](#0-6) 

## Proof of Concept

```rust
#[test]
fn test_eviction_failed_with_4_per_group() {
    use crate::peer_store::{PeerStore, ADDR_COUNT_LIMIT};
    use crate::{Flags, PeerId};
    use p2p::multiaddr::Multiaddr;

    let mut peer_store = PeerStore::default();
    // Fill with 4096 groups * 4 peers each = 16384 = ADDR_COUNT_LIMIT
    // Each group is a distinct /16 subnet: a.b.0.1 for varying a, b
    let mut count = 0;
    'outer: for a in 0u8..=255 {
        for b in 0u8..=255 {
            for port in 1u16..=4 {
                if count >= ADDR_COUNT_LIMIT { break 'outer; }
                let addr: Multiaddr = format!(
                    "/ip4/{}.{}.0.1/tcp/{}/p2p/{}",
                    a, b, port, PeerId::random().to_base58()
                ).parse().unwrap();
                peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
                count += 1;
            }
        }
    }
    assert_eq!(peer_store.addr_manager().count(), ADDR_COUNT_LIMIT);

    // Any new add_addr must now fail with EvictionFailed
    let new_addr: Multiaddr = format!(
        "/ip4/200.200.200.1/tcp/9999/p2p/{}",
        PeerId::random().to_base58()
    ).parse().unwrap();
    let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
    assert!(result.is_err(), "Expected EvictionFailed, got Ok");
}
```

The test fills the store with 4,096 groups × 4 connectable peers each (all `last_connected_at_ms=0`, `attempts_count=0`). The final `add_addr` triggers `check_purge`: Step 1 finds no non-connectable peers; Step 2 finds no groups with `len > 4`; `Err(EvictionFailed)` is returned unconditionally. [8](#0-7)

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {
                            Some(
                                addrs
                                    .iter()
                                    .choose_multiple(&mut rand::thread_rng(), 2)
                                    .into_iter()
                                    .map(|addr| addr.addr.clone())
                                    .collect::<Vec<Multiaddr>>(),
                            )
                        } else {
                            None
                        }
                    })
                    .flatten()
                    .collect()
            };

            for key in candidate_peers.iter() {
                self.addr_manager.remove(key);
            }

            if candidate_peers.is_empty() {
                return Err(PeerStoreError::EvictionFailed.into());
            }
        }
        Ok(())
    }
