### Title
Pre-Funding Config PDA Causes Permanent DoS on `initialize` — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

---

### Summary

The `initialize` function unconditionally calls `system_instruction::create_account` to allocate the config PDA. On Solana, `create_account` fails with `SystemError::AccountAlreadyInUse` if the target account already holds any lamports — even 1. Because the config PDA address is fully deterministic and publicly computable from the program ID, an unprivileged attacker can send 1 lamport to that address before the deployer calls `initialize`, permanently blocking initialization with no recovery path in the program.

---

### Finding Description

The config PDA is derived deterministically:

```
Pubkey::find_program_address(&["CONFIG".as_bytes()], program_id)
``` [1](#0-0) 

The `initialize` function then calls:

```rust
invoke_signed(
    &system_instruction::create_account(
        payer.key,
        config.key,
        lamports,
        accounts::config::SIZE.try_into()...,
        program_id,
    ),
    ...
)?;
``` [2](#0-1) 

The Solana system program's `CreateAccount` instruction checks `to_account.lamports() > 0` and returns `SystemError::AccountAlreadyInUse` if true. There is no fallback using `transfer` + `allocate` + `assign`, and `validate_config` performs no pre-check on the account's lamport balance — it only validates the PDA address and writability. [3](#0-2) 

After the `invoke_signed` fails, the function returns the error immediately with `?`. There is no alternative code path, no retry logic, and no program-level recovery mechanism. [4](#0-3) 

---

### Impact Explanation

The config account is the root of the entire program's authority model — `InitializePublisher` requires a valid, initialized config account. If `initialize` can never succeed, the program is permanently non-functional on that deployment. The deployer would be forced to redeploy with a new program ID (losing any integrations pointing to the original ID) or upgrade the program binary to use a different PDA seed.

---

### Likelihood Explanation

The program ID is observable on-chain the moment the deploy transaction lands. An attacker monitoring the network can compute the config PDA immediately and submit a 1-lamport transfer in the same slot or the next slot, before the deployer's separate `initialize` transaction. The cost to the attacker is negligible (~1 lamport + transaction fee). The deployer can avoid this only by including `initialize` atomically in the same transaction as deployment, which is not enforced or documented.

---

### Recommendation

Replace `create_account` with the three-step pattern that tolerates a pre-funded account:

1. Transfer any shortfall in lamports with `system_instruction::transfer` (only if `config.lamports() < required`).
2. Allocate space with `system_instruction::allocate`.
3. Assign ownership with `system_instruction::assign`.

This is the standard Solana pattern for initializing PDAs that may have been pre-funded. The same issue exists in `initialize_publisher` for the publisher config PDA. [5](#0-4) 

---

### Proof of Concept

```rust
// localnet test
let (config_pda, _) = Pubkey::find_program_address(&[b"CONFIG"], &program_id);

// Attacker sends 1 lamport to the config PDA before initialize is called
banks_client.process_transaction(Transaction::new_with_payer(
    &[system_instruction::transfer(&attacker.pubkey(), &config_pda, 1)],
    Some(&attacker.pubkey()),
)).await.unwrap();

// Deployer now calls initialize — this will fail with AccountAlreadyInUse
let result = banks_client.process_transaction(initialize_tx).await;
assert!(result.is_err()); // SystemError::AccountAlreadyInUse

// No recovery path exists; the program cannot be initialized on this deployment
```

pub fn validate_config<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    bump: u8,
    program_id: &Pubkey,
    require_writable: bool,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let config = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    let (config_pda, expected_bump) =
        Pubkey::find_program_address(&[CONFIG_SEED.as_bytes()], program_id);
    ensure!(ProgramError::InvalidInstructionData, bump == expected_bump);
    ensure!(
        ProgramError::InvalidArgument,
        pubkey_eq(config.key, &config_pda)
    );
    if require_writable {
        ensure!(ProgramError::InvalidArgument, config.is_writable);
    }
    Ok(config)
}
