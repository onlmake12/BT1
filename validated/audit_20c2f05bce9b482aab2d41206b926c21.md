### Title
Unbounded RPC JSON Batch Request Size Grants Unlimited Processing Allowance — (`rpc/src/server.rs`)

### Summary
The CKB RPC server uses an opt-in batch-size limit (`rpc_batch_limit`) that is `None` by default. When unconfigured, the server processes an arbitrarily large JSON-RPC batch with no cap, directly analogous to the "unlimited allowance" pattern in the external report: a resource grant is made without scoping it to the necessary amount.

### Finding Description

`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is only populated when the operator explicitly sets `rpc_batch_limit` in `ckb.toml`. [1](#0-0) 

It is set only when the config field is `Some`: [2](#0-1) 

The batch handler checks the limit only when the `OnceLock` has been populated: [3](#0-2) 

When `rpc_batch_limit` is absent (the default), `JSONRPC_BATCH_LIMIT.get()` returns `None`, the guard is never entered, and every call in the batch is dispatched unconditionally via `stream::iter(calls).then(...)`. The config field is typed `Option<usize>` with no `#[serde(default)]` fallback: [4](#0-3) 

The production default template explicitly documents the absence of a limit: [5](#0-4) 

Additionally, the HTTP server applies `CorsLayer::permissive()`, meaning any web origin can issue cross-origin batch requests if the RPC endpoint is reachable: [6](#0-5) 

### Impact Explanation

A caller submitting a single HTTP POST with a JSON array of thousands of small RPC calls (e.g., `get_tip_block_number`) forces the node to allocate a response stream entry per call, execute each method, and buffer results. With no batch cap, memory and CPU consumption grow linearly with the number of calls in the array. A sufficiently large batch can exhaust node memory and stall or crash the process, causing a denial-of-service against the node's chain-processing and P2P functions.

### Likelihood Explanation

The RPC defaults to `127.0.0.1:8114` (localhost), so remote exploitation requires the operator to expose the port — a common deployment pattern for dApp backends and exchanges. Even on localhost, any local process (including a malicious script or browser tab via `CorsLayer::permissive()`) qualifies as the "RPC caller" attacker profile explicitly listed in scope. The attack requires no credentials, no privileged keys, and no special protocol knowledge beyond standard JSON-RPC.

### Recommendation

Apply a hard default batch limit in the server initialization path rather than relying on an opt-in config value. For example, set a safe default (e.g., 2000) when `rpc_batch_limit` is `None`:

```rust
// In RpcServer::new, replace:
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}

// With:
let limit = config.rpc_batch_limit.unwrap_or(2000);
let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| limit);
```

This mirrors the external report's recommendation: approve (process) only the necessary amount per operation rather than granting an unlimited allowance.

### Proof of Concept

```bash
# Build a batch of 100,000 no-op RPC calls
python3 -c "
import json, sys
batch = [{'jsonrpc':'2.0','method':'get_tip_block_number','params':[],'id':i}
         for i in range(100000)]
sys.stdout.write(json.dumps(batch))
" > big_batch.json

# Submit to a default CKB node (localhost RPC, no rpc_batch_limit set)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  --data-binary @big_batch.json > /dev/null &

# Observe node memory spike and potential stall
watch -n1 'ps aux | grep ckb | grep -v grep'
```

Without `rpc_batch_limit` configured, the node processes all 100,000 calls. Scaling the batch size or issuing concurrent requests amplifies the impact until the node OOMs or becomes unresponsive to peers and miners.
