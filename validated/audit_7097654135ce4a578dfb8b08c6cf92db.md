### Title
User Can Select Favorable Random Number by Timing `reveal()` Within 255-Block Window - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When a user requests randomness via the legacy `request()` API with `useBlockHash = true`, the final random number incorporates `blockhash(req.blockNumber)`. Because the user controls *when* they call `reveal()` (within the 255-block window), and because the provider's secret `x_i` is publicly queryable from the Fortuna service before the reveal, the user can pre-compute the random number for every available future block and cherry-pick the most favorable outcome — giving them up to 255 effective choices instead of one.

---

### Finding Description

The `request()` function in `Entropy.sol` accepts a `useBlockHash` boolean parameter. When `true`, `requestHelper` stores `req.useBlockhash = true` and `req.blockNumber = block.number`. [1](#0-0) 

During `revealHelper`, if `req.useBlockhash` is set, the contract fetches `blockhash(req.blockNumber)` and incorporates it into the final random number: [2](#0-1) 

The code comment at line 414–418 explicitly acknowledges the design tension:

> *"The `blockhash` function will return zero if the req.blockNumber is equal to the current block number, or if it is not within the 256 most recent blocks. This allows the user to select between two random numbers by executing the reveal function in the same block as the request, or after 256 blocks. This gives each user two chances to get a favorable result on each request."*

The code reverts when `blockhash == 0` to block those two degenerate cases. However, the comment understates the real attack surface: within the valid window (blocks `N+1` through `N+255`), the user has **255 distinct blockhashes** to choose from, each producing a different random number.

The `reveal()` function enforces `msg.sender == req.requester`, meaning only the original user can call it — and they can call it at any block they choose within the window: [3](#0-2) 

Critically, in the non-callback `reveal()` flow, the **user themselves supplies `providerContribution`** (the provider's `x_i`). The provider's `x_i` is publicly available from the Fortuna service REST API at `/revelations/{sequenceNumber}` before the user calls `reveal()`: [4](#0-3) 

The final random number is `combineRandomValues(userContribution, providerContribution, blockHash)`. Since the user knows all three inputs for every future block in the window, they can pre-compute all 255 possible outcomes and call `reveal()` at the block that produces the most favorable result.

The `EntropyStructsV2.Request` struct confirms `useBlockhash` is a stored per-request flag: [5](#0-4) 

---

### Impact Explanation

Any application using the legacy `request()` API with `useBlockHash = true` is vulnerable. The user effectively has 255 independent draws of the random number and can select the best one. For applications using randomness to determine high-value outcomes (NFT trait rarity, lottery winners, game results), this completely breaks the fairness guarantee. The user pays only one fee but gets 255 chances to select a favorable outcome.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. Calling `request()` with `useBlockHash = true` — a public, permissionless function.
2. Querying the Fortuna public REST API for the provider's `x_i` — no authentication required.
3. Computing `combineRandomValues` off-chain for each of the next 255 blocks — trivial computation.
4. Submitting `reveal()` at the chosen block — one transaction.

There is a clear financial incentive whenever the random number determines a valuable outcome. The `requestWithCallback` / `requestV2` paths are not affected (they hardcode `useBlockhash = false`), but the legacy `request()` path remains live in the deployed contract. [6](#0-5) 

---

### Recommendation

1. **Remove `useBlockhash` support entirely** from the legacy `request()` path, or deprecate and disable it. The `requestWithCallback` and `requestV2` paths already correctly set `useBlockhash = false` with the comment: *"If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude."*
2. If `useBlockhash` must be retained, prevent the user from supplying `providerContribution` themselves — require the provider to submit it in a separate transaction so the user cannot pre-compute outcomes.
3. Update the code comment at line 414–418 to accurately reflect that the user has 255 choices, not two.

---

### Proof of Concept

1. Attacker calls `request(provider, keccak256(x_U), true)` paying the required fee. Contract stores `req.blockNumber = N`, `req.useBlockhash = true`.
2. Attacker queries `GET https://<fortuna-uri>/v1/chains/<chain>/revelations/<sequenceNumber>` to obtain `x_i`.
3. Attacker computes off-chain: for each block `b` in `[N+1, N+255]`, compute `r_b = keccak256(x_U, x_i, blockhash(N))` (using a local node or public RPC to read future blockhashes as blocks are produced).
4. Attacker identifies block `b*` where `r_b*` is most favorable (e.g., `r_b* % 100 == 99` for a rare NFT trait).
5. Attacker calls `reveal(provider, sequenceNumber, x_U, x_i)` at block `b*`.
6. Contract computes `randomNumber = combineRandomValues(x_U, x_i, blockhash(N))` using the attacker-chosen blockhash, delivering the pre-selected favorable outcome. [7](#0-6)

    function revealHelper(
        EntropyStructsV2.Request storage req,
        bytes32 userContribution,
        bytes32 providerContribution
    ) internal returns (bytes32 randomNumber, bytes32 blockHash) {
        bytes32 providerCommitment = constructProviderCommitment(
            req.numHashes,
            providerContribution
        );
        bytes32 userCommitment = constructUserCommitment(userContribution);
        if (
            keccak256(bytes.concat(userCommitment, providerCommitment)) !=
            req.commitment
        ) revert EntropyErrors.IncorrectRevelation();

        blockHash = bytes32(uint256(0));
        if (req.useBlockhash) {
            bytes32 _blockHash = blockhash(req.blockNumber);

            // The `blockhash` function will return zero if the req.blockNumber is equal to the current
            // block number, or if it is not within the 256 most recent blocks. This allows the user to
            // select between two random numbers by executing the reveal function in the same block as the
            // request, or after 256 blocks. This gives each user two chances to get a favorable result on
            // each request.
            // Revert this transaction for when the blockHash is 0;
            if (_blockHash == bytes32(uint256(0)))
                revert EntropyErrors.BlockhashUnavailable();

            blockHash = _blockHash;
        }

        randomNumber = combineRandomValues(
            userContribution,
            providerContribution,
            blockHash
        );

    function reveal(
        address provider,
        uint64 sequenceNumber,
        bytes32 userContribution,
        bytes32 providerContribution
    ) public override returns (bytes32 randomNumber) {
        EntropyStructsV2.Request storage req = findActiveRequest(
            provider,
            sequenceNumber
        );

        if (
            req.callbackStatus != EntropyStatusConstants.CALLBACK_NOT_NECESSARY
        ) {
            revert EntropyErrors.InvalidRevealCall();
        }

        if (req.requester != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }
        bytes32 blockHash;
        (randomNumber, blockHash) = revealHelper(
            req,
            userContribution,
            providerContribution
        );
        emit Revealed(
            EntropyStructConverter.toV1Request(req),
            userContribution,
            providerContribution,
            blockHash,
            randomNumber
        );
        clearRequest(provider, sequenceNumber);
    }

    struct Request {
        // Storage slot 1 //
        address provider;
        uint64 sequenceNumber;
        // The number of hashes required to verify the provider revelation.
        uint32 numHashes;
        // Storage slot 2 //
        // The commitment is keccak256(userCommitment, providerCommitment). Storing the hash instead of both saves 20k gas by
        // eliminating 1 store.
        bytes32 commitment;
        // Storage slot 3 //
        // The number of the block where this request was created.
        // Note that we're using a uint64 such that we have an additional space for an address and other fields in
        // this storage slot. Although block.number returns a uint256, 64 bits should be plenty to index all of the
        // blocks ever generated.
        uint64 blockNumber;
        // The address that requested this random number.
        address requester;
        // If true, incorporate the blockhash of blockNumber into the generated random value.
        bool useBlockhash;
        // Status flag for requests with callbacks. See EntropyConstants for the possible values of this flag.
        uint8 callbackStatus;
        // The gasLimit in units of 10k gas. (i.e., 2 = 20k gas). We're using units of 10k in order to fit this
        // field into the remaining 2 bytes of this storage slot. The dynamic range here is 10k - 655M, which should
        // cover all real-world use cases.
        uint16 gasLimit10k;
    }
