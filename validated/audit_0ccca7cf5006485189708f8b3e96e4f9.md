Looking at the exact code paths in question.

### Title
Rich-Indexer Rollback Panics on Blocks with ≥1000 Transactions Due to Unbounded SQL Parameter List — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

`remove_batch_by_blobs` constructs a single `DELETE … WHERE id IN ($1, …, $N)` with every transaction ID bound as a separate SQL parameter. There is no chunking. SQLite's hard limit is 999 bound variables (`SQLITE_LIMIT_VARIABLE_NUMBER`). CKB consensus imposes no per-block transaction count limit — only a byte limit (`MAX_BLOCK_BYTES = 597,000`). A valid block can therefore contain well over 999 transactions. When such a block is later rolled back during a reorg, the DELETE fails, the error propagates to the sync loop which calls `.expect("rollback block should be OK")`, panicking the indexer thread. Because the DB transaction is never committed, the indexer tip is not decremented, and every subsequent restart attempts the same failing rollback — a permanent crash loop.

---

### Finding Description

**Root cause — no chunking in `remove_batch_by_blobs`:** [1](#0-0) 

`sqlx_param_placeholders(1..ids.len())` emits one `$N` placeholder per element with no upper-bound guard: [2](#0-1) 

`rollback_block` calls `remove_batch_by_blobs` five times with the full `tx_id_list`, and also calls `query_outputs_by_tx_id_list` which has the identical unbounded parameterization pattern: [3](#0-2) [4](#0-3) 

**Panic at the call site:**

The sync loop calls rollback with `.expect()`, so any `Err` return panics the thread: [5](#0-4) 

**No per-block transaction count limit in consensus:**

`MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT = 597 × 1000 = 597,000 bytes`. There is no separate cap on the number of transactions. A minimal CKB transaction is far smaller than 597 bytes, so a valid block can contain several thousand transactions: [6](#0-5) 

---

### Impact Explanation

When a reorg touches a block that contained ≥1000 transactions (SQLite) or ≥65535 (PostgreSQL), `remove_batch_by_blobs` returns `Error::DB("too many SQL variables")`. The DB transaction is rolled back by the driver, so the indexer tip is **not** decremented. The `.expect()` at the call site panics the sync thread. On every restart the indexer sees the same stale tip, detects the same reorg, attempts the same failing rollback, and panics again — a **permanent crash loop** that renders the rich-indexer permanently non-functional until a manual database repair. [7](#0-6) 

---

### Likelihood Explanation

On mainnet, blocks with 1000+ transactions are uncommon today but are entirely valid per consensus and have been observed on high-throughput testnets. No malicious intent is required: any miner who fills a block with many small transactions (e.g., simple transfer transactions) can produce such a block. A reorg — a normal network event — then triggers the crash loop. The vulnerability requires no privileged access, no majority hashpower, and no social engineering.

---

### Recommendation

Chunk `ids` into batches of at most 999 (SQLite) or a safe constant shared across both backends before calling `remove_batch_by_blobs` and `query_outputs_by_tx_id_list`. For example:

```rust
const MAX_PARAMS: usize = 999; // safe for both SQLite and PostgreSQL

for chunk in ids.chunks(MAX_PARAMS) {
    remove_batch_by_blobs_inner(table_name, column_name, chunk, tx).await?;
}
```

Apply the same chunking to `query_outputs_by_tx_id_list`. Additionally, replace the `.expect()` in `try_loop_sync` with proper error handling so a transient DB failure does not panic the sync thread. [5](#0-4) 

---

### Proof of Concept

1. Start a CKB devnet node with the rich-indexer enabled (SQLite backend).
2. Mine a block containing 1000 minimal transactions (each ~60 bytes; well within `MAX_BLOCK_BYTES = 597,000`).
3. Let the indexer append the block successfully.
4. Mine an alternative chain of length 2 from the parent of that block to trigger a reorg.
5. Observe: the indexer sync thread panics with `"too many SQL variables"` (SQLite error code `SQLITE_ERROR`), the indexer tip remains at the rolled-back block's height, and every subsequent restart reproduces the panic. [8](#0-7)

async fn remove_batch_by_blobs(
    table_name: &str,
    column_name: &str,
    ids: &[i64],
    tx: &mut Transaction<'_, Any>,
) -> Result<(), Error> {
    if ids.is_empty() {
        return Ok(());
    }

    // build query str
    let mut query_builder = SqlBuilder::delete_from(table_name);
    let sql = query_builder
        .and_where_in(column_name, &sqlx_param_placeholders(1..ids.len())?)
        .sql()
        .map_err(|err| Error::DB(err.to_string()))?;

    // bind
    let mut query: sqlx::query::Query<'_, Any, sqlx::any::AnyArguments<'_>> = sqlx::query(&sql);
    for hash in ids {
        query = query.bind(hash);
    }

    // execute
    query
        .execute(tx.as_mut())
        .await
        .map_err(|err| Error::DB(err.to_string()))?;

    Ok(())
}
