### Title
Unbounded DB Read Amplification in `GetLastStateProofProcess::execute` via Crafted `difficulties` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged light-client peer can send a `GetLastStateProof` message containing up to 999 `difficulties` values (just under the `GET_LAST_STATE_PROOF_LIMIT` of 1000). For each difficulty, the server performs a full binary-search over the chain via `get_first_block_total_difficulty_is_not_less_than`, issuing 2 DB reads per binary-search step. On a long chain (e.g., 1 million blocks), a single crafted request triggers approximately **20,000 DB reads**, with no per-request I/O budget enforced.

---

### Finding Description

**Entrypoint**: Any peer connected on the `LightClient` P2P protocol can send a `GetLastStateProof` message. There is no authentication or privilege requirement. [1](#0-0) 

**The limit check** at the top of `execute()` only bounds the count of difficulties plus twice `last_n_blocks`: [2](#0-1) 

With `last_n_blocks = 0` and 999 difficulties, `999 + 0 = 999 < 1000`, so the check passes.

**The binary search**: For each difficulty, `get_block_numbers_via_difficulties` calls `get_first_block_total_difficulty_is_not_less_than`, which performs a binary search over `[start_block_number, end_block_number)`: [3](#0-2) 

Each loop iteration calls `get_block_total_difficulty`, which issues **2 DB reads** (one `get_block_hash`, one `get_block_ext`): [4](#0-3) 

**The range-narrowing does not help for adversarial inputs**: After finding block `num` for difficulty `d_i`, the code sets `start_block_number = num - 1`: [5](#0-4) 

If the attacker crafts difficulties that each map to consecutive blocks near the start of the chain (e.g., blocks 1, 2, 3, …, 999), then after finding block `num = k`, `start_block_number` becomes `k - 1`. The next search range is `[k-1, 1M)`, which is still nearly the full chain. The narrowing is negligible.

**No rate limiting exists**: `lib.rs` contains no per-peer request frequency limit, no per-request DB read budget, and no async yield between binary-search iterations. The only protection is `BAD_MESSAGE_BAN_TIME` for malformed messages, which does not apply here since the message is structurally valid. [6](#0-5) 

---

### Impact Explanation

On a 1M-block chain:
- Each binary search: `log2(1,000,000) ≈ 20` iterations × 2 DB reads = **~40 DB reads**
- 999 difficulties × 40 DB reads = **~40,000 DB reads per request**
- A single peer sending requests in a tight loop (no ban is triggered for valid messages) can sustain this indefinitely

This saturates the RocksDB I/O path, starving block validation, sync, and other node operations of disk bandwidth. The impact is **I/O exhaustion / effective denial of service** on the full node serving the light-client protocol.

---

### Likelihood Explanation

- No PoW, no key, no privileged role required — any peer on the `LightClient` protocol suffices
- The message is structurally valid and passes all input validation checks
- The attacker only needs to know a valid main-chain block hash (publicly observable) and craft monotonically increasing difficulty values
- No ban is triggered for a well-formed request that returns `Status::ok()`

---

### Recommendation

1. **Cap total binary-search steps per request**, not just the count of difficulties. A budget of, e.g., `MAX_DIFFICULTIES * ceil(log2(MAX_CHAIN_LENGTH))` steps should be enforced across the entire `get_block_numbers_via_difficulties` call.
2. **Alternatively**, require that the `difficulties` slice be sparse enough relative to the chain length (e.g., enforce a minimum gap between consecutive difficulty values proportional to the chain's difficulty growth rate).
3. **Add per-peer request rate limiting** in `LightClientProtocol::received` to bound the sustained request rate from any single peer.

---

### Proof of Concept

```
1. Connect to a CKB full node with the LightClient protocol enabled.
2. Obtain any valid main-chain tip block hash H (e.g., from GetLastState).
3. Read the total difficulty D_tip of the tip block.
4. Craft a GetLastStateProof message:
     last_hash         = H  (valid main-chain hash)
     start_hash        = genesis hash
     start_number      = 0
     last_n_blocks     = 0
     difficulty_boundary = D_tip
     difficulties      = [D_1, D_2, ..., D_999]
       where D_i = total_difficulty(block i) + epsilon
       (i.e., 999 values each mapping to a distinct block near the start of the chain)
5. Send the message. The server will execute 999 binary searches over ~1M blocks.
6. Instrument get_block_total_difficulty call count: expect ~40,000 calls per request.
7. Repeat in a loop; observe RocksDB read latency spike and node I/O saturation.
```
