Audit Report

## Title
`load_data_as_code` Charges Cycles on `memory_size` Instead of `content_size`, Enabling Cheap Verify-Worker Saturation — (`script/src/syscalls/load_cell_data.rs`)

## Summary

`load_data_as_code` (syscall 2091) charges `transferred_byte_cycles(memory_size)` after a successful `init_pages` call, where `memory_size` is a fully attacker-controlled register value bounded only by the VM address space. An unprivileged script author can set `memory_size` far above `content_size` and consume tens of millions of cycles per syscall, pinning all tx-pool verify workers at near-cycle-limit execution with a single cheap, reusable cell dep.

## Finding Description

In `load_data_as_code`, the cycle charge at line 163 uses `memory_size` (register A1), not `content_size` (register A3):

```rust
// script/src/syscalls/load_cell_data.rs  lines 148–163
if content_offset >= cell.len() as u64
    || content_end > cell.len() as u64
    || content_size > memory_size          // only guard: content_size ≤ memory_size
{
    machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
    return Ok(());
}
machine.memory_mut().init_pages(addr, memory_size, FLAG_EXECUTABLE | FLAG_FREEZED,
    Some(cell.slice(content_offset..content_end)), 0)?;
sc.track_pages(machine, addr, memory_size, &data_piece_id, content_offset)?;
machine.add_cycles_no_checking(transferred_byte_cycles(memory_size))?;  // memory_size, not content_size
``` [1](#0-0) 

The three guards at lines 148–150 only enforce that `content_offset` and `content_end` are within the cell's actual data, and that `content_size ≤ memory_size`. There is no upper bound on `memory_size` itself. An attacker sets `content_size = 1` (matching a 1-byte cell dep) and `memory_size = 60 MB`; all three guards pass, `init_pages` succeeds, and the cycle charge is:

```
transferred_byte_cycles(60 MB) = 60 × 1024 × 1024 / 4 = 15,728,640 cycles
``` [2](#0-1) 

`DEFAULT_MAX_TX_VERIFY_CYCLES = TWO_IN_TWO_OUT_CYCLES × 20 = 3,500,000 × 20 = 70,000,000`. [3](#0-2) [4](#0-3) 

Four non-overlapping calls with `memory_size = 17.5 MB` each exhaust the full 70 M cycle budget. The attacker tunes `addr` and `memory_size` to stay within the VM's usable address space and avoid page overlap; `init_pages` returns `Ok` each time, so the cycle charge is always applied before the script is rejected for exceeding the limit.

## Impact Explanation

This matches **High: "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

The verify manager spawns `max_tx_verify_workers` workers, each processing one transaction at a time. [5](#0-4) 

Submitting `N ≥ max_tx_verify_workers` crafted transactions simultaneously pins all workers at near-cycle-limit execution. Each worker must run `init_pages` (real page-initialization proportional to `memory_size`) before the cycle limit is hit and the transaction is rejected. Legitimate transactions queue behind them indefinitely as long as the attacker keeps flooding.

## Likelihood Explanation

- **Unprivileged**: Any user can submit transactions via RPC or P2P relay.
- **Cheap**: A 1-byte cell dep (minimum ~61 CKB capacity, reusable across unlimited transactions) plus `min_fee_rate × tx_size ≈ 300 shannons` per transaction. No fee-per-cycle mechanism exists in the tx-pool.
- **Repeatable**: The same cell dep can be referenced by unlimited transactions simultaneously.
- **No PoW or consensus bypass needed**: The attack operates entirely within the tx-pool admission path.

## Recommendation

Charge cycles based on `content_size` (actual bytes transferred into the VM), not `memory_size` (virtual address range reserved). The zero-padding of `memory_size - content_size` bytes is a VM-internal page-initialization cost and must not be exposed as a user-controllable cycle multiplier.

```rust
// script/src/syscalls/load_cell_data.rs  line 163
machine.add_cycles_no_checking(transferred_byte_cycles(content_size))?;
``` [6](#0-5) 

Alternatively, cap `memory_size` at `content_size` rounded up to the next page boundary before passing it to `init_pages`.

## Proof of Concept

1. Deploy a cell dep with exactly 1 byte of data (`0x01`).
2. Write a minimal RISC-V ELF script that:
   - Sets `A0 = 0x20000` (addr, free VM space above ELF load address)
   - Sets `A1 = 17_500_000` (memory_size ≈ 17.5 MB)
   - Sets `A2 = 0` (content_offset)
   - Sets `A3 = 1` (content_size = 1 byte)
   - Sets `A4 = 0` (index)
   - Sets `A5 = source` (CellDep)
   - Sets `A7 = 2091` (`LOAD_CELL_DATA_AS_CODE_SYSCALL_NUMBER`)
   - Issues `ecall`; repeats with non-overlapping `addr` values four times
3. Confirm: `transferred_byte_cycles(17_500_000) × 4 ≈ 70,000,000` cycles charged, hitting `DEFAULT_MAX_TX_VERIFY_CYCLES`.
4. Submit `N ≥ max_tx_verify_workers` such transactions simultaneously.
5. Observe: all verify workers are pinned at near-cycle-limit; `verify_queue_size` grows; legitimate transactions stall.
