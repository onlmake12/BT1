Audit Report

## Title
Remote Peer Can Inflate `task_count` to 128 and Indefinitely Lock 128 Block Slots in `InflightBlocks` Beyond `tip + 20` — (`sync/src/types/mod.rs`)

## Summary

An unprivileged remote sync peer can inflate its `DownloadScheduler::task_count` to `MAX_BLOCKS_IN_TRANSIT_PER_PEER=128` by responding to block requests within the `fast_time` threshold (default 1000 ms), then stop responding while keeping the TCP connection alive. This locks 128 `BlockNumberAndHash` entries in `InflightBlocks::inflight_states`, preventing any other peer from being asked for those same blocks. For entries with block numbers beyond `tip + 20`, the `prune` function's early-break means the 30-second eviction never fires, making the lock persist until the verified tip advances past those block numbers — unbounded during IBD.

## Finding Description

**Inflation path (`remove_by_block`):**

Each time a block is received, `remove_by_block` is called. It measures elapsed time and calls `time_analyzer.push_time(elapsed)`. When elapsed ≤ `fast_time` (default **1000 ms**), `TimeQuantile::MinToFast` is returned and `set.increase(2)` is called unconditionally — there is no `should_punish` guard on this branch, unlike the `NormalToUpper` and `UpperToMax` decrease branches which are both gated on `should_punish`. [9](#0-8) 

`increase` adds up to `MAX_BLOCKS_IN_TRANSIT_PER_PEER=128`: [3](#0-2) 

Starting from `INIT_BLOCKS_IN_TRANSIT_PER_PEER=32`, the gap to 128 is 96. At +2 per fast response, only **48 sub-1000ms block deliveries** are required. [10](#0-9) 

**Slot locking (`insert`):**

`InflightBlocks::insert` uses a global `BTreeMap<BlockNumberAndHash, InflightState>`. If the key is already occupied it returns `false` immediately: [5](#0-4) 

`BlockFetcher::fetch` only adds a block to the fetch list when `insert` returns `true`; a `false` return silently skips the block with no fallback to another peer: [6](#0-5) 

**`prune` timeout bypass:**

`prune` iterates `inflight_states` (a `BTreeMap` sorted by `BlockNumberAndHash`) and breaks as soon as `key.number > tip + 20`. Entries with block numbers beyond `tip + 20` are **never reached** by the eviction loop: [4](#0-3) 

`BLOCK_DOWNLOAD_TIMEOUT=30s` therefore only applies to entries within `tip + 20`. During IBD, where the victim requests blocks far ahead of the verified tip, locked slots persist until the verified tip advances past those block numbers: [11](#0-10) 

**`can_fetch` exhaustion:**

With `task_count=128` and 128 hashes in-flight, `can_fetch` returns 0, causing `reached_inflight_limit` to return `true` and `BlockFetcher::fetch` to return `None` immediately for that peer: [12](#0-11) [13](#0-12) 

**Existing mitigations are insufficient:**

`remove_by_peer` frees all slots on disconnect, but `CHAIN_SYNC_TIMEOUT` is 12 minutes — a peer that stays connected but stops sending blocks is not disconnected within the relevant timeframe: [14](#0-13) 

The `should_punish` guard that protects the decrease branches is absent from the `increase` branch, so even nodes with few peers (`should_punish=false`) are fully vulnerable to inflation: [15](#0-14) 

`time_analyzer` is a single instance shared across all peers inside `InflightBlocks`, so a fast attacker shifts the `fast_time` baseline for all peers: [7](#0-6) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single unprivileged peer can stall synchronization of up to 128 specific block slots. For blocks beyond `tip + 20` (the common case during IBD, where the verified tip lags far behind the download window), the stall is not bounded by the 30-second timeout and persists until the verified tip advances. Multiple coordinated peers covering disjoint block ranges can amplify the effect across the entire download window (`BLOCK_DOWNLOAD_WINDOW = 8192`), proportionally degrading the victim node's block sync throughput.

## Likelihood Explanation

The attacker requires only a standard P2P sync connection and the ability to serve valid blocks within 1000 ms — trivially achievable on any reasonable network. No PoW, key material, or privileged access is needed. The 48-response inflation phase is trivially achievable. The attack is repeatable: after `prune` eventually evicts entries within `tip + 20`, the attacker can reconnect and repeat. The `should_punish` guard being absent from the `increase` branch means the attack works even against nodes with few peers.

## Recommendation

1. Apply a `should_punish`-equivalent guard (or a per-peer rate limit) to the `increase` branch in `remove_by_block`, preventing inflation when the peer count is below the protection threshold.
2. Make `time_analyzer` per-peer rather than a single shared instance inside `InflightBlocks`, so one fast peer cannot shift the quantile thresholds for all peers.
3. Extend `prune` to evict timed-out entries beyond `tip + 20`, or introduce a secondary per-entry absolute deadline enforced regardless of block number relative to tip.
4. On detecting a peer with `can_fetch == 0` and no recent block deliveries, proactively re-request its inflight blocks from alternative peers after a shorter interval (e.g., `fast_time * 3`) rather than waiting for the full `BLOCK_DOWNLOAD_TIMEOUT`.

## Proof of Concept

1. Connect a malicious peer; advertise a `best_known_header` ahead of the victim's verified tip by more than 20 blocks (to exercise the `prune` bypass).
2. Respond to every `GetBlocks` message within <1000 ms with valid blocks. After 48 responses, assert `task_count == 128` via the `get_peers` RPC (`can_fetch_count` field).
3. Once the victim has 128 blocks inflight to the malicious peer, stop sending block responses (keep the TCP connection alive).
4. Assert via `get_peers` that `inflight_count == 128` and `can_fetch_count == 0` for the malicious peer.
5. Assert that honest peers are not asked for those same 128 block hashes: `InflightBlocks::insert` returns `false` for each, so `BlockFetcher::fetch` skips them.
6. For entries with block numbers > `tip + 20`: assert they remain in `inflight_states` after 30 seconds have elapsed (i.e., `prune` does not evict them), confirming the indefinite lock.
7. Disconnect the malicious peer; assert that `remove_by_peer` immediately frees all 128 slots and honest peers can resume fetching.
