### Title
Stale `total_tx_size`/`total_tx_cycles` After Ancestor-Eviction in `add_entry` Causes Tx-Pool Over-Eviction — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new pool-size statistics are computed **before** `check_and_record_ancestors` runs. When that function evicts existing entries, `update_stat_for_remove_tx` correctly decrements `self.total_tx_size`/`self.total_tx_cycles`. However, lines 218–219 then **overwrite** those correctly-updated values with the stale pre-eviction snapshot, permanently inflating the tracked totals by the size and cycles of every evicted entry. The downstream `limit_size` loop uses the inflated `total_tx_size` to decide whether to keep evicting, causing it to expel more valid transactions than the pool's actual occupancy warrants.

---

### Finding Description

`PoolMap::add_entry` (`tx-pool/src/component/pool_map.rs`, lines 200–221):

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    ...
    // Step 1 – snapshot pre-eviction totals + new entry
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211

    // Step 2 – may evict entries; each eviction calls update_stat_for_remove_tx,
    //           which correctly decrements self.total_tx_size / self.total_tx_cycles
    evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

    ...
    // Step 3 – OVERWRITES the correctly-updated values with the stale snapshot
    self.total_tx_size  = total_tx_size;                            // line 218
    self.total_tx_cycles = total_tx_cycles;                         // line 219
    Ok((true, evicts))
}
```

`check_and_record_ancestors` (lines 588–640) enters the eviction branch when:

1. `ancestors_count > max_ancestors_count` (default 25), **and**
2. `ancestors_count − cell_ref_parents.len() ≤ max_ancestors_count`

Inside that branch it calls `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly subtracts the evicted entries' sizes/cycles from `self.total_tx_size`/`self.total_tx_cycles`. But the final assignment at lines 218–219 discards those corrections.

**Concrete arithmetic:**

| Variable | Before eviction | After `update_stat_for_remove_tx` (correct) | After line 218-219 (bug) |
|---|---|---|---|
| `self.total_tx_size` | T | T − E | T + S (inflated by E) |

Where `T` = pre-eviction total, `E` = sum of evicted sizes, `S` = new entry size.

The inflated `total_tx_size` is then consumed by `limit_size` (`tx-pool/src/pool.rs`, line 298):

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evicts more entries than necessary
}
```

Each trigger of the eviction path in `add_entry` permanently inflates `total_tx_size` by `E`. Repeated triggering accumulates inflation, causing `limit_size` to continuously over-evict valid transactions.

---

### Impact Explanation

- **Tx-pool over-eviction / DoS**: `limit_size` uses the inflated counter to decide whether to keep removing entries. Valid, high-fee transactions are expelled from the pool even when actual occupancy is well below `max_tx_pool_size`.
- **Incorrect RPC reporting**: `tx_pool_info` exposes `total_tx_size` and `total_tx_cycles` directly; both become permanently wrong, misleading fee-estimation and monitoring.
- **Cascading inflation**: Each successful trigger of the eviction path in `check_and_record_ancestors` adds another `E` to the inflation. An attacker who can repeatedly trigger this path can drive `total_tx_size` arbitrarily high, causing the pool to evict every incoming transaction via `limit_size`.

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is reachable by any unprivileged tx-pool submitter. The required setup:

1. Submit a chain of 25 transactions (tx₁ → tx₂ → … → tx₂₅) so that a 26th transaction would exceed `max_ancestors_count`.
2. Submit a separate transaction `tx_dep` that uses tx₁'s output cell as a **cell dep** (not an input). This makes `tx_dep` a `cell_ref_parent` of any future transaction that spends tx₁'s output.
3. Submit a new transaction that spends tx₂₅'s output. Its ancestor count is 26 (> 25), but `cell_ref_parents = {tx_dep}`, so `26 − 1 = 25 ≤ max_ancestors_count`. The eviction branch fires, evicts `tx_dep`, and inflates `total_tx_size` by `tx_dep.size`.

This is a standard CKB cell-dep pattern, requires no privileged access, no keys, and no majority hashpower. The attacker only needs to submit valid transactions via the public RPC (`send_transaction`).

---

### Recommendation

Move the `updated_stat_for_add_tx` call to **after** `check_and_record_ancestors` completes, so the snapshot is taken from the already-eviction-adjusted state:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }

    // Perform evictions first so self.total_tx_size/cycles are up-to-date
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Now compute and assign totals against the post-eviction baseline
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size  = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, after evictions complete, re-read `self.total_tx_size` directly instead of using the pre-computed snapshot.

### Proof of Concept

**State before attack:**
- Pool contains tx₁…tx₂₅ (chain) + `tx_dep` (uses tx₁ output as cell dep)
- `total_tx_size = T` (actual sum of all entries)

**Attack step — submit `tx_new` spending tx₂₅ output:**

1. `updated_stat_for_add_tx(tx_new.size, tx_new.cycles)` → `total_tx_size_snapshot = T + tx_new.size`
2. `check_and_record_ancestors` detects 26 ancestors, `cell_ref_parents = {tx_dep}`, enters eviction branch
3. `remove_entry_and_descendants(tx_dep)` → `update_stat_for_remove_tx(tx_dep.size, tx_dep.cycles)` → `self.total_tx_size = T − tx_dep.size`
4. Lines 218–219: `self.total_tx_size = T + tx_new.size` ← stale snapshot, **ignores the subtraction**

**Result:** `self.total_tx_size = T + tx_new.size` instead of the correct `T − tx_dep.size + tx_new.size`.

**Inflation per trigger:** `tx_dep.size` (attacker-controlled; can be up to `TRANSACTION_SIZE_LIMIT = 512 KB`).

**Downstream effect:** `limit_size` at `tx-pool/src/pool.rs:298` sees `total_tx_size > max_tx_pool_size` and evicts additional valid transactions. Repeating the attack accumulates inflation until the pool continuously evicts every new submission. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);

                }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    // Remove transactions from the pool until total size <= size_limit.
    // Return a `Reject` for current inserting entry if it's removed
    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
