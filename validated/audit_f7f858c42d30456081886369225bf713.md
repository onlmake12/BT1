Audit Report

## Title
Integer Overflow in Guard Bypasses Limit, Enabling Unbounded Vec Allocation and DB Lookup DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary

`GetLastStateProofProcess::execute` contains a guard at line 201 intended to cap per-request work at `GET_LAST_STATE_PROOF_LIMIT = 1000`. The expression `(last_n_blocks as usize) * 2` wraps to `0` in a Rust release build when `last_n_blocks = 2^63`, silently bypassing the check. After the guard, the reorg branch at lines 237–247 allocates a `Vec<u64>` of up to `start_block_number` elements (bounded only by chain height), then passes it into `complete_headers` which performs one `get_ancestor` + `get_block` + `chain_root_mmr` DB lookup per element — millions of synchronous lookups per request.

## Finding Description

**Root cause — overflow in guard (line 201):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
```

`last_n_blocks` is decoded from the wire as a full `Uint64` field (schema line 336). On a 64-bit host `usize` is 64 bits. With `last_n_blocks = 2^63`:

- `2^63 as usize` = `2^63` (lossless cast)
- `2^63 * 2` = `2^64` — overflows `usize` (max = `2^64 − 1`)

Rust release builds do not check integer overflow by default; the result wraps to `0`. The guard becomes `0 + 0 > 1000` → `false`. The check is silently bypassed.

**Unbounded allocation — reorg path (lines 237–247):**

```rust
let reorg_last_n_numbers = if start_block_number == 0
    || snapshot.get_ancestor(&last_block_hash, start_block_number)
        .map(|header| header.hash() == start_block_hash)
        .unwrap_or(false)
{
    Vec::new()
} else {
    let min_block_number = start_block_number - min(start_block_number, last_n_blocks);
    (min_block_number..start_block_number).collect()   // ← unbounded
};
```

With `last_n_blocks = 2^63` and `start_block_number = N` (any valid block number ≤ chain tip):

- `min(N, 2^63) = N`
- `min_block_number = N − N = 0`
- `(0..N).collect::<Vec<u64>>()` allocates **N elements**

The only prior bound on `start_block_number` is the check at lines 231–235 that it must not exceed `last_block_number`. On CKB mainnet (~14 M blocks), this yields a ~112 MB Vec per request.

**Downstream cost — `complete_headers` (lines 350–365):**

`reorg_last_n_numbers` is chained into `block_numbers` and passed to `complete_headers`, which performs `get_ancestor` + `get_block` + `chain_root_mmr` for every element — up to 14 M sequential DB lookups per request.

**Existing checks are insufficient:**

- The guard at line 201 is the only size check on `last_n_blocks`; it is defeated by the overflow.
- The `start_block_number ≤ last_block_number` check at line 231 only bounds the allocation to chain height, not to the protocol limit.

## Impact Explanation

A single crafted `GetLastStateProof` P2P message causes the light-client protocol server to allocate hundreds of MB of heap and perform millions of synchronous DB lookups. Repeated requests from one or more peers exhaust heap memory (OOM kill) or saturate the async executor, crashing or permanently stalling the light-client server. Because the light-client server runs in the same process as the CKB node, an OOM kill terminates the entire node. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The attack requires no authentication, no PoW, and no privileged role. The attacker needs only:

1. A valid `last_hash` — trivially obtained from any public block explorer or full node.
2. `start_block_number` set to any large valid block number ≤ chain tip.
3. `start_hash` set to any value that does not match the ancestor at `start_block_number` (e.g., `[0u8; 32]`) to force the reorg branch.
4. `last_n_blocks = (u64::MAX / 2) + 1` (= `2^63`) — a single field in the packed message.
5. `difficulties = []` — empty vector.

Any peer that can open a light-client protocol connection can trigger this. The attack is repeatable at will.

## Recommendation

Replace the overflowing multiplication with a saturating operation:

```rust
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, add an explicit early cap on `last_n_blocks` before the reorg path:

```rust
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("last_n_blocks too large");
}
```

This ensures `reorg_last_n_numbers` is always bounded by `GET_LAST_STATE_PROOF_LIMIT` regardless of the reorg condition.

## Proof of Concept

```rust
// Craft a GetLastStateProof message with:
//   last_hash     = current tip hash (valid main-chain block, from any explorer)
//   start_hash    = [0u8; 32]  (wrong hash → triggers reorg branch)
//   start_number  = tip_number (e.g. 14_000_000 on mainnet)
//   last_n_blocks = (u64::MAX / 2) + 1  // = 2^63; causes (usize)*2 to wrap to 0
//   difficulty_boundary = any valid value
//   difficulties  = []  // empty → difficulties().len() = 0

let content = packed::GetLastStateProof::new_builder()
    .last_hash(tip_hash)
    .start_hash(packed::Byte32::default())
    .start_number(tip_number.pack())
    .last_n_blocks(((u64::MAX / 2) + 1).pack())
    .difficulty_boundary(U256::zero().pack())
    .build();
```
