**Key findings from the code:**

1. `deposit_fee_coins` in `price_info.move` only ever adds/joins coins into the `FEE_STORAGE_KEY` dynamic field — there is no corresponding withdrawal function anywhere in the module. [1](#0-0) 

2. `update_single_price_feed` in `pyth.move` deposits the fee into the `PriceInfoObject` and never routes it to `fee_recipient_address`. [2](#0-1) 

3. `set_fee_recipient.move` explicitly states the old fee-forwarding mechanism is gone. The `execute` function only updates the `fee_recipient_address` field in `State` — it does not transfer any coins from `PriceInfoObject`s. [3](#0-2) 

4. `State` still stores `fee_recipient_address` as a field, and `set_fee_recipient` in `state.move` only writes to that field — no coin transfer occurs. [4](#0-3) 

5. Exhaustive search of all public/entry functions across all production `.move` files in `target_chains/sui/contracts/sources/` reveals **zero** functions that read `FEE_STORAGE_KEY` from a `PriceInfoObject` and transfer or withdraw those coins anywhere.

### Title
Protocol Fee Coins Permanently Locked in `PriceInfoObject` Shared Objects With No On-Chain Withdrawal Path — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every call to `pyth::update_single_price_feed` deposits the caller-supplied SUI fee coin into the target `PriceInfoObject` via `price_info::deposit_fee_coins`. No public, friend, or entry function in the entire production contract surface ever withdraws or transfers those coins to `fee_recipient_address` or any other address. The `set_fee_recipient` governance module explicitly acknowledges the old forwarding mechanism was removed, leaving `fee_recipient_address` as a vestigial, write-only field. All accumulated SUI fees are permanently locked.

### Finding Description
`price_info::deposit_fee_coins` stores fee coins as a dynamic object field (`FEE_STORAGE_KEY`) on the shared `PriceInfoObject`: [1](#0-0) 

`update_single_price_feed` calls this unconditionally after the fee sufficiency check: [2](#0-1) 

The `set_fee_recipient` module comment reads: *"The previous version of the contract sent the fees to a recipient address but this state is not used anymore."* Its `execute` function only writes to `state.fee_recipient_address` — no coin movement: [3](#0-2) 

`state::set_fee_recipient` confirms: it only mutates the address field: [4](#0-3) 

There is no `withdraw_fee_coins`, no `collect_fees`, and no transfer call anywhere in the production source tree that touches `FEE_STORAGE_KEY`.

### Impact Explanation
Every SUI coin paid as a protocol update fee is permanently locked inside the corresponding `PriceInfoObject` shared object. The Pyth fee recipient (treasury) can never collect this revenue. The coins are not burned — they exist on-chain but are inaccessible to any party. Over time, as price feeds are updated repeatedly, the locked amount grows proportionally to `N * base_update_fee` per feed. This matches the scope criterion of **permanent locking of funds**.

### Likelihood Explanation
This is triggered by the normal, intended operation of the protocol. Every legitimate price update call by any relayer — privileged or not — contributes to the locked balance. No exploit setup is required; the condition is already active on mainnet for every update that has ever been submitted with `base_update_fee > 0`.

### Recommendation
Implement a `withdraw_fees` entry function (gated by `LatestOnly` or a governance capability) that reads the `FEE_STORAGE_KEY` dynamic field from a `PriceInfoObject`, removes it, and transfers the coin to `state::get_fee_recipient(pyth_state)`. Alternatively, route fees directly to `fee_recipient_address` inside `update_single_price_feed` instead of depositing them into the `PriceInfoObject`, restoring the behavior described in the `set_fee_recipient` comment.
