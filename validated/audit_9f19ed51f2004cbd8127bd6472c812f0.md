Audit Report

## Title
Stale `descendants_*` Fields After `remove_entry_and_descendants` Corrupt Eviction Index — (`tx-pool/src/component/pool_map.rs`)

## Summary

`remove_entry_and_descendants` pre-strips all link entries for every transaction in the removed set before calling `remove_entry` on each one. Because `update_ancestors_index_key` relies on `calc_ancestors` to find which remaining pool entries to update, and `calc_ancestors` returns an empty set once the link entry is gone, ancestors outside the removed set never have their `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` fields decremented. The stale, inflated values are baked into those entries' `EvictKey`, shielding them from eviction and causing legitimate higher-fee-rate transactions to be evicted in their place.

## Finding Description

`remove_entry_and_descendants` first collects the root and all its descendants, then calls `remove_entry_links` for every entry in that set before any `remove_entry` call: [1](#0-0) 

The comment on L256 reveals the intent: pre-removing links prevents `update_descendants_index_key` from wastefully updating entries that are themselves being removed. However, this has a fatal side effect on `update_ancestors_index_key`.

`remove_entry_links` removes the entry's own record from `self.links` via `self.links.remove(id)`: [2](#0-1) 

Only after all links are stripped does the loop call `remove_entry` for each ID. Inside `remove_entry`, `update_ancestors_index_key` is called: [3](#0-2) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`: [4](#0-3) 

`calc_ancestors` delegates to `calc_relative_ids`, which does `self.inner.get(short_id).map(...).unwrap_or_default()`: [5](#0-4) 

Because `remove_entry_links` already called `self.links.remove(id)` for the root, `self.inner.get(short_id)` returns `None`, `unwrap_or_default()` yields an empty set, and the loop over ancestors is never entered. No ancestor outside the removed set has `sub_descendant_weight` called on it, and no ancestor's `evict_key` is recomputed.

`resolve_conflict` — triggered by both RBF replacement and committed-block conflict resolution — calls `remove_entry_and_descendants` directly: [6](#0-5) 

After `resolve_conflict` returns, every pool entry that was a parent of the removed subtree retains the removed children's fee/size/cycle contribution in its `descendants_*` fields and in its indexed `evict_key`.

The `EvictKey` is computed as `max(descendants_feerate, own_feerate)`: [7](#0-6) 

A stale, inflated `descendants_feerate` raises the effective `fee_rate` in the eviction index, pushing the entry toward the "do not evict" end of `iter_by_evict_key()`.

## Impact Explanation

When the pool reaches its size limit, `next_evict_entry` iterates `iter_by_evict_key()` in ascending order and evicts the entry with the lowest `fee_rate`: [8](#0-7) 

A parent transaction whose `descendants_feerate` is stale-inflated will be ranked above its true position in the eviction order. It will not be evicted even when the pool is full and its actual fee rate is below the eviction threshold. Legitimate higher-fee-rate transactions submitted by other users will be evicted in its place. An attacker can repeat this cheaply across many nodes to keep low-fee transactions permanently resident in the pool, displacing legitimate traffic and causing **CKB network congestion with few costs** — matching the High impact class (10001–15000 points).

## Likelihood Explanation

The attack requires only three submitted transactions (parent P, child C, RBF replacement R) via the standard `send_transaction` RPC, available to any unprivileged caller. RBF is explicitly enabled when `min_rbf_rate > min_fee_rate`: [9](#0-8) 

The attacker pays fees for C but recovers C's UTXO via R, so the net cost is only the incremental RBF fee. The stale state persists until P is committed or the node restarts, and the attack is trivially repeatable across many nodes simultaneously.

## Recommendation

In `remove_entry_and_descendants`, update the ancestors of the root entry **before** stripping any links:

1. Before the `remove_entry_links` loop, compute the set of ancestors of `id` that are **not** in `removed_ids`.
2. For each such ancestor, call `sub_descendant_weight` for every entry being removed that is a descendant of that ancestor, and recompute its `evict_key`.
3. Only then proceed with `remove_entry_links` and `remove_entry` for the removed set.

Alternatively, restructure `remove_entry_and_descendants` to only remove intra-set edges (between entries all being removed), preserving the root-to-parent edge long enough for `update_ancestors_index_key` to traverse it correctly before the link is torn down.

## Proof of Concept

1. Submit parent transaction **P** with fee rate just above `min_fee_rate` (e.g., 1 001 shannons/vByte).
2. Submit child transaction **C** spending an output of P, with a very high fee rate (e.g., 1 000 000 shannons/vByte). `add_descendant_weight` is called on P; P's `descendants_fee` and `descendants_feerate` become very large.
3. Submit replacement transaction **R** spending the same input as C with fee ≥ `min_replace_fee(C)`. `resolve_conflict` → `remove_entry_and_descendants(C_id)` is called. Because C's link entry is stripped before `remove_entry` runs, `calc_ancestors(C_id)` returns ∅ and P's `descendants_*` fields are never decremented.
4. Inspect P's pool entry: `descendants_fee` = C's fee (stale), `descendants_count` = 2 (stale), `evict_key.fee_rate` = stale high `descendants_feerate`.
5. Fill the pool with transactions whose actual fee rate is between P's true rate and the stale rate. `limit_size` evicts those transactions instead of P, because P's `evict_key` ranks it as more valuable.
6. P persists in the pool indefinitely, occupying space and displacing legitimate transactions, at a cost to the attacker of only the incremental RBF fee for R.

A unit test can confirm this by asserting that after step 3, `pool_map.get(P_id).unwrap().descendants_count == 1` and `descendants_fee == P.fee` — both assertions will currently fail, demonstrating the stale state.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
