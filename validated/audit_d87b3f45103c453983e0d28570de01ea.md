### Title
Unauthenticated Miner Block-Template Notification Endpoint Allows Any HTTP Client to Redirect Mining Rewards — (`miner/src/client.rs`)

### Summary

The CKB miner's HTTP notification listener (`handle()` in `miner/src/client.rs`) accepts block template updates from any HTTP client with no authentication. When a miner operator enables notify mode by setting a `listen` address, any attacker who can reach that port can push a crafted `BlockTemplate` with an attacker-controlled coinbase lock script. The miner will then solve PoW on the attacker's template and submit it, redirecting all block rewards to the attacker's address.

### Finding Description

When the miner is started with a `listen` address configured, it spawns an HTTP server via `spawn_background()`. The handler for incoming requests is:

```rust
// miner/src/client.rs, lines 358–368
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }

    Ok(Response::new(Empty::new()))
}
``` [1](#0-0) 

There is no authentication token, IP allowlist, HMAC, or any other caller-identity check. Any HTTP POST carrying a syntactically valid `BlockTemplate` JSON body will be accepted and will immediately replace the miner's current work via `client.update_block_template(template)`.

The notify-mode path is activated in `spawn_background()`:

```rust
// miner/src/client.rs, lines 204–226
if let Some(addr) = self.config.listen {
    ...
    self.handle.spawn(async move {
        client.listen_block_template_notify(addr).await;
    });
    ...
}
``` [2](#0-1) 

The `listen_block_template_notify` function binds a Hyper HTTP server to `addr` and routes every incoming request through the unauthenticated `handle()` function above.

An attacker crafts a `BlockTemplate` JSON that is structurally identical to a legitimate template but replaces the `cellbase` transaction's output lock script with one whose `args` encodes the attacker's public-key hash. The miner's workers receive this template via `WorkerMessage::NewWork`, solve the PoW puzzle, and the miner calls `submit_block` on the CKB node with the attacker's coinbase. The node validates the block normally (the PoW is valid, the coinbase structure is valid), accepts it, and the block reward is paid to the attacker.

### Impact Explanation

**Direct financial loss.** Every block the miner finds while the attacker's template is active pays the full block reward (primary issuance + secondary issuance + transaction fees) to the attacker's address instead of the legitimate miner's address. The miner receives nothing. The attacker needs no CKB, no keys, and no privileged access to the node — only network reachability to the miner's HTTP listen port.

### Likelihood Explanation

Notify mode is the recommended high-performance configuration for production miners (the code even prints an explicit instruction to configure it). Miners are frequently co-located with pool infrastructure or run on servers with multiple network interfaces. A misconfigured firewall, a cloud security-group rule that opens the notify port, or a miner running on a shared host is sufficient for exploitation. The attack is a single HTTP POST and is immediately effective.

### Recommendation

Add a shared secret (e.g., a bearer token or HMAC-SHA256 over the body) to the notify endpoint. The miner configuration should require an `auth_token` field when `listen` is set, and `handle()` should reject any request whose `Authorization` header does not match before deserializing the body. Alternatively, restrict the listener to `127.0.0.1` only and document that as a hard requirement.

### Proof of Concept

1. Victim miner is configured with `listen = "0.0.0.0:8118"` in `ckb-miner.toml`.
2. Attacker obtains any recent legitimate `BlockTemplate` (e.g., via `get_block_template` on any public node).
3. Attacker replaces `cellbase.data.outputs[0].lock.args` with their own 20-byte public-key hash.
4. Attacker sends:
   ```
   curl -X POST http://<miner-ip>:8118 \
        -H 'Content-Type: application/json' \
        -d '<modified_block_template_json>'
   ```
5. `handle()` deserializes the body and calls `client.update_block_template(template)` with no authentication check.
6. The miner's workers immediately switch to the attacker's template.
7. The next block found is submitted with the attacker's coinbase lock script; the block reward is paid to the attacker. [1](#0-0) [2](#0-1)
