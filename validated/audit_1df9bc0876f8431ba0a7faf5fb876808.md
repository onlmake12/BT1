### Title
Concurrent `get_overview(refresh=0x1F)` Calls Cause Multiplicative `get_block_template` Invocations — (`rpc/src/module/terminal.rs`)

### Summary

`TerminalRpcImpl::get_overview` with `refresh=0x1F` (EVERYTHING) unconditionally clears the cache and then unconditionally bypasses the cache check in `get_tx_pool_info`, calling `get_block_template` on every concurrent request with no deduplication. N concurrent callers produce N concurrent `get_block_template` calls, saturating the block assembler's request queue and delaying legitimate miner block-template fetches.

### Finding Description

**Step 1 — Cache is cleared, then cache check is structurally bypassed.**

In `get_overview`, when `refresh` contains `EVERYTHING`: [1](#0-0) 

The cache is cleared. Then `get_tx_pool_info(refresh)` is called with the same `refresh` value that still contains `TX_POOL_INFO`.

**Step 2 — The cache guard in `get_tx_pool_info` is always false when `TX_POOL_INFO` is set.** [2](#0-1) 

The condition `!refresh.contains(RefreshKind::TX_POOL_INFO)` evaluates to `false` whenever the caller passes any refresh flag that includes `TX_POOL_INFO` (which `EVERYTHING=0x1F` does). The cache is **never consulted** — the function always falls through to the expensive fetch path.

**Step 3 — `get_block_template` is called unconditionally on every such request.** [3](#0-2) 

There is no in-flight deduplication, no semaphore, and no "already refreshing" flag. Each of the N concurrent requests independently calls `self.shared.get_block_template(None, None, None)`.

**Step 4 — The cache write at line 620 does not help concurrent callers.** [4](#0-3) 

Because the cache check at line 574 is structurally bypassed (not a TTL miss, but a flag-forced skip), even if one request finishes and writes to the cache, all other in-flight requests with `refresh` containing `TX_POOL_INFO` will still skip the cache and call `get_block_template` again.

### Impact Explanation

`get_block_template` is an expensive synchronous call routed through the block assembler's internal channel. N concurrent `get_overview(refresh=0x1F)` calls enqueue N `get_block_template` requests. The block assembler processes these sequentially. A legitimate miner calling `get_block_template` must wait behind all N queued requests. At sufficient concurrency (e.g., 20 simultaneous callers), the miner's response is delayed by the cumulative cost of N block-template assemblies, causing it to miss the current block window and lose the block reward — a direct economic impact.

### Likelihood Explanation

The Terminal RPC module is enabled via `enable_terminal` in `service_builder.rs`: [5](#0-4) 

If the Terminal module is exposed on the public RPC port (which is operator-configurable), any unauthenticated caller can send `get_overview` with `refresh=31`. The attack requires only the ability to send concurrent HTTP JSON-RPC requests — no keys, no PoW, no privileged access. The `RefreshKind::EVERYTHING` flag is documented publicly in the RPC README, making the attack trivially discoverable.

### Recommendation

Introduce a per-slot in-flight guard (e.g., a `Mutex<bool>` or `AtomicBool` "refresh in progress" flag per cache slot). When a refresh is already in flight for a given slot, subsequent concurrent callers should either:
- Wait on the in-flight result (coalescing), or
- Return the last cached value (stale-on-contention).

Additionally, even when `refresh` contains `TX_POOL_INFO`, the cache should be consulted first and only bypassed if the entry is actually stale, not unconditionally. The current logic conflates "force refresh" with "always re-fetch regardless of concurrent in-flight requests."
