Audit Report

## Title
Unbounded Full CKB-VM Execution via `test_tx_pool_accept` RPC With No Cache Write, No Insertion Guard, and No Rate Limiting — (`tx-pool/src/process.rs`, `rpc/src/module/pool.rs`)

## Summary
`test_tx_pool_accept` triggers full `ContextualTransactionVerifier` execution (up to `max_block_cycles`) on every call via `block_in_place`, but `_test_accept_tx` never writes the result to the verify cache and never inserts the transaction into the verify queue or orphan pool. The duplicate guard therefore always passes for the same transaction, and the cache is never populated by this path, so every repeated call unconditionally re-executes the full CKB-VM script. With no rate limiting present anywhere in the stack, an attacker with RPC access can saturate the tx-pool's tokio worker threads and stall `send_transaction` and `get_block_template`.

## Finding Description

**RPC entry point** (`rpc/src/module/pool.rs` L637–660): `test_tx_pool_accept` calls `tx_pool.test_accept_tx(tx)` with no throttle, quota, or authentication check. [1](#0-0) 

**Duplicate guard** (`tx-pool/src/process.rs` L386–399): `test_accept_tx` checks `verify_queue_contains` and `orphan_contains` before calling `_test_accept_tx`. Both checks key on `ProposalShortId` in the verify queue and orphan pool respectively. [2](#0-1) [3](#0-2) 

**`_test_accept_tx` never populates either structure** (`tx-pool/src/process.rs` L779–800): It calls `pre_check`, reads the verify cache, then calls `verify_rtx` with `command_rx = None`. It returns the result directly — no insertion into the verify queue, no insertion into the orphan pool, and critically **no write to the verify cache**. [4](#0-3) 

**Contrast with `_process_tx`** (`tx-pool/src/process.rs` L758–765): The normal submission path explicitly writes the verified result back to `txs_verify_cache` after success. `_test_accept_tx` has no equivalent write, so the cache is never populated by this path and every call re-executes from scratch. [5](#0-4) 

**Full CKB-VM execution via `block_in_place`** (`tx-pool/src/util.rs` L116–131): Because `command_rx` is `None` and `cache_entry` is `None` (never written by `_test_accept_tx`), `verify_rtx` takes the `else` branch and calls `block_in_place(ContextualTransactionVerifier::verify(max_block_cycles, false))`, blocking a tokio worker thread for the full cycle budget. [6](#0-5) 

**`max_block_cycles` used, not per-tx limit** (`tx-pool/src/process.rs` L787): `_test_accept_tx` passes `self.consensus.max_block_cycles()` — the full block cycle budget — rather than the per-transaction limit used elsewhere. [7](#0-6) 

**No rate limiting anywhere**: A grep for `rate_limit`, `RateLimit`, and `rate.limit` across the entire codebase returns zero matches. [1](#0-0) 

The combined effect: the duplicate guard always passes (nothing is ever inserted), the cache is never populated (nothing is ever written), and each call blocks a tokio worker thread for up to `max_block_cycles` of CKB-VM execution.

## Impact Explanation

Each call to `test_tx_pool_accept` with a max-cycle script blocks one tokio worker thread via `block_in_place` for the full execution duration. Sending enough concurrent requests saturates the tx-pool service thread pool. Once saturated:

- `send_transaction` submissions queue behind blocked workers and time out.
- `get_block_template` (block assembly for miners) is delayed or fails.
- All other tx-pool RPC operations stall.

This constitutes **crashing/rendering unresponsive a CKB node's tx-pool service**, matching the allowed High impact: *"Vulnerabilities which could easily crash a CKB node"* (10001–15000 points). A miner node that cannot assemble blocks is effectively taken offline as a mining participant.

## Likelihood Explanation

The RPC endpoint is part of the standard documented public API, enabled by default, and requires no authentication, staking, or privileged role. The attacker needs only network access to the RPC port and the ability to construct a transaction referencing a script cell that loops to the cycle limit. The attack is asymmetric: the attacker pays only the cost of HTTP requests while the victim node bears full CKB-VM execution per request. The attack is trivially repeatable and requires no on-chain state changes.

## Recommendation

1. **Write to the verify cache in `_test_accept_tx`**: After a successful verification, write the result to `txs_verify_cache` (keyed by `witness_hash`) exactly as `_process_tx` does at L758–765. This ensures repeated calls for the same transaction hit the cache path (`TimeRelativeTransactionVerifier`) instead of re-running `ContextualTransactionVerifier`.
2. **Add an in-progress flag or per-transaction semaphore**: Before calling `_test_accept_tx`, insert the transaction's `ProposalShortId` into a dedicated "test-in-progress" set and remove it on completion. Concurrent or repeated calls for the same transaction are rejected until the first completes.
3. **Cap cycles to per-transaction limit**: Replace `self.consensus.max_block_cycles()` with the per-transaction cycle limit in `_test_accept_tx`.
4. **Apply global or per-IP rate limiting** to `test_tx_pool_accept` at the RPC server layer.

## Proof of Concept

1. Deploy (or reference an existing) script cell containing a tight loop consuming `max_block_cycles - 1` cycles.
2. Construct transaction `T` spending any live cell with that script as the lock.
3. In a loop, send:
   ```
   POST /rpc {"method":"test_tx_pool_accept","params":[T, "passthrough"]}
   ```
   The same transaction can be reused on every iteration — `_test_accept_tx` never writes to the cache, so the cache is always empty for this path, and the duplicate guard always passes because nothing is inserted into the verify queue or orphan pool.
4. Each call enters `_test_accept_tx` → `verify_rtx` (cache miss, `command_rx = None`) → `block_in_place(ContextualTransactionVerifier::verify(max_block_cycles, false))`.
5. With enough concurrent calls, the tokio worker pool saturates; `send_transaction` and `get_block_template` calls from legitimate users begin timing out.
