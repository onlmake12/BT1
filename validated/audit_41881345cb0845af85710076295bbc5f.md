### Title
Use-After-Delete of Storage Pointer in `Echo.executeCallback()` Silently Drops Consumer Callbacks for Overflow Requests — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` obtains a `Request storage req` pointer before calling `clearRequest()`. When the request resides in `_state.requestsOverflow` (not the primary slot), `clearRequest()` issues a Solidity `delete` on the mapping entry, zeroing every field — including `req.requester` and `req.callbackGasLimit`. The function then reads those zeroed fields to invoke the consumer callback, sending it to `address(0)` with 0 gas. The try/catch silently swallows the failure, the provider has already been credited, and the consumer's callback is permanently lost.

---

### Finding Description

`executeCallback` stores a storage pointer to the active request:

```solidity
Request storage req = findActiveRequest(sequenceNumber);
``` [1](#0-0) 

`findRequest` returns a pointer to either the primary slot (`_state.requests[shortKey]`) or the overflow map (`_state.requestsOverflow[key]`): [2](#0-1) 

Later, `clearRequest` is called. For overflow requests it issues `delete _state.requestsOverflow[key]`, which zeroes every field of the struct that `req` points to: [3](#0-2) 

After `clearRequest`, the function reads `req.requester` and `req.callbackGasLimit` — both now zero — to invoke the consumer callback: [4](#0-3) 

The call becomes `IEchoConsumer(address(0))._echoCallback{gas: 0}(...)`, which fails silently inside the `try/catch`. The provider fee was already credited at line 161–162 and the request was already cleared at line 164, so there is no recovery path. [5](#0-4) 

A request lands in overflow whenever two active requests share the same `shortKey` (a `uint8` derived from `keccak256(sequenceNumber)`). With 256 slots and a growing request queue, collisions are a natural, non-adversarial occurrence. [6](#0-5) 

---

### Impact Explanation

- The consumer's `_echoCallback` is never delivered; any application logic gated on receiving the price update is permanently blocked.
- The consumer's fee (stored in `req.fee`) has already been transferred to the provider's `accruedFeesInWei` — the consumer paid for a service they did not receive.
- The provider suffers no penalty; the `PriceUpdateCallbackFailed` event is emitted, but the request is cleared and funds are gone.

Impact: **loss of user funds + permanent denial of callback service** for any request that happens to be stored in the overflow map.

---

### Likelihood Explanation

No adversarial action is required. With 256 primary slots and a continuously incrementing sequence number, any two concurrently active requests whose sequence numbers hash to the same `shortKey` will cause the older one to be pushed to overflow. Under normal load this occurs regularly. A malicious provider could also deliberately delay fulfillment of a request until a collision occurs, then fulfill the overflow request to pocket the fee without delivering the callback.

---

### Recommendation

Cache the fields that are needed after `clearRequest` into memory variables **before** calling `clearRequest`:

```solidity
address callAddress = req.requester;
uint32 gasLimit = req.callbackGasLimit;
uint128 fee = req.fee;

_state.providers[providerToCredit].accruedFeesInWei += SafeCast.toUint128((fee + msg.value) - pythFee);
clearRequest(sequenceNumber);

try IEchoConsumer(callAddress)._echoCallback{gas: gasLimit}(sequenceNumber, priceFeeds) {
    ...
}
```

This mirrors the pattern already used correctly in `Entropy.sol`'s `revealWithCallback` (the `else` branch), which explicitly copies `req.requester` to a memory variable before `clearRequest`: [7](#0-6) 

---

### Proof of Concept

1. User A submits `requestPriceUpdatesWithCallback` → assigned `sequenceNumber = N` → `shortKey = K` → stored in `_state.requests[K]`.
2. User B submits another request → assigned `sequenceNumber = M` → also hashes to `shortKey = K` → `allocRequest` moves request N to `_state.requestsOverflow[hash(N)]` and stores M in `_state.requests[K]`.
3. Provider calls `executeCallback(provider, N, ...)`.
4. `findActiveRequest(N)` returns a storage pointer to `_state.requestsOverflow[hash(N)]` — all fields valid at this point.
5. `clearRequest(N)` executes `delete _state.requestsOverflow[hash(N)]` — `req.requester` and `req.callbackGasLimit` are now `address(0)` and `0`.
6. `IEchoConsumer(address(0))._echoCallback{gas: 0}(N, priceFeeds)` is called — fails silently.
7. `PriceUpdateCallbackFailed` is emitted. Provider is credited. User A's callback is permanently lost.

        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];

    function clearRequest(uint64 sequenceNumber) internal {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        Request storage req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            req.sequenceNumber = 0;
        } else {
            delete _state.requestsOverflow[key];
        }
