### Title
Unbounded WebSocket Connection Exhaustion in Quorum Server — No Connection Limit, Rate Limit, or Max Duration - (File: apps/quorum/src/ws.rs)

### Summary
The Quorum server's `/ws` WebSocket endpoint accepts an unlimited number of connections from any unprivileged client with no per-IP rate limiting, no connection count cap, and no maximum connection duration. An attacker can open thousands of persistent connections (responding to pings to keep them alive), exhausting file descriptors and task memory, and preventing legitimate Hermes subscribers from connecting to receive VAA broadcasts.

### Finding Description

The `ws_route_handler` in `apps/quorum/src/ws.rs` upgrades any incoming WebSocket connection unconditionally and spawns a `Subscriber` task with no authentication, no IP-based rate limiting, and no connection count enforcement. [1](#0-0) 

The `WsState` struct tracks only a monotonically increasing `subscriber_counter` — it has no rate limiter, no whitelist, and no maximum connection count, unlike the Hermes WS state which carries a `DefaultKeyedRateLimiter<IpAddr>` and enforces `BYTES_LIMIT_PER_IP_PER_SECOND`. [2](#0-1) 

Compare to Hermes, which enforces per-IP byte-rate limiting and a hard 24-hour connection deadline: [3](#0-2) [4](#0-3) [5](#0-4) 

The Quorum `handle_next` loop has only a 30-second ping/pong heartbeat. A cooperative attacker simply responds to every ping, keeping each connection alive indefinitely with no upper bound: [6](#0-5) 

Additionally, `handle_new_vaa` performs a blocking `sender.send(...).await` with no timeout. If a malicious client's TCP receive window fills (by not reading), the task blocks on this send indefinitely, tying up the Tokio worker for that connection: [7](#0-6) 

Hermes WS avoids this with a configurable `ws_send_timeout` passed into every `Subscriber`: [8](#0-7) 

### Impact Explanation

The Quorum server is the VAA broadcast hub that Hermes instances subscribe to in order to receive assembled Wormhole VAAs for price feed updates: [9](#0-8) 

If an attacker exhausts the Quorum server's file descriptors or Tokio task budget by holding thousands of idle WebSocket connections open, the server cannot accept new connections from legitimate Hermes nodes. Hermes nodes that lose their Quorum WebSocket connection will fail to receive new VAAs, halting price feed aggregation and causing stale or missing price updates across all downstream consumers.

### Likelihood Explanation

The `/ws` endpoint is reachable by any unprivileged network client. No authentication is required. The attacker only needs to:
1. Open N TCP connections and complete the WebSocket handshake.
2. Respond to a WebSocket `Ping` frame every 30 seconds to prevent the heartbeat from closing the connection.

This requires no special privileges, no leaked keys, and no on-chain interaction. Standard tooling (e.g., a simple async script) can sustain thousands of idle WebSocket connections from a single host.

### Recommendation

1. **Add a maximum connection duration** (e.g., 24 hours, matching Hermes) by storing a `connection_deadline` in `Subscriber` and adding a `tokio::time::sleep_until(self.connection_deadline)` arm to `handle_next`.
2. **Add per-IP connection count or byte-rate limiting** using `governor`'s `DefaultKeyedRateLimiter`, mirroring the `WsState` in `apps/hermes/server/src/api/ws.rs`.
3. **Add a send timeout** to `handle_new_vaa` using `tokio::time::timeout`, mirroring `ws_send_timeout` in Hermes.
4. **Enforce a global maximum concurrent connection count** and reject upgrades beyond the limit.

### Proof of Concept

```python
import asyncio
import websockets

async def hold_connection(url):
    async with websockets.connect(url) as ws:
        # Respond to pings to keep alive indefinitely
        async for msg in ws:
            pass  # never close

async def main():
    url = "ws://<quorum-host>/ws"
    # Open 1000 concurrent connections
    await asyncio.gather(*[hold_connection(url) for _ in range(1000)])

asyncio.run(main())
```

Each connection holds a file descriptor and a Tokio task. At the Linux default of 1024 open file descriptors per process, ~1000 attacker connections saturate the Quorum server, causing `accept()` failures for all subsequent legitimate Hermes subscriber connections. Hermes nodes that cannot reconnect to Quorum stop receiving VAAs and stop publishing price updates.

async fn websocket_handler(state: axum::extract::State<State>, stream: WebSocket) {
    let subscriber_id = state.ws.subscriber_counter.fetch_add(1, Ordering::SeqCst);
    let (sender, receiver) = stream.split();
    let new_receiver = state.ws.broadcast_receiver.resubscribe();
    let mut subscriber = Subscriber::new(subscriber_id, new_receiver, receiver, sender);
    subscriber.run().await;

    async fn handle_next(&mut self) -> Result<()> {
        tokio::select! {
            maybe_update_event = self.notify_receiver.recv() => {
                match maybe_update_event {
                    Ok(event) => self.handle_update(event).await,
                    Err(e) => Err(anyhow!("Error receiving update event: {:?}", e)),
                }
            },
            maybe_message_or_err = self.receiver.next() => {
                self.handle_client_message(
                    maybe_message_or_err.ok_or(anyhow!("Client channel is closed"))??
                ).await
            },
            _  = self.ping_interval.tick() => {
                if !self.responded_to_ping {
                    return Err(anyhow!("Subscriber did not respond to ping. Closing connection."));
                }
                self.responded_to_ping = false;
                self.handle_update(UpdateEvent::Ping).await
            },
            _ = wait_for_exit() => {
                self.sender.close().await?;
                self.closed = true;
                Err(anyhow!("Application is shutting down. Closing connection."))
            }
        }
    }
