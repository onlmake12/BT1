### Title
Attacker Can DoS the VerifyQueue with Minimum-Fee Transactions, Blocking Higher-Fee Transactions - (`tx-pool/src/component/verify_queue.rs`)

### Summary

The `VerifyQueue` in CKB's tx-pool has a hard 256 MB size cap with no fee-rate-based eviction. An unprivileged peer or RPC caller can flood the queue with structurally valid, minimum-fee-rate transactions, filling it entirely. Because `add_tx` performs a hard reject when full and the queue stores no fee-rate metadata for eviction, any subsequent higher-fee-rate transaction is rejected with `Reject::Full` until the attacker's transactions drain through the worker pipeline.

---

### Finding Description

`VerifyQueue` is the staging area all incoming transactions must pass through before full contextual verification. It enforces a single hard cap: [1](#0-0) 

When `add_tx` is called and the queue is at capacity, it unconditionally returns `Reject::Full`: [2](#0-1) 

The queue's internal index tracks only `added_time` and `is_large_cycle` — **no fee-rate field exists**: [3](#0-2) 

There is no eviction path analogous to the main pool's `limit_size` / `next_evict_entry` (which evicts by `EvictKey` / fee-rate). The only removal operations are `remove_tx`, `remove_txs`, `remove_txs_by_peer`, and `pop_front` — none of which evict based on quality to make room for a better entry.

The critical entry path for remote transactions is `resumeble_process_tx`: [4](#0-3) 

This path calls only `non_contextual_verify` (structural checks, no fee check) before calling `enqueue_verify_queue`. The fee-rate check (`check_tx_fee`) lives in `pre_check`, which is **not** called on this path — it is deferred until after the transaction is dequeued by a worker: [5](#0-4) 

An attacker therefore submits many structurally valid transactions (passing `non_contextual_verify`) with the minimum fee rate (default `1000 shannons/KB`) and large serialized sizes. Each transaction occupies space in the VerifyQueue without any fee-rate gating at enqueue time. Once the 256 MB cap is reached, every subsequent `add_tx` call — regardless of the new transaction's fee rate — returns `Reject::Full`.

The main pool (`TxPool`) does have fee-rate-based eviction via `limit_size`: [6](#0-5) 

But the `VerifyQueue` has no equivalent mechanism.

---

### Impact Explanation

Any transaction submitted via the P2P relay path (`submit_remote_tx`) or the RPC `send_transaction` path (`notify_tx`) is blocked from entering the verification pipeline while the VerifyQueue is saturated. High-fee-rate transactions from legitimate users are rejected with `Reject::Full` and are not verified or admitted to the main pool. This constitutes a targeted DoS on the transaction admission pipeline of a CKB node, degrading its ability to process and relay transactions.

---

### Likelihood Explanation

The attacker needs to sustain ~256 MB of pending-but-unverified transaction data in the queue. Transactions need only pass `non_contextual_verify` (structural validity, no input resolution, no fee check). The attacker can craft transactions with fake inputs up to `TRANSACTION_SIZE_LIMIT` in size, each paying exactly `min_fee_rate`. Because the fee check is deferred, these transactions occupy queue space until a worker dequeues and rejects them — at which point the attacker can immediately re-flood. The attack is rate-based: the attacker must submit faster than the worker pool drains the queue. With the default worker count (3/4 of CPU cores) and a 256 MB cap, this is achievable from a single well-connected peer. No privileged access, key material, or majority hashpower is required.

---

### Recommendation

1. **Add fee-rate metadata to `VerifyEntry`**: Store the declared fee rate (or a proxy such as `declared_cycles / tx_size`) at enqueue time so that eviction decisions can be made.
2. **Implement fee-rate-based eviction in `add_tx`**: When the queue is full and the incoming transaction has a higher fee rate than the lowest-fee-rate entry currently in the queue, evict the worst entry and admit the new one — mirroring the main pool's `limit_size` / `next_evict_entry` logic.
3. **Enforce a minimum fee-rate check before enqueue**: Move the `check_tx_fee` call (or a lightweight equivalent) into `resumeble_process_tx` so that transactions below `min_fee_rate` are rejected before consuming queue space.

---

### Proof of Concept

1. Attacker opens a P2P connection to a target CKB node.
2. Attacker constructs ~N structurally valid transactions, each with serialized size near `TRANSACTION_SIZE_LIMIT`, with fake (non-existent) inputs but paying exactly `min_fee_rate` in declared fees. These pass `non_contextual_verify`.
3. Attacker relays all N transactions via the relay protocol, triggering `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue` on the target node.
4. The VerifyQueue's `total_tx_size` reaches `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB).
5. A legitimate user submits a high-fee-rate transaction via `send_transaction` RPC. The node calls `notify_tx` → `resumeble_process_tx` → `enqueue_verify_queue` → `add_tx`. `is_full` returns `true`; the call returns `Reject::Full`. The legitimate transaction is dropped.
6. As workers drain the attacker's transactions (which fail at the fee/input-resolution stage), the attacker re-submits new ones to maintain saturation. [7](#0-6) [8](#0-7)

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
