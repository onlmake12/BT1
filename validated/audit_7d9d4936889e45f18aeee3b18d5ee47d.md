### Title
Provider Fee Frontrunning Causes DoS on `requestPriceUpdatesWithCallback` — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the fee required to submit a price update request is computed on-chain at execution time using the provider's current fee parameters. Because a provider (or their fee manager) can update these parameters at any time via `setProviderFee()` with no timelock, a provider can frontrun a user's `requestPriceUpdatesWithCallback` transaction by raising their fees, causing the user's transaction to revert with `InsufficientFee` even though the user paid the correct fee at the time they queried it. The same pattern exists in `Entropy.sol` via `setProviderFee()` / `setProviderFeeAsFeeManager()`.

---

### Finding Description

`Echo.requestPriceUpdatesWithCallback` computes the required fee on-chain at execution time:

```solidity
// Echo.sol line 75-76
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
```

`getFee` reads live storage values:

```solidity
// Echo.sol lines 240-254
uint96 baseFee = _state.pythFeeInWei;
uint96 providerBaseFee = _state.providers[provider].baseFeeInWei;
uint96 providerFeedFee = SafeCast.toUint96(
    priceIds.length * _state.providers[provider].feePerFeedInWei
);
uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei;
uint256 gasFee = callbackGasLimit * providerFeeInWei;
feeAmount = baseFee + providerBaseFee + providerFeedFee + SafeCast.toUint96(gasFee);
```

Any of the three provider fee components (`baseFeeInWei`, `feePerFeedInWei`, `feePerGasInWei`) can be changed instantly by the provider or their fee manager via `setProviderFee()`:

```solidity
// Echo.sol lines 395-426
function setProviderFee(
    address provider,
    uint96 newBaseFeeInWei,
    uint96 newFeePerFeedInWei,
    uint96 newFeePerGasInWei
) external override {
    ...
    _state.providers[provider].baseFeeInWei = newBaseFeeInWei;
    _state.providers[provider].feePerFeedInWei = newFeePerFeedInWei;
    _state.providers[provider].feePerGasInWei = newFeePerGasInWei;
    ...
}
```

There is no timelock, delay, or minimum notice period before a fee change takes effect.

The same root cause exists in `Entropy.sol`:

```solidity
// Entropy.sol lines 234-235
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
```

with `setProviderFee()` and `setProviderFeeAsFeeManager()` allowing instant fee changes.

Additionally, the off-chain Fortuna keeper service (`apps/fortuna/src/keeper/fee.rs`) automatically adjusts provider fees based on profitability targets, meaning fee changes occur regularly in production without any malicious intent.

---

### Impact Explanation

A user who:
1. Calls `getFee(provider, gasLimit, priceIds)` off-chain to determine the exact required fee `F`
2. Submits `requestPriceUpdatesWithCallback{value: F}(...)`

can have their transaction reverted if the provider (or Fortuna keeper) increases any fee component between step 1 and step 2. The user loses gas and must re-query and resubmit. A malicious provider can repeat this indefinitely to permanently DoS any specific user or consumer contract, preventing them from ever successfully requesting a price update.

---

### Likelihood Explanation

- The Fortuna keeper (`apps/fortuna/src/keeper/fee.rs`) automatically adjusts fees when profitability targets are missed, making non-malicious fee-change races a realistic production scenario.
- Any registered provider or their designated fee manager can call `setProviderFee()` permissionlessly.
- On chains with a public mempool, a provider can observe a pending `requestPriceUpdatesWithCallback` transaction and frontrun it with a `setProviderFee` increase.
- Consumer contracts that cache the fee (e.g., compute it once and reuse it) are especially vulnerable.

---

### Recommendation

1. **Add a `maxFee` slippage parameter**: Allow callers to specify the maximum fee they are willing to pay. Revert only if `requiredFee > maxFee`, not if `msg.value < requiredFee`. This mirrors standard AMM slippage protection.
2. **Introduce a fee-change timelock**: Require that fee increases take effect only after a delay (e.g., one block or a configurable period), giving in-flight transactions time to complete.
3. **Refund excess**: If `msg.value > requiredFee`, refund the difference. This allows users to safely overpay as a buffer against fee increases, without losing funds.

---

### Proof of Concept

1. Provider registers with `baseFeeInWei = 1000 wei`, `feePerFeedInWei = 100 wei`, `feePerGasInWei = 1 wei`.
2. User queries `getFee(provider, 500_000, [feedA, feedB])` → returns `1000 + 200 + 500_000 + pythFee = F`.
3. User submits `requestPriceUpdatesWithCallback{value: F}(provider, publishTime, [feedA, feedB], 500_000)`.
4. Provider (or Fortuna keeper) frontruns with `setProviderFee(provider, 1001, 100, 1)` — a 1 wei increase to `baseFeeInWei`.
5. User's transaction executes: `getFee(...)` now returns `F + 1`. Since `msg.value == F < F + 1`, the check at line 76 reverts with `InsufficientFee`.
6. User's gas is wasted. Provider repeats step 4 on every retry. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
