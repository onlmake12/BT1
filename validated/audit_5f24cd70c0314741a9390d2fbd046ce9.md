### Title
`GetBlocksProcess` Missing IBD State Guard While `GetHeadersProcess` Enforces It - (File: `sync/src/synchronizer/get_blocks_process.rs`)

---

### Summary

The CKB sync protocol module explicitly documents that a node in IBD (Initial Block Download) mode must respond with `InIBD` to **both** `GetHeaders` and `GetBlocks` requests. `GetHeadersProcess` correctly enforces this with an IBD guard. `GetBlocksProcess` has no such guard and will serve full block data to any peer, even while the node is in IBD mode. This is a direct analog to the reported pattern: one function enforces a required state restriction while a sibling function that should enforce the same restriction does not.

---

### Finding Description

The module-level documentation for the synchronizer is unambiguous:

> "When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` and `GetBlocks` requests" [1](#0-0) 

`GetHeadersProcess::execute()` correctly implements this contract. When the node is in IBD, it sends an `InIBD` response and returns `Status::ignored()`:

```rust
if active_chain.is_initial_block_download() {
    info!("Ignoring getheaders from peer={} ...", self.peer);
    self.send_in_ibd();
    ...
    return Status::ignored();
}
``` [2](#0-1) 

`GetBlocksProcess::execute()` has **no IBD check at all**. It proceeds directly to serve block data to any requesting peer regardless of the node's IBD state:

```rust
pub fn execute(self) -> Status {
    let block_hashes = self.message.block_hashes();
    // ... only checks length and dedup, no IBD guard
    let active_chain = self.synchronizer.shared.active_chain();
    let iter = block_hashes.iter().take(INIT_BLOCKS_IN_TRANSIT_PER_PEER);
    for block_hash in iter {
        // serves blocks unconditionally
        if let Some(block) = active_chain.get_block(&block_hash) { ... }
    }
    Status::ok()
}
``` [3](#0-2) 

The `GetBlocks` message is dispatched through `Synchronizer::try_process` without any pre-flight IBD check at the dispatch layer either: [4](#0-3) 

The `is_initial_block_download()` function is defined on `Shared` and is accessible from `active_chain` in both handlers: [5](#0-4) 

---

### Impact Explanation

During IBD, the node is intentionally restricted: it syncs with only one selected peer and is supposed to stop responding to most P2P requests. Serving full block bodies (up to the consensus block size limit) to arbitrary peers during IBD:

1. **Bypasses the IBD isolation design**: Any inbound peer can extract full block data from a syncing node, defeating the purpose of the IBD restriction.
2. **Resource exhaustion**: An attacker can send many `GetBlocks` requests (up to `MAX_HEADERS_LEN` = 2000 hashes per message, serving up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` blocks per call), forcing the IBD node to perform expensive disk reads and network sends while it is already under maximum I/O load from its own sync process.
3. **Bandwidth amplification**: The attacker sends a small request (a list of block hashes) and receives large block bodies in return, amplifying their bandwidth consumption at the victim's expense. [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged inbound P2P peer can send a `GetBlocks` message. No authentication, whitelisting, or special privilege is required. A newly syncing CKB node spends a significant amount of time in IBD (potentially hours or days on mainnet), making the attack window large. The attacker only needs to know a few valid block hashes (which are public on-chain data) to trigger repeated block serving.

---

### Recommendation

Add the same IBD guard to `GetBlocksProcess::execute()` that exists in `GetHeadersProcess::execute()`. When the node is in IBD, it should send an `InIBD` response and return early, consistent with the documented protocol contract and the behavior of the sibling handler:

```rust
pub fn execute(self) -> Status {
    let active_chain = self.synchronizer.shared.active_chain();

    if active_chain.is_initial_block_download() {
        // send InIBD and return, matching GetHeadersProcess behavior
        return Status::ignored();
    }
    // ... rest of existing logic
}
``` [2](#0-1) 

---

### Proof of Concept

1. Start a CKB node from genesis (it will be in IBD mode since `unix_time_as_millis() - tip_header.timestamp() > MAX_TIP_AGE`).
2. Connect as an inbound P2P peer on the Sync protocol.
3. Send a `packed::SyncMessage` containing a `GetBlocks` payload with valid block hashes (e.g., the genesis block's children, obtainable from any public explorer).
4. Observe that the node responds with `SendBlock` messages containing full block data, rather than an `InIBD` message.
5. Contrast: sending a `GetHeaders` message to the same IBD node results in an `InIBD` response and no header data served.

The root cause is the absence of `active_chain.is_initial_block_download()` check in `GetBlocksProcess::execute()` at `sync/src/synchronizer/get_blocks_process.rs:33`, while the equivalent check exists at `sync/src/synchronizer/get_headers_process.rs:53`. [7](#0-6) [2](#0-1)

```rust
    pub fn execute(self) -> Status {
        let block_hashes = self.message.block_hashes();
        // use MAX_HEADERS_LEN as limit, we may increase the value of INIT_BLOCKS_IN_TRANSIT_PER_PEER in the future
        if block_hashes.len() > MAX_HEADERS_LEN {
            return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                "BlockHashes count({}) > MAX_HEADERS_LEN({})",
                block_hashes.len(),
                MAX_HEADERS_LEN,
            ));
        }
        let active_chain = self.synchronizer.shared.active_chain();

        let iter = block_hashes.iter().take(INIT_BLOCKS_IN_TRANSIT_PER_PEER);

        let mut dedup = HashSet::new();
        for block_hash in iter {
            debug!("get_blocks {} from peer {:?}", block_hash, self.peer);
            let block_hash = block_hash.to_entity();

            if block_hash == self.synchronizer.shared().consensus().genesis_hash() {
                return StatusCode::RequestGenesis.with_context("Request genesis block");
            }

            if !dedup.insert(block_hash.clone()) {
                return StatusCode::RequestDuplicate.with_context("Request duplicate block");
            }

            if !active_chain.contains_block_status(&block_hash, BlockStatus::BLOCK_VALID) {
                debug!(
                    "Ignoring get_block {} request from peer={} as it is not verified.",
                    block_hash, self.peer
                );
                continue;
            }

            if let Some(block) = active_chain.get_block(&block_hash) {
                debug!(
                    "respond_block {} {} to peer {:?}",
                    block.number(),
                    block.hash(),
                    self.peer,
                );
                let content = packed::SendBlock::new_builder().block(block.data()).build();
                let message = packed::SyncMessage::new_builder().set(content).build();

                let nc = Arc::clone(self.nc);
                self.synchronizer
                    .shared()
                    .shared()
                    .async_handle()
                    .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
            } else {
                // TODO response not found
                // TODO add timeout check in synchronizer

                // We expect that `block_hashes` is sorted descending by height.
                // So if we cannot find the current one from local, we cannot find
                // the next either.
                debug!("Stopping getblocks, since {} is not found", block_hash);
                break;
            }
        }

        Status::ok()
    }

    pub fn is_initial_block_download(&self) -> bool {
        // Once this function has returned false, it must remain false.
        if self.ibd_finished.load(Ordering::Acquire) {
            false
        } else if unix_time_as_millis().saturating_sub(self.snapshot().tip_header().timestamp())
            > MAX_TIP_AGE
        {
            true
        } else {
            self.ibd_finished.store(true, Ordering::Release);
            false
        }
    }
