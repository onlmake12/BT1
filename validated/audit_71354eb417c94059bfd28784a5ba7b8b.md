Audit Report

## Title
Incomplete Orphan Block Expiry Check Allows Unbounded Memory Growth via Single-Element `need_clean` — (`chain/src/utils/orphan_block_pool.rs`)

## Summary
`InnerPool::need_clean` samples only one arbitrary entry from a `HashMap` group of orphan blocks sharing a parent hash to decide whether to evict the entire group. An attacker inserts two competing orphan blocks under the same parent hash — one with an expired epoch and one with a far-future epoch — ensuring the group is never evicted when the non-expired block is sampled. Because non-contextual verification does not check PoW or epoch validity, crafting such blocks is trivial. Repeating this across many parent hashes causes unbounded growth of the orphan pool and associated caches, leading to node memory exhaustion.

## Finding Description

`InnerPool` stores orphan blocks as:
```rust
blocks: HashMap<ParentHash, HashMap<packed::Byte32, LonelyBlockHash>>
```
Multiple orphan blocks sharing a parent hash are stored in the same inner `HashMap`.

The expiry check at `orphan_block_pool.rs` lines 112–122 is:
```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← only ONE element checked
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
``` [1](#0-0) 

`map.iter().next()` returns an **arbitrary** entry from the inner `HashMap` (Rust `HashMap` iteration order is non-deterministic). The epoch of that single block determines whether the **entire group** is evicted. If the sampled block is not expired, all blocks in the group — including genuinely expired ones — are retained indefinitely.

`clean_expired_blocks` calls `need_clean` for every leader and removes the group only if it returns `true`: [2](#0-1) 

The cleanup timer fires every 60 seconds: [3](#0-2) 

**Why crafting attack blocks is trivial:**

The non-contextual verification path in `chain_service.rs` calls only `BlockVerifier` and `NonContextualBlockTxsVerifier`: [4](#0-3) 

`BlockVerifier::verify()` runs only `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier` — **no PoW check, no epoch check**: [5](#0-4) 

`PowVerifier` is part of the context-dependent `HeaderVerifier`, which requires the parent block to be known and is never invoked in the non-contextual path: [6](#0-5) 

`EpochVerifier` similarly requires the parent's epoch and is only reachable contextually: [7](#0-6) 

Therefore, an attacker can craft blocks with any epoch value and without solving any PoW, and they will pass non-contextual verification and be inserted into the orphan pool.

When expired orphans are not returned by `clean_expired_blocks`, `clean_expired_orphans` in `orphan_broker.rs` never calls `delete_block`, `remove_header_view`, or `remove_block_status` for them: [8](#0-7) 

The orphan pool has no enforced maximum size — the `with_capacity` argument sets only the initial allocation, and the comment explicitly forbids LRU eviction: [9](#0-8) 

## Impact Explanation

Expired orphan blocks that are not pruned permanently occupy memory in `InnerPool::blocks`, `InnerPool::parents`, `InnerPool::leaders`, and the shared `block_status_map` and `header_view` caches. Because there is no enforced size cap on the orphan pool, an attacker can grow these structures without bound across repeated 60-second cleanup cycles. Each cycle that fails to evict a group leaves the stale entries permanently, since the non-expired competing block in the group will always prevent the check from firing for that group. Sustained attack leads to OOM and node crash.

**Impact: High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

Any peer connected to a CKB node can relay blocks. The attacker crafts two blocks with the same parent hash `P` (unknown to the node), one with an old epoch and one with a far-future epoch. Neither requires valid PoW or a valid epoch relative to the chain, since `BlockVerifier` checks neither. Both are inserted into the orphan pool under parent `P`. The attack is cheap (no PoW required), repeatable with fresh parent hashes, and requires no privileged access or mining infrastructure.

## Recommendation

Replace the single-element check with an all-elements check. Only clean the group if **every** block in it is expired:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            map.values().all(|lonely_block| {
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
```

Additionally, enforce a hard maximum size on the orphan pool with FIFO or LRU eviction of the oldest entries, and consider adding a non-contextual PoW check (verifying the nonce against the block's own declared `compact_target`) before orphan pool insertion to raise the cost of the attack.

## Proof of Concept

1. Connect to a CKB node as a peer.
2. Choose a parent hash `P` not known to the node.
3. Craft **Block A**: `parent_hash = P`, `epoch = EpochNumberWithFraction::new(1, 0, 1)` (epoch 1), any nonce (PoW not checked non-contextually).
4. Craft **Block B**: `parent_hash = P`, `epoch = EpochNumberWithFraction::new(9999, 0, 1)` (epoch 9999), any nonce.
5. Send both blocks to the node. Both pass `BlockVerifier` and are inserted into `blocks[P]` as `{hash_A: BlockA, hash_B: BlockB}`.
6. Wait for the chain to advance past epoch 7 (`tip_epoch > 1 + EXPIRED_EPOCH = 7`). `clean_expired_blocks` fires every 60 seconds.
7. `need_clean(P, tip_epoch=8)` calls `map.iter().next()`. If it returns Block B (epoch 9999): `9999 + 6 < 8` → false → group not cleaned. Block A (epoch 1, genuinely expired) remains permanently.
8. Repeat steps 2–5 with fresh parent hashes `P1, P2, …`. Roughly half of groups will not be cleaned per cycle. Sustained repetition causes unbounded memory growth with no cleanup path.

**Unit test plan**: Insert two `LonelyBlockHash` entries under the same parent with epochs 1 and 9999, call `clean_expired_blocks(8)`, and assert that the returned `Vec` is empty when the `HashMap` iterates Block B first — demonstrating the incomplete eviction.

        // POW check first
        PowVerifier::new(header, self.consensus.pow_engine().as_ref()).verify()?;

        NumberVerifier::new(parent_fields.number, header).verify()?;

        TimestampVerifier::new(
            self.data_loader,
            header,

        )

        Ok(())
    }

pub struct EpochVerifier<'a> {
    parent: EpochNumberWithFraction,
    header: &'a HeaderView,

impl<'a> EpochVerifier<'a> {
    pub fn new(parent: EpochNumberWithFraction, header: &'a HeaderView) -> Self {
        EpochVerifier { parent, header }
    }

    pub fn verify(&self) -> Result<(), Error> {
        if !self.header.epoch().is_well_formed() {
            return Err(EpochError::Malformed {
                value: self.header.epoch(),
            }
            .into());
        }
        if !self.parent.is_genesis() && !self.header.epoch().is_successor_of(self.parent) {
            return Err(EpochError::NonContinuous {
                current: self.header.epoch(),
                parent: self.parent,
            }
            .into());
        }
        Ok(())
    }
