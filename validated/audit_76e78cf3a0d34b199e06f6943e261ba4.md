Audit Report

## Title
Inbound Peer Eviction Protection Tiers Gameable via Sub-Second Ping RTT Ties and Unsolicited Ping Spam — (File: `network/src/peer_registry.rs`)

## Summary
`try_evict_inbound_peer` in `network/src/peer_registry.rs` uses `as_secs()` granularity for both the Tier 1 ping-RTT comparator and the Tier 2 most-recent-message comparator. An attacker who pre-positions 8 peers with sub-second RTT and 7 peers that spam unsolicited pings can fill all 8 Tier 1 and all 7 remaining Tier 2 protection slots, leaving any honest peer as the sole eviction candidate. When the attacker's 16th connection is established, the honest peer is evicted with probability 1.0.

## Finding Description

**`sort_then_drop` mechanics** (`peer_registry.rs` L55–63): `list.sort_by(compare)` followed by `list.truncate(list.len() - n)` removes the last `n` elements from the sorted list. Those last `n` elements are the "protected" peers dropped from the eviction candidate pool; the first `list.len() - n` elements remain as candidates.

**Tier 1 — Ping RTT** (`peer_registry.rs` L151–165): The comparator sorts descending via `peer2_ping.cmp(&peer1_ping)` using `peer.ping_rtt.map(|p| p.as_secs()).unwrap_or(u64::MAX)`. Any RTT < 1 second maps to `as_secs() == 0`. After descending sort, all such peers cluster at the tail and are protected by `truncate`. Eight attacker peers with RTT of e.g. 500 ms are indistinguishable from each other (all map to 0) and trivially fill all 8 Tier 1 protection slots.

**Tier 2 — Most-recent ping message** (`peer_registry.rs` L168–183): The comparator uses `now.saturating_duration_since(t).as_secs()`. Any peer whose `last_ping_protocol_message_received_at` is within the last second yields `as_secs() == 0`. `ping_received` in `ping.rs` L62–68 sets `peer.last_ping_protocol_message_received_at = Some(Instant::now())` on every incoming ping with no rate limit, so an attacker can spam pings to keep their timestamp at `now`.

**Tier 3** (`peer_registry.rs` L185–188): `protect_peers = candidate_peers.len() >> 1`. With only 1 honest peer remaining after Tiers 1 and 2, `1 >> 1 = 0`, so zero peers are protected here.

**Correct attack sequence (max_inbound = 16):**

| Step | Action | Candidates |
|---|---|---|
| 1 | Attacker connects 8A peers (RTT=500ms) | 8 total, no eviction |
| 2 | Honest peer connects | 9 total, no eviction |
| 3 | Attacker connects 7B peers (ping spam) | 16 total, no eviction |
| 4 | 8th B attacker connects (17th) | Eviction triggered on 16 existing peers |
| Tier 1 | 8A (RTT=0) protected | 1 honest + 7B = 8 remain |
| Tier 2 | 7B (time-since=0) protected | 1 honest remains |
| Tier 3 | `1>>1=0` protected | 1 honest remains |
| Result | Honest peer evicted | p = 1.0 |

**Existing guards are insufficient**: `accept_peer` only checks `connection_status.non_whitelist_inbound >= self.max_inbound` with no per-IP cap (`peer_registry.rs` L116). `ping_received` applies no rate limiting (`ping.rs` L62–68).

**Note on submitted PoC**: The submitted PoC adds peers in the order 8A → 8B → honest → new. When the honest peer (17th) attempts to connect, eviction runs on 16 existing peers (8A + 8B). After Tier 1 (8A protected) and Tier 2 (8B protected), zero candidates remain, so `try_evict_inbound_peer` returns `None` and the honest peer receives `ReachMaxInboundLimit` — the PoC would panic at that step. The corrected sequence (8A first, then honest, then 7B, then 8th B) produces the claimed outcome.

## Impact Explanation

An attacker who pre-positions 8 sub-second-RTT peers and 7 ping-spamming peers (15 total) and then waits for an honest peer to connect can evict that honest peer with probability 1.0 on the 16th attacker connection. Repeating this pattern allows the attacker to monopolize all inbound slots, degrading the victim node's peer diversity and enabling targeted censorship of transactions and blocks on inbound relay paths. This constitutes a partial eclipse attack on inbound connections. This maps to **High** severity: a bad design that can cause CKB network degradation (reduced peer diversity, inbound relay censorship) with minimal cost to the attacker.

## Likelihood Explanation

- Establishing 16 inbound connections from distinct IPs is straightforward; no per-IP cap exists in the eviction path.
- Responding to pings within < 1 second is trivially achievable on any modern network.
- Sending unsolicited ping messages is a standard P2P protocol operation; `ping_received` imposes no rate limit and is reachable by any connected peer.
- No privileged access, leaked keys, or majority hashpower is required.
- The attack is repeatable: each time an honest peer connects while the attacker holds 15 slots, the attacker's 16th connection evicts it with probability 1.0.

## Recommendation

1. **Use sub-second RTT granularity**: Replace `p.as_secs()` with `p.as_millis()` or `p.as_micros()` in the Tier 1 comparator (`peer_registry.rs` L157) so that sub-second attacker peers cannot all tie at 0.

2. **Separate ping-send and ping-receive timestamps**: `last_ping_protocol_message_received_at` should only be updated on *pong* receipt (in `pong_received`, `ping.rs` L71–79), not on receipt of an unsolicited incoming ping (`ping_received`, `ping.rs` L62–68). This prevents an attacker from refreshing the Tier 2 timestamp by spamming pings.

3. **Add per-IP or per-/16-subnet inbound connection limits** upstream of the eviction path to raise the cost of establishing 16 simultaneous inbound connections.

4. **Add a network-group diversity check** to the protection tiers so that a single network group cannot fill an entire protection tier.

## Proof of Concept

Corrected minimal reproducible sequence:

```rust
let mut peer_store = PeerStore::default();
let mut registry = PeerRegistry::new(16, 3, false, vec![], true);

// Step 1: Attacker group A — 8 peers, ping_rtt = 500ms → as_secs() == 0 (Tier 1 protected)
for i in 0..8u32 {
    registry.accept_peer(attacker_addr_a(i), session_a(i), RawSessionType::Inbound, &mut peer_store).unwrap();
    registry.get_peer_mut(session_a(i)).unwrap().ping_rtt = Some(Duration::from_millis(500));
}

// Step 2: 1 honest peer — no ping data, no recent messages (9 total, no eviction)
registry.accept_peer(honest_addr(), session_honest(), RawSessionType::Inbound, &mut peer_store).unwrap();

// Step 3: Attacker group B — 7 peers, last_ping_protocol_message_received_at = now (16 total)
for i in 0..7u32 {
    registry.accept_peer(attacker_addr_b(i), session_b(i), RawSessionType::Inbound, &mut peer_store).unwrap();
    registry.get_peer_mut(session_b(i)).unwrap()
        .last_ping_protocol_message_received_at = Some(Instant::now());
}

// Step 4: 8th B attacker (17th connection) triggers eviction
// Tier 1: 8A protected; Tier 2: 7B protected; Tier 3: 1>>1=0; honest is sole candidate
let result = registry.accept_peer(attacker_addr_b(7), session_b(7), RawSessionType::Inbound, &mut peer_store).unwrap();
let evicted = result.unwrap();
assert_eq!(evicted.session_id, session_honest()); // honest peer evicted with p=1.0
```
