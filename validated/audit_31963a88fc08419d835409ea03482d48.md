Audit Report

## Title
Unbounded `pending_get_block_proposals` Growth Enables Resource Exhaustion via Unauthenticated Peers - (File: sync/src/relayer/get_block_proposal_process.rs, sync/src/types/mod.rs, sync/src/relayer/mod.rs)

## Summary
`SyncState::pending_get_block_proposals` is an uncapped `DashMap` that any connected peer can grow without bound by repeatedly sending `GetBlockProposal` messages containing non-existent proposal IDs. The periodic `prune_tx_proposal_request` timer drains the entire map in a single pass and forwards all accumulated IDs to the tx-pool service in one unbounded `fetch_txs` call, causing proportional memory, CPU, and tx-pool service degradation.

## Finding Description

**No size cap on the map.** `pending_get_block_proposals` is declared as a plain `DashMap` with no capacity limit: [1](#0-0) 

**Unchecked insertion.** `insert_get_block_proposals` inserts every proposal ID from every peer with no guard on total map size: [2](#0-1) 

**Per-message check is the only guard.** `GetBlockProposalProcess::execute()` rejects messages only if the count exceeds `max_block_proposals_limit * max_uncles_num` (≈3000 on mainnet). There is no per-peer rate limit, no global cap, and no validation that proposal IDs correspond to any known compact block: [3](#0-2) 

**Unconditional insertion of non-existent proposals.** Proposals absent from the local tx pool are inserted into the map without any further check: [4](#0-3) 

**Unbounded drain on every timer tick.** `drain_get_block_proposals` clones the entire map and clears it: [5](#0-4) 

`prune_tx_proposal_request` then sends the full cloned map to the tx-pool service in a single `fetch_txs` call and iterates all entries: [6](#0-5) 

The exploit path is: attacker opens N connections → each sends repeated `GetBlockProposal` messages with up to 3000 random, non-existent `ProposalShortId` values → all are inserted into the map → on each timer tick the node clones the entire map, sends all IDs to the single-threaded tx-pool service in one call, and iterates all entries.

## Impact Explanation

This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* Concretely:

- **Memory exhaustion**: the DashMap grows proportionally to `N_peers × 3000 × messages_per_tick_interval`, each entry being a 10-byte `ProposalShortId` key plus a `HashSet<PeerIndex>`.
- **CPU exhaustion**: every timer tick iterates the entire drained map and constructs per-peer response batches, all proportional to the unbounded map size.
- **Tx-pool service blocking**: `fetch_txs` sends an arbitrarily large `HashSet` to the single-threaded tx-pool service and awaits a response, stalling transaction admission and block assembly for all other callers.

## Likelihood Explanation

Any unauthenticated peer can trigger this. No stake, authentication, or special role is required. The per-message limit of ≈3000 is generous, there is no per-peer rate limit, and there is no global cap. A single attacker with a handful of connections sending messages at the timer interval rate can sustain the attack indefinitely.

## Recommendation

1. **Cap total map size** in `insert_get_block_proposals`: enforce a global `MAX_INFLIGHT_PROPOSALS` constant and drop new entries when the cap is reached.
2. **Validate against known compact blocks**: only accept proposal IDs that appear in a compact block already stored in `pending_compact_blocks`, rejecting unsolicited `GetBlockProposal` messages.
3. **Chunk the drain**: in `prune_tx_proposal_request`, process the drained map in bounded batches (e.g., 1000 entries per `fetch_txs` call) rather than forwarding the entire set at once.
4. **Per-peer rate limiting**: track per-peer injection counts and disconnect/ban peers exceeding a threshold within a time window.

## Proof of Concept

1. Attacker opens N TCP connections to the victim CKB node and completes the p2p handshake.
2. Each connection sends `GetBlockProposal` relay messages in a tight loop, each containing 3000 unique random 10-byte `ProposalShortId` values that do not exist in the victim's tx pool.
3. Because none of the IDs are in the tx pool, all pass the `filter` at line 68–71 of `get_block_proposal_process.rs` and are inserted into `pending_get_block_proposals` via `insert_get_block_proposals` with no size check.
4. Between timer ticks the map accumulates `N × 3000 × messages_sent` entries.
5. On the next `prune_tx_proposal_request` tick, the node clones the entire map (`drain_get_block_proposals`), sends all IDs to the tx-pool service in a single `fetch_txs` call, and iterates all entries — causing proportional CPU, memory, and tx-pool latency spikes.
6. Sustained attack keeps the map perpetually large, degrading relay and tx-pool performance until the node becomes unresponsive or OOM-killed.

    async fn prune_tx_proposal_request(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let get_block_proposals = self.shared().state().drain_get_block_proposals();
        let tx_pool = self.shared.shared().tx_pool_controller();

        let fetch_txs = tx_pool
            .fetch_txs(
                get_block_proposals
                    .iter()
                    .map(|kv_pair| kv_pair.key().clone())
                    .collect(),
            )
            .await;
        if let Err(err) = fetch_txs {
            debug_target!(
                crate::LOG_TARGET_RELAY,
                "relayer prune_tx_proposal_request internal error: {:?}",
                err,
            );
            return;
        }

        let txs = fetch_txs.unwrap();

        let mut peer_txs = HashMap::new();
        for (id, peer_indices) in get_block_proposals.into_iter() {
            if let Some(tx) = txs.get(&id) {
                for peer_index in peer_indices {
                    let tx_set = peer_txs.entry(peer_index).or_insert_with(Vec::new);
                    tx_set.push(tx.clone());
                }
            }
        }

        let mut relay_bytes = 0;
        let mut relay_proposals = Vec::new();
        for (peer_index, txs) in peer_txs {
            for tx in txs {
                let data = tx.data();
                let tx_size = data.total_size();
                if relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH {
                    send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals))
                        .await;
                    relay_bytes = tx_size;
                } else {
                    relay_bytes += tx_size;
                }
                relay_proposals.push(data);
            }
            if !relay_proposals.is_empty() {
                send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals)).await;
                relay_bytes = 0;
            }
        }
