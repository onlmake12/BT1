Audit Report

## Title
Unbounded Concurrent Decompression Memory: N Peers × 8 MB Per-Frame Allocation With No Global Budget — (`network/src/compress.rs`)

## Summary

`LengthDelimitedCodecWithCompress::decode` allocates a `BytesMut` buffer sized to the full declared decompressed length (up to 8 MB) for every incoming compressed frame, with no global or per-node limit on how many such allocations may be live simultaneously. An unprivileged remote peer can open up to `max_peers` connections and send one max-amplification snappy frame per connection, causing up to 125 × 8 MB ≈ 1 GB of concurrent heap allocation with no aggregate bound.

## Finding Description

**Root cause — per-frame guard with no aggregate budget:**

In `LengthDelimitedCodecWithCompress::decode` (`network/src/compress.rs`, lines 232–249), after the inner `length_delimited` codec accepts a compressed wire frame, the code reads the snappy-declared decompressed length and rejects only frames whose declared size exceeds `MAX_UNCOMPRESSED_LEN` (8 MB = `1 << 23`):

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {
    return Err(io::ErrorKind::InvalidData.into());
}
let mut buf = BytesMut::zeroed(decompressed_bytes_len);  // up to 8 MB allocated here
``` [1](#0-0) [2](#0-1) 

There is no semaphore, counter, or token bucket anywhere in the codec layer. A search across the entire codebase confirms no global decompression budget (`DECOMPRESS_IN_FLIGHT`, `Semaphore`, or equivalent) exists.

The `max_frame_length` check enforced by the inner `length_delimited` codec bounds only the **compressed wire frame**:
- Sync: 2 MB
- RelayV3: 4 MB [3](#0-2) 

This is set via the codec builder in `CKBProtocol::build()`: [4](#0-3) 

The decompressed output is separately bounded only by `MAX_UNCOMPRESSED_LEN = 8 MB`, creating a compression amplification gap. A snappy-compressed frame of ~8 KB on the wire (well within the 2 MB Sync `max_frame_length`) can declare and trigger an 8 MB decompressed allocation.

**Exploit flow:**

1. Attacker opens up to 125 TCP connections to the node's P2P port.
2. Completes the tentacle noise handshake (standard protocol, no PoW or keys required).
3. Opens the Sync or RelayV3 protocol stream.
4. On each connection, sends a snappy-compressed frame whose payload is 8 MB of zeros (~8 KB compressed — within the 2 MB Sync `max_frame_length`).
5. Each `decode` call passes the per-frame guard (`decompressed_bytes_len ≤ 8 MB`) and executes `BytesMut::zeroed(8_388_608)`.
6. With 125 connections all sending simultaneously, 125 × 8 MB ≈ 1 GB of heap is live concurrently.

**Why existing checks are insufficient:**

- The per-frame `MAX_UNCOMPRESSED_LEN` guard (line 235) only prevents a single frame from exceeding 8 MB; it does not limit aggregate concurrent allocations.
- The `max_frame_length` check bounds only the compressed wire size, not the decompressed output.
- No global decompression semaphore or budget exists anywhere in the codebase.

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Peak concurrent decompression heap usage is bounded only by `max_peers × MAX_UNCOMPRESSED_LEN`. On common CKB validator deployments (2–4 GB RAM VPS), this alone can exhaust available memory, triggering the OOM killer and crashing the node. The attacker sustains the pressure by continuously re-sending max-amplification frames on each connection.

## Likelihood Explanation

The attack requires only: (1) opening TCP connections to the public P2P port, and (2) completing the standard tentacle noise handshake and sending a crafted snappy-compressed frame — achievable with any snappy library and a tentacle client implementation. No proof of work, no cryptographic keys, no privileged role, and no victim mistakes are required. The attack is repeatable and can be sustained indefinitely by holding connections open and re-sending frames.

## Recommendation

Introduce a node-wide decompression memory budget enforced before `BytesMut::zeroed`:

```rust
static DECOMPRESS_IN_FLIGHT: AtomicUsize = AtomicUsize::new(0);
const MAX_TOTAL_DECOMPRESS: usize = 64 * 1024 * 1024; // 64 MB node-wide

// In decode(), before BytesMut::zeroed:
let prev = DECOMPRESS_IN_FLIGHT.fetch_add(decompressed_bytes_len, Ordering::Relaxed);
if prev + decompressed_bytes_len > MAX_TOTAL_DECOMPRESS {
    DECOMPRESS_IN_FLIGHT.fetch_sub(decompressed_bytes_len, Ordering::Relaxed);
    return Err(io::ErrorKind::InvalidData.into());
}
// ... allocate, decompress, then subtract on completion/error
```

Alternatively, lower `MAX_UNCOMPRESSED_LEN` per-protocol to match `max_frame_length` (eliminating the amplification gap), or use a `tokio::sync::Semaphore` with a small permit count to bound concurrent decompression operations.

## Proof of Concept

1. Implement a tentacle noise client (using the `tentacle` crate or equivalent).
2. Open 125 connections to the target node's P2P port and complete the noise handshake + Sync protocol negotiation on each.
3. On each connection, send a snappy-compressed frame: `snappy::compress(&vec![0u8; 8 * 1024 * 1024])`, prefixed with the 4-byte length and `COMPRESS_FLAG` byte (`0x80`).
4. Send all frames simultaneously.
5. Monitor target node RSS: the delta should approach 125 × 8 MB ≈ 1 GB, confirming `BytesMut::zeroed` is called 125 times concurrently with no aggregate guard in `network/src/compress.rs` lines 235–242. [5](#0-4)
