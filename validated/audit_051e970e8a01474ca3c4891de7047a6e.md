### Title
PostgreSQL-Only `script_exists_in_output` Reads Stale `row_lock` Instead of `row_type`, Permanently Corrupting Script Table on Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

In `script_exists_in_output`, the second `match` block (line 252) reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, `try_get::<bool, _>` succeeds on `row_lock` (returning `false`, since the script was already confirmed to not be a lock_script), so the function returns `Ok(false)` without ever consulting `row_type`. This causes any script that is referenced **only** as a `type_script_id` in surviving outputs to be incorrectly deleted from the `script` table during rollback. On SQLite the bug is masked because `try_get::<bool, _>` always fails (SQLite returns `BIGINT`), so the fallback `Err(_)` branch correctly reads `row_type`.

---

### Finding Description

`script_exists_in_output` is called during `rollback_block` to decide whether a script row is still referenced by any surviving output before deleting it. [1](#0-0) 

The function issues two SQL queries — one checking `lock_script_id`, one checking `type_script_id` — and is supposed to return `true` if either matches. [2](#0-1) 

After the `row_type` query is fetched (lines 237–249), the second `match` block at line 252 mistakenly re-reads `row_lock` instead of `row_type`: [3](#0-2) 

**PostgreSQL execution path for a type-script-only script S:**

| Step | Variable | Value | Reason |
|------|----------|-------|--------|
| Line 223 | `row_lock.try_get::<bool,_>(0)` | `Ok(false)` | PG returns BOOLEAN; S is not a lock_script |
| Line 225 | `r` | `false` | no early return |
| Lines 237–249 | `row_type` fetched | would be `true` | S IS a type_script in surviving outputs |
| Line 252 | `row_lock.try_get::<bool,_>(0)` | `Ok(false)` | **reads stale `row_lock` again** |
| Line 253 | return | `Ok(false)` | **wrong — should be `Ok(true)`** |

**SQLite execution path (bug masked):**

`try_get::<bool,_>` always returns `Err(_)` on SQLite (column is `BIGINT`), so both `match` blocks fall to the `Err(_)` arm. The first arm correctly reads `row_lock.get::<i64,_>(0)`, and the second arm correctly reads `row_type.get::<i64,_>(0)`. [4](#0-3) 

All existing rollback tests use `connect_sqlite(MEMORY_DB)`, so the PostgreSQL path is never exercised: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

After rollback of block N on a PostgreSQL-backed node:

1. All outputs from block N are deleted from the `output` table (line 25).
2. For each script referenced by those outputs, `script_exists_in_output` is called.
3. For any script S that was used **only** as a `type_script_id` (not as a `lock_script_id`) in any surviving output, the function incorrectly returns `false`.
4. S is added to `script_id_list_to_remove` and deleted from the `script` table.
5. Subsequent `get_cells(type_script=S)` or `get_transactions(type_script=S)` RPC calls silently return zero results, even for cells from blocks that were never rolled back. [7](#0-6) 

The rich-indexer's view of live cells permanently diverges from canonical chain state. Wallets and dApps relying on the rich-indexer RPC will see incorrect cell sets, potentially causing missed UTXOs, incorrect balance displays, or failed transactions.

---

### Likelihood Explanation

- Short reorgs (1–2 blocks) occur naturally on any PoW network without any attacker involvement.
- The PostgreSQL backend is a supported production configuration explicitly documented in the CKB rich-indexer.
- Any block containing an output whose `type_script` is not also used as a `lock_script` in any other indexed output (a common pattern for type-ID cells, UDT cells, etc.) will trigger the corruption on the next rollback.
- The fix is a one-character variable name change (`row_lock` → `row_type` at line 252), confirming this is a straightforward copy-paste error.

---

### Recommendation

Change line 252 from:


to:

Add a PostgreSQL-backed rollback test that inserts a block with an output whose `type_script` is unique (not used as a `lock_script` anywhere), appends a second block, rolls back, and asserts the script row still exists and `get_cells(type_script=S)` returns the correct result.
