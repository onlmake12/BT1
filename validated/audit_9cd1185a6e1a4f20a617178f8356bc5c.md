### Title
Duplicate Entries in `unknown_header_list` via Unchecked `insert_unknown_header_hash` During IBD Sync — (`sync/src/types/mod.rs`)

### Summary
During Initial Block Download (IBD), a malicious peer can send a `GetHeaders` message whose `block_locator_hashes` field contains duplicate unknown hashes. The receiving node inserts every entry into `PeerState::unknown_header_list` (a plain `Vec<Byte32>`) without any uniqueness check. The `BlockFetcher` then processes this list in a loop, re-inserting the same unknown hash on each iteration and breaking early, effectively stalling IBD progress for that peer slot.

### Finding Description

`PeerState::unknown_header_list` is declared as a plain `Vec<Byte32>`: [1](#0-0) 

The function that populates it, `insert_unknown_header_hash`, unconditionally calls `.push()` with no duplicate check: [2](#0-1) 

The caller, `insert_peer_unknown_header_list`, iterates over the peer-supplied `block_locator_hashes` and calls `insert_unknown_header_hash` for every hash not yet in the header map: [3](#0-2) 

`GetHeadersProcess::execute()` feeds the raw peer-supplied locator directly into this function during IBD, after only a size check (`> MAX_LOCATOR_SIZE`) but **no uniqueness check**: [4](#0-3) 

`BlockFetcher::fetch()` then drains the list from the tail. When it encounters an unknown hash it re-inserts it and breaks: [5](#0-4) 

If the list contains N copies of the same unknown hash, the fetcher pops one, re-inserts it, and breaks — every single invocation — never advancing past that entry until the block is actually downloaded.

### Impact Explanation

A malicious peer can craft a `GetHeaders` message with up to `MAX_LOCATOR_SIZE` copies of the same unknown block hash. The receiving node, while in IBD, inserts all copies into `unknown_header_list`. The `BlockFetcher` loop stalls on the repeated unknown hash, preventing the node from advancing `best_known_header` for that peer and blocking concurrent IBD block downloads from that peer slot. Because the guard `unknown_header_list_is_empty` means the list is populated only once per peer, the peer slot remains degraded until the duplicated hash is resolved.

### Likelihood Explanation

Any unprivileged peer that connects to a node in IBD mode can send a crafted `GetHeaders` message. No authentication or privilege is required. The only existing guard is a count check (`MAX_LOCATOR_SIZE`), not a uniqueness check. The attack is trivially constructable.

### Recommendation

Add a deduplication step before inserting into `unknown_header_list`. The simplest fix is to change `unknown_header_list` from `Vec<Byte32>` to a `LinkedHashSet<Byte32>` (already available in the codebase at `util/src/linked_hash_set.rs`) to preserve insertion order while enforcing uniqueness. Alternatively, add an explicit contains-check inside `insert_unknown_header_hash` before calling `.push()`, or deduplicate the incoming `header_list` in `insert_peer_unknown_header_list` before iterating.

### Proof of Concept

1. Connect to a CKB node that is in IBD mode (outbound/whitelist peer).
2. Send a `SyncMessage::GetHeaders` with `block_locator_hashes` set to `MAX_LOCATOR_SIZE` copies of a single hash that is not yet in the node's header map.
3. The node calls `insert_peer_unknown_header_list`, which iterates all entries and calls `insert_unknown_header_hash` for each — inserting `MAX_LOCATOR_SIZE` identical hashes into `unknown_header_list` (a `Vec`).
4. On every subsequent `BlockFetcher::fetch()` invocation for this peer, the fetcher pops the last copy, fails to find it in the header map, re-inserts it, and breaks — never processing the rest of the list.
5. The peer slot's `best_known_header` is never advanced, and no blocks are fetched from this peer until the duplicated hash is resolved. [2](#0-1) [6](#0-5) [5](#0-4)

    pub fn insert_peer_unknown_header_list(&self, pi: PeerIndex, header_list: Vec<Byte32>) {
        // update peer's unknown_header_list only once
        if self.state().peers.unknown_header_list_is_empty(pi) {
            // header list is an ordered list, sorted from highest to lowest,
            // so here you discard and exit early
            for hash in header_list {
                if let Some(header) = self.shared().header_map().get(&hash) {
                    self.state()
                        .peers
                        .may_set_best_known_header(pi, header.as_header_index());
                    break;
                } else {
                    self.state().peers.insert_unknown_header_hash(pi, hash)
                }
            }
        }
    }
