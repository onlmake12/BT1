Audit Report

## Title
`check_tx_fee` Minimum Fee Gate Uses Serialized Size Only, Ignoring Cycle-Based Weight — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` computes the admission minimum fee using only the transaction's serialized byte size, while every post-admission weight-sensitive operation (fee rate, eviction key, miner scoring) uses `get_transaction_weight(size, cycles) = max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`. A transaction with minimal serialized size but near-maximum cycles is admitted at a fee calibrated only to its byte size, yet its effective weight-adjusted fee rate inside the pool is orders of magnitude below `min_fee_rate`. Such transactions are never selected by miners, persist for 12 hours, and can be used to fill the 180 MB pool at a ~47× cost discount relative to the protocol's intended weight-based pricing.

## Finding Description

**Root cause — `check_tx_fee` (`tx-pool/src/util.rs` L42–53):**

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
```

The comment explicitly acknowledges the approximation. Cycles are not yet known at this call site because `check_tx_fee` is invoked during `pre_check`, before `verify_rtx` executes scripts and determines actual cycle consumption.

**Weight function used everywhere else (`util/types/src/core/tx_pool.rs` L298–303):**

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

with `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` (L279).

**Post-admission operations all use weight:**
- `TxEntry::fee_rate()` (`entry.rs` L115–118): `FeeRate::calculate(self.fee, get_transaction_weight(self.size, self.cycles))`
- `EvictKey` via `From<&TxEntry>` (`entry.rs` L234–247): uses `get_transaction_weight` for both entry and descendants weight
- `AncestorsScoreSortKey` (`entry.rs` L221–232): uses `get_transaction_weight` for miner selection scoring

**Pool size limit (`pool.rs` L292–329):**

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict by lowest EvictKey (weight-adjusted fee rate)
}
```

`total_tx_size` is the sum of serialized byte sizes, not weight. Eviction only triggers after the pool is already full.

**Exploit arithmetic (250-byte tx, 69M cycles):**
- `get_transaction_weight(250, 69_000_000) = max(250, 11769) = 11769`
- Admission min_fee (size-only): `1000 * 250 / 1000 = 250 shannons`
- True weight min_fee: `1000 * 11769 / 1000 = 11769 shannons`
- Attacker pays 251 shannons, passes `check_tx_fee`
- Effective fee rate inside pool: `251 * 1000 / 11769 ≈ 21 shannons/KW` — far below `min_fee_rate = 1000`
- Miners never select it; it persists 12 hours

**No post-verification fee check exists.** After `verify_rtx` returns with actual cycles, `submit_entry` → `_submit_entry` → `limit_size` is called, but `limit_size` only evicts when the pool is already over `max_tx_pool_size`, and the newly admitted transaction is already in the pool at that point.

## Impact Explanation

This matches **High (10001–15000 points): "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

An attacker can fill the 180 MB tx-pool (`180_000_000 / 250 = 720,000` transactions) at a total cost of ~180,720 shannons, whereas the weight-based cost should be ~8,473,680 shannons — a ~47× discount. Legitimate transactions with proper fee rates may be evicted (by `limit_size` selecting lowest `EvictKey`) or delayed. The attack is sustained (12-hour expiry) and continuously repeatable.

## Likelihood Explanation

No special privilege is required. Any RPC caller or P2P relay peer can submit transactions. The attacker needs a lock script that executes a tight RISC-V loop consuming near `max_tx_verify_cycles = 70,000,000` cycles with minimal serialized size — straightforward with CKB-VM. The fee cost per transaction is 251 shannons. The 12-hour expiry and weight-based eviction provide partial mitigation but do not prevent sustained pool pressure since the attacker can continuously resubmit as transactions expire.

## Recommendation

After `verify_rtx` completes and actual cycles are known, perform a second fee check using the true weight before finalizing pool admission:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64()));
}
```

The existing size-only check in `check_tx_fee` can remain as a cheap pre-verification filter. The post-verification check closes the gap between admission policy and actual resource cost, consistent with how `TxEntry::fee_rate()` and `AncestorsScoreSortKey` already compute effective fee rates.

## Proof of Concept

1. Deploy a CKB lock script containing a tight RISC-V loop consuming ~69,000,000 cycles (just under `max_tx_verify_cycles`).
2. Construct a minimal transaction (~250 bytes serialized) spending a cell locked by this script.
3. With default `min_fee_rate = 1000` shannons/KW, submit with fee = 251 shannons via `send_transaction` RPC.
4. `check_tx_fee` passes (251 > 250). `verify_rtx` executes the script, consuming ~69M cycles. `TxEntry` is created with `cycles ≈ 69_000_000`.
5. Verify inside the pool: `fee_rate() = FeeRate::calculate(251, get_transaction_weight(250, 69_000_000)) = 251*1000/11769 ≈ 21 shannons/KW` — far below `min_fee_rate`.
6. Confirm miners never select this transaction (block template assembly skips it due to low weight-adjusted fee rate).
7. Repeat ~720,000 times to fill the 180 MB pool (`180_000_000 / 250 = 720,000` transactions), total cost ≈ 180,720 shannons.
8. Observe that legitimate transactions with proper fee rates are evicted via `limit_size` or delayed.
