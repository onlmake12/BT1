### Title
Deposit to Stake Account Custody Bypasses `freeze` Guard, Trapping Tokens When Protocol Is Frozen — (`governance/pyth_staking_sdk/src/pyth-staking-client.ts`)

---

### Summary

The Pyth staking program's `GlobalConfig` contains a `freeze: bool` flag and a corresponding `Frozen` error (code 6017). Instructions that go through the on-chain program (e.g., `withdraw_stake`, `create_position`) load the `config` account and can enforce this flag. However, `depositTokensToStakeAccountCustody` in the SDK sends a raw SPL token `createTransferInstruction` directly to the custody PDA, bypassing the staking program entirely. When `freeze = true`, deposits are not blocked, but withdrawals (which route through the program) are. Any user who deposits during a freeze period cannot recover their tokens.

---

### Finding Description

`GlobalConfig` carries a `freeze` boolean field: [1](#0-0) 

The program defines error 6017 `Frozen` ("Protocol is frozen") to signal that state-mutating operations must be blocked: [2](#0-1) 

`withdraw_stake` and `create_position` both include `config` in their account lists, meaning the on-chain handler can (and is expected to) check `freeze` before proceeding: [3](#0-2) 

By contrast, `depositTokensToStakeAccountCustody` constructs a plain SPL token transfer instruction — it never invokes the staking program at all: [4](#0-3) 

The custody PDA is a standard SPL token account; any holder of the PYTH token can transfer into it with only their own wallet signature. The staking program has no hook to intercept incoming SPL transfers, so `freeze` is never consulted.

The same bypass exists in `createStakeAccountAndDeposit`, which also ends with a raw `createTransferInstruction`: [5](#0-4) 

---

### Impact Explanation

When governance sets `freeze = true` (e.g., before a migration or emergency halt), the intent is to freeze all staking activity. A user who deposits PYTH tokens into the custody account during this window:

1. Succeeds — the SPL transfer is accepted on-chain.
2. Cannot call `withdraw_stake` — the instruction loads `config` and will return `Frozen`.
3. Cannot call `create_position` or `close_position` — same reason.

The deposited tokens are effectively trapped in the custody PDA for the duration of the freeze (or permanently if the freeze is never lifted or the account is migrated away). This is a direct loss-of-funds risk for any staker who deposits while the protocol is frozen.

---

### Likelihood Explanation

- The freeze flag is a governance-controlled emergency mechanism; it is activated precisely when the protocol is in a sensitive state.
- Any unprivileged staking user can trigger this path — no special role is required.
- The UI (`apps/staking`) calls `deposit` which routes to `depositTokensToStakeAccountCustody`: [6](#0-5) 

A user who deposits through the normal UI while the protocol is frozen will lose access to their tokens. The likelihood is moderate: it requires the freeze to be active and a user to deposit concurrently, but the window can last hours or days during a migration.

---

### Recommendation

Replace the raw SPL transfer in `depositTokensToStakeAccountCustody` (and `createStakeAccountAndDeposit`) with a dedicated on-chain instruction (e.g., `deposit_stake`) that loads `config` and asserts `!config.freeze`. This mirrors the pattern already used by `withdraw_stake` and `create_position`.

Alternatively, add a client-side pre-flight check that reads `GlobalConfig.freeze` before constructing the deposit transaction and aborts with a clear error if the protocol is frozen — though an on-chain guard is the only trustless fix.

---

### Proof of Concept

1. Governance calls `update_config` setting `freeze = true`.
2. Attacker (or innocent user) calls `depositTokensToStakeAccountCustody(stakeAccountPositions, 1_000_000n)`.
3. The SDK emits a raw `createTransferInstruction` from the user's ATA to the custody PDA — no staking program instruction is involved: [7](#0-6) 

4. The SPL token program accepts the transfer; tokens land in the custody PDA.
5. User calls `withdrawTokensFromStakeAccountCustody` → `withdraw_stake` on-chain → program loads `config`, sees `freeze = true`, returns error 6017 `Frozen`.
6. Tokens are permanently inaccessible for the duration of the freeze. [8](#0-7)

```typescript
  public async depositTokensToStakeAccountCustody(
    stakeAccountPositions: PublicKey,
    amount: bigint,
  ) {
    const globalConfig = await this.getGlobalConfig();
    const mint = globalConfig.pythTokenMint;

    const senderTokenAccount = getAssociatedTokenAddressSync(
      mint,
      this.wallet.publicKey,
      true,
    );

    const instruction = createTransferInstruction(
      senderTokenAccount,
      getStakeAccountCustodyAddress(stakeAccountPositions),
      this.wallet.publicKey,
      amount,
    );

    return sendTransaction([instruction], this.connection, this.wallet);
  }
```
