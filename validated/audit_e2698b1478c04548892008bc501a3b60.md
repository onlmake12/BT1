### Title
`GAS_OVERHEAD` Is Hardcoded and Chain-Agnostic, Causing Systematic Keeper Underpayment or Subscriber Fund Drain Across Multi-Chain Deployments - (File: `target_chains/ethereum/pulse_sdk/solidity/SchedulerConstants.sol`)

---

### Summary

`GAS_OVERHEAD = 30000` in `SchedulerConstants.sol` is a compile-time constant used to reimburse keepers for transaction overhead in `Scheduler._processFeesAndPayKeeper`. Because it is immutable and not chain-specific, it systematically misprices keeper fees on every EVM chain where the Scheduler is deployed that has a different actual transaction overhead — directly analogous to Moonwell's `borrowRateMaxMantissa` being a single constant applied across chains with different block times.

---

### Finding Description

`SchedulerConstants.sol` declares:

```solidity
/// Fixed gas overhead component used in keeper fee calculation.
/// This is a rough estimate of the tx overhead for a keeper to call updatePriceFeeds.
uint256 public constant GAS_OVERHEAD = 30000;
``` [1](#0-0) 

This constant is consumed in `Scheduler._processFeesAndPayKeeper`:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
``` [2](#0-1) 

The total keeper fee is then:

```solidity
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [3](#0-2) 

and is deducted from the subscriber's balance and transferred to `msg.sender` (the keeper):

```solidity
status.balanceInWei -= totalKeeperFee;
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [4](#0-3) 

The `SchedulerUpgradeable` contract is a UUPS proxy with no mechanism to update `GAS_OVERHEAD` without a full contract upgrade, and the constant is baked into the inherited `SchedulerConstants` base: [5](#0-4) 

The Scheduler is an EVM contract explicitly designed for multi-chain deployment under `target_chains/ethereum/contracts`. Actual transaction base overhead varies significantly by chain:

| Chain | Approx. base tx overhead | GAS_OVERHEAD | Delta |
|---|---|---|---|
| Ethereum mainnet | ~21,000 gas | 30,000 | +9,000 (overpays keeper) |
| Arbitrum (L2) | 21,000 gas + L1 calldata fee (can be 50,000–200,000+ gas-equivalent) | 30,000 | severely underpays keeper |
| BNB Chain | ~21,000 gas | 30,000 | +9,000 (overpays keeper) |
| Optimism/Base | 21,000 gas + L1 data fee | 30,000 | underpays keeper |

---

### Impact Explanation

**Two distinct impacts depending on chain:**

1. **Keeper underpayment (L2 chains with L1 data fees):** On Arbitrum, Optimism, Base, and similar L2s, the actual gas overhead for a transaction includes an L1 data fee component that can far exceed 30,000 gas-equivalent. Keepers are systematically underpaid for their real costs. Rational keepers will stop calling `updatePriceFeeds`, causing all subscriptions on that chain to receive stale prices. Applications consuming prices via `getPricesNoOlderThan` will revert with `StalePrice`, and those using `getPricesUnsafe` will silently consume outdated data.

2. **Subscriber fund drain (chains where overhead < 30,000):** On chains where the actual overhead is lower than 30,000 gas, every keeper call overpays the keeper from the subscriber's balance. An unprivileged keeper can repeatedly call `updatePriceFeeds` with valid updates (satisfying heartbeat or deviation criteria) to drain subscription balances faster than the subscriber expects, causing premature subscription deactivation.

---

### Likelihood Explanation

The Scheduler is deployed as a multi-chain EVM contract. Pyth already operates on dozens of EVM chains including Arbitrum, Optimism, Base, BNB, and others. L2 chains with L1 data fees are among the most actively used Pyth deployment targets. Any keeper on an L2 chain will immediately observe that their actual gas cost exceeds the reimbursement, creating a concrete and immediate disincentive. The entry path requires no privilege: any address can call `updatePriceFeeds` as a keeper. [6](#0-5) 

---

### Recommendation

Replace the compile-time constant with a storage variable set at initialization and updatable by governance:

```solidity
// In SchedulerState.sol
uint256 gasOverhead;

// In SchedulerGovernance.sol
function setGasOverhead(uint256 newGasOverhead) external onlyAdmin {
    _state.gasOverhead = newGasOverhead;
}
```

Alternatively, for L2 chains, integrate a chain-specific L1 fee oracle (e.g., Optimism's `GasPriceOracle` or Arbitrum's `ArbGasInfo`) to dynamically compute the actual overhead component. At minimum, `GAS_OVERHEAD` should be set per-chain at deployment time rather than hardcoded as a universal constant.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` on Arbitrum (or any L2 with L1 data fees).
2. Create a subscription with a heartbeat update criterion.
3. As a keeper, call `updatePriceFeeds` with a valid Pyth update.
4. Observe that `gasCost = (startGas - gasleft() + 30000) * tx.gasprice` does not include the L1 data fee component.
5. The keeper's actual ETH cost (including L1 fee) exceeds `totalKeeperFee` paid out.
6. Repeat: no rational keeper will continue calling `updatePriceFeeds`; all subscriptions on that chain receive stale prices.

Conversely, on a chain where actual overhead < 30,000 gas:
1. A keeper calls `updatePriceFeeds` on every valid heartbeat interval.
2. Each call, the keeper collects `(30000 - actual_overhead) * tx.gasprice` in excess fees from the subscriber's balance.
3. The subscriber's balance depletes faster than the documented rate, causing premature subscription deactivation. [1](#0-0) [7](#0-6)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
