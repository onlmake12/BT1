### Title
Silently Ignored Return Values from Critical Tx-Pool Control Operations Cause Permanent Tx-Pool Suspension — (File: `chain/src/verify.rs`)

---

### Summary

In `chain/src/verify.rs`, the `ConsumeUnverifiedBlocks::start()` loop calls `tx_pool_controller.suspend_chunk_process()` and `tx_pool_controller.continue_chunk_process()` using the `let _ = ...` pattern, which silently discards any error returned. This is the direct Rust analog to the ERC20 `transferFrom()` unchecked-return-value class: a critical state-changing call whose failure is never observed. If `continue_chunk_process()` fails silently after a block is processed, the tx-pool's chunk processor remains permanently suspended, halting all new transaction admission without any error, log, or recovery path.

---

### Finding Description

In `chain/src/verify.rs`, inside `ConsumeUnverifiedBlocks::start()`:

```rust
// line 81
let _ = self.tx_pool_controller.suspend_chunk_process();

// ... block verification runs here ...

// line 101
let _ = self.tx_pool_controller.continue_chunk_process();
``` [1](#0-0) [2](#0-1) 

Both calls return a `Result`, but the `let _ =` binding unconditionally discards it. The `TxPoolController` communicates with the tx-pool service over an internal channel. If that channel is closed, full, or the service has exited, the send fails and the error is silently swallowed.

The critical failure mode is on line 101: if `continue_chunk_process()` fails after `suspend_chunk_process()` succeeded, the tx-pool's chunk processor is left in the suspended state with no mechanism to resume it. The block-processing loop continues normally, giving no indication that anything went wrong.

Compare this to the analogous pattern in `notify/src/lib.rs` where `Request::call` also uses `let _ =` to discard a channel send error:

```rust
// lines 51-56
let _ = sender
    .send(Request { responder, arguments })
    .await;
response.await.ok()
``` [3](#0-2) 

There, callers immediately `.expect()` on the `None` result, causing a panic. In the tx-pool case the failure is even more insidious: no panic, no log, just a permanently frozen tx-pool.

---

### Impact Explanation

Once the tx-pool chunk processor is permanently suspended, the node stops admitting or re-validating transactions. From the perspective of any RPC caller or transaction submitter:

- `send_transaction` RPC calls queue but are never processed.
- Block assembly (`get_block_template`) stalls waiting for the chunk processor.
- The node appears alive (P2P and sync continue) but is effectively a zombie for transaction processing.

This is a **local denial-of-service** against the node's transaction-processing subsystem, reachable without any privileged access.

---

### Likelihood Explanation

The `ConsumeUnverifiedBlocks::start()` loop is entered every time a valid block arrives from a peer. The tx-pool controller channel can fail if:

1. The tx-pool service has exited or is restarting (e.g., due to an OOM condition or a separate bug).
2. The internal channel buffer is exhausted under high block-relay load.

An unprivileged block relayer submitting a stream of valid blocks during a period of tx-pool instability can reliably trigger this window. Because the error is silently discarded, the operator has no indication of the failure until they notice the node is no longer processing transactions.

---

### Recommendation

Replace the `let _ =` pattern with proper error handling for both calls. At minimum, log an error and attempt recovery (e.g., retry `continue_chunk_process`). Ideally, treat a failure of `continue_chunk_process` as fatal and restart the affected service:

```rust
// Before (unsafe):
let _ = self.tx_pool_controller.suspend_chunk_process();
// ...
let _ = self.tx_pool_controller.continue_chunk_process();

// After (safe):
if let Err(e) = self.tx_pool_controller.suspend_chunk_process() {
    error!("Failed to suspend tx-pool chunk process: {}", e);
}
// ...
if let Err(e) = self.tx_pool_controller.continue_chunk_process() {
    error!("CRITICAL: Failed to resume tx-pool chunk process: {}", e);
    // trigger node shutdown or service restart
}
```

Apply the same fix to the analogous `let _ = sender.send(...).await` in `notify/src/lib.rs` `Request::call`, where a silent send failure causes downstream `.expect()` panics in all subscribers. [4](#0-3) 

---

### Proof of Concept

1. A peer relays a valid block to the target node, triggering `ConsumeUnverifiedBlocks::start()`.
2. At line 81, `suspend_chunk_process()` succeeds; the tx-pool chunk processor is paused.
3. Between lines 81 and 101, the tx-pool service exits (e.g., due to OOM, a separate panic, or a crafted sequence of heavy transactions that exhausts the service).
4. At line 101, `continue_chunk_process()` returns `Err(...)`, which is discarded by `let _ =`.
5. The tx-pool chunk processor remains suspended. No log is emitted.
6. All subsequent `send_transaction` RPC calls and block-assembly requests stall indefinitely.
7. The node continues syncing headers and relaying blocks, masking the failure from external monitoring. [5](#0-4)
