### Title
Governance VAA Sequence Gap Allows Unprivileged Relayer to Permanently Skip Governance Instructions - (File: `target_chains/ethereum/contracts/contracts/pyth/PythGovernance.sol`)

### Summary
The governance VAA sequence enforcement across all Pyth target-chain contracts uses a `sequence > lastExecutedGovernanceSequence` check rather than a strict `sequence == lastExecutedGovernanceSequence + 1` check. This allows any unprivileged relayer to submit a higher-sequence governance VAA before lower-sequence ones are executed, permanently rendering those intermediate VAAs unexecutable. The EVM `Executor.sol` additionally encodes only a single `callAddress`/`callData` per VAA, meaning complex multi-step governance operations require multiple sequential VAAs — each individually vulnerable to this ordering attack.

### Finding Description

**Root cause — sequence gap check:**

In `PythGovernance.sol::verifyGovernanceVM()`:

```solidity
if (vm.sequence <= lastExecutedGovernanceSequence())
    revert PythErrors.OldGovernanceMessage();
setLastExecutedGovernanceSequence(vm.sequence);
```

The condition only requires the incoming sequence to be *strictly greater than* the last executed one, not the immediate successor. The identical pattern appears in every Pyth target-chain implementation:

- `Executor.sol::verifyGovernanceVM()`: `if (vm.sequence <= lastExecutedSequence) revert ExecutorErrors.MessageOutOfOrder();`
- Aptos `governance.move::parse_and_verify_governance_vaa()`: `assert!(sequence > state::get_last_executed_governance_sequence(), ...)`
- Sui `governance.move::execute_governance_instruction()`: `assert!(sequence > state::get_last_executed_governance_sequence(pyth_state), ...)`
- CosmWasm `contract.rs::execute_governance_instruction()`: `if vaa.sequence <= state.governance_sequence_number { Err(PythContractError::OldGovernanceMessage)?; }`

Because governance VAAs are public Wormhole messages, any unprivileged actor can relay them in any order. If governance has issued VAAs with sequences N, N+1, N+2 and a relayer submits N+2 first, VAAs N and N+1 are permanently unexecutable (their sequences are now ≤ `lastExecutedGovernanceSequence`).

**Compounding factor — single call per EVM Executor VAA:**

`Executor.sol::GovernanceInstruction` encodes exactly one `callAddress` and one `callData` blob per VAA:

```solidity
struct GovernanceInstruction {
    ...
    address callAddress;
    uint value;
    bytes callData;
}
```

Complex multi-step operations (e.g., approve + transfer, or a two-phase contract upgrade) therefore require multiple sequential VAAs. Each intermediate VAA is independently vulnerable to the ordering attack above. By contrast, the Solana `remote_executor` supports `Vec<InstructionData>` in a single VAA payload, avoiding this limitation.

### Impact Explanation
An unprivileged relayer who monitors the Wormhole network for pending governance VAAs can submit a higher-sequence VAA before lower-sequence ones are processed. The lower-sequence VAAs are then permanently unexecutable. If those skipped VAAs contained critical actions — such as a `UpgradeContract` security patch, a `SetDataSources` correction removing a compromised oracle, or a `SetFee` adjustment — those actions are silently lost until governance issues replacement VAAs. During the window between the skip and the replacement, the protocol operates under the stale (potentially insecure) configuration.

### Likelihood Explanation
Governance VAAs are publicly observable on the Wormhole network. Any actor can call `executeGovernanceInstruction(bytes calldata encodedVM)` (EVM) or `execute(bytes memory encodedVm)` (Executor) with any valid VAA. No privileged access is required. A motivated attacker (e.g., one who benefits from delaying a fee increase or a data-source correction) can trivially front-run the intended execution order.

### Recommendation
1. Change the sequence check to require strict sequential execution: `sequence == lastExecutedGovernanceSequence + 1`. This ensures no VAA can be skipped.
2. Alternatively, maintain a set of "consumed" sequence numbers and reject any VAA whose sequence has already been consumed or whose sequence is not the next expected one.
3. For the EVM `Executor.sol`, consider supporting a list of `(callAddress, callData)` pairs per VAA (analogous to the Solana `remote_executor`'s `Vec<InstructionData>`), so that logically atomic multi-step operations can be encoded in a single VAA and are not split across multiple sequence numbers.

### Proof of Concept

**Scenario (EVM, `PythGovernance.sol`):**

1. Governance issues VAA sequence 5 (`UpgradeContract` — critical security patch) and VAA sequence 6 (`SetFee` — routine update). Both are posted to Wormhole and publicly visible.
2. Attacker calls `executeGovernanceInstruction(vaa_sequence_6)` before anyone relays VAA 5.
3. Check: `6 > lastExecutedGovernanceSequence (4)` → passes. `lastExecutedGovernanceSequence` is set to 6.
4. Attacker (or anyone) now tries to relay VAA 5: `5 <= 6` → `revert OldGovernanceMessage`. VAA 5 is permanently unexecutable.
5. The critical security patch encoded in VAA 5 is never applied. The contract remains vulnerable until governance issues a new VAA (sequence ≥ 7) with the same patch.
