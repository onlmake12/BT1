### Title
Missing Maximum Fee Guard in `requestV2()` Causes Unexpected Reverts and Unrefunded ETH Loss — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract's `requestV2()` / `requestWithCallback()` family of functions accepts an exact `msg.value` payment but provides no `maxFee` slippage guard. Because the Fortuna keeper service actively and frequently adjusts provider fees on-chain, a user who queries `getFeeV2()` and then submits `requestV2{value: fee}()` may find their transaction reverts with `InsufficientFee` if the fee was raised in the intervening block. Worse, any excess ETH a user sends to protect themselves is permanently captured as Pyth protocol fees — it is never refunded.

---

### Finding Description

Users are expected to call `getFeeV2(provider, gasLimit)` before each `requestV2()` invocation to obtain the current required fee. The interface documentation itself acknowledges the race condition:

> *"Note that provider fees can change over time. Callers of this method should explicitly compute `getFeeV2(provider, gasLimit)` prior to each invocation… Further note that excess value is **not** refunded to the caller."* [1](#0-0) 

Inside `requestHelper()`, the fee is re-computed at execution time and compared against `msg.value`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [2](#0-1) 

Any ETH above the provider's share is immediately credited to `_state.accruedPythFeesInWei` — it is not held in escrow and is not refundable:

```solidity
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [3](#0-2) 

Provider fees are mutable at any time by the provider or its designated fee manager:

```solidity
function setProviderFee(uint128 newFeeInWei) external override { ... }
function setProviderFeeAsFeeManager(address provider, uint128 newFeeInWei) external override { ... }
``` [4](#0-3) 

The Fortuna keeper service (`apps/fortuna/src/keeper/fee.rs`) is explicitly designed to adjust provider fees dynamically based on prevailing gas prices, targeting a profit window: [5](#0-4) 

This means fee changes are not exceptional — they are a routine, automated part of production operation.

---

### Impact Explanation

Two concrete loss scenarios exist:

**Scenario A — Unexpected Revert (gas loss):**
1. User calls `getFeeV2()` off-chain → receives fee `F`.
2. Fortuna keeper (or a malicious provider) calls `setProviderFeeAsFeeManager()` raising the fee to `F + δ`.
3. User's `requestV2{value: F}()` reverts with `InsufficientFee`.
4. User loses gas; must retry.

**Scenario B — Unrefunded Overpayment (ETH loss):**
1. User, aware of fee volatility, sends `requestV2{value: F + buffer}()`.
2. Fee at execution time is still `F`.
3. The `buffer` ETH is silently absorbed into `accruedPythFeesInWei` — permanently lost to the user.

A malicious provider can weaponize Scenario A into a targeted griefing DoS: register with a low fee, observe a victim's pending transaction in the mempool, frontrun it with `setProviderFee(high)`, causing the victim's transaction to revert, then restore the low fee. The provider incurs only gas costs while the victim repeatedly loses gas.

**Impact: Medium** — direct ETH loss (Scenario B) and reliable DoS of user requests (Scenario A).

---

### Likelihood Explanation

**Likelihood: Medium**

- The Fortuna keeper is documented and deployed to adjust fees on every chain where Entropy is live. Fee changes are not rare edge cases; they are continuous.
- Any address can permissionlessly register as an Entropy provider (`register()` is open), making the malicious-provider griefing path accessible to any unprivileged attacker.
- Users interacting via a UI or SDK will naturally query `getFeeV2()` and then submit a transaction — the window between these two steps is exploitable on any chain with a public mempool. [6](#0-5) 

---

### Recommendation

Add a `maxFee` parameter to `requestV2()` (and its variants) that reverts if the actual required fee exceeds the caller's stated maximum:

```solidity
function requestV2(
    address provider,
    bytes32 userRandomNumber,
    uint32 gasLimit,
    uint128 maxFee          // <-- new parameter
) external payable returns (uint64) {
    uint128 requiredFee = getFeeV2(provider, gasLimit);
    if (requiredFee > maxFee) revert EntropyErrors.FeeTooHigh();
    if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
    // refund excess: msg.value - requiredFee back to caller
    ...
}
```

This mirrors the pattern used in `sellToken()` in the referenced report and is the standard DEX slippage-protection idiom. Refunding excess ETH (rather than absorbing it into protocol fees) would also eliminate Scenario B.

The same fix should be applied to the analogous `requestPriceUpdatesWithCallback()` in the Echo contract: [7](#0-6)

### Proof of Concept

```solidity
// Demonstrates Scenario A: unexpected revert due to fee change between query and execution
function testEntropyFeeRaceCondition() public {
    address user = makeAddr("user");
    vm.deal(user, 10 ether);

    // User queries the fee off-chain (simulated here)
    uint128 fee = entropy.getFeeV2(provider, 0);

    // Fortuna keeper (or malicious provider) raises the fee before user's tx lands
    vm.prank(provider);
    entropy.setProviderFee(fee * 2);

    // User's transaction reverts even though they followed the documented pattern
    vm.prank(user);
    vm.expectRevert(EntropyErrors.InsufficientFee.selector);
    entropy.requestV2{value: fee}(provider, bytes32(uint256(42)), 0);
}

// Demonstrates Scenario B: excess ETH permanently lost to protocol fees
function testEntropyOverpaymentLost() public {
    address user = makeAddr("user");
    vm.deal(user, 10 ether);

    uint128 fee = entropy.getFeeV2(provider, 0);
    uint128 buffer = fee; // user sends 2x to protect against fee increase

    uint128 pythFeesBefore = entropy.getAccruedPythFees();

    vm.prank(user);
    entropy.requestV2{value: fee + buffer}(provider, bytes32(uint256(42)), 0);

    // The buffer is absorbed into Pyth fees, not returned to user
    assertEq(entropy.getAccruedPythFees(), pythFeesBefore + buffer);
}
```
