Audit Report

## Title
Stale `total_tx_size`/`total_tx_cycles` Counter After Ancestor Eviction Inflates Pool Accounting — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the pool's `total_tx_size` and `total_tx_cycles` counters are snapshotted into local variables before `check_and_record_ancestors` runs. When that function evicts transactions via `remove_entry_and_descendants`, it correctly decrements `self.total_tx_size` in place. However, `add_entry` then unconditionally overwrites `self.total_tx_size` with the pre-eviction snapshot, permanently inflating the counter by the total size of all evicted transactions. The inflated counter causes `limit_size` to evict legitimate pending transactions that should remain in the pool.

## Finding Description
`add_entry` in `tx-pool/src/component/pool_map.rs` (lines 200–221) follows this sequence:

```
L210-211: let (total_tx_size, total_tx_cycles) =
              self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
              // snapshots self.total_tx_size + entry.size into local var
              // self.total_tx_size is UNCHANGED at this point

L213:     evicts = self.check_and_record_ancestors(&mut entry)?;
              // may call remove_entry_and_descendants → update_stat_for_remove_tx
              // which correctly decrements self.total_tx_size for each evicted tx

L218-219: self.total_tx_size = total_tx_size;   // stale snapshot written back
          self.total_tx_cycles = total_tx_cycles; // evictions ignored
```

`updated_stat_for_add_tx` (lines 711–729) is a `&self` method — it reads `self.total_tx_size` but does not modify it. The local variable it returns is `self.total_tx_size + entry.size` at the moment of the call.

`check_and_record_ancestors` (lines 590–640) enters the eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants` for each evicted tx (line 618), which calls `update_stat_for_remove_tx` (lines 733–758), correctly decrementing `self.total_tx_size` and `self.total_tx_cycles` in place.

After evictions complete, lines 218–219 overwrite `self.total_tx_size` with the pre-eviction local variable. The net effect: `self.total_tx_size` ends up as `(initial + entry.size)` instead of `(initial + entry.size - sum_of_evicted_sizes)`. The counter is inflated by exactly the total size of all evicted transactions.

`limit_size` in `pool.rs` (lines 292–329) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting legitimate pending transactions until the (inflated) counter drops below the limit. Each evicted transaction is rejected with `Reject::Full` and recorded in the recent-reject database.

## Impact Explanation
An unprivileged submitter can permanently inflate `total_tx_size` by triggering the ancestor-eviction path. After inflation, every subsequent call to `limit_size` (which runs after every new submission) evicts legitimate pending transactions with `Reject::Full`. The attack is repeatable: each trigger inflates the counter further. This constitutes a targeted, low-cost denial-of-service against the transaction pool of any node, causing honest users' transactions to be silently dropped even when the pool has ample real capacity. This matches the **High** impact: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation
The eviction path is reachable by any unprivileged submitter with no special keys or hash power. The existing integration test `TxPoolLimitAncestorCount` in `test/src/specs/tx_pool/limit.rs` (lines 70–116) documents and exercises exactly this scenario: submit 2000 cell-dep-referencing transactions, then submit one transaction that consumes the cell dep as an input, triggering eviction of 1002 cell-ref transactions. The attack requires only standard RPC access and a small amount of CKB to cover transaction fees. It is repeatable indefinitely.

## Recommendation
Move the counter snapshot **after** all evictions complete, so it reflects the post-eviction state of `self.total_tx_size`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// Compute totals AFTER evictions have already updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, subtract the total size/cycles of all entries in `evicts` from the pre-computed `total_tx_size`/`total_tx_cycles` before the final assignment.

## Proof of Concept
The existing integration test `TxPoolLimitAncestorCount` (`test/src/specs/tx_pool/limit.rs`, lines 70–116) already exercises the exact trigger path. To confirm the accounting bug, add an assertion after the final `send_transaction_result` call:

1. Record `pool_map.total_tx_size` before submitting the consuming transaction.
2. After submission succeeds, assert `pool_map.total_tx_size == (pre_value + last_tx.size - sum_of_evicted_sizes)`.
3. Without the fix, the assertion fails: `total_tx_size` equals `pre_value + last_tx.size`, ignoring the 1002 evicted transactions' sizes.
4. Confirm that `limit_size` subsequently evicts additional legitimate transactions that were not part of the eviction set.
