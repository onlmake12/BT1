### Title
Unbounded `GetBlockFilterCheckPoints` Serving Without Per-Peer Rate Limiting — (`sync/src/filter/get_block_filter_check_points_process.rs`)

### Summary

The `BlockFilter` P2P protocol handler serves `GetBlockFilterCheckPoints` requests with no per-peer rate limiting. Each valid request triggers up to `BATCH_SIZE × 2 = 4,000` RocksDB reads and sends up to ~64 KB in response. An unprivileged remote peer can flood the node at maximum connection speed, causing sustained DB read amplification and outbound bandwidth exhaustion.

### Finding Description

`GetBlockFilterCheckPointsProcess::execute` performs up to `BATCH_SIZE=2000` loop iterations, each issuing two DB reads — `get_block_hash` (from `COLUMN_INDEX`) and `get_block_filter_hash` (from `COLUMN_BLOCK_FILTER_HASH`) — with no guard on request frequency. [1](#0-0) [2](#0-1) 

The `BlockFilter` struct holds only `shared: Arc<SyncShared>` — there is no `rate_limiter` field, unlike `Relayer` which explicitly carries one: [3](#0-2) [4](#0-3) 

The `Relayer` checks its rate limiter before dispatching any message: [5](#0-4) 

The `BlockFilter::try_process` has no equivalent check — it dispatches directly to the process handler: [6](#0-5) 

The only ban path in `BlockFilter` is for malformed (unparseable) messages. A well-formed `GetBlockFilterCheckPoints{start_number: 0}` message always satisfies `latest >= start_number` once at least one block has filter data, so it always executes the full loop and returns `Status::ok()` — never triggering a ban: [7](#0-6) 

The DB reads themselves are direct RocksDB column reads with no in-memory short-circuit for the filter hash column: [8](#0-7) 

### Impact Explanation

- **DB read amplification**: Up to 4,000 RocksDB point-reads per message. At sustained high message rate from one peer, this saturates I/O bandwidth shared with block validation and sync.
- **Outbound bandwidth**: Each response carries up to `2000 × 32 = 64 KB` of filter hashes. A single peer can force continuous large outbound frames.
- **No attacker cost**: The request message is a fixed 8-byte `start_number` field. The attacker sends tiny messages and receives large responses.

### Likelihood Explanation

The attack is reachable by any peer that connects on the Filter protocol (`SupportProtocols::Filter`). No PoW, no key, no privilege is required. The omission is structural: the `Relayer` and `HolePunching` protocols both have explicit `governor`-based rate limiters; `BlockFilter` does not. [9](#0-8) 

### Recommendation

Add a `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`) and check it at the top of `try_process` before dispatching `GetBlockFilterCheckPoints`, `GetBlockFilterHashes`, and `GetBlockFilters` messages.

### Proof of Concept

1. Connect to a node that has built filter data for at least 1 block on the `Filter` protocol.
2. In a tight loop, send `BlockFilterMessage { GetBlockFilterCheckPoints { start_number: 0 } }`.
3. Observe: no ban is issued; the node responds to every message with up to 64 KB; RocksDB read counters (via metrics or `rocksdb.stats`) show amplification proportional to send rate; outbound bandwidth is consumed at attacker-controlled rate.
