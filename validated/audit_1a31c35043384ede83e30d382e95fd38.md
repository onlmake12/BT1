### Title
`ScriptHashTypeVerifier` Omits Type Script `hash_type` Validation Against `ENABLED_SCRIPT_HASH_TYPE` Allowlist — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist (`{0,1,2,4}` — Data, Type, Data1, Data2) only on each output's **lock script** `hash_type`. It never inspects the output's **type script** `hash_type`. An unprivileged RPC caller or P2P relay peer can therefore submit a transaction whose output carries a type script with a future/unsupported `hash_type` (e.g., `Data3`=6) and bypass the cheap non-contextual pre-filter entirely, forcing the node into expensive contextual script execution before the transaction is ultimately rejected.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is the consensus-level allowlist of currently activated script hash types:

```
// util/constant/src/consensus.rs
pub static ENABLED_SCRIPT_HASH_TYPE: Set<u8> = phf_set! {
    0u8, // Data
    1u8, // Type
    2u8, // Data1
    4u8, // Data2
};
``` [1](#0-0) 

`ScriptHashTypeVerifier` is the non-contextual verifier responsible for enforcing this allowlist. Its `verify()` loop iterates over transaction outputs and checks only `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...ScriptHashTypeNotPermitted...);
            }
        } else {
            return Err(...InvalidScriptHashType...);
        }
    }
    Ok(())
}
``` [2](#0-1) 

There is no corresponding check on `output.type_().to_opt().map(|s| s.hash_type())`. The grep for `type_().*hash_type` inside `verification/` returns zero matches, confirming the omission is complete.

The `NonContextualTransactionVerifier` doc-comment even self-documents the gap: it says "Check whether output **lock** hash type within enabled range" — the type script is not mentioned. [3](#0-2) 

The structural-validity layer (`check_data` / `verify_value`) accepts any even byte or `1` as a valid `hash_type` bit-pattern, so `Data3`=6, `Data4`=8, … all pass deserialization:

```rust
pub fn verify_value(v: u8) -> bool {
    v.is_multiple_of(2) || v == 1
}
``` [4](#0-3) 

Consequently a transaction output with `lock.hash_type=0` (valid) and `type_.hash_type=6` (Data3, not in allowlist) clears every non-contextual check and enters the tx-pool. Only later, during contextual script execution, does `extract_script_and_dep_index` return `InvalidScriptHashType`:

```rust
hash_type => {
    return Err(ScriptError::InvalidScriptHashType(format!(
        "The ScriptHashType/{:?} has not been activated...", hash_type)));
}
``` [5](#0-4) 

The existing unit tests confirm the verifier only covers lock scripts — both `test_unknown_hash_type_output_lock` and `test_not_enabled_hash_type_output_lock` set the disallowed hash type on the **lock** script; no analogous test exists for the type script. [6](#0-5) 

The `CellbaseVerifier` in `block_verifier.rs` has the same pattern — it checks `output.lock().hash_type()` against `ENABLED_SCRIPT_HASH_TYPE` for cellbase outputs but never checks the type script (cellbase outputs are required to have no type script, so that specific path is safe, but it illustrates the systemic lock-only pattern). [7](#0-6) 

---

### Impact Explanation

**Tx-pool resource exhaustion / CPU DoS.** The `ScriptHashTypeVerifier` exists precisely to be a cheap, context-free gate that rejects obviously invalid transactions before the node pays the cost of cell resolution and script execution. By bypassing it with a disallowed type script `hash_type`, an attacker forces the node to:

1. Admit the transaction into the pending pool.
2. Perform full contextual verification (cell resolution, script group construction, VM invocation attempt).
3. Reject the transaction only after that expensive work.

An attacker who continuously submits such transactions can saturate the tx-pool and exhaust CPU on contextual verification, displacing legitimate transactions and degrading node responsiveness. Because the error produced at the contextual stage (`ScriptError::InvalidScriptHashType`) is not the same as the non-contextual `TransactionError::ScriptHashTypeNotPermitted`, the peer-banning / malformed-tx classification path that would normally penalise the sender is also not triggered at the earliest possible point. [8](#0-7) 

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction via the `send_transaction` RPC or via P2P relay — both are available to any unprivileged user. Constructing a transaction with a valid lock script and a type script whose `hash_type` byte is `6` (Data3) is trivial. No special keys, no mining power, and no Sybil capability are needed.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` for every output that carries one:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }

        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

Add corresponding unit tests mirroring `test_unknown_hash_type_output_lock` and `test_not_enabled_hash_type_output_lock` but targeting the type script field.

### Proof of Concept

Using the CKB Rust SDK or any raw transaction builder, construct and submit via `send_transaction` RPC:

```
Transaction {
  inputs:  [ <any live cell with a valid lock> ],
  outputs: [
    CellOutput {
      lock: Script { hash_type: 0x00 (Data), code_hash: <valid>, args: <valid> },
      type: Some(Script { hash_type: 0x06 (Data3 — not in ENABLED_SCRIPT_HASH_TYPE),
                          code_hash: <any 32 bytes>, args: b"" }),
      capacity: <sufficient>,
    }
  ],
  outputs_data: [ b"" ],
  ...
}
```

**Expected (correct) behaviour:** `NonContextualTransactionVerifier` rejects with `ScriptHashTypeNotPermitted { hash_type: 6 }` before any cell resolution.

**Actual behaviour:** `ScriptHashTypeVerifier::verify()` returns `Ok(())` (only checked the lock). The transaction enters the tx-pool and proceeds to contextual verification, where it fails inside `extract_script_and_dep_index` with `ScriptError::InvalidScriptHashType`.     pub(crate) duplicate_deps: DuplicateDepsVerifier<'a>,

            size: SizeVerifier::new(tx, consensus.max_block_bytes()),

    fn extract_script_and_dep_index(
        &self,
        script: &Script,
    ) -> Result<(&LazyData, &usize), ScriptError> {
        let script_hash_type = ScriptHashType::try_from(script.hash_type())
            .map_err(|err| ScriptError::InvalidScriptHashType(err.to_string()))?;
        match script_hash_type {
            ScriptHashType::Data | ScriptHashType::Data1 | ScriptHashType::Data2 => {
                if let Some((dep_index, lazy)) = self.binaries_by_data_hash.get(&script.code_hash())
                {
                    Ok((lazy, dep_index))
                } else {
                    Err(ScriptError::ScriptNotFound(script.code_hash()))
                }
            }
            ScriptHashType::Type => {
                if let Some(ref bin) = self.binaries_by_type_hash.get(&script.code_hash()) {
                    match bin {
                        Binaries::Unique(_, dep_index, lazy) => Ok((lazy, dep_index)),
                        Binaries::Duplicate(_, dep_index, lazy) => Ok((lazy, dep_index)),
                        Binaries::Multiple => Err(ScriptError::MultipleMatches),
                    }
                } else {
                    Err(ScriptError::ScriptNotFound(script.code_hash()))
                }
            }
            hash_type => {
                return Err(ScriptError::InvalidScriptHashType(format!(
                    "The ScriptHashType/{:?} has not been activated, and is not permitted for use.",
                    hash_type
                )));
            }
        }
    }
