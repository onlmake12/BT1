### Title
Descendant-of-Evicted-Cell-Ref-Parent Remains in `parents` Set, Causing `get_by_id_checked` Panic in `_record_ancestors` — (`tx-pool/src/component/pool_map.rs`)

### Summary

In `check_and_record_ancestors`, when a `cell_ref_parent` Q is evicted via `remove_entry_and_descendants`, all of Q's descendants (including any that are also direct parents of the incoming tx) are removed from the pool and from `links`. However, `parents.remove(next_id)` only removes Q itself — not Q's descendants — from the local `parents` set. The subsequent `calc_relation_ids` call re-includes those stale descendant IDs in the recomputed `ancestors` set (because they are still in `parents`). `_record_ancestors` then calls `get_by_id_checked` on those stale IDs, which are no longer in the pool, triggering the `expect("inconsistent pool")` panic and crashing the node.

---

### Finding Description

**Root cause — three interacting bugs:**

**Bug 1 — `remove_entry_and_descendants` removes more than `parents.remove` accounts for.** [1](#0-0) 

The eviction loop removes `next_id` (a `cell_ref_parent`) and all its descendants via `remove_entry_and_descendants`, but only calls `parents.remove(next_id)`. Descendants of `next_id` that are also direct parents of the incoming tx remain in `parents`.

**Bug 2 — `calc_relation_ids` unconditionally includes every item from its initial `stage`.** [2](#0-1) 

Every ID in the initial `stage` (i.e., `parents`) is inserted into `relation_ids` regardless of whether it exists in `self.inner`. A stale ID whose entry has been removed from `links` is still emitted in the result.

**Bug 3 — `_record_ancestors` calls `get_by_id_checked` on every ancestor without a pool-existence guard.** [3](#0-2) 

`get_by_id_checked` panics with `expect("inconsistent pool")` if the ID is absent from `entries`. [4](#0-3) 

---

### Impact Explanation

An unprivileged attacker can crash any reachable CKB full node (panic = process abort in Rust release builds) by submitting a crafted transaction chain via the public RPC (`send_transaction`) or the P2P transaction relay path. The node does not restart automatically; it requires operator intervention. This is a **remote denial-of-service** against any node that accepts unconfirmed transactions.

---

### Likelihood Explanation

The attack requires only valid, fee-paying transactions submitted through the standard submission path. No privileged access, no PoW, no key material, and no Sybil capability is needed. The chain length required equals `max_ancestors_count` (default ~25–1000 depending on deployment), which is trivially constructable. The attack is deterministic and locally reproducible.

---

### Recommendation

In the eviction loop, after calling `remove_entry_and_descendants(next_id)`, also remove all descendants of `next_id` from `parents`:

```rust
let removed = self.remove_entry_and_descendants(next_id);
for r in &removed {
    parents.remove(&r.proposal_short_id());
}
ancestors_count = ancestors_count.saturating_sub(removed.len());
```

Alternatively, after the eviction loop, filter `parents` to only retain IDs that are still present in `self.links.inner` before recomputing `ancestors`.

---

### Proof of Concept

**Preconditions:** `max_ancestors_count = N` (e.g., 25).

1. Submit **P** — a normal tx with one output `P.output[0]`.
2. Submit **Q1** — a tx with `cell_dep = P.output[0]` and one output `Q1.output[0]`. This registers `edges.deps[P.output[0]] = {Q1_id}`.
3. Submit **Q2 … Q(N-1)** — a chain where each `Qi` has `input = Q(i-1).output[0]`. Now `Q(N-1)` is a descendant of Q1.
4. Submit attacker tx **A** with two inputs: `input[0] = P.output[0]` and `input[1] = Q(N-1).output[0]`.

**Trace through `check_and_record_ancestors(A)`:**

- `get_tx_ancenstors(A)`:
  - Input `P.output[0]`: `edges.deps[P.output[0]] = {Q1_id}` → `cell_ref_parents = {Q1_id}`, `parents ∋ Q1_id`; P is in links → `parents ∋ P_id`.
  - Input `Q(N-1).output[0]`: Q(N-1) is in links → `parents ∋ Q(N-1)_id`.
  - `ancestors = {P_id, Q1_id, …, Q(N-1)_id}` (N entries); `ancestors_count = N+1`. [5](#0-4) 

- `N+1 > N` and `(N+1) − 1 = N ≤ N` → enter eviction branch.
- Evict Q1: `remove_entry_and_descendants(Q1_id)` removes Q1 through Q(N-1) from pool and links.
- `parents.remove(Q1_id)` → `parents = {P_id, Q(N-1)_id}`. **Q(N-1) is still present despite being evicted.**
- `ancestors_count` decremented to N ≤ N → loop exits. [1](#0-0) 

- Recompute: `calc_relation_ids({P_id, Q(N-1)_id}, Parents)`:
  - Q(N-1) is absent from `links.inner` → no parents traversed, but Q(N-1)_id is still inserted into `relation_ids`.
  - Result: `{P_id, Q(N-1)_id}`. [6](#0-5) 

- `_record_ancestors(A, {P_id, Q(N-1)_id}, …)`:
  - `get_by_id_checked(Q(N-1)_id)` → Q(N-1) not in `entries` → **`expect("inconsistent pool")` → panic → node crash**. [7](#0-6)

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
    }
