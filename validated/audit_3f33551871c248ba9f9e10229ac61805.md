### Summary

An unprivileged remote peer can send a `GetLastStateProof` P2P message with up to `GET_LAST_STATE_PROOF_LIMIT` (1000) strictly-increasing difficulty values, each crafted to map to a distinct block near the start of the sampled range. This forces `get_block_numbers_via_difficulties` to invoke `get_first_block_total_difficulty_is_not_less_than` up to 1000 times, each performing a binary search with O(log N) DB reads. The only size check is:

This counts the number of difficulty entries, not the number of DB reads they generate.

**Pre-call filtering (partial mitigation):**

Before calling `get_block_numbers_via_difficulties`, difficulties are filtered to those â‰¤ `total_difficulty(difficulty_boundary_block_number - 1)`:

```rust

    .into_iter()
    .take_while(|d| *d <= total_difficulty)

``` [3](#0-2) 

This does not reduce the count of difficulties â€” it only removes those above the boundary. An attacker who knows the chain's difficulty distribution (public information) can ensure all 1000 values survive this filter.

**`current_difficulty` deduplication (partial mitigation):**

Inside `get_block_numbers_via_difficulties`, entries where `current_difficulty >= difficulty` are skipped:

```rust

    continue;
}
``` [4](#0-3) 

After finding block `num` with actual total difficulty `diff`, `current_difficulty = diff`. The attacker bypasses this by sending difficulties that are each strictly greater than the actual total difficulty of the previously found block â€” achievable by querying the chain's block difficulties beforehand.

**`start_block_number` advancement (partial mitigation):**

After each found block `num`, `start_block_number = num - 1`:

```rust

    start_block_number = num - 1;
}
``` [5](#0-4) 

If the attacker crafts difficulties that each map to a block near the **start** of the remaining range (e.g., consecutive blocks 1, 2, 3, â€¦), `start_block_number` advances by only 1 per iteration. The search range [start_block_number, difficulty_boundary_block_number) barely shrinks, so each of the 1000 binary searches operates over nearly the full range.

**Per-call cost of `get_first_block_total_difficulty_is_not_less_than`:**

Each invocation performs:
- 1 DB read for `start_block_number`
- 1 DB read for `end_block_number - 1`

### Impact Explanation

With `last_n_blocks = 0` and 1000 crafted difficulties:

| Metric | Intended | Actual (1M-block chain) |
|---|---|---|
| DB reads (sampling phase) | O(1000) | O(1000 Ã— 20) â‰ˆ 20,000 |
| Amplification factor | 1Ã— | ~20Ã— |

An attacker can repeatedly send such messages to amplify I/O load on the light-client protocol server. The amplification scales with chain length: O(log N) per difficulty entry.

---

### Likelihood Explanation

- **Entrypoint:** Standard P2P `GetLastStateProof` message â€” no authentication required.
- **Knowledge required:** The chain's total difficulty per block is public and queryable. The attacker can pre-compute the exact difficulty values needed.
- **Repeatability:** The attacker can send requests in a tight loop; there is no per-peer rate limiting visible in this handler.

---

### Recommendation

Enforce a bound on the **total number of DB reads**, not just the number of difficulty entries. One approach: limit the number of difficulties that actually trigger a new binary search (i.e., those that survive the `current_difficulty` deduplication) to a constant fraction of `GET_LAST_STATE_PROOF_LIMIT`, or add a per-request DB-read counter that aborts processing once a threshold is exceeded.

---

### Proof of Concept

On a chain of N = 1,000,000 blocks:

1. Query the chain to obtain `total_difficulty(i)` for blocks i = 1, 2, â€¦, 1000.
2. Construct `difficulties = [total_difficulty(1)+1, total_difficulty(2)+1, â€¦, total_difficulty(1000)+1]` â€” strictly increasing, each just above the actual total difficulty of consecutive blocks.
3. Set `difficulty_boundary` just above `total_difficulty(1000)`, `last_n_blocks = 0`, `start_number = 0`.
4. Send the `GetLastStateProof` message.

**Expected result:** `get_block_numbers_via_difficulties` calls `get_first_block_total_difficulty_is_not_less_than` 1000 times. Each call searches over [i-1, difficulty_boundary_block_number) â‰ˆ [i-1, 1M), performing ~20 DB reads. Total: ~20,000 DB reads for a single request, versus the ~1,000 the guard was designed to allow.
