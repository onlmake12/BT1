### Title
`transfer()` Used for ETH Refund in `verifyUpdate()` Causes Permanent DoS for Contract Callers - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` refunds excess ETH to `msg.sender` using `.transfer()`, which only forwards 2300 gas. Any contract caller whose `receive()` or fallback function requires more than 2300 gas will have every `verifyUpdate()` call revert when sending excess ETH, permanently blocking Lazer price verification for that caller.

### Finding Description
In `PythLazer.sol`, the `verifyUpdate()` function is `external payable` and requires `msg.value >= verification_fee`. When `msg.value > verification_fee`, it refunds the excess using:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [1](#0-0) 

The `.transfer()` function hard-caps the forwarded gas at 2300. If `msg.sender` is a contract whose `receive()` or fallback function requires more than 2300 gas — for example, a Gnosis Safe multisig (which requires >6k gas to reach its implementation and emit an event), any DeFi protocol with storage-writing receive logic, or any smart contract wallet — the refund will revert, and because `.transfer()` propagates the revert, the entire `verifyUpdate()` call fails.

The `verification_fee` is initialized to `1 wei` in `initialize()`: [2](#0-1) 

It is owner-controlled and can be changed at any time. Callers who send slightly more than the current fee (a standard defensive pattern to avoid underpayment across fee changes) will always hit the refund path.

### Impact Explanation
Any contract-based Lazer updater that:
1. Calls `verifyUpdate()` with `msg.value > verification_fee`, **and**
2. Has a `receive()` or fallback function requiring >2300 gas

...will have every such call permanently revert. The caller cannot verify Lazer price updates, which is the sole purpose of `PythLazer`. Unlike the reference H-01, no ETH is permanently locked (the transaction reverts), but the DoS is permanent for the affected caller unless they can send exactly the right fee — which is impossible to guarantee atomically when the fee is owner-mutable.

### Likelihood Explanation
- Lazer is designed to be integrated by on-chain DeFi protocols and smart contracts, not just EOAs.
- Gnosis Safe multisigs — the dominant multisig standard used by protocols — require >6k gas for their `receive()` function, far exceeding the 2300 gas limit.
- Sending slightly more ETH than the exact fee is a standard defensive pattern in smart contract development to avoid underpayment reverts.
- The `verification_fee` is owner-mutable, making exact-fee calculation unreliable across blocks.

### Recommendation
Replace `.transfer()` with a low-level `.call()` that forwards all available gas:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "ETH refund failed");
}
```

This matches the pattern already used correctly in `Entropy.sol` for provider withdrawals: [3](#0-2) 

### Proof of Concept
1. A DeFi protocol (a contract with a non-trivial `receive()` function, e.g., one that emits an event or writes to storage) integrates `PythLazer` and calls `verifyUpdate()` with `msg.value = verification_fee + 1 wei`.
2. `verifyUpdate()` reaches line 76: `payable(msg.sender).transfer(1)`.
3. The protocol's `receive()` function requires >2300 gas; the `.transfer()` call exhausts the 2300 gas stipend and reverts.
4. The revert propagates, causing the entire `verifyUpdate()` call to fail.
5. The protocol cannot verify any Lazer price updates when sending excess ETH. Since the fee is owner-mutable, the protocol cannot reliably send the exact fee in every block, making the DoS permanent in practice.
