### Title
Missing `limit_size` Enforcement in `readd_detached_tx` After Chain Reorganization — (File: `tx-pool/src/process.rs`)

---

### Summary

The `readd_detached_tx` function, which re-inserts transactions from detached blocks back into the tx-pool after a chain reorganization, is missing the `limit_size` enforcement step that is present in the normal `submit_entry` path. This is a direct structural analog to the reported Cosmos SDK bug: a "create/re-initialize" path is missing bookkeeping that exists in the standard "submit" path, causing the pool's size cap to go unenforced after a reorg.

---

### Finding Description

**Normal transaction submission path** (`submit_entry`, lines 96–170): [1](#0-0) 

After calling `_submit_entry`, the normal path explicitly calls `limit_size` to evict low-fee-rate transactions and enforce `max_tx_pool_size`:

```rust
let evicted = _submit_entry(tx_pool, status, entry.clone(), &self.callbacks)?;
// ...
tx_pool
    .limit_size(&self.callbacks, Some(&entry.proposal_short_id()))
    .map_or(Ok(()), Err)?;
```

**Reorg re-add path** (`readd_detached_tx`, lines 878–914): [2](#0-1) 

This path calls `_submit_entry` directly but **never calls `limit_size`**:

```rust
if let Err(e) = _submit_entry(tx_pool, status, entry, &self.callbacks) {
    error!("readd_detached_tx submit_entry {} error {}", tx_hash, e);
} else {
    debug!("readd_detached_tx submit_entry {}", tx_hash);
}
```

The `_submit_entry` helper itself only calls `add_pending`, `add_gap`, or `add_proposed`, which update `total_tx_size` and `total_tx_cycles` via `add_entry`: [3](#0-2) 

The `add_entry` function correctly updates the running totals: [4](#0-3) 

So `total_tx_size` is tracked, but the enforcement gate (`limit_size`) is never called in the reorg path.

The `limit_size` function enforces the `max_tx_pool_size` bound by evicting low-fee-rate entries: [5](#0-4) 

The pool size limit is configured as `max_tx_pool_size` in `TxPoolConfig`: [6](#0-5) 

---

### Impact Explanation

After a chain reorganization, `readd_detached_tx` re-adds all detached transactions to the pool without enforcing `max_tx_pool_size`. The pool's `total_tx_size` counter is correctly incremented, but the eviction logic is never triggered. The pool remains over-limit until the next new transaction submission triggers `limit_size` via the normal `submit_entry` path. During this window, the node's tx-pool memory usage exceeds its configured bound. In a reorg involving many large transactions, this could cause a meaningful spike in memory consumption on the node.

---

### Likelihood Explanation

Small chain reorganizations (1–2 blocks) are a normal occurrence in any PoW network and can be triggered by any sync peer that presents a valid competing chain tip — no special privilege or majority hashpower is required. A peer relaying a valid block at the same height as the current tip can cause a 1-block reorg, which invokes `update_tx_pool_for_reorg` → `readd_detached_tx`. This is a reachable, unprivileged entry path. [7](#0-6) 

---

### Recommendation

Add a call to `tx_pool.limit_size(...)` at the end of `readd_detached_tx` (or after the loop), mirroring the enforcement present in `submit_entry`. This ensures the `max_tx_pool_size` invariant is maintained regardless of which code path adds entries to the pool.

---

### Proof of Concept

1. Fill the tx-pool to near `max_tx_pool_size` with pending transactions.
2. Mine a block that commits some of those transactions (reducing pool size).
3. Trigger a 1-block reorg by presenting a competing valid block at the same height via the sync protocol.
4. `update_tx_pool_for_reorg` is called: committed transactions from the new tip are removed, and detached transactions are re-added via `readd_detached_tx`.
5. Observe via `tx_pool_info` RPC that `total_tx_size` now exceeds `max_tx_pool_size` — the cap is not enforced until the next `send_transaction` call triggers `limit_size`. [2](#0-1) [1](#0-0)

fn _submit_entry(
    tx_pool: &mut TxPool,
    status: TxStatus,
    entry: TxEntry,
    callbacks: &Callbacks,
) -> Result<HashSet<TxEntry>, Reject> {
    let tx_hash = entry.transaction().hash();
    debug!("submit_entry {:?} {}", status, tx_hash);
    let (succ, evicts) = match status {
        TxStatus::Fresh => tx_pool.add_pending(entry.clone())?,
        TxStatus::Gap => tx_pool.add_gap(entry.clone())?,
        TxStatus::Proposed => tx_pool.add_proposed(entry.clone())?,
    };
    if succ {
        match status {
            TxStatus::Fresh => callbacks.call_pending(&entry),
            TxStatus::Gap => callbacks.call_pending(&entry),
            TxStatus::Proposed => callbacks.call_proposed(&entry),
        }
    }
    Ok(evicts)
}

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
