### Title
Predictable On-Chain PRNG User Contribution via `block.prevrandao` Enables Provider-Alone Manipulation — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `random()` function in `Entropy.sol` generates the user's contribution to the random number using `block.prevrandao` as a key input. Per EIP-4399, `block.prevrandao` returns the RANDAO mix from the **previous** slot of the beacon chain — a value that is fully public and finalized before the current block is proposed. Combined with the other inputs (`block.timestamp`, `msg.sender`, and the publicly readable `_state.seed`), a dishonest Entropy provider can predict the user's contribution before deciding whether to reveal their own `x_i`, enabling selective censorship and outcome manipulation **without requiring validator collusion** — contrary to the documented security model.

---

### Finding Description

The `random()` function is the on-chain PRNG used to generate the user's contribution when callers invoke the no-argument `requestV2()` variants:

```solidity
// Entropy.sol L1079-1089
function random() internal returns (bytes32) {
    _state.seed = keccak256(
        abi.encodePacked(
            block.timestamp,
            block.prevrandao,
            msg.sender,
            _state.seed
        )
    );
    return _state.seed;
}
``` [1](#0-0) 

This return value is passed directly as the `userContribution` in three public entry points:

```solidity
// L286-310
function requestV2() external payable override returns (uint64) {
    assignedSequenceNumber = requestV2(getDefaultProvider(), random(), 0);
}
function requestV2(uint32 gasLimit) external payable override returns (uint64) {
    assignedSequenceNumber = requestV2(getDefaultProvider(), random(), gasLimit);
}
function requestV2(address provider, uint32 gasLimit) external payable override returns (uint64) {
    assignedSequenceNumber = requestV2(provider, random(), gasLimit);
}
``` [2](#0-1) 

All four inputs to `random()` are observable by the provider **before** the transaction is mined:

| Input | Observability |
|---|---|
| `block.prevrandao` | Finalized from the **previous** slot; public before current block is proposed (EIP-4399) |
| `block.timestamp` | Set by the block proposer; predictable within ±1 second |
| `msg.sender` | Visible in the mempool |
| `_state.seed` | Public contract storage, readable via `eth_getStorageAt` | [3](#0-2) 

The final random number is `r = keccak256(userContribution, providerContribution, blockHash)`. [4](#0-3) 

Since the provider already knows their own `providerContribution` (it is their pre-committed hash chain value), and can now predict `userContribution` from the above inputs, the provider can compute `r` off-chain **before** submitting the `revealWithCallback` transaction. The provider can then selectively reveal only when `r` is favorable (e.g., when the outcome benefits them or harms a specific user).

---

### Impact Explanation

The Entropy protocol's security guarantee for the `requestV2()` no-argument variants is documented as:

> *"This approach modifies the security guarantees such that a dishonest validator **and** provider can collude to manipulate the result."* [5](#0-4) 

However, because `block.prevrandao` is the RANDAO mix from the **previous** slot (already finalized and public before the current block is built), the provider can predict the user's contribution **without any validator involvement**. The actual trust requirement is weaker than documented: a dishonest provider alone can:

1. Observe the pending `requestV2()` transaction in the mempool.
2. Compute `random()` output using the already-known `block.prevrandao`, `msg.sender`, and `_state.seed`.
3. Compute the final `r = hash(x_i, x_U)` before deciding to reveal.
4. Withhold revelation if `r` is unfavorable, or front-run with a different request to shift the sequence number.

This breaks the core security property of the commit-reveal protocol for all users of the no-argument `requestV2()` family, which is the default and most commonly used API path.

---

### Likelihood Explanation

- The three affected entry points (`requestV2()`, `requestV2(uint32)`, `requestV2(address, uint32)`) are the **default and recommended** API for integrators, as shown in all SDK documentation and examples.
- The provider is a permissioned but **unprivileged** role — anyone can register as a provider via `register()`.
- `block.prevrandao` has been the behavior on all post-Merge EVM chains since September 2022 (EIP-4399). The value is trivially readable from any Ethereum node before block proposal.
- `_state.seed` is a public storage slot, readable without any special access.
- No validator collusion is required; the provider alone has sufficient information.

---

### Recommendation

1. **Remove `block.prevrandao` and `block.timestamp` from the on-chain PRNG seed inputs**, as both are known before the block is finalized. These do not add unpredictability against a provider observing the mempool.
2. **Incorporate a future block hash** (e.g., `blockhash(block.number + N)`) as the unpredictable component, similar to the existing `useBlockhash` path in `revealHelper`. This forces the provider to commit before the unpredictable value is known.
3. **Alternatively**, require all callers to supply their own `userRandomNumber` and deprecate the no-argument `requestV2()` variants, or clearly document that these variants provide **no protection** against a dishonest provider (not merely a colluding validator+provider pair).

### Proof of Concept

```solidity
// Attacker is a registered Entropy provider with a known hash chain x_i.
// User calls requestV2() (no arguments) — the contract calls random() internally.

// Provider observes the pending tx in the mempool:
address user = <from mempool tx>;
bytes32 prevSeed = IEntropyState(entropyContract).getSeed(); // read _state.seed
uint256 prevrandao = block.prevrandao; // from previous slot, already public
uint256 timestamp = block.timestamp;  // predictable

// Provider computes the user's contribution off-chain:
bytes32 predictedSeed = keccak256(abi.encodePacked(timestamp, prevrandao, user, prevSeed));
bytes32 userContribution = predictedSeed; // this is what random() will return

// Provider computes the final random number:
bytes32 providerContribution = x_i; // provider knows their own hash chain
bytes32 finalRandom = keccak256(abi.encodePacked(userContribution, providerContribution, bytes32(0)));

// Provider decides: if finalRandom is unfavorable, withhold revelation.
// If favorable, call revealWithCallback to deliver the predicted outcome.
```

The provider can repeat this for every pending request, selectively fulfilling only those that produce outcomes favorable to them, with zero on-chain cost for withheld requests.
