Audit Report

## Title
`resume_from_state` Subsequent-Groups Loop Uses `consumed_cycles` for Both Limit Tracking and Returned Total, Inflating Cycle Count for Multi-Group Spawn Transactions — (File: `script/src/verify.rs`)

## Summary
In `TransactionScriptsVerifier::resume_from_state`, the loop processing script groups after the resumed group discards `used_cycles` and substitutes `consumed_cycles` into both the limit-tracking variable (`current_used`) and the returned total (`cycles`). Because `consumed_cycles` includes spawn/exec/pipe-IO overhead not visible to the VM's own cycle counter, the returned cycle count is inflated for any transaction whose post-resume script groups invoke V2 syscalls. This causes the tx-pool to reject valid transactions via the chunked verification path with `ExceededMaximumCycles` even though the same transaction passes the non-chunked `verify()` path.

## Finding Description
`ChunkState::Completed(Cycle, Cycle)` carries two distinct values: the first field (`used_cycles`) is the VM's own execution cycle count from `TerminatedResult.consumed_cycles`, and the second field (`consumed_cycles`) is `scheduler.consumed_cycles() - previous_cycles`, which additionally includes overhead cycles charged during VM suspension/resumption for spawn/exec/pipe-IO operations. [1](#0-0) [2](#0-1) 

The scheduler documentation explicitly notes that cycles charged during IO suspension/resumption are intentionally not reflected in the VM's `current_cycles` view under the Meepo hardfork, making `scheduler.consumed_cycles()` strictly greater than `TerminatedResult.consumed_cycles` for scripts using spawn/exec/pipe-IO. [3](#0-2) 

`resumable_verify` correctly separates the two values: `used_cycles` accumulates into the returned `cycles` total, while `consumed_cycles` tracks the limit budget `current_consumed_cycles`. [4](#0-3) 

`resume_from_state` also handles the resumed group correctly at lines 299–301, using `used_cycles` for `cycles` and `consumed_cycles` for `current_used`. [5](#0-4) 

However, the subsequent-groups loop at lines 322–326 discards `used_cycles` entirely (bound to `_`) and uses `consumed_cycles` for **both** `current_used` and `cycles`: [6](#0-5) 

The inflated `cycles` value is then returned as `VerifyResult::Completed(cycles)` and propagated to the tx-pool, which rejects transactions whose cycle count exceeds `max_tx_verify_cycles`. [7](#0-6) [8](#0-7) 

A secondary bug exists in `complete()` at line 395: inside the subsequent-groups loop, `current_group` (the resumed group) is used instead of `group` (the current iteration group) for `CyclesOverflow` error attribution, causing incorrect error sourcing. [9](#0-8) 

The overhead constants confirm the magnitude of inflation per spawn/exec/IO call: [10](#0-9) 

## Impact Explanation
This is an incorrect implementation of CKB-VM script cycle accounting — **High (10001–15000 points)**. A valid transaction whose scripts legitimately use spawn/exec/pipe-IO syscalls across multiple script groups will be rejected by the tx-pool via the chunked verification path with `ExceededMaximumCycles`, even though the same transaction passes the non-chunked `verify()` path. The discrepancy between the two verification paths constitutes incorrect behavior of CKB-VM system script verification.

## Likelihood Explanation
Any unprivileged user can trigger this by submitting a transaction with two or more script groups where at least one post-first group uses `ckb_spawn`/`ckb_exec`/pipe-IO syscalls, and total actual execution cycles are close to `max_tx_verify_cycles`. The chunked path is activated automatically when a script group's cycles exceed the per-step budget. Each spawn call adds at least `SPAWN_EXTRA_CYCLES_BASE` (100,000) overhead cycles to `consumed_cycles` without affecting `used_cycles`, making the inflation easy to reach. No special privileges, leaked keys, or victim mistakes are required.

## Recommendation
In `resume_from_state`, restore the correct separation between `used_cycles` (for the returned total) and `consumed_cycles` (for limit tracking), matching the pattern in `resumable_verify`:

```rust
// script/src/verify.rs lines 322–326 — fix
Ok(ChunkState::Completed(used_cycles, consumed_cycles)) => {
    current_used = wrapping_cycles_add(current_used, consumed_cycles, group)?;
    cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
}
```

Additionally, fix the error-attribution bug in `complete()` at line 395 by replacing `current_group` with `group` inside the subsequent-groups loop. [9](#0-8) 

## Proof of Concept
1. Construct a transaction with two lock-script groups. The second group's script calls `ckb_spawn` to launch a child VM, charging `SPAWN_EXTRA_CYCLES_BASE` (100,000) + `SPAWN_YIELD_CYCLES_BASE` (800) overhead per call.
2. Tune the scripts so that actual `used_cycles` ≈ `max_tx_verify_cycles - 1` but `consumed_cycles` (with overhead) > `max_tx_verify_cycles`.
3. Submit via `send_transaction` RPC. The tx-pool invokes chunked verification; `resumable_verify` suspends on the first group, then `resume_from_state` is called. After the first group completes, the subsequent-groups loop runs with the buggy accounting, inflating `cycles`.
4. Observe `ExceededMaximumCycles` rejection from the tx-pool.
5. Confirm by running the same transaction through `verifier.verify()` (non-chunked path), which accepts it — demonstrating the discrepancy.
6. A unit test mirroring `_check_typical_secp256k1_blake160_2_in_2_out_resume_load_cycles` but using spawn scripts and asserting `cycles == cycles_once` will fail on the current code and pass after the fix.
