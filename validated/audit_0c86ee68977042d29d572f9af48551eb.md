### Title
`getPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp, Allowing Arbitrarily Stale Individual Feed Prices to Pass Freshness Validation — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol`'s `getPricesNoOlderThan` (and `getEmaPricesNoOlderThan`) validates freshness using a single subscription-level timestamp (`status.priceLastUpdatedAt`), which is set to the **maximum** `publishTime` across all price feeds in the batch. Because individual feeds (e.g., closed-market equities) can have `publishTime` values arbitrarily older than this maximum, the staleness gate passes while returning prices that are far older than the caller-specified `age_seconds`. An unprivileged keeper can deliberately trigger this by submitting a valid same-slot batch containing one fresh feed and one stale feed.

---

### Finding Description

In `updatePriceFeeds`, the Scheduler calls `parsePriceFeedUpdatesWithConfig` with `minAllowedPublishTime = 0`, accepting any individual feed timestamp as long as the **maximum** timestamp in the batch is within `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` (1 hour): [1](#0-0) 

The slot check enforces that all feeds come from the same Pythnet slot, but **a shared slot does not imply a shared `publishTime`**. Closed-market feeds (e.g., US equities) carry a `publishTime` from their last trading period, which can be many hours or days old, even within a recent slot. The code itself acknowledges this: [2](#0-1) 

`updateTimestamp` (the max) is stored as `status.priceLastUpdatedAt`: [3](#0-2) 

`getPricesNoOlderThan` then checks only this subscription-level max timestamp against `age_seconds`, and if it passes, returns **all** stored prices — including those with individually stale `publishTime` values: [4](#0-3) 

The same flaw exists in `getEmaPricesNoOlderThan`: [5](#0-4) 

The `SubscriptionStatus` struct confirms `priceLastUpdatedAt` is a single scalar for the whole subscription, not per-feed: [6](#0-5) 

---

### Impact Explanation

A consumer calling `getPricesNoOlderThan(subscriptionId, priceIds, age_seconds)` receives a guarantee that is not upheld: the function name and parameter imply all returned prices are no older than `age_seconds`, but the implementation only checks when the subscription was last updated (using the max timestamp). Individual prices can be arbitrarily older. A DeFi protocol relying on this API for freshness assurance (e.g., a lending protocol checking that collateral prices are no older than 60 seconds) would silently receive stale prices for closed-market feeds, potentially enabling incorrect liquidations, mispriced loans, or other financial harm.

---

### Likelihood Explanation

Any unprivileged keeper can call `updatePriceFeeds`. A subscription containing both a crypto feed and an equity feed will naturally exhibit this behavior after equity market close. The test `testUpdatePriceFeedsSucceedsWithStaleFeedIfLatestIsValid` explicitly confirms the design accepts a batch where one feed is over 1 hour stale as long as the max timestamp is valid: [7](#0-6) 

No special privileges, leaked keys, or external oracle manipulation are required. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` constant is only 1 hour, but individual feed timestamps are unbounded below: [8](#0-7) 

---

### Recommendation

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` should validate freshness against each **individual** feed's `publishTime`, not the subscription-level `priceLastUpdatedAt`. Either:

1. Store per-feed `publishTime` and check each one against `age_seconds` before returning it, or
2. Clearly document that `age_seconds` bounds only the subscription update cycle timestamp, not individual feed publish times, and rename the function to avoid the misleading freshness guarantee.

---

### Proof of Concept

1. Create a subscription with two price IDs: `feedCrypto` (BTC/USD) and `feedEquity` (AAPL/USD).
2. After US equity market close, fetch a recent Pythnet accumulator update blob from Hermes for a slot where `feedCrypto.publishTime = now - 30 min` and `feedEquity.publishTime = now - 23 hours` (last trading period). Both feeds share the same Pythnet slot.
3. Call `scheduler.updatePriceFeeds(subscriptionId, updateData)` as an unprivileged keeper.
   - Slot check passes (same slot).
   - `updateTimestamp = now - 30 min` (max). Passes `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` check.
   - `priceLastUpdatedAt` is set to `now - 30 min`.
4. A consumer calls `scheduler.getPricesNoOlderThan(subscriptionId, [feedEquity], 3600)`.
   - `distance(block.timestamp, priceLastUpdatedAt) = 30 min < 3600 s` → **passes**.
   - Returns `feedEquity` price with `publishTime = now - 23 hours`.
5. The consumer believes the equity price is no older than 1 hour, but it is 23 hours old.

    function getPricesNoOlderThan(
        uint256 subscriptionId,
        bytes32[] calldata priceIds,
        uint256 age_seconds
    )
        external
        view
        override
        onlyWhitelistedReader(subscriptionId)
        returns (PythStructs.Price[] memory prices)
    {
        SchedulerStructs.SubscriptionStatus memory status = _state
            .subscriptionStatuses[subscriptionId];

        // Use distance (absolute difference) since pythnet timestamps
        // may be slightly ahead of this chain.
        if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
            revert PythErrors.StalePrice();

        prices = this.getPricesUnsafe(subscriptionId, priceIds);
    }
