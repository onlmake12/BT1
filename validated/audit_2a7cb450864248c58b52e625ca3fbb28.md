### Title
Entropy `useBlockhash` Requests Permanently Bricked by Sequencer Outage — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The legacy `request()` function in the Pyth Entropy contract allows callers to set `useBlockHash = true`, which binds the random number reveal to the `blockhash()` of the request block. The EVM `blockhash()` opcode only returns a non-zero value for the **256 most recent blocks**. If a sequencer outage causes more than 256 blocks to elapse before the provider can call `reveal()`, the request becomes permanently unfulfillable, the user's fee is lost, and there is no refund or cancellation path.

---

### Finding Description

When a user calls `request(provider, userCommitment, true)`, the contract stores `req.blockNumber = block.number` and `req.useBlockhash = true`. [1](#0-0) 

During `reveal()`, the `revealHelper()` function calls `blockhash(req.blockNumber)`: [2](#0-1) 

The code itself documents the 256-block constraint:

> "The `blockhash` function will return zero if the req.blockNumber is equal to the current block number, or if it is not within the 256 most recent blocks."

If `_blockHash == bytes32(0)`, the function reverts with `BlockhashUnavailable`. There is no `cancel()` or `refund()` function in the contract — `clearRequest()` is only called on a successful reveal path. A request stuck in this state occupies storage indefinitely and the user's paid fee is irrecoverable. [3](#0-2) 

The `requestV2()` path explicitly avoids this by hardcoding `useBlockhash = false`: [4](#0-3) 

However, the legacy `request()` function remains callable and is not gated or removed. [5](#0-4) 

---

### Impact Explanation

Any user who calls `request(..., true)` on an L2 chain and experiences a sequencer outage lasting longer than the 256-block window permanently loses their fee and receives no random number. There is no on-chain recovery path. The request slot is consumed from the provider's sequence, wasting a sequence number as well.

---

### Likelihood Explanation

On L2 chains with fast block times, 256 blocks is a very short window:
- **Arbitrum** (~0.25 s/block): 256 blocks ≈ **64 seconds**
- **Optimism / Base** (~2 s/block): 256 blocks ≈ **8.5 minutes**

Arbitrum suffered a documented hour-long outage in December 2023. Any outage exceeding these thresholds on a chain where Entropy is deployed would permanently brick all in-flight `useBlockhash = true` requests. The entry path is fully unprivileged — any EOA or contract can call `request()`.

---

### Recommendation

1. Add a `cancel()` / `refund()` function that allows the requester to reclaim their fee if the blockhash window has expired (i.e., `block.number > req.blockNumber + 256` and `req.useBlockhash == true`).
2. Alternatively, deprecate the `useBlockhash = true` path entirely at the contract level by reverting if `useBlockHash == true` is passed to `request()`, directing users to `requestV2()` which always uses `useBlockhash = false`.
3. Document the 256-block expiry risk prominently in the `request()` NatSpec so integrators are aware.

---

### Proof of Concept

1. User calls `request(provider, userCommitment, true)` at block N, paying the required fee.
2. Sequencer goes offline; on Arbitrum, 257 blocks (~65 seconds) elapse.
3. Provider calls `reveal(provider, sequenceNumber, userRevelation, providerRevelation)`.
4. Inside `revealHelper()`, `blockhash(N)` returns `bytes32(0)` because block N is no longer within the 256-block window.
5. The call reverts with `EntropyErrors.BlockhashUnavailable`.
6. The user's fee is permanently locked in the contract. The request cannot be cancelled or refunded. The provider's sequence number is consumed. [6](#0-5) [7](#0-6)
