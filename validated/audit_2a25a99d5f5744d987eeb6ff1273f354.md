### Title
Unauthenticated Block Template Injection via Miner HTTP Notification Endpoint — (File: miner/src/client.rs)

---

### Summary

The CKB miner process exposes an HTTP server that accepts block template updates from any caller without authentication or authorization. Any process that can reach this endpoint can silently replace the active block template with a malicious one, redirecting mining rewards to an attacker-controlled address. This is a direct analog to the missing access-control modifier on `Vault.initialize`: a state-setting function callable by anyone, with no caller check.

---

### Finding Description

The `handle` function in `miner/src/client.rs` is the HTTP request handler for the miner's notification server. It deserializes any incoming POST body as a `BlockTemplate` and immediately installs it as the active template via `client.update_block_template(template)`, with zero authentication or origin verification:

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }

    Ok(Response::new(Empty::new()))
}
``` [1](#0-0) 

The miner also polls `get_block_template` from the CKB node's RPC: [2](#0-1) 

But the HTTP notification path above is a **separate, unauthenticated write path** that any HTTP client can use to overwrite the template that the miner is actively working on.

The `BlockAssemblerConfig` that controls the `cellbase` lock script (i.e., where mining rewards go) is embedded in the block template: [3](#0-2) 

The `cellbase` lock is built from `config.args` and `config.code_hash` at template-build time. An injected template can carry a `cellbase` transaction whose lock script points to an attacker-controlled address.

---

### Impact Explanation

An attacker who can POST to the miner's HTTP notification port can:

1. **Redirect mining rewards** — replace the `cellbase` lock script with their own address; every block the miner subsequently finds pays the attacker.
2. **Censor or inject transactions** — supply a template with a hand-picked transaction set, excluding legitimate transactions or including attacker-favored ones.
3. **Cause the miner to produce invalid blocks** — supply a template with a malformed `dao`, `compact_target`, or `epoch` field, causing submitted blocks to be rejected by the network and wasting the miner's hashrate.

The impact is direct financial loss for the miner operator.

---

### Likelihood Explanation

The miner's HTTP notification server listen address is operator-configurable. If bound to `0.0.0.0` (or any non-loopback interface), the endpoint is reachable by any network peer — no privileges required. Even when bound to `127.0.0.1`, any co-located process (another container, a compromised dependency, a malicious script) can reach it. The attacker entry path is a plain HTTP POST with a JSON body — no cryptographic material, no PoW, no peer identity required. The `handle` function returns `200 OK` regardless of whether the template was accepted, giving the attacker no feedback that would deter repeated injection.

---

### Recommendation

**Short term:**
- Require a shared secret (e.g., a bearer token or HMAC-signed header) on every request to the notification endpoint, configured at miner startup.
- Default the listen address to `127.0.0.1` and document that binding to any other interface is a security risk.

**Long term:**
- Audit all miner-facing HTTP/RPC surfaces for missing caller authentication.
- Consider replacing the push-notification model with a pull model (miner polls the node exclusively), eliminating the inbound HTTP server entirely.

---

### Proof of Concept

**Precondition:** Miner is running with its HTTP notification server reachable (default or misconfigured to a non-loopback address).

**Steps:**

1. Craft a valid `BlockTemplate` JSON with the `cellbase` transaction's lock script replaced by an attacker-controlled address (e.g., `code_hash` = secp256k1, `args` = attacker's pubkey hash).
2. POST the crafted template to the miner's notification endpoint:
   ```
   POST http://<miner-host>:<notify-port>/
   Content-Type: application/json

   { "version": "0x0", "cellbase": { "data": { ... attacker lock ... } }, ... }
   ```
3. The `handle` function deserializes the body and calls `client.update_block_template(template)` with no authentication check.
4. The miner's workers immediately begin solving PoW on the injected template.
5. Any block found is submitted to the network with the attacker's address in the `cellbase` output — mining rewards are paid to the attacker. [1](#0-0)
