### Title
Predictable Ping Nonce Allows Malicious Peer to Spoof Liveness and Evade Timeout Disconnection - (File: `network/src/protocols/ping.rs`)

### Summary
The `nonce` function used to generate the ping challenge value derives its output entirely from elapsed wall-clock seconds since process start. This is deterministic and trivially predictable by any connected peer. A malicious peer can compute the expected nonce and send a spoofed pong before or without receiving a ping, permanently suppressing the timeout-disconnect mechanism and faking low RTT.

### Finding Description
In `network/src/protocols/ping.rs`, the `nonce` function is:

```rust
fn nonce(t: &Instant, start_time: Instant) -> u32 {
    t.saturating_duration_since(start_time).as_secs() as u32
}
``` [1](#0-0) 

This produces a `u32` equal to the number of seconds the node has been running. It is not random in any sense. The same nonce is then broadcast to every connected peer simultaneously:

```rust
let send_nonce = nonce(&now, self.start_time);
// ...
ps.nonce = send_nonce;
// ...
let ping_msg = PingMessage::build_ping(send_nonce);
context.filter_broadcast(TargetSession::Multi(...), proto_id, ping_msg)
``` [2](#0-1) 

The pong verification accepts a pong as valid if and only if the nonce matches and `processing == true`:

```rust
if let Some(status) = self.connected_session_ids.get_mut(&session.id)
    && (true, nonce) == (status.processing, status.nonce())
{
    status.processing = false;
    let last_ping_sent_at = status.last_ping_sent_at;
    self.pong_received(session.id, last_ping_sent_at);
    return;
}
// if nonce is incorrect or can't find ping info → disconnect
``` [3](#0-2) 

A successful pong sets `peer.ping_rtt` and `peer.last_ping_protocol_message_received_at`, and clears `processing`, preventing the timeout check from disconnecting the peer:

```rust
fn pong_received(&mut self, id: SessionId, last_ping: Instant) {
    let now = Instant::now();
    self.network_state.with_peer_registry_mut(|reg| {
        if let Some(peer) = reg.get_peer_mut(id) {
            peer.ping_rtt = Some(now.saturating_duration_since(last_ping));
            peer.last_ping_protocol_message_received_at = Some(now);
        }
    });
}
``` [4](#0-3)

### Impact Explanation
A malicious peer that can predict the nonce can:
1. Send a pong with the correct nonce immediately upon receiving a ping (or even proactively before the ping arrives), clearing `processing` and resetting `last_ping_protocol_message_received_at`.
2. Permanently avoid the `CHECK_TIMEOUT_TOKEN` disconnect path, maintaining its connection slot indefinitely regardless of actual responsiveness.
3. Fake an arbitrarily low `ping_rtt`, influencing any peer-selection or scoring logic that uses RTT.

This enables a malicious peer to occupy connection slots without being genuinely reachable, which is a direct enabler of eclipse-style attacks: the attacker keeps its slots alive while honest peers that fail real pings get evicted.

### Likelihood Explanation
The nonce is the node's uptime in seconds — a value in the range `[0, ~2^32)` that grows monotonically and is observable. Any peer that receives even one legitimate ping immediately learns the exact current nonce. Because the same nonce is broadcast to all peers, a single colluding peer can relay the nonce to others. Even without observing a ping, an attacker can estimate node uptime from public sources (e.g., peer discovery timestamps, version messages) and brute-force the small search space. No privileged access is required; any unprivileged TCP peer can exploit this.

### Recommendation
Replace the deterministic time-based nonce with a cryptographically random `u32` generated per-ping-round using `rand::thread_rng()` (which is already used elsewhere in the codebase, e.g., `network/src/peer_store/addr_manager.rs`). Each peer should ideally receive an independent random nonce rather than a shared broadcast value, to prevent nonce relay between colluding peers.

### Proof of Concept
1. Attacker connects to a CKB node as a normal peer.
2. Attacker receives the first `Ping` message; the nonce field equals `node_uptime_seconds`.
3. For every subsequent ping interval, the attacker computes `expected_nonce = observed_nonce + interval_seconds` and immediately sends `Pong(expected_nonce)` — or sends it proactively before the ping arrives.
4. The node's `received` handler matches `(true, expected_nonce) == (status.processing, status.nonce())`, calls `pong_received`, clears `processing`, and updates `last_ping_protocol_message_received_at`.
5. The `CHECK_TIMEOUT_TOKEN` handler never fires for this peer because `processing` is always `false` by the time the check runs.
6. The attacker's connection is never terminated by the ping-timeout mechanism, regardless of whether it forwards any real protocol traffic.
