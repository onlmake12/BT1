Audit Report

## Title
Orphan Pool Random Eviction Enables DoS via Pool Saturation - (File: `tx-pool/src/component/orphan.rs`)

## Summary
`OrphanPool::limit_size()` evicts entries using `HashMap::keys().next()` — an arbitrary, non-fee-ordered selection — when the pool exceeds `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100`. An attacker can saturate the orphan pool via the relay protocol's announce-request-send cycle using zero-fee, structurally valid transactions with non-existent inputs, causing legitimate orphan transactions to be permanently lost with no recovery path.

## Finding Description
**Root cause — `tx-pool/src/component/orphan.rs`, lines 119–125:**

```rust
while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
    let id = self.entries.keys().next().cloned().expect("bound checked");
    if let Some(entry) = self.remove_orphan_tx(&id) {
        evicted_txs.push(entry.tx.hash());
    }
}
```

`self.entries` is a `HashMap<ProposalShortId, Entry>`, so `keys().next()` yields entries in non-deterministic order. The `Entry` struct stores a `cycle` field (line 25) but it is never consulted during eviction. There is no fee-rate ordering, no `EvictKey`, and no per-peer slot cap.

**Reachable exploit path via relay protocol:**

The report's claim that transactions can be pushed directly via `RelayTransactions` is partially inaccurate. `TransactionsProcess::execute()` (`sync/src/relayer/transactions_process.rs`, lines 49–55) filters out any transaction not previously requested by the node:

```rust
.filter(|(tx, _)| {
    !tx_filter.contains(&tx.hash())
        && unknown_tx_hashes
            .get_priority(&tx.hash())
            .map(|priority| priority.requesting_peer() == Some(self.peer))
            .unwrap_or_default()
})
```

However, the attack remains fully feasible through the standard relay announce-request-send cycle:
1. Attacker announces crafted tx hashes via `RelayTransactionHashes`.
2. Node adds them to `unknown_tx_hashes` and sends `GetRelayTransactions` back to the attacker.
3. Attacker responds with the actual transactions (structurally valid, spending non-existent outpoints). These pass the filter and reach `submit_remote_tx`.
4. `resumeble_process_tx` → `enqueue_verify_queue` → verification runs → fails with `is_missing_input`.
5. `after_process()` (`process.rs`, lines 507–512) calls `self.add_orphan(tx, peer, declared_cycle)`.
6. `add_orphan()` (`process.rs`, lines 557–573) calls `orphan.add_orphan_tx()` → `limit_size()` with random eviction.
7. For each evicted transaction, `TxVerificationResult::Reject` is sent to the relayer, marking it as "unknown" in the relay filter.
8. When a legitimate orphan's parent arrives and `process_orphan_tx()` is called (`process.rs`, lines 591–671), the evicted child is absent and is silently lost — never promoted to the pending pool.

Note: `mark_as_known_txs` (line 76) prevents re-submission of the same tx hash. To maintain saturation after evictions, the attacker generates new unique transactions (different hashes, different non-existent outpoints) and repeats the announce-request cycle. This is stateless and requires no coordination.

Fee verification does not occur before orphan admission; it only happens after the parent is resolved. Zero-fee orphans are admitted freely.

## Impact Explanation
**High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

With a single P2P connection, no fees, no hashpower, and no key material, an attacker can permanently occupy the entire orphan pool of any targeted node. Applied across multiple nodes simultaneously, this degrades the network's ability to relay transactions whose parents are in-flight, silently dropping valid transactions from mempools and preventing their inclusion in blocks.

## Likelihood Explanation
The attack requires no privileges, no fees, no hashpower, and no key material. The announce-request-send cycle is a standard relay protocol operation. The attacker generates unlimited unique transactions (each spending a different randomly generated non-existent outpoint) to continuously re-fill evicted slots. The orphan pool limit of 100 is small enough that saturation is achieved in a single batch. The attack is repeatable and requires no coordination.

## Recommendation
1. **Fee-rate-ordered eviction**: Replace `self.entries.keys().next()` in `limit_size()` with an iteration that finds and evicts the entry with the lowest declared fee rate (derivable from `cycle` and transaction size stored in `Entry`). This mirrors the `EvictKey`-based strategy in `PoolMap::next_evict_entry()` (`tx-pool/src/component/pool_map.rs`, lines 380–385).
2. **Per-peer slot cap**: Limit the number of orphan slots any single peer can occupy (e.g., `DEFAULT_MAX_ORPHAN_TRANSACTIONS / 4 = 25`) to prevent monopolization regardless of declared fee rate.
3. **Minimum declared fee rate for orphan admission**: Reject orphan transactions below a configurable minimum declared fee rate before inserting into the pool.

## Proof of Concept
1. Attacker connects to a CKB node as a P2P peer via the relay protocol.
2. Attacker generates 100 transactions (`spam_0`…`spam_99`), each spending a randomly generated non-existent outpoint as input. These pass `non_contextual_verify` (structurally valid) but fail resolution with `is_missing_input`.
3. Attacker announces all 100 tx hashes via `RelayTransactionHashes`. Node adds them to `unknown_tx_hashes` and sends `GetRelayTransactions`. Attacker responds with the transactions. Node admits them into the orphan pool (`len == 100`).
4. Victim sends a legitimate orphan transaction `victim_tx` (child of a parent currently propagating on the network) through the same announce-request cycle.
5. Node calls `add_orphan_tx(victim_tx)` → `limit_size()` → randomly evicts one entry. With probability 100/101 ≈ 99%, the evicted entry is one of the attacker's spam transactions; with probability 1/101, `victim_tx` itself is immediately evicted.
6. Attacker generates a new unique spam transaction `spam_100` (new non-existent outpoint), announces it, waits for the node to request it, and sends it. Pool returns to 100 attacker-controlled entries.
7. On the next legitimate orphan insertion, `victim_tx` (if still present) faces another eviction probability. Over repeated cycles, `victim_tx` is eventually evicted.
8. When `victim_tx`'s parent arrives and `process_orphan_tx()` is called, `victim_tx` is absent from the orphan pool and is permanently lost — never submitted to the pending pool.
