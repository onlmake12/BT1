### Title
Unbounded `pending_get_block_proposals` DashMap Growth via Unauthenticated Peer `GetBlockProposal` Messages - (File: sync/src/types/mod.rs)

### Summary
Any connected peer can send repeated `GetBlockProposal` relay messages containing unique proposal IDs that are absent from the local tx pool. Each such message causes entries to be inserted into the `pending_get_block_proposals` DashMap with no size cap, allowing unbounded memory growth between periodic drain cycles.

### Finding Description

`SyncState` holds a `pending_get_block_proposals` field declared as a bare `DashMap` with no capacity bound: [1](#0-0) 

The insertion path in `insert_get_block_proposals` performs no size check before writing: [2](#0-1) 

This function is called unconditionally from `GetBlockProposalProcess::execute()` for every proposal ID that is not found in the local tx pool: [3](#0-2) 

The only per-message guard is a check that the proposal count does not exceed `max_block_proposals_limit * max_uncles_num` (mainnet: 1500 × 2 = 3000 IDs per message): [4](#0-3) 

There is no per-peer rate limit, no total-map size limit, and no eviction policy. The only cleanup is `drain_get_block_proposals`, which clears the entire map on a periodic timer tick inside `prune_tx_proposal_request`: [5](#0-4) 

Between consecutive timer ticks an attacker can insert an arbitrary number of unique `ProposalShortId` entries by sending back-to-back `GetBlockProposal` messages whose IDs are not present in the victim's tx pool.

The CKB changelog records an analogous fix for the related `inflight_proposals` DashMap (`#3093: Resolve inflight proposals memory bloat issue`) and for `pending_compact_blocks` (`#3110`), confirming this class of unbounded-map growth is a known concern in this subsystem. `pending_get_block_proposals` received no equivalent bound. [6](#0-5) 

### Impact Explanation

**Impact: High** — Memory exhaustion / denial of service on the victim full node.

Each `ProposalShortId` is 10 bytes; with DashMap overhead and the `HashSet<PeerIndex>` value, each entry costs on the order of ~100–200 bytes. A single peer sending 3 000-ID messages at network speed can insert millions of entries per second, exhausting available RAM and causing the node process to be OOM-killed or to stall under allocator pressure. This disrupts block relay, transaction propagation, and all other node services for the duration of the attack.

### Likelihood Explanation

**Likelihood: Medium** — Any peer that completes the CKB P2P handshake can send `GetBlockProposal` messages. No stake, bond, or privileged role is required. The attacker simply needs to maintain a connection and flood messages with fresh random `ProposalShortId` values (10 random bytes each) that will not be found in the victim's tx pool, ensuring every ID is cached. The cost to the attacker is negligible network bandwidth; the cost to the victim is unbounded memory.

### Recommendation

1. **Hard cap on `pending_get_block_proposals`**: Reject or drop insertions once the map exceeds a configurable maximum (e.g., `max_block_proposals_limit * max_connected_peers`).
2. **Per-peer entry limit**: Track how many entries each peer has contributed and refuse further insertions from peers that exceed their quota, mirroring the per-peer guard already present in `add_ask_for_txs` for `unknown_tx_hashes`.
3. **Eviction on overflow**: When the cap is reached, evict the oldest entries (or entries from the most prolific peer) rather than silently accepting them.

### Proof of Concept

```
Attacker (connected peer):
  loop:
    ids = [random_10_bytes() for _ in range(3000)]   # all absent from victim tx pool
    send GetBlockProposal { block_hash: any_valid_hash, proposals: ids }
    # Each iteration inserts up to 3000 new entries into pending_get_block_proposals
    # with no eviction until the next prune_tx_proposal_request timer tick
```

Entry path:
1. Peer sends `GetBlockProposal` → `Relayer::try_process` dispatches to `GetBlockProposalProcess::execute`.
2. Per-message count check passes (≤ 3000 IDs).
3. `tx_pool.fetch_txs(proposals)` returns empty (IDs are random / unknown).
4. All 3000 IDs are passed to `insert_get_block_proposals` → inserted into `pending_get_block_proposals` with no size guard.
5. Repeat before the next `prune_tx_proposal_request` timer fires.
