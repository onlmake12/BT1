### Title
Malicious Requester Can Permanently Grief Legacy-Path `revealWithCallback` via Reverting `_entropyCallback` — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
In `Entropy.sol`, the `revealWithCallback` function has two execution paths. When a provider has `defaultGasLimit == 0` (the legacy path, `req.gasLimit10k == 0`), the consumer's `_entropyCallback` is invoked **directly** with no revert-catching wrapper. A malicious requester can deploy a contract whose `_entropyCallback` always reverts, permanently preventing the provider from fulfilling the request and advancing their commitment chain.

### Finding Description

`revealWithCallback` branches on `req.gasLimit10k`:

- **New path** (`gasLimit10k != 0`): uses `excessivelySafeCall`, catches reverts, emits `CallbackFailed`, and transitions the request to `CALLBACK_FAILED` state — the request can be retried.
- **Legacy path** (`gasLimit10k == 0`): calls `_entropyCallback` directly with no try/catch. Any revert in the callback propagates up and reverts the entire transaction, including the `clearRequest` and the `revealHelper` state update to `currentCommitmentSequenceNumber`. [1](#0-0) 

The legacy path is entered whenever `providerInfo.defaultGasLimit == 0` at request time: [2](#0-1) 

`requestWithCallback` always passes `gasLimit = 0`, so any provider that has not called `setDefaultGasLimit` will produce requests with `gasLimit10k == 0`: [3](#0-2) 

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` confirms the revert propagates: [4](#0-3) 

### Impact Explanation

1. **Permanent request lock**: Because `clearRequest` is inside the same transaction as the direct callback call, a revert rolls back the storage clear. The request slot is permanently occupied and the user's fee is unrecoverable.
2. **Commitment chain stall**: `revealHelper` updates `providerInfo.currentCommitmentSequenceNumber` to the fulfilled sequence number. A revert rolls this back too, so the provider's commitment pointer cannot advance past the stuck sequence number.
3. **Future-request DoS**: If the provider has set `maxNumHashes`, all subsequent requests will compute `numHashes = assignedSeq - currentCommitmentSequenceNumber` against the stalled pointer. Once the gap exceeds `maxNumHashes`, every new `requestWithCallback` reverts with `LastRevealedTooOld`, effectively DoS-ing the provider. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

- Any unprivileged user can deploy a contract that reverts unconditionally in `_entropyCallback`.
- The attack targets providers with `defaultGasLimit == 0`. These are real, deployed legacy providers — the `requestWithCallback` NatDoc explicitly notes the provider's `defaultGasLimit` governs gas, and providers that have not migrated to V2 will have this field at zero.
- The attacker pays the request fee (self-funded griefing), but the cost is bounded by the fee amount, while the damage to the provider (commitment stall, potential future-request DoS) is disproportionate.

### Recommendation

1. **Wrap the legacy-path callback in a try/catch** (or `excessivelySafeCall`) identical to the new path, so a reverting callback transitions the request to `CALLBACK_FAILED` rather than reverting the entire transaction.
2. **Alternatively**, deprecate the legacy path entirely: require all providers to set a non-zero `defaultGasLimit` before accepting new `requestWithCallback` calls, ensuring all requests always use the safe new path.
3. **Short-term mitigation**: document that providers must call `setDefaultGasLimit` to protect themselves from this griefing vector.

### Proof of Concept

```solidity
// Malicious requester contract
contract MaliciousRequester is IEntropyConsumer {
    IEntropy entropy;
    constructor(address _entropy) { entropy = IEntropy(_entropy); }

    // Always reverts — blocks provider from ever clearing this request
    function entropyCallback(uint64, address, bytes32) internal override {
        revert("griefed");
    }
    function getEntropy() internal view override returns (address) {
        return address(entropy);
    }
    function attack(address provider) external payable {
        // Provider must have defaultGasLimit == 0 (legacy)
        entropy.requestWithCallback{value: msg.value}(provider, bytes32(uint(1)));
    }
}
```

1. Deploy `MaliciousRequester` and call `attack(legacyProvider)` with the required fee.
2. The provider calls `revealWithCallback` — `_entropyCallback` reverts, the whole tx reverts.
3. The request is never cleared; `currentCommitmentSequenceNumber` stays stalled.
4. If `maxNumHashes` is configured, subsequent legitimate requests to the same provider revert with `LastRevealedTooOld`. [7](#0-6)
