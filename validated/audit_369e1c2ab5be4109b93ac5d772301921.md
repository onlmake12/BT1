Audit Report

## Title
`Scheduler::resume()` Unconditionally Zeroes `iteration_cycles`, Discarding Pre-Suspension IO Cycles and Understating Total Consumed Cycles â€” (File: `script/src/scheduler.rs`)

## Summary

`Scheduler::resume()` unconditionally sets `iteration_cycles = 0` at line 242 after calling `ensure_vms_instantiated`. This also destroys the non-zero `iteration_cycles` value faithfully saved by `suspend()` â€” cycles charged by `ensure_vms_instantiated` calls inside `process_io()` during the final iteration before suspension. Those cycles are not present in `total_cycles` and are permanently lost, causing the scheduler to understate total consumed cycles by up to `N Ă— 2 Ă— SPAWN_EXTRA_CYCLES_BASE` (200,000 cycles per boundary, since `SPAWN_EXTRA_CYCLES_BASE = 100_000`) across N suspend/resume cycles.

## Finding Description

**Cycle accounting invariant.** The scheduler tracks consumed cycles across three accumulators: `total_cycles`, `iteration_cycles`, and `machine.cycles()`. The code and comments at lines 59â€“97 explicitly document this invariant and acknowledge that `iteration_cycles` will not always be zero at suspension boundaries due to the Meepo hardfork implementation.

**`iterate_outer` flow.** In `iterate_outer` (lines 417â€“456), after `iterate_inner` returns `Err(Pause)`:
1. `consume_cycles(self.iteration_cycles)` commits `iteration_cycles` to `total_cycles` (line 423)
2. `self.iteration_cycles = 0` resets the accumulator (line 428)
3. `self.process_io()` is called (line 453) â€” this calls `ensure_vms_instantiated(&[read_vm_id, write_vm_id])` at line 802, which in turn calls `resume_vm` and/or `suspend_vm`, each adding `SPAWN_EXTRA_CYCLES_BASE = 100_000` directly to `self.iteration_cycles` (lines 954â€“957 and 976â€“979)
4. `iterate_return?` propagates `Err(Pause)` â€” the function exits with `iteration_cycles` now non-zero

**`suspend()` correctly preserves the non-zero value.** `suspend()` (lines 261â€“276) saves `iteration_cycles: self.iteration_cycles` into `FullSuspendedState`. The `FullSuspendedState` documentation at lines 498â€“501 explicitly acknowledges this value will not always be zero.

**`resume()` discards the preserved value.** `resume()` (lines 204â€“244):
- Line 216: correctly restores `iteration_cycles: full.iteration_cycles`
- Line 237: calls `ensure_vms_instantiated(&full.instantiated_ids)`, adding more cycles to `iteration_cycles`
- Line 242: **unconditionally** sets `scheduler.iteration_cycles = 0`

The comment at lines 239â€“241 states the intent is to avoid charging cycles for resume overhead, but the reset also destroys the pre-suspension `iteration_cycles` representing real IO data-transfer work. Those cycles are not in `total_cycles` (they were charged after the `consume_cycles` call in `iterate_outer`) and are permanently lost.

**`chunk_run` path.** When `run()` returns `Err(Pause)`, `chunk_run` (lines 500â€“505) calls `scheduler.suspend()` and returns `ChunkState::Suspended`. On the next chunk, `resume_scheduler` calls `Scheduler::resume()`, which zeroes the saved `iteration_cycles`.

## Impact Explanation

After each resume, `total_cycles` is understated by the discarded `iteration_cycles`. The per-chunk cycle budget (`max_cycles`) is not reduced to compensate, so the script receives extra execution budget equal to the lost IO cycles on every resume. With `SPAWN_EXTRA_CYCLES_BASE = 100_000`, a single read/write VM pair in `process_io` charges at minimum 200,000 cycles (one `resume_vm` per VM) into `iteration_cycles` before suspension â€” all of which are discarded on resume. Across N suspend/resume cycles, the undercount compounds to `N Ă— 200,000+` cycles. A script declared at cycle limit `L` can actually consume `L + N Ă— 200,000+` cycles while the tx-pool reports `L`. This matches **"Incorrect implementation or behavior of CKB-VM or system scripts" (High, 10001â€“15000 points)**.

## Likelihood Explanation

No privileged access is required. Any transaction submitter can craft a script using `spawn` and `pipe` syscalls so that a read/write VM pair is in IO-wait state when the tx-pool's `ChunkCommand::Suspend` signal fires. At that moment `process_io()` calls `ensure_vms_instantiated` for the pair, charging cycles into `iteration_cycles`. The attack is repeatable across every suspend/resume boundary and requires only submitting a transaction with a script that maintains persistent pipe IO across iterations.

## Recommendation

In `resume()`, save the pre-suspension `iteration_cycles` before calling `ensure_vms_instantiated`, then restore it afterward. Only the cycles added by `ensure_vms_instantiated` itself (resume overhead) should be discarded:

```rust
let saved_iteration_cycles = full.iteration_cycles;
// ... build scheduler with iteration_cycles: full.iteration_cycles ...
scheduler.ensure_vms_instantiated(&full.instantiated_ids).unwrap();
// Discard only resume-overhead cycles; restore pre-suspension IO cycles.
scheduler.iteration_cycles = saved_iteration_cycles;
```

Alternatively, `suspend()` should flush `iteration_cycles` into `total_cycles` before saving state, ensuring `iteration_cycles` is always zero at suspension boundaries and making the reset in `resume()` safe. The existing comments flag this as a known concern requiring a future hardfork fix due to `current cycles` syscall behavior.

## Proof of Concept

1. Submit a transaction containing a script that spawns a child VM and opens a pipe between them.
2. The child VM writes data; the parent VM reads. Both are in IO-wait state (`WaitForRead`/`WaitForWrite`) at the end of an iteration.
3. The tx-pool issues `ChunkCommand::Suspend`, triggering `Err(Pause)` from `iterate_inner`.
4. `iterate_outer` commits `iteration_cycles` to `total_cycles` via `consume_cycles`, resets `iteration_cycles = 0`, then calls `process_io()`.
5. `process_io()` calls `ensure_vms_instantiated(&[read_vm_id, write_vm_id])` (line 802). Each `resume_vm` call adds `SPAWN_EXTRA_CYCLES_BASE = 100_000` to `iteration_cycles` (lines 954â€“957). With two VMs, `iteration_cycles = 200_000`.
6. `iterate_outer` propagates `Err(Pause)`; `chunk_run` calls `scheduler.suspend()`.
7. `suspend()` saves: `total_cycles = T`, `iteration_cycles = 200_000`.
8. `resume()` restores `total_cycles = T`, then sets `iteration_cycles = 0`. The 200,000 cycles are permanently lost.
9. Next chunk runs with full `max_cycles` budget; the scheduler believes only `T` cycles have been consumed, actual is `T + 200_000`.
10. Repeat across N chunks: `N Ă— 200_000` cycles of bypass accumulate. A script declared at limit `L` can actually consume `L + N Ă— 200_000` cycles while the tx-pool reports `L`.
