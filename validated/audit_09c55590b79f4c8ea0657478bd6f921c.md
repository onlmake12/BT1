### Title
Front-Runnable `updatePriceFeeds` Allows Keeper Fee Theft and Keeper DoS — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.updatePriceFeeds()` has no `msg.sender` access control and is callable by any address. An attacker can observe a legitimate keeper's pending transaction in the mempool, duplicate it with the same `subscriptionId` and `updateData`, and front-run it. The attacker's transaction succeeds first, updates the subscription's `priceLastUpdatedAt`, and collects the keeper fee. The original keeper's transaction then reverts with `TimestampOlderThanLastUpdate`. This is a direct structural analog to the EIP-2612 permit front-running pattern.

---

### Finding Description

`Scheduler.updatePriceFeeds()` is an open, permissionless function:

```solidity
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {
    uint256 startGas = gasleft();
    ...
    _processFeesAndPayKeeper(status, startGas, params.priceIds.length);
    emit PricesUpdated(subscriptionId, latestPublishTime);
}
``` [1](#0-0) 

After the Pyth oracle call succeeds and prices are stored, the contract pays a keeper fee to `msg.sender` via `_processFeesAndPayKeeper`. [2](#0-1) 

The staleness guard that causes the original keeper's transaction to revert is:

```solidity
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(
        updateTimestamp,
        status.priceLastUpdatedAt
    );
}
``` [3](#0-2) 

Once the attacker's front-run transaction sets `status.priceLastUpdatedAt` to `updateTimestamp`, the original keeper's transaction hits this exact check and reverts, because `updateTimestamp <= status.priceLastUpdatedAt` (equal). The subscription balance has already been debited for the Pyth fee and the keeper fee has been paid to the attacker. [4](#0-3) 

The `reveal()` function in `Entropy.sol` explicitly documents this exact attack class and defends against it with a `msg.sender == req.requester` check:

> "This function must be called by the same `msg.sender` that originally requested the random number. This check prevents denial-of-service attacks where another actor front-runs the requester's reveal transaction." [5](#0-4) 

`Scheduler.updatePriceFeeds()` has no equivalent protection.

---

### Impact Explanation

1. **Keeper fee theft**: The attacker collects the keeper fee (gas reimbursement + per-feed fee) that was intended for the legitimate keeper. This is a direct financial loss to keepers.
2. **Keeper griefing / DoS**: Legitimate keepers consistently lose gas without compensation. If this is automated at scale (e.g., a bot watching the mempool), keepers are economically disincentivized from operating, which can lead to subscriptions not being updated and stale prices being served to downstream consumers.
3. **Subscription balance drain**: Each front-run still debits the subscription's `balanceInWei` for the Pyth oracle fee. A subscription owner's balance is consumed at the correct rate, but the keeper fee goes to the attacker rather than the legitimate keeper.

---

### Likelihood Explanation

- `updatePriceFeeds` is a public, permissionless function with no `msg.sender` restriction.
- The `updateData` payload is fully visible in the mempool before inclusion.
- The attacker only needs to copy the calldata and submit with a higher gas price.
- On chains with a public mempool (Ethereum mainnet, most L2s with sequencer transparency), this is trivially executable by any MEV bot.
- The attacker has a direct financial incentive (keeper fee), making sustained exploitation economically rational.

---

### Recommendation

Add a whitelist or access-control mechanism for who can call `updatePriceFeeds`, similar to how `reveal()` in `Entropy.sol` restricts callers to `req.requester`. One approach:

```solidity
// Option A: restrict to a registered keeper set
modifier onlyKeeper() {
    require(_state.registeredKeepers[msg.sender], "Not a registered keeper");
    _;
}

// Option B: commit-reveal or signed update data binding msg.sender
```

Alternatively, bind the keeper fee payment to a pre-committed keeper address embedded in the update data or a separate registration step, so that front-running the call does not transfer the fee to the attacker.

### Proof of Concept

```solidity

pragma solidity ^0.8.0;

interface IScheduler {
    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external;
}

contract FrontRunKeeper {
    address schedulerAddress;

    constructor(address _scheduler) {
        schedulerAddress = _scheduler;
    }

    // Called by attacker bot after observing a keeper's pending TX in mempool.
    // Attacker copies subscriptionId and updateData from the pending TX,
    // submits this with higher gas price to front-run.
    function stealKeeperFee(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external {
        // msg.sender = attacker; keeper fee is paid to msg.sender
        IScheduler(schedulerAddress).updatePriceFeeds(subscriptionId, updateData);
        // Attacker receives keeper fee; original keeper's TX reverts with
        // TimestampOlderThanLastUpdate when it executes next.
    }
}
