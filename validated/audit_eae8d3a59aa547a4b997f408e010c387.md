### Title
Compact Block Index Misalignment via Differing `prefilled_transactions` Causes Legitimate Peer Ban — (`sync/src/relayer/compact_block_process.rs`)

---

### Summary

When multiple peers send compact blocks for the same block hash but with different `prefilled_transactions` sets, the `pending_compact_blocks` map stores only the **first** compact block received (via `or_insert_with`) while storing per-peer missing transaction indexes computed from **each peer's own** compact block layout. When `BlockTransactionsProcess` later verifies a peer's response, it uses the stored (first peer's) compact block against the responding peer's indexes, producing a layout mismatch that results in the legitimate peer being banned.

---

### Finding Description

**Root cause — `missing_or_collided_post_process` in `compact_block_process.rs`:** [1](#0-0) 

```rust
shared
    .state()
    .pending_compact_blocks()
    .await
    .entry(block_hash.clone())
    .or_insert_with(|| (compact_block, HashMap::default(), unix_time_as_millis()))
    .1
    .insert(peer, (missing_transactions.clone(), missing_uncles.clone()));
```

`or_insert_with` keeps the **first** compact block received for a given block hash. If peer B arrives after peer A, peer B's compact block is silently discarded. However, `missing_transactions` for peer B — computed from peer B's own compact block layout — are stored in the `peers_map` under peer B's index.

**Mismatch at verification — `BlockTransactionsProcess::execute` in `block_transactions_process.rs`:** [2](#0-1) 

```rust
let (compact_block, peers_map, _) = pending.get_mut();
if let Entry::Occupied(mut value) = peers_map.entry(self.peer) {
    let (expected_transaction_indexes, expected_uncle_indexes) = value.get_mut();
    attempt!(BlockTransactionsVerifier::verify(
        compact_block,                    // ← stored compact block (peer A's)
        expected_transaction_indexes,     // ← indexes computed from peer B's layout
        &received_transactions,
    ));
```

**How the layout diverges:**

A `CompactBlock` for the same block hash can legally have different `prefilled_transactions` sets across peers. The `short_ids` list and the meaning of each positional index depend entirely on which transactions are prefilled. [3](#0-2) 

Example with block `[tx0, tx1, tx2, tx3, tx4]`:

| Peer | `prefilled` | `short_ids` | Node missing (tx1 in pool) | Stored indexes |
|------|-------------|-------------|---------------------------|----------------|
| A (stored) | `{0, 2}` | `[tx1_sid, tx3_sid, tx4_sid]` | tx3@3, tx4@4 | `[3, 4]` |
| B (discarded) | `{0, 3}` | `[tx1_sid, tx2_sid, tx4_sid]` | tx2@2, tx4@4 | `[2, 4]` |

Node sends `GetBlockTransactions` to peer B with indexes `[2, 4]`. Peer B correctly responds with `[tx2, tx4]`.

**`BlockTransactionsVerifier::verify` then executes:** [4](#0-3) 

Using peer A's compact block, `block_short_ids[2]` is `None` (tx2 is prefilled in peer A's block). The `filter_map` produces only one short_id (from index 4), but `transactions.len() == 2`, triggering:

```
BlockTransactionsLengthIsUnmatchedWithPendingCompactBlock
```

This is a **4xx status code**, causing peer B to be **banned**. [5](#0-4) 

The `PendingCompactBlockMap` type confirms a single compact block is shared across all peers: [6](#0-5) 

---

### Impact Explanation

1. **Legitimate peer ban**: Any peer that sends a compact block with a different `prefilled_transactions` set than the first-arriving peer will be banned when it correctly responds to `GetBlockTransactions`.
2. **Block propagation stall**: If the first compact block stored has a prefilled set that causes reconstruction to fail (e.g., the attacker prefills nothing), and all subsequent honest peers are banned, the node cannot reconstruct the block from the relay path.
3. **Cascading effect**: The node may fall back to the slower sync path, delaying tip advancement.

---

### Likelihood Explanation

- Any unprivileged P2P peer can send a compact block for a newly announced block hash.
- The attacker only needs to relay the compact block slightly before honest peers, with a different (e.g., minimal) `prefilled_transactions` set.
- No special keys, hashpower, or Sybil capability is required — a single malicious peer connected to the victim is sufficient.
- The `contextual_check` only rejects a compact block if the **same peer** already has a pending entry, so the attacker's entry is not overwritten by honest peers. [7](#0-6) 

Likelihood: **Medium** — requires timing (arrive first) but no privileged access.

---

### Recommendation

1. **Per-peer compact block storage**: Store each peer's compact block separately (e.g., `HashMap<PeerIndex, (CompactBlock, Vec<u32>, Vec<u32>)>`) so that `BlockTransactionsVerifier::verify` always uses the compact block that generated the expected indexes for that peer.
2. **Normalize on first compact block**: When a second compact block arrives for the same block hash, recompute the missing indexes using the **stored** compact block's layout before sending `GetBlockTransactions` to the new peer, and store those recomputed indexes.
3. **Avoid banning on index mismatch**: Treat `BlockTransactionsLengthIsUnmatchedWithPendingCompactBlock` and `BlockTransactionsShortIdsAreUnmatchedWithPendingCompactBlock` as non-bannable when the mismatch can be explained by a layout difference (i.e., the peer's response is consistent with a valid alternative compact block for the same hash).

### Proof of Concept

**Attacker-controlled entry path**: unprivileged P2P relay peer → `CompactBlock` message → `CompactBlockProcess::execute` → `missing_or_collided_post_process` (stores attacker's compact block first) → honest peer's compact block discarded → honest peer's `BlockTransactions` response verified against attacker's layout → honest peer banned.

**Concrete steps**:

1. Honest miner mines block B with transactions `[tx0(cellbase), tx1, tx2]`.
2. Attacker connects to victim node and sends `CompactBlock` for B with `prefilled = {0}` (only cellbase), `short_ids = [tx1_sid, tx2_sid]`. Victim stores this compact block. Victim has `tx1` in pool, so missing = `[2]` (tx2). Victim sends `GetBlockTransactions{indexes=[2]}` to attacker.
3. Honest peer sends `CompactBlock` for B with `prefilled = {0, 1}` (cellbase + tx1), `short_ids = [tx2_sid]`. `or_insert_with` keeps attacker's compact block. Honest peer's missing = `[2]` (tx2 at index 2 in honest peer's layout). Victim sends `GetBlockTransactions{indexes=[2]}` to honest peer.
4. Honest peer responds with `BlockTransactions{transactions=[tx2]}`.
5. `BlockTransactionsVerifier::verify(attacker_compact_block, [2], [tx2])`: attacker's `block_short_ids[2]` = `tx2_sid` — this actually matches in this simple case.

Let me refine: use a case where the prefilled set causes a genuine index shift.

**Refined PoC** with block `[tx0, tx1, tx2, tx3]`:

- Attacker sends: `prefilled={0}`, `short_ids=[tx1_sid, tx2_sid, tx3_sid]`. Node has tx1 in pool. Missing = `[2, 3]`. Stored: attacker's CB.
- Honest peer sends: `prefilled={0, 1}`, `short_ids=[tx2_sid, tx3_sid]`. Missing (from honest CB) = `[2, 3]`. Stored indexes for honest peer = `[2, 3]`.
- Honest peer responds with `[tx2, tx3]`.
- Verify: attacker's `block_short_ids[2]` = `tx2_sid` ✓, `block_short_ids[3]` = `tx3_sid` ✓ — matches here too.

The divergence is clearest when the **prefilled set changes which index a given short_id occupies**:

- Block: `[tx0, tx1, tx2, tx3]`
- Attacker: `prefilled={0, 2}`, `short_ids=[tx1_sid, tx3_sid]`. Node has tx1 in pool. Missing = `[3]`. Stored: attacker's CB.
- Honest peer: `prefilled={0, 1}`, `short_ids=[tx2_sid, tx3_sid]`. Missing (from honest CB) = `[2, 3]`. Stored indexes for honest peer = `[2, 3]`.
- Honest peer responds with `[tx2, tx3]`.
- Verify: attacker's `block_short_ids[2]` = `None` (tx2 is prefilled in attacker's CB). `filter_map` yields only `[tx3_sid]` (1 item). But `transactions.len() = 2`. → `BlockTransactionsLengthIsUnmatchedWithPendingCompactBlock` → **honest peer banned**. [8](#0-7) [9](#0-8) [10](#0-9)

```rust
    pub(crate) fn verify(
        block: &packed::CompactBlock,
        indexes: &[u32],
        transactions: &[core::TransactionView],
    ) -> Status {
        let block_short_ids = block.block_short_ids();
        let missing_short_ids: Vec<packed::ProposalShortId> = indexes
            .iter()
            .filter_map(|index| {
                block_short_ids
                    .get(*index as usize)
                    .expect("should never outbound")
                    .clone()
            })
            .collect();

        if missing_short_ids.len() != transactions.len() {
            return StatusCode::BlockTransactionsLengthIsUnmatchedWithPendingCompactBlock
                .with_context(format!(
                    "Expected({}) != actual({})",
                    missing_short_ids.len(),
                    transactions.len(),
                ));
        }

        for (expected_short_id, tx) in missing_short_ids.into_iter().zip(transactions) {
            let short_id = tx.proposal_short_id();
            if expected_short_id != short_id {
                return StatusCode::BlockTransactionsShortIdsAreUnmatchedWithPendingCompactBlock
                    .with_context(format!(
                        "Expected({expected_short_id}) != actual({short_id})",
                    ));
            }
        }

        Status::ok()
    }
```
