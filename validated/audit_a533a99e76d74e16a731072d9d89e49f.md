### Title
`Indexer::prune` skips all Header/TxHash cleanup when no `ConsumedOutPoint` entries exist — (`util/indexer/src/indexer.rs`)

### Summary
`prune()` derives its Header-scan start key from `min_block_number`, which is only set inside the `ConsumedOutPoint` deletion loop. When that loop is empty (no consumed out-points exist), `min_block_number` stays at `u64::MAX`, the Header iterator starts past all stored keys, and neither `Header` nor `TxHash` entries are ever deleted.

### Finding Description
In `Indexer::prune`:

1. `min_block_number` is initialised to `BlockNumber::MAX` (line 765).
2. It is updated only inside the `ConsumedOutPoint` loop (line 777-779).
3. `ConsumedOutPoint` entries are written only for non-cellbase inputs (`tx_index > 0`, line 339). Cellbase inputs are explicitly skipped.
4. Therefore, a chain consisting entirely of cellbase-only blocks leaves the `ConsumedOutPoint` prefix empty.
5. `key_prefix_header` is built as `[KeyPrefix::Header as u8] ++ u64::MAX.to_be_bytes()` (line 784-785), placing the iterator cursor at the very end of the Header keyspace.
6. The `take_while` guard `block_number <= prune_to_block` is false immediately; the loop body never executes.
7. `batch.commit()` writes nothing for the Header/TxHash section.

Result: every `TxHash` entry written during `append` for cellbase outputs persists forever.

### Impact Explanation
The indexer's RocksDB store accumulates one `TxHash` entry per matched transaction per block, with no upper bound. Over a long-running chain with low (or zero) non-cellbase activity, this causes monotonically growing disk usage in the indexer process. Nodes running the built-in indexer (`ckb-indexer`) are affected; disk exhaustion terminates the process.

### Likelihood Explanation
Cellbase-only blocks are valid and common during low-activity periods. Any miner (even one with minimal hashpower) can produce them. The bug also fires under normal mainnet conditions whenever a stretch of blocks contains no non-cellbase transactions that touch indexed cells. No special privileges beyond block production are required.

### Recommendation
Replace the `min_block_number`-derived start key with a fixed start key of `[KeyPrefix::Header as u8]` (block 0), and let the existing `take_while` guard `block_number <= prune_to_block` bound the scan. The `ConsumedOutPoint` loop should remain independent of the Header/TxHash loop. Alternatively, initialise `min_block_number` to `0` so the Header scan always starts from the beginning of the keyspace.

### Proof of Concept
```
1. Create an Indexer with keep_num = 100.
2. Append 202 cellbase-only blocks (each block contains only a cellbase tx
   whose output matches the default filter).
3. Call indexer.prune().
4. Iterate the TxHash prefix and count entries.
5. Expected (correct): ≤ 101 entries (keep_num + 1 blocks worth).
6. Observed (buggy):   202 entries — all TxHash entries since genesis,
   none pruned.
```

The `prune_bound` test at line 1534 already constructs exactly this scenario (21 cellbase-only blocks) but never asserts on `TxHash` count, so the regression goes undetected. [7](#0-6)
