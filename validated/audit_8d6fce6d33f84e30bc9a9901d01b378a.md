Audit Report

## Title
Stale `descendants_fee` After Subtree Removal Enables Eviction Bypass and Pool-Fill Attack - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-removes all link entries for the entire subtree before calling `remove_entry` on each node. This causes `update_ancestors_index_key` to invoke `calc_ancestors` on an already-unlinked entry, receiving an empty ancestor set and silently skipping `sub_descendant_weight` on every surviving ancestor. An unprivileged attacker can exploit this via repeated RBF replacements to permanently inflate a low-fee parent's `descendants_fee`, making it immune to eviction and enabling pool-filling attacks that cause legitimate transactions to be rejected with `Reject::Full`.

## Finding Description
`remove_entry_and_descendants` first strips all link entries for the entire removed subtree, then calls `remove_entry` for each:

```rust
// pool_map.rs L252-265
for id in &removed_ids {
    self.remove_entry_links(id);  // removes from links.inner
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry_links` calls `self.links.remove(id)` (pool_map.rs L429), which removes the entry from `TxLinksMap::inner`. Inside `remove_entry`, `update_ancestors_index_key` is called with `EntryOp::Remove` (pool_map.rs L242). This calls `self.links.calc_ancestors(&child.proposal_short_id())`, which delegates to `calc_relative_ids`, which does `self.inner.get(short_id)`. Because `remove_entry_links` already called `self.links.remove(id)`, this returns `None`, so `direct` is `unwrap_or_default()` (empty `HashSet`), and the entire ancestor walk returns an empty set (links.rs L42-47). No ancestor IDs are iterated, so `sub_descendant_weight` is never called on any surviving ancestor. Their `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` remain permanently inflated.

The comment in `remove_entry_and_descendants` ("update links state for remove, so that we won't update_descendants_index_key in remove_entry") reveals the intent was only to suppress redundant descendant updates, but the pre-removal also silently breaks the ancestor update path.

`sub_descendant_weight` uses `saturating_sub` throughout (entry.rs L134-141), so any future underflow silently zeros the field rather than panicking, and there is no recompute path for `descendants_fee` analogous to the `recompute_total_stat` fallback used for `total_tx_size`.

The stale `descendants_fee` directly feeds `EvictKey` via `descendants_feerate.max(feerate)` (entry.rs L240-243, sort_key.rs L93-103). `next_evict_entry` iterates `iter_by_evict_key()` from lowest to highest (pool_map.rs L380-385); an entry with an artificially inflated `descendants_feerate` ranks higher and is never selected for eviction.

## Impact Explanation
A surviving ancestor transaction retains an inflated `descendants_fee`. Its `EvictKey` is computed as `descendants_feerate.max(feerate)`. With an artificially high `descendants_feerate`, the entry ranks as highly valuable and is never selected for eviction by `next_evict_entry`. An attacker can keep a near-zero-fee parent permanently in the pool, repeatedly inflate its apparent value by submitting and RBF-replacing high-fee children, and fill the pool with such entries, causing legitimate transactions to be rejected with `Reject::Full`. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation
Any node's RPC endpoint accepts `send_transaction` from unprivileged callers. RBF is enabled whenever `min_rbf_rate > min_fee_rate`. The attacker pays incrementally higher fees per replacement (RBF requirement), but each replacement permanently inflates the parent's `descendants_fee` by the replaced child's fee. After N replacements the inflation is `sum(child_fees[0..N-1])`, which grows without bound. No privileged access, key material, or majority hashpower is required.

## Recommendation
Before calling `remove_entry_links` for any entry in the subtree, collect the root's surviving ancestors (those not in `removed_ids`) and apply `sub_descendant_weight` to each of them. Concretely, in `remove_entry_and_descendants`, compute `calc_ancestors(id)` for the root **before** any link removal, filter out IDs that are themselves in `removed_ids`, then iterate over those ancestor IDs and call `sub_descendant_weight` with the root entry's weight. This mirrors the logic already present in `update_ancestors_index_key` but executed before links are torn down.
