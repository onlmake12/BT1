### Title
Governance VAA Sequence Check Uses `>` Instead of `== last+1`, Allowing Any Relayer to Permanently Block Lower-Sequence VAAs — (`target_chains/sui/contracts/sources/governance/governance.move`)

### Summary

The sequence guard in `execute_governance_instruction` uses a strictly-greater-than comparison (`sequence > last_executed`) rather than requiring strict consecutive ordering (`sequence == last_executed + 1`). Because Wormhole VAAs are publicly available once guardian-signed, any unprivileged relayer can submit a higher-sequence VAA before a lower-sequence one, advancing `last_executed_governance_sequence` past the skipped VAA and permanently preventing its execution.

---

### Finding Description

In `governance::execute_governance_instruction`, the replay/ordering guard is: [1](#0-0) 

```move
assert!(sequence > state::get_last_executed_governance_sequence(pyth_state),
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);

state::set_last_executed_governance_sequence(&latest_only, pyth_state, sequence);
```

The check accepts **any** sequence number strictly greater than the last executed one — not just `last + 1`. This means:

- `last_executed = 98`
- Submit seq=100 → `100 > 98` passes → `last_executed` becomes 100
- Submit seq=99 → `99 > 100` **fails** → `E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER` → permanently unexecutable

The underlying state field and setter confirm there is no additional ordering enforcement: [2](#0-1) 

The `verify_vaa` function only checks that the VAA originates from the registered governance emitter; it imposes no ordering constraint and returns a plain `WormholeVAAVerificationReceipt` that any caller can then pass to `execute_governance_instruction`: [3](#0-2) 

Because Wormhole VAAs are publicly broadcast once signed by the guardian set, any party — not just the Pyth team — can relay them in arbitrary order.

---

### Impact Explanation

A malicious relayer who observes two pending governance VAAs (both targeting Sui, i.e., `target_chain_id == SUI_CHAIN_ID` or `0`) can submit the higher-sequence one first. The skipped VAA is then permanently unexecutable on-chain. Affected governance actions include fee changes, data-source updates, stale-price-threshold changes, and fee-recipient changes. [4](#0-3) 

**Mitigating factor:** The Pyth governance team can re-issue the blocked action as a new VAA with a higher sequence number. This limits the impact to a temporary delay/disruption rather than a permanent loss of governance capability. However, the specific VAA is irrecoverably blocked, and the attack can be repeated for every new pair of pending VAAs.

---

### Likelihood Explanation

- Wormhole VAAs are publicly available once signed; no key material is needed.
- Any party can call `verify_vaa` + `execute_governance_instruction` on Sui.
- The attacker only needs two valid, guardian-signed governance VAAs targeting Sui to be simultaneously available — a realistic condition during any governance rollout that emits multiple sequential instructions.
- The attack is cheap (just a Sui transaction) and requires no special privileges.

---

### Recommendation

Change the ordering check from `>` to `== last_executed + 1` for VAAs that target the current chain. For VAAs targeting other chains (which must be skippable by design), the `from_byte_vec` / `validate` call already aborts with `E_TARGET_CHAIN_MISMATCH` before the sequence is committed, so those sequences are naturally skipped without advancing `last_executed`. [5](#0-4) 

Alternatively, maintain a set of "consumed sequence numbers" rather than a single high-water mark, so out-of-order execution is allowed but each VAA can only be used once.

---

### Proof of Concept

1. Initialize state with `last_executed_governance_sequence = 98`.
2. Obtain two valid guardian-signed governance VAAs: seq=99 (fee reduction, `target_chain_id=Sui`) and seq=100 (any action, `target_chain_id=Sui`).
3. Call `verify_vaa` + `execute_governance_instruction` with seq=100 first.
   - Assert succeeds; `last_executed_governance_sequence` is now 100.
4. Call `verify_vaa` + `execute_governance_instruction` with seq=99.
   - Assert aborts with `E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER` (error code 2).
5. The fee-reduction governance action is permanently blocked on this deployment.

        pyth_state: &State,
        verified_vaa: VAA,

        let vaa_data_source = pyth::data_source::new((vaa::emitter_chain(&verified_vaa) as u64), vaa::emitter_address(&verified_vaa));

        // The emitter chain and address must correspond to the Pyth governance emitter chain and contract.
        assert!(

            E_INVALID_GOVERNANCE_DATA_SOURCE
        );

        let digest = vaa::digest(&verified_vaa);

        let sequence = vaa::sequence(&verified_vaa);

        let payload = vaa::take_payload(verified_vaa);

        WormholeVAAVerificationReceipt { payload, digest, sequence }
    }
