### Title
Missing Per-Peer Rate Limit on `GetBlockFilters` Enables Unbounded RocksDB Read Flood â€” (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

### Summary
The `BlockFilter` P2P protocol handler processes `GetBlockFilters` messages without any rate limiting. Each message causes up to 1000 RocksDB reads. An unprivileged remote peer (or a set of peers up to the inbound connection limit) can flood the node with these messages, saturating the RocksDB read thread pool and causing the node to become unresponsive to all other protocol traffic.

### Finding Description
`CKBProtocolHandler::received` for `BlockFilter` (`sync/src/filter/mod.rs:122â€“160`) parses the message and immediately calls `self.process(...)` with no rate check. `GetBlockFiltersProcess::execute` (`sync/src/filter/get_block_filters_process.rs:33â€“85`) then iterates up to `BATCH_SIZE=1000` times, issuing `active_chain.get_block_hash(block_number)` and `active_chain.get_block_filter(&block_hash)` on each iteration â€” both are synchronous RocksDB reads. The only bound is the 1.8 MB response size cap, which limits the *response* but not the *read work* per request.

Compare this to `Relayer::try_process` (`sync/src/relayer/mod.rs:116â€“123`), which checks `self.rate_limiter.check_key(&(peer, message.item_id()))` before any processing, and `HolePunching::received` (`network/src/protocols/hole_punching/mod.rs:95â€“107`), which does the same. `BlockFilter` has no `rate_limiter` field and no equivalent check.

### Impact Explanation
With up to ~117 inbound connections (default `max_peers=125`, `max_outbound_peers=8`), each peer sending `GetBlockFilters{start_number:0}` at maximum TCP throughput triggers up to 117,000 RocksDB reads per message-round. This saturates the RocksDB read thread pool, starving all other protocols (Sync, Relay, etc.) of DB access, causing the node to stop processing blocks and transactions â€” effectively a crash-equivalent for consensus participation.

### Likelihood Explanation
The Filter protocol is enabled by default in `support_protocols` in `ckb.toml`. Any unprivileged peer that can open a TCP connection to port 8115 can exploit this. No PoW, no key, no special role required. The attack is trivially scriptable with a standard CKB P2P client library. [6](#0-5) 

### Recommendation
Add a `governor`-based per-peer rate limiter to `BlockFilter`, mirroring the pattern already used in `Relayer` and `HolePunching`:

```rust
// In BlockFilter struct:
rate_limiter: RateLimiter<(PeerIndex, u32)>,

// In received(), before calling process():
if self.rate_limiter.check_key(&(peer_index, msg.item_id())).is_err() {
    return;
}
```

A quota of ~10 req/sec per peer per message type is sufficient for legitimate light-client use and eliminates the amplification vector.

### Proof of Concept
1. Start a CKB full node with block filter data built (mine â‰Ą1 block, wait for filter build).
2. Open N TCP connections to port 8115, each performing the Filter protocol handshake.
3. On each connection, send `BlockFilterMessage { GetBlockFilters { start_number: 0 } }` in a tight loop.
4. Observe: RocksDB read thread pool saturates (visible via metrics or `perf`), node stops responding to `GetHeaders`/`SendBlock` on the Sync protocol within seconds.
