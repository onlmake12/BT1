### Title
Pending Compact Block Map Not Cleaned Up on Peer Disconnect Causing Unbounded Memory Accumulation — (`sync/src/types/mod.rs`)

---

### Summary

When a compact block relay requires missing transactions, the block is inserted into the shared `pending_compact_blocks` map keyed by block hash. If the responsible peer disconnects before supplying the missing transactions, the entry is never removed because `SyncState::disconnected()` does not clean up `pending_compact_blocks`. An unprivileged relay peer can exploit this to cause unbounded memory growth by repeatedly sending compact blocks with missing transactions and disconnecting.

---

### Finding Description

The `SyncState` struct holds a `pending_compact_blocks: tokio::sync::Mutex<PendingCompactBlockMap>` field that stores compact blocks awaiting missing transactions from peers. [1](#0-0) 

When a compact block arrives and reconstruction fails (missing transactions or short-id collision), `missing_or_collided_post_process()` inserts the compact block into this map, recording the peer index and the list of missing transaction indexes: [2](#0-1) 

The map is only cleaned up in two places:
1. When a block is successfully reconstructed and accepted — the specific entry is removed and older-epoch entries are pruned via `retain`: [3](#0-2) 

2. When `BlockTransactionsProcess::execute()` successfully reconstructs the block: [4](#0-3) 

However, `SyncState::disconnected()` — the function called when any peer disconnects — only removes inflight blocks and peer state. It does **not** remove the disconnected peer's entries from `pending_compact_blocks`: [5](#0-4) 

This means that if a peer sends a compact block requiring missing transactions, gets recorded in `pending_compact_blocks`, and then disconnects without ever supplying those transactions, the entry remains in the map indefinitely — until a block of the same or later epoch is accepted by the node.

The `compare_with_pending_compact()` function, called on every block-fetch decision, acquires the mutex and checks this map: [6](#0-5) 

A stale entry causes a 2-second delay in sync download task creation for that specific block hash. More critically, the compact block data (which can be large) and associated metadata remain allocated in memory for the lifetime of the epoch.

---

### Impact Explanation

An attacker controlling one or more relay peers can:
1. Send many compact blocks (each with a distinct valid block hash from the current epoch) that all require missing transactions.
2. Never respond with `BlockTransactions`.
3. Disconnect and reconnect to repeat.

Each such compact block is inserted into `pending_compact_blocks` and never removed until the epoch advances. Since compact blocks can be large and the map has no size cap (the `shrink_to_fit!(pending_compact_blocks, 20)` call only runs on successful block acceptance), this causes unbounded memory growth proportional to the number of such blocks sent per epoch. The mutex on `pending_compact_blocks` is also held during cleanup, causing contention with the relay and sync paths.

---

### Likelihood Explanation

Any unprivileged peer connected via the Relay protocol can send `CompactBlock` messages. The compact block only needs to pass `non_contextual_check` and `contextual_check` (header verification) before being inserted into `pending_compact_blocks`. A peer can send many such messages before disconnecting. This is reachable with no special privileges and requires only a valid P2P connection.

---

### Recommendation

In `SyncState::disconnected()`, after removing inflight blocks and peer state, also remove all `pending_compact_blocks` entries whose `peers_map` contains only the disconnecting peer (or remove the peer from the `peers_map` of each entry, and remove the entry entirely if the `peers_map` becomes empty):

```rust
pub fn disconnected(&self, pi: PeerIndex) {
    let removed_inflight_blocks_count = self.write_inflight_blocks().remove_by_peer(pi);
    // ... existing logging ...
    self.peers().disconnected(pi);

    // NEW: clean up pending_compact_blocks for the disconnected peer
    let mut pending = self.pending_compact_blocks.blocking_lock();
    pending.retain(|_, (_, peers_map, _)| {
        peers_map.remove(&pi);
        !peers_map.is_empty()
    });
}
```

Additionally, consider adding a periodic sweep (e.g., in the `TIMEOUT_EVICTION_TOKEN` notify handler) to evict entries older than a configurable threshold, as a defense-in-depth measure.

---

### Proof of Concept

1. Connect to a CKB node as a relay peer.
2. Construct a valid compact block header (passes `contextual_check`) with one or more short IDs that are not in the node's tx-pool, so reconstruction fails with `ReconstructionResult::Missing`.
3. Send the `CompactBlock` message. The node calls `missing_or_collided_post_process()`, inserting the entry into `pending_compact_blocks` and sending a `GetBlockTransactions` request back.
4. Disconnect without sending `BlockTransactions`.
5. Repeat steps 1–4 with different compact block hashes (e.g., varying the nonce).
6. Observe that `pending_compact_blocks` grows with each iteration and is not cleaned up on disconnect, as `SyncState::disconnected()` does not touch the map. [5](#0-4) [7](#0-6)

/// request missing txs and uncles from peer
async fn missing_or_collided_post_process(
    compact_block: CompactBlock,
    block_hash: Byte32,
    shared: &SyncShared,
    nc: Arc<dyn CKBProtocolContext + Sync>,
    missing_transactions: Vec<u32>,
    missing_uncles: Vec<u32>,
    peer: PeerIndex,
) {
    shared
        .state()
        .pending_compact_blocks()
        .await
        .entry(block_hash.clone())
        .or_insert_with(|| (compact_block, HashMap::default(), unix_time_as_millis()))
        .1
        .insert(peer, (missing_transactions.clone(), missing_uncles.clone()));

    let content = packed::GetBlockTransactions::new_builder()
        .block_hash(block_hash)
        .indexes(missing_transactions.as_slice())
        .uncle_indexes(missing_uncles.as_slice())
        .build();
    let message = packed::RelayMessage::new_builder().set(content).build();
    shared.shared().async_handle().spawn(async move {
        let sending = async_send_message_to(&nc, peer, &message).await;
        if !sending.is_ok() {
            ckb_logger::warn_target!(
                crate::LOG_TARGET_RELAY,
                "ignore the sending message error, error: {}",
                sending
            );
        }
    });
}
