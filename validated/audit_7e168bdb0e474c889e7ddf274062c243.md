### Title
Missing Per-Peer Rate Limiting on `GetTransactionsProof` Enables CPU/I/O Amplification DoS — (`util/light-client-protocol-server/src/lib.rs`, `src/components/get_transactions_proof.rs`)

---

### Summary

The `LightClientProtocol` handler has no per-peer rate limiter. An unprivileged peer can flood the server with `GetTransactionsProof` messages each carrying `GET_TRANSACTIONS_PROOF_LIMIT = 1000` tx_hashes spread across 1000 distinct blocks, triggering unbounded CPU and I/O work per message with no throttle.

---

### Finding Description

`LightClientProtocol` is defined with only a single `shared: Shared` field and no rate-limiting state: [1](#0-0) 

Its `received` handler dispatches directly to `try_process` with no rate check: [2](#0-1) 

Inside `GetTransactionsProofProcess::execute`, the only guard is a count check against `GET_TRANSACTIONS_PROOF_LIMIT = 1000`: [3](#0-2) [4](#0-3) 

For each unique block among the found transactions, the handler performs:
- `snapshot.get_block(&block_hash)` — full block I/O read
- `CBMT::build_merkle_proof(...)` — CPU-intensive Merkle tree computation over all block transactions
- `snapshot.get_block_uncles(&block_hash)` — additional I/O
- `snapshot.get_block_extension(&block_hash)` — additional I/O [5](#0-4) 

Then `reply_proof` calls `mmr.gen_proof(items_positions)` with up to 1000 positions: [6](#0-5) 

**Contrast with other protocols that do have rate limiters:**

- `Relayer` has a `rate_limiter` field and checks it before every message dispatch: [7](#0-6) 
- `HolePunching` has both a `rate_limiter` and a `forward_rate_limiter`, checked before dispatch: [8](#0-7) 

`LightClientProtocol` has neither.

---

### Impact Explanation

A single malicious peer, with no special privileges, can send a continuous stream of `GetTransactionsProof` messages. Each message with 1000 tx_hashes from 1000 distinct blocks forces the server to perform:
- ~3000 RocksDB reads (block body, uncles, extension per block)
- 1000 CBMT Merkle proof constructions (iterating all transactions in each block)
- 1 MMR proof generation over 1000 leaf positions

This is a high amplification ratio: a small, cheap network message triggers a large, expensive server-side computation. Repeated at high frequency, this degrades node responsiveness for all peers and can starve the async runtime.

---

### Likelihood Explanation

The attack requires only a valid P2P connection to a node running the light client server. The attacker needs to know valid tx_hashes from distinct committed blocks, which are all publicly observable on-chain. No PoW, no key, no privileged role is required.

---

### Recommendation

Add a per-peer, per-message-type rate limiter to `LightClientProtocol`, following the same pattern used in `Relayer` (`sync/src/relayer/mod.rs`) and `HolePunching` (`network/src/protocols/hole_punching/mod.rs`). The limiter should be checked in `received` before dispatching to `try_process`, keyed on `(peer_index, message_type_id)`.

Additionally, consider bounding the worst-case work per request by capping the number of distinct blocks that can appear across all tx_hashes in a single `GetTransactionsProof` message (e.g., max 10–20 distinct blocks), independent of the total tx_hash count.

---

### Proof of Concept

1. Connect to a CKB full node running the light client server protocol.
2. Collect 1000 committed tx_hashes, each from a different block (trivially obtained from any block explorer or by syncing the chain).
3. Construct a `GetTransactionsProof` message with all 1000 hashes and a valid `last_hash`.
4. Send the message in a tight loop from a single peer connection.
5. Observe: each message triggers ~3000 storage reads + 1000 CBMT proof builds + 1 MMR proof over 1000 positions on the server, with no throttle applied.
6. Benchmark: compare server CPU/latency for (a) 1000 hashes from 1 block vs (b) 1000 hashes from 1000 distinct blocks — case (b) is orders of magnitude more expensive, confirming the amplification.

    /// Init relay protocol handle
    ///
    /// This is a runtime relay protocol shared state, and any relay messages will be processed and forwarded by it
    pub fn new(chain: ChainController, shared: Arc<SyncShared>) -> Self {
        // setup a rate limiter keyed by peer and message type that lets through 30 requests per second
        // current max rps is 10 (ASK_FOR_TXS_TOKEN / TX_PROPOSAL_TOKEN), 30 is a flexible hard cap with buffer
        let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
        let rate_limiter = RateLimiter::hashmap(quota);

        Relayer {
            chain,
            shared,
            rate_limiter,
        }
    }

    /// Get shared state
    pub fn shared(&self) -> &Arc<SyncShared> {
        &self.shared
    }

    async fn try_process(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        message: packed::RelayMessageUnionReader<'_>,
    ) -> Status {
        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =
            !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));

        if should_check_rate
            && self
                .rate_limiter
                .check_key(&(peer, message.item_id()))
                .is_err()
        {
            return StatusCode::TooManyRequests.with_context(message.item_name());
        }
