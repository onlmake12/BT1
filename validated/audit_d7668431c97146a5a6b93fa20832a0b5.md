Audit Report

## Title
Tx-Pool Minimum Fee Check Uses Only Serialized Size, Ignoring Cycles Weight — Allows High-Cycles Transactions to Bypass Minimum Fee Rate - (File: `tx-pool/src/util.rs`)

## Summary
`check_tx_fee` enforces the minimum fee rate using only the transaction's serialized byte size, not the true weight `get_transaction_weight(size, cycles)`. After `verify_rtx` returns actual cycles, no second fee-rate check is performed. An attacker can submit a small-size, high-cycles transaction, pay a fee that satisfies only the size-based minimum, and have it accepted into the pool with an effective fee rate far below `min_fee_rate` while consuming significant validator CPU.

## Finding Description
`check_tx_fee` in `tx-pool/src/util.rs` at line 45 computes the minimum fee using only `tx_size`:

```rust
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The code's own comment acknowledges this is intentionally incomplete. The true transaction weight used everywhere else is computed by `get_transaction_weight` in `util/types/src/core/tx_pool.rs`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

This weight is used for `TxEntry::fee_rate()` at `tx-pool/src/component/entry.rs` lines 115–118 and for `AncestorsScoreSortKey` at lines 221–231. [3](#0-2) [4](#0-3) 

The critical gap is in `_process_tx` at `tx-pool/src/process.rs` lines 724–751: after `verify_rtx` returns `verified` with actual cycles at line 734, the code immediately creates `TxEntry::new(rtx, verified.cycles, fee, tx_size)` at line 751 with no second fee-rate check using the actual weight. [5](#0-4) 

The `check_tx_fee` call happens only in `pre_check` (line 289), before script execution. [6](#0-5) 

The same inconsistency exists in `calculate_min_replace_fee` at `tx-pool/src/pool.rs` line 103, which also uses `size as u64` only: [7](#0-6) 

## Impact Explanation
An attacker submits a transaction with small serialized size (~300 bytes) and high cycles (~69,000,000). The fee check passes because `min_fee_rate.fee(300) = 300 shannons`. The actual weight is `max(300, 69,000,000 × 0.000_170_571_4) ≈ 11,769 bytes`, giving an effective fee rate of ~25.5 shannons/KB — roughly 40× below the configured 1,000 shannons/KB minimum. Such transactions pass `check_tx_fee`, trigger expensive script execution in `verify_rtx`, and enter the pool consuming significant validator CPU at a fraction of the intended cost. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
Any unprivileged RPC caller (`send_transaction`) or P2P transaction relayer can exploit this. No special role, key, or majority hashpower is required. The attacker deploys a CKB-VM script that consumes many cycles (e.g., a tight loop) stored as a confirmed cell dep (keeping the transaction's serialized size small), then submits transactions with fees just above `min_fee_rate × tx_size`. The attack is repeatable and low-cost relative to the computational burden imposed on validators.

## Recommendation
After `verify_rtx` returns actual cycles in `_process_tx` (`tx-pool/src/process.rs` line 734), perform a second fee-rate check using the true weight before creating the `TxEntry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Apply the same fix to `calculate_min_replace_fee` in `tx-pool/src/pool.rs` line 103, replacing `self.config.min_rbf_rate.fee(size as u64)` with `self.config.min_rbf_rate.fee(get_transaction_weight(size, cycles))`, where `cycles` is the cycles of the replacing transaction's `TxEntry`.

## Proof of Concept
1. Deploy a CKB-VM script that runs a tight loop consuming ~69,000,000 cycles. Store it in a confirmed cell (cell dep, not inline — keeping the transaction's serialized size small, e.g., ~300 bytes).
2. Construct a transaction spending any live cell, referencing the loop script as a type script, with outputs capacity = inputs capacity − 300 shannons (fee = 300 shannons).
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` at `tx-pool/src/util.rs:45` computes `min_fee = min_fee_rate.fee(300) = 300 shannons` → passes.
5. `verify_rtx` executes the script, consuming ~69M cycles.
6. `TxEntry` is created at `tx-pool/src/process.rs:751` with `cycles ≈ 69,000,000`, giving actual weight `max(300, 69,000,000 × 0.000_170_571_4) ≈ 11,769 bytes` and effective fee rate `300 / 11,769 × 1,000 ≈ 25.5 shannons/KB` — far below the 1,000 shannons/KB minimum.
7. The transaction is accepted into the pool. Repeat to flood the pool with computationally expensive, underpaying transactions.
