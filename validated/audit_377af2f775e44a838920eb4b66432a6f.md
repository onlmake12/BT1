Looking at the external report's vulnerability class — **fee/refund accounting** where a protocol fee is charged to users but not properly routed to the protocol's own accumulator — I need to find an analog in the Pyth codebase.

After searching the codebase, I found a direct structural analog in the `Echo.sol` contract.

---

### Title
Echo Protocol Fee (`pythFeeInWei`) Charged to Users but Never Credited to Protocol Accumulator in `executeCallback` — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol` charges users a protocol fee (`_state.pythFeeInWei`) via `getFee()`, but `executeCallback()` credits the entire net amount to the provider's `accruedFeesInWei` rather than splitting out the protocol's share into `_state.accruedFeesInWei`. The protocol fee is silently redirected to the provider on every fulfilled request, and `withdrawFees()` can never withdraw meaningful protocol revenue.

---

### Finding Description

`getFee()` computes the total fee a user must pay:

```solidity
uint96 baseFee = _state.pythFeeInWei; // Fixed fee to Pyth
...
feeAmount = baseFee + providerBaseFee + providerFeedFee + SafeCast.toUint96(gasFee);
``` [1](#0-0) 

`_state.pythFeeInWei` is the Echo protocol's own fee (distinct from the external Pyth oracle fee). The user pays it as part of `req.fee` at request time.

In `executeCallback()`, the fee distribution is:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
...
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

Here `pythFee` is the fee paid to the **external** Pyth oracle contract (`pyth.getUpdateFee(updateData)`), not `_state.pythFeeInWei`. The entire remainder — including the Echo protocol's own `pythFeeInWei` — is credited to the provider. `_state.accruedFeesInWei` is **never incremented** anywhere in the callback path.

The state layout confirms two separate accumulators exist:

```solidity
uint96 pythFeeInWei;        // Echo protocol fee charged to users
uint128 accruedFeesInWei;   // Echo protocol fee accumulator (never filled)
``` [3](#0-2) 

`withdrawFees()` drains only `_state.accruedFeesInWei`:

```solidity
function withdrawFees(uint128 amount) external override {
    require(msg.sender == _state.admin, "Only admin can withdraw fees");
    require(_state.accruedFeesInWei >= amount, "Insufficient balance");
    _state.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [4](#0-3) 

Because `_state.accruedFeesInWei` is never incremented, `withdrawFees()` is permanently blocked from collecting the protocol's share.

---

### Impact Explanation

Every fulfilled Echo request silently transfers the Echo protocol fee (`_state.pythFeeInWei`) to the fulfilling provider instead of the protocol. The Pyth admin can never recover these fees via `withdrawFees()`. The magnitude scales with `pythFeeInWei × number_of_fulfilled_requests`. This is a direct, permanent financial loss for the Pyth protocol with no recovery path.

---

### Likelihood Explanation

This triggers on **every** `executeCallback()` invocation. Any registered provider fulfilling any request causes the protocol fee to be misrouted. No special attacker setup is required — normal protocol operation continuously drains the protocol's fee revenue to providers.

---

### Recommendation

In `executeCallback()`, explicitly credit the Echo protocol's own fee to `_state.accruedFeesInWei` before crediting the provider:

```solidity
uint128 protocolFee = SafeCast.toUint128(_state.pythFeeInWei);
_state.accruedFeesInWei += protocolFee;
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee) - protocolFee;
```

This mirrors the correct split already implemented in `Entropy.sol`'s `requestHelper()`:

```solidity
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [5](#0-4) 

---

### Proof of Concept

1. Admin sets `_state.pythFeeInWei = 0.01 ether` via governance.
2. User calls `request(provider, gasLimit, priceIds)` with `msg.value = getFee(...)` = `0.01 ether + providerFees`. The fee is stored in `req.fee`.
3. Provider calls `executeCallback(sequenceNumber, updateData, priceIds, providerToCredit)`.
4. Inside `executeCallback()`: `pythFee = pyth.getUpdateFee(updateData)` (external oracle fee, e.g. `0.001 ether`).
5. `_state.providers[providerToCredit].accruedFeesInWei += (req.fee + msg.value) - 0.001 ether` — the provider receives the full `0.01 ether` protocol fee.
6. `_state.accruedFeesInWei` remains `0`.
7. Admin calls `withdrawFees(0.01 ether)` → reverts with `"Insufficient balance"`.
8. Repeat for every request: the protocol fee is permanently lost to providers.
