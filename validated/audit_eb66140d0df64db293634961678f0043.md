### Title
Missing `hash_type` Validation for Output Type Scripts in `ScriptHashTypeVerifier` — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces that each transaction output's script `hash_type` is within the consensus-permitted set (`ENABLED_SCRIPT_HASH_TYPE`), but it only inspects the **lock script** of each output. The **type script** of each output is never checked. An unprivileged transaction sender can submit a transaction whose output cell carries a type script with a disallowed `hash_type` (e.g., `Data3 = 6`), bypassing the non-contextual verification gate entirely.

---

### Finding Description

`ScriptHashTypeVerifier::verify()` iterates over every output in a transaction and validates the lock script's `hash_type` against `ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}`:

```rust
// verification/src/transaction_verifier.rs  lines 796–814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())
}
```

There is no corresponding call to `output.type_()` anywhere in this function. A cell output's type script is optional (`ScriptOpt`), but when present, its `hash_type` field is never validated here.

The `ENABLED_SCRIPT_HASH_TYPE` constant is: [1](#0-0) 

The verifier only checks the lock script: [2](#0-1) 

The `NonContextualTransactionVerifier` doc comment itself acknowledges the gap — it says "Check whether output **lock** hash type within enabled range", with no mention of type scripts: [3](#0-2) 

The error variants `InvalidScriptHashType` and `ScriptHashTypeNotPermitted` are both described as "lock Script hash_type" errors, confirming the intent was lock-only: [4](#0-3) 

The existing tests confirm only lock-script paths are exercised — `test_unknown_hash_type_output_lock` and `test_not_enabled_hash_type_output_lock` — with no analogous tests for type scripts: [5](#0-4) 

The lower-level `check_data` path (`CellOutputReader::check_data`) does check both lock and type scripts, but it only validates structural validity via `ScriptHashType::verify_value()` (even number or 1), not membership in the **enabled** set: [6](#0-5) [7](#0-6) 

This means a type script with `hash_type = Data3 (6)` passes `check_data` (6 is even) and also passes `ScriptHashTypeVerifier` (which never reads the type script), clearing the entire non-contextual gate.

---

### Impact Explanation

A transaction with an output cell whose type script carries a disallowed `hash_type` (e.g., `Data3 = 6`, `Data4 = 8`, …, `Data127 = 254`) passes `NonContextualTransactionVerifier` and is admitted to the tx-pool. The node then performs expensive contextual verification (script execution via `select_version()`) before the transaction is finally rejected: [8](#0-7) 

Additionally, `is_malformed_tx()` marks `InvalidScriptHashType` and `ScriptHashTypeNotPermitted` as malformed — a classification used for peer banning and miner penalization. Because the type-script path is never caught by the non-contextual verifier, a block containing such a transaction would fail contextual verification without triggering the `is_malformed_tx` path, creating an inconsistency in how the node attributes blame: [9](#0-8) 

---

### Likelihood Explanation

Any unprivileged RPC caller or P2P transaction relayer can craft such a transaction with zero special privilege. The `hash_type` field is a single byte in the serialized `Script` molecule struct, trivially set to any even value ≥ 6. No key material, mining power, or social engineering is required.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` when the type script is present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }

        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }).into());
            }
        }
    }
    Ok(())
}
```

Update the `NonContextualTransactionVerifier` doc comment and add corresponding tests for type-script hash_type rejection.

### Proof of Concept

```rust
// Craft a transaction whose output has a type script with hash_type = Data3 (6),
// which is NOT in ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}.
let transaction = TransactionBuilder::default()
    .output(
        CellOutput::new_builder()
            .lock(Script::default()) // valid lock: Data (0)
            .type_(Some(
                Script::default()
                    .as_builder()
                    .hash_type(ScriptHashType::Data3) // disallowed: 6
                    .build()
            ).pack())
            .build(),
    )
    .build();

let verifier = ScriptHashTypeVerifier::new(&transaction);

// BUG: returns Ok(()) instead of Err(ScriptHashTypeNotPermitted)
assert!(verifier.verify().is_ok());
```

The transaction passes `NonContextualTransactionVerifier` and enters the tx-pool. It is only rejected later during contextual script execution, after the node has already expended resources resolving and scheduling it.
