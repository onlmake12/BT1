### Title
Unbounded Force-Refresh via `get_overview(EVERYTHING)` Enables RPC-Triggered Resource Exhaustion — (`File: rpc/src/module/terminal.rs`)

### Summary
The `get_overview` RPC endpoint accepts a caller-controlled `refresh` bitmask. Passing `refresh=31` (`EVERYTHING`) unconditionally clears all caches and re-executes every expensive sub-operation on each call with no rate limiting or minimum interval enforced. An unprivileged RPC caller can loop this call to exhaust CPU, memory, and I/O on the node.

### Finding Description
`get_overview` in `rpc/src/module/terminal.rs` accepts an optional `u32` refresh bitmask. When the caller passes `refresh=31` (`RefreshKind::EVERYTHING`), the handler immediately calls `self.cache.clear_all()` and then unconditionally re-fetches all five data categories: [1](#0-0) 

Each forced refresh triggers expensive operations:

1. **`get_sys_info`** — calls `System::new_all()`, `sys.refresh_all()`, `SysDisks::new_with_refreshed_list()`, and `SysNetworks::new_with_refreshed_list()`. These are blocking OS-level syscalls that enumerate all processes, disks, and network interfaces. [2](#0-1) 

2. **`get_tx_pool_info`** — calls `self.shared.get_block_template(None, None, None)`, which assembles a full block template from the live tx pool. This is the most expensive single operation in the CKB node: it acquires tx-pool locks, selects and orders transactions, and runs proposal/commit logic. [3](#0-2) 

3. **`get_cells_info`** — calls `estimate_num_keys_cf(COLUMN_CELL)`, a RocksDB key-count estimation scan over the cell column family. [4](#0-3) 

The TTL constants that are supposed to throttle these operations are completely bypassed when the force-refresh flag is set: [5](#0-4) 

There is no per-caller rate limit, no minimum interval between forced refreshes, and no cap on how many times `EVERYTHING` can be requested per second.

### Impact Explanation
An unprivileged RPC caller can issue `get_overview(31)` in a tight loop. Each call forces:
- A full block template assembly (holding tx-pool write locks)
- Full OS process/disk/network enumeration
- A RocksDB key-count scan

This saturates CPU, starves the tx-pool of its locks (blocking transaction submission and block assembly for miners), and can cause the node to become unresponsive. The block assembler and sync subsystems share the same async runtime, so lock contention from repeated `get_block_template` calls degrades the entire node.

### Likelihood Explanation
The RPC port is accessible to any process that can reach it (localhost by default, but commonly exposed for monitoring tools). The `get_overview` endpoint is explicitly designed for TUI dashboards that poll frequently. A malicious or compromised monitoring client, or any process with RPC access, can trigger this with a single line of code. No special privileges, keys, or network position are required.

### Recommendation
Enforce a server-side minimum interval between force-refresh operations. Specifically:

1. Track the last forced-refresh timestamp per `RefreshKind` flag inside `TerminalCache`.
2. Reject or silently downgrade `EVERYTHING` requests that arrive faster than a configurable minimum interval (e.g., matching the shortest TTL: `ttl::TX_POOL_INFO = 2s`).
3. Alternatively, remove `get_block_template` from the `get_tx_pool_info` path inside `get_overview` and use the cheaper `get_tx_pool_info` RPC data only, since block template assembly is not needed for a monitoring overview. [6](#0-5) 

### Proof of Concept
```bash
# Hammer the node with forced full refreshes; no authentication required
while true; do
  curl -s -X POST http://127.0.0.1:8114/ \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","method":"get_overview","params":[31],"id":1}' \
    > /dev/null
done
```

Each iteration forces `System::new_all()` + `get_block_template` + `estimate_num_keys_cf` with no server-side throttle. The `get_block_template` call acquires the tx-pool lock, so sustained calls will starve miners and transaction submitters of that lock, degrading block production and transaction relay.
