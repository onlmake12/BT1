### Title
Integer Division Truncation in `FeeRate::fee()` Allows Transactions to Bypass Minimum Fee Rate Enforcement - (`util/types/src/core/fee_rate.rs`)

---

### Summary

`FeeRate::fee()` uses integer (floor) division to compute the minimum fee threshold for tx-pool admission. A transaction sender can craft a transaction whose size causes the truncated `min_fee` to be strictly less than the true mathematical minimum, allowing the transaction to enter the pool with an effective fee rate below `min_fee_rate`. The effect is repeatable across every submitted transaction.

---

### Finding Description

`FeeRate::fee()` in `util/types/src/core/fee_rate.rs` computes the minimum fee for a given weight as:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
``` [1](#0-0) 

The division by `KW = 1000` truncates toward zero, so `min_fee = ⌊rate × weight / 1000⌋`.

This result is used directly in `check_tx_fee` in `tx-pool/src/util.rs`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [2](#0-1) 

Because `min_fee` is floored, a transaction paying exactly `min_fee` shannons passes the check even when its true fee rate is:

```
actual_rate = ⌊rate × tx_size / 1000⌋ × 1000 / tx_size  <  rate
```

whenever `rate × tx_size` is not divisible by 1000.

The code comment itself acknowledges the approximation:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check" [3](#0-2) 

The default `min_fee_rate` is 1000 shannons/KW: [4](#0-3) 

---

### Impact Explanation

A transaction sender can choose `tx_size` such that `min_fee_rate × tx_size mod 1000 = 999`, maximising the truncation gap. The admitted transaction's effective fee rate is up to `(KW − 1) = 999` shannons/KW below `min_fee_rate`.

**Concrete example** (`min_fee_rate = 1000`, `tx_size = 1001`):
- `min_fee = ⌊1000 × 1001 / 1000⌋ = 1001` shannons
- Transaction with fee = 1001 shannons is admitted
- Actual fee rate = `1001 × 1000 / 1001 = 999` shannons/KW — **below the configured minimum**

The underpayment per transaction is bounded at 999 shannons (≈ 0.00001 CKB). Across many transactions submitted by the same sender, the aggregate underpayment grows linearly. Unlike the bonding curve case, the error does not compound (each admission check is independent), but the bypass is systematic and reproducible for every transaction whose size is not a multiple of 1000.

The secondary effect is that miners receive slightly less fee income than the protocol's minimum-fee policy intends, and the tx-pool's fee-rate ordering is slightly distorted for affected transactions.

---

### Likelihood Explanation

Any unprivileged transaction sender reachable via the `send_transaction` RPC or P2P relay can trigger this. No special privilege is required. The attacker only needs to choose a transaction serialized size that is not a multiple of 1000 bytes — the vast majority of real transactions satisfy this condition naturally. The bypass is therefore available to essentially every transaction submitted to the network.

---

### Recommendation

Replace floor division with ceiling division in `FeeRate::fee()` so that the computed minimum fee is never less than the true mathematical minimum:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    // Ceiling division: (rate * weight + KW - 1) / KW
    let fee = (self.0.saturating_mul(weight).saturating_add(KW - 1)) / KW;
    Capacity::shannons(fee)
}
```

This ensures `min_fee ≥ rate × weight / 1000` for all inputs, so no transaction can be admitted with an effective fee rate below `min_fee_rate`.

The same fix should be applied consistently wherever `FeeRate::fee()` is used to derive a minimum threshold (including the `min_rbf_rate` path).

---

### Proof of Concept

**Setup**: Node configured with default `min_fee_rate = 1000` shannons/KW.

**Step 1**: Craft a transaction whose serialized size in block is `tx_size = 1001` bytes.

**Step 2**: Set the transaction fee to exactly `1001` shannons (= `⌊1000 × 1001 / 1000⌋`).

**Step 3**: Submit via `send_transaction` RPC.

**Result**: `check_tx_fee` computes `min_fee = 1001`, observes `fee (1001) >= min_fee (1001)`, and admits the transaction. The actual fee rate is `1001 × 1000 / 1001 = 999` shannons/KW — 1 shannons/KW below the configured minimum.

**Repeat**: Submit N such transactions. Each underpays by 1 shannon. Total underpayment = N shannons. For `tx_size = 1000999` bytes (maximising truncation), each transaction underpays by 999 shannons. [1](#0-0) [5](#0-4)
