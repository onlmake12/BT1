### Title
No Maximum Fee Guard in Entropy `requestV2`/`request` — Excess `msg.value` Permanently Captured by Protocol Treasury - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract's `requestHelper` function consumes the **entire `msg.value`** sent by the caller — any amount above the required fee is silently credited to `_state.accruedPythFeesInWei` (the Pyth protocol treasury) and never refunded. Because the provider fee can change at any time (and the Fortuna keeper adjusts it automatically based on gas prices), users who defensively overpay to avoid reverts permanently lose the excess ETH. No `maxFee` parameter exists on any `requestV2`, `request`, or `requestWithCallback` variant to let callers bound their exposure.

---

### Finding Description

In `requestHelper`, fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value` is split between the provider and the Pyth treasury. There is no refund path. The NatSpec on every public entry point explicitly acknowledges this:

> "Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2()` prior to each invocation… Further note that excess value is *not* refunded to the caller." [2](#0-1) 

The fee is composed of a Pyth flat fee plus a provider fee that scales with `callbackGasLimit`:

```solidity
function getFeeV2(address provider, uint32 gasLimit) public view override returns (uint128 feeAmount) {
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
``` [3](#0-2) 

The provider fee is mutable — any registered provider (or its designated fee manager) can call `setProviderFee` / `setProviderFeeAsFeeManager` at any time: [4](#0-3) 

The Fortuna keeper (`adjust_fee_if_necessary`) does this automatically and continuously, targeting a fee proportional to the current gas price: [5](#0-4) 

**Attack / loss scenario (no malicious actor required):**

1. Integrating contract calls `getFeeV2()` off-chain (or in a prior block) → gets `F`.
2. Keeper adjusts fee upward between that read and the user's transaction landing.
3. If the contract sent `msg.value = F` exactly → tx reverts with `InsufficientFee` (DoS).
4. To avoid this, integrators commonly add a buffer: `msg.value = F * 2` or `F + slack`.
5. The entire buffer above `providerFee` flows into `_state.accruedPythFeesInWei` — permanently lost to the user.

**Malicious provider sandwich (requires provider collusion):**

1. Provider frontruns user's `requestV2` call by calling `setProviderFee(highFee)`.
2. User's tx succeeds but pays the inflated fee; excess above `providerFee` goes to the Pyth treasury (provider does not directly pocket it, but the provider's `accruedFeesInWei` is credited the full `providerFee` at the new rate).
3. Provider backruns with `setProviderFee(originalFee)` to restore normal operation.

The same pattern exists in `Echo.sol`'s `requestPriceUpdatesWithCallback`, where excess above `pythFeeInWei` is stored in `req.fee` and later credited entirely to the fulfilling provider: [6](#0-5) 

---

### Impact Explanation

Any caller that sends `msg.value > requiredFee` — a necessary defensive pattern given that fees change continuously — permanently loses the excess ETH to the protocol treasury. For integrating contracts that batch-request randomness or price updates, the cumulative loss can be significant. There is no mechanism to recover overpaid funds.

---

### Likelihood Explanation

The Fortuna keeper is designed to update provider fees frequently in response to gas price changes. [7](#0-6) 
The IEntropyV2 NatSpec itself warns callers that "the fee can change over time," confirming this is an expected, frequent occurrence. Any integrating contract that adds a fee buffer (the only way to avoid reverts) will silently lose ETH on every request where the fee has decreased since the buffer was computed.

---

### Recommendation

Add a `maxFee` parameter to `requestV2`, `request`, and `requestWithCallback`:

```solidity
function requestV2(uint32 gasLimit, uint128 maxFee) external payable returns (uint64);
```

Inside `requestHelper`, revert if `requiredFee > maxFee`, and **refund** `msg.value - requiredFee` to `msg.sender` after crediting the provider and protocol:

```solidity
if (requiredFee > maxFee) revert EntropyErrors.FeeTooHigh();
// ... credit provider and pyth fee ...
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool ok,) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

This mirrors the slippage protection pattern recommended in the referenced report and is standard practice in fee-bearing protocols.

### Proof of Concept

```solidity
// Integrating contract that defensively overpays
contract MyGame is IEntropyConsumer {
    IEntropyV2 entropy;

    function requestRandom() external payable {
        // Query fee, add 50% buffer to survive fee increases
        uint256 fee = entropy.getFeeV2();
        uint256 buffer = fee + fee / 2;
        require(msg.value >= buffer, "send more ETH");

        // If fee has DECREASED since the buffer was computed,
        // the excess (up to fee/2) is permanently lost to the Pyth treasury.
        entropy.requestV2{value: buffer}();
        // No refund. No revert. Silent ETH loss.
    }
}
```

The test suite itself demonstrates overpayment is silently accepted and the excess accrues to the protocol: [8](#0-7)
