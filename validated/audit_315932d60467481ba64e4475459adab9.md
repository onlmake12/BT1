### Title
Entropy `revealWithCallback` Old Flow Uses Unsafe Direct Callback Without Revert Protection, Permanently Blocking Request Fulfillment - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`'s `revealWithCallback`, when `req.gasLimit10k == 0` (the legacy flow, active for any provider whose `defaultGasLimit` is 0), the callback to the requester is made via a raw direct call with no revert protection. If the requester's `_entropyCallback` reverts, the entire `revealWithCallback` transaction reverts. Because the old flow has no `CALLBACK_FAILED` state, the request is permanently stuck and can never be fulfilled. This is the direct analog of `_mint()` vs `_safeMint()`: the new flow uses `excessivelySafeCall` (safe), while the old flow uses a bare direct call (unsafe).

---

### Finding Description

`revealWithCallback` contains two execution branches:

**New flow** (`req.gasLimit10k != 0`): uses `excessivelySafeCall` to catch reverts, emits `CallbackFailed`, and transitions the request to `CALLBACK_FAILED` state, enabling recovery.

**Old flow** (`req.gasLimit10k == 0`): clears the request first, then checks `extcodesize`, then calls `_entropyCallback` directly:

```solidity
// Entropy.sol lines 661–681
address callAddress = req.requester;
clearRequest(provider, sequenceNumber);   // cleared before callback

uint len;
assembly { len := extcodesize(callAddress) }
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(  // bare call, no revert guard
        sequenceNumber, provider, randomNumber
    );
}
```

Because `clearRequest` is inside the same transaction as the bare call, a revert in `_entropyCallback` rolls back `clearRequest` too. The request is never cleared, there is no `CALLBACK_FAILED` state in this branch, and no recovery path exists. The provider's keeper will revert on every retry indefinitely.

The old flow is active whenever `providerInfo.defaultGasLimit == 0`:

```solidity
// Entropy.sol lines 268–283
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;   // old flow, no safe-call protection
} else {
    req.gasLimit10k = roundTo10kGas(...);
}
```

Any provider that has not called `setDefaultGasLimit` with a non-zero value leaves all their requests in the old flow.

---

### Impact Explanation

- A user deploys a contract whose `_entropyCallback` always reverts (e.g., missing implementation, upgraded-away logic, or intentional griefing).
- The user calls `requestWithCallback` / `requestV2` against a provider with `defaultGasLimit == 0`, paying the fee. The fee is immediately accrued to the provider (`providerInfo.accruedFeesInWei += providerFee`).
- Every subsequent `revealWithCallback` call by the provider's keeper reverts. The request is permanently stuck.
- The user's fee is permanently lost (no refund mechanism).
- The provider's keeper wastes gas indefinitely with no way to mark the request as failed or skip it.
- Unlike the new flow, there is no `CALLBACK_FAILED` → retry path; the request is frozen forever.

---

### Likelihood Explanation

- Any unprivileged Entropy user can trigger this by deploying a contract with a reverting callback and requesting from any provider with `defaultGasLimit == 0`.
- The attacker's only cost is the request fee; the damage (permanent DoS of that request slot, keeper gas drain) is asymmetric.
- Third-party providers who have not explicitly called `setDefaultGasLimit` are affected by default.
- The Pyth-operated default provider may have a non-zero gas limit, but the vulnerability is reachable through any other registered provider.

---

### Recommendation

1. **Preferred:** Apply `excessivelySafeCall` (or equivalent try/catch) in the old flow branch as well, and introduce a `CALLBACK_FAILED` state transition so stuck requests can be recovered.
2. **Alternatively:** Require all providers to set a non-zero `defaultGasLimit` before accepting requests, eliminating the old flow entirely.
3. **Minimum:** Document that providers with `defaultGasLimit == 0` are exposed to permanent request DoS and encourage migration to the new flow.

---

### Proof of Concept

```solidity
// Attacker contract
contract MaliciousConsumer is IEntropyConsumer {
    IEntropy entropy;
    constructor(address _entropy) { entropy = IEntropy(_entropy); }

    function getEntropy() internal view override returns (address) {
        return address(entropy);
    }

    // Always reverts — callback can never succeed
    function entropyCallback(uint64, address, bytes32) internal override {
        revert("blocked");
    }

    function attack(address provider, bytes32 userRandom) external payable {
        // Provider must have defaultGasLimit == 0 (old flow)
        entropy.requestWithCallback{value: msg.value}(provider, userRandom);
    }
}
```

1. Deploy `MaliciousConsumer` pointing at an Entropy contract where the target provider has `defaultGasLimit == 0`.
2. Call `attack(provider, userRandom)` with sufficient fee.
3. Provider's keeper calls `revealWithCallback(provider, seq, userRandom, providerProof)` → reverts every time.
4. Request is permanently stuck; no `CALLBACK_FAILED` state exists in the old flow to recover it.

// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

abstract contract IEntropyConsumer {
    // This method is called by Entropy to provide the random number to the consumer.
    // It asserts that the msg.sender is the Entropy contract. It is not meant to be
    // override by the consumer.
    function _entropyCallback(
        uint64 sequence,
        address provider,
        bytes32 randomNumber
    ) external {
        address entropy = getEntropy();
        require(entropy != address(0), "Entropy address not set");
        require(msg.sender == entropy, "Only Entropy can call this function");

        entropyCallback(sequence, provider, randomNumber);
    }

    // getEntropy returns Entropy contract address. The method is being used to check that the
    // callback is indeed from Entropy contract. The consumer is expected to implement this method.
    // Entropy address can be found here - https://docs.pyth.network/entropy/contract-addresses
    function getEntropy() internal view virtual returns (address);

    // This method is expected to be implemented by the consumer to handle the random number.
    // It will be called by _entropyCallback after _entropyCallback ensures that the call is
    // indeed from Entropy contract.
    function entropyCallback(
        uint64 sequence,
        address provider,
        bytes32 randomNumber
    ) internal virtual;
}
