### Title
Unbounded `last_n_numbers` allocation in `GetLastStateProofProcess::execute` bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The limit guard in `execute()` only checks `difficulties.len() + last_n_blocks * 2`, but the `last_n_numbers` slice — whose length equals `last_block_number - difficulty_boundary_block_number` — is never bounded. An unprivileged light-client peer can craft a message that forces `difficulty_boundary_block_number` to equal `start_block_number`, making `last_n_numbers` span the entire chain height. `complete_headers` then performs O(chain_height) database reads and allocates a proportionally large `Vec<VerifiableHeader>`, crashing the node with OOM.

---

### Finding Description

**Limit check (lines 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

This only bounds `difficulties.len()` and `last_n_blocks`. It does not bound the length of `last_n_numbers`. [1](#0-0) 

**`last_n_numbers` construction (lines 318–319):**

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
```

Its length is `last_block_number - difficulty_boundary_block_number`, which is unbounded. [2](#0-1) 

**How `difficulty_boundary_block_number` collapses to `start_block_number`:**

`get_first_block_total_difficulty_is_not_less_than` returns `start_block_number` immediately when `start_total_difficulty >= min_total_difficulty` (lines 30–32). Setting `difficulty_boundary = U256::from(1)` guarantees this, since every block's cumulative total difficulty is ≥ 1. [3](#0-2) 

**The adjustment guard (lines 313–316) does not help:**

```rust
if last_block_number - difficulty_boundary_block_number < last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
```

With `last_n_blocks = 0`, the condition `last_block_number - start_block_number < 0` is always false, so no adjustment occurs. [4](#0-3) 

**`complete_headers` performs O(N) work per entry (lines 132–177):**

For each of the N block numbers it calls `snapshot.get_ancestor(last_hash, number)` and `snapshot.chain_root_mmr(number - 1).get_root()`, then pushes a `VerifiableHeader` into a `Vec`. With N = 1,000,000 this exhausts memory. [5](#0-4) 

---

### Impact Explanation

The node process crashes with OOM. Even before OOM, the request monopolizes the async executor for the duration of the loop, stalling all other peer processing. Impact is local node crash (denial of service), matching the stated scope. [6](#0-5) 

---

### Likelihood Explanation

- The attacker is an unprivileged light-client peer; no key, PoW, or operator access is required.
- The message is a standard P2P `GetLastStateProof` message accepted from any connected peer.
- CKB mainnet has been running since 2019 and has well over 1M blocks, satisfying the precondition.
- The crafted values (`last_n_blocks=0`, `difficulties=[]`, `difficulty_boundary=1`) are trivially constructable.
- A single message is sufficient; no repeated requests or amplification needed. [7](#0-6) 

---

### Recommendation

After computing `last_n_numbers`, enforce that its length plus `sampled_numbers.len()` plus `reorg_last_n_numbers.len()` does not exceed `GET_LAST_STATE_PROOF_LIMIT`. The simplest fix is to add a check immediately after line 319:

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();

if last_n_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many last_n blocks");
}
```

A stricter fix bounds the combined total of all three slices before calling `complete_headers`. [8](#0-7) 

---

### Proof of Concept

```
last_hash          = <valid main-chain tip hash at height H (e.g. 1_000_000)>
start_hash         = <valid main-chain hash at height 0>
start_number       = 0
last_n_blocks      = 0
difficulty_boundary = 1   (U256)
difficulties       = []
```

**Trace:**

1. Limit check: `0 + 0*2 = 0 ≤ 1000` → passes.
2. `is_main_chain(last_hash)` → passes.
3. `start_block_number (0) ≤ last_block_number (1_000_000)` → passes.
4. Branch: `1_000_000 - 0 ≤ 0` → **false** → else branch.
5. `get_first_block_total_difficulty_is_not_less_than(0, 1_000_000, &1)`: genesis total_difficulty ≥ 1 → returns `Some((0, …))` → `difficulty_boundary_block_number = 0`.
6. Adjustment: `1_000_000 - 0 < 0` → false → no change.
7. `last_n_numbers = (0..1_000_000)` → **1,000,000 entries**.
8. `difficulty_boundary_block_number > 0` → false → `sampled_numbers = []`.
9. `complete_headers` iterates 1,000,000 entries, calling `get_ancestor` + `chain_root_mmr().get_root()` per entry → OOM crash. [9](#0-8)
