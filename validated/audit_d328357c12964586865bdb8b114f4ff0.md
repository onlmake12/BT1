### Title
Permissionless Deployment of Upgradeable Scripts Enables Silent Code Replacement and Token Theft — (File: `script/src/type_id.rs`)

---

### Summary

The CKB Type ID system allows any unprivileged user to deploy an upgradeable script cell. Once users lock their assets using a script referenced by `hash_type: Type`, the script author can silently replace the underlying bytecode with malicious code in a subsequent transaction. The `TypeIdSystemScript::verify()` function performs **zero validation on updates**, and `extract_script_and_dep_index` resolves scripts by type hash rather than data hash, so users unknowingly execute the replaced malicious bytecode. There is no pause or revocation mechanism for a specific script pair.

---

### Finding Description

**Root cause — `script/src/type_id.rs`, lines 52–75:**

The `TypeIdSystemScript::verify()` function only validates the *creation* branch (when `input_indices.is_empty()`). The update branch (1 input, 1 output with the same type ID) performs **no validation whatsoever** and immediately returns `Ok(TYPE_ID_CYCLES)`:

```rust
if self.script_group.input_indices.is_empty() {
    // creation: validates args == hash(first_input, first_output_index)
    ...
}
Ok(TYPE_ID_CYCLES)   // update: no check on new cell data
```

This means the cell owner can replace the cell's data (the script bytecode) with any arbitrary content, as long as their lock script authorizes the spend. The type script args — and therefore the type hash — remain identical across the update.

**Resolution path — `script/src/types.rs`, lines 843–852:**

When a script uses `hash_type: Type`, `extract_script_and_dep_index` resolves the executable binary by looking up the type hash in `binaries_by_type_hash`:

```rust
ScriptHashType::Type => {
    if let Some(ref bin) = self.binaries_by_type_hash.get(&script.code_hash()) {
        match bin {
            Binaries::Unique(_, dep_index, lazy) => Ok((lazy, dep_index)),
            Binaries::Duplicate(_, dep_index, lazy) => Ok((lazy, dep_index)),
            Binaries::Multiple => Err(ScriptError::MultipleMatches),
        }
    } else {
        Err(ScriptError::ScriptNotFound(script.code_hash()))
    }
}
```

The lookup is keyed on the type hash, **not** the data hash. Any cell dep whose type script hash matches is accepted as the executable binary, regardless of what bytecode it contains.

**Combined attack surface:**

Because the type hash is fixed at creation time (derived from `TYPE_ID_CODE_HASH`, `hash_type: Type`, and the immutable `args`), an attacker can:

1. Deploy a Type ID cell with legitimate bytecode; the type hash is published.
2. Induce users to lock their CKB assets using a lock/type script that references this type hash.
3. Spend the Type ID cell and re-create it with the **same type script** (identical type hash) but **malicious bytecode** — the Type ID verifier accepts this unconditionally.
4. In any subsequent transaction that spends a victim's cell, include the upgraded (malicious) cell as a cell dep. The resolver picks it up by type hash and executes the malicious binary.

There is no on-chain mechanism to pause, revoke, or blacklist a specific type-hash-identified script after it has been adopted by users.

---

### Impact Explanation

- **Direct token theft**: A victim's cell locked by `hash_type: Type` pointing to the attacker-controlled type hash will execute the malicious binary. If the malicious binary always exits with code 0 (unconditional success), any party can spend the victim's cell and redirect the capacity.
- **Permanent and irreversible**: Once the upgrade transaction is confirmed, all cells referencing that type hash are immediately at risk. There is no rollback, no pause, and no per-script emergency stop.
- **Silent**: The type hash visible in the victim's lock/type script does not change. Victims have no on-chain signal that the underlying bytecode has been replaced.

---

### Likelihood Explanation

- Any unprivileged transaction sender can deploy a Type ID cell; no admin key or special permission is required.
- The attack is realistic wherever `hash_type: Type` is used for lock or type scripts in DeFi, NFT, or DAO contracts on CKB — a common pattern for "upgradeable" contracts.
- The attacker controls the timing of the upgrade and can execute it atomically with the theft transaction, leaving no window for victims to react.

---

### Recommendation

- Protocols that use `hash_type: Type` should document the trust assumption explicitly: users are trusting the Type ID cell owner not to upgrade to malicious code.
- Consider a time-lock or multi-sig governance requirement on the lock script of any widely-adopted Type ID cell, so upgrades cannot be performed unilaterally or instantly.
- Consider an opt-in "freeze" flag in the Type ID verification that, once set, prevents further data changes (analogous to bytecode verification in the NFTBridge recommendation).
- Tooling and wallets should warn users when they are locking assets under a `hash_type: Type` script whose Type ID cell is controlled by a single key.

    pub fn verify(&self) -> Result<Cycle, ScriptError> {
        if self.max_cycles < TYPE_ID_CYCLES {
            return Err(ScriptError::ExceededMaximumCycles(self.max_cycles));
        }
        // TYPE_ID script should only accept one argument,
        // which is the hash of all inputs when creating
        // the cell.
        if self.script_group.script.args().len() != 32 {
            return Err(self.validation_failure(ERROR_ARGS));
        }

        // There could be at most one input cell and one
        // output cell with current TYPE_ID script.
        if self.script_group.input_indices.len() > 1 || self.script_group.output_indices.len() > 1 {
            return Err(self.validation_failure(ERROR_TOO_MANY_CELLS));
        }

        // If there's only one output cell with current
        // TYPE_ID script, we are creating such a cell,
        // we also need to validate that the first argument matches
        // the hash of following items concatenated:
        // 1. First CellInput of the transaction.
        // 2. Index of the first output cell in current script group.
        if self.script_group.input_indices.is_empty() {
            let first_cell_input = self
                .rtx
                .transaction
                .inputs()
                .get(0)
                .ok_or_else(|| self.validation_failure(ERROR_ARGS))?;
            let first_output_index: u64 = self
                .script_group
                .output_indices
                .first()
                .map(|output_index| *output_index as u64)
                .ok_or_else(|| self.validation_failure(ERROR_ARGS))?;

            let mut blake2b = new_blake2b();
            blake2b.update(first_cell_input.as_slice());
            blake2b.update(&first_output_index.to_le_bytes());

            blake2b.finalize(&mut ret);
