### Title
Scheduler `_storePriceUpdates` Overwrites Individual Feed Prices Without Per-Feed Timestamp Validation, Allowing Older Prices to Replace Newer Cached Ones — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.updatePriceFeeds` validates freshness using only the **maximum** `publishTime` across all feeds in a batch (`updateTimestamp`), but `_storePriceUpdates` writes every feed unconditionally. An unprivileged keeper can submit a historical Pythnet accumulator blob where one feed's `publishTime` is just barely newer than `status.priceLastUpdatedAt` (to pass the batch-level check) while another feed's `publishTime` is older than the currently stored value, rolling that feed's price back to a stale, attacker-selected value.

---

### Finding Description

`_validateShouldUpdatePrices` computes `updateTimestamp` as the maximum `publishTime` across all feeds:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [1](#0-0) 

It then rejects the batch only if this single aggregate value is not strictly greater than `priceLastUpdatedAt`:

```solidity
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(...);
}
``` [2](#0-1) 

After validation, `_storePriceUpdates` writes every feed in the batch with no per-feed timestamp guard:

```solidity
function _storePriceUpdates(...) internal {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        _state.priceUpdates[subscriptionId][priceFeeds[i].id] = priceFeeds[i];
    }
}
``` [3](#0-2) 

`parsePriceFeedUpdatesWithConfig` is called with `minAllowedPublishTime = 0`, so Pyth imposes no lower bound on individual feed timestamps:

```solidity
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0, // We enforce the past max validity ourselves ...
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
    false, true, false
);
``` [4](#0-3) 

The only additional constraint is the same-slot check, which ensures all feeds come from one Pythnet accumulator slot — but within a single slot, different feeds legitimately carry different `publishTime` values (e.g., closed equity markets retain their last-trading-period timestamp). [5](#0-4) 

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is 1 hour, applied only to the max timestamp: [6](#0-5) 

---

### Impact Explanation

**`getPricesNoOlderThan` is deceived.** The staleness check in the read path uses `status.priceLastUpdatedAt` — the batch-level max timestamp — not individual feed timestamps:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();
prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [7](#0-6) 

After a successful rollback update, `priceLastUpdatedAt` is set to the new max (which is fresh), so `getPricesNoOlderThan` passes — but the returned price for the targeted feed is the attacker-chosen older value. Downstream protocols that rely on `getPricesNoOlderThan` to guarantee per-feed freshness receive a stale price without any revert.

**Concrete impact:** A protocol using the Scheduler for, e.g., BTC/USD + ETH/USD could have its ETH/USD price rolled back to a value from up to ~1 hour ago (bounded by `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`), enabling the attacker to borrow against a stale collateral price or trigger an unjust liquidation.

---

### Likelihood Explanation

- Any address can call `updatePriceFeeds` — no privileged role required.
- Pythnet produces a new accumulator slot roughly every 400 ms. Within any 1-hour window there are ~9,000 slots. The attacker can scan historical Hermes data off-chain to find a slot where Feed A's `publishTime` is just above the current `priceLastUpdatedAt` and Feed B's `publishTime` is older and price-favorable.
- The existing test `testUpdatePriceFeedsSucceedsWithStaleFeedIfLatestIsValid` already demonstrates the contract accepts a batch where one feed's timestamp is over 1 hour 5 minutes old as long as another feed's timestamp is within the validity window — confirming the path is reachable. [8](#0-7) 

---

### Recommendation

Add a per-feed timestamp guard inside `_storePriceUpdates` (or inside `_validateShouldUpdatePrices`) that rejects any individual feed whose `publishTime` is not strictly greater than the currently stored value for that feed:

```solidity
function _storePriceUpdates(uint256 subscriptionId, PythStructs.PriceFeed[] memory priceFeeds) internal {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        PythStructs.PriceFeed storage existing = _state.priceUpdates[subscriptionId][priceFeeds[i].id];
        // Only overwrite if the incoming publishTime is strictly newer
        if (priceFeeds[i].price.publishTime > existing.price.publishTime) {
            _state.priceUpdates[subscriptionId][priceFeeds[i].id] = priceFeeds[i];
        }
    }
}
```

This mirrors the per-feed freshness guard used by the core Pyth contract (`updateLatestPriceIfNecessary`) and by every other chain implementation (Aptos `is_fresh_update`, Sui `is_fresh_update`, CosmWasm `update_price_feed_if_new`, Fuel `publish_time > latest_publish_time`). ### Proof of Concept

**Setup:** Subscription with two price IDs — `FEED_A` (active market) and `FEED_B` (active market). `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hour`.

**State after legitimate Update 1:**
- `FEED_A.publishTime = T + 3600` (current)
- `FEED_B.publishTime = T + 3600`
- `priceLastUpdatedAt = T + 3600`

**Attacker scans Hermes history and finds Pythnet slot S where:**
- `FEED_A.publishTime = T + 3601` (1 second newer — passes max-timestamp check)
- `FEED_B.publishTime = T + 1800` (30 minutes older — favorable price for attacker)
- Both feeds share the same slot (slot check passes)
- `T + 3601` is within 1 hour of `block.timestamp` (validity check passes)

**Attacker calls `updatePriceFeeds(subscriptionId, updateDataForSlotS)`:**
1. `parsePriceFeedUpdatesWithConfig` with `minAllowedPublishTime=0` accepts both feeds.
2. Slot check: both feeds from slot S — passes.
3. `updateTimestamp = max(T+3601, T+1800) = T+3601 > T+3600` — passes.
4. `_storePriceUpdates` writes both feeds unconditionally.
5. `FEED_B` is now stored with `publishTime = T+1800` (rolled back 30 minutes).
6. `priceLastUpdatedAt = T+3601`.

**Victim protocol calls `getPricesNoOlderThan(subscriptionId, [FEED_B], 120)`:**
- `distance(block.timestamp, T+3601) ≤ 120` — staleness check passes.
- Returns `FEED_B` price from `T+1800` — 30 minutes stale — without any revert.
