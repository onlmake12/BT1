Audit Report

## Title
Missing Per-Peer Rate Limiting in `BlockFilter` Enables Head-of-Line Blocking of Filter Protocol Handler — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_check_points_process.rs`)

## Summary
The `BlockFilter` protocol handler contains no per-peer rate limiter, unlike `Relayer` and `HolePunching` which both enforce per-peer quotas. A single unprivileged peer can flood `GetBlockFilterCheckPoints` messages, each triggering up to 4,000 sequential DB reads, monopolizing the sequential async handler and starving legitimate light-client peers of `GetBlockFilterHashes` and `GetBlockFilters` responses.

## Finding Description
**No rate limiter in `BlockFilter`:** The struct at `sync/src/filter/mod.rs` lines 22–25 contains only `shared: Arc<SyncShared>` — no `rate_limiter` field. `try_process` at lines 33–68 performs no quota check before dispatching to any `*Process::execute`.

**Sequential handler model:** `CKBHandler::received` in `network/src/protocols/mod.rs` line 383 calls `self.handler.received(...).await` with `&mut self`, meaning tentacle dispatches all messages from all peers through the same single async context, one at a time. `BlockFilter::received` at `sync/src/filter/mod.rs` line 152 fully awaits `self.process(...)` before returning, so no other peer's message can begin until the current one completes.

**High per-message cost:** `GetBlockFilterCheckPointsProcess::execute` at `sync/src/filter/get_block_filter_check_points_process.rs` lines 43–56 loops up to `BATCH_SIZE = 2000` iterations, each calling `get_block_hash` then `get_block_filter_hash` — up to 4,000 synchronous DB reads per message.

**Contrast with guarded handlers:** `Relayer` at `sync/src/relayer/mod.rs` lines 81, 91–92, 116–123 enforces 30 req/sec per peer per message type via `RateLimiter<(PeerIndex, u32)>`. `HolePunching` at `network/src/protocols/hole_punching/mod.rs` lines 95–107 similarly checks before any processing. `BlockFilter` has neither guard.

**Exploit flow:** Attacker connects on `SupportProtocols::Filter`, sends a tight loop of `GetBlockFilterCheckPoints` messages (e.g., `start_number = 0, 2000, 4000, ...`). Each message occupies the handler for the duration of up to 4,000 DB reads. Legitimate light-client peers sending `GetBlockFilterHashes` or `GetBlockFilters` are queued behind the flood and receive responses with latency proportional to the flood rate.

## Impact Explanation
This is a **High** severity finding: a bad design which could cause CKB network congestion with few costs. The Filter protocol is the sole mechanism by which light clients obtain compact block filter data for transaction monitoring. A single attacker peer with one TCP connection and no special privileges can render the Filter protocol handler unresponsive for all other peers on the targeted node. Since light clients depend entirely on this protocol for chain synchronization, a sustained flood effectively disconnects all light clients from the node at negligible cost to the attacker.

## Likelihood Explanation
The attack requires only a single TCP connection to a CKB node with the Filter protocol enabled. No keys, hashpower, or elevated privileges are needed. The absence of rate limiting is a concrete, verifiable code gap. The attack is locally reproducible with two mock peers and is repeatable indefinitely.

## Recommendation
Add a per-peer, per-message-type `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter`, mirroring the pattern in `Relayer` (`sync/src/relayer/mod.rs` lines 63–67, 81, 91–92). Check the limiter at the top of `try_process` before dispatching to any `*Process::execute`. A quota of 5–10 `GetBlockFilterCheckPoints` requests per second per peer is sufficient for legitimate light-client use while blocking flood attacks.

## Proof of Concept
1. Start a CKB node with the Filter protocol enabled and a populated block filter DB.
2. Connect mock peer A and mock peer B to the node on `SupportProtocols::Filter`.
3. Have peer A send `GetBlockFilterCheckPoints` messages in a tight loop (`start_number = 0, 2000, 4000, ...`).
4. Have peer B send `GetBlockFilterHashes` messages at a steady rate (e.g., 1/sec).
5. Measure peer B's response latency with and without peer A's flood active.
6. Assert that peer B's latency increases proportionally to peer A's flood rate, confirming head-of-line blocking caused by the missing rate limiter at `sync/src/filter/mod.rs` lines 33–68 and the per-message cost at `sync/src/filter/get_block_filter_check_points_process.rs` lines 43–56.
