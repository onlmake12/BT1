### Title
Expired Network Alert Replay via P2P — `clear_expired_alerts` Does Not Populate `cancel_filter` - (File: `util/network-alert/src/notifier.rs`)

---

### Summary

The `Notifier::clear_expired_alerts` function removes expired alerts from `received_alerts` and `noticed_alerts` but never adds their IDs to `cancel_filter`. The P2P relay handler (`AlertRelayer::received`) does not validate `notice_until`. Together, these two gaps allow any unprivileged P2P peer who has stored a legitimately-signed alert to replay it after it expires, causing every receiving node to re-accept, re-display, and re-broadcast the stale alert network-wide — permanently defeating the expiry/revocation mechanism.

This is the direct CKB analog of the external report: the cancel/revocation mechanism (`cancel_filter`) exists but is structurally bypassed for naturally-expired alerts, just as `Timelock.cancelTransaction` existed but was structurally unreachable from the Governor.

---

### Finding Description

**Root cause 1 — `clear_expired_alerts` does not update `cancel_filter`:** [1](#0-0) 

When `clear_expired_alerts` runs it removes expired entries from `received_alerts` and `noticed_alerts`, but it never calls `self.cancel_filter.put(id, ())` for any of the removed IDs.

**Root cause 2 — `has_received` checks only `received_alerts` and `cancel_filter`:** [2](#0-1) 

After `clear_expired_alerts` runs, `has_received(id)` returns `false` for any expired alert whose ID was not explicitly canceled — because it is absent from both maps.

**Root cause 3 — P2P relay path does not validate `notice_until`:** [3](#0-2) 

The `received` handler parses the alert, checks `has_received`, verifies signatures, and calls `notifier.add`. There is no `notice_until < now` guard. Contrast this with the RPC path, which explicitly rejects expired alerts: [4](#0-3) 

**Root cause 4 — `notifier.add` does not check `notice_until` either:** [5](#0-4) 

`add` only checks `has_received` and `is_version_effective` (version range). An expired alert with no version restriction passes both checks and is inserted into `received_alerts` and `noticed_alerts`, then re-broadcast.

**`cancel_filter` capacity is also bounded:** [6](#0-5) 

`CANCEL_FILTER_SIZE = 128`. Even for explicitly-canceled alerts, after 128 subsequent cancel operations the LRU evicts old entries, re-opening the same replay window.

---

### Impact Explanation

An unprivileged P2P peer can:

1. Observe and store any legitimately-signed alert during its active window (it is broadcast to all peers).
2. Wait for the alert to expire and for `clear_expired_alerts` to run (triggered on every new peer connection).
3. Re-send the stored raw bytes via the P2P Alert protocol.
4. The target node re-accepts the alert (`has_received` → false, signatures valid), re-inserts it into `noticed_alerts`, and re-broadcasts it to all connected peers.
5. The re-broadcast propagates network-wide; every node repeats steps 3–4.
6. The attacker repeats step 3 after each `clear_expired_alerts` cycle, keeping the stale alert alive indefinitely.

Concrete consequences:
- Stale critical alerts (e.g., "upgrade immediately") are permanently re-displayed on all nodes, causing operator confusion or panic.
- The replayed alert ID re-enters `received_alerts`, blocking any future legitimate alert with the same ID from being accepted (`has_received` returns true).
- The network sustains unnecessary alert-protocol traffic for the lifetime of the attack.

---

### Likelihood Explanation

- **Entry path is fully unprivileged**: any peer connected during the original alert broadcast has the signed bytes; no key material is needed.
- **Trigger condition is automatic**: `clear_expired_alerts` is called on every `connected` event, so the replay window opens naturally without attacker action.
- **Persistence is trivial**: a single attacker node re-sending one UDP/TCP message per peer-connection event is sufficient to keep the alert alive across the entire network.
- **No rate-limit or ban applies**: the alert passes signature verification and is not treated as a malformed message, so the peer is not banned.

---

### Recommendation

**Short term:** In `clear_expired_alerts`, add every removed alert ID to `cancel_filter` before dropping it from `received_alerts`:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    let expired_ids: Vec<u32> = self.received_alerts
        .iter()
        .filter(|(_, alert)| {
            let notice_until: u64 = alert.raw().notice_until().into();
            notice_until <= now
        })
        .map(|(id, _)| *id)
        .collect();
    for id in expired_ids {
        self.cancel_filter.put(id, ());   // <-- add this
        self.received_alerts.remove(&id);
    }
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

**Short term (defense-in-depth):** Add a `notice_until` check in the P2P relay path (`AlertRelayer::received`) to reject expired alerts before signature verification, consistent with the RPC path.

**Long term:** Replace the bounded `LruCache` for `cancel_filter` with an unbounded `HashSet` (or a time-bounded set keyed on expiry), so that cancel records are never silently evicted.

### Proof of Concept

```
1. Attacker connects to a CKB node as a normal P2P peer (SupportProtocols::Alert).

2. A legitimate signed alert (id=42, notice_until=T, valid 2-of-4 sigs) is broadcast.
   Attacker stores the raw bytes: alert_bytes.

3. Time advances past T. A new peer connects to the victim node, triggering:
     AlertRelayer::connected → clear_expired_alerts()
   This removes id=42 from received_alerts but NOT from cancel_filter.
   has_received(42) now returns false.

4. Attacker sends alert_bytes to the victim node via the Alert P2P protocol.

5. AlertRelayer::received:
   - Parses alert: OK
   - has_received(42) → false  (not in received_alerts, not in cancel_filter)
   - verify_signatures(&alert) → Ok(())  (original valid signatures)
   - notifier.add(&alert):
       - has_received(42) → false
       - received_alerts.insert(42, alert)
       - is_version_effective → true
       - noticed_alerts.push(alert)  ← stale alert re-displayed
   - quick_filter_broadcast → re-sent to all connected peers

6. All peers repeat step 5. Network-wide re-broadcast of the expired alert.

7. Attacker repeats step 4 after each clear_expired_alerts cycle.
   The expired alert remains visible on all nodes indefinitely.

    pub fn add(&mut self, alert: &Alert) {
        let alert_id = alert.raw().id().into();
        let alert_cancel = alert.raw().cancel().into();
        if self.has_received(alert_id) {
            return;
        }
        // checkout cancel_id
        if alert_cancel > 0 {
            self.cancel(alert_cancel);
        }
        // add to received alerts
        self.received_alerts.insert(alert_id, alert.clone());

        // check conditions, figure out do we need to notice this alert
        if !self.is_version_effective(alert) {
            debug!("Received a version ineffective alert {:?}", alert);
            return;
        }

        if self.noticed_alerts.contains(alert) {
            return;
        }
        self.notify_controller.notify_network_alert(alert.clone());
        self.noticed_alerts.push(alert.clone());
        // sort by priority
        self.noticed_alerts.sort_by_key(|a| {
            let priority: u32 = a.raw().priority().into();
            u32::MAX - priority
        });
    }

        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,

                if alert.raw().message().is_utf8()

                        .map(|x| x.is_utf8())

                    alert.to_entity()

                    info!(
                        "A malformed message fromP peer {} : not utf-8 string",
                        peer_index
                    );
                    nc.ban_peer(
                        peer_index,
                        BAD_MESSAGE_BAN_TIME,
                        String::from("send us a malformed message: not utf-8 string"),
                    );
                    return;
                }
            }
            Err(err) => {
                info!("A malformed message from peer {}: {:?}", peer_index, err);
                nc.ban_peer(
                    peer_index,
                    BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };
        let alert_id = alert.as_reader().raw().id().into();
        trace!("ReceiveD alert {} from peer {}", alert_id, peer_index);
        // ignore alert
        if self.notifier.lock().has_received(alert_id) {
            return;
        }

                BAD_MESSAGE_BAN_TIME,
                String::from("send us an alert with invalid signatures"),
            );
            return;

        self.mark_as_known(peer_index, alert_id);
        // broadcast message

            .into_iter()
            .filter(|peer| self.mark_as_known(*peer, alert_id))
            .collect();
        if let Err(err) = nc.quick_filter_broadcast(
            TargetSession::Multi(Box::new(selected_peers.into_iter())),
            data,
        ) {
            debug!("alert broadcast error: {:?}", err);
        }
        // add to received alerts
        self.notifier.lock().add(&alert);
    }
