Audit Report

## Title
Unauthenticated `ConnectionRequestDelivered` Triggers Unbounded Outbound TCP Connection Storm via Spoofed `from` Field — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary

Any directly connected peer can craft a `ConnectionRequestDelivered` message with `from` set to the victim's own peer ID and `to` set to a peer ID observed from the victim's `ConnectionRequest` gossip broadcast. This bypasses every validation gate and causes the victim to spawn up to 24 concurrent TCP connection loops toward attacker-controlled IPs, each looping for 30 seconds. The attack is repeatable every 5 minutes with no authentication cost.

## Finding Description

**Root cause:** The terminal branch of `ConnectionRequestDeliveredProcess::execute()` is reachable by any directly connected peer. The sole guard protecting it is `self_peer_id == content.from`, which the attacker trivially satisfies by setting `from` to the victim's own public peer ID (learned during the identify handshake).

**Exploit path:**

**Step 1 — `inflight_requests` is populated.** Every 5 minutes, `HolePunching::notify()` fires via `CHECK_INTERVAL`. When `status.non_whitelist_outbound < status.max_outbound && status.total > 0`, it calls `fetch_nat_addrs()`, gossip-broadcasts `ConnectionRequest` messages to `sqrt(total)` peers, and inserts each `to_peer_id` into `inflight_requests` with a timestamp. [1](#0-0) [2](#0-1) 

**Step 2 — Attacker learns `to` peer IDs.** The attacker is a connected peer and is among the `sqrt(total)` gossip recipients. The `to` field is read directly from the received `ConnectionRequest`. [3](#0-2) 

**Step 3 — Attacker crafts the message.** The attacker sends a `ConnectionRequestDelivered` with `from` = victim's own peer ID, `to` = observed peer ID, `route` = `[]` (empty), and `listen_addrs` = up to 24 arbitrary TCP addresses pointing to attacker-controlled IPs.

**Step 4 — All validation gates pass.**

| Check | Location | Outcome |
|---|---|---|
| Molecule decode | `mod.rs` L76–90 | Passes |
| `rate_limiter` (30 req/s per session) | `mod.rs` L95–107 | Passes — one message |
| `listen_addrs` count (1–24) | `connection_request_delivered.rs` L125–128 | Passes — attacker sets ≤24 |
| `route.len() <= MAX_HOPS` | L130–132 | Passes — empty route |
| `forward_rate_limiter` (1/s per `(from,to,item_id)`) | L134–145 | Passes — one message |
| `route.last() == None` | L147 | Passes — empty route |
| `self_peer_id == content.from` | **L150–151** | **Passes — attacker set `from` = victim's own peer ID** |
| `inflight_requests.remove(&content.to)` | **L160** | **Returns `Some(start)` — attacker used observed peer ID** | [4](#0-3) 

**Step 5 — `try_nat_traversal` spawns tasks toward attacker-controlled IPs.** The `listen_addrs` from the message are passed directly to `try_nat_traversal`. The only address validation checks that any embedded peer ID matches `content.to`; it does **not** validate that the IP addresses are associated with the `to` peer in the peer store. A single `runtime::spawn` is issued containing `select_ok(tasks)`, which polls all 24 futures concurrently. [5](#0-4) [6](#0-5) 

**Step 6 — Each future loops for 30 seconds.** Each `try_nat_traversal` future creates a new `TcpSocket`, attempts `connect` with a 200ms timeout, waits ~200ms, and repeats until 30 seconds elapse (~150 iterations per future). [7](#0-6) 

**Why existing checks are insufficient:** The `self_peer_id == content.from` check was designed to identify the originating node of a legitimate relay chain, but it is reachable from any directly connected peer session. There is no relay-chain signature, nonce, or session-origin check that would distinguish a legitimately relayed `ConnectionRequestDelivered` from one injected directly by a connected peer. The `forward_rate_limiter` (1/s per `(from, to, item_id)`) and `rate_limiter` (30/s per session) are both satisfied by a single message per `inflight_requests` entry. [8](#0-7) 

## Impact Explanation

Per triggered `inflight_requests` entry: 24 concurrent TCP connection loops run for 30 seconds, each creating a new socket every ~200ms (~150 iterations), producing ~3,600 outbound TCP SYN packets toward attacker-chosen IPs. With K=8–10 typical inflight entries per cycle, this yields ~28,800–36,000 SYN packets per 5-minute cycle directed at arbitrary third-party hosts. The victim node is turned into a repeatable TCP SYN amplifier with no authentication cost to the attacker. This exhausts the victim's outbound socket and network resources, degrading its ability to maintain legitimate CKB P2P connections and contributing to CKB network congestion.

This matches **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**: the attack requires only a single connected peer (minimal cost), repeats every 5 minutes indefinitely, consumes the victim's outbound network and socket resources, and can be directed at any IP. [9](#0-8) [10](#0-9) 

## Likelihood Explanation

- Requires only a standard P2P connection — no special privilege, no PoW, no key material
- Victim's peer ID is public (exchanged during identify handshake)
- Attacker has a non-trivial probability of being among the `sqrt(total)` gossip recipients
- `inflight_requests.remove()` limits each entry to one exploitation per cycle, but the cycle repeats every 5 minutes indefinitely
- `forward_rate_limiter` (1/s per `(from, to, item_id)`) and `rate_limiter` (30/s per session) are both satisfied by a single message per entry [11](#0-10) 

## Recommendation

1. **Authenticate the delivery path.** Include a relay-chain HMAC or nonce in `ConnectionRequestDelivered` that the originating node can verify was produced by the actual relay chain, not injected directly by a connected peer.
2. **Restrict terminal branch to relay sessions.** The `self_peer_id == content.from` branch should only be reachable from a session that is a known relay node, not from any arbitrary connected peer.
3. **Validate `listen_addrs` against the peer store.** Check that the IP addresses in `listen_addrs` are consistent with previously observed addresses for `content.to`, preventing use of arbitrary third-party IPs.
4. **Cap concurrent NAT traversal tasks.** Maintain a global semaphore or bounded task counter to limit the total number of simultaneously live NAT traversal tasks regardless of how many `inflight_requests` entries are consumed. [12](#0-11) 

## Proof of Concept

```rust
// 1. Connect attacker peer to victim node (standard P2P connection)
// 2. Wait for ConnectionRequest gossip broadcast — extract `to_peer_id` from `to` field
// 3. Craft message:

    .from(victim_peer_id_bytes)        // victim's own public peer ID (from identify)
    .to(observed_to_peer_id)           // from observed ConnectionRequest broadcast
    .route(BytesVec::default())        // empty → terminal branch
    .sync_route(BytesVec::default())
    .listen_addrs(build_24_tcp_addrs(attacker_target_ips))  // arbitrary IPs, no peer ID component
    .build();
// 4. Send to victim over HolePunching protocol session
// 5. Victim enters terminal branch: inflight_requests.remove() returns Some(start)
// 6. try_nat_traversal spawns runtime::spawn with select_ok(24 futures)
//    → 24 concurrent TCP connection loops toward attacker_target_ips for 30s
// 7. Repeat for each observed to_peer_id in the same notify() cycle (K entries)
// Assert: K*24 concurrent TCP futures active, victim sends SYN packets to attacker_target_ips
// Assert: attack repeats every 5 minutes (CHECK_INTERVAL)
```

    fn try_nat_traversal(&self, ttl: u64, remote_addrs: Vec<Multiaddr>) {
        let tasks = remote_addrs
            .into_iter()
            .filter_map(|listen_addr| match find_type(&listen_addr) {
                TransportType::Tcp => {
                    if listen_addr
                        .iter()
                        .any(|p| matches!(p, Protocol::Ip4(_) | Protocol::Ip6(_)))
                    {
                        Some(Box::pin(try_nat_traversal(self.bind_addr, listen_addr)))
                    } else {
                        None
                    }
                }
                TransportType::Memory
                | TransportType::Onion
                | TransportType::Ws
                | TransportType::Wss
                | TransportType::Tls => None,
            })
            .collect::<Vec<_>>();

        if tasks.is_empty() {
            return;
        }

        debug!("start NAT traversal");

        let control = self.p2p_control.clone();

        runtime::spawn(async move {
            runtime::delay_for(std::time::Duration::from_millis(ttl / 2)).await;
            if let Ok(((stream, addr), _)) = select_ok(tasks).await {
                debug!("NAT traversal success, addr: {:?}", addr);
                if let Some(metrics) = ckb_metrics::handle() {
                    metrics.ckb_hole_punching_active_success_count.inc();
                }
                let _ignore = control
                    .raw_session(
                        stream,
                        addr,
                        RawSessionInfo::outbound(TargetProtocol::Single(
                            SupportProtocols::Identify.protocol_id(),
                        )),
                    )
                    .await;
            }
        });
    }
