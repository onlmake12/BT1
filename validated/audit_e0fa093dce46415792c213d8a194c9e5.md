### Title
Missing Default Batch Request Limit Allows Unbounded RPC Resource Exhaustion — (`File: rpc/src/server.rs`)

### Summary
The CKB JSON-RPC server has no enforced default limit on the number of calls in a JSON-RPC batch request. The `rpc_batch_limit` configuration option is `Option<usize>` and is commented out (disabled) in the shipped default configuration. When not set, the batch-size guard is never triggered, allowing any RPC caller to submit a single HTTP request containing an arbitrarily large number of method calls bounded only by the 10 MiB body limit. This is the CKB analog of the LOB contract's missing pausable/operational-control functionality: there is no built-in, always-on mechanism to cap or halt runaway RPC workloads.

### Finding Description
In `rpc/src/server.rs`, the batch-request handler checks the limit only when `JSONRPC_BATCH_LIMIT` has been initialised:

```rust
static JSONRPC_BATCH_LIMIT: OnceLock<usize> = OnceLock::new();
...
if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
    && calls.len() > *batch_size
{
    return make_error_response(...);
}
```

`JSONRPC_BATCH_LIMIT` is populated only when `config.rpc_batch_limit` is `Some(n)`:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

The shipped default configuration explicitly leaves this option disabled:

```toml
# By default there is no limitation on the size of batch request size
# a huge batch request may cost a lot of memory or makes the RPC server slow,
# to avoid this, you may want to add a limit for the batch request size.
# rpc_batch_limit = 2000
```

`rpc_batch_limit` is typed as `Option<usize>` with no `#[serde(default)]` fallback, so it deserialises to `None` when absent. The `OnceLock` is therefore never written, `JSONRPC_BATCH_LIMIT.get()` always returns `None`, and the guard branch is never entered. Every node running the default `ckb.toml` is permanently in the unprotected state with no runtime way to enforce a limit short of restarting with a modified config.

### Impact Explanation
Within the 10 MiB body limit, a caller can pack tens of thousands of JSON-RPC calls into a single request. Each call is dispatched through the async handler. Expensive calls (`get_block`, `send_transaction`, `get_raw_tx_pool`, etc.) executed in bulk can saturate the Tokio thread pool, exhaust heap memory, and stall the RPC service for the duration of the batch. Because the RPC service shares the async runtime with the tx-pool and chain services, prolonged saturation can delay block-template generation, transaction relay, and peer-management tasks, degrading overall node liveness. There is no in-process mechanism to abort or rate-limit an in-flight batch once accepted.

### Likelihood Explanation
The RPC binds to `127.0.0.1:8114` by default, so the attacker must be a local or loopback-reachable caller — a category explicitly listed in the bounty scope ("RPC caller", "supported local CLI/RPC user"). Any process on the same host (a co-located application, a compromised dependency, or a malicious script) can reach the endpoint without credentials. The attack requires only a single HTTP POST with a large JSON array; no special knowledge or privilege is needed.

### Recommendation
Set a safe, non-`None` default for `rpc_batch_limit` in `RpcConfig` (e.g., 2000) so that the guard is always active out of the box. Alternatively, add `#[serde(default = "default_rpc_batch_limit")]` to the field and provide a sensible constant, mirroring how other pool limits are handled. Operators who need larger batches can raise the value explicitly.

### Proof of Concept
1. Start `ckb run` with the default `ckb.toml` (no `rpc_batch_limit` line).
2. Confirm `JSONRPC_BATCH_LIMIT` is never set: the guard at line 275 of `rpc/src/server.rs` is never entered.
3. Send a single HTTP POST to `http://127.0.0.1:8114`:
   ```bash
   python3 -c "
   import json, requests
   batch = [{'id':i,'jsonrpc':'2.0','method':'get_tip_header','params':[]} for i in range(50000)]
   requests.post('http://127.0.0.1:8114', json=batch, timeout=120)
   "
   ```
4. Observe the node's CPU and memory spike while processing all 50 000 calls sequentially; concurrent RPC requests stall until the batch completes.
