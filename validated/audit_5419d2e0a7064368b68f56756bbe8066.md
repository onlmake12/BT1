Audit Report

## Title
Unbounded Per-Peer DB Read Amplification in Filter Protocol — Missing Rate Limiter in `BlockFilter::try_process` - (File: `sync/src/filter/mod.rs`)

## Summary
The `BlockFilter` protocol handler processes `GetBlockFilters` and `GetBlockFilterHashes` messages with no per-peer rate limiting. Each `GetBlockFilters` message triggers up to 2,000 RocksDB reads (1,000 iterations × `get_block_hash` + `get_block_filter`), and each `GetBlockFilterHashes` message triggers up to 4,001 reads (2,000 iterations × 2 reads + 1 parent hash read). Any unprivileged inbound peer can send these 9-byte messages in a tight loop, and N such peers multiply the I/O amplification linearly, degrading or crashing the node.

## Finding Description
**Root cause:** The `BlockFilter` struct contains only a `shared: Arc<SyncShared>` field with no rate-limiter. [1](#0-0) 

`try_process` dispatches directly to process handlers with zero rate-checking logic before any handler is invoked. [2](#0-1) 

**DB amplification — `GetBlockFilters`:** `BATCH_SIZE = 1000`. Each loop iteration calls `active_chain.get_block_hash(block_number)` and `active_chain.get_block_filter(&block_hash)` — two RocksDB reads — for up to 2,000 reads per message. The 1.8 MB size cap only breaks the loop when accumulated filter data reaches that threshold; for early mainnet blocks with small filters, all 1,000 iterations complete. [3](#0-2) [4](#0-3) 

**DB amplification — `GetBlockFilterHashes`:** `BATCH_SIZE = 2000`. Each loop iteration calls `get_block_hash` chained with `get_block_filter_hash` (2 reads), plus 1 extra read for the parent hash before the loop, for up to 4,001 reads per message. [5](#0-4) [6](#0-5) 

**Contrast with Relay protocol:** The `Relayer` struct carries a `rate_limiter: RateLimiter<(PeerIndex, u32)>` initialized at 30 req/s and checked in `try_process` before every dispatch. `BlockFilter` has no equivalent. [7](#0-6) [8](#0-7) 

## Impact Explanation
Sustained high-rate `GetBlockFilters(start_number=0)` messages from multiple inbound peers cause RocksDB I/O saturation, starving block sync, tx-pool, and all other DB-dependent operations. The async executor tasks holding the full loop duration also starve other protocol handlers. This can degrade or crash a CKB node and deny service to legitimate light clients and sync peers. This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* and *Vulnerabilities which could easily crash a CKB node*. [4](#0-3) 

## Likelihood Explanation
The Filter protocol is enabled by default. The attack message is 9 bytes with no PoW, stake, or authentication requirement — any unprivileged peer can send it. The attacker only needs to establish inbound connections (trivially achievable) and send messages at maximum rate. The existing 1.8 MB size cap (added in #4972) mitigates response-side amplification but leaves the request rate entirely unbounded. The contrast with the Relay protocol's explicit rate limiter confirms this is an unintentional omission. [2](#0-1) 

## Recommendation
Mirror the Relay protocol's rate-limiting pattern in `BlockFilter`:

1. Add `rate_limiter: RateLimiter<(PeerIndex, u32)>` to the `BlockFilter` struct (same type alias as in `sync/src/relayer/mod.rs`).
2. Initialize it in `BlockFilter::new` with a quota appropriate for the higher per-message cost (e.g., 5–10 req/s given 2,000 reads/msg vs. relay's 30 req/s for cheaper operations).
3. In `try_process`, check `self.rate_limiter.check_key(&(peer, message.item_id()))` before dispatching to any handler, returning `StatusCode::TooManyRequests` on failure.
4. Apply the same guard to `GetBlockFilterHashes` (BATCH_SIZE=2000, up to 4,001 reads/msg) and `GetBlockFilterCheckPoints`.
5. Call `self.rate_limiter.retain_recent()` in `disconnected` to prevent unbounded memory growth. [9](#0-8) [10](#0-9) 

## Proof of Concept
Using the existing integration test harness at `test/src/specs/sync/block_filter.rs` as a template:

```
1. Spawn a CKB full node with block filter enabled (default config).
2. Connect N peers (e.g., N=50) on the Filter protocol.
3. Each peer sends in a tight loop:
     GetBlockFilters { start_number: 0 }
4. Observe on the target node:
   - RocksDB read IOPS spike to N × ~2,000 reads/round
   - CPU usage on async executor threads increases
   - Sync latency for honest peers degrades measurably
   - Node I/O wait increases proportionally with N
5. Apply a patch adding a 10 req/s rate limiter to BlockFilter::try_process.
6. Repeat: IOPS are now capped at N × 10 × 2,000 reads/s regardless of send rate.
``` [3](#0-2) [5](#0-4)
