Audit Report

## Title
`ScriptHashTypeVerifier` Fails to Validate Type Script `hash_type` in Transaction Outputs — (`File: verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates `hash_type` exclusively for the lock script of each output. The type script `hash_type` is never read or validated. Because `ENABLED_SCRIPT_HASH_TYPE` excludes value `3`, an unprivileged sender can embed a disallowed `hash_type` byte in an output's type script, pass `NonContextualTransactionVerifier`, and commit a permanently unspendable cell to the chain — locking capacity forever.

## Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is defined as `{0, 1, 2, 4}` — value `3` is explicitly absent. [1](#0-0) 

`ScriptHashTypeVerifier::verify()` loops over outputs and checks only `output.lock().hash_type()`. The `output.type_()` field is never consulted. [2](#0-1) 

`NonContextualTransactionVerifier::verify()` calls `self.script_hash_type.verify()` as the final step, and this is the only hash-type gate in the non-contextual path. [3](#0-2) 

A CKB cell output carries two independent scripts; the type script has its own `hash_type` byte that is completely ignored by this verifier. An attacker crafts an output with `lock.hash_type = 0` (passes the check) and `type.hash_type = 3` (never checked). The transaction passes `NonContextualTransactionVerifier` and is committed. Any subsequent spending transaction triggers type-script execution; `ScriptHashType::try_from(3)` returns `Err` because no variant is defined for `3`, causing the spending transaction to be rejected by every node uniformly — the cell is permanently unspendable.

## Impact Explanation

The concrete impact is **permanent capacity lock**: an attacker funds a cell with a disallowed type-script `hash_type`, then transfers it to any address. In CKB's UTXO model, recipients cannot refuse incoming cells. The locked capacity is irrecoverable, constituting direct economic damage to cell recipients and reducing effective circulating supply. This maps to **"Vulnerabilities which could easily damage CKB economy" (Critical, 15001–25000 points)**. A secondary impact — consensus deviation — becomes concrete if a new `ScriptHashType` variant is added to the enum but intentionally withheld from `ENABLED_SCRIPT_HASH_TYPE` pending a hardfork, since nodes at different upgrade states would resolve the hash type differently.

## Likelihood Explanation

The attack requires only the ability to submit a standard transaction — available to every RPC caller or P2P relayer. No special key, role, or victim mistake is needed. The attacker funds the cell capacity themselves (a one-time cost per attack), then transfers the unspendable cell to a target address. The bypass is deterministic and repeatable. Likelihood is **high** for the capacity-lock scenario at any time.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to validate the type script `hash_type` of every output in addition to the lock script `hash_type`. After the existing lock script check, add:

```rust
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(ht) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = ht.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        })
        .into());
    }
}
```

This closes the bypass symmetrically for both script fields.

## Proof of Concept

1. Confirm `ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}` — value `3` is absent. [1](#0-0) 
2. Build a `TransactionView` with one output: `lock: { hash_type: 0 }` (passes the verifier), `type: { hash_type: 3 }` (never read by the verifier).
3. Run `NonContextualTransactionVerifier::new(&tx, &consensus).verify()` — it returns `Ok(())` because only `output.lock().hash_type()` is checked. [4](#0-3) 
4. Submit the transaction; it is accepted into the tx-pool and committed to the chain.
5. Attempt to spend the resulting cell — script execution fails because `ScriptHashType::try_from(3)` returns `Err`, making the cell permanently unspendable.
6. Repeat targeting arbitrary recipient addresses to lock their capacity at will.
