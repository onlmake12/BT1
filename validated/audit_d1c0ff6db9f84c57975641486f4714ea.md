### Title
Unbounded Descendant Traversal and O(N) Pending-List Scan in `get_pool_tx_detail_info` RPC Enables CPU-Based DoS — (File: `tx-pool/src/pool.rs`)

---

### Summary

The `get_pool_tx_detail_info` RPC endpoint triggers three unbounded or O(N) operations inside `get_tx_detail` in `tx-pool/src/pool.rs`: (1) a full linear scan over all pending transaction hashes to compute `rank_in_pending`, (2) an unbounded BFS traversal of all descendants via `calc_descendants`, and (3) an unbounded BFS traversal of all ancestors via `calc_ancestors`. An unprivileged RPC caller can pre-populate the pool with many transactions that share a common root as a cell-dep (creating a wide descendant fan-out), then repeatedly invoke `get_pool_tx_detail_info` on that root to force expensive traversals, degrading or stalling the tx-pool service for all other users.

---

### Finding Description

**Root cause — `get_tx_detail` in `tx-pool/src/pool.rs`:** [1](#0-0) 

The function performs three expensive operations on every call:

**1. O(N) linear scan for `rank_in_pending`:**

```rust
ids.pending
    .iter()
    .enumerate()
    .find(|(_, hash)| &tx_hash == *hash)
    .map(|r| r.0)
    .unwrap_or_default()
    + 1
```

`ids.pending` is a `Vec<Byte32>` containing every pending transaction hash. The `.find()` scan is O(N) with no early-exit guarantee for a transaction near the end of the list. [2](#0-1) 

**2. Unbounded descendant BFS — `calc_descendants`:** [3](#0-2) 

This calls into `TxLinksMap::calc_relation_ids` in `tx-pool/src/component/links.rs`:

```rust
while let Some(id) = stage.iter().next().cloned() {
    //recursively
    if let Some(tx_links) = self.inner.get(&id) {
        for direct_id in tx_links.get_direct_ids(relation) {
            if !relation_ids.contains(direct_id) {
                stage.insert(direct_id.clone());
            }
        }
    }
    stage.remove(&id);
    relation_ids.insert(id);
}
``` [4](#0-3) 

This loop has **no internal bound**. It visits every reachable descendant in the pool. Critically, the pool admission logic (`check_and_record_ancestors`) only limits the **ancestor count** of a newly submitted transaction; it does **not** limit how many descendants an existing transaction may accumulate.

**3. Unbounded ancestor BFS — `calc_ancestors`:** [5](#0-4) 

Same `calc_relation_ids` loop, traversing the parent direction. While ancestors are bounded to `max_ancestors_count` (default 1000) by admission, the traversal itself has no internal guard.

**The RPC handler holds a read lock during all three operations:** [6](#0-5) 

Multiple concurrent RPC calls each hold a read lock while performing these expensive computations, amplifying the effect.

---

### Impact Explanation

An attacker who fills the pool with a wide descendant fan-out (e.g., 2000 transactions each referencing a common root as a `cell_dep`) and then repeatedly calls `get_pool_tx_detail_info` on that root forces the node to:

- Traverse up to 2000+ entries per `calc_descendants` call (as demonstrated by the `TxPoolLimitAncestorCount` integration test which successfully submits 2000 cell-dep-referencing transactions)
- Linearly scan all pending hashes for `rank_in_pending` [7](#0-6) 

This consumes significant CPU on the node's tx-pool service, degrading transaction submission, block template generation, and all other pool operations for legitimate users. The RPC endpoint is publicly documented and accessible to any configured RPC caller. [8](#0-7) 

---

### Likelihood Explanation

- `get_pool_tx_detail_info` is a standard, documented RPC method with no rate limiting in the RPC layer.
- Submitting many cell-dep-referencing transactions is cheap: each only needs to meet the minimum fee rate and minimum capacity, and they are independent (no chaining required), so the attacker pays only the minimum cost per transaction.
- The descendant count is not bounded by any pool admission rule, only by total pool byte size. With small transactions, thousands of descendants are feasible.
- The attack requires no special privilege — only RPC access (which is the documented attacker role for this scope).

---

### Recommendation

**Short term:**
- Add an internal iteration cap inside `calc_relation_ids` (e.g., bail out after visiting `max_ancestors_count` nodes) so the traversal cannot exceed a known bound regardless of pool state.
- Cache `descendants_count` and `ancestors_count` on the `TxEntry` (analogous to the already-cached `ancestors_count` field) so `get_tx_detail` reads a pre-computed value instead of re-traversing the graph.
- Replace the O(N) `rank_in_pending` linear scan with a rank maintained in the sorted index, or remove `rank_in_pending` from this endpoint.

**Long term:**
- Apply per-IP or per-connection rate limiting to expensive read-only RPC methods.
- Enforce a bound on the number of descendants a single transaction may accumulate in the pool (analogous to the existing `max_ancestors_count` limit), preventing unbounded fan-out.

---

### Proof of Concept

1. Submit `tx_root` spending a live cell.
2. Submit 2000 transactions `tx_1 … tx_2000`, each spending a distinct live cell and referencing `tx_root`'s output as a `cell_dep` (not as an input). Each passes pool admission because its own ancestor count is 1.
3. Repeatedly call:
   ```json
   {"method": "get_pool_tx_detail_info", "params": ["<tx_root_hash>"]}
   ```
4. Each call triggers:
   - `calc_descendants(tx_root_id)` → BFS over 2000 entries
   - `calc_ancestors(tx_root_id)` → BFS (small, but still traversed)
   - Linear scan of all ~2001 pending hashes for `rank_in_pending`
5. Concurrent repeated calls hold read locks simultaneously, saturating CPU on the tx-pool service and delaying all other pool operations (transaction submission, block template assembly). [9](#0-8) [1](#0-0) [10](#0-9)
