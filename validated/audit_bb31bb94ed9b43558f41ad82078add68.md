### Title
Transaction Ordering Dependency in `requestHelper` Allows Front-Running to Exhaust `maxNumHashes` Slot, Causing `LastRevealedTooOld` DoS — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract's `requestHelper` function checks a shared, incrementing counter (`sequenceNumber - currentCommitmentSequenceNumber`) against a provider-configured cap (`maxNumHashes`). Because this check depends on global state that any unprivileged caller can advance, an attacker can observe a victim's pending `requestWithCallback` / `requestV2` transaction in the mempool and front-run it with their own request, pushing the gap over the cap and causing the victim's transaction to revert with `LastRevealedTooOld`.

---

### Finding Description

In `requestHelper`, the number of hashes required for a new request is computed as:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [1](#0-0) 

`assignedSequenceNumber` is taken from `providerInfo.sequenceNumber`, which is incremented by every successful call to `requestHelper`:

```solidity
uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
    revert EntropyErrors.OutOfRandomness();
providerInfo.sequenceNumber += 1;
``` [2](#0-1) 

`currentCommitmentSequenceNumber` only advances when a reveal is processed:

```solidity
if (providerInfo.currentCommitmentSequenceNumber < req.sequenceNumber) {
    providerInfo.currentCommitmentSequenceNumber = req.sequenceNumber;
    providerInfo.currentCommitment = providerContribution;
}
``` [3](#0-2) 

The gap `sequenceNumber - currentCommitmentSequenceNumber` is therefore a shared resource that any caller can increment by submitting a request. The `maxNumHashes` field is a provider-set cap on this gap:

```solidity
uint32 maxNumHashes;
``` [4](#0-3) 

Both `requestWithCallback` and `requestV2` are fully permissionless entry points:

```solidity
function requestWithCallback(
    address provider,
    bytes32 userContribution
) public payable override returns (uint64) {
``` [5](#0-4)

### Impact Explanation

**Exploit scenario:**

1. Provider has `maxNumHashes = N` configured (a standard gas-management setting).
2. The current gap is `N − 1` (N − 1 in-flight, unrevealed requests).
3. Alice submits `requestWithCallback` — her request would produce `numHashes = N`, which satisfies `N ≤ N` (passes).
4. Bob observes Alice's pending transaction in the mempool and front-runs it with his own `requestWithCallback` using a higher gas price.
5. Bob's request is mined first; the gap becomes `N`.
6. Alice's transaction is now processed: `assignedSequenceNumber` is one higher, so `numHashes = N + 1 > N`, and the call reverts with `LastRevealedTooOld`.

Alice's entropy request fails. She loses gas (though not her fee, since the transaction reverts). If Bob repeats this every time Alice retries, she is persistently denied service from this provider until the Fortuna keeper reveals enough requests to reduce the gap — a window that can be exploited repeatedly.

The `testLastRevealedTooOld` test confirms this revert path is reachable:

```solidity
function testLastRevealedTooOld() public {
    for (uint256 i = 0; i < provider1MaxNumHashes; i++) {
        request(user1, provider1, 42, false);
    }
    assertRequestReverts(
        random.getFee(provider1),
        provider1,
        42,
        false,
        EntropyErrors.LastRevealedTooOld.selector
    );
}
``` [6](#0-5) 

---

### Likelihood Explanation

- `requestWithCallback` and `requestV2` are fully permissionless; no special role is required.
- The attacker only needs to pay the provider fee for a single request per front-run.
- The victim's `userContribution` is emitted in plaintext in the `RequestedWithCallback` event, so the attacker can also observe the victim's secret and potentially call `revealWithCallback` themselves (though the callback still goes to the original requester).
- Providers commonly set `maxNumHashes` to bound callback gas costs, making the near-limit state a realistic operational condition.
- The attack is cheap: one front-run transaction per victim retry.

---

### Recommendation

**Short term:** Document that when a provider's `maxNumHashes` cap is nearly reached, users may experience `LastRevealedTooOld` reverts due to transaction ordering. Recommend that users retry with a higher gas price or that integrators monitor the current gap before submitting.

**Long term:** Consider one or more of the following mitigations:

1. **Commit-then-request:** Allow users to pre-commit their request in one transaction and finalize it in a second, making the sequence number assignment deterministic and front-run-resistant.
2. **Per-user slot reservation:** Reserve a sequence number atomically at commit time so that the slot cannot be stolen.
3. **Soft cap with graceful degradation:** Instead of reverting, allow requests that exceed `maxNumHashes` to proceed with a higher fee, compensating the provider for the extra hashing work.

### Proof of Concept

```
State before attack:
  providerInfo.maxNumHashes = 10
  providerInfo.sequenceNumber = 11
  providerInfo.currentCommitmentSequenceNumber = 1
  gap = 11 - 1 = 10  (exactly at the limit)

Alice submits requestWithCallback (pending in mempool):
  Would get assignedSequenceNumber = 11
  numHashes = 11 - 1 = 10  → 10 ≤ 10 → PASSES

Bob front-runs with requestWithCallback (higher gas price):
  Gets assignedSequenceNumber = 11
  numHashes = 11 - 1 = 10 → PASSES, mined first
  providerInfo.sequenceNumber becomes 12

Alice's transaction is now processed:
  Gets assignedSequenceNumber = 12
  numHashes = 12 - 1 = 11 → 11 > 10 → REVERTS with LastRevealedTooOld
```
