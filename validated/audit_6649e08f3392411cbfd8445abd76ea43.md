### Title
Unchecked `_entropyCallback` Call in Legacy `revealWithCallback` Path Permanently Locks Entropy Requests - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains a legacy execution path (triggered when `req.gasLimit10k == 0`) that calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly without catching reverts. If the requester contract does not conform to the `IEntropyConsumer` interface or its callback reverts for any reason, the entire `revealWithCallback` transaction reverts. Because the request was cleared before the callback call, the revert undoes the clear, leaving the request permanently active but unfulfillable. The user loses their fee and never receives their random number.

---

### Finding Description

`revealWithCallback` in `Entropy.sol` has two execution branches:

**New path** (`req.gasLimit10k != 0 && callbackStatus == CALLBACK_NOT_STARTED`): Uses `excessivelySafeCall` to catch reverts, emits `CallbackFailed`, and transitions the request to `CALLBACK_FAILED` state for recovery.

**Legacy path** (`req.gasLimit10k == 0`, i.e., provider has `defaultGasLimit == 0`): Clears the request first, then calls `_entropyCallback` directly with no revert catching:

```solidity
clearRequest(provider, sequenceNumber);          // line 666 — cleared first
// ...
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(  // line 676 — direct call, no try/catch
        sequenceNumber,
        provider,
        randomNumber
    );
}
``` [1](#0-0) 

If `_entropyCallback` reverts (because the requester does not implement `IEntropyConsumer`, or its `entropyCallback` logic reverts), the entire transaction reverts — including the `clearRequest`. The request is restored to active state, but every subsequent `revealWithCallback` attempt will also revert for the same reason. The request is permanently stuck.

The legacy path is entered when `providerInfo.defaultGasLimit == 0`:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    // Provider doesn't support the new callback failure state flow
    req.gasLimit10k = 0;
}
``` [2](#0-1) 

The `IEntropyConsumer` interface's `_entropyCallback` itself calls `getEntropy()` and checks `msg.sender == entropy`. A requester contract that stores a stale or incorrect entropy address, or whose internal `entropyCallback` reverts, will cause this path to fail silently from the provider's perspective (the transaction reverts). [3](#0-2) 

The analog to the report is direct: just as `mintWithPermit` calls `permit()` without verifying success (assuming the token conforms to `IERC20Permit`), the legacy `revealWithCallback` path calls `_entropyCallback` without catching reverts (assuming the requester conforms to `IEntropyConsumer` and its callback never reverts).

---

### Impact Explanation

- **Permanent request lockup**: The entropy request remains active but can never be fulfilled. Every `revealWithCallback` call reverts.
- **Fee loss**: The user paid the entropy fee at request time. There is no refund mechanism for permanently stuck requests.
- **Random number never delivered**: Any application logic depending on the callback (NFT mints, games, lotteries) is permanently broken for that request.
- **No recovery path**: Unlike the new path which has `CALLBACK_FAILED` → retry recovery, the legacy path has no recovery mechanism. The request cannot be cleared by any party.

---

### Likelihood Explanation

- Any provider that has not called `setDefaultGasLimit` (i.e., `defaultGasLimit == 0`) triggers the legacy path for all their requests.
- Any user (unprivileged) can call `requestWithCallback` against such a provider.
- A requester contract whose `entropyCallback` reverts (due to a bug, out-of-gas, or a contract that does not properly inherit `IEntropyConsumer`) will trigger the lockup.
- The `extcodesize` check only verifies the requester is a contract — it does not verify `IEntropyConsumer` conformance. [4](#0-3) 

---

### Recommendation

Apply the same `excessivelySafeCall` pattern used in the new path to the legacy path as well. If the callback reverts, emit a failure event and leave the request in a recoverable state rather than reverting the entire transaction. Alternatively, deprecate the legacy path entirely by requiring all providers to set a non-zero `defaultGasLimit` before accepting new callback requests.

---

### Proof of Concept

1. Deploy a provider with `defaultGasLimit == 0` (never calls `setDefaultGasLimit`).
2. Deploy a requester contract that calls `requestWithCallback` but whose `entropyCallback` always reverts (e.g., `revert("not ready")`).
3. The requester calls `requestWithCallback{value: fee}(provider, userRandomNumber)` — succeeds, fee paid, sequence number assigned.
4. Provider/keeper calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`.
5. Execution enters the legacy path (`gasLimit10k == 0`), clears the request, then calls `_entropyCallback` on the requester.
6. The requester's callback reverts → entire transaction reverts → `clearRequest` is undone → request remains active.
7. All subsequent `revealWithCallback` calls revert identically.
8. The request is permanently stuck; the user's fee is lost; the random number is never delivered. [5](#0-4)

        uint64 sequence,
        address provider,
        bytes32 randomNumber

        address entropy = getEntropy();
