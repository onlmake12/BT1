### Title
`PythAggregatorV3.latestRoundData()` / `latestAnswer()` Use `getPriceUnsafe()` With No Staleness Check, Returning Arbitrarily Stale Prices to Chainlink-Compatible Consumers - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink `AggregatorV3Interface`-compatible adapter, distributed as part of `@pythnetwork/pyth-sdk-solidity` and actively recommended for protocols migrating from Chainlink. Every price-reading function in the contract — `latestRoundData()`, `latestAnswer()`, `getRoundData()`, `latestTimestamp()`, and `decimals()` — calls `pyth.getPriceUnsafe(priceId)`, which explicitly performs **no staleness check** and can return a price from arbitrarily far in the past. There is no revert, no error signal, and no `updatedAt` divergence that would alert a downstream Chainlink-compatible consumer that the price is stale.

---

### Finding Description

`PythAggregatorV3` implements the Chainlink `AggregatorV3Interface` so that existing Chainlink-integrated protocols can swap in Pyth prices with minimal code changes. The contract is the canonical migration path documented at `apps/developer-hub/content/docs/price-feeds/core/migrate-an-app-to-pyth/chainlink.mdx`.

All five price-reading functions delegate to `pyth.getPriceUnsafe(priceId)`:

```solidity
// latestRoundData — line 110
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
roundId = uint80(price.publishTime);
return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);

// latestAnswer — line 54
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
return int256(price.price);

// getRoundData — line 89
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
return (_roundId, int256(price.price), price.publishTime, price.publishTime, _roundId);
```

`getPriceUnsafe` is documented in `IPyth.sol` as: *"This function returns the most recent price update in this contract without any recency checks. This function is unsafe as the returned price update may be arbitrarily far in the past."*

The analog to the original report is direct: the original report found that a Chainlink oracle wrapper called `latestRoundData()` without checking `answeredInRound >= roundId`. Here, Pyth's own Chainlink-compatible wrapper calls `getPriceUnsafe()` — the equivalent of skipping all recency validation — and returns the result as if it were a fresh, authoritative price.

Protocols that call `latestRoundData()` on `PythAggregatorV3` and check `updatedAt` against a heartbeat threshold will find that `updatedAt == price.publishTime`, which is the Pyth `publishTime` — potentially hours or days old — with no revert to signal the problem.

---

### Impact Explanation

Any DeFi protocol (lending, derivatives, liquidation engine) that integrates `PythAggregatorV3` as a drop-in Chainlink replacement and calls `latestRoundData()` or `latestAnswer()` will silently receive stale prices when the Pyth feed has not been updated recently (e.g., during network congestion, market closure, or a push-feed scheduling gap). Concrete impacts:

- **Incorrect liquidations**: A borrower's collateral is valued at a stale (higher) price, preventing a valid liquidation; or a stale (lower) price triggers an unjust liquidation.
- **Undercollateralized loans**: A stale high price allows a borrower to take out more debt than the current collateral value supports.
- **Mispriced derivatives**: Stale settlement prices cause incorrect P&L calculations.

These are the same impacts enumerated in the original M-7 report.

---

### Likelihood Explanation

Pyth is a pull oracle. On chains or feeds without an active price pusher, the on-chain price can become arbitrarily stale. The migration guide explicitly tells protocols to deploy `PythAggregatorV3` and point their existing Chainlink app at it. A protocol that does so without adding its own `updatedAt` staleness check (which many Chainlink-compatible protocols do not, because they rely on the oracle adapter to revert on stale data) is immediately vulnerable. The entry path requires no privilege: any user of the integrated protocol triggers the stale-price path simply by submitting a transaction when the feed is stale.

---

### Recommendation

Replace `getPriceUnsafe` in all price-reading functions of `PythAggregatorV3` with `getPriceNoOlderThan`, parameterized by a configurable `maxAge` set at construction time:

```solidity
uint256 public maxAge; // e.g. 3600 seconds

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    // reverts with StalePrice() if price.publishTime is too old
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

This mirrors the fix in the original report (`require(answeredInRound >= roundID, "round not complete")`): the adapter should revert rather than silently return a stale price.

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing at a Pyth contract where feed `X` was last updated 2 hours ago.
2. Call `latestRoundData()` on the adapter.
3. Observe: the call succeeds and returns `answer = <2-hour-old price>`, `updatedAt = <2-hour-old timestamp>`, with no revert.
4. A Chainlink-compatible lending protocol that does not independently validate `updatedAt` will use this stale price for collateral valuation, enabling incorrect liquidations or undercollateralized borrowing.
