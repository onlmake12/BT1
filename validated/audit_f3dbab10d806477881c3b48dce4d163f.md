### Title
Missing Maximum Fee Bound in `requestPriceUpdatesWithCallback` Enables Provider Front-Running to Drain User Overpayment — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo.sol` contract's `requestPriceUpdatesWithCallback` function accepts `msg.value` as payment but provides no `maxFee` parameter for the caller to bound how much they are willing to pay. Because providers can update their fees at any time via `setProviderFee`, and because any `msg.value` beyond the Pyth base fee is stored verbatim as the provider's claimable fee (`req.fee = msg.value - pythFee`), a malicious provider can front-run a user's request transaction to inflate their fee and capture the user's entire ETH overpayment.

---

### Finding Description

`requestPriceUpdatesWithCallback` enforces only a minimum fee floor:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [1](#0-0) 

All ETH beyond the Pyth protocol base fee is then stored as the provider's claimable fee:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

When `executeCallback` is later called, the provider receives `req.fee` in full:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [3](#0-2) 

Providers can change their fee schedule at any time with no delay or cap:

```solidity
function setProviderFee(
    address provider,
    uint96 newBaseFeeInWei,
    uint96 newFeePerFeedInWei,
    uint96 newFeePerGasInWei
) external override {
    ...
    _state.providers[provider].baseFeeInWei = newBaseFeeInWei;
    _state.providers[provider].feePerFeedInWei = newFeePerFeedInWei;
    _state.providers[provider].feePerGasInWei = newFeePerGasInWei;
``` [4](#0-3) 

The interface documentation explicitly warns that fees change over time and that excess value is not refunded, but provides no mechanism for callers to protect themselves: [5](#0-4) 

---

### Impact Explanation

A user who sends `msg.value = getFee(...) + buffer` (a common defensive pattern to avoid reverts due to fee changes) can have the entire buffer extracted by a front-running provider. The provider captures `buffer` wei per sandwiched request with zero cost beyond two `setProviderFee` transactions. There is no on-chain mechanism for the user to express a maximum acceptable fee, so the attack is undetectable and unpreventable at the contract level.

---

### Likelihood Explanation

Any registered Echo provider can execute this attack permissionlessly. Providers are unprivileged actors who self-register via `registerProvider`. MEV infrastructure on all supported EVM chains makes mempool-visible `requestPriceUpdatesWithCallback` calls trivially front-runnable. The attack is profitable whenever the user's buffer exceeds the gas cost of two `setProviderFee` calls.

---

### Recommendation

**Short term:** Add a `maxFee` parameter to `requestPriceUpdatesWithCallback` and revert if `msg.value > maxFee` or if `getFee(...) > maxFee`. This gives callers a slippage-equivalent bound:

```solidity
function requestPriceUpdatesWithCallback(
    address provider,
    uint64 publishTime,
    bytes32[] calldata priceIds,
    uint32 callbackGasLimit,
    uint96 maxFee          // <-- new parameter
) external payable returns (uint64) {
    uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
    if (msg.value < requiredFee) revert InsufficientFee();
    if (msg.value > maxFee) revert ExceedsMaxFee();
    ...
}
```

**Long term:** Consider adding a time-lock or rate-limit on `setProviderFee` changes to reduce the viability of fee-manipulation front-running.

---

### Proof of Concept

1. Alice calls `getFee(provider, gasLimit, priceIds)` off-chain → returns `X` wei.
2. Alice submits `requestPriceUpdatesWithCallback{value: X + 0.01 ether}(...)` with a 0.01 ETH buffer to avoid reverts.
3. The provider monitors the mempool, sees Alice's transaction with `msg.value = X + 0.01 ether`.
4. Provider front-runs with `setProviderFee(provider, newBase, newPerFeed, newPerGas)` such that `getFee(...) = X + 0.01 ether`.
5. Alice's transaction executes: `req.fee = (X + 0.01 ether) - pythFee` — the full buffer is now the provider's claimable fee.
6. Provider back-runs with `setProviderFee(...)` to restore original fees.
7. Provider calls `executeCallback(...)` and withdraws `req.fee`, capturing Alice's 0.01 ETH buffer.

Alice receives her price update callback but pays 0.01 ETH more than the fair fee, with no recourse.
