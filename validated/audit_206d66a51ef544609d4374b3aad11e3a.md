### Title
Underpaid `callbackGasLimit` in Echo Permanently Prevents Callback Execution With No Recovery Path — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` credits provider fees and clears the request **before** invoking the user callback. If the user-provided `callbackGasLimit` is too low, the callback fails silently via `try/catch`, but the request is already gone — there is no retry or recovery mechanism. The user's callback is permanently lost.

---

### Finding Description

In `Echo.executeCallback`, the execution order is:

1. Provider fees are credited unconditionally (line 161–162)
2. The request is cleared from storage (line 164)
3. The callback is attempted with exactly `req.callbackGasLimit` gas (line 177–179)
4. If the callback runs out of gas or reverts, the `catch` block emits `PriceUpdateCallbackFailed` and returns silently

```solidity
// fees credited BEFORE callback
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

clearRequest(sequenceNumber);   // request DELETED before callback

try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit   // user-controlled gas cap
    }(sequenceNumber, priceFeeds)
{
    emitPriceUpdate(...);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(...);   // silent failure, no retry
} catch {
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
```

The developer comment at line 155–156 explicitly acknowledges the danger:

> `// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract.`

Unlike Entropy's `revealWithCallback`, which transitions to `CALLBACK_FAILED` state and allows re-invocation, Echo has **no retry path**. Once `clearRequest` runs, the sequence number is gone.

The `callbackGasLimit` is set by the requester at request time and is directly used as the gas stipend for the callback call. The fee paid by the user is proportional to `callbackGasLimit` (`callbackGasLimit * feePerGasInWei`), creating an economic incentive to set it as low as possible. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Any consumer contract whose `echoCallback` requires more gas than the `callbackGasLimit` it specified at request time will have its callback silently dropped with no recovery. For DeFi protocols using Echo for on-chain price-triggered logic (liquidations, conditional orders, oracle-gated state transitions), the critical application logic never executes. The fee paid is also permanently transferred to the provider. There is no mechanism to re-request or retry the same sequence number. [4](#0-3) 

---

### Likelihood Explanation

- The fee formula `callbackGasLimit * feePerGasInWei` directly incentivizes users to minimize `callbackGasLimit` to reduce costs.
- Gas usage of a callback is not static: it depends on storage slot warmth, EVM opcode repricing (EIPs), and the number of price feeds returned. A limit that was sufficient at test time may be insufficient in production.
- The `callbackGasLimit` is a `uint32` with no enforced minimum beyond the provider's base fee, so values of `0` or near-zero are accepted.
- The `try/catch` pattern masks the failure entirely from the caller's perspective — no revert propagates, making the failure invisible unless the `PriceUpdateCallbackFailed` event is monitored. [5](#0-4) [6](#0-5) 

---

### Recommendation

Apply the same pattern used in Entropy's `revealWithCallback`:

1. **Do not clear the request before the callback succeeds.** Move `clearRequest` inside the success branch of the `try` block.
2. **Introduce a `CALLBACK_FAILED` state** so that a failed callback can be retried (with a higher gas limit or after fixing the consumer contract).
3. **Alternatively**, revert `executeCallback` entirely if the callback fails due to out-of-gas, so the provider must supply sufficient gas to the outer transaction (as Entropy does with `InsufficientGas`).

Reference: Entropy's recovery pattern: [7](#0-6) [8](#0-7) 

---

### Proof of Concept

1. Consumer contract deploys with an `echoCallback` that performs 5 storage writes (~25,000 gas each = ~125,000 gas total).
2. Consumer calls `requestPriceUpdatesWithCallback(..., callbackGasLimit=50000)` to save on fees.
3. Provider calls `executeCallback(...)`.
4. Inside `executeCallback`:
   - `accruedFeesInWei` is incremented for the provider.
   - `clearRequest(sequenceNumber)` deletes the request.
   - `_echoCallback{gas: 50000}(...)` is called — runs out of gas.
   - `catch` block emits `PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)")`.
5. Consumer's `echoCallback` never ran. The sequence number no longer exists in storage.
6. There is no function in `IEcho` to retry a failed callback. The consumer's application state is permanently stale. [9](#0-8) [10](#0-9)
