Audit Report

## Title
Missing Deduplication in `GetTransactionsProofProcess::execute` Enables 1000x CPU/IO Amplification — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`GetTransactionsProofProcess::execute` accepts up to 1000 tx hashes with no deduplication guard, while the structurally identical `GetBlocksProofProcess::execute` explicitly rejects duplicates via a `HashSet`. An unprivileged remote peer can send 1000 identical copies of one valid tx hash, forcing 2000 RocksDB reads, a `CBMT::build_merkle_proof` call with 1000 duplicate indices (with a `.expect()` that may panic and crash the node), and a response containing 1000 copies of the same transaction — all at negligible attacker cost and with no ban triggered.

## Finding Description

`GetBlocksProofProcess::execute` builds a `HashSet` and returns `MalformedProtocolMessage` on any duplicate (`get_blocks_proof.rs` lines 62–70). `GetTransactionsProofProcess::execute` has no equivalent guard. After the length check at lines 37–39, `tx_hashes` are iterated directly through a `partition` call that invokes `snapshot.get_transaction_info` once per hash (line 61), then a second loop calls `snapshot.get_transaction_with_info` once per hash (line 69). Both calls are issued for every duplicate. All 1000 identical hashes resolve to the same block hash key in `txs_in_blocks`, producing a `Vec` of 1000 `(tx, index)` pairs with the same index. `CBMT::build_merkle_proof` is then called with a 1000-element slice of the same index (lines 86–97), followed by `.expect("build proof with verified inputs should be OK")`. If the `merkle_cbt` crate returns `None` for duplicate indices, this panics and crashes the node. Even if it does not panic, the response is serialized with 1000 copies of the same transaction. The message is well-formed, so `try_process` returns `Status::ok()` and `should_ban()` is never triggered, leaving the attacker free to repeat indefinitely.

## Impact Explanation

**High** — matches "Vulnerabilities or bad designs which could cause CKB network congestion with few costs" and potentially "Vulnerabilities which could easily crash a CKB node." Per single malformed-but-well-formed request: 1000× `get_transaction_info` (RocksDB reads), 1000× `get_transaction_with_info` (RocksDB reads), `CBMT::build_merkle_proof` with 1000 duplicate indices (CPU), and a response payload with 1000 copies of the same transaction (bandwidth amplification). If `build_merkle_proof` returns `None` for duplicate indices, the `.expect()` panics and crashes the node outright. A single persistent connection sustaining this at the server's processing rate constitutes a low-cost, high-amplification resource exhaustion or crash attack.

## Likelihood Explanation

The light client protocol is a supported production P2P protocol. Any peer that can connect and send a `LightClientMessage::GetTransactionsProof` can trigger this — no authentication, stake, or proof-of-work is required. The message dispatch path has no per-peer rate limiting. A well-formed message with 1000 duplicate hashes passes all current checks cleanly and returns `Status::ok()`, so no ban is ever applied to the attacker.

## Recommendation

Add a deduplication check immediately after the length check at line 39, mirroring the pattern in `GetBlocksProofProcess`:

```rust
let tx_hashes: Vec<_> = self.message.tx_hashes().to_entity().into_iter().collect();
let mut uniq = std::collections::HashSet::new();
if !tx_hashes.iter().all(|h| uniq.insert(h)) {
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate tx hash exists");
}
```

This is exactly the pattern already used in `GetBlocksProofProcess` at `get_blocks_proof.rs` lines 62–70. Additionally, the `.expect()` on `CBMT::build_merkle_proof` at line 97 should be replaced with proper error handling to avoid a panic-based crash path even after deduplication is added.

## Proof of Concept

1. Connect to a CKB node with the light client protocol enabled.
2. Obtain any confirmed tx hash `H` on the main chain.
3. Send a `GetTransactionsProof` message with `tx_hashes = [H; 1000]` and a valid `last_hash` pointing to a main-chain block.
4. Instrument `snapshot.get_transaction_info` and `snapshot.get_transaction_with_info`: each will be called exactly 1000 times.
5. Observe the `SendTransactionsProof` response contains 1000 copies of the same transaction under one `FilteredBlock`, or observe a node panic if `CBMT::build_merkle_proof` returns `None` for 1000 duplicate indices.
6. Repeat in a tight loop from a single connection to sustain amplified load; no ban is triggered because the message is well-formed and `Status::ok()` is returned, which does not satisfy `should_ban()`.
