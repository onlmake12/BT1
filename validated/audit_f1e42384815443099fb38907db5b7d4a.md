Audit Report

## Title
VerifyQueue Hard-Cap with No Fee-Rate Eviction Enables DoS of Transaction Admission Pipeline - (`tx-pool/src/component/verify_queue.rs`)

## Summary

`VerifyQueue` enforces a hard 256 MB size cap with no fee-rate metadata and no quality-based eviction. An unprivileged peer can flood the queue with structurally valid, minimum-fee-rate transactions with fake inputs, filling it entirely. Because `add_tx` unconditionally returns `Reject::Full` when the cap is reached and no eviction path exists, all subsequent transactions — including high-fee-rate ones — are rejected until the attacker's transactions drain through the worker pipeline.

## Finding Description

**Hard cap with unconditional rejection:**
`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` is set to 256 MB. [1](#0-0) 

When `add_tx` is called and `is_full` returns true, it unconditionally returns `Reject::Full` with no eviction attempt: [2](#0-1) 

**No fee-rate metadata in `VerifyEntry`:**
`VerifyEntry` stores only `id`, `added_time`, `is_large_cycle`, `is_proposal_tx`, and `inner`. There is no fee-rate field, making quality-based eviction structurally impossible: [3](#0-2) 

**Fee check is deferred past the enqueue point:**
`resumeble_process_tx` — the entry point for both P2P relay (`submit_remote_tx`) and RPC (`notify_tx`) — calls only `non_contextual_verify` before calling `enqueue_verify_queue`: [4](#0-3) 

`check_tx_fee` is called inside `pre_check`, which is only invoked by `_process_tx` — the worker path that runs *after* a transaction is dequeued: [5](#0-4) [6](#0-5) 

**No eviction operations exist:**
The only removal methods on `VerifyQueue` are `remove_tx`, `remove_txs`, `remove_txs_by_peer`, and `pop_front`. None evict based on fee-rate to make room for a better entry: [7](#0-6) 

**Attacker is not banned:**
`ban_malformed` is only triggered when `reject.is_malformed_tx()` returns true inside `non_contextual_verify`. Transactions with fake inputs pass structural validation and fail later at `resolve_tx` in `pre_check` with `Reject::Resolve(OutPointError::Unknown)`, which does not trigger a ban: [8](#0-7) 

**Exploit flow:**
1. Attacker crafts structurally valid transactions with fake (non-existent) inputs and minimum declared fee rate. These pass `non_contextual_verify` because that check is purely structural.
2. Attacker relays them via P2P, triggering `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue` → `add_tx`.
3. `total_tx_size` reaches 256 MB. All subsequent `add_tx` calls return `Reject::Full`.
4. Workers dequeue attacker transactions, which fail at `resolve_tx` in `pre_check` (unknown inputs). The peer is not banned and transactions may be routed to the orphan pool.
5. Attacker re-floods to maintain saturation. Legitimate high-fee-rate transactions are rejected with `Reject::Full`.

## Impact Explanation

Any transaction submitted via the P2P relay path or the RPC `send_transaction` path is blocked from entering the verification pipeline while the VerifyQueue is saturated. High-fee-rate transactions from legitimate users are rejected with `Reject::Full` and never reach the main pool. This constitutes a targeted DoS on the transaction admission pipeline of a CKB node. **Impact: High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

No privileged access, key material, or majority hashpower is required. The attacker needs only a P2P connection. Transactions need only pass `non_contextual_verify` (structural validity, no input resolution, no fee check). The attacker can craft transactions with fake inputs up to `TRANSACTION_SIZE_LIMIT` in size, each paying exactly `min_fee_rate` in declared fees. Because the fee check is deferred, these transactions occupy queue space until a worker dequeues and rejects them. The attacker is not banned because fake-input transactions fail at resolution (not at structural validation). The attack is repeatable and maintainable from a single well-connected peer.

## Recommendation

1. **Add fee-rate metadata to `VerifyEntry`**: Store the declared fee rate (or a proxy such as `declared_cycles / tx_size`) at enqueue time.
2. **Implement fee-rate-based eviction in `add_tx`**: When the queue is full and the incoming transaction has a higher fee rate than the lowest-fee-rate entry currently in the queue, evict the worst entry and admit the new one — mirroring the main pool's `limit_size` / `next_evict_entry` logic.
3. **Enforce a minimum fee-rate check before enqueue**: Move a lightweight fee-rate check (using declared cycles and tx size) into `resumeble_process_tx` so that transactions below `min_fee_rate` are rejected before consuming queue space.

## Proof of Concept

1. Attacker opens a P2P connection to a target CKB node.
2. Attacker constructs N structurally valid transactions, each with serialized size near `TRANSACTION_SIZE_LIMIT`, with fake (non-existent) inputs but declaring exactly `min_fee_rate` in cycles. These pass `non_contextual_verify`.
3. Attacker relays all N transactions via the relay protocol, triggering `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue` → `add_tx` on the target node.
4. The VerifyQueue's `total_tx_size` reaches `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB).
5. A legitimate user submits a high-fee-rate transaction via `send_transaction` RPC. The node calls `notify_tx` → `resumeble_process_tx` → `enqueue_verify_queue` → `add_tx`. `is_full` returns `true`; the call returns `Reject::Full`. The legitimate transaction is dropped.
6. As workers drain the attacker's transactions (which fail at input resolution in `pre_check`), the attacker re-submits new ones to maintain saturation. The attacker is not banned because the failure mode (`Reject::Resolve` / missing input) does not trigger `ban_malformed`.

    /// Remove a tx from the queue
    pub fn remove_tx(&mut self, id: &ProposalShortId) -> Option<Entry> {
        self.inner.remove_by_id(id).map(|e| {
            let tx_size = e.inner.tx.data().serialized_size_in_block();
            if let Some(total_tx_size) = self.total_tx_size.checked_sub(tx_size) {
                self.total_tx_size = total_tx_size;
            } else if let Some(total_tx_size) = self.recompute_total_tx_size() {
                error!(
                    "verify_queue total_tx_size {} underflowed by sub {}, recomputed {}",
                    self.total_tx_size, tx_size, total_tx_size
                );
                self.total_tx_size = total_tx_size;
            } else {
                error!(
                    "verify_queue total_tx_size {} underflowed by sub {}, and recomputing overflowed",
                    self.total_tx_size, tx_size
                );
            }
            self.shrink_to_fit();
            e.inner
        })
    }

    /// Remove multiple txs from the queue
    pub fn remove_txs(&mut self, ids: impl Iterator<Item = ProposalShortId>) {
        for id in ids {
            self.remove_tx(&id);
        }
    }

    /// Remove multiple txs from the queue from a specified peer
    pub fn remove_txs_by_peer(&mut self, peer: &PeerIndex) {
        let ids: Vec<_> = self
            .inner
            .iter()
            .filter(|&(_cycle, entry)| entry.inner.remote.as_ref().is_some_and(|(_, p)| p == peer))
            .map(|(_cycle, entry)| entry.id.clone())
            .collect();

        self.remove_txs(ids.into_iter());
    }
