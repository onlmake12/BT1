### Title
Off-by-One Epoch Boundary in MMR Activation Check Causes Missing Chain Root Extension at RFC0044 Activation Epoch — (`tx-pool/src/block_assembler/mod.rs` and `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

The `build_extension` function in the block assembler and the `BlockExtensionVerifier` both check RFC0044 (MMR) activation using the **parent block's epoch number** instead of the **new block's epoch number**. This is an off-by-one at the epoch boundary: the first block of the RFC0044 activation epoch is assembled without the mandatory MMR chain root extension, and the verifier accepts it without the extension. The code itself contains an explicit comment acknowledging this as a known off-by-one that is preserved for consensus consistency.

---

### Finding Description

In `build_extension`:

```rust
pub(crate) fn build_extension(snapshot: &Snapshot) -> Result<Option<packed::Bytes>, AnyError> {
    let tip_header = snapshot.tip_header();
    // The use of the epoch number of the tip here leads to an off-by-one bug,
    // so be careful, it needs to be preserved for consistency reasons and not fixed directly.
    let mmr_activate = snapshot
        .consensus()
        .rfc0044_active(tip_header.epoch().number()); // ← uses TIP epoch, not new block's epoch
```

The new block being assembled is at `tip_header.number() + 1`. At an epoch boundary, the new block belongs to epoch `N+1` while the tip belongs to epoch `N`. The activation check `rfc0044_active(N)` evaluates to `false` when `N = RFC0044_ACTIVE_EPOCH - 1`, even though the new block is the first block of the activation epoch.

The `BlockExtensionVerifier` has the identical off-by-one:

```rust
let mmr_active = self
    .context
    .consensus
    .rfc0044_active(self.parent.epoch().number()); // ← uses parent epoch, not block's epoch
```

Because both the assembler and verifier use the parent's epoch number, they are mutually consistent — but both are wrong at the epoch boundary. The result is that the first block of the RFC0044 activation epoch (epoch 8651 on mainnet, epoch 5711 on testnet) is:

1. Assembled **without** the MMR chain root extension.
2. Accepted by all full nodes **without** the extension.
3. The extension content is not validated even if a miner includes one, because `mmr_active = false` skips the root hash check.

The `rfc0044_active` function:

```rust
pub fn rfc0044_active(&self, target: EpochNumber) -> bool {
    let rfc0044_active_epoch = match self.id.as_str() {
        mainnet::CHAIN_SPEC_NAME => softfork::mainnet::RFC0044_ACTIVE_EPOCH, // 8651
        testnet::CHAIN_SPEC_NAME => softfork::testnet::RFC0044_ACTIVE_EPOCH, // 5711
        _ => 0,
    };
    target >= rfc0044_active_epoch
}
```

When `target = 8650` (the tip's epoch at the boundary), `8650 >= 8651` is `false`, so the first block of epoch 8651 is treated as pre-activation.

---

### Impact Explanation

**Consensus layer:** All full nodes agree (both assembler and verifier have the same off-by-one), so there is no chain split among full nodes.

**Light client layer (RFC0044 scope):** RFC0044 light clients rely on the MMR chain root embedded in the block extension to verify the chain without downloading all headers. The first block of the activation epoch is missing a verified chain root. A light client that expects a valid MMR root starting at the activation epoch will encounter a block with either no extension or an unverified extension at that boundary. This breaks the security guarantee of the light client protocol for that specific block.

**Miner-exploitable:** Because `mmr_active = false` at the boundary, the verifier skips the chain root hash check even if an extension is present. A miner can include an **arbitrary, incorrect** 32-byte MMR root in the extension of the first activation-epoch block, and it will be accepted by all full nodes. Light clients that attempt to use this block's extension as a trust anchor will receive a forged root.

---

### Likelihood Explanation

This is not theoretical — it has already occurred on mainnet (epoch 8651) and testnet (epoch 5711). The code comment explicitly confirms the bug exists and was preserved. Any miner operating at the epoch boundary will produce a block without a valid MMR root, and any light client bootstrapping from that block will be affected. The entry path is the standard `get_block_template` RPC call made by any miner at the epoch boundary.

---

### Recommendation

The correct fix is to use the **new block's epoch number** for the activation check in both `build_extension` and `BlockExtensionVerifier`. The new block's epoch is already computed via `next_epoch_ext(tip_header, ...)` in the assembler. However, as the code comment notes, this fix requires a coordinated consensus change (hardfork), because changing the activation boundary for the extension check would alter which blocks are considered valid. A soft-fork or hardfork that re-specifies the activation boundary at `RFC0044_ACTIVE_EPOCH` (using the block's own epoch, not the parent's) would resolve the issue.

---

### Proof of Concept

**Scenario (mainnet):**

1. Chain reaches block `B` — the last block of epoch 8650 (tip epoch = 8650).
2. Miner calls `get_block_template`. `build_extension` evaluates `rfc0044_active(8650)` → `false`. Template is returned **without** the MMR extension.
3. Miner submits block `B+1` (first block of epoch 8651) without the extension.
4. `BlockExtensionVerifier` evaluates `rfc0044_active(parent.epoch().number())` = `rfc0044_active(8650)` → `false`. Extension is not required. Block is accepted.
5. Alternatively, miner includes an extension with an arbitrary 32-byte value. Since `mmr_active = false`, the root hash check is skipped. Block is accepted with a forged MMR root.
6. An RFC0044 light client that uses block `B+1`'s extension as a chain root anchor receives either no root or a forged root, breaking its security guarantee. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
