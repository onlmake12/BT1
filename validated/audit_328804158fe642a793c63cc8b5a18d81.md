Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Growth via Peer-Supplied `GetBlockProposal` Messages — (File: `sync/src/types/mod.rs`)

## Summary
`insert_get_block_proposals` inserts peer-supplied `ProposalShortId` values into the shared `pending_get_block_proposals: DashMap` with no size cap. Any unprivileged P2P peer can flood this map with fabricated proposal IDs absent from the tx pool, causing unbounded memory growth between periodic drain cycles. The drain itself clones the entire map, transiently doubling peak memory, and sustained flooding from multiple peers can exhaust node memory and crash the process.

## Finding Description
**Root cause — no size guard in `insert_get_block_proposals`:**

`GetBlockProposalProcess::execute()` validates only the per-message proposal count against `max_block_proposals_limit * max_uncles_num`. Proposals absent from the tx pool are forwarded unconditionally to `insert_get_block_proposals`:

```rust
// sync/src/relayer/get_block_proposal_process.rs L68-77
let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
    .into_iter()
    .filter(|short_id| !fetched_transactions.contains_key(short_id))
    .collect();
self.relayer.shared().state()
    .insert_get_block_proposals(self.peer, not_exist_proposals);
``` [1](#0-0) 

`insert_get_block_proposals` performs no size check before inserting:

```rust
// sync/src/types/mod.rs L1594-1600
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [2](#0-1) 

The map is declared as an unbounded `DashMap::new()`: [3](#0-2) 

The only relief valve is `drain_get_block_proposals`, which performs a full O(n) clone before clearing — doubling peak memory at every drain tick:

```rust
// sync/src/types/mod.rs L1586-1592
pub fn drain_get_block_proposals(&self) -> DashMap<...> {
    let ret = self.pending_get_block_proposals.clone(); // full clone
    self.pending_get_block_proposals.clear();
    ret
}
``` [4](#0-3) 

**Rate limiter is insufficient:**

The relayer applies a 30 req/s per-peer rate limit keyed by `(PeerIndex, message_item_id)`: [5](#0-4) [6](#0-5) 

This slows but does not prevent the attack. With 30 req/s × up to 3,000 proposals/req × N Sybil peers, the insertion rate scales linearly with peer count. The drain fires periodically, so the steady-state map size between drains grows proportionally to the number of attacker connections.

## Impact Explanation
Memory exhaustion leading to OOM kill of the CKB node process. This matches the allowed High impact: **"Vulnerabilities which could easily crash a CKB node"** (10001–15000 points). The `drain_get_block_proposals` clone creates a transient memory spike that can trigger OOM even before steady-state is reached. CPU time spent cloning and clearing the large DashMap also delays the relayer timer, allowing further accumulation — a positive feedback loop.

## Likelihood Explanation
- Requires only a standard P2P connection — no authentication, stake, or miner role.
- Fabricating `ProposalShortId` values absent from the tx pool is trivial (random 10-byte values).
- The 30 req/s rate limit per peer is easily bypassed by opening multiple Sybil connections; the effect multiplies linearly.
- No existing hard cap, eviction policy, or per-peer accounting guards `pending_get_block_proposals`.
- Analogous fixes were applied to related structures (`#3093`, `#3094`, `#3110`) but `pending_get_block_proposals` was not addressed.

## Recommendation
Add a hard cap inside `insert_get_block_proposals`:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    const MAX_PENDING_PROPOSALS: usize = 10_000;
    if self.pending_get_block_proposals.len() >= MAX_PENDING_PROPOSALS {
        return;
    }
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
```

Additionally, replace the full-clone drain with an atomic swap (e.g., `std::mem::take` on a `Mutex<HashMap>` or `DashMap::drain()`) to eliminate the transient memory doubling.

## Proof of Concept
1. Connect to a victim CKB node as a standard P2P peer (no special privileges required).
2. Repeatedly send `GetBlockProposal` relay messages containing the maximum allowed number of random `ProposalShortId` values (none present in the victim's tx pool) at 30 messages/second.
3. Open additional Sybil connections to multiply the insertion rate.
4. Observe `pending_get_block_proposals` growing without bound between drain ticks via RSS monitoring.
5. With sufficient connections, peak RSS climbs until OOM; the victim node crashes and cannot relay or confirm transactions.
