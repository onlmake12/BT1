### Title
Rich Indexer `script_exists_in_output` reads `row_lock` instead of `row_type` in second match arm, causing type-only scripts to be incorrectly deleted on rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

`script_exists_in_output` contains a copy-paste error: the second `match` arm (line 252) re-reads `row_lock` instead of `row_type`. Any script referenced **only** as a `type_script_id` in surviving outputs is therefore always reported as absent, causing it to be deleted from the `script` table during block rollback. Subsequent `get_cells` RPC calls that join on `script.id` silently return empty results for those cells.

### Finding Description

In `util/rich-indexer/src/indexer/remove.rs`, `script_exists_in_output` performs two SQL `EXISTS` queries:

- `row_lock` — checks `WHERE lock_script_id = $1` (lines 208–220)
- `row_type` — checks `WHERE type_script_id = $1` (lines 237–249)

The first match arm (line 223) correctly short-circuits on `row_lock`. But the second match arm at line 252 re-reads `row_lock` instead of `row_type`:

```rust
// line 252 — BUG: should be row_type.try_get::<bool, _>(0)
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

The `Err(_)` branch (SQLite path) accidentally uses `row_type`, but the `Ok(r)` branch (PostgreSQL path) returns the stale `row_lock` boolean. On both backends, when a script is **not** a lock script but **is** a type script, the function returns `false`, triggering deletion.

The caller in `rollback_block` unconditionally deletes any script for which `script_exists_in_output` returns `false`: [2](#0-1) 

### Impact Explanation

After any block rollback (natural reorg), every script that appears **only** as a `type_script_id` in surviving outputs is deleted from the `script` table. Because `get_cells` and related RPC endpoints join on `script.id`, those cells become invisible to callers — the indexer reports them as non-existent even though they are live on-chain. This is silent data corruption: no error is returned, just empty result sets.

### Likelihood Explanation

Reorgs are a normal, unprivileged network event. No attacker action is required; a natural 1-block reorg on mainnet is sufficient to trigger the bug for any node running the rich indexer with type-scripted outputs in the rolled-back block.

### Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// correct
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [3](#0-2) 

### Proof of Concept

1. Start a CKB node with the rich indexer enabled (SQLite or PostgreSQL).
2. Submit a transaction whose output has a `type_script` (and a different `lock_script`).
3. Append the block containing that transaction.
4. Roll back that block (trigger a reorg or call the internal rollback path).
5. Query `SELECT COUNT(*) FROM script WHERE id = <type_script_id>`.
6. Observe count = 0 — the type script was deleted despite the output still existing in surviving state.
7. Call `get_cells` filtering by that type script hash; observe an empty result set for a live cell.
