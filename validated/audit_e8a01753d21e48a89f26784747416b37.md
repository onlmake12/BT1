Audit Report

## Title
Non-Deterministic Orphan Expiry via Single-Entry HashMap Sampling in `need_clean` — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::need_clean` calls `map.iter().next()` on a `HashMap<packed::Byte32, LonelyBlockHash>` to decide whether to evict all blocks sharing a `parent_hash`. Because Rust's `HashMap` iteration order is non-deterministic, an attacker who injects a crafted block (no PoW required) sharing a `parent_hash` with a legitimate orphan but carrying a heterogeneous `epoch_number` can cause either premature eviction of the legitimate orphan or indefinite on-disk retention of the crafted block. Both outcomes constitute a suboptimal CKB state storage mechanism defect exploitable by any unprivileged P2P peer.

## Finding Description

**Root cause — single arbitrary sample in `need_clean`:**

`InnerPool::need_clean` retrieves only the first iterator entry from the inner `HashMap`:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← arbitrary sample
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
``` [1](#0-0) 

The inner structure explicitly allows multiple blocks per `parent_hash`:

```rust
blocks: HashMap<ParentHash, HashMap<packed::Byte32, LonelyBlockHash>>,
``` [2](#0-1) 

**Attacker injection path — no PoW required:**

`asynchronous_process_block` calls `non_contextual_verify`, which runs only `BlockVerifier` and `NonContextualBlockTxsVerifier`: [3](#0-2) 

`BlockVerifier` checks only proposals limit, block bytes, cellbase structure, duplicates, and merkle root — it does **not** check PoW or epoch continuity: [4](#0-3) 

PoW and epoch continuity are only checked in `HeaderVerifier`, which is contextual and requires the parent header from the chain store: [5](#0-4) 

`epoch_number` stored in `LonelyBlockHash` is taken directly from the block header with no prior validation: [6](#0-5) 

After passing `non_contextual_verify`, the block is written to disk and inserted into the orphan pool: [7](#0-6) 

**Trigger — 60-second cleanup timer:**

`clean_expired_orphans` fires every 60 seconds, fetches `tip_epoch_number`, and calls `clean_expired_blocks`: [8](#0-7) 

`clean_expired_blocks` iterates leaders and calls `need_clean` per leader, then calls `remove_blocks_by_parent` if it returns `true` — evicting the entire sibling group: [9](#0-8) 

**Cleanup path — both blocks deleted or neither:**

When `need_clean` returns `true`, `clean_expired_orphans` deletes the block from disk, removes its header view, and removes its block status — affecting every sibling under that `parent_hash`: [10](#0-9) 

## Impact Explanation

**Medium (2001–10000 points) — Suboptimal implementation of CKB state storage mechanism.**

Two concrete outcomes depending on which HashMap entry is sampled:

| HashMap seed picks | `need_clean` returns | Outcome |
|---|---|---|
| Crafted block (expired epoch) | `true` | Legitimate orphan evicted; raw block data deleted from store, header view and block status stripped; sync stalls until re-requested |
| Legitimate block (non-expired epoch) | `false` | Crafted block and its on-disk record retained indefinitely; `block_status_map` entry never reclaimed; storage and memory consumed permanently |

In the indefinite-retention case, an attacker flooding many crafted blocks with future `epoch_number` values under distinct `parent_hash` values causes unbounded growth of both the on-disk unverified block store and the in-memory `block_status_map`, as `clean_expired_blocks` will never evict them. This is a concrete suboptimal CKB state storage mechanism defect.

## Likelihood Explanation

- The attacker is an unprivileged P2P sync peer; no key or privilege is required.
- Crafting a block with an arbitrary `epoch_number` requires only constructing a valid `BlockBuilder` payload with correct cellbase `since` = block number, valid merkle root, and any desired epoch field — no PoW solution is needed.
- The target `parent_hash` can be any hash not present in the chain store (random hashes suffice for the indefinite-retention variant); for the premature-eviction variant, the attacker observes a known orphan's `parent_hash` via P2P.
- The attack is repeatable and amplifiable: the peer can flood the node with many such crafted blocks.
- The 60-second cleanup timer provides a reliable trigger window.

## Recommendation

Replace the single-entry sample with a check over **all** entries under the parent. Evict only when every sibling block is expired:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            !map.is_empty()
                && map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
        })
        .unwrap_or_default()
}
```

Alternatively, track the minimum `epoch_number` per `parent_hash` group at insertion time to avoid iterating all entries on every cleanup tick.

    pub fn clean_expired_blocks(&mut self, tip_epoch: EpochNumber) -> Vec<LonelyBlockHash> {
        let mut result = vec![];

    pub(crate) fn clean_expired_orphans(&self) {
        debug!("clean expired orphans");
        let tip_epoch_number = self
            .shared
            .store()
            .get_tip_header()
            .expect("tip header")
            .epoch()
            .number();
        let expired_orphans = self
            .orphan_blocks_broker
            .clean_expired_blocks(tip_epoch_number);
        for expired_orphan in expired_orphans {
            self.delete_block(&expired_orphan);
            self.shared.remove_header_view(&expired_orphan.hash());
            self.shared.remove_block_status(&expired_orphan.hash());
            info!(
                "cleaned expired orphan: {}-{}",
                expired_orphan.number(),
                expired_orphan.hash()
            );
        }
    }
