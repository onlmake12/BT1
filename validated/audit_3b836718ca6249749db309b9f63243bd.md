### Title
Missing Signer Verification on `owner` Account in `create_position` Instruction Allows Unauthorized Position Creation — (`governance/pyth_staking_sdk/src/idl/staking.json`)

---

### Summary

The staking program's instruction that creates staking positions verifies the `owner` account key against the stored value in `stake_account_metadata`, but does **not** enforce that `owner` is a transaction signer. When the optional `pool_authority` signer is omitted, the instruction has **no signer requirement at all**. An unprivileged attacker can force-create staking positions on behalf of any user, locking their tokens without consent.

---

### Finding Description

In the staking program IDL, the instruction following `join_dao_llc` (contextually `create_position`) defines its accounts as follows:

```json
{
  "docs": [
    "CHECK : This AccountInfo is safe because it's checked against stake_account_metadata"
  ],
  "name": "owner",
  "relations": ["stake_account_metadata"],
  "writable": true
},
...
{
  "name": "pool_authority",
  "optional": true,
  "signer": true
}
``` [1](#0-0) 

The `owner` account has:
- `relations: ["stake_account_metadata"]` — Anchor enforces **key equality** only (the account key must match `stake_account_metadata.owner`)
- **No** `signer: true` constraint

The `pool_authority` account is `optional` and `signer: true`. When it is not provided (`None`), the Anchor framework does not enforce any signer at all, because `owner` has no signer constraint.

The developer comment — *"CHECK : This AccountInfo is safe because it's checked against stake_account_metadata"* — explicitly confirms that only key equality is verified, not signing authority. This is the exact same reasoning flaw as the Phantasia Sports bug: checking `sell_order_data.seller_wallet != *seller_wallet_account.key` without checking `seller_wallet_account.is_signer`.

Compare to `withdraw_stake`, which correctly enforces both:

```json
{
  "name": "owner",
  "signer": true,
  "relations": ["stakeAccountMetadata"]
}
``` [2](#0-1) 

The `withdraw_stake` instruction has `signer: true` on `owner`. The `create_position` instruction does not.

---

### Impact Explanation

An attacker who knows any user's `stake_account_positions` address (publicly derivable from the user's wallet) can:

1. Call `create_position` without providing `pool_authority` (omitting the optional signer).
2. Pass the victim's correct `owner` pubkey and `stake_account_metadata` PDA — satisfying the `relations` key-equality check without signing.
3. Force-create a staking position that locks the victim's tokens in `stake_account_custody` into a governance or OIS staking target.

The victim's tokens become locked for the cooldown period. They cannot call `withdraw_stake` until they first close the forced position and wait out the unbonding window. An attacker can repeatedly re-create positions as the victim closes them, causing a sustained denial-of-service against any staking user's ability to withdraw funds.

**Impact: High** — user funds are locked without consent; withdrawal is blocked for the cooldown duration.

---

### Likelihood Explanation

- The entry point is a public, permissionless instruction.
- No privileged key, governance majority, or leaked secret is required.
- The target's `stake_account_positions` address is a PDA derivable from their wallet pubkey — fully public.
- The attacker only needs to submit a valid transaction with the victim's accounts and omit `pool_authority`.

**Likelihood: High** — trivially reachable by any on-chain transaction sender.

---

### Recommendation

Add `signer: true` to the `owner` account constraint in the `create_position` instruction, matching the pattern already used in `withdraw_stake` and `close_position`:

```rust
#[account(
    mut,
    constraint = owner.key() == stake_account_metadata.owner,
)]
pub owner: Signer<'info>,
```

Alternatively, if `pool_authority` is intended as a delegated signer path, enforce that **at least one** of `owner` (as signer) or `pool_authority` (as signer) must be present — not that both are optional simultaneously.

---

### Proof of Concept

1. Alice has a stake account with 1000 PYTH tokens in `stake_account_custody`, no active positions.
2. Attacker derives Alice's `stake_account_positions` PDA from her wallet pubkey.
3. Attacker submits `create_position` with:
   - `owner` = Alice's pubkey (satisfies `relations` key check, no signer required)
   - `stake_account_metadata` = Alice's metadata PDA
   - `pool_authority` = omitted (optional)
   - `target_with_parameters` = `{ voting: {} }` (governance staking)
   - `amount` = 1000 PYTH
4. Transaction succeeds — Alice's 1000 PYTH are now locked in a governance position she did not create.
5. Alice attempts `withdraw_stake` → fails because tokens are staked.
6. Alice calls `close_position` → must wait the full cooldown period before tokens are unlocked.
7. Attacker repeats step 3 after cooldown, indefinitely blocking Alice's withdrawals. [3](#0-2)
