### Title
Aptos Accumulator Parser Uses Hardcoded Major Version Literal Instead of Named Constant - (File: target_chains/aptos/contracts/sources/pyth.move)

### Summary

The Aptos Pyth contract's `parse_and_verify_accumulator_message` function validates the accumulator message major version by comparing against the hardcoded literal `1` rather than a named constant. Every other Pyth chain implementation (Ethereum, Sui, Fuel, TON) defines and uses a `MAJOR_VERSION` constant for this check. If the protocol ever increments the major version, the Aptos contract would be the only chain silently rejecting all valid price updates.

### Finding Description

In `target_chains/aptos/contracts/sources/pyth.move`, `parse_and_verify_accumulator_message` checks the accumulator header major version with a bare literal:

```move
let major = deserialize::deserialize_u8(cursor);
assert!(major == 1, error::invalid_accumulator_payload());
``` [1](#0-0) 

The file defines only two module-level constants:

```move
const PYTHNET_ACCUMULATOR_UPDATE_MAGIC: u64 = 1347305813;
const ACCUMULATOR_UPDATE_WORMHOLE_VERIFICATION_MAGIC: u64 = 1096111958;
``` [2](#0-1) 

There is no `MAJOR_VERSION` constant defined anywhere in the Aptos contract. By contrast, every other chain defines and uses one:

- **Ethereum**: `uint8 constant MAJOR_VERSION = 1;` checked as `if (majorVersion != MAJOR_VERSION)` [3](#0-2) 
- **Sui**: `const MAJOR_VERSION: u8 = 1;` checked as `assert!(major == MAJOR_VERSION, ...)` [4](#0-3) 
- **Fuel**: `const MAJOR_VERSION = 1;` checked as `major_version.unwrap() == MAJOR_VERSION` [5](#0-4) 
- **TON**: `const int MAJOR_VERSION = 1;` checked as `major_version == MAJOR_VERSION` [6](#0-5) 

The same Aptos function also hardcodes `0` for the WormholeMerkle update type and PriceFeedMessage type, where no named constants exist:

```move
assert!(wh_message_payload_type == 0, error::invalid_wormhole_message()); // Merkle variant
...
assert!(message_type == 0, error::invalid_accumulator_message()); // PriceFeedMessage variant
...
assert!(proof_type == 0, error::invalid_accumulator_payload());
``` [7](#0-6) [8](#0-7) [9](#0-8) 

### Impact Explanation

If the Pyth protocol increments the accumulator major version (e.g., to `2`) as part of a protocol upgrade, all other chain contracts would accept the new format because they reference the `MAJOR_VERSION` constant that would be updated. The Aptos contract would continue to reject every accumulator update with `error::invalid_accumulator_payload()`, silently breaking all price feed updates on Aptos. Any DeFi protocol on Aptos consuming Pyth prices would receive stale data or revert, depending on their staleness checks.

### Likelihood Explanation

A major version bump is a realistic protocol upgrade event. The inconsistency is already present across chains — Aptos is the only chain without the constant — making it likely to be missed during a coordinated upgrade. The entry path requires no special privileges: any unprivileged caller submitting a valid accumulator update with the new major version triggers the rejection.

### Recommendation

Define named constants in the Aptos contract mirroring the other chain implementations and replace all hardcoded literals with them:

```move
const MAJOR_VERSION: u8 = 1;
const WORMHOLE_MERKLE_UPDATE_TYPE: u8 = 0;
const PRICE_FEED_MESSAGE_TYPE: u8 = 0;
```

Then replace:
- `assert!(major == 1, ...)` → `assert!(major == MAJOR_VERSION, ...)`
- `assert!(wh_message_payload_type == 0, ...)` → `assert!(wh_message_payload_type == WORMHOLE_MERKLE_UPDATE_TYPE, ...)`
- `assert!(message_type == 0, ...)` → `assert!(message_type == PRICE_FEED_MESSAGE_TYPE, ...)`
- `assert!(proof_type == 0, ...)` → `assert!(proof_type == WORMHOLE_MERKLE_UPDATE_TYPE, ...)`

### Proof of Concept

1. Pyth protocol upgrades accumulator format to major version 2 and updates `MAJOR_VERSION = 2` in Ethereum, Sui, Fuel, and TON contracts.
2. Aptos contract is not updated (the literal `1` is not a named constant, so there is no single place to update).
3. A price updater calls `update_price_feeds_with_funder` on Aptos with a valid v2 accumulator message.
4. `parse_and_verify_accumulator_message` reads `major = 2`, hits `assert!(major == 1, ...)`, and aborts with `error::invalid_accumulator_payload()`.
5. All Aptos price feeds become permanently stale while every other chain continues to update normally. [10](#0-9)
