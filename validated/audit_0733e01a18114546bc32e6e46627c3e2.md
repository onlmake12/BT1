### Title
Missing Maximum Fee Guard and No Excess Refund in Entropy Request Functions — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

All Entropy randomness request entry points (`request`, `requestWithCallback`, `requestV2` and its overloads) route through `requestHelper()`, which accepts `msg.value` as the fee but provides no `maxFee` parameter and permanently captures every wei of excess value. Because provider fees are mutable and can be raised at any time by a permissionlessly registered provider, a user's pending transaction can be griefed into a revert, and any user who defensively over-sends ETH to avoid that revert loses the surplus permanently to the Pyth protocol fee pool.

---

### Finding Description

`requestHelper()` performs the following fee accounting:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire surplus `msg.value - providerFee` is credited to `accruedPythFeesInWei` — it is never returned to the caller. This is explicitly documented in `IEntropyV2.sol`:

> "Note that the fee can change over time … Further note that excess value is *not* refunded to the caller." [2](#0-1) 

Provider fees are mutable via `setProviderFee()` (callable by any registered provider for their own entry) and `setProviderFeeAsFeeManager()`: [3](#0-2) 

Provider registration is permissionless — anyone can call `register()`: [4](#0-3) 

None of the public `requestV2` overloads accept a `maxFee` parameter: [5](#0-4) 

---

### Impact Explanation

Two concrete harms arise:

**1. Griefing / DoS via fee front-run.**
A user reads `getFeeV2(provider, gasLimit)` off-chain, constructs a transaction with `msg.value = fee`, and submits it. Before the transaction is mined, the provider calls `setProviderFee(fee + 1)`. The user's transaction reverts with `InsufficientFee`. The provider can repeat this indefinitely, permanently blocking a specific user or contract from obtaining randomness from that provider at no cost to the attacker beyond gas.

**2. Permanent loss of excess ETH.**
A user (or integrating smart contract) who sends `msg.value > requiredFee` — a common defensive pattern to avoid the revert above — loses the surplus permanently. The surplus is credited to `accruedPythFeesInWei` and is only withdrawable by the Pyth admin, not the caller. [6](#0-5) 

---

### Likelihood Explanation

Provider registration is permissionless. Any actor can register as a provider and attract users (e.g., by advertising a low fee), then repeatedly front-run their requests. For the zero-argument `requestV2()` variant, the default provider is admin-controlled, limiting that specific vector; however, the three-argument `requestV2(provider, userRandomNumber, gasLimit)` and `requestWithCallback(provider, ...)` variants expose every registered provider as a potential griefing actor against users who explicitly chose them. Integrating smart contracts that hardcode a provider address are particularly vulnerable because they cannot switch providers in response to griefing. [7](#0-6) 

---

### Recommendation

1. **Add a `maxFee` parameter** to all public request functions. Revert if `requiredFee > maxFee`. This gives callers a slippage-like bound analogous to AMM `amountOutMin` guards.

2. **Refund excess `msg.value`** to `msg.sender` after deducting `requiredFee`, rather than crediting the surplus to `accruedPythFeesInWei`. The current design punishes defensive over-payment.

3. Optionally, **add a `deadline` parameter** so that a pending request transaction automatically reverts if it is not mined within the caller's acceptable window, preventing stale requests from executing at an unexpectedly higher fee after a long mempool delay.

---

### Proof of Concept

```
1. Alice deploys a consumer contract that calls requestV2(providerEve, userRandom, 0)
   with msg.value = getFeeV2(providerEve, 0) = 1000 wei.

2. Eve (providerEve) monitors the mempool and sees Alice's pending transaction.

3. Eve calls setProviderFee(1001) — a single cheap transaction.

4. Alice's transaction executes: requiredFee = 1001 > msg.value = 1000 → revert InsufficientFee.

5. Eve calls setProviderFee(1000) to restore the original fee.

6. Alice retries with msg.value = 1001 to be safe.

7. Eve repeats step 3 with setProviderFee(1002), causing another revert.

8. If Alice instead sends msg.value = 10000 to guarantee success, and Eve does not front-run,
   Alice's request succeeds but 9000 wei (= 10000 - providerFee) is permanently credited
   to accruedPythFeesInWei and is unrecoverable by Alice.
``` [8](#0-7) [9](#0-8)

    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
