### Title
Excess `msg.value` Is Permanently Absorbed Into Protocol Fees Instead of Being Refunded to Callers â€” (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `requestHelper()` uses a `<` (less-than) inequality to validate the fee, allowing callers to overpay. Any excess ETH above the required fee is silently credited to `accruedPythFeesInWei` rather than refunded. The contract's own NatSpec explicitly acknowledges this: *"Note that excess value is not refunded to the caller."* This is the direct analog of the Hinkal `>=` vs `==` balance-check vulnerability.

---

### Finding Description

In `requestHelper()`, the fee gate is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The check `msg.value < requiredFee` only rejects underpayment. When `msg.value > requiredFee`, the entire surplus is swept into `accruedPythFeesInWei`:

- `providerFee` is credited to the provider (correct, fixed amount).
- `msg.value - providerFee` is credited to Pyth fees â€” this includes `pythFeeInWei` (correct) **plus any excess** (incorrect).

No refund path exists. The same inequality is present in all three public entry points â€” `request()`, `requestWithCallback()`, and `requestV2()` â€” all of which route through `requestHelper()`. [2](#0-1) [3](#0-2) 

The same pattern exists in `Echo.sol`'s `requestPriceUpdatesWithCallback()`, where excess above `pythFeeInWei` is stored in `req.fee` and later credited entirely to the provider rather than refunded:

```solidity
if (msg.value < requiredFee) revert InsufficientFee();
...
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [4](#0-3) 

---

### Impact Explanation

Any caller who sends `msg.value > getFeeV2(provider, gasLimit)` permanently loses the excess ETH. The surplus is irrecoverably credited to `accruedPythFeesInWei` (controlled by the Pyth admin) and is never returned to the user. This constitutes a direct, unrecoverable financial loss for the caller with no compensating benefit.

---

### Likelihood Explanation

This is realistically triggered by unprivileged users in at least two ways:

1. **Fee estimation buffer:** Callers commonly add a small ETH buffer when calling payable functions to avoid reverts from minor fee fluctuations. Every such call silently donates the buffer to Pyth.
2. **Provider fee front-run:** A provider can call `setProviderFee()` to lower their fee between the moment a user calls `getFeeV2()` off-chain and the moment the user's transaction lands on-chain. The user's `msg.value` was computed against the old (higher) fee; the new lower fee means the excess is absorbed.

Both scenarios require no privileged access and are reachable by any unprivileged Entropy user.

---

### Recommendation

Replace the inequality check with an equality check, or add an explicit refund of excess ETH:

```solidity
// Option A: strict equality
if (msg.value != requiredFee) revert EntropyErrors.IncorrectFee();

// Option B: refund excess
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

Apply the same fix to `Echo.sol`'s `requestPriceUpdatesWithCallback()`.

---

### Proof of Concept

1. Provider registers with `feeInWei = 1000 wei`; Pyth fee is `100 wei`; `getFeeV2()` returns `1100 wei`.
2. User calls `requestV2{value: 2000}(provider, 0)` â€” sending `900 wei` excess.
3. Inside `requestHelper()`:
   - `requiredFee = 1100`, `msg.value = 2000` â†’ check passes.
   - `providerFee = 1000` â†’ `providerInfo.accruedFeesInWei += 1000` âœ“
   - `accruedPythFeesInWei += (2000 - 1000) = 1000` â€” Pyth receives `1000` instead of `100`.
4. User's `900 wei` excess is permanently locked in `accruedPythFeesInWei`. No refund is issued. [5](#0-4)
