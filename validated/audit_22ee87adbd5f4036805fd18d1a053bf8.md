### Title
Unprivileged Caller Can Permanently Brick `InitializePublisher` by Setting Uncontrollable Authority in `initialize` — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary

The `initialize` instruction accepts a caller-supplied `args.authority: [u8; 32]` with no validation that the value corresponds to a controllable key. Any unprivileged account can race the deployer, call `Initialize` first, write `[0u8; 32]` (or any uncontrollable pubkey) as the authority, and permanently prevent `InitializePublisher` from ever succeeding, because the config PDA is a write-once singleton.

### Finding Description

**Entrypoint — `initialize`:**

The `initialize` function validates only that the payer is a signer and the config PDA is correct. It places `args.authority` directly into the config account with zero constraints on its value: [1](#0-0) [2](#0-1) 

`validate_payer` only checks `is_signer` and `is_writable` — it does **not** check that the payer is the program upgrade authority or any privileged role: [3](#0-2) 

**Write-once singleton — `accounts::config::create`:**

The config account is a singleton PDA. Once written, `create` rejects any re-initialization attempt via the `AlreadyInitialized` guard: [4](#0-3) 

**Downstream lock — `validate_authority`:**

Every call to `InitializePublisher` passes through `validate_authority`, which requires the submitted account to both be a signer **and** match the stored `config.authority` bytes exactly: [5](#0-4) 

If `config.authority` is `[0u8; 32]`, no valid Ed25519 keypair can satisfy this check, so `InitializePublisher` is permanently blocked.

**No recovery path exists.** There is no `UpdateAuthority`, `Reinitialize`, or admin-reset instruction anywhere in the program: [6](#0-5) 

### Impact Explanation

- `InitializePublisher` is permanently bricked — no publisher can ever be registered.
- `SubmitPrices` depends on a publisher config that can only be created via `InitializePublisher`, so price submission is also permanently blocked for all new publishers.
- Recovery requires a program upgrade (redeployment), which is an operational emergency.

### Likelihood Explanation

The attack requires only a funded Solana account and knowledge of the program ID. It is a classic front-running / initialization race: the window exists between program deployment and the deployer's own `Initialize` call. An attacker monitoring the chain for new program deployments can automate this. No privileged access, leaked keys, or governance majority is needed.

### Recommendation

1. **Require the authority to be a signer of the `Initialize` transaction.** Add a check in `initialize` that `args.authority == payer.key.to_bytes()`, or add the authority as an explicit signer account and validate it with `is_signer`.
2. **Alternatively**, derive the authority from the program upgrade authority (BPFLoader's upgrade authority) so only the deployer can set it.
3. **Add an `UpdateAuthority` instruction** gated on the current authority's signature to allow recovery from misconfiguration.

### Proof of Concept

```rust
// 1. Attacker calls Initialize with authority = [0u8; 32]
let mut data = vec![Instruction::Initialize as u8, config_bump];
data.extend_from_slice(&[0u8; 32]); // authority = all-zeros

// 2. Config PDA is now permanently locked to uncontrollable key.

// 3. Legitimate deployer's Initialize call now returns AlreadyInitialized.

// 4. Any InitializePublisher call returns MissingRequiredSignature
//    because no one can sign as pubkey([0u8; 32]).
```

The existing test in `initialize.rs` (lines 75–76) already demonstrates that `authority` is accepted as arbitrary bytes — it sets `authority[..3] = [1,2,3]` with no signer check on that value, confirming the missing guard. [7](#0-6)

```rust
pub fn validate_payer<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,

    let payer = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, payer.is_signer);
    ensure!(ProgramError::InvalidArgument, payer.is_writable);
    Ok(payer)
}
```

```rust
pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)
```
