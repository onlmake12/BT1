### Title
Excess ETH Sent to `Entropy.requestHelper()` Is Permanently Captured as Protocol Fees Instead of Being Refunded to the Caller тАФ (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper()` internal function in `Entropy.sol` checks that `msg.value >= requiredFee` but never refunds any overpayment. Instead, the entire surplus above `providerFee` is silently credited to `_state.accruedPythFeesInWei`. Any user who sends more ETH than the exact `getFeeV2()` amount permanently loses the excess to the Pyth protocol treasury with no mechanism for recovery.

---

### Finding Description

`requestHelper()` performs the following fee accounting:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);   // providerFee + pythFeeInWei
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The Pyth fee credited to `accruedPythFeesInWei` is computed as `msg.value - providerFee`, **not** the fixed `_state.pythFeeInWei`. When `msg.value > requiredFee`, the excess `(msg.value - requiredFee)` is silently absorbed into the Pyth treasury rather than returned to `msg.sender`. There is no refund path anywhere in `requestHelper`, `requestV2`, `request`, or `requestWithCallback`.

`getFeeV2` confirms the intended fee is exactly `providerFee + pythFeeInWei`:

```solidity
function getFeeV2(address provider, uint32 gasLimit) public view override returns (uint128 feeAmount) {
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
``` [2](#0-1) 

The interface documentation acknowledges this behavior explicitly:

> "Further note that excess value is *not* refunded to the caller." [3](#0-2) 

The same note appears inline in `Entropy.sol` for both `request()` and `requestWithCallback()`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

Any caller of `requestV2`, `request`, or `requestWithCallback` who sends `msg.value` even 1 wei above `getFeeV2()` permanently loses that excess. The funds are not locked in a dead address тАФ they are credited to `accruedPythFeesInWei` and become withdrawable only by the Pyth admin, not the original sender. From the user's perspective the loss is irreversible.

Realistic overpayment scenarios:
- Provider fee changes between the block where the user reads `getFeeV2()` and the block where the transaction is mined.
- Integrator contracts that add a small buffer (e.g., `getFeeV2() * 110 / 100`) to avoid reverts.
- Front-ends that hardcode a fee value that becomes stale.

---

### Likelihood Explanation

The interface itself warns callers that fees can change over time and instructs them to recompute `getFeeV2()` before every call. This implies the protocol team anticipates fee volatility. Any integrator that adds a safety margin to avoid `InsufficientFee` reverts will routinely overpay and lose the excess. The entry path requires no privilege тАФ any EOA or contract can call `requestV2`.

---

### Recommendation

Refund any excess ETH to `msg.sender` at the end of `requestHelper`, analogous to the pattern already used in `PythLazer.sol`:

```solidity
// In requestHelper, after fee accounting:
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
// Credit only the fixed pythFeeInWei, not the full surplus:
_state.accruedPythFeesInWei += (requiredFee - providerFee); // == pythFeeInWei
``` [6](#0-5) 

---

### Proof of Concept

1. Provider fee is currently `0.001 ETH`; Pyth fee is `0.0001 ETH`; `getFeeV2()` returns `0.0011 ETH`.
2. An integrator contract calls `requestV2(provider, gasLimit)` with `msg.value = 0.002 ETH` (a 1.8├Ч buffer to avoid reverts if fees rise).
3. `requestHelper` executes:
   - `requiredFee = 0.0011 ETH`
   - `msg.value (0.002) >= requiredFee` тЖТ no revert
   - `providerInfo.accruedFeesInWei += 0.001 ETH`
   - `_state.accruedPythFeesInWei += (0.002 - 0.001) = 0.001 ETH` тЖР captures `0.0009 ETH` excess
4. The integrator's `0.0009 ETH` overpayment is permanently transferred to the Pyth treasury. The user has no recourse. [7](#0-6)
