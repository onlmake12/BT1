Audit Report

## Title
Cheap Block-Byte Stuffing via Oversized Low-Cycle Transactions — (`tx-pool/src/util.rs`, `util/types/src/core/tx_pool.rs`, `spec/src/consensus.rs`)

## Summary
A single transaction up to 512,000 bytes with a trivial lock script can consume ~86% of a block's byte budget (`MAX_BLOCK_BYTES = 597,000`) while exhausting only ~0.03% of the cycles budget (`MAX_BLOCK_CYCLES = 3,500,000,000`). Because the minimum fee gate is byte-proportional only and the fee rate is a static node-local config, the cost to sustain this attack is ~512,000 shannons (~0.005 CKB) per block with no economic feedback loop to deter the attacker.

## Finding Description
**Block and transaction size constants:**
`MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT = 597 × 1,000 = 597,000 bytes` (`spec/src/consensus.rs` L72–83). The tx-pool enforces `TRANSACTION_SIZE_LIMIT = 512 * 1_000 = 512,000 bytes` as the per-transaction ceiling (`util/types/src/core/tx_pool.rs` L305–309), and `non_contextual_verify` rejects anything above that limit but accepts anything below it (`tx-pool/src/util.rs` L67–73).

**Fee gate is byte-only:**
`check_tx_fee` computes `min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64)` — raw serialized size, no cycles component (`tx-pool/src/util.rs` L44–52). The code comment itself acknowledges: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check."* The default `min_fee_rate = 1,000 shannons/KB` is a static node-local config value (`resource/ckb.toml` L212).

**Weight formula does not impose a cycles surcharge:**
`get_transaction_weight` returns `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)` (`util/types/src/core/tx_pool.rs` L298–303). For a trivial lock script consuming ~1,000,000 cycles: `1,000,000 × 0.000_170_571 ≈ 170 bytes`, so `weight = max(512,000, 170) = 512,000`. The weight collapses to the byte size, meaning no cycles-based fee premium applies.

**Block assembler stops on byte exhaustion:**
`TxSelector.txs_to_commit` breaks out of the selection loop when `next_cycles > cycles_limit || next_size > size_limit` (`tx-pool/src/component/tx_selector.rs` L149–162). After a 512 KB transaction is selected, only ~83 KB of byte budget remains, leaving the 3.5 billion cycles budget almost entirely unused.

**Byte limit is a hard consensus rule:**
`BlockBytesVerifier` enforces the byte limit at consensus level — it is not merely a miner preference (`verification/src/block_verifier.rs` L251–262).

**No dynamic fee pressure:**
There is no EIP-1559-style base-fee adjustment. Repeated byte-stuffed blocks do not raise the fee floor, so the attacker's cost per block never increases.

## Impact Explanation
This matches the **High** impact category: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* An attacker can continuously submit 512 KB transactions (large witness, trivial always-success lock) to ensure every block is byte-full but cycles-empty. Legitimate transactions are delayed indefinitely. Time-sensitive operations — oracle updates, DAO withdrawals, contract interactions with deadlines — are blocked. The cost is ~512,000 shannons (~0.005 CKB) per block, which at any realistic CKB price is negligible.

## Likelihood Explanation
The attack requires no special privilege: any account with a small CKB balance can submit transactions via the public `send_transaction` RPC. The technique is straightforward — craft a transaction with a large witness blob and a minimal lock script. The static fee floor means there is no economic feedback loop to deter the attacker. The gap between `TRANSACTION_SIZE_LIMIT` (512 KB) and `MAX_BLOCK_BYTES` (597 KB) is intentional for packing flexibility but inadvertently enables this attack. The attack is fully repeatable every block.

## Recommendation
1. **Reduce `TRANSACTION_SIZE_LIMIT`** to a value that ensures no single transaction can consume a dominant fraction of the block's byte budget (e.g., cap at 64 KB or 10% of `MAX_BLOCK_BYTES`).
2. **Introduce a cycles-floor per byte**: require that a transaction's declared cycles be at least proportional to its serialized size, so that a large-byte transaction must also consume a meaningful share of the cycles budget, making it expensive to waste both resources simultaneously.
3. **Dynamic fee-rate adjustment**: implement a protocol-level mechanism analogous to EIP-1559 that raises the minimum fee rate when recent blocks are consistently byte-full but cycles-empty, creating economic pressure against byte-stuffing attacks.

## Proof of Concept
1. Deploy an always-success lock script cell on-chain (standard devnet fixture).
2. Construct a transaction spending one cell locked by that script, with a single output returning capacity to the same lock, and a witness field padded to ~511,900 bytes of arbitrary data.
3. Verify serialized size: ~512,000 bytes — below `TRANSACTION_SIZE_LIMIT`, accepted by the pool.
4. Pay minimum fee: `1,000 shannons/KB × 512 KB = 512,000 shannons`.
5. Submit via `send_transaction` RPC. The transaction enters the pool, is proposed in the next proposal window, and committed in the following block.
6. After commitment, the block's byte usage is ~514,000 bytes out of 597,000 — leaving only ~83,000 bytes for all other transactions.
7. Cycles used: ~1,000,000 out of 3,500,000,000 — 0.03% utilization.
8. Repeat every block. Cost: ~512,000 shannons/block ≈ 0.005 CKB/block.
