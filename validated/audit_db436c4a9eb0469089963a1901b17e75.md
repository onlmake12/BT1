### Title
`TxSelector::txs_to_commit` Premature Loop Termination Due to Non-Resetting Failure Counter Excludes Valid Transactions from Block Templates — (`File: tx-pool/src/component/tx_selector.rs`)

---

### Summary

`TxSelector::txs_to_commit` uses a counter named `consecutive_failed` that is **never reset to zero on a successful transaction inclusion**. It accumulates every failure across the entire selection loop. Once it exceeds `MAX_CONSECUTIVE_FAILURES = 4000`, the loop breaks unconditionally. This means the block-template assembly algorithm can terminate early — leaving valid, fee-paying transactions unincluded — even when the block is far from full and valid candidates remain in the pool.

---

### Finding Description

In `tx-pool/src/component/tx_selector.rs`, the `txs_to_commit` function drives block-template transaction selection:

```
let mut consecutive_failed = 0;          // line 104 — initialized once, never reset
```

Two distinct failure paths both increment this counter:

**Path 1 — size/cycles overflow** (lines 152–162):
```rust
if next_cycles > cycles_limit || next_size > size_limit {
    consecutive_failed += 1;
    ...
    if consecutive_failed > MAX_CONSECUTIVE_FAILURES { break; }
    continue;
}
```

**Path 2 — ancestor not in proposed pool** (lines 176–189):
```rust
if ancestors_ids.iter().any(|id| !self.pool_map.has_proposed(id)) {
    ...
    consecutive_failed += 1;
    if consecutive_failed > MAX_CONSECUTIVE_FAILURES { break; }
    continue;
}
```

On a **successful** inclusion (lines 207–218), `consecutive_failed` is **never reset**. The variable name implies "consecutive" failures, but the implementation counts **total** failures across the entire run.

The pre-loop filter (lines 109–111) compounds this:

```rust
.filter(|entry| {
    entry.ancestors_size <= size_limit && entry.ancestors_cycles <= cycles_limit
})
```

`size_limit` and `cycles_limit` here are the **original full-block limits** captured before the loop begins. As transactions are added and `size`/`cycles` accumulate, the remaining capacity shrinks — but the filter never tightens. Transactions that individually fit within the full block limit but cannot fit in the **remaining** space still pass the filter, enter the loop, and fail at line 152, each time incrementing `consecutive_failed`.

**Concrete scenario:**

1. The proposed pool contains 5,000 transactions sorted by descending fee rate.
2. The first 1,000 are included, filling the block to ~95% capacity.
3. The next 4,001 transactions each have `ancestors_size` that fits within the original `size_limit` (pass the filter) but exceeds the remaining space (fail at line 152).
4. `consecutive_failed` reaches 4,001 → `break` fires at line 158.
5. The remaining transactions — including small ones that would fit in the residual 5% — are **never evaluated**.

The same early-exit can be triggered entirely by Path 2: if 4,001 proposed-pool entries have at least one ancestor that is not yet in the proposed pool (a normal transient state in CKB's two-phase commit), the counter hits the limit and the loop exits, regardless of how full the block is.

---

### Impact Explanation

- **Block templates produced by `get_block_template` RPC are suboptimal**: valid, fee-paying transactions that fit in the remaining block space are silently excluded.
- **Miner revenue loss**: fees from excluded transactions are forfeited.
- **Transaction confirmation delay**: users whose transactions are excluded must wait for a subsequent block.
- **Targeted exclusion**: an attacker who controls enough mempool entries can reliably push `consecutive_failed` past the threshold before a victim's transaction is evaluated, effectively censoring it from block templates without any privileged access.

---

### Likelihood Explanation

The entry path is fully unprivileged. Any node reachable via the `send_transaction` RPC endpoint accepts transactions from external callers. An attacker needs only to:

1. Submit enough transactions (≥ 4,001) that will fail either the size/cycles check or the ancestor-proposed check during a given assembly round.
2. Ensure those transactions sort ahead of the victim transaction by fee rate (trivially achieved by paying a slightly higher fee).

CKB's two-phase commit model makes Path 2 naturally reachable: during the window between a transaction being proposed and its ancestors being confirmed, ancestor-not-proposed failures occur organically. A large mempool under normal load can accumulate these failures without any attacker involvement, making this a reliability issue even absent a deliberate attack.

---

### Recommendation

Reset `consecutive_failed` to zero each time a transaction is successfully included in the block, matching the intent of the "consecutive" name and the behavior of analogous implementations (e.g., Bitcoin Core's `CreateNewBlock`):

```rust
// After the successful inclusion block (after line 220):
consecutive_failed = 0;
```

Additionally, tighten the pre-loop filter to use the **remaining** capacity rather than the original limits, so that transactions that cannot possibly fit are skipped without consuming failure budget:

```rust
.filter(|entry| {
    entry.ancestors_size <= size_limit && entry.ancestors_cycles <= cycles_limit
    // Note: size_limit/cycles_limit should reflect remaining capacity, updated each iteration
})
```

Or, equivalently, skip entries whose `ancestors_size > (size_limit - size)` before incrementing `consecutive_failed`.

### Proof of Concept

**Setup**: A proposed pool with 5,001 entries, all with fee rate 1000 shannons/byte.

- Entries 1–1,000: `ancestors_size = 9_000` bytes each → included successfully, total `size` = 9,000,000 bytes (block ~90% full, limit = 10,000,000).
- Entries 1,001–5,001: `ancestors_size = 1,500` bytes each → pass the filter (`1,500 ≤ 10,000,000`) but fail at line 152 (`9,000,000 + 1,500 > 10,000,000`... wait, 9,000,000 + 1,500 = 9,001,500 which is < 10,000,000). Let me adjust.

**Adjusted Setup**: limit = 10,000,000 bytes.

- Entries 1–9: `ancestors_size = 1,000,000` bytes each → included, total `size` = 9,000,000.
- Entries 10–4,010: `ancestors_size = 1,500,000` bytes each → pass filter (`1,500,000 ≤ 10,000,000`) but fail at line 152 (`9,000,000 + 1,500,000 = 10,500,000 > 10,000,000`). Each increments `consecutive_failed`. After 4,001 such entries, `consecutive_failed > 4000` → `break`.
- Entry 4,011: `ancestors_size = 900,000` bytes → would fit (`9,000,000 + 900,000 = 9,900,000 < 10,000,000`) and would be included — **but the loop has already exited**.

The block template is produced with only 9 transactions instead of 10. The 10th valid transaction is excluded despite fitting within the block limits.
