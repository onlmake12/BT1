Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Persistent Connection with Unique `(from, to)` PeerId Pairs — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

`HolePunching::forward_rate_limiter` uses `governor`'s `HashMapStateStore<(PeerId, PeerId, u32)>`, which accumulates one entry per unique key and relies on explicit `retain_recent()` calls for eviction. The `disconnected()` handler calls `retain_recent()`, but the periodic `notify()` handler (every 5 minutes) does not. An attacker maintaining a persistent TCP connection and sending `ConnectionRequest` messages with unique attacker-controlled `(from, to)` PeerId pairs can grow the HashMap without bound, eventually exhausting heap memory and crashing the node.

## Finding Description

`forward_rate_limiter` is declared as `RateLimiter<(PeerId, PeerId, u32)>` backed by `HashMapStateStore`: [1](#0-0) 

`retain_recent()` is called on `forward_rate_limiter` only inside `disconnected()`: [2](#0-1) 

The `notify()` handler, which fires every `CHECK_INTERVAL` (5 minutes), cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter: [3](#0-2) 

In `ConnectionRequestProcess::execute`, every message that passes the per-session `rate_limiter` check calls `check_key` on `forward_rate_limiter` with attacker-controlled `content.from` and `content.to`: [4](#0-3) 

The per-session `rate_limiter` is keyed by `(session_id, msg.item_id())` and allows 30 messages/sec from a single session regardless of how many unique `(from, to)` pairs are used: [5](#0-4) 

The `forward_rate_limiter` quota is 1 request/second per `(from, to, msg_item_id)` tuple: [6](#0-5) 

Since each new `(from, to)` pair is a fresh key, it is never rate-limited on first use — it simply inserts a new entry. With 30 unique pairs/sec and no periodic eviction, the HashMap grows at 30 entries/sec indefinitely as long as the session remains open. The attacker can craft messages that pass all validation checks (valid `listen_addrs`, `max_hops ≤ 6`, `route` not containing the victim's peer ID) to reliably reach the `check_key` call. [7](#0-6) 

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.** At 30 entries/sec sustained over 24 hours, the attacker inserts ~2.6 million entries. Each entry holds two `PeerId` values (~39 bytes each) plus a `u32` plus `HashMap` overhead (~100–120 bytes/entry), yielding ~300 MB of heap growth from a single connection. Multiple concurrent attackers scale this linearly. The node eventually OOM-crashes, terminating block relay, transaction propagation, and consensus participation.

## Likelihood Explanation

The attack requires only a single persistent TCP connection — no proof-of-work, no stake, no privileged role. The attacker generates random `PeerId` bytes locally (trivial). The per-session rate limiter is the only throttle and does not bound total HashMap size. The attack is fully automatable, replicable across the network, and requires no victim interaction or mistake.

## Recommendation

Add `retain_recent()` calls for both rate limiters inside the `notify()` handler, which already fires every 5 minutes, so stale entries are evicted on a time-based schedule independent of peer disconnection:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();          // add
    self.forward_rate_limiter.retain_recent();  // add
    let now = unix_time_as_millis();
    self.pending_delivered.retain(|_, (_, t)| (now - *t) < TIMEOUT);
    self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);
    ...
}
```

## Proof of Concept

```rust
// Single persistent session, never disconnects
let session = connect_to_victim_node();
loop {
    // 30 unique (from, to) pairs per second — stays within per-session rate limit
    for _ in 0..30 {
        let from = PeerId::random();
        let to   = PeerId::random();
        session.send(ConnectionRequest {
            from, to, max_hops: 1,
            listen_addrs: [valid_tcp_addr],
            route: [],
        });
    }
    sleep(Duration::from_secs(1));
    // forward_rate_limiter HashMap grows by 30 entries/sec
    // retain_recent() is never called because session stays open
    // After 24h: ~2.6M entries, ~300MB heap consumed → OOM crash
}
```

To verify: instrument `forward_rate_limiter` to log its `len()` before and after each `received()` call, run the PoC for 60 seconds, and confirm monotonic growth of ~1800 entries with no eviction.
