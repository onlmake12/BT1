### Title
Unbounded per-request amplification in `GetTransactionsProofProcess::execute` with no rate limiting on `LightClientProtocol` — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

---

### Summary

`LightClientProtocol` has no per-peer rate limiter. A single valid `GetTransactionsProof` message carrying `GET_TRANSACTIONS_PROOF_LIMIT` (1000) tx_hashes, each from a distinct block, forces the server to perform up to ~6000 storage reads, 1000 CBMT proof builds, and one MMR `gen_proof` over 1000 positions — all synchronously, with no throttle. An unprivileged peer can repeat this indefinitely.

---

### Finding Description

**Entry point** — `LightClientProtocol::received` dispatches directly to `try_process` with no rate check: [1](#0-0) 

There is no `rate_limiter` field anywhere in `LightClientProtocol`: [2](#0-1) 

Contrast with `Relayer`, which gates every non-PoW message through a per-peer/message-type rate limiter before processing: [3](#0-2) 

And `HolePunching`, which does the same: [4](#0-3) 

**The only guard in `execute`** is a count ceiling: [5](#0-4) 

`GET_TRANSACTIONS_PROOF_LIMIT` is 1000: [6](#0-5) 

A request with exactly 1000 hashes passes this check. If each hash belongs to a distinct block, the per-block loop runs 1000 times, each iteration performing:

- `snapshot.get_block()` — full block I/O read
- `CBMT::build_merkle_proof()` — hashes all transactions in the block
- `snapshot.get_block_uncles()` — uncle I/O read
- `snapshot.get_block_extension()` — extension I/O read [7](#0-6) 

Then `reply_proof` calls `mmr.gen_proof(1000_positions)` — an O(N log N) MMR traversal over the full chain: [8](#0-7) 

**No ban is triggered** for a well-formed request. `MalformedProtocolMessage` (400-range) bans the peer, but a request with 1000 valid hashes is not malformed: [9](#0-8) 

---

### Impact Explanation

Each maximally-crafted request causes ~4000 storage I/O operations plus 1000 CBMT hash-tree builds plus one large MMR proof generation. With no rate limiting, a single attacker peer can saturate the server's I/O and CPU by pipelining requests. This degrades sync and relay performance for all honest peers sharing the same node.

---

### Likelihood Explanation

The attack requires only a standard P2P connection to a node that has the light-client protocol enabled. All 1000 tx_hashes can be harvested from public chain data. No key, no PoW, no privileged role is needed. The amplification ratio (1 small request → thousands of server-side operations) makes it practical even over a single connection.

---

### Recommendation

Add a per-peer, per-message-type rate limiter to `LightClientProtocol` following the same pattern used in `Relayer` (`sync/src/relayer/mod.rs`) and `HolePunching` (`network/src/protocols/hole_punching/mod.rs`). Additionally, consider capping the number of distinct blocks that a single `GetTransactionsProof` request may span (e.g., 50 blocks), independent of the total tx_hash count.

---

### Proof of Concept

1. Start a CKB full node with the light-client protocol enabled on a chain with ≥1000 committed transactions across ≥1000 distinct blocks.
2. Connect as a light-client peer.
3. Collect 1000 tx_hashes, one from each of 1000 distinct blocks.
4. Send a `GetTransactionsProof` message with all 1000 hashes and the current tip as `last_hash`.
5. Measure server CPU and I/O during processing; repeat in a tight loop.
6. Compare against a baseline request with 1000 hashes all from the same block (single `get_block` call, single CBMT build, MMR proof for 1 position).

The per-request cost in step 4 will be ~1000× higher than the baseline in step 6, with no server-side throttle preventing repetition.

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
