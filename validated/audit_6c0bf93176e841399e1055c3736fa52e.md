### Title
`getPricesNoOlderThan` Staleness Check Uses Batch-Level Max Timestamp Instead of Per-Feed `publishTime`, Allowing Stale Individual Prices to Pass - (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Scheduler contract's `getPricesNoOlderThan` (and `getEmaPricesNoOlderThan`) functions check staleness against `status.priceLastUpdatedAt`, which is set to the **maximum** `publishTime` across all feeds in the batch. Individual feeds in the same subscription can have `publishTime` values that are arbitrarily older (e.g., closed-market assets). A whitelisted reader calling `getPricesNoOlderThan` with a tight age threshold (e.g., 60 seconds) believes all returned prices satisfy that threshold, but individual feeds can be hours or days old.

---

### Finding Description

In `Scheduler.sol`, `status.priceLastUpdatedAt` is set during `updatePriceFeeds` to the **maximum** `publishTime` across all feeds in the batch:

```solidity
// _validateShouldUpdatePrices
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
// ...
status.priceLastUpdatedAt = latestPublishTime; // = max(publishTime)
``` [1](#0-0) [2](#0-1) 

The `getPricesNoOlderThan` function then checks staleness only against this single batch-level value:

```solidity
function getPricesNoOlderThan(...) external view override onlyWhitelistedReader(subscriptionId) {
    SchedulerStructs.SubscriptionStatus memory status = _state.subscriptionStatuses[subscriptionId];
    if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
        revert PythErrors.StalePrice();
    prices = this.getPricesUnsafe(subscriptionId, priceIds);
}
``` [3](#0-2) 

The code explicitly acknowledges that closed-market feeds will carry old `publishTime` values from their last trading period, and intentionally uses the max timestamp to avoid rejecting such updates:

> "Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period." [4](#0-3) 

The test `testUpdatePriceFeedsSucceedsWithStaleFeedIfLatestIsValid` confirms this behavior: a batch with one feed at `stalePublishTime` (1 hour 5 minutes ago) and one at `validPublishTime` (30 minutes ago) succeeds, and `priceLastUpdatedAt` is set to `validPublishTime`. [5](#0-4) 

This means `getPricesNoOlderThan(subscriptionId, [feedA_BTC, feedB_GOLD], 60)` will:
1. Check `distance(block.timestamp, BTC_publishTime) ≤ 60` → passes
2. Return GOLD's price with `publishTime` from 8+ hours ago without any per-feed check

The same flaw exists in `getEmaPricesNoOlderThan`: [6](#0-5) 

---

### Impact Explanation

A whitelisted reader (e.g., a DeFi lending protocol) calls `getPricesNoOlderThan(subscriptionId, priceIds, 60)` expecting all returned prices to be no older than 60 seconds. The function returns without reverting, but individual feeds for closed-market assets (equities, commodities, FX) can carry `publishTime` values from their last trading session — potentially hours or days old. The consumer has no indication that the staleness guarantee was not met per-feed. This can lead to incorrect collateral valuations, liquidations, or trades based on stale prices.

---

### Likelihood Explanation

The Scheduler is explicitly designed to support mixed subscriptions containing both 24/7 crypto assets and session-based assets (equities, commodities). The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is only 1 hour, but closed-market assets can have `publishTime` from their last session close, which can be 16+ hours ago on weekdays or 64+ hours over weekends. Any subscription mixing these asset types triggers the condition. The `onlyWhitelistedReader` modifier means the affected party is a legitimate consumer of the Scheduler service, not an external attacker. [7](#0-6) 

---

### Recommendation

Change `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` to check each individual feed's `publishTime` against `age_seconds`, not the batch-level `priceLastUpdatedAt`:

```solidity
function getPricesNoOlderThan(...) external view override onlyWhitelistedReader(subscriptionId) {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

Alternatively, document clearly that `age_seconds` applies to the batch update time, not individual feed `publishTime`, and rename the function to avoid the misleading implication that all returned prices satisfy the age constraint.

---

### Proof of Concept

1. Create a subscription with two feeds: `BTC/USD` (24/7) and `GOLD/USD` (session-based).
2. Submit an update where `BTC/USD.publishTime = block.timestamp - 30s` and `GOLD/USD.publishTime = block.timestamp - 8 hours`. Both feeds share the same Pythnet slot, so the slot check passes.
3. `_validateShouldUpdatePrices` computes `updateTimestamp = max(now-30s, now-8h) = now-30s`. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` check passes (30s < 1h). `status.priceLastUpdatedAt = now - 30s`.
4. Call `getPricesNoOlderThan(subscriptionId, [BTC_id, GOLD_id], 60)`.
5. The check `distance(block.timestamp, now-30s) = 30 ≤ 60` passes. The function returns both prices without reverting.
6. The returned `GOLD/USD` price has `publishTime = now - 8 hours`, violating the caller's expectation of a 60-second freshness guarantee. [8](#0-7) [9](#0-8)
