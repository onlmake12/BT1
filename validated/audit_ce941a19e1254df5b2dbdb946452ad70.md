### Title
`Scheduler.getMinimumBalance` Omits Pyth Protocol Fee Component, Causing Subscriptions Funded at Minimum Balance to Fail During Updates - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getMinimumBalance` returns only `numPriceFeeds * minimumBalancePerFeed`, omitting the Pyth protocol fee (`pythFee`) that is also deducted from the subscription balance during every `updatePriceFeeds` call. Subscriptions funded at exactly the value returned by `getMinimumBalance` will revert during updates because the balance is insufficient to cover both the Pyth fee and the keeper fee.

---

### Finding Description

`getMinimumBalance` is the canonical function used to enforce minimum balance requirements at subscription creation, update, top-up, and withdrawal:

```solidity
function getMinimumBalance(
    uint8 numPriceFeeds
) external view override returns (uint256 minimumBalanceInWei) {
    return uint256(numPriceFeeds) * this.getMinimumBalancePerFeed();
}
``` [1](#0-0) 

However, `updatePriceFeeds` deducts **two** separate fee components from `status.balanceInWei`:

**Step 1 — Pyth protocol fee deducted first:**
```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
if (status.balanceInWei < pythFee) {
    revert SchedulerErrors.InsufficientBalance();
}
status.balanceInWei -= pythFee;
``` [2](#0-1) 

**Step 2 — Keeper fee deducted from the already-reduced balance:**
```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [3](#0-2) 

The total actual cost per update is `pythFee + totalKeeperFee`, but `getMinimumBalance` only accounts for the keeper-side component (`numPriceFeeds * minimumBalancePerFeed`). The Pyth fee is entirely absent from the minimum balance formula.

The same `getMinimumBalance` is used as the enforcement gate in `createSubscription`, `updateSubscription`, `addFunds`, and `withdrawFunds`: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

Any subscription manager who funds their subscription at exactly `getMinimumBalance(n)` will have a subscription that is accepted as valid by the contract but will revert on every `updatePriceFeeds` call with `InsufficientBalance` once the Pyth fee is deducted. The subscription appears active and funded, yet no keeper can successfully update it. Managers who call `withdrawFunds` down to the minimum balance floor are also left with a non-functional subscription. The Pyth fee is non-zero on all production deployments (1 wei minimum per update, scaling with the number of price feeds in the update data), so this is not a theoretical edge case.

---

### Likelihood Explanation

The `getMinimumBalance` function is the public, documented API for determining how much to deposit. Any subscription manager following the contract interface will use it. The Pyth fee is always present and always deducted before the keeper fee check, so every subscription funded at exactly the minimum will fail on its first update attempt. This is a high-likelihood, low-effort trigger requiring no special privileges.

---

### Recommendation

Include the Pyth protocol fee in `getMinimumBalance`. Since the Pyth fee scales with the number of price feeds, a per-feed estimate should be added:

```solidity
function getMinimumBalance(
    uint8 numPriceFeeds
) external view override returns (uint256 minimumBalanceInWei) {
+   uint256 pythFeePerFeed = IPyth(_state.pyth).singleUpdateFeeInWei(); // or a governance-set estimate
    return uint256(numPriceFeeds) * (this.getMinimumBalancePerFeed() + pythFeePerFeed);
}
```

Alternatively, add a separate `pythFeeReservePerFeed` governance parameter and include it in the minimum balance formula, ensuring the returned value always covers the full cost of one update cycle.

---

### Proof of Concept

1. Admin deploys `Scheduler` with `minimumBalancePerFeed = X` (set to cover keeper gas + `singleUpdateKeeperFeeInWei`).
2. Subscription manager calls `createSubscription` with `msg.value = getMinimumBalance(2) = 2 * X`.
3. Contract accepts the subscription as valid.
4. Keeper calls `updatePriceFeeds(subscriptionId, updateData)`.
5. `pythFee = pyth.getUpdateFee(updateData)` returns `P > 0`.
6. `status.balanceInWei -= pythFee` → remaining = `2X - P`.
7. `_processFeesAndPayKeeper` computes `totalKeeperFee ≈ 2X` (since `minimumBalancePerFeed` was sized to cover it).
8. Check `2X - P < 2X` → `InsufficientBalance` revert.
9. The subscription is permanently stuck: it passes all balance checks at creation/withdrawal but fails every update.
