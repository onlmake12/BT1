Audit Report

## Title
Attacker-Controlled Snappy Varint Enables Up-to-8MB Pre-Decompression Heap Allocation Per Connection — (File: network/src/compress.rs)

## Summary
In `LengthDelimitedCodecWithCompress::decode`, `BytesMut::zeroed(decompressed_bytes_len)` is allocated unconditionally at line 242 based on the attacker-supplied snappy varint before any validation of the compressed payload. The only guard is the 8MB ceiling at line 235, which an attacker can trivially satisfy with a ~11-byte wire frame. With 117 default max inbound peers, simultaneous crafted frames produce ~924MB peak RSS, sufficient to OOM-kill nodes with ≤1–2GB RAM.

## Finding Description
At `network/src/compress.rs` line 233, `decompress_len(&data[1..])` reads the uncompressed length from the snappy stream varint — a value entirely under attacker control. The guard at line 235 rejects values exceeding `MAX_UNCOMPRESSED_LEN` (`1 << 23` = 8,388,608 bytes), but any value up to 8MB passes. Line 242 then executes `BytesMut::zeroed(decompressed_bytes_len)` unconditionally, committing the full claimed size to physical memory. Only at line 243 does actual decompression occur; a garbage payload causes `SnapDecoder::decompress` to return `Err`, which propagates as `io::ErrorKind::InvalidData` — but the allocation has already been made and is only freed when the buffer is dropped after the error path.

The `max_frame_length` guard (`support_protocols.rs` lines 129–130: Sync = 2MB, RelayV3 = 4MB) is enforced by the `length_delimited` codec on the **compressed wire frame size**. A crafted frame of ~11 bytes (1 flag byte + 4-byte snappy varint claiming 7.9MB + 3 garbage bytes) trivially passes this check while triggering a ~7.9MB allocation on the target.

When `decode` returns `Err`, tentacle closes the connection at the transport layer. The application-layer `ban_peer` path at `protocols/mod.rs` line 684 is never reached for codec-level errors, so the attacker can reconnect immediately and repeat. The same pre-allocation pattern exists in `Message::decompress` at line 81 (`vec![0; decompressed_bytes_len]`).

Default config (`resource/ckb.toml`): `max_peers = 125`, `max_outbound_peers = 8`, giving `max_inbound_peers = 117` (`util/app-config/src/configs/network.rs` line 356: `max_peers.saturating_sub(max_outbound_peers)`).

## Impact Explanation
**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

- Per-connection: one ~11-byte compressed frame → `BytesMut::zeroed(7,864,320)` → decompression error → disconnect → memory freed. Peak RSS spike per connection.
- Simultaneous: 117 inbound connections × 7.9MB ≈ **924MB peak RSS**. On nodes with ≤1–2GB RAM this is sufficient to trigger the OS OOM killer, crashing the node.
- Sustained: no codec-layer ban prevents immediate reconnect. Continuous allocator churn degrades all P2P protocol processing (Sync, RelayV3, LightClient, Filter).

## Likelihood Explanation
The attack requires only a TCP connection to the default P2P port 8115 (publicly exposed per `resource/ckb.toml` line 70). The attacker must complete a tentacle/secio handshake per connection, which adds modest per-connection cost but is not a meaningful barrier. No authentication, PoW, key material, or prior relationship is required. The crafted frame is trivial to construct. The default `max_inbound = 117` caps simultaneous connections but an attacker can open all 117 from a single host or distributed IPs. The attack is fully repeatable with no per-attempt cost beyond a TCP+secio handshake.

## Recommendation
Move the allocation **after** lightweight structural pre-validation of the compressed payload (e.g., verify the snappy stream identifier block before trusting the varint), or replace `BytesMut::zeroed(decompressed_bytes_len)` with a streaming/lazy decompressor that does not pre-allocate the full claimed size. At minimum, add a codec-layer connection ban or exponential backoff when `decode` returns `Err` for a compressed frame to prevent immediate reconnect-and-repeat. The same issue in `Message::decompress` at line 81 (`vec![0; decompressed_bytes_len]`) must be fixed in both paths.

## Proof of Concept
Minimal Rust unit test:
```rust
#[test]
fn test_pre_alloc_dos() {
    use tokio_util::codec::Decoder;
    use p2p::bytes::BytesMut;

    // Build a length-delimited codec with max_frame_length = 4MB (RelayV3)
    let ld = tokio_util::codec::length_delimited::Builder::new()
        .max_frame_length(4 * 1024 * 1024)
        .length_field_length(4)
        .new_codec();
    let mut codec = LengthDelimitedCodecWithCompress::new(true, ld, 0.into());

    // Snappy varint encoding of 7,864,320 (~7.9MB, just under MAX_UNCOMPRESSED_LEN)
    let mut val = 7_864_320usize;
    let mut varint = vec![];
    while val > 0x7F { varint.push(((val & 0x7F) | 0x80) as u8); val >>= 7; }
    varint.push(val as u8);

    // Frame: COMPRESS_FLAG=0x80 | varint | 3 garbage bytes
    let mut payload = vec![0x80u8];
    payload.extend_from_slice(&varint);
    payload.extend_from_slice(&[0xff, 0xff, 0xff]);

    // 4-byte big-endian length prefix
    let mut frame = BytesMut::new();
    frame.extend_from_slice(&(payload.len() as u32).to_be_bytes());
    frame.extend_from_slice(&payload);

    // Measure allocator before/after: expect ~7.9MB allocation then Err return
    let result = codec.decode(&mut frame);
    assert!(result.is_err()); // allocation occurred, then freed
}
```
The ~7.9MB allocation followed by `Err` return is directly observable via allocator instrumentation (e.g., `jemalloc` stats or `valgrind --tool=massif`). For the full DoS, open 117 simultaneous connections each sending this frame and observe peak RSS on the target node.
