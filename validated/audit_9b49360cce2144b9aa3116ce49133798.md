### Title
`Echo.executeCallback()` Permanently Consumes User Fees When Consumer Callback Fails, With No Refund or Retry Mechanism — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, `executeCallback()` credits provider fees and clears the request **before** attempting the consumer callback. When the callback fails (caught by `try/catch`), the user's fee is permanently consumed with no refund path and no retry mechanism. A registered provider can deliberately trigger this by supplying insufficient gas to the callback, collecting the fee while delivering no service.

---

### Finding Description

`executeCallback()` performs state mutations in this order:

1. **Credits provider fees** (line 161–162):
   ```solidity
   _state.providers[providerToCredit].accruedFeesInWei += SafeCast
       .toUint128((req.fee + msg.value) - pythFee);
   ```

2. **Clears the request** (line 164):
   ```solidity
   clearRequest(sequenceNumber);
   ```

3. **Attempts the callback** (lines 176–201) with `try/catch`:
   ```solidity
   try
       IEchoConsumer(req.requester)._echoCallback{
           gas: req.callbackGasLimit
       }(sequenceNumber, priceFeeds)
   { ... }
   catch Error(string memory reason) {
       emit PriceUpdateCallbackFailed(...);
   } catch {
       emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
   }
   ```

When the callback fails, the request is already cleared and the provider has already received the fee. The user:
- Cannot retry (request is gone)
- Cannot cancel (no cancellation function exists)
- Cannot recover their fee (no refund path)

The developers themselves flag this in a TODO comment at line 155–156:
> `// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract.`

The try/catch prevents the outer transaction from reverting, but it does not address the case where the callback fails — the user's funds are still consumed.

**Concrete attack path via gas manipulation:**

The contract does not enforce that the provider supplies sufficient gas to the outer transaction. The `{gas: req.callbackGasLimit}` call forwards at most 63/64 of the remaining gas (EVM CALL opcode rule). A malicious registered provider can call `executeCallback` with just enough gas to pass the pre-callback logic but not enough to satisfy `callbackGasLimit` inside the callback. The callback fails with an out-of-gas error, the try/catch catches it, the provider collects the fee, and the user receives no price update.

Additionally, in `requestPriceUpdatesWithCallback()` (line 76–84), there is no refund of excess `msg.value`:
```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
// ...
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```
Any overpayment above `requiredFee` is silently captured as provider fee, compounding the loss.

---

### Impact Explanation

**Impact: Medium.** A user who calls `requestPriceUpdatesWithCallback` and pays the required fee loses their entire fee payment if the callback fails, with no mechanism to recover funds or retry. In the malicious-provider scenario, the provider profits from the fee while delivering no service. This is a direct, permanent loss of user funds.

---

### Likelihood Explanation

**Likelihood: High.** Callback failures are common in practice (out-of-gas, logic errors, reentrancy guards). Any registered provider (permissionless registration via `registerProvider()`) can deliberately trigger this by supplying insufficient gas. The contract provides no protection against this pattern.

---

### Recommendation

1. **Attempt the callback before crediting fees and clearing the request.** If the callback reverts, the entire transaction reverts, preserving the user's funds and the request state.
2. **Alternatively, implement a `CALLBACK_FAILED` state** (as Entropy does) that keeps the request alive and allows retry without additional payment.
3. **Enforce minimum gas** in `executeCallback` to ensure the provider supplies at least `callbackGasLimit * 64/63` gas before the callback is attempted, preventing deliberate out-of-gas attacks.
4. **Refund excess `msg.value`** in `requestPriceUpdatesWithCallback` to prevent silent overpayment capture.

---

### Proof of Concept

```
1. Attacker registers as a provider via registerProvider().

2. Victim calls requestPriceUpdatesWithCallback{value: requiredFee}(
       attacker_provider, publishTime, priceIds, callbackGasLimit=1_000_000
   );
   → req.fee = requiredFee - pythFeeInWei is stored; request is active.

3. Attacker calls executeCallback{gas: ~500_000}(
       attacker_provider, sequenceNumber, updateData, priceIds
   );
   → pythFee is paid to Pyth contract.
   → attacker's accruedFeesInWei += (req.fee + 0) - pythFee  [credited].
   → clearRequest(sequenceNumber)  [request deleted].
   → _echoCallback{gas: 1_000_000} is attempted but only ~470k gas is
     available (63/64 rule); callback reverts with out-of-gas.
   → try/catch catches it; emits PriceUpdateCallbackFailed.
   → executeCallback returns successfully.

4. Victim's fee is consumed. Request is gone. No retry. No refund.
   Attacker withdraws accrued fees via withdrawAsFeeManager().
