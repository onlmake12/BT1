### Title
`LoadCellData` Syscall Loads Full Cell with `u64::MAX` When Actual Size Is Already Known — (File: `script/src/syscalls/load_cell_data.rs`)

---

### Summary

Both `load_data` and `load_data_as_code` in the `LoadCellData` syscall handler call `sc.load_data(…, u64::MAX)` to fetch the entire cell, even when the script has already supplied the exact byte count it needs. Cycles are charged only for the bytes actually written back to the VM, not for the full cell materialization. This is a direct structural analog to the reported `approveAndCall` pattern: the required amount is known at the call site, yet the maximum value is used instead.

---

### Finding Description

When `size != 0`, the returned `cell` binding is entirely unused; `store_bytes` re-fetches the needed slice independently. The `u64::MAX` load is therefore redundant for the common non-zero-size path, yet it materialises the entire cell into a `Bytes` allocation before the size is even inspected.

### Impact Explanation

`Snapshot2Context::load_data` with `u64::MAX` materialises the entire cell data into a heap-allocated `Bytes` object. Cycles are charged only for `wrote_size` (bytes written to VM memory) in `load_data`, and only for `memory_size` (the mapped page region) in `load_data_as_code`. Neither accounts for the cost of reading and allocating the full cell.

A script author can therefore craft a transaction referencing a large cell (capacity-bounded, but cells can hold hundreds of KB) and issue a `ckb_load_cell_data` syscall with `size = 1`. The node performs O(cell\_size) allocation and data-copy work while the script pays O(1) cycles. Repeated across multiple cells or multiple syscall invocations within the cycle budget, this creates a systematic cycle-accounting mismatch that allows a script to impose disproportionate node-side resource consumption.

---

### Likelihood Explanation

The entry path requires only a valid transaction submission — an unprivileged tx-pool submitter. No special role, key, or majority hashpower is needed. The syscall is part of the standard CKB script ABI and is exercised by virtually every non-trivial script. Any script author who knows the cell data layout can deliberately target a large dep or input cell while requesting a single byte, making this straightforward to exploit deliberately or stumble upon accidentally.

---

### Recommendation

In `load_data`, restructure the logic so that `sc.load_data` is called with `u64::MAX` **only** on the `size == 0` branch (where the full length is genuinely needed). On the `size != 0` branch, skip the preliminary full-cell load entirely and call `store_bytes` directly — it already handles the `SnapshotDataLoadError` → `INDEX_OUT_OF_BOUND` mapping. This mirrors the fix recommended for `approveAndCall`: use the known amount rather than the maximum.

In `load_data_as_code`, pass `content_offset + content_size` (the actual end boundary) as the length argument to `sc.load_data` instead of `u64::MAX`, so only the required slice is fetched and allocated.

---

### Proof of Concept

1. Publish a dep cell containing 500 KB of arbitrary data (within CKB capacity rules).
2. Write a lock script that calls `ckb_load_cell_data` with `addr = some_buf`, `size = 1`, `offset = 0`, pointing at the 500 KB dep cell.
3. Submit the transaction to the tx-pool.
4. The node's `load_data` handler calls `sc.load_data(…, u64::MAX)`, allocating ~500 KB on the heap.
5. `store_bytes` writes exactly 1 byte; `transferred_byte_cycles(1)` is charged.
6. The script pays ~1 cycle for data transfer while the node performs ~500 KB of allocation and I/O work.
7. Repeat the syscall up to the script's cycle limit to amplify the imbalance within a single transaction execution. [3](#0-2)
