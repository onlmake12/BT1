Audit Report

## Title
ELF Parsing and Memory Allocation Performed Before Cycle-Limit Enforcement in VM Boot Path ‚Äî (File: `script/src/scheduler.rs`)

## Summary
When a new VM is booted via `boot_vm()` ‚Äî either for the root script or via a `ckb_spawn` syscall ‚Äî the expensive operations of ELF parsing (`parse_elf`) and program loading (`load_program_with_metadata`) execute with no cycle pre-check. The cycle cost is charged to the new machine's internal counter via `add_cycles_no_checking`, which does not enforce the limit. Limit enforcement only occurs in the next scheduler iteration when `set_max_cycles` is called. An attacker can craft transactions that force full ELF parsing and heap allocation before the cycle check fires, making sustained CPU/memory pressure cheaper than the cycle model intends.

## Finding Description
**Root VM path:** `run()` calls `boot_root_vm_if_needed()` at L292 before `limit_cycles` is extracted from `mode` (L294‚Äì297) and before the cycle-enforcement loop begins (L299‚Äì301). The root ELF is fully parsed and loaded into VM memory with zero cycle budget enforcement.

**Spawn path call chain:** `iterate_inner()` (L461) ‚Üí `iterate_process_results()` (L478) ‚Üí `process_message_box()` (L358) ‚Üí `Message::Spawn` handler (L523) ‚Üí `boot_vm()` (L539) ‚Üí `load_vm_program()` (L1023).

Inside `boot_vm()` (L1015‚Äì1039):
- `create_dummy_vm()` allocates a new RISC-V machine initialized with `max_cycles = u64::MAX` (L1083‚Äì1084), meaning no limit is active during boot.
- `sc.load_data(...)` reads the full ELF binary (L1021).
- `load_vm_program()` (L1042‚Äì1075) calls `parse_elf` (L1050), `load_program_with_metadata` (L1058/L1061, heap allocation proportional to ELF size), and `sc.mark_program` (L1065‚Äì1070) ‚Äî all before any cycle check.

The cycle charge is applied only after all this work:
```rust
machine.inner_mut().add_cycles_no_checking(transferred_byte_cycles(bytes))?;
```
(L1071‚Äì1073). `add_cycles_no_checking` accumulates cycles on the new machine but does **not** enforce the limit.

The limit is only enforced in the next iteration inside `iterate_inner` (L467):
```rust
vm.inner_mut().set_max_cycles(limit_cycles);
let result = vm.run();
```

`iterate_outer()` checks `limit_cycles.checked_sub(self.iteration_cycles)` (L424‚Äì426), but `iteration_cycles` only contains cycles from the spawning VM's execution ‚Äî not the new machine's pre-loaded ELF cost. If the new machine's pre-loaded cycle count already exceeds `limit_cycles`, it fails immediately on its first instruction, but all expensive setup work has already completed.

**Existing guards are insufficient:** `MAX_VMS_COUNT = 16` (L34) limits spawned VMs per transaction but does not prevent the per-spawn ELF loading cost from being incurred before the check. The `MAX_VMS_COUNT` guard at L532‚Äì538 fires before `boot_vm` is called, but only limits the count ‚Äî not the per-spawn ELF loading cost. The flat fee `SPAWN_EXTRA_CYCLES_BASE` (imported at L4) is charged to `iteration_cycles` in `resume_vm`/`suspend_vm`, not in the spawn boot path itself, and does not cover the variable cost of loading a large ELF.

## Impact Explanation
**High severity** ‚Äî matches: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

An attacker submitting transactions with `max_cycles` tuned to the gap between the flat spawn fee and the ELF loading cost forces the node to perform O(ELF size) heap allocation and ELF segment mapping per transaction, while the transaction is rejected for cycle exhaustion. The node's tx-pool verification workers and block verification path both execute this code path. Repeated submission of such transactions sustains CPU and memory pressure disproportionate to the cycle budget the attacker nominally consumes.

## Likelihood Explanation
The attack requires only the ability to submit transactions to the tx-pool, which is open to any network peer with no special privileges, keys, or hashpower. The attacker deploys a large cell dep ELF once (a normal on-chain operation) and then submits arbitrarily many transactions referencing it. The attack is repeatable, cheap relative to the node work induced, and requires no victim mistakes or external dependencies.

## Recommendation
1. In `process_message_box` for `Message::Spawn` (L523), after `sc.load_data` returns the program bytes, compute `transferred_byte_cycles(program.len())` and check it against the remaining cycle budget before proceeding to `boot_vm`. If the budget is insufficient, set the spawning VM's return register to an error code and skip the boot.
2. Alternatively, charge `transferred_byte_cycles` to `iteration_cycles` (the scheduler-level counter checked against the limit in `iterate_outer` at L424‚Äì426) rather than to the new machine's unchecked counter via `add_cycles_no_checking`, so the cost is enforced in the same iteration that triggers the spawn.
3. Apply the same fix to the root VM boot path in `boot_root_vm_if_needed` (L993), ensuring the ELF loading cost is checked against `limit_cycles` before `boot_vm` is called.

## Proof of Concept
1. Deploy a cell dep containing a valid ELF binary of ~1 MB.
2. Submit a transaction whose lock script is a minimal program that immediately calls `ckb_spawn(cell_dep_index, ...)` targeting the large ELF.
3. Set `max_cycles` to `~150,000` (above `SPAWN_EXTRA_CYCLES_BASE + SPAWN_YIELD_CYCLES_BASE`, below `SPAWN_EXTRA_CYCLES_BASE + SPAWN_YIELD_CYCLES_BASE + transferred_byte_cycles(1_048_576)`).
4. The node executes the spawning script, issues the spawn syscall (charges flat fees, yields), then enters `process_message_box` ‚Üí `boot_vm`:
   - `create_dummy_vm` allocates the machine with `max_cycles = u64::MAX` (L1083‚Äì1084)
   - `load_data` reads the 1 MB ELF (L1021)
   - `parse_elf` parses it (L1050)
   - `load_program_with_metadata` maps it into VM memory (L1058/L1061)
   - `add_cycles_no_checking(transferred_byte_cycles(1_048_576))` charges to the new machine ‚Äî no limit enforced (L1071‚Äì1073)
5. Next iteration: `set_max_cycles(remaining ~49,200)` is called (L467); the machine already has ~262,144 cycles accumulated; it fails immediately with `CyclesExceeded`.
6. Transaction is rejected, but the node has performed full ELF parsing and memory allocation for the spawned VM.
7. Repeat with many such transactions to sustain CPU/memory pressure on the node.
