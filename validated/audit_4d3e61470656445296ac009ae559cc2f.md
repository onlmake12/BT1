### Title
Unguarded `_entropyCallback` Invocation in Legacy Path Permanently Blocks Entropy Fulfillment - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains two execution paths. The new path (when `req.gasLimit10k != 0`) uses `excessivelySafeCall` to catch callback reverts gracefully. However, the legacy path (when `req.gasLimit10k == 0`, i.e., providers with no `defaultGasLimit` set) invokes `_entropyCallback` directly with no error handling. If the consumer's callback reverts, the entire `revealWithCallback` transaction reverts, and since there is no recovery mechanism for this path, the request is permanently stuck and can never be fulfilled.

### Finding Description

In `revealWithCallback`, the branching logic at line 574 separates requests into two paths:

**New path** (`req.gasLimit10k != 0 && CALLBACK_NOT_STARTED`): uses `excessivelySafeCall`, catches reverts, emits `CallbackFailed`, and transitions the request to `CALLBACK_FAILED` state for later recovery. [1](#0-0) 

**Legacy/else path** (`req.gasLimit10k == 0`): clears the request from storage first, then calls `_entropyCallback` directly with no try-catch: [2](#0-1) 

If `_entropyCallback` reverts in the legacy path, the entire transaction reverts (including `clearRequest`), returning the request to `CALLBACK_NOT_STARTED`. There is no `CALLBACK_FAILED` state transition and no recovery mechanism for this path. Every subsequent `revealWithCallback` attempt will also revert, permanently blocking fulfillment.

A provider enters the legacy path when `defaultGasLimit == 0`. `requestWithCallback` explicitly passes `gasLimit = 0`, which maps to `req.gasLimit10k = 0` when the provider's `defaultGasLimit` is also 0: [3](#0-2) [4](#0-3) 

The `EntropyStatusConstants` confirms there is no `CALLBACK_FAILED` state reachable from the legacy path: [5](#0-4) 

### Impact Explanation

Any Entropy request made against a provider with `defaultGasLimit == 0` (the legacy configuration) where the consumer's `entropyCallback` reverts — whether due to a bug, an invariant violation, or intentional logic — results in a permanently unfulfillable request. The consumer contract never receives its random number. Any application state machine depending on the callback (e.g., a lottery, NFT mint, or game outcome) is permanently blocked. The user's fee is already credited to the provider at request time and is not refunded.

### Likelihood Explanation

The `requestWithCallback` function is the primary legacy entry point and is still callable by any unprivileged user. Providers that have not called `setDefaultGasLimit` retain `defaultGasLimit == 0`. Consumer callbacks that revert are a realistic scenario (e.g., due to reentrancy guards, paused contracts, or logic bugs introduced after the request was made). The Fortuna off-chain relayer calls `revealWithCallback` automatically, meaning repeated failed attempts will occur without any on-chain recovery path. [6](#0-5) 

### Recommendation

Wrap the direct `_entropyCallback` invocation in the legacy else-branch with a try-catch (or `excessivelySafeCall`), and introduce a `CALLBACK_FAILED` state transition analogous to the new path. This would allow the request to be retried or manually recovered rather than being permanently stuck:

```solidity
if (len != 0) {
    try IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    ) {
        // success: emit RevealedWithCallback
    } catch {
        // re-store request in CALLBACK_FAILED state for recovery
        emit CallbackFailed(...);
    }
}
```

Alternatively, migrate all legacy requests to the new `gasLimit10k`-based flow by enforcing a non-zero minimum `defaultGasLimit` for all providers.

### Proof of Concept

1. Provider registers with `defaultGasLimit == 0` (legacy, never called `setDefaultGasLimit`).
2. User calls `requestWithCallback(provider, userRandomNumber)` — this sets `req.gasLimit10k = 0`.
3. Consumer contract's `entropyCallback` contains a `revert` (e.g., a paused modifier or a bug).
4. Fortuna/relayer calls `revealWithCallback(provider, seq, userRandom, providerRandom)`.
5. Execution enters the `else` branch at line 661; `clearRequest` executes, then `_entropyCallback` reverts.
6. The entire transaction reverts; `clearRequest` is undone; request stays at `CALLBACK_NOT_STARTED`.
7. Steps 4–6 repeat indefinitely. The request is permanently stuck with no on-chain recovery path. [7](#0-6)
