### Title
Coarse-Grained Staleness Check in `getPricesNoOlderThan` Allows Return of Arbitrarily Stale Per-Feed Prices - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getPricesNoOlderThan` validates freshness using a single subscription-level timestamp (`priceLastUpdatedAt`) that equals the **maximum** `publishTime` across all feeds in the subscription. When a caller requests a specific feed whose individual `publishTime` is far older than `age_seconds` (e.g., a closed-market equity feed), the staleness gate passes because the max timestamp is fresh, but the returned price data is arbitrarily stale. Any whitelisted reader (or any caller when whitelist is disabled) can trigger this path.

---

### Finding Description

In `Scheduler.sol`, `updatePriceFeeds` records the subscription-level freshness marker as the **maximum** `publishTime` across all parsed feeds:

```solidity
// _validateShouldUpdatePrices
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
// ...
status.priceLastUpdatedAt = latestPublishTime;  // = max publishTime
``` [1](#0-0) [2](#0-1) 

`getPricesNoOlderThan` then checks freshness against this single max value:

```solidity
function getPricesNoOlderThan(..., uint256 age_seconds) external view ... {
    SchedulerStructs.SubscriptionStatus memory status = _state.subscriptionStatuses[subscriptionId];
    if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
        revert PythErrors.StalePrice();
    prices = this.getPricesUnsafe(subscriptionId, priceIds);  // returns per-feed stored prices
}
``` [3](#0-2) 

`getPricesUnsafe` returns the raw stored `PriceFeed.price` for each requested ID, which carries its own individual `publishTime`: [4](#0-3) 

The contract explicitly accommodates closed-market feeds (e.g., equities) whose `publishTime` reflects the last trading period, potentially days in the past:

> *"some asset markets may be closed. Closed markets will have a publishTime from their last trading period."* [5](#0-4) 

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is only 1 hour and is enforced against the **max** timestamp, not individual feed timestamps: [6](#0-5) 

**Concrete scenario:**

A subscription contains two feeds:
- Feed A (BTC/USD, 24/7): `publishTime = now - 30s`
- Feed B (AAPL/USD, market closed): `publishTime = now - 8 hours`

After `updatePriceFeeds`, `priceLastUpdatedAt = now - 30s` (the max).

A caller invokes `getPricesNoOlderThan(subscriptionId, [feedB_id], 60)`:
1. Check: `distance(now, now-30s) = 30s ≤ 60s` → **passes**
2. Returns Feed B's price with `publishTime = now - 8 hours` — **8 hours stale**

The `age_seconds` parameter provides no actual per-feed freshness guarantee.

---

### Impact Explanation

Any DeFi protocol (lending, derivatives, perpetuals) integrating with the Scheduler via `getPricesNoOlderThan` and relying on its staleness guarantee for individual feeds will silently receive prices that are arbitrarily old. This leads to incorrect collateral valuations, mispriced liquidations, or incorrect settlement prices — the exact class of harm described in the reference report. The impact is **incorrect valuations and decision-making** within downstream systems.

---

### Likelihood Explanation

The Scheduler is explicitly designed to support mixed subscriptions containing both 24/7 crypto feeds and market-hours-limited feeds. Any subscription with at least one non-24/7 feed is affected whenever that market is closed. An unprivileged caller (any whitelisted reader, or any address when `whitelistEnabled = false`) can call `getPricesNoOlderThan` and receive stale per-feed data. No special access is required. [7](#0-6) 

---

### Recommendation

Replace the subscription-level staleness check in `getPricesNoOlderThan` (and `getEmaPricesNoOlderThan`) with a per-feed check against each individual feed's `publishTime`:

```solidity
function getPricesNoOlderThan(..., uint256 age_seconds) external view ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

Alternatively, document clearly that `getPricesNoOlderThan` only guarantees the subscription was updated within `age_seconds` (not that each individual feed's price is that fresh), and require callers to inspect `publishTime` on each returned `Price` struct.

---

### Proof of Concept

1. Create a subscription with two feeds: a 24/7 crypto feed and a market-hours equity feed.
2. Call `updatePriceFeeds` with valid update data where the equity feed's `publishTime` is 8 hours ago (last trading period) and the crypto feed's `publishTime` is `now - 30s`. Both share the same Pythnet slot, so `PriceSlotMismatch` is not triggered.
3. `priceLastUpdatedAt` is set to `now - 30s`.
4. Call `getPricesNoOlderThan(subscriptionId, [equityFeedId], 60)`.
5. The staleness check passes (`distance(now, now-30s) = 30 ≤ 60`).
6. The returned `Price.publishTime` is `now - 8 hours`, demonstrating that a price 8 hours stale is returned through a function named `getPricesNoOlderThan` with `age_seconds = 60`. [3](#0-2) [8](#0-7)
