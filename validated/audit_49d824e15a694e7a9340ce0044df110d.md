### Title
Fallback `estimate_fee_rate` Includes Already-Proposed Transactions, Inflating Fee Rate Estimates - (`tx-pool/src/component/pool_map.rs`)

### Summary

The fallback fee rate estimator in `pool_map.rs` iterates over **all** pool entries (including `Gap` and `Proposed` status transactions) when its own documented assumption states that all transactions are "waiting to be proposed." This causes the estimate to treat already-proposed transactions as if they are still competing for block space, inflating the returned fee rate.

### Finding Description

The fallback `estimate_fee_rate` in `PoolMap` simulates filling blocks with pool transactions sorted by fee rate to find the minimum fee rate needed to be included within `target_blocks` blocks.

The documented assumption for this algorithm is explicit:

> "all transactions in the pool are waiting to be proposed" [1](#0-0) 

However, the implementation iterates over **all** entries regardless of status:

```rust
pub(crate) fn estimate_fee_rate(
    &self,
    mut target_blocks: usize,
    ...
) -> FeeRate {
    let iter = self.entries.iter_by_score().rev();  // ALL statuses: Pending, Gap, Proposed
    ...
}
``` [2](#0-1) 

The pool has three distinct statuses:
- `Pending` â€” waiting to be proposed
- `Gap` â€” proposed but not yet in the committable window
- `Proposed` â€” proposed and committable in the next block(s) [3](#0-2) 

`Gap` and `Proposed` transactions are **already past the proposal stage**. They are not "waiting to be proposed." Including them in the simulation makes the algorithm believe there are more transactions competing for block space than there actually are.

Compare with `get_proposals`, which correctly filters to only `Status::Pending`: [4](#0-3) 

And `score_sorted_iter_by_status`, which provides the correct filtered iterator: [5](#0-4) 

The fallback is invoked from `TxPool::estimate_fee_rate` when the primary estimator (`ConfirmationFraction` or `WeightUnitsFlow`) fails: [6](#0-5) 

Which is triggered via the public `estimate_fee_rate` RPC: [7](#0-6) 

### Impact Explanation

When `Gap` and `Proposed` transactions are included in the simulation, they consume "virtual" block space in the fee-rate calculation. This makes the algorithm conclude that a higher fee rate is needed to be included within the target number of blocks. The result is an **inflated fee rate estimate** returned to callers.

Users and external tools (wallets, dApps) that rely on `estimate_fee_rate` will overpay transaction fees. The inflation is proportional to the number of proposed/gap transactions in the pool. During periods of high activity (many proposed transactions), the overestimate can be significant.

### Likelihood Explanation

The fallback algorithm is used whenever the primary estimator returns an error (e.g., insufficient historical data, node just started, or IBD state). The `enable_fallback` parameter defaults to `true`, making this the common path for new or recently-restarted nodes. [8](#0-7) 

Any unprivileged RPC caller can trigger this path by calling `estimate_fee_rate`. A transaction submitter who floods the pool with transactions that get proposed can amplify the inflation effect.

### Recommendation

Change `pool_map.estimate_fee_rate` to iterate only over `Pending` (and optionally `Gap`) entries, consistent with the documented assumption that all pool transactions are "waiting to be proposed":

```rust
pub(crate) fn estimate_fee_rate(
    &self,
    mut target_blocks: usize,
    max_block_bytes: usize,
    max_block_cycles: Cycle,
    min_fee_rate: FeeRate,
) -> FeeRate {
    debug_assert!(target_blocks > 0);
-   let iter = self.entries.iter_by_score().rev();
+   let iter = self.entries.iter_by_score().rev()
+       .filter(|entry| entry.status == Status::Pending);
    ...
}
```

### Proof of Concept

1. Submit a large number of transactions to the pool so they get proposed (status transitions to `Proposed`).
2. Call `estimate_fee_rate` via RPC with `enable_fallback: true` while the primary estimator is not ready (e.g., on a freshly started node).
3. The returned fee rate will be higher than if only `Pending` transactions were counted, because the proposed transactions consume virtual block space in the simulation.
4. Compare the result against a call made when the pool has the same transactions but all in `Pending` status â€” the estimate will differ despite the same underlying fee competition.

```rust
    pub(crate) fn estimate_fee_rate(
        &self,
        mut target_blocks: usize,
        max_block_bytes: usize,
        max_block_cycles: Cycle,
        min_fee_rate: FeeRate,
    ) -> FeeRate {
        debug_assert!(target_blocks > 0);
        let iter = self.entries.iter_by_score().rev();
        let mut current_block_bytes = 0;
        let mut current_block_cycles = 0;
        for entry in iter {
            current_block_bytes += entry.inner.size;
            current_block_cycles += entry.inner.cycles;
            if current_block_bytes >= max_block_bytes || current_block_cycles >= max_block_cycles {
                target_blocks -= 1;
                if target_blocks == 0 {
                    return entry.inner.fee_rate();
                }
                current_block_bytes = entry.inner.size;
                current_block_cycles = entry.inner.cycles;
            }
        }

        min_fee_rate
    }
