### Title
Permanent Rent Lock of Relayer's `SignatureSet` via `post_vaa` Front-Running — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

### Summary
An unprivileged attacker can permanently lock a relayer's `SignatureSet` rent lamports by racing to call `post_vaa` first with a different `signature_set` keypair for the same VAA. Because `close_signature_set` requires the existing `posted_vaa` to point (`has_one`) to the victim's `signature_set`, and the PDA slot is already occupied by the attacker's version, the victim's `signature_set` account can never be closed.

### Finding Description
The legacy `post_vaa` flow works as follows:

1. A relayer creates a fresh keypair for a `SignatureSet` account and calls `verify_signatures` (multiple transactions) to accumulate guardian signature verifications into it.
2. The relayer calls `post_vaa`, which `init`-creates a `PostedVaaV1` PDA at `[b"PostedVAA", message_hash]` and writes the relayer's `signature_set` pubkey into `PostedVaaV1Info.signature_set`.

Because the `PostedVaaV1` PDA address depends only on the VAA body hash (public data), and because `signature_set` is a free keypair (not a PDA), any party can independently create a second valid `SignatureSet` for the same VAA and race to call `post_vaa` first. VAA data is publicly available from Wormhole guardians, so the attacker does not need to observe the victim's mempool — they can prepare their competing `SignatureSet` at any time.

Once the attacker's `post_vaa` lands first:
- The `PostedVaaV1` PDA exists with `signature_set = attacker_keypair`.
- The victim's `post_vaa` reverts (Anchor `init` on an already-initialized account).
- `close_signature_set` enforces `has_one = signature_set`, so it can only be called with the attacker's `signature_set`, not the victim's.
- No other instruction in the program closes a `SignatureSet`.
- The victim's `SignatureSet` rent is permanently frozen.

The attacker's own `signature_set` can be closed normally (it is the one referenced in `posted_vaa`), so the attacker's net cost is only transaction fees.

### Impact Explanation
The victim relayer permanently loses the SOL rent deposited in their `SignatureSet` account. A `SignatureSet` for a 19-guardian set is approximately `8 + 4 + 19 + 32 + 4 = 67` bytes plus discriminator overhead, costing roughly 0.001–0.002 SOL in rent. The attack can be repeated for every VAA a target relayer attempts to post, compounding the loss. The invariant that "all rent paid by a relayer must be recoverable" is violated.

### Likelihood Explanation
The attack requires no privileged access. VAA data is public. The attacker only needs to:
1. Observe that a relayer is in the middle of `verify_signatures` for a given VAA (or simply race any VAA they see on-chain).
2. Create their own `SignatureSet` keypair and call `verify_signatures` + `post_vaa` before the victim.

On Solana, transaction ordering within a slot is not guaranteed to favor the original submitter, and validators can reorder transactions. The attack is cheap (a few transaction fees) and repeatable.

### Recommendation
Two complementary fixes:

1. **Bind `signature_set` to the payer/relayer at creation time**, or derive it as a PDA that includes the payer pubkey, so two relayers cannot produce competing accounts for the same VAA.
2. **Add a fallback close path** in `close_signature_set` (or a new instruction) that allows closing an orphaned `SignatureSet` whose message hash matches a `posted_vaa` that does not reference it — returning lamports to the original payer stored in the `SignatureSet` account.

Alternatively, change `post_vaa` to use `init_if_needed` and verify that the existing account's `signature_set` field matches the caller's, reverting gracefully rather than failing the entire transaction.

### Proof of Concept
State-transition test:

```
1. Alice: create keypair sig_set_alice; call verify_signatures(vaa_data, sig_set_alice) → quorum reached
2. Bob:   create keypair sig_set_bob;   call verify_signatures(vaa_data, sig_set_bob)   → quorum reached
3. Bob:   call post_vaa(sig_set_bob, vaa_data) → PostedVaaV1 PDA created; posted_vaa.signature_set = sig_set_bob
4. Alice: call post_vaa(sig_set_alice, vaa_data) → FAILS: account already initialized
5. Alice: call close_signature_set(posted_vaa, sig_set_alice) → FAILS: has_one constraint (posted_vaa.signature_set ≠ sig_set_alice)
6. Assert: sig_set_alice.lamports > 0 and no instruction exists to recover them → permanent rent lock confirmed
```
