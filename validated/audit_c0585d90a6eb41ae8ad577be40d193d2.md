### Title
Compressed Frame Bypasses Per-Protocol `max_frame_length`, Delivering Up to 8 MB to Upper Handlers ŌĆö (`network/src/compress.rs`)

### Summary
`LengthDelimitedCodecWithCompress::decode` enforces `max_frame_length` only on the compressed wire frame. After decompression, the result is bounded solely by `MAX_UNCOMPRESSED_LEN` (8 MB). An unprivileged remote peer can send a snappy-compressed frame whose wire size is within any protocol's declared limit but whose decompressed output approaches 8 MB, violating the invariant that upper handlers never receive a message larger than the protocol's declared `max_frame_length`.

### Finding Description

`CKBProtocol::build()` constructs a `LengthDelimitedCodecWithCompress` with `compress: true` (the default for all protocols) and passes the per-protocol `max_frame_length` to the inner `LengthDelimitedCodec`: [1](#0-0) 

The per-protocol limits are small for lightweight protocols: [2](#0-1) 

In `decode()`, the inner codec enforces `max_frame_length` on the wire frame, but after decompression the only guard is against `MAX_UNCOMPRESSED_LEN` (8 MB): [3](#0-2) 

The decompressed buffer is returned directly to the upper handler with no check against `max_frame_length`: [4](#0-3) 

By contrast, the encoder-side `process()` does enforce `max_frame_length` before writing: [5](#0-4) 

This asymmetry means the decoder accepts decompressed payloads up to 8 MB regardless of the protocol's declared limit.

### Impact Explanation

For protocols with small `max_frame_length` values (Ping=1 KB, Identify=2 KB, Time=1 KB, Feeler=1 KB, DisconnectMessage=1 KB), an attacker can deliver up to ~8 MB per frame to the upper handler. Effects include:

- **Memory pressure**: each malicious frame allocates up to 8 MB (`BytesMut::zeroed(decompressed_bytes_len)` at line 242), scalable across many connections.
- **Unbounded handler processing**: handlers like `PingHandler` and `IdentifyProtocol` receive buffers orders of magnitude larger than expected, potentially triggering downstream flatbuffers/molecule parsing panics or excessive CPU work.
- **No per-connection rate limiting** prevents an attacker from repeating this across many frames.

### Likelihood Explanation

- Compression is enabled by default (`compress: true`) for all `CKBProtocol` instances.
- Any unprivileged peer can connect on any protocol ŌĆö no authentication required.
- Snappy achieves very high compression ratios on repetitive data; compressing ~8 MB of repeated bytes to under 1 KB is straightforward.
- The attack requires only a single crafted P2P frame.

### Recommendation

After decompression, add a bounds check against `max_frame_length` before returning the buffer:

```rust
// In decode(), after decompression succeeds:
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
    return Err(io::Error::new(io::ErrorKind::InvalidData, "decompressed data too large"));
}
```

This mirrors the existing encoder-side guard in `process()`. [6](#0-5) 

### Proof of Concept

For the Ping protocol (`max_frame_length = 1024`):

1. Craft a snappy stream compressing ~8 MB ŌłÆ 1 bytes of `0x41` (repetitive) to under 1023 bytes (trivially achievable; snappy compresses such input to ~0.01% of original size).
2. Prepend `COMPRESS_FLAG` (0x80) as the first byte.
3. Prepend a 4-byte big-endian length header for the total wire frame (Ōēż 1024 bytes).
4. Send this single frame to a CKB node on the `/ckb/ping` protocol.
5. Assert that `decode()` returns `Ok(Some(buf))` where `buf.len() Ōēł 8 MB >> 1024`.

The `length_delimited.decode()` call passes (wire size Ōēż 1024), the `MAX_UNCOMPRESSED_LEN` check passes (< 8 MB), and the 8 MB buffer is returned to `PingHandler`. [7](#0-6) [8](#0-7)
