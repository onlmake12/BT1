The code in `kill_process` has a concrete, verifiable logic defect. Let me trace it precisely.

The match arm ordering makes the SIGKILL branch structurally unreachable while the target process is alive. Here is the precise analysis:

**Match arm evaluation order in Rust is top-to-bottom, first match wins.**

- Arm 1 `Ok(_)` — fires whenever `check_process` returns `Ok` (i.e., process is alive). `wait_time` is decremented with **no guard**. This arm has unconditional priority.
- Arm 2 `_ if wait_time <= 0` — fires only when `check_process` returns `Err` (process is gone) AND `wait_time <= 0`. It is **never reached while the process is alive**.
- Arm 3 `_` — fires when `check_process` returns `Err` and `wait_time > 0` (process died normally).

As long as the target process is alive, `check_process` returns `Ok(_)`, arm 1 always wins, `wait_time` decrements from 60 → 0 → -1 → -2 → ... indefinitely. The SIGKILL branch is dead code for a live, SIGTERM-ignoring process. [1](#0-0) [2](#0-1)

### Title
`ckb daemon --stop` infinite loop — SIGKILL branch unreachable for SIGTERM-ignoring process — (`ckb-bin/src/subcommand/daemon.rs`)

### Summary
`kill_process` sends SIGTERM then enters a loop polling `check_process`. The SIGKILL escalation arm is guarded by `_ if wait_time <= 0`, which only matches `Err` results (process gone). While the process is alive, `check_process` returns `Ok(_)`, which always matches the first arm unconditionally, decrementing `wait_time` past zero forever. The command never exits.

### Finding Description
In `kill_process` (`ckb-bin/src/subcommand/daemon.rs`, lines 59–86):

```rust
let mut wait_time = 60;
loop {
    let res = check_process(pid_file);
    match res {
        Ok(_) => {                        // arm 1: process alive — always wins
            wait_time -= 1;               // decrements past 0 with no bound check
            std::thread::sleep(...1s);
        }
        _ if wait_time <= 0 => {          // arm 2: only reachable when process is GONE
            kill(...SIGKILL)...;          // ← structurally dead while process is alive
            break;
        }
        _ => { break; }                   // arm 3: process gone, normal exit
    }
}
```

The intended invariant — "send SIGKILL after 60 seconds if process has not exited" — is never enforced. The `wait_time <= 0` guard is on the wrong match arm. It applies to `Err` (process not found), not to `Ok` (process still running). [3](#0-2) 

### Impact Explanation
`ckb daemon --stop` hangs indefinitely. The operator loses the ability to stop the node via the CLI. The pid file is never removed (line 23 is never reached), so subsequent `--stop` invocations also hang. The only recovery is a manual `kill -9`. [4](#0-3) 

### Likelihood Explanation
Any scenario where the CKB process is slow or unable to handle SIGTERM (e.g., blocked in a syscall, signal handler not yet installed at startup, or a future regression in signal handling) triggers this. An unprivileged local operator running `ckb daemon --stop` is the exact intended use case. No special privileges, no network access, and no cryptographic material are required.

### Recommendation
Move the timeout check into the `Ok(_)` arm:

```rust
Ok(_) if wait_time > 0 => {
    wait_time -= 1;
    std::thread::sleep(Duration::from_secs(1));
}
Ok(_) => {
    // wait_time exhausted, process still alive → SIGKILL
    kill(Pid::from_raw(pid), Some(Signal::SIGKILL))...;
    break;
}
_ => { break; } // process exited normally
```

### Proof of Concept
```bash
# 1. Start a SIGTERM-ignoring process
bash -c 'trap "" TERM; sleep 3600' &
VICTIM_PID=$!

# 2. Write its PID to the CKB pid file
echo $VICTIM_PID > /tmp/ckb.pid

# 3. Run daemon --stop (supply the pid file path via config)
timeout 120 ckb daemon --stop --pid-file /tmp/ckb.pid
echo "exit code: $?"   # prints 124 (timeout), command never returned
```

The command never returns; `timeout` kills it after 120 s, confirming the infinite loop.
