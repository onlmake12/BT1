### Title
Unbounded While Loop in `executeCallback` Enables Gas-Exhaustion DoS — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past all fulfilled (inactive) sequence numbers. An unprivileged attacker can pre-fill the sequence-number space by creating many requests and fulfilling all but the earliest one, causing any subsequent `executeCallback` for that earliest request to exhaust the block gas limit and revert — permanently locking the requester's funds.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs the following loop:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a
// storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

The loop iterates one storage read (`findRequest`) per sequence number until it finds the next active request. The developers themselves flag this as a gas concern in the inline TODO. `firstUnfulfilledSeq` is a plain `uint64` with no upper bound enforced by the contract. [2](#0-1) 

`requestPriceUpdatesWithCallback` is open to any caller; it only requires a fee payment and a registered provider. [3](#0-2) 

After `_state.exclusivityPeriodSeconds` elapses, `executeCallback` is also open to any caller — no provider check is enforced. [4](#0-3) 

**Attack path:**

1. Attacker calls `requestPriceUpdatesWithCallback` N times, obtaining sequence numbers 1 … N.
2. After the exclusivity window, the attacker calls `executeCallback` for sequences 2 … N (providing valid historical Pyth update data, which is publicly available via Hermes). Each call advances `firstUnfulfilledSeq` only up to sequence 1, which remains active.
3. `firstUnfulfilledSeq` is now pinned at 1.
4. When the legitimate provider (or anyone) attempts to fulfill sequence 1, the while loop must iterate through all N−1 already-fulfilled slots, performing N−1 cold storage reads. For a sufficiently large N this exceeds the block gas limit and the entire transaction reverts.

The developers already note the consequence of a revert in a sibling TODO:

> "if this effect occurs here, we need to guarantee that `executeCallback` can never revert. If `executeCallback` can revert, then funds can be permanently locked in the contract." [5](#0-4) 

---

### Impact Explanation

- `executeCallback` for the targeted sequence number reverts on every attempt, so the requester's callback is never delivered.
- The requester's fee (stored in `req.fee`) is never returned; the contract has no refund path for a permanently un-fulfillable request.
- The provider assigned to that request also cannot collect their fee, and no penalty mechanism exists to compensate them.

This renders the Echo price-update-with-callback service non-functional for any request that an attacker chooses to target, matching the "contract rendered useless" severity of the reference report.

---

### Likelihood Explanation

- Both entry points (`requestPriceUpdatesWithCallback` and post-exclusivity `executeCallback`) require no privileged role — any EOA can call them.
- Historical Pyth price data for arbitrary past timestamps is freely available from Hermes, so the attacker can supply valid `updateData` for each of the N−1 fulfillment calls.
- The only cost to the attacker is N request fees plus N−1 `executeCallback` gas costs. If the configured `pythFeeInWei` and provider fees are low (as they are in typical deployments), the attack is economically viable.
- The attack is permanent: once `firstUnfulfilledSeq` is pinned behind a targeted sequence, no on-chain mechanism resets it without fulfilling that sequence — which itself reverts.

---

### Recommendation

Replace the unbounded while loop with a bounded or lazy advancement strategy:

1. **Doubly-linked list** (suggested by the in-code TODO): maintain a linked list of active requests so `firstUnfulfilledSeq` can be updated in O(1).
2. **Lazy pointer**: remove the while loop entirely; let `getFirstActiveRequests` scan forward off-chain and have keepers pass the correct sequence number directly.
3. **Cap iterations**: add a hard upper bound (e.g., `MAX_SCAN = 256`) to the while loop so a single `executeCallback` call can never scan more than a fixed number of slots.

---

### Proof of Concept

```solidity
// Attacker script (pseudo-Solidity)
uint64 VICTIM_SEQ = 1; // first request, created by a legitimate user

// Step 1: create N requests (attacker pays fees)
for (uint i = 0; i < N; i++) {
    echo.requestPriceUpdatesWithCallback{value: fee}(
        provider, block.timestamp, priceIds, callbackGas
    );
}
// Sequence numbers: VICTIM_SEQ=1, attacker owns 2..N+1

// Step 2: wait for exclusivity period to expire, then fulfill 2..N+1
for (uint64 seq = 2; seq <= N + 1; seq++) {
    bytes[] memory updateData = fetchHermesUpdate(seq); // public API
    echo.executeCallback(anyAddress, seq, updateData, priceIds);
    // firstUnfulfilledSeq stays at VICTIM_SEQ=1 after each call
}

// Step 3: victim's provider tries to fulfill sequence 1
// The while loop now iterates N times → OOG → revert
echo.executeCallback(provider, VICTIM_SEQ, victimUpdateData, priceIds); // REVERTS
