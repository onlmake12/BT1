Audit Report

## Title
Tx-Pool `min_fee_rate` Admission Check Uses Serialized Size Instead of Weight, Enabling Low-Cost CPU Exhaustion — (File: `tx-pool/src/util.rs`)

## Summary
`check_tx_fee` enforces `min_fee_rate` using only the transaction's serialized byte size, while post-verification fee-rate accounting uses `get_transaction_weight(size, cycles)`. For any transaction where `cycles × DEFAULT_BYTES_PER_CYCLES > size`, the transaction is admitted at a fee far below the configured `min_fee_rate` when measured by weight. This allows an unprivileged attacker to submit high-cycle, small-size transactions that pass the admission gate cheaply while forcing the node to spend up to `max_tx_verify_cycles` of CPU per transaction.

## Finding Description

**Root cause — size-only admission check:**

`check_tx_fee` in `tx-pool/src/util.rs` lines 42–45 computes the minimum required fee using only `tx_size`. The code comment explicitly acknowledges the discrepancy:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

This check is called during `pre_check` in `tx-pool/src/process.rs` before `verify_rtx` has run and before actual cycles are known: [2](#0-1) 

After `verify_rtx` completes, the `TxEntry` is created with the actual verified cycles: [3](#0-2) 

The entry's actual fee rate is then computed using `get_transaction_weight`: [4](#0-3) 

Where `get_transaction_weight` is: [5](#0-4) 

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`: [6](#0-5) 

The weight exceeds the size whenever `cycles > size / 0.000_170_571_4 ≈ size × 5862`. For a 1000-byte transaction, this threshold is ~5.86M cycles — well within the default `max_tx_verify_cycles = 70_000_000`.

**Exploit flow:**

1. Attacker crafts a transaction: `tx_size = 1000` bytes, script consumes `cycles = 70_000_000`.
2. For remote submission, `declared_cycles = 70_000_000` must be declared (lines 736–748 enforce `declared == verified.cycles` or the tx is rejected). For local RPC, no declaration is needed.
3. `check_tx_fee` computes `min_fee = min_fee_rate × 1000 / 1000 = 1000 shannons`. Attacker sets `fee = 1000 shannons`. Check passes.
4. `verify_rtx` runs, consuming 70M cycles of node CPU. Verification succeeds.
5. `TxEntry` is created with `cycles = 70_000_000`. Actual weight = `max(1000, 70_000_000 × 0.000_170_571_4) = 11,940`. Actual fee rate = `1000 × 1000 / 11940 ≈ 83 shannons/KW` — **12× below `min_fee_rate`**.
6. The entry is admitted to the pool.

**Why existing guards are insufficient:**

The verify queue size limit (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256_000_000`) is based on serialized tx size, not cycles: [7](#0-6) [8](#0-7) 

A 1000-byte transaction occupies only 1000 bytes of queue budget, so up to ~256,000 such transactions can be queued simultaneously, each requiring 70M cycles of CPU to verify.

`limit_size` evicts by weight-based fee rate (lowest first), so these transactions are evicted eventually — but only after verification CPU has already been consumed: [9](#0-8) 

`estimate_fee_rate` iterates entries by score (weight-based), so admitted low-effective-fee-rate entries skew the estimate downward: [10](#0-9) 

## Impact Explanation

This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

The attack is asymmetric: the attacker pays only the size-based minimum fee (e.g., 1000 shannons per transaction) while forcing the node to spend up to 70M cycles of CPU per transaction for verification. The verify queue can hold ~256,000 such 1000-byte transactions simultaneously. The CPU cost to the node is unbounded relative to the attacker's fee expenditure, enabling sustained CPU exhaustion of individual nodes and, if applied broadly, network-wide congestion.

## Likelihood Explanation

The exploit path is fully open to any RPC caller (`send_transaction`) or relay peer (`submit_remote_tx`). No special privilege is required. A RISC-V script that loops to consume cycles without increasing transaction size is trivial to write. The discrepancy grows linearly with cycles, so the bypass is maximally effective at `max_tx_verify_cycles`. The attacker does need live cells to spend (one per transaction), which is a non-trivial but achievable constraint for a sustained attack.

## Recommendation

Replace the size-only proxy in `check_tx_fee` with a weight-based check. Since cycles are not yet known at pre-check time, use the declared cycles (for remote transactions) or `max_tx_verify_cycles` as a conservative upper bound:

```rust
// Use declared_cycles if available, else use max_tx_verify_cycles as upper bound
let weight = get_transaction_weight(tx_size, declared_cycles.unwrap_or(max_tx_verify_cycles));
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

Additionally, the verify queue size limit should account for declared cycles (not just serialized size) to prevent CPU exhaustion via queue flooding.

## Proof of Concept

1. Deploy a CKB script that loops for ~60M cycles (tight RISC-V loop), keeping the script binary small (~200 bytes).
2. Construct a transaction spending a live cell, referencing the script as a lock, with `tx_size ≈ 1000` bytes.
3. Set `fee = min_fee_rate × tx_size / 1000` (e.g., 1000 shannons at `min_fee_rate = 1000 shannons/KW`).
4. Submit via `send_transaction` RPC (or relay with `declared_cycles = actual_cycles`).
5. Observe: transaction is accepted into the pool.
6. Query `get_raw_tx_pool`, compute `fee / get_transaction_weight(size, cycles)` — actual fee rate is ~83 shannons/KW, far below `min_fee_rate = 1000`.
7. Repeat with many such transactions (each spending a different live cell); observe node CPU saturation and `estimate_fee_rate` returning artificially low values.
