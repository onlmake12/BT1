### Title
Unbounded Growth of `pending_get_block_proposals` via Unauthenticated `GetBlockProposal` P2P Messages — (`sync/src/types/mod.rs`, `sync/src/relayer/get_block_proposal_process.rs`)

---

### Summary

The `SyncState::insert_get_block_proposals` function inserts peer-supplied proposal IDs into the `pending_get_block_proposals` `DashMap` with no total-size guard. Any connected peer can send repeated `GetBlockProposal` relay messages containing proposal IDs that do not exist in the local tx-pool. Each such message passes the per-message length check, but the accumulated entries in `pending_get_block_proposals` are never bounded. The periodic `drain_get_block_proposals` call then clones and iterates the entire unbounded map, creating both a memory-exhaustion and a CPU-overhead DoS path.

---

### Finding Description

**Root cause — `insert_get_block_proposals` has no size cap**

`SyncState::insert_get_block_proposals` unconditionally inserts every supplied `ProposalShortId` into the `pending_get_block_proposals` `DashMap`:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [1](#0-0) 

There is no check on `self.pending_get_block_proposals.len()` before or after insertion, and no per-peer quota.

**Attacker-controlled entry path — `GetBlockProposalProcess::execute`**

When a peer sends a `GetBlockProposal` relay message, `GetBlockProposalProcess::execute` validates only the *per-message* proposal count against the consensus limit:

```rust
let limit = shared.consensus().max_block_proposals_limit()
    * (shared.consensus().max_uncles_num() as u64);
if message_len as u64 > limit {
    return StatusCode::ProtocolMessageIsMalformed...
}
``` [2](#0-1) 

Proposals that are not found in the local tx-pool are then forwarded to `insert_get_block_proposals`:

```rust
let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
    .into_iter()
    .filter(|short_id| !fetched_transactions.contains_key(short_id))
    .collect();

self.relayer
    .shared()
    .state()
    .insert_get_block_proposals(self.peer, not_exist_proposals);
``` [3](#0-2) 

There is no rate limit, no per-peer quota, and no total-map-size guard across successive messages.

**Unbounded iteration — `drain_get_block_proposals`**

The relayer's periodic timer calls `drain_get_block_proposals`, which clones the entire map (O(n) allocation) and clears it:

```rust
pub fn drain_get_block_proposals(
    &self,
) -> DashMap<packed::ProposalShortId, HashSet<PeerIndex>> {
    let ret = self.pending_get_block_proposals.clone();
    self.pending_get_block_proposals.clear();
    ret
}
``` [4](#0-3) 

If the map has been inflated by an attacker, this clone is proportionally expensive in both memory and CPU.

**Contrast with protected structures**

The analogous `unknown_tx_hashes` structure is explicitly bounded — `add_ask_for_txs` enforces `MAX_UNKNOWN_TX_HASHES_SIZE` and `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`: [5](#0-4) 

`pending_get_block_proposals` has no equivalent guard.

---

### Impact Explanation

An attacker who connects as a peer can send a continuous stream of `GetBlockProposal` messages, each carrying up to `max_block_proposals_limit × max_uncles_num` (1 500 × 2 = 3 000 on mainnet) fabricated `ProposalShortId` values (10 bytes each). Because none of these IDs exist in the local tx-pool, all of them are inserted into `pending_get_block_proposals`. At 3 000 entries per message and no rate limit, the map grows at ~30 KB per message. Sustained flooding exhausts node memory and causes the periodic drain to perform an increasingly expensive clone-and-iterate, degrading or halting normal relay and block-assembly operations for all legitimate peers.

---

### Likelihood Explanation

The attack requires only a standard P2P connection — no privileged key, no majority hashpower, no social engineering. The `GetBlockProposal` message type is part of the normal relay protocol (`RelayV3`), so any peer can send it. The per-message length check is the only guard, and it does not prevent repeated messages. The CHANGELOG records prior memory-bloat fixes in the same area (`#3093` inflight proposals, `#3094` inflight blocks, `#3110` compact blocks), confirming that unbounded-growth bugs in sync-state collections are a realistic and recurring concern in this codebase.

---

### Recommendation

Apply a total-size cap inside `insert_get_block_proposals`, analogous to the guard already present in `add_ask_for_txs`. For example:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    const MAX_PENDING_GET_BLOCK_PROPOSALS: usize = 10_000;
    if self.pending_get_block_proposals.len() >= MAX_PENDING_GET_BLOCK_PROPOSALS {
        return;
    }
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
```

Additionally, consider adding a per-peer quota so a single peer cannot monopolise the budget, and consider disconnecting or temporarily banning peers that repeatedly trigger the cap.

---

### Proof of Concept

1. Connect to a CKB node as a relay peer (standard `RelayV3` handshake).
2. In a tight loop, send `GetBlockProposal` messages each containing `max_block_proposals_limit × max_uncles_num` (3 000) randomly generated `ProposalShortId` values that do not correspond to any real transaction.
3. Because none of the IDs exist in the node's tx-pool, all are inserted into `pending_get_block_proposals` without any size check.
4. Observe node memory growing proportionally to the number of messages sent.
5. When the relayer's periodic timer fires, `drain_get_block_proposals` clones the entire inflated map, causing a CPU spike and potentially blocking the relay thread long enough to stall block propagation for legitimate peers.
