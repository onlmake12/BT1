Audit Report

## Title
Unauthenticated `from` PeerId in `ConnectionRequest` Allows `pending_delivered` Poisoning with Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

The `respond_delivered` function in `ConnectionRequestProcess` inserts attacker-supplied `listen_addrs` into `pending_delivered` keyed by the message's `from` PeerId, which is parsed purely from message bytes and never verified against the actual sending session's PeerId. Any connected peer can spoof an arbitrary `from` PeerId and inject attacker-controlled TCP addresses into the map. By subsequently sending a `ConnectionSync` with the same spoofed `from`, the victim node spawns unbounded concurrent 30-second NAT traversal tasks to attacker-controlled addresses via `runtime::spawn`, exhausting async task and TCP socket resources and crashing the node.

## Finding Description

**Root cause — unauthenticated `from` field:**

In `connection_request.rs` lines 36–38, `from` is deserialized from message bytes with no binding to the actual session:

```rust
let from = PeerId::from_bytes(value.from().raw_data().to_vec()).map_err(|_| {
    StatusCode::InvalidFromPeerId.with_context("the from peer id is invalid")
})?;
```

The `ConnectionRequestProcess` struct holds `peer: PeerIndex` (the actual session ID) but `execute()` at line 145 routes to `respond_delivered` solely based on `content.to == self_peer_id`, passing `content.from` directly without checking it against the session's actual PeerId:

```rust
if self_peer_id == &content.to {
    self.respond_delivered(content.from, &content.to, content.listen_addrs)
        .await
```

**Insufficient guard in `respond_delivered`:**

Lines 161–167 only reject re-insertion when an existing entry is less than `HOLE_PUNCHING_INTERVAL` (2 minutes) old. For a fresh `from_peer_id` (any spoofed PeerId the attacker has not used in the last 2 minutes), the guard is entirely absent. The unconditional overwrite at lines 234–237 then executes:

```rust
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
```

The `remote_listens` are the attacker-supplied `listen_addrs` filtered only to TCP/IPv4/IPv6 — a filter that does not prevent attacker-controlled addresses from passing.

**Consumption of the poisoned entry:**

In `connection_sync.rs`, `SyncContent.from` is also parsed from message bytes with no session binding (lines 42–44). `ConnectionSyncProcess` has no `peer` field at all, making session-based verification structurally impossible. At lines 111–115, `pending_delivered` is looked up by `content.from`:

```rust
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
```

These addresses are passed to `try_nat_traversal` and each spawned as an independent `runtime::spawn` task (lines 119–124, 145). Each task retries TCP connections for up to 30 seconds.

**Full exploit flow:**

1. Attacker (connected peer E) sends `ConnectionRequest { from: A, to: V, listen_addrs: [attacker_addr] }` directly to V.
2. V's `execute()` sees `content.to == self_peer_id`, calls `respond_delivered(A, V, [attacker_addr])`.
3. Guard passes (no prior entry for A). V inserts `pending_delivered[A] = ([attacker_addr], now)`.
4. Attacker sends `ConnectionSync { from: A, to: V, route: [] }` to V.
5. V looks up `pending_delivered[A]`, gets `[attacker_addr]`, spawns `runtime::spawn(try_nat_traversal(..., attacker_addr))` for 30 seconds.
6. Repeating with distinct spoofed `from` PeerIds (B, C, …, up to 30/sec per the `rate_limiter`) accumulates 30 × 30 = 900 concurrent tasks per attacker session after 30 seconds, each creating a new TCP socket every ~200ms.

**Why existing guards fail:**

- The `rate_limiter` (30 req/sec per `(session_id, msg_item_id)`) bounds the injection rate but does not prevent accumulation: 30 tasks/sec × 30 sec lifetime = 900 concurrent tasks per session.
- The `forward_rate_limiter` (1 req/sec per `(from, to, msg_item_id)`) is bypassed by using distinct spoofed `from` PeerIds — each new PeerId is a fresh key with no prior rate limit state.
- The `HOLE_PUNCHING_INTERVAL` guard only prevents rapid re-poisoning of the same `from_peer_id`; it does not prevent initial poisoning or poisoning with fresh PeerIds.
- The `notify` cleanup retains entries for 5 minutes (`TIMEOUT`), meaning poisoned entries persist and can be triggered by a delayed `ConnectionSync`.

## Impact Explanation

The concrete impact is resource exhaustion leading to a node crash. Each `try_nat_traversal` task is spawned unconditionally via `runtime::spawn` with no semaphore or task counter. Each task runs for up to 30 seconds and creates a new `TcpSocket` every ~200ms (~150 socket operations per task). With 900 concurrent tasks per attacker session, this generates ~135,000 TCP connection attempts over 30 seconds, exhausting the process's file descriptor limit and async runtime task memory. A single connected attacker with one session can sustain this indefinitely. This maps to the **High** impact class: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation

- The attacker requires only a single established p2p connection to the victim — no special privilege, no cryptographic material, no PoW.
- The victim's PeerId is publicly exchanged during peer identification; spoofed `from` PeerIds need not correspond to any real peer.
- The attack is immediately repeatable: distinct spoofed `from` PeerIds have no cooldown at all under the `forward_rate_limiter`.
- The `rate_limiter` of 30 req/sec per session is the only practical bound, and it is sufficient to accumulate hundreds of concurrent tasks within seconds.
- No victim interaction or mistake is required.

## Recommendation

1. **Verify `from` against the session.** In `respond_delivered`, look up the actual PeerId of `self.peer` from the peer registry (e.g., `network_state.with_peer_registry(|r| r.get_peer(self.peer).map(|p| p.peer_id.clone()))`) and reject the message if `content.from` does not match it. This eliminates spoofing entirely.
2. **Bound concurrent NAT traversal tasks.** Maintain an `Arc<AtomicUsize>` counter or `tokio::sync::Semaphore` on active `try_nat_traversal` tasks. Reject new `ConnectionSync` messages when the limit is reached.
3. **Do not allow overwrite of an unconsumed entry.** If `pending_delivered` already contains an entry for `from_peer_id` (regardless of age), reject the new `ConnectionRequest` until the entry is consumed or explicitly expired.
