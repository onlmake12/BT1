### Title
Keeper DoS and Fee Theft via Front-Running Permissionless `updatePriceFeeds()` — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.updatePriceFeeds()` carries no access control and reverts when the submitted update's timestamp is not strictly newer than the last stored timestamp. An unprivileged attacker can copy a legitimate keeper's pending transaction from the public mempool and front-run it with identical calldata. The attacker's call succeeds, sets `priceLastUpdatedAt`, and collects the keeper fee. The legitimate keeper's call then reverts because the timestamp equality check `updateTimestamp <= priceLastUpdatedAt` is now true. Repeated front-running permanently denies legitimate keepers their fees and accelerates subscription balance drain, eventually causing price staleness for downstream consumers.

---

### Finding Description

`updatePriceFeeds()` is declared `external` with no keeper whitelist or access-control modifier: [1](#0-0) 

Inside the function, the Pyth fee is deducted from the subscription balance before any timestamp validation occurs: [2](#0-1) 

`_validateShouldUpdatePrices()` then enforces a strict "newer than last stored" rule using `<=`: [3](#0-2) 

If the check passes, `_processFeesAndPayKeeper()` pays `msg.sender` the full keeper fee (gas cost + per-feed fee): [4](#0-3) 

Because the entire call reverts on a timestamp collision, the Pyth fee deduction at line 305 is also rolled back — but the legitimate keeper still loses all transaction gas and receives nothing.

---

### Impact Explanation

- **Keeper fee theft**: Every time a legitimate keeper submits `updatePriceFeeds()`, an attacker can front-run with the same calldata, collect the keeper fee, and cause the legitimate keeper's transaction to revert. The attacker profits on every update cycle.
- **Subscription balance drain**: The attacker is paid `pythFee + keeperFee` per front-run. Subscription balances are exhausted at the same rate as normal operation, but all keeper revenue flows to the attacker.
- **Price staleness for consumers**: Once a subscription's balance reaches zero, `updatePriceFeeds()` reverts at the `pythFee` balance check (line 295–297). Consumers calling `getPricesNoOlderThan()` will receive `StalePrice` reverts, breaking any protocol that depends on fresh Scheduler-delivered prices.

---

### Likelihood Explanation

- `updatePriceFeeds()` is explicitly designed to be permissionless (no keeper registry exists anywhere in the Pulse contracts).
- All EVM chains with a public mempool (Ethereum, Arbitrum, BNB Chain, etc.) expose pending transactions before inclusion.
- The attacker only needs to copy `subscriptionId` and `updateData` verbatim — zero additional knowledge required.
- The attack is profitable: the attacker earns `keeperFee` minus their own gas cost per front-run. On low-gas chains this margin is large.
- The Fortuna keeper service already handles `NoSuchRequest` reverts from `revealWithCallback()` front-running (see `apps/fortuna/src/keeper/process_event.rs` lines 153–174), confirming that the Pyth team is aware of mempool-based front-running as a realistic threat model.

---

### Recommendation

1. **Soft-fail on stale timestamp**: Instead of reverting when `updateTimestamp <= priceLastUpdatedAt`, return early without paying the keeper. This eliminates the griefing surface while preserving permissionless participation.
2. **Alternatively, introduce a keeper whitelist or commit-reveal**: Allow only registered keepers to call `updatePriceFeeds()`, or require keepers to commit to an update before revealing it, preventing mempool copying.
3. **Deduct the Pyth fee only after timestamp validation**: Move the `status.balanceInWei -= pythFee` deduction (line 305) to after `_validateShouldUpdatePrices()` returns, so a reverted call does not temporarily corrupt balance accounting mid-execution (defense-in-depth).

---

### Proof of Concept

```
Setup:
  - Subscription S has balance = pythFee + keeperFee (enough for exactly one update)
  - Legitimate keeper K prepares:
      updatePriceFeeds(S, updateData)   // updateData has publishTime = T

Attack:
  1. Attacker A monitors the mempool, sees K's pending transaction.
  2. A submits the identical call with higher gas price:
       updatePriceFeeds(S, updateData)
  3. A's transaction is mined first:
       - _validateShouldUpdatePrices: priceLastUpdatedAt=0, T > 0 → passes
       - status.priceLastUpdatedAt = T
       - A receives keeperFee
  4. K's transaction is mined next:
       - _validateShouldUpdatePrices: T <= T → revert TimestampOlderThanLastUpdate
       - K loses gas, receives nothing.

Repeat every heartbeat interval:
  - A earns keeperFee on every cycle.
  - K's transactions always revert.
  - After N cycles the subscription balance hits zero.
  - All subsequent updatePriceFeeds calls revert at the pythFee balance check.
  - Consumers receive StalePrice on every getPricesNoOlderThan() call.
``` [5](#0-4)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
