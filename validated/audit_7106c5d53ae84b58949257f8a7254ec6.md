### Title
`RecentReject` Store Grows Unbounded Due to Non-Incremented Counter and Random Shard Eviction — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

Two compounding bugs in `RecentReject` allow any peer to exhaust node disk space by flooding the node with rejection-inducing transactions. The primary bug is that `self.total_keys_num` is **never incremented** in `put`, so `shrink` is essentially never triggered from a fresh start. The secondary bug is that `shrink` selects a shard **uniformly at random**, so an attacker who concentrates entries in one shard survives eviction with probability `(shard_num-1)/shard_num` per call.

---

### Finding Description

**Bug 1 — Counter never incremented in `put`:**

In `put`, the overflow check computes a local variable `total_keys_num` via `self.total_keys_num.checked_add(1)`, but **never writes it back** to `self.total_keys_num`: [1](#0-0) 

`self.total_keys_num` is only updated inside `shrink` (line 111). On a fresh node, `self.total_keys_num` starts at 0 (estimated from an empty DB at construction): [2](#0-1) 

Every subsequent `put` computes `local = 0 + 1 = 1`, which is never `> count_limit` (default is tens of thousands), so `shrink` is **never called**. The RocksDB store grows without bound.

**Bug 2 — Random shard eviction (amplification):**

Even when `shrink` is triggered (e.g., after a node restart where `total_keys_num` is re-estimated from a full DB), it picks a shard uniformly at random: [3](#0-2) 

With `DEFAULT_SHARDS = 5`, an attacker who concentrates all entries in shard 0 (by controlling the first 4 bytes of the tx hash via `u32::from_le_bytes(hash[0..4]) % shard_num == 0`) survives each `shrink` call with probability 4/5. After dropping an empty shard, `total_keys_num` is re-estimated and still exceeds `count_limit`, so the next `put` triggers `shrink` again — but the attacker's shard is still intact. [4](#0-3) 

**Attacker entry path:**

Any peer submitting transactions that get rejected with `should_recorded() == true` (e.g., `Reject::Malformed`, `Reject::Verification`, etc.) triggers `put_recent_reject`: [5](#0-4) 

This is reachable via P2P transaction relay or the `send_transaction` RPC, both unprivileged paths.

---

### Impact Explanation

The `recent_reject` store is a RocksDB instance on disk. With no effective eviction, an attacker can grow it to exhaust available disk space, causing the node to crash or become unable to write new blocks/transactions. This is a remote DoS against any CKB full node.

---

### Likelihood Explanation

The attack requires only submitting transactions that get rejected — no PoW, no keys, no privileged access. The attacker can submit them via P2P relay or RPC. Crafting hashes to target a specific shard is straightforward (brute-force the first 4 bytes of the transaction content). The counter bug alone (Bug 1) is sufficient to trigger unbounded growth with no shard targeting needed.

---

### Recommendation

1. **Fix the counter**: In `put`, write back the incremented value to `self.total_keys_num` after a successful DB write:
   ```rust
   self.total_keys_num = total_keys_num; // after self.db.put(...)
   ```
2. **Fix shard eviction**: Instead of picking a random shard, identify and drop the **largest** shard (or the shard containing the inserted key), ensuring the invariant `total_keys_num <= count_limit` is actually maintained.
3. **Rate-limit** rejected transaction recording per peer to reduce the attack surface.

---

### Proof of Concept

```rust
// Fresh RecentReject, count_limit = 10, shard_num = 5
let mut rr = RecentReject::build(path, 5, 10, -1).unwrap();
// self.total_keys_num = 0

// Submit 10000 rejected txs with hash[0..4] mapping to shard 0
for i in 0..10000u64 {
    let mut hash = [0u8; 32];
    // Set low 4 bytes so u32::from_le_bytes % 5 == 0, e.g. bytes = [0,0,0,0]
    hash[4..12].copy_from_slice(&i.to_le_bytes());
    let key = Byte32::new(hash);
    rr.put(&key, Reject::Malformed("x".into(), Default::default())).unwrap();
    // Each call: local total_keys_num = 0+1 = 1, never > 10, shrink never called
}

// Verify: shard 0 has 10000 entries, count_limit = 10 is violated
assert!(rr.estimate_total_keys_num().unwrap() > 10); // passes — invariant broken
```

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;
        self.db.create_cf_with_ttl(&shard, self.ttl)?;

        let total_keys_num = self.estimate_total_keys_num()?;
        self.total_keys_num = total_keys_num;
        Ok(total_keys_num)
    }
