### Title
Division-by-Zero Node Panic via Crafted `since` Epoch Field with Zero Epoch Length — (`verification/src/transaction_verifier.rs`, `util/types/src/core/extras.rs`)

---

### Summary

`EpochNumberWithFraction::to_rational()` panics with a denominator-zero panic when called on a value whose epoch length is zero but whose packed representation is non-zero. The transaction verifier uses `from_full_value_unchecked` — the variant that explicitly skips normalization — to parse the epoch encoded in a transaction input's `since` field. An unprivileged transaction sender can craft a `since` value encoding epoch number ≥ 1 with epoch length = 0, causing the node to panic during transaction verification.

---

### Finding Description

`EpochNumberWithFraction` packs three sub-fields into a single `u64`:

| Bits | Field |
|------|-------|
| 0–23 | epoch number |
| 24–39 | epoch index |
| 40–55 | epoch length | [1](#0-0) 

`to_rational()` converts this to a `RationalU256` by calling `RationalU256::new(index, length)`:

```rust
pub fn to_rational(self) -> RationalU256 {
    if self.0 == 0 {
        RationalU256::zero()
    } else {
        RationalU256::new(self.index().into(), self.length().into()) + U256::from(self.number())
    }
}
``` [2](#0-1) 

`RationalU256::new` unconditionally panics when its denominator is zero:

```rust
pub fn new(numer: U256, denom: U256) -> RationalU256 {
    if denom.is_zero() {
        panic!("denominator == 0");
    }
    ...
}
``` [3](#0-2) 

The safe constructor `from_full_value` calls `normalize()` to rewrite zero-length epochs before any use:

```rust
pub fn from_full_value(value: u64) -> Self {
    Self::from_full_value_unchecked(value).normalize()
}
``` [4](#0-3) 

But `from_full_value_unchecked` — used in `verification/src/transaction_verifier.rs` — skips normalization entirely, and its own documentation warns of the risk:

> The `EpochNumberWithFraction` constructed by this method has a potential risk that when call `self.to_rational()` may lead to a panic if the user specifies a zero epoch length. [5](#0-4) 

The `since` field of a transaction input is a raw `u64` fully controlled by the transaction sender. When it encodes an epoch-based time-lock (flag bits set to epoch mode), the verifier parses it via `from_full_value_unchecked`. If the packed value has epoch number ≥ 1 and epoch length = 0, `self.0 != 0` so the genesis short-circuit is skipped, and `RationalU256::new(..., 0)` panics.

---

### Impact Explanation

A Rust integer-division-by-zero or an explicit `panic!` in a synchronous call stack terminates the thread. In CKB's architecture the chain/verification service runs on a dedicated thread; a panic there crashes the node process. The attacker achieves **remote node crash** (denial of service) against any peer that attempts to verify the malicious transaction — including nodes receiving it via P2P relay, nodes processing a block containing it, or nodes responding to an RPC `send_transaction` call.

---

### Likelihood Explanation

The `since` field is a plain `u64` in the transaction wire format. No cryptographic material is required to set it to an arbitrary value. Any unprivileged actor who can submit a transaction (via RPC `send_transaction`, P2P relay, or by including it in a mined block) can trigger the panic. The crafted value is trivial to construct: set bits 40–55 to 0 (epoch length = 0) and bits 0–23 to any non-zero value (epoch number ≥ 1), with the epoch-mode flag bits set.

---

### Recommendation

Replace `from_full_value_unchecked` with `from_full_value` (which calls `normalize()`) wherever the source is untrusted external data — specifically in `verification/src/transaction_verifier.rs`. Alternatively, add an explicit `is_well_formed` / `is_well_formed_increment` guard before any call to `to_rational()` on a value derived from the `since` field, and return a `TransactionError` rather than panicking.

The safe path already exists in the codebase:

```rust
pub fn normalize(self) -> Self {
    if self.length() == 0 {
        Self::new(self.number(), 0, 1)
    } else {
        self
    }
}
``` [6](#0-5) 

---

### Proof of Concept

Craft a transaction input whose `since` field encodes:

```
bits 63    = 1  (relative flag)
bits 62    = 1  (epoch metric)
bits 40-55 = 0  (epoch length = 0)   ← triggers the panic
bits 24-39 = 0  (epoch index  = 0)
bits 0-23  = 1  (epoch number = 1)   ← makes self.0 != 0, bypassing genesis guard
```

Packed value: `0xC000000000000001`

Submit via `send_transaction` RPC or relay over P2P. When the node's `SinceVerifier` calls `from_full_value_unchecked(0xC000000000000001)` and subsequently `to_rational()`, execution reaches:

```rust
RationalU256::new(0_u256, 0_u256)  // length == 0 → panic!("denominator == 0")
``` [7](#0-6) 

causing the node process to crash.
