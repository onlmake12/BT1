### Title
Missing Emitter Authority Signer Check in `handle_post_prepared_message` Allows Unauthorized Message Publication - (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_message/mod.rs`)

---

### Summary

`handle_post_prepared_message` never verifies that the caller is the `emitter_authority` stored in the message account. Any unprivileged attacker who observes a finalized (`ReadyForPublishing`) message account on-chain can call `post_message` with that account and publish the message under the victim's emitter address, consuming their sequence number.

---

### Finding Description

The three-step prepared-message flow is:

1. `init_message_v1` — requires `emitter_authority` as a `Signer`; stores `emitter_authority: ctx.accounts.emitter_authority.key()` in `PostedMessageV1Info`.
2. `write_message_v1` / `finalize_message_v1` — both require `emitter_authority` as a `Signer` and verify it against the stored field via `require_draft_message`.
3. `post_message` (legacy) — **does not require the emitter authority to sign**.

In `finalize_message_v1`, the constraint is: [1](#0-0) 

After finalization the status becomes `ReadyForPublishing`. At that point, the `emitter_authority` pubkey is still stored in the message account at bytes 5–37: [2](#0-1) 

In `PostMessage`, the `emitter` field is **optional** and only needs to be a signer if provided: [3](#0-2) 

Inside `handle_post_prepared_message`, the only checks performed are:

- Message is owned by Core Bridge
- `info.status == MessageStatus::ReadyForPublishing`
- Commitment matches
- Payload argument is empty
- **If** an emitter is provided, it matches `info.emitter` (the emitter address, not the authority) [4](#0-3) 

The `emitter_authority` stored in the message is **never checked against any signer**. It is simply zeroed out unconditionally: [5](#0-4) 

---

### Impact Explanation

An attacker can publish a Wormhole message under any victim's emitter address by supplying a finalized message account they did not create. This:

- Consumes the victim's emitter sequence number (sequence slot griefing / DoS)
- Publishes an attacker-triggered VAA attributed to the victim's emitter, which downstream Pyth cross-chain consumers (price feed relayers, governance) would treat as originating from the legitimate emitter [6](#0-5) 

---

### Likelihood Explanation

The attack requires only:
- A funded payer account (any Solana wallet)
- Knowledge of the victim's message account address (publicly observable on-chain once `finalize_message_v1` is called)

No privileged access, leaked keys, or governance majority is needed. The window is the time between `finalize_message_v1` and the victim's own `post_message` call — a standard front-running window on Solana.

---

### Recommendation

In `handle_post_prepared_message`, add a check that the `emitter_authority` stored in the message matches a required signer. Concretely:

- Add `emitter_authority: Signer<'info>` to the `PostMessage` account context (or a separate prepared-message context).
- After deserializing `info`, assert `require_keys_eq!(emitter_authority.key(), info.emitter_authority, ...)`.

This mirrors the guard already present in `finalize_message_v1` and `write_message_v1`. [7](#0-6) 

---

### Proof of Concept

```
1. Alice calls init_message_v1 (emitter_authority = Alice), write_message_v1, finalize_message_v1.
   → message account status = ReadyForPublishing, emitter_authority = Alice's pubkey stored at bytes 5-37.

2. Bob (attacker) observes the message account address on-chain.

3. Bob calls post_message with:
   - message = Alice's message account
   - emitter = None  (optional, omitted)
   - payer = Bob's account

4. handle_post_prepared_message runs:
   - owner check: passes (Core Bridge owns the account)
   - status check: passes (ReadyForPublishing)
   - emitter authority check: ABSENT
   - message published under Alice's emitter address with Alice's next sequence number

5. Assert: Bob's transaction succeeds; Alice's sequence number is consumed;
   the resulting VAA carries Alice's emitter address but was triggered by Bob.
``` [8](#0-7)

    pub(crate) fn require_draft_message(
        acc_info: &AccountInfo,
        emitter_authority: &Signer,
    ) -> Result<bool> {
        let data = acc_info.try_borrow_data()?;
        require!(
            data.len() > 4 && data[..4] == *Self::LEGACY_DISCRIMINATOR,
            ErrorCode::AccountDidNotDeserialize
        );

        require_keys_eq!(
            Self::emitter_authority_unsafe(&data),
            emitter_authority.key(),
            CoreBridgeError::EmitterAuthorityMismatch
        );

        require!(
            Self::status_unsafe(&data) == MessageStatus::Writing,
            CoreBridgeError::NotInWritingStatus
        );

        Ok(true)
    }

fn handle_post_prepared_message(ctx: Context<PostMessage>, args: PostMessageArgs) -> Result<()> {
    msg!("MessageStatus: ReadyForPublishing");

    // Because the message account was prepared by the Core Bridge, we need to double-check that
    // the Core Bridge is the owner.
    require_keys_eq!(
        *ctx.accounts.message.owner,
        crate::ID,
        ErrorCode::ConstraintOwner
    );

    // Check message header. This is mutable because we will rewrite to the mesasge account with
    // some modified values later on.
    let mut info = PostedMessageV1::try_deserialize_info(&ctx.accounts.message)?;

    // Make sure the message is ready to be published.
    require!(
        info.status == MessageStatus::ReadyForPublishing,
        CoreBridgeError::NotReadyForPublishing
    );

    // Check emitter sequence account. If it does not exist, create it. Otherwise realloc the
    // account if it is a legacy emitter sequence account.
    let mut emitter_sequence = create_or_realloc_emitter_sequence(
        &ctx.accounts.emitter_sequence,
        &ctx.accounts.payer,
        &ctx.accounts.system_program,
        &info.emitter,
        ctx.bumps.emitter_sequence,
    )?;

    // If the emitter is the same as the emitter authority, this message's emitter is a legacy
    // emitter. Otherwise it is an executable.
    if info.emitter == info.emitter_authority {
        require!(
            emitter_sequence.emitter_type != EmitterType::Executable,
            CoreBridgeError::ExecutableEmitter
        );
        emitter_sequence.emitter_type = EmitterType::Legacy;
    } else {
        require!(
            emitter_sequence.emitter_type != EmitterType::Legacy,
            CoreBridgeError::LegacyEmitter
        );
        emitter_sequence.emitter_type = EmitterType::Executable;
    }

    let PostMessageArgs {
        nonce: _,
        payload: unnecessary_payload,
        commitment,
    } = args;

    // Verify that the commitment provided agrees with the one encoded in the message account.
    // Performing unwrap here is safe because the consistency level written to this account was
    // determined by the commitment level set when this message was initialized.
    require_eq!(
        commitment,
        Commitment::try_from(info.consistency_level).unwrap(),
        CoreBridgeError::InvalidInstructionArgument
    );

    // The payload argument is not allowed if the message has been prepared beforehand.
    require!(
        unnecessary_payload.is_empty(),
        CoreBridgeError::InvalidInstructionArgument
    );

    // If the emitter is passed into the account context, revert if it does not agree with the one
    // encoded in the message account. This check is important for deriving the correct emitter
    // sequence PDA address, which is determined by the account passed into this context.
    if let Some(emitter) = ctx.accounts.emitter.as_ref() {
        require_keys_eq!(
            emitter.key(),
            info.emitter,
            CoreBridgeError::EmitterMismatch
        );
    }

    // Set other values to reflect published state.
    info.emitter_authority = Default::default();
    info.status = MessageStatus::Published;
    info.posted_timestamp = Clock::get().map(Into::into)?;
    info.sequence = emitter_sequence.value;

    // Even though integrators should be reading the message account data to determine its sequence
    // number instead of using program logs, we need to keep this log here for backwards
    // compatibility.
    msg!("Sequence: {}", emitter_sequence.value);

    // Update emitter sequence account with incremented value.
    {
        emitter_sequence.value += 1;

        let acc_data: &mut [_] = &mut ctx.accounts.emitter_sequence.data.borrow_mut();
        let mut writer = std::io::Cursor::new(acc_data);
        emitter_sequence.try_serialize(&mut writer)?;
    }

    let msg_acc_data: &mut [_] = &mut ctx.accounts.message.data.borrow_mut();
    let mut writer = std::io::Cursor::new(msg_acc_data);

    std::io::Write::write_all(&mut writer, PostedMessageV1::LEGACY_DISCRIMINATOR)?;
    info.serialize(&mut writer)?;

    // Done.
    Ok(())
}
