### Title
Missing Maximum Fee Protection in `requestWithCallback` / `requestV2` — No Refund of Excess `msg.value` - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The Entropy contract's `requestHelper` function absorbs all `msg.value` in excess of the provider fee into `accruedPythFeesInWei` without refunding the caller. There is no `maxFee` parameter in `requestWithCallback` or `requestV2` that would allow users to bound the fee they are willing to pay. Because providers can update their fee at any time via `register()`, a race condition exists between a user's off-chain fee query and on-chain execution, causing users to either have their transaction revert or silently overpay with no recourse.

---

### Finding Description

`Entropy.sol`'s `requestHelper` is the shared execution path for `requestWithCallback`, `requestV2`, and related entry points. The fee accounting logic is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire surplus `msg.value - providerFee` is credited to the Pyth protocol fee pool — it is never returned to the caller. The `IEntropyV2` interface explicitly documents this: *"excess value is not refunded to the caller"*, and advises callers to re-query `getFeeV2` before each invocation. [2](#0-1) 

However, no `maxFee` parameter exists in any of the request functions:

```solidity
function requestV2(address provider, bytes32 userRandomNumber, uint32 gasLimit)
    external payable returns (uint64 assignedSequenceNumber);
``` [3](#0-2) 

A provider can update their fee at any time by calling `register()` with a new `feeInWei`:

```solidity
provider.feeInWei = feeInWei;
``` [4](#0-3) 

This creates two failure modes for users:

1. **Revert path**: User queries `getFeeV2`, provider increases fee before the user's tx lands → `msg.value < requiredFee` → `InsufficientFee` revert. User must retry, paying gas twice.
2. **Silent overpayment path**: User sends a buffer (e.g., 1.5× the queried fee) to avoid reverts. If the fee did not increase, the surplus is permanently absorbed into `accruedPythFeesInWei`. The user has no way to recover it and no way to have specified a maximum acceptable fee.

Integrating contracts (e.g., a DeFi protocol that wraps Entropy) are particularly exposed: they typically hardcode or cache a fee value and forward it with `msg.value`, making the overpayment path the common case.

---

### Impact Explanation

**Medium.** Users or integrating contracts that send any ETH beyond the exact current fee permanently lose that ETH to the Pyth protocol fee pool. There is no refund mechanism and no `maxFee` guard. For integrating contracts that buffer fees to avoid reverts, this is a guaranteed, silent loss on every request where the fee is lower than the buffered amount.

---

### Likelihood Explanation

**Medium.** Provider fees are mutable and can be updated at any time by any registered provider via `register()`. The Fortuna keeper actively adjusts fees in response to gas price changes (`adjust_fee_if_necessary`). [5](#0-4) 

Any integrating contract that caches a fee value or sends a buffer will silently overpay on every call where the live fee is lower than the sent value. This is a routine operational condition, not a rare edge case.

---

### Recommendation

Add a `maxFee` parameter to `requestWithCallback` and `requestV2` that reverts if `msg.value > maxFee`, and refund any excess `msg.value` beyond `requiredFee` back to `msg.sender`:

```solidity
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
_state.accruedPythFeesInWei += (requiredFee - providerFee);
```

Alternatively, document and enforce that callers must pass `msg.value == getFeeV2(provider, gasLimit)` exactly, and revert on overpayment.

---

### Proof of Concept

1. Provider registers with `feeInWei = 1 ether`.
2. User queries `getFeeV2(provider, gasLimit)` off-chain → returns `1.001 ether` (including Pyth fee).
3. User submits `requestV2{value: 1.5 ether}(provider, userRandom, gasLimit)` with a 0.5 ETH buffer.
4. `requestHelper` executes: `requiredFee = 1.001 ether`, `providerFee = 1 ether`.
5. `accruedPythFeesInWei += 1.5 ether - 1 ether = 0.5 ether + 0.001 ether = 0.501 ether`.
6. User receives their random number callback but has permanently lost `0.499 ether` to the Pyth fee pool with no recourse. [1](#0-0) [6](#0-5)
