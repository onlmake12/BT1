### Title
Unchecked Return Value of `submit_remote_tx` Silently Discards Tx Rejection — (File: `tx-pool/src/service.rs`)

### Summary
In `tx-pool/src/service.rs`, the `Message::SubmitRemoteTx` handler discards the `Result` returned by `service.submit_remote_tx(...)` and unconditionally sends `()` back to the caller. This means the network relay layer always receives `Ok(())` regardless of whether the transaction was actually accepted or rejected by the pool, preventing proper peer penalization for invalid transactions.

### Finding Description
Inside the tx-pool service message dispatch loop, the handler for `Message::SubmitRemoteTx` is:

```rust
// tx-pool/src/service.rs, lines 844–851
Message::SubmitRemoteTx(Request {
    responder,
    arguments: (tx, declared_cycles, peer),
}) => {
    let _result = service.submit_remote_tx(tx, declared_cycles, peer).await;
    if let Err(e) = responder.send(()) {
        error!("Responder sending submit_tx result failed {:?}", e);
    };
}
```

`service.submit_remote_tx(...)` returns a `Result<(), AnyError>` that encodes whether the transaction passed validation and was admitted to the pool. This result is bound to `_result` — a Rust convention for explicitly discarding a value — and is never inspected. The `responder.send(())` call then unconditionally signals success to the waiting caller.

The public controller method that drives this path is:

```rust
// tx-pool/src/service.rs
pub async fn submit_remote_tx(
    &self,
    tx: TransactionView,
    declared_cycles: Cycle,
    peer: PeerIndex,
) -> Result<(), AnyError> {
    send_message!(self, SubmitRemoteTx, (tx, declared_cycles, peer))
}
```

Because the service handler always responds with `()`, this method always resolves to `Ok(())` from the caller's perspective, even when the underlying submission failed (e.g., invalid script, capacity mismatch, double-spend, declared-cycles mismatch).

A parallel pattern exists for `Message::NotifyTxs`:

```rust
// tx-pool/src/service.rs, lines 853–857
Message::NotifyTxs(Notify { arguments: txs }) => {
    for tx in txs {
        let _ret = service.notify_tx(tx).await;
    }
}
```

Here too, `notify_tx`'s `Result` is silently discarded with no logging or propagation.

### Impact Explanation
The network relay layer calls `TxPoolController::submit_remote_tx` when a remote peer relays a transaction. If the tx is rejected (invalid, over-limit, double-spend, etc.), the relay layer receives `Ok(())` and cannot distinguish rejection from acceptance. Consequences:

- **Peer scoring is bypassed**: CKB's relay protocol uses rejection outcomes to penalize or ban peers that send invalid transactions. With the result discarded, a malicious peer can repeatedly relay invalid or malformed transactions without triggering any ban score increment.
- **Silent resource consumption**: Each invalid tx still goes through full validation (script execution, capacity checks) before being rejected. Because the caller never learns of the rejection, it cannot apply back-pressure or rate-limit the offending peer.
- **Incorrect relay state**: The relay layer may re-broadcast or re-request transactions it believes were accepted, causing unnecessary network traffic.

### Likelihood Explanation
Any unprivileged P2P peer can send arbitrary transactions to a CKB node. The entry path is fully attacker-controlled: connect to a node, relay a crafted invalid transaction, and the node's tx-pool will silently swallow the rejection. No special privileges, keys, or majority hashpower are required. This is a realistic and low-effort attack vector.

### Recommendation
Propagate the `Result` from `service.submit_remote_tx(...)` back through the responder instead of sending `()`:

```rust
Message::SubmitRemoteTx(Request {
    responder,
    arguments: (tx, declared_cycles, peer),
}) => {
    let result = service.submit_remote_tx(tx, declared_cycles, peer).await;
    if let Err(e) = responder.send(result) {
        error!("Responder sending submit_tx result failed {:?}", e);
    };
}
```

This requires changing the responder type from `oneshot::Sender<()>` to `oneshot::Sender<Result<(), AnyError>>` and updating `TxPoolController::submit_remote_tx` accordingly. For `notify_tx`, at minimum log the error so operators can observe silent failures.

### Proof of Concept
1. Connect a peer to a CKB node.
2. Relay a transaction with an invalid script or a declared cycle count that does not match the actual execution cost.
3. The tx-pool internally rejects the transaction and `service.submit_remote_tx` returns `Err(...)`.
4. The service handler at line 848 discards the error via `let _result = ...`.
5. The responder sends `()` at line 849, so `TxPoolController::submit_remote_tx` resolves to `Ok(())`.
6. The relay layer receives `Ok(())`, does not increment the peer's ban score, and the peer suffers no penalty despite sending an invalid transaction.
7. The peer can repeat this indefinitely, consuming node validation resources without consequence. [1](#0-0) [2](#0-1)
