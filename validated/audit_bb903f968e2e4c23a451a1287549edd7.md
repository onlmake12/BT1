Audit Report

## Title
Unverified `from` Peer Identity in Hole-Punching `ConnectionRequest` Enables NAT-Traversal Hijacking and Rate-Limiter Bypass — (File: network/src/protocols/hole_punching/component/connection_request.rs)

## Summary
`ConnectionRequestProcess::execute()` decodes the `from` field exclusively from the attacker-controlled message payload and never cross-checks it against the actual session identity (`self.peer`). This unverified `from` is used as the key for both the `forward_rate_limiter` and the `pending_delivered` cache. An attacker can poison `pending_delivered[victim_peer_id]` with their own listen addresses and bypass the forward rate limiter by cycling through arbitrary `from` values, enabling NAT-traversal hijacking and unbounded gossip amplification across the network.

## Finding Description

**Root cause — unverified `from` field:**

`ConnectionRequestProcess` holds `peer: PeerIndex` (the actual session identity) at [1](#0-0)  but never uses it to validate `content.from`. The `from` field is decoded entirely from the raw message bytes: [2](#0-1) 

No comparison against the actual session's `PeerId` is performed anywhere in `execute()`.

**Attack path 1 — `pending_delivered` cache poisoning and NAT-traversal hijacking:**

When `self_peer_id == &content.to`, `respond_delivered(content.from, ...)` is called: [3](#0-2) 

Inside `respond_delivered`, the only guard is a 2-minute deduplication check on the same `from_peer_id`: [4](#0-3) 

This does not prevent the initial insertion. The attacker-controlled `from_peer_id` and their own listen addresses are unconditionally inserted: [5](#0-4) 

`ConnectionSyncProcess` has no `peer` field and performs no session-level check. It looks up the cache using the equally unverified `content.from` from the sync message: [6](#0-5) 

On a cache hit, `try_nat_traversal` is spawned toward the addresses stored in the poisoned entry: [7](#0-6) 

The `try_nat_traversal` function retries TCP connections for up to 30 seconds per address: [8](#0-7) 

**Attack path 2 — forward rate-limiter bypass and gossip amplification:**

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`: [9](#0-8) 

Its quota is 1 request/second per key: [10](#0-9) 

The outer `rate_limiter` is keyed by `(session_id, msg_item_id)` and capped at 30/s per session: [11](#0-10) 

An attacker sending 30 `ConnectionRequest` messages/second with 30 distinct fake `from` values passes both limiters: the outer limiter allows 30/s per session, and each message produces a unique `forward_rate_limiter` key. Each passing message is broadcast to `sqrt(N)` peers for up to `MAX_HOPS = 6` hops: [12](#0-11) 

The same `forward_rate_limiter` bypass applies to `ConnectionRequestDelivered` and `ConnectionSync`, both of which use the same key structure: [13](#0-12) [14](#0-13) 

**Attack path 3 — `pending_delivered` memory exhaustion:**

`pending_delivered` is a plain unbounded `HashMap`: [15](#0-14) 

It is pruned only every 5 minutes: [16](#0-15) 

At 30 messages/second, an attacker inserts up to 9,000 entries per eviction window, each holding a `PeerId` and up to `ADDRS_COUNT_LIMIT = 24` `Multiaddr` values: [17](#0-16) 

**Why `TooManyRequests` does not ban the attacker:**

`StatusCode::TooManyRequests` has code 110 (informational range). The ban logic only triggers for 4xx codes: [18](#0-17) 

The attacker is never banned for rate-limit violations.

## Impact Explanation

**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single connected peer sending 30 `ConnectionRequest` messages/second with distinct fake `from` values bypasses the `forward_rate_limiter` at every relay node. Each message is broadcast to `sqrt(N)` peers and forwarded up to 6 hops. With N=100 peers, total message deliveries across the network at the deepest hop reach `30 × 10^6` per second, saturating the gossip layer. The NAT-traversal hijacking additionally causes victim nodes to spawn 30-second TCP connection tasks to attacker-controlled endpoints, wasting file descriptors and CPU.

## Likelihood Explanation

Any peer connected to a CKB node over the HolePunching protocol can trigger this. No special privilege, key material, or hashpower is required — only one active session. The outer rate limiter (30 msg/s per session) is the only throttle, and it is insufficient: 30 distinct fake `from` values per second sustains both the cache poisoning and the rate-limiter bypass indefinitely. The attack is repeatable and requires no victim interaction. The attacker is never banned because `TooManyRequests` is a non-banning status code.

## Recommendation

In `ConnectionRequestProcess::execute()`, resolve the actual `PeerId` of the sending session from the peer registry using `self.peer` (the `PeerIndex`) before accepting `content.from`. Assert that the resolved `PeerId` equals `content.from`; if they differ, return a 4xx status code (which triggers a ban) and reject the message. Apply the same check in `ConnectionSyncProcess::execute()` for `content.from` — add a `peer: PeerIndex` field to `ConnectionSyncProcess` and verify it against `content.from`. Additionally, cap the size of `pending_delivered` with an LRU eviction policy to bound memory consumption independent of the rate limiter.

## Proof of Concept

**Cache poisoning + NAT hijacking:**
1. Attacker peer `A` connects to victim node `V` over the HolePunching protocol.
2. `A` sends a `ConnectionRequest` with `from = peer_id_of_B` (any known peer), `to = V.peer_id`, `listen_addrs = [A_tcp_addr]`, `max_hops = 6`, empty `route`.
3. `V` receives it: `self_peer_id == content.to`, so `respond_delivered(peer_id_of_B, ...)` is called. `pending_delivered[peer_id_of_B] = ([A_tcp_addr], now)` is inserted.
4. `A` immediately sends a `ConnectionSync` with `from = peer_id_of_B`, `to = V.peer_id`, empty `route`.
5. `V` receives it: `self_peer_id == content.to`, route is empty, looks up `pending_delivered[peer_id_of_B]`, finds `[A_tcp_addr]`, spawns `try_nat_traversal(ttl, [A_tcp_addr])` — connecting to `A` instead of `B`.

**Forward rate-limiter bypass:**
1. `A` connects to `V`.
2. `A` sends 30 `ConnectionRequest` messages/second, each with a distinct randomly-generated `from` PeerId, `to = any_peer_id`, valid `listen_addrs`, `max_hops = 6`.
3. Each message passes the outer `rate_limiter` (30/s per session) and the `forward_rate_limiter` (unique key per message), and is broadcast to `sqrt(N)` peers for up to 6 hops, amplifying traffic across the gossip network.
