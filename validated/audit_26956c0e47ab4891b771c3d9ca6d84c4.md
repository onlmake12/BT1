### Title
Silent u128→u64 Truncating Cast in DAO Withdrawal Capacity Calculation Produces Wrong Maximum-Withdraw Value — (File: `util/dao/src/lib.rs`)

### Summary
`DaoCalculator::calculate_maximum_withdraw` computes the interest-bearing withdrawal capacity using a u128 intermediate value but converts it back to u64 with a bare `as u64` cast. This silently truncates on overflow instead of returning `DaoError::Overflow`, producing a wrong (lower) maximum-withdraw value. Every other analogous u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, making this the one unsafe outlier. The result propagates into tx-pool fee admission and the `calculate_dao_maximum_withdraw` RPC, causing valid large-capacity DAO withdrawals to be incorrectly rejected.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
// lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← bare truncating cast
        .safe_add(occupied_capacity)?;
```

`withdraw_counted_capacity` is a `u128`. The expression `counted_capacity * withdrawing_ar / deposit_ar` can exceed `u64::MAX` when `counted_capacity` is large (close to `u64::MAX` shannons) and `withdrawing_ar > deposit_ar` (i.e., any non-zero interest has accrued). The `as u64` cast silently wraps the value modulo 2⁶⁴, producing a value that is far smaller than the true result.

Every other u128→u64 narrowing in the same file uses the safe pattern:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)

// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
```

This is the direct CKB analog to the SafeERC20 report: the code uses a wider intermediate type (u128) that *appears* to make the arithmetic safe, but the final narrowing step is done with an unsafe bare cast instead of the proper checked conversion — exactly as `transfer` wrapped in `require` appears safe but misses "bad ERC20" edge cases.

### Impact Explanation

`calculate_maximum_withdraw` is called from two production paths:

1. **tx-pool fee admission** — `DaoCalculator::transaction_fee` (called by `check_tx_fee` in `tx-pool/src/util.rs` and by `FeeCalculator::transaction_fee` in `verification/src/transaction_verifier.rs`) subtracts `outputs_capacity` from the result of `transaction_maximum_withdraw`, which itself calls `calculate_maximum_withdraw`. When the truncated value is smaller than `outputs_capacity`, `safe_sub` returns `DaoError::Overflow`, and the transaction is rejected from the tx-pool with a `Reject::Malformed` error even though it is protocol-valid.

2. **RPC** — `calculate_dao_maximum_withdraw` in `rpc/src/module/experiment.rs` calls `calculate_maximum_withdraw` directly and returns the wrong (truncated) capacity to the caller, silently misleading wallets and tooling about the actual withdrawable amount.

The on-chain DAO script performs its own independent capacity check, so consensus safety is preserved. The impact is confined to: (a) valid DAO withdrawal transactions being permanently unsubmittable through the node's tx-pool, and (b) incorrect RPC responses.

### Likelihood Explanation

A DAO depositor who holds a cell with capacity close to `u64::MAX` shannons (≈184 billion CKB) and waits for any non-trivial interest accumulation (`withdrawing_ar > deposit_ar`) will trigger the overflow. The accumulation rate `ar` starts at `10^16` and only increases, so the condition `withdrawing_ar > deposit_ar` is true for every withdrawal after the first block. The overflow threshold is:

```
counted_capacity * withdrawing_ar > u64::MAX * deposit_ar
```

For a cell with `counted_capacity ≈ u64::MAX` and even a tiny `withdrawing_ar / deposit_ar` ratio above 1, the product overflows. While cells of this size are rare today, the code path is reachable by any tx-pool submitter or RPC caller without any privileged access.

### Recommendation

Replace the bare `as u64` cast with the same checked conversion used everywhere else in the file:

```rust
// util/dao/src/lib.rs, lines 155-156
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
```

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` (lines 296–349) already constructs a cell with `capacity = 18_446_744_073_709_550_000` shannons (near `u64::MAX`) and asserts `result.is_err()`. With the current `as u64` cast, when `withdraw_counted_capacity` overflows u64 the truncated value is silently used and `safe_add` may succeed, returning `Ok(wrong_value)` instead of `Err(DaoError::Overflow)` — meaning the test's `assert!(result.is_err())` assertion can fail depending on the exact `ar` ratio used, demonstrating the unsafe cast is the root cause.

    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
