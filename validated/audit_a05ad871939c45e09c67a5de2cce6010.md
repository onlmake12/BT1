### Title
Unsafe Native ETH Refund via `.transfer()` Causes DoS for Smart Contract Callers - (File: lazer/contracts/evm/src/PythLazer.sol)

### Summary
`PythLazer.verifyUpdate` refunds excess ETH to the caller using `payable(msg.sender).transfer(...)`, which forwards only 2300 gas. Any smart contract caller whose `receive()` or `fallback()` function consumes more than 2300 gas will have the entire `verifyUpdate` call revert, making the function permanently unusable for that integrator when overpaying.

### Finding Description
In `PythLazer.sol`, the `verifyUpdate` function accepts a fee and refunds any excess:

```solidity
// lazer/contracts/evm/src/PythLazer.sol, line 74-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [1](#0-0) 

Solidity's `.transfer()` hard-caps the gas forwarded to the recipient at 2300. This is insufficient for any smart contract recipient that performs even a single storage write (e.g., EIP-1967 proxies, multi-sig wallets, contracts that emit events in `receive()`). When the refund fails, the EVM reverts the entire transaction, including the price verification that was the caller's actual goal.

No `SafeERC20` or equivalent safe-call pattern is used anywhere in the production Solidity contracts — confirmed by the absence of any `safeTransfer`, `SafeERC20`, or `call{value:}` pattern in the `.sol` files.

### Impact Explanation
Smart contract integrators of Pyth Lazer (e.g., DeFi protocols, on-chain bots, proxy-based contracts) that call `verifyUpdate` with `msg.value > verification_fee` will have every such call revert. The integrator cannot obtain a verified Lazer price update on-chain, breaking any protocol that depends on Lazer for price feeds. Funds sent are not lost (the revert returns them), but the service is rendered inaccessible.

### Likelihood Explanation
Smart contract callers are the primary integrators of Lazer's on-chain verification. It is common practice for contracts to forward `msg.value` from an outer call or to add a buffer to ensure the fee is covered. Any such contract with a non-trivial `receive()` function (e.g., one that writes to storage or emits an event) will trigger the revert. EIP-1967 transparent proxies and many multi-sig wallets already exceed the 2300 gas limit.

### Recommendation
Replace `.transfer()` with a low-level `call` and check the return value:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Refund failed");
}
```

This forwards all available gas to the recipient and explicitly checks for failure, matching the intent of OpenZeppelin's `Address.sendValue`.

### Proof of Concept
1. Deploy a smart contract `Caller` with a `receive()` function that writes to a storage variable (costs ~5000 gas, exceeding the 2300 limit).
2. From `Caller`, call `PythLazer.verifyUpdate{value: verification_fee + 1 wei}(validUpdate)`.
3. The signature verification succeeds, but the refund of `1 wei` via `.transfer()` runs out of gas.
4. The entire transaction reverts with an out-of-gas error.
5. `Caller` can never successfully call `verifyUpdate` with any `msg.value` above `verification_fee`, even with a perfectly valid update payload. [2](#0-1)
