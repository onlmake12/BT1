### Title
Missing Empty-List Guard in `reset_spent_cells` Causes Panic on Rollback of Filter-Excluded Blocks — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

`reset_spent_cells` in the rich-indexer lacks the same empty-slice early-return guard that every other similar function in the same file has. When a block filter is configured and a block that does not match the filter is stored as a "simple block" (header-only, no transactions), rolling back that block passes an empty `tx_id_list` to `reset_spent_cells`, which generates invalid SQL (`IN ()`), returns `Err`, and causes `try_loop_sync` to panic via an unconditional `.expect()`.

---

### Finding Description

**Step 1 — Simple block storage.**
In `AsyncRichIndexer::append`, when `is_block_filter_match` returns `false`, only the block hash and number are written via `bulk_insert_blocks_simple`. No rows are inserted into `ckb_transaction`. [1](#0-0) 

**Step 2 — Rollback queries an empty transaction list.**
`rollback_block` calls `query_tx_id_list_by_block_id`, which queries `ckb_transaction WHERE block_id = $1`. For a simple block, this returns `vec![]`. [2](#0-1) [3](#0-2) 

**Step 3 — `reset_spent_cells` has no empty-list guard.**
`remove_batch_by_blobs` (line 60) and `query_outputs_by_tx_id_list` (line 167) both guard against empty slices with an early `return Ok(())`. `reset_spent_cells` does not: [4](#0-3) [5](#0-4) [6](#0-5) 

When `tx_id_list` is `&[]`, `and_where_in("consumed_tx_id", &[])` generates `WHERE consumed_tx_id IN ()` — a syntax error in both SQLite and PostgreSQL. The `.query()` call either returns an error (caught by `map_err`) or produces invalid SQL that fails at execution. Either way, `reset_spent_cells` returns `Err(Error::DB(...))`.

**Step 4 — Unconditional `.expect()` panics the process.**
`try_loop_sync` calls `rollback()` with `.expect()`, not with error handling: [7](#0-6) 

The `Err` propagates: `reset_spent_cells` → `rollback_block` → `AsyncRichIndexer::rollback` → `RichIndexer::rollback` → `.expect("rollback block should be OK")` → **process panic**. [8](#0-7) 

---

### Impact Explanation

The indexer sync thread panics, crashing the node process. The node goes offline and requires a manual restart. Any service depending on the rich-indexer (RPC queries, cell/transaction indexing) is unavailable until the operator intervenes.

---

### Likelihood Explanation

The block filter is a documented, supported production feature of the rich-indexer. Any operator who configures one (e.g., filtering by block number range or script type) is exposed. A reorg — which is a normal network event and can also be deliberately triggered by an attacker mining a competing chain — is sufficient to hit this path. No privileged access is required; the attacker only needs to relay valid blocks through P2P.

---

### Recommendation

Add the same empty-list guard to `reset_spent_cells` that already exists in `remove_batch_by_blobs` and `query_outputs_by_tx_id_list`:

```rust
async fn reset_spent_cells(tx_id_list: &[i64], tx: &mut Transaction<'_, Any>) -> Result<(), Error> {
    if tx_id_list.is_empty() {
        return Ok(());
    }
    // ... existing logic
}
```

Additionally, consider replacing the `.expect("rollback block should be OK")` in `try_loop_sync` with proper error handling (log and retry, or graceful shutdown) so that any future DB errors in rollback do not crash the process.

---

### Proof of Concept

1. Configure rich-indexer with a block filter that excludes block N (e.g., `block.header.number.to_uint() != N.to_uint()`).
2. Append block N — it is stored as a simple block (no `ckb_transaction` rows).
3. Trigger a reorg: append a competing block at height N with a different hash, causing `try_loop_sync` to detect `block.parent_hash() != tip_hash` and call `indexer.rollback()`.
4. `rollback_block` → `query_tx_id_list_by_block_id` returns `[]` → `reset_spent_cells(&[], tx)` → invalid SQL `IN ()` → `Err(Error::DB(...))` → `.expect(...)` → **panic**.

Reproducible as a unit test against both the SQLite and PostgreSQL backends using `AsyncRichIndexer` directly, as the existing rollback tests in `util/rich-indexer/src/tests/rollback.rs` demonstrate the test harness pattern. [9](#0-8)
