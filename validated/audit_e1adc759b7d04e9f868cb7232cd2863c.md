Audit Report

## Title
`RecentReject::put` never writes back incremented key count, disabling shrink and allowing unbounded RocksDB growth — (File: `tx-pool/src/component/recent_reject.rs`)

## Summary
In `RecentReject::put`, the incremented key count is computed into a local `if let` binding but is never assigned back to `self.total_keys_num` when within the limit. Because `self.total_keys_num` never advances, `shrink()` is never triggered, and the underlying `DBWithTTL` (RocksDB) grows without bound as rejected transactions accumulate. An attacker can exhaust disk space on a target node by spamming rejected transactions.

## Finding Description
In `put()` at lines 62–69 of `tx-pool/src/component/recent_reject.rs`:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // ← no else: total_keys_num is a local binding, never written back
} else {
    self.shrink()?;
}
```

`total_keys_num` is a local variable introduced by the `if let Some(...)` pattern. When `total_keys_num <= self.count_limit`, neither `self.total_keys_num = total_keys_num` nor `self.shrink()` is executed. The only write-back to `self.total_keys_num` occurs inside `shrink()` at lines 110–111:

```rust
let total_keys_num = self.estimate_total_keys_num()?;
self.total_keys_num = total_keys_num;
```

Since `self.total_keys_num` is initialized from a RocksDB estimate at `build()` time (lines 39–51, typically 0 for a fresh node) and is never incremented in the normal path, the condition `total_keys_num > self.count_limit` evaluates to `1 > count_limit` on every call — which is false for any reasonable `count_limit`. `shrink()` is therefore never called. The unconditional `self.db.put(...)` at line 60 writes every rejected transaction to RocksDB with no compensating removal.

The existing test at `tx-pool/src/component/tests/recent_reject.rs` line 39 asserts `recent_reject.total_keys_num < 100` after inserting 160 entries against a limit of 100. This assertion passes trivially because `total_keys_num` stays at 0 (its initial value) — the test does not verify actual DB entry count and provides no evidence the limit is enforced.

While `DBWithTTL` provides TTL-based RocksDB compaction that will eventually expire entries, this is a background process that cannot keep pace with a sustained spam attack generating distinct transaction hashes. The `shrink()` path (dropping and recreating a column family shard) is the intended active enforcement mechanism and is completely bypassed.

## Impact Explanation
Every rejected transaction causes an unconditional RocksDB write with no compensating `shrink()`. The `count_limit` invariant is completely broken. An attacker can exhaust disk space on a target node, causing the node process to crash or become unable to write further data (blocks, transactions, state). This matches **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation
Any unprivileged network peer can submit transactions that are rejected (e.g., `Reject::Full`, `Reject::Malformed`, `Reject::Resolve`, `Reject::Verification`). No proof-of-work, private keys, or special role is required. The attacker only needs to generate distinct transaction hashes at a rate faster than TTL-based RocksDB compaction reclaims space. This is trivially achievable with a script generating random transactions.

## Recommendation
Write the incremented value back to `self.total_keys_num` in the non-shrink branch:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    } else {
        self.total_keys_num = total_keys_num;  // ← fix
    }
} else {
    self.shrink()?;
}
```

## Proof of Concept
```rust
#[test]
fn test_unbounded_growth() {
    let tmp_dir = tempfile::Builder::new().tempdir().unwrap();
    let mut rr = RecentReject::build(tmp_dir.path(), 2, 10, 3600).unwrap();
    let initial = rr.total_keys_num; // 0 on fresh DB

    for i in 0u64..10_000 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        rr.put(&key, Reject::Malformed(i.to_string(), Default::default())).unwrap();
    }

    // Bug: counter never advanced, shrink never called
    assert_eq!(rr.total_keys_num, initial); // passes — still 0
    // DB on disk contains ~10000 entries, far beyond count_limit=10
    // Verify: rr.estimate_total_keys_num() >> 10
}
```

The existing test at line 39 of `tx-pool/src/component/tests/recent_reject.rs` already inadvertently demonstrates the bug: `recent_reject.total_keys_num < 100` passes after 160 insertions only because the counter is stuck at 0, not because shrink was enforced.
