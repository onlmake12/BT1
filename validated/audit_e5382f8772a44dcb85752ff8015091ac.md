### Title
DAO Withdrawal Interest Silently Zeroed for Genesis-Block Deposits Due to Ambiguous Zero Block-Number Check — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw` uses `deposited_block_number > 0` to distinguish a **deposit cell** (phase-1, cell data = 8 zero bytes) from a **withdrawing cell** (phase-2, cell data = deposit block number as u64 LE). When the original deposit was committed at block 0 (genesis), the withdrawing cell's data is also `[0,0,0,0,0,0,0,0]`, making it indistinguishable from a deposit cell. The function silently returns the raw cell capacity with no interest, exactly mirroring the Morpho pattern where `userIndex == 0` caused `getRewards` to return 0 for a legitimately-zero starting index.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` reads the 8-byte cell data of every DAO-typed input and interprets it as the deposit block number:

```rust
let deposited_block_number =
    match self.data_loader.load_cell_data(cell_meta) {
        Some(data) if data.len() == 8 => LittleEndian::read_u64(&data),
        _ => 0,
    };
if deposited_block_number > 0 {
    // … full interest calculation via calculate_maximum_withdraw …
} else {
    Ok(output.capacity().into())   // ← raw capacity, no interest
}
```

The intended invariant is:

| Cell phase | Cell data | `deposited_block_number` |
|---|---|---|
| Deposit (phase 1) | `[0; 8]` | 0 |
| Withdrawing (phase 2) | deposit block number | > 0 |

The invariant breaks when `deposit block number == 0` (genesis). A withdrawing cell for a genesis deposit stores `0u64` in its data, so the guard `> 0` is false and the else-branch returns the raw capacity, discarding all accrued DAO interest.

The same `transaction_maximum_withdraw` is called from two paths:

1. **`transaction_fee`** — used by `FeeCalculator` during transaction verification. An underestimated `maximum_withdraw` causes `maximum_withdraw.safe_sub(outputs_capacity)` to underflow when the user's output correctly includes interest, returning `DaoError` and rejecting a valid transaction.

2. **`withdrawed_interests`** — used by `dao_field_with_current_epoch` to update the global DAO accumulator field. An underestimated `withdrawed_interests` inflates the `s` field (`current_s = parent_s + nervosdao_issuance - withdrawed_interests`), corrupting the on-chain DAO state for all subsequent depositors.

---

### Impact Explanation

A transaction sender holding a DAO withdrawing cell whose deposit block number is 0 submits a valid phase-2 → phase-3 withdrawal transaction. The node's `FeeCalculator` calls `transaction_fee`, which calls `transaction_maximum_withdraw`. Because `deposited_block_number == 0`, the function returns the raw capacity instead of `capacity + interest`. The subtraction `maximum_withdraw.safe_sub(outputs_capacity)` underflows (the user's output legitimately exceeds the raw capacity), the node returns a `DaoError`, and the transaction is permanently rejected despite being consensus-valid. Additionally, if the block assembler processes such a transaction, the DAO field written to the block header is incorrect, silently corrupting the global accumulation-rate state.

---

### Likelihood Explanation

On the CKB mainnet genesis block, no cells carry the DAO type script, so the condition is not triggered in the current mainnet deployment. However:

- The CKB chain-spec format explicitly supports arbitrary genesis transactions, and any operator or chain deploying a custom network (testnet, devnet, sidechain) with DAO cells in the genesis block will hit this bug.
- The `calculate_dao_maximum_withdraw` RPC endpoint calls `DaoCalculator::calculate_maximum_withdraw` directly (bypassing `transaction_maximum_withdraw`) and is therefore unaffected, meaning the RPC will return the correct interest amount while the node simultaneously rejects the withdrawal transaction — a confusing and exploitable discrepancy for any RPC caller.

---

### Recommendation

Replace the ambiguous sentinel check with an explicit structural distinction. The deposit block number stored in a withdrawing cell's data is the block number of the **deposit header**, which can be verified against the header dep. The correct guard is to check whether the witness provides a valid `input_type` field (the header-dep index), not whether the stored block number is non-zero:

```rust
// Instead of:
if deposited_block_number > 0 { … }

// Use: attempt to parse the witness; if it succeeds, treat as withdrawing cell
// regardless of the numeric value of deposited_block_number.
let is_withdrawing_cell = rtx.transaction.witnesses()
    .get(i)
    .and_then(|w| WitnessArgs::from_slice(&w.raw_data()).ok())
    .and_then(|w| w.input_type().to_opt())
    .map(|d| d.len() == 8)
    .unwrap_or(false);

if is_withdrawing_cell {
    // full interest path, verify deposit_header.number() == deposited_block_number
    …
} else {
    Ok(output.capacity().into())
}
```

This correctly handles `deposited_block_number == 0` (genesis deposit) while preserving the existing behaviour for all other cases.

---

### Proof of Concept

1. Construct a genesis block containing a DAO-typed cell (capacity 1 000 000 CKB, cell data `[0;8]`).
2. Mine several epochs so the accumulation rate `ar` grows from `10^16` to, say, `1.05 × 10^16`.
3. Create a phase-2 (prepare-withdrawal) transaction spending the genesis DAO cell; the output cell data is `0u64.to_le_bytes()` (deposit block = 0).
4. Create a phase-3 (withdraw) transaction spending the phase-2 cell, with output capacity = `calculate_dao_maximum_withdraw(genesis_outpoint, phase2_block_hash)` (RPC returns correct value ≈ 1 050 000 CKB).
5. Submit the phase-3 transaction. The node calls `transaction_maximum_withdraw`, reads `deposited_block_number = 0`, enters the else-branch, returns `1 000 000 CKB` (raw capacity). `transaction_fee` computes `1 000 000 - 1 050 000` → underflow → `DaoError::InvalidOutPoint` → transaction rejected.
6. The RPC `calculate_dao_maximum_withdraw` simultaneously returns `1 050 000 CKB`, confirming the discrepancy. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    /// Calculates the new dao field with specified [`EpochExt`].
    pub fn dao_field_with_current_epoch(
        &self,
        rtxs: impl Iterator<Item = &'a ResolvedTransaction> + Clone,
        parent: &HeaderView,
        current_block_epoch: &EpochExt,
    ) -> Result<Byte32, DaoError> {
        // Freed occupied capacities from consumed inputs
        let freed_occupied_capacities =
            rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
                self.input_occupied_capacities(rtx)
                    .and_then(|c| capacities.safe_add(c))
            })?;
        let added_occupied_capacities = self.added_occupied_capacities(rtxs.clone())?;
        let withdrawed_interests = self.withdrawed_interests(rtxs)?;

        let (parent_ar, parent_c, parent_s, parent_u) = extract_dao_data(parent.dao());

        // g contains both primary issuance and secondary issuance,
        // g2 is the secondary issuance for the block, which consists of
        // issuance for the miner, NervosDAO and treasury.
        // When calculating issuance in NervosDAO, we use the real
        // issuance for each block(which will only be issued on chain
        // after the finalization delay), not the capacities generated
        // in the cellbase of current block.
        let current_block_number = parent.number() + 1;
        let current_g2 = current_block_epoch.secondary_block_issuance(
            current_block_number,
            self.consensus.secondary_epoch_reward(),
        )?;
        let current_g = current_block_epoch
            .block_reward(current_block_number)
            .and_then(|c| c.safe_add(current_g2))?;

        let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
            / u128::from(parent_c.as_u64());
        let miner_issuance =
            Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
        let nervosdao_issuance = current_g2.safe_sub(miner_issuance)?;

        let current_c = parent_c.safe_add(current_g)?;
        let current_u = parent_u
            .safe_add(added_occupied_capacities)
            .and_then(|u| u.safe_sub(freed_occupied_capacities))?;
        let current_s = parent_s
            .safe_add(nervosdao_issuance)
            .and_then(|s| s.safe_sub(withdrawed_interests))?;

        let ar_increase128 =
            u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
        let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
        let current_ar = parent_ar
            .checked_add(ar_increase)
            .ok_or(DaoError::Overflow)?;

        Ok(pack_dao_data(current_ar, current_c, current_s, current_u))
    }

    fn withdrawed_interests(
        &self,
        mut rtxs: impl Iterator<Item = &'a ResolvedTransaction> + Clone,
    ) -> Result<Capacity, DaoError> {
        let maximum_withdraws = rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
            self.transaction_maximum_withdraw(rtx)
                .and_then(|c| capacities.safe_add(c).map_err(Into::into))
        })?;
        let input_capacities = rtxs.try_fold(Capacity::zero(), |capacities, rtx| {
            let tx_input_capacities = rtx.resolved_inputs.iter().try_fold(
                Capacity::zero(),
                |tx_capacities, cell_meta| {
                    let output_capacity: Capacity = cell_meta.cell_output.capacity().into();
                    tx_capacities.safe_add(output_capacity)
                },
            )?;
            capacities.safe_add(tx_input_capacities)
        })?;
        maximum_withdraws
            .safe_sub(input_capacities)
            .map_err(Into::into)
    }
