### Title
Unbounded `while` Loop in `Echo.executeCallback` Can Cause Out-of-Gas Revert, Permanently Blocking Callback Fulfillment — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that iterates over all fulfilled (inactive) requests between `_state.firstUnfulfilledSeq` and `_state.currentSequenceNumber`. An attacker who creates and fulfills a sufficient number of requests can cause this loop to exhaust the block gas limit, permanently preventing any subsequent `executeCallback` call from succeeding.

---

### Finding Description

Inside `executeCallback`, after `clearRequest` is called at line 164, the following loop runs:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs a cold `SLOAD` costing approximately 2,100 gas. If N inactive (fulfilled) requests exist between `firstUnfulfilledSeq` and the next active request, the loop runs N times, consuming ~2,100 × N gas.

The developers themselves flagged this in a TODO comment immediately above the loop:

> "TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. a better solution would be a doubly-linked list of active requests." [2](#0-1) 

**Critical ordering issue:** `clearRequest(sequenceNumber)` is called at line 164 *before* the loop. If the loop causes an out-of-gas revert, the entire transaction reverts — including `clearRequest`. The request remains active, but `firstUnfulfilledSeq` is also not advanced. Every subsequent `executeCallback` call faces the same oversized loop, creating a **permanent DoS**.

**Attack path (unprivileged user):**
1. Attacker calls `requestPriceUpdatesWithCallback` N times, paying the required fee each time (sequence numbers 1…N).
2. Attacker (or anyone) calls `executeCallback` for each of those N requests, fulfilling them. `firstUnfulfilledSeq` stays at its initial value because a single long-lived active request at a lower sequence number pins it.
3. A legitimate user's request at sequence number N+1 is now pending.
4. When the provider's keeper calls `executeCallback` for sequence N+1, the `while` loop must iterate over all N inactive slots before stopping. If N × 2,100 gas exceeds the block gas limit (~30 M on Ethereum mainnet, giving N ≈ 14,285), the transaction reverts.
5. `firstUnfulfilledSeq` is never advanced; every future `executeCallback` call also reverts. The Echo contract is permanently DoS'd. [3](#0-2) 

---

### Impact Explanation

**High.** If `executeCallback` cannot complete, consumer contracts never receive their price-feed callbacks. Funds paid by users at request time are locked in the contract with no recovery path, because there is no function to manually advance `firstUnfulfilledSeq` or cancel a stuck request. All pending requests for all users are permanently blocked.

---

### Likelihood Explanation

**Low.** The attacker must pay the Echo fee for each of the N requests needed to overflow the gas budget. At Ethereum's ~30 M block gas limit and ~2,100 gas per loop iteration, roughly 14,000 paid requests are required. This is a non-trivial economic cost. However, on chains with lower block gas limits or higher per-gas costs the threshold is lower, and a well-funded adversary can execute the attack on any chain.

---

### Recommendation

1. **Replace the linear scan with a doubly-linked list** of active requests (as the TODO comment already suggests), so `firstUnfulfilledSeq` advancement is O(1).
2. **Alternatively**, cap the number of loop iterations per call (e.g., advance at most 100 slots per `executeCallback` invocation) and expose a separate `advanceFirstUnfulfilledSeq(uint256 maxSteps)` function so the pointer can be advanced incrementally.
3. **Add an explicit gas check** before the callback invocation (analogous to Entropy's `InsufficientGas` guard) so that insufficient-gas conditions fail fast and predictably rather than silently consuming all remaining gas. [4](#0-3) 

---

### Proof of Concept

```solidity
// Attacker script (pseudocode)
uint256 N = 15_000; // exceeds ~14,285 threshold on Ethereum
for (uint i = 0; i < N; i++) {
    // Each call pays the required fee; sequence numbers 1..N are assigned
    echo.requestPriceUpdatesWithCallback{value: fee}(provider, block.timestamp, priceIds, gasLimit);
}
// Fulfill requests 2..N (leave request 1 active to pin firstUnfulfilledSeq)
for (uint i = 2; i <= N; i++) {
    echo.executeCallback(provider, i, updateData, priceIds);
}
// Now firstUnfulfilledSeq == 1 (pinned by active request 1)
// currentSequenceNumber == N+1

// Legitimate user creates request N+1
echo.requestPriceUpdatesWithCallback{value: fee}(provider, block.timestamp, priceIds, gasLimit);

// Provider keeper tries to fulfill request N+1 — the while loop iterates N times and OOGs
echo.executeCallback(provider, N+1, updateData, priceIds); // REVERTS with OOG

// All future executeCallback calls also revert — permanent DoS
``` [5](#0-4)
