Audit Report

## Title
`RecentReject::put` Never Writes Back Incremented `total_keys_num`, Making `shrink()` Permanently Unreachable — (File: tx-pool/src/component/recent_reject.rs)

## Summary
In `RecentReject::put`, `self.total_keys_num.checked_add(1)` produces a value bound only to a local shadow variable; `self.total_keys_num` is never updated. The field stays at its DB-estimated initial value (0 on a fresh node) for the entire process lifetime. The guard `total_keys_num > self.count_limit` therefore always evaluates `1 > count_limit`, which is permanently false for any reasonable limit, so `shrink()` is never invoked and the `recent_reject` RocksDB store grows without bound until disk exhaustion crashes the node.

## Finding Description
`RecentReject` is a sharded RocksDB-with-TTL store. It is initialized with `total_keys_num` estimated from the DB at construction time (0 for a fresh node): [1](#0-0) 

In `put()`, lines 62–69: [2](#0-1) 

`self.total_keys_num.checked_add(1)` returns a new `u64` bound to the local name `total_keys_num`. `self.total_keys_num` is never assigned. After `put()` returns, `self.total_keys_num` is identical to its pre-call value. On a fresh node this is 0, so every call evaluates `0 + 1 = 1 > count_limit` (e.g., 20,000), which is always false. `shrink()` is therefore unreachable through the normal guard.

`shrink()` does correctly re-estimate and write back `self.total_keys_num`, but it is never reached: [3](#0-2) 

The existing test inadvertently confirms the bug: after 160 insertions against a limit of 100, it asserts `recent_reject.total_keys_num < 100`. This passes trivially because `total_keys_num` was never incremented from 0, not because the limit was correctly enforced: [4](#0-3) 

The stale value is also what the RPC exposes as `total_recent_reject_num`, giving operators no warning of the growing DB.

## Impact Explanation
Any unprivileged user can submit transactions via RPC (`send_transaction`) or P2P relay. Rejected transactions (invalid scripts, double-spends, below-minimum fee rate) are recorded in the `recent_reject` DB. Because `shrink()` is never called, the DB grows without bound. Disk exhaustion will crash the node. TTL provides partial mitigation, but at a sufficiently high submission rate, disk exhaustion occurs before RocksDB TTL-based compaction reclaims space. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attack requires no privilege. Any RPC caller or P2P peer can submit a stream of unique transactions that fail validation. Rejected transactions are a normal operational occurrence. The attack is repeatable, requires no special knowledge of node state, and can be sustained indefinitely at low cost (e.g., transactions with invalid scripts or below-minimum fee rates).

## Recommendation
In `put()`, write the incremented value back to `self.total_keys_num` before the limit check:

```rust
match self.total_keys_num.checked_add(1) {
    Some(n) => {
        self.total_keys_num = n;  // ← missing assignment
        if n > self.count_limit {
            self.shrink()?;
        }
    }
    None => {
        self.shrink()?;
    }
}
```

## Proof of Concept
The existing test at `tx-pool/src/component/tests/recent_reject.rs` line 39 already demonstrates the bug: after 160 insertions against a limit of 100, `total_keys_num` is still 0 and `assert!(recent_reject.total_keys_num < 100)` passes trivially.

Targeted reproduction:
1. Build `RecentReject` with `count_limit = 10`.
2. Insert 50 unique rejected transactions via `put()`.
3. Assert `rr.total_keys_num == 0` — passes, proving the counter was never incremented.
4. Call `estimate_total_keys_num()` directly — returns ~50, far exceeding the limit of 10, proving `shrink()` was never called. [5](#0-4)

            if total_keys_num > self.count_limit {
                self.shrink()?;
            }
        } else {
            // overflow occurred, try shrink
            self.shrink()?;
        }
        Ok(())
    }
