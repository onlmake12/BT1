### Title
Governance `UpgradeContract` Action Always Reverts, Permanently Disabling Emergency Upgrade Safeguard — (`target_chains/fuel/contracts/pyth-contract/src/main.sw`)

---

### Summary

The Fuel Pyth contract's `execute_governance_instruction` function recognizes the `UpgradeContract` governance action but unconditionally reverts its execution. Any governance VAA carrying an `UpgradeContract` action will always fail on Fuel, making it impossible for Pyth governance to upgrade or patch the deployed contract — the exact analog of the zkSync STM force-upgrade safeguard being non-functional.

---

### Finding Description

In `target_chains/fuel/contracts/pyth-contract/src/main.sw`, the `execute_governance_instruction` function dispatches on `GovernanceAction::UpgradeContract` and immediately calls `revert(0u64)` after logging a message:

```sway
match gi.action {
    GovernanceAction::UpgradeContract => {
        require(gi.target_chain_id != 0, PythError::InvalidGovernanceTarget);
        // TODO: implement upgrade_upgradeable_contract(uc) when Fuel releases the upgrade standard library;
        log("Upgrade functionality not implemented");
        revert(0u64);   // <-- unconditional revert
    },
    ...
}
``` [1](#0-0) 

The governance instruction parser in `target_chains/fuel/contracts/pyth-interface/src/data_structures/governance_instruction.sw` correctly maps action byte `0` to `GovernanceAction::UpgradeContract` and does **not** reject it — the action is parsed successfully and passed to the dispatcher:

```sway
0 => GovernanceAction::UpgradeContract, // Not implemented
``` [2](#0-1) 

This means the governance sequence number is consumed (incremented) by `verify_governance_vm` before the revert, so the VAA cannot even be replayed after a fix without issuing a new governance VAA with a higher sequence number. [3](#0-2) 

A second instance of the same pattern exists in the Stylus receiver:

```rust
fn upgrade_contract(&self, _new_implementation: FixedBytes<32>) {
    unimplemented!("Upgrade contract not yet implemented");
}
``` [4](#0-3) 

---

### Impact Explanation

Pyth governance cannot upgrade the Fuel Pyth contract under any circumstances. If a critical vulnerability is discovered in the deployed Fuel contract (e.g., a price manipulation bug, fee drain, or data-source bypass), the designed emergency upgrade path is completely non-functional. The only remediation would be a full redeployment and manual migration of all state, which is far slower and more disruptive than a governance-controlled upgrade. This directly mirrors the zkSync finding: a designed safeguard mechanism is rendered inoperative by an incomplete implementation.

---

### Likelihood Explanation

The likelihood of needing an emergency upgrade is low in any given time window, but the consequence when it is needed is severe. The Fuel contract is a live production contract processing real price feeds. Any exploitable bug discovered after deployment cannot be patched via the standard governance upgrade path, leaving the contract permanently vulnerable until a full redeployment is coordinated. The governance message submitter (an externally reachable role explicitly in scope) can trigger this failure path with any valid `UpgradeContract` VAA.

---

### Recommendation

1. Implement the `UpgradeContract` governance action on Fuel once the Fuel upgrade standard library is available. Until then, document clearly that the Fuel deployment is non-upgradeable and treat it accordingly in the security model.
2. Apply the same fix to the Stylus receiver (`target_chains/stylus/contracts/pyth-receiver/src/governance.rs`), replacing the `unimplemented!()` panic with a real upgrade dispatch.
3. Consider whether the sequence-number consumption on a failed `UpgradeContract` call should be reverted (i.e., the sequence should not advance if the action cannot be executed), to avoid wasting governance sequence slots.

---

### Proof of Concept

1. Pyth governance issues a valid Wormhole VAA with:
   - Emitter matching the configured Pyth governance data source
   - Sequence number greater than `last_executed_governance_sequence`
   - Payload: governance magic `PTGM` + module `Target` (1) + action `UpgradeContract` (0) + Fuel chain ID + new implementation hash
2. Any caller submits this VAA to `execute_governance_instruction` on the Fuel Pyth contract.
3. `verify_governance_vm` succeeds and advances `last_executed_governance_sequence`.
4. The `match` arm for `GovernanceAction::UpgradeContract` is reached and `revert(0u64)` is called unconditionally.
5. The transaction reverts. The governance VAA is consumed (sequence advanced) but the upgrade never occurs.
6. Governance must issue a new VAA with a higher sequence number to retry — but the retry will also revert, indefinitely. [5](#0-4)
