### Title
Pre-PoW O(proposals) Hash Computation in `SendBlock` Processing Enables CPU Exhaustion — (`sync/src/synchronizer/block_process.rs`, `util/types/src/extension.rs`)

### Summary

When a `SendBlock` P2P message is received, `BlockProcess::execute()` immediately calls `into_view()` on the raw packed block, which internally calls `reset_header_with_hashes()` → `calc_proposals_hash()`. This performs O(proposals_count) blake2b hashing over all proposal short IDs **before** any PoW verification occurs. An unprivileged peer can send many blocks with max proposals but invalid PoW, forcing the victim node to perform this hashing work on every block before the PoW check rejects them.

### Finding Description

**Step 1 — Entry point and immediate O(proposals) work:**

In `block_process.rs`, the very first operation on a received `SendBlock` message is:

```rust
let block = Arc::new(self.message.block().to_entity().into_view());
``` [1](#0-0) 

`into_view()` is implemented in `views.rs` as:

```rust

    let tx_hashes = self.calc_tx_hashes();
    let tx_witness_hashes = self.calc_tx_witness_hashes();

    Self::block_into_view_internal(block, tx_hashes, tx_witness_hashes)

**Step 3 — `calc_proposals_hash` is O(proposals_count):**

```rust
pub fn calc_proposals_hash(&self) -> packed::Byte32 {
    if self.is_empty() {
        packed::Byte32::zero()
    } else {
        let mut blake2b = new_blake2b();
        for id in self.iter() {
            blake2b.update(id.as_slice());
        }
        blake2b.finalize(&mut ret);
        ret.into()

``` [4](#0-3) 

**Step 4 — PoW check happens much later:**

PoW verification is inside `PowVerifier`, which is invoked by `HeaderVerifier::verify()` — called only during the chain verification pipeline triggered by `asynchronous_process_remote_block`, well after `into_view()` has already completed. [5](#0-4) [6](#0-5) 

The `new_block_received` dedup check at line 43 only guards against the **same block hash** being processed twice. Since `into_view()` recalculates the header hash from the block content, an attacker can vary the nonce to produce a unique hash per block, bypassing this dedup guard for each new block sent. [7](#0-6) 

### Impact Explanation

Each malicious `SendBlock` message with `max_block_proposals_limit` proposals (1,500 × 10 bytes = 15,000 bytes) forces the victim to run a full blake2b pass over all proposals, plus compute all transaction hashes (`calc_tx_hashes`, `calc_tx_witness_hashes`), before the block is rejected for invalid PoW. The attacker pays only network bandwidth; the victim pays CPU for every block received. Sustained at high message rate from one or more peers, this degrades node performance and can contribute to network congestion. [8](#0-7) 

### Likelihood Explanation

Any unprivileged P2P peer can send `SendBlock` messages. No authentication, stake, or hashpower is required. The attacker only needs to craft blocks with a valid structure (correct field sizes, max proposals, arbitrary nonce). The dedup check is trivially bypassed by varying the nonce. This is a low-cost, low-skill attack.

### Recommendation

Perform a lightweight PoW check on the raw packed header **before** calling `into_view()`. Since `PowVerifier::verify()` only needs `self.header.data()` (the packed `Header`), it can be applied directly to `self.message.block().header()` before any O(block_size) computation. Only if PoW passes should `into_view()` (and thus `reset_header_with_hashes` / `calc_proposals_hash`) be invoked.

### Proof of Concept

1. Craft a `packed::Block` with `proposals` filled to `max_block_proposals_limit` (1,500 entries), a valid structure, but an invalid nonce.
2. Send it repeatedly as `SendBlock` P2P messages, varying the nonce each time to produce unique block hashes.
3. Observe that each message triggers `calc_proposals_hash` (15,000-byte blake2b pass) + full tx hash computation before PoW rejection.
4. Benchmark: measure CPU time in `reset_header_with_hashes` vs. `PowVerifier::verify()` per block; assert that the former is invoked before the latter in the call stack. [9](#0-8) [10](#0-9) [11](#0-10)

    fn reset_header_with_hashes(
        self,
        tx_hashes: &[packed::Byte32],
        tx_witness_hashes: &[packed::Byte32],
    ) -> packed::Block {
        let raw_transactions_root = merkle_root(tx_hashes);
        let witnesses_root = merkle_root(tx_witness_hashes);
        let transactions_root = merkle_root(&[raw_transactions_root, witnesses_root]);
        let proposals_hash = self.as_reader().calc_proposals_hash();
        let extra_hash = self.as_reader().calc_extra_hash().extra_hash();
        let raw_header = self
            .header()
            .raw()
            .as_builder()
            .transactions_root(transactions_root)
            .proposals_hash(proposals_hash)
            .extra_hash(extra_hash)
            .build();
        let header = self.header().as_builder().raw(raw_header).build();
        if let Some(extension) = self.extension() {
            packed::BlockV1::new_builder()
                .header(header)
                .uncles(self.uncles())
                .transactions(self.transactions())
                .proposals(self.proposals())
                .extension(extension)
                .build()
                .as_v0()
        } else {
            self.as_builder().header(header).build()
        }
    }
