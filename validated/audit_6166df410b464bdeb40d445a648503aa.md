### Title
Off-by-One in RFC0044 MMR Activation Epoch Check Causes 1-Block Gap in MMR Extension Coverage — (`tx-pool/src/block_assembler/mod.rs` and `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

`build_extension` in the block assembler and `BlockExtensionVerifier` both check RFC0044 (MMR chain-root extension) activation using the **parent block's** epoch number instead of the **candidate/new block's** epoch number. This is a direct off-by-one at the epoch activation boundary: the first block of the RFC0044 activation epoch is assembled without the MMR extension and accepted by the verifier without it, creating a 1-block gap in MMR coverage at the activation boundary.

---

### Finding Description

`build_extension` determines whether to include the MMR chain-root extension by calling:

```rust
let mmr_activate = snapshot
    .consensus()
    .rfc0044_active(tip_header.epoch().number());
```

`tip_header` is the current chain tip — the **parent** of the block being assembled. The candidate block's epoch number is `current_epoch.number()`, which at an epoch boundary equals `tip_header.epoch().number() + 1`.

The activation check is:

```rust
pub fn rfc0044_active(&self, target: EpochNumber) -> bool {
    target >= rfc0044_active_epoch   // mainnet: 8651, testnet: 5711
}
```

At the epoch boundary where RFC0044 activates (epoch `E = RFC0044_ACTIVE_EPOCH`):
- The tip (parent) is the last block of epoch `E − 1`.
- `tip_header.epoch().number()` = `E − 1`.
- `rfc0044_active(E − 1)` = **false**.
- The assembler omits the MMR extension from the first block of epoch `E`.

`BlockExtensionVerifier` has the identical off-by-one:

```rust
let mmr_active = self
    .context
    .consensus
    .rfc0044_active(self.parent.epoch().number());
```

Because both sides use the parent's epoch number, they are mutually consistent — the verifier accepts the extension-less first block of epoch `E`. The code comment in `build_extension` explicitly acknowledges this:

> "The use of the epoch number of the tip here leads to an off-by-one bug, so be careful, it needs to be preserved for consistency reasons and not fixed directly."

The correct check in both places should use the **new block's** epoch number (`current_epoch.number()` in the assembler; the block-under-verification's own epoch in the verifier).

---

### Impact Explanation

The first block of the RFC0044 activation epoch (block at the epoch-`E` boundary) is permanently committed to the canonical chain **without** an MMR chain-root extension. Every subsequent block carries the extension. This creates a 1-block hole in the MMR proof chain:

- Light clients (RFC0044 consumers) that rely on the MMR to verify chain history cannot obtain a valid MMR proof for that specific block.
- The MMR root stored in block `E+1` covers blocks up to `E+1 − 1`, but block `E` itself has no MMR root, breaking the continuous proof chain at the activation boundary.
- Any cross-chain bridge or light-client protocol that assumes the MMR covers all blocks from the activation epoch onward will encounter an unverifiable gap at exactly this block.

---

### Likelihood Explanation

This is deterministic and unconditional: every CKB mainnet node will produce and accept the first block of epoch 8651 without the MMR extension. No special attacker action is required — the gap is baked into the consensus rules as deployed. A miner calling `get_block_template` (the standard RPC entry point) at the epoch boundary will receive a template without the extension, and the resulting block will pass all verifiers.

---

### Recommendation

The correct fix is to use the **candidate block's** epoch number in both places:

In `build_extension` (`tx-pool/src/block_assembler/mod.rs`):
```rust
// Use current_epoch.number() (the new block's epoch), not tip_header.epoch().number()
let mmr_activate = snapshot
    .consensus()
    .rfc0044_active(current_epoch.number());
```

In `BlockExtensionVerifier::verify` (`verification/contextual/src/contextual_block_verifier.rs`):
```rust
// Use block.epoch().number() (the block being verified), not self.parent.epoch().number()
let mmr_active = self
    .context
    .consensus
    .rfc0044_active(block.epoch().number());
```

Because the bug is already committed to mainnet history (the first block of epoch 8651 is already mined without the extension), any fix must be applied as a forward-only rule change and must not retroactively invalidate the already-accepted block.

---

### Proof of Concept

1. Observe `RFC0044_ACTIVE_EPOCH = 8651` (mainnet).
2. At the last block of epoch 8650 (tip), call `get_block_template` via RPC.
3. The assembler calls `rfc0044_active(8650)` → `8650 >= 8651` → **false** → no extension field in the template.
4. The miner submits the block; `BlockExtensionVerifier` calls `rfc0044_active(parent.epoch().number())` = `rfc0044_active(8650)` → **false** → extension not required → block accepted.
5. The first block of epoch 8651 is on-chain with no MMR extension.
6. The second block of epoch 8651 calls `rfc0044_active(8651)` → **true** → extension required and present.
7. A light client requesting an MMR proof for the first block of epoch 8651 finds no chain-root extension at that position, breaking the continuous MMR proof chain. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    pub(crate) fn build_extension(snapshot: &Snapshot) -> Result<Option<packed::Bytes>, AnyError> {

        // The use of the epoch number of the tip here leads to an off-by-one bug,
        // so be careful, it needs to be preserved for consistency reasons and not fixed directly.

                .get_root()

            Ok(Some(bytes))
        } else {
            Ok(None)
        }
    }

        let extra_fields_count = block.data().count_extra_fields();

        let mmr_active = self
            .context
            .consensus
            .rfc0044_active(self.parent.epoch().number());
        match extra_fields_count {
            0 => {
                if mmr_active {
                    return Err(BlockErrorKind::NoBlockExtension.into());
                }
            }
            1 => {
                let extension = if let Some(data) = block.extension() {
                    data
                } else {
                    return Err(BlockErrorKind::UnknownFields.into());
                };
                if extension.is_empty() {
                    return Err(BlockErrorKind::EmptyBlockExtension.into());
                }
                if extension.len() > 96 {
                    return Err(BlockErrorKind::ExceededMaximumBlockExtensionBytes.into());
