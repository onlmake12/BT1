### Title
Unbounded Indexer RPC Iteration via Default `usize::MAX` Request Limit Enables Memory Exhaustion — (File: `util/indexer/src/service.rs`)

---

### Summary

The CKB indexer's `IndexerService` defaults `request_limit` to `usize::MAX` when the operator has not explicitly configured it. Because the RPC-exposed `limit` parameter is a `Uint32` (max ~4.29 billion), and `u32::MAX < usize::MAX` on all 64-bit targets, the guard that is supposed to cap query size is permanently bypassed under the default configuration. An unprivileged RPC caller can pass `limit = 0xFFFFFFFF` to `get_cells` or `get_transactions`, forcing the node to iterate and accumulate billions of RocksDB records into a heap-allocated `Vec` for up to the full 10-second timeout window per request. Concurrent requests amplify memory and CPU pressure.

---

### Finding Description

In `util/indexer/src/service.rs`, `IndexerService::new` initialises `request_limit` as:

```rust
request_limit: config.request_limit.unwrap_or(usize::MAX),
``` [1](#0-0) 

The corresponding config field `request_limit` in `IndexerConfig` defaults to `None`:

```rust
pub request_limit: Option<usize>,
// ...
request_limit: None,
``` [2](#0-1) [3](#0-2) 

So unless an operator explicitly sets `request_limit` in their config, the effective cap is `usize::MAX`.

The guard in `get_cells` is:

```rust
if limit > self.request_limit {
    return Err(Error::invalid_params(...));
}
``` [4](#0-3) 

Because `limit` is cast from `Uint32` (max `4_294_967_295`) and `self.request_limit` is `usize::MAX` (`18_446_744_073_709_551_615` on 64-bit), the condition `limit > self.request_limit` is **always false**. The same guard exists identically in `get_transactions`: [5](#0-4) 

After the guard, both functions collect results into a `Vec` bounded only by `limit` and the `TimeoutIterator`:

```rust
.take(limit)
.collect::<Vec<_>>();
``` [6](#0-5) 

The `TimeoutIterator` stops iteration after `timeout_limit` seconds (default: 10 seconds):

```rust
timeout_limit: Duration::from_secs(config.timeout_limit.unwrap_or(10)),
``` [7](#0-6) 

However, during those 10 seconds, the node is continuously reading RocksDB records and pushing them into a heap-allocated `Vec`. On a node with millions of indexed cells, this can allocate gigabytes of memory per request before the timeout fires.

---

### Impact Explanation

- **Memory exhaustion**: Each request with `limit = u32::MAX` causes the node to allocate a `Vec` that grows for up to 10 seconds of continuous RocksDB reads. On a mainnet node with tens of millions of indexed cells, this can reach gigabytes per request.
- **CPU exhaustion**: RocksDB iteration, key parsing, and filter evaluation run at full speed for the entire timeout window.
- **Amplification**: An attacker can send many concurrent requests. Each occupies a thread and accumulates its own `Vec`, multiplying memory pressure.
- **Node unavailability**: Sustained memory pressure can trigger OOM-kill of the CKB process or cause severe latency for all other RPC consumers and internal node operations.

---

### Likelihood Explanation

Any unprivileged caller with network access to the node's RPC port can trigger this. The indexer RPC (`get_cells`, `get_transactions`) is documented and publicly accessible when the indexer module is enabled. No authentication, special role, or key material is required. The attacker only needs to craft a standard JSON-RPC request with `limit` set to `4294967295`.

---

### Recommendation

1. **Set a safe default for `request_limit`**: Replace `usize::MAX` with a bounded default (e.g., `1000` or `10000`) that matches realistic use cases:
   ```rust
   request_limit: config.request_limit.unwrap_or(1000),
   ```
2. **Document the configuration**: Make clear in the default config template that `request_limit` must be set to a finite value when the indexer is exposed publicly.
3. **Consider a hard cap**: Enforce an absolute maximum (e.g., `u16::MAX`) regardless of operator configuration, so that even a misconfigured node cannot be trivially exhausted.

---

### Proof of Concept

Send the following JSON-RPC request to a CKB node with the indexer enabled and default configuration:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells",
  "params": [
    {
      "script": {
        "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
        "hash_type": "type",
        "args": "0x"
      },
      "script_type": "lock"
    },
    "asc",
    "0xffffffff",
    null
  ]
}
```

With `request_limit` unset (default `usize::MAX`), the guard at line 216 of `util/indexer/src/service.rs` passes because `0xffffffff < usize::MAX`. The node then iterates RocksDB for 10 seconds, accumulating all matching cells into memory. Repeating this request concurrently (e.g., 20 parallel connections) will multiply memory consumption proportionally, potentially exhausting available RAM and crashing the node process.
