### Title
`consecutive_failed` Never Reset on Successful Package Inclusion Causes Premature Block-Template Loop Termination — (`tx-pool/src/component/tx_selector.rs`)

---

### Summary

`TxSelector::txs_to_commit` uses a `consecutive_failed` counter as a heuristic to stop iterating when the block is nearly full. The counter is incremented on every size/cycles or ancestor-check failure, but is **never reset to zero after a successful package inclusion**. This means the counter accumulates across successes and failures alike, causing the selection loop to terminate after `MAX_CONSECUTIVE_FAILURES + 1` (4001) total failures — not 4001 *consecutive* failures — even when high-fee transactions that individually fit the remaining block space are still available.

---

### Finding Description

In `txs_to_commit`, `consecutive_failed` is initialized once and only ever incremented: [1](#0-0) 

It is incremented on size/cycles overflow: [2](#0-1) 

And on ancestor-not-proposed failure: [3](#0-2) 

The successful inclusion path (lines 207–220) adds entries to the block and calls `update_modified_entries`, but **contains no `consecutive_failed = 0` reset**: [4](#0-3) 

The constant is defined as: [5](#0-4) 

The comment explicitly says this is a heuristic to stop "when the block is close to full" — implying the counter should track *consecutive* failures, not *total* failures. The variable name `consecutive_failed` confirms the intended semantics. The missing reset is a deviation from that invariant.

The call chain from an unprivileged submitter is:

`send_transaction` (RPC) → pending pool → proposed pool (after proposal window) → `package_txs` → `TxSelector::txs_to_commit` [6](#0-5) 

This is called from `BlockAssembler::update_full` and `update_transactions` on every block template refresh: [7](#0-6) 

---

### Impact Explanation

An attacker who submits 4001+ transactions to the proposed pool with `ancestors_size` carefully sized to be just above `size_limit - S` (where `S` is the space consumed by the first few committed high-fee transactions) can cause the selection loop to exhaust `consecutive_failed > 4000` and break early, skipping all remaining proposed transactions — including small, high-fee transactions that would individually fit the remaining block space. Every miner on every node calling `get_block_template` receives a systematically under-filled block template. This reduces network throughput and enables low-cost congestion by occupying proposal slots cheaply.

---

### Likelihood Explanation

- An unprivileged attacker submits transactions via the standard `send_transaction` RPC — no special access required.
- CKB's two-phase commit means transactions move from pending to proposed after the proposal window (default 2 blocks); miners naturally include proposals since they are cheap (10-byte short IDs).
- The attacker needs to craft ~4001 transactions with `ancestors_size` in the range `(size_limit - S, size_limit]`. Since `S` (the size of the first committed tx) is observable from the mempool, this is predictable.
- The attack cost is ~4001 minimum-fee transactions per block, which is low relative to the impact.
- The attack must be repeated each block (proposed pool is refreshed), but the cost remains low.

---

### Recommendation

Add `consecutive_failed = 0;` immediately after a successful package inclusion, at the end of the success path in `txs_to_commit`, just before or after the `update_modified_entries` call:

```rust
self.update_modified_entries(&ancestors);
consecutive_failed = 0; // reset: a successful inclusion means the block is not yet "full"
```

This restores the intended semantics: the counter tracks *consecutive* failures, and a successful inclusion resets the streak. [8](#0-7) 

---

### Proof of Concept

```
Setup:
  size_limit = 600_000 bytes
  S = 100 bytes  (size of first committed tx, fee_rate = 1000)
  attacker_ancestors_size = 599_950  (passes filter: <= 600_000; fails check after first tx: 100 + 599_950 > 600_000)
  attacker_fee_rate = 999  (just below first tx, above all other legitimate txs)
  attacker_tx_count = 4001

Pool state (sorted by fee rate, descending):
  [0]   legitimate_tx_A  fee_rate=1000  ancestors_size=100   → COMMITTED (size becomes 100)
  [1..4001] attacker_tx_i fee_rate=999  ancestors_size=599_950 → FAIL size check (100+599_950 > 600_000)
                                                                  consecutive_failed increments each time
  [4002] legitimate_tx_B  fee_rate=500  ancestors_size=100   → NEVER REACHED (loop breaks at consecutive_failed=4001)

Expected (correct) result: tx_A and tx_B both committed (total size 200 bytes).
Actual (buggy) result:     only tx_A committed; tx_B skipped; block is 599,900 bytes under-filled.

Patched version (consecutive_failed = 0 after success):
  After tx_A is committed, consecutive_failed resets to 0.
  Attacker txs increment it to 4001 again, but tx_B is reached before that if it appears before the 4001st attacker tx.
  (Attacker would need to interleave legitimate txs with attacker txs to still cause harm, which is harder.)
```
