### Title
Missing Staleness Validation in `PythAggregatorV3` Causes Stale Prices to Be Silently Returned - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatible adapter contract, calls `getPriceUnsafe()` in every price-returning function (`latestAnswer`, `latestRoundData`, `getRoundData`) without any staleness check. Any protocol using this adapter as a drop-in Chainlink replacement will silently receive arbitrarily stale prices, directly analogous to the reported `latestAnswer`-returns-0 pattern.

---

### Finding Description

`PythAggregatorV3` is the official Pyth SDK contract for protocols migrating from Chainlink. It implements `AggregatorV3Interface` by wrapping the Pyth oracle. Every price-returning function delegates to `pyth.getPriceUnsafe(priceId)`:

```solidity
// Line 53-56
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// Line 99-119
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}

// Line 76-97
function getRoundData(uint80 _roundId) external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return (_roundId, int256(price.price), price.publishTime, price.publishTime, _roundId);
}
```

`getPriceUnsafe` is explicitly documented in `IPyth.sol` as returning "the most recent price update in this contract without any recency checks" and that "the returned price update may be arbitrarily far in the past." No staleness check against `publishTime` is performed anywhere in `PythAggregatorV3`.

A second compounding issue: `latestRoundData()` sets `answeredInRound = roundId = uint80(price.publishTime)`. The standard Chainlink staleness guard used by integrators — `require(answeredInRound >= roundId)` — will always pass since both values are identical, even when the price is hours or days old.

Pyth is a pull oracle. Prices must be actively pushed on-chain. If the push mechanism fails (network outage, market closure, service disruption), the on-chain price becomes stale. `PythAggregatorV3` provides no protection against this condition. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**High.** Any protocol that deploys `PythAggregatorV3` as a Chainlink replacement and calls `latestAnswer()` or `latestRoundData()` will receive a stale price without any revert or error signal. This directly affects:

- Lending protocols: stale collateral valuations allow under-collateralized borrows or block valid liquidations.
- Swap/AMM protocols: stale prices cause incorrect exchange rates, enabling arbitrage at the protocol's expense.
- Derivatives protocols: stale settlement prices cause incorrect P&L and margin calculations.

The `PythAggregatorV3` contract is the official migration path documented by Pyth for Chainlink users, making its misuse surface large. [4](#0-3) 

---

### Likelihood Explanation

**Low.** The condition requires the on-chain Pyth price to become stale, which happens when the push mechanism fails. This is an uncommon but realistic scenario: network outages, market closures for equity/commodity feeds, or a lapse in the push service. The `PythAggregatorV3` migration guide explicitly notes that Chainlink-compatible apps "may need to schedule these price updates themselves," meaning many integrators may not have a robust push mechanism in place. [5](#0-4) 

---

### Recommendation

Replace all `getPriceUnsafe(priceId)` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`, where `maxAge` is a configurable staleness threshold set at construction time. This mirrors the behavior of Chainlink's `latestRoundData` with a proper `updatedAt` staleness check.

```solidity
uint256 public maxAge; // set in constructor

function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    return int256(price.price);
}
``` [6](#0-5) 

---

### Proof of Concept

1. A lending protocol deploys `PythAggregatorV3(pythAddress, ETH_USD_FEED_ID)` as its ETH/USD price oracle.
2. The protocol's push service goes offline for 2 hours (network outage or market closure).
3. The on-chain Pyth ETH/USD price is now 2 hours stale (e.g., ETH was $3,000 but is now $2,500).
4. An attacker calls the lending protocol's borrow function. The protocol calls `latestRoundData()` on `PythAggregatorV3`.
5. `PythAggregatorV3.latestRoundData()` calls `pyth.getPriceUnsafe(priceId)` — no revert, returns the 2-hour-old price of $3,000.
6. The attacker borrows against the inflated $3,000 collateral valuation, extracting more than the current $2,500 value allows.
7. The protocol suffers a loss of funds.

The standard Chainlink staleness guard `require(answeredInRound >= roundId)` does not protect against this because `PythAggregatorV3.latestRoundData()` always returns `answeredInRound == roundId`. [7](#0-6)

    function getPriceNoOlderThan(
        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
        price = getPriceUnsafe(id);

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();

        return price;
    }
