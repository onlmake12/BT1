### Title
Third-party descendant stuffing forces inflated replacement fee or completely blocks RBF — (`tx-pool/src/pool.rs`)

---

### Summary

`TxPool::check_rbf` automatically accumulates all descendant transaction fees into the minimum replacement fee and hard-rejects any RBF attempt when the descendant count exceeds `MAX_REPLACEMENT_CANDIDATES (100)`. A third party who controls the outputs of a pending transaction (e.g., the recipient of a payment) can pre-stuff the pool with up to 100 descendants before the original sender attempts RBF, either inflating the required replacement fee beyond what the sender planned or completely blocking the sender from replacing their transaction at all.

---

### Finding Description

In `TxPool::check_rbf` (`tx-pool/src/pool.rs`), when a user submits a replacement transaction, two automatic behaviors occur unconditionally:

**1. Descendant count hard-block (Rule #5)**

```rust
// tx-pool/src/pool.rs lines 617–623
let descendants = self.pool_map.calc_descendants(&conflict.id);
replace_count += descendants.len() + 1;
if replace_count > MAX_REPLACEMENT_CANDIDATES {
    return Err(Reject::RBFRejected(format!(
        "Tx conflict with too many txs, conflict txs count: {}, expect <= {}",
        replace_count, MAX_REPLACEMENT_CANDIDATES,
    )));
}
```

If the conflicted transaction has ≥ 100 descendants in the pool, the RBF is rejected outright — regardless of how high the replacement fee is.

**2. Automatic descendant fee accumulation (Rules #3/#4)**

```rust
// tx-pool/src/pool.rs lines 645, 665–670
all_conflicted.extend(entries);   // descendants appended to conflict set
...
if let Some(min_replace_fee) = self.calculate_min_replace_fee(&all_conflicted, entry.size) {
    if fee < min_replace_fee {
        return Err(Reject::RBFRejected(format!(
            "Tx's current fee is {}, expect it to >= {} to replace old txs",
            fee, min_replace_fee,
        )));
    }
}
```

`calculate_min_replace_fee` sums the fees of every entry in `all_conflicted` (original tx + all descendants):

```rust
// tx-pool/src/pool.rs lines 101–113
/// min_replace_fee = sum(replaced_txs.fee) + extra_rbf_fee
fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
    let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
    let replaced_fees: HashMap<_, _> = conflicts
        .iter()
        .map(|c| (c.id.clone(), c.inner.fee))
        .collect();
    let replaced_sum_fee = replaced_fees
        .values()
        .try_fold(Capacity::zero(), |acc, x| acc.safe_add(*x));
    ...
}
```

Because `all_conflicted` is built by automatically extending with every descendant, a third party who submits high-fee descendants before the sender attempts RBF directly inflates `min_replace_fee` without the sender's knowledge or consent.

The analog to the external report is exact:
- **External**: `drawableFunds > 0` → `_updateAndRelease(drawableFunds)` fires automatically inside `pay`, increasing the borrower's debt.
- **CKB**: `descendants.len() > 0` → descendant fees are automatically folded into `min_replace_fee` inside `check_rbf`, increasing the sender's required replacement fee; and if `descendants.len() ≥ 100`, the replacement is blocked entirely.

---

### Impact Explanation

**Scenario A — Fee inflation**: An attacker who controls the outputs of a pending transaction (e.g., the payment recipient) submits N descendants with high fees before the sender attempts RBF. The sender's replacement transaction must now cover `fee(original) + sum(N descendant fees) + extra_rbf_fee`. The sender is forced to pay significantly more than they planned, without any consent.

**Scenario B — Complete RBF block**: The attacker submits a chain of exactly 100 descendants (B₁ → B₂ → … → B₁₀₀, each spending the previous output). `replace_count` becomes 101, exceeding `MAX_REPLACEMENT_CANDIDATES = 100`. Every subsequent RBF attempt by the sender is rejected with `"Tx conflict with too many txs"`. The sender's transaction is stuck in the pool for up to 12 hours (default `expiry_hours`), unable to be accelerated or cancelled. This is a direct economic DoS: the attacker pays only the fees for 100 minimal transactions; the sender loses the ability to manage their own transaction.

---

### Likelihood Explanation

The attack requires the attacker to control at least one output of the target transaction — the most common case being the payment recipient. This is a realistic, low-cost attack:

- The attacker pays fees for ≤ 100 transactions (each can carry the minimum fee rate).
- No special privilege, mining power, or key material is needed beyond controlling the recipient address.
- The attack is repeatable: every time the sender's transaction expires and they resubmit, the attacker can re-pin it.
- The `max_ancestors_count` default is 1,000, so the pool readily accepts a 100-deep descendant chain.

---

### Recommendation

1. **Implement a descendant-count carve-out**: Allow the sender's replacement transaction to bypass the `MAX_REPLACEMENT_CANDIDATES` limit when the replacement spends only inputs that were already in the conflicted set (i.e., no new unconfirmed inputs). This is analogous to Bitcoin's CPFP carve-out (BIP-125 Rule #5 exception).

2. **Cap per-output descendant depth**: Limit how many unconfirmed descendants can be created from a single output, reducing the attacker's ability to pin a transaction cheaply.

3. **Exclude descendant fees from `min_replace_fee` when descendants are attacker-controlled**: Only require the replacement fee to exceed the fee of the directly conflicted transaction plus `extra_rbf_fee`, not the entire descendant subtree. This mirrors the fix in the external report — removing the implicit auto-draw from the payment path.

---

### Proof of Concept

```
1. User submits Tx_A:
     inputs:  [confirmed_cell_X]
     outputs: [100 CKB → attacker_address]
     fee:     1000 shannons

2. Attacker (controls attacker_address) submits chain:
     Tx_B1: inputs=[Tx_A:output0], outputs=[99.9 CKB → attacker]
     Tx_B2: inputs=[Tx_B1:output0], outputs=[99.8 CKB → attacker]
     ...
     Tx_B100: inputs=[Tx_B99:output0], outputs=[90 CKB → attacker]
   (100 descendants, each accepted because max_ancestors_count=1000)

3. User attempts RBF with Tx_A':
     inputs:  [confirmed_cell_X]
     outputs: [99 CKB → recipient]   ← higher fee
     fee:     2000 shannons

4. check_rbf executes:
     conflict_ids = {Tx_A}
     descendants  = {Tx_B1, ..., Tx_B100}  (100 entries)
     replace_count = 100 + 1 = 101 > MAX_REPLACEMENT_CANDIDATES(100)
     → Err(RBFRejected("Tx conflict with too many txs, conflict txs count: 101, expect <= 100"))

5. Tx_A remains in pool for up to 12 hours; sender cannot replace or cancel it.
```

    fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
        let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
        // don't account for duplicate txs
        let replaced_fees: HashMap<_, _> = conflicts
            .iter()
            .map(|c| (c.id.clone(), c.inner.fee))
            .collect();
        let replaced_sum_fee = replaced_fees
            .values()
            .try_fold(Capacity::zero(), |acc, x| acc.safe_add(*x));
        let res = replaced_sum_fee.map_or(Err(CapacityError::Overflow), |sum| {
            sum.safe_add(extra_rbf_fee)
        });
        if let Ok(res) = res {
            Some(res)
        } else {
            let fees = conflicts.iter().map(|c| c.inner.fee).collect::<Vec<_>>();
            error!(
                "conflicts: {:?} replaced_sum_fee {:?} overflow by add {}",
                conflicts.iter().map(|e| e.id.clone()).collect::<Vec<_>>(),
                fees,
                extra_rbf_fee
            );
            None
        }
    }

pub struct RbfTooManyDescendants;

// RBF Rule #5
impl Spec for RbfTooManyDescendants {
    fn run(&self, nodes: &mut Vec<Node>) {
        let node0 = &nodes[0];

        node0.mine_until_out_bootstrap_period();

        // build txs chain
        let tx0 = node0.new_transaction_spend_tip_cellbase();
        let tx0_temp = tx0.clone();
        let mut txs = vec![tx0];
        let max_count = 101;
        while txs.len() <= max_count {
            let parent = txs.last().unwrap();
            let child = parent
                .as_advanced_builder()
                .set_inputs(vec![{
                    CellInput::new_builder()
                        .previous_output(OutPoint::new(parent.hash(), 0))
                        .build()
                }])
                .set_outputs(vec![parent.output(0).unwrap()])
                .build();
            txs.push(child);
        }
        assert_eq!(txs.len(), max_count + 1);
        // send tx chain
        for tx in txs[..=max_count - 1].iter() {
            let ret = node0.rpc_client().send_transaction_result(tx.data().into());
            assert!(ret.is_ok());
        }

        // Set tx2 fee to a higher value
        let output2 = CellOutputBuilder::default()
            .capacity(capacity_bytes!(70))
            .build();

        let tx2 = tx0_temp
            .as_advanced_builder()
            .set_outputs(vec![output2])
            .build();

        let res = node0
            .rpc_client()
            .send_transaction_result(tx2.data().into());
        assert!(res.is_err(), "tx2 should be rejected");
        assert!(
            res.err()
                .unwrap()
                .to_string()
                .contains("Tx conflict with too many txs")
        );

        // local submit tx RBF check failed, will not in conflicts pool
        assert_eq!(get_tx_pool_conflicts(node0), vec![tx2.hash().into()]);
    }

    fn modify_app_config(&self, config: &mut ckb_app_config::CKBAppConfig) {
        config.tx_pool.min_rbf_rate = ckb_types::core::FeeRate(1500);
    }
}
