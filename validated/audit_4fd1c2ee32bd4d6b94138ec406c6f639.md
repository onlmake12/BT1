### Title
Fee Manager Can Frontrun `setFeeManager` to Drain All Accrued Provider Fees - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.setFeeManager()` directly overwrites `provider.feeManager` in a single transaction with no timelock or two-step revocation. A fee manager who observes a pending `setFeeManager` call in the mempool can frontrun it with `withdrawAsFeeManager()` to drain the provider's entire accrued fee balance before their access is revoked. The same pattern exists in `Echo.setFeeManager()`.

---

### Finding Description

`setFeeManager` directly replaces the fee manager address atomically:

```solidity
provider.feeManager = manager;
``` [1](#0-0) 

`withdrawAsFeeManager` allows the current fee manager to drain any amount up to the full `accruedFeesInWei` balance:

```solidity
if (providerInfo.feeManager != msg.sender) {
    revert EntropyErrors.Unauthorized();
}
...
providerInfo.accruedFeesInWei -= amount;
(bool sent, ) = msg.sender.call{value: amount}("");
``` [2](#0-1) 

There is no delay, pending-state, or two-step confirmation between revoking a fee manager and the revocation taking effect. The same structural issue exists in the Echo contract: [3](#0-2) [4](#0-3) 

---

### Impact Explanation

A provider who decides to revoke or rotate their fee manager (e.g., due to a key compromise or a change in business relationship) submits `setFeeManager(newManager)`. The outgoing fee manager, monitoring the mempool, frontruns this transaction with `withdrawAsFeeManager(provider, accruedFeesInWei)`, draining the provider's entire accumulated fee balance to themselves before the revocation lands. The provider loses 100% of their accrued fees with no recourse. The attack is atomic and irreversible.

---

### Likelihood Explanation

Any registered Entropy or Echo provider that has ever assigned a fee manager faces this risk whenever they attempt to rotate or revoke that manager. On public EVM chains (Ethereum mainnet, Arbitrum, etc.) the mempool is observable. A fee manager running a simple mempool watcher can automate this attack. The scenario is realistic whenever a provider relationship sours or a fee manager key is suspected compromised — precisely the moment the provider would call `setFeeManager`.

---

### Recommendation

Implement a two-step fee manager rotation with a mandatory delay:

1. Provider calls `proposeFeeManager(newManager)` which records the pending change and a timestamp.
2. After a timelock (e.g., 24–48 hours), either the provider or the new manager calls `acceptFeeManager()` to finalize.
3. During the pending window, the outgoing fee manager retains their current rights, but the provider can also call `emergencyRevokeFeeManager()` which immediately zeroes the fee manager but simultaneously locks withdrawals for the same timelock period.

Alternatively, add a `maxWithdrawableByFeeManager` cap or require the provider to co-sign fee manager withdrawals above a threshold.

---

### Proof of Concept

```
Block N (mempool):
  Provider broadcasts: setFeeManager(newManager)
  accruedFeesInWei = 10 ETH

Block N (frontrun, higher gas):
  OldFeeManager broadcasts: withdrawAsFeeManager(provider, 10 ETH)
  → passes: providerInfo.feeManager == msg.sender ✓
  → providerInfo.accruedFeesInWei = 0
  → 10 ETH sent to OldFeeManager

Block N+1:
  setFeeManager(newManager) lands
  → provider.feeManager = newManager
  → accruedFeesInWei = 0 (already drained)

Result: Provider loses 10 ETH; new fee manager inherits an empty balance.
``` [5](#0-4) [6](#0-5)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer
        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "withdrawal to msg.sender failed");

        emit EntropyEvents.Withdrawal(provider, msg.sender, amount);
        emit EntropyEventsV2.Withdrawal(
            provider,
            msg.sender,
            amount,
            bytes("")
        );
    }

    function setFeeManager(address manager) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        address oldFeeManager = provider.feeManager;
        provider.feeManager = manager;
        emit ProviderFeeManagerUpdated(msg.sender, oldFeeManager, manager);
        emit EntropyEventsV2.ProviderFeeManagerUpdated(
            msg.sender,
            oldFeeManager,
            manager,
            bytes("")
        );
    }
