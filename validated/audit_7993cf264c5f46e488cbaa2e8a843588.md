Audit Report

## Title
Unauthenticated `from` Field in `ConnectionRequestProcess::execute()` Enables Forward Rate-Limiter Bypass and Amplified Network Flooding — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

`ConnectionRequestProcess::execute()` parses `content.from` directly from the attacker-controlled message payload and never verifies it matches the authenticated peer ID of the sending session. Because the `forward_rate_limiter` is keyed on `(content.from, content.to, msg_item_id)`, an attacker can rotate spoofed `from` values to obtain a fresh rate-limit bucket per message, bypassing the 1 req/s per-pair cap while staying within the 30 req/s session cap. Each bypassed message is then broadcast to `sqrt(N)` peers, producing amplified network flooding with a single P2P connection.

## Finding Description

In `execute()`, `content.from` is parsed from the attacker-supplied payload with no subsequent comparison to the actual session's peer ID: [1](#0-0) 

`self.peer` (a `PeerIndex` for the authenticated session) is available in the struct but is never used to validate `content.from`. The forward rate limiter is then checked against the spoofed value: [2](#0-1) 

The `forward_rate_limiter` is configured at 1 req/s per `(PeerId, PeerId, u32)` key: [3](#0-2) 

By sending each message with a distinct spoofed `from`, the attacker gets a fresh bucket every time, making the forward rate limiter ineffective. The only real constraint is the session-level limiter at 30 req/s, keyed on `(session_id, msg.item_id())`: [9](#0-8) [4](#0-3) 

When `to` is not directly connected, `forward_message()` broadcasts to `sqrt(total_peers)` peers: [5](#0-4) 

`forward_request()` only appends the relay's own ID to `route` and decrements `max_hops`, leaving `from` unchanged: [6](#0-5) 

Each receiving relay node applies its own `forward_rate_limiter` against the same spoofed `(from, to)` key — also a fresh bucket — and re-broadcasts to another `sqrt(N)` peers. The route-based loop guard (`content.route.contains(self_peer_id)`) prevents a single node from processing the same message twice, but does not prevent the fan-out amplification across distinct nodes. With `MAX_HOPS = 6`, the cascade can reach `sqrt(N)^6` total message deliveries per attacker message in the worst case.

Additionally, when the relay is the `to` target, `respond_delivered()` stores the spoofed `from` into `pending_delivered`, poisoning NAT traversal state: [7](#0-6) 

The same `forward_rate_limiter` bypass applies identically to `ConnectionRequestDelivered`: [8](#0-7) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single attacker connection can sustain 30 spoofed `ConnectionRequest` messages/second (session cap). Each message is broadcast to `sqrt(N)` peers, each of which re-broadcasts to another `sqrt(N)` peers, up to 6 hops. For a network of N=100 peers, one attacker message generates up to `10^6 = 1,000,000` message deliveries in the worst-case hop chain. At 30 messages/second, this is a severe amplification attack requiring only a standard P2P connection and no special privileges.

## Likelihood Explanation

Requires only a single standard P2P connection to any relay node — no keys, no hashpower, no majority stake. The `ConnectionRequest` message is a normal production message type. The missing check is a single equality assertion. The attack is trivially scriptable and repeatable indefinitely.

## Recommendation

After parsing `content`, resolve the actual peer ID for `self.peer` from the peer registry and assert it equals `content.from`. If they differ, return a ban-triggering status:

```rust
let actual_peer_id = self.protocol.network_state
    .with_peer_registry(|reg| reg.get_peer(self.peer).map(|p| p.peer_id.clone()));
if actual_peer_id.as_ref() != Some(&content.from) {
    return StatusCode::InvalidFromPeerId
        .with_context("from peer id does not match the actual sender");
}
```

Apply the same check to `ConnectionRequestDelivered` and `ConnectionSync`. This ensures `content.from` is always the cryptographically authenticated identity of the sending session, making the `forward_rate_limiter` key unforgeable.

## Proof of Concept

1. Attacker connects to relay node R, authenticated as peer ID `A`.
2. Attacker sends 30 `ConnectionRequest` messages/second, each with a distinct spoofed `from` value (`B1`, `B2`, ..., `B30`) and a non-connected `to` target `C`.
3. `execute()` parses `content.from = Bi`, never checks `Bi == A`.
4. `forward_rate_limiter.check_key(&(Bi, C, item_id))` passes for each `i` (fresh bucket).
5. `forward_message()` broadcasts each message to `sqrt(N)` peers.
6. Each receiving peer repeats steps 3–5 with its own fresh buckets, re-broadcasting to `sqrt(N)` more peers, up to 6 hops.
7. Network message volume grows as `30 × sqrt(N)^hop_depth` per second from a single attacker connection.
8. To verify: instrument a test network with N nodes, send 30 spoofed-`from` `ConnectionRequest` messages/second from one node, and measure total message deliveries across the network vs. the expected 30/s cap.

pub(crate) fn forward_request(
    request: packed::ConnectionRequestReader<'_>,
    current_id: &PeerId,
) -> packed::ConnectionRequest {
    let max_hops: u8 = request.max_hops().into();
    let message = request.to_entity();
    let new_route = message
        .route()
        .as_builder()
        .push(current_id.as_bytes())
        .build();
    message
        .as_builder()
        .max_hops(max_hops.saturating_sub(1))
        .route(new_route)
        .build()
}
