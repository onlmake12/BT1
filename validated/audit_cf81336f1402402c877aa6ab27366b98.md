### Title
Non-Atomic Drain-Before-Write in `save_into_file` Causes Permanent Loss of All Pending Transactions on Write Failure — (File: `tx-pool/src/persisted.rs`)

---

### Summary

`TxPool::save_into_file` truncates the on-disk persisted-data file and **drains every in-memory transaction out of the pool** before the write succeeds. If the subsequent `write_all` call fails (e.g., disk full, I/O error), all pending transactions are permanently gone from both memory and disk with no recovery path — a direct structural analog to the Solidity bug where old tokens became unrecoverable after a state change with no withdraw-by-address mechanism.

---

### Finding Description

In `tx-pool/src/persisted.rs`, `save_into_file` performs three sequential, non-atomic, destructive steps:

**Step 1 — Truncate the existing file (old persisted data destroyed):** [1](#0-0) 

**Step 2 — Drain every transaction from the in-memory pool (pool emptied):** [2](#0-1) 

**Step 3 — Write to disk (may fail after steps 1 and 2 are already irreversible):** [3](#0-2) 

`drain_all_transactions` removes all entries from the `PoolMap` (pending, gap, proposed) and clears it: [4](#0-3) 

After `drain_all_transactions` returns, the pool is empty in memory. After `truncate(true)`, the file is empty on disk. If `write_all` then fails, there is **no rollback, no backup file, and no restore path**. The error handler in `save_pool` only logs the failure: [5](#0-4) 

On the next node start, `load_from_file` finds an empty or zero-byte file and returns an empty transaction list: [6](#0-5) 

**Structural analog to the reported Solidity bug:**
- Solidity: `setTokenAddress` changed the token reference → old tokens unrecoverable because `withdraw` had no mechanism to specify the old token address.
- CKB: `save_into_file` changes pool state (truncates file + drains pool) → old transactions unrecoverable because there is no restore path after a failed write.

Both share the same root cause: a destructive state change is made irreversible before the new state is durably committed, with no recovery function.

---

### Impact Explanation

All pending, gap, and proposed transactions held in the tx-pool at shutdown time are permanently lost. Transaction submitters must resubmit their transactions after the node restarts. While the underlying cells/capacity remain live on-chain (no direct financial loss), the operational impact is significant:

- High-fee transactions pending inclusion are silently dropped.
- The node operator has no way to recover them — the pool is empty and the file is empty.
- Any downstream service relying on tx-pool persistence (e.g., fee estimators, relay state) loses its state.

**Impact: 3 / 5**

---

### Likelihood Explanation

`save_into_file` is called automatically on every graceful node shutdown via the SIGTERM/SIGINT signal handler: [7](#0-6) 

It is also callable directly by any local RPC user via the `save_pool` RPC: [8](#0-7) 

A write failure can occur due to:
- Disk full (a local attacker or co-located process can fill the disk before shutdown)
- Underlying storage I/O error
- Process interruption mid-write

The `save_pool` RPC caller and the local CLI/RPC user are both explicitly in scope as valid entry points.

**Likelihood: 2 / 5**

---

### Recommendation

Replace the non-atomic drain-then-write pattern with an atomic write-then-swap pattern:

1. Collect transactions **without draining** (iterate, do not remove from pool).
2. Write to a **temporary file** first.
3. Only after `write_all` and `sync_all` succeed, atomically rename the temp file to the final path.
4. Only then clear the in-memory pool (if shutdown is the goal).

This ensures that if the write fails, both the old persisted file and the in-memory pool remain intact and recoverable.

---

### Proof of Concept

```
1. Start a CKB node with a nearly-full disk (or fill the disk after startup).
2. Submit N transactions to the tx-pool via `send_transaction` RPC.
3. Trigger save: send SIGTERM to the node process, or call `save_pool` RPC.
4. Inside save_into_file:
   a. File opened with truncate(true) → old persisted data destroyed.
   b. drain_all_transactions() called → pool_map cleared, all N txs removed from memory.
   c. write_all(txs.as_slice()) fails → disk full error returned.
   d. save_pool logs "failed to save pool" and returns.
5. Node restarts.
6. load_from_file finds an empty/zero-byte file → returns Vec::new().
7. load_persisted_data receives 0 transactions → pool starts empty.
8. All N previously pending transactions are permanently lost.
   Users must resubmit; no recovery mechanism exists in the node.
```

    pub(crate) fn load_from_file(&self) -> Result<Vec<TransactionView>, AnyError> {
        let mut persisted_data_file = self.config.persisted_data.clone();
        persisted_data_file.set_extension(format!("v{VERSION}"));

        if persisted_data_file.exists() {
            let mut file = OpenOptions::new()
                .read(true)
                .open(&persisted_data_file)
                .map_err(|err| {
                    let errmsg = format!(
                        "Failed to open the tx-pool persisted data file [{persisted_data_file:?}], cause: {err}"
                    );
                    OtherError::new(errmsg)
                })?;
            let mut buffer = Vec::new();
            file.read_to_end(&mut buffer).map_err(|err| {
                let errmsg = format!(
                    "Failed to read the tx-pool persisted data file [{persisted_data_file:?}], cause: {err}"
                );
                OtherError::new(errmsg)
            })?;

            let persisted_data = TransactionVecReader::from_slice(&buffer)
                .map_err(|err| {
                    let errmsg = format!(
                        "The tx-pool persisted data file [{persisted_data_file:?}] is broken, cause: {err}"
                    );
                    OtherError::new(errmsg)
                })?
                .to_entity();

            Ok(persisted_data
                .into_iter()
                .map(|tx| tx.into_view())
                .collect())
        } else {
            Ok(Vec::new())
        }
    }

    pub(crate) fn drain_all_transactions(&mut self) -> Vec<TransactionView> {
        let mut txs = TxSelector::new(&self.pool_map)
            .txs_to_commit(usize::MAX, Cycle::MAX)
            .0
            .into_iter()
            .map(|tx_entry| tx_entry.into_transaction())
            .collect::<Vec<_>>();
        let mut pending = self
            .pool_map
            .entries
            .remove_by_status(&Status::Pending)
            .into_iter()
            .map(|e| e.inner.into_transaction())
            .collect::<Vec<_>>();
        txs.append(&mut pending);
        let mut gap = self
            .pool_map
            .entries
            .remove_by_status(&Status::Gap)
            .into_iter()
            .map(|e| e.inner.into_transaction())
            .collect::<Vec<_>>();
        txs.append(&mut gap);
        self.pool_map.clear();
        txs
    }
