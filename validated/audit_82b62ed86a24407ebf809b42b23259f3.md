### Title
`max_tx_verify_cycles` Limit Not Enforced for Cached Transactions on Reorg Re-admission — (`tx-pool/src/util.rs`)

---

### Summary

The `verify_rtx` function in the CKB tx-pool has two code paths: one for cache hits and one for cache misses. The `max_tx_verify_cycles` cycle limit is enforced only in the cache-miss path. When a cached `Completed` entry exists, the limit is silently ignored and the stored cycle count is returned directly. This allows transactions with cycles exceeding `max_tx_verify_cycles` to be re-admitted to the pool after a reorg, bypassing the DoS protection the limit is designed to provide.

---

### Finding Description

The tx-pool enforces a per-transaction cycle limit (`max_tx_verify_cycles`) to prevent high-cost transactions from consuming excessive verification resources. However, this limit is only applied when a transaction is freshly verified. When a cached `Completed` entry exists, the limit is never checked.

The `verify_rtx` function in `tx-pool/src/util.rs` has the following structure:

```rust
pub(crate) async fn verify_rtx(
    ...
    cache_entry: &Option<CacheEntry>,
    max_tx_verify_cycles: Cycle,
    ...
) -> Result<Completed, Reject> {
    if let Some(completed) = cache_entry {
        // Cache hit: max_tx_verify_cycles is NEVER checked
        TimeRelativeTransactionVerifier::new(rtx, consensus, data_loader, tx_env)
            .verify()
            .map(|_| *completed)   // cached cycles returned unconditionally
            .map_err(Reject::Verification)
    } else if let Some(command_rx) = command_rx {
        ContextualTransactionVerifier::new(...)
            .verify_with_pause(max_tx_verify_cycles, command_rx)  // limit enforced
            ...
    } else {
        ContextualTransactionVerifier::new(...)
            .verify(max_tx_verify_cycles, false)  // limit enforced
            ...
    }
}
```

The cache-miss paths enforce `max_tx_verify_cycles` via `ContextualTransactionVerifier::verify(max_tx_verify_cycles, ...)`. The cache-hit path ignores it entirely.

The divergence between the two cycle limits used by callers makes this exploitable:

- **`_process_tx`** (the main submission path) uses `max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles())`. For local RPC-submitted transactions where `declared_cycles = None`, this resolves to `max_block_cycles` — a consensus constant orders of magnitude larger than `max_tx_verify_cycles` (default ~66M vs ~3.5B).
- **`readd_detached_tx`** (the reorg re-admission path) uses `max_cycles = self.tx_pool_config.max_tx_verify_cycles` — the intended DoS limit.

The attack flow:

1. An RPC caller submits a transaction with cycles C where `max_tx_verify_cycles < C < max_block_cycles`.
2. `_process_tx` verifies it with `max_cycles = max_block_cycles`; the tx passes and is cached as `Completed { cycles: C }`.
3. The tx is included in a block.
4. A natural reorg detaches the block; `update_tx_pool_for_reorg` → `readd_detached_tx` is called.
5. `readd_detached_tx` sets `max_cycles = max_tx_verify_cycles` and calls `verify_rtx` with the cached entry.
6. The cache-hit branch returns the stored `C` cycles without checking against `max_tx_verify_cycles`.
7. The tx is re-admitted to the pool with `C > max_tx_verify_cycles`, bypassing the DoS protection. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Medium-High Impact.** The `max_tx_verify_cycles` parameter is explicitly documented as a DoS mitigation ("to reduce DDOS vulnerability", per CHANGELOG). Bypassing it allows transactions with arbitrarily high cycle counts (up to `max_block_cycles`) to persist in the pool after a reorg. Such transactions:

- Occupy pool capacity with entries that should have been rejected under the current limit.
- Can be selected by the block assembler, causing the miner's block template to include a single transaction consuming a disproportionate share of the block cycle budget.
- Undermine the operator's ability to tune `max_tx_verify_cycles` for resource management, since the limit is silently bypassed for any tx that was previously cached. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

**High Likelihood.** The conditions required are:

1. A transaction with cycles between `max_tx_verify_cycles` (~66M default) and `max_block_cycles` (~3.5B) is submitted via RPC — this is a normal, valid transaction.
2. A reorg occurs — reorgs are a routine part of PoW blockchain operation and require no attacker action.
3. The tx is in the `txs_verify_cache` at the time of reorg — the cache holds up to 30,000 entries and is populated on every successful tx verification.

No privileged access, no config change, and no majority hashpower is required. Any RPC caller can craft a high-cycle transaction and wait for a natural reorg. [6](#0-5) [7](#0-6) 

---

### Recommendation

In the cache-hit branch of `verify_rtx`, add an explicit check that the cached cycle count does not exceed the `max_tx_verify_cycles` limit passed by the caller:

```rust
if let Some(completed) = cache_entry {
    // Add: reject if cached cycles exceed the current limit
    if completed.cycles > max_tx_verify_cycles {
        return Err(Reject::DeclaredWrongCycles(max_tx_verify_cycles, completed.cycles));
    }
    TimeRelativeTransactionVerifier::new(rtx, consensus, data_loader, tx_env)
        .verify()
        .map(|_| *completed)
        .map_err(Reject::Verification)
}
```

Additionally, consider whether `_process_tx` should use `max_tx_verify_cycles` rather than `max_block_cycles` as the default limit for locally submitted transactions, so that the pool consistently enforces the configured DoS limit regardless of the code path taken. [5](#0-4) [8](#0-7) 

---

### Proof of Concept

1. Configure a CKB node with `max_tx_verify_cycles = 66_000_000` (default).

2. Craft a transaction whose script execution consumes ~200M cycles (well above `max_tx_verify_cycles` but below `max_block_cycles`).

3. Submit it via `send_transaction` RPC. It is accepted because `_process_tx` uses `max_block_cycles` (~3.5B) as the limit, not `max_tx_verify_cycles`. The tx is cached with `cycles = 200_000_000`.

4. Mine a block that includes the transaction.

5. Trigger a reorg (e.g., by connecting a competing chain of equal or greater work). The block is detached; `update_tx_pool_for_reorg` → `readd_detached_tx` is called.

6. Inside `readd_detached_tx`, `max_cycles = self.tx_pool_config.max_tx_verify_cycles = 66_000_000`. `verify_rtx` is called with the cached entry.

7. The cache-hit branch returns `Completed { cycles: 200_000_000 }` without comparing against `66_000_000`.

8. The transaction is re-admitted to the pool. Querying `get_raw_tx_pool` confirms the tx is present with `cycles = 200_000_000 > max_tx_verify_cycles`. [3](#0-2) [1](#0-0)

        for tx in txs {

            {

                let snapshot = tx_pool.cloned_snapshot();

                if let Ok(verified) = verify_rtx(
                    snapshot,
                    Arc::clone(&rtx),
                    tx_env,
                    &verify_cache,
                    max_cycles,
                    None,

                    let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
                    if let Err(e) = _submit_entry(tx_pool, status, entry, &self.callbacks) {
                        error!("readd_detached_tx submit_entry {} error {}", tx_hash, e);
                    } else {
                        debug!("readd_detached_tx submit_entry {}", tx_hash);
                    }
                }
            }
        }
    }
