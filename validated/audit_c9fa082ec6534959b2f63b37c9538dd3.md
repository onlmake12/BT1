### Title
`total_tx_size`/`total_tx_cycles` Inflated by Stale Snapshot Overwrite After Ancestor-Eviction in `add_entry` — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the pool's aggregate accounting variables `total_tx_size` and `total_tx_cycles` are snapshotted **before** `check_and_record_ancestors` runs. When that function evicts existing pool entries (via `remove_entry_and_descendants`) to satisfy the ancestor-count limit, each eviction correctly decrements `self.total_tx_size`/`self.total_tx_cycles` through `update_stat_for_remove_tx`. However, the final two lines of `add_entry` then **overwrite** those correctly-decremented values with the stale pre-eviction snapshot, re-inflating the totals by the combined size/cycles of every evicted entry. The inflated `total_tx_size` is subsequently used by `limit_size` to decide whether to evict further transactions, causing legitimate pool entries to be dropped even when the pool is not actually full.

---

### Finding Description

`PoolMap::add_entry` follows this sequence:

```
// Step 1 – snapshot new totals BEFORE any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211

// Step 2 – may evict entries; each eviction calls update_stat_for_remove_tx,
//           which correctly DECREMENTS self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

// ... insert new entry ...

// Step 3 – OVERWRITES the post-eviction value with the stale pre-eviction snapshot
self.total_tx_size  = total_tx_size;                            // line 218
self.total_tx_cycles = total_tx_cycles;                         // line 219
```

`updated_stat_for_add_tx` (lines 711–728) reads `self.total_tx_size` at call time and returns `self.total_tx_size + new_entry.size`. It does **not** modify `self.total_tx_size`.

`check_and_record_ancestors` (lines 588–640) enters its eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants` (line 618), which calls `remove_entry` → `update_stat_for_remove_tx` (line 247), which correctly subtracts each evicted entry's size and cycles from `self.total_tx_size` / `self.total_tx_cycles`.

After evictions, `self.total_tx_size` equals `T_original − Σ(evicted_sizes)`. The final assignment at line 218 then sets it back to `T_original + new_entry.size`, discarding the eviction credits. The net inflation is `Σ(evicted_sizes)`.

`cell_ref_parents` are pool transactions that hold a cell dep pointing to the same output that the incoming transaction spends as an input. This is a normal, reachable condition whenever a popular script cell is being consumed.

---

### Impact Explanation

`limit_size` (pool.rs lines 292–329) loops while `self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts the lowest-fee-rate entries. With `total_tx_size` inflated by `Σ(evicted_sizes)`, the loop evicts that many additional bytes of legitimate, fee-paying transactions that would otherwise have remained in the pool. The evicted transactions receive a `Reject::Full` callback and are broadcast as rejected to peers.

Concretely:
- **Legitimate transactions are silently dropped** from the mempool even when actual pool occupancy is below `max_tx_pool_size`.
- **Fee-rate ordering is distorted**: the lowest-fee-rate transactions are evicted first, so an attacker can selectively displace competing transactions by engineering the inflation.
- **The inflation persists** until enough subsequent removals bring `total_tx_size` back to the true value; there is no self-correcting path short of a full pool clear or node restart.

---

### Likelihood Explanation

The trigger requires:
1. Enough existing pool transactions that share a cell dep with one of the attacker's input cells to form `cell_ref_parents`.
2. The attacker's new transaction has an ancestor chain long enough to exceed `max_ancestors_count` (default 25 in `ckb.toml`, 1 000 in the legacy default) when `cell_ref_parents` are counted.

Both conditions are achievable by any unprivileged RPC or P2P transaction submitter without any special privilege. The attacker first floods the pool with transactions that reference a chosen cell dep, then submits a transaction spending that cell with a long ancestor chain. No key material, majority hashpower, or operator access is required.

---

### Recommendation

Move the stat snapshot **after** `check_and_record_ancestors` completes, so it reflects the post-eviction pool state:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, Default::default()));
    }
    // Validate that adding this entry won't overflow totals, but do NOT
    // commit the snapshot yet.
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    let evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Compute and commit totals AFTER all evictions have already
    // decremented self.total_tx_size / self.total_tx_cycles.
    self.total_tx_size  = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

Alternatively, accumulate the evicted sizes inside `check_and_record_ancestors` and subtract them from the pre-computed snapshot before the final assignment.

### Proof of Concept

**Setup** (pool with `max_ancestors_count = 25`, `max_tx_pool_size = 1 000 000` bytes):

1. Submit 26 transactions `A₁ → A₂ → … → A₂₆` (chain), each 1 000 bytes, all using cell `C` as a cell dep. Pool size = 26 000 bytes. `cell_ref_parents` for any tx spending `C` = {A₁…A₂₆}.

2. Submit transaction `B` that **spends cell `C`** as an input and has 0 ancestors of its own. `ancestors_count = 1 + 26 = 27 > 25`. `cell_ref_parents.len() = 26`, so `27 − 26 = 1 ≤ 25` → eviction branch entered.

3. `check_and_record_ancestors` evicts 2 entries (e.g., A₁ and A₂, 2 000 bytes total) to bring `ancestors_count` to 25. `update_stat_for_remove_tx` is called twice: `self.total_tx_size` drops from 26 000 to 24 000.

4. Back in `add_entry`, `total_tx_size` (the stale snapshot) = 26 000 + size(B) = 27 000. Line 218 sets `self.total_tx_size = 27 000` instead of the correct 24 000 + size(B) = 25 000.

5. Inflation = 2 000 bytes (the evicted entries). `limit_size` now sees `27 000 > max_tx_pool_size`? If `max_tx_pool_size` were set to 26 500 for this test, it would evict 500+ bytes of legitimate transactions that should have stayed.

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );

                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
