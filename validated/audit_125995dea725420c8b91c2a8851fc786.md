### Title
Self-Wait Deadlock in `process_message_box` Escalates to `Error::Unexpected` Instead of `WAIT_FAILURE` ‚Äî (`script/src/scheduler.rs`)

### Summary

A script running under `ScriptVersion::V2` can call `ckb_wait` with `A0` set to its own `VmId`. The scheduler's `process_message_box` lacks a self-wait guard, so it transitions the only runnable VM into `VmState::Wait { target_vm_id: self }`. On the next iteration, `iterate_prepare_machine` finds no runnable VM and returns `Error::Unexpected("A deadlock situation has been reached!")` ‚Äî an internal error that should never be reachable via script-controlled input.

---

### Finding Description

**Step 1 ‚Äî Attacker entrypoint:** An unprivileged script author deploys a lock/type script that executes `ckb_wait(own_vm_id, addr)` (sets `A7=WAIT`, `A0=own_pid`, `A1=valid_addr`). This is a standard CKB-VM script execution path, reachable via transaction submission.

**Step 2 ‚Äî `Wait::ecall`** (`script/src/syscalls/wait.rs:33-51`):

`target_id` is read directly from `A0` with no validation. A `Message::Wait(self.id, WaitArgs { target_id: self.id, ... })` is pushed to the message box, then `Err(VMError::Yield)` is returned. [1](#0-0) 

**Step 3 ‚Äî `process_message_box`** (`script/src/scheduler.rs:564-592`):

The handler has two guards before inserting `VmState::Wait`:

- **Guard 1** (line 565): Is `args.target_id` in `terminated_vms`? ‚Üí **No** (the VM just yielded, it is not terminated).
- **Guard 2** (line 578): Is `args.target_id` absent from `self.states`? ‚Üí **No** ‚Äî the calling VM is still present in `self.states` as `VmState::Runnable` at this point. [2](#0-1) 

Both guards pass without triggering. The handler falls through to the unconditional insert:

```rust
self.states.insert(vm_id, VmState::Wait { target_vm_id: args.target_id, ... });
```

where `vm_id == args.target_id`. The VM is now waiting for itself.

**Step 4 ‚Äî Back in `iterate_process_results`** (line 408): `Err(Error::Yield)` is silently swallowed as `Ok(())`. No error is surfaced yet. [3](#0-2) 

**Step 5 ‚Äî Next call to `iterate_prepare_machine`** (lines 335-346):

The scheduler scans `self.states` for any `VmState::Runnable` entry. The only VM is now in `VmState::Wait`. No runnable VM is found, so:

```rust
Error::Unexpected("A deadlock situation has been reached!".to_string())
```

is returned. [4](#0-3) 

This propagates up through `iterate_inner` ‚Üí `iterate_outer` ‚Üí `run`, surfacing as an internal unexpected error from script verification ‚Äî instead of the expected `WAIT_FAILURE` (error code 5).

---

### Impact Explanation

The invariant violated is: **script-controlled inputs must never escalate to `Error::Unexpected`**. `Error::Unexpected` is reserved for internal logic errors (e.g., corrupted state, programming bugs). Returning it from a path reachable by any unprivileged script author means:

- Script verification produces a wrong error type for a self-wait, which is incorrect CKB-VM behavior.
- Any upstream code that distinguishes `Error::Unexpected` from normal script failure (e.g., for logging, error reporting, or future error-code-based branching) will misclassify this as an internal node fault rather than a script-level rejection.

The node does not panic (no `unwrap`/`panic!` is hit), but the error semantics are wrong and the invariant is broken.

---

### Likelihood Explanation

Any script author can trigger this with a single instruction sequence. No special privileges, no PoW, no key material, no majority hashpower required. The attack surface is the standard transaction submission path.

---

### Recommendation

Add a self-wait guard in `process_message_box` before the `VmState::Wait` insert:

```rust
// In the Message::Wait branch, after the two existing guards:
if args.target_id == vm_id {
    let (_, machine) = self.ensure_get_instantiated(&vm_id)?;
    machine.inner_mut().set_register(A0, Self::u8_to_reg(WAIT_FAILURE));
    continue;
}
``` [5](#0-4)

### Proof of Concept

Unit test sketch:
1. Create a `ScriptVersion::V2` scheduler with a single root VM.
2. Set `A7=WAIT`, `A0=ROOT_VM_ID` (own ID), `A1=valid_writable_addr`.
3. Run the scheduler.
4. Assert the result is `WAIT_FAILURE` (error code 5), **not** `Error::Unexpected("A deadlock situation has been reached!")`.

Under the current code, step 4 will fail ‚Äî the scheduler returns `Error::Unexpected` instead of `WAIT_FAILURE`.
