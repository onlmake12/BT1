Audit Report

## Title
`GetBlocksProcess` Missing IBD State Check Contradicts Documented Design Contract - (File: sync/src/synchronizer/get_blocks_process.rs)

## Summary
`sync/src/synchronizer/mod.rs` L4 explicitly documents that an IBD-mode node must respond with `InIBD` to both `GetHeaders` and `GetBlocks` requests. `GetHeadersProcess::execute()` enforces this correctly, but `GetBlocksProcess::execute()` contains no IBD check, allowing any peer to force the node to perform RocksDB lookups and serve full block data during IBD, directly violating the stated design contract.

## Finding Description
The design contract is stated at `sync/src/synchronizer/mod.rs` L4: [1](#0-0) 

`GetHeadersProcess::execute()` enforces this at `sync/src/synchronizer/get_headers_process.rs` L53–65: [2](#0-1) 

`GetBlocksProcess::execute()` has no such check. It immediately proceeds to process block hashes, performing `contains_block_status` RocksDB lookups and `get_block` retrievals, and spawning async send tasks: [3](#0-2) [4](#0-3) 

The dispatch at `sync/src/synchronizer/mod.rs` L407–411 applies no top-level IBD guard before calling `GetBlocksProcess`: [5](#0-4) 

Exploit path: any peer opens a Sync protocol connection and sends a `GetBlocks` message with up to `MAX_HEADERS_LEN` (2000) hashes. The node processes up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32) per call — performing `contains_block_status` RocksDB lookups for each, `get_block` retrievals for validated blocks, and spawning async tasks to send full block data — all while its own IBD pipeline competes for the same disk I/O and CPU resources. The `BLOCK_VALID` status check at L60 does not prevent the RocksDB lookup itself; it only gates the subsequent block retrieval and send.

## Impact Explanation
**Low (501–2000 points): Any other important performance improvement for CKB.** The missing check violates the explicitly documented IBD design contract and forces an IBD node to perform unnecessary disk I/O (RocksDB status lookups and block retrievals) and spawn async send tasks in response to attacker-controlled messages. This degrades IBD throughput. It does not crash the node, cause consensus deviation, or damage the CKB economy.

## Likelihood Explanation
Any peer that opens a Sync protocol connection can trigger this. No special privilege, key, or hashpower is required. Block hashes are public on-chain data. The attack is repeatable at negligible cost and requires only a TCP connection. During IBD, the node accepts inbound connections and processes Sync messages normally.

## Recommendation
Add an IBD check at the start of `GetBlocksProcess::execute()`, mirroring `GetHeadersProcess`. When `active_chain.is_initial_block_download()` returns `true`, send an `InIBD` response using a `send_in_ibd()` helper (identical to the one in `GetHeadersProcess` at `get_headers_process.rs` L101–115) and return `Status::ignored()`. [6](#0-5) 

## Proof of Concept
1. Start a CKB node from genesis; confirm `is_initial_block_download()` returns `true` (tip timestamp is far behind wall clock).
2. Connect a custom peer that opens the Sync protocol.
3. Send a `GetBlocks` message containing the hash of block #1 (publicly known).
4. Observe the node responds with a `SendBlock` message containing full block data instead of `InIBD`.
5. Send a `GetHeaders` message to the same node; observe it correctly returns `InIBD` and logs: *"Ignoring getheaders from peer=X because the node is in initial block download stage."*
6. The asymmetry between steps 4 and 5 confirms the missing guard.
