### Title
Unbounded `tx.gasprice` in Keeper Fee Calculation Allows Subscription Balance Drain — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `_processFeesAndPayKeeper` function in `Scheduler.sol` calculates the keeper's reimbursement using `tx.gasprice` with no upper bound. Because `updatePriceFeeds` is a permissionless function callable by any address, a malicious keeper can submit the transaction with an arbitrarily inflated gas price and drain the subscription's entire ETH balance in a single call.

### Finding Description
In `_processFeesAndPayKeeper`, the gas cost component of the keeper fee is computed as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
```

`tx.gasprice` is fully controlled by the transaction sender (the keeper). There is no maximum gas price cap enforced anywhere in the contract — neither in `SubscriptionParams`, `SchedulerConstants`, nor in `_processFeesAndPayKeeper` itself. The only guard is:

```solidity
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
```

This check does not protect the subscription manager; it merely prevents the keeper from extracting *more* than the balance. A keeper can set `tx.gasprice` high enough so that `totalKeeperFee` equals the entire subscription balance, transferring it all to themselves in one transaction.

The keeper network is explicitly permissionless per the README: *"Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`."* [1](#0-0) [2](#0-1) 

### Impact Explanation
A malicious keeper who satisfies the update trigger conditions (heartbeat elapsed or price deviation threshold crossed) can call `updatePriceFeeds` with `tx.gasprice` set to `subscriptionBalance / (gasUsed + GAS_OVERHEAD)`. The contract will compute `totalKeeperFee ≈ subscriptionBalance`, pass the balance check, deduct the full balance, and transfer it to the keeper. The subscription manager loses their entire deposited ETH. This is a direct theft of user funds with no recovery path. [3](#0-2) 

### Likelihood Explanation
The attack requires only that the keeper wait for a legitimate update trigger (heartbeat or deviation) to be satisfied — a condition that occurs regularly by design. No privileged access, leaked key, or governance majority is needed. The keeper network is permissionless, so any attacker can register as a keeper. On EVM chains, a transaction sender freely sets `tx.gasprice` (or `maxFeePerGas` on EIP-1559 chains). The attack is straightforward and economically rational for any subscription holding significant ETH. [4](#0-3) 

### Recommendation
Enforce a maximum gas price cap in the subscription parameters or as a global admin-configurable constant. For example:

1. Add a `maxGasPriceInWei` field to `SubscriptionParams` that the manager sets at subscription creation time.
2. In `_processFeesAndPayKeeper`, cap the effective gas price: `uint256 effectiveGasPrice = tx.gasprice > params.maxGasPriceInWei ? params.maxGasPriceInWei : tx.gasprice;`
3. Alternatively, add a global `maxGasPriceInWei` admin parameter and apply it as a ceiling.

This mirrors the standard slippage-protection pattern: the user (subscription manager) specifies the worst-case price they are willing to pay, and the contract enforces it. [5](#0-4) 

### Proof of Concept

1. Subscription manager calls `createSubscription` and deposits 1 ETH.
2. Attacker deploys a keeper contract and waits for the heartbeat interval to elapse (e.g., 60 seconds).
3. Attacker calls `updatePriceFeeds(subscriptionId, validUpdateData)` with `tx.gasprice = 1 ETH / (estimatedGasUsed + GAS_OVERHEAD)`.
4. Inside `_processFeesAndPayKeeper`:
   - `gasCost = (gasUsed + GAS_OVERHEAD) * tx.gasprice ≈ 1 ETH`
   - `totalKeeperFee ≈ 1 ETH`
   - `status.balanceInWei (1 ETH) >= totalKeeperFee (1 ETH)` → check passes
   - `msg.sender.call{value: 1 ETH}("")` → attacker receives the full subscription balance
5. Subscription balance is now 0; the manager's funds are stolen. [6](#0-5)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }

- Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`. The main goal of making this component a permissionless network rather a set of permissioned nodes is to enhance reliability for the feeds -- if one provider fails, others should be available to service the subscriptions. We can improve this reliability by sourcing independent providers, and by making it profitable to push updates, paid out by the users of the feeds.

- Keepers are paid directly by the subscription's funds held in this contract for each successful update they perform. The payment covers gas costs plus a premium, and payment is sent directly to `msg.sender` (the keeper) at the end of `updatePriceFeeds`. The first transaction included in a block that passes checks will succeed and receive the payment. Subsequent attempts for the same update interval will revert since we verify the update criteria on-chain. By only allowing updates when they are needed, we keep costs predictable for the users.
