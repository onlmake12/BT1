Audit Report

## Title
Unbounded `pending_delivered` HashMap Growth and Uncapped NAT Traversal Task Spawning via Spoofed `from` PeerIds — (`network/src/protocols/hole_punching/`)

## Summary
An unprivileged remote peer with a single valid P2P connection can flood the victim's `pending_delivered` HashMap by sending `ConnectionRequest` messages with unique, attacker-generated `from` PeerIds. Because no size cap exists on `pending_delivered` and cleanup only runs every 5 minutes, the HashMap accumulates up to 9,000 entries per attacker session per window. A follow-up stream of `ConnectionSync` messages — one per accumulated `from_i` — then causes the victim to call `runtime::spawn` for each entry, launching long-lived NAT traversal tasks that each loop for up to 30 seconds making repeated TCP connection attempts, leading to async-task and file-descriptor exhaustion that can crash the node.

## Finding Description

**Unbounded `pending_delivered` insertion**

`pending_delivered` is declared as a plain `HashMap` with no size bound: [1](#0-0) 

In `connection_request.rs`, `respond_delivered()` only guards against re-insertion of the **same** `from` PeerId within 2 minutes (`HOLE_PUNCHING_INTERVAL`): [2](#0-1) 

For N **distinct** `from_i` PeerIds, this check always passes and each is unconditionally inserted: [3](#0-2) 

**Rate limiters do not bound aggregate HashMap size**

The session-level `rate_limiter` is keyed by `(session_id, msg.item_id())` and allows 30 req/sec per session: [4](#0-3) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1 req/sec per key: [5](#0-4) 

With N unique `from_i` values, each gets its own independent bucket. The session limiter caps total throughput at 30/sec, so the attacker inserts 30 new entries/sec. Over the 5-minute cleanup window (`CHECK_INTERVAL`), this yields **9,000 entries** before the first eviction: [6](#0-5) 

Cleanup only runs in `notify()` every 5 minutes: [7](#0-6) 

Each entry stores up to `ADDRS_COUNT_LIMIT = 24` `Multiaddr` values: [8](#0-7) 

**Uncapped `runtime::spawn` via `ConnectionSync`**

When the victim receives a `ConnectionSync` with `from = from_i` present in `pending_delivered`, it unconditionally calls `runtime::spawn` with no cap on concurrent tasks: [9](#0-8) [10](#0-9) 

The `forward_rate_limiter` for `ConnectionSync` is also keyed by `(from, to, msg_item_id)`, so N unique `from_i` values each get their own bucket, providing no aggregate protection.

Each spawned task runs `try_nat_traversal`, which loops for up to **30 seconds** making TCP connection attempts every ~200ms: [11](#0-10) 

At 30 `ConnectionSync` messages/sec, after 30 seconds there are **900 concurrent NAT traversal tasks** at steady state, each holding a `TcpSocket` and making ~5 TCP connection attempts/sec.

**`from` field is unauthenticated**

The `from` field is parsed directly from the message body with no validation against the actual session's peer identity: [12](#0-11) 

Generating N unique `PeerId` values is trivial (random 32-byte keys).

## Impact Explanation

This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

- **Memory exhaustion**: 9,000 entries × up to 24 `Multiaddr` values (~1–2 KB/entry) = ~10–18 MB per attacker connection per 5-minute window. Multiple attacker connections multiply this linearly.
- **Async task / file-descriptor exhaustion**: 900 concurrent `runtime::spawn` tasks at steady state, each holding a `TcpSocket` and looping for 30 seconds. This exhausts the Tokio thread pool, OS file descriptors, and outbound port space, causing the victim node to become unresponsive to legitimate peers and ultimately crash or be killed by the OS.

## Likelihood Explanation

The attacker requires only a single valid P2P connection to the victim — a standard precondition for any peer in the CKB network. The `from` field is a free-form message body field with no authentication. Generating unique `PeerId` values is trivial. The attack is fully automatable, requires no special privileges, and is reproducible in a local test environment with a single malicious peer.

## Recommendation

1. **Cap `pending_delivered` size**: Enforce a maximum entry count (e.g., 256 or 1024). Reject new insertions when the cap is reached, or evict the oldest entry (LRU).
2. **Rate-limit NAT traversal task spawning globally**: Track the total number of live NAT traversal tasks (e.g., via an `Arc<AtomicUsize>`) and refuse to spawn new ones beyond a threshold (e.g., 16–32 concurrent tasks).
3. **Authenticate `from` PeerId**: Validate that the `from` field in `ConnectionRequest` matches the actual session's authenticated peer identity, preventing spoofed `from` values from consuming victim-side state.
4. **Reduce `CHECK_INTERVAL`**: Reduce the cleanup interval from 5 minutes to 30–60 seconds to limit peak HashMap size.

## Proof of Concept

```rust
// Phase 1: Insert 9,000 unique `from` entries into pending_delivered
// by sending ConnectionRequest messages with unique from_i PeerIds
// at 30/sec (session rate limit) over 300 seconds (5-minute window).
//
// Each insertion passes the HOLE_PUNCHING_INTERVAL check because
// no existing entry exists for the new from_i.
//
// Demonstrates unbounded growth:
let mut pending_delivered: HashMap<PeerId, (Vec<Multiaddr>, u64)> = HashMap::new();
let now = unix_time_as_millis();
for _ in 0..9_000 {
    let from = PeerId::random(); // unique each iteration
    pending_delivered.insert(from, (vec![dummy_multiaddr()], now));
}
assert_eq!(pending_delivered.len(), 9_000); // no cap exists

// Phase 2: Send ConnectionSync with each from_i → triggers runtime::spawn
// for each entry → 900 concurrent NAT traversal tasks at steady state
// (30/sec × 30-second task lifetime), each holding a TcpSocket and
// looping for 30 seconds → file-descriptor and thread-pool exhaustion.
```

Minimal manual reproduction: connect a single peer to the victim, send `ConnectionRequest` messages in a loop with `to = victim_peer_id` and a fresh random `from` each time at 30/sec, then send matching `ConnectionSync` messages. Monitor victim's memory usage and open file descriptors to observe unbounded growth and eventual crash.
