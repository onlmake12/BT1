Audit Report

## Title
TOCTOU Race Between `OrphanBroker` and `ConsumeUnverifiedBlockProcessor` Causes Preload Thread Panic and Permanent Block-Processing Halt — (`chain/src/preload_unverified_blocks_channel.rs`, `chain/src/orphan_broker.rs`)

## Summary

A time-of-check/time-of-use race between the `ChainService` thread and the `verify_blocks` thread allows child block C to be dispatched to the preload thread after parent P's `COLUMN_BLOCK_HEADER` entry has been deleted by a concurrent failed verification. The preload thread calls `.expect("parent header stored")` on the missing header, panics unconditionally, and because no `catch_unwind` guards the preload loop, the panic propagates: the `unverified_block_tx` sender is dropped, the verify thread's receiver returns `Err` and exits, permanently halting block processing on the node until restart.

## Finding Description

**TOCTOU window.** In `process_lonely_block`, the `ChainService` thread reads `is_pending_verify` and then unconditionally forwards C to the preload channel with no lock held across the two operations: [1](#0-0) 

`process_descendant` inserts C into `is_pending_verify` and sends it to the preload channel: [2](#0-1) 

**Concurrent failure path.** When P fails contextual verification on the `verify_blocks` thread, `consume_unverified_blocks` calls `self.delete_unverified_block(&block)` which commits a RocksDB transaction deleting `COLUMN_BLOCK_HEADER` for P's hash, then calls `self.is_pending_verify.remove(&block_hash)`: [3](#0-2) 

`delete_unverified_block` calls `db_txn.delete_block(&block)` which explicitly removes `COLUMN_BLOCK_HEADER`: [4](#0-3) [5](#0-4) 

**Panic site.** The preload thread calls `load_full_unverified_block_by_hash(C)`, which reads P's header from the same RocksDB column via `get_block_header`. `get_block_header` checks an LRU cache first, but cache eviction is not guaranteed — if P's header is not cached, the DB read returns `None` and the unconditional `.expect` panics: [6](#0-5) [7](#0-6) 

**No `catch_unwind` in preload loop.** The preload thread's `start()` loop calls `preload_unverified_channel` directly with no panic guard, unlike the `verify_blocks` thread which wraps `consume_unverified_blocks` in `catch_unwind`: [8](#0-7) [9](#0-8) 

**Cascade.** When the preload thread panics, `PreloadUnverifiedBlocksChannel` is dropped, which drops `unverified_block_tx`. The verify thread's `select!` loop receives `Err` on `unverified_block_rx` and returns, terminating the verify thread: [10](#0-9) 

The channel wiring is confirmed in `init.rs`: `unverified_block_tx` is owned exclusively by the preload thread, so its drop immediately disconnects the verify thread's receiver: [11](#0-10) 

## Impact Explanation

The node permanently stops processing new blocks. P2P connections remain active and RPC responds, but the chain tip is frozen. This is a complete block-processing denial-of-service requiring a node restart to recover. This matches the **High** impact class: "Vulnerabilities which could easily crash a CKB node."

## Likelihood Explanation

The race window is wide: contextual verification of P (CKB-VM script execution, transaction resolution) can take hundreds of milliseconds to seconds for a block with complex scripts, while the `ChainService` thread dispatches C to the preload channel in microseconds. An attacker who mines a single PoW-valid block P containing a contextually invalid transaction (e.g., a script that always aborts, a double-spend, or an invalid capacity) and immediately sends child C can reliably hit this window. Mining one block does not require majority hashpower and the attack is repeatable. The LRU header cache (`store/src/store.rs:74-78`) provides no reliable protection since cache eviction is non-deterministic and `delete_block` does not invalidate the cache entry, meaning the race is only exploitable on a cache miss — but this is achievable under memory pressure or with sufficient block volume.

## Recommendation

1. **Add `catch_unwind` in the preload thread loop** mirroring the pattern already used in `verify_blocks` (`verify.rs:86-96`), so a panic on one task does not kill the thread.
2. **Replace `.expect("parent header stored")` with graceful error handling** — if the parent header is absent, log an error and drop the task; C will be discarded as invalid once P's `BLOCK_INVALID` status propagates.
3. **Re-check parent validity immediately before dispatch** — after the `is_pending_verify.contains` check, re-read `get_block_status(&parent_hash)` inside a short critical section or re-check after acquiring the `is_pending_verify` shard lock, so a concurrently-failed parent is detected before C is forwarded to the preload channel.

## Proof of Concept

```
1. Attacker mines block P (valid PoW, contextually invalid — e.g., script always aborts).
2. Attacker sends P to victim node via P2P SendBlock.
   → P passes non-contextual checks, stored in DB, inserted into is_pending_verify,
     sent to preload channel, then to verify channel.
3. Attacker immediately sends child C (parent_hash = P.hash) via P2P SendBlock.
   → C passes non-contextual checks, stored in DB.
   → OrphanBroker.process_lonely_block(C):
       is_pending_verify.contains(P.hash) == true  [orphan_broker.rs:111]
       → process_descendant(C) → C sent to preload channel  [orphan_broker.rs:118]
4. Concurrently, verify_blocks thread finishes verifying P:
   → P fails → delete_unverified_block(P) commits
     → COLUMN_BLOCK_HEADER[P.hash] deleted from RocksDB  [transaction.rs:216]
   → is_pending_verify.remove(P.hash)
5. Preload thread dequeues C:
   → load_full_unverified_block_by_hash(C)
   → get_block_header(P.hash): cache miss → DB read → None
   → .expect("parent header stored") → PANIC  [preload_unverified_blocks_channel.rs:95]
6. Preload thread terminates → unverified_block_tx dropped
   → verify thread's unverified_block_rx returns Err → verify thread exits.
   [verify.rs:103-106]
7. Node tip frozen; no further blocks processed until restart.
```

    fn get_block_header(&self, hash: &packed::Byte32) -> Option<HeaderView> {
        if let Some(cache) = self.cache()
            && let Some(header) = cache.headers.lock().get(hash)
        {
            return Some(header.clone());
        };
        let ret = self.get(COLUMN_BLOCK_HEADER, hash.as_slice()).map(|slice| {
            let reader = packed::HeaderViewReader::from_slice_should_be_ok(slice.as_ref());
            Into::<HeaderView>::into(reader)
        });

        if let Some(cache) = self.cache() {
            ret.inspect(|header| {
                cache.headers.lock().put(hash.clone(), header.clone());
            })
        } else {
            ret
        }
    }
