### Title
Unauthenticated `from` Field in Hole-Punching Protocol Enables Attacker-Directed Outbound TCP Connection Storms — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

---

### Summary

A directly-connected unprivileged peer can send two crafted hole-punching protocol messages — a `ConnectionRequest` with a spoofed `from` PeerId and attacker-controlled TCP addresses, followed by a `ConnectionSync` referencing the same spoofed `from` — to force the victim node to spawn persistent outbound TCP connection tasks targeting arbitrary attacker-controlled addresses. The root cause is that neither `ConnectionRequestProcess` nor `ConnectionSyncProcess` ever verifies that the `from` field in the message matches the authenticated PeerId of the actual sending peer.

---

### Finding Description

**Step 1 — Poisoning `pending_delivered` via `ConnectionRequest`:**

In `ConnectionRequestProcess::execute()`, when `content.to == self_peer_id`, the handler calls `respond_delivered(content.from, ...)`: [1](#0-0) 

Inside `respond_delivered`, the `from_peer_id` (taken verbatim from the message, not from the authenticated session) is used as the key to insert into `pending_delivered`: [2](#0-1) 

There is no check anywhere that `from_peer_id` equals the PeerId of `self.peer` (the actual authenticated sender). The attacker sets `from` to any arbitrary `FAKE_ID` and `listen_addrs` to attacker-controlled TCP addresses. After filtering to TCP-only addresses: [3](#0-2) 

...the victim inserts `pending_delivered[FAKE_ID] = ([attacker_tcp_addrs], now)`.

**Step 2 — Triggering passive NAT traversal via `ConnectionSync`:**

In `ConnectionSyncProcess::execute()`, when `route` is empty and `content.to == self_peer_id`, the handler looks up `pending_delivered` keyed by `content.from`: [4](#0-3) 

Again, `content.from` is taken verbatim from the message with no verification against the actual sender. If the lookup succeeds (it will, because Step 1 populated it with `FAKE_ID`), the victim spawns a `runtime::spawn` task that calls `try_nat_traversal` for each stored address: [5](#0-4) 

**The `try_nat_traversal` function** retries TCP connections in a loop for up to 30 seconds with ~200ms intervals (~150 attempts per address): [6](#0-5) 

With `ADDRS_COUNT_LIMIT = 24` addresses per request, each two-message exchange spawns up to 24 concurrent tasks, each making ~150 TCP SYN attempts. The attacker can repeat this with different `FAKE_ID` values to bypass the `pending_delivered` deduplication check: [7](#0-6) 

The rate limiter key is `(content.from, content.to, msg_item_id)`: [8](#0-7) 

Since `content.from` is attacker-controlled, the attacker trivially bypasses the rate limiter by rotating `FAKE_ID` values.

---

### Impact Explanation

The victim node spawns unbounded outbound TCP connection storms toward attacker-controlled addresses. Each spawned task runs for up to 30 seconds making ~150 TCP SYN attempts per address, consuming file descriptors, socket buffers, and CPU. The attacker can also direct these connections at third-party hosts (using their IPs as `listen_addrs`), turning the victim into an unwitting port-scanner or TCP flood source. This constitutes a concrete, attacker-directed outbound connection amplification attack.

---

### Likelihood Explanation

The attacker only needs a single direct P2P connection to the victim — a normal, unprivileged peer relationship. No PoW, no key material, no privileged role is required. The two-message sequence is trivially constructable from the public molecule schema. The missing guard (sender PeerId authentication) is a single absent check.

---

### Recommendation

In `respond_delivered`, verify that `from_peer_id` equals the authenticated PeerId of the actual sending session (`self.peer`). The session's PeerId is available via the peer registry:

```rust
// In respond_delivered, before inserting into pending_delivered:
let actual_sender_peer_id = self.protocol.network_state
    .peer_registry.read()
    .get_peer(self.peer)
    .map(|p| p.connected_addr.peer_id.clone()); // or equivalent

if actual_sender_peer_id.as_ref() != Some(&from_peer_id) {
    return StatusCode::InvalidFromPeerId.with_context("from does not match sender");
}
```

Similarly, in `ConnectionSyncProcess::execute()`, verify that `content.from` matches the authenticated PeerId of the actual sender before performing the `pending_delivered` lookup and spawning NAT traversal tasks.

---

### Proof of Concept

Two-step integration test:

1. Establish a direct P2P connection from attacker node to victim node.
2. Send a `ConnectionRequest` message with `from = FAKE_PEER_ID`, `to = VICTIM_PEER_ID`, `listen_addrs = [attacker_ip:attacker_port]`, `route = []`, `max_hops = 1`.
3. Victim inserts `pending_delivered[FAKE_PEER_ID] = ([attacker_ip:attacker_port], now)`.
4. Send a `ConnectionSync` message with `from = FAKE_PEER_ID`, `to = VICTIM_PEER_ID`, `route = []`.
5. Assert: victim spawns outbound TCP connections to `attacker_ip:attacker_port` (observable via a TCP listener on the attacker side receiving SYN packets from the victim).
6. Repeat steps 2–5 with a new `FAKE_PEER_ID` each iteration to confirm unbounded amplification.
