### Title
Unbounded Async Task Spawning via Compact Block Flood — (`sync/src/relayer/mod.rs`, `sync/src/relayer/compact_block_process.rs`, `util/runtime/src/native.rs`)

### Summary

An attacker who pre-mines a chain with a low `compact_target` (minimum difficulty) can send thousands of compact blocks to a victim node. Each block that passes header verification unconditionally spawns an async task via `request_proposal_txs`. Because `CompactBlock` messages are **explicitly exempted from the rate limiter** (the code comment says "CompactBlock will be verified by POW, it's OK to skip rate limit checking"), and because `spawn_task` has no admission control, the global Tokio runtime's task queue grows without bound, starving legitimate block propagation tasks.

---

### Finding Description

**Step 1 — Rate limiter is bypassed for CompactBlock:**

In `try_process`, the rate limiter is explicitly skipped for `CompactBlock` messages: [1](#0-0) 

The design assumption is that PoW cost is the natural rate limiter. This assumption breaks on a private chain with a low `compact_target`.

**Step 2 — PoW is verified only against the block's own `compact_target`:**

`HeaderVerifier::verify` calls `PowVerifier`, which checks the hash against `header.raw().compact_target()` — the value the attacker controls: [2](#0-1) [3](#0-2) 

There is no check that `compact_target` meets any minimum threshold relative to the main chain's epoch difficulty at the compact block processing stage. The `EpochVerifier` only receives `parent_fields.epoch` (an `EpochNumberWithFraction`) and cannot verify the compact_target against the expected epoch difficulty without the full epoch extension.

**Step 3 — Every valid compact block unconditionally spawns a task:**

After passing `non_contextual_check` and `contextual_check`, `request_proposal_txs` is called unconditionally: [4](#0-3) 

Inside `request_proposal_txs`, a task is spawned with no backpressure: [5](#0-4) 

If reconstruction fails (missing transactions), a second task is also spawned: [6](#0-5) 

**Step 4 — `spawn_task` has no admission control:** [7](#0-6) 

`spawn_task` delegates directly to `Handle::spawn` → `TokioHandle::spawn`. There is no maximum pending task count, no semaphore, and no backpressure mechanism.

**Step 5 — Timestamp check does not prevent pre-mining:**

The `TimestampVerifier` only requires `timestamp > median_time_of_ancestors` and `timestamp <= now + 15s`. An attacker can pre-mine a chain of N blocks with monotonically increasing past timestamps (e.g., genesis_time + i * 1000ms). All blocks pass the timestamp check. [8](#0-7) 

**Step 6 — `insert_valid_header` enables chained block acceptance:**

After each block passes checks, its header is inserted into the valid header index: [9](#0-8) 

This allows the next block in the attacker's chain to pass the parent check, enabling sequential processing of the entire pre-mined chain.

---

### Impact Explanation

Each of the N compact blocks spawns at least one async task in the global Tokio runtime. With N=10,000 tasks queued, the runtime's worker threads are saturated processing `fresh_proposals_filter` calls and state mutations. Legitimate tasks — including block propagation to downstream peers — are delayed or starved. The node fails to relay new blocks within the block interval (~10 seconds on mainnet), causing it to fall behind and disrupting the network.

---

### Likelihood Explanation

- **Cost**: Mining a chain with `compact_target = 0x207fffff` (maximum target) requires near-zero computational work. 10,000 blocks can be pre-mined in milliseconds on commodity hardware.
- **Access**: Only a standard P2P connection is required. No privileged role needed.
- **Detectability**: The attack is silent — no banning occurs because the blocks are structurally valid.
- **Repeatability**: The attacker can reconnect and repeat after disconnection.

---

### Recommendation

1. **Enforce a minimum `compact_target` during compact block header verification.** Reject blocks whose `compact_target` is below the current epoch's expected difficulty (or a fraction thereof). This restores the PoW-as-rate-limiter assumption.
2. **Apply a per-peer rate limit to `CompactBlock` messages.** The existing `rate_limiter` infrastructure is already in place; remove the exemption or add a separate, higher-quota limiter for compact blocks.
3. **Add a bounded semaphore before `spawn_task` calls** in `request_proposal_txs` and `missing_or_collided_post_process` to cap the number of concurrently queued tasks per peer.

---

### Proof of Concept

```
1. Generate a private CKB chain with compact_target = 0x207fffff (minimum difficulty).
2. Pre-mine 10,000 blocks with timestamps genesis_time + i*1000ms.
3. Connect to a victim node via P2P.
4. Send all 10,000 CompactBlock messages in rapid succession.
5. Observe: the victim node's Tokio runtime queues 10,000+ tasks from
   request_proposal_txs; block relay latency to downstream peers
   exceeds the 10-second block interval.
```
