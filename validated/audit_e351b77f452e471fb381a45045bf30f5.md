Audit Report

## Title
Unbounded Per-Message RocksDB + MMR Amplification DoS in Light Client `GetBlocksProof` Handler — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary

The `GetBlocksProof` handler in `LightClientProtocol` accepts messages containing up to 1,000 valid block hashes and performs 3,000+ synchronous RocksDB reads plus one MMR proof generation per message. Because `LightClientProtocol` has no rate limiter and well-formed max-size requests always return `Status::ok()` (which never triggers a ban), an attacker can flood the server in a tight loop from a single unauthenticated connection, saturating I/O and CPU.

## Finding Description

`GET_BLOCKS_PROOF_LIMIT` is set to 1,000 in `util/light-client-protocol-server/src/constant.rs` (L5). The guard in `execute()` uses a strict `>` comparison, so a message with exactly 1,000 hashes passes all checks (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`, L38–40).

For every hash in `found`, the handler performs three synchronous RocksDB reads — `get_block_header`, `get_block_uncles`, `get_block_extension` — in addition to the earlier `is_main_chain` partition scan (`get_blocks_proof.rs`, L81–94). `reply_proof()` then calls `mmr.gen_proof(items_positions)` with up to 1,000 leaf positions (`util/light-client-protocol-server/src/lib.rs`, L210).

`LightClientProtocol` has no `rate_limiter` field (`lib.rs`, L26–35) and `try_process()` performs no rate check before dispatching any handler (`lib.rs`, L96–125). A successful request returns `Status::ok()` (code 200); `should_ban()` only fires for codes in the 400–499 range (`util/light-client-protocol-server/src/status.rs`, L95–101), and the `StatusCode` enum has no `TooManyRequests` variant (`status.rs`, L16–38). No ban is ever issued for a valid max-size request.

By contrast, `Relayer` explicitly rate-limits every non-PoW message per `(peer, item_id)` before processing (`sync/src/relayer/mod.rs`, L88–123), using `sync/src/status.rs`'s `TooManyRequests = 110` variant. The light client protocol server has no equivalent guard.

## Impact Explanation

Each max-size `GetBlocksProof` message forces the server to execute:
- 1,000 × `is_main_chain()` (index lookup)
- 1,000 × `get_block_header()` (RocksDB read)
- 1,000 × `get_block_uncles()` (RocksDB read)
- 1,000 × `get_block_extension()` (RocksDB read)
- 1 × `mmr.gen_proof(1,000 positions)` (CPU-bound MMR traversal)

A single attacker connection sending these messages in a tight loop saturates server I/O and CPU, degrading or halting normal sync and relay throughput. This maps to the **High** impact class: *"Vulnerabilities which could easily crash a CKB node"* and *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"* (10,001–15,000 points).

## Likelihood Explanation

The light client protocol is opt-in on production CKB full nodes, which limits the attack surface to nodes that have explicitly enabled it. However, for any such node, any peer that can establish a TCP connection and speak the protocol can execute this attack — no authentication, no PoW, no stake, and no prior trust is required. The attack is trivially scriptable: collect 1,000 valid main-chain block hashes once, then replay the same `GetBlocksProof` message in a loop. No ban is ever issued, so the attacker connection persists indefinitely.

## Recommendation

Add a per-peer, per-message-type rate limiter to `LightClientProtocol`, mirroring the pattern already used in `Relayer` (`sync/src/relayer/mod.rs`, L63–67, L81–82, L116–123):

1. Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`.
2. Add a `TooManyRequests = 429` variant to `util/light-client-protocol-server/src/status.rs`'s `StatusCode` enum (4xx range so `should_ban()` fires).
3. In `try_process()`, check the limiter keyed on `(peer_index, message.item_id())` before dispatching any handler; return `StatusCode::TooManyRequests` on breach.
4. Additionally consider reducing `GET_BLOCKS_PROOF_LIMIT` or splitting large requests into paginated responses.

## Proof of Concept

1. Run a CKB full node with light client protocol enabled and a chain of height ≥ 1,000.
2. From a separate process, connect as a light client peer.
3. Collect 1,000 valid main-chain block hashes and the current tip hash.
4. In a tight loop, send `GetBlocksProof { block_hashes: [h1…h1000], last_hash: tip }`.
5. Observe: server CPU and RocksDB I/O spike to saturation; no ban is issued against the sending peer; normal sync/relay throughput degrades.
