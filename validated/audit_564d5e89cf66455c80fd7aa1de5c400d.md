### Title
Expired Network Alert Signature Replay via P2P — (`util/network-alert/src/alert_relayer.rs`)

---

### Summary

The `AlertRelayer::received` P2P handler accepts and re-processes any alert whose ID has been evicted from the in-memory dedup map by `clear_expired_alerts`. Because `clear_expired_alerts` removes expired alert IDs from `received_alerts` without adding them to `cancel_filter`, and because the P2P handler performs no `notice_until` expiry check, any unprivileged P2P peer can replay a previously-seen, legitimately-signed alert after it expires. The replayed alert passes signature verification, is re-added to the notifier, re-triggers `notify_network_alert` (including any configured `network_alert_notify_script`), and is re-broadcast to all connected peers.

---

### Finding Description

**Root cause — two cooperating gaps:**

**Gap 1: `clear_expired_alerts` removes the dedup entry but does not tombstone it.**

`Notifier::clear_expired_alerts` removes expired alerts from `received_alerts` (the HashMap used for deduplication) and from `noticed_alerts`, but does NOT insert the expired ID into `cancel_filter`:

```rust
// util/network-alert/src/notifier.rs
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|_id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        notice_until > now          // expired entries are dropped
    });
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
    // cancel_filter is never updated here
}
```

After this runs, `has_received(id)` returns `false` for the expired alert ID:

```rust
pub fn has_received(&self, id: u32) -> bool {
    self.received_alerts.contains_key(&id) || self.cancel_filter.contains(&id)
    // both are now false for the expired alert
}
```

`clear_expired_alerts` is called every time a new peer connects:

```rust
// util/network-alert/src/alert_relayer.rs
async fn connected(&mut self, nc: Arc<dyn CKBProtocolContext + Sync>, peer_index: PeerIndex, _version: &str) {
    self.clear_expired_alerts();   // evicts expired IDs from dedup map
    // ...
}
```

**Gap 2: The P2P `received` handler performs no `notice_until` check.**

The RPC path (`send_alert`) explicitly rejects expired alerts:

```rust
// rpc/src/module/alert.rs
let now_ms = ckb_systemtime::unix_time_as_millis();
let notice_until: u64 = alert.raw().notice_until().into();
if notice_until < now_ms {
    return Err(RPCError::invalid_params(...));
}
```

The P2P `received` handler has no equivalent check:

```rust
// util/network-alert/src/alert_relayer.rs
async fn received(&mut self, nc: ..., peer_index: PeerIndex, data: Bytes) {
    // parse alert ...
    let alert_id = alert.as_reader().raw().id().into();
    if self.notifier.lock().has_received(alert_id) {   // only ID dedup, no expiry check
        return;
    }
    if let Err(err) = self.verifier.verify_signatures(&alert) {  // sig check only
        nc.ban_peer(...);
        return;
    }
    // broadcast + add — no notice_until validation at all
    self.notifier.lock().add(&alert);
}
```

`Notifier::add` also performs no expiry check — it inserts the alert into `received_alerts`, calls `notify_controller.notify_network_alert`, pushes to `noticed_alerts`, and re-sorts.

**Exploit flow:**

1. Developers broadcast a valid alert (ID=X, `notice_until=T`, 2-of-4 signatures). All nodes accept it; the alert is public on the P2P network.
2. Time advances past `T`. The alert expires.
3. A new peer connects to a victim node, triggering `clear_expired_alerts`. Alert ID=X is removed from `received_alerts`. `has_received(X)` now returns `false`.
4. Attacker (any P2P peer) sends the original, unmodified alert bytes for ID=X to the victim node.
5. `received` handler: `has_received(X)` → `false` → proceeds.
6. `verifier.verify_signatures` → passes (same valid developer signatures, unchanged message).
7. `notifier.add(&alert)` → re-inserts into `received_alerts`, calls `notify_network_alert` again, pushes to `noticed_alerts`, re-sorts.
8. Alert is re-broadcast to all connected peers, which repeat steps 3–8.

---

### Impact Explanation

- **False/stale emergency messages re-displayed on all nodes**: Operators and monitoring systems see an outdated alert as if it were newly issued, potentially causing incorrect incident response.
- **`network_alert_notify_script` re-triggered**: If a node operator has configured a shell script to run on alert receipt (the `network_alert_notify_script` config option, invoked in `notify/src/lib.rs`), that script executes again for every replay. Depending on the script's action (e.g., automated shutdown, paging on-call staff, posting to external systems), this can cause operational disruption.
- **Network-wide re-propagation**: Because the replayed alert is re-broadcast to all connected peers, a single attacker with one P2P connection can cause the expired alert to re-propagate across the entire network.

---

### Likelihood Explanation

- **Preconditions**: The attacker only needs (a) a P2P connection to any CKB node and (b) the bytes of any previously-broadcast alert, which are public and observable on the network.
- **No privileged access required**: The attacker does not need developer signing keys. The signatures in the replayed alert are the original, legitimate ones.
- **Trigger condition is automatic**: `clear_expired_alerts` is called on every new peer connection, so the dedup entry is reliably evicted without any attacker action.
- **Repeatable**: After the replayed alert is re-added and then expires again, the cycle can repeat indefinitely.

---

### Recommendation

1. **Add a `notice_until` check in `AlertRelayer::received`** before the `has_received` check, mirroring the RPC path:
   ```rust
   let notice_until: u64 = alert.as_reader().raw().notice_until().into();
   let now_ms = ckb_systemtime::unix_time_as_millis();
   if notice_until < now_ms {
       // silently drop or disconnect peer
       return;
   }
   ```
2. **Tombstone expired IDs in `clear_expired_alerts`**: When removing an expired alert from `received_alerts`, also insert its ID into `cancel_filter` so `has_received` continues to return `true` for that ID even after expiry.
3. **Add a `notice_until` guard in `Notifier::add`** as a defense-in-depth measure, so that even if the P2P handler is bypassed, expired alerts cannot be re-inserted.

---

    pub fn add(&mut self, alert: &Alert) {
        let alert_id = alert.raw().id().into();
        let alert_cancel = alert.raw().cancel().into();
        if self.has_received(alert_id) {
            return;
        }
        // checkout cancel_id
        if alert_cancel > 0 {
            self.cancel(alert_cancel);
        }
        // add to received alerts
        self.received_alerts.insert(alert_id, alert.clone());

        // check conditions, figure out do we need to notice this alert
        if !self.is_version_effective(alert) {
            debug!("Received a version ineffective alert {:?}", alert);
            return;
        }

        if self.noticed_alerts.contains(alert) {
            return;
        }
        self.notify_controller.notify_network_alert(alert.clone());
        self.noticed_alerts.push(alert.clone());
        // sort by priority
        self.noticed_alerts.sort_by_key(|a| {
            let priority: u32 = a.raw().priority().into();
            u32::MAX - priority
        });
    }
