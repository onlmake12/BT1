### Title
Missing Per-Peer Rate Limiting in `LightClientProtocol::received` Allows Unbounded MMR Root Computation — (`util/light-client-protocol-server/src/lib.rs`)

### Summary
`LightClientProtocol` has no rate limiter. Every well-formed `GetLastState` message from any peer unconditionally triggers `get_verifiable_tip_header()`, which performs `snapshot.chain_root_mmr(tip_number - 1).get_root()` — an O(log N) RocksDB read + hash-merge sequence. A single unprivileged peer can flood this handler with no throttle, no ban, and no backpressure, saturating the node's I/O and CPU.

### Finding Description

`LightClientProtocol::received` parses the message and immediately dispatches to `try_process` with no rate check: [1](#0-0) 

The struct itself carries no `rate_limiter` field — only `shared: Shared`: [2](#0-1) 

`try_process` routes `GetLastState` directly to `GetLastStateProcess::execute`: [3](#0-2) 

`execute` unconditionally calls `get_verifiable_tip_header()` on every message: [4](#0-3) 

`get_verifiable_tip_header()` calls `snapshot.chain_root_mmr(tip_block.number() - 1).get_root()` on every invocation: [5](#0-4) 

`chain_root_mmr` creates an `MMR` backed by the `Snapshot`'s `MMRStore`, whose `get_elem` issues a live RocksDB read per MMR node: [6](#0-5) [7](#0-6) 

For a chain of N blocks the MMR has O(log N) peaks; `get_root()` must read and merge all of them. On mainnet (~10 M blocks) that is ~23 RocksDB point-reads plus hash merges per `GetLastState` message.

**Contrast with protocols that do have rate limiting:**

`Relayer::try_process` gates every non-PoW message at 30 req/s per (peer, message-type): [8](#0-7) 

`HolePunching::received` applies the same governor-based check before any processing: [9](#0-8) 

`LightClientProtocol` has neither construct. The grep for `rate_limit|rate_limiter|RateLimiter|TooManyRequests|throttle` across the entire `util/light-client-protocol-server/` tree returns zero matches.

### Impact Explanation
A single TCP-connected peer can send a tight loop of minimal `GetLastState{subscribe:false}` messages (each ~2 bytes on the wire). Each message forces the server to:
1. Acquire a snapshot Arc clone.
2. Perform O(log N) RocksDB reads from `COLUMN_CHAIN_ROOT_MMR`.
3. Compute hash merges to produce the MMR root.
4. Serialize and enqueue a `SendLastState` reply.

There is no ban, no disconnect, and no throttle triggered by this valid-but-repeated request. Under sustained flood the node's RocksDB I/O bandwidth and async-runtime CPU are consumed, degrading or blocking service for all other peers (sync, relay, other light clients).

### Likelihood Explanation
The attack requires only a standard P2P connection to a node that has the light-client protocol enabled. No key, no stake, no PoW. The message is trivially constructable. The absence of any rate-limiting field in `LightClientProtocol` (unlike `Relayer` and `HolePunching`) indicates this was an oversight rather than an intentional design choice.

### Recommendation
Add a `governor`-based `RateLimiter` field to `LightClientProtocol` (mirroring `Relayer`), keyed by `(PeerIndex, message_item_id)`, and check it at the top of `received` before dispatching to `try_process`. A limit of 30 req/s per peer per message type (matching the existing protocols) would eliminate the amplification while allowing legitimate light-client polling.

### Proof of Concept
1. Construct a mock `LightClientProtocol` with a chain of N blocks (e.g. N = 1000).
2. In a tight loop, call `protocol.received(nc.context(), peer_index, get_last_state_msg).await` M times (e.g. M = 10 000) from a single `PeerIndex` with no inter-message delay.
3. Assert that `nc.sent_messages().borrow().len() == M` (all M MMR computations completed).
4. Assert that `nc.not_banned(peer_index)` is `true` (no ban was applied).
5. Measure wall-clock time; compare against a version with a 30 req/s governor guard — the unguarded version will be orders of magnitude slower under load and will monopolize the async executor.

The existing test harness in `util/light-client-protocol-server/src/tests/` already provides `MockChain`, `MockNetworkContext`, and `protocol.received(...)` — the PoC requires only adding the loop and the assertion on message count.

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
