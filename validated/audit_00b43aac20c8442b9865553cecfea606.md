### Title
Zero-Fee `transferFrom` Call in `transfer_fee` Causes DoS When Fee Is Set to Zero — (File: `target_chains/starknet/contracts/src/pyth.cairo`)

---

### Summary

The `transfer_fee` helper in Pyth's Starknet contract unconditionally calls `transferFrom(caller, contract, 0)` when governance has set the per-update fee to zero. ERC20 tokens that reject zero-value transfers will cause every `update_price_feeds` call to revert, making price feed updates impossible on Starknet whenever the protocol fee is waived.

---

### Finding Description

Governance can set `single_update_fee1` or `single_update_fee2` to `0` via the `SetFee` / `SetFeeInToken` governance actions: [1](#0-0) 

When either fee is `0`, `update_price_feeds_internal` still calls `transfer_fee` unconditionally: [2](#0-1) 

Inside `transfer_fee`, `total_fee = single_update_fee * num_updates = 0`. The allowance and balance checks both pass trivially (`u256 >= 0` is always true), so execution reaches the `transferFrom` call with `amount = 0`: [3](#0-2) 

The contract documentation explicitly states that **any ERC20-compatible token** can be used as the fee token: [4](#0-3) 

Several ERC20 implementations (including some versions of USDT-style tokens and custom tokens) revert or return `false` on zero-value `transferFrom` calls. When `transferFrom(caller, contract, 0)` returns `false`, `transfer_fee` returns `false`. The fallback to fee token 2 is then attempted; if `single_update_fee2` is also `0`, the same failure occurs, and the transaction panics with `InsufficientFeeAllowance`: [5](#0-4) 

This is structurally identical to the reported vulnerability: a fee-validation/transfer function is called unconditionally even when the fee amount is zero, causing a revert that blocks all price feed updates.

---

### Impact Explanation

All public entry points that call `update_price_feeds_internal` — namely `update_price_feeds`, `parse_price_feed_updates`, and `parse_unique_price_feed_updates` — become permanently reverted for any caller when both configured fee tokens reject zero-value transfers. Price feed data on Starknet cannot be updated, breaking every downstream consumer (DeFi protocols, liquidation bots, etc.) that depends on fresh Pyth prices.

---

### Likelihood Explanation

- Governance legitimately sets fees to `0` to waive charges (e.g., promotional periods, subsidized deployments).
- The contract explicitly supports arbitrary ERC20 tokens as fee tokens, and zero-value transfer rejection is a known behavior in non-standard or older ERC20 implementations.
- The attacker-controlled entry path is simply calling `update_price_feeds` after governance has set the fee to `0` — no special privilege is required.

---

### Recommendation

Add an early-return guard in `transfer_fee` when `total_fee` is zero, mirroring the pattern already used in the non-Osmosis CosmWasm contract:

```cairo
fn transfer_fee(
    num_updates: u8,
    caller: ContractAddress,
    contract: ContractAddress,
    fee_token: ContractAddress,
    single_update_fee: u256,
) -> bool {
    let total_fee = single_update_fee * num_updates.into();
    if total_fee == 0 {
        return true;  // No fee required; skip transferFrom entirely
    }
    let fee_contract = IERC20CamelDispatcher { contract_address: fee_token };
    if fee_contract.allowance(caller, contract) < total_fee {
        return false;
    }
    if fee_contract.balanceOf(caller) < total_fee {
        return false;
    }
    if !fee_contract.transferFrom(caller, contract, total_fee) {
        return false;
    }
    true
}
```

The analogous CosmWasm contract already handles this correctly by short-circuiting when the configured fee amount is zero: [6](#0-5) 

---

### Proof of Concept

1. Governance submits a `SetFee` VAA setting `single_update_fee1 = 0` and `single_update_fee2 = 0` for the Starknet deployment.
2. Any unprivileged user calls `update_price_feeds(valid_update_data)`.
3. `update_price_feeds_internal` reads `single_update_fee1 = 0`, computes `total_fee = 0`, and calls `transfer_fee(num_updates, caller, contract, fee_token1, 0)`.
4. Inside `transfer_fee`, `total_fee = 0`; allowance/balance checks pass; `fee_token1.transferFrom(caller, contract, 0)` is called.
5. If `fee_token1` is an ERC20 that returns `false` (or reverts) on zero-value transfers, `transfer_fee` returns `false`.
6. The fallback path calls `transfer_fee(num_updates, caller, contract, fee_token2, 0)` — same result.
7. Both return `false`; the contract panics with `InsufficientFeeAllowance` at line 677.
8. All price feed updates on Starknet are blocked despite the fee being intentionally set to zero. [7](#0-6)
