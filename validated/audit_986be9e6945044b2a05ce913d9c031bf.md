### Title
Excess ETH Sent to `requestPriceUpdatesWithCallback` Is Not Refunded — Silently Transferred to Provider - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `requestPriceUpdatesWithCallback` is `payable` and enforces only a minimum fee (`msg.value >= requiredFee`). Any ETH sent above `requiredFee` is not refunded to the caller — it is silently absorbed into `req.fee` and later credited to the provider's accrued balance. There is no refund path and no cap on accepted `msg.value`.

---

### Finding Description

In `requestPriceUpdatesWithCallback`, the fee check is one-sided:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [1](#0-0) 

After this check, the provider's stored fee is computed as:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

Only `_state.pythFeeInWei` is credited to Pyth's accrued balance:

```solidity
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [3](#0-2) 

This means if a user sends `msg.value = requiredFee + X`, the excess `X` is silently folded into `req.fee`. When `executeCallback` is later called, the provider is credited with:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [4](#0-3) 

The user's overpayment `X` is permanently transferred to the provider. There is no refund mechanism and no upper bound on accepted `msg.value`.

The `IEcho` interface documents the function as `payable` without any refund guarantee: [5](#0-4) 

---

### Impact Explanation

Any user who sends more ETH than `getFee(provider, callbackGasLimit, priceIds)` permanently loses the excess to the provider. This is a direct, unrecoverable loss of user funds with no admin recovery path. The `withdrawFees` function only covers `_state.accruedFeesInWei` (Pyth's portion), not provider-accrued fees: [6](#0-5) 

Provider-accrued fees are only withdrawable by the provider or its fee manager: [7](#0-6) 

---

### Likelihood Explanation

Medium. Overpayment is a common occurrence in EVM interactions:
- Front-end fee estimation may add a buffer
- Users calling the contract directly (e.g., via Etherscan or scripts) may round up
- Gas price fluctuations between fee estimation and transaction submission can cause overpayment
- The `getFee` function returns a `uint96`, and callers may send a rounded-up `uint256` value

The entry path is fully unprivileged — any external caller of `requestPriceUpdatesWithCallback` is affected.

---

### Recommendation

Refund excess ETH to the caller at the end of `requestPriceUpdatesWithCallback`:

```solidity
uint256 excess = msg.value - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "Refund failed");
}
```

Alternatively, enforce an exact fee: `if (msg.value != requiredFee) revert IncorrectFee();`

---

### Proof of Concept

1. Provider registers with `baseFeeInWei = 100`, `feePerFeedInWei = 10`, `feePerGasInWei = 1`.
2. User calls `getFee(provider, 100000, [priceId])` → returns `requiredFee = 100 + 10 + 100000 + pythFeeInWei`.
3. User calls `requestPriceUpdatesWithCallback{value: requiredFee + 1 ether}(...)`.
4. Contract stores `req.fee = (requiredFee + 1 ether) - pythFeeInWei` — the 1 ETH overpayment is absorbed.
5. Provider calls `executeCallback(...)` and is credited with `req.fee - pythFee`, which includes the user's 1 ETH overpayment.
6. User has no mechanism to recover the 1 ETH.
