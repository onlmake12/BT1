### Title
`MAX_DEPOSIT_LIMIT` Cap Bypassable in Permanent Subscriptions via Incremental `addFunds` Calls — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract enforces a `MAX_DEPOSIT_LIMIT` cap on permanent subscriptions, but the check in `addFunds` validates only the single incoming `msg.value` rather than the resulting cumulative balance. An unprivileged subscription manager can bypass the cap entirely by calling `addFunds` multiple times with individually sub-limit amounts.

---

### Finding Description

In `createSubscription`, the deposit limit is checked correctly against the initial deposit: [1](#0-0) 

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

In `addFunds`, the same pattern is repeated — but it still only checks `msg.value`, not the post-deposit balance: [2](#0-1) 

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

Because `status.balanceInWei` is updated **after** the check, a manager whose permanent subscription already holds `MAX_DEPOSIT_LIMIT` can call `addFunds` with any amount `<= MAX_DEPOSIT_LIMIT` and the check passes, pushing the cumulative balance arbitrarily above the cap.

Permanent subscriptions explicitly block withdrawals: [3](#0-2) 

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

So any ETH deposited beyond `MAX_DEPOSIT_LIMIT` is permanently locked with no recovery path.

---

### Impact Explanation

Funds deposited into a permanent subscription beyond `MAX_DEPOSIT_LIMIT` are irrecoverably locked in the contract. The cap exists as a safety bound; bypassing it allows a manager to permanently lock an unbounded amount of ETH. While the manager is locking their own funds, the protocol's stated invariant is violated, and the contract accumulates ETH it can never return.

---

### Likelihood Explanation

Any subscription manager of a permanent subscription can trigger this by calling `addFunds` in a loop with amounts just below `MAX_DEPOSIT_LIMIT`. No special privilege, leaked key, or external dependency is required — only the manager role, which is self-assigned at subscription creation.

---

### Recommendation

Change the check in `addFunds` to validate the **post-deposit cumulative balance**:

```solidity
// Before (incorrect — only checks single deposit):
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {

// After (correct — checks resulting balance):
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
```

---

### Proof of Concept

1. Deploy `Scheduler` with `MAX_DEPOSIT_LIMIT = 100 ether`.
2. Call `createSubscription{value: 100 ether}(permanentParams)` → succeeds, balance = 100 ETH.
3. Call `addFunds{value: 50 ether}(subscriptionId)`:
   - Check: `50 ether > 100 ether` → **false**, no revert.
   - `status.balanceInWei` becomes **150 ETH**, exceeding the cap.
4. Call `withdrawFunds` → reverts with `CannotUpdatePermanentSubscription`.
5. The 50 ETH above the cap is permanently locked. [4](#0-3)

```text
    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
```
