Audit Report

## Title
Missing `BLOCK_INVALID` Guard Allows Repeated Re-verification of Known-Invalid Headers — (`sync/src/synchronizer/headers_process.rs`)

## Summary

In `HeaderAcceptor::accept()`, the block status is fetched and used to short-circuit only the `HEADER_VALID` case. The `BLOCK_INVALID` flag is fetched but never tested, causing the function to fall through and re-execute all header verification steps — including `HeaderVerifier::verify()` (PoW check) — on every re-delivery of a known-invalid header. A developer `FIXME` comment at lines 301–302 explicitly acknowledges this gap. Any unprivileged P2P peer can exploit this by repeatedly sending `SendHeaders` messages containing a pre-crafted invalid header, forcing the node to re-run verification on each delivery with no early-exit guard.

## Finding Description

In `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::accept()` at lines 301–322:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer state and return early
    return result;
}
```

After this block, the function unconditionally falls through to:

1. `prev_block_check` (line 324) — checks if parent is `BLOCK_INVALID`
2. `non_contextual_check` (line 334) — calls `self.verifier.verify(self.header)`, which includes Eaglesong PoW verification and other header checks
3. `version_check` (line 346)

All three checks call `shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID)` on failure, but on the next invocation of `accept()` for the same header, the stored `BLOCK_INVALID` status is fetched and then silently ignored. The `HEADER_VALID` branch exits early; the `BLOCK_INVALID` case has no corresponding branch.

The contrast with `compact_block_process.rs` lines 259–261 is direct — that path correctly guards:

```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
```

The `HeadersProcess::execute()` loop at lines 154–179 iterates over all headers in a `SendHeaders` message (up to `MAX_HEADERS_LEN` = 2000) with no per-hash deduplication for the `BLOCK_INVALID` case, calling `acceptor.accept()` for each.

## Impact Explanation

A malicious peer can force the victim node to re-execute `HeaderVerifier::verify()` (including Eaglesong PoW verification and all other non-contextual checks) on every delivery of a known-invalid header. With `MAX_HEADERS_LEN` = 2000 headers per message and no rate limit specific to this path, a single peer sending a tight loop of `SendHeaders` messages causes sustained, unbounded CPU consumption proportional to the send rate, with no useful work performed. This matches: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

- Any unauthenticated P2P peer can send `SendHeaders` messages; no privilege, key, or special role is required.
- The attacker only needs to craft one header that fails any of the three checks (invalid PoW nonce, invalid parent, wrong version). The first delivery marks it `BLOCK_INVALID`; every subsequent delivery re-triggers the full check path.
- The `FIXME` comment confirms this is a known, unresolved gap in the code, not a speculative edge case.
- No per-hash deduplication guard exists for the `BLOCK_INVALID` case in the processing loop.

## Recommendation

Add an explicit early-return for `BLOCK_INVALID` immediately after the `HEADER_VALID` check in `HeaderAcceptor::accept()`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing logic
    return result;
}
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}
```

This mirrors the pattern already used in `compact_block_process.rs` at lines 259–261.

## Proof of Concept

1. Connect to a CKB node as an unprivileged P2P peer using the sync protocol.
2. Craft a `SendHeaders` message containing a header with a valid structure but an invalid PoW nonce (does not satisfy the Eaglesong target).
3. Send the message once. The node runs `non_contextual_check` → `HeaderVerifier::verify()` → PoW fails → `insert_block_status(hash, BLOCK_INVALID)` (line 341).
4. In a tight loop, re-send the identical `SendHeaders` message (optionally with up to 2000 copies of the same invalid header per message).
5. On each iteration, `accept()` fetches `BLOCK_INVALID` from the status map, skips the `HEADER_VALID` branch (line 304), and falls through to re-run `non_contextual_check` → `HeaderVerifier::verify()` again.
6. Observe sustained CPU consumption on the victim node with no corresponding useful work, confirming the missing guard.
