### Title
Unchecked ETH Refund Return Value in `updateFeeds()` Silently Traps Caller's Excess ETH â€” (`target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3.updateFeeds()` attempts to refund excess ETH to `msg.sender` after paying the Pyth fee, but the return value of the low-level `.call{}` is captured into a `success` variable that is never checked. The `// solhint-disable-next-line no-unused-vars` comment explicitly acknowledges this. If `msg.sender` is a contract without a `receive()` or `fallback()` function, the refund silently fails and the excess ETH is permanently trapped in the `PythAggregatorV3` contract until the next caller drains it.

---

### Finding Description

In `PythAggregatorV3.sol`, the `updateFeeds()` function is a `public payable` entry point that:
1. Queries the required Pyth fee via `pyth.getUpdateFee(priceUpdateData)`.
2. Calls `pyth.updatePriceFeeds{value: fee}(priceUpdateData)`.
3. Attempts to refund the remaining contract balance (`address(this).balance`) to `msg.sender`.

The refund at step 3 is:

```solidity
// refund remaining eth
// solhint-disable-next-line no-unused-vars
(bool success, ) = payable(msg.sender).call{
    value: address(this).balance
}("");
``` [1](#0-0) 

The `success` boolean is declared but never inspected. If the call reverts (e.g., because `msg.sender` is a contract without `receive()`/`fallback()`), execution continues silently and the ETH remains in the `PythAggregatorV3` contract.

Because the refund sends `address(this).balance` (the entire contract balance), any subsequently trapped ETH will be swept to the *next* caller who successfully receives the refund â€” meaning the original caller permanently loses their overpayment and an unrelated future caller gains it.

All other ETH-transfer sites in the Pyth codebase correctly check the return value:

- `Echo.sol` `withdrawFees`: `require(sent, "Failed to send fees")` [2](#0-1) 
- `Echo.sol` `withdrawAsFeeManager`: `require(sent, "Failed to send fees")` [3](#0-2) 
- `Scheduler.sol` `withdrawFunds`: `require(sent, "Failed to send funds")` [4](#0-3) 
- `Scheduler.sol` `_processFeesAndPayKeeper`: `revert SchedulerErrors.KeeperPaymentFailed()` [5](#0-4) 
- `PythGovernance.sol` `withdrawFee`: `require(success, "Failed to withdraw fees")` [6](#0-5) 
- `EntropyGovernance.sol` `withdrawFee`: `require(success, "Failed to withdraw fees")` [7](#0-6) 

`PythAggregatorV3.updateFeeds()` is the sole exception. [8](#0-7) 

---

### Impact Explanation

Any caller of `updateFeeds()` that:
- Sends more ETH than the required Pyth fee (overpayment), **and**
- Is a contract without a `receive()` or `fallback()` function

will silently lose the excess ETH. The ETH is not returned and not reverted â€” it accumulates in `PythAggregatorV3`. The next EOA or contract-with-`receive()` that calls `updateFeeds()` will receive the accumulated trapped ETH as part of their own refund, constituting a direct transfer of funds from the original caller to an unrelated party.

---

### Likelihood Explanation

`PythAggregatorV3` is a Chainlink-compatible adapter contract intended to be deployed and called by other smart contracts (DeFi protocols, aggregators, keepers). Many such integrating contracts do not implement `receive()`/`fallback()`. The `updateFeeds()` function is `public payable` and callable by any unprivileged address. No special role or key is required. The scenario is realistic whenever an integrating contract calls `updateFeeds()` with a slightly over-estimated ETH value (a common defensive pattern to avoid underpayment reverts).

---

### Recommendation

Check the return value of the refund call and revert on failure, consistent with every other ETH-transfer site in the codebase:

```solidity
function updateFeeds(bytes[] calldata priceUpdateData) public payable {
    uint fee = pyth.getUpdateFee(priceUpdateData);
    pyth.updatePriceFeeds{value: fee}(priceUpdateData);

    // refund remaining eth
    if (address(this).balance > 0) {
        (bool success, ) = payable(msg.sender).call{
            value: address(this).balance
        }("");
        require(success, "ETH refund failed");
    }
}
```

---

### Proof of Concept

1. Deploy `PythAggregatorV3` wrapping a live Pyth contract.
2. Deploy an attacker-controlled contract `Caller` that has **no** `receive()` or `fallback()` function.
3. From `Caller`, call `PythAggregatorV3.updateFeeds{value: fee + 1 ether}(data)`.
4. The Pyth fee is paid; `address(this).balance == 1 ether`.
5. The refund call `payable(Caller).call{value: 1 ether}("")` reverts internally (no `receive()`), but `success` is never checked â€” execution continues.
6. `Caller` loses 1 ether; it remains in `PythAggregatorV3`.
7. Any subsequent EOA caller of `updateFeeds()` who sends even 1 wei will receive `1 ether + 1 wei` as their refund, draining the trapped funds. [8](#0-7)
