### Title
Division by Zero in `calculateTwap` When Start and End TWAP Messages Share the Same `publishSlot` — (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary
`Pyth.sol#calculateTwap` divides by `slotDiff`, which is computed as `twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot`. The upstream validation in `validateTwapPriceInfo` only rejects the case where `start.publishSlot > end.publishSlot`, but **not** the equal case. When both messages carry the same `publishSlot`, `slotDiff == 0` and all three subsequent divisions revert with a division-by-zero panic, making `parseTwapPriceFeedUpdates` permanently unusable for that input pair.

### Finding Description

`validateTwapPriceInfo` enforces slot ordering with a non-strict inequality:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

This allows `start.publishSlot == end.publishSlot` to pass validation. `calculateTwap` then computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // division by zero → revert
uint128 twapConf  = confDiff / uint128(slotDiff);           // division by zero → revert
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // division by zero → revert
``` [2](#0-1) 

The Solana receiver correctly uses a **strict** greater-than check (`end_msg.publish_slot > start_msg.publish_slot`) to prevent this:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The EVM implementation is inconsistent with the Solana implementation and lacks this guard.

### Impact Explanation
Any call to `parseTwapPriceFeedUpdates` that supplies two valid, Wormhole-attested TWAP messages with identical `publishSlot` values will unconditionally revert. This is a **denial-of-service** on the TWAP price feed update path for EVM chains. Downstream protocols relying on `parseTwapPriceFeedUpdates` for TWAP data will be unable to consume price updates for the affected slot pair, potentially blocking liquidations, oracle-dependent state transitions, or any time-sensitive DeFi operation.

### Likelihood Explanation
Pythnet can publish multiple TWAP messages within a single slot. A user who observes two valid on-chain TWAP attestations sharing the same `publishSlot` (e.g., a re-published or re-attested message within the same slot) can submit them to `parseTwapPriceFeedUpdates` and trigger the revert. No privileged access is required — only the ability to call a public payable function with valid Wormhole-attested data. The scenario is realistic whenever the Wormhole guardian set attests two TWAP messages from the same Pythnet slot.

### Recommendation
Change the slot validation in `validateTwapPriceInfo` from a non-strict to a strict inequality, matching the Solana receiver's behavior:

```diff
- if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
+ if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
      revert PythErrors.InvalidTwapUpdateDataSet();
  }
``` [1](#0-0) 

This ensures `slotDiff >= 1` before `calculateTwap` is entered, eliminating the division-by-zero path.

### Proof of Concept

1. Obtain two valid Wormhole-attested TWAP messages for the same price feed where both have `publishSlot = S` (e.g., `S = 1000`), with `prevPublishTime < publishTime` satisfied for each (so the timestamp checks pass).
2. Call `parseTwapPriceFeedUpdates(updateData, priceIds, minTime, maxTime)` on the EVM Pyth contract.
3. `validateTwapPriceInfo` passes because `1000 > 1000` is `false`.
4. `calculateTwap` computes `slotDiff = 1000 - 1000 = 0`.
5. Line 731 executes `priceDiff / int128(0)` → EVM division-by-zero panic → transaction reverts.
6. All callers of `parseTwapPriceFeedUpdates` with this slot pair are permanently blocked. [4](#0-3) [5](#0-4)

    function validateTwapPriceInfo(
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure {
        // First validate each individual price's uniqueness
        if (
            twapPriceInfoStart.prevPublishTime >= twapPriceInfoStart.publishTime
        ) {
            revert PythErrors.InvalidTwapUpdateData();
        }
        if (twapPriceInfoEnd.prevPublishTime >= twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateData();
        }

        // Then validate the relationship between the two data points
        if (twapPriceInfoStart.expo != twapPriceInfoEnd.expo) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishTime > twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
    }

```text
    function calculateTwap(
        bytes32 priceId,
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure returns (PythStructs.TwapPriceFeed memory twapPriceFeed) {
        twapPriceFeed.id = priceId;
        twapPriceFeed.startTime = twapPriceInfoStart.publishTime;
        twapPriceFeed.endTime = twapPriceInfoEnd.publishTime;

        // Calculate differences between start and end points for slots and cumulative values
        uint64 slotDiff = twapPriceInfoEnd.publishSlot -
            twapPriceInfoStart.publishSlot;
        int128 priceDiff = twapPriceInfoEnd.cumulativePrice -
            twapPriceInfoStart.cumulativePrice;
        uint128 confDiff = twapPriceInfoEnd.cumulativeConf -
            twapPriceInfoStart.cumulativeConf;

        // Calculate time-weighted average price (TWAP) and confidence by dividing
        // the difference in cumulative values by the number of slots between data points
        int128 twapPrice = priceDiff / int128(uint128(slotDiff));
        uint128 twapConf = confDiff / uint128(slotDiff);

        // The conversion from int128 to int64 is safe because:
        // 1. Individual prices fit within int64 by protocol design
        // 2. TWAP is essentially an average price over time (cumulativePrice₂-cumulativePrice₁)/slotDiff
        // 3. This average must be within the range of individual prices that went into the calculation
        // We use int128 only as an intermediate type to safely handle cumulative sums
        twapPriceFeed.twap.price = int64(twapPrice);
        twapPriceFeed.twap.conf = uint64(twapConf);
        twapPriceFeed.twap.expo = twapPriceInfoStart.expo;
        twapPriceFeed.twap.publishTime = twapPriceInfoEnd.publishTime;

        // Calculate downSlotsRatio as a value between 0 and 1,000,000
        // 0 means no slots were missed, 1,000,000 means all slots were missed
        uint64 totalDownSlots = twapPriceInfoEnd.numDownSlots -
            twapPriceInfoStart.numDownSlots;
        uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff;

        // Safely downcast to uint32 (sufficient for value range 0-1,000,000)
        twapPriceFeed.downSlotsRatio = uint32(downSlotsRatio);

        return twapPriceFeed;
    }
