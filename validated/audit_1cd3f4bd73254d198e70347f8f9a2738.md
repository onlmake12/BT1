### Title
`BlockAssembler::update_uncles()` Prepares Uncle Blocks Without Snapshot Staleness Check - (File: `tx-pool/src/block_assembler/mod.rs`)

---

### Summary

`BlockAssembler::update_uncles()` prepares uncle blocks for the current block template using `current.snapshot` without verifying that this snapshot still reflects the current chain tip. Every other template-update function in the same struct performs this staleness guard, but `update_uncles()` silently omits it. A miner's block template can therefore contain uncle references that are invalid for the actual chain tip, causing submitted blocks to be rejected.

---

### Finding Description

`BlockAssembler` maintains a `CurrentTemplate` that holds a cached `Arc<Snapshot>`. When a new block is committed, a `BlockAssemblerMessage::Reset(snapshot)` message is enqueued, which calls `update_blank()` to replace the cached snapshot. Between the commit event and the processing of that `Reset` message, the cached snapshot is stale.

All three other incremental update functions guard against this window:

- `update_full()` — checks `current.snapshot.tip_hash() != tx_pool_reader.snapshot().tip_hash()` and returns early if stale.
- `update_proposals()` — same guard.
- `update_transactions()` — same guard.

`update_uncles()` has no such guard:

```rust
pub(crate) async fn update_uncles(&self) {
    let mut current = self.current.lock().await;
    // ... no tip_hash staleness check ...
    let uncles = self.prepare_uncles(&current.snapshot, &current.epoch).await;
    // proceeds unconditionally with potentially stale snapshot
```

`prepare_uncles()` uses the snapshot to evaluate uncle validity via `snapshot.is_main_chain()` and `snapshot.is_uncle()`. With a stale snapshot, a block that has since been incorporated into the main chain may still appear as a valid uncle candidate, and a block whose parent is no longer reachable may pass the parent-chain check.

---

### Impact Explanation

A miner calling `get_block_template` RPC receives a template containing uncle references that are invalid under the current chain tip. When the miner submits the solved block, consensus verification rejects it (uncle validity is checked during block verification). The miner's proof-of-work is wasted. Under rapid block production or high uncle rates, this window is wider and the probability of a stale-uncle template being served increases. The impact is **medium**: it degrades miner efficiency and can cause repeated block rejections without any direct fund loss, but it is a correctness failure in a security-sensitive path (block assembly).

---

### Likelihood Explanation

The race window exists whenever a `BlockAssemblerMessage::Uncle` message is enqueued and processed before the corresponding `BlockAssemblerMessage::Reset` message that follows a new block commit. Uncle messages are triggered by P2P uncle relay, which any connected peer can initiate. Under normal mainnet conditions with non-trivial uncle rates, this race is regularly reachable. Likelihood is **medium**.

---

### Recommendation

Add the same staleness guard used by the other update functions. Since `update_uncles()` does not currently accept a `tx_pool` reference, the simplest fix is to pass `&RwLock<TxPool>` as a parameter (matching the signature of `update_proposals` and `update_transactions`) and add the check:

```rust
pub(crate) async fn update_uncles(&self, tx_pool: &RwLock<TxPool>) {
    let mut current = self.current.lock().await;
    {
        let tx_pool_reader = tx_pool.read().await;
        if current.snapshot.tip_hash() != tx_pool_reader.snapshot().tip_hash() {
            return;
        }
    }
    // ... rest of function unchanged
```

Alternatively, store the latest snapshot in a shared atomic reference that `update_uncles()` can read without taking the tx_pool lock.

---

### Proof of Concept

1. Node is at tip N. `CurrentTemplate.snapshot` = snapshot_N.
2. A peer relays an uncle block U (valid under snapshot_N). The chain service enqueues `BlockAssemblerMessage::Uncle`.
3. Before the `Uncle` message is processed, a new block N+1 is committed. The chain service enqueues `BlockAssemblerMessage::Reset(snapshot_N+1)`. Block U is now part of the main chain at height N+1.
4. The `Uncle` message is processed first (FIFO queue, but `Uncle` was enqueued before `Reset`). `update_uncles()` calls `prepare_uncles(&current.snapshot /* = snapshot_N */, ...)`.
5. `snapshot_N.is_main_chain(&U.hash())` returns `false` (U was not yet on the main chain at snapshot_N), so U passes the uncle validity filter and is added to the template.
6. `Reset` is processed; `update_blank()` replaces the snapshot. But the uncle U is already embedded in the template's uncle list from step 5.
7. Miner calls `get_block_template`, receives the template with U as an uncle.
8. Miner solves the block and submits it. Block verification checks uncle validity against the current chain state (snapshot_N+1), where U is already on the main chain — the block is rejected.

    pub(crate) async fn update_uncles(&self) {
        let mut current = self.current.lock().await;
        let consensus = current.snapshot.consensus();
        let max_block_bytes = consensus.max_block_bytes() as usize;
        let max_uncles_num = consensus.max_uncles_num();
        let current_uncles_num = current.template.uncles.len();
        if current_uncles_num < max_uncles_num {
            let remain_size = max_block_bytes.saturating_sub(current.size.total);

            if remain_size > UncleBlockView::serialized_size_in_block() {
                let uncles = self.prepare_uncles(&current.snapshot, &current.epoch).await;

                let new_uncle_size = uncles.len() * UncleBlockView::serialized_size_in_block();
                let new_total_size = current.size.calc_total_by_uncles(new_uncle_size);

                if new_total_size < max_block_bytes {
                    let mut builder = BlockTemplateBuilder::from_template(&current.template);
                    builder
                        .set_uncles(uncles)
                        .work_id(self.work_id.fetch_add(1, Ordering::SeqCst))
                        .current_time(cmp::max(
                            unix_time_as_millis(),
                            current.template.current_time,
                        ));
                    current.template = builder.build();
                    current.size.uncles = new_uncle_size;
                    current.size.total = new_total_size;

                    trace!(
                        "[BlockAssembler] update_uncles-{} epoch-{} uncles-{} proposals-{} txs-{}",
                        current.template.number,
                        current.template.epoch.number(),
                        current.template.uncles.len(),
                        current.template.proposals.len(),
                        current.template.transactions.len(),
                    );
                }
            }
        }
    }
