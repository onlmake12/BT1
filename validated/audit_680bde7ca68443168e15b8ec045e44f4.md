### Title
Permanent Freezing of All Protocol Update Fees — No Extraction Path Exists for `FEE_STORAGE_KEY` Coins — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary

`deposit_fee_coins` is declared `public` (not `public(friend)`), and there is no corresponding withdrawal or transfer function anywhere in the module tree. Every SUI fee deposited into a `PriceInfoObject`'s `FEE_STORAGE_KEY` dynamic field — whether via the normal `update_single_price_feed` flow or by a direct arbitrary call — is permanently frozen. The `fee_recipient_address` stored in `State` is vestigial and never used to extract coins.

### Finding Description

**1. `deposit_fee_coins` is unrestricted public:** [1](#0-0) 

The function is declared `public fun deposit_fee_coins(...)`, not `public(friend)`. Any unprivileged transaction sender can call it directly on any shared `PriceInfoObject` with an arbitrary `Coin<SUI>` amount.

**2. Normal fee collection path also deposits into the same frozen field:** [2](#0-1) 

Every legitimate call to `update_single_price_feed` asserts a minimum fee and then calls `price_info::deposit_fee_coins`, accumulating SUI in `FEE_STORAGE_KEY`.

**3. No withdrawal function exists anywhere:**

A search across all `.move` files under `target_chains/sui/contracts/sources/` for `withdraw`, `dynamic_object_field::remove.*FEE`, `coin::split`, and `transfer.*fee` returns zero hits in `price_info.move`, `state.move`, `pyth.move`, and all `governance/*.move` files. There is no function that removes or transfers `Coin<SUI>` from `FEE_STORAGE_KEY`.

**4. `fee_recipient_address` in `State` is explicitly acknowledged as unused:** [3](#0-2) 

The comment reads: *"The previous version of the contract sent the fees to a recipient address but this state is not used anymore."* The field is stored in `State`: [4](#0-3) 

but `get_fee_recipient` is a read-only getter: [5](#0-4) 

It is never used to route coins out of any `PriceInfoObject`.

**5. Governance actions provide no fee-drain path:** [6](#0-5) 

The six defined governance actions are: `CONTRACT_UPGRADE`, `SET_GOVERNANCE_DATA_SOURCE`, `SET_DATA_SOURCES`, `SET_UPDATE_FEE`, `SET_STALE_PRICE_THRESHOLD`, `SET_FEE_RECIPIENT`. None involve transferring or withdrawing `Coin<SUI>` from `PriceInfoObject`s.

### Impact Explanation

Every SUI fee paid by every user calling `update_single_price_feed` is permanently locked inside the corresponding `PriceInfoObject`. The protocol has no on-chain mechanism to recover these funds. Additionally, any unprivileged actor can call `deposit_fee_coins` directly to add arbitrary SUI to the frozen pool (burning their own funds, but also permanently removing them from circulation). The total locked amount grows monotonically with no drain path, constituting permanent freezing of all accumulated update fees (unclaimed yield).

### Likelihood Explanation

This is not a theoretical risk. The fee accumulation happens on every single price feed update. The `deposit_fee_coins` function is already being called in production via `update_single_price_feed`. The absence of a withdrawal path is a structural property of the current code, not a configuration issue.

### Recommendation

1. Change `deposit_fee_coins` from `public fun` to `public(friend) fun` to prevent arbitrary external deposits.
2. Implement a privileged `withdraw_fees(price_info_object: &mut PriceInfoObject, ctx: &mut TxContext): Coin<SUI>` function (gated by `LatestOnly` or a governance capability) that removes the `FEE_STORAGE_KEY` dynamic field and transfers the coins to `state.fee_recipient_address`.
3. Alternatively, route fees directly to `fee_recipient_address` at deposit time using `sui::transfer::public_transfer`, eliminating the need for a separate withdrawal step.

### Proof of Concept

```move
// Attacker calls deposit_fee_coins directly — no privilege required
let attacker_coin = coin::mint_for_testing<SUI>(1_000_000, ctx);
price_info::deposit_fee_coins(&mut price_info_object, attacker_coin);
// Coins are now permanently locked; get_balance grows, no drain path exists

// Verify no public function in any module can extract them:
// grep -r "FEE_STORAGE_KEY" sources/ -> only deposit/read paths, zero remove/transfer paths
assert!(price_info::get_balance(&price_info_object) == 1_000_000, 0);
// No function exists to bring this back to 0
```

        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
