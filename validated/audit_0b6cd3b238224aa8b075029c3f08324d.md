### Title
Orphan Pool Admits Unlimited Double-Spending Transactions Without Aggregate Cell-Consumption Check — (File: `tx-pool/src/component/orphan.rs`)

---

### Summary

`OrphanPool::add_orphan_tx` admits every orphan transaction that has a unique proposal short ID, with no check for whether multiple orphans collectively double-spend the same parent output cell. An unprivileged peer or RPC caller can flood the orphan pool with arbitrarily many transactions that each individually appear valid (their parent is simply unknown) but collectively can only ever produce one valid execution. This mirrors the op-geth sponsor-balance bug exactly: individual admission passes, aggregate resource accounting is absent, and the pool fills with entries that are mutually exclusive at execution time.

---

### Finding Description

`OrphanPool::add_orphan_tx` in `tx-pool/src/component/orphan.rs` performs only one admission guard:

```rust
if self.entries.contains_key(&tx.proposal_short_id()) {
    return vec![];
}
``` [1](#0-0) 

After that single duplicate-ID check, the transaction is unconditionally inserted and the pool is trimmed to `DEFAULT_MAX_ORPHAN_TRANSACTIONS` by random eviction:

```rust
while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
    let id = self.entries.keys().next().cloned().expect("bound checked");
    if let Some(entry) = self.remove_orphan_tx(&id) {
        evicted_txs.push(entry.tx.hash());
    }
}
``` [2](#0-1) 

There is no check that the incoming orphan's inputs conflict with any already-stored orphan's inputs. The `by_out_point` index maps each input `OutPoint` to a *set* of orphan IDs, explicitly supporting multiple orphans that reference the same parent output:

```rust
for out_point in tx.input_pts_iter() {
    self.by_out_point
        .entry(out_point)
        .or_default()
        .insert(tx.proposal_short_id());
}
``` [3](#0-2) 

The codebase's own unit test confirms this is the actual runtime behavior:

```rust
fn test_orphan_allows_double_spends_of_unknown_input() {
    let tx1 = build_tx(vec![(&parent_hash, 0)], 1);
    let tx2 = build_tx(vec![(&parent_hash, 0)], 2);
    orphan.add_orphan_tx(tx1.clone(), 0.into(), 0);
    orphan.add_orphan_tx(tx2.clone(), 0.into(), 0);
    assert_eq!(orphan.len(), 2);   // both admitted
``` [4](#0-3) 

An integration-level spec (`TxPoolOrphanDoubleSpend`) further confirms that two relay-submitted orphans spending the same cell index of the same parent are both accepted into the orphan pool:

```rust
let tx11 = node0.new_transaction_with_capacity_and_index(tx1.hash(), ..., 0, 0);
let tx12 = node0.new_transaction_with_capacity_and_index(tx1.hash(), ..., 0, 0);
assert!(run_replay_tx(&net, node0, tx11, 1, 0), "tx11 in orphan pool");
assert!(run_replay_tx(&net, node0, tx12, 2, 0), "tx12 in orphan pool");
``` [5](#0-4) 

The shared resource here is the parent transaction's output cell (a CKB live cell). Each orphan individually passes admission because the parent is absent from the pool and the snapshot, so no conflict is detectable at that moment. But only one of the N orphans can ever be valid — they are mutually exclusive consumers of the same cell.

---

### Impact Explanation

**Orphan pool saturation / eviction of legitimate orphans.** An attacker who controls a single UTXO (or fabricates a plausible parent hash) can generate `DEFAULT_MAX_ORPHAN_TRANSACTIONS` distinct orphan transactions all spending the same output, submit them via the P2P relay or the `send_transaction` RPC, and fill the orphan pool entirely with mutually-exclusive, collectively-invalid entries. Legitimate orphan transactions from honest users are then evicted by the random-eviction path. When the parent is eventually submitted, `process_orphan_tx` iterates all matching orphans:

```rust
let orphans = self.find_orphan_by_previous(&previous).await;
for orphan in orphans.into_iter() {
    // _process_tx called for each — only the first succeeds
``` [6](#0-5) 

The node wastes verification cycles on N−1 orphans that will all fail with a dead-cell conflict, and honest users whose orphan transactions were evicted must resubmit them. The main `TxPool` (pending/proposed) is not directly corrupted, but the orphan subsystem — which is the only mechanism for handling out-of-order transaction relay — is rendered unreliable for the duration of the attack.

---

### Likelihood Explanation

The attack requires only an unprivileged transaction sender or P2P peer. No keys, no hashpower, no Sybil capability is needed beyond the ability to craft transactions. The attacker needs one real or synthetic parent `OutPoint` (even a non-existent one works, since orphan admission does not verify the parent exists) and the ability to produce `DEFAULT_MAX_ORPHAN_TRANSACTIONS` distinct child transactions spending the same output (varying witnesses or other fields to produce distinct tx hashes). This is trivially scriptable. The cost is negligible: no fees are paid because the transactions never reach the main pool.

---

### Recommendation

Before inserting an orphan, check whether any of its input `OutPoint`s already appear in `by_out_point` with a non-empty set of existing orphan IDs. If a conflict is found, either reject the incoming orphan or apply a fee-rate comparison and evict the lower-fee conflicting orphan (analogous to RBF in the main pool). This mirrors the fix applied in op-geth PR #43: compute the aggregate resource consumption (here: the set of cells already claimed by orphans) and reject new entries that would exceed what the shared resource can satisfy.

---

### Proof of Concept

1. Pick any 32-byte value as a synthetic parent tx hash `P`.
2. For `i` in `1..DEFAULT_MAX_ORPHAN_TRANSACTIONS+1`, construct transaction `Ti`:
   - Single input: `OutPoint { tx_hash: P, index: 0 }` (same cell for every `Ti`)
   - Vary the witness bytes so each `Ti` has a distinct tx hash and proposal short ID.
3. Submit each `Ti` via `send_transaction` RPC or P2P relay.
4. Observe via `get_raw_tx_pool` that the orphan count reaches the maximum and legitimate orphan transactions submitted by honest users are evicted.
5. Submit the real parent transaction `P` (or any transaction whose hash matches the synthetic `P`). Observe that `process_orphan_tx` is invoked for all `N` orphans, N−1 fail with a dead-cell error, and the node has wasted N−1 full verification cycles.
