Audit Report

## Title
Missing Deduplication in `GetTransactionsProofProcess::execute` Allows Per-Request DB Lookup Amplification — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`GetTransactionsProofProcess::execute` performs no deduplication on the incoming `tx_hashes` list. An attacker can send 1000 copies of a single known tx hash, triggering 1000 `get_transaction_info` calls and 1000 `get_transaction_with_info` calls, plus a `CBMT::build_merkle_proof` invocation over 1000 duplicate indices — all per message. Because the handler returns `OK` (200), `should_ban` never fires, and the attacker can repeat indefinitely with no cost.

## Finding Description

**Root cause:** No `HashSet` deduplication guard exists between the length check and the `partition` loop in `get_transactions_proof.rs`.

**Code path:**

1. The length check at L37–39 passes for exactly 1000 identical hashes: [1](#0-0) 

2. All 1000 hashes (including duplicates) are iterated in the `partition`, each calling `snapshot.get_transaction_info(tx_hash)`: [2](#0-1) 

3. All 1000 entries in `found` are iterated, each calling `snapshot.get_transaction_with_info(&tx_hash)`. Since all 1000 hashes are identical, all entries land in the same `txs_in_blocks` bucket, and `CBMT::build_merkle_proof` is called with 1000 duplicate indices: [3](#0-2) [4](#0-3) 

4. By contrast, `GetBlocksProofProcess::execute` explicitly deduplicates and returns a ban-triggering 400: [5](#0-4) 

5. `should_ban` only triggers on 4xx codes. Since `GetTransactionsProofProcess` returns `OK` (200) on success, no ban is issued: [6](#0-5) 

6. The limit constant confirms 1000 is the maximum: [7](#0-6) 

## Impact Explanation

A single malicious light-client peer can send repeated `GetTransactionsProof` messages each containing 1000 copies of a known tx hash. Each message triggers 2000 DB lookups plus a heavy Merkle proof computation over 1000 duplicate indices, versus the expected 2 lookups and trivial proof for 1 unique tx — a 1000× amplification factor. Because no ban is issued, the peer can repeat indefinitely, exhausting I/O and CPU on the node running the light-client server. This matches **High — Vulnerabilities or bad designs which could cause CKB network congestion / crash a CKB node with few costs.**

## Likelihood Explanation

The entrypoint is the P2P light-client protocol, reachable by any unprivileged peer with no PoW, key, or privileged role required. The asymmetry with `GetBlocksProofProcess` (which has the fix) strongly indicates an unintentional omission. The attack is trivially repeatable with no cost to the attacker.

## Recommendation

Add a `HashSet` deduplication check in `GetTransactionsProofProcess::execute` immediately after the length check, mirroring the pattern in `GetBlocksProofProcess::execute`:

```rust
let mut uniq = HashSet::new();
if !tx_hashes_vec.iter().all(|h| uniq.insert(h)) {
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate tx hash exists");
}
```

This returns a 4xx status, which `should_ban` will convert into a peer ban.

## Proof of Concept

Send a `GetTransactionsProof` P2P message with `tx_hashes = [known_tx_hash; 1000]` and a valid `last_hash` pointing to a main-chain block. Instrument `snapshot.get_transaction_info` and `snapshot.get_transaction_with_info` with call counters. Assert both counters reach 1000 rather than 1. Confirm the peer is not banned after the response (status is `OK`). Repeat the message in a loop and observe sustained DB I/O amplification with no ban applied.
