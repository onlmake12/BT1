### Title
Indexer Key Namespace Collision via Variable-Length `args` Concatenation Without Length Encoding — (`util/indexer/src/indexer.rs`)

---

### Summary

The CKB built-in indexer constructs RocksDB storage keys by concatenating `code_hash || hash_type || args` (via `extract_raw_data`) without encoding the length of the variable-length `args` field. Because `args` is user-controlled and length-unbounded, a crafted RPC query can produce a search prefix that is byte-identical to the stored key prefix of a *different* script (one with shorter `args`), causing the prefix scan to return cells and transactions belonging to the wrong script. This is a direct analog to the RocketPool settings-key collision: user-controlled input appended without a delimiter causes one namespace to bleed into another.

---

### Finding Description

`extract_raw_data` in `util/indexer/src/indexer.rs` produces the raw key material for every script-indexed entry:

```rust
pub fn extract_raw_data(script: &Script) -> Vec<u8> {
    [
        script.code_hash().as_slice(),   // always 32 bytes
        script.hash_type().as_slice(),   // always 1 byte
        &script.args().raw_data(),       // VARIABLE, user-controlled
    ]
    .concat()
}
``` [1](#0-0) 

`append_key` then appends fixed-width fields after the variable-length `args`:

```
[prefix_byte(1)] [code_hash(32)] [hash_type(1)] [args(N)] [block_number(8)] [tx_index(4)] [io_index(4)]
``` [2](#0-1) 

Because `args` has no length prefix or delimiter, the byte sequence `args_A || block_number_A` for Script A with shorter args is indistinguishable from `args_B` for Script B whose args happen to equal `args_A || block_number_A_bytes`.

`build_query_options` builds the search prefix the same way:

```rust
prefix.extend_from_slice(extract_raw_data(&script).as_slice());
``` [3](#0-2) 

The prefix scan then uses `key.starts_with(&prefix)` to collect results:

```rust
.take_while(|(key, _value)| key.starts_with(&prefix))
``` [4](#0-3) 

**Concrete collision example:**

| | Script A | Script B (attacker query) |
|---|---|---|
| `code_hash` | `X` | `X` |
| `hash_type` | `H` | `H` |
| `args` | `[]` (empty) | `[0x00,0x00,0x00,0x00,0x00,0x00,0x00,0x01]` (8 bytes) |

Stored key for Script A at `block_number=1, tx_index=0, output_index=0`:
```
[prefix][X(32)][H(1)][0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x01][0x00 0x00 0x00 0x00][0x00 0x00 0x00 0x00]
```

Query prefix for Script B (args = 8 bytes encoding of `1`):
```
[prefix][X(32)][H(1)][0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x01]
```

The query prefix for Script B is a byte-for-byte prefix of Script A's stored key. `key.starts_with(&prefix)` returns `true`, so Script A's cells are returned in the response for Script B.

The same collision applies to `get_transactions` and `get_cells_capacity`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

An unprivileged RPC caller can query `get_cells`, `get_transactions`, or `get_cells_capacity` with a crafted `IndexerSearchKey` whose `args` equals the big-endian encoding of a target block number. The indexer returns cells and transactions belonging to a *different* script (one with shorter args) as if they matched the queried script. This produces false-positive results that:

- Cause wallet or dApp clients to display incorrect live-cell sets and balances for a script
- Corrupt capacity calculations returned by `get_cells_capacity`
- Mislead any application that relies on the indexer for UTXO enumeration or script-based accounting

The `Exact` search mode (`script_search_mode: "exact"`) does mitigate this by checking `key.len() != prefix.len() + 16`, but this mode is **opt-in**; the default is prefix search, which is the vulnerable path. [7](#0-6) 

---

### Likelihood Explanation

The collision is trivially constructable. Block number 1 is always present on any live chain. Any script with empty `args` that has cells at block 1 is a collision target: an attacker queries with `args = [0x00,0x00,0x00,0x00,0x00,0x00,0x00,0x01]` against the same `code_hash`/`hash_type`. No special privileges, no key material, and no on-chain transactions are required from the attacker — only an RPC call to a node running the built-in indexer.

---

### Recommendation

Encode the length of `args` before appending it in `extract_raw_data` (or in `append_key`), so that the key encoding is unambiguous:

```rust
fn append_key(encoded: &mut Vec<u8>, script: &Script, ...) {
    let args = script.args().raw_data();
    encoded.extend_from_slice(script.code_hash().as_slice());
    encoded.extend_from_slice(script.hash_type().as_slice());
    // Prepend args length as a fixed-width field to prevent ambiguity
    encoded.extend_from_slice(&(args.len() as u32).to_be_bytes());
    encoded.extend_from_slice(&args);
    encoded.extend_from_slice(&block_number.to_be_bytes());
    encoded.extend_from_slice(&tx_index.to_be_bytes());
    encoded.extend_from_slice(&io_index.to_be_bytes());
}
```

This requires a database migration for existing indexer stores. Alternatively, enforce that `Exact` mode is the default for all queries, and document that prefix mode is only safe when the caller understands the ambiguity.

---

### Proof of Concept

1. Start a CKB node with the built-in indexer enabled.
2. Identify any script `S_A` with `code_hash=X`, `hash_type=H`, `args=[]` that has live cells at `block_number=1`.
3. Call `get_cells` via RPC with:
   ```json
   {
     "search_key": {
       "script": {
         "code_hash": "<X>",
         "hash_type": "<H>",
         "args": "0x0000000000000001"
       },
       "script_type": "lock"
     },
     "order": "asc",
     "limit": "0x64"
   }
   ```
4. Observe that the response includes cells whose actual lock script has `args=0x` (empty), not `args=0x0000000000000001`. The indexer has returned cells belonging to Script A in response to a query for Script B. [8](#0-7) [9](#0-8)

        IndexerScriptType::Type => vec![type_prefix as u8],
