### Title
Stale `proposedAdmin` Not Cleared on Ownership Transfer Allows Unauthorized Admin Takeover — (`EntropyGovernance.sol` / `SchedulerGovernance.sol`)

---

### Summary

Both `EntropyGovernance` and `SchedulerGovernance` implement a two-step admin transfer pattern via `proposeAdmin()` / `acceptAdmin()`. When contract ownership is transferred via OpenZeppelin's `Ownable2Step` (`transferOwnership` / `acceptOwnership`), the `_state.proposedAdmin` value is **never cleared**. A previously proposed admin — proposed by the old owner or old admin — can call `acceptAdmin()` after the ownership transfer and seize admin control without the new owner's knowledge or consent.

---

### Finding Description

`EntropyGovernance.proposeAdmin()` sets `_state.proposedAdmin` and can be called by either the current owner **or** the current admin:

```solidity
function proposeAdmin(address newAdmin) public virtual {
    require(newAdmin != address(0), "newAdmin is zero address");
    _authoriseAdminAction();          // passes for owner OR admin
    _state.proposedAdmin = newAdmin;
    emit NewAdminProposed(_state.admin, newAdmin);
}
```

`acceptAdmin()` clears `_state.proposedAdmin` only when the proposed party accepts:

```solidity
function acceptAdmin() external {
    if (msg.sender != _state.proposedAdmin) revert EntropyErrors.Unauthorized();
    address oldAdmin = _state.admin;
    _state.admin = msg.sender;
    _state.proposedAdmin = address(0);   // cleared here only
    emit NewAdminAccepted(oldAdmin, msg.sender);
}
```

The OZ `Ownable2Step` `transferOwnership` / `acceptOwnership` path **never touches** `_state.proposedAdmin`. Therefore the following attack sequence is possible:

1. Old admin (or old owner) calls `proposeAdmin(X)` — sets `_state.proposedAdmin = X`.
2. Old owner calls `transferOwnership(new_owner)` — `_state.proposedAdmin` is **not cleared**.
3. New owner calls `acceptOwnership()` — `_state.proposedAdmin` is **still X**.
4. X calls `acceptAdmin()` — becomes admin without the new owner's knowledge or consent.

There is no `cancelAdminProposal()` function. The only way to overwrite the stale value is to call `proposeAdmin()` again with a non-zero address, but `proposeAdmin(address(0))` is explicitly blocked:

```solidity
require(newAdmin != address(0), "newAdmin is zero address");
```

So the new owner cannot zero out the stale proposal directly; they can only overwrite it — but only if they are aware it exists.

The identical pattern exists in `SchedulerGovernance.sol`.

---

### Impact Explanation

Once X calls `acceptAdmin()`, they hold the `admin` role. As admin, X can:

- Call `setPythFee()` — manipulate protocol fee collection
- Call `setDefaultProvider()` — redirect all default-provider randomness requests
- Call `withdrawFee()` — drain all accrued Pyth protocol fees from the contract
- Call `proposeAdmin()` — propose yet another admin, perpetuating control

This constitutes unauthorized control over critical Entropy (and Scheduler/Pulse) contract parameters and treasury.

---

### Likelihood Explanation

The scenario is realistic during governance transitions (e.g., multisig rotation, protocol upgrade). The old admin may have legitimately proposed a new admin that had not yet accepted. If ownership is transferred before acceptance, the new owner inherits the stale proposal with no direct way to cancel it. The new owner is not notified of the pending proposal on-chain. The proposed party (X) can monitor the chain and call `acceptAdmin()` at any time after the ownership transfer completes.

---

### Recommendation

Clear `_state.proposedAdmin` whenever ownership changes. Override `_transferOwnership` in `EntropyUpgradeable` and `SchedulerUpgradeable`:

```solidity
function _transferOwnership(address newOwner) internal override {
    _state.proposedAdmin = address(0);
    super._transferOwnership(newOwner);
}
```

Alternatively, add a `cancelAdminProposal()` function callable by the owner, and document that it must be called before any ownership transfer.

---

### Proof of Concept

```solidity
// 1. Setup: old owner proposes attacker as admin
vm.prank(oldOwner);
entropy.proposeAdmin(attacker);
assertEq(entropy.proposedAdmin(), attacker);

// 2. Old owner transfers ownership to new owner
vm.prank(oldOwner);
entropy.transferOwnership(newOwner);
vm.prank(newOwner);
entropy.acceptOwnership();
assertEq(entropy.owner(), newOwner);

// 3. proposedAdmin is NOT cleared — stale state persists
assertEq(entropy.proposedAdmin(), attacker); // still attacker!

// 4. Attacker accepts admin without new owner's consent
vm.prank(attacker);
entropy.acceptAdmin();
assertEq(entropy.getAdmin(), attacker); // attacker is now admin

// 5. Attacker drains fees
vm.prank(attacker);
entropy.withdrawFee(attacker, entropy.getAccruedPythFees());
```
