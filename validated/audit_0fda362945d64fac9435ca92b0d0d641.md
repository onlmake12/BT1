### Title
DAO Withdrawal Header-Dep Index Value Mismatch Between Rust Verifier and C On-Chain Script — (`File: util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator` in `util/dao/src/lib.rs` reads the deposit-header index from the witness `input_type` field as a full 8-byte little-endian `u64`. The deployed C DAO script (`dao.c`) reads the same field using only its **lowest byte** (effectively a `u8`). When a transaction encodes an index value greater than 255 (e.g., `257 = 0x0000000000000101`), the two layers resolve to **different entries** in `header_deps`, producing a consensus split: the C script accepts the transaction while the Rust node rejects it.

---

### Finding Description

In `transaction_maximum_withdraw`, the witness `input_type` field is decoded as a full `u64`:

```rust
// util/dao/src/lib.rs  line 91
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

That value is then used directly as an array index into `header_deps`:

```rust
// util/dao/src/lib.rs  lines 94-98
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
    .and_then(|hash| header_deps.get(&hash))
    .ok_or(DaoError::InvalidOutPoint)
``` [1](#0-0) 

The C DAO script, however, interprets the same 8-byte field by reading only its lowest byte. For an `input_type` value of `257` (little-endian bytes `[0x01, 0x01, 0x00, …]`):

- **C script** reads byte `0x01` → resolves `header_deps[1]` (the deposit block).
- **Rust code** reads `u64 = 257` → resolves `header_deps[257]` (a different block, e.g., the withdraw block).

The Rust code then performs a block-number consistency check:

```rust
// util/dao/src/lib.rs  line 105-107
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
``` [2](#0-1) 

Because `header_deps[257]` is the withdraw block (number 200) while the cell data records the deposit block number (100), the check fails and the Rust node rejects the transaction. The C script, resolving to `header_deps[1]` (the correct deposit block), would accept it.

The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this divergence:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
// Rust resolves index 257 → withdraw block (number 200), but cell data
// says deposited at block 100. Block number check catches the mismatch.
assert!(result.is_err(), "expected Err, got {result:?}");
``` [3](#0-2) 

---

### Impact Explanation

`DaoCalculator::transaction_fee` is invoked during tx-pool admission to compute the fee for DAO withdrawal transactions. When it returns `Err`, the node rejects the transaction from the mempool. Because the C DAO script (the authoritative on-chain validator) would accept the same transaction, the result is:

1. **Consensus split**: the on-chain script accepts the transaction; the Rust node rejects it.
2. **Mempool denial**: a valid DAO withdrawal with `input_type` index > 255 cannot be submitted through any standard CKB RPC endpoint.
3. **Fund lock-up**: a depositor who constructs (or is tricked into constructing) such a transaction cannot withdraw their NervosDAO funds through the normal path, even though the transaction is protocol-valid.

---

### Likelihood Explanation

The attack surface is reachable by any **RPC caller** or **tx-pool submitter** (unprivileged roles). A transaction sender needs only to:

1. Hold a NervosDAO deposit cell.
2. Construct a withdrawal transaction with ≥ 258 `header_deps` entries.
3. Place the deposit block hash at position 1 and set `input_type` to `257` (u64 LE).

No privileged access, key material, or majority hashpower is required. The transaction is structurally valid and passes all other checks; only the Rust index-resolution diverges from the C script.

---

### Recommendation

In `transaction_maximum_withdraw` (`util/dao/src/lib.rs`), truncate the decoded index to its lowest byte before using it as a `header_deps` offset, matching the C DAO script's behavior:

```rust
// Align with dao.c: only the lowest byte of the 8-byte field is the index.
let header_dep_index = LittleEndian::read_u64(&header_deps_index_data.unwrap()) as u8 as usize;
```

Alternatively, add an explicit validation that rejects any `input_type` value whose upper 7 bytes are non-zero, so that the Rust node and the C script are guaranteed to agree on the resolved index.

---

### Proof of Concept

1. Construct a DAO withdrawal `ResolvedTransaction` with 258 `header_deps`:
   - `header_deps[1]` = deposit block hash (block number 100).
   - `header_deps[257]` = withdraw block hash (block number 200).
2. Set cell data to `100u64` (deposit block number).
3. Set `WitnessArgs.input_type` to `257u64` in little-endian (bytes `[0x01, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`).
4. Call `DaoCalculator::transaction_fee(&rtx)`.

**Expected (C script behavior):** index = lowest byte = 1 → `header_deps[1]` = deposit block → number matches cell data → **accept**.

**Actual (Rust behavior):** index = 257 → `header_deps[257]` = withdraw block → number 200 ≠ cell data 100 → `Err(DaoError::InvalidOutPoint)` → **reject**.

This is confirmed by the existing test at `util/dao/src/tests.rs:475–537`. [4](#0-3) [5](#0-4)

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
