### Title
Fee Permanently Locked in `PriceInfoObject` When Update `publish_time` Equals Cached Timestamp — (`target_chains/sui/contracts/sources/pyth.move`)

---

### Summary

In `update_single_price_feed`, the update fee is deposited into the `PriceInfoObject` **unconditionally** before the freshness check. Because `is_fresh_update` uses a strict `>` comparison, a price update whose `publish_time` exactly equals the cached timestamp is silently discarded — but the fee has already been taken and is permanently locked inside the `PriceInfoObject` with no withdrawal path.

---

### Finding Description

**Step 1 — Fee deposited unconditionally.**

`update_single_price_feed` charges the caller and stores the coins in the `PriceInfoObject` before any freshness logic runs:

```
assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);
price_info::deposit_fee_coins(price_info_object, fee);   // ← fee taken here, unconditionally
``` [1](#0-0) 

**Step 2 — Strict `>` in `is_fresh_update`.**

The freshness gate uses strict greater-than, so `update_timestamp == cached_timestamp` returns `false`:

```move
update_timestamp > cached_timestamp
``` [2](#0-1) 

When `is_fresh_update` returns `false`, neither the cache update nor the `PriceFeedUpdateEvent` emission occurs:

```move
if (is_fresh_update(update, price_info_object)){
    pyth_event::emit_price_feed_update(...);
    price_info::update_price_info_object(...);
}
``` [3](#0-2) 

**Step 3 — Fees are permanently locked; no withdrawal path exists.**

`deposit_fee_coins` accumulates SUI into a dynamic object field on the `PriceInfoObject`:

```move
coin::join(current_fee, fee_coins);
``` [4](#0-3) 

There is no `withdraw_fee_coins`, no `collect_fees`, and no transfer-to-recipient function anywhere in the Sui contract sources. The `set_fee_recipient` module itself carries the comment: *"The previous version of the contract sent the fees to a recipient address but this state is not used anymore."* [5](#0-4) 

The `fee_recipient_address` field exists in `State` but is never used to pull funds out of any `PriceInfoObject`. [6](#0-5) 

---

### Impact Explanation

Any caller of `update_single_price_feed` who submits a VAA whose price `publish_time` equals the currently cached timestamp:

- Pays `base_update_fee` SUI (or more).
- Receives no cache update and no `PriceFeedUpdateEvent`.
- Has no on-chain evidence of the payment (no event).
- Cannot recover the fee — it is permanently frozen inside the `PriceInfoObject`.

This matches the scoped impact: **permanent freezing of unclaimed yield**. The locked SUI is inaccessible to the fee recipient, the protocol, and the payer.

---

### Likelihood Explanation

- `base_update_fee > 0` is the normal production configuration.
- Equal-timestamp collisions are realistic: two independent relayers racing to submit the same Pythnet slot, or a retry of a transaction that already landed.
- The attacker needs no privilege — `update_single_price_feed` is a fully public entry point.
- The condition is locally testable with a single unit test (set cached `publish_time = T`, submit update with `publish_time = T`, assert fee deposited and event count unchanged).

---

### Recommendation

Move the fee deposit **after** the freshness check, or refund the fee when `is_fresh_update` returns `false`:

```move
// Option A: deposit only on a fresh update
if (is_fresh_update(latest_only, cur_price_info, price_info_object, clock)) {
    price_info::deposit_fee_coins(price_info_object, fee);
    update_cache(...);
} else {
    sui::transfer::public_transfer(fee, tx_context::sender(ctx));
}
```

Alternatively, change `is_fresh_update` to use `>=` so equal-timestamp updates are accepted (matching the semantics of every other Pyth chain deployment, e.g. Aptos line 395: `update_timestamp > cached_timestamp` is also strict, but Aptos fees go to a module-level balance, not a per-object field with no drain). [7](#0-6)

    }

    fun is_fresh_update(update: &PriceInfo, price_info_object: &PriceInfoObject): bool {
        // Get the timestamp of the update's current price
        let price_feed = price_info::get_price_feed(update);
        let update_timestamp = price::get_timestamp(&price_feed::get_price(price_feed));

        // Get the timestamp of the cached data for the price identifier
        let cached_price_info = price_info::get_price_info_from_price_info_object(price_info_object);
        let cached_price_feed =  price_info::get_price_feed(&cached_price_info);
        let cached_timestamp = price::get_timestamp(&price_feed::get_price(cached_price_feed));

        update_timestamp > cached_timestamp
    }
