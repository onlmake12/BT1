### Title
Out-of-Bounds Index Panic in `TxConfirmStat::remove_unconfirmed_tx` Crashes Tx-Pool Service Thread — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

### Summary
When a tracked transaction remains unconfirmed for more than `MAX_CONFIRM_BLOCKS` (1000) blocks and is then evicted from the tx-pool with `count_failure = true`, the `confirm_blocks_to_failed_txs[tx_age - 1]` index access in `remove_unconfirmed_tx` exceeds the array's length of 1000, causing a Rust index-out-of-bounds panic that crashes the tx-pool service thread.

### Finding Description

`TxConfirmStat` is initialized with both `confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` as vectors of length `MAX_CONFIRM_BLOCKS = 1000`. [1](#0-0) 

In `remove_unconfirmed_tx`, the guard at line 208 correctly redirects the `block_unconfirmed_txs` access to `old_unconfirmed_txs` when `tx_age >= 1000`, but the subsequent `count_failure` branch at line 214–216 has **no corresponding bounds check**: [2](#0-1) 

When `tx_age >= 1001`, `tx_age - 1 >= 1000`, which is out of bounds for a 1000-element vector, causing a panic.

The call chain is:

1. Attacker submits a valid minimum-fee tx via RPC/P2P → `register_pending` callback fires → `fee_estimator.accept_tx(tx_hash, entry_info)` [3](#0-2) 
2. `accept_tx` calls `track_tx(tx_hash, fee_rate, self.current_tip)`, recording `entry_height = current_tip` [4](#0-3) 
3. 1001+ blocks pass; `best_height` advances with each `commit_block` call [5](#0-4) 
4. Tx is evicted (expiry, pool size limit, or reorg) → `register_reject` callback fires → `fee_estimator.reject_tx(&tx_hash)` [6](#0-5) 
5. `reject_tx` → `drop_tx(count_failure=true)` → `drop_tx_inner` → `remove_unconfirmed_tx(entry_height, best_height, bucket_index, true)` [7](#0-6) 
6. `tx_age = best_height - entry_height >= 1001` → `confirm_blocks_to_failed_txs[1000]` → **panic**

### Impact Explanation
A Rust index-out-of-bounds panic unwinds and terminates the tx-pool service thread. The tx-pool is a critical node component; its crash renders the node unable to accept or relay transactions. If the `ConfirmationFraction` estimator is enabled and the node is widely deployed, any unprivileged actor can trigger this crash by submitting a single valid tx and waiting ~2.2+ hours (1001 blocks at 8 s/block), well within the default 12-hour expiry window.

### Likelihood Explanation
The attack requires only a valid transaction submission (no special privileges, no PoW, no key material) and patience. The default tx expiry of 12 hours makes the 1001-block threshold easily reachable in normal operation, even without deliberate attack — a low-fee tx that is never proposed will naturally age past 1000 blocks.

### Recommendation
Add a bounds check before the `confirm_blocks_to_failed_txs` access in `remove_unconfirmed_tx`:

```rust
if count_failure && tx_age - 1 < self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This mirrors the existing guard for `block_unconfirmed_txs` and silently drops failure samples for txs older than the tracking window, which is the correct semantic (they are already counted as `old_unconfirmed_txs`).
