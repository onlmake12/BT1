Audit Report

## Title
Full CKB-VM Execution Before Cycle Mismatch Check Enables Asymmetric CPU DoS via Relay Protocol — (`tx-pool/src/process.rs`)

## Summary

In `_process_tx`, `verify_rtx` (full CKB-VM script execution) is called unconditionally before the `declared_cycles != verified.cycles` check. Because the verify cache is only populated on the success path — after the `DeclaredWrongCycles` early return — every resubmission of the same transaction with `declared_cycles = actual + 1` forces a full VM execution. Combined with `remove_from_known_txs` clearing the tx_hash filter on rejection, an attacker with one valid UTXO and multiple IPs can sustain repeated full-VM occupations of the victim's verify queue at negligible cost.

## Finding Description

The execution order in `_process_tx` is confirmed in `tx-pool/src/process.rs` (L715–749):

1. `pre_check` (L715) — cheap input resolution and fee check
2. `fetch_tx_verify_cache` (L719) — cache lookup by `witness_hash`
3. `max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles())` (L720)
4. **`verify_rtx(...)` with `max_cycles`** (L724–732) — full CKB-VM execution runs here
5. **Only then**: `if declared != verified.cycles` → `Reject::DeclaredWrongCycles` (L736–748)

With `declared_cycles = actual + 1`, `max_cycles` is set to `actual + 1`, which is sufficient for the VM to complete. The VM returns `verified.cycles = actual`, the mismatch is detected, and `DeclaredWrongCycles` is returned. [1](#0-0) 

The cache update at L758–765 is placed **after** the `DeclaredWrongCycles` early return and only on the success path. So the cache is never populated for a `DeclaredWrongCycles` rejection, and every resubmission of the same witness triggers full VM execution. [2](#0-1) 

The cache fast-path (cheap `TimeRelativeTransactionVerifier`) in `tx-pool/src/util.rs` is only taken when `cache_entry` is `Some`. Since the cache is never populated for `DeclaredWrongCycles` rejections, this path is never taken for repeated submissions. [3](#0-2) 

On rejection, `TxVerificationResult::Reject` causes `remove_from_known_txs` to remove the `tx_hash` from the `tx_filter` in `sync/src/relayer/mod.rs` (L673–675), meaning the same `tx_hash` can be re-announced after reconnection since the filter no longer blocks it. [4](#0-3) 

The only pre-VM guard in `TransactionsProcess::execute` is `declared_cycles > max_block_cycles` (immediate ban, no VM). A declared value of `actual + 1` passes this guard trivially. [5](#0-4) 

## Impact Explanation

Each relay submission with `declared_cycles = actual + 1` forces the victim node to run full `ContextualTransactionVerifier` including secp256k1 CKB-VM execution, occupying the verify queue for the full duration. The attacker spends O(1) CPU (send a P2P message); the victim spends O(secp256k1_cycles) CPU. With N IPs, the attacker can sustain N parallel verify-queue occupations, delaying legitimate transaction admission across the network.

This matches the **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

**Preconditions:**
- Attacker must own at least one valid UTXO to pass `pre_check`
- Attacker needs multiple IPs to bypass the 3-day ban per IP triggered on `DeclaredWrongCycles`

**Feasibility:** The UTXO is never consumed (tx is always rejected), so one UTXO suffices indefinitely. Cost scales with number of IPs, not CKB holdings. The existing integration test `DeclaredWrongCycles` at `test/src/specs/tx_pool/declared_wrong_cycles.rs` (L26) uses `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1` (i.e., `actual + 1`) and confirms the tx is rejected after full verification, proving the attack path is reachable. [6](#0-5) 

## Recommendation

Populate `txs_verify_cache` on a `DeclaredWrongCycles` rejection as well as on success. Subsequent resubmissions of the same witness would then take the cheap `TimeRelativeTransactionVerifier` path. Alternatively, move the `declared_cycles` plausibility check before `verify_rtx` — since `max_cycles` is already set to `declared_cycles`, a pre-execution check comparing `declared_cycles` against a fast estimate (or simply rejecting if `declared_cycles` is implausibly small) would prevent full VM execution for obviously wrong declarations.

## Proof of Concept

```
1. Attacker owns UTXO U, creates valid secp256k1-signed tx T (actual cycles = C)
2. Connect to victim node as relay peer
3. Send RelayTransactionHashes([T.hash])
4. Receive GetRelayTransactions([T.hash])
5. Send RelayTransactions([{cycles: C+1, transaction: T}])
   → victim sets max_cycles = C+1, runs full secp256k1 VM (C cycles of CPU)
   → verified.cycles = C, declared = C+1 → DeclaredWrongCycles
   → victim bans attacker IP, removes T.hash from tx_filter
   → cache NOT populated
6. Reconnect from new IP → goto step 3
   (UTXO U is still unspent; T.hash is no longer filtered)
```

Confirmed by integration test `DeclaredWrongCycles` (`test/src/specs/tx_pool/declared_wrong_cycles.rs`, L26) which uses `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1` and asserts the tx is rejected after full verification. [7](#0-6)
