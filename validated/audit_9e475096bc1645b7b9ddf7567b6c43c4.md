### Title
Unprotected `setup()` Can Be Front-Run to Hijack Guardian Set and Implementation - (File: `target_chains/ethereum/contracts/contracts/wormhole-receiver/ReceiverSetup.sol`)

---

### Summary

`ReceiverSetup.setup()` is a `public` function with no access control and no "already initialized" guard. During the deployment window between proxy creation and the `setup()` call, any unprivileged transaction sender can front-run the deployer, inject malicious guardian keys and a malicious implementation address, and take full control of the Pyth wormhole receiver.

---

### Finding Description

The deployment flow for the Pyth wormhole receiver on EVM chains is:

1. Deploy `ReceiverSetup` (initial implementation).
2. Deploy the ERC1967 proxy pointing to `ReceiverSetup`.
3. Call `setup()` on the proxy (delegatecall into `ReceiverSetup`) to configure guardian keys, chain IDs, governance contract, and upgrade to `ReceiverImplementation`.

`ReceiverSetup.setup()` is declared `public` with no caller restriction and no idempotency guard:

```solidity
// target_chains/ethereum/contracts/contracts/wormhole-receiver/ReceiverSetup.sol
contract ReceiverSetup is ReceiverSetters, ERC1967Upgrade {
    function setup(
        address implementation,
        address[] memory initialGuardians,
        uint16 chainId,
        uint16 governanceChainId,
        bytes32 governanceContract
    ) public {
        require(initialGuardians.length > 0, "no guardians specified");
        ...
        _upgradeTo(implementation);
    }
}
``` [1](#0-0) 

If steps 2 and 3 are in separate transactions (as is common in multi-step deployment scripts), an attacker observing the mempool can front-run step 3 and call `setup()` with:
- `initialGuardians` = attacker-controlled keys → the receiver will accept VAAs signed by the attacker.
- `implementation` = attacker-controlled contract → full proxy takeover.
- `governanceContract` = attacker-controlled address → governance actions can be forged.

The identical pattern exists in the vendored `Setup.sol`:

```solidity
// target_chains/ethereum/contracts/contracts/wormhole/Setup.sol
contract Setup is Setters, ERC1967Upgrade {
    function setup(...) public { ... _upgradeTo(implementation); }
}
``` [2](#0-1) 

Compare with contracts that correctly protect initialization — `PythLazer` uses `_disableInitializers()` in its constructor, and `SchedulerUpgradeable` also calls `_disableInitializers()`: [3](#0-2) [4](#0-3) 

`ReceiverSetup` has neither protection.

---

### Impact Explanation

If an attacker successfully front-runs `setup()`:

1. **Fake price updates accepted**: With attacker-controlled guardian keys, the attacker can forge arbitrary VAAs. The `ReceiverImplementation` verifies VAAs against the stored guardian set. Forged VAAs can carry arbitrary Pyth price payloads, allowing the attacker to manipulate any price feed served by this receiver on the target chain.
2. **Full proxy takeover**: By supplying a malicious `implementation` address, the attacker gains control of the proxy's logic, enabling arbitrary state manipulation, fund theft (if the contract holds fees), and permanent denial of service.
3. **Governance hijack**: A malicious `governanceContract` allows the attacker to submit governance actions (e.g., upgrade the contract again, change data sources).

---

### Likelihood Explanation

- **Trigger**: Any unprivileged EOA can call `setup()` on the proxy between proxy deployment and the legitimate `setup()` call.
- **Window**: Exists whenever deployment steps 2 and 3 are in separate transactions. Multi-step deployment scripts (e.g., Hardhat/Foundry scripts that broadcast transactions individually) create this window on every new chain deployment.
- **Detection difficulty**: A front-runner only needs to watch the mempool for a transaction targeting the newly deployed proxy address and submit a competing `setup()` call with higher gas.
- **Likelihood**: Medium — the window is short but the attack is trivially scriptable and the payoff (control of a price oracle) is high.

---

### Recommendation

Add an "already initialized" guard to `setup()`, analogous to OpenZeppelin's `initializer` modifier, or restrict the caller to the deployer (e.g., check `msg.sender == tx.origin` of the deployment transaction, or use a deployer-owned capability pattern). The simplest fix mirrors what `PythLazer` already does: call `_disableInitializers()` in the constructor of the setup contract, or add a one-time-use flag:

```solidity
bool private _setupDone;

function setup(...) public {
    require(!_setupDone, "already initialized");
    _setupDone = true;
    ...
}
```

---

### Proof of Concept

1. Attacker monitors the mempool.
2. Deployer broadcasts: deploy proxy pointing to `ReceiverSetup`.
3. Deployer broadcasts: call `proxy.setup(realImpl, realGuardians, ...)`.
4. Attacker front-runs step 3 with: `proxy.setup(attackerImpl, [attackerKey], chainId, govChain, attackerGovContract)`.
5. Attacker's `setup()` executes first; `_upgradeTo(attackerImpl)` installs the malicious implementation.
6. Deployer's `setup()` call now hits `attackerImpl`, which has no `setup()` function — it reverts harmlessly, but the proxy is already compromised.
7. Attacker uses `attackerKey` to sign forged VAAs containing arbitrary Pyth price data and submits them to the now-attacker-controlled receiver.
