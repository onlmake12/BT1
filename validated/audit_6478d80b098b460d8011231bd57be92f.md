### Title
Keeper Can Drain Subscription Balance by Replaying Multiple Historical Price Updates When Heartbeat Periods Are Missed - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

`Scheduler.sol::updatePriceFeeds()` pays a keeper fee on every successful call. The heartbeat trigger condition compares the price feed's `publishTime` (`updateTimestamp`) against `priceLastUpdatedAt + heartbeatSeconds`, not `block.timestamp`. When a subscription has been inactive for multiple heartbeat periods, a keeper can fetch multiple historical signed price updates from Hermes (each with a `publishTime` spaced `heartbeatSeconds` apart) and submit them one by one in rapid succession, collecting a full keeper fee on each call, draining the subscription balance far faster than the subscription owner intended.

---

### Finding Description

`updatePriceFeeds` is the permissionless keeper entry point in `Scheduler.sol`. After parsing the price update, `_validateShouldUpdatePrices` enforces the heartbeat condition:

```solidity
// Reject updates if they're older than the latest stored ones
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(...);
}

// If updateOnHeartbeat is enabled and the heartbeat interval has passed, trigger update
if (params.updateCriteria.updateOnHeartbeat) {
    uint256 lastUpdateTime = status.priceLastUpdatedAt;
    if (
        lastUpdateTime == 0 ||
        updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds
    ) {
        return updateTimestamp;
    }
}
``` [1](#0-0) 

After a successful update, `status.priceLastUpdatedAt` is set to `latestPublishTime` (the price feed's `publishTime`):

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

The keeper is then paid:

```solidity
_processFeesAndPayKeeper(status, startGas, params.priceIds.length);
``` [3](#0-2) 

The payment logic transfers `gasCost + keeperSpecificFee` to `msg.sender` on every successful call:

```solidity
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [4](#0-3) 

**The flaw:** The heartbeat guard compares `updateTimestamp` (the price feed's `publishTime`, sourced from Hermes historical data) against `priceLastUpdatedAt + heartbeatSeconds`. It does **not** compare `block.timestamp` against `priceLastUpdatedAt + heartbeatSeconds`. Hermes exposes a public historical price API. If a subscription has been inactive for N heartbeat periods, there exist N valid signed price updates in Hermes history, each with a `publishTime` spaced `heartbeatSeconds` apart. A keeper can submit all N in the same block (or back-to-back transactions), each satisfying:

1. `updateTimestamp > priceLastUpdatedAt` (strictly newer)
2. `updateTimestamp >= priceLastUpdatedAt + heartbeatSeconds` (heartbeat met)
3. `updateTimestamp >= block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` (within validity window)

Each call succeeds and pays the keeper a full fee. The subscription's balance is drained at N× the intended rate.

The `parsePriceFeedUpdatesWithConfig` call is made with `minTimestamp = 0`, explicitly allowing old price data:

```solidity
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
    ...
)
``` [5](#0-4) 

The README itself acknowledges this design: *"Note: We don't want to reject update data if it contains a price from a market that closed a few days ago"* — confirming `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` spans days, making many historical heartbeat slots available. [6](#0-5) 

---

### Impact Explanation

A keeper can collect N keeper fees (gas reimbursement + `singleUpdateKeeperFeeInWei × numPriceIds` per call) in a single block by replaying N historical price updates, where N = `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD / heartbeatSeconds`. For a subscription with `heartbeatSeconds = 60` and a validity window of 1 day, N ≈ 1440. The subscription's ETH balance is drained at up to 1440× the intended rate, causing premature subscription deactivation and denial of service to the subscription's readers. The keeper profits at the direct expense of the subscription manager.

---

### Likelihood Explanation

- Any permissionless keeper can trigger this; no privileged access is required.
- Hermes provides a public historical price API, making historical VAAs freely accessible.
- Subscriptions that are temporarily inactive (e.g., due to low keeper competition, network congestion, or a newly created subscription) are vulnerable.
- The attack is economically rational: the keeper profits from each extra fee collected.

---

### Recommendation

Normalize `priceLastUpdatedAt` using modular arithmetic analogous to the Olympus fix, so that even if multiple heartbeat periods have elapsed, only one update is credited per period:

```solidity
// Instead of:
status.priceLastUpdatedAt = latestPublishTime;

// Use:
status.priceLastUpdatedAt = latestPublishTime
    - (latestPublishTime - status.priceLastUpdatedAt)
      % params.updateCriteria.heartbeatSeconds;
```

Alternatively, enforce that `updateTimestamp < priceLastUpdatedAt + 2 * heartbeatSeconds` to cap the number of missed periods that can be replayed in one call, or track the last update using `block.timestamp` rather than the price feed's `publishTime` for the heartbeat guard.

---

### Proof of Concept

1. Subscription is created with `heartbeatSeconds = 60` and funded with 1 ETH.
2. No keeper updates the subscription for 1 hour (60 missed heartbeat periods).
3. Attacker fetches 60 historical signed price updates from Hermes, each with `publishTime` spaced 60 seconds apart, all within `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`.
4. Attacker calls `updatePriceFeeds(subscriptionId, updateData_1)` → succeeds, `priceLastUpdatedAt = T1`, keeper receives fee.
5. Attacker calls `updatePriceFeeds(subscriptionId, updateData_2)` where `publishTime_2 = T1 + 60` → succeeds, keeper receives fee again.
6. Repeat 58 more times in the same block.
7. Attacker has collected 60× the intended single-update keeper fee, draining the subscription balance in one block instead of over one hour. [7](#0-6) [8](#0-7) [9](#0-8)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }
