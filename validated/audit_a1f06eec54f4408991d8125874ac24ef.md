### Title
Excess `msg.value` Permanently Credited to Protocol Instead of Refunded in Entropy Request Functions - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
In `Entropy.sol`, the internal `requestHelper` function accepts `msg.value` for fee payment but silently absorbs any excess ETH above the required fee into `_state.accruedPythFeesInWei` (Pyth protocol treasury) rather than refunding it to the caller. All public entry points — `request()`, `requestWithCallback()`, `requestV2()` — share this behavior. Users who overpay (due to stale fee estimates, fee increases between estimation and submission, or simple miscalculation) permanently lose the excess ETH with no recovery path.

### Finding Description
In `requestHelper`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The expression `SafeCast.toUint128(msg.value) - providerFee` assigns the **entire remainder** of `msg.value` (after the provider's share) to the Pyth protocol treasury. If `msg.value = requiredFee + X`, then `X` is silently taken by the protocol rather than returned to `msg.sender`. There is no refund path anywhere in `request()`, `requestWithCallback()`, or `requestV2()`. [2](#0-1) [3](#0-2) [4](#0-3) 

The interface documentation acknowledges this: "excess value is *not* refunded to the caller." [5](#0-4) [6](#0-5) [7](#0-6) 

The excess is recoverable only by the Pyth admin via `withdrawFee`, not by the original caller. [8](#0-7) 

### Impact Explanation
Any user who sends `msg.value > getFeeV2(provider, gasLimit)` permanently loses the excess ETH. The excess is credited to the Pyth protocol treasury (`accruedPythFeesInWei`) and can only be withdrawn by the privileged admin — not by the original caller. This is a direct, unrecoverable financial loss for the user.

### Likelihood Explanation
Fee changes are a realistic trigger. The `pythFeeInWei` and provider `feeInWei` can be updated at any time by the admin or provider respectively. A user who calls `getFeeV2()` off-chain and then submits a transaction may find the fee has increased in the interim (front-run by a fee update), causing their previously-correct `msg.value` to become an overpayment on the next request. Additionally, integrators who hardcode fee values or add a safety buffer to avoid `InsufficientFee` reverts will routinely overpay. The test suite itself demonstrates overpayment as a normal usage pattern: [9](#0-8) 

### Recommendation
Add a refund of excess `msg.value` at the end of `requestHelper`:

```solidity
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee); // only the protocol's share
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

Alternatively, enforce `msg.value == requiredFee` exactly, reverting on overpayment.

### Proof of Concept
1. Provider registers with `feeInWei = 100 wei`; Pyth fee is `50 wei`; `requiredFee = 150 wei`.
2. User calls `requestV2(provider, userRandom, 0){value: 200 wei}` (overpays by 50 wei).
3. `requestHelper` executes: `providerInfo.accruedFeesInWei += 100`; `_state.accruedPythFeesInWei += (200 - 100) = 100` (should be 50).
4. User's 50 wei excess is now in `accruedPythFeesInWei` — only the Pyth admin can withdraw it via `withdrawFee`. The user has no recourse. [1](#0-0)

        address provider,
        bytes32 userContribution,
        uint32 gasLimit
    ) public payable override returns (uint64) {
        EntropyStructsV2.Request storage req = requestHelper(
            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.
            // If we remove the blockHash from this, the provider would have no choice but to provide its committed
            // random number. Hence, useBlockHash is set to false.
            false,
            true,
            gasLimit
        );

        emit RequestedWithCallback(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            EntropyStructConverter.toV1Request(req)
        );
        emit EntropyEventsV2.Requested(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            uint32(req.gasLimit10k) * TEN_THOUSAND,
            bytes("")
        );
        return req.sequenceNumber;
    }

    function withdrawFee(address targetAddress, uint128 amount) external {
        require(targetAddress != address(0), "targetAddress is zero address");
        _authoriseAdminAction();

        if (amount > _state.accruedPythFeesInWei)
            revert EntropyErrors.InsufficientFee();

        _state.accruedPythFeesInWei -= amount;

        (bool success, ) = targetAddress.call{value: amount}("");
        require(success, "Failed to withdraw fees");

        emit FeeWithdrawn(targetAddress, amount);
    }
