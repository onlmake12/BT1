Audit Report

## Title
Sync Layer `version_check` Ignores RFC-0048 Hardfork Gate, Permanently Rejecting Valid Post-Hardfork Non-Zero-Version Headers — (`sync/src/synchronizer/headers_process.rs`)

## Summary

`HeaderAcceptor::version_check` unconditionally rejects any header with a non-zero `version` field, with no reference to the RFC-0048 hardfork switch (`is_remove_header_version_reservation_rule_enabled`). RFC-0048 is activated on mainnet at epoch 12293 and explicitly lifts the version-zero reservation. The hardfork gate function is defined but confirmed by grep to have zero call sites anywhere in the codebase, meaning it is never consulted. Any node receiving a syntactically valid, PoW-correct block with a non-zero header version after epoch 12293 will permanently mark it `BLOCK_INVALID` and never reconsider it, causing a consensus split against nodes that correctly implement RFC-0048.

## Finding Description

`HeaderAcceptor::version_check` at `sync/src/synchronizer/headers_process.rs:286–293` enforces a hardcoded version-zero rule with no epoch or hardfork awareness:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    if self.header.version() != 0 {
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
```

When this returns `Err(())`, `accept()` at line 346–353 immediately writes `BlockStatus::BLOCK_INVALID` into the shared block-status map and returns, permanently blacklisting the block hash with no mechanism for reconsideration.

The RFC-0048 hardfork gate `is_remove_header_version_reservation_rule_enabled` is generated via `define_methods!` macro in `util/types/src/core/hardfork/ckb2023.rs:66–73`. A codebase-wide grep for this function name returns exactly one match — the macro definition site — confirming it is never invoked. By contrast, the analogous RFC-0049 gate `is_vm_version_2_and_syscalls_3_enabled` has 5 call sites in `script/src/types.rs`, demonstrating the pattern was applied for RFC-0049 but omitted for RFC-0048.

The execution order in `accept()` is: `prev_block_check` → `non_contextual_check` (which includes `PowVerifier`) → `version_check`. A header must pass valid PoW before reaching `version_check`, so the triggering block must be a legitimately mined block with non-zero version — exactly the class of blocks RFC-0048 permits. `HeaderVerifier::verify()` checks only PoW, block number, epoch continuity, and timestamp; it performs no version check and provides no safety net.

## Impact Explanation

This is a **consensus deviation** vulnerability matching the Critical bounty impact: *"Vulnerabilities which could easily cause consensus deviation."* After RFC-0048 activation at epoch 12293 (already live on mainnet), a miner is permitted by consensus rules to produce a block with a non-zero `version` field. Any CKB node running this code will permanently reject such a block at the sync layer. The affected node silently stalls on a stale fork while nodes correctly implementing RFC-0048 follow the canonical chain. The `BLOCK_INVALID` status is permanent — there is no retry or re-evaluation path.

## Likelihood Explanation

RFC-0048 is already active on mainnet. No current miner software produces non-zero-version headers, so the bug is latent today. The primary realistic trigger is a miner updating block-assembly software to set a non-zero `version` field, which is a natural and permitted use of the now-unreserved bits under RFC-0048. At that point, every unpatched node that receives the block via the Sync protocol `SendHeaders` message will split from the canonical chain. A malicious relayer crafting a fake header with `version = 1` cannot trigger the bug cheaply because `non_contextual_check` (including `PowVerifier`) runs before `version_check`, requiring a valid Eaglesong PoW solution.

## Recommendation

In `HeaderAcceptor::version_check`, consult the hardfork switch before enforcing the version-zero rule:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    let epoch = self.header.epoch().number();
    let hardfork = self.active_chain.sync_shared()
        .shared()
        .consensus()
        .hardfork_switch();
    if !hardfork.ckb2023.is_remove_header_version_reservation_rule_enabled(epoch)
        && self.header.version() != 0
    {
        state.invalid(Some(ValidationError::Version));
        return Err(());
    }
    Ok(())
}
```

This mirrors the existing pattern used for VM-version and syscall gating in `script/src/types.rs`, where `is_vm_version_2_and_syscalls_3_enabled` is consulted before applying post-hardfork rules.

## Proof of Concept

1. Confirm RFC-0048 is active: call `get_consensus` RPC and verify `hardfork_features` contains `{"rfc":"0048","epoch_number":"0x3005"}`.
2. Mine (or obtain) a block at epoch ≥ 12293 with `header.version = 1` and a valid Eaglesong PoW solution for the current chain tip.
3. Relay the block to an unpatched CKB node via the Sync protocol `SendHeaders` P2P message.
4. Observe that `HeaderAcceptor::accept()` at `sync/src/synchronizer/headers_process.rs:346` calls `version_check`, which returns `Err(())`, causing `shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID)` at line 352.
5. Confirm the node never syncs this block even if it becomes the canonical chain tip, while a patched node or a node implementing RFC-0048 correctly accepts it — demonstrating the consensus split.
