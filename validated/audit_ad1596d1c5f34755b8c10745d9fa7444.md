### Title
Unbounded DB Amplification via `GetTransactionsProof` with 1000 Tx Hashes Across 1000 Blocks — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

### Summary

An unprivileged remote peer can send a single `GetTransactionsProof` message with 1000 `tx_hashes` (at the `GET_TRANSACTIONS_PROOF_LIMIT`), each confirmed in a distinct block, forcing the server to perform thousands of DB reads with no ban triggered and no rate limiting.

### Finding Description

`GET_TRANSACTIONS_PROOF_LIMIT` is set to 1000. [1](#0-0) 

`execute()` accepts up to 1000 `tx_hashes` before rejecting: [2](#0-1) 

For each found transaction, `get_transaction_with_info()` is called: [3](#0-2) 

For each unique block containing those transactions, `get_block()`, `CBMT::build_merkle_proof()`, and `get_block_uncles()` are all called: [4](#0-3) 

Finally, `reply_proof()` is called with all 1000 block positions, which performs `mmr.get_root()` and `mmr.gen_proof(items_positions)` — the latter costing O(N_positions × log(chain_length)) DB reads: [5](#0-4) 

A valid request returns `Status::ok()` (code 200). `should_ban()` only triggers for 4xx codes: [6](#0-5) 

The `received()` handler applies no rate limiting, no per-peer request throttling, and no cooldown: [7](#0-6) 

### Impact Explanation

On a 1M-block chain (log₂(1M) ≈ 20), a single message with 1000 tx hashes spread across 1000 blocks causes approximately:
- 1000 `get_transaction_with_info()` reads
- 1000 `get_block()` reads
- 1000 `CBMT::build_merkle_proof()` computations
- 1000 `get_block_uncles()` reads
- ~20,000 MMR DB reads from `gen_proof(1000 positions)` on a 1M-block chain

Total: ~24,000 RocksDB reads per single P2P message. An attacker can send these messages in a tight loop from multiple peers with no ban, causing sustained node performance degradation and congestion.

### Likelihood Explanation

The attack requires only a valid P2P connection to a node running the light client protocol server. The attacker needs to know 1000 confirmed tx hashes (trivially obtained from a block explorer or by scanning the chain). No special privileges, no PoW, no key material required.

### Recommendation

1. **Rate-limit per peer**: Track the number of `GetTransactionsProof` requests per peer per time window and disconnect/ban peers that exceed a threshold.
2. **Reduce the limit**: `GET_TRANSACTIONS_PROOF_LIMIT = 1000` is too high given the O(N × log(chain)) MMR cost. A limit of 50–100 would reduce amplification significantly.
3. **Cap unique blocks**: Independently limit the number of distinct blocks referenced in a single request (e.g., max 50 unique blocks), since the MMR proof cost scales with unique block positions, not just tx count.
4. **Charge work back to the peer**: Consider tracking cumulative DB work per peer and banning peers that consistently trigger high-cost requests.

### Proof of Concept

1. Connect to a CKB full node with the light client protocol enabled.
2. Collect 1000 confirmed tx hashes, each from a different block, from a chain with ≥1M blocks.
3. Send a single `GetTransactionsProof` message with all 1000 hashes and a valid `last_hash`.
4. Observe: server performs ~24,000 RocksDB reads, returns `Status::ok()`, no ban triggered.
5. Repeat in a tight loop from a single peer — no throttle or ban will stop it.
6. Measure RocksDB read latency and CPU on the server; assert amplification factor grows with chain length.

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
