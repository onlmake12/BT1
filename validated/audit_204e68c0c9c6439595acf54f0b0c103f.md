### Title
Unbounded `announce=true` Nodes messages allow unlimited peer-store pollution per session — (`network/src/protocols/discovery/mod.rs`)

### Summary
The `DuplicateFirstNodes` guard in `DiscoveryProtocol::received` only tracks `announce=false` Nodes messages via the `received_nodes` flag. `announce=true` messages are subject only to a per-message item cap (`ANNOUNCE_THRESHOLD = 10`), with no per-session message count or total-address limit. A single connected peer can send an unbounded stream of `announce=true` Nodes messages, each injecting up to 30 addresses into the peer store.

### Finding Description
In `mod.rs` lines 181–205, the Nodes handler checks `!nodes.announce && state.received_nodes` to detect duplicate non-announce messages. For `announce=true` the check is skipped entirely and `state.received_nodes` is never updated. `verify_nodes_message` enforces only a per-message ceiling (10 items, 3 addrs/item). There is no session-scoped counter, no total-address budget, and no time-based rate limit for announce messages. The attacker rotates through unique public-IP multiaddreses to defeat the `addr_known` bloom filter dedup.

### Impact Explanation
The peer store is finite. Flooding it with attacker-controlled addresses displaces legitimate peer records, degrading or preventing peer discovery for the victim node. The effect persists until the peer store's own eviction policy reclaims space, which may take hours given typical TTLs.

### Likelihood Explanation
Any peer that completes the P2P handshake can execute this. No special role, key, or hashpower is required. The attack is cheap: sending thousands of small UDP/TCP frames is trivial.

### Recommendation
Track a per-session `announce_received_count` (or total addresses received via announce) and trigger `Misbehavior::TooManyItems` / disconnect once a session-level threshold is exceeded, mirroring the existing `DuplicateFirstNodes` protection for `announce=false`.

### Proof of Concept
1. Establish a session with a CKB node.
2. In a loop, send `Nodes(announce=true, items=[10 nodes × 3 unique public addrs])`.
3. After N iterations, assert `peer_store.addr_count() ≈ N × 30` and that no disconnect occurred.
4. Confirm `received_nodes` remains `false` throughout, so a final `Nodes(announce=false)` is still accepted normally.
