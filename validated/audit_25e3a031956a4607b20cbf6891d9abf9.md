### Title
Unbounded `while` Loop in `executeCallback` Enables Gas DoS on Request Fulfillment — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` by scanning through all cleared (inactive) requests. An unprivileged attacker can inflate the number of fulfilled-but-unscanned requests so that the next `executeCallback` call for the oldest pending request exhausts the block gas limit, permanently preventing that request from being fulfilled.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs the following loop:

```solidity
// Echo.sol lines 169–174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

The loop advances `firstUnfulfilledSeq` one step at a time for every consecutive inactive (already-fulfilled) slot. There is no cap on how many iterations it may perform in a single transaction.

`requestPriceUpdatesWithCallback` imposes no limit on the total number of outstanding requests:

```solidity
// Echo.sol lines 52–102 — no cap on total requests
requestSequenceNumber = _state.currentSequenceNumber++;
``` [2](#0-1) 

`executeCallback` is callable by any address:

```solidity
function executeCallback(
    address providerToCredit,
    uint64 sequenceNumber,
    ...
) external payable override {
``` [3](#0-2) 

**Attack path:**

1. Attacker calls `requestPriceUpdatesWithCallback` N times, obtaining sequence numbers `1 … N`. Each call pays the required fee.
2. Attacker calls `executeCallback` for sequence numbers `2 … N` (all except `1`). After each call the while loop stops immediately because `firstUnfulfilledSeq = 1` is still active.
3. A legitimate user (or the attacker) now calls `executeCallback` for sequence number `1`.
4. `clearRequest(1)` marks slot 1 inactive.
5. The while loop starts at `firstUnfulfilledSeq = 1` and must iterate through all N now-inactive slots before stopping at `N+1 = currentSequenceNumber`. Each iteration calls `findRequest` (a storage lookup) and increments a storage variable — O(N) storage operations in a single transaction.
6. The transaction reverts with out-of-gas, rolling back `clearRequest(1)` as well. Request 1 is permanently stuck.

---

### Impact Explanation

- **Denial of Service on `executeCallback`**: Any request whose sequence number equals `firstUnfulfilledSeq` can be made permanently un-fulfillable. The requester's funds (fee) are locked in the contract with no recovery path, because `executeCallback` is the only way to clear a request and trigger the callback.
- **Locked user funds**: The fee paid by the victim at request time (`req.fee`) is never credited to the provider and cannot be reclaimed by the requester.
- **Protocol liveness**: Because `firstUnfulfilledSeq` never advances past the stuck slot, `getFirstActiveRequests` (which also iterates from `firstUnfulfilledSeq`) will always return the stuck request first, degrading keeper discovery for all subsequent requests. [4](#0-3) 

---

### Likelihood Explanation

- `requestPriceUpdatesWithCallback` and `executeCallback` are both permissionless; no privileged role is required.
- On low-fee EVM chains (Polygon, BSC, Arbitrum, Base) where Echo is deployed, the cost of submitting N requests is proportional to N × (baseFee + providerFee). With small provider fees and cheap gas, N in the thousands is economically feasible.
- The attacker recovers most of their cost because they are the `providerToCredit` for the N−1 callbacks they execute themselves, accruing those fees back.
- The code comment at line 166–167 explicitly acknowledges the problem: *"I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number."* [5](#0-4) 

---

### Recommendation

Replace the unbounded `while` loop with a mapping-based O(1) lookup, as the code comment itself suggests (a doubly-linked list of active requests). Concretely:

1. Maintain a `mapping(uint64 => uint64) nextActive` linked list so that `firstUnfulfilledSeq` can jump directly to the next active request in O(1) when a request is cleared.
2. Alternatively, remove the `firstUnfulfilledSeq` advancement from `executeCallback` entirely and let keepers call a separate, gas-bounded `advanceFirstUnfulfilledSeq(uint256 maxSteps)` function.
3. At minimum, add a `maxSteps` parameter to the while loop so a single `executeCallback` call cannot iterate more than a fixed number of slots.

---

### Proof of Concept

```
Setup (on any low-fee EVM chain):
  N = 5000  (adjustable; gas limit ~30M on most chains)

Step 1 – Inflate the queue:
  for i in 1..N:
    echo.requestPriceUpdatesWithCallback{value: fee}(
        attackerProvider, block.timestamp, [priceId], gasLimit
    )
  // sequence numbers 1..N are now active; firstUnfulfilledSeq = 0

Step 2 – Fulfill all but the first:
  for i in 2..N:
    echo.executeCallback(attackerProvider, i, updateData, priceIds)
  // firstUnfulfilledSeq stays at 0/1 because seq 1 is still active

Step 3 – Victim (or attacker) tries to fulfill seq 1:
  echo.executeCallback{gas: 30_000_000}(
      attackerProvider, 1, updateData, priceIds
  )
  // clearRequest(1) executes, then the while loop iterates ~N times
  // → OOG revert; clearRequest(1) is rolled back
  // → seq 1 is permanently stuck; victim's fee is locked
```

The `while` loop performs one `findRequest` (two `sload`s via `_state.requests[shortKey]`) plus one `sstore` (`firstUnfulfilledSeq++`) per iteration — roughly 5 000 gas per step × 5 000 steps = 25 M gas, exceeding the 30 M block gas limit when combined with the rest of the function's overhead.
