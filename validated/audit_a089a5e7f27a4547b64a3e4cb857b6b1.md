### Title
Division-by-Zero in `calculateTwap` via Missing Equal-Slot Guard in `validateTwapPriceInfo` — (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

`validateTwapPriceInfo` uses a strict `>` comparison for `publishSlot`, allowing equal slots to pass. `calculateTwap` then unconditionally divides by `slotDiff`, which is zero when both slots are equal, causing an EVM division-by-zero panic revert on every such call.

---

### Finding Description

In `validateTwapPriceInfo`, the slot ordering check is:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This only rejects `start > end`. When `start.publishSlot == end.publishSlot`, the condition is `false` and validation passes silently. [1](#0-0) 

`calculateTwap` then computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // panic: division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // panic: division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panic: division by zero
```

All three divisions are unguarded. Any one of them triggers an EVM `Panic(0x12)` revert. [2](#0-1) 

The Solana implementation of the same logic correctly uses a strict greater-than `require!`, which rejects equal slots:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The Ethereum contract is inconsistent with the Solana reference implementation.

---

### Impact Explanation

Any caller of `parseTwapPriceFeedUpdates` — a fully public, payable function — can trigger this revert by submitting the same valid guardian-signed TWAP VAA as both `updateData[0]` (start) and `updateData[1]` (end). Both messages carry the same `publishSlot`. The function reverts with an unhandled panic instead of a clean `InvalidTwapUpdateDataSet` error, making TWAP price publication permanently unavailable for that slot pair. Downstream integrators relying on TWAP prices receive no price data. [4](#0-3) 

---

### Likelihood Explanation

The attack requires only one valid guardian-signed TWAP VAA (freely available from Hermes/Wormhole relayers). The attacker submits it twice — once as start, once as end. No privileged access, no key compromise, no Sybil attack is needed. The call path is direct: `parseTwapPriceFeedUpdates` → `validateTwapPriceInfo` (passes) → `calculateTwap` (panics). This is trivially reproducible in a unit test. [5](#0-4) 

---

### Recommendation

Change the slot comparison in `validateTwapPriceInfo` from strict `>` to `>=`:

```solidity
// Before (buggy):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After (correct):
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This matches the Solana implementation's semantics and eliminates the zero-denominator path entirely. [1](#0-0)
