### Title
Whitelist Check Applied to Wrong `msg.sender` in `getPricesNoOlderThan` / `getEmaPricesNoOlderThan` Causes Permanent DoS When Whitelist Is Enabled — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol` implements a `readerWhitelist` feature via the `onlyWhitelistedReader` modifier. Two functions — `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` — pass the modifier for the original caller, but then internally call `this.getPricesUnsafe(...)` and `this.getEmaPricesUnsafe(...)` respectively. Because `this.f()` is an external call in Solidity, `msg.sender` inside those inner calls becomes `address(this)` (the Scheduler contract itself). The `onlyWhitelistedReader` modifier is re-applied to that inner call and the Scheduler contract is neither the subscription manager nor in the `readerWhitelist`, so the call always reverts with `Unauthorized` whenever `whitelistEnabled == true`. The result is a permanent, protocol-level DoS on both freshness-checked getter functions for every subscription that has the whitelist enabled.

---

### Finding Description

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` are guarded by `onlyWhitelistedReader`, which correctly validates the original `msg.sender`. However, both functions delegate to the unsafe variants via an external `this.` call:

```solidity
// Scheduler.sol lines 535–555
function getPricesNoOlderThan(...)
    external view override
    onlyWhitelistedReader(subscriptionId)   // ← checks original msg.sender ✓
    returns (PythStructs.Price[] memory prices)
{
    ...
    prices = this.getPricesUnsafe(subscriptionId, priceIds);  // ← external call; msg.sender = address(this)
}

// Scheduler.sol lines 578–598
function getEmaPricesNoOlderThan(...)
    external view override
    onlyWhitelistedReader(subscriptionId)   // ← checks original msg.sender ✓
    returns (PythStructs.Price[] memory prices)
{
    ...
    prices = this.getEmaPricesUnsafe(subscriptionId, priceIds);  // ← external call; msg.sender = address(this)
}
```

Inside `getPricesUnsafe` / `getEmaPricesUnsafe`, `onlyWhitelistedReader` fires again:

```solidity
// Scheduler.sol lines 750–779
modifier onlyWhitelistedReader(uint256 subscriptionId) {
    if (_state.subscriptionManager[subscriptionId] == msg.sender) { _; return; }
    if (!_state.subscriptionParams[subscriptionId].whitelistEnabled) { _; return; }

    address[] storage whitelist = _state.subscriptionParams[subscriptionId].readerWhitelist;
    bool isWhitelisted = false;
    for (uint i = 0; i < whitelist.length; i++) {
        if (whitelist[i] == msg.sender) { isWhitelisted = true; break; }
    }
    if (!isWhitelisted) { revert SchedulerErrors.Unauthorized(); }
    _;
}
```

At this point `msg.sender == address(this)` (the Scheduler proxy). The Scheduler contract is never the subscription manager and is never added to `readerWhitelist`, so the modifier always reverts when `whitelistEnabled == true`. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Any subscription that sets `whitelistEnabled = true` (the security-hardened configuration) permanently loses access to `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` for **all callers**, including the subscription manager and every address in `readerWhitelist`. These are the only functions that enforce a freshness bound on returned prices. Consumers that depend on them for safety (e.g., DeFi protocols that must reject stale prices) are forced to fall back to `getPricesUnsafe` / `getEmaPricesUnsafe` and implement staleness checks themselves, or simply cannot use the Scheduler at all with whitelist enabled. The whitelist feature and the freshness-checked getters are mutually exclusive due to this bug, which is the opposite of the intended design.

---

### Likelihood Explanation

Any subscription owner who enables `whitelistEnabled = true` (a documented, intentional feature) immediately triggers the DoS. No special attacker is needed — the broken state is entered by normal, legitimate configuration. The Scheduler is a deployed production contract; any integrator enabling the whitelist will encounter this.

---

### Recommendation

Replace the external `this.` delegation with an internal helper that does not re-apply the modifier:

```solidity
// Extract shared logic into an internal function (no modifier)
function _getPricesUnsafeInternal(
    uint256 subscriptionId,
    bytes32[] calldata priceIds
) internal view returns (PythStructs.Price[] memory prices) {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        prices[i] = priceFeeds[i].price;
    }
}

function getPricesNoOlderThan(...) external view override onlyWhitelistedReader(subscriptionId) ... {
    ...
    prices = _getPricesUnsafeInternal(subscriptionId, priceIds); // internal call, no re-check
}
```

The same pattern applies to `getEmaPricesNoOlderThan` / `getEmaPricesUnsafe`.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` (or use the existing deployment).
2. Call `createSubscription` with `whitelistEnabled = true` and `readerWhitelist = [alice]`.
3. Fund the subscription and push a price update.
4. As `alice` (a whitelisted address), call `getPricesNoOlderThan(subscriptionId, [], 3600)`.
5. Observe revert with `SchedulerErrors.Unauthorized` — even though `alice` is whitelisted.
6. As `alice`, call `getPricesUnsafe(subscriptionId, [])` directly — succeeds.

The revert in step 5 occurs because `getPricesNoOlderThan` internally calls `this.getPricesUnsafe(...)`, making `msg.sender = address(scheduler)` inside `getPricesUnsafe`, which fails the `onlyWhitelistedReader` check since the Scheduler contract itself is not in the whitelist.
