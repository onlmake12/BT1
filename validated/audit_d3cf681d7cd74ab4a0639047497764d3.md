### Title
Missing Duplicate Index Validation in `GetBlockTransactions` Handler Enables Bandwidth Amplification - (`sync/src/relayer/get_block_transactions_process.rs`)

---

### Summary

The `GetBlockTransactions` P2P message handler does not validate that the `indexes` or `uncle_indexes` fields are free of duplicates. A malicious peer can send a single small request containing repeated indexes, causing the honest node to look up and transmit the same transactions or uncles multiple times, amplifying outbound bandwidth by up to `MAX_RELAY_TXS_NUM_PER_BATCH`-fold.

---

### Finding Description

In `sync/src/relayer/get_block_transactions_process.rs`, the handler for the `GetBlockTransactions` relay message validates only that the count of indexes does not exceed protocol limits: [1](#0-0) 

After those count checks, the handler iterates directly over the peer-supplied `indexes` and `uncle_indexes` arrays without any uniqueness check: [2](#0-1) 

Because `filter_map` + `get()` happily returns the same element for every repeated index, the resulting `transactions` and `uncles` vectors can contain many copies of the same item. The node then serialises and sends all of them back to the requesting peer.

The analogous `GetRelayTransactions` handler **does** perform this check and explicitly rejects duplicate hashes: [3](#0-2) 

The `GetBlockTransactions` message schema carries two index arrays: [4](#0-3) 

Both arrays are bounded only by count, not by uniqueness.

---

### Impact Explanation

An attacker who is a connected relay peer can craft a `GetBlockTransactions` message with `indexes = [0, 0, 0, …, 0]` (up to `MAX_RELAY_TXS_NUM_PER_BATCH` entries) pointing at the largest transaction in any known block. The honest node responds with that many copies of the same transaction, multiplying the attacker's effective bandwidth consumption by the full batch limit. Repeated from multiple connections or in rapid succession, this constitutes a targeted bandwidth-exhaustion attack against individual nodes, degrading their ability to participate in block propagation and sync.

---

### Likelihood Explanation

Any unauthenticated peer that has completed the relay handshake can send `GetBlockTransactions`. No special privilege, key, or majority hash-power is required. The attack is trivially constructable from the public protocol schema and requires only a single malformed message per amplification cycle.

---

### Recommendation

Add a duplicate-index check immediately after the count validation, mirroring the pattern already used in `get_transactions_process.rs`:

```rust
// After existing count checks:
let indexes_set: HashSet<u32> = get_block_transactions.indexes().iter()
    .map(Into::into).collect();
if indexes_set.len() != get_block_transactions.indexes().len() {
    return StatusCode::ProtocolMessageIsMalformed
        .with_context("Duplicate transaction indexes");
}

let uncle_indexes_set: HashSet<u32> = get_block_transactions.uncle_indexes().iter()
    .map(Into::into).collect();
if uncle_indexes_set.len() != get_block_transactions.uncle_indexes().len() {
    return StatusCode::ProtocolMessageIsMalformed
        .with_context("Duplicate uncle indexes");
}
```

---

### Proof of Concept

1. Connect to a CKB node as a relay peer (standard P2P handshake).
2. Obtain any block hash `H` that the node has stored.
3. Send a `GetBlockTransactions` message:
   ```
   block_hash:    H
   indexes:       [0, 0, 0, …, 0]   // MAX_RELAY_TXS_NUM_PER_BATCH repetitions of index 0
   uncle_indexes: []
   ```
4. The node executes the loop at lines 61–71 of `get_block_transactions_process.rs`, calling `block.transactions().get(0)` once per entry, collecting `MAX_RELAY_TXS_NUM_PER_BATCH` identical copies of the cellbase transaction.
5. It serialises and transmits all copies in a single `BlockTransactions` response.

The attacker's outbound cost is one small fixed-size request; the honest node's outbound cost is `MAX_RELAY_TXS_NUM_PER_BATCH × sizeof(cellbase_tx)`. The inconsistency with the duplicate-rejection logic already present in `get_transactions_process.rs` confirms this is an unintentional omission rather than a deliberate design choice. [5](#0-4)

    pub async fn execute(self) -> Status {
        let shared = self.relayer.shared();
        {

            if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "Indexes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    get_block_transactions.indexes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
            if get_block_transactions.uncle_indexes().len() > shared.consensus().max_uncles_num() {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "UncleIndexes count({}) > consensus max_uncles_num({})",
                    get_block_transactions.uncle_indexes().len(),
                    shared.consensus().max_uncles_num(),
                ));
            }
        }

        let block_hash = self.message.block_hash().to_entity();
        debug_target!(
            crate::LOG_TARGET_RELAY,
            "get_block_transactions {}",
            block_hash

        if let Some(block) = shared.store().get_block(&block_hash) {
            let transactions = self
                .message
                .indexes()
                .iter()
                .filter_map(|i| {
                    block
                        .transactions()
                        .get(Into::<u32>::into(i) as usize)
                        .cloned()
                })
                .collect::<Vec<_>>();

                .transactions(

                        .into_iter()
                        .map(|uncle| uncle.data())
                        .collect::<Vec<_>>(),
                )
                .build();
            let message = packed::RelayMessage::new_builder().set(content).build();

            return async_send_message_to(&self.nc, self.peer, &message).await;
        }

        Status::ok()
    }
