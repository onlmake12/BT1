### Title
Missing Upper Bound Validation on `heartbeatSeconds` and `deviationThresholdBps` in `Scheduler._validateSubscriptionParams` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler._validateSubscriptionParams` only enforces lower-bound (non-zero) checks on `UpdateCriteria.heartbeatSeconds` and `UpdateCriteria.deviationThresholdBps`. No upper-bound constraints exist. Any user can create or update a subscription with values that make the update conditions permanently impossible to satisfy, causing the subscription to never push price updates and locking deposited funds.

---

### Finding Description

`_validateSubscriptionParams` is called by both `createSubscription` and `updateSubscription`. The relevant validation block is:

```solidity
if (
    params.updateCriteria.updateOnHeartbeat &&
    params.updateCriteria.heartbeatSeconds == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
if (
    params.updateCriteria.updateOnDeviation &&
    params.updateCriteria.deviationThresholdBps == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [1](#0-0) 

Both fields are `uint32`. No maximum is enforced:

- `heartbeatSeconds` can be set to `type(uint32).max` ≈ 136 years. The heartbeat condition `updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds` can never be satisfied within any realistic timeframe.
- `deviationThresholdBps` can be set to `type(uint32).max` = 4,294,967,295 bps ≈ 42,949,673%. The deviation check `deviationBps >= params.updateCriteria.deviationThresholdBps` can never be satisfied for any real asset price. [2](#0-1) 

The `SchedulerConstants` contract defines no maximum for either field: [3](#0-2) 

The `UpdateCriteria` struct accepts arbitrary `uint32` values for both fields: [4](#0-3) 

---

### Impact Explanation

**Impact: Medium**

A subscription created with `updateOnHeartbeat = true, heartbeatSeconds = type(uint32).max, updateOnDeviation = false` (or the equivalent deviation-only variant) will never satisfy its update condition. Consequences:

1. **Permanent subscription DoS**: The subscription is active but `updatePriceFeeds` always reverts with `UpdateConditionsNotMet`. Prices stored in the subscription are never refreshed.
2. **Fund lock**: The deposited ETH balance is consumed only by the Pyth fee on each failed attempt (none succeed), but the balance cannot be withdrawn below the minimum while the subscription is active. If `isPermanent = true`, the subscription cannot be updated at all, permanently locking funds.
3. **Stale price delivery to readers**: Any downstream protocol reading prices via `getPricesNoOlderThan` will revert or receive stale data, breaking integrations. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Low**

The subscription manager is the one who supplies the parameters. Exploitation requires either:
- A misconfiguration error (e.g., passing milliseconds instead of seconds, or accidentally setting `deviationThresholdBps` to a raw integer instead of a BPS value), or
- A malicious manager who intentionally creates a broken subscription and convinces a downstream protocol to rely on it.

Both scenarios are realistic but not common. The absence of any upper-bound guard means there is no on-chain protection against either case.

---

### Recommendation

Add upper-bound guards inside `_validateSubscriptionParams`:

```solidity
uint32 constant MAX_HEARTBEAT_SECONDS = 7 days;   // e.g. 604800
uint32 constant MAX_DEVIATION_BPS     = 10_000;   // 100%

if (params.updateCriteria.updateOnHeartbeat) {
    if (params.updateCriteria.heartbeatSeconds == 0 ||
        params.updateCriteria.heartbeatSeconds > MAX_HEARTBEAT_SECONDS)
        revert SchedulerErrors.InvalidUpdateCriteria();
}
if (params.updateCriteria.updateOnDeviation) {
    if (params.updateCriteria.deviationThresholdBps == 0 ||
        params.updateCriteria.deviationThresholdBps > MAX_DEVIATION_BPS)
        revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [6](#0-5)

        bytes[] calldata updateData

        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    /// for which a price update timestamp is considered valid

    uint64 public constant PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours;
