### Title
Fee Sufficiency Check Ignores Token Price — Raw Amounts Summed Across Different Denominations Without USD Conversion - (File: `target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

### Summary

The Osmosis-specific `is_fee_sufficient` function in the Pyth CosmWasm contract sums raw token amounts across multiple allowed denominations without converting them to a common USD value. This mirrors the GlmRouter bug class exactly: raw token units are compared against a threshold without accounting for the actual market price of each token.

### Finding Description

On Osmosis, the Pyth CosmWasm contract accepts multiple token denominations as fee payment for `update_price_feeds`. The `is_fee_sufficient` function (Osmosis feature flag) accumulates raw `coin.amount` values across all submitted coins into a single `total_amount`, then compares that against `base_denom_fee.amount` — which is always `1` per VAA regardless of which denom is used.

The code itself acknowledges the missing price conversion with explicit `FIXME`/`NOTE` comments:

```rust
// NOTE: the base fee denom right now is = denom: 'uosmo', amount: 1, which is almost negligible
// It's not important to convert the price right now. For now
// we are keeping the base fee amount same for each valid denom -> 1
// but this logic will be updated to use spot price for different valid tokens in future
Ok(base_denom_fee.amount.u128() <= total_amount)
``` [1](#0-0) 

The companion function `get_update_fee_for_denom` also returns the same raw amount (`1` per VAA) for any allowed denom, confirming no price normalization occurs:

```rust
// for other valid denoms too we are using the base amount as 1
// base amount is multiplied to number of vaas to get the total amount
// this will be change later on to add custom logic using spot price for valid tokens
Ok(coin(get_update_fee_amount(deps, vaas)?, denom))
``` [2](#0-1) 

Two distinct sub-issues exist:

1. **Cross-denom substitution**: A user pays `1 uion` instead of `1 uosmo`. If `uion` is worth $0.01 and `uosmo` is worth $0.50, the user pays 50× less USD value but the check `1 <= 1` passes.

2. **Cross-denom aggregation**: A user sends `0.5 uosmo + 0.5 uion`. `total_amount = 1`, the check passes, but the actual USD value is less than `1 uosmo`. [3](#0-2) 

### Impact Explanation

Any unprivileged caller of `update_price_feeds` or `parse_price_feed_updates` on the Osmosis deployment can pay fees in the lowest-value allowed token (or split across denoms) and satisfy the fee check while delivering far less USD value than intended. The protocol loses fee revenue proportional to the price differential between the base denom and the substituted token. The `is_allowed_tx_fees_denom` check only verifies that the token is listed in Osmosis's txFee module — it does not bound the token's market price. [4](#0-3) 

### Likelihood Explanation

The entry path is fully permissionless: any account can call `update_price_feeds` with arbitrary `info.funds`. The Osmosis txFee module lists multiple tokens with varying market prices. The vulnerability is exploitable whenever any allowed non-base token trades at a lower price per unit than `uosmo`. This is a realistic and persistent market condition. [5](#0-4) 

### Recommendation

Before comparing against `base_denom_fee.amount`, convert each coin's amount to the base denomination (`uosmo`) using the Osmosis spot price from the txFee/gamm module. The existing comment already describes the correct fix:

```
// convert each of them to the base token that is 'uosmo'
// combine all the converted token
// check with `has_coins`
```

Only after this conversion should `total_amount` be compared against `base_denom_fee.amount`. Additionally, do not aggregate amounts across different denoms before conversion. [6](#0-5) 

### Proof of Concept

1. Osmosis deployment has `fee = Coin { denom: "uosmo", amount: 1 }` per VAA.
2. `uion` is listed in the Osmosis txFee module (passes `is_allowed_tx_fees_denom`).
3. Attacker calls `update_price_feeds` with 1 VAA, sending `info.funds = [Coin { denom: "uion", amount: 1 }]`.
4. `total_amount = 1`. `base_denom_fee.amount = 1`. Check: `1 <= 1` → `true`.
5. Price feed is updated. Attacker paid `1 uion` (e.g., $0.01) instead of `1 uosmo` (e.g., $0.50).
6. Protocol receives ~98% less USD value than intended per update call. [7](#0-6)

/// Else it will throw error
pub fn get_update_fee_for_denom(deps: &Deps, vaas: &[Binary], denom: String) -> StdResult<Coin> {
    let config = config_read(deps.storage).load()?;

    // if the denom is not a base denom it should be an allowed one

    // the base fee is set to -> denom = base denom of a chain, amount = 1
    // which is very minimal
    // for other valid denoms too we are using the base amount as 1
    // base amount is multiplied to number of vaas to get the total amount

    // this will be change later on to add custom logic using spot price for valid tokens
    Ok(coin(get_update_fee_amount(deps, vaas)?, denom))
}
