### Title
Entropy Provider Sequence Number Exhaustion Enables Temporary DOS — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

An unprivileged attacker can exhaust an Entropy provider's committed sequence numbers by repeatedly calling `requestV2()` and paying the required fee, without ever revealing the requests. Once `providerInfo.sequenceNumber >= providerInfo.endSequenceNumber`, every subsequent randomness request for that provider reverts with `OutOfRandomness()`, blocking all legitimate users until the provider re-registers.

---

### Finding Description

The Entropy contract assigns each provider a finite window of sequence numbers bounded by `endSequenceNumber`, set at registration time as `provider.sequenceNumber + chainLength`. [1](#0-0) 

Every call to `requestHelper()` increments `providerInfo.sequenceNumber` by 1 and reverts if the window is exhausted:

```solidity
uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
    revert EntropyErrors.OutOfRandomness();
providerInfo.sequenceNumber += 1;
``` [1](#0-0) 

There is no per-caller rate limit, no minimum time between requests, and no cap on how many requests a single address can make. An attacker simply calls `requestV2()` in a loop, paying the fee each time, until the provider's entire chain is consumed. The attacker never needs to call `reveal()` or `revealWithCallback()`.

The `requestsOverflow` mapping absorbs requests that collide in the fixed-size `requests[NUM_REQUESTS]` array, so the fixed array size does not protect against this attack. [2](#0-1) 

The `ProviderInfo` struct stores `endSequenceNumber` and `sequenceNumber` as `uint64` fields with no on-chain mechanism to pause or throttle consumption: [3](#0-2) 

---

### Impact Explanation

Once the sequence is exhausted, every call to `requestV2()`, `requestWithCallback()`, or `request()` for that provider reverts with `OutOfRandomness()`. All legitimate users — DeFi protocols, games, lotteries — that depend on that provider for randomness are completely blocked. The provider must call `register()` again with a new commitment to restore service. Until that transaction is mined, the provider is fully non-functional.

If the attacker monitors the mempool and front-runs the provider's re-registration with another burst of requests, the DOS can be sustained indefinitely at a cost proportional to `chainLength × feeInWei` per cycle.

---

### Likelihood Explanation

The attack entry point is the public, payable `requestV2()` function — no special role is required. [4](#0-3) 

The attacker's cost is `(endSequenceNumber - currentSequenceNumber) × feeInWei`. For a provider with a short chain (e.g., 10,000 entries) and a low fee (e.g., 0.0001 ETH), the total cost is only 1 ETH. On low-fee chains (Polygon, BNB, Base), this is economically viable. The Fortuna keeper's default provider likely uses a long chain, but third-party providers registered with shorter chains are immediately vulnerable. [5](#0-4) 

---

### Recommendation

1. **Per-block or per-address request rate limiting**: Limit the number of requests a single `msg.sender` can make within a block or sliding window.
2. **Minimum chain length enforcement**: Require providers to register with a `chainLength` above a protocol-defined minimum to raise the cost of exhaustion.
3. **Emergency sequence extension**: Allow providers (or a privileged role) to extend `endSequenceNumber` without a full re-registration, reducing the recovery window.
4. **Fee floor tied to chain length**: Scale the minimum `feeInWei` to the chain length so that exhausting a short chain is always economically irrational.

### Proof of Concept

```solidity
// Attacker contract
contract EntropyExhauster {
    IEntropy entropy;
    address provider;

    constructor(address _entropy, address _provider) {
        entropy = IEntropy(_entropy);
        provider = _provider;
    }

    // Call with enough ETH to cover N * fee
    function exhaust(uint256 count) external payable {
        uint128 fee = entropy.getFee(provider);
        for (uint256 i = 0; i < count; i++) {
            // Never reveal — just consume sequence numbers
            entropy.requestWithCallback{value: fee}(
                provider,
                bytes32(i) // arbitrary user contribution
            );
        }
    }
}
```

After `exhaust()` consumes all remaining sequence numbers, any call by a legitimate user to `requestV2()` or `requestWithCallback()` for that provider reverts:

```
EntropyErrors.OutOfRandomness()
``` [6](#0-5) 

The provider must call `register()` with a new commitment to restore service, during which all randomness requests are blocked. [7](#0-6)
