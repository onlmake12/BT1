### Title
`ckb_debug` Syscall (2177) Propagates `VMError::External` on Non-UTF8 Input, Causing Consensus-Visible Transaction Rejection — (`script/src/syscalls/debugger.rs`)

---

### Summary

`Debugger::ecall` calls `String::from_utf8` on the raw byte buffer read from VM memory. Any non-UTF8 byte before the null terminator causes `String::from_utf8` to return `Err`, which is immediately mapped to `VMError::External` and propagated via `?`. This error is not intercepted by `map_vm_internal_error` and becomes `ScriptError::VMInternalError`, causing the transaction to be rejected with an `ErrorKind::Script` error. Because `Debugger` is unconditionally registered for every `ScriptVersion`, this affects all scripts on all nodes uniformly.

---

### Finding Description

**Step 1 — Syscall registration (unconditional).**
`generate_ckb_syscalls` in `script/src/syscalls/generator.rs` always pushes `Debugger` into the syscall vector before any version-gated syscalls are added. [1](#0-0) 

**Step 2 — Non-UTF8 byte triggers `VMError::External`.**
`Debugger::ecall` reads bytes from VM memory until a null terminator, then calls `String::from_utf8(buffer)`. If any byte is not valid UTF-8 (e.g., `0xFF`), the call returns `Err`, which is mapped to `VMError::External(format!("String from buffer {e:?}"))` and propagated immediately via `?`. [2](#0-1) 

**Step 3 — `map_vm_internal_error` does not intercept this variant.**
The only special-cased `External` value is the literal string `"stopped"` (used for the `ChunkCommand::Stop` signal). The message `"String from buffer ..."` does not match, so the error falls through to the catch-all arm `_ => ScriptError::VMInternalError(error)`. [3](#0-2) 

**Step 4 — `ScriptError::VMInternalError` becomes a consensus-visible rejection.**
`From<TransactionScriptError> for Error` converts every `ScriptError` except `Interrupts` into `ErrorKind::Script`, which is the same error kind used for `ValidationFailure`. The transaction is rejected by every node running this code. [4](#0-3) 

**Step 5 — `DEBUG_PRINT_SYSCALL_NUMBER` is 2177, confirming the syscall number cited in the question.** [5](#0-4) 

---

### Impact Explanation

The `ckb_debug` syscall is a debug facility; its contract is that it must never cause a consensus-visible script execution failure regardless of what bytes the script places in the buffer. The current implementation breaks this contract: any script that writes a single non-UTF8 byte to a debug buffer and calls syscall 2177 will be rejected with `ScriptError::VMInternalError` on every CKB node. This is incorrect CKB-VM behavior within the stated scope ("Incorrect implementation or behavior of CKB-VM or system scripts").

---

### Likelihood Explanation

The path is fully local-testable with no privileged access required. Any unprivileged script author can trigger it by writing `[0xFF, 0x00]` to a memory region and issuing `ecall` with `a7 = 2177`. The `Debugger` is always present regardless of `ScriptVersion`, so there is no version-gating that could prevent the path from being reached.

---

### Recommendation

Replace `String::from_utf8` with `String::from_utf8_lossy` (which substitutes `U+FFFD` for invalid sequences) so the syscall always succeeds and never propagates an error:

```rust
// script/src/syscalls/debugger.rs, lines 55-57
let s = String::from_utf8_lossy(&buffer);
(self.printer)(&self.sg_info.script_hash, s.as_ref());
```

This eliminates the `VMError::External` path entirely while preserving the debug-print semantics.
