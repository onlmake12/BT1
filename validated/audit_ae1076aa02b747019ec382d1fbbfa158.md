### Title
Keeper Gas Compensation Misses Post-Snapshot Operations in `_processFeesAndPayKeeper` — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler._processFeesAndPayKeeper` takes a `gasleft()` snapshot to measure gas consumed, but several gas-consuming operations (storage writes and the ETH transfer to the keeper) execute **after** that snapshot. Those operations are not included in the measured `gasCost`, so the subscription is charged less than the actual gas consumed and the keeper is systematically underpaid.

---

### Finding Description

`updatePriceFeeds` captures a gas snapshot at the very start:

```solidity
uint256 startGas = gasleft();   // line 279
``` [1](#0-0) 

After all price-feed parsing, validation, and storage, it calls:

```solidity
_processFeesAndPayKeeper(status, startGas, params.priceIds.length);  // line 345
``` [2](#0-1) 

Inside `_processFeesAndPayKeeper`, the gas cost is frozen at the very first line:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;  // line 846
``` [3](#0-2) 

Everything that executes **after** this `gasleft()` call is not included in `gasCost`:

| Operation | Approx. gas (warm storage) |
|---|---|
| SLOAD `_state.singleUpdateKeeperFeeInWei` (line 847) | ~100 |
| SLOAD `status.balanceInWei` (line 852) | ~100 |
| SSTORE `status.balanceInWei -= totalKeeperFee` (line 856) | ~5,000 |
| SSTORE `status.totalSpent += totalKeeperFee` (line 857) | ~5,000 |
| `msg.sender.call{value: totalKeeperFee}("")` (line 860) | ~2,300+ | [4](#0-3) 

Total unaccounted gas: **~12,500–15,000 gas** per `updatePriceFeeds` call.

The `GAS_OVERHEAD` constant is documented as covering **transaction overhead** (base cost + calldata), not these post-snapshot operations:

```solidity
/// Fixed gas overhead component used in keeper fee calculation.
/// This is a rough estimate of the tx overhead for a keeper to call updatePriceFeeds.
uint256 public constant GAS_OVERHEAD = 30000;
``` [5](#0-4) 

`GAS_OVERHEAD = 30000` is already consumed by the 21,000 base transaction cost plus calldata (each non-zero byte costs 16 gas; a typical multi-feed update payload is hundreds of bytes). There is no remaining budget in `GAS_OVERHEAD` to cover the ~12,500–15,000 post-snapshot gas.

---

### Impact Explanation

Every successful `updatePriceFeeds` call underpays the keeper by approximately 12,500–15,000 gas × `tx.gasprice`. At 10 gwei this is ~0.000125–0.00015 ETH per update. Across a high-frequency subscription (e.g., 1 update/minute), this accumulates to ~0.18–0.22 ETH/day of uncompensated keeper cost. Keepers operating at thin margins may become unprofitable, degrading the liveness of the permissionless keeper network that Pulse depends on.

The subscription manager (who funds the subscription) benefits from the underpayment — they pay less per update than the true gas cost — but cannot amplify this beyond the fixed per-call gap.

---

### Likelihood Explanation

This affects **every** successful `updatePriceFeeds` call on every active subscription. No special conditions are required; any permissionless keeper calling the function triggers the accounting gap. The entry path is fully unprivileged.

---

### Recommendation

Move the `gasleft()` snapshot to the **last possible point** before the keeper payment, or account for the post-snapshot operations explicitly. One approach is to add a `POST_SNAPSHOT_GAS` constant (analogous to the report's `BankSendGasConsumption`) that covers the known fixed gas of the storage writes and ETH transfer, and add it to `GAS_OVERHEAD`:

```solidity
// Approximate gas consumed after the gasleft() snapshot:
// SLOAD x2 (~200) + SSTORE x2 (~10000) + CALL (~3000) + overhead
uint256 constant POST_SNAPSHOT_GAS = 14000;

uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD + POST_SNAPSHOT_GAS) * tx.gasprice;
```

Alternatively, restructure `_processFeesAndPayKeeper` so that the `gasleft()` snapshot is taken after all storage writes and only the ETH transfer remains, minimising the unaccounted window.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` with `gasPrice = 10 gwei`.
2. Create a subscription with 5 price feeds and fund it.
3. Call `updatePriceFeeds` from a keeper address.
4. Record `keeperBalanceBefore` and `keeperBalanceAfter`.
5. Independently measure the actual gas consumed by the full transaction (via `gasUsed` from the receipt × `tx.gasprice`).
6. Observe that `keeperBalanceAfter - keeperBalanceBefore < actualGasCost`: the keeper receives less than the true gas expenditure because the ~12,500–15,000 gas consumed inside `_processFeesAndPayKeeper` after the `gasleft()` call (lines 847–863) is not included in `gasCost`.

The existing test `testUpdatePriceFeedsPaysKeeperCorrectly` already acknowledges this gap in its comment: *"The real cost is more because of the gas used in the updatePriceFeeds function"* — but it does not account for the post-snapshot operations specifically. [6](#0-5)
