### Title
Unbounded O(L) Work in `GetLastStateProofProcess::execute` via `last_n_blocks=0` + Low `difficulty_boundary` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The per-message work guard in `GetLastStateProofProcess::execute` checks only the *input* fields `difficulties.len() + last_n_blocks * 2`, but the actual number of blocks processed by `complete_headers` is determined by `last_n_numbers.len()`, which in the else-branch equals `last_block_number - difficulty_boundary_block_number`. When an attacker sends `last_n_blocks=0` and a `difficulty_boundary` that resolves to block 0, `last_n_numbers` becomes `(0..L)` — the entire chain — while the guard evaluates to `0 > 1000 = false` and passes. The `GET_LAST_STATE_PROOF_LIMIT` constant is never consulted for the actual output size.

### Impact Explanation

A single malicious peer can send one `GetLastStateProof` message that forces the server to perform O(L) `get_ancestor` traversals and O(L) MMR root computations, where L is the full chain height (mainnet: ~13M+ blocks). Each `get_ancestor` is itself O(log L) in the skip-list implementation. Repeated at high frequency, this exhausts CPU and RocksDB I/O, causing node resource exhaustion and denial of service. No PoW, no stake, no privileged role is required — any peer can send this message.

---

### Likelihood Explanation

The attack requires knowing the genesis block's total difficulty (public, constant, derivable from the genesis block header) and the current tip hash (broadcast by the node). Both are trivially obtainable. The message is structurally valid and passes all existing input validation. The attack is repeatable with no rate limit specific to this message type beyond general peer connection limits.

---

### Recommendation

The guard must bound the *actual output size*, not just the input parameters. After computing `last_n_numbers` (and `sampled_numbers`, `reorg_last_n_numbers`), enforce:

```rust
if sampled_numbers.len() + last_n_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, cap `last_n_numbers` at `GET_LAST_STATE_PROOF_LIMIT` before calling `complete_headers`, and reject requests where the computed set would exceed the limit.
