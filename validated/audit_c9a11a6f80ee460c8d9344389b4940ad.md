### Title
Inaccurate Staleness Check in `getPricesNoOlderThan` Uses Aggregate MAX Timestamp Instead of Per-Feed Timestamps — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

`Scheduler.getPricesNoOlderThan` and `Scheduler.getEmaPricesNoOlderThan` validate freshness using `status.priceLastUpdatedAt`, which is the **maximum** `publishTime` across all feeds in the subscription. Individual feeds may carry timestamps far older than the requested age bound. The function then returns those stale per-feed prices without any per-feed timestamp check, silently violating the freshness guarantee implied by the function name.

### Finding Description

In `updatePriceFeeds`, after parsing and validating the update batch, the contract records:

```solidity
status.priceLastUpdatedAt = latestPublishTime;   // MAX publishTime across all feeds
_storePriceUpdates(subscriptionId, priceFeeds);  // stores each feed's actual (possibly old) price
``` [1](#0-0) 

`latestPublishTime` is computed as the maximum `publishTime` across all feeds:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [2](#0-1) 

The code explicitly allows feeds with arbitrarily old timestamps (e.g., closed equity markets) to pass the parse step:

```solidity
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
    ...
``` [3](#0-2) 

Then `getPricesNoOlderThan` checks only the aggregate MAX timestamp:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [4](#0-3) 

`getPricesUnsafe` returns the raw stored per-feed prices with no timestamp validation:

```solidity
for (uint i = 0; i < priceFeeds.length; i++) {
    prices[i] = priceFeeds[i].price;
}
``` [5](#0-4) 

The same flaw exists in `getEmaPricesNoOlderThan`: [6](#0-5) 

**Analogy to the reported bug:** In `_limitClose`, `_limitPrice` is set to the stored threshold (`_trade.tpPrice`) instead of the actual current price (`_price`). Here, the staleness check uses the stored aggregate MAX timestamp (`status.priceLastUpdatedAt`) instead of each feed's actual individual timestamp — the same pattern of substituting a stored/aggregate value for the true per-item value after a check passes.

### Impact Explanation

A consumer protocol calling `getPricesNoOlderThan(subscriptionId, priceIds, 60)` expects every returned price to be no older than 60 seconds. If the subscription contains a mix of crypto feeds (always active) and equity feeds (closed overnight/weekends), the staleness check passes because the crypto feed's recent timestamp satisfies `age_seconds`, but the equity feed's price — potentially hours or days old — is returned without any per-feed freshness check. Protocols making financial decisions (liquidations, collateral valuation, settlement) based on these prices would operate on stale data while believing it is fresh, potentially leading to incorrect liquidations, mispriced collateral, or exploitable arbitrage.

### Likelihood Explanation

The Scheduler is explicitly designed to support mixed-market subscriptions (the code comments acknowledge closed-market feeds). Any subscription combining crypto and equity/commodity feeds will exhibit this behavior on every update during non-trading hours. A keeper (unprivileged, anyone can call `updatePriceFeeds`) submits a valid batch; the MAX timestamp from the active crypto feed sets `priceLastUpdatedAt`; any whitelisted reader then calls `getPricesNoOlderThan` and receives stale equity prices. No special privileges or adversarial setup are required — this is a normal operational scenario.

### Recommendation

Replace the single aggregate timestamp check with a per-feed timestamp check inside `getPricesNoOlderThan` and `getEmaPricesNoOlderThan`:

```solidity
function getPricesNoOlderThan(
    uint256 subscriptionId,
    bytes32[] calldata priceIds,
    uint256 age_seconds
) external view override onlyWhitelistedReader(subscriptionId) returns (PythStructs.Price[] memory prices) {
    prices = this.getPricesUnsafe(subscriptionId, priceIds);
    for (uint i = 0; i < prices.length; i++) {
        if (distance(block.timestamp, prices[i].publishTime) > age_seconds)
            revert PythErrors.StalePrice();
    }
}
```

This mirrors how the standard `AbstractPyth.getPriceNoOlderThan` checks the individual price's `publishTime`: [7](#0-6) 

### Proof of Concept

1. Create a Scheduler subscription with two price IDs: `CRYPTO/USD` (always active) and `EQUITY/USD` (market hours only).
2. At 9 PM UTC (equity market closed), a keeper calls `updatePriceFeeds` with valid update data. The crypto feed has `publishTime = T_now`, the equity feed has `publishTime = T_now - 8h` (last close).
3. `_validateShouldUpdatePrices` computes `updateTimestamp = T_now` (MAX). The heartbeat check passes. `status.priceLastUpdatedAt = T_now`.
4. Both prices are stored: crypto at `T_now`, equity at `T_now - 8h`.
5. A consumer calls `getPricesNoOlderThan(subscriptionId, [EQUITY_ID], 60)`.
6. `distance(block.timestamp, status.priceLastUpdatedAt) = distance(T_now, T_now) = 0 ≤ 60` → check passes.
7. `getPricesUnsafe` returns the equity price with `publishTime = T_now - 8h` — 8 hours stale — with no revert. [8](#0-7)
