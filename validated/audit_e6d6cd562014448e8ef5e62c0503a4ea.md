Audit Report

## Title
Unbounded `vec!` Allocation in `WeightUnitsFlow::do_estimate` via High-Fee-Rate Transaction — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary

`Algorithm::max_bucket_index_by_fee_rate` maps a `FeeRate` to a bucket index with no upper bound. A transaction carrying a fee of ~10,000 CKB at minimum weight produces a fee rate of ~1.67×10¹³ shannons/KW, which maps to a bucket index of ~1.67×10⁸. `do_estimate` then attempts two `vec![0u64; max_bucket_index + 1]` allocations of ~1.34 GB each, causing the node process to be terminated by the OS OOM killer.

## Finding Description

**Root cause 1 — unbounded `max_bucket_index_by_fee_rate`:**

The final match arm in `max_bucket_index_by_fee_rate` applies a linear formula with no ceiling:

```rust
x => (x + t * 11_500) / (100 * t),   // t = 1000
```

For `x ≈ 1.67×10¹³` (fee rate from a 10,000 CKB fee / 60-byte tx), this yields `≈ 1.67×10⁸`. No cap or guard exists anywhere in the function.

**Root cause 2 — unchecked allocation in `do_estimate`:**

```rust
let mut buckets = vec![0u64; max_bucket_index + 1];   // line 219
...
let mut buckets = vec![0u64; max_bucket_index + 1];   // line 248
```

Both allocations use the raw index with no bounds check. At index ~1.67×10⁸, each allocation is ~1.34 GB (1.67×10⁸ × 8 bytes), totaling ~2.68 GB.

**Root cause 3 — `FeeRate::calculate` uses `saturating_mul`:**

```rust
FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
```

For `fee = 10,000 CKB = 10¹² shannons`, `weight = 60`: `fee_rate = 10¹² × 1000 / 60 ≈ 1.67×10¹³`. This is a valid, non-saturating result that passes through to the unbounded index formula.

**Root cause 4 — no fee-rate cap in tx-pool admission:** No fee-rate ceiling is enforced when a transaction enters the mempool. Any valid transaction with a high fee is admitted and stored in `all_entry_info`, which is passed directly to `do_estimate`.

**Exploit path:**

1. Attacker submits a valid transaction with fee ≥ 10,000 CKB and minimum weight (60 bytes).
2. `TxStatus::new_from_entry_info` computes `fee_rate ≈ 1.67×10¹³`.
3. Attacker (or any caller) invokes the `estimate_fee_rate` RPC (no authentication required).
4. `ExperimentRpcImpl::estimate_fee_rate` → `TxPoolController::estimate_fee_rate` → `process.rs::estimate_fee_rate` calls `get_all_entry_info` then `FeeEstimator::estimate_fee_rate` → `do_estimate`.
5. `max_bucket_index_by_fee_rate` returns `≈ 1.67×10⁸`; `vec![0u64; 1.67×10⁸+1]` is attempted twice (~2.68 GB total).
6. OOM killer terminates the node process.

**Existing guards are insufficient:** The only guard in `do_estimate` before the allocations is a `LackData` check on historical block count, which is unrelated to fee-rate magnitude. No bounds check on `max_bucket_index` exists anywhere in the call chain.

## Impact Explanation

On any node with `WeightUnitsFlow` configured, a single `estimate_fee_rate` RPC call while a high-fee-rate transaction is in the mempool causes the tx-pool service thread to attempt a multi-GB heap allocation, crashing the node process via OOM. This matches the allowed impact: **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**.

## Likelihood Explanation

`WeightUnitsFlow` requires explicit opt-in in `[fee_estimator]` config, limiting the affected population to nodes that have enabled it. For nodes that do enable it: the attacker needs ~10,000 CKB temporarily in the mempool (not permanently burned if the node crashes before block confirmation). The `estimate_fee_rate` RPC is unauthenticated and publicly accessible with no parameters required. The attack is repeatable within each ~28-second block interval. The economic barrier is low relative to the impact on affected nodes.

## Recommendation

Cap the result of `max_bucket_index_by_fee_rate` to a safe constant before use:

```rust
const MAX_BUCKET_INDEX: usize = 1000;

fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
    let t = FEE_RATE_UNIT;
    let index = match fee_rate.as_u64() {
        x if x <= 10_000 => x / t,
        x if x <= 50_000 => (x + t * 10) / (2 * t),
        x if x <= 200_000 => (x + t * 100) / (5 * t),
        x if x <= 500_000 => (x + t * 400) / (10 * t),
        x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
        x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
        x => (x + t * 11_500) / (100 * t),
    };
    (index as usize).min(MAX_BUCKET_INDEX)
}
```

Alternatively, add a guard in `do_estimate` before both `vec!` allocations:

```rust
let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
    .min(MAX_BUCKET_INDEX);
```

The constant `MAX_BUCKET_INDEX = 1000` is consistent with the existing test coverage range (fee rates up to 3,000,000 shannons/KW map to indices ≤ ~135 in normal operation) and bounds the maximum allocation to 8 KB per `vec!`.

## Proof of Concept

```rust
#[test]
fn test_no_oom_on_high_fee_rate() {
    // fee = 10_000 CKB = 10^12 shannons, weight = 60
    // fee_rate = 10^12 * 1000 / 60 ≈ 1.67e13
    // max_bucket_index ≈ 1.67e8 → ~1.34 GB per vec → OOM
    let fee_rate = FeeRate::calculate(
        ckb_types::core::Capacity::shannons(1_000_000_000_000u64),
        60,
    );
    let idx = Algorithm::max_bucket_index_by_fee_rate(fee_rate);
    assert!(idx < 10_000, "bucket index {} is unbounded", idx);
}
```

Running this test on an unpatched node will either OOM-kill the test process or confirm the index is ~167,000,000, demonstrating the allocation would be ~1.34 GB per `vec!` call. The test can be added directly to the existing `#[cfg(test)]` block in `util/fee-estimator/src/estimator/weight_units_flow.rs`.
