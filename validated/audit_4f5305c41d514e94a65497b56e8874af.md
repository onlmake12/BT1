### Title
Unbounded SQL OFFSET in `get_tx_with_cell` Allows O(n) DB Scan via Crafted Cursor — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

### Summary
The ungrouped `get_transactions` RPC path decodes the `after` cursor into a `(last_id: i64, offset: i32)` pair and passes the `offset` field directly to a SQL `OFFSET` clause with no upper-bound validation. Any unprivileged caller can craft a cursor whose offset field encodes an arbitrarily large value, forcing the SQL engine to scan and discard that many rows on every request.

### Finding Description

**Cursor parsing — no range check on offset:**

In `get_transactions`, the `after` bytes are split and decoded: [1](#0-0) 

`decode_i32` only validates that the slice is exactly 4 bytes; it performs no range check on the resulting value: [2](#0-1) 

**Offset injected directly into SQL:**

`get_tx_with_cell` passes the decoded offset straight to `query_builder.offset()`: [3](#0-2) 

This produces a query of the form:
```sql
SELECT ... ORDER BY tx_id ASC LIMIT <limit> OFFSET <attacker_value>
```

SQL `OFFSET` semantics require the engine to materialize and discard `offset` rows before returning results. With SQLite or PostgreSQL, this is O(offset) work regardless of the actual result set size.

**Contrast with grouped mode:**

The grouped path (`get_tx_with_cells`) uses a keyset cursor (`WHERE tx_id > after`) with no OFFSET, making it O(1) per page: [4](#0-3) 

The ungrouped path was not given the same treatment.

### Impact Explanation
An unprivileged caller submits repeated `get_transactions` requests with `group_by_transaction=false` and a crafted `after` cursor encoding `offset = 2^30`. Each request forces the DB to scan ~10^9 rows. On a node with a widely-used lock script (e.g., the secp256k1 default lock), this saturates the DB thread pool and degrades indexer responsiveness for all users. No authentication, PoW, or privileged access is required.

### Likelihood Explanation
The RPC is exposed over HTTP with no authentication by default. The cursor format (12 bytes, last 4 = offset LE) is documented implicitly by the returned `last_cursor` value and is trivially reversible. The attack requires a single crafted JSON-RPC call.

### Recommendation
Replace the SQL `OFFSET`-based pagination with a keyset approach, mirroring the grouped path. The cursor should encode `(last_tx_id, last_io_type, last_io_index)` as a composite key, and the query should use `WHERE (tx_id, io_type, io_index) > (last_tx_id, last_io_type, last_io_index)` instead of `OFFSET`. If OFFSET must be retained, cap it at `limit` (its only legitimate value is at most `limit - 1` to re-anchor within a tie group).

### Proof of Concept

1. Enable rich-indexer on a CKB node with a script that appears in many transactions.
2. Call `get_transactions` with `group_by_transaction: false`, `limit: 1`, no `after` → receive a valid `last_cursor`.
3. Decode the 12-byte cursor; replace the last 4 bytes with `\x00\x00\x00\x40` (offset = 2^30 in LE).
4. Re-encode and submit as `after` in a new `get_transactions` call.
5. Observe that query latency grows proportionally to the crafted offset value, confirming O(offset) DB scan cost.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
