### Title
Admin Role Can Prevent Its Own Removal by Overwriting Owner's Pending Admin Proposal — (`EntropyGovernance.sol` / `SchedulerGovernance.sol`)

---

### Summary

Both `EntropyGovernance` and `SchedulerGovernance` implement a two-step admin transfer: the owner (or admin) calls `proposeAdmin(newAdmin)`, then `newAdmin` calls `acceptAdmin()`. Because `proposeAdmin` is gated by `_authoriseAdminAction()` / `_authorizeAdminAction()`, which permits **either the owner or the current admin** to call it, the current admin can front-run or overwrite any pending owner-initiated admin replacement, preventing their own removal indefinitely through the standard governance path.

---

### Finding Description

In `EntropyGovernance.sol`:

```solidity
function proposeAdmin(address newAdmin) public virtual {
    require(newAdmin != address(0), "newAdmin is zero address");
    _authoriseAdminAction();          // passes for owner OR current admin
    _state.proposedAdmin = newAdmin;  // unconditionally overwrites any pending proposal
    emit NewAdminProposed(_state.admin, newAdmin);
}
```

`_authoriseAdminAction()` in `EntropyUpgradable` is:

```solidity
function _authoriseAdminAction() internal view override {
    if (msg.sender != owner() && msg.sender != _state.admin)
        revert EntropyErrors.Unauthorized();
}
```

The identical pattern exists in `SchedulerGovernance.sol` / `SchedulerUpgradeable.sol` via `_authorizeAdminAction()`.

The `proposedAdmin` slot is a single storage variable. There is no access-control distinction between "owner proposing a replacement" and "admin proposing a replacement." The current admin can therefore:

1. Watch the mempool (or simply react on-chain) for the owner's `proposeAdmin(newAdmin)` call.
2. Immediately call `proposeAdmin(self_or_ally)`, overwriting `_state.proposedAdmin`.
3. Repeat this every time the owner retries.

The owner's only escape hatch is a full UUPS contract upgrade (`upgradeTo`), which requires deploying a new implementation that passes `magicCheck`. This is a significant operational burden and may not be feasible under time pressure or in chains where the upgrade key is held in a multisig with a slow signing process.

---

### Impact Explanation

- The admin role in `EntropyUpgradable` and `SchedulerUpgradeable` can entrench itself by continuously overwriting the owner's `proposeAdmin` calls.
- The owner cannot replace the admin through the intended governance path (`proposeAdmin` → `acceptAdmin`) as long as the admin is willing to front-run.
- Admin controls `setPythFee`, `setDefaultProvider`, `withdrawFee` (Entropy) and `setSingleUpdateKeeperFeeInWei`, `setMinimumBalancePerFeed` (Scheduler). A malicious entrenched admin can drain fees or manipulate protocol parameters indefinitely.
- The owner's only recourse — a UUPS upgrade — is a heavyweight operation that may be blocked by operational constraints.

---

### Likelihood Explanation

The admin is a semi-trusted role that is explicitly designed to be replaceable by the owner. If the admin key is compromised or the admin acts adversarially, the owner has no lightweight path to remove them. The attack requires only that the admin (or whoever controls the admin key) monitors the chain and calls `proposeAdmin` to overwrite the owner's proposal — a single transaction with no special resources.

---

### Recommendation

Restrict `proposeAdmin` so that **only the owner** can call it. The admin should not be able to nominate their own successor or overwrite the owner's nomination:

```solidity
function proposeAdmin(address newAdmin) public virtual {
    require(newAdmin != address(0), "newAdmin is zero address");
    require(msg.sender == owner(), "only owner can propose admin");
    _state.proposedAdmin = newAdmin;
    emit NewAdminProposed(_state.admin, newAdmin);
}
```

This mirrors the correct design intent stated in the NatSpec ("The owner will have all the power over it") and is consistent with how `_authorizeUpgrade` is already restricted to `onlyOwner`.

---

### Proof of Concept

```
1. Deploy EntropyUpgradable with owner=Alice, admin=Bob.
2. Alice calls proposeAdmin(Carol).          // _state.proposedAdmin = Carol
3. Bob calls proposeAdmin(Bob).              // _state.proposedAdmin = Bob  (overwrites)
4. Carol calls acceptAdmin() → reverts (she is no longer proposedAdmin).
5. Bob calls acceptAdmin() → succeeds; Bob remains admin.
6. Alice retries proposeAdmin(Carol) → Bob overwrites again.
7. Loop continues indefinitely; Bob is permanently entrenched.
```
