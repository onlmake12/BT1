### Title
Keeper Fees Can Drain Subscription Balance Below `minimumBalance`, Locking Manager Funds in Active Subscription - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

In `Scheduler.sol`, the `_processFeesAndPayKeeper` function deducts keeper fees from a subscription's `balanceInWei` without enforcing the `minimumBalance` floor. Once fees push the balance below `minimumBalance`, the subscription manager cannot call `withdrawFunds` on the still-active subscription for any nonzero amount, because the post-withdrawal balance check always fails. The manager's only recovery path is to deactivate the subscription first — sacrificing live price-feed delivery — before withdrawing the stranded ETH.

### Finding Description

`_processFeesAndPayKeeper` deducts the full keeper fee (gas cost + per-feed fee) from `status.balanceInWei` with only a "balance ≥ fee" check: [1](#0-0) 

There is no lower bound preventing the post-fee balance from falling below `minimumBalance`. After this deduction, `withdrawFunds` enforces for active subscriptions: [2](#0-1) 

If `status.balanceInWei` is already below `minimumBalance`, then `status.balanceInWei - amount < minimumBalance` is true for every `amount > 0`, so every withdrawal attempt reverts with `InsufficientBalance`.

`addFunds` also enforces the minimum after crediting the deposit: [3](#0-2) 

So the manager cannot top up incrementally; they must deposit enough in a single call to clear the minimum, or deactivate first.

`getMinimumBalance` is `numPriceFeeds × minimumBalancePerFeed`: [4](#0-3) 

The minimum is a non-trivial ETH amount that scales with the number of subscribed feeds.

### Impact Explanation

A subscription manager whose balance has been drained below `minimumBalance` by keeper fees cannot withdraw any ETH while the subscription remains active. Their only recovery path is to call `updateSubscription` with `isActive = false` (deactivating the subscription), then call `withdrawFunds`. Deactivation removes the subscription from the active list and stops all price-feed updates, which can break downstream protocols that depend on the subscription for live prices. The manager must choose between losing access to their remaining ETH or disrupting their protocol's price-feed availability.

### Likelihood Explanation

This condition arises naturally for any subscription that was funded close to the minimum balance. Each `updatePriceFeeds` call deducts `gasCost + singleUpdateKeeperFeeInWei × numFeeds` from the balance: [5](#0-4) 

Over time, repeated keeper updates will push any subscription toward and then below the minimum. The existing test `testAddFundsEnforcesMinimumBalance` explicitly demonstrates that keeper updates can drain the balance below minimum: [6](#0-5) 

### Recommendation

In `withdrawFunds`, allow the manager to withdraw the full remaining balance even when it is already below `minimumBalance` (i.e., only enforce the minimum floor when the pre-withdrawal balance is at or above the minimum):

```solidity
if (params.isActive) {
    uint256 minimumBalance = this.getMinimumBalance(uint8(params.priceIds.length));
    // Only enforce minimum if balance is currently above it
    if (status.balanceInWei >= minimumBalance &&
        status.balanceInWei - amount < minimumBalance) {
        revert SchedulerErrors.InsufficientBalance();
    }
}
```

This allows managers to recover stranded funds without being forced to deactivate their subscription, while still preventing voluntary withdrawals that would drop a healthy balance below the minimum.

### Proof of Concept

1. Manager creates a subscription with 2 price feeds, depositing exactly `minimumBalance` (e.g., `2 × minimumBalancePerFeed`).
2. A keeper calls `updatePriceFeeds` repeatedly. Each call deducts `gasCost + 2 × singleUpdateKeeperFeeInWei` from `balanceInWei` via `_processFeesAndPayKeeper`.
3. After enough updates, `balanceInWei` falls below `minimumBalance` (confirmed by the existing test at line 750–758 of `PulseScheduler.t.sol`).
4. Manager calls `withdrawFunds(subscriptionId, balanceInWei)` — reverts with `InsufficientBalance` because `balanceInWei - balanceInWei (= 0) < minimumBalance`.
5. Manager calls `withdrawFunds(subscriptionId, 1 wei)` — also reverts for the same reason.
6. Manager is forced to call `updateSubscription(..., isActive=false)` to deactivate, then `withdrawFunds` — recovering funds but permanently stopping price-feed updates for their protocol. [7](#0-6)

```text
    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
```
