Looking at the exact code path in `pyth.move` and `price_info.move`:

### Title
Fee Taken Without State Update on Equal-Timestamp Submission, Enabling Relayer Fee Griefing — (`target_chains/sui/contracts/sources/pyth.move`)

---

### Summary

`update_single_price_feed` unconditionally deposits the caller's fee into the `PriceInfoObject` before checking freshness. When the submitted update's `publish_time` equals the cached timestamp, `is_fresh_update` returns `false` (strict `>`), `update_cache` silently no-ops, and the fee is permanently lost with no state change and no error signal. An attacker can front-run any legitimate relayer by replaying the same valid VAA first, causing the legitimate relayer's subsequent identical submission to hit the equal-timestamp path and lose their fee.

---

### Finding Description

The fee-before-freshness ordering in `update_single_price_feed` is the root cause:

```
// pyth.move line 277 — fee deposited unconditionally
price_info::deposit_fee_coins(price_info_object, fee);

// pyth.move line 287 — freshness checked only after fee is taken
update_cache(latest_only, cur_price_info, price_info_object, clock);
``` [1](#0-0) 

Inside `update_cache`, the freshness gate uses strict greater-than:

```move
update_timestamp > cached_timestamp   // line 337
``` [2](#0-1) 

When `update_timestamp == cached_timestamp`, `is_fresh_update` returns `false`, the `if` body is skipped entirely, and `update_cache` returns without calling `update_price_info_object` or aborting: [3](#0-2) 

`deposit_fee_coins` accumulates coins into the `PriceInfoObject` with no withdrawal path: [4](#0-3) 

There is no `withdraw_fee_coins` or refund function anywhere in `price_info.move` or `pyth.move`. Once deposited, the fee is irrecoverable by the caller.

---

### Impact Explanation

Any relayer who submits a price update whose `publish_time` equals the currently cached timestamp loses their `base_update_fee` with no state change and no error. The fee is permanently locked in the `PriceInfoObject`. The attacker's cost is one `base_update_fee` to grief the legitimate relayer out of one `base_update_fee` — a 1:1 griefing ratio. At scale (many price feeds, many relayers), this is a viable sustained griefing campaign.

---

### Likelihood Explanation

The attack requires no privileged access and no key material. The attacker only needs to:

1. Observe a pending `update_single_price_feed` transaction (Sui validators see pending transactions before ordering).
2. Copy the identical accumulator message / VAA bytes verbatim — no forgery needed.
3. Submit first. Their transaction succeeds and advances the cached timestamp to `T+1`.
4. The legitimate relayer's transaction executes next: `update_timestamp (T+1) == cached_timestamp (T+1)` → fee taken, no update, no abort.

The VAA is already Wormhole-verified and data-source-checked before `update_single_price_feed` is called, so the attacker passes all upstream guards by construction (they are replaying a valid message). Sui's object-locking model means both transactions cannot execute in the same checkpoint, so the attacker's submission ordering is sufficient.

---

### Recommendation

Two complementary fixes:

1. **Move fee collection after the freshness check**, or make it conditional on a successful update. If `is_fresh_update` returns `false`, return the fee coin to the caller rather than depositing it.

2. **Change the freshness comparison to `>=`** if equal-timestamp updates should be treated as no-ops without fee collection, or add an explicit abort (`E_STALE_PRICE_UPDATE`) so the transaction reverts and the fee is never taken.

The cleanest fix is to restructure `update_single_price_feed` so that `deposit_fee_coins` is only called when `is_fresh_update` returns `true`, preserving the invariant that a fee payment always results in either a state update or a full transaction revert (and thus an automatic coin refund via Sui's abort semantics).

---

### Proof of Concept

State-transition test (pseudocode matching the Sui Move test framework):

```move
// 1. Initialize with cached timestamp T
let price_info_object = create_price_info_object_with_timestamp(T);

// 2. Attacker submits valid VAA with publish_time = T+1 → succeeds, cached = T+1
let vaa = get_valid_vaa_with_timestamp(T + 1);
update_single_price_feed(&pyth_state, vaa, &mut price_info_object, fee_attacker, &clock);
assert!(get_cached_timestamp(&price_info_object) == T + 1);

// 3. Legitimate relayer submits the SAME VAA (publish_time = T+1 == cached T+1)
update_single_price_feed(&pyth_state, vaa, &mut price_info_object, fee_relayer, &clock);

// 4. Assert: fee was taken
assert!(price_info::get_balance(&price_info_object) == 2 * base_update_fee);

// 5. Assert: price info unchanged (still T+1, same values as after step 2)
assert!(get_cached_timestamp(&price_info_object) == T + 1);
// No abort was raised — relayer has no indication their fee was wasted
```

The fee at step 4 is `2 × base_update_fee` (attacker + relayer), both locked in the object. The relayer's fee is gone with no state change and no error.

    fun is_fresh_update(update: &PriceInfo, price_info_object: &PriceInfoObject): bool {
        // Get the timestamp of the update's current price
        let price_feed = price_info::get_price_feed(update);
        let update_timestamp = price::get_timestamp(&price_feed::get_price(price_feed));

        // Get the timestamp of the cached data for the price identifier
        let cached_price_info = price_info::get_price_info_from_price_info_object(price_info_object);
        let cached_price_feed =  price_info::get_price_feed(&cached_price_info);
        let cached_timestamp = price::get_timestamp(&price_feed::get_price(cached_price_feed));

        update_timestamp > cached_timestamp
    }
