Audit Report

## Title
Insert-Before-Check Ordering in `add_ask_for_txs` Allows Global Queue Exhaustion and Transaction Relay Disruption - (File: sync/src/types/mod.rs)

## Summary
`SyncState::add_ask_for_txs` unconditionally inserts up to 32,767 hashes per peer into the global `unknown_tx_hashes` queue before evaluating whether the global limit (`MAX_UNKNOWN_TX_HASHES_SIZE = 50,000`) has been exceeded. An attacker using two P2P connections can fill the queue with fake hashes, causing all subsequent `RelayTransactionHashes` messages from legitimate peers to return `Status::ignored()`, silently preventing the victim node from requesting those transactions. The attack window per cycle is ~30 seconds and can be sustained with IP rotation.

## Finding Description
In `add_ask_for_txs` ( [1](#0-0) ), all incoming hashes are inserted into `unknown_tx_hashes` unconditionally before any limit check. The global-size and per-peer checks only execute after insertion ( [2](#0-1) ).

The per-peer ban (`TooManyUnknownTransactions`) fires only when the current peer's contribution reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32,767`. A second attacker peer whose per-peer count is below that threshold receives only `Status::ignored()` — no ban, no disconnect — while its fake hashes remain in the queue. Legitimate peers arriving after the queue is full also receive `Status::ignored()`, so the victim node never issues `GetRelayTransactions` for their hashes.

Fake entries do drain: `pop_ask_for_txs` ( [3](#0-2) ) only re-queues an entry when `next_request_peer()` returns `Some`; for a single-peer entry with `requested=true`, `next_request_peer()` returns `None` ( [4](#0-3) ), so the entry is silently dropped after ~30 seconds (`RETRY_ASK_TX_TIMEOUT_INCREASE`). The attacker sustains the DoS by reconnecting with a new IP every ~30 seconds; the ban duration is only 5 minutes per identity ( [5](#0-4) ).

The entry path `TransactionHashesProcess::execute` enforces only a per-message size check before calling `add_ask_for_txs` ( [6](#0-5) ), providing no pre-insertion global limit guard.

Constants confirmed: `MAX_RELAY_TXS_NUM_PER_BATCH = MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32,767`, `MAX_UNKNOWN_TX_HASHES_SIZE = 50,000` ( [7](#0-6) ).

## Impact Explanation
When the queue is saturated, the victim node silently drops all incoming transaction hash announcements from legitimate peers. Miners on the victim node miss mempool transactions; users whose transactions propagate only through the victim's peers experience delayed or missed confirmations. Targeted at multiple nodes simultaneously, this disrupts CKB network-wide transaction propagation at negligible cost, matching the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* (10,001–15,000 points).

## Likelihood Explanation
- Any unauthenticated P2P peer can send `RelayTransactionHashes`; no stake, PoW, or privilege is required.
- Two connections suffice to fill the 50,000-entry queue (32,767 + 17,233 unique fake hashes, ~2 MB total).
- The attack cycle is ~30 seconds. Sustaining it requires one new source IP per cycle — feasible with a modest botnet or IP-rotation service.
- The existing integration test `TooManyUnknownTransactions` ( [8](#0-7) ) covers only the single-peer ban path and does not test cross-peer queue exhaustion.

## Recommendation
1. **Check limits before inserting**: Move the global and per-peer limit checks to the top of `add_ask_for_txs`, before the insertion loop. Reject or truncate input if limits would be exceeded.
2. **Evict entries from banned peers**: When a peer is banned, remove all `unknown_tx_hashes` entries exclusively associated with that peer (`priority.peers == [banned_peer]`), eliminating the ~30-second pollution window.
3. **Rate-limit `RelayTransactionHashes` per peer**: Apply the existing `rate_limiter` in `Relayer` to this message type to reduce the fill rate.

## Proof of Concept
1. Attacker connects as peer A, sends one `RelayTransactionHashes` with 32,767 unique non-existent hashes. `add_ask_for_txs` inserts all 32,767 → `peer_unknown_counter = 32,767 >= MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` → peer A banned (5 min). 32,767 fake hashes remain in queue.
2. Attacker connects as peer B (new IP), sends 17,233 more unique fake hashes. Queue reaches 50,000. Peer B's per-peer count = 17,233 < 32,767 → `Status::ignored()` (no ban).
3. Legitimate peer C sends real tx hashes → `unknown_tx_hashes.len() >= 50,000` → `Status::ignored()` → victim never fetches those transactions.
4. After ~30 seconds, `pop_ask_for_txs` drains the fake entries (single-peer entries with `requested=true` and `peers.len()==1` are dropped via `next_request_peer()` returning `None`).
5. Attacker connects as peer D (new IP) and repeats steps 1–2 to refill the queue, sustaining the DoS indefinitely.

    pub fn execute(self) -> Status {
        let state = self.relayer.shared().state();
        {
            let relay_transaction_hashes = self.message;
            if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "TxHashes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    relay_transaction_hashes.tx_hashes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
        }

        let tx_hashes: Vec<_> = {
            let mut tx_filter = state.tx_filter();
            tx_filter.remove_expired();
            self.message
                .tx_hashes()
                .iter()
                .map(|x| x.to_entity())
                .filter(|tx_hash| !tx_filter.contains(tx_hash))
                .collect()
        };

        state.add_ask_for_txs(self.peer, tx_hashes)
    }
