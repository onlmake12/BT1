Audit Report

## Title
Shared `VerifyQueue` Has No Per-Sender Quota, Allowing a Single Actor to Exhaust the 256 MB Global Limit and Block All Legitimate Transaction Submissions — (`tx-pool/src/component/verify_queue.rs`)

## Summary

The `VerifyQueue` enforces a single global 256 MB cap (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`) with no per-peer or per-RPC-caller quota. A single P2P peer can submit ~500 structurally valid transactions carrying fabricated (non-existent) input `OutPoint`s to fill the entire queue. Once full, every subsequent submission from any source is rejected with `Reject::Full` until workers drain the attacker's entries, and the attacker can immediately re-fill to sustain the denial indefinitely without being banned.

## Finding Description

**Root cause — flat global admission check with no per-peer accounting:**

`VerifyQueue` holds only a single `total_tx_size: usize` counter and no per-peer map: [1](#0-0) 

`is_full` is a flat global comparison: [2](#0-1) 

When full, `add_tx` immediately rejects with `Reject::Full`: [3](#0-2) 

**Exploit path — fabricated OutPoints bypass the only pre-queue check:**

All remote submission paths converge on `resumeble_process_tx`, which calls only `non_contextual_verify` before `enqueue_verify_queue`: [4](#0-3) 

`non_contextual_verify` validates structure, enforces the 512 KB per-transaction size ceiling, and rejects cellbase transactions — but performs no on-chain `OutPoint` resolution: [5](#0-4) 

`TRANSACTION_SIZE_LIMIT = 512 * 1_000 = 512,000 bytes`, so `256,000,000 / 512,000 = 500` transactions fill the queue: [6](#0-5) 

**Why existing defenses are insufficient:**

The fee-rate check (`check_tx_fee`) is performed inside `_process_tx` → `pre_check`, which runs only after a worker dequeues the transaction — not before admission. Transactions with fabricated OutPoints are admitted to the queue without any fee check.

`remove_txs_by_peer` exists but is only called inside `ban_malformed`: [7](#0-6) 

`ban_malformed` is only triggered when `reject.is_malformed_tx()` returns `true`. A transaction whose OutPoint is unknown resolves to `Reject::Resolve(OutPointError::Unknown)`, which is explicitly not in the malformed set: [8](#0-7) 

Consequently, the attacker is never banned and `remove_txs_by_peer` is never invoked during the attack. In `after_process`, unknown-OutPoint failures are treated as missing-input orphans, not malformed transactions: [9](#0-8) 

## Impact Explanation

**High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single malicious P2P peer can exhaust the 256 MB `VerifyQueue` of any targeted node, preventing that node from accepting any new transactions — blocking its participation in transaction relay and block assembly. Applied to multiple nodes simultaneously, this causes network-wide transaction relay congestion at negligible cost. [10](#0-9) 

## Likelihood Explanation

**High for P2P attacker.** Any connected peer can relay transactions via `submit_remote_tx`: [11](#0-10) 

Filling 256 MB requires ~500 transactions (each up to 512 KB), trivial to construct with one fabricated input `OutPoint` and a large witness blob. There is no connection-level rate limiting on transaction relay. The attacker is never banned (unknown OutPoints are not malformed), can reconnect after disconnect, and can continuously re-fill as workers drain entries. **Medium for local RPC attacker**, since the RPC is bound to `127.0.0.1` by default, requiring co-location.

## Recommendation

1. **Introduce per-peer quota in `VerifyQueue`**: Add a `HashMap<PeerIndex, usize>` tracking each peer's contribution to `total_tx_size`. Cap each peer's share (e.g., `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE / max_peers`). The existing `remove_txs_by_peer` infrastructure already iterates by peer — extend it to enforce admission limits. [12](#0-11) 

2. **Per-peer eviction when full**: Instead of a flat `Reject::Full`, evict the lowest-priority transaction from the peer holding the largest queue share, analogous to how `limit_size` evicts by fee rate in the main pool.

3. **Apply minimum fee-rate check before queue admission**: Move the fee-rate floor check ahead of `enqueue_verify_queue` so zero-fee or dust transactions cannot cheaply fill the queue.

## Proof of Concept

1. Construct ~500 transactions, each with a witness blob approaching 512 KB and a fabricated (non-existent) input `OutPoint`. Each passes `non_contextual_verify` (valid structure, size ≤ `TRANSACTION_SIZE_LIMIT`, not cellbase).
2. Submit all 500 via P2P relay (`submit_remote_tx`) in rapid succession. Each reaches `enqueue_verify_queue` and is accepted, accumulating ~256 MB of `total_tx_size`.
3. The 501st `send_transaction` from any caller returns `PoolIsFull: verify_queue total_tx_size exceeded`.
4. Workers process the attacker's transactions; each fails contextual resolution (`OutPointError::Unknown`) and is treated as an orphan — the attacker is never banned. The attacker immediately re-submits to refill.
5. Legitimate transactions are continuously rejected for as long as the attacker maintains the flood. [13](#0-12)

```rust
    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }
```

```rust
pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
```
