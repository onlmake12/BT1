### Title
Unbounded `pending_get_block_proposals` Map Growth via Repeated `GetBlockProposal` Relay Messages — (`sync/src/types/mod.rs`)

---

### Summary

An unprivileged P2P peer can send a continuous stream of `GetBlockProposal` relay messages, each carrying up to `max_block_proposals_limit × max_uncles_num` (1 500 × 2 = 3 000) unique `ProposalShortId` values that are absent from the local tx-pool. Every such ID is unconditionally inserted into the global `pending_get_block_proposals` `DashMap` with no per-peer quota and no total-size cap, causing unbounded memory growth until the periodic drain fires. This is a direct analog to the Union Finance griefing attack: an attacker-controlled list grows without a safe upper bound, exhausting a shared resource.

---

### Finding Description

**Root cause — `insert_get_block_proposals` has no size guard**

`sync/src/types/mod.rs` lines 1594–1600:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [1](#0-0) 

Every `ProposalShortId` in `ids` is inserted into the shared `DashMap<ProposalShortId, HashSet<PeerIndex>>` with no check on the map's current size and no per-peer quota.

**Caller — `GetBlockProposalProcess::execute`**

`sync/src/relayer/get_block_proposal_process.rs` lines 67–77:

```rust
// Transactions that do not exist on this node
let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
    .into_iter()
    .filter(|short_id| !fetched_transactions.contains_key(short_id))
    .collect();

    .shared()
    .state()

``` [2](#0-1) 

The only guard before this call is a per-message length check against `max_block_proposals_limit × max_uncles_num`: [3](#0-2) 

That check limits a **single message** to ≤ 3 000 IDs, but places no restriction on how many messages a peer may send, nor on the cumulative size of the map across messages.

**Contrast with the analogous `add_ask_for_txs`**

The structurally identical `unknown_tx_hashes` map is protected by both a per-peer `.take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)` truncation and a global `MAX_UNKNOWN_TX_HASHES_SIZE` ceiling that returns `TooManyUnknownTransactions` when exceeded: [4](#0-3) 

`pending_get_block_proposals` has neither protection.

**Drain is periodic, not immediate**

`drain_get_block_proposals` clones and clears the map on a timer: [5](#0-4) 

Between two consecutive drain ticks an attacker can insert an unbounded number of entries.

---

### Impact Explanation

Each `ProposalShortId` key is 10 bytes; each `HashSet<PeerIndex>` value adds heap overhead. A single peer sending 3 000-ID messages at network speed can grow the map to millions of entries between drain ticks, exhausting heap memory and causing the node process to be killed by the OS OOM killer or to stall under allocation pressure. This denies service to all honest peers connected to the victim node.

---

### Likelihood Explanation

The attack requires only a single connected peer sending standard relay protocol messages. No stake, no key, no majority hash-power is needed. The per-message limit (3 000 IDs) is large enough that even a modest message rate (e.g., 100 msg/s) inserts 300 000 entries per second. The proposal IDs need not correspond to real transactions — any 10-byte values not present in the local tx-pool suffice.

---

### Recommendation

Apply the same two-layer defence used by `add_ask_for_txs`:

1. **Per-peer quota**: inside `insert_get_block_proposals`, count how many entries in `pending_get_block_proposals` already reference `pi`; reject or truncate once a per-peer ceiling (e.g., `MAX_BLOCK_PROPOSALS_LIMIT`) is reached.
2. **Global cap**: before inserting, check `pending_get_block_proposals.len()` against a configurable maximum (e.g., `MAX_BLOCK_PROPOSALS_LIMIT × max_peers`); return an error status and optionally ban the peer if the cap is exceeded.
3. **Rate-limit** `GetBlockProposal` messages at the network layer, consistent with how other relay messages are rate-limited.

---

### Proof of Concept

1. Connect a single peer to a victim CKB node.
2. In a tight loop, send `RelayMessage::GetBlockProposal` messages, each containing `max_block_proposals_limit × max_uncles_num` (3 000) freshly generated `ProposalShortId` values that are not present in the victim's tx-pool.
3. Each message passes the per-message length check at `get_block_proposal_process.rs:40`.
4. All 3 000 IDs per message are forwarded to `insert_get_block_proposals` and inserted into `pending_get_block_proposals` with no size check.
5. Observe the victim node's RSS grow monotonically between drain ticks; at sufficient message rate the node exhausts available memory and crashes or becomes unresponsive.

The absence of any size guard in `insert_get_block_proposals` (contrast with the explicit `MAX_UNKNOWN_TX_HASHES_SIZE` guard in `add_ask_for_txs`) confirms the root cause is in production code, reachable by any unprivileged relay peer.
