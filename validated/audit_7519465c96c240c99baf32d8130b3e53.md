### Summary

The `execute_governance_instruction` function uses a strictly-greater-than (`>`) sequence check instead of a strict-equality (`==`) check. Any permissionless relayer who possesses a legitimately-signed governance VAA with a high sequence number can submit it first, permanently rendering all lower-sequence governance VAAs unexecutable.

### Finding Description

In `governance.move`, the sequence guard is:

```move

    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);
``` [1](#0-0) 

Immediately after, the state is updated to the submitted sequence:

```move

``` [2](#0-1)

The `last_executed_governance_sequence` is initialized to `0` in `state.move`: [6](#0-5)

### Impact Explanation

If governance issues VAAs with sequences 1 through N (e.g., a batch of data-source updates, fee changes, or a contract upgrade followed by parameter resets), a malicious relayer who observes all of them on the Wormhole network can submit VAA #N first. After that transaction, `last_executed_governance_sequence = N`. Every VAA with sequence 1 through N-1 now permanently fails the `> N` check with `E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER`. Those governance actions â€” which may include critical data-source additions, fee corrections, or stale-price-threshold changes â€” are irreversibly censored with no recovery path short of issuing entirely new governance VAAs.

---

### Likelihood Explanation

- Both entry points (`verify_vaa`, `execute_governance_instruction`) are `public`; no privileged role is required to relay.
- Wormhole VAAs are publicly observable once signed by the guardian network; the attacker needs no special access.
- The attack is most likely during governance batches (e.g., a contract upgrade VAA followed by several parameter-update VAAs issued in the same governance cycle), which is a realistic operational scenario.
- The attack is a single transaction and is locally testable exactly as described in the proof of concept.

---

### Recommendation

Replace the `>` check with a strict-equality check (`==`) so that governance VAAs must be executed in exact sequence order:

```move
assert!(sequence == state::get_last_executed_governance_sequence(pyth_state) + 1,
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);
```

Apply the same fix to `contract_upgrade.move`. This matches the design intent stated in the comment ("Require that new sequence number is greater than last executed sequence number") while closing the gap-skipping attack surface.

---

### Proof of Concept

State-transition test (as described):

1. Deploy with `last_executed_governance_sequence = 0`.
2. Obtain two legitimately-signed governance VAAs: `seq=500` (e.g., a data-source update) and `seq=1000` (e.g., a fee update). Both are valid Wormhole VAAs from the Pyth governance emitter.
3. Call `verify_vaa` + `execute_governance_instruction` with `seq=1000`. Passes: `1000 > 0`. State updates to `last_executed_governance_sequence = 1000`.
4. Call `verify_vaa` + `execute_governance_instruction` with `seq=500`. Fails: `500 > 1000` is false â†’ aborts with `E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER`.
5. The data-source update in `seq=500` is permanently unexecutable.
