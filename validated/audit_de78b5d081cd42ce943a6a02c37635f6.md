### Title
Verification Fee Permanently Locked in `verifyUpdate` — No Withdrawal Mechanism - (File: lazer/contracts/evm/src/PythLazer.sol)

### Summary
The `verifyUpdate` function in `PythLazer.sol` collects a `verification_fee` from callers on every invocation, but the contract contains no function to withdraw or forward those accumulated native-token fees to any treasury or owner address. Every fee payment is permanently locked in the contract.

### Finding Description
In `PythLazer.sol`, the `verifyUpdate` function is `external payable` and enforces a minimum fee:

```solidity
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    // Require fee and refund excess
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
        payable(msg.sender).transfer(msg.value - verification_fee);
    }
    // ... signature verification ...
}
```

The excess above `verification_fee` is correctly refunded to the caller. However, the exact `verification_fee` amount is retained in the contract's balance on every call. The contract (111 lines total) exposes no `withdraw`, `withdrawFee`, or treasury-transfer function. There is no state variable tracking accrued fees, no admin sweep function, and no `receive`/`fallback` that routes funds elsewhere. The collected fees accumulate in the contract's ETH balance with no reachable path to recover them.

This is structurally identical to the reported game-fee bug: a fee is computed, deducted from the caller's payment, and silently retained in the contract rather than forwarded to the intended recipient.

### Impact Explanation
Every call to `verifyUpdate` with a non-zero `verification_fee` permanently locks `verification_fee` wei in the contract. Over time, as Lazer price-feed consumers integrate and call `verifyUpdate` at high frequency, the cumulative locked amount grows without bound. The Pyth protocol loses all revenue that was intended to be collected through this fee mechanism. There is no admin escape hatch, no governance action, and no upgrade path visible in the contract that could recover the funds.

**Impact: Medium** — Protocol revenue is permanently lost; user funds are not at risk.

### Likelihood Explanation
**Likelihood: High** — `verifyUpdate` is the primary entry point for all Lazer price-feed consumers. Any deployment with `verification_fee > 0` will lock funds on every single call. No special conditions or attacker coordination are required; normal protocol usage is sufficient to trigger the issue continuously.

### Recommendation
Add a privileged withdrawal function that transfers the contract's accumulated ETH balance to a designated treasury address:

```solidity
address public treasury;

function withdrawFees(uint256 amount) external onlyOwner {
    (bool sent, ) = treasury.call{value: amount}("");
    require(sent, "Fee withdrawal failed");
}
```

Alternatively, forward the `verification_fee` directly to the treasury inside `verifyUpdate` at the point of collection, eliminating the need for a separate withdrawal step.

### Proof of Concept

1. Deploy `PythLazer` with `verification_fee = 1 ether`.
2. Call `verifyUpdate{value: 1 ether}(validUpdate)` from any EOA.
3. Observe: `address(pythLazer).balance == 1 ether`.
4. Attempt to call any withdrawal function — none exists.
5. Repeat step 2 N times; `address(pythLazer).balance == N ether`, permanently locked.
