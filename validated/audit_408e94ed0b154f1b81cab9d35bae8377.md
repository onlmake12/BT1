Audit Report

## Title
Orphan Block Pool Memory Exhaustion via Single-Child Sampling in `need_clean` — (`chain/src/utils/orphan_block_pool.rs`)

## Summary
`InnerPool::need_clean` samples only the first child returned by `map.iter().next()` to decide whether to evict an entire sibling group. An attacker can insert orphan blocks with a far-future `epoch_number` under arbitrary unknown parent hashes; because non-contextual verification never runs `PowVerifier`, these blocks require zero mining work. The orphan pool has no hard size cap, so the pool grows without bound until the node runs out of memory.

## Finding Description
`InnerPool::need_clean` at `chain/src/utils/orphan_block_pool.rs:113-122` calls `map.iter().next()` on the inner `HashMap<packed::Byte32, LonelyBlockHash>` to pick one representative child. If that child satisfies `epoch_number + EXPIRED_EPOCH >= tip_epoch`, the function returns `false` and `clean_expired_blocks` (lines 98-110) skips the entire group permanently. [1](#0-0) 

The `epoch_number` stored in `LonelyBlockHash` is read verbatim from the attacker-controlled block header at `chain/src/lib.rs:97` with no validation: [2](#0-1) 

`ChainService::non_contextual_verify` (`chain/src/chain_service.rs:72-89`) calls only `BlockVerifier` and `NonContextualBlockTxsVerifier`. `BlockVerifier::verify` (`verification/src/block_verifier.rs:39-47`) runs only cellbase, bytes, proposals-limit, duplicate, and merkle-root checks — no `PowVerifier`, no epoch check: [3](#0-2) 

`PowVerifier` is only invoked inside `HeaderVerifier` (`verification/src/header_verifier.rs:34`), which requires the parent header and is never called for orphan blocks: [4](#0-3) 

After passing non-contextual verification, `asynchronous_process_block` calls `orphan_broker.process_lonely_block` (`chain/src/chain_service.rs:143`). When the parent is unknown, `process_lonely_block` calls `self.orphan_blocks_broker.insert(lonely_block)` (`chain/src/orphan_broker.rs:122`) with no size guard: [5](#0-4) 

`OrphanBlockPool::insert` (`chain/src/utils/orphan_block_pool.rs:141-143`) has no capacity check. The `with_capacity` argument is only a HashMap pre-allocation hint. By contrast, the tx orphan pool enforces `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` and actively evicts on overflow (`tx-pool/src/component/orphan.rs:119-125`): [6](#0-5) 

The cleanup timer fires every 60 seconds (`chain/src/chain_service.rs:40-41`), but `clean_expired_blocks` is permanently defeated for any leader whose sole child carries a far-future epoch: [7](#0-6) 

## Impact Explanation
An unprivileged remote attacker can grow the orphan block pool without bound, consuming all available memory and crashing the targeted CKB node. This matches the allowed bounty impact: **High (10001–15000 points) — "Vulnerabilities which could easily crash a CKB node."**

## Likelihood Explanation
- **Zero mining work required.** `non_contextual_verify` never invokes `PowVerifier`; any nonce is accepted for orphan blocks.
- **No privileged access.** The attack path is the standard P2P block-relay path: received block → `asynchronous_process_block` → `process_lonely_block` → `orphan_blocks_broker.insert`.
- **Permanent shielding.** Setting `epoch_number = u64::MAX - EXPIRED_EPOCH - 1` guarantees `need_clean` always returns `false` for that leader for the lifetime of the process.
- **Repeatable at CPU speed.** One crafted block per fresh random parent hash; the pool grows by one entry per iteration with no cleanup ever occurring.

## Recommendation
1. **Fix `need_clean` to check all children:** return `true` if *any* child satisfies `epoch_number + EXPIRED_EPOCH < tip_epoch`, or evict individual expired children rather than the whole group atomically.
2. **Enforce a hard cap on orphan block pool size** analogous to `DEFAULT_MAX_ORPHAN_TRANSACTIONS` in the tx orphan pool, with LRU or oldest-epoch eviction when the cap is reached.
3. **Add a minimum `compact_target` check in non-contextual verification** to reject blocks whose declared difficulty is implausibly low relative to the genesis target, preventing zero-work orphan spam even if the cleanup logic is fixed.

## Proof of Concept
```rust
// Mirrors the existing test harness in chain/src/tests/orphan_block_pool.rs
let pool = OrphanBlockPool::with_capacity(10);
let tip_epoch: EpochNumber = 20;

// Each iteration: craft one block with unknown parent and far-future epoch.
// No PoW needed — BlockVerifier does not call PowVerifier.
loop {
    let parent = random_byte32(); // unknown to victim node
    let block = make_lonely_block_hash(parent, u64::MAX - EXPIRED_EPOCH - 1);
    pool.insert(block);
    // clean_expired_blocks(tip_epoch) returns [] every time:
    // need_clean checks the one child, finds epoch+EXPIRED_EPOCH >= tip_epoch, returns false.
    assert_eq!(pool.clean_expired_blocks(tip_epoch).len(), 0);
    // pool.len() grows by 1 every iteration — unbounded.
}
```
