### Title
Misleading `tx_hash` Parameter in `FeeEstimator::accept_tx` and Silent No-op `reject_tx` for `WeightUnitsFlow` Algorithm — (File: `util/fee-estimator/src/estimator/mod.rs`)

---

### Summary

`FeeEstimator::accept_tx` accepts a `tx_hash: Byte32` parameter that is silently dropped when the `WeightUnitsFlow` algorithm is active. Simultaneously, `FeeEstimator::reject_tx` is a complete no-op for `WeightUnitsFlow`. Callers in the tx-pool pass `tx_hash` expecting full lifecycle tracking, but the estimator's internal state diverges from reality: accepted transactions that are later rejected (e.g., via RBF) are never removed. This inflates fee rate estimates returned to RPC callers.

---

### Finding Description

In `util/fee-estimator/src/estimator/mod.rs`, the public `FeeEstimator` dispatch methods have the following behavior:

```rust
// Line 75-81
pub fn accept_tx(&self, tx_hash: Byte32, info: TxEntryInfo) {
    match self {
        Self::Dummy => {}
        Self::ConfirmationFraction(algo) => algo.write().accept_tx(tx_hash, info),
        Self::WeightUnitsFlow(algo) => algo.write().accept_tx(info),  // tx_hash silently dropped
    }
}

// Line 84-89
pub fn reject_tx(&self, tx_hash: &Byte32) {
    match self {
        Self::Dummy | Self::WeightUnitsFlow(_) => {}  // complete no-op for WeightUnitsFlow
        Self::ConfirmationFraction(algo) => algo.write().reject_tx(tx_hash),
    }
}
``` [1](#0-0) 

Two distinct problems exist:

1. **`accept_tx`: `tx_hash` is accepted but silently dropped for `WeightUnitsFlow`.** The `ConfirmationFraction` variant forwards `tx_hash` to its inner algorithm for per-transaction tracking. The `WeightUnitsFlow` variant calls `algo.write().accept_tx(info)` — the `tx_hash` is consumed by the function signature but never forwarded. The inner `WeightUnitsFlow::Algorithm::accept_tx` has no knowledge of which transaction hash corresponds to the tracked `info`.

2. **`reject_tx`: entire call is a no-op for `WeightUnitsFlow`.** When a transaction is rejected from the pool (e.g., via RBF eviction, pool size limit, or explicit removal), `reject_tx` is called with the transaction's hash. For `WeightUnitsFlow`, this call does nothing — the previously accepted transaction data is never removed from the estimator's internal state.

The tx-pool calls `fee_estimator.accept_tx(tx_hash, info)` when a transaction enters the pool and `fee_estimator.reject_tx(&tx_hash)` when it is evicted. For `WeightUnitsFlow`, the accept is recorded but the reject is silently swallowed, causing the estimator's internal flow model to permanently retain stale entries for evicted transactions. ---

### Impact Explanation

**Impact: Low–Medium**

The `WeightUnitsFlow` fee estimator's internal state becomes permanently inflated whenever transactions are accepted and then rejected. Because `reject_tx` is a no-op, every RBF replacement, pool eviction, or explicit `remove_transaction` RPC call leaves a ghost entry in the estimator. Over time — or under deliberate manipulation — the estimator's view of mempool fee pressure diverges upward from reality.

The `estimate_fee_rate` RPC endpoint (reachable by any unprivileged RPC caller) returns results derived from this stale state. Users and wallets relying on `estimate_fee_rate` to set transaction fees will receive inflated estimates and overpay fees. This is a correctness and economic harm issue, not a consensus or fund-loss issue, matching the "Low" impact tier of the reference report. [3](#0-2) 

---

### Likelihood Explanation

**Likelihood: High**

Any node configured to use the `WeightUnitsFlow` fee estimator will exhibit this behavior unconditionally. The divergence is not probabilistic — every rejected transaction (RBF, pool eviction, explicit removal) silently fails to update the estimator. RBF is a standard, documented feature of the CKB tx-pool. An unprivileged tx-pool submitter can trigger the divergence simply by submitting a transaction and then replacing it via RBF. No special privileges, keys, or majority hashpower are required.

---

### Recommendation

1. **Remove the `tx_hash` parameter from `FeeEstimator::accept_tx`** if `WeightUnitsFlow` is the canonical algorithm and per-hash tracking is not needed, making the interface honest about what is actually used. Alternatively, pass `tx_hash` through to `WeightUnitsFlow::Algorithm::accept_tx` and implement proper per-hash tracking.

2. **Implement `reject_tx` for `WeightUnitsFlow`**: The `WeightUnitsFlow` arm of `reject_tx` must remove the corresponding entry from the algorithm's internal state. Leaving it as a no-op means the estimator's state is permanently monotonically increasing and never self-corrects.

### Proof of Concept

**Setup:** Configure a CKB node with the `WeightUnitsFlow` fee estimator.

**Steps:**

1. Submit transaction `Tx_A` via `send_transaction` RPC. The tx-pool calls `fee_estimator.accept_tx(tx_a_hash, info_a)`. For `WeightUnitsFlow`, `tx_a_hash` is dropped; `info_a` is recorded in the estimator's internal flow state.

2. Submit transaction `Tx_B` that spends the same input as `Tx_A` with a higher fee rate (RBF). The tx-pool evicts `Tx_A` and calls `fee_estimator.reject_tx(&tx_a_hash)`. For `WeightUnitsFlow`, this is a no-op — `info_a` remains in the estimator.

3. Call `estimate_fee_rate` RPC. The estimator returns a fee rate inflated by the ghost entry for `Tx_A`, even though `Tx_A` is no longer in the pool.

4. Repeat steps 1–2 many times. The estimator's internal state accumulates unboundedly many ghost entries, causing `estimate_fee_rate` to return progressively higher (incorrect) fee rates to all RPC callers.

**Root cause lines:**

- `accept_tx` drops `tx_hash` for `WeightUnitsFlow`: [4](#0-3) 
- `reject_tx` is a no-op for `WeightUnitsFlow`: [5](#0-4)
