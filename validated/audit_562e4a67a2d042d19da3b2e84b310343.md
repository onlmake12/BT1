### Title
Unbounded Per-Request DB I/O in `GetTransactionsProofProcess::execute` with No Rate Limiting — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

---

### Summary

The light client protocol server accepts `GetTransactionsProof` messages from any unauthenticated peer and processes up to `GET_TRANSACTIONS_PROOF_LIMIT = 1000` transaction hashes per request. When those hashes span 1000 distinct blocks (worst case), the handler synchronously loads 1000 full blocks from the DB, builds 1000 CBMT Merkle proofs, computes 1000 `witnesses_root` hashes, loads 1000 uncle sets, and computes 1000 uncle hashes — all with zero rate limiting. The `LightClientProtocol` handler has no rate limiter, unlike the `Relayer` protocol which explicitly guards every message type.

---

### Finding Description

**Entrypoint:** Any peer connected on the `LightClient` protocol can send a `GetTransactionsProof` P2P message.

**Dispatch path:**

`LightClientProtocol::received` → `try_process` → `GetTransactionsProofProcess::execute`

In `lib.rs`, `try_process` dispatches directly with no rate-limit check: [1](#0-0) 

Compare this to `Relayer::try_process` in `sync/src/relayer/mod.rs`, which explicitly checks a per-peer-per-message-type rate limiter before dispatching: [2](#0-1) 

The `LightClientProtocol` struct has no `rate_limiter` field at all: [3](#0-2) 

**The only guard** in `execute` is a count check against `GET_TRANSACTIONS_PROOF_LIMIT = 1000`: [4](#0-3) [5](#0-4) 

**The expensive loop:** For each unique block containing a requested tx, the handler performs:
- `snapshot.get_block(&block_hash)` — full block deserialization from RocksDB
- `CBMT::build_merkle_proof(...)` — iterates all txs in the block to build the proof
- `block.calc_witnesses_root()` — hashes all witnesses
- `snapshot.get_block_uncles(&block_hash)` — DB read
- `uncles.data().calc_uncles_hash()` — hash computation
- `snapshot.get_block_extension(&block_hash)` — DB read [6](#0-5) 

After the loop, `reply_proof` calls `mmr.gen_proof(positions)` over up to 1000 MMR positions: [7](#0-6) 

All of these DB reads are synchronous blocking calls executed inside an `async fn`, which can starve the Tokio async runtime for the duration of the handler.

---

### Impact Explanation

An attacker with knowledge of 1000 confirmed transaction hashes from 1000 different blocks (all public on-chain data) can send a single `GetTransactionsProof` message that forces the server to perform ~6000 synchronous DB reads plus cryptographic hash computations. With no rate limit, the attacker can flood the server with back-to-back requests from one or more peers, saturating the DB I/O and blocking the async runtime, causing the light client server to become unresponsive to all peers — effectively congesting the CKB light client network at negligible attacker cost (a few bytes per message).

---

### Likelihood Explanation

- No PoW, no stake, no authentication required — any peer on the `LightClient` protocol can send this message.
- All required tx hashes are publicly available from any block explorer or full node.
- The attack is repeatable indefinitely since a successful request returns `Status::ok()` and does not ban the peer.
- The asymmetry with `Relayer` (which has a rate limiter) confirms this is an oversight, not an intentional design.

---

### Recommendation

Add a per-peer rate limiter to `LightClientProtocol`, mirroring the pattern in `Relayer`:

1. Add a `rate_limiter: RateLimiter<(PeerIndex, u32), ...>` field to `LightClientProtocol`.
2. In `try_process`, check `rate_limiter.check_key(&(peer_index, message.item_id()))` before dispatching, returning `StatusCode::TooManyRequests` on failure.
3. Additionally, consider reducing `GET_TRANSACTIONS_PROOF_LIMIT` or adding a secondary limit on the number of distinct blocks (e.g., cap at 100 distinct blocks per request).

---

### Proof of Concept

```
1. Run a CKB node with the light client server enabled.
2. Mine 1000+ blocks, each containing one unique transaction.
3. Collect all 1000 tx hashes (public chain data).
4. Connect as a light client peer.
5. Send a GetTransactionsProof message with all 1000 tx hashes and a valid last_hash.
6. Repeat in a tight loop from one or more peers.
7. Observe: server CPU/DB I/O saturates; other light client peers receive no responses.
```

The handler returns `Status::ok()` for a valid request, so the peer is never banned, and the loop can continue indefinitely. [8](#0-7)
