### Title
Unauthenticated `from` PeerId in Hole-Punching Protocol Allows Attacker-Directed Outbound TCP Connection Storms — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

---

### Summary

An unprivileged peer directly connected to a victim CKB node can forge the `from` field in `ConnectionRequest` and `ConnectionSync` messages to inject arbitrary TCP addresses into the victim's `pending_delivered` map and subsequently trigger passive NAT traversal tasks that make outbound TCP connections to attacker-controlled addresses. No authentication, PoW, or privileged role is required.

---

### Finding Description

The hole-punching protocol processes two message types in sequence. Neither verifies that the `from` PeerId in the message matches the actual session sending it.

**Step 1 — Poisoning `pending_delivered` via `ConnectionRequest`:**

In `respond_delivered()`, when the victim is the `to` target, the handler inserts the message's `from` field and the supplied `listen_addrs` directly into `pending_delivered` without checking whether `from` corresponds to the actual connected peer (`self.peer`): [1](#0-0) 

The `remote_listens` stored here are filtered only to TCP/IP addresses — attacker-controlled TCP addresses pass this filter: [2](#0-1) 

**Step 2 — Triggering passive NAT traversal via `ConnectionSync`:**

When a `ConnectionSync` arrives with `route` empty and `to == self_peer_id`, the handler looks up `pending_delivered` using the message's `from` field — again with no check that `from` matches the actual sender — and spawns a `runtime::spawn` task that calls `try_nat_traversal` for every stored address: [3](#0-2) [4](#0-3) [5](#0-4) 

**Rate-limit bypass:**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`: [6](#0-5) 

Because the attacker controls the `from` field and can use a fresh, randomly generated PeerId for each pair of messages, the per-`(from, to)` rate limit is trivially bypassed. The only binding constraint is the per-session `rate_limiter` at 30 req/sec: [7](#0-6) 

The `HOLE_PUNCHING_INTERVAL` deduplication check in `respond_delivered` is similarly bypassed by using a different `from` PeerId each time: [8](#0-7) 

---

### Impact Explanation

Each `ConnectionRequest`/`ConnectionSync` pair causes the victim to spawn one `runtime::spawn` task that attempts outbound TCP connections to up to `ADDRS_COUNT_LIMIT` (24) attacker-controlled addresses. At 30 pairs/sec (the per-session cap), a single attacker session can drive up to 720 concurrent outbound TCP connection attempts per second toward arbitrary targets, exhausting file descriptors, OS connection-tracking state, and network bandwidth on the victim node. The victim node's IP address becomes the apparent source of the connection flood, enabling third-party amplification/abuse.

---

### Likelihood Explanation

The attacker only needs a single direct P2P connection to the victim — a normal, unprivileged peer connection. No key material, PoW, or special role is required. The `HolePunching` protocol is enabled by default in production CKB nodes. The two-message sequence is straightforward to construct from the public `packed::ConnectionRequest` / `packed::ConnectionSync` schema.

---

### Recommendation

1. **Authenticate `from`**: In `respond_delivered`, assert that `from_peer_id` equals the PeerId of the actual session (`self.peer`). Reject messages where `from` does not match the sender's authenticated PeerId.
2. **Authenticate `from` in `ConnectionSync`**: Similarly, verify `content.from` against the actual session's PeerId before looking up `pending_delivered`.
3. **Bound `pending_delivered` size**: Cap the total number of entries to prevent unbounded memory growth from forged `from` PeerIds.

---

### Proof of Concept

Two-step integration test (pseudocode):

```rust
// Step 1: send ConnectionRequest to victim with forged `from` and attacker TCP addr
let forged_from = PeerId::random();
let attacker_addr: Multiaddr = "/ip4/1.2.3.4/tcp/9999".parse().unwrap();
let req = packed::ConnectionRequest::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())
    .listen_addrs(vec![attacker_addr.to_vec()].pack())
    .max_hops(6u8.pack())
    .route(vec![].pack())
    .build();
attacker_session.send(req);

// Step 2: send ConnectionSync with same forged `from`
let sync = packed::ConnectionSync::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())
    .route(vec![].pack())
    .build();
attacker_session.send(sync);

// Assert: victim node makes outbound TCP connection attempt to 1.2.3.4:9999
// Repeat with fresh forged_from PeerIds at 30/sec to produce connection storm
```
