Audit Report

## Title
Hole-Punching `ConnectionRequest` O(N^1.5) Amplification via Unique Spoofed `(from, to)` Pairs — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
An unprivileged remote peer can inject `ConnectionRequest` messages with unique spoofed `(from, to)` PeerId pairs targeting nonexistent `to` peers. Because the `forward_rate_limiter` is keyed on `(from, to, msg_item_id)` and is local to each node's `HolePunching` instance, every intermediate node sees each unique pair for the first time and forwards to `sqrt(N)` peers. This produces O(N^1.5) total messages per injected message, with no ban path triggered for the attacker.

## Finding Description

**Entry-point rate limiter (30 req/sec per session):**
The `received` handler applies a per-session rate limiter keyed on `(session_id, msg.item_id())` at 30 req/sec. [1](#0-0) 
This allows the attacker to inject 30 unique `ConnectionRequest` messages per second before being silently dropped (no ban).

**`forward_rate_limiter` is per-node, not global:**
The `forward_rate_limiter` is a field of the `HolePunching` struct — one instance per node, not shared across the network. [2](#0-1) 
It is initialized with a quota of 1 req/sec keyed on `(content.from, content.to, self.msg_item_id)`. [3](#0-2) 

**Rate limiter check in `execute()` bypassed by unique pairs:**
The check uses the unique `(from, to)` pair as the key. [4](#0-3) 
With unique `(from_i, to_i)` pairs per injected message, every node's rate limiter sees each key for the first time and passes unconditionally.

**Forward path when `to` is not found:**
When `self_peer_id != content.to` and `max_hops > 0`, `forward_message` is called. [5](#0-4) 

**sqrt(N) fan-out in `forward_message()`:**
When `target_sid` is `None` (nonexistent `to` peer), the node broadcasts to `sqrt(N)` peers via `filter_broadcast`. [6](#0-5) 

**Route check prevents loops, not fan-out:**
The route check only prevents a node from forwarding a message whose route already contains its own peer_id. [7](#0-6) 
This prevents a single chain from looping back, but does not prevent N distinct nodes from each forwarding the same `(from, to)` pair once. The per-node `forward_rate_limiter` drops the second copy at any given node, meaning each node forwards at most once per second per `(from, to)` pair — but with unique pairs, this cap is never reached.

**Attacker is never banned:**
`TooManyRequests = 110` (1xx) and `ReachedMaxHops = 504` (5xx) do not trigger `should_ban()`, which only fires for 4xx codes. [8](#0-7) [9](#0-8) 

**`MAX_HOPS = 6`:** [10](#0-9) 

## Impact Explanation
This matches the allowed CKB bounty impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points).**

Each injected message with a unique `(from_i, to_i)` pair causes up to N nodes to each forward to `sqrt(N)` peers, yielding O(N^1.5) total protocol messages per injected message. At N=10,000 nodes and 30 injected messages/sec, this produces approximately 30,000,000 forwarded messages/sec network-wide, saturating P2P bandwidth and starving legitimate traffic.

| N nodes | Messages/sec (30 injected/sec) |
|---------|-------------------------------|
| 1,000   | ~930,000                      |
| 10,000  | ~30,000,000                   |

## Likelihood Explanation
- Requires only one direct P2P connection to any CKB node — no authentication, no stake, no PoW.
- Spoofed `from`/`to` PeerIds need only be syntactically valid bytes; they need not correspond to real peers.
- `listen_addrs` must be non-empty with a valid TCP multiaddr — trivially satisfied.
- The attacker is never banned: `TooManyRequests` (110) and `ReachedMaxHops` (504) are 1xx/5xx codes, outside the 4xx ban range.
- The attack is repeatable indefinitely at 30 unique pairs/sec per connection; multiple attacker connections multiply the effect linearly.

## Recommendation
1. **Global message deduplication:** Maintain a shared, bounded LRU cache of recently-seen `(from, to, hash(message))` tuples across all sessions on a node, so that duplicate messages arriving via different paths are deduplicated before the fan-out decision.
2. **Move `forward_rate_limiter` to a shared `Arc<Mutex<...>>`:** The limiter already exists per-node but must be shared across all sessions so that messages arriving from different peers for the same `(from, to)` pair consume the same budget.
3. **Reduce fan-out or cap it to a small constant:** The `sqrt(N)` fan-out combined with `MAX_HOPS=6` is the root cause. Capping the fan-out to a small constant (e.g., 3–5 peers) regardless of N bounds the amplification to O(constant^MAX_HOPS) per message.
4. **Ban on repeated nonexistent-target forwarding:** Track per-session counts of forwarded messages where `target_sid` was `None`; ban sessions exceeding a threshold.

## Proof of Concept
```
1. Attacker establishes one TCP connection to any CKB mainnet node (Node A).
2. In a loop at 30 msg/sec, attacker sends:
     ConnectionRequest {
       from: random_peer_id(),   // unique each iteration
       to:   random_peer_id(),   // unique each iteration, not in any registry
       max_hops: 6,
       route: [],
       listen_addrs: ["/ip4/1.2.3.4/tcp/8115/p2p/<from_peer_id>"]
     }
3. Node A: rate_limiter[(session_A, ConnectionRequest_type)] passes (≤30/sec).
4. Node A: forward_rate_limiter[(from_i, to_i, type)] passes (first time for this unique pair).
5. Node A: target_sid = None → filter_broadcast to sqrt(N) peers.
6. Each of those sqrt(N) peers: forward_rate_limiter[(from_i, to_i, type)] passes (first time on that node).
7. Each broadcasts to sqrt(N) more peers, up to MAX_HOPS=6.
8. Instrument total_messages_sent_network_wide: observe O(N^1.5) per attacker message.
9. At N=10,000: 30 injected msg/sec → ~30,000,000 forwarded msg/sec observed across all nodes.

Verification: Add a counter to filter_broadcast calls in forward_message(); confirm it grows as N×sqrt(N) as N increases.
```
