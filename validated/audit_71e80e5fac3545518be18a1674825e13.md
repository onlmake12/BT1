### Title
`check_tx_fee` Admission Uses Size-Only Weight, Allowing Cycle-Heavy Transactions to Bypass the Minimum Fee Rate — (`tx-pool/src/util.rs`)

---

### Summary

The tx-pool admission check in `check_tx_fee` computes the minimum required fee using only the serialized byte size of the transaction, explicitly ignoring the cycles dimension. The actual fee rate used everywhere else in the codebase — sorting, eviction, fee estimation — is computed from `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. A transaction sender or P2P relayer can craft a transaction with a small byte size but very high cycle consumption that passes the admission check while its true fee rate is far below `min_fee_rate`.

---

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code itself acknowledges the problem with a comment:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [1](#0-0) 

Everywhere else in the codebase, the canonical weight formula is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`, derived from `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES`. [3](#0-2) 

`TxEntry::fee_rate()` uses this weight for sorting and eviction:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [4](#0-3) 

The pool's internal `estimate_fee_rate` also uses both `size` and `cycles` as independent block-filling dimensions: [5](#0-4) 

The gap: `check_tx_fee` is the **only** place that uses `tx_size` alone as the weight for the minimum fee check.

---

### Impact Explanation

**Concrete example** with default `min_fee_rate = 1000 shannons/KW`:

| Parameter | Value |
|---|---|
| Transaction byte size | 100 bytes |
| Transaction cycles | 10,000,000 |
| Actual weight | `max(100, 10_000_000 × 0.000_170_571_4)` = **1705** |
| Fee required by `check_tx_fee` | `1000 × 100 / 1000` = **100 shannons** |
| Fee required by actual weight | `1000 × 1705 / 1000` = **1705 shannons** |
| Effective fee rate if 100 shannons paid | `100 × 1000 / 1705` ≈ **58 shannons/KW** |

A transaction paying only 100 shannons passes admission but has an effective fee rate of ~58 shannons/KW — 17× below the configured minimum. Such transactions:

1. Are admitted to the pool and consume pool memory.
2. Are sorted to the bottom of the pool by actual fee rate, so miners will not include them.
3. Occupy pool space until evicted, potentially displacing legitimate transactions.
4. Can be used to fill the pool with low-effective-fee-rate transactions that appear to meet the minimum, degrading pool quality and miner revenue.

---

### Likelihood Explanation

The attack is reachable by any unprivileged actor via:
- **RPC `send_transaction`**: a local or remote RPC caller submits a transaction with high cycles and small byte size.
- **P2P relay** (`TransactionsProcess::execute`): a peer relays such a transaction with a declared cycle count up to `max_block_cycles`. [6](#0-5) 

No special privilege is required. The attacker only needs to construct a valid CKB transaction whose script execution consumes many cycles (e.g., a script with a tight computation loop) while keeping the serialized transaction small. This is straightforward with CKB-VM scripts.

---

### Recommendation

Replace the size-only weight in `check_tx_fee` with the canonical `get_transaction_weight`. Since cycles are not yet known at `pre_check` time (verification happens after), the check should be deferred or a declared-cycles value should be used as an upper bound. Specifically:

- For remote transactions, use the peer-declared `declared_cycles` as the weight input for the minimum fee check before full verification.
- For local transactions, either require a declared cycles hint or perform the weight-based check after verification completes (before `submit_entry`).

This mirrors how `TxEntry::fee_rate()` and `estimate_fee_rate` already operate. [7](#0-6) 

---

### Proof of Concept

1. Construct a CKB transaction with:
   - A single input/output (small byte size, e.g., ~100–200 bytes serialized).
   - A lock script that runs a tight loop consuming ~5,000,000 cycles.
   - Fee = `min_fee_rate × tx_size / 1000` (e.g., 200 shannons for a 200-byte tx at 1000 shannons/KW).

2. Submit via `send_transaction` RPC or relay via P2P.

3. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`. The transaction passes.

4. After verification, `TxEntry::fee_rate()` computes actual weight = `max(200, 5_000_000 × 0.000_170_571_4)` = `max(200, 852)` = **852**. Effective fee rate = `200 × 1000 / 852` ≈ **234 shannons/KW** — well below the 1000 shannons/KW minimum.

5. The transaction sits in the pool with a sub-minimum effective fee rate, consuming pool resources and being ignored by miners.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
