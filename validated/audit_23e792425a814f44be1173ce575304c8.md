### Title
Unbounded `pending_delivered` HashMap Growth via Spoofed `from` PeerIds in ConnectionRequest — (`network/src/protocols/hole_punching/component/connection_request.rs`)

### Summary
The `HolePunching` protocol's `pending_delivered` HashMap has no size cap. An attacker with a single P2P connection can insert an unbounded number of entries by sending `ConnectionRequest` messages with `to = local_peer_id` and a distinct, attacker-chosen `from` PeerId per message. The only throttle is the per-session rate limiter (30 msg/s), allowing up to ~9,000 entries per session before the 5-minute cleanup fires.

### Finding Description

**Insertion path** — `ConnectionRequestProcess::respond_delivered` is reached whenever the local node is the `to` target of a `ConnectionRequest`. The `from` field is taken directly from the message bytes with no validation that it matches the actual session peer: [1](#0-0) 

The deduplication guard only blocks re-insertion for the **same** `from` PeerId: [2](#0-1) 

Each message with a **new** `from` PeerId bypasses this guard and unconditionally inserts: [3](#0-2) 

**Rate limiting** — The primary `rate_limiter` is keyed by `(session_id, msg_item_id)` at 30 msg/s per session: [4](#0-3) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1/s per pair: [5](#0-4) 

Since `from` varies per message, the `forward_rate_limiter` creates a new bucket per unique `from` — it does not bound total insertions and itself grows unboundedly.

**Cleanup** — `retain` runs every `CHECK_INTERVAL = 5 min` and removes entries older than `TIMEOUT = 5 min`: [6](#0-5) 

There is no size cap anywhere on `pending_delivered`: [7](#0-6) 

### Impact Explanation

Over a 5-minute window (TIMEOUT), a single session can insert `30 × 300 = 9,000` entries. Each entry stores up to `ADDRS_COUNT_LIMIT = 24` `Multiaddr` values (~720 bytes) plus a `PeerId` key (~39 bytes) and a `u64` timestamp. That is ~760 bytes × 9,000 ≈ **6.8 MB per session**. With the default max peer count (~125), the map can reach **~850 MB**, degrading node performance. The `forward_rate_limiter` HashMap also grows at the same rate, compounding the effect. [8](#0-7) 

### Likelihood Explanation

The attack requires only a single valid P2P connection to the target node — no special privileges, no PoW, no key material. The `from` field is never authenticated against the session. The attacker simply cycles through fresh random PeerId bytes in each message.

### Recommendation

1. **Cap `pending_delivered`**: Enforce a hard maximum (e.g., 1,024 entries) and reject new insertions when the cap is reached.
2. **Validate `from` against session peer ID**: Require that the `from` field in a `ConnectionRequest` matches the actual peer ID of the sending session, eliminating spoofed-`from` insertions entirely.
3. **Cap `forward_rate_limiter` state**: Call `retain_recent()` on `forward_rate_limiter` in `notify()` alongside `rate_limiter`, or switch to a fixed-size LRU structure.

### Proof of Concept

```
1. Connect to target node via P2P.
2. In a loop at ≤30 msg/s:
     send ConnectionRequest {
       from: random_peer_id(),   // new PeerId each iteration
       to:   target_local_peer_id,
       listen_addrs: [valid_tcp_multiaddr × 24],
       max_hops: 6,
       route: [],
     }
3. After 5 minutes, pending_delivered contains ~9,000 entries.
4. Measure RSS growth; assert it exceeds a configurable cap.
``` [9](#0-8)
