### Title
`max_tx_verify_cycles` DoS Limit Bypassed via Attacker-Controlled `declared_cycles` in Remote Transaction Relay — (`tx-pool/src/process.rs`)

---

### Summary

A remote peer relaying a transaction can supply an arbitrarily large `declared_cycles` value. The tx-pool uses this attacker-controlled value as the actual cycle-execution ceiling during script verification, completely bypassing the operator-configured `max_tx_verify_cycles` DoS guard. An unprivileged P2P peer can force the node to spend up to `max_block_cycles` (~3.5 billion) cycles verifying a single transaction, roughly 50× the intended limit of 70 million cycles.

---

### Finding Description

The vulnerability class in the external report is **resource-limit bypass via attacker-controlled input**: a caller inflates resource consumption to defeat a guard that was supposed to cap it. The CKB analog is structurally identical but in the tx-pool cycle-accounting layer.

**Root cause — `_process_tx` in `tx-pool/src/process.rs`:**

```rust
// tx-pool/src/process.rs  line 720
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
```

When a remote peer relays a transaction, `declared_cycles` is the cycle count the peer *claims* the transaction uses. This value is passed directly as the `max_cycles` ceiling to `verify_rtx`:

```rust
// tx-pool/src/process.rs  lines 724-732
let verified_ret = verify_rtx(
    Arc::clone(&snapshot),
    Arc::clone(&rtx),
    tx_env,
    &verify_cache,
    max_cycles,          // ← attacker-supplied declared_cycles
    command_rx,
)
.await;
```

The operator-configured `max_tx_verify_cycles` (default 70 million, purpose: "limit max verify cycles on single tx, to reduce DDOS vulnerability") is **never consulted** in this path. It is only used as a queue-classification threshold (`large_cycle_threshold`) in `VerifyQueue::add_tx` and as the actual limit in the unrelated `readd_detached_tx` path.

After verification, the only check performed is an exact-match guard:

```rust
// tx-pool/src/process.rs  lines 736-748
if let Some(declared) = declared_cycles
    && declared != verified.cycles
{
    return Some((Err(Reject::DeclaredWrongCycles(declared, verified.cycles)), snapshot));
}
```

An attacker who crafts a transaction that genuinely consumes `N` cycles (where `max_tx_verify_cycles < N ≤ max_block_cycles`) and declares exactly `N` passes this check. The node verifies with `max_cycles = N`, the declared-vs-actual check passes, and the transaction is accepted into the pool.

**Entry path (fully unprivileged P2P):**

```
P2P RelayV3 message
  → sync/relay handler
  → TxPoolService::submit_remote_tx(tx, declared_cycles, peer)
  → resumeble_process_tx → enqueue_verify_queue
  → VerifyMgr Worker::process_inner
  → TxPoolService::_process_tx(tx, Some(declared_cycles), command_rx)
  → verify_rtx(..., max_cycles = declared_cycles, ...)   ← DoS here
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

### Impact Explanation

`max_tx_verify_cycles` was introduced specifically to prevent CPU-exhaustion DoS from expensive scripts (CHANGELOG: *"The purpose is to limit max verify cycles on single tx, to reduce DDOS vulnerability"*). With this bypass, a single remote peer can force the node to execute up to `max_block_cycles` ≈ 3.5 billion cycles of RISC-V script per transaction relay message. At the default `max_tx_verify_cycles = 70,000,000`, this is a **50× amplification** of intended CPU budget per relayed transaction. Multiple concurrent peers multiplies the effect linearly, causing node unresponsiveness or starvation of legitimate transaction processing. [5](#0-4) 

---

### Likelihood Explanation

The attack requires only a standard P2P connection and the ability to craft a transaction whose script consumes a known, large number of cycles. An attacker can pre-measure the exact cycle count of their script offline (e.g., using the `estimate_cycles` RPC or local execution), then relay it with the matching `declared_cycles`. No privileged access, no key material, and no majority hashpower is required. The relay protocol (`RelayV3`) is open to any peer. [6](#0-5) [7](#0-6) 

---

### Recommendation

In `_process_tx`, cap `max_cycles` at `max_tx_verify_cycles` when processing remote transactions:

```rust
// tx-pool/src/process.rs
let max_cycles = declared_cycles
    .map(|d| d.min(self.tx_pool_config.max_tx_verify_cycles))
    .unwrap_or_else(|| self.tx_pool_config.max_tx_verify_cycles);
```

This preserves the declared-cycles exact-match check (a tx that genuinely uses more than `max_tx_verify_cycles` will fail verification and be rejected with `ExceededMaximumCycles`) while ensuring the node never spends more than the operator-configured budget per transaction. [8](#0-7) 

---

### Proof of Concept

1. Craft a CKB transaction whose lock script executes exactly `N` cycles, where `N = 500_000_000` (well above the default `max_tx_verify_cycles = 70_000_000` but below `max_block_cycles`).
2. Measure `N` locally via `estimate_cycles` RPC or offline execution.
3. Connect to the target node as a P2P peer supporting `RelayV3`.
4. Send a `RelayTransactionHashes` followed by a `RelayTransactions` message with `cycles = N` in the `RelayTransaction` struct.
5. Observe via node metrics (`ckb_tx_pool_async_process`) that the node spends ~7× longer verifying this single transaction than it would for a `max_tx_verify_cycles`-bounded transaction.
6. Repeat from multiple peers concurrently to saturate the verify worker pool.

The `DeclaredWrongCycles` rejection guard does not fire because `declared == verified.cycles` exactly. [6](#0-5) [9](#0-8)

    snapshot: Arc<Snapshot>,
    rtx: Arc<ResolvedTransaction>,
    tx_env: Arc<TxVerifyEnv>,
    cache_entry: &Option<CacheEntry>,
    max_tx_verify_cycles: Cycle,
    command_rx: Option<&mut watch::Receiver<ChunkCommand>>,
) -> Result<Completed, Reject> {
    let consensus = snapshot.cloned_consensus();
    let data_loader = snapshot.as_data_loader();

    if let Some(completed) = cache_entry {
        TimeRelativeTransactionVerifier::new(rtx, consensus, data_loader, tx_env)
            .verify()
            .map(|_| *completed)
            .map_err(Reject::Verification)
    } else if let Some(command_rx) = command_rx {
        ContextualTransactionVerifier::new(
            Arc::clone(&rtx),
            consensus,
            data_loader,
            Arc::clone(&tx_env),
        )
        .verify_with_pause(max_tx_verify_cycles, command_rx)
        .await
        .and_then(|result| {
            DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
                .verify()?;
            Ok(result)
        })
        .map_err(Reject::Verification)
    } else {
        block_in_place(|| {
            ContextualTransactionVerifier::new(Arc::clone(&rtx), consensus, data_loader, tx_env)
                .verify(max_tx_verify_cycles, false)
                .and_then(|result| {
                    DaoScriptSizeVerifier::new(
                        rtx,
                        snapshot.cloned_consensus(),
                        snapshot.as_data_loader(),
                    )
                    .verify()?;
                    Ok(result)
                })
                .map_err(Reject::Verification)
        })
    }
