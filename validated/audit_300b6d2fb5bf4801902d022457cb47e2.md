### Title
Unvalidated `listen_port` in GetNodes Allows Inbound Peer to Poison Peer Store with Forged Port — (`network/src/protocols/discovery/state.rs`, `network/src/protocols/discovery/mod.rs`)

---

### Summary

An unprivileged inbound peer can send a `GetNodes` message with an arbitrary `listen_port` value. The node unconditionally replaces the TCP port in the peer's recorded address with the attacker-supplied value via `update_port()`, stores the resulting forged address in the peer store via `add_new_addr()`, and subsequently propagates it to other peers through `get_random()` responses. No validation is performed to confirm the advertised port matches the actual TCP source port of the connection.

---

### Finding Description

When an inbound session is established, the peer's address is recorded as `RemoteAddress::Init(actual_IP:ephemeral_port)`. [1](#0-0) 

When a `GetNodes` message is received, `update_port()` is called with the attacker-supplied `listen_port` value: [2](#0-1) 

`update_port()` operates on any `RemoteAddress::Init` variant and replaces the TCP port component with the caller-supplied value, with no bounds check, no comparison against the actual source port, and no privileged-port rejection: [3](#0-2) 

The resulting `RemoteAddress::Listen(actual_IP:attacker_port)` is then passed directly to `add_new_addr()`, which stores it in the peer store after only an IP-reachability check (no port validation): [4](#0-3) 

The forged address is now in the peer store and will be returned in all future `get_random()` responses sent to other peers. [5](#0-4) 

Note: the developers were aware that the forged address must not appear in the *immediate* response to the attacker (the comment on line 118–119 explains the `get_random` call is placed before `update_port` for this reason), but the address is still durably stored and propagated to all subsequent peers. [6](#0-5) 

---

### Impact Explanation

Any peer that receives a `Nodes` response containing the forged address will attempt to connect to `attacker_real_IP:attacker_chosen_port`. If the attacker does not listen on that port, the connection attempt fails silently, degrading peer discovery. If the attacker does listen on that port (e.g., a non-CKB service), it can fingerprint or mislead connecting nodes. Because each victim node propagates up to 2500 addresses per `Nodes` response, a single poisoned entry fans out across the network rapidly.

The attacker can only forge their own IP with a different port — they cannot inject arbitrary third-party IPs. However, they can open many inbound connections (one forged entry per connection) to amplify the effect.

---

### Likelihood Explanation

The exploit requires only an inbound TCP connection and a single crafted `GetNodes` message — no authentication, no PoW, no privileged role. The `received_get_nodes` flag prevents duplicate `GetNodes` on the same session, but the attacker can open many sessions. The code path is exercised on every inbound peer connection that participates in discovery.

---

### Recommendation

In `update_port()` (or at the call site in the `GetNodes` handler), validate that the supplied `listen_port` matches the actual source port of the TCP connection, or at minimum reject well-known privileged ports (< 1024) and require the port to be non-zero. The actual source port is available as `context.session.address` at the time the `GetNodes` message is processed. [3](#0-2) 

---

### Proof of Concept

```
1. Attacker opens an inbound TCP connection to victim from 1.2.3.4:54321.
2. Victim records RemoteAddress::Init(/ip4/1.2.3.4/tcp/54321).
3. Attacker sends GetNodes { listen_port: Some(80), version: 1, count: 1000, required_flags: 0 }.
4. Victim calls update_port(80) → RemoteAddress::Listen(/ip4/1.2.3.4/tcp/80).
5. Victim calls add_new_addr(session_id, (/ip4/1.2.3.4/tcp/80, flags)).
6. /ip4/1.2.3.4/tcp/80 is now in the peer store.
7. Any subsequent peer that sends GetNodes to the victim receives /ip4/1.2.3.4/tcp/80
   in the Nodes response and attempts to connect to it.
```

Unit test assertion: after step 5, `peer_store.fetch_random_addrs(2500, flags)` must contain `/ip4/1.2.3.4/tcp/80` and must NOT contain `/ip4/1.2.3.4/tcp/54321`.

    pub(crate) fn update_port(&mut self, port: u16) {
        if let RemoteAddress::Init(addr) = self {
            let addr = addr
                .into_iter()
                .map(|proto| {
                    match proto {
                        // TODO: other transport, UDP for example
                        Protocol::Tcp(_) => Protocol::Tcp(port),
                        value => value,
                    }
                })
                .collect();
            *self = RemoteAddress::Listen(addr);
        }
    }

    fn get_random(&mut self, n: usize, flags: Flags) -> Vec<(Multiaddr, Flags)> {
        let fetch_random_addrs = self
            .network_state
            .with_peer_store_mut(|peer_store| peer_store.fetch_random_addrs(n, flags));
        let addrs = fetch_random_addrs
            .into_iter()
            .filter_map(|paddr| {
                if !self.is_valid_addr(&paddr.addr) {
                    return None;
                }
                let f = Flags::from_bits_truncate(paddr.flags);
                Some((paddr.addr, f))
            })
            .collect();
        trace!("Discovered random addrs: {:?}", addrs);
        addrs
    }
