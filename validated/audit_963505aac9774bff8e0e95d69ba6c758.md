### Title
`PythLazer.verifyUpdate()` Accepts Arbitrarily Stale Price Updates — No Payload Timestamp Validation Against `block.timestamp` - (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` is the sole on-chain verification primitive for Pyth Lazer price updates on EVM chains. It validates the ECDSA signature and checks that the signer's certificate has not expired, but it **never compares the timestamp embedded in the signed payload against `block.timestamp`**. As a result, any signed Lazer update — regardless of how old it is — passes verification and is returned to the consumer as valid, as long as the signer key has not expired. This is a direct structural analog to the reported `feed_prices()` vulnerability, where the oracle accepted delayed transactions with stale prices as fresh.

---

### Finding Description

`verifyUpdate()` in `lazer/contracts/evm/src/PythLazer.sol` performs three checks:

1. Fee sufficiency (`msg.value >= verification_fee`)
2. EVM format magic number validation
3. ECDSA signature recovery and signer certificate expiry check (`isValidSigner`) [1](#0-0) 

The signed payload contains a `timestamp` field (in microseconds) that is parsed by `PythLazerLib.parsePayloadHeader()` and exposed as `update.timestamp`. This timestamp represents when the Lazer server generated the price update. [2](#0-1) 

However, `verifyUpdate()` never reads or validates this timestamp. It returns the raw `payload` bytes to the caller without any staleness enforcement. The contract has no concept of a maximum acceptable age for the price data. [3](#0-2) 

The only time-based check in the entire contract is `isValidSigner`, which checks whether the signer's certificate has expired — not whether the price data itself is fresh: [4](#0-3) 

Unlike Pyth Core, which provides `getPriceNoOlderThan()` as a built-in staleness guard, Lazer has no equivalent on-chain primitive. `verifyUpdate()` is the only on-chain entry point, and it provides no staleness protection.

The official integration documentation example only checks monotonicity (`_timestamp > timestamp`) — not freshness relative to `block.timestamp`: [5](#0-4) 

---

### Impact Explanation

Any consumer contract that calls `verifyUpdate()` and uses the returned price without independently comparing the payload timestamp to `block.timestamp` will silently accept arbitrarily stale prices. Because `verifyUpdate()` is the only on-chain verification step and it provides no staleness signal, consumers have no on-chain indication that the price is stale — the call succeeds and returns a valid-looking payload.

Concrete impact: a relayer or attacker holding a previously captured (but still validly signed) Lazer update can submit it to a consumer contract during a period of network congestion or after a price movement. The consumer contract will accept the old price as current, enabling incorrect liquidations, mispriced trades, or manipulation of any protocol that uses Lazer prices for financial decisions.

---

### Likelihood Explanation

Lazer is marketed as an ultra-low-latency feed (sub-50ms). Consumer contracts are expected to act on prices in near-real-time. The integration guide does not show a `block.timestamp` staleness check in its example code. Any consumer following the documented pattern is vulnerable. Network congestion on EVM chains (which can delay transactions by minutes) makes this scenario realistic without any attacker involvement — a legitimately submitted update can land stale. An adversarial relayer can also deliberately delay submission.

---

### Recommendation

`verifyUpdate()` should decode the payload timestamp and compare it against `block.timestamp` with a configurable maximum staleness window (e.g., `MAX_PRICE_AGE_SECONDS`). Reject updates whose payload timestamp is older than `block.timestamp - MAX_PRICE_AGE_SECONDS` or newer than `block.timestamp + GRACE_FUTURE_PERIOD`. This mirrors the fix recommended in the original report:

```solidity
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    // ... existing fee, magic, signature checks ...

    // Decode timestamp from payload (bytes 4..11 after PAYLOAD_FORMAT_MAGIC)
    uint64 payloadTimestampUs = uint64(bytes8(payload[4:12]));
    uint64 payloadTimestampSec = payloadTimestampUs / 1_000_000;

    if (block.timestamp > payloadTimestampSec) {
        require(
            block.timestamp - payloadTimestampSec <= MAX_PRICE_AGE_SECONDS,
            "stale price update"
        );
    } else {
        require(
            payloadTimestampSec - block.timestamp <= GRACE_FUTURE_PERIOD,
            "future price update"
        );
    }
}
```

`MAX_PRICE_AGE_SECONDS` should be a governance-configurable parameter. Given Lazer's sub-second design, a value of 5–10 seconds is appropriate for real-time channels.

### Proof of Concept

Given:
- A Lazer trusted signer with `expiresAt` far in the future
- ETH price is `$3,000`; a consumer DeFi protocol uses Lazer for liquidation pricing

1. At `T=0`, the Lazer signer emits a signed update with `timestamp = T` and `ETH price = $2,500` (a dip). A relayer captures this update but does not submit it.
2. At `T=30s`, ETH price recovers to `$3,000`. Alice borrows against her ETH collateral at the current price.
3. At `T=60s`, the relayer submits the captured update from `T=0` to the consumer contract. `verifyUpdate()` succeeds — the signature is valid and the signer has not expired. The payload timestamp (`T=0`) is never checked against `block.timestamp` (`T=60s`).
4. The consumer contract sees ETH at `$2,500` and liquidates Alice's position, even though the current price is `$3,000`.

The attack requires only a valid signed Lazer update (obtainable by any subscriber) and the ability to delay its on-chain submission — no privileged access is needed. [1](#0-0)
