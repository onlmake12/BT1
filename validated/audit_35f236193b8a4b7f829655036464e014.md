### Title
Unbounded Iteration Over All Tx-Pool Entries in `get_raw_tx_pool(verbose=true)` Can Lead to Node DoS — (`rpc/src/module/pool.rs`)

---

### Summary

The `get_raw_tx_pool` RPC endpoint, when called with `verbose=true`, triggers an unbounded iteration over every transaction in the tx-pool (pending, gap, proposed, and conflicted) with no pagination, no result cap, and no per-call timeout. An attacker who can reach the RPC port can first flood the pool with many small transactions, then repeatedly call this endpoint to exhaust CPU and memory, stalling the node's RPC thread pool and delaying block processing.

---

### Finding Description

`get_raw_tx_pool` is implemented in `rpc/src/module/pool.rs`:

```rust
fn get_raw_tx_pool(&self, verbose: Option<bool>) -> Result<RawTxPool> {
    let tx_pool = self.shared.tx_pool_controller();
    let raw = if verbose.unwrap_or(false) {
        let info = tx_pool
            .get_all_entry_info()          // ← unbounded scan
            ...
        RawTxPool::Verbose(info.into())
    } else {
        let ids = tx_pool
            .get_all_ids()                 // ← unbounded scan
            ...
        RawTxPool::Ids(ids.into())
    };
    Ok(raw)
}
```

`get_all_entry_info()` in `tx-pool/src/pool.rs` iterates over every entry in three separate unbounded passes:

```rust
pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
    let pending = self
        .pool_map
        .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
        .map(|entry| (entry.transaction().hash(), entry.to_info()))
        .collect();                        // full scan, no limit

    let proposed = self
        .pool_map
        .sorted_proposed_iter()
        .map(|entry| (entry.transaction().hash(), entry.to_info()))
        .collect();                        // full scan, no limit

    let conflicted = self
        .conflicts_cache
        .iter()
        .map(|(_id, tx)| tx.hash())
        .collect();                        // up to CONFLICTES_CACHE_SIZE = 10_000
    ...
}
```

`get_ids()` (the non-verbose path) performs the same unbounded scans over pending and proposed entries.

The tx-pool is bounded only by `max_tx_pool_size` **in bytes** (default 180 MB). A minimal CKB transaction is ~100–200 bytes, so the pool can legally hold hundreds of thousands of entries. Each `to_info()` call computes ancestor statistics. The entire result is then JSON-serialized in one synchronous response with no streaming, no pagination parameter, and no server-side limit.

There is no rate limit on this specific RPC method beyond the global per-connection rate limiter (which governs P2P relay, not RPC).

---

### Impact Explanation

A caller who can reach the RPC port can:

1. Flood the pool with many minimum-size transactions (via `send_transaction` or P2P relay) until `max_tx_pool_size` is reached.
2. Repeatedly call `get_raw_tx_pool(verbose=true)`.

Each call forces the node to:
- Acquire the tx-pool read lock for the full duration of the scan.
- Compute ancestor statistics for every entry.
- Allocate and serialize a potentially hundreds-of-megabytes JSON payload.

This can saturate the RPC thread pool, delay or stall block template assembly (which also acquires the pool lock), and cause the node to become unresponsive to all other RPC callers. In the worst case, repeated calls can trigger OOM conditions on memory-constrained nodes.

---

### Likelihood Explanation

The RPC port is exposed to localhost by default and optionally to a configured listen address. Any process or user with access to the RPC endpoint — including a misconfigured public-facing node, a compromised local process, or a legitimate operator script that loops on this call — can trigger the condition. Filling the pool is straightforward: minimum-fee transactions are accepted by the pool up to the byte limit, and no PoW or privileged key is required to submit them via `send_transaction`.

---

### Recommendation

- Add a `limit` and `after` (cursor) parameter to `get_raw_tx_pool`, mirroring the pagination pattern already used by `get_cells` and `get_transactions` in the indexer RPC.
- Enforce a hard server-side cap (e.g., 1,000 entries per call) even when no limit is supplied.
- Consider making the verbose scan asynchronous with a per-request timeout so it cannot block the RPC thread pool indefinitely.

---

### Proof of Concept

```python
import requests, json, time

RPC = "http://127.0.0.1:8114"

# Step 1: flood the pool with many small transactions via send_transaction
# (omitted for brevity — standard CKB transaction construction)

# Step 2: repeatedly call get_raw_tx_pool(verbose=true)
while True:
    t0 = time.time()
    r = requests.post(RPC, json={
        "id": 1, "jsonrpc": "2.0",
        "method": "get_raw_tx_pool",
        "params": [True]
    })
    elapsed = time.time() - t0
    count = len(r.json()["result"]["pending"])
    print(f"returned {count} entries in {elapsed:.2f}s")
    # With a full pool, elapsed grows to tens of seconds per call,
    # blocking all concurrent RPC requests.
```
