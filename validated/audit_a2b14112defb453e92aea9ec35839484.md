Audit Report

## Title
Missing `BLOCK_INVALID` Early-Return in `HeaderAcceptor::accept()` Allows Sync State Corruption via `SendHeaders` - (File: `sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::accept()` contains an acknowledged FIXME that omits an early-return guard for `BLOCK_INVALID` block status. Because `BLOCK_INVALID` (`1 << 12`) and `HEADER_VALID` (`1`) are completely disjoint bits, a header whose block was previously marked invalid bypasses the `HEADER_VALID` branch, passes all three lightweight checks, and is re-inserted into `header_map` via `insert_valid_header`, creating an inconsistent sync state. The same block is correctly rejected via the `CompactBlock` path.

## Finding Description
In `HeaderAcceptor::accept()` (lines 295–358 of `sync/src/synchronizer/headers_process.rs`), the acknowledged FIXME at lines 301–302 notes the missing guard. The code reads the block status and branches only on `HEADER_VALID` (line 304). Since `BLOCK_INVALID = 1 << 12` and `HEADER_VALID = 1` share no bits (`shared/src/block_status.rs` lines 11 and 16), `status.contains(BlockStatus::HEADER_VALID)` is false for an invalid block, and execution falls through to:

1. `prev_block_check` (lines 244–253) — checks only the **parent** hash for `BLOCK_INVALID`, not the header itself.
2. `non_contextual_check` (lines 255–283) — runs `HeaderVerifier`. A block marked invalid due to body failure (e.g., script execution) retains a structurally valid header that passes this check.
3. `version_check` (lines 286–293) — checks only the version field.

If all three pass, `sync_shared.insert_valid_header(self.peer, self.header)` is called at line 356, which inserts the header into `header_map` (line 1129 of `sync/src/types/mod.rs`) and updates both the peer's best-known header and `shared_best_header` (lines 1130–1140). The `block_status_map` entry remains `BLOCK_INVALID`, but the sync layer's view is now inconsistent with it.

`BLOCK_INVALID` is set by `chain/src/verify.rs` lines 175–177 after full block verification failure. The `CompactBlockProcess` path correctly rejects such blocks, as confirmed by the test at lines 60–71 of `sync/src/relayer/tests/compact_block_process.rs`.

## Impact Explanation
This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**. Concrete impacts: (1) `header_map` and `shared_best_header` are updated to reference a block the node has already determined is invalid, creating an inconsistent sync state; (2) the synchronizer may re-request the full block body for a hash it already rejected, wasting bandwidth and CPU; (3) if the invalid chain has higher total difficulty, `shared_best_header` being advanced to it can mislead chain-selection logic, potentially causing the node to repeatedly attempt to sync an invalid chain.

## Likelihood Explanation
Triggering the bug requires an attacker to first get a block marked `BLOCK_INVALID` on the target node, which requires PoW to craft a block with a valid header but invalid body. The initial PoW cost is non-trivial, but once a block is marked invalid, the exploit can be repeated at negligible cost by any connected peer via a standard `SendHeaders` P2P message. No special privileges are required beyond a standard P2P connection.

## Recommendation
Add an explicit early-return for `BLOCK_INVALID` immediately after reading the block status in `HeaderAcceptor::accept()`, resolving the acknowledged FIXME at line 301:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing logic
}
```

This mirrors the behavior already implemented in `CompactBlockProcess`.

## Proof of Concept
1. Connect a peer to a CKB node.
2. Send a block with a valid header but invalid body (e.g., a failing script) via any block-submission path, causing the node to mark it `BLOCK_INVALID` (confirmed path: `chain/src/verify.rs` lines 175–177).
3. Re-send the same header via a `SendHeaders` P2P message.
4. Observe that `HeaderAcceptor::accept()` falls through all three lightweight checks and calls `insert_valid_header`.
5. Verify that `shared_best_header` and the peer's best-known header now point to the previously-invalid block, while `block_status_map` still contains `BLOCK_INVALID` — confirming the inconsistent state.

A unit test analogous to the existing `test_in_block_status_map` in `sync/src/relayer/tests/compact_block_process.rs` (lines 25–71) can be written for `HeaderAcceptor::accept()`: insert a `BLOCK_INVALID` status, call `accept()`, and assert the result is `ValidationState::Invalid` rather than `ValidationState::Valid`.
