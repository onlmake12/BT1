### Title
Missing Per-Peer Rate Limiting on `GetLastStateProof` Enables Sustained DB I/O Exhaustion — (`util/light-client-protocol-server/src/lib.rs`, `get_last_state_proof.rs`)

---

### Summary

`LightClientProtocol` has no per-peer rate limiter. A single unprivileged remote peer can flood the server with `GetLastStateProof` messages, each of which triggers up to ~1,497 synchronous DB reads inside `complete_headers`, exhausting CPU and RocksDB I/O and degrading service for all legitimate light clients.

---

### Finding Description

**Limit check is insufficient — it only bounds a single request, not request rate.**

The only guard in `execute()` is a per-message size check: [1](#0-0) 

With `last_n_blocks=499` and `difficulties=[]`, the expression evaluates to `0 + 998 = 998 ≤ 1000`, so the check passes. [2](#0-1) 

**Short-chain branch produces up to 499 block numbers per request.**

When `last_block_number - start_block_number <= last_n_blocks`, the server takes all blocks in that range with no sampled reduction: [3](#0-2) 

**`complete_headers` performs 3 DB operations per block.**

For each of the up to 499 block numbers, `complete_headers` calls:
- `snapshot.get_ancestor()` — header DB read
- `snapshot.get_block()` — full block DB read
- `snapshot.chain_root_mmr(*number - 1).get_root()` — MMR DB read [4](#0-3) 

That is up to **~1,497 synchronous DB operations per single request**.

**`LightClientProtocol` has no rate limiter — unlike `Relayer`.**

The `received` handler dispatches directly to `try_process` with zero throttling: [5](#0-4) 

The `LightClientProtocol` struct carries only a `Shared` field — no `RateLimiter`: [6](#0-5) 

By contrast, `Relayer` explicitly holds a keyed `RateLimiter<(PeerIndex, u32)>`: [7](#0-6) 

This asymmetry confirms the omission is unintentional.

---

### Impact Explanation

A single peer sending a continuous stream of `GetLastStateProof` messages (each with `last_n_blocks=499`, `difficulties=[]`, `start_number = last_block_number - 499`) forces the server to execute up to ~1,497 RocksDB reads per message with no back-pressure. This saturates the DB I/O thread pool and CPU, causing response latency for all other light clients to grow unboundedly. The server cannot disconnect the attacker because no ban condition is triggered — every request is structurally valid.

---

### Likelihood Explanation

The attack requires only a valid P2P connection to a node running the light client protocol server. No special privileges, keys, or hashpower are needed. The crafted message is trivially constructable from the public schema. The attacker sustains the load indefinitely since there is no connection-level throttle, no per-peer message counter, and no ban triggered by well-formed requests.

---

### Recommendation

Add a keyed per-peer rate limiter to `LightClientProtocol`, mirroring the pattern already used in `Relayer`: [8](#0-7) 

Apply the limiter at the top of `received()` (or inside `try_process`) before dispatching to any `GetLastStateProof` handler. A reasonable limit is on the order of a few requests per second per peer, consistent with the legitimate sync use-case.

---

### Proof of Concept

1. Connect to a CKB node with the light client protocol enabled.
2. Obtain the current tip hash and tip block number `N`.
3. In a tight loop, send `GetLastStateProof` messages:
   - `last_hash` = tip hash
   - `start_hash` = hash of block `N-499`
   - `start_number` = `N-499`
   - `last_n_blocks` = `499`
   - `difficulty_boundary` = `U256::zero()`
   - `difficulties` = `[]`
4. Each message passes the limit check (`998 ≤ 1000`), enters the short-chain branch, and causes `complete_headers` to execute ~1,497 DB reads.
5. Concurrently measure response latency for a second peer sending `GetLastState` requests; assert latency remains bounded — it will not.

        &mut self,

        let msg = match packed::LightClientMessageReader::from_slice(&data) {

                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(

                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
