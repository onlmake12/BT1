### Title
User Commitment Not Bound to Requester Address Enables Commitment Copying in Entropy Protocol - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The `constructUserCommitment` function in `Entropy.sol` computes `keccak256(userRandomness)` without including the requester's address. Combined with the `request` function accepting a raw `userCommitment` hash directly from the caller, any attacker can copy another user's on-chain commitment and submit it as their own. Once the original user reveals their secret `x_U`, the attacker can use the same value to fulfill their own copied request, obtaining a random number derived from a contribution they did not independently choose.

### Finding Description

`constructUserCommitment` at line 944 computes:

```solidity
userCommitment = keccak256(bytes.concat(userRandomness));
``` [1](#0-0) 

No requester address is mixed into this hash. The `request` function (line 322) accepts `userCommitment` directly from the caller and stores it in `req.commitment = keccak256(bytes.concat(userCommitment, providerInfo.currentCommitment))`: [2](#0-1) [3](#0-2) 

Because `userCommitment` is a plain `bytes32` with no address binding, an attacker who observes User A's `request` transaction on-chain can submit an identical `userCommitment` in their own `request` call, obtaining a separate sequence number `j` backed by the same commitment value.

When User A later calls `reveal(provider, i, x_A, x_i)`, `x_A` becomes public. The attacker then calls `reveal(provider, j, x_A, x_j)`. The `reveal` function only checks `req.requester == msg.sender`: [4](#0-3) 

Since the attacker is the requester of request `j`, this check passes. The commitment verification also passes because `constructUserCommitment(x_A) == hash(x_A)` matches what was stored. The attacker receives `hash(x_A, x_j, 0)` — a random number derived from a contribution they did not independently generate.

The same root cause applies to `requestWithCallback` / `requestV2`. Here the raw `userContribution` (not its hash) is passed and immediately emitted in the `RequestedWithCallback` event: [5](#0-4) 

Since `revealWithCallback` has no `msg.sender` restriction ("Anyone can call this method"): [6](#0-5) 

An attacker who sees `x_A` in the event can submit `requestWithCallback(provider, x_A)`, then call `revealWithCallback` themselves. Because the provider's hash chain values are also public, the attacker can compute `hash(x_A, x_j, 0)` before even submitting their request and selectively participate only when the outcome is favorable.

### Impact Explanation

The Entropy protocol's security guarantee is: *"the result is random as long as either A or B are honest."* Commitment copying breaks this guarantee. An attacker who copies User A's commitment is not independently sampling their own `x_U`; they are reusing A's contribution. After A reveals, the attacker has full knowledge of `x_A` and `x_j` (provider's public hash chain), so they can compute the outcome before deciding to reveal. This gives the attacker a selective-outcome advantage over any application (e.g., on-chain lottery, game) that uses Entropy for randomness. For the `requestWithCallback` path the advantage is even stronger: the attacker can predict the outcome before committing at all.

### Likelihood Explanation

The attack is straightforward: watch the mempool or emitted events for `userCommitment` / `userContribution` values, submit a duplicate request (paying only the standard fee), and wait for the original user's reveal. No privileged access, leaked keys, or off-chain collusion is required. Any unprivileged Entropy user can execute this.

### Recommendation

Bind the user commitment to the requester's address. Change `constructUserCommitment` to include `msg.sender`:

```solidity
// In requestHelper, before storing:
bytes32 userCommitment = keccak256(abi.encodePacked(msg.sender, rawUserCommitment));
```

For the `request` path, the contract should re-hash the caller-supplied commitment together with `msg.sender` inside `requestHelper` rather than storing it verbatim. For `requestWithCallback`, the same binding should be applied to `constructUserCommitment`. The `reveal` / `revealWithCallback` functions must then reconstruct the commitment using `req.requester` instead of `msg.sender` (since `revealWithCallback` is permissionless).

### Proof of Concept

```
1. User A calls request(provider, keccak256(x_A), false) → sequence i
   (keccak256(x_A) is visible on-chain)

2. Attacker calls request(provider, keccak256(x_A), false) → sequence j
   (same commitment, different sequence number)

3. User A calls reveal(provider, i, x_A, x_i) → x_A is now public

4. Attacker observes x_A, computes hash(x_A, x_j, 0) off-chain.
   If favorable: calls reveal(provider, j, x_A, x_j) → succeeds.
   If unfavorable: does not reveal (forfeits fee only).

Result: Attacker obtains a random number derived from x_A with
        look-ahead knowledge of the outcome, violating the protocol's
        randomness guarantee.
```
