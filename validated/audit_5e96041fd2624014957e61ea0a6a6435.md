Audit Report

## Title
Integer Truncation in `FeeRate::fee()` Silently Zeroes Minimum Fee on Sub-Default `min_fee_rate` Configurations — (File: `util/types/src/core/fee_rate.rs`)

## Summary
`FeeRate::fee(weight)` computes `(self.0 * weight) / 1000` using integer division, which truncates to `0` whenever `fee_rate * weight < 1000`. The tx-pool admission gate in `check_tx_fee` uses this result directly: when `min_fee = 0`, the guard `fee < min_fee` is `fee < 0` — impossible for a `u64` — so every transaction, including zero-fee ones, is unconditionally admitted. Any node operator who sets `min_fee_rate` to a supported value in `[1, 999]` is silently exposed to zero-fee mempool flooding.

## Finding Description
`FeeRate::fee()` is defined at `util/types/src/core/fee_rate.rs` lines 34–37:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
```

`KW = 1000` (line 7). Integer division truncates toward zero: any `(fee_rate, weight)` pair where `fee_rate * weight < 1000` produces `Capacity::shannons(0)`.

This value flows directly into the admission gate at `tx-pool/src/util.rs` lines 45–52:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    let reject = Reject::LowFeeRate(...);
    return Err(reject);
}
```

When `min_fee == Capacity::shannons(0)`, the condition `fee < min_fee` is `u64 < 0`, which is always `false`. No rejection occurs regardless of the actual fee paid.

Concrete truncation cases:

| `min_fee_rate` | `tx_size` | `min_fee` computed | Zero-fee admitted? |
|---|---|---|---|
| 1000 (default) | 242 | 242 shannons | No |
| 1 | 999 | `1 * 999 / 1000 = 0` | **Yes** |
| 500 | 1 | `500 * 1 / 1000 = 0` | **Yes** |
| 5 | 199 | `5 * 199 / 1000 = 0` | **Yes** |

The configuration is fully supported: `resource/ckb.toml` line 212 documents `min_fee_rate = 1_000` as the default, and `util/app-config/src/legacy/tx_pool.rs` line 10 defines `DEFAULT_MIN_FEE_RATE: FeeRate = FeeRate::from_u64(1000)` with no lower-bound validation or truncation warning. Any operator-supplied value in `[1, 999]` is accepted silently.

## Impact Explanation
An attacker who knows a target node's `min_fee_rate` is below 1000 can craft valid transactions whose serialized size falls below `1000 / min_fee_rate` bytes and submit them with zero fee via the `send_transaction` RPC. `check_tx_fee` computes `min_fee = 0` and admits them unconditionally. This allows the attacker to flood the mempool (`max_tx_pool_size = 180 MB` by default) with zero-cost transactions, evicting or crowding out legitimate fee-paying transactions and degrading relay/mining service on the affected node. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, applicable to any deployment (testnet, private network, or mainnet nodes with operator-lowered fee rates) where `min_fee_rate < 1000`.

## Likelihood Explanation
The default `min_fee_rate = 1000` is not affected. However, operators running private chains, testnets, or low-traffic nodes commonly lower this value to attract transactions. No documentation warns that values below 1000 cause silent truncation. The attacker entry path requires no special privilege: submit a small zero-fee transaction via the public `send_transaction` RPC. The attack is repeatable at negligible cost (zero fee per transaction) and requires no cryptographic material beyond a valid UTXO.

## Recommendation
Replace floor division with ceiling division in `FeeRate::fee()` (`util/types/src/core/fee_rate.rs`):

```rust
pub fn fee(self, weight: u64) -> Capacity {
    // Ceiling division: (a + b - 1) / b
    let fee = self.0
        .saturating_mul(weight)
        .saturating_add(KW - 1)
        / KW;
    Capacity::shannons(fee)
}
```

This guarantees that any non-zero `fee_rate` applied to any non-zero `weight` produces at least 1 shannon, preserving the operator's intent regardless of the configured rate. Alternatively, add a post-condition: if `self.0 > 0 && weight > 0 && fee == 0 { return Capacity::one(); }`.

## Proof of Concept
1. Start a CKB node with `min_fee_rate = 1` in `ckb.toml`.
2. Construct a valid transaction with serialized size ≤ 999 bytes (achievable with a minimal 1-input/1-output structure).
3. Set outputs' total capacity equal to inputs' total capacity (zero fee).
4. Submit via `send_transaction` RPC.
5. Observe: node accepts the transaction. `FeeRate(1).fee(999) = 1 * 999 / 1000 = 0`, so `min_fee = 0` and `0 < 0` is false — no rejection.
6. Repeat in a loop to exhaust `max_tx_pool_size = 180 MB`.

Expected (correct) behavior: rejected with `LowFeeRate`. Actual behavior: admitted unconditionally.
