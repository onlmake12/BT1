### Title
Unauthenticated RPC Exposes Privileged Node-Control Methods to Any Local or Remote Caller — (`rpc/src/module/net.rs`, `rpc/src/module/pool.rs`, `rpc/src/server.rs`)

---

### Summary

The CKB JSON-RPC server implements no authentication or per-caller authorization mechanism. Any process that can reach the RPC port — any local process by default, or any remote host if the operator binds to a non-loopback address — can invoke privileged administrative methods such as `set_network_active`, `clear_tx_pool`, `remove_transaction`, `set_ban`, and `clear_banned_addresses` without restriction. This is the direct analog of the `flexGovernanceAdapter` finding: functions that should be callable only by a privileged role (the node operator) are callable by any unprivileged caller.

---

### Finding Description

**Root cause — no authentication in the RPC server**

`rpc/src/server.rs` builds the Axum router with `CorsLayer::permissive()` and no authentication middleware: [1](#0-0) 

The `RpcConfig` struct has no authentication fields whatsoever: [2](#0-1) 

The miner client sends HTTP Basic Auth headers when a username/password is embedded in the RPC URL: [3](#0-2) 

However, the RPC server never validates these headers — there is no corresponding server-side middleware that checks them. The server simply ignores any `Authorization` header.

**Privileged methods with no caller check**

`set_network_active` — disables all P2P network message processing with a single boolean: [4](#0-3) 

`clear_tx_pool` — atomically removes every pending transaction from the pool: [5](#0-4) 

`remove_transaction` — removes a specific transaction and all its dependents: [6](#0-5) 

`clear_banned_addresses` — erases the entire IP ban list: [7](#0-6) 

`set_ban` — inserts or deletes arbitrary IP/subnet bans: [8](#0-7) 

`add_node` / `remove_node` — forces or severs peer connections: [9](#0-8) 

None of these methods perform any caller identity check. The trait definitions carry no access-control annotation: [10](#0-9) [11](#0-10) 

**Attacker entry path**

The default bind address is `127.0.0.1:8114`: [12](#0-11) 

Any process running on the same host — a malicious dApp, a compromised dependency, a co-located container — can reach this port without any credential. If the operator changes `listen_address` to a non-loopback address (a supported and documented configuration option), the attack surface extends to any host on the network. The RPC documentation explicitly acknowledges this risk but provides no enforcement mechanism: [13](#0-12) 

---

### Impact Explanation

| Method | Impact |
|---|---|
| `set_network_active(false)` | Completely silences the node's P2P layer. The node stops syncing, stops relaying transactions and blocks, and becomes invisible to the network. Persistent calls prevent recovery. |
| `clear_tx_pool()` | Destroys all pending transactions. Users who paid fees to submit transactions lose their place in the queue; time-sensitive transactions (e.g., DAO withdrawals near deadlines) may miss their window. |
| `remove_transaction(hash)` | Targeted eviction of a specific transaction and its entire descendant chain, enabling griefing of individual users. |
| `clear_banned_addresses()` | Removes all IP bans, re-admitting previously banned malicious peers and undoing operator-applied network hygiene. |
| `set_ban(ip, "insert", ...)` | Bans legitimate peers, including the operator's own trusted nodes or whitelist peers, causing network partition. |
| `add_node(attacker_peer, ...)` | Forces the node to connect to an attacker-controlled peer, enabling eclipse-attack setup. |

The combination of `set_network_active(false)` and `clear_tx_pool()` can silently isolate a node and erase its mempool, causing it to miss blocks and lose all unconfirmed transactions — a severe availability and integrity impact.

---

### Likelihood Explanation

- **Default exposure:** Any process on the same host can reach `127.0.0.1:8114` without credentials. This includes browser-based JavaScript (via `localhost` fetch), malicious npm/cargo dependencies, or co-located services.
- **Non-default but supported exposure:** Operators who bind to `0.0.0.0` or a LAN address expose these methods to the entire network with zero authentication.
- **No exploit complexity:** A single HTTP POST with a JSON body is sufficient. No cryptographic material, no prior state, no race condition.
- **Attacker profile in scope:** "RPC caller" and "supported local CLI/RPC user" are explicitly listed as in-scope attacker profiles.

---

### Recommendation

1. **Add an authentication layer to the RPC server.** Implement token-based or HTTP Basic Auth validation in the Axum middleware stack in `rpc/src/server.rs`, reading a secret from `RpcConfig`.
2. **Introduce per-method privilege tiers.** Separate read-only methods (e.g., `get_block`, `tx_pool_info`) from write/administrative methods (e.g., `set_network_active`, `clear_tx_pool`, `set_ban`) and require authentication only for the latter.
3. **Add an `rpc_token` field to `RpcConfig`** (`util/app-config/src/configs/rpc.rs`) so operators can configure a secret that must be presented in the `Authorization` header.
4. **Document that the miner's Basic Auth URL credential is not validated by the server** and remove the false sense of security it implies.

---

### Proof of Concept

With a default CKB node running on `127.0.0.1:8114`, any local process can execute:

```bash
# Disable all P2P networking — node is now isolated
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":1}'

# Clear the entire transaction pool
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":2}'

# Remove a specific pending transaction by hash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"],"id":3}'

# Erase all IP bans, re-admitting previously banned peers
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_banned_addresses","params":[],"id":4}'
```

All four calls succeed with `"result": null` and no authentication challenge. No credentials, no signature, no privileged key required. The node's network is silenced and its mempool is empty after two HTTP requests.

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}

    fn set_ban(
        &self,
        address: String,
        command: String,
        ban_time: Option<Timestamp>,
        absolute: Option<bool>,
        reason: Option<String>,
    ) -> Result<()> {
        let ip_network = address.parse().map_err(|_| {
            RPCError::invalid_params(format!(
                "Expected `params[0]` to be a valid IP address, got {address}"
            ))
        })?;

        match command.as_ref() {
            "insert" => {
                let ban_until = if absolute.unwrap_or(false) {
                    ban_time.unwrap_or_default().into()
                } else {
                    unix_time_as_millis()
                        + ban_time
                            .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
                            .value()
                };
                self.network_controller
                    .ban(ip_network, ban_until, reason.unwrap_or_default());
                Ok(())
            }
            "delete" => {
                self.network_controller.unban(&ip_network);
                Ok(())
            }
            _ => Err(RPCError::invalid_params(format!(
                "Expected `params[1]` to be in the list [insert, delete], got {address}"
            ))),
        }

    fn add_node(&self, peer_id: String, address: String) -> Result<()> {
        let multiaddr = address.parse::<Multiaddr>().map_err(|e| {
            RPCError::invalid_params(format!("Error parsing address '{}': {}", address, e))
        })?;

        // Check if the parsed multiaddr already contains a peer ID
        let multiaddr_with_peer_id = if extract_peer_id(&multiaddr).is_some() {
            multiaddr
        } else {
            // Add the peer ID to the address
            format!("{}/p2p/{}", address, peer_id)
                .parse()
                .map_err(|e| {
                    RPCError::invalid_params(format!(
                        "Error adding peer ID to address: {}, err: {}",
                        address, e
                    ))
                })?
        };

        self.network_controller.add_node(multiaddr_with_peer_id);
        Ok(())
    }
