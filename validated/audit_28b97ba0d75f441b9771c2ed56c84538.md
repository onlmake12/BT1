Audit Report

## Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Returns Incorrect Withdrawal Capacity — (File: `util/dao/src/lib.rs`)

## Summary
`DaoCalculator::calculate_maximum_withdraw` computes a `u128` intermediate value `withdraw_counted_capacity` and narrows it to `u64` with a bare `as u64` cast (line 156). When the intermediate result exceeds `u64::MAX`, the upper 64 bits are silently discarded and the function returns a drastically incorrect (too-small) capacity with no error. Every other `u128→u64` narrowing in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern. The inconsistency is the root cause.

## Finding Description
In `util/dao/src/lib.rs` lines 152–156:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
```

The `as u64` cast silently discards the upper 64 bits if `withdraw_counted_capacity > u64::MAX`. No error is propagated; the function proceeds with a wrong value and returns `Ok(wrong_capacity)`.

All three other `u128→u64` narrowings in the same file use the safe pattern:
- Line 204: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- Lines 244–245: `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?`
- Line 258: `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?`

The existing overflow test (`check_withdraw_calculation_overflows`, `tests.rs` lines 295–350) uses `withdrawing_ar/deposit_ar ≈ 1.0000000001`. With that ratio, `withdraw_counted_capacity` stays just below `u64::MAX` after `occupied_capacity` is subtracted, so the overflow is caught downstream by `safe_add`. The test does **not** exercise the silent-truncation path where the cast itself produces a wrong value that `safe_add` then accepts.

## Impact Explanation
`calculate_maximum_withdraw` feeds three downstream paths:

1. **`withdrawed_interests` → `dao_field_with_current_epoch`**: The NervosDAO savings pool accumulator `current_s` is computed as `parent_s + nervosdao_issuance - withdrawed_interests`. A silently truncated (too-small) `maximum_withdraw` makes `withdrawed_interests` too small, inflating `current_s`. This value is embedded in every block header and is consensus-critical. Any future node with the corrected cast would compute a different `current_s` and diverge from the chain — **consensus deviation**.

2. **`transaction_fee`**: Fee = `maximum_withdraw - outputs_capacity`. A silently truncated `maximum_withdraw` that is smaller than `outputs_capacity` causes `safe_sub` to return `DaoError`, causing valid DAO withdrawal transactions to be incorrectly rejected by the tx-pool and block verifier — **economic damage / network disruption**.

3. **RPC `calculate_dao_maximum_withdraw`**: Users receive a silently wrong withdrawal amount and construct transactions with incorrect output capacities — **economic damage**.

This maps to: **Critical — Vulnerabilities which could easily cause consensus deviation** and **Critical — Vulnerabilities which could easily damage CKB economy**.

## Likelihood Explanation
The truncation requires `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. Since `ar` starts at `10^16` and grows monotonically via secondary issuance, the ratio `withdrawing_ar/deposit_ar` must exceed approximately 5× for a cell holding a near-maximum CKB balance. On mainnet this is a long-horizon condition (decades at current issuance rates). However:

- The condition is immediately and trivially reachable in devnet/testnet by setting `ar` values directly in the DAO field of block headers (as the existing test infrastructure already does).
- The defect is demonstrably present and inconsistent with the defensive pattern used everywhere else in the file.
- There is no defense-in-depth: no checked cast, no assertion, no comment acknowledging the assumption.

## Recommendation
Replace the bare `as u64` cast with the checked conversion already used everywhere else in the file:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

Add a test case where `withdraw_counted_capacity` itself (before `safe_add`) exceeds `u64::MAX` and verify that `DaoError::Overflow` is returned rather than a silently wrong value.

With the current code the assertion fails (returns `Ok` with a wrong value); with the fix it passes.
