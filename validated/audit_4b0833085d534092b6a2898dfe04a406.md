### Title
Duplicate Price IDs in `requestPriceUpdatesWithCallback` Permanently Lock User Funds - (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol`'s `requestPriceUpdatesWithCallback` accepts a `priceIds` array without checking for duplicate entries. When a request is created with duplicate price IDs (e.g., `[A, A]`), the downstream `parsePriceFeedUpdates` call inside `executeCallback` will always revert, making the request permanently unfulfillable and locking the user's deposited funds in the contract forever.

### Finding Description

`requestPriceUpdatesWithCallback` validates only that `priceIds.length <= MAX_PRICE_IDS` but performs no uniqueness check: [1](#0-0) 

The request is stored with the duplicate prefixes and the user's fee is immediately split: the Pyth protocol fee is accrued, and the remainder is locked inside the request struct: [2](#0-1) 

When `executeCallback` is later called, it invokes `parsePriceFeedUpdates` with the same duplicate `priceIds`: [3](#0-2) 

Inside `Pyth.sol`, `_parseSingleMerkleUpdate` scans `context.priceIds` for the first matching index `k` and fills `context.priceFeeds[k]` only if it is still zero: [4](#0-3) 

With `priceIds = [A, A]`, the loop always finds `k=0` first. `context.priceFeeds[0]` is filled, but `context.priceFeeds[1]` remains `id == 0`. The final validation loop then unconditionally reverts: [5](#0-4) 

Because `parsePriceFeedUpdates` reverts, `executeCallback` reverts entirely, `clearRequest` is never reached, and the request remains active forever. There is no cancel or refund path in the contract. The developers themselves flagged this exact risk in a TODO comment: [6](#0-5) 

### Impact Explanation

A user who submits a request with duplicate price IDs loses the full `msg.value` they paid. The provider fee portion (`req.fee`) is locked inside the request struct and can never be withdrawn because the only code path that credits it (`executeCallback`) always reverts for this request. There is no administrative rescue or cancel function in `Echo.sol`. The locked amount equals `msg.value` minus the already-accrued Pyth protocol fee.

### Likelihood Explanation

Any caller of `requestPriceUpdatesWithCallback` can trigger this, either by mistake (copy-paste error in a `priceIds` array) or deliberately (self-griefing). The function is fully permissionless. The `MAX_PRICE_IDS` limit of 10 means up to 10 slots can be filled with duplicates of a single ID, maximising the locked fee. No privileged role is required.

### Recommendation

Add a duplicate-check loop in `requestPriceUpdatesWithCallback` before storing the request, mirroring the pattern already used in `Scheduler.sol`: [7](#0-6) 

```solidity
for (uint i = 0; i < priceIds.length; i++) {
    for (uint j = i + 1; j < priceIds.length; j++) {
        if (priceIds[i] == priceIds[j]) revert DuplicatePriceId(priceIds[i]);
    }
}
```

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback(provider, t, [A, A], gasLimit)` paying the required fee.
2. Contract stores `req.priceIdPrefixes = [prefix(A), prefix(A)]` and locks `req.fee = msg.value - pythFee`.
3. Provider calls `executeCallback(provider, seq, updateData, [A, A])`.
4. Prefix check passes (both prefixes match).
5. `parsePriceFeedUpdates(updateData, [A, A], t, t)` is called.
6. `_parseSingleMerkleUpdate` fills `context.priceFeeds[0]` for price ID `A`; `context.priceFeeds[1].id` stays `0`.
7. Final loop at `Pyth.sol:391-394` reverts with `PriceFeedNotFoundWithinRange`.
8. `executeCallback` reverts; `clearRequest` is never called.
9. No other code path can clear the request or return `req.fee` to the user.
10. Funds are permanently locked.
