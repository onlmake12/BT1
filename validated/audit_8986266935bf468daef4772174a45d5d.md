Audit Report

## Title
Missing `request_limit` Guard in `get_cells_capacity` Enables Unbounded RocksDB Scan — (`util/indexer/src/service.rs`)

## Summary
`IndexerHandle::get_cells_capacity` iterates over all matching cells in RocksDB with no count-based limit, bounded only by a 10-second `TimeoutIterator`. Unlike `get_cells` and `get_transactions`, which both reject requests exceeding `self.request_limit` and stop early with `.take(limit)`, `get_cells_capacity` performs a full prefix scan on every call. An unprivileged caller can issue concurrent broad-prefix requests, each holding a worker thread for the full timeout duration, making the node's RPC interface unresponsive.

## Finding Description
In `util/indexer/src/service.rs`, `get_cells_capacity` (line 686) builds a RocksDB prefix iterator and folds over every matching cell:

```rust
// Line 720 — only bound is a 10-second timeout
let mut iter = TimeoutIterator::new(snapshot.iterator(mode).skip(skip), self.timeout_limit);

let capacity: u64 = iter
    .by_ref()
    .take_while(|(key, _value)| key.starts_with(&prefix))
    .filter_map(...)
    .sum();
```

There is no `request_limit` check and no `.take(N)` early-stop. By contrast, `get_cells` (lines 212–221) and `get_transactions` (lines 388–397) both enforce:

```rust
if limit > self.request_limit {
    return Err(Error::invalid_params(...));
}
```

and terminate the iterator with `.take(limit)`.

The default configuration sets `request_limit = usize::MAX` (line 98) and `timeout_limit = 10s` (line 99), meaning each `get_cells_capacity` call can scan millions of cells for the full 10 seconds. The `TimeoutIterator::next()` checks elapsed time only at the top of each call (lines 55–61), so a single expensive RocksDB read inside the loop body can overshoot the boundary. The RPC trait exposes `get_cells_capacity` with no `limit` parameter (lines 879–883), and the implementation passes directly to `IndexerHandle::get_cells_capacity` with no additional guard (lines 929–936).

## Impact Explanation
Concurrent `get_cells_capacity` requests with a broad script prefix (e.g., empty `args`) each hold a worker thread for up to 10 seconds while scanning the full matching cell set. With enough concurrent requests the RPC thread pool is saturated, causing all subsequent RPC calls to queue indefinitely. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash / unresponsiveness**. The evidence does not conclusively prove that P2P message processing shares the same thread pool and is also blocked, so the impact is bounded at the Note level based on what is proven.

## Likelihood Explanation
The `get_cells_capacity` RPC is part of the standard public CKB indexer API, requires no authentication, and is reachable by any HTTP client that can connect to the RPC port. The attack requires only repeated JSON-RPC POST requests. The default configuration (`request_limit = usize::MAX`, `timeout_limit = 10s`) maximises exposure. Any node with the indexer enabled and the RPC port reachable is affected.

## Recommendation
1. Add the same `request_limit` guard to `get_cells_capacity` that exists in `get_cells` and `get_transactions` — reject requests that would scan more than `self.request_limit` cells.
2. Alternatively, add an internal `.take(self.request_limit)` to the iterator chain so the scan stops after at most `request_limit` cells, returning a partial result or an explicit error.
3. Lower the default `timeout_limit` and document that `request_limit` should be configured for any publicly reachable node.

## Proof of Concept
Send 20–50 concurrent HTTP POST requests to the CKB RPC port:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells_capacity",
  "params": [{
    "script": {
      "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
      "hash_type": "type",
      "args": "0x"
    },
    "script_type": "lock"
  }]
}
```

Empty `args` matches every cell whose lock script uses this `code_hash`. Each request scans the full matching set (bounded only by the 10-second timeout). Subsequent RPC calls (`get_cells`, `get_block_template`, etc.) will queue indefinitely until the saturating requests time out.
