### Title
Peer Store Monopolization via Unbounded `announce=true` Nodes Messages — (`network/src/peer_store/peer_store_impl.rs`, `network/src/protocols/discovery/mod.rs`)

---

### Summary

A connected remote peer can send unlimited `Nodes(announce=true)` messages, each carrying up to 30 routable addresses (10 items × 3 addresses), filling the peer store to `ADDR_COUNT_LIMIT=16384` with addresses from a single /16 subnet. Once full, `check_purge`'s network-group eviction path computes `take(len/2)` where `len=1`, selecting **zero** groups for eviction, returns `EvictionFailed`, and permanently blocks all new address additions. The peer store is monopolized; legitimate addresses cannot enter.

---

### Finding Description

**No rate limiting on `announce=true` messages.**

In `received()`, the `state.received_nodes` guard only applies to `announce=false` messages: [1](#0-0) 

For `announce=true`, there is no per-session counter or cooldown. `verify_nodes_message` only disconnects if items exceed `ANNOUNCE_THRESHOLD=10` or addresses per item exceed `MAX_ADDRS=3`: [2](#0-1) 

So an attacker can send an unlimited stream of `Nodes(announce=true)` messages, each with exactly 10 items × 3 addresses = 30 addresses, without triggering any misbehavior response.

**`add_new_addrs` passes each address to `add_addr`:** [3](#0-2) 

The only filter is `is_valid_addr` (routability check). Addresses from any globally routable /16 subnet pass. The attacker does not need to control those IPs — they only need to advertise them.

**`add_addr` calls `check_purge` before inserting:** [4](#0-3) 

**The fatal flaw in `check_purge`'s network-group eviction:** [5](#0-4) 

When all 16384 stored addresses belong to the same /16 network group (as defined by `Group::IP4([bits[0], bits[1]])`): [6](#0-5) 

Then `peers_by_network_group.len() == 1`, so `take(len / 2)` = `take(0)` — **zero groups are selected for eviction**. `candidate_peers` is empty, and the function returns `Err(PeerStoreError::EvictionFailed)`. `add_addr` propagates this error, and the new address is silently dropped.

**`AddrManager.add` deduplicates by address:** [7](#0-6) 

So the attacker needs 16384 unique addresses from a /16 subnet (which has 65536 addresses — easily achievable).

**Freshly advertised addresses are always connectable** (`last_connected_at_ms=0`, `attempts_count=0`), so `check_purge`'s first eviction pass (non-connectable addresses) finds nothing: [8](#0-7) 

---

### Impact Explanation

Once the peer store is monopolized:
- `fetch_addrs_to_feeler` and `fetch_addrs_to_attempt` return only attacker-controlled addresses
- The node dials only attacker-controlled peers, enabling eclipse attack preconditions
- Legitimate peer addresses discovered via discovery are permanently rejected with `EvictionFailed`
- The `get_peers` RPC reflects only attacker-supplied addresses

The impact is scoped to peer diversity degradation and eclipse attack precondition, matching the stated scope.

---

### Likelihood Explanation

- Requires only a single active P2P connection (unprivileged)
- ~547 messages of 30 addresses each fills the store (16384 / 30 ≈ 547)
- No rate limiting, no per-session announce counter
- Attacker need not control the advertised IPs — any routable /16 subnet suffices
- The mathematical flaw (`take(len/2)` with `len=1`) is deterministic and always triggered

---

### Recommendation

1. **Rate-limit `announce=true` messages per session** — add a per-session counter or cooldown (e.g., max N announce messages per hour).
2. **Fix the `take(len/2)` eviction formula** — when `len == 1`, the formula produces `take(0)` and evicts nothing. Use `take(len.saturating_sub(1).max(1))` or always evict from the largest group regardless of `len/2`.
3. **Cap per-/16-group entries** in `add_addr` before insertion, rejecting addresses that would push any single group beyond a threshold (e.g., `ADDR_COUNT_LIMIT / expected_groups`).

---

### Proof of Concept

**Invalid as stated**: The PoC uses `225.0.0.0/24` (Class D multicast). `is_valid_addr` calls `is_reachable()` from the p2p library, which filters multicast addresses. Those addresses never reach `add_addr`.

**Corrected PoC** (conceptual):
1. Establish a P2P session with the victim node.
2. Send 547 `Nodes(announce=true)` messages, each containing 10 items with 3 unique routable addresses from `1.2.0.0/16` (e.g., `1.2.0.1` through `1.2.63.255`).
3. All 16384 addresses are stored; all share `Group::IP4([1, 2])`.
4. Send one more `Nodes(announce=true)` with a legitimate address from `8.8.8.0/24`.
5. `check_purge` runs: `len=1`, `take(0)` → no evictions → `EvictionFailed` → address rejected.
6. Assert: `fetch_addrs_to_feeler` returns only `1.2.x.x` addresses; the `8.8.8.x` address is absent.

The core vulnerability — unbounded announce ingestion combined with the `take(len/2)` eviction dead zone — is real and reachable by any connected peer.
