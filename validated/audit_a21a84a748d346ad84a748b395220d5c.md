Audit Report

## Title
Unbounded `NOT IN` SQL clause from `pool.dead_cells()` causes persistent DoS of rich-indexer RPC — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`, `get_cells_capacity.rs`)

## Summary
When `index_tx_pool: true` is configured, `get_cells` and `get_cells_capacity` collect every entry from the unbounded `Pool.dead_cells` `HashSet` and embed one SQL bind parameter per entry into a `NOT IN (…)` clause with no upper bound. An unprivileged attacker who submits ~1,000 single-input mempool transactions pushes the parameter count past SQLite's `SQLITE_MAX_VARIABLE_NUMBER` (default 999), causing every subsequent `get_cells` and `get_cells_capacity` call to return `Error::DB` for as long as the mempool remains large.

## Finding Description
`Pool.dead_cells` is a plain `HashSet<OutPoint>` with no capacity cap. [1](#0-0) 

`Pool::new_transaction` inserts every input of every accepted mempool transaction into that set with no guard. [2](#0-1) 

`get_cells` iterates the full set and constructs one `$N` placeholder per entry in a `NOT IN (…)` clause: [3](#0-2) 

`get_cells_capacity` has the identical pattern: [4](#0-3) 

SQLite enforces `SQLITE_MAX_VARIABLE_NUMBER` (default 999) at query execution time; exceeding it returns a driver error that is propagated as `Error::DB`. There are no existing checks or guards anywhere in this code path that limit the size of `dead_cells` before query construction.

## Impact Explanation
All callers of the `get_cells` and `get_cells_capacity` RPC methods receive persistent `Error::DB` responses for the duration of the attack. This constitutes a sustained, remotely-triggered crash of those RPC API endpoints, matching the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**.

The node's consensus and block-production paths are entirely unaffected; only the rich-indexer query surface is rendered non-functional.

## Likelihood Explanation
- Requires `index_tx_pool: true` in the node configuration (non-default, but a documented and supported option).
- Submitting ~1,000 minimal single-input transactions is trivially achievable with no PoW, no privileged access, and negligible cost (well under 1 MB of pool space against the 180 MB default `max_tx_pool_size`). [5](#0-4) 
- The attack is repeatable and self-sustaining as long as the attacker's transactions remain in the mempool.

## Recommendation
Replace the inline `NOT IN (…)` expansion with a bounded approach:

1. **Temp table / CTE**: Insert dead cells into a temporary table or CTE and join against it — both SQLite and PostgreSQL support this without parameter-count limits.
2. **Chunked exclusion**: Split dead cells into chunks of ≤500 and apply multiple `NOT IN` clauses per chunk.
3. **Hard cap with clear error**: If `dead_cells().count()` exceeds a configurable threshold (e.g., 500), return `Error::Params` immediately rather than attempting an oversized query.

## Proof of Concept
```rust
// Populate a mock Pool with 1,000 dead cells (SQLite limit = 999)
let mut pool = Pool::default();
for i in 0u32..1_000 {
    let tx = TransactionBuilder::default()
        .input(CellInput::new(OutPoint::new(h256!("0xdead").pack(), i), 0))
        .build();
    pool.new_transaction(&tx.into_view());
}
// pool.dead_cells().count() == 1,000
// Calling get_cells() now builds a NOT IN clause with 1,000 bind parameters,
// exceeding SQLite's SQLITE_MAX_VARIABLE_NUMBER (999) → returns Error::DB.
```

With a node configured with `index_tx_pool = true` and SQLite backend, submit 1,000 transactions each spending a distinct UTXO via `send_transaction` RPC, then call `get_cells` — it will return a DB error on every subsequent invocation until the mempool drains.
