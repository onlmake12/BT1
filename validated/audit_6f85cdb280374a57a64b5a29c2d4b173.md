### Title
On-Chain Revert Misclassified as Transient Error Causes Futile 5-Minute Retry Loop in Fortuna Keeper — (`apps/fortuna/src/eth_utils/utils.rs`)

---

### Summary

In Fortuna's `submit_tx` function, a confirmed-on-chain transaction that reverts (EVM status = 0) is classified as `backoff::Error::transient`. This causes `submit_tx_with_backoff` to retry the same call for up to 5 minutes, submitting and paying gas for each retry, even though the revert is deterministic and permanent. The `error_mapper` in `process_event_with_backoff` only intercepts `GasUsageEstimateError(NoSuchRequest)` during pre-flight estimation; it never intercepts `ReceiptError`, leaving all confirmed on-chain reverts to be retried as transient.

---

### Finding Description

In `apps/fortuna/src/eth_utils/utils.rs`, `submit_tx` checks the receipt status after a transaction is confirmed:

```rust
if receipt.status == Some(U64::from(0)) {
    return Err(backoff::Error::transient(SubmitTxError::ReceiptError(
        transaction.clone(),
        receipt.clone(),
    )));
}
```

`backoff::Error::transient` signals the `ExponentialBackoff` loop in `submit_tx_with_backoff` to retry. The backoff is configured with `max_elapsed_time: Some(Duration::from_secs(300))` — 5 minutes. Every retry re-submits the identical transaction to the chain, which will revert again for the same reason, consuming gas each time.

The `error_mapper` closure in `process_event_with_backoff` only matches one specific pattern:

```rust
if let backoff::Error::Transient {
    err: SubmitTxError::GasUsageEstimateError(tx, ContractError::Revert(revert)),
    ..
} = &e
{
    if let Ok(PythRandomErrorsErrors::NoSuchRequest(_)) = PythRandomErrorsErrors::decode(revert) { ... }
}
```

`ReceiptError` is never matched by this mapper, so all confirmed on-chain reverts fall through and remain `transient`.

On-chain reverts of `revealWithCallback` that are deterministically permanent include:
- `IncorrectRevelation` — provider's hash chain doesn't match commitment
- `AssertionFailure` — contract invariant broken
- `InvalidRevealCall` — wrong reveal method used
- `InsufficientGas` — the Entropy contract itself reverts when the calling context lacks gas (distinct from the user callback running out of gas)
- `NoSuchRequest` — if the request was cleared between gas estimation and submission

---

### Impact Explanation

Each retry of a permanently-reverting `revealWithCallback` transaction:
1. Pays gas for a transaction that will revert on-chain (keeper/provider funds drained).
2. Holds the spawned task for up to 5 minutes, during which the same sequence number is locked in `fulfilled_requests_cache` and cannot be re-processed by a backup replica.
3. Delays the keeper's nonce progression, potentially stalling other pending transactions queued behind it.

A user who creates an entropy request whose fulfillment deterministically reverts (e.g., by exploiting a state condition that causes `IncorrectRevelation` or `AssertionFailure`) can force the keeper to burn gas for 5 minutes per request. Multiple such requests compound the effect.

---

### Likelihood Explanation

Any entropy user can submit a `requestV2` call. The Fortuna keeper will attempt `revealWithCallback` for every such request. If the on-chain call reverts for any permanent reason (including edge cases in the Entropy contract's own logic), the retry loop activates automatically. No special privilege is required; the trigger is a normal user action.

---

### Recommendation

Classify `ReceiptError` as `backoff::Error::Permanent` in `submit_tx`, since a confirmed on-chain revert with status=0 cannot be resolved by resubmitting the same transaction:

```rust
if receipt.status == Some(U64::from(0)) {
    return Err(backoff::Error::permanent(SubmitTxError::ReceiptError(
        transaction.clone(),
        receipt.clone(),
    )));
}
```

Alternatively, extend the `error_mapper` in `process_event_with_backoff` to intercept `ReceiptError` and decode the revert reason, returning `backoff::Error::Permanent` for known-permanent error selectors (`IncorrectRevelation`, `AssertionFailure`, `InvalidRevealCall`, etc.) and `backoff::Error::Transient` only for errors that may resolve on retry.

    let error_mapper = |num_retries, e| {
        if let backoff::Error::Transient {
            err: SubmitTxError::GasUsageEstimateError(tx, ContractError::Revert(revert)),
            ..
        } = &e
        {
            if let Ok(PythRandomErrorsErrors::NoSuchRequest(_)) =
                PythRandomErrorsErrors::decode(revert)
            {
                let err = SubmitTxError::GasUsageEstimateError(
                    tx.clone(),
                    ContractError::Revert(revert.clone()),
                );
                // Slow down the retries if the request is not found.
                // This probably means that the request is already fulfilled via another process.
                // After 5 retries, we return the error permanently.
                if num_retries >= 5 {
                    return backoff::Error::Permanent(err);
                }
                let retry_after_seconds = match num_retries {
                    0 => 5,
                    1 => 10,
                    _ => 60,
                };
                return backoff::Error::Transient {
                    err,
                    retry_after: Some(Duration::from_secs(retry_after_seconds)),
                };
            }
        }
        e
    };
