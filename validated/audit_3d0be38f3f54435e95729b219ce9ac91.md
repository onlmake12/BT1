### Title
Expired OIS Rewards Permanently Locked in `pool_reward_custody` With No Recovery Mechanism — (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The Integrity Pool program distributes PYTH rewards from a `pool_reward_custody` token account. Earned rewards expire after approximately 53 epochs (~1 year). Once expired, a staker can no longer claim them. However, the integrity pool IDL exposes no instruction to recover or recycle those expired/unclaimed PYTH tokens from `pool_reward_custody`, leaving them permanently locked in the PDA.

---

### Finding Description

The OIS reward flow works as follows:

1. PYTH tokens are deposited into the `pool_reward_custody` associated token account (a PDA of the `PoolConfig`).
2. Stakers call `advance_delegation_record` to claim their earned rewards; this transfers tokens from `pool_reward_custody` to the staker's custody account.
3. Rewards expire after 53 epochs from the epoch in which they were earned.

The expiry is computed in the SDK as:

```ts
expiry: epochToDate(instructions.lowestEpoch + 53n)
``` [1](#0-0) 

The UI warns users that "Rewards expire one year from the epoch in which they were earned." [2](#0-1) 

After expiry, `advance_delegation_record` will no longer transfer those tokens to the staker. The tokens remain in `pool_reward_custody`. Reviewing the full integrity pool IDL, the only instructions present are:

- `advance` — advances the pool epoch
- `advance_delegation_record` — claims rewards for a delegator
- `update_y` — updates the reward rate
- Slash-related instructions (`create_slash_event`, `slash`)
- Publisher/pool configuration instructions [3](#0-2) 

There is **no** `withdraw_unclaimed_rewards`, `recover_expired_rewards`, or equivalent instruction. Once rewards expire, the PYTH tokens that were allocated to those rewards remain in `pool_reward_custody` indefinitely with no on-chain path to recover them.

The `pool_reward_custody` PDA is derived from `pool_config` and the PYTH token mint: [4](#0-3) 

---

### Impact Explanation

PYTH tokens deposited into `pool_reward_custody` for OIS rewards that are earned but not claimed before the 53-epoch expiry window become permanently locked in the PDA. There is no governance instruction, admin function, or permissionless path to recover them. This is a direct loss of protocol-owned PYTH tokens.

**Severity qualifier:** The reward rate `y` is currently set to 0 per OP-PIP-103, so no rewards are presently being distributed and no tokens are currently at risk. However, the structural lockup will apply immediately when `y` is set to a non-zero value via `update_y`. [5](#0-4) 

---

### Likelihood Explanation

- Likelihood is **low-to-medium** currently because `y = 0` and no rewards are flowing.
- Once `y > 0` is set by governance, any staker who earns rewards and fails to call `advance_delegation_record` within 53 epochs will cause those tokens to be permanently locked.
- Stakers who are inactive, lose access to their wallet, or are unaware of the expiry window are realistic vectors. At scale, the cumulative locked amount could be significant.

---

### Recommendation

Add a permissioned instruction (callable by the `pool_authority` or governance) to the integrity pool program that transfers expired/unclaimed reward tokens from `pool_reward_custody` back to a designated treasury or recycles them into the reward pool. This mirrors the fix applied in the referenced `PublicSaleINTX.sol` report (`withdrawUnallocatedIntx`).

---

### Proof of Concept

1. Governance sets `y > 0` via `update_y`.
2. Staker S delegates to publisher P and earns rewards in epoch E.
3. Staker S does not call `advance_delegation_record` for 53 epochs (e.g., wallet lost, user inactive).
4. After epoch `E + 53`, the reward expiry is reached: `epochToDate(lowestEpoch + 53n)`.
5. `advance_delegation_record` no longer transfers the expired reward tokens to S.
6. The PYTH tokens remain in `pool_reward_custody` (PDA of `PoolConfig`).
7. No instruction exists in the integrity pool program to recover these tokens.
8. Result: PYTH tokens permanently locked in `pool_reward_custody`. [6](#0-5) [7](#0-6)

  public async getClaimableRewards(
    stakeAccountPositions: PublicKey,
    simulationPayer?: PublicKey,
  ) {
    const instructions = await this.getAdvanceDelegationRecordInstructions(
      stakeAccountPositions,
      simulationPayer,
    );

    let totalRewards = 0n;

    for (const instruction of instructions.advanceDelegationRecordInstructions) {
      const tx = new Transaction().add(instruction);
      tx.feePayer = simulationPayer ?? this.wallet.publicKey;
      // eslint-disable-next-line @typescript-eslint/no-deprecated
      const res = await this.connection.simulateTransaction(tx);
      const val = res.value.returnData?.data[0];
      if (val === undefined) {
        continue;
      }
      const buffer = Buffer.from(val, "base64").reverse();
      totalRewards += BigInt("0x" + buffer.toString("hex"));
    }

    return {
      expiry:
        instructions.lowestEpoch === undefined
          ? undefined
          : epochToDate(instructions.lowestEpoch + 53n),
      totalRewards,
    };
  }
