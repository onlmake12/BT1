### Title
Silent u128→u64 Truncating Cast in `DaoCalculator::calculate_maximum_withdraw` Produces Wrong Withdrawal Amounts and Can Block NervosDAO Withdrawals — (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the DAO interest-adjusted withdrawal capacity using a `u128` intermediate (`withdraw_counted_capacity`), then casts it to `u64` with the **silent truncating** `as u64` operator. If the intermediate exceeds `u64::MAX`, the lower 64 bits are silently kept, producing a drastically wrong (much smaller) result. This is structurally inconsistent with every other analogous u128→u64 narrowing in the same file, which all use the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern. The wrong result causes the node to reject a valid DAO withdrawal transaction, permanently blocking the depositor from reclaiming their funds.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
// lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

`withdraw_counted_capacity` is a `u128`. The `as u64` cast in Rust is defined to **truncate** (keep the low 64 bits) with no error, no panic, and no `None`. If `withdraw_counted_capacity > u64::MAX`, the result is `withdraw_counted_capacity % 2^64` — which can be arbitrarily small (e.g., 0 or 1000 shannons) even when the true value is many times `u64::MAX`.

Contrast this with every other u128→u64 narrowing in the same file:

```rust
// line 245 — miner_issuance128
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)

// line 258 — ar_increase128
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) 

Both use `u64::try_from(...)` and propagate `DaoError::Overflow`. The `calculate_maximum_withdraw` function is the only place in the DAO calculator that uses the unsafe `as u64` cast.

The existing test `check_withdraw_calculation_overflows` (lines 295–350) only covers the sub-case where the truncated value, when added to `occupied_capacity`, itself overflows `u64` (triggering `safe_add`'s error path). It does **not** cover the silent-wrong-result sub-case where the truncated value is small but `safe_add` succeeds — in that case the function returns `Ok(wrong_small_value)` with no indication of error. [4](#0-3) 

---

### Impact Explanation

`calculate_maximum_withdraw` is called in two critical paths:

1. **Transaction fee verification** — `DaoCalculator::transaction_fee` → `transaction_maximum_withdraw` → `calculate_maximum_withdraw`. The fee is `maximum_withdraw - outputs_capacity`. If `maximum_withdraw` is silently truncated to a tiny value (e.g., 1000 shannons) while the user's output is the correct large amount, the subtraction underflows and the transaction is rejected with `Malformed Overflow transaction`. The depositor cannot withdraw their CKB. [5](#0-4) 

2. **RPC `calculate_dao_maximum_withdraw`** — returns the wrong (tiny) capacity to the caller, causing wallets and tooling to construct transactions that will always be rejected. [6](#0-5) 

In the worst sub-case (truncated value + `occupied_capacity` overflows `u64`), `safe_add` returns `DaoError::Overflow`, which the RPC surfaces as `CKBInternalError (-1) — Mathematics overflow`, also blocking withdrawal.

---

### Likelihood Explanation

The overflow condition is `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`, i.e., `withdrawing_ar / deposit_ar > u64::MAX / counted_capacity`. Given:

- Total CKB supply ≈ 3.36 × 10¹⁸ shannons → `counted_capacity` ≤ ~3.36 × 10¹⁸
- `ar` starts at 10¹⁶ and grows at ~`g2/c` per block ≈ 4 × 10⁻¹¹ per block on mainnet

For the ratio to exceed `u64::MAX / 3.36×10¹⁸ ≈ 5.5`, `ar` would need to grow by 5.5×, requiring an astronomically long time under normal issuance. **However**, the structural defect is real and confirmed by the inconsistency with `u64::try_from` used everywhere else in the same function's sibling calculations. Any future change to secondary issuance parameters, a custom chain spec with higher `secondary_epoch_reward`, or a deposit that is a large fraction of total supply held for many epochs, moves this from theoretical to practical. The bug is also reachable today on any chain where `ar` has grown sufficiently relative to the deposit epoch's `ar`.

---

### Recommendation

Replace the truncating cast with the same checked pattern used for `miner_issuance128` and `ar_increase128` in the same file:

```diff
-        let withdraw_capacity =
-            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
+        let withdraw_capacity =
+            Capacity::shannons(
+                u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
+            )
+            .safe_add(occupied_capacity)?;
```

This makes the overflow explicit and consistent with the rest of `DaoCalculator`.

---

### Proof of Concept

Construct a scenario where `counted_capacity * withdrawing_ar / deposit_ar` just exceeds `2^64`:

- `deposit_ar = 10_000_000_000_000_000` (10¹⁶, plausible initial value)
- `withdrawing_ar = 55_000_000_000_000_000` (5.5× growth)
- `counted_capacity = 3_355_443_200_000_000_000` shannons (~33.5 billion CKB, near total supply)
- `withdraw_counted_capacity = 3_355_443_200_000_000_000 * 55_000_000_000_000_000 / 10_000_000_000_000_000`
  `= 3_355_443_200_000_000_000 * 5.5 = 18_454_937_600_000_000_000`
  `> u64::MAX (18_446_744_073_709_551_615)`

`as u64` truncates to `18_454_937_600_000_000_000 - 2^64 = 8_193_526_290_448_384`.

`calculate_maximum_withdraw` returns `Ok(8_193_526_290_448_384 + occupied_capacity)` — roughly 81 million CKB — instead of the correct ~184 billion CKB. The depositor's withdrawal transaction, which correctly claims ~184 billion CKB, is rejected by the node's fee verifier because `maximum_withdraw (81M CKB) < outputs_capacity (184B CKB)`. [7](#0-6)

    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }

#[test]
fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
}
