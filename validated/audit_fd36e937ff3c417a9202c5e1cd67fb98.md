### Title
`ScriptHashTypeVerifier` validates only lock script hash type, silently skipping type script hash type — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and enforces that the hash type is within the consensus-permitted set (`ENABLED_SCRIPT_HASH_TYPE`), but it only inspects `output.lock()`. It never inspects `output.type_()`. A transaction output carrying a type script with a forbidden or reserved hash type passes this non-contextual gate entirely unchecked.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, `ScriptHashTypeVerifier::verify()` is the non-contextual guard that rejects transactions whose scripts use hash types outside the currently enabled consensus range:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())   // ← lock only
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

Every `CellOutput` in CKB has two scripts: a mandatory `lock` and an optional `type_`. The verifier checks `output.lock().hash_type()` but never calls `output.type_().to_opt()` to validate the type script's hash type. This is the direct analog of the ShortLongSpell bug: a derived/nested field (`vault.uToken`) is used where the direct field (`vault`) was required — here `output.lock()` is checked where `output.type_()` must also be checked.

The same omission appears in `CellbaseVerifier` in `block_verifier.rs` (lines 135–144), though it is harmless there because cellbase outputs are separately required to carry no type script at all (lines 127–133). [1](#0-0) [2](#0-1) 

---

### Impact Explanation

`ScriptHashTypeVerifier` is part of `NonContextualTransactionVerifier`, which is the earliest rejection gate — called before any database lookup, script execution, or fee check. Its purpose is to prevent transactions with consensus-illegal scripts from ever entering the tx-pool or being relayed.

Because the type script hash type is never validated here, a transaction whose output carries a type script with a reserved or future hash byte (any value not in `ENABLED_SCRIPT_HASH_TYPE`) passes this gate and is admitted to the tx-pool. Downstream effects:

1. **Tx-pool pollution / resource exhaustion**: The node expends CPU and memory resolving and partially verifying transactions that should have been dropped at the door. An attacker can flood the pool with such transactions at low cost.
2. **Block-template contamination**: The block assembler draws from the tx-pool. A transaction with an invalid type-script hash type can be selected for a block template. The resulting block will fail contextual/script verification on all peers, wasting miner work and potentially causing temporary chain-tip divergence.
3. **Consensus-rule bypass window**: `ENABLED_SCRIPT_HASH_TYPE` is the mechanism by which new hash types are gated behind hard forks. Bypassing the check for type scripts means the enforcement is asymmetric: lock scripts are gated, type scripts are not. A future hash type that is intentionally disabled could be used in a type script position without triggering the guard. [3](#0-2) 

---

### Likelihood Explanation

The attack requires no privilege. Any unprivileged actor can:

1. Construct a `TransactionView` whose output has a type script with `hash_type` set to a byte value outside `ENABLED_SCRIPT_HASH_TYPE` (e.g., a reserved value such as `3`).
2. Submit it via the public `send_transaction` JSON-RPC endpoint.

The transaction passes `NonContextualTransactionVerifier` (including `ScriptHashTypeVerifier`) and enters the pool. No key, no stake, no special peer relationship is required. [4](#0-3) 

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type when a type script is present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Validate lock script hash type (existing check)
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }

        // NEW: Validate type script hash type when present
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) =
                TryInto::<ScriptHashType>::try_into(type_script.hash_type())
            {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                })
                .into());
            }
        }
    }
    Ok(())
}
``` [1](#0-0)

### Proof of Concept

1. Build a transaction with one output whose `lock` script uses a valid hash type (e.g., `Data1 = 2`) and whose `type_` script uses a reserved/forbidden hash type byte (e.g., `0x03`).
2. Call `NonContextualTransactionVerifier::new(&tx, &consensus).verify()`.
3. Observe: the call returns `Ok(())` — the invalid type script hash type is never checked.
4. Submit the transaction via `send_transaction` RPC; the node accepts it into the tx-pool.
5. Observe the tx-pool now holds a transaction that will fail script execution on every peer, wasting pool capacity and potentially polluting block templates.

The root cause is at: [5](#0-4) 

where `output.lock().hash_type()` is the only hash type ever read — `output.type_()` is never consulted.
