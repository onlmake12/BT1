### Title
High-Cycle Orphan Transaction Permanently Stuck in Orphan Pool When Verify Queue Is Full at Parent Processing Time — (File: tx-pool/src/process.rs)

### Summary

When a high-cycle orphan transaction's parent is successfully processed but the verify queue is full at that exact moment, the orphan remains stuck in the `OrphanPool` with no automatic retry mechanism. The orphan will not be re-attempted when queue space becomes available, and will only be removed upon expiry — analogous to the `deposit_failed` stuck state caused by a failing `swapTokensForExactTokens` recovery operation.

---

### Finding Description

In `process_orphan_tx` (`tx-pool/src/process.rs`), when a parent transaction is successfully processed, the function performs a BFS over the orphan pool to find and promote orphan transactions that depend on it. For high-cycle orphans (those with `orphan.cycle > self.tx_pool_config.max_tx_verify_cycles`), the code attempts to enqueue them into the `VerifyQueue` rather than processing them inline:

```rust
// tx-pool/src/process.rs lines 598–624
if orphan.cycle > self.tx_pool_config.max_tx_verify_cycles {
    let orphan_id = orphan.tx.proposal_short_id();
    match self
        .enqueue_verify_queue(
            orphan.tx.clone(),
            false,
            Some((orphan.cycle, orphan.peer)),
        )
        .await
    {
        Ok(_) => {
            self.remove_orphan_tx(&orphan_id).await;
        }
        Err(reject) => {
            warn!(
                "process_orphan {} failed to enqueue verify queue: {}; keep orphan from {}",
                orphan.tx.hash(),
                reject,
                tx.hash(),
            );
            // orphan stays in pool — no retry scheduled
        }
    }
``` [1](#0-0) 

`enqueue_verify_queue` calls `VerifyQueue::add_tx`, which returns `Err(Reject::Full(...))` when the queue's `total_tx_size` is at or above `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`:

```rust
// tx-pool/src/component/verify_queue.rs lines 215–219
if self.is_full(tx_size) {
    return Err(Reject::Full(format!(
        "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
        tx.hash()
    )));
}
``` [2](#0-1) 

When this `Err` branch is taken, the orphan is kept in the `OrphanPool` but **no retry is scheduled**. The only callers of `process_orphan_tx` are:

1. `after_process` — triggered when a transaction is successfully processed (lines 500, 536)
2. `remove_orphan_txs_by_attach` — triggered when a block is attached (line 872) [3](#0-2) [4](#0-3) 

For a typical orphan with a single parent, once the parent is processed (or confirmed in a block), `process_orphan_tx` will not be called again with that parent. The orphan is therefore stuck in the `OrphanPool` until it expires after `ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL`:

```rust
// tx-pool/src/component/orphan.rs lines 15–16
pub(crate) const ORPHAN_TX_EXPIRE_TIME: u64 = 100 * MAX_BLOCK_INTERVAL;
pub(crate) const DEFAULT_MAX_ORPHAN_TRANSACTIONS: usize = 100;
``` [5](#0-4) 

The existing test `process_orphan_tx_keeps_high_cycle_orphan_when_verify_queue_is_full` confirms this behavior — the orphan stays in the pool and is not enqueued — but does not verify that the orphan will ever be retried:

```rust
// tx-pool/src/component/tests/chunk.rs lines 347–373
assert!(service.orphan.read().await.contains_key(&orphan_id));
assert!(!service.verify_queue.read().await.contains_key(&orphan_id));
``` [6](#0-5) 

---

### Impact Explanation

- **Orphan pool slot exhaustion**: The `OrphanPool` is capped at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries. Stuck high-cycle orphans occupy these slots, causing legitimate orphan transactions to be randomly evicted via `limit_size`.
- **Transaction loss**: A user's high-cycle orphan transaction is effectively lost for the duration of the expiry window (up to `100 * MAX_BLOCK_INTERVAL`), requiring resubmission.
- **DoS on orphan pool**: An attacker can deliberately fill the orphan pool with stuck high-cycle orphans, preventing legitimate orphan transactions from being stored or promoted.

---

### Likelihood Explanation

An unprivileged relay peer can:
1. Submit up to 100 high-cycle orphan transactions (without parents) to fill the `OrphanPool`.
2. Flood the `VerifyQueue` with large transactions to keep it at capacity (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`).
3. Submit the parents of the orphans.
4. The parents are processed, `process_orphan_tx` is called, enqueue fails, orphans remain stuck.

No privileged access, leaked keys, or majority hashpower is required. The attack is reachable via the standard P2P relay protocol.

---

### Recommendation

Implement a retry mechanism for high-cycle orphan transactions that failed to enqueue due to a full verify queue. Options include:

1. **Periodic retry**: Maintain a "pending retry" list of orphan IDs that failed to enqueue, and retry them on a timer or when the verify queue drains below a threshold.
2. **Queue drain notification**: Subscribe to a notification when the verify queue has available space and retry pending orphans at that point.
3. **Inline processing fallback**: When the verify queue is full and the orphan's declared cycle count is only slightly above `max_tx_verify_cycles`, fall through to inline `_process_tx` processing instead of silently keeping the orphan.

---

### Proof of Concept

1. Peer A submits a high-cycle orphan transaction `child_tx` (referencing `parent_tx` output, declared cycles > `max_tx_verify_cycles`) via the relay protocol. `child_tx` enters `OrphanPool`.
2. Peer B (or Peer A) floods the `VerifyQueue` with many large transactions, keeping `total_tx_size` near `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`.
3. Peer A submits `parent_tx`. It is processed successfully. `after_process` calls `process_orphan_tx(&parent_tx)`.
4. Inside `process_orphan_tx`, `child_tx` is found. Since `child_tx.cycle > max_tx_verify_cycles`, `enqueue_verify_queue` is called. It returns `Err(Reject::Full(...))`.
5. `child_tx` remains in `OrphanPool`. No retry is scheduled.
6. The `VerifyQueue` drains. `child_tx` is never retried because `process_orphan_tx` is not called again with `parent_tx`.
7. `child_tx` expires after `ORPHAN_TX_EXPIRE_TIME` and is silently dropped. The user's transaction is lost.
