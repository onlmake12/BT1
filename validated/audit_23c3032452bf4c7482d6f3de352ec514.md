### Title
`transfer()` Used for ETH Refund in `verifyUpdate()` Causes DoS for Contract Callers - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` uses `payable(msg.sender).transfer(...)` to refund excess ETH to callers. The 2300-gas stipend imposed by `transfer()` will cause the entire call to revert when `msg.sender` is a smart contract whose `receive()`/`fallback()` function consumes more than 2300 gas, permanently blocking those callers from using `verifyUpdate()` with any overpayment.

### Finding Description
In `PythLazer.verifyUpdate()`, after checking that `msg.value >= verification_fee`, the contract attempts to refund the surplus using `transfer()`:

```solidity
// lazer/contracts/evm/src/PythLazer.sol, lines 74-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
```

`transfer()` hard-caps the gas forwarded to the recipient at 2300. Any smart contract caller whose `receive()` or `fallback()` function performs even a minimal state write (e.g., emitting an event, updating a counter) will exceed this limit. The resulting out-of-gas revert propagates back through `verifyUpdate()`, causing the entire transaction to fail. The caller cannot work around this by sending exactly `verification_fee` because `verification_fee` is a storage variable that can be changed by the owner at any time, making exact-fee submission unreliable in practice.

The `Governance.sol` instance of `transfer()` (line 118) is disqualified because it requires a trusted guardian-set governance action and is not reachable by an unprivileged actor.

### Impact Explanation
Any smart contract integrating `PythLazer` as a Lazer price-update verifier — e.g., a DeFi protocol that calls `verifyUpdate()` and whose `receive()` function does more than a bare ETH accept — will be permanently unable to call `verifyUpdate()` whenever it sends more than `verification_fee`. Because `verification_fee` is mutable by the owner, even a caller that previously sent the exact fee may begin overpaying after a fee reduction, triggering the refund path and the DoS. This breaks the core Lazer verification flow for an entire class of on-chain integrators.

### Likelihood Explanation
Lazer is designed to be called by on-chain contracts that consume verified price payloads. It is standard practice for such contracts to send a small ETH buffer above the expected fee to avoid reverts from fee increases. The combination of a mutable `verification_fee` and a `transfer()`-based refund makes this a realistic, reachable failure mode for any non-trivial contract caller.

### Recommendation
Replace `transfer()` with a low-level `call()` and validate the return value:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Refund failed");
}
```

This forwards all available gas to the recipient, eliminating the 2300-gas restriction while still reverting on failure.

### Proof of Concept

1. Deploy a contract `Attacker` with a `receive()` function that emits an event (costs >2300 gas).
2. `Attacker` calls `PythLazer.verifyUpdate{value: verification_fee + 1}(validUpdate)`.
3. `verifyUpdate` reaches line 76 and calls `payable(msg.sender).transfer(1)`.
4. The EVM forwards only 2300 gas to `Attacker.receive()`, which runs out of gas emitting the event.
5. The `transfer()` reverts, propagating the revert through `verifyUpdate()`.
6. `Attacker` can never successfully call `verifyUpdate()` with any overpayment, regardless of the validity of the update payload.
