### Title
Entropy Provider Sequence Number Exhaustion via Abandoned Non-Callback Requests — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract's `requestHelper` permanently increments `providerInfo.sequenceNumber` on every `request()` / `requestV2()` call. There is no mechanism to cancel a request and restore the consumed sequence number. An unprivileged attacker can make many non-callback requests and deliberately never call `reveal()`, permanently consuming sequence numbers from the provider's finite hash chain. When `maxNumHashes` is configured, only that many abandoned non-callback requests are needed to trigger `LastRevealedTooOld` for all subsequent legitimate requests, creating a repeatable DoS.

---

### Finding Description

Every call to `requestHelper` unconditionally increments the provider's `sequenceNumber`:

```solidity
uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
    revert EntropyErrors.OutOfRandomness();
providerInfo.sequenceNumber += 1;
``` [1](#0-0) 

The `clearRequest` function, called only on successful `reveal` / `revealWithCallback`, zeroes the storage slot but **never decrements `sequenceNumber`**:

```solidity
function clearRequest(address provider, uint64 sequenceNumber) internal {
    ...
    req.sequenceNumber = 0;
    ...
}
``` [2](#0-1) 

For non-callback requests made via `request()`, only the original `msg.sender` (the requester) can call `reveal()`:

```solidity
if (req.requester != msg.sender) {
    revert EntropyErrors.Unauthorized();
}
``` [3](#0-2) 

This means the provider (Fortuna) **cannot** advance `currentCommitmentSequenceNumber` by fulfilling the attacker's requests. The `numHashes` field for every new request is computed as:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber - providerInfo.currentCommitmentSequenceNumber
);
if (providerInfo.maxNumHashes != 0 && req.numHashes > providerInfo.maxNumHashes) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [4](#0-3) 

Once the attacker has `maxNumHashes` unfulfilled non-callback requests outstanding, every subsequent legitimate `request()` call reverts with `LastRevealedTooOld`. The provider's only recourse is to call `advanceProviderCommitment()` (which requires revealing their own hash chain values) or re-register entirely.

The `endSequenceNumber` bound is set at registration time as `provider.sequenceNumber + chainLength`: [5](#0-4) 

The Fortuna service already anticipates full exhaustion and re-registers automatically when `endSequenceNumber <= sequenceNumber`: [6](#0-5) 

But re-registration does not help against the `LastRevealedTooOld` DoS path, which only requires `maxNumHashes` abandoned requests — a much smaller number than the full chain length.

---

### Impact Explanation

- **`LastRevealedTooOld` DoS**: If the provider has set `maxNumHashes` (recommended to bound gas costs), an attacker needs only `maxNumHashes` non-callback requests left unrevealed to block all new requests. The provider must call `advanceProviderCommitment()` to recover, which requires on-chain transactions and introduces service interruption.
- **Full chain exhaustion**: With enough capital, an attacker can exhaust the entire `chainLength`, forcing a provider re-registration. Any in-flight legitimate requests at re-registration time still resolve, but new requests are blocked until re-registration completes.
- **Repeated attack**: After the provider recovers (via `advanceProviderCommitment` or re-registration), the attacker can repeat the cycle.

The impact is a **denial of service on the Entropy randomness service** for all users of the targeted provider, including DeFi protocols relying on on-chain randomness.

---

### Likelihood Explanation

- Any unprivileged address can call `request()` by paying the required fee. The attack is **not free** — the attacker permanently loses fees paid. This is the key difference from the Alchemix M-14 case.
- For the `LastRevealedTooOld` path, the cost is `maxNumHashes × feeInWei`. If `maxNumHashes` is small (e.g., a few hundred), the cost is low relative to the disruption caused.
- The attack is repeatable: after the provider recovers, the attacker can immediately repeat.
- On low-fee chains (e.g., L2s), the per-request cost is negligible, making the attack economically viable.

---

### Recommendation

1. **Add a request cancellation mechanism**: Allow the original requester to cancel a non-callback request and receive a partial refund, while restoring the sequence number slot (or at minimum, advancing `currentCommitmentSequenceNumber` past the cancelled slot).
2. **Alternatively, allow the provider to forcibly advance past stale requests**: Extend `advanceProviderCommitment` to be callable permissionlessly (or by the provider) to skip over abandoned non-callback requests without requiring the requester's cooperation.
3. **Charge a non-refundable cancellation fee or time-lock requests**: Introduce a time-based expiry after which abandoned requests are automatically cleared and `currentCommitmentSequenceNumber` can be advanced past them.

---

### Proof of Concept

```solidity
// Attacker contract
contract EntropyGriefer {
    IEntropy entropy;
    address provider;

    constructor(address _entropy, address _provider) {
        entropy = IEntropy(_entropy);
        provider = _provider;
    }

    // Step 1: Call this maxNumHashes times to exhaust the LastRevealedTooOld limit
    function consumeSequenceNumber(bytes32 userCommitment) external payable {
        uint128 fee = entropy.getFee(provider);
        // Use non-callback request so only this contract can reveal
        entropy.request{value: fee}(provider, userCommitment, false);
        // Deliberately never call reveal() -- sequence number is permanently consumed
    }

    // After maxNumHashes calls, all new requests from legitimate users revert with LastRevealedTooOld
}
```

The attacker calls `consumeSequenceNumber` exactly `maxNumHashes` times. After this, any call to `request()` or `requestV2()` against the targeted provider reverts with `EntropyErrors.LastRevealedTooOld`, blocking all legitimate users until the provider calls `advanceProviderCommitment()`.
