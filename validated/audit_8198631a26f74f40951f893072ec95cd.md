### Title
No Timeout on Miner RPC HTTP Client Causes Indefinite Hang and Task Accumulation - (File: miner/src/client.rs)

---

### Summary

The `Rpc` HTTP client in the CKB miner is built without any request timeout. Every call to `get_block_template` and `submit_block` can hang indefinitely if the CKB node RPC endpoint stops responding. In `block_on_submit` mode this blocks the calling thread permanently; in async mode, spawned Tokio tasks accumulate without bound, exhausting runtime resources.

---

### Finding Description

In `Rpc::new()`, the `hyper_util` legacy HTTP client is constructed with no timeout:

```rust
let client = HttpClient::builder(TokioExecutor::new())
    .build::<_, Full<Bytes>>(https);
``` [1](#0-0) 

Each incoming RPC request is dispatched by spawning a new Tokio task that calls `client.request(req).await` with no `tokio::time::timeout` wrapper and no cancellation token:

```rust
loop_handle.spawn(async move {
    let request = match client.request(req).await ...
``` [2](#0-1) 

The `MinerClientConfig` struct exposes `poll_interval`, `rpc_url`, `block_on_submit`, and `listen`, but defines no timeout field whatsoever: [3](#0-2) 

This means both the `get_block_template` polling path and the `submit_block` path are unprotected. Contrast this with `tx-pool/src/block_assembler/mod.rs`, where the block assembler's outbound notify HTTP calls are explicitly wrapped with `tokio::time::timeout(notify_timeout, client.request(req))`: [4](#0-3) 

No equivalent protection exists in the miner client.

---

### Impact Explanation

**Thread blocking (high severity path):** When `block_on_submit = true`, `submit_block` calls `self.handle.block_on(future)`, which synchronously blocks the calling thread until the future resolves: [5](#0-4) 

If the RPC server never responds, this thread is permanently blocked. The miner stops submitting valid blocks and stops processing new work, halting mining entirely.

**Task accumulation (async path):** When `block_on_submit = false`, each poll cycle spawns a new Tokio task via `loop_handle.spawn(...)`. With `poll_interval` firing repeatedly and each spawned task hanging indefinitely, tasks accumulate without bound, exhausting Tokio runtime memory and thread pool capacity. [2](#0-1) 

The net result is a complete availability loss for the miner process: it can no longer fetch block templates or submit solved blocks.

---

### Likelihood Explanation

The entry path requires only that the CKB node RPC endpoint (configured via `rpc_url`) becomes unresponsive — a realistic condition under network partition, node crash, or a misconfigured endpoint. No privileged access, no cryptographic material, and no majority hashpower is required. The miner is a supported local CLI tool explicitly listed in the bounty scope. The `poll_interval` loop means the condition is triggered automatically and repeatedly without any attacker action after initial setup.

---

### Recommendation

1. Set a connection and response timeout on the `HttpClient` builder, e.g. using `hyper-timeout` (already present in `Cargo.lock`).
2. Wrap each `client.request(req).await` call with `tokio::time::timeout(duration, ...)`, mirroring the pattern already used in `tx-pool/src/block_assembler/mod.rs` line 701.
3. Add a `rpc_timeout_millis` field to `MinerClientConfig` so the timeout is user-configurable, with a safe default (e.g. 30 seconds).

---

### Proof of Concept

1. Configure `ckb-miner` with `rpc_url` pointing to a host that accepts TCP connections but never sends an HTTP response (e.g., `nc -l 8114` with no output).
2. Start `ckb-miner`. The `get_block_template` RPC call is issued immediately.
3. With `block_on_submit = true`: the miner thread blocks on `handle.block_on(future)` and never returns. The miner produces no output and submits no blocks indefinitely.
4. With `block_on_submit = false`: each `poll_interval` tick spawns a new hung Tokio task. After N ticks, N tasks are stuck in `client.request(req).await`. Memory and runtime thread usage grow monotonically until the process is killed or OOMs. [6](#0-5)

    pub fn new(url: Uri, handle: Handle) -> Rpc {
        let (sender, mut receiver) = mpsc::channel(65_535);
        let stop_rx: CancellationToken = new_tokio_exit_rx();

        let https = hyper_tls::HttpsConnector::new();
        let client = HttpClient::builder(TokioExecutor::new()).build::<_, Full<Bytes>>(https);
        let loop_handle = handle.clone();
        handle.spawn(async move {
            loop {
                tokio::select! {
                    Some(item) = receiver.recv() => {
                        let (sender, call): RpcRequest = item;
                        let req_url = url.clone();
                        let request_json = serde_json::to_vec(&call).expect("valid rpc call");

                        let mut req = Request::builder().uri(req_url).method("POST").header(CONTENT_TYPE, "application/json");

                        if let Some(value) = parse_authorization(&url) {
                            req = req
                                .header(hyper::header::AUTHORIZATION, value);
                        }
                        let req = req.body(Full::new(Bytes::from(request_json))).unwrap();
                        let client = client.clone();
                        loop_handle.spawn(async move {
                            let request = match client
                                .request(req)
                                .await
                                .map(|res|res.into_body())
                            {
                                Ok(body) => BodyExt::collect(body).await.map_err(RpcError::Http).map(|t| t.to_bytes()),
                                Err(err) => Err(RpcError::Client(err)),
                            };
                            if sender.send(request).is_err() {
                                error!("rpc response send back error")
                            }
                        });
                    },
                    _ = stop_rx.cancelled() => {
                        info!("Rpc server received exit signal, exit now");
                        break
                    },
                    else => break
                }
            }
        });

        Rpc { sender }
    }
