Audit Report

## Title
Unbounded `dead_cells` NOT IN Clause Causes Persistent Indexer RPC DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`, `util/indexer-sync/src/pool.rs`)

## Summary
When `index_tx_pool=true`, `Pool::new_transaction` inserts every input of every accepted pending transaction into an unbounded `HashSet<OutPoint>` with no size cap. Both `get_cells` and `get_cells_capacity` then collect the entire set and emit a single SQL `NOT IN (…)` clause with one bound parameter per dead cell. SQLite rejects queries exceeding 999 bound variables; PostgreSQL rejects queries exceeding 65 535 parameters. An attacker who submits enough valid pending transactions pushes the parameter count past either threshold, causing every subsequent `get_cells`/`get_cells_capacity` RPC call to return a DB error for as long as those transactions remain pending.

## Finding Description
`Pool::new_transaction` inserts all inputs of every accepted pending transaction into `dead_cells` with no upper bound: [1](#0-0) 

`get_cells` collects the entire `dead_cells` set and builds a single `NOT IN (…)` clause with one `$N` placeholder per entry, with no limit on the number of placeholders emitted: [2](#0-1) 

`get_cells_capacity` has an identical unbounded construction: [3](#0-2) 

The `request_limit` field only caps the number of rows returned and provides no guard on the NOT IN clause size: [4](#0-3) 

`TxPoolConfig` enforces `max_tx_pool_size` (bytes) and `max_tx_verify_cycles`, but neither directly caps the total number of inputs across all pending transactions, so `dead_cells` can grow without bound: [5](#0-4) 

## Impact Explanation
Once `dead_cells` exceeds 999 entries (SQLite) or 65 535 entries (PostgreSQL), every call to `get_cells` or `get_cells_capacity` returns a DB error. The indexer is effectively unavailable for live-cell queries for the duration the offending transactions remain pending. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation
An attacker needs only to own ≥1 000 live UTXOs (achievable by splitting funds with cheap transactions) and submit ~100 pending transactions each spending ~10 distinct inputs. This is within default pool size limits, requires no privileged access, and uses only standard `send_transaction` RPC calls. The attacker's funds are not lost; they remain in pending transactions. The failure persists until the transactions are committed or expire (hours, given the default `expiry_hours`).

## Recommendation
1. **Cap the NOT IN clause**: Before building the clause, truncate `dead_cells` to a safe maximum (e.g., 500 for SQLite, 30 000 for PostgreSQL) or chunk the query into multiple batches.
2. **Use a temporary table / CTE**: Insert dead cells into a temporary table and JOIN against it instead of using a variable-length NOT IN list.
3. **Enforce a hard limit on `Pool.dead_cells`**: Reject or evict entries once the set exceeds a configurable threshold, and expose that threshold as a node operator setting.

## Proof of Concept
1. Enable `index_tx_pool = true` in the indexer config.
2. Pre-fund a wallet with ≥1 000 live cells.
3. Submit 100 valid pending transactions, each spending 10 distinct inputs → `Pool.dead_cells` grows to 1 000 entries.
4. Call `get_cells` with any valid search key.
5. **Expected result (SQLite)**: the node returns a DB error such as `"too many SQL variables"` because the generated NOT IN clause contains 1 000 `$N` placeholders, exceeding SQLite's 999-variable limit.
6. The error persists for every `get_cells`/`get_cells_capacity` call until the pending transactions are committed or expire.
