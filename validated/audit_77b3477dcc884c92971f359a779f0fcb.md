### Title
`addFunds()` Deposit Cap Bypass Allows Permanent Subscriptions to Accumulate Balance Far Exceeding `MAX_DEPOSIT_LIMIT` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `MAX_DEPOSIT_LIMIT` cap on permanent subscriptions in `Scheduler.sol` is enforced only against the **per-transaction deposit amount** (`msg.value`), not against the **resulting cumulative balance** (`status.balanceInWei + msg.value`). An unprivileged user can call `addFunds()` repeatedly with amounts at or just below the limit, accumulating a balance that is an arbitrary multiple of `MAX_DEPOSIT_LIMIT` in a permanent subscription from which withdrawals are permanently blocked.

---

### Finding Description

`MAX_DEPOSIT_LIMIT` is set to `100 ether` in `SchedulerConstants.sol`. [1](#0-0) 

The `addFunds()` function in `Scheduler.sol` checks:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [2](#0-1) 

The guard compares only `msg.value` (the single deposit) against the cap. It does **not** compare `status.balanceInWei + msg.value` against the cap. After the check passes, `status.balanceInWei` is unconditionally incremented. The same flaw exists in `createSubscription()`, though there the initial balance equals `msg.value` so the first deposit is correctly bounded. [3](#0-2) 

Withdrawals from permanent subscriptions are permanently blocked:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [4](#0-3) 

---

### Impact Explanation

Any user who creates a permanent subscription can call `addFunds()` N times with `msg.value = MAX_DEPOSIT_LIMIT` (100 ether each), locking `N × 100 ether` permanently in the contract. Because permanent subscriptions cannot withdraw, the ETH is irrecoverably locked beyond the intended cap. The `MAX_DEPOSIT_LIMIT` is rendered completely ineffective as a protection mechanism for permanent subscriptions.

---

### Likelihood Explanation

The entry path is fully permissionless — any address can create a permanent subscription and call `addFunds()` with no privileged role required. The `addFunds()` function is `external payable` with no access control beyond subscription activity status. The bypass requires only repeated standard ETH transfers, which is trivially achievable. [5](#0-4) 

---

### Recommendation

Replace the per-transaction check with a post-addition cumulative balance check:

```solidity
status.balanceInWei += msg.value;
if (params.isPermanent && status.balanceInWei > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

This ensures the total locked balance in any permanent subscription never exceeds `MAX_DEPOSIT_LIMIT`, regardless of how many individual deposits are made.

---

### Proof of Concept

1. Alice calls `createSubscription{value: 100 ether}(params)` with `params.isPermanent = true`. The check `100 ether > 100 ether` is **false**, so it passes. `status.balanceInWei = 100 ether`.

2. Alice calls `addFunds{value: 100 ether}(subscriptionId)`. The check `100 ether > 100 ether` is **false** again. `status.balanceInWei = 200 ether`.

3. Alice repeats step 2 ten times. `status.balanceInWei = 1100 ether`, permanently locked, with `MAX_DEPOSIT_LIMIT = 100 ether` completely bypassed.

4. `withdrawFunds()` reverts for permanent subscriptions unconditionally, so the excess ETH is irrecoverable. [6](#0-5)
