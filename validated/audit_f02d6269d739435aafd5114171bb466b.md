### Title
WeightUnitsFlow Fee Estimator Permanently Retains Evicted Transactions, Enabling Attacker-Controlled Fee Rate Inflation — (File: `util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

The `WeightUnitsFlow` fee estimator records every accepted transaction in its historical flow data but has no mechanism to remove entries when transactions are evicted from the pool. An unprivileged attacker can exploit this by repeatedly submitting and RBF-evicting transactions to inflate the historical flow data, causing `estimate_fee_rate` to return artificially elevated fee rate recommendations to all users querying the node.

---

### Finding Description

**Root cause — `reject_tx` is a no-op for `WeightUnitsFlow`:**

In `util/fee-estimator/src/estimator/mod.rs`, the `FeeEstimator::reject_tx` dispatch explicitly does nothing for the `WeightUnitsFlow` variant:

```rust
pub fn reject_tx(&self, tx_hash: &Byte32) {
    match self {
        Self::Dummy | Self::WeightUnitsFlow(_) => {}   // ← no-op
        Self::ConfirmationFraction(algo) => algo.write().reject_tx(tx_hash),
    }
}
``` [1](#0-0) 

By contrast, `ConfirmationFraction` properly calls `drop_tx` → `remove_unconfirmed_tx` with `count_failure: true`, fully reversing the effect of `accept_tx`. [2](#0-1) 

**`accept_tx` permanently writes to `self.txs`:**

Every transaction that enters the pending pool is recorded in `self.txs` keyed by the current tip block number:

```rust
pub fn accept_tx(&mut self, info: TxEntryInfo) {
    if self.current_tip == 0 { return; }
    let item = TxStatus::new_from_entry_info(info);
    self.txs
        .entry(self.current_tip)
        .and_modify(|items| items.push(item))
        .or_insert_with(|| vec![item]);
}
``` [3](#0-2) 

Because `TxStatus` stores only `{ weight, fee_rate }` with no tx hash, there is no structural way to remove a specific transaction from `self.txs` even if `reject_tx` were implemented. [4](#0-3) 

**How `self.txs` drives fee estimates:**

`do_estimate` calls `sorted_flowed`, which collects **all** entries from `self.txs` within the historical window (2× `target_blocks`) and uses them to compute `flow_speed_buckets`:

```rust
fn sorted_flowed(&self, historical_tip: BlockNumber) -> Vec<TxStatus> {
    self.txs
        .iter()
        .filter(|&(&num, _)| num >= historical_tip)
        .flat_map(|(_, statuses)| statuses.to_owned())
        .collect()
    // ...
}
``` [5](#0-4) 

The flow speed is then used to predict future transaction arrivals:

```rust
let added_weight = flow_speed_buckets[bucket_index] * target_blocks;
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
let passed = current_weight + added_weight <= removed_weight;
``` [6](#0-5) 

Inflating `flow_speed_buckets` increases `added_weight`, causing lower fee-rate buckets to fail the `passed` condition, which forces the estimator to return a higher fee rate.

**The callback wiring confirms the asymmetry:**

In `shared/src/shared_builder.rs`, `accept_tx` is called on every `register_pending` event, while `reject_tx` is called on every `register_reject` event — but for `WeightUnitsFlow` the latter is silently discarded: [7](#0-6) 

---

### Impact Explanation

An attacker who inflates `self.txs` causes `estimate_fee_rate` (exposed via the `experiment_estimate_fee_rate` RPC) to return fee rates higher than the true pool state warrants. Every user or wallet that relies on this RPC to set transaction fees will overpay. The effect persists for the full historical window (2× `MAX_TARGET` blocks ≈ hundreds of blocks) and can be continuously renewed. This is a direct financial harm to all fee-estimator users on the node, analogous to the Morpho "delta" that socializes a worse rate to all P2P borrowers.

---

### Likelihood Explanation

The attack is reachable by any unprivileged tx-pool submitter via the public `send_transaction` RPC or P2P relay. The attacker submits transactions (recorded in `self.txs`), then evicts them via RBF (since `min_rbf_rate > min_fee_rate` is the default config). The RBF fee premium is the only cost; the replacement transactions remain in the pool, so the attacker retains their capital. The attack can be repeated each block to maintain inflated flow data indefinitely. No privileged access, key material, or majority hashpower is required.

---

### Recommendation

1. Change `WeightUnitsFlow::accept_tx` to store the tx hash alongside `TxStatus` so individual entries can be identified.
2. Implement a real `reject_tx` for `WeightUnitsFlow` that removes the corresponding entry from `self.txs`.
3. Alternatively, record a negative-weight "cancellation" entry when a transaction is evicted, so the net flow contribution is zero.
4. Consider mitigations for flash-submit-and-evict attacks (e.g., only recording transactions that survive at least one block in the pool before contributing to flow statistics).

---

### Proof of Concept

```
1. Attacker calls send_transaction(tx_A[1..N], fee_rate=F)
   → accept_tx fires for each → self.txs[tip] gains N entries with weight W and fee_rate F

2. Attacker calls send_transaction(tx_B[1..N], fee_rate=F+min_rbf_rate)
   where tx_B[i] spends the same inputs as tx_A[i]
   → RBF evicts tx_A[1..N] → reject_tx fires → NO-OP for WeightUnitsFlow
   → self.txs[tip] still contains the N original entries

3. self.txs now records 2N entries worth of flow but only N txs are in the pool.

4. estimate_fee_rate(target=T) calls sorted_flowed → sees 2N entries
   → flow_speed = 2 * actual_flow
   → added_weight = 2 * actual_added_weight
   → lower fee-rate buckets fail the `passed` condition
   → returned fee rate is higher than warranted

5. Repeat each block to maintain the inflated state.
   All users querying estimate_fee_rate overpay fees.
```
