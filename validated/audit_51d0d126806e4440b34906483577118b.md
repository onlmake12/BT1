### Title
Callback Fires Before Pool Size Enforcement in `_submit_entry`, Causing Inconsistent External State — (File: `tx-pool/src/process.rs`)

### Summary
In `tx-pool/src/process.rs`, the `_submit_entry` function fires the `call_pending`/`call_proposed` callback immediately after inserting a transaction into the pool, but before the caller `submit_entry` completes the remaining state-enforcement steps — specifically `limit_size`. If `limit_size` subsequently evicts the just-inserted transaction, external subscribers receive both a "new transaction accepted" notification and a "transaction rejected" notification for the same transaction. This violates the Checks-Effects-Interactions pattern and produces an inconsistent observable state for the indexer and RPC subscription clients.

### Finding Description

`_submit_entry` (`tx-pool/src/process.rs`, lines 1016–1037) inserts the transaction into the pool and immediately fires the registered callback:

```rust
fn _submit_entry(tx_pool, status, entry, callbacks) -> Result<HashSet<TxEntry>, Reject> {
    let (succ, evicts) = match status {
        TxStatus::Fresh    => tx_pool.add_pending(entry.clone())?,   // state written
        TxStatus::Gap      => tx_pool.add_gap(entry.clone())?,
        TxStatus::Proposed => tx_pool.add_proposed(entry.clone())?,
    };
    if succ {
        match status {
            TxStatus::Fresh | TxStatus::Gap => callbacks.call_pending(&entry),   // ← callback fires HERE
            TxStatus::Proposed              => callbacks.call_proposed(&entry),
        }
    }
    Ok(evicts)
}
```

The caller `submit_entry` (lines 96–170) still has critical state-enforcement work to do **after** `_submit_entry` returns:

```rust
let evicted = _submit_entry(tx_pool, status, entry.clone(), &self.callbacks)?;
// call_reject for evicted txs
for evict in evicted { self.callbacks.call_reject(tx_pool, &evict, reject); }
// conflict cache cleanup
tx_pool.remove_conflict(&entry.proposal_short_id());
// ← pool-size enforcement: may evict the just-added tx
tx_pool.limit_size(&self.callbacks, Some(&entry.proposal_short_id()))
       .map_or(Ok(()), Err)?;
```

`limit_size` (`tx-pool/src/pool.rs`, lines 292–329) selects the lowest-fee-rate entry for eviction. If the just-inserted transaction has the lowest fee rate, it is removed and `call_reject` is fired for it:

```rust
callbacks.call_reject(self, &entry, reject);   // fires for the same tx that call_pending already fired for
```

The registered `call_pending` callback (`shared/src/shared_builder.rs`, lines 559–566) performs two externally visible actions before `limit_size` runs:

1. `notify_pending.notify_new_transaction(notify_tx_entry)` — broadcasts to all RPC subscription clients and the indexer overlay.
2. `fee_estimator_clone.accept_tx(tx_hash, entry_info)` — records the transaction as accepted in the fee estimator.

The registered `call_reject` callback (`shared/src/shared_builder.rs`, lines 576–602) then fires from inside `limit_size`, sending `notify_reject_transaction` and calling `fee_estimator.reject_tx`.

### Impact Explanation

**RPC subscription clients** receive a `new_transaction` event followed by a `rejected_transaction` event for the same transaction hash. Clients that act on the first event (e.g., marking inputs as spent, updating UI state) must handle an unexpected reversal.

**Indexer dead-cell overlay** (`util/indexer-sync/src/pool.rs`, lines 125–135): upon receiving `new_transaction`, the indexer marks the transaction's input `OutPoint`s as dead cells. During the window before `rejected_transaction` arrives, any indexer query for those cells returns "consumed by pool tx" — an incorrect answer. A script author or application relying on the indexer to check cell liveness before constructing a transaction may build on a false premise.

**Fee estimator**: `accept_tx` is called before `reject_tx` for the same transaction. Depending on the estimator's internal accounting, this can skew fee-rate recommendations upward or downward, affecting the fee advice given to all subsequent RPC callers of `estimate_fee_rate`.

### Likelihood Explanation

Any unprivileged tx-pool submitter — via `send_transaction` RPC or P2P relay — can trigger this condition whenever the pool is at or near `max_tx_pool_size`. An attacker can deliberately saturate the pool with minimum-fee transactions to bring it to the limit, then submit a target transaction with a fee rate just below the eviction threshold. The eviction path through `limit_size` is a normal, reachable code path, not an edge case.

### Recommendation

Move `call_pending`/`call_proposed` to after all state-enforcement steps are complete. Concretely, fire the callback only after `limit_size` confirms the transaction will be retained (i.e., `limit_size` returns `None` for the current entry). This mirrors the correct CEI ordering: complete all pool state mutations first, then notify external observers.

### Proof of Concept

1. Fill the tx pool to `max_tx_pool_size` with low-fee-rate transactions.
2. Submit a new transaction via `send_transaction` RPC whose fee rate is the lowest in the pool.
3. Inside `submit_entry` → `_submit_entry`: the transaction is inserted via `add_pending`; `call_pending` fires immediately, sending `notify_new_transaction` to all subscribers and calling `fee_estimator.accept_tx`.
4. Control returns to `submit_entry`; `limit_size` is called, selects the just-inserted transaction as the eviction candidate, removes it, and calls `call_reject` — sending `notify_reject_transaction` to all subscribers and calling `fee_estimator.reject_tx`.
5. `submit_entry` returns `Err(Reject::Full)`.
6. **Observable result**: RPC subscription clients and the indexer have seen "transaction X accepted" followed by "transaction X rejected". During the window between steps 3 and 4, the indexer incorrectly reports the transaction's input cells as consumed.

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
