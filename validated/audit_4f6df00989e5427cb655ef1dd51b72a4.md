### Title
`ScriptHashTypeVerifier` Enforces Enabled Hash-Type Constraint Only on Lock Scripts, Not Type Scripts — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates that each output's **lock script** `hash_type` is within the consensus-permitted set (`ENABLED_SCRIPT_HASH_TYPE`). It never performs the same check on the output's **type script**. This mirrors the external report's pattern exactly: a restriction is applied to one surface (lock scripts / marketplace approvals) but silently omitted from a parallel surface (type scripts / OTC transfers), leaving a gap that any unprivileged transaction sender can exploit.

---

### Finding Description

`ScriptHashTypeVerifier` is part of `NonContextualTransactionVerifier` and is intended to enforce the consensus rule that only enabled `ScriptHashType` variants may appear in transaction outputs. The implementation reads:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

The loop body calls `output.lock().hash_type()` exclusively. `output.type_().to_opt()` is never consulted. A transaction output carrying a type script whose `hash_type` is a defined-but-not-yet-enabled variant (e.g., `ScriptHashType::Data3`, confirmed by the existing test `test_not_enabled_hash_type_output_lock`) passes this verifier without error.

The existing unit test confirms the asymmetry: `test_not_enabled_hash_type_output_lock` verifies that `Data3` in a **lock** script is rejected, but there is no corresponding test for `Data3` in a **type** script, because the verifier never checks it.

The `check_data` path in `util/gen-types/src/extension/check_data.rs` does validate both lock and type scripts for *valid enum membership*, but it does not enforce the *enabled* subset — that is the sole responsibility of `ScriptHashTypeVerifier`, which fails to cover type scripts.

---

### Impact Explanation

**Impact: Medium**

1. **Bypass of non-contextual gate**: Any unprivileged RPC caller or P2P relay peer can submit a transaction whose type script carries a disallowed `hash_type`. The transaction passes `NonContextualTransactionVerifier` (the cheap, early gate) and enters the tx-pool, forcing the node to run the full, expensive `ContextualTransactionVerifier` / script execution path before the transaction is ultimately rejected.

2. **Cheap DoS amplification**: Because the non-contextual check is the intended cheap filter, bypassing it allows an attacker to force disproportionate CPU work (script group resolution, VM scheduling) per submitted transaction.

3. **Consensus inconsistency surface**: If a miner assembles a block containing such a transaction (e.g., because the miner's node also relies on the same incomplete non-contextual check), the block will be invalid on other nodes that catch the error during contextual script execution. Different nodes may produce different error codes (`ScriptHashTypeNotPermitted` vs. a script-execution error), creating divergent rejection reasons for the same invalid transaction.

4. **Protocol invariant violation**: The stated invariant — "Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules" — is only half-enforced, directly analogous to the NFT contract's `onlyApprovedContracts` modifier that restricts one transfer path while leaving another open.

---

### Likelihood Explanation

**Likelihood: Low**

Exploiting this requires a transaction sender to deliberately craft an output with a type script using a not-yet-enabled `hash_type` variant. Such variants (`Data3`, etc.) are defined in the enum but not in `ENABLED_SCRIPT_HASH_TYPE`. Normal wallets and SDK users will never produce such transactions. The gap is reachable by any unprivileged RPC caller (`send_transaction`) or P2P relay peer without any special privilege.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` for each output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

Add a corresponding unit test analogous to `test_not_enabled_hash_type_output_lock` that places `ScriptHashType::Data3` in a type script and asserts rejection.

3. Transaction enters the tx-pool.
4. `verify_rtx` → `ContextualTransactionVerifier` → script execution fails on the `Data3` type script.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
