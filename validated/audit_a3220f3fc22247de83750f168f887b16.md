### Title
Off-by-one in `MAX_VMS_COUNT` spawn guard allows 17th VM creation - (`script/src/scheduler.rs`)

### Summary

The spawn guard at line 532 uses strict `>` instead of `>=`, allowing one extra VM to be created beyond the intended `MAX_VMS_COUNT = 16` limit.

### Finding Description

`MAX_VMS_COUNT` is defined as `16`: [1](#0-0) 

The guard in `Scheduler::process_message_box` for the `Message::Spawn` arm reads: [2](#0-1) 

When `suspended.len() + instantiated.len()` is exactly `16`, the expression `16 > 16` evaluates to `false`, so the guard does **not** fire and `boot_vm` is called at line 539, creating a 17th VM. The correct operator is `>=`. [3](#0-2) 

### Impact Explanation

A script author submitting a V2 (Meepo hardfork) transaction can:
1. Spawn exactly 16 child VMs (filling `suspended + instantiated` to 16).
2. Issue one additional `spawn` syscall.
3. The guard passes, and a 17th VM is instantiated — exceeding the resource limit by one.

The impact is **limited**: all nodes running the same binary have the same bug, so there is no consensus divergence between peers. The excess is exactly one VM (one extra ~4 MB CKB-VM address space allocation per script group execution). This is a resource-limit bypass, not a catastrophic exploit.

### Likelihood Explanation

The path is fully attacker-controlled: submit a transaction whose lock/type script is a V2 ELF that calls `spawn` 17 times. No privileged access, no PoW, no social engineering required. The bug is reachable on any node that has activated the Meepo hardfork.

### Recommendation

Change `>` to `>=` on line 532:

```rust
// Before (buggy):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

### Proof of Concept

Write a unit test that:
1. Boots a scheduler with a V2 script.
2. Spawns exactly `MAX_VMS_COUNT` (16) child VMs, keeping all alive (e.g., blocked on a pipe read).
3. Issues one more `spawn` syscall.
4. With the current `>` check, the call succeeds (returns `SUCCESS`).
5. With the corrected `>=` check, the call returns `MAX_VMS_SPAWNED` (error code 8). [4](#0-3)
