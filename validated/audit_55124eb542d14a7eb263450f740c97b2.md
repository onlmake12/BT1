### Title
Missing Chain Reorganization (Detached Block) Notification in RPC Subscription System — (`rpc/src/module/subscription.rs`, `chain/src/verify.rs`)

---

### Summary

The CKB RPC subscription system exposes `new_tip_header` and `new_tip_block` topics that fire only for the single new canonical tip block. During a chain reorganization, the system emits no notification for detached (rolled-back) blocks and no notification for intermediate attached blocks in the fork chain. An RPC subscriber relying on these events to track confirmed transactions will silently accumulate incorrect state across any reorg, with no mechanism to detect or recover from the gap.

---

### Finding Description

When a new best block is found and a fork/reorg occurs, `ConsumeUnverifiedBlockProcessor` in `chain/src/verify.rs` computes a `ForkChanges` struct containing both `attached_blocks` and `detached_blocks`: [1](#0-0) 

After committing the database transaction, the code calls `update_tx_pool_for_reorg` with the full fork information (both detached and attached blocks), but then calls `notify_new_block` with **only the single new tip block**: [2](#0-1) 

`notify_new_block` fans out to all internal subscribers and ultimately to the RPC subscription layer: [3](#0-2) 

In `SubscriptionRpcImpl::new`, the single block received from `new_block_receiver` is published to both `new_tip_header_sender` and `new_tip_block_sender`: [4](#0-3) 

The `Topic` enum has no `DetachedBlock`, `Reorg`, or equivalent topic: [5](#0-4) 

Consequently, during a reorg where blocks `A1..An` are detached and `B1..Bm` are attached, a subscriber receives exactly one event — for `Bm` (the new tip). The subscriber has no way to learn:
- Which blocks were detached (and thus which previously "confirmed" transactions are now unconfirmed)
- Which intermediate blocks `B1..B(m-1)` were attached

Additionally, when the broadcast channel lags, messages are silently dropped and the subscriber only logs an error and continues, with no indication of which specific events were lost: [6](#0-5) 

---

### Impact Explanation

An RPC caller (exchange, wallet, DeFi application) subscribing to `new_tip_block` or `new_tip_header` to track transaction finality will:

1. Mark transactions in detached blocks as "confirmed" when they are not.
2. Receive no signal that a reorg occurred, so it cannot re-query chain state.
3. Have no way to reconstruct the correct canonical chain state from subscription events alone.

The concrete security consequence is **double-spend acceptance**: an application that credits a user upon seeing a transaction in a `new_tip_block` event will not learn that the block was later detached, allowing the transaction sender to spend the same outputs on the new canonical chain.

---

### Likelihood Explanation

Short reorgs (1–2 blocks) are a normal occurrence on any PoW network due to propagation latency and require no attacker. Any block relayer or sync peer can trigger a reorg by propagating a competing chain with higher total difficulty. The CKB documentation itself explicitly acknowledges reorgs as a routine event: [7](#0-6) 

Applications that rely on the subscription API for finality tracking — a natural and documented use case — are exposed to this gap on every reorg, however caused.

---

### Recommendation

1. Add a `DetachedBlock` (or `Reorg`) subscription topic that emits one event per detached block during a reorganization, including the detached block's header and hash.
2. Emit `new_tip_block` events for each block in `fork.attached_blocks()`, not only the final tip, so subscribers can reconstruct the full fork.
3. When `RecvError::Lagged` occurs, notify the subscriber explicitly (e.g., via a special gap-indicator message) so it knows to re-sync state rather than silently continuing.

---

### Proof of Concept

1. Connect a WebSocket or TCP client and subscribe to `new_tip_block`.
2. Arrange for two competing chains to exist (e.g., via a network partition or by submitting two valid blocks at the same height to different nodes).
3. When the longer chain wins and a reorg occurs, observe that the subscriber receives exactly one `new_tip_block` event (the new tip), with no events for the detached blocks.
4. The subscriber's view of "confirmed" transactions in the detached blocks is now permanently incorrect, with no notification to trigger a re-query.

The relevant code path is:

- `chain/src/verify.rs`: `find_fork` → `rollback` → `reconcile_main_chain` → `notify_new_block(block)` (only the tip)
- `notify/src/lib.rs`: `handle_notify_new_block` fans out only the single block
- `rpc/src/module/subscription.rs`: `new_block_receiver.recv()` publishes only that one block to `new_tip_block_sender` [8](#0-7)

    fn handle_notify_new_block(&self, block: BlockView) {
        trace!("New block event {:?}", block);
        let block_hash = block.hash();
        // notify all subscribers
        for subscriber in self.new_block_subscribers.values() {
            let block = block.clone();
            let subscriber = subscriber.clone();
            self.handle.spawn(async move {
                if let Err(e) = subscriber.send(block).await {
                    error!("Failed to notify new block, error: {}", e);
                }
            });
        }

        // notify all watchers
        for watcher in self.new_block_watchers.values() {
            if let Err(e) = watcher.send(block_hash.clone()) {
                error!("Failed to notify new block watcher, error: {}", e);
            }
        }

        // notify script
        if let Some(script) = self.config.new_block_notify_script.clone() {
            let script_timeout = self.timeout.script;
            self.handle.spawn(async move {
                let args = [format!("{block_hash:#x}")];
                match timeout(script_timeout, Command::new(&script).args(&args).status()).await {
                    Ok(ret) => match ret {
                        Ok(status) => debug!("The new_block_notify script exited with: {status}"),
                        Err(e) => error!(
                            "Failed to run new_block_notify_script: {} {:?}, error: {}",
                            script, args[0], e
                        ),
                    },
                    Err(_) => ckb_logger::warn!("new_block_notify_script {script} timed out"),
                }
            });
        }
