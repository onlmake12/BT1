Audit Report

## Title
Zero-cycle DB reads via `size=0` probe in data-loading syscalls — (`script/src/syscalls/load_header.rs`)

## Summary
`LoadHeader::ecall` performs a full RocksDB header lookup unconditionally before inspecting the requested buffer size. Cycle charging is computed from `store_data`'s return value (`real_size = min(size, full_size)`). When a script sets the size field to 0, `real_size=0`, `transferred_byte_cycles(0)=0`, and the DB read is entirely unmetered. The same pattern exists in every other data-loading syscall in the codebase, enabling a ~7.5× amplification of RocksDB I/O relative to metered cycle cost.

## Finding Description
In `script/src/syscalls/load_header.rs` lines 163–168, `fetch_header` is called unconditionally before any size inspection: [1](#0-0) 

`fetch_header` for `Source::Transaction(SourceEntry::HeaderDep)` calls `self.sg_data.tx_info.data_loader.get_header(&block_hash)`: [2](#0-1) 

`data_loader.get_header` resolves through `DataLoaderWrapper` directly to `ChainStore::get_block_header` — a live RocksDB read with no caching layer: [3](#0-2) 

After the unconditional DB read, `load_full` delegates to `store_data`, which computes `real_size = cmp::min(size, full_size)` and returns `real_size`: [4](#0-3) 

The cycle charge at line 175 uses this return value: [5](#0-4) 

`transferred_byte_cycles(0)` returns 0: [6](#0-5) 

There is no fixed base cost per syscall invocation — no `SYSCALL_CYCLES_BASE` or equivalent constant exists anywhere in the codebase. The only cycle cost charged is from `transferred_byte_cycles(len)`. When `size=0`, the entire RocksDB read is unmetered.

The same pattern is confirmed in `load_block_extension.rs`: [7](#0-6) 

And `grep_search` confirms `transferred_byte_cycles` applied to `wrote_size` (not `full_size`) in `load_cell.rs`, `load_input.rs`, `load_tx.rs`, `load_witness.rs`, and `load_script.rs`.

Note: `load_cell_data.rs` has a partial mitigation — it explicitly checks `if size == 0` and returns early at lines 62–69 — but this does not apply to `load_header`, `load_block_extension`, or the other syscalls. [8](#0-7) 

## Impact Explanation
The 70,000,000-cycle budget is the primary mechanism bounding total node work per script. A normal `load_header` call on a ~208-byte header costs ~52 data-transfer cycles plus a small number of RISC-V instruction cycles (~8), allowing roughly 1,166,667 reads per budget. With `size=0`, only the RISC-V instruction cycles are charged, allowing roughly 8,750,000 reads per budget — a ~7.5× amplification of RocksDB I/O relative to metered cost. Because the same flaw exists across all data-loading syscalls (except `load_cell_data`), an attacker can combine them for maximum I/O pressure. A transaction submitted via `send_transaction` RPC triggers script verification on every receiving node in the tx-pool, making this a network-wide DoS vector. This matches: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The attack requires only a valid transaction with at least one `header_dep`. No privileged access, no key material, and no majority hashpower is needed. The size-probe pattern (`*size_addr = 0`) is a standard two-phase load idiom in CKB scripts, so it is indistinguishable from legitimate usage at the API level. The transaction can be submitted by any unprivileged user via the public RPC, and every node that receives it will execute the amplified DB reads during tx-pool admission. The attack is trivially repeatable and scriptable.

## Recommendation
Charge cycles based on `full_size` (bytes fetched from the DB), not `real_size` (bytes written to the script buffer). Modify `store_data` in `script/src/syscalls/utils.rs` to return both `real_size` and `full_size`, or introduce a separate helper. In `load_header.rs` and all analogous syscalls, replace:

```rust
machine.add_cycles_no_checking(transferred_byte_cycles(len))?;
```

with a charge based on the serialized size of the fetched object regardless of how many bytes the script requested. Alternatively, add a fixed per-syscall base cost that covers the DB lookup independently of the transfer size. The partial mitigation already present in `load_cell_data.rs` (early return on `size==0` without charging) should also be revisited — it avoids the write but still performs the DB read.

## Proof of Concept
Construct a transaction with one `header_dep`. In the lock/type script, execute the following RISC-V loop:

```asm
    la   a1, size_word      # size_word initialized to 0
    li   a7, 2072           # LOAD_HEADER_SYSCALL_NUMBER
    li   a3, 0              # index 0
    li   a4, 4              # Source::HeaderDep
    la   a0, buf
    li   a2, 0              # offset = 0
loop:
    ecall                   # DB read, 0 data-transfer cycles charged
    sw   zero, 0(a1)        # reset size_word to 0
    addi t0, t0, 1
    blt  t0, 10000, loop
```

Each of the 10,000 iterations triggers one `ChainStore::get_block_header` RocksDB read and charges 0 data-transfer cycles. Instrument RocksDB read counters before and after script verification to confirm 10,000 reads at ~0 metered data cost. Scale the loop count to the full 70M-cycle budget to demonstrate the full amplification ratio (~8.75M reads vs. the expected ~1.17M).
