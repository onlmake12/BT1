Audit Report

## Title
Unbounded `runtime::spawn` of NAT Traversal Tasks via `ConnectionSync` with Distinct `item_id`s — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

## Summary
An unprivileged connected peer can exhaust the async task pool, CPU, and socket file descriptors by first seeding `pending_delivered` with a single `ConnectionRequest`, then flooding the victim with `ConnectionSync` messages each carrying a distinct `item_id`. Both rate limiters include `item_id` in their key, so each distinct value creates an independent bucket, bypassing the intended throttle. Every accepted `ConnectionSync` unconditionally calls `runtime::spawn`, launching a task that runs up to 24 concurrent TCP retry loops for 30 seconds each, with no bound on total concurrent tasks.

## Finding Description

**Step 1 — Seed `pending_delivered`**

The attacker sends one `ConnectionRequest` with `from = attacker_peer_id`, `to = victim_peer_id`, and 1–24 valid TCP listen addresses. In `respond_delivered()`, the victim inserts the attacker's peer ID into `pending_delivered`: [1](#0-0) 

The `HOLE_PUNCHING_INTERVAL` guard only blocks re-insertion within 2 minutes; the first insertion is unconditional. The entry persists for `TIMEOUT = 5 minutes`: [2](#0-1) 

**Step 2 — Bypass both rate limiters with distinct `item_id`s**

The outer rate limiter in `received()` is keyed by `(session_id, msg.item_id())` at 30/second: [3](#0-2) 

The `forward_rate_limiter` in `ConnectionSyncProcess::execute()` is keyed by `(from, to, item_id)` at 1/second: [4](#0-3) 

Because `item_id` (a `u32`) is part of both keys, each distinct value creates an independent bucket. With N distinct `item_id`s the attacker passes N messages/second through the `forward_rate_limiter` and up to 30×N through the outer limiter.

**Step 3 — Unconditional `runtime::spawn` per accepted message**

Once both rate limiters pass and `pending_delivered` contains the attacker's `from` peer ID, `execute()` unconditionally calls `runtime::spawn` with no cap on concurrent tasks: [5](#0-4) 

**Step 4 — Each task runs up to 24 TCP retry loops for 30 seconds**

`try_nat_traversal` loops for up to 30 seconds, creating a new TCP socket on every ~200 ms iteration: [6](#0-5) 

Up to `ADDRS_COUNT_LIMIT = 24` such futures run concurrently inside each spawned task via `select_ok`: [7](#0-6) [8](#0-7) 

## Impact Explanation

In a 30-second window with N = 1,000 distinct `item_id`s: 1,000 async tasks are spawned, each holding 24 concurrent `try_nat_traversal` futures that each create a new TCP socket every ~400 ms. This produces tens of thousands of socket operations per second, rapidly exhausting the OS file-descriptor limit (typically 65,535 on Linux), the Tokio thread/task pool, and CPU from continuous TCP retry loops — causing the node to become unresponsive or crash. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (10,001–15,000 points).

## Likelihood Explanation

The attacker requires only a standard unprivileged P2P connection to the victim. The `ConnectionRequest` used to seed `pending_delivered` is a normal protocol message requiring no special privilege, no proof-of-work, and no cryptographic material beyond a valid peer identity. Sending thousands of `ConnectionSync` messages with incrementing `item_id` values is trivially scriptable. The attack is repeatable as long as the connection is maintained and the `pending_delivered` entry has not expired.

## Recommendation

1. **Fix the rate limiter key**: Key `forward_rate_limiter` by `(from, to)` only, dropping `item_id`, so the 1/second limit applies to the entire `(from, to)` pair regardless of `item_id`. Apply the same fix to the outer `rate_limiter` key.
2. **Bound concurrent tasks per `(from, to)` pair**: Track active NAT traversal tasks per session and reject new spawns if a task is already running for that pair (e.g., using a `DashMap<(PeerId, PeerId), JoinHandle<_>>`).
3. **Global task semaphore**: Introduce a bounded `tokio::sync::Semaphore` to cap the total number of concurrent NAT traversal tasks node-wide.

## Proof of Concept

```
1. Connect to victim node as peer A (attacker_peer_id).

2. Send ConnectionRequest{
       from = attacker_peer_id,
       to   = victim_peer_id,
       listen_addrs = [<1–24 valid TCP IPv4/IPv6 addrs>]
   }.
   → victim calls respond_delivered(), inserts pending_delivered[attacker_peer_id].

3. In a tight loop for i in 0..N:
       Send ConnectionSync{
           from    = attacker_peer_id,
           to      = victim_peer_id,
           route   = [],
           item_id = i          // distinct per message
       }.
   → outer rate_limiter: new bucket per (session_id, i) → passes
   → forward_rate_limiter: new bucket per (attacker, victim, i) → passes
   → pending_delivered lookup succeeds
   → runtime::spawn fires unconditionally

4. After 30 seconds:
   - N async tasks running, each with up to 24 try_nat_traversal futures
   - Each future creates a new TCP socket every ~400 ms
   - Assert: open fd count > system ulimit → node crashes or becomes unresponsive.
```
