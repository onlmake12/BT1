Audit Report

## Title
Unbounded GetHeaders I/O Amplification via Missing Sync Protocol Rate Limiter — (File: `sync/src/synchronizer/get_headers_process.rs`)

## Summary
An unprivileged inbound peer can send repeated `GetHeaders` messages with only the genesis hash as the locator. Each message bypasses all guards, triggers up to 2,000 sequential RocksDB reads via `get_locator_response()`, and spawns an unbounded tokio task. No per-peer rate limiter exists on the Sync protocol, and the `getheaders_received()` hook intended to throttle this is a confirmed no-op stub.

## Finding Description

**`getheaders_received()` is a confirmed no-op stub:** [1](#0-0) 

**No rate limiter exists in the Synchronizer:** grep confirms zero matches for `rate_limiter`, `RateLimiter`, or `governor` in `sync/src/synchronizer/`. The Relayer, by contrast, enforces 30 req/sec per `(peer, message_item_id)`: [2](#0-1) 

**Exploit flow in `GetHeadersProcess::execute()` with a genesis-terminated locator under non-IBD conditions:**

1. Locator size check passes — `[genesis_hash]` has length 1, well under `MAX_LOCATOR_SIZE = 101`: [3](#0-2) 

2. IBD check passes under non-IBD conditions: [4](#0-3) 

3. `locate_latest_common_block()` with `[genesis_hash]`: the last element equals the genesis hash (line 1867 check passes), `find` resolves it at `index == 0` with `latest_common == Some(0)`, triggering the early return — this is cheap: [5](#0-4) 

4. `get_locator_response(0, &zero_hash)` iterates from block 1 to tip, performing up to `MAX_HEADERS_LEN = 2,000` sequential `get_block_hash` + `get_block_header` DB reads: [6](#0-5) [7](#0-6) 

5. An unbounded `async_handle().spawn()` is issued per message with no backpressure: [8](#0-7) 

**No ban is triggered:** `GetHeadersMissCommonAncestors` is only returned when `locate_latest_common_block` returns `None`, which requires an empty locator or one not ending with the genesis hash. A genesis-terminated locator always returns `Some(0)`, so the attacker is never banned: [9](#0-8) [10](#0-9) 

## Impact Explanation

Each valid `GetHeaders` message causes up to ~2,000 RocksDB reads plus one tokio task spawn against the shared `ChainDB`. With no rate limiter, a single inbound peer can sustain this at TCP line rate. On a synced mainnet node with a large chain, this translates to sustained I/O saturation degrading block processing, transaction relay, and RPC responsiveness. A coordinated multi-peer attack could crash the targeted node entirely. This matches two High-severity allowed impacts: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** and **"Vulnerabilities which could easily crash a CKB node"** (High, 10001–15000 points).

## Likelihood Explanation

The attack requires only a TCP connection to the node's P2P port (default 8115) and knowledge of the genesis hash (public, hardcoded). No PoW, no keys, no privileged access. The `// TODO:` comment in `getheaders_received()` and the complete absence of any rate-limiting infrastructure in `sync/src/synchronizer/` confirm this is an unfinished mitigation. The attack is repeatable indefinitely from a single peer without triggering any ban.

## Recommendation

1. Add a `governor::RateLimiter` to `Synchronizer` keyed by `(PeerIndex, message_item_id)`, mirroring the Relayer's existing pattern at `sync/src/relayer/mod.rs:89-99`.
2. Implement `getheaders_received()` to track per-peer request timestamps and enforce a minimum inter-request interval.
3. Consider capping `get_locator_response` output or adding a short-circuit when the requesting peer has recently been served.

## Proof of Concept

```rust
// In a unit test, call in a tight loop from a single peer:
for _ in 0..1000 {
    let locator = vec![genesis_hash.clone()]; // valid, cheap, never banned
    let msg = packed::GetHeaders::new_builder()
        .block_locator_hashes(locator)
        .hash_stop(packed::Byte32::zero())
        .build();
    GetHeadersProcess::new(msg.as_reader(), &synchronizer, peer, &nc).execute();
    // Each call: ~2000 DB reads + 1 tokio::spawn, no ban, no rate limit
}
// Observe: DB read counter grows by ~2,000,000; tokio task queue depth grows unboundedly
```
