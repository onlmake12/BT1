### Title
Unbounded `tx.gasprice` in Keeper Fee Calculation Drains Subscription Balances — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `_processFeesAndPayKeeper` function in `Scheduler.sol` computes the keeper reimbursement using the raw `tx.gasprice` with no upper-bound cap. Because any unprivileged keeper controls the gas price of their own transaction, a keeper can submit `updatePriceFeeds` with an artificially inflated gas price, draining a subscription's balance far beyond what the subscription manager anticipated.

### Finding Description
In `_processFeesAndPayKeeper`, the gas cost charged to the subscription is:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
``` [1](#0-0) 

`tx.gasprice` is entirely under the caller's control — it is the gas price the keeper chose when submitting the transaction. The `SubscriptionParams` struct contains no `maxGasPrice` field, and `_validateSubscriptionParams` imposes no gas-price constraint: [2](#0-1) [3](#0-2) 

The full `updatePriceFeeds` flow is permissionless — any address may call it: [4](#0-3) 

The keeper receives `totalKeeperFee = gasCost + keeperSpecificFee` directly from the subscription balance: [5](#0-4) 

### Impact Explanation
A subscription manager deposits native tokens into a subscription expecting fees proportional to normal network gas prices. A malicious keeper submitting `updatePriceFeeds` at 100× the prevailing gas price will extract 100× the expected gas reimbursement from the subscription balance in a single transaction. The keeper profits by `GAS_OVERHEAD * tx.gasprice` (the overhead component is pure profit above actual network cost), while the subscription manager suffers direct, unrecoverable loss of deposited funds. Permanent subscriptions (`isPermanent = true`) are especially at risk because the manager cannot update parameters to react.

### Likelihood Explanation
The entry point is fully permissionless — no registration, whitelist, or privileged role is required to call `updatePriceFeeds`. The attack requires only that the trigger conditions for the subscription are met (heartbeat elapsed or price deviation threshold crossed), which is a normal operational state. The attacker's only cost is paying the inflated gas price to the network, but they recover that cost plus the `GAS_OVERHEAD` premium from the subscription. This makes the attack economically rational whenever a subscription holds a balance large enough to cover the inflated fee.

### Recommendation
Add a `maxGasPrice` field to `SubscriptionParams` and enforce it inside `_processFeesAndPayKeeper`:

```solidity
// In SubscriptionParams:
uint256 maxGasPrice; // 0 = no cap (backward-compatible)

// In _processFeesAndPayKeeper:
uint256 effectiveGasPrice = tx.gasprice;
if (params.maxGasPrice > 0 && effectiveGasPrice > params.maxGasPrice) {
    effectiveGasPrice = params.maxGasPrice;
}
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, enforce a global protocol-level maximum gas price cap settable by the admin, similar to how `singleUpdateKeeperFeeInWei` is a global parameter.

### Proof of Concept

1. Alice creates a subscription with 1 ETH balance, heartbeat of 60 seconds, tracking 2 price feeds.
2. 60 seconds elapse — the heartbeat trigger condition is met.
3. Bob (malicious keeper) calls `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = 10_000 gwei` (≈100× the prevailing 100 gwei base fee).
4. Inside `_processFeesAndPayKeeper`, `gasCost = (gasUsed + GAS_OVERHEAD) * 10_000 gwei`. If `gasUsed + GAS_OVERHEAD ≈ 500_000`, the keeper extracts `500_000 * 10_000 gwei = 5 ETH` — but Alice only has 1 ETH, so the call reverts with `InsufficientBalance`. However, at `1_000 gwei` (10× normal), the keeper extracts `0.5 ETH` in one update instead of the expected `0.05 ETH`, draining Alice's subscription 10× faster than anticipated.
5. Bob profits by `GAS_OVERHEAD * 1_000 gwei` above his actual network cost, and Alice's subscription is drained in 2 updates instead of the expected 20.

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }
