### Title
Fixed-Size Nested Struct Array Blocks Safe Future Upgrades in UUPS-Upgradeable Contract - (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer` is a UUPS-upgradeable contract that stores a fixed-size array of a nested struct (`TrustedSignerInfo[100]`) as a direct state variable. Because Solidity lays out fixed arrays contiguously in storage, any future upgrade that adds a field to `TrustedSignerInfo` will shift every subsequent state variable, silently corrupting the contract's storage and breaking signer validation.

---

### Finding Description

`PythLazer.sol` declares its state as:

```solidity
// lazer/contracts/evm/src/PythLazer.sol
TrustedSignerInfo[100] internal trustedSigners;   // slots 0–199
uint256 public verification_fee;                   // slot 200
mapping(address => uint256) trustedSignerToExpiresAtMapping; // slot 201
```

where `TrustedSignerInfo` is:

```solidity
struct TrustedSignerInfo {
    address pubkey;    // 1 slot (20 bytes, padded)
    uint256 expiresAt; // 1 slot
}
```

Each element currently occupies **2 storage slots**, so the array consumes slots 0–199. `verification_fee` lands at slot 200 and `trustedSignerToExpiresAtMapping` at slot 201.

If a future upgrade extends `TrustedSignerInfo` with even one new field (e.g., `uint256 activatedAt`), each element grows to **3 slots**, the array expands to slots 0–299, and:

- `verification_fee` is now read from slot 300 (zero or garbage → fee becomes 0 or an attacker-controlled value if a prior write landed there).
- `trustedSignerToExpiresAtMapping` is now at slot 301 → all existing expiry records are lost, making every previously registered signer appear invalid.

The contract is explicitly designed to be upgraded (the README states *"Be careful of changing storage slots when upgrading"* and the deploy script includes an upgrade path). The `TrustedSignerInfo` struct is minimal today and is a natural candidate for extension (e.g., adding activation timestamps, signer metadata, or rotation counters).

This is the same class of issue as M-18: a fixed-size array of a value-type struct placed before other state variables in an upgradeable contract, with no storage gap or mapping indirection to absorb future struct growth.

---

### Impact Explanation

If the inner struct is extended during an upgrade:

1. **All existing trusted signers become unrecognized** — `trustedSignerToExpiresAtMapping` is read from the wrong slot, returning 0 for every address, so `isValidSigner` returns `false` for all signers.
2. **`verifyUpdate` reverts for every caller** — no signer passes validation, breaking the entire Lazer price-feed verification flow for all consumers.
3. **`verification_fee` is corrupted** — could become 0 (allowing free verification) or an arbitrary large value (DoS by making verification unaffordable).

The impact is a complete, silent breakage of the Lazer on-chain verification contract after a routine upgrade, with no on-chain warning.

---

### Likelihood Explanation

- The contract is deployed behind a UUPS proxy and is explicitly intended to be upgraded.
- `TrustedSignerInfo` contains only two fields; adding rotation metadata, activation time, or a revocation flag is a foreseeable and reasonable future change.
- The README already warns about storage slot care, indicating the team is aware upgrades will happen.
- No storage gap or mapping indirection currently protects against this.

---

### Recommendation

Replace the fixed-size nested struct array with a mapping, which allocates each value at an independent `keccak256`-derived slot and is immune to struct-size changes:

```solidity
// Instead of:
TrustedSignerInfo[100] internal trustedSigners;

// Use:
mapping(uint256 => TrustedSignerInfo) internal trustedSigners;
uint256 internal trustedSignersCount;
```

Alternatively, if a bounded array is required, add a storage gap after the array:

```solidity
TrustedSignerInfo[100] internal trustedSigners;
uint256[50] private __gap; // absorbs future TrustedSignerInfo growth
```

The mapping approach is strongly preferred because it is unconditionally safe regardless of how many fields are added to `TrustedSignerInfo`.

---

### Proof of Concept

**Current storage layout** (`TrustedSignerInfo` = 2 slots/element):

| Slot | Variable |
|------|----------|
| 0–199 | `trustedSigners[0..99]` (2 slots each) |
| 200 | `verification_fee` |
| 201 | `trustedSignerToExpiresAtMapping` |

**After upgrade adding `uint256 activatedAt` to `TrustedSignerInfo`** (3 slots/element):

| Slot | Variable read by new code | Actual content |
|------|--------------------------|----------------|
| 0–299 | `trustedSigners[0..99]` (3 slots each) | old 2-slot entries + garbage |
| 300 | `verification_fee` | 0 (was never written) |
| 301 | `trustedSignerToExpiresAtMapping` | empty mapping (was never written) |

`isValidSigner(signer)` calls `trustedSignerToExpiresAtMapping[signer]` which now reads from slot 301 — always returning 0 — so `block.timestamp < 0` is always false, and every `verifyUpdate` call reverts with `"invalid signer"`. [1](#0-0) [2](#0-1) [3](#0-2)
