### Title
Unprivileged Caller Can Steal `signature_set` Rent Lamports via `close_signature_set` — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

### Summary

`CloseSignatureSet` requires only that `sol_destination` is a `Signer`. It imposes no constraint that `sol_destination` is the original payer who funded the `signature_set` account. Any arbitrary signer can supply themselves as `sol_destination` and drain the rent-exempt lamports from any victim's `signature_set`.

### Finding Description

In `verify_signatures.rs`, the `signature_set` account is created with `payer = payer` (the caller who funds it): [1](#0-0) 

The `SignatureSet` state struct stores only `sig_verify_successes`, `message_hash`, and `guardian_set_index` — **the original payer's pubkey is never recorded**: [2](#0-1) 

In `close_signature_set.rs`, the `CloseSignatureSet` account context closes the `signature_set` to `sol_destination` with no authority check: [3](#0-2) 

The only constraints are:
- `sol_destination` is a `Signer` (line 9–10) — any arbitrary signer qualifies.
- `posted_vaa` is a valid PDA with `has_one = signature_set` (lines 13–21) — this only ensures the `signature_set` matches the one referenced by the VAA, not that the caller owns it.
- `signature_set` is closed to `sol_destination` (lines 26–29) — lamports go to whoever the attacker names. [4](#0-3) 

### Impact Explanation

Any attacker who can sign a transaction can call `close_signature_set` with:
- `sol_destination` = attacker's own pubkey
- `posted_vaa` = any existing posted VAA on-chain (publicly readable)
- `signature_set` = the `signature_set` pubkey stored in that posted VAA (also publicly readable)

The `signature_set` account is closed and all its lamports (rent-exempt deposit paid by the original payer) are transferred to the attacker. The original payer receives nothing. This is direct, unconditional theft of SOL.

### Likelihood Explanation

All inputs needed to construct the attack are publicly visible on-chain. No privileged access, leaked keys, or governance majority is required. The attacker only needs to be able to sign a transaction. Any `posted_vaa` with a non-zero `signature_set` (i.e., created via the legacy `verify_signatures` + `post_vaa` path) is a valid target.

### Recommendation

Record the original payer's pubkey inside the `SignatureSet` account at creation time (in `verify_signatures`), and add a constraint in `CloseSignatureSet` requiring `sol_destination` to match that stored payer:

```rust
// In SignatureSet state:
pub payer: Pubkey,

// In CloseSignatureSet context:
#[account(
    mut,
    close = sol_destination,
    constraint = signature_set.inner().payer == sol_destination.key()
        @ CoreBridgeError::InvalidFeeRecipient
)]
signature_set: Account<'info, AccountVariant<SignatureSet>>,
```

Alternatively, require `sol_destination` to match the `payer` field stored in the associated `posted_vaa` account, if that is already recorded there.

### Proof of Concept

1. Alice calls `verify_signatures` with her keypair as `payer`, funding a `signature_set` account with rent-exempt lamports (~0.002 SOL for a 19-guardian set).
2. Alice (or anyone) calls the legacy `post_vaa` instruction, creating a `posted_vaa` PDA that stores `signature_set = alice_sig_set_pubkey`.
3. Bob (arbitrary attacker, no prior relationship to Alice or the VAA) constructs a transaction calling `close_signature_set` with:
   - `sol_destination` = Bob's pubkey (Bob signs the transaction)
   - `posted_vaa` = the publicly known PDA for Alice's VAA
   - `signature_set` = `alice_sig_set_pubkey` (read from `posted_vaa` on-chain)
4. The instruction passes all account validation checks. Anchor's `close = sol_destination` macro transfers all lamports from `alice_sig_set` to Bob.
5. Assert: Bob's balance increased by the `signature_set` rent amount; Alice received nothing.
