Audit Report

## Title
Unbounded DB Read Amplification in `GetLastStateProof` Handler via Count-Only Guard — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The `GET_LAST_STATE_PROOF_LIMIT` guard bounds only the count of difficulty samples, not the computational cost of resolving them. Each of up to 999 difficulty entries independently triggers a full binary search over the entire block range, each step issuing two DB reads (`get_block_hash` + `get_block_ext`). An unprivileged peer can craft a valid `GetLastStateProof` message causing ~46,000 DB reads per request, with no rate limiting and no ban triggered for valid responses.

## Finding Description
The entry-point guard at lines 201–204 checks:

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT  // = 1000
``` [1](#0-0) [2](#0-1) 

With `difficulties.len()=999` and `last_n_blocks=0`, the expression evaluates to `999 ≤ 1000` and passes. Each of the 999 difficulties then drives a call to `get_first_block_total_difficulty_is_not_less_than` via `get_block_numbers_via_difficulties`: [3](#0-2) 

Each binary search step calls `get_block_total_difficulty`, which issues exactly two DB reads: [4](#0-3) 

The `start_block_number` advance optimization at lines 90–91 only shrinks the search range by 1 per iteration when the attacker targets consecutive early blocks (e.g., blocks 1, 2, 3, …, 999) on a chain of height H ≈ 10M: [5](#0-4) 

Tracing the optimization: iteration k starts at `start_block_number = k-2`, so the range is `[k-2, difficulty_boundary_block_number)` ≈ `[k-2, H)`. For k ≤ 999 and H ≈ 10M, this is effectively the full chain range, costing ⌈log₂(10M)⌉ ≈ 23 steps × 2 reads = 46 reads per difficulty. Total: 999 × 46 ≈ **45,954 DB reads per single message**.

The `current_difficulty` deduplication guard at lines 82–83 also does not help, since the attacker supplies strictly increasing difficulties.

The actual binary search range for the 999 difficulties is `[start_block_number, difficulty_boundary_block_number)`, computed at lines 335–344: [6](#0-5) 

To maximize this range, the attacker sets `difficulty_boundary = total_diff(block_{H-1})` (second-to-last block), causing `difficulty_boundary_block_number = H-1`, making the search range `[0, H-1)` ≈ `[0, H)`.

All existing validation checks pass: difficulties are monotonically increasing (line 254), the last difficulty is less than `difficulty_boundary` (line 259), and the first-difficulty predecessor check at lines 268–288 is skipped entirely when `start_block_number = 0`. [7](#0-6) 

The response is `StatusCode::OK` (200). `should_ban` only triggers for 4xx codes: [8](#0-7) 

No rate limiting, per-peer throttling, or concurrency cap exists anywhere in the light client protocol server.

## Impact Explanation
A single attacker peer can continuously send crafted `GetLastStateProof` requests, each causing ~46,000 synchronous RocksDB reads, saturating the node's I/O path and degrading performance for all other operations (block sync, tx relay, RPC). This matches the **High (10001–15000 points)** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*. The attacker sustains the attack indefinitely at zero cost beyond maintaining a P2P connection, with no ban or backpressure mechanism to stop them.

## Likelihood Explanation
The attacker requires only a valid P2P connection to a node running the light client server. The total difficulty values for the first ~999 blocks are trivially obtained by syncing the chain or querying a public explorer. No special privileges, keys, or hashpower are required. The crafted message is indistinguishable from a legitimate request and returns `OK`, so the attacker is never banned and can repeat the attack at maximum throughput.

## Recommendation
Replace the flat count limit with a cost-aware limit. The most targeted fix is to cap the total estimated binary search steps before processing: compute `difficulties.len() × ⌈log₂(difficulty_boundary_block_number - start_block_number)⌉` and reject requests exceeding a threshold (e.g., 10,000 estimated steps). Alternatively, replace the per-difficulty binary search in `get_block_numbers_via_difficulties` with a single linear scan over the block range that emits block numbers as each difficulty threshold is crossed — O(H) total but with a single pass and no repeated range traversal. A complementary defense is per-peer request rate limiting (e.g., N `GetLastStateProof` requests per peer per time window) enforced before any DB access.

## Proof of Concept
Construct a `GetLastStateProof` message with:
- `last_n_blocks = 0`
- `start_number = 0`, `start_hash = genesis hash`
- `last_hash = current tip hash` (valid main-chain block at height H)
- `difficulty_boundary = total_diff(block_{H-1})` (second-to-last block, ensuring `difficulty_boundary_block_number = H-1`)
- `difficulties = [total_diff(block_1), total_diff(block_2), …, total_diff(block_999)]`

Verification:
1. Guard check: `999 + 0×2 = 999 ≤ 1000` → passes
2. Monotonically increasing: ✓ (total difficulty is non-decreasing per block)
3. Last difficulty < boundary: ✓ (`total_diff(block_999)` << `total_diff(block_{H-1})`)
4. First difficulty > predecessor of start block: ✓ (`start_number=0`, check skipped at lines 268–270)
5. `difficulty_boundary_block_number` resolves to `H-1` via binary search over `[0, H)`
6. Each of 999 difficulties triggers `get_block_numbers_via_difficulties(0, H-1, diff_i)` → binary search over `[0, H-1)` → ~23 steps × 2 reads = 46 reads
7. Total: ~45,954 DB reads; response is `OK`; no ban; repeat indefinitely
