### Title
Peer Store Eviction Bypass via Uniform /16-Group Distribution — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

`PeerStore::check_purge` contains two sequential eviction strategies. When both fail to produce candidates, it returns `PeerStoreError::EvictionFailed`, which propagates up through `add_addr` and permanently blocks new address ingestion. An unprivileged attacker who pre-fills the store with exactly 4 connectable addresses per IPv4 /16 group across 4096 distinct /16 groups (totalling `ADDR_COUNT_LIMIT = 16384`) can deterministically trigger this failure on every subsequent `add_addr` call.

---

### Finding Description

**Step 1 — Non-connectable eviction (lines 341–355):**

`is_connectable` returns `false` only when `last_connected_at_ms == 0 && attempts_count >= ADDR_MAX_RETRIES (3)`, or when the address has timed out with ≥ 10 failures. Freshly-advertised addresses (`attempts_count=0`) are always connectable, so step 1 finds zero candidates. [1](#0-0) 

**Step 2 — Network-group eviction (lines 358–393):**

Addresses are grouped by `Group`, which for IPv4 uses only the **first two octets** (a /16 network, not /24 as the question states — but the attack works identically at the /16 level). [2](#0-1) 

The eviction loop only removes entries from groups where `addrs.len() > 4` (strictly greater). Groups of exactly 4 return `None` from `flat_map`. [3](#0-2) 

If all groups have `len == 4`, `candidate_peers` is empty and `EvictionFailed` is returned unconditionally. [4](#0-3) 

**Attack construction:**

- 4096 distinct /16 subnets × 4 addresses each = 16384 = `ADDR_COUNT_LIMIT`
- All entries are connectable (fresh, `attempts_count=0`)
- Every subsequent `add_addr` call hits `check_purge`, both steps fail, `EvictionFailed` is returned [5](#0-4) 

**Entry path:**

The discovery protocol's `add_new_addrs` calls `peer_store.add_addr()` for every address in a received `Nodes` message, with no per-session rate limit visible in the handler. The error is silently logged at `debug` level and discarded. [6](#0-5) 

**Persistence:**

The peer store is serialized to disk on shutdown and reloaded on startup, so the poisoned state survives node restarts. [7](#0-6) 

---

### Impact Explanation

Once the store is poisoned, no new peer addresses can be learned via discovery. The node is limited to its existing 16384 entries. As those peers churn or become unreachable, the node's connectivity degrades without any ability to replenish its peer set. This is a peer-discovery freeze that worsens over time.

---

### Likelihood Explanation

The attacker needs only a single P2P connection to the victim. Sending ~16384 addresses across multiple `Nodes` messages is trivially achievable. The attack requires no special privileges, no PoW, and no Sybil capability beyond one connection. The poisoned state persists across restarts.

---

### Recommendation

Replace the strict `> 4` threshold with `>= 1` (evict from any non-empty group), or guarantee at least one eviction candidate regardless of group distribution — for example, always evict the oldest-tried address from the largest group, falling back to a random address if all groups are equal-sized. Additionally, enforce a per-session cap on the number of addresses accepted from a single `Nodes` message source.

        for host in 1u8..=4 {

                subnet_a, subnet_b, host,
                PeerId::random().to_base58()

            peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();

    }
}
// Store is now at ADDR_COUNT_LIMIT with 4096 groups of exactly 4

// Any new address fails permanently

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
