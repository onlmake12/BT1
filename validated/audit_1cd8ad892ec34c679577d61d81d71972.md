### Title
`n_sync_started` Counter Permanently Inflated When Peer Disconnects Between Increment and `start_sync` Call — (`sync/src/synchronizer/mod.rs`)

### Summary

In `start_sync_headers`, the `n_sync_started` counter is incremented unconditionally before the corresponding peer state is updated via `start_sync`. If the peer disconnects in the window between the counter increment and the `get_mut` call, the counter is permanently inflated. Because all decrement paths (`suspend_sync`, `tip_synced`, `disconnected`) only decrement when `sync_started()` is true — which is only set by `start_sync` — the leaked count can never be recovered. In IBD mode this permanently blocks the node from starting header sync with any peer.

### Finding Description

`start_sync_headers` in `sync/src/synchronizer/mod.rs` first collects a snapshot of eligible peers, then for each peer atomically increments `n_sync_started`, and only afterwards attempts to update the peer's state:

```rust
// peers list collected here — snapshot, not live
let peers: Vec<PeerIndex> = self
    .peers()
    .state
    .iter()
    .filter(|kv_pair| kv_pair.value().can_start_sync(now, ibd))
    .map(|kv_pair| *kv_pair.key())
    .collect();
// ...
for peer in peers {
    if self
        .shared()
        .state()
        .n_sync_started()
        .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
            if ibd && x != 0 { None } else { Some(x + 1) }  // ← counter incremented here
        })
        .is_err()
    {
        break;
    }
    {
        if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
            peer_state.start_sync(HeadersSyncController::from_header(&tip)); // ← state set here
        }
        // if get_mut returns None, start_sync is never called
    }
``` [1](#0-0) 

If the peer disconnects between the snapshot collection and the `get_mut` call, `disconnected()` removes the peer from `self.state` first. Then `get_mut` returns `None`, `start_sync` is never called, and `sync_started()` remains `false` for that peer.

All three decrement sites guard on `sync_started()`:

```rust
pub fn disconnected(&self, peer: PeerIndex) {
    if let Some(peer_state) = self.state.remove(&peer)... {
        if peer_state.sync_started() {          // ← false → no decrement
            self.n_sync_started.fetch_sub(1, ...);
        }
    }
}
``` [2](#0-1) 

```rust
pub(crate) fn suspend_sync(&self, peer_state: &mut PeerState) {
    if peer_state.sync_started() {              // ← false → no decrement
        self.peers.n_sync_started.fetch_sub(1, ...);
    }
    peer_state.suspend_sync(SUSPEND_SYNC_TIME);
}
``` [3](#0-2) 

Because `sync_started()` checks `HeadersSyncState::Started` — which is only set by `ChainSyncState::start()` inside `PeerState::start_sync()` — and `start_sync` was never called, the counter is permanently inflated with no recovery path. [4](#0-3) 

### Impact Explanation

`n_sync_started` is the sole gate for IBD single-peer enforcement:

```rust
.fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
    if ibd && x != 0 { None } else { Some(x + 1) }
})
``` [5](#0-4) 

A permanently inflated counter (≥ 1) causes every subsequent `fetch_update` in IBD mode to return `Err`, breaking out of the loop immediately. The node stops initiating header sync with any peer and remains stuck in Initial Block Download indefinitely. There is no self-healing mechanism; the node must be restarted to recover. This is a remote denial-of-service against the sync subsystem.

### Likelihood Explanation

The race window exists between two non-atomic operations: the `fetch_update` increment and the subsequent `self.peers().state.get_mut(&peer)`. The network layer runs disconnect callbacks concurrently. An unprivileged inbound or outbound peer can trigger this by:

1. Connecting to the victim node and waiting to appear in the `can_start_sync` filter.
2. Disconnecting immediately after the `peers` snapshot is taken but before `get_mut` is reached for their entry.

The window is small (microseconds to low milliseconds) but the attacker can repeat the connect/disconnect cycle rapidly to increase probability. No special privileges, keys, or majority hashpower are required — only the ability to open and close TCP connections to the node's P2P port.

### Recommendation

Move the counter increment to occur only after `start_sync` is confirmed to have been called:

```rust
for peer in peers {
    let mut started = false;
    if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
        // Only increment if the peer still exists
        if self
            .shared()
            .state()
            .n_sync_started()
            .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
                if ibd && x != 0 { None } else { Some(x + 1) }
            })
            .is_ok()
        {
            peer_state.start_sync(HeadersSyncController::from_header(&tip));
            started = true;
        } else {
            break;
        }
    }
    if started {
        active_chain.send_getheaders_to_peer(nc, peer, tip.number_and_hash());
    }
}
```

This ensures `n_sync_started` is only incremented when `sync_started()` will subsequently return `true`, keeping the counter and the peer state consistent.

### Proof of Concept

```
1. Victim node enters IBD (is_initial_block_download() == true).
2. Attacker peer connects; node adds it to peers.state with SyncProtocolConnected state.
3. start_sync_headers() timer fires; attacker peer passes can_start_sync() filter
   and is collected into `peers` Vec.
4. Attacker peer immediately disconnects; Peers::disconnected() removes it from
   peers.state (sync_started() == false → no decrement).
5. Loop reaches attacker's PeerIndex:
   - fetch_update succeeds: n_sync_started becomes 1.
   - get_mut(&peer) returns None (peer already removed).
   - start_sync() is never called; sync_started() remains false.
6. n_sync_started == 1 permanently.
7. Next start_sync_headers() call: fetch_update sees ibd && x != 0 → returns Err
   → breaks immediately for every peer.
8. Node never initiates header sync again; IBD stalls indefinitely.
```
