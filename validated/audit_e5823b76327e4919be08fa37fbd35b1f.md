### Title
Wrapping Integer Overflow in `GetLastStateProofProcess::execute` Bypasses Limit Guard, Enabling Unbounded O(N) Work Per Request — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged remote peer can send a `GetLastStateProof` message with `last_n_blocks = 2^63` and `difficulties = []`. On a 64-bit release build, the guard expression `(last_n_blocks as usize) * 2` wraps to `0`, making the limit check evaluate to `false`. The code then falls into the "not enough blocks" branch and collects every block number from `start_block_number` to `last_block_number`, performing O(chain_height) database lookups per message with no meaningful cost to the attacker.

### Impact Explanation

A single crafted message causes the server to perform O(chain_height) disk I/O and CPU work (MMR root per block). An attacker with a persistent peer connection can flood the server with such messages at negligible cost, saturating I/O and CPU, degrading or halting block propagation and sync for all other peers — matching the stated scope of **CKB network congestion with few costs**.

---

### Likelihood Explanation

- The entry point is the standard P2P light-client protocol, reachable by any unprivileged peer.
- No PoW, key material, or privileged role is required.
- The attacker only needs to know any valid main-chain block hash (publicly observable).
- The overflow is deterministic on every 64-bit release build; it does not depend on timing or race conditions.

---

### Recommendation

Replace the unchecked multiply with a saturating or checked variant so that an overflow is treated as exceeding the limit:

```rust
// Before (vulnerable):
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT

// After (safe):
let n = (last_n_blocks as usize).saturating_mul(2);
if self.message.difficulties().len().saturating_add(n)
    > constant::GET_LAST_STATE_PROOF_LIMIT
``` [5](#0-4)

1. Start a CKB node with the light-client protocol enabled (release build).
2. Connect as a peer and send a `LightClientMessage::GetLastStateProof` with:
   - `last_hash` = current chain tip hash (any valid main-chain hash)
   - `start_hash` = genesis block hash
   - `start_number` = `0`
   - `last_n_blocks` = `0x8000000000000000`
   - `difficulty_boundary` = `U256::MAX`
   - `difficulties` = `[]`
3. Observe that the server does **not** return `MalformedProtocolMessage` (guard bypassed).
4. Observe that the server iterates over all `N` blocks and performs `N` DB lookups.
5. Repeat in a tight loop; measure CPU/IO saturation on the server.
6. Confirm fix: replacing `* 2` with `.saturating_mul(2)` causes the guard to fire and return `MalformedProtocolMessage` immediately.
