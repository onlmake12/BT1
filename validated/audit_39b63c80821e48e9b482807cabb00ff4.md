### Title
Unbounded O(N × log H) DB Read Amplification in `get_block_numbers_via_difficulties` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

An unprivileged remote peer can send a `GetLastStateProof` message with up to **1000** strictly-increasing difficulty values crafted to map to blocks near the start of a long chain. Each entry triggers a full binary search over `[prev_found − 1, chain_tip)`, costing O(log H) DB reads. The total per-request DB work is O(N × log H), not O(N), enabling sustained IO exhaustion with no rate limiting in this path.

### Finding Description

**Limit check** — the guard at line 201 is:

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // = 1000
``` [1](#0-0) [2](#0-1) 

With `last_n_blocks = 0`, the attacker may supply up to **1000** difficulty entries.

**Monotonicity check** — the server enforces *strict* monotonic increase:

```rust
if difficulties.windows(2).any(|d| d[0] >= d[1]) { … reject … }
``` [3](#0-2) 

This prevents duplicates but does **not** prevent 1000 distinct values that all map to blocks near the chain's genesis.

**Binary search per entry** — `get_block_numbers_via_difficulties` calls `get_first_block_total_difficulty_is_not_less_than` for every difficulty: [4](#0-3) 

After finding block `num`, the next search starts from `num − 1`:

```rust
if num > start_block_number {
    start_block_number = num - 1;
}
``` [5](#0-4) 

If the attacker maps all 1000 difficulties to blocks 1, 2, 3, …, 1000 on a 1 M-block chain, each subsequent search starts from `i − 1` and ends at 1 M, covering nearly the full range. Each binary search costs O(log H) iterations.

**DB reads per binary search step** — each iteration calls `get_block_total_difficulty`, which performs two DB lookups (`get_block_hash` + `get_block_ext`): [6](#0-5) 

**The "skip" guard does not help** — the `current_difficulty >= *difficulty` skip at line 82 only fires when the previously found block's total difficulty already exceeds the next requested difficulty. An attacker who sets `d_{i+1} = D_i + 1` (where `D_i` is the exact total difficulty of block `i`) ensures this never triggers. [7](#0-6) 

### Impact Explanation

On a 1 M-block chain (log₂(1 M) ≈ 20):

| Parameter | Value |
|---|---|
| Max difficulties per request | 1000 |
| Binary search depth | ~20 |
| DB reads per search step | 2 |
| **Total DB reads per request** | **~40 000** |

A single persistent peer sending back-to-back crafted `GetLastStateProof` messages can saturate the node's storage IO, degrading block sync and validation for all other peers.

### Likelihood Explanation

- No authentication or rate limiting is applied to this message type in the visible code path.
- All required information (block total difficulties) is publicly readable from the chain.
- The attack is trivially scriptable: read difficulties at blocks 1…1000, increment each by 1, send the request in a loop.

### Recommendation

1. **Cap binary search work, not just entry count.** Track the total number of `get_block_total_difficulty` calls across the entire `get_block_numbers_via_difficulties` invocation and abort once a constant budget (e.g., `2 × GET_LAST_STATE_PROOF_LIMIT`) is exceeded.
2. **Tighten the limit formula.** The current check `difficulties.len() + last_n_blocks * 2 ≤ 1000` allows `difficulties.len() = 1000` when `last_n_blocks = 0`. Enforce a separate, lower cap on `difficulties.len()` alone.
3. **Add per-peer request rate limiting** for `GetLastStateProof` messages.

### Proof of Concept

```
chain_height = 1_000_000
# Read total_difficulty at blocks 1..1000 from the chain
difficulties = [total_difficulty(i) for i in range(1, 1001)]
# Each d[i] is strictly greater than d[i-1] (valid chain property)

msg = GetLastStateProof {
    last_n_blocks: 0,
    difficulties: difficulties,   # len = 1000, passes the limit check
    start_number: 0,
    difficulty_boundary: total_difficulty(1_000_000),
    …
}

# Server performs:
#   for each of 1000 difficulties → binary_search([i-1, 1_000_000))
#   each binary search: ~20 iterations × 2 DB reads = 40 reads
#   total: 1000 × 40 = 40_000 DB reads per single request
```
