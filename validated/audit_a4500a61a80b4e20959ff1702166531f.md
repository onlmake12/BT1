### Title
Unbounded `pending_delivered` HashMap Growth via Spoofed `from_peer_id` in `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

The `HolePunching` protocol's `pending_delivered` HashMap has no size cap. An attacker who is directly connected to a victim node can send `ConnectionRequest` messages with `to = victim_peer_id` and a fresh, unique `from_peer_id` per message. Each message that passes validation inserts a new entry into `pending_delivered`. The only eviction is a periodic `retain` that runs every 5 minutes. The per-session rate limiter (30 req/s) limits the insertion rate but not the total size, allowing an attacker to accumulate thousands of entries per connection before any cleanup occurs.

---

### Finding Description

`HolePunching` stores a `pending_delivered: HashMap<PeerId, PendingDeliveredInfo>` where `PendingDeliveredInfo = (Vec<Multiaddr>, u64)`. [1](#0-0) 

The only deduplication guard in `respond_delivered` checks whether the **same** `from_peer_id` was seen recently: [2](#0-1) 

This guard is bypassed entirely when each message carries a distinct `from_peer_id`. After a successful `send_message_to` back to the attacker's session, the new entry is unconditionally inserted: [3](#0-2) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`. Because each message uses a unique `from`, every message gets its own fresh bucket and the limiter never fires: [4](#0-3) 

The outer rate limiter is keyed by `(session_id, msg.item_id())` and allows 30 req/s per session — it throttles the rate but does not cap the total map size: [5](#0-4) 

Cleanup only runs every `CHECK_INTERVAL = 5 minutes` and only removes entries older than `TIMEOUT = 5 minutes`: [6](#0-5) 

---

### Impact Explanation

**Memory growth per session per 5-minute window:**
- 30 req/s × 300 s = **9,000 entries**
- Each entry: `PeerId` (~39 B) + `Vec<Multiaddr>` (up to 24 × ~50 B = 1,200 B) + `u64` (8 B) + HashMap overhead (~50 B) ≈ **~1,300 B**
- Per session: **~11.7 MB** before the first cleanup tick

With `N` simultaneous attacker connections the map grows to `N × 11.7 MB`. A node with a typical inbound connection limit of 125 peers could accumulate **~1.46 GB** of heap memory in a single 5-minute window, causing OOM on resource-constrained nodes.

---

### Likelihood Explanation

- HolePunching must be enabled (it is a production feature, not gated by a flag in the code).
- The attacker only needs a standard P2P connection — no authentication, no PoW, no privileged role.
- The attacker must supply at least one valid TCP/IPv4 or IPv6 `listen_addr` (trivially satisfied with any routable address). [7](#0-6) 

- The `send_message_to` back to the attacker's own session will succeed as long as the session is alive, which the attacker controls.

---

### Recommendation

1. **Add a hard size cap** on `pending_delivered` (e.g., 1,024 entries). Reject or evict LRU entries when the cap is reached.
2. **Key the `forward_rate_limiter` on the sending session** (or on `(session_id, to)`) rather than on `(from, to, msg_item_id)`, so that a single session cannot bypass it by rotating `from_peer_id`.
3. **Validate that `from_peer_id` matches the actual connected peer** (i.e., the session's authenticated peer ID), preventing spoofed `from` values entirely.

---
