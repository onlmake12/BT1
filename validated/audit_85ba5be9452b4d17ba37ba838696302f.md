Audit Report

## Title
Decompression Size Check Uses Global Constant Instead of Per-Protocol Frame Limit, Allowing Memory Amplification — (`network/src/compress.rs`)

## Summary

In `LengthDelimitedCodecWithCompress::decode`, the decompressed-size guard at line 235 compares against the module-level constant `MAX_UNCOMPRESSED_LEN = 8MB` rather than `self.length_delimited.max_frame_length()`. For any protocol whose wire limit is below 8MB (RelayV3 at 4MB, Sync/LightClient/Filter at 2MB, Discovery at 512KB, Alert at 128KB), an attacker can send a legitimately compressed frame that decompresses to 8MB — far exceeding the protocol's intended ceiling — while keeping the session alive for continuous repetition.

## Finding Description

`MAX_UNCOMPRESSED_LEN` is defined as a module-level constant of 8MB: [1](#0-0) 

Per-protocol wire limits are defined in `SupportProtocols::max_frame_length()`, e.g. RelayV3 at 4MB, Sync at 2MB, Discovery at 512KB: [2](#0-1) 

In `LengthDelimitedCodecWithCompress::decode`, after `self.length_delimited.decode(src)` enforces the wire limit, the decompressed-size check uses the global 8MB constant rather than the codec's own `max_frame_length`: [3](#0-2) 

By contrast, the encoder's `process` method correctly consults `self.length_delimited.max_frame_length()`: [4](#0-3) 

`CKBProtocol::build()` wires the per-protocol `max_frame_length` into the inner `LengthDelimitedCodec`, but the decoder never reads it back: [5](#0-4) 

All `CKBProtocol::new_with_support_protocol` instances set `compress: true` by default: [6](#0-5) 

**Exploit path (RelayV3, 4MB wire limit):**
1. Attacker connects as a RelayV3 peer.
2. Attacker compresses 8MB of zeros with snappy → ~8KB compressed payload.
3. Attacker prepends `COMPRESS_FLAG` byte and wraps in a length-delimited frame (~8KB total, well under the 4MB wire limit).
4. `self.length_delimited.decode(src)` accepts the frame (8KB ≤ 4MB). ✓
5. `decompress_len(&data[1..])` returns 8,388,608. ✓
6. `8_388_608 > MAX_UNCOMPRESSED_LEN` (8MB) → false → check passes. ✓
7. `BytesMut::zeroed(8_388_608)` allocates 8MB.
8. `SnapDecoder::new().decompress(...)` succeeds; 8MB buffer returned to handler.
9. Session stays alive; attacker repeats continuously.

The same path applies to Sync (4× amplification), Discovery (16×), and Alert (64×) — any protocol whose wire limit is below 8MB and whose compressed payload fits within that wire limit. The Ping (1KB) case requires a malformed snappy header with a false claimed decompressed size; decompression then fails and the connection closes, limiting that variant to one allocation per connection attempt rather than continuous exploitation.

## Impact Explanation

This matches **High: Vulnerabilities which could easily crash a CKB node**. An attacker with only an established P2P session can continuously stream compressed frames that cause allocations far exceeding the protocol's intended limit. For RelayV3, each ~8KB wire frame causes an 8MB allocation — a 2× amplification that can be sustained indefinitely on a single session. Across many concurrent peers (each streaming at maximum rate), this drives unbounded memory growth and crashes the node. Protocols with smaller wire limits (Discovery at 512KB, Alert at 128KB) exhibit even higher amplification ratios (16× and 64× respectively) for valid compressed payloads.

## Likelihood Explanation

- Requires only an established P2P session — no privilege, no PoW, no key material.
- Compression is enabled by default for all `CKBProtocol` instances via `compress: true`.
- The snappy frame format is public; constructing a valid compressed payload (e.g., snappy-compressed zero-filled buffer) is trivial.
- The session remains alive after successful decompression, allowing continuous exploitation without reconnection.
- No rate limiting or per-session memory accounting exists to bound the damage.

## Recommendation

In `LengthDelimitedCodecWithCompress::decode`, replace the check against the global `MAX_UNCOMPRESSED_LEN` with a check against `self.length_delimited.max_frame_length()`:

```rust
// current (line 235):
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {

// fixed:
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
```

This enforces the invariant that decompressed output never exceeds the per-protocol wire limit, eliminating the amplification uniformly across all protocols.

## Proof of Concept

**Unit test (RelayV3, 2× amplification, continuous):**

```rust
#[test]
fn test_decompress_amplification_relay_v3() {
    use tokio_util::codec::{Decoder, Encoder};
    use p2p::bytes::{Bytes, BytesMut};
    use snap::raw::Encoder as SnapEncoder;

    let max_frame = 4 * 1024 * 1024; // RelayV3: 4MB wire limit
    let mut codec = LengthDelimitedCodecWithCompress::new(
        true,
        tokio_util::codec::length_delimited::Builder::new()
            .max_frame_length(max_frame)
            .new_codec(),
        101.into(), // RelayV3 protocol id
    );

    // Compress 8MB of zeros → ~8KB compressed
    let plaintext = vec![0u8; 8 * 1024 * 1024];
    let compressed = SnapEncoder::new().compress_vec(&plaintext).unwrap();
    assert!(compressed.len() < max_frame, "compressed fits within wire limit");

    // Build wire frame: COMPRESS_FLAG + compressed payload, length-prefixed
    let mut frame = BytesMut::new();
    frame.extend_from_slice(&(compressed.len() as u32 + 1).to_be_bytes());
    frame.put_u8(0x80); // COMPRESS_FLAG
    frame.extend_from_slice(&compressed);

    let result = codec.decode(&mut frame).unwrap().unwrap();
    // Bug: result.len() == 8MB > max_frame_length (4MB) — invariant violated
    assert!(result.len() > max_frame,
        "decompressed {} bytes, exceeding wire limit {}", result.len(), max_frame);
}
```

Run this test against the current code to confirm it passes (demonstrating the bug). After applying the fix, the test should fail at `decode()` returning `Err(InvalidData)` instead.
