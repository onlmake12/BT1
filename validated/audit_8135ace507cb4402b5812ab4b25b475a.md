### Title
Excess Fee Permanently Locked in `PriceInfoObject` Due to Missing Refund Logic — (`target_chains/sui/contracts/sources/pyth.move`)

---

### Summary

`update_single_price_feed` enforces only a **minimum** fee check (`<=`), then deposits the **entire** passed coin into `PriceInfoObject`. No change/refund is returned. Any value above `base_update_fee` is permanently locked with no recovery path — the fee-recipient mechanism is explicitly deprecated and no withdrawal function exists anywhere in the Sui contracts.

---

### Finding Description

In `update_single_price_feed`, the fee guard is:

```move
assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);
``` [1](#0-0) 

Immediately after, the **entire** coin is deposited without splitting:

```move
price_info::deposit_fee_coins(price_info_object, fee);
``` [2](#0-1) 

`deposit_fee_coins` unconditionally joins the full coin value into the `PriceInfoObject`'s dynamic fee field:

```move
coin::join(current_fee, fee_coins);
``` [3](#0-2) 

A grep across all Sui contract sources for `withdraw`, `refund`, `fee_recipient`, and `collect_fee` confirms **zero withdrawal functions** exist in the production Sui contracts. The `set_fee_recipient` governance module explicitly states:

> "The previous version of the contract sent the fees to a recipient address but this state is not used anymore. This module is kept for backward compatibility." [4](#0-3) 

The `State` struct retains a `fee_recipient_address` field but no code path ever transfers fees to it. [5](#0-4) 

---

### Impact Explanation

Any caller who passes a `Coin<SUI>` with value `N * base_update_fee` (N > 1) loses `(N-1) * base_update_fee` SUI permanently. The excess is locked inside the `PriceInfoObject` shared object as a dynamic field under the private `FEE_STORAGE_KEY` constant — inaccessible to any external module. Because `PriceInfoObject` is a shared object, it cannot be transferred. Because `FEE_STORAGE_KEY` is a private constant, no external module can construct the key to access the field. The funds are irrecoverable.

Additionally, even the **exact** fee paid per update is permanently frozen — the protocol itself has no way to collect any fees from `PriceInfoObject` since the fee-recipient mechanism is deprecated and no replacement exists.

---

### Likelihood Explanation

The official SDK correctly pre-splits coins to exactly `base_update_fee`:

```typescript
const coins = tx.splitCoins(
  tx.gas,
  feedIds.map(() => tx.pure.u64(baseUpdateFee)),
);
``` [6](#0-5) 

However, any caller constructing transactions directly (e.g., via PTBs, custom integrations, or scripts) who passes an unsplit coin triggers the issue. The contract provides no protection. The likelihood of accidental overpayment is moderate; the likelihood of the broader "all fees are permanently frozen" issue is certain and already in effect on every deployed instance.

---

### Recommendation

1. **Enforce exact fee**: Replace `<=` with `==` in the fee assertion, or split the coin inside the function and return the remainder:
   ```move
   let exact_fee = coin::split(&mut fee, state::get_base_update_fee(pyth_state), ctx);
   price_info::deposit_fee_coins(price_info_object, exact_fee);
   transfer::public_transfer(fee, tx_context::sender(ctx)); // refund excess
   ```
2. **Add a fee withdrawal function**: Implement a privileged `withdraw_fee_coins` function callable by the fee recipient to drain accumulated fees from `PriceInfoObject`.

---

### Proof of Concept

```move
#[test]
fun test_overpayment_locks_excess() {
    // setup with base_update_fee = 1
    let fee_coins = coin::mint_for_testing<SUI>(10, ctx); // 10x the required fee
    let vec = update_single_price_feed(&pyth_state, price_updates, &mut pio, fee_coins, &clock);
    // Assert: entire 10 units locked, not just 1
    assert!(price_info::get_balance(&pio) == 10, 0);
    // Assert: no coin returned to caller, no refund event emitted
    hot_potato_vector::destroy(vec);
}
```

The test at line 817 already implicitly confirms the mechanic — it asserts `get_balance == DEFAULT_BASE_UPDATE_FEE` only because the test itself splits the coin to exactly that amount first. Passing an unsplit coin would show the full excess absorbed. [7](#0-6)

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);

                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
