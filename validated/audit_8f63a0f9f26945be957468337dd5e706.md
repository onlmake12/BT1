Audit Report

## Title
`remove_expired` Removes Parent Transactions Without Evicting Dependent Descendants, Leaving Zombie Entries in the Pool - (File: tx-pool/src/pool.rs)

## Summary
`TxPool::remove_expired` calls `pool_map.remove_entry` for each expired transaction, which removes only the individual entry without cascading to child transactions. Children submitted after the parent remain in the pool in a permanently uncommittable state, holding pool size/cycle budget and locking their input outpoints in `edges.inputs`, contradicting the function's own comment ("and their dependencies") and every other eviction path in the codebase.

## Finding Description
In `tx-pool/src/pool.rs` at L270–288, `remove_expired` collects entries satisfying `self.expiry + entry.inner.timestamp < now_ms`, then calls `self.pool_map.remove_entry(&entry.proposal_short_id())` for each one. [1](#0-0) 

`remove_entry` at `pool_map.rs` L235–250 removes only the single entry: it calls `remove_entry_edges` (releases the parent's own inputs from `edges.inputs`) and `remove_entry_links` (severs parent↔child links in `links`), but does not touch child transactions. [2](#0-1) 

`remove_entry_edges` at L642–653 only removes the expired entry's own inputs — the child's inputs (referencing the now-gone parent's output outpoints) remain registered in `edges.inputs` because the child is still in the pool. [3](#0-2) 

After parent eviction, child transactions:
1. Remain in `entries` with `Status::Pending/Gap/Proposed`.
2. Retain their input outpoints in `edges.inputs`, pointing to the child's `ProposalShortId`.
3. Have their `links.parents` severed, so `calc_ancestors` returns an empty set — the pool treats them as root transactions.
4. Can never be committed: the parent's outputs are neither in the pool nor on-chain.
5. Will not be re-evicted until their own timestamps also cross the expiry threshold.

The `test_remove_entry` test in `score_key.rs` L157–167 explicitly confirms this: after `map.remove_entry(&tx1_id)`, both `tx2` and `tx3` remain in the pool. [4](#0-3) 

The correct function, `remove_entry_and_descendants` at L252–265, first collects all descendants via `calc_descendants`, then removes all of them — this is what every other eviction path uses. [5](#0-4) 

All other eviction paths use `remove_entry_and_descendants`:
- `limit_size` → L307
- `remove_tx` → L359
- `remove_by_detached_proposal` → L343
- `resolve_conflict` → L310, L321
- `resolve_conflict_header_dep` → L285

`remove_expired` is the sole eviction path using the single-entry variant. [6](#0-5) [7](#0-6) 

**Double-spend blocking:** `edges.insert_input` at `edges.rs` L33–54 returns `Err(Reject::RBFRejected(...))` if an outpoint is already occupied. Since zombie children still hold their input outpoints in `edges.inputs`, any new transaction attempting to spend the same outpoints is rejected as a double-spend (or must satisfy RBF rules), even though the parent was never mined and is no longer in the pool. [8](#0-7) 

**Exploit flow:**
1. Attacker submits parent `P` at time `T` with minimum fee.
2. Attacker submits `N` children `C₁…Cₙ` spending `P`'s outputs at time `T + δ` (where `δ` approaches `expiry_hours`).
3. At `T + expiry`, `remove_expired` removes `P` but leaves `C₁…Cₙ` in the pool.
4. `C₁…Cₙ` remain as zombies until `T + expiry + δ`, consuming `total_tx_size` and `total_tx_cycles` budget.
5. Any new transaction attempting to spend the same outpoints as `C₁…Cₙ` is rejected with `RBFRejected`, blocking legitimate users.
6. Repeating this pattern fills a significant fraction of the pool with permanently unspendable transactions.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Zombie entries consume `pool_map.total_tx_size` and `total_tx_cycles` budget, reducing available pool capacity for legitimate transactions. The double-spend rejection on zombie-held outpoints further blocks legitimate users from spending those outputs. With a default `expiry_hours` of 12 hours and `δ` approaching that value, an attacker can maintain persistent pool congestion across repeated cycles at the cost of minimum transaction fees.

## Likelihood Explanation
The attack requires only the ability to submit transactions — available to any unprivileged peer or RPC caller. No special privileges, leaked keys, or victim mistakes are required. The cost is minimum fees for parent and child transactions. The 12-hour default expiry window gives a long zombie persistence period per cycle. The attack is repeatable and automatable.

## Recommendation
Replace `remove_entry` with `remove_entry_and_descendants` inside `remove_expired`, consistent with every other eviction path:

```rust
// tx-pool/src/pool.rs  remove_expired()
for entry in removed {
    let tx_hash = entry.transaction().hash();
    debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
-   self.pool_map.remove_entry(&entry.proposal_short_id());
+   let evicted = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
    let reject = Reject::Expiry(entry.timestamp);
-   callbacks.call_reject(self, &entry, reject);
+   for e in evicted {
+       callbacks.call_reject(self, &e, reject.clone());
+   }
}
```

This aligns the implementation with the existing comment ("and their dependencies") and with the behaviour of all other eviction paths.

## Proof of Concept
A unit test analogous to `test_remove_entry` in `score_key.rs` can confirm the zombie state:
1. Create a `PoolMap` and add parent `tx1` and child `tx2` (spending `tx1`'s output).
2. Call `pool_map.remove_entry(&tx1_id)` (simulating `remove_expired`).
3. Assert `tx2` still exists in `entries` (`map.contains_key(&tx2_id)` is `true`).
4. Assert `tx2`'s input outpoint is still registered in `edges.inputs` (`map.inputs_len() > 0`).
5. Attempt to insert a new transaction spending the same outpoint as `tx2` — assert it is rejected with `RBFRejected`.

Manual steps:
1. Submit parent `P` with minimum fee at time `T`.
2. Submit child `C` spending an output of `P` at time `T + δ` (e.g., `δ` = 11 hours with `expiry_hours` = 12).
3. Advance the node clock past `T + expiry_hours` but before `T + expiry_hours + δ`.
4. Observe via `tx_pool_info` that `C` remains in the pending pool even though `P` is gone.
5. Attempt to submit a new transaction `C'` spending the same output of `P` as `C` — it is rejected with a double-spend error.

```rust
    // Expire all transaction (and their dependencies) in the pool.
    pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }
```

```rust
    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
```
