### Title
Unbounded Gas Forwarding in Legacy `revealWithCallback` Path Allows Malicious Entropy Requesters to Grief Providers — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When an Entropy provider has `defaultGasLimit == 0`, the `revealWithCallback` function invokes `_entropyCallback` on the requester contract with **no gas limit**. A malicious requester can deploy a contract whose callback consumes all available gas, forcing the provider (e.g., Fortuna) to pay far more in gas costs than the fixed fee the user paid — a direct analog to the M-02 gas-griefing pattern.

---

### Finding Description

`Entropy.sol` contains two distinct execution paths inside `revealWithCallback`:

**New path** (`req.gasLimit10k != 0`): uses `excessivelySafeCall` with an explicit upper-bound gas limit derived from the stored `gasLimit10k` field. [1](#0-0) 

**Legacy path** (`req.gasLimit10k == 0`): calls `_entropyCallback` directly with **no gas stipend at all**. [2](#0-1) 

The `gasLimit10k` field is set to `0` unconditionally whenever the provider's `defaultGasLimit` is `0`: [3](#0-2) 

This means **every request** made to a provider with `defaultGasLimit == 0` — regardless of any `gasLimit` argument the user passes — will use the legacy path. The test suite explicitly confirms this behavior:

> "A provider that hasn't upgraded to the callback failure flow can never cause a callback to fail because it runs out of gas."

The Fortuna service syncs `defaultGasLimit` from `chain_config.gas_limit`: [4](#0-3) 

If `chain_config.gas_limit` is `0` for any deployed chain, Fortuna operates in the legacy path on that chain. Additionally, any third-party provider that has not called `setDefaultGasLimit` starts with `defaultGasLimit == 0` by default (zero-initialized storage).

---

### Impact Explanation

A malicious user deploys a requester contract whose `_entropyCallback` performs expensive-but-non-reverting computation (e.g., a tight loop writing to storage). When the provider calls `revealWithCallback`, the callback consumes all remaining gas in the transaction. The user paid only the fixed base fee (`provider.feeInWei`), while the provider bears the full gas cost of the transaction. With no cap, the callback can consume an entire block's worth of gas (~30M on Ethereum mainnet). Repeated across many requests, this drains the provider's ETH balance far beyond what fees cover, potentially making the provider unable to fulfill legitimate requests.

---

### Likelihood Explanation

The attack is reachable by any unprivileged user who:
1. Identifies a provider with `defaultGasLimit == 0` (the default for any newly registered provider, and any provider that has not explicitly called `setDefaultGasLimit`).
2. Deploys a malicious consumer contract.
3. Calls `requestWithCallback` or `requestV2` paying only the base fee.

The Fortuna service is configured to fulfill **all** pending requests automatically, making it a natural target. The `gas_limit` field in Fortuna's chain config is optional and may be absent (defaulting to `0`) for some deployments.

---

### Recommendation

Apply a reasonable gas cap in the legacy path as well. The simplest fix is to forward a provider-configurable or protocol-defined maximum gas limit even when `gasLimit10k == 0`, rather than forwarding all remaining gas:

```solidity
// Legacy path: apply a safe default cap instead of forwarding unlimited gas
uint256 legacyGasLimit = 500_000; // or a protocol constant
(bool success, ) = callAddress.call{gas: legacyGasLimit}(
    abi.encodeWithSelector(
        IEntropyConsumer._entropyCallback.selector,
        sequenceNumber,
        provider,
        randomNumber
    )
);
```

Alternatively, require all providers to set a non-zero `defaultGasLimit` before accepting callback requests, eliminating the legacy path entirely.

---

### Proof of Concept

```solidity
// Malicious requester contract
contract GasGriefConsumer is IEntropyConsumer {
    IEntropy entropy;
    constructor(address _entropy) { entropy = IEntropy(_entropy); }

    function getEntropy() internal view override returns (address) { return address(entropy); }

    // Step 1: request entropy from a provider with defaultGasLimit == 0
    function attack(address provider, bytes32 userNum) external payable {
        entropy.requestWithCallback{value: msg.value}(provider, userNum);
    }

    // Step 2: when provider calls revealWithCallback, consume all gas
    function entropyCallback(uint64, address, bytes32) internal override {
        // Burn all forwarded gas without reverting
        uint256 i = 0;
        while (gasleft() > 5000) {
            // expensive SSTORE loop
            assembly { sstore(i, i) }
            unchecked { i++; }
        }
    }
}
```

1. Deploy `GasGriefConsumer` pointing at the Entropy contract.
2. Call `attack(providerWithZeroGasLimit, userNum)` with the base fee.
3. Fortuna detects the pending request and calls `revealWithCallback`.
4. `entropyCallback` consumes all remaining gas; Fortuna pays the full gas cost.
5. Repeat to drain Fortuna's ETH balance.
