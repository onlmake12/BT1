Audit Report

## Title
Wrong Row Variable in `script_exists_in_output` Causes Incorrect Script Deletion on PostgreSQL During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary
In `script_exists_in_output`, the second `match` block at line 252 reads `row_lock` instead of `row_type`. On PostgreSQL, this causes the type-script existence check to always return the already-known-false lock-script result, so every script referenced only as a `type_script_id` in surviving outputs is unconditionally deleted from the `script` table during `rollback_block`. SQLite is unaffected because its integer type causes `try_get::<bool, _>` to fail, falling through to the correct `row_type` branch.

## Finding Description
`script_exists_in_output` executes two `EXISTS` queries: `row_lock` (checks `lock_script_id = $1`) and `row_type` (checks `type_script_id = $1`). After the first match block at lines 222–235 short-circuits with `return Ok(true)` if the lock query is positive, execution reaches line 252 only when `row_lock` is already `false`. The second match block at line 252 mistakenly calls `row_lock.try_get::<bool, _>(0)` again instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, `EXISTS` returns `BOOLEAN`, so `try_get::<bool, _>(0)` succeeds and returns `Ok(false)` — the already-known lock result — without ever consulting `row_type`. The `rollback_block` caller at lines 33–37 then pushes the `type_script_id` into `script_id_list_to_remove` and deletes it from the `script` table, even when other surviving `output` rows still hold a foreign key `type_script_id` pointing to it. This leaves dangling `type_script_id` references in the `output` table, causing any subsequent indexer query that JOINs `output` with `script` on `type_script_id` to silently return `NULL` for type-script fields. ## Impact Explanation
This is a data-integrity corruption in the CKB rich-indexer state storage mechanism. The `script` table is the authoritative store for script data used by all indexer queries. Incorrect deletion of still-referenced scripts produces permanently inconsistent indexer state after any qualifying reorg, causing all consumers of the rich-indexer API to receive silently wrong results for affected outputs. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**, as the indexer's rollback logic incorrectly mutates persistent state, breaking the invariant that `output.type_script_id` always references a valid `script` row.

## Likelihood Explanation
The bug fires on every PostgreSQL-backed `rollback_block` call where any rolled-back output had a type script still referenced by a surviving output. Chain reorgs are a normal, unauthenticated network event. An unprivileged peer can also deliberately trigger reorgs by relaying valid competing-chain blocks over P2P — a standard, zero-cost operation. PostgreSQL is an explicitly supported production backend with dedicated DDL files. No special privileges or victim mistakes are required.

## Recommendation
Change line 252 from `row_lock` to `row_type`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // ← fix: was row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

## Proof of Concept
1. Start a CKB node with the rich-indexer backed by PostgreSQL.
2. Index a block B1 containing output A (type script S, lock L1) and a pre-existing output C (type script S, lock L3) indexed in an earlier non-rolled-back block.
3. Trigger rollback of block B1 (e.g., feed a longer competing chain).
4. Query `SELECT id FROM script WHERE id = <S_id>` — the row for S is deleted despite output C still holding `type_script_id = S_id`.
5. Query `get_cells` filtering by script S — type-script fields for output C are returned as `NULL`, confirming the inconsistency.
