### Title
DAO Withdrawal `header_dep_index` Interpreted as `u64` by Rust but as `u8` by C VM, Causing Node-Level Rejection of Valid Withdrawals — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw` reads the DAO withdrawal `header_dep_index` from the witness as a full `u64`, while the on-chain C VM DAO script reads only the **lowest byte** of that same 8-byte field. When an attacker (or any user) places the deposit-block hash at `header_deps[N & 0xFF]` and a different block hash at `header_deps[N]` for any `N > 255`, the Rust node and the C VM resolve to **different block headers** from the same witness value. The Rust node then rejects the transaction (or the block containing it) even though the C VM script execution succeeds.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` decodes the deposit-block index from the witness `input_type` field as a full 64-bit integer:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// …
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // uses full u64
```

The C VM DAO script, however, reads only the **lowest byte** of the same 8-byte value (treating it as a `u8` index). The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this split:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
header_deps[1]   = deposit_block.hash();
header_deps[257] = withdraw_block.hash();
// witness input_type = 257  (0x0000000000000101)
```

With `header_dep_index = 257`:
- **C VM** reads lowest byte → `1` → `header_deps[1]` = deposit block (number 100) → matches cell data → **ACCEPTS**
- **Rust** reads full u64 → `257` → `header_deps[257]` = withdraw block (number 200) → block-number check `200 ≠ 100` → **REJECTS**

The test asserts `result.is_err()`, confirming the Rust code rejects what the C VM would accept.

`DaoCalculator::transaction_fee` is called in two consensus-critical paths:

1. **Tx-pool admission** — `check_tx_fee` in `tx-pool/src/util.rs` maps any `DaoError` to `Reject::Malformed`, blocking the transaction from entering the pool.
2. **Block verification** — `FeeCalculator::transaction_fee` inside `ContextualTransactionVerifier::verify()` in `verification/src/transaction_verifier.rs` is called after C VM script execution succeeds; a `DaoError` here causes the entire block to be rejected.

Additionally, `DaoHeaderVerifier` in `verification/contextual/src/contextual_block_verifier.rs` calls `DaoCalculator::dao_field`, which internally calls `transaction_maximum_withdraw` (via `withdrawed_interests`). An incorrect deposit-header resolution also corrupts the computed DAO accumulation-rate field, causing `dao != self.header.dao()` and a `BlockErrorKind::InvalidDAO` rejection.

---

### Impact Explanation

An unprivileged transaction sender can craft a DAO withdrawal with ≥ 258 distinct `header_deps` and set `input_type = N` where `N > 255` and `N & 0xFF` points to the real deposit block while `N` points to a different block. The C VM script execution succeeds (valid withdrawal), but the Rust node rejects the transaction at the tx-pool level and, if a miner bypasses the pool, rejects the containing block at verification time. This creates:

- **Denial-of-service on DAO withdrawals**: any withdrawal crafted with `header_dep_index > 255` is permanently unspendable through a standard Rust node.
- **Block invalidation**: a miner who includes such a transaction produces a block that all Rust nodes reject, wasting proof-of-work.
- **DAO field corruption**: the block-level DAO accumulation-rate field is computed using the wrong deposit header, causing `InvalidDAO` rejection independently of the per-transaction fee check.

---

### Likelihood Explanation

The attack requires constructing a transaction with ≥ 258 distinct canonical block hashes in `header_deps` (no duplicates are allowed per `DuplicateDepsVerifier`) and setting `input_type` to an index whose low byte differs from the full value. This is unusual for legitimate users but trivially constructable by an attacker. The `header_deps` list has no enforced upper-bound count beyond the block-size limit, so 258 entries are feasible. The attacker does not need any privileged role — only the ability to submit a transaction or convince a miner to include one.

---

### Recommendation

Align the Rust `DaoCalculator` with the C VM DAO script's index interpretation. If the C VM reads only the lowest byte, truncate the index in Rust before use:

```rust
// In util/dao/src/lib.rs, transaction_maximum_withdraw:
let raw_index = LittleEndian::read_u64(&header_deps_index_data.unwrap());
// Match C VM behavior: use only the lowest byte as the array index.
let header_dep_index = (raw_index & 0xFF) as usize;
```

Alternatively, update the C VM DAO script to read the full `u64` and add a consensus rule rejecting transactions where `header_dep_index ≥ header_deps.len()` or `header_dep_index > 255` to close the ambiguity window entirely.

---

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
