### Title
Accumulated SUI Update Fees Are Permanently Frozen in `PriceInfoObject` with No Withdrawal Path â€” (`target_chains/sui/contracts/sources/price_info.move`, `pyth.move`)

### Summary
Every call to `update_single_price_feed` deposits `Coin<SUI>` into a `PriceInfoObject` dynamic field under `FEE_STORAGE_KEY`. No function in any production module ever withdraws those coins. The `fee_recipient_address` stored in `State` is explicitly noted as unused. All protocol revenue is permanently frozen.

### Finding Description

`update_single_price_feed` in `pyth.move` unconditionally calls `price_info::deposit_fee_coins`, which merges the caller's `Coin<SUI>` into the `PriceInfoObject`'s `FEE_STORAGE_KEY` dynamic object field: [1](#0-0) 

`deposit_fee_coins` is `public` (deposit-only) and has no corresponding withdrawal counterpart anywhere in the module: [2](#0-1) 

`State` stores a `fee_recipient_address` field: [3](#0-2) 

But the governance module `set_fee_recipient.move` itself documents that this field is vestigial â€” the comment on line 1 reads: *"The previous version of the contract sent the fees to a recipient address but this state is not used anymore. This module is kept for backward compatibility."* [4](#0-3) 

`set_fee_recipient::execute` only updates the address field in `State`; it never touches the `Coin<SUI>` stored in any `PriceInfoObject`: [5](#0-4) 

A full search of all production Move sources under `target_chains/sui/contracts/sources/` confirms there is no `withdraw_fees`, `transfer_to_recipient`, or any function that calls `dynamic_object_field::remove` or `coin::split` on the `FEE_STORAGE_KEY` field. The governance dispatch table in `governance.move` lists every executable action and none of them touch `PriceInfoObject` balances: [6](#0-5) 

### Impact Explanation
Every `Coin<SUI>` paid as an update fee since contract deployment is permanently locked inside the respective `PriceInfoObject` shared object. The balance can only increase (`get_balance` is monotonically non-decreasing). There is no on-chain code path â€” privileged or otherwise â€” that can move those coins to `fee_recipient_address` or anywhere else. This constitutes a complete, permanent loss of all protocol revenue.

### Likelihood Explanation
This is not a probabilistic exploit â€” it is a structural invariant violation present from the first `update_single_price_feed` call. Every user who pays an update fee contributes to the frozen balance. The condition is already true on mainnet if the contract is deployed as-is.

### Recommendation
Add a privileged withdrawal function (gated by `LatestOnly` and callable only by `fee_recipient_address` or via a governance VAA) that calls `dynamic_object_field::remove<vector<u8>, Coin<SUI>>` on `FEE_STORAGE_KEY` from a given `PriceInfoObject` and transfers the coin to `fee_recipient_address`. Alternatively, redirect fees at deposit time directly to `fee_recipient_address` using `sui::transfer::public_transfer` instead of storing them in the `PriceInfoObject`.

### Proof of Concept
Static invariant proof (no exploit transaction needed):

1. Call `update_single_price_feed(...)` with any valid VAA and fee â‰Ą `base_update_fee`. Observe `price_info::get_balance(price_info_object)` increases.
2. Enumerate every `public` and `public(friend)` function across all modules in `target_chains/sui/contracts/sources/`. None of them call `dynamic_object_field::remove` on `FEE_STORAGE_KEY` or reduce the balance of any `PriceInfoObject`.
3. Therefore `get_balance` is strictly monotonically non-decreasing for all reachable execution paths â€” the frozen-fee invariant holds unconditionally. [7](#0-6)

        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);

        };
