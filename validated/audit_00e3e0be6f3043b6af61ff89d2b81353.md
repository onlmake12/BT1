### Title
Exponent-Blind Deviation Calculation Drains Subscription Balances via False Positive Updates — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `_validateShouldUpdatePrices` function in `Scheduler.sol` computes price deviation by comparing raw Pyth price mantissas directly, without normalizing for their respective exponents. This is the exact decimal-unit discrepancy class from the reference report: two values that carry different implicit decimal scales are subtracted and divided as if they share the same scale. When the Pyth network changes the exponent for a feed (a legitimate, documented operation), any unprivileged keeper can submit the new valid update data and cause the Scheduler to compute a wildly incorrect deviation, triggering a false-positive update, paying the keeper, and draining the subscription's ETH balance.

---

### Finding Description

In `Scheduler.sol`, `_validateShouldUpdatePrices` reads the raw mantissa fields from both the incoming and the stored price feed and computes deviation entirely in mantissa space:

```solidity
// Lines 425–442
int64 currentPrice  = priceFeeds[i].price.price;   // raw mantissa, expo ignored
int64 previousPrice = previousFeed.price.price;    // raw mantissa, expo ignored

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
```

A Pyth price is defined as `actual_price = mantissa × 10^expo`. The exponent fields `priceFeeds[i].price.expo` and `previousFeed.price.expo` are never read or used in this calculation. The code implicitly assumes both prices share the same exponent, but this is not guaranteed or enforced.

**Concrete arithmetic:**

Suppose BTC/USD exponent changes from `-8` to `-9` (a one-digit precision increase, which Pyth publishers can legitimately emit):

| | mantissa | expo | actual price |
|---|---|---|---|
| stored (previous) | `9_742_350_000_000` | `-8` | $97,423.50 |
| incoming (current) | `97_423_500_000_000` | `-9` | $97,423.50 |

The code computes:
```
numerator   = |97_423_500_000_000 - 9_742_350_000_000| = 87_681_150_000_000
denominator = 9_742_350_000_000
deviationBps = 87_681_150_000_000 × 10_000 / 9_742_350_000_000 = 90_000 bps  (900%)
```

The actual price deviation is **0%**, but the contract sees **900%**, which exceeds any reasonable `deviationThresholdBps` and returns `updateTimestamp`, authorizing the update.

The inverse case (exponent increases by 1, mantissa shrinks 10×) produces a computed deviation of 9 000 bps (90%), still a false positive for any threshold below 9 000 bps.

---

### Impact Explanation

`_processFeesAndPayKeeper` is called unconditionally after a successful `_validateShouldUpdatePrices` return. It deducts `gasCost + keeperSpecificFee` from `status.balanceInWei` and transfers that ETH to `msg.sender` (the keeper). A false-positive deviation check therefore:

1. **Drains subscription ETH balances** — every exponent-change event allows a keeper to collect a full keeper fee for a spurious update.
2. **Causes repeated spurious updates** — because the stored price now has the new exponent, the next update with the old exponent (if the network reverts) triggers the same false positive again, creating a loop.
3. **Breaks subscriber SLAs** — subscriptions configured with tight deviation thresholds (e.g., 50 bps) will be triggered on every exponent change regardless of actual price movement, exhausting their balance and eventually deactivating.

---

### Likelihood Explanation

- Pyth price feed exponents are not immutable. The Pyth network documentation explicitly states that `expo` is part of the price data and can change when publishers adjust precision (e.g., after a large price move that shifts the significant digits).
- The `updatePriceFeeds` entry point in `Scheduler.sol` has **no access control** — any address can call it with valid Pyth accumulator update data obtained from Hermes.
- The attacker needs only to observe an exponent change on-chain (or anticipate one) and submit the valid signed update data. No privileged key, governance majority, or oracle manipulation is required.
- The Pyth accumulator proof fully validates the data, so the update is accepted by the Pyth contract and passed to the Scheduler.

---

### Recommendation

Normalize both prices to a common exponent before computing the deviation ratio. The existing `PythUtils.convertToUint` library function already handles this correctly:

```solidity
// Normalize both prices to the same target decimals (e.g., 18) before comparing
uint256 currentNorm  = PythUtils.convertToUint(
    priceFeeds[i].price.price, priceFeeds[i].price.expo, 18);
uint256 previousNorm = PythUtils.convertToUint(
    previousFeed.price.price,  previousFeed.price.expo,  18);

uint256 numerator    = currentNorm > previousNorm
    ? currentNorm - previousNorm
    : previousNorm - currentNorm;
uint256 deviationBps = Math.mulDiv(numerator, 10_000, previousNorm);
```

This mirrors the correct pattern already used in `PythPriceOracleGetter.sol` (lines 72–86), which explicitly applies `price.expo` when normalizing Pyth prices to a target decimal base.

### Proof of Concept

1. Deploy `SchedulerUpgradeable` and create a subscription with `updateOnDeviation = true`, `deviationThresholdBps = 100` (1%).
2. Submit an initial price update for BTC/USD with `price = 9_742_350_000_000`, `expo = -8` ($97,423.50). This is stored as `previousFeed`.
3. Obtain a valid Pyth accumulator update from Hermes where the BTC/USD feed has been published with `price = 97_423_500_000_000`, `expo = -9` (same $97,423.50, different exponent).
4. Call `Scheduler.updatePriceFeeds(subscriptionId, updateData)` as any EOA.
5. `_validateShouldUpdatePrices` computes `deviationBps = 90_000`, which is ≥ 100, and returns `updateTimestamp`.
6. `_processFeesAndPayKeeper` pays the caller `gasCost + singleUpdateKeeperFeeInWei × numPriceIds` from the subscription balance.
7. The subscription balance is drained by a full keeper fee for a zero-deviation price event. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }

    function convertToUint(
        int64 price,
        int32 expo,
        uint8 targetDecimals
    ) public pure returns (uint256) {
        if (price < 0) {
            revert PythErrors.NegativeInputPrice();
        }
        if (expo < -255) {
            revert PythErrors.InvalidInputExpo();
        }

        // If targetDecimals is 6, we want to multiply the final price by 10 ** -6
        // So the delta exponent is targetDecimals + currentExpo
        int32 deltaExponent = int32(uint32(targetDecimals)) + expo;

        // Bounds check: prevent overflow/underflow with base 10 exponentiation
        // Calculation: 10 ** n <= (2 ** 256 - 63) - 1
        //              n <= log10((2 ** 193) - 1)
        //              n <= 58.2
        if (deltaExponent > 58 || deltaExponent < -58)
            revert PythErrors.ExponentOverflow();

        // We can safely cast the price to uint256 because the above condition will revert if the price is negative
        uint256 unsignedPrice = uint256(uint64(price));

        if (deltaExponent > 0) {
            (bool success, uint256 result) = Math.tryMul(
                unsignedPrice,
                10 ** uint32(deltaExponent)
            );
            // This condition is unreachable since we validated deltaExponent bounds above.
            // But keeping it here for safety.
            if (!success) {
                revert PythErrors.CombinedPriceOverflow();
            }
            return result;
        } else {
            (bool success, uint256 result) = Math.tryDiv(
                unsignedPrice,
                10 ** uint(Math.abs(deltaExponent))
            );
            // This condition is unreachable since we validated deltaExponent bounds above.
            // But keeping it here for safety.
            if (!success) {
                revert PythErrors.CombinedPriceOverflow();
            }
            return result;
        }
    }
