Audit Report

## Title
Integer Overflow in `GET_LAST_STATE_PROOF_LIMIT` Guard Bypasses DoS Protection — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary

The guard at line 201 of `GetLastStateProofProcess::execute` computes `(last_n_blocks as usize) * 2` using unchecked multiplication. When a peer sends `last_n_blocks = 2^63`, this multiplication wraps to zero in Rust release mode, bypassing the `GET_LAST_STATE_PROOF_LIMIT` check entirely. The subsequent branch unconditionally collects every block number in the chain, forcing the server to perform O(N) MMR root computations and an O(N log N) `gen_proof` call per unauthenticated P2P message.

## Finding Description

**Root cause — line 201:** [1](#0-0) 

`last_n_blocks` is a peer-controlled `u64`. On a 64-bit target, `usize` is also 64 bits, so `2^63 as usize` = `9223372036854775808` with no truncation. The subsequent `* 2` computes `2^64`, which wraps to `0` in Rust release mode (overflow checks are disabled). With `difficulties.len() = 0`, the guard evaluates `0 > 1000 = false` and does not reject the message.

**Downstream consequence — lines 291–297:** [2](#0-1) 

Since no real chain has `2^63` blocks, `last_block_number - start_block_number <= 2^63` is always true. The "not enough blocks" branch is unconditionally taken, and `last_n_numbers` collects every block number from `start_block_number` to `last_block_number` — the entire chain.

**Work per block in `complete_headers` — lines 132–163:** [3](#0-2) 

For each of the N collected block numbers, the server calls `snapshot.get_ancestor` (chain traversal) and `snapshot.chain_root_mmr(*number - 1).get_root()` (O(log N) MMR root computation). There is no secondary length guard on `numbers`.

**Final `reply_proof` call — lines 207–216:** [4](#0-3) 

`items_positions` contains N entries (one per chain block). `gen_proof` over N positions on an MMR of size N is O(N log N). No secondary limit on `items_positions.len()` exists in `reply_proof`.

**Limit constant:** [5](#0-4) 

`GET_LAST_STATE_PROOF_LIMIT = 1000` is the only guard, and it is bypassed by the overflow.

## Impact Explanation

A single unauthenticated peer can force a full node to allocate a `Vec` of N block numbers, perform N MMR `get_root()` calls (each reading O(log N) DB nodes), and execute one `gen_proof` call over N positions. For a node with 100,000 blocks this is ~100,000 DB reads plus a large MMR proof computation in a single async handler. Repeated requests cause CPU exhaustion and/or OOM, crashing the node.

**Impact: High — Vulnerabilities which could easily crash a CKB node (10001–15000 points).**

## Likelihood Explanation

- The attacker requires only a valid tip block hash (publicly observable from the chain) and `start_number = 0`.
- No proof-of-work, no key, and no privileged role is required.
- The molecule encoding of `Uint64 = 2^63` is a standard 8-byte little-endian value, trivially constructable.
- The overflow is deterministic and reproducible in any standard `cargo build --release`.
- The handler is reachable via the standard light-client P2P protocol message path in `try_process`. [6](#0-5) 

## Recommendation

Replace the plain `*` with a saturating multiply to prevent wrap-to-zero:

```rust
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, reject `last_n_blocks` before the cast:

```rust
if last_n_blocks > (constant::GET_LAST_STATE_PROOF_LIMIT / 2) as u64 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

As defense-in-depth, add a secondary length guard in `complete_headers` or before the `gen_proof` call in `reply_proof` to cap `items_positions.len()`.

```rust
// Verify the overflow in Rust release mode:
let last_n_blocks: u64 = 1u64 << 63;
let x: usize = (last_n_blocks as usize).wrapping_mul(2);
assert_eq!(x, 0); // true on 64-bit in release mode

// Craft the malicious message:
// last_n_blocks        = 0x8000000000000000 (little-endian 8 bytes)
// difficulties         = [] (empty)
// last_hash            = <any valid main-chain tip hash>
// start_number         = 0
// start_hash           = <genesis hash>
// difficulty_boundary  = <any U256>
//
// Expected server behavior on a node with N=100_000 blocks:
// 1. Limit check: 0 + 0 = 0 > 1000 → false (bypassed)
// 2. last_n_numbers = (0..100_000).collect() → 100_000 entries
// 3. complete_headers: 100_000 × (get_ancestor + chain_root_mmr.get_root())
// 4. reply_proof: gen_proof(100_000 positions) → CPU exhaustion / OOM
```
