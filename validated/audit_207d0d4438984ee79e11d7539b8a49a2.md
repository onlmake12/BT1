### Title
Wrong Hardcoded Hardfork Activation Epoch Causes Consensus Divergence on Historical Sync - (File: `util/constant/src/hardfork/mainnet.rs`, `util/constant/src/hardfork/testnet.rs`)

---

### Summary

`CKB2021_START_EPOCH` is hardcoded to `0` in both the mainnet and testnet hardfork constant files, while the correct activation epochs are `5414` (mainnet) and `3113` (testnet). The correct values are even present as commented-out lines directly above the wrong ones. This wrong constant is consumed by the hardfork switch builder and propagates into consensus validation, causing any node syncing from genesis to apply CKB2021 rules from epoch 0 instead of the correct activation epoch, producing a consensus split against the rest of the network.

---

### Finding Description

In `util/constant/src/hardfork/mainnet.rs`:

```rust
/// First epoch number for CKB v2021, at about 2022/05/10 1:00 UTC
// pub const CKB2021_START_EPOCH: u64 = 5414;
pub const CKB2021_START_EPOCH: u64 = 0;
``` [1](#0-0) 

And in `util/constant/src/hardfork/testnet.rs`:

```rust
/// First epoch number for CKB v2021, at about 2021/10/24 3:15 UTC.
// pub const CKB2021_START_EPOCH: u64 = 3113;
pub const CKB2021_START_EPOCH: u64 = 0;
``` [2](#0-1) 

The correct values are explicitly present as commented-out lines. The active constant is wrong.

This constant is consumed in two places:

**1. `CKB2021::new_mirana()`** in `util/types/src/core/hardfork/ckb2021.rs` (lines 87–99), which sets the activation epoch for RFC-0029, RFC-0030, RFC-0031, RFC-0036, and RFC-0038 to `hardfork::mainnet::CKB2021_START_EPOCH` (i.e., `0`): [3](#0-2) 

**2. `HardForkConfig::complete_mainnet()` and `complete_testnet()`** in `spec/src/hardfork.rs`, which pass `mainnet::CKB2021_START_EPOCH` / `testnet::CKB2021_START_EPOCH` into `update_2021()`, setting the same RFC activation epochs: [4](#0-3) 

The affected RFCs and their consensus-critical behavior changes:

| RFC | Feature | Effect of wrong epoch |
|-----|---------|----------------------|
| RFC-0029 | Allow multiple cell dep matches | Accepts pre-fork txs that should be rejected |
| RFC-0030 | Stricter `since` epoch validation | Rejects valid pre-fork txs |
| RFC-0031 | `uncles_hash` reused as `extra_hash` | Rejects valid pre-fork blocks |
| RFC-0036 | Remove header deps immature rule | Accepts pre-fork txs that should be rejected |
| RFC-0038 | Disallow over max dep expansion limit | Rejects valid pre-fork txs | [5](#0-4) 

The RPC documentation itself already exposes the wrong values — the `get_consensus` example shows `"epoch_number": "0x0"` for RFC-0029 through RFC-0038, when mainnet should show `"0x0"` only for RFC-0028/0032 and `"0x1526"` (5414) for the rest: [6](#0-5) 

---

### Impact Explanation

A node running this code and syncing the mainnet chain from genesis applies CKB2021 consensus rules (RFC-0029, RFC-0030, RFC-0031, RFC-0036, RFC-0038) starting at epoch 0 instead of epoch 5414. Historical blocks (epochs 0–5413) were produced under pre-fork rules. The node will:

- **Reject valid historical blocks** that conform to pre-fork RFC-0031 (`uncles_hash`) format, because it expects the post-fork `extra_hash` format from genesis.
- **Reject valid historical transactions** that pass pre-fork RFC-0030 `since` validation but fail the stricter post-fork check.
- **Accept invalid historical transactions** that were correctly rejected pre-fork under RFC-0029 and RFC-0036 rules.

The result is a hard consensus split: the node cannot correctly sync the mainnet chain from genesis and will diverge from the honest network.

---

### Likelihood Explanation

Any new node joining mainnet or testnet and syncing from genesis is affected automatically — no attacker action is required to trigger the divergence. A sync peer (unprivileged) simply relays historical blocks (epochs 0–5413) through the normal sync protocol. The node applies wrong validation rules and either rejects valid blocks or accepts invalid ones. The CHANGELOG confirms mainnet CKB2021 activated at epoch 5414 and testnet at epoch 3113, so the gap between the wrong epoch (0) and the correct epoch is large and covers thousands of real historical blocks.

---

### Recommendation

Restore the correct activation epochs:

In `util/constant/src/hardfork/mainnet.rs`:
```rust
pub const CKB2021_START_EPOCH: u64 = 5414;
```

In `util/constant/src/hardfork/testnet.rs`:
```rust
pub const CKB2021_START_EPOCH: u64 = 3113;
```

These are the values present in the commented-out lines directly above the wrong constants, and match the values documented in the CHANGELOG ("Activate ckb2021 in mainnet since epoch 5414", "Activate ckb2021 in testnet since epoch 3113"). [7](#0-6) [8](#0-7) 

---

### Proof of Concept

1. Build a CKB node from this repository and configure it for mainnet.
2. Start syncing from genesis.
3. Call `get_consensus` RPC — observe `"rfc": "0029", "epoch_number": "0x0"` (wrong; should be `"0x1526"`).
4. The node applies RFC-0031 `extra_hash` validation from epoch 0. Historical mainnet blocks (epoch 0–5413) use the pre-fork `uncles_hash` format. The node rejects them as invalid, halting sync before epoch 5414.
5. Alternatively, submit a pre-fork transaction (epoch < 5414) that uses a `since` value valid under pre-fork RFC-0030 rules but invalid under post-fork rules — the node incorrectly rejects it.

The root cause is solely the wrong constant at: [9](#0-8) [10](#0-9)
