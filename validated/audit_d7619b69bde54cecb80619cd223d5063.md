### Title
Unregistered Address Can Steal Provider Fees via Unchecked `providerToCredit` in `executeCallback` - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
`Echo.executeCallback` credits provider fees to a caller-supplied `providerToCredit` address with no validation that the address is a registered provider or the original request provider. After the exclusivity period expires, any unprivileged caller can redirect the original provider's accrued fees to an arbitrary address they control, then withdraw those funds.

### Finding Description
In `Echo.sol`, `requestPriceUpdatesWithCallback` stores the provider's fee portion in `req.fee` and credits `_state.accruedFeesInWei` with only the Pyth platform fee: [1](#0-0) 

When `executeCallback` is called, the provider fee (`req.fee`) is credited to the caller-supplied `providerToCredit`: [2](#0-1) 

The only guard on `providerToCredit` is the exclusivity period check, which enforces that `providerToCredit == req.provider` only during the exclusivity window: [3](#0-2) 

After the exclusivity period, there is **no check** that `providerToCredit` is a registered provider, or even that it is the original `req.provider`. Any address — including an attacker's — can be credited.

The `EchoState.ProviderInfo.accruedFeesInWei` mapping is written to regardless of whether the address is registered: [4](#0-3) 

The attacker can then register as a provider, set themselves as their own fee manager, and drain the credited fees via `withdrawAsFeeManager`: [5](#0-4) 

### Impact Explanation
The original provider loses their rightful fees for fulfilling a request. The attacker receives ETH that belongs to the provider. The admin's `_state.accruedFeesInWei` (Pyth platform fees) is unaffected, but provider-side fee accounting is fully compromised. Any pending request whose exclusivity period has elapsed is exploitable.

### Likelihood Explanation
Pyth price update data is publicly available from the Pyth network. Any transaction sender can fetch valid `updateData` and `priceIds` and call `executeCallback`. The only prerequisite is waiting for the exclusivity period to expire. If the original provider is slow or offline, the attacker can front-run the legitimate fulfillment. The exclusivity period is configurable and can be set to zero, making every request immediately exploitable.

### Recommendation
Add a check in `executeCallback` that `providerToCredit` is either the original `req.provider` or a registered provider:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit must be a registered provider"
);
```

Additionally, consider restricting `providerToCredit` to only the original `req.provider` unless a formal provider substitution mechanism is intended.

### Proof of Concept
1. User calls `requestPriceUpdatesWithCallback` paying `pythFeeInWei + providerFees`. `req.fee = providerFees` is stored.
2. Attacker waits for `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`.
3. Attacker fetches valid Pyth price update data from the public Pyth network.
4. Attacker calls `executeCallback(attacker_address, sequenceNumber, updateData, priceIds)`.
5. `_state.providers[attacker_address].accruedFeesInWei += req.fee + msg.value - pythFee` — attacker is credited.
6. Attacker calls `registerProvider(...)` to register their address.
7. Attacker calls `setFeeManager(attacker_address)` to set themselves as fee manager.
8. Attacker calls `withdrawAsFeeManager(attacker_address, amount)` — ETH is transferred to attacker.

The original provider's fees are fully drained with no privileged access required.

        );

            _state.providers[provider].accruedFeesInWei >= amount,
