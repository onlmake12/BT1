Audit Report

## Title
Admission Check Uses Serialized Size Only While Fee Estimator and Pool Ordering Use Actual Weight, Allowing Sub-`min_fee_rate` Transactions to Poison Fee Estimator — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` gates pool admission using only the transaction's serialized byte size, while every downstream component (pool ordering, eviction, fee estimation) uses `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. An unprivileged sender can craft a high-cycle, low-size transaction that passes the size-only admission check while carrying an effective fee rate orders of magnitude below `min_fee_rate`. Such transactions enter the pool and are recorded by the fee estimator at their true depressed effective fee rate, poisoning fee rate recommendations for all users.

## Finding Description
**Admission check — size only (`tx-pool/src/util.rs` L42-52):**
The code itself documents the discrepancy with the comment "here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly." `check_tx_fee` is called inside `pre_check` before script verification, so `cycles` are not yet known. The check passes for any transaction paying `fee ≥ min_fee_rate × tx_size`.

**Actual weight formula (`util/types/src/core/tx_pool.rs` L298-303):**
`get_transaction_weight` returns `max(tx_size, cycles * 0.000_170_571_4)`. For 70 M cycles the cycle-equivalent weight is ~11 940 bytes.

**No second check after verification (`tx-pool/src/process.rs` L734-751):**
After `verify_rtx` returns `verified.cycles`, the code immediately constructs `TxEntry::new(rtx, verified.cycles, fee, tx_size)` with no weight-based fee rate re-check. The only post-verification check is a `declared_cycles` mismatch guard.

**Fee estimator uses actual weight (`util/fee-estimator/src/estimator/weight_units_flow.rs` L97-100):**
`TxStatus::new_from_entry_info` calls `get_transaction_weight(info.size, info.cycles)` and computes the true effective fee rate — far below `min_fee_rate` for the crafted transaction — then records it in historical flow buckets via `accept_tx`.

**Pool ordering also uses actual weight (`tx-pool/src/component/entry.rs` L115-118, L221-231):**
`fee_rate()`, `AncestorsScoreSortKey::from`, and `EvictKey::from` all call `get_transaction_weight`, so these entries are correctly ranked low and evicted first — but not before being recorded in the fee estimator's historical data.

**Concrete numbers** (`min_fee_rate = 1 000 shannons/KW`, `tx_size = 200 B`, `cycles = 70 000 000`):

| Step | Value |
|---|---|
| Admission threshold | `1000 × 200 / 1000 = 200 shannons` |
| Actual weight | `max(200, 11 940) = 11 940` |
| Effective fee rate | `200 × 1000 / 11 940 ≈ 16.7 shannons/KW` |
| Ratio below floor | ~60× |

## Impact Explanation
**High — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (10001–15000 points).**
The `weight_units_flow` algorithm stores per-block flow data keyed by `current_tip`. Each admitted transaction is recorded via `accept_tx` with its true depressed effective fee rate. Historical flow at very low fee rates accumulates, causing `estimate_fee_rate` to return artificially low recommendations. Legitimate users following these recommendations underpay, causing their transactions to be stuck or evicted, degrading network throughput. The attacker's per-transaction cost is only `min_fee_rate × tx_size = 200 shannons` — ~60× cheaper than a legitimately weighted transaction at the same cycle count.

## Likelihood Explanation
Any unprivileged user can trigger this via `send_transaction` RPC or P2P relay. The attacker deploys a lock/type script that consumes close to `max_tx_verify_cycles` (70 M) while keeping the serialized transaction small (~200 bytes). Script deployment is a one-time on-chain cost. The attack is repeatable at low marginal cost and requires no special privileges.

## Recommendation
After `verify_rtx` completes and `verified.cycles` is known, perform a second fee rate check using the actual transaction weight before constructing `TxEntry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((
        Err(Reject::LowFeeRate(
            tx_pool.config.min_fee_rate,
            min_fee_by_weight.as_u64(),
            fee.as_u64(),
        )),
        snapshot,
    ));
}
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

This closes the gap between the pre-verification size-only gate and the post-verification weight-based accounting used everywhere else.

## Proof of Concept
1. Deploy a CKB lock script that loops until it consumes ~70 M cycles (a tight VM loop with no useful work).
2. Create a transaction spending a cell locked by that script. Keep the serialized size small (~200 bytes by minimizing witnesses/outputs).
3. Submit via `send_transaction` RPC with fee = 200 shannons (`min_fee_rate × tx_size`).
4. Observe: `check_tx_fee` passes (200 ≥ 200). Script verification runs and records `cycles ≈ 70 000 000`. `TxEntry` is created with those cycles. No second fee check fires.
5. Query `estimate_fee_rate` before and after flooding with N such transactions. The returned fee rate recommendation drops significantly below `min_fee_rate` as the estimator's flow buckets fill with 16.7 shannons/KW entries.
6. Confirm via `get_pool_info` that the transactions are present in the pool with their true cycles, and via the fee estimator's internal bucket trace logs that low-fee-rate buckets dominate.
