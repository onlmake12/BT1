### Title
`blockhash(req.blockNumber)` Reveal Window Collapses on Fast-Block Chains — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol` stores `block.number` at request time and later calls `blockhash(req.blockNumber)` during reveal. The EVM `blockhash` opcode only returns a non-zero value for the 256 most recent blocks. On Ethereum this window is ~51 minutes, but on chains with fast block production (zkSync, Arbitrum Nova, etc.) 256 blocks can elapse in seconds. Any user who calls `request()` with `useBlockHash = true` on such a chain will find their request permanently un-revealable once the window closes, locking their paid fee in the contract with no recovery path.

---

### Finding Description

`requestHelper` records the current block number into every request: [1](#0-0) 

During `reveal`, `revealHelper` calls `blockhash(req.blockNumber)` and reverts with `BlockhashUnavailable` if the result is zero: [2](#0-1) 

The inline comment explicitly acknowledges the 256-block constraint and frames the valid reveal window as "blocks 1–255 after the request block." On Ethereum (~12 s/block) this is roughly 3–51 minutes. On zkSync Era, where a new L2 block can be produced every 1–2 seconds (or even faster during high throughput), 256 blocks elapse in 4–8 minutes. On chains that batch many L2 blocks per L1 block the window can be even shorter.

The public `request()` entry point accepts `useBlockHash` as a caller-controlled boolean: [3](#0-2) 

Once the 256-block window closes the request is permanently stuck: `reveal()` always reverts, `clearRequest` is never called, and the fee already credited to the provider is non-refundable. There is no timeout or fallback path that lets the user recover funds or obtain a random number through an alternative route.

The newer `requestV2` / `requestWithCallback` path hard-codes `useBlockhash = false` and is not affected: [4](#0-3) 

---

### Impact Explanation

A user who calls `request()` with `useBlockHash = true` on a fast-block chain:

1. Pays the full provider + Pyth fee (non-refundable).
2. Has a window of only seconds to minutes to call `reveal()` before `blockhash` returns zero.
3. After the window closes, every `reveal()` call reverts with `BlockhashUnavailable`.
4. The request slot is never cleared; the user receives no random number and cannot recover their fee.

This constitutes permanent loss of user funds and permanent denial of the randomness service for affected requests.

---

### Likelihood Explanation

- Pyth Entropy is already deployed on multiple EVM-compatible chains, several of which produce blocks far faster than Ethereum mainnet.
- Any user or integrating contract that calls the legacy `request()` API with `useBlockHash = true` is affected automatically — no special attacker action is required; the chain's own block production rate triggers the failure.
- The failure is silent from the user's perspective: the request appears valid on-chain until they attempt to reveal, at which point it is already too late.

---

### Recommendation

1. **Document the incompatibility**: Add an explicit NatSpec warning on `request()` stating that `useBlockHash = true` is unsafe on chains where 256 blocks can elapse in less than the expected user-reveal latency.
2. **Add a chain-specific guard or disable the flag**: On deployment to fast-block chains, either force `useBlockhash = false` in `requestHelper`, or revert if `useBlockHash = true` is passed.
3. **Provide a recovery path**: Add a timeout mechanism (e.g., after N blocks or T seconds) that allows a user to cancel a stuck `useBlockhash` request and reclaim their fee.
4. **Prefer the newer API**: Deprecate the `request()` / `reveal()` path in favour of `requestV2` / `revealWithCallback`, which already hard-codes `useBlockhash = false`.

---

### Proof of Concept

1. Deploy `Entropy` on a chain where blocks are produced every ~1 second (e.g., zkSync Era).
2. Call `request(provider, userCommitment, true)` — `useBlockHash = true`.
3. Wait ~260 seconds (≈ 260 blocks on a 1 s/block chain).
4. Call `reveal(provider, sequenceNumber, userContribution, providerContribution)`.
5. `revealHelper` executes `blockhash(req.blockNumber)` → returns `bytes32(0)` because the block is older than 256 blocks.
6. The call reverts with `EntropyErrors.BlockhashUnavailable`.
7. The user's fee is permanently locked; the request can never be fulfilled. [5](#0-4)

    function revealHelper(
        EntropyStructsV2.Request storage req,
        bytes32 userContribution,
        bytes32 providerContribution
    ) internal returns (bytes32 randomNumber, bytes32 blockHash) {
        bytes32 providerCommitment = constructProviderCommitment(
            req.numHashes,
            providerContribution
        );
        bytes32 userCommitment = constructUserCommitment(userContribution);
        if (
            keccak256(bytes.concat(userCommitment, providerCommitment)) !=
            req.commitment
        ) revert EntropyErrors.IncorrectRevelation();

        blockHash = bytes32(uint256(0));
        if (req.useBlockhash) {
            bytes32 _blockHash = blockhash(req.blockNumber);

            // The `blockhash` function will return zero if the req.blockNumber is equal to the current
            // block number, or if it is not within the 256 most recent blocks. This allows the user to
            // select between two random numbers by executing the reveal function in the same block as the
            // request, or after 256 blocks. This gives each user two chances to get a favorable result on
            // each request.
            // Revert this transaction for when the blockHash is 0;
            if (_blockHash == bytes32(uint256(0)))
                revert EntropyErrors.BlockhashUnavailable();

            blockHash = _blockHash;
        }

        randomNumber = combineRandomValues(
            userContribution,
            providerContribution,
            blockHash
        );

        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            req.provider
        ];
        if (providerInfo.currentCommitmentSequenceNumber < req.sequenceNumber) {
            providerInfo.currentCommitmentSequenceNumber = req.sequenceNumber;
            providerInfo.currentCommitment = providerContribution;
        }
    }
