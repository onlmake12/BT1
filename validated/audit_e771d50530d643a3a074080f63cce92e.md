Audit Report

## Title
Tx-Pool Size Limit Bypassed During Reorg Re-add Path — (`File: tx-pool/src/process.rs`)

## Summary
`readd_detached_tx` re-inserts transactions from detached blocks back into the pool after a chain reorganization by calling `_submit_entry` in a loop, but never calls `limit_size` afterward. Because it executes after `_update_tx_pool_for_reorg` (which does enforce the limit), the pool's `total_tx_size` can exceed `config.max_tx_pool_size` by the cumulative size of all re-added detached transactions, violating the operator-configured memory bound.

## Finding Description
The normal submission path in `submit_entry` enforces the size limit immediately after inserting an entry: [1](#0-0) 

`_update_tx_pool_for_reorg` also enforces the limit at its conclusion: [2](#0-1) 

However, the call sequence in `update_tx_pool_for_reorg` runs `_update_tx_pool_for_reorg` first (which brings the pool to ≤ `max_tx_pool_size`), then immediately calls `readd_detached_tx` with no subsequent `limit_size`: [3](#0-2) 

Inside `readd_detached_tx`, each successful `_submit_entry` call adds to `pool_map.total_tx_size` with no eviction check: [4](#0-3) 

`limit_size` enforces the bound via a `while total_tx_size > max_tx_pool_size` loop that evicts lowest-fee-rate entries: [5](#0-4) 

Since this loop is never entered after `readd_detached_tx`, the pool can hold `max_tx_pool_size + Σ(detached_tx_sizes)` bytes of data, where the sum is over all transactions in the detached chain segment that are not present in the newly attached segment.

## Impact Explanation
The `max_tx_pool_size` configuration is the sole memory-bounding mechanism for the tx-pool. Bypassing it allows heap growth beyond the operator's configured limit. With `TRANSACTION_SIZE_LIMIT = 512 KB` per transaction and blocks containing many transactions, even a single-block reorg can push the pool meaningfully over its limit. A multi-block reorg amplifies this proportionally. On resource-constrained nodes where `max_tx_pool_size` is set close to available memory, this can cause an OOM kill of the node process. This maps to **High: Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
Reorgs are a routine part of CKB network operation and require no attacker involvement — they occur naturally when two miners find blocks at the same height. The `update_tx_pool_for_reorg` path is exercised on every reorg. Any peer relaying a valid competing block triggers this path. A miner with modest hashpower can deliberately and repeatedly trigger reorgs to amplify pool growth. No special privileges or victim mistakes are required.

## Recommendation
After `readd_detached_tx` completes, call `limit_size` to restore the invariant, mirroring the pattern in `_update_tx_pool_for_reorg`:

```rust
self.readd_detached_tx(&mut tx_pool, retain, fetched_cache).await;
// Enforce size limit after re-adding detached transactions
let _ = tx_pool.limit_size(&self.callbacks, None);
```

This should be inserted at line 851 of `tx-pool/src/process.rs`, inside the same write-lock scope.

## Proof of Concept
1. Configure a CKB node with a small `max_tx_pool_size` (e.g., 2 MB).
2. Submit enough large transactions (each near `TRANSACTION_SIZE_LIMIT = 512 KB`) to fill the pool to its limit. Confirm via `get_tx_pool_info` that `total_tx_size ≈ max_tx_pool_size`.
3. Mine a block on a second node that includes those transactions but does not broadcast it. On the first node, mine a competing block at the same height that does not include those transactions.
4. Relay the second node's chain to the first node, triggering a reorg. The chain service calls `update_tx_pool_for_reorg`.
5. `_update_tx_pool_for_reorg` runs and calls `limit_size`, bringing the pool to ≤ `max_tx_pool_size`. Then `readd_detached_tx` re-adds all detached transactions without calling `limit_size`.
6. Query `get_tx_pool_info` and observe that `total_tx_size` now exceeds `max_tx_pool_size` by the size of the re-added transactions, confirming the invariant violation.
