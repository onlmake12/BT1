Audit Report

## Title
Unbounded `pending_get_block_proposals` Map Growth via Unauthenticated Peer Messages Causes Memory/CPU Exhaustion - (File: sync/src/types/mod.rs)

## Summary
Any peer that completes the relay handshake can repeatedly send `GetBlockProposal` messages containing up to 3,000 unique fabricated `ProposalShortId` values. Because none of these IDs exist in the local tx-pool, all are inserted unconditionally into the unbounded `pending_get_block_proposals` `DashMap` in `SyncState`. On each relay timer tick, `drain_get_block_proposals` clones the entire map in O(n) time and memory, stalling the relay service and eventually exhausting node memory.

## Finding Description
`SyncState` holds an unbounded `DashMap` for caching unanswered proposal requests:

```rust
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>,
``` [1](#0-0) 

`insert_get_block_proposals` inserts every supplied ID with no cap check:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [2](#0-1) 

This is called from `GetBlockProposalProcess::execute()` for every proposal ID not found in the tx-pool: [3](#0-2) 

The only guard is a per-message size check capped at `max_block_proposals_limit * max_uncles_num` (3,000 on mainnet): [4](#0-3) 

There is no rate limit, per-peer quota, or cumulative map-size cap. On the relay timer, `drain_get_block_proposals` clones the entire map before clearing it:

```rust
pub fn drain_get_block_proposals(
    &self,
) -> DashMap<packed::ProposalShortId, HashSet<PeerIndex>> {
    let ret = self.pending_get_block_proposals.clone();
    self.pending_get_block_proposals.clear();
    ret
}
``` [5](#0-4) 

Cloning a `DashMap` with millions of entries is O(n) in both time and memory and acquires locks on every internal shard, stalling all concurrent readers/writers. The CHANGELOG confirms analogous fixes were applied to `inflight_proposals` (#3093) and `inflight_blocks` (#3094), but `pending_get_block_proposals` was not addressed by those fixes: [6](#0-5) 

## Impact Explanation
A single unauthenticated peer can drive unbounded memory growth and cause O(n) CPU stalls on the relay timer, which can crash the victim CKB node via OOM or render the relay subsystem unresponsive, degrading block-proposal and transaction propagation. This matches the **High** impact class: "Vulnerabilities which could easily crash a CKB node" (10,001–15,000 points).

## Likelihood Explanation
The `GetBlockProposal` message is part of the standard `RelayV3` protocol reachable by any peer after handshake. Fabricating unique 10-byte `ProposalShortId` values is trivial. No rate limiting, connection-level quota, or map-size cap exists on this path. The CHANGELOG already records analogous fixes for `inflight_proposals` (#3093) and `inflight_blocks` (#3094) memory bloat, confirming the pattern is known and exploitable; `pending_get_block_proposals` was not addressed by those fixes. [6](#0-5) 

## Recommendation
1. **Cap the map size**: in `insert_get_block_proposals`, refuse to insert new entries once `pending_get_block_proposals` exceeds a configurable maximum (e.g., `max_block_proposals_limit * max_uncles_num * MAX_CONNECTED_PEERS`).
2. **Per-peer quota**: track how many pending entries each peer has contributed and reject further insertions once a per-peer limit is reached; ban peers that exceed it.
3. **Avoid full clone on drain**: swap `pending_get_block_proposals` with a fresh empty `DashMap` using `std::mem::replace` instead of cloning, eliminating the O(n) allocation under shard locks.

## Proof of Concept
```
Attacker peer → victim node (RelayV3 protocol)

loop forever:
    ids = [random_10_byte_id() for _ in range(3000)]  # unique, non-existent
    msg = GetBlockProposal { block_hash: tip_hash, proposals: ids }
    send(msg)
    # passes per-message limit check (3000 ≤ 3000)
    # victim inserts all 3000 IDs into pending_get_block_proposals
    # no rate limit → repeat immediately

After N iterations:
    pending_get_block_proposals.len() == 3000 * N

On next relay timer tick:
    drain_get_block_proposals() clones 3000*N-entry DashMap
    → O(N) memory allocation + O(N) CPU under shard locks
    → relay timer stalls; node OOMs or relay service becomes unresponsive
```

Entry point: `GetBlockProposalProcess::execute` in `sync/src/relayer/get_block_proposal_process.rs`, reachable from any peer via the `RelayV3` protocol handler. [7](#0-6)

```rust
    pub async fn execute(self) -> Status {
        let shared = self.relayer.shared();
        let message_len = self.message.proposals().len();
        {
            // The block proposal request is separate from uncles,
            // so here the limit is only used to calculate the maximum value of uncles
            let limit = shared.consensus().max_block_proposals_limit()
                * (shared.consensus().max_uncles_num() as u64);
            if message_len as u64 > limit {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "GetBlockProposal proposals count({message_len}) > consensus max_block_proposals_limit({limit})"
                ));
            }
        }

        let proposals: HashSet<packed::ProposalShortId> =
            self.message.proposals().to_entity().into_iter().collect();

        if proposals.len() != message_len {
            return StatusCode::RequestDuplicate.with_context("Request duplicate proposal");
        }

        let fetched_transactions = {
            let tx_pool = self.relayer.shared.shared().tx_pool_controller();
            let fetch_txs = tx_pool.fetch_txs(proposals.clone()).await;
            if let Err(e) = fetch_txs {
                debug_target!(
                    crate::LOG_TARGET_RELAY,
                    "relayer tx_pool_controller send fetch_txs error: {:?}",
                    e
                );
                return Status::ok();
            }
            fetch_txs.unwrap()
        };
        // Transactions that do not exist on this node
        let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
            .into_iter()
            .filter(|short_id| !fetched_transactions.contains_key(short_id))
            .collect();

        // Cache request, try process on timer
        self.relayer
            .shared()
            .state()
            .insert_get_block_proposals(self.peer, not_exist_proposals);
```
