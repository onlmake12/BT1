### Title
Permanent Subscription Balance Bypasses `MAX_DEPOSIT_LIMIT` via Repeated `addFunds` Calls — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.sol` contract enforces a `MAX_DEPOSIT_LIMIT` on permanent subscriptions by checking each individual `msg.value` against the cap. However, neither `createSubscription` nor `addFunds` checks whether the **resulting cumulative balance** (`status.balanceInWei + msg.value`) exceeds `MAX_DEPOSIT_LIMIT`. An unprivileged caller can bypass the cap entirely by making multiple `addFunds` calls, each individually within the limit, accumulating a total balance that is an arbitrary multiple of `MAX_DEPOSIT_LIMIT`.

---

### Finding Description

In `Scheduler.sol`, the deposit cap for permanent subscriptions is enforced as follows:

**`createSubscription`** (lines 47–50):
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**`addFunds`** (lines 612–617):
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

Both checks validate only the **per-call deposit amount** (`msg.value`), not the **post-deposit total balance** (`status.balanceInWei + msg.value`). Since permanent subscriptions prohibit withdrawals and cannot be deactivated, the `MAX_DEPOSIT_LIMIT` is the sole mechanism preventing unbounded ETH from being locked in a single permanent subscription. The missing check on the cumulative balance renders this protection ineffective.

This is structurally identical to M-04: the function validates the input quantity (per-call deposit) but not the derived/resulting quantity (total balance), allowing the intended cap to be bypassed. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Permanent subscriptions are designed to be irrevocable: funds cannot be withdrawn, the subscription cannot be deactivated, and price feeds cannot be removed. The `MAX_DEPOSIT_LIMIT` is the only guard against locking an excessive amount of ETH in such a subscription. By bypassing it, an attacker (or a misconfigured integrator) can:

1. Lock an arbitrarily large amount of ETH permanently in a single subscription with no recovery path.
2. Cause the keeper system to process a subscription with an anomalously large balance, potentially disrupting fee accounting or keeper incentive logic that assumes balances are bounded.
3. Grief the subscription owner by front-running their `addFunds` calls with additional deposits that push the balance far beyond what was intended, since `addFunds` has no access control — any address can call it for any subscription.

The third point is particularly impactful: because `addFunds` is permissionless (any address can fund any subscription), a third party can permanently lock excess ETH in someone else's permanent subscription without their consent, and the victim has no recourse since withdrawals are blocked for permanent subscriptions. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

The entry path requires no privileges: any externally owned account can call `addFunds` on any active subscription. The bypass requires only multiple sequential transactions, each sending `msg.value = MAX_DEPOSIT_LIMIT`. This is trivially executable on-chain. The permissionless nature of `addFunds` means even a third party (not the subscription manager) can trigger this against a victim's permanent subscription. [5](#0-4) 

---

### Recommendation

Replace the per-call `msg.value` check with a post-addition cumulative balance check in both `createSubscription` and `addFunds`:

```solidity
// In addFunds:
uint256 newBalance = status.balanceInWei + msg.value;
if (params.isPermanent && newBalance > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei = newBalance;
```

Apply the same pattern in `createSubscription` (though at creation time `status.balanceInWei` is zero, so `msg.value` and the cumulative balance are equal — the existing check there is sufficient; the critical fix is in `addFunds`).

---

### Proof of Concept

```solidity
// Attacker bypasses MAX_DEPOSIT_LIMIT on a permanent subscription

uint256 maxLimit = scheduler.MAX_DEPOSIT_LIMIT(); // e.g., 100 ether

// Step 1: Create a permanent subscription at the limit (passes check)
params.isPermanent = true;
uint256 subId = scheduler.createSubscription{value: maxLimit}(params);

// Step 2: Add funds N times, each call passes the per-call check
for (uint i = 0; i < 9; i++) {
    scheduler.addFunds{value: maxLimit}(subId); // msg.value == MAX_DEPOSIT_LIMIT, passes check
}

// Result: balanceInWei == 10 * MAX_DEPOSIT_LIMIT, 10x the intended cap
// Funds are permanently locked; no withdrawal is possible
(, SchedulerStructs.SubscriptionStatus memory status) = scheduler.getSubscription(subId);
assert(status.balanceInWei == 10 * maxLimit); // cap bypassed
``` [6](#0-5) [3](#0-2)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
