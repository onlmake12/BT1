### Title
`HeaderAcceptor::accept()` Skips `BLOCK_INVALID` Check, Allowing Re-Acceptance of Known-Invalid Block Headers — (`File: sync/src/synchronizer/headers_process.rs`)

---

### Summary

`HeaderAcceptor::accept()` contains a self-acknowledged `FIXME` that skips the `BLOCK_INVALID` status check. When a peer sends a `SendHeaders` message containing a header for a block already marked `BLOCK_INVALID`, the node accepts the header, inserts it into the header map, and unconditionally updates the `shared_best_header` to point to the invalid block. This drives the sync engine to re-download and re-process the invalid block, creating a repeatable DoS loop reachable by any unprivileged sync peer.

---

### Finding Description

In `HeaderAcceptor::accept()`, the code reads the block status but only acts on `HEADER_VALID`. The `BLOCK_INVALID` case is explicitly left unhandled with a `FIXME` comment:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ...
    return result;
}
``` [1](#0-0) 

If the block is `BLOCK_INVALID` (bit `1 << 12`, disjoint from `HEADER_VALID`), the code falls through all three sub-checks (`prev_block_check`, `non_contextual_check`, `version_check`). If the header has valid PoW, a correct block number, valid epoch, and an acceptable timestamp, all three pass and `insert_valid_header` is called:

```rust
sync_shared.insert_valid_header(self.peer, self.header);
``` [2](#0-1) 

`insert_valid_header` computes the header's cumulative total difficulty and calls `may_set_shared_best_header`:

```rust
self.shared.header_map().insert(header_view.clone());
self.state.peers().may_set_best_known_header(peer, header_view.as_header_index());
self.state.may_set_shared_best_header(header_view);
``` [3](#0-2) 

`may_set_shared_best_header` unconditionally overwrites the global best-known header if the incoming header's total difficulty is higher — with no `BLOCK_INVALID` guard:

```rust
pub fn may_set_shared_best_header(&self, header: HeaderIndexView) {
    let mut shared_best_header = self.shared_best_header.write();
    if !header.is_better_than(shared_best_header.total_difficulty()) {
        return;
    }
    *shared_best_header = header;
}
``` [4](#0-3) 

`shared_best_header` is the primary signal used by `better_tip_header()` to drive block download scheduling:

```rust
let best_known = self.shared.state().shared_best_header();
if total_difficulty > *best_known.total_difficulty() {
    (header, total_difficulty).into()
} else {
    best_known
}
``` [5](#0-4) 

By contrast, `compact_block_process.rs` correctly guards against this case:

```rust
if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
``` [6](#0-5) 

The `BlockStatus` flags confirm `BLOCK_INVALID` (`1 << 12`) is entirely disjoint from `HEADER_VALID` (`1`), so the existing check never catches it:

```rust
const HEADER_VALID  =     1;
const BLOCK_INVALID =     1 << 12;
``` [7](#0-6) 

---

### Impact Explanation

An attacker who has previously caused a block to be marked `BLOCK_INVALID` (e.g., by submitting a valid-PoW block with invalid transactions that fail contextual verification in `ConsumeUnverifiedBlocks`) can then send the header for that block to the victim node via the `SendHeaders` sync message from a fresh peer connection. The node will:

1. Accept the header (no `BLOCK_INVALID` guard).
2. Insert it into the header map.
3. Overwrite `shared_best_header` with the invalid block's header if its total difficulty is higher.
4. Begin scheduling block downloads toward the invalid chain tip.
5. Re-download the invalid block, re-process it, re-mark it `BLOCK_INVALID`, and delete it from the DB.
6. Repeat from step 1 indefinitely.

This creates a sustained DoS loop: the node wastes CPU (contextual verification), disk I/O (write then delete), and network bandwidth (re-downloading the block) on every cycle. The `ConsumeUnverifiedBlocks` thread is single-threaded, so stalling it with repeated invalid-block processing degrades the node's ability to process legitimate blocks. [8](#0-7) 

---

### Likelihood Explanation

The attacker needs to mine one valid-PoW block whose body fails contextual verification (e.g., a double-spend, an invalid cellbase reward, or a two-phase-commit violation). This is a one-time cost. After that, the header-replay attack is free: sending a `SendHeaders` message costs nothing, and the victim node has no rate-limit or ban logic for this path because `HeaderAcceptor::accept()` returns a `ValidationResult` with `state = Valid` (the default) when it falls through to `insert_valid_header` — the caller in `HeadersProcess::execute()` sees no error and does not penalize the peer. [9](#0-8) 

---

### Recommendation

Add an early-return guard for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept()`, resolving the existing `FIXME`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

// Resolve FIXME: reject headers for blocks already known to be invalid.
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing path
}
```

This mirrors the guard already present in `compact_block_process.rs` and prevents the `shared_best_header` from being poisoned by a known-invalid block's header.

---

### Proof of Concept

1. **Mine** a block `B` with valid PoW but with a transaction that double-spends a cell (fails `ContextualBlockVerifier`).
2. **Submit** `B` to the victim node via the sync `SendBlock` message. The node stores it, runs `ConsumeUnverifiedBlocks`, marks it `BLOCK_INVALID`, and deletes it from the DB.
3. **Open a new peer connection** to the victim node.
4. **Send** a `SendHeaders` message containing only the header of `B`.
5. **Observe**: `HeaderAcceptor::accept()` passes all sub-checks (PoW valid, number valid, epoch valid, timestamp valid). `insert_valid_header` is called. `shared_best_header` is updated to `B`'s header.
6. **Observe**: The sync engine schedules a download of `B`. The node re-downloads `B`, re-runs `ConsumeUnverifiedBlocks`, re-marks it `BLOCK_INVALID`, and deletes it again.
7. **Repeat** step 4 from the same or a new peer connection to sustain the loop. [10](#0-9) [11](#0-10)

    pub fn accept(&self) -> ValidationResult {
        let mut result = ValidationResult::default();
        let sync_shared = self.active_chain.sync_shared();
        let state = self.active_chain.state();
        let shared = sync_shared.shared();

        // FIXME If status == BLOCK_INVALID then return early. But which error
        // type should we return?
        let status = self.active_chain.get_block_status(&self.header.hash());
        if status.contains(BlockStatus::HEADER_VALID) {
            let header_index = sync_shared
                .get_header_index_view(
                    &self.header.hash(),
                    status.contains(BlockStatus::BLOCK_STORED),
                )
                .unwrap_or_else(|| {
                    panic!(
                        "header {}-{} with HEADER_VALID should exist",
                        self.header.number(),
                        self.header.hash()
                    )
                })
                .as_header_index();
            state
                .peers()
                .may_set_best_known_header(self.peer, header_index);
            return result;
        }

        if self.prev_block_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-parent header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        if let Some(is_invalid) = self.non_contextual_check(&mut result).err() {
            debug!(
                "HeadersProcess rejected non-contextual header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            if is_invalid {
                shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            }
            return result;
        }

        if self.version_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-version header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        sync_shared.insert_valid_header(self.peer, self.header);
        result
    }

    pub fn insert_valid_header(&self, peer: PeerIndex, header: &core::HeaderView) {
        let tip_number = self.active_chain().tip_number();
        let store_first = tip_number >= header.number();
        // We don't use header#parent_hash clone here because it will hold the arc counter of the SendHeaders message
        // which will cause the 2000 headers to be held in memory for a long time
        let parent_hash = Byte32::from_slice(header.data().raw().parent_hash().as_slice())
            .expect("checked slice length");
        let parent_header_index = self
            .get_header_index_view(&parent_hash, store_first)
            .expect("parent should be verified");
        let mut header_view = HeaderIndexView::new(
            header.hash(),
            header.number(),
            header.epoch(),
            header.timestamp(),
            parent_hash,
            parent_header_index.total_difficulty() + header.difficulty(),
        );

        let snapshot = Arc::clone(&self.shared.snapshot());
        header_view.build_skip(
            tip_number,
            |hash, store_first| self.get_header_index_view(hash, store_first),
            |number, current| {
                // shortcut to return an ancestor block
                if current.number <= snapshot.tip_number() && snapshot.is_main_chain(&current.hash)
                {
                    snapshot
                        .get_block_hash(number)
                        .and_then(|hash| self.get_header_index_view(&hash, true))
                } else {
                    None
                }
            },
        );
        self.shared.header_map().insert(header_view.clone());
        self.state
            .peers()
            .may_set_best_known_header(peer, header_view.as_header_index());
        if header_view.number().is_multiple_of(10000) {
            info!(
                "inserted valid header: header {}-{}",
                header_view.number(),
                header_view.hash()
            );
        }
        self.state.may_set_shared_best_header(header_view);
    }
