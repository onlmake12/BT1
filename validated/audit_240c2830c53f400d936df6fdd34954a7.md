Audit Report

## Title
Global `total_tx_size`/`total_tx_cycles` Counters Inflated When Eviction Occurs During `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the updated global counters are computed into local variables before `check_and_record_ancestors` runs. When that function evicts existing transactions, `remove_entry` correctly decrements `self.total_tx_size`/`self.total_tx_cycles` via `update_stat_for_remove_tx`. However, the stale pre-eviction local variables are then unconditionally written back to `self`, silently overwriting those decrements. The result is permanently inflated counters for the lifetime of the pool, until an underflow triggers `recompute_total_stat`.

## Finding Description
In `add_entry` (lines 200–221 of `tx-pool/src/component/pool_map.rs`):

```rust
// Step 1: snapshot new totals into LOCAL variables (pre-eviction)
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;  // line 210-211

// Step 2: may call remove_entry_and_descendants → remove_entry
//         → update_stat_for_remove_tx → self.total_tx_{size,cycles} -= evicted.{size,cycles}
evicts = self.check_and_record_ancestors(&mut entry)?;         // line 213

// Step 3: OVERWRITES the decrements with the stale pre-eviction snapshot
self.total_tx_size  = total_tx_size;   // line 218
self.total_tx_cycles = total_tx_cycles; // line 219
```

The eviction path inside `check_and_record_ancestors` (lines 603–625) is triggered when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants` (line 618), which calls `remove_entry` (line 247), which calls `update_stat_for_remove_tx` (line 247) and correctly decrements `self.total_tx_size`/`self.total_tx_cycles`. Those decrements are then immediately overwritten at lines 218–219 with the stale local variables.

The fallback `recompute_total_stat` (lines 698–708) is only triggered on underflow in `update_stat_for_remove_tx` (line 742), not on this inflation path. There is no guard that detects or corrects the overwrite.

**Concrete example:**

| Step | `self.total_tx_size` |
|------|---------------------|
| Initial | 1000 |
| `updated_stat_for_add_tx(100, ...)` → local var = 1100 | (local only) |
| Eviction of tx(size=200) via `update_stat_for_remove_tx` | 800 |
| Overwrite with local var | **1100** (expected: 900) |

## Impact Explanation
The most concrete impact is on `limit_size` (pool.rs line 298):
```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
```
An inflated `total_tx_size` causes `limit_size` to evict valid, higher-fee transactions from the mempool even when the pool is not actually full. Each attacker-triggered eviction event inflates the counter by the evicted transaction's size, and `limit_size` then removes legitimate transactions to compensate for the phantom inflation. This constitutes unnecessary eviction of valid transactions and degraded mempool quality. Secondary impacts include incorrect `tx_pool_info` RPC reporting and, in extreme cases (many repeated eviction events), premature `Reject::Full` for new transactions. This matches **Low (501–2000 points): Any other important performance improvements for CKB**, with a secondary argument for Medium given the mempool accounting invariant violation.

## Likelihood Explanation
The eviction path requires: (1) a chain of transactions in the pool whose ancestor count exceeds `max_ancestors_count` (default 25), and (2) some ancestors are "cell-ref parents" sharing the same cell dep as the new transaction. An unprivileged submitter via RPC `send_transaction` or P2P relay can craft a sequence of transactions all referencing the same widely-used cell dep (e.g., a popular lock script), forming a chain that exceeds `max_ancestors_count`. Submitting a new transaction into this chain triggers the eviction path. Each such submission inflates the counters by the evicted transaction's size/cycles. Repeated submissions compound the inflation. The cost to the attacker is only the transaction fees for the submitted transactions.

## Recommendation
Move the counter update to after `check_and_record_ancestors` completes, so evictions are already reflected in `self.total_tx_size`/`self.total_tx_cycles` before the new entry's contribution is added:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Add only the new entry's contribution to already-eviction-adjusted counters
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

The overflow pre-check (`updated_stat_for_add_tx`) should either be moved to after eviction or account for the sizes/cycles of transactions that will be evicted.

## Proof of Concept
1. Create a `PoolMap` with `max_ancestors_count = 3`.
2. Insert transactions `T1`, `T2`, `T3` that all reference the same cell dep `D` (making them cell-ref parents of each other). Record `pool.total_tx_size` after each insertion.
3. Submit `T4` that spends an output of `T3` and also references cell dep `D`. Its ancestor count = 4 > 3, but `cell_ref_parents` contains `{T1, T2, T3}`, so `4 - 3 = 1 <= 3` satisfies the eviction condition.
4. `check_and_record_ancestors` evicts `T1` (lowest fee) via `remove_entry_and_descendants` → `update_stat_for_remove_tx` decrements `self.total_tx_size` by `T1.size`.
5. Lines 218–219 overwrite with the pre-eviction local variable, re-inflating by `T1.size`.
6. Assert: `pool.total_tx_size == actual_sum_of_entry_sizes(pool)` — this assertion fails, confirming the inflation.
7. Repeat step 3–6 with new transactions to compound the inflation and observe `limit_size` evicting valid transactions unnecessarily.
