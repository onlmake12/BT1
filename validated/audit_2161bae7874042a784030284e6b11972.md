### Title
Unconditional 1-Byte Frame Triggers Immediate 300-Second IP Ban via `LengthDelimitedCodecWithCompress::decode` — (`network/src/compress.rs`)

### Summary

The `data.len() < 2` guard in `LengthDelimitedCodecWithCompress::decode` is unconditional. A remote peer that completes the secio handshake and sends a single 5-byte TCP payload (4-byte length prefix `[0x00,0x00,0x00,0x01]` + 1 flag byte `[0x00]`) causes the codec to return `Err(io::ErrorKind::InvalidData)`, which the tentacle framework surfaces as `ServiceError::ProtocolError`, which `handle_error` in `network.rs` converts into an unconditional 300-second IP ban — with no prior misbehavior scoring required.

**Step 1 — Codec decode path** [1](#0-0) 

The check at line 228 is unconditional: it fires regardless of `self.enable_compress`. A frame whose length-delimited payload is exactly 1 byte (the flag byte only, no application payload) satisfies `max_frame_length` for every protocol but fails `data.len() < 2`, returning `Err(io::ErrorKind::InvalidData)`.

The existing test at line 154 explicitly demonstrates this exact byte sequence: [2](#0-1) 

**Step 2 — Codec is always installed for CKB application protocols**

`CKBProtocol::build()` always wraps the inner `LengthDelimitedCodec` with `LengthDelimitedCodecWithCompress`, regardless of the `compress` flag (which only controls encoding, not decoding): [3](#0-2) 

This applies to Sync, RelayV3, Time, Alert, LightClient, and Filter protocols. All have `max_frame_length` values far above 1 byte: [4](#0-3)

### Impact Explanation

An unprivileged attacker who controls N distinct IP addresses can:
1. Open one TCP connection per IP to the victim node.
2. Complete the secio handshake (no special key required).
3. Open any CKB application protocol substream.
4. Send 5 bytes: `[0x00, 0x00, 0x00, 0x01, 0x00]`.
5. Cause the victim to ban each of those N IPs for 300 seconds.

During the ban window, any legitimate peer sharing one of those IPs cannot connect. Repeated cycling across IP pools degrades the victim's peer diversity and inbound connection capacity continuously.

**Correction to the question's claim**: The ban is per exact IP (`/32`), not per `/24` subnet. The claim that "a legitimate peer from the same /24 is subsequently rejected" is not supported by `ip_to_network`'s implementation. The impact is exact-IP banning, not subnet-wide banning.

---

### Likelihood Explanation

- Requires only a TCP connection and a valid secio handshake — no privileged access, no PoW, no keys.
- The attack payload is 5 bytes and is confirmed by the existing unit test.
- The `is_whitelist` guard in `ban_session` only protects explicitly whitelisted peers; ordinary inbound peers have no protection.
- No rate-limiting or misbehavior scoring precedes the ban decision for codec-level errors.

---

### Recommendation

1. **Do not ban on codec errors without prior scoring.** Codec-level `InvalidData` should increment a misbehavior score; only ban after the score exceeds a threshold.
2. **Alternatively**, treat a codec error as a disconnect (not a ban), consistent with how `ServiceError::MuxerError` and `ServiceError::SessionTimeout` are handled.
3. **Or**, add a minimum-frame-size check inside the tentacle codec layer before the application-level `data.len() < 2` guard, so that a 1-byte frame is silently dropped rather than treated as peer misbehavior.
