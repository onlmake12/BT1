### Title
Inbound Peer Can Inject Arbitrary Addresses with Attacker-Controlled Flags into Peer Store via Identify `listen_addrs` — (`network/src/protocols/identify/mod.rs`)

---

### Summary

An unprivileged inbound peer that passes identify verification (correct network name, non-zero flags) can include arbitrary third-party IP addresses in its `listen_addrs` field. The `add_remote_listen_addrs` function stores all of those addresses into the peer store using the **attacker's own flags** from their identify message, with no validation that the reported addresses belong to the connecting peer. This allows flag spoofing for any address in the peer store, which is then broadcast via the discovery protocol.

---

### Finding Description

The `received` handler in `IdentifyProtocol` executes this sequence unconditionally for both inbound and outbound sessions: [1](#0-0) 

1. `received_identify` is called first. It calls `verify()`, which only checks that the network name matches and `flag != 0`. On success, it stores the attacker-supplied `flags` into `peer.identify_info`: [2](#0-1) 

2. The code at lines 415–451 is gated on `is_outbound()` — so for an **inbound** peer, the only effect of `received_identify` is writing the attacker's flags into `peer.identify_info`. The function returns `Continue`. [3](#0-2) 

3. `process_listens` is then called with the attacker's `listen_addrs`. It filters for globally reachable IPs and calls `add_remote_listen_addrs`: [4](#0-3) 

4. `add_remote_listen_addrs` reads `flags` from `peer.identify_info` (set in step 1 from the attacker's message) and calls `peer_store.add_addr` for **every address in the attacker's list**, with no check that those addresses belong to the connecting peer: [5](#0-4) 

5. `PeerStore::add_addr` passes `last_connected_at_ms = 0` to `AddrManager::add`: [6](#0-5) 

6. `AddrManager::add` overwrites an existing entry when `addr_info.last_connected_at_ms >= exist_last_connected_at_ms`. Since the attacker's entry has `last_connected_at_ms = 0`, it overwrites any existing entry that also has `last_connected_at_ms = 0` (i.e., addresses discovered but never yet connected to), and inserts fresh entries for addresses not yet in the store: [7](#0-6) 

The developer comment at lines 416–419 explicitly acknowledges the concern about broadcasting unverified inbound addresses — but this guard only prevents the inbound peer's **own** connection address from being stored. It does not protect the `listen_addrs` path, which is the attack vector here.

---

### Impact Explanation

- **Flag downgrade**: Attacker sends `flags = Flags::COMPATIBILITY` (0b1) and includes addresses of known full-node bootnodes or mining pool nodes in `listen_addrs`. Those nodes are stored with only `COMPATIBILITY` flags. Other nodes querying the peer store with `required_flags = SYNC | RELAY` will not receive those addresses from discovery, effectively eclipsing them from the network's view.
- **Flag upgrade / misdirection**: Attacker sends `flags = SYNC | RELAY` and includes addresses of light-client-only nodes. Those nodes are stored with `SYNC | RELAY` flags. Other nodes will attempt to open sync/relay protocols with them and fail, wasting connection slots.
- **Persistence**: The peer store is serialized to disk (`addr_manager.db`). Poisoned entries survive node restarts.
- **Propagation**: The discovery protocol's `GetNodes` / `Nodes` exchange broadcasts peer store contents to the wider network, spreading the poisoned entries. [8](#0-7) 

---

### Likelihood Explanation

- Requires only a standard inbound TCP connection and knowledge of the target network's name string (publicly known for mainnet: `"ckb"`).
- No cryptographic material, hashpower, or privileged access is needed.
- A single attacker node can continuously reconnect and re-inject entries, since the peer store eviction policy is based on connection quality metrics, not flag integrity.
- The attack is locally testable with a minimal P2P client.

---

### Recommendation

In `add_remote_listen_addrs`, restrict storage to addresses that match the connecting peer's own identity (same peer ID prefix or same IP), or decouple the flags used for `listen_addrs` from the peer's self-reported identify flags. The flags stored for a third-party address should only be set after a direct connection to that address is verified (as is already done for outbound peers via `add_outbound_addr`).

Alternatively, apply the same guard already present for the inbound peer's own address (lines 415–423) to the `listen_addrs` path: skip `peer_store.add_addr` entirely for addresses reported by inbound peers, or store them with a fixed `Flags::COMPATIBILITY` regardless of the reporter's flags.

---

### Proof of Concept

```
1. Start a CKB node (victim) on mainnet or testnet.
2. Connect to it as an inbound peer using a custom P2P client.
3. Send an IdentifyMessage with:
     identify = { name: "ckb", flag: SYNC|RELAY, client_version: "0.1" }
     listen_addrs = [ <known_bootnode_addr_1>, <known_bootnode_addr_2>, ... ]
     observed_addr = <any valid addr>
4. After the session, query the victim's peer store (via RPC get_peers or
   by reading addr_manager.db on disk).
5. Assert that known_bootnode_addr_1 is now stored with flags = SYNC|RELAY
   (attacker-supplied), regardless of the bootnode's actual capabilities.
6. Observe that subsequent discovery responses from the victim broadcast
   the bootnode addresses with the attacker-supplied flags.
```
