### Title
Unvalidated `providerToCredit` in `executeCallback` Permanently Locks User Fees — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, the `executeCallback` function accepts a caller-supplied `providerToCredit` address and credits the user's escrowed fee to it without validating that the address is a registered provider. After the exclusivity period elapses, any unprivileged caller can invoke `executeCallback` with `providerToCredit = address(0)` (or any address that has no fee manager set), permanently locking the user's fees inside the contract with no recovery path.

### Finding Description

`requestPriceUpdatesWithCallback` escrows the user's fee minus the Pyth protocol fee into `req.fee`: [1](#0-0) 

`executeCallback` then credits that escrowed fee to the caller-supplied `providerToCredit`: [2](#0-1) 

The only guard on `providerToCredit` is the exclusivity-period check, which enforces that `providerToCredit == req.provider` only while `block.timestamp < req.publishTime + exclusivityPeriodSeconds`. Once that window closes, the parameter is completely unconstrained: [3](#0-2) 

There is no subsequent check that `providerToCredit` is a registered provider. The two withdrawal paths available in the contract are:

- `withdrawFees` — admin-only, drains `_state.accruedFeesInWei` (Pyth protocol fees only, not provider fees). [4](#0-3) 
- `withdrawAsFeeManager` — requires `msg.sender == _state.providers[provider].feeManager`. [5](#0-4) 

For `address(0)`, `_state.providers[address(0)].feeManager` is `address(0)` by default (zero-initialized storage). `withdrawAsFeeManager` would require `msg.sender == address(0)`, which is impossible. The fees credited to `address(0)` are permanently irrecoverable.

The `ProviderInfo` struct confirms `accruedFeesInWei` is stored per-address in the `providers` mapping: [6](#0-5) 

### Impact Explanation

A griefing attacker waits for the exclusivity period to expire, then calls `executeCallback(address(0), sequenceNumber, validUpdateData, priceIds)`. The user's escrowed fee is credited to `address(0)` and permanently locked. The legitimate provider receives nothing. The user's callback may succeed (price update delivered) but their fee is destroyed. There is no admin recovery path because `withdrawFees` only covers `_state.accruedFeesInWei`, not per-provider balances.

**Impact: High** — user funds permanently locked with no recovery mechanism.

### Likelihood Explanation

**Likelihood: Medium.** The attack requires:
1. Waiting for the exclusivity period to elapse (configurable; 15 seconds in test deployments).
2. Submitting a valid `executeCallback` transaction before the legitimate provider does.

Both conditions are easily achievable by a motivated attacker monitoring the mempool. No privileged access is required.

### Recommendation

Add a check in `executeCallback` that `providerToCredit` is a registered provider before crediting fees:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit not registered"
);
```

This mirrors the pattern already used in `requestPriceUpdatesWithCallback`: [7](#0-6) 

### Proof of Concept

1. Alice calls `requestPriceUpdatesWithCallback{value: fee}(defaultProvider, block.timestamp, priceIds, gasLimit)`. Her fee (minus `pythFeeInWei`) is stored in `req.fee`.
2. The exclusivity period (`exclusivityPeriodSeconds`) elapses.
3. Attacker calls `executeCallback(address(0), sequenceNumber, validUpdateData, priceIds)` with no `msg.value`.
4. Inside `executeCallback`, line 161–162 executes: `_state.providers[address(0)].accruedFeesInWei += req.fee - pythFee`. Alice's fee is credited to `address(0)`.
5. The request is cleared (line 164). Alice cannot retry.
6. `withdrawAsFeeManager(address(0), amount)` requires `msg.sender == address(0)` — impossible. Funds are permanently locked.
7. `withdrawFees` (admin) only drains `_state.accruedFeesInWei`, not `_state.providers[address(0)].accruedFeesInWei`. No recovery path exists. [8](#0-7)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);
