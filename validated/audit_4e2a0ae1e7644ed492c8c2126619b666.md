### Title
Concurrent `HeaderMapKernel::get()` Calls Race on Destructive `backend.remove()`, Causing `unwrap_or_else(panic!)` on `HEADER_VALID` Header — (`shared/src/types/header_map/kernel_lru.rs`)

---

### Summary

`HeaderMapKernel::get()` promotes a sled-backend entry to memory via a destructive `backend.remove()`. Two concurrent callers for the same hash that is only in the sled backend will race: one wins the remove and gets `Some(view)`, the other gets `None` and returns it to the caller. When the losing thread is executing `HeaderAcceptor::accept()` in `headers_process.rs`, the subsequent `unwrap_or_else(|| panic!(...))` fires because the header has `HEADER_VALID` status but is no longer findable in either the header map or the chain store.

---

### Finding Description

**Root cause — `kernel_lru.rs` lines 132–141:** [1](#0-0) 

`get()` first checks `memory.get_refresh()` (a write-locked `LinkedHashMap`), then, on a miss, calls `self.backend.remove(hash)` — a destructive sled operation that atomically deletes and returns the entry. There is **no lock or coordination** between the memory-miss check and the backend remove. Two threads that both miss the memory check will both call `backend.remove()` for the same key; sled's atomicity guarantees only one returns `Some`, the other returns `None`.

**Sled `remove()` — `backend_sled.rs` lines 90–100:** [2](#0-1) 

The sled `remove` is atomic per-key, so exactly one thread wins. The losing thread returns `None` from `HeaderMapKernel::get()`.

**Exact race interleaving:**

| Step | Thread A | Thread B |
|------|----------|----------|
| 1 | `memory.get_refresh(H)` → `None` | |
| 2 | | `memory.get_refresh(H)` → `None` |
| 3 | `backend.remove(H)` → `Some(view)` | |
| 4 | | `backend.remove(H)` → `None` ← **loses** |
| 5 | `memory.insert(view)` | |
| 6 | returns `Some(view)` | returns `None` ← **propagates up** |

Thread B's `memory.insert` by Thread A at step 5 is too late — Thread B already passed the memory check at step 2.

**Panic site — `headers_process.rs` lines 304–316:** [3](#0-2) 

`HeaderAcceptor::accept()` checks `BlockStatus::HEADER_VALID` and then calls `sync_shared.get_header_index_view()` with `store_first = status.contains(BlockStatus::BLOCK_STORED)`. For a header that is `HEADER_VALID` but not yet `BLOCK_STORED`, `store_first` is `false`.

**`get_header_index_view()` — `sync/src/types/mod.rs` lines 1158–1166:** [4](#0-3) 

With `store_first = false`, it calls `header_map().get(hash)` first. Thread B's call returns `None` (race). The fallback to the chain store also returns `None` because the header is not yet stored on-chain. The entire function returns `None`, and the `unwrap_or_else(|| panic!(...))` fires.

**How entries reach the sled backend:**

`limit_memory()` runs every 5 seconds in a background task and evicts the oldest entries from the `MemoryMap` into sled: [5](#0-4) 

During IBD, the memory limit is routinely exceeded, making sled-only entries common.

---

### Impact Explanation

The panic terminates the CKB node process. Any attacker who can cause two sync threads to concurrently call `get_header_index_view()` for the same `HEADER_VALID`, non-stored header hash can crash the node. Because the panic is unconditional (`unwrap_or_else(|| panic!(...))`) and the process is not restarted automatically by default, this is a **remote denial-of-service** against any syncing node.

---

### Likelihood Explanation

During IBD or rapid header sync, the node processes `SendHeaders` messages from multiple peers concurrently. The memory limit is routinely exceeded, pushing entries to sled. An attacker with two or more peer connections can send the same valid header batch simultaneously, reliably triggering concurrent `get()` calls for the same hash. The race window (between the memory miss and the sled remove) is narrow but reproducible under load. No PoW forgery or privileged access is required — only standard P2P peer connections sending valid (real-chain) headers.

---

### Recommendation

Replace the destructive `backend.remove()` promotion in `HeaderMapKernel::get()` with a non-destructive `backend.get()`. Promotion to memory (and eviction from sled) should be done separately and idempotently, or the entire get-and-promote sequence should be serialized per key (e.g., with a per-key mutex or a single coarse lock around the backend lookup + memory insert). A simpler fix: use `backend.get()` instead of `backend.remove()` in the read path, and only call `backend.remove()` lazily (e.g., during the next `limit_memory()` sweep or on explicit eviction).

//    a header whose block_status is HEADER_VALID — observe the panic.

let r1 = h1.join().unwrap();
let r2 = h2.join().unwrap();
