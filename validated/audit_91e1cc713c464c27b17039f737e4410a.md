### Title
Provider Fees Permanently Stuck When `executeCallback` Called With `providerToCredit = address(0)` - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, the `executeCallback` function accepts a caller-supplied `providerToCredit` address and unconditionally credits provider fees to `_state.providers[providerToCredit].accruedFeesInWei`. There is no check that `providerToCredit != address(0)` and no requirement that it be a registered provider. After the exclusivity period expires, any unprivileged relayer can call `executeCallback` with `providerToCredit = address(0)`, permanently locking the provider fee portion of the request in the contract with no recovery path.

### Finding Description

`executeCallback` in `Echo.sol` enforces that `providerToCredit == req.provider` only during the exclusivity window: [1](#0-0) 

After that window, any address — including `address(0)` — is accepted as `providerToCredit`. The fee accrual then writes to the zero-address provider slot: [2](#0-1) 

The only withdrawal path for provider-accrued fees is `withdrawAsFeeManager`, which requires `msg.sender == _state.providers[provider].feeManager`: [3](#0-2) 

Because `address(0)` is never registered and its `feeManager` slot defaults to `address(0)`, no real account can ever satisfy that check for `provider = address(0)`. The `setFeeManager` function also cannot rescue this because it keys on `msg.sender` as the provider: [4](#0-3) 

The admin-only `withdrawFees` function only drains `_state.accruedFeesInWei` (Pyth protocol fees), not per-provider balances: [5](#0-4) 

There is no other code path that can recover funds credited to `_state.providers[address(0)]`.

### Impact Explanation

Provider fees (`req.fee`, the portion of `msg.value` above the Pyth fee) are permanently locked in the `Echo` contract. The legitimate provider who fulfilled the request receives nothing. Any additional ETH sent by the relayer via `msg.value` in `executeCallback` is also lost. Funds are irrecoverable without a contract upgrade.

### Likelihood Explanation

`executeCallback` is a public, permissionless function callable by any relayer after the exclusivity period. No special privilege is required. A malicious or buggy relayer can trivially pass `address(0)` as `providerToCredit`. The exclusivity period is a configurable short window, so the attack surface is always available for any sufficiently old unfulfilled request.

### Recommendation

Add a zero-address guard and/or a registered-provider check at the top of `executeCallback`:

```solidity
require(providerToCredit != address(0), "providerToCredit is zero address");
require(_state.providers[providerToCredit].isRegistered, "Provider not registered");
```

This mirrors the pattern already used in `EntropyGovernance.sol`: [6](#0-5) 

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback(provider, publishTime, priceIds, callbackGasLimit)` with sufficient ETH. `req.fee` is set to `msg.value - pythFeeInWei`.
2. The exclusivity period (`_state.exclusivityPeriodSeconds`) elapses without the assigned provider fulfilling the request.
3. Attacker calls `executeCallback(address(0), sequenceNumber, updateData, priceIds)`.
4. The exclusivity check passes (period expired). Fees are credited: `_state.providers[address(0)].accruedFeesInWei += req.fee`.
5. No account can call `withdrawAsFeeManager(address(0), amount)` because `_state.providers[address(0)].feeManager == address(0)`.
6. Funds are permanently stuck. [7](#0-6)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);
