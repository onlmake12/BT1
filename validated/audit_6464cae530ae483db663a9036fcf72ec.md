### Title
`PythAggregatorV3.latestRoundData()` / `latestAnswer()` Return Arbitrarily Stale Prices With No Staleness Check — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink-compatible adapter contract, shipped in the production Solidity SDK and explicitly recommended in the Chainlink migration guide. Every price-reading function in the contract calls `getPriceUnsafe()` with no staleness validation. Any protocol using this adapter for liquidations or collateral valuation will silently receive prices from arbitrarily far in the past.

---

### Finding Description

`PythAggregatorV3` implements Chainlink's `AggregatorV3Interface` as a drop-in replacement for Chainlink feeds. It is the official migration path Pyth recommends to protocols moving from Chainlink.

Every price-reading function in the contract calls `pyth.getPriceUnsafe(priceId)`:

- `latestAnswer()` — line 54
- `latestRoundData()` — line 110
- `getRoundData()` — line 89
- `decimals()` — line 41
- `latestTimestamp()` — line 59

`getPriceUnsafe()` is explicitly documented as returning a price "from arbitrarily far in the past" with no recency guarantee. The `PythAggregatorV3` contract has no `maxAge` constructor parameter, no staleness threshold, and no call to `getPriceNoOlderThan()`. There is no revert, no flag, and no warning returned to the caller when the price is stale.

The `latestRoundData()` return value is the primary function Chainlink-compatible protocols use for liquidations and collateral valuation. Protocols that migrated from Chainlink using this adapter will call `latestRoundData()` expecting a fresh price, but will silently receive a price that may be hours or days old. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Any lending, derivatives, or liquidation protocol that uses `PythAggregatorV3` as a Chainlink feed replacement is exposed to stale-price exploitation:

- **False liquidations**: If a collateral asset's price dropped and then recovered, an attacker can trigger a liquidation using the stale low price.
- **Over-borrowing**: If a collateral asset's price rose and then fell, a borrower can borrow against the stale high price.
- **Price manipulation via inaction**: An attacker simply waits for the on-chain Pyth price to go stale (no `updateFeeds()` call is required from the attacker), then interacts with the victim protocol.

The impact is identical to the original report: stale oracle prices enabling false liquidations and over-borrowing. [3](#0-2) 

---

### Likelihood Explanation

- `PythAggregatorV3` is a production contract in the official Pyth Solidity SDK, deployed by real protocols.
- The migration guide explicitly instructs protocols to deploy and use it as a Chainlink replacement.
- Pyth is a pull oracle — prices only update when someone calls `updateFeeds()`. If no keeper is running, prices go stale immediately.
- An unprivileged attacker needs only to wait for staleness and then interact with a victim protocol. No special access, no key, no governance majority required.
- The contract is deployed on Arbitrum (address `0xff1a0f4744e8582DF1aE09D5611b887B6a12925C` is listed in the docs), making the L2-sequencer-downtime scenario from the original report directly applicable as one trigger. [4](#0-3) 

---

### Recommendation

Add a configurable `maxAge` staleness threshold to `PythAggregatorV3` and replace all `getPriceUnsafe()` calls with `getPriceNoOlderThan(priceId, maxAge)`. The `maxAge` should be set in the constructor and should revert if the cached price is too old, consistent with how Chainlink's `latestRoundData()` is expected to behave.

```solidity
uint256 public maxAge;

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    // ...
}
``` [5](#0-4)

### Proof of Concept

1. Protocol X migrates from Chainlink to Pyth using `PythAggregatorV3` for ETH/USD collateral valuation, following the official Pyth migration guide.
2. ETH price is $3000. The last `updateFeeds()` call was 2 hours ago. No keeper is running.
3. ETH price drops to $2000 on-chain (Pyth VAA is available on Hermes but not yet submitted).
4. Attacker calls `latestRoundData()` on `PythAggregatorV3` — it returns $3000 (stale, via `getPriceUnsafe()`), no revert.
5. Attacker borrows the maximum amount against their ETH collateral at the stale $3000 price.
6. The actual collateral is worth $2000; the protocol is immediately undercollateralized.

No privileged access is required. The attacker's only action is submitting a borrow transaction. The root cause is entirely within Pyth's own `PythAggregatorV3` contract. [6](#0-5) [7](#0-6)
