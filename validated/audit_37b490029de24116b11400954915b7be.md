### Title
Keeper Can Drain Subscription Balance via Inflated `tx.gasprice` in `_processFeesAndPayKeeper` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `_processFeesAndPayKeeper` function calculates the keeper's gas reimbursement using `tx.gasprice` without any upper bound. Because the keeper is the transaction submitter (`msg.sender`) and also the payment recipient, they fully control `tx.gasprice`. A malicious keeper can inflate this value to drain a subscription's entire balance in a single `updatePriceFeeds` call.

---

### Finding Description

In `Scheduler._processFeesAndPayKeeper`, the keeper fee is computed as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [1](#0-0) 

The only guard is a balance check:

```solidity
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [2](#0-1) 

The keeper then receives `totalKeeperFee` directly:

```solidity
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [3](#0-2) 

There is no cap on `tx.gasprice`. The keeper controls this value when submitting the transaction. On legacy chains they set `gasPrice` directly; on EIP-1559 chains they set `maxPriorityFeePerGas` to inflate the effective `tx.gasprice` (`baseFee + tip`).

The keeper's net profit from the gas component is `GAS_OVERHEAD * tx.gasprice` (30,000 gas units × gas price), because the actual gas they spend is approximately `(startGas - gasleft()) * tx.gasprice`, which cancels out. By setting `tx.gasprice` to an extreme value, the keeper can make `totalKeeperFee` approach `status.balanceInWei`, draining the subscription in one call. [4](#0-3) 

The `updatePriceFeeds` function is permissionless — any address can call it: [5](#0-4) 

---

### Impact Explanation

A malicious keeper can drain a subscription's entire deposited balance in a single `updatePriceFeeds` call by submitting the transaction with an artificially inflated gas price. The subscription manager loses their deposited funds and the subscription becomes inoperable (balance falls below `getMinimumBalance`). This is a direct, irreversible loss of user funds held in the contract.

---

### Likelihood Explanation

The `updatePriceFeeds` function is intentionally permissionless — any address can act as a keeper. No registration, stake, or privileged role is required. The attacker only needs to:
1. Identify a subscription with a meaningful balance.
2. Call `updatePriceFeeds` with a valid price update and an inflated `tx.gasprice` (or `maxPriorityFeePerGas` on EIP-1559 chains).

The attack is straightforward, requires no special knowledge beyond reading the contract, and is profitable as long as the subscription balance exceeds the attacker's actual gas cost.

---

### Recommendation

Add a maximum gas price cap to `_processFeesAndPayKeeper`. The admin-configurable `maxGasPriceInWei` parameter should be checked before computing `gasCost`:

```solidity
uint256 effectiveGasPrice = tx.gasprice < _state.maxGasPriceInWei
    ? tx.gasprice
    : _state.maxGasPriceInWei;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, cap `totalKeeperFee` at a configurable fraction of the subscription balance per call, preventing full drainage in a single transaction.

---

### Proof of Concept

1. Subscription manager creates a subscription and deposits 1 ETH.
2. Attacker calls `scheduler.updatePriceFeeds(subscriptionId, validUpdateData)` with `tx.gasprice = 1_000_000 gwei`.
3. Inside `_processFeesAndPayKeeper`:
   - `gasUsed ≈ 200,000` (typical for the function)
   - `gasCost = (200,000 + 30,000) * 1_000_000 gwei = 230,000 * 1e9 * 1e9 wei = 0.23 ETH`
   - Attacker pays `200,000 * 1_000_000 gwei = 0.2 ETH` in actual gas
   - Attacker receives `0.23 ETH + keeperSpecificFee` from the subscription
   - Net profit: `0.03 ETH + keeperSpecificFee` per call
4. By increasing `tx.gasprice` further (e.g., to `4,000,000 gwei`), `totalKeeperFee` approaches 1 ETH, draining the subscription in one transaction while the attacker's actual gas cost is `200,000 * 4,000,000 gwei = 0.8 ETH` — still profitable if the subscription balance is large enough.     function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
