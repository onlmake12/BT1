### Title
Unbounded Accumulation in `pending_get_block_proposals` Enables Peer-Triggered O(N) Processing DoS — (File: `sync/src/types/mod.rs`, `sync/src/relayer/get_block_proposal_process.rs`)

---

### Summary

A connected P2P peer can send repeated `GetBlockProposal` relay messages containing proposal short IDs that do not exist in the local tx-pool. Each such message causes non-existent IDs to be inserted into the `pending_get_block_proposals` `DashMap` with no size cap. On every `TX_PROPOSAL_TOKEN` timer tick, `prune_tx_proposal_request` drains the entire accumulated map, clones it, and issues a single `fetch_txs` call to the tx-pool service with all accumulated IDs. An attacker can inflate this map to an arbitrarily large size, causing O(N) CPU and memory pressure on each timer tick and blocking the tx-pool service for legitimate operations.

---

### Finding Description

**Root cause — unbounded insertion in `insert_get_block_proposals`:**

`SyncState::insert_get_block_proposals` inserts every non-existent proposal ID into a `DashMap` with no size limit:

```rust

    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [1](#0-0) 

**Trigger path — `GetBlockProposalProcess::execute`:**

When a peer sends a `GetBlockProposal` message, the handler fetches matching transactions from the tx-pool and then caches all IDs that were *not found* into `pending_get_block_proposals`:

```rust

    .into_iter()
    .filter(|short_id| !fetched_transactions.contains_key(short_id))
    .collect();

self.relayer
    .shared()
    .state()
    .insert_get_block_proposals(self.peer, not_exist_proposals);
``` [2](#0-1) 

An attacker sends proposal IDs for transactions that do not exist anywhere (random 10-byte IDs). All of them pass the per-message count check and all of them are classified as "not found," so all are inserted into the unbounded map.

**Per-message count check is insufficient:**

The handler enforces a per-message limit of `max_block_proposals_limit * max_uncles_num` (on mainnet: 1500 × 2 = 3000 IDs per message): [3](#0-2) 

The relayer rate-limiter allows 30 messages per second per `(PeerIndex, message_type)` pair: [4](#0-3) 

This means one peer can inject up to **30 × 3000 = 90,000 unique proposal IDs per second** into `pending_get_block_proposals`. With `MAX_RELAY_PEERS = 128` peers, the theoretical rate is ~11.5 million entries/second. [5](#0-4) 

**O(N) processing on timer tick — `prune_tx_proposal_request`:**

On every `TX_PROPOSAL_TOKEN` timer tick, the relayer calls `prune_tx_proposal_request`, which:
1. Clones the entire `pending_get_block_proposals` DashMap (O(N) memory allocation)
2. Calls `tx_pool.fetch_txs(all_N_ids)` — a blocking channel call to the tx-pool service with all accumulated IDs
3. Iterates over all N results to build per-peer response batches

```rust

---

### Impact Explanation

- **Memory exhaustion**: Each `ProposalShortId` is 10 bytes; with `HashSet<PeerIndex>` overhead, an attacker can grow the map to hundreds of megabytes before the next timer tick drains it.
- **CPU/latency spike on timer tick**: The `fetch_txs` call with N IDs blocks the tx-pool service for O(N) time, delaying legitimate transaction submission and block assembly for miners.
- **Repeated attack**: Because the map is drained (not permanently corrupted) on each tick, the attacker can sustain the attack indefinitely, causing recurring O(N) spikes every timer interval.
- **No authentication required**: Any connected P2P peer can send `GetBlockProposal` messages. No keys, no stake, no privileged role needed.

---

### Likelihood Explanation

The attack is straightforward: connect as a peer, send `GetBlockProposal` messages at the rate-limit ceiling (30/sec) with 3000 random proposal IDs each. This is a standard P2P relay message that any node accepts from any connected peer. The attacker needs only a single TCP connection to a victim node. The rate limiter does not prevent the accumulation — it only slows the injection rate, which is still 90,000 entries/second per peer.

---

### Recommendation

Apply a hard cap on the total size of `pending_get_block_proposals`. When the cap is reached, either drop new entries or evict the oldest ones. Additionally, consider per-peer caps on how many pending proposal IDs can be cached, analogous to the existing `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` guard applied to `unknown_tx_hashes`: [9](#0-8) 

A similar per-peer and global cap should be enforced inside `insert_get_block_proposals`.

---

### Proof of Concept

1. Connect to a victim CKB node as a P2P peer using the `RelayV3` protocol.
2. In a loop, send `GetBlockProposal` messages, each containing 3000 random `ProposalShortId` values (10 random bytes each) that do not correspond to any real transaction.
3. Send at the rate-limit ceiling: 30 messages/second.
4. After ~10 seconds, `pending_get_block_proposals` contains ~900,000 entries.
5. On the next `TX_PROPOSAL_TOKEN` timer tick, `prune_tx_proposal_request` clones the 900,000-entry map and issues a `fetch_txs` call with 900,000 IDs to the tx-pool service.
6. Observe: elevated CPU usage, memory spike, and delayed tx-pool responses for legitimate users during the timer tick processing.
7. Repeat indefinitely to sustain the degradation.

The per-message limit check at line 40 of `get_block_proposal_process.rs` is bypassed because it only limits a single message, not the total accumulated state across many messages over time. [10](#0-9)

        let get_block_proposals = self.shared().state().drain_get_block_proposals();
        let tx_pool = self.shared.shared().tx_pool_controller();

        let fetch_txs = tx_pool
            .fetch_txs(
                get_block_proposals
                    .iter()
                    .map(|kv_pair| kv_pair.key().clone())
                    .collect(),
            )
            .await;
        if let Err(err) = fetch_txs {
            debug_target!(
                crate::LOG_TARGET_RELAY,
                "relayer prune_tx_proposal_request internal error: {:?}",
                err,
            );
            return;
        }

        let txs = fetch_txs.unwrap();

        let mut peer_txs = HashMap::new();
        for (id, peer_indices) in get_block_proposals.into_iter() {
            if let Some(tx) = txs.get(&id) {
                for peer_index in peer_indices {
                    let tx_set = peer_txs.entry(peer_index).or_insert_with(Vec::new);
                    tx_set.push(tx.clone());

            }

        let mut relay_proposals = Vec::new();
