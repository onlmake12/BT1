### Title
`max_tx_verify_cycles` Not Enforced as Hard Upper Bound in Async Worker Path — (`tx-pool/src/process.rs`, `tx-pool/src/util.rs`)

### Summary
In `_process_tx`, the cycle limit passed to `verify_rtx` (and ultimately to `verify_with_pause`) is `declared_cycles`, not `min(declared_cycles, max_tx_verify_cycles)`. A remote peer that declares cycles equal to the script's actual consumption — even when that value exceeds `max_tx_verify_cycles` — will have its transaction accepted into the pool, because the only post-verification guard checks `declared == actual`, not `actual <= max_tx_verify_cycles`.

### Finding Description
`_process_tx` computes:
```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
``` [6](#0-5) 

This value is passed directly to `verify_rtx` as the VM cycle budget. When a peer relays a transaction with `declared_cycles = 80_000_000` and the script actually consumes exactly 80M cycles, the VM runs to completion within its budget, `verified.cycles = 80M = declared`, and the `DeclaredWrongCycles` guard does not fire. No other guard in `_process_tx` checks whether `declared_cycles > max_tx_verify_cycles`. [3](#0-2) 

The spawn syscall (RFC0044) is one concrete mechanism to construct a script that consumes exactly 80M cycles across child VMs, but any script consuming exactly `declared_cycles > max_tx_verify_cycles` cycles triggers the same path.

### Impact Explanation
Transactions exceeding the operator-configured `max_tx_verify_cycles` enter the pending pool. This causes:
- Excessive CPU consumption during verification (the VM runs for up to `max_block_cycles` instead of `max_tx_verify_cycles`)
- Such transactions cannot be included in a block (block cycle limit is separate), so they permanently occupy pool space
- Block assembly may be degraded if the pool fills with unmineable high-cycle transactions

### Likelihood Explanation
The attack requires only a valid transaction whose script consumes exactly `declared_cycles > max_tx_verify_cycles` cycles. This is straightforward to construct with a loop-counting script or via spawn. No privileged access, hashpower, or key material is needed — only P2P relay access.

### Recommendation
In `_process_tx`, cap `max_cycles` at `max_tx_verify_cycles` when a declared value is present:
```rust
let max_cycles = declared_cycles
    .map(|d| d.min(self.tx_pool_config.max_tx_verify_cycles))
    .unwrap_or_else(|| self.consensus.max_block_cycles());
```
Additionally, add an explicit early rejection:
```rust
if let Some(declared) = declared_cycles {
    if declared > self.tx_pool_config.max_tx_verify_cycles {
        return Some((Err(Reject::ExceededMaximumCycles(declared, self.tx_pool_config.max_tx_verify_cycles)), snapshot));
    }
}
``` [7](#0-6) 

### Proof of Concept
A unit test asserting that a transaction with `declared_cycles = MAX_TX_VERIFY_CYCLES + 1` where the script actually consumes `MAX_TX_VERIFY_CYCLES + 1` cycles is rejected with `ExceededMaximumCycles` when processed through the async worker path (`Worker::process_inner` → `_process_tx`) would demonstrate the missing guard. The existing test infrastructure in `tx-pool/src/component/tests/chunk.rs` already sets `MAX_TX_VERIFY_CYCLES = 70_000_000` and constructs a `TxPoolService` with that config. [8](#0-7)
