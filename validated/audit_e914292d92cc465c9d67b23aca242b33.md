### Title
Fortuna Keeper Loop Freezes Permanently on Persistent `eth_getLogs` RPC Error Due to Missing Adaptive Chunking — (`apps/fortuna/src/keeper/block.rs`)

### Summary

Fortuna's `process_single_block_batch` retries a fixed 100-block `eth_getLogs` query indefinitely on any RPC error, with no logic to reduce the block range or classify the error. If the log response for a 100-block window exceeds the RPC node's response-size limit (typically 10 MB), the keeper thread stalls permanently on that batch, halting all Entropy request fulfillment for the affected chain.

### Finding Description

`process_single_block_batch` in `apps/fortuna/src/keeper/block.rs` issues a single `eth_getLogs` call via `get_request_with_callback_events` and, on any `Err`, sleeps for `RETRY_INTERVAL` (5 s) and retries the **identical** block range forever:

```rust

    let events_res = process_params
        .chain_state
        .contract
        .get_request_with_callback_events(
            block_range.from,
            block_range.to,
            process_params.chain_state.provider_address,
        )
        .await;
    ...
    match events_res {
        Ok(events) => { ... break; }
        Err(e) => {
            tracing::error!("Error while getting events...");
            time::sleep(RETRY_INTERVAL).await;
            // ← no range reduction, no error classification
        }
    }
}

The batch size is a hardcoded constant of 100 blocks:

The underlying RPC call in `get_request_with_callback_events` is a direct `query_with_meta()` with no size-limit handling:

```rust
let res: Vec<(Requested2Filter, LogMeta)> = event.query_with_meta().await?;
``` [3](#0-2) 

The `Requested` (V2) event emitted on each Entropy request contains a dynamic `bytes extraArgs` field alongside three 32-byte indexed topics and three 32-byte non-indexed fields:

```solidity
event Requested(
    address indexed provider,
    address indexed caller,
    uint64 indexed sequenceNumber,
    bytes32 userContribution,
    uint32 gasLimit,
    bytes extraArgs          // ← dynamic, user-supplied
);
``` [4](#0-3) 

Each log entry is at minimum ~450–500 bytes (4 topics × 32 B + ABI-encoded data + log metadata). With a 10 MB RPC limit, approximately 20,000–22,000 events per 100-block window would trigger the limit. On high-throughput L2s (Arbitrum, BNB Chain, Polygon) where blocks are produced every 0.25–3 s, 100 blocks span only 25 s–5 min, making a flood of requests feasible. If `extraArgs` is non-empty and user-controlled, each log grows proportionally, reducing the threshold further.

### Impact Explanation

When the size-limit error is returned by the RPC node, `process_single_block_batch` enters an infinite retry loop on the same 100-block range. Because `process_block_range` calls `process_single_block_batch` sequentially and awaits each call, the entire block-processing thread stalls at that batch. No subsequent blocks are ever processed. All pending Entropy requests in the affected window and all future requests go unfulfilled indefinitely. The keeper must be manually restarted (and the block range manually adjusted) to recover. This constitutes a complete denial-of-service against the Entropy fulfillment service on the affected chain. [5](#0-4) 

### Likelihood Explanation

An unprivileged user can call `requestV2` on the Entropy contract permissionlessly, paying only the provider fee per request. On low-cost L2s (Arbitrum, BNB Chain, Polygon), the fee per request is small and blocks are produced rapidly. A sustained flood of requests over a short window (100 blocks) can push the `eth_getLogs` response over the 10 MB limit. The attack is self-sustaining: once the keeper is stuck, the attacker does not need to continue submitting requests. The keeper remains frozen until manually restarted. [6](#0-5) 

### Recommendation

1. **Classify errors**: Detect "response size limit exceeded" errors (e.g., `-32005` JSON-RPC code or message substrings like `"query returned more than"` / `"result window too large"`) and distinguish them from transient RPC failures.
2. **Adaptive chunking**: On a size-limit error, halve the block range and retry each half independently, combining results — exactly as recommended in H-04.
3. **Hard cap on `BLOCK_BATCH_SIZE`**: Make the batch size configurable per chain and document the relationship between expected event volume and the RPC node's limit.

### Proof of Concept

1. Deploy or use an existing Entropy contract on a low-cost L2 (e.g., Arbitrum).
2. Submit ~25,000 `requestV2` calls within a 100-block window, each paying the minimum provider fee.
3. Observe that the Fortuna keeper's `process_single_block_batch` begins logging `"Error while getting events"` repeatedly for that batch.
4. Confirm that no subsequent Entropy requests (in later blocks) are ever fulfilled — the `process_event_block_number` metric stops advancing.
5. The keeper recovers only after a manual restart with a reduced `backlog_range` that skips the saturated window. [7](#0-6) [8](#0-7) [9](#0-8)

    let BlockRange {
        from: first_block,
        to: last_block,
    } = block_range;
    let mut current_block = first_block;
    while current_block <= last_block {

/// Process a batch of blocks for a chain. It will fetch events for all the blocks in a single call for the provided batch
/// and then try to process them one by one. It checks the `fulfilled_request_cache`. If the request was already fulfilled.
/// It won't reprocess it. If the request was already processed, it will reprocess it.
/// If the process fails, it will retry indefinitely.
#[tracing::instrument(name = "batch", skip_all, fields(
    batch_from_block = block_range.from, batch_to_block = block_range.to
))]

pub async fn process_single_block_batch(block_range: BlockRange, process_params: ProcessParams) {
    let label = ChainIdLabel {
        chain_id: process_params.chain_state.id.clone(),
    };
    loop {
        let events_res = process_params
            .chain_state
            .contract
            .get_request_with_callback_events(
                block_range.from,
                block_range.to,
                process_params.chain_state.provider_address,
            )
            .await;

        // Only update metrics if we successfully retrieved events.
        if events_res.is_ok() {
            // Track the last time blocks were processed. If anything happens to the processing thread, the
            // timestamp will lag, which will trigger an alert.
            let server_timestamp = SystemTime::now()
                .duration_since(UNIX_EPOCH)
                .map(|duration| duration.as_secs() as i64)
                .unwrap_or(0);
            process_params
                .metrics
                .process_event_timestamp
                .get_or_create(&label)
                .set(server_timestamp);

            let current_block = process_params
                .metrics
                .process_event_block_number
                .get_or_create(&label)
                .get();
            if block_range.to > current_block as u64 {
                process_params
                    .metrics
                    .process_event_block_number
                    .get_or_create(&label)
                    .set(block_range.to as i64);
            }
        }

        match events_res {
            Ok(events) => {
                tracing::info!(num_of_events = &events.len(), "Processing",);
                for event in &events {
                    // the write lock guarantees we spawn only one task per sequence number
                    let newly_inserted = process_params
                        .fulfilled_requests_cache
                        .write()
                        .await
                        .insert(event.sequence_number);
                    if newly_inserted {
                        spawn(
                            process_event_with_backoff(event.clone(), process_params.clone())
                                .in_current_span(),
                        );
                    }
                }
                tracing::info!(num_of_events = &events.len(), "Processed",);
                break;
            }
            Err(e) => {
                tracing::error!(
                    "Error while getting events. Waiting for {} seconds before retry. error: {:?}",
                    RETRY_INTERVAL.as_secs(),
                    e
                );
                time::sleep(RETRY_INTERVAL).await;
            }
        }
    }
