Audit Report

## Title
Unbounded `last_n_numbers` via Low `difficulty_boundary` Bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The upfront limit check in `GetLastStateProofProcess::execute` bounds only `last_n_blocks` and `difficulties.len()`, but the actual work performed is determined by `last_n_numbers`, whose size is controlled by the attacker-supplied `difficulty_boundary`. By setting `difficulty_boundary` to the total difficulty of block 1, an attacker forces `last_n_numbers = (1..N)` — N−1 entries — causing O(N log N) synchronous DB reads per single crafted message, completely bypassing `GET_LAST_STATE_PROOF_LIMIT = 1000`.

## Finding Description
The limit check at L201–205 evaluates `difficulties.len() + last_n_blocks * 2`. With `difficulties=[]` and `last_n_blocks=1`, this is `0 + 2 = 2`, well under 1000, so the check passes.

In the else-branch (L291–348), `difficulty_boundary_block_number` is resolved via `get_first_block_total_difficulty_is_not_less_than(start_block_number, last_block_number, &difficulty_boundary)`. When `difficulty_boundary` equals the total difficulty of block 1, this binary search returns block number 1.

The adjustment guard at L313–316 checks `last_block_number - difficulty_boundary_block_number < last_n_blocks`, i.e., `N - 1 < 1`. This is false for any N > 2, so `difficulty_boundary_block_number` remains 1.

`last_n_numbers` is then collected as `(1..last_block_number)` at L318–319, yielding N−1 entries with no size check. `complete_headers` at L356–366 iterates every entry, calling `self.snapshot.chain_root_mmr(*number - 1).get_root()` for each — O(log N) DB reads per block — for a total of O(N log N) DB reads per request. On mainnet with millions of blocks, the `last_n_numbers` Vec alone consumes tens to hundreds of MB of memory, and the subsequent DB reads saturate I/O and CPU on the light-client server thread.

The existing `GET_LAST_STATE_PROOF_LIMIT` constant is never applied to `last_n_numbers.len()`.

## Impact Explanation
A single malicious peer can send one crafted `GetLastStateProof` message that forces the server to allocate an unbounded vector and perform O(N log N) synchronous DB reads before returning. On a mainnet node with millions of blocks, this can exhaust memory and saturate I/O, effectively crashing or hanging the light-client server. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation
The attack requires only two pieces of public information: the current tip block hash (observable from the P2P network) and block 1's total difficulty (a fixed public constant on mainnet). No privileged role, no key material, no PoW, and no victim mistake is required. Any peer can craft and send this message. The condition is deterministic and reproducible on every mainnet node running the light-client server.

## Recommendation
After computing `last_n_numbers` and before calling `complete_headers`, add a hard bound:

```rust
if last_n_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage
        .with_context("last_n_numbers exceeds limit");
}
```

This must be applied in both the if-branch (L296) and the else-branch (L318–319) of the `(sampled_numbers, last_n_numbers)` computation. Alternatively, clamp `difficulty_boundary_block_number` from below before collecting `last_n_numbers`:

```rust
let min_boundary = last_block_number
    .saturating_sub((constant::GET_LAST_STATE_PROOF_LIMIT as u64)
    .saturating_sub(last_n_blocks));
if difficulty_boundary_block_number < min_boundary {
    difficulty_boundary_block_number = min_boundary;
}
```

The fix must be applied **after** the difficulty boundary is resolved and **before** `last_n_numbers` is collected.

## Proof of Concept
1. Observe the current tip hash `H_tip` and block number `N` from the P2P network.
2. Obtain `total_difficulty[1]` (public constant for mainnet block 1).
3. Send:
   ```
   GetLastStateProof {
       last_hash:           H_tip,
       last_n_blocks:       1,
       difficulty_boundary: total_difficulty[1],
       difficulties:        [],
       start_hash:          genesis_hash,
       start_number:        0,
   }
   ```
4. Server resolves `difficulty_boundary_block_number = 1`.
5. Guard `N - 1 < 1` is false → no adjustment.
6. `last_n_numbers = (1..N)`, length = N−1.
7. `complete_headers` iterates N−1 blocks, each performing O(log N) MMR DB reads.
8. Differential test: measure response latency for `difficulty_boundary=total_difficulty[1]` vs `difficulty_boundary=total_difficulty[N-1]`. The former scales as O(N log N); the latter is O(log N).
