### Title
Entropy Provider Can Front-Run User's `request`/`requestWithCallback`/`requestV2` via `setProviderFee()` to Cause Permanent `InsufficientFee` DoS — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

A registered Entropy provider can front-run any user's randomness request by calling `setProviderFee()` (or `register()`) to raise their fee immediately before the user's transaction is mined. Because the fee is read from live storage at execution time and there is no user-supplied maximum-fee guard, the user's transaction reverts with `InsufficientFee`. A malicious provider can repeat this indefinitely to permanently deny randomness to targeted users or to all users of the default provider.

---

### Finding Description

`requestHelper()` reads the provider's current fee from storage at the moment of execution:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [1](#0-0) 

`getFeeV2` resolves to `provider.feeInWei + _state.pythFeeInWei`, where `provider.feeInWei` is the live storage value: [2](#0-1) 

`setProviderFee()` lets any registered provider update `feeInWei` unconditionally and immediately:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
``` [3](#0-2) 

`register()` also updates `feeInWei` and can be used identically: [4](#0-3) 

The same TOCTOU pattern exists in the Echo contract's `requestPriceUpdatesWithCallback`:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [5](#0-4) 

with `Echo.setProviderFee()` as the front-running vector: [6](#0-5) 

**Exploit path:**

1. Provider is registered with `feeInWei = F`.
2. User calls `getFee(provider)` off-chain → observes `F + pythFee`.
3. User broadcasts `requestWithCallback{value: F + pythFee}(provider, userContribution)`.
4. Provider detects the pending transaction in the mempool and broadcasts `setProviderFee(F + 1)` with a higher gas price.
5. `setProviderFee` mines first; `requiredFee` is now `F + 1 + pythFee`.
6. User's transaction reverts with `InsufficientFee`.
7. Provider resets fee to `F` and repeats for every retry.

No special privilege beyond being a registered provider is required. Provider registration is permissionless.

---

### Impact Explanation

- Any registered provider can permanently deny randomness to a targeted user or to all users of the default provider by looping the front-run.
- The user loses gas on every failed attempt with no recourse other than switching providers.
- If the default provider is the attacker, the entire `requestV2()` / `requestWithCallback()` surface is affected for users who rely on `getDefaultProvider()`.
- No ETH is permanently stolen (reverted transactions return ETH), but the service is rendered unavailable — a sustained DoS on the randomness product.

---

### Likelihood Explanation

- Any entity that can call `register()` (permissionless) becomes a provider and gains the ability to execute this attack.
- Front-running is straightforward on EVM chains with a public mempool; the attacker only needs to observe the pending `request*` call and respond with a higher-gas `setProviderFee`.
- The attack is cheap: one `setProviderFee` call per blocked request, and the fee can be reset immediately after.

---

### Recommendation

Add a `maxFee` parameter to `request`, `requestWithCallback`, and `requestV2` so users can specify the maximum fee they are willing to pay, analogous to slippage protection in AMMs:

```solidity
function requestV2(
    address provider,
    bytes32 userContribution,
    uint32 gasLimit,
    uint128 maxFee          // <-- new
) public payable returns (uint64) {
    uint128 requiredFee = getFeeV2(provider, gasLimit);
    if (requiredFee > maxFee) revert EntropyErrors.FeeTooHigh();
    if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
    ...
}
```

Apply the same fix to `Echo.requestPriceUpdatesWithCallback`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache-2.0
pragma solidity ^0.8.0;

// Assumes provider1 is a registered provider with feeInWei = 10
// and pythFeeInWei = 1 (total = 11 wei).

contract FrontRunPoC {
    IEntropy entropy;
    address provider1;

    // Step 1: user observes fee = 11 wei and submits request
    function userRequest() external payable {
        // msg.value == 11 wei (exactly getFee(provider1))
        entropy.requestWithCallback{value: msg.value}(
            provider1,
            bytes32(uint256(42))
        );
        // This reverts with InsufficientFee if provider front-ran
    }

    // Step 2: provider front-runs with higher gas price
    function providerFrontRun() external {
        // provider1 calls setProviderFee(11) → requiredFee becomes 12 wei
        IEntropy(entropy).setProviderFee(11);
        // userRequest() now reverts; provider resets fee afterward
    }
}
```

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.

        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;

        provider.sequenceNumber += 1;

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[

        if (provider.sequenceNumber == 0) {

        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
