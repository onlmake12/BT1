### Title
Stale `row_lock` Re-Read in `script_exists_in_output` Causes Silent Type-Script Deletion on PostgreSQL During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

A copy-paste bug in `script_exists_in_output` causes the function to re-read the `row_lock` result variable instead of `row_type` in its final `match` branch. On PostgreSQL this silently returns `false` for any script that is referenced only as a `type_script_id` (not as a `lock_script_id`) in surviving output rows, causing `rollback_block` to delete that script row from the `script` table even though live outputs still hold a foreign-key reference to it.

---

### Finding Description

`script_exists_in_output` is called during every chain rollback to decide whether a script row can be safely deleted. It performs two SQL `EXISTS` queries:

1. **`row_lock`** — checks `output.lock_script_id = $1` [1](#0-0) 
2. **`row_type`** — checks `output.type_script_id = $1` [2](#0-1) 

If `row_lock` is true the function returns early with `Ok(true)` (correct). [3](#0-2) 

However, the final `match` that is supposed to return the `row_type` result instead reads `row_lock` again:

```rust
// line 252 — BUG: should be row_type.try_get, not row_lock.try_get
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [4](#0-3) 

**PostgreSQL** returns `BOOLEAN` for `EXISTS`, so `row_lock.try_get::<bool, _>(0)` succeeds and returns the already-known-false value from `row_lock`. The `row_type` result is completely discarded. The function always returns `false` when the script is only a `type_script_id` in surviving rows.

**SQLite** returns `BIGINT` for `EXISTS`, so `try_get::<bool, _>(0)` fails, the `Err` arm is taken, and `row_type.get::<i64, _>(0)` is read correctly. The bug is **silent on SQLite**, which is exactly the backend used by all existing rollback tests. [5](#0-4) 

The caller in `rollback_block` collects script IDs to delete based on this return value:

```rust
if let Some(type_script_id) = type_script_id
    && !script_exists_in_output(type_script_id, tx).await?
{
    script_id_list_to_remove.push(type_script_id);
}
``` [6](#0-5) 

Note that the `output` rows for the rolled-back block are deleted **before** these checks run (line 25), so the only surviving references are from outputs in earlier blocks. [7](#0-6) 

---

### Impact Explanation

**Concrete scenario (PostgreSQL deployment):**

1. Block N (surviving) contains an output with `type_script_id = X` and `lock_script_id = Y`.
2. Block N+1 (rolled back) contains an output with the **same** `type_script_id = X` but a different `lock_script_id = Z`.
3. During rollback of block N+1, `script_exists_in_output(X)` is called after block N+1's outputs are deleted.
   - `row_lock` → `EXISTS(lock_script_id = X)` → `false` (X is not a lock script anywhere).
   - Early-return check passes (false), so execution continues.
   - `row_type` → `EXISTS(type_script_id = X)` → `true` (block N's output still references X).
   - Final match re-reads `row_lock` → returns `Ok(false)`.
4. Script row X is added to `script_id_list_to_remove` and deleted.
5. Block N's output now holds a dangling `type_script_id` foreign key.
6. All subsequent `get_cells` / `get_transactions` queries filtered by type script X return empty results or produce NULL-dereference errors, breaking SUDT balance queries, DAO deposit queries, and any other type-script-indexed lookup.

---

### Likelihood Explanation

- Chain reorgs are a normal part of CKB operation and require no special privilege to trigger — any peer that presents a valid longer chain causes the node to roll back.
- The condition (a type script shared between a rolled-back block and a surviving block, where the script is not simultaneously a lock script) is routine for SUDT tokens and DAO cells.
- The bug is completely invisible in the test suite because all rollback tests use SQLite. [8](#0-7) 
- PostgreSQL is the recommended production backend for the rich-indexer.

---

### Recommendation

Change line 252 from `row_lock.try_get` to `row_type.try_get`:

```rust
// BEFORE (buggy)
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}

// AFTER (correct)
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [9](#0-8) 

Additionally, add a PostgreSQL-backed rollback test that verifies a shared type script is **not** deleted when surviving outputs still reference it.

### Proof of Concept

```
1. Start a CKB node with rich-indexer backed by PostgreSQL.
2. Append block_A containing tx_A with:
     output_0: lock=LOCK_A, type=TYPE_X
3. Append block_B (child of block_A) containing tx_B with:
     output_0: lock=LOCK_B, type=TYPE_X   (same TYPE_X, different lock)
4. Trigger rollback of block_B (present a competing chain of equal height).
5. Query: SELECT id FROM script WHERE id = <TYPE_X id>
   → Expected: row exists (TYPE_X still referenced by block_A's output)
   → Actual:   row is gone (deleted by the buggy rollback)
6. Call get_cells(search_key={script: TYPE_X, script_type: type})
   → Expected: returns output_0 from block_A
   → Actual:   returns empty list (script row deleted, JOIN finds nothing)
```
