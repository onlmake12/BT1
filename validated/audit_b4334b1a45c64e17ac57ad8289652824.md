### Title
`DaoCalculator::transaction_maximum_withdraw` reads `header_dep_index` as `u64` while the on-chain C DAO script reads it as `u8`, creating a consensus divergence for DAO withdrawals with index > 255 — (File: `util/dao/src/lib.rs`)

---

### Summary

The Nervos DAO withdrawal protocol requires the transaction sender to encode the `header_deps` array index of the deposit block header inside the witness `input_type` field as an 8-byte little-endian integer. The on-chain C DAO script (`dao.c`) reads only the **lowest byte** (treating the index as `u8`), while the Rust `DaoCalculator::transaction_maximum_withdraw` reads the **full 8-byte `u64`**. For any DAO withdrawal where the encoded index exceeds 255, the two implementations resolve to different `header_deps` entries, producing divergent DAO field computations and enabling a consensus split or incorrect interest accounting.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the deposit-header index from the witness as a full `u64`:

```rust
// dao contract stores header deps index as u64 in the input_type field of WitnessArgs
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

It then uses that value directly as a `usize` index into `header_deps`:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
    .and_then(|hash| header_deps.get(&hash))
    .ok_or(DaoError::InvalidOutPoint)
```

The on-chain C DAO script, however, reads only the **lowest byte** of the same field (treating it as `u8`). This is explicitly documented in the codebase's own test:

```rust
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
// input_type = 257, lowest byte = 1
let witness = WitnessArgs::new_builder()
    .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
    .build();
```

For `input_type = 257` (LE bytes `[0x01, 0x01, 0x00, …]`):
- **C VM** reads byte 0 → index **1** → `header_deps[1]`
- **Rust** reads full u64 → index **257** → `header_deps[257]`

The two implementations resolve to entirely different block headers whenever the encoded index exceeds 255. [1](#0-0) [2](#0-1)

### Impact Explanation

**Consensus split / incorrect DAO field**: `DaoCalculator::dao_field` (called during block assembly and block validation) internally calls `withdrawed_interests` → `transaction_maximum_withdraw`. If a DAO withdrawal transaction with `input_type > 255` is included in a block:

- The C VM (on-chain script execution) resolves the deposit header via the **lowest byte** of the index and computes interest using that header's accumulation rate (`AR`).
- The Rust node resolves the deposit header via the **full u64** index and computes a different `AR`.
- The DAO field written into the block header by the miner (using C-VM-consistent logic) will not match the DAO field the Rust node independently computes during validation, causing the Rust node to reject a block that the C VM considers valid — a **consensus split**.

**DAO interest theft**: An attacker who controls the transaction can place at `header_deps[full_u64_index]` a canonical block at the same height as the deposit block but with a higher `AR` (e.g., a block from a different epoch boundary). The Rust node computes inflated interest using that higher `AR`, while the C VM computes the correct (lower) interest using the actual deposit block. The block's DAO field is then set to the inflated value, allowing the attacker to extract more CKB than they are entitled to.

**Tx-pool denial of service**: The Rust tx-pool calls `DaoCalculator::transaction_fee` before admitting DAO withdrawals. A legitimate user who encodes `input_type > 255` (e.g., because they have more than 255 `header_deps`) will have their valid withdrawal rejected by every Rust node's tx-pool, even though the C VM would accept it. ---

### Likelihood Explanation

The attacker-controlled entry path is a standard DAO withdrawal transaction submitted via the `send_transaction` RPC or relayed over P2P. No privileged role is required. The attacker only needs to:

1. Hold a DAO deposit cell.
2. Construct a withdrawal transaction with `input_type` encoding an index > 255 (e.g., `257u64.to_le_bytes()`).
3. Arrange `header_deps` so that `header_deps[lowest_byte]` is the actual deposit block and `header_deps[full_u64_index]` is a different canonical block at the same height with a higher `AR`.
4. Have the transaction included in a block (either via a cooperative miner or by bypassing the tx-pool).

The protocol imposes no upper bound on the number of `header_deps` in a transaction, so indices > 255 are syntactically valid. The discrepancy is reachable on mainnet by any unprivileged transaction sender. [5](#0-4) [6](#0-5) 

---

### Recommendation

Align the Rust `DaoCalculator` with the on-chain C DAO script by reading only the lowest byte of the `input_type` field when extracting the `header_dep_index`:

```rust
// Match the C DAO script: read only the lowest byte (u8) as the index
let header_dep_index = header_deps_index_data.unwrap()[0] as usize;
```

Alternatively, update the C DAO script to read the full 8-byte little-endian `u64` and enforce that the index fits within the actual `header_deps` length. Whichever direction is chosen, both the C script and the Rust `DaoCalculator` must agree on the same interpretation. Add a consensus-level transaction validation rule that rejects any DAO withdrawal whose `input_type` encodes an index > 255 until the C script is updated, preventing the divergence from being exploited on-chain. [7](#0-6)

### Proof of Concept

The codebase's own test in `util/dao/src/tests.rs` (`check_dao_withdraw_header_dep_index_exceeds_u8`) demonstrates the divergence:

1. `header_deps` is padded to 258 entries.
2. `header_deps[1]` = deposit block (block number 100) — what the C VM resolves.
3. `header_deps[257]` = withdraw block (block number 200) — what the Rust code resolves.
4. `input_type = 257u64` (LE: `[0x01, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`); lowest byte = 1.
5. The Rust `DaoCalculator` resolves index 257 → withdraw block (number 200), which does not match the cell data (deposited at block 100) → returns `Err`.
6. The C VM resolves index 1 → deposit block (number 100) → matches cell data → **accepts**.

For the interest-theft variant, replace `header_deps[257]` with a canonical block at height 100 that has a higher `AR` than the actual deposit block. The Rust node then computes inflated interest and writes an inflated DAO field into the block header, while the C VM computes the correct (lower) interest — a direct theft of DAO funds from the protocol's secondary issuance pool.     fn transaction_maximum_withdraw(
        &self,
        rtx: &ResolvedTransaction,
    ) -> Result<Capacity, DaoError> {
        let header_deps: HashSet<Byte32> = rtx.transaction.header_deps_iter().collect();
        rtx.resolved_inputs.iter().enumerate().try_fold(
            Capacity::zero(),
            |capacities, (i, cell_meta)| {
                let capacity: Result<Capacity, DaoError> = {
                    let output = &cell_meta.cell_output;
                    let is_dao_type_script = |type_script: Script| {
                        Into::<u8>::into(type_script.hash_type())
                            == Into::<u8>::into(ScriptHashType::Type)
                            && type_script.code_hash() == self.consensus.dao_type_hash()
                    };
                    let is_dao_output = output
                        .type_()
                        .to_opt()
                        .map(is_dao_type_script)
                        .unwrap_or(false);
                    if is_dao_output {
                        // A withdrawing DAO cell has 8 bytes of cell data storing the
                        // block number of the original deposit.
                        let deposited_block_number =
                            match self.data_loader.load_cell_data(cell_meta) {
                                Some(data) if data.len() == 8 => LittleEndian::read_u64(&data),
                                _ => 0,
                            };
                        if deposited_block_number > 0 {
                            let withdrawing_header_hash = cell_meta
                                .transaction_info
                                .as_ref()
                                .map(|info| &info.block_hash)
                                .filter(|hash| header_deps.contains(hash))
                                .ok_or(DaoError::InvalidOutPoint)?;
                            let deposit_header_hash = rtx
                                .transaction
                                .witnesses()
                                .get(i)
                                .ok_or(DaoError::InvalidOutPoint)
                                .and_then(|witness_data| {
                                    // dao contract stores header deps index as u64 in the input_type field of WitnessArgs
                                    let witness =
                                        WitnessArgs::from_slice(&Into::<Bytes>::into(witness_data))
                                            .map_err(|_| DaoError::InvalidDaoFormat)?;
                                    let header_deps_index_data: Option<Bytes> =
                                        witness.input_type().to_opt().map(|witness| witness.into());
                                    if header_deps_index_data.is_none()
                                        || header_deps_index_data.clone().map(|data| data.len())
                                            != Some(8)
                                    {
                                        return Err(DaoError::InvalidDaoFormat);
                                    }
                                    Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
                                })
                                .and_then(|header_dep_index| {
                                    rtx.transaction
                                        .header_deps()
                                        .get(header_dep_index as usize)
                                        .and_then(|hash| header_deps.get(&hash))
                                        .ok_or(DaoError::InvalidOutPoint)
                                })?;

                            let deposit_header = self
                                .data_loader
                                .get_header(deposit_header_hash)
                                .ok_or(DaoError::InvalidHeader)?;
                            if deposit_header.number() != deposited_block_number {
                                return Err(DaoError::InvalidOutPoint);
                            }
                            self.calculate_maximum_withdraw(
                                output,
                                Capacity::bytes(cell_meta.data_bytes as usize)?,
                                deposit_header_hash,
                                withdrawing_header_hash,
                            )
                        } else {
                            Ok(output.capacity().into())
                        }
                    } else {
                        Ok(output.capacity().into())
                    }
                };
                capacity.and_then(|c| c.safe_add(capacities).map_err(Into::into))
            },
        )
    }

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
