### Title
Delegator Rewards Permanently Locked When Publisher Is Removed From Integrity Pool - (`governance/pyth_staking_sdk/src/utils/pool.ts`, `governance/pyth_staking_sdk/src/pyth-staking-client.ts`)

---

### Summary

When a publisher is removed from the OIS integrity pool (their slot in `pool_data.publishers` is zeroed to `PublicKey.default`), any delegator who had staked to that publisher loses all unclaimed rewards permanently. The `advance_delegation_record` instruction — the only mechanism to settle and distribute rewards — can no longer be invoked for a removed publisher, leaving the delegator's `delegation_record` PDA frozen at a stale epoch and their accrued rewards unclaimable.

---

### Finding Description

The OIS integrity pool tracks active publishers in a fixed-size array `pool_data.publishers`. The client-side helper `extractPublisherData` filters this array to only return entries where the pubkey is not `PublicKey.default`:

```ts
// governance/pyth_staking_sdk/src/utils/pool.ts
return poolData.publishers
  .filter((publisher) => !publisher.equals(PublicKey.default))
  ...
```

`getAdvanceDelegationRecordInstructions` then builds `advance_delegation_record` instructions **only** for publishers present in this filtered active list:

```ts
// governance/pyth_staking_sdk/src/pyth-staking-client.ts
const allPublishers = extractPublisherData(poolData);
const publishers = allPublishers
  .map(...)
  .filter(({ lowestEpoch }) => lowestEpoch !== undefined);
...
const advanceDelegationRecordInstructions = await Promise.all(
  filteredPublishers.map(({ pubkey, stakeAccount }) =>
    this.integrityPoolProgram.methods
      .advanceDelegationRecord()
      ...
  ),
);
```

The on-chain `advance_delegation_record` instruction validates the `publisher` account against `pool_data` (per the IDL: `"CHECK : The publisher will be checked against data in the pool_data"`). Once a publisher's slot is zeroed, this check fails for that publisher key, making it impossible to advance the delegation record for any delegator who was staked to that publisher.

The `delegation_record` PDA (keyed by `[publisher, stake_account_positions]`) persists on-chain with a stale `last_epoch`. The delegator's positions in `stake_account_positions` still reference the removed publisher's pubkey. There is no fallback path to settle these rewards or advance the record.

**Scenario:**
1. Delegator D stakes to Publisher P. Rewards accrue over several epochs.
2. Publisher P is removed from the pool (slot zeroed in `pool_data.publishers`).
3. D calls `advance_delegation_record` — the instruction fails because P is no longer in `pool_data`.
4. D's `delegation_record` is permanently stuck. Rewards earned while P was active are unclaimable.
5. D cannot undelegate cleanly either, since `undelegate` also requires the publisher to be validated against `pool_data`.

---

### Impact Explanation

Delegators who had active positions with a removed publisher permanently lose all unclaimed OIS rewards accrued up to the epoch of removal. The `pool_reward_custody` retains those tokens with no mechanism to redistribute or recover them. This is a direct, irreversible loss of user funds (PYTH tokens) for any delegator affected by a publisher removal event.

---

### Likelihood Explanation

Publisher removal is a realistic governance/operational event — publishers can be deregistered, reassigned, or opt out. The `set_publisher_stake_account` instruction (callable by the publisher or reward authority) can zero out a publisher's stake account, and the pool data can be updated to remove a publisher entry. Any delegator who did not call `advance_delegation_record` before the removal loses their pending rewards. Since `advance_delegation_record` is not automatically triggered on removal, this is a silent loss that users may not notice until they attempt to claim.

---

### Recommendation

Before zeroing a publisher's slot in `pool_data.publishers`, the protocol should either:
1. Force-settle all outstanding `delegation_record` accounts for that publisher (similar to MarinateV2's fix of sending rewards before whitelist removal), or
2. Retain the publisher entry in a "tombstoned" state that still allows `advance_delegation_record` to be called up to the removal epoch, then prevents new delegation but permits reward settlement.

Additionally, `undelegate` should be callable for removed publishers so delegators can exit their positions.

---

### Proof of Concept

```
Epoch N:   Delegator D stakes 1000 PYTH to Publisher P.
           delegation_record[P][D].last_epoch = N

Epoch N+5: Publisher P is removed (pool_data.publishers[i] = PublicKey::default).
           delegation_record[P][D].last_epoch still = N  (5 epochs of rewards unclaimed)

Epoch N+6: D calls advance_delegation_record(publisher=P):
           → on-chain check: P not found in pool_data.publishers → ERROR PublisherNotFound
           → delegation_record[P][D].last_epoch remains N forever
           → 5 epochs of rewards permanently locked in pool_reward_custody
