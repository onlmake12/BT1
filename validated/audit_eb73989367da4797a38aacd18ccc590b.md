Audit Report

## Title
Remote Peer Can Inflate `task_count` to 128 and Lock Block Slots in `InflightBlocks` for Up to 30 Seconds — (File: `sync/src/types/mod.rs`)

## Summary

A remote sync peer can inflate its `DownloadScheduler::task_count` to `MAX_BLOCKS_IN_TRANSIT_PER_PEER` (128) by responding to block requests within `fast_time` (default 1000 ms), then stop responding while staying connected. This locks up to 128 `BlockNumberAndHash` slots in `InflightBlocks::inflight_states`, preventing any other peer from being asked for those same blocks for up to 30 seconds per `BLOCK_DOWNLOAD_TIMEOUT`, with no fallback re-request path. All code claims in the submission are confirmed by direct inspection.

## Finding Description

**Inflation path — confirmed:**

`remove_by_block` at lines 797–811 of `sync/src/types/mod.rs` calls `set.increase(2)` unconditionally when `time_analyzer.push_time(elapsed)` returns `TimeQuantile::MinToFast` (elapsed ≤ `fast_time`). The `should_punish` guard only wraps the `NormalToUpper` and `UpperToMax` (decrease) branches, not the increase branches:

```rust
TimeQuantile::MinToFast => set.increase(2),          // no guard
TimeQuantile::FastToNormal => set.increase(1),        // no guard
TimeQuantile::NormalToUpper => {
    if should_punish { set.decrease(1) }              // guarded
}
TimeQuantile::UpperToMax => {
    if should_punish { set.decrease(2) }              // guarded
}
```

`increase` caps at `MAX_BLOCKS_IN_TRANSIT_PER_PEER`. Starting from `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32), 48 fast responses suffice: (128 − 32) / 2 = 48.

`fast_time` defaults to 1000 ms (line 449), not 750 ms as stated in the original report title — the submission correctly identifies this discrepancy, which does not affect exploitability.

**Slot locking — confirmed:**

`InflightBlocks::insert` at lines 748–751 uses a global `BTreeMap<BlockNumberAndHash, InflightState>`. If the key is already present, it returns `false` immediately with no fallback:

```rust
Entry::Occupied(_entry) => return false,
```

`BlockFetcher::fetch` at lines 271–276 only adds a block to the fetch list when `insert` returns `true`. A `false` return silently skips the block — no alternative peer is tried.

**Timeout and prune limitation — confirmed:**

`prune` at line 666 evicts entries only where `value.timestamp + BLOCK_DOWNLOAD_TIMEOUT < now`. The loop at lines 661–664 breaks at `key.number > tip + 20`, so inflight entries for blocks beyond `tip + 20` are never pruned until the tip advances — which it cannot do if those blocks are locked. This makes the effective stall potentially unbounded for blocks above `tip + 20`.

**`can_fetch` exhaustion — confirmed:**

`can_fetch` at lines 504–506 returns `task_count.saturating_sub(hashes.len())`. With `task_count=128` and 128 hashes inflight, `can_fetch=0`. `peer_can_fetch_count` at lines 613–618 delegates to this, and `BlockFetcher::fetch` at lines 219–222 caps `n_fetch` to `peer_can_fetch_count`, so no further requests are issued to the malicious peer.

**Existing mitigations — confirmed insufficient:**

- `should_punish` (line 786) guards only the decrease branches, not the increase branches.
- `remove_by_peer` (lines 766–783) clears inflight blocks on disconnect, but the attacker need not disconnect — staying connected while not responding is sufficient to hold slots.
- `protect_num` (lines 649, 786) suppresses punishment when fewer than `MAX_OUTBOUND_PEERS_TO_PROTECT_FROM_DISCONNECT` peers are connected, making the attack easier in low-peer-count scenarios.
- `time_analyzer` is a single shared field in `InflightBlocks` (line 540), not per-peer, so a fast attacker shifts the `fast_time` baseline downward for all peers over time.

## Impact Explanation

Block synchronization for up to 128 specific block hashes stalls for at least 30 seconds (and potentially longer for blocks above `tip + 20`). Multiple coordinated malicious peers can cover disjoint block ranges, amplifying the stall across the victim's entire sync window. This concretely matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** — **High severity (10001–15000 points)**.

## Likelihood Explanation

The attacker requires only a standard P2P sync connection and the ability to respond to 48 block requests within 1000 ms each. This is trivially achievable on any low-latency or co-located network. No PoW, key material, or privileged access is required. The attack is repeatable: after the 30-second timeout expires for entries within `tip + 20`, the attacker can reconnect and repeat. The shared `time_analyzer` means a single fast attacker can also shift the `fast_time` baseline downward over time, making the threshold easier to beat for subsequent rounds.

## Recommendation

1. Add a `should_punish`-style guard or a per-checkpoint cap on `task_count` increases, preventing a single peer from reaching `MAX_BLOCKS_IN_TRANSIT_PER_PEER` without sustained good behavior across multiple checkpoint windows.
2. Allow re-requesting a block from a different peer after a shorter unresponsiveness interval (e.g., `low_time`) rather than waiting the full `BLOCK_DOWNLOAD_TIMEOUT` (30 s).
3. Make `time_analyzer` per-peer rather than global to prevent one fast peer from skewing quantile thresholds for all peers.
4. Extend `prune` to cover the full inflight range (not just `tip + 20`) so that timed-out entries beyond that range are also evicted promptly.

## Proof of Concept

1. Connect a malicious peer to the victim node and advertise a `best_known_header` ahead of the victim's tip.
2. Respond to every `GetBlocks` request within <1000 ms with valid blocks. After 48 responses, `task_count` reaches 128 (`INIT=32`, `increase(2)` × 48 = 96, `32+96=128`).
3. Once the victim has 128 blocks inflight to the malicious peer, stop responding while remaining connected.
4. Observe that `peer_can_fetch_count` returns 0 for the malicious peer and that `InflightBlocks::insert` returns `false` for all 128 locked hashes, preventing honest peers from being asked for those blocks.
5. After 30 seconds, `prune` evicts entries within `tip + 20`; entries beyond that range remain locked until the tip advances.
6. Unit test: insert 128 entries for a peer into `InflightBlocks`, call `insert` for the same hashes from a second peer index, assert all return `false`, then assert they remain `false` until `unix_time_as_millis()` advances past `timestamp + BLOCK_DOWNLOAD_TIMEOUT`.
