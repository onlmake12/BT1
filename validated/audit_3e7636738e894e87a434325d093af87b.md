### Title
Incorrect Guard Condition in `transaction_maximum_withdraw` Silently Skips Interest for Genesis-Block DAO Deposits, Causing Withdrawal Rejection - (File: `util/dao/src/lib.rs`)

### Summary
`DaoCalculator::transaction_maximum_withdraw` uses `deposited_block_number > 0` to distinguish a withdrawing DAO cell from a deposit cell. This guard incorrectly treats a DAO withdrawal cell whose original deposit was committed in block 0 (the genesis block) as a plain deposit cell, skipping interest calculation entirely. The resulting underestimated `maximum_withdraw` causes `transaction_fee` to produce a `DaoError::Overflow` (via `safe_sub` underflow), which the tx pool maps to `Reject::Malformed`, permanently blocking the withdrawal transaction from entering the pool.

### Finding Description
In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` reads the 8-byte cell data of a DAO-type input cell and interprets it as the block number at which the original deposit was made:

```rust
let deposited_block_number =
    match self.data_loader.load_cell_data(cell_meta) {
        Some(data) if data.len() == 8 => LittleEndian::read_u64(&data),
        _ => 0,
    };
if deposited_block_number > 0 {
    // ... interest calculation path (withdrawal phase 2)
} else {
    Ok(output.capacity().into())   // treated as deposit cell
}
```

The intent is to distinguish:
- **Deposit cell** (phase 0 → 1): cell data is all zeros → `deposited_block_number == 0`
- **Withdrawal cell** (phase 1 → 2): cell data stores the deposit block number → `deposited_block_number > 0`

The guard `> 0` conflates two distinct cases: (a) data is all zeros (genuine deposit cell) and (b) data encodes block number `0` (a withdrawal cell whose deposit was in the genesis block). When `deposited_block_number == 0` because the deposit was in block 0, the code falls into the `else` branch and returns `output.capacity().into()` — the original deposit amount with no interest.

The correct discriminant is used elsewhere in the codebase: `DaoScriptSizeVerifier` at `verification/src/transaction_verifier.rs` line 868 checks `input_data.into_iter().any(|b| b != 0)` to identify deposit cells by all-zero data, which correctly handles a block-0 deposit.

### Impact Explanation
When a user submits a DAO withdrawal (phase 2) transaction for a genesis-block deposit via the `send_transaction` RPC:

1. `check_tx_fee` in `tx-pool/src/util.rs` calls `DaoCalculator::transaction_fee`.
2. `transaction_fee` calls `transaction_maximum_withdraw`, which returns `output.capacity()` (no interest) instead of the correct interest-bearing amount.
3. `transaction_fee` then computes `maximum_withdraw.safe_sub(outputs_capacity)`. Since the user's output includes earned interest (outputs_capacity > original deposit), `safe_sub` underflows and returns `DaoError::Overflow`.
4. `check_tx_fee` maps this to `Reject::Malformed(…, "expect (outputs capacity) <= (inputs capacity)")`.
5. The transaction is permanently rejected from the tx pool. The user cannot claim their DAO interest.

The on-chain DAO C script would compute the correct interest and accept the transaction, but the Rust-side fee gate blocks it before script execution ever runs.

### Likelihood Explanation
A DAO deposit in block 0 requires a genesis-issued cell carrying the DAO type script. While the standard CKB genesis does not include such cells, the code path is reachable: any chain spec that issues a cell with the DAO type script in the genesis block (e.g., a custom devnet or a future mainnet upgrade) would trigger this bug. The condition is a clear logic error — the correct check is whether the 8 bytes are all zero, not whether the decoded integer is nonzero.

### Recommendation
Replace the numeric guard with an explicit all-zero check, mirroring the approach in `DaoScriptSizeVerifier`:

```rust
// Before
if deposited_block_number > 0 {

// After
let is_withdrawal_cell = match self.data_loader.load_cell_data(cell_meta) {
    Some(ref data) if data.len() == 8 => data.iter().any(|b| *b != 0),
    _ => false,
};
if is_withdrawal_cell {
```

This correctly distinguishes a deposit cell (all-zero data) from a withdrawal cell whose deposit block number happens to be 0.
