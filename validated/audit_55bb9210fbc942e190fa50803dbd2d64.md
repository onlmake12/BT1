### Title
Protocol and Provider Fees Are Immediately and Irrevocably Credited at Request Time With No Refund Path for Unfulfilled Requests — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, both the provider fee and the Pyth protocol fee are fully credited to their respective accumulators at the moment a randomness request is created, before any service is rendered. No cancellation or refund function exists anywhere in the contract. If a provider fails to fulfill a request — or if the caller overpays — all funds are permanently lost to the user with no on-chain recourse.

---

### Finding Description

Inside `requestHelper`, the fee accounting is performed unconditionally and immediately:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;                                    // line 237
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);   // lines 238-239
``` [1](#0-0) 

At this point the provider's `accruedFeesInWei` and the protocol's `accruedPythFeesInWei` have both been incremented. The request is stored, but no fulfillment has occurred.

The `request()` and `requestWithCallback()` / `requestV2()` entry points carry an explicit acknowledgement of the non-refund policy:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) 

A complete audit of the contract confirms there is no `cancel`, `refund`, or `withdrawRequest` function. The only resolution paths are `reveal` and `revealWithCallback`, both of which require the provider to supply a valid hash-chain preimage. [3](#0-2) [4](#0-3) 

The `allocRequest` comment further acknowledges that unfulfilled requests can persist indefinitely:

> "There is a chance that some requests never get revealed and remain active forever." [5](#0-4) 

---

### Impact Explanation

**Permanent loss of user funds in two distinct scenarios:**

1. **Provider non-fulfillment.** A provider that goes offline, is deregistered, or selectively ignores requests leaves the user's full fee (provider portion + Pyth portion) permanently credited to the provider and protocol with no on-chain recovery path. The user paid for a random number they will never receive.

2. **Overpayment is silently absorbed.** Because `msg.value - providerFee` is credited to `accruedPythFeesInWei` rather than the exact `pythFeeInWei`, any overpayment (e.g., from a fee estimate race condition between `getFeeV2` and the actual transaction) is permanently transferred to the Pyth protocol treasury. The user has no way to recover the surplus.

In both cases the funds are not frozen — they are immediately withdrawable by the provider (`withdraw`) or the Pyth admin (`withdrawFees` / `withdrawAsFeeManager`). The loss is therefore one-sided: the protocol and provider are overpaid relative to the service actually rendered. [6](#0-5) 

---

### Likelihood Explanation

- Any registered provider can stop fulfilling requests at any time; there is no on-chain slashing or bond mechanism that forces fulfillment.
- Fee changes between a user's `getFeeV2` call and their `request` transaction are a normal race condition on any live network, causing routine overpayment.
- The `allocRequest` comment explicitly acknowledges that permanently unfulfilled requests are an expected operational reality.
- An unprivileged user triggers this path simply by calling `requestWithCallback` or `requestV2` — no special role required.

---

### Recommendation

1. **Introduce a user-callable cancellation function** that, after a configurable timeout (e.g., N blocks with no fulfillment), allows the original requester to reclaim the provider fee and the Pyth fee, crediting them back from the respective accumulators.
2. **Refund excess `msg.value`** by crediting exactly `_state.pythFeeInWei` to `accruedPythFeesInWei` and returning `msg.value - requiredFee` to `msg.sender` when `msg.value > requiredFee`.
3. Alternatively, hold fees in escrow within the request struct and only release them to the provider/protocol upon successful fulfillment, mirroring the approach already used for the provider fee in the `Echo` contract. [7](#0-6) 

---

### Proof of Concept

```
1. Alice calls requestV2(provider, userContribution, gasLimit) with msg.value = getFeeV2(provider, gasLimit).
   → requestHelper executes:
       providerInfo.accruedFeesInWei += providerFee          // provider credited immediately
       accruedPythFeesInWei += (msg.value - providerFee)     // Pyth credited immediately
   → Request stored on-chain.

2. Provider goes offline (or deliberately ignores Alice's sequence number).
   → revealWithCallback is never called.
   → Alice's request remains active forever (acknowledged in allocRequest comment).

3. Alice has no cancel/refund function to call.
   → Provider withdraws their accrued fees via withdraw().
   → Pyth admin withdraws protocol fees via withdrawFees().
   → Alice has permanently lost her full payment for a random number she never received.

Overpayment variant:
1. Alice calls getFeeV2() → 100 wei.
2. Fee increases to 110 wei before her tx lands.
3. Alice sends 110 wei; requiredFee = 110 wei; msg.value == requiredFee, no revert.
   OR Alice sends 120 wei; msg.value - providerFee = 120 - providerFee, all excess goes to Pyth.
4. No refund is issued; excess is permanently credited to accruedPythFeesInWei.
```
