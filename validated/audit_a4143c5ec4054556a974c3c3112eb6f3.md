### Title
Wrong Column Traversal in `migrate_transaction_info` Corrupts `COLUMN_TRANSACTION_INFO` on Database Upgrade - (File: `util/migrate/src/migrations/table_to_struct.rs`)

---

### Summary

The `ChangeMoleculeTableToStruct` database migration (v0.35.0, version `20200703124523`) contains a copy-paste error in `migrate_transaction_info`: it iterates over `COLUMN_UNCLES` instead of `COLUMN_TRANSACTION_INFO`. This is a direct analog to the reported storage-collision class: a schema-layout change is applied to the wrong data store, causing the target column to remain in the old (incompatible) binary format while foreign data from a different column is injected into it. Any node upgrading from a pre-v0.35.0 database runs this migration and ends up with a permanently corrupted `COLUMN_TRANSACTION_INFO`, causing all subsequent reads of historical transaction metadata to silently return garbage field values.

---

### Finding Description

In `util/migrate/src/migrations/table_to_struct.rs`, the function `migrate_transaction_info` is supposed to convert every record in `COLUMN_TRANSACTION_INFO` from the old molecule `table` encoding (68 bytes, with a 16-byte offset header) to the new molecule `struct` encoding (52 bytes, no header). The closure strips the first 16 bytes and writes the result back to `COLUMN_TRANSACTION_INFO`.

The bug: the `db.traverse(...)` call on line 93 passes `COLUMN_UNCLES` as the source column instead of `COLUMN_TRANSACTION_INFO`:

```rust
// util/migrate/src/migrations/table_to_struct.rs  lines 77-101
fn migrate_transaction_info(&self, db: &RocksDB) -> Result<()> {
    const TRANSACTION_INFO_SIZE: usize = 52;
    ...
    let mut transaction_info_migration = |key: &[u8], value: &[u8]| -> Result<()> {
        if value.len() != TRANSACTION_INFO_SIZE {
            wb.put(COLUMN_TRANSACTION_INFO, key, &value[16..])?;  // writes to tx-info col
        }
        Ok(())
    };
    ...
    let (_count, nk) =
        db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;  // reads from UNCLES col
    ...
}
``` [1](#0-0) 

Compare with the correct pattern used by every other sub-migration in the same file, e.g. `migrate_header` which correctly traverses `COLUMN_BLOCK_HEADER` and writes back to `COLUMN_BLOCK_HEADER`: [2](#0-1) 

Two concrete effects result:

**Effect 1 – `COLUMN_TRANSACTION_INFO` is never migrated.**
All pre-migration transaction-info records remain in the old 68-byte molecule `table` format. The new code reads them with `packed::TransactionInfoReader::from_slice_should_be_ok`, which is a fixed-size struct reader that blindly interprets the first 52 bytes: [3](#0-2) 

The old 68-byte layout is:
```
bytes  0- 3: total_size  = 0x44 (68)
bytes  4- 7: offset[0]   = 0x10 (16)
bytes  8-11: offset[1]   = 0x18 (24)
bytes 12-15: offset[2]   = 0x20 (32)
bytes 16-23: block_number (actual u64)
bytes 24-31: block_epoch  (actual u64)
bytes 32-67: key          (block_hash 32B + index 4B)
```

The new struct reader interprets the first 52 bytes as:
```
block_number ← bytes 0-7  = 0x1000000044000000  (garbage ~1.15×10¹⁸)
block_epoch  ← bytes 8-15 = 0x2000000018000000  (garbage ~2.31×10¹⁸)
key          ← bytes 16-51 = actual block_number || actual block_epoch || first 20B of block_hash
```

`block_hash` inside `key` is therefore truncated and wrong; `block_number` and `block_epoch` are completely wrong. [4](#0-3) 

**Effect 2 – Uncle header data is injected into `COLUMN_TRANSACTION_INFO`.**
Uncle records stored in `COLUMN_UNCLES` are keyed by uncle hash (32 bytes) and valued by a serialized `HeaderView` (variable size, typically ≠ 52 bytes). The closure strips 16 bytes and writes the remainder into `COLUMN_TRANSACTION_INFO` under the uncle-hash key. This pollutes the transaction-info index with entries whose keys are uncle hashes, not transaction hashes. [5](#0-4) 

---

### Impact Explanation

**Corrupted `TransactionInfo` silently breaks relative time-lock (`since`) verification.**

`verify_relative_lock` in `verification/src/transaction_verifier.rs` reads `info.block_number` and `info.block_epoch` from the `TransactionInfo` stored in `COLUMN_TRANSACTION_INFO` to enforce relative time-locks on cell inputs: [6](#0-5) 

After migration, `info.block_number` for any pre-migration cell is read as `~1.15×10¹⁸`. The `checked_add` on line 682 overflows and returns `TransactionError::InvalidSince`, causing every transaction that spends a pre-migration cell with a relative block-number `since` lock to be **permanently and incorrectly rejected** by upgraded nodes.

For epoch-based `since`, `info.block_epoch` is similarly garbage (`~2.31×10¹⁸`), causing `b = info.block_epoch.to_rational() + delta` to be astronomically large, so `a < b` is always true and the transaction is always rejected as `Immature`.

For timestamp-based `since`, `info.block_hash` is truncated/wrong, so `get_header_fields(&info.block_hash)` either panics (`expect("header exist")`) or returns a wrong timestamp, producing an incorrect `base_timestamp` for the lock calculation.

**Secondary impact – `get_transaction_with_info` returns wrong or no transaction.**

`get_transaction_with_info` uses `tx_info.block_hash` (corrupted) to look up the transaction body in `COLUMN_BLOCK_BODY`. The lookup key `tx_info.key().as_slice()` encodes the wrong block hash and wrong index, so the body lookup returns `None`, making the transaction appear non-existent to callers such as the RPC `get_transaction` endpoint. [7](#0-6) 

**Consensus split between upgraded and non-upgraded nodes.**

Nodes that ran this migration reject transactions spending pre-migration cells with relative `since` locks. Nodes that have not yet run it (or fresh nodes that synced from genesis) accept the same transactions. This creates a consensus split: a block containing such a transaction is valid on non-migrated nodes and invalid on migrated nodes.

---

### Likelihood Explanation

- The migration runs automatically on any node upgrading from a database version older than `20200703124523` (pre-v0.35.0). The migration code is still present and active in the current codebase.
- The `ChangeMoleculeTableToStruct` migration is registered unconditionally in `Migrate::new` and is applied whenever the stored DB version is below its version string.
- Any operator who kept a pre-v0.35.0 database and upgraded to a later binary would have silently corrupted their `COLUMN_TRANSACTION_INFO`. The node continues to start and operate normally; there is no error or warning.
- A transaction sender who holds a cell created before the migration and uses a relative `since` lock can trigger the rejection path deterministically by submitting a valid (by protocol rules) transaction. [8](#0-7) 

---

### Recommendation

1. **Fix the traversal column**: Change `COLUMN_UNCLES` to `COLUMN_TRANSACTION_INFO` on line 93 of `util/migrate/src/migrations/table_to_struct.rs`.

2. **Add a follow-up migration** to detect and repair nodes that already ran the broken migration. The repair migration should: (a) delete all entries in `COLUMN_TRANSACTION_INFO` whose key length is 32 bytes but whose value does not decode as a valid `TransactionInfo` struct, and (b) re-derive correct `TransactionInfo` records by replaying `attach_block` for all historical blocks.

3. **Add a CI test** that runs `ChangeMoleculeTableToStruct` against a synthetic pre-migration database and asserts that `COLUMN_TRANSACTION_INFO` records are correctly converted and that `COLUMN_UNCLES` records are unchanged.

---

### Proof of Concept

**Step 1 – Construct a pre-migration database** containing at least one block with a transaction and at least one uncle. Write the transaction info in the old molecule `table` format (68 bytes) and the uncle header in the old `table` format (variable, ≠ 52 bytes).

**Step 2 – Run `ChangeMoleculeTableToStruct::migrate`.**

**Step 3 – Read `COLUMN_TRANSACTION_INFO`** for the known transaction hash. Observe:
- The value is still 68 bytes (not migrated).
- `packed::TransactionInfoReader::from_slice_should_be_ok` reads bytes 0–7 as `block_number` = `0x1000000044000000` ≈ 1.15×10¹⁸.
- `block_epoch` = `0x2000000018000000` ≈ 2.31×10¹⁸.
- `block_hash` inside `key` = first 20 bytes of the actual block hash concatenated with garbage.

**Step 4 – Attempt to verify a transaction** that spends the pre-migration cell with `since = 0x2000000000000001` (relative block-number lock of 1). `verify_relative_lock` computes `required_block_number = 1.15×10¹⁸ + 1`, which overflows `checked_add` and returns `TransactionError::InvalidSince`, rejecting the transaction even though the lock is satisfied by protocol rules.

**Step 5 – Read `COLUMN_TRANSACTION_INFO`** for the uncle hash. Observe a spurious entry: the uncle's `HeaderView` bytes with the first 16 bytes stripped, written under the uncle-hash key — data that should never appear in the transaction-info column. [1](#0-0) [3](#0-2) [9](#0-8)

    fn verify_relative_lock(
        &self,
        index: usize,
        since: Since,
        cell_meta: &CellMeta,
    ) -> Result<(), Error> {
        if since.is_relative() {
            let info = match cell_meta.transaction_info {
                Some(ref transaction_info) => Ok(transaction_info),
                None => Err(TransactionError::Immature { index }),
            }?;
            match since.extract_metric() {
                Some(SinceMetric::BlockNumber(block_number)) => {
                    let proposal_window = self.consensus.tx_proposal_window();
                    let required_block_number = info
                        .block_number
                        .checked_add(block_number)
                        .ok_or(TransactionError::InvalidSince { index })?;
                    if self.tx_env.block_number(proposal_window) < required_block_number {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                Some(SinceMetric::EpochNumberWithFraction(epoch_number_with_fraction)) => {
                    if !epoch_number_with_fraction.is_well_formed_increment() {
                        return Err((TransactionError::InvalidSince { index }).into());
                    }
                    let a = self.tx_env.epoch().to_rational();
                    let b = info.block_epoch.to_rational()
                        + epoch_number_with_fraction.normalize().to_rational();
                    if a < b {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                Some(SinceMetric::Timestamp(timestamp)) => {
                    // pass_median_time(current_block) starts with tip block, which is the
                    // parent of current block.
                    // pass_median_time(input_cell's block) starts with cell_block_number - 1,
                    // which is the parent of input_cell's block
                    let proposal_window = self.consensus.tx_proposal_window();
                    let parent_hash = self.tx_env.parent_hash();
                    let epoch_number = self.tx_env.epoch_number(proposal_window);
                    let hardfork_switch = self.consensus.hardfork_switch();
                    let base_timestamp = if hardfork_switch
                        .ckb2021
                        .is_block_ts_as_relative_since_start_enabled(epoch_number)
                    {
                        self.data_loader
                            .get_header_fields(&info.block_hash)
                            .expect("header exist")
                            .timestamp
                    } else {
                        self.parent_median_time(&info.block_hash)
                    };
                    let current_median_time = self.block_median_time(&parent_hash);
                    let required_timestamp = base_timestamp
                        .checked_add(timestamp)
                        .ok_or(TransactionError::InvalidSince { index })?;
                    if current_median_time < required_timestamp {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                None => {
                    return Err((TransactionError::InvalidSince { index }).into());
                }
            }
        }
        Ok(())
    }
