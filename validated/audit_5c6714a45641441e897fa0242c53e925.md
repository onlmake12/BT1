### Title
Unbounded `while` Loop in `constructProviderCommitment` Can Exhaust Gas, Permanently Locking Entropy Requests - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

`Entropy.constructProviderCommitment` iterates `numHashes` times in an unbounded `while` loop. `numHashes` for each request equals `assignedSequenceNumber - currentCommitmentSequenceNumber`. When a provider has `maxNumHashes == 0` (the default, which disables the guard) and has not called `advanceProviderCommitment` for a large number of requests, `numHashes` grows without bound. Any subsequent call to `reveal` or `revealWithCallback` for such a request will run out of gas, permanently locking the request.

### Finding Description

`constructProviderCommitment` in `Entropy.sol` hashes a value `numHashes` times in a tight loop:

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
``` [1](#0-0) 

This function is called from `revealHelper`, which is invoked by both `reveal` and `revealWithCallback`:

```solidity
bytes32 providerCommitment = constructProviderCommitment(
    req.numHashes,
    providerContribution
);
``` [2](#0-1) 

`req.numHashes` is set at request time as:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
``` [3](#0-2) 

The only guard against a large `numHashes` is the `maxNumHashes` check:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [4](#0-3) 

Critically, `maxNumHashes` defaults to `0` in the `ProviderInfo` struct (Solidity zero-initializes storage), and `0` **disables** the check entirely — the condition `providerInfo.maxNumHashes != 0` short-circuits to false. [5](#0-4) 

This is confirmed by the test `testZeroMaxNumHashesDisableChecks`, which explicitly verifies that `maxNumHashes == 0` bypasses the guard. [6](#0-5) 

A provider that has not called `setMaxNumHashes` (or has set it to `0`) and has not called `advanceProviderCommitment` for many requests will accumulate a large gap. Any new request created in this state stores a large `numHashes`. When `revealWithCallback` is later called, `constructProviderCommitment` loops that many times, consuming gas proportional to `numHashes`. At ~30 gas per `keccak256`, ~1,000,000 iterations exhausts a 30M gas block limit.

### Impact Explanation

Any in-flight Entropy request whose stored `numHashes` exceeds the gas budget of a single transaction becomes permanently unrevealable. The provider cannot fulfill the callback, the user cannot receive their random number, and the request is stuck forever. This directly impairs the availability and correctness of the Entropy protocol for affected users.

### Likelihood Explanation

`maxNumHashes` is opt-in and disabled by default (`0`). A provider that registers without calling `setMaxNumHashes` has no protection. If the provider's off-chain service (`fortuna`) fails to call `advanceProviderCommitment` regularly — due to a bug, downtime, or negligence — the gap grows with every new request. Any user who requests randomness during this window receives a permanently stuck request. The Fortuna service does support `advanceProviderCommitment` but it is not enforced on-chain. [7](#0-6) 

### Recommendation

1. **Enforce a non-zero `maxNumHashes` at provider registration.** Require providers to set a meaningful `maxNumHashes` when calling `register`, rather than allowing `0` to silently disable the guard.
2. **Add a hard on-chain cap** on `numHashes` (e.g., `type(uint32).max` is too large; cap at a value that fits within a reasonable gas budget, e.g., 10,000).
3. **Apply the `maxNumHashes` check in `advanceProviderCommitment` as well**, since that function also calls `constructProviderCommitment` with an unbounded `numHashes`. [8](#0-7) 

### Proof of Concept

1. Provider registers with default `maxNumHashes == 0` (no `setMaxNumHashes` call).
2. Provider's off-chain service stops calling `advanceProviderCommitment`. `currentCommitmentSequenceNumber` stays at 0.
3. 1,000,000 users call `requestWithCallback`. Each request stores `numHashes = sequenceNumber - 0 = sequenceNumber`.
4. The 1,000,000th user's request stores `numHashes = 1,000,000`.
5. Provider calls `revealWithCallback` for that request. `constructProviderCommitment` loops 1,000,000 times (~30M gas). Transaction reverts with out-of-gas.
6. The request is permanently stuck. The user never receives their random number.
