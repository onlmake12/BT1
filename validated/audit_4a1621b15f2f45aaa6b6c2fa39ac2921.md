### Title
Stale `descendants_*` Accounting in `remove_entry_and_descendants` Allows Tx-Pool Eviction Manipulation — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all parent/child links for the removed subtree are cleared **before** `remove_entry` is called on each entry. Because `remove_entry` relies on those links to locate ancestors via `update_ancestors_index_key`, the ancestors of the removed subtree never have their `descendants_fee`, `descendants_size`, `descendants_cycles`, `descendants_count`, or `evict_key` decremented. An unprivileged tx submitter can exploit this through repeated RBF replacements to inflate a low-fee transaction's apparent descendant fee rate, preventing it from being evicted when the pool is full and causing legitimate high-fee transactions to be displaced instead.

---

### Finding Description

**Root cause — `remove_entry_and_descendants` clears links before updating ancestors**

`remove_entry_and_descendants` first removes all links for every entry in the subtree, then calls `remove_entry` for each: [1](#0-0) 

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← ALL links cleared here
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← remove_entry called after links gone
        .collect()
}
```

Inside `remove_entry`, the first thing done is: [2](#0-1) 

```rust
pub(crate) fn remove_entry(&mut self, id: &ProposalShortId) -> Option<TxEntry> {

        self.update_descendants_index_key(&entry.inner, EntryOp::Remove);
        self.remove_entry_edges(&entry.inner);
        self.remove_entry_links(id);   // ← links already gone; this is a no-op
        ...
    })
}
```

`update_ancestors_index_key` calls `self.links.calc_ancestors(...)` to find which pool entries are ancestors of the removed entry: [3](#0-2) 

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());  // returns ∅ — links already gone
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                EntryOp::Add    => e.inner.add_descendant_weight(child),
            };
            e.evict_key = e.inner.as_evict_key();   // ← never reached for ancestors
        });
    }
}
```

Because the links were already cleared, `calc_ancestors` returns an empty set. No ancestor entry has `sub_descendant_weight` called, and no ancestor's `evict_key` is recomputed. The `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` fields of every ancestor of the removed subtree remain permanently inflated.

**Compounding via RBF re-insertion**

When a replacement transaction B′ (child of A) is subsequently added, `record_entry_descendants` calls `update_ancestors_index_key(B′, Add)`: [4](#0-3) 

At this point the links for A are intact, so A is correctly found as an ancestor and B′'s fee is **added** to A's `descendants_fee`. But the old B's fee was **never subtracted**. After one RBF cycle, A's `descendants_fee = fee(B_old) + fee(B′)` instead of just `fee(B′)`. After N cycles, `descendants_fee = (N+1) × fee(B)`.

**Effect on eviction ordering**

The `EvictKey` for each entry is computed as: [5](#0-4) 

```rust
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        let feerate = FeeRate::calculate(entry.fee, weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),   // ← inflated by stale descendants_fee
            ...
        }
    }
}
```

`limit_size` uses `next_evict_entry`, which iterates by `evict_key` to pick the cheapest entry to drop: [6](#0-5) 

With an artificially high `fee_rate` in its `evict_key`, the ancestor A is skipped during eviction even though its true fee rate is low.

---

### Impact Explanation

**Impact: High**

An attacker can keep an arbitrarily low-fee transaction alive in the tx-pool indefinitely by inflating its `descendants_fee` through repeated RBF replacements of a high-fee child. When the pool reaches `max_tx_pool_size`, `limit_size` evicts entries with the lowest `evict_key.fee_rate`. Because A's `evict_key` is stale and inflated, legitimate high-fee transactions submitted by other users are evicted in A's place. This degrades tx-pool quality, reduces effective miner revenue, and can be used to selectively displace competing transactions.

---

### Likelihood Explanation

**Likelihood: High**

- The RBF path (`process_rbf`) is the primary trigger and is reachable by any unprivileged tx submitter via the `send_transaction` JSON-RPC endpoint.
- `remove_entry_and_descendants` is also called from `resolve_conflict` (block commit), `resolve_conflict_header_dep`, and `remove_by_detached_proposal`, so the stale accounting accumulates through normal node operation even without deliberate exploitation.
- No special privileges, keys, or majority hash power are required. [7](#0-6) 

---

### Recommendation

In `remove_entry_and_descendants`, collect and update the ancestors of the root entry **before** clearing any links. Concretely:

1. Before the `remove_entry_links` loop, call `self.links.calc_ancestors(id)` to get the set of surviving ancestors.
2. For each entry being removed, call `sub_descendant_weight` on every surviving ancestor and recompute `evict_key`.
3. Only then proceed to clear links and remove entries.

Alternatively, restructure `remove_entry` so that it accepts a pre-computed ancestor set, allowing `remove_entry_and_descendants` to pass the correct ancestors even after links are cleared.

---

### Proof of Concept

```
1. Submit tx A  (fee = 1 shannon/byte, low fee)
2. Submit tx B  (fee = 1000 shannons/byte, spends A's output)
   → A.descendants_fee = fee(A) + fee(B)
   → A.evict_key.fee_rate = max(A_rate, B_rate) = B_rate  (high)

3. RBF-replace B with B′ (same fee as B, also spends A's output)
   → remove_entry_and_descendants(B) called
   → A.descendants_fee NOT decremented  (bug)
   → B′ added: A.descendants_fee += fee(B′)
   → A.descendants_fee = fee(B) + fee(B′) = 2 × fee(B)  (double-counted)

4. Repeat step 3 N times:
   → A.descendants_fee = (N+1) × fee(B)  (arbitrarily large)
   → A.evict_key.fee_rate grows unboundedly

5. Fill pool with other transactions.
   → limit_size() calls next_evict_entry()
   → A is never selected for eviction despite its true fee rate being 1 shannon/byte
   → Legitimate high-fee transactions from other users are evicted instead.
```

    fn record_entry_descendants(&mut self, entry: &TxEntry) {
        let tx_short_id: ProposalShortId = entry.proposal_short_id();
        let outputs = entry.transaction().output_pts();
        let mut children = HashSet::new();

        // collect children
        for o in outputs {
            if let Some(ids) = self.edges.get_deps_ref(&o).cloned() {
                children.extend(ids);
            }
            if let Some(id) = self.edges.get_input_ref(&o).cloned() {
                children.insert(id);
            }
        }
        // update children
        if !children.is_empty() {
            for child in &children {
                self.links.add_parent(child, tx_short_id.clone());
            }
            if let Some(links) = self.links.inner.get_mut(&tx_short_id) {
                links.children.extend(children);
            }
            self.update_descendants_index_key(entry, EntryOp::Add);
        }
        // update ancestor's index key for adding new entry
        self.update_ancestors_index_key(entry, EntryOp::Add);
    }
