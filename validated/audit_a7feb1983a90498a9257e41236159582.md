## Title
`migrate_transaction_info` Traverses `COLUMN_UNCLES` Instead of `COLUMN_TRANSACTION_INFO`, Leaving All Pre-Migration Transaction Info Unmigrated â€” (`File: util/migrate/src/migrations/table_to_struct.rs`)

## Summary
The `ChangeMoleculeTableToStruct` migration's `migrate_transaction_info` function calls `db.traverse(COLUMN_UNCLES, ...)` at line 93 instead of `db.traverse(COLUMN_TRANSACTION_INFO, ...)`. As a result, every `TransactionInfo` record stored in the old 68-byte molecule Table encoding is never rewritten to the 52-byte Struct encoding. Any node that ran this migration against pre-v0.35.0 data retains stale, mis-encoded transaction info for every historical transaction, causing corrupted or missing results from `get_transaction_info` and the `get_transaction` RPC for all affected transactions.

## Finding Description
In `util/migrate/src/migrations/table_to_struct.rs`, the `migrate_transaction_info` function is defined at lines 77â€“102. The closure `transaction_info_migration` correctly targets `COLUMN_TRANSACTION_INFO` for writes (`wb.put(COLUMN_TRANSACTION_INFO, key, &value[16..])`), but the `db.traverse` call at line 93 feeds it records from `COLUMN_UNCLES`:

```rust
let (_count, nk) =
    db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
```

By the time `migrate_transaction_info` runs, `migrate_uncles` has already completed, so all uncle records in `COLUMN_UNCLES` are 240 bytes. The guard `value.len() != TRANSACTION_INFO_SIZE` (52) is always true for 240-byte uncle records, so the closure fires for every uncle â€” writing 224-byte uncle-derived slices into `COLUMN_TRANSACTION_INFO` under uncle hash keys. The actual `COLUMN_TRANSACTION_INFO` entries (keyed by tx hash, 68 bytes in old Table format) are never visited and remain unconverted.

After migration, `get_transaction_info` in `store/src/store.rs` (lines 307â€“313) calls `packed::TransactionInfoReader::from_slice_should_be_ok` on the raw 68-byte slice. This function does not validate total length; it wraps the slice and reads at fixed Struct offsets:
- `block_number()` â†’ bytes `[0..8]` â†’ reads `total_size` + first offset table entry (garbage)
- `block_epoch()` â†’ bytes `[8..16]` â†’ reads remaining offset bytes (garbage)
- `key()` â†’ bytes `[16..52]` â†’ reads actual `block_number` + `block_epoch` + partial `key` (shifted, wrong)

The three other sub-migrations all correctly traverse their own column: `migrate_header` traverses `COLUMN_BLOCK_HEADER`, `migrate_uncles` traverses `COLUMN_UNCLES`, and `migrate_epoch_ext` traverses `COLUMN_EPOCH`. Only `migrate_transaction_info` uses the wrong column.

## Impact Explanation
Every call to `get_transaction_info` for a pre-migration transaction returns a `TransactionInfo` with a corrupted `block_hash` and wrong `block_number`/`block_epoch`. `get_transaction_with_info` (lines 316â€“342 of `store/src/store.rs`) uses the corrupted `tx_info.key()` to look up `COLUMN_BLOCK_BODY`, which returns `None` because the key is wrong â€” causing `get_transaction` to silently return `None` for confirmed transactions. This directly causes the `get_transaction` RPC to return incorrect results for any transaction confirmed before the migration ran, fitting the **Note (0â€“500 points): Any local RPC API crash** impact class. The corruption is deterministic and affects every pre-migration transaction on any node that performed the v0.35.0 upgrade.

## Likelihood Explanation
The migration is triggered automatically on node startup when the stored database version is behind `20200703124523`, or explicitly via `ckb migrate`. Any operator who upgraded from a version prior to v0.35.0 to v0.35.0 or later would have run this migration. No attacker action, special configuration, or external dependency is required â€” the bug is triggered by the standard, documented upgrade procedure. The migration version is stamped in the database after running, so re-running the migration does not fix already-affected nodes.

## Recommendation
Change line 93 of `util/migrate/src/migrations/table_to_struct.rs` from:
```rust
db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
```
to:
```rust
db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
```
Because the migration version `20200703124523` is already stamped in affected databases, a new migration (with a new version string) must also be added to re-scan `COLUMN_TRANSACTION_INFO` and rewrite any remaining 68-byte Table-format entries to the 52-byte Struct format.
