Audit Report

## Title
Unbounded Response Amplification via Duplicate Indexes in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` accepts up to 32,767 transaction indexes from a remote peer with no deduplication and no byte-size cap on the resulting `BlockTransactions` response. An unprivileged peer can send a single `GetBlockTransactions` message with 32,767 identical valid indexes pointing to a stored transaction, causing the victim node to allocate, serialize, and transmit a response orders of magnitude larger than the request. The analogous handler `GetBlockProposalProcess` has both protections; `GetBlockTransactionsProcess` has neither.

## Finding Description
The only guard in `execute` checks the raw count of indexes against `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767):

```rust
// sync/src/relayer/get_block_transactions_process.rs L37-43
if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
    return StatusCode::ProtocolMessageIsMalformed...
}
```

After passing this check, the code iterates every index — including duplicates — and clones the corresponding transaction for each occurrence:

```rust
// L61-71
let transactions = self.message.indexes().iter()
    .filter_map(|i| block.transactions().get(Into::<u32>::into(i) as usize).cloned())
    .collect::<Vec<_>>();
```

The resulting `Vec<Transaction>` is packed directly into a `BlockTransactions` message and sent with no size check:

```rust
// L80-97
let content = packed::BlockTransactions::new_builder()
    .block_hash(block_hash)
    .transactions(
        transactions.into_iter().map(|tx| tx.data()).collect::<Vec<_>>(),
    )
    ...
    .build();
return async_send_message_to(&self.nc, self.peer, &message).await;
```

`MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MiB) is defined in `mod.rs` at line 61 but is never imported or used in `get_block_transactions_process.rs` (line 1 imports only `MAX_RELAY_TXS_NUM_PER_BATCH`).

By contrast, `GetBlockProposalProcess` deduplicates via `HashSet` and rejects if `proposals.len() != message_len`, and batches responses flushing when `relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH`. Neither protection exists in `GetBlockTransactionsProcess`.

The rate limiter in `Relayer::try_process` (30 req/s per `(peer, message_type)`) is insufficient: at 30 req/s, even a modest 32 KB transaction yields 30 × 32,767 × 32 KB ≈ 30 GB/s outbound per peer. Multiple malicious peers multiply the effect linearly.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node / cause CKB network congestion with few costs.**

A single malicious peer can force the victim to allocate and serialize a response of up to `32,767 × max_tx_size` bytes per request, exhausting memory and outbound bandwidth. The relay async task is blocked for the duration of each allocation and send. With the 30 req/s rate limit, a single attacker peer can sustain continuous memory pressure and bandwidth saturation. Multiple coordinated peers amplify this proportionally. Node crash via OOM or sustained bandwidth exhaustion is a realistic outcome.

## Likelihood Explanation
No authentication, PoW, or privileged access is required. The attacker only needs to connect as a normal P2P peer and know any block hash on-chain (trivially obtained) and the index of any transaction within it. The `GetBlockTransactions` message is a standard relay protocol message accepted from any connected peer. The attack is repeatable at the rate limiter ceiling (30 req/s per peer) and scales with the number of attacker-controlled peers.

## Recommendation
Apply both protections present in `GetBlockProposalProcess` to `GetBlockTransactionsProcess::execute`:

1. **Deduplicate indexes before processing**: Collect indexes into a `HashSet<u32>`; if `deduped.len() != original_len`, return `StatusCode::RequestDuplicate`.
2. **Enforce a byte-size cap on the response**: Accumulate `tx.data().total_size()` for each transaction to be included; stop adding transactions once the running total exceeds `MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MiB), or batch and flush as `GetBlockProposalProcess` does.

## Proof of Concept
```
1. Attacker connects to victim as a normal P2P peer.
2. Attacker identifies any block hash H on-chain containing a large transaction at index 0
   (e.g., a transaction with large witness data; block hashes are public).
3. Attacker sends:
     GetBlockTransactions {
       block_hash: H,
       indexes: [0, 0, 0, ..., 0],   // 32,767 entries, all identical
       uncle_indexes: []
     }
4. Victim's execute():
   - Passes the count check (32767 <= 32767).
   - Loads the block from store.
   - filter_map produces 32,767 clones of the same transaction.
   - Builds and sends a BlockTransactions message containing 32,767 copies.
5. If the transaction is T bytes, the response is ~32,767 × T bytes,
   allocated and serialized from a single ~200-byte request.
6. Attacker repeats at 30 req/s (rate limiter ceiling) for sustained impact.
   Multiple attacker peers multiply the effect.

Unit test plan:
- Construct a mock block with one large transaction.
- Call execute() with 32,767 identical index-0 entries.
- Assert the response Vec length is 32,767 and total serialized size >> MAX_RELAY_TXS_BYTES_PER_BATCH.
```
