Audit Report

## Title
Unbounded Dead-Cell `NOT IN` Clause Exceeds SQLite Bind-Parameter Limit in `get_cells` — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

## Summary
When `RichIndexer` is enabled with `index_tx_pool = true` and the SQLite backend is used, `get_cells` and `get_cells_capacity` dynamically construct a `NOT IN ($1, $2, …, $N)` clause with one bind parameter per dead cell in the pool. SQLite's default `SQLITE_MAX_VARIABLE_NUMBER` is 999. If the pool accumulates ≥ 1000 distinct dead cells, every call to these RPC endpoints returns a `DB`-level error until the pool drains.

## Finding Description
`Pool.dead_cells` is an unbounded `HashSet<OutPoint>`: [1](#0-0) 

In `get_cells`, all dead cells are collected without any count bound: [2](#0-1) 

One `$N` placeholder is emitted per entry: [3](#0-2) 

One `.bind(tx_hash)` call is made per entry (note: `output_index` is inlined into the SQL string, so each dead cell contributes exactly one bind parameter): [4](#0-3) 

Because `param_index` starts at 1 and is already incremented for script parameters before the dead-cell loop, the total bind count exceeds 999 with slightly fewer than 999 dead cells. When SQLite rejects the query with `SQLITE_RANGE`, the error propagates as: [5](#0-4) 

The identical pattern exists in `get_cells_capacity`: [6](#0-5) 

The codebase already recognises this limit in the insert path with an explicit comment, but the guard was never applied to the query path: [7](#0-6) 

## Impact Explanation
The `get_cells` and `get_cells_capacity` RPC endpoints become non-functional (returning `Error::DB` to all callers) for the duration the pool holds ≥ ~995 dead cells. The node itself does not crash. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation
Requires: (1) operator has explicitly enabled the `RichIndexer` module (not in default `modules` list) and `index_tx_pool = true` — a documented production configuration: [8](#0-7) 
(2) an attacker controls ≥ ~995 live cells on-chain and submits transactions spending them. The 180 MB pool size limit: [9](#0-8) 
imposes no transaction-count ceiling that would prevent accumulating ~1000 distinct inputs. On testnet the cost is negligible; on mainnet it requires real CKB but is not prohibitive. Once triggered, the condition persists until the pool drains.

## Recommendation
Before constructing the `NOT IN` clause, check `dead_cells.len()` and either: (a) chunk into batches of at most `BATCH_SIZE_THRESHOLD - <script_param_count>` and issue multiple queries intersecting results, or (b) insert dead-cell hashes into a temporary table and join against it. Apply the same fix to `get_cells_capacity`. The existing `BATCH_SIZE_THRESHOLD` constant in `insert.rs` should be reused as the reference limit.

## Proof of Concept
Populate the indexer `Pool` with ~1000 synthetic `OutPoint` entries via `pool.new_transaction(...)` (each transaction spending one input), then call `get_cells` against a SQLite-backed `AsyncRichIndexerHandle`. Assert the call returns `Err(Error::DB(…))` rather than a successful result. The existing `BATCH_SIZE_THRESHOLD` constant and its comment in `insert.rs` confirm the developers are aware of the SQLite bind limit; the omission in the query path is the root cause.
