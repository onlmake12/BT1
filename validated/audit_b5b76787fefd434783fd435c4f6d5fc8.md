### Title
`check_tx_fee` Uses Serialized Size Instead of Actual Weight, Allowing Min-Fee-Rate Bypass for High-Cycle Transactions - (File: `tx-pool/src/util.rs`)

### Summary

The tx-pool's minimum fee rate admission check (`check_tx_fee`) computes the required minimum fee using only the transaction's serialized byte size. However, the actual "weight" used for block-space accounting and miner prioritization is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. A transaction submitter can craft a small-serialized-size transaction that consumes near-maximum cycles, pass the size-only fee gate, and enter the tx-pool while paying a fee rate up to ~60× below the configured `min_fee_rate`. The code itself acknowledges this discrepancy with an inline comment but performs no corrective check after the actual cycle count is verified.

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` enforces the minimum fee rate using only `tx_size`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The comment directly above this line reads:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [1](#0-0) 

The actual weight used everywhere else in the codebase (miner selection, fee estimation, eviction) is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

The processing pipeline in `_process_tx` calls `pre_check` (which calls `check_tx_fee` with size only) **before** `verify_rtx` (which produces the actual cycle count). After `verify_rtx` returns the verified cycle count, no second fee-rate check is performed against the true weight: [3](#0-2) 

The `TxEntry` is created with the verified cycles, but the fee adequacy is never re-evaluated: [4](#0-3) 

### Impact Explanation

An attacker submits a transaction with:
- Minimal serialized size (~200 bytes, the smallest valid transaction)
- Cycles near `max_tx_verify_cycles` (default 70,000,000)

The effective weight is `max(200, 70_000_000 × 0.000_170_571_4)` ≈ 11,940 bytes. The fee check requires only `min_fee_rate × 200 / 1000` shannons, but the correct requirement is `min_fee_rate × 11,940 / 1000` shannons — a ~60× discrepancy.

Consequences:
1. **Tx-pool spam at 1/60th the intended cost.** The attacker can fill the 180 MB pool with ~60× more transactions than the fee rate was designed to allow, evicting legitimate higher-fee-rate transactions.
2. **Miner deprioritization.** These transactions will never be competitively mined because their true fee rate (by weight) is far below `min_fee_rate`, yet they occupy pool slots.
3. **Fee estimation poisoning.** The `estimate_fee_rate` fallback path iterates pool entries by score; injecting many low-true-fee-rate entries skews the estimate downward. [5](#0-4) 

### Likelihood Explanation

The attack requires only the ability to submit transactions via the public `send_transaction` RPC or the P2P relay protocol — both are reachable by any unprivileged user. No special keys, majority hashpower, or privileged access are needed. The attacker must pay a small fee (the size-based minimum), but the cost is ~60× lower than intended, making sustained spam economically viable. [6](#0-5) 

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee-rate check using the true weight:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

This mirrors how `TxEntry::fee_rate()` and all miner-selection code already compute the weight: [7](#0-6) 

### Proof of Concept

1. Craft a transaction spending a live cell with a lock script that consumes ~70,000,000 cycles (e.g., a tight loop in a RISC-V script).
2. Set the output capacity so that `inputs_capacity − outputs_capacity` = `min_fee_rate × tx_size / 1000` shannons (satisfying the size-only check).
3. Submit via `send_transaction` RPC. The node accepts it into the pending pool.
4. Observe via `get_pool_transaction` that the entry's `cycles` field is ~70,000,000 while the fee paid corresponds to a weight of only ~200 bytes.
5. Repeat to fill the pool; legitimate transactions with correct fee rates are evicted. [8](#0-7) [9](#0-8)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
