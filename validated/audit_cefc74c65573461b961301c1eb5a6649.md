### Title
Missing Payload Timestamp Validation Enables Replay of Stale Signed Price Updates - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate` authenticates the ECDSA signature and signer expiry of a Lazer price payload but never validates the timestamp embedded in the payload itself. Any unprivileged caller can replay an arbitrarily old signed payload — captured when prices were at a favorable level — and the contract will return it as fully verified. Consumer contracts that rely solely on `verifyUpdate` for price authenticity will receive stale prices, enabling the same class of economic attack described in the reference report: open a position using a favorable (replayed) price, then close it after the real market has moved.

### Finding Description
`PythLazer.verifyUpdate` performs three checks:

1. Fee payment (`msg.value >= verification_fee`) [1](#0-0) 
2. EVM format magic (`evm_magic != EVM_FORMAT_MAGIC`) [2](#0-1) 
3. ECDSA signature recovery and signer expiry (`isValidSigner`) [3](#0-2) 

After these checks the raw `payload` bytes are returned to the caller with no further validation. [4](#0-3) 

The payload format (confirmed by the test helper `buildPayload`) encodes a `uint64 timestamp` as its second field, immediately after the magic. [5](#0-4) 

`verifyUpdate` never reads or bounds-checks this timestamp field. Because Lazer payloads are broadcast publicly over WebSocket, an attacker can trivially capture and store any historically signed payload. As long as the signing key has not yet expired (`block.timestamp < trustedSignerToExpiresAtMapping[signer]`), the replayed payload passes every on-chain check and is returned as a valid, verified update. [6](#0-5) 

### Impact Explanation
Consumer contracts that call `verifyUpdate` and then parse the returned payload for price data will silently accept prices from an arbitrarily old point in time. An attacker who:

1. Saves a Lazer payload captured when a target asset's price was at a favorable extreme,
2. Waits for the real market price to diverge significantly, then
3. Calls `verifyUpdate` with the saved payload (paying only the `verification_fee`),

can feed the consumer contract a stale price that differs materially from the current market. Any derivative, lending, or perp protocol built on top of `PythLazer` that does not independently enforce a freshness window is directly exploitable for financial loss — exactly the oracle-manipulation-then-position-close attack described in the reference report.

### Likelihood Explanation
- **Entry path is fully unprivileged**: any EOA can call `verifyUpdate` with a fee of 1 wei (the default `verification_fee`). [7](#0-6) 
- **Signed payloads are public**: Lazer distributes signed updates over a public WebSocket stream; no privileged access is needed to collect historical payloads.
- **Signer keys are long-lived**: `trustedSignerToExpiresAtMapping` stores expiry timestamps that are set far in the future (the test uses `3000000000000000`), so captured payloads remain replayable for the entire lifetime of a key. [8](#0-7) 
- **No other on-chain guard**: `verifyUpdate` is the sole on-chain verification primitive for Lazer; there is no companion `checkFreshness` function.

### Recommendation
Add a configurable maximum age parameter (e.g., `maxAgeSeconds`) to `verifyUpdate`. After recovering the signer, decode the `uint64 timestamp` from the payload (bytes `[4:12]` per the payload format) and require:

```solidity
uint64 payloadTimestamp = uint64(bytes8(payload[4:12]));
require(
    block.timestamp <= payloadTimestamp + maxAgeSeconds,
    "payload too old"
);
```

Alternatively, expose a separate `verifyUpdateWithMaxAge(bytes calldata update, uint64 maxAgeSeconds)` variant so latency-sensitive consumers can enforce strict freshness while others retain flexibility. This mirrors the pattern already used in Pyth Core (`getPriceNoOlderThan`) and the Pulse Scheduler's `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` guard. [9](#0-8) 

### Proof of Concept

```
Preconditions:
  - trustedSigner S is registered with expiresAt = far future
  - S signs payload P at time T with price = $100 (favorable to attacker)
  - At time T+3600 the real price is $120

Attack:
  1. Attacker calls PythLazer.verifyUpdate{value: 1 wei}(oldUpdate)
     where oldUpdate contains payload P signed by S at time T.
  2. verifyUpdate:
       - fee check passes (1 wei >= 1 wei)
       - magic check passes
       - ECDSA recovers signer = S
       - isValidSigner(S) → true (key not expired)
       - returns (payload=P, signer=S)   ← timestamp T never checked
  3. Consumer contract reads price = $100 from P.
  4. Attacker opens a leveraged long at $100, immediately closes at real price $120.
  5. Net profit = leverage × $20 per unit, funded by the consumer's liquidity pool.
``` [4](#0-3)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
