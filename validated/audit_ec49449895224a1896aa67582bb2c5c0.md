### Title
Partial Price ID Verification in `executeCallback` Allows Wrong Price Data Delivery - (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol`'s `executeCallback` stores only the first 8 bytes (`bytes8`) of each requested price ID at request time, and at callback time only verifies those 8 bytes against the caller-supplied `priceIds`. An unprivileged caller can supply a different full 32-byte Pyth price ID that shares the same 8-byte prefix, causing the consumer contract to receive price data for the wrong asset and permanently consuming the request.

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, the contract stores only the first 8 bytes of each price ID:

```solidity
// Copy only the first 8 bytes of each price ID to storage
for (uint8 i = 0; i < priceIds.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    req.priceIdPrefixes[i] = prefix;
}
``` [1](#0-0) 

At callback time, `executeCallback` only checks those 8 bytes:

```solidity
for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    if (prefix != req.priceIdPrefixes[i]) {
        revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
    }
}
``` [2](#0-1) 

After the exclusivity period, `executeCallback` is callable by anyone — there is no `msg.sender` restriction: [3](#0-2) 

The function then calls `parsePriceFeedUpdates` with the attacker-supplied `priceIds` and delivers the resulting price feeds to the consumer via `_echoCallback`: [4](#0-3) 

The request is then cleared (`clearRequest`), so the legitimate callback can never be re-executed.

### Impact Explanation

An attacker who finds (or waits for) two legitimate Pyth price IDs sharing the same first 8 bytes can:

1. Observe a pending `requestPriceUpdatesWithCallback` for price ID `A` (e.g., BTC/USD).
2. Wait for the exclusivity period to elapse.
3. Call `executeCallback` supplying price ID `B` (a different asset whose first 8 bytes match `A`'s prefix).
4. `parsePriceFeedUpdates` succeeds and returns price data for asset `B`.
5. The consumer's `_echoCallback` receives wrong price data and may execute trades/liquidations based on it.
6. The request is permanently cleared — the correct callback for asset `A` can never be delivered.

With 500+ Pyth price feeds, prefix collisions are plausible. Even without a current collision, the design is structurally broken: any future price feed addition could create one.

### Likelihood Explanation

- `executeCallback` is permissionless after the exclusivity period — any EOA can call it.
- The attacker only needs to find two Pyth price IDs sharing the same first 8 bytes among the 500+ live feeds.
- The attack requires no privileged access, no leaked keys, and no governance majority.
- The attacker can monitor the mempool for pending requests and act after the exclusivity window.

### Recommendation

Store and verify the **full 32-byte price IDs** instead of only the first 8 bytes. Replace `bytes8 priceIdPrefixes` with `bytes32[] priceIds` in the `Request` struct and compare the full IDs in `executeCallback`. The gas savings from truncation do not justify the security risk.

### Proof of Concept

1. Alice calls `requestPriceUpdatesWithCallback` for price ID `A = 0xAAAAAAAABBBBBBBB<24 more bytes>`, paying the required fee.
2. The contract stores `req.priceIdPrefixes[0] = 0xAAAAAAAABBBBBBBB`.
3. Attacker finds legitimate Pyth price ID `B = 0xAAAAAAAABBBBBBBB<different 24 bytes>`.
4. After the exclusivity period, attacker calls:
   ```solidity
   echo.executeCallback(attacker, sequenceNumber, updateData, [B]);
   ```
5. The 8-byte prefix check passes (`B`'s prefix == stored prefix).
6. `parsePriceFeedUpdates` returns price data for asset `B`.
7. Alice's consumer receives wrong price data; the request is cleared.
8. Alice's legitimate callback for asset `A` can never be fulfilled.
