Audit Report

## Title
Minimum Fee Rate Admission Check Uses Raw Byte Size Instead of Transaction Weight, Allowing Cycle-Heavy Transactions to Bypass Fee Rate Enforcement - (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate by passing raw serialized byte size to `FeeRate::fee()`, which is defined as shannons per kilo-**weight**. For cycle-heavy transactions, the correct weight (`max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`) can be orders of magnitude larger than the byte size alone, making the admission gate far too lenient. No subsequent weight-based fee rate gate exists in the admission path after cycles become known from `verify_rtx`.

## Finding Description
`FeeRate` is explicitly documented as "shannons per kilo-weight": [1](#0-0) 

`FeeRate::fee` computes `self.0.saturating_mul(weight) / KW` where `KW = 1000`: [2](#0-1) 

The sole admission-time fee rate gate in `check_tx_fee` passes raw `tx_size` (bytes) as the weight argument, with a comment explicitly acknowledging this is an approximation: [3](#0-2) 

The correct weight is computed by `get_transaction_weight`: [4](#0-3) 

In `_process_tx`, `check_tx_fee` is called during `pre_check` (before cycles are known), and after `verify_rtx` returns actual cycles, there is no post-verification fee rate gate — the entry is created and submitted directly: [5](#0-4) 

`TxEntry::fee_rate()` already uses the correct weight-based calculation, but only for sorting and eviction after admission: [6](#0-5) 

For a transaction with 200 serialized bytes and 70,000,000 cycles, the correct weight is `max(200, 70_000_000 × 0.000_170_571_4) ≈ 11,940`, but the admission check uses only `200`. With `min_fee_rate = 1000 shannons/KW` (the production default), the check requires only 200 shannons instead of the correct 11,940 shannons. A transaction paying 201 shannons passes the gate with an actual fee rate of `≈ 16.8 shannons/KW` — roughly 60× below the configured minimum. [7](#0-6) 

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker can flood the tx-pool and relay network with computationally expensive transactions (near-maximum VM cycles) while paying fees ~60× below the configured minimum. All peers apply the same flawed size-based check, so the spam propagates network-wide. This degrades node performance by forcing full CKB-VM execution of high-cycle transactions that paid inadequate fees, and reduces miner revenue by allowing below-minimum-fee-rate transactions to be mined.

## Likelihood Explanation
The attack requires no special privileges — any RPC caller or relay peer can submit transactions. The maximum cycle limit (70,000,000) is publicly documented in the default config: [8](#0-7) 
A script author can write a loop script consuming near-maximum cycles with minimal transaction bytes by placing the script in a cell dep (referenced by hash, not inlined). The discrepancy factor between bytes and weight reaches up to `11,940 / 200 ≈ 60×` for maximally cycle-heavy transactions. The attack is repeatable and systematic with negligible cost per transaction.

## Recommendation
Replace the bytes-based minimum fee computation in `check_tx_fee` with a weight-based computation. Since cycles are not available before script verification, the admission pipeline should be restructured so the fee rate check occurs after `verify_rtx` returns actual cycles in `_process_tx`:
```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```
This matches how `TxEntry::fee_rate()` already correctly computes the fee rate for sorting and eviction.

## Proof of Concept
1. Deploy a lock script cell that loops for ~70,000,000 cycles. The script cell is stored on-chain; the transaction referencing it via cell dep can be ~200 bytes serialized.
2. Craft a CKB transaction spending a cell locked by that script, with fee = 201 shannons.
3. Submit via RPC `send_transaction`.
4. `pre_check` → `check_tx_fee` computes `min_fee = 1000 * 200 / 1000 = 200 shannons`. Since `201 >= 200`, the transaction passes the admission gate.
5. `verify_rtx` executes the script, consuming ~70,000,000 cycles. No post-verification fee rate check is performed.
6. Actual weight = `max(200, 70_000_000 × 0.000_170_571_4) = 11,940`. Actual fee rate = `201 × 1000 / 11940 ≈ 16.8 shannons/KW`, which is ~60× below the 1,000 shannons/KW minimum.
7. The transaction is admitted to the pool and relayed to all peers (which apply the same flawed check). Repeat to flood the mempool and relay network with high-cycle, low-fee transactions.
