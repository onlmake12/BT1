### Title
Silent u128→u64 Truncation in DAO Withdrawal Capacity Calculation Produces Wrong Withdrawal Amount — (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the interest-bearing withdrawal capacity using a `u128` intermediate value, then silently truncates it to `u64` with an unchecked `as u64` cast. When the intermediate product exceeds `u64::MAX`, the high bits are discarded without any error, returning a drastically wrong (far too small) withdrawal capacity as a successful `Ok(...)` result. This is the CKB analog of the BigNumber carry/overflow calculation error: both are arithmetic errors in a financial computation that silently produce incorrect results instead of propagating an error.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The multiplication `counted_capacity * withdrawing_ar` is correctly widened to `u128` to avoid overflow. However, after the division, the result is cast back to `u64` with `as u64` — a **truncating, non-panicking cast** in Rust. If `withdraw_counted_capacity > u64::MAX`, the upper bits are silently discarded, producing a value that is `withdraw_counted_capacity mod 2^64` — potentially orders of magnitude smaller than the correct result.

The subsequent `safe_add(occupied_capacity)` only catches overflow of the *truncated* value, not the original overflow. If the truncated value is small (which it will be when the true result is just above a multiple of `2^64`), `safe_add` succeeds and the function returns `Ok(wrong_capacity)` with no indication of error.

The correct fix is to use `u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?`, consistent with how `miner_issuance128` and `ar_increase128` are handled elsewhere in the same file:

```rust
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) 

```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

The `calculate_maximum_withdraw` function is the only u128→u64 narrowing in this file that uses an unchecked `as u64` cast.

---

### Impact Explanation

`calculate_maximum_withdraw` is called in two security-critical paths:

1. **Transaction verification** — `transaction_maximum_withdraw` → `transaction_fee` is used during DAO withdrawal (phase 2) transaction verification to confirm the output capacity does not exceed the maximum withdrawable amount. [4](#0-3) 

   If `withdraw_counted_capacity` silently truncates, the computed "maximum withdraw" is far too small. `transaction_fee = maximum_withdraw - outputs_capacity` would underflow and return `Err`, causing a **valid DAO withdrawal transaction to be permanently rejected** by the node. The depositor cannot recover their funds through the normal withdrawal path.

2. **`calculate_dao_maximum_withdraw` RPC** — returns the wrong (too small) capacity to the caller, causing wallets and dApps to construct withdrawal transactions with insufficient output capacity, which are then rejected on-chain. [5](#0-4) 

The net effect is **permanent loss of access to DAO interest** (and potentially principal) for affected depositors, without any on-chain error that would indicate a bug — the node simply rejects the withdrawal as "Overflow" or the RPC returns a wrong value.

---

### Likelihood Explanation

The overflow condition requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX  (≈ 1.844 × 10¹⁹ shannons)
```

Since `withdrawing_ar ≥ deposit_ar` always holds (the accumulation rate is monotonically non-decreasing), the condition simplifies to needing a large `counted_capacity` and/or a large `withdrawing_ar / deposit_ar` ratio.

- CKB's total supply is ~33.6 billion CKB = ~3.36 × 10¹⁸ shannons, so a single cell's `counted_capacity` is bounded by the total supply. For overflow with the full supply deposited, `withdrawing_ar / deposit_ar > 5.5` is required — achievable only after many decades of compounding at typical secondary issuance rates.
- However, the `ar` ratio is stored as a raw `u64` in the DAO field and is extracted without bounds checking. A crafted or corrupted block header with an abnormally large `withdrawing_ar` value (e.g., from a chain reorganization or a malicious miner) could trigger the condition immediately.
- The existing test `check_withdraw_calculation_overflows` does **not** cover this path — it tests the `safe_add` overflow at the end (line 156), not the `as u64` truncation. The truncation path returns `Ok(wrong_value)` and is entirely untested. [6](#0-5) 

Likelihood is **low under normal mainnet conditions** but **medium under adversarial conditions** (malicious miner controlling `ar` values in block headers, or very large deposits over long time horizons).

---

### Recommendation

Replace the unchecked cast on line 156 with a checked conversion, consistent with the pattern used for `miner_issuance128` and `ar_increase128` in the same function:

```rust
// Before (vulnerable):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After (safe):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

Additionally, add a unit test that specifically exercises the `as u64` truncation path (i.e., where `withdraw_counted_capacity > u64::MAX` but `withdraw_counted_capacity as u64 + occupied_capacity ≤ u64::MAX`), confirming the function returns `Err(DaoError::Overflow)` rather than a silently wrong `Ok` value.

---

### Proof of Concept

**Trigger condition** (values chosen so truncation produces a small, non-overflowing result):

```
counted_capacity  = u64::MAX = 18_446_744_073_709_551_615
deposit_ar        = 10_000_000_000_000_000   (initial ar = 10^16)
withdrawing_ar    = 10_000_000_001_000_000   (ar after some time)
```

**Computation:**

```
withdraw_counted_capacity (u128)
  = 18_446_744_073_709_551_615 × 10_000_000_001_000_000
    / 10_000_000_000_000_000
  ≈ 18_446_744_073_709_551_615 + 18_446_744_073_709
  = 18_446_762_520_453_625_324   (exceeds u64::MAX by 17_446_744_073_708)

withdraw_counted_capacity as u64
  = 18_446_762_520_453_625_324 mod 2^64
  = 17_446_744_073_708           ← drastically wrong (should be ~18.4 × 10^18)
```

The function returns `Ok(Capacity::shannons(17_446_744_073_708 + occupied_capacity))` — approximately **1,000,000× less** than the correct withdrawal amount — with no error signal. The depositor's DAO withdrawal transaction, constructed using this value, would be rejected by the chain verifier because the actual maximum withdraw (computed again during verification) would differ from the output capacity.     /// Calculate maximum withdraw capacity of a deposited dao output
    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }

#[test]
fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
}
