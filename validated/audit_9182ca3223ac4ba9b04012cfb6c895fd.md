### Title
Admin Role Can Halt All Price Feed Updates via Unbounded `setSingleUpdateKeeperFeeInWei` — (File: `target_chains/ethereum/contracts/contracts/pulse/SchedulerGovernance.sol`)

---

### Summary

In `SchedulerUpgradeable`, the `admin` role (a lower-privileged operational role, distinct from the `owner`/governance) is permitted to call `setSingleUpdateKeeperFeeInWei()` with no upper-bound validation. Setting this value to `type(uint128).max` causes every subsequent `updatePriceFeeds()` call across every subscription to revert with `InsufficientBalance`, permanently halting all price feed updates in the Pulse/Scheduler protocol until the `owner` intervenes.

---

### Finding Description

`SchedulerUpgradeable` defines two privilege levels:

- **`owner`** — the highest-privilege governance role, set at initialization, capable of upgrading the contract.
- **`admin`** — a lower-privilege operational role, set at initialization and transferable via a two-step proposal.

The `_authorizeAdminAction()` hook in `SchedulerUpgradeable` grants **both** roles equal access to critical fee-setting functions:

```solidity
// SchedulerUpgradeable.sol L57-60
function _authorizeAdminAction() internal view override {
    if (msg.sender != owner() && msg.sender != _state.admin)
        revert SchedulerErrors.Unauthorized();
}
``` [1](#0-0) 

This means the `admin` can call `setSingleUpdateKeeperFeeInWei()`:

```solidity
// SchedulerGovernance.sol L68-75
function setSingleUpdateKeeperFeeInWei(uint128 newFee) external {
    _authorizeAdminAction();
    uint oldFee = _state.singleUpdateKeeperFeeInWei;
    _state.singleUpdateKeeperFeeInWei = newFee;
    emit SingleUpdateKeeperFeeSet(oldFee, newFee);
}
``` [2](#0-1) 

There is no upper-bound check on `newFee`. The fee is consumed in `_processFeesAndPayKeeper()`:

```solidity
// Scheduler.sol L846-853
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;

if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [3](#0-2) 

If `singleUpdateKeeperFeeInWei` is set to `type(uint128).max`, then `keeperSpecificFee = uint256(type(uint128).max) * numPriceIds` becomes an astronomically large value that no subscription balance can ever satisfy. Every call to `updatePriceFeeds()` for every subscription will revert.

The same pattern applies to `setMinimumBalancePerFeed()`, which if set to `type(uint128).max` prevents new subscriptions from being created, prevents existing active subscriptions from adding funds, and prevents withdrawals from active subscriptions. [4](#0-3) 

---

### Impact Explanation

A malicious or compromised `admin` can call `setSingleUpdateKeeperFeeInWei(type(uint128).max)` in a single transaction. This immediately causes every `updatePriceFeeds()` call across all subscriptions to revert with `InsufficientBalance`, halting all automated price feed delivery in the Pulse/Scheduler system. Subscribers' DeFi protocols that depend on Pulse for price updates will receive no further updates until the `owner` (governance) resets the fee. The `admin` can also combine this with `setMinimumBalancePerFeed(type(uint128).max)` to additionally lock all subscription funds (preventing withdrawals from active subscriptions), compounding the damage. [5](#0-4) 

---

### Likelihood Explanation

The `admin` role is a distinct, lower-privileged operational address. The scenario is directly analogous to M-14: if the `owner` (governance) initiates an admin replacement via `proposeAdmin()`, the current `admin` can observe this on-chain and front-run the transition by setting the keeper fee to `type(uint128).max`, halting the protocol before being replaced. The `admin` can also act maliciously at any time without any triggering event. The attack requires only a single transaction with no special preconditions beyond holding the `admin` key. [6](#0-5) 

---

### Recommendation

Restrict `setSingleUpdateKeeperFeeInWei()` and `setMinimumBalancePerFeed()` to `onlyOwner` (governance), removing `admin` access from these critical protocol-parameter functions. The `admin` role should retain only operational actions (e.g., `proposeAdmin`), while fee parameters that can halt the protocol must require the highest privilege level.

```solidity
function setSingleUpdateKeeperFeeInWei(uint128 newFee) external onlyOwner {
    ...
}

function setMinimumBalancePerFeed(uint128 newMinimumBalance) external onlyOwner {
    ...
}
```

Alternatively, add a reasonable upper-bound cap on both parameters to prevent extreme values from being set even by the `owner`.

---

### Proof of Concept

1. `owner` deploys `SchedulerUpgradeable` with `admin = address(A)`.
2. `owner` calls `proposeAdmin(address(B))` to replace the admin.
3. `address(A)` observes the pending replacement and calls `setSingleUpdateKeeperFeeInWei(type(uint128).max)`.
4. All subsequent calls to `updatePriceFeeds(subscriptionId, updateData)` revert:
   - `keeperSpecificFee = uint256(type(uint128).max) * numPriceIds` → enormous value
   - `status.balanceInWei < totalKeeperFee` → always true → `revert InsufficientBalance()`
5. All Pulse subscriptions stop receiving price updates until `owner` resets the fee. [7](#0-6) [8](#0-7)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
