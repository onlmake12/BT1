Audit Report

## Title
HolePunching Per-Peer Rate Limit Bypassed by Disconnect-Reconnect (Session ID Reset) — (File: network/src/protocols/hole_punching/mod.rs)

## Summary
The `HolePunching` protocol's primary rate limiter is keyed by `(PeerIndex, u32)` where `PeerIndex` is a type alias for the ephemeral `SessionId`. Because a new `SessionId` is assigned on every TCP reconnection, an attacker can exhaust their 30 msg/sec quota and then immediately reconnect to receive a fresh, empty bucket. The `disconnected` handler only calls `retain_recent()` — a stale-entry sweep — rather than removing the disconnecting peer's entry, so the old exhausted entry lingers while the new session starts unconstrained.

## Finding Description
`PeerIndex` is a pure alias for `SessionId`: [1](#0-0) 

The `HolePunching` struct holds two rate limiters with different key types: [2](#0-1) 

Every received message is checked against `rate_limiter` using the session-scoped `session_id`: [3](#0-2) 

On disconnect, only `retain_recent()` is called — this evicts globally stale entries but does **not** remove the specific disconnecting peer's entry: [4](#0-3) 

The quota is 30 requests/second per `(PeerIndex, msg_item_id)` pair: [5](#0-4) 

The `forward_rate_limiter` is keyed by `(PeerId, PeerId, u32)` — cryptographic identity — and is therefore **not** bypassed by reconnection. Only the primary `rate_limiter` is vulnerable.

**Bypass sequence:**
1. Attacker connects → assigned `SessionId = N`.
2. Attacker sends 30 HolePunching messages in <1 second → bucket for `(N, item_id)` exhausted.
3. Attacker disconnects → `retain_recent()` runs but the entry for `(N, item_id)` is still "recent" and remains.
4. Attacker reconnects → assigned `SessionId = N+1`.
5. No entry exists for `(N+1, item_id)` → fresh bucket → 30 more messages pass immediately.
6. Repeat indefinitely.

Rate-limited messages are silently dropped (not banned), so there is no ban mechanism triggered by this pattern. [6](#0-5) 

## Impact Explanation
An unprivileged remote peer can flood a CKB node with HolePunching messages (`ConnectionRequest`, `ConnectionRequestDelivered`, `ConnectionSync`) at rates far exceeding the intended 30/second cap. The `notify` handler broadcasts `ConnectionRequest` messages to `sqrt(total)` peers via gossip, and sustained inbound flooding drives repeated processing and potential outbound amplification across the network. This matches: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
Any peer that can establish a TCP connection to a CKB node can exploit this. No special privileges, keys, or majority hashpower are required. The only cost per bypass cycle is a TCP handshake, which is negligible. The HolePunching protocol is compiled and enabled for all non-WASM targets and is reachable from the public internet. The attack is trivially repeatable in a tight loop.

## Recommendation
Key the primary rate limiter by `PeerId` (stable cryptographic identity) rather than `PeerIndex` (ephemeral session ID), matching the approach already used by `forward_rate_limiter`:

```rust
// Change:
rate_limiter: RateLimiter<(PeerIndex, u32)>,
// To:
rate_limiter: RateLimiter<(PeerId, u32)>,
```

In the `received` handler, extract the `PeerId` from the session context and use it as the rate-limit key. This ensures reconnection does not reset the rate-limit bucket, as the bucket is tied to cryptographic identity rather than the transient session handle.

## Proof of Concept
```
1. Attacker opens TCP connection to victim CKB node.
   → Assigned SessionId = 100 (PeerIndex = 100).

2. Attacker sends 30 ConnectionRequest messages in <1 second.
   → rate_limiter[(100, item_id)] bucket exhausted.
   → 31st message is silently dropped (no ban triggered).

3. Attacker closes TCP connection.
   → disconnected() fires; retain_recent() runs but entry (100, item_id)
     is still "recent" and remains in the hashmap.

4. Attacker immediately reconnects.
   → Assigned SessionId = 101 (PeerIndex = 101).
   → rate_limiter has NO entry for (101, item_id).

5. Attacker sends 30 more ConnectionRequest messages in <1 second.
   → All 30 pass the rate check (fresh bucket).

6. Repeat from step 3 at will.
   → Effective throughput: 30 × reconnect_rate messages/second,
     unbounded by the intended 30/second cap.
```
