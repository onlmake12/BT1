Audit Report

## Title
TOCTOU Race in `_process_tx` Enables Redundant CKB-VM Script Verification via Concurrent Duplicate Submissions — (File: tx-pool/src/process.rs)

## Summary
`_process_tx` performs the duplicate-transaction check (`check_txid_collision`) under a read lock in `pre_check`, releases all locks, runs expensive CKB-VM script verification (`verify_rtx`) with no lock held, then re-acquires a write lock in `submit_entry` without re-checking the duplicate condition. With multiple verify workers active, a second submission of the same transaction timed to arrive after the first is dequeued can pass `pre_check` and trigger a full redundant CKB-VM execution. The secondary effect inserts the valid duplicate into `conflicts_pool`.

## Finding Description

**Root cause — no atomic check-then-insert:**

`pre_check` (process.rs lines 269–316) acquires a read lock via `with_tx_pool_read_lock` (line 277), calls `check_txid_collision` (util.rs lines 20–26) which checks `tx_pool.contains_proposal_id(&short_id)`, then drops the lock before returning. [1](#0-0) [2](#0-1) 

`verify_rtx` (process.rs lines 724–732) then runs the full `ContextualTransactionVerifier` with **no lock held**. [3](#0-2) 

`submit_entry` (process.rs lines 96–170) acquires a write lock and re-checks for conflicting inputs via `check_rbf`/`find_conflict_outpoint` (lines 105–116), but **never calls `check_txid_collision`** — there is no guard against a duplicate that passed `pre_check` while the first copy was in `verify_rtx`. [4](#0-3) 

**Why the verify_queue duplicate guard is insufficient:**

`add_tx` (verify_queue.rs lines 204–209) only rejects a second submission if the first copy is **still in the queue**. Once a worker calls `pop_front` (verify_mgr.rs line 132) and removes the entry, the queue no longer contains the tx, so a second submission passes `add_tx` and is enqueued for a second worker. [5](#0-4) [6](#0-5) 

**Multiple concurrent workers exist by design:**

`VerifyMgr` spawns `max_tx_verify_workers` workers (verify_mgr.rs lines 179–203), each independently calling `pop_front` and `_process_tx`. Two workers can therefore hold different copies of the same transaction simultaneously. [7](#0-6) 

**Exploit flow:**
1. Worker A dequeues tx T; T is no longer in the verify_queue.
2. Attacker sends T again; `add_tx` passes (T absent from queue), T is enqueued.
3. Worker B dequeues the second copy; both workers enter `_process_tx` concurrently.
4. Both pass `pre_check` (T not yet in pool — A has not reached `submit_entry`).
5. Both execute `verify_rtx` concurrently, each consuming up to `max_block_cycles` CPU.
6. Worker A completes `submit_entry` first, inserting T into the pool.
7. Worker B reaches `submit_entry`; `find_conflict_outpoint` returns T's inputs as dead, returning `Err(Reject::Resolve(OutPointError::Dead(...)))`.
8. `after_process` (lines 479–487) matches this error, calls `find_conflict_outpoint` (returns Some, since T is in pool), and calls `record_conflict(tx)` — inserting the valid duplicate T into `conflicts_pool`. [8](#0-7) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker holding a transaction whose lock script consumes close to `max_block_cycles` can force N full CKB-VM executions at the cost of N P2P relay messages. Each wasted execution occupies a verify worker for hundreds of milliseconds, starving legitimate transactions of verification capacity. Sustained across multiple transactions or multiple target nodes, this degrades network-wide transaction throughput. The secondary `conflicts_pool` pollution evicts legitimate conflict records from the LRU cache, degrading RBF conflict tracking accuracy.

## Likelihood Explanation

**Medium.** Any unprivileged peer can submit transactions via the P2P relay protocol (`RelayTransactions`). The TOCTOU window spans the entire CKB-VM execution time for the first copy — potentially hundreds of milliseconds for max-cycles scripts. The attacker needs only to send the same transaction twice in rapid succession; the second submission will be enqueued once the first is dequeued. No special privileges, keys, or majority hashpower are required. The attack is repeatable indefinitely and scales linearly with the number of concurrent verify workers.

## Recommendation

Re-check `check_txid_collision` (or equivalently `contains_proposal_id`) **inside `submit_entry` under the write lock**, before calling `_submit_entry`. This mirrors the existing pattern where `check_rbf` is explicitly placed inside the write lock (process.rs line 104) to prevent concurrent issues:

```rust
// Inside submit_entry's write-lock closure, before _submit_entry:
if tx_pool.contains_proposal_id(&entry.proposal_short_id()) {
    return Err(Reject::Duplicated(entry.transaction().hash()));
}
```

This ensures the duplicate check and pool insertion are atomic with respect to concurrent submissions. [9](#0-8) 

## Proof of Concept

1. Construct a valid transaction T whose lock script consumes close to `max_block_cycles` cycles (e.g., a tight loop in CKB-VM).
2. Configure the target node with `max_tx_verify_workers > 1` (the default for production nodes).
3. Open a P2P connection and send T via `RelayTransactions`. Monitor node logs for T being dequeued from `verify_queue` (log entry from `pop_front` in verify_mgr.rs line 132).
4. Immediately send T again via `RelayTransactions` while the first verification is still running (within the `verify_rtx` window).
5. Observe node logs: both submissions enter `_process_tx` and both call `verify_rtx`. Only one succeeds in `submit_entry`; the second is rejected with `Reject::Resolve(OutPointError::Dead(...))`.
6. Confirm `conflicts_pool` pollution: query the node's conflict cache and observe T present despite being a valid (non-conflicting) transaction.
7. Repeat steps 3–6 in a loop to sustain CPU exhaustion. Each iteration forces one extra full CKB-VM execution at the cost of one relay message.

pub(crate) fn check_txid_collision(tx_pool: &TxPool, tx: &TransactionView) -> Result<(), Reject> {
    let short_id = tx.proposal_short_id();
    if tx_pool.contains_proposal_id(&short_id) {
        return Err(Reject::Duplicated(tx.hash()));
