### Title
Malicious Entropy Provider Collects Fees and Permanently Blocks Callbacks Without Penalty — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any address can permissionlessly register as an Entropy provider. When a user submits a `requestV2` / `requestWithCallback`, the provider's fee is **immediately and irrevocably credited** to the provider's balance before any fulfillment occurs. If the provider never calls `revealWithCallback`, the user's callback is permanently blocked, the fee is permanently lost, and the provider faces zero on-chain penalty.

---

### Finding Description

**Step 1 — Permissionless provider registration.**

`register()` imposes no stake, bond, or whitelist: [1](#0-0) 

Any EOA can call `register(feeInWei, commitment, …)` with a low fee to attract users.

**Step 2 — Fee credited to provider at request time, before fulfillment.**

Inside `requestHelper()`, the provider's share of `msg.value` is added to `providerInfo.accruedFeesInWei` unconditionally, in the same transaction as the user's request: [2](#0-1) 

The fee is not held in escrow pending fulfillment — it is immediately spendable by the provider via `withdraw()`.

**Step 3 — No timeout, expiry, or refund mechanism.**

The contract stores the request in `CALLBACK_NOT_STARTED` state indefinitely: [3](#0-2) 

There is no function in the contract that allows a user to cancel a pending request, reclaim their fee, or trigger a timeout. A search of the contract confirms no `refund`, `cancel`, or user-initiated `withdraw` path exists.

**Step 4 — Provider withdraws fees with no obligation.**

The provider can call `withdraw()` at any time after fees accrue: [4](#0-3) 

**Attack flow:**
1. Attacker calls `register(lowFee, validCommitment, …)` — permissionless, no stake required.
2. Users call `requestV2{value: fee}(attackerAddress, …)` — fee immediately credited to attacker.
3. Attacker never calls `revealWithCallback` — user contract stuck in `CALLBACK_NOT_STARTED` forever.
4. Attacker calls `withdraw(accruedFees)` — collects all fees with zero cost.

---

### Impact Explanation

- **User fees are permanently lost**: credited to the malicious provider at request time with no refund path.
- **User application contract is permanently bricked**: any contract whose logic depends on `entropyCallback` (e.g., NFT mints, lottery draws, game outcomes) is stuck indefinitely in `CALLBACK_NOT_STARTED` state.
- **No on-chain recovery**: unlike the Connext case where funds are returned after expiry, Pyth Entropy has no expiry — the request and its blocked state persist forever.

---

### Likelihood Explanation

- Registration is fully permissionless and costless (no stake).
- The attacker only needs to register and accept requests — no special knowledge or privilege required.
- The attack is economically rational: collect fees, never spend gas on `revealWithCallback`.
- Any user who selects a non-default provider (e.g., via `requestV2(address provider, …)`) is exposed.

---

### Recommendation

1. **Escrow provider fees**: hold the provider's fee portion in a pending state and only release it upon successful `revealWithCallback`. Refund the user if the provider fails to fulfill within a deadline.
2. **Add a request expiry / user cancellation path**: allow users to cancel an unfulfilled request after a timeout (e.g., N blocks) and reclaim their fee.
3. **Require provider stake / bond**: require providers to lock collateral on registration that can be slashed for non-fulfillment, analogous to the penalty mechanism suggested in the Connext finding.

---

### Proof of Concept

```solidity
// 1. Attacker registers as a provider with a valid (but throwaway) commitment
entropy.register{value: 0}(
    1 wei,                  // low fee to attract users
    keccak256("seed"),      // valid commitment
    "",
    1000,                   // chainLength
    ""
);

// 2. Victim calls requestV2 — fee immediately credited to attacker
uint64 seq = entropy.requestV2{value: entropy.getFeeV2(attacker, 0)}(attacker, 0);
// At this point: attacker.accruedFeesInWei += providerFee
// Victim's contract is now waiting for _entropyCallback(seq, attacker, randomNumber)

// 3. Attacker never calls revealWithCallback — victim stuck forever

// 4. Attacker withdraws fees
entropy.withdraw(entropy.getProviderInfoV2(attacker).accruedFeesInWei);
// Attacker has collected fees; victim's callback will never fire
```

The root cause is in `requestHelper` at the fee-crediting line: [5](#0-4) 

combined with the absence of any expiry or refund path anywhere in `Entropy.sol`.
