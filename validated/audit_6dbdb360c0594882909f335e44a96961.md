### Title
Off-by-One in `isValidSigner` Rejects Valid Lazer Updates at Exact Expiry Block — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.isValidSigner` uses a strict `<` comparison against `expiresAt`, causing a trusted signer to be treated as invalid at the exact block where `block.timestamp == expiresAt`. Any `verifyUpdate` call in that block reverts with `"invalid signer"` even though the signer's validity period has not yet elapsed.

---

### Finding Description

In `PythLazer.sol`, the signer validity check is:

```solidity
// lazer/contracts/evm/src/PythLazer.sol  line 67
function isValidSigner(address signer) public view returns (bool) {
    return block.timestamp < trustedSignerToExpiresAtMapping[signer];
}
```

`expiresAt` is set by the owner via `updateTrustedSigner` and represents the intended last moment of validity. When `block.timestamp == expiresAt`, the expression `block.timestamp < expiresAt` evaluates to `false`, so the signer is rejected. The correct boundary should be `<=` to include the expiry block itself as valid.

`verifyUpdate` calls `isValidSigner` and reverts on `false`:

```solidity
// lazer/contracts/evm/src/PythLazer.sol  lines 103-105
if (!isValidSigner(signer)) {
    revert("invalid signer");
}
```

---

### Impact Explanation

Any Lazer price update submitted in the exact block where `block.timestamp == expiresAt` is rejected with `"invalid signer"`, even though the signer's configured validity window has not yet closed. Downstream protocols consuming Lazer prices via `verifyUpdate` in that block receive no price data, which can cause stale-price failures or missed liquidations/settlements in DeFi integrations.

---

### Likelihood Explanation

Moderate-low. EVM `block.timestamp` advances in discrete steps (roughly every 12 seconds on Ethereum mainnet). The owner sets `expiresAt` to a specific future timestamp. There is a non-trivial probability that a block lands exactly on that value, particularly on chains with 1-second block times (e.g., Arbitrum, Base, Optimism) where Lazer is deployed. The window is a single block, but on high-frequency chains it is a realistic collision.

---

### Recommendation

Change the strict `<` to `<=` in `isValidSigner`:

```solidity
function isValidSigner(address signer) public view returns (bool) {
    return block.timestamp <= trustedSignerToExpiresAtMapping[signer];
}
```

This makes `expiresAt` inclusive, consistent with the natural reading of "expires at time T" meaning the signer is valid through and including block T.

---

### Proof of Concept

1. Owner calls `updateTrustedSigner(signerAddr, T)` where `T` is a future timestamp.
2. A Lazer pusher signs a price payload at block time `T` and calls `verifyUpdate(update)`.
3. Inside `verifyUpdate`, `isValidSigner(signerAddr)` evaluates `T < T` → `false`.
4. The call reverts with `"invalid signer"`.
5. The identical update submitted one block earlier (timestamp `T-1`) would succeed, and one block later (timestamp `T+1`) would also correctly fail — only the boundary block is incorrectly rejected. [1](#0-0) [2](#0-1) [3](#0-2)
