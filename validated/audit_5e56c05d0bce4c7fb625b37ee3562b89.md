### Title
Entropy Provider DOS via Fee-Paid Spam Requests Inflating `numHashes` in `constructProviderCommitment` - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Pyth Entropy contract's `constructProviderCommitment` function performs a `while` loop that iterates `numHashes` times — where `numHashes` equals the gap between the current request's sequence number and the provider's last revealed sequence number. An attacker who pays the minimum fee can flood the provider with many outstanding requests, forcing the Fortuna keeper's every subsequent `revealWithCallback` call to execute an ever-growing number of keccak256 hashes on-chain. This directly maps to the Aleo analog: a fee-paying attacker can make the provider's reveal transactions prohibitively expensive or cause them to exceed the block gas limit, halting the Entropy service.

### Finding Description

**Root cause — `constructProviderCommitment` loop is proportional to outstanding requests:** [1](#0-0) 

Every call to `revealHelper` (invoked by both `reveal` and `revealWithCallback`) calls `constructProviderCommitment(req.numHashes, providerContribution)`, which loops `numHashes` times computing `keccak256`. [2](#0-1) 

**`numHashes` is set at request time as the gap between the new sequence number and the provider's last-revealed sequence number:** [3](#0-2) 

The contract itself documents this: *"Verification therefore may require computing multiple hashes (~the number of concurrent requests)."* [4](#0-3) 

**Attack path:**

1. Attacker registers as a provider with `feeInWei = 0` (no minimum fee enforced by the contract — `register` accepts any `feeInWei` including zero). [5](#0-4) 

2. Alternatively, the attacker simply pays the legitimate provider's fee (which is the Pyth protocol fee + provider fee) and submits thousands of requests against the default provider without ever triggering a reveal. Each request increments `providerInfo.sequenceNumber` but `currentCommitmentSequenceNumber` stays fixed.

3. When the Fortuna keeper eventually calls `revealWithCallback` for any of these requests, `req.numHashes` equals the full gap (potentially thousands), causing `constructProviderCommitment` to loop thousands of times.

4. If `maxNumHashes` is set to 0 (disabled), there is no cap on `numHashes`: [6](#0-5) 

5. Even with `maxNumHashes` set, the attacker can keep the outstanding count just below the threshold, since the Fortuna keeper's `update_commitments_if_necessary` only fires at 95% of `maxNumHashes`: [7](#0-6) 

**The `advanceProviderCommitment` mitigation is reactive, not preventive.** The keeper calls it only after the threshold is already reached, and the attacker can continuously re-fill the queue faster than the keeper can drain it. [8](#0-7) 

### Impact Explanation

- **Gas exhaustion on reveal transactions**: With thousands of outstanding requests, each `revealWithCallback` call requires thousands of keccak256 operations. At ~30 gas per keccak256 and 10,000 outstanding requests, that is ~300,000 extra gas per reveal — potentially exceeding block gas limits or making reveals economically unviable for the Fortuna keeper.
- **Entropy service unavailability**: If the keeper's reveal transactions consistently run out of gas or revert, legitimate users' randomness requests are never fulfilled. This is a direct service-level DOS against the Pyth Entropy product.
- **Provider fee drain**: The attacker pays the fee per request, but the cost to the attacker (fee × N requests) may be far less than the cost imposed on the provider (gas × N hashes × all future reveals).

### Likelihood Explanation

- Any unprivileged user can call `requestV2` or `request` by paying the provider fee.
- The attack requires no special access — only ETH to pay fees.
- The Fortuna keeper explicitly monitors for this condition and logs a warning, confirming the developers are aware of the risk but rely on a reactive mitigation: [9](#0-8) 
- Providers that do not set `maxNumHashes` (i.e., `maxNumHashes == 0`) have no on-chain protection at all.

### Recommendation

1. **Enforce a non-zero `maxNumHashes` at the contract level** for all providers, or cap `numHashes` in `constructProviderCommitment` to a protocol-defined maximum.
2. **Charge a fee proportional to `numHashes`** at request time, so that creating high-`numHashes` requests is economically costly to the attacker.
3. **Proactively advance the commitment** in the keeper before the threshold is reached, rather than reactively at 95%.
4. **Require providers to set `maxNumHashes > 0`** during registration, making `LastRevealedTooOld` a hard safety valve.

### Proof of Concept

```
1. Attacker calls requestV2(defaultProvider, randomBytes, 0) × 10,000 times,
   paying the minimum fee each time. No reveals are triggered.

2. providerInfo.sequenceNumber = 10,001
   providerInfo.currentCommitmentSequenceNumber = 1
   → All 10,000 requests have numHashes ≈ 10,000

3. Fortuna keeper calls revealWithCallback(provider, seqNum=1, ...).
   → constructProviderCommitment(10000, providerContribution) loops 10,000 times.
   → Gas cost: ~300,000+ extra gas just for the hash loop.

4. For seqNum=2: numHashes=9,999 → still ~300,000 gas.
   The keeper must pay this inflated gas for every single reveal.

5. If maxNumHashes == 0, there is no on-chain cap. The attacker can push
   numHashes to the uint32 max (4,294,967,295), making any reveal
   transaction physically impossible within block gas limits.

    function constructProviderCommitment(
        uint64 numHashes,
        bytes32 revelation
    ) internal pure returns (bytes32 currentHash) {
        currentHash = revelation;
        while (numHashes > 0) {
            currentHash = keccak256(bytes.concat(currentHash));
            numHashes -= 1;
        }

    let provider_address = chain_state.provider_address;
    let provider_info = contract
        .get_provider_info_v2(provider_address)
        .block(latest_safe_block) // To ensure we are not revealing sooner than we should
        .call()
        .await
        .map_err(|e| {
            anyhow!(
                "Error while getting provider info at block {}. error: {:?}",
                latest_safe_block,
                e
            )
        })?;
    if provider_info.max_num_hashes == 0 {
        return Ok(());
    }
    let threshold =
        ((provider_info.max_num_hashes as f64) * UPDATE_COMMITMENTS_THRESHOLD_FACTOR) as u64;
    let outstanding_requests =
        provider_info.sequence_number - provider_info.current_commitment_sequence_number;
    if outstanding_requests > threshold {
        // NOTE: This log message triggers a grafana alert. If you want to change the text, please change the alert also.
        tracing::warn!("Update commitments threshold reached -- possible outage or DDOS attack. Number of outstanding requests: {:?} Threshold: {:?}", outstanding_requests, threshold);
        let seq_number = provider_info.sequence_number - 1;
        let provider_revelation = chain_state
            .state
            .reveal(seq_number)
            .map_err(|e| anyhow!("Error revealing: {:?}", e))?;
        let contract_call =
            contract.advance_provider_commitment(provider_address, seq_number, provider_revelation);
        send_and_confirm(contract_call).await?;
    }
    Ok(())
