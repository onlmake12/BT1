Audit Report

## Title
Stale `EvictKey` on Parent After Child Removed via `remove_entry_and_descendants` Allows Low-Fee Tx to Evade Pool Eviction — (`tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-strips all link entries for every ID in the removal set before calling `remove_entry`, causing `update_ancestors_index_key` to resolve an empty ancestor set for each removed transaction. Ancestors of the removed subtree that remain in the pool never have `sub_descendant_weight` called on them and their `evict_key` is never recalculated, leaving their `evict_key.fee_rate` permanently inflated. This allows a low-fee parent transaction to evade eviction by retaining a stale, elevated fee-rate contributed by a since-removed high-fee child.

## Finding Description
`remove_entry_and_descendants` (L252–265) collects all descendants of the target ID, then iterates over the full removal set calling `remove_entry_links` on each ID (L257–259). `remove_entry_links` (L418–430) removes the ID from its parents' children sets, removes its parents from its own parents set, and finally calls `self.links.remove(id)` at L429, deleting the link entry from `TxLinksMap::inner` entirely.

Subsequently, `remove_entry` is called for each removed ID (L263). At L242, `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` is invoked. That function calls `self.links.calc_ancestors(&child.proposal_short_id())` (L433–434), which delegates to `calc_relative_ids` (links.rs L37–50). There, `self.inner.get(short_id)` returns `None` because the link entry was already deleted; `direct` becomes an empty set via `.unwrap_or_default()` (L47), and `calc_relation_ids` returns an empty `HashSet`. The `for anc_id in &ancestors` loop (L435–444) never executes: `sub_descendant_weight` is never called on any surviving parent P, and `e.evict_key = e.inner.as_evict_key()` (L442) is never re-run for P.

The `EvictKey` is computed as `fee_rate: descendants_feerate.max(feerate)` (entry.rs L243). When C was added, P's `descendants_fee/descendants_size/descendants_cycles` were incremented via `add_descendant_weight` and P's `evict_key.fee_rate` was elevated to reflect C's high fee rate. After C is removed via `remove_entry_and_descendants`, those fields and the indexed key are never corrected. The comment at L256 ("update links state for remove, so that we won't update_descendants_index_key in remove_entry") confirms the pre-deletion is intentional for the descendant direction but is an unintended side-effect for the ancestor direction. ## Impact Explanation
An unprivileged attacker can permanently inflate P's `evict_key.fee_rate` in the mempool. `next_evict_entry` (L380–385) selects the lowest-ranked entry by `evict_key`; P is skipped because its stale key shows a high fee rate. Any legitimate transaction M whose true fee rate falls between P's actual fee rate and P's inflated stale fee rate will be evicted before P. The attacker can repeat the inflation cycle at negligible cost, keeping P in the pool indefinitely while displacing legitimate transactions. This constitutes a low-cost, deterministic, and repeatable attack that corrupts the pool's eviction ordering across all affected nodes. Impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. [6](#0-5) 

## Likelihood Explanation
The attack requires no special privileges, no hashpower, and no social engineering. The `resolve_conflict` path (L305–332) is always enabled and triggered by any standard double-spend submission. The conflicting transaction C' does not need to spend P's output — it only needs to conflict with C on one input. The cost is two valid unconfirmed transactions plus one conflicting submission. The bug is deterministic and reproducible in a local test environment. The attacker can repeat the cycle indefinitely to maintain the inflation. [7](#0-6) 

## Recommendation
Collect each removed entry's ancestor set *before* calling `remove_entry_links`. In `remove_entry_and_descendants`, before the link-stripping loop, compute the ancestors of the root ID that lie outside the removal set. After all removals, iterate over those surviving ancestors and call `sub_descendant_weight` plus `as_evict_key()` recalculation for each removed entry. Alternatively, add an optional pre-computed ancestor set parameter to `remove_entry` that bypasses the `calc_ancestors` call when links have already been torn down.

#[test]
fn test_evict_key_stale_after_child_removed_via_conflict() {
    let mut pool = PoolMap::new(1000);

    // P: low fee (1 shannon, size 100)
    let tx_p = build_tx(vec![(&Byte32::zero(), 0)], 2);
    // C: high-fee child of P, also spends external UTXO_C
    let tx_c = build_tx(vec![(&tx_p.hash(), 0), (&h256!("0x2").into(), 0)], 1);
    // C': conflicts with C on UTXO_C, does NOT spend P's output
    let tx_c_prime = build_tx(vec![(&h256!("0x2").into(), 0)], 1);
    // M: medium fee, independent
    let tx_m = build_tx(vec![(&h256!("0x3").into(), 0)], 1);

    let entry_p = TxEntry::dummy_resolve(tx_p.clone(), 2, Capacity::shannons(1), 100);
    let entry_c = TxEntry::dummy_resolve(tx_c.clone(), 2, Capacity::shannons(1000), 100);
    let entry_m = TxEntry::dummy_resolve(tx_m.clone(), 2, Capacity::shannons(50), 100);

    pool.add_entry(entry_p, Status::Pending).unwrap();
    pool.add_entry(entry_c, Status::Pending).unwrap();

    // Simulate conflict: remove C via resolve_conflict path
    pool.resolve_conflict(&tx_c_prime);

    pool.add_entry(entry_m, Status::Pending).unwrap();

    // With the bug: P's evict_key.fee_rate is still ~1000, so M is evicted first
    let first_evict = pool.next_evict_entry(Status::Pending).unwrap();
    // Expected (correct): first_evict == tx_p (fee_rate 1 < 50)
    // Actual (buggy):     first_evict == tx_m (P's stale evict_key shows ~1000)
    assert_eq!(
        first_evict,
        tx_p.proposal_short_id(),
        "P should be evicted first (lowest true fee rate), not M"
    );
}

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    pub(crate) fn resolve_conflict(&mut self, tx: &TransactionView) -> Vec<ConflictEntry> {
        let mut conflicts = Vec::new();

        for i in tx.input_pts_iter() {
            if let Some(id) = self.edges.remove_input(&i) {
                let entries = self.remove_entry_and_descendants(&id);
                if !entries.is_empty() {
                    let reject = Reject::Resolve(OutPointError::Dead(i.clone()));
                    let rejects = std::iter::repeat_n(reject, entries.len());
                    conflicts.extend(entries.into_iter().zip(rejects));
                }
            }

            // deps consumed
            if let Some(x) = self.edges.remove_deps(&i) {

                    let entries = self.remove_entry_and_descendants(&id);

                        let reject = Reject::Resolve(OutPointError::Dead(i.clone()));
                        let rejects = std::iter::repeat_n(reject, entries.len());
                        conflicts.extend(entries.into_iter().zip(rejects));
                    }
                }
            }
        }

        conflicts

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
