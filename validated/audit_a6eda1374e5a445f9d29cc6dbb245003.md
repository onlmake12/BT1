### Title
Permissionless `Initialize` allows front-running to permanently hijack program authority — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary
`initialize()` has no deployer-only access control. Any signer can call it first with an attacker-controlled `authority` pubkey. Because the config PDA is deterministic and can only be initialized once, the legitimate deployer's call will always fail after a successful front-run, permanently locking the attacker's authority into the program.

### Finding Description

`validate_payer` only checks `is_signer` and `is_writable` — it does **not** verify the payer is the program deployer or any specific identity: [1](#0-0) 

`validate_config` only verifies the PDA derivation from `CONFIG_SEED` and the bump — no caller identity check: [2](#0-1) 

The `args.authority` field is a free, unconstrained parameter passed directly into storage: [3](#0-2) 

`config::create()` enforces a one-time-only write: if `format != 0`, it returns `AlreadyInitialized` and there is no re-initialization path: [4](#0-3) 

The config PDA is derived from `CONFIG_SEED` alone — fully deterministic and predictable by any observer: [5](#0-4) 

### Impact Explanation

Once the attacker's `Initialize` lands first:
- The config PDA stores the attacker's pubkey as `authority`.
- `validate_authority` in `initialize_publisher` enforces that the signer must match `config.authority`: [6](#0-5) 

- No legitimate publisher can ever be registered (`InitializePublisher` will always fail with `MissingRequiredSignature`).
- The deployer's `Initialize` fails with `AlreadyInitialized` and there is no recovery path.
- The program is permanently inoperable for legitimate use.

### Likelihood Explanation

- Solana validators expose pending transactions; front-running is well-known on Solana.
- The attack requires zero privileges — any funded wallet suffices.
- The config PDA address is fully computable off-chain before deployment.
- The attack window is the gap between program deployment and the deployer's `Initialize` transaction landing.

### Recommendation

Restrict `initialize()` to the program's upgrade authority (the deployer). On Solana, the upgrade authority is stored in the program's `ProgramData` account. The `initialize` processor should:
1. Accept the `ProgramData` account as an additional input.
2. Verify that `payer.key` matches the `upgrade_authority_address` stored in the `ProgramData` account.
3. Only then proceed to create the config PDA.

Alternatively, embed the expected authority pubkey as a program constant at compile time.

pub fn validate_config<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    bump: u8,
    program_id: &Pubkey,
    require_writable: bool,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let config = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    let (config_pda, expected_bump) =
        Pubkey::find_program_address(&[CONFIG_SEED.as_bytes()], program_id);
    ensure!(ProgramError::InvalidInstructionData, bump == expected_bump);
    ensure!(
        ProgramError::InvalidArgument,
        pubkey_eq(config.key, &config_pda)
    );
    if require_writable {
        ensure!(ProgramError::InvalidArgument, config.is_writable);
    }
    Ok(config)
}

pub fn initialize(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    args: &InitializeArgs,
) -> ProgramResult {
    let mut accounts = accounts.iter();
    let payer = validate_payer(accounts.next())?;
    let config = validate_config(accounts.next(), args.config_bump, program_id, true)?;
    let system = validate_system(accounts.next())?;

    let lamports = (Rent::get()?).minimum_balance(accounts::config::SIZE);

    invoke_signed(
        &system_instruction::create_account(
            payer.key,
            config.key,
            lamports,
            accounts::config::SIZE
                .try_into()
                .expect("unexpected overflow"),
            program_id,
        ),
        &[payer.clone(), config.clone(), system.clone()],
        &[&[CONFIG_SEED.as_bytes(), &[args.config_bump]]],
    )?;

    accounts::config::create(*config.data.borrow_mut(), args.authority)?;

    Ok(())
}
