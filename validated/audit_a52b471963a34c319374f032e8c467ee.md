### Title
Integer Overflow in `GetLastStateProofProcess::execute` Bypasses `GET_LAST_STATE_PROOF_LIMIT`, Enabling Unbounded Chain Traversal DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The limit guard at line 201 uses unchecked arithmetic. In Rust release mode, integer overflow wraps silently. An unprivileged light-client peer can craft a `GetLastStateProof` message with `last_n_blocks = 2^63` (i.e., `u64::MAX/2 + 1`) and an empty `difficulties` list. The multiplication `(last_n_blocks as usize) * 2` wraps to `0`, the guard evaluates `0 > 1000 = false`, and the server proceeds to traverse and build headers for every block in the chain.

---

### Finding Description

**The guard (lines 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) 

`GET_LAST_STATE_PROOF_LIMIT` is `1000`. [2](#0-1) 

**The overflow:**

- `last_n_blocks = 2^63` (fits in `u64`)
- `last_n_blocks as usize = 2^63` (on 64-bit)
- `2^63 * 2 = 2^64` → wraps to `0` in release mode (Rust wraps on overflow in release builds by default)
- `difficulties.len() = 0` (empty list)
- Guard: `0 + 0 > 1000` → **false** → guard is bypassed

**What executes after the bypass:**

At line 291, the branch condition `last_block_number - start_block_number <= last_n_blocks` is trivially true for any real chain (chain length << 2^63), so the server takes the "not enough blocks" path:

```rust
let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
``` [3](#0-2) 

This collects **every block number** from `start_block_number` to `last_block_number` into a `Vec`. Then `complete_headers` iterates over every entry, calling `snapshot.get_ancestor()`, `snapshot.get_block()`, and `snapshot.chain_root_mmr()` for each — all disk-backed operations. [4](#0-3) 

Additionally, the `reorg_last_n_numbers` path at line 245 also uses `last_n_blocks` but is bounded by `min(start_block_number, last_n_blocks)`, so it is also bounded by chain length. [5](#0-4)

### Impact Explanation

For a long-running mainnet node with N blocks:
- Allocates a `Vec<u64>` of N entries (~8N bytes)
- Performs N disk reads (`get_ancestor`, `get_block`, `chain_root_mmr`) per request
- Builds N `VerifiableHeader` structs in memory

A single crafted message forces O(chain_length) work. Multiple concurrent requests from different peers multiply the effect. The intended limit of 1000 blocks is completely nullified. Impact: CPU exhaustion, disk I/O saturation, and potential OOM on nodes with very long chains or under concurrent attack.

---

### Likelihood Explanation

The light-client protocol server is a production component. Any peer that can open a light-client protocol connection can send this message — no authentication, no PoW, no stake required. The crafted value `last_n_blocks = 2^63` is a single field in a flatbuffers-encoded message. The overflow is deterministic and reproducible in any release build.

---

### Recommendation

Replace the unchecked arithmetic with overflow-safe operations:

```rust
let total = self.message.difficulties().len()
    .checked_add((last_n_blocks as usize).checked_mul(2).unwrap_or(usize::MAX))
    .unwrap_or(usize::MAX);
if total > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Or add an explicit early rejection of `last_n_blocks` values that exceed the limit before any arithmetic:

```rust
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```
