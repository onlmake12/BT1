Audit Report

## Title
Malicious Peer Can Occupy `InflightBlocks` Slots to Delay Block Synchronization — (`sync/src/types/mod.rs`)

## Summary

`InflightBlocks::insert` enforces a strict one-peer-per-block rule: once a block hash is registered as in-flight for any peer, no other peer can claim that slot until the original entry is pruned by timeout. A malicious peer can advertise blocks it will never deliver, occupying in-flight slots and preventing legitimate peers from being assigned those blocks for the full `BLOCK_DOWNLOAD_TIMEOUT` (30 seconds). The behavior is confirmed by code and by an explicit test comment ("don't allow 2 peer for one block").

## Finding Description

`InflightBlocks::insert` at `sync/src/types/mod.rs:748–764` checks `inflight_states` via `entry()`. If the entry is `Occupied`, it returns `false` immediately — no reassignment, no timestamp comparison, no fallback peer:

```rust
Entry::Occupied(_entry) => return false,   // slot taken, no reassignment
Entry::Vacant(entry) => entry.insert(InflightState::new(peer)),
```

In `sync/src/synchronizer/block_fetcher.rs:271–284`, when `insert` returns `false`, the block is silently skipped — it is not added to the fetch list for the current peer.

`prune` at `sync/src/types/mod.rs:638–686` only clears entries where `value.timestamp + BLOCK_DOWNLOAD_TIMEOUT < now` (`BLOCK_DOWNLOAD_TIMEOUT = 30s`, confirmed at `util/constant/src/sync.rs:48`). Additionally, `prune` only iterates up to `tip + 20`, so blocks beyond that range are not pruned at all in a given cycle.

`remove_by_peer` at `sync/src/types/mod.rs:766–783` does free all slots for a peer on disconnection — but the attacker does not need to disconnect. They can stay connected and simply ignore `GetBlocks` requests, keeping their slots occupied for the full timeout window.

`MAX_BLOCKS_IN_TRANSIT_PER_PEER = 128` (`util/constant/src/sync.rs:16`) bounds how many slots one peer can occupy simultaneously, but multiple attacker connections multiply the effect linearly.

The test at `sync/src/tests/inflight_blocks.rs:11–13` explicitly documents and asserts this behavior as intentional design, confirming the code path is stable and not an accidental regression.

## Impact Explanation

A node under attack cannot download targeted blocks from legitimate peers until `BLOCK_DOWNLOAD_TIMEOUT` (30 s) expires per cycle. An attacker controlling multiple inbound connections (up to the node's peer limit) can stagger timeouts across different block ranges, sustaining a degraded or near-zero sync rate during IBD. After reconnecting, the attacker can repeat the pattern indefinitely. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (High, 10001–15000 points), since the attacker's cost is only maintaining connections and sending valid headers — both trivially cheap.

## Likelihood Explanation

Any unprivileged peer reachable by the victim can perform this attack. No special keys, hashpower, or privileged access is required. The attacker only needs to:
1. Establish one or more connections to the victim node.
2. Send valid `SendHeaders` messages for publicly known block hashes.
3. Ignore subsequent `GetBlocks` requests.

Block hashes are fully public, making target slot selection entirely predictable. The attacker can reconnect after being pruned/disconnected and repeat.

## Recommendation

When `insert` encounters an `Occupied` entry, compare the existing entry's `timestamp` against a fraction of `BLOCK_DOWNLOAD_TIMEOUT`. If the assigned peer has been unresponsive for a significant portion of the timeout, either:
- Transfer the in-flight assignment to the new peer (evict the stale entry and re-insert for the new peer), or
- Allow a secondary peer to be registered for the same block so that if the primary times out, the secondary can serve the request immediately without waiting for the next `prune` cycle.

Additionally, consider tightening `prune` to cover a wider block range, or triggering an early prune when a new peer advertises a block already in-flight for a peer that has been unresponsive.

## Proof of Concept

1. Attacker peer connects to victim CKB node.
2. Attacker sends `SendHeaders` for blocks `B1..Bn` (all publicly known hashes).
3. Victim's `BlockFetcher` calls `inflight_blocks.insert(attacker_peer, Bi)` for each `Bi` — all succeed (slots vacant).
4. Victim sends `GetBlocks([B1..Bn])` to attacker; attacker ignores the message.
5. Legitimate peer advertises the same `B1..Bn`; victim calls `inflight_blocks.insert(legit_peer, Bi)` — all return `false` (slots `Occupied` by attacker).
6. Victim cannot download `B1..Bn` from the legitimate peer until `BLOCK_DOWNLOAD_TIMEOUT` (30 s) elapses and `prune` fires.
7. Attacker reconnects and repeats, sustaining the delay.

Reproducible via unit test: extend `sync/src/tests/inflight_blocks.rs` to insert for a "malicious" peer, advance a mock clock by less than `BLOCK_DOWNLOAD_TIMEOUT`, then assert that `insert` for a "legitimate" peer returns `false` and that `prune` does not yet clear the entry.
