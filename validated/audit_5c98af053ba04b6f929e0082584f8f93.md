### Title
Provider Can Pre-Compute User Contribution in `requestV2()` No-UserRandomNumber Variants, Enabling Selective Reveal Censorship — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `random()` internal function used to generate the user's contribution in the no-`userRandomNumber` variants of `requestV2()` derives its output exclusively from publicly observable or estimable on-chain values. A registered Entropy provider, who possesses their full pre-committed hash chain, can predict the user's contribution before deciding whether to reveal their own value — enabling selective censorship of unfavorable outcomes without requiring validator collusion.

---

### Finding Description

The `random()` function at lines 1079–1089 of `Entropy.sol` generates the user's contribution as:

```solidity
function random() internal returns (bytes32) {
    _state.seed = keccak256(
        abi.encodePacked(
            block.timestamp,
            block.prevrandao,
            msg.sender,
            _state.seed
        )
    );
    return _state.seed;
}
``` [1](#0-0) 

This value is used as the user's contribution when callers invoke the no-`userRandomNumber` overloads:

```solidity
function requestV2() external payable override returns (uint64 assignedSequenceNumber) {
    assignedSequenceNumber = requestV2(getDefaultProvider(), random(), 0);
}
``` [2](#0-1) 

All four inputs to `random()` are either already public or estimable before the transaction is mined:

| Input | Observability |
|---|---|
| `block.prevrandao` | The **previous** block's RANDAO — already finalized and publicly known before the current block is built |
| `_state.seed` | A `bytes32` field in contract storage — readable by any observer via `eth_getStorageAt` |
| `msg.sender` | The requester's address — visible in the mempool before inclusion |
| `block.timestamp` | Set by the current block's validator; estimable within a narrow range (typically ±15 s) | [3](#0-2) 

The Entropy security model requires that the user's contribution be **secret to the provider** until after the provider has committed to their hash-chain value. The commit-reveal protocol comment in `Entropy.sol` explicitly states:

> "This protocol has the same security properties as the 2-party randomness protocol above: as long as either the provider or user is honest, the number r is random." [4](#0-3) 

When the user's contribution is derived from `random()`, this property is broken: the provider can compute `x_U` off-chain before submitting their reveal transaction, then compute `r = keccak256(x_U, x_i, 0)` using their known hash-chain value `x_i`, and selectively withhold the reveal if the outcome is unfavorable.

The `IEntropyV2` interface documentation acknowledges a weaker form of this risk:

> "This approach modifies the security guarantees such that a **dishonest validator and provider** can collude to manipulate the result." [5](#0-4) 

However, the actual attack does **not** require validator collusion. Because `block.prevrandao` is already finalized before the current block is built, the provider can compute the user's contribution for any candidate `block.timestamp` value independently. The provider only needs to enumerate a small range of plausible timestamps (e.g., ±30 seconds) to determine the set of possible user contributions and corresponding final random numbers.

The protocol design documentation also acknowledges the censorship attack vector:

> "Providers are trusted to reveal their random number regardless of what the final result is. Providers can compute r off-chain before they reveal x_i, which permits a censorship attack." [6](#0-5) 

With user-supplied randomness, the provider cannot compute `r` before the user reveals `x_U`. With the in-contract PRNG, the provider can compute `r` before deciding to reveal, because `x_U` is derivable from public state.

---

### Impact Explanation

Any registered Entropy provider serving requests made via the no-`userRandomNumber` `requestV2()` variants can:

1. Monitor the mempool for pending `requestV2()` calls.
2. Read `_state.seed` from contract storage.
3. Know `block.prevrandao` (already finalized).
4. Enumerate plausible `block.timestamp` values.
5. Compute the user's contribution `x_U` for each candidate timestamp.
6. Compute the final random number `r = keccak256(x_U, x_i, 0)` using their known hash-chain value `x_i`.
7. Withhold the reveal if `r` is unfavorable (e.g., in a lottery, NFT mint, or game outcome).

This breaks the core randomness guarantee for all consumers using the default `requestV2()` entry points, which are the simplest and most commonly used variants. Downstream contracts (lotteries, games, NFT mints) that rely on these variants receive manipulable randomness.

---

### Likelihood Explanation

**Medium-High.** The attack requires only that the provider:
- Be a registered provider (permissionless — anyone can register via `register()`).
- Monitor the mempool (standard infrastructure).
- Enumerate ~60 candidate `block.timestamp` values (trivial computation).

No validator collusion, key leakage, or governance access is required. The provider can execute this attack independently on any EVM chain where `block.prevrandao` is available (post-Merge Ethereum and forks).

---

### Recommendation

1. **Deprecate or clearly gate the no-`userRandomNumber` variants** for any use case where the provider is not fully trusted. The current documentation warning is insufficient — it understates the risk by implying validator collusion is required.
2. **Do not use `block.prevrandao` or `block.timestamp` as entropy sources** in the in-contract PRNG, as both are known before the transaction executes. If an in-contract PRNG is needed, incorporate a value that is unpredictable at request time, such as a hash of the previous block's hash combined with a contract-internal counter that is not readable before the transaction.
3. **Consider removing the no-`userRandomNumber` variants entirely** and requiring callers to supply their own `userRandomNumber`, preserving the full security guarantee of the commit-reveal protocol.

### Proof of Concept

```solidity
// Attacker is a registered provider with known hash chain x[0..N]
// User calls requestV2() (no userRandomNumber) — transaction visible in mempool

// Step 1: Read public state
bytes32 currentSeed = /* eth_getStorageAt(_state.seed slot) */;
bytes32 prevRandao  = /* block.prevrandao of the previous block (already finalized) */;
address sender      = /* msg.sender from the pending mempool tx */;

// Step 2: Enumerate candidate timestamps
for (uint256 ts = expectedTime - 30; ts <= expectedTime + 30; ts++) {
    bytes32 userContribution = keccak256(abi.encodePacked(ts, prevRandao, sender, currentSeed));
    
    // Step 3: Compute final random number using known hash chain value x_i
    bytes32 r = keccak256(abi.encodePacked(userContribution, x_i, bytes32(0)));
    
    // Step 4: If r is unfavorable (e.g., user wins lottery), withhold reveal
    if (isFavorable(r)) {
        // Submit reveal transaction
    }
    // Otherwise, do not reveal — user's request is censored
}
```

The provider calls `revealWithCallback()` only when the computed `r` is favorable, and withholds the reveal otherwise. The user's request remains permanently unfulfilled (or fulfilled only when the provider benefits). [7](#0-6)
