Looking at the actual code carefully:

### Title
Targeted Legitimate Peer Eviction via `protect_peers=0` in Connection-Time Pass of `try_evict_inbound_peer` — (`network/src/peer_registry.rs`)

---

### Summary

The connection-time protection pass computes `protect_peers = candidate_peers.len() >> 1`. When an attacker engineers exactly one legitimate peer to survive the ping and message passes, `protect_peers` becomes `0`, the `sort_then_drop` call is a no-op, and the sole remaining legitimate peer is deterministically evicted. The attacker repeats this on every reconnect attempt, achieving a persistent eclipse.

---

### Finding Description

**Root cause — `protect_peers = 0` when one candidate remains:** [1](#0-0) 

```rust
let protect_peers = candidate_peers.len() >> 1;   // line 185
sort_then_drop(&mut candidate_peers, protect_peers, |peer1, peer2| {
    peer2.connected_time.cmp(&peer1.connected_time)
});
```

When `candidate_peers.len() == 1`, `protect_peers = 1 >> 1 = 0`.

`sort_then_drop` with `n = 0`: [2](#0-1) 

```rust
if list.len() > n {                        // 1 > 0 → true
    list.truncate(list.len() - n);         // truncate(1 - 0) = truncate(1) → no-op
}
```

The single remaining peer is **not** removed from the candidate list and proceeds to the network-group eviction step, where it is the only member of its group and is selected for eviction.

**Attack preconditions (all attacker-controlled):**

| Parameter | Value |
|---|---|
| `max_inbound` | `2 * EVICTION_PROTECT_PEERS + 1 = 17` |
| Attacker peers | 16, with `ping_rtt = Some(Duration::from_secs(0))` and `last_ping_protocol_message_received_at` set to "just now" |
| Legitimate peers | 1, with normal (higher) ping and older last-message timestamp | [3](#0-2) 

**Step-by-step eviction trace:**

1. Attacker connects 16 peers; legitimate peer connects as peer 17 (slots full).
2. Attacker connects peer 18 → `non_whitelist_inbound (17) >= max_inbound (17)` → `try_evict_inbound_peer` called. [4](#0-3) 

3. **Ping pass** (`n = EVICTION_PROTECT_PEERS = 8`): sorts descending by ping; truncates last 8 (lowest ping = attacker peers). Remaining: 8 attacker peers + 1 legitimate peer = **9 candidates**. [5](#0-4) 

4. **Message pass** (`n = 8`): sorts descending by seconds-since-last-message; truncates last 8 (most recent = attacker peers). Remaining: **1 legitimate peer**. [6](#0-5) 

5. **Connection-time pass**: `protect_peers = 1 >> 1 = 0` → no-op → legitimate peer stays.
6. **Network-group step**: 1 peer, 1 group → that peer is chosen → legitimate peer evicted. [7](#0-6) 

**Why the attacker can reliably control both passes:**

- *Ping*: `ping_rtt` is measured in whole seconds (`p.as_secs()`). Attacker nodes respond to pings in <1 ms → `ping_rtt = 0 s`. Legitimate peers with any measurable latency are sorted ahead of attacker peers and remain as candidates.
- *Messages*: Attacker nodes send a ping-protocol message immediately before the eviction fires, setting `last_ping_protocol_message_received_at` to "now". Legitimate peers' last-message timestamps are not attacker-controlled and will be older.

---

### Impact Explanation

The attacker can prevent **any** legitimate peer from maintaining an inbound connection to the victim node. Every time the legitimate peer reconnects, the same eviction sequence fires and ejects it. This enables a persistent **eclipse attack**: the victim node's inbound peer set is entirely attacker-controlled, allowing the attacker to filter blocks, transactions, and peer-discovery messages.

---

### Likelihood Explanation

The attack requires controlling `max_inbound − 1` IP addresses (16 with the minimal configuration). This is a non-trivial but realistic capability for a motivated adversary (e.g., cloud VMs, rented IPs). The attack is **not** probabilistic once the attacker's peers have established ping measurements; the ping-pass and message-pass outcomes are deterministic given the attacker's control over response timing. The `protect_peers = 0` path is reachable on every eviction call when the setup conditions hold.

---

### Recommendation

Replace the right-shift with a formula that guarantees at least one peer is protected when candidates remain:

```rust
// Before (vulnerable):
let protect_peers = candidate_peers.len() >> 1;

// After (safe):
let protect_peers = (candidate_peers.len() + 1) / 2;
// or equivalently:
let protect_peers = candidate_peers.len().div_ceil(2);
```

This ensures that when `candidate_peers.len() == 1`, `protect_peers = 1`, so `sort_then_drop` removes the sole remaining peer from the candidate list, causing `evict_group` to be empty and `try_evict_inbound_peer` to return `None` — correctly refusing the new connection rather than evicting the only long-connected legitimate peer.

### Proof of Concept

```rust
#[test]
fn test_protect_peers_zero_evicts_legitimate_peer() {
    use std::time::{Duration, Instant};
    let mut peer_store = PeerStore::default();
    // max_inbound = 2*8+1 = 17
    let mut registry = PeerRegistry::new(17, 3, false, vec![], true);

    // Connect 16 attacker peers
    for sid in 0..16u32 {
        registry.accept_peer(random_addr(), sid.into(),
            RawSessionType::Inbound, &mut peer_store).unwrap();
    }
    // Set attacker peers: 0-second ping + very recent message
    let now = Instant::now();
    for peer in registry.peers_mut().values_mut()
        .filter(|p| p.is_inbound() && !p.is_whitelist)
    {
        peer.ping_rtt = Some(Duration::from_millis(1));
        peer.last_ping_protocol_message_received_at = Some(now);
    }

    // Connect legitimate peer (slot 17)
    let legit_addr = random_addr();
    registry.accept_peer(legit_addr.clone(), 100u32.into(),
        RawSessionType::Inbound, &mut peer_store).unwrap();
    // Give legitimate peer older message timestamp
    let legit_id = extract_peer_id(&legit_addr).unwrap();
    let sid = registry.get_key_by_peer_id(&legit_id).unwrap();
    registry.get_peer_mut(sid).unwrap()
        .last_ping_protocol_message_received_at =
            Some(now - Duration::from_secs(60));

    // Trigger eviction (18th connection)
    let result = registry.accept_peer(random_addr(), 200u32.into(),
        RawSessionType::Inbound, &mut peer_store).unwrap();

    // Assert: the evicted peer is the legitimate peer
    let evicted = result.expect("eviction must occur");
    assert_eq!(
        extract_peer_id(&evicted.connected_addr).unwrap(),
        legit_id,
        "legitimate peer was evicted — protect_peers=0 bug confirmed"
    );
}
```
