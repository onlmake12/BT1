### Title
Unbounded `last_n_numbers` Vec Allocation via Crafted `difficulty_boundary` in `GetLastStateProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged light-client peer can send a `GetLastStateProof` message with `last_n_blocks = 0`, an empty `difficulties` list, and a `difficulty_boundary` of `1`. This causes `difficulty_boundary_block_number` to resolve to block `0` (or `start_block_number`), and the guard at line 313 never fires, so `last_n_numbers` is allocated as a Vec of every block number from `0` to `last_block_number` — potentially millions of entries on mainnet. The server then performs a DB lookup per entry in `complete_headers`, exhausting memory and CPU.

---

### Finding Description

**Entry guard (lines 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

With `difficulties = []` and `last_n_blocks = 0`, this evaluates to `0 + 0 = 0 > 1000` → **false**. The guard passes. [1](#0-0) 

**Branch selection (lines 291–292):**

```rust
if last_block_number - start_block_number <= last_n_blocks {
```

With `last_n_blocks = 0` and any non-genesis tip, this is `N <= 0` → **false**. We enter the `else` branch. [2](#0-1) 

**Boundary resolution (lines 299–311):**

`get_first_block_total_difficulty_is_not_less_than(start_block_number=0, last_block_number, &difficulty_boundary=1)` checks block 0 first. Since the genesis block's total difficulty is ≥ 1, it immediately returns `(0, genesis_difficulty)`. So `difficulty_boundary_block_number = 0`. [3](#0-2) 

**The broken guard (lines 313–316):**

```rust
if last_block_number - difficulty_boundary_block_number < last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
```

With `difficulty_boundary_block_number = 0` and `last_n_blocks = 0`, this is `last_block_number < 0` on a `u64` — **always false**. The guard never fires. [4](#0-3) 

**Unbounded allocation (lines 318–319):**

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
```

With `difficulty_boundary_block_number = 0` and `last_block_number = 10_000_000` (realistic mainnet), this allocates a `Vec<u64>` of 10 million entries (~80 MB), then `complete_headers` performs multiple snapshot/DB lookups for each entry. [5](#0-4) 

The `GET_LAST_STATE_PROOF_LIMIT = 1000` constant is never actually applied to the size of `last_n_numbers` itself — only to the sum `difficulties.len() + last_n_blocks * 2` at the entry point. [6](#0-5) 

---

### Impact Explanation

A single malicious peer can trigger O(N) memory allocation and O(N) sequential DB reads (where N = chain height, ~10M on mainnet) per message. This is a **remote memory exhaustion / CPU DoS** against any CKB full node running the light-client protocol server. No authentication, PoW, or privileged role is required — any connected light-client peer can send this message.

---

### Likelihood Explanation

The light-client protocol is enabled in production CKB nodes. The crafted message is trivially constructable: valid `last_hash`/`start_hash` pointing to real main-chain blocks are the only non-trivial requirement, and those are publicly observable. The attack is repeatable and can be sent from multiple peers simultaneously.

---

### Recommendation

After computing `difficulty_boundary_block_number`, enforce an explicit upper bound on the size of `last_n_numbers` before the `collect()`:

```rust
let span = last_block_number.saturating_sub(difficulty_boundary_block_number) as usize;
if span > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many last_n blocks");
}
let last_n_numbers = (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
```

Additionally, enforce a minimum value for `last_n_blocks` (e.g., ≥ 1) or rewrite the entry guard to bound `last_n_numbers` size directly rather than relying on `last_n_blocks * 2`.

---

### Proof of Concept

On a chain with height H (e.g., H = 10,000,000):

1. Observe the current tip hash `T` and genesis hash `G`.
2. Craft a `GetLastStateProof` message:
   - `last_hash = T`
   - `start_hash = G`
   - `start_number = 0`
   - `last_n_blocks = 0`
   - `difficulties = []`
   - `difficulty_boundary = 1` (any value ≤ genesis total difficulty)
3. Send to a CKB node with the light-client server enabled.
4. The server allocates `(0..H).collect::<Vec<u64>>()` = H entries, then calls `complete_headers` with H block lookups.
5. Repeat from multiple peers to amplify memory/CPU exhaustion.
