Audit Report

## Title
ExecV2 Missing Bounds Check Causes Hard VM Termination Instead of `SLICE_OUT_OF_BOUND` — (`script/src/scheduler.rs`, `script/src/syscalls/exec_v2.rs`)

## Summary
`ExecV2::ecall` performs no bounds validation on `offset`/`length` before queuing `Message::ExecV2` and yielding. When `process_message_box` subsequently calls `sc.load_data` with the raw user-supplied values, any out-of-bounds access returns `VMError::SnapshotDataLoadError`, which the `?` operator propagates as a hard scheduler error rather than setting A0 to `SLICE_OUT_OF_BOUND` and allowing the script to continue. Both `Exec::ecall` and `Spawn::ecall` explicitly perform this bounds check and return `SLICE_OUT_OF_BOUND`, establishing a clear invariant that `ExecV2` breaks.

## Finding Description
In `ExecV2::ecall` (`exec_v2.rs` lines 46–67), `offset` and `length` are extracted from the `bounds` register and placed directly into `DataLocation` with no validation before being pushed to the message box. The `ExecV2` struct holds only `id` and `message_box` — no `snapshot2_context` — so it structurally cannot perform the probe call that `Spawn::ecall` uses.

In `Scheduler::process_message_box` (`scheduler.rs` lines 496–504), the `Message::ExecV2` arm calls `sc.load_data(&args.location.data_piece_id, args.location.offset, args.location.length)?`. The `?` propagates `VMError::SnapshotDataLoadError` (returned when offset exceeds cell data length) as a hard error through `process_message_box` → `iterate_process_results` → `iterate_inner` → `iterate_outer` → `run`, terminating the scheduler with an error.

By contrast, `Spawn::ecall` (`spawn.rs` lines 112–131) first calls `sc.load_data(&data_piece_id, 0, 0)` to obtain `full_length`, catches `SnapshotDataLoadError` → `INDEX_OUT_OF_BOUND`, then checks `offset >= full_length` and `offset+length > full_length`, returning `SLICE_OUT_OF_BOUND` in both cases before queuing any message. `Exec::ecall` (`exec.rs` lines 138–152) performs the same checks synchronously. `LoadCellData` also catches `SnapshotDataLoadError` and converts it to `INDEX_OUT_OF_BOUND` rather than propagating it.

The exploit path: script calls ExecV2 with `bounds = (cell_data_length + 1) << 32` → `ExecV2::ecall` queues the message without validation → `process_message_box` calls `sc.load_data` with out-of-bounds offset → `SnapshotDataLoadError` is returned → `?` propagates it as a hard error → scheduler terminates with an error → transaction verification fails. The script never receives `SLICE_OUT_OF_BOUND` in A0 and cannot handle the condition gracefully.

## Impact Explanation
This is an incorrect implementation of CKB-VM system syscall behavior, matching the allowed impact: **Incorrect implementation or behavior of CKB-VM or system scripts — High (10001–15000 points)**. Any `ScriptVersion::V2` script that calls ExecV2 with an out-of-bounds offset and expects to receive `SLICE_OUT_OF_BOUND` in A0 (to implement fallback logic) will instead cause the entire VM to terminate with a hard error, making the transaction fail verification. This breaks the documented and implemented invariant of the exec-family syscalls.

## Likelihood Explanation
Any unprivileged script author deploying a `ScriptVersion::V2` script can trigger this. The trigger requires only crafting a transaction where the script calls ExecV2 with `offset >= cell_data_length`. This is trivially constructable and locally testable with no special privileges or external conditions required. The condition is deterministic and 100% reproducible.

## Recommendation
In `process_message_box` for `Message::ExecV2` (`scheduler.rs` around line 496), replicate the bounds-checking pattern from `Spawn::ecall`:
1. Call `sc.load_data(&args.location.data_piece_id, 0, 0)` to obtain `full_length`.
2. Catch `SnapshotDataLoadError` → set A0 to `INDEX_OUT_OF_BOUND` on the old machine, then `continue`.
3. Check `args.location.offset >= full_length` → set A0 to `SLICE_OUT_OF_BOUND`, then `continue`.
4. If `args.location.length > 0`, check `args.location.offset + args.location.length > full_length` → set A0 to `SLICE_OUT_OF_BOUND`, then `continue`.
5. Only then proceed with the validated `sc.load_data` call and VM replacement.

## Proof of Concept
1. Deploy a `ScriptVersion::V2` lock script that:
   - Calls ExecV2 with `index=0` (a valid cell dep), `source=CellDep`, `place=CellData`, and `bounds = (cell_data_length + 1) << 32` (offset one past the end, length=0).
   - After the ExecV2 call returns, checks if A0 == `SLICE_OUT_OF_BOUND` (3); if so, exits with code 0 (success).
2. Submit a transaction using this lock script.
3. **Expected (per invariant):** A0 = 3, script exits 0, transaction validates.
