### Title
Unbounded `pending_get_block_proposals` Map Growth via Unauthenticated `GetBlockProposal` Messages Leads to Memory and CPU DoS — (`File: sync/src/types/mod.rs`)

---

### Summary

`SyncState::pending_get_block_proposals` is an unbounded `DashMap` that any connected peer can inflate by repeatedly sending `GetBlockProposal` relay messages containing unique proposal IDs absent from the tx pool. There is no per-peer rate limit, no cap on the map's total size, and no cleanup on peer disconnect. The map is only drained by a periodic timer callback (`prune_tx_proposal_request`) that performs an O(n) clone-and-iterate over the entire accumulated map, making both memory and CPU exhaustion reachable.

---

### Finding Description

**Unbounded insertion path**

`SyncState` declares the map with no capacity bound:

```rust

``` [1](#0-0) 

The insertion function performs no size check before writing:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [2](#0-1) 

**Attacker-controlled entry point**

`GetBlockProposalProcess::execute()` enforces only a per-message size ceiling (`max_block_proposals_limit × max_uncles_num`), then unconditionally calls `insert_get_block_proposals` with every proposal ID that is absent from the local tx pool:

```rust
let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
    .into_iter()
    .filter(|short_id| !fetched_transactions.contains_key(short_id))
    .collect();

// Cache request, try process on timer
self.relayer
    .shared()
    .state()
    .insert_get_block_proposals(self.peer, not_exist_proposals);
``` [3](#0-2) 

There is no rate limit on how many `GetBlockProposal` messages a peer may send, and no cap on how many entries a single peer may contribute to the map.

**No cleanup on disconnect**

`SyncState::disconnected` removes inflight blocks and peer state, but never touches `pending_get_block_proposals`:

```rust

    let removed_inflight_blocks_count = self.write_inflight_blocks().remove_by_peer(pi);
    // ...

}
``` [4](#0-3) 

Entries from a disconnected (or banned) peer persist in the map indefinitely until the next timer drain.

**O(n) drain on timer**

`drain_get_block_proposals` clones the entire map before clearing it, and `prune_tx_proposal_request` then iterates over every accumulated entry to issue tx-pool lookups and relay responses:

---

### Impact Explanation

An attacker who fills `pending_get_block_proposals` with N entries causes:

1. **Memory exhaustion** — each `ProposalShortId` (10 bytes) plus `HashSet<PeerIndex>` overhead accumulates in process memory with no eviction between timer ticks.
2. **CPU spike on every timer tick** — `drain_get_block_proposals` performs an O(n) clone; `prune_tx_proposal_request` then performs an O(n) iteration plus a bulk tx-pool lookup proportional to n. If the timer fires while the map is maximally inflated, the relay service stalls for the duration of that O(n) work, delaying compact-block processing and transaction relay for all peers.

In the worst case, if the timer interval is long relative to the attacker's message rate, the map can grow to `(timer_interval_seconds × messages_per_second) × max_proposals_per_message` entries before being drained, with each drain cycle consuming proportional CPU.

---

### Likelihood Explanation

The attack requires only a standard inbound or outbound P2P connection — no stake, no PoW, no privileged role. `GetBlockProposal` is a normal relay protocol message. The per-message ceiling (`max_block_proposals_limit × max_uncles_num`, typically 1500 × 2 = 3000 on mainnet) is the only throttle, and there is no per-peer or global rate limit on how many such messages may be sent. A single malicious peer can sustain a high message rate for the duration of its connection, and entries survive peer disconnection.

---

### Recommendation

1. **Hard cap on map size**: In `insert_get_block_proposals`, reject new entries once `pending_get_block_proposals.len()` exceeds a configurable maximum (e.g., `max_block_proposals_limit × max_uncles_num`).
2. **Per-peer quota**: Track how many entries each peer has contributed and reject further insertions once a per-peer limit is reached.
3. **Cleanup on disconnect**: In `SyncState::disconnected`, remove all entries whose `HashSet<PeerIndex>` contains only the disconnecting peer.
4. **Rate-limit `GetBlockProposal` messages** at the protocol layer, consistent with how other relay messages are rate-limited.

---

### Proof of Concept

```
1. Establish a P2P connection to a CKB node (standard inbound peer).

2. In a tight loop, send GetBlockProposal relay messages:
      - Each message contains max_block_proposals_limit * max_uncles_num
        unique ProposalShortId values (10-byte random values not present
        in the target node's tx pool).
      - No valid PoW or stake is required.

3. For each message, GetBlockProposalProcess::execute() calls
   insert_get_block_proposals() with all IDs that are absent from the
   tx pool, inserting them into pending_get_block_proposals with no
   size check.

4. Between timer ticks of prune_tx_proposal_request, the map grows
   proportionally to the message rate.

5. When the timer fires, drain_get_block_proposals() clones the entire
   inflated map (O(n) allocation), clears it (O(n)), and
   prune_tx_proposal_request iterates over every entry (O(n) tx-pool
   lookups), stalling the relay service for the duration.

6. Repeating across the connection lifetime causes sustained memory
   pressure and periodic CPU spikes proportional to the accumulated
   map size.
```

    async fn prune_tx_proposal_request(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let get_block_proposals = self.shared().state().drain_get_block_proposals();
        let tx_pool = self.shared.shared().tx_pool_controller();

        let fetch_txs = tx_pool
            .fetch_txs(
                get_block_proposals
                    .iter()
                    .map(|kv_pair| kv_pair.key().clone())
                    .collect(),
            )
            .await;
        if let Err(err) = fetch_txs {
            debug_target!(
                crate::LOG_TARGET_RELAY,
                "relayer prune_tx_proposal_request internal error: {:?}",
                err,
            );
            return;
        }

        let txs = fetch_txs.unwrap();

        let mut peer_txs = HashMap::new();
        for (id, peer_indices) in get_block_proposals.into_iter() {
            if let Some(tx) = txs.get(&id) {
                for peer_index in peer_indices {
                    let tx_set = peer_txs.entry(peer_index).or_insert_with(Vec::new);
                    tx_set.push(tx.clone());
                }
            }
        }

        let mut relay_bytes = 0;
        let mut relay_proposals = Vec::new();
        for (peer_index, txs) in peer_txs {
            for tx in txs {
                let data = tx.data();
                let tx_size = data.total_size();
                if relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH {
                    send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals))
                        .await;
                    relay_bytes = tx_size;
                } else {
                    relay_bytes += tx_size;
                }
                relay_proposals.push(data);
            }
            if !relay_proposals.is_empty() {
                send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals)).await;
                relay_bytes = 0;
            }
        }
