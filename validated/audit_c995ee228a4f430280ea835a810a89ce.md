### Title
`BlockFilter` Protocol Handler Implements Peer-Ban Mechanism but Fails to Apply It for Unsolicited Response Messages - (File: `sync/src/filter/mod.rs`)

### Summary

The `BlockFilter` protocol handler has a fully functional peer-ban mechanism (`nc.ban_peer()`) that is correctly applied for malformed messages. However, when a remote peer sends unsolicited response messages (`BlockFilters`, `BlockFilterHashes`, `BlockFilterCheckPoints`) — which are server-to-client responses that should only arrive after an explicit request — the handler returns `Status::ignored()` instead of a ban-triggering status code. The code even contains a `// TODO: ban remote peer` comment explicitly acknowledging this gap. As a result, any unprivileged peer can flood a CKB full node with unsolicited filter data indefinitely without being banned.

### Finding Description

In `sync/src/filter/mod.rs`, the `BlockFilter` struct implements `CKBProtocolHandler`. Its `received()` method correctly bans peers that send malformed (unparseable) messages: [1](#0-0) 

The `process()` method also has a ban-dispatch path that calls `nc.ban_peer()` when `status.should_ban()` returns a duration: [2](#0-1) 

However, in `try_process()`, when a peer sends any of the three unsolicited response message types, the handler returns `Status::ignored()` with only a warning log and an unimplemented TODO: [3](#0-2) 

`Status::ignored()` maps to `StatusCode::Ignored = 101`, which is in the 1xx informational range: [4](#0-3) 

The `should_ban()` method only triggers for 4xx codes (400–499), so `Status::ignored()` never produces a ban: [5](#0-4) 

Unlike the `HolePunching` protocol handler, which has a `rate_limiter` field to throttle per-peer message rates, `BlockFilter` has no rate limiter: [6](#0-5) 

The ban mechanism is fully available — `nc.ban_peer()` is wired through the `CKBProtocolContext` trait and used in the same file — but it is simply never invoked for this message class.

### Impact Explanation

A malicious unprivileged peer connected to the Filter protocol can send an unbounded stream of `BlockFilters`, `BlockFilterHashes`, or `BlockFilterCheckPoints` messages without ever being banned. Each message is deserialized (via `packed::BlockFilterMessageReader::from_compatible_slice`), dispatched through `try_process()`, and logged as a warning. Because no ban is issued and no rate limiter exists in `BlockFilter`, the attacker can sustain this indefinitely, consuming CPU (deserialization), memory (logging), and I/O bandwidth on the victim node. The node has no automated defense mechanism to cut off the misbehaving peer.

### Likelihood Explanation

The Filter protocol is a server-side protocol exposed by full nodes to serve block filter data. Any peer that successfully connects and negotiates the Filter protocol can immediately begin sending unsolicited response messages. No special privilege, key, or majority hashpower is required. The attack is trivially scriptable: connect, send `BlockFilters` in a loop, never get banned.

### Recommendation

In the unsolicited-message match arm of `try_process()`, return a ban-triggering 4xx status code instead of `Status::ignored()`:

```rust
packed::BlockFilterMessageUnionReader::BlockFilters(_)
| packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)
| packed::BlockFilterMessageUnionReader::BlockFilterCheckPoints(_) => {
    // remote peer should not send block filter to us without asking
    StatusCode::ProtocolMessageIsMalformed
        .with_context("received unsolicited block filter response")
}
```

This causes `process()` to call `nc.ban_peer(peer, BAD_MESSAGE_BAN_TIME, ...)` automatically through the existing dispatch logic at lines 88–97, consistent with how all other protocol handlers in the codebase treat protocol violations.

### Proof of Concept

1. Connect a custom peer to a CKB full node that supports the Filter protocol (`SupportProtocols::Filter`).
2. In a tight loop, send a valid `BlockFilterMessage` containing a `BlockFilters` variant (a response message) without any prior `GetBlockFilters` request.
3. Observe that the node logs `"Received unexpected message from peer"` warnings repeatedly but never disconnects or bans the peer.
4. Confirm via `get_banned_addresses` RPC that the peer's address never appears in the ban list, while the node's CPU and log I/O are continuously consumed.
