### Title
Bootnode-Mode `inbound_eviction` Bypasses Group-Diversity Protection, Enabling Single-Subnet Slot Monopolization — (`network/src/network.rs`, `network/src/peer_registry.rs`)

---

### Summary

In `bootnode_mode`, the `inbound_eviction` function evicts inbound peers purely by HashMap iteration index (`index & 0x1 != 0`) with zero group-diversity consideration. This is a completely separate code path from `try_evict_inbound_peer`, which has sophisticated group-diversity protection. An attacker controlling many peers from a single `/16` subnet can exploit the random 50% survival rate to maintain disproportionate slot occupancy across repeated eviction cycles.

---

### Finding Description

**Two separate eviction mechanisms exist and they are not equivalent:**

**Path 1 — `try_evict_inbound_peer` (peer_registry.rs:142–211):**
Called from `accept_peer` when `non_whitelist_inbound >= max_inbound`. It:
- Protects 8 lowest-ping peers [1](#0-0) 
- Protects 8 most-recently-messaging peers [2](#0-1) 
- Protects half of remaining by longest connection time [3](#0-2) 
- Groups remaining by `network_group()`, picks the **largest group**, evicts one randomly — explicit group-diversity enforcement [4](#0-3) 

**Path 2 — `inbound_eviction` (network.rs:580–604):**
Called on **every `SessionOpen` event** (line 734), before `accept_peer`. In bootnode mode, when `max_inbound <= non_whitelist_inbound + 10`: [5](#0-4) 

It collects all non-whitelist inbound peers and evicts every peer at an odd HashMap iteration index:
```rust
.filter_map(|(index, peer)| if index & 0x1 != 0 { Some(peer) } else { None })
``` [6](#0-5) 

**No group lookup. No diversity check. No protection tiers.** The eviction is effectively random (Rust `HashMap` uses a randomized seed), giving every peer — regardless of subnet — a 50% survival probability.

**Call site confirms `inbound_eviction` fires first, before `accept_peer`:** [7](#0-6) 

Then `accept_peer` (which calls `try_evict_inbound_peer`) runs afterward: [8](#0-7)

### Impact Explanation

The claimed impact of **"consensus deviation"** is **overstated and inaccurate**. Bootnodes are peer-discovery nodes, not consensus participants. They do not validate blocks, relay transactions for inclusion, or vote on chain tips.

The **actual impact** is:

- A single-subnet attacker can maintain ~N/2 inbound slots on a bootnode across repeated eviction cycles, because `inbound_eviction` gives them the same 50% survival rate as diverse honest peers, and the attacker can reconnect evicted peers faster than organic honest peers.
- The bootnode then serves **biased peer address lists** to new nodes performing initial peer discovery.
- New nodes bootstrapping through a compromised bootnode receive a skewed view of the network, increasing the feasibility of **eclipse attack setup** against those new nodes — but this is indirect and requires additional steps beyond what is described.

This is a real design flaw: the group-diversity invariant enforced by `try_evict_inbound_peer` is completely absent from `inbound_eviction`. However, the scope impact of "consensus deviation" does not follow directly from bootnode slot monopolization alone.

---

### Likelihood Explanation

- Requires `bootnode_mode = true` (an explicit operator configuration, not the default for regular nodes).
- Requires the attacker to sustain N/2 simultaneous connections from a single `/16` subnet — resource-intensive but not infeasible for a motivated attacker with a cloud subnet.
- The trigger condition (`max_inbound <= non_whitelist_inbound + 10`) is easily reached on a busy bootnode.
- The attack is repeatable and self-reinforcing across cycles.

---

### Recommendation

In `inbound_eviction`, replace the index-parity eviction with the same group-diversity logic used in `try_evict_inbound_peer`: group surviving peers by `network_group()`, identify over-represented groups, and preferentially evict from those groups. Alternatively, call `try_evict_inbound_peer` directly (or a batch variant of it) instead of the flat index-based eviction.

### Proof of Concept

```
1. Configure node: bootnode_mode=true, max_inbound=N
2. Attacker connects N/2 peers from 1.2.x.x (/16), honest peers fill remaining N/2 from diverse /16s
3. A new connection arrives → inbound_eviction fires (non_whitelist_inbound = N >= max_inbound - 10)
4. ~N/2 peers evicted by index parity (random 50% each, no group preference)
5. Attacker immediately reconnects evicted peers from 1.2.x.x
6. Honest peers from diverse subnets reconnect more slowly (organic)
7. After reconnection race: attacker holds ~N/2 slots, honest holds ~N/4
8. Repeat from step 3 → attacker maintains structural majority on bootnode
9. New nodes bootstrapping via this bootnode receive peer lists dominated by 1.2.x.x addresses
```

    fn inbound_eviction(&self) -> Vec<PeerIndex> {
        if self.network_state.config.bootnode_mode {
            let status = self.network_state.connection_status();

            if status.max_inbound <= status.non_whitelist_inbound.saturating_add(10) {
                self.network_state
                    .with_peer_registry(|registry| {
                        registry
                            .peers()
                            .values()
                            .filter(|peer| peer.is_inbound() && !peer.is_whitelist)
                            .map(|peer| peer.session_id)
                            .collect::<Vec<SessionId>>()
                    })
                    .into_iter()
                    .enumerate()
                    .filter_map(|(index, peer)| if index & 0x1 != 0 { Some(peer) } else { None })
                    .collect()
            } else {
                Vec::new()
            }
        } else {
            Vec::new()
        }
    }
