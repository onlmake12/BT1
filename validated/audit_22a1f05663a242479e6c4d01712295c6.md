Audit Report

## Title
Duplicate Indexes in `GetBlockTransactions` Bypass Uniqueness Check, Enabling Bandwidth Amplification Attack — (File: `sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute()` validates only the raw length of the `indexes` array against `MAX_RELAY_TXS_NUM_PER_BATCH` (32767) but performs no deduplication. An attacker can send a `GetBlockTransactions` message with 32767 identical indexes, causing the victim node to serialize and transmit that transaction 32767 times in a single `BlockTransactions` response, with no outbound byte-size cap applied in this code path.

## Finding Description
The length guard at line 37 of `get_block_transactions_process.rs` uses the raw (pre-dedup) count:

```rust
if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
    return StatusCode::ProtocolMessageIsMalformed ...
}
``` [1](#0-0) 

No deduplication follows. The `filter_map` at lines 61–71 resolves every index independently against the stored block, producing up to 32767 identical `TransactionView` clones:

```rust
let transactions = self.message.indexes().iter()
    .filter_map(|i| block.transactions().get(Into::<u32>::into(i) as usize).cloned())
    .collect::<Vec<_>>();
``` [2](#0-1) 

All clones are serialized into a single `BlockTransactions` message and sent with no `MAX_RELAY_TXS_BYTES_PER_BATCH` guard: [3](#0-2)

## Impact Explanation
**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

With 30 requests/second, each carrying 32767 duplicate indexes to a 1 KB transaction, the victim node emits approximately 30 × 32 MB ≈ 960 MB/s of outbound traffic per attacker connection. `MAX_RELAY_PEERS` is 128, so the theoretical ceiling is far higher. This directly exhausts the victim node's outbound bandwidth, degrading its ability to serve legitimate peers, propagate blocks, and relay transactions. [6](#0-5) 

## Likelihood Explanation
Any connected peer can trigger this immediately without special privilege, leaked keys, or victim mistakes. The attacker only needs any valid block hash stored by the victim — trivially obtained from the public chain. The attack is repeatable, stateless, and cheap: a single low-bandwidth connection suffices to saturate the victim's uplink at 30 req/s within the rate limit. [7](#0-6) 

## Recommendation
Add a deduplication check in `GetBlockTransactionsProcess::execute()` immediately after the length check, mirroring the pattern in `GetTransactionsProcess` and `GetBlockProposalProcess`:

```rust
let indexes_set: HashSet<u32> = self.message.indexes().iter()
    .map(Into::<u32>::into).collect();
if indexes_set.len() != self.message.indexes().len() {
    return StatusCode::RequestDuplicate.with_context("Duplicate transaction index");
}

let uncle_set: HashSet<u32> = self.message.uncle_indexes().iter()
    .map(Into::<u32>::into).collect();
if uncle_set.len() != self.message.uncle_indexes().len() {
    return StatusCode::RequestDuplicate.with_context("Duplicate uncle index");
}
```

Additionally, apply a `MAX_RELAY_TXS_BYTES_PER_BATCH` guard on the outbound `BlockTransactions` response, consistent with the batching logic in `get_transactions_process.rs` (lines 87–101) and `get_block_proposal_process.rs` (lines 79–95). [8](#0-7) 

## Proof of Concept
**Minimal manual steps:**

1. Connect a custom peer to a CKB full node via the RelayV3 protocol.
2. Identify any stored block hash `H` and a valid transaction index `i` within that block.
3. Construct a `GetBlockTransactions` message: `block_hash = H`, `indexes = [i; 32767]`.
4. Send the message. Observe the victim node transmitting a `BlockTransactions` response containing 32767 copies of the same transaction.
5. Repeat at 30 req/s (within the rate limit) and measure outbound bandwidth on the victim.
