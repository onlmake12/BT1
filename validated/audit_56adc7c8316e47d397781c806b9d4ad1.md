Audit Report

## Title
`InflightBlocks::prune` Skips Timeout Eviction for Inflight Entries Above `tip+20`, Enabling Temporary Chain Stall — (`sync/src/types/mod.rs`)

## Summary

`InflightBlocks::prune` iterates only up to `tip + 20` in a BTreeMap, leaving all inflight entries with `key.number > tip + 20` unvisited on every prune cycle. The secondary cleanup path (`mark_slow_block` / `trace.retain`) covers only entries at `key.number <= tip + 1`. A malicious peer that silently drops `GetBlocks` requests for blocks in `(tip+20, tip+8192]` will have its inflight slots occupied without timeout eviction until the tip naturally advances to bring those entries into the prune window. During this window, no other peer can be scheduled to fetch those same blocks, causing a chain stall of up to 30 seconds per block slot.

## Finding Description

**Root cause — `prune` hard-stops at `tip + 20`:** [1](#0-0) 

The BTreeMap iteration breaks immediately when `key.number > tip + 20`. Entries above this threshold are never visited for timeout eviction.

**Secondary path — `mark_slow_block` hard-stops at `tip + 1`:** [2](#0-1) 

Only entries `<= tip + 1` are added to `trace_number`. The `trace.retain` path at lines 713–742 only evicts entries previously inserted into `trace_number`, so it also covers at most `<= tip + 1`.

**Gap:** entries in `(tip+1, tip+20]` are covered by the `prune` timeout loop; entries in `(tip+20, tip+8192]` are covered by neither path.

**`BlockFetcher` can insert up to `tip + BLOCK_DOWNLOAD_WINDOW`:** [3](#0-2) [4](#0-3) 

`BLOCK_DOWNLOAD_WINDOW = 8192`, so inflight entries can be inserted up to `tip + 8192`.

**`insert` blocks other peers from fetching the same block:** [5](#0-4) 

If a block is already in `inflight_states`, `insert` returns `false`. In `block_fetcher.rs` lines 271–276, when `insert` returns `false`, the block is simply skipped — no other peer is scheduled to fetch it.

**`can_fetch` is zero while slots are occupied:** [6](#0-5) 

With 32 entries inflight and `task_count = INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32`, `can_fetch = 0`. [7](#0-6) 

**Exploit flow:**
1. Malicious peer connects and advertises headers for blocks `tip+21` through `tip+52`.
2. Victim's `BlockFetcher` inserts those 32 blocks into `inflight_states` and sends `GetBlocks` to the malicious peer.
3. Malicious peer silently drops all requests.
4. Chain advances to `tip+20` using honest peers (blocks `tip+1`–`tip+20` are within the prune window and will be evicted/re-requested if timed out).
5. Block `tip+21` is needed to advance further, but it is stuck in the malicious peer's inflight slot. No other peer can be scheduled to fetch it. Chain stalls.
6. After `BLOCK_DOWNLOAD_TIMEOUT = 30s`, the tip is at `tip+20`, so `tip+21` is now within the prune window (`tip+20 + 20 = tip+40`). The entry is evicted and the peer is punished via `punish(2)`: `task_count = 32 >> 2 = 8`.
7. After 3 such evictions (≈90s), `task_count` reaches 0 and the peer is disconnected. [8](#0-7) [9](#0-8) 

The peer can reconnect immediately (fresh `task_count = 32`) and repeat the attack indefinitely.

**Correction to the "permanent" claim:** The stall is not permanent. As the tip advances to `tip+20` via honest peers, the malicious peer's entries begin falling into the prune window and are evicted one by one (each after 30s). However, the chain advances at ~1 block per 30s during the attack instead of the normal rate, and the attacker can sustain this by reconnecting after disconnection.

## Impact Explanation

A single malicious peer causes the chain to advance at approximately 1 block per 30 seconds (instead of normal throughput) for ~90 seconds before being disconnected. With multiple coordinated malicious peers reconnecting in rotation, this degradation can be sustained indefinitely with minimal attacker cost (just maintaining TCP connections and dropping packets). This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points).**

## Likelihood Explanation

Any unprivileged peer can execute this attack. It requires no special knowledge, no cryptographic material, and no on-chain resources. The attacker only needs to connect to a victim node, send valid headers for future blocks, and then drop `GetBlocks` messages. The attack is repeatable after each disconnection with zero cost beyond maintaining a TCP connection.

## Recommendation

Extend the prune timeout loop to cover all inflight entries, not just those within `tip + 20`. The `tip + 20` optimization was intended to avoid iterating the full BTreeMap on every prune cycle, but it creates the eviction gap. Options:

1. **Remove the `tip + 20` cap entirely** and rely on the BTreeMap's sorted order to terminate early once entries are too recent to have timed out (i.e., break when `value.timestamp + BLOCK_DOWNLOAD_TIMEOUT >= now` for entries well above tip).
2. **Track per-peer insertion timestamps separately** so that a peer's oldest inflight entry can be checked regardless of block number.
3. **Extend `mark_slow_block`** to cover up to `tip + 20` (or higher) so that the `trace.retain` path also covers the gap range.

## Proof of Concept

**Minimal unit test plan:**

```rust
// In sync/src/types/mod.rs tests:
// 1. Create an InflightBlocks with one peer.
// 2. Insert 32 blocks at numbers tip+21 through tip+52 for the malicious peer.
// 3. Advance time by BLOCK_DOWNLOAD_TIMEOUT + 1ms.
// 4. Call prune(tip).
// 5. Assert: inflight_states still contains all 32 entries (not evicted).
// 6. Assert: download_schedulers still contains the peer (not disconnected).
// 7. Assert: insert(honest_peer, tip+21) returns false (slot still occupied).
```

This directly demonstrates that timed-out entries above `tip+20` survive `prune` and block honest peers from fetching the same blocks.
