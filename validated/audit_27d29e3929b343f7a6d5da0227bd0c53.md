Audit Report

## Title
Decompression Bypasses Per-Protocol `max_frame_length` Bound, Allowing Up to 4× Memory Amplification per Frame — (`network/src/compress.rs`)

## Summary

`LengthDelimitedCodecWithCompress::decode` enforces `max_frame_length` on compressed wire bytes via `length_delimited.decode`, but after detecting `COMPRESS_FLAG` it bounds the decompressed allocation only by the global `MAX_UNCOMPRESSED_LEN` (8 MB), never by the per-protocol `max_frame_length`. The encoder path (`process`) already performs this check, but the decoder does not. Any unprivileged TCP peer can exploit this asymmetry to force allocations up to 4× the intended per-protocol limit per frame.

## Finding Description

`MAX_UNCOMPRESSED_LEN` is `1 << 23` = 8 MB. [1](#0-0) 

`SupportProtocols::Sync` sets `max_frame_length` = 2 MB; `RelayV3` sets it to 4 MB. [2](#0-1) 

`CKBProtocol::build()` wires `LengthDelimitedCodecWithCompress` with the per-protocol `max_frame_length` for all protocols built via `new_with_support_protocol` (compress defaults to `true`): [3](#0-2) 

In `decode()`, after `length_delimited.decode(src)` accepts the compressed frame (enforcing `max_frame_length` on wire bytes), the only guard on the decompressed size is the global constant:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN { … return Err(…); }
let mut buf = BytesMut::zeroed(decompressed_bytes_len);
``` [4](#0-3) 

There is no check `decompressed_bytes_len > self.length_delimited.max_frame_length()`. The encoder path (`process`) already performs exactly this check: [5](#0-4) 

but it is absent from the decoder path. The gap is structural: the encoder and decoder enforce different upper bounds on message size.

**Amplification ratios per protocol:**

| Protocol | `max_frame_length` | Max decompressed | Amplification |
|---|---|---|---|
| Sync | 2 MB | 8 MB | **4×** |
| RelayV3 | 4 MB | 8 MB | **2×** |
| LightClient/Filter | 2 MB | 8 MB | **4×** |

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.** With N concurrent peers each pipelining compressed frames whose wire size is ≤ `max_frame_length` but whose snappy header claims ~8 MB decompressed, the node allocates N × 8 MB per batch. The per-protocol frame limits exist precisely to bound this memory consumption; bypassing them with a 4× amplification factor across many concurrent connections creates a realistic OOM path. The allocation happens before decompression succeeds or fails, so even a malformed payload triggers the allocation.

## Likelihood Explanation

The exploit requires only a TCP connection. No proof-of-work, no keys, no privileged role. The snappy format is public and `decompress_len` reads a varint from the stream header that the attacker fully controls. The path is reachable on both `Sync` and `RelayV3`, the two highest-traffic protocols. Snappy compresses repetitive data (e.g., zeros) to well under 10% of original size, so an 8 MB decompressed payload easily fits within a 2–4 MB wire frame limit.

## Recommendation

In `decode()`, after reading `decompressed_bytes_len`, add a check against the per-protocol limit before allocating:

```rust
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
    return Err(io::ErrorKind::InvalidData.into());
}
```

This mirrors the guard already present in `process()` and closes the asymmetry between the encoder and decoder paths. [6](#0-5) 

## Proof of Concept

1. Construct ~8 MB of zeros.
2. Snappy-compress it — the compressed form will be well under 4 MB (snappy achieves >90% compression on repetitive data).
3. Prepend `COMPRESS_FLAG` (0x80) and send as a single `RelayV3` frame; wire length ≤ 4 MB passes `length_delimited.decode`.
4. On the receiver, `decompress_len` returns ~8 MB; the check `> MAX_UNCOMPRESSED_LEN` passes (8 MB − 1 < 8 MB); `BytesMut::zeroed(~8 MB)` is allocated.
5. Repeat across multiple connections or in a tight loop to amplify memory pressure toward OOM.

A unit test can confirm: construct a `LengthDelimitedCodecWithCompress` with `max_frame_length = 4 MB`, craft a frame whose compressed size < 4 MB but whose snappy header claims decompressed size > 4 MB, call `decode()`, and assert it returns `Err` — currently it returns `Ok` with an 8 MB buffer. [7](#0-6)
