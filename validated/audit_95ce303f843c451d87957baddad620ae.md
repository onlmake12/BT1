### Title
Unbounded While Loop in `executeCallback` Can Be Griefed to Permanently Block Request Fulfillment - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol`'s `executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past all fulfilled requests. An unprivileged attacker can inflate the number of fulfilled requests ahead of a target request, causing the loop to exhaust all available gas and permanently prevent that request from ever being fulfilled.

### Finding Description

Inside `executeCallback`, after clearing the fulfilled request, the following loop runs unconditionally:

```solidity
// Echo.sol lines 169–174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest(_state.firstUnfulfilledSeq)`, which computes a `keccak256` hash, performs an SLOAD on the fixed-size `requests[32]` array, and potentially a second SLOAD on the `requestsOverflow` mapping. The result is then passed to `isActive(Request memory req)`, which forces a full storage-to-memory copy of the `Request` struct (3+ SLOADs per iteration). [2](#0-1) [3](#0-2) 

The number of iterations equals the number of consecutive fulfilled (inactive) sequence numbers starting from `firstUnfulfilledSeq`. This count is fully attacker-controlled.

**Attack path:**

1. Attacker calls `registerProvider` with zero fees (`baseFeeInWei = 0`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`). [4](#0-3) 

2. Attacker calls `requestPriceUpdatesWithCallback` N times, paying only `pythFeeInWei` per request (which may be zero or very low). This creates requests with sequence numbers 1, 2, …, N. [5](#0-4) 

3. Attacker calls `executeCallback` for requests 2, 3, …, N (all except request 1). `firstUnfulfilledSeq` stays at 1 because request 1 is still active.

4. When the legitimate provider (or anyone) attempts to call `executeCallback` for request 1:
   - `clearRequest(1)` executes successfully.
   - The while loop starts at `firstUnfulfilledSeq = 1` and must scan through all N−1 fulfilled sequence numbers before finding the next active request (or reaching `currentSequenceNumber`).
   - With N large enough, the loop exhausts the block gas limit and the entire transaction reverts.
   - Because the revert undoes `clearRequest(1)`, request 1 remains permanently active but unfulfillable. [6](#0-5) 

The `executeCallback` function has no access control, so any address can submit requests and execute callbacks. [7](#0-6) 

The `EchoState` confirms `currentSequenceNumber` is a `uint64`, so the sequence space is effectively unbounded. [8](#0-7) 

The IEcho interface itself documents the linear gas scaling of this scan:

> "Gas Usage: This function's gas cost scales linearly with the number of requests between firstUnfulfilledSeq and currentSequenceNumber." [9](#0-8) 

### Impact Explanation

A targeted request can be permanently locked: its fee is trapped in the contract, the requester's callback is never delivered, and the provider can never claim the fee. If the attacker targets the very first outstanding request, all subsequent `executeCallback` calls for that request will also revert, since `firstUnfulfilledSeq` can never advance past it. This constitutes a permanent, targeted DoS of the Echo fulfillment path for any specific request.

### Likelihood Explanation

`executeCallback` is permissionless and `registerProvider` allows zero-fee providers. The attacker's cost is N × `pythFeeInWei` (protocol fee) plus gas for N−1 callback executions. With `pythFeeInWei` set to zero or a small value, and with warm SLOADs costing ~100 gas each, roughly 100,000–300,000 fulfilled requests are needed to exhaust a 30M-gas block. At low fee settings this is economically feasible for a motivated attacker. The code comment at line 166–167 even acknowledges the gas concern, indicating the developers are aware of the scaling issue. [10](#0-9) 

### Recommendation

Replace the unbounded while loop with a counter-based approach: maintain a separate `uint64 fulfilledCount` or use a doubly-linked list of active requests (as the TODO comment at line 167 already suggests). Alternatively, cap the number of iterations the while loop may perform per transaction and defer the remainder to a separate `advanceFirstUnfulfilledSeq()` call, so that `executeCallback` cannot be made to OOG by the size of the fulfilled-request gap.

### Proof of Concept

```solidity
// 1. Attacker registers as a zero-fee provider
echo.registerProvider(0, 0, 0);

// 2. Create N requests (only pythFeeInWei cost each, possibly 0)
for (uint i = 0; i < N; i++) {
    echo.requestPriceUpdatesWithCallback{value: pythFee}(
        attackerProvider, block.timestamp, priceIds, callbackGasLimit
    );
}
// Sequence numbers 1..N are now active; firstUnfulfilledSeq = 1

// 3. Fulfill requests 2..N, leaving request 1 active
for (uint64 seq = 2; seq <= N; seq++) {
    echo.executeCallback(attackerProvider, seq, updateData, priceIds);
}
// firstUnfulfilledSeq is still 1 (request 1 still active)

// 4. Provider attempts to fulfill request 1 — while loop must scan N-1 entries → OOG revert
echo.executeCallback{gas: 30_000_000}(provider, 1, updateData, priceIds);
// Transaction reverts; request 1 remains permanently unfulfillable
```

    function findRequest(
        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];
        }
    }
