### Title
No Manager Transfer Mechanism in Scheduler Contract Causes Permanently Stuck Subscription Funds — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract (Pyth Pulse) permanently binds the subscription manager role to `msg.sender` at creation time with no mechanism to transfer or update that role. When a smart contract creates a subscription on behalf of users — a common and explicitly documented use case — and that contract later loses the ability to call manager-gated functions, all deposited ETH is permanently locked with no recovery path.

---

### Finding Description

In `Scheduler.sol`, `createSubscription` unconditionally assigns `msg.sender` as the subscription manager:

```solidity
// Map subscription ID to manager
_state.subscriptionManager[subscriptionId] = msg.sender;
``` [1](#0-0) 

The `onlyManager` modifier gates both fund withdrawal and subscription modification:

```solidity
modifier onlyManager(uint256 subscriptionId) {
    if (_state.subscriptionManager[subscriptionId] != msg.sender) {
        revert SchedulerErrors.Unauthorized();
    }
    _;
}
``` [2](#0-1) 

`withdrawFunds` and `updateSubscription` are both exclusively gated by this modifier: [3](#0-2) [4](#0-3) 

A search across all `.sol` files confirms there is **no** `transferManager`, `setManager`, `transferSubscription`, or any equivalent function anywhere in the codebase. The `IScheduler` interface also exposes no such function: [5](#0-4) 

The `SchedulerGovernance` contract provides no admin override for manager-gated functions either — admin actions are limited to fee configuration: [6](#0-5) 

The Pulse README explicitly documents that the Manager role is expected to be held by protocols: *"Typically an EOA or a protocol controlled by its governance."* [7](#0-6) 

---

### Impact Explanation

When a smart contract (DeFi protocol, multisig, factory, or governance-controlled proxy) calls `createSubscription`, it becomes the permanent, irrevocable manager. If that contract:

1. Does not implement a forwarding function to call `withdrawFunds` or `updateSubscription` on the Scheduler (e.g., a simple multisig or governance contract that was not designed with Scheduler integration in mind),
2. Is upgraded via a proxy pattern and the new implementation drops those capabilities, or
3. Is a factory/router that creates subscriptions on behalf of users but does not forward manager calls back to the user,

then all ETH deposited into the subscription's balance is permanently stuck. `withdrawFunds` sends funds to `msg.sender` (the manager), so only the manager can recover them: [8](#0-7) 

Additionally, `updateSubscription` being manager-only means the subscription parameters (feeds, triggers, whitelist) can never be changed, and the subscription can never be deactivated to allow full balance withdrawal.

---

### Likelihood Explanation

The Scheduler is explicitly designed for protocol-level use. Any protocol that integrates Pulse by having its own contract call `createSubscription` — without also implementing a complete set of forwarding functions for every manager action — is permanently exposed. This is a realistic and common integration pattern. The probability increases over time as protocols upgrade their contracts.

---

### Recommendation

Add a `transferManager` function gated by `onlyManager`:

```solidity
function transferManager(uint256 subscriptionId, address newManager) external onlyManager(subscriptionId) {
    require(newManager != address(0), "new manager is zero address");
    address oldManager = _state.subscriptionManager[subscriptionId];
    _state.subscriptionManager[subscriptionId] = newManager;
    emit ManagerTransferred(subscriptionId, oldManager, newManager);
}
```

This mirrors the pattern used by the budget contracts in the referenced Boost report and is consistent with how `Ownable2StepUpgradeable` handles ownership transfers in the rest of the Pyth codebase. [9](#0-8) 

---

### Proof of Concept

1. Deploy a simple factory contract `SubscriptionFactory` that calls `scheduler.createSubscription{value: minimumBalance}(params)` and stores the returned `subscriptionId`, but does not implement any function to call `withdrawFunds` or `updateSubscription`.
2. A user calls `SubscriptionFactory.create(...)`. The factory becomes the manager.
3. The user attempts to call `scheduler.withdrawFunds(subscriptionId, amount)` directly — reverts with `Unauthorized` because `msg.sender` is the user, not the factory.
4. The factory has no `withdrawFunds` forwarding function, so the ETH is permanently locked.
5. Even the Scheduler admin/owner cannot recover the funds — `withdrawFunds` has no admin override path. [2](#0-1) [10](#0-9)

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
