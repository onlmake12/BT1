Looking at the exact code in `remove_unconfirmed_tx` and the array sizes:

### Title
Out-of-Bounds Index Panic in `remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

### Summary

`remove_unconfirmed_tx` guards the `block_unconfirmed_txs` array access with a `tx_age >= len` check, but then unconditionally indexes `confirm_blocks_to_failed_txs[tx_age - 1]` without a matching bound check. When a tracked transaction is evicted from the pool after more than `MAX_CONFIRM_BLOCKS` (1000) blocks, `tx_age - 1` exceeds the array's last valid index (999), causing a Rust index-out-of-bounds panic and crashing the node.

### Finding Description

`TxConfirmStat` is initialized with both `block_unconfirmed_txs` and `confirm_blocks_to_failed_txs` of length `MAX_CONFIRM_BLOCKS = 1000`: [1](#0-0) 

In `remove_unconfirmed_tx`, the code correctly detects when a tx has aged beyond the sliding window: [2](#0-1) 

However, the subsequent `count_failure` branch indexes `confirm_blocks_to_failed_txs` using the **unclamped** `tx_age - 1` value: [3](#0-2) 

`confirm_blocks_to_failed_txs` has length 1000 (valid indices `0..=999`). When `tx_age = 1001`, `tx_age - 1 = 1000` is out of bounds. Rust panics unconditionally on out-of-bounds slice indexing in release builds, crashing the process.

### Impact Explanation

A node crash (panic) terminates the CKB node process. Any unprivileged peer or RPC caller who can submit a transaction that (a) enters the fee estimator's tracking and (b) is later evicted with `count_failure=true` after 1001+ blocks can trigger this. This is a **denial-of-service** against any node running the `ConfirmationFraction` fee estimator.

### Likelihood Explanation

The path is fully reachable without any privilege:

1. **Transaction submission** — attacker submits a valid low-fee-rate tx via P2P relay (`submit_remote_tx`) or RPC. The tx is accepted into the pool and tracked by `accept_tx` → `track_tx` → `add_unconfirmed_tx`. [4](#0-3) 

2. **Block processing** — each new block calls `commit_block` → `process_block` → `move_track_window`. After exactly 1000 blocks, the tx's slot is rotated into `old_unconfirmed_txs`. [5](#0-4) 

3. **Eviction** — after block 1001, any eviction path that calls `reject_tx` (pool size limit, RBF replacement, explicit removal) invokes `drop_tx_inner(tx_hash, true)` → `remove_unconfirmed_tx(..., count_failure=true)`. [6](#0-5) [7](#0-6) 

4. **Panic** — `tx_age = 1001`, `confirm_blocks_to_failed_txs[1000]` panics. [8](#0-7) 

Low-fee transactions routinely sit in pools for hundreds of blocks. No special privilege, hashpower, or key material is required.

### Recommendation

Add a bounds check before indexing `confirm_blocks_to_failed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This mirrors the existing guard on `block_unconfirmed_txs` and silently drops the failure sample for transactions older than the tracking window, which is the correct semantic (they are already counted as `old_unconfirmed_txs`).

### Proof of Concept

```rust
// Boundary test: tx_age = 999, 1000, 1001 with count_failure=true
let buckets = vec![FeeRate::from_u64(1000)];
let mut stat = TxConfirmStat::new(buckets, 1000, 0.993);

// Simulate tx tracked at entry_height=0, bucket_index=0
stat.add_unconfirmed_tx(0, FeeRate::from_u64(1000));

// tx_age=999: confirm_blocks_to_failed_txs[998] — valid, no panic
stat.remove_unconfirmed_tx(0, 999, 0, true);  // OK

// Re-add for next test
stat.add_unconfirmed_tx(0, FeeRate::from_u64(1000));
// tx_age=1000: confirm_blocks_to_failed_txs[999] — last valid index, no panic
stat.remove_unconfirmed_tx(0, 1000, 0, true); // OK

// Re-add for next test
stat.add_unconfirmed_tx(0, FeeRate::from_u64(1000));
// tx_age=1001: confirm_blocks_to_failed_txs[1000] — OUT OF BOUNDS → PANIC
stat.remove_unconfirmed_tx(0, 1001, 0, true); // thread 'main' panicked: index out of bounds
```
