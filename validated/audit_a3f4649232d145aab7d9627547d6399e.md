Audit Report

## Title
`ScriptHashTypeVerifier` Skips Type Script Hash-Type Validation, Enabling Cheap Contextual-Verification DoS — (File: `verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and enforces `ENABLED_SCRIPT_HASH_TYPE` only for each output's **lock script**, never for the optional **type script**. A transaction carrying an output whose type script uses a disallowed hash type (e.g., `Data3` = 6) passes `NonContextualTransactionVerifier::verify()` and is forwarded into the contextual pipeline — cell-dep resolution, input resolution, and script-group construction — before being rejected by `select_version()`. An attacker who owns any live cells can exploit this repeatedly at zero ongoing fee cost.

## Finding Description

`ScriptHashTypeVerifier::verify()` (lines 796–814) reads:

```rust
for output in self.transaction.outputs() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { … }
    } else { … }
}
``` [1](#0-0) 

The loop body calls `output.lock().hash_type()` exclusively. `output.type_().to_opt()` is never called; the type script's hash type is never validated here.

`NonContextualTransactionVerifier::verify()` calls `self.script_hash_type.verify()` as its final gate before the transaction enters the contextual pipeline: [2](#0-1) 

The downstream check that does catch the bad hash type is `select_version()` in `script/src/types.rs`:

```rust
hash_type => {
    return Err(ScriptError::InvalidScriptHashType(format!(
        "The ScriptHashType/{:?} has not been activated, and is not permitted for use.",
        hash_type
    )));
}
``` [3](#0-2) 

This check is reached only after cell-dep and input resolution (RocksDB lookups) and script-group construction — work that the non-contextual gate was designed to avoid. Note that the VM itself is **not** instantiated (the error is returned before execution begins), but the cell-resolution and script-group-building overhead is real and measurable.

**Exploit path:**
1. Attacker owns any live cell (valid input).
2. Attacker crafts a transaction spending that cell with an output whose type script has `hash_type = 6` (`Data3`).
3. Transaction passes `NonContextualTransactionVerifier::verify()` (the broken gate).
4. Transaction enters contextual verification: inputs and cell-deps are resolved from RocksDB, script groups are built.
5. `select_version()` returns `InvalidScriptHashType` — transaction is rejected, no fee is charged.
6. The input cell is still unspent; the attacker resubmits in a tight loop.

The existing unit tests confirm only the lock-script path is exercised: [4](#0-3) 

There is no test for a disallowed type-script hash type.

## Impact Explanation

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. The attacker forces repeated contextual-verification cycles (RocksDB cell lookups, script-group construction) on the tx-pool verification worker at zero ongoing fee cost. The attacker's cells are never consumed (transactions are always rejected), so a single small UTXO sustains an indefinite stream of expensive-to-verify, free-to-submit transactions.

## Likelihood Explanation

Any user who holds a live CKB cell can trigger this. Crafting a `Script` with `hash_type = 6` is trivial via the JSON-RPC `send_transaction` endpoint. No mining power, no privileged access, and no key material beyond the attacker's own wallet is required. The attack is repeatable with no cooldown and no fee drain.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script when present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        check_hash_type(output.lock().hash_type())?;
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the existing `TryInto<ScriptHashType>` + `ENABLED_SCRIPT_HASH_TYPE` logic. Add corresponding unit tests for both the `InvalidScriptHashType` and `ScriptHashTypeNotPermitted` error paths on the type script slot.

## Proof of Concept

1. Obtain any live cell (any wallet with CKB).
2. Build a transaction:
   - Input: the live cell.
   - Output: a `CellOutput` whose `type_` field is a `Script` with `hash_type = 6` (`Data3`) and any 32-byte `code_hash`.
3. Submit via `send_transaction` JSON-RPC.
4. Observe: the node does **not** return `ScriptHashTypeNotPermitted` at the non-contextual stage; it proceeds to cell-dep/input resolution and script-group construction before returning `InvalidScriptHashType`.
5. Because the transaction is rejected, the input cell remains unspent. Repeat step 3 in a tight loop to sustain the load on the verification worker.

A unit test analogous to `test_not_enabled_hash_type_output_lock` (lines 100–122) but targeting the type script slot will reproduce the missing rejection at the `ScriptHashTypeVerifier` layer.

#[test]
pub fn test_unknown_hash_type_output_lock() {
    let transaction = TransactionBuilder::default()
        .output(
            CellOutput::new_builder()
                .lock(Script::default().as_builder().hash_type(3).build())
                .build(),
        )
        .build();
    let verifier = ScriptHashTypeVerifier::new(&transaction);

    assert_error_eq!(
        verifier.verify().unwrap_err(),
        TransactionError::InvalidScriptHashType {
            hash_type: 3.into(),

    );
}
