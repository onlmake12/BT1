Audit Report

## Title
Off-by-One in VM Count Guard Allows Spawning Beyond `MAX_VMS_COUNT` — (`script/src/scheduler.rs`)

## Summary
The `Scheduler` in `script/src/scheduler.rs` uses `>` instead of `>=` when comparing the current VM count against `MAX_VMS_COUNT` (16). When exactly 16 VMs exist, the condition `16 > 16` evaluates to `false`, the guard is skipped, and `boot_vm` is called, producing a 17th VM. This is a confirmed off-by-one that allows any script author to exceed the consensus-defined VM count limit.

## Finding Description
`MAX_VMS_COUNT` is defined as `16` at line 34 of `script/src/scheduler.rs`. In the `Message::Spawn` handler at line 532, the guard is:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
```

When `suspended.len() + instantiated.len()` equals exactly `16`, the expression `16 > 16` is `false`. The guard body (which sets the return register to `MAX_VMS_SPAWNED` and `continue`s) is never entered. Execution falls through to `boot_vm` at line 539, which instantiates a 17th VM. The correct operator is `>=`: once the total reaches `MAX_VMS_COUNT`, no further spawn should be permitted. No other guard exists between the check and the `boot_vm` call.

## Impact Explanation
This constitutes **incorrect implementation or behavior of CKB-VM** (High, 10001–15000 points). `MAX_VMS_COUNT` is a consensus-level resource cap. Bypassing it — even by one — means the VM runtime does not enforce its own declared invariant. Every node running this code will accept the 17th VM, so there is no consensus split, but the resource bound that the constant was designed to enforce is violated. Memory, register state, and snapshot data for the extra VM are allocated beyond what the cap was intended to allow.

## Likelihood Explanation
Any unprivileged script author can trigger this by submitting a transaction containing a script that recursively calls `ckb_spawn` until 16 VMs exist, then issues one additional spawn. No special keys, privileges, or majority hashpower are required — only the ability to submit a transaction with a crafted script. The condition is deterministic and reproducible on every node.

## Recommendation
Change the comparison operator at line 532 from `>` to `>=`:

```diff
- if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
+ if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

This ensures that once the total VM count reaches `MAX_VMS_COUNT`, no further spawns are permitted.

## Proof of Concept
Write a CKB script in C that:
1. Reads a counter argument passed at spawn time.
2. If the counter is less than 16, calls `ckb_spawn` on itself with `counter + 1`, then waits.
3. When the counter reaches 16 (meaning 16 VMs already exist including root), issues one more `ckb_spawn`.
4. Assert the return value of that final spawn is `SUCCESS` (0) rather than `MAX_VMS_SPAWNED`.

The existing test pattern in `script/src/verify/tests/ckb_latest/features_since_v2023.rs` (e.g., `check_spawn_index_out_of_bound`) provides the scaffolding for a Rust-level integration test that loads such a binary and verifies the 17th spawn succeeds, confirming the bypass.
