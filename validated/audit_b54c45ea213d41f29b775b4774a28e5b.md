### Title
Fee Credited and Request Cleared Before Callback Execution With No Rollback on Failure — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol::executeCallback`, the provider's fee is credited (`accruedFeesInWei += ...`) and the request is permanently cleared (`clearRequest`) **before** the consumer callback is attempted. When the callback fails (caught by `try/catch`), neither state change is rolled back. The consumer permanently loses their fee with no service delivered and no refund or retry path. This is a direct analog to the LeXscrow incomplete-state-reset class: one state variable (the request) is cleared while a dependent accounting variable (the fee credit) is not conditionally reversed on failure.

### Finding Description
In `executeCallback`, the execution order is:

1. **Line 161–162**: Provider fee is credited unconditionally.
2. **Line 164**: Request is permanently cleared via `clearRequest(sequenceNumber)`.
3. **Lines 176–201**: Consumer callback is attempted inside a `try/catch`. On failure, only an event is emitted — no state is rolled back.

```solidity
// Fee credited BEFORE callback
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // line 161-162

clearRequest(sequenceNumber);                        // line 164

try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{
    emitPriceUpdate(...);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(...);             // no refund, no retry
} catch {
    emit PriceUpdateCallbackFailed(...);             // no refund, no retry
}
```

The developers themselves flag this ordering as dangerous in a TODO comment at line 155–156:

> `// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract.`

However, the comment only addresses the case where `executeCallback` itself reverts. The actual issue is the opposite: `executeCallback` **succeeds** (does not revert), but the consumer's callback fails — and the fee has already been irrevocably credited to the provider with the request cleared. [1](#0-0) 

### Impact Explanation
Any consumer whose `_echoCallback` reverts — due to a bug, an out-of-gas condition from an underestimated `callbackGasLimit`, a contract upgrade, or a self-destruct — permanently loses the fee they paid at request time. The fee (`req.fee + msg.value - pythFee`) is transferred to `providerToCredit`'s `accruedFeesInWei` balance with no mechanism for:
- Refund to the consumer
- Retry of the callback
- Dispute of the payment

A malicious provider can also exploit this deliberately: after the exclusivity period expires, any address can call `executeCallback` as `providerToCredit`. If a provider knows a consumer's callback will revert (e.g., the consumer contract has a known revert path or has been upgraded), they can call `executeCallback` to collect the fee without delivering the service. [2](#0-1) [3](#0-2) 

### Likelihood Explanation
- **Medium-High.** Consumer contracts routinely have bugs in callback handlers, or set `callbackGasLimit` too low. The `callbackGasLimit` is set by the consumer at request time and is not validated against actual gas usage. Any consumer contract that reverts in `_echoCallback` for any reason triggers this loss. The path is fully reachable by an unprivileged transaction sender (the provider calling `executeCallback`, or any address after the exclusivity period). [4](#0-3) 

### Recommendation
Move the fee credit **after** a successful callback, or implement a conditional rollback: if the callback fails, reverse the `accruedFeesInWei` increment and either re-activate the request (allowing retry) or refund the consumer directly. At minimum, do not clear the request until the callback has succeeded, mirroring the pattern used in `Entropy.sol` where `clearRequest` is only called on the success path. [5](#0-4) 

### Proof of Concept
1. Consumer contract `C` calls `requestPriceUpdatesWithCallback{value: fee}(provider, publishTime, priceIds, gasLimit)`. Request is stored with `req.fee = fee - pythFeeInWei`.
2. Consumer contract `C`'s `_echoCallback` is known to revert (e.g., it calls `require(false)`).
3. Provider (or any address after exclusivity period) calls `executeCallback(providerToCredit, sequenceNumber, updateData, priceIds)`.
4. Inside `executeCallback`:
   - `_state.providers[providerToCredit].accruedFeesInWei += (req.fee + msg.value) - pythFee` — provider is paid.
   - `clearRequest(sequenceNumber)` — request is deleted.
   - `IEchoConsumer(C)._echoCallback(...)` reverts — caught by `catch`, only an event emitted.
5. Transaction succeeds. Provider has received the fee. Consumer's request is gone. Consumer has no recourse. [6](#0-5)
