### Title
Deviation Check in `_validateShouldUpdatePrices` Compares Raw Price Mantissas Without Exponent Normalization - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `Scheduler._validateShouldUpdatePrices` function computes price deviation by subtracting and dividing raw `int64` mantissa values from two price feeds without accounting for their respective `expo` fields. When the exponent of a Pyth price feed changes between the stored (previous) update and the incoming (current) update, the deviation calculation produces a wildly incorrect result ŌĆö either falsely triggering updates that drain subscription balances, or suppressing updates that should fire.

### Finding Description

In `Scheduler.sol`, the `_validateShouldUpdatePrices` function is responsible for deciding whether a price update meets the deviation threshold configured by a subscription. The relevant code is:

```solidity
// Calculate the deviation percentage
int64 currentPrice = priceFeeds[i].price.price;
int64 previousPrice = previousFeed.price.price;

// Calculate absolute deviation basis points (scaled by 1e4)
uint256 numerator = SignedMath.abs(
    currentPrice - previousPrice
);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(
    numerator,
    10_000,
    denominator
);
``` [1](#0-0) 

A Pyth price is represented as `price ├Ś 10^expo`. The `price` field is a raw integer mantissa; the actual value is only meaningful when combined with `expo`. The code above reads only `priceFeeds[i].price.price` and `previousFeed.price.price` ŌĆö the raw mantissas ŌĆö and computes their ratio directly, **ignoring both exponents entirely**.

If the exponent of a feed changes between the stored update and the incoming update (a legitimate occurrence in the Pyth protocol), the two mantissas are in different scales and their ratio is meaningless. For example:

| Scenario | Previous | Current | Actual price change |
|---|---|---|---|
| Expo: -8 ŌåÆ -6 | mantissa=`100000000` ($1.00) | mantissa=`1000000` ($1.00) | 0% |
| Computed deviation | `|1000000 - 100000000| / 100000000 ├Ś 10000` | = **9900 bps (99%)** | ŌåÉ wrong |

| Scenario | Previous | Current | Actual price change |
|---|---|---|---|
| Expo: -6 ŌåÆ -8 | mantissa=`1000000` ($1.00) | mantissa=`100000000` ($1.00) | 0% |
| Computed deviation | `|100000000 - 1000000| / 1000000 ├Ś 10000` | = **990000 bps (9900%)** | ŌåÉ wrong |

The correct approach is to normalize both prices to the same exponent before computing the ratio, as is done in `PythUtils.convertToUint` and `PythUtils.deriveCrossRate`. [2](#0-1) 

Notably, the Solana receiver already recognizes this class of problem: `validate_twap_messages` explicitly requires that the start and end messages share the same exponent before performing arithmetic on their mantissas. [3](#0-2) 

The `Scheduler` deviation check has no such guard.

### Impact Explanation

**Spurious updates / fund drainage:** When the exponent decreases (e.g., -8 ŌåÆ -6), the new mantissa is 100├Ś smaller for the same real price. The computed deviation is ~99%, far above any reasonable threshold. Every subsequent update will be accepted as a deviation trigger, causing the subscription's ETH balance to be drained by keeper fees even though the real price has not moved.

**Missed updates / stale prices:** When the exponent increases (e.g., -6 ŌåÆ -8), the new mantissa is 100├Ś larger for the same real price. The computed deviation is ~9900%, which always exceeds the threshold ŌĆö but this means the *first* post-change update always fires. After that, the stored previous price has the new exponent, so subsequent legitimate price moves are computed correctly. The primary harm is the spurious first trigger.

**Suppressed updates:** If a subscription's `deviationThresholdBps` is set very high and the exponent change causes the computed deviation to be artificially *low* (e.g., a large real move is masked by an exponent increase that shrinks the mantissa), legitimate updates could be suppressed, leaving subscribers with stale prices.

### Likelihood Explanation

Pyth price feeds do change their exponent over time as the network adjusts precision. Any pusher (unprivileged) who submits a valid Pyth price update immediately after an exponent change will trigger the broken deviation path. The pusher does not need to forge any data ŌĆö they simply submit a legitimately signed Pyth update that happens to carry a different exponent than the one stored in the subscription's state. This is a realistic, low-effort trigger condition.

### Recommendation

Before computing the deviation, normalize both prices to the same exponent. The simplest approach is to use `PythUtils.convertToUint` with a fixed `targetDecimals` for both the current and previous price, then compute the ratio on the normalized values:

```solidity
// Normalize both prices to 18 decimals before comparing
uint256 normalizedCurrent = PythUtils.convertToUint(
    priceFeeds[i].price.price,
    priceFeeds[i].price.expo,
    18
);
uint256 normalizedPrevious = PythUtils.convertToUint(
    previousFeed.price.price,
    previousFeed.price.expo,
    18
);

if (normalizedPrevious == 0 || normalizedCurrent == 0) continue;

uint256 numerator = normalizedCurrent > normalizedPrevious
    ? normalizedCurrent - normalizedPrevious
    : normalizedPrevious - normalizedCurrent;
uint256 deviationBps = Math.mulDiv(numerator, 10_000, normalizedPrevious);
```

Alternatively, add an explicit check that `priceFeeds[i].price.expo == previousFeed.price.expo` and skip (or always trigger) the deviation check when they differ, similar to the guard in `validate_twap_messages`.

### Proof of Concept

1. Deploy `Scheduler` and create a subscription with `updateOnDeviation = true`, `deviationThresholdBps = 100` (1%).
2. Submit a first update for feed `F` with `price = 100000000`, `expo = -8` (actual price = $1.00). This is stored as the previous price.
3. The Pyth network changes feed `F`'s exponent to `-6`. Submit a second update with `price = 1000000`, `expo = -6` (actual price = $1.00 ŌĆö no real change).
4. `_validateShouldUpdatePrices` computes:
   - `numerator = |1000000 - 100000000| = 99000000`
   - `denominator = 100000000`
   - `deviationBps = 9900` (99%)
5. `9900 >= 100` ŌåÆ update is accepted and keeper is paid, despite zero real price movement.
6. Repeat step 3ŌĆō5 indefinitely: every update drains the subscription balance. [4](#0-3)
