### Title
Unsanitized Provider URI Stored On-Chain Without Validation Enables Phishing via Official Entropy Explorer - (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Entropy contract's `register()` and `setProviderUri()` functions accept a `uri` parameter as raw `bytes calldata` with **zero content or format validation**. Any unprivileged actor can permissionlessly register as a provider and store an arbitrary URI on-chain. The contract's own documentation states this URI is used by "Client SDKs to automatically determine how to retrieve random values for each provider." The URI is decoded and surfaced by official Pyth tooling (`contract_manager`), making it a vector for phishing or injection attacks against users and developers who trust the official Entropy Explorer.

### Finding Description

In `Entropy.sol`, the `register()` function stores the caller-supplied `uri` bytes directly into `provider.uri` with no validation beyond `chainLength != 0`:

```solidity
function register(
    uint128 feeInWei,
    bytes32 commitment,
    bytes calldata commitmentMetadata,
    uint64 chainLength,
    bytes calldata uri   // ← no validation whatsoever
) public override {
    if (chainLength == 0) revert EntropyErrors.AssertionFailure();
    ...
    provider.uri = uri;  // stored verbatim
```

Similarly, `setProviderUri()` updates the URI with no validation:

```solidity
function setProviderUri(bytes calldata newUri) external override {
    ...
    provider.uri = newUri;  // stored verbatim
```

The `ProviderInfo` struct documents the URI's purpose explicitly:

```
// Optional URI where clients can retrieve revelations for the provider.
// Client SDKs can use this field to automatically determine how to retrieve random values for each provider.
```

The official `contract_manager` tooling decodes this URI and surfaces it:

```typescript
uri: Web3.utils.toAscii(info.uri),
```

There are no checks for:
- URI scheme (e.g., `https://` vs `javascript:` or `data:`)
- Minimum or maximum length
- Character set restrictions
- Domain allowlisting

By contrast, the analogous Astroport vulnerability at least enforced `MIN_LINK_LENGTH` / `MAX_LINK_LENGTH` bounds. Pyth's Entropy contract enforces nothing.

### Impact Explanation

A malicious actor registers as an Entropy provider (permissionless) and sets `uri` to:
1. A phishing URL mimicking the official Fortuna endpoint (e.g., `https://fortuna-staging.dourolabs.app.evil.com/...`) — this URI appears in the official Entropy Explorer and in `getProviderInfo` responses consumed by developer tooling, giving it false legitimacy.
2. A `javascript:` or `data:` URI — if any frontend renders the URI as a clickable link without sanitization, this becomes an XSS vector.
3. An arbitrarily long byte sequence — no length cap means storage bloat and potential DoS on indexers/frontends that process provider info.

Users and developers who query `getProviderInfo` or view the Entropy Explorer may trust the displayed URI because it originates from the official on-chain contract, not knowing it was set by a malicious provider.

**Impact: 3/5** — Phishing/social engineering risk; no direct on-chain fund theft, but users directed to a malicious URI could lose funds off-chain.

### Likelihood Explanation

Registration is fully permissionless — anyone can call `register()` with any URI. No stake, no approval, no governance vote is required. The attack requires only a small gas payment.

**Likelihood: 3/5** — Low barrier to execute; effectiveness depends on whether the official frontend renders the URI as a trusted link.

### Recommendation

Add URI validation in both `register()` and `setProviderUri()`:

1. **Enforce a maximum byte length** (e.g., 256 bytes) to prevent storage abuse.
2. **Validate the URI scheme** — reject any URI that does not begin with `https://`.
3. **Restrict to printable ASCII** — reject any byte outside the range `0x20–0x7E`.
4. **Frontend hardening** — ensure the Entropy Explorer and all official tooling HTML-encode the URI before rendering and never construct clickable `<a href>` elements directly from the on-chain `uri` field without allowlist validation.

### Proof of Concept

```solidity
// Attacker registers as a provider with a phishing URI
bytes memory maliciousUri = bytes("https://fortuna-staging.dourolabs.app.attacker.com/steal");

entropyContract.register(
    1 wei,                    // minimal fee
    someCommitment,           // valid commitment
    hex"",                    // empty metadata
    1,                        // chainLength = 1 (minimum valid)
    maliciousUri              // ← accepted with no validation
);

// Now getProviderInfo(attacker) returns uri = "https://fortuna-staging.dourolabs.app.attacker.com/steal"
// This URI appears in the official Entropy Explorer and in SDK responses
// Developers/users who see it may trust it as a legitimate Fortuna endpoint
```

    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }

    // Set provider uri. It will revert if provider is not registered.
    function setProviderUri(bytes calldata newUri) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        bytes memory oldUri = provider.uri;
        provider.uri = newUri;
        emit ProviderUriUpdated(msg.sender, oldUri, newUri);
        emit EntropyEventsV2.ProviderUriUpdated(
            msg.sender,
            oldUri,
            newUri,
            bytes("")
        );
    }
