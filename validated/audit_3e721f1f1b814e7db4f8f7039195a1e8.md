### Title
Excess Fee Refund Sent to `msg.sender` Instead of Originating User in `verifyUpdate` ŌĆö (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` refunds excess ETH to `msg.sender`. When the function is called by an integrating contract (the primary Lazer use-case), `msg.sender` is the calling contract, not the end user who funded the transaction. This causes either a silent loss of excess ETH for the user, or a full revert/DoS of the verification call if the calling contract cannot receive ETH.

---

### Finding Description

In `PythLazer.sol`, `verifyUpdate` is a `payable` function that collects a `verification_fee` and explicitly attempts to refund any overpayment:

```solidity
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    // Require fee and refund excess
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
        payable(msg.sender).transfer(msg.value - verification_fee);
    }
    ...
}
``` [1](#0-0) 

The refund is unconditionally sent to `msg.sender`. Lazer is designed to be consumed by on-chain contracts that call `pythLazer.verifyUpdate{value: fee}(update)` from within their own logic. In that call chain:

- `msg.sender` = the **calling contract** (e.g., a DeFi protocol or oracle consumer)
- The **end user** who funded the transaction is `tx.origin`, not `msg.sender`

The calling contract receives the excess refund, not the user who paid for it. This is the same receiver mismatch class as the BendDAO finding: a token/ETH operation uses the wrong address (`msg.sender` / `msgSender`) instead of the intended recipient (`receiver` / end user).

The existing test only validates the EOA-calls-directly path:

```solidity
// Alice overpays and is refunded
vm.prank(alice);
pythLazer.verifyUpdate{value: 0.5 ether}(update);
assertEq(alice.balance, 1 ether - fee - fee);
``` [2](#0-1) 

No test covers the case where a contract calls `verifyUpdate` on behalf of a user.

---

### Impact Explanation

Two concrete impacts when a contract calls `verifyUpdate` with `msg.value > verification_fee`:

1. **DoS / Revert**: `transfer()` forwards only 2300 gas. Most non-trivial contracts (e.g., multisigs, proxy contracts, DeFi protocols) cannot execute any logic within 2300 gas. The `transfer()` call reverts, which reverts the entire `verifyUpdate` call. Any integrating contract that overpays by even 1 wei will have its verification permanently blocked unless it sends the exact fee ŌĆö a fragile requirement.

2. **Silent ETH loss**: If the calling contract does have a `receive()` fallback that accepts ETH, the excess is silently deposited into the calling contract. The end user who funded the transaction loses the excess ETH with no recourse.

Both impacts are triggered by any unprivileged Lazer updater/relayer contract that overpays the fee.

---

### Likelihood Explanation

- Lazer is explicitly designed for on-chain contract integration; the `verifyUpdate` function is the primary entry point for consumer contracts.
- Contracts routinely overpay fees to avoid underpayment reverts (e.g., using `getFee() + buffer` or passing `msg.value` through from a user).
- The `verification_fee` is mutable (owner-controlled), so a fee increase after a contract hardcodes a slightly-higher value will trigger the refund path.
- The DoS variant requires zero attacker action ŌĆö it is triggered by normal, well-intentioned usage.

---

### Recommendation

Remove the in-function refund entirely, or accept an explicit `refundTo` address parameter:

```diff
function verifyUpdate(
-   bytes calldata update
+   bytes calldata update,
+   address refundTo
) external payable returns (bytes calldata payload, address signer) {
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
-       payable(msg.sender).transfer(msg.value - verification_fee);
+       payable(refundTo).transfer(msg.value - verification_fee);
    }
    ...
}
```

Alternatively, use `call` instead of `transfer` to avoid the 2300-gas stipend limitation, and document that callers are responsible for handling the refund themselves.

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.13;

interface IPythLazer {
    function verifyUpdate(bytes calldata update) external payable
        returns (bytes calldata payload, address signer);
    function verification_fee() external view returns (uint256);
}

// Simulates a typical Lazer consumer contract (no receive() function)
contract LazerConsumer {
    IPythLazer public pythLazer;

    constructor(address _pythLazer) {
        pythLazer = IPythLazer(_pythLazer);
    }

    // Called by a user who sends ETH to cover the fee + buffer
    function consumePrice(bytes calldata update) external payable {
        uint256 fee = pythLazer.verification_fee();
        // User sends fee + 1 wei buffer to be safe
        // This call REVERTS because transfer() to this contract fails (no receive())
        (bytes memory payload, ) = pythLazer.verifyUpdate{value: fee + 1}(update);
        // ... use payload
    }
}
```

When `LazerConsumer.consumePrice` is called with `msg.value = fee + 1`, `PythLazer.verifyUpdate` attempts `payable(address(lazerConsumer)).transfer(1)`. Since `LazerConsumer` has no `receive()` function, the `transfer` reverts, and the entire `consumePrice` call fails ŌĆö permanently blocking the consumer from using Lazer unless it sends the exact fee to the wei.
