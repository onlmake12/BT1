Audit Report

## Title
Write Lock Held Across Unbounded Synchronous Script Execution During Reorg Re-verification — (File: tx-pool/src/process.rs)

## Summary

`update_tx_pool_for_reorg` acquires the `tx_pool` write lock at line 836 and holds it across the entire `readd_detached_tx` call. Inside that call, `verify_rtx` is invoked with `command_rx = None` for every detached transaction, which unconditionally takes the `block_in_place` synchronous script execution path. The write lock is never released between transactions, blocking all concurrent pool operations — tx submission, block template generation, and pool-touching RPCs — for the full duration of re-verification.

## Finding Description

**Lock acquisition and scope** (`tx-pool/src/process.rs`, L836–851): The write guard `tx_pool` is acquired at L836 and lives until the closing brace at L851. `readd_detached_tx` is called at L849 while the guard is still in scope, receiving `&mut TxPool` — the guard is not dropped before or during the call.

**`readd_detached_tx` passes `None` as `command_rx`** (`tx-pool/src/process.rs`, L895–902): Every iteration of the detached-tx loop calls `verify_rtx(..., None)`. There is no yield point, no cancellation, and no lock release between iterations.

**`verify_rtx` with `command_rx = None` takes the `block_in_place` branch** (`tx-pool/src/util.rs`, L116–131): The `else` branch runs `ContextualTransactionVerifier::verify` synchronously via `block_in_place`. `block_in_place` moves other tasks off the current thread but does **not** release the Tokio `RwLock` guard held in the caller's scope. Every concurrent `self.tx_pool.write().await` or `self.tx_pool.read().await` is blocked for the entire loop.

**Single reorg consumer** (`tx-pool/src/service.rs`, L698–731): The `reorg_receiver` loop is a single async task; a second reorg cannot be processed while the first is stalled.

No existing guard limits the number of transactions re-verified, caps per-transaction cycle consumption during reorg, or releases the lock between verifications.

## Impact Explanation

While the write lock is held, `submit_entry` (all tx submissions), `get_block_template` (miners cannot produce new templates), and all pool-touching RPC calls are blocked. Stall duration is `N_detached_txs × T_verify_per_tx`. At the default `max_tx_verify_cycles` of 70,000,000, a block of dozens of max-cycle transactions produces a multi-second to multi-tens-of-second stall per reorg event. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (High, 10001–15000 points).

## Likelihood Explanation

A 1-block reorg is a routine network event requiring no majority hashpower — any miner can privately mine a single block and release it to cause one. The attacker fills that block with valid transactions each consuming `max_tx_verify_cycles` cycles (valid scripts, valid PoW, valid chain). No special privilege, no leaked key, and no majority hashpower is required for the 1-block case. The attack is repeatable: each subsequent 1-block reorg re-triggers the stall.

## Recommendation

1. **Release the lock before verification.** Collect the set of transactions to re-add, drop the write guard, verify each transaction independently without holding any pool lock, then re-acquire the write lock only to insert verified entries.
2. **Use the `verify_with_pause` path.** Pass a `command_rx` to `verify_rtx` in `readd_detached_tx` so verification yields to the Tokio scheduler and can be cancelled, consistent with how normal tx submission works.
3. **Cap the number of transactions re-verified per reorg event** to bound worst-case stall time regardless of block contents.

## Proof of Concept

```
1. Craft transaction T with a lock script consuming exactly max_tx_verify_cycles cycles.
2. Mine block B1 (privately) at height H containing K copies of T (distinct inputs).
3. Let the honest network produce a competing block B1' at height H.
4. Release B1 to a victim node that already has B1' as tip → 1-block reorg.
5. Victim calls update_tx_pool_for_reorg(detached=[B1], attached=[B1']).
6. readd_detached_tx iterates K transactions, calling verify_rtx(..., None) → block_in_place for each.
7. tx_pool write lock is held for K × T_verify_per_tx.
8. During this window, issue get_block_template or send_transaction RPC and observe it hangs until re-verification completes.
```
