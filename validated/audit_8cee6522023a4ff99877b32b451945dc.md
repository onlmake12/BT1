Audit Report

## Title
Peer Store Eviction Bypass via Uniform /16-Group Distribution — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
`PeerStore::check_purge` contains two sequential eviction strategies, both of which produce zero candidates when the address store holds exactly 4 addresses per IPv4 /16 group across 4096 distinct groups (totaling `ADDR_COUNT_LIMIT` = 16384 entries). In this state, every subsequent `add_addr` call returns `PeerStoreError::EvictionFailed`, permanently freezing peer address ingestion. The poisoned state is serialized to disk and survives node restarts.

## Finding Description
`check_purge` (`peer_store_impl.rs` lines 327–404) is called before every `add_addr` insertion. When the store is at capacity, it runs two sequential eviction passes:

**Pass 1 (lines 341–355):** Collects addresses where `!is_connectable(now_ms)`. `AddrInfo::new` initializes `last_tried_at_ms = 0`, `attempts_count = 0`, `last_connected_at_ms = 0` (`types.rs` lines 71–75). For any real `now_ms > 60000`, `tried_in_last_minute` is false, and neither failure branch in `is_connectable` (`types.rs` lines 95–103) is triggered. All freshly-advertised addresses are connectable; Pass 1 yields zero candidates.

**Pass 2 (lines 357–402):** Groups addresses by `Group`, which for IPv4 uses only the first two octets (`network_group.rs` lines 26–28), forming /16 buckets. The loop takes the top `len / 2` groups by size (line 376), then applies a strict `addrs.len() > 4` guard (line 378). Groups of exactly 4 return `None` from `flat_map`. With 4096 groups of exactly 4, `candidate_peers` is empty and `EvictionFailed` is returned at lines 399–401.

**Attack construction:** An attacker with a single P2P connection sends `Nodes` messages containing addresses from 4096 distinct globally-routable /16 subnets, 4 addresses per subnet (16384 total). `add_new_addrs` (`discovery/mod.rs` lines 347–363) calls `add_addr` for each, silently discarding errors at `debug` level. Once the store reaches `ADDR_COUNT_LIMIT` in this uniform distribution, both eviction passes fail for every subsequent insertion. `dump_to_dir`/`load_from_dir_or_default` (`peer_store_db.rs` lines 112–132) serialize and reload the poisoned state across restarts.

## Impact Explanation
Once poisoned, the node cannot ingest any new peer addresses via discovery. The existing 16384 entries churn over time (connections fail, timeouts expire) with no ability to replenish, causing the node's reachable peer set to degrade monotonically. This is a permanent, attacker-induced freeze of the peer address storage mechanism, matching **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation
The attacker requires only a single P2P connection to the victim. Sending ~16384 addresses across multiple `Nodes` messages requires no PoW, no special privileges, and no Sybil capability beyond one connection. The IPv4 space contains far more than 4096 globally-routable /16 subnets, making address construction trivial. The poisoned state persists across restarts, requiring manual deletion of `addr_manager.db` to remediate. The attack is deterministic and fully repeatable.

## Recommendation
Replace the strict `> 4` threshold at `peer_store_impl.rs` line 378 with `>= 1` so that any non-empty group is eligible for eviction. As a defense-in-depth measure, guarantee at least one eviction candidate regardless of group distribution — for example, always evict the oldest-tried address from the largest group, falling back to a random address if all groups are equal-sized. Additionally, enforce a per-session cap on the number of addresses accepted from a single `Nodes` message source in `add_new_addrs` to limit the rate at which an attacker can fill the store.
