### Title
Unbounded `runtime::spawn` of 24-concurrent-TCP-retry tasks via `ConnectionSync` on poisoned `pending_delivered` — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

An unprivileged connected peer can exhaust the victim node's file descriptors and TCP connection slots by (1) poisoning `pending_delivered` with up to 24 attacker-controlled TCP addresses per entry using fabricated `from` peer IDs in `ConnectionRequest` messages, then (2) sending `ConnectionSync` messages that cause the node to `runtime::spawn` an unbounded number of tasks, each running `select_ok` over 24 concurrent `try_nat_traversal` futures that each retry TCP connections for 30 seconds.

---

### Finding Description

**Phase 1 — Poisoning `pending_delivered`**

`ConnectionRequestProcess::execute` is called for any message received from any connected peer. When `self_peer_id == &content.to`, it calls `respond_delivered`: [1](#0-0) 

Inside `respond_delivered`, the only deduplication guard is a per-`from_peer_id` cooldown of `HOLE_PUNCHING_INTERVAL` (2 minutes): [2](#0-1) 

The `from` field in the `ConnectionRequest` message is **not** verified to match the actual sender's session peer ID — it is taken verbatim from the message payload. An attacker can craft messages with arbitrarily many distinct fake `from` peer IDs, each carrying 24 attacker-controlled TCP/IP addresses. Each unique `from` bypasses the cooldown check and inserts a fresh entry: [3](#0-2) 

There is **no cap on the size of `pending_delivered`**. The only cleanup is in `notify()`, which runs every 5 minutes and removes entries older than 5 minutes: [4](#0-3) 

The per-session `rate_limiter` allows 30 `ConnectionRequest` messages per second: [5](#0-4) 

So within one 5-minute window, a single connected peer can insert up to 30/s × 300s = **9,000 entries** into `pending_delivered`.

---

**Phase 2 — Triggering unbounded spawns via `ConnectionSync`**

`ConnectionSyncProcess::execute` processes any `ConnectionSync` message. When `route` is empty and `content.to == self_peer_id`, it looks up `pending_delivered` by `content.from`: [6](#0-5) 

If found, it builds one `try_nat_traversal` future per stored address and unconditionally calls `runtime::spawn`: [7](#0-6) 

Each `try_nat_traversal` future loops for up to **30 seconds**, creating a new TCP socket and attempting a connection on every ~200 ms iteration: [8](#0-7) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1/second: [9](#0-8) 

Because the attacker uses a different `from` peer ID for each `ConnectionSync`, every message is a distinct key and the rate limiter does not aggregate them. The only effective cap is the per-session `rate_limiter` at 30/second.

---

### Impact Explanation

- **30 `ConnectionSync` messages/second** → 30 `runtime::spawn` tasks/second
- Each task runs `select_ok` over **24 concurrent `try_nat_traversal` futures**
- Each future holds a TCP socket open and retries for **30 seconds**
- Steady-state concurrent TCP sockets: 30 × 24 × 30 = **21,600**

A typical Linux process has a default file-descriptor limit of 1,024 (soft) or 65,535 (hard). Even at the hard limit, the attacker saturates it within ~3 seconds of sustained sending. The spawned tasks are never cancelled; they run to their 30-second timeout regardless of what happens to the originating session.

---

### Likelihood Explanation

The attack requires only a single P2P connection to the victim node — no authentication, no PoW, no privileged role. The `ConnectionRequest` and `ConnectionSync` messages are standard HolePunching protocol messages accepted from any peer. The fabricated `from` peer ID is never cross-checked against the actual session identity. The attack is fully local-testable and requires no Sybil capability.

---

### Recommendation

1. **Bind `from` to the actual session peer ID** in `ConnectionRequest` processing: reject any message where `content.from` does not match the peer ID of the sending session.
2. **Cap `pending_delivered`** to a small fixed size (e.g., 64 entries total), evicting the oldest entry on overflow.
3. **Track and cap active NAT-traversal spawns**: maintain a counter of live spawned tasks and refuse to spawn more when a threshold (e.g., 8) is reached.
4. **Remove the entry from `pending_delivered` after the first `ConnectionSync` triggers it**, so each poisoned entry can only produce one spawn.

---

### Proof of Concept

```
1. Attacker connects to victim node via the HolePunching protocol.

2. Attacker sends 30 ConnectionRequest messages per second, each with:
   - from  = freshly generated peer ID (different each time)
   - to    = victim's local peer ID
   - listen_addrs = 24 valid TCP/IP multiaddrs (e.g., 192.0.2.1:1–192.0.2.24:1)
   - max_hops = 1, route = []
   Result: pending_delivered grows by 30 entries/second.

3. Attacker sends 30 ConnectionSync messages per second, each with:
   - from  = one of the poisoned peer IDs from step 2
   - to    = victim's local peer ID
   - route = []
   Result: each message triggers runtime::spawn with select_ok over 24
           try_nat_traversal futures (30-second TCP retry loops each).

4. After ~3 seconds: 30 × 3 = 90 spawned tasks × 24 sockets = 2,160 open
   TCP sockets. After ~30 seconds: 21,600 concurrent sockets.
   File-descriptor table exhausted; new connections and I/O fail with EMFILE.
```
