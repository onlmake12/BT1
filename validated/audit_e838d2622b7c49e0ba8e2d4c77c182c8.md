Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Growth via Malicious `GetBlockProposal` P2P Messages — (File: `sync/src/types/mod.rs`)

## Summary
`SyncState::pending_get_block_proposals` is an uncapped `DashMap` that any peer can fill by sending `GetBlockProposal` messages containing up to 3,000 fake `ProposalShortId` values. The map has no size guard, the `disconnected()` handler does not remove a peer's entries, and the only relief is a periodic drain that performs an O(n) clone of the entire map. With enough Sybil peers the map grows large enough within each drain window to cause severe heap pressure or OOM.

## Finding Description

**Unbounded map, no capacity limit.**
`pending_get_block_proposals` is declared as a plain `DashMap::new()` with no maximum size: [1](#0-0) [2](#0-1) 

**Per-message count check is the only guard in the handler.**
`GetBlockProposalProcess::execute()` rejects messages whose proposal count exceeds `max_block_proposals_limit × max_uncles_num` (1,500 × 2 = 3,000), but applies no total-map size check: [3](#0-2) 

For every proposal ID absent from the tx-pool, the handler calls `insert_get_block_proposals` unconditionally: [4](#0-3) 

**`insert_get_block_proposals` has no size guard.**
Every ID is inserted without checking the current map size: [5](#0-4) 

**`drain_get_block_proposals` performs an O(n) clone.**
The drain clones the entire map before clearing it, allocating a full copy proportional to the current map size: [6](#0-5) 

The drain is triggered by `TX_PROPOSAL_TOKEN` every 100 ms: [7](#0-6) 

**`disconnected()` does not clean up `pending_get_block_proposals`.**
On peer disconnect, only inflight blocks are removed and `peers().disconnected()` is called; no entries are removed from `pending_get_block_proposals`: [8](#0-7) 

**Rate limiter is per-peer per-message-type at 30 req/s.**
The limiter key is `(PeerIndex, message_item_id)`, so each peer is independently capped at 30 `GetBlockProposal` messages per second: [9](#0-8) [10](#0-9) 

**Exploit arithmetic.**
Within each 100 ms drain window, `k` Sybil peers can insert `k × 3 × 3,000 = 9,000k` entries. At `k = 100` peers that is 900,000 entries. Each entry carries a 10-byte key plus a `HashSet<PeerIndex>` and DashMap shard overhead (~100 bytes total), yielding ~90 MB of live heap per drain cycle. The O(n) clone doubles the peak allocation to ~180 MB per cycle. At `k = 1,000` peers the peak reaches ~1.8 GB, sufficient to trigger OOM on typical node deployments. Entries from peers that disconnect mid-window are not removed and persist until the next drain, slightly extending the window of accumulation.

## Impact Explanation

This matches the **High** severity impact: *"Vulnerabilities which could easily crash a CKB node."* With a sufficient number of Sybil peers (reachable without any authentication or stake), the node's heap grows to OOM within seconds. Even below the OOM threshold, the O(n) clone every 100 ms on a large map introduces a recurring CPU and allocator spike in the relay timer loop, delaying processing of legitimate compact blocks and transactions.

## Likelihood Explanation

Any unprivileged TCP peer can send `GetBlockProposal` messages. No valid transaction, stake, or authentication is required. Generating 3,000 random 10-byte IDs is negligible CPU work. The per-peer rate limit (30 req/s) is easily saturated and multiplies linearly with the number of Sybil connections an attacker controls. The attack is stateless and repeatable. The CKB changelog records prior fixes for the same class of resource-exhaustion bug (`#3093`, `#3094`, `#3110`), confirming the pattern is realistic.

## Recommendation

1. **Cap `pending_get_block_proposals`**: Enforce a maximum entry count (e.g., `max_block_proposals_limit × max_peers`). Silently drop insertions once the cap is reached.
2. **Per-peer sub-map quota**: Track how many pending IDs each peer has contributed; evict the oldest entries from that peer when its quota is exceeded.
3. **Clean up on disconnect**: In `SyncState::disconnected()`, iterate `pending_get_block_proposals` and remove the disconnected `PeerIndex` from every `HashSet`; drop keys whose sets become empty.
4. **Avoid full clone in drain**: Replace the `clone()` + `clear()` pattern in `drain_get_block_proposals` with `std::mem::take` or an atomic swap to eliminate the O(n) duplicate allocation.
5. **Reduce drain interval or make it event-driven**: The current 100 ms interval already limits worst-case accumulation, but combining it with a size cap removes the residual risk entirely.

## Proof of Concept

```
1. Establish N TCP connections to the victim node (N Sybil peers).
2. For each peer, loop at ≤30 msg/s:
   a. Generate 3,000 random 10-byte ProposalShortId values (all fake).
   b. Build a GetBlockProposal message with any known tip block_hash
      and the 3,000 fake IDs.
   c. Send over the RelayV3 protocol.
3. None of the fake IDs exist in the victim's tx-pool, so all 3,000 are
   inserted into pending_get_block_proposals on every message.
4. Within each 100 ms drain window the map accumulates N × 3 × 3,000 entries.
5. Monitor victim RSS; it grows with N. At N ≈ 100–1,000 the peak heap
   (map + O(n) clone) reaches hundreds of MB to several GB.
6. Verify by adding a log of pending_get_block_proposals.len() immediately
   before drain_get_block_proposals() is called; confirm it grows linearly
   with N between ticks and that the O(n) clone causes a measurable
   latency spike in the relay timer loop.
```
