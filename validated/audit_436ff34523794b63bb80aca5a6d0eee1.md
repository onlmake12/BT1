### Title
Incomplete Fee Pre-Validation in `updatePriceFeeds` Causes Unexpected Reverts for Keepers — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `updatePriceFeeds` function in `Scheduler.sol` only pre-validates that the subscription balance covers the Pyth oracle fee before executing the combined operation (pay Pyth fee → validate → store prices → pay keeper). The keeper fee is only checked *after* the Pyth contract has already been called. When a subscription's balance falls between `pythFee` and `pythFee + totalKeeperFee`, the transaction reverts at the keeper payment step, wasting the keeper's gas and leaving the subscription unupdated.

### Finding Description

`updatePriceFeeds` performs a two-phase fee operation:

**Phase 1 — Pyth fee (pre-checked):** [1](#0-0) 

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
if (status.balanceInWei < pythFee) {
    revert SchedulerErrors.InsufficientBalance();
}
status.balanceInWei -= pythFee;
status.totalSpent += pythFee;
// ... Pyth contract is called here with {value: pythFee}
```

**Phase 2 — Keeper fee (checked only after Pyth call):** [2](#0-1) 

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
```

There is no upfront check that `balanceInWei >= pythFee + totalKeeperFee`. The function uses a "pay first, check later" approach for the keeper fee — analogous to the external report's `ONLY_SECOND_STEP` modality that approves funds without ensuring they are available for the full combined operation.

### Impact Explanation

When a subscription's balance satisfies `pythFee <= balanceInWei < pythFee + totalKeeperFee`:

1. The initial check passes (line 295)
2. `pythFee` is deducted and the Pyth contract is called (lines 305–319)
3. Price validation and storage succeed
4. `_processFeesAndPayKeeper` reverts with `InsufficientBalance` (line 852)
5. The entire transaction reverts — the keeper's gas is consumed with no compensation

The revert undoes all state changes (no fund loss to the subscription), but the keeper suffers an uncompensated gas loss. Subscriptions that have drained below the combined threshold become permanently un-updatable by any keeper until the manager adds funds, causing stale prices for downstream readers.

### Likelihood Explanation

Subscriptions naturally drain over time through repeated successful updates. The `getMinimumBalance` check at creation time does not guarantee the balance remains sufficient for future updates. The test suite explicitly demonstrates this drain scenario: [3](#0-2) 

Additionally, the test `testUpdatePriceFeedsRevertsInsufficientBalanceForKeeperFee` confirms that funding with exactly `mockPythFee + minKeeperFee` is insufficient because the actual gas cost of `updatePriceFeeds` exceeds `GAS_OVERHEAD`: [4](#0-3) 

Any permissionless keeper calling `updatePriceFeeds` on a subscription in this balance range will revert and lose gas.

### Recommendation

Add a pre-flight check for the minimum total fee before executing any state changes or external calls:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
uint256 minKeeperFee = GAS_OVERHEAD * tx.gasprice +
    uint256(_state.singleUpdateKeeperFeeInWei) * params.priceIds.length;
if (status.balanceInWei < pythFee + minKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
```

This mirrors the fix recommended in the external report: validate the full combined fee requirement upfront before committing to any step of the operation.

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
