### Title
One-Time Provider Registration With No Direct Withdrawal Path Permanently Locks Accrued Fees - (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol` enforces a one-time, irrevocable provider registration via `registerProvider()`. Unlike Entropy's `register()` which allows re-registration, Echo's `registerProvider()` permanently binds a provider address with no direct `withdraw()` function for the provider. The only fee-withdrawal path is `withdrawAsFeeManager()`, which requires a fee manager to have been pre-set. If no fee manager is configured (the default), all accrued provider fees are permanently locked in the contract with no recovery path.

### Finding Description

`registerProvider()` enforces a strict one-time registration guard:

```solidity
require(!provider.isRegistered, "Provider already registered");
``` [1](#0-0) 

Once registered, the `ProviderInfo` struct stores `accruedFeesInWei` and `feeManager` (defaulting to `address(0)`): [2](#0-1) 

Fees are credited to the provider in `executeCallback()`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [3](#0-2) 

The `IEcho` interface exposes **no** `withdraw(uint128 amount)` function for providers. The only withdrawal paths are:

1. `withdrawFees()` — admin-only, drains Pyth protocol fees only, not provider fees.
2. `withdrawAsFeeManager()` — requires `msg.sender == _state.providers[provider].feeManager`. [4](#0-3) 

If `feeManager` is `address(0)` (the default), the check `msg.sender == address(0)` can never pass, making `withdrawAsFeeManager` permanently uncallable for that provider. There is no admin rescue function for provider-specific `accruedFeesInWei`, and no de-register/re-register mechanism exists.

This contrasts directly with Entropy's `register()`, which explicitly preserves `accruedFeesInWei` across re-registrations and provides a direct `withdraw()` for the provider address: [5](#0-4) [6](#0-5) 

### Impact Explanation

A provider who registers without calling `setFeeManager()` — or whose fee manager address becomes inaccessible (key loss, compromised wallet, contract self-destruct) — permanently loses all accrued fees stored in `accruedFeesInWei`. Because `registerProvider()` is one-time and irrevocable, the provider cannot re-register under a new address to recover. The ETH is locked in the contract indefinitely with no admin escape hatch for provider balances.

### Likelihood Explanation

`setFeeManager` is an optional, separate call after registration. Any provider who registers and begins accruing fees without explicitly calling `setFeeManager` is immediately exposed. Since `feeManager` defaults to `address(0)`, this is the default state for every newly registered provider. Key rotation (a standard operational security practice) is impossible without first setting a fee manager, making this a realistic operational risk for any Echo provider. [7](#0-6) 

### Recommendation

1. Add a direct `withdraw(uint128 amount)` function callable by `msg.sender` when `msg.sender` is a registered provider (mirroring Entropy's design).
2. Allow re-registration (update fees/config) without blocking on `isRegistered`, preserving `accruedFeesInWei` across re-registrations.
3. Alternatively, add an admin-controlled emergency rescue function scoped to provider balances, or require `feeManager` to be set non-zero at registration time.

### Proof of Concept

```solidity
// 1. Provider registers without setting a fee manager
vm.prank(provider);
echo.registerProvider(1000, 1000, 1000);
// feeManager is address(0) by default

// 2. User requests price update, fees accrue to provider
vm.deal(user, 1 ether);
vm.prank(user);
echo.requestPriceUpdatesWithCallback{value: fee}(provider, publishTime, priceIds, gasLimit);

// 3. Provider executes callback, fees credited
echo.executeCallback(provider, seqNum, updateData, priceIds);
// provider.accruedFeesInWei > 0

// 4. Provider tries to withdraw — no withdraw() function exists
// Only path: withdrawAsFeeManager, but feeManager == address(0)
vm.prank(provider);
// echo.withdraw(amount); // DOES NOT EXIST in Echo

vm.prank(address(0)); // impossible
echo.withdrawAsFeeManager(provider, amount); // always reverts: "Only fee manager"

// 5. Provider cannot re-register under new address
vm.prank(newProvider);
echo.registerProvider(1000, 1000, 1000); // new address, but old fees are stuck forever
// require(!provider.isRegistered) passes for new address, but old fees remain locked
``` [8](#0-7) [4](#0-3) [9](#0-8)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
    }
