### Title
Incorrect `timeout_count` Accumulation in `DownloadScheduler::decrease` Causes Overly Aggressive Peer Penalization — (File: `sync/src/types/mod.rs`)

---

### Summary

The `DownloadScheduler::decrease` function in the block-download scheduler overwrites `self.timeout_count` with `self.task_count.saturating_add(num)` instead of accumulating `self.timeout_count.saturating_add(num)`. Because `task_count` is always initialized to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (a value well above 2), the guard condition `timeout_count > 2` is satisfied on every single call, causing `task_count` to be decremented on every slow block response rather than every two slow responses as intended. This is a direct analog to the "uninitialized variable" class: a running accumulator (`timeout_count`) is never actually accumulated — it is reset to the wrong base value on each invocation, producing behavior that diverges from the design on every call.

---

### Finding Description

In `sync/src/types/mod.rs`, `DownloadScheduler` is the per-peer structure that controls how many concurrent block-download tasks a peer is allowed. When a block arrives slowly (response time falls in the `NormalToUpper` or `UpperToMax` quantile), `remove_by_block` calls `set.decrease(1)` or `set.decrease(2)` respectively.

The intended design of `decrease` is to accumulate a `timeout_count` and only decrement `task_count` after two accumulated slow responses, providing a tolerance buffer before penalizing a peer:

```rust
// sync/src/types/mod.rs  lines 521-527
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: uses task_count, not timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

The correct line should be:
```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

Because `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (a constant larger than 2), the expression `self.task_count.saturating_add(num)` always produces a value greater than 2. The guard `if self.timeout_count > 2` is therefore always true, `task_count` is decremented on every call, and `timeout_count` is immediately reset to 0. The accumulator is never actually accumulated — it is overwritten with the wrong variable on every invocation, exactly mirroring the "uninitialized variable" pattern where a state field never holds the value the subsequent logic expects.

The `Default` for `DownloadScheduler` initializes `timeout_count` to 0:

```rust
// sync/src/types/mod.rs  lines 489-496
impl Default for DownloadScheduler {
    fn default() -> Self {
        Self {
            hashes: HashSet::default(),
            task_count: INIT_BLOCKS_IN_TRANSIT_PER_PEER,
            timeout_count: 0,
        }
    }
}
```

Once `task_count` reaches 0, `prune()` disconnects the peer:

```rust
// sync/src/types/mod.rs  lines 692-700
download_schedulers.retain(|k, v| {
    if v.task_count == 0 {
        disconnect_list.insert(*k);
        false
    } else {
        true
    }
});
```

The full call chain triggered by any remote peer sending a block:

```
receive block → remove_by_block (line 785)
  → time_analyzer.push_time(elapsed) (line 798)
  → TimeQuantile::NormalToUpper → set.decrease(1)   (line 803)
     or TimeQuantile::UpperToMax → set.decrease(2)   (line 808)
  → task_count decremented immediately (bug)
→ prune() → task_count == 0 → peer disconnected
```

---

### Impact Explanation

Every peer whose block-delivery time falls in the `NormalToUpper` or `UpperToMax` range (i.e., between `normal_time` 1250 ms and above `low_time` 1500 ms by default) has its `task_count` decremented on every single slow response instead of every two. A peer starting at `INIT_BLOCKS_IN_TRANSIT_PER_PEER` will reach `task_count == 0` in exactly `INIT_BLOCKS_IN_TRANSIT_PER_PEER` slow responses and be forcibly disconnected. In high-latency or congested network conditions — which are common during IBD — legitimate peers are systematically and prematurely evicted. If all connected peers experience latency in this range, the node loses all sync peers and stalls block download entirely, constituting a denial-of-sync condition reachable without any privileged access.

---

### Likelihood Explanation

The `NormalToUpper` and `UpperToMax` quantiles represent the majority of real-world network conditions under load. The default `normal_time` is 1250 ms and `low_time` is 1500 ms; any peer responding between these values triggers `decrease`. Because the bug causes `task_count` to drop on every such response, a peer with consistently moderate latency (e.g., a cross-continental peer) will be disconnected after a small number of block downloads. This is not a rare edge case — it is the normal operating range for many legitimate peers.

---

### Recommendation

Fix `decrease` to accumulate `timeout_count` rather than overwrite it with `task_count`:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // was: self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

This restores the intended two-strike tolerance before penalizing a peer's download quota.

---

### Proof of Concept

1. Connect to a CKB node as an unprivileged sync peer.
2. Respond to `GetBlocks` requests with block delivery times consistently between 1250 ms and 1500 ms (the `NormalToUpper` range).
3. Each delivered block triggers `remove_by_block` → `time_analyzer.push_time` → `TimeQuantile::NormalToUpper` → `set.decrease(1)`.
4. Due to the bug, `task_count` is decremented on every response (not every two).
5. After `INIT_BLOCKS_IN_TRANSIT_PER_PEER` such responses, `task_count` reaches 0.
6. The next `prune()` call disconnects this peer.
7. If all connected peers exhibit similar latency (e.g., in a geographically distributed network under load), the victim node disconnects all peers and stalls IBD.
