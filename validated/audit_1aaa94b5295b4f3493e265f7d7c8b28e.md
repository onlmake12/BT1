Audit Report

## Title
Missing `BLOCK_INVALID` Early-Return in `HeaderAcceptor::accept()` Causes Redundant PoW Re-verification — (File: sync/src/synchronizer/headers_process.rs)

## Summary
`HeaderAcceptor::accept()` checks cached block status but only short-circuits on `HEADER_VALID`, not `BLOCK_INVALID`. A developer-acknowledged `FIXME` at lines 301–302 confirms the gap. Any peer can repeatedly send `SendHeaders` messages containing a previously-rejected header, forcing one Eaglesong PoW hash computation per message with no rate-limit applied to the Sync protocol's `SendHeaders` handler.

## Finding Description
In `sync/src/synchronizer/headers_process.rs` lines 301–304, `accept()` reads the cached status and returns early only for `HEADER_VALID`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) { ... return result; }
```

`BLOCK_INVALID = 1 << 12` is disjoint from `HEADER_VALID = 1` (`shared/src/block_status.rs` lines 11–16), so `status.contains(BlockStatus::HEADER_VALID)` is always `false` for an invalid header and the early-return is never triggered.

Execution falls through to `non_contextual_check`, which calls `HeaderVerifier::verify()`. That function runs `PowVerifier` as its **first** step (`verification/src/header_verifier.rs` line 34) before any cheaper checks. The `execute()` loop (`headers_process.rs` lines 154–179) does break on the first `ValidationState::Invalid`, so only one Eaglesong hash is computed per message — but the attacker can send a new `SendHeaders` message immediately with no back-pressure, as the Sync protocol handler (`sync/src/synchronizer/mod.rs` lines 381–422) has no rate limiter for `SendHeaders` (unlike the Relay protocol which has a 30 req/s limiter at `sync/src/relayer/mod.rs` lines 89–92).

By contrast, `compact_block_process.rs` lines 259–261 correctly short-circuits on `BLOCK_INVALID` before any verification.

## Impact Explanation
The concrete impact is repeated, unnecessary CPU work — one Eaglesong hash per `SendHeaders` message — with no rate-limit or peer-ban applied to senders of already-invalid headers. Eaglesong is a single hash computation, fast in absolute terms. This does not crash a node or cause network-wide congestion. The impact maps to **Low (501–2000 points): any other important performance improvements for CKB**.

## Likelihood Explanation
Triggerable by any unprivileged peer with a standard sync connection. The attacker needs one previously-rejected header hash, obtainable by sending a single malformed header and observing the rejection. The `FIXME` comment in production source confirms developer awareness. No special privileges, key material, or hashpower are required. The attack is repeatable at the rate the attacker can send TCP packets.

## Recommendation
Add an explicit `BLOCK_INVALID` early-return at the top of `accept()`, mirroring `compact_block_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing peer-state update
    return result;
}
```

## Proof of Concept
1. Connect to a CKB node as a standard sync peer.
2. Send a `SendHeaders` message with one header `H` whose PoW nonce is invalid. The node runs `PowVerifier`, fails, and marks `H` as `BLOCK_INVALID` in the shared status cache.
3. Repeatedly send `SendHeaders` messages each containing `H` as the first header.
4. For each message, `accept()` reads `status = BLOCK_INVALID`, does not short-circuit (FIXME path), and re-executes `PowVerifier::verify()` (one Eaglesong hash) before returning `ValidationState::Invalid`.
5. CPU cost per message is one Eaglesong hash; attacker cost is one TCP packet. The ratio is bounded but non-zero and unbounded in aggregate across message rate, with no Sync-protocol rate limiter to constrain it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
