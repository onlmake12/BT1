### Title
`PythLazer.sol` `verifyUpdate` Accepts Arbitrarily Old Price Updates ‚Äî No Payload Timestamp Freshness Check (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

The `verifyUpdate` function in `PythLazer.sol` is the sole on-chain security gate for Pyth Lazer price updates on EVM chains. It verifies the ECDSA signature and checks that the signer has not expired, but it **never inspects the timestamp embedded in the payload**. Any unprivileged caller can replay a validly-signed Lazer price update from arbitrarily far in the past, and the contract will accept it and return the stale payload as verified.

---

### Finding Description

`verifyUpdate` in `PythLazer.sol` performs exactly three checks:

1. EVM format magic bytes match.
2. ECDSA signature recovers to a non-zero address.
3. The recovered address is a currently-valid trusted signer (`isValidSigner`).

```solidity
// lazer/contracts/evm/src/PythLazer.sol  lines 70-106
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    ...
    payload = update[71:71 + payload_len];
    bytes32 hash = keccak256(payload);
    (signer, , ) = ECDSA.tryRecover(hash, ...);
    if (signer == address(0)) { revert("invalid signature"); }
    if (!isValidSigner(signer)) { revert("invalid signer"); }
    // ‚Üê NO timestamp check on `payload`
}
```

The payload's first field is a `uint64` timestamp (microseconds), as documented and parsed by `PythLazerLib.parsePayloadHeader`:

```solidity
// PythLazerLib.sol ‚Äì parsePayloadHeader
(uint64 _timestamp, Channel channel, uint8 feedsLen, uint16 pos) = parsePayloadHeader(payload);
```

`verifyUpdate` never reads or bounds-checks `_timestamp`. A Lazer update signed by a trusted key that was valid at any point in the past will pass `verifyUpdate` today, as long as the signer's `expiresAt` has not yet elapsed. Trusted signers are configured with long-lived expiry windows (the test fixture uses `expiresAt` values far in the future).

This is structurally identical to the ThecosomataETH finding: the oracle verification layer (here `verifyUpdate`; there `price_oracle`) returns a price without any freshness guarantee, leaving every downstream consumer exposed.

---

### Impact Explanation

Any consumer contract that calls `verifyUpdate` and then uses the returned `payload` without independently checking `_timestamp` will silently accept a price from an arbitrary point in the past. Because `verifyUpdate` is the only on-chain security primitive Pyth Lazer exposes on EVM, and because the documentation example does not enforce a staleness check in the contract itself, real consumer deployments are likely to omit the check.

Concrete consequences:
- A lending protocol using Lazer for collateral valuation can be fed a price from hours ago, enabling under-collateralised borrows or wrongful liquidations.
- A perp/derivatives protocol can be fed a stale price to open or close positions at a favourable historical price, extracting value from the protocol.
- The attack requires no privileged access and no capital: old signed Lazer messages are broadcast publicly and can be captured by any observer.

---

### Likelihood Explanation

Lazer price updates are streamed publicly over the Lazer WebSocket. Any observer can record them. A trusted signer key is valid for an extended period (governance-controlled `expiresAt`). The attacker's only requirement is to have captured a signed update at a time when the price differed materially from the current price ‚Äî trivially achievable during any volatile period. The attack is permissionless, requires no on-chain capital, and is repeatable.

---

### Recommendation

Add an optional `maxAge` parameter (in seconds or microseconds) to `verifyUpdate`, or expose a separate `verifyUpdateNoOlderThan(bytes calldata update, uint64 maxAgeUs)` variant that parses the first 8 bytes of the payload as the timestamp and reverts if `block.timestamp * 1e6 - timestamp > maxAgeUs`. This mirrors the `getPriceNoOlderThan` pattern already present in Pyth Core's EVM contracts.

```solidity
// Suggested addition inside verifyUpdate or a new wrapper:
uint64 payloadTimestampUs = uint64(bytes8(payload[0:8]));
require(
    block.timestamp * 1_000_000 <= payloadTimestampUs + maxAgeUs,
    "stale Lazer update"
);
```

---

### Proof of Concept

1. Subscribe to the Lazer WebSocket and record a signed update `U_old` at time `T_old` when asset price was `P_old`.
2. Wait until the current price `P_now` differs materially from `P_old` (e.g., >5%).
3. Call `pythLazer.verifyUpdate{value: fee}(U_old)` on-chain. The call succeeds and returns `payload` containing timestamp `T_old` and price `P_old`.
4. Pass the returned `payload` to any consumer contract that does not independently check `_timestamp`. The consumer uses `P_old` as if it were the current price.
5. Exploit the price discrepancy (e.g., open a position at the stale price, immediately close at the real price).

No special privileges, no flash loan, and no on-chain capital are required beyond the `verification_fee`. [1](#0-0) [2](#0-1)
