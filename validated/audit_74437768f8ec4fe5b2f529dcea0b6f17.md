### Title
Stale `descendants_*` Accounting in `remove_entry_and_descendants` Leaves Ancestor Eviction Keys Permanently Inflated — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

`remove_entry_and_descendants` strips all link records from every entry in the removal set **before** calling `remove_entry` on each one. Because `update_ancestors_index_key` discovers ancestors through those same link records, it finds an empty ancestor set for every removed entry and never decrements the `descendants_size / descendants_cycles / descendants_fee / descendants_count` fields of entries that remain in the pool. Those ancestors permanently carry inflated descendant accounting, and their `EvictKey` is never corrected.

---

### Finding Description

`remove_entry_and_descendants` is the only bulk-removal path in the pool. It is called from `resolve_conflict`, `resolve_conflict_header_dep`, `limit_size`, and `check_and_record_ancestors`.

```
// tx-pool/src/component/pool_map.rs  lines 252-265
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← strips ALL link records first
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← links already gone
        .collect()
}
```

Inside `remove_entry` (lines 235-250), the first thing done after extracting the entry is:

```
self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
```

`update_ancestors_index_key` (lines 432-445) resolves ancestors through `self.links.calc_ancestors`:

```
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());   // ← always empty here
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
```

Because `remove_entry_links` already deleted the link record for every entry in `removed_ids`, `calc_ancestors` returns an empty set for each of them. No ancestor's `descendants_*` fields are decremented, and no ancestor's `evict_key` is recomputed.

The single-entry path (`remove_entry` called directly, not via `remove_entry_and_descendants`) is **correct**: links are still intact when `update_ancestors_index_key` runs, so ancestors are found and updated. The bug is exclusive to the bulk path.

The `EvictKey` is computed from the stale fields:

```
// tx-pool/src/component/entry.rs  lines 234-247
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_weight =
            get_transaction_weight(entry.descendants_size, entry.descendants_cycles);
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        let feerate = FeeRate::calculate(entry.fee, weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),   // ← inflated
            ...
            descendants_count: entry.descendants_count,   // ← inflated
        }
    }
}
```

---

### Impact Explanation

An ancestor whose high-fee descendants were removed via conflict resolution retains an inflated `EvictKey`. When `limit_size` runs, it evicts entries in ascending `EvictKey` order. The ancestor with the stale (high) key is skipped in favour of legitimate transactions that have accurate, lower keys. A low-fee root transaction can therefore occupy pool space indefinitely while correctly-accounted higher-fee transactions are displaced.

The stale accounting also accumulates across repeated replacement cycles: each time a new descendant is added (`add_descendant_weight` is called correctly on add), the ancestor's `descendants_fee` grows by the new descendant's fee, but it is never reduced when that descendant is later removed via `remove_entry_and_descendants`. Over multiple cycles the ancestor's apparent fee rate diverges arbitrarily from its real fee rate.

---

### Likelihood Explanation

The attack path is fully unprivileged and reachable via the public `send_transaction` RPC:

1. Submit root tx with a low fee rate (e.g., 1 shannon/byte).
2. Submit a child tx with a high fee rate (e.g., 1 000 shannons/byte) spending the root's output.
3. Submit a second tx that spends the same output as the child (conflict). The child is removed via `resolve_conflict` → `remove_entry_and_descendants`. The root's `descendants_*` are not decremented.
4. Root's `EvictKey` remains inflated by the child's fee. When the pool fills, the root is not evicted; legitimate mid-fee transactions are evicted instead.

No privileged access, no majority hash power, and no social engineering are required.

---

### Recommendation

Before stripping link records, compute and apply the descendant-weight correction to all ancestors that lie **outside** the removal set:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // NEW: update ancestors of the root entry before links are torn down
    if let Some(root_entry) = self.get(id).cloned() {
        self.update_ancestors_index_key(&root_entry, EntryOp::Remove);
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

Alternatively, restructure `remove_entry_and_descendants` so that it only removes links **between** entries in the removal set, leaving the links from the root to its external ancestors intact until `update_ancestors_index_key` has run.
