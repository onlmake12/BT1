Audit Report

## Title
Stale Pre-Eviction Snapshot Overwrites Correctly-Decremented `total_tx_size` in `PoolMap::add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures `self.total_tx_size + entry.size` into a local variable before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts cell-dep parent transactions, those evictions correctly decrement `self.total_tx_size` via `update_stat_for_remove_tx`. However, the stale pre-eviction snapshot is then unconditionally written back to `self.total_tx_size`, permanently inflating the counter by the aggregate size of all evicted transactions. Repeated exploitation causes `limit_size` to evict valid transactions and reject new submissions with `Reject::Full` even when the pool is nearly empty.

## Finding Description
In `add_entry` (lines 200–221 of `tx-pool/src/component/pool_map.rs`):

- **Lines 210–211**: `updated_stat_for_add_tx(entry.size, entry.cycles)` reads `self.total_tx_size` and returns `self.total_tx_size + entry.size` as a local — a snapshot taken before any evictions.
- **Line 213**: `check_and_record_ancestors(&mut entry)` is called. At lines 603–625, when `ancestors_count > self.max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, `remove_entry_and_descendants` is called in a loop. Each call reaches `remove_entry` (line 263), which calls `update_stat_for_remove_tx` (line 247), correctly decrementing `self.total_tx_size` (lines 738–740).
- **Lines 218–219**: The stale pre-eviction snapshot is unconditionally written back, overwriting every decrement made during evictions.

`update_stat_for_remove_tx`'s self-healing path (`recompute_total_stat`, lines 742–756) is only triggered on underflow, never on overflow, so the inflation is never corrected.

**Net effect per invocation:**

| Counter | Correct value | Actual value written |
|---|---|---|
| `total_tx_size` | `original − Σ(evicted.size) + entry.size` | `original + entry.size` |
| `total_tx_cycles` | `original − Σ(evicted.cycles) + entry.cycles` | `original + entry.cycles` |

## Impact Explanation
`limit_size` in `tx-pool/src/pool.rs` (lines 298–326) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting transactions and issuing `Reject::Full` until the counter drops below the cap. An inflated `total_tx_size` causes this loop to fire even when actual pool occupancy is far below the limit, evicting legitimate pending transactions and rejecting new valid submissions. The inflation accumulates across repeated attack invocations. This matches the **High** severity bounty impact: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation
The trigger condition — a new transaction whose ancestor count via cell-dep references exceeds `max_ancestors_count` but can be reduced by evicting cell-dep parents — is reachable by any unprivileged caller of the `send_transaction` RPC or any P2P relay peer. No key material, privileged access, or majority hash power is required. The attacker pre-populates the pool with a cell-dep chain, then submits a crafted transaction to trigger the eviction branch. The attack is repeatable and the inflation compounds with each iteration.

## Recommendation
Move the stat computation to after `check_and_record_ancestors` completes, so evictions are already reflected in `self.total_tx_size` before the snapshot is taken:

```diff
- let (total_tx_size, total_tx_cycles) =
-     self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
  trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
  evicts = self.check_and_record_ancestors(&mut entry)?;
  self.record_entry_edges(&entry)?;
  self.insert_entry(&entry, status);
  self.record_entry_descendants(&entry);
  self.track_entry_statics(None, Some(status));
+ let (total_tx_size, total_tx_cycles) =
+     self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
  self.total_tx_size = total_tx_size;
  self.total_tx_cycles = total_tx_cycles;
```

Alternatively, remove the snapshot pattern entirely and increment `self.total_tx_size` / `self.total_tx_cycles` inline after all evictions complete, eliminating the stale local variable.

## Proof of Concept
1. Configure a node with `max_ancestors_count = N` and a modest `max_tx_pool_size`.
2. Submit transactions `T1 … TN` where each `Ti+1` references `Ti` as a cell dep, forming a chain of depth `N`. Record `total_tx_size_before`.
3. Submit `Tnew` referencing `T1` as a cell dep. Its ancestor count is `N+1 > max_ancestors_count`, but `ancestors_count - cell_ref_parents.len() = N <= max_ancestors_count`, so the eviction branch at lines 603–625 fires. `T1` (and its descendants) are evicted; `self.total_tx_size` is decremented by their aggregate size. Then line 218 writes back the pre-eviction snapshot, re-inflating `total_tx_size` by that same aggregate.
4. Query pool stats and observe `total_tx_size > total_tx_size_before + Tnew.size` — the inflation equals the evicted transactions' sizes.
5. Repeat steps 2–4. Each iteration inflates `total_tx_size` further.
6. After enough iterations, `total_tx_size > max_tx_pool_size` despite the pool being nearly empty. `limit_size` begins evicting all remaining transactions and rejecting new submissions with `Reject::Full`.

A unit test can be written against `PoolMap::add_entry` directly: assert that after a triggered eviction, `pool_map.total_tx_size` equals the sum of sizes of entries actually present in `pool_map.entries`, and verify the current code fails this invariant.
