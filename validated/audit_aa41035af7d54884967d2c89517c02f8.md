### Title
Unguarded `initialize` Allows Attacker to Front-Run Config Account Creation and Permanently DoS the `pyth-price-store` Program — (File: `target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

---

### Summary

The `initialize` function in the `pyth-price-store` Solana program creates the singleton config PDA using a raw `system_instruction::create_account` call via `invoke_signed`. The payer account is validated only as a signer — there is no check that the payer is a specific privileged admin. The `authority` field is passed freely as an argument. Any unprivileged actor can call `initialize` before the legitimate deployer, supply a malicious `authority`, and permanently prevent the legitimate admin from ever initializing the program.

---

### Finding Description

In `initialize.rs`, the program creates the config account as follows:

```rust
pub fn initialize(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    args: &InitializeArgs,
) -> ProgramResult {
    let mut accounts = accounts.iter();
    let payer = validate_payer(accounts.next())?;          // any signer
    let config = validate_config(accounts.next(), args.config_bump, program_id, true)?;
    let system = validate_system(accounts.next())?;

    let lamports = (Rent::get()?).minimum_balance(accounts::config::SIZE);

    invoke_signed(
        &system_instruction::create_account(
            payer.key,
            config.key,
            lamports,
            accounts::config::SIZE.try_into().expect("unexpected overflow"),
            program_id,
        ),
        &[payer.clone(), config.clone(), system.clone()],
        &[&[CONFIG_SEED.as_bytes(), &[args.config_bump]]],
    )?;

    accounts::config::create(*config.data.borrow_mut(), args.authority)?;
    Ok(())
}
``` [1](#0-0) 

The config PDA is derived solely from the constant `CONFIG_SEED`, making its address fully deterministic and publicly computable. The `validate_payer` call enforces only that the account is a transaction signer — the test confirms the payer is simply the transaction fee-payer with no admin identity check: [2](#0-1) 

The `args.authority` value is entirely attacker-controlled. Because `system_instruction::create_account` fails if the target account already exists, whichever caller reaches the chain first wins permanently.

---

### Impact Explanation

An attacker who calls `initialize` before the legitimate deployer:

1. **Permanently DoS's the legitimate admin's initialization** — the config PDA already exists; the admin's `create_account` call returns an error and can never succeed.
2. **Installs a malicious authority** — all downstream privileged operations (`initialize_publisher`, publisher config updates, price submissions) are gated on the authority stored in the config account. With a malicious authority installed, the attacker controls the entire publisher permission system of the price store.

The config account is a singleton; there is no recovery path once it is created with the wrong authority. [3](#0-2) 

---

### Likelihood Explanation

The attack requires only:
- Monitoring the mempool (or simply racing the deployment transaction) for the `initialize` instruction.
- Submitting a transaction with the same accounts but an attacker-controlled `authority` pubkey.

No privileged access, leaked keys, or oracle manipulation is needed. The config PDA address is deterministic and computable off-chain from `CONFIG_SEED` and the program ID. This is a realistic, low-cost attack executable by any unprivileged actor at deployment time. [4](#0-3) 

---

### Recommendation

Restrict the `initialize` instruction so that only a hardcoded or upgrade-authority-derived admin can call it. Two standard mitigations:

1. **Hardcode the expected authority pubkey** and verify `payer.key == &EXPECTED_ADMIN` inside `validate_payer` (or a dedicated check before `create_account`).
2. **Derive the expected payer from the program's upgrade authority** (via the BPF Upgradeable Loader's `ProgramData` account) and assert the payer matches.

Either approach ensures that even if an attacker submits the transaction first, the program rejects it because the payer is not the legitimate admin.

---

### Proof of Concept

```
1. Compute config PDA:
   (config, bump) = find_program_address([CONFIG_SEED], PYTH_PRICE_STORE_PROGRAM_ID)

2. Attacker constructs InitializeArgs { config_bump: bump, authority: attacker_pubkey }

3. Attacker submits `initialize` instruction with:
   - accounts[0]: attacker keypair (signer, payer)
   - accounts[1]: config PDA
   - accounts[2]: system_program

4. Transaction succeeds; config account is created with authority = attacker_pubkey.

5. Legitimate admin submits the same instruction:
   - system_instruction::create_account fails: "account already in use"
   - Program is permanently initialized with attacker's authority.

6. All subsequent `initialize_publisher` calls require the authority to sign;
   the attacker now controls which publishers are registered.
``` [5](#0-4)
