Audit Report

## Title
Missing Per-Peer Rate Limit on Light Client Protocol Enables Unbounded MMR Database Reads — (`util/light-client-protocol-server/src/lib.rs`)

## Summary
`LightClientProtocol` carries no rate limiter of any kind. Every `GetLastState` message from any peer unconditionally invokes `get_verifiable_tip_header()`, which performs a RocksDB block-body read and O(log N) MMR reads from `COLUMN_CHAIN_ROOT_MMR`. The `Relayer` and `HolePunching` protocols both enforce a `governor`-based 30 req/s per-peer rate limit before any handler dispatch; the light-client protocol has no equivalent guard anywhere in its codebase.

## Finding Description
`LightClientProtocol` is defined with only a `shared: Shared` field and no `rate_limiter`: [1](#0-0) 

`try_process` dispatches to all handlers with zero rate-limit checks: [2](#0-1) 

`GetLastStateProcess::execute()` calls `get_verifiable_tip_header()` unconditionally on every invocation, regardless of the `subscribe` flag: [3](#0-2) 

`get_verifiable_tip_header()` performs: (1) `self.shared.snapshot()` — Arc clone of chain snapshot; (2) `snapshot.get_block(&tip_hash)` — RocksDB read from `COLUMN_BLOCK_BODY`; (3) `snapshot.chain_root_mmr(tip_block.number() - 1).get_root()` — O(log N) RocksDB reads from `COLUMN_CHAIN_ROOT_MMR`: [4](#0-3) 

By contrast, `Relayer` carries an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` field: [5](#0-4) 

And checks it before any handler dispatch: [6](#0-5) 

`HolePunching` applies the same pattern: [7](#0-6) 

A grep for `rate_limiter` under `util/light-client-protocol-server/` returns zero matches, confirming no guard exists anywhere in the light-client protocol codebase.

## Impact Explanation
A single unprivileged peer connecting on the `LightClient` protocol can send `GetLastState` messages at the maximum rate the TCP connection allows. Each message unconditionally triggers one RocksDB block-body read and O(log N) RocksDB reads from `COLUMN_CHAIN_ROOT_MMR` (where N is the current chain height — on mainnet, millions of blocks, yielding ~20–23 peak MMR node reads per message). Under sustained attack from even a single peer, this produces unbounded CPU and I/O amplification with no per-peer throttle, which can degrade or crash the full node. This matches the allowed bounty impact: **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
Any peer that can open a connection on the `LightClient` protocol (no authentication, no PoW, no privileged role required) can trigger this. The `subscribe` flag value is irrelevant — the MMR read occurs on every `GetLastState` message unconditionally. The attack is trivially scriptable, requires no special knowledge, and is repeatable indefinitely. The impact scales linearly with the number of attacking peers and with chain height (larger MMR = more reads per `get_root()` call).

## Recommendation
Add a `governor`-based `RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`, mirroring the pattern in `Relayer`: [8](#0-7) 

Check it at the top of `try_process` before dispatching to any handler, using the same 30 req/s per-peer-per-message-type quota already established in the codebase. Call `rate_limiter.retain_recent()` in the `disconnected` handler to prevent unbounded memory growth.

## Proof of Concept
1. Connect a peer to the node's `LightClient` protocol endpoint (no credentials required).
2. In a tight loop, send `GetLastState { subscribe: false }` (or `true` — the work is identical).
3. Observe via RocksDB statistics or `perf` that `COLUMN_CHAIN_ROOT_MMR` read IOPS and CPU time scale linearly with message rate.
4. Assert that no `TooManyRequests` status is ever returned (unlike `Relayer`, which returns it after 30 req/s per peer per message type).
5. Confirm node degradation or crash under sustained load from a single peer.

    pub(crate) fn get_verifiable_tip_header(&self) -> Result<packed::VerifiableHeader, String> {
        let snapshot = self.shared.snapshot();

        let tip_hash = snapshot.tip_hash();
        let tip_block = snapshot
            .get_block(&tip_hash)
            .expect("checked: tip block should be existed");
        let parent_chain_root = if tip_block.is_genesis() {
            Default::default()
        } else {
            let mmr = snapshot.chain_root_mmr(tip_block.number() - 1);
            match mmr.get_root() {
                Ok(root) => root,
                Err(err) => {
                    let errmsg = format!("failed to generate a root since {err:?}");
                    return Err(errmsg);
                }
            }
        };
