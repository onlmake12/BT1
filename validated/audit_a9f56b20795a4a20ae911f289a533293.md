### Title
Permanent Subscription Deposit Limit Bypassed via Cumulative `addFunds` Calls - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `MAX_DEPOSIT_LIMIT` guard in `Scheduler.addFunds` checks only the per-transaction `msg.value` instead of the resulting cumulative balance. Because `addFunds` has no access-control restriction, any caller can make repeated deposits each ≤ `MAX_DEPOSIT_LIMIT` and accumulate an arbitrarily large balance in a permanent subscription — permanently locking those funds with no withdrawal path.

---

### Finding Description

`Scheduler.addFunds` enforces the deposit cap for permanent subscriptions as follows:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [1](#0-0) 

The check is on `msg.value` (the single-call deposit), not on `status.balanceInWei + msg.value` (the post-deposit cumulative balance). `MAX_DEPOSIT_LIMIT` is 100 ETH: [2](#0-1) 

Because `addFunds` has **no access control** — any address may call it for any subscription — a caller can invoke it N times with `msg.value == MAX_DEPOSIT_LIMIT` each time, growing the balance to `N × 100 ETH`. The same per-transaction-only check exists in `createSubscription`: [3](#0-2) 

Permanent subscriptions explicitly prohibit withdrawals: [4](#0-3) 

So any ETH deposited beyond the intended cap is permanently locked in the contract with no recovery path.

The analogy to the reported vulnerability class is direct: the check is applied to the **wrong quantity** (`msg.value` per call instead of the cumulative balance), just as the original report's `checkLastBlockAction` tracked the wrong identity (`msg.sender` of the intermediary instead of the actual end user).

---

### Impact Explanation

- **Griefing / permanent fund lock**: Because `addFunds` is permissionless, a third party can deposit large amounts of ETH into any permanent subscription, permanently locking those funds. The subscription manager has no mechanism to recover them.
- **Deposit-limit safety mechanism is ineffective**: The `MAX_DEPOSIT_LIMIT` constant is rendered meaningless for permanent subscriptions.
- **No direct theft**, but irreversible loss of deposited ETH and potential accounting confusion for subscription managers that track their own balances.

---

### Likelihood Explanation

- `addFunds` is a public, payable function with no caller restriction (confirmed by the test `testAnyoneCanAddFunds`). [5](#0-4) 
- The bypass requires only two sequential calls, each with `msg.value == MAX_DEPOSIT_LIMIT`.
- The attacker loses their own ETH, which limits motivation for profit-driven attackers but not for griefing.

---

### Recommendation

Replace the per-transaction check with a cumulative-balance check in both `addFunds` and `createSubscription`:

```solidity
// addFunds
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// createSubscription
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) { // already correct here since balance starts at 0
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

The `createSubscription` check is already correct (balance starts at zero), but `addFunds` must account for the existing balance.

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription with MAX_DEPOSIT_LIMIT (100 ETH)
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);

// 2. Any address calls addFunds twice — each call passes the per-tx check
scheduler.addFunds{value: 100 ether}(subId);  // balance = 200 ETH, passes (100 == 100, not > 100)
scheduler.addFunds{value: 100 ether}(subId);  // balance = 300 ETH, passes

// 3. withdrawFunds reverts — permanent subscription
scheduler.withdrawFunds(subId, 1 ether);  // CannotUpdatePermanentSubscription

// 300 ETH is now permanently locked, far exceeding the intended 100 ETH cap.
``` [6](#0-5)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
