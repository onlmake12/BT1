### Title
Uncapped `tx.gasprice` in Keeper Fee Reimbursement Drains Subscription Balances — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `_processFeesAndPayKeeper` function reimburses keepers for gas costs using `tx.gasprice` with no upper bound. A malicious keeper can inflate `tx.gasprice` to drain a subscription's balance far faster than the manager intended, analogous to the external report's "inverted slippage" pattern where the protocol absorbs costs that should be borne by the user.

---

### Finding Description

In `_processFeesAndPayKeeper`, the keeper reimbursement is calculated as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [1](#0-0) 

The keeper's net profit from the gas overhead component alone is `GAS_OVERHEAD * tx.gasprice`. Because `tx.gasprice` is fully attacker-controlled (via the EIP-1559 priority fee), a keeper can set an arbitrarily high gas price. The subscription balance is then debited at that inflated rate:

```solidity
status.balanceInWei -= totalKeeperFee;
``` [2](#0-1) 

There is no cap on `tx.gasprice` anywhere in `updatePriceFeeds` or `_processFeesAndPayKeeper`. The only guard is an `InsufficientBalance` revert, which does not prevent the drain — it only stops it once the balance is exhausted. [3](#0-2) 

The entry point is fully permissionless — `updatePriceFeeds` has no access control:

```solidity
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {
    uint256 startGas = gasleft();
``` [4](#0-3)

### Impact Explanation

A subscription manager deposits funds expecting to pay for N updates at prevailing gas prices. A malicious keeper submitting valid updates at an inflated `tx.gasprice` (e.g., 1000× normal) drains the subscription in N/1000 updates. The keeper profits from `GAS_OVERHEAD * inflated_gasprice` per call, which is pure extraction from the subscription balance. Subscription managers lose deposited funds faster than expected, and their subscriptions become inactive prematurely.

**Impact: Medium** — limited funds lost from subscription balances; no protocol-wide treasury at risk.

---

### Likelihood Explanation

**Likelihood: Low** — the attacker must pay the inflated gas price to the network themselves (though they recover it plus the overhead from the subscription). On chains with low base fees, a keeper can set a very high priority fee, pay a small base cost, and extract a large overhead reimbursement. The attack is economically rational when `GAS_OVERHEAD * (inflated_gasprice - baseFee) > 0`, which is always true if the priority fee exceeds zero. Requires a keeper willing to front gas costs, but the overhead profit makes it self-funding.

---

### Recommendation

Introduce a `maxGasPriceInWei` parameter (configurable by admin) and cap the gas price used in the reimbursement calculation:

```solidity
uint256 effectiveGasPrice = tx.gasprice;
if (_state.maxGasPriceInWei > 0 && effectiveGasPrice > _state.maxGasPriceInWei) {
    effectiveGasPrice = _state.maxGasPriceInWei;
}
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, subscription managers could set their own `maxGasPrice` per subscription, so they control the maximum rate at which their balance can be drained.

---

### Proof of Concept

1. Subscription manager creates a subscription with 1 ETH balance, expecting 1000 updates at 1 gwei.
2. Malicious keeper calls `updatePriceFeeds` with valid update data but sets `tx.gasprice = 1000 gwei`.
3. `gasCost = (actualGasUsed + GAS_OVERHEAD) * 1000 gwei` — 1000× the expected cost.
4. After ~1 update, the subscription balance is drained.
5. The keeper's net profit per call: `GAS_OVERHEAD * 1000 gwei` (minus the priority fee paid to the network, which is recovered from the subscription).
6. The subscription becomes inactive after a single update instead of 1000. [5](#0-4)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
