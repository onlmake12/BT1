### Title
Incorrect Epoch Ordering Check in `HeaderDigest::verify()` Allows Temporally Inverted Digest to Pass Validation â€” (`util/types/src/utilities/merkle_mountain_range.rs`)

---

### Summary

`HeaderDigest::verify()` uses a raw integer index comparison (`start_epoch.index() > end_epoch.index()`) when both epochs share the same epoch number, instead of the rational cross-multiplication comparison implemented in `EpochNumberWithFraction::cmp()`. This allows a crafted digest where `start_epoch` is fractionally greater than `end_epoch` (i.e., temporally inverted) to pass validation and return `Ok(())`.

---

### Finding Description

The guard at lines 84â€“93 of `merkle_mountain_range.rs` reads:

```rust
if start_epoch != end_epoch
    && ((start_epoch_number > end_epoch_number)
        || (start_epoch_number == end_epoch_number
            && start_epoch.index() > end_epoch.index()))
``` [1](#0-0) 

When `start_epoch_number == end_epoch_number`, the code only compares raw `.index()` values. It does **not** account for differing epoch lengths.

The correct ordering is implemented in `EpochNumberWithFraction::cmp()`:

```rust
let a = self.index() * other.length();
let b = other.index() * self.length();
a.cmp(&b)
``` [2](#0-1) 

This is a cross-multiplication rational comparison. The `verify()` function bypasses it entirely for the same-epoch case.

**Concrete counterexample:**

| Field | Value |
|---|---|
| `start_epoch` | `(number=1, index=5, length=10)` â†’ fraction = 5/10 = **0.5** |
| `end_epoch` | `(number=1, index=6, length=20)` â†’ fraction = 6/20 = **0.3** |

- `start_epoch_number == end_epoch_number` â†’ true (both = 1)
- `start_epoch.index() > end_epoch.index()` â†’ `5 > 6` â†’ **false**
- Guard does not fire â†’ `verify()` returns `Ok(())`

But via `Ord::cmp`: `a = 5 * 20 = 100`, `b = 6 * 10 = 60`, `a > b` â†’ `start_epoch > end_epoch` â†’ this is a **temporally inverted** digest.

---

### Impact Explanation

`HeaderDigest` is the core data structure of the light client MMR proof system (CKB RFC 0044). A malicious full node serving light clients over P2P can craft a `HeaderDigest` with an inverted epoch range that passes `verify()`. The light client accepts this digest as structurally valid, undermining the integrity of MMR-based chain state proofs â€” including total difficulty accounting and block range attestation. [3](#0-2) 

---

### Likelihood Explanation

Any unprivileged peer acting as a full node in the P2P network can send crafted `SendLastStateProof`, `SendBlocksProof`, or `SendTransactionsProof` messages containing a malformed `HeaderDigest`. No special privileges, keys, or majority hashpower are required. The scenario is directly constructible and locally testable.

---

### Recommendation

Replace the raw index comparison in `verify()` with the proper `Ord`-based comparison:

```rust
// Before (incorrect):
start_epoch_number == end_epoch_number
    && start_epoch.index() > end_epoch.index()

// After (correct):
start_epoch > end_epoch
```

This delegates to `EpochNumberWithFraction::cmp()`, which performs the correct cross-multiplied rational comparison and handles differing epoch lengths within the same epoch number. [1](#0-0)

### Proof of Concept

```rust
use ckb_types::{
    core::EpochNumberWithFraction,
    packed,
    prelude::*,
    utilities::merkle_mountain_range::HeaderDigest,
};

let start_epoch = EpochNumberWithFraction::new_unchecked(1, 5, 10); // 0.5 through epoch 1
let end_epoch   = EpochNumberWithFraction::new_unchecked(1, 6, 20); // 0.3 through epoch 1

// Confirm rational ordering: start > end
assert!(start_epoch > end_epoch, "start is fractionally after end");

let digest = packed::HeaderDigest::new_builder()
    .start_number(10u64.pack())
    .end_number(11u64.pack())   // start_number <= end_number passes check #1
    .start_epoch(start_epoch.pack())
    .end_epoch(end_epoch.pack())
    // ... other fields set to consistent values
    .build();

// Bug: verify() returns Ok(()) despite start_epoch > end_epoch
let result = digest.verify();
assert!(result.is_err(), "expected Err for inverted epoch, got Ok");
// This assertion FAILS â€” demonstrating the bug.
```

The raw index comparison `5 > 6` is false, so the guard at line 87 does not trigger, and `verify()` returns `Ok(())` for a digest whose epoch range is inverted. [1](#0-0) [4](#0-3)
