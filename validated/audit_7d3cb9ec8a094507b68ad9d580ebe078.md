## Critical Prerequisite Assessment

The attack requires the Wormhole guardians to sign both switch VAAs. However, the dangerous scenario does **not** require malicious intent:

- Governance legitimately switches from source A → B (e.g., key rotation). `initial_sequence=0` is a natural default for a brand-new source.
- Governance later switches back from B → A (e.g., rollback, emergency). Again sets `initial_sequence=0`.
- At this point, an **unprivileged relayer** holding any old VAA from source A with sequence 1..4 can replay it with no further guardian involvement.

The relayer is unprivileged. The guardians acted in good faith. The code provides no protection.

### Title
Governance Sequence Reset via `SET_GOVERNANCE_DATA_SOURCE` Enables Replay of Old Governance VAAs — (`target_chains/sui/contracts/sources/governance/set_governance_data_source.move`)

### Summary
`set_governance_data_source::execute` unconditionally overwrites `last_executed_governance_sequence` with the caller-supplied `initial_sequence` field (which may be 0), destroying the monotonic sequence invariant enforced by `execute_governance_instruction`. Any subsequent source switch-back leaves old VAAs from the reinstated source replayable by an unprivileged relayer.

### Finding Description
In `execute_governance_instruction`, the sequence monotonicity check (`sequence > last_executed`) is performed and the counter is updated to the current VAA's sequence. Control then passes to `set_governance_data_source::execute`, which calls `state::set_last_executed_governance_sequence(latest_only, pyth_state, initial_sequence)` — a second, unconditional write that replaces the just-written value with whatever `initial_sequence` the VAA payload encodes. [5](#0-4) 

There is no check that `initial_sequence >= last_executed_governance_sequence` for the case where the new source was previously used. The `state::set_last_executed_governance_sequence` function is a blind write with no guard. [4](#0-3) 

### Impact Explanation
After a source A → B → A round-trip with `initial_sequence=0`, any VAA previously executed from source A (sequences 1..N-1) satisfies `sequence > 0` and passes the monotonicity check. An unprivileged relayer can replay `contract_upgrade`, `set_data_sources`, `set_fee_recipient`, or any other governance action that was previously executed, achieving unauthorized governance voting result manipulation and potential unauthorized contract control.

### Likelihood Explanation
The trigger is a legitimate governance source switch-back, which is a plausible operational event (key rotation rollback, emergency recovery). The `initial_sequence=0` value is the natural default for a "fresh" source. No malicious guardian collusion is required — only a good-faith governance action followed by an unprivileged relay of a previously-seen VAA.

### Recommendation
In `set_governance_data_source::execute`, enforce that `initial_sequence` is at least as large as the current `last_executed_governance_sequence` before overwriting it:

```move
let current = state::get_last_executed_governance_sequence(pyth_state);
assert!(initial_sequence >= current, E_INITIAL_SEQUENCE_TOO_LOW);
state::set_last_executed_governance_sequence(latest_only, pyth_state, initial_sequence);
```

This ensures the sequence counter can only move forward, even across source changes.
