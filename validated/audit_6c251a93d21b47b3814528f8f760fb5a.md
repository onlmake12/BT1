### Title
Missing Maximum Fee Guard in `requestV2()` Causes Irrecoverable Overpayment When Fees Change Between Query and Execution - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `requestV2()` family of functions accepts a dynamic fee computed at execution time, but provides no `maxFee` parameter for callers to cap what they are willing to pay. All `msg.value` above the provider's share is permanently credited to Pyth's accrued fees with no refund. Because fees are adjusted continuously by the keeper and can be changed at any time by the provider, users who send a buffer above the queried fee to avoid reverts silently and irrecoverably overpay.

---

### Finding Description

The fee required to call `requestV2()` is computed at execution time inside `requestHelper()`:

```solidity
// Entropy.sol lines 234–239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value` is consumed: `providerFee` goes to the provider and the remainder (`msg.value - providerFee`) goes to `_state.accruedPythFeesInWei`. **No excess is refunded.** This is explicitly documented in the interface:

> "Further note that excess value is *not* refunded to the caller." [2](#0-1) 

The fee is dynamically computed from two mutable components:

1. **Provider fee** (`provider.feeInWei`) — changeable at any time by the provider via `setProviderFee()` or by the designated fee manager via `setProviderFeeAsFeeManager()`. [3](#0-2) 

2. **Pyth base fee** (`_state.pythFeeInWei`) — governance-controlled and also variable. [4](#0-3) 

Additionally, the Fortuna keeper (`apps/fortuna/src/keeper/fee.rs`) continuously adjusts `provider.feeInWei` on-chain based on prevailing gas prices, meaning the fee can change between any two blocks. [5](#0-4) 

None of the `requestV2()` overloads accept a `maxFee` parameter:

```solidity
function requestV2() external payable returns (uint64);
function requestV2(uint32 gasLimit) external payable returns (uint64);
function requestV2(address provider, uint32 gasLimit) external payable returns (uint64);
function requestV2(address provider, bytes32 userRandomNumber, uint32 gasLimit) external payable returns (uint64);
``` [6](#0-5) 

This creates an inescapable dilemma for callers:

- **Send exactly `getFeeV2()`**: If the fee increases before the tx executes, the tx reverts with `InsufficientFee`. The user must retry, paying gas again.
- **Send `getFeeV2() + buffer`**: The buffer is silently and permanently consumed by `_state.accruedPythFeesInWei`. There is no mechanism to recover it.

There is no third option. Users cannot express "execute only if fee ≤ X."

---

### Impact Explanation

Any caller of `requestV2()` who sends `msg.value` greater than the exact fee at execution time permanently loses the excess to Pyth's accrued fee pool. The loss is proportional to the buffer sent. For consumer contracts that wrap `requestV2()` and forward `msg.value` from end users (a common pattern shown in the official docs), the end user bears this loss with no recourse. The funds are not recoverable by the user — they accrue to Pyth's treasury. [7](#0-6) 

---

### Likelihood Explanation

The official documentation explicitly warns that fees change over time and instructs callers to query `getFeeV2()` before each invocation. This warning implicitly encourages sending a buffer to avoid reverts. The Fortuna keeper adjusts fees on every chain continuously based on gas prices, so fee changes between query and execution are routine, not exceptional. Any consumer contract or EOA that follows the recommended pattern of sending a small buffer will silently overpay on every request where the fee has decreased since the query. [8](#0-7) 

---

### Recommendation

**Short term:** Add a `maxFee` parameter to all `requestV2()` variants. Inside `requestHelper()`, revert if `requiredFee > maxFee`. This allows callers to express an upper bound on what they are willing to pay, mirroring standard slippage-protection patterns in DeFi.

**Long term:** Refund excess `msg.value` to the caller. Change the accounting in `requestHelper()` so that only `requiredFee` is consumed and `msg.value - requiredFee` is returned to `msg.sender`. This eliminates the overpayment risk entirely and removes the need for callers to compute the exact fee.

---

### Proof of Concept

1. Alice's consumer contract calls `getFeeV2()` off-chain and gets `1000 wei`.
2. Alice's contract sends `requestV2{value: 1100 wei}()` (10% buffer to avoid reverts).
3. At execution time, `getFeeV2()` returns `900 wei` (fee decreased due to lower gas prices).
4. `requestHelper` executes: `requiredFee = 900`, `msg.value (1100) >= requiredFee (900)` — passes.
5. `providerFee = 800`, `providerInfo.accruedFeesInWei += 800`, `_state.accruedPythFeesInWei += (1100 - 800) = 300`.
6. Alice paid `1100 wei` but only `900 wei` was required. The extra `200 wei` is permanently in Pyth's treasury. Alice has no way to recover it and had no way to prevent it. [9](#0-8)
