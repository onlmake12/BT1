### Title
User Fee Permanently Locked in Echo Contract When Request Is Never Fulfilled — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo` contract splits the user's payment into a Pyth protocol fee (credited immediately to `_state.accruedFeesInWei`) and a provider fee (stored in `req.fee`). The provider fee is only credited to the provider's `accruedFeesInWei` when `executeCallback` is called. There is no cancellation or refund mechanism. If a request is never fulfilled, the provider fee portion is permanently locked in the contract's ETH balance — inaccessible to the user, the provider, and the admin.

---

### Finding Description

In `requestPriceUpdatesWithCallback`, the user pays `msg.value`. The contract immediately credits the Pyth protocol fee:

```solidity
_state.accruedFeesInWei += _state.pythFeeInWei;
```

and stores the remainder as the provider fee in the request:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [1](#0-0) 

The provider fee is only credited when `executeCallback` is called:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

If `executeCallback` is never called, `req.fee` remains in the contract's ETH balance but is tracked in no withdrawable accounting variable. The only admin-withdrawable balance is `_state.accruedFeesInWei`:

```solidity
function withdrawFees(uint128 amount) external override {
    require(msg.sender == _state.admin, "Only admin can withdraw fees");
    require(_state.accruedFeesInWei >= amount, "Insufficient balance");
    _state.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [3](#0-2) 

Provider withdrawals only access `_state.providers[provider].accruedFeesInWei`, which is never incremented for unfulfilled requests. There is no `cancelRequest`, timeout, or emergency-rescue function anywhere in the contract. [4](#0-3) 

---

### Impact Explanation

For every unfulfilled request, the ETH corresponding to `req.fee` is permanently locked in the contract. Neither the user (no refund path), the provider (their `accruedFeesInWei` is never incremented), nor the admin (can only withdraw `_state.accruedFeesInWei`) can recover it. This results in **permanent, irrecoverable loss of user funds** proportional to the number of unfulfilled requests.

---

### Likelihood Explanation

A malicious provider can register via `registerProvider`, attract users to pay fees via `requestPriceUpdatesWithCallback`, and then simply never call `executeCallback`. The exclusivity period only restricts *which* provider can fulfill during the window — it does not force fulfillment or enable refunds after the window expires. Any provider going offline (technical failure or deliberate) produces the same outcome. The entry path is fully unprivileged: any registered provider can trigger this for any user. [5](#0-4) 

---

### Recommendation

Add a request expiry mechanism (e.g., based on `publishTime` + a grace period) and a `cancelRequest` function that refunds `req.fee` to `req.requester` when the deadline has passed without fulfillment. Alternatively, credit the provider fee to `providerInfo.accruedFeesInWei` at request time (as Entropy does) so that at minimum the funds are not permanently locked, and separately implement a slashing/penalty mechanism for non-fulfillment.

---

### Proof of Concept

1. Malicious provider calls `registerProvider(baseFee, feedFee, gasFee)`.
2. User calls `requestPriceUpdatesWithCallback{value: totalFee}(maliciousProvider, ...)`.
   - `_state.accruedFeesInWei += pythFeeInWei` ✓ (admin can withdraw this)
   - `req.fee = totalFee - pythFeeInWei` stored in request slot
3. Malicious provider never calls `executeCallback`.
4. User attempts to recover funds — no `cancelRequest` function exists.
5. Admin attempts to withdraw — `withdrawFees` only covers `accruedFeesInWei`; `req.fee` is not included.
6. Provider attempts to withdraw — `accruedFeesInWei` for provider is 0; nothing to withdraw.
7. `req.fee` worth of ETH is permanently locked in the contract. [6](#0-5)
