Audit Report

## Title
Unbounded `block_status_map` Growth via Permanent `BLOCK_INVALID` Insertions Without Eviction or Peer Banning — (`shared/src/shared.rs`, `sync/src/synchronizer/headers_process.rs`)

## Summary

`block_status_map` (`Arc<DashMap<Byte32, BlockStatus>>`) in `shared/src/shared.rs` has no capacity bound and no eviction policy. Every unique invalid header received via the sync protocol causes a permanent `BLOCK_INVALID` insertion through `insert_block_status()`. Because `remove_block_status()` is never called for `BLOCK_INVALID` entries on the header-validation failure paths, and because `HeadersProcess` never calls `ban_peer`, an unprivileged remote peer can drive unbounded heap growth and crash the node via OOM.

## Finding Description

The map is declared with no size limit at `shared/src/shared.rs:72`:

```rust
pub(crate) block_status_map: Arc<DashMap<Byte32, BlockStatus>>,
```

`insert_block_status()` at `shared/src/shared.rs:455-457` performs an unconditional insert with no guard:

```rust
pub fn insert_block_status(&self, block_hash: Byte32, status: BlockStatus) {
    self.block_status_map.insert(block_hash, status);
}
```

In `HeaderAcceptor::accept()` at `sync/src/synchronizer/headers_process.rs:295-358`, the only early-return guard checks `BlockStatus::HEADER_VALID` (line 304). There is an explicit `FIXME` comment at line 301 acknowledging that no early return exists for `BLOCK_INVALID`. Three distinct failure branches each call `insert_block_status(..., BLOCK_INVALID)` with no corresponding removal:

- `prev_block_check` failure → line 330
- `non_contextual_check` failure (when `is_invalid == true`) → line 341
- `version_check` failure → line 352

`remove_block_status()` is only called on the success path (`chain/src/verify.rs:143`) and on the internal-DB-error path (`chain/src/verify.rs:180`). For every block that fails contextual verification, `BLOCK_INVALID` is inserted and the entry is never removed.

A grep for `ban_peer` in `sync/src/synchronizer/headers_process.rs` returns zero matches, confirming no peer is disconnected or banned after submitting invalid headers.

## Impact Explanation

This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

`block_status_map` resides entirely in process heap memory. Each entry is a `Byte32` key (32 bytes) plus `BlockStatus` metadata plus `DashMap` internal overhead (~100–200 bytes per entry). Sending 10 million unique invalid header hashes consumes roughly 1–2 GB of heap. At sufficient scale the OS OOM killer terminates the node process, constituting a complete node crash.

## Likelihood Explanation

Any unprivileged remote peer acting as a header relayer can trigger this. The attacker constructs `SendHeaders` messages containing headers with unique hashes that fail `version_check` (version field != 0, which is trivially constructable without any PoW capability) or `non_contextual_check` (invalid PoW, varying the nonce to produce a unique hash each iteration). Because no banning occurs in `HeadersProcess`, the attacker is not disconnected after a single invalid submission. Even if the connection is dropped, reconnection is trivial. IP rotation makes address-based banning ineffective. No rate limiting was found in the sync path. The attack accumulates entries across sessions.

## Recommendation

1. **Cap the map size**: Enforce a maximum entry count (e.g., 1 million entries). When the cap is reached, evict the oldest or least-recently-used `BLOCK_INVALID` entries before inserting new ones.
2. **Periodic TTL-based cleanup**: Add a background task that removes `BLOCK_INVALID` entries older than a configurable TTL (e.g., 24 hours).
3. **Peer banning on invalid headers**: In `HeaderAcceptor::accept()` and `HeadersProcess::execute()`, call `ban_peer` when a header fails validation with `ValidationState::Invalid`, preventing the same peer from repeatedly injecting entries.
4. **Resolve the FIXME**: Add an early return in `HeaderAcceptor::accept()` when `status == BLOCK_INVALID` (line 301–302), preventing redundant re-processing of already-known-invalid hashes.
5. **Per-peer rate limiting**: Enforce a per-peer limit on the number of invalid headers accepted per time window before banning.

## Proof of Concept

1. Establish a sync protocol connection to a CKB mainnet node.
2. In a loop, construct `SendHeaders` messages each containing a single header with `version != 0` (e.g., version = 1) and a unique nonce, producing a unique hash each iteration. This requires no PoW capability — the header fails `version_check` before any PoW check is needed, or alternatively use headers with invalid PoW that fail `non_contextual_check`.
3. Send each message. After each batch, observe `/proc/<pid>/status` (`VmRSS`) on the node host — resident memory grows monotonically.
4. Confirm via debug logging that `insert_block_status(..., BLOCK_INVALID)` is called at `sync/src/synchronizer/headers_process.rs:352` (version failure) or line 341 (non-contextual failure) for each header, and that no corresponding `remove_block_status` is ever called.
5. After sufficient iterations (scale depends on attacker bandwidth and node RAM), the node process is terminated by the OS OOM killer or becomes unresponsive due to memory pressure.
