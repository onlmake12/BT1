### Title
Zero-Tolerance Publish Time Window in `executeCallback()` Causes Permanent User Fund Lock — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback()` calls `IPyth.parsePriceFeedUpdates()` with `minPublishTime == maxPublishTime == req.publishTime`. This is the direct analog of the M-09 zero-slippage pattern: just as `amountADesired == amountAMin` in `addLiquidity()` leaves no tolerance for deviation, setting both timestamp bounds to the same value leaves zero tolerance for the price feed's actual `publishTime`. Any price feed data whose `publishTime` differs by even one second from `req.publishTime` causes `parsePriceFeedUpdates` to revert with `PriceFeedNotFoundWithinRange`, making the request permanently unfulfillable and locking the user's fee in the contract with no refund path.

---

### Finding Description

In `Echo.sol`, a user calls `requestPriceUpdatesWithCallback(provider, publishTime, priceIds, callbackGasLimit)` and pays a fee. The `publishTime` is stored in the request struct. Later, a provider (or anyone after the exclusivity period) calls `executeCallback()` to fulfill the request. Inside `executeCallback()`, the Pyth oracle is queried:

```solidity
// Echo.sol lines 144–153
IPyth pyth = IPyth(_state.pyth);
uint256 pythFee = pyth.getUpdateFee(updateData);
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
    value: pythFee
}(
    updateData,
    priceIds,
    SafeCast.toUint64(req.publishTime),   // minPublishTime
    SafeCast.toUint64(req.publishTime)    // maxPublishTime  ← same value
);
```

`parsePriceFeedUpdates` in `Pyth.sol` (lines 320–399) accepts a price feed only when `publishTime >= minAllowedPublishTime && publishTime <= maxAllowedPublishTime`. When both bounds are identical, only an exact timestamp match is accepted. If no price feed in `updateData` has `publishTime == req.publishTime`, the function reverts with `PriceFeedNotFoundWithinRange` at line 393, propagating the revert up through `executeCallback()`.

Because `executeCallback()` reverts before `clearRequest()` is called (line 164) and before the provider's `accruedFeesInWei` is credited (line 161–162), the request remains active and the user's fee (`req.fee`) stays locked in the contract. There is no `cancelRequest`, no user-facing refund function, and no timeout-based recovery mechanism anywhere in `Echo.sol` or `IEcho.sol`.

The developer's own inline TODO at lines 155–156 acknowledges this exact risk:
> *"TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract."*

---

### Impact Explanation

- **User funds permanently locked.** The fee paid via `requestPriceUpdatesWithCallback` (stored as `req.fee`) cannot be recovered. There is no refund or cancel path.
- **Callback never executed.** The consumer contract's `_echoCallback` is never invoked, breaking any downstream application logic that depends on receiving the price update.
- **Denial of service on the request.** The request slot remains active indefinitely (or until overwritten by a new request in the ring buffer), but can never be fulfilled.

---

### Likelihood Explanation

Pyth price feeds are published at discrete timestamps driven by the Pythnet slot cadence (~400 ms slots). A user who calls `requestPriceUpdatesWithCallback` with `publishTime = block.timestamp` (the natural choice, as shown in the test at `Echo.t.sol` line 225) is requesting data at a specific EVM block timestamp. The Pyth network's `publishTime` values are derived from Pythnet slot times, which are not synchronized to EVM block timestamps. It is routine for the nearest available price feed to have a `publishTime` that is 1–2 seconds off from the requested value. Additionally, the interface explicitly allows `publishTime` up to 60 seconds in the future (`require(publishTime <= block.timestamp + 60)`), making exact matches even less likely for future-dated requests. Any such mismatch causes a permanent lock.

---

### Recommendation

Pass a tolerance window instead of a point value. For example, allow a configurable `±N` second window:

```solidity
pyth.parsePriceFeedUpdates{value: pythFee}(
    updateData,
    priceIds,
    SafeCast.toUint64(req.publishTime),
    SafeCast.toUint64(req.publishTime + PUBLISH_TIME_TOLERANCE)  // e.g., +5 seconds
);
```

Additionally, implement a user-facing `cancelRequest(uint64 sequenceNumber)` function that allows refunds after a timeout period, so that even if `executeCallback` is permanently blocked, user funds are not locked forever.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback(provider, block.timestamp, priceIds, gasLimit)` paying fee `F`. Request is stored with `req.publishTime = T`.
2. Pyth network publishes price data at `publishTime = T+1` (one slot later).
3. Provider calls `executeCallback(provider, seqNum, updateData, priceIds)` with the only available update data (published at `T+1`).
4. Inside `executeCallback`, `parsePriceFeedUpdates` is called with `minPublishTime = T, maxPublishTime = T`.
5. `parsePriceFeedUpdates` checks `T+1 >= T && T+1 <= T` → false. Price feed slot remains `id == 0`.
6. At line 392–394 of `Pyth.sol`, `revert PythErrors.PriceFeedNotFoundWithinRange()` is thrown.
7. `executeCallback` reverts. `clearRequest` is never called. `req.fee = F - pythFeeInWei` remains locked.
