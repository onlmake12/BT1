Audit Report

## Title
Wrong Variable Reference in `script_exists_in_output` Causes Script Deletion During Block Rollback on PostgreSQL - (File: util/rich-indexer/src/indexer/remove.rs)

## Summary
In `script_exists_in_output`, after fetching `row_type` (the type-script EXISTS query result), the final match statement on line 252 reads from `row_lock` instead of `row_type`. On PostgreSQL, `row_lock.try_get::<bool, _>(0)` always succeeds and always returns `false` at that point (since `true` would have caused an early return), so the function unconditionally returns `Ok(false)` for any script checked as a type script. During `rollback_block`, this causes every type-script-only script ID to be added to the deletion list and removed from the `script` table, corrupting the Rich Indexer's relational database.

## Finding Description
`script_exists_in_output` performs two SQL `EXISTS` queries: one for `lock_script_id` and one for `type_script_id`. [1](#0-0) 

The first query result is stored in `row_lock`. If it is `true`, the function returns early. [2](#0-1) 

The second query result is stored in `row_type`. However, the final match on line 252 reads `row_lock` again instead of `row_type`: [3](#0-2) 

On PostgreSQL, `try_get::<bool, _>(0)` succeeds on `row_lock` and returns `false` (since `true` would have triggered the early return above). The `Ok(r)` arm therefore always returns `Ok(false)`, completely ignoring `row_type`. The SQLite path is unaffected because `try_get::<bool, _>` fails on SQLite's `BIGINT` result, falling through to the `Err` branch which correctly reads `row_type`.

`rollback_block` calls `script_exists_in_output` for every type script ID encountered in the rolled-back outputs, and unconditionally deletes all IDs for which it returns `false`: [4](#0-3) 

## Impact Explanation
This is a concrete incorrect implementation of the CKB Rich Indexer's state storage mechanism. During any block rollback, script rows that are still referenced by surviving outputs via `type_script_id` are deleted from the `script` table. This corrupts the indexer's relational state: subsequent Rich Indexer RPC calls (e.g., `get_cells`, `get_transactions`) filtering by type script return incorrect or empty results for affected scripts. If foreign-key constraints are enforced, subsequent inserts or queries may produce database errors. Full re-indexing from genesis is required to recover. This matches **Medium (2001–10000 points): Suboptimal/incorrect implementation of CKB state storage mechanism**.

## Likelihood Explanation
Block rollbacks occur naturally during any chain reorganization, which is a routine event on a live network. No attacker action is required — any reorg of depth ≥ 1 involving outputs with type scripts will trigger the faulty path. The bug is deterministic and reproducible on any PostgreSQL-backed Rich Indexer deployment that experiences a reorg. The Rich Indexer is a production-supported component, and PostgreSQL is the recommended backend for production deployments.

## Recommendation
On line 252, replace `row_lock` with `row_type`:

```rust
// BEFORE (buggy):
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}

// AFTER (correct):
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

## Proof of Concept
1. Run a CKB node with the Rich Indexer configured to use PostgreSQL.
2. Submit a transaction whose output has a `type_script` (e.g., a Simple UDT cell). Let it be included in block N.
3. Trigger a chain reorganization that rolls back block N.
4. Observe that `rollback_block` is called, which calls `script_exists_in_output(type_script_id, tx)`.
5. Because `row_lock` is `false` at line 252 and the match reads `row_lock` instead of `row_type`, the function returns `false` even though the type script is still referenced by surviving outputs.
6. The script row is deleted from the `script` table.
7. Subsequent `get_cells` RPC calls filtering by that type script return empty results, confirming database corruption.

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
