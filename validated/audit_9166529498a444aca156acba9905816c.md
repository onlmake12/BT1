### Title
Wrong Row Variable Used in `script_exists_in_output` Causes Silent Script Table Corruption on PostgreSQL During Block Rollback - (File: `util/rich-indexer/src/indexer/remove.rs`)

### Summary

In `script_exists_in_output`, the final match block at line 252 reads from `row_lock` instead of `row_type` when evaluating the PostgreSQL boolean result for the type-script existence check. On PostgreSQL, this causes the function to return the lock-script query result instead of the type-script query result. During `rollback_block`, any script that is referenced only as a `type_script_id` (not as a `lock_script_id`) in surviving outputs is incorrectly reported as absent and is permanently deleted from the `script` table, while the `output` rows that reference it via `type_script_id` remain. This silently corrupts the rich-indexer database on every reorg when PostgreSQL is the backend.

### Finding Description

`script_exists_in_output` is called from `rollback_block` to decide whether a script can be safely garbage-collected after the rolled-back block's outputs are removed. [1](#0-0) 

The function issues two SQL queries:

1. `row_lock` — `SELECT EXISTS (… WHERE lock_script_id = $1)` — checks whether the script is still referenced as a lock script in any remaining output.
2. `row_type` — `SELECT EXISTS (… WHERE type_script_id = $1)` — checks whether the script is still referenced as a type script in any remaining output. [2](#0-1) 

After the second query, the code should evaluate `row_type` to return the type-script existence result. Instead, line 252 re-reads `row_lock`:

```rust
// Line 252 — BUG: row_lock used instead of row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [3](#0-2) 

On **PostgreSQL**, `try_get::<bool, _>(0)` succeeds (PostgreSQL returns `BOOLEAN`), so the `Ok(r)` arm is always taken. The value returned is the lock-script existence result, not the type-script existence result. On **SQLite**, `try_get::<bool, _>` fails (SQLite returns `BIGINT`), so the `Err` arm falls through to `row_type.get::<i64, _>(0) == 1`, which is correct.

**Consequence in `rollback_block`:**

```
rollback_block
  └─ query_outputs_by_tx_id_list  → collects (output_id, lock_script_id, type_script_id)
  └─ remove outputs from DB
  └─ for each type_script_id:
       script_exists_in_output(type_script_id)
         → row_lock query: script NOT a lock script → false
         → row_type query: script IS a type script → true (correct answer)
         → but returns row_lock result (false) on PostgreSQL  ← BUG
       → script incorrectly added to script_id_list_to_remove
  └─ DELETE FROM script WHERE id IN (…)  ← deletes still-referenced scripts
``` [4](#0-3) 

After the rollback, surviving `output` rows still hold `type_script_id` values pointing to now-deleted `script` rows. The `script` table has no foreign-key constraint enforcing referential integrity: [5](#0-4) 

All subsequent rich-indexer RPC calls that join `output` with `script` on `type_script_id` (`get_cells`, `get_transactions`, `get_cells_capacity` with `script_type: Type`) will silently return empty or incorrect results for any script that was only used as a type script in surviving outputs.

### Impact Explanation

Any CKB node operator running the rich indexer with a PostgreSQL backend experiences silent, permanent corruption of the `script` table on every block rollback (reorg). After corruption:

- `get_cells` / `get_transactions` / `get_cells_capacity` RPC calls keyed on a type script return empty results for cells that are actually live.
- Users and applications relying on the rich indexer to locate their type-script-bearing cells (e.g., DAO deposit cells, UDT cells, NFT cells) cannot find them through the RPC, even though the cells exist on-chain.
- The corruption is permanent and cumulative: each reorg that involves type-script-only scripts worsens the state. The only recovery is a full re-index.

### Likelihood Explanation

- Reorgs are a normal, frequent occurrence on CKB mainnet and testnet; no attacker is required.
- The rich indexer with PostgreSQL is an officially documented and supported deployment mode.
- Any block containing outputs with type scripts (DAO, UDT, NFT, TYPE_ID cells — extremely common) is sufficient to trigger the bug on the next rollback of that block.
- The SQLite path is unaffected, so the bug is invisible in the default embedded-SQLite configuration and in all existing tests (which use SQLite). [6](#0-5) 

### Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// Correct fix
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Additionally, add a PostgreSQL-backed integration test for `rollback_block` that includes outputs with type scripts to prevent regression.

### Proof of Concept

**Setup:** CKB node with `--rich-indexer` and PostgreSQL backend.

1. Submit a block containing a transaction with an output that has a type script (e.g., a TYPE_ID cell or DAO deposit). The script is inserted into the `script` table; the output references it via `type_script_id`.
2. Trigger a reorg that rolls back this block (e.g., a competing chain tip arrives).
3. `rollback_block` is called → `script_exists_in_output(type_script_id)` is invoked.
4. `row_lock` query returns `false` (the script is not a lock script). `row_type` query returns `true` (the script is a type script in a surviving output). But the function returns `row_lock`'s result (`false`) on PostgreSQL.
5. The script is added to `script_id_list_to_remove` and deleted from the `script` table.
6. Query `get_cells` with the type script → returns empty, even though the cell is live on-chain. [1](#0-0)

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
