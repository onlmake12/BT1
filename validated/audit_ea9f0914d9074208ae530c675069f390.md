Audit Report

## Title
Wrong Column Traversal in `migrate_transaction_info` Leaves `COLUMN_TRANSACTION_INFO` Unmigrated, Causing Panic on Any Transaction Lookup — (`util/migrate/src/migrations/table_to_struct.rs`)

## Summary

`ChangeMoleculeTableToStruct::migrate_transaction_info()` traverses `COLUMN_UNCLES` instead of `COLUMN_TRANSACTION_INFO` at line 93, leaving all pre-migration 68-byte table-format transaction info entries untouched. After the migration bumps the DB version to `"20200703124523"`, any call to `get_transaction_info()` reads a 68-byte entry and passes it to `from_slice_should_be_ok`, which panics unconditionally because `TransactionInfoReader::verify` requires exactly 52 bytes (`TOTAL_SIZE`). The node process crashes and cannot recover without manual DB repair or full re-sync.

## Finding Description

**Root cause — line 93:**

The closure `transaction_info_migration` is defined to read from and write to `COLUMN_TRANSACTION_INFO`, but the `db.traverse` call iterates over `COLUMN_UNCLES` instead: [1](#0-0) 

Every other sub-migration correctly names its own column: [2](#0-1) [3](#0-2) 

**Execution sequence:**

1. `migrate_uncles()` runs first (line 153) and rewrites uncle `HeaderView` entries from 240 → 228 bytes in `COLUMN_UNCLES`. [4](#0-3) 

2. `migrate_transaction_info()` then traverses `COLUMN_UNCLES` (now 228-byte entries). Since `228 != TRANSACTION_INFO_SIZE (52)`, the guard at line 84 passes for every uncle entry, and the closure writes `uncle_value[16:]` (212 bytes) into `COLUMN_TRANSACTION_INFO` under each uncle's hash key. [5](#0-4) 

3. The actual `COLUMN_TRANSACTION_INFO` entries (68-byte old table format) are **never read or rewritten**.

4. Migration returns `Ok(db)` and the DB version is bumped to `"20200703124523"`. [6](#0-5) 

**Panic path after restart:**

`get_transaction_info` calls `from_slice_should_be_ok` on the stored bytes: [7](#0-6) 

`from_slice_should_be_ok` panics unconditionally on any verification error: [8](#0-7) 

`TransactionInfoReader::verify` rejects any slice that is not exactly `TOTAL_SIZE` (52) bytes. The unmigrated entries are 68 bytes (52 + 16 bytes of molecule table overhead), so `verify` returns `TotalSizeNotMatch(52, 68)` and the node panics with:
```
failed to convert from slice: reason: TotalSizeNotMatch(52, 68); data: 0x<68 bytes>
```

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.** `get_transaction_info` is called from RPC handlers, chain sync, and script verification paths. A Rust panic in any of these threads crashes the node process. The node is rendered permanently inoperable on the migrated database until the DB is manually repaired or re-synced from scratch.

## Likelihood Explanation

The precondition is trivially met on any real mainnet or testnet node: uncle blocks exist in every epoch, and their post-migration serialized size (228 bytes) is never 52 bytes, so the guard always passes. Any node operator who ran `ckb migrate` for the v0.35.0 upgrade on a pre-existing database would trigger this. No attacker capability is required — the bug is self-inflicted by the migration tool on legitimate node operators.

## Recommendation

Fix line 93 to traverse `COLUMN_TRANSACTION_INFO` instead of `COLUMN_UNCLES`:

```rust
let (_count, nk) =
    db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
```

Additionally, introduce a follow-up repair migration for nodes that already ran the broken migration: it should delete all spurious entries in `COLUMN_TRANSACTION_INFO` whose keys are uncle hashes (identifiable by 212-byte values) and re-migrate the actual 68-byte transaction info entries.

## Proof of Concept

1. Start a CKB node on a pre-v0.35.0 database containing at least one uncle block (true of any mainnet/testnet DB).
2. Run `ckb migrate`.
3. Observe migration completes with no error and DB version is bumped to `"20200703124523"`.
4. Restart the node and call `ckb rpc get_transaction --hash <any_committed_tx_hash>`.
5. Node panics: `failed to convert from slice: reason: TotalSizeNotMatch(52, 68); data: 0x...`
6. Confirm by directly dumping `COLUMN_TRANSACTION_INFO`: entries still have 68-byte values (old table format), plus spurious entries with uncle hashes as keys containing 212-byte slices of uncle data.
