### Title
Unbounded `pending_delivered` HashMap Growth via Spoofed `from` PeerIds in HolePunching Protocol — (`network/src/protocols/hole_punching/mod.rs`, `component/connection_request.rs`)

---

### Summary

The `HolePunching` protocol handler stores one entry per unique `from` PeerId in `pending_delivered: HashMap<PeerId, PendingDeliveredInfo>` with no size cap. The map is pruned only once every 5 minutes in `notify()`. An attacker with one or more connected sessions can flood the map with distinct spoofed `from` PeerIds at up to 30 req/sec per session, exhausting process heap memory before the pruning tick fires.

---

### Finding Description

`HolePunching` declares two unbounded collections:

```rust
pending_delivered: HashMap<PeerId, PendingDeliveredInfo>,
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,  // HashMapStateStore internally
``` [1](#0-0) 

The only pruning of `pending_delivered` happens in `notify()`, which fires on a 5-minute timer:

```rust
const CHECK_INTERVAL: Duration = Duration::from_secs(5 * 60);
...
self.pending_delivered.retain(|_, (_, t)| (now - *t) < TIMEOUT);
``` [2](#0-1) [3](#0-2) 

When a `ConnectionRequest` arrives with `content.to == self_peer_id`, `respond_delivered` is called. It deduplicates only on repeated `from` PeerIds:

```rust
if let Some((_, t)) = self.protocol.pending_delivered.get(&from_peer_id) {
    if now - t < HOLE_PUNCHING_INTERVAL { return StatusCode::Ignore... }
}
``` [4](#0-3) 

With a **distinct** `from` PeerId on every message, this guard never fires. The insertion at the end of `respond_delivered` is unconditional for new keys:

```rust
self.protocol.pending_delivered.insert(from_peer_id, (remote_listens, now));
``` [5](#0-4) 

The `rate_limiter` is keyed by `(session_id, msg_item_id)` — it caps one session at 30 req/sec but does not bound the total across sessions: [6](#0-5) [7](#0-6) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`. With unique `from` PeerIds, every message gets its own fresh bucket — it does not limit total insertion rate into `pending_delivered`: [8](#0-7) 

The `forward_rate_limiter`'s internal `HashMapStateStore<(PeerId, PeerId, u32)>` also grows unboundedly with each unique `(from, to)` pair, compounding memory pressure. `retain_recent()` is only called on session disconnect, not during active flooding: [9](#0-8) 

---

### Impact Explanation

**Memory exhaustion math (conservative):**

| Parameter | Value |
|---|---|
| Rate per session | 30 req/sec |
| Window before pruning | 300 sec (5 min) |
| Entries per session | 9,000 |
| Max inbound sessions (typical) | ~125 |
| Total entries | ~1,125,000 |
| Bytes per entry (PeerId + 24 TCP addrs) | ~1,239 bytes |
| **Total heap growth** | **~1.4 GB** |

`ADDRS_COUNT_LIMIT = 24` addresses are accepted per message; the attacker controls `listen_addrs` and can fill all 24 slots with valid TCP addresses, maximizing per-entry size: [10](#0-9) [11](#0-10) 

Result: process OOM → node crash → local RPC becomes unresponsive.

---

### Likelihood Explanation

- HolePunching is an opt-in protocol but is included in the production binary and enabled via config.
- The attacker only needs one or more inbound P2P connections — no authentication, no PoW, no privileged role.
- The victim's peer ID is publicly advertised via the P2P identify protocol.
- The attack is fully automatable with a simple loop sending `ConnectionRequest` with random `from` bytes.
- Even a single session (9,000 entries × 1,239 bytes ≈ 11 MB per 5-min window) causes steady unbounded growth if the attacker reconnects repeatedly.

---

### Recommendation

1. **Cap `pending_delivered`**: enforce a maximum size (e.g., 1,024 entries) and evict the oldest entry on overflow before inserting.
2. **Cap `inflight_requests`** similarly — it has the same structural issue.
3. **Periodic `retain_recent()` on rate limiters**: call `forward_rate_limiter.retain_recent()` inside `notify()`, not only on disconnect.
4. **Validate `from` is a currently-connected peer**: reject `ConnectionRequest` messages whose `from` PeerId does not correspond to a known connected session, eliminating spoofed-identity flooding entirely.

---

### Proof of Concept

```rust
// Drive respond_delivered with 10^6 distinct from PeerIds and assert HashMap stays bounded.
// In a unit test harness:
for i in 0..1_000_000u64 {
    let fake_from = PeerId::random();  // unique each iteration
    let msg = build_connection_request(from=fake_from, to=victim_peer_id, listen_addrs=[valid_tcp_addr]);
    hole_punching.received(ctx, msg).await;
}
assert!(hole_punching.pending_delivered.len() <= SOME_BOUND);  // FAILS: len == 1_000_000
```

The `rate_limiter` throttles this to 30/sec per session in production, but the HashMap size is unbounded — it grows to `min(rate × elapsed, unique_froms_sent)` with no ceiling.
