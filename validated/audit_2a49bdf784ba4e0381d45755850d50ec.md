### Title
Blocklist Bypass via Fresh Signature-Set Keypair — (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_vaa.rs`)

---

### Summary

The `INVALID_SIGNATURE_SET_KEYS` blocklist in `post_vaa` gates on the **keypair pubkey** of the `SignatureSet` account, not on the **VAA message hash**. Because `SignatureSet` accounts are created from arbitrary relayer-generated keypairs, any attacker can generate a fresh keypair, re-run `verify_signatures` with the original guardian secp256k1 signatures (which are public), and call `post_vaa` with the new keypair. The new keypair is not in the 16-entry blocklist, so the check passes and a `PostedVaaV1` is created for the blocked VAA.

---

### Finding Description

`post_vaa.rs` defines a 16-entry string blocklist and checks it as follows:

```rust
const INVALID_SIGNATURE_SET_KEYS: [&str; 16] = [
    "18eK1799CaNMGCUnnCt1Kq2uwKkax6T2WmtrDsZuVFQ",
    // … 15 more …
];

require!(
    !INVALID_SIGNATURE_SET_KEYS.contains(&signature_set.key().to_string().as_str()),
    CoreBridgeError::InvalidSignatureSet
);
``` [1](#0-0) [2](#0-1) 

The `SignatureSet` account is a **relayer-generated keypair** — there is no PDA derivation or program-controlled address for it. The `verify_signatures` instruction accepts any new keypair via `init_if_needed`: [3](#0-2) 

When a fresh keypair is used, `verify_signatures` initializes the new `SignatureSet` with whatever `message_hash` the Sig Verify native program produces from the provided guardian signatures: [4](#0-3) 

The `posted_vaa` PDA is derived solely from `signature_set.message_hash`, not from the keypair: [5](#0-4) 

So the blocklist protects 16 specific keypairs, but the PDA address space it was meant to protect is keyed on the message hash. Any fresh keypair carrying the same message hash produces the same PDA and is not blocked.

---

### Impact Explanation

A `PostedVaaV1` account is created for a VAA the protocol intended to permanently reject. Downstream consumers (governance executor, price-feed receiver) that gate on the existence of a `PostedVaaV1` PDA will accept the previously-blocked VAA, potentially executing a malicious governance action or accepting a manipulated price update.

---

### Likelihood Explanation

The attack requires only:
1. The original VAA bytes (publicly available from any Wormhole relayer/API).
2. A fresh Solana keypair (trivially generated).
3. Two permissionless on-chain calls: `verify_signatures` then `post_vaa`.

No privileged access, leaked keys, or guardian collusion is needed. The guardian secp256k1 signatures are embedded in the original VAA bytes and can be replayed verbatim into a new `verify_signatures` call.

---

### Recommendation

Replace the keypair-pubkey blocklist with a **message-hash blocklist**. Check `signature_set.message_hash` against a set of blocked hashes:

```rust
const INVALID_MESSAGE_HASHES: [MessageHash; N] = [ /* blocked hashes */ ];

require!(
    !INVALID_MESSAGE_HASHES.contains(&signature_set.message_hash),
    CoreBridgeError::InvalidSignatureSet
);
```

This makes the block unconditional regardless of which keypair is used to carry the signatures.

---

### Proof of Concept

1. Obtain the raw bytes of a blocked VAA (e.g., one whose original signature-set keypair is `18eK1799CaNMGCUnnCt1Kq2uwKkax6T2WmtrDsZuVFQ`).
2. Generate a fresh Solana keypair `new_sig_set`.
3. Build a transaction:
   - Instruction 0: `secp256k1_program` — verify the original guardian signatures against `keccak(body)`.
   - Instruction 1: `verify_signatures` — pass `new_sig_set` as the signature-set account; the program initializes it with `message_hash = keccak(body)`.
4. Build a second transaction:
   - `post_vaa` — pass `new_sig_set`; the blocklist check passes (new pubkey not in list); quorum check passes (guardian sigs verified); message-hash check passes; `PostedVaaV1` PDA `["PostedVAA", message_hash]` is created.
5. Assert the `PostedVaaV1` account now exists on-chain for the blocked VAA's message hash. [6](#0-5) [7](#0-6)

    pub fn constraints(ctx: &Context<Self>, args: &PostVaaArgs) -> Result<()> {
        let guardian_set = ctx.accounts.guardian_set.inner();

        // Check that the guardian set is still active.
        let timestamp = Clock::get().map(Into::into)?;
        require!(
            guardian_set.is_active(&timestamp),
            CoreBridgeError::GuardianSetExpired
        );

        let signature_set = &ctx.accounts.signature_set;
        require!(
            !INVALID_SIGNATURE_SET_KEYS.contains(&signature_set.key().to_string().as_str()),
            CoreBridgeError::InvalidSignatureSet
        );

        // Number of verified signatures in the signature set account must be at least quorum with
        // the guardian set.
        require!(
            signature_set.num_verified() >= utils::quorum(guardian_set.keys.len()),
            CoreBridgeError::NoQuorum
        );

        // Recompute the message hash and compare it to the one in the signature set account.
        let recomputed = solana_program::keccak::hashv(&[
            &args.timestamp.to_be_bytes(),
            &args.nonce.to_be_bytes(),
            &args.emitter_chain.to_be_bytes(),
            &args.emitter_address,
            &args.sequence.to_be_bytes(),
            &[args.consistency_level],
            &args.payload,
        ]);
        require_eq!(
            MessageHash::from(recomputed),
            signature_set.message_hash,
            CoreBridgeError::InvalidMessageHash
        );

        // Done.
        Ok(())
    }

#[access_control(VerifySignatures::constraints(&ctx))]
fn verify_signatures(ctx: Context<VerifySignatures>, args: VerifySignaturesArgs) -> Result<()> {
    // It would have been nice to be able to perform this check in `access_control`, but there
    // is no data from the instruction sysvar loaded by that point. We have to load it and perform
    // the safety checks in this instruction handler.
    let instructions_sysvar = &ctx.accounts.instructions;

    // We grab the index of the instruction before this instruction, which should be the sig verify
    // program.
    let sig_verify_index = u16::checked_sub(
        sysvar::instructions::load_current_index_checked(instructions_sysvar)?,
        1,
    )
    .ok_or(CoreBridgeError::InstructionAtWrongIndex)?;

    // And here we verify that the previous instruction is actually the Sig Verify native program.
    let SigVerifyParameters {
        eth_pubkeys: signers,
        message,
    } = sysvar::instructions::load_instruction_at_checked(
        usize::from(sig_verify_index),
        instructions_sysvar,
    )
    .map_err(Into::into)
    .and_then(|ix| deserialize_secp256k1_ix(sig_verify_index, &ix))?;

    let VerifySignaturesArgs { signer_indices } = args;

    // Number of specified signers must equal the number of signatures verified in the Sig Verify
    // native program instruction.
    let guardian_indices: Vec<_> = signer_indices
        .iter()
        .enumerate()
        .filter_map(|(i, &value)| if value >= 0 { Some(i) } else { None })
        .collect();
    require_eq!(
        signers.len(),
        guardian_indices.len(),
        CoreBridgeError::SignerIndicesMismatch
    );

    // We use this message hash later on.
    let message_hash = MessageHash::from(message);
    let signature_set = &mut ctx.accounts.signature_set;
    let guardian_set = ctx.accounts.guardian_set.inner();
    let guardians = &guardian_set.keys;

    // If the signature set account has not been initialized yet, establish the expected account
    // data (guardian set index used, hash and which indices have been verified).
    if signature_set.is_initialized() {
        // Otherwise, verify that the guardian set index is what we expect from
        // the last time we wrote to the signature set account.
        require_eq!(
            guardian_set.index,
            signature_set.guardian_set_index,
            CoreBridgeError::GuardianSetMismatch
        );

        // And verify that the message hash is the same as the one already encoded in the signature
        // set.
        require_eq!(
            message_hash,
            signature_set.message_hash,
            CoreBridgeError::MessageMismatch
        );
    } else {
        // We are assuming that the signature set has not been "initialized" if there is no
        // indication of verified signatures (via `sig_verify_successes`) written to this account
        // yet. If we reach this condition, we set the message hash and guardian set index because
        // we are assuming that the account is created with this instruction invocation.
        signature_set.set_inner(SignatureSet {
            sig_verify_successes: vec![false; guardians.len()],
            message_hash,
            guardian_set_index: guardian_set.index,
        });
    }

    // Attempt to write `true` to represent verified guardian eth pubkey.
    for (i, &signer_index) in guardian_indices.iter().enumerate() {
        require!(
            signers.get(i) == guardians.get(signer_index),
            CoreBridgeError::InvalidGuardianKeyRecovery
        );

        // Overwritten content should be zeros except double signs by the
        // signer or harmless replays.
        signature_set.sig_verify_successes[signer_index] = true;
    }

    // Done.
    Ok(())
}
