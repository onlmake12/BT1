Audit Report

## Title
Unbounded Iteration in `get_raw_tx_pool` RPC Causes Node DoS via Memory Exhaustion and Lock Contention - (File: `rpc/src/module/pool.rs`)

## Summary
`get_raw_tx_pool` unconditionally collects the entire tx-pool into unbounded heap structures with no pagination or size guard. With the default 180 MB pool cap, a single `verbose=true` call can trigger gigabytes of JSON serialization overhead while holding the tx-pool read lock and blocking the service event loop, stalling transaction admission and block assembly. Any caller with RPC access can repeat this at negligible cost.

## Finding Description
`get_raw_tx_pool` at [1](#0-0)  delegates unconditionally to `get_all_entry_info()` or `get_all_ids()` with no size guard, limit, or pagination.

In `tx-pool/src/pool.rs`, `get_ids()` and `get_all_entry_info()` both call `score_sorted_iter_by_statuses` and `sorted_proposed_iter` and `.collect()` the full result into unbounded structures: [2](#0-1) 

The result types `TxPoolIds` and `TxPoolEntryInfo` are plain unbounded `Vec<Byte32>` and `HashMap<Byte32, TxEntryInfo>` with no capacity cap: [3](#0-2) 

In `tx-pool/src/service.rs`, the async read lock is acquired before the call and held for the entire duration of the scan and channel send: [4](#0-3) 

The service processes messages sequentially in a single async task loop. While `GetAllEntryInfo` is executing, the loop does not yield, so all queued messages (`SubmitLocalTx`, `BlockTemplate`, etc.) are stalled until the scan completes. The pool is bounded by total serialized byte size, not entry count: [5](#0-4) 

The default `max_tx_pool_size = 180_000_000` (180 MB) is confirmed both in the config template and in the legacy defaults: [6](#0-5) 

No rate limit, pagination, or per-call cap exists anywhere in this call chain. The indexer already documents the identical risk and applies a `request_limit = 400` guard with the explicit note that "serde json serialization consumes too much memory (10x), it may cause the physical machine to become unresponsive": [7](#0-6) 

That guard was never applied to `get_raw_tx_pool`.

## Impact Explanation
This matches **High: Vulnerabilities which could easily crash a CKB node**. With 180 MB of pool data and a documented 10× JSON serialization overhead (acknowledged in `ckb.toml` for the indexer), a single `verbose=true` call can allocate ~1.8 GB of heap. The OS OOM killer may terminate the node process outright. Even before OOM, the service event loop stall blocks `get_block_template` and `send_transaction`, halting mining and relay. Multiple concurrent calls compound both effects. Impact is scoped to a single node, not the whole network, placing this in the High tier. [8](#0-7) 

## Likelihood Explanation
The RPC binds to `127.0.0.1:8114` by default, limiting direct remote exploitation on default-configured nodes: [9](#0-8) 

However, mining pools, exchanges, and public RPC providers routinely expose this port. The pool-filling precondition is economically trivial: `min_fee_rate = 1_000` shannons/KB means filling 180 MB costs approximately 1.8 CKB. A busy mainnet node may reach a large pool state organically, requiring no attacker-controlled transactions. No special privileges or protocol knowledge beyond standard JSON-RPC are required. The attack is stateless and repeatable. [10](#0-9) 

## Recommendation
Add pagination parameters (`limit` and `after` cursor) to `get_raw_tx_pool`. Enforce a hard per-call cap on entries returned, matching the `request_limit = 400` pattern already documented for the indexer in `ckb.toml`. The existing indexer RPC design in `rpc/src/module/indexer.rs` provides a ready template. Additionally, consider releasing the read lock between pages or streaming results rather than collecting the entire pool into memory before responding. [11](#0-10) 

## Proof of Concept
1. Configure a CKB node with RPC accessible (localhost or exposed).
2. Submit minimum-size transactions until the pool approaches `max_tx_pool_size` (180 MB). At `min_fee_rate = 1,000` shannons/KB this costs ~1.8 CKB total.
3. Issue concurrent JSON-RPC calls: `{"method":"get_raw_tx_pool","params":[true],"id":1,"jsonrpc":"2.0"}`.
4. Observe: node memory climbs rapidly as `TxPoolEntryInfo` `HashMap`s are allocated; the tx-pool read lock is held for the duration of each call; concurrent `send_transaction` and `get_block_template` calls stall or time out; the node process may be killed by the OOM killer.
5. Reproducible as a unit test by constructing a `TxPool` with a large number of mock entries and calling `get_all_entry_info()` while measuring allocation and lock-hold duration, analogous to the existing chunk tests in `tx-pool/src/component/tests/`. [12](#0-11)

    fn get_raw_tx_pool(&self, verbose: Option<bool>) -> Result<RawTxPool> {
        let tx_pool = self.shared.tx_pool_controller();

        let raw = if verbose.unwrap_or(false) {
            let info = tx_pool
                .get_all_entry_info()
                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Verbose(info.into())
        } else {
            let ids = tx_pool
                .get_all_ids()
                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Ids(ids.into())
        };
        Ok(raw)
    }

    pub(crate) fn get_ids(&self) -> TxPoolIds {
        let pending = self
            .pool_map
            .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
            .map(|entry| entry.transaction().hash())
            .collect();

        let proposed = self
            .pool_map
            .sorted_proposed_iter()
            .map(|entry| entry.transaction().hash())
            .collect();

        TxPoolIds { pending, proposed }
    }

    pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
        let pending = self
            .pool_map
            .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let proposed = self
            .pool_map
            .sorted_proposed_iter()
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let conflicted = self
            .conflicts_cache
            .iter()
            .map(|(_id, tx)| tx.hash())
            .collect();
        TxPoolEntryInfo {
            pending,
            proposed,
            conflicted,
        }
    }
