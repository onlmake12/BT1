### Title
Signature Verification Runs Before Data-Source Check in `post_update_atomic`, Enabling Near-Zero-Cost Block Stuffing — (`target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

`post_update_atomic` verifies all guardian signatures (each costing ~38k CU) before checking whether the VAA's emitter is in `config.valid_data_sources`. An unprivileged attacker who possesses any real Wormhole VAA signed by the current guardian set can craft a call that burns ~722k CU per transaction while paying only the ~5,000-lamport Solana base fee, because CU is consumed even when the transaction reverts.

---

### Finding Description

The execution order inside `post_update_atomic` is:

1. **Signature verification loop** (lines 163–180) — iterates over every signature in the VAA and calls `verify_guardian_signature`, which invokes `secp256k1_recover` (~25k CU) and `keccak::hash` (~13k CU) per signature.
2. `post_price_update_from_vaa` is called (line 194), which internally calls:
   - `pay_single_update_fee` (line 459) — lamport transfer, **rolled back on revert**
   - `verify_vaa_data_source` (line 460) — returns `InvalidDataSource` if the emitter is not whitelisted [1](#0-0) [2](#0-1) 

The code's own comment documents the per-signature cost: [3](#0-2) 

With 19 signatures: 19 × (25k + 13k) CU = **722k CU consumed before the data-source gate fires**.

On Solana, a failed transaction rolls back all lamport transfers but does **not** refund consumed CU — those CU are permanently deducted from the block's 48M CU budget.

---

### Impact Explanation

An attacker submitting ~66 such transactions per block (48M / 722k ≈ 66) saturates the entire block CU budget at a cost of roughly 66 × 5,000 lamports ≈ 330,000 lamports (~$0.05) per block. Legitimate Pyth price-update transactions are crowded out, delaying oracle updates and potentially causing cascading failures in downstream DeFi protocols that depend on fresh Pyth prices.

---

### Likelihood Explanation

- **Attacker prerequisites**: possession of any real Wormhole VAA signed by the current guardian set with an emitter not in `valid_data_sources`. Wormhole VAAs are public and observable on-chain from any Wormhole-integrated protocol (not just Pyth). The attacker does not need to forge signatures.
- **No privileged access required**: the call path is fully open to any Solana account.
- **Repeatable indefinitely**: the same VAA bytes can be resubmitted in every block.

---

### Recommendation

Move `verify_vaa_data_source` to **before** the signature verification loop in `post_update_atomic`. The emitter address and chain are available directly from `vaa.body()` without any cryptographic work:

```rust
// Check data source FIRST — cheap, no CU cost
let vaa_components = VaaComponents {
    verification_level,
    emitter_address: vaa.body().emitter_address(),
    emitter_chain: vaa.body().emitter_chain(),
};
verify_vaa_data_source(config, &vaa_components)?;  // <-- moved up

// Only then run the expensive signature loop
for sig in vaa.signatures() { ... }
```

This makes the cheap O(1) whitelist check the first gate, so invalid-emitter transactions are rejected before any `secp256k1_recover` call is made. [4](#0-3)

fn verify_vaa_data_source(
    config: &Account<'_, Config>,
    vaa_components: &VaaComponents,
) -> Result<()> {
    let valid_data_source = config.valid_data_sources.iter().any(|x| {
        *x == DataSource {
            chain: vaa_components.emitter_chain,
            emitter: Pubkey::from(vaa_components.emitter_address),
        }
    });
    if !valid_data_source {
        return err!(ReceiverError::InvalidDataSource);
    }
    Ok(())
}
