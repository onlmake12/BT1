Audit Report

## Title
Ambiguous Variable-Length `args` Encoding in Indexer RocksDB Keys Enables Cross-Script Result Pollution — (`util/indexer/src/indexer.rs`)

## Summary
`extract_raw_data` concatenates the variable-length `args` field of a `Script` directly against fixed-length positional fields (`block_number`, `tx_index`, `io_index`) with no length delimiter. Because `build_query_options` builds the scan prefix using the same ambiguous encoding, and the default scan path in `get_cells`, `get_transactions`, and `get_cells_capacity` applies only a `starts_with` check without a key-length guard, a stored key for script A (shorter `args`) at a specific block/tx/output position is byte-identical to the search prefix for a crafted script B (longer `args`), causing the attacker's cell to appear in indexer results for the target script.

## Finding Description
`extract_raw_data` at `util/indexer/src/indexer.rs:171-178` produces:
```
code_hash(32) || hash_type(1) || args(N bytes, no length prefix)
```
`append_key` at lines 157-168 appends:
```
extract_raw_data(script) || block_number(8 BE) || tx_index(4 BE) || io_index(4 BE)
```
So the full stored RocksDB key for a cell is:
```
[prefix_byte] || code_hash(32) || hash_type(1) || args_A(N) || block_number(8) || tx_index(4) || io_index(4)
```
`build_query_options` at `util/indexer/src/service.rs:878` calls `extract_raw_data` with the same no-length-prefix encoding to build the scan prefix:
```
[prefix_byte] || code_hash(32) || hash_type(1) || args_B
```
Because there is no length delimiter, if `args_B = args_A || block_number_bytes || tx_index_bytes[0..k]` for the appropriate overlap length, the stored key for script A `starts_with` the search prefix for script B.

The scan in `get_cells` at lines 249-258 uses `key.starts_with(&prefix)` unconditionally. The key-length guard (`key.len() != prefix.len() + 16`) is only applied when `script_search_exact` is true, i.e., when the caller explicitly passes `IndexerSearchMode::Exact`. When `script_search_mode` is `None` (the default), `script_search_exact` is false and no length check is applied. The identical pattern repeats in `get_transactions` at lines 469-476 (guard only when `script_search_exact`, key length check `prefix.len() + 17`).

## Impact Explanation
Applications (wallets, dApps, bridges) that rely on `get_cells`, `get_transactions`, or `get_cells_capacity` in the default prefix scan mode receive incorrect data: phantom UTXOs, inflated/deflated capacity, or spurious transaction associations. This is a concrete suboptimal implementation of the CKB state storage mechanism (the indexer's RocksDB key space), matching **Medium (2001–10000 points) — Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation
An unprivileged attacker controls `args` entirely at transaction construction time and controls `output_index` by choosing how many outputs to include. `tx_index` within a block is partially controllable by submitting multiple transactions. `block_number` is predictable. A retroactive variant requires no position control: after a cell is confirmed at `(N, T, I)`, the attacker computes exactly which target `args_B` their cell collides with and advertises that script to victims. No privileged access, no majority hashpower, and no social engineering is required. The attack is repeatable and cheap.

## Recommendation
Add a fixed-width length prefix for `args` inside `extract_raw_data` so the boundary between `args` and the positional fields is unambiguous:
```rust
pub fn extract_raw_data(script: &Script) -> Vec<u8> {
    let args = script.args().raw_data();
    let args_len = (args.len() as u32).to_le_bytes();
    [
        script.code_hash().as_slice(),
        script.hash_type().as_slice(),
        &args_len,
        &args,
    ]
    .concat()
}
```
This requires a full re-index. As a weaker short-term mitigation, the key-length guard already present in exact mode should be applied unconditionally to all scan modes, but this does not fix the underlying key-space collision.

## Proof of Concept
1. Run a CKB devnet node with the built-in indexer enabled.
2. Deploy a cell with lock script `code_hash = X`, `hash_type = Y`, `args = 0xAA` (1 byte). Note the confirmed `block_number = N`, `tx_index = T`, `output_index = I`.
3. Compute the 16-byte suffix `S = N.to_be_bytes(8) || T.to_be_bytes(4) || I.to_be_bytes(4)`.
4. Call `get_cells` RPC with:
   ```json
   {
     "script": { "code_hash": "X", "hash_type": "Y", "args": "0xAA<S_hex>" },
     "script_type": "lock"
   }
   ```
   (no `script_search_mode` field — uses default prefix scan)
5. Observe that the attacker's cell (which has `args = 0xAA`, not the 17-byte args in the query) is returned, confirming cross-script result pollution. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
