### Title
No-Cancellation Mechanism for Expired `useBlockhash` Entropy Requests Causes Permanent Fee Loss and Irrecoverable Request State — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Entropy contract's `request()` function accepts a `useBlockhash` flag. When set to `true`, the request incorporates the block hash of the originating block into the random number. Because `blockhash()` only returns a non-zero value for the 256 most recent blocks (~53 minutes on Ethereum), any such request not revealed by the provider within that window becomes permanently irrecoverable. There is no `cancelRequest` or refund function anywhere in the contract, so the user's fee is permanently locked and the request slot persists in the `requestsOverflow` mapping indefinitely.

### Finding Description

In `Entropy.sol`, the `request()` function stores `req.blockNumber = block.number` and `req.useBlockhash = true` when the caller passes `useBlockHash = true`. [1](#0-0) 

During reveal, `revealHelper` calls `blockhash(req.blockNumber)`. If the block is older than 256 blocks, `blockhash()` returns `bytes32(0)`, and the function reverts with `BlockhashUnavailable`: [2](#0-1) 

Both `reveal()` and `revealWithCallback()` call `revealHelper`, so neither path can complete after the 256-block window. The `IEntropy` interface exposes no `cancelRequest`, `expireRequest`, or refund function: [3](#0-2) 

When a new request with the same `shortKey` arrives, `allocRequest` evicts the stuck request into the unbounded `requestsOverflow` mapping, where it remains permanently: [4](#0-3) 

The `requestsOverflow` mapping is declared with no size bound and no deletion path for expired entries: [5](#0-4) 

The `clearRequest` helper is only called from `reveal` and `revealWithCallback` upon successful completion — never on expiry: [6](#0-5) 

### Impact Explanation

Any user who calls `request(provider, commitment, true)` and whose provider does not reveal within 256 blocks loses their fee permanently. The request occupies a slot in the ring buffer or the overflow mapping forever. The provider retains the accrued fee with no obligation to refund. Over time, a provider that goes offline (or a malicious provider) causes all pending `useBlockhash=true` requests to become permanently irrecoverable, with no on-chain remedy for affected users.

### Likelihood Explanation

The `request()` function with `useBlockhash=true` is a publicly callable, permissionless entry point. The Fortuna off-chain service is the primary revealer; any downtime exceeding ~53 minutes on Ethereum (256 blocks) triggers the condition. A malicious provider can register, collect fees, and deliberately withhold reveals to permanently lock all users' fees. The legacy `request()` path remains live in the deployed contract.

### Recommendation

1. Add a `cancelRequest(address provider, uint64 sequenceNumber)` function that allows the original requester to cancel and receive a refund after the 256-block window has passed for `useBlockhash=true` requests (or after a configurable timeout for all requests).
2. Alternatively, deprecate the `useBlockhash=true` path entirely, since `requestV2` / `requestWithCallback` already hardcode `useBlockhash = false`. [7](#0-6) 

### Proof of Concept

1. User calls `request(provider, keccak256(secret), true)` at block N, paying the required fee.
2. Provider does not call `reveal()` for 256+ blocks.
3. User attempts `reveal(provider, seqNum, secret, providerProof)` — reverts with `BlockhashUnavailable`.
4. User attempts `revealWithCallback(...)` — same revert path through `revealHelper`.
5. No other function exists to cancel the request or recover the fee.
6. A subsequent request with the same `shortKey` pushes the stuck request into `requestsOverflow`, where it remains permanently with no deletion path.

        EntropyStructsV2.Request storage req = _state.requests[shortKey];
        if (req.provider == provider && req.sequenceNumber == sequenceNumber) {

        } else {
