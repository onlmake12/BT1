Audit Report

## Title
Per-Peer `unknown_tx_hashes` Limit Bypass via Sub-Threshold Batch Submission — (`sync/src/types/mod.rs`)

## Summary
`add_ask_for_txs` unconditionally inserts up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767) hashes before evaluating any per-peer limit. The per-peer ban scan is gated behind a global-length check that a single peer sending 32,766 hashes never satisfies. After the 30-second `RETRY_ASK_TX_TIMEOUT_INCREASE` window, `pop_ask_for_txs` permanently drops all entries for the peer, resetting its effective count to zero. The attacker can repeat this cycle indefinitely without ever being banned.

## Finding Description
**Phase 1 — unconditional insertion** (`sync/src/types/mod.rs`, lines 1486–1504): `add_ask_for_txs` iterates the incoming hashes with `.take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)` and inserts every entry into `unknown_tx_hashes` with no per-peer guard before insertion. [1](#0-0) 

**Phase 2 — conditional per-peer check** (lines 1506–1529): only after insertion does the code check `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) or `>= peers_count × 32,767`. With one connected peer and a batch of 32,766 hashes the queue length is 32,766, satisfying neither condition (`32,766 < 50,000` and `32,766 < 1 × 32,767`), so the per-peer counter scan and ban are never reached and `Status::ok()` is returned. [2](#0-1) 

**Constants confirm the off-by-one boundary**: `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = MAX_RELAY_TXS_NUM_PER_BATCH = 32,767` and `MAX_UNKNOWN_TX_HASHES_SIZE = 50,000`. [3](#0-2) 

**Entry expiry** (`pop_ask_for_txs`, lines 1466–1479): when `next_request_peer()` is called on an entry whose `requested` flag is `true` and `peers.len() == 1`, it returns `None`. The entry is then not re-pushed to the queue and is permanently dropped. After the 30-second timeout all 32,766 entries are silently discarded, resetting the attacker's effective per-peer count to zero. [4](#0-3) [5](#0-4) 

**Existing test gap**: the integration test sends exactly `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = 32,767 hashes, which triggers the global check (`32,767 >= 1 × 32,767`) and correctly bans the peer. Sending 32,766 hashes bypasses this test's coverage entirely. [6](#0-5) 

## Impact Explanation
A single unprivileged peer causes the victim node to continuously emit `GetRelayTransactions` messages for ~32,766 non-existent transactions every ~31 seconds, wasting outbound bandwidth and CPU. The `unknown_tx_hashes` queue is kept perpetually occupied, crowding out legitimate transaction-hash announcements from honest peers and degrading transaction propagation. The attacker is never banned. This matches **Low (501–2000 points) — important performance degradation for a CKB node**.

## Likelihood Explanation
Any peer that can establish a relay connection can execute this attack. No special keys, hashpower, or Sybil capability is required. The rate limiter is not a barrier: the attacker needs only one `RelayTransactionHashes` message per 31-second cycle. The attack is fully automated, repeatable indefinitely, and leaves no ban record on the attacker.

## Recommendation
Replace the post-insertion global-length gate with a **monotonic per-peer counter** stored in `PeerState`. Increment it by the number of newly inserted (`Vacant`) entries during Phase 1 of `add_ask_for_txs` and never decrement it (or decay it only over a long window). Reject or ban the peer as soon as its cumulative counter reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, regardless of whether earlier entries have already been consumed from the queue. This ensures the accounting variable is independent of the mutable queue state.

## Proof of Concept
Modify the existing `TooManyUnknownTransactions` integration test to send `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER - 1` (32,766) hashes instead of 32,767, then assert that the peer is **not** banned after 60 seconds. The assertion will pass, confirming the bypass. Additionally, assert that after a second identical batch sent after 31 seconds the peer is still not banned, confirming the cyclic reset via entry expiry.
