Audit Report

## Title
Missing `BLOCK_INVALID` Early-Return Causes Redundant Header Re-Verification — (`sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::accept()` lacks an early-return for headers whose status is already `BLOCK_INVALID`. Because `BLOCK_INVALID = 1 << 12` does not set the `HEADER_VALID = 1` bit, the guard at line 304 is false for `BLOCK_INVALID` headers, causing the code to fall through and redundantly re-execute `prev_block_check`, `non_contextual_check` (including full Eaglesong PoW verification via `HeaderVerifier::verify()`), and `version_check`. The developers acknowledged this with the `FIXME` at lines 301–302. The peer is banned after the first offending message via `StatusCode::HeadersIsInvalid` → `should_ban()`, so the impact is bounded redundant CPU per peer connection, not a sustained DoS.

## Finding Description
In `HeaderAcceptor::accept()` (lines 295–358 of `sync/src/synchronizer/headers_process.rs`), the status check at line 303–304 only fast-paths on `HEADER_VALID`. Since `BLOCK_INVALID = 1 << 12 = 4096` and `HEADER_VALID = 1` share no bits, `status.contains(BlockStatus::HEADER_VALID)` evaluates to `false` for any `BLOCK_INVALID` header, and execution falls through to lines 324–354. There, `prev_block_check` (line 324), `non_contextual_check` (line 334, which calls `self.verifier.verify(self.header)` at line 256 — full PoW verification), and `version_check` (line 346) are all re-executed for a header already known to be invalid. The checks fail again, `result.state` is set to `ValidationState::Invalid`, and `execute()` returns `StatusCode::HeadersIsInvalid` (415). `should_ban()` in `status.rs` lines 165–179 returns `Some(BAD_MESSAGE_BAN_TIME)` for all 4xx codes except `GetHeadersMissCommonAncestors`, so the peer is banned after this single message. The `FIXME` at lines 301–302 is the developers' own acknowledgment of the missing path.

The claimed "peer is never banned" and "no Sybil capability needed for sustained DoS" are incorrect: the ban mechanism is functional and limits each attacker connection to one offending message before a ban. Sustained attack requires reconnection after each ban, which requires Sybil capability. The "state confusion after restart" scenario is also not caused by this bug, as `block_status_map` is cleared on restart, returning `UNKNOWN` for all entries.

## Impact Explanation
The concrete impact is redundant CPU computation — re-running `HeaderVerifier::verify()` including Eaglesong PoW verification for headers already cached as `BLOCK_INVALID` — before the peer is banned. This does not crash a node, does not cause consensus deviation, and does not cause network-wide congestion. The impact fits **Low (501–2000 points): Any other important performance improvements for CKB**. The claimed High DoS impact is not proven because the ban mechanism is functional and limits each attacker connection to one offending message.

## Likelihood Explanation
Any P2P peer can send a `SendHeaders` message containing a known-`BLOCK_INVALID` header. The peer is banned after the first such message via `StatusCode::HeadersIsInvalid` → `should_ban()` → `BAD_MESSAGE_BAN_TIME`. Sustained attack requires reconnection after each ban, requiring either many IPs (Sybil) or repeated reconnections. The redundant PoW computation per message is real but bounded and not catastrophic.

## Recommendation
Add an explicit early-return for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept()`, resolving the `FIXME` at line 301 of `sync/src/synchronizer/headers_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent));
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early-return logic
}
```
