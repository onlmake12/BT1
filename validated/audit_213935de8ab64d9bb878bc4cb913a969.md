### Title
Missing Staleness Validation in `PythAggregatorV3.latestRoundData()` Exposes Integrators to Stale Price Exploitation - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, the official Pyth Solidity SDK adapter implementing Chainlink's `AggregatorV3Interface`, calls `pyth.getPriceUnsafe()` in every price-reading function — including `latestRoundData()`, `latestAnswer()`, `getRoundData()`, `latestTimestamp()`, and `decimals()` — without performing any staleness check. This is a direct analog to the external report's `_getLastPrice` vulnerability: a price-reading function that returns arbitrarily old data with no recency enforcement.

---

### Finding Description

`PythAggregatorV3` is the official Pyth-provided Chainlink compatibility adapter, distributed as part of `@pythnetwork/pyth-sdk-solidity` and actively recommended in Pyth's migration documentation for protocols moving from Chainlink to Pyth. Every price-reading function in the contract calls `getPriceUnsafe`:

```solidity
// latestAnswer() — line 53-56
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// latestRoundData() — line 99-119
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

`getPriceUnsafe` is explicitly documented in `IPyth.sol` as returning "the most recent price update in this contract without any recency checks" and that "the returned price update may be arbitrarily far in the past." No staleness check (e.g., comparing `price.publishTime` against `block.timestamp`) is performed anywhere in the contract.

The `updatedAt` field returned by `latestRoundData()` is set to `price.publishTime`, which is the Pyth publish timestamp — but the contract never validates that this timestamp is recent before returning the price. Downstream Chainlink-compatible protocols that call `latestRoundData()` and do not independently check `updatedAt` will silently consume stale prices.

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a drop-in Chainlink replacement and calls `latestRoundData()` or `latestAnswer()` will receive stale price data during periods when the Pyth feed has not been recently updated (e.g., market closures, network outages, or simply when no keeper has pushed a fresh update). Chainlink-compatible protocols typically do not add their own staleness checks because they expect the oracle adapter to enforce recency — exactly the assumption `PythAggregatorV3` violates.

Concrete impact: stale prices used for collateral valuation, liquidation thresholds, or swap pricing in any DeFi protocol using this adapter. An attacker can time transactions to exploit the price discrepancy between the stale on-chain price and the true market price.

**Impact: Medium** — financial loss to users of protocols built on this adapter; the severity depends on how stale the price is and the protocol's use case.

---

### Likelihood Explanation

Pyth is a pull oracle; prices are only updated when a keeper submits fresh data. During market closures (equities, FX), network congestion, or keeper downtime, the cached price can become arbitrarily stale. The `PythAggregatorV3` is the officially recommended migration path for Chainlink integrators, making widespread deployment likely. Any unprivileged user can trigger the vulnerable path by simply interacting with a protocol that uses this adapter.

**Likelihood: Medium** — requires a period of price staleness, which is a normal operational condition for pull oracles.

---

### Recommendation

Replace `getPriceUnsafe` calls in all price-reading functions with `getPriceNoOlderThan` using a configurable staleness threshold, or revert if the `publishTime` is too old:

```solidity
uint256 public maxStaleness; // configurable, e.g. 60 seconds

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxStaleness);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

Alternatively, add an explicit staleness check after `getPriceUnsafe`:

```solidity
require(
    block.timestamp - price.publishTime <= maxStaleness,
    "Stale price"
);
```

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to any Pyth price feed.
2. Allow time to pass without calling `updateFeeds()` (simulating a stale cache).
3. Call `latestRoundData()` — it returns the old `price.price` and `updatedAt = price.publishTime` (a timestamp far in the past) with no revert.
4. A downstream lending protocol using this adapter will accept the stale price for collateral valuation, allowing an attacker to borrow against over-valued or under-valued collateral.
