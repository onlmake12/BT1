### Title
`total_tx_size` and `total_tx_cycles` Overwritten with Stale Pre-Eviction Values After Ancestor Eviction in `add_entry` - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the global pool accounting counters `total_tx_size` and `total_tx_cycles` are computed before a conditional eviction step (`check_and_record_ancestors`). When that eviction step removes transactions from the pool — correctly updating the counters in-place via `update_stat_for_remove_tx` — the final assignment overwrites those correctly updated values with stale pre-eviction totals. The result is permanently inflated `total_tx_size` and `total_tx_cycles` counters, which drive pool eviction policy and pool-full rejection decisions.

---

### Finding Description

`PoolMap::add_entry` follows this sequence:

```rust
// Step 1: compute new totals BEFORE eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Step 2: may call remove_entry_and_descendants → remove_entry
//         → update_stat_for_remove_tx, which mutates self.total_tx_size
//         and self.total_tx_cycles IN PLACE
evicts = self.check_and_record_ancestors(&mut entry)?;

// ... insert entry ...

// Step 3: OVERWRITE the correctly updated in-place values
//         with the stale pre-eviction snapshot
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` captures a snapshot of the new totals based on the state at call time: [2](#0-1) 

`check_and_record_ancestors` can evict existing pool entries when the new transaction's ancestor count exceeds `max_ancestors_count` but some ancestors are cell-dep parents that can be removed. Each eviction calls `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly subtracts the evicted entry's size and cycles from `self.total_tx_size` and `self.total_tx_cycles` in place: [3](#0-2) 

After `check_and_record_ancestors` returns, the final two assignments in `add_entry` blindly overwrite `self.total_tx_size` and `self.total_tx_cycles` with the stale snapshot computed before any evictions occurred. The evicted transactions' sizes and cycles are never subtracted from the final counters.

**Concrete example:**
- Pool state: `total_tx_size = 100`, `total_tx_cycles = 1000`
- New entry: `size = 50`, `cycles = 500`
- `updated_stat_for_add_tx` computes local vars: `total_tx_size = 150`, `total_tx_cycles = 1500`
- `check_and_record_ancestors` evicts a tx with `size = 30`, `cycles = 300`
  - `update_stat_for_remove_tx(30, 300)` sets `self.total_tx_size = 70`, `self.total_tx_cycles = 700`
- Final assignment: `self.total_tx_size = 150` (stale), `self.total_tx_cycles = 1500` (stale)
- **Correct values should be:** `total_tx_size = 120`, `total_tx_cycles = 1200`
- **Inflation:** `+30` size, `+300` cycles per eviction event

---

### Impact Explanation

`total_tx_size` is the primary gate for two critical pool decisions:

1. **Pool eviction (`limit_size`):** The loop `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` runs and evicts valid pending transactions even when the pool has actual free space. Each eviction call to `remove_entry_and_descendants` will correctly subtract from the counter, but the pool will have already discarded valid transactions unnecessarily. [4](#0-3) 

2. **False "pool full" rejection of new transactions:** `updated_stat_for_add_tx` uses the inflated `self.total_tx_size` as the base for overflow/capacity checks, causing legitimate new transactions to be rejected with `Reject::Full` even when the pool has capacity. [5](#0-4) 

3. **Incorrect RPC reporting:** `total_tx_size` and `total_tx_cycles` are directly exposed via `get_tx_pool_info`, misleading operators and downstream tooling. [6](#0-5) 

The inflation is permanent until the pool is cleared or enough subsequent evictions bring the counter back down. Each trigger event inflates the counter by the sum of evicted transactions' sizes.

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is triggered when:
1. A new transaction's ancestor count exceeds `max_ancestors_count` (default 25), AND
2. Some of those ancestors are "cell-ref parents" — existing pool transactions whose outputs are referenced as cell deps by the new transaction's ancestors.

An unprivileged transaction sender can craft this scenario by:
- Submitting a chain of transactions where intermediate transactions reference earlier outputs as cell deps
- Submitting a final transaction that has >25 ancestors including those cell-dep-referencing transactions

This is reachable via the standard `send_transaction` RPC endpoint with no privileged access required. The attacker only needs to pay transaction fees for the crafted chain.

---

### Recommendation

Compute the new totals **after** `check_and_record_ancestors` completes, so that any in-place modifications from evictions are already reflected:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Validate that adding this entry won't overflow, but don't capture stale totals yet
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Compute final totals AFTER evictions have updated self.total_tx_size/cycles in place
    self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

Alternatively, recompute via `recompute_total_stat()` after the eviction step, though that is O(n) in pool size.

### Proof of Concept

1. Fill the pool with a chain of 26 transactions `T1 → T2 → ... → T26` where `T2` references `T1`'s output as a cell dep (making `T1` a cell-ref parent of `T2`).
2. Submit a new transaction `T_new` that spends `T26`'s output and also references `T1`'s output as a cell dep.
3. `add_entry` is called for `T_new`. `ancestors_count = 26 > max_ancestors_count = 25`. `cell_ref_parents = {T1}`. Since `26 - 1 = 25 <= 25`, the eviction branch fires and `T1` (and its descendants) are removed via `remove_entry_and_descendants`.
4. `update_stat_for_remove_tx` correctly subtracts `T1`'s size/cycles from `self.total_tx_size`/`self.total_tx_cycles`.
5. `add_entry` then assigns `self.total_tx_size = total_tx_size` (the stale pre-eviction value), inflating the counter by `T1`'s size.
6. Call `get_tx_pool_info` via RPC and observe `total_tx_size` is larger than the sum of actual pool entries' sizes. Observe that `limit_size` subsequently evicts valid transactions unnecessarily. [7](#0-6) [8](#0-7)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

            ))
        })?;

    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
