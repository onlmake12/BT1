### Title
State Mutated Before Size Check in `add_ask_for_txs` Allows Unbounded Growth of `unknown_tx_hashes` — (`sync/src/types/mod.rs`)

### Summary

`SyncState::add_ask_for_txs` inserts all incoming tx hashes into the shared `unknown_tx_hashes` priority queue **before** checking whether the queue has exceeded its size limits. When the post-insertion check fires and returns `TooManyUnknownTransactions` or `Status::ignored()`, the newly inserted entries are **never removed**. Any unprivileged peer can exploit this to grow `unknown_tx_hashes` well beyond its intended cap, causing unbounded memory growth and inflated transaction-fetch traffic on the victim node.

### Finding Description

In `sync/src/types/mod.rs`, `add_ask_for_txs` follows this order:

1. **Insert** up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` hashes into `unknown_tx_hashes` (permanent state mutation).
2. **Then** check whether `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` or the per-peer cap is exceeded.
3. If the check fires, return an error — but the entries inserted in step 1 **remain in the map**. [1](#0-0) 

The comment on line 1506 even acknowledges the ordering: *"Check `unknown_tx_hashes`'s length **after** inserting the arrival `tx_hashes`"*. This is the root cause. [2](#0-1) 

The intended invariant is that `unknown_tx_hashes` stays below `MAX_UNKNOWN_TX_HASHES_SIZE` and that no single peer contributes more than `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` entries. Because insertion precedes the guard, both invariants are violated on every call that triggers the overflow path: the map is already over the limit by the time the check runs, and the excess entries are never rolled back.

The analogous structure to the `BalancerPriceOracle` bug is exact:

| BalancerPriceOracle | CKB `add_ask_for_txs` |
|---|---|
| `cumulativePrice += averagePrice * elapsedTime` (state mutated) | entries inserted into `unknown_tx_hashes` (state mutated) |
| slippage check fires → `return false` | size check fires → `return TooManyUnknownTransactions` |
| `cumulativePrice` stays corrupted | inserted entries stay in the map |
| future TWAP calculations are wrong | future fetch-request batches are inflated |

### Impact Explanation

- **Memory exhaustion**: Each call from a malicious peer inserts up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` new entries regardless of the current map size. Repeated calls accumulate entries without bound.
- **Inflated transaction-fetch traffic**: `unknown_tx_hashes` drives the periodic `get_ask_for_txs` loop that issues `GetTransactions` requests to peers. A bloated map causes the node to request large numbers of fabricated or useless tx hashes from its peers, wasting bandwidth and CPU.
- **Per-peer cap bypass**: The per-peer limit (`MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`) is checked only after insertion, so a single peer can permanently exceed its quota on every call that crosses the threshold.

### Likelihood Explanation

Any connected peer can send `SendBlockTransactions` / compact-block announcement messages that carry tx hashes. The `add_ask_for_txs` path is reached for every such message. No authentication, stake, or special role is required. A single malicious peer making repeated announcements with fresh (never-seen) tx hashes will continuously grow the map.

### Recommendation

Move the size check **before** the insertion loop, or use a local staging buffer and only commit to `unknown_tx_hashes` after the check passes:

```rust
pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
    let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

    // Guard BEFORE mutation
    if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE
        || unknown_tx_hashes.len()
            >= self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER
    {
        // per-peer check, then return error/ignored
        ...
        return StatusCode::TooManyUnknownTransactions.into();
    }

    for tx_hash in tx_hashes.into_iter().take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER) {
        // insert only after guard passes
        ...
    }
    Status::ok()
}
```

### Proof of Concept

1. Attacker connects to a CKB node as a normal sync peer.
2. Attacker repeatedly sends compact-block announcements or `SendBlockTransactions` messages, each carrying `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` fresh, random tx hashes that the victim has never seen.
3. Each call to `add_ask_for_txs` inserts all hashes into `unknown_tx_hashes` before the check fires.
4. The check returns `TooManyUnknownTransactions` or `Status::ignored()`, but the entries remain.
5. After N calls, `unknown_tx_hashes` contains `N × MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` entries — far beyond the intended cap.
6. The victim node's periodic fetch loop issues `GetTransactions` requests for all of these fabricated hashes, wasting memory and network resources proportional to N. [3](#0-2) [4](#0-3)

```rust
    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }

        // Check `unknown_tx_hashes`'s length after inserting the arrival `tx_hashes`
        if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE
            || unknown_tx_hashes.len()
                >= self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER
        {
            warn!(
                "unknown_tx_hashes is too long, len: {}",
                unknown_tx_hashes.len()
            );

            let mut peer_unknown_counter = 0;
            for (_hash, priority) in unknown_tx_hashes.iter() {
                for peer in priority.peers.iter() {
                    if *peer == peer_index {
                        peer_unknown_counter += 1;
                    }
                }
            }
            if peer_unknown_counter >= MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER {
                return StatusCode::TooManyUnknownTransactions.into();
            }

            return Status::ignored();
        }

        Status::ok()
    }
```
