### Title
Silent u128→u64 Truncation in DAO Withdrawal Capacity Calculation Produces Withdrawal Amount Less Than Deposit — (File: util/dao/src/lib.rs)

### Summary
`DaoCalculator::calculate_maximum_withdraw` computes the maximum withdrawal capacity using a u128 intermediate value but casts it to u64 with a silent truncating `as u64` cast. If the intermediate product overflows u64::MAX, the result wraps to a small value, causing the returned withdrawal capacity to be far less than the original deposit — a direct analog to the external report's "free balance from report can be less than free balance from previous era" discrepancy.

### Finding Description
In `calculate_maximum_withdraw` (`util/dao/src/lib.rs`), the withdrawal amount is computed as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The cast `withdraw_counted_capacity as u64` is a **silent truncating cast**. If `withdraw_counted_capacity` exceeds `u64::MAX`, the value wraps modulo 2^64 to a small number. The subsequent `safe_add(occupied_capacity)` may then succeed without error, returning a `withdraw_capacity` that is orders of magnitude smaller than the original deposit.

This is inconsistent with every other u128→u64 conversion in the same codebase, which all use `u64::try_from(...).map_err(|_| DaoError::Overflow)?`:

```rust
// secondary_block_reward — correct pattern
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch — correct pattern
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) [3](#0-2) 

The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

Since `withdrawing_ar ≥ deposit_ar` always holds (AR is monotonically increasing, enforced by the block-number check at line 142), the ratio `withdrawing_ar / deposit_ar` grows over time. For a depositor holding a large fraction of the total CKB supply, this threshold is reachable once the accumulation rate has grown sufficiently.

The existing test `check_withdraw_calculation_overflows` only covers the case where `safe_add(occupied_capacity)` itself overflows; it does not cover the case where `withdraw_counted_capacity` wraps to a small value and `safe_add` silently succeeds with the wrong result. [4](#0-3) 

### Impact Explanation
1. **Loss of depositor funds**: A DAO depositor whose withdrawal triggers the truncation receives a withdrawal capacity far below their deposit principal plus accrued interest. The difference is permanently lost.
2. **Incorrect DAO field propagated to block headers**: `calculate_maximum_withdraw` feeds into `withdrawed_interests`, which feeds into `current_s` in `dao_field_with_current_epoch`. A truncated (too-small) `withdrawed_interests` causes `current_s` to be inflated — the NervosDAO savings counter in every subsequent block header is wrong. [5](#0-4) [6](#0-5) 

### Likelihood Explanation
The overflow requires `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. With the current total CKB supply (~3.36 × 10^18 shannons) and an initial AR of 10^16, the AR must grow approximately 5.5× before a maximum-supply deposit overflows. This is a long-horizon condition. However:
- The code path is consensus-critical and executed on every DAO phase-2 withdrawal.
- The bug is latent and silent — no error is returned, so no node detects the miscalculation.
- The fix is a one-line change to match the pattern already used everywhere else in the same file.

### Recommendation
Replace the silent truncating cast with the checked conversion already used in all other arithmetic paths in this file:

```rust
// Before (unsafe):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After (safe, consistent with rest of codebase):
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
```

This makes the function return `DaoError::Overflow` instead of silently returning a wrong (smaller) value, consistent with `secondary_block_reward` and `dao_field_with_current_epoch`.

### Proof of Concept
Construct a DAO deposit where `counted_capacity × withdrawing_ar` overflows u64 but `(counted_capacity × withdrawing_ar % 2^64) + occupied_capacity` does not:

```rust
// counted_capacity chosen so that:
//   counted_capacity * withdrawing_ar / deposit_ar  wraps to a small value
// e.g., counted_capacity = u64::MAX - occupied_capacity + 1
//        deposit_ar      = 10_000_000_000_000_000   (initial AR)
//        withdrawing_ar  = 20_000_000_000_000_000   (AR doubled)
//
// withdraw_counted_capacity (u128) = (u64::MAX - occ + 1) * 2 = 2*u64::MAX - 2*occ + 2
// as u64 truncates to: (2*u64::MAX - 2*occ + 2) % 2^64 = u64::MAX - 2*occ + 1
//   (still large — adjust ratio to get a small wrapped value)
//
// With withdrawing_ar = 10_000_000_000_000_001 (just above deposit_ar):
//   withdraw_counted_capacity ≈ u64::MAX + small_delta
//   as u64 → small_delta   (a few shannons)
//   safe_add(occupied_capacity) succeeds
//   returned withdraw_capacity ≈ occupied_capacity  (far below deposit)
```

The existing test at `util/dao/src/tests.rs:295-349` demonstrates the overflow path but only catches the `safe_add` overflow; a crafted AR ratio that wraps `withdraw_counted_capacity` to a sub-`u64::MAX` value bypasses that check and returns a silently wrong result. [7](#0-6) [8](#0-7)

fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
