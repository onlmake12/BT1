### Title
Unauthenticated `clear_tx_pool` RPC Allows Any Caller to Wipe the Entire Transaction Pool — (`rpc/src/module/pool.rs`)

### Summary
The `clear_tx_pool()` RPC method in the Pool module performs a fully destructive, irreversible state-changing operation — flushing every pending and proposed transaction from the node's memory pool — with no per-method authorization check. Any caller who can reach the RPC endpoint (the default attacker profile explicitly listed in scope as "RPC caller") can invoke it unconditionally. This is a direct analog to the reported `setTolerancePercentageForRewards()` pattern: a privileged, state-mutating function that is missing an access restriction.

### Finding Description

`clear_tx_pool()` is defined in the `PoolRpc` trait and implemented in `PoolRpcImpl`: [1](#0-0) [2](#0-1) 

The implementation calls `tx_pool.clear_pool(snapshot)` directly with no caller identity check, no token, no IP allowlist enforcement at the method level, and no rate limit. The Pool module is enabled in the default node configuration: [3](#0-2) 

The only protection is the network-level binding to `127.0.0.1:8114` by default. However:
- Operators routinely expose the RPC port to non-loopback addresses for remote management.
- Any local process on the same host (another user, a compromised service, a malicious script) can call it without any credential.
- The RPC server itself has no per-method authentication layer; there is no equivalent of an `onlyAdmin` guard anywhere in the method body or trait definition.

The same pattern applies to `clear_tx_verify_queue()` and `remove_transaction()` in the same file: [4](#0-3) [5](#0-4) 

### Impact Explanation

A successful call to `clear_tx_pool` immediately evicts every unconfirmed transaction from the node's memory pool. Consequences:
- All pending user transactions are silently dropped; senders receive no notification and must resubmit.
- Transactions that were about to be proposed or committed in the next block are lost, delaying confirmation and potentially causing time-sensitive transactions (e.g., those with `since` locks near expiry) to miss their window.
- A targeted attacker can repeatedly call this in a loop, keeping the pool perpetually empty and effectively censoring all transactions on that node.
- For a mining node, this prevents the node from including any transactions in blocks it assembles, reducing miner revenue.

Severity is **Medium**: no direct theft of on-chain funds, but concrete operational disruption and potential economic harm to users and operators.

### Likelihood Explanation

- The Pool RPC module is **enabled by default**.
- Many node operators expose the RPC port beyond localhost for tooling, dashboards, or remote management — a common real-world deployment pattern explicitly warned against in the docs but widely practiced.
- No credential, signature, or secret is required; a single unauthenticated HTTP POST suffices.
- The call is trivially scriptable and can be repeated at high frequency.

### Recommendation

Add a per-method authorization mechanism for destructive Pool RPC operations. Options in increasing strength:
1. **HTTP Basic Auth gate**: extend the existing `parse_authorization` pattern (already present in the miner client) to the RPC server, and require a configured secret for state-mutating Pool methods.
2. **IP allowlist enforcement at method level**: reject `clear_tx_pool`, `clear_tx_verify_queue`, and `remove_transaction` calls from any source not in an explicit operator-configured allowlist, independent of the listen address.
3. **Separate privileged RPC socket**: expose destructive methods only on a separate, permission-restricted Unix socket or a second listen address, analogous to how `Debug` module is kept separate from the default module list.

### Proof of Concept

With a default or network-exposed CKB node (Pool module enabled):

```bash
# Submit a transaction to populate the pool
curl -s -X POST http://<node>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"send_transaction","params":[{...}],"id":1}'

# Any unauthenticated caller wipes the entire pool
curl -s -X POST http://<node>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":2}'

# Verify pool is empty
curl -s -X POST http://<node>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":3}'
# → "pending":"0x0","proposed":"0x0"
```

The call succeeds with `"result": null` and no authentication challenge. Repeating it in a loop keeps the pool perpetually empty, effectively halting transaction propagation on the targeted node.

    fn clear_tx_pool(&self) -> Result<()> {
        let snapshot = Arc::clone(&self.shared.snapshot());
        let tx_pool = self.shared.tx_pool_controller();
        tx_pool
            .clear_pool(snapshot)
            .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;

        Ok(())
    }
