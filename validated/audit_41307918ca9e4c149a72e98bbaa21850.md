### Title
`PythAggregatorV3#latestRoundData` Returns Stale Price Without Freshness Validation - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is a Chainlink `AggregatorV3`-compatible adapter that wraps Pyth price feeds. Every price-reading function in the contract — `latestAnswer()`, `latestRoundData()`, `getRoundData()`, and `decimals()` — calls `pyth.getPriceUnsafe(priceId)` with no staleness check. Any downstream protocol that uses this adapter as a drop-in Chainlink replacement can silently consume arbitrarily stale prices.

---

### Finding Description

`PythAggregatorV3.sol` implements the Chainlink `AggregatorV3` interface so that protocols already integrated with Chainlink can switch to Pyth with minimal code changes. However, all four price-reading functions delegate to `getPriceUnsafe`, which is explicitly documented to return a price from **arbitrarily far in the past**:

```solidity
// latestAnswer — line 53-56
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// latestRoundData — line 99-119
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

Neither function checks whether `price.publishTime` is within an acceptable window of `block.timestamp`. The `updatedAt` field returned by `latestRoundData()` is simply `price.publishTime`, which may be hours or days old. No revert or error is raised.

The Pyth core contract's `getPriceUnsafe` is intentionally unchecked:

```solidity
// Pyth.sol line 184-194
function getPriceUnsafe(bytes32 id) public view override returns (PythStructs.Price memory price) {
    PythInternalStructs.PriceInfo storage info = _state.latestPriceInfo[id];
    price.publishTime = info.publishTime;
    ...
    if (price.publishTime == 0) revert PythErrors.PriceFeedNotFound();
}
```

The only check is that the price has ever been set (`publishTime != 0`). There is no maximum-age enforcement.

---

### Impact Explanation

Protocols that integrate `PythAggregatorV3` as a Chainlink drop-in and call `latestRoundData()` or `latestAnswer()` will receive stale prices during:

- Periods of network congestion where no one has called `updateFeeds()` recently.
- Oracle downtime or Hermes service interruption.
- Any gap between price updates on the target chain.

Stale prices can cause incorrect liquidations, mispriced collateral, wrong trade execution, or acceptance of under-collateralized positions — the same class of loss described in the external report. Because the adapter is presented as a Chainlink-compatible interface, integrators may reasonably assume the adapter enforces freshness (as Chainlink's own heartbeat mechanism does), and may not add an independent `updatedAt` check.

---

### Likelihood Explanation

The `PythAggregatorV3` contract is part of Pyth's official Solidity SDK and is actively promoted for Chainlink-to-Pyth migrations. Any protocol that deploys it and does not independently validate `updatedAt < block.timestamp - MAX_DELAY` is exposed. The Pyth pull-oracle model requires an explicit `updateFeeds()` call before reading; if no keeper or user submits a fresh update, the on-chain price can become arbitrarily stale. This is a realistic scenario during congestion or downtime.

---

### Recommendation

Add a configurable `MAX_DELAY` staleness guard in `latestRoundData()` and `latestAnswer()`:

```solidity
uint256 public constant MAX_DELAY = 3600; // configurable

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    require(
        block.timestamp - price.publishTime <= MAX_DELAY,
        "Stale price"
    );
    ...
}
```

Alternatively, replace `getPriceUnsafe` with `getPriceNoOlderThan(priceId, MAX_DELAY)`, which already enforces this check internally via `AbstractPyth.sol`:

```solidity
// AbstractPyth.sol line 50-60
function getPriceNoOlderThan(bytes32 id, uint age) public view virtual override
    returns (PythStructs.Price memory price) {
    price = getPriceUnsafe(id);
    if (diff(block.timestamp, price.publishTime) > age)
        revert PythErrors.StalePrice();
    return price;
}
```

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth EVM contract.
2. Call `updateFeeds()` once to seed the price.
3. Wait longer than any reasonable heartbeat (e.g., 2 hours) without calling `updateFeeds()` again.
4. Call `latestRoundData()`. It returns `(roundId, answer, startedAt, updatedAt, answeredInRound)` where `updatedAt` is 2+ hours in the past.
5. No revert occurs. A downstream lending protocol calling `latestRoundData()` and not independently checking `updatedAt` will use the 2-hour-old price for collateral valuation or liquidation decisions. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
