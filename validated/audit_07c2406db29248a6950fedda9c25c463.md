### Title
`getPricesNoOlderThan` Staleness Check Uses Max Subscription Timestamp Instead of Per-Feed `publishTime`, Allowing Stale Individual Prices to Pass Freshness Guard - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getPricesNoOlderThan()` checks freshness against `status.priceLastUpdatedAt`, which is set to the **maximum** `publishTime` across all feeds in a subscription batch. For subscriptions that mix always-active feeds (e.g., crypto) with closed-market feeds (e.g., equities), the max timestamp reflects the crypto feed's recency while individual equity feed prices can be days old. A consumer calling `getPricesNoOlderThan` with a tight `age_seconds` will pass the staleness guard and receive prices that are actually far older than `age_seconds`.

---

### Finding Description

In `_validateShouldUpdatePrices()`, the representative timestamp for the entire subscription update is computed as the **maximum** `publishTime` across all feeds:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

This max value is then stored as the subscription-level freshness marker:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
```

The design rationale is explicitly documented in the code — closed markets (equities, FX on weekends) will have a `publishTime` from their last trading period, so the max is used to avoid rejecting valid updates.

However, `getPricesNoOlderThan()` then uses this same subscription-level max to gate per-feed price reads:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
```

`getPricesUnsafe` returns the raw stored `PriceFeed.price` structs, each carrying their own individual `publishTime`. The staleness check never inspects individual feed `publishTime` values — only the subscription-level max. A consumer requesting a closed-market feed with `age_seconds = 3600` will pass the check (because the max publishTime from an open-market feed is recent) but receive a price whose actual `publishTime` is 60+ hours old.

The same flaw exists in `getEmaPricesNoOlderThan()`:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getEmaPricesUnsafe(subscriptionId, priceIds);
```

---

### Impact Explanation

A DeFi protocol (lending, derivatives, structured products) that:
1. Subscribes to a mixed feed set (e.g., ETH/USD + AAPL/USD)
2. Calls `getPricesNoOlderThan(subscriptionId, [AAPL_USD_ID], 3600)` expecting a price no older than 1 hour

…will receive an AAPL price that is 60+ hours old (last Friday's close, submitted on Monday morning) while the function does **not** revert. The protocol believes the price is fresh and proceeds with incorrect collateral valuations, liquidation thresholds, or settlement prices.

This is a silent correctness failure: no revert, no warning, wrong price age.

---

### Likelihood Explanation

The Scheduler's own code comments explicitly anticipate mixed-market subscriptions containing closed-market feeds. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hour` constant in `SchedulerConstants.sol` is specifically designed to allow old closed-market timestamps through. Any integrator building a protocol that tracks both crypto and traditional-asset feeds via a single Scheduler subscription is exposed. The entry path requires only being a whitelisted reader — no privileged role.

---

### Recommendation

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` should validate freshness against each **individual** feed's `publishTime`, not the subscription-level max. For example:

```solidity
for (uint i = 0; i < priceFeeds.length; i++) {
    if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
        revert PythErrors.StalePrice();
}
```

Alternatively, the function's documentation should prominently warn that `age_seconds` is checked against the subscription's max publishTime, not per-feed publishTime, and consumers must inspect individual `publishTime` fields themselves when using mixed-market subscriptions.

---

### Proof of Concept

**Setup:** Subscription with two feeds — ETH/USD (crypto, always active) and AAPL/USD (equity).

**Monday 09:00 UTC — keeper submits update:**
- ETH/USD `publishTime` = `T` (≈ block.timestamp)
- AAPL/USD `publishTime` = `T − 60 hours` (last Friday close)
- `_validateShouldUpdatePrices` computes `updateTimestamp = max(T, T−60h) = T`
- `status.priceLastUpdatedAt = T`

**Monday 09:01 UTC — consumer calls:**
```solidity
getPricesNoOlderThan(subscriptionId, [AAPL_USD_ID], 3600)
```

**Check in `getPricesNoOlderThan`:**
```
distance(block.timestamp, status.priceLastUpdatedAt)
= distance(T + 60s, T)
= 60 seconds  ≤  3600  → does NOT revert
```

**Returned price:** AAPL/USD with `publishTime = T − 60 hours` — **60 hours stale**, silently accepted.
