### Title
`get_overview` RPC Returns Raw `u64` System Metrics as Plain JSON Numbers, Causing Silent Precision Loss in IEEE 754 Clients — (File: `util/jsonrpc-types/src/terminal.rs`)

### Summary

The `get_overview` RPC endpoint returns multiple `u64` system-metric fields (memory, disk, network byte counters) as plain JSON numbers rather than the 0x-prefixed hex strings mandated by CKB's own RPC policy. Standard JSON clients (browsers, Node.js, jq) that use IEEE 754 double-precision floats will silently corrupt any value exceeding 2^53 − 1 (≈ 9 × 10^15). Cumulative counters such as `total_written_bytes`, `total_read_bytes`, `total_received`, and `total_transmitted` can reach this threshold on a long-running mainnet node, causing monitoring tools to silently read wrong values.

### Finding Description

CKB's own RPC documentation explicitly states:

> "CKB RPC does not use JSON numbers because of the precision problem. Float point numbers are not used in the RPC, and integers are encoded as 0x-prefixed hex string such as `0x10` for decimal value 16." [1](#0-0) 

All standard CKB numeric types (`Uint32`, `Uint64`, `Uint128`) enforce this via `JsonUint<T>`, which calls `deserializer.deserialize_str()` and rejects bare JSON numbers. [2](#0-1) 

However, the structs introduced for the Terminal RPC module bypass this entirely. The following fields in `util/jsonrpc-types/src/terminal.rs` are declared as raw `u64` (or `usize`) and derive `Serialize`/`Deserialize` directly, causing serde to emit them as plain JSON integers:

| Struct | Field | Type |
|---|---|---|
| `SysInfo` | `memory`, `virtual_memory` | `u64` |
| `Global` | `total_memory`, `used_memory` | `u64` |
| `Disk` | `total_space`, `available_space` | `u64` |
| `DiskUsage` | `total_written_bytes`, `written_bytes`, `total_read_bytes`, `read_bytes` | `u64` |
| `Network` | `received`, `total_received`, `transmitted`, `total_transmitted` | `u64` |
| `PeerInfo` | `peer_id` | `usize` | [3](#0-2) 

The `get_overview` RPC method in `rpc/src/module/terminal.rs` populates these fields directly from `sysinfo` crate values and serializes them into the JSON response: [4](#0-3) 

The RPC documentation example itself embeds these as bare integers:

```json
"total_space": 1000000000000,
"total_memory": 8589934592,
"total_read_bytes": 1073741824
``` [5](#0-4) 

### Impact Explanation

Any RPC caller using a JavaScript browser client, Node.js, or jq to call `get_overview` will silently receive corrupted values for any field exceeding 2^53 − 1 (9,007,199,254,740,991). The corruption is silent — no error is raised; the number is rounded to the nearest representable IEEE 754 double.

The most realistic fields to exceed 2^53 on a long-running mainnet node are the cumulative I/O and network counters (`total_written_bytes`, `total_read_bytes`, `total_received`, `total_transmitted`). A node that has transmitted ~9 PB of data total would have `total_transmitted > 2^53`. Even `total_memory` on a server with more than 8 PB of RAM (theoretical) would be affected, though memory values are bounded in practice.

The impact is incorrect monitoring data silently delivered to RPC consumers — a direct analog to the DeGate report where large numeric fields were silently corrupted by standard JSON parsers.

### Likelihood Explanation

Medium. The `get_overview` endpoint is specifically designed for TUI monitoring dashboards and is reachable by any unprivileged RPC caller. The cumulative byte counters (`total_written_bytes`, `total_received`, `total_transmitted`) grow monotonically and will eventually exceed 2^53 on a busy, long-running node. The inconsistency with CKB's own stated RPC policy means any client written to the documented standard (hex strings) will be surprised by these fields.

### Recommendation

Replace all raw `u64`/`usize` fields in the Terminal RPC types with `Uint64` (i.e., `JsonUint<u64>`) from `ckb_jsonrpc_types`, consistent with every other numeric field in the CKB RPC surface. This will serialize them as 0x-prefixed hex strings and eliminate the IEEE 754 precision hazard.

```rust
// Before (util/jsonrpc-types/src/terminal.rs)
pub memory: u64,
pub virtual_memory: u64,
pub total_memory: u64,
// ...

// After
pub memory: Uint64,
pub virtual_memory: Uint64,
pub total_memory: Uint64,
// ...
```

`PeerInfo.peer_id` should similarly be changed from `usize` to `Uint64`.

### Proof of Concept

Call `get_overview` from a Node.js client on a node where `total_transmitted` has accumulated past 2^53:

```js
const res = await fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ jsonrpc: "2.0", method: "get_overview", params: [null], id: 1 })
});
const data = await res.json();
// data.result.sys.global.networks[0].total_transmitted
// will be rounded to the nearest multiple of 2048 once it exceeds 2^53
console.log(data.result.sys.global.networks[0].total_transmitted);
// Expected: 9007199254740993  (2^53 + 1)
// Actual:   9007199254740992  (silently rounded down)
```

The root cause is the plain `u64` field declarations in `util/jsonrpc-types/src/terminal.rs` lines 37–93, serialized without the `JsonUint` wrapper that all other CKB RPC numeric types use. [6](#0-5) [7](#0-6)
