### Title
PeerStore `check_purge` Permanently Fails to Evict When All Addresses Are Connectable and Distributed Across Small Network Groups — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

The `PeerStore` enforces a hard cap of `ADDR_COUNT_LIMIT = 16384` discovered peer addresses. When the store is full, `check_purge` attempts eviction in two steps: first removing non-connectable (failed/expired) addresses, then removing addresses from over-represented network groups. The second step only evicts from groups with **strictly more than 4** peers. If an attacker fills all 16384 slots with connectable addresses spread across 4096+ distinct `/24` subnets (≤4 per group), both eviction steps produce an empty candidate set and `check_purge` returns `PeerStoreError::EvictionFailed`. Every subsequent call to `add_addr` — the only path for discovered peer addresses — propagates this error and silently discards the new address. The peer store is permanently exhausted: no new legitimate peer addresses can ever be added without a node restart.

---

### Finding Description

`PeerStore::check_purge` is called at the top of `add_addr` before inserting any new discovered address:

```rust
pub fn add_addr(&mut self, addr: Multiaddr, flags: Flags) -> Result<()> {
    if self.ban_list.is_addr_banned(&addr) { return Ok(()); }
    self.check_purge()?;          // ← propagates EvictionFailed
    ...
}
``` [1](#0-0) 

Inside `check_purge`, the first eviction pass collects addresses where `!addr.is_connectable(now_ms)`. If all stored addresses are connectable, this set is empty and the code falls through to the second pass: [2](#0-1) 

The second pass groups addresses by `/24` network segment, sorts groups by size, and then applies:

```rust
if addrs.len() > 4 {
    Some(addrs.iter().choose_multiple(&mut rand::thread_rng(), 2)...)
} else {
    None   // ← groups with ≤4 peers are NEVER candidates for eviction
}
``` [3](#0-2) 

If every network group contains ≤4 addresses, `candidate_peers` is empty and the function returns:

```rust
if candidate_peers.is_empty() {
    return Err(PeerStoreError::EvictionFailed.into());
}
``` [4](#0-3) 

The fixed limit is defined as:

```rust
pub(crate) const ADDR_COUNT_LIMIT: usize = 16384;
``` [5](#0-4) 

The two callers of `add_addr` in the P2P stack both silently swallow the error:

- `DiscoveryAddressManager::add_new_addrs` logs at `debug` level and continues: [6](#0-5) 

- `IdentifyProtocol::add_remote_listen_addrs` logs at `error` level and continues: [7](#0-6) 

Note that `add_outbound_addr` (used for confirmed outbound connections) bypasses `check_purge` entirely and can push the store past the limit, but `add_addr` — the only path for *discovered* addresses — is permanently blocked once the limit is reached under the described conditions. [8](#0-7) 

---

### Impact Explanation

Once the peer store is exhausted in this way, the node can no longer learn about new peers via the discovery or identify protocols. If all currently connected peers eventually disconnect, the node has no fresh candidates to dial and becomes progressively isolated from the network. Peer discovery — the mechanism by which a node bootstraps and maintains connectivity — is permanently impaired until the node is restarted. This is a direct analog to the MultisigManager finding: a fixed-size registry becomes permanently full because the eviction algorithm has a blind spot for a specific distribution of entries, and there is no replacement path.

---

### Likelihood Explanation

The discovery protocol allows up to `MAX_ADDR_TO_SEND = 1000` addresses per `Nodes` message. [9](#0-8) 

Filling 16384 slots requires roughly 17 such messages. An attacker controlling addresses across 4096+ distinct `/24` subnets (e.g., via a cloud provider or botnet) can advertise these addresses to a target node. Because `add_addr` sets `last_connected_at_ms = 0` for discovered addresses, freshly inserted entries are connectable by default (they have not yet failed any connection attempt), satisfying the condition that makes the first eviction pass produce an empty set. The attacker does not need to run actual nodes at those addresses — they only need the addresses to be accepted into the store before any connection attempt marks them non-connectable.

---

### Recommendation

Apply the same fix suggested in the analogous GoGoPool finding: count only *connectable* addresses toward the limit, or lower the group-size threshold for eviction from `> 4` to `>= 1` so that at least one address can always be evicted when the store is full. A minimal targeted fix:

```rust
// Change:
if addrs.len() > 4 {
// To:
if addrs.len() >= 1 {
```

Alternatively, add a final fallback eviction step that randomly removes one address from the largest group when both primary strategies fail, ensuring `check_purge` never returns `EvictionFailed` as long as the store is non-empty.

---

### Proof of Concept

1. Connect to a target CKB node as a peer.
2. Send repeated `Nodes` (discovery) messages, each containing up to 1000 addresses drawn from 4096+ distinct `/24` subnets, with at most 4 addresses per subnet.
3. After ~17 such messages, `addr_manager.count()` reaches `ADDR_COUNT_LIMIT = 16384`.
4. All stored addresses have `last_connected_at_ms = 0` and zero failed attempts, so `is_connectable` returns `true` for all of them.
5. Every subsequent `add_addr` call hits `check_purge`, which finds zero non-connectable addresses (first pass empty) and zero groups with `> 4` members (second pass empty), and returns `Err(PeerStoreError::EvictionFailed)`.
6. The error is logged and discarded by `DiscoveryAddressManager::add_new_addrs`.
7. The node can no longer add any new peer addresses. Peer discovery is permanently blocked until restart. [2](#0-1) [10](#0-9)

```rust
    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {
                            Some(
                                addrs
                                    .iter()
                                    .choose_multiple(&mut rand::thread_rng(), 2)
                                    .into_iter()
                                    .map(|addr| addr.addr.clone())
                                    .collect::<Vec<Multiaddr>>(),
                            )
                        } else {
                            None
                        }
                    })
                    .flatten()
                    .collect()
            };

            for key in candidate_peers.iter() {
                self.addr_manager.remove(key);
            }

            if candidate_peers.is_empty() {
                return Err(PeerStoreError::EvictionFailed.into());
            }
        }
        Ok(())
    }
