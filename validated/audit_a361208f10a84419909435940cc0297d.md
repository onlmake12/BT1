Audit Report

## Title
Integer Overflow in `GetLastStateProofProcess::execute` Bypasses `GET_LAST_STATE_PROOF_LIMIT`, Enabling Node DoS via Unbounded Allocation — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The limit guard at line 201 computes `difficulties.len() + (last_n_blocks as usize) * 2`. In release mode, Rust uses wrapping arithmetic; an attacker sending `last_n_blocks = 2^63` with an empty `difficulties` list causes the multiplication to wrap to `0`, making the guard evaluate `0 > 1000 = false`. The check is silently bypassed. Downstream, the server unconditionally collects every block number from `start_block_number` to the chain tip into a `Vec`, then calls `complete_headers` for each entry — performing multiple DB lookups per block — proportional to the full chain length, causing OOM or a multi-minute CPU stall on a production node.

## Finding Description
**Overflow site** — `get_last_state_proof.rs`, lines 199–205:
```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```
`last_n_blocks` is a raw `u64` decoded from the peer message with no prior bound check. On a 64-bit host, `usize` is 64 bits. With `last_n_blocks = 9223372036854775808` (`2^63`):
```
(9223372036854775808_usize) * 2  →  wraps to 0  (release mode)
0 + 0 = 0  >  1000  →  false   ← guard never fires
```

**Unbounded allocation site** — lines 291–297:
```rust
let (sampled_numbers, last_n_numbers) = if last_block_number - start_block_number
    <= last_n_blocks
{
    let sampled_numbers = Vec::new();
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
    (sampled_numbers, last_n_numbers)
```
Because `last_n_blocks` is astronomically large, the condition is always true for any real chain. The server collects every block number from `start_block_number` to the chain tip into a `Vec<BlockNumber>`.

**Per-entry DB work** — `complete_headers`, lines 132–163: each iteration performs at minimum three DB operations (`get_ancestor` O(log n), `get_block`, `chain_root_mmr`).

**Existing guards are insufficient**: The only guard is the overflowable check at line 201. There is no separate bound check on `last_n_blocks` itself before the arithmetic, and no rate limit at the message-processing layer.

## Impact Explanation
For a CKB mainnet node at ~12 million blocks, a single crafted peer message triggers allocation of ~96 MB for the `Vec<BlockNumber>`, followed by construction of a `Vec<VerifiableHeader>` (several GB) and 36M+ RocksDB lookups. This is sufficient to cause an OOM kill or multi-minute CPU stall on a production full node running the light-client server. This matches the allowed bounty impact: **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The light-client protocol server is a production feature. The attacker requires only a TCP connection to the node's P2P port — no key, no PoW, no privileged role. The crafted message is a single valid molecule-encoded `GetLastStateProof` with `last_n_blocks = 9223372036854775808` and `difficulties = []`. Release mode (wrapping overflow) is the standard deployment configuration. The attack is repeatable with no rate limit at the message-processing layer.

## Recommendation
Replace the unchecked arithmetic with saturating operations, or reject `last_n_blocks` before the arithmetic:

```rust
// Option A: saturating arithmetic
let total = self.message.difficulties().len()
    .saturating_add((last_n_blocks as usize).saturating_mul(2));
if total > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option B: early rejection
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Either fix closes the bypass before any snapshot or Vec allocation occurs.

## Proof of Concept
```rust
// Unit test — no chain needed
let last_n_blocks: u64 = (usize::MAX / 2 + 1) as u64;  // = 2^63
let difficulties_len: usize = 0;

// Simulate release-mode wrapping arithmetic
let total = difficulties_len.wrapping_add((last_n_blocks as usize).wrapping_mul(2));
assert_eq!(total, 0);          // overflow → 0
assert!(total <= 1000);        // guard is bypassed
// → server proceeds to (0..chain_tip).collect() with no bound
```
To confirm end-to-end: connect a peer to a node with the light-client server enabled, send a molecule-encoded `GetLastStateProof` message with `last_n_blocks = 9223372036854775808`, `difficulties = []`, and a valid `last_hash` on the main chain. Observe memory growth proportional to chain length and eventual OOM or stall.
