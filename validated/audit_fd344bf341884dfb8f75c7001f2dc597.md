### Title
Missing Ancestor-Bound Check in `GetBlocksProofProcess::execute` Enables Unbanned Repeated `InternalError` DoS — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

---

### Summary

`GetBlocksProofProcess::execute` partitions requested block hashes only by `is_main_chain`, with no check that each found block's number is ≤ `last_block.number()`. An attacker can supply `last_hash = block[N]` and `block_hashes = [block[N+1], …, block[N+K]]` — all valid main-chain hashes — causing `reply_proof` to call `chain_root_mmr(N-1).gen_proof(positions)` where every position is outside the MMR's range `[0, N-1]`. This returns `StatusCode::InternalError` (5xx), which `should_ban()` explicitly excludes from banning. The peer is never banned and can repeat the loop indefinitely.

---

### Finding Description

**Step 1 — Attacker-controlled entrypoint.**
Any peer connected on the light-client P2P protocol can send a `GetBlocksProof` message. No authentication or privilege is required.

**Step 2 — `execute()` validates `last_hash` but not the relative height of `block_hashes`.**

In `execute()`, `last_block_hash` is checked to be on the main chain, and `block_hashes` are partitioned into `found` (on main chain) vs `missing`. There is no check that a found block's `header.number()` is ≤ `last_block.number()`: [1](#0-0) 

For each found hash, `leaf_index_to_pos(header.number())` is pushed into `positions`. If the attacker supplies hashes for blocks N+1..N+K, those positions correspond to MMR leaf indices ≥ N.

**Step 3 — `reply_proof()` builds an MMR rooted at `last_block.number() - 1`.** [2](#0-1) 

The MMR is `chain_root_mmr(N-1)`, which covers only leaf indices 0..N-1. Calling `gen_proof` with positions ≥ N is out-of-range and returns an error, causing the function to return `StatusCode::InternalError`.

**Step 4 — `InternalError` (500) is a 5xx code; `should_ban()` only bans 4xx codes.** [3](#0-2) 

`InternalError = 500` falls in the 5xx range, so `should_ban()` returns `None`. The peer receives only a warning log entry. [4](#0-3) 

**Step 5 — No rate limiting on the light-client protocol handler.**

Unlike the hole-punching protocol which has an explicit `rate_limiter`, the `LightClientProtocol::received` handler has no per-peer rate limiting. The attacker can send up to `GET_BLOCKS_PROOF_LIMIT = 1000` hashes per message: [5](#0-4) 

---

### Impact Explanation

Each crafted request forces the server to:
1. Look up `last_block` from the store.
2. Look up headers for up to 1000 main-chain block hashes.
3. Compute `mmr.get_root()` (succeeds, does real work).
4. Attempt `mmr.gen_proof(positions)` (fails, does real work).

The attacker pays only network send cost. The server pays store lookups + MMR computation per request, with no ban and no rate limit enforced. This is a zero-cost amplified DoS loop against the light-client server.

---

### Likelihood Explanation

The attack requires only:
- Knowledge of any two consecutive main-chain block hashes (trivially obtained from any public CKB node or explorer).
- A P2P connection to the target light-client server (open by design).

No PoW, no key, no privileged role. The scenario is directly reproducible in a local testnet.

---

### Recommendation

In `execute()`, after partitioning into `found`, reject any found block whose `header.number()` exceeds `last_block.number()`. For example:

```rust
for block_hash in found {
    let header = snapshot
        .get_block_header(&block_hash)
        .expect("header should be in store");
    if header.number() > last_block.number() {
        return StatusCode::MalformedProtocolMessage
            .with_context("block hash is not an ancestor of last_hash");
    }
    positions.push(leaf_index_to_pos(header.number()));
    // ...
}
```

This converts the out-of-range case into a 4xx `MalformedProtocolMessage`, which `should_ban()` will catch and ban the peer.

```

2. Connect to the light-client server as a peer.
3. Send GetBlocksProof {
       last_hash:    block[N].hash,

5. Observe: peer is NOT banned (nc.not_banned(peer) == true).
6. Repeat step 3 indefinitely — server performs MMR work each time, peer is never disconnected.
```
