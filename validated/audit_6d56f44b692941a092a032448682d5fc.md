### Title
`lastPushAttempt` Set Before Simulation Causes Spurious Gas Escalation on Every `NoFreshUpdate` — (`File: apps/price_pusher/src/evm/evm.ts`)

### Summary

In `EvmPricePusher.updatePriceFeed()`, `this.lastPushAttempt` (which stores the gas price and nonce of the "last sent transaction") is written to memory **before** the simulation and transaction are attempted. When the simulation fails — most commonly with `NoFreshUpdate` (price already fresh on-chain, the normal steady-state) — the function returns early without clearing `lastPushAttempt`. On every subsequent call, the pusher incorrectly believes a transaction is stuck in the mempool and escalates the gas price by `overrideGasPriceMultiplier` (default 1.1×). Because `NoFreshUpdate` is the most frequent outcome, this escalation compounds on every push cycle until the cap (`overrideGasPriceMultiplierCap`, default 5×) is reached. When a real price update is finally needed, the pusher pays up to 5× the market gas price. If the wallet balance is insufficient at the escalated price, the pusher throws and crashes, causing price feed staleness.

### Finding Description

In `apps/price_pusher/src/evm/evm.ts`, `EvmPricePusher.updatePriceFeed()` follows this ordering:

```
// 1. Compute gasPrice (possibly already escalated from prior lastPushAttempt)
// 2. Compute txNonce = lastExecutedNonce + 1

// 3. *** SET lastPushAttempt BEFORE any I/O ***
this.lastPushAttempt = { gasPrice, nonce: txNonce };   // line 257-260

try {
  // 4. Simulate — may throw NoFreshUpdate, ContractFunctionExecutionError, etc.
  const { request } = await this.pythContract.simulate.updatePriceFeedsIfNecessary(...);
  // 5. Send tx
  const hash = await this.client.writeContract(request);
} catch (error) {
  if (NoFreshUpdate) return;          // lastPushAttempt NOT cleared
  if (ContractFunctionExecutionError) return;  // lastPushAttempt NOT cleared
  if (TransactionExecutionError) return;       // lastPushAttempt NOT cleared
  if (nonce errors) return;                    // lastPushAttempt NOT cleared
  ...
}
```

Because no transaction was broadcast, `lastExecutedNonce` on-chain is unchanged. On the next call:

- `lastPushAttempt.nonce (= N+1) > lastExecutedNonce (= N)` → the pusher thinks a tx is stuck
- `gasPriceToOverride = scaleGasPrice(lastPushAttempt.gasPrice, 1.1)`
- `gasPrice = escalateGasPrice(fresh, gasPriceToOverride, cap)` → gas price is bumped
- `lastPushAttempt` is overwritten with the **already-escalated** gas price

This compounds: each `NoFreshUpdate` cycle raises the stored gas price by 1.1×, up to the 5× cap. The correct pattern (as in the ZetaChain Ton fix) is to set `lastPushAttempt` only **after** the transaction is successfully submitted.

The `FeeCapTooLowError` / `replacement transaction underpriced` path is intentionally designed to retain `lastPushAttempt` (the comment at line 374-386 confirms this). But `NoFreshUpdate`, `ContractFunctionExecutionError`, and `TransactionExecutionError` paths are not — no transaction was ever sent in those cases, so there is nothing to escalate.

### Impact Explanation

1. **Gas overpayment**: Every real price update after a run of `NoFreshUpdate` cycles pays up to 5× the market gas price. This is a direct financial loss to the price pusher operator.
2. **Pusher crash / price feed staleness**: If the wallet balance is insufficient at the escalated price, the `InsufficientFundsError` branch throws (line 310), crashing the pusher process. This causes the price feeds served by that pusher to go stale, breaking downstream consumers.

### Likelihood Explanation

`NoFreshUpdate` is the **most common** simulation outcome in normal operation — it fires every push cycle where the on-chain price is already fresh (i.e., the majority of cycles). The escalation therefore begins immediately after the first `NoFreshUpdate` and reaches the 5× cap within ~17 consecutive `NoFreshUpdate` cycles (`1.1^17 ≈ 5`). On a pusher with a short `pushingFrequency`, this cap is reached within minutes.

### Recommendation

Move the `this.lastPushAttempt` assignment to **after** `writeContract` succeeds, mirroring the correct pattern from the ZetaChain Ton fix. For error paths where a transaction **was** submitted but failed to land (e.g., `FeeCapTooLowError`), retain `lastPushAttempt` explicitly. For paths where no transaction was ever sent (`NoFreshUpdate`, `ContractFunctionExecutionError`, nonce errors), clear `lastPushAttempt` before returning.

```ts
// After writeContract succeeds:
const hash = await this.client.writeContract(request);
this.lastPushAttempt = { gasPrice, nonce: txNonce }; // only set here
```

And in the `NoFreshUpdate` / nonce-error / `ContractFunctionExecutionError` catch branches, add:

```ts
this.lastPushAttempt = undefined;
return;
```

### Proof of Concept

1. Price pusher starts. `lastPushAttempt = undefined`.
2. Push cycle 1: price is fresh on-chain. Simulation reverts with `NoFreshUpdate`. `lastPushAttempt` is set to `{gasPrice: G, nonce: N+1}`. No tx sent.
3. Push cycle 2: `lastExecutedNonce = N` (unchanged). `lastPushAttempt.nonce (N+1) > N` → `gasPriceToOverride = G * 1.1`. `gasPrice = max(fresh, G*1.1)`. Simulation reverts with `NoFreshUpdate` again. `lastPushAttempt = {gasPrice: G*1.1, nonce: N+1}`.
4. Push cycles 3–17: same pattern. After 17 cycles, `gasPrice = G * 1.1^17 ≈ G * 5` (cap reached).
5. Push cycle 18: price finally goes stale. A real tx is submitted at `5× G`. Operator pays 5× the market rate.
6. If wallet balance < `5G * gasLimit`, `InsufficientFundsError` is thrown at line 310, the pusher crashes, and price feeds go stale.
