### Title
Missing Type Script Hash Type Validation in Non-Contextual Transaction Verifier — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist only against each output's **lock** script. It never inspects the output's **type** script hash type. Any transaction sender can craft an output whose type script carries a disallowed hash type (e.g., `Data3` = `3`) and the non-contextual verifier will accept it silently, allowing the transaction to enter the tx-pool and be relayed network-wide.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is defined as the set `{0, 1, 2, 4}` (Data, Type, Data1, Data2). Values such as `3` (`Data3`) are explicitly excluded. [1](#0-0) 

`ScriptHashTypeVerifier::verify()` iterates over every output and checks only `output.lock().hash_type()`. The `output.type_()` field is never read: [2](#0-1) 

The verifier is wired into `NonContextualTransactionVerifier`, which is the first gate a transaction must pass before entering the tx-pool or being relayed: [3](#0-2) 

The existing test suite only exercises the lock-script path (`test_not_enabled_hash_type_output_lock`); there is no corresponding test for the type-script path, confirming the gap is undetected: [4](#0-3) 

---

### Impact Explanation

A transaction whose output carries a type script with a disallowed hash type (e.g., raw byte `3`) bypasses `ScriptHashTypeVerifier` entirely. Such a transaction:

1. Is admitted to the local tx-pool.
2. Is relayed to every connected peer, which also admits it to their tx-pools.
3. Consumes tx-pool slots and relay bandwidth across the network.

If the script execution engine does not uniformly reject the unknown hash type at contextual verification time, the transaction could also be included in a block, potentially causing consensus divergence between nodes that handle the unknown hash type differently.

---

### Likelihood Explanation

The entry path requires only the ability to submit a transaction via the RPC (`send_transaction`) or the P2P relay protocol — both are available to any unprivileged user. Constructing such a transaction is trivial: set any output's type script hash type byte to `3`. No special keys, no privileged access, and no majority hashpower are required.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type for every output that carries one:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        let lock_ht = output.lock().hash_type();
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(lock_ht) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType { hash_type: lock_ht }.into());
        }

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_ht = type_script.hash_type();
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_ht) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType { hash_type: type_ht }.into());
            }
        }
    }
    Ok(())
}
```

Add a corresponding test (`test_not_enabled_hash_type_output_type`) mirroring the existing lock-script test.

---

### Proof of Concept

1. Build a transaction with one output whose type script has `hash_type = 3` (raw byte, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via `send_transaction` RPC or inject via the P2P relay.
3. Observe that `NonContextualTransactionVerifier::verify()` returns `Ok(())` — the transaction is accepted into the tx-pool and relayed to peers without any warning.

The root cause is the single-field check at `verification/src/transaction_verifier.rs:798` (`output.lock().hash_type()`) with no corresponding check on `output.type_()`. [2](#0-1)

    /// Creates a new NonContextualTransactionVerifier
    pub fn new(tx: &'a TransactionView, consensus: &'a Consensus) -> Self {
        NonContextualTransactionVerifier {
            version: VersionVerifier::new(tx, consensus.tx_version()),
            size: SizeVerifier::new(tx, consensus.max_block_bytes()),
            empty: EmptyVerifier::new(tx),

            script_hash_type: ScriptHashTypeVerifier::new(tx),

        self.empty.verify()?;
        self.duplicate_deps.verify()?;
        self.outputs_data_verifier.verify()?;
        self.script_hash_type.verify()?;
        Ok(())
    }

#[test]
pub fn test_not_enabled_hash_type_output_lock() {
    let transaction = TransactionBuilder::default()
        .output(
            CellOutput::new_builder()
                .lock(
                    Script::default()
                        .as_builder()
                        .hash_type(ScriptHashType::Data3)
                        .build(),
                )
                .build(),
        )
        .build();
    let verifier = ScriptHashTypeVerifier::new(&transaction);

    assert_error_eq!(
        verifier.verify().unwrap_err(),
        TransactionError::ScriptHashTypeNotPermitted {
            hash_type: ScriptHashType::Data3.into(),
        },
    );
}
