### Title
Ping-RTT Eviction Protection Bypassed via `as_secs()` Truncation — (`network/src/peer_registry.rs`)

### Summary

`try_evict_inbound_peer` uses `Duration::as_secs()` to compare peer ping RTTs for eviction protection. This truncates all sub-second RTTs to `0`, collapsing the entire `[1ms, 999ms]` range into a single bucket indistinguishable from `0`. An attacker who controls 8+ inbound connections and responds to pings immediately (sub-second RTT) can fill all `EVICTION_PROTECT_PEERS = 8` low-ping protection slots when honest peers have RTTs ≥ 1 s, ensuring only honest peers remain as eviction candidates.

### Finding Description

In `try_evict_inbound_peer`, the first protection pass sorts candidates by ping and retains the 8 lowest-ping peers: [1](#0-0) 

The comparator extracts ping via `.map(|p| p.as_secs())`: [2](#0-1) 

`Duration::as_secs()` returns only the whole-second component, discarding all sub-second precision. A peer with RTT = 1 ms and a peer with RTT = 999 ms both produce `0`. A peer with RTT = 1 000 ms produces `1`.

`sort_then_drop` sorts descending (`peer2_ping.cmp(&peer1_ping)`) and drops the front (highest-ping) entries, keeping the last `n = 8`: [3](#0-2) 

If 8 attacker peers all have `as_secs() = 0` and honest peers have `as_secs() ≥ 1`, the sort places honest peers at the front (dropped) and attacker peers at the back (protected). The same truncation defect also affects the second protection pass (`last_ping_protocol_message_received_at.as_secs()`): [4](#0-3) 

The `ping_rtt` field is set by `pong_received` using `now.saturating_duration_since(last_ping)`, which is fully attacker-controlled — the attacker simply replies to the ping message immediately: [5](#0-4) 

### Impact Explanation

With all 8 low-ping protection slots occupied by attacker peers, honest inbound peers are unprotected by the ping criterion. They then compete only in the "most recent message" and "longest connection time" passes. An attacker who also sends frequent pings and connects early can defeat those passes too, leaving honest peers as the sole eviction candidates. Repeated eviction cycles allow the attacker to eclipse the victim node, enabling network partition and consensus deviation.

### Likelihood Explanation

The precondition — honest peers having RTT ≥ 1 s — is unusual for typical LAN/regional peers but is achievable for cross-continental or congested paths. The attacker has full control over their own RTT (respond to ping immediately, guaranteed sub-second). No privileged access, PoW, or key material is required; only 8 inbound TCP connections are needed, which is well within reach of a single host.

### Recommendation

Replace `as_secs()` with `as_millis()` (or `subsec_millis`-aware comparison) in both the ping-RTT and last-message-time comparators so that sub-second differences are preserved and the protection criterion cannot be trivially saturated:

```rust
// Before (lossy):
.map(|p| p.as_secs())

// After (preserves sub-second precision):
.map(|p| p.as_millis())
```

Apply the same fix to the `last_ping_protocol_message_received_at` comparator at line 175.

### Proof of Concept

```rust
// 8 attacker peers: ping_rtt = Duration::from_millis(999) → as_secs() = 0
// 8 honest peers:   ping_rtt = Duration::from_millis(1001) → as_secs() = 1
// After sort_then_drop(8): honest peers are dropped (as_secs=1 sorted to front),
// attacker peers are kept (as_secs=0 sorted to back).
// Honest peers become eviction candidates; attacker peers are protected.
// Run try_evict_inbound_peer 1000 times: only honest peers are ever evicted.
```

```rust
fn sort_then_drop<T, F>(list: &mut Vec<T>, n: usize, compare: F)
where
    F: FnMut(&T, &T) -> std::cmp::Ordering,
{
    list.sort_by(compare);
    if list.len() > n {
        list.truncate(list.len() - n);
    }
}
```
