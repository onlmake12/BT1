Audit Report

## Title
`HeaderAcceptor::accept()` Missing `BLOCK_INVALID` Early-Return Allows Invalid Headers to Be Accepted as `HEADER_VALID` - (File: `sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` at lines 301–302 noting that a `BLOCK_INVALID` status should trigger an early return but does not. Because `BLOCK_INVALID` (`1 << 12`) does not contain the `HEADER_VALID` bit (`1`), the only early-return guard is bypassed, and a header previously marked invalid passes all three subsequent sub-checks before `insert_valid_header` is called, corrupting the peer's best-known-header state and potentially triggering repeated `GetBlocks` cycles for invalid-chain blocks.

## Finding Description
In `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::accept()` reads the block status at line 303 and immediately checks `status.contains(BlockStatus::HEADER_VALID)` at line 304. `BLOCK_INVALID = 1 << 12 = 4096` shares no bits with `HEADER_VALID = 1`, so this guard does not fire for an invalid header.

The three subsequent checks all pass for a header with valid PoW but an invalid body:
- `prev_block_check` (line 324): checks only whether the **parent** is `BLOCK_INVALID`, not the header itself.
- `non_contextual_check` (line 334): calls `HeaderVerifier::verify()`, which validates PoW, block number, epoch, and timestamp — all of which are valid for a well-formed header with an invalid body.
- `version_check` (line 346): checks `version == 0`.

If all three pass, `sync_shared.insert_valid_header(self.peer, self.header)` is called at line 356, which writes the header into `header_map` and updates the peer's best-known header to the invalid chain.

`get_block_status` in `shared/src/shared.rs` checks `block_status_map` first (line 426), then `header_map` (line 429). While `block_status_map` retains the `BLOCK_INVALID` entry during the current session, `insert_valid_header` has already written to `header_map`. If `remove_block_status` is subsequently called for this hash (5 call sites exist in `chain/src/verify.rs`), `get_block_status` will return `HEADER_VALID` from `header_map` instead of `BLOCK_INVALID`, silently re-legitimizing the invalid header within the same session.

Contrast with `compact_block_process.rs` line 259, which has an explicit `BLOCK_INVALID` guard, and `orphan_broker.rs` line 119, which also checks for `BLOCK_INVALID` — confirming the pattern is known and applied elsewhere but missing here.

## Impact Explanation
An attacker mines a single block with a valid header (valid PoW, correct number/epoch/timestamp) but an invalid body. With one mining operation, the attacker can:
1. Send the block via `SendBlock` to many nodes → each node marks it `BLOCK_INVALID`.
2. Immediately re-send the same header via `SendHeaders` → each node calls `insert_valid_header`, updating its peer best-known-header to the invalid chain.
3. Each node may then issue `GetBlocks` for the invalid block; the attacker responds with the same invalid block; nodes reject the body and the cycle can repeat.

This maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, since a single mining operation enables repeated `GetBlocks`/`SendBlock` cycles across many nodes, wasting bandwidth and CPU at scale.

## Likelihood Explanation
The attack is reachable by any unprivileged P2P peer. The only prerequisite is mining one block with a valid header and an invalid body (e.g., a transaction with zero inputs, as used in the existing integration test `test/src/specs/sync/invalid_block.rs`). The same mined block can be replayed to arbitrarily many nodes. The `FIXME` comment in production code confirms developer awareness of the gap. No special privileges, leaked keys, or victim mistakes are required.

## Recommendation
In `HeaderAcceptor::accept()`, add an explicit early-return guard for `BLOCK_INVALID` immediately after reading the status, before the `HEADER_VALID` check:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

if status.contains(BlockStatus::BLOCK_INVALID) {
    state.invalid(Some(ValidationError::InvalidParent));
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing logic
}
```

This mirrors the guard already present in `compact_block_process.rs` (line 259) and `orphan_broker.rs` (line 119).

## Proof of Concept
1. Connect a malicious peer to a CKB node.
2. Mine block `B` at height `N` with a valid header (valid PoW, correct number/epoch/timestamp) but an invalid body (e.g., `TransactionBuilder::default().build()` — zero inputs, same pattern as `test/src/specs/sync/invalid_block.rs` lines 35–41).
3. Send block `B` via `SendBlock`. The node runs `non_contextual_verify` in `chain/src/chain_service.rs` (line 120), which fails; `insert_block_status(B.hash, BLOCK_INVALID)` is called (line 127).
4. Immediately send the header of `B` via `SendHeaders`.
5. `HeaderAcceptor::accept()` is called. `get_block_status(B.hash)` returns `BLOCK_INVALID` (from `block_status_map`). The `HEADER_VALID` guard at line 304 does not fire. `prev_block_check`, `non_contextual_check` (header-only PoW/number/epoch/timestamp), and `version_check` all pass.
6. `insert_valid_header` is called at line 356: `header_map[B.hash]` is set; the peer's best-known header is updated to `B`.
7. The node now believes the peer's best chain tip is the invalid block `B` and may issue `GetBlocks` for it.
8. Repeat steps 4–7 across many nodes using the same mined block `B` to amplify bandwidth waste.
