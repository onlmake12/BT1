Audit Report

## Title
Peer-Supplied `declared_cycles` Used as VM Cap Enables Permanent Known-TX Filter Poisoning — (`tx-pool/src/process.rs`, `sync/src/relayer/transactions_process.rs`)

## Summary
A remote peer relaying a transaction via `RelayTransactions` can supply `declared_cycles = 1`, which passes the only guard (`> max_block_cycles`) and is then used verbatim as the `max_cycles` cap for CKB-VM script verification. The VM halts with `ExceededMaximumCycles`, which is classified as a malformed-tx Script error, causing `is_allowed_relay()` to return `false`. As a result, `TxVerificationResult::Reject` is never sent to the relayer, `remove_from_known_txs` is never called, and the tx hash — already inserted into the known-tx filter before verification — remains there permanently. All subsequent honest relay attempts for the same transaction are silently dropped.

## Finding Description

**Step 1 — Hash inserted before verification** (`sync/src/relayer/transactions_process.rs` L76):
After the only guard (banning `declared_cycles > max_block_cycles`), the tx hash is unconditionally inserted into the known-tx filter. A value of `declared_cycles = 1` trivially passes this check.

**Step 2 — Attacker-controlled cap used for VM verification** (`tx-pool/src/process.rs` L720):
`_process_tx` uses the peer-supplied value directly as `max_cycles`:
```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
```
With `max_cycles = 1` and actual script cost of hundreds of cycles, `verify_rtx` returns `ExceededMaximumCycles`. The `try_or_return_with_snapshot!` macro at L734 returns early — the `DeclaredWrongCycles` check at L736–748 is never reached.

**Step 3 — Misclassification of the error** (`script/src/error.rs` L195–202, `util/types/src/core/tx_pool.rs` L69–85):
`ScriptError::ExceededMaximumCycles` is converted to `ErrorKind::Script` (all non-`Interrupts` script errors take this path). Its display string `"ExceededMaximumCycles: expect cycles <= {0}"` does not contain `ARGV_TOO_LONG_TEXT`, so `is_malformed_from_verification` returns `true`, making `is_malformed_tx()` return `true` and `is_allowed_relay()` return `false`.

**Step 4 — `TxVerificationResult::Reject` never sent** (`tx-pool/src/process.rs` L517–521):
Because `is_allowed_relay()` is `false`, the `send_result_to_relayer(TxVerificationResult::Reject {...})` branch is skipped.

**Step 5 — Hash never removed from filter** (`sync/src/relayer/mod.rs` L673–675):
`remove_from_known_txs` is only called when `TxVerificationResult::Reject` is received. Since it is never sent, the hash persists in the filter for the full entry lifetime.

**Step 6 — Honest peers silently dropped** (`sync/src/relayer/transactions_process.rs` L49–55):
Any subsequent honest peer relaying the same tx with correct cycles is filtered out by `!tx_filter.contains(&tx.hash())`.

**Contrast with over-declaration**: When `declared_cycles > actual`, `verify_rtx` succeeds and the `DeclaredWrongCycles` check fires, producing `Reject::DeclaredWrongCycles` for which `is_allowed_relay()` returns `true` and the hash is correctly cleared. The under-declaration path is entirely unhandled.

## Impact Explanation
An attacker with only a standard P2P connection can suppress arbitrary valid transactions from propagating through targeted nodes. Applied at scale across many nodes and many transactions, this constitutes **CKB network congestion with few costs**, matching the High (10001–15000 points) bounty impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* The attacker pays only the cost of P2P connections and a 3-day ban per IP, while the suppression effect persists for the full filter entry lifetime.

## Likelihood Explanation
The attack requires no privileged access, no key material, and no hashpower. The attacker only needs to know a target transaction hash (observable from any node's mempool broadcast) and send it with `declared_cycles = 1` via `RelayV3`. The 3-day IP ban is the only friction; it is trivially bypassed by rotating IPs or waiting for expiry. The `tx_filter.remove_expired()` call does eventually expire entries, but the attacker can re-inject before expiry to maintain suppression indefinitely.

## Recommendation
1. **Do not use `declared_cycles` as `max_cycles` for VM verification.** Always verify with `consensus.max_block_cycles()` as the actual cap. Use `declared_cycles` only as a hint for queue prioritization.
2. **Treat `ExceededMaximumCycles` caused by a peer-supplied cap as `DeclaredWrongCycles`**: detect when `verify_rtx` fails with `ExceededMaximumCycles` and `declared_cycles` was the binding cap, then return `Reject::DeclaredWrongCycles` so `is_allowed_relay()` returns `true` and the hash is cleared.
3. **Alternatively, mark the tx hash as known only after successful verification**, or unconditionally remove it from the filter on any non-`DeclaredWrongCycles` failure.

## Proof of Concept
1. Observe a valid pending transaction `T` (e.g., standard secp256k1 lock, actual script cost ~537 cycles) on the network.
2. Connect to victim node as a peer via `RelayV3`.
3. Announce `T`'s hash via `RelayTransactionHashes`; wait for `GetRelayTransactions`.
4. Send `RelayTransactions` with `T` and `cycles = 1`.
5. Victim node: inserts `T`'s hash into known-tx filter (`mark_as_known_txs`), runs VM with `max_cycles = 1`, gets `ExceededMaximumCycles` → `Reject::Verification(Script error)`, bans attacker peer, but does **not** send `TxVerificationResult::Reject` → hash stays in filter.
6. Honest peer B now tries to relay `T` with correct `cycles = 537`. The filter check `!tx_filter.contains(&T.hash())` returns `false` → `T` is silently dropped at `transactions_process.rs` L49–55.
7. `T` never enters the victim's tx-pool.
8. Attacker reconnects (after ban expiry or via new IP) and repeats step 4 to refresh the filter entry before it expires.

The existing integration test `DeclaredWrongCyclesChunk` covers only the over-declaration path; the under-declaration path (cycles below actual) produces a different error code that bypasses the re-relay mechanism and is not tested.
