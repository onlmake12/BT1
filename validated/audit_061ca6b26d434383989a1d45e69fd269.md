### Title
Stale Ancestor Stats Emitted in `new_transaction` Notification Due to Read-After-Write on Cloned Entry - (File: tx-pool/src/process.rs)

### Summary
In `_submit_entry`, a `TxEntry` is cloned and passed to `add_pending`/`add_gap`/`add_proposed`, which mutates the clone's ancestor stats (`ancestors_count`, `ancestors_size`, `ancestors_cycles`, `ancestors_fee`) via `check_and_record_ancestors`. The callback (`call_pending`/`call_proposed`) is then invoked with the **original, unmodified entry**, which still carries the initial default ancestor stats (`ancestors_count=1`, `ancestors_size=self.size`, etc.). As a result, every `new_transaction` and `proposed_transaction` notification emitted to RPC subscribers contains incorrect ancestor stats for any child transaction whose parent is already in the pool.

### Finding Description

In `tx-pool/src/process.rs`, the function `_submit_entry` performs the following sequence:

```rust
let (succ, evicts) = match status {
    TxStatus::Fresh => tx_pool.add_pending(entry.clone())?,   // clone is mutated
    TxStatus::Gap   => tx_pool.add_gap(entry.clone())?,
    TxStatus::Proposed => tx_pool.add_proposed(entry.clone())?,
};
if succ {
    match status {
        TxStatus::Fresh    => callbacks.call_pending(&entry),  // original (stale) entry used
        TxStatus::Gap      => callbacks.call_pending(&entry),
        TxStatus::Proposed => callbacks.call_proposed(&entry),
    }
}

### Impact Explanation

Every `new_transaction` and `proposed_transaction` subscription event emitted to RPC subscribers carries incorrect `ancestors_count`, `ancestors_size`, `ancestors_cycles`, and `ancestors_fee` for any child transaction whose parent is already in the pool. The values are always reported as if the transaction has no ancestors (`ancestors_count=1`). External tools — fee estimators, mempool monitors, wallet backends — that rely on these subscription events to compute CPFP (child-pays-for-parent) fee rates or ancestor package sizes will receive systematically wrong data, leading to incorrect fee recommendations and misjudged transaction priority.

### Likelihood Explanation

This is a deterministic, unconditional bug triggered by any RPC caller or P2P relayer that submits a child transaction to the pool while its parent is already present. No special privileges or timing are required. The code path executes on every successful `submit_entry` call for a child transaction.

### Recommendation

Pass the **modified** entry (the one returned from or stored in the pool after `add_entry`) to the callback, rather than the original pre-insertion clone. One approach: have `add_entry` return the mutated entry, or retrieve the entry from the pool by its short ID immediately after insertion and pass that to the callback.

### Proof of Concept

1. Submit parent transaction `P` to the tx-pool via RPC `send_transaction`.
2. Subscribe to `new_transaction` events via RPC `subscribe`.
3. Submit child transaction `C` (spending an output of `P`) via RPC `send_transaction`.
4. Observe the `new_transaction` event for `C`: `ancestors_count` is `1` and `ancestors_size` equals only `C`'s own size, even though `P` is a live ancestor in the pool.
5. Query `get_pool_tx_detail_info` for `C`: `ancestors_count` correctly reflects `2` (P + C), confirming the pool's internal state is accurate but the notification was emitted with stale data.

    /// Update ancestor state for add an entry
    pub fn add_ancestor_weight(&mut self, entry: &TxEntry) {
        self.ancestors_count = self.ancestors_count.saturating_add(1);
        self.ancestors_size = self.ancestors_size.saturating_add(entry.size);
        self.ancestors_cycles = self.ancestors_cycles.saturating_add(entry.cycles);
        self.ancestors_fee = Capacity::shannons(
            self.ancestors_fee
                .as_u64()
                .saturating_add(entry.fee.as_u64()),
        );
    }
