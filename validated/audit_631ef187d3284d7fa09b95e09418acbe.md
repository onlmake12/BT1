### Title
Unbounded `runtime::spawn` via ConnectionSync with Attacker-Populated `pending_delivered` — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

An unprivileged remote peer can exhaust the async runtime and OS file descriptors of a CKB node by (1) flooding `pending_delivered` with spoofed `from` peer IDs via `ConnectionRequest` messages, then (2) sending `ConnectionSync` messages that each bypass the `forward_rate_limiter` (because the limiter key includes the attacker-controlled `from` field) and unconditionally call `runtime::spawn` with no concurrency cap, each spawning a 30-second NAT traversal task that opens up to 24 concurrent TCP sockets.

---

### Finding Description

**Phase 1 — Populate `pending_delivered` (no bound)**

`ConnectionRequestProcess::respond_delivered` inserts an entry into `pending_delivered` for every unique `from` peer ID it receives: [1](#0-0) 

The deduplication guard only blocks re-insertion of the *same* `from` within 2 minutes: [2](#0-1) 

The `forward_rate_limiter` for `ConnectionRequest` is keyed by `(from, to, msg_item_id)`: [3](#0-2) 

Because the attacker supplies a *different* `from` peer ID in each message, every message gets its own rate-limit bucket and passes. The only binding constraint is the per-session `rate_limiter` at 30 msg/s: [4](#0-3) 

`pending_delivered` is only cleaned up every 5 minutes (`CHECK_INTERVAL`) with a 5-minute TTL (`TIMEOUT`): [5](#0-4) 

Result: a single connected peer can accumulate up to 30/s × 300 s = **9,000 entries** in `pending_delivered` before the first cleanup pass.

---

**Phase 2 — Trigger unbounded `runtime::spawn`**

`ConnectionSyncProcess::execute` checks the `forward_rate_limiter` with the same `(from, to, msg_item_id)` key: [6](#0-5) 

Because each message uses a *different* `from` (one of the 9,000 keys in `pending_delivered`), each message gets its own bucket and passes at 1/s. The per-session cap of 30/s is the only real throttle.

When `self_peer_id == content.to` and `content.from` is found in `pending_delivered`, the code unconditionally calls `runtime::spawn` with **no semaphore, no task counter, and no concurrency cap**: [7](#0-6) 

Each spawned task calls `select_ok(tasks)` where `tasks` contains up to `ADDRS_COUNT_LIMIT = 24` concurrent `try_nat_traversal` futures: [8](#0-7) [9](#0-8) 

Each `try_nat_traversal` call loops for up to **30 seconds**, creating a new `TcpSocket` on every ~200 ms retry: [10](#0-9) [11](#0-10)

### Impact Explanation

After 60 seconds of phase-2 flooding at 30 msg/s:
- **1,800 spawned tasks** are alive simultaneously, each holding up to 24 open `TcpSocket` file descriptors.
- Peak concurrent sockets: **~43,200**, far exceeding the default OS `ulimit` (1,024–65,536).
- File-descriptor exhaustion causes all subsequent `socket()` / `accept()` calls to fail, breaking every protocol the node serves (sync, relay, RPC).
- Memory pressure from 1,800 live async tasks (each with a 30-second timer and socket state) can additionally OOM the process.

The impact is a **full CKB node crash / unresponsiveness** affecting all connected peers and all node services.

---

### Likelihood Explanation

- Requires only a standard P2P connection — no privilege, no PoW, no key material.
- Phase 1 (populating `pending_delivered`) is a normal protocol flow; the victim node responds cooperatively.
- Phase 2 messages are well-formed and pass all structural validation.
- The attack is self-contained from a single peer session and completes in under 2 minutes.
- No existing guard (rate limiter, size cap, task semaphore) prevents it.

---

### Recommendation

1. **Cap `pending_delivered` size** — enforce a hard maximum (e.g., 256 entries) and reject new `ConnectionRequest` insertions when the cap is reached.
2. **Bound concurrent NAT traversal tasks** — use a `tokio::sync::Semaphore` (or an atomic counter) before calling `runtime::spawn`; drop the message if the limit is exceeded.
3. **Re-key `forward_rate_limiter` on the *receiving session***, not on the attacker-controlled `from` field, for the `ConnectionSync` path. The current `(from, to, msg_item_id)` key is trivially bypassed by rotating `from`.
4. **Reduce `try_nat_traversal` timeout** or limit the number of retry iterations per task to bound per-task socket lifetime.
