### Title
Requester Contract Can Permanently DoS Provider Keeper via Reverting `_entropyCallback` in Legacy `revealWithCallback` Path - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
In `Entropy.sol`, the `revealWithCallback` function has two execution branches. When a provider has `defaultGasLimit == 0` (the legacy path), the callback to the requester contract is made with a bare direct call — no `try/catch`, no `excessivelySafeCall`. A malicious requester contract that always reverts in `_entropyCallback` causes every `revealWithCallback` attempt to revert, permanently locking the request in storage and permanently DoS-ing the provider's keeper for that sequence number.

### Finding Description

`revealWithCallback` branches on `req.gasLimit10k`:

**New path (`gasLimit10k != 0`):** Uses `excessivelySafeCall` which catches reverts, emits `CallbackFailed`, and sets `callbackStatus = CALLBACK_FAILED`. The request can be retried or recovered. This path is safe.

**Legacy path (`gasLimit10k == 0`):** Calls the requester directly with no error handling:

```solidity
} else {
    address callAddress = req.requester;
    ...
    clearRequest(provider, sequenceNumber);   // ← cleared first
    ...
    if (len != 0) {
        IEntropyConsumer(callAddress)._entropyCallback(  // ← bare call, no try/catch
            sequenceNumber,
            provider,
            randomNumber
        );
    }
``` [1](#0-0) 

Because `clearRequest` is called before the callback, and the callback reverts, the entire transaction reverts — including the `clearRequest`. The request is never cleared. There is no recovery path: no `CALLBACK_FAILED` state, no retry mechanism, no admin escape hatch.

The legacy path is entered whenever `providerInfo.defaultGasLimit == 0`:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;  // opts out of new failure-state flow
``` [2](#0-1) 

The `requestWithCallback` API always passes `gasLimit = 0`, so any request made through it against a provider with `defaultGasLimit == 0` lands in the legacy path:

```solidity
function requestWithCallback(...) public payable override returns (uint64) {
    return requestV2(provider, userContribution, 0);
}
``` [3](#0-2) 

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` confirms this revert behavior is reproducible:

```solidity
vm.expectRevert();
random.revealWithCallback(...);
``` [4](#0-3) 

### Impact Explanation

- The stuck request permanently occupies a storage slot with no on-chain mechanism to clear it.
- The provider's off-chain keeper (Fortuna) will retry `revealWithCallback` indefinitely, burning gas on every attempt with no possibility of success.
- The provider's `currentCommitmentSequenceNumber` cannot advance past the stuck sequence number through normal reveal, degrading hash-chain efficiency for all future requests (more hashes required per request).
- If the provider has `maxNumHashes` configured, future requests may eventually revert with `LastRevealedTooOld` as the gap grows.

Fees are credited at request time so there is no direct ETH lock, but the keeper DoS and storage leak are permanent and unrecoverable without a contract upgrade.

### Likelihood Explanation

Any unprivileged user can:
1. Deploy a contract whose `_entropyCallback` unconditionally reverts.
2. Call `requestWithCallback` against any provider with `defaultGasLimit == 0`, paying only the standard fee.
3. Walk away — the keeper is now permanently stuck on that sequence number.

The attacker loses only the request fee. The cost is low and the attack is trivially repeatable across multiple sequence numbers to amplify the keeper DoS.

### Recommendation

Apply the same `try/catch` (or `excessivelySafeCall`) pattern used in the new path to the legacy path as well. If the callback reverts, emit a failure event and clear the request rather than reverting the entire transaction. Alternatively, deprecate the legacy path entirely by requiring all providers to set a non-zero `defaultGasLimit` before accepting new `requestWithCallback` calls.

### Proof of Concept

```solidity
// Malicious requester
contract MaliciousRequester is IEntropyConsumer {
    address entropy;
    constructor(address _entropy) { entropy = _entropy; }
    function getEntropy() internal view override returns (address) { return entropy; }
    function entropyCallback(uint64, address, bytes32) internal override {
        revert("always revert");
    }
    function attack(address provider, bytes32 userRandom) external payable {
        // provider must have defaultGasLimit == 0
        IEntropy(entropy).requestWithCallback{value: msg.value}(provider, userRandom);
    }
}

// After the provider's keeper calls revealWithCallback:
// → transaction reverts every time
// → request never cleared
// → keeper permanently DoS'd for this sequence number
```

The provider's only partial mitigation is `advanceProviderCommitment` to skip the stuck sequence number, but the storage slot remains occupied and the keeper must be manually configured to skip it — there is no automatic recovery. [5](#0-4) [6](#0-5)

            bytes memory ret;
            uint256 startingGas = gasleft();
            (success, ret) = req.requester.excessivelySafeCall(
                // Warning: the provided gas limit below is only an *upper bound* on the gas provided to the call.
                // At most 63/64ths of the current context's gas will be provided to a call, which may be less
                // than the indicated gas limit. (See CALL opcode docs here https://www.evm.codes/?fork=cancun#f1)
                // Consequently, out-of-gas reverts need to be handled carefully to ensure that the callback
                // was truly provided with a sufficient amount of gas.
                uint256(req.gasLimit10k) * TEN_THOUSAND,

                    sequenceNumber,
                    provider,
                    randomNumber

            req.callbackStatus = EntropyStatusConstants.CALLBACK_NOT_STARTED;
