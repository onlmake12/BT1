### Title
Unbounded Startup Loop Over Persisted Transactions Causes Proportional Node Initialization Delay — (`tx-pool/src/service.rs`)

### Summary
`TxPoolServiceBuilder::start()` synchronously re-verifies every persisted transaction via `load_persisted_data` before the node finishes initializing. Because the loop is proportional to the number of transactions saved in the pool and each iteration performs full CKB-VM script execution, an unprivileged tx-pool submitter can pre-fill the pool before a node restart and cause startup to stall for an arbitrarily long period bounded only by `max_tx_pool_size`.

### Finding Description

During node startup, `TxPoolServiceBuilder::start()` loads all previously persisted transactions from disk and then synchronously re-submits each one for full verification: [1](#0-0) 

```rust
let tx_pool = TxPool::new(self.tx_pool_config, self.snapshot);
let txs = match tx_pool.load_from_file() {
    Ok(txs) => txs,
    ...
};
```

After spawning all async service tasks, `start()` calls `load_persisted_data` **synchronously** — blocking the caller until every transaction is processed: [2](#0-1) 

`load_persisted_data` iterates over the entire loaded set and calls `submit_local_tx` for each entry: [3](#0-2) 

`submit_local_tx` is a synchronous blocking RPC into the tx-pool service that triggers `process_tx`, which in turn calls `_process_tx` — performing full CKB-VM script verification via `verify_rtx` for every transaction: [4](#0-3) 

The persisted data file is written from the entire pool on graceful shutdown: [5](#0-4) 

There is no cap on the number of transactions re-verified during startup beyond the pool's configured `max_tx_pool_size` (default 180 MB). With minimum-size transactions each consuming up to `max_tx_verify_cycles` (default 3.5 billion cycles), the total startup verification work is:

```
startup_cost ≈ N_txs × avg_verification_cycles
```

This is structurally identical to the reported Solidity bug: an initialization routine that iterates over a collection whose size is proportional to prior protocol activity, with no bound on per-iteration cost.

### Impact Explanation

The node's `start()` function does not return until `load_persisted_data` completes. During this window the node is partially unavailable: it cannot relay blocks, respond to sync peers, or submit new transactions. On a heavily loaded node with a full tx-pool, startup can be delayed by minutes to hours. Peers that detect the node as unresponsive will disconnect it, potentially causing it to fall behind the chain tip and requiring a full re-sync.

**Impact: Medium** — node availability is degraded proportional to attacker-controlled pool fill level; no funds are at risk.

### Likelihood Explanation

Any unprivileged RPC caller with access to `send_transaction` can submit transactions up to the pool limit. Nodes restart regularly (software upgrades, crashes, operator maintenance). The attacker only needs to submit enough transactions to fill the pool before a restart — no special privileges, hashpower, or Sybil capability required.

**Likelihood: Medium** — requires pool pre-fill and a subsequent restart, both of which are realistic conditions on a public node.

### Recommendation

1. **Decouple startup from re-verification.** Load persisted transactions into the pool data structure directly (bypassing script execution) and mark them as "pending re-verification." Verify them lazily in the background after the node is fully operational.
2. **Limit re-verification batch size.** Process persisted transactions in bounded batches with yield points so the startup path is not blocked indefinitely.
3. **Skip re-verification for cache hits.** Transactions that have a valid entry in `txs_verify_cache` do not need to be re-executed; reuse the cached `Completed` result directly.

### Proof of Concept

1. Connect to a CKB node with RPC access.
2. Submit transactions up to `max_tx_pool_size` using `send_transaction` RPC — each with a script that consumes close to `max_tx_verify_cycles`.
3. Wait for the node to shut down gracefully (or trigger a restart). The pool is persisted via `save_into_file`.
4. Restart the node. Observe that `TxPoolServiceBuilder::start()` blocks in `load_persisted_data`, re-verifying every transaction sequentially. The node does not accept P2P connections or RPC calls until the loop completes.

Relevant code path:

```

    fn load_persisted_data(&self, txs: Vec<TransactionView>) -> Result<(), AnyError> {
        if !txs.is_empty() {
            info!("Loading persistent tx-pool data, total {} txs", txs.len());
            let mut failed_txs = 0;
            for tx in txs {
                if self.submit_local_tx(tx)?.is_err() {
                    failed_txs += 1;
                }
            }
            if failed_txs == 0 {
                info!("Persistent tx-pool data is loaded");
            } else {
                info!(
                    "Persistent tx-pool data is loaded, {} stale txs are ignored",
                    failed_txs
                );
            }
        }
        Ok(())
    }

    pub(crate) fn load_from_file(&self) -> Result<Vec<TransactionView>, AnyError> {
        let mut persisted_data_file = self.config.persisted_data.clone();
        persisted_data_file.set_extension(format!("v{VERSION}"));

        if persisted_data_file.exists() {
            let mut file = OpenOptions::new()
                .read(true)
                .open(&persisted_data_file)
                .map_err(|err| {
                    let errmsg = format!(
                        "Failed to open the tx-pool persisted data file [{persisted_data_file:?}], cause: {err}"
                    );
                    OtherError::new(errmsg)
                })?;
            let mut buffer = Vec::new();
            file.read_to_end(&mut buffer).map_err(|err| {
                let errmsg = format!(
                    "Failed to read the tx-pool persisted data file [{persisted_data_file:?}], cause: {err}"
                );
                OtherError::new(errmsg)
            })?;

            let persisted_data = TransactionVecReader::from_slice(&buffer)
                .map_err(|err| {
                    let errmsg = format!(
                        "The tx-pool persisted data file [{persisted_data_file:?}] is broken, cause: {err}"
                    );
                    OtherError::new(errmsg)
                })?
                .to_entity();

            Ok(persisted_data
                .into_iter()
                .map(|tx| tx.into_view())
                .collect())
        } else {
            Ok(Vec::new())
        }
    }

    pub(crate) fn save_into_file(&mut self) -> Result<(), AnyError> {
        let mut persisted_data_file = self.config.persisted_data.clone();
        persisted_data_file.set_extension(format!("v{VERSION}"));

        let mut file = OpenOptions::new()
            .create(true)
            .write(true)
            .truncate(true)
            .open(&persisted_data_file)
            .map_err(|err| {
                let errmsg = format!(
                    "Failed to open the tx-pool persisted data file [{persisted_data_file:?}], cause: {err}"
                );
                OtherError::new(errmsg)
            })?;

        let txs = TransactionVec::new_builder()
            .extend(self.drain_all_transactions().iter().map(|tx| tx.data()))
            .build();

        file.write_all(txs.as_slice()).map_err(|err| {
            let errmsg = format!(
                "Failed to write the tx-pool persisted data into file [{persisted_data_file:?}], cause: {err}"
            );
            OtherError::new(errmsg)
        })?;
        file.sync_all().map_err(|err| {
            let errmsg = format!(
                "Failed to sync the tx-pool persisted data file [{persisted_data_file:?}], cause: {err}"
            );
            OtherError::new(errmsg)
        })?;
        Ok(())
    }
