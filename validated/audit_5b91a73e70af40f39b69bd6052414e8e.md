### Title
Wrong Index Used After Filtering Default Publishers in `extractPublisherData` — (`File: governance/pyth_staking_sdk/src/utils/pool.ts`)

---

### Summary

`extractPublisherData` in the Pyth staking SDK filters out default (empty) publisher slots from the `PoolData.publishers` fixed-size array, then uses the **filtered array's sequential index** to access all parallel storage arrays (`delState`, `selfDelState`, `publisherStakeAccounts`, `delegationFees`, `events[*].eventData`). Those parallel arrays are indexed by the **original storage position**, not the filtered position. When gaps exist in the publishers array (i.e., a publisher has been removed, leaving a `PublicKey.default` slot in the middle), every publisher after the gap is read from the wrong slot in every parallel array.

---

### Finding Description

`PoolData` stores publisher data in parallel fixed-size arrays of 1024 entries, all indexed by the same storage position:

```
publishers[1024]            ← publisher pubkeys
delState[1024]              ← delegation states
selfDelState[1024]          ← self-delegation states
publisherStakeAccounts[1024]← publisher stake accounts
delegationFees[1024]        ← delegation fees
events[52].eventData[1024]  ← per-publisher reward event data
```

`extractPublisherData` filters out empty slots and then maps with a fresh sequential `index`:

```typescript
// governance/pyth_staking_sdk/src/utils/pool.ts  lines 12-48
return poolData.publishers
  .filter((publisher) => !publisher.equals(PublicKey.default))  // removes gaps
  .map((publisher, index) => ({                                  // index = filtered position
    ...
    delegationFee: poolData.delegationFees[index] ?? 0n,         // BUG: should be storage index
    selfDelegation: poolData.selfDelState[index]?.totalDelegation ?? 0n,
    stakeAccount: poolData.publisherStakeAccounts[index]...,     // BUG: wrong account
    totalDelegation:
      (poolData.delState[index]?.totalDelegation ?? 0n) + ...,
    apyHistory: poolData.events.filter(...).map((event) => ({
      apy: ... event.eventData[index]?.otherRewardRatio ...      // BUG: wrong event slot
    }))
  }));
```

If the storage layout is `[pub_A, DEFAULT, pub_B]` (pub at slot 0, gap at slot 1, pub at slot 2):

| Publisher | Filtered index | Storage index | Arrays read |
|-----------|---------------|---------------|-------------|
| pub_A | 0 | 0 | Correct |
| pub_B | 1 | 2 | **Wrong — reads slot 1 (the gap's data)** |

The `removePublisher` function exists in the SDK (confirmed in `governance/pyth_staking_sdk/src/pyth-staking-client.ts`), meaning gaps are a reachable state. [1](#0-0) 

---

### Impact Explanation

**Incorrect on-chain transaction construction:** `getAdvanceDelegationRecordInstructions` calls `extractPublisherData` and uses the returned `stakeAccount` field to populate `publisherStakeAccountPositions` and `publisherStakeAccountCustody` in `advanceDelegationRecord` instructions.

```typescript
// governance/pyth_staking_sdk/src/pyth-staking-client.ts  lines 739-756
filteredPublishers.map(({ pubkey, stakeAccount }) =>
  this.integrityPoolProgram.methods
    .advanceDelegationRecord()
    .accountsPartial({
      publisher: pubkey,
      publisherStakeAccountCustody: stakeAccount
        ? getStakeAccountCustodyAddress(stakeAccount) : null,
      publisherStakeAccountPositions: stakeAccount,   // ← wrong account if index is off
      ...
    })
```

When `stakeAccount` belongs to a different publisher (due to the index mismatch), the on-chain program receives a mismatched `publisher` / `publisherStakeAccountPositions` pair and will reject the transaction. This prevents any staking user whose publisher falls after a gap from advancing their delegation record and claiming rewards. [2](#0-1) 

**Incorrect display data:** `delegationFees`, `selfDelState`, `delState`, and `events[*].eventData` are all read from the wrong slot, so APY history, delegation fees, and pool utilization figures shown to users are silently wrong for every publisher after a gap. [3](#0-2) 

---

### Likelihood Explanation

The precondition is at least one publisher removal that leaves a `PublicKey.default` gap in the middle of the `publishers` array. `removePublisher` is an exposed SDK method callable by the pool authority. Once any publisher is removed from a non-tail position, every subsequent publisher is affected. The bug is silent — no error is thrown; wrong data is returned and wrong accounts are silently embedded in transactions. [4](#0-3) 

---

### Recommendation

Preserve the original storage index through the filter by using `map`-before-`filter`:

```typescript
export const extractPublisherData = (poolData: PoolDataAccount): PublisherData => {
  return poolData.publishers
    .map((publisher, storageIndex) => ({ publisher, storageIndex }))
    .filter(({ publisher }) => !publisher.equals(PublicKey.default))
    .map(({ publisher, storageIndex }) => ({
      delegationFee: poolData.delegationFees[storageIndex] ?? 0n,
      selfDelegation: poolData.selfDelState[storageIndex]?.totalDelegation ?? 0n,
      stakeAccount: poolData.publisherStakeAccounts[storageIndex]...,
      totalDelegation:
        (poolData.delState[storageIndex]?.totalDelegation ?? 0n) + ...,
      apyHistory: poolData.events.filter(...).map((event) => ({
        apy: ... event.eventData[storageIndex]?.otherRewardRatio ...
      }))
    }));
};
```

---

### Proof of Concept

1. Pool has three publishers at storage slots 0, 1, 2 with distinct `delegationFees` [100, 200, 300].
2. Publisher at slot 1 is removed → storage becomes `[pub_A, DEFAULT, pub_C]`.
3. `extractPublisherData` filters to `[pub_A, pub_C]` with filtered indices 0 and 1.
4. `pub_C` reads `delegationFees[1]` (= 200, the removed publisher's fee) instead of `delegationFees[2]` (= 300).
5. `pub_C` reads `publisherStakeAccounts[1]` (= removed publisher's stake account or default) instead of `publisherStakeAccounts[2]`.
6. `getAdvanceDelegationRecordInstructions` embeds the wrong `publisherStakeAccountPositions` for `pub_C`.
7. The `advanceDelegationRecord` transaction for `pub_C` is rejected on-chain; `pub_C`'s delegators cannot claim rewards. [5](#0-4) [6](#0-5)
