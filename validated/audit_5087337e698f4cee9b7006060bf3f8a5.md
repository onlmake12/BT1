Audit Report

## Title
Peer Store Permanent Saturation via `check_purge` Eviction Dead Zone — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

`check_purge` contains a logical dead zone: when all network groups in the peer store hold exactly ≤4 entries and all entries are connectable, neither eviction phase removes anything and the function returns `PeerStoreError::EvictionFailed`. An unprivileged attacker can engineer this state via ~1638 short-lived inbound P2P connections carrying crafted listen addresses, permanently blocking `add_addr` for all honest peers and causing progressive node isolation.

## Finding Description

**Phase 1** (lines 341–355) collects non-connectable entries for removal. Fresh entries created by `add_addr` have `last_connected_at_ms=0` and `attempts_count=0` (types.rs L65–76). `is_connectable` returns `true` for these because neither rejection branch fires: `0 >= ADDR_MAX_RETRIES(3)` is false, and `0 >= ADDR_MAX_FAILURES(10)` is false (types.rs L89–104). Phase 1 therefore finds nothing.

**Phase 2** (lines 357–401) is entered only when Phase 1 removes nothing. It groups entries by `/16` network prefix (network_group.rs L26–28: `Group::IP4([bits[0], bits[1]])`), sorts groups by descending size, takes the top half, and for each group applies the guard `if addrs.len() > 4` (line 378). Groups with exactly 4 entries are silently skipped — `None` is returned from `flat_map`, contributing zero candidates. If every group has ≤4 entries, `candidate_peers` is empty and line 400 returns `Err(PeerStoreError::EvictionFailed)`.

**Attack entry point:** `process_listens` (identify/mod.rs L123–150) accepts up to `MAX_ADDRS=10` listen addresses per inbound connection and passes them to `add_remote_listen_addrs` (L472–495), which calls `peer_store.add_addr` for each address with no per-session rate limit. The only filter is `is_reachable` (globally routable IPs), which the attacker satisfies trivially by advertising real public IP ranges — no ownership required.

**Saturation state:** `ADDR_COUNT_LIMIT=16384` (mod.rs L26). Filling 4096 distinct `/16` groups with exactly 4 entries each (4096 × 4 = 16384) reaches the limit. Every subsequent `add_addr` call triggers `check_purge`, which returns `EvictionFailed`. The error is logged but swallowed in `add_remote_listen_addrs` (L490–492) and in the Discovery protocol's `add_new_addrs`. The node can no longer record any new peer address.

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** After saturation, the victim node's peer discovery via both Identify and Discovery protocols is permanently blocked. The node cannot replenish its peer pool after natural churn. Once all existing connections are lost, the node becomes fully isolated from the network and cannot recover without a restart. A coordinated attacker targeting multiple nodes simultaneously could degrade network-wide peer discovery.

## Likelihood Explanation

The attack requires approximately 1638 short-lived inbound TCP connections (16384 ÷ 10 addresses per connection). The attacker does not need to hold connections open — connect, send Identify with 10 crafted globally-routable listen addresses, disconnect, repeat. Addresses survive disconnection. No special privileges, no PoW, no key material. A single machine with one IP can execute this by cycling connections. The injected listen addresses need only be globally routable (e.g., `1.x.y.z`, `2.x.y.z`, …) — the node never verifies reachability before storing them. The effect is permanent until node restart.

## Recommendation

1. **Remove the `> 4` hard floor** in Phase 2 or replace it with unconditional eviction: always evict from the largest group regardless of its size, ensuring at least one candidate is always produced.
2. **Unconditionally evict** the oldest/least-recently-tried entry (preferring entries with `last_connected_at_ms=0`) when no other candidate exists, rather than returning `EvictionFailed`.
3. **Rate-limit per-session address insertions** in `add_remote_listen_addrs` — cap the total number of addresses accepted from a single session across its lifetime.
4. **Prefer evicting never-connected entries** (`last_connected_at_ms=0`) before entries with a real connection history in both eviction phases.

## Proof of Concept

```rust
// Fill 4096 distinct /16 groups with exactly 4 entries each = 16384 total.
// All entries: last_connected_at_ms=0, attempts_count=0 → is_connectable=true.
let mut peer_store = PeerStore::default();
let mut count = 0u32;
'outer: for a in 1u8..=255 {
    for b in 0u8..=255 {
        for port in 1u16..=4 {
            let addr: Multiaddr = format!(
                "/ip4/{}.{}.0.1/tcp/{}/p2p/{}",
                a, b, port, PeerId::random().to_base58()
            ).parse().unwrap();
            peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
            count += 1;
            if count == 16384 { break 'outer; }
        }
    }
}
// Peer store is now at ADDR_COUNT_LIMIT with 4096 groups × 4 entries each.
// All entries connectable. Phase 1 finds nothing. Phase 2: 4 > 4 is false for all groups.
let new_addr: Multiaddr = "/ip4/8.8.8.8/tcp/42/p2p/<new_peer_id>"
    .parse().unwrap();
let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
assert!(matches!(result, Err(e) if e.to_string().contains("EvictionFailed")));
```
