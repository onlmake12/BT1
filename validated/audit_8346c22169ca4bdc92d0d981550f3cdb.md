Audit Report

## Title
Miner-Controlled Epoch Tail Block Timestamp Corrupts Difficulty Adjustment via Zero Epoch Duration - (File: `traits/src/epoch_provider.rs`)

## Summary
A miner who wins the epoch tail block can legally set its timestamp to just above the 37-block median, which may be lower than the epoch start block's timestamp. The epoch duration is computed with `saturating_sub`, silently producing zero. This zero is clamped to 1 second in `next_epoch_ext`, inflating the computed hash rate by orders of magnitude. The TAU dampening filter caps the effect at 2× per epoch, but all nodes accept the resulting corrupted epoch parameters as valid consensus state, permanently degrading chain throughput.

## Finding Description

**Root cause — `traits/src/epoch_provider.rs` lines 36–40**

`epoch_duration_in_milliseconds` is computed as:

```rust
let epoch_duration_in_milliseconds = header.timestamp().saturating_sub(
    self.get_block_header(&last_block_hash_in_previous_epoch)
        .expect("stored block header")
        .timestamp(),
);
``` [1](#0-0) 

If the tail block's timestamp ≤ the epoch start block's timestamp, `saturating_sub` returns `0`. No lower-bound validation is applied before this value is returned in `BlockEpoch::TailBlock`. [2](#0-1) 

**Propagation — `spec/src/consensus.rs` lines 853–860**

The zero flows into `next_epoch_ext` (the non-`permanent_difficulty` path, active on mainnet):

```rust
let last_epoch_duration = U256::from(cmp::max(
    epoch_duration_in_milliseconds / MILLISECONDS_IN_A_SECOND,
    1,
));
``` [3](#0-2) 

Zero is clamped to 1 second rather than rejected. The hash rate is then:

```rust
let last_epoch_hash_rate = last_difficulty
    * (epoch.length() + epoch_uncles_count)
    / &last_epoch_duration;
``` [4](#0-3) 

With `last_epoch_duration = 1`, the computed hash rate is `difficulty × (epoch_length + uncles)` — roughly 14,400× the real value for a 1800-block epoch at 8-second intervals.

**TAU dampening — `spec/src/consensus.rs` lines 789–792 and 802–803**

`bounding_hash_rate` caps the result at `previous_hash_rate × TAU` (2×): [5](#0-4) 

`bounding_epoch_length` caps the next epoch length at `last_epoch_length × TAU` (2×): [6](#0-5) 

Both caps are accepted as valid consensus state by all nodes.

**Why the timestamp is valid — `verification/src/header_verifier.rs` lines 76–94**

`TimestampVerifier` only enforces two constraints: the timestamp must be strictly greater than the median of the previous 37 blocks, and no more than 15 seconds in the future: [7](#0-6) 

For a 1800-block epoch, the epoch start block is ~1799 blocks before the tail block — far outside the 37-block median window. A miner can legally set the tail block's timestamp to `median_of_37_blocks + 1 ms`, which passes `TimestampVerifier` but may be lower than the epoch start block's timestamp, triggering the zero-duration path. There is no verification of `epoch_duration_in_milliseconds` against any minimum threshold anywhere in the block verification pipeline.

## Impact Explanation

**Critical — Vulnerabilities which could easily cause consensus deviation and damage CKB economy.**

Each triggered epoch causes the next epoch difficulty and length to be up to 2× higher than the previous epoch. These are accepted by all nodes as valid consensus state — there is no fork, but the chain's throughput is permanently degraded. Compounded over successive epochs, difficulty and epoch length grow exponentially, slowing block production for all participants. The constants confirm the scale: `MAX_EPOCH_LENGTH = DEFAULT_EPOCH_DURATION_TARGET / MIN_BLOCK_INTERVAL = 1800` blocks. [8](#0-7) 

## Likelihood Explanation

Any miner who successfully mines an epoch tail block can trigger this. No special privilege or majority hash power is required — only a valid PoW solution for the tail block with a crafted timestamp. Any pool or solo miner with non-trivial hash power has a realistic probability of mining epoch tail blocks over time. The attack is repeatable: each epoch tail block is an independent opportunity, and the attacker need not control a majority of hash power.

## Recommendation

1. **Validate `epoch_duration_in_milliseconds` against a minimum floor** before using it in the difficulty adjustment. If the computed duration is below `MIN_BLOCK_INTERVAL × epoch_length / 2` (in milliseconds), reject the block or clamp to the floor rather than silently using 1 second.
2. **Add an explicit check in `get_block_epoch`** (`traits/src/epoch_provider.rs`) that the tail block's timestamp is strictly greater than the epoch start block's timestamp, returning an error if not.
3. **Add epoch duration verification to block contextual verification** (analogous to `TimestampVerifier`), so that a block producing an implausible epoch duration is rejected at the consensus layer before `next_epoch_ext` is called.

## Proof of Concept

1. Standard epoch length is 1800 blocks (`MAX_EPOCH_LENGTH = DEFAULT_EPOCH_DURATION_TARGET / MIN_BLOCK_INTERVAL`). [8](#0-7) 
2. A miner targets the epoch tail block (block `start + length - 1`). They set `header.timestamp = block_median_time(parent, 37) + 1` — the minimum valid timestamp per `TimestampVerifier`. On a normally-running chain, this value is lower than the epoch start block's timestamp (which was set ~4 hours earlier in real time). [9](#0-8) 
3. In `get_block_epoch` (`traits/src/epoch_provider.rs` line 36), `epoch_duration_in_milliseconds = tail_ts.saturating_sub(start_ts) = 0`. [1](#0-0) 
4. In `next_epoch_ext` (`spec/src/consensus.rs` line 853), `last_epoch_duration = max(0 / 1000, 1) = 1`. [3](#0-2) 
5. `last_epoch_hash_rate = difficulty × (1800 + uncles) / 1` — ~14,400× the real hash rate. [4](#0-3) 
6. `bounding_hash_rate` clamps to `previous_hash_rate × 2`; `bounding_epoch_length` clamps to `last_epoch_length × 2`. [10](#0-9) 
7. All nodes accept the next epoch with 2× difficulty and 2× length as valid consensus state. Repeating this over successive epochs compounds the effect exponentially.
