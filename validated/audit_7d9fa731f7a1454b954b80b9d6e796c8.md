### Title
Underflow in `executeCallback` Fee Accounting Permanently Locks User Funds - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, the `executeCallback` function computes the provider's credit as `(req.fee + msg.value) - pythFee`. Because `req.fee` was stored at request time as `msg.value_at_request - _state.pythFeeInWei`, and the actual Pyth oracle fee (`pythFee`) is computed dynamically at callback time from `updateData`, an attacker can craft `updateData` with extra price feeds to inflate `pythFee` above `req.fee + msg.value`, causing an arithmetic underflow revert. Since `clearRequest` is called after this line, the revert prevents the request from ever being cleared, permanently locking the user's deposited funds.

### Finding Description

At request time in `requestPriceUpdatesWithCallback`, the stored fee is:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;  // Pyth's flat fee credited immediately
``` [1](#0-0) 

At callback time in `executeCallback`, the actual Pyth oracle fee is fetched dynamically:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
// ...
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The `getUpdateFee` function in `Pyth.sol` charges `singleUpdateFeeInWei * totalNumUpdates`, where `totalNumUpdates` is counted from all feeds present in `updateData`, not just the requested `priceIds`. [3](#0-2) 

The `priceIds` verification in `executeCallback` only checks the first 8 bytes of each requested feed prefix — it does **not** restrict the total number of feeds in `updateData`. [4](#0-3) 

This creates a gap: the contract's ETH balance for a given request is `req.fee + _state.pythFeeInWei + msg.value`, but the subtraction only accounts for `req.fee + msg.value`. An attacker can craft `updateData` with extra feeds such that:

```
req.fee < pythFee <= req.fee + _state.pythFeeInWei
```

Under this condition:
- `parsePriceFeedUpdates{value: pythFee}(...)` **succeeds** (contract has enough ETH)
- `(req.fee + msg.value) - pythFee` **underflows** → revert

Since `clearRequest(sequenceNumber)` is called on line 164, after the underflowing line 162, the revert prevents the request from ever being cleared. [5](#0-4) 

The `Request` struct stores `fee` as `uint96`, and `EchoState` stores `accruedFeesInWei` as `uint128`, both operating under Solidity 0.8's checked arithmetic. [6](#0-5) 

### Impact Explanation

**Impact: High.** User funds paid at request time (`req.fee + _state.pythFeeInWei`) are permanently locked in the contract. The request can never be fulfilled or cancelled because `executeCallback` always reverts for the targeted sequence number. There is no recovery path — no admin function exists to cancel individual requests or refund users.

### Likelihood Explanation

**Likelihood: Medium.** The attack requires no privileged access. Any unprivileged caller of `executeCallback` (which is permissionless after the exclusivity period) can supply crafted `updateData`. The attacker only needs to include enough extra price feeds in `updateData` to push `pythFee` into the vulnerable range `(req.fee, req.fee + _state.pythFeeInWei]`. The attacker sends `msg.value = 0`. The attack is also reachable inadvertently if Pyth governance raises `singleUpdateFeeInWei` between request and fulfillment.

### Recommendation

The fee accounting must include `_state.pythFeeInWei` in the available amount, since that ETH is still in the contract's balance at callback time:

```solidity
// Option 1: Include pythFeeInWei in the available pool
uint256 totalAvailable = uint256(req.fee) + _state.pythFeeInWei + msg.value;
uint256 providerAmount = totalAvailable > pythFee ? totalAvailable - pythFee : 0;
_state.providers[providerToCredit].accruedFeesInWei += SafeCast.toUint128(providerAmount);
// Also adjust _state.accruedFeesInWei to not double-count pythFeeInWei
```

Or, do not pre-credit `_state.accruedFeesInWei` at request time and instead settle all fee accounting atomically in `executeCallback`. Additionally, add a minimum `msg.value` check in `executeCallback` to ensure the caller covers any shortfall between `req.fee` and the actual `pythFee`.

### Proof of Concept

1. Echo contract is deployed with `_state.pythFeeInWei = 1000 wei`, `singleUpdateFeeInWei = 100 wei`.
2. User calls `requestPriceUpdatesWithCallback` with `msg.value = 1500 wei` (1000 Pyth fee + 500 provider fee), requesting 1 price feed.
   - `req.fee = 500 wei`
   - `_state.accruedFeesInWei += 1000`
3. Attacker calls `executeCallback(providerToCredit, sequenceNumber, updateData, priceIds)` with `msg.value = 0`, where `updateData` contains 6 extra price feeds (total 7 feeds × 100 wei = 700 wei = `pythFee`).
4. `pythFee = 700 wei`. Contract balance = 1500 wei ≥ 700 wei → `parsePriceFeedUpdates` succeeds.
5. Line 162: `(500 + 0) - 700` → **arithmetic underflow → revert**.
6. `clearRequest` is never reached. The request remains active forever.
7. User's 1500 wei is permanently locked. No recovery path exists.

        bytes[] calldata updateData
    ) public view override returns (uint feeAmount) {
        uint totalNumUpdates = 0;
        for (uint i = 0; i < updateData.length; i++) {
            if (
                updateData[i].length > 4 &&
                UnsafeCalldataBytesLib.toUint32(updateData[i], 0) ==
                ACCUMULATOR_MAGIC
            ) {
                (
                    uint offset,
                    UpdateType updateType
                ) = extractUpdateTypeFromAccumulatorHeader(updateData[i]);
                if (updateType != UpdateType.WormholeMerkle) {
                    revert PythErrors.InvalidUpdateData();
                }
                totalNumUpdates += parseWormholeMerkleHeaderNumUpdates(
                    updateData[i],
                    offset
                );
            } else {
                revert PythErrors.InvalidUpdateData();
            }
        }
        return getTotalFee(totalNumUpdates);
    }
