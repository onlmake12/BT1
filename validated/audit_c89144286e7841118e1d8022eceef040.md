Audit Report

## Title
Unvalidated Attacker-Controlled `listen_addrs` with Rotatable `from` PeerId Enables Unbounded Async Task Spawning and Node Crash — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary

An attacker with a standard P2P connection can exhaust the victim node's async runtime by sending crafted `ConnectionRequest` messages with arbitrary `listen_addrs` keyed to synthetic `from` peer IDs, then following up with matching `ConnectionSync` messages. Because `pending_delivered` is keyed by the message-level `content.from` field rather than the actual session peer ID, and because all per-key cooldowns are keyed the same way, the attacker can rotate fresh random `from` values to bypass every rate guard and sustain unbounded `try_nat_traversal` future spawning, crashing the node.

## Finding Description

**Root cause 1 — No private/loopback/link-local filtering in `respond_delivered()`:** [1](#0-0) 

The filter accepts any TCP address containing `Ip4` or `Ip6` protocol components. Private ranges (`10.x`, `192.168.x`, `172.16.x`), loopback (`127.x`), link-local (`169.254.x`), and cloud metadata addresses all pass through. Up to 24 addresses per message are accepted (`ADDRS_COUNT_LIMIT = 24`). [2](#0-1) 

**Root cause 2 — `pending_delivered` keyed by message-level `content.from`, not session peer ID:** [3](#0-2) 

`ConnectionRequestProcess` has `self.peer: PeerIndex` (the actual session), but it is never used to verify or replace `from_peer_id`. An attacker can supply any syntactically valid `PeerId` bytes as `from`.

**Root cause 3 — `HOLE_PUNCHING_INTERVAL` cooldown keyed by `from_peer_id`:** [4](#0-3) 

The 2-minute cooldown is per `from_peer_id`. Rotating fresh random `from` values trivially bypasses it.

**Root cause 4 — `forward_rate_limiter` keyed by `(from, to, msg_item_id)`:** [5](#0-4) [6](#0-5) 

Both `ConnectionRequest` and `ConnectionSync` share the same bypass: rotating `content.from` creates a fresh rate-limit bucket each time.

**Root cause 5 — `ConnectionSyncProcess` has no session peer ID field and performs no session-to-`from` binding:** [7](#0-6) 

There is no `peer` field. When `content.to == self_peer_id` and `content.route` is empty, the process looks up `pending_delivered` by `content.from` and spawns tasks unconditionally.

**Root cause 6 — Unbounded `runtime::spawn` per `ConnectionSync`:** [8](#0-7) [9](#0-8) 

One Tokio task is spawned per `ConnectionSync` message. Inside it, `select_ok(tasks)` polls all 24 `try_nat_traversal` futures concurrently. Each future runs a retry loop for up to 30 seconds. [10](#0-9) 

**Binding constraint — per-session rate limiter at 30 req/sec:** [11](#0-10) [12](#0-11) 

This is the only guard that cannot be bypassed by rotating `from`. It allows 30 `ConnectionRequest` + 30 `ConnectionSync` per second from one session.

**Exploit arithmetic:**
- 30 `ConnectionRequest`/sec × 24 addresses → 30 new `pending_delivered` entries/sec
- 30 `ConnectionSync`/sec → 30 spawned Tokio tasks/sec, each running 24 concurrent `try_nat_traversal` futures via `select_ok`
- Each future lives up to 30 seconds → **21,600 concurrent `try_nat_traversal` futures at steady state** from a single attacker connection
- Each future creates TCP sockets every ~200ms, exhausting file descriptors and socket resources

**Secondary vector — unbounded `pending_delivered` map growth:** [13](#0-12) 

Cleanup runs every 5 minutes with a 5-minute TIMEOUT. At 30 inserts/sec, up to 9,000 entries accumulate per window, each holding up to 24 `Multiaddr` values — an additional memory exhaustion path.

## Impact Explanation

**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

Sustaining 21,600 concurrent `try_nat_traversal` futures from a single connection exhausts the Tokio runtime's task capacity and the OS file-descriptor limit (each future holds open TCP sockets). The node process becomes unresponsive or OOMs. Multiple attacker connections multiply the effect linearly. The victim node is taken fully offline, losing participation in block propagation and transaction relay.

## Likelihood Explanation

Preconditions are minimal: a standard P2P connection (no special privileges) and the victim's peer ID (publicly discoverable via DHT/peer exchange). The attack requires only generating fresh random `PeerId` bytes — syntactic validity is the only requirement, as confirmed by `PeerId::from_bytes` at: [14](#0-13) 

The two-message sequence is trivial to automate, repeatable indefinitely, and effective from a single connection.

## Recommendation

1. **Reject private/loopback/link-local addresses** in `respond_delivered()` before inserting into `pending_delivered`. In the `TransportType::Tcp` branch, add checks on `Protocol::Ip4(addr)` for `addr.is_private() || addr.is_loopback() || addr.is_link_local()` and equivalent IPv6 checks (`is_loopback()`, `is_unicast_link_local()`).

2. **Key `pending_delivered` by the actual session peer ID**, not `content.from`. In `ConnectionRequestProcess`, resolve the session peer ID from `self.peer` via the peer registry and use it as the map key instead of `from_peer_id`.

3. **Pass the session peer ID into `ConnectionSyncProcess`** and assert it matches the `content.from` lookup key before retrieving from `pending_delivered`.

4. **Cap concurrent `try_nat_traversal` tasks** with a bounded semaphore or a fixed-size task pool to limit blast radius even if other guards are bypassed.

5. **Bound `pending_delivered` map size** with an explicit capacity limit to prevent secondary memory exhaustion.

## Proof of Concept

```
1. Attacker establishes a standard P2P connection to victim (victim_peer_id known).

2. For i in 1..30 per second (within per-session rate limit):
   fresh_id[i] = PeerId::random()
   Send ConnectionRequest {
     from: fresh_id[i],
     to:   victim_peer_id,
     max_hops: 6,
     route: [],
     listen_addrs: [
       /ip4/1.2.3.4/tcp/8114,   // 24 routable addresses
       /ip4/1.2.3.5/tcp/8114,
       ... (24 total)
     ]
   }
   // victim stores 24 addrs in pending_delivered[fresh_id[i]]

3. For i in 1..30 per second:
   Send ConnectionSync {
     from: fresh_id[i],
     to:   victim_peer_id,
     route: []
   }
   // victim spawns 1 Tokio task running 24 concurrent try_nat_traversal futures

4. After 30 seconds:
   - 900 spawned Tokio tasks active
   - 21,600 concurrent try_nat_traversal futures running
   - Each future holds TCP sockets, retrying every ~200ms
   - Node FD limit exhausted → process crashes or becomes unresponsive

Verification: monitor victim's /proc/<pid>/fd count and memory; both grow
linearly at ~720 new futures/sec until crash. A unit test can mock
try_nat_traversal with a 30-second sleep, send N message pairs, and assert
the spawned task count exceeds a threshold.
```
