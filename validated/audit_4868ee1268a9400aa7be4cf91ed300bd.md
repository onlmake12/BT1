### Title
Excess ETH Sent to Entropy Request Functions Is Silently Absorbed Into Pyth Fee Pool — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any user who calls `request()`, `requestWithCallback()`, or any `requestV2()` variant with `msg.value` exceeding the required fee permanently loses the excess ETH. Rather than being refunded, the surplus is silently credited to `_state.accruedPythFeesInWei` — Pyth's own treasury — with no mechanism for the user to recover it.

---

### Finding Description

In `Entropy.sol`, the internal `requestHelper` function handles fee accounting for all randomness request entry points. The fee split logic is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);   // = providerFee + pythFeeInWei
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

`getFeeV2` is defined as `getProviderFee(provider, gasLimit) + _state.pythFeeInWei`. [2](#0-1) 

When `msg.value == requiredFee`, the math is correct: `msg.value - providerFee == pythFeeInWei`. But when `msg.value > requiredFee` (i.e., the user overpays), the entire surplus is added to `accruedPythFeesInWei` instead of being returned. The correct Pyth portion (`pythFeeInWei`) should be credited, and the remainder (`msg.value - requiredFee`) should be refunded to `msg.sender`.

This behavior is present across all public entry points that route through `requestHelper`:

- `request(address, bytes32, bool)` — documented: *"excess value is not refunded"*
- `requestWithCallback(address, bytes32)` — documented: *"excess value is not refunded"*
- `requestV2()`, `requestV2(uint32)`, `requestV2(address, uint32)`, `requestV2(address, bytes32, uint32)` [3](#0-2) [4](#0-3) [5](#0-4) 

The interface NatSpec explicitly acknowledges this behavior:

> *"further note that excess value is \*not\* refunded to the caller"* [6](#0-5) 

This confirms the behavior is present in production code, not just a documentation artifact.

---

### Impact Explanation

Any ETH sent above `getFeeV2(...)` is permanently transferred to Pyth's `accruedPythFeesInWei` pool. The user has no mechanism to recover it. The loss is proportional to the overpayment amount and is irreversible once the transaction is confirmed.

**Impact class:** Direct, permanent loss of user funds (ETH) to the protocol treasury.

---

### Likelihood Explanation

Several realistic scenarios cause overpayment:

1. **Fee changes between estimation and submission.** The documentation explicitly warns fees change over time. A user who calls `getFeeV2()` off-chain and then submits with a buffer (e.g., `fee * 110 / 100`) to avoid revert will lose the buffer.

2. **Integration contracts forwarding `msg.value` directly.** A common pattern (shown in the official docs) is `entropy.requestV2{value: fee}()` where `fee` is computed on-chain. If the calling contract passes `msg.value` directly without exact trimming, any surplus is lost.

3. **Provider fee increase between `getFeeV2()` call and `requestV2()` submission.** A provider can call `setProviderFee()` at any time. A user who estimated the fee before the increase will overpay if they added a safety margin. [7](#0-6) 

---

### Recommendation

Replace the current fee credit line with an explicit split that credits only the correct Pyth portion and refunds the remainder:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += _state.pythFeeInWei;  // credit only the exact Pyth fee

uint256 excess = msg.value - requiredFee;
if (excess > 0) {
    (bool sent, ) = payable(msg.sender).call{value: excess}("");
    require(sent, "refund failed");
}
```

Alternatively, revert if `msg.value != requiredFee` to prevent any accidental overpayment.

---

### Proof of Concept

```solidity
// Setup: provider fee = 0.001 ETH, pythFee = 0.0001 ETH, requiredFee = 0.0011 ETH
// User sends 0.002 ETH (e.g., added a safety buffer)

entropy.requestV2{value: 0.002 ether}();

// Inside requestHelper:
//   providerFee = 0.001 ETH  → credited to provider ✓
//   accruedPythFeesInWei += (0.002 - 0.001) = 0.001 ETH
//   But pythFeeInWei = 0.0001 ETH
//   → Pyth absorbs 0.001 ETH instead of 0.0001 ETH
//   → User loses 0.0009 ETH (the excess) permanently
``` [1](#0-0) [2](#0-1)
