Audit Report

## Title
Unauthenticated `ConnectionRequestDelivered` Relay Allows Targeted Peer Flooding via Bypassed Rate Limiter — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary
`ConnectionRequestDeliveredProcess::execute()` unconditionally forwards messages to any attacker-specified `route.last()` peer without verifying a prior `ConnectionRequest` was ever relayed for that `(from, to)` pair. The `forward_rate_limiter` intended to throttle this path is trivially bypassed because its key is derived from attacker-controlled `from`/`to` fields in the message body. The only binding constraint is the outer `rate_limiter` at 30 req/s per session, allowing an attacker with one relay connection to flood any peer connected to that relay with 30 HolePunching messages/second, scalable linearly with relay connections.

## Finding Description
In `execute()`, after the `forward_rate_limiter` check, when `content.route` is non-empty, the relay immediately forwards without any state check:

```rust
match content.route.last() {
    Some(next_peer_id) => self.forward_delivered(next_peer_id).await,
``` [1](#0-0) 

`forward_delivered` resolves the attacker-supplied `PeerId` to a live session and sends the message unconditionally: [2](#0-1) 

The `inflight_requests` and `pending_delivered` state are only consulted in the `route.is_empty()` (`None`) branch where `self_peer_id == content.from` — never when acting as a relay: [3](#0-2) 

The `forward_rate_limiter` check at L134–145 is keyed by `(content.from, content.to, msg_item_id)`: [4](#0-3) 

Since `from` and `to` are free fields in the message body, the attacker simply varies them per request, making this limiter ineffective. The only binding constraint is the outer `rate_limiter` keyed by `(session_id, msg_item_id)` at 30 req/s: [5](#0-4) 

The `forward_rate_limiter` is configured at 1 req/s per `(from, to)` pair, but this is entirely attacker-controlled: [6](#0-5) 

## Impact Explanation
An attacker with a single P2P connection to relay R can direct R to forward up to 30 `HolePunching` messages/second to any peer V connected to R. With N relay connections, the rate scales to 30N messages/second targeting V. Each message forces V to execute `DeliverdContent::try_from` (parsing), peer registry lookups, and rate-limiter key checks before returning `StatusCode::Ignore`. At scale across multiple relays, this constitutes targeted network congestion of specific CKB peers at very low attacker cost (one TCP connection per relay, no keys or hashpower required). This matches the **High** bounty impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The exploit requires only a standard P2P connection to any relay running the HolePunching protocol — no special privileges, cryptographic keys, or hashpower. The victim's `PeerId` is public information on the CKB P2P network. The `forward_rate_limiter` bypass is trivial: the attacker simply increments a counter in the `from` or `to` field per message. The attack is repeatable, persistent, and scalable with the number of relay connections the attacker maintains.

## Recommendation
Before forwarding a `ConnectionRequestDelivered`, the relay should verify it previously forwarded a `ConnectionRequest` for the same `(from, to)` pair:

- In `ConnectionRequestProcess::forward_message`, record `(from, to)` with a timestamp in a new `forwarded_requests: HashMap<(PeerId, PeerId), u64>` field on `HolePunching`.
- In `ConnectionRequestDeliveredProcess::execute`, when `route.last()` is `Some`, check that `forwarded_requests` contains a recent entry for `(content.from, content.to)` before calling `forward_delivered`. Return `StatusCode::Ignore` if no such entry exists.
- Remove the entry after forwarding the delivered message (or on timeout) to prevent replay.

This enforces the invariant that `ConnectionRequestDelivered` can only travel back along a route established by a legitimate `ConnectionRequest`.

## Proof of Concept
1. Attacker A establishes a standard P2P connection to relay R.
2. A observes (via peer exchange or network scanning) that victim V is connected to R.
3. A crafts `ConnectionRequestDelivered` messages with:
   - `route = [V.peer_id]`
   - `from = random_peer_id_i` (different each iteration, bypasses `forward_rate_limiter`)
   - `to = random_peer_id_i'`
   - `listen_addrs = [any valid multiaddr embedding to's peer_id]`
   - `sync_route = []`
4. A sends these at ~30/s to R (capped by outer `rate_limiter` at `(session_id, msg_item_id)`).
5. R executes `ConnectionRequestDeliveredProcess::execute()`: `route.last() = V.peer_id` → `forward_delivered(V.peer_id)` → `peer_registry.get_key_by_peer_id(V.peer_id)` → `send_message_to(V.session_id, HolePunching, message)`.
6. V receives 30 HolePunching messages/second from R, each requiring parse and processing, without R ever having forwarded a `ConnectionRequest` for any of these `(from, to)` pairs.
7. Repeat with additional relay connections to scale the flood linearly.

            Some(next_peer) => {
