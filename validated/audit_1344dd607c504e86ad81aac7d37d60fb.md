### Title
Any Signer Can Remove or Reassign Any Publisher's Stake Account — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

### Summary

The `set_publisher_stake_account` instruction in the Integrity Pool program accepts an unconstrained `signer` account with no role or ownership check. Any arbitrary transaction sender can call this instruction to remove or replace any registered publisher's stake account association, disrupting self-staking reward accounting for that publisher.

### Finding Description

The `set_publisher_stake_account` instruction's account list contains a `signer` field that carries no `relations`, `constraint`, or ownership check tying it to the `publisher` account being modified:

```json
{
  "name": "signer",
  "signer": true
},
{
  "docs": ["CHECK : The publisher will be checked against data in the pool_data"],
  "name": "publisher"
},
{
  "name": "pool_data",
  "relations": ["pool_config"],
  "writable": true
},
{
  "name": "new_stake_account_positions_option",
  "optional": true
},
{
  "name": "current_stake_account_positions_option",
  "optional": true
}
``` [1](#0-0) 

The `publisher` account is only validated to be a registered publisher inside `pool_data` (per the `CHECK` comment), but the `signer` is never verified to be that publisher or any privileged authority. Compare this to `create_slash_event`, which correctly enforces `"relations": ["pool_config"]` on `reward_program_authority`: [2](#0-1) 

The SDK client exposes `removePublisherStakeAccount` which passes `undefined` (i.e., `None`) for `newStakeAccountPositions`, effectively clearing the publisher's stake account: [3](#0-2) 

Any transaction sender can invoke this path for any publisher without owning or controlling that publisher's key.

### Impact Explanation

Removing a publisher's stake account from `pool_data` causes:

1. The publisher's self-staking positions to be invisible to the reward calculation in `advance_delegation_record`, zeroing out their self-reward ratio for affected epochs.
2. Incorrect `delta_delegation` accounting in `pool_data`, corrupting the global delegation state used to compute all delegator rewards.
3. Permanent loss of accrued but unclaimed rewards for the targeted publisher, since `advance_delegation_record` reads the publisher's custody account only when `publisher_stake_account_positions` is present. [4](#0-3) 

**Impact: 4/5** — Direct loss of staking rewards for a targeted publisher; corrupted pool accounting affects all delegators of that publisher.

### Likelihood Explanation

The instruction is on-chain and callable by any funded Solana wallet. No special key, governance vote, or privileged role is required. The attacker only needs to know the target publisher's public key (publicly visible on-chain). The attack is cheap (one transaction) and repeatable.

**Likelihood: 4/5** — Trivially reachable by any unprivileged actor.

### Recommendation

Add an ownership constraint requiring `signer == publisher` (i.e., the publisher must sign their own stake account reassignment), or add a `relations: ["pool_config"]` constraint tying `signer` to a designated authority stored in `pool_config`. The analogous `create_slash_event` instruction already demonstrates the correct pattern with `reward_program_authority`. [5](#0-4) 

### Proof of Concept

1. Observe that publisher `P` has a registered stake account `S` in `pool_data`.
2. Attacker constructs a `set_publisher_stake_account` transaction with:
   - `signer` = attacker's own keypair (any funded wallet)
   - `publisher` = `P`
   - `pool_data` = the live pool data account
   - `new_stake_account_positions_option` = omitted (None)
   - `current_stake_account_positions_option` = `S`
3. Transaction succeeds; `pool_data` now records no stake account for publisher `P`.
4. All subsequent `advance_delegation_record` calls for `P`'s delegators skip the publisher self-stake custody transfer, zeroing `P`'s self-reward ratio and corrupting `delta_delegation` in `pool_data`.
5. Publisher `P` and their delegators permanently lose rewards for all epochs after the attack. [6](#0-5)

```typescript
  async setPublisherStakeAccount(
    publisher: PublicKey,
    stakeAccountPositions: PublicKey,
    newStakeAccountPositions: PublicKey | undefined,
  ) {
    const instruction = await this.integrityPoolProgram.methods
      .setPublisherStakeAccount()
      .accounts({
        currentStakeAccountPositionsOption: stakeAccountPositions,
        // eslint-disable-next-line unicorn/no-null
        newStakeAccountPositionsOption: newStakeAccountPositions ?? null,
        publisher,
      })
      .instruction();

    await sendTransaction([instruction], this.connection, this.wallet);
    return;
  }
```
