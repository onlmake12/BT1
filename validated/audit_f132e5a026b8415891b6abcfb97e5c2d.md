### Title
SDK `PreparedMessage` Directive Hardcodes `Commitment::Finalized`, Permanently Freezing Lamports for Messages Prepared with `Commitment::Confirmed` — (`target_chains/solana/programs/core-bridge/src/sdk/publish_message.rs`)

### Summary
The `PublishMessageDirective::PreparedMessage` arm of `publish_message` passes a hardcoded `Commitment::Finalized` to the `post_message` CPI. However, `handle_post_prepared_message` unconditionally validates that the passed commitment matches the commitment encoded in the message account at initialization time. Any message prepared with `Commitment::Confirmed` can never be successfully published, and because `close_message_v1` only accepts `Writing` status, the message account lamports are permanently irrecoverable.

### Finding Description
In `publish_message.rs` lines 153–157, the `PreparedMessage` directive constructs `PostMessageArgs` with `commitment: Commitment::Finalized` (hardcoded) and the comment `// not checked`. This assumption is wrong. In `post_message/mod.rs` lines 281–285, `handle_post_prepared_message` calls `require_eq!(commitment, Commitment::try_from(info.consistency_level).unwrap(), CoreBridgeError::InvalidInstructionArgument)`. If the message was initialized via `init_message_v1` with `Commitment::Confirmed` (consistency_level = 1), the CPI passes 32 but the account stores 1, causing an unconditional revert. After `finalize_message_v1` sets the status to `ReadyForPublishing`, the account is in a terminal state: it cannot be published (CPI always reverts) and cannot be closed (`close_message_v1` requires `Writing` status per lines 145–148 of `posted_message_v1/mod.rs`).

### Impact Explanation
Lamports locked in the message account are permanently frozen. There is no recovery path: the only closing instruction (`close_message_v1`) is gated on `Writing` status, and the only publishing instruction (`post_message`) always fails for `Confirmed` messages when called via the `PreparedMessage` directive. This matches the "permanent freezing of unclaimed yield/funds" impact category.

### Likelihood Explanation
Any integrating program that:
1. Calls `prepare_message` (or `init_message_v1`) with `Commitment::Confirmed`, and
2. Subsequently calls `publish_message` with `PublishMessageDirective::PreparedMessage`

will trigger this bug on every invocation. The `PreparedMessage` directive is explicitly documented for use after message preparation, making this a realistic integration pattern. An unprivileged relayer who can invoke the integrating program's publish instruction can repeatedly trigger the failure, but the freeze is caused by the first attempt.

### Recommendation
Replace the hardcoded `Commitment::Finalized` in the `PreparedMessage` arm with the actual commitment read from the prepared message account before constructing the CPI args. Specifically, deserialize `PostedMessageV1Info` from the message account, extract `consistency_level`, convert it to `Commitment`, and pass that value. Alternatively, remove the commitment check in `handle_post_prepared_message` for the prepared-message path (since the commitment is already encoded in the account and cannot be changed), which aligns with the original developer intent expressed by the `// not checked` comment.

### Proof of Concept
1. Integrating program calls `init_message_v1` with `commitment: Commitment::Confirmed` → `consistency_level = 1` written to account, status = `Writing`.
2. Integrating program calls `write_message_v1` and `finalize_message_v1` → status = `ReadyForPublishing`.
3. Relayer (or anyone) invokes the integrating program's publish instruction, which calls `publish_message(..., PublishMessageDirective::PreparedMessage)`.
4. SDK constructs CPI with `commitment: Commitment::Finalized` (= 32).
5. `handle_post_prepared_message` executes `require_eq!(Finalized, Confirmed)` → fails with `InvalidInstructionArgument`. Transaction reverts.
6. Message account remains in `ReadyForPublishing` state.
7. Call `close_message_v1` → fails with `NotInWritingStatus`.
8. Lamports are permanently frozen. No recovery path exists.
