### Title
Missing Zero-Amount Guard Before ETH Transfer Allows Spurious Withdrawals and Event Pollution - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `withdraw()` and `withdrawAsFeeManager()` functions in `Entropy.sol` check that the caller's accrued balance is **at least** the requested `amount`, but never verify that `amount > 0`. Any registered provider (or their fee manager) can call either function with `amount = 0`, causing a zero-value ETH call to be dispatched and two spurious `Withdrawal` events to be emitted per invocation.

---

### Finding Description

`Entropy.withdraw()` enforces only:

```solidity
require(
    providerInfo.accruedFeesInWei >= amount,
    "Insufficient balance"
);
```

Because `uint128(0) >= uint128(0)` is always `true`, the guard is trivially bypassed when `amount == 0`. The function then executes:

```solidity
providerInfo.accruedFeesInWei -= amount;          // no-op
(bool sent, ) = msg.sender.call{value: amount}(""); // zero-value call
require(sent, "withdrawal to msg.sender failed");
emit EntropyEvents.Withdrawal(msg.sender, msg.sender, amount);
emit EntropyEventsV2.Withdrawal(msg.sender, msg.sender, amount, bytes(""));
```

The identical pattern exists in `withdrawAsFeeManager()`:

```solidity
require(
    providerInfo.accruedFeesInWei >= amount,
    "Insufficient balance"
);
providerInfo.accruedFeesInWei -= amount;
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "withdrawal to msg.sender failed");
emit EntropyEvents.Withdrawal(provider, msg.sender, amount);
emit EntropyEventsV2.Withdrawal(provider, msg.sender, amount, bytes(""));
``` [1](#0-0) [2](#0-1) 

---

### Impact Explanation

1. **Spurious on-chain events**: Each zero-amount call emits two `Withdrawal` events (`EntropyEvents.Withdrawal` and `EntropyEventsV2.Withdrawal`) with `amount = 0`. Off-chain indexers, dashboards, and alerting systems that track provider withdrawals will record phantom withdrawal activity, degrading the integrity of fee-accounting observability.

2. **Zero-value external call**: `msg.sender.call{value: 0}("")` is dispatched unconditionally. If the caller is a contract whose `receive()` or `fallback()` performs state changes or emits its own events on any incoming call (including zero-value), those side-effects are triggered without any actual fund movement.

3. **Gas griefing**: An adversary who has registered as a provider (permissionless via `register()`) can spam `withdraw(0)` in a tight loop, bloating the event log and wasting gas for anyone replaying or indexing the chain.

---

### Likelihood Explanation

Registration as a provider is **fully permissionless** — anyone can call `register()` with any parameters. Once registered, calling `withdraw(0)` requires no special privilege, no fee, and no prior balance. The entry path is therefore reachable by any unprivileged transaction sender at negligible cost. [3](#0-2) 

---

### Recommendation

Add an explicit non-zero guard at the top of both `withdraw()` and `withdrawAsFeeManager()`:

```solidity
// In withdraw()
if (amount == 0) revert EntropyErrors.AssertionFailure();

// In withdrawAsFeeManager()
if (amount == 0) revert EntropyErrors.AssertionFailure();
```

This mirrors the existing `chainLength == 0` guard already present in `register()` and eliminates the zero-value transfer path entirely. [4](#0-3) 

---

### Proof of Concept

1. Call `register(0, bytes32(0), "", 1, "")` from any EOA — this registers `msg.sender` as a provider with zero accrued fees.
2. Call `withdraw(0)` from the same EOA.
3. Observe:
   - Transaction succeeds (no revert).
   - `EntropyEvents.Withdrawal(caller, caller, 0)` is emitted.
   - `EntropyEventsV2.Withdrawal(caller, caller, 0, "")` is emitted.
   - A zero-value ETH call was dispatched to `msg.sender`.
4. Repeat step 2 arbitrarily many times to flood the event log with zero-amount withdrawal records.

    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
