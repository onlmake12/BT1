Audit Report

## Title
Missing Rate Limiting on `GetBlockFilterCheckPoints` Enables Peer-Triggered DB Read Exhaustion — (`sync/src/filter/get_block_filter_check_points_process.rs`)

## Summary
The `BlockFilter` protocol handler processes `GetBlockFilterCheckPoints` messages with no per-peer rate limiting. Each request triggers up to 4,000 synchronous RocksDB reads (2 reads × 2,000 loop iterations). The `Relayer` protocol applies an identical class of protection via a keyed `RateLimiter<(PeerIndex, u32)>`, but this was never applied to the filter protocol, leaving it open to low-cost DB read flooding that degrades sync for all peers.

## Finding Description
In `sync/src/filter/get_block_filter_check_points_process.rs`, `BATCH_SIZE = 2000` and `CHECK_POINT_INTERVAL = 2000` (lines 9–10). The `execute()` method (lines 43–56) loops up to 2,000 times; each iteration calls `get_block_hash(block_number)` and then `get_block_filter_hash(&block_hash)` — two synchronous RocksDB point-lookups — for a maximum of 4,000 DB reads per request. The loop only exits early if a hash is missing; on a fully synced node with complete filter data, all 2,000 iterations complete.

`filter/mod.rs` `try_process()` (lines 50–54) dispatches directly to `GetBlockFilterCheckPointsProcess::execute()` with no rate check. The only guard in `received()` (lines 128–143) bans peers that send unparseable bytes; a well-formed `GetBlockFilterCheckPoints { start_number: 0 }` is never banned and never throttled.

By contrast, `relayer/mod.rs` defines `RateLimiter<(PeerIndex, u32)>` (lines 63–67, field at line 81) and checks it before every non-CompactBlock message (lines 116–123), returning `StatusCode::TooManyRequests` on excess. A grep across the entire `sync/src/filter/` directory for `rate_limit`, `RateLimiter`, and `governor` returns zero matches.

## Impact Explanation
Matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker opening multiple P2P connections and sending `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop forces the node to execute hundreds of thousands of RocksDB point-lookups per second from this path alone. This saturates DB read capacity, increases latency for block/header sync DB operations, and causes sync degradation visible to all connected peers — constituting network congestion at very low attacker cost.

## Likelihood Explanation
The attack requires only a valid P2P connection to a node with the filter protocol enabled. The message contains a single `BlockNumber` field, costs the attacker essentially nothing to send repeatedly, and is indistinguishable from a legitimate light client request. No privilege escalation, leaked keys, or victim mistakes are required. The filter protocol is enabled on nodes serving light clients, a standard mainnet deployment configuration.

## Recommendation
1. Add a `RateLimiter<(PeerIndex, u32)>` field to the `BlockFilter` struct, mirroring the pattern in `sync/src/relayer/mod.rs` (lines 63–82), and check it in `try_process()` before dispatching to `GetBlockFilterCheckPointsProcess::execute()`.
2. Optionally reduce `BATCH_SIZE` from 2,000 to a smaller value (e.g., 100–200) to reduce per-request DB cost, though this alone does not prevent flooding.
3. Ban peers that exceed a configurable request-per-second threshold for filter messages, consistent with the relayer's `TooManyRequests` status path.

## Proof of Concept
On a node with filter data built for a mature chain:
1. Open 50 TCP connections to the node's filter protocol port.
2. From each connection, send `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop.
3. Monitor RocksDB read IOPS via `rocksdb.stats` or `/proc` metrics.
4. Simultaneously send `GetHeaders`/`GetBlocks` sync messages from a separate legitimate peer and measure response latency.
5. Expected: DB IOPS spike proportional to the number of concurrent senders; sync message response latency increases measurably, confirming DB read saturation from the filter protocol path.
