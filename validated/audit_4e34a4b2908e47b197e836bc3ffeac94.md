### Title
Silent `u128 → u64` Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Capacity — (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes an intermediate `withdraw_counted_capacity` as a `u128` but converts it to `u64` with a silent truncating `as u64` cast. When the product exceeds `u64::MAX`, the result wraps silently to a much smaller value, producing an incorrect (too small) maximum withdrawal capacity. This is the direct CKB analog of the Solidity `startingPrice - priceDeduction` underflow: a growing quantity eventually exceeds a fixed bound, and the arithmetic silently produces a wrong result instead of erroring.

---

### Finding Description

In `util/dao/src/lib.rs` lines 152–156:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
```

The cast `withdraw_counted_capacity as u64` silently truncates the lower 64 bits when the value exceeds `u64::MAX`. There is no checked conversion, no error return, and no panic — the wrong value propagates silently into `Capacity::shannons(...)`.

This is structurally identical to the reported Solidity bug: a quantity that grows over time (`withdrawing_ar / deposit_ar` ratio increases as the accumulation rate rises) eventually causes an arithmetic operation to produce an incorrect result. In Solidity the result was an underflow revert; here the result is a silently wrong capacity value.

The inconsistency is explicit within the same file. The analogous computation in `secondary_block_reward` (lines 202–204) uses the correct checked pattern:

```rust
let reward128 = u128::from(target_g2.as_u64()) * u128::from(target_parent_u.as_u64())
    / u128::from(target_parent_c.as_u64());
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
```

And in `dao_field_with_current_epoch` (lines 242–245):

```rust
let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
    / u128::from(parent_c.as_u64());
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
```

Both use `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. The `calculate_maximum_withdraw` function does not. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

The truncated `withdraw_capacity` flows into `transaction_maximum_withdraw`, which is consumed by `withdrawed_interests`:

```rust
maximum_withdraws
    .safe_sub(input_capacities)
    .map_err(Into::into)
``` [4](#0-3) 

If `withdraw_counted_capacity` wraps to a value smaller than `occupied_capacity`, `safe_add` at line 156 may still succeed (with a wrong small value). Then `maximum_withdraws` (the sum of truncated values) can fall below `input_capacities` (the original cell capacities), causing `safe_sub` to return `DaoError::Overflow`.

This error propagates through `dao_field_with_current_epoch` → `DaoHeaderVerifier::verify()`:

```rust
.dao_field(self.resolved.iter().map(AsRef::as_ref), self.parent)
.map_err(|e| { ... e })?;
``` [5](#0-4) 

`DaoHeaderVerifier` is called unconditionally (unless `disable_daoheader` switch is set) during `ContextualBlockVerifier::verify()`: [6](#0-5) 

**Result**: A block containing a large DAO withdrawal — one that the DAO type script (running in CKB-VM) correctly accepts — is rejected by the Rust node's contextual verifier. This is a **consensus split**: nodes that process the block accept it at the script level but reject it at the DAO field level, while the block producer's own calculation (which also uses `calculate_maximum_withdraw`) produces the same wrong value and embeds it in the block header's DAO field. The mismatch between the CKB-VM script result and the Rust node's DAO field check causes the block to be permanently orphaned on nodes that hit the truncation.

Alternatively, if the truncated value is still ≥ `input_capacities`, `current_s` is computed too large (underreporting withdrawn interest), silently corrupting the DAO savings accumulator for all subsequent blocks.

---

### Likelihood Explanation

The trigger condition is `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`.

- `counted_capacity = output_capacity − occupied_capacity`. A deposit of ~184 million CKB (1.84 × 10^18 shannons, about 0.9 × u64::MAX) is large but not impossible for an institutional depositor.
- `withdrawing_ar / deposit_ar` must exceed ~1.0 for the product to overflow. The AR starts at 10^16 and grows with each block's secondary issuance. A ratio of 2× (AR doubling) would require many decades at current issuance rates, but a ratio of 1.001× with a near-maximum deposit is sufficient.
- The condition becomes reachable much sooner for deposits close to u64::MAX shannons combined with any non-trivial AR growth.

Likelihood is **low in the near term** but **increases monotonically** as the chain ages — exactly the same temporal dynamic as the Dutch auction's `priceDeduction` growing past `startingPrice`.

The attacker-controlled entry path is: submit a DAO deposit transaction with a very large capacity cell, wait for sufficient AR growth, then submit a DAO withdrawal transaction. No privileged access is required; any CKB holder can execute this.

---

### Recommendation

Replace the silent cast with the same checked pattern used elsewhere in the file:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This is consistent with `secondary_block_reward` (line 204) and `dao_field_with_current_epoch` (line 245), and ensures the error is surfaced as `DaoError::Overflow` rather than silently producing a wrong value. [7](#0-6)

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` (lines 296–350) tests only the `safe_add` overflow path (where `withdraw_capacity > u64::MAX` after adding `occupied_capacity`). It does **not** test the silent truncation path where `withdraw_counted_capacity` itself exceeds `u64::MAX` before the cast. [8](#0-7) 

A test that demonstrates the silent truncation:

```rust
// deposit_ar = 10^16 (genesis AR)
// withdrawing_ar = 2 * 10^16 (AR has doubled — long-term scenario)
// counted_capacity = 10^19 shannons (close to u64::MAX ≈ 1.844 × 10^19)
//
// withdraw_counted_capacity = 10^19 * (2 * 10^16) / 10^16 = 2 * 10^19
// 2 * 10^19 > u64::MAX (1.844 * 10^19)
//
// `as u64` truncates: (2 * 10^19) % 2^64 ≈ 1.55 * 10^18  (a small fraction)
// Capacity::shannons(1.55e18).safe_add(occupied_capacity) SUCCEEDS with wrong value
// maximum_withdraw is now far below input_capacity → safe_sub underflows → DaoError::Overflow
// DaoHeaderVerifier rejects the block → consensus split

let output = CellOutput::new_builder()
    .capacity(Capacity::shannons(10_000_000_000_000_000_000u64))  // ~10^19 shannons
    .build();
let deposit_ar:     u64 = 10_000_000_000_000_000;   // 10^16
let withdrawing_ar: u64 = 20_000_000_000_000_000;   // 2 × 10^16

// withdraw_counted_capacity (u128) = (10^19 - occupied) * 2×10^16 / 10^16
//                                  ≈ 2 × 10^19  >  u64::MAX
// `as u64` silently wraps → wrong small value → downstream safe_sub fails
```

The fix (`u64::try_from`) would return `DaoError::Overflow` at the correct location, consistent with the rest of the codebase.

fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
}
