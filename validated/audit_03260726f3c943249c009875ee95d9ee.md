### Title
Publisher Removal Index Mismatch Causes Incorrect Delegation Accounting and Broken Reward Claims - (`File: governance/pyth_staking_sdk/src/utils/pool.ts`)

---

### Summary

The `extractPublisherData` function in the Pyth staking SDK filters out removed publishers (those set to `PublicKey.default`) from the `publishers` array, but then uses the **post-filter index** to access parallel on-chain arrays (`delState`, `selfDelState`, `delegationFees`, `publisherStakeAccounts`, `numSlashEvents`). These parallel arrays are indexed by the **original** position in the `publishers` array. After any publisher removal, all publishers occupying slots after the removed slot will have their delegation state, APY history, delegation fees, and stake accounts read from the wrong index. This directly corrupts reward-claiming instruction construction.

---

### Finding Description

`PoolData` stores publisher data in parallel fixed-size arrays of 1024 entries. When a publisher is removed, their slot in `publishers[]` is set to `PublicKey.default`, but the corresponding entries in `delState[]`, `selfDelState[]`, `delegationFees[]`, `publisherStakeAccounts[]`, and `numSlashEvents[]` remain at their original indices.

`extractPublisherData` does:

```typescript
return poolData.publishers
    .filter((publisher) => !publisher.equals(PublicKey.default))
    .map((publisher, index) => ({
      ...
      delegationFee: poolData.delegationFees[index] ?? 0n,
      stakeAccount: poolData.publisherStakeAccounts[index] === undefined || ...
      totalDelegation:
        (poolData.delState[index]?.totalDelegation ?? 0n) +
        (poolData.selfDelState[index]?.totalDelegation ?? 0n),
```

After `.filter()`, `index` in `.map()` is the position in the **filtered** array. But `delState[index]`, `selfDelState[index]`, `delegationFees[index]`, and `publisherStakeAccounts[index]` are all indexed by the **original** slot position.

**Example:** If `publishers[3]` is removed (set to `PublicKey.default`):
- Filtered array: `[pub0, pub1, pub2, pub4, pub5, ...]`
- `pub4` is now at filtered index `3`
- `delState[3]` is read → but this is the removed publisher's delegation state, not `pub4`'s
- `publisherStakeAccounts[3]` is read → wrong stake account used for `pub4` [1](#0-0) 

This corrupted `stakeAccount` value is then passed directly into `advanceDelegationRecord` instruction construction:

```typescript
.accountsPartial({
    publisherStakeAccountCustody: stakeAccount
      ? getStakeAccountCustodyAddress(stakeAccount)
      : null,
    publisherStakeAccountPositions: stakeAccount,
    ...
})
``` [2](#0-1) 

Additionally, `sumDelegations` in both the staking app and insights app sums over the **entire** raw `delState` and `selfDelState` arrays (all 1024 entries), including stale entries for removed publishers whose delegation state was never zeroed:

```typescript
totalStaked:
  sumDelegations(poolData.delState) + sumDelegations(poolData.selfDelState),
``` [3](#0-2) [4](#0-3) 

This is the direct analog to the Derby `totalAllocatedTokens` bug: removing a publisher zeroes their `publishers[]` slot but does not zero their `delState[]` entry, so aggregate totals remain inflated.

---

### Impact Explanation

1. **Broken reward claiming**: `getClaimableRewards` and `advanceDelegationRecord` use the wrong `publisherStakeAccountPositions` and `publisherStakeAccountCustody` for all publishers occupying slots after any removed publisher. The on-chain `advanceDelegationRecord` instruction will either fail (account mismatch) or process rewards against the wrong stake account, causing stakers to be unable to claim rewards.

2. **Inflated total staked display**: `sumDelegations` over raw `delState` includes stale amounts from removed publishers, permanently inflating the reported "OIS Total Staked" figure.

3. **Incorrect APY history and pool utilization**: `event.eventData[index]` and `delState[index]` are read at wrong indices, producing incorrect APY history and pool utilization for all publishers after a removed slot. [5](#0-4) 

---

### Likelihood Explanation

The `publishers` array has 1024 slots and the filter for `PublicKey.default` is explicitly present, confirming that publisher removal is a supported operation. Any governance action that removes a publisher from the pool triggers this bug for all stakers delegated to publishers occupying higher-indexed slots. The `setPublisherStakeAccount` / `removePublisherStakeAccount` path in the SDK confirms this is a live operation. [6](#0-5) 

---

### Recommendation

Replace the chained `.filter().map()` with a single `.reduce()` or use `flatMap` that preserves the original index when accessing parallel arrays:

```typescript
return poolData.publishers
    .map((publisher, originalIndex) => ({ publisher, originalIndex }))
    .filter(({ publisher }) => !publisher.equals(PublicKey.default))
    .map(({ publisher, originalIndex }) => ({
      delegationFee: poolData.delegationFees[originalIndex] ?? 0n,
      stakeAccount: poolData.publisherStakeAccounts[originalIndex] === undefined || ...
      totalDelegation:
        (poolData.delState[originalIndex]?.totalDelegation ?? 0n) +
        (poolData.selfDelState[originalIndex]?.totalDelegation ?? 0n),
      ...
    }));
```

Additionally, `sumDelegations` should only sum over active publisher slots (non-default), not the entire raw array.

---

### Proof of Concept

1. Publisher pool has 3 publishers at slots 0, 1, 2 with non-zero `delState`.
2. Governance removes publisher at slot 1 (sets `publishers[1] = PublicKey.default`). `delState[1]` is **not** zeroed.
3. `extractPublisherData` returns `[pub0, pub2]` with filtered indices `[0, 1]`.
4. For `pub2` (filtered index 1): `delState[1]` is read → returns removed publisher's stale delegation state.
5. `publisherStakeAccounts[1]` is read → returns removed publisher's stake account.
6. `advanceDelegationRecord` is called for `pub2` with the wrong `publisherStakeAccountPositions` → instruction fails or processes against wrong account.
7. Staker delegated to `pub2` cannot claim rewards.
8. `sumDelegations(poolData.delState)` includes `delState[1]` (stale, non-zero) → total staked is permanently inflated. [7](#0-6) [8](#0-7)

export const extractPublisherData = (
  poolData: PoolDataAccount,
): PublisherData => {
  return poolData.publishers
    .filter((publisher) => !publisher.equals(PublicKey.default))
    .map((publisher, index) => ({
      apyHistory: poolData.events
        .filter((event) => event.epoch > 0n)
        .map((event) => ({
          apy:
            convertEpochYieldToApy(
              (event.y * (event.eventData[index]?.otherRewardRatio ?? 0n)) /
                FRACTION_PRECISION_N,
            ) *
            computeDelegatorRewardPercentage(
              poolData.delegationFees[index] ?? 0n,
            ),
          epoch: event.epoch,
          selfApy: convertEpochYieldToApy(
            (event.y * (event.eventData[index]?.selfRewardRatio ?? 0n)) /
              FRACTION_PRECISION_N,
          ),
        }))
        .sort((a, b) => Number(a.epoch) - Number(b.epoch)),
      delegationFee: poolData.delegationFees[index] ?? 0n,
      pubkey: publisher,
      selfDelegation: poolData.selfDelState[index]?.totalDelegation ?? 0n,
      selfDelegationDelta: poolData.selfDelState[index]?.deltaDelegation ?? 0n,
      stakeAccount:
        poolData.publisherStakeAccounts[index] === undefined ||
        poolData.publisherStakeAccounts[index].equals(PublicKey.default)
          ? null // eslint-disable-line unicorn/no-null
          : poolData.publisherStakeAccounts[index],
      totalDelegation:
        (poolData.delState[index]?.totalDelegation ?? 0n) +
        (poolData.selfDelState[index]?.totalDelegation ?? 0n),
      totalDelegationDelta:
        (poolData.delState[index]?.deltaDelegation ?? 0n) +
        (poolData.selfDelState[index]?.deltaDelegation ?? 0n),
    }));
