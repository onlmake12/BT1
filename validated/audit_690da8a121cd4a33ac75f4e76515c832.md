### Title
User Funds Permanently Locked When External Pyth Oracle Call Fails in `Echo.executeCallback` — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo.executeCallback` function makes an external call to `IPyth.parsePriceFeedUpdates`. If this call permanently reverts — for example, because price data for the requested `publishTime` is unavailable — the user's fee paid via `requestPriceUpdatesWithCallback` is permanently locked in the Echo contract. There is no cancel, refund, or emergency-exit function for users.

---

### Finding Description

In `Echo.executeCallback`, the function calls `pyth.parsePriceFeedUpdates{value: pythFee}(...)` as an external call before any state changes that would allow recovery: [1](#0-0) 

If `parsePriceFeedUpdates` reverts (e.g., no Pyth price data exists for the exact `publishTime` stored in the request), the entire `executeCallback` transaction reverts and the request remains active in storage. However, there is no user-callable function to cancel the request and recover the fee. The code itself acknowledges this risk in an inline TODO: [2](#0-1) 

The `requestPriceUpdatesWithCallback` function stores the user's fee in `req.fee` and the Pyth protocol fee in `_state.accruedFeesInWei`: [3](#0-2) 

The `publishTime` is constrained to at most 60 seconds in the future at request time: [4](#0-3) 

This means the price data window is narrow. If Pyth has an outage or the data for that specific timestamp is pruned or unavailable, `parsePriceFeedUpdates` will always revert for that request. The only withdrawal functions in the contract are admin-only (`withdrawFees`) and provider-only (`withdrawAsFeeManager`): [5](#0-4) 

There is no user-facing cancel or refund path in `IEcho`: [6](#0-5) 

---

### Impact Explanation

A user who calls `requestPriceUpdatesWithCallback` and pays the required fee has their ETH locked in the Echo contract. If the Pyth oracle call in `executeCallback` permanently reverts for that request's `publishTime`, the request can never be cleared and the user's fee can never be recovered. This is a direct loss of user funds with no on-chain remedy.

---

### Likelihood Explanation

Pyth network outages, data gaps, or price feed unavailability for a specific narrow time window are realistic operational scenarios. The 60-second `publishTime` constraint creates a tight window: if the Pyth network fails to publish data during that window, the request becomes permanently unserviceable. Any user who submitted a request during such a window loses their fee.

---

### Recommendation

Add a user-callable `cancelRequest` function that allows the original requester to cancel their pending request and receive a refund of `req.fee` after a configurable timeout (e.g., if the request has not been fulfilled within N seconds of `publishTime`). This mirrors the emergency-exit pattern recommended in the reference report and is consistent with how Entropy's `revealWithCallback` handles stuck requests via `CALLBACK_FAILED` state recovery.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback(provider, block.timestamp, priceIds, gasLimit)` paying the required fee. The fee is stored in `req.fee`.
2. Pyth network experiences an outage during that timestamp window; no price data is published for `publishTime`.
3. Provider calls `executeCallback(...)` with the best available update data.
4. `pyth.parsePriceFeedUpdates{value: pythFee}(updateData, priceIds, publishTime, publishTime)` reverts because no price data exists for the exact `publishTime`.
5. The entire `executeCallback` transaction reverts; the request remains active in `_state.requests`.
6. No subsequent call can ever succeed for this request because the Pyth data for that timestamp is gone.
7. The user has no function to call to cancel the request or recover their fee — it is permanently locked in the Echo contract. [7](#0-6)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);

        // TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
        // a better solution would be a doubly-linked list of active requests.
        // After successful callback, update firstUnfulfilledSeq if needed
        while (
            _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
            !isActive(findRequest(_state.firstUnfulfilledSeq))
        ) {
            _state.firstUnfulfilledSeq++;
        }

        try
            IEchoConsumer(req.requester)._echoCallback{
                gas: req.callbackGasLimit
            }(sequenceNumber, priceFeeds)
        {
            // Callback succeeded
            emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
        } catch Error(string memory reason) {
            // Explicit revert/require
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                reason
            );
        } catch {
            // Out of gas or other low-level errors
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                "low-level error (possibly out of gas)"
            );
        }
    }
