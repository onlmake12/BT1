### Title
Off-by-One in RFC0044 MMR Activation Check Causes First Block of Activation Epoch to Omit Extension — (`tx-pool/src/block_assembler/mod.rs`)

---

### Summary

In `build_extension`, the block assembler checks whether RFC0044 (MMR chain root extension) is active using the **tip block's epoch number** rather than the **candidate block's epoch number**. Because the candidate block is always one block ahead of the tip, this is an off-by-one error: at the epoch boundary where RFC0044 activates, the first block of the activation epoch is assembled without the required MMR extension.

---

### Finding Description

`build_extension` in `tx-pool/src/block_assembler/mod.rs` is called to decide whether to include the MMR chain root in the block extension field:

```rust
pub(crate) fn build_extension(snapshot: &Snapshot) -> Result<Option<packed::Bytes>, AnyError> {
    let tip_header = snapshot.tip_header();
    // The use of the epoch number of the tip here leads to an off-by-one bug,
    // so be careful, it needs to be preserved for consistency reasons and not fixed directly.
    let mmr_activate = snapshot
        .consensus()
        .rfc0044_active(tip_header.epoch().number());
``` [1](#0-0) 

The candidate block being assembled is `tip.number() + 1`. When the tip is the **last block of epoch N**, the candidate block is the **first block of epoch N+1**. If RFC0044 activates at epoch N+1, the check `rfc0044_active(N)` returns `false` (not yet active), so no extension is included — even though the candidate block is the first block of the activation epoch and should carry the MMR root.

The code comment explicitly acknowledges this: *"The use of the epoch number of the tip here leads to an off-by-one bug, so be careful, it needs to be preserved for consistency reasons and not fixed directly."* [2](#0-1) 

The phrase "preserved for consistency reasons" indicates the contextual block verifier (`verification/contextual/src/contextual_block_verifier.rs`) carries the same off-by-one when validating the extension field, making both sides consistently wrong: neither the assembler nor the verifier expects the extension in the first block of the activation epoch. [3](#0-2) 

This is structurally identical to the Pico M-5 bug: a counter (`current_chunk`) is incremented **before** the boundary check, so the check always operates on the already-advanced value and the intended guard is bypassed. Here, the epoch number used for the activation guard is the tip's epoch (one behind the candidate), so the guard is bypassed for the first block of the activation epoch.

---

### Impact Explanation

RFC0044 introduces the MMR chain root extension to support light clients. Due to this off-by-one, the **first block of the RFC0044 activation epoch** is assembled and accepted by the network **without the MMR extension**, even though the protocol intends it to be present from the first block of the activation epoch onward.

Consequences:
- The MMR chain is missing a commitment for the first block of the activation epoch.
- Light clients that rely on the MMR extension for chain verification cannot verify the first block of the activation epoch using the standard MMR path.
- The effective activation of RFC0044 is silently delayed by one block, creating an inconsistency between the declared activation epoch and the actual first block carrying the extension.

---

### Likelihood Explanation

This is triggered deterministically at the RFC0044 activation epoch boundary. Any miner calling `get_block_template` when the tip is the last block of the activation epoch's predecessor will receive a template without the extension. The block they mine and submit will be accepted by all nodes (since the verifier has the same off-by-one), permanently committing the gap into the canonical chain. No special attacker capability is required — this is a normal miner/block-template-caller scenario.

---

### Recommendation

The fix requires updating both sides atomically:

1. In `build_extension`, derive the candidate block's epoch number from the snapshot's current epoch extension rather than from `tip_header.epoch().number()`:

```rust
// Instead of:
let mmr_activate = snapshot.consensus().rfc0044_active(tip_header.epoch().number());

// Use the candidate block's epoch number:
let candidate_epoch = snapshot.epoch_ext().number();
let mmr_activate = snapshot.consensus().rfc0044_active(candidate_epoch);
```

2. Apply the same correction in the contextual block verifier's RFC0044 activation check so both sides remain consistent. [4](#0-3) 

---

### Proof of Concept

1. Configure a dev chain with RFC0044 activating at epoch 2.
2. Mine blocks until the tip is the last block of epoch 1 (i.e., `tip.epoch().index() + 1 == tip.epoch().length()`).
3. Call `get_block_template` RPC. The returned template will have no `extension` field.
4. Mine and submit this block. It is accepted by the network.
5. The first block of epoch 2 — the RFC0044 activation epoch — is now on-chain without the MMR chain root extension.
6. Attempt to verify this block's MMR commitment as a light client: no extension is present, breaking the expected MMR chain continuity from the activation epoch. [4](#0-3)

/// EpochVerifier

/// Check for block epoch
pub struct EpochVerifier<'a> {
    epoch: &'a EpochExt,

impl<'a> EpochVerifier<'a> {
    pub fn new(epoch: &'a EpochExt, block: &'a BlockView) -> Self {
        EpochVerifier { epoch, block }
    }

    pub fn verify(&self) -> Result<(), Error> {
        let header = self.block.header();
        let actual_epoch_with_fraction = header.epoch();
        let block_number = header.number();
        let epoch_with_fraction = self.epoch.number_with_fraction(block_number);
        if actual_epoch_with_fraction != epoch_with_fraction {
            return Err(EpochError::NumberMismatch {
                expected: epoch_with_fraction.full_value(),
                actual: actual_epoch_with_fraction.full_value(),
            }
            .into());
        }
        let actual_compact_target = header.compact_target();
        if self.epoch.compact_target() != actual_compact_target {
            return Err(EpochError::TargetMismatch {
                expected: self.epoch.compact_target(),
                actual: actual_compact_target,
            }
            .into());
        }
        Ok(())
    }
