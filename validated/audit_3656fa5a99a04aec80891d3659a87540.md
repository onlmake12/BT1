Audit Report

## Title
Snappy Decompression Allocation Bypasses `max_frame_length` Bound in `LengthDelimitedCodecWithCompress::decode()` — (File: network/src/compress.rs)

## Summary
`LengthDelimitedCodecWithCompress::decode()` enforces `max_frame_length` only on the compressed wire bytes, but unconditionally allocates a buffer sized by the attacker-controlled snappy stream header value (up to `MAX_UNCOMPRESSED_LEN` = 8 MB) before any decompression occurs. For Sync (wire limit 2 MB) this yields a 4× memory amplification per frame, enabling any unprivileged peer to exhaust node memory by sustaining crafted compressed frames across multiple connections.

## Finding Description
`MAX_UNCOMPRESSED_LEN` is hardcoded to `1 << 23` (8 MB) at `network/src/compress.rs:13` with no relationship to `max_frame_length`.

In `decode()` at line 226, `self.length_delimited.decode(src)` enforces `max_frame_length` on the wire frame (2 MB for Sync, 4 MB for RelayV3). This check is correct and passes only frames within the wire limit.

When the compress flag is set (line 232), `decompress_len(&data[1..])` at line 233 reads the declared uncompressed length from the snappy stream header — a value the attacker fully controls. The only guard is `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` (line 235), which allows values up to 8,388,608 bytes.

At line 242, `BytesMut::zeroed(decompressed_bytes_len)` allocates up to 8 MB unconditionally, before any actual decompression and before any check against `max_frame_length`. The decompression at line 243 may subsequently fail, but the allocation has already occurred.

`CKBProtocol::build()` in `network/src/protocols/mod.rs` (lines 280–288) wires `LengthDelimitedCodecWithCompress` with `max_frame_length` from the protocol, and `new_with_support_protocol()` (lines 207–221) sets `compress: true` by default for all protocols including Sync and RelayV3. The same allocation pattern exists in `Message::decompress()` at lines 72–89.

The exploit path: attacker sends a snappy-flagged frame whose compressed payload is just under `max_frame_length` (passes wire check) and whose snappy header declares `MAX_UNCOMPRESSED_LEN - 1` bytes of output (passes the `> MAX_UNCOMPRESSED_LEN` guard). `BytesMut::zeroed(~8 MB)` executes regardless. The `max_frame_length` invariant is completely bypassed for the decompressed allocation.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

For Sync (`max_frame_length` = 2 MB), each crafted frame causes an 8 MB allocation — 4× amplification. With N concurrent peers each streaming crafted frames, the node allocates N × 8 MB instead of the intended N × 2 MB. On a node with 100 inbound peers, this is 800 MB of peak allocation versus the expected 200 MB. On nodes with 1–2 GB RAM, sustained adversarial traffic can exhaust available memory and crash the node.

## Likelihood Explanation
Any unprivileged peer that can establish a P2P connection can trigger this. No proof-of-work, no key, and no special role is required. Sync and RelayV3 are open to all peers by default. The attack is repeatable and sustained: a peer can continuously stream crafted frames, each triggering an 8 MB allocation. An attacker coordinating 50–100 peers can sustain the memory pressure needed to crash a target node.

## Recommendation
In `LengthDelimitedCodecWithCompress::decode()` (line 242 of `network/src/compress.rs`), after reading `decompressed_bytes_len` from the snappy header, add a second guard:

```rust
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
    return Err(io::ErrorKind::InvalidData.into());
}
```

This ensures the decompressed allocation is also bounded by `max_frame_length`, restoring the intended invariant. The identical pattern in `Message::decompress()` (lines 72–89) should receive the same fix, with `max_frame_length` passed as a parameter.

## Proof of Concept
1. Using the `snap` crate, craft a snappy stream whose header declares `MAX_UNCOMPRESSED_LEN - 1` (7,999,999) bytes of uncompressed output, with a minimal compressed payload (a few hundred bytes — well under the 2 MB Sync wire limit).
2. Wrap it in a length-delimited frame and feed it to `LengthDelimitedCodecWithCompress::decode()` configured with `max_frame_length = 2 * 1024 * 1024` (Sync).
3. Observe that `BytesMut::zeroed(7_999_999)` is called — allocating ~8 MB — despite the wire frame being only a few hundred bytes.
4. Assert `allocated_bytes > max_frame_length` — the invariant is broken.
