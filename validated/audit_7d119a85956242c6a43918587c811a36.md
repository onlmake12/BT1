Audit Report

## Title
Unbounded TCP RPC Connection Acceptance Enables RPC Denial-of-Service - (File: rpc/src/server.rs)

## Summary
The `start_tcp_server` function in `rpc/src/server.rs` accepts TCP connections in an unbounded loop, spawning an unconstrained `tokio::spawn` task per connection with no semaphore, counter, or per-IP limit. When `tcp_listen_address` is enabled, an unprivileged caller can exhaust file descriptors and Tokio task capacity by opening thousands of persistent connections, rendering the RPC subscription endpoint unresponsive.

## Finding Description
The accept loop at lines 176–194 of `rpc/src/server.rs` is unconditional:

```rust
while let Ok((stream, _)) = listener.accept().await {
    ...
    tokio::spawn(async move {          // spawned with no guard
        serve_stream_sink(&rpc, w, r, stream_config).await
    });
}
```

Each spawned task holds the TCP stream open for the lifetime of the subscription session. There is no maximum connection count, per-IP limit, idle timeout, or backpressure mechanism anywhere in `start_tcp_server`. The `RpcConfig` struct (`util/app-config/src/configs/rpc.rs`, lines 26–61) has no field for a TCP connection cap, confirming the absence of any such limit at the configuration layer. The `rpc_batch_limit` guard (`resource/ckb.toml`, lines 205–208) is commented out by default and applies only to HTTP batch requests, not to TCP connection count.

Exploit path:
1. Operator enables `tcp_listen_address` (opt-in, disabled by default).
2. Attacker opens N TCP connections to the subscription port (no authentication required).
3. Each connection holds a Tokio task and OS file descriptor.
4. At the process file-descriptor limit, `listener.accept()` begins returning errors; legitimate RPC callers time out.

## Impact Explanation
The impact is RPC unavailability on the local node — the node's P2P, consensus, and block-validation functions continue, but the RPC endpoint becomes unresponsive to legitimate callers (miners, wallets, indexers). This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**. The claim's assertion of High severity is not supported: the node itself does not crash, and `tcp_listen_address` is opt-in and bound to `127.0.0.1` by default, requiring explicit operator configuration before any remote attacker can reach the port.

## Likelihood Explanation
Triggerable by any unprivileged TCP client once `tcp_listen_address` is enabled. No authentication, no special protocol knowledge, and no victim mistake is required beyond the operator having enabled the feature. The default bind address (`127.0.0.1`) limits remote exploitation unless the operator also changes the bind to `0.0.0.0`. The attack is repeatable and requires only a standard TCP socket loop.

## Recommendation
1. Introduce a `tokio::sync::Semaphore` or atomic counter in `start_tcp_server` to cap concurrent connections; close connections that exceed the cap.
2. Add a per-IP connection limit to prevent a single source from monopolizing the slot pool.
3. Add an idle-connection timeout for TCP sessions that have not sent a valid JSON-RPC frame within a configurable window.
4. Uncomment `rpc_batch_limit` in `resource/ckb.toml` to enable the batch-size guard by default.
