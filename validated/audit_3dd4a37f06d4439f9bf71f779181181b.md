### Title
Non-Atomic Tx-Pool Persistence Write Destroys Existing Backup Before Confirming Write Success - (File: tx-pool/src/persisted.rs)

---

### Summary

`TxPool::save_into_file()` opens the persisted data file with `truncate(true)`, destroying the existing on-disk backup, and then calls `drain_all_transactions()` to remove all transactions from the in-memory pool — both destructive operations — before the new data is written to disk. If the subsequent `file.write_all()` fails (e.g., disk full, I/O error), the on-disk backup is empty and the in-memory pool is already drained. On the next node startup, `load_from_file()` encounters an empty or corrupt file, fails molecule deserialization, and silently starts with an empty pool, permanently losing all previously pending transactions.

---

### Finding Description

In `tx-pool/src/persisted.rs`, `save_into_file()` performs three sequential operations with no atomicity guarantee:

```rust
// Step 1: Truncate the existing backup file (destructive)
let mut file = OpenOptions::new()
    .create(true)
    .write(true)
    .truncate(true)   // <-- existing persisted data is destroyed here
    .open(&persisted_data_file)...?;

// Step 2: Drain all transactions from the in-memory pool (destructive)
let txs = TransactionVec::new_builder()
    .extend(self.drain_all_transactions().iter().map(|tx| tx.data()))
    .build();

// Step 3: Write to the now-empty file — if this fails, both backup and memory are gone
file.write_all(txs.as_slice())...?;
``` [1](#0-0) 

If `file.write_all` returns an error (disk full, I/O error, process killed mid-write), the file has already been truncated to zero bytes and the in-memory pool has already been drained. There is no recovery path.

On the next startup, `load_from_file()` checks `persisted_data_file.exists()` (the file exists, but is empty), reads zero bytes, and passes them to `TransactionVecReader::from_slice(&[])`, which fails molecule deserialization. [2](#0-1) 

The error is caught in `TxPoolServiceBuilder::start()` and silently replaced with an empty `Vec::new()`:

```rust
let txs = match tx_pool.load_from_file() {
    Ok(txs) => txs,
    Err(e) => {
        error!("{}", e.to_string());
        error!("Failed to load txs from tx-pool persistent data file, all txs are ignored");
        Vec::new()
    }
};
``` [3](#0-2) 

All pending transactions are permanently lost with no mechanism to recover them.

---

### Impact Explanation

All transactions that were in the tx-pool at shutdown time are permanently discarded. Users who submitted transactions expecting them to be proposed and committed in upcoming blocks will find their transactions silently dropped. The node operator has no indication beyond a log error. Transactions must be resubmitted by users, and any time-sensitive transactions (e.g., those with expiry constraints) may be permanently unprocessable. This is a data-loss impact analogous to the reported `fallback_backup_kernel` issue: an operation that assumes a write will succeed destroys the prior valid state before confirming the new state is safely persisted.

---

### Likelihood Explanation

The save is triggered during graceful shutdown via the cancellation signal path in `tx-pool/src/service.rs`: [4](#0-3) 

A disk-full condition at shutdown time is realistic in production environments where:
- A tx-pool submitter (RPC caller) floods the node with large valid transactions up to the `max_tx_pool_size` limit (180 MB default), consuming disk space when the pool is saved
- The node's data directory (blocks, logs, freezer) grows to fill the disk over time
- An operator restarts the node under low-disk conditions

The `max_tx_pool_size` is 180 MB by default, meaning a single save attempt can require up to 180 MB of free disk space. If the disk has less than that available, the write will fail after truncation. [5](#0-4) 

---

### Recommendation

Replace the truncate-then-write pattern with an atomic write:

1. Write the new data to a temporary file (e.g., `persisted_data.v1.tmp`) in the same directory.
2. Only after `file.sync_all()` succeeds, atomically rename/move the temp file to the final path.
3. If the write fails, the original file is untouched.

This mirrors the safe pattern already used in `PeerStore::dump_to_dir()`, which writes to a `tmp/` subdirectory and then calls `move_file()` to atomically replace the destination: [6](#0-5) 

---

### Proof of Concept

1. Start a CKB node with a nearly-full disk (leave < 180 MB free).
2. Submit transactions via RPC (`send_transaction`) until the tx-pool is populated.
3. Send `SIGTERM` to the node process to trigger graceful shutdown.
4. `save_pool()` is called → `save_into_file()` truncates the file → `write_all` fails with `ENOSPC`.
5. Restart the node. Observe the log: `"Failed to load txs from tx-pool persistent data file, all txs are ignored"`.
6. Query `get_raw_tx_pool` — the pool is empty; all previously pending transactions are gone.
