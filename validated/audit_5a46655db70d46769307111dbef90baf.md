### Title
Unbounded Hash-Chain Loop in `constructProviderCommitment` Enables DoS on Entropy Provider Reveals — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `constructProviderCommitment` function in `Entropy.sol` contains an unbounded `while` loop that iterates `numHashes` times. When a provider has `maxNumHashes == 0` (the default for providers that have not explicitly set it), there is no on-chain cap on `numHashes`. An unprivileged Entropy user can spam requests to drive `numHashes` to an arbitrarily large value, making the provider's subsequent `revealWithCallback` / `reveal` transactions exceed the block gas limit and permanently fail.

---

### Finding Description

`constructProviderCommitment` hashes a revelation value `numHashes` times in a plain `while` loop:

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
``` [1](#0-0) 

`numHashes` for a request is computed at request-creation time as:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
``` [2](#0-1) 

The only on-chain guard against a large `numHashes` is:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [3](#0-2) 

When `maxNumHashes == 0` (the default for any provider that has not called `setMaxNumHashes`), the condition short-circuits and the check is **entirely skipped**. `numHashes` is stored as `uint32`, allowing values up to ~4.29 billion.

`constructProviderCommitment` is called inside `revealHelper`, which is invoked by both `reveal` and `revealWithCallback`: [4](#0-3) 

---

### Impact Explanation

An attacker who makes N sequential requests against a provider with `maxNumHashes == 0` causes the N-th request to store `numHashes ≈ N`. When the provider attempts to call `revealWithCallback` for that request, `constructProviderCommitment` must execute N keccak256 iterations. At ~30 gas per keccak256, N ≈ 1,000,000 already costs ~30 million gas — near or above the block gas limit on most EVM chains. The reveal transaction reverts with an out-of-gas error, permanently blocking fulfillment of that request and all later ones that share the same gap. This constitutes a **Denial of Service** against the Entropy provider's ability to fulfill randomness requests.

---

### Likelihood Explanation

- Any provider that has not explicitly called `setMaxNumHashes` has `maxNumHashes == 0` by default.
- The attack requires the attacker to pay the provider's fee per request, but the cost is linear while the gas damage is also linear — the attacker can choose the cheapest provider and spam at low cost.
- The provider can mitigate by calling `advanceProviderCommitment`, but if they are not actively monitoring, the attack can succeed before they react.
- The `fortuna` keeper already monitors for this condition and warns when `outstanding_requests > threshold`, confirming the Pyth team is aware this is a real operational risk. [5](#0-4) 

---

### Recommendation

1. Remove the `maxNumHashes != 0` short-circuit. The guard should always enforce a cap; providers that want no cap should set `maxNumHashes` to `type(uint32).max`, not `0`.
2. Alternatively, set a protocol-level hard cap on `numHashes` independent of the provider's configuration.
3. Consider storing `numHashes` as a smaller type (e.g., `uint24`) to bound the worst-case loop count at the type level.

---

### Proof of Concept

1. Deploy or use an existing Entropy provider that has `maxNumHashes == 0`.
2. Attacker calls `requestV2(provider, gasLimit)` N times (e.g., N = 2,000,000), paying the required fee each time. Each call increments `providerInfo.sequenceNumber` by 1 without advancing `currentCommitmentSequenceNumber`.
3. The N-th request is stored with `req.numHashes = N`.
4. Provider calls `revealWithCallback(provider, sequenceNumber_N, userRandom, providerRevelation)`.
5. Inside `revealHelper`, `constructProviderCommitment(2_000_000, providerRevelation)` loops 2,000,000 times (~60 million gas for keccak256 alone), exceeding the block gas limit.
6. The transaction reverts out-of-gas. The provider cannot fulfill this request or any later request with a similarly large `numHashes`. [6](#0-5) [7](#0-6)
