### Title
Block Stuffing Permanently Locks User Fees via Expired `blockhash` Window in Legacy `request()` with `useBlockHash=true` - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The legacy `request()` function in `Entropy.sol` allows callers to set `useBlockHash = true`, which binds the reveal to a 256-block window. A malicious actor can stuff blocks with dummy transactions to push past that window, causing every subsequent `reveal()` call to permanently revert with `BlockhashUnavailable`. Because no refund or cancellation path exists, the user's fee is permanently locked in the contract.

---

### Finding Description

When a user calls `request(provider, userCommitment, true)`, the contract stores the current block number in `req.blockNumber` and sets `req.useBlockhash = true`. [1](#0-0) 

During `reveal()`, `revealHelper` is called, which executes:

```solidity
if (req.useBlockhash) {
    bytes32 _blockHash = blockhash(req.blockNumber);
    if (_blockHash == bytes32(uint256(0)))
        revert EntropyErrors.BlockhashUnavailable();
    blockHash = _blockHash;
}
``` [2](#0-1) 

The EVM `blockhash` opcode returns `0` for any block older than 256 blocks. This creates a hard 256-block reveal window. If the window expires, the revert fires and the request can never be fulfilled.

`clearRequest` is only called on a **successful** reveal path: [3](#0-2) 

There is no refund, cancellation, or recovery function for a request whose blockhash has expired. The fee paid at request time — split between the provider (`providerInfo.accruedFeesInWei`) and the Pyth protocol (`_state.accruedPythFeesInWei`) — is permanently retained: [4](#0-3) 

The newer `requestWithCallback` / `requestV2` path explicitly avoids this by hardcoding `useBlockHash = false`: [5](#0-4) 

However, the legacy `request()` function remains live and callable by any user or integrating contract: [6](#0-5) 

---

### Impact Explanation

A user who calls `request()` with `useBlockHash = true` and pays a fee permanently loses that fee if the 256-block window expires before they can reveal. On chains with cheap gas (BSC ~3 s/block → 256 blocks ≈ 12.8 minutes; Polygon ~2 s/block → 256 blocks ≈ 8.5 minutes), an attacker can fill every block in the window with dummy transactions for a few dollars, permanently bricking the target's request. The user receives no random number and no refund. In lottery, NFT-mint, or game contexts where the attacker is a competing participant, this translates to a direct financial gain for the attacker (the victim cannot claim a prize or favorable outcome).

---

### Likelihood Explanation

- The legacy `request()` function is still deployed and callable by any address.
- On low-fee EVM chains (BSC, Polygon, Avalanche C-Chain), filling 256 blocks costs on the order of $10–$30 USD — comparable to the original report's $14–$15 per block on BSC.
- Any on-chain application (lottery, NFT raffle, game) that uses `request()` with `useBlockHash = true` is a realistic target.
- The attacker only needs to know the victim's sequence number (emitted in the `Requested` event) to time the attack.

---

### Recommendation

1. **Add a refund path**: Allow the original requester to reclaim their fee after the blockhash window has expired (e.g., after block `req.blockNumber + 257`), clearing the request from storage.
2. **Deprecate `useBlockHash = true`**: The codebase already acknowledges the risk ("allows a scenario in which the provider and miner can collude"). Emit a deprecation warning or revert on `useBlockHash = true` in `request()`.
3. **Document the window risk**: If the path is kept, warn integrators that the 256-block window can be closed by block stuffing on low-fee chains.

---

### Proof of Concept

1. User calls `request(provider, userCommitment, true)` at block N, paying fee F. The `Requested` event reveals the sequence number.
2. Attacker (a competing lottery participant) observes the event and begins submitting high-gas dummy transactions to fill every block from N+1 onward.
3. After 256 blocks (≈ 12.8 minutes on BSC), `blockhash(N)` returns `0`.
4. User calls `reveal(provider, seqNum, userContribution, providerContribution)`.
5. `revealHelper` evaluates `blockhash(req.blockNumber)` → `0` → reverts `BlockhashUnavailable`.
6. No refund path exists; fee F is permanently split between provider and Pyth protocol.
7. The attacker's competing lottery entry is now the only valid one for that round.
