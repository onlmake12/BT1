### Title
Locally-Submitted Duplicate Transaction Triggers Incorrect `TxVerificationResult::Ok` Broadcast to All Peers - (File: `tx-pool/src/process.rs`)

### Summary

In `tx-pool/src/process.rs`, the `after_process` function handles a locally-submitted (RPC) transaction that is rejected as `Reject::Duplicated` by emitting `TxVerificationResult::Ok` to the relayer. This is the same signal used for a genuinely newly-accepted transaction. The relayer then broadcasts the transaction hash to **all** connected peers and calls `mark_as_known_tx`, as if the transaction were freshly accepted — even though it was already in the pool and was rejected.

### Finding Description

In `after_process` (`tx-pool/src/process.rs`), the `remote = None` (local RPC) branch has a special case for `Reject::Duplicated`:

```rust
Err(Reject::Duplicated(_)) => {
    debug!("after_process {} duplicated", tx_hash);
    // re-broadcast tx when it's duplicated and submitted through local rpc
    self.send_result_to_relayer(TxVerificationResult::Ok {
        original_peer: None,
        tx_hash,
    });
}
``` [1](#0-0) 

This sends `TxVerificationResult::Ok { original_peer: None, tx_hash }` — the exact same signal as a genuinely newly-accepted transaction — to the relayer.

In `send_bulk_of_tx_hashes` (`sync/src/relayer/mod.rs`), when `TxVerificationResult::Ok` with `original_peer: None` is received, the relayer:
1. Pushes the tx hash into the broadcast queue for **every** connected peer.
2. Calls `mark_as_known_tx(tx_hash)` on the tx filter. [2](#0-1) 

This is structurally identical to the reported Solidity vulnerability: an already-processed item (already-pooled transaction ≡ already-filled order) is re-submitted, and the system emits an incorrect "success" notification instead of a neutral or rejection signal.

The asymmetry is confirmed by comparing the remote path: when a **remote** peer sends a duplicate transaction, `Reject::Duplicated` falls into the generic `Err(reject)` branch, and because `is_allowed_relay()` returns `true` for `Duplicated`, the relayer correctly receives `TxVerificationResult::Reject`, which removes the tx from the known-tx filter. [3](#0-2) [4](#0-3) 

For local submissions, the opposite happens: `TxVerificationResult::Ok` is sent, causing a broadcast to all peers.

### Impact Explanation

A local RPC caller can repeatedly invoke `send_transaction` with the same transaction hash (already in the pool). Each call causes the node to enqueue a `TxVerificationResult::Ok` result, which on the next `send_bulk_of_tx_hashes` timer tick causes the node to:

- Broadcast the tx hash to every connected full-relay peer via `RelayTransactionHashes`.
- Mark the tx as known in the tx filter (potentially resetting deduplication state).

Connected peers that receive the hash announcement will issue `GetRelayTransactions` requests back to the node, consuming inbound bandwidth and CPU on both sides. This is a P2P amplification loop driven entirely from the local RPC interface. [5](#0-4) 

### Likelihood Explanation

The entry point is the `send_transaction` JSON-RPC method, which is accessible to any process with access to the node's RPC port. By default the RPC binds to localhost, so the attacker must be a local user or any process running on the same host (e.g., a wallet, a script, a compromised dependency). The scope explicitly includes "supported local CLI/RPC user" as a valid entry point. No special privileges, keys, or majority hashpower are required — only the ability to call `send_transaction` with a transaction already in the pool.

### Recommendation

Replace the `TxVerificationResult::Ok` emission for the `Reject::Duplicated` local case with a dedicated signal (or no signal), consistent with how the remote path handles duplicates. If re-broadcast is genuinely desired, a separate `TxVerificationResult::Rebroadcast` variant should be introduced so that `send_bulk_of_tx_hashes` can handle it without the side-effect of calling `mark_as_known_tx` and without treating the tx as a brand-new acceptance. At minimum, the `mark_as_known_tx` call must not be triggered for a duplicate submission.

```rust
// Current (incorrect):
Err(Reject::Duplicated(_)) => {
    self.send_result_to_relayer(TxVerificationResult::Ok {
        original_peer: None,
        tx_hash,
    });
}

// Suggested fix: do not emit Ok for an already-pooled tx,
// or emit a dedicated Rebroadcast variant that skips mark_as_known_tx.
Err(Reject::Duplicated(_)) => {
    // no-op, or send a Rebroadcast variant
}
```

### Proof of Concept

1. Start a CKB node with default RPC on `localhost:8114`.
2. Submit a valid transaction `T` via `send_transaction` RPC → accepted into the pool.
3. Immediately call `send_transaction` again with the same `T`.
4. The RPC returns `PoolRejectedDuplicatedTransaction` to the caller (correct), but internally `after_process` fires `TxVerificationResult::Ok { original_peer: None, tx_hash: T.hash() }`.
5. On the next `send_bulk_of_tx_hashes` timer tick, the node broadcasts `RelayTransactionHashes([T.hash()])` to every connected peer.
6. Each connected peer responds with `GetRelayTransactions([T.hash()])`, consuming bandwidth.
7. Repeat step 3 in a tight loop to amplify the effect proportionally to the number of connected peers. [6](#0-5) [2](#0-1)
