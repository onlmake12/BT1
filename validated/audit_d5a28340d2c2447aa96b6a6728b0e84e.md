### Title
Permissionless `updatePriceFeeds` with 1-Hour Validity Window Enables Adversarial Price Selection in Pyth Pulse Scheduler — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.updatePriceFeeds` function carries no access control and accepts any cryptographically valid Pyth price update whose `publishTime` falls within the last `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` (1 hour). An unprivileged attacker acting as a "keeper" can query the Hermes historical price API, cherry-pick a favorable price from within that 1-hour window, push it to the Scheduler, and immediately exploit the resulting on-chain price in a downstream protocol — the exact adversarial-price-selection pattern described in the reference report.

---

### Finding Description

`Scheduler.updatePriceFeeds` is declared `external` with no `onlyKeeper`, `onlyWhitelisted`, or any other caller restriction:

```solidity
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {
``` [1](#0-0) 

The only temporal guard is enforced inside `_validateShouldUpdatePrices`. It rejects updates whose `updateTimestamp` is older than `block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`:

```solidity
uint256 minAllowedTimestamp = (block.timestamp >
    PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    : 0;

if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(...);
}
``` [2](#0-1) 

`PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is hardcoded to **1 hour**:

```solidity
uint64 public constant PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours;
``` [3](#0-2) 

The only other temporal constraint is that the new update must be strictly newer than the last stored one:

```solidity
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(...);
}
``` [4](#0-3) 

Crucially, the Pyth core contract is called with `minPublishTime = 0`, so it imposes no lower-bound timestamp check of its own — the Scheduler is solely responsible for the 1-hour floor:

```solidity
) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
        updateData,
        params.priceIds,
        0, // We enforce the past max validity ourselves
        curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
        ...
    );
``` [5](#0-4) 

The deviation check compares the pushed price against the **stored** price, not the current market price:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;
int64 previousPrice = previousFeed.price.price;
...
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
    return updateTimestamp;
}
``` [6](#0-5) 

This means an attacker can scan the Hermes historical-price API for any signed update in the last hour whose price deviates from the stored price by at least `deviationThresholdBps`, push it, and have it accepted as a legitimate keeper update — while also receiving the keeper payment from the subscription balance.

---

### Impact Explanation

Any downstream protocol (lending, perp, options) that reads prices from a Scheduler subscription is exposed. The attacker can:

1. Observe the current stored price `P_stored` in the Scheduler.
2. Retrieve a signed Pyth price update `P_favorable` from the Hermes historical API (up to 1 hour old, newer than `P_stored`) that is maximally favorable to their open position.
3. Call `updatePriceFeeds` — which is permissionless — to overwrite the stored price with `P_favorable`.
4. In the same block, interact with the downstream protocol at `P_favorable` (e.g., borrow against over-valued collateral, liquidate an under-valued position, or open/close a leveraged trade).

The attacker is also **paid** a keeper fee from the subscription's balance for performing this manipulation, making the attack self-funding.

---

### Likelihood Explanation

- **No privilege required**: any EOA can call `updatePriceFeeds`.
- **Data freely available**: Pyth's Hermes API exposes signed historical price updates for any timestamp within the last hour at no cost.
- **Economically incentivized**: the attacker receives a keeper fee on top of the trading profit.
- **Broad applicability**: every active Scheduler subscription with a deviation-based or heartbeat-based trigger is a potential target.

---

### Recommendation

1. **Tighten the validity window**: Reduce `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` from 1 hour to a value consistent with the downstream protocol's staleness tolerance (e.g., 60–120 seconds). Closed-market feeds can be handled by a separate, explicit exception rather than a blanket 1-hour window.
2. **Restrict caller or require a commit-reveal**: Require keepers to register, or introduce a two-step commit-reveal so that the price used cannot be known before the commitment is on-chain.
3. **Enforce a maximum timestamp gap between consecutive updates**: Reject updates whose `publishTime` is more than N seconds newer than `priceLastUpdatedAt`, preventing large jumps caused by cherry-picked historical prices.

---

### Proof of Concept

```
Setup:
  - Subscription S: updateOnDeviation=true, deviationThresholdBps=100 (1%)
  - Stored price: ETH = $2000 (publishTime = T-30min)
  - Current market price: ETH = $2000

Attack:
  1. Attacker queries Hermes: GET /v2/updates/price/{T-25min}
     → Returns signed VAA with ETH = $2025 (+1.25%, above threshold)
  2. Attacker calls Scheduler.updatePriceFeeds(subscriptionId, vaa)
     → Passes: publishTime(T-25min) > priceLastUpdatedAt(T-30min) ✓
     → Passes: T-25min > block.timestamp - 1hour ✓
     → Deviation = 1.25% ≥ 1% threshold ✓
     → Stored price updated to $2025; attacker receives keeper fee
  3. Attacker immediately borrows max USDC against ETH collateral
     valued at $2025 instead of the true $2000
  4. Attacker walks away with ~1.25% excess borrowing capacity
     (amplified by position size) plus the keeper fee
```
