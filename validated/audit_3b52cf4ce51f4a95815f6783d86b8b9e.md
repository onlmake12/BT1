### Title
Excess Native Token Sent to `requestHelper` Is Silently Absorbed Into Pyth Fee Pool Instead of Being Refunded — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s internal `requestHelper` function (called by every public request entry point) checks `msg.value < requiredFee` to gate insufficient payments, but accepts any overpayment silently. The entire surplus above `providerFee` is credited to `_state.accruedPythFeesInWei` rather than returned to the caller. Any user who sends more ETH than the exact required fee permanently loses the excess to Pyth's protocol fee pool.

---

### Finding Description

`requestHelper` computes the required fee and validates payment with a strict-less-than guard:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The split is: `providerFee` → provider balance, `msg.value - providerFee` → Pyth balance. When `msg.value > requiredFee`, the surplus is entirely absorbed into `_state.accruedPythFeesInWei`. No refund path exists. The public-facing `request`, `requestWithCallback`, and `requestV2` functions all route through `requestHelper` and carry the same comment: *"Note that excess value is *not* refunded to the caller."* [2](#0-1) [3](#0-2) 

By contrast, `PythLazer.sol`'s `verifyUpdate` demonstrates the correct pattern: it checks `msg.value >= verification_fee` and immediately refunds the surplus with `payable(msg.sender).transfer(msg.value - verification_fee)`. [4](#0-3) 

The same `>=`-without-refund pattern also appears in `Pyth.sol`'s `updatePriceFeeds`, `parsePriceFeedUpdatesWithConfig`, and `parseTwapPriceFeedUpdates`, where excess ETH is not tracked in any accounting variable at all and simply accumulates in the contract balance. [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any caller of `request`, `requestWithCallback`, or `requestV2` who sends even 1 wei above the exact required fee permanently loses that surplus. The surplus is credited to Pyth's protocol fee pool (`_state.accruedPythFeesInWei`) and is only withdrawable by Pyth governance — the original sender has no recourse. For `Pyth.sol`'s update functions the excess is not even tracked; it is irrecoverably locked in the contract balance. This constitutes a direct, irreversible loss of user funds with no protocol benefit proportional to the overpayment. [8](#0-7) 

---

### Likelihood Explanation

Fees in Entropy are dynamic: providers can call `setProviderFee` at any time, and the Pyth protocol fee can change via governance. A caller who queries `getFeeV2` off-chain and then submits a transaction with a small buffer to avoid reverts (a common defensive pattern) will routinely overpay. Integrating contracts that hardcode a fee estimate or add a safety margin are especially exposed. The entry points are fully permissionless and require no privileged role. [9](#0-8) 

---

### Recommendation

Adopt the pattern already used in `PythLazer.sol`: after crediting the exact required amounts to provider and protocol balances, refund any surplus to `msg.sender`.

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee);   // use requiredFee, not msg.value
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

Apply the same fix to `updatePriceFeeds`, `parsePriceFeedUpdatesWithConfig`, and `parseTwapPriceFeedUpdates` in `Pyth.sol`.

---

### Proof of Concept

1. Provider registers with `feeInWei = 1 ether`.
2. User calls `getFeeV2(provider, 0)` off-chain → returns `1 ether`.
3. User submits `requestV2(provider, secret, 0){value: 1.1 ether}` with a 10 % buffer.
4. `requestHelper` executes: `providerInfo.accruedFeesInWei += 1 ether`; `_state.accruedPythFeesInWei += 0.1 ether`.
5. User's 0.1 ETH surplus is now in Pyth's fee pool. The user receives no refund and has no mechanism to recover it. [10](#0-9)
