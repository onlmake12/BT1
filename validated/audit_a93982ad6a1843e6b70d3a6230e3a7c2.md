### Title
Off-by-One in Spawn VM Count Check Allows One Extra VM Beyond `MAX_VMS_COUNT` - (File: `script/src/scheduler.rs`)

---

### Summary

In `Scheduler::process_message_box`, the guard that enforces the maximum number of concurrent VMs uses a strict `>` comparison instead of `>=`. This allows a script to spawn exactly one VM beyond the intended `MAX_VMS_COUNT` limit of 16, bypassing the resource cap enforced by the CKB-VM spawn subsystem.

---

### Finding Description

`MAX_VMS_COUNT` is defined as `16` in `script/src/scheduler.rs`:

```rust
pub const MAX_VMS_COUNT: u64 = 16;
``` [1](#0-0) 

When a `Message::Spawn` is processed, the guard is:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
    // return MAX_VMS_SPAWNED error to the script
    continue;
}
let spawned_vm_id = self.boot_vm(...)?;
``` [2](#0-1) 

Because the condition is `>` (strictly greater than) rather than `>=`, when `suspended.len() + instantiated.len() == 16` (exactly at the limit), the guard does **not** fire. `boot_vm` is then called, inserting a 17th VM into `self.instantiated` and `self.states`. The enforced limit is therefore `MAX_VMS_COUNT + 1 = 17`, not 16.

This is directly analogous to the BusLib::ride bug: the condition permits the count to equal the capacity, allowing one extra element beyond the intended maximum.

The inconsistency is visible by comparing with the `MAX_FDS` guard in the same function, which correctly uses `>=`:

```rust
if self.fds.len() as u64 >= MAX_FDS {
    // return MAX_FDS_CREATED error
    continue;
}
``` [3](#0-2) 

---

### Impact Explanation

A script author can craft a lock or type script that calls `ckb_spawn` in a loop. When the total active VM count reaches `MAX_VMS_COUNT` (16), the guard passes and one additional VM is created, making the real ceiling 17. This:

1. Violates the protocol-specified resource limit for spawned VMs.
2. Allows scripts to consume more memory and scheduler state than the consensus rules intend.
3. If `MAX_VMS_COUNT` is a consensus-critical boundary (which it is, as it is a hard-coded constant in the node's script verifier), any future tightening of the limit or reliance on the exact value in downstream logic could produce divergent behavior.

The `states` map, `instantiated` map, and `suspended` map all grow by one extra entry, consuming additional heap memory per script execution beyond the intended cap.

---

### Likelihood Explanation

Any unprivileged transaction sender can submit a transaction whose lock or type script calls `ckb_spawn` up to 16 times. The 16th call succeeds (creating VM #17) instead of returning `MAX_VMS_SPAWNED`. No special privileges, keys, or majority hashpower are required. The entry path is fully attacker-controlled via a crafted script in a submitted transaction.

---

### Recommendation

Change the comparison from `>` to `>=` to match the semantics of the `MAX_FDS` guard and enforce the intended limit:

```rust
// Before (incorrect):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
``` [4](#0-3) 

---

### Proof of Concept

1. Author a CKB script (lock or type) that calls `ckb_spawn` in a loop, each time spawning a child that immediately exits.
2. On the 16th spawn call (when `suspended + instantiated == 16`), the guard `> 16` evaluates to `false`, so `boot_vm` is invoked and a 17th VM is created successfully — returning `SUCCESS` instead of `MAX_VMS_SPAWNED`.
3. The test data file `spawn_create_17_spawn.c` already encodes the expectation that the 17th spawn (index 16, 0-based) should fail with `CKB_MAX_VMS_SPAWNED`, but due to this bug it succeeds:

```c
if ((i < 16) || (ret != CKB_MAX_VMS_SPAWNED)) {
    printf("Unexpected spawn error!\n");
    return -1;
}
``` [5](#0-4) 

The off-by-one means the 17th VM (i == 16) is spawned successfully, causing this check to fail and confirming the boundary is not enforced correctly.
