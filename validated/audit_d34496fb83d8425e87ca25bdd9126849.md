Audit Report

## Title
VerifyQueue Can Be Flooded with Structurally Valid but Semantically Invalid Transactions, Causing Legitimate Transaction Rejection - (File: `tx-pool/src/component/verify_queue.rs`, `tx-pool/src/process.rs`)

## Summary

`resumeble_process_tx` admits transactions into the 256 MB verify queue after only a structural `non_contextual_verify` check, with no fee floor and no input-existence check at admission time. An unprivileged remote peer can flood the queue with transactions referencing fabricated `OutPoint` values at zero on-chain cost, causing all subsequent legitimate transactions to receive `Reject::Full`. The attacker is never banned because missing-input rejections route to the orphan pool rather than `ban_malformed`.

## Finding Description

**Admission path — no semantic gate:**
`resumeble_process_tx` (process.rs L335–353) calls `non_contextual_verify` (util.rs L56–83), which runs only `NonContextualTransactionVerifier` (size, cellbase, script-group format checks) and a size-limit check. If that passes, the transaction is immediately inserted via `enqueue_verify_queue` (process.rs L352 → L860–868 → verify_queue.rs `add_tx` L198–237). No fee check and no input-existence check occur at this stage.

**Queue admission gate — size only:**
`add_tx` (verify_queue.rs L198–237) has a single guard: `is_full` (L103–106), which rejects new entries once `total_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256,000,000 bytes, L18). There is no per-peer byte budget, no minimum fee floor, and no priority eviction.

**Deferred semantic checks:**
`pre_check` (process.rs L269–316), which calls `resolve_tx` (L286) and `check_tx_fee` (L289), executes only inside `_process_tx` (process.rs L705–777, specifically L715) — after the transaction has already been dequeued by a background worker. A transaction with a randomly fabricated `OutPoint` passes `non_contextual_verify`, occupies queue space, and is only rejected at `pre_check` when the worker processes it.

**No ban for missing-input rejections:**
`after_process` (process.rs L458–555) handles the post-processing path. When a worker rejects a transaction for `OutPointError::Unknown`, `is_missing_input` (util.rs L150–152) returns `true`, and the code at L507–512 routes the transaction to the orphan pool via `add_orphan` — it does **not** call `ban_malformed`. The `ban_malformed` path (L514–515) is only reached for `reject.is_malformed_tx()`. Transactions with fake inputs are not structurally malformed, so the attacker peer is never disconnected and `remove_txs_by_peer` (verify_queue.rs L159–168), which is only invoked from `ban_malformed` (process.rs L702), is never called to clean up their entries.

**Entry point for remote peers:**
`submit_remote_tx` (process.rs L371–379) calls `resumeble_process_tx_and_notify_full_reject` → `resumeble_process_tx`, confirming any standard relay peer can reach this path.

## Impact Explanation

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

Filling the 256 MB queue with ~200-byte transactions requires roughly 1.28 million crafted transactions. Each costs only network bandwidth — no on-chain funds, no key material, no hashpower. While the queue is full, every legitimate transaction submitted via relay or RPC receives `Reject::Full` and is dropped. Workers drain attacker entries in FIFO order (`pop_front`/`peek`, verify_queue.rs L171–194), each consuming a snapshot read and cell-resolution attempt before rejection. After draining, the attacker can immediately re-flood. The orphan pool also fills with the rejected entries, compounding resource pressure.

## Likelihood Explanation

Any unprivileged remote peer reachable via the standard relay protocol (`submit_remote_tx`, process.rs L371–379) can trigger this. No privileged access is required. Crafting structurally valid transactions with fake `OutPoint` values is trivial and requires no on-chain funds. The attack is repeatable with no cooldown: after the worker drains the queue, the attacker reconnects (or uses the same connection, since no ban is issued) and re-floods. There is no per-peer admission throttle at the verify-queue level.

## Recommendation

1. **Enforce a minimum fee rate at queue admission**: Before `enqueue_verify_queue`, perform a lightweight fee-rate plausibility check using declared output capacity vs. a flat minimum-byte-fee floor. This mirrors the existing `check_tx_fee` logic in `util.rs` but applied earlier, without requiring full input resolution.
2. **Per-peer queue slot limit**: Track bytes-in-queue per `PeerIndex` inside `VerifyQueue` and reject new submissions from a peer once their share exceeds a configurable threshold.
3. **Priority eviction**: When the queue is full and an incoming transaction has a higher declared fee rate than the lowest-priority entry, evict the lowest entry rather than rejecting the new one.
4. **Ban or penalize peers whose transactions are repeatedly rejected for missing inputs**: Rather than silently routing to the orphan pool, apply a score penalty or temporary ban after N consecutive missing-input rejections from the same peer.

## Proof of Concept

1. Connect to a CKB node as a remote peer.
2. Craft transactions each spending a randomly generated, non-existent `OutPoint` (32-byte random cell tx hash + arbitrary index). Each transaction is ~200 bytes serialized and passes `non_contextual_verify` because that check does not resolve inputs.
3. Relay all transactions via `submit_remote_tx`. Each reaches `resumeble_process_tx` → `non_contextual_verify` (passes) → `enqueue_verify_queue` (admitted, queue grows).
4. After ~1.28 million transactions, `total_tx_size` reaches 256,000,000 bytes. `is_full` returns `true`.
5. Submit a legitimate transaction from any peer or RPC. It receives `Reject::Full("verify_queue total_tx_size exceeded...")`.
6. Workers begin draining: each attacker transaction triggers `pre_check` → `resolve_tx` → `OutPointError::Unknown` → `is_missing_input` returns `true` → routed to orphan pool. No ban is issued.
7. After the queue drains, repeat from step 2. The attacker peer remains connected.
