### Title
Excess Fee Coin Not Returned to Caller in `update_price_feeds` — (File: target_chains/aptos/contracts/sources/pyth.move)

### Summary

In `target_chains/aptos/contracts/sources/pyth.move`, the `update_price_feeds` function accepts a `Coin<AptosCoin>` fee argument and deposits the **entire coin** to the Pyth contract, even when the actual required fee is less than the coin's value. The excess is never returned to the caller.

### Finding Description

`update_price_feeds` computes `total_updates` by iterating over the provided VAAs and calling `update_price_feed_from_single_vaa` for each. That inner function returns `0` for any VAA whose price is not newer than the stored price (i.e., a stale or duplicate update). The fee is then computed as `state::get_base_update_fee() * total_updates`.

```move
public fun update_price_feeds(vaas: vector<vector<u8>>, fee: Coin<AptosCoin>) {
    let total_updates = 0;
    while (!vector::is_empty(&vaas)) {
        total_updates = total_updates + update_price_feed_from_single_vaa(vector::pop_back(&mut vaas));
    };
    let update_fee = state::get_base_update_fee() * total_updates;
    assert!(update_fee <= coin::value(&fee), error::insufficient_fee());
    coin::deposit(@pyth, fee);   // <-- entire coin deposited, not just update_fee
}
``` [1](#0-0) 

The caller cannot know `total_updates` before calling the function, because some VAAs may be skipped (price already up-to-date). A caller who passes a coin sized for `N` VAAs but only `M < N` result in actual updates will have `(N - M) * base_update_fee` permanently transferred to the Pyth contract with no mechanism for recovery.

By contrast, `update_price_feeds_with_funder` correctly withdraws **exactly** `update_fee` from the signer's account after computing `total_updates`:

```move
let update_fee = state::get_base_update_fee() * total_updates;
let fee = coin::withdraw<AptosCoin>(account, update_fee);
coin::deposit(@pyth, fee);
``` [2](#0-1) 

The `update_price_feeds` variant does not perform this split — it deposits the entire caller-supplied coin.

### Impact Explanation

Any caller of `update_price_feeds` who supplies a coin larger than the actual required fee permanently loses the excess APT to the Pyth contract. Because `total_updates` is only known after execution, callers must over-estimate the fee to avoid reverting on the `assert!`. The over-estimated amount is silently absorbed by the protocol. There is no withdrawal path for the original caller to recover these funds.

### Likelihood Explanation

This is reachable by any unprivileged updater calling `update_price_feeds` directly (it is a `public fun`, callable from any module). The scenario is common: a caller submits a batch of VAAs where some prices are already fresher on-chain (e.g., another updater raced ahead), causing `total_updates < len(vaas)`. The caller's coin was sized for `len(vaas)` updates, so the excess is lost. This is a normal operational condition, not an edge case.

### Recommendation

After computing `update_fee`, split the coin and return the remainder to the caller:

```move
public fun update_price_feeds(vaas: vector<vector<u8>>, fee: Coin<AptosCoin>): Coin<AptosCoin> {
    let total_updates = 0;
    while (!vector::is_empty(&vaas)) {
        total_updates = total_updates + update_price_feed_from_single_vaa(vector::pop_back(&mut vaas));
    };
    let update_fee = state::get_base_update_fee() * total_updates;
    assert!(update_fee <= coin::value(&fee), error::insufficient_fee());
    let exact_fee = coin::extract(&mut fee, update_fee);
    coin::deposit(@pyth, exact_fee);
    fee  // return excess to caller
}
```

Alternatively, mirror the pattern of `update_price_feeds_with_funder` and accept a `&signer` to withdraw exactly the computed fee.

### Proof of Concept

1. On-chain, price feed `X` has publish time `T`.
2. Caller prepares 3 VAAs: two with publish time `T+1` (fresh) and one with publish time `T-1` (stale).
3. Caller computes expected fee as `3 * base_update_fee` and creates a coin of that value.
4. Caller calls `update_price_feeds([vaa1, vaa2, vaa3], coin_3x_fee)`.
5. Inside the function, `update_price_feed_from_single_vaa` returns `1, 1, 0` for the three VAAs → `total_updates = 2`.
6. `update_fee = 2 * base_update_fee`.
7. `assert!(2 * base_update_fee <= 3 * base_update_fee)` passes.
8. `coin::deposit(@pyth, fee)` deposits the full `3 * base_update_fee` coin.
9. Caller loses `1 * base_update_fee` with no recourse.
