### Title
Malicious Entropy Requester Can Permanently DoS Provider Fulfillment via Reverting Callback in Legacy `revealWithCallback` Path — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In the legacy execution branch of `revealWithCallback`, when `req.gasLimit10k == 0`, the callback to the requester is made with a bare, uncaught external call. A malicious requester can deploy a contract whose `_entropyCallback` always reverts. Every provider attempt to fulfill the request will revert, permanently locking the request in storage and blocking the provider's `currentCommitmentSequenceNumber` from advancing. If the provider has `maxNumHashes` configured, this eventually prevents all new requests from being accepted.

---

### Finding Description

`revealWithCallback` in `Entropy.sol` has two execution paths, selected by whether `req.gasLimit10k != 0`:

**New path (gasLimit10k != 0):** Uses `excessivelySafeCall` to catch reverts, emits `CallbackFailed`, and transitions the request to `CALLBACK_FAILED` state — the request remains fulfillable.

**Legacy path (gasLimit10k == 0):** Entered when the provider has `defaultGasLimit == 0` (has not opted into the new callback-failure flow). The request is cleared first, then the callback is called with no error catching:

```solidity
} else {
    address callAddress = req.requester;
    ...
    clearRequest(provider, sequenceNumber);
    // WARNING: DO NOT USE req BELOW HERE AS ITS CONTENTS HAS BEEN CLEARED

    if (len != 0) {
        IEntropyConsumer(callAddress)._entropyCallback(   // <-- bare call, no try/catch
            sequenceNumber,
            provider,
            randomNumber
        );
    }
```

If `_entropyCallback` reverts, the **entire transaction reverts**, including the `clearRequest` and the `revealHelper` update to `providerInfo.currentCommitmentSequenceNumber`. The request is never cleared and can never be cleared, because every subsequent `revealWithCallback` attempt will also revert.

The attacker path:
1. Deploy a contract whose `_entropyCallback` unconditionally reverts.
2. Call `requestWithCallback(provider, userContribution)` (which passes `gasLimit=0`, landing in the legacy path when the provider's `defaultGasLimit == 0`).
3. Every provider keeper call to `revealWithCallback` for this sequence number reverts.
4. The request occupies a storage slot permanently.
5. `providerInfo.currentCommitmentSequenceNumber` cannot advance past this sequence number.
6. If `providerInfo.maxNumHashes != 0`, future requests whose `numHashes` exceeds `maxNumHashes` revert with `LastRevealedTooOld`, blocking new requests. [1](#0-0) 

---

### Impact Explanation

- **Immediate:** The provider's Fortuna keeper wastes gas indefinitely retrying the stuck request.
- **Medium-term:** `providerInfo.currentCommitmentSequenceNumber` is stuck; the provider cannot advance it via normal fulfillment.
- **Long-term (if `maxNumHashes` is set):** After `maxNumHashes` additional requests are made, all new requests revert with `LastRevealedTooOld`, completely halting the provider's service. The provider must call `advanceProviderCommitment` to manually skip the stuck slot — but an attacker can create multiple stuck requests to overwhelm this mitigation.

The new V2 path explicitly avoids this by using `excessivelySafeCall` and the `CALLBACK_FAILED` state, confirming the Pyth team is aware of the class of issue but the legacy path remains unprotected. [2](#0-1) 

---

### Likelihood Explanation

- Any unprivileged user can call `requestWithCallback` or `requestV2(..., gasLimit=0)` against any provider whose `defaultGasLimit == 0`.
- Providers that have not yet called `setDefaultGasLimit` have `defaultGasLimit == 0` by default (storage zero-initialization).
- The attack requires only a small fee payment (the provider's `feeInWei`).
- No privileged access, leaked keys, or external oracle manipulation is required. [3](#0-2) [4](#0-3) 

---

### Recommendation

Apply the same protection used in the new path to the legacy path: wrap the bare `_entropyCallback` call in a `try/catch` (or `excessivelySafeCall`) and, on revert, either:
- Transition the request to a `CALLBACK_FAILED` state (allowing retry), or
- Clear the request and emit a `CallbackFailed` event so the provider's fees are not permanently locked.

Alternatively, deprecate the legacy path entirely by requiring all providers to set a non-zero `defaultGasLimit` before accepting new callback requests. [5](#0-4) 

---

### Proof of Concept

```solidity
// Malicious requester contract
contract MaliciousRequester is IEntropyConsumer {
    address entropy;
    constructor(address _entropy) { entropy = _entropy; }

    function getEntropy() internal view override returns (address) { return entropy; }

    // Always reverts — blocks every revealWithCallback attempt
    function entropyCallback(uint64, address, bytes32) internal override {
        revert("blocked");
    }

    function attack(address provider, bytes32 userNum) external payable {
        // Provider must have defaultGasLimit == 0 for legacy path
        IEntropy(entropy).requestWithCallback{value: msg.value}(provider, userNum);
    }
}
```

1. Deploy `MaliciousRequester` against a provider with `defaultGasLimit == 0`.
2. Call `attack(provider, someBytes32)` with sufficient fee.
3. Observe that every `revealWithCallback` call for the assigned sequence number reverts.
4. The request is never cleared; `currentCommitmentSequenceNumber` is stuck.
5. If `maxNumHashes` is set to e.g. 100, after 100 more requests, all new requests revert with `LastRevealedTooOld`. [6](#0-5) [7](#0-6)
