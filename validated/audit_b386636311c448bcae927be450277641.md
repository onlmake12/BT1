Audit Report

## Title
Fee Rate Minimum Check Uses Transaction Size Instead of Weight, Bypassing Cycle-Weighted Admission Control - (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using raw serialized byte size rather than the canonical transaction weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). Because cycles are only known after script execution and no weight-based fee check is performed post-execution, an unprivileged submitter can craft a transaction with maximum cycles and minimal byte size that passes the size-based gate with a fee far below what the weight-based `min_fee_rate` would require. The same unit mismatch exists in `calculate_min_replace_fee` for the RBF path.

## Finding Description

`get_transaction_weight` in `util/types/src/core/tx_pool.rs` defines weight as:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [1](#0-0) 

`FeeRate` is defined as shannons per kilo-weight, so the correct minimum fee for a 200-byte, 70 000 000-cycle transaction is `1000 × 11940 / 1000 = 11 940 shannons`.

The sole admission-time fee gate, `check_tx_fee`, instead computes:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [2](#0-1) 

The code's own comment concedes this is theoretically incorrect. `check_tx_fee` is called inside `pre_check`, which runs **before** script execution: [3](#0-2) 

After `verify_rtx` returns the actual cycles, the code constructs a `TxEntry` and calls `submit_entry` with no subsequent weight-based fee check: [4](#0-3) 

`TxEntry::fee_rate()` correctly uses weight for sorting/eviction: [5](#0-4) 

But this is not a rejection gate — it only governs which entries are evicted when the pool exceeds `max_tx_pool_size` (measured in bytes, not weight), meaning cycle-heavy, byte-light transactions occupy minimal pool space and survive eviction longer than their effective fee rate warrants.

The same mismatch appears in `calculate_min_replace_fee`:

```rust
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
``` [6](#0-5) 

## Impact Explanation

With `min_fee_rate = 1 000` shannons/KW and a transaction of 200 bytes consuming 70 000 000 cycles:

| Metric | Value |
|---|---|
| Actual weight | `max(200, 70 000 000 × 0.000170571)` ≈ **11 940** |
| `min_fee` via size check | `1 000 × 200 / 1 000` = **200 shannons** |
| `min_fee` via weight | `1 000 × 11 940 / 1 000` = **11 940 shannons** |

A transaction paying 201 shannons passes the gate despite an effective fee rate of ≈ 16 shannons/KW — roughly 60× below the configured threshold. Because the pool size limit is byte-based, many such transactions fit simultaneously. Each forces the node to execute 70 000 000 cycles of script work before insertion. An attacker can continuously flood the pool with cycle-expensive, fee-cheap transactions, exhausting node CPU and degrading service for legitimate users.

This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation

The attack requires only an unprivileged `send_transaction` RPC call. No special role, key, or majority hashpower is needed. The attacker controls both transaction byte size (keep minimal) and script complexity (maximize cycles up to `max_tx_verify_cycles`, default 70 000 000). The discrepancy between size-based gate and weight-based reality is largest precisely at the maximum-cycles boundary, which is trivially targetable. The attack is repeatable and cheap to sustain.

## Recommendation

After script execution, when cycles are known, perform a second fee rate check using the actual weight:

```rust
// In _process_tx, after verify_rtx returns verified.cycles:
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Apply the same correction to `calculate_min_replace_fee` in `tx-pool/src/pool.rs`, substituting `get_transaction_weight(size, entry.cycles)` for the raw `size` argument passed to `min_rbf_rate.fee(...)`.

## Proof of Concept

1. Craft a CKB transaction whose lock/type script runs a tight loop consuming ≈ 70 000 000 cycles but whose serialized size is ≈ 200 bytes (single input/output, compact witness).
2. Set the fee to 201 shannons (just above the size-based threshold of 200 shannons).
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`; the transaction passes the gate.
5. `verify_rtx` executes scripts consuming 70 000 000 cycles; the entry is inserted with weight ≈ 11 940.
6. The effective fee rate stored in the pool is `201 × 1000 / 11940 ≈ 16 shannons/KW`, far below the 1 000 shannons/KW threshold.
7. Repeat to fill the pool with cycle-expensive, fee-cheap transactions. Each submission forces the node to execute 70 000 000 cycles of script verification. Because pool size is byte-bounded, many such 200-byte transactions fit simultaneously, crowding out legitimate higher-fee-rate transactions and continuously wasting node verification resources.
