### Title
Network Alert Signed Message Replay Across CKB Networks — (`File: util/network-alert/src/verifier.rs`)

---

### Summary

The `RawAlert` message hash used in `Verifier::verify_signatures` does not include any chain/network identifier. Because the same Nervos Foundation signing keys are hard-coded for all CKB networks (mainnet, testnet), a validly-signed alert from one network can be replayed verbatim on another network and will pass signature verification.

---

### Finding Description

`calc_alert_hash()` is computed exclusively over the serialized bytes of `RawAlert`, whose schema is:

```
table RawAlert {
    notice_until:   Uint64,
    id:             Uint32,
    cancel:         Uint32,
    priority:       Uint32,
    message:        Bytes,
    min_version:    BytesOpt,
    max_version:    BytesOpt,
}
``` [1](#0-0) 

There is no `chain_id`, `genesis_hash`, or any network-scoping field in `RawAlert`. The hash function simply serializes the raw struct:

```rust
pub fn calc_alert_hash(&self) -> packed::Byte32 {
    self.calc_hash()   // hash of self.as_slice() — no chain context
}
``` [2](#0-1) 

`verify_signatures` then uses this chain-agnostic hash directly as the signing message:

```rust
let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
``` [3](#0-2) 

The signing keys are identical across all networks — they are embedded in the binary via a single `alert_signature.toml` loaded as the `Default` for `NetworkAlertConfig`:

```rust
impl Default for Config {
    fn default() -> Self {
        let alert_config = include_bytes!("./alert_signature.toml");
        toml::from_slice(&alert_config[..]).expect("alert system config")
    }
}
``` [4](#0-3) 

The same four Nervos Foundation public keys are therefore active on both mainnet and testnet: [5](#0-4) 

---

### Impact Explanation

An alert legitimately signed and broadcast on CKB testnet (e.g., warning about a testnet-specific bug) produces a fully valid `packed::Alert` blob. Because the alert hash contains no network discriminator, that exact blob passes `verify_signatures` on mainnet without modification. The alert is then stored by the `Notifier`, displayed to all mainnet node operators, and re-broadcast to every connected peer via `AlertRelayer::received`. [6](#0-5) 

Conversely, a mainnet alert can be replayed on testnet. The alert system is explicitly described as being for "critical problems" — a false alert on mainnet could cause unnecessary operator panic, emergency node restarts, or loss of confidence in the network.

---

### Likelihood Explanation

- CKB is deployed on both mainnet (Mirana) and testnet (Pudge), and both use the same embedded signing keys.
- Any alert ever issued on testnet is publicly observable on the P2P network or via RPC.
- Replaying requires zero cryptographic capability: the attacker simply captures the raw `packed::Alert` bytes from testnet and submits them to a mainnet node via the `send_alert` RPC or directly over the P2P Alert protocol.
- The `send_alert` RPC is accessible to any local or remote caller with the Alert RPC module enabled. [7](#0-6) 

---

### Recommendation

Include a network-scoping value in the signed message. The most natural choice is the genesis block hash, which is unique per CKB network and already available in the node. The `RawAlert` schema should be extended with a `chain_id` (or `genesis_hash`) field, or the `Verifier` should mix the genesis hash into the message before verification:

```rust
// Conceptual fix in verify_signatures:
let mut hasher = new_blake2b();
hasher.update(alert.calc_alert_hash().as_slice());
hasher.update(consensus.genesis_hash().as_slice()); // network-specific
let mut hash = [0u8; 32];
hasher.finalize(&mut hash);
let message = Message::from_slice(&hash)?;
```

This ensures a signature produced for testnet is cryptographically invalid on mainnet and vice versa.

---

### Proof of Concept

1. On a CKB testnet node, call `send_alert` with a valid alert (signed by 2-of-4 Nervos Foundation keys). Capture the raw `packed::Alert` bytes from the P2P wire or from the RPC response.

2. On a CKB mainnet node with the Alert RPC module enabled, submit the identical bytes:

```json
{
  "jsonrpc": "2.0",
  "method": "send_alert",
  "params": [<testnet_alert_json_verbatim>],
  "id": 1
}
```

3. The mainnet node calls `verify_signatures`, computes `calc_alert_hash()` over the same `RawAlert` bytes (identical on both networks), recovers the same Nervos Foundation public keys (identical embedded config), and returns `Ok(())`.

4. The alert is stored and broadcast to all connected mainnet peers. Every mainnet node displays the testnet alert message to its operator.

The root cause is confirmed at:
- Schema: `RawAlert` has no chain identifier field. [1](#0-0) 
- Hash: `calc_alert_hash` hashes only `RawAlert` bytes. [8](#0-7) 
- Verification: message is derived solely from that chain-agnostic hash. [3](#0-2) 
- Keys: same keys embedded for all networks. [4](#0-3)

#[async_trait]
impl AlertRpc for AlertRpcImpl {
    fn send_alert(&self, alert: Alert) -> Result<()> {
        let alert: packed::Alert = alert.into();
        let now_ms = ckb_systemtime::unix_time_as_millis();
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until < now_ms {
            return Err(RPCError::invalid_params(format!(
                "Expected `params[0].notice_until` in the future (> {now_ms}), got {notice_until}",
            )));
        }

        let result = self.verifier.verify_signatures(&alert);

        match result {
            Ok(()) => {
                // set self node notifier
                self.notifier.lock().add(&alert);

                self.network_controller.broadcast_with_handle(
                    SupportProtocols::Alert.protocol_id(),
                    alert.as_bytes(),
                    &self.handle,
                );
                Ok(())
            }
            Err(e) => Err(RPCError::custom_with_error(
                RPCError::AlertFailedToVerifySignatures,
                e,
            )),
        }
    }
