Audit Report

## Title
Missing `pending_compact_blocks` Guard in `GetBlockTransactionsProcess::execute` Enables Amplified Resource Exhaustion — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary

`GetBlockTransactionsProcess::execute` performs only two input-size guards before unconditionally querying the persistent chain store for any block hash supplied by a remote peer. There is no check that the hash corresponds to an active compact-block reconstruction in `pending_compact_blocks`. Any unprivileged peer can send `GetBlockTransactions` for arbitrary committed blocks at the rate-limiter ceiling, forcing repeated disk reads, serialization, and large outbound sends — a classic small-request/large-response amplification pattern.

## Finding Description

In `sync/src/relayer/get_block_transactions_process.rs`, `execute` applies two guards:

1. `indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH` (L37–43)
2. `uncle_indexes().len() > shared.consensus().max_uncles_num()` (L44–50)

After these checks it immediately calls `shared.store().get_block(&block_hash)` (L60) with no verification that `block_hash` is present in `pending_compact_blocks`. If the block exists in the store, it collects up to 32,767 transactions and all requested uncles, serializes them into a `BlockTransactions` message, and sends it back to the peer (L60–97). [1](#0-0) [2](#0-1) 

By contrast, `BlockTransactionsProcess::execute` — the receiver-side handler — gates all processing on an `Entry::Occupied` lookup in `pending_compact_blocks` before doing any work: [3](#0-2) 

The rate limiter in `Relayer::try_process` is keyed by `(PeerIndex, message_item_id)` at 30 req/s per peer: [4](#0-3) 

With `MAX_RELAY_PEERS = 128` and `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`: [5](#0-4) 

An attacker controlling connections up to the inbound peer limit sustains thousands of `GetBlockTransactions` requests per second, each triggering a full block read from disk, collection and serialization of up to 32,767 transactions, and an outbound send proportional to the full block size.

The existing integration test `MissingUncleRequest` confirms this path is reachable: it submits a block via `node.submit_block()` (committed to the chain store, no compact-block reconstruction in progress) and then queries it directly with `GetBlockTransactions`, receiving a full `BlockTransactions` response: [6](#0-5) 

## Impact Explanation

This is a **High** severity issue matching: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

The request/response asymmetry is the core amplifier: a peer sends a small, fixed-size `GetBlockTransactions` message and forces the node to perform a full disk read of a historical block, collect and serialize up to 32,767 transactions, and transmit the result. Targeting epoch-boundary or high-throughput blocks maximizes per-request cost. With `max_peers = 125` and `max_outbound_peers = 8`, an attacker can fill up to 117 inbound slots; at 30 req/s each that is ~3,510 req/s of forced disk I/O and large outbound sends, sufficient to saturate the relay I/O thread and outbound bandwidth, degrading or crashing the node's ability to participate in block relay. [7](#0-6) 

## Likelihood Explanation

Any peer that can establish a P2P connection can execute this. Block hashes for committed blocks are publicly available from any chain explorer or synced node. No key material, hashpower, or operator access is required. The attack is repeatable, cheap for the attacker (small outbound messages, large inbound responses), and requires no coordination.

## Recommendation

Add a guard at the top of `execute` (after the existing input checks) that verifies `block_hash` is present in `pending_compact_blocks` before querying the store. If the hash is not pending, return `Status::ok()` silently, consistent with how `BlockTransactionsProcess` handles unknown hashes:

```rust
let is_pending = shared
    .state()
    .pending_compact_blocks()
    .await
    .contains_key(&block_hash);
if !is_pending {
    return Status::ok();
}
```

This aligns the sender-side guard with the receiver-side invariant already enforced in `BlockTransactionsProcess::execute`.

## Proof of Concept

1. Connect to a non-IBD CKB node as a relay peer (requires only a valid P2P handshake).
2. Obtain the hash of any committed block (e.g., genesis block hash, publicly known from any explorer).
3. Send a `RelayMessage` containing `GetBlockTransactions { block_hash: <committed_hash>, indexes: [0..N-1], uncle_indexes: [] }` where N ≤ 32,767.
4. Observe the node responds with a `BlockTransactions` message containing full transaction data — confirmed by the existing integration test `MissingUncleRequest` which does exactly this with a committed block (no `pending_compact_blocks` entry exists at any point).
5. Repeat at 30 req/s per connection; open up to 117 inbound connections to multiply throughput to ~3,510 req/s.
6. Assert: no `pending_compact_blocks` entry for the queried hash exists at any point, yet the node serves the full response — confirming the missing guard.
