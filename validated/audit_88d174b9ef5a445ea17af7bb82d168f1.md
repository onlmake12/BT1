### Title
Unsafe Truncating Cast of `withdraw_counted_capacity` from `u128` to `u64` Silently Corrupts DAO Withdrawal Capacity — (`File: util/dao/src/lib.rs`)

---

### Summary

In `calculate_maximum_withdraw`, the intermediate `u128` result `withdraw_counted_capacity` is cast to `u64` using a bare `as u64` truncating cast. If the value exceeds `u64::MAX`, the high bits are silently discarded, producing a drastically understated withdrawal capacity instead of returning a `DaoError::Overflow`. Every other analogous `u128 → u64` narrowing in the same file uses the safe `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern, making this a clear inconsistency.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The expression `withdraw_counted_capacity as u64` is a **truncating cast**: if `withdraw_counted_capacity > u64::MAX`, the upper bits are silently dropped and no error is returned.

Compare this to every other `u128 → u64` narrowing in the same file, all of which use the checked form:

- `let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;` [2](#0-1) 
- `Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)` [3](#0-2) 
- `let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;` [4](#0-3) 

The overflow condition is: `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`. Since `deposit_ar` starts at `10^16` and `withdrawing_ar` grows monotonically, this triggers when the AR ratio `withdrawing_ar / deposit_ar` exceeds approximately `u64::MAX / counted_capacity`.

The existing test `check_withdraw_calculation_overflows` is designed to catch exactly this case — it uses a capacity of `18_446_744_073_709_550_000` (near `u64::MAX`) and asserts `result.is_err()`. However, because the cast is bare `as u64`, the overflow silently truncates to a small value (~2 billion shannons), `safe_add` succeeds, and the function returns `Ok(wrong_small_value)` instead of `Err(Overflow)` — meaning the test itself is broken by this bug. [5](#0-4) 

`calculate_maximum_withdraw` is called from two paths:

1. **Consensus path**: `transaction_maximum_withdraw` → `transaction_fee`, used during block verification to check that DAO withdrawal outputs do not exceed the computed maximum. [6](#0-5) 
2. **RPC path**: `calculate_dao_maximum_withdraw` RPC, callable by any unprivileged RPC user. [7](#0-6) 

---

### Impact Explanation

If `withdraw_counted_capacity` overflows `u64` and truncates:

- The returned `withdraw_capacity` is drastically smaller than the true value.
- In the consensus path, `transaction_fee = maximum_withdraw - outputs_capacity` would underflow (since `outputs_capacity > truncated_maximum_withdraw`), causing the node to reject a legitimate DAO withdrawal transaction as invalid.
- A user with a sufficiently large DAO deposit would be permanently unable to withdraw through any node running this code.
- The RPC `calculate_dao_maximum_withdraw` would return a silently wrong (much smaller) value to callers, misleading wallet software.

---

### Likelihood Explanation

**Low in practice.** The overflow requires:

```
counted_capacity * withdrawing_ar / deposit_ar > u64::MAX ≈ 1.844 × 10^19
```

The total CKB issuance is ~33.6 billion CKB = ~3.36 × 10^18 shannons, so `counted_capacity` is bounded well below `u64::MAX`. For the product to overflow, the AR ratio `withdrawing_ar / deposit_ar` would need to exceed ~5.49×, implying a cumulative NervosDAO return of ~449%. At current secondary issuance rates this would take centuries. The bug is therefore not exploitable on mainnet in any realistic timeframe, but the code is demonstrably incorrect and the overflow-protection test is silently broken.

---

### Recommendation

Replace the bare truncating cast with the same checked pattern used everywhere else in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with the rest of the file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [8](#0-7)

### Proof of Concept

Using the values from the existing test `check_withdraw_calculation_overflows`:

- `output capacity = 18_446_744_073_709_550_000` shannons (no data, no lock → occupied ≈ 0)
- `deposit_ar = 10_000_000_000_123_456`
- `withdrawing_ar = 10_000_000_001_123_456`

```
withdraw_counted_capacity (u128)
  = 18_446_744_073_709_550_000 * 10_000_000_001_123_456
    / 10_000_000_000_123_456
  ≈ 18_446_744_075_781_548_384   // exceeds u64::MAX = 18_446_744_073_709_551_615
```

With `as u64` (truncation):
```
18_446_744_075_781_548_384 mod 2^64 ≈ 2_071_996_768
```

`Capacity::shannons(2_071_996_768).safe_add(0)` → `Ok(Capacity::shannons(2_071_996_768))`

The function returns `Ok` with ~20 CKB instead of the correct ~184 billion CKB, and the test `assert!(result.is_err())` fails silently. [1](#0-0) [9](#0-8)

                    .get(out_point.index().into())

                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),

                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
