### Title
Missing Per-Peer Rate Limiting in `LightClientProtocol` Enables Unbounded MMR Root Computation via `GetLastState` Flooding — (`util/light-client-protocol-server/src/lib.rs`)

---

### Summary

`LightClientProtocol` has no per-peer rate limiter. Any unauthenticated remote peer can send `GetLastState` messages at maximum TCP throughput, forcing a synchronous MMR root computation (`chain_root_mmr(tip_number-1).get_root()`) and a full `SendLastState` reply on every single message. The `Relayer` protocol handler in the same codebase explicitly guards against this with a `governor::RateLimiter`, but `LightClientProtocol` has no equivalent guard.

### Impact Explanation

A single unauthenticated peer can send `GetLastState` (subscribe=false) at maximum TCP throughput. Each message forces:
1. A snapshot acquisition (`self.shared.snapshot()`)
2. An MMR root computation reading `O(log N)` RocksDB entries (at mainnet height ~10M blocks, ~23 DB reads per call)
3. Serialization and transmission of a `SendLastState` reply

At thousands of messages per second, this saturates the node's RocksDB I/O and async executor threads, degrading or blocking service to all other peers (sync, relay, legitimate light clients). The attacker's cost is a single TCP connection and outbound bandwidth.

---

### Likelihood Explanation

The light client protocol is a production feature enabled on full nodes that opt in. The attack requires only a TCP connection — no authentication, no PoW, no stake. The `GetLastState` message is the simplest possible message in the protocol (a single boolean field). The absence of rate limiting is a straightforward oversight evidenced by the fact that the developers explicitly added rate limiting to `Relayer` for the same reason.

---

### Recommendation

Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`, mirroring the pattern in `Relayer`. Check the rate limit at the top of `try_process` before dispatching any message. A quota of 10–30 req/s per `(peer, message_type)` pair is consistent with the existing `Relayer` policy. Additionally, consider caching the last computed MMR root with a short TTL so that even un-rate-limited calls do not each hit RocksDB.

    pub(crate) async fn execute(self) -> Status {
        let subscribe: bool = self.message.subscribe().into();
        if subscribe {
            self.nc.with_peer_mut(
                self.peer,
                Box::new(|peer| {
                    peer.if_lightclient_subscribed = true;
                }),
            );
        }

        let tip_header = match self.protocol.get_verifiable_tip_header() {
            Ok(tip_state) => tip_state,
            Err(errmsg) => {
                return StatusCode::InternalError.with_context(errmsg);
            }
        };

        let content = packed::SendLastState::new_builder()
            .last_header(tip_header)
            .build();
        let message = packed::LightClientMessage::new_builder()
            .set(content)
            .build();

        self.nc.reply(self.peer, &message).await
    }
