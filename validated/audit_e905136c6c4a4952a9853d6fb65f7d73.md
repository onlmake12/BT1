### Title
Missing rate limiting for `GetHeaders` in `Synchronizer` allows CPU/IO amplification by any connected peer — (`sync/src/synchronizer/get_headers_process.rs`)

### Summary
Any connected peer can flood `GetHeaders` messages at an unbounded rate post-IBD. Each message triggers up to ~2,101 DB reads with no per-peer throttle, because `getheaders_received` is an unimplemented TODO stub and the `Synchronizer` has no rate limiter (unlike `Relayer`, which does).

### Finding Description
`GetHeadersProcess::execute` performs two expensive DB operations per message: `locate_latest_common_block` (up to 101 hash lookups) and `get_locator_response` (up to 2,000 sequential block header reads). The call to `self.synchronizer.peers().getheaders_received(self.peer)` is a no-op stub. The `Synchronizer` struct has no `rate_limiter` field and no equivalent of the `Relayer`'s per-peer-per-message-type governor check. An attacker sends a valid locator (any real block hashes + genesis hash) in a tight loop; the IBD guard is bypassed since the node is out of IBD; `locate_latest_common_block` succeeds and `get_locator_response` is called every iteration.

### Impact Explanation
Sustained flooding saturates the RocksDB I/O path and the tokio `block_in_place` thread pool used for sync message processing, degrading block download throughput for all legitimate peers. Scoped impact: Low (501–2000) — measurable sync performance degradation.

### Likelihood Explanation
Any connected peer (inbound or outbound, no special flag required) can execute this. The attacker only needs a valid TCP connection and knowledge of the genesis hash (public). No PoW, no key, no privilege required.

### Recommendation
Add a per-peer rate limiter to `Synchronizer` mirroring the one in `Relayer` (30 req/s keyed by `(PeerIndex, message_item_id)`), and implement `getheaders_received` to track and enforce a per-peer `GetHeaders` budget.

### Proof of Concept
1. Connect a peer to a non-IBD CKB node.
2. In a tight loop, send `GetHeaders` with `block_locator_hashes = [<any valid tip hash>, <genesis hash>]`.
3. Each message passes the `MAX_LOCATOR_SIZE` check, bypasses the IBD guard, calls `locate_latest_common_block` (DB reads), then `get_locator_response` (up to 2,000 DB reads), and spawns an async send task.
4. Benchmark legitimate peer sync throughput (blocks/s) with and without the flood; assert measurable degradation.

    pub fn locate_latest_common_block(
        &self,
        _hash_stop: &Byte32,
        locator: &[Byte32],
    ) -> Option<BlockNumber> {
        if locator.is_empty() {
            return None;
        }

        let locator_hash = locator.last().expect("empty checked");
        if locator_hash != &self.sync_shared.consensus().genesis_hash() {
            return None;
        }

        // iterator are lazy
        let (index, latest_common) = locator
            .iter()
            .enumerate()
            .map(|(index, hash)| (index, self.snapshot.get_block_number(hash)))
            .find(|(_index, number)| number.is_some())
            .expect("locator last checked");

        if index == 0 || latest_common == Some(0) {
            return latest_common;
        }

        if let Some(header) = locator
            .get(index - 1)
            .and_then(|hash| self.sync_shared.store().get_block_header(hash))
        {
            let mut block_hash = header.data().raw().parent_hash();
            loop {
                let block_header = match self.sync_shared.store().get_block_header(&block_hash) {
                    None => break latest_common,
                    Some(block_header) => block_header,
                };

                if let Some(block_number) = self.snapshot.get_block_number(&block_hash) {
                    return Some(block_number);
                }

                block_hash = block_header.data().raw().parent_hash();
            }
        } else {
            latest_common
        }
    }
