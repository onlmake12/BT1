### Title
Entropy Provider Randomness Pool Permanently Exhausted by Permissionless Request Spam, Rendering Provider Unavailable — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

An unprivileged attacker can exhaust any Entropy provider's bounded randomness pool by repeatedly calling `requestV2()` or `requestWithCallback()` while paying the required fee. Once `providerInfo.sequenceNumber >= providerInfo.endSequenceNumber`, every subsequent randomness request to that provider reverts with `OutOfRandomness`, making the provider permanently unavailable until it re-registers. This is a direct on-chain analog to the "pool becomes unavailable" vulnerability class in the external report.

---

### Finding Description

Each Entropy provider registers with a finite hash chain of length `chainLength`. At registration, `endSequenceNumber` is set to `provider.sequenceNumber + chainLength`: [1](#0-0) 

The `ProviderInfo` struct documents the invariant explicitly:

> "If `sequenceNumber == endSequenceNumber`, the provider must rotate their commitment to add additional random values." [2](#0-1) 

Inside `requestHelper()`, every accepted request unconditionally increments `sequenceNumber`: [3](#0-2) 

The guard that blocks new requests once the pool is exhausted: [4](#0-3) 

The error definition: [5](#0-4) 

Because `requestV2()` and `requestWithCallback()` are fully permissionless — any address may call them by paying the fee — an attacker can submit `chainLength - currentSequenceNumber` requests, consuming every remaining slot. The attacker does **not** need to reveal the random numbers; sequence numbers are consumed and `sequenceNumber` is incremented regardless of whether the requests are ever fulfilled.

A secondary, cheaper path exists when `maxNumHashes` is non-zero. An attacker only needs to submit `maxNumHashes + 1` unfulfilled requests to trigger `LastRevealedTooOld` for all subsequent callers: [6](#0-5) 

---

### Impact Explanation

Once `sequenceNumber == endSequenceNumber`, every call to `requestV2()`, `requestWithCallback()`, or `request()` for that provider reverts with `OutOfRandomness`. The default provider — used by all callers of the zero-argument `requestV2()` — is the highest-value target: [7](#0-6) 

All downstream contracts that depend on Entropy for randomness (e.g., NFT mints, lotteries, gaming protocols) are completely blocked from requesting new random numbers until the provider calls `register()` again to extend the chain. The outage is permanent until the provider acts, mirroring the "pool becomes unavailable" pattern in the external report.

---

### Likelihood Explanation

The attack is permissionless and requires no privileged access. The economic cost scales with `chainLength × feeInWei`. For providers with small chain lengths or low fees, the attack is cheap. For the default Fortuna provider with a large chain, the cost is higher but not infinite — a well-funded attacker targeting a high-value protocol that depends on Entropy can absorb the cost. The attacker also accrues no benefit from the consumed sequence numbers, making this a pure griefing/DoS vector. Likelihood is rated **high** for small-chain providers and **moderate** for large-chain providers.

---

### Recommendation

1. **Providers**: Register with very large `chainLength` values (e.g., 2^32) to make exhaustion economically infeasible.
2. **Protocol**: Add a minimum `chainLength` floor enforced in `register()` to prevent providers from accidentally registering with small chains.
3. **Protocol**: Consider allowing providers to extend `endSequenceNumber` in-place (without full re-registration) so recovery is faster and does not disrupt in-flight requests.
4. **Protocol**: Consider per-address rate limiting or a maximum number of unfulfilled requests per address to raise the cost of the `LastRevealedTooOld` variant.

---

### Proof of Concept

```solidity
// Attacker exhausts a provider with chainLength = C
// Assumes attacker has sufficient ETH to pay C * fee

address provider = entropy.getDefaultProvider();
uint128 fee = entropy.getFee(provider);
uint64 seqNum;

for (uint i = 0; i < C; i++) {
    seqNum = entropy.requestV2{value: fee}();
    // Do NOT reveal — sequence numbers are consumed regardless
}

// Now sequenceNumber == endSequenceNumber
// All subsequent calls revert with OutOfRandomness:
entropy.requestV2{value: fee}(); // reverts: OutOfRandomness
```
