### Title
Unbounded `GetHeaders` Request Processing Without Per-Peer Rate Limiting or Response Caching Enables Resource Exhaustion — (`sync/src/synchronizer/get_headers_process.rs`)

---

### Summary

The `GetHeadersProcess::execute()` handler processes every incoming `GetHeaders` P2P message by fetching up to 2,000 block headers from the database and transmitting them to the requesting peer, with no per-peer rate limiting and no caching of responses. An unprivileged sync peer can repeatedly send `GetHeaders` messages to force the node to perform expensive, redundant database reads and network transmissions indefinitely.

---

### Finding Description

In `sync/src/synchronizer/get_headers_process.rs`, the `execute()` function handles incoming `GetHeaders` sync protocol messages. For each valid request, it calls `active_chain.get_locator_response()`: [1](#0-0) 

`get_locator_response` is defined in `sync/src/types/mod.rs` and fetches up to `MAX_HEADERS_LEN` (2,000) block headers from the database by iterating through block numbers and loading each header individually from the store: [2](#0-1) 

The constant `MAX_HEADERS_LEN` is 2,000: [3](#0-2) 

The only input validation performed before this expensive operation is:
1. Checking that the locator size does not exceed `MAX_LOCATOR_SIZE` (101 entries)
2. Checking if the node is in IBD mode (sends `InIBD` instead) [4](#0-3) 

There is **no per-peer rate limiting** on incoming `GetHeaders` requests and **no caching** of the response. The same 2,000 headers are fetched from the database on every request, even if the chain tip has not changed since the last identical request.

By contrast, the node's *outgoing* `GetHeaders` requests do have a rate-limiting cache (`pending_get_headers` with `GET_HEADERS_TIMEOUT = 15s`): [5](#0-4) 

This asymmetry means the node protects itself from sending redundant outgoing requests but applies no equivalent protection against receiving redundant incoming ones.

The `Synchronizer` registers the `SEND_GET_HEADERS_TOKEN` timer at a 1-second interval, but the `GetHeaders` *handler* itself is invoked on every received message with no throttle: [6](#0-5) 

---

### Impact Explanation

A single unprivileged sync peer can send a continuous stream of valid `GetHeaders` messages (each with a locator of up to 101 hashes). For each message, the node:

1. Performs up to 2,000 sequential database lookups (`get_block_hash` + `get_block_header`) from RocksDB
2. Serializes the resulting headers into a `SendHeaders` message
3. Transmits the message back to the peer

Since there is no caching, the same headers are re-fetched from disk on every request even when the chain tip is unchanged — directly analogous to the report's "re-process the same data repeatedly, despite having a local cache." This causes sustained CPU and I/O load, degrading the node's ability to serve legitimate peers and process new blocks.

---

### Likelihood Explanation

Any peer that completes the sync protocol handshake can send `GetHeaders` messages. No special privileges, keys, or majority hashpower are required. The attack is trivially scriptable using the standard CKB sync wire protocol. The `BAD_MESSAGE_BAN_TIME` ban is only applied for *malformed* messages; a well-formed but high-frequency `GetHeaders` stream is never penalized. [7](#0-6) 

---

### Recommendation

1. **Add per-peer rate limiting** on incoming `GetHeaders` requests (e.g., a token-bucket or sliding-window counter per `PeerIndex`). Peers exceeding the limit should be penalized or disconnected.
2. **Cache `get_locator_response` results** keyed on `(block_number, hash_stop)` and invalidated on chain tip change, so repeated identical requests do not re-read the database.
3. Consider applying the same `GET_HEADERS_TIMEOUT` guard that already exists for outgoing requests to incoming ones as well.

---

### Proof of Concept

1. Establish a sync protocol connection to a CKB full node as an unprivileged peer.
2. In a tight loop, send `GetHeaders` messages with a valid block locator (e.g., the genesis hash as the single locator entry, `hash_stop = 0x00…00`).
3. Each message passes all validation checks in `GetHeadersProcess::execute()` and triggers a fetch of up to 2,000 headers from the database.
4. Monitor the node's RocksDB I/O and CPU usage; both will climb proportionally to the request rate with no self-limiting behavior on the node's side. [8](#0-7) [9](#0-8)

    pub fn execute(self) -> Status {
        let active_chain = self.synchronizer.shared.active_chain();

        let block_locator_hashes = self
            .message
            .block_locator_hashes()
            .iter()
            .map(|x| x.to_entity())
            .collect::<Vec<Byte32>>();
        let hash_stop = self.message.hash_stop().to_entity();
        let locator_size = block_locator_hashes.len();
        if locator_size > MAX_LOCATOR_SIZE {
            return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                "Locator count({locator_size}) > MAX_LOCATOR_SIZE({MAX_LOCATOR_SIZE})"
            ));
        }

        if active_chain.is_initial_block_download() {
            info!(
                "Ignoring getheaders from peer={} because the node is in initial block download stage.",
                self.peer
            );
            self.send_in_ibd();
            let shared = self.synchronizer.shared();
            if let Some(flag) = shared.state().peers().get_flag(self.peer)
                && (flag.is_outbound || flag.is_whitelist || flag.is_protect)
            {
                shared.insert_peer_unknown_header_list(self.peer, block_locator_hashes);
            };
            return Status::ignored();
        }

        if let Some(block_number) =
            active_chain.locate_latest_common_block(&hash_stop, &block_locator_hashes[..])
        {
            debug!(
                "headers latest_common={} tip={} begin",
                block_number,
                active_chain.tip_header().number(),
            );

            self.synchronizer.peers().getheaders_received(self.peer);
            let headers: Vec<core::HeaderView> =
                active_chain.get_locator_response(block_number, &hash_stop);
            // response headers

            debug!("headers len={}", headers.len());

            let content = packed::SendHeaders::new_builder()
                .headers(headers.into_iter().map(|x| x.data()).collect::<Vec<_>>())
                .build();
            let message = packed::SyncMessage::new_builder().set(content).build();
            let nc = Arc::clone(self.nc);
            self.synchronizer
                .shared()
                .shared()
                .async_handle()
                .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
        } else {
            return StatusCode::GetHeadersMissCommonAncestors
                .with_context(format!("{block_locator_hashes:#x?}"));
        }
        Status::ok()
    }

    pub fn get_locator_response(
        &self,
        block_number: BlockNumber,
        hash_stop: &Byte32,
    ) -> Vec<core::HeaderView> {
        let tip_number = self.tip_header().number();
        let Some(start_number) = block_number.checked_add(1) else {
            return Vec::new();
        };
        std::iter::successors(Some(start_number), |number| number.checked_add(1))
            .take_while(|number| *number <= tip_number)
            .take(MAX_HEADERS_LEN)
            .filter_map(|block_number| self.snapshot.get_block_hash(block_number))
            .take_while(|block_hash| block_hash != hash_stop)
            .filter_map(|block_hash| self.sync_shared.store().get_block_header(&block_hash))
            .collect()
    }
