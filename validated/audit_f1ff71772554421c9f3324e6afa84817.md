Audit Report

## Title
Unconditional Orphan Retention on `Reject::Full` in High-Cycle Path Allows Orphan Pool Saturation — (`tx-pool/src/process.rs`)

## Summary
In `process_orphan_tx`, the high-cycle branch (orphan declared cycle > `max_tx_verify_cycles`) calls `enqueue_verify_queue` and, on any error, unconditionally retains the orphan in the pool. The low-cycle branch correctly removes orphans on non-missing-input errors. An attacker can fill the verify queue to its 256 MB hard cap with structurally valid, fee-invalid transactions (fee validation is deferred), then submit orphan parents — causing every high-cycle orphan to receive `Reject::Full` and remain permanently in the 100-slot orphan pool, evicting legitimate orphan transactions.

## Finding Description

**High-cycle error branch — no eviction:** [1](#0-0) 

The `Err(reject)` arm only logs a warning. There is no call to `remove_orphan_tx`, so the orphan is unconditionally retained regardless of the rejection reason.

**Low-cycle error branch — correct eviction:** [2](#0-1) 

The low-cycle path calls `is_missing_input` and removes the orphan on any non-missing-input error. The high-cycle path never calls `is_missing_input` at all.

**`is_missing_input` definition:** [3](#0-2) 

`Reject::Full` does not match `Reject::Resolve(...)`, so `is_missing_input` would return `false` for it — but the high-cycle path never reaches this check.

**`enqueue_verify_queue` returns `Reject::Full` at 256 MB:** [4](#0-3) [5](#0-4) 

**Orphan pool hard cap and random eviction:** [6](#0-5) [7](#0-6) 

**Fee check is deferred past `enqueue_verify_queue`:** [8](#0-7) 

`resumeble_process_tx` only calls `non_contextual_verify` (structure only) before enqueuing. Fee validation (`check_tx_fee`) runs inside `_process_tx → pre_check`, which executes only after dequeue. This means the attacker can fill 256 MB of verify queue with structurally valid, fee-invalid transactions at near-zero cost.

## Impact Explanation

Once the verify queue is full and the orphan parent is accepted, every high-cycle orphan in the pool receives `Reject::Full` from `enqueue_verify_queue`. Because the error branch has no eviction logic, all 100 orphan slots remain occupied by attacker-controlled entries whose parent is already confirmed but which can never advance to the verify queue. Legitimate orphan transactions submitted by honest peers are immediately evicted by `limit_size` random eviction. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The attack is cheap (fee-invalid transactions cost nothing to produce), repeatable, and can be applied to multiple nodes simultaneously to disrupt orphan transaction relay across the network.

## Likelihood Explanation

All preconditions are reachable by an unprivileged peer via standard P2P transaction relay:
1. Submit enough structurally valid (but fee-invalid) transactions to fill 256 MB of verify queue — cost is near-zero since fee validation is deferred.
2. Pre-stage up to 100 orphan transactions with `declared_cycle > max_tx_verify_cycles` spending outputs of a not-yet-submitted parent.
3. Submit the parent transaction; `process_orphan_tx` fires and all orphans receive `Reject::Full`, remaining stranded.

No keys, hashpower, or privileged access are required. The only mitigating factor is that orphans expire after `ORPHAN_TX_EXPIRE_TIME = 100 × MAX_BLOCK_INTERVAL`, so the attacker must periodically re-stage the attack to sustain the effect. [9](#0-8) 

## Recommendation

In the high-cycle error branch of `process_orphan_tx`, apply the same eviction logic used by the low-cycle path:

```rust
Err(reject) => {
    warn!(
        "process_orphan {} failed to enqueue verify queue: {}; keep orphan from {}",
        orphan.tx.hash(),
        reject,
        tx.hash(),
    );
    if !is_missing_input(&reject) {
        self.remove_orphan_tx(&orphan_id).await;
    }
}
```

This ensures that a transient `Reject::Full` (or any other non-missing-input error) does not permanently strand orphans whose parent has already been resolved.

## Proof of Concept

```
1. Connect to a CKB node as a peer via standard P2P.
2. Submit ~N structurally valid transactions (pass non_contextual_verify,
   fail fee check) large enough to fill verify_queue to 256 MB.
   These remain in the queue since fee validation is deferred.
3. Submit 100 orphan transactions O_1..O_100, each:
   - spending an output of a not-yet-submitted parent P
   - with declared_cycle > max_tx_verify_cycles
   These enter the orphan pool (DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100).
4. Submit parent P. P passes verification and is accepted.
   after_process calls process_orphan_tx(P).
5. For each O_i: enqueue_verify_queue returns Err(Reject::Full).
   The Err branch logs a warning and does NOT call remove_orphan_tx.
6. Assert: orphan pool still contains all 100 O_i entries.
7. Submit a legitimate orphan L: it is evicted immediately by limit_size
   because the pool is at capacity with attacker-controlled entries.
``` [10](#0-9) [7](#0-6)
