### Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Amounts - (File: `util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes an intermediate `u128` result for the withdrawal amount and then casts it to `u64` using the silent truncating `as u64` operator. When the intermediate value exceeds `u64::MAX`, the result is silently wrong — producing a drastically under-reported withdrawal capacity — rather than returning an error. This is directly analogous to the `pow_u32` bug: a numeric computation that silently yields incorrect output for a specific class of inputs (large DAO deposits with high accumulated-rate ratios), without any error being raised.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The expression `withdraw_counted_capacity as u64` is a **wrapping/truncating cast** in Rust. If `withdraw_counted_capacity` (a `u128`) exceeds `u64::MAX` (≈ 1.84 × 10¹⁹), the high bits are silently discarded, producing a value that is orders of magnitude smaller than the correct result. No error is returned; the function returns `Ok` with a wrong capacity.

This is inconsistent with every other analogous computation in the same codebase. Both `dao_field_with_current_epoch` and `secondary_block_reward` use checked conversions:

```rust
// dao_field_with_current_epoch — checked:
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

```rust
// secondary_block_reward — checked:
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

`calculate_maximum_withdraw` is the sole outlier that uses the unsafe `as u64` cast.

The existing overflow test (`check_withdraw_calculation_overflows`) only catches the case where the truncated value plus `occupied_capacity` happens to overflow `u64` in the subsequent `safe_add`. For inputs where the truncated value is small enough that `safe_add` succeeds, the function returns `Ok` with a silently incorrect result. [4](#0-3) 

### Impact Explanation

`calculate_maximum_withdraw` is called from two production paths:

**1. RPC path — `calculate_dao_maximum_withdraw`:**
Any RPC caller can invoke this endpoint with a valid DAO cell out-point and a withdrawing block hash. If the deposit is large enough and the AR ratio has grown sufficiently, the RPC returns a silently wrong (too-small) withdrawal amount. A depositor relying on this value to construct their withdrawal transaction would build a transaction with an incorrect output capacity. [5](#0-4) 

**2. Tx-pool admission path — `transaction_fee` → `transaction_maximum_withdraw`:**
When a DAO withdrawal transaction is submitted to the tx-pool, `transaction_fee` calls `transaction_maximum_withdraw`, which calls `calculate_maximum_withdraw`. If the truncation produces a value smaller than the transaction's output capacity, `maximum_withdraw.safe_sub(outputs_capacity)` fails, and the node **incorrectly rejects a valid DAO withdrawal transaction**. This is a consensus-relevant tx-pool admission error: a transaction that the on-chain DAO script would accept is rejected by the node's fee calculation. [6](#0-5) 

### Likelihood Explanation

The overflow condition requires:

```
counted_capacity × withdrawing_ar > u64::MAX × deposit_ar
```

- `counted_capacity` can be up to ~3.36 × 10¹⁸ shannons (total CKB supply)
- `deposit_ar` starts at 10¹⁶ (genesis)
- For overflow: `withdrawing_ar > (1.84 × 10¹⁹ × deposit_ar) / counted_capacity`

With a large deposit (e.g., 10¹⁸ shannons) and `deposit_ar = 10¹⁶`, overflow occurs when `withdrawing_ar > ~1.84 × 10¹⁷`, i.e., when AR has grown by ~18×. AR grows continuously with each block's secondary issuance. Over a multi-year horizon with large locked deposits, this threshold is reachable. The RPC path is reachable immediately by any caller who can reference a block with a sufficiently high AR value.

### Recommendation

Replace the silent truncating cast with a checked conversion, consistent with the rest of the codebase:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with other functions):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [1](#0-0) 

### Proof of Concept

```
deposit_ar      = 10_000_000_000_000_000   (10^16, genesis AR)
withdrawing_ar  = 200_000_000_000_000_000  (2×10^17, ~20× growth)
counted_capacity = 3_000_000_000_000_000_000  (3×10^18 shannons, ~30 billion CKB)

withdraw_counted_capacity (u128) =
    3_000_000_000_000_000_000 × 200_000_000_000_000_000
    / 10_000_000_000_000_000
  = 60_000_000_000_000_000_000   (6×10^19)

u64::MAX = 18_446_744_073_709_551_615   (≈1.84×10^19)

6×10^19 > u64::MAX  →  truncation occurs

withdraw_counted_capacity as u64 =
    60_000_000_000_000_000_000 % 2^64
  = 60_000_000_000_000_000_000 - 3 × 18_446_744_073_709_551_616
  = 4_659_767_779_871_344_752   (≈4.66×10^18)

Correct result:  6.0×10^19 shannons
Returned result: 4.66×10^18 shannons  (≈92% loss of funds reported)
```

The function returns `Ok(Capacity::shannons(4_659_767_779_871_344_752 + occupied_capacity))` — a value that is ~12.9× smaller than the correct withdrawal amount — with no error, no log, and no indication that truncation occurred. [7](#0-6) [8](#0-7)

                            match self.data_loader.load_cell_data(cell_meta) {

                                .transaction

                                .ok_or(DaoError::InvalidOutPoint)
                                .and_then(|witness_data| {
                                    // dao contract stores header deps index as u64 in the input_type field of WitnessArgs
                                    let witness =

                                            .map_err(|_| DaoError::InvalidDaoFormat)?;
                                    let header_deps_index_data: Option<Bytes> =
                                        witness.input_type().to_opt().map(|witness| witness.into());
                                    if header_deps_index_data.is_none()
                                        || header_deps_index_data.clone().map(|data| data.len())
                                            != Some(8)
                                    {
                                        return Err(DaoError::InvalidDaoFormat);
                                    }
                                    Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
                                })
                                .and_then(|header_dep_index| {
                                    rtx.transaction
                                        .header_deps()
                                        .get(header_dep_index as usize)
                                        .and_then(|hash| header_deps.get(&hash))
                                        .ok_or(DaoError::InvalidOutPoint)
                                })?;

                            let deposit_header = self
                                .data_loader
                                .get_header(deposit_header_hash)
                                .ok_or(DaoError::InvalidHeader)?;
                            if deposit_header.number() != deposited_block_number {
                                return Err(DaoError::InvalidOutPoint);
                            }
                            self.calculate_maximum_withdraw(
                                output,
                                Capacity::bytes(cell_meta.data_bytes as usize)?,
                                deposit_header_hash,
                                withdrawing_header_hash,
                            )
                        } else {
                            Ok(output.capacity().into())
                        }
                    } else {
                        Ok(output.capacity().into())
                    }
                };
                capacity.and_then(|c| c.safe_add(capacities).map_err(Into::into))
            },
        )
    }

    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }
