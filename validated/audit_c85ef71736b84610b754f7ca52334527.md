### Title
Inconsistent Fee Metric in Tx-Pool Admission: `check_tx_fee` Uses Raw Size Instead of Weight — (`tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` enforces the minimum fee rate using raw serialized byte size (`tx_size`) as the weight denominator, but `FeeRate` is defined and documented as **shannons per kilo-weight**, where `weight = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For compute-heavy transactions where cycles dominate, `weight >> size`, so the admission gate underestimates the true minimum fee. An unprivileged tx-pool submitter can craft a small-but-cycle-intensive transaction that passes the fee check while paying an effective fee rate far below `min_fee_rate`. The same inconsistency appears in `calculate_min_replace_fee` for RBF.

---

### Finding Description

`FeeRate` is explicitly typed as "shannons per kilo-weight": [1](#0-0) 

The canonical weight function is: [2](#0-1) 

`weight = max(tx_size, cycles × 0.000_170_571_4)`. For a compute-heavy transaction, cycles dominate and `weight >> size`.

However, `check_tx_fee` — the sole admission gate — computes the minimum fee using raw `tx_size`, not weight: [3](#0-2) 

The code comment itself acknowledges the inconsistency: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly."* There is no subsequent weight-based fee rate check before the transaction is admitted to the pool.

By contrast, once a transaction is inside the pool, its fee rate is computed correctly using weight: [4](#0-3) 

The same size-only metric is used in the RBF extra-fee calculation: [5](#0-4) 

---

### Impact Explanation

An attacker submits a transaction with:
- Small serialized size (e.g., 100 bytes)
- High cycle consumption (e.g., 10,000,000 cycles via a CKB-VM script)

Concrete numbers with default `min_fee_rate = 1000 shannons/KW`:

| Metric | Value |
|---|---|
| `tx_size` | 100 bytes |
| `cycles` | 10,000,000 |
| `weight` = max(100, 10M × 0.000_170_571_4) | **1,705** |
| `min_fee` via `check_tx_fee` (uses size) | 100 shannons |
| True `min_fee` (uses weight) | 1,705 shannons |
| Effective fee rate of admitted tx | **~58 shannons/KW** (5.8% of minimum) |

A transaction paying only 100 shannons passes `check_tx_fee` and is admitted to the pool, even though its true fee rate is ~58 shannons/KW — far below the configured floor of 1,000 shannons/KW.

Consequences:
1. **Mempool pollution**: Underpriced compute-heavy transactions fill the pool, displacing legitimately-priced transactions.
2. **Relay propagation**: Admitted transactions are relayed to peers, spreading the underpriced load across the network.
3. **RBF bypass**: The `calculate_min_replace_fee` function also uses raw `size`, so the extra RBF fee for a compute-heavy replacement transaction is similarly underestimated, allowing cheaper-than-intended replacements.

---

### Likelihood Explanation

CKB-VM is a general-purpose RISC-V VM. Any script author can deploy a lock or type script that consumes many cycles while keeping the transaction's serialized size small (e.g., a tight computational loop in a compact binary). This is a standard capability of the platform, not an exotic attack. The attacker only needs to submit a transaction via the public `send_transaction` RPC — no privileged access is required. [6](#0-5) 

The ratio `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` means that at `max_tx_verify_cycles = 70,000,000`, the weight contribution from cycles alone is ~11,940 bytes. A transaction with 70M cycles but only 200 bytes of serialized size would have `weight ≈ 11,940`, meaning the size-based fee check requires only ~200 shannons while the weight-based minimum would be ~11,940 shannons — a **~60× undercharge**.

---

### Recommendation

Replace the size-only fee check in `check_tx_fee` with a weight-based check. Since cycles are not yet known at the pre-verification stage (they are computed during script execution), the check can be deferred to after `verify_rtx` returns the `Completed` entry containing the actual cycle count, or a conservative upper-bound estimate using `max_tx_verify_cycles` can be applied as a pre-filter:

```rust
// After verify_rtx returns cycles:
let weight = get_transaction_weight(tx_size, completed.cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

Apply the same fix to `calculate_min_replace_fee` in `tx-pool/src/pool.rs`: [7](#0-6) 

Use `get_transaction_weight(size, entry.cycles)` instead of raw `size as u64`.

---

### Proof of Concept

1. Deploy a CKB-VM lock script that runs a tight loop consuming ~10,000,000 cycles but compiles to ≤ 200 bytes.
2. Construct a transaction spending a cell locked by that script. Serialized size ≈ 200 bytes.
3. Set the transaction fee to 200 shannons (passes `check_tx_fee`: `1000 × 200 / 1000 = 200`).
4. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000`.
5. Observe the transaction is accepted into the pool.
6. Verify the actual fee rate: `weight = max(200, 10_000_000 × 0.000_170_571_4) = 1705`; effective fee rate = `1000 × 200 / 1705 ≈ 117 shannons/KW` — well below the 1000 shannons/KW minimum.
7. Repeat with many such transactions to fill the mempool with underpriced compute-heavy entries, crowding out legitimately-priced transactions. [8](#0-7) [2](#0-1)
