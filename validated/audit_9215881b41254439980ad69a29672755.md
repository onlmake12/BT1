### Title
Unrestricted `initialize_pool` Allows Attacker to Hijack Integrity Pool Authority — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `initialize_pool` instruction of the Pyth integrity pool program accepts any signer as `payer` with no authority constraint linking it to the staking program's governance authority. An attacker who calls this instruction before the protocol can set an arbitrary `reward_program_authority`, permanently seizing control of the slashing mechanism for all stakers.

---

### Finding Description

The `initialize_pool` instruction in the integrity pool program is the one-time bootstrapping call that writes the global `pool_config` PDA (seed `pool_config`). It accepts two critical arguments:

- `reward_program_authority` — the pubkey that is later required as a signer for `create_slash_event`
- `y` — the reward-rate parameter

The IDL account list for this instruction is:

| Account | Constraint |
|---|---|
| `payer` | `signer: true, writable: true` |
| `config_account` | PDA from staking program (read-only, no `relations`) |
| `pool_data` | `writable: true` |
| `pool_config` | PDA `[pool_config]`, `writable: true` |
| `slash_custody` | (read-only) |
| `system_program` | — |

The `payer` account carries **no `relations` constraint** tying it to `config_account.governance_authority` or any other privileged key. The `config_account` (the staking program's global config, which stores the governance authority) is passed in but is never used as an Anchor-level authority gate on `payer`.

Because `pool_config` is a PDA that can only be `init`-ialized once, the first caller wins permanently. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Once `pool_config` is initialized with an attacker-controlled `reward_program_authority`, the attacker becomes the sole account authorized to call `create_slash_event`. The `create_slash_event` instruction enforces `rewardProgramAuthority` as a required signer via `relations: ["poolConfig"]`: [3](#0-2) 

This means the attacker can:

1. **Create arbitrary slash events** against any publisher, triggering slashing of delegated PYTH tokens from legitimate stakers.
2. **Block legitimate slashing** by never calling `create_slash_event`, undermining the integrity of the OIS mechanism.
3. **Permanently brick the pool** — there is no `update_reward_program_authority` instruction visible in the IDL, so recovery requires a full program redeployment.

The `y` parameter (reward rate) can also be set to zero or an extreme value, corrupting reward distribution for all stakers.

---

### Likelihood Explanation

Solana has no public mempool, so classic front-running is not possible. However, the attack window is the gap between program deployment and the protocol's own `initialize_pool` call. Any account monitoring the chain for the new program deployment can race to call `initialize_pool` first. The call requires only a small SOL balance for rent (the `payer` pays for the `pool_config` account). No special privileges, tokens, or insider knowledge are needed. [4](#0-3) 

---

### Recommendation

Add an authority constraint to the `initialize_pool` instruction that requires `payer` to equal the `governance_authority` stored in `config_account`. In Anchor this is expressed as a `has_one` or `constraint` on the `payer` account:

```rust
#[account(
    constraint = payer.key() == config_account.governance_authority
        @ IntegrityPoolError::Unauthorized
)]
pub payer: Signer<'info>,
```

Alternatively, make the deployment and initialization atomic in a single transaction so no race window exists.

---

### Proof of Concept

1. Protocol deploys the integrity pool program on-chain.
2. Attacker observes the new program account on-chain (no mempool needed).
3. Attacker submits a transaction calling `initialize_pool` with:
   - `payer` = attacker's keypair
   - `pool_data` = any freshly created account
   - `reward_program_authority` = attacker's pubkey
   - `y` = 0 (or any value)
4. Because `pool_config` PDA does not yet exist, Anchor's `init` constraint succeeds and the account is created with attacker's address as `reward_program_authority`.
5. Protocol's own `initialize_pool` call now reverts (PDA already exists).
6. Attacker calls `create_slash_event` for any publisher, slashing delegated staker funds at will. [5](#0-4) [6](#0-5)

  public async initializePool({
    rewardProgramAuthority,
    poolData,
    y,
  }: {
    rewardProgramAuthority: PublicKey;
    poolData: PublicKey;
    y: bigint;
  }) {
    const yAnchor = convertBigIntToBN(y);
    const instruction = await this.integrityPoolProgram.methods
      .initializePool(rewardProgramAuthority, yAnchor)
      .accounts({
        poolData,
        slashCustody: getStakeAccountCustodyAddress(poolData),
      })
      .instruction();

    return sendTransaction([instruction], this.connection, this.wallet);
  }
