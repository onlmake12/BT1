### Title
Unrestricted `executeCallback` Allows Fee Theft and Forced Consumer Callback Execution - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol`'s `executeCallback` function has no `msg.sender` authorization check. Any unprivileged caller can invoke it after the exclusivity period expires, redirect the request fee to an attacker-controlled provider address, and force the consumer's `_echoCallback` to execute with attacker-supplied (but Pyth-valid) price data.

### Finding Description

`executeCallback` is the fulfillment entry point for Echo price-update requests. Its only caller restriction is an exclusivity-period guard that checks the `providerToCredit` *argument*, not `msg.sender`:

```solidity
// Echo.sol lines 113-121
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
``` [1](#0-0) 

After `req.publishTime + exclusivityPeriodSeconds` elapses, the guard is skipped entirely. The function then:

1. Credits fees to the caller-supplied `providerToCredit`:

```solidity
// Echo.sol lines 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

2. Calls the consumer's `_echoCallback` via the Echo contract (which the consumer trusts unconditionally):

```solidity
// Echo.sol lines 176-179
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
``` [3](#0-2) 

The consumer's guard in `IEchoConsumer._echoCallback` only checks `msg.sender == echo`:

```solidity
// IEcho.sol lines 17-21
address echo = getEcho();
require(echo != address(0), "Echo address not set");
require(msg.sender == echo, "Only Echo can call this function");
echoCallback(sequenceNumber, priceFeeds);
``` [4](#0-3) 

Because `msg.sender` is the Echo contract (not the original caller), this check passes regardless of who triggered `executeCallback`. The consumer has no way to distinguish a legitimate provider fulfillment from an attacker-triggered one.

Provider registration is permissionless:

```solidity
// Echo.sol lines 381-393
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    ...
    provider.isRegistered = true;
``` [5](#0-4) 

Fee withdrawal is available to any registered provider via `withdrawAsFeeManager` after calling `setFeeManager(self)`: [6](#0-5) 

### Impact Explanation

**Fee theft (direct financial loss):** An attacker registers as a provider, waits for the exclusivity period to expire, calls `executeCallback(attacker_address, sequenceNumber, validUpdateData, priceIds)`, and receives the full request fee that was meant for the legitimate provider. The legitimate provider is permanently deprived of their earned fee.

**Forced consumer callback execution:** Even during the exclusivity period, any third party can call `executeCallback(req.provider, sequenceNumber, validUpdateData, priceIds)` — the fee goes to the legitimate provider but the callback fires from an unauthorized trigger. Consumer contracts that rely on the callback being invoked only by the designated provider (e.g., for sequencing, reentrancy guards, or state-machine transitions) can be manipulated.

### Likelihood Explanation

Pyth price data for any given `publishTime` is publicly available from the Hermes price service. An attacker needs only to:
1. Register as a provider (no cost beyond gas).
2. Observe a pending Echo request on-chain.
3. Fetch valid Pyth update data for `req.publishTime` from Hermes.
4. Wait for `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`.
5. Call `executeCallback` with `providerToCredit = attacker_address`.

No privileged access, leaked keys, or oracle manipulation is required. The attack is fully automatable and profitable whenever request fees exceed gas costs.

### Recommendation

Add an explicit `msg.sender` authorization check in `executeCallback`. During the exclusivity period, require `msg.sender == req.provider`. After the exclusivity period, require `msg.sender` to be a registered provider and use `msg.sender` (not the caller-supplied `providerToCredit`) as the address to credit fees, or at minimum verify that `providerToCredit == msg.sender` to prevent fee redirection:

```solidity
function executeCallback(...) external payable override {
    Request storage req = findActiveRequest(sequenceNumber);

    if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
        require(msg.sender == req.provider, "Only assigned provider during exclusivity period");
    }
    require(providerToCredit == msg.sender, "providerToCredit must be caller");
    ...
}
```

### Proof of Concept

```
1. Alice (legitimate provider) is registered and assigned to fulfill request #42
   with req.publishTime = T and req.fee = 1 ETH.

2. Attacker calls echo.registerProvider(0, 0, 0) — permissionless, no cost.

3. Attacker calls echo.setFeeManager(attacker_address) — sets self as fee manager.

4. At block.timestamp = T + exclusivityPeriodSeconds + 1:
   Attacker fetches valid Pyth updateData for timestamp T from Hermes.

5. Attacker calls:
   echo.executeCallback(attacker_address, 42, updateData, priceIds)

6. Echo credits 1 ETH to attacker's provider account.
   Echo calls consumer._echoCallback(42, priceFeeds) — consumer accepts
   because msg.sender == echo.

7. Attacker calls echo.withdrawAsFeeManager(attacker_address, 1 ETH).
   Attacker receives 1 ETH. Alice receives nothing.
```

        require(

        require(
            _state.providers[provider].accruedFeesInWei >= amount,
