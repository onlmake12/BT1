### Title
Keeper Queue Flooding DoS via Unbounded `requestPriceUpdatesWithCallback` Spam on Low-Fee Networks — (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

---

### Summary

The `Echo` contract's `requestPriceUpdatesWithCallback` function imposes no per-user rate limit or request cap. Any unprivileged caller can submit an unbounded number of price-update callback requests by paying only the protocol fee. On low-fee, high-throughput networks (e.g., Arbitrum), the cost per request is negligible. The off-chain Echo keeper must scan and process every pending request via `getFirstActiveRequests`, whose gas cost scales **linearly** with the number of unfulfilled requests. An attacker can therefore flood the keeper's processing queue, indefinitely delaying legitimate users' callbacks.

---

### Finding Description

`Echo.requestPriceUpdatesWithCallback` is a `payable external` function with no access control beyond a fee check and a 60-second `publishTime` cap: [1](#0-0) 

The only guard against spam is the fee requirement at line 75–76. On Arbitrum or similar L2s, `tx.gasprice` is on the order of 0.01–0.1 gwei, making `feePerGasInWei * callbackGasLimit` trivially small. An attacker can submit thousands of requests for a few dollars.

The keeper discovers pending work by calling `getFirstActiveRequests`. The interface documents the linear gas scaling explicitly: [2](#0-1) 

Each unfulfilled request between `firstUnfulfilledSeq` and `currentSequenceNumber` costs ~2 100 gas (cold SLOAD). With thousands of attacker requests queued, the keeper's discovery call becomes prohibitively expensive or hits block gas limits, stalling fulfillment for all users.

Additionally, the contract stores requests in a fixed-size ring buffer of `NUM_REQUESTS` slots (bounded by `uint8`, so ≤ 255): [3](#0-2) 

`allocRequest` assigns slots via `sequenceNumber % NUM_REQUESTS`. If an attacker submits exactly `NUM_REQUESTS` requests after a legitimate user's request at sequence number N, the slot at `(N + NUM_REQUESTS) % NUM_REQUESTS` wraps back to the legitimate user's slot and **silently overwrites it**. The legitimate user's callback is permanently lost; their fee is not refunded (the contract explicitly states excess value is not returned).

The analogous comment in the Entropy contract confirms this overwrite behavior is by design: [4](#0-3) 

---

### Impact Explanation

- **Keeper queue flooding**: Legitimate users' `requestPriceUpdatesWithCallback` callbacks are delayed indefinitely while the keeper processes the attacker's queue.
- **Ring buffer overwrite**: A legitimate user's pending request can be silently overwritten by an attacker submitting `NUM_REQUESTS` subsequent requests. The user's callback never fires and their fee is not refunded.
- **Severity**: Medium–High. Delayed or lost price-update callbacks in DeFi protocols (e.g., liquidation triggers, settlement) can cause direct financial loss to downstream integrators.

---

### Likelihood Explanation

- The attack requires no privileged role — any EOA or contract can call `requestPriceUpdatesWithCallback`.
- On Arbitrum (a primary Echo deployment target), gas fees are ~0.01–0.1 gwei. Filling a 255-slot ring buffer costs well under $1 at current prices.
- The 60-second `publishTime` cap prevents far-future request spam but does not limit the total number of concurrent requests. [5](#0-4) 

---

### Recommendation

1. **Short term**: Introduce a per-address request cap (e.g., max N concurrent unfulfilled requests per `msg.sender`). Revert if the cap is exceeded.
2. **Short term**: In `allocRequest`, check whether the slot being overwritten contains an active (unfulfilled) request and revert rather than silently overwriting it.
3. **Long term**: Analyze the fee model on each supported chain and set a minimum fee floor that makes bulk spam economically infeasible regardless of network gas price.

---

### Proof of Concept

```
1. Deploy Echo on Arbitrum (low gas price ~0.01 gwei).
2. Legitimate user Alice calls requestPriceUpdatesWithCallback(provider, now, [priceId], gasLimit)
   → assigned sequenceNumber = N, stored at slot N % NUM_REQUESTS.
3. Attacker Eve calls requestPriceUpdatesWithCallback NUM_REQUESTS times in rapid succession
   (sequence numbers N+1 … N+NUM_REQUESTS).
   → Each call costs ~$0.001 on Arbitrum.
4. The request at sequence number N+NUM_REQUESTS is stored at slot
   (N + NUM_REQUESTS) % NUM_REQUESTS = N % NUM_REQUESTS,
   overwriting Alice's request.
5. Alice's callback is never executed; her fee is not refunded.
6. The keeper's getFirstActiveRequests must now scan NUM_REQUESTS+1 slots,
   all occupied by Eve's requests, before it can service any other user.
``` [6](#0-5) [7](#0-6)
