### Title
Keeper Fee Theft and Gas Griefing via Front-Running `Scheduler.updatePriceFeeds` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.updatePriceFeeds` function is callable by any unprivileged address (keeper). A keeper who observes a subscription that meets update conditions and submits a transaction to the mempool can be front-run by an attacker who submits the identical update data first. The attacker's transaction succeeds and collects the keeper fee; the original keeper's transaction then reverts on the `TimestampOlderThanLastUpdate` check, wasting all gas with no reimbursement.

---

### Finding Description

`Scheduler.updatePriceFeeds` is a permissionless function that any address can call to push a price update for a subscription and receive a keeper fee paid from the subscription's balance.

The function flow is:

1. Records `startGas = gasleft()`.
2. Deducts `pythFee` from `status.balanceInWei`.
3. Calls `pyth.parsePriceFeedUpdatesWithConfig` (expensive Wormhole VAA verification + Merkle proof parsing).
4. Validates all slots match.
5. Calls `_validateShouldUpdatePrices`, which **reverts** with `TimestampOlderThanLastUpdate` if `updateTimestamp <= status.priceLastUpdatedAt`.
6. Pays the keeper via `_processFeesAndPayKeeper`. [1](#0-0) 

Because the entire function is atomic, a revert at step 5 rolls back the `pythFee` deduction at step 2 and skips the keeper payment at step 6. The keeper receives nothing but has already spent gas on the expensive VAA verification in step 3.

**Attack path:**

1. Attacker monitors the mempool for a keeper's `updatePriceFeeds(subscriptionId, updateData)` transaction.
2. Attacker copies the exact same `subscriptionId` and `updateData` and submits with a higher gas price (front-run).
3. Attacker's transaction executes first: VAA is verified, prices are stored, `status.priceLastUpdatedAt` is set to the update's timestamp, and the attacker receives the full keeper fee.
4. Original keeper's transaction executes: VAA verification and Merkle proof parsing succeed (same data), but `_validateShouldUpdatePrices` reverts with `TimestampOlderThanLastUpdate` because `updateTimestamp == status.priceLastUpdatedAt` (already set by the attacker).
5. Keeper's transaction reverts entirely — no reimbursement, gas is lost.

---

### Impact Explanation

- **Keeper gas theft**: The keeper pays for expensive Wormhole VAA verification and Merkle proof parsing (significant gas) but receives zero reimbursement because the transaction reverts.
- **Keeper fee theft**: The attacker collects the keeper fee (gas cost reimbursement + `singleUpdateKeeperFeeInWei × numPriceIds`) from the subscription balance.
- **Subscription balance drain**: The subscription's balance is charged the keeper fee by the attacker, accelerating depletion.
- **Keeper liveness degradation**: Repeated griefing makes operating a keeper unprofitable, degrading the liveness of the Scheduler system. [3](#0-2) 

---

### Likelihood Explanation

- `updatePriceFeeds` is fully permissionless — any EOA can call it.
- Keeper transactions are publicly visible in the mempool on all EVM chains where the Scheduler is deployed.
- The attack is **directly profitable**: the attacker receives the keeper fee minus their own gas cost. On chains with low base fees, the profit margin is significant.
- No special knowledge, keys, or privileged access is required.
- The attack is trivially automatable with a mempool-watching bot.

---

### Recommendation

1. **Commit-reveal or subscription-specific access**: Allow subscriptions to designate a whitelisted keeper address, or implement a commit-reveal scheme so the update data cannot be copied from the mempool.
2. **Non-reverting stale-update handling**: Instead of reverting when `updateTimestamp <= priceLastUpdatedAt`, return gracefully (emit an event, refund the caller's gas via a partial payment). This mirrors the fix suggested in the original report — remove the hard revert from the internal validation path.
3. **Partial gas reimbursement on stale revert**: If a revert is unavoidable, reimburse the caller's gas cost up to the point of the stale-check before reverting, so keepers are not penalized for racing conditions.

---

### Proof of Concept

```
Precondition:
  - Subscription S has heartbeatSeconds elapsed since last update.
  - Keeper K fetches updateData from Hermes and submits:
      scheduler.updatePriceFeeds(S, updateData)  // gas price: 10 gwei

Attack:
  - Attacker A sees K's tx in mempool.
  - A submits identical call with gas price 11 gwei (front-run):
      scheduler.updatePriceFeeds(S, updateData)

Result:
  - A's tx mines first:
      parsePriceFeedUpdatesWithConfig succeeds
      _validateShouldUpdatePrices passes (priceLastUpdatedAt < updateTimestamp)
      status.priceLastUpdatedAt = updateTimestamp
      A receives keeper fee from subscription balance
  - K's tx mines second:
      parsePriceFeedUpdatesWithConfig succeeds (same VAA, same Merkle proofs)
      _validateShouldUpdatePrices REVERTS: updateTimestamp <= priceLastUpdatedAt
      Entire tx reverts — K loses ~200k+ gas, receives nothing
``` [4](#0-3)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }
