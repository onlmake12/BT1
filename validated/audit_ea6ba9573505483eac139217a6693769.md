### Title
`MockPyth.parsePriceFeedUpdatesUnique` Silently Ignores Uniqueness Constraint — (`File: target_chains/ethereum/sdk/solidity/MockPyth.sol`)

---

### Summary

`MockPyth.sol`, the official SDK testing mock distributed as `@pythnetwork/pyth-sdk-solidity`, implements `parsePriceFeedUpdatesUnique` by passing `checkUniqueness = false` to the shared `parsePriceFeedUpdatesWithConfig` helper — the same value used by `parsePriceFeedUpdates`. The uniqueness parameter is silently dropped, making both functions behaviorally identical in the mock. The production `Pyth.sol` correctly passes `checkUniqueness = true`. This is a direct analog to the external report: a function's documented semantic (enforce `prevPublishTime < minPublishTime`) is not implemented, and the distinguishing parameter is ignored.

---

### Finding Description

**Interface contract (`IPyth.sol`)** documents `parsePriceFeedUpdatesUnique` as:

> "ensures the updates returned are the first updates published in `minPublishTime`. That is, if there are multiple updates for a given timestamp, this method will return the first update."

The uniqueness invariant is: `prevPublishTime < minPublishTime <= publishTime <= maxPublishTime`. [1](#0-0) 

**`MockPyth.sol` implementation** (the SDK mock):

```solidity
function parsePriceFeedUpdatesUnique(...) external payable override returns (...) {
    (feeds, ) = parsePriceFeedUpdatesWithConfig(
        updateData, priceIds, minPublishTime, maxPublishTime,
        false,   // ← checkUniqueness: WRONG, should be true
        true,
        false
    );
}
``` [2](#0-1) 

Compare with `parsePriceFeedUpdates` in the same file, which also passes `false`: [3](#0-2) 

Both functions are now identical in the mock. The `checkUniqueness` flag is the only difference between them in the shared helper: [4](#0-3) 

**Production `Pyth.sol`** correctly passes `true`:

```solidity
function parsePriceFeedUpdatesUnique(...) {
    (priceFeeds, ) = parsePriceFeedUpdatesWithConfig(
        ..., true, false, false   // checkUniqueness = true ✓
    );
}
``` [5](#0-4) 

---

### Impact Explanation

`MockPyth.sol` is the canonical testing mock shipped in the official `@pythnetwork/pyth-sdk-solidity` SDK package. Protocols that use `parsePriceFeedUpdatesUnique` to enforce settlement uniqueness (e.g., to guarantee a price is used only once per time window, preventing replay-based manipulation) will write tests against `MockPyth`. Because `MockPyth` does not enforce `prevPublishTime < minPublishTime`, those tests will pass even when the uniqueness constraint is violated. The protocol's uniqueness-dependent logic is never exercised in tests, creating a false sense of correctness. When deployed against the real Pyth contract, the behavior diverges: the real contract enforces uniqueness and will revert on inputs that `MockPyth` accepted, or — more dangerously — a protocol that was designed to rely on uniqueness but was never tested for it may have a logical gap that an attacker can exploit by submitting a non-first update within the window.

---

### Likelihood Explanation

`MockPyth` is the standard, officially recommended testing utility for all EVM protocols integrating Pyth. Any protocol that calls `parsePriceFeedUpdatesUnique` and tests with `MockPyth` is affected. The function name explicitly signals a uniqueness guarantee, so developers who choose it over `parsePriceFeedUpdates` do so precisely because they need that guarantee — making them the most likely to be harmed by its absence in tests.

---

### Recommendation

In `MockPyth.sol`, change `parsePriceFeedUpdatesUnique` to pass `checkUniqueness = true`:

```solidity
function parsePriceFeedUpdatesUnique(
    bytes[] calldata updateData,
    bytes32[] calldata priceIds,
    uint64 minPublishTime,
    uint64 maxPublishTime
) external payable override returns (PythStructs.PriceFeed[] memory feeds) {
    (feeds, ) = parsePriceFeedUpdatesWithConfig(
        updateData,
        priceIds,
        minPublishTime,
        maxPublishTime,
        true,   // ← was false, must be true to match interface semantics
        true,
        false
    );
}
```

### Proof of Concept

1. Deploy `MockPyth` with any valid parameters.
2. Create two price feed updates for the same `priceId` with:
   - Update A: `publishTime = T`, `prevPublishTime = T - 10` (i.e., `prevPublishTime >= minPublishTime` if `minPublishTime = T - 5`)
   - Update B: `publishTime = T`, `prevPublishTime = T - 20` (i.e., `prevPublishTime < minPublishTime`)
3. Call `MockPyth.parsePriceFeedUpdatesUnique(updateData, priceIds, T-5, T+5)` with Update A.
4. **Expected (per interface):** Revert with `PriceFeedNotFoundWithinRange` because `prevPublishTime (T-10) >= minPublishTime (T-5)`.
5. **Actual:** Returns successfully — the uniqueness check is never applied because `checkUniqueness = false`.
6. Call the same function on the real `Pyth.sol` with the same data: it reverts as expected.

The divergence between mock and production behavior is confirmed by comparing:
- `MockPyth.sol` line 175: `false` for `checkUniqueness`
- `Pyth.sol` line 628: `true` for `checkUniqueness` [6](#0-5) [7](#0-6)
