### Title
Unimplemented `todo!()` Stubs in Production Argus Adapters Cause Panic and Service Crash — (`apps/argus/src/adapters/hermes.rs`, `apps/argus/src/adapters/contract.rs`)

---

### Summary

`HermesClient::get_latest_prices` and `PythPulse<M>::update_price_feeds` — the **only** concrete implementations of their respective traits — are stubbed with `todo!()`. Both are called unconditionally from production service loops (`PythPriceService` and `PricePusherService`). On the first timer tick or push request, the service panics and crashes, permanently halting all on-chain price updates.

---

### Finding Description

In `apps/argus/src/adapters/hermes.rs`, the sole implementation of `ReadPythPrices` is `HermesClient`, and both required methods are stubs:

```rust
async fn get_latest_prices(&self, _feed_ids: &[PriceId]) -> Result<Vec<Vec<u8>>> {
    todo!()
}

async fn subscribe_to_price_updates(&self, _feed_ids: &[PriceId]) -> Result<()> {
    todo!()
}
``` [1](#0-0) 

In `apps/argus/src/adapters/contract.rs`, the sole implementation of `UpdateChainPrices` is `PythPulse<M>`, and its core method is also a stub:

```rust
async fn update_price_feeds(...) -> Result<H256> {
    // ...
    todo!()
}
``` [2](#0-1) 

`GetChainPrices::get_price_unsafe` on `PythPulse<M>` is likewise `todo!()`: [3](#0-2) 

These stubs are called directly from two production service loops:

**`PythPriceService::start`** — calls `get_latest_prices` on every timer tick: [4](#0-3) 

**`PricePusherService::handle_request`** — calls both `get_latest_prices` and `update_price_feeds` for every push request: [5](#0-4) 

The `todo!()` macro in Rust unconditionally panics with `"not yet implemented"`. In an async Tokio task, this unwinds the task, terminating the service loop.

---

### Impact Explanation

The argus price-pusher service crashes on its first operational cycle — either the first timer tick in `PythPriceService` or the first push request processed by `PricePusherService`. Once crashed, no further price updates are submitted to the on-chain Pulse contract. All subscriptions depending on argus receive permanently stale prices. The impact is equivalent to the original report: **price attestations/updates are frozen until the stubs are implemented**.

---

### Likelihood Explanation

**Certain.** There is no conditional guard, feature flag, or alternative code path that bypasses the `todo!()` calls. `HermesClient` and `PythPulse<M>` are the only concrete implementations of their traits. The `PythPriceService` timer fires automatically after startup; no external actor needs to do anything. The crash is deterministic and immediate.

---

### Recommendation

Implement the four stubbed methods:

- `HermesClient::get_latest_prices` — issue an HTTP request to the Hermes `/v2/updates/price/latest` endpoint and return the encoded update data.
- `HermesClient::subscribe_to_price_updates` — open a streaming SSE/WebSocket connection to Hermes for the given feed IDs.
- `PythPulse<M>::update_price_feeds` — encode and submit the `updatePriceFeeds` transaction to the Pulse contract via the `Middleware`.
- `PythPulse<M>::get_price_unsafe` — call the `getPriceUnsafe` view function on the Pulse contract.

Until these are implemented, the argus service must not be deployed in any environment where it would be instantiated with `HermesClient` or `PythPulse<M>`.

---

### Proof of Concept

1. Start the argus service; `PythPriceService` is instantiated with `HermesClient` and `PythPulse<M>`.
2. `PythPriceService::start` enters its loop; the interval timer fires.
3. `feed_ids` is non-empty (at least one subscription exists).
4. `self.pyth_price_client.get_latest_prices(&feed_ids_vec).await` is called on `HermesClient`.
5. `HermesClient::get_latest_prices` executes `todo!()` → **panic: "not yet implemented"**.
6. The Tokio task unwinds; the service loop terminates.
7. No further price updates are ever pushed; all on-chain prices go stale.

The same panic is triggered independently via `PricePusherService::handle_request` on the first `PushRequest` received from the controller. [6](#0-5) [7](#0-6)
