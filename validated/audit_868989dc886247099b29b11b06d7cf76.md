### Title
Expired Network Alert Accepted and Re-broadcast via P2P Without Expiry Check - (File: `util/network-alert/src/alert_relayer.rs`)

### Summary
The P2P alert handler in `AlertRelayer::received()` verifies cryptographic signatures on incoming alerts but never checks whether the alert's `notice_until` timestamp has already passed. Any unprivileged P2P peer can replay a previously-valid, now-expired alert to inject it into nodes that have not yet seen it. The RPC entry point (`send_alert`) correctly enforces the expiry check, but the P2P path does not, creating an asymmetric enforcement gap.

### Finding Description
The `AlertRelayer::received()` handler processes alerts arriving over the P2P network. It parses the message, deduplicates by alert ID, verifies the multi-signature, then broadcasts the alert to connected peers and stores it in the `Notifier`. [1](#0-0) 

The signature verification step delegates to `Verifier::verify_signatures()`, which only checks cryptographic validity — it never inspects `notice_until`. [2](#0-1) 

By contrast, the RPC entry point `AlertRpcImpl::send_alert()` explicitly rejects alerts whose `notice_until` is in the past before doing anything else: [3](#0-2) 

The `Notifier::add()` method also performs no expiry check when storing the alert: [4](#0-3) 

Expiry cleanup only happens lazily in `clear_expired_alerts()`, which is called from `AlertRelayer::connected()` — i.e., only when a new peer connects, not when an alert is received: [5](#0-4) 

The `RawAlert` molecule schema confirms `notice_until` is a plain `Uint64` field that is part of the signed payload but is never validated for currency during P2P receipt: [6](#0-5) 

### Impact Explanation
An unprivileged P2P peer who observed any past legitimate alert (all alerts are broadcast publicly on the P2P network) can replay it to nodes that have not yet seen it. The consequences are:

1. **Stale alert display**: The replayed expired alert is added to `noticed_alerts` and surfaced to users via `get_blockchain_info`, causing confusion about the node's health or required upgrades.
2. **Alert ID slot poisoning**: Because `has_received()` returns `true` for any alert ID stored in `received_alerts` or `cancel_filter`, a replayed old alert with ID `X` causes any future legitimate alert with the same ID `X` to be silently dropped by all affected nodes — effectively suppressing a critical network-wide notification.
3. **Unnecessary network amplification**: The expired alert is re-broadcast to all connected peers, propagating the stale data further. [7](#0-6) 

### Likelihood Explanation
The attack requires no privileged keys and no special capability beyond being a connected P2P peer. Past alerts are publicly observable on the CKB mainnet P2P network. The attacker simply stores a previously-broadcast alert and re-sends it to target nodes (e.g., newly-synced nodes that have never seen the alert). The attack is trivially repeatable and requires no brute force or cryptographic work.

### Recommendation
Add an expiry check in `AlertRelayer::received()` immediately after parsing the alert, mirroring the check already present in the RPC handler:

```rust
// In AlertRelayer::received(), after parsing the alert:
let notice_until: u64 = alert.as_reader().raw().notice_until().into();
let now_ms = ckb_systemtime::unix_time_as_millis();
if notice_until < now_ms {
    debug!(
        "Received an expired alert {} from peer {}, ignoring",
        alert_id, peer_index
    );
    return; // do NOT ban; expiry is not a protocol violation
}
```

Additionally, `Verifier::verify_signatures()` should optionally accept a `now_ms` parameter so expiry can be enforced at the verification layer as a single, auditable check point.

### Proof of Concept

1. Observe any past legitimate CKB network alert broadcast on the P2P network and record its raw bytes (alert ID, `notice_until` in the past, valid 2-of-4 signatures).
2. Connect to a CKB node as a P2P peer using the Alert protocol (`SupportProtocols::Alert`).
3. Send the raw bytes of the expired alert to the target node.
4. The target node's `AlertRelayer::received()` will:
   - Parse the alert successfully.
   - Find `has_received(alert_id)` returns `false` (node has not seen this alert).
   - Call `verify_signatures()` — passes, because the signatures are cryptographically valid.
   - Broadcast the expired alert to all of the target node's connected peers.
   - Call `notifier.add(&alert)` — stores the expired alert in `received_alerts` and `noticed_alerts`.
5. Query `get_blockchain_info` on the target node and observe the expired alert appearing in the `alerts` field.
6. Any future legitimate alert with the same ID will be silently dropped by the target node because `has_received(alert_id)` now returns `true`. [8](#0-7)

    pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
        trace!("Verifying alert {:?}", alert);
        let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
        let signatures: Vec<Signature> = alert
            .signatures()
            .into_iter()
            .filter_map(
                |sig_data| match Signature::from_slice(sig_data.as_reader().raw_data()) {
                    Ok(sig) => {
                        if sig.is_valid() {
                            Some(sig)
                        } else {
                            debug!("invalid signature: {:?}", sig);
                            None
                        }
                    }
                    Err(err) => {
                        debug!("signature error: {}", err);
                        None
                    }
                },
            )
            .collect();
        verify_m_of_n(
            &message,
            self.config.signatures_threshold,
            &signatures,
            &self.pubkeys,
        )
        .map_err(|err| err.kind())?;
        Ok(())
    }

    pub fn add(&mut self, alert: &Alert) {
        let alert_id = alert.raw().id().into();
        let alert_cancel = alert.raw().cancel().into();
        if self.has_received(alert_id) {
            return;
        }
        // checkout cancel_id
        if alert_cancel > 0 {
            self.cancel(alert_cancel);
        }
        // add to received alerts
        self.received_alerts.insert(alert_id, alert.clone());

        // check conditions, figure out do we need to notice this alert
        if !self.is_version_effective(alert) {
            debug!("Received a version ineffective alert {:?}", alert);
            return;
        }

        if self.noticed_alerts.contains(alert) {
            return;
        }
        self.notify_controller.notify_network_alert(alert.clone());
        self.noticed_alerts.push(alert.clone());
        // sort by priority
        self.noticed_alerts.sort_by_key(|a| {
            let priority: u32 = a.raw().priority().into();
            u32::MAX - priority
        });
    }
