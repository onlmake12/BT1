### Title
Insufficient Gas Causes Silent Permanent Callback Loss in Echo.sol — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` function irreversibly clears the request and credits the provider **before** making the consumer callback, then wraps the callback in a bare `try/catch` that silently swallows out-of-gas (OOG) exceptions. An attacker (or any relayer) can supply a precisely calculated gas amount that allows all pre-callback state changes to complete but causes the inner `_echoCallback` call to OOG. The request is permanently lost with no retry path, and the consumer never receives their price update.

---

### Finding Description

In `Echo.sol`, `executeCallback` performs the following sequence:

1. **Credits the provider** (line 161–162)
2. **Clears the request** (line 164) — irreversible, no retry state
3. **Advances `firstUnfulfilledSeq`** (lines 169–174) — variable gas consumption loop
4. **Calls the consumer callback** with a fixed gas cap (lines 176–179)
5. **Catches all exceptions** including OOG in a bare `catch` (lines 192–200)

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // provider credited

clearRequest(sequenceNumber);                        // request permanently deleted

while (...) { _state.firstUnfulfilledSeq++; }       // variable gas burn

try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit                    // 63/64 rule applies here
    }(sequenceNumber, priceFeeds)
{ ... }
catch Error(string memory reason) { emit PriceUpdateCallbackFailed(...); }
catch {
    // Out of gas or other low-level errors          // OOG silently swallowed
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
``` [1](#0-0) 

Per the EVM CALL opcode, at most 63/64 of the outer context's remaining gas is forwarded to the inner call, regardless of the `{gas: X}` stipend specified. If the outer context has just enough gas to complete the pre-callback logic but not enough to actually forward `callbackGasLimit` gas, the inner call silently OOGs. The bare `catch` fires, emitting an event, and execution continues — but the request is already gone.

**Contrast with `Entropy.sol`'s `revealWithCallback`**, which explicitly guards against this:

```solidity
uint256 startingGas = gasleft();
(success, ret) = req.requester.excessivelySafeCall(...);
...
} else if ((startingGas * 31) / 32 > uint256(req.gasLimit10k) * TEN_THOUSAND) {
    req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED; // retryable
} else {
    revert EntropyErrors.InsufficientGas(); // forces caller to retry with more gas
}
``` [2](#0-1) 

`Entropy.sol` uses `excessivelySafeCall`, records `startingGas`, distinguishes OOG from genuine revert, and either marks the request as `CALLBACK_FAILED` (retryable) or reverts with `InsufficientGas`. `Echo.sol` has none of these protections and has no `CALLBACK_FAILED` retry state at all. [3](#0-2) 

---

### Impact Explanation

- The consumer's `_echoCallback` is never executed, yet the request is permanently cleared and cannot be resubmitted.
- The consumer paid fees (including `callbackGasLimit`-scaled gas fees) but receives no price update.
- The provider is credited regardless of whether the callback succeeded.
- There is no `CALLBACK_FAILED` state or retry mechanism in `Echo.sol`, unlike `Entropy.sol`.
- This constitutes a **permanent, irrecoverable DoS** on the consumer's price update callback.

---

### Likelihood Explanation

- `executeCallback` is a `public payable` function callable by any address — no access control.
- An attacker can calculate the gas consumed by the pre-callback path (validation, Pyth fee parsing, storage writes, `firstUnfulfilledSeq` loop) and submit a transaction with just enough gas to complete that path but not enough to forward `callbackGasLimit` to the inner call.
- The `firstUnfulfilledSeq` loop (lines 169–174) adds variable but bounded gas consumption, which an attacker can account for by observing on-chain state.
- A malicious relayer competing for the provider fee has direct economic incentive to grief a consumer this way (e.g., to prevent a competitor's callback from executing).
- The attack requires no privileged access, no leaked keys, and no third-party oracle manipulation. [4](#0-3) 

---

### Recommendation

1. **Short term:** Before the try-catch, verify that the outer context has sufficient gas to forward `callbackGasLimit` to the inner call. Mirror `Entropy.sol`'s pattern:
   ```solidity
   uint256 startingGas = gasleft();
   // ... make the call ...
   if (!success) {
       if ((startingGas * 31) / 32 <= req.callbackGasLimit) {
           revert InsufficientGas(); // force caller to retry with more gas
       }
       // genuine callback failure — emit event
   }
   ```
2. **Short term:** Do not clear the request before the callback. Either keep the request alive until the callback succeeds, or introduce a `CALLBACK_FAILED` retry state (as `Entropy.sol` does) so that a failed callback can be retried.
3. **Long term:** Use `excessivelySafeCall` (as `Entropy.sol` does) instead of a Solidity `try/catch`, which provides better control over return data copying and gas accounting.

---

### Proof of Concept

1. Alice calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 200_000`, paying the required fee.
2. Attacker observes the pending request on-chain.
3. Attacker calls `executeCallback` with a gas amount calculated to:
   - Complete all pre-callback logic (validation, Pyth parsing, fee crediting, `clearRequest`, `firstUnfulfilledSeq` loop) — approximately 80,000–120,000 gas
   - Leave fewer than `200_000 * 64/63 ≈ 203_175` gas remaining at the point of the try-call, so the inner call receives less than `200_000` gas
4. The inner `_echoCallback{gas: 200_000}` call OOGs because the EVM can only forward 63/64 of the remaining outer gas.
5. The bare `catch` fires, emitting `PriceUpdateCallbackFailed("low-level error (possibly out of gas)")`.
6. Alice's request is permanently cleared. She paid fees but her callback was never executed and cannot be retried. [5](#0-4)
