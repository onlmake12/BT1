### Title
Unbounded Lifetime of `unknown_tx_hashes` Entries Enables Persistent Tx-Relay Queue Exhaustion — (File: `sync/src/types/mod.rs`)

---

### Summary

The `unknown_tx_hashes` queue in `SyncState` tracks transaction hashes announced by peers but not yet fetched. Entries carry a 30-second retry cooldown (`RETRY_ASK_TX_TIMEOUT_INCREASE`) but have **no time-based expiration**. A malicious peer can keep entries alive indefinitely by re-announcing the same fake transaction hashes before they are naturally dropped, exhausting the global queue and blocking legitimate transaction relay for all peers.

---

### Finding Description

**Entry creation — no expiration timestamp recorded**

When a peer sends a `RelayTransactionHashes` message, `TransactionHashesProcess::execute` calls `state.add_ask_for_txs(peer, tx_hashes)`: [1](#0-0) 

Each new entry is created with `request_time = Instant::now()` and `requested = false`. No deadline or maximum lifetime is stored.

**Retry cooldown exists, but no expiration**

`pop_ask_for_txs` enforces a 30-second cooldown before retrying a request: [2](#0-1) 

The constant is `RETRY_ASK_TX_TIMEOUT_INCREASE = Duration::from_secs(30)`: [3](#0-2) 

**Entries are only dropped when all peers are exhausted — not by time**

`next_request_peer` removes one peer per retry cycle and returns `None` only when the last peer has already been tried: [4](#0-3) 

In `pop_ask_for_txs`, when `next_request_peer()` returns `None` the entry is silently dropped (not pushed back). This is the **only** removal path other than `mark_as_known_txs`.

**`push_peer` allows a peer to re-add itself without deduplication**

When the same peer re-announces the same tx hash (via a new `RelayTransactionHashes` message), `add_ask_for_txs` hits the `Occupied` branch and calls `priority.push_peer(peer_index)`: [5](#0-4) 

`push_peer` appends unconditionally with no duplicate check: [6](#0-5) 

This resets the effective lifetime of the entry: the peer is appended to the tail of `peers`, so `next_request_peer` will reach it again after cycling through the existing list.

**`disconnected()` does not clean up `unknown_tx_hashes`**

When a peer disconnects, only `inflight_blocks` and `peers` state are cleaned: [7](#0-6) 

Entries in `unknown_tx_hashes` that reference the disconnected peer are **not removed**. The node will attempt to request the transaction from the now-gone peer, fail silently, and the entry will linger until the next `pop_ask_for_txs` cycle exhausts the peer list — at which point the attacker can reconnect and re-announce.

**Soft size limits do not prevent the attack**

The global cap is `MAX_UNKNOWN_TX_HASHES_SIZE = 50_000` and per-peer cap is `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32_767`: [8](#0-7) 

When the queue is full, new announcements from honest peers return `Status::ignored()` — they are silently discarded: [9](#0-8) 

A single malicious peer can occupy up to 32,767 slots; two peers can saturate the entire 50,000-entry queue and hold it there indefinitely.

---

### Impact Explanation

- **Tx-relay starvation**: When `unknown_tx_hashes` is saturated, `RelayTransactionHashes` messages from honest peers are silently dropped. The node stops fetching new transactions from the network, degrading mempool propagation.
- **Bandwidth waste**: The node continuously emits `GetRelayTransactions` messages for non-existent transactions, consuming outbound bandwidth proportional to queue size × retry rate.
- **Memory pressure**: 50,000 entries, each holding a `Byte32` key and a `Vec<PeerIndex>` that grows with each re-announcement, constitute unbounded heap growth for the duration of the attack.

---

### Likelihood Explanation

The attack requires only a standard P2P connection. No cryptographic material, no mining power, and no privileged role is needed. The attacker sends `RelayTransactionHashes` messages (a normal relay protocol message) containing fabricated 32-byte hashes. The node cannot distinguish fake hashes from real unconfirmed transaction hashes before fetching them. The attacker must re-announce every ~30 seconds per peer slot, which is trivially automated. A single attacker-controlled node is sufficient to saturate the per-peer quota; two nodes saturate the global queue.

---

### Recommendation

1. **Add a wall-clock deadline to each `UnknownTxHashPriority` entry** (e.g., `created_at: Instant`). In `pop_ask_for_txs`, drop entries whose age exceeds a configurable maximum (e.g., 5 minutes), regardless of remaining peers.
2. **Clean up `unknown_tx_hashes` on peer disconnect** inside `SyncState::disconnected`, removing all entries whose `peers` list becomes empty after the disconnected peer is filtered out.
3. **Deduplicate in `push_peer`**: before appending, check whether `peer_index` is already present in `self.peers` to prevent a single peer from artificially extending an entry's lifetime.

---

### Proof of Concept

```
1. Attacker connects to a CKB node as a standard P2P relay peer.

2. Attacker sends RelayTransactionHashes containing 32,767 fabricated
   Byte32 hashes (none correspond to real transactions).

3. The node calls add_ask_for_txs, inserting 32,767 entries into
   unknown_tx_hashes with requested=false.

4. pop_ask_for_txs immediately dispatches GetRelayTransactions for all
   entries. Attacker ignores every request.

5. After 30 seconds (RETRY_ASK_TX_TIMEOUT_INCREASE), pop_ask_for_txs
   calls next_request_peer() on each entry. Since peers.len()==1 and
   requested==true, it returns None — entries are about to be dropped.

6. Before step 5 completes, attacker re-sends the same
   RelayTransactionHashes message. add_ask_for_txs hits the Occupied
   branch and calls push_peer, appending the attacker's PeerIndex again.
   peers is now [attacker, attacker]; the entry survives.

7. Repeat from step 4 indefinitely. The queue remains saturated at
   32,767 entries. Honest peers' RelayTransactionHashes messages are
   silently ignored (Status::ignored()) for the duration of the attack.
```

                self.peers.swap_remove(0);
                self.peers.first().cloned()
