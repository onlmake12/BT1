Audit Report

## Title
`descendants_*` Fields Permanently Stale After `remove_entry_and_descendants` Due to Pre-Removal of Links — (`tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::remove_entry_and_descendants` tears down all link entries for the removed subtree before calling `remove_entry` on each node. Because `update_ancestors_index_key` discovers ancestors via `self.links.calc_ancestors`, and those link entries are already erased, it returns an empty set and never calls `sub_descendant_weight` on surviving ancestors. The `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` fields of every surviving ancestor are permanently inflated, corrupting eviction-key ordering for the lifetime of those entries.

## Finding Description
`remove_entry_and_descendants` operates in two phases:

**Phase 1 — link teardown:** [1](#0-0) 

`remove_entry_links` removes the entry from its parents' children sets, its children's parents sets, and then calls `self.links.remove(id)`, fully erasing the `ProposalShortId → TxLinks` entry from `TxLinksMap::inner`: [2](#0-1) 

**Phase 2 — entry removal:** [3](#0-2) 

`remove_entry` calls `update_ancestors_index_key` at L242: [4](#0-3) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`: [5](#0-4) 

`calc_ancestors` delegates to `calc_relative_ids`, which starts with `self.inner.get(short_id)`: [6](#0-5) 

Because Phase 1 already called `self.links.remove(id)` for every removed ID, `self.inner.get(short_id)` returns `None`, `direct` is empty, and `calc_relation_ids` returns an empty set. `sub_descendant_weight` is never called on any surviving ancestor.

The comment at L256 acknowledges the pre-removal is intentional to suppress `update_descendants_index_key` (correct — all descendants are being removed), but it inadvertently also suppresses `update_ancestors_index_key` for ancestors that are **not** being removed.

`recompute_total_stat` only recomputes pool-wide `total_tx_size`/`total_tx_cycles`, not per-entry `descendants_*` fields: [7](#0-6) 

There is no periodic recomputation that would heal the stale fields.

## Impact Explanation
`EvictKey` is computed directly from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`: [8](#0-7) 

An ancestor whose `descendants_fee` is inflated (because removed high-fee descendants were never subtracted) appears to have a higher `descendants_feerate` than it actually does, so it is deprioritised for eviction. This allows low-fee transactions to occupy pool space beyond their fair share, corrupting the pool's eviction ordering for the remainder of the ancestor's lifetime. This constitutes a **suboptimal implementation of the CKB state storage mechanism** (Medium, 2001–10000 points), as the pool's incremental accounting invariant is permanently broken for affected entries.

## Likelihood Explanation
`remove_entry_and_descendants` is called from `resolve_conflict` (L305-332), `resolve_conflict_header_dep` (L267-292), `check_and_record_ancestors` (L618), and `limit_size`. The `resolve_conflict` path is reachable by any unprivileged user by submitting a double-spend: [9](#0-8) 

The attacker submits parent A (low fee), children B and C (high fee), then a conflicting transaction spending B's input. This triggers `resolve_conflict` → `remove_entry_and_descendants(B_id)`, leaving A's `descendants_*` permanently inflated. The attack is cheap, deterministic, and repeatable.

## Recommendation
Collect and update surviving ancestors **before** erasing links:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Update surviving ancestors BEFORE links are torn down.
    for rid in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(rid) {
            let inner = entry.inner.clone();
            self.update_ancestors_index_key(&inner, EntryOp::Remove);
        }
    }

    // Now safe to remove links.
    for rid in &removed_ids {
        self.remove_entry_links(rid);
    }

    removed_ids
        .iter()
        .filter_map(|rid| self.remove_entry_without_ancestor_update(rid))
        .collect()
}
```

Alternatively, add a flag to `remove_entry` to skip `update_ancestors_index_key` and call it explicitly before link removal.

## Proof of Concept
Write a unit test in `tx-pool/src/component/tests/` following the pattern of the existing `test_remove_entry` test in `score_key.rs`: [10](#0-9) 

1. Build chain A → B → C (A is parent, C is grandchild), all added to the pool via `add_proposed`.
2. Assert `A.descendants_count == 3` and `A.descendants_fee == fee_A + fee_B + fee_C`.
3. Call `pool_map.remove_entry_and_descendants(&B_id)`.
4. Assert B and C are absent from the pool.
5. Assert `A.descendants_count == 1` and `A.descendants_fee == fee_A` — **this assertion will fail**, demonstrating the stale fields.
