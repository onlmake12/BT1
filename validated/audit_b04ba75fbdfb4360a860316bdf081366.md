### Title
Anyone Can Force-Merge Another User's Staking Positions — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`, `governance/pyth_staking_sdk/src/idl/staking.json`)

---

### Summary

The `merge_delegation_positions` instruction in the Integrity Pool program and the `merge_target_positions` instruction in the Staking program both accept an `owner` account that is **writable but not required to sign**. Because the `owner` constraint is only a data-equality check (Anchor `relations`), any unprivileged caller can supply the correct `owner` pubkey (publicly readable on-chain) and force-merge another user's staking positions without their consent.

---

### Finding Description

In `integrity-pool.json`, the `merge_delegation_positions` instruction is explicitly documented as permissionless:

> `"CHECK : This instruction is permissionless, this account will be checked against stake_account_metadata in the CPI"`

The `owner` account is `writable` but carries **no `signer: true` constraint**. [1](#0-0) 

Similarly, in `staking.json`, the `merge_target_positions` instruction lists `owner` as writable with only a `relations` check against `stake_account_metadata`, and the only signer is `pool_authority` which is **optional**:

> `"CHECK : This AccountInfo is safe because it's checked against stake_account_metadata"` [2](#0-1) 

The `pool_authority` optional signer means the instruction succeeds even when `pool_authority` is absent. [3](#0-2) 

An attacker needs only to:
1. Read `stake_account_metadata.owner` from on-chain state (public).
2. Derive the `stake_account_positions` and `stake_account_metadata` PDAs.
3. Call `merge_delegation_positions` or `merge_target_positions` with the victim's `owner` pubkey.

The `owner` pubkey satisfies the `relations` constraint without signing. [4](#0-3) 

---

### Impact Explanation

`merge_delegation_positions` and `merge_target_positions` consolidate multiple staking positions (potentially with different amounts and activation epochs) into a single position. A user who deliberately maintains separate positions — for example, to selectively unstake a specific tranche, to track different activation epochs, or to manage partial cooldowns independently — can have those positions force-merged by an attacker at any time. This disrupts the user's position management strategy and may prevent them from executing intended partial-unstake operations, mirroring the M-3 impact where LP tokens were transferred at an inappropriate time. [5](#0-4) 

---

### Likelihood Explanation

The attack requires no special privilege, no leaked key, and no governance majority. The victim's `owner` pubkey and all required PDA seeds are publicly readable on-chain. Any Solana transaction sender can construct and submit the instruction at any time. The only cost is the transaction fee. Likelihood is **high**.

---

### Recommendation

Add `signer: true` to the `owner` account in both `merge_delegation_positions` and `merge_target_positions`. This ensures only the account owner (or an explicitly authorized delegate) can trigger a merge of their own positions. If permissionless merging is intentionally desired for protocol efficiency, the instruction should be restricted to a no-op when the merge would change the user's effective position structure (e.g., only allow merging fully-equivalent positions), or emit an event so the owner can detect and respond to unsolicited merges.

---

### Proof of Concept

```
1. Alice holds two delegation positions for publisher P:
   - Position A: 100 PYTH, activationEpoch = 10
   - Position B: 50 PYTH, activationEpoch = 12
   Alice intends to unstake Position B (50 PYTH) next epoch while keeping Position A locked.

2. Bob (attacker) reads Alice's stake_account_metadata on-chain, obtains her owner pubkey.

3. Bob calls merge_delegation_positions with:
   - owner = Alice's pubkey  (no signature from Alice required)
   - publisher = P
   - stakeAccountPositions = Alice's positions account

4. The program merges Position A and Position B into a single position of 150 PYTH
   with activationEpoch = 10.

5. Alice can no longer selectively unstake the 50 PYTH tranche; she must unstake
   from the merged position, disrupting her intended strategy.
``` [6](#0-5) [7](#0-6)
