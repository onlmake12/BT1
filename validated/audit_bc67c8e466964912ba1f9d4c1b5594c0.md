### Title
DAO Withdrawal Header-Dep Index Interpretation Mismatch Between Rust and CKB-VM Causes Consensus Split — (`util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator` reads the full `u64` value from the witness `input_type` field to resolve the `header_deps` index for DAO withdrawal transactions. The CKB-VM DAO C script reads only the **lowest byte** of that same 8-byte field. When a transaction encodes an index whose full `u64` value differs from its lowest byte (e.g., 257, whose lowest byte is 1), Rust and the CKB-VM resolve different `header_deps` entries. This causes Rust to reject a transaction (and any block containing it) that the CKB-VM would accept, producing a consensus split.

---

### Finding Description

In `util/dao/src/lib.rs`, `DaoCalculator::transaction_maximum_withdraw` reads the deposit header index from the witness as a full `u64`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
``` [1](#0-0) 

It then uses this value directly as a `usize` index into `header_deps`:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
``` [2](#0-1) 

The CKB-VM DAO C script, however, reads only the **lowest byte** of the same 8-byte witness field when resolving the `header_deps` index. This is explicitly documented in the production test suite:

```rust
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
let mut header_deps = vec![dummy; 258];
header_deps[1] = deposit_block.hash();
header_deps[257] = withdraw_block.hash();
// input_type = 257, lowest byte = 1
let witness = WitnessArgs::new_builder()
    .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
    .build();
``` [3](#0-2) 

The test asserts that Rust returns an error for this transaction:

```rust
// Rust resolves index 257 → withdraw block (number 200), but cell data
// says deposited at block 100. Block number check catches the mismatch.
assert!(result.is_err(), "expected Err, got {result:?}");
``` [4](#0-3) 

The block-number cross-check that triggers the error is:

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
``` [5](#0-4) 

`transaction_maximum_withdraw` is called from `withdrawed_interests`, which is called from `dao_field_with_current_epoch` — the function used to compute and verify the DAO field embedded in every block header: [6](#0-5) [7](#0-6) 

Additionally, `DaoCalculator::transaction_fee` (which also calls `transaction_maximum_withdraw`) is invoked by `check_tx_fee` in the tx-pool: [8](#0-7) 

---

### Impact Explanation

**Consensus split.** A transaction sender crafts a DAO withdrawal whose witness `input_type` encodes index 257 (lowest byte = 1). The CKB-VM DAO script resolves `header_deps[1]` (the correct deposit block) and accepts the transaction. The Rust node resolves `header_deps[257]` (a different block), the block-number check fails, and `dao_field_with_current_epoch` returns `DaoError::InvalidOutPoint`. Because the DAO field cannot be computed, the Rust node rejects any block containing this transaction. CKB-VM-based nodes accept the same block. The two populations of nodes diverge onto different chains.

A secondary effect is **tx-pool denial of service**: the Rust tx-pool's `check_tx_fee` rejects the transaction before it can even be relayed, so legitimate DAO withdrawals using indices > 255 are silently dropped by Rust nodes.

---

### Likelihood Explanation

The attack requires only a transaction sender who can construct a DAO withdrawal with a crafted witness (index 257 or any value whose lowest byte differs from its full `u64` value) and a miner willing to include it. No privileged access, key material, or majority hash power is needed. The witness field is fully attacker-controlled, and the `header_deps` list can be padded to 258+ entries to make the high index valid. The discrepancy is already documented in the production test suite, indicating it is a known, reachable code path.

---

### Recommendation

Align the Rust index-reading logic with the CKB-VM DAO script. Either:

1. **Truncate to one byte in Rust**: read only the lowest byte of the 8-byte witness field (matching the C VM behavior), or
2. **Enforce a maximum index in consensus rules**: reject any DAO withdrawal whose witness encodes an index ≥ 256, so both Rust and the CKB-VM agree on the valid range.

The fix should be applied in `DaoCalculator::transaction_maximum_withdraw` in `util/dao/src/lib.rs` at the point where `LittleEndian::read_u64` is called. [1](#0-0) 

---

### Proof of Concept

1. Construct a DAO withdrawal transaction:
   - `header_deps` has 258 entries; `header_deps[1]` = deposit block hash (block number 100); `header_deps[257]` = withdraw block hash (block number 200).
   - Cell data encodes deposited block number = 100.
   - Witness `input_type` = `257u64` in little-endian (8 bytes).
2. Submit to a miner who includes it in a block.
3. **CKB-VM**: reads lowest byte of 257 = 1 → `header_deps[1]` = deposit block (number 100) → block-number check 100 == 100 → DAO script passes → block accepted.
4. **Rust node**: reads full u64 = 257 → `header_deps[257]` = withdraw block (number 200) → block-number check 200 ≠ 100 → `DaoError::InvalidOutPoint` → `dao_field_with_current_epoch` fails → block rejected.
5. Result: Rust nodes and CKB-VM-based nodes are on different chains.

This exact scenario is encoded in the existing production test `check_dao_withdraw_header_dep_index_exceeds_u8`: [9](#0-8)

    /// Calculates the new dao field with specified [`EpochExt`].
    pub fn dao_field_with_current_epoch(
        &self,
        rtxs: impl Iterator<Item = &'a ResolvedTransaction> + Clone,
        parent: &HeaderView,
        current_block_epoch: &EpochExt,
    ) -> Result<Byte32, DaoError> {
        // Freed occupied capacities from consumed inputs
        let freed_occupied_capacities =
            rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
                self.input_occupied_capacities(rtx)
                    .and_then(|c| capacities.safe_add(c))
            })?;
        let added_occupied_capacities = self.added_occupied_capacities(rtxs.clone())?;
        let withdrawed_interests = self.withdrawed_interests(rtxs)?;

        let (parent_ar, parent_c, parent_s, parent_u) = extract_dao_data(parent.dao());

        // g contains both primary issuance and secondary issuance,
        // g2 is the secondary issuance for the block, which consists of
        // issuance for the miner, NervosDAO and treasury.
        // When calculating issuance in NervosDAO, we use the real
        // issuance for each block(which will only be issued on chain
        // after the finalization delay), not the capacities generated
        // in the cellbase of current block.
        let current_block_number = parent.number() + 1;
        let current_g2 = current_block_epoch.secondary_block_issuance(
            current_block_number,
            self.consensus.secondary_epoch_reward(),
        )?;
        let current_g = current_block_epoch
            .block_reward(current_block_number)
            .and_then(|c| c.safe_add(current_g2))?;

        let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
            / u128::from(parent_c.as_u64());
        let miner_issuance =
            Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
        let nervosdao_issuance = current_g2.safe_sub(miner_issuance)?;

        let current_c = parent_c.safe_add(current_g)?;
        let current_u = parent_u
            .safe_add(added_occupied_capacities)
            .and_then(|u| u.safe_sub(freed_occupied_capacities))?;
        let current_s = parent_s
            .safe_add(nervosdao_issuance)
            .and_then(|s| s.safe_sub(withdrawed_interests))?;

        let ar_increase128 =
            u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
        let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
        let current_ar = parent_ar
            .checked_add(ar_increase)
            .ok_or(DaoError::Overflow)?;

        Ok(pack_dao_data(current_ar, current_c, current_s, current_u))
    }

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
