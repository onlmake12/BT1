Audit Report

## Title
Insert-Before-Check with O(N) Mutex-Held Scan in `add_ask_for_txs` Enables Sustained Tx-Relay Degradation — (`sync/src/types/mod.rs`)

## Summary
`SyncState::add_ask_for_txs` unconditionally inserts up to 32,767 hashes into the global `unknown_tx_hashes` queue before checking the 50,000-entry soft limit. Once the queue exceeds `MAX_UNKNOWN_TX_HASHES_SIZE`, every subsequent call performs a full O(N) nested linear scan over all queue entries while holding the `unknown_tx_hashes` mutex, blocking `pop_ask_for_txs` and `mark_as_known_txs` for the duration. The overflow path returns `Status::ignored()` with no peer penalty, allowing any two standard P2P connections to permanently keep the queue above the threshold at negligible cost.

## Finding Description
**Entry point** (`sync/src/relayer/transaction_hashes_process.rs`, L25–50): `TransactionHashesProcess::execute` enforces only a per-message size cap (`> MAX_RELAY_TXS_NUM_PER_BATCH`). There is no per-peer rate limit, cooldown, or connection-level quota. After filtering through `tx_filter`, it calls `state.add_ask_for_txs(self.peer, tx_hashes)` unconditionally. [1](#0-0) 

**Insert-before-check** (`sync/src/types/mod.rs`, L1483–1504): The mutex is acquired at L1484 and held for the entire function. Lines 1486–1503 insert up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767) new entries into the priority queue unconditionally. The global-limit check only occurs after all insertions complete. [2](#0-1) 

**O(N) scan on overflow** (`sync/src/types/mod.rs`, L1507–1529): When `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE (50,000)`, the code iterates over every entry in the queue and every peer inside each entry's `peers` vector to count the calling peer's contributions. This scan executes while the mutex is held. [3](#0-2) 

**No penalty on overflow** (L1528): The overflow branch returns `Status::ignored()`. The peer is never banned or disconnected, so the attacker can repeat the trigger indefinitely. [4](#0-3) 

**Constants** (`util/constant/src/sync.rs`, L67–72): `MAX_RELAY_TXS_NUM_PER_BATCH = MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32,767` and `MAX_UNKNOWN_TX_HASHES_SIZE = 50,000`. Two peers each sending a full batch push the queue to ~65,534, permanently above the threshold. [5](#0-4) 

**Mutex contention confirmed**: Both `mark_as_known_txs` (L1443–1451) and `pop_ask_for_txs` (L1453–1481) acquire the same `unknown_tx_hashes` mutex, so they are blocked for the full duration of the O(N) scan in `add_ask_for_txs`. [6](#0-5) 

The existing `tx_filter` guard does not prevent this because the attacker uses fresh, unique hashes on each message.

## Impact Explanation
The `unknown_tx_hashes` mutex serializes three callers: `add_ask_for_txs` (inbound relay messages), `pop_ask_for_txs` (the periodic timer dispatching `GetRelayTransactions` requests), and `mark_as_known_txs` (called on transaction verification). While `add_ask_for_txs` holds the mutex executing the O(N) scan, both other callers block. Sustained flooding from multiple connections creates a serialization bottleneck: the dispatch timer stalls, verified transactions cannot be removed from the queue, the queue stays inflated, and the O(N) scan fires on every subsequent call. This constitutes **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation
The attack requires only two standard P2P connections (no PoW, stake, or privilege) and approximately 2 MB of unique 32-byte hashes to push the queue past 50,000. Maintaining the inflated state requires only a low-rate stream of fresh hashes, since `pop_ask_for_txs` drains the queue gradually. The attacker receives no feedback (`Status::ignored()`) and is never disconnected, making the attack indefinitely repeatable at negligible cost.

## Recommendation
1. **Check before insert**: Evaluate both the per-peer and global limits before inserting any hashes; reject the batch early if either limit would be exceeded.
2. **Replace the O(N) scan with O(1) accounting**: Maintain a per-peer counter in `PeerState` that is incremented and decremented alongside insertions and removals, eliminating the need to scan the entire queue.
3. **Enforce a hard per-peer cap at insertion time**: Once a peer's counter reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, reject further hashes from that peer without touching the global queue.
4. **Apply a peer penalty on overflow**: Return a ban or disconnect status rather than `Status::ignored()` when a peer repeatedly triggers the overflow path.

## Proof of Concept
```
1. Connect two attacker peers A and B to the target node.
2. Peer A sends RelayTransactionHashes with 32,767 unique fake 32-byte hashes.
   → queue size ≈ 32,767 (below threshold, Status::ok()).
3. Peer B sends RelayTransactionHashes with 32,767 different unique fake hashes.
   → queue size ≈ 65,534; overflow branch fires; Status::ignored() returned.
4. Send any further RelayTransactionHashes from any peer.
   → add_ask_for_txs acquires the mutex, inserts hashes, then iterates all
     ~65,534 entries in the O(N) scan before returning.
5. Benchmark: measure latency of add_ask_for_txs with queue empty vs. inflated.
6. Flood with concurrent messages from multiple connections; observe that
   pop_ask_for_txs and mark_as_known_txs are delayed proportionally.
7. Confirm the queue remains inflated indefinitely (attacker is never disconnected).
```
