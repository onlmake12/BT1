### Title
Verify-Queue Transaction Sizes Not Deducted from Pool-Map Size Limit Check, Causing Unexpected Eviction of Accepted Transactions - (File: tx-pool/src/pool.rs)

---

### Summary

The `limit_size` function in `tx-pool/src/pool.rs` enforces `max_tx_pool_size` by checking only `pool_map.total_tx_size`, which excludes transactions currently sitting in the `verify_queue`. Because the verify_queue can hold up to 256 MB of transactions that are committed to be added to the pool_map once verification completes, the pool_map's size-limit check systematically underestimates actual committed pool occupancy. When verify_queue transactions graduate into the pool_map, `limit_size` evicts previously accepted transactions to compensate, causing silent, unexpected drops of legitimate transactions.

---

### Finding Description

CKB's tx-pool is split into two stages:

1. **`verify_queue`** (`tx-pool/src/component/verify_queue.rs`) — holds transactions awaiting script verification. Its own size cap is `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256_000_000` bytes (256 MB). [1](#0-0) 

2. **`pool_map`** (`tx-pool/src/component/pool_map.rs`) — holds verified transactions in Pending/Gap/Proposed states. Its running total is `pool_map.total_tx_size`. [2](#0-1) 

The size-limit enforcement in `limit_size` reads:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee-rate entries …
}
``` [3](#0-2) 

`pool_map.total_tx_size` is updated only when a transaction is inserted into or removed from the `pool_map`. It never includes the bytes of transactions still sitting in the `verify_queue`. [4](#0-3) 

The verify_queue's own fullness check is independent and uses a separate counter:

```rust
pub fn is_full(&self, add_tx_size: usize) -> bool {
    add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size
}
``` [5](#0-4) 

These two counters are never combined when deciding whether the pool has room. The default `max_tx_pool_size` is 180 MB while `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` is 256 MB, so the verify_queue alone can hold more bytes than the entire pool_map limit. [1](#0-0) 

---

### Impact Explanation

**Scenario:**

1. `pool_map.total_tx_size` = 170 MB (10 MB below the 180 MB limit).
2. An attacker (or burst of peers) floods the verify_queue with 100 MB of large transactions. The verify_queue accepts them because 100 MB < 256 MB. The pool_map size check is not consulted at this stage.
3. A legitimate user submits a valid transaction. `pool_map.total_tx_size` (170 MB) < `max_tx_pool_size` (180 MB), so it is accepted and added to the pool_map (now 170 MB + tx_size).
4. The 100 MB of verify_queue transactions complete verification and are inserted into the pool_map one by one. Each insertion calls `limit_size`.
5. `limit_size` now sees `pool_map.total_tx_size` ≫ 180 MB and evicts the lowest-fee-rate entries — which may include the legitimate user's transaction that was accepted in step 3.

The legitimate transaction is silently dropped from the pool after having been accepted, with no indication to the submitter other than a `Reject::Full` callback. [6](#0-5) 

The `tx_pool_info` RPC exposes `total_tx_size` (pool_map only) and `verify_queue_size` (count, not bytes) as separate fields, so callers cannot easily compute the true committed occupancy. [7](#0-6) 

---

### Likelihood Explanation

Any unprivileged peer or RPC caller can submit transactions. The verify_queue is reachable via `submit_transaction` (local RPC), `submit_remote_tx` (relay), and `notify_tx` (internal). Under normal mainnet load the pool rarely sits near its 180 MB limit, but a targeted flood of large transactions (each up to the 512 KB `TRANSACTION_SIZE_LIMIT`) can fill the verify_queue while the pool_map appears to have headroom. The attacker does not need any special privilege, key, or majority hashpower. [8](#0-7) 

---

### Recommendation

When deciding whether to accept a new transaction into the verify_queue or the pool_map, include the current `verify_queue.total_tx_size` in the effective occupancy check against `max_tx_pool_size`. Concretely, the `is_full` guard in `VerifyQueue::add_tx` (and the `limit_size` loop) should treat `pool_map.total_tx_size + verify_queue.total_tx_size` as the combined committed size and compare it against `max_tx_pool_size`. This mirrors the fix recommended in the external report: account for committed-but-not-yet-executed amounts before advertising available capacity.

---

### Proof of Concept

```
1. Start a CKB node with default config (max_tx_pool_size = 180 MB).
2. Fill pool_map to ~170 MB with valid pending transactions.
3. Submit 90 MB of large transactions (each ~512 KB) via relay/RPC.
   → verify_queue accepts all (90 MB < 256 MB); pool_map unchanged at 170 MB.
4. Submit one legitimate transaction T_victim via send_transaction RPC.
   → pool_map.total_tx_size (170 MB) < 180 MB → T_victim accepted into pool_map.
5. Wait for the 90 MB of verify_queue transactions to complete verification.
   → Each is inserted into pool_map; limit_size fires repeatedly.
   → pool_map exceeds 180 MB; T_victim (lowest fee rate) is evicted with Reject::Full.
6. Query get_transaction(T_victim.hash()) → status "rejected" or "unknown".
   T_victim was accepted but silently dropped due to the accounting mismatch.
```
