Audit Report

## Title
TOCTOU Race in Tx-Pool Admission Allows Redundant Script Verification CPU Exhaustion — (File: `tx-pool/src/process.rs`)

## Summary
The `_process_tx` pipeline calls `check_txid_collision` under a read lock in `pre_check()`, releases that lock, then runs full script verification in `verify_rtx()` with no lock held and no in-flight marker. `submit_entry()` acquires a write lock but never re-calls `check_txid_collision`. N concurrent submissions of the same transaction each independently pass the duplicate check, each execute full script verification, and are only deduplicated via dead-outpoint detection in `submit_entry()` — after all N verifications have already completed.

## Finding Description
The three-phase pipeline in `_process_tx` (lines 705–753) is:

**Phase 1 — Read-locked check, then released:**
`pre_check()` (lines 269–316) acquires `with_tx_pool_read_lock`, calls `check_txid_collision` at line 282, and returns. The read lock is dropped when the closure exits at line 314. No in-flight marker is set.

**Phase 2 — Expensive work, no lock held:**
`verify_rtx()` is called at lines 724–732 with no lock held. It runs full contextual script verification via `ContextualTransactionVerifier::verify` up to `max_block_cycles` of CPU. Any number of concurrent goroutines can be executing this simultaneously for the same transaction.

**Phase 3 — Write-locked state update without txid re-check:**
`submit_entry()` (lines 96–170) acquires `with_tx_pool_write_lock`. It calls `check_rbf` or `find_conflict_outpoint` (lines 105–115) but never calls `check_txid_collision`. A duplicate transaction is rejected only because its inputs are already consumed (`OutPointError::Dead`) — but only after `verify_rtx()` has already completed for all N concurrent submissions.

The `check_txid_collision` function in `util.rs` (lines 20–26) checks `tx_pool.contains_proposal_id(&short_id)`. This check is authoritative only at the moment the read lock is held. Because no write to the pool occurs between Phase 1 and Phase 3, and because no in-flight set exists, all N concurrent submissions observe an empty pool during Phase 1 and proceed independently to Phase 2.

The same structural race exists in `resumeble_process_tx` (lines 335–353): `orphan_contains` (line 344) and `verify_queue_contains` (line 349) are separate, non-atomic lock acquisitions before `enqueue_verify_queue` (line 352). Similarly, `process_tx` (lines 401–426) checks `verify_queue_contains` at line 409 non-atomically before calling `_process_tx` at line 413.

## Impact Explanation
This is a sustained CPU exhaustion attack. An attacker submits the same high-cycle transaction N times concurrently via the RPC endpoint. The node performs N × `max_block_cycles` of wasted script verification CPU work. The verification worker pool is saturated, delaying or blocking legitimate transaction processing and propagation. A node unable to process and relay transactions in a timely manner contributes to CKB network congestion. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The attack requires no privileged access, no keys, and no special network position. Any entity that can reach the node's RPC port can submit transactions. Concurrent HTTP submissions are trivially achievable with standard tooling (e.g., `curl`, `ab`, any async HTTP client). The vulnerable window spans the entire duration of script verification — potentially hundreds of milliseconds for complex scripts. The attack is repeatable indefinitely with no per-attempt cost beyond a cheap RPC call.

## Recommendation
Re-check for txid collision inside `submit_entry()` under the write lock, before any other work, and return `Reject::Duplicated` immediately if the tx is already present:

```rust
pub(crate) async fn submit_entry(...) {
    self.with_tx_pool_write_lock(move |tx_pool, snapshot| {
        // Authoritative re-check under write lock
        check_txid_collision(tx_pool, entry.transaction())?;
        // ... existing RBF / conflict checks ...
    }).await
}
```

Additionally, maintain an in-flight set (a `HashSet<ProposalShortId>` protected by its own lock) populated atomically at the end of `pre_check()` and cleared at the end of `submit_entry()`, so that concurrent duplicate submissions are rejected before entering the expensive verification phase entirely. The `verify_queue_contains` / `enqueue_verify_queue` sequence in `resumeble_process_tx` should similarly be made atomic under a single lock acquisition.

## Proof of Concept
1. Obtain any valid transaction `tx` with scripts that consume close to `max_block_cycles` cycles.
2. Open N concurrent HTTP connections to the node's RPC endpoint.
3. Send `send_transaction(tx)` on all N connections simultaneously.
4. Observe via node metrics or CPU profiling that `verify_rtx` (lines 724–732 of `tx-pool/src/process.rs`) is invoked N times for the same transaction hash.
5. Confirm that only one submission succeeds; N−1 are rejected with `Reject::Resolve(OutPointError::Dead(...))` from `submit_entry()` (line 113), but only after all N verifications have completed.
6. Repeat indefinitely to sustain CPU saturation of the verification worker pool.
