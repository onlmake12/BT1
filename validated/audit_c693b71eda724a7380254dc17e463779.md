### Title
Unbounded `while` Loop in `executeCallback` Scales with Fulfilled Request Count, Enabling OOG and Permanent Fee Lock - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that linearly scans all cleared (fulfilled) sequence numbers to advance `_state.firstUnfulfilledSeq`. An attacker can create a large number of requests and fulfill all but the lowest-sequence-number one, causing every subsequent attempt to fulfill that first request to exceed the block gas limit. The victim's fee is permanently locked in the contract.

---

### Finding Description

In `Echo.sol`, after clearing a request, `executeCallback` runs:

```solidity
clearRequest(sequenceNumber);

// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration of the loop calls `findRequest`, which performs:
1. A `keccak256` hash to compute `shortKey`
2. An SLOAD on `_state.requests[shortKey]` (warm, ~100 gas)
3. If not matching, an SLOAD on `_state.requestsOverflow[key]` (cold, ~2,100 gas) [2](#0-1) 

For cleared requests, the overflow mapping key is unique per sequence number, so each iteration costs approximately **~2,230 gas** in cold SLOADs. With a 30M gas block limit, OOG occurs at roughly **~13,400 iterations**.

The `while` loop runs **before** the consumer callback and **after** `clearRequest`. If the loop OOGs, the entire transaction reverts — including `clearRequest` — so the request remains active but can never be fulfilled: every retry also OOGs.

The code's own TODO comment acknowledges this design flaw, and the gas benchmark test explicitly documents the worst-case scenario:

```
// The last fulfillment will be the most expensive since it needs
// to linearly scan through all the fulfilled requests in storage
// in order to update _state.lastUnfulfilledReq
``` [3](#0-2) 

The `State` struct stores `firstUnfulfilledSeq` as a `uint64` with no upper bound on the gap it must scan: [4](#0-3) 

---

### Impact Explanation

A victim who submitted request sequence number `S` (the lowest unfulfilled sequence number) has their fee permanently locked. Every call to `executeCallback(S)` OOGs in the while loop before `clearRequest` can take effect. The fee stored in `req.fee` is irrecoverable because there is no refund or cancellation path. The `firstUnfulfilledSeq` pointer is also permanently stuck, degrading `getFirstActiveRequests` for all future callers. [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged address can call both `requestPriceUpdatesWithCallback` and `executeCallback` (after the exclusivity period). An attacker creates ~14,000 requests, waits for or self-fulfills all but the lowest-sequence-number one, then the victim's request (or the attacker's own first request) is permanently stuck. The provider fee per request can be set to 1 wei by a registered provider, making the attack economically feasible. The gas benchmark test already demonstrates the linear scaling behavior with only 10 requests. [6](#0-5) 

---

### Recommendation

Replace the unbounded `while` loop with a doubly-linked list of active requests (as the TODO comment suggests), or track `firstUnfulfilledSeq` lazily without scanning in `executeCallback`. A minimal fix is to cap the number of iterations in the while loop per call (e.g., advance at most 64 steps), accepting that `firstUnfulfilledSeq` may lag but never causing OOG.

---

### Proof of Concept

1. Attacker registers as a provider with `baseFeeInWei = 1 wei`, `feePerFeedInWei = 1 wei`, `feePerGasInWei = 1 wei`.
2. Attacker calls `requestPriceUpdatesWithCallback` 14,000 times, obtaining sequence numbers 1 through 14,000. Sequence number 1 is the victim's (or attacker's own) target.
3. Attacker calls `executeCallback` for sequence numbers 2 through 14,000 (all except 1). Each call succeeds; the while loop stops immediately at `firstUnfulfilledSeq = 1` because request 1 is still active.
4. Now `firstUnfulfilledSeq = 1` and requests 2–14,000 are all cleared.
5. Attacker (or anyone) calls `executeCallback(1)`. After `clearRequest(1)`, the while loop starts at `firstUnfulfilledSeq = 1` and must scan 14,000 cleared slots: `14,000 × 2,230 gas ≈ 31.2M gas` → exceeds 30M block gas limit → OOG revert.
6. The transaction reverts entirely. Request 1 is not cleared. `firstUnfulfilledSeq` remains at 1.
7. Every subsequent call to `executeCallback(1)` also OOGs. The fee paid for request 1 is permanently locked in the contract. [1](#0-0) [7](#0-6)

    function findRequest(
        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];
        }
    }
