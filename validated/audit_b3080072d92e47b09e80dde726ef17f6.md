Audit Report

## Title
Missing Per-Peer Rate Limiting on `GetBlockFilters` Enables Resource Exhaustion DoS — (`sync/src/filter/get_block_filters_process.rs`)

## Summary
The `BlockFilter` protocol handler performs no per-peer rate limiting before dispatching `GetBlockFilters` messages to `GetBlockFiltersProcess::execute()`. Each such message triggers up to 1,000 sequential RocksDB reads and up to ~1.8 MB of heap allocation. The `Relayer` protocol, handling analogous request-response patterns, explicitly enforces a 30 req/s per-peer rate limit via `governor::RateLimiter`; no equivalent guard exists anywhere in the filter module. Any unprivileged peer with a valid P2P connection can exploit this to exhaust disk I/O, CPU, and memory on the target node.

## Finding Description
`GetBlockFiltersProcess::execute()` iterates up to `BATCH_SIZE = 1000` times, calling `active_chain.get_block_hash(block_number)` and `active_chain.get_block_filter(&block_hash)` per iteration — two RocksDB reads per block — accumulating results into `block_hashes` and `filters` Vecs until the 1.8 MB size cap is hit, then serializes and sends the full response: [1](#0-0) [2](#0-1) [3](#0-2) 

The `BlockFilter` struct carries only a `shared: Arc<SyncShared>` field — no rate limiter — and `try_process` dispatches directly to `execute()` with zero rate checking: [4](#0-3) [5](#0-4) 

By contrast, `Relayer` defines a `RateLimiter<(PeerIndex, u32)>` keyed by peer and message type, enforcing 30 req/s, and checks it before any message dispatch: [6](#0-5) [7](#0-6) [8](#0-7) 

No analogous guard exists anywhere in `sync/src/filter/`.

## Impact Explanation
Each `GetBlockFilters(start_number=0)` request causes up to 2,000 RocksDB reads, up to ~1.8 MB heap allocation, molecule serialization, and an async send. With no rate limit, a single peer can saturate the node's async task queue and disk I/O subsystem. Multiple concurrent peers amplify this linearly. The result is CPU/I/O exhaustion and memory pressure leading to node unresponsiveness or OOM crash.

This matches the allowed CKB bounty impact: **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points). A node with `block_filter` enabled and a long chain (normal production state) is fully vulnerable.

## Likelihood Explanation
The attack requires only a valid P2P connection and the ability to send well-formed `GetBlockFilters` messages. The molecule struct is a single `Uint64` field — trivial to construct: [9](#0-8) 

No authentication, PoW, stake, or special privilege is required. The precondition — a node with `block_filter` enabled and filters built for a long chain — is the normal production state of any full node with this feature active. The attack is repeatable, scalable with peer count, and requires no victim mistakes.

## Recommendation
Add a `RateLimiter<(PeerIndex, u32)>` field to the `BlockFilter` struct, mirroring the pattern in `Relayer`. In `try_process`, before dispatching to any `*Process::execute()`, call `rate_limiter.check_key(&(peer, message.item_id()))` and return `StatusCode::TooManyRequests` (with optional peer ban) on violation. The 30 req/s quota used by `Relayer` is a reasonable starting point; a lower value (e.g., 5–10 req/s) may be more appropriate given the higher per-request cost of filter reads versus relay message processing.

## Proof of Concept
```rust
// Attacker: connect N peers, each sending GetBlockFilters(0) in a tight loop
let msg = packed::BlockFilterMessage::new_builder()
    .set(packed::GetBlockFilters::new_builder()
        .start_number(0u64.pack())
        .build())
    .build();
loop {
    net.send(&node, SupportProtocols::Filter, msg.as_bytes());
    // No sleep — no server-side rate limit will stop this.
    // Each iteration causes the server to perform up to 2,000 RocksDB reads
    // and allocate up to ~1.8 MB. With N peers, exhaustion is linear.
}
```

To reproduce: spin up a CKB full node with `block_filter = true` and a synced chain of >1,000 blocks. Connect N attacker peers and have each send `GetBlockFilters { start_number: 0 }` in a tight loop. Monitor RocksDB read latency, async task queue depth, and RSS memory growth. The node will become unresponsive or OOM-killed as N and request rate increase.
