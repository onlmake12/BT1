### Title
Legacy `gasLimit10k == 0` Path in `revealWithCallback` Propagates Callback Revert, Permanently Locking Requests — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When a provider has `defaultGasLimit == 0` (the default state for any provider that has never called `setDefaultGasLimit`), any `requestWithCallback` request stores `req.gasLimit10k = 0`. In `revealWithCallback`, this causes the legacy `else` branch to execute, which calls `_entropyCallback` directly with no error catching. If the consumer's callback always reverts, the entire `revealWithCallback` transaction reverts — including the preceding `clearRequest` — leaving the request permanently active and unfulfillable.

---

### Finding Description

**Step 1 — `gasLimit10k` is set to 0 for legacy providers.**

In `requestHelper`, when `providerInfo.defaultGasLimit == 0`:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;
}
``` [1](#0-0) 

This is the default state for every provider that has never called `setDefaultGasLimit`. The comment says this "disables" the new callback failure state flow.

**Step 2 — `revealWithCallback` takes the legacy branch.**

The new safe path is only entered when `req.gasLimit10k != 0 && callbackStatus == CALLBACK_NOT_STARTED`:

```solidity
if (
    req.gasLimit10k != 0 &&
    req.callbackStatus == EntropyStatusConstants.CALLBACK_NOT_STARTED
) {
    // excessivelySafeCall — catches reverts
} else {
    clearRequest(provider, sequenceNumber);   // ← cleared first
    // ...
    IEntropyConsumer(callAddress)._entropyCallback(...);  // ← direct, uncaught
}
``` [2](#0-1) 

**Step 3 — Revert in callback undoes `clearRequest`.**

`clearRequest` is called at line 666 **before** the callback at line 676. If `_entropyCallback` reverts, the EVM unwinds the entire transaction, restoring the request to its active state. There is no `try/catch`, no `excessivelySafeCall`, and no `CALLBACK_FAILED` state transition in this branch. [3](#0-2) 

**Step 4 — No recovery path exists.**

`reveal` is gated to `CALLBACK_NOT_NECESSARY` requests and will revert for callback requests: [4](#0-3) 

`revealWithCallback` will always revert for this request. There is no timeout, no cancellation function, and no admin escape hatch. The request is permanently active.

**Step 5 — The test confirms the revert.**

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` (with `provider1` having `defaultGasLimit == 0`) explicitly asserts `vm.expectRevert()` on `revealWithCallback` when the consumer reverts — but does not assert the request is cleared afterward, confirming the stuck state is the actual outcome: [5](#0-4) 

---

### Impact Explanation

- Every `revealWithCallback` attempt by the provider's keeper reverts, wasting keeper gas indefinitely.
- The request slot remains active forever; the provider's sequence number is consumed and unrecoverable.
- An attacker can repeat this for every sequence number the provider has, at the cost of the request fee per slot.
- There is no on-chain mechanism to clear or cancel the stuck request.

This matches the scoped impact: **griefing without profit motive** — the attacker pays fees but permanently degrades keeper liveness for a targeted provider.

---

### Likelihood Explanation

- Any provider that has not explicitly called `setDefaultGasLimit` is vulnerable by default.
- The attacker needs only to deploy a contract with a reverting `_entropyCallback` and pay the request fee — no privileged access required.
- The attack is repeatable and cheap relative to the keeper gas wasted.

---

### Recommendation

In the legacy `else` branch of `revealWithCallback`, wrap the `_entropyCallback` call in a `try/catch` or use `excessivelySafeCall` (as the new path already does), and emit a `CallbackFailed`-equivalent event rather than propagating the revert. Alternatively, enforce that `defaultGasLimit != 0` is required for `requestWithCallback` to be accepted, forcing all providers into the safe path.
