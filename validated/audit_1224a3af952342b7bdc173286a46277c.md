### Title
`check_tx_fee` Enforces Minimum Fee Rate Against Size Only, Ignoring Cycles — Allows Effective Fee Rate Bypass - (File: `tx-pool/src/util.rs`)

### Summary

The `check_tx_fee` function in the CKB tx-pool enforces the minimum fee rate using only the transaction's serialized byte size as the weight. However, the canonical transaction weight used everywhere else in the system — sorting, eviction, fee estimation, and block assembly — is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. A transaction sender can craft a small-size, high-cycles transaction that passes the size-only fee gate but whose actual effective fee rate is far below the configured minimum.

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee as:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The code itself acknowledges this is theoretically incorrect. The canonical weight function used everywhere else is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

This weight function is used for:
- **Fee rate computation** in `TxEntry::fee_rate()`: [3](#0-2) 
- **Sorting and eviction** via `AncestorsScoreSortKey` and `EvictKey`: [4](#0-3) 
- **Fee rate statistics** returned by the RPC: [5](#0-4) 
- **Fee estimation algorithms**: [6](#0-5) 

The pool eviction is triggered only when `total_tx_size` exceeds `max_tx_pool_size`:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
``` [7](#0-6) 

There is no corresponding `total_tx_cycles > max_cycles` eviction trigger in the current codebase. High-cycles, low-size transactions do not pressure the size-based eviction boundary.

### Impact Explanation

**Concrete bypass with default config** (`min_fee_rate = 1000 shannons/KW`, `max_tx_verify_cycles = 70,000,000`, `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`):

| Parameter | Value |
|---|---|
| `tx_size` | 1,000 bytes |
| `cycles` | 70,000,000 |
| Weight used by `check_tx_fee` | 1,000 |
| Actual canonical weight | `max(1000, 70,000,000 × 0.000_170_571_4)` = **11,940** |
| Fee required by gate | **1,000 shannons** |
| Fee that should be required | **11,940 shannons** |
| Effective fee rate paid | ~83.8 shannons/KW (vs. 1,000 minimum) |

An attacker pays ~8.4% of the fee that the minimum fee rate policy intends to require. They can flood the mempool with high-cycles transactions at a fraction of the expected cost. Because these transactions have small size, they do not trigger the size-based eviction loop. They consume block cycle budget, potentially crowding out legitimate transactions from block templates.

### Likelihood Explanation

This is reachable by any unprivileged actor who can submit a transaction via the `send_transaction` RPC or via P2P relay. No special role, key, or majority hashpower is required. The attacker only needs to construct a transaction whose lock/type script consumes many cycles (e.g., a computationally intensive RISC-V script) while keeping the serialized transaction small. The `max_tx_verify_cycles` config (default 70,000,000) bounds the per-transaction cycle count but does not prevent the bypass — it only limits the magnitude of the discount.

### Recommendation

Replace the size-only weight in `check_tx_fee` with the canonical `get_transaction_weight(tx_size, cycles)`. Since cycles are not known at the pre-check stage (before script execution), the check should be applied **after** verification completes and the actual cycle count is known, using the same weight function used for sorting and eviction:

```rust
// After verification, cycles are known
let weight = get_transaction_weight(tx_size, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

Alternatively, apply a conservative upper-bound cycle estimate at pre-check time and re-validate after execution.

### Proof of Concept

1. Construct a CKB transaction with a lock script that loops for ~70,000,000 cycles but whose serialized size is ~1,000 bytes.
2. Set the transaction fee to exactly `min_fee_rate * tx_size / 1000` = 1,000 shannons (with `min_fee_rate = 1000`).
3. Submit via `send_transaction` RPC.
4. Observe: the transaction is accepted into the mempool despite its actual effective fee rate being ~83.8 shannons/KW, far below the 1,000 shannons/KW minimum.
5. Repeat with many such transactions. Each occupies ~1,000 bytes of pool size but ~11,940 bytes of effective block weight. The mempool fills with cycle-heavy transactions at 8.4% of the intended minimum cost.
