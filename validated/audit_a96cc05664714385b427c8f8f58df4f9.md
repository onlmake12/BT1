### Title
Ghost ancestor IDs from evicted descendants cause `get_by_id_checked` panic in `_record_ancestors` ‚Äî (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`check_and_record_ancestors` evicts a `cell_ref_parent` and all its descendants via `remove_entry_and_descendants`, but only removes the direct evicted ID from `parents`. If a descendant of the evicted tx is also a direct parent of the incoming tx, it remains in `parents` as a ghost ID. `calc_relation_ids` unconditionally inserts every ID from `stage` into `relation_ids` regardless of pool membership, so the ghost ID propagates into `ancestors`. `_record_ancestors` then calls `get_by_id_checked` on the ghost ID, which panics with `"inconsistent pool"`, crashing the node.

---

### Finding Description

**Step 1 ‚Äî `calc_relation_ids` unconditionally inserts IDs not in `self.inner`** [1](#0-0) 

The `if let Some(tx_links) = self.inner.get(&id)` at line 61 only gates the inner loop. Lines 68‚Äì69 (`stage.remove(&id); relation_ids.insert(id);`) execute unconditionally. Any ID in the initial `stage` that is absent from `self.inner` is silently included in the returned set.

**Step 2 ‚Äî Eviction loop removes descendants but only prunes the root from `parents`** [2](#0-1) 

`remove_entry_and_descendants(next_id)` removes `next_id` **and all its descendants** from `self.links.inner`: [3](#0-2) 

But the cleanup on line 620 is only `parents.remove(next_id)` ‚Äî it does not remove descendants of `next_id` that were also deleted from the pool.

**Step 3 ‚Äî Ghost ID survives into `ancestors`** [4](#0-3) 

`calc_relation_ids` is called with the stale `parents` set. Any descendant of the evicted tx that was also a direct parent of the incoming tx is now a ghost ID in `parents` ‚Äî absent from `self.links.inner` but still present in `stage`, so it gets inserted into `relation_ids`.

**Step 4 ‚Äî Panic in `_record_ancestors`** [5](#0-4) 

`get_by_id_checked` calls `.expect("inconsistent pool")` on the ghost ID: [6](#0-5) 

---

### Impact Explanation

A Rust `expect` panic unwinds the tx-pool service thread. Because CKB's tx-pool runs as an actor/service without a `catch_unwind` boundary around this path, the panic propagates and terminates the node process. Any peer that can submit transactions to the mempool (P2P `send_transaction` or RPC `send_transaction`) can trigger this.

---

### Likelihood Explanation

The attacker must craft a transaction graph satisfying two conditions simultaneously:
1. A `cell_ref_parent` tx_C exists (tx_C's output is used as a cell dep by some in-pool tx whose output the new tx spends as an input).
2. A descendant of tx_C (tx_D, which spends tx_C's output) is also a direct parent of the new tx (the new tx spends tx_D's output).
3. The total ancestor count exceeds `max_ancestors_count` to enter the eviction branch.

All three conditions are achievable by an unprivileged attacker who submits a carefully ordered sequence of transactions via the public RPC or P2P relay. No privileged access, key material, or majority hashpower is required.

---

### Recommendation

In the eviction loop, after calling `remove_entry_and_descendants(next_id)`, remove **all** IDs that were actually deleted from `parents`, not just `next_id`:

```rust
let removed = self.remove_entry_and_descendants(next_id);
let removed_count = removed.len();
for r in &removed {
    parents.remove(&r.proposal_short_id());
}
ancestors_count = ancestors_count.saturating_sub(removed_count);
evicted.extend(removed);
```

Additionally, `calc_relation_ids` should skip inserting IDs absent from `self.inner` to make the function safe against stale inputs:

```rust
stage.remove(&id);
if self.inner.contains_key(&id) {
    relation_ids.insert(id);
}
```

### Proof of Concept

```
Pool state before new tx submission:
  tx_A  (root, no parents)
  tx_C  (spends tx_A output; tx_C's output is used as cell dep by tx_X)
  tx_D  (spends tx_C output)
  tx_X  (uses tx_C output as cell dep; produces output O_X)
  ... (enough ancestors to exceed max_ancestors_count)

New tx (tx_NEW):
  inputs:  [O_X (from tx_X), O_D (from tx_D)]
  ‚Üí tx_X is a cell_ref_parent (its input O_X is produced by tx_X which cell-deps tx_C)
  ‚Üí tx_D is a direct parent (tx_NEW spends tx_D's output)

Trigger:
  ancestors_count > max_ancestors_count  ‚Üí enters eviction branch
  evict_candidates = [tx_X]  (lowest-fee cell_ref_parent)
  remove_entry_and_descendants(tx_X) removes tx_X from self.links.inner
    (tx_X has no descendants here, so only tx_X is removed)

Wait ‚Äî let me correct the PoC to match the exact condition:

Pool state:
  tx_C  (cell_ref_parent; tx_C's output used as cell dep by tx_X)
  tx_D  (spends tx_C output directly ‚Üí descendant of tx_C in links)
  tx_X  (uses tx_C output as cell dep; produces O_X)
  tx_NEW spends O_X (making tx_X a cell_ref_parent) AND spends O_D (making tx_D a direct parent)

  evict_candidates = [tx_C]  (tx_C is the cell_ref_parent with lowest fee)
  remove_entry_and_descendants(tx_C):
    ‚Üí removes tx_C from self.links.inner
    ‚Üí calc_descendants(tx_C) returns {tx_D, tx_X}
    ‚Üí removes tx_D and tx_X from self.links.inner
  parents.remove(tx_C)  ‚Üí tx_C gone from parents
  tx_D still in parents (ghost)

  calc_relation_ids({tx_D, ...}, Parents)
    ‚Üí tx_D not in self.inner ‚Üí skips inner loop, but inserts tx_D into relation_ids
  _record_ancestors iterates ancestors containing tx_D
    ‚Üí get_by_id_checked(tx_D) ‚Üí .expect("inconsistent pool") ‚Üí PANIC ‚Üí node crash
```
