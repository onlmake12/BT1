Audit Report

## Title
Unauthenticated GetHeaders Bandwidth Amplification via At-Limit Locator — (`sync/src/synchronizer/get_headers_process.rs`)

## Summary
A remote peer can send a `GetHeaders` message with exactly `MAX_LOCATOR_SIZE` (101) locator entries — 100 all-zero hashes followed by the real genesis hash — to force a fully-synced node to respond with up to 2,000 block headers (~416 KB) per request. No ban is triggered, no rate limit exists, and the attack is indefinitely repeatable over a single persistent TCP connection, yielding a ~126× byte amplification ratio.

## Finding Description
**Step 1 — Size guard bypassed at the boundary.**
`execute` checks `locator_size > MAX_LOCATOR_SIZE` (strict `>`). With `MAX_LOCATOR_SIZE = 101`, a locator of exactly 101 entries passes unconditionally. [1](#0-0) [2](#0-1) 

**Step 2 — `locate_latest_common_block` returns `Some(0)` without banning.**
The function requires the last locator entry to equal the genesis hash; the attacker satisfies this by placing the real genesis hash at index 100. The preceding 100 zero hashes all return `None` from `get_block_number`. The iterator finds the first `Some` at the genesis entry (index 100), and the early-exit condition `latest_common == Some(0)` fires immediately, returning `Some(0)`. [3](#0-2) 

**Step 3 — `get_locator_response(0, Byte32::zero())` emits up to 2,000 headers.**
Starting from block 1, the iterator runs up to `MAX_HEADERS_LEN = 2000`. The `hash_stop` is `Byte32::zero()`, which never matches any real block hash, so the stop condition is never triggered on a synced node. [4](#0-3) [5](#0-4) 

**Step 4 — No ban, no rate limit.**
`execute` returns `Status::ok()` (code 100) on the success path. `should_ban()` only acts on codes in `400..500`; code 100 returns `None`. `TooManyRequests = 110` is defined but never applied in this handler. No per-peer request counter or cooldown exists anywhere in the `GetHeaders` processing path. [6](#0-5) [7](#0-6) [8](#0-7) 

**Step 5 — Existing test does not cover the at-limit case.**
`InvalidLocatorSize` uses `0..=MAX_LOCATOR_SIZE` which produces 102 entries (correctly banned), leaving the exactly-101-entry case with a valid genesis tail completely untested. [9](#0-8) 

## Impact Explanation
An attacker with a single persistent connection can saturate the victim node's upload bandwidth and storage I/O by sending the crafted message in a tight loop. Each iteration forces 101 DB reads (locator lookups) plus up to 4,000 additional reads (hash + header per block), and transmits ~416 KB back to the attacker for a ~3.3 KB input. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker operating from multiple IPs or connections can simultaneously target many nodes, amplifying the network-wide effect.

## Likelihood Explanation
The attack requires only a standard P2P connection — no proof-of-work, no keys, no privileges. The crafted locator is trivially constructable from public knowledge of the genesis hash. The off-by-one in the size guard is a stable, deterministic property of the current code. The attack is fully repeatable with no cooldown or backoff imposed by the node.

## Recommendation
1. **Change the size check to `>=`**: `if locator_size >= MAX_LOCATOR_SIZE` rejects the maximally-crafted 101-entry case at the gate.
2. **Rate-limit `GetHeaders` per peer**: Apply `StatusCode::TooManyRequests` (already defined at code 110) when a peer exceeds N requests per second, returning early before any DB access.
3. **Treat genesis-anchored responses as suspicious**: If `locate_latest_common_block` resolves to block 0 and `locator_size` is at or near the maximum, apply a per-peer cooldown or reduce the response window.
4. **Add a regression test**: Extend `InvalidLocatorSize` to cover the exactly-101-entry case with a valid genesis tail and assert that no 2,000-header response is returned.

## Proof of Concept
```rust
// Requires: a running non-IBD CKB node with tip >> 2000
let genesis_hash = node.consensus().genesis_hash();
let mut hashes: Vec<Byte32> = (0..100).map(|_| Byte32::zero()).collect();
hashes.push(genesis_hash);
assert_eq!(hashes.len(), MAX_LOCATOR_SIZE); // exactly 101

let msg = SyncMessage::new_builder()
    .set(GetHeaders::new_builder()
        .block_locator_hashes(hashes)
        .hash_stop(Byte32::zero())
        .build())
    .build()
    .as_bytes();

// Send in a tight loop; assert:
// 1. Each iteration receives a SendHeaders response with 2,000 headers (~416 KB)
// 2. The peer is never banned (connection remains open indefinitely)
// 3. No rate-limit response is ever returned
loop {
    net.send(node, SupportProtocols::Sync, msg.clone());
    let response = net.receive(); // expect SendHeaders with len == 2000
    assert_eq!(response.headers().len(), MAX_HEADERS_LEN);
}
```
