### Title
Fee Sufficiency Check Compares Raw Amounts Across Different Token Denominations Without Conversion - (`target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

---

### Summary

The Osmosis-specific `is_fee_sufficient()` function in the Pyth CosmWasm contract sums raw token amounts from multiple different denominations (e.g., `uosmo`, `uion`, IBC tokens) into a single `total_amount` and compares it directly against `base_denom_fee.amount`, which is denominated exclusively in `uosmo`. Because no exchange-rate conversion is applied, a caller can satisfy the fee check by paying in a lower-value allowed token, underpaying the required fee in economic terms.

---

### Finding Description

In `target_chains/cosmwasm/contracts/pyth/src/contract.rs`, the Osmosis build of `is_fee_sufficient()` iterates over all coins in `info.funds`, validates each denom against the allowed list, and accumulates their raw amounts into `total_amount`:

```rust
let mut total_amount = 0u128;
for coin in &info.funds {
    if coin.denom != state.fee.denom && !is_allowed_tx_fees_denom(deps, &coin.denom) {
        return Err(PythContractError::InvalidFeeDenom { ... })?;
    }
    total_amount = total_amount.checked_add(coin.amount.u128())...;
}
let base_denom_fee = get_update_fee(deps, data)?;
Ok(base_denom_fee.amount.u128() <= total_amount)
```

`base_denom_fee` is always returned in the contract's configured denom (`uosmo`), while `total_amount` is the raw sum of amounts from any mix of `uosmo`, `uion`, and IBC-wrapped tokens. These are compared as plain integers with no spot-price conversion. The code itself acknowledges this is unfinished:

> *"It's not important to convert the price right now. For now we are keeping the base fee amount same for each valid denom → 1 but this logic will be updated to use spot price for different valid tokens in future"* [1](#0-0) 

The `get_update_fee_for_denom()` function has the same design: it returns the same raw `amount` regardless of which denom is requested, explicitly noting this will be corrected later. [2](#0-1) 

---

### Impact Explanation

Any unprivileged caller of `update_price_feeds` (or `parse_price_feed_updates`) on the Osmosis deployment can pay the required fee using any Osmosis-allowed denom (e.g., `uion`) instead of `uosmo`. Because the check compares raw amounts without conversion, 1 `uion` satisfies a fee requirement of 1 `uosmo` regardless of the actual market exchange rate between the two tokens. If the fee is ever raised to a non-negligible amount, callers can systematically underpay by choosing the cheapest allowed denom. The contract collects no `uosmo` in such cases, and the fee revenue is economically less than intended.

---

### Likelihood Explanation

The entry path is fully permissionless: any address can call `update_price_feeds` on the Osmosis mainnet deployment with `info.funds` denominated in `uion` or any IBC token accepted by Osmosis's txfee module. No privileged role, leaked key, or governance action is required. The current fee is set to 1 `uosmo` per VAA (negligible), but the structural bug is live and exploitable at any fee level. [3](#0-2) 

---

### Recommendation

Before summing `total_amount`, convert each non-base-denom coin to its `uosmo` equivalent using Osmosis's spot-price or TWAP oracle (e.g., via `TxfeesQuerier` or the Osmosis `PoolManager` query). Only after all amounts are expressed in the same unit should they be summed and compared against `base_denom_fee.amount`. The existing `TODO` comment already identifies this as the intended fix.

---

### Proof of Concept

1. The Pyth contract on Osmosis is configured with `fee = Coin { denom: "uosmo", amount: N }`.
2. Attacker calls `update_price_feeds` with `info.funds = [Coin { denom: "uion", amount: N }]`.
3. `is_allowed_tx_fees_denom` returns `true` for `"uion"` (it has a pool ID in Osmosis's txfee module).
4. `total_amount` accumulates to `N`.
5. `base_denom_fee.amount` is also `N` (in `uosmo`).
6. `Ok(N <= N)` → `true` — fee check passes.
7. The attacker has paid `N uion` instead of `N uosmo`. If `uion < uosmo` in market value, the attacker has underpaid the required fee.

The test suite confirms this behavior is intentional/known:

```rust
// a valid denom other than base denom with sufficient fee
info.funds = coins(100, "uion");
let result = is_fee_sufficient(deps, info.clone(), data);
assert_eq!(result, Ok(true));  // passes with uion == uosmo raw amount
``` [4](#0-3)

    // base amount is multiplied to number of vaas to get the total amount

    // this will be change later on to add custom logic using spot price for valid tokens
