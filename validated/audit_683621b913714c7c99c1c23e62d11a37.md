Audit Report

## Title
Minimum Fee Admission Check Uses Only Byte Size, Ignoring Cycles Dimension — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` enforces the minimum fee rate using only the transaction's serialized byte size, completely ignoring VM execution cycles. Because CKB blocks are constrained by two independent limits (`MAX_BLOCK_BYTES` and `MAX_BLOCK_CYCLES`), a transaction that is small in bytes but consumes near-maximum cycles can be admitted to the pool while paying a fee calibrated only to its byte footprint — up to ~119× below the weight-based minimum. This allows an unprivileged attacker to flood the verification pipeline with high-cycle, minimum-fee transactions at negligible cost, causing sustained CPU exhaustion on nodes and distorting the fee market.

## Finding Description
**Root cause — `check_tx_fee` ignores cycles:**

`check_tx_fee` in `tx-pool/src/util.rs` (L28–54) computes the minimum fee using only `tx_size`: [1](#0-0) 

The comment at L42–44 explicitly acknowledges the theoretical incorrectness. The function signature takes no `cycles` parameter, so it structurally cannot account for the cycles dimension. [2](#0-1) 

**`get_transaction_weight` exists but is never called in the admission path:**

`util/types/src/core/tx_pool.rs` (L298–303) provides the correct unified weight: [3](#0-2) 

This function is imported and used in `tx-pool/src/component/entry.rs` for fee-rate sorting and prioritization inside the pool, but is never invoked during the admission gate. [4](#0-3) 

**No post-verification fee cross-check:**

In `_process_tx` (`tx-pool/src/process.rs`, L705–777), the flow is:
1. `pre_check` → `check_tx_fee(tx_pool, snapshot, rtx, tx_size)` — size-only gate passes [5](#0-4) 
2. `verify_rtx(...)` — executes scripts, returns `verified.cycles` [6](#0-5) 
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with actual cycles, no weight-based fee cross-check [7](#0-6) 

After `verify_rtx` returns the actual cycle count, there is no step that cross-checks the fee against `get_transaction_weight(tx_size, verified.cycles)`. The cycles are verified only against `max_tx_verify_cycles`, never against the fee.

**Concrete underpayment with default config (`max_tx_verify_cycles = 70,000,000`):**

| Parameter | Value |
|---|---|
| Transaction size | 100 bytes |
| Transaction cycles | 70,000,000 |
| Cycles-equivalent bytes | 70M × 0.000_170_571_4 ≈ 11,940 bytes |
| Actual weight | max(100, 11940) = **11,940** |
| Minimum fee (size-only, admitted) | 1,000 × 100/1,000 = **100 shannons** |
| Minimum fee (weight-based, correct) | 1,000 × 11,940/1,000 = **11,940 shannons** |
| Underpayment ratio | **~119×** |

**Existing guards are insufficient:**

- `max_tx_verify_cycles` caps cycles per transaction but does not enforce fee proportionality.
- The byte-based pool size limit (`max_tx_pool_size`) caps total bytes, not total verification CPU consumed.
- The `limit_size` eviction mechanism evicts low-weight-fee-rate entries only after they are already admitted and have already consumed verification CPU.

## Impact Explanation
This matches the allowed CKB bounty impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker submitting 100-byte, 70M-cycle transactions at 100 shannons each forces each receiving node to execute 70M cycles of CKB-VM work per transaction during `verify_rtx`. With `max_tx_verify_workers` parallel workers and a byte-bounded pool of 180 MB, up to ~1.8 million such transactions can be admitted before the byte limit is hit. Each transaction consumes significant CPU during verification while paying a fee 119× below the weight-based threshold. This can saturate the verification queue, delay legitimate transaction processing, and cause sustained CPU load across nodes that relay and verify these transactions — constituting network congestion at negligible attacker cost.

Secondary impact: miners who include these transactions receive fees calibrated to ~100 bytes of block space while the transaction consumes ~11,940 bytes-equivalent of cycle space, breaking the fee market signal that `min_fee_rate` is intended to enforce.

## Likelihood Explanation
The exploit requires only an unprivileged `send_transaction` RPC call or P2P relay. No special keys, majority hashpower, or victim mistakes are needed. The attacker only needs to:
1. Construct a valid CKB transaction with minimal serialized size (~100 bytes)
2. Include a lock script that loops for ~70M cycles (standard script authoring capability)
3. Pay `min_fee_rate × tx_size / 1000` shannons (100 shannons at default config)

This is repeatable indefinitely and is straightforward for any script author. The discrepancy is largest for the class of transactions with high cycles and small byte size — a class that includes many legitimate use cases (complex lock scripts with minimal data), making the attack indistinguishable from normal traffic.

## Recommendation
Split the fee check into two stages:

1. **Pre-verification (current):** Keep the size-only check as a cheap early gate to reject obviously underpaying transactions before incurring verification cost.

2. **Post-verification:** After `verify_rtx` returns `verified.cycles`, add a weight-based check in `_process_tx` (`tx-pool/src/process.rs`) between L734 and L751:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Alternatively, redefine `min_fee_rate` in terms of weight-units rather than bytes, and update `check_tx_fee` to accept a `cycles` parameter (defaulting to 0 pre-verification, actual value post-verification).

## Proof of Concept
1. Craft a CKB transaction:
   - One input cell, one output cell (serialized size ≈ 100 bytes)
   - Lock script that loops for ~70,000,000 cycles
   - Fee = `1000 * 100 / 1000 = 100 shannons`

2. Submit via `send_transaction` RPC.

3. `pre_check` calls `check_tx_fee(tx_pool, snapshot, rtx, tx_size=100)`:
   - `min_fee = 1000.fee(100) = 100 shannons`
   - `fee = 100 shannons` → `fee >= min_fee` → **passes** (`tx-pool/src/util.rs` L45–52)

4. `verify_rtx` executes the script consuming 70,000,000 cycles. Cycles are checked only against `max_tx_verify_cycles` — no fee cross-check (`tx-pool/src/process.rs` L724–734).

5. `TxEntry::new(rtx, 70_000_000, 100_shannons, 100)` is created and submitted to pool (`tx-pool/src/process.rs` L751–753).

6. Transaction enters pool with effective fee rate ≈ 8 shannons/KW — ~119× below `min_fee_rate = 1,000 shannons/KW`.

7. Repeat to fill the verification queue and pool with high-CPU, minimum-fee transactions, each having consumed 70M cycles of node verification work for 100 shannons of fee.
