### Title
`MAX_DEPOSIT_LIMIT` Checked Per-Transaction Instead of Cumulatively, Allowing Permanent Subscriptions to Accumulate Unbounded Locked ETH — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `MAX_DEPOSIT_LIMIT` (100 ETH) guard for permanent subscriptions in the Pyth Pulse `Scheduler` contract validates only the **current transaction's `msg.value`**, not the **cumulative `status.balanceInWei`**. Because permanent subscriptions permanently block all withdrawals, any ETH deposited beyond what the subscription will ever consume is irrecoverably locked. An unprivileged subscription manager can bypass the cap entirely by calling `addFunds` repeatedly with amounts ≤ 100 ETH each.

---

### Finding Description

In `addFunds`, the deposit cap guard is:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

The condition checks `msg.value > MAX_DEPOSIT_LIMIT` (100 ETH), not `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`. The same per-transaction-only check appears in `createSubscription`:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

`withdrawFunds` unconditionally reverts for permanent subscriptions:

```solidity
// Prevent withdrawals from permanent subscriptions
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

`updateSubscription` also unconditionally reverts for permanent subscriptions:

```solidity
if (currentParams.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

There is therefore no path to recover ETH from a permanent subscription once deposited. The cumulative balance is unbounded.

---

### Impact Explanation

A subscription manager of a permanent subscription can:

1. Create a permanent subscription with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH) — passes the check.
2. Call `addFunds` with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH) — passes the check again; `balanceInWei` is now 200 ETH.
3. Repeat N times — `balanceInWei` grows to N × 100 ETH.

All deposited ETH beyond what keeper fees will ever consume is permanently locked in the contract with no recovery path. The `MAX_DEPOSIT_LIMIT` constant is documented as "Maximum deposit limit for permanent subscriptions in wei", confirming the intent is a cumulative cap, not a per-transaction cap. The bypass renders the safety limit meaningless.

---

### Likelihood Explanation

The entry path requires only that the caller be the subscription manager of a permanent subscription — an unprivileged role any user can self-assign by calling `createSubscription` with `isPermanent = true`. No special access, leaked key, or governance majority is needed. The bypass is a direct, deterministic sequence of `addFunds` calls.

---

### Recommendation

Change both the `createSubscription` and `addFunds` guards to check the **post-deposit cumulative balance**:

```solidity
// In createSubscription:
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// In addFunds — replace with:
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

This mirrors the correct pattern: the cap must be enforced against the running total, not the incremental deposit.

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription at the cap
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);
// status.balanceInWei == 100 ether — at the limit

// 2. Add funds again at the cap — check passes because msg.value == MAX_DEPOSIT_LIMIT (not >)
scheduler.addFunds{value: 100 ether}(subId);
// status.balanceInWei == 200 ether — cap bypassed

// 3. Attempt withdrawal — permanently blocked
scheduler.withdrawFunds(subId, 100 ether);
// reverts: CannotUpdatePermanentSubscription

// 4. Repeat step 2 arbitrarily — balance grows to N * 100 ETH, all irrecoverable
```
