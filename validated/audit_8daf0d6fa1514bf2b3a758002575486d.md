### Title
Unbounded NAT Traversal Task Accumulation via Non-Consuming `pending_delivered` Read in `ConnectionSyncProcess::execute` — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

`ConnectionSyncProcess::execute()` reads `pending_delivered` with `.get()` but never calls `.remove()`. A single connected peer can populate the entry once via a `ConnectionRequest`, then send `ConnectionSync` messages at the rate-limiter ceiling (1/s) indefinitely, each triggering a fresh `runtime::spawn` of up to 24 long-lived TCP tasks. Over the 5-minute entry lifetime this accumulates thousands of concurrent tasks and open sockets, exhausting file descriptors and memory.

---

### Finding Description

**How `pending_delivered` is populated:**

In `ConnectionRequestProcess::respond_delivered()`, when the local node is the `to` target of a `ConnectionRequest`, it stores the attacker's listen addresses: [1](#0-0) 

A re-insertion guard exists, but it only prevents refreshing the entry within `HOLE_PUNCHING_INTERVAL` (2 minutes): [2](#0-1) 

The entry lives for `TIMEOUT` = 5 minutes before the periodic `notify()` cleanup removes it: [3](#0-2) 

**The non-consuming read in `ConnectionSyncProcess::execute()`:**

When a `ConnectionSync` arrives targeting the local node, the handler reads the entry with `.get()` — it is never removed: [4](#0-3) 

Each successful read immediately spawns a new async batch: [5](#0-4) 

**Contrast with the correct pattern in `ConnectionRequestDeliveredProcess::execute()`**, which consumes `inflight_requests` with `.remove()`: [6](#0-5) 

**Rate limiting does not bound total task count:**

The `forward_rate_limiter` is keyed by `(from, to, item_id)` at 1 request/second: [7](#0-6) 

`item_id` is the fixed message-type discriminant, not a per-message nonce, so the attacker is capped at 1 spawn/second — not 0 after the first.

**Each spawned task is long-lived:**

`try_nat_traversal` runs a retry loop for up to 30 seconds per task: [8](#0-7) 

`ADDRS_COUNT_LIMIT` = 24 tasks per spawn: [9](#0-8) 

---

### Impact Explanation

- 1 spawn/second × 24 tasks/spawn × 30-second task lifetime = **720 concurrent tasks and open `TcpSocket` file descriptors at steady state**
- Over the full 5-minute `pending_delivered` lifetime: **300 spawns × 24 = 7,200 tasks**
- Default Linux soft fd limit is 1,024; at 24 new sockets/second, exhaustion occurs in ~43 seconds
- After fd exhaustion, `create_socket()` returns `Err`, but already-spawned tasks continue holding their sockets; new connection attempts across the entire node (P2P, RPC) fail
- Memory exhaustion from thousands of stacked async task frames is a secondary crash vector

---

### Likelihood Explanation

Any unprivileged peer that can establish a P2P connection can execute this attack. The `ConnectionRequest` path requires no special privilege — the local node accepts it whenever `self_peer_id == &content.to`. The attacker controls the `listen_addrs` field and can supply 24 non-routable addresses (e.g., `10.0.0.x`) that will reliably hang for the full 30-second timeout. No PoW, no key material, no Sybil attack is required — a single connected peer suffices.

---

### Recommendation

In `ConnectionSyncProcess::execute()`, replace the `.get()` call with `.remove()` to consume the `pending_delivered` entry on first use, matching the pattern already used for `inflight_requests` in `ConnectionRequestDeliveredProcess::execute()`:

```rust
// connection_sync.rs, line 111
let listens_info = self
    .protocol
    .pending_delivered
    .remove(&content.from)   // was: .get(&content.from)
    .map(|info| info.0);
```

This enforces the invariant that a single hole-punching handshake produces at most one NAT traversal attempt.

### Proof of Concept

**State test sketch:**

1. Construct a `HolePunching` instance with a mock `NetworkState` whose `local_peer_id` = `NODE`.
2. Inject `pending_delivered[ATTACKER] = ([addr1..addr24], now)` directly.
3. Deliver N `ConnectionSync` messages with `from=ATTACKER`, `to=NODE`, empty route, passing the rate limiter (spaced ≥1 s apart or with a mocked clock).
4. Assert: `pending_delivered` still contains `ATTACKER` after each message (confirming no removal).
5. Assert: N independent `runtime::spawn` calls were made (via a spawn counter or mock executor).
6. Compare: repeat with `ConnectionRequestDelivered` — `inflight_requests` is empty after the first message.
