Audit Report

## Title
Off-by-One in `check_purge` Group Eviction Threshold Allows Peer Store Freeze via Crafted Discovery Messages — (File: `network/src/peer_store/peer_store_impl.rs`)

## Summary

The group-based eviction path in `check_purge` uses `> 4` at line 378 instead of `>= 4`, meaning groups of exactly 4 entries are never selected as eviction candidates. An attacker with minimal P2P access can fill all 16384 `ADDR_COUNT_LIMIT` slots with 4096 groups of exactly 4 addresses each, causing every subsequent `add_addr` call to return `Err(PeerStoreError::EvictionFailed)` and permanently freezing peer address discovery until the node is restarted.

## Finding Description

**Root cause — line 378:** The condition `if addrs.len() > 4` excludes groups of exactly 4 from eviction. The comment at line 338 documents the intent as "more than 4 peer," but the boundary is ambiguous; the per-subnet cap is 4, so groups at exactly 4 should be eligible. With 4096 groups each of size exactly 4, `flat_map` returns `None` for every group, producing an empty `candidate_peers` list in step 2.

**Step 1 (non-connectable eviction) is bypassed:** Fresh entries inserted by `add_addr` have `last_connected_at_ms = 0` and `attempts_count = 0`. In `is_connectable`, the first guard fires only when `attempts_count >= ADDR_MAX_RETRIES (3)` and `last_connected_at_ms == 0`; with `attempts_count = 0`, it does not fire. The second guard fires only when `attempts_count >= ADDR_MAX_FAILURES (10)`; again, it does not fire. All fresh entries return `true` from `is_connectable`, so step 1 removes nothing.

**`AddrManager::add` resets `attempts_count` on re-advertisement:** The overwrite condition is `addr_info.last_connected_at_ms >= exist_last_connected_at_ms`. Since both the new and existing entry have `last_connected_at_ms = 0`, the condition `0 >= 0` is true and the entry is overwritten with `attempts_count = 0`, preventing natural expiry through failed dial accumulation.

**`EvictionFailed` is silently swallowed:** In `add_new_addrs` in `discovery/mod.rs`, the error is logged only at `debug!` level with no rate-limiting, session penalty, or operator alert.

**Entry point — P2P Discovery protocol:** With `MAX_ADDR_TO_SEND = 1000` and `MAX_ADDRS = 3`, one non-announce `Nodes` message delivers up to 3000 addresses. Filling 16384 slots requires approximately 6 connections. The attacker does not need to own or control the advertised IPs; `is_valid_addr` only filters private/loopback addresses via `is_reachable`, so any public IP across 4096 distinct `/16` subnets passes.

**Group key is `/16` subnet:** `network_group.rs` confirms IPv4 addresses are grouped by `[bits[0], bits[1]]`, so 4096 distinct `/16` subnets yield 4096 distinct groups.

## Impact Explanation

Once the store is filled with 4096 groups of exactly 4 connectable addresses, every subsequent `add_addr` call triggers `check_purge`, which finds no eviction candidates in either step and returns `Err(PeerStoreError::EvictionFailed)`. The peer store is frozen: no legitimate peer addresses discovered via the network can be recorded. Peer discovery degrades to zero new entries until the node is restarted or the peer store file is manually deleted. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation

The attacker requires only one accepted inbound or outbound P2P connection per ~3000 addresses injected. No proof-of-work, no key material, and no privileged access are required. The crafted addresses are arbitrary public IPs the attacker does not need to control. Approximately 6 connections are sufficient to fill the store. The attack is repeatable and maintainable indefinitely by periodically re-advertising the same addresses to reset `attempts_count` via the `AddrManager::add` overwrite path (`0 >= 0`).

## Recommendation

Change the strict inequality to `>=` at line 378 of `network/src/peer_store/peer_store_impl.rs`:

```rust
// Before
if addrs.len() > 4 {
// After
if addrs.len() >= 4 {
```

This ensures any group at or above the per-subnet cap of 4 is eligible for eviction, closing the off-by-one gap and aligning the code with the documented intent at line 338.

## Proof of Concept

```rust
let mut peer_store = PeerStore::default();
// Fill store: 4096 subnets × 4 addresses = 16384 = ADDR_COUNT_LIMIT
for subnet in 0u16..4096 {
    for host in 1u8..=4 {
        let a = (subnet >> 8) as u8;
        let b = (subnet & 0xff) as u8;
        let addr: Multiaddr = format!(
            "/ip4/{}.{}.0.{}/tcp/8115/p2p/{}",
            a + 1, b, host, PeerId::random().to_base58()
        ).parse().unwrap();
        peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    }
}
// All groups have len == 4; check_purge step 2 produces no candidates
let new_addr: Multiaddr = format!(
    "/ip4/200.200.200.1/tcp/8115/p2p/{}", PeerId::random().to_base58()
).parse().unwrap();
// Returns Err(EvictionFailed) — peer store is frozen
assert!(peer_store.add_addr(new_addr, Flags::COMPATIBILITY).is_ok()); // FAILS
```
