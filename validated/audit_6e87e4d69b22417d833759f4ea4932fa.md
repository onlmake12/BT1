### Title
Unbounded `while` Loop in `constructProviderCommitment` Enables Out-of-Gas DOS on `reveal`/`revealWithCallback` - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol` contains an unbounded `while` loop in `constructProviderCommitment` that iterates `numHashes` times per call. When a provider's `maxNumHashes` is `0` (the default), no upper bound is enforced on `numHashes` at request time. An unprivileged user can inflate a provider's `sequenceNumber` by making many requests, causing `numHashes` for subsequent requests to grow until `reveal`/`revealWithCallback` permanently reverts with out-of-gas for any request carrying a large `numHashes`.

---

### Finding Description

`constructProviderCommitment` hashes a revelation value `numHashes` times in a plain `while` loop: [1](#0-0) 

`numHashes` is recorded in each request inside `requestHelper` as:

```
req.numHashes = assignedSequenceNumber − providerInfo.currentCommitmentSequenceNumber
``` [2](#0-1) 

The guard that rejects oversized `numHashes` is conditional on `maxNumHashes != 0`:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [3](#0-2) 

`maxNumHashes` is a provider-set field that defaults to `0` (zero-initialized storage). A provider that never calls `setMaxNumHashes` leaves this field at `0`, which **disables the guard entirely**. There is no protocol-level minimum or fallback cap.

`currentCommitmentSequenceNumber` only advances when the provider calls `advanceProviderCommitment` or when a successful `reveal`/`revealWithCallback` updates it via `revealHelper`. If the provider is slow to advance, the gap `assignedSequenceNumber − currentCommitmentSequenceNumber` grows with every new request. [4](#0-3) 

---

### Impact Explanation

When `reveal` or `revealWithCallback` is called for a request whose stored `numHashes` exceeds roughly 800 000 (≈ 30 M gas limit ÷ ~36 gas per `keccak256` round), the transaction reverts with out-of-gas. The request remains in storage but is permanently unrevealable:

- The user's paid fee is trapped in the contract and cannot be recovered.
- The Entropy service becomes permanently broken for all requests at or above the gas-exhausting `numHashes` threshold for that provider.
- The default provider (set at initialization) is the highest-impact target because the majority of users route through it. [5](#0-4) 

---

### Likelihood Explanation

- `maxNumHashes` defaults to `0` for every newly registered provider, including the protocol's own default provider. Any provider that does not explicitly call `setMaxNumHashes` is permanently exposed.
- An unprivileged attacker can call `requestV2()` (or `request()`) in a loop, paying only the provider fee per call, to inflate `providerInfo.sequenceNumber` without the provider advancing `currentCommitmentSequenceNumber`.
- Even without a deliberate attacker, organic high-volume usage of a provider that infrequently calls `advanceProviderCommitment` will naturally push `numHashes` into the danger zone over time.
- The attack is cheap relative to the damage: on a low-fee provider, a few hundred thousand fee payments suffice to permanently brick reveals for all subsequent users. [6](#0-5) 

---

### Recommendation

1. **Enforce a protocol-level cap on `numHashes` unconditionally**, independent of `maxNumHashes`:

```solidity
uint32 constant MAX_NUM_HASHES = 10_000; // tune to safe gas budget

req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber - providerInfo.currentCommitmentSequenceNumber
);
if (req.numHashes > MAX_NUM_HASHES) {
    revert EntropyErrors.LastRevealedTooOld();
}
// existing per-provider check can remain as a stricter override
if (providerInfo.maxNumHashes != 0 && req.numHashes > providerInfo.maxNumHashes) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

2. **Require providers to set `maxNumHashes` during `register`**, or default it to a safe non-zero value instead of `0`.

3. **Add an `unchecked` block and use `++i` / `--numHashes` in `constructProviderCommitment`** to reduce per-iteration gas cost as a secondary mitigation.

---

### Proof of Concept

1. Provider P registers with default `maxNumHashes = 0` and `chainLength = 10_000_000`.
2. Attacker calls `requestV2{value: fee}()` in a loop ~900 000 times against provider P, paying the minimum fee each time. `providerInfo.sequenceNumber` is now ~900 001; `currentCommitmentSequenceNumber` remains at its initial value (e.g., 1).
3. Victim V calls `requestV2{value: fee}()` — their request is assigned `sequenceNumber ≈ 900 001` and `req.numHashes ≈ 900 000`. No revert at request time because `maxNumHashes == 0`.
4. Provider P submits `revealWithCallback(P, 900001, userContribution, providerContribution)`.
5. Inside `revealHelper`, `constructProviderCommitment(900000, providerContribution)` loops 900 000 times → transaction reverts with out-of-gas.
6. Victim V's fee is permanently locked; their random number is never delivered. [1](#0-0) [7](#0-6)

        uint64 numHashes,
        bytes32 revelation
