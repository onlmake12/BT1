### Title
`ScriptHashTypeVerifier` Skips `hash_type` Validation for Output Type Scripts — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist only against the **lock script** of each transaction output. The **type script** of every output is never checked. An unprivileged RPC caller can submit a transaction whose output type script carries a not-yet-enabled `hash_type` (e.g., `Data3`, `Data4`, …), bypass the cheap non-contextual gate, and force the node into expensive contextual verification before the transaction is ultimately rejected deep inside the script executor.

---

### Finding Description

`ScriptHashTypeVerifier` is the dedicated non-contextual verifier that enforces which `ScriptHashType` values are currently permitted by consensus. It is composed into `NonContextualTransactionVerifier` and is the first line of defence against transactions that carry future or invalid hash types.

Its `verify()` implementation iterates over outputs and inspects only `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  L796-L814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

`output.type_()` is never touched. Every output cell can carry an optional type script (`CellOutput` has both `lock` and `type_` fields), and the type script has its own independent `hash_type` byte.

The lower-level structural check (`check_data` in `util/gen-types/src/extension/check_data.rs`) does cover both scripts:

```rust
// util/gen-types/src/extension/check_data.rs  L24-L28
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()
    }
}
```

But `check_data` only validates the **bit-pattern** of the byte (even vs. odd), not whether the value is within the **consensus-enabled** set. `ENABLED_SCRIPT_HASH_TYPE` is a stricter allowlist that gates future VM versions. A value such as `Data3 = 6` is structurally valid (even, non-zero) and passes `check_data`, but is not yet enabled by consensus and should be rejected by `ScriptHashTypeVerifier`. Because `ScriptHashTypeVerifier` never reads `output.type_().hash_type()`, a type script with `Data3` silently passes the non-contextual gate.

The gap is directly analogous to the reported Lombard finding: certain operations (here, type-script hash-type validation) skip the role/type check that other operations (lock-script hash-type validation) correctly perform.

---

### Impact Explanation

A transaction that passes `NonContextualTransactionVerifier` is admitted to the tx-pool and queued for contextual verification, which includes full script execution via the CKB-VM. The script executor's `select_version` will eventually reject the unenabled hash type:

```rust
// script/src/types.rs  L930-L935
hash_type => {
    return Err(ScriptError::InvalidScriptHashType(format!(
        "The ScriptHashType/{:?} has not been activated, and is not permitted for use.",
        hash_type
    )));
}
```

But by that point the node has already:
- Resolved all cell deps and inputs from the database
- Allocated scheduler and VM state
- Consumed CPU and memory for contextual verification

An attacker can flood the node with such transactions at the rate the RPC accepts them, causing repeated expensive contextual-verification cycles that all terminate in a cheap error, degrading throughput for legitimate transactions. Because the non-contextual check is the intended cheap rejection point, its incompleteness undermines the defence-in-depth design.

---

### Likelihood Explanation

The attack requires only an RPC call to `send_transaction` with a crafted transaction. No privileged access, no key material, no majority hashpower, and no Sybil capability is needed. The `ScriptHashType` enum already defines `Data3` through `Data127` as valid Rust variants; constructing a transaction with such a type script is trivial using the public Molecule serialization API. The gap exists on mainnet today for any `DataN` value with N ≥ 3.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of the type script of every output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        check_hash_type(output.lock().hash_type())?;
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the existing `TryInto` + `ENABLED_SCRIPT_HASH_TYPE` logic. This keeps the rejection at the cheap, non-contextual stage, consistent with the verifier's stated purpose.

---

### Proof of Concept

1. Build a transaction with one output whose **type script** has `hash_type = ScriptHashType::Data3` (byte value `6`). The lock script uses the standard `Data` hash type.
2. Submit via `send_transaction` RPC.
3. Observe that `NonContextualTransactionVerifier` (including `ScriptHashTypeVerifier`) returns `Ok(())` — the transaction is admitted to the pool.
4. The transaction is later rejected during contextual verification with `ScriptError::InvalidScriptHashType`, but only after full cell resolution and VM initialisation.
5. Repeat in a tight loop to demonstrate resource exhaustion on the verifier thread pool.
