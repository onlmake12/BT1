### Title
Fee Deposited Before Freshness Check Enables Fee-for-Nothing via Front-Running — (`target_chains/sui/contracts/sources/pyth.move`)

---

### Summary

In `update_single_price_feed`, the caller's fee is deposited into the `PriceInfoObject` unconditionally before the freshness check occurs. An attacker who front-runs a victim with a newer accumulator message causes the victim's fee to be permanently locked in the object while the price feed is not updated, violating the invariant that a fee payment must yield a price feed update.

---

### Finding Description

The execution order in `update_single_price_feed` is:

1. **Fee sufficiency check** (line 274)
2. **`deposit_fee_coins`** — fee is irrevocably transferred into the `PriceInfoObject` (line 277)
3. **`update_cache`** is called (line 287), which internally calls `is_fresh_update` (line 316)
4. If `is_fresh_update` returns `false`, the price is silently skipped — but the fee is already gone [1](#0-0) 

Inside `update_cache`, the freshness gate is: [2](#0-1) 

`is_fresh_update` returns `true` only when `update_timestamp > cached_timestamp`: [3](#0-2) 

`deposit_fee_coins` simply accumulates coins into a dynamic object field with no withdrawal path: [4](#0-3) 

A grep across all Sui contract sources confirms there is **no** `withdraw`, `refund`, or `reclaim` function anywhere in the module. Fees deposited into `FEE_STORAGE_KEY` are permanently locked. [5](#0-4) 

---

### Impact Explanation

A victim who calls `update_single_price_feed` with a hot potato carrying timestamp `T` loses their entire SUI fee if the `PriceInfoObject` was already updated to timestamp `T+1` (by an attacker or any concurrent updater). The price feed is unchanged from the victim's perspective, and the fee is permanently locked in the object. There is no code path to recover it.

---

### Likelihood Explanation

The attack requires only a publicly available, more recent accumulator message for the same price identifier — Pyth publishes these continuously. On Sui, shared-object transactions are sequenced by validators; an attacker who submits a transaction touching the same `PriceInfoObject` with a higher-timestamp update can reliably be ordered before the victim. Even without deliberate front-running, two independent users submitting updates for the same feed in the same epoch will trigger this: the one whose update arrives second (with the older timestamp) loses their fee silently.

---

### Recommendation

Move `deposit_fee_coins` to **after** the freshness check, so the fee is only collected when the price feed is actually updated:

```move
// Inside update_cache or update_single_price_feed:
if (is_fresh_update(update, price_info_object)) {
    price_info::deposit_fee_coins(price_info_object, fee);  // collect fee only on actual update
    price_info::update_price_info_object(price_info_object, update);
    pyth_event::emit_price_feed_update(...);
} else {
    // return fee to caller or abort
    transfer::public_transfer(fee, tx_context::sender(ctx));
}
```

Alternatively, abort the entire transaction when the update is stale, so the fee transfer never settles.
