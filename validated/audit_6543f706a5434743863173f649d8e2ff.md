Audit Report

## Title
Unbounded `last_n_numbers` allocation via `difficulty_boundary=U256::zero()` bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
An unprivileged light-client peer can send a crafted `GetLastStateProof` message with `difficulty_boundary=U256::zero()`, `last_n_blocks=499`, `start_number=0`, and a valid tip hash. The upfront limit check passes (998 ≤ 1000), but `difficulty_boundary=0` causes `get_first_block_total_difficulty_is_not_less_than` to immediately return block 0 as the boundary, setting `difficulty_boundary_block_number=0`. The subsequent adjustment guard does not fire, so `last_n_numbers = (0..last_block_number)` — a Vec proportional to the full chain height — is allocated and passed to `complete_headers` for N sequential DB lookups and MMR root computations, entirely bypassing `GET_LAST_STATE_PROOF_LIMIT`.

## Finding Description
**Root cause:** The limit check at L201–205 only validates input fields (`difficulties.len()` and `last_n_blocks`), but the *derived* quantity `last_n_numbers.len()` — which equals `last_block_number - difficulty_boundary_block_number` — is never bounded.

**Exact code path:**

1. **Limit check passes** (L201–205): `0 + 499×2 = 998 ≤ 1000`. [1](#0-0) 

2. **Empty-difficulties validation passes** (L259–263): `difficulties.last()` returns `None`, so `unwrap_or(false)` = `false`. [2](#0-1) 

3. **Outer branch taken** (L291–292): `last_block_number - 0 = N > 499`, so the else branch executes. [3](#0-2) 

4. **`get_first_block_total_difficulty_is_not_less_than(0, N, &U256::zero())` short-circuits** (L30–33): genesis total difficulty ≥ 0 is always true, so it immediately returns `Some((0, genesis_diff))`, setting `difficulty_boundary_block_number = 0`. [4](#0-3) 

5. **Adjustment guard does not fire** (L313–316): `N - 0 = N ≥ 499`, so the condition `< last_n_blocks` is false. [5](#0-4) 

6. **Unbounded `last_n_numbers` allocated** (L318–319): `(0..N).collect()` — N entries with no cap. [6](#0-5) 

7. **`difficulty_boundary_block_number > 0` is false** (L321, L345–346): execution falls to `(Vec::new(), last_n_numbers)`, returning the unbounded vec directly. [7](#0-6) 

8. **`complete_headers` called with all N block numbers** (L356–366): N `get_ancestor` + `get_block` + `chain_root_mmr` DB lookups execute synchronously. [8](#0-7) 

`GET_LAST_STATE_PROOF_LIMIT = 1000` is entirely bypassed. [9](#0-8) 

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

For a chain of height N:
- `last_n_numbers` allocates a `Vec<u64>` of N entries (8N bytes).
- `complete_headers` performs N `get_ancestor` + `get_block` + `chain_root_mmr` calls — all synchronous DB/MMR operations on the server's snapshot.
- The response (`SendLastStateProof`) contains N `VerifiableHeader` structs, potentially hundreds of MB of serialized data.

On a chain of 1M blocks, the DB I/O and response serialization dominate. The server thread handling this peer is blocked for the duration. Repeated requests from multiple peers can exhaust memory and I/O capacity, causing OOM or extreme latency — crashing the light-client protocol server component of the CKB node.

## Likelihood Explanation
The attack requires only a valid tip hash (publicly observable on the P2P network) and a single crafted `GetLastStateProof` message. No proof-of-work, no key, no privileged role is needed. Any peer connected to the light-client protocol port can trigger it. The attack is trivially repeatable and can be parallelized across multiple connections.

## Recommendation
After computing `difficulty_boundary_block_number`, enforce that `last_block_number - difficulty_boundary_block_number <= last_n_blocks` (or `<= GET_LAST_STATE_PROOF_LIMIT`). If this invariant is violated, return `StatusCode::InvalidRequest`. Additionally, consider explicitly rejecting `difficulty_boundary=U256::zero()`, since every block satisfies it and it serves no legitimate client purpose.

## Proof of Concept
1. Mine a chain of 100,000 blocks on a local devnet with the light-client protocol enabled.
2. Obtain the tip hash `tip_hash` and genesis hash `genesis_hash`.
3. Send:
   ```
   GetLastStateProof {
     last_hash: tip_hash,
     start_hash: genesis_hash,
     start_number: 0,
     last_n_blocks: 499,
     difficulty_boundary: U256::zero(),
     difficulties: [],
   }
   ```
4. Observe that `last_n_numbers.len() == 100_000` (not ≤ 1000), the server performs 100,000 DB lookups, and the response is hundreds of MB.
5. Assert `last_n_numbers.len() <= 1000` — this assertion fails, confirming the invariant violation.
