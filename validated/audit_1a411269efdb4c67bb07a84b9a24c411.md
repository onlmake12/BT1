### Title
`verifyUpdate()` Reverts for Contract Callers That Cannot Receive ETH Refund — (`File: lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.verifyUpdate()` uses `payable(msg.sender).transfer(...)` to refund excess ETH to the caller. Because `.transfer()` forwards only 2300 gas and reverts on failure, any contract caller that sends `msg.value > verification_fee` but lacks a `receive()` / `fallback()` function will have the entire `verifyUpdate()` call revert. This permanently blocks those contracts from using Pyth Lazer price verification.

### Finding Description

In `PythLazer.verifyUpdate()`, after checking that the caller paid at least `verification_fee`, the contract attempts to push the excess ETH back to `msg.sender`:

```solidity
// lazer/contracts/evm/src/PythLazer.sol, lines 73-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
```

`address.transfer()` is a Solidity built-in that:
- Forwards exactly **2300 gas** to the recipient
- **Reverts the entire calling transaction** if the transfer fails

If `msg.sender` is a contract without a `receive()` or `fallback()` function (or one whose `receive()` consumes more than 2300 gas), the `.transfer()` call reverts, causing `verifyUpdate()` to revert entirely — even though the price update data and signature are valid.

The `verification_fee` is initialized to `1 wei` and there is no setter function in the current contract version, so any caller sending `2 wei` or more triggers the refund path. Contracts that integrate Lazer price feeds commonly send a small buffer above the minimum fee to guard against fee changes, making this a realistic trigger condition.

### Impact Explanation

Any contract integrating `PythLazer.verifyUpdate()` that:
1. Sends `msg.value > verification_fee` (e.g., sends a fixed amount like `0.001 ether` to ensure coverage), **and**
2. Does not implement a `receive()` or `fallback()` function

...will have **every call to `verifyUpdate()` revert**. This is a complete denial-of-service on Pyth Lazer price verification for that contract. Since `verifyUpdate()` is the sole entry point for consuming Lazer price data on-chain, affected integrators lose all access to Lazer feeds with no workaround short of redeploying their contract.

### Likelihood Explanation

- Many on-chain contracts (e.g., pure logic contracts, proxy implementations, contracts that only send ETH intentionally) do not implement `receive()`.
- The `verification_fee` is 1 wei — an extremely small value. Contracts that hardcode a fee amount (e.g., `0.001 ether`) or use `msg.value` forwarded from a user will almost always send more than 1 wei, triggering the refund branch.
- The issue is triggered by normal, unprivileged usage of the public `verifyUpdate()` function — no special role or key is required.

### Recommendation

Replace `.transfer()` with a low-level `.call{value: ...}("")` that does not revert on failure, or adopt a pull-payment pattern where excess ETH is credited to the caller's balance for later withdrawal:

```solidity
// Option A: use .call (does not revert on failure)
if (msg.value > verification_fee) {
    (bool sent, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(sent, "Refund failed");
}

// Option B: pull-payment — store excess, let caller withdraw
// pendingRefunds[msg.sender] += msg.value - verification_fee;
```

Option B is the closest analog to the recommended fix in the original report and eliminates the push-payment DoS entirely.

### Proof of Concept

1. Deploy a contract `LazerConsumer` that calls `pythLazer.verifyUpdate{value: 0.001 ether}(update)` but has **no** `receive()` function.
2. Call `LazerConsumer.consumePrice()`.
3. The call to `verifyUpdate` reaches line 76, attempts `payable(msg.sender).transfer(0.001 ether - 1)`, which reverts because `LazerConsumer` cannot receive ETH.
4. The entire transaction reverts — the contract can never consume a Lazer price update.
