Audit Report

## Title
Out-of-Bounds Index Panic in `remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary

`TxConfirmStat::remove_unconfirmed_tx` guards the `block_unconfirmed_txs` access for `tx_age >= len`, but performs an unguarded index into `confirm_blocks_to_failed_txs[tx_age - 1]` immediately after. When a tracked transaction has been unconfirmed for more than `MAX_CONFIRM_BLOCKS` (1000) blocks, `tx_age - 1` exceeds the vec's last valid index (999), causing a Rust index-out-of-bounds panic that crashes the tx-pool service thread and the node.

## Finding Description

`confirm_blocks_to_failed_txs` is initialized with length `MAX_CONFIRM_BLOCKS = 1000`, giving valid indices `0..=999`. In `remove_unconfirmed_tx`, `tx_age` is computed as `tip_height.saturating_sub(entry_height) as usize`. The guard at line 208 (`if tx_age >= self.block_unconfirmed_txs.len()`) correctly handles the `block_unconfirmed_txs` access for large `tx_age`, but the subsequent `count_failure` branch at lines 214–216 has no corresponding bounds check:

```rust
if count_failure {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

When `tx_age = 1001`, `tx_age - 1 = 1000` is an out-of-bounds index on a `Vec` of length 1000, causing an unconditional panic.

The full call chain: `remove_expired` / `limit_size` (called on every block update) → `callbacks.call_reject` → `fee_estimator.reject_tx` → `drop_tx` (always passes `count_failure = true`) → `drop_tx_inner` → `remove_unconfirmed_tx`. A tx is tracked at `entry_height == best_height`; `tx_age` grows as `best_height` advances. After 1001 blocks, any eviction of that tx triggers the panic.

## Impact Explanation

A Rust index-out-of-bounds panic is unrecoverable and terminates the thread. Since this occurs inside the tx-pool service thread (holding the fee estimator write lock), it crashes the CKB node. This matches the **High (10001–15000 points)** impact: *Vulnerabilities which could easily crash a CKB node*. The crash is deterministic and unconditional once triggered.

## Likelihood Explanation

An unprivileged attacker submits a valid low-fee-rate transaction via P2P or RPC. The tx gets tracked by the fee estimator. After 1001 blocks (~2.2 hours at 8-second block time), the tx is still in the pool (default expiry is 12 hours). The next block update triggers `remove_expired` or `limit_size`, which evicts the tx and calls `reject_tx`, triggering the panic. No special privileges, no victim interaction, and no external dependencies are required. The attack is repeatable after a node restart.

## Recommendation

Add a bounds check before the `confirm_blocks_to_failed_txs` access, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

## Proof of Concept

```rust
#[test]
fn test_no_panic_on_old_drop() {
    let mut stat = TxConfirmStat::default();
    // MAX_CONFIRM_BLOCKS = 1000, so tx_age = 1001 triggers the bug
    let entry_height: u64 = 0;
    let tip_height: u64 = 1001;
    let bucket_index: usize = 0;
    // This must not panic:
    stat.remove_unconfirmed_tx(entry_height, tip_height, bucket_index, true);
}
```

With unpatched code, `tx_age = 1001`, `confirm_blocks_to_failed_txs[1000]` on a `Vec` of length 1000 → index-out-of-bounds panic. With the fix, the branch is skipped and the test passes.
