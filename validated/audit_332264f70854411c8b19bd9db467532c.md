Audit Report

## Title
Missing Per-Peer Rate Limiter in `BlockFilter` Enables Head-of-Line Blocking DoS — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_check_points_process.rs`)

## Summary
`BlockFilter` processes all incoming messages sequentially with no per-peer rate limiting. A single unprivileged peer can flood `GetBlockFilterCheckPoints` messages, each triggering up to 4,000 synchronous DB reads per message, monopolizing the handler and starving all other light-client peers of responses. The root cause is a concrete, verifiable absence of a rate limiter that is present in `Relayer` and `HolePunching`.

## Finding Description
**Sequential dispatch:** `CKBHandler::received` in `network/src/protocols/mod.rs` (L365–384) calls `self.handler.received(...).await`, and Tentacle's `ServiceProtocol` contract means all messages for a given protocol instance are dispatched one at a time through the same async context.

**Full await before next message:** `BlockFilter::received` at `sync/src/filter/mod.rs` L152 calls `self.process(nc, peer_index, msg).await` and does not return until the call completes. No other peer's message can begin processing until the current one finishes.

**Per-message cost:** `GetBlockFilterCheckPointsProcess::execute` in `sync/src/filter/get_block_filter_check_points_process.rs` runs a loop of up to `BATCH_SIZE = 2000` iterations (L9, L43), each performing two synchronous DB reads — `get_block_hash` followed by `get_block_filter_hash` (L44–46) — with no `.await` yield point inside the loop. This totals up to 4,000 blocking DB reads per message.

**No rate limiter in `BlockFilter`:** The struct at `sync/src/filter/mod.rs` L22–25 contains only `shared: Arc<SyncShared>` — no `rate_limiter` field, no per-peer quota check anywhere in `try_process`.

**Contrast with `Relayer`:** `sync/src/relayer/mod.rs` L81 declares `rate_limiter: RateLimiter<(PeerIndex, u32)>`, checked at L116–123 before any dispatch, enforcing 30 req/sec per peer per message type.

**Contrast with `HolePunching`:** `network/src/protocols/hole_punching/mod.rs` L45–46 declares both `rate_limiter` and `forward_rate_limiter`, checked at L95–107 before any processing.

Exploit path: attacker opens one TCP connection on `SupportProtocols::Filter`, sends a continuous stream of `GetBlockFilterCheckPoints` messages with `start_number = 0`. Each message occupies the handler for the duration of up to 4,000 synchronous DB reads. All other peers' messages — including `GetBlockFilterHashes` and `GetBlockFilters` from legitimate light clients — queue behind the flood and are delayed proportionally.

## Impact Explanation
**High severity** — matches: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* The Filter protocol is the primary data-serving infrastructure for CKB light clients. A single attacker peer with one TCP connection can continuously saturate the handler, rendering the filter service non-functional for all connected light clients on the targeted node.

## Likelihood Explanation
The attack requires only a single TCP connection to a CKB node with the Filter protocol enabled. No special privileges, keys, or hashpower are needed. The `start_number` field is attacker-controlled and unchecked. The absence of rate limiting is a concrete, verifiable code gap. The attack is continuously repeatable and self-sustaining.

## Recommendation
Add a per-peer, per-message-type rate limiter to `BlockFilter`, mirroring the pattern in `Relayer`:

```rust
pub struct BlockFilter {
    shared: Arc<SyncShared>,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

Check the limiter at the top of `try_process` before dispatching to any `*Process::execute`. A quota of 5–10 `GetBlockFilterCheckPoints` requests per second per peer is sufficient for legitimate light-client use, as checkpoint data changes only with new blocks.

## Proof of Concept
1. Start a CKB node with the Filter protocol enabled and a populated block filter DB.
2. Connect two mock peers (peer A and peer B) to the node on `SupportProtocols::Filter`.
3. Have peer A send `GetBlockFilterCheckPoints` messages in a tight loop with `start_number = 0`.
4. Have peer B send `GetBlockFilterHashes` messages at a steady rate (e.g., 1/sec).
5. Record peer B's response latency with peer A idle vs. peer A flooding.
6. Assert that peer B's latency increases proportionally to peer A's flood rate, confirming head-of-line blocking.

The root cause is confirmed at `sync/src/filter/mod.rs` L22–25 (no rate limiter field), `sync/src/filter/get_block_filter_check_points_process.rs` L9–10 and L43–56 (2000-iteration loop with two DB reads per iteration, no yield), and `sync/src/filter/mod.rs` L152 (full await before next message).
