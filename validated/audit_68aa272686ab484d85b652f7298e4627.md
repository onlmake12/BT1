Audit Report

## Title
Unbounded TCP Connection Acceptance in `start_tcp_server` Enables FD Exhaustion DoS — (File: `rpc/src/server.rs`)

## Summary
The `start_tcp_server` function accepts TCP connections in an unbounded loop with no connection cap, no idle timeout, and a `while let Ok(...)` pattern that permanently exits on any accept error including `EMFILE`. A local attacker can open enough idle connections to exhaust the process file-descriptor limit, causing the TCP RPC accept loop to exit permanently and refuse all further connections until the process is restarted.

## Finding Description
In `rpc/src/server.rs`, `start_tcp_server` (lines 156–202) contains three compounding defects:

**1. No connection limit.** The accept loop at line 176 calls `tokio::spawn` unconditionally for every accepted connection. There is no semaphore, counter, or backpressure mechanism anywhere in `start_tcp_server`. The `stream_config` values `with_channel_size(4)` and `with_pipeline_size(4)` (lines 166–171) are per-connection pipeline limits, not a global connection cap.

**2. No idle timeout.** Each spawned task calls `serve_stream_sink(&rpc, w, r, stream_config).await` (line 190) with no `tokio::time::timeout` wrapper. A client that connects and sends nothing holds the task and its file descriptor indefinitely. Contrast with the HTTP/WS path, which applies `TimeoutLayer::with_status_code(..., Duration::from_secs(30))` (lines 125–128).

**3. Accept loop exits permanently on any error.** `while let Ok((stream, _)) = listener.accept().await` (line 176) breaks out of the loop on any `Err` variant, including the transient `EMFILE` (too many open files). Once the FD limit is reached, `listener.accept()` returns `Err(EMFILE)`, the loop exits, and the TCP RPC server stops accepting connections entirely. The `tokio::select!` block (lines 174–199) then falls through with no restart logic.

**Exploit flow:**
1. Attacker (local process) opens idle TCP connections to `127.0.0.1:18114` in a loop.
2. Each connection is accepted and spawned as a task holding an open FD.
3. At ~1024 connections (default Linux `ulimit -n`), `accept()` returns `Err(EMFILE)`.
4. `while let Ok(...)` exits; the TCP RPC accept loop terminates permanently.
5. All subsequent legitimate TCP RPC calls receive connection refused until process restart.

## Impact Explanation
The TCP RPC accept loop is permanently halted without a process restart. This matches the in-scope bounty impact: **"Any local RPC API crash" (0–500 points)**. The impact is limited to the TCP RPC transport; the HTTP/WS RPC path is unaffected.

## Likelihood Explanation
- TCP RPC is opt-in but a documented production feature; when enabled it binds to `127.0.0.1:18114` with no authentication.
- Any local process can reach `127.0.0.1:18114` without privileges.
- The default Linux per-process FD limit is 1024 (`ulimit -n`); opening 1024 idle TCP connections requires only a trivial script.
- The attack is repeatable: once the loop exits, it stays exited until process restart.
- No authentication, rate limiting, or connection cap stands in the way.

## Recommendation
1. Add a global connection semaphore before spawning:
   ```rust
   let sem = Arc::new(tokio::sync::Semaphore::new(MAX_TCP_RPC_CONNECTIONS));
   // acquire permit before tokio::spawn; drop permit when task ends
   ```
2. Wrap `serve_stream_sink` with `tokio::time::timeout` to close idle connections, mirroring the HTTP server's `TimeoutLayer`.
3. Change the accept loop to handle transient errors (e.g., `EMFILE`) with a brief sleep and retry rather than breaking permanently:
   ```rust
   loop {
       match listener.accept().await {
           Ok((stream, _)) => { /* spawn */ }
           Err(e) if is_transient(&e) => {
               tokio::time::sleep(Duration::from_millis(100)).await;
           }
           Err(_) => break,
       }
   }
   ```
