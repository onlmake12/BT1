### Title
Division-by-Zero in `calculateTwap` When Start and End Messages Share the Same `publishSlot` — (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary

`validateTwapPriceInfo` in `Pyth.sol` uses a non-strict `>` check for slot ordering, allowing `startSlot == endSlot`. When `calculateTwap` is subsequently called, `slotDiff` is zero, causing a Solidity `Panic(0x12)` division-by-zero revert. Any unprivileged caller who submits two valid TWAP messages sharing the same `publishSlot` can permanently DoS `parseTwapPriceFeedUpdates` for that window.

### Finding Description

`validateTwapPriceInfo` rejects only the case where the start slot is strictly greater than the end slot:

```solidity
// Pyth.sol line 604
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

Equal slots (`startSlot == endSlot`) pass this check. [1](#0-0) 

`calculateTwap` then computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // Panic: division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // Panic: division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // Panic: division by zero
``` [2](#0-1) 

The Solana receiver correctly uses a strict `>` guard that prevents this:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The EVM contract is missing this equivalent strict check.

### Impact Explanation

Any call to `parseTwapPriceFeedUpdates` with a start/end pair whose `publishSlot` values are equal will revert with a Solidity panic. Consumers relying on this function to obtain TWAP prices for a specific window cannot do so. Because the revert is deterministic for that input, the window is permanently inaccessible on-chain via the EVM contract. [4](#0-3) 

### Likelihood Explanation

Pythnet can legitimately publish multiple TWAP messages within the same slot (e.g., when multiple price updates arrive in the same slot). A user who obtains two such valid Wormhole-signed TWAP messages for the same feed and same `publishSlot` — both of which individually pass the `prevPublishTime < publishTime` uniqueness check — can trigger this revert. No privileged access or key compromise is required; the attacker only needs to call `parseTwapPriceFeedUpdates` with valid on-chain data. [5](#0-4) 

### Recommendation

Change the slot ordering check in `validateTwapPriceInfo` from a non-strict to a strict inequality, mirroring the Solana implementation:

```solidity
// Before (allows equal slots → slotDiff == 0 → division by zero)
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After (rejects equal slots)
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
``` [1](#0-0) 

### Proof of Concept

1. Obtain two valid Wormhole-signed TWAP accumulator messages for the same feed where both have `publishSlot = S`, `prevPublishTime < publishTime` (individually valid), and `publishTime_start <= publishTime_end`.
2. Call `parseTwapPriceFeedUpdates([startUpdate, endUpdate], [priceId])` with sufficient fee.
3. `validateTwapPriceInfo` passes: `startSlot > endSlot` is `false` (they are equal), `startTime > endTime` is `false`.
4. `calculateTwap` executes `slotDiff = S - S = 0`, then `priceDiff / 0` triggers `Panic(0x12)`.
5. The transaction reverts; the TWAP for this window is inaccessible. [6](#0-5)

    function validateTwapPriceInfo(
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure {
        // First validate each individual price's uniqueness
        if (
            twapPriceInfoStart.prevPublishTime >= twapPriceInfoStart.publishTime
        ) {
            revert PythErrors.InvalidTwapUpdateData();
        }
        if (twapPriceInfoEnd.prevPublishTime >= twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateData();
        }

        // Then validate the relationship between the two data points
        if (twapPriceInfoStart.expo != twapPriceInfoEnd.expo) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishTime > twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
    }
