### Title
`HeaderAcceptor::accept()` Skips `BLOCK_INVALID` Status Check, Allowing Previously-Invalidated Headers to Be Re-Accepted as Valid — (`sync/src/synchronizer/headers_process.rs`)

---

### Summary

In `HeaderAcceptor::accept()`, the code checks whether a header's status contains `HEADER_VALID` to short-circuit processing, but never checks whether the status is `BLOCK_INVALID`. Because `BLOCK_INVALID` (bit 12) does not share any bits with `HEADER_VALID` (bit 0), a header previously marked invalid falls through all structural checks and, if it passes them, is re-inserted as a valid header via `insert_valid_header`. The developer acknowledged this with an explicit `FIXME` comment but left it unresolved. This is the direct CKB analog of the external report: an invalid state is treated as valid because the validity check uses an insufficient proxy indicator rather than an explicit invalidity flag.

---

### Finding Description

**Root cause — `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::accept()`, lines 301–356:**

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer's best known header and return early
    return result;
}
// Falls through to prev_block_check, non_contextual_check, version_check
// ...
sync_shared.insert_valid_header(self.peer, self.header);
```

The `BlockStatus` flags are defined in `shared/src/block_status.rs`:

```rust
const HEADER_VALID  =     1;
const BLOCK_INVALID =     1 << 12;   // = 4096
```

`BLOCK_INVALID` (4096) does not contain the `HEADER_VALID` bit (1), so `status.contains(BlockStatus::HEADER_VALID)` evaluates to `false` for any header with `BLOCK_INVALID` status. The code then proceeds to run three structural checks:

1. **`prev_block_check`** — only rejects if the *parent* is `BLOCK_INVALID`. A header marked invalid for any other reason (e.g., its block body was invalid, it was part of a rejected chain, or its parent's invalid status was evicted from the in-memory map) passes this check.
2. **`non_contextual_check`** — runs `HeaderVerifier` (PoW, timestamp, epoch). A structurally valid header passes.
3. **`version_check`** — passes if `version == 0`.

If all three pass, `sync_shared.insert_valid_header(self.peer, self.header)` is called, inserting the header into the `header_map` and calling `may_set_best_known_header` for the peer. The returned `ValidationResult` has `state = ValidationState::Valid` (the default), so `HeadersProcess::execute()` treats the header as valid and continues processing subsequent headers in the batch.

**Contrast with the external report:** In Perennial v2, `KeeperOracle.at()` checks `valid = (price != 0)` — an indirect proxy — instead of an explicit validity flag, so expired versions with a non-zero price are treated as valid. In CKB, `HeaderAcceptor::accept()` checks `status.contains(HEADER_VALID)` — an indirect proxy — instead of explicitly checking `status == BLOCK_INVALID`, so previously-invalidated headers are treated as valid.

**Attacker-controlled entry path:**

Any peer can send a `SendHeaders` P2P message (protocol `Sync`). The message is deserialized and dispatched to `HeadersProcess::execute()`, which calls `HeaderAcceptor::accept()` for each header in the batch. No privilege is required.

---

### Impact Explanation

1. **Sync-state corruption:** The peer's best known header is updated to a previously-invalidated header via `may_set_best_known_header`. If this header has higher total difficulty than the current tip, the synchronizer may prioritize downloading blocks from this peer, following an invalid chain tip.

2. **Resource exhaustion:** The node may issue `GetBlocks` requests for blocks on the invalid chain, consuming CPU, memory, and bandwidth to download and re-verify blocks that will ultimately be rejected by the block verifier.

3. **IBD disruption:** During Initial Block Download, the node selects sync peers based on their best known header. Corrupting this value with an invalid header can stall or misdirect IBD.

4. **Cascading re-acceptance:** Because `execute()` processes a batch of up to `MAX_HEADERS_LEN` (2000) headers and treats each `Valid` result as a signal to continue, an attacker can re-accept an entire chain of previously-invalidated headers in a single message.

---

### Likelihood Explanation

- Any unprivileged peer can send `SendHeaders` messages at will.
- The attacker only needs to know the hash of a header that was previously marked `BLOCK_INVALID` on the target node (e.g., from a prior failed sync attempt, a known-invalid chain, or a block whose body was rejected).
- The `block_status_map` is an in-memory `DashMap`; entries are evicted on node restart, meaning a header invalidated in a prior session has no persistent `BLOCK_INVALID` record and is even more easily re-accepted.
- No cryptographic capability is required; the attacker simply replays a structurally valid header.

---

### Recommendation

In `HeaderAcceptor::accept()`, add an explicit early-return for `BLOCK_INVALID` status immediately after the status is read, resolving the existing `FIXME`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early-return path
}
```

### Proof of Concept

1. Node A receives and rejects a block `B` (e.g., invalid transaction). `B`'s header hash is stored in `block_status_map` as `BLOCK_INVALID`.
2. Attacker peer connects to Node A and sends a `SendHeaders` message containing `B`'s header (structurally valid: correct PoW, timestamp > median, version 0, parent is a valid block).
3. `HeadersProcess::execute()` → `HeaderAcceptor::accept()`:
   - `get_block_status(B.hash)` → `BLOCK_INVALID` (4096).
   - `status.contains(HEADER_VALID)` → `false` (4096 & 1 = 0). No early return.
   - `prev_block_check`: parent of `B` is valid → passes.
   - `non_contextual_check`: `B`'s header has valid PoW/timestamp → passes.
   - `version_check`: version == 0 → passes.
   - `insert_valid_header` is called: `B` enters `header_map`; peer's best known header is set to `B`.
4. `ValidationResult.state == Valid` → `execute()` continues, accepting subsequent headers chained from `B`.
5. Node A's synchronizer now believes the attacker peer has a valid chain tip at `B`, and may request blocks from this peer, wasting resources and corrupting sync state. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn accept(&self) -> ValidationResult {
        let mut result = ValidationResult::default();
        let sync_shared = self.active_chain.sync_shared();
        let state = self.active_chain.state();
        let shared = sync_shared.shared();

        // FIXME If status == BLOCK_INVALID then return early. But which error
        // type should we return?
        let status = self.active_chain.get_block_status(&self.header.hash());
        if status.contains(BlockStatus::HEADER_VALID) {
            let header_index = sync_shared
                .get_header_index_view(
                    &self.header.hash(),
                    status.contains(BlockStatus::BLOCK_STORED),
                )
                .unwrap_or_else(|| {
                    panic!(
                        "header {}-{} with HEADER_VALID should exist",
                        self.header.number(),
                        self.header.hash()
                    )
                })
                .as_header_index();
            state
                .peers()
                .may_set_best_known_header(self.peer, header_index);
            return result;
        }

        if self.prev_block_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-parent header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        if let Some(is_invalid) = self.non_contextual_check(&mut result).err() {
            debug!(
                "HeadersProcess rejected non-contextual header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            if is_invalid {
                shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            }
            return result;
        }

        if self.version_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-version header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        sync_shared.insert_valid_header(self.peer, self.header);
        result
    }
