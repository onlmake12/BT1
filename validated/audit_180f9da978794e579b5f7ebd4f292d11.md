### Title
Unauthenticated `remove_transaction` / `clear_tx_pool` RPC Methods Allow Any Caller to Evict Arbitrary Pending Transactions — (`rpc/src/module/pool.rs`)

---

### Summary

The Pool RPC module exposes `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` as fully public, unauthenticated JSON-RPC methods. Any caller who can reach the RPC endpoint — including via a CSRF-style cross-origin request from a malicious web page, because the server applies `CorsLayer::permissive()` — can remove any pending transaction (or the entire pool) without owning or signing anything. This is a direct structural analog to H-04: a function that should be restricted to the node operator is callable by any unprivileged party, and it manipulates shared state (the tx pool) that belongs to other users.

---

### Finding Description

**Root cause — `rpc/src/module/pool.rs`:**

`remove_transaction` is declared as a plain public RPC trait method with no authentication guard: [1](#0-0) 

Its implementation directly forwards the caller-supplied `tx_hash` to the pool controller with no ownership or authorization check: [2](#0-1) 

`clear_tx_pool` and `clear_tx_verify_queue` are equally unguarded: [3](#0-2) 

**Root cause — `rpc/src/server.rs`:**

The HTTP server is built with `CorsLayer::permissive()`, which sets `Access-Control-Allow-Origin: *` and allows all methods and headers. This means a malicious web page loaded in any browser can issue a cross-origin POST to `http://127.0.0.1:8114` and invoke any of these methods, even when the node is bound to localhost: [4](#0-3) 

The default configuration explicitly binds to `127.0.0.1:8114`, but the permissive CORS policy defeats that network-level boundary for browser-based attackers: [5](#0-4) 

There is no authentication token, IP allowlist enforcement at the application layer, or per-method access control anywhere in `ServiceBuilder` or `RpcServer`. [6](#0-5) 

---

### Impact Explanation

**`remove_transaction(tx_hash)`** removes the target transaction *and all of its descendants* from the pending pool: [2](#0-1) 

**`clear_tx_pool()`** wipes the entire pool in one call: [7](#0-6) 

Concrete impacts:

1. **Targeted griefing of a specific user's transaction.** An attacker who observes a pending tx hash (visible via `get_raw_tx_pool`) can call `remove_transaction` repeatedly to prevent it from ever being proposed or committed. The victim must keep resubmitting.

2. **Time-sensitive transaction denial.** CKB DAO withdrawals and `since`-locked inputs are only valid within specific epoch windows. If an attacker continuously evicts a DAO withdrawal transaction, the user may miss their unlock epoch and be forced to wait another full lock period (180 epochs ≈ 30 days), losing accrued interest or suffering indefinite delay. [8](#0-7) 

3. **Mass pool wipe.** `clear_tx_pool` removes every pending transaction from every user simultaneously, causing a complete service disruption for all submitters.

4. **CSRF attack surface.** Because `CorsLayer::permissive()` is unconditional, any web page the node operator visits can silently POST to `http://127.0.0.1:8114` and invoke these methods without the operator's knowledge.

---

### Likelihood Explanation

- **High** if the operator has exposed the RPC to a non-localhost address (common in mining pools, exchanges, and infrastructure operators who need remote access).
- **Medium** for localhost-only deployments: the permissive CORS policy means any malicious web page visited by the node operator can trigger the attack via a browser fetch/XHR. No special privileges are required — only the ability to serve a web page to the victim.
- The tx hashes needed to target specific transactions are publicly observable via `get_raw_tx_pool`, which is also unauthenticated. [9](#0-8) 

---

### Recommendation

1. **Add an authentication token** to the RPC server (e.g., a bearer token in the `Authorization` header or a cookie) and reject requests that do not present it. This is the standard mitigation for local RPC endpoints (used by Bitcoin Core, Geth, etc.).

2. **Replace `CorsLayer::permissive()` with a restrictive CORS policy** that only allows the configured `listen_address` origin, or disables CORS entirely for non-browser use cases.

3. **Separate destructive pool-management methods** (`remove_transaction`, `clear_tx_pool`, `clear_tx_verify_queue`) into a distinct RPC module that is disabled by default and requires explicit opt-in, analogous to the `IntegrationTest` module pattern already used in the codebase. [10](#0-9) 

---

### Proof of Concept

**Attack 1 — Direct RPC (node exposed to network):**

```bash
# Enumerate pending transactions
curl -s http://<node-ip>:8114 -X POST -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false],"id":1}'

# Remove a specific victim transaction and all its descendants
curl -s http://<node-ip>:8114 -X POST -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"],"id":2}'

# Or wipe the entire pool
curl -s http://<node-ip>:8114 -X POST -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":3}'
```

**Attack 2 — CSRF via malicious web page (localhost node):**

```html
<!-- Served from attacker.com; victim visits while running a CKB node -->
<script>
fetch('http://127.0.0.1:8114', {
  method: 'POST',
  headers: {'Content-Type': 'application/json'},
  body: JSON.stringify({
    jsonrpc: '2.0', method: 'clear_tx_pool', params: [], id: 1
  })
});
</script>
```

Because `CorsLayer::permissive()` responds to the browser's preflight with `Access-Control-Allow-Origin: *`, the browser executes the request. The entire tx pool is cleared without the node operator's knowledge or consent. [4](#0-3)
