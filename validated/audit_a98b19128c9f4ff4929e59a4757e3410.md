### Title
Index-Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` via Aged Transaction Eviction — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

### Summary

`remove_unconfirmed_tx` contains a missing bounds check on the `confirm_blocks_to_failed_txs` vector. When a tracked transaction is evicted from the tx-pool after more than `MAX_CONFIRM_BLOCKS` (1000) blocks, the function panics with an index-out-of-bounds, crashing the node process.

### Finding Description

`confirm_blocks_to_failed_txs` is allocated with exactly `max_confirm_blocks = 1000` entries (indices `0..=999`): [1](#0-0) 

In `remove_unconfirmed_tx`, there is a guard that routes "old" transactions (age ≥ 1000) to the `old_unconfirmed_txs` counter — but this guard only protects the `block_unconfirmed_txs` access. The subsequent `count_failure` branch has **no bounds check**: [2](#0-1) 

When `tx_age = 1001`, `tx_age - 1 = 1000`, which is out of bounds for a vec of length 1000. Rust panics unconditionally, terminating the node process.

### Impact Explanation

A Rust index-out-of-bounds panic aborts the process. Any node running the `ConfirmationFraction` fee estimator that evicts a transaction aged ≥ 1001 blocks will crash. This is a remotely triggerable node crash causing network disruption.

### Likelihood Explanation

The tx-pool expiry is **time-based** (default 12 hours), not block-based: [3](#0-2) 

At CKB's ~10-second block time, 12 hours ≈ 4320 blocks — far exceeding the 1000-block threshold. The `remove_expired` path calls `callbacks.call_reject` for every expired tx: [4](#0-3) 

The reject callback unconditionally calls `fee_estimator.reject_tx`: [5](#0-4) 

Which routes to `drop_tx_inner(hash, true)` → `remove_unconfirmed_tx(..., count_failure=true)`: [6](#0-5) 

The `limit_size` eviction path (pool full) also triggers the same callback: [7](#0-6) 

An unprivileged attacker submits a valid low-fee-rate transaction, waits for 1001+ blocks (≈2.8 hours), then causes pool eviction (e.g., by flooding the pool with higher-fee txs to trigger `limit_size`). The panic fires deterministically.

### Recommendation

Add a bounds check before the `count_failure` branch in `remove_unconfirmed_tx`:

```rust
if count_failure && tx_age - 1 < self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

Transactions aged beyond `MAX_CONFIRM_BLOCKS` should be silently dropped from failure accounting (they are already moved to `old_unconfirmed_txs`), consistent with how `add_confirmed_tx` caps at `max_confirms()`.

### Proof of Concept

```rust
#[test]
fn test_oob_panic() {
    let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
    let mut stat = TxConfirmStat::new(buckets, 1000, 0.993);
    // Track tx at height 0
    let bucket_index = stat.add_unconfirmed_tx(0, FeeRate::from_u64(1500)).unwrap();
    // Simulate 1001 blocks passing, then eviction with count_failure=true
    // tx_age = 1001 - 0 = 1001; confirm_blocks_to_failed_txs[1000] → PANIC
    stat.remove_unconfirmed_tx(0, 1001, bucket_index, true);
}
```

Running this test panics: `index out of bounds: the len is 1000 but the index is 1000`. [8](#0-7)
