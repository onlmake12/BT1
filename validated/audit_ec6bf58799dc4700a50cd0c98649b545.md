### Title
Fee Locked at Request Time Becomes Economically Stale After Provider Fee Update, Permanently Locking User Funds — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.sol`'s `requestPriceUpdatesWithCallback` fixes the provider fee at request time and stores it in `req.fee`. Because `setProviderFee()` can be called at any time and there is no cancel/refund path and no penalty for non-fulfillment, a provider fee increase after a request is submitted makes old requests economically unviable to fulfill, permanently locking user funds.

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, the fee is computed against the provider's **current** fee schedule and stored:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
...
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [1](#0-0) 

`getFee` uses the provider's live `baseFeeInWei`, `feePerFeedInWei`, and `feePerGasInWei`:

```solidity
uint96 providerBaseFee = _state.providers[provider].baseFeeInWei;
uint96 providerFeedFee = SafeCast.toUint96(
    priceIds.length * _state.providers[provider].feePerFeedInWei
);
uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei;
uint256 gasFee = callbackGasLimit * providerFeeInWei;
``` [2](#0-1) 

The provider can update all three rates at any time via `setProviderFee()`: [3](#0-2) 

At execution time, the provider is credited only `req.fee + msg.value - pythFee`. If the provider raised their rates after the request was submitted, `req.fee` is now below their minimum, giving them no economic incentive to execute. The code itself acknowledges the missing penalty mechanism:

```solidity
// TODO: there also needs to be some penalty mechanism in case the expected provider
// doesn't execute the callback.
``` [4](#0-3) 

There is no cancel or refund function anywhere in `Echo.sol` or `EchoState.sol`. The only fund-exit paths are `withdrawFees` (admin only, for Pyth protocol fees) and `withdrawAsFeeManager` (provider's fee manager only, for accrued provider fees). Neither allows the original requester to recover their locked ETH. [5](#0-4) [6](#0-5) 

A secondary compounding issue: `executeCallback` calls `parsePriceFeedUpdates` with `minPublishTime = maxPublishTime = req.publishTime` (a single exact timestamp). If the provider delays execution and Hermes no longer retains data for that exact timestamp, `parsePriceFeedUpdates` reverts, making the request permanently unexecutable even if the provider is willing. [7](#0-6) 

### Impact Explanation

User ETH paid for a price-update request can be permanently locked in the Echo contract with no recovery path. The provider collects nothing (since they never execute), and the user loses their entire fee. For protocols that depend on Echo for time-sensitive price callbacks (e.g., liquidations, settlement), the stuck request also means the downstream action never fires, causing secondary protocol-level harm.

### Likelihood Explanation

The trigger requires only a registered provider calling `setProviderFee()` after accumulating pending requests — a single permissionless transaction. A provider acting in bad faith (or simply repricing to market rates during a gas spike) can strand all in-flight requests. The exclusivity period (`exclusivityPeriodSeconds`) makes it worse: during that window no alternative provider can step in, so even a temporary fee increase is sufficient to strand requests until the window expires, by which time the `publishTime` may be too old for Hermes to serve. [8](#0-7) 

### Recommendation

1. **Add a user-cancellable refund path**: Allow the requester to cancel an unfulfilled request after a timeout and recover `req.fee + _state.pythFeeInWei`.
2. **Snapshot provider fees into the request**: Store the provider's fee schedule at request time and enforce it at execution time, so the provider cannot retroactively make a request uneconomical.
3. **Widen the `parsePriceFeedUpdates` time window**: Use `[req.publishTime, req.publishTime + delta]` instead of a single exact timestamp to tolerate minor Hermes data gaps.
4. **Implement a penalty/escalation mechanism**: As the existing TODO notes, penalize providers that fail to fulfill within the exclusivity window and allow any provider to step in with the stored fee as incentive.

### Proof of Concept

1. Provider registers with `baseFeeInWei = 1 wei`, `feePerFeedInWei = 1 wei`, `feePerGasInWei = 1 wei`.
2. User calls `requestPriceUpdatesWithCallback{value: getFee(...)}(provider, block.timestamp, priceIds, gasLimit)`. `req.fee` is stored at the low rate.
3. Provider calls `setProviderFee(provider, 1e18, 1e18, 1e18)` — raising rates to 1 ETH each.
4. Provider ignores the request (executing would credit them only the old `req.fee`, far below their new minimum).
5. During `exclusivityPeriodSeconds`, no other provider can execute.
6. After the exclusivity period, no other provider will execute either, since `req.fee` is below any rational provider's minimum.
7. The user has no cancel function to call. Their ETH is permanently locked in the contract.
