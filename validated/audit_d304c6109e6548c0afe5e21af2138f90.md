Audit Report

## Title
Unbounded response amplification via duplicate indexes in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` accepts up to `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) transaction indexes from an untrusted peer but performs no deduplication and no byte-cap check on the assembled response. An attacker sending 32,767 copies of index `0` causes the node to clone and serialize the same transaction (e.g., the cellbase) 32,767 times into a single `BlockTransactions` message. The sibling handlers `GetTransactionsProcess` and `GetBlockProposalProcess` both implement these protections; `GetBlockTransactionsProcess` does not.

## Finding Description

**Constants confirmed** in `sync/src/relayer/mod.rs`: [1](#0-0) 

`GetBlockTransactionsProcess` imports only `MAX_RELAY_TXS_NUM_PER_BATCH`, never `MAX_RELAY_TXS_BYTES_PER_BATCH`: [2](#0-1) 

The only guard is a count check — no deduplication: [3](#0-2) 

After passing that check, every index (including duplicates) is mapped directly to a transaction: [4](#0-3) 

All clones are serialized into one message and sent with no size limit: [5](#0-4) 

**Contrast with sibling handlers:**

`GetTransactionsProcess` deduplicates via `HashSet` and enforces `MAX_RELAY_TXS_BYTES_PER_BATCH`: [6](#0-5) [7](#0-6) 

`GetBlockProposalProcess` deduplicates via `HashSet` and enforces `MAX_RELAY_TXS_BYTES_PER_BATCH`: [8](#0-7) [9](#0-8) 

The rate limiter is keyed by `(PeerIndex, message_type)` and limits to 30 messages/second — it bounds message *count*, not message *size*: [10](#0-9) 

## Impact Explanation

A CKB mainnet cellbase transaction with a full witness is several kilobytes. At 32,767 copies × ~4 KB = ~128 MB per response message. The rate limiter permits 30 such messages per second per peer. With `MAX_RELAY_PEERS = 128` peers each sending 30 requests/second, outbound bandwidth reaches 128 × 30 × 128 MB ≈ hundreds of GB/s — far beyond any node's capacity. Concurrently, each in-flight response allocates the full cloned `Vec<TransactionView>` before serialization, causing rapid OOM. This matches two allowed High impacts: **"Vulnerabilities which could easily crash a CKB node"** and **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation

- Requires only a valid P2P connection — no key, no PoW, no stake.
- The attacker needs only one stored block hash, trivially obtained from any public block explorer or by passively observing the network.
- The exploit is deterministic and repeatable; a single malicious peer can sustain it indefinitely within the rate limit.
- The rate limiter does not bound response size, only request count.

## Recommendation

1. **Deduplicate indexes** before lookup, mirroring `GetTransactionsProcess`:
   ```rust
   let indexes_set: HashSet<u32> = self.message.indexes().iter()
       .map(Into::<u32>::into).collect();
   if indexes_set.len() != self.message.indexes().len() {
       return StatusCode::RequestDuplicate.with_context("Duplicate indexes");
   }
   ```
2. **Import and enforce `MAX_RELAY_TXS_BYTES_PER_BATCH`** on the assembled response, aborting or splitting once the total serialized size exceeds 1 MiB, consistent with `GetTransactionsProcess` and `GetBlockProposalProcess`.

## Proof of Concept

```
1. Connect to a CKB node over P2P (RelayV3 protocol).
2. Obtain any stored block hash H (e.g., genesis block hash, publicly known).
3. Craft a GetBlockTransactions message:
     block_hash    = H
     indexes       = [0] * 32767   # 32767 copies of index 0
     uncle_indexes = []
4. Send the message.
5. Observe: the node allocates and serializes 32767 copies of the cellbase
   transaction and sends a single BlockTransactions response.
6. Assert: response_bytes >> MAX_RELAY_TXS_BYTES_PER_BATCH (1 MiB).
7. Repeat at 30 req/s per peer connection to sustain bandwidth/memory exhaustion.
8. Open 128 peer connections to multiply the effect by MAX_RELAY_PEERS.
```
