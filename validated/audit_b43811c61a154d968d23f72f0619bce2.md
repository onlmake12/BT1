### Title
No Refund of Excess ETH Overpayment in `updatePriceFeeds` and Related Payable Functions - (File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol)

### Summary
The Pyth EVM contract accepts any `msg.value >= requiredFee` in its payable price-update functions but never refunds the excess ETH to the caller. Any ETH above the required fee is permanently locked in the contract, recoverable only via a privileged governance `withdrawFee` VAA. An unprivileged updater who overpays (e.g., by adding a safety buffer, using a stale fee estimate, or being front-run by a governance fee change) permanently loses the excess funds.

---

### Finding Description

Every payable entry point in `Pyth.sol` enforces only a lower-bound fee check:

**`updatePriceFeeds`** (lines 77–78):
```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
```

**`parsePriceFeedUpdatesWithConfig`** (lines 336–337):
```solidity
if (msg.value < getUpdateFee(updateData))
    revert PythErrors.InsufficientFee();
```

**`parseTwapPriceFeedUpdates`** (lines 505–506):
```solidity
uint requiredFee = getTwapUpdateFee(updateData);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
```

None of these functions compute `msg.value - requiredFee` and return the difference to `msg.sender`. The entire `msg.value` is absorbed by the contract. The only recovery path is the governance-controlled `withdrawFee` function in `PythGovernance.sol` (lines 264–272), which requires a signed governance VAA and sends funds to an arbitrary governance-chosen address — not back to the original caller. The `getTotalFee` helper confirms the exact required amount:
```solidity
function getTotalFee(uint totalNumUpdates) private view returns (uint requiredFee) {
    return (totalNumUpdates * singleUpdateFeeInWei()) + transactionFeeInWei();
}

---

### Impact Explanation

Any ETH sent above `requiredFee` is permanently locked in the Pyth contract. The caller has no on-chain mechanism to recover it. The governance `withdrawFee` path sends funds to a governance-chosen address, not the original overpaying caller. This constitutes a direct, irreversible loss of user funds for every overpaying transaction.

**Impact: High** — direct ETH loss with no user-accessible recovery path.

---

### Likelihood Explanation

Several realistic scenarios cause overpayment:

1. **Stale fee estimate**: A caller queries `getUpdateFee` off-chain, then a governance `SetFee` VAA executes before the update transaction lands, changing `singleUpdateFeeInWei`. The caller's pre-computed fee is now higher than required.
2. **Safety buffer**: Integrators and relayers commonly add a small ETH buffer (e.g., `fee * 110 / 100`) to avoid `InsufficientFee` reverts under fee volatility.
3. **Smart contract pass-through**: A DeFi protocol that calls `updatePriceFeeds` and forwards `msg.value` directly without computing the exact fee will routinely overpay.
4. **`updatePriceFeedsIfNecessary` with empty update**: The function forwards the full `msg.value` to `updatePriceFeeds` via an internal call; if the update is not needed the call reverts, but if it is needed the full value is consumed regardless of the actual fee. [6](#0-5) 

**Likelihood: Medium** — common integration patterns and fee-change races make overpayment a realistic, recurring event.

---

### Recommendation

After computing `requiredFee`, refund the surplus to `msg.sender` in each payable function:

```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
uint excess = msg.value - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "Refund failed");
}
```

Apply the same pattern to `parsePriceFeedUpdatesWithConfig` and `parseTwapPriceFeedUpdates`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

interface IPyth {
    function updatePriceFeeds(bytes[] calldata updateData) external payable;
    function getUpdateFee(bytes[] calldata updateData) external view returns (uint);
}

contract OverpaymentPoC {
    IPyth pyth;

    constructor(address _pyth) { pyth = IPyth(_pyth); }

    /// Caller sends 10x the required fee.
    /// After the call, the excess 9x fee is locked in the Pyth contract
    /// and the caller's balance is reduced by the full 10x amount.
    function triggerOverpayment(bytes[] calldata updateData) external payable {
        uint required = pyth.getUpdateFee(updateData);
        uint overpay  = required * 10;
        require(msg.value >= overpay, "send 10x fee");

        uint callerBefore = address(this).balance;
        pyth.updatePriceFeeds{value: overpay}(updateData);
        uint callerAfter  = address(this).balance;

        // callerBefore - callerAfter == overpay (not required)
        // Excess = overpay - required is permanently lost to the caller.
        assert(callerBefore - callerAfter == overpay);
    }
}
```

The `assert` passes — the full `overpay` amount is consumed, not just `required`. The difference (`overpay - required`) is irrecoverably locked in the Pyth contract until a governance VAA is issued.
