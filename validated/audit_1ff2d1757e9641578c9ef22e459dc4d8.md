### Title
`MAX_DEPOSIT_LIMIT` Bypass for Permanent Subscriptions via Cumulative `addFunds` Calls — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `MAX_DEPOSIT_LIMIT` (100 ETH) cap for permanent subscriptions in `Scheduler.sol` is enforced only against the per-transaction `msg.value`, not against the cumulative `balanceInWei + msg.value`. Any unprivileged user who owns a permanent subscription can call `addFunds` repeatedly with amounts ≤ 100 ETH each time, accumulating a balance far exceeding the intended cap and permanently locking excess ETH in an irrecoverable subscription.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
```

Both `createSubscription` and `addFunds` enforce this limit only against the current transaction's `msg.value`:

**`createSubscription` (line 48):**
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**`addFunds` (line 613):**
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Neither check accounts for the subscription's existing `status.balanceInWei`. The correct guard should be:

```solidity
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**Attack path:**
1. Create a permanent subscription with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH). This passes the check.
2. Call `addFunds` with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH). `msg.value (100 ETH) > MAX_DEPOSIT_LIMIT (100 ETH)` is **false**, so the check passes.
3. `status.balanceInWei` is now 200 ETH — double the intended cap.
4. Repeat indefinitely. The balance grows without bound.

Since permanent subscriptions cannot be deactivated (`CannotUpdatePermanentSubscription`) and cannot have funds withdrawn (`withdrawFunds` reverts for permanent subscriptions at line 640), all excess ETH is permanently locked.

---

### Impact Explanation

- **Funds permanently locked beyond the intended cap.** The `MAX_DEPOSIT_LIMIT` is the protocol's safety bound on how much ETH can be irrecoverably committed to a single permanent subscription. Bypassing it allows an attacker (or a mistaken user) to lock an unbounded amount of ETH with no recovery path.
- **Protocol invariant violation.** The cap exists to bound the maximum ETH exposure per permanent subscription. Its bypass undermines the protocol's economic safety guarantees.
- **Severity: Medium.** The attacker must own the subscription (self-inflicted for a user, but a malicious actor could grief the protocol's ETH accounting or exploit downstream logic that reads `balanceInWei`).

---

### Likelihood Explanation

- Any subscription manager of a permanent subscription can trigger this with no special privileges.
- The entry path is a standard `payable` external function (`addFunds`) callable by any EOA or contract.
- The bypass requires only multiple transactions, each individually valid.
- Likelihood is **medium**: it requires intentional multi-step action, but no privileged access or external dependency.

---

### Recommendation

Change both `createSubscription` and `addFunds` to check the **cumulative** balance against `MAX_DEPOSIT_LIMIT`:

In `createSubscription`:
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```
(This is already correct for creation since `balanceInWei` starts at 0.)

In `addFunds`, replace:
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
```
with:
```solidity
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
```

---

### Proof of Concept

```solidity
// 1. Create permanent subscription at the cap
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);
// status.balanceInWei == 100 ether

// 2. Add funds again — msg.value == MAX_DEPOSIT_LIMIT, check passes (100 > 100 is false)
scheduler.addFunds{value: 100 ether}(subId);
// status.balanceInWei == 200 ether — cap bypassed

// 3. Funds are permanently locked; withdrawFunds reverts for permanent subscriptions
scheduler.withdrawFunds(subId, 1 ether); // reverts: CannotUpdatePermanentSubscription
```
