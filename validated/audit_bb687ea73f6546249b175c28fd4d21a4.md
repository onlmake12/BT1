### Title
`executeCallback` Credits Fees to Unverified `providerToCredit` After Exclusivity Period — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, the `executeCallback` function accepts a caller-supplied `providerToCredit` address that is credited with the request fee. After the exclusivity period expires, there is no check that `providerToCredit` is a registered provider. An unprivileged attacker can call `executeCallback` with `providerToCredit` set to an arbitrary unregistered address, permanently locking the user-paid fees in the contract and denying the legitimate provider their expected payment.

### Finding Description
`requestPriceUpdatesWithCallback` stores the assigned provider in `req.provider` and collects a fee from the user. When `executeCallback` is called, it accepts a `providerToCredit` parameter and credits `_state.providers[providerToCredit].accruedFeesInWei` with the request fee.

During the exclusivity window, the contract enforces `providerToCredit == req.provider`. However, once the exclusivity period elapses, this constraint is dropped entirely and `providerToCredit` is never validated against the registered providers list: [1](#0-0) 

The fee is then unconditionally credited to the unverified address: [2](#0-1) 

The only withdrawal path for provider fees is `withdrawAsFeeManager`, which requires `msg.sender == _state.providers[provider].feeManager`. For any unregistered address, `feeManager` is `address(0)`, making the credited fees permanently irrecoverable: [3](#0-2) 

This is structurally identical to the reported bug: a caller-supplied parameter (`providerToCredit`) is used to drive a state-changing operation (fee accounting) without being verified against the actual on-chain registered state (`req.provider` / `isRegistered`).

### Impact Explanation
- **Fee locking**: An attacker calls `executeCallback` after the exclusivity period with `providerToCredit = address(0xdead)` (any unregistered address). The `req.fee` (user-paid provider fee) is credited to an address with no `feeManager`, making it permanently unwithdrawable.
- **Fee theft**: An attacker pre-registers as a provider (permissionless via `registerProvider`), monitors pending requests, and after the exclusivity period calls `executeCallback` crediting themselves, stealing the fee from the original assigned provider.

In both cases, the legitimate provider is denied their expected compensation for fulfilling the callback.

### Likelihood Explanation
The exclusivity period is a short configurable window (default 15 seconds per the contract initialization). Any pending unfulfilled request is publicly visible on-chain. After the window expires, the attack requires only a single transaction with no special privileges. The `registerProvider` function is fully permissionless: [4](#0-3) 

The attacker entry path is: (1) register as a provider, (2) watch for requests whose exclusivity period has elapsed, (3) call `executeCallback` with `providerToCredit = self`. This is trivially automatable.

### Recommendation
After the exclusivity period, `providerToCredit` should still be required to be a registered provider:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit must be a registered provider"
);
```

This mirrors the fix in the referenced report: derive the displayed/credited entity from verified on-chain state rather than accepting it as a raw caller-supplied parameter.

### Proof of Concept

1. Deploy `Echo` with `exclusivityPeriodSeconds = 15`.
2. Attacker calls `echo.registerProvider(...)` — permissionless.
3. User calls `echo.requestPriceUpdatesWithCallback{value: fee}(legitimateProvider, ...)` — `req.fee` is stored.
4. `block.timestamp` advances past `req.publishTime + 15`.
5. Attacker calls:
   ```solidity
   echo.executeCallback(
       address(0xdead),   // providerToCredit — unregistered, no feeManager
       sequenceNumber,
       updateData,
       priceIds
   );
   ```
6. `_state.providers[address(0xdead)].accruedFeesInWei` is incremented by `req.fee`.
7. `address(0xdead).feeManager == address(0)` → `withdrawAsFeeManager` reverts for any caller.
8. Fees are permanently locked; `legitimateProvider` receives nothing. [5](#0-4)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

    function registerProvider(
        uint96 baseFeeInWei,
        uint96 feePerFeedInWei,
        uint96 feePerGasInWei
    ) external override {
        ProviderInfo storage provider = _state.providers[msg.sender];
        require(!provider.isRegistered, "Provider already registered");
        provider.baseFeeInWei = baseFeeInWei;
        provider.feePerFeedInWei = feePerFeedInWei;
        provider.feePerGasInWei = feePerGasInWei;
        provider.isRegistered = true;
        emit ProviderRegistered(msg.sender, feePerGasInWei);
    }
