### Title
Excess `msg.value` Permanently Captured as Protocol Fees Instead of Refunded to Caller — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.requestHelper` credits **all** of `msg.value - providerFee` to `_state.accruedPythFeesInWei` rather than only the Pyth portion of the required fee. Any ETH sent above `requiredFee` is silently captured by the protocol instead of being returned to the caller. The same pattern exists in `Echo.requestPriceUpdatesWithCallback`, where excess above `_state.pythFeeInWei` is stored as `req.fee` and later credited to the provider.

---

### Finding Description

In `Entropy.sol`, `requestHelper` performs the following fee accounting:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The check at line 235 only enforces a **minimum** (`msg.value >= requiredFee`). However, the accounting at lines 237–239 consumes **all** of `msg.value`: `providerFee` goes to the provider and the entire remainder (`msg.value - providerFee`) goes to `_state.accruedPythFeesInWei`. If `msg.value > requiredFee`, the surplus above `requiredFee` is credited to Pyth's withdrawable fee pool rather than refunded to the caller.

The contract's own NatSpec comment on `request()` and `requestWithCallback()` explicitly acknowledges this:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

All three public entry points delegate to `requestHelper` and are therefore affected:

- `request()` — line 322 [4](#0-3) 
- `requestWithCallback()` — line 346 [5](#0-4) 
- `requestV2()` — line 358 [6](#0-5) 

**Secondary instance — `Echo.sol`:**

`Echo.requestPriceUpdatesWithCallback` stores `req.fee = msg.value - _state.pythFeeInWei` (line 84). When `msg.value > requiredFee`, the excess above `pythFeeInWei` is silently added to `req.fee` and later credited to the provider in `executeCallback` (line 162), not returned to the requester. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

Any caller who sends `msg.value` above the exact `requiredFee` permanently loses the surplus. In the Entropy case the surplus is credited to `_state.accruedPythFeesInWei`, which is withdrawable only by the Pyth admin — the user has no mechanism to recover it. In the Echo case the surplus is credited to the provider's accrued fees. In both cases the user's ETH is irreversibly transferred to a third party without consent.

**Impact class:** Direct, permanent loss of user funds (ETH). Severity: Medium (funds lost per-call, but bounded by the overpayment amount per transaction).

---

### Likelihood Explanation

This is triggered by any unprivileged caller of `request`, `requestWithCallback`, or `requestV2` who sends more than the exact fee. This is a realistic and common scenario because:

1. Fee estimation (`getFeeV2`) is a view call made before the transaction is mined; the fee can change between estimation and execution if the provider or Pyth admin updates fees.
2. Smart contract integrators routinely add a small buffer (e.g., `fee * 110 / 100`) to avoid `InsufficientFee` reverts.
3. Wallets and SDKs that call `getFee` and then add a safety margin will silently overpay on every request.

---

### Recommendation

Compute the exact Pyth portion of the required fee and credit only that amount to `_state.accruedPythFeesInWei`. Refund any surplus to `msg.sender` at the end of `requestHelper`:

```solidity
uint128 pythFee = requiredFee - providerFee;
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += pythFee;

uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

Apply the same pattern to `Echo.requestPriceUpdatesWithCallback`: store only `requiredFee - _state.pythFeeInWei` as `req.fee` and refund `msg.value - requiredFee` to `msg.sender`.

---

### Proof of Concept

1. Provider registers with `feeInWei = 100 wei`; Pyth fee is `50 wei`; `requiredFee = 150 wei`.
2. User calls `requestV2{value: 200 wei}(provider, userContribution, gasLimit)`.
3. Inside `requestHelper`:
   - `requiredFee = 150`, `providerFee = 100`
   - `providerInfo.accruedFeesInWei += 100` ✓
   - `_state.accruedPythFeesInWei += (200 - 100) = 100` ← should be `50`
4. The user paid `200 wei` but the protocol required only `150 wei`. The extra `50 wei` is credited to Pyth's fee pool and is permanently lost to the user.
5. Pyth admin calls `withdrawFee` and extracts the `50 wei` surplus.
