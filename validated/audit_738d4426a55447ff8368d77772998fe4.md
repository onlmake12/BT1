### Title
Unrecoverable Node Shutdown via `panic!` in `HeaderAcceptor::accept()` Propagating to `broadcast_exit_signals()` — (`sync/src/synchronizer/headers_process.rs`)

---

### Summary

A `panic!()` embedded in `HeaderAcceptor::accept()` fires when the node's block-status map records a header as `HEADER_VALID` but the corresponding header index view is absent (a TOCTOU race between concurrent block processing threads). Because the sync protocol handler runs this code inside `tokio::task::block_in_place()` with no `catch_unwind`, the panic propagates to the async task, the tentacle P2P framework raises `ProtocolHandleError::AbnormallyClosed`, and the network service responds by calling `broadcast_exit_signals()` — an irreversible, node-wide shutdown. There is no recovery path analogous to LayerZero's `forceResumeReceive`; a single crafted `SendHeaders` message delivered at the right moment terminates the node.

---

### Finding Description

**Step 1 — The reachable `panic!` in the `SendHeaders` handler**

`HeaderAcceptor::accept()` first reads the block status for the incoming header, then immediately tries to fetch the header index view from a separate concurrent map:

```rust
// sync/src/synchronizer/headers_process.rs  lines 303-316
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    let header_index = sync_shared
        .get_header_index_view(
            &self.header.hash(),
            status.contains(BlockStatus::BLOCK_STORED),
        )
        .unwrap_or_else(|| {
            panic!(
                "header {}-{} with HEADER_VALID should exist",
                self.header.number(),
                self.header.hash()
            )
        })
```

`get_block_status()` reads from `shared.block_status_map` (a `DashMap`) and `get_header_index_view()` reads from `sync_shared.header_map` (a separate concurrent structure). These two reads are **not atomic**. A concurrent thread (e.g., the `ConsumeUnverifiedBlocks` pipeline or a reorg) can evict or overwrite the header index view between the two reads, leaving the status as `HEADER_VALID` while the view is `None`. The `panic!` fires unconditionally in that window. [1](#0-0) 

**Step 2 — The panic propagates through `block_in_place` with no catch**

`SendHeaders` messages are dispatched via:

```rust
// sync/src/synchronizer/mod.rs  lines 402-405
packed::SyncMessageUnionReader::SendHeaders(reader) => {
    tokio::task::block_in_place(|| {
        HeadersProcess::new(reader, self, peer, &nc).execute()
    })
}
```

`block_in_place` does not catch panics; the panic unwinds through the async task. [2](#0-1) 

**Step 3 — The network service calls `broadcast_exit_signals()` with no recovery**

When the async task panics, tentacle raises `ProtocolHandleError::AbnormallyClosed`. The `CKBNetworkService::handle_error` handler responds:

```rust
// network/src/network.rs  lines 688-718
ServiceError::ProtocolHandleError { proto_id, error } => {
    let ProtocolHandleErrorKind::AbnormallyClosed(opt_session_id) = error;
    {
        if let Some(id) = opt_session_id {
            self.network_state.ban_session(...);
        }
        // ...
        broadcast_exit_signals();   // ← entire node shuts down
    }
}
```

`broadcast_exit_signals()` cancels the tokio runtime and sends stop signals to every registered thread. There is no mechanism to skip the offending message and resume processing — the node terminates. [3](#0-2) 

---

### Impact Explanation

Any unprivileged peer that can establish a sync connection can send a `SendHeaders` P2P message. If the message contains a header whose hash is already recorded as `HEADER_VALID` in the node's block-status map (e.g., a recently validated header the attacker observed on-chain) and the delivery is timed to coincide with concurrent block-processing activity that transiently removes or has not yet populated the header index view, the `panic!` fires and the node shuts down completely. This is a remote, unauthenticated denial-of-service against any CKB full node.

---

### Likelihood Explanation

The race window exists between `insert_block_status(..., HEADER_VALID)` and the corresponding `insert_valid_header()` call (or during a reorg that clears the header map). An attacker who monitors the public chain can identify recently validated headers and flood `SendHeaders` messages during IBD or reorg events to maximize the probability of hitting the window. The attack requires no special privilege, no key material, and no majority hash power.

---

### Recommendation

1. **Replace `panic!` with a graceful error return** in `HeaderAcceptor::accept()`. Return a `ValidationState::TemporaryInvalid` or a new `ValidationState::InternalError` variant instead of panicking when `get_header_index_view()` returns `None` for a `HEADER_VALID` header.

2. **Wrap `block_in_place` closures with `catch_unwind`** in `Synchronizer::try_process()` so that any future panics in sync message handlers are converted to error statuses rather than propagating to the async task.

3. **Replace `broadcast_exit_signals()` in the `ProtocolHandleError` arm** with a targeted response (disconnect the offending peer, log the error) so that a single misbehaving peer cannot terminate the node.

---

### Proof of Concept

```
1. Attacker connects to a CKB full node as a sync peer.
2. Attacker observes a header H that the node has recently validated
   (HEADER_VALID set, visible via sync state or block propagation).
3. Attacker sends a SendHeaders P2P message containing header H,
   timed to arrive while the node is concurrently processing a reorg
   or during IBD when the header_map is being updated.
4. HeaderAcceptor::accept() reads HEADER_VALID from block_status_map,
   then reads None from header_map (race window).
5. panic!("header {}-{} with HEADER_VALID should exist", ...) fires.
6. Panic propagates through block_in_place → Synchronizer::received().
7. Tentacle raises ProtocolHandleError::AbnormallyClosed.
8. network/src/network.rs:717 calls broadcast_exit_signals().
9. Node shuts down; no recovery is possible without operator restart.
```
