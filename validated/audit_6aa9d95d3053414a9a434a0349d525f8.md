### Title
Overly Broad DAO Withdrawal Exemption in `CapacityVerifier` Allows Non-DAO Cell Capacity Inflation - (`verification/src/transaction_verifier.rs`)

### Summary

`CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` if **any** input uses the DAO type script, causing `CapacityVerifier::verify()` to skip the entire `OutputsSumOverflow` check for the transaction. However, the on-chain DAO type script only validates DAO cells â€” it does not enforce capacity conservation for non-DAO cells in the same transaction. This inconsistency allows an attacker to inflate non-DAO output capacity beyond non-DAO input capacity, creating CKB tokens out of thin air, bounded by the DAO interest earned.

### Finding Description

`CapacityVerifier::verify()` in `verification/src/transaction_verifier.rs` skips the `OutputsSumOverflow` check when `valid_dao_withdraw_transaction()` returns true: [1](#0-0) 

The exemption is justified by the comment "DAO withdraw transaction is verified via the type script of DAO cells." However, `valid_dao_withdraw_transaction()` uses a coarse criterion â€” it returns `true` if **any** input cell uses the DAO type script: [2](#0-1) 

When this returns `true`, the **entire** `OutputsSumOverflow` check is skipped for the whole transaction, including all non-DAO inputs and outputs.

The DAO type script is a CKB type script. In CKB's execution model, type scripts only run for cells they are attached to. The DAO script validates that the DAO output capacity equals the maximum withdrawal amount â€” it does not validate non-DAO cells in the same transaction.

The `FeeCalculator` (also inside `ContextualTransactionVerifier`) calls `DaoCalculator::transaction_fee()`: [3](#0-2) 

`transaction_fee` computes `maximum_withdraw - outputs_capacity`, where `maximum_withdraw` for non-DAO inputs is simply `output.capacity()` (the raw input capacity): [4](#0-3) 

And for DAO inputs it is the interest-adjusted amount: [5](#0-4) 

So `maximum_withdraw` = `DAO_max_withdraw` + `non_DAO_input_capacity`. If an attacker inflates non-DAO outputs by Î” (where Î” â‰¤ DAO interest), the fee is still `(DAO_interest - Î”) > 0`, which passes the fee check in `check_tx_fee`: [6](#0-5) 

No other verifier enforces per-cell capacity conservation for non-DAO cells in a DAO withdrawal transaction.

### Impact Explanation

An attacker who holds a DAO withdrawal cell (phase 2) can craft a transaction that:
1. Includes the DAO input and non-DAO inputs
2. Sets non-DAO outputs to exceed non-DAO inputs by up to the DAO interest amount
3. Keeps total outputs â‰¤ total `maximum_withdraw` so the fee check passes

This inflates the total CKB supply by up to the DAO interest amount per withdrawal transaction. The `CapacityVerifier` skips the sum check, the DAO script ignores non-DAO cells, and the `FeeCalculator` only checks the aggregate fee â€” none of them catch the per-cell non-DAO imbalance. This is a direct violation of CKB's capacity conservation invariant and constitutes unauthorized token creation.

### Likelihood Explanation

Any user who has completed DAO phase 1 (prepare) and is eligible for phase 2 withdrawal can exploit this. No privileged access is required â€” depositing into the DAO is a normal user action. The attacker simply crafts the withdrawal transaction with inflated non-DAO outputs. The exploit is reachable via the standard `send_transaction` RPC path.

### Recommendation

`CapacityVerifier::verify()` should not skip the `OutputsSumOverflow` check for the entire transaction. Instead, it should either:

1. **Perform a partial check**: compute `non_DAO_inputs_sum` and `non_DAO_outputs_sum` separately and enforce `non_DAO_outputs_sum â‰¤ non_DAO_inputs_sum`, while still skipping the check for DAO cells (whose capacity increase is legitimately validated by the DAO type script).

2. **Remove the exemption entirely**: rely solely on the DAO type script to validate DAO cell capacity, and enforce the standard `OutputsSumOverflow` check for all inputs/outputs. The DAO type script already ensures the DAO output does not exceed `maximum_withdraw`, so the aggregate check would still pass for valid DAO withdrawals.

### Proof of Concept

Concrete numbers:
- DAO input (phase 2): 1000 CKB deposited, `maximum_withdraw` = 1010 CKB (10 CKB interest)
- Non-DAO input: 100 CKB
- DAO output: 1004 CKB (6 CKB left as fee from DAO interest)
- Non-DAO output: **105 CKB** (5 CKB more than the 100 CKB non-DAO input)
- Total `maximum_withdraw` = 1010 + 100 = 1110 CKB
- Total outputs = 1004 + 105 = 1109 CKB
- Fee = 1110 âˆ’ 1109 = 1 CKB â†’ passes `check_tx_fee`

`CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` (DAO input present), so the `OutputsSumOverflow` check is skipped entirely. The DAO type script validates only the DAO output (1004 CKB â‰¤ 1010 CKB maximum). The non-DAO imbalance (100 CKB in â†’ 105 CKB out) is never checked. The attacker creates 5 CKB out of thin air.     tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,

    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
