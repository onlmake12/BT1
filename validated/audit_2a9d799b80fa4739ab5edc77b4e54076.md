### Title
Missing Staleness Check in `PythAggregatorV3` Exposes Integrators to Stale Price Exploitation - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatible adapter contract, uses `getPriceUnsafe()` in every price-reading function without any staleness check. Any protocol that migrates from Chainlink to Pyth using this adapter will silently consume arbitrarily stale prices, enabling unprivileged attackers to exploit the stale data for profit.

---

### Finding Description

`PythAggregatorV3` is the production Solidity contract Pyth provides to allow Chainlink-dependent protocols to migrate to Pyth price feeds. It implements `AggregatorV3Interface` and is explicitly documented as the migration path.

Every price-reading function in the contract calls `pyth.getPriceUnsafe(priceId)`:

- `latestAnswer()` — line 54
- `latestTimestamp()` — line 59
- `latestRound()` — delegates to `latestTimestamp()`
- `getAnswer(uint256)` — delegates to `latestAnswer()`
- `getTimestamp(uint256)` — delegates to `latestTimestamp()`
- `getRoundData(uint80)` — line 89
- `latestRoundData()` — line 110
- `decimals()` — line 41

`getPriceUnsafe()` is explicitly documented to return prices from **arbitrarily far in the past** with no recency guarantee. The Pyth SDK provides `getPriceNoOlderThan(id, age)` as the safe alternative, but `PythAggregatorV3` never calls it.

The analog to the Chainlink report is direct:
- Chainlink: `timeStamp == 0` (incomplete round) and `answeredInRound < roundId` (carried-over answer) are missing checks.
- Pyth: `publishTime` staleness is never validated in `PythAggregatorV3`, meaning a price from hours or days ago is returned as if it were current. [1](#0-0) [2](#0-1) 

The `getPriceUnsafe` function itself is documented to make no recency guarantees: [3](#0-2) 

---

### Impact Explanation

Any DeFi protocol (lending, derivatives, stablecoin) that uses `PythAggregatorV3` as a Chainlink drop-in replacement will receive stale prices without any revert or warning. An attacker can:

1. Wait for the Pyth price feed to go stale (e.g., keeper outage, market closure, network congestion).
2. Interact with the victim protocol while it reads a stale price via `latestRoundData()` or `latestAnswer()`.
3. Exploit the price discrepancy — e.g., borrow against overvalued collateral, avoid liquidation on undervalued debt, or arbitrage against the stale price.

The impact is **loss of funds** for the protocol and its users. The severity is amplified because `PythAggregatorV3` is Pyth's official migration adapter, so many protocols will deploy it without auditing the staleness behavior.

---

### Likelihood Explanation

- `PythAggregatorV3` is the officially documented Chainlink migration path, making it widely deployed.
- Pyth is a pull oracle; price feeds only update when someone calls `updatePriceFeeds()`. If the keeper is down or the subscription runs out of funds, prices go stale.
- The Pyth documentation explicitly warns about stale prices and adversarial selection, but `PythAggregatorV3` provides no protection.
- No privileged access is required; any unprivileged transaction sender can trigger the exploit by interacting with a protocol that uses the adapter. [4](#0-3) 

---

### Recommendation

Replace all `getPriceUnsafe()` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`, where `maxAge` is a configurable parameter set at construction time. Add a `maxAge` state variable and expose a setter (owner-only) so deployers can tune the staleness threshold for their use case.

```solidity
contract PythAggregatorV3 {
    bytes32 public priceId;
    IPyth public pyth;
    uint public maxAge; // e.g., 3600 seconds

    constructor(address _pyth, bytes32 _priceId, uint _maxAge) {
        priceId = _priceId;
        pyth = IPyth(_pyth);
        maxAge = _maxAge;
    }

    function latestAnswer() public view virtual returns (int256) {
        PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
        require(price.price > 0, "Invalid non-positive price");
        return int256(price.price);
    }

    function latestRoundData() external view returns (
        uint80 roundId, int256 answer, uint256 startedAt,
        uint256 updatedAt, uint80 answeredInRound
    ) {
        PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
        require(price.price > 0, "Invalid non-positive price");
        roundId = uint80(price.publishTime);
        return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
    }
    // ... same for getRoundData, decimals, etc.
}
```

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a price feed (e.g., ETH/USD).
2. Call `updateFeeds()` to push a fresh price at time `T`.
3. Wait `N` hours without calling `updateFeeds()` again (simulating keeper downtime).
4. Call `latestRoundData()` — it returns the price from time `T` with no revert, despite being `N` hours stale.
5. A lending protocol reading this price will use the `T`-era price for collateral valuation, allowing an attacker to borrow against a price that no longer reflects market reality.

```solidity
// Foundry PoC
function testStalePriceNoRevert() public {
    // Deploy adapter
    PythAggregatorV3 adapter = new PythAggregatorV3(address(pyth), ETH_USD_FEED_ID);

    // Push a fresh price
    adapter.updateFeeds{value: fee}(updateData);

    // Advance time by 24 hours (no keeper update)
    skip(86400);

    // latestRoundData() returns the 24-hour-old price with NO revert
    (, int256 answer,,,) = adapter.latestRoundData();
    // answer is 24 hours stale — protocol will use it as if current
    assertTrue(answer > 0); // succeeds, no StalePrice revert
}
