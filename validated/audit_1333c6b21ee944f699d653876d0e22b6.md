### Title
Incorrect `total_tx_size`/`total_tx_cycles` Accounting in `add_entry` When Ancestor-Limit Eviction Occurs — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the updated pool-size totals are **pre-computed before** `check_and_record_ancestors` runs, but **written back after** it returns. When `check_and_record_ancestors` evicts transactions to satisfy the ancestor-count limit, those evictions correctly decrement `self.total_tx_size` / `self.total_tx_cycles` via `update_stat_for_remove_tx`. The pre-computed snapshot then **overwrites** those decrements, leaving `total_tx_size` and `total_tx_cycles` permanently inflated by the sizes and cycles of the evicted entries.

---

### Finding Description

`add_entry` in `pool_map.rs` follows this sequence:

```
(1) let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
        // snapshot = current_total + new_entry (evictions not yet known)

(2) evicts = self.check_and_record_ancestors(&mut entry)?;
        // may call remove_entry_and_descendants → remove_entry
        //   → update_stat_for_remove_tx  ← modifies self.total_tx_size in-place

(3) self.total_tx_size  = total_tx_size;   // overwrites step-2 modifications
    self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` only adds the new entry's size/cycles to the **current** totals: [2](#0-1) 

`check_and_record_ancestors` can evict pool entries when the incoming transaction would exceed `max_ancestors_count` but the excess is attributable to `cell_ref_parents`: [3](#0-2) 

Each eviction goes through `remove_entry`, which calls `update_stat_for_remove_tx` and **directly mutates** `self.total_tx_size` / `self.total_tx_cycles`: [4](#0-3) [5](#0-4) 

After `check_and_record_ancestors` returns, the pre-computed snapshot from step (1) is written back unconditionally, erasing the correct decrements. The result:

```
actual pool entries  = old_entries − evicted + new_entry
total_tx_size stored = old_total   + new_entry.size          (evicted sizes NOT subtracted)
correct value        = old_total   − evicted_sizes + new_entry.size
```

`total_tx_size` is inflated by `Σ(evicted_entry.size)` after every such event.

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size()` to decide whether to evict further transactions: [6](#0-5) 

An inflated `total_tx_size` causes `limit_size()` to believe the pool is over capacity and evict **legitimate, fee-paying transactions** that should remain. This degrades pool quality, harms honest users whose transactions are dropped, and can be exploited to selectively clear competing transactions. The incorrect value is also surfaced directly to callers of the `get_tx_pool_info` RPC (`total_tx_size` field). [7](#0-6) 

---

### Likelihood Explanation

The eviction branch in `check_and_record_ancestors` is reachable by any unprivileged transaction sender. The attacker must craft a transaction graph where:

1. A chain of ≥ `max_ancestors_count` (default 25) in-pool transactions exists.
2. One or more of those ancestors are also referenced as `cell_dep` outputs by another in-pool transaction (`cell_ref_parents`).
3. A new transaction is submitted whose total ancestor count exceeds the limit but whose non-`cell_ref_parent` ancestor count does not — triggering the eviction path.

This requires only standard CKB transaction submission (RPC `send_transaction` or P2P relay) and no privileged access. The inflation accumulates across repeated submissions, progressively worsening pool accounting until the pool is cleared on a new block.

---

### Recommendation

Move the stat update to **after** `check_and_record_ancestors` completes, so it reflects the post-eviction state of `self.total_tx_size` / `self.total_tx_cycles`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute and apply totals AFTER evictions have already updated self.total_tx_*
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, call `recompute_total_stat()` at the end of `add_entry` whenever `evicts` is non-empty.

---

### Proof of Concept

```
State: pool contains 25 chained txs T1→T2→…→T25 (total_tx_size = S)
       pool also contains TX_dep (size = D) whose output is used as a
       cell_dep by T25 (making TX_dep a cell_ref_parent of any tx
       spending T25's output).

Step 1: submit TX_new that spends T25's output.
        ancestors_count = 26 > max_ancestors_count (25)
        cell_ref_parents = {TX_dep}, so 26 - 1 = 25 ≤ 25 → eviction branch taken.

Step 2: check_and_record_ancestors evicts TX_dep:
        remove_entry(TX_dep) → update_stat_for_remove_tx(D, …)
        self.total_tx_size = S − D   ← correct intermediate value

Step 3: add_entry writes back pre-computed snapshot:
        self.total_tx_size = S + TX_new.size   ← D is NOT subtracted

Result: total_tx_size is inflated by D (TX_dep's size).
        limit_size() now sees a pool that appears D bytes larger than reality
        and evicts a legitimate pending transaction to compensate.
``` [8](#0-7)

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
