### Title
Unbounded `constructProviderCommitment` Loop When `maxNumHashes = 0` Causes `revealWithCallback`/`reveal` to Always Revert - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The `constructProviderCommitment` function in `Entropy.sol` contains an unbounded `while` loop that iterates `numHashes` times. The `numHashes` value stored per request is the gap between the current sequence number and the provider's last advanced commitment. The `maxNumHashes` parameter is intended to cap this gap, but when `maxNumHashes = 0` (the default for all newly registered providers), the cap check is entirely skipped. If the gap grows large enough, `revealWithCallback` and `reveal` will always revert due to out-of-gas, permanently preventing fulfillment of in-flight requests.

### Finding Description

`constructProviderCommitment` runs a `while` loop proportional to `numHashes`: [1](#0-0) 

`numHashes` is set at request time as `assignedSequenceNumber - currentCommitmentSequenceNumber`: [2](#0-1) 

The guard that enforces a maximum is gated on `maxNumHashes != 0`. When `maxNumHashes == 0` (the default for every newly registered provider), the check is **completely skipped**: [3](#0-2) 

`revealHelper`, called by both `revealWithCallback` and `reveal`, passes the stored `req.numHashes` directly into `constructProviderCommitment`: [4](#0-3) 

`advanceProviderCommitment` (permissionlessly callable by anyone) also calls `constructProviderCommitment` with an unbounded `numHashes`: [5](#0-4) 

The `setMaxNumHashes` function accepts any `uint32` value with no validation, and `0` is explicitly treated as "disable the check": [6](#0-5) 

The keeper's off-chain mitigation (`update_commitments_loop`) also short-circuits when `max_num_hashes == 0`, meaning no automatic commitment advancement occurs: [7](#0-6) 

### Impact Explanation

When `maxNumHashes = 0` and the provider's `currentCommitmentSequenceNumber` falls far behind `sequenceNumber` (e.g., due to high request volume or keeper downtime), new requests are stored with a very large `numHashes`. Each `keccak256` in the loop costs ~30 gas; at `numHashes = 1,000,000` that is ~30M gas, exceeding typical block gas limits. Consequently:

- `revealWithCallback` and `reveal` revert for all requests with large `numHashes`.
- Users who paid fees receive no random number and cannot recover their funds.
- The Entropy service is effectively DoS'd for that provider.

### Likelihood Explanation

- Every provider starts with `maxNumHashes = 0` by default. Providers that do not explicitly call `setMaxNumHashes` are permanently exposed.
- An unprivileged user can spam requests (paying the fee each time) to a provider with `maxNumHashes = 0`, causing `sequenceNumber` to grow while the keeper may not keep pace.
- Organic high-traffic scenarios (no attacker needed) can also trigger this: if the keeper is temporarily down, the gap grows, and requests made during that window get large `numHashes` values that can never be revealed.
- `revealWithCallback` is permissionlessly callable by anyone, so the DoS is triggered by the reveal step, not the request step.

### Recommendation

1. **Enforce a non-zero `maxNumHashes` at registration time**, or set a protocol-level default cap so that `maxNumHashes = 0` does not mean "unlimited."
2. **Add a hard upper bound** on `numHashes` inside `constructProviderCommitment` (e.g., revert if `numHashes > MAX_HASHES`) to prevent out-of-gas reverts regardless of provider configuration.
3. **Validate `maxNumHashes` in `setMaxNumHashes`** to reject values that would allow loops exceeding a safe gas budget.
4. **Remove the `if (provider_info.max_num_hashes == 0) { return Ok(()); }` short-circuit** in the keeper's `update_commitments_if_necessary`, or apply a default threshold when `maxNumHashes` is zero.

### Proof of Concept

1. Provider registers with default `maxNumHashes = 0`.
2. Attacker (or organic users) submit N requests (e.g., N = 500,000), each paying the required fee. Provider's keeper does not advance the commitment.
3. `currentCommitmentSequenceNumber` remains at 0; `sequenceNumber` is now 500,001.
4. The last request has `numHashes = 500,000` stored in `req.numHashes`.
5. Anyone calls `revealWithCallback(provider, 500000, ...)`. Inside `revealHelper`, `constructProviderCommitment(500000, providerContribution)` runs the `while` loop 500,000 times (~15M gas for keccak256 alone), exceeding the block gas limit.
6. The transaction reverts. The request remains in `CALLBACK_NOT_STARTED` state. The user's fee is consumed and the random number is never delivered. [8](#0-7)

    function constructProviderCommitment(
        uint64 numHashes,
        bytes32 revelation
    ) internal pure returns (bytes32 currentHash) {
        currentHash = revelation;
        while (numHashes > 0) {
            currentHash = keccak256(bytes.concat(currentHash));
            numHashes -= 1;
        }
    }
