Audit Report

## Title
`n_sync_started` Counter Permanently Inflated via Peer Disconnect Race in `start_sync_headers` — (File: `sync/src/synchronizer/mod.rs`)

## Summary
In `start_sync_headers`, `n_sync_started` is atomically incremented via `fetch_update` before the peer's state is transitioned to `Started` via a separate `get_mut` call. A peer disconnect arriving between these two non-atomic steps causes `disconnected()` to skip the counter decrement (because `sync_started()` returns `false`), permanently inflating `n_sync_started`. In IBD mode this prevents any future header sync from starting, stalling the node indefinitely until restart.

## Finding Description
In `start_sync_headers` (L670–691), for each candidate peer the code:

1. Calls `n_sync_started().fetch_update(...)` with closure `|x| if ibd && x != 0 { None } else { Some(x + 1) }` — this atomically increments the counter.
2. Separately calls `self.peers().state.get_mut(&peer)` and, only if `Some`, calls `peer_state.start_sync(...)` to set `headers_sync_state = Started`.

These are two independent operations on a `DashMap`. Between step 1 and step 2, a concurrent `SessionClose` event invokes `Peers::disconnected()` (L901–924), which calls `self.state.remove(&peer)`. At that point `peer_state.sync_started()` checks `headers_sync_state == Started` (L107–109) — but since `start_sync` was never called, it returns `false`, and the `fetch_sub` decrement is skipped entirely. Back in `start_sync_headers`, `get_mut` returns `None`, so `start_sync` is never called. The counter is now +1 with no peer in `Started` state and no path to decrement it.

The `DashMap` usage and `AtomicUsize` for `n_sync_started` confirm concurrent access is expected. The test file at L1090 explicitly acknowledges: *"There may be competition between header sync and eviction, it will case assert panic"* — confirming the race class is known.

## Impact Explanation
In IBD mode, the `fetch_update` closure returns `None` (error) whenever `x != 0`, causing the loop to `break` immediately. A permanently inflated `n_sync_started` means `send_getheaders_to_peer` is never called again. The node remains stuck in Initial Block Download indefinitely — it cannot sync headers, cannot advance its chain, and cannot participate in the network as a full node. This constitutes a remotely-triggered, persistent denial-of-sync against the local node, matching **High impact: Vulnerabilities which could easily crash/render non-functional a CKB node**.

## Likelihood Explanation
- No authentication or privilege is required; any inbound or outbound peer can trigger this.
- The attacker only needs to connect and immediately disconnect in a tight loop to maximize the probability of hitting the narrow race window between `fetch_update` and `get_mut`.
- The node's periodic sync timer repeatedly calls `start_sync_headers`, giving the attacker continuous opportunities.
- The race window, while small, is repeatable and the effect is permanent (no self-healing mechanism exists).
- The codebase itself acknowledges the race class in test comments.

## Recommendation
Increment `n_sync_started` only after successfully setting the peer state to `Started`, or roll back the increment if `get_mut` returns `None`:

```rust
// After fetch_update succeeds:
let started = if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
    true
} else {
    false
};

if !started {
    self.shared().state().n_sync_started()
        .fetch_sub(1, Ordering::AcqRel);
    continue;
}
```

Alternatively, hold a `DashMap` write guard across both the counter increment and the state transition to eliminate the window entirely.

## Proof of Concept
1. Attacker peer connects → `sync_connected` adds peer to `self.state` in `SyncProtocolConnected` state.
2. Node's `start_sync_headers` timer fires; peer passes `can_start_sync` filter and is collected into `peers`.
3. `fetch_update` succeeds → `n_sync_started` becomes 1.
4. **Race**: attacker peer disconnects → `Peers::disconnected` calls `state.remove(&peer)`; `sync_started()` returns `false` → counter NOT decremented.
5. `state.get_mut(&peer)` returns `None`; `start_sync` is never called.
6. `n_sync_started` is permanently 1 with no peer in `Started` state.
7. All subsequent `start_sync_headers` calls in IBD: `fetch_update` closure sees `x == 1 != 0`, returns `None`, loop breaks immediately — `send_getheaders_to_peer` is never called.
8. Node is stuck in IBD until restarted.

Reproducible as a unit test: mock two concurrent goroutines/threads — one running `start_sync_headers` and one calling `peers.disconnected(peer)` — and assert that `n_sync_started` returns to 0 after the disconnect. The current code will fail this assertion.
