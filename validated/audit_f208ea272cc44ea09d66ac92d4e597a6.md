Audit Report

## Title
Missing Per-Peer Rate Limiter on BlockFilter Protocol Enables I/O Amplification DoS — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_hashes_process.rs`)

## Summary
The `BlockFilter` protocol handler exposes `GetBlockFilterHashes`, `GetBlockFilters`, and `GetBlockFilterCheckPoints` with no per-peer rate limiting. Any connected peer can repeatedly send `GetBlockFilterHashes(start_number=0)`, each triggering up to 4,000 RocksDB point-reads and producing a ~64 KB response. The `Relayer` handler has an explicit `governor`-based per-peer rate limiter; `BlockFilter` has none, confirmed by direct code inspection.

## Finding Description
`GetBlockFilterHashesProcess::execute` defines `BATCH_SIZE = 2000` and iterates up to that many times, performing two sequential store lookups per iteration (`get_block_hash` + `get_block_filter_hash`), totaling up to 4,000 RocksDB point-reads per message and producing a response of up to 2,000 × 32 = 64 KB.

The `BlockFilter` struct carries no rate-limiter field:
```rust
// sync/src/filter/mod.rs, lines 22-25
pub struct BlockFilter {
    shared: Arc<SyncShared>,
}
```

The `try_process` dispatch path performs no rate check before calling `execute` on any of the three request variants. By contrast, `Relayer` carries a `RateLimiter<(PeerIndex, u32)>` (lines 63–67, 81 of `sync/src/relayer/mod.rs`) and rejects requests exceeding 30 req/s per peer per message type before any dispatch (lines 116–123). The `disconnected` handler in `Relayer` also calls `rate_limiter.retain_recent()` (line 934) to bound memory; `BlockFilter::disconnected` does nothing analogous. A search over `sync/src/filter/` returns zero matches for `rate_limit`, `RateLimiter`, `governor`, or `TooManyRequests`.

## Impact Explanation
With N peers (up to `max_peers`, typically 125) each sending `GetBlockFilterHashes(start_number=0)` in a tight loop: N × 4,000 RocksDB reads per cycle (500,000 reads/round at 125 peers) easily saturates spinning disks and significantly loads SSDs. Outbound bandwidth reaches N × 64 KB per round (8 MB/round at 125 peers). The sync and relay protocols share the same async executor and RocksDB instance, so saturating either resource delays block/header propagation and transaction relay, degrading consensus participation and potentially crashing the node under sustained load. This matches: **High — Vulnerabilities which could easily crash a CKB node** and **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The attack path is entirely unprivileged and reachable from the public P2P interface. The attacker needs only standard TCP connectivity to the node's P2P port — no PoW, no stake, no authentication, and no victim mistake is required. The cost is negligible: open N TCP connections and send a small fixed message in a tight loop. The asymmetry with `Relayer` (which has an explicit rate limiter added intentionally) confirms the omission in `BlockFilter` is unintentional. The attack is repeatable and deterministic.

## Recommendation
Apply the same `governor`-based per-peer rate limiter pattern already used in `Relayer` to the `BlockFilter` handler:

1. Add a `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter`, mirroring `sync/src/relayer/mod.rs` lines 63–67 and 81.
2. In `BlockFilter::try_process`, check the limiter keyed by `(peer, message.item_id())` before dispatching any of the three request variants, returning `StatusCode::TooManyRequests` on limit breach.
3. Call `rate_limiter.retain_recent()` in the `disconnected` handler to bound memory growth.
4. Consider a global concurrency cap on in-flight Filter requests to bound aggregate RocksDB read pressure regardless of peer count.
