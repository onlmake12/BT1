### Title
Unchecked `u128`ŌåÆ`u64` Cast in `calculate_maximum_withdraw` Silently Truncates DAO Withdrawal Amount - (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` uses an unchecked `as u64` cast on the intermediate `u128` result `withdraw_counted_capacity`. Every other analogous arithmetic operation in the same module uses the checked `u64::try_from(...)` conversion that returns `DaoError::Overflow` on overflow. The inconsistency means that if `withdraw_counted_capacity` exceeds `u64::MAX`, the value is silently truncated to a wrong (too-small) number instead of returning an error, producing an incorrect maximum-withdrawal figure. This is the direct analog of the external report's "formula implementation does not match the spec" class: the code diverges from the correct checked-arithmetic pattern used everywhere else in the same file.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ŌåÉ unchecked cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

Every other `u128`ŌåÆ`u64` narrowing in the same file uses the checked form:

```rust
// miner_issuance
u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?
// ar_increase
u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?
// secondary_block_reward
u64::try_from(reward128).map_err(|_| DaoError::Overflow)?
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The existing overflow test `check_withdraw_calculation_overflows` only exercises the path where `safe_add(occupied_capacity)` overflows ŌĆö it does **not** exercise the path where `withdraw_counted_capacity` itself exceeds `u64::MAX` but the subsequent `safe_add` does not overflow, leaving the truncated (wrong) value silently accepted. [5](#0-4) 

The function is called both from the public RPC (`calculate_dao_maximum_withdraw`) and from `transaction_maximum_withdraw`, which feeds `transaction_fee` used during tx-pool admission of DAO withdrawal transactions. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Two concrete consequences:

1. **Wrong withdrawal amount returned to the user.** `calculate_dao_maximum_withdraw` (RPC) and the on-chain DAO script both rely on this value. A truncated result means a depositor is told they can withdraw less than they are entitled to, or the node rejects a valid withdrawal transaction because `transaction_fee = maximum_withdraw - outputs_capacity` underflows (appears negative) when `maximum_withdraw` is truncated below `outputs_capacity`.

2. **Denial-of-service against DAO withdrawals.** Because `transaction_fee` calls `transaction_maximum_withdraw` ŌåÆ `calculate_maximum_withdraw`, a truncated result causes `safe_sub` to return `DaoError::Overflow`, causing the tx-pool to reject a legitimate DAO withdrawal transaction submitted by any user. [7](#0-6) 

---

### Likelihood Explanation

For `withdraw_counted_capacity` to exceed `u64::MAX`, the accumulate-rate ratio `withdrawing_ar / deposit_ar` must be large enough that `counted_capacity ├Ś (withdrawing_ar / deposit_ar) > u64::MAX`. Since `ar` starts at `10^16` and grows by `ar ├Ś g2 / C` per block (a very small fraction), this requires many years of chain operation. The likelihood is therefore **low in the near term** but the defect is a real code inconsistency that will eventually become reachable as the chain matures, and it is already a correctness bug relative to the checked-arithmetic contract established by every other calculation in the same module.

---

### Recommendation

Replace the unchecked cast with the same checked pattern used everywhere else in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (consistent with rest of module):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
```

Also add a test case where `withdraw_counted_capacity` itself (before `safe_add`) exceeds `u64::MAX` to cover the truncation path that the existing overflow test misses.

---

### Proof of Concept

Construct a scenario where `ar` has grown to approximately `2 ├Ś deposit_ar`:

```
counted_capacity  = u64::MAX - 1  = 18_446_744_073_709_551_614
deposit_ar        = 10_000_000_000_000_000   (genesis ar = 10^16)
withdrawing_ar    = 20_000_000_000_000_000   (ar doubled)

withdraw_counted_capacity (u128)
  = (u64::MAX - 1) ├Ś 20_000_000_000_000_000 / 10_000_000_000_000_000
  = (u64::MAX - 1) ├Ś 2
  = 36_893_488_147_419_103_228   ŌåÉ exceeds u64::MAX

withdraw_counted_capacity as u64
  = 36_893_488_147_419_103_228 mod 2^64
  = 18_446_744_073_709_551_612   ŌåÉ silently truncated, 2 shannons short

If occupied_capacity is small (e.g. 4_100_000_000 shannons for a minimal cell),
safe_add succeeds and returns a wrong (too-small) withdraw_capacity
instead of returning DaoError::Overflow.
```

The attacker-controlled entry path is any unprivileged RPC caller invoking `calculate_dao_maximum_withdraw`, or any transaction sender submitting a DAO phase-2 withdrawal transaction to the tx-pool. [8](#0-7) [6](#0-5)

    /// Calculate maximum withdraw capacity of a deposited dao output
    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
            DaoWithdrawingCalculationKind::WithdrawingOutPoint(withdrawing_out_point) => {
                let (_tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
