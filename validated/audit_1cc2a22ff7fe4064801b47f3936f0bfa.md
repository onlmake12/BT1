### Title
BLOCK_INVALID Status Not Checked in `HeaderAcceptor::accept` — (`sync/src/synchronizer/headers_process.rs`)

### Summary
`HeaderAcceptor::accept()` retrieves a header's `BlockStatus` but only checks for `BLOCK_INVALID` in a developer-acknowledged `FIXME` comment. When a header is already marked `BLOCK_INVALID` (e.g., because its block body failed contextual verification), the function falls through all three sub-checks and calls `sync_shared.insert_valid_header()`, re-accepting the header as valid. An unprivileged P2P peer can exploit this to corrupt the node's best-known-header tracking.

### Finding Description

In `HeaderAcceptor::accept()`, the node fetches the current `BlockStatus` for the incoming header:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    ...
    return result;
}
``` [1](#0-0) 

The code only short-circuits on `HEADER_VALID`. If `status == BLOCK_INVALID`, execution falls through to `prev_block_check`, `non_contextual_check`, and `version_check`. A block that failed **contextual** verification (invalid scripts, capacity errors, etc.) will have a header that passes all three of those checks — valid PoW, valid parent hash, version 0 — and the function reaches:

```rust
sync_shared.insert_valid_header(self.peer, self.header);
``` [2](#0-1) 

This inserts the previously-invalid header into the `header_map` and calls `may_set_best_known_header`, updating the peer's tracked chain tip to a block the node already knows is invalid.

The `BLOCK_INVALID` flag is set in `block_status_map` (an in-memory `DashMap`) when contextual verification fails:

```rust
self.shared
    .insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
``` [3](#0-2) 

The `BlockStatus` bitflag definitions confirm `BLOCK_INVALID` is disjoint from `HEADER_VALID`:

```rust
const BLOCK_INVALID = 1 << 12;
``` [4](#0-3) 

So `status.contains(BlockStatus::HEADER_VALID)` is `false` when `status == BLOCK_INVALID`, and the guard does not fire.

### Impact Explanation

An attacker who can send `SendHeaders` P2P messages (any unprivileged peer) can:

1. Submit a block with a valid header (valid PoW, valid parent, version 0) but invalid body (e.g., script failure, capacity violation).
2. The victim node stores the block, runs contextual verification, marks it `BLOCK_INVALID`.
3. The attacker re-sends just the header via `SendHeaders`.
4. `HeaderAcceptor::accept()` skips the `BLOCK_INVALID` guard, passes all three sub-checks, and calls `insert_valid_header`.
5. The victim node's `shared_best_header` and per-peer best-known-header are updated to point to the invalid chain tip.

Consequences:
- The node may preferentially sync from the attacker peer, requesting blocks on an invalid chain and wasting resources.
- During IBD, the node uses best-known-header comparisons to decide which outbound peers to keep or disconnect. Corrupting this view can cause the node to disconnect legitimate peers.
- The node's `unverified_tip` tracking and block-fetch scheduling are driven by best-known-header state, so this can stall or misdirect block download.

### Likelihood Explanation

Any peer connected to the victim node can send `SendHeaders` messages. Mining a block with a valid header but invalid body (e.g., a script that always fails) requires only the attacker's own hashpower for one block — no majority hashpower is needed. The `block_status_map` is populated after the first submission, and the re-sent header is processed by `HeadersProcess::execute` on the second `SendHeaders` message. This is a straightforward, low-cost attack.

### Recommendation

At the top of `HeaderAcceptor::accept()`, add an explicit early-return for `BLOCK_INVALID`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    ...
}
```

This resolves the developer-acknowledged `FIXME` and mirrors the pattern already used in `compact_block_process.rs`:

```rust
shared
    .shared()
    .insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
return StatusCode::CompactBlockHasInvalidHeader
    .with_context(format!("{block_hash} {err}"));
``` [5](#0-4) 

### Proof of Concept

1. Attacker connects to victim as a P2P peer.
2. Attacker mines block `B` at height `N+1` with valid PoW/parent/version but a transaction whose lock script always returns error.
3. Attacker sends `B` via `SendBlock`. Victim stores it, runs `ConsumeUnverifiedBlockProcessor::consume_unverified_blocks`, contextual verification fails, `block_status_map[B.hash] = BLOCK_INVALID`.
4. Attacker sends `SendHeaders([B.header])`.
5. `HeadersProcess::execute` calls `HeaderAcceptor::accept(B.header)`.
6. `get_block_status(B.hash)` returns `BLOCK_INVALID`. The guard `status.contains(HEADER_VALID)` is `false` — no early return.
7. `prev_block_check` passes (parent is valid). `non_contextual_check` passes (PoW/timestamp valid). `version_check` passes (version 0).
8. `sync_shared.insert_valid_header(peer, B.header)` is called — the invalid header is now tracked as the peer's best known header. [6](#0-5)
