### Title
`get_block_economic_state` Uses Shallow Protocol Delay (11 Blocks) as "Finalization" Threshold, Offering No Reorg Guarantees - (`rpc/src/module/chain.rs`)

---

### Summary

The `get_block_economic_state` RPC uses `finalization_delay_length()` — only **11 blocks** on mainnet — as the threshold to declare a block's economic state "finalized" and return it, including a `finalized_at` field. Because CKB is a PoW chain with no BFT finality, this 11-block depth offers no meaningful reorg protection. Any downstream system (miner payout processor, exchange deposit confirmation, keeper) that treats a non-null response as "safe to act on" is exposed to incorrect financial decisions if a reorg deeper than 11 blocks occurs.

---

### Finding Description

In `rpc/src/module/chain.rs`, the `get_block_economic_state` implementation computes:

```rust
let delay_length = snapshot.consensus().finalization_delay_length();
let finalized_at_number = block_number + delay_length;
if block_number == 0 || snapshot.tip_number() < finalized_at_number {
    return Ok(None);
}
``` [1](#0-0) 

`finalization_delay_length()` is defined as `tx_proposal_window.farthest() + 1`: [2](#0-1) 

With the mainnet default `TX_PROPOSAL_WINDOW = ProposalWindow(2, 10)`, this yields `10 + 1 = 11 blocks`. [3](#0-2) 

The response includes a `finalized_at` field (the hash of block `N + 11`) that is named and documented as if it represents true finality: [4](#0-3) 

The RPC documentation reinforces this framing:

> "Because of the delay, this RPC returns null if the block rewards are not finalized yet." [5](#0-4) 

The term "finalized" here refers only to the **protocol-level reward commitment delay** (the cellbase of block N+11 pays out the reward for block N), not to any reorg-safety guarantee. CKB is a Nakamoto PoW chain with no BFT finality mechanism. A reorg deeper than 11 blocks is theoretically achievable and would invalidate the returned `miner_reward`, `txs_fee`, and `issuance` values.

For comparison, `CELLBASE_MATURITY` — the threshold for spending cellbase outputs — is set to **4 epochs** (~4,000 blocks), explicitly to reduce reorg risk: [6](#0-5) 

No equivalent depth protection exists for `get_block_economic_state`.

---

### Impact Explanation

An RPC caller — such as a miner payout system, an exchange crediting miner rewards, or an automated keeper — that uses `get_block_economic_state` returning non-null as a signal that reward data is safe to act on can be deceived. After a reorg deeper than 11 blocks:

- The `miner_reward` values change (different transactions committed in the reorged block).
- The `txs_fee` values change.
- The `finalized_at` block hash no longer exists on the canonical chain.

Any funds distributed or actions taken based on the pre-reorg response are based on incorrect data. The `finalized_at` field name specifically encourages callers to treat this as a finality signal.

---

### Likelihood Explanation

CKB mainnet uses Nakamoto PoW consensus. A reorg of 11 blocks requires an attacker to temporarily outpace the honest chain by 11 blocks — achievable with a sustained hashrate advantage well below 51% (selfish mining strategies can cause reorgs with ~33% hashrate). The threshold of 11 blocks is significantly shallower than the 30-block threshold flagged as insufficient in the reference report, and far shallower than the 4-epoch cellbase maturity used for actual fund safety in the same codebase. The misleading `finalized_at` naming increases the probability that callers will treat this as a strong finality signal.

---

### Recommendation

1. Rename `finalized_at` to `reward_committed_at` or `reward_locked_at` to accurately reflect that this is a protocol reward-commitment event, not a reorg-safety guarantee.
2. Add explicit documentation to `get_block_economic_state` stating that the returned data is subject to reorg and that callers requiring reorg safety should wait for additional confirmations beyond `finalization_delay_length`.
3. Consider adding an optional `min_confirmations` parameter to the RPC so callers can specify their own reorg-safety depth.
4. Align the RPC's safety framing with `CELLBASE_MATURITY` (4 epochs), which is the existing protocol-level reorg-safety threshold for funds.

---

### Proof of Concept

1. Attacker controls ~35% of CKB hashrate and begins selfish mining from block N.
2. Honest chain advances to block N+11. An RPC caller queries `get_block_economic_state(hash_of_block_N)` and receives a non-null response with `finalized_at = hash_of_block_N+11`, `miner_reward = X CKB`, `txs_fee = Y CKB`.
3. The caller's payout system treats this as "finalized" and credits the miner X+Y CKB.
4. The attacker releases their private chain (length > N+11), causing a reorg. Block N on the new canonical chain has different committed transactions and a different miner.
5. A second call to `get_block_economic_state(hash_of_block_N)` now returns `null` (block N is no longer on the canonical chain) or returns different values.
6. The payout system has already acted on stale data; the credited funds do not correspond to the canonical chain state.

The entry path is a standard unprivileged RPC call to `get_block_economic_state`, which is publicly accessible to any RPC caller. [7](#0-6)

    fn get_block_economic_state(&self, block_hash: H256) -> Result<Option<BlockEconomicState>> {
        let snapshot = self.shared.snapshot();

        let block_number =
            if let Some(block_number) = snapshot.get_block_number(&(&block_hash).into()) {
                block_number
            } else {
                return Ok(None);
            };

        let delay_length = snapshot.consensus().finalization_delay_length();
        let finalized_at_number = block_number + delay_length;
        if block_number == 0 || snapshot.tip_number() < finalized_at_number {
            return Ok(None);
        }

        let block_hash = (&block_hash).into();
        let finalized_at = if let Some(block_hash) = snapshot.get_block_hash(finalized_at_number) {
            block_hash
        } else {
            return Ok(None);
        };

        let issuance = if let Some(issuance) = snapshot
            .get_block_epoch_index(&block_hash)
            .and_then(|index| snapshot.get_epoch_ext(&index))
            .and_then(|epoch_ext| {
                let primary = epoch_ext.block_reward(block_number).ok()?;
                let secondary = epoch_ext
                    .secondary_block_issuance(
                        block_number,
                        snapshot.consensus().secondary_epoch_reward(),
                    )
                    .ok()?;
                Some(core::BlockIssuance { primary, secondary })
            }) {
            issuance
        } else {
            return Ok(None);
        };

        let txs_fee = if let Some(txs_fee) =
            snapshot.get_block_ext(&block_hash).and_then(|block_ext| {
                block_ext
                    .txs_fees
                    .iter()
                    .try_fold(core::Capacity::zero(), |acc, tx_fee| acc.safe_add(*tx_fee))
                    .ok()
            }) {
            txs_fee
        } else {
            return Ok(None);
        };

        Ok(snapshot.get_block_header(&block_hash).and_then(|header| {
            RewardCalculator::new(snapshot.consensus(), snapshot.as_ref())
                .block_reward_for_target(&header)
                .ok()
                .map(|(_, block_reward)| core::BlockEconomicState {
                    issuance,
                    miner_reward: block_reward.into(),
                    txs_fee,
                    finalized_at,
                })
                .map(Into::into)
        }))
    }
```
