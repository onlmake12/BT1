### Title
Gas Griefing via Malicious `_entropyCallback` in Legacy `revealWithCallback` Path — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains a legacy execution path (triggered when `req.gasLimit10k == 0`) that invokes the requester's `_entropyCallback` with no gas cap and no `try/catch` protection. A malicious requester can implement a callback that consumes all forwarded gas and reverts, causing the entire `revealWithCallback` transaction to revert. Because `clearRequest` is called before the callback (checks-effects-interactions), the revert undoes the request clearing, leaving the request permanently active. The provider (who pays gas) can be griefed on every fulfillment attempt indefinitely.

---

### Finding Description

`revealWithCallback` has two execution branches:

**Branch 1 (new path, `req.gasLimit10k != 0`):** Uses `excessivelySafeCall` with a bounded gas limit and handles failures gracefully via a `CALLBACK_FAILED` state.

**Branch 2 (legacy path, `req.gasLimit10k == 0`):** Calls the requester's callback directly with no gas cap and no error handling:

The legacy path is reachable via the old `requestWithCallback` function (which does not set `gasLimit10k`) or via `requestV2` when the provider's `defaultGasLimit` is 0. The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` explicitly confirms that a reverting callback causes the entire `revealWithCallback` to revert: [2](#0-1) 

Because `clearRequest` is called before the callback (line 666), a revert from the callback undoes the request clearing via EVM state rollback. The request remains active and the provider can be griefed on every subsequent fulfillment attempt.

The `testCallbackProvidedGas` test further confirms that when `defaultGasLimit == 0`, the legacy path is used and the callback is unconstrained: [3](#0-2) 

---

### Impact Explanation

- The provider (who submits and pays for `revealWithCallback` transactions) loses gas on every griefed attempt.
- The request is never cleared, so the attacker's position remains in state and the provider is forced to retry indefinitely or abandon the request.
- No provider fees are accrued from the griefed request (fee accounting only completes on successful `clearRequest`).
- A single attacker can register multiple requests via `requestWithCallback` pointing to the same malicious contract, multiplying the griefing surface.
- The only recourse is off-chain blacklisting of the sequence number, which is not enforced on-chain.

---

### Likelihood Explanation

- The legacy path is reachable by any unprivileged user calling `requestWithCallback` (no special role required).
- The attack requires only deploying a malicious contract and paying the Entropy request fee — a low-cost, repeatable operation.
- Providers that have not called `setDefaultGasLimit` (i.e., `defaultGasLimit == 0`) are fully exposed to this path.
- The attack is persistent: because the request is never cleared on revert, the provider will be griefed on every retry.

---

### Recommendation

Apply the same protection used in the new path to the legacy path:

1. **Wrap the legacy callback in a `try/catch`** so that a reverting callback does not propagate and undo `clearRequest`.
2. **Alternatively, deprecate the legacy path entirely** by requiring all new requests to set a non-zero gas limit, and migrating existing zero-gas-limit requests to the new failure-state flow.
3. **At minimum**, add a `gasleft()` check before the callback to ensure the provider has provided sufficient gas, mirroring the `InsufficientGas` revert in the new path.

### Proof of Concept

```solidity
contract MaliciousEntropyConsumer is IEntropyConsumer {
    IEntropy entropy;

    constructor(address _entropy) {
        entropy = IEntropy(_entropy);
    }

    // Step 1: Request using legacy path (no gas limit set)
    function attack(address provider, bytes32 userRandom) external payable {
        uint fee = entropy.getFee(provider);
        entropy.requestWithCallback{value: fee}(provider, userRandom);
    }

    // Step 2: Malicious callback — consumes all gas and reverts
    function _entropyCallback(
        uint64 sequenceNumber,
        address provider,
        bytes32 randomNumber
    ) external override {
        uint256 threshold = 5000;
        while (gasleft() > threshold) { /* spin */ }
        revert("grief");
    }

    function getEntropy() internal view override returns (address) {
        return address(entropy);
    }
}
```

When the provider calls `revealWithCallback` for this request:
1. `clearRequest` executes (line 666).
2. `_entropyCallback` is called with no gas cap (line 676).
3. The callback spins until near-OOG, then reverts.
4. The revert propagates, undoing `clearRequest`.
5. The request remains active; the provider lost gas.
6. The provider retries → step 1 repeats. [4](#0-3)
