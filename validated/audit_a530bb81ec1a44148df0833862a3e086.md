### Title
Entropy Users Cannot Specify a Maximum Fee, Enabling Provider Front-Running to Extract User ETH Buffers - (File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol)

---

### Summary

`Entropy.requestHelper()` accepts `msg.value` and enforces `msg.value >= requiredFee`, but provides no `maxFee` parameter. Any ETH above `providerFee` is permanently credited to `accruedPythFeesInWei`, and any ETH up to `providerFee` is credited to the provider. Because providers can call `setProviderFee()` at any time with no delay, a malicious provider can front-run a user's pending request transaction to raise their fee to exactly the user's `msg.value`, capturing the user's safety buffer.

---

### Finding Description

In `requestHelper()`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value` is consumed — excess above `requiredFee` goes to `accruedPythFeesInWei`, and the provider captures their full `providerFee`. The NatSpec explicitly acknowledges this: *"excess value is not refunded to the caller."* [2](#0-1) 

Providers can update their fee at any time via `setProviderFee()` with immediate effect:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
``` [3](#0-2) 

Provider registration is permissionless — anyone can call `register()` and set any fee: [4](#0-3) 

Because `getFeeV2` reads live on-chain state, the fee a user observes off-chain can differ from the fee charged when the transaction lands. There is no `maxFee` guard the user can set to bound their exposure. [5](#0-4) 

---

### Impact Explanation

A malicious provider can extract the ETH buffer users send above the quoted fee:

1. Provider registers with a competitive low fee (e.g., 90 wei; `requiredFee` = 100 wei with 10 wei Pyth fee).
2. User queries `getFee(provider)` = 100 wei, sends `msg.value = 150 wei` as a safety buffer.
3. Provider sees the pending transaction in the mempool and front-runs it with `setProviderFee(140)`, making `requiredFee` = 150 wei.
4. User's transaction executes: provider accrues 140 wei, Pyth accrues 10 wei. User paid 50 wei more than the quoted price.
5. Provider calls `setProviderFee(90)` to restore the original fee and repeat.

The user has no recourse — there is no `maxFee` parameter to bound the acceptable fee, and no refund mechanism for excess ETH.

---

### Likelihood Explanation

- Provider registration is permissionless; any attacker can register.
- `setProviderFee()` takes effect immediately with no timelock or delay.
- Users routinely send buffers above the quoted fee to avoid reverts from minor fee fluctuations (a pattern the SDK encourages by advising callers to re-query the fee before each call).
- Mempool monitoring is standard for MEV bots on all EVM chains where Entropy is deployed.

---

### Recommendation

Add a `maxFee` parameter to `request()`, `requestWithCallback()`, and `requestV2()`:

```solidity
function requestV2(
    address provider,
    bytes32 userContribution,
    uint32 gasLimit,
    uint128 maxFee          // <-- new
) public payable override returns (uint64) {
    uint128 requiredFee = getFeeV2(provider, gasLimit);
    if (requiredFee > maxFee) revert EntropyErrors.FeeTooHigh();
    ...
}
```

Additionally, refund excess `msg.value` above `requiredFee` to `msg.sender` rather than silently crediting it to `accruedPythFeesInWei`.

---

### Proof of Concept

```solidity
// Attacker is a registered provider with feeInWei = 90 wei, pythFee = 10 wei
// requiredFee = 100 wei

// Step 1: User queries fee off-chain: getFeeV2(attacker, 0) = 100 wei
// Step 2: User submits tx with msg.value = 150 wei (50 wei buffer)

// Step 3: Attacker front-runs in same block:
entropy.setProviderFee(140);  // requiredFee now = 150 wei

// Step 4: User's tx lands:
//   providerFee = 140, accruedPythFeesInWei += 10
//   attacker.accruedFeesInWei += 140  (gained 50 extra wei)

// Step 5: Attacker restores fee:
entropy.setProviderFee(90);

// Net: user paid 150 wei for a service quoted at 100 wei, with no recourse.

        if (_state.providers[provider].sequenceNumber == 0)

        // Assign a sequence number to the request
        uint64 assignedSequenceNumber = providerInfo.sequenceNumber;

    // Set provider fee. It will revert if provider is not registered.
    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
