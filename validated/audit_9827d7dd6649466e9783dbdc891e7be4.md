### Title
Entropy Provider Fees Credited Before Fulfillment With No Refund Mechanism — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

In `Entropy.sol`, the provider's fee is credited to `providerInfo.accruedFeesInWei` immediately at request time — before the provider has revealed the random number. No refund or cancellation path exists for users whose requests are never fulfilled. A provider that goes offline (or is malicious) after collecting fees leaves users with permanently lost funds and no recourse, directly analogous to the whitelisted maker that cannot be liquidated accumulating bad debt against traders.

### Finding Description

In `requestHelper`, the provider fee accounting occurs unconditionally at request time:

```solidity
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;                          // line 237
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The provider can immediately withdraw these accrued fees via `withdraw()` or `withdrawAsFeeManager()`:

```solidity
providerInfo.accruedFeesInWei -= amount;
(bool sent, ) = msg.sender.call{value: amount}("");
``` [2](#0-1) 

There is no function in the contract that allows a user to cancel an unfulfilled request and recover their fee. The `reveal` / `revealWithCallback` path is the only way to resolve a request, and it requires the provider's secret pre-image — which only the provider knows. If the provider never calls `revealWithCallback`, the request sits in storage indefinitely and the user's ETH is gone.

The `Echo.sol` contract (the newer push-oracle callback system) contains an explicit TODO acknowledging the same missing mechanism:

```solidity
// TODO: there also needs to be some penalty mechanism in case the expected provider
// doesn't execute the callback. This should take funds from the expected provider
// and give to providerToCredit.
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [3](#0-2) 

This confirms the design gap is recognized but unresolved across both the Entropy and Echo subsystems.

### Impact Explanation

A user who calls `requestV2()` or `requestWithCallback()` pays a fee that is immediately credited to the provider. If the provider:

- Goes offline permanently (e.g., Fortuna service failure),
- Exhausts their committed hash chain and stops rotating,
- Is malicious and deliberately withholds the reveal,

…the user's ETH is permanently locked in the provider's `accruedFeesInWei` balance (or already withdrawn). The user cannot close/cancel the request, cannot recover their fee, and cannot obtain the random number. Any downstream contract logic that depends on the callback (e.g., NFT mint, lottery resolution, game outcome) is permanently blocked.

This is structurally identical to the whitelisted maker scenario: the privileged entity (provider) collects value upfront, cannot be penalized for non-performance, and the counterparty (user) suffers an irrecoverable loss.

**Impact**: Direct loss of user funds (ETH paid as fees) + permanent denial of service for any contract awaiting the callback.

### Likelihood Explanation

- Any registered provider can stop operating at any time; registration is permissionless.
- The default provider (Fortuna) is a centralized off-chain service — any outage, key loss, or deliberate shutdown leaves all in-flight requests permanently unfulfilled.
- A malicious provider can register, attract requests (possibly with a low fee to maximize volume), collect fees, withdraw them, and then go silent — all without any on-chain penalty.
- The `OutOfRandomness` guard only prevents *new* requests once the sequence is exhausted; it does not protect *existing* in-flight requests. [4](#0-3) 

### Recommendation

1. **Escrow provider fees** — hold the provider fee in a per-request escrow rather than crediting `accruedFeesInWei` immediately. Release to the provider only upon successful `revealWithCallback`.
2. **Add a user-callable refund path** — after a configurable timeout (e.g., N blocks), allow the requester to cancel the request and recover the escrowed provider fee.
3. **Implement a slashing/penalty mechanism** — as the Echo TODO already notes, a time-scaled penalty that transfers escrowed funds to whoever eventually fulfills the request (or back to the user) would align incentives.

### Proof of Concept

```
1. Provider registers with a small fee (e.g., 1 wei) to attract users.
2. User calls requestV2{value: fee}(provider, userRandom, gasLimit).
   → providerInfo.accruedFeesInWei += providerFee  (credited immediately)
3. Provider calls withdraw() to drain their accrued balance.
4. Provider goes silent — never calls revealWithCallback.
5. User's request remains in CALLBACK_NOT_STARTED state forever.
6. User has no function to call to recover their ETH.
7. Any contract awaiting the entropyCallback is permanently blocked.
```

The fee credit at step 2 and the absence of any cancellation/refund function are both verifiable directly in `Entropy.sol` `requestHelper` (line 237) and the contract's public interface, which exposes `reveal`/`revealWithCallback` as the only request-resolution paths with no user-initiated cancellation counterpart.
