Audit Report

## Title
Unbounded O(L) Work in `GetLastStateProofProcess::execute` via `last_n_blocks=0` + Low `difficulty_boundary` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The per-message work guard in `GetLastStateProofProcess::execute` checks only the *input* fields `difficulties.len() + last_n_blocks * 2`, but the actual number of blocks processed by `complete_headers` is determined by `last_n_numbers.len()`, which in the else-branch equals `last_block_number - difficulty_boundary_block_number`. Sending `last_n_blocks=0` and a `difficulty_boundary` at or below the genesis block's total difficulty causes `last_n_numbers` to span the entire chain (0..L), while the guard evaluates to `0 > 1000 = false` and passes. A single malicious peer can force O(L) `get_ancestor` traversals and O(L) MMR root computations per message, exhausting CPU and RocksDB I/O.

## Finding Description

**Guard bypass (lines 201–205):**
```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```
With `difficulties=[]` and `last_n_blocks=0`, this evaluates to `0 > 1000` → **false**. The guard is bypassed entirely. [1](#0-0) 

**Branch selection (lines 291–292):**
With `start_block_number=0`, `last_n_blocks=0`, and chain height L: `L - 0 <= 0` is false for any L > 0. Execution falls into the else-branch. [2](#0-1) 

**`difficulty_boundary_block_number` resolves to 0 (lines 30–32):**
`get_first_block_total_difficulty_is_not_less_than(0, L, &U256::one())` is called. Since the genesis block's total difficulty is ≥ 1, `start_total_difficulty >= *min_total_difficulty` is immediately true, returning `Some((0, genesis_difficulty))`. So `difficulty_boundary_block_number = 0`. [3](#0-2) 

**Unbounded `last_n_numbers` (lines 313–319):**
`L - 0 < 0` is false (since `last_n_blocks=0`), so no adjustment occurs. `last_n_numbers = (0..L)` → **L elements**. [4](#0-3) 

**`difficulty_boundary_block_number > 0` check (lines 321–346):**
`0 > 0` is false, so the else-branch returns `(Vec::new(), last_n_numbers)` — the full L-element vector is returned unchanged. [5](#0-4) 

**`complete_headers` called with L block numbers (lines 132–154):**
For each of the L block numbers, `complete_headers` calls `snapshot.get_ancestor()` (O(log L) skip-list traversal) and `snapshot.chain_root_mmr(*number - 1).get_root()` (O(log L) MMR computation). Total per-message cost: O(L log L) DB reads. [6](#0-5) 

`GET_LAST_STATE_PROOF_LIMIT = 1000` is defined but never applied to the actual output size. [7](#0-6) 

## Impact Explanation
A single malicious peer can send one structurally valid `GetLastStateProof` message that forces the server to perform O(L) `get_ancestor` traversals and O(L) MMR root computations, where L is the full chain height (mainnet: ~13M+ blocks). Each `get_ancestor` is O(log L) in the skip-list implementation, yielding ~300M+ operations per request. Repeated at high frequency, this exhausts CPU and RocksDB I/O, causing the node to become unresponsive or crash. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation
The attack requires only: (1) the genesis block's total difficulty (public constant, derivable from the genesis block header), and (2) the current tip hash (broadcast by the node via `GetLastState`). Both are trivially obtainable. The message is structurally valid and passes all existing input validation. No PoW, stake, or privileged role is required — any peer that establishes a light-client connection can send this message. The attack is repeatable with no rate limit specific to this message type beyond general peer connection limits.

## Recommendation
After computing `last_n_numbers` (and `sampled_numbers`, `reorg_last_n_numbers`), enforce a bound on the *actual output size* before calling `complete_headers`:

```rust
if sampled_numbers.len() + last_n_numbers.len() + reorg_last_n_numbers.len()
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, cap `last_n_numbers` at `GET_LAST_STATE_PROOF_LIMIT` before collecting, and reject requests where the computed set would exceed the limit. The existing guard at lines 201–205 should also be updated or removed, as it only bounds input parameters and not actual work.

## Proof of Concept
```
1. Connect to a CKB node as a light-client peer.
2. Query the tip block hash H_tip (block number L) via GetLastState.
3. Obtain genesis block's total difficulty D_genesis (public constant from genesis block header).
4. Send GetLastStateProof {
       start_hash:          genesis_hash,
       start_number:        0,
       last_hash:           H_tip,
       last_n_blocks:       0,
       difficulty_boundary: U256::one(),   // <= D_genesis, resolves to block 0
       difficulties:        [],
   }
5. Observe: server performs L get_ancestor() + L mmr.get_root() calls.
6. Assert: number of DB reads >> GET_LAST_STATE_PROOF_LIMIT (1000).
7. Repeat in a tight loop to exhaust node CPU and RocksDB I/O.
```
