Audit Report

## Title
Unbounded `last_n_numbers` allocation bypasses `GET_LAST_STATE_PROOF_LIMIT` in `GetLastStateProofProcess::execute` - (File: `util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The limit check in `execute()` only validates `difficulties.len() + last_n_blocks * 2`, which evaluates to `0` when an attacker sends `last_n_blocks=0` and `difficulties=[]`. In the `else` branch, `difficulty_boundary_block_number` collapses to `start_block_number` when `difficulty_boundary=U256::from(1)`, causing `last_n_numbers` to span `(start_block_number..last_block_number)` — an unbounded range proportional to chain height. No secondary bound check exists before `complete_headers` iterates every entry, performing DB reads and heap allocations per block, leading to OOM crash.

## Finding Description

**Root cause — insufficient limit check (lines 201–205):**

The guard only accounts for `difficulties.len()` and `last_n_blocks * 2`: [1](#0-0) 

With `last_n_blocks=0` and `difficulties=[]`, the expression is `0 + 0 = 0`, which is not `> 1000`, so the check passes unconditionally.

**Branch selection (lines 291–292):**

When `last_block_number > start_block_number` (i.e., the chain has advanced), the condition `last_block_number - start_block_number <= 0` is false, so the `else` branch is taken: [2](#0-1) 

**`difficulty_boundary_block_number` collapses to `start_block_number` (lines 29–36):**

`get_first_block_total_difficulty_is_not_less_than` checks if `start_total_difficulty >= min_total_difficulty` and immediately returns `Some((start_block_number, …))` if true: [3](#0-2) 

Since every mined block has cumulative difficulty ≥ 1, passing `difficulty_boundary=U256::from(1)` guarantees this early return, setting `difficulty_boundary_block_number = start_block_number`.

**Adjustment check does not help (lines 313–316):**

With `last_n_blocks=0`, the condition `last_block_number - start_block_number < 0` is false, so `difficulty_boundary_block_number` is not adjusted: [4](#0-3) 

**Unbounded `last_n_numbers` (lines 318–319):**

`last_n_numbers` becomes `(start_block_number..last_block_number)` — millions of entries on mainnet: [5](#0-4) 

**No bound check before `complete_headers` (lines 350–365):**

`block_numbers` is assembled and passed directly to `complete_headers` with no size assertion: [6](#0-5) 

`complete_headers` performs one `get_ancestor` DB read, one full block fetch, one `chain_root_mmr(n-1).get_root()` DB read, and one `VerifiableHeader` heap allocation per entry: [7](#0-6) 

The constant `GET_LAST_STATE_PROOF_LIMIT = 1000` is never applied to the final `block_numbers` slice: [8](#0-7) 

## Impact Explanation

A single malicious P2P peer can crash the light-client-protocol-server process with OOM. On a chain of height H, the attack allocates O(H) `VerifiableHeader` objects and issues O(H) DB reads in a single request. This matches the **High** impact category: *"Vulnerabilities which could easily crash a CKB node"* (10001–15000 points).

## Likelihood Explanation

CKB mainnet has been live since 2019 with block height in the millions. The attack requires only a TCP connection to a node with the light-client protocol server enabled and a single crafted `GetLastStateProof` message. No authentication, PoW, or stake is required. The attack is trivially repeatable and can be automated.

## Recommendation

After assembling `block_numbers`, assert its length against `GET_LAST_STATE_PROOF_LIMIT` before calling `complete_headers`:

```rust
let block_numbers = reorg_last_n_numbers
    .into_iter()
    .chain(sampled_numbers)
    .chain(last_n_numbers)
    .collect::<Vec<_>>();

if block_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage
        .with_context("too many blocks to prove");
}
```

This enforces the invariant `|block_numbers| ≤ GET_LAST_STATE_PROOF_LIMIT` regardless of how `difficulty_boundary`, `last_n_blocks`, or chain height interact.

## Proof of Concept

```
Preconditions:
  - Node synced to chain height H >> 1 (e.g., H = 2_000_000)
  - Light-client protocol server enabled

Craft GetLastStateProof:
  last_n_blocks       = 0
  difficulties        = []
  difficulty_boundary = U256::from(1)
  start_number        = 1
  start_hash          = hash(block #1)   // valid main-chain block
  last_hash           = hash(tip block)  // valid main-chain tip

Execution path:
  1. Limit check: 0 + 0*2 = 0 ≤ 1000                          → passes
  2. is_main_chain(last_hash)                                   → true
  3. start_hash matches ancestor at start_number                → reorg_last_n_numbers = []
  4. last_block_number - start_block_number = ~2M > 0           → else branch
  5. get_first_block_total_difficulty_is_not_less_than(1, 2M, 1)
       → td[1] ≥ 1 → returns Some((1, td[1]))
       → difficulty_boundary_block_number = 1
  6. 2M - 1 < 0 is false                                       → no adjustment
  7. last_n_numbers = (1..2_000_000)                           → ~2M entries
  8. difficulties = [] → sampled_numbers = []
  9. block_numbers = ~2M entries, no bound check
 10. complete_headers iterates ~2M times                        → OOM crash
```
