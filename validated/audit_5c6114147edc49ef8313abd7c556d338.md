### Title
Unauthenticated `from` Field in `ConnectionRequest` Enables Attacker-Directed NAT Traversal TCP Connections — (`network/src/protocols/hole_punching/component/connection_request.rs`)

### Summary
An unprivileged remote peer can cause a CKB node to initiate outbound TCP connections to arbitrary attacker-chosen IP:port targets by sending two crafted hole-punching protocol messages. The `from` field in `ConnectionRequest` is never validated against the authenticated session peer ID, allowing the attacker to populate `pending_delivered` with any PeerId key mapped to attacker-controlled listen addresses, then trigger NAT traversal to those addresses via a `ConnectionSync` message.

### Finding Description

The `ConnectionRequestProcess::execute()` function parses the `from` field directly from the message payload without ever checking that it matches the authenticated peer ID of the session that sent the message.

In `connection_request.rs`, `from` is taken from the wire message: [1](#0-0) 

The `peer` field stored in `ConnectionRequestProcess` is a `PeerIndex` (session index), used only to route the reply — it is never compared against `content.from`: [2](#0-1) 

When `self_peer_id == content.to`, `respond_delivered` is called with the attacker-supplied `from_peer_id` and `remote_listens` (also attacker-supplied via `listen_addrs`), and inserts them directly into `pending_delivered`: [3](#0-2) 

In `connection_sync.rs`, when a `ConnectionSync` message arrives with `to=local_node` and an empty route, the handler looks up `pending_delivered[content.from]` — again using the unauthenticated `from` field from the wire message — and spawns `try_nat_traversal` to whatever addresses are stored there: [4](#0-3) 

The `HolePunching` struct stores `pending_delivered` as a plain `HashMap<PeerId, PendingDeliveredInfo>` with no binding to the session that created each entry: [5](#0-4) 

### Impact Explanation

The target node initiates real outbound TCP connections (via `try_nat_traversal`) to attacker-chosen `SocketAddr` targets. This turns the node into a TCP port-scanner or connection amplifier against arbitrary hosts. An attacker behind NAT can use this to probe internal network services unreachable from the public internet, or to generate connection-flood traffic toward a victim host using the CKB node as a proxy.

### Likelihood Explanation

Any peer that can establish a P2P connection to the target node can execute this in two messages. No special privileges, no leaked keys, no majority hashpower. The target node's peer ID is public. The rate limiter (`forward_rate_limiter` keyed on `(from, to, item_id)`) allows 1 request per second per `(from, to)` pair, but the attacker can rotate the spoofed `from` PeerId to bypass this, or simply wait one second between attempts.

### Recommendation

In `respond_delivered`, validate that `from_peer_id` matches the authenticated peer ID of the session (`self.peer`). Concretely, resolve `self.peer` (a `PeerIndex`) to its `PeerId` via the peer registry and assert equality with `content.from` before inserting into `pending_delivered`. The same check should be applied in `ConnectionSync::execute()` — verify that `content.from` matches the authenticated peer ID of the session delivering the `ConnectionSync` message.

### Proof of Concept

```
1. Attacker establishes a normal P2P connection to the target node (session S, attacker's real PeerId = A).
2. Attacker sends:
     ConnectionRequest {
       from = <any arbitrary PeerId V>,
       to   = <target node's PeerId>,
       listen_addrs = [/ip4/VICTIM_IP/tcp/VICTIM_PORT],
       max_hops = 1,
       route = [],
     }
3. Target node: self_peer_id == content.to → calls respond_delivered(V, target, [VICTIM_IP:PORT])
   → pending_delivered[V] = ([/ip4/VICTIM_IP/tcp/VICTIM_PORT], now)
   → sends ConnectionRequestDelivered back to session S (ignored by attacker)
4. Attacker sends:
     ConnectionSync {
       from  = <same PeerId V>,
       to    = <target node's PeerId>,
       route = [],
     }
5. Target node: self_peer_id == content.to, route empty
   → listens_info = pending_delivered[V] = [/ip4/VICTIM_IP/tcp/VICTIM_PORT]
   → spawns try_nat_traversal(bind_addr, /ip4/VICTIM_IP/tcp/VICTIM_PORT)
   → target node initiates TCP SYN to VICTIM_IP:VICTIM_PORT
```
