### Title
WeightUnitsFlow Fee Estimator Uses Hardcoded `MAX_BLOCK_BYTES` Constant Instead of Runtime Consensus Parameter, Producing Systematically Wrong Fee Estimates — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

The `WeightUnitsFlow` fee estimation algorithm hardcodes the compile-time constant `MAX_BLOCK_BYTES` (imported from `ckb_chain_spec::consensus`) when computing how much weight is removed from the mempool per block. It never receives or consults the actual runtime `consensus.max_block_bytes` field, which is a configurable per-chain-spec parameter. Any network whose `max_block_bytes` differs from the hardcoded constant will receive systematically wrong fee-rate estimates from the `estimate_fee_rate` RPC, causing transactions to be stuck in the mempool (underestimate) or users to overpay (overestimate).

---

### Finding Description

The `WeightUnitsFlow::do_estimate` function, at the core of the fee estimation loop, computes:

```rust
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
``` [1](#0-0) 

`MAX_BLOCK_BYTES` is a compile-time constant imported from `ckb_chain_spec::consensus`: [2](#0-1) 

However, the `Consensus` struct exposes `max_block_bytes` as a **runtime-configurable field** set from the chain spec `Params`: [3](#0-2) 

This field is populated at node startup from the chain spec and can differ from the hardcoded constant on testnet, devnet, or any custom network: [4](#0-3) 

The `Algorithm` struct is constructed with no consensus reference and receives none at estimation time: [5](#0-4) 

Its `estimate_fee_rate` method only accepts `target_blocks` and `TxPoolEntryInfo` — no snapshot, no consensus: [6](#0-5) 

By contrast, the **fallback** fee estimator in `pool_map.rs` correctly reads the runtime consensus value: [7](#0-6) 

The `WeightUnitsFlow` algorithm is the primary (non-fallback) estimator. Its `removed_weight` term — the simulated per-block drain of mempool weight — is the single most important parameter in the estimation loop. Using the wrong value here directly corrupts the output fee rate.

---

### Impact Explanation

- **Underestimate** (actual `max_block_bytes` < `MAX_BLOCK_BYTES`): The algorithm assumes blocks drain more weight than they actually do, so it returns a fee rate that is too low. Transactions submitted at the estimated rate will not be confirmed within the expected number of blocks and may be stuck in the mempool indefinitely.
- **Overestimate** (actual `max_block_bytes` > `MAX_BLOCK_BYTES`): The algorithm assumes blocks drain less weight than they actually do, so it returns a fee rate that is too high. Users overpay for every transaction.

The `estimate_fee_rate` RPC is the primary fee-guidance mechanism for wallets and dApps. Systematic miscalculation here affects every user of the node's RPC on any network where `max_block_bytes` deviates from the hardcoded constant.

---

### Likelihood Explanation

Any RPC caller — an unprivileged user, wallet, or dApp — can invoke `estimate_fee_rate` and receive the wrong result. The bug is always active on networks (testnet, devnet, custom chains) where `max_block_bytes` is configured differently from the hardcoded constant. No special privileges, keys, or majority hashpower are required to trigger the wrong output. [8](#0-7) 

---

### Recommendation

Pass the runtime `max_block_bytes` from the consensus snapshot into `WeightUnitsFlow::Algorithm` at construction time or at each `estimate_fee_rate` call, mirroring how the fallback estimator reads `self.snapshot.consensus().max_block_bytes()`. Remove the dependency on the compile-time `MAX_BLOCK_BYTES` constant from the estimation loop.

---

### Proof of Concept

1. Start a CKB devnet node with `max_block_bytes` set to a value different from the compile-time `MAX_BLOCK_BYTES` constant (e.g., half the default).
2. Configure the node to use the `WeightUnitsFlow` fee estimator.
3. Fill the mempool with transactions.
4. Call `estimate_fee_rate` via RPC.
5. Observe that the returned fee rate is computed using the wrong `removed_weight` (based on the hardcoded constant), not the actual block capacity. The estimate will be systematically lower than correct, causing submitted transactions to miss their target confirmation window.
