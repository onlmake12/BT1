Audit Report

## Title
Unbounded Relay Amplification via Duplicate RPC Transaction Submission — (File: `tx-pool/src/process.rs`)

## Summary
The `after_process` function unconditionally fires a `TxVerificationResult::Ok` relay event whenever a local RPC submission returns `Reject::Duplicated`, with no rate-limiting or deduplication guard. The relay consumer `send_bulk_of_tx_hashes` then pushes the tx hash to every connected full-relay peer for each such event without a prior known-tx check in the `original_peer = None` branch. An attacker with RPC access can submit the same pooled transaction N times and generate N × P relay broadcasts at negligible cost.

## Finding Description
**Root cause — `after_process`, lines 538–545:**

The `None` (local RPC) arm of `after_process` handles `Err(Reject::Duplicated(_))` by unconditionally calling `send_result_to_relayer(TxVerificationResult::Ok { original_peer: None, tx_hash })`. There is no cooldown, counter, or deduplication check before this send. [1](#0-0) 

**Why the Duplicated path is reachable from the main pool:**

`process_tx`'s early-return guard at lines 409–411 only checks `verify_queue_contains` and `orphan_contains`. A transaction already accepted into the main pool passes both checks, proceeds to `_process_tx`, and hits `check_txid_collision`, which returns `Err(Reject::Duplicated(...))`. This result is then passed to `after_process`. [2](#0-1) 

**Channel send has no back-pressure:**

`send_result_to_relayer` is a non-blocking channel send that only logs on failure. There is no rate-limiting, submission counter, or per-hash cooldown. [3](#0-2) 

**Relay amplification — `send_bulk_of_tx_hashes`, lines 662–669:**

In the `None` branch, the tx hash is pushed into `selected` for every connected peer unconditionally. `mark_as_known_tx` is called only *after* the push and is never checked *before* the push in this branch. Each new relay event from a duplicate submission therefore results in another full broadcast round to all P peers. [4](#0-3) 

`MAX_RELAY_TXS_NUM_PER_BATCH` only controls how many items are drained per tick; it does not prevent the channel from accumulating arbitrarily many events from repeated duplicate submissions. [5](#0-4) 

## Impact Explanation
Each duplicate RPC call costs the attacker only a cheap local round-trip with no PoW, no fee, and no new transaction. Each call injects one `TxVerificationResult::Ok` into the relay channel. The relayer drains up to `MAX_RELAY_TXS_NUM_PER_BATCH` items per tick and sends a `RelayTransactionHashes` message to every full-relay peer. Submitting the same tx N times generates N × P relay messages (P = connected peers), congesting the P2P relay channel and degrading throughput for honest peers. This matches the **High** impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
The RPC endpoint is the standard, documented interface for submitting transactions. Any operator exposing the RPC (wallets, exchanges, public nodes) is vulnerable. The attacker needs no special privilege, no key material, and no new valid transaction — only one transaction already accepted into the pool. The exploit is trivially scriptable with a simple loop of identical RPC calls.

## Recommendation
1. In the `Err(Reject::Duplicated(_))` branch for `remote = None`, suppress the relay broadcast entirely, or gate it behind a per-`tx_hash` cooldown (e.g., only re-broadcast if the last broadcast for this hash was more than N seconds ago).
2. In `send_bulk_of_tx_hashes`, before pushing to `selected` in the `None` branch, check whether the tx hash is already in the known-tx filter and skip if so — mirroring the deduplication that the `Some(peer)` path implicitly relies on.

## Proof of Concept
```python
import requests

RPC = "http://127.0.0.1:8114"
tx = { ... }  # any valid, fee-paying transaction already accepted into the pool

# Step 1: confirm tx is in pool (first submission)
requests.post(RPC, json={"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[tx,"passthrough"]})

# Step 2: submit same tx N times — each returns PoolRejectedDuplicatedTransaction (-1107)
# but internally fires send_result_to_relayer(TxVerificationResult::Ok) each time
for i in range(1000):
    requests.post(RPC, json={"id":i+2,"jsonrpc":"2.0","method":"send_transaction","params":[tx,"passthrough"]})

# Result: relay channel accumulates 1000 Ok events → 1000 × P RelayTransactionHashes
# broadcasts to all connected peers, congesting the P2P relay channel
```
