### Title
Unsafe `as u64` Truncating Cast in `calculate_maximum_withdraw` Silently Corrupts NervosDAO Withdrawal Accounting - (File: `util/dao/src/lib.rs`)

---

### Summary

In `util/dao/src/lib.rs`, the function `calculate_maximum_withdraw` computes an intermediate `u128` result (`withdraw_counted_capacity`) and then narrows it to `u64` using the silent truncating `as u64` cast. Every other analogous `u128`→`u64` narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, which propagates an error on overflow. The inconsistent use of `as u64` here means that if the product exceeds `u64::MAX`, the high bits are silently discarded and a wrong (too small) withdrawal capacity is returned — directly mirroring the M-05 pattern of unsafe downcasting causing wrong accounting.

---

### Finding Description

**Root cause — the unsafe cast:**

In `calculate_maximum_withdraw`:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← unsafe truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The multiplication is correctly widened to `u128` to avoid overflow during the intermediate computation. However, the final narrowing back to `u64` uses `as u64`, which in Rust silently wraps/truncates — it does **not** panic or return an error if `withdraw_counted_capacity > u64::MAX`.

**Contrast with the safe pattern used everywhere else in the same file:**

`secondary_block_reward` (line 204):
```rust
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

`dao_field_with_current_epoch` (line 245):
```rust
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
``` [3](#0-2) 

`dao_field_with_current_epoch` (line 258):
```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [4](#0-3) 

`calculate_maximum_withdraw` is the **only** function in this file that uses the unsafe `as u64` cast for a `u128` intermediate result.

**Overflow condition:**

The formula is:

```
withdraw_counted_capacity = counted_capacity × withdrawing_ar / deposit_ar
```

where:
- `counted_capacity` is a `u64` (cell capacity minus occupied capacity, in shannons)
- `withdrawing_ar` and `deposit_ar` are the NervosDAO accumulate rates (`u64`), with `withdrawing_ar ≥ deposit_ar`

Since `withdrawing_ar ≥ deposit_ar`, the result is always `≥ counted_capacity`. For the result to exceed `u64::MAX ≈ 1.84 × 10¹⁹`, the ratio `withdrawing_ar / deposit_ar` must exceed `u64::MAX / counted_capacity`. With CKB's total supply ≈ 3.36 × 10¹⁸ shannons, the ratio would need to exceed ~5.5×. The initial `ar` is `10_000_000_000_000_000` (10¹⁶) and grows slowly per block. [5](#0-4) 

**Exploit path:**

1. A user deposits a large amount of CKB into NervosDAO via a standard DAO deposit transaction (no privilege required).
2. After a very long lock period during which `ar` has grown sufficiently relative to the deposit-time `ar`, the user submits a withdrawal transaction.
3. `calculate_maximum_withdraw` is called during verification (via `transaction_maximum_withdraw` → `withdrawed_interests` → `dao_field_with_current_epoch`) and also directly via the RPC `calculate_dao_maximum_withdraw`.
4. `withdraw_counted_capacity as u64` silently truncates, returning a value far smaller than the correct withdrawal amount.
5. The on-chain verifier accepts the block with the truncated accounting; the depositor's excess CKB is permanently unrecoverable. [6](#0-5) 

---

### Impact Explanation

- **Wrong capacity accounting**: The returned `withdraw_capacity` is silently smaller than the true entitled amount. The depositor loses the truncated portion permanently — it cannot be withdrawn because the on-chain state records the wrong (lower) maximum.
- **No error surfaced**: Unlike the safe `try_from` pattern, no `DaoError::Overflow` is returned; the caller receives a plausible-looking but incorrect `Capacity` value and proceeds normally.
- **Consensus / RPC divergence**: The RPC endpoint `calculate_dao_maximum_withdraw` also calls this function. Off-chain tooling relying on the RPC would compute a different (correct) value if it independently implements the formula, creating a discrepancy between what users expect and what the chain enforces. [7](#0-6) 

---

### Likelihood Explanation

Low but non-zero, and structurally identical to M-05. The overflow requires `withdrawing_ar / deposit_ar > ~5.5` relative to a maximum-supply deposit. Given the slow growth rate of `ar` from its genesis value of `10^16`, this threshold would only be crossed after an extremely long chain lifetime. However:

- The defect is a **real code inconsistency** — the same file applies the safe pattern to every other analogous computation.
- The M-05 reference finding was kept at medium severity under the same reasoning ("unlikely in practice, but technically real").
- No attacker privilege is required; any NervosDAO depositor is the affected party.

---

### Recommendation

Replace the unsafe `as u64` cast with the same checked pattern used everywhere else in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with rest of file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [8](#0-7)

### Proof of Concept

**Scenario**: Alice deposits `N` shannons into NervosDAO at block `B_deposit` where `deposit_ar = 10^16`. She withdraws at block `B_withdraw` where `withdrawing_ar = 5.5 × 10^16` (ar has grown 5.5×).

```
counted_capacity  = N  (e.g., u64::MAX / 5 ≈ 3.69 × 10^18 shannons)
withdraw_counted_capacity (u128) = N × 5.5 × 10^16 / 10^16
                                 = N × 5.5
                                 ≈ 2.03 × 10^19  >  u64::MAX (1.84 × 10^19)

withdraw_counted_capacity as u64
  = (2.03 × 10^19) mod 2^64
  ≈ 1.9 × 10^18   ← silently truncated, ~10× smaller than correct value
```

`calculate_maximum_withdraw` returns `~1.9 × 10^18 + occupied_capacity` instead of the correct `~2.03 × 10^19 + occupied_capacity`. Alice's withdrawal transaction is validated against the wrong (lower) maximum, and the difference is permanently locked in the contract. [9](#0-8)

    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }
