Audit Report

## Title
Unbounded `GetBlocksProof` Request Rate with O(k·log(N)) MMR DB Reads Enables Low-Rate DoS on Light Client Servers — (`util/light-client-protocol-server/src/lib.rs`)

## Summary
`LightClientProtocol` contains no per-peer rate limiter, unlike `Relayer` and `HolePunching` which both enforce one. Any peer can repeatedly send `GetBlocksProof` messages with up to 1000 valid block hashes, each triggering O(k·log(N)) RocksDB reads via `mmr.gen_proof(items_positions)`, allowing a single attacker to saturate the node's storage I/O and degrade or crash the light client server.

## Finding Description
`LightClientProtocol` holds only `pub shared: Shared` with no rate limiter field. The `received` handler parses the message and immediately calls `try_process` with no rate check of any kind. `try_process` dispatches `GetBlocksProof` directly to `GetBlocksProofProcess::execute`. `execute()` enforces only a size cap of 1000 hashes, then calls `reply_proof` with up to 1000 MMR leaf positions. `reply_proof` calls `mmr.gen_proof(items_positions)` on an MMR backed by the snapshot store. The `MMRStore` implementation for `&Snapshot` issues one RocksDB read per MMR node accessed via `get_header_digest`. Standard MMR proof generation for k leaves in a tree of height N requires O(k·log(N)) node reads. With k=1000 and a mainnet chain of ~10M blocks, each request triggers ~23,000 RocksDB reads. There is no guard preventing a peer from issuing these requests in a tight loop. By contrast, `Relayer` explicitly initializes a per-peer, per-message-type rate limiter at 30 req/sec and rejects excess requests before any processing occurs. `HolePunching` applies the same pattern. `LightClientProtocol` has no equivalent protection.

## Impact Explanation
A single unprivileged peer can drive continuous high-volume RocksDB I/O on the target node. At 10 requests/second with 1000 hashes each, this produces ~230,000 RocksDB reads/second from one connection. This can saturate disk I/O, stall the chain processing pipeline that shares the same RocksDB instance, and render the node unresponsive — matching the allowed impact: **High: Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation
The attack requires only a valid P2P connection to a node with the `LightClient` protocol enabled (listed in the default `support_protocols` config) and 1000 valid main-chain block hashes, which are trivially obtained from any public block explorer or by syncing headers. No PoW, keys, or privileged access are needed. The attack is repeatable indefinitely from a single peer.

## Recommendation
Add a per-peer rate limiter to `LightClientProtocol` following the same pattern as `Relayer`:

```rust
pub struct LightClientProtocol {
    pub shared: Shared,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

In `try_process`, check the rate before dispatching any message handler and return `StatusCode::TooManyRequests` on excess. Additionally, consider reducing `GET_BLOCKS_PROOF_LIMIT` from 1000 or adding a global async work queue with backpressure for MMR proof generation.

## Proof of Concept
1. Connect to a mainnet CKB node with `LightClient` in `support_protocols` via P2P.
2. Collect 1000 valid main-chain block hashes spread across the full chain height (e.g., blocks 0, 10000, 20000, …, 9,990,000) from any public explorer.
3. In a loop, send `GetBlocksProof { last_hash: <current tip>, block_hashes: [1000 hashes] }` as fast as the connection allows.
4. Monitor the node's RocksDB I/O (e.g., via `iostat` or RocksDB statistics); observe I/O saturation and degraded block processing latency.
5. To confirm O(k·log(N)) scaling: benchmark `mmr.gen_proof(k_positions)` on chains of height 100, 10,000, and 1,000,000 and verify read count grows with log(N).

    async fn try_process(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        message: packed::RelayMessageUnionReader<'_>,
    ) -> Status {
        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =
            !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));

        if should_check_rate
            && self
                .rate_limiter
                .check_key(&(peer, message.item_id()))
                .is_err()
        {
            return StatusCode::TooManyRequests.with_context(message.item_name());
        }
