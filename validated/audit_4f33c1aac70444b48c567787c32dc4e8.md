Audit Report

## Title
Unvalidated `observed_addr` Injection Poisons Node's Own Address Advertisement - (File: `network/src/protocols/identify/mod.rs`)

## Summary
The Identify protocol's `received` handler accepts a remote-supplied `observed_addr` and stores it unconditionally in `NetworkState::observed_addrs` without verifying it against the session's actual transport-layer source IP. These stored addresses are appended to the node's own `listen_addrs` and broadcast to every subsequent peer, and also propagated via the hole-punching protocol. A single unprivileged peer can inject an arbitrary public IP into the victim's address advertisement with one message.

## Finding Description

**Root cause:** `process_observed` at [1](#0-0)  passes the remote-supplied `observed: Multiaddr` directly to `add_observed_addr` with no comparison against `context.session.address`. The session's actual source IP is available in scope but never consulted.

`IdentifyCallback::add_observed_addr` at [2](#0-1)  appends the local peer-id if absent and writes unconditionally to `NetworkState::add_observed_addr`.

`NetworkState::add_observed_addr` at [3](#0-2)  inserts the attacker-controlled address directly into the shared `observed_addrs` map keyed by session ID, with no validation.

**Propagation — identify:** `local_listen_addrs` at [4](#0-3)  appends observed addresses whenever the real listen count is below `MAX_RETURN_LISTEN_ADDRS` (10). This result is sent to every new peer in `connected` at [5](#0-4) , filtered only by `is_reachable`, which rejects RFC-1918/loopback but passes any routable public IP.

**Propagation — hole-punching:** `observed_addrs` is also included in `ConnectionRequest` broadcasts at [6](#0-5) , propagating the poisoned address to additional peers without further attacker involvement.

**Existing guards are insufficient:**
- `check_duplicate` at [7](#0-6)  prevents a second `IdentifyMessage` per session, but the attacker needs only one message per session and can reconnect to refresh the entry.
- `global_ip_only` / `is_reachable` filtering only rejects private/loopback addresses; any routable public IP the attacker controls passes.
- There is no cross-session corroboration requirement before an observed address is trusted.

## Impact Explanation

**High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The victim advertises the attacker's IP as its own to every peer it subsequently connects to. Receiving peers write the address into their peer stores under the victim's peer-id and later attempt to dial it. The P2P peer-id handshake fails (the attacker lacks the victim's private key), wasting connection slots and bandwidth across the network. Via hole-punching gossip, the poisoned address reaches O(√N) additional peers per notify interval. The cost to the attacker is one TCP connection and one crafted message.

## Likelihood Explanation

- **Entry path:** Any peer that can establish a TCP connection (inbound or outbound) qualifies. No authentication beyond the standard P2P handshake is required.
- **Ease:** The attacker sends exactly one crafted `IdentifyMessage` immediately after the protocol opens.
- **Persistence:** The poisoned entry remains in `observed_addrs` for the entire session lifetime and is broadcast to every new peer the victim connects to during that window.
- **Repeatability:** The attacker can reconnect to refresh the entry after disconnection, bypassing `check_duplicate`.

## Recommendation

**Short term:** In `process_observed`, extract the IP component of the remote-supplied `observed` address and compare it against the transport-layer source IP of the session (`context.session.address`). Reject (return `MisbehaveResult::Disconnect` or silently ignore) any `observed_addr` whose IP does not match the session's actual remote IP.

**Long term:** Treat `observed_addrs` as unconfirmed hints. Require corroboration from at least N independent peers reporting the same address before including it in outbound `listen_addrs`, consistent with the approach used by Bitcoin Core's address manager and libp2p's identify-push extension.

## Proof of Concept

1. Attacker Eve establishes a TCP connection to victim node V (inbound or outbound).
2. After the secio handshake, Eve sends an `IdentifyMessage` with `observed_addr` = `/ip4/<attacker-public-ip>/tcp/8114`, valid `listen_addrs`, and a valid network identifier.
3. V's `received` handler calls `process_observed` → `add_observed_addr` → `NetworkState::add_observed_addr`, storing `<attacker-public-ip>:8114/p2p/<V-peer-id>` in `observed_addrs[eve_session_id]`.
4. V subsequently connects to peer P. `connected` calls `local_listen_addrs`, which appends the poisoned address (it passes `is_reachable` because it is a public IP), and sends it to P in V's `IdentifyMessage`.
5. P calls `add_remote_listen_addrs`, writing `<attacker-public-ip>:8114` into its peer store under V's peer-id.
6. P later dials `<attacker-public-ip>:8114` expecting V; the connection reaches Eve's server. The P2P peer-id handshake fails, wasting P's connection slot.
7. Steps 4–6 repeat for every peer V connects to during Eve's session. Hole-punching gossip propagates the address to additional peers without further attacker involvement.

            let observe_addrs = self
                .network_state
                .observed_addrs(MAX_RETURN_LISTEN_ADDRS - listens.len());
