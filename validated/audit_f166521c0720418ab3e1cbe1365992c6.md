### Title
Irreversible Request Clearance Before Callback Execution Permanently Locks User Funds — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` calls `clearRequest()` and credits the provider's fee **before** attempting the consumer callback. When the callback fails (caught by `try/catch`), the request is permanently deleted with no retry path, and the user's consumer contract never receives the price update it paid for.

---

### Finding Description

In `Echo.executeCallback()`, the execution order is:

1. **Line 161–162**: Provider is credited the full fee (`accruedFeesInWei += ...`)
2. **Line 164**: `clearRequest(sequenceNumber)` permanently deletes the request from storage
3. **Lines 176–201**: The consumer callback is attempted inside a `try/catch`

When the callback reverts (out-of-gas, consumer bug, or any revert), the `catch` branches emit `PriceUpdateCallbackFailed` and return normally — the transaction does **not** revert. But the request has already been cleared at step 2. Because `findActiveRequest()` checks `isActive(req)` (i.e., `req.sequenceNumber != 0`), any subsequent call to `executeCallback` for the same sequence number will revert with `NoSuchRequest`. There is no `CALLBACK_FAILED` state analogous to `EntropyStatusConstants.CALLBACK_FAILED` in `Entropy.sol`, and no retry function.

The contract's own TODO comment at line 155–156 acknowledges the risk: *"if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract."* However, the comment misframes the problem: even without a revert (thanks to `try/catch`), a failed callback still permanently closes the request.

This is structurally identical to the reported `retryMessage()` bug: a terminal state transition (clearing the request) is applied unconditionally regardless of whether the operation (callback) succeeded, making the function callable only once and preventing any retry.

---

### Impact Explanation

- The user's consumer contract never receives the price update it paid for.
- The provider is credited the full fee despite the callback failing.
- The request is permanently cleared; no retry is possible.
- User funds (the fee paid at request time) are effectively lost with no recourse.

This constitutes permanent loss of user funds and permanent denial of service for the specific request.

---

### Likelihood Explanation

Any consumer contract whose `_echoCallback` reverts — due to out-of-gas (the `callbackGasLimit` is an upper bound, not a guarantee due to the 63/64 rule), a bug in the consumer, or any revert condition — triggers this path. The entry point (`executeCallback`) is callable by anyone after the exclusivity period, so an unprivileged relayer can trigger it. The failure scenario is realistic and requires no special privileges.

---

### Recommendation

Move `clearRequest()` and the fee credit **after** a confirmed successful callback, or introduce a `CALLBACK_FAILED` state (as `Entropy.sol` does with `EntropyStatusConstants.CALLBACK_FAILED`) that keeps the request active and allows re-invocation. The fee credit should only be finalized on success, or a refund/retry mechanism must be provided on failure.

---

### Proof of Concept

```
1. User calls requestPriceUpdatesWithCallback{value: fee}(...) → sequenceNumber = N
2. Provider calls executeCallback(provider, N, updateData, priceIds)
   a. Line 161: provider.accruedFeesInWei += fee  ← fee credited
   b. Line 164: clearRequest(N)                   ← request deleted
   c. Line 176: try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)
      → callback reverts (out-of-gas or consumer bug)
   d. catch: emit PriceUpdateCallbackFailed(...)   ← tx succeeds, no revert
3. Anyone calls executeCallback(provider, N, ...) again
   → findActiveRequest(N) → isActive(req) == false → revert NoSuchRequest()
4. User's consumer never received the price update.
   Provider keeps the fee. No retry possible.

        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        Request storage req = _state.requests[shortKey];
