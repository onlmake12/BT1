Audit Report

## Title
Integer Overflow in `GET_LAST_STATE_PROOF_LIMIT` Guard Bypasses DoS Protection — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
`GetLastStateProofProcess::execute` computes `(last_n_blocks as usize) * 2` using unchecked multiplication at line 201. A peer-supplied `last_n_blocks = 2^63` causes this to wrap to zero in Rust release mode, bypassing the `GET_LAST_STATE_PROOF_LIMIT = 1000` guard entirely. The handler then collects every block number in the chain and performs O(N) MMR root computations plus an O(N log N) `gen_proof` call per unauthenticated P2P message, enabling CPU exhaustion and OOM on the full node.

## Finding Description

**Root cause — line 201:**
`last_n_blocks` is a peer-controlled `u64`. On a 64-bit target `usize` is also 64 bits, so `2^63 as usize = 9223372036854775808` with no truncation. The subsequent `* 2` computes `2^64`, which wraps to `0` in Rust release mode (overflow checks are disabled). With an empty `difficulties` list, the guard evaluates `0 + 0 = 0 > 1000 → false` and does not reject the message. [1](#0-0) 

**Downstream consequence — lines 291–297:**
With `last_n_blocks = 2^63` and `start_block_number = 0`, the condition `last_block_number - 0 <= 2^63` is always true for any realistic chain. The "not enough blocks" branch is unconditionally taken, and `last_n_numbers` collects every block number from `start_block_number` to `last_block_number`. [2](#0-1) 

**Prerequisite checks all pass with the crafted message:**
- Line 210: `is_main_chain(&last_block_hash)` — attacker supplies any publicly observable tip hash. [3](#0-2) 
- Line 231: `start_block_number > last_block_number` — false with `start_block_number = 0`. [4](#0-3) 
- Lines 252–289: all difficulty validation guards short-circuit to no-op with an empty `difficulties` list. [5](#0-4) 

**Work per block in `complete_headers` — lines 132–163:**
For each of the N collected block numbers, the server calls `snapshot.get_ancestor` (chain traversal) and `snapshot.chain_root_mmr(*number - 1).get_root()` (O(log N) MMR root computation). There is no secondary length guard on `numbers`. [6](#0-5) 

**Final `reply_proof` call — lines 207–216:**
`items_positions` contains N entries (one per chain block). `gen_proof` over N positions on an MMR of size N is O(N log N). No secondary limit on `items_positions.len()` exists. [7](#0-6) 

**Limit constant:**
`GET_LAST_STATE_PROOF_LIMIT = 1000` is the only guard, and it is bypassed by the overflow. [8](#0-7) 

**No overflow protection exists anywhere in the module** — no `saturating_mul`, `checked_mul`, `wrapping_mul`, or any overflow annotation is present in `util/light-client-protocol-server/src/`.


## Impact Explanation
A single unauthenticated peer can force a full node to allocate a `Vec` of N block numbers, perform N MMR `get_root()` calls (each reading O(log N) DB nodes), and execute one `gen_proof` call over N positions. For a node with 100,000 blocks this is ~100,000 DB reads plus a large MMR proof computation in a single async handler. Repeated requests cause CPU exhaustion and/or OOM, crashing the node.

**Impact: High — Vulnerabilities which could easily crash a CKB node (10001–15000 points).**

## Likelihood Explanation
- The attacker requires only a valid tip block hash, which is publicly observable from the chain.
- No proof-of-work, no key, and no privileged role is required.
- The molecule encoding of `Uint64 = 2^63` is a standard 8-byte little-endian value, trivially constructable.
- The overflow is deterministic and reproducible in any standard `cargo build --release`.
- The handler is reachable via the standard light-client P2P protocol message path in `try_process`. [9](#0-8) 

## Recommendation
Replace the plain `*` with a saturating multiply to prevent wrap-to-zero:

```rust
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, reject `last_n_blocks` before the cast:

```rust
if last_n_blocks > (constant::GET_LAST_STATE_PROOF_LIMIT / 2) as u64 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

As defense-in-depth, add a secondary length guard in `complete_headers` or before the `gen_proof` call in `reply_proof` to cap `items_positions.len()`.
