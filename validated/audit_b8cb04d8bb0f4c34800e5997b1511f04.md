### Title
`uint64` Overflow in `deriveCrossRate` Denominator Causes Unexpected Panic Revert for Valid Inputs тАФ (File: `target_chains/ethereum/sdk/solidity/PythUtils.sol`)

### Summary

`PythUtils.deriveCrossRate` uses a custom subset of OpenZeppelin's Math library (`Math.sol`) for price arithmetic. In the negative-`deltaExponent` branch, the denominator is computed as `10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)`. Because the literal `10` is inferred as `uint64` (not `uint256`), the exponentiation overflows `uint64` for any `|deltaExponent| >= 20`, causing an arithmetic panic revert (Solidity panic code `0x11`) instead of the documented `PythErrors.ExponentOverflow`. The bounds check at line 107 permits `deltaExponent` down to `-58`, so a large portion of the documented valid input range silently breaks.

### Finding Description

In `PythUtils.deriveCrossRate`:

```solidity
// line 107 тАФ allows deltaExponent in [-58, 58]
if (deltaExponent > 58 || deltaExponent < -58)
    revert PythErrors.ExponentOverflow();

// line 117-122 тАФ else branch (deltaExponent <= 0)
result = Math.mulDiv(
    uint64(price1),
    1,
    10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // тЖР BUG
);
```

`Math.abs(deltaExponent)` returns `uint256`. After the explicit cast `uint64(Math.abs(deltaExponent))`, the literal `10` is inferred as `uint64` by the Solidity compiler (because the result is multiplied by `uint64(price2)`, forcing the whole sub-expression to `uint64`). The exponentiation `10 ** uint64(x)` is therefore a `uint64` operation.

`uint64.max тЙИ 1.844 ├Ч 10^19`. So `10 ** 20` already overflows `uint64`. For any `deltaExponent` in `[-58, -20]`, Solidity 0.8+ reverts with arithmetic panic `0x11` тАФ not with `ExponentOverflow`.

By contrast, `convertToUint` correctly uses `uint` (i.e., `uint256`) for the same computation:

```solidity
// line 61 тАФ correct
10 ** uint(Math.abs(deltaExponent))
```

The test suite (`PythTestUtils.t.sol`) only exercises small negative `deltaExponent` values (e.g., `-8`), so the broken range `[-58, -20]` is never covered.

### Impact Explanation

Any on-chain contract that calls `deriveCrossRate` with parameters producing `deltaExponent` in `[-58, -20]` will receive an unexpected arithmetic panic revert instead of a valid cross-rate. This is a denial-of-service for downstream DeFi protocols (e.g., DEX routers, lending protocols, options vaults) that rely on this SDK function for price computation. Liquidations, swaps, or oracle reads that depend on the cross-rate will revert, potentially freezing protocol operations.

### Likelihood Explanation

The trigger condition is `expo1 - expo2 - targetExponent <= -20`. This is reachable with realistic Pyth feed parameters. For example:
- `expo1 = -8` (standard Pyth feed), `expo2 = 0`, `targetExponent = 12` тЖТ `deltaExponent = -20` тЖТ panic.
- `expo1 = -8`, `expo2 = 0`, `targetExponent = 50` тЖТ `deltaExponent = -58` тЖТ panic.

Any developer following the documented API who chooses a positive `targetExponent` or uses a feed with `expo2 >= 0` will hit this silently.

### Recommendation

Replace `uint64` with `uint256` (or bare `uint`) in the denominator computation, matching the pattern already used correctly in `convertToUint`:

```solidity
// Before (broken):
10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)

// After (correct):
10 ** uint(Math.abs(deltaExponent)) * uint256(uint64(price2))
```

Add test cases for `deltaExponent` values of `-20`, `-40`, and `-58` in `testDeriveCrossRate` to cover the full documented range.

### Proof of Concept

```solidity
// expo1=-8, expo2=0, targetExponent=12 тЖТ deltaExponent = -8 - 0 - 12 = -20

    100,   // price1
    -8,    // expo1
    100,   // price2
    0,     // expo2  (valid: >= -255)
    12     // targetExponent (valid: >= -255)
    // deltaExponent = -8 - (0 + 12) = -20, passes bounds check [-58,58]
    // else branch: 10 ** uint64(20) * uint64(100) тЖТ uint64 overflow тЖТ panic
);
```
