Audit Report

## Title
`n_sync_started` Counter Permanently Inflated via Peer Disconnect Race in `start_sync_headers` — (File: sync/src/synchronizer/mod.rs)

## Summary

In `start_sync_headers`, `n_sync_started` is atomically incremented before the corresponding per-peer `HeadersSyncState` is transitioned to `Started`. A peer disconnect racing between these two non-atomic steps causes `disconnected()` to skip the counter decrement (because `sync_started()` returns `false`), permanently inflating the counter by 1. During IBD, this causes every subsequent `fetch_update` call to return `Err`, breaking the sync loop and stalling initial block download indefinitely until node restart.

## Finding Description

**Root cause — two-step non-atomic update:**

In `start_sync_headers` (`sync/src/synchronizer/mod.rs`, lines 672–691), the counter is incremented first:

```rust
// Step 1: counter incremented atomically
if self.shared().state().n_sync_started()
    .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
        if ibd && x != 0 { None } else { Some(x + 1) }
    })
    .is_err()
{
    break;
}
// Step 2: peer state set to Started — separate, non-atomic step
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
}
```

Between Step 1 and Step 2, `Peers::disconnected()` (`sync/src/types/mod.rs`, lines 901–924) can execute on a concurrent thread:

```rust
pub fn disconnected(&self, peer: PeerIndex) {
    if let Some(peer_state) = self.state.remove(&peer)
                                        .map(|(_, peer_state)| peer_state) {
        if peer_state.sync_started() {   // returns false: state is SyncProtocolConnected
            // n_sync_started.fetch_sub(1) ← SKIPPED
        }
    }
}
```

`sync_started()` delegates to `ChainSyncState::started()`, which returns `true` only for `HeadersSyncState::Started`. Since Step 2 has not yet executed, the state is still `SyncProtocolConnected`, so the decrement is skipped. After `disconnected()` removes the peer, Step 2's `get_mut(&peer)` returns `None`, so `start_sync` is never called. The counter is now `1` with no peer in `Started` state — permanently leaked.

**IBD stall:**

The IBD guard at line 677:
```rust
if ibd && x != 0 { None }
```
With `x == 1` permanently, every `fetch_update` call returns `Err`, causing `break` on the first iteration of the peer loop. No header sync is ever initiated again until the node is restarted.

**Concurrency is by design:** `Peers.state` is a `DashMap<PeerIndex, PeerState>` and `n_sync_started` is an `AtomicUsize` — both explicitly designed for concurrent multi-threaded access with no shared lock protecting their joint invariant.

## Impact Explanation

A victim node in IBD with a permanently inflated `n_sync_started` counter cannot initiate header synchronization with any peer. It is stuck at genesis, unable to advance its chain tip, and cannot participate in the network. This is functionally equivalent to a crashed node from the perspective of network participation. This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

During IBD, `can_start_sync` requires the peer to be `is_protect` (outbound peer within the protected count) or `is_whitelist`. An attacker running a publicly reachable node can be discovered and connected to by the victim as an outbound peer, satisfying `is_protect`. The race window is between two consecutive non-async Rust statements on separate threads (network event thread vs. sync timer thread), making it a genuine multi-threaded race. The `SEND_GET_HEADERS_TOKEN` timer fires every 1 second (`SYNC_NOTIFY_INTERVAL`), giving the attacker a repeatable, predictable trigger. A single successful race permanently stalls the node; the attacker can retry cheaply by reconnecting on each timer tick.

## Recommendation

The increment of `n_sync_started` and the transition of `HeadersSyncState` to `Started` must be performed as a single atomic operation under the same `DashMap` entry write guard. Concretely: acquire `state.get_mut(&peer)` first, verify `can_start_sync` still holds, then increment `n_sync_started` and call `start_sync` while holding the write guard. This ensures `disconnected()` either sees the full `Started` state (and decrements) or sees the peer already removed (and the counter was never incremented). Alternatively, add a compensating decrement in `start_sync_headers` when `get_mut(&peer)` returns `None` after a successful `fetch_update`.

## Proof of Concept

1. Victim node is in IBD (`is_initial_block_download() == true`), `n_sync_started == 0`.
2. Attacker runs a publicly reachable node; victim connects to it outbound → peer P is marked `is_protect`.
3. `SEND_GET_HEADERS_TOKEN` fires → `start_sync_headers` runs → P passes `can_start_sync`.
4. `n_sync_started.fetch_update(...)` succeeds → counter becomes `1`.
5. Attacker immediately closes the TCP connection → `Peers::disconnected(P)` fires on the network thread.
6. `state.remove(&P)` returns `PeerState` with `HeadersSyncState::SyncProtocolConnected`; `sync_started()` → `false`; counter NOT decremented. Counter stays at `1`.
7. Back in `start_sync_headers`: `state.get_mut(&P)` → `None`; `start_sync` never called.
8. Counter is `1`, no peer is in `Started` state.
9. On every subsequent `SEND_GET_HEADERS_TOKEN` tick: `fetch_update` returns `Err` (`ibd && x != 0`), loop breaks immediately. No sync ever starts.
10. Node is permanently stalled in IBD until restarted.

**Unit test plan:** Construct a `Peers` instance; call `n_sync_started.fetch_update` directly to simulate Step 1; call `peers.disconnected(P)` before calling `peer_state.start_sync()`; assert `n_sync_started.load() == 1` with no peer in `Started` state; verify that a subsequent simulated `start_sync_headers` call with `ibd=true` never starts sync for any peer.
