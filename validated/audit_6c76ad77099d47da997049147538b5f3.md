### Title
`RecentReject` Count Limit Never Enforced Due to Stale `total_keys_num` — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

The `put()` method in `RecentReject` performs a limit check against `self.total_keys_num`, but **never updates `self.total_keys_num` after writing a new entry**. The result of `checked_add(1)` is bound only to a local variable; the struct field stays at its initial estimated value forever. This makes the `count_limit` check permanently ineffective, allowing the rejected-transaction database to grow without bound.

---

### Finding Description

In `tx-pool/src/component/recent_reject.rs`, the `put()` function writes a rejected transaction hash to a RocksDB-backed store and then checks whether the total count exceeds `count_limit`:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ...
    self.db.put(&shard, hash_slice, json_string)?;   // entry written to DB

    if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
        if total_keys_num > self.count_limit {
            self.shrink()?;                          // only reached if initial estimate was already at limit
        }
    } else {
        self.shrink()?;
    }
    Ok(())
    // self.total_keys_num is NEVER updated here
}
``` [1](#0-0) 

`self.total_keys_num.checked_add(1)` produces a value bound to the local variable `total_keys_num` inside the `if let Some(...)` arm. `self.total_keys_num` itself is **never assigned** the incremented value. On a fresh node (empty DB), `self.total_keys_num` is initialized to `0` at startup: [2](#0-1) 

So every call to `put()` evaluates `0 + 1 > count_limit`, which is always `false` for any reasonable `count_limit`. `shrink()` is never triggered. The only time `self.total_keys_num` is updated is inside `shrink()` itself: [3](#0-2) 

But `shrink()` is unreachable because the guard condition is never satisfied. The existing unit test inadvertently confirms this: it asserts `recent_reject.total_keys_num < 100` after 160 `put()` calls with `limit = 100`, and the assertion passes only because `total_keys_num` is still `0` — not because the limit was enforced. [4](#0-3) 

The `count_limit` is configured via `keep_rejected_tx_hashes_count` in `TxPoolConfig` and is intended to cap the on-disk size of the recent-reject store.

---

### Impact Explanation

The `recent_reject` database accumulates every qualifying rejected transaction indefinitely. An attacker who can cause repeated rejections will exhaust the node's disk space, leading to a denial-of-service: the node crashes or becomes unable to write new data (blocks, chain state, tx-pool). The impact is **disk resource exhaustion / node DoS**.

---

### Likelihood Explanation

The attacker entry path is fully unprivileged. Any peer or RPC caller can submit transactions via `send_transaction`. Transactions rejected with reasons such as `Malformed`, `DeclaredWrongCycles`, `Verification`, or `LowFeeRate` are recorded via `put_recent_reject()`: [5](#0-4) [6](#0-5) 

Submitting a large volume of cheaply-crafted invalid transactions (e.g., transactions with a declared cycle count that mismatches actual execution, or transactions with invalid witnesses) is straightforward and requires no special privilege. The bug is present on every node that has `recent_reject` enabled (the default).

---

### Recommendation

Assign the incremented value back to `self.total_keys_num` inside `put()` before the limit check:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ...
    self.db.put(&shard, hash_slice, json_string)?;

    if let Some(new_total) = self.total_keys_num.checked_add(1) {
        self.total_keys_num = new_total;          // ← persist the increment
        if self.total_keys_num > self.count_limit {
            self.shrink()?;
        }
    } else {
        self.shrink()?;
    }
    Ok(())
}
```

---

### Proof of Concept

1. Start a CKB node with `recent_reject` storage enabled (default).
2. Repeatedly submit transactions via `send_transaction` RPC that will be rejected (e.g., transactions with a `declared_cycles` value that does not match actual script execution — `DeclaredWrongCycles`).
3. Observe that `self.total_keys_num` never advances past its initial value of `0`, so `shrink()` is never called.
4. The on-disk RocksDB store for `recent_reject` grows without bound, consuming all available disk space and eventually crashing the node.

The existing test at `tx-pool/src/component/tests/recent_reject.rs:39` demonstrates the bug: after 160 `put()` calls against a limit of 100, `total_keys_num` is still `0` (not `≥ 100`), confirming the counter is never updated and the limit is never enforced. [1](#0-0) [7](#0-6)
