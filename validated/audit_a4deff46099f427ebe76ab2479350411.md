Audit Report

## Title
Stale Descendant-Weight Statistics After Batch Removal Causes Incorrect Eviction Scoring and Tx-Pool DoS - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` erases all parent-child links for every removed entry before calling `remove_entry` on any of them. Because `remove_entry` relies on those links (via `update_ancestors_index_key` → `calc_ancestors`) to decrement surviving ancestors' `descendants_fee/size/cycles/count`, the decrement never happens. Surviving ancestors permanently carry inflated descendant-weight statistics, inflating their `EvictKey` and suppressing their eviction, allowing an attacker to fill the pool with phantom-fee entries at low cost.

## Finding Description
`remove_entry_and_descendants` (pool_map.rs L252-265) first iterates all removed IDs and calls `remove_entry_links(id)` for each. [1](#0-0) 

`remove_entry_links` (L418-430) removes the entry from every parent's children set and then calls `self.links.remove(id)`, which deletes the entry's link record entirely from `TxLinksMap`. [2](#0-1) 

Only after all links are gone does `remove_entry_and_descendants` call `remove_entry(id)` for each ID. `remove_entry` (L235-250) calls `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` at L242. [3](#0-2) 

`update_ancestors_index_key` (L432-445) calls `self.links.calc_ancestors(&child.proposal_short_id())`. [4](#0-3) 

`calc_ancestors` delegates to `calc_relative_ids`, which calls `self.inner.get(short_id)` and returns `unwrap_or_default()` — an empty set — when the link record is absent. [5](#0-4) 

Because the link record for the removed entry was already deleted in the first loop, `calc_ancestors` returns `{}`. The `for anc_id in &ancestors` loop body — which calls `e.inner.sub_descendant_weight(child)` and recomputes `e.evict_key` — never executes. Every ancestor of the removed subtree that was **not** itself removed retains stale (inflated) values for `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`.

`EvictKey` (entry.rs L234-247) is computed as `max(descendants_feerate, own_feerate)`, so inflated descendant fields produce an artificially high evict key. [6](#0-5) 

`EvictKey` ordering (sort_key.rs L92-103) sorts by ascending `fee_rate` first — the comment explicitly states "select the smallest fee_rate... for eviction." [7](#0-6) 

`next_evict_entry` (L380-385) iterates `iter_by_evict_key()` in ascending order and returns the first matching entry. Entries with inflated (high) evict keys appear to have high fee rates and are skipped. [8](#0-7) 

`limit_size` (pool.rs L292-329) loops calling `next_evict_entry` and `remove_entry_and_descendants`. Phantom-fee ancestors are never selected; instead, legitimate high-fee transactions are evicted first, and new legitimate transactions are rejected with `Reject::Full`. [9](#0-8) 

The existing `remove_entry_links` call inside `remove_entry` itself (L245) is a no-op for the already-cleared entries and provides no correction. [10](#0-9) 

## Impact Explanation
An attacker submits a low-fee parent A and a high-fee child B, then displaces B via a cheap RBF-conflicting transaction C. `resolve_conflict` calls `remove_entry_and_descendants(B)`, leaving A in the pool with B's fee contribution permanently embedded in A's `descendants_fee`. A's `EvictKey` remains artificially high. When the pool reaches `max_tx_pool_size`, `limit_size` evicts legitimate high-fee transactions before A, and new legitimate transactions are rejected with `Reject::Full`. By repeating the pattern with many parent transactions, the attacker can sustain CKB network congestion at low cost. This matches the allowed **High** impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
The attack requires no privileged access. Any P2P peer or RPC caller can submit transactions. `remove_entry_and_descendants` is triggered by `resolve_conflict` (RBF), `resolve_conflict_header_dep` (header-dep invalidation), and `limit_size` (size eviction) — all reachable without authentication. [11](#0-10) [12](#0-11) 

The attacker's cost per phantom-fee slot is the fee for parent A plus the minimum RBF fee bump for the conflicting transaction C. The stale state persists until A is committed on-chain or the node restarts, giving the attacker a long window per slot. The attack is repeatable and scalable.

## Recommendation
Compute the set of surviving ancestors of the root entry **before** clearing any links, then explicitly call `sub_descendant_weight` and recompute `evict_key` for each of them. Concretely:

1. Before the `remove_entry_links` loop in `remove_entry_and_descendants`, call `self.links.calc_ancestors(id)` for the root entry to obtain the set of ancestors that will survive the removal.
2. After all entries are removed, iterate that ancestor set and apply `sub_descendant_weight` for each removed entry that was a descendant of each ancestor, then recompute `evict_key`.

Alternatively, restructure the function so that `remove_entry_links` is called only for the strict descendants (not the root), allowing `remove_entry`'s existing `update_ancestors_index_key` call to correctly find and update the root's surviving ancestors before the root's own links are cleared.

## Proof of Concept
```
1. Submit tx A: low fee (100 shannons), size 200 bytes.
   Pool state: A.descendants_fee=100, A.descendants_size=200.

2. Submit tx B: child of A, high fee (10_000 shannons), size 200 bytes.
   Pool state: A.descendants_fee=10_100, A.descendants_size=400.
   A.evict_key.fee_rate ≈ 25_250 shannons/kB (high → not evicted).

3. Submit tx C: spends same input as B (RBF conflict, minimum fee bump).
   resolve_conflict → remove_entry_and_descendants(B):
     remove_entry_links(B): removes B from A's children, deletes B's link record.
     remove_entry(B): update_ancestors_index_key(B, Remove):
       calc_ancestors(B) → {} (link already gone)
       → A.sub_descendant_weight(B) is NEVER called.

4. After removal:
   A.descendants_fee   = 10_100  (should be 100)
   A.descendants_size  = 400     (should be 200)
   A.evict_key.fee_rate ≈ 25_250 (should be ~500 shannons/kB)

5. Fill pool to max_tx_pool_size with transactions having fee_rate < 25_250.
   limit_size loops: next_evict_entry skips A (inflated evict key).
   Legitimate tx D (fee=5_000 shannons, fee_rate < A's phantom rate) → evicted or rejected Reject::Full.
```

A unit test can assert that after `remove_entry_and_descendants(&tx2_id)` in a chain tx1→tx2, `pool_map.get(&tx1_id).unwrap().descendants_fee == tx1.fee` and `pool_map.get(&tx1_id).unwrap().descendants_count == 1`.

        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        for id in &removed_ids {

        let mut conflicts = Vec::new();

        for i in tx.input_pts_iter() {
            if let Some(id) = self.edges.remove_input(&i) {
                let entries = self.remove_entry_and_descendants(&id);
                if !entries.is_empty() {
                    let reject = Reject::Resolve(OutPointError::Dead(i.clone()));
                    let rejects = std::iter::repeat_n(reject, entries.len());

                        let rejects = std::iter::repeat_n(reject, entries.len());
                        conflicts.extend(entries.into_iter().zip(rejects));

                }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

/// First compare fee_rate, select the smallest fee_rate,
/// and then select the latest timestamp, for eviction,
/// the latest timestamp which also means that the fewer descendants may exist.
#[derive(Eq, PartialEq, Clone, Debug)]
pub struct EvictKey {
    pub fee_rate: FeeRate,
    pub timestamp: u64,
    pub descendants_count: usize,
}

impl PartialOrd for EvictKey {
    fn partial_cmp(&self, other: &Self) -> Option<Ordering> {
        Some(self.cmp(other))
    }
}

impl Ord for EvictKey {
    fn cmp(&self, other: &Self) -> Ordering {
        if self.fee_rate == other.fee_rate {
            if self.descendants_count == other.descendants_count {
                self.timestamp.cmp(&other.timestamp)
            } else {
                self.descendants_count.cmp(&other.descendants_count)
            }
        } else {
            self.fee_rate.cmp(&other.fee_rate)
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
