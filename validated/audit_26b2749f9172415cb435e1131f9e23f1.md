### Title
Underestimated Minimum Fee Threshold in `check_tx_fee` Omits Cycles-Based Weight Component — (File: `tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` calculates the minimum required fee using only the transaction's serialized byte size, but the canonical transaction weight is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. When a transaction's cycle consumption is high enough that the cycles-derived weight exceeds its byte size, the minimum fee check underestimates the required fee. An unprivileged transaction sender can submit a transaction whose actual effective fee rate is below the configured `min_fee_rate`, bypassing the admission gate and polluting the mempool.

---

### Finding Description

The minimum fee gate in `check_tx_fee` is:

```rust
// tx-pool/src/util.rs:42-52
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0) 

The canonical weight function used everywhere else in the codebase is:

```rust
// util/types/src/core/tx_pool.rs:298-303
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` is derived from `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES`. [3](#0-2) 

`check_tx_fee` passes `tx_size` directly to `FeeRate::fee`, which computes `min_fee_rate * tx_size / 1000`. It never calls `get_transaction_weight`. The cycles count is not available at the point `check_tx_fee` is called (it is determined by script execution later), but the result is that the admission gate is structurally weaker than the fee-rate metric used for pool ordering and block assembly.

The same `get_transaction_weight` is used for:
- Pool entry fee-rate scoring (`TxEntry::fee_rate`, `AncestorsScoreSortKey`)
- Block-level fee-rate statistics (`rpc/src/util/fee_rate.rs`)
- Fee estimator (`util/fee-estimator`) [4](#0-3) [5](#0-4) 

So the admission check and the actual fee-rate metric are inconsistent: admission uses `size`, everything else uses `max(size, cycles × factor)`.

---

### Impact Explanation

**Mempool pollution below the minimum fee-rate floor.** A transaction sender can craft a transaction that:

1. Has a small serialized size (e.g., 1 000 bytes).
2. Executes a script with high cycle consumption (e.g., 10 000 000 cycles).

With `min_fee_rate = 1 000 shannons/KB`:

| Metric | Value |
|---|---|
| `tx_size` | 1 000 bytes |
| `cycles` | 10 000 000 |
| `weight = max(1000, 10 000 000 × 0.000_170_571_4)` | **1 705** |
| `min_fee` checked at admission | `1 000 × 1 000 / 1 000 = 1 000 shannons` |
| Actual minimum fee at `min_fee_rate` | `1 000 × 1 705 / 1 000 = 1 705 shannons` |
| Effective fee rate if fee = 1 000 shannons | `1 000 × 1 000 / 1 705 ≈ 586 shannons/KB` |

The transaction passes `check_tx_fee` with 1 000 shannons but its true effective fee rate is ~586 shannons/KB — 41 % below the configured floor. The node operator's `min_fee_rate` policy is silently violated.

Consequences:
- The pool accumulates transactions whose effective fee rate is below the operator-configured minimum.
- These transactions occupy pool space (up to `max_tx_pool_size = 180 MB`) and consume relay bandwidth.
- They are sorted to the bottom of the pool by actual weight-based fee rate and are unlikely to be mined, creating persistent pool bloat.
- The discrepancy between the admission check and the eviction/ordering metric means the pool can fill with low-quality transactions that the operator intended to exclude.

---

### Likelihood Explanation

The entry path is fully open to any unprivileged actor:

- Via RPC `send_transaction` (local or remote if RPC is exposed).
- Via P2P transaction relay from any connected peer.

No special privilege, key, or majority hash power is required. The attacker only needs to submit a valid transaction with a script that consumes many cycles. CKB-VM scripts are user-defined, so any script author can produce a high-cycle, small-size transaction. The `max_tx_verify_cycles` limit (default 70 000 000) bounds the maximum cycle count, but even at 5 000 000 cycles the weight exceeds size for transactions under ~853 bytes, which is achievable.

The code comment at line 42–44 explicitly acknowledges the approximation, confirming this is a known design gap rather than an oversight in the analysis. [6](#0-5) 

---

### Recommendation

Replace the size-only minimum fee check with a weight-based check. Because cycles are not yet known at the point `check_tx_fee` is called (they are determined by `verify_rtx`), the fix requires one of:

1. **Move `check_tx_fee` after `verify_rtx`** so the actual cycle count is available, then compute `min_fee = min_fee_rate.fee(get_transaction_weight(tx_size, cycles))`.
2. **Apply a conservative upper-bound** at admission: use `get_transaction_weight(tx_size, max_tx_verify_cycles)` as the weight for the minimum fee check, ensuring no transaction with cycles up to the limit can bypass the floor.
3. **Re-check fee rate after cycle verification**: after `verify_rtx` returns the actual cycles, perform a second fee-rate check using the true weight and reject if below `min_fee_rate`.

Option 1 or 3 is most accurate. Option 2 is overly conservative but safe.

### Proof of Concept

**Configuration:** `min_fee_rate = 1_000` shannons/KB (default), `max_tx_verify_cycles = 70_000_000`.

**Craft a transaction:**
- One input cell, one output cell (small serialized size, e.g., ~500 bytes).
- Lock script that loops for 8 000 000 cycles (well within the limit).
- Fee = `min_fee_rate.fee(tx_size)` = `1_000 × 500 / 1_000 = 500 shannons`.

**Admission check in `check_tx_fee`:**
```
min_fee = min_fee_rate.fee(500) = 500 shannons
fee = 500 shannons
fee >= min_fee  →  ACCEPTED
```

**Actual effective fee rate (used for pool ordering):**
```
weight = max(500, 8_000_000 × 0.000_170_571_4) = max(500, 1_364) = 1_364
effective_fee_rate = 500 × 1_000 / 1_364 ≈ 366 shannons/KB
```

The transaction is admitted with an effective fee rate of ~366 shannons/KB, 63 % below the 1 000 shannons/KB minimum. Repeating this with many such transactions fills the pool with below-minimum-fee-rate entries, violating the operator's admission policy. [7](#0-6) [8](#0-7)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
