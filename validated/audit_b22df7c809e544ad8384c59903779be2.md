Audit Report

## Title
Unbounded `NOT IN` SQL clause from `Pool.dead_cells` causes DoS of rich-indexer RPC — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`, `get_cells_capacity.rs`)

## Summary
When `index_tx_pool = true`, `get_cells` and `get_cells_capacity` collect every entry from the unbounded `Pool.dead_cells` `HashSet` and embed one SQL bind parameter per entry into a `NOT IN (…)` clause with no upper bound. An attacker who submits ~1,000 single-input mempool transactions pushes the parameter count past SQLite's default `SQLITE_MAX_VARIABLE_NUMBER` limit of 999, causing every subsequent call to these two RPC methods to return `Error::DB` for as long as the mempool remains large.

## Finding Description
`Pool.dead_cells` is a plain `HashSet<OutPoint>` with no capacity cap. [1](#0-0) 

`Pool::new_transaction` inserts every input of every accepted mempool transaction into this set with no guard or limit. [2](#0-1) 

In `get_cells`, the entire set is iterated and one `$N` placeholder is emitted per entry into a `NOT IN (…)` clause with no pre-check on set size: [3](#0-2) 

`get_cells_capacity` has the identical unbounded pattern: [4](#0-3) 

SQLite enforces `SQLITE_MAX_VARIABLE_NUMBER` (default 999) at query execution time. When exceeded, `sqlx` returns an error propagated as `Error::DB`. A grep across the entire codebase confirms no existing check limits `dead_cells` size before SQL construction or execution. [5](#0-4) 

## Impact Explanation
All callers of `get_cells` and `get_cells_capacity` receive persistent `Error::DB` responses for the duration of the attack. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**. The node's consensus and block-production paths are entirely unaffected; only the rich-indexer query surface is rendered non-functional. [6](#0-5) 

## Likelihood Explanation
- Requires `index_tx_pool = true` and the `RichIndexer` RPC module to be enabled — both are opt-in and off by default. [7](#0-6) 
- Submitting transactions to the mempool is an open, unprivileged operation over RPC/P2P.
- SQLite's 999-parameter limit is crossed with as few as ~1,000 single-input transactions, which occupy well under 1 MB against the 180 MB default `max_tx_pool_size`.
- No PoW, no privileged access, and no special configuration beyond the two opt-in flags is required.
- The condition is persistent: the DoS lasts as long as the mempool holds ≥1,000 inputs, and the attacker can continuously resubmit to maintain it. [2](#0-1) 

## Recommendation
Replace the inline `NOT IN (…)` expansion with a bounded approach:

1. **Temp table / CTE**: Insert dead cells into a temporary table or CTE and join against it — both SQLite and PostgreSQL support this without parameter-count limits.
2. **Chunked exclusion**: Split dead cells into chunks of ≤500 and apply multiple `NOT IN` clauses per chunk, staying under the SQLite limit.
3. **Hard cap with clear error**: If `dead_cells.len()` exceeds a configurable threshold (e.g., 500), return `Error::invalid_params` rather than attempting an oversized query.

## Proof of Concept
```rust
// Populate a mock Pool with 1,000 dead cells (SQLite limit = 999)
let mut pool = Pool::default();
for i in 0u32..1_000 {
    let tx = TransactionBuilder::default()
        .input(CellInput::new(OutPoint::new(h256!("0xdead").pack(), i), 0))
        .build();
    pool.new_transaction(&tx.into_view());
}
// pool.dead_cells().count() == 1,000
// Calling get_cells() now builds a NOT IN clause with 1,000 bind parameters,
// exceeding SQLite's SQLITE_MAX_VARIABLE_NUMBER (999) → returns Error::DB.
```

Manual steps:
1. Start a CKB node with `index_tx_pool = true` and the `RichIndexer` RPC module enabled.
2. Submit 1,000 distinct single-input transactions to the mempool via RPC/P2P.
3. Call `get_cells` or `get_cells_capacity` — both return `Error::DB` for the duration the mempool holds ≥1,000 inputs.
