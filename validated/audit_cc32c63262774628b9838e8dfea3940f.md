### Title
Silent `u128`→`u64` Truncating Cast in `calculate_maximum_withdraw` Produces Wrong DAO Withdrawal Amounts Without Error — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes an intermediate `u128` value (`withdraw_counted_capacity`) and then casts it to `u64` using the Rust `as u64` operator — a **silent truncating cast** that wraps on overflow without returning an error. Every analogous `u128`→`u64` narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, making this a clear inconsistency. When the intermediate value exceeds `u64::MAX`, the result is silently truncated to a wrong (much smaller) withdrawal capacity, and the function returns `Ok(wrong_value)` instead of `Err(DaoError::Overflow)`.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the DAO withdrawal capacity as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits if `withdraw_counted_capacity > u64::MAX`. The function then returns `Ok(truncated_wrong_value)` with no indication of error.

Compare this to the two structurally identical patterns in the same file that correctly use checked conversion:

```rust
// dao_field_with_current_epoch (line 244-245)
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) 

```rust
// secondary_block_reward (line 204)
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

The pattern is identical — multiply two `u64` values promoted to `u128`, divide, then narrow back to `u64` — but only `calculate_maximum_withdraw` uses the unsafe `as u64` cast.

The existing test `check_withdraw_calculation_overflows` expects `result.is_err()` for a near-maximum-capacity cell, but the `as u64` truncation means the function may silently return a wrong `Ok` value rather than the expected `Err`, making the test's overflow detection unreliable for this specific path. [4](#0-3) 

`calculate_maximum_withdraw` is called from two production paths:

1. `transaction_maximum_withdraw` → `transaction_fee` — used during transaction verification to check that DAO withdrawal outputs do not exceed the maximum withdrawable amount. [5](#0-4) 

2. The `calculate_dao_maximum_withdraw` RPC endpoint, directly reachable by any RPC caller. [6](#0-5) 

---

### Impact Explanation

If `withdraw_counted_capacity` overflows `u64`, the `as u64` cast silently wraps it to a much smaller value. Two consequences follow:

- **Wrong RPC result**: `calculate_dao_maximum_withdraw` returns a silently truncated (far too small) withdrawal capacity to the caller, misleading users about how much CKB they can withdraw.
- **Incorrect transaction verification**: `transaction_fee = maximum_withdraw - outputs_capacity`. With a truncated `maximum_withdraw`, a valid DAO withdrawal transaction may be incorrectly rejected (if truncated value < outputs_capacity), or the fee calculation is wrong, corrupting the economic accounting of the block.

---

### Likelihood Explanation

The overflow condition requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

Since `withdrawing_ar ≥ deposit_ar` (the accumulation rate only increases), overflow requires the AR ratio to exceed `u64::MAX / counted_capacity`. For a cell with a very large capacity (e.g., `counted_capacity ≈ u64::MAX / 2 ≈ 9.2 × 10¹⁸` shannons), the AR would only need to grow to roughly 2× its initial value (`10¹⁶`), which is plausible over a sufficiently long chain lifetime. For typical cell sizes the threshold is much higher and practically unreachable. Likelihood is therefore **low but non-zero**, and the severity of silent wrong-value return (rather than a clean error) makes it worth fixing.

---

### Recommendation

Replace the silent `as u64` cast with the same checked conversion used elsewhere in the file:

```rust
// Before (silent truncation):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After (consistent with the rest of the file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
``` [1](#0-0)

### Proof of Concept

The bug is demonstrated by the structural inconsistency within the same file. The three u128→u64 narrowings in `util/dao/src/lib.rs` are:

| Location | Method | Narrowing |
|---|---|---|
| Line 204 | `secondary_block_reward` | `u64::try_from(reward128).map_err(...)` ✓ |
| Line 244–245 | `dao_field_with_current_epoch` | `u64::try_from(miner_issuance128).map_err(...)` ✓ |
| Line 156 | `calculate_maximum_withdraw` | `withdraw_counted_capacity as u64` ✗ |

A concrete triggering scenario: construct a DAO cell with `output_capacity` near `u64::MAX` (e.g., `18_446_744_073_709_550_000` shannons as used in the existing test), deposit it at block A with `deposit_ar = AR_0`, and withdraw at block B where `withdrawing_ar` is large enough that `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. The function returns `Ok(truncated_value)` instead of `Err(DaoError::Overflow)`, silently producing a wrong withdrawal amount.
