Audit Report

## Title
Unauthenticated `from` PeerId in Hole-Punching Protocol Enables Attacker-Directed Outbound TCP Connection Storms — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
The hole-punching protocol's `ConnectionRequest` and `ConnectionSync` handlers accept an arbitrary `from` PeerId from message content without verifying it matches the authenticated session sender. A single unprivileged peer can forge `from` to inject attacker-controlled TCP addresses into the victim's `pending_delivered` map, then trigger unbounded outbound TCP connection attempts to those addresses via a follow-up `ConnectionSync`. The per-session rate limit of 30 req/sec is the only binding constraint, enabling up to 720 outbound TCP connection attempts per second from the victim node.

## Finding Description

**Root cause — no `from` authentication in `respond_delivered`:**

In `connection_request.rs`, `respond_delivered` is invoked when `self_peer_id == &content.to`. The `from_peer_id` parameter is taken directly from the message content and is never compared against `self.peer` (the actual authenticated session index): [1](#0-0) 

The deduplication guard at L161–167 is keyed by `from_peer_id` from the message, so using a fresh random `from` PeerId per request trivially bypasses it: [2](#0-1) 

After filtering to TCP/IP addresses (which attacker-supplied addresses pass), the forged `from_peer_id` and attacker-controlled `remote_listens` are inserted unconditionally into `pending_delivered`: [3](#0-2) 

**Root cause — no `from` authentication in `ConnectionSync`:**

In `connection_sync.rs`, when `route` is empty and `self_peer_id == &content.to`, the handler looks up `pending_delivered` using `content.from` from the message — again with no check that `content.from` matches the actual session sender: [4](#0-3) 

For every stored address, a `runtime::spawn` task is created that calls `try_nat_traversal`, making a real outbound TCP connection attempt: [5](#0-4) [6](#0-5) 

**Rate-limit bypass:**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`: [7](#0-6) 

It is configured at 1 req/sec per `(from, to, msg_item_id)` tuple: [8](#0-7) 

Since the attacker controls `from`, using a fresh random PeerId per pair bypasses this limiter entirely. The only binding constraint is the per-session `rate_limiter` at 30 req/sec keyed by `(session_id, msg.item_id())`: [9](#0-8) 

**Unbounded `pending_delivered` growth:**

The cleanup in `notify()` runs every 5 minutes (`CHECK_INTERVAL`). With 30 req/sec, an attacker can insert up to 9,000 entries per cleanup window, each holding up to `ADDRS_COUNT_LIMIT` (24) `Multiaddr` values, causing unbounded memory growth between cleanup cycles: [10](#0-9) 

## Impact Explanation

**High — Vulnerability which could easily crash a CKB node.**

At 30 message pairs/sec (the per-session cap), each pair spawns a `runtime::spawn` task that attempts outbound TCP connections to up to 24 attacker-controlled addresses. This yields up to 720 concurrent outbound TCP connection attempts per second. Sustained over seconds, this exhausts the victim node's file descriptors and OS connection-tracking state, causing the node process to crash or become unable to accept/initiate legitimate connections. Additionally, the victim's IP address becomes the apparent source of a TCP SYN flood toward arbitrary third-party targets.

## Likelihood Explanation

The attacker requires only a single standard P2P connection to the victim — no key material, PoW, or privileged role. The victim's PeerId (needed for the `to` field) is publicly observable from the P2P network. The `HolePunching` protocol is enabled by default. The two-message sequence is straightforward to construct from the public `packed::ConnectionRequest` / `packed::ConnectionSync` schema. The attack is repeatable and sustainable for the duration of the session.

## Recommendation

1. **Authenticate `from` in `respond_delivered`**: Assert that `from_peer_id` equals the PeerId of the actual session. Resolve the session's PeerId from `self.peer` via `network_state.peer_registry` and reject any message where `content.from` does not match.
2. **Authenticate `from` in `ConnectionSyncProcess`**: Pass the session's PeerId into `ConnectionSyncProcess` and verify `content.from` against it before looking up `pending_delivered`.
3. **Cap `pending_delivered` size**: Enforce a maximum entry count (e.g., equal to `max_outbound`) to bound memory growth from any remaining unauthenticated paths.

## Proof of Concept

```rust
// Step 1: send ConnectionRequest to victim with forged `from` and attacker TCP addr
// Repeat with fresh forged_from PeerIds at 30/sec to bypass HOLE_PUNCHING_INTERVAL and forward_rate_limiter
let forged_from = PeerId::random();
let attacker_addr: Multiaddr = "/ip4/1.2.3.4/tcp/9999".parse().unwrap();
let req = packed::ConnectionRequest::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())   // victim's known PeerId
    .listen_addrs(vec![attacker_addr.to_vec()].pack())
    .max_hops(6u8.pack())
    .route(vec![].pack())        // empty route → triggers respond_delivered
    .build();
attacker_session.send(req);

// Step 2: send ConnectionSync with same forged `from`
let sync = packed::ConnectionSync::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())
    .route(vec![].pack())        // empty route → triggers pending_delivered lookup
    .build();
attacker_session.send(sync);

// Result: victim spawns runtime::spawn task → outbound TCP connect to 1.2.3.4:9999
// Repeat at 30 pairs/sec with fresh forged_from each time → 720 TCP SYNs/sec to arbitrary targets
// Sustained attack exhausts victim's file descriptors → node crash
```
