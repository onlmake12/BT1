Audit Report

## Title
Peer Store Permanent Saturation via Off-by-One in `check_purge` Group Eviction Threshold — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

`check_purge` in `peer_store_impl.rs` uses a strictly-greater-than threshold (`addrs.len() > 4`) when deciding whether to evict addresses from an over-represented network group. An attacker who fills all 16 384 peer-store slots with exactly 4 addresses per /16 network group causes both eviction passes to produce empty candidate lists, returning `Err(EvictionFailed)`. The discovery protocol silently drops this error at `debug` level, so every subsequent honest address is permanently rejected and the store remains saturated with attacker-controlled entries.

## Finding Description

**Pass 1** collects every address for which `is_connectable(now_ms)` returns `false`. [1](#0-0) 

Freshly-added attacker addresses always pass `is_connectable`: `attempts_count` starts at 0 (below `ADDR_MAX_RETRIES = 3`) and `last_connected_at_ms` starts at 0, so neither false-branch fires. [2](#0-1) 

**Pass 2** (only reached when pass 1 removes nothing) groups addresses by network segment, sorts descending by group size, takes the top half of groups, and evicts 2 addresses per group **only if `addrs.len() > 4`** (strictly greater than): [3](#0-2) 

The network group key for IPv4 is `Group::IP4([bits[0], bits[1]])` — the first two octets (/16), not /24: [4](#0-3) 

An attacker using 4 096 distinct /16 subnets (varying both the first and second octet) with exactly 4 addresses each fills all 16 384 slots (`ADDR_COUNT_LIMIT`) [5](#0-4) 
while keeping every group at exactly 4 — the threshold `> 4` never fires. Both passes produce empty candidate lists, and `Err(PeerStoreError::EvictionFailed)` is returned.

`add_addr` propagates that error with `?`: [6](#0-5) 

The discovery protocol silently swallows the error at `debug` level: [7](#0-6) 

**Note on the submitted PoC:** The original PoC used `10.{subnet}.{host}.1` addresses, which all share first octet `10` and vary only the second octet — producing groups of ~64 entries each, not 4. The eviction threshold `> 4` fires correctly for those large groups, so the original test would not demonstrate the bug. The corrected PoC (varying both first and second octets across 64×64 = 4 096 distinct /16 subnets, 4 addresses each) is structurally correct and would reproduce the failure.

## Impact Explanation

With the store permanently saturated, no new honest peer address can be stored. `fetch_addrs_to_feeler` returns only attacker-controlled addresses (all have `last_connected_at_ms = 0`, satisfying the "never connected" filter). If the attacker operates nodes at those addresses, feeler connections succeed and those entries are promoted into `fetch_addrs_to_attempt` candidates. After a node restart the victim makes all outbound connections to attacker nodes — a full eclipse. An eclipsed node can be fed a fake chain, have its transactions censored, or be used to facilitate double-spend attacks. This matches **Critical (15 001 – 25 000 points): Vulnerabilities which could easily damage CKB economy**.

## Likelihood Explanation

The attacker needs only one TCP connection to the victim to inject discovery `Nodes` messages. Addresses do not need to be reachable at injection time — they only need to be synthesized from 4 096 distinct /16 subnets with exactly 4 addresses each. Approximately 17 discovery messages of 1 000 addresses each suffice to fill 16 384 slots. No rate-limit on discovery address ingestion is present. The attack is low-cost, requires no special privilege, and is repeatable.

## Recommendation

Change the eviction threshold from strictly greater than 4 to greater than or equal to 4 in `check_purge`:

```rust
// network/src/peer_store/peer_store_impl.rs, line 378
// Before:
if addrs.len() > 4 {
// After:
if addrs.len() >= 4 {
```

Additionally, consider adding a per-session rate limit on the number of discovery addresses accepted per time window, and logging `EvictionFailed` at `warn` or `error` level rather than `debug` so operators are alerted when the store cannot be purged.

## Proof of Concept

```rust
#[test]
fn test_eclipse_via_exact_group_saturation() {
    use crate::peer_store::{ADDR_COUNT_LIMIT, PeerStore};
    use crate::Flags;
    use p2p::{multiaddr::Multiaddr, secio::PeerId};

    let mut peer_store = PeerStore::default();

    // Fill store: 4096 /16 groups × 4 addresses = 16384 (ADDR_COUNT_LIMIT)
    // Vary BOTH first and second octets to get distinct Group::IP4([a, b]) keys.
    let mut count = 0usize;
    'outer: for a in 1u8..=64u8 {
        for b in 0u8..=63u8 {
            for port in 1u16..=4 {
                if count >= ADDR_COUNT_LIMIT { break 'outer; }
                let addr: Multiaddr = format!(
                    "/ip4/{}.{}.1.1/tcp/{}/p2p/{}",
                    a, b, port,
                    PeerId::random().to_base58()
                ).parse().unwrap();
                peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
                count += 1;
            }
        }
    }
    assert_eq!(peer_store.addr_manager().count(), ADDR_COUNT_LIMIT);

    // Now try to add an honest address — must succeed but currently fails
    let honest: Multiaddr = format!(
        "/ip4/8.8.8.8/tcp/8333/p2p/{}", PeerId::random().to_base58()
    ).parse().unwrap();
    let result = peer_store.add_addr(honest.clone(), Flags::COMPATIBILITY);
    // Fails on current code: check_purge returns Err(EvictionFailed)
    assert!(result.is_ok(), "honest address must always be accepted");
}
```

With this corrected structure every group has exactly 4 entries, `addrs.len() > 4` is false for all of them, pass 2 produces an empty candidate list, and `Err(EvictionFailed)` is returned — confirming the bug.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times

            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
