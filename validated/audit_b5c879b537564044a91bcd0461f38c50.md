### Title
Unauthenticated Block Template Injection via Miner Notify HTTP Server - (`miner/src/client.rs`)

### Summary

The CKB miner client's notify-mode HTTP server (`listen_block_template_notify`) accepts block template updates from any HTTP client without any authentication or source verification. The `handle` function processes every incoming request and calls `update_block_template`, immediately replacing the miner's active work. Any attacker who can reach the miner's configured listen port can inject arbitrary block templates, causing the miner to waste all hashpower on attacker-controlled (invalid or stale) work, resulting in complete mining revenue loss for the duration of the attack.

### Finding Description

When the miner is configured in notify mode (`listen` option set in `ckb-miner.toml`), `spawn_background` calls `listen_block_template_notify` which binds a TCP listener and serves every connection through the `handle` function: [1](#0-0) 

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();
    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }
    Ok(Response::new(Empty::new()))
}
```

There is no check on the request source IP, no `Authorization` header validation, no shared secret, and no HMAC. Any HTTP POST with a valid `BlockTemplate` JSON body is accepted unconditionally.

`update_block_template` then replaces the current work and notifies all worker threads: [2](#0-1) 

The listen address is a user-configurable `SocketAddr`: [3](#0-2) 

When set to a non-loopback address (e.g., `0.0.0.0:8888`), the server is reachable by any network peer. The `parse_authorization` function exists only for the miner's **outgoing** RPC calls to the CKB node: [4](#0-3) 

It is never applied to the **incoming** notify server, creating an asymmetric and exploitable gap.

The CKB node's block assembler sends legitimate templates to this endpoint: [5](#0-4) 

An attacker who can reach the same port can send competing, attacker-crafted templates at any time.

**Exploit flow:**

1. Miner operator enables notify mode: `listen = "0.0.0.0:8888"` in `ckb-miner.toml`.
2. Attacker discovers the open port (port scan or knowledge of the default example address).
3. Attacker sends a crafted HTTP POST with a `BlockTemplate` JSON body containing a stale `parent_hash`, wrong `transactions_root`, or a `work_id` chosen to prevent future legitimate updates.
4. `handle` deserializes the body and calls `update_block_template` with no validation.
5. All miner workers immediately switch to the attacker's template and begin solving PoW on it.
6. Every solved nonce is submitted to the CKB node via `submit_block`, which rejects it (wrong parent, invalid merkle root, etc.).
7. The attacker repeats the injection continuously, keeping the miner permanently on invalid work.

### Impact Explanation

The miner wastes 100% of its hashpower on blocks that the CKB node will always reject. The miner earns zero block rewards for the duration of the attack. Because the attacker only needs to send a small HTTP POST periodically (far cheaper than the miner's electricity cost), the attack is asymmetric and sustainable. The miner has no way to distinguish the injected template from a legitimate one at the `handle` layer.

The `CellbaseVerifier` and `HeaderVerifier` in the CKB node will reject the submitted blocks, but the damage (wasted hashpower, lost revenue) is already done before submission. [6](#0-5) 

### Likelihood Explanation

The `listen` option is documented and supported. Mining pools and professional miners commonly expose the miner process on a non-loopback address to allow the CKB node (running on a separate machine) to push templates. Any attacker who can reach the miner's network port — via internet exposure, a compromised co-located service, or a malicious peer on the same LAN — can trigger this with a single HTTP request. No credentials, keys, or privileged access are required.

### Recommendation

The `handle` function must verify the request source before accepting a template update. Options include:

1. **Source IP allowlist**: Accept connections only from the configured CKB node's IP address.
2. **Shared secret / HMAC**: The CKB node signs the template body with a pre-shared key; the miner verifies the signature in `handle` before calling `update_block_template`.
3. **Bearer token**: Require an `Authorization` header with a secret token, consistent with the existing `parse_authorization` pattern already used for outgoing RPC calls.

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}
