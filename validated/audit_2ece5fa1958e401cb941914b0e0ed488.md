### Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` — (`sync/src/types/mod.rs`)

### Summary

An unprivileged remote peer can cause unbounded heap memory growth by repeatedly announcing the same set of tx hashes via `RelayTransactionHashes` messages. The `push_peer()` method appends unconditionally, and the only size guard in `add_ask_for_txs` is gated on the *number of distinct hashes* in the map — which never grows when the attacker reuses the same hashes. The result is a memory-exhaustion DoS.

---

### Finding Description

**Entry point**: `TransactionHashesProcess::execute()` in `sync/src/relayer/transaction_hashes_process.rs`.

Each incoming `RelayTransactionHashes` message is checked for count (`> MAX_RELAY_TXS_NUM_PER_BATCH = 32767`) and then filtered through `tx_filter` (which only contains *resolved/known* hashes). Hashes still pending in `unknown_tx_hashes` are **not** in `tx_filter`, so they pass through every time. [1](#0-0) 

The filtered hashes are passed to `SyncState::add_ask_for_txs()`. For each hash already present in the `unknown_tx_hashes` priority queue (the `Occupied` branch), it calls `push_peer()` unconditionally: [2](#0-1) 

`push_peer()` has no deduplication and no cap: [3](#0-2) 

The only guard against abuse is the size check at lines 1507–1529, which only activates when `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE (50000)` or `>= peers.len() * 32767`: [4](#0-3) 

**Critical bypass**: When the attacker reuses the same set of hashes (e.g., 100 hashes), the map never grows beyond 100 entries. The condition `100 >= 50000` is always false, so the guard **never executes**. The `peers` Vec for each of those 100 hashes grows by one entry per message, without limit.

The rate limiter in the relayer allows 30 messages/second per `(PeerIndex, message_type)` pair: [5](#0-4) 

At 30 msg/sec × 32767 hashes/msg × 4 bytes/`PeerIndex`, memory grows at ~3.9 MB/second per attacking peer. After one hour from a single peer: ~14 GB.

---

### Impact Explanation

**Memory exhaustion / OOM crash** of the CKB node. The `unknown_tx_hashes` Mutex-protected structure grows without bound in heap memory. The OS will eventually OOM-kill the process, causing a full node crash and denial of service. No block production, no transaction relay, no sync.

---

### Likelihood Explanation

- Requires only a standard P2P connection — no privileges, no PoW, no keys.
- The attacker needs to stay connected and send 30 messages/second (the rate limit), which is trivially achievable.
- The bypass of the size guard is structural: reusing the same hashes keeps `unknown_tx_hashes.len()` constant, so the guard is permanently skipped.
- A single peer is sufficient; multiple peers accelerate the attack.

---

### Recommendation

1. **Deduplicate in `push_peer()`**: Before appending, check `if !self.peers.contains(&peer_index)`.
2. **Cap `peers` Vec length**: Enforce a hard maximum (e.g., 128 or `MAX_RELAY_PEERS`) inside `push_peer()`.
3. **Move the per-peer guard before the push**: Count how many times `peer_index` already appears in the entry's `peers` Vec before calling `push_peer()`, not only when the total map size threshold is exceeded.

---

### Proof of Concept

1. Connect to a CKB node as a peer.
2. Construct 100 random (non-existent) tx hashes `[h1..h100]`.
3. Send `RelayTransactionHashes { tx_hashes: [h1..h100] }` at 30 messages/second (within the rate limit).
4. After N seconds, assert `unknown_tx_hashes[h_i].peers.len() == N * 30 + 1` for each hash.
5. Monitor node RSS memory — it grows linearly at ~3.9 MB/sec.
6. After sufficient time, the node OOM-crashes.

The hashes never enter `tx_filter` (they are never resolved), so the filter provides no protection. The map size stays at 100, so the guard at line 1507 never fires. [6](#0-5) [7](#0-6)

        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }
