### Title
Hardcoded `TRANSACTION_SIZE_LIMIT` Diverges from Consensus `max_block_bytes`, Causing Valid Transactions to Be Silently Rejected by Tx-Pool — (File: `util/types/src/core/tx_pool.rs`)

---

### Summary

The tx-pool enforces a hardcoded per-transaction size ceiling (`TRANSACTION_SIZE_LIMIT = 512 * 1_000` bytes) that is never derived from the consensus-configured `max_block_bytes` (597,000 bytes on mainnet). This creates a split-brain admission policy: the consensus-aware `SizeVerifier` inside `NonContextualTransactionVerifier` accepts transactions up to 597,000 bytes, but the subsequent hardcoded check in `non_contextual_verify` rejects any transaction larger than 512,000 bytes. A transaction sender who submits a consensus-valid transaction in the 512,001–597,000 byte range receives a spurious `ExceededTransactionSizeLimit` rejection from the tx-pool, even though the same transaction would be accepted in a mined block.

---

### Finding Description

**Root cause — hardcoded constant not derived from consensus:**

`TRANSACTION_SIZE_LIMIT` is defined as a bare compile-time constant:

```rust
// util/types/src/core/tx_pool.rs  line 309
pub const TRANSACTION_SIZE_LIMIT: u64 = 512 * 1_000;
```

The comment above it acknowledges the relationship to block size but never reads the actual consensus value:

```rust
// lines 305-309
/// The maximum size of the tx-pool to accept transactions
/// The ckb consensus does not limit the size of a single transaction,
/// but if the size of the transaction is close to the limit of the block,
/// it may cause the transaction to fail to be packed
pub const TRANSACTION_SIZE_LIMIT: u64 = 512 * 1_000;
```

**Two-stage size check with inconsistent limits:**

`non_contextual_verify` in `tx-pool/src/util.rs` runs two independent size checks back-to-back:

```rust
// tx-pool/src/util.rs  lines 56-73
pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)   // ← uses consensus.max_block_bytes()
        .verify()
        .map_err(Reject::Verification)?;

    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {                  // ← uses hardcoded 512_000
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    ...
}
```

Inside `NonContextualTransactionVerifier::new`, the `SizeVerifier` is constructed with the consensus value:

```rust
// verification/src/transaction_verifier.rs  lines 82-91
pub fn new(tx: &'a TransactionView, consensus: &'a Consensus) -> Self {
    NonContextualTransactionVerifier {
        ...
        size: SizeVerifier::new(tx, consensus.max_block_bytes()),  // 597_000 on mainnet
        ...
    }
}
```

`SizeVerifier::verify` accepts the transaction if `tx_size <= 597_000`. The subsequent hardcoded check then rejects it if `tx_size > 512_000`. The two checks use different, unrelated limits derived from different sources.

**Mainnet numbers:**

```
MAX_BLOCK_BYTES  = TWO_IN_TWO_OUT_BYTES * TWO_IN_TWO_OUT_COUNT
                 = 597 * 1_000 = 597_000   (spec/src/consensus.rs lines 72-83)

TRANSACTION_SIZE_LIMIT = 512 * 1_000 = 512_000
```

The gap is 85,000 bytes. Any transaction in the range [512,001 – 597,000] bytes is consensus-valid but tx-pool-rejected.

---

### Impact Explanation

A transaction sender (RPC caller / tx-pool submitter) who constructs a consensus-valid transaction whose serialized block size falls between 512,001 and 597,000 bytes will receive `Reject::ExceededTransactionSizeLimit` from the tx-pool. The transaction is permanently unsubmittable through the standard `send_transaction` RPC path. Because the rejection reason is `ExceededTransactionSizeLimit` rather than a consensus error, the sender has no way to distinguish this from a genuine protocol violation. The transaction is valid and could be included in a block by a miner who bypasses the tx-pool, but no normal propagation path exists for it. This is a direct analog to the pyUmbral scenario: a hardcoded size assumption that does not track the actual configured parameter causes correct inputs to be silently discarded.

---

### Likelihood Explanation

The discrepancy exists on mainnet today (597,000 vs 512,000). Any script author or application developer who constructs a large transaction — for example, a cell containing a large RISC-V binary or a batch of outputs — and whose serialized transaction size lands in the 85 KB gap will hit this rejection. The entry path is the standard `send_transaction` JSON-RPC call, requiring no special privileges.

---

### Recommendation

Replace the hardcoded constant with a value derived from the consensus parameter. The simplest fix is to pass `consensus.max_block_bytes()` (or a fraction thereof to preserve the intended headroom) into `non_contextual_verify` and remove the separate `TRANSACTION_SIZE_LIMIT` constant, or redefine it as:

```rust
// derive from consensus instead of hardcoding
let tx_size_limit = consensus.max_block_bytes() * 512 / 597; // preserve ~85 KB headroom
if tx_size > tx_size_limit { ... }
```

This mirrors the pyUmbral recommendation: replace the hardcoded `197` with `6 * key_size + 5` — i.e., derive the limit from the actual configured parameter rather than assuming a fixed default.

---

### Proof of Concept

1. Start a CKB node with the default mainnet chain spec (`max_block_bytes = 597_000`).
2. Construct a transaction whose `serialized_size_in_block()` is, e.g., 550,000 bytes (e.g., a cell output with a ~549 KB data blob).
3. Submit via `send_transaction` RPC.
4. **Expected (per consensus):** accepted — `550_000 ≤ 597_000`.
5. **Actual:** rejected with `ExceededTransactionSizeLimit(550000, 512000)` because `550_000 > TRANSACTION_SIZE_LIMIT`.
6. Confirm the transaction is consensus-valid by having a miner include it directly in a block template — it will pass `BlockBytesVerifier` and all contextual checks.

```rust
impl<'a> SizeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView, block_bytes_limit: u64) -> Self {
        SizeVerifier {
            transaction,
            block_bytes_limit,
        }
    }

    pub fn verify(&self) -> Result<(), Error> {
        let size = self.transaction.data().serialized_size_in_block() as u64;
        if size <= self.block_bytes_limit {
            Ok(())
        } else {
            Err(TransactionError::ExceededMaximumBlockBytes {
                actual: size,
                limit: self.block_bytes_limit,
            }
            .into())
        }
    }
}
```
