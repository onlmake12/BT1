### Title
Unrestricted `create_stake_account` Allows Anyone to Impose an Arbitrary Vesting Lock on Any User's Stake Account - (File: `governance/pyth_staking_sdk/src/idl/staking.json`)

---

### Summary

The `create_stake_account` instruction in the Pyth staking program is explicitly documented as "trustless" — any transaction sender can call it to create a stake account for an arbitrary `owner` pubkey with an arbitrary `VestingSchedule` lock. Because the vesting schedule is immutable after account creation and directly restricts token withdrawals and Integrity Staking eligibility, a malicious actor can create a poisoned stake account for any victim with a maximally restrictive vesting schedule, locking any tokens the victim deposits into that account indefinitely.

---

### Finding Description

The `create_stake_account` instruction accepts two caller-controlled arguments — `owner` (pubkey) and `lock` (VestingSchedule) — with no authentication check requiring the `owner` to sign the transaction. [1](#0-0) 

The instruction is explicitly labeled "Trustless instruction that creates a stake account for a user." The only required signer is `payer`, which can be any account. [2](#0-1) 

The `stakeAccountPositions` account passed to the instruction is an unconstrained writable account — it is not a PDA derived from the `owner`'s pubkey. Any attacker can create an account owned by the staking program (via `SystemProgram.createAccount`) and pass it as `stakeAccountPositions`. [3](#0-2) 

The resulting `StakeAccountMetadataV2` PDA stores the attacker-supplied `owner` and `lock` fields permanently: [4](#0-3) 

The `VestingSchedule` enum supports `PeriodicVesting` with arbitrary `initial_balance`, `start_date`, `period_duration`, and `num_periods`: [5](#0-4) 

There is no instruction in the staking program to update the `lock` field after account creation. The vesting schedule is set once and is permanent.

The `lock` field directly controls how much of the custody balance is considered "locked." The UI computes `availableToStakeIntegrity` as `total - locked - ...`, and `withdraw_stake` enforces `InsufficientWithdrawableBalance` (error 6013) when the requested amount exceeds the unlocked balance: [6](#0-5) 

The vesting schedule unlock logic in the SDK shows that `PeriodicVesting` with a far-future `start_date` and large `initial_balance` would report the entire balance as locked for the full vesting duration: [7](#0-6) 

The UI subtracts the locked amount from the total to determine what is available to withdraw or stake: [8](#0-7) 

---

### Impact Explanation

An attacker who creates a stake account for a victim with `PeriodicVesting { initial_balance: u64::MAX, start_date: <far future>, period_duration: u64::MAX, num_periods: 1 }` causes the following for any tokens the victim deposits into that account:

- **Withdrawal blocked**: `withdraw_stake` will revert with `InsufficientWithdrawableBalance` because the entire balance is reported as locked.
- **Integrity Staking blocked**: The locked amount is subtracted from `availableToStakeIntegrity`, preventing OIS participation.
- **No recovery path**: There is no instruction to update or override the vesting schedule.

The victim's tokens remain permanently locked until the vesting schedule expires (which the attacker can set to an arbitrarily distant date).

---

### Likelihood Explanation

The attacker entry path requires only:
1. Knowledge of the victim's pubkey (publicly available on-chain).
2. Payment of rent for a new positions account (~0.002 SOL).
3. The victim depositing tokens into the attacker-created account.

Step 3 is realistic because:
- Any UI that discovers stake accounts by querying `StakeAccountMetadataV2` accounts filtered by `owner == victim_pubkey` (a standard Solana `getProgramAccounts` pattern) will surface the attacker-created account alongside legitimate ones.
- The attacker can also front-run a victim's `create_stake_account` call if the victim creates the positions account in a separate transaction from the metadata initialization — the attacker observes the positions account on-chain and calls `create_stake_account` on it first with a restrictive lock, causing the victim's subsequent call to fail (metadata PDA already initialized). [9](#0-8) 

---

### Recommendation

1. **Require the `owner` to sign `create_stake_account`**: Add a constraint that the `owner` argument must match a signer in the transaction. This eliminates the ability to create accounts on behalf of others without consent.
2. **Restrict trustless creation to `FullyVested`**: If permissionless creation is required for airdrop/distribution scenarios, enforce that the `lock` argument must be `FullyVested` when the `owner` is not a signer.
3. **Whitelist callers**: Introduce a set of authorized callers (e.g., a distribution program) that are permitted to set non-`FullyVested` schedules on behalf of others.

export const getPeriodicUnlockSchedule = (options: {
  balance: bigint;
  startDate: bigint;
  periodDuration: bigint;
  numPeriods: bigint;
  includePastPeriods: boolean;
}): UnlockSchedule["schedule"] => {
  const { balance, startDate, periodDuration, numPeriods, includePastPeriods } =
    options;

  const unlockSchedule: UnlockSchedule["schedule"] = [];
  const currentTimeStamp = Date.now() / 1000;

  for (let i = 0; i < numPeriods; i++) {
    const unlockTimeStamp =
      Number(startDate) + Number(periodDuration) * (i + 1);
    if (currentTimeStamp < unlockTimeStamp || includePastPeriods) {
      unlockSchedule.push({
        amount:
          ((numPeriods - BigInt(i)) * balance) / numPeriods -
          ((numPeriods - BigInt(i + 1)) * balance) / numPeriods,
        date: new Date(unlockTimeStamp * 1000),
      });
    }
  }

  return unlockSchedule;
