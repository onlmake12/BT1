### Title
Semantic Overloading of `TxStatus::Unknown` Conflates "Never Seen" and "Rejected-but-Evicted" Transaction States — (`util/types/src/core/tx_pool.rs`)

---

### Summary

The `TxStatus::Unknown` variant in CKB's tx-pool type system carries two semantically distinct meanings within a single enum variant. This is a direct analog to the Audius `No`-outcome overload: a single discriminant is used to represent two fundamentally different situations, making it impossible for RPC callers to distinguish them.

---

### Finding Description

In `util/types/src/core/tx_pool.rs`, the `TxStatus` enum is defined as:

```rust
/// Tx-pool transaction status
#[derive(Clone, Debug, PartialEq, Eq)]
pub enum TxStatus {
    Pending,
    Proposed,
    Committed(BlockNumber, H256, u32),
    /// Status "unknown". The node has not seen the transaction,
    /// or it should be rejected but was cleared due to storage limitations.
    Unknown,
    Rejected(String),
}
``` [1](#0-0) 

The comment on `Unknown` explicitly documents two distinct semantic states collapsed into one variant:

1. **"The node has not seen the transaction"** — the transaction was never submitted to or observed by this node.
2. **"it should be rejected but was cleared due to storage limitations"** — the transaction *was* submitted, *was* rejected, but the bounded recent-reject store evicted the rejection record.

The identical dual meaning is mirrored in the JSON-RPC surface type `Status::Unknown` in `util/jsonrpc-types/src/blockchain.rs`:

```rust
/// Status "unknown". The node has not seen the transaction,
/// or it should be rejected but was cleared due to storage limitations.
Unknown,
``` [2](#0-1) 

The conversion from the internal type to the JSON-RPC type maps `TxStatus::Unknown` directly to `TxStatus::unknown()` with no disambiguation: [3](#0-2) 

The `TransactionWithStatus::with_unknown()` constructor is the single code path used for both cases: [4](#0-3) 

---

### Impact Explanation

Any RPC caller — a wallet, exchange, dApp, or monitoring system — that queries `get_transaction` or `get_transaction_with_status` for a transaction that was **rejected and then evicted from the recent-reject store** will receive `status: "unknown"`. This is indistinguishable from a transaction that was **never submitted at all**.

Concrete consequences:
- A wallet that submitted a double-spend transaction, received a rejection, but then re-queries after the rejection record is evicted, will see `Unknown` and may incorrectly conclude the transaction was never received — and resubmit it indefinitely.
- An exchange monitoring deposit transactions cannot distinguish "we never saw this tx" from "we saw it, rejected it, and forgot." Both look identical.
- Future protocol or application code that branches on `TxStatus::Unknown` will silently apply the same logic to both cases, opening the door to regressions — exactly the pattern the Audius report warns about.

---

### Likelihood Explanation

The recent-reject store in CKB's tx-pool is **bounded by storage limits** (as documented in the `Rejected` variant's own comment: *"Due to storage limitations, the node can only hold the most recently removed transactions"*). Under normal mainnet operation — high transaction volume, node restarts, or pool evictions — rejection records are routinely cleared. Any caller that queries status after eviction hits the overloaded `Unknown` path. This is a realistic, non-adversarial operational scenario reachable by any unprivileged RPC caller. [5](#0-4) 

---

### Recommendation

Split `TxStatus::Unknown` into two distinct variants:

```rust
pub enum TxStatus {
    Pending,
    Proposed,
    Committed(BlockNumber, H256, u32),
    /// The node has genuinely never seen this transaction.
    Unknown,
    /// The transaction was rejected, but the rejection record was evicted
    /// from the bounded recent-reject store due to storage limitations.
    RejectedAndEvicted,
    Rejected(String),
}
```

Apply the same split to the JSON-RPC `Status` enum and update all construction sites (`with_unknown()`, pool eviction paths) to use the correct variant. This allows callers to unambiguously distinguish "never seen" from "seen, rejected, record cleared."

---

### Proof of Concept

1. Submit a transaction `T` that is invalid (e.g., double-spend). The node rejects it and records the rejection in the bounded recent-reject store with `TxStatus::Rejected(reason)`.
2. Submit enough subsequent rejected transactions to fill and evict `T`'s rejection record from the store (or restart the node, which clears the in-memory recent-reject store).
3. Query `get_transaction` or `get_transaction_with_status` for `T`'s hash via RPC.
4. The node finds `T` neither in the pool, nor in the chain, nor in the recent-reject store. It returns `TransactionWithStatus::with_unknown()`.
5. The RPC response is `{"status": "unknown"}` — identical to a transaction that was never submitted. The caller has no way to know `T` was previously rejected. [6](#0-5) [4](#0-3)
