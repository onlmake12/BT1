### Title
Stale Descendant Statistics on Ancestors After `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

When a transaction subtree is removed from the CKB tx-pool via `remove_entry_and_descendants` (triggered during RBF replacement, pool size eviction, or conflict resolution), the **ancestors** of the removed root retain stale `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` values. This is because all link entries are pre-cleared before `remove_entry` can walk the link graph to update ancestor statistics. The result is that surviving ancestor transactions carry an inflated `evict_key`, making them resistant to eviction even when their fee rate no longer justifies pool residency.

---

### Finding Description

`remove_entry_and_descendants` first strips all link entries for every node in the subtree, then calls `remove_entry` on each: [1](#0-0) 

Inside `remove_entry`, the call to `update_ancestors_index_key` is supposed to walk the link graph upward and subtract the removed entry's weight from each surviving ancestor's `descendants_*` fields: [2](#0-1) 

`update_ancestors_index_key` resolves ancestors via `self.links.calc_ancestors`: [3](#0-2) 

Because `remove_entry_links` was already called for the root in the pre-clearing loop, the root's link entry is gone by the time `update_ancestors_index_key` runs. `calc_ancestors` therefore returns an empty set, and **no ancestor has its descendant statistics decremented**.

`remove_entry_links` removes the root from its parents' children lists and deletes the root's own link record: [4](#0-3) 

The `evict_key` of each surviving ancestor is computed from `descendants_fee` and `descendants_size`: [5](#0-4) 

Because `descendants_fee` is never decremented, the `evict_key` remains inflated after the subtree is removed.

---

### Impact Explanation

The `limit_size` eviction loop selects the next victim by `iter_by_evict_key()` (lowest key first): [6](#0-5) 

A surviving ancestor with a stale, inflated `evict_key` is ranked as if it still has high-fee descendants. It is therefore skipped during eviction even when its own fee rate is low. Legitimate high-fee transactions submitted by other users may be rejected with `Reject::Full` while the low-fee ancestor persists indefinitely.

---

### Likelihood Explanation

RBF is enabled whenever `min_rbf_rate > min_fee_rate`: [7](#0-6) 

An unprivileged tx-pool submitter can trigger this path by:
1. Submitting a low-fee parent transaction (tx_A).
2. Submitting a high-fee child transaction (tx_B, spending tx_A's output).
3. Replacing tx_B with tx_C via RBF — `process_rbf` calls `remove_entry_and_descendants` on tx_B.

After step 3, tx_A's `descendants_fee` still includes tx_B's fee. The pool is near-full in practice on mainnet, making the eviction path active. The default `min_rbf_rate` of 1500 shannons/KB vs `min_fee_rate` of 1000 shannons/KB means RBF is on by default. [8](#0-7) 

---

### Recommendation

In `remove_entry_and_descendants`, update the ancestors of the subtree root **before** clearing any links. Specifically, before the pre-clearing loop, call `update_ancestors_index_key` for the root entry using the still-intact link graph, so that each surviving ancestor has its `descendants_*` fields and `evict_key` correctly decremented. Alternatively, restructure `remove_entry` to accept an optional pre-computed ancestor set so that the link graph does not need to be traversed after it has been cleared.

---

### Proof of Concept

```
Pool state: tx_A (fee=1, parent) → tx_B (fee=1000, child)
tx_A.evict_key reflects descendants_fee = 1001

Step 1: Attacker submits tx_C (fee=1500, same inputs as tx_B) → RBF accepted
Step 2: process_rbf calls remove_entry_and_descendants(tx_B)
  - remove_entry_links(tx_B): removes tx_B from tx_A's children list, deletes tx_B's link record
  - remove_entry(tx_B): update_ancestors_index_key(tx_B, Remove)
      → calc_ancestors(tx_B) returns {} (link record gone)
      → tx_A.descendants_fee NOT decremented (still = 1001)
      → tx_A.evict_key NOT updated

Step 3: Pool fills up; limit_size runs
  - tx_A.evict_key is ranked as if descendants_fee = 1001
  - tx_A is skipped for eviction
  - A legitimate tx_D (fee=500, no children) is evicted instead
  - tx_A (fee=1) persists in the pool indefinitely
```

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

    // Remove transactions from the pool until total size <= size_limit.
    // Return a `Reject` for current inserting entry if it's removed
    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
