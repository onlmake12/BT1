### Title
Stale `descendants_*` Accounting in `remove_entry_and_descendants` Leaves Surviving Ancestors with Inflated Eviction Keys — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link records for every entry in the removal batch before calling `remove_entry` on each one. Because `remove_entry` relies on the live link graph to find and update surviving ancestors' `descendants_*` fields and `evict_key`, those fields are never decremented. Surviving ancestors permanently carry inflated descendant-fee/size/cycle counts and a stale eviction key, corrupting the pool's eviction ordering.

---

### Finding Description

`remove_entry_and_descendants` collects the target entry and all its descendants, then calls `remove_entry_links` for every entry in that set **before** calling `remove_entry` on any of them:

```rust
// tx-pool/src/component/pool_map.rs
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ← ALL links torn down here
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← remove_entry now sees empty link graph
        .collect()
}
``` [1](#0-0) 

Inside `remove_entry`, the first thing done is `update_ancestors_index_key`, which walks the live link graph to find every surviving ancestor and call `sub_descendant_weight` + refresh `evict_key` on each:

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();   // ← evict_key refreshed here
        });
    }
}
``` [2](#0-1) 

Because `remove_entry_links` already removed the entry from the link map, `calc_ancestors` returns an empty set. No surviving ancestor ever has `sub_descendant_weight` called on it, and no surviving ancestor's `evict_key` is refreshed.

By contrast, the single-entry `remove_entry` path calls `remove_entry_links` **after** `update_ancestors_index_key`, so the link graph is still intact when the ancestor walk happens — the correct order: [3](#0-2) 

The fields left stale on surviving ancestors are `descendants_count`, `descendants_fee`, `descendants_size`, `descendants_cycles`, and the derived `evict_key`: [4](#0-3) 

The `EvictKey` is computed directly from those fields: [5](#0-4) 

---

### Impact Explanation

`evict_key` is the sole ordering criterion used by `next_evict_entry`, which is called inside `limit_size` whenever `total_tx_size > max_tx_pool_size`: [6](#0-5) 

A surviving ancestor whose child was removed retains the child's fee/size/cycles in its own `descendants_fee`/`descendants_size`/`descendants_cycles`. Its apparent `descendants_feerate` is therefore computed over a larger (phantom) package than actually exists. Depending on whether the removed child had a higher or lower fee rate than the ancestor, the ancestor's eviction priority is silently skewed in either direction. The pool will evict the wrong transactions when it is full, and the stale state persists for the lifetime of the ancestor entry.

---

### Likelihood Explanation

The bug is triggered by every code path that calls `remove_entry_and_descendants`:

- **RBF replacement** (`process_rbf` in `tx-pool/src/process.rs`) — any tx-pool submitter can trigger this by submitting a replacement transaction.
- **Conflict resolution** (`resolve_conflict` / `resolve_conflict_header_dep`) — triggered by block relay or header relay.
- **Expiry eviction** (`remove_expired`) — triggered automatically by the pool's expiry timer.
- **Size-limit eviction** (`limit_size`) — triggered whenever the pool is full.
- **Manual removal** (`remove_tx` RPC) — callable by any local RPC user. [1](#0-0) 

Any unprivileged transaction sender can deliberately trigger this: submit a parent transaction A (low fee rate) and a child transaction B (high fee rate), then replace B via RBF. After B is removed, A's `evict_key` still reflects B's fee, making A appear to have a higher fee rate than it actually does. A is then systematically skipped during eviction, allowing a low-fee-rate transaction to occupy pool space indefinitely and crowd out legitimate higher-fee-rate transactions.

---

### Recommendation

Move the ancestor-index update **before** tearing down the link graph. One correct approach:

1. For each entry in `removed_ids`, call `update_ancestors_index_key(entry, EntryOp::Remove)` while links are still intact (restricting updates to ancestors **not** in `removed_ids`).
2. Only then call `remove_entry_links` and `remove_entry` for each entry.

Alternatively, replicate the single-entry `remove_entry` ordering (links removed last) and guard against double-updates within the batch explicitly.

---

### Proof of Concept

Given chain A → B → C (A is parent, B is child, C is grandchild):

1. Submit A, B, C to the tx-pool (all accepted).
2. Submit a conflicting transaction B′ that replaces B via RBF.
3. `process_rbf` calls `remove_entry_and_descendants(B)`, removing B and C.
4. Inspect A's entry: `descendants_count` is still 3 (should be 1), `descendants_fee` still includes B.fee + C.fee, `evict_key` is stale.
5. Fill the pool to capacity with many medium-fee-rate transactions.
6. Observe that `limit_size` skips evicting A (because A's apparent `descendants_feerate` is inflated by B and C's fees), even though A's actual fee rate is below the eviction threshold.
7. Legitimate transactions at A's true fee rate are rejected while A remains in the pool. [7](#0-6) [8](#0-7)
