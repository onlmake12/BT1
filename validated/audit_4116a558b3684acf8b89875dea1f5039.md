### Title
Stale Ancestor Descendant-Weight Tracking After `remove_entry_and_descendants` Corrupts Pool Eviction Ordering - (`tx-pool/src/component/pool_map.rs`)

### Summary

`PoolMap::remove_entry_and_descendants` removes all parent-child links **before** calling `remove_entry` on each removed entry. Because `update_ancestors_index_key` resolves ancestors through those same links, it finds an empty ancestor set and never decrements the `descendants_*` fields of transactions that remain in the pool. Those remaining ancestors permanently carry inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`, which corrupts their `EvictKey`. An unprivileged tx-pool submitter can exploit this to make a low-fee transaction effectively immune to eviction, forcing the pool to evict legitimate higher-fee transactions instead.

### Finding Description

**Root cause — link removal order in `remove_entry_and_descendants`:**

```rust
// tx-pool/src/component/pool_map.rs  lines 252-265
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ← ALL links torn down here
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← remove_entry called after links are gone
        .collect()
}
```

`remove_entry` then calls `update_ancestors_index_key`:

```rust
// lines 432-445
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());  // ← returns ∅ because links gone
    for anc_id in &ancestors {          // ← loop body never executes
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
```

Because the ancestor set is empty, `sub_descendant_weight` is never called on any ancestor that remains in the pool. Their `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` are never decremented.

**How `EvictKey` is computed from these stale fields:**

```rust
// tx-pool/src/component/entry.rs  lines 234-247

**Exploit path (RBF-enabled node):**

1. Attacker submits `tx_A` with a low fee rate.
2. Attacker submits `tx_B` spending `tx_A`'s output with a very high fee. `tx_A.descendants_fee` and `tx_A.descendants_feerate` are now large; `tx_A.evict_key.fee_rate = descendants_feerate` (high).
3. Attacker submits `tx_C` that conflicts with `tx_B` via RBF. `resolve_conflict` calls `remove_entry_and_descendants(&tx_B_id)`. Links are torn down first; `tx_A`'s `descendants_*` are never decremented.
4. `tx_A` now has `descendants_count = 2`, `descendants_fee = fee_A + fee_B`, `evict_key.fee_rate` = the high rate of the now-removed `tx_B`.
5. Steps 2–3 can be repeated to further inflate `tx_A`'s `descendants_count` without bound (saturating arithmetic in `add_descendant_weight` allows this).
6. When the pool is full, `tx_A` is never selected for eviction; legitimate transactions with correct (lower) `EvictKey` values are evicted instead.

The same divergence occurs without RBF whenever `remove_entry_and_descendants` is triggered by pool-size eviction of a child transaction.

### Impact Explanation

An unprivileged attacker can permanently anchor a low-fee transaction in the pool by temporarily attaching a high-fee child and then removing it via RBF (or by letting the child be evicted). The ancestor's `EvictKey` becomes permanently inflated, making it immune to the pool's size-limit eviction logic. Legitimate transactions submitted by other users are evicted in its place. Repeated application across multiple attacker-controlled transactions can degrade the pool's ability to enforce its fee-rate floor, constituting a sustained DoS against the tx-pool admission and eviction subsystem.

### Likelihood Explanation

RBF is a configurable feature (`min_rbf_rate > min_fee_rate`). On nodes where it is enabled, the attack requires only two ordinary `send_transaction` RPC calls per inflation cycle — no privileged access, no cryptographic break, and no majority hash power. The attacker pays the fee for `tx_B` but recovers it via `tx_C` (which replaces `tx_B`). The net cost per inflation cycle is only the fee difference required by RBF rules, making repeated exploitation economically feasible.

### Recommendation

In `remove_entry_and_descendants`, update ancestors' descendant weights **before** tearing down links. Specifically, for the root entry being removed (not its descendants, which are also being removed), call `update_ancestors_index_key` while links are still intact, then remove links:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Fix: update ancestors' descendant weights BEFORE removing links
    if let Some(root_entry) = self.get(id).cloned() {
        self.update_ancestors_index_key(&root_entry, EntryOp::Remove);
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

Alternatively, recompute `evict_key` for all remaining ancestors after the removal batch completes.

    pub fn add_descendant_weight(&mut self, entry: &TxEntry) {
        self.descendants_count = self.descendants_count.saturating_add(1);
        self.descendants_size = self.descendants_size.saturating_add(entry.size);
        self.descendants_cycles = self.descendants_cycles.saturating_add(entry.cycles);
        self.descendants_fee = Capacity::shannons(
            self.descendants_fee
                .as_u64()
                .saturating_add(entry.fee.as_u64()),
        );
    }

    // Return a `Reject` for current inserting entry if it's removed
    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));

                        ret = Some(reject.clone());
                    }

                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
