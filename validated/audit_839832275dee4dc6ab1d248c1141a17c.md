### Title
Excessive `msg.value` Permanently Lost to Pyth Fee Pool in Entropy Requests - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
`Entropy.requestHelper()` accepts any `msg.value >= requiredFee` without refunding the excess. The surplus is silently credited to `_state.accruedPythFeesInWei` (Pyth's treasury), permanently transferring user funds to the protocol with no recovery path.

### Finding Description
In `requestHelper()`, the fee check uses a strict less-than comparison:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The accounting on line 238–239 distributes the **entire** `msg.value` between the provider and Pyth: `providerFee` goes to the provider, and `msg.value - providerFee` goes to Pyth's accrued fees. Since `requiredFee = providerFee + pythFee`, any excess above `requiredFee` is silently absorbed into `accruedPythFeesInWei` rather than being returned to the caller.

The codebase itself acknowledges this behavior with explicit comments on the public-facing wrappers:

```solidity
// Note that excess value is *not* refunded to the caller.
function request(...) public payable override returns (uint64 assignedSequenceNumber) {
``` [2](#0-1) 

```solidity
// Note that excess value is *not* refunded to the caller.
function requestWithCallback(...) public payable override returns (uint64) {
``` [3](#0-2) 

All three public entry points — `request()`, `requestWithCallback()`, and `requestV2()` — route through `requestHelper()` and are affected. [4](#0-3) 

### Impact Explanation
**Impact: High** — Any ETH sent above `requiredFee` is permanently transferred to Pyth's fee pool (`accruedPythFeesInWei`). There is no user-facing refund or withdrawal mechanism. The funds are irrecoverable by the original sender. This is a direct, permanent loss of user funds.

### Likelihood Explanation
**Likelihood: Low** — Requires a user mistake: sending more ETH than the exact fee. This can realistically occur when:
- A frontend or SDK over-estimates the fee and pads the `msg.value`
- A user manually constructs a transaction with a rounded-up value
- A smart contract integration uses a stale or conservative fee estimate

The Entropy SDK and integrations that call `getFee()` / `getFeeV2()` and pass the result directly are safe, but any caller that adds a buffer (a common defensive pattern) silently loses the buffer amount.

### Recommendation
Change the fee check to require an exact match, or explicitly refund the excess:

```diff
- if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
+ if (msg.value != requiredFee) revert EntropyErrors.InsufficientFee();
```

Or alternatively, refund the surplus after accounting:

```solidity
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

### Proof of Concept
1. Provider is registered with `feeInWei = 1000 wei`, Pyth fee is `100 wei`. `requiredFee = 1100 wei`.
2. User calls `requestV2(provider, userContribution, gasLimit)` with `msg.value = 2000 wei` (e.g., due to a conservative SDK estimate).
3. `requestHelper` passes the check (`2000 >= 1100`).
4. `providerInfo.accruedFeesInWei += 1000` (correct provider share).
5. `_state.accruedPythFeesInWei += (2000 - 1000) = 1000` — Pyth receives `1000 wei` instead of the correct `100 wei`.
6. The user's `900 wei` excess is permanently absorbed into Pyth's treasury with no refund. [5](#0-4)
