### Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Silently Corrupts DAO Withdrawal Capacity — (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes a u128 intermediate value and casts it to u64 with a silent truncating `as u64`. Every other u128→u64 conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When the intermediate value eventually exceeds u64::MAX (reachable after the accumulate-rate `ar` grows ~5.5× from genesis, estimated ~43+ years of chain operation), the function silently returns a wrong withdrawal capacity instead of an error. This corrupts the DAO field computed for any block containing a DAO withdrawal, causing `DaoHeaderVerifier` to reject those blocks and effectively locking all DAO deposits.

---

### Finding Description

In `calculate_maximum_withdraw`:

```rust
// util/dao/src/lib.rs lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the high 64 bits when `withdraw_counted_capacity > u64::MAX`. Every other u128→u64 conversion in the same file uses the checked form:

- `secondary_block_reward` line 204: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- `dao_field_with_current_epoch` line 245: `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?`
- `dao_field_with_current_epoch` line 258: `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?` [2](#0-1) [3](#0-2) 

The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

`counted_capacity` is bounded by the total CKB supply (~3.36 × 10¹⁸ shannons). `u64::MAX ≈ 1.84 × 10¹⁹`. Therefore overflow occurs when `withdrawing_ar / deposit_ar > ~5.5`. Since `ar` starts at `DEFAULT_GENESIS_ACCUMULATE_RATE = 10¹⁶` and grows multiplicatively at ~10⁻⁸ per block (observed from mainnet data: block 0 `ar = 10¹⁶`, block 5892 `ar ≈ 1.0006 × 10¹⁶`), the 5.5× threshold is reached in approximately `log(5.5) / 10⁻⁸ ≈ 1.7 × 10⁸` blocks (~43 years). [4](#0-3) 

The genesis guard only protects `c` from being zero, not `ar`: [5](#0-4) 

---

### Impact Explanation

`calculate_maximum_withdraw` feeds into two critical paths:

1. **`withdrawed_interests` → `dao_field_with_current_epoch`**: The wrong withdrawal capacity produces a wrong `withdrawed_interests` sum, which corrupts the DAO field (`current_s`) packed into the block header. `DaoHeaderVerifier::verify` then rejects every block containing a DAO withdrawal transaction because `dao != self.header.dao()`. [6](#0-5) 

2. **`transaction_fee` → `check_tx_fee`**: The wrong capacity causes the fee to be computed incorrectly, potentially causing valid DAO withdrawal transactions to be rejected from the tx pool. [7](#0-6) 

3. **RPC `calculate_dao_maximum_withdraw`**: Returns a silently wrong value to callers. [8](#0-7) 

Net result: once `ar` crosses the threshold, no block containing a DAO withdrawal can be accepted by any honest node. All CKB locked in NervosDAO becomes unwithdrawable at the consensus layer.

---

### Likelihood Explanation

The overflow is not attacker-triggered — it is a deterministic consequence of chain age. Based on the observed `ar` growth rate (~10⁻⁸ per block), the 5.5× threshold is reached in approximately 43+ years of continuous chain operation. This is within the expected operational lifetime of the CKB mainnet. Any user submitting a DAO withdrawal transaction (a supported, unprivileged RPC/tx-pool action) after that point triggers the silent corruption path. No special privileges, keys, or majority hashpower are required.

---

### Recommendation

Replace the silent cast on line 156 with the same checked conversion used everywhere else in the file:

```rust
// Before (silent truncation):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (consistent with secondary_block_reward and dao_field_with_current_epoch):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [9](#0-8)

### Proof of Concept

The inconsistency is directly visible by diffing the four u128→u64 conversions in `util/dao/src/lib.rs`:

| Location | Conversion used |
|---|---|
| `calculate_maximum_withdraw` line 156 | `withdraw_counted_capacity as u64` ← **silent truncation** |
| `secondary_block_reward` line 204 | `u64::try_from(reward128).map_err(\|_\| DaoError::Overflow)?` |
| `dao_field_with_current_epoch` line 245 | `u64::try_from(miner_issuance128).map_err(\|_\| DaoError::Overflow)?` |
| `dao_field_with_current_epoch` line 258 | `u64::try_from(ar_increase128).map_err(\|_\| DaoError::Overflow)?` | [10](#0-9) 

A concrete numeric trigger: if a depositor holds `counted_capacity = 3.36 × 10¹⁸` shannons (the entire CKB supply) and `withdrawing_ar / deposit_ar ≥ 5.5`, then `withdraw_counted_capacity ≥ 1.848 × 10¹⁹ > u64::MAX`, and `withdraw_counted_capacity as u64` wraps to a value near zero, producing a `withdraw_capacity` far below the actual entitlement. The `DaoHeaderVerifier` then rejects any block that includes this withdrawal, as the DAO field computed by `dao_field_with_current_epoch` will not match the header's DAO field. [11](#0-10)

    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {

                format!("{err}"),

            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,

    // reject txs which fee lower than min fee rate

            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);

    }
