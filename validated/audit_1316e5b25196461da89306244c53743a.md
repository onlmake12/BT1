Audit Report

## Title
Missing `send_timeout` in `handle_notify_new_block` Allows Unbounded Task Accumulation Under Block Flood - (File: `notify/src/lib.rs`)

## Summary
`handle_notify_new_block` dispatches to each subscriber via `subscriber.send(block).await` with no timeout. Every other subscriber-dispatch handler in the same file uses `send_timeout`. When a subscriber's 128-slot channel is saturated, each additional block spawns a new tokio task that suspends indefinitely, retaining a live `Arc<BlockView>`. Sustained block delivery against a slow subscriber can exhaust node memory and trigger an OOM kill.

## Finding Description
`notify/src/lib.rs` lines 261–273 confirm the inconsistency:

```rust
fn handle_notify_new_block(&self, block: BlockView) {
    for subscriber in self.new_block_subscribers.values() {
        let block = block.clone();
        let subscriber = subscriber.clone();
        self.handle.spawn(async move {
            if let Err(e) = subscriber.send(block).await {   // ← no timeout
                error!("Failed to notify new block, error: {}", e);
            }
        });
    }
``` [1](#0-0) 

All other subscriber-dispatch handlers use `send_timeout`:
- `handle_notify_new_transaction` line 324 [2](#0-1) 
- `handle_notify_proposed_transaction` line 354 [3](#0-2) 
- `handle_notify_reject_transaction` line 384 [4](#0-3) 
- `handle_notify_network_alert` line 417 [5](#0-4) 

`NotifyTimeout` (lines 76–80) has `tx`, `alert`, and `script` fields but no `block` field. [6](#0-5) 

`NotifyConfig` has `notify_tx_timeout`, `notify_alert_timeout`, and `script_timeout` but no `notify_block_timeout`. [7](#0-6) 

The event loop (lines 196–219) is non-blocking with respect to spawned tasks: it calls `handle_notify_new_block`, which spawns tasks and returns immediately, then processes the next message. [8](#0-7) 

`notify_new_block` (lines 483–490) is fire-and-forget — it spawns a task sending to `new_block_notifier` (cap 128) and returns without blocking the caller. [9](#0-8) 

The 128-slot `new_block_notifier` channel provides a fixed buffer but no relief valve. Once the subscriber channel fills, every additional block spawns a new suspended task holding a distinct `Arc<BlockView>` reference, preventing GC of that block's data. Tokio's `mpsc::Sender::send` waits indefinitely until capacity is available or the receiver is dropped; since the subscriber remains active, tasks never self-cancel.

## Impact Explanation
A node with at least one registered `new_block_subscriber` (the RPC WebSocket subscription service registers one whenever any client subscribes to new blocks) processing blocks faster than the subscriber can drain will accumulate suspended tokio tasks, each retaining a full `BlockView`. With CKB's `max_block_bytes` limit, thousands of such tasks can exhaust available RAM, causing the OS OOM killer to terminate the node process. This maps to **High: Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation
Preconditions: (1) at least one active `new_block_subscriber` — satisfied whenever any WebSocket client subscribes to new blocks, a common production configuration; (2) block delivery rate exceeds subscriber drain rate — achievable during IBD when a fast sync peer delivers valid blocks at the maximum rate the sync protocol allows, and the WebSocket client or RPC layer is slow or overloaded. No special privilege is required: any unprivileged sync peer can provide valid blocks at maximum rate. The inconsistency with all other handlers strongly suggests an oversight rather than intentional design. Likelihood is **Low** but non-negligible for nodes with active RPC subscribers during IBD.

## Recommendation
Add a `block` field to `NotifyTimeout` (notify/src/lib.rs lines 76–80) and a corresponding `notify_block_timeout` field to `NotifyConfig` (util/app-config/src/configs/notify.rs), then change the dispatch loop to use `send_timeout`:

```rust
fn handle_notify_new_block(&self, block: BlockView) {
    let block_timeout = self.timeout.block;
    for subscriber in self.new_block_subscribers.values() {
        let block = block.clone();
        let subscriber = subscriber.clone();
        self.handle.spawn(async move {
            if let Err(e) = subscriber.send_timeout(block, block_timeout).await {
                error!("Failed to notify new block, error: {}", e);
            }
        });
    }
    // ...
}
```

## Proof of Concept
1. Start a CKB node with at least one WebSocket client subscribed to `new_block` (registers a `new_block_subscriber` with a 128-slot channel).
2. During IBD, connect a sync peer that delivers valid blocks at the maximum rate the `SendBlock` sync message allows.
3. Ensure the WebSocket subscriber is slow to drain (e.g., simulate a slow client or many clients).
4. After 128 blocks, the subscriber channel is full. Each subsequent block causes `handle_notify_new_block` to spawn a new task suspended on `subscriber.send(block).await`.
5. Monitor tokio task count and RSS memory: both grow proportionally to `(blocks_processed − 128) × avg_block_size`.
6. With sustained delivery, RSS grows until the OOM killer terminates the process.

A unit test can reproduce this by: creating a `NotifyService` with one subscriber, never reading from the subscriber's receiver, sending >128 blocks via `notify_new_block`, and asserting that spawned task count grows unboundedly via `tokio::runtime::Handle::metrics().num_alive_tasks()`.
