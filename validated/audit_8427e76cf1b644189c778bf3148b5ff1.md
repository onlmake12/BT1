Audit Report

## Title
Stale `ancestors_size` Guard in `txs_to_commit` Causes Valid Transactions to Be Excluded from Block Templates — (File: tx-pool/src/component/tx_selector.rs)

## Summary
In `TxSelector::txs_to_commit`, the size/cycles guard computes `next_size = size + tx_entry.ancestors_size`, where `ancestors_size` is the total cumulative size of the transaction plus all its ancestors. However, `size` is a running accumulator that already includes ancestors committed in prior iterations, and the actual addition loop only adds incremental bytes. When `update_modified_entries` fails to move a descendant into `modified_entries` — an inconsistency explicitly acknowledged in the source — that descendant remains in `proposed_pool` with a stale `ancestors_size` that double-counts already-committed ancestors, causing the guard to reject it even though its actual incremental contribution fits within the block limits.

## Finding Description

**Guard (lines 148–162):** [1](#0-0) 

`tx_entry.ancestors_size` is the total cumulative size (tx + all ancestors). `size` is a running total already including ancestors added in previous iterations. The guard therefore over-estimates the incremental cost for any transaction whose ancestors are already in `fetched_txs`.

**Actual addition (lines 207–218):** [2](#0-1) 

Already-fetched ancestors are skipped via `is_new = self.fetched_txs.insert(short_id.clone())`, so only the incremental bytes are actually added. The guard and the addition are structurally inconsistent.

**Mitigation and its acknowledged gap (lines 241–262):** [3](#0-2) 

`update_modified_entries` calls `sub_ancestor_weight` on descendants of newly-added transactions and moves them into `modified_entries` with a corrected `ancestors_size`. However, lines 250–251 contain an explicit acknowledgment:

```
// Note: since https://github.com/nervosnetwork/ckb/pull/3706
// calc_descendants() may not consistent
```

When `calc_descendants()` misses a descendant, that descendant stays in `proposed_pool` with its original stale `ancestors_size`.

**`skip_proposed_entry` does not rescue it (lines 235–239):** [4](#0-3) 

A missed descendant is in none of `fetched_txs`, `modified_entries`, or `failed_txs`, so it is not skipped — it is evaluated from `proposed_pool` with the stale value.

**`TxEntry.ancestors_size` semantics confirmed (entry.rs lines 27, 66, 157–160):** [5](#0-4) [6](#0-5) 

`ancestors_size` is initialized to the tx's own size and accumulates each ancestor's `size` via `add_ancestor_weight` / `sub_ancestor_weight`, confirming it is a cumulative total, not an incremental value.

**Exploit path:**
1. `size_limit = 350`, `cycles_limit = ∞`.
2. Tx A (`size=100`, no ancestors → `ancestors_size=100`), Tx B (`size=100`, ancestor={A} → `ancestors_size=200`, higher fee-rate), Tx C (`size=100`, ancestor={A} → `ancestors_size=200`, lower fee-rate; C is **not** a descendant of B).
3. All three are in `proposed_pool`.
4. **Iteration 1:** B is selected (higher fee-rate). Guard: `0+200=200≤350` → passes. A and B added; `size=200`, `fetched_txs={A,B}`. `update_modified_entries` called; `calc_descendants(A)` returns only `{B}` (misses C per acknowledged inconsistency). C is never moved to `modified_entries`.
5. **Iteration 2:** C is selected from `proposed_pool` with stale `ancestors_size=200`. Guard: `200+200=400>350` → **C is rejected**.
6. **Correct behavior:** C's incremental size is 100 (A already fetched); `200+100=300≤350` → C should be included.

## Impact Explanation

The bug causes valid transactions to be silently excluded from block templates, reducing miner fee revenue and delaying confirmation for affected senders. No funds are at risk, no consensus deviation occurs, and no node crash results. This matches the allowed bounty impact: **Low (501–2000 points): Any other important performance improvements for CKB.**

## Likelihood Explanation

The precondition — two transactions sharing a common ancestor (CPFP/sibling pattern) — is routine on mainnet. The additional condition — `calc_descendants()` missing one of them — is explicitly acknowledged in the source as a known inconsistency introduced by PR #3706. No special privileges or attacker capabilities are required; any transaction sender who submits a CPFP chain under normal mempool conditions can trigger this scenario.

## Recommendation

Replace the guard's use of `tx_entry.ancestors_size` with the actual incremental cost: before the guard, compute the set of ancestors not yet in `fetched_txs`, sum only their sizes and cycles, and use that sum for the limit check. This makes the guard consistent with the actual addition loop at lines 207–218. Alternatively, ensure `update_modified_entries` is always consistent — i.e., every descendant of a newly-added transaction is moved to `modified_entries` with a corrected `ancestors_size` before the next outer-loop iteration — so that no descendant is ever evaluated from `proposed_pool` with a stale value.

## Proof of Concept

**Unit test outline:**
- `size_limit = 350`, `cycles_limit = u64::MAX`
- Tx A: `size=100`, no ancestors → `ancestors_size=100`
- Tx B: `size=100`, ancestor={A} → `ancestors_size=200`, fee-rate > C
- Tx C: `size=100`, ancestor={A} → `ancestors_size=200`, fee-rate < B; C is **not** a descendant of B

**Steps:**
1. Add A, B, C to `proposed_pool`.
2. Patch or mock `pool_map.calc_descendants` so that `calc_descendants(A)` returns only `{B}` (omitting C, as acknowledged possible since PR #3706).
3. Call `txs_to_commit(350, u64::MAX)`.
4. **Iteration 1:** B selected (higher fee-rate). Guard: `0+200=200≤350` → passes. A and B added; `size=200`. `update_modified_entries` called; C is missed.
5. **Iteration 2:** C selected from `proposed_pool` with stale `ancestors_size=200`. Guard: `200+200=400>350` → C rejected.
6. **Assert:** returned `entries` contains only A and B; C is absent despite fitting within the limit (`300≤350`).
7. **Regression assert:** applying the fix (incremental size computation) causes C to be included in the output.

                    self.modified_entries.insert_entry(desc);
