Audit Report

## Title
Unbounded `indices` and `lemmas` Arrays in `verify_transaction_proof` / `verify_transaction_and_witness_proof` Enable CPU DoS — (File: `rpc/src/module/chain.rs`)

## Summary
Both `verify_transaction_proof` and `verify_transaction_and_witness_proof` RPC methods construct `MerkleProof` objects directly from caller-supplied `indices` and `lemmas` arrays with no length validation. An attacker with access to the RPC port can supply arrays up to the 10 MiB body limit, forcing proportional blake2b hashing work per request, sustaining CPU saturation and degrading or halting node operation.

## Finding Description
In `verify_transaction_proof`, `MerkleProof::new` is called directly with caller-controlled arrays:

```rust
// rpc/src/module/chain.rs lines 1948–1961
let merkle_proof = MerkleProof::new(
    tx_proof.proof.indices.into_iter().map(|index| index.value()).collect(),
    tx_proof.proof.lemmas.into_iter().map(|lemma| lemma.into()).collect(),
);
CBMT::retrieve_leaves(block.tx_hashes(), &merkle_proof)
    .and_then(|tx_hashes| merkle_proof.root(&tx_hashes) ...
``` [1](#0-0) 

In `verify_transaction_and_witness_proof`, this is done **twice** — once for `transactions_proof` and once for `witnesses_proof`: [2](#0-1) 

The `MergeByte32::merge()` function invokes `new_blake2b()` and performs a full blake2b hash per node pair during `merkle_proof.root()`: [3](#0-2) 

The only pre-computation check is that the referenced `block_hash` exists in the chain — trivially satisfiable with any known block hash. A `grep` search confirms no `indices.len()` or `lemmas.len()` check exists anywhere in the RPC call path. The `max_request_body_size` of 10 MiB is the only constraint, permitting ~150,000 lemmas per proof struct. [4](#0-3) 

By contrast, the light-client P2P handler correctly enforces a limit before doing any work: [5](#0-4) 

The constant `GET_TRANSACTIONS_PROOF_LIMIT` is 1000: [6](#0-5) 

The `rpc_batch_limit` is opt-in, disabled by default, and limits the number of JSON-RPC calls in a batch — not the size of arrays within a single call's parameters. [7](#0-6) 

## Impact Explanation
A caller with access to the RPC port can sustain high CPU load on the CKB node by continuously sending `verify_transaction_and_witness_proof` requests with maximally-sized `indices`/`lemmas` arrays (~150,000 lemmas × 2 proof structs = ~300,000 blake2b operations per request). Since the RPC server shares the node process, this saturates a CPU core and degrades block processing, P2P message handling, and transaction relay — effectively halting node operation. This matches **High — Vulnerabilities which could easily crash a CKB node** when the RPC port is exposed to untrusted callers, or at minimum **Note — Any local RPC API crash** in the default localhost-only configuration.

## Likelihood Explanation
The default configuration binds the RPC to `127.0.0.1:8114`, restricting access to local processes. However, many node operators expose the RPC port for wallets, dApps, and tooling. No authentication, API key, or rate limit is applied to the Chain RPC module. The attack requires only a valid block hash (any known block hash from the public chain suffices), no tokens or keys, and is trivially scriptable and repeatable. The `Chain` module is enabled by default. [8](#0-7) 

## Recommendation
Add an explicit upper-bound check on `indices.len()` and `lemmas.len()` at the start of both `verify_transaction_proof` and `verify_transaction_and_witness_proof`, before any `MerkleProof::new` or `CBMT::retrieve_leaves` call. A reasonable limit mirrors the light-client constant (`GET_TRANSACTIONS_PROOF_LIMIT = 1000`) or is derived from the maximum transactions per block. Similarly, add an upper-bound check on `tx_hashes.len()` in `get_tx_indices`, which currently only checks `is_empty()`: [9](#0-8) 

## Proof of Concept
```python
import requests

block_hash = "0x7978ec7ce5b507cfb52e149e36b1a23f6062ed150503c85bbf825da3599095ed"
large_lemmas = ["0x" + "00" * 32] * 100_000
large_indices = ["0x0"] * 100_000

payload = {
    "jsonrpc": "2.0", "id": 1,
    "method": "verify_transaction_and_witness_proof",
    "params": [{
        "block_hash": block_hash,
        "transactions_proof": {"indices": large_indices, "lemmas": large_lemmas},
        "witnesses_proof":    {"indices": large_indices, "lemmas": large_lemmas}
    }]
}

while True:
    requests.post("http://127.0.0.1:8114", json=payload)
```

Each iteration forces ~200,000+ blake2b hash operations before returning `"Invalid transaction_and_witness proof"`. Sustained in a loop, this saturates a CPU core and degrades node responsiveness to peers and miners.
