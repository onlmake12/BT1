### Title
`is_well_known_script` Prefix-Only Args Match Allows Arbitrary-Length Args to Bypass `WellKnownScriptsOnly` Output Validator — (`File: rpc/src/module/pool.rs`)

---

### Summary

The `is_well_known_script` helper in `rpc/src/module/pool.rs` matches a transaction output's lock/type script against the well-known script list using a **prefix check on `args`** rather than an exact-length check. Because the well-known scripts (anyone-can-pay, cheque, Simple UDT) are registered with empty args (`"args": "0x"`), any script with the correct `code_hash` and `hash_type` passes validation regardless of what args it carries — including args of arbitrary length or content. This is the CKB analog of the ENS bug: a wrong-format (wrong-length) value is silently accepted instead of being rejected.

---

### Finding Description

`WellKnownScriptsOnlyValidator` is the guard applied when a caller submits a transaction via `send_transaction` with `outputs_validator = "well_known_scripts_only"`. It is also the guard used when `reject_ill_transactions` is enabled on the node.

For lock scripts that are not secp256k1-blake160 (i.e., anyone-can-pay and cheque), validation falls through to `validate_well_known_lock_scripts`, which calls `is_well_known_script`:

```rust
fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script
            .args()
            .as_slice()
            .starts_with(well_known_script.args().as_slice())
}
```

The well-known scripts are registered with `"args": "0x"` (empty args):

```json
{
    "code_hash": "0xd369597ff47f29fbc0d47d2e3775370d1250b85140c670e4718af712983a2354",
    "hash_type": "type",
    "args": "0x"
}
```

Because `well_known_script.args()` is empty, `script.args().as_slice().starts_with(&[])` is **always true** for any args value. Therefore, any script with the matching `code_hash` and `hash_type` passes validation regardless of the length or content of its `args` field.

The same flaw applies to `validate_well_known_type_scripts` for the Simple UDT script, which is also registered with empty args.

By contrast, the secp256k1-blake160 validators correctly enforce exact args length:

```rust
} else if script.args().len() != BLAKE160_LEN {
    Err(DefaultOutputsValidatorError::ArgsLen)
```

The well-known-script path has no equivalent length guard.

---

### Impact Explanation

An RPC caller submitting a transaction with `outputs_validator = "well_known_scripts_only"` can include output cells whose lock script uses the anyone-can-pay or cheque `code_hash` but carries **arbitrary args** — for example, args that do not correspond to any real public key hash, or args that encode a different spending condition entirely. The validator accepts the transaction silently.

The purpose of `WellKnownScriptsOnly` is to protect users from accidentally locking funds under scripts that cannot be spent. A script with the correct code hash but malformed or wrong-length args may be unspendable (e.g., if the on-chain script logic requires exactly 20 bytes for a pubkey hash but receives 22 bytes, it will abort). Funds sent to such an output are permanently lost.

Additionally, when `reject_ill_transactions` is enabled in the node config, the node applies this same validator to all incoming transactions. The bypass means the node's ill-transaction filter provides weaker protection than intended for anyone-can-pay and cheque outputs.

---

### Likelihood Explanation

The entry path is the public `send_transaction` RPC, reachable by any RPC caller (local or remote depending on node config). The caller only needs to:
1. Set `outputs_validator` to `"well_known_scripts_only"`.
2. Construct a transaction output with the anyone-can-pay or cheque `code_hash` and `hash_type = "type"`, but with args of incorrect length (e.g., 22 bytes instead of 20).

This is a low-effort mistake that a developer can make accidentally when constructing transactions programmatically, and the validator provides no warning. The likelihood of accidental triggering is moderate; deliberate exploitation to cause fund loss in a third party requires the attacker to convince the victim to send to the malformed output, which is lower likelihood but still realistic.

---

### Recommendation

Replace the prefix match in `is_well_known_script` with an exact args match, or add an explicit args-length check for each well-known script. For anyone-can-pay and cheque, the expected args length (20 bytes for a blake160 pubkey hash) should be enforced, mirroring the secp256k1-blake160 validators:

```rust
fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script.args().as_slice() == well_known_script.args().as_slice()
        // or: enforce a known minimum/exact args length per script
}
```

If prefix matching is intentional (to allow scripts with optional extra args), the well-known script registry should store the **minimum required args** rather than empty args, and the validator should additionally check that `script.args().len() >= expected_min_len`.

### Proof of Concept

An RPC caller sends the following JSON-RPC request to a mainnet/testnet node with the Pool module enabled:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "send_transaction",
  "params": [
    {
      "version": "0x0",
      "cell_deps": [...],
      "header_deps": [],
      "inputs": [...],
      "outputs": [
        {
          "capacity": "0x2540be400",
          "lock": {
            "code_hash": "0xd369597ff47f29fbc0d47d2e3775370d1250b85140c670e4718af712983a2354",
            "hash_type": "type",
            "args": "0x9fD6E51AaD88f6f4CE6aB8827279CFFFb92266332265"
          }
        }
      ],
      "outputs_data": ["0x"],
      "witnesses": [...]
    },
    "well_known_scripts_only"
  ]
}
```

The `args` field is 22 bytes (wrong length for anyone-can-pay, which expects 20 bytes). The validator accepts this transaction because `is_well_known_script` only checks that `script.args()` starts with the empty bytes of the registered well-known script. The transaction passes the `WellKnownScriptsOnly` guard and enters the pool. If mined, the output cell may be permanently unspendable depending on the on-chain script logic.
