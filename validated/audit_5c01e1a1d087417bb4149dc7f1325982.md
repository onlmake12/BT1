Audit Report

## Title
`check_tx_fee` Enforces Minimum Fee Using Raw Byte Size Instead of Transaction Weight, Enabling Cycle-Heavy Spam at Fraction of Intended Cost — (File: tx-pool/src/util.rs)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee by passing raw serialized byte size (`tx_size`) to `FeeRate::fee()`, while `FeeRate` is defined as shannons per kilo-**weight**, where weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. No subsequent weight-based fee check is performed after `verify_rtx` resolves the actual cycle count. An unprivileged attacker can craft transactions that are small in serialized size but consume the maximum allowed cycles, paying up to ~119× less than the weight-based minimum fee, and have them admitted to the pool and relayed across the network.

## Finding Description

`FeeRate` is explicitly typed as "shannons per kilo-weight": [1](#0-0) 

The canonical transaction weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`: [2](#0-1) 

`TxEntry::fee_rate()` correctly uses this weight for post-admission accounting: [3](#0-2) 

However, `check_tx_fee` — the sole admission gate — passes only raw `tx_size` to `FeeRate::fee()`, and the code comment explicitly acknowledges the mismatch: [4](#0-3) 

In `_process_tx`, the call sequence is:
1. `pre_check` → calls `check_tx_fee(tx_size)` — size-only fee gate fires here
2. `verify_rtx` → actual cycle count is determined
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles
4. `submit_entry` — no weight-based fee re-check occurs [5](#0-4) 

There is no second fee check after cycles are known. The size-only check is the only admission gate.

## Impact Explanation

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

With `tx_size = 100 bytes`, `cycles = 70,000,000`, and `min_fee_rate = 1,000 shannons/KW`:
- Admission check requires: `1000 × 100 / 1000 = 100 shannons`
- Weight-based minimum should require: `1000 × max(100, 70,000,000 × 0.000_170_571_4) / 1000 = 1000 × 11,940 / 1000 = 11,940 shannons`

The attacker pays ~119× less than the weight-based rate demands. Such transactions are admitted to the pool, relayed to peers (propagating the underpriced load), and consume the block's cycle budget when committed without paying proportional fees. The pool size limit is 180MB; at 100 bytes per transaction, up to 1.8 million cycle-heavy transactions could be injected, each consuming the full 70M-cycle block budget at a fraction of the intended cost. [6](#0-5) 

## Likelihood Explanation

The attack requires only the ability to submit a transaction via the standard `send_transaction` RPC or P2P relay — no privileged access, no key material, no majority hashpower. The attacker pre-deploys a cycle-intensive script cell on-chain (a one-time cost), then crafts transactions with a small serialized body (single input/output referencing that cell dep via a compact witness) that execute the script for ~70M cycles. The default `max_tx_verify_cycles = 70,000,000` and `min_fee_rate = 1,000` are live mainnet values. [7](#0-6) 

## Recommendation

After `verify_rtx` returns the `Completed` entry with actual cycles, perform a weight-based fee check before calling `submit_entry`:

```rust
// In _process_tx, after verify_rtx:
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, min_fee.as_u64(), fee.as_u64())), snapshot));
}
```

The size-only pre-check in `check_tx_fee` can be retained as a cheap early rejection for obviously underfunded transactions, but it must be followed by a definitive weight-based check once cycles are known. [3](#0-2) 

## Proof of Concept

1. Deploy a script cell on-chain whose execution loops for ~70,000,000 cycles.
2. Craft a transaction: 1 input (a UTXO you own), 1 output, a witness invoking the loop script, and a cell dep pointing to the deployed script. Serialized size will be ≤ 200 bytes.
3. Set the transaction fee to `1000 × 200 / 1000 = 200 shannons` (satisfies the size-based check in `check_tx_fee`).
4. Submit via `send_transaction` RPC. Observe the transaction is accepted into the pool.
5. The correct weight-based minimum fee would be `1000 × 11,940 / 1000 = 11,940 shannons` — the attacker paid ~1.7% of this.
6. Repeat with many such transactions. Each consumes ~70M cycles of block budget while paying only the size-based fee. The pool fills with cycle-heavy, underpriced transactions that are relayed to all peers. [4](#0-3)
