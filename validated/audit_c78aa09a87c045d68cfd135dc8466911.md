### Title
Missing Bounds Check on `tx_age` in Fee Estimator Causes Out-of-Bounds Panic on Tx Removal — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

`TxConfirmStat::remove_unconfirmed_tx` does not validate that `tx_age - 1` is within the bounds of `confirm_blocks_to_failed_txs` before indexing into it. When a tracked transaction ages beyond `max_confirm_blocks` blocks and is then removed from the pool with `count_failure = true`, the function performs an out-of-bounds index access, causing a Rust panic and crashing the node. Additionally, the `old_unconfirmed_txs` counter is decremented with a bare `-=` operator without a zero-check, mirroring the "same item removed multiple times decrements the counter" pattern from the report.

---

### Finding Description

In `confirmation_fraction.rs`, `remove_unconfirmed_tx` computes `tx_age` as the number of blocks a transaction has been in the pool:

```rust
let tx_age = tip_height.saturating_sub(entry_height) as usize;
```

It then branches on whether `tx_age >= self.block_unconfirmed_txs.len()` (i.e., `>= max_confirm_blocks`). In the "old" branch it decrements `old_unconfirmed_txs` with a bare subtraction:

```rust
self.bucket_stats[bucket_index].old_unconfirmed_txs -= 1;   // line 209
```

Immediately after the branch, with no additional guard, it indexes into `confirm_blocks_to_failed_txs`:

```rust
if count_failure {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;  // line 215
}
```

`confirm_blocks_to_failed_txs` is allocated with length `max_confirm_blocks` (same as `block_unconfirmed_txs`). When `tx_age > max_confirm_blocks`, `tx_age - 1 >= max_confirm_blocks`, which is out of bounds. Rust's slice indexing always panics on out-of-bounds access in both debug and release builds, crashing the process.

The second issue is that `old_unconfirmed_txs -= 1` (line 209) is an unchecked subtraction on a `usize`. If the counter is already `0` (e.g., due to a `move_track_window` race or a double-removal path), this panics in debug mode and wraps to `usize::MAX` in release mode, corrupting the estimator's statistics permanently. [1](#0-0) 

The two arrays are allocated with the same `max_confirm_blocks` length at construction time: [2](#0-1) 

---

### Impact Explanation

A Rust `index out of bounds` panic is an unrecoverable process abort. Any CKB node that has the fee estimator enabled and that receives a transaction which:

1. Enters the tx pool and is tracked by `add_unconfirmed_tx`, and
2. Remains unconfirmed for more than `max_confirm_blocks` blocks, and
3. Is subsequently evicted/expired with `count_failure = true`

…will crash. Because the tx pool expiry path is triggered automatically on every new block, the crash is deterministic once the age threshold is crossed. The node cannot recover without a restart, and the condition will recur on the next long-lived unconfirmed transaction.

---

### Likelihood Explanation

An unprivileged RPC caller can submit a low-fee transaction via `send_transaction`. Such a transaction will naturally age in the pool if the fee is below the current market rate. After `max_confirm_blocks` blocks (a fixed constant), the expiry sweep calls `remove_unconfirmed_tx` with `count_failure = true`, triggering the panic. No special privileges, keys, or majority hashpower are required. The attack is repeatable after each node restart.

---

### Recommendation

1. Add an explicit bounds check before the `confirm_blocks_to_failed_txs` access:
   ```rust
   if count_failure && tx_age - 1 < self.confirm_blocks_to_failed_txs.len() {
       self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
   }
   ```
2. Replace the bare `old_unconfirmed_txs -= 1` with a saturating subtraction:
   ```rust
   self.bucket_stats[bucket_index].old_unconfirmed_txs =
       self.bucket_stats[bucket_index].old_unconfirmed_txs.saturating_sub(1);
   ```
3. Validate `bucket_index < self.bucket_stats.len()` at the entry of `remove_unconfirmed_tx` and return early if out of range, consistent with how `add_unconfirmed_tx` returns `None` on an unknown fee rate.

---

### Proof of Concept

1. Start a CKB node with the fee estimator enabled.
2. Submit a transaction with a fee rate below the minimum market rate via `send_transaction` RPC.
3. Mine (or wait for) more than `max_confirm_blocks` blocks without the transaction being committed.
4. The tx pool expiry sweep fires, calls `remove_unconfirmed_tx` with `tx_age > max_confirm_blocks` and `count_failure = true`.
5. `confirm_blocks_to_failed_txs[tx_age - 1]` panics with `index out of bounds: the len is max_confirm_blocks but the index is tx_age - 1`, crashing the node. [3](#0-2)

    fn remove_unconfirmed_tx(
        &mut self,
        entry_height: u64,
        tip_height: u64,
        bucket_index: usize,
        count_failure: bool,
    ) {
        let tx_age = tip_height.saturating_sub(entry_height) as usize;
        if tx_age < 1 {
            return;
        }
        if tx_age >= self.block_unconfirmed_txs.len() {
            self.bucket_stats[bucket_index].old_unconfirmed_txs -= 1;
        } else {
            let block_index = (entry_height % self.block_unconfirmed_txs.len() as u64) as usize;
            self.block_unconfirmed_txs[block_index][bucket_index] -= 1;
        }
        if count_failure {
            self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
        }
    }
