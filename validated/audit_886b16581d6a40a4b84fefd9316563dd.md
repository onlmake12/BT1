### Title
`ScriptHashTypeVerifier` Only Validates Lock Script Hash Type, Ignoring Type Script Hash Type in Transaction Outputs — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` is documented to verify that the `ScriptHashType` of **all** transaction outputs is within the range permitted by current consensus rules. However, the implementation only checks the **lock script** hash type of each output and completely ignores the **type script** hash type. This mirrors the external report's root cause: code that handles a "basic" variant (lock script) but silently skips an "advanced" variant (type script), causing the feature to be incomplete.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, the `ScriptHashTypeVerifier` struct is described as:

> "Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules."

Its `verify()` implementation iterates over outputs and checks only `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

The check `output.type_().to_opt()` is never consulted. A type script with a hash type value such as `6` (`Data3`, not in `ENABLED_SCRIPT_HASH_TYPE`) passes this verifier without error. [2](#0-1) 

The enabled set is `{0, 1, 2, 4}` (Data, Type, Data1, Data2). Values like `6` (Data3), `8` (Data4), etc. are structurally valid `ScriptHashType` values (even numbers pass `verify_value`) but are not yet activated. [3](#0-2) 

When such an output is later consumed as an input in a future transaction, `select_version()` in `script/src/types.rs` will reject the type script with `InvalidScriptHashType`: [4](#0-3) 

Because type scripts on **outputs** are only executed when those outputs are **consumed** in a future transaction, the creating transaction passes all verification and is committed to the chain. The output cell becomes permanently unspendable.

---

### Impact Explanation

A transaction sender can create outputs whose type scripts carry a non-permitted `ScriptHashType` (e.g., `Data3 = 6`). The `NonContextualTransactionVerifier` — which runs `ScriptHashTypeVerifier` — accepts the transaction. The output is committed to the chain. Any future transaction attempting to spend that output will fail at script execution time with `InvalidScriptHashType`, making the capacity locked in that cell permanently irrecoverable. This constitutes a **permanent capacity loss** for the cell owner. [5](#0-4) 

---

### Likelihood Explanation

The entry path is fully unprivileged: any user submitting a transaction via the `send_transaction` RPC can trigger this. A user who accidentally (or through a buggy SDK/tool) constructs an output with an unsupported type script hash type will have their CKB permanently locked with no recourse. The likelihood is moderate — the scenario is realistic for developers experimenting with future VM versions or for users of tooling that does not validate hash types before submission.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type of each output, mirroring the lock script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Check lock script hash type
        let lock_hash_type: u8 = output.lock().hash_type().into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&lock_hash_type) { ... }

        // Also check type script hash type
        if let Some(type_script) = output.type_().to_opt() {
            let type_hash_type: u8 = type_script.hash_type().into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&type_hash_type) { ... }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Construct a transaction with one output whose type script has `hash_type = 6` (Data3, not in `ENABLED_SCRIPT_HASH_TYPE`) and a valid lock script.
2. Submit via `send_transaction` RPC.
3. Observe: `ScriptHashTypeVerifier` passes (only checks lock script). Transaction is committed.
4. Construct a second transaction spending that output.
5. Observe: Script execution fails with `InvalidScriptHashType` for the type script. The output is permanently unspendable. [6](#0-5) [7](#0-6)
