Audit Report

## Title
Silent Pagination Restart from Zero `last_cursor` on Empty `get_transactions` Result — (File: `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
`AsyncRichIndexerHandle::get_transactions` initializes its cursor tracking variables (`last_id`, `count` in ungrouped mode; `last_cursor` in grouped mode) to integer zero. When the query returns no rows, these defaults are serialized unconditionally into the returned `last_cursor`, producing an all-zero byte sequence. A caller that feeds this cursor back as `after` receives results starting from the very beginning of the transaction table, identical to passing `after = null`. This is inconsistent with `get_cells`, which correctly returns an empty cursor on empty results.

## Finding Description
**Ungrouped mode:** `last_id: i64 = 0` and `count: i32 = 0` are declared before the iterator (L66–67). On an empty result the iterator body never executes, so both remain zero. Lines 91–93 unconditionally serialize them: `last_cursor = [0u8;8] ++ [0u8;4]` = 12 zero bytes. When this is passed back as `after`, lines 44–53 decode it to `Some((0i64, 0i32))`. Line 171 then emits `WHERE tx_id >= 0 OFFSET 0`, which is always satisfied since auto-increment `tx_id` starts at 1, restarting the scan from the first row.

**Grouped mode:** `last_cursor: i64 = 0` (L111) is serialized as 8 zero bytes (L138). The `get_tx_with_cells` path at L320–326 uses strict `>`, but `WHERE tx_id > 0` is equally always true, producing the same restart.

**Contrast with `get_cells`:** `last_cursor` is initialized to `Vec::new()` (L228 of `get_cells.rs`) and only written inside the row-mapping closure. An empty result returns an empty byte slice; passing it back as `after` causes `decode_i64` to fail with a parse error, making misuse visible.

## Impact Explanation
The concrete impact is incorrect RPC pagination behavior: callers receive a misleading non-empty cursor on empty results, causing silent restart from offset 0 on the next call. This can lead to repeated full-table scans on the indexer's SQL database, constituting a performance degradation issue. This fits **Low (501–2000 points): Any other important performance improvements for CKB**, as each spurious full-table scan triggered by any number of concurrent callers imposes growing I/O and CPU cost on the indexer node as the transaction table grows.

## Likelihood Explanation
The `get_transactions` RPC is publicly documented and reachable by any unprivileged caller. The API does not document that `last_cursor` is meaningless on an empty result. Callers using narrow `block_range` filters or rare scripts routinely encounter empty pages during normal operation. The behavior is inconsistent with `get_cells`, so developers who test with `get_cells` will not discover the issue until switching to `get_transactions`. Triggering the bug requires only two sequential RPC calls with no special privileges.

## Recommendation
Mirror the `get_cells` pattern in both modes:
1. Initialize `last_id`/`last_cursor` to `None` or `Vec::new()`, not `0`.
2. Populate the cursor only inside the row-mapping closure, so an empty result produces an empty byte slice.
3. In ungrouped mode, switch the SQL predicate from `>=` to `>` (matching grouped mode and `get_cells`) to eliminate the offset-based deduplication that makes the zero default dangerous.
