### Title
Unauthenticated `ConnectionSync` Relay Forwarding Enables Unsolicited NAT Traversal on Any Connected Peer — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

### Summary
`ConnectionSyncProcess::execute` forwards a `ConnectionSync` message to the peer named in `content.to` whenever `route` is empty and the local node is not the `to` peer, with no verification that the sender is part of a legitimate hole-punching flow. Any peer connected to a relay can deliver a `ConnectionSync` to any other peer connected to that relay.

### Finding Description
In `connection_sync.rs` lines 98–104, when `route=[]`, the relay unconditionally calls `forward_sync(&content.to)`: [1](#0-0) 

`forward_sync` (lines 178–184) resolves the target session from the peer registry and delivers the message: [2](#0-1) 

The only guards are a route-length check and a `forward_rate_limiter` keyed on `(from, to, msg_item_id)`: [3](#0-2) 

Because `from` is fully attacker-controlled, the attacker rotates `from` values to bypass the `forward_rate_limiter`. The per-session limiter (30/sec) in `mod.rs` limits throughput but does not prevent the attack: [4](#0-3) 

On the victim V, the received `ConnectionSync` triggers NAT traversal if `pending_delivered` contains the `from` peer ID: [5](#0-4) 

On success, V calls `raw_session` to establish a new inbound P2P session: [6](#0-5) 

The `pending_delivered` map is populated in `connection_request.rs` when V receives a `ConnectionRequest` with `to=V`: [7](#0-6) 

The attacker can pre-populate this map by sending a `ConnectionRequest { from=X, to=V, listen_addrs=[attacker_addr] }` through the same relay, since `from` is not verified against the actual sender identity.

### Impact Explanation
An attacker connected to relay R can:
1. Pre-populate V's `pending_delivered[X]` with attacker-controlled TCP addresses by sending a spoofed `ConnectionRequest`.
2. Trigger V to make outbound TCP connections to those addresses via a spoofed `ConnectionSync`.
3. On success, cause V to call `raw_session`, establishing a new inbound P2P session from attacker-controlled infrastructure.

This is an SSRF-class primitive: the attacker causes V to initiate TCP connections to arbitrary addresses and potentially accept a new P2P session from them.

### Likelihood Explanation
The attack requires only that the attacker be a connected peer to a relay R that is also connected to V — a normal network topology condition. No privileged access, leaked keys, or majority hashpower is needed. The two-step sequence (ConnectionRequest then ConnectionSync) is straightforward to implement.

### Recommendation
- Verify that the sender of a `ConnectionSync` is the peer identified in `content.to` (i.e., only the `to` peer should be able to initiate the sync back).
- Alternatively, maintain a relay-side state table of in-flight hole-punching flows and only forward `ConnectionSync` messages that correspond to a known flow.
- Do not allow `from` to be an arbitrary, unverified peer ID in either `ConnectionRequest` or `ConnectionSync`.

### Proof of Concept
```
1. Attacker A connects to relay R (R is also connected to victim V).
2. A sends: ConnectionRequest { from=X, to=V_peer_id, listen_addrs=[attacker_ip:port] }
   → R forwards to V (V is directly connected to R).
   → V stores pending_delivered[X] = ([attacker_ip:port/p2p/X], now).
3. A sends: ConnectionSync { route=[], from=X, to=V_peer_id }
   → R: route.last() == None, self_peer_id != V_peer_id → calls forward_sync(V_peer_id).
   → V receives ConnectionSync, finds pending_delivered[X], spawns try_nat_traversal to attacker_ip:port.
   → If attacker_ip:port accepts the TCP connection, V calls raw_session() establishing a new inbound session.
Assert: attacker_ip:port receives a TCP connection from V.
```
