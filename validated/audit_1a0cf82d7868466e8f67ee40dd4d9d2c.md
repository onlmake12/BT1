Audit Report

## Title
Stale Descendant-Weight Accounting in `remove_entry_and_descendants` Leaves Ancestor Entries with Inflated Eviction Score — (`tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::remove_entry_and_descendants` calls `remove_entry_links` on every entry in the removal set before iterating to call `remove_entry`. Because `update_ancestors_index_key` resolves ancestors through those same link records, it finds an empty ancestor set and never calls `sub_descendant_weight` on ancestor transactions that remain in the pool. Those ancestors permanently retain inflated `descendants_fee / descendants_size / descendants_cycles / descendants_count`, corrupting the `EvictKey` used to rank transactions for eviction.

## Finding Description
`remove_entry_and_descendants` collects the target and all its descendants, then pre-calls `remove_entry_links` on every one of them before iterating to call `remove_entry` (L252–265). Inside `remove_entry`, `update_ancestors_index_key` is called at L242. `update_ancestors_index_key` resolves the ancestor set via `self.links.calc_ancestors(&child.proposal_short_id())` (L433–434), which calls `calc_relative_ids` in `links.rs` (L37–50), which looks up `self.inner.get(short_id)`. Because `remove_entry_links` already called `self.links.remove(id)` for every entry in the removal set during the pre-pass, `self.inner.get(&A_id)` returns `None`, `direct` is empty, and `calc_relation_ids` returns an empty set. `sub_descendant_weight` is therefore never called on any ancestor that remains in the pool. The single-entry `remove_entry` path works correctly because links are still intact at call time. The existing test `test_remove_entry_and_descendants` (L171–230 in `score_key.rs`) only asserts that removed entries are absent; it does not assert that the remaining ancestor's `descendants_count` or `descendants_fee` are decremented, so the regression goes undetected.

## Impact Explanation
`EvictKey` is computed directly from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` (entry.rs L234–247). An ancestor whose removed descendants had a higher fee rate than itself retains an inflated `descendants_feerate`, making its `fee_rate` field in `EvictKey` appear higher than it truly is. `next_evict_entry` selects the lowest-`EvictKey` entry to drop; the inflated ancestor is ranked as more valuable than it deserves, so other legitimate transactions are evicted in its place. An unprivileged attacker can cheaply and repeatedly cause legitimate users' transactions to be evicted from the mempool, preventing confirmation. This constitutes **damage to CKB economy** — Critical (15001–25000 points).

## Likelihood Explanation
No special privileges are required. An attacker submits a low-fee root transaction X, then high-fee descendants A → B spending X's output, then a conflicting transaction A′ spending the same input as A. `resolve_conflict` calls `remove_entry_and_descendants(A)`, leaving X with permanently inflated descendant stats. The only cost is the fee for A′. The attack is repeatable: each round inflates a different ancestor. No majority hash power, no social engineering, and no victim mistakes are required.

## Recommendation
Collect and apply ancestor updates **before** tearing down links. For each entry being removed, resolve its ancestors while links are still intact and call `sub_descendant_weight` only on ancestors that are **not** themselves in the removal set. Only after all ancestor updates are applied should `remove_entry_links` be called. This mirrors the logic already used by the single-entry `remove_entry` path and ensures `EvictKey` values remain accurate for entries that stay in the pool.

## Proof of Concept
**Setup**: pool contains chain `X → A → B`.

After insertion, X's tracked state:
- `descendants_fee = 600`, `descendants_size = 500`, `descendants_cycles = 500`, `descendants_count = 3`

**Trigger**: submit tx A′ spending the same input as A. `resolve_conflict` calls `remove_entry_and_descendants(A)`.

**Expected state of X**:
- `descendants_fee = 100`, `descendants_size = 100`, `descendants_cycles = 100`, `descendants_count = 1`

**Actual state of X (bug)**:
- `descendants_fee = 600`, `descendants_size = 500`, `descendants_cycles = 500`, `descendants_count = 3`

A regression test can be added to `tx-pool/src/component/tests/score_key.rs` following the pattern of `test_remove_entry_and_descendants`: after calling `map.remove_entry_and_descendants(&tx2_id)`, assert `map.get(&tx1_id).unwrap().descendants_count == 1` and `map.get(&tx1_id).unwrap().descendants_fee == Capacity::shannons(100)`. Under the current code, both assertions fail.
