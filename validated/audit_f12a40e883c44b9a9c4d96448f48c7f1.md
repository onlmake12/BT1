Audit Report

## Title
Truncated 32-bit Genesis Hash in `identify_name()` Allows Cross-Chain Peers to Bypass Network Identity Filter — (File: `spec/src/consensus.rs`)

## Summary
`Consensus::identify_name()` constructs the P2P identity string using only the first 8 hex characters (4 bytes / 32 bits) of the 32-byte genesis hash. An attacker who creates a custom chain spec with `name = "ckb"` and iterates the genesis nonce until the first 4 bytes of the genesis hash match the mainnet prefix can pass `Identify::verify()` and be admitted as a legitimate peer, enabling peer-slot exhaustion and resource consumption against any reachable CKB node.

## Finding Description
`Consensus::identify_name()` at `spec/src/consensus.rs` lines 964–968 formats the identity string as `"/{id}/{genesis_hash[..8]}"`, using only 8 of 64 hex characters of the genesis hash. `Identify::verify()` at `network/src/protocols/identify/mod.rs` lines 541–551 performs a full string equality check against this truncated identifier. Because `Consensus.id` is set directly from `ChainSpec.name` (via `spec/src/lib.rs` line 573), and the mainnet name `"ckb"` is freely copyable (`resource/specs/mainnet.toml` line 1), an attacker can:

1. Create a custom chain spec with `name = "ckb"`.
2. Iterate the genesis `nonce` (a `u128` field) and recompute the genesis hash until `genesis_hash_hex[..8]` matches the mainnet prefix. This is a 32-bit preimage search requiring ~2^32 Blake2b computations (~43 seconds at 100M hashes/s on one CPU core).
3. Start a node on this custom chain and connect to a mainnet peer.
4. `Identify::verify()` returns `Some(...)` because `"/ckb/<4-byte-prefix>" == "/ckb/<4-byte-prefix>"`.
5. The attacker node is admitted and can occupy peer slots and inject sync/relay messages (`GetHeaders`, `SendBlock`) that consume CPU and memory before being rejected at the block-validation layer.

The identify protocol is the only network-layer chain-identity gate and cannot be disabled (`network/src/network.rs` line 867). No other guard checks the full genesis hash at the connection layer.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker running multiple nodes with colliding genesis hash prefixes can exhaust inbound peer slots on targeted nodes (preventing legitimate peers from connecting) and force processing of sync/relay messages that pass the identity gate and enter the processing pipeline before being rejected, consuming CPU and memory per message. At scale, this constitutes network-wide congestion with negligible attacker cost.

## Likelihood Explanation
No privileged access is required. The chain name `"ckb"` is freely settable. The 32-bit prefix collision requires ~2^32 Blake2b computations, achievable in under a minute on commodity hardware. The genesis `nonce` field (`u128`) provides ample search space. The attack is repeatable: the 5-minute ban applies per session, not per genesis hash, and the attacker can reconnect with new sessions or rotate IPs. Pre-computed colliding specs can be reused indefinitely.

## Recommendation
Replace the 8-character genesis hash slice with the full 64-character hex string in `identify_name()` in `spec/src/consensus.rs`:

```rust
pub fn identify_name(&self) -> String {
    let genesis_hash = format!("{:x}", Into::<H256>::into(&self.genesis_hash));
    format!("/{}/{}", self.id, &genesis_hash)   // full 256-bit hash
}
```

This is a non-breaking change for honest nodes (all derive the same full hash from the same genesis block) and raises the collision cost from ~2^32 to ~2^256.

## Proof of Concept
1. Clone the CKB repository. Create a custom chain spec with `name = "ckb"`.
2. Write a script that iterates `genesis.nonce` and calls `ChainSpec::build_genesis()` + `consensus.genesis_hash()` until `&genesis_hash_hex[..8]` matches the mainnet prefix (obtainable from any public mainnet node).
3. Start a CKB node on this custom chain.
4. Connect to a mainnet peer. `Identify::verify()` passes because the identity strings are equal.
5. Send `GetHeaders` or `SendBlock` messages; observe they enter `HeadersProcess`/`BlockProcess` and consume CPU/memory before being rejected at the ancestry check, confirming resource consumption past the identity gate.
