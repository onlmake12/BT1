### Title
No Authentication on Privileged RPC Methods Allows Any Caller to Disable P2P Networking and Clear the Transaction Pool — (File: `rpc/src/server.rs`)

---

### Summary

The CKB RPC server registers no authentication middleware. Any caller who can reach the RPC port — including any local process on the same machine or any remote host if the operator has bound the server to a non-loopback address — can invoke privileged state-mutating methods such as `set_network_active(false)`, `clear_tx_pool()`, `set_ban()`, and `clear_banned_addresses()` without supplying any credential. The miner client ships HTTP Basic Auth support that is never validated server-side, creating a false impression that the server enforces caller identity.

---

### Finding Description

**Root cause — no auth middleware in the server**

`rpc/src/server.rs` builds the axum router and attaches only `CorsLayer::permissive()` and `TimeoutLayer`. No authentication layer is present:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())          // ← permissive CORS, not auth
    .layer(TimeoutLayer::with_status_code(…))
    .layer(Extension(stream_config));
```

Every incoming POST is dispatched directly to `handle_jsonrpc`, which calls `io.handle_call(call, T::default())`. The metadata type is `Option<Session>` and is always `None` — no token, no IP check, no credential.

**`service_builder.rs` — no per-method guard**

`rpc/src/service_builder.rs` registers all modules (Net, Pool, Miner, etc.) via `set_rpc_module_methods!`. The macro only checks whether a module is *enabled* in config; it applies no caller-identity check to any individual method.

**Privileged methods with no caller check**

`rpc/src/module/net.rs` — `set_network_active`:
```rust
fn set_network_active(&self, state: bool) -> Result<()> {
    self.network_controller.set_active(state);
    Ok(())
}
```
No guard. Any caller can pass `false` to halt all P2P activity.

`rpc/src/module/pool.rs` — `clear_tx_pool`:
```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
```
No guard. Any caller can evict every pending transaction.

`rpc/src/module/net.rs` — `clear_banned_addresses` / `set_ban`:
```rust
fn clear_banned_addresses(&self) -> Result<()> {
    self.network_controller.clear_banned_addrs();
    Ok(())
}
```
No guard. Any caller can erase the entire ban list, re-admitting previously banned malicious peers.

**Miner client ships auth code the server never validates**

`miner/src/client.rs` contains `parse_authorization()` which reads HTTP Basic Auth credentials from the miner URL and attaches them as an `Authorization` header. The server never inspects this header. The auth path is dead on the server side, giving operators a false sense of security.

---

### Impact Explanation

| Method | Impact |
|---|---|
| `set_network_active(false)` | Node stops syncing, relaying blocks and transactions, and participating in consensus. Complete P2P isolation until the operator manually re-enables networking. |
| `clear_tx_pool()` | All pending user transactions are dropped. Users must resubmit, potentially at higher fee rates. Miners lose queued fee revenue. |
| `clear_tx_verify_queue()` | Transactions awaiting script verification are discarded silently. |
| `set_ban` / `clear_banned_addresses` | Attacker can ban legitimate peers or unban previously-blocked malicious ones, degrading or manipulating the node's peer set. |
| `remove_transaction(hash)` | Targeted removal of a specific pending transaction, enabling fee-sniping or griefing. |

The most severe single call is `set_network_active(false)`: it causes an immediate, complete denial-of-service on the node's network participation with a single unauthenticated HTTP POST.

---

### Likelihood Explanation

**Default configuration (localhost):** The default `listen_address = "127.0.0.1:8114"` restricts access to the local machine. Any process running under any user account on that machine — including malware, a compromised co-located service, or another user on a shared server — can reach the port and invoke these methods. This is the minimum exposure.

**Non-default configuration (0.0.0.0):** The config file explicitly documents that operators *can* bind to `0.0.0.0`. The README warns against it but provides no enforcement. When bound to a non-loopback address, any network-reachable host becomes an unauthenticated caller. Exchange nodes, mining pool nodes, and infrastructure nodes are common targets for this misconfiguration.

**No credential required:** The attacker needs only TCP connectivity to the RPC port. No key, no token, no prior knowledge of the node state.

---

### Recommendation

Add an authentication middleware layer to the axum router in `rpc/src/server.rs` that validates a configurable bearer token or HTTP Basic Auth credential before dispatching any request. The miner client already implements the client side of HTTP Basic Auth (`parse_authorization` in `miner/src/client.rs`); the server must enforce it. A new `rpc_auth_token` field in `RpcConfig` (`util/app-config/src/configs/rpc.rs`) should gate all state-mutating methods. Read-only methods (e.g., `get_block`, `get_peers`) may optionally remain unauthenticated.

---

### Proof of Concept

With a default CKB node running (`listen_address = "127.0.0.1:8114"`, Net and Pool modules enabled):

**Step 1 — Disable all P2P networking (no credentials required):**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H "Content-Type: application/json" \
  -d '{"id":1,"jsonrpc":"2.0","method":"set_network_active","params":[false]}'
# → {"jsonrpc":"2.0","result":null,"id":1}
```
The node immediately stops all P2P activity. `local_node_info` will show `"active": false`.

**Step 2 — Clear the entire transaction pool (no credentials required):**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H "Content-Type: application/json" \
  -d '{"id":2,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}'
# → {"jsonrpc":"2.0","result":null,"id":2}
```
All pending transactions are evicted. `tx_pool_info` will show `"pending": "0x0"`.

Both calls succeed with zero authentication. The server code path is:

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}
