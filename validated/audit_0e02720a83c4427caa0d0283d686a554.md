Audit Report

## Title
Unbounded Vec Allocation via Uncapped `max_bucket_index_by_fee_rate` in `do_estimate` Enables Node Crash - (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary

`WeightUnitsFlow::do_estimate` derives `max_bucket_index` directly from the highest fee rate in the live mempool snapshot and allocates two `Vec<u64>` of that size with no upper-bound guard. Because `check_tx_fee` enforces only a minimum fee rate and `max_bucket_index_by_fee_rate` has no ceiling, an attacker who places a single high-fee transaction in the pool and then calls the public `estimate_fee_rate` RPC can force the node to attempt a multi-gigabyte heap allocation, crashing the process via OOM abort.

## Finding Description

`do_estimate` reads `max_fee_rate` from the first element of `sorted_current_txs` (the live mempool, sorted descending by fee rate) and passes it directly to `max_bucket_index_by_fee_rate`:

```rust
let max_fee_rate = sorted_current_txs.first().map(|tx| tx.fee_rate) ...
let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate);
``` [1](#0-0) 

`max_bucket_index_by_fee_rate` has no ceiling; for any fee rate above 2,000,000 shannons/KW it returns `(x + 11_500_000) / 100_000`, growing linearly without bound: [2](#0-1) 

Two `Vec<u64>` are then allocated with length `max_bucket_index + 1`, with no guard: [3](#0-2) [4](#0-3) 

`FeeRate::calculate` uses `saturating_mul`, so the fee rate is bounded by `u64::MAX / weight`. For a 100-byte transaction with a 1,000,000 CKB fee (10¹⁴ shannons):
```
fee_rate = 10¹⁴ × 1000 / 100 = 10¹⁵ shannons/KW
max_bucket_index ≈ 10¹⁵ / 100_000 = 10¹⁰
allocation per vec = 10¹⁰ × 8 bytes = 80 GB  (×2 = 160 GB)
``` [5](#0-4) 

Note: fee rates at or near `u64::MAX` wrap in the addition `x + 11_500_000` (release-mode wrapping), producing a small index — the dangerous range is fee rates large enough to produce a huge index but small enough not to overflow the addition (i.e., fee rates up to `u64::MAX - 11_500_000`). A 1,000,000 CKB fee on a 100-byte transaction falls squarely in this range.

`check_tx_fee` enforces only a minimum fee rate; there is no maximum: [6](#0-5) 

`TxPoolConfig` has no `max_fee_rate` field: [7](#0-6) 

The `estimate_fee_rate` RPC is unauthenticated: [8](#0-7) 

Call chain: `estimate_fee_rate` RPC → `ExperimentRpcImpl::estimate_fee_rate` → `TxPoolService::estimate_fee_rate` → `FeeEstimator::estimate_fee_rate` → `WeightUnitsFlow::do_estimate` → `vec![0u64; max_bucket_index + 1]` → OOM abort. [9](#0-8) 

## Impact Explanation

When the allocation fails, Rust's global allocator calls `handle_alloc_error`, which aborts the process. On Linux the OOM killer terminates the node before the allocation completes. Either path crashes the node process. The crash is repeatable: the attacker's UTXO is never spent (the transaction is never mined because the node crashes before mining), so the attacker resubmits the same transaction after each restart and calls the RPC again at zero marginal cost.

This matches the allowed impact: **High — Vulnerability which could easily crash a CKB node.**

## Likelihood Explanation

- The `estimate_fee_rate` RPC requires no authentication. While the default config binds to `127.0.0.1:8114`, many infrastructure operators (exchanges, explorers, public RPC providers) expose this port externally, and any process on the same host can reach it regardless.
- The tx-pool admission gate has no upper fee-rate bound, so any valid UTXO with sufficient capacity can produce an arbitrarily high fee rate.
- The attacker needs only one UTXO with ~1,000,000 CKB (or less for smaller but still crash-inducing allocations). The UTXO is never consumed because the transaction is never mined, making the attack indefinitely repeatable.
- The `WeightUnitsFlow` estimator is a supported, documented algorithm variant enabled by default.

## Recommendation

1. **Cap `max_bucket_index`** before any allocation in `do_estimate`:
   ```rust
   const MAX_BUCKET_INDEX: usize = 200; // matches the algorithm's designed range
   let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
       .min(MAX_BUCKET_INDEX);
   ```

2. **Add overflow safety** in `max_bucket_index_by_fee_rate` using `checked_add`, returning a capped value on overflow:
   ```rust
   x => x.checked_add(t * 11_500)
         .map(|v| v / (100 * t))
         .unwrap_or(MAX_BUCKET_INDEX as u64),
   ```

3. **Optionally clamp fee rate at pool admission** to a protocol-defined maximum to prevent other future misuse of unbounded fee rates.

## Proof of Concept

```bash
# 1. Attacker submits one transaction:
#    input capacity  = 1_000_000 CKB = 10^14 shannons (attacker's UTXO)
#    output capacity = 61 shannons (minimum cell)
#    fee             ≈ 10^14 shannons
#    serialized size ≈ 100 bytes
#    fee_rate        = 10^14 * 1000 / 100 = 10^15 shannons/KW
#    max_bucket_index ≈ 10^10
#    vec![0u64; 10^10 + 1] × 2 ≈ 160 GB → OOM / process killed

# 2. Attacker calls estimate_fee_rate:
curl -X POST http://localhost:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"estimate_fee_rate","params":[],"id":1}'
# Node crashes.

# 3. Node restarts; attacker's UTXO is still unspent (tx was never mined).
# 4. Repeat from step 1 indefinitely.
```

Unit test to confirm the bucket index growth:
```rust
#[test]
fn test_large_fee_rate_bucket_index() {
    // 10^15 shannons/KW → bucket index ≈ 10^10
    let fee_rate = FeeRate::from_u64(1_000_000_000_000_000u64);
    let idx = Algorithm::max_bucket_index_by_fee_rate(fee_rate);
    assert!(idx > 1_000_000_000, "idx={idx}"); // 10^9 elements × 8 bytes = 8 GB
}

    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }

    pub(crate) async fn estimate_fee_rate(
        &self,
        estimate_mode: EstimateMode,
        enable_fallback: bool,
    ) -> Result<FeeRate, AnyError> {
        let all_entry_info = self.tx_pool.read().await.get_all_entry_info();
        match self
            .fee_estimator
            .estimate_fee_rate(estimate_mode, all_entry_info)
        {
            Ok(fee_rate) => Ok(fee_rate),
            Err(err) => {
                if enable_fallback {
                    let target_blocks =
                        FeeEstimator::target_blocks_for_estimate_mode(estimate_mode);
                    self.tx_pool
                        .read()
                        .await
                        .estimate_fee_rate(target_blocks)
                        .map_err(Into::into)
                } else {
                    Err(err.into())
                }
            }
        }
    }
