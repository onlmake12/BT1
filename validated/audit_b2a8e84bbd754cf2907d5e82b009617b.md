### Title
Unprotected Direct Callback in Legacy `revealWithCallback` Path Permanently Bricks Entropy Requests — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `revealWithCallback` contains two execution paths. The **new path** (when `req.gasLimit10k != 0`) uses `excessivelySafeCall` to safely catch callback reverts and transitions the request to a recoverable `CALLBACK_FAILED` state. The **legacy path** (when `req.gasLimit10k == 0`, i.e., the provider has never called `setDefaultGasLimit`) makes a **bare, unprotected direct call** to the requester's `_entropyCallback`. If that callback reverts for any reason, the entire `revealWithCallback` transaction reverts, the request remains permanently stuck in `CALLBACK_NOT_STARTED` state, and there is no recovery mechanism.

---

### Finding Description

In `revealWithCallback`, after verifying the provider's proof, the contract branches on `req.gasLimit10k`: [1](#0-0) 

When `gasLimit10k == 0` (legacy path), the request is cleared first (checks-effects-interactions), then a **bare external call** is made to the requester's contract: [2](#0-1) 

If `_entropyCallback` reverts, the entire transaction reverts — including the `clearRequest` call at line 666. The request returns to `CALLBACK_NOT_STARTED` state. Since `revealWithCallback` only accepts requests in `CALLBACK_NOT_STARTED` or `CALLBACK_FAILED` state, the provider can retry, but every retry will revert again. There is no mechanism to mark the request as permanently failed or to refund the requester.

This is set up at request time: when a provider has `defaultGasLimit == 0`, `requestHelper` explicitly sets `req.gasLimit10k = 0`, opting into the legacy path: [3](#0-2) 

The new safe path uses `excessivelySafeCall` and a `CALLBACK_FAILED` recovery state, but this is only available when `gasLimit10k != 0`: [4](#0-3) 

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` confirms this behavior — it explicitly `vm.expectRevert()` when the callback reverts under the legacy path (provider with no `defaultGasLimit` set): [5](#0-4) 

---

### Impact Explanation

1. **Permanently stuck request:** The request occupies a storage slot indefinitely with no way to clear it.
2. **Provider keeper DoS:** The Fortuna keeper service (`apps/fortuna`) will repeatedly attempt `revealWithCallback` for the stuck request, burning gas on every attempt with no success path.
3. **Fee loss:** The requester's fee was credited to the provider at request time and cannot be refunded, so the requester permanently loses their fee without receiving randomness. [6](#0-5) 

---

### Likelihood Explanation

Any user can deploy a consumer contract whose `_entropyCallback` unconditionally reverts, then call `requestWithCallback` against a provider that has not set `defaultGasLimit` (i.e., `defaultGasLimit == 0`). This is a zero-cost setup: the attacker pays the request fee but permanently DoS's the provider's keeper for that slot. Multiple such requests amplify the keeper gas drain. The `requestWithCallback` function is permissionless and callable by anyone: [7](#0-6) 

---

### Recommendation

1. **Deprecate the legacy path entirely.** Require all providers to set `defaultGasLimit > 0` before accepting new `requestWithCallback` calls. Reject requests when `providerInfo.defaultGasLimit == 0`.
2. **Add a timeout/cancellation mechanism.** Allow a request to be cancelled (with partial refund) after a configurable number of failed reveal attempts, regardless of path.
3. **Wrap the legacy callback in a try/catch.** As a minimal fix, wrap the bare call at line 676 in a `try/catch` and transition to a failed state analogous to `CALLBACK_FAILED`.

---

### Proof of Concept

1. Deploy a provider with `defaultGasLimit == 0` (never calls `setDefaultGasLimit`).
2. Deploy a malicious consumer contract:
   ```solidity
   function _entropyCallback(uint64, address, bytes32) external pure override {
       revert("always fails");
   }
   ```
3. Call `requestWithCallback{value: fee}(provider, userRandomNumber)` from the malicious consumer.
4. Attempt `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)` — it reverts every time.
5. The request is permanently stuck. The provider's Fortuna keeper burns gas on every retry. The requester's fee is unrecoverable. [8](#0-7)
