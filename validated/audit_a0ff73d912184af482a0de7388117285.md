### Title
Cross-Chain Signature Replay in `verifyUpdate()` Due to Missing Chain Binding — (`File: lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` hashes only the raw payload bytes with no chain ID or contract address included in the signed digest. Because the same trusted signer keys are registered on every EVM deployment and the contract is deployed to the same deterministic address on all chains, a valid signed Lazer update from one EVM chain is cryptographically valid on every other EVM chain.

---

### Finding Description

In `PythLazer.verifyUpdate()`, the signature verification digest is computed as:

```solidity
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` is structured as `PAYLOAD_FORMAT_MAGIC | timestamp | channel | feedsData` — it contains no `block.chainid` and no `address(this)`:

```solidity
bytes memory payload = abi.encodePacked(
    uint32(2479346549), // PAYLOAD_FORMAT_MAGIC
    timestamp,
    uint8(channel),
    uint8(feedsData.length)
    // ... feed data
);
``` [2](#0-1) 

The deployment script explicitly deploys `PythLazer` to the **same deterministic address** (`0xACeA761c27A909d4D3895128EBe6370FDE2dF481`) on every EVM chain using CreateX's `create3`: [3](#0-2) 

This is confirmed by the contract store, which lists the same proxy address across Ethereum, Arbitrum, BSC, Monad, Berachain, HyperEVM, Fantom Sonic, and others: [4](#0-3) 

The same trusted signer keys are registered on all these deployments. Because the signed digest is `keccak256(payload)` with no chain-specific context, a signature that is valid on chain A is equally valid on chain B.

---

### Impact Explanation

An unprivileged relayer or Lazer updater can:

1. Observe a valid `update` blob accepted on chain A (e.g., Ethereum mainnet).
2. Submit the **identical bytes** to `verifyUpdate()` on chain B (e.g., Arbitrum, BSC, Monad, etc.).
3. `verifyUpdate()` will return `(payload, signer)` with `isValidSigner(signer) == true` on chain B.

Consumer contracts that call `verifyUpdate()` and trust its return value will accept the replayed price data. Concretely:

- If the Lazer service is not actively pushing to chain B (e.g., a newly added chain, a chain experiencing downtime, or a chain with lower priority), an attacker can replay updates from chain A to chain B, injecting price data that was never intended for chain B at that moment.
- Consumer contracts that gate actions on a successful `verifyUpdate()` call (e.g., liquidations, settlements) can be triggered with replayed data.
- The `signer` return value, which some consumers may use for access control, is identical across all chains for the same update. [5](#0-4) 

---

### Likelihood Explanation

- The `PythLazer` contract is live on 10+ EVM chains at the same address with the same trusted signers.
- All `update` blobs are publicly observable on-chain (calldata is public).
- No special access is required — any address can call `verifyUpdate()` with any `update` bytes.
- The attacker only needs to copy calldata from a transaction on chain A and resubmit it on chain B.
- The deployment script comment `uint8(0), // No crosschain replay protection` in the salt generation confirms this protection is absent at the contract level. [3](#0-2) 

---

### Recommendation

Include `block.chainid` (and optionally `address(this)`) in the signed digest inside `verifyUpdate()`:

```solidity
bytes32 hash = keccak256(abi.encodePacked(block.chainid, address(this), payload));
```

This binds each signature to a specific chain and contract instance, preventing cross-chain replay. The Lazer signing service would need to produce chain-specific signatures accordingly. Alternatively, adopt EIP-712 with a domain separator that includes `chainId` and `verifyingContract`.

---

### Proof of Concept

1. Deploy `PythLazer` on two EVM chains (e.g., Ethereum fork at chainId 1 and Arbitrum fork at chainId 42161) with the same trusted signer.
2. On chain A, call `verifyUpdate{value: fee}(update)` with a valid signed update — it succeeds.
3. Copy the exact `update` bytes.
4. On chain B, call `verifyUpdate{value: fee}(update)` with the **same bytes** — it also succeeds and returns the same `(payload, signer)`.
5. A consumer contract on chain B that calls `verifyUpdate()` and uses the result to update prices will accept the replayed data, even though the Lazer service never sent an update to chain B. [6](#0-5) [7](#0-6)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
```
