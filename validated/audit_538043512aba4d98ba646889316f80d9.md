### Title
Missing Zero-Price Check Causes Division-by-Zero Revert in `deriveCrossRate` - (File: `target_chains/ethereum/sdk/solidity/PythUtils.sol`)

---

### Summary

`PythUtils.deriveCrossRate` validates that `price2` is not negative but does not check whether `price2 == 0`. Both computation branches divide by `uint64(price2)` (directly or as part of a product), so passing `price2 = 0` triggers a Solidity panic revert (division by zero) rather than a clean, descriptive error. Any on-chain protocol that feeds a zero-valued Pyth price into this function will have its transaction silently reverted.

---

### Finding Description

`deriveCrossRate` in `PythUtils.sol` computes `price1 / price2` scaled to a target exponent. The only guard on `price2` is:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
``` [1](#0-0) 

This rejects negative values but silently allows `price2 == 0`. Both downstream branches then pass `uint64(price2)` as the denominator to `Math.mulDiv`:

**Branch 1 (`deltaExponent > 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    10 ** uint64(deltaExponent),
    uint64(price2)          // denominator = 0 → panic
);
``` [2](#0-1) 

**Branch 2 (`deltaExponent <= 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    1,
    10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // product = 0 → panic
);
``` [3](#0-2) 

`Math.mulDiv` explicitly panics when the denominator is zero:

```solidity
if (high == 0) {
    return low / denominator;   // Solidity reverts on denominator == 0
}
if (denominator <= high) {
    panic(ternary(denominator == 0, DIVISION_BY_ZERO, UNDER_OVERFLOW));
}
``` [4](#0-3) 

The function's own NatSpec documents only the negative-price revert, not a zero-price revert, so integrators have no reason to guard against it:

```
/// @dev This function will revert if either price is negative or if the exponents are less than -255.
``` [5](#0-4) 

---

### Impact Explanation

Any protocol that calls `deriveCrossRate` with a Pyth price feed whose current value is `0` (e.g., a newly listed asset, a depegged stablecoin, or a feed that has not yet been initialized) will receive an opaque panic revert instead of a descriptive error. This breaks the calling contract's execution flow and can cause a denial-of-service for any function that depends on the cross-rate result — including liquidations, swaps, or collateral valuations — for as long as the denominator feed reports zero.

---

### Likelihood Explanation

Pyth price feeds are `int64` values and can legitimately be `0`. The `deriveCrossRate` function is `public`, callable by any address, and is the recommended SDK utility for computing synthetic pairs (e.g., ETH/EUR from ETH/USD and EUR/USD). The official documentation and example code show direct use of live `getPriceNoOlderThan` results as inputs without any zero-price guard on the caller side, because the function's own documentation implies it handles all non-negative inputs. [6](#0-5) 

---

### Recommendation

Add an explicit zero check for `price2` before the division, mirroring the existing negative-price guard:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
if (price2 == 0) {
    revert PythErrors.InvalidInputPrice(); // or a new ZeroDenominatorPrice error
}
```

Update the NatSpec to document this revert condition so integrators are aware.

---

### Proof of Concept

```solidity
// Any caller can trigger this:
PythUtils.deriveCrossRate(
    1_000_000,   // price1 = some valid price
    -8,          // expo1
    0,           // price2 = 0  ← passes the `< 0` check, reaches mulDiv
    -8,          // expo2
    -8           // targetExponent
);
// Result: Solidity panic (division by zero), not PythErrors.NegativeInputPrice
```

With `deltaExponent = (-8) - ((-8) + (-8)) = 8 > 0`, execution enters Branch 1 and calls `Math.mulDiv(1_000_000, 10**8, 0)`, which hits the `low / denominator` path in `Math.mulDiv` and reverts with a raw arithmetic panic instead of a descriptive Pyth error. [7](#0-6)
