### Title
Fee Charged Without Price State Update When Submitted Timestamp Is Not Strictly Greater Than Cached Timestamp — (`target_chains/sui/contracts/sources/pyth.move`)

### Summary
`update_single_price_feed` unconditionally deposits the caller's fee coins into the `PriceInfoObject` before checking whether the submitted price is actually fresher than the cached one. When `is_fresh_update` returns false (i.e., `update_timestamp <= cached_timestamp`), the price write is silently skipped but the fee is permanently retained, violating the invariant that fees should only be consumed when a state change occurs.

### Finding Description
In `update_single_price_feed`:

```
assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);
price_info::deposit_fee_coins(price_info_object, fee);   // ← fee taken unconditionally
...
update_cache(latest_only, cur_price_info, price_info_object, clock);  // ← may be a no-op
``` [1](#0-0) 

Inside `update_cache`, the write is gated on `is_fresh_update`:

```move
if (is_fresh_update(update, price_info_object)){
    price_info::update_price_info_object(price_info_object, update);
}
// else: silent no-op, fee already gone
``` [2](#0-1) 

`is_fresh_update` uses strict `>`, so equal timestamps also produce a no-op: [3](#0-2) 

`deposit_fee_coins` is `public` and has no paired withdrawal function in `price_info.move`: [4](#0-3) 

There is no `withdraw_fee_coins` or any other mechanism to recover coins from the `FEE_STORAGE_KEY` dynamic field. Deposited coins are permanently locked. [5](#0-4) 

### Impact Explanation
Any caller of `update_single_price_feed` who submits a price update whose timestamp is ≤ the currently cached timestamp loses their fee SUI coins permanently. The coins are locked in the `PriceInfoObject` with no recovery path. This is a direct, permanent loss of user-paid funds with zero corresponding state change.

### Likelihood Explanation
This is reachable without any privilege. Two realistic paths:

1. **Race condition / MEV**: User A fetches a VAA and prepares a transaction. Before it lands, User B (or a bot) submits a newer update. User A's transaction executes with a now-stale timestamp, fee is taken, no update occurs.
2. **Repeated submission**: A user re-submits the same VAA (e.g., retrying a failed transaction) after the price has already been updated with an equal or newer timestamp. Fee is charged again, no update occurs.

Both paths require only a public entry point and no privileged access.

### Recommendation
Move the fee deposit **after** the freshness check, or refund the fee if `is_fresh_update` returns false. For example, in `update_single_price_feed`, only call `deposit_fee_coins` inside the branch where `update_cache` actually writes state, or return the `fee` coin to the caller when the update is skipped. Alternatively, abort with a descriptive error when the submitted update is not fresh, consistent with how `E_PRICE_UPDATE_NOT_FOUND_FOR_PRICE_INFO_OBJECT` is handled.

### Proof of Concept
1. Initialize Pyth with `base_update_fee = 50`.
2. Create a `PriceInfoObject` with a cached price at timestamp `T`.
3. Construct a `HotPotatoVector<PriceInfo>` (via `create_price_infos_hot_potato`) containing a price with timestamp `T` (equal, not greater).
4. Call `update_single_price_feed` with 50 MIST fee coins.
5. Assert: `price_info::get_balance(&price_info_object) == 50` (fee deposited).
6. Assert: the price data in `price_info_object` is unchanged (timestamp still `T`, same price values).

This directly confirms fee drain with no state change. The existing test at line 817 already asserts `get_balance` increases on a successful update — the same assertion holds for a stale update, demonstrating the bug. [6](#0-5)

    fun is_fresh_update(update: &PriceInfo, price_info_object: &PriceInfoObject): bool {

    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
