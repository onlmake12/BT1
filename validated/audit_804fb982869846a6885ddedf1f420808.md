### Title
Permanent Freezing of Protocol Update Fees â€” (`target_chains/sui/contracts/sources/price_info.move`)

---

### Summary

Every call to `update_single_price_feed` deposits `Coin<SUI>` into a `PriceInfoObject` dynamic field (`FEE_STORAGE_KEY`). No function in any module can remove those coins. All update fees paid since deployment are permanently frozen.

---

### Finding Description

`pyth::update_single_price_feed` unconditionally calls `price_info::deposit_fee_coins`, which either creates or `coin::join`s into the `FEE_STORAGE_KEY` dynamic field of the `PriceInfoObject`: [1](#0-0) [2](#0-1) 

`price_info.move` exposes only two public functions touching `FEE_STORAGE_KEY`: `deposit_fee_coins` (write, additive only) and `get_balance` (read-only). There is no `withdraw_fee_coins`, no `take_fee_coins`, and no function that calls `dynamic_object_field::remove` or `coin::split` on that field anywhere in the contract sources: [3](#0-2) 

`state.move` stores a `fee_recipient_address` field and exposes `get_fee_recipient`, but no code path ever reads that address to route coins out of any `PriceInfoObject`: [4](#0-3) [5](#0-4) 

The governance module enumerates six actions (`CONTRACT_UPGRADE` through `SET_FEE_RECIPIENT`); none of them withdraw fees from `PriceInfoObject`: [6](#0-5) 

The `set_fee_recipient` module itself contains an explicit admission that fee routing was removed:

> "The previous version of the contract sent the fees to a recipient address but **this state is not used anymore**. This module is kept for backward compatibility." [7](#0-6) 

---

### Impact Explanation

Every `Coin<SUI>` paid as an update fee since deployment is locked inside individual `PriceInfoObject` shared objects with no on-chain code path to recover them. The `fee_recipient_address` is a dead field. Protocol revenue is 100% unclaimable under the current bytecode.

---

### Likelihood Explanation

This is not a probabilistic risk â€” it is a structural invariant violation present in every transaction that calls `update_single_price_feed`. Any caller (unprivileged) triggers it. The balance can only increase, never decrease, as confirmed by static analysis of all public and `friend` functions in the module.

---

### Recommendation

Add a privileged withdrawal function (gated by `LatestOnly` or a governance VAA) to `price_info.move` that calls `dynamic_object_field::remove` or `coin::split` on `FEE_STORAGE_KEY` and transfers the resulting `Coin<SUI>` to `state::get_fee_recipient(pyth_state)`. Alternatively, route fees directly to `fee_recipient_address` inside `update_single_price_feed` instead of storing them in the `PriceInfoObject`.

---

### Proof of Concept

Static invariant proof:

1. Search all functions in `target_chains/sui/contracts/sources/` for any call that removes or splits the `FEE_STORAGE_KEY` dynamic field â€” result: zero matches.
2. `get_balance(price_info_object)` can only return a value â‰Ą its previous value after any sequence of public calls.
3. Invariant test: after N calls to `update_single_price_feed` with fee `f` each, `get_balance(pio) == N * f` and no public function reduces it. [8](#0-7) 

The existing test itself asserts fees accumulate and never checks for a withdrawal path, confirming the design omission.

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
