Audit Report

## Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Computed Values After In-Flight Evictions ‚Äî (`tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, the new aggregate totals are computed as local variables before `check_and_record_ancestors` is called. `check_and_record_ancestors` can evict existing pool entries via `remove_entry_and_descendants` ‚Üí `remove_entry` ‚Üí `update_stat_for_remove_tx`, which decrements `self.total_tx_size` in place. The final unconditional assignment then overwrites the correctly-decremented live field with the stale pre-mutation snapshot, permanently overcounting `total_tx_size` by the sum of evicted entries' sizes. This overcount causes `limit_size` to evict additional legitimate transactions that would not otherwise have been removed.

## Finding Description

**Root cause ‚Äî stale snapshot overwrite:**

In `add_entry` (pool_map.rs L210‚Äì219):
```
L210: let (total_tx_size, total_tx_cycles) =
          self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
      // Snapshot: total_tx_size = self.total_tx_size + entry.size
      // self.total_tx_size is NOT yet modified.

L213: evicts = self.check_and_record_ancestors(&mut entry)?;
      // May call remove_entry_and_descendants ‚Üí remove_entry ‚Üí
      // update_stat_for_remove_tx, which DECREMENTS self.total_tx_size in place.

L218: self.total_tx_size = total_tx_size;   // ‚Üê OVERWRITES the decremented value
L219: self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (L711‚Äì729) only performs an overflow check and returns candidate totals; it does not commit them to `self`.

`check_and_record_ancestors` (L588‚Äì640) enters the eviction branch when `ancestors_count > self.max_ancestors_count` but `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count`. In that branch it calls `self.remove_entry_and_descendants(next_id)` (L618), which calls `remove_entry` for each removed entry (L252‚Äì264), which calls `self.update_stat_for_remove_tx(entry.inner.size, entry.inner.cycles)` (L247), which decrements `self.total_tx_size` in place (L738‚Äì740).

After `check_and_record_ancestors` returns, `self.total_tx_size` correctly reflects `original ‚àí evicted_sizes`. The assignment at L218 then sets it back to `original + entry.size`, discarding the eviction credits:

```
correct value  = original ‚àí evicted_sizes + entry.size
reported value = original + entry.size          (overcounted by evicted_sizes)
```

**Eviction path is confirmed reachable:** `cell_ref_parents` are populated at L531‚Äì533 when the new transaction's input's `previous_output` is found in `self.edges.deps` ‚Äî i.e., when some pool transaction has declared that same output as a `cell_dep`. This is a standard, documented scenario.

**Downstream effect:** `limit_size` (pool.rs L298) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting entries until the condition is false. An overcounted `total_tx_size` causes it to evict `evicted_sizes` worth of additional legitimate transactions that should have remained in the pool.

## Impact Explanation

The overcounted `total_tx_size` causes `limit_size` to evict legitimate pending/proposed transactions with `Reject::Full`, even though the pool has physical space. An attacker can exploit this to selectively or broadly evict valid transactions from the mempool. During reorgs, `_update_tx_pool_for_reorg` calls `limit_size(callbacks, None)` (process.rs L1113), meaning the overcount can cause arbitrary pending/proposed transactions to be evicted on every reorg without any current-entry guard. Repeated triggering keeps the mempool in a degraded state, forcing honest users to continuously resubmit transactions and contributing to **network congestion with few costs** ‚Äî matching the High (10001‚Äì15000 points) impact class.

## Likelihood Explanation

The trigger requires submitting a transaction whose input spends an output that is simultaneously referenced as a `cell_dep` by one or more existing pool transactions, and where the resulting ancestor count exceeds `max_ancestors_count` (default 25) but the non-cell-ref ancestor count does not. An attacker can deliberately construct this scenario: submit N transactions each declaring a known on-chain output `O` as a `cell_dep`, then submit a transaction spending `O` as an input. This is an unprivileged `send_transaction` RPC call. The setup cost scales with N but the damage (overcount = sum of evicted sizes) also scales with N, making the attack economically viable for disruption purposes.

## Recommendation

Remove the pre-computation of `(total_tx_size, total_tx_cycles)` before `check_and_record_ancestors`. After all mutations are complete, update the live fields directly:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Overflow check only ‚Äî do not capture snapshot totals here
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Now add against the live (already eviction-decremented) fields:
    self.total_tx_size = self.total_tx_size
        .checked_add(entry.size)
        .expect("overflow already checked above");
    self.total_tx_cycles = self.total_tx_cycles
        .checked_add(entry.cycles)
        .expect("overflow already checked above");
    Ok((true, evicts))
}
```

Alternatively, add a post-insertion invariant assertion:
```rust
debug_assert_eq!(self.total_tx_size, self.recompute_total_stat().unwrap().0);
```

## Proof of Concept

1. Create a `PoolMap` with `max_ancestors_count = 25`.
2. Submit 26 transactions `T1‚Ä¶T26`, each declaring output `O` of an on-chain transaction `P` as a `cell_dep`. Each `Ti` has no pool-internal parents, so `ancestors_count = 1` for each. Record `total_tx_size` after each insertion; it should equal `26 * MOCK_SIZE`.
3. Submit `T_attack` whose **input** spends output `O` of `P`. Inside `add_entry`:
   - `updated_stat_for_add_tx` snapshots `total_tx_size = 26*S + S_attack`.
   - `check_and_record_ancestors` finds `cell_ref_parents = {T1‚Ä¶T26}`, `ancestors_count = 27 > 25`. It evicts enough `Ti` entries (e.g., 2) via `remove_entry_and_descendants`, decrementing `self.total_tx_size` by `2*S` to `24*S`.
   - The assignment at L218 sets `self.total_tx_size = 26*S + S_attack` ‚Äî overcounted by `2*S`.
4. Assert `pool_map.total_tx_size == 26*S + S_attack` (overcounted) vs. correct value `24*S + S_attack`.
5. Call `limit_size` with a `max_tx_pool_size` just above `24*S + S_attack`; observe that it evicts additional legitimate transactions due to the inflated counter, even though the pool is physically within limits.
