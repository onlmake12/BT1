### Title
Block Assembler Effective Size/Cycles Cap Is Lower Than Intended Due to Shared Ancestor Accumulator - (File: tx-pool/src/component/tx_selector.rs)

---

### Summary

In `TxSelector::txs_to_commit`, the check that decides whether a transaction fits within the block's size and cycles limits uses `tx_entry.ancestors_size` / `tx_entry.ancestors_cycles`, which includes ancestors that have **already been fetched and counted** in the running `size`/`cycles` accumulators. This double-counts already-included ancestors, making the effective block capacity cap lower than the protocol-intended limit. Miners assemble blocks smaller than the maximum allowed, losing fee revenue.

---

### Finding Description

`TxSelector::txs_to_commit` selects transactions from the proposed pool to fill a block up to `size_limit` and `cycles_limit`. It maintains two running accumulators:

```rust
let mut size: usize = 0;
let mut cycles: Cycle = 0;
```

These are incremented **only for newly-fetched (not yet seen) transactions**:

```rust
for (short_id, entry) in &ancestors {
    let is_new = self.fetched_txs.insert(short_id.clone());
    if !is_new {
        continue;          // already counted — skip
    }
    cycles = cycles.saturating_add(entry.cycles);
    size   = size.saturating_add(entry.size);
```

However, the **cap check** that guards whether to include a transaction uses `tx_entry.ancestors_size` and `tx_entry.ancestors_cycles`, which are the **total** size/cycles of the transaction **plus all its in-pool ancestors**, regardless of whether those ancestors are already in `fetched_txs`:

```rust
let next_size   = size.saturating_add(tx_entry.ancestors_size);
let next_cycles = cycles.saturating_add(tx_entry.ancestors_cycles);

if next_cycles > cycles_limit || next_size > size_limit {
    consecutive_failed += 1;
    ...
    continue;
}
```

`ancestors_size` is defined as the sum of the tx's own size plus every in-pool ancestor's size (set at entry creation and updated as ancestors are added):

```rust
pub ancestors_size: usize,
...
ancestors_size: size,   // initialized to own size
...
pub fn add_ancestor_weight(&mut self, entry: &TxEntry) {
    self.ancestors_size = self.ancestors_size.saturating_add(entry.size);
```

So when transaction B (with ancestor A already in `fetched_txs`) is evaluated:

- `size` already includes A's size (say 100 bytes)
- `tx_entry.ancestors_size` = size(B) + size(A) = 200 bytes
- `next_size` = 100 + 200 = 300 bytes — but the **actual new bytes** to be added is only size(B) = 100 bytes

The check rejects B if `next_size > size_limit`, even though the real new contribution is only size(B). The effective cap is therefore `size_limit - already_fetched_ancestor_sizes`, which is lower than the intended `size_limit`.

---

### Impact Explanation

Miners assemble blocks that are smaller (in bytes and cycles) than the protocol-permitted maximum. Transactions that would fit within the block are incorrectly excluded because the cap check double-counts already-included ancestors. This directly reduces miner fee revenue per block. The chain remains consensus-valid (blocks are within limits), but miners are systematically under-compensated relative to what the protocol allows.

**Impact: Low** (no funds at risk, no consensus violation; miner revenue is reduced)

---

### Likelihood Explanation

Transaction chains (where tx B spends an output of tx A, both in the pool) are the normal operating mode of CKB's two-phase commit system. Any tx-pool submitter who creates a chain of two or more transactions triggers this code path during block assembly. On a busy mempool with many chained transactions, the effective block capacity can be significantly below the limit.

**Likelihood: High** (triggered by ordinary transaction chains, which are ubiquitous)

---

### Recommendation

Replace the cap check with the **actual new size/cycles** that would be added — i.e., only the ancestors not yet present in `fetched_txs`:

```rust
// Compute only the truly new contribution before the check
let new_size   = ancestors_ids.iter()
    .filter(|id| !self.fetched_txs.contains(*id))
    .map(|id| self.pool_map.get(id).map_or(0, |e| e.size))
    .sum::<usize>()
    .saturating_add(tx_entry.size);

let new_cycles = /* analogous */;

let next_size   = size.saturating_add(new_size);
let next_cycles = cycles.saturating_add(new_cycles);
```

Alternatively, track a separate "projected size" that accounts for already-fetched ancestors when evaluating each candidate.

---

### Proof of Concept

Consider `size_limit = 250`, two transactions A (size=100) and B (size=100, ancestor=A):

1. A is evaluated first: `next_size = 0 + 100 = 100 ≤ 250` → accepted. `size = 100`, A added to `fetched_txs`.
2. B is evaluated next: `tx_entry.ancestors_size = 200` (A+B). `next_size = 100 + 200 = 300 > 250` → **rejected**.
3. But the actual new bytes B would add = 100. `100 + 100 = 200 ≤ 250` → B **should** have been accepted.

The miner's block is 100 bytes instead of 200 bytes, losing B's fee.
