### Title
Missing Strict Slot Ordering in `validateTwapPriceInfo` Enables Division-by-Zero DoS in `parseTwapPriceFeedUpdates` — (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary

`parseTwapPriceFeedUpdates` validates the start/end TWAP pair with a non-strict slot comparison (`>`), allowing equal `publishSlot` values to pass. `calculateTwap` then unconditionally divides by `slotDiff`, which is `0` when slots are equal, causing a Solidity 0.8 arithmetic panic (revert). Any unprivileged caller can trigger this by submitting the same valid Wormhole-signed VAA as both `updateData[0]` and `updateData[1]`.

### Finding Description

In `validateTwapPriceInfo`, the slot ordering check is:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

This only reverts when `start > end`. When `start == end` (equal slots), validation passes. The Solana receiver's equivalent check is strict:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [2](#0-1) 

After passing validation, `calculateTwap` computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot;
// slotDiff == 0 when slots are equal
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // division by zero
``` [3](#0-2) 

Solidity 0.8+ checked arithmetic causes a panic revert on any of these three divisions when `slotDiff == 0`.

The attacker path is straightforward: obtain any single legitimate Wormhole-signed TWAP VAA for a price feed (these are publicly available from Hermes), then call `parseTwapPriceFeedUpdates` with that same VAA passed as both `updateData[0]` and `updateData[1]`. The per-feed price ID check at lines 537–541 passes (same data), the individual message validity checks at lines 591–598 pass (valid `prevPublishTime < publishTime`), the exponent check passes, and the slot check passes because `start == end` is not rejected. [4](#0-3) 

### Impact Explanation

Any protocol that integrates `parseTwapPriceFeedUpdates` and allows user-supplied `updateData` (e.g., a lending protocol that lets borrowers trigger their own TWAP liquidation check) can be DoS'd: the attacker submits the same VAA twice, the call reverts with a division-by-zero panic, and the protocol action (liquidation, price check, etc.) is blocked. The fee paid via `msg.value` is returned on revert, so the attacker's only cost is gas. The protocol's TWAP-dependent logic is rendered non-functional for that call.

**Impact: Medium** — targeted DoS of TWAP-dependent protocol actions; no direct fund loss from the Pyth contract itself, but downstream protocols relying on TWAP for time-sensitive operations (liquidations, settlement) can be blocked.

### Likelihood Explanation

**Likelihood: Medium** — the attack requires only a single valid public VAA (freely available from Hermes) and a single transaction. No privileged access, no key compromise, no Sybil attack. The inconsistency between the EVM and Solana implementations (non-strict vs. strict slot comparison) is the root cause and is not guarded by any other check.

### Recommendation

Change the slot comparison in `validateTwapPriceInfo` to use strict inequality, matching the Solana implementation:

```solidity
// Before (non-strict — allows equal slots → slotDiff = 0 → division by zero)
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}

// After (strict — rejects equal slots, consistent with Solana receiver)
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

Alternatively, add an explicit `slotDiff > 0` guard at the top of `calculateTwap` before any division.
