### Title
Unbounded `while` Loop in `Echo.executeCallback()` Can Cause OOG, Blocking Callback Fulfillment â€” (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past all already-cleared requests. An attacker can pre-fill the sequence space with many fulfilled requests so that a subsequent `executeCallback` call iterates over all of them in a single transaction, exhausting gas and reverting â€” permanently blocking that request from being fulfilled and locking the requester's fee.

---

### Finding Description

Inside `executeCallback`, after clearing the current request, the following loop runs:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

Each iteration performs a storage read (`findRequest` + `isActive`) and a storage write (`_state.firstUnfulfilledSeq++`). There is no cap on how many iterations this loop can execute. The developer even left a self-acknowledgment in the code:

> *"TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number."*

The loop advances the pointer past every *inactive* (already-cleared) request. If a large contiguous block of requests has been cleared before the request at `firstUnfulfilledSeq` is cleared, the loop must scan the entire block in one transaction.

**Attack path:**

1. Attacker creates N requests (sequence numbers `k+1` through `k+N`), paying the required fee for each. `firstUnfulfilledSeq` is currently `k`.
2. After the exclusivity period expires, anyone may call `executeCallback`. The attacker (or a cooperating party) calls `executeCallback` for requests `k+1` through `k+N-1`, clearing them. `firstUnfulfilledSeq` remains at `k` because request `k` is still active.
3. The attacker calls `executeCallback` for request `k`. After `clearRequest(k)`, the `while` loop must now advance `firstUnfulfilledSeq` from `k` through `k+N`, performing N storage reads and N storage writes. With a sufficiently large N, this exhausts the block gas limit and the entire transaction reverts.
4. Request `k` is never cleared; its fee remains locked in the contract; the consumer never receives its callback. [1](#0-0) 

---

### Impact Explanation

- **Callback permanently blocked**: Because the OOG revert undoes `clearRequest`, the request stays active but can never be successfully fulfilled â€” every subsequent attempt to fulfill it will re-trigger the same oversized loop and revert again.
- **Fee locked**: The requester's fee (`req.fee`) is held in the contract and is never returned, since there is no refund path for a stuck active request.
- **Provider fee withheld**: The provider cannot collect their accrued fee for this request. [2](#0-1) 

---

### Likelihood Explanation

**Medium.** After the configurable `exclusivityPeriodSeconds` elapses, `executeCallback` is callable by any address â€” not just the assigned provider. An attacker therefore controls both the creation of requests and their out-of-order fulfillment. The cost is the sum of fees for N requests, which is bounded only by the attacker's budget. The `NUM_REQUESTS` fixed-size array and overflow mapping impose no effective cap on the total number of live sequence numbers, so N can be made arbitrarily large. [3](#0-2) [4](#0-3) 

---

### Recommendation

Replace the unbounded `while` loop with a bounded alternative. Two options:

1. **Cap the loop**: advance `firstUnfulfilledSeq` by at most a fixed number of steps per call (e.g., 50), accepting that the pointer may lag.
2. **Remove the pointer entirely**: replace `firstUnfulfilledSeq` with a doubly-linked list of active requests (as the developer's own TODO suggests), so that clearing a request is O(1) and no scan is needed. [5](#0-4)
