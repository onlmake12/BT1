### Title
Missing Block-Height Bound Check in `GetTransactionsProofProcess::execute` Allows Unbounded Server-Side MMR Computation With No Peer Ban — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

---

### Summary

An unprivileged light-client peer can craft a `GetTransactionsProof` message with `last_hash` pointing to an old main-chain block at height H and `tx_hashes` referencing transactions committed in blocks at height > H. Because `execute()` never checks that a found transaction's block number is ≤ `last_block.number()`, it passes out-of-range MMR positions to `reply_proof()`, which calls `mmr.gen_proof()` on an MMR that is too small to contain those positions. `gen_proof` returns `Err`, the server returns `StatusCode::InternalError` (HTTP-500-class), and — critically — the 500-class code is **never** a ban trigger. The peer is not disconnected and can repeat the request indefinitely.

---

### Finding Description

**Step 1 — Entry point.**
Any peer connected on the light-client protocol can send a `GetTransactionsProof` message. The only size guard is `GET_TRANSACTIONS_PROOF_LIMIT = 1000`. [1](#0-0) 

**Step 2 — `last_hash` validation.**
The server checks only that `last_hash` is on the main chain. It does **not** check that the referenced transactions' blocks are at height ≤ `last_block.number()`. [2](#0-1) 

**Step 3 — `partition()` has no height guard.**
The `partition()` call classifies a tx as "found" if and only if it is on the main chain. A transaction in a block at height 10 is on the main chain even when `last_hash` is at height 5. [3](#0-2) 

**Step 4 — Positions are computed from the actual block number, not clamped to `last_block.number()`.**
`leaf_index_to_pos(block.number())` is called unconditionally. If `block.number() > last_block.number()`, the resulting position is outside the MMR that `reply_proof` will build. [4](#0-3) 

**Step 5 — `reply_proof` builds an MMR of size `leaf_index_to_mmr_size(H-1)` and calls `gen_proof` with out-of-range positions.**
The MMR covers only blocks 0 … H−1. Positions for blocks > H are outside its range; `gen_proof` returns `Err`. [5](#0-4) 

**Step 6 — `InternalError` (500) never triggers a ban.**
`should_ban()` only returns `Some(ban_time)` for codes in `[400, 500)`. Code 500 falls outside that range, so the peer is never banned or disconnected — only a `warn!` log is emitted. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Each crafted request forces the server to:
1. Perform up to 1,000 DB lookups (`get_transaction_info`, `get_transaction_with_info`, `get_block`).
2. Build a CBMT Merkle proof for each affected block.
3. Compute the MMR root (`mmr.get_root()` — succeeds).
4. Attempt `mmr.gen_proof()` with out-of-range positions — fails, discarding all prior work.

The attacker pays only the cost of sending a small fixed-size P2P message. The server pays CPU + I/O proportional to `GET_TRANSACTIONS_PROOF_LIMIT`. With no ban and no rate limit enforced at this layer, the attacker can loop this indefinitely, exhausting server resources.

---

### Likelihood Explanation

The attack requires no privilege, no key, no PoW, and no Sybil capability. Any peer that can open a light-client protocol connection can trigger it. The crafted message is trivially constructable: pick any old confirmed block as `last_hash`, then supply hashes of transactions confirmed after that block. Both are publicly observable on-chain.

---

### Recommendation

In `execute()`, after the `partition()` step, filter out any "found" transaction whose block number exceeds `last_block.number()`, moving it to `missing` instead:

```rust
// After partition, before building txs_in_blocks:
let (found, extra_missing): (Vec<_>, Vec<_>) = found.into_iter().partition(|tx_hash| {
    snapshot
        .get_transaction_info(tx_hash)
        .map(|info| {
            snapshot.get_block_number(&info.block_hash)
                .map(|n| n <= last_block.number())
                .unwrap_or(false)
        })
        .unwrap_or(false)
});
missing.extend(extra_missing);
```

The same missing guard exists in `get_blocks_proof.rs` at line 85 and should receive an identical fix. [8](#0-7)

### Proof of Concept

```
1. Mine a chain to height 10.
2. Record tx_hash T = hash of any transaction in block at height 10.
3. Record block_hash B5 = hash of the main-chain block at height 5.
4. Send GetTransactionsProof { last_hash: B5, tx_hashes: [T] }.
5. Observe server log: "process GetTransactionsProof from <peer>; result is
   InternalError(500): failed to generate a proof since ..."
6. Observe peer is NOT banned.
7. Repeat step 4 in a tight loop.
```

The server performs full DB + CBMT work on every iteration before failing at `gen_proof`.
