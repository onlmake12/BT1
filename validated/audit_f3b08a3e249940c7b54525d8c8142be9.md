### Title
Excess ETH Sent to `requestPriceUpdatesWithCallback` Is Permanently Transferred to Provider Instead of Refunded - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

---

### Summary

`Echo.sol`'s `requestPriceUpdatesWithCallback` only enforces a lower-bound check on `msg.value` (`msg.value < requiredFee` reverts) but imposes no upper-bound check. Any ETH sent above the required fee is silently stored as the provider's fee and paid out to the provider upon `executeCallback`. The caller permanently loses the excess ETH with no refund mechanism.

---

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback` computes the required fee and validates only the minimum:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [1](#0-0) 

It then stores the entire `msg.value` minus only the fixed Pyth protocol fee as the provider's fee:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

Only `_state.pythFeeInWei` is credited to Echo's accrued fees:

```solidity
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [3](#0-2) 

When `executeCallback` is later called, the provider receives:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [4](#0-3) 

Because `req.fee` already encodes the full `msg.value_request − pythFeeInWei`, any excess above `requiredFee` flows directly into the provider's withdrawable balance. There is no path for the original caller to recover it.

The interface NatDoc states `"The msg.value must be equal to getFee(callbackGasLimit)"` — implying exact payment is expected — but this is never enforced on-chain: [5](#0-4) 

By contrast, `PythLazer.sol`'s `verifyUpdate` correctly refunds excess:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [6](#0-5) 

---

### Impact Explanation

Any caller who sends `msg.value > getFee(provider, callbackGasLimit, priceIds)` permanently loses the excess ETH to the provider. The provider can withdraw it via `withdrawAsFeeManager`. The user has no recourse. For a single overpayment of even a small amount (e.g., a frontend rounding up by 1 gwei or a user manually specifying a round number), the funds are irrecoverable by the payer.

---

### Likelihood Explanation

`getFee` is a multi-component calculation (`pythFeeInWei + baseFeeInWei + feePerFeedInWei × N + feePerGasInWei × gasLimit`). Frontends, scripts, or integrators that round up, add a buffer, or hardcode a "safe" value will routinely overpay. The pattern is common in practice and the contract provides no protection against it.

---

### Recommendation

Add an exact-match check (or refund excess) in `requestPriceUpdatesWithCallback`:

```solidity
// Option A: strict equality
if (msg.value != requiredFee) revert IncorrectFee();

// Option B: refund excess (mirrors PythLazer pattern)
if (msg.value > requiredFee) {
    payable(msg.sender).transfer(msg.value - requiredFee);
}
```

---

### Proof of Concept

1. Provider is registered with `baseFeeInWei = 1000`, `feePerFeedInWei = 100`, `feePerGasInWei = 1`. `pythFeeInWei = 500`.
2. User requests 2 price feeds with `callbackGasLimit = 100_000`. `requiredFee = 500 + 1000 + 200 + 100_000 = 101_700 wei`.
3. User sends `msg.value = 0.001 ether = 1_000_000_000_000_000 wei` (a common round number).
4. `req.fee = 1_000_000_000_000_000 − 500 = 999_999_999_999_500`.
5. Provider calls `executeCallback`; `pythFee` (Pyth price-feed update fee) ≈ 1 wei.
6. Provider accrues `999_999_999_999_500 + 0 − 1 = 999_999_999_999_499 wei`.
7. User has permanently lost `≈ 0.001 ETH − 101_700 wei ≈ 0.000999... ETH` to the provider with no refund path.
