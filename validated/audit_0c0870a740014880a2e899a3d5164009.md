### Title
Silent Truncating Cast `withdraw_counted_capacity as u64` Missing Overflow Check in `DaoCalculator::calculate_maximum_withdraw` - (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes a `u128` intermediate value `withdraw_counted_capacity` and then casts it to `u64` using the Rust `as u64` operator, which **silently truncates** the high bits on overflow. Every other analogous `u128 → u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, which propagates a proper error. The missing overflow guard means that when `withdraw_counted_capacity` exceeds `u64::MAX`, the returned `withdraw_capacity` is silently wrong (far too small), causing any DAO withdrawal transaction whose cell capacity is large enough to trigger the overflow to be incorrectly rejected by the node.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the maximum withdrawable capacity for a NervosDAO cell:

```rust
// util/dao/src/lib.rs  lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a **truncating** (wrapping) cast in Rust: if `withdraw_counted_capacity > u64::MAX`, the upper bits are silently discarded and the result wraps to a small value with no error returned.

Every other `u128 → u64` narrowing in the same file uses the checked form:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)

// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The inconsistency is the root cause. The formula is:

```
withdraw_counted_capacity = counted_capacity × withdrawing_ar / deposit_ar
```

where `withdrawing_ar` and `deposit_ar` are the NervosDAO accumulation-rate fields extracted from committed block headers. [5](#0-4) 

`counted_capacity` is itself a `u64` (cell capacity minus occupied capacity). If `withdrawing_ar / deposit_ar > u64::MAX / counted_capacity`, the `u128` product overflows `u64` and the `as u64` cast silently wraps it to a small value.

The truncated `withdraw_capacity` is then used by `transaction_maximum_withdraw` → `transaction_fee`:

```rust
// util/dao/src/lib.rs  lines 30-36
pub fn transaction_fee(&self, rtx: &ResolvedTransaction) -> Result<Capacity, DaoError> {
    let maximum_withdraw = self.transaction_maximum_withdraw(rtx)?;
    rtx.transaction
        .outputs_capacity()
        .and_then(|y| maximum_withdraw.safe_sub(y))   // ← underflows if maximum_withdraw is wrong
        .map_err(Into::into)
}
``` [6](#0-5) 

When `maximum_withdraw` is silently truncated to a small value, `safe_sub` returns `DaoError::Overflow`, which propagates up through `check_tx_fee` in the tx-pool:

```rust
// tx-pool/src/util.rs  lines 34-41
let fee = DaoCalculator::new(...)
    .transaction_fee(rtx)
    .map_err(|err| {
        Reject::Malformed(
            format!("{err}"),
            "expect (outputs capacity) <= (inputs capacity)".to_owned(),
        )
    })?;
``` [7](#0-6) 

The transaction is rejected as malformed even though it is structurally valid.

---

### Impact Explanation

A legitimate DAO withdrawal transaction is permanently rejected by any honest node that evaluates it. The user cannot recover their deposited CKB + interest. Because the rejection is based on a deterministic arithmetic error in `calculate_maximum_withdraw`, every node in the network will reject the same transaction, making the DoS network-wide and not bypassable by resubmission. The `calculate_maximum_withdraw` function is also exposed directly via the `calculate_dao_maximum_withdraw` RPC endpoint, so the RPC caller also receives a silently wrong (too small) value. [8](#0-7) 

---

### Likelihood Explanation

The overflow condition requires:

```
counted_capacity × withdrawing_ar > u64::MAX × deposit_ar
```

`ar` starts at `10_000_000_000_000_000` (10¹⁶) and grows slowly with secondary issuance. For the product to overflow `u64::MAX ≈ 1.84 × 10¹⁹`, a cell's `counted_capacity` would need to be close to `u64::MAX` **and** the accumulation rate would need to have grown by a factor proportional to `u64::MAX / counted_capacity`. Given the total CKB supply (~3.36 × 10¹⁸ shannons) and the slow growth of `ar`, this is extremely unlikely on mainnet under normal economic conditions. However, the code inconsistency is real and constitutes a latent correctness bug: the same file applies the safe pattern everywhere else, making this a clear omission rather than an intentional design choice.

---

### Recommendation

Replace the truncating cast with the same checked conversion used everywhere else in the file:

```rust
// Before (unsafe truncating cast):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (consistent with lines 204, 245, 258):
Capacity::shannons(
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
).safe_add(occupied_capacity)?
``` [9](#0-8) 

---

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` (lines 295–350) already sets up a cell with capacity near `u64::MAX` and asserts `result.is_err()`. [10](#0-9)  That test passes today only because the `safe_add(occupied_capacity)` at the end overflows — not because the `as u64` truncation is caught. A targeted test can demonstrate the silent truncation:

1. Construct a `CellOutput` with `capacity = u64::MAX` and zero occupied capacity (e.g., empty lock/type).
2. Set `deposit_ar = 1` and `withdrawing_ar = 2` in the block headers (achievable in a test environment by directly calling `pack_dao_data`).
3. Call `calculate_maximum_withdraw`.
4. With the current code, `withdraw_counted_capacity = u64::MAX * 2 / 1 = 2 * u64::MAX` which truncates via `as u64` to `u64::MAX - 1` (wraps), returning `Ok(Capacity::shannons(u64::MAX - 1))` — a silently wrong result instead of `Err(DaoError::Overflow)`.
5. With the fix (`u64::try_from`), the call correctly returns `Err(DaoError::Overflow)`.

fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
}
