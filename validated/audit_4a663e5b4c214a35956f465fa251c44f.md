### Title
`PythAggregatorV3.latestRoundData()` Returns Unvalidated Stale Price via `getPriceUnsafe()` — (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink `AggregatorV3Interface`-compatible adapter, designed as a drop-in replacement for protocols migrating from Chainlink. Every price-reading function in the contract — including `latestRoundData()`, `latestAnswer()`, `getRoundData()`, and `decimals()` — calls `pyth.getPriceUnsafe()`, which returns the last stored on-chain price with **no staleness check and no price validity check**. This is the direct structural analog to the Chainlink `minAnswer`/`maxAnswer` circuit-breaker issue: both result in a consuming protocol receiving and acting on a wrong price due to missing validation in the oracle adapter layer.

---

### Finding Description

`PythAggregatorV3.sol` implements the Chainlink `AggregatorV3Interface` by wrapping a Pyth price feed. All price-reading entry points use `getPriceUnsafe()`:

```solidity
// line 110 — latestRoundData()
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
roundId = uint80(price.publishTime);
return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);

// line 54 — latestAnswer()
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
return int256(price.price);

// line 89 — getRoundData()
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
return (_roundId, int256(price.price), price.publishTime, price.publishTime, _roundId);
```

`IPyth.getPriceUnsafe()` is explicitly documented as returning "the most recent price update in this contract without any recency checks" and warns that "the returned price update may be arbitrarily far in the past." [1](#0-0) 

No staleness check is applied. No check for `price.price <= 0` is applied. The `updatedAt` field returned to callers is simply `price.publishTime`, which can be arbitrarily old. [2](#0-1) 

By contrast, Pyth's own Aave adapter (`PythPriceOracleGetter.sol`) correctly uses `getPriceNoOlderThan()` with a configured `validTimePeriodSeconds` and explicitly reverts on `price.price <= 0`: [3](#0-2) 

The `PythAggregatorV3` adapter provides no equivalent protection.

---

### Impact Explanation

Protocols that deploy `PythAggregatorV3` as a Chainlink drop-in replacement and call `latestRoundData()` will receive a price that may be arbitrarily stale. In Pyth's pull-oracle model, on-chain prices are only updated when someone calls `updatePriceFeeds()`. If updates stop (network congestion, pusher failure, or deliberate attacker inaction), the stored price diverges from the real market price.

An attacker can exploit this divergence to:

- **Borrow against overvalued collateral**: If an asset crashes in real price but the stale on-chain Pyth price is still high, the attacker borrows the maximum against the inflated collateral value — exactly the Venus/LUNA scenario cited in the original report.
- **Avoid liquidation**: A position that should be liquidated at the real price is not liquidatable at the stale price.
- **Execute trades at wrong prices**: Any DeFi protocol using `PythAggregatorV3` for pricing (AMMs, perpetuals, options) will misprice at the stale value.

Additionally, `int256(price.price)` is returned without a positivity check. Pyth prices are `int64`; a negative price (which Pyth can represent) would be returned as a negative `int256 answer`, which most Chainlink-consuming protocols do not expect and do not guard against, potentially causing division-by-zero, underflow, or incorrect logic downstream. [4](#0-3) 

---

### Likelihood Explanation

- `PythAggregatorV3` is the **official, documented migration path** for Chainlink users. The Pyth developer hub explicitly instructs protocols to deploy it as a drop-in replacement. [5](#0-4) 
- Protocols migrating from Chainlink are accustomed to `latestRoundData()` returning a fresh price; they will not add their own staleness checks because they expect the adapter to handle it (as Chainlink's own circuit-breaker logic does).
- In Pyth's pull model, price staleness is a normal operational condition — any gap in the price pusher service creates the window.
- The attacker needs no privileged access: they only need to observe that the on-chain price is stale and then interact with the victim protocol.

---

### Recommendation

Replace `getPriceUnsafe()` with `getPriceNoOlderThan()` using a configurable `maxAge` parameter, and add a non-positive price guard:

```solidity
uint256 public maxAge; // set in constructor

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    require(price.price > 0, "Invalid non-positive price");
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

This mirrors the pattern already used in `PythPriceOracleGetter.sol`. [6](#0-5) 

---

### Proof of Concept

1. Protocol deploys `PythAggregatorV3` pointing to the ETH/USD Pyth feed and uses it as a Chainlink `AggregatorV3Interface` for collateral pricing.
2. The Pyth price pusher stops updating the on-chain ETH/USD price (e.g., ETH crashes from $3000 to $500; the pusher fails or the attacker simply does not call `updatePriceFeeds()`).
3. The on-chain Pyth price remains at $3000 (stale).
4. Attacker calls `latestRoundData()` on `PythAggregatorV3` — it calls `getPriceUnsafe()` and returns `answer = 3000e8`, `updatedAt = <old timestamp>`.
5. The protocol, not checking `updatedAt`, accepts the $3000 price.
6. Attacker deposits ETH as collateral, borrows the maximum against the $3000 valuation, and walks away with the borrowed assets. The collateral is worth only $500 at real market price.
7. The protocol is left with undercollateralized debt — the same outcome as the Venus/LUNA incident.
