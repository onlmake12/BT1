### Title
Redundant Repeated `block_median_time` Store Lookups in `SinceVerifier` Per-Input Loop - (File: `verification/src/transaction_verifier.rs`)

### Summary

`SinceVerifier::verify()` loops over every transaction input and, for each input carrying a timestamp-based `since` field, calls `self.block_median_time(&parent_hash)` — a function that itself performs up to 37 sequential RocksDB header lookups. Because `parent_hash` is derived from `self.tx_env` and is identical for every input in the same transaction, the same 37-lookup chain is re-executed once per input rather than once per transaction. An unprivileged sender can craft a large transaction with many timestamp-`since` inputs to force O(N × 37) store reads during tx-pool admission and block verification.

---

### Finding Description

**Root cause — `SinceVerifier::verify()` (outer loop):**

`SinceVerifier::verify()` iterates over every resolved input:

**Root cause — `block_median_time` is an O(37) store walk:**

```rust
// traits/src/header_provider.rs  L32-L50
fn block_median_time(&self, block_hash: &Byte32, median_block_count: usize) -> u64 {
    let mut timestamps: Vec<u64> = Vec::with_capacity(median_block_count);
    let mut block_hash = block_hash.clone();
    for _ in 0..median_block_count {          // ← up to 37 iterations
        let header_fields = self
            .get_header_fields(&block_hash)   // ← RocksDB read each time
            .expect("parent header exist");
        timestamps.push(header_fields.timestamp);
        block_hash = header_fields.parent_hash;
        if header_fields.number == 0 { break; }
    }
    timestamps.sort_unstable();
    timestamps[timestamps.len() >> 1]
}

**Scale of the redundancy:**

`max_block_bytes = 0x91c08 = 597,000 bytes`. A `CellInput` is 44 bytes (36-byte `OutPoint` + 8-byte `since`), so a maximum-size transaction can carry roughly **~13,500 inputs**. If every input uses an absolute timestamp `since`:

- `block_median_time(&parent_hash)` is called **13,500 times**, each doing **37 store reads** = **~499,500 store reads** for what is a single invariant value.
- Pre-CKB2021 relative timestamp `since` doubles this to **~999,000 store reads** per transaction.

The `SinceVerifier` is invoked in both the tx-pool admission path (`verify_rtx` in `tx-pool/src/util.rs`) and the block verification path (`BlockTxsVerifier` in `verification/contextual/src/contextual_block_verifier.rs`). [6](#0-5) [7](#0-6) 

---

### Impact Explanation

An unprivileged transaction sender can force the node to perform hundreds of thousands of sequential RocksDB reads during a single transaction's `SinceVerifier` pass. This amplifies CPU and I/O cost for the verifying node with no corresponding cost increase for the attacker (the transaction fee is proportional to byte size, not to the number of timestamp-since inputs). Under sustained submission of such transactions, the tx-pool worker thread and block-verification thread can be saturated, degrading or halting normal transaction processing and block acceptance — a resource-exhaustion DoS.

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction via the public `send_transaction` RPC or P2P relay — no privileged access, no key material, no majority hashpower. Constructing a transaction with many inputs referencing existing live cells and setting timestamp-based `since` values is straightforward. The transaction will be rejected (inputs are immature), but the expensive verification runs before rejection is returned. The attacker can repeat this at low cost.

---

### Recommendation

Cache the invariant `current_median_time` result before the input loop in `SinceVerifier::verify()`, since `self.tx_env.parent_hash()` does not change between inputs:

```rust
pub fn verify(&self) -> Result<(), Error> {
    // Compute once; reuse across all inputs.
    let cached_current_median_time: Option<u64> = None; // lazy-init on first timestamp since
    for (index, (cell_meta, input)) in ... {
        ...
        self.verify_absolute_lock(index, since, cached_current_median_time)?;
        self.verify_relative_lock(index, since, cell_meta, cached_current_median_time)?;
    }
    Ok(())
}
```

Similarly, `parent_median_time(&info.block_hash)` in the relative-timestamp branch varies per input (it depends on the cell's block hash), but `current_median_time` (from `self.tx_env.parent_hash()`) is always the same and should be computed once and passed in.

---

### Proof of Concept

1. Obtain ~13,500 live cells (or use a devnet with pre-funded outputs).
2. Construct a transaction spending all of them, setting each input's `since` field to an absolute timestamp value in the future (e.g., `0x4000_0000_FFFF_FFFF`).
3. Submit via `send_transaction` RPC.
4. The node runs `SinceVerifier::verify()`, which calls `block_median_time(&parent_hash)` 13,500 times, each performing 37 RocksDB reads (~499,500 reads total).
5. The transaction is ultimately rejected as `Immature`, but the node has already paid the full verification cost.
6. Repeat in a tight loop to sustain the load.
