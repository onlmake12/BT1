### Title
Malicious Outbound Peer Can Indefinitely Reset Chain-Sync Eviction Timeout - (File: `sync/src/synchronizer/mod.rs`)

---

### Summary

The `eviction()` function in `sync/src/synchronizer/mod.rs` resets the 12-minute `CHAIN_SYNC_TIMEOUT` whenever a peer's best-known header difficulty reaches the difficulty stored at the time the timeout was first set. A malicious outbound peer can exploit this by repeatedly relaying valid real-chain headers that are just one step behind the current local tip, resetting the eviction timer indefinitely and occupying an outbound connection slot without providing useful sync data.

---

### Finding Description

`eviction()` implements a chain-sync timeout to disconnect outbound peers that are not providing useful sync data. When a peer's best-known header has less work than the local tip, a timeout is set:

```rust
state.chain_sync.timeout = now + CHAIN_SYNC_TIMEOUT;   // 12 minutes
state.chain_sync.work_header = Some(tip_header);
state.chain_sync.total_difficulty = Some(local_total_difficulty);  // snapshot D_old
state.chain_sync.sent_getheaders = false;
```

The timeout is **reset** (extended by another 12 minutes, and `sent_getheaders` cleared back to `false`) when the peer's best-known header difficulty reaches the snapshotted difficulty:

```rust
} else if state.chain_sync.timeout == 0
    || (best_known_header.is_some()
        && best_known_header
            .map(|header_index| header_index.total_difficulty().clone())
            >= state.chain_sync.total_difficulty)   // >= D_old, not >= D_current
{
    state.chain_sync.timeout = now + CHAIN_SYNC_TIMEOUT;
    state.chain_sync.total_difficulty = Some(local_total_difficulty);  // now D_current
    state.chain_sync.sent_getheaders = false;
```

The comparison is against `state.chain_sync.total_difficulty` (the **old** snapshot), not against the **current** local tip difficulty. The first branch (which clears the timeout entirely) requires `peer_diff >= current_local_difficulty`. So the reset fires whenever:

```
D_old  <=  peer_best_known_difficulty  <  D_current
```

Because `sent_getheaders` is also reset to `false` on every reset, the peer is never actually challenged and the eviction branch (`eviction.push(*peer)`) is never reached.

**Attack steps:**

1. Attacker's node is discovered and connected to as an outbound peer by the victim.
2. Attacker's best-known header is behind the local tip → timeout set, `D_old = D_victim_tip`.
3. The real chain grows; local tip advances to `D_current > D_old`.
4. Attacker relays a valid real-chain header with difficulty `D_old ≤ peer_diff < D_current` (trivially obtained by following the canonical chain).
5. Reset condition fires: timeout extended 12 more minutes, `D_old` updated to `D_current`, `sent_getheaders = false`.
6. Repeat from step 3 indefinitely.

The attacker performs **zero PoW work**; they only relay headers that already exist on the canonical chain.

---

### Impact Explanation

A malicious peer can hold an outbound connection slot indefinitely without ever being evicted. CKB nodes have a fixed number of outbound slots (typically 8). If an attacker controls enough discoverable nodes to fill all outbound slots, the victim node is eclipsed: it cannot sync new blocks from honest peers, enabling transaction censorship or double-spend attacks against the isolated node.

---

### Likelihood Explanation

Moderate. The attacker only needs to:
1. Run one or more CKB nodes with public IPs and get them into the victim's peer store (standard peer-discovery propagation).
2. Automate sending one valid canonical header every ~11 minutes per occupied slot.

No PoW, no key material, no privileged access is required. The attack is fully scriptable and cheap to sustain.

---

### Recommendation

Once the timeout has been set for a peer, do not reset it based on incremental header improvements that still fall short of the **current** local tip. The reset condition should compare against `local_total_difficulty` (current), not `state.chain_sync.total_difficulty` (snapshot):

```rust
// Only fully clear the timeout if the peer has caught up to the CURRENT tip.
// Do not reset the timer for partial progress.
} else if state.chain_sync.timeout == 0 {
    state.chain_sync.timeout = now + CHAIN_SYNC_TIMEOUT;
    ...
}
```

Alternatively, once `sent_getheaders` has been set to `true`, never reset it back to `false` for the same timeout cycle, mirroring the fix applied to the 0x MultiSig issue (allow "over-progress" without resetting the clock).

### Proof of Concept

```
T=0:   victim tip = D0; attacker best_known = D_low < D0
       → timeout = T+12min, chain_sync.total_difficulty = D0, sent_getheaders = false

T=5min: victim tip grows to D1 > D0
        attacker sends valid header H1 with total_difficulty = D0
        → D0 >= D0 (reset condition) AND D0 < D1 (not-clear condition)
        → timeout = T+5min+12min, chain_sync.total_difficulty = D1, sent_getheaders = false

T=10min: victim tip grows to D2 > D1
         attacker sends valid header H2 with total_difficulty = D1
         → D1 >= D1 AND D1 < D2
         → timeout reset again, sent_getheaders = false

    pub fn eviction(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let active_chain = self.shared.active_chain();
        let mut eviction = Vec::new();
        let better_tip_header = self.better_tip_header();
        for mut kv_pair in self.peers().state.iter_mut() {
            let (peer, state) = kv_pair.pair_mut();
            let now = unix_time_as_millis();

            if let Some(ref mut controller) = state.headers_sync_controller {
                let better_tip_ts = better_tip_header.timestamp();
                if let Some(is_timeout) = controller.is_timeout(better_tip_ts, now) {
                    if is_timeout {
                        eviction.push(*peer);
                        continue;
                    }
                } else {
                    active_chain.send_getheaders_to_peer(
                        nc,
                        *peer,
                        better_tip_header.number_and_hash(),
                    );
                }
            }

            // On ibd, node should only have one peer to sync headers, and it's state can control by
            // headers_sync_controller.
            //
            // The header sync of other nodes does not matter in the ibd phase, and parallel synchronization
            // can be enabled by unknown list, so there is no need to repeatedly download headers with
            // multiple nodes at the same time.
            if active_chain.is_initial_block_download() {
                continue;
            }
            if state.peer_flags.is_outbound {
                let best_known_header = state.best_known_header.as_ref();
                let (tip_header, local_total_difficulty) = {
                    (
                        active_chain.tip_header().to_owned(),
                        active_chain.total_difficulty().to_owned(),
                    )
                };
                if best_known_header
                    .map(|header_index| header_index.total_difficulty().clone())
                    .unwrap_or_default()
                    >= local_total_difficulty
                {
                    if state.chain_sync.timeout != 0 {
                        state.chain_sync.timeout = 0;
                        state.chain_sync.work_header = None;
                        state.chain_sync.total_difficulty = None;
                        state.chain_sync.sent_getheaders = false;
                    }
                } else if state.chain_sync.timeout == 0
                    || (best_known_header.is_some()
                        && best_known_header
                            .map(|header_index| header_index.total_difficulty().clone())
                            >= state.chain_sync.total_difficulty)
                {
                    // Our best block known by this peer is behind our tip, and we're either noticing
                    // that for the first time, OR this peer was able to catch up to some earlier point
                    // where we checked against our tip.
                    // Either way, set a new timeout based on current tip.
                    state.chain_sync.timeout = now + CHAIN_SYNC_TIMEOUT;
                    state.chain_sync.work_header = Some(tip_header);
                    state.chain_sync.total_difficulty = Some(local_total_difficulty);
                    state.chain_sync.sent_getheaders = false;
                } else if state.chain_sync.timeout > 0 && now > state.chain_sync.timeout {
                    // No evidence yet that our peer has synced to a chain with work equal to that
                    // of our tip, when we first detected it was behind. Send a single getheaders
                    // message to give the peer a chance to update us.
                    if state.chain_sync.sent_getheaders {
                        if state.peer_flags.is_protect || state.peer_flags.is_whitelist {
                            if state.sync_started() {
                                self.shared().state().suspend_sync(state);
                            }
                        } else {
                            eviction.push(*peer);
                        }
                    } else {
                        state.chain_sync.sent_getheaders = true;
                        state.chain_sync.timeout = now + EVICTION_HEADERS_RESPONSE_TIME;
                        active_chain.send_getheaders_to_peer(
                            nc,
                            *peer,
                            state
                                .chain_sync
                                .work_header
                                .as_ref()
                                .expect("work_header be assigned")
                                .into(),
                        );
                    }
                }
            }
        }
        for peer in eviction {
            info!("Timeout eviction peer={}", peer);
            if let Err(err) = nc.disconnect(peer, "sync timeout eviction") {
                debug!("synchronizer disconnect error: {:?}", err);
            }
        }
    }

// State used to enforce CHAIN_SYNC_TIMEOUT
// Only in effect for connections that are outbound, non-manual,
// non-protected and non-whitelist.
// Algorithm: if a peer's best known block has less work than our tip,
// set a timeout CHAIN_SYNC_TIMEOUT seconds in the future:
//   - If at timeout their best known block now has more work than our tip
//     when the timeout was set, then either reset the timeout or clear it
//     (after comparing against our current tip's work)
//   - If at timeout their best known block still has less work than our
//     tip did when the timeout was set, then send a getheaders message,
//     and set a shorter timeout, HEADERS_RESPONSE_TIME seconds in future.
//     If their best known block is still behind when that new timeout is
//     reached, disconnect.

#[derive(Clone, Debug, Default)]
pub struct ChainSyncState {
    pub timeout: u64,
    pub work_header: Option<core::HeaderView>,
    pub total_difficulty: Option<U256>,
    pub sent_getheaders: bool,
    headers_sync_state: HeadersSyncState,
}
