### Title
Missing `commitment != bytes32(0)` Validation in `Entropy.register()` Allows Permanently Unfulfillable Requests - (File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol)

### Summary

`Entropy.sol`'s `register()` function accepts a `bytes32 commitment` parameter with only a `chainLength == 0` guard. No check prevents `commitment == bytes32(0)`. A provider who registers with a zero commitment can never fulfill any request assigned to them, because fulfillment requires finding a preimage of zero under keccak256 — computationally infeasible. Every user who pays a fee to request from this provider loses their fee and receives no random number.

### Finding Description

The `register()` function in `Entropy.sol` stores the caller-supplied `commitment` directly as both `originalCommitment` and `currentCommitment` with no zero-value guard:

```solidity
function register(
    uint128 feeInWei,
    bytes32 commitment,       // ← no check: commitment != bytes32(0)
    bytes calldata commitmentMetadata,
    uint64 chainLength,
    bytes calldata uri
) public override {
    if (chainLength == 0) revert EntropyErrors.AssertionFailure();  // only guard

    ...
    provider.originalCommitment = commitment;
    provider.currentCommitment  = commitment;   // stored as-is
    ...
}
``` [1](#0-0) 

During reveal, `revealHelper` verifies the provider's contribution by computing `constructProviderCommitment(req.numHashes, providerContribution)` and requiring it to equal `currentCommitment`:

```solidity
bytes32 providerCommitment = constructProviderCommitment(
    req.numHashes,
    providerContribution
);
// providerCommitment must equal the stored currentCommitment
if (
    keccak256(bytes.concat(userCommitment, providerCommitment)) !=
    req.commitment
) revert EntropyErrors.IncorrectRevelation();
``` [2](#0-1) 

`constructProviderCommitment(numHashes, x)` computes `keccak256^{numHashes}(x)`. For the first request after registration `numHashes = 1`, so the provider must supply `x` such that `keccak256(x) == bytes32(0)`. This is computationally infeasible. Every subsequent request has `numHashes >= 1`, so the invariant holds: **no valid revelation exists for a provider whose `currentCommitment` is `bytes32(0)`**.

The `IEntropy` interface comment for `setFeeManager` explicitly documents that `address(0)` is a valid sentinel to disable the fee manager role, confirming the codebase does use zero-value sentinels intentionally in some places — but `commitment = bytes32(0)` has no such documented intent and no protocol-level escape hatch. [3](#0-2) 

### Impact Explanation

Any user who pays the fee to request randomness from a provider whose `currentCommitment` is `bytes32(0)` will:

1. Have their request stored on-chain with a commitment that can never be satisfied.
2. Lose the fee paid (`pythFeeInWei + provider.feeInWei`) with no refund mechanism.
3. Never receive a callback or a random number — the request is permanently stuck.

If the admin later sets such a provider as the `defaultProvider`, every new request routed to the default provider is affected. [4](#0-3) 

### Likelihood Explanation

`register()` is a public, permissionless function — any address can call it. A malicious actor can:

1. Call `register(lowFee, bytes32(0), ..., chainLength, uri)` to appear as a cheap, legitimate provider.
2. Attract users who select the cheapest provider or who are directed to it by an integrating dApp.
3. Collect fees from every request while being structurally unable to fulfill any of them.

A legitimate provider could also trigger this accidentally (human error during deployment), which is the exact scenario the analogous AeraVault report highlighted.

### Recommendation

Add a zero-value guard at the top of `register()`:

```solidity
if (commitment == bytes32(0)) revert EntropyErrors.AssertionFailure();
```

This mirrors the existing `chainLength == 0` guard and prevents both accidental and malicious zero-commitment registrations. [5](#0-4) 

### Proof of Concept

```solidity
// 1. Attacker registers with zero commitment
vm.prank(attacker);
entropy.register(
    1 wei,          // low fee to attract users
    bytes32(0),     // zero commitment — no check prevents this
    hex"",
    100,
    bytes("https://attacker.example.com")
);

// 2. User pays fee and requests randomness from attacker's provider
vm.deal(user, entropy.getFee(attacker));
vm.prank(user);
uint64 seq = entropy.requestV2{value: entropy.getFee(attacker)}(
    attacker,
    entropy.constructUserCommitment(bytes32(uint256(42))),
    gasLimit
);

// 3. Attacker attempts to reveal — must find x s.t. keccak256(x) == bytes32(0)
//    This is computationally infeasible. The request is permanently stuck.
//    User's fee is lost; no callback is ever delivered.
vm.prank(attacker);
vm.expectRevert(EntropyErrors.IncorrectRevelation.selector);
entropy.revealWithCallback(attacker, seq, bytes32(uint256(42)), bytes32(0));
