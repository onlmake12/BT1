### Title
Unbounded Orphan Block Pool Growth via Non-Contextual Verification Bypass — (`chain/src/utils/orphan_block_pool.rs`, `chain/src/chain_service.rs`)

### Summary

The `OrphanBlockPool` has no enforced size cap. `with_capacity` is only a `HashMap` pre-allocation hint, not a limit. `InnerPool::insert` performs no size check. `BlockVerifier` (used in `non_contextual_verify`) does not include PoW verification. An attacker can flood the pool with structurally valid, PoW-invalid blocks at the current epoch, which are never evicted by the epoch-based cleanup, causing unbounded memory growth.

### Finding Description

**No hard cap on pool size.**
`ORPHAN_BLOCK_SIZE = BLOCK_DOWNLOAD_WINDOW as usize` is passed to `OrphanBlockPool::with_capacity`, which calls `HashMap::with_capacity` — a pre-allocation hint only. [1](#0-0) [2](#0-1) 

`InnerPool::insert` has no size guard: [3](#0-2) 

**PoW is not checked in `non_contextual_verify`.**
`BlockVerifier::verify` only runs `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier` — no `PowVerifier`: [4](#0-3) 

PoW verification lives in `HeaderVerifier` (context-dependent, requires parent), not in the non-contextual path used by `ChainService`: [5](#0-4) 

**Insertion path has no size check.**
After passing `non_contextual_verify` and `insert_block`, `process_lonely_block` inserts into the orphan pool unconditionally when the parent is unknown: [6](#0-5) 

**Expiry logic does not protect against current-epoch blocks.**
`need_clean` only evicts blocks where `epoch_number + 6 < tip_epoch`. Blocks at `epoch = tip_epoch` satisfy `tip_epoch + 6 < tip_epoch = false` and are never cleaned: [7](#0-6) 

The cleanup timer fires every 60 seconds and is the only eviction mechanism: [8](#0-7) 

The code comment explicitly confirms no LRU eviction exists: [9](#0-8) 

**Double impact: each block is also written to the DB before orphan insertion:** [10](#0-9) 

### Impact Explanation

An attacker relaying structurally valid (but PoW-invalid) blocks with unknown parents at the current epoch causes:
1. Unbounded growth of the in-memory `OrphanBlockPool` (`parents` HashMap, `blocks` HashMap, `leaders` HashSet)
2. Unbounded disk growth (each block is written to the DB before orphan insertion)
3. Node OOM crash or disk exhaustion

### Likelihood Explanation

The attack requires only P2P block relay. Blocks must pass `BlockVerifier` (valid merkle root, valid cellbase structure, within byte limits) but do **not** need valid PoW. This is achievable by any unprivileged peer. The epoch-based cleanup provides zero protection as long as the attacker uses the current epoch. The 60-second timer is irrelevant since cleanup never fires for current-epoch blocks.

### Recommendation

1. Enforce a hard maximum on `OrphanBlockPool` size in `InnerPool::insert`, rejecting or evicting (e.g., oldest-first) when the cap is reached.
2. Consider adding a non-contextual PoW check (PoW verification itself is stateless — it only needs the header) before accepting blocks into the orphan pool.
3. Alternatively, add a per-peer rate limit on orphan block insertion.

### Proof of Concept

```
1. Craft N blocks with:
   - Valid cellbase (dummy input with block number in `since`)
   - Valid merkle root over transactions
   - epoch = tip_epoch (current epoch)
   - parent_hash = random unknown hashes (different per block)
   - Any nonce (PoW not checked)
2. Relay all N blocks via P2P SendBlock messages.
3. Each block passes non_contextual_verify (no PoW check).
4. Each block is written to DB via insert_block.
5. Each block enters orphan pool via process_lonely_block (parent unknown).
6. Wait 60 seconds; clean_expired_orphans fires.
7. need_clean returns false for all blocks (tip_epoch + 6 < tip_epoch is false).
8. Pool remains at size N. Repeat indefinitely.
9. Assert: orphan_block_pool.len() grows without bound → OOM.
```

    fn insert(&mut self, lonely_block: LonelyBlockHash) {
        let hash = lonely_block.hash();
        let parent_hash = lonely_block.parent_hash();
        self.blocks
            .entry(parent_hash.clone())
            .or_default()
            .insert(hash.clone(), lonely_block);
        // Out-of-order insertion needs to be deduplicated
        self.leaders.remove(&hash);
        // It is a possible optimization to make the judgment in advance,
        // because the parent of the block must not be equal to its own hash,
        // so we can judge first, which may reduce one arc clone
        if !self.parents.contains_key(&parent_hash) {
            // Block referenced by `parent_hash` is not in the pool,
            // and it has at least one child, the new inserted block, so add it to leaders.
            self.leaders.insert(parent_hash.clone());
        }
        self.parents.insert(hash, parent_hash);
    }
