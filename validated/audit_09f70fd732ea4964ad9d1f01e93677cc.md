### Title
Excess ETH Sent by Executor in `executeCallback()` Is Credited to `providerToCredit` Instead of Being Refunded to `msg.sender` â€” (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol::executeCallback()`, the executor (`msg.sender`) sends `msg.value` to cover the Pyth oracle update fee. After paying the actual Pyth fee (`pythFee`), the entire remainder `msg.value - pythFee` is credited to `providerToCredit`'s accrued balance rather than being refunded to the executor. When `msg.sender != providerToCredit`, any overpayment by the executor is silently transferred to a different party.

---

### Finding Description

`executeCallback()` is called by an executor (keeper) who supplies `updateData` and pays `msg.value` to cover the Pyth oracle fee. The contract computes the actual Pyth fee on-chain via `pyth.getUpdateFee(updateData)` and pays it to the Pyth contract. The remainder is then credited entirely to `providerToCredit`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [1](#0-0) 

Here `req.fee` is the provider's fee stored from the original request (= `msg.value_at_request - pythFeeInWei`), and `msg.value` is the ETH the executor sends at callback time to cover the Pyth oracle fee. The formula credits `req.fee + (msg.value - pythFee)` to `providerToCredit`. If `msg.value > pythFee`, the excess `msg.value - pythFee` goes to `providerToCredit`, not back to `msg.sender`.

The function signature accepts a caller-supplied `providerToCredit` that is explicitly allowed to differ from `msg.sender` after the exclusivity period elapses:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
``` [2](#0-1) 

After the exclusivity window, any address can call `executeCallback` with any `providerToCredit`. The executor who pays `msg.value` is not guaranteed to be the same as `providerToCredit`, so any overpayment is misdirected.

The Pyth oracle fee is computed on-chain at execution time from `updateData`:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
``` [3](#0-2) 

Because the executor must estimate this fee off-chain before submitting the transaction, they routinely send a small buffer above the expected fee to avoid reverts. That buffer is silently absorbed by `providerToCredit`.

---

### Impact Explanation

Any executor who calls `executeCallback` with `msg.value` slightly above the actual Pyth oracle fee loses the excess ETH to `providerToCredit`. When `msg.sender != providerToCredit` (which is always possible after the exclusivity period), the executor suffers a direct ETH loss with no recourse. A provider who is aware of this can register as `providerToCredit` and benefit from executors' overpayments. Over many fulfilled requests this constitutes a steady, unintended wealth transfer from executors to providers.

---

### Likelihood Explanation

Executors must estimate `pyth.getUpdateFee(updateData)` off-chain before submitting the transaction. Because the Pyth fee depends on the number of price feed updates in `updateData` and can change between estimation and execution, executors routinely add a small buffer. After the exclusivity period any unprivileged address can call `executeCallback` with an arbitrary `providerToCredit`, making the `msg.sender != providerToCredit` case common in practice. No privileged access is required to trigger the loss.

---

### Recommendation

After paying the Pyth oracle fee, compute the actual excess and refund it to `msg.sender`:

```solidity
uint256 excess = msg.value - pythFee;
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128(req.fee);
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "Refund failed");
}
```

This mirrors the pattern used in `PythLazer.sol::verifyUpdate()` where excess fee is explicitly refunded to `msg.sender`:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [4](#0-3) 

---

### Proof of Concept

1. Alice registers as a provider and sets `providerToCredit = Alice`.
2. Bob (executor, `msg.sender`) calls `executeCallback(Alice, seqNum, updateData, priceIds)` with `msg.value = 0.001 ETH`.
3. `pythFee = pyth.getUpdateFee(updateData)` returns `0.0009 ETH`.
4. The contract executes: `Alice.accruedFeesInWei += req.fee + 0.001 - 0.0009 = req.fee + 0.0001 ETH`.
5. Bob's `0.0001 ETH` overpayment is credited to Alice, not refunded to Bob.
6. Bob has no way to recover the excess. Alice can call `withdrawAsFeeManager` to extract it. [5](#0-4)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(

            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );
