### Title
Unenforced `max_request_body_size` + Unbounded Heap Allocation in `JsonBytes::visit_str` Enables Local OOM via Concurrent RPC Requests — (`util/jsonrpc-types/src/bytes.rs`, `rpc/src/server.rs`)

---

### Summary

The `max_request_body_size` configuration field is **dead code**: it is stored in `RpcConfig` but never passed to or applied in the axum HTTP server. No `DefaultBodyLimit` or `RequestBodyLimitLayer` is present anywhere in the codebase. Combined with `JsonBytes::visit_str` allocating a heap buffer proportional to the attacker-controlled hex string length before any tx-pool gate runs, N concurrent `send_transaction` requests each carrying a near-maximum-size witness can exhaust node heap memory and crash the process.

---

### Finding Description

**Dead config — `max_request_body_size` is never enforced.**

`RpcConfig` carries the field: [1](#0-0) 

In `RpcServer::new()`, the value is never forwarded to `start_server()`: [2](#0-1) 

`start_server()` builds the axum `Router` with `CorsLayer` and `TimeoutLayer` but **no body-limit layer**: [3](#0-2) 

A grep for `max_request_body_size`, `DefaultBodyLimit`, and `RequestBodyLimitLayer` across all of `rpc/src/` returns zero matches. The config value is read from TOML and then silently discarded.

**Unbounded allocation in `JsonBytes::visit_str`.**

The handler receives the raw body as `Bytes` with no size check, then `serde_json` drives deserialization into `JsonBytes`. Inside `visit_str`, before any validity or tx-pool check: [4](#0-3) 

Line 104 allocates `vec![0; bytes.len() >> 1]` — exactly half the hex-string byte count — proportional to attacker input, with no upper bound.

**Call path:**

```
HTTP POST /  (no body limit enforced)
  → handle_jsonrpc (rpc/src/server.rs:218)
    → serde_json::from_str::<Request>
      → JsonBytes::deserialize
        → BytesVisitor::visit_str
          → vec![0; bytes.len() >> 1]   ← attacker-controlled allocation
            → tx-pool admission (never reached if OOM)
``` [5](#0-4) 

---

### Impact Explanation

Each concurrent request carrying a witness hex string of length L allocates ~L/2 bytes on the heap before any tx-pool check. With no body-size enforcement and no per-connection concurrency limit, N simultaneous requests each with L ≈ 10 MiB (or larger, since the limit is unenforced) produce N × 5 MiB+ of live heap allocations. On a typical node with 8–16 GB RAM, a few hundred concurrent connections suffice to trigger OOM, crashing the process, terminating all peer connections, and halting block propagation.

---

### Likelihood Explanation

The default RPC binding is `127.0.0.1:8114`: [6](#0-5) 

Any unprivileged local process can reach this. Operators who expose the RPC port (common for infrastructure nodes) extend the attack surface to the network. The `Pool` module is enabled by default: [7](#0-6) 

The bug is exploitable with a trivial HTTP client loop; no cryptographic material, PoW, or privileged access is required.

---

### Recommendation

1. **Enforce `max_request_body_size` in the axum server.** Add `tower_http::limit::RequestBodyLimitLayer::new(config.max_request_body_size)` to the router layer stack in `start_server()`. Axum has no implicit body limit.
2. **Add a size gate in `JsonBytes::visit_str`** (or at the deserialization entry point) that rejects strings exceeding a configurable maximum before allocating the decode buffer.
3. **Add a per-IP or per-connection concurrency limit** at the axum layer to bound simultaneous in-flight requests.

---

### Proof of Concept

```python
import threading, requests, time

WITNESS = "0x" + "aa" * 4_999_998   # ~10 MiB hex → ~5 MiB alloc per call

payload = {
    "id": 1, "jsonrpc": "2.0", "method": "send_transaction",
    "params": [{
        "version": "0x0", "cell_deps": [], "header_deps": [],
        "inputs": [{"previous_output": {"tx_hash": "0x" + "00"*32, "index": "0x0"}, "since": "0x0"}],
        "outputs": [], "outputs_data": [],
        "witnesses": [WITNESS]   # JsonBytes::visit_str allocates vec![0; 4_999_998]
    }, "passthrough"]
}

def send():
    try:
        requests.post("http://127.0.0.1:8114", json=payload, timeout=60)
    except Exception:
        pass

threads = [threading.Thread(target=send) for _ in range(200)]
for t in threads: t.start()
for t in threads: t.join()
# Expected: node process OOM-killed; all peer connections dropped
```

`max_request_body_size` is never enforced, so requests larger than 10 MiB are also accepted, amplifying the per-request allocation further.
