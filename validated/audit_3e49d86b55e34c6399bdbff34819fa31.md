### Title
Permissionless `initialize_pool` Allows Anyone to Front-Run Pool Initialization and Set Malicious `pool_data` and `reward_program_authority` - (File: governance/pyth_staking_sdk/src/idl/integrity-pool.json)

---

### Summary

The `initialize_pool` instruction of the integrity-pool program accepts an arbitrary `payer` (any signer) with no authority constraint, and accepts `pool_data` as a plain writable account with no PDA derivation or ownership check. Because `pool_config` is a singleton PDA that can only be initialized once, any unprivileged actor who submits the transaction first permanently sets the pool's `pool_data` address and `reward_program_authority` for the entire protocol.

---

### Finding Description

The `initialize_pool` instruction is defined in the integrity-pool IDL as follows:

```
accounts:
  payer          – signer, writable  (no authority constraint)
  config_account – PDA from staking program (read-only validation)
  pool_data      – writable          (arbitrary account, no PDA seeds, no owner check)
  pool_config    – writable PDA      (seeds: ["pool_config"])
  slash_custody  – unconstrained
  system_program

args:
  reward_program_authority: pubkey   (fully attacker-controlled)
  y: u64
``` [1](#0-0) 

The `payer` field carries only `"signer": true` and `"writable": true` — there is no `relations`, `has_one`, or `constraint` attribute tying it to any authority stored in `config_account` or elsewhere. `pool_data` is a plain writable account with no PDA seeds, meaning the caller supplies any account they own. [2](#0-1) 

`pool_config` is a singleton PDA (`seeds: ["pool_config"]`). Anchor's `init` constraint prevents re-initialization, so the first transaction that executes `initialize_pool` wins permanently. [3](#0-2) 

All subsequent instructions that read pool state use `pool_config` as the source of truth for `pool_data` via Anchor `relations`: [4](#0-3) 

The SDK's `initializePool` client method confirms no privileged wallet is enforced — it simply sends the transaction from whoever holds `this.wallet`: [5](#0-4) 

---

### Impact Explanation

An attacker who front-runs `initialize_pool` before the legitimate deployer:

1. Supplies a `pool_data` account they control, permanently recorded in `pool_config.pool_data`.
2. Sets themselves as `reward_program_authority`, gaining the ability to call privileged instructions such as `update_delegation_fee` and `advance` that require `reward_program_authority` as a signer via `relations: ["pool_config"]`. [6](#0-5) 

All staking users who subsequently `delegate`, `undelegate`, or `advance` will interact with the attacker-controlled `pool_data`, enabling reward manipulation, incorrect slash accounting, and potential theft of staked PYTH tokens.

---

### Likelihood Explanation

The attack window is the gap between contract deployment and the legitimate `initialize_pool` call. On Solana, a mempool-watching bot can observe the deployer's `initialize_pool` transaction and submit an identical one with higher priority fees (front-run). This is a well-known Solana deployment-time attack pattern requiring no special privileges — only the ability to submit a transaction.

---

### Recommendation

Restrict `initialize_pool` so that only a designated authority can call it. The standard Anchor pattern is to derive `pool_config` with a seed that includes the deployer's public key, or to add a `constraint = payer.key() == expected_authority` check validated against a value stored in `config_account` (the staking program's config PDA, which is already passed to this instruction and is controlled by governance).

---

### Proof of Concept

1. Deploy the integrity-pool program (or observe its deployment on-chain).
2. Before the Pyth team broadcasts `initialize_pool`, construct the same instruction with:
   - `payer` = attacker keypair
   - `pool_data` = attacker-owned account pre-funded with enough lamports
   - `reward_program_authority` = attacker pubkey
   - `y` = any valid value
3. Submit with a higher compute-unit price to win the race.
4. `pool_config` is now permanently initialized with attacker-controlled `pool_data` and `reward_program_authority`.
5. All subsequent `delegate`/`undelegate`/`advance` calls route through the attacker's `pool_data`, and only the attacker can call `update_delegation_fee` or other `reward_program_authority`-gated instructions. [7](#0-6)

  public async initializePool({
    rewardProgramAuthority,
    poolData,
    y,
  }: {
    rewardProgramAuthority: PublicKey;
    poolData: PublicKey;
    y: bigint;
  }) {
    const yAnchor = convertBigIntToBN(y);
    const instruction = await this.integrityPoolProgram.methods
      .initializePool(rewardProgramAuthority, yAnchor)
      .accounts({
        poolData,
        slashCustody: getStakeAccountCustodyAddress(poolData),
      })
      .instruction();

    return sendTransaction([instruction], this.connection, this.wallet);
  }
