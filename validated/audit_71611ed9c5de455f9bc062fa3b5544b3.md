Audit Report

## Title
Missing Duplicate-Hash Deduplication in `GetTransactionsProofProcess::execute` Causes Redundant Processing — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary
`GetTransactionsProofProcess::execute` performs no deduplication of incoming `tx_hashes`, unlike `GetBlocksProofProcess::execute` which explicitly bans peers sending duplicate hashes. An unprivileged peer can send 1000 copies of the same valid on-chain tx hash, causing 2000 redundant DB API calls, accumulation of 1000 duplicate `(tx, index)` entries in the block map, and CBMT proof construction over 1000 duplicate indices — with no ban or rate-limit consequence.

## Finding Description
`GetBlocksProofProcess::execute` collects hashes and immediately returns `StatusCode::MalformedProtocolMessage` (4xx → peer ban) if any duplicate is detected: [1](#0-0) 

`GetTransactionsProofProcess::execute` has no equivalent guard. After the size check (`> 1000` → reject), it directly partitions the raw iterator: [2](#0-1) 

With 1000 copies of the same valid tx hash:
- The `partition` closure calls `snapshot.get_transaction_info` 1000 times (DB call #1 × 1000).
- The `for tx_hash in found` loop calls `snapshot.get_transaction_with_info` 1000 times (DB call #2 × 1000).
- All 1000 entries share the same `block_hash` key, so the Vec accumulates 1000 `(tx, same_index)` entries.
- `CBMT::build_merkle_proof` is then called with 1000 duplicate indices.

The handler concludes with `reply_proof::<packed::SendTransactionsProofV1>`, returning `Status::ok()` (code 200). In `lib.rs` lines 81–86, `status.should_ban()` is called; since `should_ban` only returns `Some` for 4xx codes, the peer is never banned: [3](#0-2) [4](#0-3) 

## Impact Explanation
**Low (501–2000 points): Any other important performance improvements for CKB.**

The claimed "High: network congestion" severity is overstated. RocksDB's block cache means that after the first read of a given key, subsequent reads of the same key on the same snapshot are served from cache — so the 2000 function calls do not translate to 2000 disk reads. The actual disk I/O amplification is ~1×, not 1000×. The real overhead is CPU: 2000 RocksDB API calls (key hashing, cache lookup, deserialization), accumulation of 1000 duplicate Vec entries, and CBMT proof construction over 1000 duplicate indices. This is a genuine performance waste and a design inconsistency relative to `GetBlocksProofProcess`, but it does not rise to "saturating the RocksDB read path" or causing network-wide congestion. The light-client protocol server is also an optional component, further limiting blast radius.

## Likelihood Explanation
The light-client protocol endpoint is reachable by any unprivileged peer. The message is structurally valid (passes the `> 1000` size check with exactly 1000 hashes), requires no special knowledge beyond one valid on-chain tx hash, and the peer is never banned. The attack is trivially repeatable in a tight loop from a single connection. [5](#0-4) 

## Recommendation
Add a deduplication check in `GetTransactionsProofProcess::execute` immediately after the size check, mirroring the pattern already present in `GetBlocksProofProcess::execute`:

```rust
let tx_hashes: Vec<_> = self.message.tx_hashes().to_entity().into_iter().collect();
let mut uniq = HashSet::new();
if !tx_hashes.iter().all(|h| uniq.insert(h)) {
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate tx hash exists");
}
```

This returns a 4xx status, which `should_ban` converts into a 5-minute peer ban, consistent with the blocks-proof handler's behavior.

## Proof of Concept
1. Connect to a CKB node's light-client P2P endpoint.
2. Identify any valid on-chain transaction hash `H`.
3. Build a `GetTransactionsProof` message with `tx_hashes = [H] * 1000` and a valid `last_hash`.
4. Send the message repeatedly in a loop.
5. Observe: the server performs 2000 DB API calls per message, returns `SendTransactionsProofV1` (not a ban), and the peer connection remains open.
6. A unit test can be added in `util/light-client-protocol-server/src/tests/components/get_transactions_proof.rs`, constructing a `GetTransactionsProof` message with 1000 copies of the same hash and asserting that the peer is banned (`nc.banned(peer_index)` returns `true`).
