### Title
Excess `msg.value` in `requestHelper` Is Silently Absorbed by Pyth Instead of Being Refunded — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `requestHelper` function credits `accruedPythFeesInWei` with `msg.value - providerFee` rather than the fixed `pythFeeInWei`. When a caller sends more than the required fee, the entire surplus is permanently credited to Pyth's fee balance instead of being refunded to the caller. This is the direct analog of the external report's "incorrect arithmetic causes one party to pay more than it should" pattern.

---

### Finding Description

`requestHelper` splits the incoming payment between the provider and Pyth:

```solidity
// target_chains/ethereum/contracts/contracts/entropy/Entropy.sol  lines 233-239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

`getFeeV2` is defined as:

```solidity
// lines 760-765
function getFeeV2(address provider, uint32 gasLimit) public view override returns (uint128 feeAmount) {
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
``` [2](#0-1) 

So `requiredFee = providerFee + pythFeeInWei`. When `msg.value == requiredFee` the split is correct. But when `msg.value > requiredFee` (overpayment by `excess`):

- Provider receives: `providerFee` (correct)
- Pyth receives: `msg.value - providerFee = pythFeeInWei + excess` (**incorrect**)
- Caller loses: `excess` with no refund path

The correct Pyth credit should always be `_state.pythFeeInWei`, not `msg.value - providerFee`. The existing test in `Entropy.t.sol` explicitly confirms this behavior — an overpayment of 10 000 wei is absorbed into `accruedPythFees`:

```solidity
// Entropy.t.sol lines 682-699
requestWithFee(user2, pythFeeInWei + provider2FeeInWei + 10000, provider2, 42, false);
assertEq(random.getAccruedPythFees(), pythFeeInWei * 5 + 10000);
``` [3](#0-2) 

By contrast, `PythLazer.sol` correctly refunds excess:

```solidity
// lazer/contracts/evm/src/PythLazer.sol lines 74-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [4](#0-3) 

---

### Impact Explanation

Any caller of `requestV2` (or the legacy `request`) who sends `msg.value` even 1 wei above `getFeeV2()` permanently loses the surplus to the Pyth fee pool. Callers cannot recover it. Integrating contracts that add a small buffer to avoid `InsufficientFee` reverts, or that use a round-number `msg.value`, will silently donate funds to the protocol on every request.

---

### Likelihood Explanation

The `InsufficientFee` revert incentivises callers to send slightly more than the quoted fee as a safety margin, especially since the fee can change between the `getFeeV2` call and the transaction landing on-chain. Smart-contract integrations that compute fees off-chain or add a fixed buffer are particularly exposed. The entry path requires no privilege — any unprivileged Entropy user triggers it.

---

### Recommendation

Replace the variable `msg.value - providerFee` credit with the fixed `_state.pythFeeInWei`, and refund any excess to the caller:

```solidity
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += _state.pythFeeInWei;
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

This matches the pattern already used in `PythLazer.sol`.

---

### Proof of Concept

1. Deploy `Entropy` with `pythFeeInWei = 100` and a provider with `feeInWei = 200`.
2. Call `getFeeV2(provider, 0)` → returns `300`.
3. Call `requestV2` with `msg.value = 400` (100 wei overpayment).
4. Observe: `getAccruedPythFees()` returns `200` instead of `100`; the caller's 100 wei surplus is gone with no refund.
5. The existing test at `Entropy.t.sol:699` already asserts this exact behaviour (`pythFeeInWei * 5 + 10000`), confirming the root cause is present in production code. [5](#0-4)
