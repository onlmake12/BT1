### Title
Unprivileged Pre-initialization of Push Oracle PDA Permanently Blocks Price Feed Updates — (`target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

### Summary
An attacker can call `pyth_solana_receiver::post_update` directly, passing the push oracle's PDA (`seeds = [shard_id, feed_id]`) as `price_update_account` and their own key as `write_authority`. Because `PostUpdate` uses `init_if_needed` with no restriction on the account address, the account is created with `write_authority = attacker_key`. All subsequent CPIs from `update_price_feed` fail with `WrongWriteAuthority` because the push oracle signs as the PDA, not as the attacker.

### Finding Description
`pyth_solana_receiver::post_update` accepts any writable account as `price_update_account` and initializes it on first use with `write_authority` set to the caller-supplied signer. [1](#0-0) 

The push oracle's `update_price_feed` derives a PDA with seeds `[shard_id, feed_id]` and passes it as both `price_feed_account` and `write_authority` in the CPI, signing with the PDA's seeds. [2](#0-1) 

Because `PostUpdate` places no restriction on which address may be used as `price_update_account`, an attacker can target the exact PDA the push oracle will later use, initialize it first with their own `write_authority`, and permanently block the push oracle's CPI. [3](#0-2) 

The only recovery path is `reclaim_rent`, which requires the stored `write_authority` to sign — meaning only the attacker can close the account. [4](#0-3) 

### Impact Explanation
Any `(shard_id, feed_id)` pair whose PDA has not yet been written by the push oracle can be permanently frozen. After the attack, every call to `update_price_feed` for that feed reverts with `WrongWriteAuthority`. Price consumers reading that feed account will either see stale data (if the attacker posted a real price) or no data at all. The attacker can target high-value feeds (e.g., BTC/USD, ETH/USD) before they are first published on a new shard, causing a permanent denial-of-service for that feed.

### Likelihood Explanation
The attack requires only a valid Wormhole-verified VAA (freely obtainable from the Wormhole guardian network) and the ability to pay the update fee. No privileged access is needed. The window of vulnerability is any time before the push oracle first writes to a given `(shard_id, feed_id)` PDA — including at program deployment, shard expansion, or new feed onboarding.

### Recommendation
In `PostUpdate`, add a constraint that restricts `price_update_account` to accounts **not** derived from the push oracle's PDA namespace, or — more robustly — add a dedicated `init` instruction to the push oracle that creates and owns the PDA before any `post_update` call is possible. Alternatively, the push oracle should use a separate PDA derivation that includes the push oracle program ID as a seed, and the receiver should enforce that the `write_authority` matches the account's own address when the account is a PDA signer.

### Proof of Concept

```
1. Derive target PDA:
   pda = find_program_address([shard_id_bytes, feed_id_bytes], PUSH_ORACLE_PROGRAM_ID)

2. Attacker calls:
   pyth_solana_receiver::post_update(
       price_update_account = pda,   // no address restriction in PostUpdate
       write_authority      = attacker_keypair,
       encoded_vaa          = any valid verified VAA,
       ...
   )
   → account created, write_authority = attacker_pubkey

3. Legitimate relayer calls:
   pyth_push_oracle::update_price_feed(
       price_feed_account = pda,
       shard_id, feed_id, params, ...
   )
   → CPI to post_update with write_authority = pda (PDA signer)
   → constraint: pda.write_authority (attacker_pubkey) != Pubkey::default()
                 AND != pda_pubkey
   → FAILS: WrongWriteAuthority

4. All future update_price_feed calls for this (shard_id, feed_id) fail identically.
   reclaim_rent requires attacker_keypair to sign → attacker controls recovery.
```
