### Title
Sequential `subscriptionId` Counter Enables Re-Org Injection of Malicious Subscription, Allowing Fund Theft — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol` assigns subscription IDs using a monotonically increasing counter (`_state.subscriptionNumber++`). During a chain re-org, a malicious actor can race to create a new subscription that claims the same `subscriptionId` as a legitimate one. Any subsequent `addFunds()` calls from the original manager's application are credited to the attacker-controlled subscription, which the attacker can immediately drain via `withdrawFunds()`.

---

### Finding Description

`createSubscription()` assigns IDs sequentially:

```solidity
// Scheduler.sol L55
subscriptionId = _state.subscriptionNumber++;
``` [1](#0-0) 

The counter is a plain `uint256` stored in `SchedulerState`:

```solidity
// SchedulerState.sol L11
/// Monotonically increasing counter for subscription IDs
uint256 subscriptionNumber;
``` [2](#0-1) 

There is no hash-of-parameters check, no nonce tied to the creator, and no binding between the returned `subscriptionId` and the `SubscriptionParams` that were supplied. The subscription's manager is simply `msg.sender` at creation time:

```solidity
// Scheduler.sol L69
_state.subscriptionManager[subscriptionId] = msg.sender;
``` [3](#0-2) 

`addFunds()` has **no manager check** — it accepts ETH from any caller for any active subscription:

```solidity
// Scheduler.sol L602-617
function addFunds(uint256 subscriptionId) external payable override {
    ...
    if (!params.isActive) { revert ...; }
    ...
    status.balanceInWei += msg.value;
``` [4](#0-3) 

`withdrawFunds()` is restricted to the manager:

```solidity
// Scheduler.sol L630-633
function withdrawFunds(
    uint256 subscriptionId,
    uint256 amount
) external override onlyManager(subscriptionId) {
``` [5](#0-4) 

---

### Impact Explanation

**Direct fund theft:** A Manager's application that stores a `subscriptionId` off-chain and later calls `addFunds(subscriptionId)` will deposit ETH into the attacker-controlled subscription. The attacker (who is the `subscriptionManager` of the injected subscription) immediately calls `withdrawFunds()` to drain those funds.

**Price feed manipulation:** Any DeFi protocol that reads prices from a hardcoded `subscriptionId` via `getPricesUnsafe()` or `getPricesNoOlderThan()` will now consume prices for the attacker's chosen `priceIds` instead of the intended ones, enabling oracle manipulation attacks downstream. [6](#0-5) 

---

### Likelihood Explanation

- `createSubscription()` is fully permissionless — no role or whitelist required.
- Re-orgs occur naturally on EVM chains (especially L2s and sidechains with short finality windows). The attacker only needs to observe the mempool for a pending `createSubscription` transaction, wait for a re-org, and front-run the re-submission with their own malicious subscription.
- The attack is profitable: the attacker can lure large deposits by creating a subscription that looks identical to the legitimate one (same `priceIds`, same `updateCriteria`), then drain `addFunds` calls.
- Permanent subscriptions with `MAX_DEPOSIT_LIMIT` caps make this especially attractive as a high-value target. [7](#0-6) 

---

### Recommendation

Replace the sequential counter with a hash-based subscription ID that commits to both the creator and the subscription parameters:

```solidity
subscriptionId = uint256(keccak256(abi.encode(
    msg.sender,
    subscriptionParams,
    block.chainid,
    _state.subscriptionNumber++  // nonce to allow same params twice
)));
```

This ensures that even if a re-org occurs and the counter resets, an attacker cannot produce a subscription whose ID collides with the legitimate one, because the ID is bound to the creator address and the exact parameters.

---

### Proof of Concept

1. Alice calls `createSubscription({priceIds: [ETH/USD], ...})` with `1 ETH`. Transaction lands in block B. She receives `subscriptionId = 7`. Her application stores `7` off-chain.

2. A re-org reverts block B. `_state.subscriptionNumber` resets to `6`.

3. Attacker observes Alice's pending transaction in the mempool. Before Alice's transaction is re-included, the attacker submits `createSubscription({priceIds: [ETH/USD], whitelistEnabled: false, ...})` with the minimum balance. Attacker receives `subscriptionId = 7` and becomes its `subscriptionManager`.

4. Alice's transaction is re-included and receives `subscriptionId = 8` (counter is now `7` after attacker's tx). Alice's application still references `7`.

5. Alice's application calls `addFunds(7){value: 5 ETH}`. Because `addFunds` has no manager check and subscription `7` is active, the 5 ETH is credited to the attacker's subscription.

6. Attacker calls `withdrawFunds(7, 5 ETH)` and steals Alice's funds. [8](#0-7) [9](#0-8)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
