Audit Report

## Title
Unbounded DB Read Amplification via Unauthenticated `GetTransactionsProof` with No Per-Peer Rate Limiting — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary
`LightClientProtocol` contains no rate limiter field and performs no rate-limit check before dispatching `GetTransactionsProof`. Any peer can send messages containing up to 1000 tx_hashes at arbitrary speed, each unconditionally triggering up to 1000 synchronous `get_transaction_info` DB reads plus full response construction and transmission. The `Relayer` protocol has an explicit per-peer rate limiter capped at 30 req/s; the light client protocol has none.

## Finding Description
`LightClientProtocol` struct contains only `pub shared: Shared` — no `rate_limiter` field exists. [1](#0-0) 

`try_process` dispatches `GetTransactionsProof` directly to `GetTransactionsProofProcess::execute()` with zero rate-limit check. [2](#0-1) 

Inside `execute()`, the only guards are an empty-check and a `> GET_TRANSACTIONS_PROOF_LIMIT` check. A message with exactly 1000 hashes passes both. The handler then calls `snapshot.get_transaction_info(tx_hash)` for every hash in a `partition` loop. [3](#0-2) 

`GET_TRANSACTIONS_PROOF_LIMIT` is 1000. [4](#0-3) 

By contrast, `Relayer` has an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` field capped at 30 req/s. [5](#0-4) 

This rate limiter is checked before any dispatch in `Relayer::try_process`. [6](#0-5) 

## Impact Explanation
Each request costs the server up to 1000 synchronous DB reads plus response construction and transmission. A single attacker sending requests at high rate imposes unbounded DB read IOPS and outbound bandwidth. This can saturate disk I/O and network egress on a full node running the light client server, causing denial of service to legitimate light clients and degrading full-node operation. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
The attack requires only a valid main-chain block hash (publicly available via RPC) and the ability to open a P2P connection. No key, stake, or privileged role is needed. The light client protocol server is the intended production interface for light clients. The attack is trivially scriptable and repeatable.

## Recommendation
Add a per-peer rate limiter to `LightClientProtocol`, mirroring the pattern in `Relayer`:
- Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`.
- In `try_process`, check `rate_limiter.check_key(&(peer_index, message.item_id()))` before dispatching, returning `StatusCode::TooManyRequests` on failure.
- Consider lowering `GET_TRANSACTIONS_PROOF_LIMIT` (e.g., to 100) to reduce per-request DB amplification.

## Proof of Concept
```
1. Obtain any valid tip block hash H from the CKB chain (public RPC).
2. Connect to a CKB full node with light client protocol enabled.
3. In a tight loop:
   a. Build GetTransactionsProof { last_hash: H, tx_hashes: [rand_32_bytes(); 1000] }
   b. Send over the P2P connection.
4. Observe: server performs 1000 get_transaction_info DB reads per message,
   constructs and sends a SendTransactionsProofV1 response per message,
   no TooManyRequests rejection is ever returned.
5. Measure DB read IOPS and outbound bandwidth — both scale linearly with
   request rate, unbounded by any server-side control.
```
