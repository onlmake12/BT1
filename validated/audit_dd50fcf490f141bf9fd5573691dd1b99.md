### Title
Missing IBD Guard and Rate Limiter in `BlockFilter::received` Allows Cheap DB-Read Flooding During Initial Block Download — (`sync/src/filter/mod.rs`)

### Summary
`BlockFilter::received` processes all inbound filter messages unconditionally during IBD. Unlike `Relayer::received` (which returns immediately during IBD) and `GetHeadersProcess::execute` (which ignores the request during IBD), the Filter protocol handler has no IBD guard and no per-peer rate limiter. An unprivileged remote peer can flood `GetBlockFilterCheckPoints` messages, each triggering up to 4 000 RocksDB reads, competing with IBD block-download writes at the worst possible time.

### Finding Description

**Missing IBD guard in `BlockFilter::received`**

`Relayer::received` returns immediately during IBD: [1](#0-0) 

`GetHeadersProcess::execute` also returns early during IBD: [2](#0-1) 

`BlockFilter::received` has no such guard — it parses the message and calls `self.process()` unconditionally: [3](#0-2) 

**Missing rate limiter in `BlockFilter`**

`Relayer` initialises a per-peer, per-message-type rate limiter (30 req/s hard cap): [4](#0-3) 

`BlockFilter` has no equivalent. The `try_process` dispatch is bare: [5](#0-4) 

**DB reads per `GetBlockFilterCheckPoints` request**

`BATCH_SIZE = 2000` and `CHECK_POINT_INTERVAL = 2000`. Each loop iteration calls `get_block_hash()` then `get_block_filter_hash()` — two RocksDB point-reads — for a theoretical maximum of 4 000 reads per message: [6](#0-5) 

The same pattern applies to `GetBlockFilterHashes` (`BATCH_SIZE = 2000`, 2 reads/iteration = up to 4 000 reads): [7](#0-6) 

And `GetBlockFilters` (`BATCH_SIZE = 1000`, 2 reads/iteration = up to 2 000 reads): [8](#0-7) 

**Practical read count during IBD**

The guard `if latest >= start_number` limits reads to blocks for which filter data has already been built. With `start_number = 0` the condition is always satisfied (`latest` is a `u64`). The background filter-builder service runs continuously during IBD, so `latest` grows over time, and the number of reads per request grows proportionally. Even at genesis, `get_block_hash(0)` and `get_block_filter_hash(genesis_hash)` are executed before the loop breaks. [9](#0-8) 

### Impact Explanation
During IBD — the most I/O-intensive node state — an attacker with a single inbound connection on the Filter protocol can issue a continuous stream of `GetBlockFilterCheckPoints{start_number:0}` messages. Each message triggers synchronous RocksDB reads that share the same column families and block cache as the IBD block-write path. With no rate limiter and no IBD guard, the attacker can saturate RocksDB read I/O, degrading IBD block-processing throughput. The same three filter message types are all affected, multiplying the available attack surface.

### Likelihood Explanation
The Filter protocol is a standard supported protocol; any peer can connect to it without any privilege. The attack requires only a TCP connection and a trivially small message. The cost to the attacker is negligible; the cost to the victim is measurable IBD slowdown. The missing guard is a straightforward oversight relative to the explicit guards present in `Relayer::received` and `GetHeadersProcess::execute`.

### Recommendation
1. Add an IBD guard at the top of `BlockFilter::received` (or `try_process`) mirroring `Relayer::received`:
   ```rust
   if self.shared.active_chain().is_initial_block_download() {
       return;
   }
   ```
2. Add a per-peer rate limiter to `BlockFilter` analogous to the one in `Relayer::new`.

### Proof of Concept
```
1. Start a CKB node from genesis (IBD state).
2. Connect a second process to the node's Filter protocol port.
3. In a tight loop, send GetBlockFilterCheckPoints { start_number: 0 }.
4. Measure IBD block-processing throughput (blocks/s) vs. a baseline run
   without the filter flood.
5. Expected: measurable throughput degradation due to RocksDB read
   contention; IBD completion time increases proportionally to flood rate.
```

        filter: &'a BlockFilter,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
    ) -> Self {
        Self {
            message,
            nc,
            filter,
            peer,
        }
    }

    pub async fn execute(self) -> Status {
        let active_chain = self.filter.shared.active_chain();
        let start_number: BlockNumber = self.message.to_entity().start_number().into();
        let latest: BlockNumber = active_chain.get_latest_built_filter_block_number();

        if latest >= start_number {
            let mut block_hashes = Vec::new();
            let mut filters = Vec::new();
            let mut current_content_size = 0;
            current_content_size += 8; // Size of start_number
            current_content_size += 4 * 2; // Size of the header field `full-size` of `block_hash` and `block_filter`
            let mut block_number = start_number;
            for _ in 0..BATCH_SIZE {
                if let Some(block_hash) = active_chain.get_block_hash(block_number) {
                    if let Some(block_filter) = active_chain.get_block_filter(&block_hash) {
                        if current_content_size
                            + block_hash.as_slice().len()
                            + 4
                            + block_filter.as_slice().len()
                            + 4
                            >= (1.8 * 1024.0 * 1024.0) as usize
                        {
                            // Break if the encoded size of `block_hash` + `block_filter` + `start_number` + molecule header increase reaches 1.8MB, to avoid frame size too large
                            break;
                        }
                        current_content_size +=
                            block_hash.as_slice().len() + block_filter.as_slice().len() + 4;
                        block_hashes.push(block_hash);
                        filters.push(block_filter);
                    } else {
                        break;
                    }
                } else {
                    break;
                }
                let Some(next_block_number) = block_number.checked_add(1) else {
                    break;
                };
                block_number = next_block_number;
            }
