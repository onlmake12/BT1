### Title
Missing `callbackGasLimit` Enforcement in `executeCallback` Allows Fee Collection Without Callback Delivery — (`Echo.sol`)

---

### Summary

In `Echo.sol`, a user pays a fee proportional to their specified `callbackGasLimit` when calling `requestPriceUpdatesWithCallback`. However, `executeCallback` clears the request and credits the provider **before** invoking the callback, with no check that `gasleft() >= req.callbackGasLimit`. A relayer (or anyone after the exclusivity window) can call `executeCallback` with insufficient gas, causing the callback to fail silently while the provider collects the full fee and the user has no retry path.

---

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, they specify `callbackGasLimit` and pay a fee that includes a per-gas component:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
...
req.callbackGasLimit = callbackGasLimit;
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [1](#0-0) 

In `executeCallback`, the execution order is:

1. Provider is credited (`accruedFeesInWei += (req.fee + msg.value) - pythFee`)
2. Request is cleared (`clearRequest(sequenceNumber)`)
3. Callback is attempted with `gas: req.callbackGasLimit` — **no pre-check on `gasleft()`** [2](#0-1) 

If the callback fails (e.g., because the caller provided insufficient total gas), the `try/catch` silently emits `PriceUpdateCallbackFailed`. The request is already gone and the provider is already paid. The user has no retry mechanism. [3](#0-2) 

**Contrast with `Entropy.sol#revealWithCallback`**, which explicitly guards against this:

```solidity
uint256 startingGas = gasleft();
// ... excessivelySafeCall with gasLimit ...
} else {
    // Calling context did not have enough gas to run the callback
    revert EntropyErrors.InsufficientGas();
}
``` [4](#0-3) 

Entropy also preserves the request in a `CALLBACK_FAILED` state on failure, allowing retry. Echo has no equivalent. [5](#0-4) 

The `InsufficientGas` error is defined in `EntropyErrors.sol` precisely for this scenario: [6](#0-5) 

Echo.sol has no analogous error or guard.

---

### Impact Explanation

A user pays a fee that includes a `callbackGasLimit`-proportional component, expecting their `_echoCallback` to execute with at least `callbackGasLimit` gas. Because the request is cleared and the provider is credited before the callback, a failed callback results in:

- **User's funds permanently transferred to the provider** with no recourse
- **User's callback never executes** — the price update the user paid for is not delivered to their contract
- **No retry path** — unlike Entropy's `CALLBACK_FAILED` state, Echo has no mechanism to re-attempt

This is a direct financial loss to the user: they paid for a service (callback execution with guaranteed gas) that was not delivered.

---

### Likelihood Explanation

`executeCallback` is callable by **anyone** after the `exclusivityPeriodSeconds` window:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(providerToCredit == req.provider, "Only assigned provider during exclusivity period");
}
``` [7](#0-6) 

After this window, any relayer — including a malicious one — can call `executeCallback` with a gas amount sufficient to clear the request and credit themselves, but insufficient to run the user's callback. The EVM's 63/64 rule (at most 63/64ths of `gasleft()` is forwarded to a sub-call) makes it straightforward to craft a transaction where `gasleft()` is just above the overhead cost but below `callbackGasLimit`. The test suite itself acknowledges this scenario: [8](#0-7) 

---

### Recommendation

Before invoking the callback, add a gas sufficiency check analogous to `Entropy.sol`:

```solidity
// Ensure the calling context has enough gas to forward callbackGasLimit to the callback.
// Due to EVM's 63/64 rule, we need (64/63) * callbackGasLimit available.
require(
    gasleft() * 63 / 64 >= req.callbackGasLimit,
    "Insufficient gas to execute callback"
);
```

Additionally, adopt Entropy's pattern of **not clearing the request on callback failure** — instead, move to a `CALLBACK_FAILED` state that allows retry. This ensures the user retains recourse if the callback fails for any reason.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 1_000_000`, paying the corresponding fee. `req.fee` is stored.
2. The exclusivity period expires.
3. A malicious relayer calls `executeCallback{gas: 200_000}(...)`. This is enough gas to:
   - Parse price feeds
   - Credit the provider: `accruedFeesInWei += req.fee + msg.value - pythFee` ✓
   - Clear the request: `clearRequest(sequenceNumber)` ✓
   - Attempt the callback with `gas: 1_000_000` — but `gasleft()` at this point is ~50k, so the EVM forwards only ~49k gas, far below `callbackGasLimit`
4. The callback reverts out-of-gas. The `catch` block emits `PriceUpdateCallbackFailed`.
5. The request is gone. The provider (relayer) has collected `req.fee`. The user's `_echoCallback` never ran.
