Audit Report

## Title
Stale Descendant Statistics in `PoolMap::remove_entry_and_descendants` Corrupt Eviction Ordering - (File: tx-pool/src/component/pool_map.rs)

## Summary

`remove_entry_and_descendants` pre-removes all link graph entries for the entire removed batch before delegating to `remove_entry` for each member. Because `update_ancestors_index_key` relies on the live link graph to find ancestors outside the removed batch, it finds none and never calls `sub_descendant_weight` on them. Their `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` fields remain permanently inflated, and the stored `evict_key` — which encodes `fee_rate` derived from those stale fields — is never refreshed, corrupting the eviction order used when the pool is full.

## Finding Description

**Root cause — link teardown precedes ancestor update:**

`remove_entry_and_descendants` (lines 252–265) first collects the target and all its descendants, then calls `remove_entry_links` for every member in the batch, and only then calls `remove_entry` for each:

```rust
// pool_map.rs lines 252-265
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ← all links torn down for every member
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← links already gone
        .collect()
}
```

The comment reveals the intent: pre-removing links prevents `update_descendants_index_key` (which updates children's ancestor stats) from running inside `remove_entry` for entries that are themselves being removed. However, it also prevents `update_ancestors_index_key` (line 242 of `remove_entry`) from working correctly for ancestors that are **not** being removed.

`remove_entry_links` (lines 418–430) removes the entry from `self.links` entirely and severs all parent/child back-references. After the first loop, the link graph contains no record of any entry in `removed_ids`.

`remove_entry` (line 242) then calls `update_ancestors_index_key`, which queries `self.links.calc_ancestors(id)`. Because the links are already gone, `calc_ancestors` returns an empty set. The `modify_by_id` closure that calls `sub_descendant_weight` and refreshes `evict_key` (lines 437–443) is never executed for any ancestor that lies outside the removed batch.

**Stale fields in `EvictKey`:**

`EvictKey` is constructed from `TxEntry` (entry.rs lines 234–247) using `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`. Both the primary sort field (`fee_rate`) and the tiebreaker (`descendants_count`) can be wrong.

Consider chain **A (fee=100) → B (fee=1000) → C (fee=1000)**. After `remove_entry_and_descendants(B)`:
- A's `descendants_fee` stays at 2100 (should be 100).
- A's `descendants_feerate` stays at 7 shannons/byte (should be 1).
- A's `EvictKey.fee_rate` stays at `max(7, 1) = 7` (should be 1).
- A's `EvictKey.descendants_count` stays at 3 (should be 1).

The `evict_key` field is a stored, indexed field in `MultiIndexPoolEntryMap` (line 55 of pool_map.rs). It is never recomputed after the fact; no code path corrects it.

**Eviction path:**

`limit_size` (pool.rs lines 292–329) calls `next_evict_entry` → `iter_by_evict_key()` (pool_map.rs lines 380–385) to select the next victim. With a stale high `fee_rate`, A ranks as harder to evict than it actually is, so a genuinely higher-fee transaction is evicted instead.

**Existing test gap:**

`test_remove_entry_and_descendants` (score_key.rs lines 170–230) only asserts that B and C are absent and that A's `calc_descendants` set is empty. It never checks A's `descendants_count`, `descendants_fee`, or `evict_key` after the call, so the corruption goes undetected.

## Impact Explanation

This is a **Medium** impact: suboptimal implementation of the CKB tx-pool state storage mechanism. The eviction comparator permanently carries inflated descendant statistics for ancestors of any removed subtree. When the pool reaches capacity, `limit_size` selects the wrong eviction victim — a low-fee ancestor with stale high `EvictKey.fee_rate` survives rounds it should lose, while a legitimately higher-fee transaction is evicted instead. The corruption persists until the ancestor itself is eventually removed; no self-healing path exists.

## Likelihood Explanation

Any code path that calls `remove_entry_and_descendants` on an entry that has a pool-resident ancestor triggers the bug. This includes:

- `resolve_conflict` (pool_map.rs lines 305–332): fires on every committed block that double-spends a mid-chain pool entry — a routine, unprivileged event.
- `process_rbf` (process.rs): fires on any RBF replacement targeting a mid-chain entry.
- `limit_size` (pool.rs lines 292–329): fires whenever the pool exceeds `max_tx_pool_size`.
- `check_and_record_ancestors` (pool_map.rs lines 588–640): fires when ancestor-count limits are exceeded.

An unprivileged submitter needs only to submit a parent–child chain (A → B → C) and then submit a conflicting transaction that causes B and C to be evicted. No special privilege is required.

## Recommendation

Snapshot each entry's ancestor set **before** any links are torn down, then apply the weight subtraction using the snapshot:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Update ancestor stats BEFORE removing links so outside ancestors are reachable.
    for id in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(id).map(|e| e.inner.clone()) {
            self.update_ancestors_index_key(&entry, EntryOp::Remove);
        }
    }

    // Now safe to tear down links; update_ancestors_index_key won't be called again
    // inside remove_entry because links are gone (empty ancestor set — no-op).
    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

A regression test should assert that after `remove_entry_and_descendants(B)` on chain A→B→C, A's `descendants_count` equals 1, A's `descendants_fee` equals A's own fee, and A's `evict_key` reflects those corrected values.

## Proof of Concept

**Minimal unit test (extend `tx-pool/src/component/tests/score_key.rs`):**

1. Build chain tx1 (fee=100) → tx2 (fee=1000) → tx3 (fee=1000) with proper `OutPoint` linkage.
2. `map.add_proposed(tx1); map.add_proposed(tx2); map.add_proposed(tx3);`
3. Assert `map.get(&tx1_id).unwrap().descendants_count == 3`.
4. `map.remove_entry_and_descendants(&tx2_id);`
5. Assert `map.get(&tx1_id).unwrap().descendants_count == 1` — **this assertion currently fails**, confirming the bug.
6. Assert `map.get(&tx1_id).unwrap().descendants_fee == Capacity::shannons(100)` — **also fails**.
7. Assert `map.next_evict_entry(Status::Proposed) == Some(tx1_id)` when a lower-fee tx with no descendants is also present — **fails** because tx1's stale high `evict_key` ranks it as harder to evict.
