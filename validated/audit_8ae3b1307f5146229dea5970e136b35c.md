Audit Report

## Title
VerifyQueue Saturation via Zero-Fee Transactions Before Fee Check Enables P2P Transaction Relay DoS — (`tx-pool/src/process.rs`, `tx-pool/src/component/verify_queue.rs`)

## Summary

The `VerifyQueue` admits transactions based solely on `non_contextual_verify`, which cannot detect zero-fee or below-minimum-fee-rate transactions without chain state. The fee check (`check_tx_fee`) is deferred to worker threads that dequeue and call `pre_check` after admission. An unprivileged P2P peer can continuously relay structurally valid, zero-fee transactions to saturate the 256 MB queue, causing all subsequent legitimate P2P transactions to be rejected with `Reject::Full`. The attack is repeatable: workers drain the queue quickly (unresolvable inputs fail fast), but the attacker can immediately refill it.

## Finding Description

The P2P relay path is: `submit_remote_tx` → `resumeble_process_tx_and_notify_full_reject` → `resumeble_process_tx` → `enqueue_verify_queue`. [1](#0-0) 

The only gate before queuing is `non_contextual_verify`, which checks only structure-level properties: version, size ≤ 512 KB (`TRANSACTION_SIZE_LIMIT`), non-empty inputs/outputs, duplicate deps, and cellbase rejection. [2](#0-1) 

It performs **no fee check**. The fee check lives in `check_tx_fee`, called only from `pre_check`, which is invoked by verify workers **after** a transaction is already dequeued from the queue. [3](#0-2) [4](#0-3) 

`check_tx_fee` requires a `ResolvedTransaction` (`rtx`) to compute actual fee from input capacities — information unavailable without chain state and therefore impossible to check at the `non_contextual_verify` stage.

The `VerifyQueue` is bounded at 256 MB with no per-peer limit: [5](#0-4) 

`add_tx` enforces only the total-size cap and a duplicate check — no fee gate, no per-peer accounting: [6](#0-5) 

When the queue is full, any new transaction (including legitimate ones) is rejected with `Reject::Full`: [7](#0-6) 

While `remove_txs_by_peer` exists and is called on peer disconnect, it does not prevent the attack — the attacker reconnects and refills immediately. [8](#0-7) 

## Impact Explanation

This matches the allowed CKB bounty impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

When the 256 MB verify queue is saturated, every subsequent `submit_remote_tx` call from any peer returns `Reject::Full`. The relayer is notified and marks the transaction as unknown in its filter, blocking the entire P2P transaction relay pipeline for all peers connected to the victim node. Legitimate users cannot get their transactions into the pool for verification or eventual block inclusion. The attack costs only bandwidth — no CKB tokens are spent, and the default `min_fee_rate` of 1,000 shannons/KB is entirely bypassed at the admission stage. [9](#0-8) 

## Likelihood Explanation

Any unprivileged P2P peer can trigger this. The attacker needs only to:
1. Connect to the victim node as a standard P2P peer.
2. Craft transactions that pass `non_contextual_verify`: valid version, at least one input and one output, no duplicate deps, size ≤ 512 KB.
3. Set output capacity equal to input capacity (zero fee) — undetectable without chain state.
4. Use arbitrary non-existent input outpoints — these also pass non-contextual checks.

With transactions near the 512 KB size limit, only ~500 transactions are needed to fill the 256 MB queue. The attack is repeatable: workers drain the queue quickly (each transaction fails at `pre_check` due to unresolvable inputs), but the attacker immediately re-sends to refill. No special privileges, no CKB tokens, no cryptographic material beyond a valid transaction structure are required.

## Recommendation

1. **Per-peer slot cap in `enqueue_verify_queue`**: Track per-peer transaction count/size in the `VerifyQueue` and reject new entries from a peer that exceeds a threshold (analogous to `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` in the orphan pool).
2. **Stateless fee lower-bound pre-screen**: Before admission, if the transaction's total declared output capacity is ≥ total declared input capacity (computable from the transaction's own fields without chain state), reject it as provably zero-fee.
3. **Peer banning on repeated `Reject::Full` contributions**: Track which peer's transactions are draining the queue and apply a temporary ban after repeated offenses.

## Proof of Concept

1. Connect to a CKB node as a P2P peer.
2. Construct ~500 transactions, each ~512 KB, with:
   - Valid structure (non-empty inputs/outputs, correct version, no duplicate deps).
   - Arbitrary non-existent input outpoints.
   - Output capacity equal to a plausible input capacity (zero fee).
3. Relay all transactions via the P2P relay protocol, triggering `submit_remote_tx` on the victim node.
4. Each transaction passes `non_contextual_verify` and is admitted to the `VerifyQueue` until `total_tx_size` reaches 256 MB.
5. Observe that any subsequent `submit_remote_tx` call from any peer returns `Reject::Full`.
6. Workers drain the queue (each transaction fails at `pre_check` due to unresolvable inputs or zero fee); attacker immediately re-sends to refill.
7. Legitimate transactions from honest peers are continuously rejected for the duration of the attack.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
