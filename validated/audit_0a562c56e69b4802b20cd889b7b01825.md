## Verification of Each Claim

**Claim 1: `parse_and_verify` does NOT consume the VAA.**

Confirmed. [1](#0-0) 

`parse_and_verify` only verifies guardian signatures and returns a `VAA` object. It never calls `consumed_vaas::consume`. The `consume` function is a separate, explicitly-called public function. [2](#0-1) 

**Claim 2: `governance::verify_vaa` does NOT add to `consumed_vaas`.**

Confirmed. `verify_vaa` checks the emitter data source, extracts `digest`/`sequence`/`payload`, and returns a `WormholeVAAVerificationReceipt`. It never touches `consumed_vaas`. [3](#0-2) 

**Claim 3: The only replay guard is sequence ordering.**

Confirmed. `execute_governance_instruction` only checks `sequence > state::get_last_executed_governance_sequence(pyth_state)`. [4](#0-3) 

**Claim 4: `set_governance_data_source` resets `last_executed_governance_sequence` to an arbitrary value.**

Confirmed. It calls `state::set_last_executed_governance_sequence(latest_only, pyth_state, initial_sequence)` where `initial_sequence` is attacker-relayable governance payload data. [5](#0-4) 

**Claim 5: `WormholeVAAVerificationReceipt` is a hot potato (no abilities).**

Confirmed. The struct has no `store`, `key`, `copy`, or `drop` abilities, so it must be consumed in the same PTB it is created. However, within a single PTB, multiple hot potatoes can coexist. [6](#0-5) 

**Claim 6: `VAA` is also a hot potato — but two can be created in one PTB.**

Confirmed. The `VAA` struct has no abilities. [7](#0-6)  Since `parse_and_verify` does not mark the VAA as consumed, calling it twice with the same bytes in a single Sui PTB produces two independent `VAA` objects, both valid.

## Attack Path (Concrete, Single PTB)

The attacker needs two legitimate governance VAAs already on-chain:
- **VAA_S** (sequence `S`): any governance action to replay
- **VAA_T** (sequence `T > S`): a `set_governance_data_source` action with `initial_sequence < S`

Within one Programmable Transaction Block:

1. `parse_and_verify(wormhole_state, bytes_S, clock)` → `vaa_S_1`
2. `parse_and_verify(wormhole_state, bytes_S, clock)` → `vaa_S_2` *(no consumed check, succeeds)*
3. `governance::verify_vaa(pyth_state, vaa_S_1)` → `receipt_1`
4. `governance::verify_vaa(pyth_state, vaa_S_2)` → `receipt_2`
5. `execute_governance_instruction(pyth_state, receipt_1)` → action executes; `last_seq = S`
6. `parse_and_verify(wormhole_state, bytes_T, clock)` → `vaa_T`
7. `governance::verify_vaa(pyth_state, vaa_T)` → `receipt_T`
8. `execute_governance_instruction(pyth_state, receipt_T)` → `set_governance_data_source` executes; `last_seq = initial_sequence` (< S)
9. `execute_governance_instruction(pyth_state, receipt_2)` → check: `S > initial_sequence` ✓ → **action executes a second time**

## Root Cause

The Pyth governance system on Sui has no VAA-hash-based replay protection for governance messages. The `consumed_vaas` field exists in `State` [8](#0-7)  but is never used in the governance execution path. The sole guard — a monotonically increasing sequence number — is defeatable because `set_governance_data_source` can reset it to an arbitrary lower value. [9](#0-8)

### Title
Governance VAA Replay via Sequence Reset in `set_governance_data_source` — (`target_chains/sui/contracts/sources/governance/governance.move`)

### Summary
`governance::verify_vaa` and `wormhole::vaa::parse_and_verify` both lack VAA-hash consumption. The only replay guard is a sequence-number monotonicity check. Because `set_governance_data_source` can reset `last_executed_governance_sequence` to an arbitrary `initial_sequence`, an attacker who holds two `WormholeVAAVerificationReceipt` objects derived from the same VAA bytes (obtainable in a single PTB) can replay a governance action after the sequence is reset.

### Finding Description
`wormhole::vaa::parse_and_verify` verifies guardian signatures but does not call `consumed_vaas::consume`, so the same raw VAA bytes can be parsed twice in one PTB to yield two independent `VAA` hot-potato objects. `governance::verify_vaa` converts each into a `WormholeVAAVerificationReceipt` without recording the digest in any consumed set. `execute_governance_instruction` enforces only `sequence > last_executed_governance_sequence`. When a `set_governance_data_source` governance VAA is executed, it unconditionally overwrites `last_executed_governance_sequence` with the payload's `initial_sequence` field, which can be any value, including one below a previously executed VAA's sequence. An attacker who pre-obtains two receipts for VAA_S, executes the first, then triggers the sequence reset via VAA_T, can execute the second receipt and replay the governance action.

### Impact Explanation
A governance action (e.g., `set_data_sources`, `set_update_fee`, `set_stale_price_threshold`, `set_fee_recipient`, or `contract_upgrade`) executes twice. This deviates execution from the single voted outcome, constituting governance voting result manipulation. Depending on the replayed action, consequences include unauthorized fee changes, unauthorized data-source changes, or double-application of a contract upgrade.

### Likelihood Explanation
The precondition is that governance has ever issued a `set_governance_data_source` VAA with `initial_sequence` lower than some prior governance VAA's sequence. This is a realistic operational scenario (e.g., migrating to a new governance emitter and resetting the counter). Once such a VAA exists on-chain, any unprivileged relayer can execute the attack in a single PTB with no special privileges.

### Recommendation
In `governance::verify_vaa`, after extracting the digest, call `vaa::consume` against the Pyth state's `consumed_vaas` (which already exists in `State`). This records the VAA hash and prevents the same VAA from producing a second receipt regardless of sequence resets:

```move
// After: let digest = vaa::digest(&verified_vaa);
vaa::consume(state::borrow_mut_consumed_vaas(&latest_only, pyth_state), &verified_vaa);
```

The same fix should be applied in `contract_upgrade::authorize_upgrade`.

### Proof of Concept
Unit test outline:
1. Initialize Pyth state with governance data source G, `last_executed_governance_sequence = 0`.
2. Craft VAA_S (sequence 5, action = `set_update_fee`).
3. Craft VAA_T (sequence 6, action = `set_governance_data_source`, `initial_sequence = 3`).
4. In a single PTB: call `parse_and_verify` twice on VAA_S bytes → `vaa_S_1`, `vaa_S_2`; call `verify_vaa` twice → `r1`, `r2`; execute `r1` (fee changes, `last_seq = 5`); parse/verify/execute VAA_T (`last_seq = 3`); execute `r2` (fee changes again, `last_seq = 5`).
5. Assert the fee was applied twice and `last_executed_governance_sequence = 5`.

    public fun verify_vaa(
        pyth_state: &State,
        verified_vaa: VAA,
    ): WormholeVAAVerificationReceipt {
        state::assert_latest_only(pyth_state);

        let vaa_data_source = pyth::data_source::new((vaa::emitter_chain(&verified_vaa) as u64), vaa::emitter_address(&verified_vaa));

        // The emitter chain and address must correspond to the Pyth governance emitter chain and contract.
        assert!(
            pyth::state::is_valid_governance_data_source(pyth_state, vaa_data_source),
            E_INVALID_GOVERNANCE_DATA_SOURCE
        );

        let digest = vaa::digest(&verified_vaa);

        let sequence = vaa::sequence(&verified_vaa);

        let payload = vaa::take_payload(verified_vaa);

        WormholeVAAVerificationReceipt { payload, digest, sequence }
    }
