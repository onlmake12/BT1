### Title
`verifyUpdate` Lacks On-Chain Replay Protection, Allowing Stale Signed Price Updates to Be Re-submitted Indefinitely - (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` performs ECDSA signature verification and trusted-signer validation but never records which update payloads have already been processed. Any valid signed Lazer price update observed on-chain can be re-submitted by an unprivileged caller at any future time (while the signer certificate is still valid), causing consuming contracts to accept arbitrarily stale price data as freshly verified.

---

### Finding Description

`verifyUpdate` in `PythLazer.sol` only checks:

1. That the fee is paid.
2. That the EVM format magic bytes are correct.
3. That the ECDSA signature over the payload is valid.
4. That the recovered signer address is in the trusted-signer list and has not expired. [1](#0-0) 

There is no mapping, set, or any other structure that records previously accepted update hashes. The contract never compares the payload's embedded timestamp against `block.timestamp`. As a result, the exact same `update` bytes — with the exact same signature — can be passed to `verifyUpdate` an unlimited number of times and will succeed every time, as long as the signer's `expiresAt` has not passed. [2](#0-1) 

The same structural gap exists in the Aptos contract (`verify_message`) and the Sui contract (`parse_and_verify_le_ecdsa_update_v2` / `verify_le_ecdsa_message`): neither tracks consumed messages nor enforces payload-timestamp freshness relative to the chain clock. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

An attacker who observes a valid Lazer price update on-chain (or intercepts it off-chain) can replay it at any later time while the signer certificate is still valid. Any consuming contract that relies solely on `verifyUpdate` returning without reverting — without independently enforcing payload-timestamp freshness — will accept the stale price as current. This enables:

- **Stale-price injection**: replaying an update from minutes or hours ago when the real price has moved significantly, triggering incorrect liquidations or enabling profitable arbitrage against the protocol.
- **Repeated state manipulation**: submitting the same update multiple times within a single block or across blocks to repeatedly trigger price-dependent logic (e.g., repeated settlement, repeated fee accrual, repeated oracle callbacks).

The Lazer payload format does embed a `timestamp` field, but `verifyUpdate` never reads it. The contract's own documentation and the `PythLazerLib.parsePayloadHeader` helper expose this timestamp to consuming contracts, but there is no enforcement at the verification layer. [5](#0-4) 

---

### Likelihood Explanation

The entry path requires no privileged access. Any address can call `verifyUpdate` with `msg.value >= 1 wei`. Valid update bytes are publicly observable on-chain from any prior transaction that called `verifyUpdate`. The attacker's only cost is the nominal `verification_fee` (currently 1 wei) per replay. Signer certificates are set to expire far in the future (e.g., the test fixture uses `3000000000000000` as `expiresAt`), giving a very long replay window. [6](#0-5) 

---

### Recommendation

Add on-chain replay protection to `verifyUpdate` (and the equivalent Aptos/Sui entry points) using one or both of the following approaches:

1. **Consumed-update tracking**: maintain a `mapping(bytes32 => bool) private _usedUpdates` keyed on `keccak256(update)` (or `keccak256(payload)`). Revert if the hash has already been seen; set it to `true` after the first successful verification.

2. **Payload-timestamp freshness check**: parse the `uint64 timestamp` from the payload header and require `block.timestamp` to be within an acceptable staleness window (e.g., ±60 seconds). This mirrors how Wormhole VAA-based Pyth price feeds enforce freshness. [7](#0-6) 

---

### Proof of Concept

The absence of replay protection is directly readable from the contract source. The `verifyUpdate` function contains no storage write, no mapping lookup, and no timestamp comparison against `block.timestamp`. The test in `PythLazer.t.sol` confirms that the same `update` bytes can be submitted twice by Alice and both calls succeed:

```solidity
// Alice provides appropriate fee — succeeds
vm.prank(alice);
pythLazer.verifyUpdate{value: fee}(update);
assertEq(alice.balance, 1 ether - fee);

// Alice submits the IDENTICAL update again — also succeeds (no replay guard)
vm.prank(alice);
pythLazer.verifyUpdate{value: 0.5 ether}(update);
assertEq(alice.balance, 1 ether - fee - fee);
``` [8](#0-7) 

The test was written to demonstrate overpayment refund, but it simultaneously proves that the same signed update is accepted on every call — the contract has no mechanism to reject a previously seen update.

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }

        message: vector<u8>,

    ) acquires Storage {

        // Verify fee amount

        // Transfer fee to treasury
        coin::deposit(storage.treasury, fee);

        // Verify signer is trusted and not expired
        let i = 0;
        let valid = false;
        while (i < storage.trusted_signers.length()) {
            let signer_info = vector::borrow(&storage.trusted_signers, (i as u64));
            if (&signer_info.pubkey == &trusted_signer
                && signer_info.expires_at > timestamp::now_seconds()) {
                valid = true;
                break
            };
            i = i + 1;

        assert!(

            EINVALID_SIGNATURE

public(package) fun verify_le_ecdsa_message(
    state: &State,
    clock: &Clock,
    signature: &vector<u8>,
    payload: &vector<u8>,
) {
    let current_cap = state.current_cap();

    // 0 stands for keccak256 hash
    let pubkey = secp256k1_ecrecover(signature, payload, 0);

    // Check if the recovered pubkey is in the trusted signers list
    let trusted_signers = state.trusted_signers(&current_cap);
    let mut maybe_idx = trusted_signers.find_index!(|signer|
        signer.public_key() == &pubkey
    );

    assert!(maybe_idx.is_some(), ESignerNotTrusted);
    let idx = maybe_idx.extract();
    let expires_at_ms = trusted_signers[idx].expires_at_ms();
    assert!(clock.timestamp_ms() < expires_at_ms, ESignerExpired);
}
