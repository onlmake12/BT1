### Title
Unbounded Light-Client Subscription Amplification via `GetLastState{subscribe:true}` — (`util/light-client-protocol-server/src/components/get_last_state.rs`, `sync/src/relayer/mod.rs`)

---

### Summary

Any unauthenticated remote peer can send `GetLastState{subscribe:true}` to set `if_lightclient_subscribed = true` on its peer record. On every accepted block, `build_and_broadcast_compact_block` iterates all connected peers, collects every subscribed one with no cap, and broadcasts a `SendLastState` to all of them. There is no rate limit on the subscription request, no cap on the number of subscribed peers, and no per-block throttle on the resulting `SendLastState` fan-out. An attacker holding N connections can force the server to emit N `SendLastState` messages per block.

---

### Finding Description

**Step 1 — Subscription with no guard.**

`GetLastStateProcess::execute` unconditionally sets the flag on any peer that sends `subscribe: true`: [1](#0-0) 

There is no rate limit, no authentication, no cap on how many peers may hold this flag simultaneously.

**Step 2 — Per-block fan-out with no cap.**

`build_and_broadcast_compact_block` (called on every successfully verified block) collects *all* connected peers whose `if_lightclient_subscribed` is `true` and broadcasts `SendLastState` to every one of them: [2](#0-1) 

There is no `take(MAX_RELAY_PEERS)` or equivalent limit here. Compare this with the compact-block fan-out immediately above it, which is explicitly capped: [3](#0-2) 

`MAX_RELAY_PEERS = 128` is applied to the compact-block broadcast but is **absent** from the `SendLastState` broadcast.

**Step 3 — The Relayer's rate limiter does not cover this path.**

The `RateLimiter` in `Relayer::try_process` only guards relay-protocol messages (CompactBlock, RelayTransactions, etc.): [4](#0-3) 

`GetLastState` is handled by the separate `LightClientProtocol` handler, which has no rate limiter of its own.

---

### Impact Explanation

Each new block causes the server to send one `SendLastState` (containing a `VerifiableHeader` with block header + MMR chain root) to every subscribed peer. An attacker holding N connections, each having sent one `GetLastState{subscribe:true}`, forces N outbound messages per block. The server's outbound bandwidth on the light-client protocol scales linearly with the number of attacker-controlled connections, with no server-side bound other than the global `max_peers` configuration. This constitutes a disproportionate server-side outbound bandwidth amplification triggered by a single logical action (subscribing) per connection.

---

### Likelihood Explanation

The attack requires only the ability to open TCP connections to the node's P2P port and send a single valid light-client protocol message per connection. No PoW, no keys, no privileged role. The `GetLastState` message is a standard, documented part of the light-client protocol. The attack is fully local-testable and requires no mainnet access.

---

### Recommendation

1. **Cap subscribed peers.** In `build_and_broadcast_compact_block`, apply a `take(MAX_RELAY_PEERS)` (or a dedicated `MAX_LIGHT_CLIENT_SUBSCRIBED_PEERS` constant) to the `light_client_peers` iterator before broadcasting, mirroring the existing cap on the compact-block fan-out.
2. **Rate-limit `GetLastState` in the light-client handler.** Add a per-peer rate limiter to `LightClientProtocol`'s message dispatcher, analogous to the one in `Relayer::try_process`.
3. **Optionally enforce a global subscription counter.** Reject `GetLastState{subscribe:true}` once the total number of subscribed peers reaches a configured ceiling.

---

### Proof of Concept

```
1. Start a CKB full node with default config.
2. Open 1000 TCP connections to its P2P port, each negotiating the LightClient protocol.
3. On each connection send: LightClientMessage { GetLastState { subscribe: true } }.
4. Mine (or submit) one valid block to the node.
5. Observe: the node emits exactly 1000 SendLastState messages on the LightClient protocol,
   one per subscribed connection, with no server-side throttle.
6. Measure total outbound bytes = 1000 × sizeof(SendLastState).
   A single legitimate block event has caused O(N) server outbound traffic
   where N is entirely attacker-controlled.
```
