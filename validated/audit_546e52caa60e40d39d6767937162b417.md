Audit Report

## Title
TOCTOU Race in `HeaderMapKernel::get` Causes Unconditional Node Panic During IBD — (`shared/src/types/header_map/kernel_lru.rs`)

## Summary
`HeaderMapKernel::get` releases the memory `RwLock` write guard before calling `SledBackend::remove`, creating a window where two concurrent callers for the same sled-resident key both observe a memory miss and race on the atomic sled remove. The losing thread receives `None`, which propagates to `insert_valid_header`'s unconditional `.expect("parent should be verified")`, panicking and crashing the node. This is reachable from two production P2P message handlers (`HeadersProcess` and `CompactBlockProcess`) with no special privileges required.

## Finding Description
`HeaderMapKernel::get` (`kernel_lru.rs:109–142`) performs a two-phase lookup with no atomicity guarantee across the phases:

1. `self.memory.get_refresh(hash)` (`memory.rs:87–93`) acquires a `RwLock` write guard via `self.0.write()` and drops it on return. After the function returns, no lock is held.
2. `self.backend.remove(hash)` (`backend_sled.rs:90–100`) is then called with no synchronization. Sled's `db.remove` is atomic at the DB level: the first caller receives `Some(v)`, the second receives `None`.

Two threads concurrently processing headers whose parent H is sled-resident (evicted from the memory map by `limit_memory`) can both pass the `memory.get_refresh` check with `None` and then race on `backend.remove`. The losing thread returns `None` from `get()`.

This `None` propagates through `get_header_index_view` (`sync/src/types/mod.rs:1143–1167`) — which only falls back to the chain store if the block is already stored, which it is not during IBD (the `store_first` flag is `tip_number >= header.number()`, which is `false` during IBD) — into `insert_valid_header` (`sync/src/types/mod.rs:1101–1103`), which calls `.expect("parent should be verified")` unconditionally, panicking the node.

The `HEADER_VALID` early-exit in `HeaderAcceptor::accept` (`headers_process.rs:303–322`) is not atomic with the subsequent `insert_valid_header` call at line 356, so two threads processing distinct children C1 and C2 of the same sled-resident parent H can both pass the check (neither C1 nor C2 has `HEADER_VALID` status yet) and both enter `insert_valid_header`, triggering the race on H.

The same `insert_valid_header` call is also present in `CompactBlockProcess` (`sync/src/relayer/compact_block_process.rs:78`), providing a second P2P entry point.

## Impact Explanation
A Rust `panic!` in a tokio task crashes the node process. This maps directly to the **High** bounty impact: *Vulnerabilities which could easily crash a CKB node* (10001–15000 points). The crash is deterministic once the race fires: there is no recovery path, no error return, and no catch around the `.expect`.

## Likelihood Explanation
The preconditions are all met during normal IBD:
- `limit_memory` runs on a 5-second timer (`shared/src/types/header_map/mod.rs:29`, `57–76`) and evicts entries to sled whenever the memory map exceeds `memory_limit` — this is the designed behavior during long-chain sync.
- Two peers (honest or attacker-controlled) sending headers that share a sled-resident parent H is routine; an attacker needs only two simultaneous connections and knowledge of any recently-evicted header hash (observable from the P2P protocol).
- The race window between `memory.get_refresh` returning and `backend.remove` completing is small but non-zero and grows under load. It is reproducible without any PoW, privileged access, or majority hashpower.

## Recommendation
Make the check-and-promote sequence atomic. The minimal fix is to hold the memory write lock across the entire `get_refresh → backend.remove → memory.insert` sequence, or to use a per-key entry API so no other thread can interleave. A lower-cost mitigation is to re-check memory after a failed `backend.remove` (the winning thread will have re-inserted the entry via `memory.insert` at `kernel_lru.rs:137`), returning `Some` instead of `None` in that case. The winning thread's `memory.insert` makes this re-check safe and sufficient to eliminate the panic.

      backend.remove(&H) → Some(v)              backend.remove(&H) → None  ← H lost

**Manual reproduction steps:**
1. Run a CKB node in IBD mode with default `memory_limit` so that `limit_memory` evicts entries to sled.
2. Connect two peers simultaneously; have both send headers whose parent H has been evicted to sled.
3. Time the two `Headers` (or compact block) messages to arrive concurrently (e.g., via a proxy that delays and then releases both at the same instant).
4. Observe the node process exit with `thread panicked at 'parent should be verified'`.

A stress test can instrument `backend.remove` with a `std::thread::yield_now()` after the sled call to widen the race window and make it deterministic in CI.
