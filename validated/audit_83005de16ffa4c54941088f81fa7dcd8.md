Based on my investigation of the code, I can now provide a definitive assessment.

**Key findings:**

1. `LightClientProtocol` struct has **no rate limiter field** — confirmed at `lib.rs` lines 26-29. Compare with `Relayer` which explicitly carries `rate_limiter: RateLimiter<(PeerIndex, u32)>`.

2. `try_process()` dispatches `GetLastState` directly to `GetLastStateProcess::execute()` with **zero rate-limit check** before dispatch.

3. `get_verifiable_tip_header()` does real, non-trivial work per call: snapshot acquisition → DB read for tip block → `chain_root_mmr(tip_number-1).get_root()`.

4. On success, `execute()` returns `Status::ok()`, so `received()` never bans the peer.

5. Other protocols (Relayer at `sync/src/relayer/mod.rs:116-123`, HolePunching at `network/src/protocols/hole_punching/mod.rs:251-257`) have explicit per-peer rate limiters. The light client protocol has none.

### Title
Unbounded Per-Peer `GetLastState` Requests Trigger Unthrottled MMR Root Computation — (`util/light-client-protocol-server/src/components/get_last_state.rs`)

### Summary
Any unprivileged peer connected to a CKB node with the light client protocol enabled can send `GetLastState` messages at maximum rate. Each message unconditionally invokes `get_verifiable_tip_header()`, which acquires a snapshot, reads the tip block from the database, and computes `chain_root_mmr(tip_number-1).get_root()`. There is no per-peer rate limit and no ban on success, allowing a single attacker to drive sustained CPU and I/O overhead proportional to request rate.

### Finding Description
`GetLastStateProcess::execute()` performs no rate-limit check before calling `get_verifiable_tip_header()`: [1](#0-0) 

`get_verifiable_tip_header()` does real work on every call — snapshot acquisition, DB read, and MMR root computation: [2](#0-1) 

The `received()` handler only bans on malformed messages or when `status.should_ban()` is true. A successful `GetLastState` returns `Status::ok()`, so no ban is ever triggered: [3](#0-2) 

`LightClientProtocol` carries no rate limiter field at all: [4](#0-3) 

By contrast, `Relayer` has an explicit per-peer-per-message-type rate limiter checked before any processing: [5](#0-4) 

And `HolePunching` similarly has a rate limiter: [6](#0-5) 

### Impact Explanation
A single malicious peer can flood the server with `GetLastState(subscribe=false)` messages. Each triggers a snapshot acquisition, a DB read for the tip block, and an MMR root computation over `tip_number - 1` leaves. On a long chain, this is a non-trivial I/O and CPU operation. Sustained flooding degrades node responsiveness for legitimate peers and can starve other protocol handlers sharing the async runtime.

### Likelihood Explanation
The attack requires only a valid P2P connection to a node with the light client protocol enabled. No authentication, no PoW, no special state. The message is minimal (a single boolean field). Any peer can execute this immediately after connecting.

### Recommendation
Add a per-peer rate limiter to `LightClientProtocol`, mirroring the pattern used in `Relayer`:
- Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`.
- In `try_process()`, check the rate limiter keyed by `(peer_index, message.item_id())` before dispatching to any handler.
- Return `StatusCode::TooManyRequests` (and optionally ban after repeated violations) when the limit is exceeded.

### Proof of Concept
```
1. Connect a peer to a CKB node with LightClient protocol enabled.
2. In a tight loop, send 10,000 LightClientMessage { GetLastState { subscribe: false } } messages.
3. Measure CPU time spent in get_verifiable_tip_header() (snapshot + DB read + chain_root_mmr().get_root()).
4. Observe that CPU time scales linearly with request count, with no throttling or ban applied.
``` [7](#0-6)

            .set(content)
            .build();
