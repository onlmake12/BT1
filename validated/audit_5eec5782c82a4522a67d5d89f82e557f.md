### Title
DAO Withdrawal Incorrectly Rejected Due to u64 vs. Lowest-Byte Index Width Mismatch Between Rust Verifier and C VM — (`util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` reads the `header_dep_index` from the witness as a full `u64`, while the on-chain C DAO script (`dao.c`) reads only the **lowest byte** of that same field. When a DAO withdrawal transaction has more than 256 `header_deps` and the deposit header sits at an index whose lowest byte differs from the full index value (e.g., index `257`, lowest byte `1`), the two layers resolve **different** `header_deps` entries. The Rust layer then fails its block-number consistency check and rejects the transaction — even though the C VM would correctly resolve the deposit header and accept it.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the deposit header by reading the witness `input_type` field as a full 8-byte little-endian `u64`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

and immediately uses that value to index into `header_deps`:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
``` [1](#0-0) 

The on-chain C DAO script, however, reads the same field using only its **lowest byte** (a `uint8_t` cast), so it resolves `header_deps[index & 0xFF]` rather than `header_deps[index]`.

When `index > 255`, the two layers index into different positions of the `header_deps` array. The Rust layer then performs the consistency check:

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
``` [2](#0-1) 

Because the Rust layer resolved the **wrong** header (not the actual deposit header), this check fails and the transaction is rejected — even though the C VM would have resolved the correct deposit header via the lowest byte and accepted the transaction.

The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this divergence. With `index = 257` (lowest byte = `1`):

- `header_deps[1]` = deposit block (what C VM resolves)
- `header_deps[257]` = withdraw block (what Rust resolves)

The test comment reads: *"Rust resolves index 257 → withdraw block (number 200), but cell data says deposited at block 100. Block number check catches the mismatch."* [3](#0-2) 

This is the direct analog to the external report: the Rust layer uses a **different parameter** (full u64 index) than the authoritative execution layer (lowest-byte index) in its boundary/consistency check, causing legitimate operations to be incorrectly rejected.

---

### Impact Explanation

**Impact: Medium**

Any DAO withdrawal transaction whose witness `input_type` encodes an index `> 255` is rejected by the Rust `DaoCalculator` (called from `check_tx_fee` in the tx-pool admission path) even though the C VM DAO script would accept it. This constitutes a **denial-of-service against legitimate DAO withdrawals**: users with many DAO cells in a single withdrawal transaction — where the deposit header ends up at a `header_deps` position whose full u64 value exceeds 255 but whose lowest byte points to the correct deposit header — cannot withdraw their locked CKB through the normal tx-pool path.

Additionally, the mismatch means the Rust layer computes the wrong maximum withdrawal amount (using the wrong header's `ar` value), producing an incorrect fee calculation for any such transaction. [4](#0-3) 

---

### Likelihood Explanation

**Likelihood: Low**

Triggering this requires a single DAO withdrawal transaction with more than 256 `header_deps` entries, where the deposit header for at least one input lands at a position `> 255`. This is unusual in practice — most users withdraw a small number of DAO cells at once — but it is a reachable condition for power users or scripts that batch many DAO withdrawals. No privileged access, key material, or majority hashpower is required; any transaction sender via the `send_transaction` RPC can trigger the rejection.

---

### Recommendation

The Rust `DaoCalculator` should resolve the `header_dep_index` using the **same width** as the C VM DAO script. If the C VM reads only the lowest byte, the Rust layer must also mask to the lowest byte before indexing:

```rust
// Current (wrong):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.get(header_dep_index as usize)

// Fixed (match C VM behavior):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()) & 0xFF)
// ...
.get(header_dep_index as usize)
```

Alternatively, if the intent is for the C VM to support full u64 indices, the C DAO script should be updated to read the full 8-byte value, and the Rust layer's behavior is already correct. Either way, the two layers must agree on the same parameter. [5](#0-4) 

---

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` is a direct proof of concept:

1. Build a DAO withdrawal transaction with 258 `header_deps`.
2. Place the deposit block hash at `header_deps[1]` (C VM resolves here via lowest byte of `257`).
3. Place the withdraw block hash at `header_deps[257]` (Rust resolves here via full u64 `257`).
4. Set witness `input_type` = `257u64` in little-endian.
5. Set cell data = deposit block number (`100`).

**C VM path**: reads lowest byte `1` → `header_deps[1]` = deposit block (number `100`) → `100 == 100` → **accepts**.

**Rust path**: reads full u64 `257` → `header_deps[257]` = withdraw block (number `200`) → `200 != 100` → `DaoError::InvalidOutPoint` → **rejects**.

The test asserts `result.is_err()`, confirming the Rust layer rejects what the C VM would accept. [6](#0-5)

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}

    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,

    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
