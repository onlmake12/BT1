### Title
`PythLazer.verifyUpdate` Accepts Arbitrarily Old Signed Price Payloads Without Timestamp Freshness Check — (`File: lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.sol`'s `verifyUpdate()` function verifies that the ECDSA signature belongs to a currently-trusted signer, but performs no check that the payload's embedded `timestamp` is recent relative to `block.timestamp`. Any unprivileged relayer who captures a valid signed Lazer update can replay it indefinitely in the future (for as long as the signer key has not expired), causing consumer contracts to receive arbitrarily stale price data.

---

### Finding Description

`verifyUpdate` in `PythLazer.sol` performs three checks:

1. Fee payment (`msg.value >= verification_fee`)
2. EVM format magic bytes
3. Signer validity via `isValidSigner(signer)` — which only checks `block.timestamp < trustedSignerToExpiresAtMapping[signer]` [1](#0-0) 

The payload returned by `verifyUpdate` contains a `timestamp` field (microseconds since Unix epoch) embedded in its header, as defined in `PythLazerStructs.Update`: [2](#0-1) 

This timestamp is parsed by `PythLazerLib.parsePayloadHeader` and `parseUpdateFromPayload`, but `verifyUpdate` itself never parses or validates it against `block.timestamp`. [3](#0-2) 

The result is that a signed Lazer update is valid for the entire lifetime of the signer key (which can be set to arbitrarily far in the future by the owner), not just for the moment it was produced.

---

### Impact Explanation

Consumer contracts that call `verifyUpdate` and use the returned payload to update their on-chain price state can be fed prices from arbitrarily far in the past. This is directly analogous to the ForwardRequest deadline issue: just as a signed meta-transaction without a deadline can be executed long after the signer intended, a signed Lazer price update without a freshness check can be submitted long after the price was valid.

Concrete impact:
- A DeFi protocol using Lazer prices for liquidations or collateral valuation could be manipulated by replaying a price update from a period when the asset was at a significantly different value.
- There is no mechanism for a consumer to distinguish a freshly-submitted update from a replayed one at the `verifyUpdate` level — the function returns `(payload, signer)` with no freshness guarantee.

---

### Likelihood Explanation

The attack requires only that an unprivileged actor capture a valid signed Lazer update (trivially available from the public Lazer WebSocket stream or from on-chain transaction history) and submit it later. No privileged access, leaked keys, or oracle manipulation is required. The window of opportunity is bounded only by the signer key's `expiresAt`, which can be set years into the future. [4](#0-3) 

---

### Recommendation

Add a `maxAge` parameter (or a hardcoded staleness bound) to `verifyUpdate`. After recovering the signer and before returning, parse the payload's `timestamp` field and require it to be within an acceptable window of `block.timestamp`:

```solidity
// After recovering signer and validating it:
uint64 payloadTimestampUs = uint64(bytes8(payload[4:12])); // offset after PAYLOAD_MAGIC
uint64 payloadTimestampSec = payloadTimestampUs / 1_000_000;
require(
    block.timestamp <= payloadTimestampSec + MAX_PAYLOAD_AGE_SECONDS,
    "payload too old"
);
```

Alternatively, expose a `verifyUpdate(bytes calldata update, uint256 maxAgeSeconds)` overload so consumers can enforce their own freshness requirements at the verification layer rather than relying on post-hoc timestamp checks in application code.

---

### Proof of Concept

1. At time `T`, a Lazer relayer receives a valid signed update for BTC/USD at price `$100,000` and records the raw `update` bytes.
2. At time `T + 6 months`, BTC/USD is at `$50,000`.
3. The attacker calls:
   ```solidity
   pythLazer.verifyUpdate{value: fee}(capturedUpdate);
   ```
4. `verifyUpdate` succeeds — the signer key has not expired, the signature is valid, and no timestamp check is performed.
5. The returned `payload` contains `timestamp = T` and `price = $100,000`.
6. A consumer contract that trusts `verifyUpdate`'s return value updates its stored price to `$100,000`, enabling the attacker to exploit the inflated price (e.g., over-borrow against collateral, avoid liquidation, or trigger incorrect liquidations of other users). [5](#0-4)
