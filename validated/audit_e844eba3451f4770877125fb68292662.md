### Title
Copy-Paste Bug in `script_exists_in_output` Causes Silent Script Deletion on PostgreSQL After Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

`script_exists_in_output` contains a concrete copy-paste bug on line 252: the final `match` block reads from `row_lock` (the lock-script query result) instead of `row_type` (the type-script query result). On PostgreSQL this silently returns the wrong boolean, causing scripts that are only referenced as `type_script` to be deleted from the `script` table during rollback even when surviving outputs still reference them. SQLite is unaffected because its integer return type causes the `try_get::<bool,_>` call to fail and fall through to the correct `row_type` branch.

### Finding Description

The function queries the `output` table twice:

1. `row_lock` — checks `WHERE lock_script_id = $1`
2. `row_type` — checks `WHERE type_script_id = $1`

The early-return path (lines 223–235) is correct: if `row_lock` is true it returns immediately. But the final return path (lines 252–256) re-matches on `row_lock` instead of `row_type`:

```rust
// line 252 — BUG: row_lock used instead of row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),          // returns lock result, not type result
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),  // SQLite: correct
}
``` [1](#0-0) 

On PostgreSQL, `row_lock.try_get::<bool, _>(0)` always succeeds (PostgreSQL `EXISTS` returns `BOOLEAN`), so the `Ok(r)` arm always fires and returns the **lock-script** existence result — not the type-script result. On SQLite, `try_get::<bool,_>` fails (SQLite returns `BIGINT`), so the `Err(_)` arm fires and correctly reads `row_type`.

**Execution path during rollback:**

1. `rollback_block` deletes block N's `output` rows first (line 25).
2. It then iterates `output_lock_type_list` (collected before deletion) and calls `script_exists_in_output` for each `lock_script_id` / `type_script_id`.
3. For script S that is only a `type_script` (never a `lock_script`): `row_lock` → `false`; `row_type` → `true` (block N-1 outputs still exist). Due to the bug, the function returns `false`.
4. S is pushed into `script_id_list_to_remove` and deleted (line 39).
5. Block N-1's `output` rows still hold `type_script_id = S`, but S no longer exists in the `script` table — a dangling reference. [2](#0-1) 

### Impact Explanation

All subsequent `get_cells` / `get_transactions` RPC calls that filter by `type_script = S` will return empty results, because the script row is gone. The indexer's view of the canonical chain permanently diverges: block N-1's outputs are invisible to any script-based query. Any application (wallet, dApp, bridge) relying on the rich-indexer for cell discovery will silently miss live cells.

### Likelihood Explanation

Chain reorganizations are a normal part of CKB operation. An attacker who can produce a competing fork (even a short one) can deterministically trigger this path. No privileged access is required — only the ability to broadcast valid blocks. The precondition (script S used as `type_script` in two consecutive blocks, one of which is rolled back) is trivially satisfiable in any real deployment. The PostgreSQL backend is the recommended production backend for the rich-indexer.

### Recommendation

Change line 252 from `row_lock` to `row_type`:

```rust
// correct
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [3](#0-2) 

### Proof of Concept

```
1. Start node with PostgreSQL rich-indexer backend.
2. append(block_N_minus_1)  — contains output with type_script = S, lock_script = L1 (L1 ≠ S)
3. append(block_N)          — contains output with type_script = S, lock_script = L2 (L2 ≠ S)
4. rollback()               — removes block_N outputs; S should still exist for block_N_minus_1
5. get_cells(type_script=S) — returns [] on PostgreSQL, correct result on SQLite
```

The differential between SQLite (correct) and PostgreSQL (empty) is directly observable and confirms the bug. The root cause is the single wrong variable name on line 252. [4](#0-3)

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
