### Title
Hard-Coded `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` Blocks Keeper Updates for Long-Heartbeat Price Feeds in Scheduler — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.sol` contract enforces a hard-coded `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` constant when validating incoming price update data. For subscriptions that contain only price feeds with heartbeats longer than 1 hour (e.g., Pyth equity feeds during market closures, or any asset with a ≥ 24-hour heartbeat), the maximum `publishTime` across all feeds in the update will always be older than 1 hour. This causes every keeper call to `updatePriceFeeds` to revert with `TimestampTooOld`, permanently blocking updates and leaving stored prices stale.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint64 public constant PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours;
``` [1](#0-0) 

In `_validateShouldUpdatePrices`, the Scheduler computes the **maximum** `publishTime` across all price feeds in the submitted update data, then checks it against this constant:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}

uint256 minAllowedTimestamp = (block.timestamp > PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    : 0;

if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(updateTimestamp, block.timestamp);
}
``` [2](#0-1) 

The comment in `SchedulerConstants.sol` explicitly acknowledges that some markets close for multiple days and that the Pyth contract parsing layer intentionally does **not** apply this check for that reason:

> "We don't use this when parsing update data from the Pyth contract because [we] don't want to reject update data if it contains a price from a market that closed a few days ago." [3](#0-2) 

Despite this acknowledgment, the Scheduler then applies a 1-hour hard-coded window to the **maximum** timestamp across all feeds in the batch. For a subscription containing **only** long-heartbeat feeds (e.g., equity feeds like AAPL/USD that only update during market hours), the maximum `publishTime` will be from the last trading session — potentially 16+ hours ago on a weekday evening, or 64+ hours ago after a weekend. Every keeper call will revert with `TimestampTooOld`.

The constant is not configurable per subscription and cannot be overridden without a contract upgrade.

---

### Impact Explanation

Any whitelisted keeper calling `updatePriceFeeds` on a subscription containing only long-heartbeat assets is permanently blocked from pushing updates. The subscription's stored prices become stale. Downstream consumers using `getPricesNoOlderThan` will revert with `StalePrice`, and those using `getPricesUnsafe` will silently receive arbitrarily old prices. [4](#0-3) 

The Scheduler's core value proposition — guaranteed push-based price freshness — is entirely defeated for affected subscriptions.

---

### Likelihood Explanation

Pyth Network publishes equity price feeds (stocks, ETFs, commodities) with heartbeats tied to market hours. Any subscription manager who creates a Scheduler subscription for equity-only feeds will encounter this revert during every market closure. The Scheduler interface imposes no restriction on which Pyth price feed IDs can be subscribed to, so this is a reachable condition for any unprivileged user who creates such a subscription. The keeper (also unprivileged) is the transaction sender who triggers the revert.

---

### Recommendation

1. Make `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` a per-subscription configurable parameter (set at subscription creation time), so that subscriptions with long-heartbeat feeds can specify an appropriate window.
2. Alternatively, increase the global constant to accommodate the longest supported heartbeat (e.g., 7 days to cover weekend closures), since the max-timestamp logic already ensures at least one feed in a mixed subscription is fresh.
3. Add validation at subscription creation time to warn or reject configurations where the heartbeat of the subscribed feeds is known to exceed `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`.

---

### Proof of Concept

1. Deploy or use the existing `Scheduler` contract on a network where Pyth equity feeds are available.
2. Create a subscription containing only an equity price feed (e.g., AAPL/USD, which has a heartbeat tied to NYSE market hours).
3. Wait until the market closes (e.g., after 4 PM ET on a Friday).
4. After more than 1 hour has elapsed since the last equity feed update, fetch the latest available update data from Hermes for that feed.
5. Call `scheduler.updatePriceFeeds(subscriptionId, updateData)` as a keeper.

**Expected result:**

```
revert SchedulerErrors.TimestampTooOld(
    updateTimestamp,   // e.g., last Friday 4 PM ET
    block.timestamp    // e.g., Saturday 10 AM ET
)
```

The subscription can never be updated until the market reopens and a fresh price is published within the 1-hour window — but even then, the keeper must call within 1 hour of market open, creating a narrow and fragile update window. [5](#0-4)
