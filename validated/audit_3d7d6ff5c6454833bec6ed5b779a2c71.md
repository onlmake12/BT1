The execution path is fully traceable and confirmed. Here is the complete analysis:

**Execution trace for the attack:**

1. `execute()` checks: non-empty ✓, `len() > 1000` (exactly 1000 passes) ✓, `last_hash` on main chain ✓, no duplicates ✓
2. `partition(|h| snapshot.is_main_chain(h))` — all 1000 random hashes → `missing`, `found` is empty
3. The `for block_hash in found` loop body never executes; `positions` is empty
4. `reply_proof()` is called with empty `positions`
5. Inside `reply_proof()`: `last_block.is_genesis()` is false → acquires a **second snapshot**, calls `snapshot.chain_root_mmr(last_block.number() - 1).get_root()` (O(log N) RocksDB reads), skips `gen_proof` (positions empty), returns `Status::ok()`
6. Back in `received()`: `status.should_ban()` checks `if !(400..500).contains(&code)` — code is 200, so **no ban**

**Missing guard — no rate limiter on `LightClientProtocol`:**

`Relayer::new()` installs a `governor::RateLimiter` at 30 req/sec per peer. `HolePunching::new()` installs two rate limiters. `LightClientProtocol::new()` stores only `shared: Shared` — no rate limiter field, no rate check in `try_process()`.

### Title
Unbounded `GetBlocksProof` Requests with All-Missing Hashes Trigger Repeated MMR Root Computation and Snapshot Acquisition Without Rate Limiting or Peer Ban — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

### Summary
An unprivileged remote peer can repeatedly send well-formed `GetBlocksProof` messages containing up to 1000 non-main-chain hashes and a valid `last_hash`, causing the server to perform 1000 RocksDB `is_main_chain()` lookups, acquire two chain snapshots, and compute an MMR root per message, with no rate limiting and no peer ban.

### Finding Description

`GetBlocksProofProcess::execute()` accepts any message where `block_hashes` is non-empty, `len() ≤ GET_BLOCKS_PROOF_LIMIT` (1000), `last_hash` is on the main chain, and all hashes are unique. [1](#0-0) 

When all 1000 `block_hashes` are absent from the main chain, `found` is empty and `missing` holds all 1000 entries. [2](#0-1) 

`reply_proof()` is then called with an empty `positions` vector. Because `last_block` is not genesis, it acquires a fresh snapshot and calls `mmr.get_root()` unconditionally before checking whether `positions` is empty. [3](#0-2) 

The function returns `Status::ok()` (code 200). The `received()` dispatcher only bans peers for 4xx status codes, so no ban is issued. [4](#0-3) 

Unlike `Relayer` (30 req/sec governor rate limiter) and `HolePunching` (two rate limiters), `LightClientProtocol` carries no rate limiter and performs no rate check in `try_process()`. [5](#0-4) 

The constant `GET_BLOCKS_PROOF_LIMIT = 1000` sets the per-message work ceiling. [6](#0-5) 

The existing test explicitly asserts `nc.not_banned(peer_index)` for the all-missing case, confirming this is the intended (but unguarded) behavior. [7](#0-6) 

### Impact Explanation
Each message forces: 2 RocksDB snapshot acquisitions, 1000 `is_main_chain()` point-reads (each a `rocksdb_get_pinned_cf` call), and one MMR root computation reading O(log N) peak nodes from RocksDB. A single attacker connection can sustain this at network/async throughput with no back-pressure. On a long chain (mainnet height ~14M), the MMR has ~24 peaks to read per call. Sustained flooding degrades RocksDB read throughput and increases I/O latency for the node's normal chain operations.

### Likelihood Explanation
The attack requires only a valid P2P connection to a node with the light client server enabled. The attacker needs only the current tip hash (publicly broadcast via `SendLastState`). No PoW, no keys, no privileged access. The attack is trivially scriptable.

### Recommendation
1. Add a per-peer rate limiter to `LightClientProtocol` analogous to `Relayer`'s governor-based limiter.
2. Optionally, short-circuit `reply_proof()` before calling `mmr.get_root()` when `items_positions` is empty and `missing_items` is non-empty — the MMR root is still needed for the `VerifiableHeader`, but the proof generation can be skipped (already done); the root computation itself could be deferred or cached.
3. Consider returning a 4xx status (triggering a ban) when the ratio of missing to requested hashes exceeds a threshold, as a fully-missing response is a strong signal of a malicious or buggy client.
