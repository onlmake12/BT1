Audit Report

## Title
Unbounded JSON-RPC Batch Request Causes RPC Server Resource Exhaustion (DoS) — (File: rpc/src/server.rs)

## Summary
`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is only initialized when `config.rpc_batch_limit` is explicitly set. The shipped default configuration leaves `rpc_batch_limit` commented out, so the guard in `handle_jsonrpc` is never entered and batch requests are processed with no upper bound on call count. A local process can send a single HTTP POST with a JSON array of tens of thousands of calls (within the 10 MiB body limit) and force the server to execute them all sequentially, degrading RPC responsiveness for legitimate callers.

## Finding Description
`static JSONRPC_BATCH_LIMIT: OnceLock<usize> = OnceLock::new()` is declared at line 34 of `rpc/src/server.rs`. It is populated only inside the conditional at lines 53–55:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

When `rpc_batch_limit` is absent (the default), `JSONRPC_BATCH_LIMIT.get()` returns `None`. The guard at lines 275–282 of `handle_jsonrpc`:

```rust
if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
    && calls.len() > *batch_size
```

is therefore never entered. The batch is dispatched via `.then()` at lines 284–289, processing each call sequentially within the async task. The field `rpc_batch_limit` in `util/app-config/src/configs/rpc.rs` line 44 is `Option<usize>` with no forced default. `resource/ckb.toml` lines 205–208 explicitly comment out the option with a comment acknowledging the risk (`# rpc_batch_limit = 2000`). The only body-size constraint is `max_request_body_size = 10485760` (10 MiB, line 187), allowing ~130,000–200,000 minimal calls per request. The `TimeoutLayer` at lines 125–128 bounds individual request duration to 30 seconds but does not prevent sustained flooding via concurrent connections.

## Impact Explanation
The RPC server's async task for a large batch is occupied for up to 30 seconds processing sequential calls, and concurrent large-batch connections can contend on shared resources (CPU, tx-pool read locks via methods like `get_block_template`), degrading or blocking responses to legitimate callers. The RPC endpoint defaults to `127.0.0.1:8114` (localhost only). This matches **Note (0–500 points) — Any local RPC API crash** from the allowed CKB bounty impact scope.

## Likelihood Explanation
Any local process — a compromised application, malicious dependency, or script run by the node operator — can reach `127.0.0.1:8114` without credentials. The attack requires only a single HTTP POST with a crafted JSON array; no special protocol knowledge beyond JSON-RPC 2.0 batch specification is needed. The default configuration ships with the protection disabled, and the config comment itself acknowledges the risk. The attack is repeatable and can be sustained with concurrent connections across the 30-second timeout window.

## Recommendation
Set a safe hard-coded default for `JSONRPC_BATCH_LIMIT` that applies even when `rpc_batch_limit` is absent from the configuration:

```rust
const DEFAULT_BATCH_LIMIT: usize = 2000;
let limit = config.rpc_batch_limit.unwrap_or(DEFAULT_BATCH_LIMIT);
let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| limit);
```

This ensures the guard at lines 275–282 of `server.rs` is always active. Additionally, consider processing batch calls with bounded concurrency (e.g., `buffer_unordered`) rather than purely sequentially via `.then()`, so that a single slow call does not block the entire batch response stream.

## Proof of Concept
```python
import json, requests

batch = [
    {"jsonrpc": "2.0", "method": "get_tip_block_number", "params": [], "id": i}
    for i in range(100_000)
]
payload = json.dumps(batch)
# ~5.5 MiB — within the 10 MiB max_request_body_size limit

resp = requests.post(
    "http://127.0.0.1:8114",
    data=payload,
    headers={"Content-Type": "application/json"},
    timeout=120,
)
print(resp.status_code)
```

Run multiple concurrent instances to sustain pressure across the 30-second timeout window. Substituting `get_block_template` amplifies impact by contending on the tx-pool read lock.
