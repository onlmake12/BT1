Audit Report

## Title
Stale `descendants_*` / `evict_key` on surviving ancestors after `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

## Summary

`remove_entry_and_descendants` pre-removes all link entries for the target subtree in Phase 1, then calls `remove_entry` (which internally calls `update_ancestors_index_key`) in Phase 2. Because the links are already gone, `calc_ancestors` returns an empty set and surviving ancestors outside the removed subtree never have their `descendants_count / descendants_fee / descendants_size / descendants_cycles` decremented. Their `evict_key` is permanently inflated, causing the eviction selector to skip them even when they should be the next eviction candidate.

## Finding Description

**Root cause — two-phase removal in `remove_entry_and_descendants` (lines 252–265):** [1](#0-0) 

Phase 1 calls `remove_entry_links` for every node in the subtree. `remove_entry_links` (lines 418–430) removes the target from `links.inner` entirely: [2](#0-1) 

Phase 2 calls `remove_entry`, which at line 242 calls `update_ancestors_index_key`: [3](#0-2) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`: [4](#0-3) 

`calc_ancestors` calls `calc_relative_ids`, which does `self.inner.get(short_id)`: [5](#0-4) 

Since `short_id` was already removed from `links.inner` in Phase 1, `get` returns `None`, `direct` is empty, the BFS returns `{}`, and the loop body in `update_ancestors_index_key` never executes. No ancestor's `descendants_*` fields are decremented.

**Exact trace for chain tx0 → tx1 → tx2, calling `remove_entry_and_descendants(tx1)`:**

- Phase 1 `remove_entry_links(tx1)`: removes tx1 from tx0's `children` set, removes tx1 from tx2's `parents` set, removes tx1 from `links.inner`.
- Phase 1 `remove_entry_links(tx2)`: tx2's `parents` is now `{}` (tx1 already removed), removes tx2 from `links.inner`.
- After Phase 1: tx0 is still in `links.inner` with an empty `children` set; tx1 and tx2 are gone.
- Phase 2 `remove_entry(tx1)` → `update_ancestors_index_key(tx1, Remove)` → `calc_ancestors(tx1_id)` → `inner.get(tx1_id)` returns `None` → `{}` → loop never runs → tx0's `descendants_*` fields unchanged.
- Same for `remove_entry(tx2)`.

**Result:** tx0 remains in the pool with stale fields:

| Field | Expected | Actual |
|---|---|---|
| `descendants_count` | 1 | 3 |
| `descendants_fee` | fee(tx0) | fee(tx0)+fee(tx1)+fee(tx2) |
| `descendants_size` | size(tx0) | size(tx0)+size(tx1)+size(tx2) |
| `descendants_cycles` | cycles(tx0) | cycles(tx0)+cycles(tx1)+cycles(tx2) |

The comment on line 256 confirms the intent was only to suppress `update_descendants_index_key` for the removed set, but the implementation also silences `update_ancestors_index_key` for surviving ancestors.

**Trigger path — `resolve_conflict` (lines 305–332):** [6](#0-5) 

Submitting a transaction that double-spends an in-pool child is a standard, unprivileged P2P/RPC path that directly triggers `remove_entry_and_descendants`.

## Impact Explanation

`EvictKey` is computed from the stale `descendants_*` fields: [7](#0-6) 

With inflated `descendants_fee` and `descendants_size/cycles`, `descendants_feerate` is inflated. Since `fee_rate = descendants_feerate.max(feerate)`, the `fee_rate` field in `EvictKey` is inflated.

`EvictKey` ordering (sort_key.rs lines 92–103): lower `fee_rate` → evicted first: [8](#0-7) 

With tx0's `fee_rate` inflated, it sorts later in eviction order and is skipped by `next_evict_entry`: [9](#0-8) 

An attacker fills the pool with low-fee tx0-class transactions, each with a descendant chain. After double-spending each child to trigger `remove_entry_and_descendants`, every surviving tx0 has an inflated `evict_key`. The pool becomes full of low-fee transactions that appear more valuable than they are, preventing legitimate high-fee transactions from entering. This matches **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

The trigger path is entirely unprivileged: submit a transaction that double-spends an in-pool child. This is a standard P2P/RPC transaction submission path. No PoW, no key material, no operator access is required. The attacker pays fees for the initial chain and the double-spend, but the cost is proportional to the number of tx0 entries they want to pin, not to the pool size. The attack is repeatable and cheap relative to the pool-pinning effect achieved.

## Recommendation

In `remove_entry_and_descendants`, update surviving ancestors **before** removing links. Specifically, for each entry being removed, call `update_ancestors_index_key(entry, Remove)` while the links are still intact, then remove the links. Alternatively, after the pre-removal loop, explicitly walk the set of surviving ancestors (those in the parent set of `removed_ids` but not in `removed_ids` themselves) and call `sub_descendant_weight` on each for every removed descendant, then refresh their `evict_key`.

## Proof of Concept

```rust
// Setup: tx0 -> tx1 -> tx2
map.add_pending(tx0); map.add_pending(tx1); map.add_pending(tx2);

// Trigger: remove tx1 and its descendants (via resolve_conflict double-spend)
map.remove_entry_and_descendants(&tx1_id);

// tx0 must still be in pool
assert!(map.contains_key(&tx0_id));

// Invariant: descendants_count of tx0 must equal actual live descendants
let live_desc = map.calc_descendants(&tx0_id).len() + 1; // +1 for self
let stored = map.get(&tx0_id).unwrap().descendants_count;
assert_eq!(stored, live_desc);
// FAILS: stored == 3, live_desc == 1

// Eviction invariant: tx0 with low fee_rate must be evicted before a high-fee tx
// FAILS: tx0's inflated evict_key causes it to be skipped
```

The `remove_entry_links` pre-removal loop (lines 257–259) is the precise point where the ancestor link information needed by `update_ancestors_index_key` (lines 432–445) is destroyed before it can be used.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
