### Title
Wrong File Opened in Freezer Repair Loop File-Slip Branch — (`freezer/src/freezer_files.rs`)

---

### Summary

In `FreezerFilesBuilder::build`, the crash-repair loop contains a one-line mistake: when it detects that stripping the last index entry has slipped back into an earlier data file (`new_index.file_id != head_index.file_id`), it re-opens `head_index.file_id` (the file it just abandoned) instead of `new_index.file_id` (the earlier file that must now become the head). Every subsequent comparison and possible truncation therefore operates on the wrong file, leaving the `FreezerFiles` struct in a permanently inconsistent state.

---

### Finding Description

**The bug — line 458:**

```rust
// slipped back into an earlier head-file
if new_index.file_id != head_index.file_id {
    let head_file_name = helper::file_name(head_index.file_id);  // ← BUG: should be new_index.file_id
    let (new_head, size) = self.open_append(self.file_path.join(head_file_name))?;
    head = new_head;
    head_size = size;
}
expect_head_size = new_index.offset;
head_index = new_index;           // head_index now has file_id = new_index.file_id
``` [1](#0-0) 

The intent of the branch is: "we stripped the last index entry and the new last entry belongs to an earlier blk file, so switch `head` to that earlier file." The code instead re-opens the file that was just stripped, which is the wrong file.

**Concrete two-file scenario:**

| State | Value |
|---|---|
| INDEX last entry | `{file_id=2, offset=Y}` |
| INDEX second-to-last | `{file_id=1, offset=X}` |
| `blk000002` actual size | `S < Y` (crash left it short) |

1. Pre-loop: `head_index={file_id=2,offset=Y}`, `head`=blk000002, `head_size=S`, `expect_head_size=Y`.
2. `Y > S` → strip last index entry; read `new_index={file_id=1,offset=X}`.
3. `new_index.file_id(1) != head_index.file_id(2)` → slip branch fires.
4. **Bug**: opens `helper::file_name(head_index.file_id)` = `blk000002` again.
5. `head_size` = size of blk000002 = `S`; `expect_head_size` = `X`.
6. `head_index` is updated to `{file_id=1,offset=X}` (line 464).

After the loop exits (when `X == S` by coincidence, or after further wrong truncations):

- `head` file handle → blk000002 [2](#0-1) 
- `head_id` → 1 (from `head_index.file_id`) [3](#0-2) 

The struct is constructed with a mismatched `head` (file handle for blk000002) and `head_id=1`. All subsequent `append` calls write data into blk000002 while writing index entries that record `file_id=1`, making every future `retrieve` call read from the wrong file or wrong offset.

If `X < S` the loop additionally truncates blk000002 to length `X`, destroying valid data that belongs to blk000001's logical range. [4](#0-3) 

---

### Impact Explanation

- **Frozen block data is silently corrupted.** `retrieve()` reads from the file and offset recorded in the index; after the mismatch, those coordinates point into the wrong blk file. [5](#0-4) 
- **Subsequent appends compound the corruption.** New blocks are written to blk000002 but indexed under `file_id=1`, so every block appended after restart is unreadable or returns garbage.
- **Consensus deviation.** Peers syncing historical blocks from this node receive wrong data; nodes that rely on their own frozen store for reorg resolution will apply incorrect block data.

---

### Likelihood Explanation

The trigger is a crash (power loss, OOM kill, SIGKILL) that occurs after the node has rolled over to a new blk file and written at least one index entry for it, but before the data file is fully flushed. This is a normal operating condition for any long-running node. No attacker action is required to create the precondition; the repair loop is supposed to handle exactly this case. The entry point is "database restart/reorg state," which is explicitly in scope.

---

### Recommendation

Change line 458 from `head_index.file_id` to `new_index.file_id`:

```rust
if new_index.file_id != head_index.file_id {
    let head_file_name = helper::file_name(new_index.file_id);  // ← correct
    let (new_head, size) = self.open_append(self.file_path.join(head_file_name))?;
    head = new_head;
    head_size = size;
}
``` [6](#0-5) 

---

### Proof of Concept

1. Create a freezer directory with `max_file_size` set small enough that two blk files exist (e.g., blk000001 and blk000002).
2. Craft the INDEX file so the last entry is `{file_id=2, offset=Y}` and the second-to-last is `{file_id=1, offset=X}`, where `Y > actual size of blk000002`.
3. Call `FreezerFilesBuilder::build`.
4. Observe: the slip branch opens blk000002 instead of blk000001; `head_size` = size(blk000002); `expect_head_size` = X.
5. If `X < size(blk000002)`, blk000002 is truncated to X bytes (wrong file truncated).
6. The returned `FreezerFiles` has `head_id=1` but `head` file handle pointing to blk000002.
7. Call `retrieve()` on a block whose index entry says `file_id=1`; it reads from blk000002 at the wrong offset, returning corrupt data.
