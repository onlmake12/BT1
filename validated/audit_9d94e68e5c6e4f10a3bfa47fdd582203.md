### Title
Missing Request Cancellation/Refund Mechanism Permanently Locks User Funds in Echo Contract - (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol` has no cancellation or refund function for unfulfilled price update requests. When a user calls `requestPriceUpdatesWithCallback`, the provider fee portion of their payment (`req.fee`) is held in the contract and can only be released when the provider calls `executeCallback`. If the provider never fulfills the request, the user's funds are permanently locked with no recovery path.

---

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback` splits the user's payment into two parts:

1. The Pyth protocol fee is immediately credited to `_state.accruedFeesInWei`
2. The remaining provider fee is stored as `req.fee` inside the `Request` struct [1](#0-0) 

The only code path that ever releases `req.fee` is inside `executeCallback`, which credits the provider: [2](#0-1) 

A complete audit of the `Echo.sol` and `IEcho.sol` interfaces confirms there is **no** `cancelRequest`, `refundRequest`, or any timeout-based expiry function: [3](#0-2) 

The admin's `withdrawFees` function only covers `_state.accruedFeesInWei` (the Pyth protocol fee), not the per-request `req.fee` held in individual `Request` structs: [4](#0-3) 

The `Request` struct confirms `fee` is stored per-request with no expiry field: [5](#0-4) 

The code itself acknowledges the missing mechanism with a TODO comment: [6](#0-5) 

Provider registration is **permissionless** — anyone can call `registerProvider`: [7](#0-6) 

---

### Impact Explanation

Any user who calls `requestPriceUpdatesWithCallback` against a provider that never calls `executeCallback` permanently loses the provider-fee portion of their payment (`req.fee = msg.value - pythFeeInWei`). There is no on-chain mechanism to cancel the request or recover the funds. This is a direct, permanent loss of user ETH.

---

### Likelihood Explanation

The attack path requires no privileged access:

1. A malicious actor calls `registerProvider` with attractive fees.
2. Users call `requestPriceUpdatesWithCallback`, paying fees to this provider.
3. The provider never calls `executeCallback`.
4. All `req.fee` values are permanently locked.

Even without malicious intent, a legitimate provider going offline produces the same outcome. After the exclusivity period expires, any third party *can* call `executeCallback` — but only if they possess valid Pyth price data for the exact `publishTime` of the request. For requests with stale `publishTime` values where historical data is unavailable, no one can fulfill them, and the funds remain locked indefinitely.

---

### Recommendation

Add a `cancelRequest` function that allows the original requester to reclaim `req.fee` after a configurable timeout period has elapsed past `publishTime`. For example:

```solidity
function cancelRequest(uint64 sequenceNumber) external {
    Request storage req = findActiveRequest(sequenceNumber);
    require(msg.sender == req.requester, "Only requester can cancel");
    require(
        block.timestamp > req.publishTime + CANCELLATION_TIMEOUT,
        "Cancellation timeout not elapsed"
    );
    uint96 refundAmount = req.fee;
    clearRequest(sequenceNumber);
    (bool sent, ) = msg.sender.call{value: refundAmount}("");
    require(sent, "Refund failed");
    emit RequestCancelled(sequenceNumber, msg.sender, refundAmount);
}
```

This mirrors the recommended fix in H-01: add a dedicated built-in operation for the missing claim path rather than relying on an external workaround.

---

### Proof of Concept

```
Attack steps:
1. Attacker calls registerProvider(baseFee, feePerFeed, feePerGas) — permissionless.
2. Victim calls requestPriceUpdatesWithCallback{value: getFee(...)}(attackerProvider, publishTime, priceIds, gasLimit).
   - req.fee = msg.value - pythFeeInWei is stored in the Request struct.
   - _state.accruedFeesInWei += pythFeeInWei (Pyth fee credited immediately).
3. Attacker never calls executeCallback.
4. Victim has no cancelRequest, no refund, no timeout expiry.
5. req.fee is permanently locked in the Echo contract.
6. Victim calls withdrawFees — reverts because they are not admin.
7. Admin calls withdrawFees — only recovers _state.accruedFeesInWei (Pyth fee), not req.fee.
8. req.fee is irrecoverable.
```
