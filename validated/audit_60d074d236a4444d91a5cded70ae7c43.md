### Title
Unvalidated Stale Price Returned by `PythAggregatorV3.latestRoundData()` — (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink `AggregatorV3Interface`-compatible adapter, explicitly documented as the migration path for protocols moving from Chainlink to Pyth. Every price-reading function in the contract — including `latestRoundData()`, `latestAnswer()`, `getRoundData()`, and `decimals()` — calls `pyth.getPriceUnsafe(priceId)` without any staleness check or price validity check. Any protocol that deploys this adapter and calls `latestRoundData()` will silently receive an arbitrarily stale price with no on-chain revert.

---

### Finding Description

`PythAggregatorV3.sol` implements the Chainlink `AggregatorV3Interface` by wrapping a single Pyth price feed. The contract is distributed as part of the official `@pythnetwork/pyth-sdk-solidity` NPM package and is the recommended drop-in replacement for Chainlink feeds.

Every price-reading function delegates to `pyth.getPriceUnsafe(priceId)`:

```solidity
// PythAggregatorV3.sol lines 110–118
function latestRoundData() external view returns (
    uint80 roundId, int256 answer,
    uint256 startedAt, uint256 updatedAt, uint80 answeredInRound
) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId); // ← no staleness check
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

`getPriceUnsafe()` is explicitly documented as returning prices "from arbitrarily far in the past." The underlying `Pyth.sol` implementation only reverts if `publishTime == 0` (feed never initialized), but performs no staleness check and no `price > 0` check:

```solidity
// Pyth.sol lines 184–194
function getPriceUnsafe(bytes32 id) public view override returns (PythStructs.Price memory price) {
    PythInternalStructs.PriceInfo storage info = _state.latestPriceInfo[id];
    price.publishTime = info.publishTime;
    price.expo = info.expo;
    price.price = info.price;
    price.conf = info.conf;
    if (price.publishTime == 0) revert PythErrors.PriceFeedNotFound(); // only check
}
```

Missing validations in `latestRoundData()` (and all other price-reading functions):

| Check | Chainlink Analog | Present in `PythAggregatorV3`? |
|---|---|---|
| Staleness (`updatedAt != 0` / age bound) | `require(updateTime != 0, "Incomplete round")` | **No** |
| Price validity (`answer > 0`) | `require(price > 0, "Chainlink Malfunction")` | **No** |

The `updatedAt` field returned is `price.publishTime`, which can be hours or days old. The `answeredInRound` field is set equal to `roundId` (both are `uint80(price.publishTime)`), so any downstream `answeredInRound >= roundId` check will trivially pass, masking the staleness.

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a Chainlink-compatible oracle and calls `latestRoundData()` will:

1. **Receive stale prices silently** — during a Pyth network outage, Hermes downtime, or simply when no keeper has pushed a recent update, the adapter returns the last cached price with no revert. The `updatedAt` timestamp will be old, but most protocols do not check it.
2. **Receive zero or negative prices** — Pyth's `price.price` is `int64`; a zero or negative value is not rejected by `getPriceUnsafe()`. Downstream protocols that cast this to `uint256` for collateral valuation will compute incorrect (potentially zero or wrapped) values.

Concrete protocol-level impact: incorrect collateral valuation → under-collateralized borrows go unliquidated (insolvency) or healthy positions get liquidated (user loss). This matches the exact impact class of the reference Chainlink report.

---

### Likelihood Explanation

- `PythAggregatorV3` is the **official, documented migration path** for Chainlink users. The developer hub guide at `apps/developer-hub/content/docs/price-feeds/core/migrate-an-app-to-pyth/chainlink.mdx` instructs users to deploy it and pass its address directly to their existing Chainlink-consuming app.
- Protocols migrating from Chainlink are likely to retain their existing `latestRoundData()` call pattern without adding Pyth-specific staleness guards, because the adapter is presented as a transparent drop-in.
- Pyth is a pull oracle: if no keeper calls `updateFeeds()`, the on-chain price does not update. During any network disruption or keeper failure, the stale price window opens immediately.
- An attacker needs only to wait for a staleness window and submit a transaction to the victim protocol — no privileged access required.

---

### Recommendation

Replace `getPriceUnsafe` with `getPriceNoOlderThan` using a configurable `maxAge` parameter, and add a `price > 0` guard:

```solidity
uint256 public maxAge; // configurable staleness threshold, e.g. 60 seconds

function latestRoundData() external view returns (
    uint80 roundId, int256 answer,
    uint256 startedAt, uint256 updatedAt, uint80 answeredInRound
) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    require(price.price > 0, "Pyth: non-positive price");
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

Apply the same fix to `latestAnswer()`, `getRoundData()`, and `decimals()`.

---

### Proof of Concept

1. A lending protocol deploys `PythAggregatorV3` for ETH/USD following the official migration guide.
2. The protocol's collateral valuation calls `latestRoundData()` and uses the returned `answer` as the ETH price.
3. The Pyth keeper stops updating the feed (network outage, keeper bug, or deliberate griefing by withholding updates).
4. The on-chain ETH price in the Pyth contract is now 1 hour old. The real ETH price has dropped 20%.
5. An attacker calls the lending protocol's borrow function. `latestRoundData()` returns the stale (inflated) ETH price — no revert occurs because `getPriceUnsafe` has no staleness check.
6. The attacker borrows against overvalued collateral, extracting funds the protocol cannot recover once the price corrects.
