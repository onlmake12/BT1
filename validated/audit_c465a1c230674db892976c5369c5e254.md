### Title
Predictable On-Chain PRNG in `random()` Allows Provider to Pre-Compute and Manipulate Entropy Outcomes - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The internal `random()` function in `Entropy.sol` generates the user's entropy contribution using only publicly known or derivable on-chain values. Three no-argument `requestV2` overloads call this function as a substitute for a user-supplied secret. Because every input to `random()` is observable or computable by the provider before they reveal their hash-chain value, the 2-party security guarantee ("random as long as either party is honest") is broken for all callers of these convenience overloads.

---

### Finding Description

`random()` at line 1079 of `Entropy.sol`:

```solidity
function random() internal returns (bytes32) {
    _state.seed = keccak256(
        abi.encodePacked(
            block.timestamp,
            block.prevrandao,
            msg.sender,
            _state.seed
        )
    );
    return _state.seed;
}
```

Every input is attacker-observable:

| Input | Why it is known |
|---|---|
| `block.timestamp` | Set by the block proposer; known once the block is mined |
| `block.prevrandao` | The RANDAO reveal from the **previous** block; already committed and public before the current block is produced |
| `msg.sender` | The caller's address; visible in the mempool |
| `_state.seed` | Private Solidity field, but all EVM storage is readable via `eth_getStorageAt` |

This return value is used directly as `userContribution` in three public entry points:

```solidity
// line 292
assignedSequenceNumber = requestV2(getDefaultProvider(), random(), 0);

// line 300
assignedSequenceNumber = requestV2(getDefaultProvider(), random(), gasLimit);

// line 309
assignedSequenceNumber = requestV2(provider, random(), gasLimit);
```

Each of these calls `requestV2(address, bytes32, uint32)`, which stores `constructUserCommitment(userContribution)` and **emits `userContribution` in plaintext** in the `RequestedWithCallback` event (line 378). The inline comment at line 366–368 asserts:

> "If we remove the blockHash from this, the provider would have no choice but to provide its committed random number."

This assertion is only valid when `userContribution` is a secret the provider cannot know before they reveal. With `random()`, the provider can compute `userContribution` from public state, invalidating the comment's security claim.

---

### Impact Explanation

A malicious provider can:

1. Read `_state.seed` from contract storage at any time.
2. Observe a pending `requestV2()` transaction in the mempool (learning `msg.sender`).
3. Compute `userContribution = keccak256(block.timestamp, block.prevrandao, msg.sender, seed)` for the expected block (all inputs are known or bounded).
4. Compute the final random number `r = keccak256(userContribution, x_i, 0)` where `x_i` is the provider's current hash-chain value (which the provider already knows).
5. If `r` is unfavorable, call `register()` to rotate their commitment **before** the user's transaction is included, shifting which `x_i` will be used and thus changing `r`.

The result is that the provider can steer the random number to any value they choose for users of the no-arg `requestV2` overloads. Downstream consumers (NFT mints, games, lotteries) built on these overloads receive a manipulated result.

---

### Likelihood Explanation

The attack requires a registered provider who:
- Monitors the mempool (standard infrastructure for any active provider).
- Can call `register()` to rotate their commitment (a permissionless, always-available operation).

No validator collusion, key leak, or governance action is needed. Any provider who wishes to manipulate outcomes for users of the convenience overloads can do so with only public on-chain data and a mempool subscription. Likelihood is **medium-high** for any provider with a financial incentive tied to the random outcome.

---

### Recommendation

1. **Remove the no-arg `requestV2` overloads** or require callers to supply their own `userContribution`. The protocol already provides `requestV2(address, bytes32, uint32)` for users who generate their own secret off-chain.
2. If convenience overloads are retained, add explicit NatSpec warnings that they rely on an on-chain PRNG and do **not** provide the 2-party security guarantee.
3. If an on-chain fallback is truly needed, use a VRF or commit-reveal scheme that does not expose all entropy inputs before the provider's reveal step.

### Proof of Concept

```solidity
// Attacker is a registered provider.
// Step 1: read current seed from storage slot of _state.seed
bytes32 currentSeed = /* eth_getStorageAt(entropyAddr, seedSlot) */;

// Step 2: observe pending requestV2() tx from victim address `victim`
// Step 3: compute predicted userContribution
bytes32 predictedUserContribution = keccak256(
    abi.encodePacked(
        expectedBlockTimestamp,   // known: prevrandao already committed
        block.prevrandao,         // already public
        victim,                   // from mempool
        currentSeed               // from storage
    )
);

// Step 4: compute final random number using provider's own x_i
bytes32 predictedR = keccak256(
    abi.encodePacked(predictedUserContribution, myX_i, bytes32(0))
);

// Step 5: if predictedR is unfavorable, rotate commitment before victim's tx lands
if (predictedR != desiredOutcome) {
    entropy.register(fee, newCommitment, metadata, chainLength, uri);
    // now a different x_i will be used, changing the outcome
}
```

The `random()` function is the necessary vulnerable step: it is the only reason `userContribution` is computable from public state. Users calling `requestV2(address, bytes32, uint32)` with a self-generated secret are not affected. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
