### Title
Unaccounted I/O Overhead from `MAX_DEP_EXPANSION_LIMIT = 2048` Dep Group Expansion Enables Node DoS — (`util/types/src/core/cell.rs`)

### Summary
CKB's `MAX_DEP_EXPANSION_LIMIT = 2048` allows a single transaction to force up to 2048 database reads during cell-dep resolution. This work is entirely outside the cycle-limit accounting system. An unprivileged transaction submitter can craft transactions whose dep groups expand to the maximum 2048 entries, imposing disproportionate I/O load on every validating node during both tx-pool admission and block validation, with no protocol-level bound on that cost.

### Finding Description
In `util/types/src/core/cell.rs`, the constant `MAX_DEP_EXPANSION_LIMIT` is set to 2048. [1](#0-0) 

During transaction resolution, `resolve_transaction_deps_with_system_cell_cache` initialises a counter at this limit and decrements it for every expanded dep entry. For each `DepGroup`-typed cell dep, `resolve_transaction_dep` parses the group's data into a list of sub-out-points and then calls `cell_resolver` (a DB read) for every sub-out-point before the counter is checked: [2](#0-1) 

The counter check (`checked_sub`) only prevents accepting a transaction that exceeds 2048 total expanded deps; it does not prevent the node from performing up to 2048 individual DB reads for a transaction that sits exactly at the limit. The same resolution path is executed:

1. **During tx-pool admission** — every transaction submitted via RPC or relayed by a peer is resolved before entering the pool.
2. **During block validation** — `resolve_transaction` is called for every committed transaction in a block. [3](#0-2) 

The CKB cycle limit (`max_block_cycles`) accounts only for script-execution cycles, not for the I/O cost of dep resolution. There is no protocol parameter that bounds the number of DB reads a single transaction may impose during resolution.

The integration test suite explicitly confirms that a transaction with exactly 2048 expanded deps is accepted (`PASS`) while 2049 is rejected: [4](#0-3) 

### Impact Explanation
An attacker who creates a dep-group cell whose data encodes 2048 out-points (2048 × 36 = 73,728 bytes of cell data, feasible within cell capacity rules) can submit many transactions that each reference this dep group. Each submission forces the receiving node to perform 2048 DB reads during `resolve_transaction`. Because the tx-pool admission path is synchronous with respect to resolution, a sustained stream of such transactions saturates the node's I/O and delays or blocks legitimate transaction processing and block validation. A block filled with transactions that each expand to 2048 deps multiplies this overhead across every peer that validates the block, with no cycle-limit protection.

Impact: **service degradation / denial of service** on tx-pool admission and block validation for all full nodes that receive the crafted transactions or blocks.

### Likelihood Explanation
The attack requires no privileged access. Any RPC caller or P2P relay peer can submit transactions. The only cost to the attacker is the CKB capacity locked in the dep-group cell and the transaction fees, both of which are small relative to the I/O burden imposed. The dep-group cell can be reused across an unlimited number of attacking transactions, making the attack cheap to sustain.

### Recommendation
1. **Reduce `MAX_DEP_EXPANSION_LIMIT`** to a value whose worst-case DB-read overhead is acceptable (e.g., 64 or 128), consistent with the approach recommended in the referenced Velodrome/Spearbit report for `MAX_DELEGATES`.
2. **Account for dep-expansion cost in the fee or cycle model** — introduce a per-expanded-dep fee surcharge or a synthetic "resolution cycle" cost so that transactions with large dep groups pay proportionally more, making the attack economically unattractive.
3. **Cache resolved deps across transactions within the same block** — a block-scoped dep-resolution cache would amortise repeated reads of the same dep-group cells across multiple transactions in one block, reducing the per-transaction I/O amplification.

### Proof of Concept
**Attacker-controlled entry path:**

1. Attacker calls `ckb_send_transaction` (RPC) or relays via P2P to submit a transaction `T` whose `cell_deps` field contains one entry of type `DepGroup` pointing to a live cell whose data encodes exactly 2048 distinct out-points.
2. The node's tx-pool service calls `resolve_transaction` → `resolve_transaction_deps_with_system_cell_cache` → `resolve_transaction_dep`.
3. Inside `resolve_transaction_dep`, `parse_dep_group_data` deserialises the 2048 out-points, then the loop at line 829 issues 2048 `cell_resolver` calls, each hitting RocksDB.
4. `remaining_dep_slots` reaches 0 exactly (2048 − 2048 = 0), so `checked_sub` succeeds and the transaction is accepted into the pool.
5. Attacker repeats step 1 with different inputs but the same dep-group cell, generating a continuous stream of 2048-DB-read resolutions.
6. A miner that includes N such transactions in a block forces every validating node to perform up to N × 2048 DB reads outside any cycle-limit protection. [5](#0-4) [6](#0-5)

fn resolve_transaction_dep<F: FnMut(&OutPoint, bool) -> Result<CellMeta, OutPointError>>(
    cell_dep: &CellDep,
    cell_resolver: &mut F,
    resolved_cell_deps: &mut Vec<CellMeta>,
    resolved_dep_groups: &mut Vec<CellMeta>,
    eager_load: bool,
    remaining_dep_slots: &mut usize,
) -> Result<(), OutPointError> {
    if cell_dep.dep_type() == DepType::DepGroup.into() {
        let outpoint = cell_dep.out_point();
        let dep_group = cell_resolver(&outpoint, true)?;
        let data = dep_group
            .mem_cell_data
            .as_ref()
            .expect("Load cell meta must with data");
        let sub_out_points =
            parse_dep_group_data(data).map_err(|_| OutPointError::InvalidDepGroup(outpoint))?;

        *remaining_dep_slots = remaining_dep_slots
            .checked_sub(sub_out_points.len())
            .ok_or(OutPointError::OverMaxDepExpansionLimit)?;

        for sub_out_point in sub_out_points.into_iter() {
            resolved_cell_deps.push(cell_resolver(&sub_out_point, eager_load)?);
        }
        resolved_dep_groups.push(dep_group);
    } else {
        *remaining_dep_slots = remaining_dep_slots
            .checked_sub(1)
            .ok_or(OutPointError::OverMaxDepExpansionLimit)?;

        resolved_cell_deps.push(cell_resolver(&cell_dep.out_point(), eager_load)?);
    }
    Ok(())
}
