### Title
Silent u128→u64 Truncation in DAO Maximum Withdraw Calculation Produces Wrong Capacity — (`File: util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes an intermediate u128 result and casts it to u64 with a bare `as u64` truncating cast. When the intermediate value exceeds `u64::MAX`, the result is silently wrapped to a much smaller number instead of returning `DaoError::Overflow`. Every other analogous u128→u64 conversion in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern. The inconsistency means the function can return a silently wrong (too-small) capacity, causing downstream `safe_sub` calls to underflow and reject valid DAO withdrawal transactions, or corrupt the DAO field written into a new block.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← bare truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently wraps any value above `u64::MAX` to `value % 2^64`. Every other u128→u64 narrowing in the same file uses the checked form:

```rust
// secondary_block_reward
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch – miner_issuance
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)

// dao_field_with_current_epoch – ar_increase
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The existing unit test `check_withdraw_calculation_overflows` (lines 295–350) constructs a cell with capacity `18_446_744_073_709_550_000` shannons and expects `result.is_err()`. With the truncating cast the function instead returns `Ok(wrong_small_value)`, so the test assertion `assert!(result.is_err())` fails — confirming the defect is present and untested-against. [5](#0-4) 

`calculate_maximum_withdraw` is called from two production paths:

1. **`transaction_maximum_withdraw` → `transaction_fee`** — used by the tx-pool admission check (`check_tx_fee` in `tx-pool/src/util.rs`) and by `FeeCalculator::transaction_fee` inside `ContextualTransactionVerifier::verify`. A silently truncated (too-small) `maximum_withdraw` causes `maximum_withdraw.safe_sub(outputs_capacity)` to underflow and return `DaoError::Overflow`, which is mapped to `Reject::Malformed` — permanently rejecting a valid DAO withdrawal transaction. [6](#0-5) [7](#0-6) 

2. **`withdrawed_interests` → `dao_field_with_current_epoch`** — used during block assembly and block verification. A wrong `maximum_withdraws` value causes `maximum_withdraws.safe_sub(input_capacities)` to underflow, corrupting the DAO field or causing block validation to fail. [8](#0-7) [9](#0-8) 

3. **`calculate_dao_maximum_withdraw` RPC** — returns the silently wrong value to any RPC caller, misleading wallets about the correct withdrawal amount. [10](#0-9) 

### Impact Explanation

A DAO depositor whose cell's `counted_capacity * withdrawing_ar / deposit_ar` overflows u64 will have their valid withdrawal transaction permanently rejected by the node's tx-pool and block verifier. The node returns `Malformed … expect (outputs capacity) <= (inputs capacity)` even though the transaction is protocol-correct. The user cannot recover their principal plus interest. Additionally, if such a transaction were somehow included in a block (e.g., by a miner that bypasses the check), the block verifier would compute a wrong DAO field, potentially causing a consensus split.

### Likelihood Explanation

The overflow requires `counted_capacity * withdrawing_ar > u64::MAX * deposit_ar`. Because `withdrawing_ar / deposit_ar` is always slightly above 1 (it encodes the DAO interest rate, which is a small annual percentage), `counted_capacity` itself must be close to `u64::MAX` shannons (~184 billion CKB) to trigger the overflow. The total CKB issuance is ~33.6 billion CKB, so no single cell on mainnet today can hold enough capacity to trigger this. However, as the accumulation rate `ar` grows over decades and if large cells are consolidated, the margin shrinks. The defect is also a correctness regression relative to the rest of the codebase and causes the existing unit test to silently pass with a wrong result.

### Recommendation

Replace the bare truncating cast with the same checked pattern used everywhere else in the file:

```rust
// Before (line 155-156):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After:
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
``` [11](#0-10) 

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` already demonstrates the defect. With the current code the test assertion `assert!(result.is_err())` fails because the function returns `Ok(truncated_wrong_value)` instead of `Err(DaoError::Overflow)`:

```
output.capacity  = 18_446_744_073_709_550_000 shannons
occupied_capacity ≈ 0  (no lock/type script in test)
counted_capacity  = 18_446_744_073_709_550_000

withdraw_counted_capacity (u128)
  = 18_446_744_073_709_550_000 * 10_000_000_001_123_456
    / 10_000_000_000_123_456
  ≈ 20_291_418_477_450_505_000   ← exceeds u64::MAX (18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 20_291_418_477_450_505_000 % 2^64
  ≈ 1_844_674_403_740_953_384   ← silently truncated, no error

safe_add(0) succeeds → Ok(Capacity::shannons(1_844_674_403_740_953_384))
``` [5](#0-4) 

A transaction sender submitting a DAO withdrawal with such a cell would have their transaction rejected by `check_tx_fee` with `Reject::Malformed` because `maximum_withdraw (truncated) < outputs_capacity`, even though the transaction is valid. [12](#0-11)

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
