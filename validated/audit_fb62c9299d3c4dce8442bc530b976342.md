### Title
Signed Integer Division Truncates Towards Zero in TWAP Calculation, Producing Inflated Prices for Negative-Price Assets — (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol` and `target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

Both the EVM and Solana TWAP calculation implementations divide a signed `int128`/`i128` `priceDiff` by an unsigned slot count using native integer division, which in both Solidity and Rust truncates towards zero. When the cumulative price difference is negative (i.e., the asset's average price over the window is negative), the result is rounded towards zero — producing a TWAP that is **higher** than the true mathematical average. Downstream protocols consuming this TWAP may make incorrect decisions (e.g., delayed liquidations, incorrect collateral valuations) when the true TWAP is negative.

---

### Finding Description

In `calculateTwap` in `Pyth.sol`:

```solidity
int128 priceDiff = twapPriceInfoEnd.cumulativePrice -
    twapPriceInfoStart.cumulativePrice;
// ...
int128 twapPrice = priceDiff / int128(uint128(slotDiff));
```

`priceDiff` is `int128` (signed). When it is negative and not exactly divisible by `slotDiff`, Solidity's `/` operator truncates towards zero. For example, `-7 / 2 = -3` in Solidity, whereas the mathematically correct floor is `-4`.

The same pattern exists in the Solana receiver:

```rust
let price = i64::try_from(price_diff / i128::from(slot_diff))
    .map_err(|_| ReceiverError::TwapCalculationOverflow)?;
```

Rust's integer division also truncates towards zero for signed types.

`cumulativePrice` is defined as `int128` in `PythStructs.TwapPriceInfo` and `i128` in `TwapMessage`, explicitly supporting negative values. Pyth documentation acknowledges that prices can be negative (e.g., for certain derivative products). When the average price over a TWAP window is negative and non-integer in the price unit, the reported TWAP is off by up to +1 unit compared to the true floor-rounded value.

---

### Impact Explanation

Any protocol consuming `parseTwapPriceFeedUpdates` (EVM) or `post_twap_update` (Solana) for a negative-price feed receives a TWAP that is at most 1 price unit higher than the true time-weighted average. For a feed with `expo = -8`, this is a $0.00000001 error per unit; for `expo = -2` (e.g., a 2-decimal stablecoin-denominated derivative), the error is $0.01. Protocols using the TWAP as a threshold check (e.g., "liquidate if TWAP < X") may fail to trigger when the true TWAP is just below the threshold but the reported TWAP rounds up to meet or exceed it. The error is bounded at 1 unit and cannot be amplified by an attacker, but it is systematic and deterministic.

---

### Likelihood Explanation

Negative Pyth prices are rare but explicitly supported by the protocol. Any feed for a derivative product (e.g., a basis swap, funding rate, or spread) can have a negative price. The rounding error occurs on every TWAP calculation where `priceDiff % slotDiff != 0` and `priceDiff < 0`. No attacker action is required — the error is inherent to the calculation. Any unprivileged caller submitting valid Wormhole-attested TWAP update pairs for a negative-price feed will receive a biased result.

---

### Recommendation

Apply floor division for negative `priceDiff`. In Solidity:

```solidity
int128 twapPrice;
if (priceDiff < 0 && priceDiff % int128(uint128(slotDiff)) != 0) {
    twapPrice = priceDiff / int128(uint128(slotDiff)) - 1;
} else {
    twapPrice = priceDiff / int128(uint128(slotDiff));
}
```

In Rust, use `i128::div_floor` (stable since Rust 1.73) or an equivalent manual implementation:

```rust
fn div_floor(a: i128, b: i128) -> i128 {
    let d = a / b;
    let r = a % b;
    if (r != 0) && ((r < 0) != (b < 0)) { d - 1 } else { d }
}
let price = i64::try_from(div_floor(price_diff, i128::from(slot_diff)))
    .map_err(|_| ReceiverError::TwapCalculationOverflow)?;
```

Add unit tests covering negative `priceDiff` with non-zero remainder.

### Proof of Concept

**EVM:**
- `cumulativePrice_start = 0`, `cumulativePrice_end = -7`, `slotDiff = 2`
- `priceDiff = -7`
- Solidity: `-7 / 2 = -3` (truncates towards zero)
- Correct floor: `-4`
- Reported TWAP: `-3` instead of `-4` → 1 unit inflated

**Solana:**
- Same arithmetic: `price_diff = -7i128`, `slot_diff = 2u64`
- Rust: `-7i128 / 2i128 = -3` (truncates towards zero)
- Correct floor: `-4`

A protocol with a liquidation threshold at TWAP ≤ -4 would not liquidate, even though the true TWAP is -3.5 (floor -4), because the reported TWAP is -3. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
