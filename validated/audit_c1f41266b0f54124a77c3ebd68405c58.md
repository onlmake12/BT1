Audit Report

## Title
Uncharged O(N) `header_deps` Linear Scan in `LoadHeader::load_header` Enables Cycle-Budget Bypass DoS â€” (`script/src/syscalls/load_header.rs`)

## Summary
`LoadHeader::load_header` performs a full linear scan over all `header_deps` on every invocation for `Source::Input` and `Source::CellDep` paths, but charges cycles only for the bytes written to VM memory. A script that calls `ckb_load_header` or `ckb_load_header_by_field` in a tight loop against a transaction carrying many `header_deps` can consume arbitrarily more CPU time than the cycle budget implies, breaking the invariant that `max_block_cycles` bounds verification wall time and enabling effective DoS against any node that processes the transaction.

## Finding Description
In `script/src/syscalls/load_header.rs`, the `load_header` method (lines 56â€“70) performs a full linear scan over the transaction's `header_deps` list via `.any()` before fetching the header:

```rust
fn load_header(&self, cell_meta: &CellMeta) -> Option<HeaderView> {
    let block_hash = &cell_meta.transaction_info.as_ref()?.block_hash;
    if self
        .header_deps()
        .into_iter()
        .any(|hash| &hash == block_hash)
    {
        self.sg_data.tx_info.data_loader.get_header(block_hash)
    } else {
        None
    }
}
```

This path is triggered for `Source::Transaction(Input)`, `Source::Transaction(CellDep)`, and `Source::Group(Input)` in `fetch_header` (lines 74â€“105). The only cycle charge after the scan is at line 175:

```rust
machine.add_cycles_no_checking(transferred_byte_cycles(len))?;
```

This charges only for bytes written to VM memory â€” 2 cycles for `LOAD_HEADER_BY_FIELD` (8 bytes / 4 = 2) or ~52 cycles for `LOAD_HEADER` (~208 bytes / 4). The O(N) scan work over `header_deps` is never charged.

The `DuplicateDepsVerifier` (lines 437â€“458 of `verification/src/transaction_verifier.rs`) prevents duplicate `header_deps` but imposes no count limit. With `MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES * TWO_IN_TWO_OUT_COUNT = 597 * 1000 = 597,000 bytes` and each `header_dep` being 32 bytes, a single transaction can carry up to ~18,000 distinct `header_deps`. All must reference valid canonical-chain blocks (`HeaderChecker::check_valid`), which is trivially satisfiable on mainnet with millions of blocks.

At ~20 cycles per `LOAD_HEADER_BY_FIELD` call and `MAX_BLOCK_CYCLES = 3,500,000,000`, a script can invoke the syscall ~175 million times. Each invocation triggers an ~18,000-element Ă— 32-byte scan. The total comparison work is orders of magnitude beyond what the cycle budget implies, completely decoupling actual CPU time from the cycle model.

The same pattern exists identically in `load_block_extension.rs` (lines 51â€“59), which has the same uncharged linear scan over `header_deps`.

## Impact Explanation
This matches the **High** impact category: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* During script verification â€” both at tx-pool admission and block verification â€” a node executes CKB-VM for each script group. A single crafted transaction causes the verifier to spend orders of magnitude more CPU time than `max_block_cycles` implies. Any node that processes the transaction (or a block containing it) is subject to severe slowdown. Since block verification is on the critical path for consensus, this can stall block propagation and cause network-wide congestion. The attacker's cost is minimal: one transaction with a small script and many `header_deps`.

## Likelihood Explanation
The attack is fully unprivileged. The attacker needs only: (1) N distinct canonical-chain block hashes as `header_deps` (freely available on mainnet), (2) a deployed minimal looping script (~50 bytes of RISC-V), and (3) one live UTXO to spend. No special keys, hashpower, or privileged access are required. The transaction can be submitted via the standard `send_transaction` RPC or P2P relay. The attack is repeatable and cheap.

## Recommendation
Charge cycles proportional to the number of `header_deps` scanned on each `load_header` invocation. Before or during the scan, add a base cost of `header_deps().len()` cycles (one cycle per element checked). Alternatively, pre-build a `HashSet<Byte32>` from `header_deps` once per script group execution and use O(1) lookups, eliminating the scan entirely. The same fix must be applied to `load_block_extension.rs` which has the identical pattern.

## Proof of Concept
```
Transaction layout:
  header_deps: [h1, h2, ..., h_N]   (N = ~10,000 distinct canonical block hashes)
  inputs: [one live UTXO]

  witnesses: [...]

Script (RISC-V, ~50 bytes):
  loop:
    li a3, 0                  // index = 0 (first input)

        if let Some(dep) = transaction

            return Err(TransactionError::DuplicateCellDeps {
                out_point: dep.out_point(),
            }
            .into());
        }
        if let Some(hash) = transaction
            .header_deps_iter()
            .find_map(|hash| seen_headers.replace(hash))
        {
            return Err(TransactionError::DuplicateHeaderDeps { hash }.into());
        }
        Ok(())
    }
