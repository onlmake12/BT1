## Vulnerability Assessment

The call chain is concrete and fully traceable. This is a **real vulnerability**.

---

### Title
Tx-pool write lock held during blocking RocksDB `drop_cf`/`create_cf_with_ttl` in `RecentReject::shrink`, enabling remote-peer-triggered lock starvation — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

An unprivileged remote peer can submit a high volume of distinct transactions that are cheaply rejected (e.g., `LowFeeRate` with fee=0). Each rejection triggers `put_recent_reject`, which acquires the tokio `tx_pool` write lock and calls `RecentReject::put`. Once `total_keys_num > count_limit`, `shrink()` fires, executing synchronous blocking RocksDB `drop_cf` + `create_cf_with_ttl` while the write lock is held. This blocks the tokio runtime thread and starves all concurrent operations that need the write lock: tx admission, block assembly, and RBF checks.

---

### Finding Description

**Step 1 — Attacker entry point:**

A remote peer submits transactions via P2P relay. These reach `process_tx`: [1](#0-0) 

`process_tx` calls `after_process` after the submission attempt.

**Step 2 — `should_recorded()` includes `LowFeeRate`:**

`should_recorded()` returns `true` for every reject variant except `Duplicated`: [2](#0-1) 

A fee=0 transaction produces `Reject::LowFeeRate`, which passes this check. Crucially, `is_malformed_tx()` is false for `LowFeeRate`, so the peer is **not banned**.

**Step 3 — `after_process` calls `put_recent_reject`:** [3](#0-2) 

**Step 4 — `put_recent_reject` acquires the write lock and calls `RecentReject::put`:** [4](#0-3) 

The tokio `RwLock` write guard is held for the entire duration of the call.

**Step 5 — `RecentReject::put` may call `shrink()`:** [5](#0-4) 

**Step 6 — `shrink()` executes blocking RocksDB operations while the write lock is held:** [6](#0-5) 

`drop_cf` and `create_cf_with_ttl` are synchronous, blocking RocksDB column-family management calls: [7](#0-6) 

Neither uses `spawn_blocking` nor `block_in_place` (despite `service.rs` importing `block_in_place` at line 46 for other uses). The tokio runtime thread is blocked for the full duration of the RocksDB operation.

**Step 7 — The write lock is the global tx-pool lock:**

`with_tx_pool_write_lock` is used for block assembly, RBF checks, and tx admission: [8](#0-7) 

All of these are blocked while `shrink()` runs.

---

### Impact Explanation

- **Tx admission starvation**: No new transactions can be admitted while the write lock is held during `shrink()`.
- **Block assembly blocked**: Block template generation requires the write lock and is delayed.
- **RBF checks blocked**: `check_rbf` is explicitly noted to require the write lock to avoid concurrent issues.
- **Tokio thread blocking**: Synchronous RocksDB I/O inside an async context blocks the tokio worker thread, degrading the entire async runtime.
- **Repeatable**: The attacker can continuously trigger `shrink()` by keeping `total_keys_num` near `count_limit` with a steady stream of fee=0 transactions, causing repeated lock-hold spikes.

---

### Likelihood Explanation

- No privileges required — standard P2P transaction relay.
- Fee=0 transactions are structurally valid and pass non-contextual verification.
- `LowFeeRate` does not trigger peer banning (`is_malformed_tx()` = false).
- `should_recorded()` = true for `LowFeeRate`, so every such rejection feeds `recent_reject`.
- The attacker only needs to exceed `count_limit` (a finite, configurable threshold) with distinct tx hashes, which is cheap to do with fee=0 transactions.

---

### Recommendation

1. Move `shrink()` out of the write-lock critical section: record the reject first (releasing the lock), then perform the shrink asynchronously or via `spawn_blocking`.
2. Wrap `drop_cf` and `create_cf_with_ttl` in `tokio::task::block_in_place` or `spawn_blocking` to avoid blocking the async runtime thread.
3. Apply per-peer rate limiting on transaction submissions to bound the rate at which `recent_reject` can be filled.
4. Consider not recording `LowFeeRate` rejections (or rate-limiting their recording), since they are the cheapest rejection to trigger at scale.

---

### Proof of Concept

```
1. Connect to a CKB node as a P2P peer.
2. Construct N distinct transactions each with fee=0 (outputs consume all capacity, no fee left).
3. Submit them via the RelayTransactions P2P message.
4. Each tx passes non_contextual_verify, fails at fee-rate check → Reject::LowFeeRate.
5. after_process → put_recent_reject → tx_pool.write() → recent_reject.put().
6. Once total_keys_num > count_limit, shrink() fires: drop_cf + create_cf_with_ttl execute
   synchronously while the write lock is held.
7. Benchmark: measure tx_pool write lock hold time when shrink() fires vs. normal path.
   Expected: lock hold time spikes by the duration of RocksDB CF drop/create (tens to
   hundreds of milliseconds), during which block assembly and tx admission are fully stalled.
```

        &self,
        tx: TransactionView,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<Completed, Reject> {
        // non contextual verify first
        self.non_contextual_verify(&tx, remote).await?;

        if self.verify_queue_contains(&tx).await || self.orphan_contains(&tx).await {
            return Err(Reject::Duplicated(tx.hash()));
        }

        if let Some((ret, snapshot)) = self
            ._process_tx(tx.clone(), remote.map(|r| r.0), None)
            .await
        {
            self.after_process(tx, remote, &snapshot, &ret).await;
            ret
        } else {
            // currently, the returned cycles is not been used, mock 0 if delay
            Ok(Completed {
                cycles: 0,
                fee: Capacity::zero(),
            })
        }
    }
