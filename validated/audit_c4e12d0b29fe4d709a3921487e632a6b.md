### Title
Expired Alert Replay via P2P: `clear_expired_alerts` Removes Dedup Guard Without Blocking Re-admission - (`util/network-alert/src/notifier.rs`)

### Summary

The `Notifier::clear_expired_alerts` function removes expired alerts from `received_alerts` and `noticed_alerts` but does not add their IDs to `cancel_filter`. After expiry, `has_received(alert_id)` returns `false`, and the P2P `AlertRelayer::received` handler has no `notice_until` check of its own. Any unprivileged P2P peer holding the original signed alert bytes can replay them after expiry, causing the node to re-accept, re-notify all subscribers (including executing `network_alert_notify_script`), and re-broadcast the expired alert to the entire network.

### Finding Description

**Root cause — `clear_expired_alerts` does not tombstone expired IDs:** [1](#0-0) 

`clear_expired_alerts` removes expired entries from `received_alerts` and `noticed_alerts` but never calls `self.cancel_filter.put(id, ())`. After this runs, `has_received(id)` consults both maps: [2](#0-1) 

Because the expired alert is gone from `received_alerts` and was never added to `cancel_filter`, `has_received` returns `false` for the now-expired ID.

**P2P received handler has no `notice_until` guard:** [3](#0-2) 

The handler checks `has_received` (which now returns `false`) and then calls `verifier.verify_signatures`. The original signatures are cryptographically valid — they cover the `RawAlert` hash including the past `notice_until` value — so verification passes. There is no check that `notice_until` is in the future, unlike the RPC path: [4](#0-3) 

**Re-admission fires notification and re-broadcasts:**

After passing both checks, `notifier.add()` is called: [5](#0-4) 

`received_alerts.insert` re-stores the alert, `notify_controller.notify_network_alert` fires again (triggering all subscribers and the optional `network_alert_notify_script`), and the alert is pushed back into `noticed_alerts`. The relayer then re-broadcasts to all connected peers: [6](#0-5) 

**`clear_expired_alerts` is called only on peer connect and `get_blockchain_info`:** [7](#0-6) [8](#0-7) 

This means the replay window opens every time expiry cleanup runs, and the attacker can immediately replay before the next cleanup cycle.

### Impact Explanation

1. **Repeated notification execution**: `notify_controller.notify_network_alert` fires for every replay, delivering the expired alert to all RPC subscribers and executing `network_alert_notify_script` (if configured). A script that performs expensive operations (e.g., external HTTP calls, restarts) can be triggered repeatedly, constituting a DoS against the node operator.
2. **Network-wide re-broadcast**: Each replaying peer causes the expired alert to propagate to all connected peers, which in turn re-accept it (same `has_received` gap), creating a network-wide spam amplification loop until all nodes independently run `clear_expired_alerts` again.
3. **Integrity violation**: Operators and monitoring systems receive spurious "active alert" signals for alerts that have already expired and been acted upon, undermining the integrity of the alert system.

### Likelihood Explanation

- **Attacker preconditions**: Only requires being a connected P2P peer (no keys, no privilege). The alert bytes are public — any node that received the original alert stores them in `received_alerts` and re-sends them to newly connected peers via `connected()`.
- **Trigger**: Trivially reproducible: connect to any CKB node, wait for an alert to expire, send the stored alert bytes over the Alert protocol.
- **Persistence**: The replay can be repeated indefinitely as long as the attacker reconnects or sends the bytes before the next `clear_expired_alerts` call re-adds the alert (which itself gets cleared again on the next cycle, reopening the window).

### Recommendation

In `clear_expired_alerts`, tombstone each expired alert ID into `cancel_filter` before removing it from `received_alerts`:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    let expired_ids: Vec<u32> = self.received_alerts
        .iter()
        .filter_map(|(id, alert)| {
            let notice_until: u64 = alert.raw().notice_until().into();
            if notice_until <= now { Some(*id) } else { None }
        })
        .collect();
    for id in expired_ids {
        self.cancel_filter.put(id, ());  // tombstone so has_received stays true
        self.received_alerts.remove(&id);
    }
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

Additionally, add a `notice_until` check in the P2P `received()` handler (mirroring the RPC guard) as defense-in-depth:

```rust
let notice_until: u64 = alert.as_reader().raw().notice_until().into();
let now_ms = ckb_systemtime::unix_time_as_millis();
if notice_until < now_ms {
    // drop silently or ban peer
    return;
}
```

### Proof of Concept

1. Node A receives a valid alert (ID=42, `notice_until = T`). It is stored in `received_alerts`.
2. Time advances past `T`. `clear_expired_alerts` is triggered (e.g., via `get_blockchain_info` RPC). Alert ID=42 is removed from `received_alerts`. `cancel_filter` is unchanged. `has_received(42)` now returns `false`.
3. Attacker (any P2P peer) sends the original alert bytes (obtained from any node before expiry, or from the P2P gossip during the alert's active period) to Node A over the Alert protocol.
4. `AlertRelayer::received()`: `has_received(42)` → `false`; `verify_signatures` → `Ok` (signatures are still cryptographically valid); `notifier.add()` is called.
5. `notify_controller.notify_network_alert` fires; `network_alert_notify_script` executes; alert is re-broadcast to all of Node A's peers.
6. Each peer that had also cleared the alert repeats steps 4–5, propagating the replay network-wide.
7. Repeat from step 3 indefinitely. [9](#0-8) [10](#0-9)

    pub fn add(&mut self, alert: &Alert) {
        let alert_id = alert.raw().id().into();
        let alert_cancel = alert.raw().cancel().into();
        if self.has_received(alert_id) {
            return;
        }
        // checkout cancel_id
        if alert_cancel > 0 {
            self.cancel(alert_cancel);
        }
        // add to received alerts
        self.received_alerts.insert(alert_id, alert.clone());

        // check conditions, figure out do we need to notice this alert
        if !self.is_version_effective(alert) {
            debug!("Received a version ineffective alert {:?}", alert);
            return;
        }

        if self.noticed_alerts.contains(alert) {
            return;

        self.noticed_alerts.push(alert.clone());
        // sort by priority
        self.noticed_alerts.sort_by_key(|a| {
            let priority: u32 = a.raw().priority().into();
            u32::MAX - priority
        });
    }
