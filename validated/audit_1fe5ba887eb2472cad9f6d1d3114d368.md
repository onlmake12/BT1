### Title
Zero `ban_time` with `absolute=true` Silently Creates an Immediately-Expired Ban Entry - (File: `rpc/src/module/net.rs`)

### Summary

The `set_ban` RPC handler accepts `ban_time = 0` (or `ban_time = null`) combined with `absolute = true` without any validation. This causes `ban_until` to be set to `0`, which is always in the past, so the resulting ban entry is immediately expired and the targeted IP is never actually banned. The node operator receives a success response but the ban is silently ineffective.

### Finding Description

In `rpc/src/module/net.rs`, the `set_ban` implementation computes `ban_until` as follows:

```rust
let ban_until = if absolute.unwrap_or(false) {
    ban_time.unwrap_or_default().into()   // line 708: no zero-check
} else {
    unix_time_as_millis()
        + ban_time
            .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
            .value()
};
self.network_controller
    .ban(ip_network, ban_until, reason.unwrap_or_default());
```

When `absolute = true` and `ban_time` is `0` or `null`, `ban_time.unwrap_or_default()` resolves to `Timestamp(0)`, so `ban_until = 0`. There is no check that `ban_until` is in the future before the entry is inserted.

The ban check in `network/src/peer_store/ban_list.rs` uses a strict greater-than comparison:

```rust
fn is_ip_banned_until(&self, ip: IpAddr, now_ms: u64) -> bool {
    ...
    banned_addr.ban_until.gt(&now_ms)   // 0 > now_ms is always false
    ...
}
```

Since `now_ms` (current Unix time in milliseconds) is always far greater than `0`, the condition `0 > now_ms` is always `false`. The IP is inserted into the ban list but is never considered banned. [1](#0-0) [2](#0-1) 

### Impact Explanation

A node operator who calls `set_ban("1.2.3.4", "insert", Some(0), Some(true), None)` receives a `null` success response and sees the entry appear in `get_banned_addresses` with `ban_until: "0x0"`. However, the peer at `1.2.3.4` is never rejected by `is_addr_banned`, so it can continue to connect, relay invalid messages, or spam the node. The operator has no indication the ban is ineffective unless they manually inspect the `ban_until` value. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

The `set_ban` RPC is exposed to any local RPC caller (the default binding is `127.0.0.1:8114`). A supported local CLI/RPC user — the exact entry path in scope — can trigger this by passing `ban_time: "0x0"` with `absolute: true`. This is a realistic operator mistake (e.g., intending to unban by setting an expired timestamp, or scripting a ban with a miscalculated timestamp). The RPC documentation does not warn against zero values. [5](#0-4) 

### Recommendation

Add a validation check before inserting the ban entry. When `absolute = true`, verify that the computed `ban_until` is strictly greater than the current time:

```rust
"insert" => {
    let ban_until = if absolute.unwrap_or(false) {
        ban_time.unwrap_or_default().into()
    } else {
        unix_time_as_millis()
            + ban_time
                .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
                .value()
    };
    // Add this check:
    if ban_until <= unix_time_as_millis() {
        return Err(RPCError::invalid_params(
            "ban_time must be a future timestamp when absolute is true"
        ));
    }
    self.network_controller
        .ban(ip_network, ban_until, reason.unwrap_or_default());
    Ok(())
}
``` [6](#0-5) 

### Proof of Concept

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "set_ban",
  "params": ["1.2.3.4", "insert", "0x0", true, "test"]
}
```

Response: `{"result": null}` — success.

Follow-up call to `get_banned_addresses` returns:
```json
[{"address": "1.2.3.4/32", "ban_until": "0x0", "ban_reason": "test", "created_at": "0x..."}]
```

The peer at `1.2.3.4` is **not** blocked: `is_ip_banned` evaluates `0 > now_ms` → `false` → peer is accepted. [2](#0-1) [7](#0-6)
