### Title
Stale Descendant-Weight Accounting in `remove_entry_and_descendants` Allows Tx-Pool Eviction Manipulation — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link information for every entry in the batch **before** calling `remove_entry` on each one. Because the root entry's links are erased first, the subsequent call to `update_ancestors_index_key` inside `remove_entry` finds no ancestors and silently skips updating their `descendants_count / descendants_size / descendants_fee` fields. Entries that remain in the pool are left with permanently inflated descendant-weight accounting, corrupting their `evict_key` and causing incorrect eviction ordering for the lifetime of those entries.

---

### Finding Description

`remove_entry_and_descendants` collects the root id and all descendant ids, then calls `remove_entry_links` on **all** of them before iterating to call `remove_entry`:

```
removed_ids = [root] + calc_descendants(root)

for id in &removed_ids {
    self.remove_entry_links(id);          // ← erases root's parent links too
}

removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
``` [1](#0-0) 

Inside `remove_entry`, the call to `update_ancestors_index_key` is supposed to walk up to every surviving ancestor and call `sub_descendant_weight`:

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
``` [2](#0-1) 

Because `remove_entry_links(root_id)` already removed the root's parent pointers from `self.links`, `calc_ancestors(root_id)` returns an empty set. No surviving ancestor entry ever has `sub_descendant_weight` called on it, and no `evict_key` is recomputed. The comment in the code acknowledges the pre-removal of links is intentional to suppress `update_descendants_index_key` for the removed descendants, but it silently also suppresses the necessary ancestor update for entries that **stay** in the pool. [3](#0-2) 

The analogous correct decrement (`update_stat_for_remove_tx`) is called and does work correctly for the global `total_tx_size` / `total_tx_cycles` counters, but the per-entry descendant-weight fields are never decremented. [4](#0-3) 

---

### Impact Explanation

Every surviving ancestor of the removed root retains inflated `descendants_count`, `descendants_size`, and `descendants_fee`. These fields feed directly into `as_evict_key()`, which governs which transactions are evicted when the pool exceeds `max_tx_pool_size`. [5](#0-4) 

A transaction with an inflated evict key appears more valuable than it is and is therefore evicted later (or not at all) relative to other pool entries. This allows an attacker to keep a low-fee transaction alive in the pool beyond its fair lifetime, displacing legitimate higher-fee transactions that should have survived eviction instead. Repeated triggering accumulates unbounded inflation across multiple surviving ancestors.

---

### Likelihood Explanation

The trigger is `resolve_conflict`, which is called every time a newly submitted transaction spends an input already claimed by a pool transaction. Any unprivileged RPC caller or P2P peer can submit a transaction via `send_transaction` / the relay protocol. The attacker needs only to:

1. Submit a parent transaction P.
2. Submit one or more child transactions spending P's outputs.
3. Submit a conflicting transaction spending the same inputs as the children.

Step 3 causes `resolve_conflict` → `remove_entry_and_descendants` on the children, leaving P with inflated descendant weights. No special privilege, key material, or majority hash power is required. [6](#0-5) 

---

### Recommendation

Before erasing the root entry's links, snapshot its ancestor set and update those ancestors' descendant weights:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // NEW: update surviving ancestors BEFORE links are erased
    if let Some(root_entry) = self.get(id).cloned() {
        self.update_ancestors_index_key(&root_entry, EntryOp::Remove);
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

Alternatively, save the ancestor set before calling `remove_entry_links` and apply `sub_descendant_weight` manually to each surviving ancestor.

---

### Proof of Concept

1. Submit parent transaction **P** (moderate fee-rate).
2. Submit child transactions **C1**, **C2** spending P's outputs (any fee-rate).
3. Submit conflicting transaction **X** spending the same input as C1.
4. `resolve_conflict(X)` calls `remove_entry_and_descendants(C1_id)`, which also removes C2 (descendant of C1).
5. Before removal: P's `descendants_count = 2`, `descendants_size = size(C1)+size(C2)`, `descendants_fee = fee(C1)+fee(C2)`.
6. After removal: P's fields are **unchanged** — still show 2 descendants with their combined size and fee.
7. P's `evict_key` is recomputed from stale data; P is ranked as if it still has two high-value descendants.
8. When the pool fills, P survives eviction rounds it should have lost, displacing legitimate transactions. [1](#0-0) [2](#0-1)

    /// Update size and cycles statistics for remove tx

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
