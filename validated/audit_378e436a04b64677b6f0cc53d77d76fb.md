Audit Report

## Title
Unauthenticated `from` Field in `ConnectionRequest`/`ConnectionSync` Enables Attacker-Controlled NAT Traversal — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
The hole-punching protocol's `ConnectionRequestProcess::execute` and `ConnectionSyncProcess::execute` never verify that the `from` field in received messages matches the actual sending peer's identity. An attacker with a single legitimate P2P connection can spoof `from=target_peer_id` with `listen_addrs=[attacker_ip]`, causing the victim to insert the attacker's IP into `pending_delivered[target_peer_id]`. A subsequent spoofed `ConnectionSync` then causes the victim to spawn a NAT traversal task that calls `raw_session` to the attacker's IP, establishing a new P2P session to an attacker-controlled endpoint. Repeating with fresh spoofed `target_peer_id` values fills the victim's connection slots, enabling a targeted eclipse attack.

## Finding Description

**Root cause — `ConnectionRequestProcess::execute` (L145–147):**

When `self_peer_id == &content.to`, the code calls `respond_delivered(content.from, ...)` directly:

```rust
if self_peer_id == &content.to {
    self.respond_delivered(content.from, &content.to, content.listen_addrs)
        .await
```

`content.from` is taken verbatim from the wire message. The actual sender is `self.peer` (the session ID), but there is no lookup to verify that the peer ID associated with `self.peer` equals `content.from`.

**`respond_delivered` (L161–167, L234–237):**

The only guard is a 2-minute cooldown keyed by `from_peer_id` (the spoofed value):

```rust
if let Some((_, t)) = self.protocol.pending_delivered.get(&from_peer_id) {
    let now = unix_time_as_millis();
    if now - t < HOLE_PUNCHING_INTERVAL {
        return StatusCode::Ignore ...
    }
}
```

After the cooldown check, the attacker's addresses are unconditionally inserted:

```rust
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
```

Using a different spoofed `target_peer_id` each time trivially bypasses the 2-minute cooldown.

**`ConnectionSyncProcess::execute` (L111–115, L144–160):**

`ConnectionSyncProcess` does not even receive the sender's session ID (compare `ConnectionRequestProcess::new` which takes `peer: PeerIndex` vs `ConnectionSyncProcess::new` which does not). It looks up `pending_delivered` using the spoofed `content.from`:

```rust
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
```

This returns the attacker's IP inserted in step 1. The code then spawns a NAT traversal task and calls `raw_session` with the attacker's IP:

```rust
let _ignore = control
    .raw_session(
        stream,
        addr,
        RawSessionInfo::inbound(listen_addr),
    )
    .await;
```

**Rate limiter analysis:**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`. `ConnectionRequest` and `ConnectionSync` have different `item_id` values, so they occupy separate rate limiter slots. Both can be sent within the same second without triggering the rate limit. The per-session `rate_limiter` allows 30 messages per second per `(session_id, item_id)`, so the attacker can send 30 `ConnectionRequest` + 30 `ConnectionSync` messages per second, each with a fresh spoofed `target_peer_id`.

## Impact Explanation

The victim node establishes raw TCP sessions to attacker-controlled IPs. By repeating the two-message sequence with many different spoofed `target_peer_id` values (each bypasses the 2-minute cooldown), the attacker can fill the victim's outbound connection slots with attacker-controlled nodes. This constitutes a targeted eclipse attack: the victim's view of the network is dominated by attacker-controlled peers, enabling selective withholding or manipulation of block/transaction relay and causing consensus deviation.

**Severity: Critical (15001–25000 points)** — Vulnerabilities which could easily cause consensus deviation.

## Likelihood Explanation

The attack requires only a single legitimate P2P connection to the victim — no special privileges, no key material, no majority hashpower. The spoofed fields (`from`, `listen_addrs`) are accepted verbatim from the wire message with no cryptographic binding to the sending session. The two-message sequence (one `ConnectionRequest`, one `ConnectionSync`) is sufficient to trigger `raw_session` to an arbitrary attacker-controlled IP. The rate limiter (30/s per session) allows rapid slot exhaustion. The cooldown bypass (fresh spoofed `target_peer_id` per attempt) makes the attack repeatable indefinitely.

## Recommendation

In `ConnectionRequestProcess::execute` (before calling `respond_delivered`), verify that `content.from` matches the actual peer ID of the sending session by looking up `self.peer` in the peer registry:

```rust
let actual_from = self.protocol.network_state
    .peer_registry.read()
    .get_peer(self.peer)
    .and_then(|p| p.connected_addr.peer_id());
if actual_from.as_ref() != Some(&content.from) {
    return StatusCode::InvalidFromPeerId.with_context("from does not match sender");
}
```

Similarly, pass the sender's session ID into `ConnectionSyncProcess` and verify `content.from` against the actual sending peer before consuming `pending_delivered`.

## Proof of Concept

```
1. Attacker connects to victim (legitimate session, session_id=S, attacker_peer_id=A)

2. Attacker sends ConnectionRequest {
       from = target_peer_id  (arbitrary, not A),
       to   = victim_peer_id,
       listen_addrs = [attacker_ip:port],
       max_hops = 1,
       route = []
   }
   → victim: self_peer_id == content.to → calls respond_delivered(target_peer_id, ...)
   → no check that target_peer_id == A
   → pending_delivered[target_peer_id] = ([attacker_ip:port], now)
   → ConnectionRequestDelivered sent back to session S

3. Attacker sends ConnectionSync {
       from  = target_peer_id,
       to    = victim_peer_id,
       route = []
   }
   → victim: route is empty, self_peer_id == content.to
   → pending_delivered.get(&target_peer_id) = [attacker_ip:port]
   → spawns try_nat_traversal(bind_addr, attacker_ip:port)
   → attacker's listener accepts TCP connection
   → control.raw_session(stream, attacker_ip, RawSessionInfo::inbound(listen_addr))
   → new P2P session established to attacker-controlled node

4. Repeat steps 2–3 with fresh target_peer_id values (bypasses 2-min cooldown).
   At 30 req/s, victim's outbound slots are filled within seconds.
```
