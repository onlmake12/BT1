Audit Report

## Title
`RecentReject::put` Never Increments `self.total_keys_num`, Disabling Shrink Guard and Allowing Unbounded Disk Growth — (File: `tx-pool/src/component/recent_reject.rs`)

## Summary
In `RecentReject::put`, the incremented key count is bound to a local variable `total_keys_num` but is never written back to `self.total_keys_num`. Because `self.total_keys_num` stays at 0 permanently, the shrink guard condition `total_keys_num > self.count_limit` evaluates to `1 > count_limit` on every call and never triggers `shrink()`. An unprivileged attacker can spam low-fee-rate transactions indefinitely, causing the underlying RocksDB TTL store to grow without bound and exhaust node disk space.

## Finding Description
In `RecentReject::put` at lines 62–69:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // self.total_keys_num is never updated here
} else {
    self.shrink()?;
}
```

The local binding `total_keys_num` holds the incremented value, but `self.total_keys_num` is never assigned. On a fresh DB, `self.total_keys_num` is initialized to 0 via `estimate_num_keys_cf` (lines 39–52), which returns 0 for an empty column family. Every subsequent call to `put()` re-computes `0.checked_add(1) = Some(1)`, which never exceeds any `count_limit > 1`, so `shrink()` is never triggered. The only code path that updates `self.total_keys_num` is inside `shrink()` itself at lines 110–111:

```rust
let total_keys_num = self.estimate_total_keys_num()?;
self.total_keys_num = total_keys_num;
```

Since `shrink()` is unreachable, `self.total_keys_num` stays frozen at 0 forever and the DB grows without bound.

The existing test at line 39 (`assert!(recent_reject.total_keys_num < 100)`) passes trivially because `total_keys_num` is always 0, not because `shrink()` worked correctly.

The exploit path is fully reachable:
- `Reject::LowFeeRate` satisfies `should_recorded()` (line 100–102: returns `true` for everything except `Duplicated`), so `put_recent_reject` is called for every low-fee-rate rejection (lines 522–524 of `process.rs`).
- `Reject::LowFeeRate` does NOT satisfy `is_malformed_tx()` (lines 89–97: only `Malformed`, `DeclaredWrongCycles`, `Verification`, and `Resolve(OverMaxDepExpansionLimit)` return `true`), so the attacker is never banned (line 514–515 of `process.rs`).

## Impact Explanation
The `RecentReject` RocksDB TTL store accumulates entries indefinitely with no upper bound. With a sufficiently long TTL (or TTL = -1 as used in tests), this causes unbounded disk consumption on the node, leading to disk exhaustion and node crash/unavailability. This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation
The attack requires no privileges. Any peer connected via P2P can submit structurally valid transactions with a fee rate below the minimum threshold. The node processes each one, records the `LowFeeRate` rejection into `RecentReject`, and never bans the peer. The attacker can sustain the spam indefinitely at negligible cost (transactions need only be structurally valid with a fee below the minimum threshold), making this a practical and repeatable denial-of-service.

## Recommendation
Add the missing assignment in `RecentReject::put` so that `self.total_keys_num` is updated on every call:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;  // <-- missing line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

The existing test in `tx-pool/src/component/tests/recent_reject.rs` should also be updated to assert that `shrink()` was actually invoked (e.g., by verifying that `total_keys_num` is non-zero and below `count_limit` after inserting more entries than the limit), rather than relying on the trivially-true assertion `total_keys_num < 100` when the field is always 0.

## Proof of Concept
```rust
#[test]
fn test_total_keys_num_never_increments() {
    let tmp_dir = tempfile::Builder::new().tempdir().unwrap();
    let count_limit = 10u64;
    let mut rr = RecentReject::build(tmp_dir.path(), 2, count_limit, -1).unwrap();

    assert_eq!(rr.total_keys_num, 0);

    for i in 0..10000u64 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        rr.put(&key, Reject::LowFeeRate(FeeRate::from_u64(0), 0, 0)).unwrap();
    }

    // BUG: total_keys_num is still 0, shrink() was never called
    assert_eq!(rr.total_keys_num, 0);
    // The underlying DB actually contains ~10000 keys, far exceeding count_limit
}
```
