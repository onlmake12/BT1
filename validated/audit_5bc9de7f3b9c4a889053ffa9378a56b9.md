### Title
Unbounded Null-Terminator Scan Loop with Deferred Cycle Accounting in `Debugger` Syscall - (File: `script/src/syscalls/debugger.rs`)

---

### Summary

The `Debugger::ecall` handler in `script/src/syscalls/debugger.rs` contains an unbounded loop that scans VM memory byte-by-byte looking for a null terminator (`0x00`). There is no maximum-length guard on the scan, and cycle accounting is deferred until after the loop completes. A script author can invoke syscall `2177` (`ckb_debug`) with a pointer into a region of VM memory that contains no null byte, forcing the loop to traverse the entire addressable VM memory space before any cycle cost is assessed.

---

### Finding Description

The `ecall` implementation for the `Debugger` syscall reads:

```rust
let mut addr = machine.registers()[A0].to_u64();
let mut buffer = Vec::new();

loop {
    let byte = machine
        .memory_mut()
        .load8(&Mac::REG::from_u64(addr))?
        .to_u8();
    if byte == 0 {
        break;
    }
    buffer.push(byte);
    addr = checked_add_addr(addr, 1)?;
}

machine.add_cycles_no_checking(transferred_byte_cycles(buffer.len() as u64))?;
``` [1](#0-0) 

The loop has two structural problems that mirror the original report's pattern:

1. **No upper-bound guard on the scan.** The loop terminates only when it reads a `0x00` byte or when `load8` returns an out-of-bounds error at the VM memory ceiling. There is no `max_len` check analogous to the buffer-length check that was missing in `innerModExp`/`privateRightShift`. If the script places the pointer at a region with no null byte before the VM memory boundary, the loop walks the entire remaining VM address space.

2. **Cycle accounting is deferred past the loop.** `machine.add_cycles_no_checking(...)` is called only after the loop exits. During the entire scan the machine's cycle counter is not incremented. This means the script performs unbounded host-side work (byte reads + `Vec` pushes) before the cycle budget is consulted.

The analogous pattern in the original report: the assembly loop in `innerModExp` decremented a length counter without checking whether it had passed the end of the result buffer; here the Rust loop increments an address counter without checking whether it has passed the end of the intended string.

---

### Impact Explanation

- **Host memory exhaustion.** `buffer` is a heap-allocated `Vec<u8>` that grows with every non-zero byte read. A script that points `A0` at a region filled with `0xFF` bytes forces the host process to allocate up to `RISCV_MAX_MEMORY` bytes (the CKB-VM hard ceiling) on the heap before the loop terminates with a VM out-of-bounds error.

- **Cycle-budget bypass during the scan.** Because `add_cycles_no_checking` is called after the loop, the script pays zero cycles while the host performs the scan. A script that has already consumed nearly its full cycle budget can still trigger the full scan; the cycle check only fires after all the host work is done. This breaks the invariant that cycle cost is proportional to host work performed.

- **Validator/node resource consumption.** Every full node that validates a transaction containing such a script must execute the scan. An attacker can craft a transaction whose script repeatedly calls `ckb_debug` with unterminated strings, multiplying the per-call cost across all validators.

---

### Likelihood Explanation

- The `DEBUG_PRINT_SYSCALL_NUMBER` (`2177`) syscall is unconditionally registered in the script verifier for every transaction; it is not gated behind a debug flag at the Rust level.
- Any unprivileged script author can deploy a lock/type script that calls `ckb_debug` with a crafted pointer. No special privilege, key, or majority hash power is required.
- The attack is cheap to construct: a single cell whose script body issues one `ecall 2177` with `a0` pointing to a non-null-terminated region suffices.

---

### Recommendation

Add a maximum-length guard inside the loop, consistent with how `exec.rs` enforces `MAX_ARGV_LENGTH`:

```rust
const MAX_DEBUG_STRING_LEN: usize = 4096; // or another reasonable bound

loop {
    if buffer.len() >= MAX_DEBUG_STRING_LEN {
        return Err(VMError::External("debug string too long".to_string()));
    }
    let byte = machine
        .memory_mut()
        .load8(&Mac::REG::from_u64(addr))?
        .to_u8();
    if byte == 0 {
        break;
    }
    buffer.push(byte);
    addr = checked_add_addr(addr, 1)?;
}
```

Additionally, move cycle accounting inside the loop (charge per byte) so that the cycle budget is enforced incrementally rather than after the fact, matching the pattern used by other syscalls.

---

### Proof of Concept

A RISC-V script compiled for CKB-VM that triggers the vulnerability:

```c
// Fills the stack with 0xFF bytes, then calls ckb_debug pointing into it.
// The Debugger loop will scan from the pointer to the VM memory ceiling
// without charging any cycles during the scan.
#include "ckb_syscalls.h"
#include <string.h>

int main() {
    // Allocate a large buffer with no null terminator
    char buf[1024 * 1024];
    memset(buf, 0xFF, sizeof(buf));
    // syscall 2177: ckb_debug(buf)
    // The Debugger loop will scan past buf until VM memory boundary
    ckb_debug(buf);  // no null byte → loop walks entire remaining VM memory
    return 0;
}
```

When this script is placed in a transaction's lock or type script and the transaction is submitted to the tx-pool or included in a block, every validating node executes the unbounded scan, allocating up to `RISCV_MAX_MEMORY` bytes on the host heap and performing the full scan before any cycle accounting occurs.
