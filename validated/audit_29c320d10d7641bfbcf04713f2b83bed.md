### Title
Permissionless `Initialize` Allows Front-Running to Hijack Config Authority — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary
The `initialize` instruction has no access-control check on the caller. Any unprivileged account can invoke it before the legitimate deployer, supply an attacker-controlled pubkey as `authority`, and permanently seize control of the `InitializePublisher` gate.

### Finding Description

`validate_payer` only verifies that the payer is a signer and writable: [1](#0-0) 

There is no check that the payer is the program upgrade authority, a hardcoded deployer key, or any other privileged identity. Any account that can sign a transaction satisfies this check.

`validate_config` only verifies the PDA derivation and writability: [2](#0-1) 

No caller-identity constraint is imposed here either.

`accounts::config::create` writes `args.authority` verbatim with no validation of its value: [3](#0-2) 

The only re-initialization guard is `data.format != 0`, which prevents a second call but does nothing to prevent the *first* call from being made by an attacker. [4](#0-3) 

The `initialize` processor wires these together with no additional privilege check: [5](#0-4) 

### Impact Explanation

`validate_authority` (used in `InitializePublisher`) enforces that the signer's key matches `config.authority`: [6](#0-5) 

If an attacker controls `config.authority`, they exclusively control which publishers are registered. They can register arbitrary publishers or permanently deny all legitimate ones. The Config PDA is a singleton; whoever creates it first wins and the state cannot be corrected without a program upgrade.

### Likelihood Explanation

The attack requires no special privilege — only enough SOL to pay rent (~0.001 SOL). On Solana, transactions are publicly visible before finalization. An attacker watching for program deployment (or simply submitting the call speculatively) can reliably win the race. This is a one-time, permanent action.

### Recommendation

Add a constraint that the payer must be the program's upgrade authority (retrievable from the `BPFLoaderUpgradeable` program data account), or hardcode an expected deployer pubkey checked at initialization time. A minimal fix is to pass the program data account and assert `payer.key == upgrade_authority`.

### Proof of Concept

```rust
// In a solana-program-test:
// 1. Deploy the program.
// 2. Attacker submits Initialize with authority = attacker_keypair.pubkey()
//    before the legitimate deployer.
// 3. Legitimate deployer submits Initialize — receives AccountAlreadyInitialized
//    because system_instruction::create_account fails on an existing account.
// 4. Read config.authority — it equals attacker_keypair.pubkey().
// 5. Attacker calls InitializePublisher signing with attacker_keypair — succeeds.
// 6. Legitimate deployer calls InitializePublisher signing with deployer_keypair
//    — fails with MissingRequiredSignature.
```

The existing test in `processor/initialize.rs` already demonstrates the happy path; a second transaction with a different authority in the same test would confirm the loser gets `AccountAlreadyInitialized`. [7](#0-6)

pub fn validate_config<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    bump: u8,
    program_id: &Pubkey,
    require_writable: bool,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let config = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    let (config_pda, expected_bump) =
        Pubkey::find_program_address(&[CONFIG_SEED.as_bytes()], program_id);
    ensure!(ProgramError::InvalidInstructionData, bump == expected_bump);
    ensure!(
        ProgramError::InvalidArgument,
        pubkey_eq(config.key, &config_pda)
    );
    if require_writable {
        ensure!(ProgramError::InvalidArgument, config.is_writable);
    }
    Ok(config)
}

pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)

    if data.len() < size_of::<Config>() {
        return Err(ReadAccountError::DataTooShort);

    let data: &mut Config = from_bytes_mut(&mut data[..size_of::<Config>()]);

pub fn initialize(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    args: &InitializeArgs,
) -> ProgramResult {
    let mut accounts = accounts.iter();
    let payer = validate_payer(accounts.next())?;
    let config = validate_config(accounts.next(), args.config_bump, program_id, true)?;
    let system = validate_system(accounts.next())?;

    let lamports = (Rent::get()?).minimum_balance(accounts::config::SIZE);

    invoke_signed(
        &system_instruction::create_account(
            payer.key,
            config.key,
            lamports,
            accounts::config::SIZE
                .try_into()
                .expect("unexpected overflow"),
            program_id,
        ),
        &[payer.clone(), config.clone(), system.clone()],
        &[&[CONFIG_SEED.as_bytes(), &[args.config_bump]]],
    )?;

    accounts::config::create(*config.data.borrow_mut(), args.authority)?;

    Ok(())
}
