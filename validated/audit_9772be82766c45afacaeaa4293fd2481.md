Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Growth via Peer-Supplied `GetBlockProposal` Messages — (File: `sync/src/types/mod.rs`)

## Summary
`insert_get_block_proposals` inserts peer-supplied `ProposalShortId` values into the shared `pending_get_block_proposals` DashMap with no size cap. Any unprivileged P2P peer can flood this map with fabricated proposal IDs absent from the tx pool, causing unbounded memory growth between periodic drain cycles. The `drain_get_block_proposals` function clones the entire map before clearing it, transiently doubling peak memory and potentially triggering OOM.

## Finding Description
`GetBlockProposalProcess::execute()` in `sync/src/relayer/get_block_proposal_process.rs` validates only the per-message proposal count against `max_block_proposals_limit * max_uncles_num`. Proposals absent from the tx pool are forwarded unconditionally to `insert_get_block_proposals` (lines 68–77). [1](#0-0) 

`insert_get_block_proposals` performs no size check before inserting into the shared DashMap. [2](#0-1) 

The map is declared as an unbounded `DashMap::new()` with no capacity argument or eviction policy. [3](#0-2) 

`drain_get_block_proposals` performs a full O(n) `.clone()` before `.clear()`, doubling peak memory at every drain tick when the map is large. [4](#0-3) 

The rate limiter at 30 req/s per `(PeerIndex, message_id)` slows but does not prevent the attack; multiple Sybil connections multiply the insertion rate linearly. [5](#0-4) 

## Impact Explanation
This matches the allowed CKB bounty impact: **"Vulnerabilities which could easily crash a CKB node" (High, 10001–15000 points)**. The victim node process is OOM-killed, taking it fully offline and preventing it from relaying or confirming transactions. The `drain_get_block_proposals` clone creates a transient memory spike that can trigger OOM even before steady-state is reached.

## Likelihood Explanation
Triggering this requires only a standard P2P connection — no authentication, no stake, no miner role. Fabricating `ProposalShortId` values absent from the tx pool is trivial (random 10-byte values). The rate limiter (30 req/s per peer) slows but does not prevent the attack; multiple Sybil connections multiply the effect linearly. No existing hard cap or eviction policy guards `pending_get_block_proposals`.

## Recommendation
Add a hard cap inside `insert_get_block_proposals`:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    const MAX_PENDING_PROPOSALS: usize = 10_000;
    if self.pending_get_block_proposals.len() >= MAX_PENDING_PROPOSALS {
        return;
    }
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
```

Additionally, replace the full-clone drain with `DashMap::drain()` or an atomic swap (e.g., `std::mem::replace` on a `Mutex<HashMap>`) to eliminate the transient memory doubling in `drain_get_block_proposals`.

## Proof of Concept
1. Connect to a victim CKB node as a standard P2P peer.
2. Repeatedly send `GetBlockProposal` relay messages containing 3,000 random `ProposalShortId` values (none present in the victim's tx pool) at 30 messages/second per connection.
3. Open 10 such connections simultaneously (900,000 entries/s insertion rate).
4. Monitor the victim process RSS — it climbs by hundreds of MB per drain interval.
5. With sustained flooding, the victim process is OOM-killed and goes offline.
