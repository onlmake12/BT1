Audit Report

## Title
Missing Duplicate-Hash Deduplication in `GetTransactionsProofProcess::execute` Enables Unbounded DB Read Amplification — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`GetTransactionsProofProcess::execute` accepts up to 1000 tx hashes with no duplicate-hash check, allowing any unprivileged light-client peer to trigger up to 2000 redundant DB reads per message by sending 1000 copies of the same valid on-chain tx hash. The sibling handler `GetBlocksProofProcess::execute` explicitly guards against this with a `HashSet` check. Because the handler returns `Status::ok()`, the peer is never banned, and no rate limiter exists on the light-client path, the attack is freely repeatable.

## Finding Description

**Asymmetry confirmed:**

`GetBlocksProofProcess::execute` rejects duplicate hashes at lines 62–70 via a `HashSet` before any DB access: [1](#0-0) 

`GetTransactionsProofProcess::execute` has only a size guard; no equivalent dedup check exists: [2](#0-1) 

**Phase 1 — up to 1000 `get_transaction_info` DB calls:**

The `partition` closure iterates the raw (undeduped) iterator and calls `snapshot.get_transaction_info(tx_hash)` for every element, including all duplicates: [3](#0-2) 

**Phase 2 — up to 1000 `get_transaction_with_info` DB calls:**

The `for tx_hash in found` loop calls `snapshot.get_transaction_with_info(&tx_hash)` for every element in `found`, again including all duplicates. The `txs_in_blocks` HashMap is keyed by `block_hash`, not by `tx_hash`, so it does not deduplicate the per-tx DB reads: [4](#0-3) 

**Phase 3 — potential panic (additional severity):**

With 1000 duplicate tx hashes all resolving to the same block, `txs_and_tx_indices` will contain 1000 entries with the same `tx_info.index`. `CBMT::build_merkle_proof` is then called with 1000 duplicate indices. If the external `merkle_cbt` crate returns `None` for duplicate indices, the `.expect()` at line 97 panics and crashes the node thread — a more severe outcome than the DB amplification alone: [5](#0-4) 

**No rate limiter on the light-client path:**

`LightClientProtocol::received` calls `try_process` directly with no rate-limiting check: [6](#0-5) 

**Peer is never banned:**

`should_ban()` only triggers on 4xx status codes. The duplicate request completes successfully and returns `Status::ok()`, so the peer is never disconnected: [7](#0-6) 

**Limit constant is 1000:** [8](#0-7) 

## Impact Explanation

A single unprivileged peer can send `GetTransactionsProof` messages with 1000 copies of the same valid on-chain tx hash in a tight loop. Each message causes up to 2000 DB reads (and potentially a node-thread panic via the `.expect()` on `build_merkle_proof` with duplicate indices) with no back-pressure, no rate limiting, and no peer ban. This constitutes a low-cost, repeatable DoS against the light-client server's DB read path and potentially the node process itself.

**Matched allowed impact: High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

If the `build_merkle_proof` panic path is confirmed, the impact escalates to **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The attack requires only: (1) a valid on-chain tx hash (publicly observable from any block explorer), and (2) the ability to connect as a light-client peer (no privilege, key, or hashpower required). The asymmetry with `GetBlocksProofProcess` confirms this is an unintentional omission. The attack is trivially repeatable with no penalty to the attacker.

## Recommendation

Add a duplicate-hash check immediately after the size guard, mirroring `GetBlocksProofProcess`:

```rust
let tx_hashes: Vec<_> = self.message.tx_hashes().to_entity().into_iter().collect();
let mut uniq = HashSet::new();
if !tx_hashes.iter().all(|h| uniq.insert(h)) {
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate tx hash exists");
}
```

This returns a 4xx status, which `should_ban()` converts into a peer ban, eliminating both the redundant DB work and the attack vector. Additionally, consider adding a rate limiter to `LightClientProtocol::received` analogous to the one in the relayer.

## Proof of Concept

1. Obtain any valid on-chain tx hash `H` (e.g., from block 1, publicly visible on any block explorer).
2. Build a `GetTransactionsProof` message: `last_hash = <current tip hash>`, `tx_hashes = [H; 1000]`.
3. Connect as a light-client peer and send the message repeatedly in a loop.
4. Observe: server performs up to 2000 DB reads per message, peer is never banned, no rate limiting applies.
5. Compare DB read count against a request with `tx_hashes = [H]` (1 unique hash → 2 DB reads).
6. Contrast with sending `GetBlocksProof` with 2 duplicate block hashes — that returns `MalformedProtocolMessage` and bans the peer immediately.
7. To test the panic path: send `tx_hashes = [H; 1000]` where `H` is a confirmed on-chain tx, observe whether `build_merkle_proof` with 1000 duplicate indices returns `None` and triggers the `.expect()` panic at line 97.
