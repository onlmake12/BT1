Audit Report

## Title
Permanent Ancestor Descendant-Weight Corruption via Pre-Cleared Links in `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-clears all link nodes for every transaction in the removal set before calling `remove_entry` on each. Because `update_ancestors_index_key` inside `remove_entry` resolves ancestors via `calc_ancestors`, which reads from the now-empty `TxLinksMap`, it returns an empty set for every removed transaction. As a result, surviving ancestor P's `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` fields are never decremented, permanently inflating P's `EvictKey` and preventing it from being correctly selected for eviction.

## Finding Description
**Root cause:** `remove_entry_and_descendants` (lines 252–265) runs a pre-clear loop calling `remove_entry_links` for every id in `removed_ids` before the removal loop. `remove_entry_links` (lines 418–430) removes the tx from its parents' children sets, removes it from its children's parents sets, and then deletes its own link node via `self.links.remove(id)`. After this loop, C1 and C2 have no link nodes at all, and P's children set is empty.

When `remove_entry` is subsequently called for C1, it invokes `update_ancestors_index_key(&C1, Remove)` (line 242), which calls `self.links.calc_ancestors(&C1_id)` (line 434). `calc_relative_ids` calls `self.inner.get(&C1_id)` which returns `None` (link node was deleted), so `direct` is `unwrap_or_default()` — an empty set. `calc_relation_ids` with an empty stage returns `{}`. The loop over ancestors is empty, so `P.sub_descendant_weight(C1)` is never called. The same applies to C2.

The comment on lines 256–258 reveals the intent was only to suppress `update_descendants_index_key` (to avoid decrementing C2's ancestor weights when C1 is removed), but the pre-clear also silences `update_ancestors_index_key`, which is the defect.

**Execution trace for P → C1 → C2, `remove_entry_and_descendants(C1)`:**

Pre-clear loop:
- `remove_entry_links(C1)` → P.children = {}, C2.parents = {}, C1 link node deleted
- `remove_entry_links(C2)` → C2 link node deleted

Removal loop:
- `remove_entry(C1)` → `calc_ancestors(C1)` → {} → P not updated
- `remove_entry(C2)` → `calc_ancestors(C2)` → {} → P not updated

**Resulting state of P:**

| Field | Expected | Actual |
|---|---|---|
| `descendants_count` | 1 | 3 |
| `descendants_fee` | P.fee | P.fee + C1.fee + C2.fee |
| `descendants_size` | P.size | P.size + C1.size + C2.size |
| `descendants_cycles` | P.cycles | P.cycles + C1.cycles + C2.cycles |

**Why existing checks fail:** There is no reconciliation path. P's descendant fields are only updated via `update_ancestors_index_key` / `update_descendants_index_key`, both of which are now no-ops for P's ghost descendants. The corruption persists until P is mined or the pool is restarted.

## Impact Explanation
`EvictKey.fee_rate` is computed as `max(descendants_feerate, feerate)` (entry.rs lines 240–243). `EvictKey` ordering is ascending by `fee_rate` — the lowest `fee_rate` entry is evicted first (sort_key.rs lines 92–103). `next_evict_entry` iterates ascending by evict_key and returns the first match (pool_map.rs lines 380–385).

With inflated `descendants_fee` and `descendants_size/cycles`, P's `descendants_feerate` is higher than reality, so P's `fee_rate` is inflated. P is never selected by `next_evict_entry` when the pool is full. Concrete effects:
- P permanently occupies a pool slot with a falsely high evict priority
- When the pool is full, P is skipped during eviction; lower-fee transactions that should be evicted remain
- Higher-fee incoming transactions are rejected as the pool stays full
- The attack is repeatable: each new chain submission corrupts another ancestor entry

This matches: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The attacker path is fully unprivileged and reachable via standard P2P transaction submission:
1. Submit P, C1, C2 (a valid dependency chain) to the target node's tx-pool
2. Submit a new transaction T that double-spends one of C1's inputs

Step 2 triggers `resolve_conflict` → `remove_entry_and_descendants(&C1_id)` (pool_map.rs lines 305–332). The same trigger fires via `resolve_conflict_header_dep` (lines 267–292) and via the ancestor-count eviction path inside `check_and_record_ancestors` (lines 614–625). No special privileges, no PoW, no key material required. The attack is repeatable and cheap — four transactions per corruption event.

## Recommendation
Remove the pre-clear loop entirely. Without it, `remove_entry` processes each tx in topological order and `update_ancestors_index_key` correctly decrements P's descendant weights via the still-intact link graph. The concern about double-decrementing C2's ancestor weights is moot because C2 is also being removed in the same pass — its `entries` record is deleted before any further use.

Alternatively, if the pre-clear is kept for the `update_descendants_index_key` suppression, the fix must manually call `update_ancestors_index_key` for each removed tx **before** its link node is erased, or collect and apply the ancestor weight deltas in a separate pass before any link nodes are removed.
