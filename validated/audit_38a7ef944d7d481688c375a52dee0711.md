Audit Report

## Title
`Debugger::ecall` Terminates VM with `VMInternalError` on Non-UTF-8 Debug Strings — (`script/src/syscalls/debugger.rs`)

## Summary
`Debugger::ecall` calls `String::from_utf8(buffer)` on bytes read from VM memory and propagates any `Err` as `VMError::External`, which `map_vm_internal_error` converts to `ScriptError::VMInternalError`. Any script that passes a pointer to a buffer containing non-UTF-8 bytes to syscall 2177 (`DEBUG_PRINT_SYSCALL_NUMBER`) will be terminated with a VM internal error instead of continuing execution, causing the transaction to be rejected by all nodes.

## Finding Description
In `script/src/syscalls/debugger.rs` lines 55–56, after reading bytes from VM memory until a null terminator, the code attempts UTF-8 decoding:

```rust
let s = String::from_utf8(buffer)
    .map_err(|e| VMError::External(format!("String from buffer {e:?}")))?;
```

The `?` operator propagates `VMError::External(...)` out of `ecall`, terminating the VM. In `script/src/verify.rs`, `detailed_run` at line 550 maps this error via `map_vm_internal_error`. That function at lines 566–572 maps `VMInternalError::External("String from buffer ...")` to `ScriptError::VMInternalError` because the reason string does not equal `"stopped"`. The `Debugger` syscall handler is unconditionally registered for all script versions (V0 and above) in `generate_ckb_syscalls` (`generator.rs` lines 23–33). The UTF-8 check executes before the `DebugPrinter` callback is invoked, so even a no-op printer does not prevent the failure.

## Impact Explanation
This is an incorrect implementation of a CKB-VM syscall (High: 10001–15000 points). A script that reads arbitrary bytes from cell data, witnesses, or script args and passes a pointer to those bytes to `DEBUG_PRINT_SYSCALL_NUMBER` for diagnostic output will fail with `ScriptError::VMInternalError` if any byte is not valid UTF-8, even if the script's logic is otherwise correct and should return exit code 0. The behavior is deterministic across all nodes (no consensus split), but causes valid transactions to be rejected. Lock scripts that fail this way would permanently prevent spending of cells.

## Likelihood Explanation
Any script author who uses the debug syscall with data read from external inputs (cell data, witnesses, args) rather than string literals risks triggering this. The syscall is available in all script versions with no documented UTF-8 requirement. The trigger is trivially reproducible: write `[0xFF, 0x00]` to a stack buffer and call ecall 2177 with a pointer to it. No special privileges are required — any script deployed on-chain can exercise this path.

## Recommendation
Replace `String::from_utf8` with `String::from_utf8_lossy` in `script/src/syscalls/debugger.rs` line 55, which replaces invalid bytes with `U+FFFD` instead of returning an error:

```rust
let s = String::from_utf8_lossy(&buffer);
(self.printer)(&self.sg_info.script_hash, s.as_ref());
```

This preserves the debug output intent while making the syscall robust to arbitrary byte sequences.

## Proof of Concept
A RISC-V script that writes `[0xFF, 0x00]` to a stack buffer and calls ecall 2177 with a pointer to it will terminate with `ScriptError::VMInternalError(External("String from buffer ..."))` instead of returning exit code 0. A unit test using `TransactionScriptsVerifier` with such a script binary (analogous to existing tests in `script/src/verify/tests/`) would confirm the incorrect rejection. The exact error string `"String from buffer"` is embedded in the `map_err` closure and would appear in the `VMInternalError` message.
