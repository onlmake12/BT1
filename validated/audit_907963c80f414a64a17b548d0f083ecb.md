Looking at the exact code paths involved:

### Title
Unguarded `Initialize` Allows Permanent DoS of `InitializePublisher` via Unsignable Authority — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

---

### Summary

The `initialize` instruction accepts any arbitrary 32-byte value as `authority` with no validation that it corresponds to a keypair-backed account. Because the config PDA can only be initialized once and there is no authority-update instruction, an attacker who calls `Initialize` first — setting `authority` to a PDA address — permanently prevents `InitializePublisher` from ever succeeding.

---

### Finding Description

**Step 1 — No access control on `Initialize`.**

`initialize` only requires that the `payer` is a signer and writable. There is no check that the caller is privileged, and no check that `args.authority` has signed or is even a valid keypair address. [1](#0-0) 

**Step 2 — Authority is stored as raw bytes with no keypair validation.**

`accounts::config::create` writes `args.authority` (a caller-supplied `[u8; 32]`) directly into the config account. Any 32-byte value is accepted — including the bytes of a PDA that has no private key. [2](#0-1) [3](#0-2) 

**Step 3 — Config can only be initialized once.**

`create` checks `data.format != 0` and returns `AlreadyInitialized` on any subsequent call. Once the config account exists, its authority is immutable. [4](#0-3) 

**Step 4 — `validate_authority` requires `is_signer` on the stored authority.**

`InitializePublisher` calls `validate_authority`, which enforces both `authority.is_signer` and `authority.key.to_bytes() == config.authority`. A PDA can never satisfy `is_signer` because it has no private key. [5](#0-4) 

**Step 5 — No recovery path exists.**

The program exposes exactly three instructions (`Initialize`, `SubmitPrices`, `InitializePublisher`) with no `UpdateAuthority` or admin-reset instruction. [6](#0-5) 

---

### Impact Explanation

If an attacker front-runs the Pyth team's `Initialize` call and sets `authority` to any PDA (e.g., the config PDA itself), `InitializePublisher` is permanently blocked. No publisher can ever be registered, making the price-store program permanently non-functional. The program would need to be redeployed and all downstream integrations updated.

---

### Likelihood Explanation

The attack window is the gap between program deployment and the first `Initialize` transaction. On Solana this is a real, observable window — any account watching the chain can submit a competing transaction. If the Pyth team does not initialize in the same transaction as deployment (or via a single atomic bundle), the attack is trivially executable by any unprivileged account with enough SOL to pay rent.

---

### Recommendation

1. **Require the authority to co-sign `Initialize`**: add `ensure!(ProgramError::MissingRequiredSignature, authority_account.is_signer)` where `authority_account.key == args.authority`, so the stored authority must prove it can sign before being recorded.
2. **Or restrict `Initialize` to the program upgrade authority**: validate that the payer is the program's upgrade authority (via `BpfUpgradeableLoader` account checks).
3. **Initialize in the same transaction as deployment** as a short-term operational mitigation.

---

### Proof of Concept

```
1. Deploy the pyth-price-store program → program_id
2. Compute config_pda = find_program_address(["CONFIG"], program_id)
3. Before Pyth calls Initialize, submit:
     Initialize {
       payer:      attacker_keypair   (signer, writable)
       config:     config_pda         (writable)
       system:     system_program
       args.authority = config_pda.to_bytes()   // PDA — no private key
     }
4. Config account is now created with authority = config_pda bytes.
5. Attempt InitializePublisher with any account as authority:
   → validate_authority checks authority.is_signer → always false for config_pda
   → returns MissingRequiredSignature
6. No valid signer for config_pda can ever exist → InitializePublisher is permanently blocked.
```

```rust
pub fn create(data: &mut [u8], authority: [u8; 32]) -> Result<&mut Config, ReadAccountError> {
    if data.len() < size_of::<Config>() {
        return Err(ReadAccountError::DataTooShort);
    }
    let data: &mut Config = from_bytes_mut(&mut data[..size_of::<Config>()]);
    if data.format != 0 {
        return Err(ReadAccountError::AlreadyInitialized);
    }
    data.format = FORMAT;
    data.authority = authority;
    Ok(data)
```

```rust
pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)
```
