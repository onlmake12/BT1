### Title
RBF `min_replace_fee` TOCTOU Race Condition Allows Griefing of Replacement Transactions — (File: `tx-pool/src/pool.rs`, `tx-pool/src/service.rs`)

---

### Summary

The `get_transaction_with_verbosity` RPC exposes a `min_replace_fee` field computed from a point-in-time snapshot of the tx-pool. An unprivileged attacker who controls the conflicting transaction can front-run a replacement submission by appending descendant transactions between the moment the user reads `min_replace_fee` and the moment the replacement is submitted. This inflates the actual `min_replace_fee` enforced at submission time, causing the replacement to be rejected. The attacker can repeat this indefinitely to block any RBF replacement of their transaction.

---

### Finding Description

**`min_replace_fee` is a stale snapshot value**

`TxPool::min_replace_fee` in `tx-pool/src/pool.rs` computes the minimum fee required to replace a pending transaction by summing the fees of the target transaction and all its current descendants, then adding an extra RBF surcharge:

```rust
// tx-pool/src/pool.rs  lines 86-99
pub fn min_replace_fee(&self, tx: &TxEntry) -> Option<Capacity> {
    if !self.enable_rbf() { return None; }
    let mut conflicts = vec![self.get_pool_entry(&tx.proposal_short_id()).unwrap()];
    let descendants = self.pool_map.calc_descendants(&tx.proposal_short_id());
    let descendants = descendants.iter()
        .filter_map(|id| self.get_pool_entry(id)).collect::<Vec<_>>();
    conflicts.extend(descendants);
    self.calculate_min_replace_fee(&conflicts, tx.size)
}
```

This value is returned to the caller via the `GetTransactionWithStatus` message handler:

```rust
// tx-pool/src/service.rs  lines 916-919
let (tx_status, min_replace_fee) = if status == &Status::Proposed {
    (TxStatus::Proposed, None)
} else {
    (TxStatus::Pending, tx_pool.min_replace_fee(entry))
};
```

**The actual enforcement happens later, under a write lock**

The real RBF fee check occurs inside `submit_entry`, which acquires the write lock and calls `check_rbf`:

```rust
// tx-pool/src/process.rs  lines 103-106
let conflicts = if tx_pool.enable_rbf() {
    tx_pool.check_rbf(&snapshot, &entry)?
} ...
```

Inside `check_rbf`, `calculate_min_replace_fee` is called again with the **current** pool state at submission time:

```rust
// tx-pool/src/pool.rs  lines 664-671
let fee = entry.fee;
if let Some(min_replace_fee) = self.calculate_min_replace_fee(&all_conflicted, entry.size) {
    if fee < min_replace_fee {
        return Err(Reject::RBFRejected(format!(
            "Tx's current fee is {}, expect it to >= {} to replace old txs",
            fee, min_replace_fee,
        )));
    }
}
```

**The race window**

Between the RPC query (read lock, releases immediately) and `submit_entry` (write lock), the pool is unlocked. During this window, any unprivileged peer or RPC caller can submit additional descendant transactions that spend outputs of the conflicting transaction. Each descendant's fee is added to `min_replace_fee`. The replacement transaction, constructed with the stale value, is then rejected.

---

### Impact Explanation

An attacker who controls transaction T1 in the pool can prevent any third party from replacing T1 via RBF:

1. User B reads `min_replace_fee(T1)` = X via `get_transaction_with_verbosity`.
2. User B constructs replacement T2 with fee = X.
3. Attacker submits T1_child (a descendant of T1) with fee Y before T2 is submitted.
4. `min_replace_fee` at submission time = X + Y + extra_rbf_fee > X.
5. T2 is rejected with `RBFRejected`.

The attacker can repeat step 3 continuously, keeping `min_replace_fee` perpetually above whatever fee User B is willing to pay. In the worst case, T1 is eventually committed on-chain before User B can replace it, permanently preventing User B's intended action. No funds are directly stolen, but the attacker achieves a targeted, sustained denial-of-service against RBF replacement for their own transactions.

---

### Likelihood Explanation

**Medium.** The attack requires only that the attacker:
- Have a transaction already in the pending pool (achievable by any tx-pool submitter).
- Be able to submit additional transactions spending their own outputs (no special privilege required).
- Monitor the mempool for incoming replacement attempts (observable via standard P2P relay or RPC polling).

The attack is cheap: the attacker only needs to pay the fee for each descendant they add. The cost is bounded by the number of descendants they can add before hitting `MAX_REPLACEMENT_CANDIDATES` (100), but the attacker can cycle descendants using their own RBF replacements to reset the count.

---

### Recommendation

**Short term:** Document in the RPC specification that `min_replace_fee` is a point-in-time estimate and may be stale by the time a replacement is submitted. Advise callers to add a safety margin above the returned value.

**Long term:** Consider adding an optional `max_fee` parameter to `send_transaction` for RBF replacements. If the actual `min_replace_fee` at submission time exceeds `max_fee`, the node rejects the submission with a clear error rather than silently accepting a transaction that will fail. This mirrors the recommendation in the LendingPool report: "consider requesting the user input an upper bound of the price that he is willing to repay."

---

### Proof of Concept

```
1. Attacker submits T1 (spending cell C) → accepted into pending pool.

2. User B calls get_transaction_with_verbosity(T1.hash, 2)
   → response: { min_replace_fee: X }

3. Attacker submits T1_child (spending T1's output 0, fee = Y)
   → accepted into pending pool as descendant of T1.

4. User B constructs T2 (spending cell C, fee = X) and calls send_transaction(T2).

5. Inside submit_entry (write lock):
     check_rbf computes min_replace_fee = X + Y + extra_rbf_fee
     T2.fee (= X) < min_replace_fee
     → Reject::RBFRejected("Tx's current fee is X, expect it to >= X+Y+extra to replace old txs")

6. T2 is rejected. Attacker repeats step 3 with a new descendant to keep
   min_replace_fee elevated indefinitely.
```
