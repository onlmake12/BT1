### Title
Cache-Bypass via `refresh=0x1` Allows Unbounded `System::new_all()` Invocations — (`rpc/src/module/terminal.rs`)

### Summary
The `get_overview` RPC endpoint accepts a caller-controlled `refresh` bitmask. When bit `0x1` (`SYSTEM_INFO`) is set, `get_sys_info` unconditionally skips the TTL cache and calls `System::new_all()` + `sys.refresh_all()` on every invocation. There is no rate limiting, cooldown, or per-caller throttle to compensate. The 5-second TTL that is supposed to bound the rate of expensive OS-level syscalls provides zero protection when the caller supplies `refresh=0x1`.

### Finding Description

**Cache bypass path** — `get_sys_info` at lines 501–506:

```rust
if !refresh.contains(RefreshKind::SYSTEM_INFO)   // ← false when refresh=0x1
    && let Some(cached) = self.cache.get_sys_info()
{
    return Ok(cached);
}
// Falls through unconditionally:
let mut sys = System::new_all();   // enumerates all OS processes
sys.refresh_all();                 // reads /proc for every process
``` [1](#0-0) 

When `refresh` contains `SYSTEM_INFO`, the entire `if` block is skipped. The cache is never consulted, and `System::new_all()` is called on every request regardless of how recently the last refresh occurred.

**No rate limiting in the server** — `server.rs` applies only a batch-size cap (`JSONRPC_BATCH_LIMIT`) and a 30-second HTTP timeout. There is no per-method, per-IP, or per-second call rate limit. [2](#0-1) [3](#0-2) 

**TTL is documented as the protection** — the module explicitly states the 5-second TTL is the mechanism that bounds expensive syscall frequency: [4](#0-3) 

That invariant is completely defeated by the `refresh` flag.

**`RefreshKind::SYSTEM_INFO = 0b00000001`** — the flag is publicly documented in the RPC spec and requires no privilege to use: [5](#0-4) 

### Impact Explanation
An unprivileged caller who can reach the RPC port (default: localhost; optionally exposed) can send `get_overview` with `refresh=1` in a tight loop. Each call triggers `System::new_all()` which allocates a new `System` struct and enumerates all OS processes, followed by `sys.refresh_all()` which re-reads `/proc/<pid>/` entries for every process. On a host with 1000+ processes this is measurably slow. Because `get_overview` is a synchronous function dispatched on the RPC thread pool, concurrent flood requests can saturate the pool, delaying or blocking all other RPC calls and degrading node responsiveness. Impact is scoped to RPC availability; consensus and P2P are unaffected.

### Likelihood Explanation
The `refresh` parameter is publicly documented with explicit values. Any caller with RPC access (local or remote depending on operator configuration) can trigger this with a trivial one-liner. No authentication, no privilege, no key material required.

### Recommendation
1. **Enforce a per-flag cooldown inside `get_sys_info`**: track the last forced-refresh timestamp separately from the TTL cache. Reject or silently serve stale data if a forced refresh was performed within the last N seconds (e.g., 5 s).
2. **Alternatively, treat `refresh` flags as advisory**: cap the effective refresh rate server-side regardless of the caller-supplied flag.
3. **Add per-method or per-IP rate limiting** at the server layer for expensive endpoints.
