Audit Report

## Title
Orphan Pool Monopolization via Missing Per-Peer Quota in `add_orphan_tx` — (`tx-pool/src/component/orphan.rs`)

## Summary

`OrphanPool` enforces a global cap of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` but contains no per-peer slot quota or fairness mechanism. A single unprivileged relay peer can fill all 100 slots with crafted orphan transactions via the standard relay handshake, then continuously refresh them, causing legitimate orphan entries from all other peers to be evicted with ~99% probability on each insertion. This breaks orphan transaction resolution for the victim node at negligible attacker cost.

## Finding Description

**Root cause — no per-peer accounting in `OrphanPool`:**

`OrphanPool` stores only a flat `HashMap<ProposalShortId, Entry>` and a reverse index; there is no per-peer counter, cap, or fairness structure anywhere in the type. [1](#0-0) 

`add_orphan_tx` inserts any non-duplicate transaction unconditionally, then calls `limit_size()`. [2](#0-1) 

`limit_size()` first removes expired entries, then evicts by calling `.keys().next()` on the `HashMap` — effectively random due to Rust's hash randomization — with no regard for which peer contributed the entry. [3](#0-2) 

**Exploit path:**

The relay path that feeds orphans is: `TransactionsProcess::execute()` → `submit_remote_tx()` → `after_process()` → `add_orphan()` → `add_orphan_tx()`. [4](#0-3) 

The `unknown_tx_hashes` filter in `TransactionsProcess::execute()` requires that the victim node previously requested the transaction from the sending peer. [5](#0-4) 

This is satisfied by the normal relay flow: attacker sends `RelayTransactionHashes` → victim adds hashes to `unknown_tx_hashes` and sends `GetRelayTransactions` back to the attacker → attacker responds with `RelayTransactions` containing orphan transactions (valid transactions referencing fabricated outpoints). The `requesting_peer()` check confirms the victim did request from this peer, so the filter passes. [6](#0-5) 

**Why existing guards are insufficient:**

The rate limiter in `Relayer` is keyed by `(PeerIndex, message_type)` at 30 req/s. This limits message frequency, not the number of transactions per message. `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`, so 100 orphans can be delivered in a single `RelayTransactions` message, consuming only 1 of the 30 allowed requests per second. [7](#0-6) [8](#0-7) 

`add_ask_for_txs` has a `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` limit that may slow the attack to multiple rounds, but does not prevent it — the attacker simply repeats the `RelayTransactionHashes` → `GetRelayTransactions` → `RelayTransactions` cycle with fresh hashes. [9](#0-8) 

`ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL` means attacker entries do not expire quickly, so the attacker does not need to race against expiry. [10](#0-9) 

## Impact Explanation

Once the attacker holds all 100 slots, any new orphan from a legitimate peer causes the pool to reach 101 entries, then one random entry is evicted. With 100/101 entries belonging to the attacker, the legitimate entry is evicted with ~99% probability. The attacker maintains near-100% occupancy by continuously advertising fresh orphan hashes. The practical effect is that legitimate orphan transactions from all other peers are evicted before their parents can arrive, permanently breaking the orphan resolution mechanism on the victim node. Applied simultaneously to many nodes (each requiring only one P2P connection), this constitutes a **bad design which could cause CKB network congestion with few costs** — matching the **High (10001–15000 points)** impact class.

## Likelihood Explanation

The attack requires only a standard P2P connection and the ability to craft transactions referencing fabricated outpoints (trivially done with any valid key pair). No PoW, no fees, no privileged access. The attacker can deliver all 100 orphans in a single `RelayTransactions` message after one `RelayTransactionHashes` / `GetRelayTransactions` round-trip. Refreshing monopoly requires only one additional round-trip per evicted slot. The attack is cheap, repeatable, and fully automatable.

## Recommendation

Introduce a per-peer slot quota (e.g., `DEFAULT_MAX_ORPHAN_TRANSACTIONS / max_peers`). Track a `HashMap<PeerIndex, usize>` counter inside `OrphanPool`. In `limit_size()`, prefer evicting entries from the peer with the most slots before falling back to random eviction. Optionally, reject insertion in `add_orphan_tx` when the submitting peer has already reached its quota. This is the approach adopted by Bitcoin Core's orphan pool since 2022.

## Proof of Concept

1. Attacker connects as peer A via standard P2P.
2. Peer A sends `RelayTransactionHashes` with 100 hashes of transactions whose inputs reference non-existent outpoints.
3. Victim node adds hashes to `unknown_tx_hashes` (keyed to peer A) and sends `GetRelayTransactions` back to peer A.
4. Peer A responds with a single `RelayTransactions` message containing all 100 orphan transactions (within `MAX_RELAY_TXS_NUM_PER_BATCH`).
5. Each transaction passes the `requesting_peer == peer A` filter, fails `pre_check` with a missing-input error, and triggers `add_orphan_tx`. After 100 insertions the orphan pool is 100% occupied by peer A's entries.
6. Peer B (honest) relays one orphan. `add_orphan_tx` inserts it (pool = 101), then `limit_size` evicts one random entry. Probability peer B's entry survives: 1/101 ≈ 1%.
7. Peer A repeats steps 2–4 with fresh orphan hashes to replace any evicted entries, maintaining monopoly at the cost of one round-trip per refresh cycle.

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
