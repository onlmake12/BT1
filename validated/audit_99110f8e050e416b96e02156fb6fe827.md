### Title
Delegators Can Front-Run `slash` by Calling `undelegate` Before Slash Is Applied — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

In Pyth's Oracle Integrity Staking (OIS) program, slashing a delegator is a two-step process: first `createSlashEvent` creates an on-chain PDA recording the slash ratio, then `slash` must be called separately per delegator to actually deduct tokens. The `undelegate` instruction does not include `delegationRecord` in its accounts and therefore cannot check whether pending slash events exist for the caller. A delegator who observes a confirmed `createSlashEvent` on-chain can call `undelegate` before `slash` is executed against their account, escaping the slash entirely.

---

### Finding Description

The OIS integrity pool uses a two-phase slashing model:

1. **`createSlashEvent`** — called by `rewardProgramAuthority`, creates a `SlashEvent` PDA keyed by `(publisher, index)` containing `epoch`, `slashRatio`, and `slashCustody`. This is an on-chain, publicly observable state change.

2. **`slash`** — called by any `signer`, applies the slash to a specific delegator's `stakeAccountCustody` and advances `delegationRecord.next_slash_event_index`.

The `DelegationRecord` struct tracks two fields:
- `last_epoch` — last epoch when reward accounting was advanced
- `next_slash_event_index` — the next slash event index yet to be applied to this delegator [1](#0-0) 

The `PoolData` account tracks `numSlashEvents` per publisher: [2](#0-1) 

The `slash` instruction correctly includes `delegationRecord` (writable) and updates `next_slash_event_index`: [3](#0-2) 

However, the `undelegate` instruction's account list contains **no `delegationRecord`**: [4](#0-3) 

Without `delegationRecord` in scope, `undelegate` cannot compare `delegationRecord.next_slash_event_index` against `poolData.numSlashEvents[publisher_index]`. There is no guard preventing a delegator from undelegating while a slash event exists that has not yet been applied to their account.

The `OutdatedDelegatorAccounting` error (code 6008) exists in the program: [5](#0-4) 

But this error cannot be raised inside `undelegate` for pending slash events because `delegationRecord` is not passed to that instruction.

---

### Impact Explanation

A delegator who successfully front-runs `slash` retains their full staked balance. The Pyth DAO receives less slashed tokens than intended, directly undermining the economic security guarantee of OIS. The slashing rulebook states slashing is capped at 5% of total staked; if many delegators escape, the effective slash approaches zero. This breaks the protocol's stated invariant that delegators share in publisher misbehavior losses pro-rata. [6](#0-5) 

---

### Likelihood Explanation

- `createSlashEvent` is confirmed on-chain and immediately visible to all participants.
- The Pythian Council has up to one full epoch (7 days) after the incident epoch to execute slashing, meaning the `slash` calls per delegator may happen well after `createSlashEvent`.
- `slash` must be called individually per delegator; with many delegators, the window between `createSlashEvent` and each individual `slash` call is non-trivial.
- Any delegator running a monitoring bot can detect `createSlashEvent` and submit `undelegate` in the same or next block.
- No privileged access is required — `undelegate` is callable by any stake account owner. [7](#0-6) 

---

### Recommendation

Before executing `undelegate`, the integrity pool program should verify that the delegator's `delegationRecord.next_slash_event_index` equals `poolData.numSlashEvents[publisher_index]`. If pending slash events exist, the instruction should revert with `OutdatedDelegatorAccounting`. This requires adding `delegationRecord` (read-only) to the `undelegate` accounts and adding the check at the start of the instruction handler, analogous to how `advanceDelegationRecord` enforces accounting freshness before distributing rewards. [8](#0-7) 

---

### Proof of Concept

1. Delegator D stakes 1,000,000 PYTH to publisher P's pool.
2. Publisher P misbehaves; the Pythian Council calls `createSlashEvent` with `slashRatio = 500` (5%). The `SlashEvent` PDA is now on-chain.
3. Delegator D's monitoring bot detects the confirmed `createSlashEvent` transaction.
4. Before the Pythian Council (or anyone) calls `slash` targeting D's `stakeAccountPositions`, D submits `undelegate` for their full position.
5. `undelegate` succeeds — it does not read `delegationRecord` and cannot detect `next_slash_event_index < numSlashEvents`.
6. D's 1,000,000 PYTH enters cooldown, fully intact. The subsequent `slash` call targeting D's account either fails (position already closed) or slashes zero tokens.
7. D avoids the 50,000 PYTH slash. The DAO treasury receives less than the intended slashed amount. [9](#0-8)

```text
## Slashing

Slashing is an important aspect of the OIS protocol to ensure the integrity of the system.

The slashed amount for each pool is calculated as follows:

$$
\large{\bold{SL_p} = w \cdot S_p = w \cdot (S^{p}_{p} + S^{d}_{p})}
$$

Where:

- $SL_p$ is the slashed amount for the publisher $p$ pool.
- $w$ is the slashing rate.
- $S_p$ is the stake assigned to the publisher $p$ pool, made of self-staked amount $S^{p}_{p}$ and delegated stake $S^{d}_{p}$, or $S_{p} = S^{p}_{p} + S^{d}_{p}$.

Here $SL_p$ is uniformly allocated to both the self-staking publisher and delegators in the pool, pro-rata to their respective stake.
```

```typescript
  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }
```
