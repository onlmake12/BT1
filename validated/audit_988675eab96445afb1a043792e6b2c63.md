### Title
Entropy Request Permanently Stuck in `CALLBACK_FAILED` State With No Cancellation Path — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
When an Entropy callback request enters the `CALLBACK_FAILED` state and the requester's callback function permanently reverts (e.g., due to a contract bug, self-destruct, or intentional design), the request is permanently stuck with no mechanism to cancel or clear it. This is a direct analog to the reported "rejected auction is always in default state" pattern: a state machine reaches a terminal state from which there is no valid exit transition.

### Finding Description
The `revealWithCallback` function implements a two-phase callback failure flow. On the first attempt, if the callback reverts and sufficient gas was provided, the request transitions to `CALLBACK_FAILED` and the transaction succeeds (emitting `CallbackFailed`). On subsequent calls when `callbackStatus == CALLBACK_FAILED`, the code skips the safe-call block (which only runs when `callbackStatus == CALLBACK_NOT_STARTED`) and calls the callback directly without error catching. [1](#0-0) 

The safe-call block is gated on `callbackStatus == CALLBACK_NOT_STARTED`: [1](#0-0) 

When `callbackStatus == CALLBACK_FAILED`, this block is bypassed and the callback is invoked directly. If the callback always reverts, the entire transaction reverts, leaving the request permanently in `CALLBACK_FAILED`. The test suite confirms this behavior: [2](#0-1) 

There is no `cancelRequest`, `clearRequest`, or timeout function available to the requester or any permissionless caller to clear a permanently stuck `CALLBACK_FAILED` request. [3](#0-2) 

### Impact Explanation
A request stuck in `CALLBACK_FAILED` occupies on-chain storage indefinitely. More critically, any application contract that holds user funds contingent on receiving the entropy callback (e.g., a lottery, a game, a commit-reveal scheme) will have those funds permanently locked at the application layer, with no protocol-level escape hatch. The Pyth Entropy contract provides no mechanism for the requester to recover from a permanently broken callback.

### Likelihood Explanation
Any `requestWithCallback` or `requestV2` call where the requester's `_entropyCallback` function permanently reverts — due to a bug introduced post-request, a contract upgrade, or a self-destruct — triggers this condition. This is reachable by any unprivileged user who makes an entropy request. The provider's `defaultGasLimit` must be non-zero (opting into the new callback failure flow) for this path to be active. [4](#0-3) 

### Recommendation
Add a `cancelRequest` function (callable by the original requester after a timeout, or permissionlessly after a sufficiently long period) that clears a request in `CALLBACK_FAILED` state. Alternatively, document that `CALLBACK_FAILED` is a permanent terminal state when the callback is irrecoverably broken, and provide a protocol-level escape hatch analogous to the recommendation in the original report.

### Proof of Concept
1. Provider registers with `defaultGasLimit > 0` (opts into callback failure flow).
2. User calls `requestV2(provider, userRandom, gasLimit)` — request is created with `callbackStatus = CALLBACK_NOT_STARTED`.
3. Provider calls `revealWithCallback` — callback reverts, request moves to `callbackStatus = CALLBACK_FAILED`, transaction succeeds.
4. The callback contract is now permanently broken (bug, upgrade, self-destruct).
5. Any subsequent call to `revealWithCallback` for this sequence number reverts entirely (callback called directly, no error catching).
6. No `cancelRequest` or equivalent exists. The request and any application-level funds awaiting the callback are permanently stuck. [5](#0-4) [6](#0-5)

    function revealWithCallback(
        address provider,
        uint64 sequenceNumber,
        bytes32 userContribution,
        bytes32 providerContribution
    ) public override {
        EntropyStructsV2.Request storage req = findActiveRequest(
            provider,
            sequenceNumber
        );

        if (
            !(req.callbackStatus ==
                EntropyStatusConstants.CALLBACK_NOT_STARTED ||
                req.callbackStatus == EntropyStatusConstants.CALLBACK_FAILED)
        ) {
            revert EntropyErrors.InvalidRevealCall();
        }
