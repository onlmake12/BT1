### Title
Missing `emitter_authority` Signer Check in `handle_post_prepared_message` Allows Unauthorized Publication and Permanent Account Locking — (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_message/mod.rs`)

---

### Summary

`handle_post_prepared_message` never verifies that the caller is the `emitter_authority` stored in the message account. Because the `emitter` account is declared `Option<Signer<'info>>`, an attacker can pass `emitter = None`, bypassing the only conditional key-equality check, and force-publish any victim message that is in `ReadyForPublishing` status. The resulting `Published` account cannot be closed (lamports permanently locked) and the victim's sequence number is consumed without consent.

---

### Finding Description

The legacy `post_message` instruction routes to `handle_post_prepared_message` whenever the message account already exists (i.e., was prepared via `init_message_v1` → `write_message_v1` → `finalize_message_v1`).

The `PostMessage` account struct declares the emitter as optional:

```rust
emitter: Option<Signer<'info>>,
``` [1](#0-0) 

Inside `handle_post_prepared_message`, the only emitter-related check is:

```rust
if let Some(emitter) = ctx.accounts.emitter.as_ref() {
    require_keys_eq!(
        emitter.key(),
        info.emitter,
        CoreBridgeError::EmitterMismatch
    );
}
``` [2](#0-1) 

This check is entirely skipped when `emitter` is `None`. Critically, there is **no check anywhere in the function** that the transaction signer matches `info.emitter_authority` — the field that was set during `init_message_v1` to record who owns the draft message.

The `emitter_sequence` PDA seed is derived from the emitter address read directly out of the message account data when `emitter` is `None`:

```rust
None => {
    // ...
    message
        .try_borrow_data()
        .map(|data| PostedMessageV1::emitter_unsafe(&data))
        .map_err(Into::into)
}
``` [3](#0-2) 

So an attacker can construct a fully valid transaction with no privileged accounts — only their own `payer` keypair — and publish any victim message that has reached `ReadyForPublishing`.

---

### Impact Explanation

**Permanent lamport lock:** After force-publishing, the message status is `Published`. `close_message_v1` delegates to `require_draft_message`, which hard-requires `MessageStatus::Writing`:

```rust
require!(
    Self::status_unsafe(&data) == MessageStatus::Writing,
    CoreBridgeError::NotInWritingStatus
);
``` [4](#0-3) 

A `Published` account satisfies neither `Writing` nor any other closeable state. The Core Bridge program has no other instruction to reclaim a `Published` message account. The victim's rent lamports are permanently locked.

**Unauthorized sequence consumption:** The victim's `emitter_sequence` is incremented:

```rust
emitter_sequence.value += 1;
``` [5](#0-4) 

**Unauthorized cross-chain message emission:** The victim's payload is published to Wormhole Guardians under the victim's emitter address and sequence number without the victim's consent. This can have downstream cross-chain consequences depending on what the payload encodes.

---

### Likelihood Explanation

The attack requires only:
1. Knowledge of the victim's message account address (public on-chain state).
2. The correct `commitment` value (readable from the message account data).
3. An empty `payload` argument.
4. A funded `payer` keypair.

No privileged access, leaked keys, or governance majority is needed. Any transaction sender can execute this against any `ReadyForPublishing` message account at any time.

---

### Recommendation

In `handle_post_prepared_message`, add an explicit check that the `emitter_authority` stored in the message is a signer. The simplest fix is to require that when `emitter` is `None` (prepared-message path), the `emitter_authority` field from the deserialized message info must match a required signer account in the context. Alternatively, make the `emitter` account non-optional for the prepared-message path and require it to match `info.emitter_authority` (not just `info.emitter`).

---
