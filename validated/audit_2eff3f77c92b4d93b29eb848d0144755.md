### Title
Off-by-One in Decompression Size Guard Allows 8 MB Heap Allocation per Crafted P2P Frame — (`network/src/compress.rs`)

### Summary

Both decompression paths in `network/src/compress.rs` use a strictly-greater-than check (`>`) against `MAX_UNCOMPRESSED_LEN`. This allows the boundary value of exactly 8,388,608 bytes to pass the guard, causing an 8 MB heap allocation before any decompression is attempted. An unprivileged remote peer can craft a minimal snappy frame (a few bytes) whose varint header claims exactly `MAX_UNCOMPRESSED_LEN` bytes, triggering the allocation. With up to 125 simultaneous peer connections each sending such frames, this produces up to ~1 GB of concurrent transient heap allocations.

### Finding Description

`MAX_UNCOMPRESSED_LEN` is defined as `1 << 23` (8,388,608 bytes). [1](#0-0) 

In `LengthDelimitedCodecWithCompress::decode`, the guard is: [2](#0-1) 

The check `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` is `FALSE` when `decompressed_bytes_len == 8_388_608`, so execution falls through to `BytesMut::zeroed(decompressed_bytes_len)` — allocating exactly 8 MB. The identical off-by-one exists in `Message::decompress`: [3](#0-2) 

The snappy format encodes the uncompressed length as a varint at the start of the stream. For 8,388,608 = `0x800000`, the varint is only 4 bytes (`[0x80, 0x80, 0x80, 0x04]`). An attacker can send a frame containing just this 4-byte varint plus a minimal (or invalid) body — well within any protocol's `max_frame_length`. The `decompress_len` call returns 8,388,608, the guard passes, 8 MB is allocated, and then `SnapDecoder::new().decompress()` fails on the invalid body, returning `Err` and freeing the buffer. The connection is dropped.

The `LengthDelimitedCodecWithCompress` is wired into every `CKBProtocol` with compression enabled by default: [4](#0-3) [5](#0-4) 

Affected protocols include Sync (2 MB frame limit) and RelayV3 (4 MB frame limit) — both well above the ~5-byte crafted payload needed: [6](#0-5) 

### Impact Explanation

Each crafted frame causes a transient 8 MB heap allocation. With `max_peers = 125` connections each sending one such frame concurrently, the node faces up to ~1 GB of simultaneous transient allocations. While each allocation is freed after the decode error, the burst can cause significant memory pressure, potentially triggering OOM on memory-constrained nodes or degrading performance through allocator contention. The attacker must reconnect after each attempt (connection is dropped on `Err`), but reconnection is cheap.

### Likelihood Explanation

The attack requires only a valid TCP connection and knowledge of the snappy varint format — no authentication, no PoW, no privileged role. Compression is enabled by default for all `CKBProtocol` instances. The crafted payload is trivially constructable (4-byte varint + 1 byte body).

### Recommendation

Change both `>` comparisons to `>=` so that `decompressed_bytes_len == MAX_UNCOMPRESSED_LEN` is also rejected:

```rust
// In both Message::decompress and LengthDelimitedCodecWithCompress::decode:
if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN {
``` [7](#0-6) [8](#0-7) 

### Proof of Concept

```rust
// Craft a snappy stream claiming exactly MAX_UNCOMPRESSED_LEN (8,388,608) bytes:
// Snappy varint for 8_388_608 = [0x80, 0x80, 0x80, 0x04] (4 bytes)
// Followed by a single invalid literal byte to form a minimal body.
let crafted_compressed: &[u8] = &[0x80, 0x80, 0x80, 0x04, 0x00];

// Prepend COMPRESS_FLAG (0x80) as the frame's first byte:
let mut frame = BytesMut::new();
frame.put_u8(0x80); // COMPRESS_FLAG
frame.extend_from_slice(crafted_compressed);

// decompress_len returns 8_388_608; guard (> not >=) passes;
// BytesMut::zeroed(8_388_608) is allocated; decompress fails; Err returned.
let result = decompress(frame);
assert!(result.is_err()); // true, but 8 MB was allocated and freed

// Repeat across 125 peer connections simultaneously for ~1 GB transient pressure.
```
