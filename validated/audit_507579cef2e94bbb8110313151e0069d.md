### Title
Unbounded Hash Loop in `constructProviderCommitment` Enables Permanent Fund Lockup via Gas Exhaustion — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `constructProviderCommitment` function in `Entropy.sol` contains an unbounded `while` loop that iterates `numHashes` times. Because `numHashes` is stored immutably in each request at creation time and the guard (`maxNumHashes`) is only enforced when a provider explicitly opts in, an unprivileged attacker can inflate the gap between `assignedSequenceNumber` and `currentCommitmentSequenceNumber` by submitting many requests to a provider that has not set `maxNumHashes`. Any subsequent legitimate user's request will be assigned a `numHashes` large enough to exhaust the block gas limit during `revealWithCallback` / `reveal`, permanently locking that user's funds with no on-chain refund path.

---

### Finding Description

**Root cause — unbounded loop:** [1](#0-0) 

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
```

Each iteration costs ~30 gas (keccak256). At 1 000 000 iterations that is ~30 M gas — at or above the Ethereum block gas limit, and trivially above limits on many L2/alt-L1 deployments.

**`numHashes` is fixed at request time and never updated:** [2](#0-1) 

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

The guard is a no-op when `maxNumHashes == 0` (the default after `register()`). Calling `advanceProviderCommitment` after the request is stored does **not** retroactively reduce `req.numHashes`; the stored value is permanent.

**`constructProviderCommitment` is called unconditionally in both reveal paths:** [3](#0-2) 

Both `reveal` and `revealWithCallback` call `revealHelper`, which calls `constructProviderCommitment(req.numHashes, ...)`. If the loop exceeds the gas limit the transaction reverts, and because there is no refund or cancellation mechanism for stuck requests, the user's deposited fee is permanently locked.

---

### Impact Explanation

A user who submits a `requestWithCallback` / `requestV2` against a provider with `maxNumHashes == 0` and a large sequence gap will have their fee locked forever. Neither the user nor the provider can recover the funds: `revealWithCallback` always reverts out-of-gas, and there is no `cancelRequest` or refund function in the contract.

---

### Likelihood Explanation

1. **Default state is vulnerable.** `maxNumHashes` defaults to `0` after `register()`; a provider must make a separate `setMaxNumHashes` call to opt in to protection. Many providers may never do this.
2. **Attacker cost is bounded.** On low-fee chains (Polygon, BNB Chain, Base, etc.) an attacker can submit tens of thousands of requests for a few dollars, pushing the sequence gap past the gas-exhaustion threshold.
3. **Provider cannot retroactively fix existing requests.** Even if the provider calls `advanceProviderCommitment` or `setMaxNumHashes` after the attack, the stored `req.numHashes` in the victim's request is immutable.
4. **No privileged access required.** Any address can call `requestV2` / `requestWithCallback`.

---

### Recommendation

1. **Enforce a protocol-level hard cap on `numHashes` unconditionally** — not gated on `maxNumHashes != 0`:
   ```solidity
   uint32 constant MAX_NUM_HASHES = 10_000; // tune to safe gas budget
   if (req.numHashes > MAX_NUM_HASHES) revert EntropyErrors.LastRevealedTooOld();
   ```
2. **Require providers to set `maxNumHashes > 0` during `register()`**, or default it to a safe value rather than `0`.
3. **Add a user-callable `cancelRequest` / refund path** so that requests that can never be fulfilled do not permanently lock funds.

---

### Proof of Concept

```
1. Provider P calls register(...) without ever calling setMaxNumHashes().
   → providerInfo.maxNumHashes == 0  (guard disabled)
   → providerInfo.currentCommitmentSequenceNumber == 0

2. Attacker calls requestV2(P, ...) × N times (N ≈ 1 000 000 on a cheap chain).
   → providerInfo.sequenceNumber == N
   → providerInfo.currentCommitmentSequenceNumber still == 0
     (provider has not called advanceProviderCommitment)

3. Victim calls requestV2(P, ...).
   → assignedSequenceNumber = N
   → req.numHashes = N - 0 = N  (stored immutably)
   → msg.value (victim's fee) is credited to the contract

4. Provider calls revealWithCallback(P, N, ...).
   → revealHelper → constructProviderCommitment(N, revelation)
   → while loop runs N times → OUT OF GAS → revert

5. Victim's fee is permanently locked; no refund path exists.
``` [1](#0-0) [2](#0-1) [4](#0-3)
