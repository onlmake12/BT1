### Title
Unvalidated `observed_addr` Injection Poisons Victim Node's Advertised Listen Addresses — (`network/src/protocols/identify/mod.rs`)

### Summary
Any unprivileged remote peer can send an identify message whose `observed_addr` field contains an arbitrary globally-routable `Multiaddr`. Because no reachability or ownership validation is performed at any point in the call chain, the victim node stores the attacker-controlled address and subsequently broadcasts it as one of its own listen addresses to every new peer that connects, violating the invariant that a node must only advertise its own verified listen addresses.

### Finding Description

The full call chain is:

**Step 1 — `process_observed` accepts any address without validation.**

`process_observed` (lines 152–169) takes the raw `observed` `Multiaddr` from the incoming identify message and immediately calls `add_observed_addr`. There is no IP reachability check, no size limit, and no ownership verification. The function unconditionally returns `MisbehaveResult::Continue`. [1](#0-0) 

**Step 2 — `IdentifyCallback::add_observed_addr` stores the address verbatim.**

The only transformation is appending the victim's own peer ID if absent. The address is then written into `NetworkState::observed_addrs` keyed by `session_id`. Always returns `MisbehaveResult::Continue`. [2](#0-1) 

**Step 3 — `NetworkState::add_observed_addr` performs no validation.**

The address is inserted into the `observed_addrs: RwLock<HashMap<PeerIndex, Multiaddr>>` with no further checks. [3](#0-2) 

**Step 4 — `local_listen_addrs` pads with attacker-controlled addresses.**

When the victim has fewer than `MAX_RETURN_LISTEN_ADDRS` (10) public addresses — the common case for NAT-ed nodes — `local_listen_addrs` fills the remainder from `observed_addrs`, directly including the attacker's injected address. [4](#0-3) 

**Step 5 — `connected` broadcasts the poisoned list to every new peer.**

When any new peer connects, the victim immediately sends an identify message whose `listen_addrs` is built from `local_listen_addrs()`. The only filter applied is `is_reachable`, which merely checks that the IP is globally routable — a condition the attacker trivially satisfies by using any real public IP (e.g., `8.8.8.8`). [5](#0-4) 

**Step 6 — Cleanup is too late.**

The `observed_addrs` entry is only removed on `SessionClose`. While the attacker remains connected, every new peer that connects to the victim receives the poisoned address. [6](#0-5) 

### Impact Explanation

- **Address poisoning at scale:** The victim node advertises an attacker-chosen IP:port (with the victim's own peer ID appended) as one of its listen addresses to every subsequent peer. Receiving peers store this in their peer stores and may re-broadcast it via the discovery protocol, amplifying the pollution network-wide.
- **Connectivity disruption:** Peers that attempt to reach the victim via the poisoned address will fail, degrading the victim's effective reachability.
- **NAT-ed nodes are fully exposed:** A node with zero configured public addresses relies entirely on `observed_addrs` to fill its advertised list, meaning the attacker can control the entire advertised address set with a single connection.
- **Low cost:** A single TCP connection and one identify message is sufficient. The attacker can maintain the connection indefinitely to keep the poisoned entry alive.

### Likelihood Explanation

The attack requires only a standard P2P connection — no special privileges, no PoW, no key material. The identify protocol is mandatory for all peers. The precondition (victim has fewer than 10 public addresses) is the default state for any node behind NAT, which is the majority of CKB nodes. The `is_reachable` filter in `connected()` is the only guard, and it is trivially bypassed by using any real public IP.

### Recommendation

1. **Validate ownership of `observed_addr`:** Only accept an `observed_addr` if it matches the actual transport address of the session (i.e., `session.address`). The observed address is supposed to tell the local node what its own external address looks like — it must match the session's remote endpoint, not an arbitrary value.
2. **Apply `is_reachable` at insertion time** in `add_observed_addr` / `NetworkState::add_observed_addr`, not only at broadcast time.
3. **Require consensus across multiple peers:** Only promote an observed address to the advertised list if multiple independent peers report the same address (quorum-based confirmation), preventing a single attacker from injecting an address.

    fn local_listen_addrs(&mut self) -> Vec<Multiaddr> {
        let mut listens = self.listen_addrs();

        if listens.len() < MAX_RETURN_LISTEN_ADDRS {
            let observe_addrs = self
                .network_state
                .observed_addrs(MAX_RETURN_LISTEN_ADDRS - listens.len());
            listens.extend(observe_addrs);
            listens
        } else {
            listens
        }
    }
