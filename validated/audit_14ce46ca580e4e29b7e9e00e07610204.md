### Title
Fee Rate Estimate Manipulable via Lazy-Accumulated Historical Flow Snapshot — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

### Summary

The `WeightUnitsFlow` fee estimator accumulates historical transaction-flow data in a lazy snapshot (`self.txs`) that is keyed by `self.current_tip`. Because `current_tip` is only advanced when `commit_block` is called, all transactions submitted between two consecutive blocks are recorded at the same block height. An unprivileged tx-pool submitter can flood the pool with artificially high-fee-rate transactions to inflate the historical flow snapshot, causing `estimate_fee_rate` to return an elevated fee rate that persists for up to `2 × MAX_TARGET` (≈ 202) blocks — even after the attacker's transactions are evicted or expire.

### Finding Description

`weight_units_flow::Algorithm` maintains two inputs to its estimate:

1. **`sorted_current_txs`** — the live mempool state at query time.
2. **`self.txs: HashMap<BlockNumber, Vec<TxStatus>>`** — a historical record of every transaction accepted into the pool, keyed by the block height at which it was accepted (`self.current_tip`).

`accept_tx` appends to `self.txs[self.current_tip]`:

```rust
pub fn accept_tx(&mut self, info: TxEntryInfo) {
    if self.current_tip == 0 { return; }
    let item = TxStatus::new_from_entry_info(info);
    self.txs
        .entry(self.current_tip)
        .and_modify(|items| items.push(item))
        .or_insert_with(|| vec![item]);
}
```

`self.current_tip` is **only updated** when a block is committed:

```rust
pub fn commit_block(&mut self, block: &BlockView) {
    let tip_number = block.number();
    if self.boot_tip == 0 { self.boot_tip = tip_number; }
    self.current_tip = tip_number;
    self.expire();
}
```

The historical data is retained for `historical_blocks = 2 × target_blocks` and is used to compute `flow_speed_buckets`:

```rust
buckets.into_iter()
    .map(|value| value / historical_blocks)
    .collect::<Vec<_>>()
```

The estimation decision is:

```rust
let passed = current_weight + added_weight <= removed_weight;
// added_weight = flow_speed_buckets[bucket_index] * target_blocks
```

An attacker who submits N large-weight, high-fee-rate transactions between two blocks inflates `self.txs[current_tip]`. This inflates `flow_speed_buckets` for all subsequent calls to `estimate_fee_rate` for the next `2 × target_blocks` blocks, regardless of whether those transactions are still in the pool.

The `expire()` function only prunes entries older than `current_tip - historical_blocks`, so the poisoned entry at `current_tip = K` survives until block `K + 2 × MAX_TARGET` (≈ K + 202).

### Impact Explanation

Any node operator running the `WeightUnitsFlow` estimator (configured via `fee_estimator.algorithm = "WeightUnitsFlow"`) exposes its `estimate_fee_rate` RPC to manipulation. Wallets and tooling that call this RPC to determine the fee rate for their transactions will receive an inflated estimate and overpay fees for up to ~202 blocks. The attacker pays only the minimum fee to submit the flood transactions; the economic harm falls entirely on honest users who trust the estimate.

The fallback path (`pool_map.estimate_fee_rate`) is also manipulable via the same mechanism (current pool state), but it self-corrects as soon as the attacker's transactions leave the pool. The `weight_units_flow` historical snapshot does not self-correct until the poisoned block entry expires.

### Likelihood Explanation

- The `WeightUnitsFlow` estimator is **not enabled by default** (`FeeEstimator::Dummy` is the default); it must be explicitly configured.
- Any unprivileged tx-pool submitter (local RPC caller or remote peer relaying a transaction) can trigger `accept_tx`.
- The attacker needs enough CKB to pay the minimum fee for N transactions; no special privilege or hashpower is required.
- The attack is repeatable: the attacker can re-submit a new flood batch every `~202` blocks to keep the estimate elevated indefinitely.

Likelihood is **medium** given the opt-in nature of the estimator, but **high** for any node that has enabled it.

### Recommendation

1. **Do not persist raw transaction submissions in the historical flow snapshot without bounding per-block contribution.** Cap the total weight recorded per block height (e.g., `MAX_BLOCK_BYTES`) so a single block's entry cannot dominate the flow average.
2. **Apply a decay factor** to historical entries (as the `ConfirmationFraction` algorithm does) so that a single manipulated block's influence diminishes over time rather than persisting at full weight until expiry.
3. **Exclude transactions that were evicted, expired, or replaced (RBF)** from the historical flow data, or subtract their weight when they leave the pool.
4. Consider a time-weighted average rate (TWAR) approach analogous to the Morpho recommendation: weight each block's contribution by how recently it occurred.

### Proof of Concept

1. Node is configured with `fee_estimator.algorithm = "WeightUnitsFlow"`.
2. Attacker calls `send_transaction` (local RPC or P2P relay) to submit 1 000 transactions each with fee rate = 2 000 000 shannons/kB (the highest bucket, index ≈ 135+) and maximum allowed size. All are accepted into the pool between two consecutive blocks; `accept_tx` records all 1 000 entries at `self.txs[current_tip]`.
3. Honest user calls `estimate_fee_rate` with `NoPriority` mode. `do_estimate` computes `flow_speed_buckets` from `sorted_flowed`, which now includes the 1 000 attacker entries. The inflated `added_weight` causes all low-fee buckets to fail the `passed` check, returning a fee rate far above the true market rate.
4. Attacker waits for their transactions to expire (after `expiry_hours`). The pool clears, but `self.txs[current_tip]` still holds the 1 000 entries.
5. `estimate_fee_rate` continues to return the inflated rate for the next `2 × target_blocks` blocks (up to 202 blocks for `NoPriority` mode with `DEFAULT_TARGET = 21`, `historical_blocks = 42`; up to 202 blocks for `MAX_TARGET = 101`).

    fn sorted_flowed(&self, historical_tip: BlockNumber) -> Vec<TxStatus> {
        let mut statuses: Vec<_> = self
            .txs
            .iter()
            .filter(|&(&num, _)| num >= historical_tip)
            .flat_map(|(_, statuses)| statuses.to_owned())
            .collect();
        statuses.sort_unstable_by(|a, b| b.cmp(a));
        ckb_logger::trace!(">>> sorted flowed length: {}", statuses.len());
        statuses
    }

                }
            }
        }
