### Summary

`parseTwapPriceFeedUpdates` in `Pyth.sol` and `validate_twap_messages` in the Solana receiver program impose no minimum window size between the start and end TWAP data points. Any unprivileged caller can submit two valid Wormhole-signed VAAs that are only 1 slot (~400 ms on Pythnet) apart. The contract accepts this as a valid TWAP and returns a single-slot spot price, completely defeating the manipulation-resistance property that TWAP is supposed to provide.

---

### Finding Description

`parseTwapPriceFeedUpdates` in `Pyth.sol` calls `validateTwapPriceInfo` before computing the TWAP: [1](#0-0) 

The validation checks:
- `prevPublishTime < publishTime` for each individual message (uniqueness)
- `expo` matches between start and end
- `startSlot <= endSlot` (allows equal slots â€” a separate division-by-zero DoS)
- `startTime <= endTime`

**There is no check that `endSlot - startSlot` (or `endTime - startTime`) meets any minimum threshold.** A `slotDiff` of 1 passes all checks.

`calculateTwap` then divides cumulative price differences by `slotDiff`: [2](#0-1) 

With `slotDiff = 1`, the result is `cumulativePrice[end] - cumulativePrice[start]` â€” a single-slot spot price, not a time-weighted average.

The same root cause exists on Solana. `validate_twap_messages` enforces `end_slot > start_slot` (strictly, so `slotDiff â‰Ą 1`) but imposes no minimum: [3](#0-2) 

`calculate_twap` then divides by `slot_diff`: [4](#0-3) 

The Solana SDK's `get_twap_no_older_than` does enforce an exact `window_seconds` match at the consumer level: [5](#0-4) 

However, `get_twap_unchecked` bypasses this entirely: [6](#0-5) 

The SDK's warning explicitly acknowledges this risk ("arbitrary window length twap updates"), but the on-chain program itself provides no protection.

---

### Impact Explanation

Any protocol that calls `parseTwapPriceFeedUpdates` (EVM) or reads a `TwapUpdate` account (Solana) without independently verifying `endTime - startTime` will receive a single-slot spot price instead of a manipulation-resistant TWAP. An attacker can:

1. Monitor Pythnet for a slot where the price deviates significantly from fair value (e.g., due to a large DEX trade used as a Pythnet data source).
2. Obtain the two valid Wormhole-signed VAAs for that slot and the immediately preceding slot.
3. Call `parseTwapPriceFeedUpdates` with those two VAAs.
4. The contract returns a "TWAP" that is actually the manipulated single-slot price.
5. The attacker uses this price to exploit a lending protocol, perpetuals exchange, or any other DeFi protocol that relies on the TWAP for manipulation resistance.

The `TwapPriceFeed` struct does expose `startTime` and `endTime` to consumers, but there is no on-chain enforcement, and the `downSlotsRatio` field (intended to signal data quality) will be 0 for a 1-slot window with no missed slots â€” giving a false signal of high quality. [7](#0-6) 

---

### Likelihood Explanation

- The entry point (`parseTwapPriceFeedUpdates`) is permissionless â€” any address can call it with any two valid VAAs.
- Valid Wormhole-signed Pythnet VAAs for consecutive slots are always available from Hermes; no key compromise or privileged access is required.
- The attacker only needs to pay the standard update fee.
- Protocols that integrate TWAP specifically to resist manipulation are the primary targets, and many will not independently re-validate the window size since the contract is expected to enforce it.

---

### Recommendation

Add a minimum window size check in `validateTwapPriceInfo` (EVM) and `validate_twap_messages` (Solana). For example, enforce a minimum of 300 seconds (5 minutes) or a configurable governance-controlled minimum:

**EVM (`Pyth.sol`):**
```solidity
uint64 constant MIN_TWAP_WINDOW_SLOTS = 750; // ~5 min at 400ms/slot

function validateTwapPriceInfo(...) private pure {
    ...
    if (twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot < MIN_TWAP_WINDOW_SLOTS) {
        revert PythErrors.InvalidTwapUpdateDataSet();
    }
}
```

**Solana (`lib.rs`):**
```rust
const MIN_TWAP_WINDOW_SLOTS: u64 = 750;

fn validate_twap_messages(start_msg: &TwapMessage, end_msg: &TwapMessage) -> Result<()> {
    ...
    require!(
        end_msg.publish_slot.saturating_sub(start_msg.publish_slot) >= MIN_TWAP_WINDOW_SLOTS,
        ReceiverError::TwapWindowTooShort
    );
    Ok(())
}
```

Additionally, fix the EVM `validateTwapPriceInfo` slot check from `>` to `>=` to prevent the equal-slot division-by-zero panic:

```solidity
// Change:
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot)
// To:
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot)
```

### Proof of Concept

1. Obtain two valid Wormhole-signed Pythnet VAAs for consecutive slots `N` and `N+1` from Hermes, where slot `N+1` has a spike in `cumulativePrice` (e.g., `cumulativePrice[N+1] - cumulativePrice[N]` = manipulated spot price Ă— 1).
2. Call `parseTwapPriceFeedUpdates{value: fee}([vaa_N, vaa_N+1], [priceId])`.
3. `validateTwapPriceInfo` passes: `prevPublishTime < publishTime` for both, `expo` matches, `publishSlot[N] < publishSlot[N+1]`, `publishTime[N] <= publishTime[N+1]`.
4. `calculateTwap` computes `slotDiff = 1`, `twapPrice = priceDiff / 1 = manipulated spot price`.
5. The returned `TwapPriceFeed.twap.price` equals the single-slot manipulated price, with `downSlotsRatio = 0` (no missed slots), falsely indicating a high-quality TWAP.
6. Any protocol consuming this result without checking `endTime - startTime >= MIN_WINDOW` is exploited. [8](#0-7) [9](#0-8)

fn validate_twap_messages(start_msg: &TwapMessage, end_msg: &TwapMessage) -> Result<()> {

    require!(
        start_msg.feed_id == end_msg.feed_id,
        ReceiverError::FeedIdMismatch
    );

    // Validate exponents match
    require!(
        start_msg.exponent == end_msg.exponent,
        ReceiverError::ExponentMismatch

    require!(
        end_msg.publish_slot > start_msg.publish_slot,
        ReceiverError::InvalidTwapSlots

    require!(
        start_msg.prev_publish_time < start_msg.publish_time,
        ReceiverError::InvalidTwapStartMessage
    );
    require!(
        end_msg.prev_publish_time < end_msg.publish_time,
        ReceiverError::InvalidTwapEndMessage
    );

fn calculate_twap(start_msg: &TwapMessage, end_msg: &TwapMessage) -> Result<(i64, u64, u32)> {
    let slot_diff = end_msg
        .publish_slot
        .checked_sub(start_msg.publish_slot)
        .ok_or(ReceiverError::TwapCalculationOverflow)?;

    let price_diff = end_msg
        .cumulative_price
        .checked_sub(start_msg.cumulative_price)
        .ok_or(ReceiverError::TwapCalculationOverflow)?;

    let conf_diff = end_msg
        .cumulative_conf
        .checked_sub(start_msg.cumulative_conf)
        .ok_or(ReceiverError::TwapCalculationOverflow)?;

    // Calculate time averaged price and confidence
    let price = i64::try_from(price_diff / i128::from(slot_diff))
        .map_err(|_| ReceiverError::TwapCalculationOverflow)?;
    let conf = u64::try_from(conf_diff / u128::from(slot_diff))
        .map_err(|_| ReceiverError::TwapCalculationOverflow)?;
