### Title
Stale On-Chain State Used for Uniqueness Check Instead of Message-Embedded `prevPublishTime` - (File: `target_chains/stylus/contracts/pyth-receiver/src/lib.rs`)

---

### Summary

The Stylus (Arbitrum) Pyth receiver contract's `parse_price_feed_updates_internal` function, when `check_uniqueness` is enabled, reads the on-chain stored `latest_price_info` to obtain `prev_publish_time` for the uniqueness gate. The EVM reference implementation uses the `prevPublishTime` field embedded in the price message itself. Because the on-chain stored price can be arbitrarily stale, the Stylus uniqueness check can be bypassed: an attacker submits a price update that is **not** the first update in the requested window, yet the contract accepts it as unique because the stale on-chain price predates `minAllowedPublishTime`.

---

### Finding Description

In `target_chains/stylus/contracts/pyth-receiver/src/lib.rs`, the uniqueness check inside `parse_price_feed_updates_internal` is:

```rust
if check_uniqueness {
    let prev_price_info = self.latest_price_info.get(price_id_fb);
    let prev_publish_time = prev_price_info.publish_time.get().to::<u64>();

    if prev_publish_time > 0 && min_allowed_publish_time <= prev_publish_time {
        return Err(PythReceiverError::PriceFeedNotFoundWithinRange);
    }
}
``` [1](#0-0) 

`prev_publish_time` is sourced from `self.latest_price_info` — the most recently stored on-chain price — not from the `prevPublishTime` field carried inside the accumulator message itself.

The EVM implementation (`Pyth.sol` / `PythAccumulator.sol`) extracts `prevPublishTime` directly from the encoded price feed message via `extractPriceInfoFromMerkleProof`, and uses that message-embedded value for the uniqueness gate: [2](#0-1) 

The message-embedded `prevPublishTime` represents the publish time of the immediately preceding update in the Pythnet accumulator sequence — the correct reference for "is this the first update after `minPublishTime`?" The on-chain `latest_price_info.publish_time` is an unrelated value: it reflects the last time anyone called `updatePriceFeeds` on this particular chain, which can lag arbitrarily behind the Pythnet sequence.

---

### Impact Explanation

`parsePriceFeedUpdatesUnique` / `parsePriceFeedUpdatesWithConfig(checkUniqueness=true)` is the function used by derivative protocols for **deterministic settlement at a fixed timestamp** — the caller expects exactly the first price update published after `minPublishTime`. If the uniqueness invariant is broken, an attacker can:

1. Obtain any price update whose `publishTime` falls in `[minPublishTime, maxPublishTime]` from Hermes (multiple such updates exist because Pythnet publishes at sub-second frequency).
2. Submit a later update in the window (not the first), bypassing the uniqueness check because the on-chain stored price is stale (before `minAllowedPublishTime`).
3. Receive a price of their choosing within the window, enabling adversarial price selection for settlement — equivalent to the "adversarial selection" attack described in Pyth's own best-practices documentation.

Financial loss to protocols using this function for options/futures settlement on Arbitrum Stylus is the concrete impact. [3](#0-2) 

---

### Likelihood Explanation

Three conditions are required, all routinely satisfied:

1. **On-chain price is stale before `minAllowedPublishTime`**: Any time no one has called `updatePriceFeeds` recently, the stored price lags. This is the normal state for most chains between user-driven updates.
2. **Multiple updates exist in the window**: Pythnet publishes prices multiple times per second; any 1-second window contains many updates.
3. **Attacker can retrieve non-first updates**: Hermes exposes historical price data at arbitrary timestamps via the `/v2/updates/price/{publish_time}` endpoint, making any update in the window freely obtainable.

No privileged access, leaked keys, or governance majority is required. Any unprivileged transaction sender can exploit this.

---

### Recommendation

Replace the on-chain `latest_price_info` lookup with the `prevPublishTime` value decoded from the accumulator message itself, mirroring the EVM implementation. The `prevPublishTime` field is already present in the encoded `PriceFeedMessage` and is parsed during Merkle proof verification. Pass it through to the uniqueness check instead of reading `self.latest_price_info`.

---

### Proof of Concept

1. Deploy a settlement contract on Arbitrum Stylus that calls `parsePriceFeedUpdatesUnique(updateData, [BTC/USD], T, T+5)` for a 5-second settlement window starting at `T`.
2. Ensure the on-chain BTC/USD price was last updated at time `T - 100` (stale, before `T`).
3. From Hermes, fetch **two** BTC/USD updates with `publishTime` in `[T, T+5]`: call them `update_first` (earliest) and `update_later` (a later one, with `prevPublishTime >= T`).
4. Submit `update_later` to `parsePriceFeedUpdatesUnique`.
5. The Stylus contract reads `latest_price_info.publish_time = T - 100 < T = minAllowedPublishTime`, so the uniqueness check passes.
6. The contract returns `update_later`'s price — not the first price in the window — allowing the attacker to select a favorable price for settlement. [4](#0-3) [5](#0-4)

```text
## Adversarial selection

Pull updates give users of Pyth Network some ability to select which price to use in a transaction.
This ability is highly circumscribed by various constraints: on-chain prices must move forward in time and cannot be from too far in the past.
However, users can still choose any price update that satisfies these constraints.
This ability is functionally equivalent to latency: it allows users to see the price in the future before using a price from the past.

The simplest way to guard against this attack vector is to incorporate a **staleness check** to ensure that the price used in a transaction is sufficiently recent.

The Pyth SDK provides the [getPriceNoOlderThan()](https://api-reference.pyth.network/price-feeds/evm/getPriceNoOlderThan) method to help users guard against this attack vector. This method returns the most recent price update that is not older than a specified threshold.
Highly latency-sensitive protocols may wish to reduce the threshold to a few seconds to better suit their needs.
Please also see the section below on [latency mitigations](#latency) for additional ideas on how latency-sensitive protocols can minimize the impact of oracle latency.
