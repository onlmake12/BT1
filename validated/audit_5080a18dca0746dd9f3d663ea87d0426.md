### Title
Unregistered `providerToCredit` Address in `executeCallback()` Enables Fee Theft After Exclusivity Period - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, the `executeCallback()` function accepts a caller-controlled `providerToCredit` address and credits fees to it without verifying that the address is a registered provider. After the exclusivity period expires, any caller can supply an arbitrary address as `providerToCredit`, redirecting the legitimate provider's earned fees to an attacker-controlled account.

### Finding Description

`Echo.executeCallback()` enforces that `providerToCredit == req.provider` only during the exclusivity window:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

After that window closes, the check is skipped entirely. The function then unconditionally credits fees to whatever address was passed:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

There is no subsequent check that `providerToCredit` is a registered provider (`_state.providers[providerToCredit].isRegistered`). Compare this with `requestPriceUpdatesWithCallback()`, which does enforce registration:

```solidity
require(
    _state.providers[provider].isRegistered,
    "Provider not registered"
);
```

The invariant that only registered providers should accrue fees is therefore broken in `executeCallback()` after the exclusivity period.

### Impact Explanation

An attacker who:
1. Registers as a provider (permissionless via `registerProvider()`),
2. Sets themselves as their own fee manager via `setFeeManager()`,
3. Waits for the exclusivity period on any pending request to expire,
4. Calls `executeCallback()` with their own address as `providerToCredit`,

will have the full `req.fee + msg.value - pythFee` credited to their `accruedFeesInWei` slot. They can then drain it via `withdrawAsFeeManager()`. The legitimate provider (`req.provider`) receives nothing for fulfilling the request. This is a direct, repeatable theft of provider fees.

### Likelihood Explanation

The exclusivity period is a configurable but finite window (`_state.exclusivityPeriodSeconds`). Any unfulfilled request that outlives this window is exploitable by any registered address. Because provider registration is permissionless and the attack requires only a single transaction after the window expires, the likelihood is high for any request that the legitimate provider is slow to fulfill.

### Recommendation

Add a registration check inside `executeCallback()` before crediting fees:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit is not a registered provider"
);
```

This mirrors the guard already present in `requestPriceUpdatesWithCallback()` and closes the gap.

### Proof of Concept

1. Attacker calls `registerProvider(0, 0, 0)` to register their address.
2. Attacker calls `setFeeManager(attackerAddress)` to set themselves as fee manager.
3. User calls `requestPriceUpdatesWithCallback(legitimateProvider, ...)`, paying a fee. `req.provider = legitimateProvider`, `req.fee = F`.
4. Attacker waits until `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`.
5. Attacker calls `executeCallback(attackerAddress, sequenceNumber, updateData, priceIds)`.
6. The exclusivity check is skipped. `_state.providers[attackerAddress].accruedFeesInWei += F - pythFee`.
7. Attacker calls `withdrawAsFeeManager(attackerAddress, F - pythFee)` and receives the funds.
