Looking at the vulnerability class from the external report — **shared/single storage slot overwritten by new state, making old state's funds/service permanently inaccessible** — I need to find an analog in Pyth where a hash-table slot is shared between concurrent states and a new entry evicts an old one, causing the old state's funds or callback to be permanently lost.

The `Echo.sol` contract contains exactly this pattern.

---

### Title
Use-After-Delete on Overflow Request in `executeCallback` Causes Silent Callback Drop and User Fee Loss — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol` uses a fixed-size 32-slot array (`_state.requests[NUM_REQUESTS]`) as a hash table for in-flight requests. When a new request collides with an existing active request at the same short key, `allocRequest` evicts the existing request to `_state.requestsOverflow`. In `executeCallback`, the storage reference `req` is obtained via `findActiveRequest`, then `clearRequest` is called. For overflow requests, `clearRequest` executes `delete _state.requestsOverflow[key]`, which **zeros all fields of the struct in storage**. The storage reference `req` still points to the now-zeroed slot. The subsequent callback invocation `IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` therefore calls `address(0)` with 0 gas — silently succeeding — while the provider has already been credited the user's fee.

### Finding Description

In `Echo.sol::requestPriceUpdatesWithCallback`, each request is stored via `allocRequest`: [1](#0-0) 

When a new request's short key collides with an existing active request, the existing request is evicted to `_state.requestsOverflow`. The evicted request is still valid and findable via `findRequest` / `findActiveRequest`.

In `executeCallback`, the flow is:

1. `req = findActiveRequest(sequenceNumber)` — for an overflow request, this returns a storage reference to `_state.requestsOverflow[key]`.
2. The provider fee is credited: `_state.providers[providerToCredit].accruedFeesInWei += ...` using `req.fee`.
3. `clearRequest(sequenceNumber)` is called. [2](#0-1) 

For overflow requests, `clearRequest` executes `delete _state.requestsOverflow[key]`, which **zeros every field** of the struct at that storage location.

4. The storage reference `req` now reads all-zero values: `req.requester == address(0)`, `req.callbackGasLimit == 0`.
5. The callback is invoked: [3](#0-2) 

Calling `address(0)` with 0 gas does not revert — it silently succeeds. The `try` block succeeds, `emitPriceUpdate` fires, and the protocol considers the request fulfilled.

The root cause is that `clearRequest` is called before the callback, and for overflow requests it destroys the storage the `req` reference points to. [4](#0-3) 

### Impact Explanation

- The user pays the full fee (`msg.value`) for a price update callback.
- The provider is credited the fee before `clearRequest`.
- After `clearRequest` zeros the overflow entry, the callback is called on `address(0)` with 0 gas — it silently does nothing.
- The user's consumer contract **never receives its `_echoCallback`**, but the fee is permanently transferred to the provider.
- There is no refund mechanism. The user's funds are effectively lost and the service is never delivered.

This is a direct analog to the external report: a shared storage slot (the 32-slot array indexed by a 5-bit short key) is overwritten by a new request, evicting the old one to overflow. The subsequent delete of the overflow entry makes the old request's callback permanently undeliverable while the fee is already gone.

### Likelihood Explanation

With only 32 slots and a 5-bit short key (`NUM_REQUESTS_MASK = 0x1f`), hash collisions are expected after ~32 concurrent active requests (birthday-bound ~23 requests for 50% collision probability). [5](#0-4) 

Any unprivileged user can:
1. Submit enough requests to fill the 32-slot array.
2. Submit one more request that collides with a victim's request, evicting it to overflow.
3. Call `executeCallback` for the victim's overflow request, triggering the use-after-delete.

No privileged access is required. The attacker only needs to pay the request fees.

### Recommendation

Move the callback invocation **before** `clearRequest`, or save the necessary fields (`requester`, `callbackGasLimit`) into memory variables before clearing:

```solidity
// Save fields before clearing
address callbackTarget = req.requester;
uint32 gasLimit = req.callbackGasLimit;

clearRequest(sequenceNumber);

try IEchoConsumer(callbackTarget)._echoCallback{gas: gasLimit}(sequenceNumber, priceFeeds) {
    emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
} catch { ... }
```

This mirrors the pattern already used correctly in `Entropy.sol`'s `revealWithCallback` path: [6](#0-5) 

### Proof of Concept

1. 32 concurrent requests are active, filling all slots in `_state.requests[32]`.
2. User A submits request with sequence number `N` such that `keccak256(N)[0] & 0x1f` collides with an existing slot. User A's request is evicted to `_state.requestsOverflow[keccak256(N)]`.
3. Provider calls `executeCallback(providerAddr, N, updateData, priceIds)`.
4. `findActiveRequest(N)` returns a storage reference to `_state.requestsOverflow[keccak256(N)]` — valid, contains `req.requester = UserA`, `req.fee = X`, `req.callbackGasLimit = G`.
5. `_state.providers[providerAddr].accruedFeesInWei += X + msg.value - pythFee` — provider credited.
6. `clearRequest(N)` → `delete _state.requestsOverflow[keccak256(N)]` → all fields zeroed.
7. `req.requester == address(0)`, `req.callbackGasLimit == 0`.
8. `IEchoConsumer(address(0))._echoCallback{gas: 0}(N, priceFeeds)` — silent no-op, `try` succeeds.
9. `emitPriceUpdate` fires. Protocol considers request fulfilled.
10. User A's consumer contract never receives its callback. Fee is permanently lost. [7](#0-6)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);

        // TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
        // a better solution would be a doubly-linked list of active requests.
        // After successful callback, update firstUnfulfilledSeq if needed
        while (
            _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
            !isActive(findRequest(_state.firstUnfulfilledSeq))
        ) {
            _state.firstUnfulfilledSeq++;
        }

        try
            IEchoConsumer(req.requester)._echoCallback{
                gas: req.callbackGasLimit
            }(sequenceNumber, priceFeeds)
        {
            // Callback succeeded
            emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
        } catch Error(string memory reason) {
            // Explicit revert/require
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                reason
            );
        } catch {
            // Out of gas or other low-level errors
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                "low-level error (possibly out of gas)"
            );
        }
    }

    function clearRequest(uint64 sequenceNumber) internal {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        Request storage req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            req.sequenceNumber = 0;
        } else {
            delete _state.requestsOverflow[key];
        }
    }
