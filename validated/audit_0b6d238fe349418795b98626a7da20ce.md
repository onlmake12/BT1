### Title
Permanent Subscription Deposit Limit Bypassed via Incremental `addFunds` Calls — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `addFunds` function in `Scheduler.sol` enforces `MAX_DEPOSIT_LIMIT` (100 ETH) by checking only the current transaction's `msg.value` against the limit, not the cumulative `status.balanceInWei + msg.value`. This allows any caller to bypass the deposit cap for permanent subscriptions through multiple sub-limit deposits, permanently locking an arbitrarily large amount of ETH in the contract.

### Finding Description

`Scheduler.sol` defines `MAX_DEPOSIT_LIMIT = 100 ether` as a hard cap on deposits into permanent subscriptions. Permanent subscriptions are irreversible: funds cannot be withdrawn and the subscription cannot be deactivated.

The `addFunds` function enforces this cap as follows:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

status.balanceInWei += msg.value;
```

The guard checks only `msg.value > MAX_DEPOSIT_LIMIT`, not `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`. The same flawed pattern exists in `createSubscription`:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Because `addFunds` is **permissionless** (no `onlyManager` modifier), any external address can call it for any active subscription. An attacker can:

1. Create a permanent subscription depositing exactly `MAX_DEPOSIT_LIMIT` (100 ETH).
2. Repeatedly call `addFunds` with `msg.value = MAX_DEPOSIT_LIMIT - 1 wei` (99.999... ETH), each call passing the guard.
3. After N such calls, `status.balanceInWei = 100 ETH + N × ~100 ETH`, far exceeding the intended cap.

The circuit breaker (the deposit limit) is checked per-transaction rather than against the post-state cumulative balance, exactly mirroring the Bond SDA pattern where `term.maxDebt` was checked after `market.totalDebt` was already updated.

### Impact Explanation

- `MAX_DEPOSIT_LIMIT` is the protocol's only safeguard against unbounded ETH being permanently locked in permanent subscriptions. Bypassing it defeats this invariant entirely.
- Since permanent subscriptions prohibit withdrawals (`CannotUpdatePermanentSubscription` is thrown on `withdrawFunds`), any ETH deposited beyond the limit is **permanently locked** in the contract.
- Because `addFunds` is permissionless, a third party can force-lock their own ETH into any active permanent subscription, inflating its balance without the subscription manager's consent. While the attacker bears the ETH cost, the protocol's accounting invariant is broken and the contract holds more ETH than governance intended.
- The excess balance is consumed by keepers over time, but at a rate bounded by the subscription's update frequency — meaning large excesses persist for extended periods.

### Likelihood Explanation

- The entry path is fully permissionless and requires no privileged role: any EOA or contract can call `addFunds` on any active subscription.
- The bypass requires only multiple transactions, each below `MAX_DEPOSIT_LIMIT`. No special conditions, oracle manipulation, or governance access are needed.
- The test suite (`testPermanentSubscriptionDepositLimit`) only tests that a single call with `msg.value > MAX_DEPOSIT_LIMIT` reverts; it does not test the cumulative bypass path, confirming the gap was not considered.

### Recommendation

Replace the per-transaction check with a cumulative post-state check in both `addFunds` and `createSubscription`:

```solidity
// In addFunds:
if (params.isPermanent && (status.balanceInWei + msg.value) > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;

// In createSubscription:
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
// (createSubscription is safe as-is since balanceInWei starts at 0, but should be consistent)
```

### Proof of Concept

```
MAX_DEPOSIT_LIMIT = 100 ETH

Step 1: createSubscription{value: 100 ETH}(permanentParams)
  → status.balanceInWei = 100 ETH  ✓ (passes: 100 ETH == MAX_DEPOSIT_LIMIT)

Step 2: addFunds{value: 99.9 ETH}(subscriptionId)
  → check: 99.9 ETH > 100 ETH? NO → passes
  → status.balanceInWei = 199.9 ETH  ← EXCEEDS LIMIT

Step 3: addFunds{value: 99.9 ETH}(subscriptionId)
  → check: 99.9 ETH > 100 ETH? NO → passes
  → status.balanceInWei = 299.8 ETH  ← EXCEEDS LIMIT FURTHER

Repeat N times → status.balanceInWei = 100 + N × 99.9 ETH (permanently locked, non-withdrawable)
```

The root cause is in `addFunds` at line 613 and `createSubscription` at line 48 of `Scheduler.sol`, where the guard validates `msg.value` in isolation rather than the resulting cumulative balance. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
