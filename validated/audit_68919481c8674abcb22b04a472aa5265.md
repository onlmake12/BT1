### Title
Silent u128→u64 Truncating Cast in `DaoCalculator::calculate_maximum_withdraw` Produces Wrong Withdrawal Capacity - (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the maximum withdrawable capacity for a NervosDAO cell using a u128 intermediate, then silently truncates it to u64 with `as u64`. Every other analogous u128→u64 conversion in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern. When the u128 result exceeds `u64::MAX`, the `as u64` cast wraps the value to a completely wrong (orders-of-magnitude too small) number instead of returning an error, corrupting the computed withdrawal capacity.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits of the u128 result. If `withdraw_counted_capacity` exceeds `u64::MAX`, the lower 64 bits are taken, producing a value that can be billions of times smaller than the correct answer — directly analogous to the CompoundV2 `10**28` vs `10**18` magnitude error.

All other u128→u64 conversions in the same file use the safe pattern:

```rust
// secondary_block_reward (line 204)
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch (line 258)
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) 

`calculate_maximum_withdraw` is the only site that uses the unsafe `as u64` cast.

---

### Impact Explanation

`calculate_maximum_withdraw` is called from three paths:

**Path 1 — RPC `calculate_dao_maximum_withdraw`** (`rpc/src/module/experiment.rs`): Returns a silently wrong (catastrophically small) withdrawal estimate to any RPC caller, misleading users about their entitled DAO interest.

**Path 2 — `transaction_fee`** (line 31): Used by the tx-pool to compute the fee of a DAO withdrawal transaction. A truncated `maximum_withdraw` makes `maximum_withdraw - outputs_capacity` wrong, corrupting fee accounting.

**Path 3 — `withdrawed_interests` → `dao_field_with_current_epoch`** (lines 316–317, 222): The `withdrawed_interests` value fed into the DAO field calculation becomes too small, causing `current_s` (the secondary issuance accumulator stored in every block header) to be inflated. This corrupts the on-chain DAO state, which is a consensus-level impact. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The overflow condition requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

`deposit_ar` starts at `10^16` and only grows; `withdrawing_ar ≥ deposit_ar`. The total CKB supply caps `counted_capacity` at roughly `3.36 × 10^18` shannons. For the product to exceed `u64::MAX ≈ 1.84 × 10^19`, the ratio `withdrawing_ar / deposit_ar` must exceed ~5.5×. Given the secondary issuance rate, this would take thousands of years on mainnet. [6](#0-5) 

On mainnet the likelihood is effectively zero. However:
- On a devnet or testnet with a high `secondary_epoch_reward` or low `initial_primary_epoch_reward`, the `ar` ratio can grow much faster, making the condition reachable in testing environments.
- The code inconsistency is a latent defect: every other arithmetic path in the same module is protected; this one is not.

---

### Recommendation

Replace the silent truncating cast with the same checked conversion used everywhere else in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64)

// After (consistent with the rest of the file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64)
``` [7](#0-6) 

---

### Proof of Concept

Numeric demonstration (no fork needed):

```
deposit_ar      = 10_000_000_000_000_000   (initial ar, 10^16)
withdrawing_ar  = 55_000_000_000_000_000   (ar after extreme growth, 5.5×)
counted_capacity = 3_360_000_000_000_000_000  (≈ total CKB supply in shannons)

withdraw_counted_capacity (u128)
  = 3_360_000_000_000_000_000 × 55_000_000_000_000_000
    / 10_000_000_000_000_000
  = 18_480_000_000_000_000_000   ← exceeds u64::MAX (18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 18_480_000_000_000_000_000 mod 2^64
  = 33_255_926_290_448_385       ← ~555× too small, no error returned
```

The function returns `Ok(Capacity::shannons(33_255_926_290_448_385 + occupied_capacity))` — a value hundreds of times smaller than the correct withdrawal amount — with no indication of error, directly mirroring the CompoundV2 magnitude scalar truncation.
