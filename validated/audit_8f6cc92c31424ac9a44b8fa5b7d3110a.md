### Title
No Per-Caller Authentication on Destructive RPC Methods Allows Unauthorized Tx-Pool Wipe and Network Manipulation — (File: `rpc/src/module/pool.rs`, `rpc/src/module/net.rs`)

---

### Summary

The CKB JSON-RPC server exposes several destructive, state-mutating methods — `clear_tx_pool`, `clear_tx_verify_queue`, `remove_transaction`, `set_ban`, `clear_banned_addresses`, and `set_network_active` — with no per-caller authentication or authorization check of any kind. The only access control is a coarse module-level enable/disable flag in the config. When the RPC server is bound to a non-loopback address (a fully supported configuration), any unauthenticated HTTP client can invoke these methods and wipe the mempool, remove targeted transactions, manipulate the peer ban list, or disable P2P networking entirely.

---

### Finding Description

**Root cause — no authentication layer in the RPC server**

The `Config` struct that governs the RPC server contains fields for `listen_address`, `modules`, `max_request_body_size`, and similar options, but has no authentication token, no IP allowlist enforcement, and no per-method authorization field. [1](#0-0) 

The module-level checks (`pool_enable()`, `net_enable()`, etc.) only gate whether a module's methods are registered at all — they do not restrict which callers may invoke them. [2](#0-1) 

**Affected methods — Pool module**

`clear_tx_pool` and `clear_tx_verify_queue` are registered as plain `#[rpc]` handlers with no guard, token check, or caller identity check: [3](#0-2) [4](#0-3) 

Their implementations call directly into the tx-pool service with no authorization step: [5](#0-4) [6](#0-5) 

`remove_transaction` is equally unguarded: [7](#0-6) 

**Affected methods — Net module**

`set_ban`, `clear_banned_addresses`, and `set_network_active` are all declared in `NetRpc` with no authentication guard. The grep confirms all three are present in `rpc/src/module/net.rs`. [8](#0-7) 

**Supported non-localhost binding**

The default `listen_address` is `127.0.0.1:8114`, but the config field is a plain `String` that operators routinely change for remote management. The documentation warns against public exposure but provides no code-level enforcement: [9](#0-8) 

The `Pool` and `Net` modules are enabled by default in the production config: [10](#0-9)

### Impact Explanation

| Method | Impact |
|---|---|
| `clear_tx_pool` | Wipes every pending and proposed transaction from the mempool. All user-submitted transactions are silently discarded and must be resubmitted. |
| `clear_tx_verify_queue` | Drops all transactions currently being verified, causing them to be lost without confirmation. |
| `remove_transaction(hash)` | Surgically evicts a targeted transaction (e.g., a high-value or time-sensitive one) from the pool. |
| `set_ban(addr, "insert")` | Bans legitimate peers, degrading or severing P2P connectivity. |
| `clear_banned_addresses` | Removes all bans, re-admitting previously blocked malicious peers. |
| `set_network_active(false)` | Disconnects the node from the entire P2P network. |

The most severe single-call impact is `clear_tx_pool`: it causes complete loss of all unconfirmed transactions on the targeted node, directly harming users whose transactions were pending. Combined with `set_network_active(false)`, an attacker can isolate the node and prevent resubmission.

---

### Likelihood Explanation

Many CKB node operators expose their RPC port for legitimate remote management (mining pools, monitoring dashboards, automated tooling). The configuration explicitly supports non-localhost binding via `listen_address`. There is no authentication mechanism to add even if an operator wanted one — the `Config` struct has no such field. Any attacker who can reach the RPC port over HTTP (including via SSRF from a co-located service) can trigger these methods with a single JSON-RPC call.

---

### Recommendation

1. Add an optional `auth_token` or `allowed_ips` field to `Config` in `util/app-config/src/configs/rpc.rs`. Enforce it as middleware in the RPC server before dispatching any method.
2. Separate destructive/administrative methods (`clear_tx_pool`, `set_ban`, `set_network_active`) into a distinct module (e.g., `Admin`) that is **disabled by default** and requires explicit opt-in, analogous to how `IntegrationTest` and `Debug` modules are disabled by default.
3. At minimum, document that `clear_tx_pool`, `clear_tx_verify_queue`, `remove_transaction`, `set_ban`, `clear_banned_addresses`, and `set_network_active` are administrative operations that must never be reachable from untrusted callers, and enforce this with a compile-time or startup-time warning when `listen_address` is not loopback.

---

### Proof of Concept

With the RPC exposed on any non-loopback address, an unprivileged attacker sends:

```bash
# Wipe the entire mempool
curl -X POST http://<node-ip>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}'

# Wipe the verification queue
curl -X POST http://<node-ip>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_verify_queue","params":[],"id":2}'

# Disable P2P networking entirely
curl -X POST http://<node-ip>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":3}'

# Ban a legitimate peer
curl -X POST http://<node-ip>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_ban","params":["<peer-ip>","insert",null,null,"attacker"],"id":4}'
```

No credentials, tokens, or signatures are required. Each call succeeds immediately and takes effect on the live node state.
