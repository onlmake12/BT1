### Title
`reclaim_rent` / `reclaim_twap_rent` hard-couple SOL refund recipient and `write_authority`, permanently locking rent for any PDA-owned price update account — (`File: target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

`ReclaimRent` and `ReclaimTwapRent` require a single `payer: Signer` to simultaneously be the SOL lamport refund recipient **and** the stored `write_authority` of the price update account. Any `PriceUpdateV2` or `TwapUpdate` account whose `write_authority` is a PDA — including every account created by the production `pyth-push-oracle` program — can never have its rent reclaimed. The SOL is permanently locked.

---

### Finding Description

In `pyth-solana-receiver`, the `ReclaimRent` account struct enforces:

```rust
#[derive(Accounts)]
pub struct ReclaimRent<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(mut, close = payer,
        constraint = price_update_account.write_authority == payer.key()
            @ ReceiverError::WrongWriteAuthority)]
    pub price_update_account: Account<'info, PriceUpdateV2>,
}
``` [1](#0-0) 

`ReclaimTwapRent` is identical in structure:

```rust
#[account(mut, close = payer,
    constraint = twap_update_account.write_authority == payer.key()
        @ ReceiverError::WrongWriteAuthority)]
``` [2](#0-1) 

The `payer` must be a `Signer<'info>` (i.e., must have signed the top-level transaction) **and** must equal `price_update_account.write_authority`. These are two distinct roles:

- **SOL refund recipient** (`close = payer`): the wallet that receives the lamports when the account is closed.
- **Data authority** (`write_authority`): the entity authorized to write to or close the price update account, stored on-chain by `post_price_update_from_vaa`:

```rust
price_update_account.write_authority = write_authority.key();
``` [3](#0-2) 

The production `pyth-push-oracle` program calls `post_update` via CPI and explicitly sets `write_authority` to the `price_feed_account` PDA (seeds `[shard_id, feed_id]`):

```rust
write_authority: ctx.accounts.price_feed_account.to_account_info().clone(),
``` [4](#0-3) 

with PDA seeds:

```rust
#[account(mut, seeds = [&shard_id.to_le_bytes(), &feed_id], bump)]
pub price_feed_account: UncheckedAccount<'info>,
``` [5](#0-4) 

After this CPI, the `write_authority` field stored in every push-oracle-created `PriceUpdateV2` account is the PDA's pubkey. A PDA cannot sign a top-level Solana transaction — it can only sign via CPI with seeds. Because `reclaim_rent` is a top-level instruction (not a CPI wrapper in `pyth-push-oracle`), there is no code path that can satisfy `payer: Signer` with `payer.key() == PDA_pubkey`. The constraint is permanently unsatisfiable for all such accounts.

The same applies to any third-party integrator that calls `post_update` or `post_update_atomic` via CPI and passes a PDA as `write_authority` — a standard and documented CPI pattern.

---

### Impact Explanation

Every `PriceUpdateV2` account created through `pyth-push-oracle` has its rent permanently locked. The `pyth-push-oracle` is a production program that creates one persistent price feed account per `(shard_id, feed_id)` pair. None of these accounts can ever be closed via `reclaim_rent`. The same applies to `TwapUpdate` accounts via `reclaim_twap_rent` for any CPI caller using a PDA write authority.

The locked amount per account is the rent-exempt minimum for `PriceUpdateV2::LEN = 134` bytes (~0.00189 SOL at current rent rates). Across all active price feeds and shards, this accumulates to a non-trivial and irrecoverable SOL balance. Additionally, any third-party protocol that integrates `post_update` via CPI with a PDA write authority faces the same permanent rent lock, blocking a standard and expected integration pattern.

---

### Likelihood Explanation

The `pyth-push-oracle` program is already deployed and actively creates price feed accounts with PDA write authorities. Every such account is already affected. Any new integrator following the CPI pattern demonstrated by `pyth-push-oracle` will encounter the same issue. The root cause is structural — it requires no special conditions, no attacker, and no race. It is triggered by the normal operation of the push oracle.

---

### Recommendation

Decouple the SOL refund recipient from the `write_authority` check in both `ReclaimRent` and `ReclaimTwapRent`. Introduce a separate `write_authority: Signer<'info>` that is checked against `price_update_account.write_authority`, while `payer` (the lamport refund destination) remains a separate, unconstrained mutable account:

```rust
#[derive(Accounts)]
pub struct ReclaimRent<'info> {
    #[account(mut)]
    pub payer: SystemAccount<'info>,          // receives lamports, need not sign
    #[account(mut, close = payer,
        constraint = price_update_account.write_authority == write_authority.key()
            @ ReceiverError::WrongWriteAuthority)]
    pub price_update_account: Account<'info, PriceUpdateV2>,
    pub write_authority: Signer<'info>,       // authorizes the close
}
```

`pyth-push-oracle` would then need a wrapper instruction that calls `reclaim_rent` via CPI, signing with the PDA seeds, and directing lamports to a caller-specified refund address.

---

### Proof of Concept

1. `pyth-push-oracle::update_price_feed` is called for feed `F` on shard `S`. It CPIs into `pyth-solana-receiver::post_update` with `write_authority = price_feed_account` (PDA with seeds `[S, F]`). [6](#0-5) 

2. Inside `post_price_update_from_vaa`, the stored field is set: `price_update_account.write_authority = write_authority.key()` — i.e., the PDA pubkey. [3](#0-2) 

3. Any caller now attempts `reclaim_rent` on this account. The Anchor constraint `price_update_account.write_authority == payer.key()` requires `payer` to equal the PDA pubkey. [7](#0-6) 

4. `payer` is `Signer<'info>`, requiring a valid transaction signature from that key. A PDA cannot produce a transaction signature. The instruction reverts with `WrongWriteAuthority` for any non-PDA payer, and is structurally impossible to satisfy with the PDA itself.

5. The rent in the price feed account is permanently locked with no recovery path.
