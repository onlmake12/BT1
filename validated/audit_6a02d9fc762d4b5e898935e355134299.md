Audit Report

## Title
Infinite loop in `kill_process` when process ignores SIGTERM — (`ckb-bin/src/subcommand/daemon.rs`)

## Summary
`kill_process` sends SIGTERM and polls `check_process` in a loop, decrementing `wait_time` in the `Ok` arm. The force-kill guard `_ if wait_time <= 0` is placed in a catch-all arm that is only reachable when `check_process` returns `Err` (process already exited). If the process stays alive and ignores SIGTERM, `wait_time` decrements past zero indefinitely and SIGKILL is never sent, causing `ckb daemon --stop` to hang forever.

## Finding Description
In `kill_process` (`ckb-bin/src/subcommand/daemon.rs`, lines 61–86), the match has three arms:

1. `Ok(_)` — fires when `check_process` returns `Ok` (process alive); decrements `wait_time`, sleeps 1 second. This arm exhaustively matches all `Ok` variants.
2. `_ if wait_time <= 0` — catch-all with guard; only reachable when `check_process` returns `Err`. The guard is evaluated only after arm 1 fails to match, which requires the process to have already exited.
3. `_` — catch-all; only reachable when `check_process` returns `Err` and `wait_time > 0`.

Because arm 1 consumes every `Ok` result, arms 2 and 3 are only reachable post-exit. The intended escalation to SIGKILL after 60 seconds is structurally unreachable in the scenario it was designed for. `wait_time` goes negative without bound and the loop never breaks.

`check_process` uses `kill(pid, None)` (signal 0) to probe liveness: `Ok` = alive, `Err` = gone. A process that ignores SIGTERM remains alive, so `check_process` keeps returning `Ok`, arm 1 keeps firing, and the SIGKILL path is never reached.

## Impact Explanation
`ckb daemon --stop` hangs indefinitely when the target process defers or ignores SIGTERM. The operator loses the ability to stop the node through the normal CLI path. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local command line crash/hang**.

## Likelihood Explanation
Any CKB process that defers SIGTERM handling (e.g., during a long RocksDB flush, a blocked syscall, or a custom signal handler that delays shutdown) triggers this path. The operator runs `ckb daemon --stop`, SIGTERM is sent but not acted on within 60 seconds, and the CLI hangs. This requires only local access to the node host and is a realistic operational scenario.

## Recommendation
Move the timeout check into the `Ok` arm so it is evaluated while the process is still alive:

```rust
Ok(_) => {
    wait_time -= 1;
    if wait_time <= 0 {
        kill(Pid::from_raw(pid), Some(Signal::SIGKILL)).map_err(|_| ExitCode::Failure)?;
        break;
    }
    eprint!("{}", ".".yellow());
    let _ = io::stderr().flush();
    std::thread::sleep(std::time::Duration::from_secs(1));
}
_ => { break; }
```

## Proof of Concept
Mock `check_process` to always return `Ok(pid)`. Call `kill_process`. Assert the loop exits within 61 iterations and that SIGKILL was sent. With the current code, the loop never exits and the assertion fails.

Alternatively: start a process that installs a SIGTERM handler that does nothing (`signal(SIGTERM, SIG_IGN)`), write its PID to a pid file, and run `ckb daemon --stop` — the CLI will hang indefinitely with dots printing forever.
