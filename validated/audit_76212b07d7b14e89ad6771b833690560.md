### Title
Registered Echo providers can never be deregistered, leaving compromised providers permanently active — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `registerProvider()` sets `isRegistered = true` on a provider's `ProviderInfo` struct with no corresponding deregistration path. Neither the provider itself nor the admin can ever set `isRegistered = false`. This is a direct structural analog to M-02: a one-way registry that can only grow, never shrink.

---

### Finding Description

`registerProvider()` in `Echo.sol` permanently marks a provider as registered:

```solidity
// Echo.sol lines 381–393
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    provider.feePerFeedInWei = feePerFeedInWei;
    provider.feePerGasInWei = feePerGasInWei;
    provider.isRegistered = true;
    emit ProviderRegistered(msg.sender, feePerGasInWei);
}
``` [1](#0-0) 

The `isRegistered` flag lives in `ProviderInfo` in `EchoState.sol`:

```solidity
// EchoState.sol lines 31–46
struct ProviderInfo {
    uint96 baseFeeInWei;
    uint96 feePerFeedInWei;
    uint96 feePerGasInWei;
    uint128 accruedFeesInWei;
    address feeManager;
    bool isRegistered;
}
``` [2](#0-1) 

The entire `Echo.sol` and its admin surface (`setDefaultProvider`, `setExclusivityPeriod`, `setProviderFee`) contain **no function that sets `isRegistered = false`** or otherwise disables a provider. A grep across the entire `echo/` directory confirms zero matches for `unregister`, `deregister`, `setRegistered`, or `isRegistered.*false`. [3](#0-2) 

The gate that enforces provider validity at request time is:

```solidity
// Echo.sol lines 58–60
require(
    _state.providers[provider].isRegistered,
    "Provider not registered"
);
``` [4](#0-3) 

Because `isRegistered` can never be cleared, any provider address that was ever registered will pass this check forever.

---

### Impact Explanation

If a registered provider's private key is compromised, or the provider turns malicious:

1. The attacker controls the provider address and can receive new `requestPriceUpdatesWithCallback` calls directed at it.
2. The attacker can choose not to call `executeCallback`, causing the request to go unfulfilled. The user's fee is already collected and non-refundable.
3. The admin's only lever is `setDefaultProvider`, which redirects the *default* path — but users who explicitly pass the compromised provider address in `requestPriceUpdatesWithCallback` are unaffected by that change.
4. There is no emergency circuit-breaker: the protocol cannot prevent new requests from being routed to the compromised provider, and cannot force-fulfill or cancel in-flight requests.

The result is permanent, irremovable exposure: the compromised provider remains a valid target for user requests indefinitely, with no on-chain remedy short of a contract upgrade.

---

### Likelihood Explanation

Provider key compromise is a realistic operational risk for any long-lived service. The Entropy/Echo model explicitly anticipates provider rotation (Entropy's `register()` supports re-registration to rotate commitments), yet Echo provides no analogous escape hatch. Any provider that loses key control, is abandoned, or is taken over by an attacker becomes a permanent fee-drain trap for users who continue to route requests to it.

---

### Recommendation

Add an admin-controlled deregistration function:

```solidity
function deregisterProvider(address provider) external {
    require(msg.sender == _state.admin, "Only admin");
    _state.providers[provider].isRegistered = false;
    emit ProviderDeregistered(provider);
}
```

Alternatively, mirror the pattern used in `Entropy.sol` where a provider can signal retirement by setting their fee or commitment to a sentinel value, or add a boolean `isActive` flag that the admin can toggle independently of `isRegistered`, so that historical accounting is preserved while new requests are blocked.

---

### Proof of Concept

1. Provider `P` calls `registerProvider(fee, fee, fee)` → `_state.providers[P].isRegistered = true`. [5](#0-4) 

2. Attacker compromises `P`'s private key.

3. Admin calls `setDefaultProvider(safeProvider)` to redirect the default path — but this does **not** affect `P.isRegistered`. [6](#0-5) 

4. User calls `requestPriceUpdatesWithCallback(P, publishTime, priceIds, gasLimit)` with explicit provider `P`. The check `require(_state.providers[P].isRegistered)` passes because `isRegistered` is still `true`. [4](#0-3) 

5. The attacker, controlling `P`, never calls `executeCallback`. The user's fee is consumed; the callback is never delivered. No on-chain mechanism exists to prevent step 4 from repeating indefinitely.
