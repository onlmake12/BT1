Audit Report

## Title
`DownloadScheduler::decrease` Uses Wrong Operand, Bypassing Slow-Response Accumulator — (File: sync/src/types/mod.rs)

## Summary
In `sync/src/types/mod.rs`, `DownloadScheduler::decrease` writes `self.task_count.saturating_add(num)` into `self.timeout_count` instead of `self.timeout_count.saturating_add(num)`. Because `task_count` initialises to 32 (`INIT_BLOCKS_IN_TRANSIT_PER_PEER`), the result is always ≥ 33, which always exceeds the guard threshold of 2. Consequently `task_count` is decremented on every slow block response rather than only after accumulating three or more slow responses, causing legitimate high-latency peers to be disconnected ~3× faster than the protocol intends.

## Finding Description
`DownloadScheduler` defaults to `task_count = 32`, `timeout_count = 0` (lines 491–495). The `decrease` method at lines 521–527 reads:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num); // wrong operand
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

The intended accumulator is `self.timeout_count`, but the code reads from `self.task_count`. Since `task_count` starts at 32, `32 + 1 = 33 > 2` is always true, so the branch fires unconditionally and `task_count` is decremented on every call. `timeout_count` is reset to 0 immediately after, so it never accumulates anything.

`decrease` is called from `remove_by_block` (lines 801–810) whenever a received block's elapsed time falls in the `NormalToUpper` (penalty 1) or `UpperToMax` (penalty 2) quantile and `should_punish` is true (peer count > `protect_num`, default 4). When `task_count` reaches 0, `prune` adds the peer to `disconnect_list` (lines 692–700).

## Impact Explanation
The intended design gives each peer a grace window of at least 3 slow responses before `task_count` is decremented. The bug removes that window entirely: every single slow response decrements `task_count`. Starting from 32, a peer reaches `task_count = 0` after 32 slow responses instead of the intended ~96+. Legitimate peers on high-latency links are disconnected ~3× faster than the protocol intends, degrading sync reliability and peer-set stability. This maps to **Low (501–2000 points): important performance/behaviour improvement for CKB** — the peer-management invariant is broken, harming sync throughput and connection stability.

## Likelihood Explanation
The condition `should_punish` (peer count > 4) is satisfied on any normally-connected mainnet node. The `adjustment` flag defaults to `true`. Any peer whose block responses fall in the upper latency quantiles triggers this path. An adversarial peer can deliberately pace responses into the slow quantile (above `normal_time`, below the 30 s hard timeout) to exhaust its own `task_count` in 32 responses and force disconnection, then reconnect and repeat. No special privileges are required.

## Recommendation
Change line 522 in `sync/src/types/mod.rs` from:

```rust
self.timeout_count = self.task_count.saturating_add(num);
```

to:

```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

This restores the intended accumulator semantics so that `task_count` is only decremented after the accumulated slow-response count exceeds 2.

## Proof of Concept
1. Construct a `DownloadScheduler` with default values (`task_count = 32`, `timeout_count = 0`).
2. Call `decrease(1)` once.
3. Assert `task_count == 31` — confirms the branch fires immediately (bug present).
4. With the fix applied, assert `task_count == 32` and `timeout_count == 1` after the first call, and `task_count == 31` only after the third call (`timeout_count` accumulates to 3 > 2).
