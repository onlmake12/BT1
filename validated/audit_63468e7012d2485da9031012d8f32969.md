### Title
DAO Withdrawal False Rejection via `header_dep_index` Interpretation Mismatch Between Rust `DaoCalculator` and C DAO Script — (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw()` reads the full u64 `header_dep_index` from the witness and uses it directly as a `usize` array index into `header_deps`. The deployed C DAO script (`dao.c`) reads only the **lowest byte** of that same u64. When `header_dep_index >= 256`, the two implementations index into different slots of `header_deps`, causing the Rust node to reject a DAO withdrawal transaction that the C VM would accept.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw()` decodes the deposit-block pointer from the witness `input_type` field and resolves it:

```rust
// line 91-98
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // ← full u64 used as index
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

The C DAO script (referenced at `test/src/specs/dao/dao_user.rs:14`) interprets the same 8-byte field using only its **lowest byte** as the index. For any `header_dep_index` whose lowest byte is `b` but whose full value is `b + 256k` (k ≥ 1):

| Layer | Index used | `header_deps` slot resolved |
|---|---|---|
| C DAO script | `b` (lowest byte) | deposit block hash |
| Rust `DaoCalculator` | `b + 256k` (full u64) | a different hash (e.g., withdraw block) |

The Rust node then performs a block-number consistency check:

```rust
// line 105-107
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
```

Because the Rust code resolved the wrong slot (e.g., the withdraw block at position 257 instead of the deposit block at position 1), the block number does not match the cell data, and the function returns `DaoError::InvalidOutPoint`.

This error propagates through `check_tx_fee` in `tx-pool/src/util.rs`:

```rust
// line 34-41
let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
    .transaction_fee(rtx)
    .map_err(|err| {
        Reject::Malformed(format!("{err}"), "expect (outputs capacity) <= (inputs capacity)".to_owned())
    })?;
```

The transaction is rejected with `Reject::Malformed` before it ever reaches script execution.

The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents and confirms this behavior:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
// ...
// Rust resolves index 257 → withdraw block (number 200), but cell data
// says deposited at block 100. Block number check catches the mismatch.
assert!(result.is_err(), "expected Err, got {result:?}");
```

---

### Impact Explanation

A DAO depositor who constructs a phase-2 withdrawal transaction where `header_dep_index >= 256` (i.e., the deposit block hash sits at position `>= 256` in the transaction's `header_deps` list) will have their transaction permanently rejected by the Rust node's tx-pool with `Malformed("InvalidOutPoint", ...)`. The C DAO script would accept the same transaction. The user cannot complete their DAO withdrawal through the normal RPC/tx-pool path. Funds remain locked in the DAO cell; no funds are lost, but the withdrawal is blocked until the user reconstructs the transaction with a `header_dep_index < 256`.

---

### Likelihood Explanation

The `header_deps` list is constructed from a `HashSet` in the reference `DAOUser::withdraw()` implementation (`test/src/specs/dao/dao_user.rs:155-161`), making the ordering non-deterministic. A user withdrawing from many DAO cells simultaneously (deposit + prepare headers for each cell) can easily produce a `header_deps` list with more than 256 entries, placing the deposit block hash at an index `>= 256`. Any wallet or script that builds withdrawal transactions with large `header_deps` sets is affected. The scenario is realistic for power users or automated withdrawal scripts.

---

### Recommendation

In `transaction_maximum_withdraw()` (`util/dao/src/lib.rs`, line 96), mask the index to its lowest byte before indexing, matching the C DAO script's behavior:

```rust
.and_then(|header_dep_index| {
    let index = (header_dep_index & 0xFF) as usize;  // match dao.c lowest-byte semantics
    rtx.transaction
        .header_deps()
        .get(index)
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

Update the test `check_dao_withdraw_header_dep_index_exceeds_u8` to assert `result.is_ok()` after the fix, confirming the Rust node now agrees with the C VM.

**Concrete scenario:**

1. User has ≥ 256 unique header deps in a DAO phase-2 withdrawal transaction (e.g., many DAO cells being withdrawn simultaneously, with deposit and prepare block hashes collected into a `HashSet` whose iteration order places the deposit block hash at index 257).
2. Witness `input_type` = `257u64` in little-endian (lowest byte = 1, pointing to the deposit block at `header_deps[1]`).
3. User submits via `send_transaction` RPC.
4. `check_tx_fee` → `DaoCalculator::transaction_fee()` → `transaction_maximum_withdraw()` reads full index 257 → resolves `header_deps[257]` (the withdraw block, block number 200) → block number check: 200 ≠ 100 (cell data) → `DaoError::InvalidOutPoint` → `Reject::Malformed`.
5. Transaction is rejected. The C DAO script, using index `257 & 0xFF = 1`, would resolve `header_deps[1]` (deposit block, block number 100) → block number check: 100 == 100 → accept.
6. User's valid DAO withdrawal is permanently blocked at the tx-pool layer.
