### Title
Unvalidated `providerToCredit` Address in `executeCallback` Enables Fee Theft After Exclusivity Period — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary
`Echo.executeCallback` accepts a caller-supplied `providerToCredit` address and unconditionally credits the full request fee to `_state.providers[providerToCredit].accruedFeesInWei`. The exclusivity-period guard only enforces the legitimate provider during a short window; once that window expires the check is dropped entirely and any address — including one pre-registered by an attacker — may be supplied. Because `registerProvider` is permissionless, an attacker can pre-register, front-run the legitimate provider's fulfillment transaction, and drain the fee to their own balance.

---

### Finding Description
`executeCallback` enforces `providerToCredit == req.provider` only while `block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds`.

```solidity
// Echo.sol lines 114-121
if (
    block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

After that window the parameter is completely unconstrained:

```solidity
// Echo.sol lines 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [1](#0-0) [2](#0-1) 

Provider registration is permissionless:

```solidity
// Echo.sol lines 381-393
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    ...
    provider.isRegistered = true;
``` [3](#0-2) 

A registered provider can set any address as fee manager and withdraw accrued fees:

```solidity
// Echo.sol lines 360-378
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(msg.sender == _state.providers[provider].feeManager, "Only fee manager");
    ...
    _state.providers[provider].accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [4](#0-3) 

There is no check anywhere in `executeCallback` that `providerToCredit` is a registered provider, that it matches `req.provider`, or that it is non-zero after the exclusivity window closes.

---

### Impact Explanation
The legitimate provider who is assigned to a request loses the entire `req.fee` they are owed. The fee is credited to an attacker-controlled address and can be fully withdrawn. For a zero-address supply the fee is permanently locked in the contract. In both cases the requester's payment is misdirected and the legitimate provider receives nothing for fulfilling the request.

---

### Likelihood Explanation
`registerProvider` is permissionless and costs only gas. `executeCallback` is a public `payable` function callable by anyone. An attacker only needs to monitor the mempool for pending `executeCallback` transactions from the legitimate provider and front-run them with `providerToCredit` set to their own pre-registered address. On chains with a public mempool this is straightforward. Even without front-running, if the legitimate provider delays past the exclusivity period the attacker can call first with no competition.

---

### Recommendation
After the exclusivity period, restrict `providerToCredit` to `req.provider` (the address that was assigned the request), or at minimum require that `providerToCredit` is a registered provider and emit a warning when it differs from `req.provider`. The simplest fix is to remove the conditional and always enforce `providerToCredit == req.provider`:

```solidity
require(
    providerToCredit == req.provider,
    "providerToCredit must be the assigned provider"
);
```

If the intent is to allow a different relayer to be credited after the exclusivity period, the contract must at minimum validate `providerToCredit != address(0)` and that it is a registered provider, and the design should be reconsidered to prevent fee theft from the originally assigned provider.

---

### Proof of Concept

1. **Setup**: Attacker calls `registerProvider(0, 0, 0)` to register `attacker` as a provider. Attacker calls `setFeeManager(attacker)` so they can later withdraw.

2. **User request**: A user calls `requestPriceUpdatesWithCallback(legitimateProvider, publishTime, priceIds, gasLimit)` paying `req.fee` in ETH. The request is stored with `req.provider = legitimateProvider`.

3. **Wait**: Attacker waits until `block.timestamp >= req.publishTime + _state.exclusivityPeriodSeconds`. The exclusivity guard is now inactive.

4. **Front-run**: Attacker calls:
   ```solidity
   echo.executeCallback(
       attacker,          // providerToCredit — attacker's own address
       sequenceNumber,
       updateData,
       priceIds
   );
   ```
   The check at line 114 is skipped. Line 161 executes:
   ```
   _state.providers[attacker].accruedFeesInWei += req.fee - pythFee
   ```

5. **Drain**: Attacker calls `withdrawAsFeeManager(attacker, amount)`. The `msg.sender.call{value: amount}("")` at line 375 transfers the stolen fee to the attacker. [5](#0-4) [6](#0-5) [7](#0-6)
