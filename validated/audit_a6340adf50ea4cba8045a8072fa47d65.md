Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Permanently Over-Counted via Stale Snapshot Overwrite After Cell-Ref-Parent Evictions — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` computes and returns new totals as local values without modifying `self.total_tx_size`/`self.total_tx_cycles`. When `check_and_record_ancestors` subsequently evicts cell-ref-parent transactions, `remove_entry` → `update_stat_for_remove_tx` correctly decrements those fields in place. However, `add_entry` then unconditionally overwrites both fields with the stale pre-eviction locals, permanently inflating pool occupancy counters by the aggregate size and cycles of all evicted transactions. The over-count accumulates across repeated invocations and is never self-correcting.

## Finding Description
In `add_entry` at lines 210–219: [1](#0-0) 

`updated_stat_for_add_tx` (lines 711–729) computes `self.total_tx_size + tx_size` and returns it as a local — it does **not** write to `self.total_tx_size`: [2](#0-1) 

`check_and_record_ancestors` (lines 603–625) fires when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count`, calling `remove_entry_and_descendants` → `remove_entry`: [3](#0-2) 

`remove_entry` calls `update_stat_for_remove_tx` at line 247, which **directly mutates** `self.total_tx_size` and `self.total_tx_cycles`: [4](#0-3) [5](#0-4) 

Those in-place decrements are then immediately discarded when `add_entry` writes the stale pre-eviction locals back at lines 218–219. The `recompute_total_stat` safety net inside `update_stat_for_remove_tx` only fires on underflow (`checked_sub` failure); an over-count never triggers underflow, so the phantom bytes persist indefinitely.

## Impact Explanation
`total_tx_size` is the sole guard in `limit_size` deciding whether to evict further transactions: [6](#0-5) 

An inflated `total_tx_size` causes spurious evictions of legitimate pending/proposed transactions and spurious `Reject::Full` rejections of new submissions via `updated_stat_for_add_tx`. Because the over-count accumulates with each attack iteration, an attacker can drive `total_tx_size` arbitrarily high until the pool permanently rejects all new transactions from all users. This matches **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**: the mempool of targeted nodes becomes effectively non-functional, blocking transaction propagation across the network.

## Likelihood Explanation
The trigger requires no privileged access, no majority hash power, and no victim mistakes. An attacker needs only a single live on-chain cell and slightly more than `max_ancestors_count` (default: 125) transactions each referencing that cell as a `cell_dep`. These are standard, low-fee transactions. The attack is fully repeatable: each round inflates the counters by the aggregate size of the evicted transactions, and rounds can be chained until the pool is permanently saturated. Any unprivileged P2P or RPC submitter can execute this.

## Recommendation
Move the `updated_stat_for_add_tx` call to **after** `check_and_record_ancestors` completes, so it reads the already-adjusted `self.total_tx_size`/`self.total_tx_cycles`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Moving `updated_stat_for_add_tx` after evictions is safe because evictions only reduce the totals, making the overflow check less likely to fire, not more.

## Proof of Concept
1. Configure a node with `max_ancestors_count = 125` and `max_tx_pool_size = 130_500` bytes.
2. Publish a live cell `C` on-chain.
3. Submit 130 transactions `T1…T130`, each with `cell_dep = C` and `size = 1000` bytes. Pool: `total_tx_size = 130_000`.
4. Submit transaction `TX` (size = 1000) spending `C` as an input.
   - `get_tx_ancenstors` returns `ancestors = {T1…T130}`, `cell_ref_parents = {T1…T130}`, `ancestors_count = 131`.
   - `131 - 130 = 1 ≤ 125` → eviction branch fires.
   - `updated_stat_for_add_tx` snapshots `total_tx_size_local = 131_000` (pre-eviction).
   - `check_and_record_ancestors` evicts 6 `Ti` entries; `update_stat_for_remove_tx` runs 6×: `self.total_tx_size` becomes `≈125_000`.
   - `add_entry` writes stale snapshot: `self.total_tx_size = 131_000` (phantom 6_000 bytes injected).
5. `limit_size` fires (`131_000 > 130_500`), evicting legitimate transactions even though actual occupancy is `≈125_000` bytes.
6. Repeat steps 3–4 to accumulate phantom bytes until `total_tx_size` permanently exceeds `max_tx_pool_size`, causing all subsequent submissions to be rejected with `Reject::Full`.
