### Title
`Echo.executeCallback` Only Verifies 8-Byte Price ID Prefix, Allowing Price Feed Substitution — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` stores only the first 8 bytes of each requested price ID to save gas. `Echo.executeCallback` then verifies only those 8 bytes against the caller-supplied `priceIds`, and passes the **caller-supplied** (not the originally stored) full price IDs directly to `IPyth.parsePriceFeedUpdates`. Because the full 32-byte price ID is never stored and never re-verified, any caller of `executeCallback` can substitute a different Pyth price feed that shares the same 8-byte prefix, causing the consumer contract to receive price data for the wrong asset.

---

### Finding Description

**Step 1 — Request stores only 8-byte prefix:**

In `requestPriceUpdatesWithCallback`, the contract deliberately truncates each 32-byte price ID to its first 8 bytes before writing to storage:

```solidity
// Copy only the first 8 bytes of each price ID to storage
for (uint8 i = 0; i < priceIds.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    req.priceIdPrefixes[i] = prefix;
}
``` [1](#0-0) 

The full 32-byte price IDs are only emitted in an event (`PriceUpdateRequested`) and are **never persisted on-chain**. [2](#0-1) 

**Step 2 — Callback verifies only the 8-byte prefix:**

In `executeCallback`, the verification loop extracts only the first 8 bytes of each caller-supplied price ID and compares it to the stored prefix:

```solidity
for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    if (prefix != req.priceIdPrefixes[i]) {
        revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
    }
}
``` [3](#0-2) 

**Step 3 — Caller-controlled price IDs are forwarded to Pyth:**

Immediately after the prefix-only check, the **caller-supplied** `priceIds` (not the original ones) are passed directly to `parsePriceFeedUpdates`:

```solidity
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(
    updateData,
    priceIds,   // <-- attacker-controlled
    SafeCast.toUint64(req.publishTime),
    SafeCast.toUint64(req.publishTime)
);
``` [4](#0-3) 

The consumer's `_echoCallback` then receives `priceFeeds` derived from the substituted price IDs, not the originally requested ones. [5](#0-4) 

---

### Impact Explanation

A caller of `executeCallback` (any address after the exclusivity period, or the assigned provider during it) can:

1. Find or construct a Pyth price feed ID `B` whose first 8 bytes match those of the originally requested feed `A`.
2. Supply valid Wormhole-attested `updateData` for feed `B` along with `priceIds = [B]`.
3. The 8-byte prefix check passes; `parsePriceFeedUpdates` returns price data for feed `B`.
4. The consumer's `_echoCallback` receives price data for feed `B` instead of feed `A`.

The consumer contract has no way to detect the substitution — it only receives the `priceFeeds` array. Any financial logic (liquidations, settlements, options pricing) that depends on the correct feed identity is silently corrupted. The invariant that "the delivered price feed matches the requested price feed" is broken, directly analogous to the RuniverseLand invariant that "plotsMinted tracks the correct plotSize."

---

### Likelihood Explanation

`executeCallback` is a public, permissionless function with no access control beyond the configurable exclusivity window. After that window, **any address** can call it. The attacker needs only to find a second Pyth price feed whose first 8 bytes collide with the target feed. Pyth currently publishes hundreds of price feeds; with 8-byte (64-bit) prefixes, the birthday-paradox collision probability across ~500 feeds is non-negligible (~0.001%). More critically, the design structurally cannot enforce the invariant regardless of collision probability, because the full price ID is never stored. A malicious provider operating within the exclusivity window can exploit this trivially by choosing any feed with a matching prefix.

---

### Recommendation

Store the full 32-byte price IDs on-chain instead of only the 8-byte prefix. Replace `bytes8[] priceIdPrefixes` in the `Request` struct with `bytes32[] priceIds`, and compare the full IDs in `executeCallback`. The gas savings from truncation do not justify the loss of the feed-identity invariant.

```solidity
// In EchoState.sol — store full price IDs
bytes32[] priceIds;  // was: bytes8[] priceIdPrefixes

// In executeCallback — compare full IDs
require(priceIds[i] == req.priceIds[i], "Price ID mismatch");
``` [6](#0-5) 

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `priceIds = [0xAABBCCDDEEFF001122334455667788990011223344556677889900112233445566]`. The contract stores prefix `0xAABBCCDDEEFF0011`.

2. Attacker finds Pyth feed `B` with ID `0xAABBCCDDEEFF0011FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF` (same first 8 bytes, different remaining 24 bytes). Attacker obtains valid Wormhole-attested `updateData` for feed `B` from Hermes.

3. After the exclusivity period, attacker calls:
   ```solidity
   echo.executeCallback(
       anyProvider,
       sequenceNumber,
       updateData_for_B,
       [0xAABBCCDDEEFF0011FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF]
   );
   ```

4. The prefix check at lines 128–141 passes (`0xAABBCCDDEEFF0011 == 0xAABBCCDDEEFF0011`).

5. `parsePriceFeedUpdates` returns price data for feed `B`. The consumer's `_echoCallback` is invoked with feed `B`'s price, believing it is feed `A`'s price. [7](#0-6)

        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds

        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds

            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
