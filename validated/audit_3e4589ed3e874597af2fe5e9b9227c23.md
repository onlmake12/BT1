### Title
Silent `u128`ŌåÆ`u64` Truncation in `calculate_maximum_withdraw` Returns Incorrect DAO Withdrawal Amount - (File: `util/dao/src/lib.rs`)

---

### Summary

In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `u128` result `withdraw_counted_capacity` is narrowed to `u64` via a silent `as u64` cast. If the value exceeds `u64::MAX`, the function silently returns an incorrect (too-small) withdrawal capacity instead of propagating an overflow error. Every other analogous `u128`ŌåÆ`u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, making this a structurally inconsistent and unsafe cast ŌĆö the direct CKB analog of the `wfrac` silent-incorrect-result class.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the maximum amount a DAO depositor may withdraw:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ŌåÉ silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits when `withdraw_counted_capacity > u64::MAX`, producing a wrong (too-small) result with no error. This is structurally identical to the `wfrac` bug: just as `wfrac` silently returned `_INT256_MIN` unchanged when negating it (due to two's complement limits), `calculate_maximum_withdraw` silently returns a truncated value when the `u128` product overflows `u64`.

The three other `u128`ŌåÆ`u64` conversions in the same file all use checked conversion:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

Only line 156 uses the silent `as u64` cast, making this a clear inconsistency and an unguarded edge case.

The `calculate_maximum_withdraw` function is called from:
1. `transaction_maximum_withdraw` ŌåÆ `withdrawed_interests` ŌåÆ `dao_field_with_current_epoch` (DAO field consensus computation)
2. `transaction_maximum_withdraw` ŌåÆ `transaction_fee` (fee estimation)
3. Directly via the RPC `calculate_dao_maximum_withdraw` [3](#0-2) 

The `dao_field_with_current_epoch` result is verified by `DaoHeaderVerifier`, which rejects any block whose computed DAO field does not match the header:

```rust
if dao != self.header.dao() {
    return Err((BlockErrorKind::InvalidDAO).into());
}
``` [4](#0-3) 

---

### Impact Explanation

If `withdraw_counted_capacity` overflows `u64`, the function silently returns a wrong (too-small) withdrawal capacity. This propagates into:

- **DAO field `s` (secondary issuance accumulator)**: `withdrawed_interests` feeds into `current_s = parent_s.safe_add(nervosdao_issuance).and_then(|s| s.safe_sub(withdrawed_interests))`. A truncated `withdrawed_interests` makes `current_s` too large, producing an incorrect DAO field. Since both block producers and verifiers run the same code, they agree on the wrong value ŌĆö the incorrect DAO field is accepted by consensus, permanently corrupting the `s` accumulator used for future DAO interest calculations.
- **RPC `calculate_dao_maximum_withdraw`**: Returns a wrong (too-small) value to callers, causing depositors to under-claim their entitled interest.
- **Fee estimation**: `transaction_fee` returns a wrong value. [5](#0-4) 

---

### Likelihood Explanation

The overflow condition is `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`. Since `ar` starts at `10^16` (genesis) and grows proportionally to secondary issuance divided by total capacity, and the total CKB supply is bounded at approximately `3.36 ├Ś 10^18` shannons, the overflow requires `ar` to grow to roughly `5.36 ├Ś 10^16` ŌĆö a ~5.36├Ś increase from genesis. Given the secondary issuance halving schedule, this would take centuries on mainnet under normal conditions. Likelihood is therefore very low in the near term, but the code defect is real, the fix is trivial, and the inconsistency with the three surrounding checked conversions is a clear oversight.

---

### Recommendation

Replace the silent cast on line 156 with the same checked conversion pattern used everywhere else in the function:

```rust
let withdraw_capacity = Capacity::shannons(
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
).safe_add(occupied_capacity)?;
``` [6](#0-5)
