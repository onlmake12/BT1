### Title
Missing Fraction Validation in `EpochNumberWithFraction` Deserialization Allows Node Panic via Malformed Block/Transaction - (File: `util/types/src/conversion/blockchain.rs`)

### Summary

The `Unpack` deserialization impl for `EpochNumberWithFraction` uses `from_full_value_unchecked`, bypassing all fraction validity checks. A non-zero packed value with a zero `length` field will cause `to_rational()` to call `RationalU256::new(index, 0)`, which unconditionally panics with `"denominator == 0"`. This is the direct CKB analog of the Bancor `FractionLibrary` / `fromFraction112` missing-validation class.

---

### Finding Description

`EpochNumberWithFraction` is a packed 64-bit fraction `index / length` used in block headers (epoch field) and transaction `since` fields. Two construction paths exist:

- `from_full_value(v)` — safe, calls `.normalize()` which rewrites zero-length to `(number, 0, 1)`.
- `from_full_value_unchecked(v)` — no check, stores the raw bits.

The `Unpack` trait implementation — the **only** deserialization path for all wire-format and database-stored `EpochNumberWithFraction` values — uses the unchecked variant: [1](#0-0) 

`to_rational()` explicitly documents and implements a panic when the stored value is non-zero but `length == 0`: [2](#0-1) 

`RationalU256::new` panics unconditionally on a zero denominator: [3](#0-2) 

The `Since::extract_metric()` path also constructs an unvalidated `EpochNumberWithFraction` directly from attacker-controlled transaction input bits: [4](#0-3) 

The `SinceVerifier` does call `is_well_formed_increment()` before `to_rational()`, and `normalize()` is applied, so the `since` path is currently guarded. However, the `Unpack` deserialization path for block headers has no such guard at the deserialization layer — validation is deferred entirely to the downstream `EpochVerifier`: [5](#0-4) 

Any code path that calls `to_rational()` on a deserialized `EpochNumberWithFraction` **before** the `EpochVerifier` runs will panic. The `MaturityVerifier` calls `to_rational()` on `info.block_epoch` (from stored transaction info) and on `self.epoch` (from the current block header): [6](#0-5) 

The `EpochExt::set_primary_reward` also divides by `self.length` without a zero-check: [7](#0-6) 

---

### Impact Explanation

If a malicious peer relays a block whose header encodes a non-zero epoch with `length == 0` (e.g., packed value `0x0000_0000_0000_0001` = number=1, index=0, length=0), and any processing step calls `to_rational()` on that deserialized value before the `EpochVerifier` gate executes, the node panics and crashes. Even where the `EpochVerifier` gate is present, the lack of validation at the deserialization boundary means any future code path or ordering change silently introduces a reachable panic. The `EpochExt::set_primary_reward` division by `self.length` is an additional unguarded arithmetic path.

---

### Likelihood Explanation

Any unprivileged P2P peer can relay a crafted block header with a malformed epoch field. The attacker-controlled entry point is the standard block relay / sync protocol. The packed `u64` epoch field is fully attacker-controlled. The only barrier is the ordering of the `EpochVerifier` check relative to `to_rational()` calls in the processing pipeline — a fragile, implicit ordering dependency rather than a structural invariant enforced at construction time.

---

### Recommendation

Replace `from_full_value_unchecked` with `from_full_value` (which calls `.normalize()`) in the `Unpack` impl: [8](#0-7) 

Alternatively, add an explicit `is_well_formed()` guard inside `to_rational()` itself, returning an error or a safe default instead of panicking, consistent with the `is_genesis` special-case already present. The `EpochExt::set_primary_reward` division should also guard against `self.length == 0`.

---

### Proof of Concept

1. Craft a block header with epoch packed value `0x0000_0000_0000_0001` (number=1, index=0, length=0; full_value ≠ 0).
2. Relay it to a CKB node via the P2P block relay protocol.
3. The node deserializes the epoch via `Unpack` → `from_full_value_unchecked(1)` → `EpochNumberWithFraction(1)` with `length() == 0`.
4. Any call to `.to_rational()` on this value before the `EpochVerifier` gate executes reaches `RationalU256::new(0.into(), 0.into())` → `panic!("denominator == 0")` → node crash.
