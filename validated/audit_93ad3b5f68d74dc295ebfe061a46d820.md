The bug is real and the execution path is concrete. Here is the analysis:

---

### Title
`terminated_vms` Entry Consumed by First Waiter, Causing Spurious `WAIT_FAILURE` for All Subsequent Waiters on the Same Target — (`script/src/scheduler.rs`)

---

### Summary

In `Scheduler::process_message_box`, when a `Message::Wait` is processed and the target VM is found in `terminated_vms`, the entry is unconditionally removed via `terminated_vms.retain(|id, _| id != &args.target_id)` at line 575. Because the scheduler is single-threaded and processes one VM per iteration, a second VM that later issues `ckb_wait` on the same already-terminated target finds neither `terminated_vms` nor `states` containing the target, and receives `WAIT_FAILURE` (code 5) instead of `SUCCESS` (code 0).

---

### Finding Description

The scheduler maintains two relevant data structures:

- `states: BTreeMap<VmId, VmState>` — tracks live VMs
- `terminated_vms: BTreeMap<VmId, i8>` — tracks VMs that have exited but whose exit code has not yet been consumed

When a non-root VM terminates (`iterate_process_results`, lines 362–406):

1. `terminated_vms.insert(vm_id_to_run, code)` is called. [1](#0-0) 
2. Any VMs **already** in `VmState::Wait { target_vm_id == vm_id_to_run }` are resolved immediately (written exit code, marked `Runnable`). [2](#0-1) 
3. The target is removed from `states`. [3](#0-2) 

When a waiter issues `ckb_wait` **after** the target has already terminated, `process_message_box` handles `Message::Wait` as follows:

- If target is in `terminated_vms`: write exit code, mark waiter `Runnable`, then **remove the entry** with `terminated_vms.retain(|id, _| id != &args.target_id)`. [4](#0-3) 
- If target is **not** in `terminated_vms` and **not** in `states`: return `WAIT_FAILURE`. [5](#0-4) 

The `retain` call at line 575 is the root cause. It destroys the only record that the target ever existed, making it invisible to any subsequent waiter.

---

### Impact Explanation

**Concrete execution trace** (all single-threaded, deterministic):

1. Root VM spawns `target`, `waiter_A`, `waiter_B`.
2. `target` runs and exits with code 0 → `terminated_vms = {target: 0}`, target removed from `states`.
3. `waiter_A` runs, calls `ckb_wait(target, addr_A)` → yields, `Message::Wait(waiter_A, target)` queued.
4. `process_message_box`: target found in `terminated_vms` → SUCCESS written to `waiter_A`, **`terminated_vms` entry for `target` removed**.
5. `waiter_B` runs, calls `ckb_wait(target, addr_B)` → yields, `Message::Wait(waiter_B, target)` queued.
6. `process_message_box`: target **not** in `terminated_vms`, **not** in `states` → `WAIT_FAILURE` written to `waiter_B`. [5](#0-4) 

`waiter_B` receives return code 5 (`WAIT_FAILURE`) for a target that legitimately terminated with code 0. Any script logic that branches on the `ckb_wait` return value — including authorization gates in lock scripts — will execute the wrong branch for `waiter_B`.

---

### Likelihood Explanation

This is reachable by any unprivileged script author. CKB-VM script execution is a standard production path (transaction validation). No special privileges, keys, or majority hashpower are required. The trigger condition — two VMs waiting on the same child where the child terminates before either waiter issues its syscall — is a natural fan-out pattern and requires no adversarial timing; the single-threaded scheduler makes the ordering fully deterministic and reproducible.

---

### Recommendation

Remove the `terminated_vms.retain` call from `process_message_box`. The entry in `terminated_vms` should persist until it is certain no further waiters can reference the target. Since the target is already removed from `states` at termination time, the `!self.states.contains_key(&args.target_id)` guard at line 578 already correctly distinguishes "never existed / invalid ID" from "terminated and consumed." The entry should only be removed once all legitimate waiters have been resolved, or simply never removed (relying on the fact that `states` no longer contains the target as the authoritative "VM is gone" signal).

---

### Proof of Concept

```rust
// Pseudocode for a CKB script triggering the bug:
// Root VM:
let target = ckb_spawn(target_program);   // spawns target
let wa     = ckb_spawn(waiter_a_program); // spawns waiter_A
let wb     = ckb_spawn(waiter_b_program); // spawns waiter_B
// target_program: exits immediately with code 0
// waiter_a_program: calls ckb_wait(target) — receives SUCCESS (correct)
// waiter_b_program: calls ckb_wait(target) — receives WAIT_FAILURE (BUG)
//   because terminated_vms entry was removed after waiter_A consumed it
assert!(wait_result_b == SUCCESS); // FAILS — receives 5 (WAIT_FAILURE)
```

The invariant that "every VM that successfully terminated must be waitable by any VM that holds its ID" is violated for the second and all subsequent waiters. [6](#0-5)
