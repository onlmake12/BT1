### Title
Deviation Check Incorrectly Skips Zero-Price Feeds, Marking Valid Updates as Invalid — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

In `Scheduler._validateShouldUpdatePrices`, the `updateOnDeviation` branch silently skips any feed whose `previousPrice` or `currentPrice` is zero to avoid division by zero. When **all** feeds in a subscription are skipped this way, the function falls through to `revert UpdateConditionsNotMet()`, permanently blocking a valid update even though the price has moved significantly (e.g., from 0 → X, or X → 0).

### Finding Description

`_validateShouldUpdatePrices` evaluates whether a keeper-submitted update satisfies the subscription's trigger criteria:

```solidity
// If updateOnDeviation is enabled, check if any price has deviated enough
if (params.updateCriteria.updateOnDeviation) {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        ...
        // Skip if either price is zero to avoid division by zero
        if (previousPrice == 0 || currentPrice == 0) {
            continue;          // ← silently skips the feed
        }
        ...
        if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
            return updateTimestamp;
        }
    }
}

revert SchedulerErrors.UpdateConditionsNotMet();   // ← reached when ALL feeds are skipped
``` [1](#0-0) 

When every feed in the batch has `previousPrice == 0` or `currentPrice == 0`, the inner loop completes without ever returning, and the function unconditionally reverts. The check is applied to the **full set** of feeds when it should only gate feeds that have a meaningful non-zero denominator; feeds transitioning to or from zero represent the largest possible deviation (infinite or 100%) and should always trigger an update.

The analogous flaw in the reference report (M-08) was `doItemsIntersect(sell.nfts, buy.nfts)` being evaluated on the full NFT sets instead of only on `constructedNfts`, causing valid orders to be rejected. Here, the deviation check is evaluated on the full feed set instead of only on feeds with non-zero denominators, causing valid updates to be rejected.

### Impact Explanation

A subscription configured with `updateOnDeviation` only (no heartbeat) whose feeds carry a zero `previousPrice` or `currentPrice` will **never** be updated by any keeper, regardless of how large the actual price movement is. The subscription's stored prices become permanently stale. Downstream consumers reading via `getPricesNoOlderThan` will receive `StalePrice` reverts; consumers using `getPricesUnsafe` will read outdated data. The subscription's balance continues to be locked with no path to recovery short of the manager deactivating and recreating the subscription.

Even for subscriptions that also have `updateOnHeartbeat` enabled, the deviation-triggered fast-path is silently disabled for all zero-price feeds, defeating the purpose of the deviation criterion during market disruptions (exactly the scenario where zero prices are most likely).

### Likelihood Explanation

`int64` prices in Pyth can legitimately be zero for:
- Newly listed assets before their first non-zero aggregation
- Assets whose price rounds to zero at the stored exponent
- Market-disruption or circuit-breaker events

Any keeper calling `updatePriceFeeds` on such a subscription will trigger the revert. No special privilege is required; the entry path is the public `updatePriceFeeds` function callable by any address. [2](#0-1) 

### Recommendation

Instead of `continue`-ing when a price is zero, treat the transition as an unconditional trigger (infinite or 100% deviation always exceeds any finite threshold):

```solidity
// Skip if either price is zero to avoid division by zero
// A transition to/from zero is always a maximum-deviation event; trigger immediately.
if (previousPrice == 0 || currentPrice == 0) {
    return updateTimestamp;   // treat as infinite deviation
}
```

Alternatively, only skip the feed if **both** prices are zero (no change), and trigger otherwise.

### Proof of Concept

1. Create a subscription with `updateOnDeviation = true`, `updateOnHeartbeat = false`, `deviationThresholdBps = 100` (1%).
2. Submit a first update where all feeds have `price = 0`. This succeeds because `previousFeed.id == bytes32(0)` (no prior stored feed) causes an early return.
3. Submit a second update where all feeds have `price = 1_000_000` (a massive move from 0).
4. In `_validateShouldUpdatePrices`, for each feed: `previousPrice = 0` → `continue`. The loop ends without returning.
5. The function reverts with `UpdateConditionsNotMet`, even though the price moved from 0 to 1,000,000 — a deviation that should unambiguously trigger an update. [3](#0-2)
