### Title
Copy-Paste Bug in `script_exists_in_output` Causes Incorrect Type-Script Deletion on PostgreSQL During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

A copy-paste error at line 252 of `script_exists_in_output` causes the function to evaluate the **lock-script** existence result a second time instead of the **type-script** existence result. On PostgreSQL this silently returns the wrong boolean, causing type-scripts that are only referenced as `type_script_id` (not `lock_script_id`) in surviving outputs to be incorrectly deleted from the `script` table during every `rollback_block` call. SQLite is unaffected because its type mismatch forces the correct fallback path.

---

### Finding Description

In `script_exists_in_output`: [1](#0-0) 

`row_lock` is populated by querying `WHERE lock_script_id = $1`. [2](#0-1) 

`row_type` is populated by querying `WHERE type_script_id = $1`.

Then the final return: [3](#0-2) 

Line 252 reads `row_lock.try_get::<bool, _>(0)` — it should read `row_type.try_get::<bool, _>(0)`. On **PostgreSQL**, `EXISTS(...)` returns a native `BOOLEAN`, so `try_get::<bool, _>(0)` succeeds on `row_lock` and the `Ok(r)` arm is taken, returning the lock-script check result instead of the type-script check result. On **SQLite**, `EXISTS(...)` returns `BIGINT`, so `try_get::<bool, _>` fails, the `Err(_)` arm is taken, and `row_type.get::<i64, _>(0)` is correctly used.

The consequence inside `rollback_block`: [4](#0-3) 

For every output in the rolled-back block whose `type_script_id` is **not** also used as a `lock_script_id` in any surviving output, `script_exists_in_output` returns `false` on PostgreSQL (because it re-reads `row_lock`, which is `false`). That script ID is pushed into `script_id_list_to_remove` and deleted from the `script` table, even though surviving outputs still reference it via `type_script_id`.

---

### Impact Explanation

After the incorrect deletion, `get_cells` and `get_cells_capacity` perform JOINs against the `script` table: [5](#0-4) 

Because the script row no longer exists, the JOIN produces no match and those cells are silently omitted from results. The indexer returns fewer cells and lower capacity than the canonical chain state, and diverges from a parallel SQLite-backed node processing the same reorg sequence. Each additional reorg compounds the divergence.

---

### Likelihood Explanation

- Reorgs are a normal part of CKB network operation; no attacker action is required to trigger them.
- The bug fires on every rollback where any output has a `type_script_id` that is not simultaneously used as a `lock_script_id` in a surviving output — a common real-world condition.
- The bug is silently backend-specific: PostgreSQL operators will observe divergence from SQLite operators with no error logged.

---

### Recommendation

Fix line 252 to use `row_type` instead of `row_lock`:

```rust
// was: match row_lock.try_get::<bool, _>(0) {
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [3](#0-2) 

---

### Proof of Concept

1. Start a PostgreSQL-backed rich-indexer node.
2. Append a block containing one output with a `type_script` that is not used as a `lock_script` anywhere.
3. Call `rollback_block` once.
4. Query `SELECT * FROM script WHERE id = <type_script_id>` — the row is gone despite no surviving output referencing it being removed.
5. Call `get_cells` filtering by that type script — returns empty.
6. Repeat with a SQLite-backed node on the same sequence — `get_cells` returns the correct cell.
