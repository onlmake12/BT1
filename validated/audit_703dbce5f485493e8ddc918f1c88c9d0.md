### Title
Entropy Provider Can Front-Run `requestWithCallback`/`requestV2` to DoS Users via Instant Fee Increase — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any registered Entropy provider (or their designated fee manager) can call `setProviderFee` to instantly raise their fee with no timelock. A user who queries `getFee(provider)` off-chain and then submits a `requestWithCallback` / `requestV2` transaction can have that transaction reverted by the provider front-running with a fee increase. No `maxFee` guard parameter exists anywhere in the request path, leaving users with no on-chain protection.

---

### Finding Description

**Step 1 — Instant, unconstrained fee change.**

`setProviderFee` writes the new fee directly to storage with no delay, cap, or timelock: [1](#0-0) 

`setProviderFeeAsFeeManager` gives the same power to a delegated fee manager: [2](#0-1) 

**Step 2 — Fee is re-read at execution time, not at submission time.**

Inside `requestHelper`, the required fee is computed from the *current* on-chain state at the moment the transaction executes: [3](#0-2) 

If `msg.value` (calculated from the stale `getFee` result) is now less than `requiredFee`, the call reverts with `InsufficientFee`.

**Step 3 — No `maxFee` parameter exists.**

Neither `requestWithCallback` nor `requestV2` accepts a caller-specified maximum fee. The IEntropy interface comment explicitly warns: *"Note that excess value is not refunded to the caller."* Users therefore cannot express consent to a specific fee level, and cannot protect themselves from a race-condition fee increase. [4](#0-3) 

The same structural gap exists in the Echo (`Pulse`) contract's `requestPriceUpdatesWithCallback` and its `setProviderFee`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

A malicious or compromised Entropy provider can repeatedly front-run every pending `requestWithCallback` / `requestV2` transaction in the mempool by raising their fee by 1 wei before each one lands. Every user request targeting that provider reverts with `InsufficientFee`. The provider can then lower the fee again, making the service appear available off-chain while blocking all on-chain requests. This is a targeted, sustained DoS against any user or protocol that relies on a specific provider for randomness or price-update callbacks (Echo). Funds are not directly stolen, but the service is rendered completely unavailable to affected users.

---

### Likelihood Explanation

The attacker role is an **Entropy provider**, explicitly listed as a valid attacker in the audit scope. Any address that has called `register` qualifies. Monitoring the public mempool for `requestWithCallback` transactions and front-running them with a `setProviderFee` call is straightforward on EVM chains with a public mempool. The attack requires no special privilege beyond being a registered provider, and can be automated. The default provider (set by admin) is the most impactful target because most users route through it.

---

### Recommendation

1. **Add a `maxFee` parameter** to `requestWithCallback`, `requestV2`, and Echo's `requestPriceUpdatesWithCallback`. Revert if the computed fee exceeds the caller's stated maximum, mirroring the fix applied to the analogous Teller vulnerability.
2. **Introduce a timelock or notice period** for `setProviderFee` / `setProviderFeeAsFeeManager` (e.g., a two-step commit-reveal with a minimum delay), so users can observe a pending fee change and decide whether to proceed.

---

### Proof of Concept

```
1. Provider registers with feeInWei = F.
2. User calls getFee(provider) off-chain → receives F + pythFeeInWei = T.
3. User broadcasts requestWithCallback{value: T}(provider, userContribution).
4. Provider (or fee manager) sees the pending tx in the mempool and broadcasts
   setProviderFee(F + 1) with a higher gas price (front-run).
5. setProviderFee executes first; provider.feeInWei = F + 1.
6. User's requestWithCallback executes:
     requiredFee = getFeeV2(provider, 0) = (F+1) + pythFeeInWei = T + 1
     msg.value = T < T + 1  →  revert InsufficientFee
7. User's ETH is returned, but the request fails.
8. Provider resets fee to F. Repeat indefinitely.
```

The same sequence applies to `requestV2` (with a `gasLimit` argument) and to Echo's `requestPriceUpdatesWithCallback`, where `getFee` is also read at execution time from mutable provider state.

    // Set provider fee. It will revert if provider is not registered.
    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
