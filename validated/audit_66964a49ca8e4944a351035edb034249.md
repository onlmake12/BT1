### Title
Decompressed frame size not validated against per-protocol `max_frame_length`, allowing any peer to deliver up to 8 MB payloads to handlers with sub-MB limits — (`network/src/compress.rs`)

### Summary
`LengthDelimitedCodecWithCompress::decode` enforces `max_frame_length` only on the compressed wire bytes. After decompression, the resulting buffer is bounded only by the global `MAX_UNCOMPRESSED_LEN = 8 MB`. An unprivileged remote peer can send a compressed frame just below a protocol's `max_frame_length` that decompresses to nearly 8 MB, bypassing the per-protocol size limit entirely and delivering an oversized payload to the upper-layer handler.

### Impact Explanation
- **Memory amplification**: A single peer can force a `BytesMut::zeroed(~8 MB)` allocation per message on any protocol. With N peers each sending at their protocol's compressed frame rate, heap usage grows as `N × 8 MB × message_rate`.
- **Protocols with tiny limits are most affected**: Ping (1 KB limit → up to 8000× amplification), Identify (2 KB → 4000×), Discovery (512 KB → 16×). These handlers are not designed to process multi-MB payloads.
- **CPU exhaustion**: Molecule deserialization of 8 MB on every Sync/Relay message is significantly more expensive than the intended 2 MB / 4 MB maximum.
- **No panic required for impact**: sustained OOM pressure or CPU saturation across all connected peers is sufficient for a denial-of-service.

### Likelihood Explanation
Any unauthenticated peer on the P2P network can open the Sync, Relay, Ping, or any other protocol and send snappy-compressed frames. Snappy compression ratios of 4–8× on structured/repetitive data (e.g., zero-padded molecule buffers) are easily achievable. No special privileges, keys, or hashpower are required.

### Recommendation
In `LengthDelimitedCodecWithCompress::decode`, after obtaining `decompressed_bytes_len`, add a check against `self.length_delimited.max_frame_length()`:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN
    || decompressed_bytes_len > self.length_delimited.max_frame_length()
{
    return Err(io::ErrorKind::InvalidData.into());
}
```

This mirrors the existing guard in `process` (the encoder path) and ensures the decompressed payload respects the same per-protocol size limit as uncompressed messages.

### Proof of Concept
1. Connect to a CKB node as a peer on the Sync protocol (`max_frame_length` = 2 MB).
2. Construct a snappy-compressed payload: take ~8 MB of repetitive bytes (e.g., `vec![0u8; 8_000_000]`), compress with `snap::raw::Encoder`. The compressed output will be well under 2 MB.
3. Prepend `COMPRESS_FLAG` (0x80) and a 4-byte length prefix, send over the wire.
4. The `length_delimited` codec accepts it (compressed size < 2 MB). `decompress_len` returns ~8 MB < `MAX_UNCOMPRESSED_LEN`. `BytesMut::zeroed(8_000_000)` is allocated. The Sync handler's `received` is called with an 8 MB slice.
5. Repeat from all available peer slots. Each message forces an 8 MB allocation and a full molecule parse attempt, exhausting heap and CPU.
