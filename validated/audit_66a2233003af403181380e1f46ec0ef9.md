### Title
Uncharged Linear `header_deps` Scan in `LoadHeader` / `LoadBlockExtension` Syscalls Violates Cycle-Cost Invariant — (`script/src/syscalls/load_header.rs`, `script/src/syscalls/load_block_extension.rs`)

### Summary

Both `LoadHeader::load_header` and `LoadBlockExtension::load_block_extension` perform an O(N) linear scan over all `header_deps` in Rust host code before charging any cycles. The cycle charge covers only the bytes transferred to VM memory, not the scan itself. This violates the CKB invariant that cycle cost must cover all node computation, enabling a bounded but real CPU-amplification attack.

### Finding Description

In `load_header.rs` lines 56–70:

```rust
fn load_header(&self, cell_meta: &CellMeta) -> Option<HeaderView> {
    let block_hash = &cell_meta.transaction_info.as_ref()?.block_hash;
    if self
        .header_deps()          // returns Byte32Vec over raw tx bytes
        .into_iter()
        .any(|hash| &hash == block_hash)   // O(N) scan — uncharged
    {
        self.sg_data.tx_info.data_loader.get_header(block_hash)
    } else {
        None
    }
}
``` [1](#0-0) 

The identical pattern appears in `load_block_extension.rs` lines 46–60: [2](#0-1) 

Cycles are charged only after the scan completes, for the bytes written to VM memory:

```rust
machine.add_cycles_no_checking(transferred_byte_cycles(len))?;
``` [3](#0-2) 

The cost model charges 1 cycle per 4 bytes transferred: [4](#0-3) 

The scan itself — iterating over N × 32-byte hashes in Rust — is entirely outside the VM cycle accounting.

### Impact Explanation

**Amplification ratio per syscall call:**
- Rust-side work: N × 32 bytes of hash comparison (e.g., N=16,000 → 512,000 bytes)
- Cycles charged: `header_size / 4` ≈ 52 cycles (for a ~208-byte header)
- Ratio: ~9,846× in bytes processed vs cycles charged

**Practical worst-case bounds:**

The `TRANSACTION_SIZE_LIMIT` is 512 KB: [5](#0-4) 

Each `header_dep` is 32 bytes, giving ≈ 16,000 header_deps maximum per transaction. The default `max_tx_verify_cycles` is 70,000,000: [6](#0-5) 

With ~1,000 VM cycles overhead per syscall call, the script can make ≈ 70,000 syscall calls. Total uncharged Rust-side work: 70,000 × 16,000 = **1.12 billion hash comparisons** (~1–2 seconds of CPU per transaction per node), while cycles charged remain proportional only to the header size.

The scan also fires even when the result is `None` (cell's block hash not in header_deps), so the attacker does not need the cell to actually match any header dep.

### Likelihood Explanation

- **Attacker entry point**: Unprivileged RPC `send_transaction` — no privileged access required.
- **Constraint**: Each `header_dep` must be a valid main-chain block hash (enforced during resolution). On mainnet with millions of blocks, collecting 16,000 valid hashes is trivially feasible.
- **Economic cost**: Fee rate is based on tx serialized size. A 512 KB transaction costs ~512,000 shannons (~0.005 CKB at current rates) — negligible relative to the CPU work imposed on every node that verifies it.
- **Duplicate header_deps** are rejected by `DuplicateDepsVerifier`, so all 16,000 must be distinct — still easily satisfied on mainnet. [7](#0-6) 

An attacker can sustain this by continuously submitting such transactions, causing persistent amplified CPU load on all verifying nodes.

### Recommendation

Charge cycles proportional to the number of `header_deps` scanned before returning from the syscall. One approach: add a fixed per-element scan cost (e.g., 1 cycle per `header_dep` entry examined) inside `load_header` and `load_block_extension` before the `.any()` call, debited via `machine.add_cycles_no_checking(header_deps_scanned_count)`. Alternatively, pre-build a `HashSet<Byte32>` of `header_deps` at script group initialization time (charged once) and use O(1) lookup per syscall call.

### Proof of Concept

1. Collect 16,000 distinct valid main-chain block hashes.
2. Craft a transaction with those 16,000 `header_deps`, one input cell, and a lock script that loops calling `LOAD_HEADER` (syscall 2072) for `Source::Input[0]` until cycles are exhausted.
3. Submit via `send_transaction` RPC.
4. Measure wall-clock time for script verification vs cycles charged.
5. Assert: wall-clock time scales with `header_deps` count while cycles charged remain constant (only `transferred_byte_cycles(header_size)` per call).

    pub fn verify(&self) -> Result<(), Error> {
        let transaction = self.transaction;
        let mut seen_cells = HashSet::with_capacity(self.transaction.cell_deps().len());
        let mut seen_headers = HashSet::with_capacity(self.transaction.header_deps().len());

        if let Some(dep) = transaction
            .cell_deps_iter()
            .find_map(|dep| seen_cells.replace(dep))
        {
            return Err(TransactionError::DuplicateCellDeps {
                out_point: dep.out_point(),
            }
            .into());
        }
        if let Some(hash) = transaction
            .header_deps_iter()
            .find_map(|hash| seen_headers.replace(hash))
        {
            return Err(TransactionError::DuplicateHeaderDeps { hash }.into());
        }
        Ok(())
    }
