Audit Report

## Title
`HeaderAcceptor::accept()` Skips `BLOCK_INVALID` Guard, Allowing Header Map Pollution via `SendHeaders` - (File: `sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::accept()` contains an acknowledged `FIXME` where a header whose block is already marked `BLOCK_INVALID` bypasses the early-return guard because `BLOCK_INVALID` (`1 << 12 = 4096`) does not contain the `HEADER_VALID` (`1`) bit. The function then re-runs only lightweight non-contextual checks, which pass for blocks invalidated for contextual reasons, and calls `insert_valid_header`, inserting the invalid block's header into `header_map` and updating the peer's `best_known_header` to a known-invalid block.

## Finding Description
In `sync/src/synchronizer/headers_process.rs` at line 301ŌĆō322, `accept()` checks `status.contains(BlockStatus::HEADER_VALID)` as its only early-return guard. Since `BLOCK_INVALID = 1 << 12` shares no bits with `HEADER_VALID = 1`, the check evaluates to `false` for `BLOCK_INVALID` blocks and the guard is skipped. The three subsequent checks ŌĆö `prev_block_check` (checks only the *parent's* status), `non_contextual_check` (PoW, timestamp, epoch, via `HeaderVerifier`), and `version_check` (version == 0) ŌĆö all pass for a block that was invalidated for contextual body reasons (invalid cellbase reward, invalid transactions, invalid DAO header, etc.). Execution reaches line 356: `sync_shared.insert_valid_header(self.peer, self.header)`. This function (lines 1094ŌĆō1141 of `sync/src/types/mod.rs`) inserts the `HeaderIndexView` into `header_map`, calls `may_set_best_known_header` to update the peer's chain tip, and calls `may_set_shared_best_header` to potentially update the global best header ŌĆö all for a block the node already knows is invalid. By contrast, `CompactBlockProcess` at line 259 of `sync/src/relayer/compact_block_process.rs` correctly guards with `status.contains(BlockStatus::BLOCK_INVALID)` and returns an error.

## Impact Explanation
The concrete impact is sync disruption and bandwidth waste for affected nodes. `BlockFetcher` (lines 159ŌĆō169 of `sync/src/synchronizer/block_fetcher.rs`) uses `best_known_header` to determine which blocks to request; with it pointing at an invalid block, the node issues `GetBlocks` for blocks on a chain it already knows is invalid. If `shared_best_header` is also updated, the node's IBD state machine may stall legitimate sync or loop in futile download attempts. `header_map` is polluted with entries for invalid blocks, affecting ancestor-traversal and skip-pointer logic. This maps to **Low (501ŌĆō2000 points): "Any other important performance improvements for CKB"** ŌĆö the node does not crash, no consensus deviation occurs, and no economic damage results; the impact is wasted bandwidth, CPU, and degraded sync scheduling.

## Likelihood Explanation
The entry path is the standard `SendHeaders` P2P message, reachable by any connected peer without privilege. However, the precondition requires the attacker to have previously caused a block to be marked `BLOCK_INVALID`, which in the most realistic scenario means mining a block with a valid PoW header but an invalid body (e.g., wrong cellbase reward). Mining a valid PoW block is a significant one-time cost, limiting this to well-resourced attackers. Once such a block exists, the attacker can repeatedly send its header to any number of nodes at negligible marginal cost. The FIXME comment in the source confirms developer awareness of the gap.

## Recommendation
Add an explicit `BLOCK_INVALID` guard at the top of `HeaderAcceptor::accept()`, before the `HEADER_VALID` check, mirroring what `CompactBlockProcess` already does:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent));
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing logic
}
```

## Proof of Concept
1. Connect a malicious peer to a CKB node.
2. Construct and relay a block whose header is valid (correct PoW, timestamp, epoch, version) but whose body is contextually invalid (e.g., cellbase output exceeds the allowed reward). The node processes it, fails contextual verification, and records `block_status_map[block_hash] = BLOCK_INVALID`.
3. Send a `SendHeaders` P2P message containing only that block's header.
4. `HeadersProcess::execute()` calls `HeaderAcceptor::accept()`.
5. `get_block_status` returns `BLOCK_INVALID` (4096). `status.contains(HEADER_VALID)` = `(4096 & 1) != 0` = `false`. The early-return is skipped.
6. `prev_block_check` passes (parent is valid). `non_contextual_check` passes (header PoW/timestamp/epoch are valid). `version_check` passes (version == 0).
7. `insert_valid_header` is called: the invalid block's header is inserted into `header_map`, and `best_known_header` for the peer is updated to the invalid block.
8. Confirm via the `get_peers` RPC that `best_known_header_hash` now points to the known-invalid block, and observe `GetBlocks` requests being issued for blocks on the invalid chain.
