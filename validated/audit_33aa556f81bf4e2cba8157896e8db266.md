### Title
Provider Fees Permanently Locked Due to Uninitialized `feeManager` Defaulting to `address(0)` — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the only mechanism for a provider to withdraw accrued fees is `withdrawAsFeeManager`, which requires `msg.sender == feeManager`. Because `feeManager` is never set during `registerProvider` and defaults to `address(0)`, any provider that does not explicitly call `setFeeManager` after registration can never withdraw their fees. There is no fallback `withdraw` function for providers in Echo (unlike Entropy). All accrued provider fees are permanently locked in the contract.

---

### Finding Description

When a provider registers via `registerProvider`, the `ProviderInfo` struct is partially initialized — `feeManager` is left at its zero-value default (`address(0)`):

```solidity
// Echo.sol lines 381-393
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    provider.feePerFeedInWei = feePerFeedInWei;
    provider.feePerGasInWei = feePerGasInWei;
    provider.isRegistered = true;
    emit ProviderRegistered(msg.sender, feePerGasInWei);
}
```

`feeManager` is never assigned here. [1](#0-0) 

The `ProviderInfo` struct confirms `feeManager` is a plain `address` field with no default:

```solidity
struct ProviderInfo {
    ...
    address feeManager;
    bool isRegistered;
    ...
}
``` [2](#0-1) 

The only provider fee withdrawal path in Echo is `withdrawAsFeeManager`:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    ...
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [3](#0-2) 

Since `feeManager == address(0)` by default, the check `msg.sender == address(0)` can never be satisfied (EVM guarantees `msg.sender != address(0)`). There is **no** plain `withdraw(amount)` function for providers in Echo — unlike `Entropy.sol` which provides both `withdraw` (provider-direct) and `withdrawAsFeeManager`. [4](#0-3) 

The `IEcho` interface exposes only `withdrawFees` (admin-only, for Pyth protocol fees) and `withdrawAsFeeManager` (fee-manager-only, for provider fees) — no provider-direct withdrawal exists. [5](#0-4) 

Fees accrue to the provider's balance during `executeCallback`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [6](#0-5) 

These fees are permanently locked for any provider that does not call `setFeeManager` with a non-zero address after registration.

---

### Impact Explanation

**High.** All provider fees accrued in `Echo.sol` are permanently locked in the contract for any provider that registers without subsequently calling `setFeeManager`. There is no recovery path — no admin override, no governance mechanism, and no direct `withdraw` for providers. The ETH is irrecoverably stuck.

---

### Likelihood Explanation

**High.** The `registerProvider` function gives no indication that a separate `setFeeManager` call is required before fees can be withdrawn. A provider following the natural registration flow will accrue fees they can never retrieve. The `setFeeManager` function is not documented as a prerequisite in `registerProvider` or in the `IEcho` interface. Any provider operating without a fee manager — which is the default state — is affected from the first fulfilled request onward.

---

### Recommendation

Add a direct `withdraw` function for providers in `Echo.sol`, mirroring `Entropy.sol`:

```solidity
function withdraw(uint128 amount) external {
    ProviderInfo storage providerInfo = _state.providers[msg.sender];
    require(providerInfo.isRegistered, "Provider not registered");
    require(providerInfo.accruedFeesInWei >= amount, "Insufficient balance");
    providerInfo.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "Withdrawal failed");
    emit FeesWithdrawn(msg.sender, amount);
}
```

Alternatively, add a zero-address guard in `setFeeManager` and require providers to set a non-zero fee manager at registration time, but the direct `withdraw` approach is simpler and eliminates the dependency entirely.

---

### Proof of Concept

1. Provider calls `registerProvider(baseFee, feedFee, gasFee)`. `feeManager` defaults to `address(0)`.
2. A user calls `requestPriceUpdatesWithCallback{value: fee}(provider, ...)`. Fee is split: `_state.accruedFeesInWei += pythFeeInWei` (Pyth) and `req.fee = msg.value - pythFeeInWei` (provider pending).
3. Provider calls `executeCallback(...)`. `_state.providers[provider].accruedFeesInWei += (req.fee + msg.value - pythFee)`.
4. Provider calls `withdrawAsFeeManager(provider, amount)`. Reverts with `"Only fee manager"` because `_state.providers[provider].feeManager == address(0)` and `msg.sender != address(0)`.
5. Provider has no other withdrawal function. Fees are permanently locked.
