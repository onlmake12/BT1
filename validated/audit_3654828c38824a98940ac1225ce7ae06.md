### Title
Missing Timestamp Validation in `verifyUpdate` Enables Stale Lazer Price Replay - (File: lazer/contracts/evm/src/PythLazer.sol)

### Summary
The `verifyUpdate` function in `PythLazer.sol` verifies the ECDSA signature and signer validity of a Lazer price update but performs **no validation of the payload's embedded timestamp**. Any old, validly-signed Lazer update can be replayed by an unprivileged caller at any future time, and `verifyUpdate` will succeed. Consumer contracts that rely on `verifyUpdate` as their security gate — without adding their own timestamp check — will accept arbitrarily stale prices.

### Finding Description
`PythLazer.sol` is the on-chain verification contract for Pyth Lazer price updates on EVM chains. Its `verifyUpdate` function is the sole entry point for processing incoming Lazer oracle messages. It performs three checks:

1. Fee payment
2. EVM format magic
3. ECDSA signature validity and signer expiry [1](#0-0) 

The payload returned by `verifyUpdate` contains a `uint64 timestamp` field at bytes `[4:12]` of the payload (after the `PAYLOAD_FORMAT_MAGIC`), as parsed by `parsePayloadHeader`: [2](#0-1) 

However, `verifyUpdate` never reads or validates this timestamp. There is no check that:
- `timestamp` is within an acceptable distance of `block.timestamp` (staleness check), or
- `timestamp` is strictly greater than any previously accepted update timestamp (monotonicity check).

The only time-based protection is `isValidSigner`, which checks that the signer key has not expired: [3](#0-2) 

This is a key-rotation mechanism, not a price-freshness mechanism. A signer key may remain valid for months or years, during which any historical Lazer update signed by that key can be replayed successfully.

### Impact Explanation
An attacker who obtains any old Lazer update blob (e.g., captured from the public WebSocket stream) can call `verifyUpdate` with that blob at any future time. `verifyUpdate` will return the old payload and the signer address without error. Any consumer contract that:
- calls `verifyUpdate` to authenticate the update, and
- does not independently validate the payload timestamp against `block.timestamp`

will update its on-chain price state with the stale price. This directly mirrors the reported vulnerability class: stale oracle data being accepted due to missing timestamp validation in the receiver/verification function.

Concrete consequences:
- Liquidation engines using Lazer prices could be triggered or blocked using a favorable historical price.
- Derivative protocols could be manipulated by replaying a price from a period of high volatility.
- Any protocol that treats a successful `verifyUpdate` call as proof of price freshness is vulnerable.

### Likelihood Explanation
- The attack is **permissionless**: any EOA can call `verifyUpdate`.
- Old Lazer update blobs are **publicly observable** from the Lazer WebSocket stream; an attacker can archive them.
- The signer key expiry (`expiresAt`) is set by governance and is typically far in the future, so historical updates remain replayable for extended periods.
- The official integration documentation example shows a consumer-side timestamp guard (`if (feedId == 2 && _timestamp > timestamp)`), but this is advisory, not enforced by the protocol contract. Consumers that omit or implement it incorrectly are silently vulnerable. [1](#0-0) 

### Recommendation
Add a staleness check inside `verifyUpdate` that reads the `uint64 timestamp` from the payload and reverts if it is older than a configurable `maxStalenessSeconds` threshold relative to `block.timestamp`:

```solidity
// After extracting payload:
uint64 payloadTimestamp = uint64(bytes8(payload[4:12]));
require(
    block.timestamp <= payloadTimestamp + maxStalenessSeconds,
    "stale price update"
);
```

Alternatively, enforce a monotonicity invariant by storing the last accepted timestamp per signer (or globally) and rejecting updates with a timestamp ≤ the stored value. This mirrors the protection already present in the Solana push oracle: [4](#0-3) 

### Proof of Concept
1. At time `T`, a Lazer price update is broadcast. An attacker captures the raw `update` bytes (EVM-encoded, signed by a trusted signer).
2. At time `T + N` (e.g., N = 1 hour), the attacker calls:
   ```solidity
   pythLazer.verifyUpdate{value: fee}(oldUpdateBytes);
   ```
3. `verifyUpdate` succeeds: the signature is valid, the signer has not expired.
4. The returned `payload` contains the timestamp from time `T`.
5. A consumer contract that does not check `payload.timestamp` against `block.timestamp` stores the hour-old price as current.
6. The attacker uses this stale price to trigger a favorable liquidation or manipulate a derivative position. [1](#0-0) [2](#0-1)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
