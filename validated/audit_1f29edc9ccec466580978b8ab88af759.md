Audit Report

## Title
Tx-Pool Admission Uses Size-Only Fee Check, Allowing Cheap Cycle-Budget Exhaustion Griefing - (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` gates pool admission using only serialized byte size, while actual resource cost is measured by `weight = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. A transaction with ~200 bytes but ~70M cycles pays only the size-based minimum fee yet forces the node to execute full script verification consuming ~70M cycles. No weight-based fee check is applied after `verify_rtx` returns, and `total_tx_cycles` is tracked but never used as an eviction trigger, enabling sustained CPU exhaustion at ~60× below the true cost.

## Finding Description

**Size-only admission gate (`tx-pool/src/util.rs` lines 42–45):**

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The comment explicitly acknowledges the gap. The fee check passes for any transaction whose fee ≥ `min_fee_rate × tx_size`, regardless of cycle consumption.

**No post-verification weight-based check (`tx-pool/src/process.rs` lines 724–754):**

After `verify_rtx` returns `verified.cycles`, the code immediately constructs the entry and submits it:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
// declared_cycles mismatch check only
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
```

There is no gate comparing `fee` against `min_fee_rate.fee(get_transaction_weight(tx_size, verified.cycles))`.

**Weight formula (`util/types/src/core/tx_pool.rs` lines 279, 298–303):**

```rust
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

For `size=200`, `cycles=70_000_000`: `weight = max(200, 11_940) = 11_940`. Weight-based fee rate = `200 / 11_940 × 1000 ≈ 16.7 shannons/KW` — ~60× below the 1,000 shannons/KW minimum.

**Weight-based fee rate used only for eviction, not admission (`tx-pool/src/component/entry.rs` lines 114–118, `sort_key.rs` lines 80–103):**

`fee_rate()` and `EvictKey` both use `get_transaction_weight`, but these only apply after the transaction is already in the pool. The attacker's transaction is correctly deprioritized for eviction, but the CPU cost of script execution has already been paid by the node.

**No cycle-based eviction trigger (`tx-pool/src/pool.rs` line 298, `tx-pool/src/component/pool_map.rs` lines 69–71):**

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size { ... }
```

`total_tx_cycles` is tracked in `pool_map` but is never used as an eviction condition. The pool can accumulate unbounded cycles from high-cycle/low-size transactions.

**Exploit flow:**
1. Deploy a script consuming ~69,999,999 cycles in a live cell dep (keeping tx size small).
2. Craft a ~200-byte transaction referencing that cell dep. Fee = 200 shannons.
3. Submit via `send_transaction` RPC. `check_tx_fee` passes (200 ≥ 1000 × 200 / 1000 = 200). `verify_rtx` executes the script consuming ~70M cycles. `TxEntry` is admitted.
4. The admitted entry has `fee_rate ≈ 16.7 shannons/KW`, far below minimum, but admission already occurred.
5. Repeat with different UTXOs. Each submission forces full script execution. Evicted transactions can be resubmitted continuously.

## Impact Explanation

**High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Each submitted transaction forces the node to spend CPU executing up to 70M cycles of script before the transaction can be evicted. The ~60× underpayment ratio means the attacker's cost per unit of CPU disruption is ~60× lower than for legitimate transactions. Continuous resubmission of evicted transactions sustains pool churn and CPU saturation at minimal marginal cost.

## Likelihood Explanation

- **Entry path**: Any unprivileged user via `send_transaction` RPC or P2P relay. No special role required.
- **Setup cost**: One high-cycle script deployed in a cell dep; attacker needs a moderate supply of UTXOs to sustain resubmission.
- **Persistence**: Evicted transactions can be immediately resubmitted, maintaining continuous CPU pressure.
- **Naturally occurring**: Complex lock scripts (ZK verifiers, multi-sig aggregation) submitted with minimum fees can trigger this without malicious intent.

## Recommendation

After `verify_rtx` returns `verified.cycles` in `_process_tx` (`tx-pool/src/process.rs`), apply a weight-based minimum fee check before constructing the `TxEntry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64())), snapshot));
}
```

The existing size-only check in `check_tx_fee` can be retained as a fast pre-filter before script execution. The post-verification check closes the gap between admission cost and actual resource cost.

## Proof of Concept

1. Deploy a CKB script that loops for exactly 69,999,999 cycles. Store it in a live cell on-chain.
2. Craft a transaction: one input (spendable UTXO), one output (capacity minus fee), one cell dep referencing the high-cycle script. Serialized size ≈ 200 bytes. Fee = 200 shannons.
3. Submit via `send_transaction` RPC.
   - `check_tx_fee`: `min_fee = 1000 × 200 / 1000 = 200 shannons` ✓ passes.
   - `verify_rtx` executes the script: ~70M cycles consumed.
   - `TxEntry` created: `size=200`, `cycles=70_000_000`, `fee=200`.
   - `fee_rate() = FeeRate::calculate(200, max(200, 11940)) ≈ 16.7 shannons/KW` — 60× below minimum.
4. Repeat with different UTXOs. Observe via `tx_pool_info` that `total_tx_cycles` grows rapidly while node CPU is saturated with script verification.
5. Confirm: attacker transactions are deprioritized for block inclusion (correct) but each submission still forces full script execution on the node.
