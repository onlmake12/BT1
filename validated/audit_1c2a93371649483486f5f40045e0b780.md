Audit Report

## Title
Unbounded Per-Request Resource Exhaustion via `GetBlocksProofProcess::execute` with No Rate Limiting — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary
Any unprivileged P2P peer can send a structurally valid `GetBlocksProof` message with up to 1000 distinct main-chain block hashes. The server unconditionally performs 3 synchronous DB reads per hash (3,000 total) plus O(N × log(chain_length)) MMR proof generation, with no per-peer rate limit. Because every valid request returns `Status::ok()`, the banning logic never fires, allowing indefinite repetition by the same peer.

## Finding Description
`GET_BLOCKS_PROOF_LIMIT` is set to 1000 with no accompanying throttle. [1](#0-0) 

`execute` enforces only structural validity — non-empty, ≤ 1000 items, no duplicates — and performs no per-peer accounting before proceeding to DB work. [2](#0-1) 

For every hash that passes `is_main_chain`, three synchronous DB reads are issued unconditionally: `get_block_header`, `get_block_uncles`, and `get_block_extension`. [3](#0-2) 

All collected positions are then passed to `mmr.gen_proof`, which reads O(N × log(chain_length)) MMR nodes from storage. [4](#0-3) 

A structurally valid request always returns `Status::ok()`. [5](#0-4) 

The banning logic only fires on 4xx status codes; a 200 response never triggers a ban. [6](#0-5) [7](#0-6) 

A grep across the entire `util/light-client-protocol-server` directory confirms zero occurrences of any rate-limiting, token-bucket, quota, or per-peer request-counting mechanism.


## Impact Explanation
A single attacker peer sending a continuous stream of max-size `GetBlocksProof` messages forces up to 3,000 synchronous RocksDB reads plus O(N × log(chain_length)) MMR node reads per request. At even modest request rates this saturates the node's DB I/O and CPU, starving the consensus reactor (block relay, IBD) of the DB access it needs. The node degrades or stalls without the attacker ever being banned. This matches the **High** bounty impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"* and *"Vulnerabilities which could easily crash a CKB node."*

## Likelihood Explanation
The attack requires only a valid P2P connection and 1000 main-chain block hashes, which are trivially obtained from any block explorer or by syncing headers. No proof-of-work, no key, and no privileged role is needed. The attacker can open multiple simultaneous connections to multiply the effect. The peer is never banned, so the attack can run indefinitely from a single IP.

## Recommendation
1. **Per-peer rate limit**: enforce a maximum number of `GetBlocksProof` requests per peer per time window (e.g., 1 request/second) using a token-bucket counter stored in peer state, checked at the top of `execute` before any DB work.
2. **Return a 4xx on rate-limit breach**: use an existing or new `StatusCode` in the 400–499 range so that `should_ban` fires and the offending peer is banned via `BAD_MESSAGE_BAN_TIME`. [6](#0-5) 
3. **Reduce the effective limit or charge proportional cost**: lower `GET_BLOCKS_PROOF_LIMIT` or require a cost proportional to the number of items before performing any DB work. [1](#0-0) 

## Proof of Concept
1. Connect to a CKB node with the light-client protocol enabled.
2. Obtain 1000 distinct main-chain block hashes (e.g., blocks 1–1000) from any block explorer.
3. In a tight loop, send `GetBlocksProof { last_hash: <tip>, block_hashes: [h1..h1000] }`.
4. Observe: RocksDB read IOPS spike; block relay latency increases; consensus participation degrades.
5. Confirm the peer is never banned by monitoring peer state — every response carries `Status::ok()` (code 200), which `should_ban` explicitly ignores. [6](#0-5)
