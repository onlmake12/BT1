### Title
Lazer `verifyUpdate` Does Not Validate Payload Timestamp Against Current Block Time - (`lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.verifyUpdate` verifies the ECDSA signature and checks that the signer key has not expired, but never compares the timestamp embedded in the signed payload against `block.timestamp`. Any unprivileged caller can replay an arbitrarily old but validly-signed Lazer price update, and the contract will accept it.

### Finding Description

`verifyUpdate` in `PythLazer.sol` performs two checks before returning the payload:

1. The ECDSA signature is valid.
2. The recovered signer address is currently trusted (`isValidSigner` checks `block.timestamp < trustedSignerToExpiresAtMapping[signer]`). [1](#0-0) 

What it does **not** check is whether the `timestamp` field embedded inside the payload is recent. The payload format includes a `uint64 timestamp` as its first field after the magic bytes, as parsed by `PythLazerLib.parsePayloadHeader`: [2](#0-1) 

Because `verifyUpdate` returns the raw `payload` bytes without any freshness assertion, a caller who captured a legitimately-signed Lazer update from days or weeks ago can submit it today. The contract will:
- Recover the signer from the signature ✓
- Confirm the signer is still in the trusted-signer list and not expired ✓
- Return the payload — which contains a stale price — without complaint ✓

The Sui Lazer contract checks signer expiry against the clock in `verify_le_ecdsa_message`, but similarly never validates the payload's own timestamp field against the current time: [3](#0-2) 

The Aptos Lazer contract has the same pattern — it checks `signer_info.expires_at > timestamp::now_seconds()` but never checks the payload timestamp: [4](#0-3) 

### Impact Explanation

A Lazer updater or any party who has stored historical signed Lazer payloads can submit an old price update to any consumer contract that calls `verifyUpdate`. If the consumer contract only checks `_timestamp > storedTimestamp` (as shown in the official integration example) rather than also checking `_timestamp` against `block.timestamp`, the consumer will accept the stale price as valid. This enables price manipulation: an attacker waits for a favorable historical price, then replays that signed payload to a lending protocol, perpetuals exchange, or other DeFi consumer, causing incorrect liquidations, mispriced trades, or fund extraction. [5](#0-4) 

### Likelihood Explanation

Lazer updates are broadcast over a WebSocket subscription. Any subscriber — including an attacker — can record every signed payload indefinitely. The attacker only needs to wait for a price that is favorable for their exploit, then replay it. No privileged access is required; the only cost is the `verification_fee` (currently 1 wei). The attack is straightforward and requires no cryptographic break.

### Recommendation

Add a staleness check inside `verifyUpdate` by parsing the payload timestamp and asserting it falls within an acceptable window of `block.timestamp`:

```solidity
// After extracting payload:
uint64 payloadTimestamp = uint64(bytes8(payload[4:12])); // after 4-byte magic
require(
    block.timestamp <= payloadTimestamp + MAX_STALENESS_SECONDS,
    "payload timestamp too old"
);
```

Alternatively, expose a `verifyUpdateWithMaxAge(bytes calldata update, uint256 maxAge)` variant that enforces freshness, and deprecate the unchecked form. The Sui and Aptos contracts should apply the same fix.

### Proof of Concept

1. At time `T`, attacker subscribes to the Lazer WebSocket and records a signed payload `P` containing a favorable price (e.g., ETH at $5000 when market is $3000).
2. At time `T + N days`, attacker calls `PythLazer.verifyUpdate{value: 1 wei}(P)`.
3. `isValidSigner` passes because the signer key has not expired.
4. `verifyUpdate` returns `payload` with no staleness error.
5. Attacker passes the returned payload to a victim DeFi contract that calls `parsePayloadHeader` and uses the price without checking `_timestamp` vs `block.timestamp`.
6. The victim contract processes the stale $5000 price, allowing the attacker to, e.g., borrow against inflated collateral. [1](#0-0)

public(package) fun verify_le_ecdsa_message(
    state: &State,
    clock: &Clock,
    signature: &vector<u8>,
    payload: &vector<u8>,
) {
    let current_cap = state.current_cap();

    // 0 stands for keccak256 hash
    let pubkey = secp256k1_ecrecover(signature, payload, 0);

    // Check if the recovered pubkey is in the trusted signers list
    let trusted_signers = state.trusted_signers(&current_cap);
    let mut maybe_idx = trusted_signers.find_index!(|signer|
        signer.public_key() == &pubkey
    );

    assert!(maybe_idx.is_some(), ESignerNotTrusted);
    let idx = maybe_idx.extract();
    let expires_at_ms = trusted_signers[idx].expires_at_ms();
    assert!(clock.timestamp_ms() < expires_at_ms, ESignerExpired);
