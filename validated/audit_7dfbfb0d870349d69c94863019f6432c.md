### Title
MMR `get_root()` I/O Amplification in `GetLastStateProof` Handler — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

An unprivileged remote peer can craft a valid `GetLastStateProof` message that causes the server to perform O(N × log H) RocksDB reads per request, where N ≤ 500 (bounded by `GET_LAST_STATE_PROOF_LIMIT`) and H is the chain height. The per-request limit is designed to bound work to O(N) but the `complete_headers` loop calls `chain_root_mmr(number-1).get_root()` once per block, each of which reads O(log H) MMR nodes from disk. No ban or rate-limit is applied to valid requests.

### Finding Description

**Limit check** at [1](#0-0) 

```
difficulties.len() + last_n_blocks * 2 <= GET_LAST_STATE_PROOF_LIMIT (1000)
```

With `difficulties=[]` and `last_n_blocks=499`: `0 + 998 = 998 ≤ 1000` — passes.

**`last_n_numbers` construction** (lines 291–319): with `difficulty_boundary=tip_total_difficulty`, the boundary block resolves to the tip itself, triggering the `< last_n_blocks` guard that sets `difficulty_boundary_block_number = tip - 499`, producing a 499-entry `last_n_numbers` vector. [2](#0-1) 

**`complete_headers` inner loop**: for every block number in that vector, the code unconditionally calls: [3](#0-2) 

```rust
let mmr = self.snapshot.chain_root_mmr(*number - 1);
match mmr.get_root() { ... }
```

**`chain_root_mmr` and `get_root()` cost**: `chain_root_mmr(n)` creates a `ChainRootMMR` of size `leaf_index_to_mmr_size(n)`. [4](#0-3) 

`get_root()` traverses the MMR tree, calling `get_elem(pos)` → `store.get_header_digest(pos)` for each of O(log H) internal nodes — each a separate RocksDB read. [5](#0-4) 

**No rate limiting or ban**: `lib.rs` only bans peers for malformed messages. A valid `GetLastStateProof` that passes all checks returns `Status::ok()` with no penalty. [6](#0-5) 

### Impact Explanation

For a mainnet chain at height H ≈ 10⁶, log₂(H) ≈ 20. A single max-sample request triggers:
- 499 × O(log H) ≈ ~10,000 RocksDB point reads for MMR roots alone
- Plus 499 block loads (`get_block`) and uncle hash computations

An attacker maintaining a persistent connection and sending back-to-back requests sustains continuous I/O amplification with no ban risk, degrading the node's ability to serve other peers and process chain data.

### Likelihood Explanation

Any peer that can open a light-client protocol connection can send this message. No PoW, stake, or privileged role is required. The crafted parameters are trivially derived from the tip state (which the server freely advertises via `GetLastState`). The attack is repeatable indefinitely.

### Recommendation

1. **Cache MMR roots per request**: compute `chain_root_mmr(tip-1).get_root()` once and reuse it for all headers in the same response, since all sampled blocks share the same tip MMR.
2. **Alternatively**, store pre-computed `parent_chain_root` values alongside block headers at write time, eliminating per-block MMR traversal entirely at read time.
3. **Add per-peer request rate limiting** in `try_process` for `GetLastStateProof` messages.

### Proof of Concept

```
peer → server: GetLastStateProof {
    last_n_blocks: 499,
    difficulties: [],
    last_hash: <tip_hash>,        // on main chain → passes is_main_chain check
    start_hash: <genesis_hash>,
    start_number: 0,              // reorg_last_n_numbers = []
    difficulty_boundary: <tip_total_difficulty>
}
```

Limit check: `0 + 499*2 = 998 ≤ 1000` → passes. [7](#0-6) 

Server builds `last_n_numbers = [tip-499, tip-498, …, tip-1]` (499 entries), then `complete_headers` calls `chain_root_mmr(n-1).get_root()` 499 times. Instrument `get_header_digest` call count: expect ~499 × log₂(tip) calls per request. Send at maximum rate; observe RocksDB read IOPS scale linearly with request rate × log(H), with no ban applied.
