### Title
Deviation-Based Update Permanently Blocked When Price Is Zero — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler._validateShouldUpdatePrices`, when either the stored previous price or the incoming current price is zero, the deviation check is silently skipped via `continue`. For subscriptions configured with only deviation-based update criteria (`updateOnDeviation: true`, `updateOnHeartbeat: false`), this means no update can ever be triggered when any subscribed price feed transitions to or from zero — even though such a transition represents an infinite or 100% price change, far exceeding any configured threshold.

---

### Finding Description

The `_validateShouldUpdatePrices` function in `Scheduler.sol` evaluates whether a price update satisfies the subscription's configured trigger criteria. When `updateOnDeviation` is enabled, it iterates over all price feeds and computes the deviation between the stored previous price and the incoming current price.

At line 429, to avoid a division-by-zero in the `Math.mulDiv` call, the code silently skips the deviation check for any feed where either price is zero:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;  // <-- silently skips this feed's deviation check
}
``` [1](#0-0) 

If the loop completes without any feed triggering the deviation threshold, the function falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [2](#0-1) 

This creates the following inconsistency, directly analogous to the reference report:

| Scenario | Previous Price | Current Price | Actual Deviation | Behavior |
|---|---|---|---|---|
| Normal | 100 | 200 | 100% | Deviation computed, update triggered if ≥ threshold |
| Zero-to-nonzero | 0 | 100 | Infinite | **Skipped**, update NOT triggered |
| Nonzero-to-zero | 100 | 0 | 100% | **Skipped**, update NOT triggered |
| Both zero | 0 | 0 | 0% | **Skipped**, update NOT triggered |

For a subscription with `updateOnDeviation: true` and `updateOnHeartbeat: false`, if all feeds in the subscription have a zero stored price or a zero incoming price, the loop completes without returning, and the function always reverts with `UpdateConditionsNotMet`. The subscription becomes permanently unable to update via the deviation path. [3](#0-2) 

The `updatePriceFeeds` function is callable by any unprivileged keeper (no access control): [4](#0-3) 

---

### Impact Explanation

**Subscriptions with deviation-only criteria are permanently DoS'd when any price feed has a zero value.** Downstream protocols relying on the Scheduler for price updates will receive stale prices indefinitely, since no keeper can successfully call `updatePriceFeeds` to satisfy the deviation condition. This is especially severe because:

1. The subscription owner has no recourse other than adding a heartbeat criterion (a parameter change requiring a separate `updateSubscription` call, which may not be possible for `isPermanent` subscriptions).
2. A price transition from non-zero to zero (e.g., an asset being delisted or a market halt) is precisely the scenario where downstream protocols most urgently need an updated price.
3. Pyth price feeds use `int64` for the price field, which can legitimately be zero. [5](#0-4) 

---

### Likelihood Explanation

Pyth price feeds can legitimately publish a price of zero (e.g., for certain synthetic assets, during market close, or for feeds that have been deprecated). The `price.price` field is `int64` and is not validated to be non-zero anywhere in the Scheduler contract before storage. Once a zero price is stored via a first update, all subsequent deviation-only updates for that feed are permanently skipped. [5](#0-4) 

---

### Recommendation

Instead of skipping the deviation check when either price is zero, treat a transition involving zero as always exceeding the deviation threshold (since the deviation is mathematically infinite or undefined). Replace the `continue` with an immediate return:

```solidity
// A transition to or from zero represents an infinite deviation — always trigger update
if (previousPrice == 0 || currentPrice == 0) {
    return updateTimestamp;
}
```

This ensures consistent behavior: any price change involving zero is treated as a significant deviation, matching the intent of the deviation-based update criterion. [1](#0-0) 

---

### Proof of Concept

1. Deploy `Scheduler` with a subscription configured as:
   - `updateOnHeartbeat: false`
   - `updateOnDeviation: true`, `deviationThresholdBps: 100` (1%)
   - One price feed, e.g. `feedId = 0xABC...`

2. Perform a first update with `price.price = 0` for `feedId`. This succeeds because `previousFeed.id == bytes32(0)` triggers the "first update" early return at line 420. [6](#0-5) 

3. Now attempt a second update with `price.price = 1000` (a large non-zero value, representing infinite deviation from 0).

4. In `_validateShouldUpdatePrices`, `previousPrice = 0`, so the condition at line 429 fires and `continue` is executed. The loop ends. The function reverts with `UpdateConditionsNotMet`.

5. The subscription is now permanently unable to update via the deviation path, regardless of how large the price change is. [3](#0-2)
