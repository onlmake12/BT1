Audit Report

## Title
Unbounded `pending_compact_blocks` Growth via Trivially-Mined Compact Blocks — (`sync/src/relayer/compact_block_process.rs`)

## Summary
The `HeaderVerifier` used during compact block processing verifies PoW against the header's own `compact_target` field without cross-checking it against the epoch's expected difficulty. An attacker can set `compact_target = 0x207fffff` (near-maximum target), trivially satisfy PoW with any nonce, and insert unlimited entries into the unbounded `pending_compact_blocks` map by referencing missing transactions. With no size cap, TTL, or per-peer limit, this causes unbounded heap growth leading to OOM and node crash.

## Finding Description

**PoW verification uses the header's own `compact_target`, not the epoch's expected difficulty.**

In `pow/src/eaglesong.rs`, the verifier derives the target directly from the header field: [1](#0-0) 

Setting `compact_target = 0x207fffff` yields a near-maximum target, so the Eaglesong output of virtually any nonce satisfies `output <= block_target`.

**The `HeaderVerifier` used in `contextual_check` does not validate `compact_target` against the epoch.**

`contextual_check` in `compact_block_process.rs` uses `HeaderVerifier` from `verification/src/header_verifier.rs`: [2](#0-1) 

This `HeaderVerifier::verify` calls `PowVerifier` (which accepts any nonce at max target), `NumberVerifier` (attacker sets correct number), `EpochVerifier` (only checks `is_well_formed()` and `is_successor_of()` — no `compact_target` check), and `TimestampVerifier` (attacker sets valid timestamp): [3](#0-2) 

The `EpochVerifier` in `header_verifier.rs` explicitly only checks structural well-formedness and epoch succession, never the `compact_target` value: [4](#0-3) 

By contrast, the `EpochVerifier` in `contextual_block_verifier.rs` (used only when a full block is accepted into the chain) does check `compact_target` against the epoch: [5](#0-4) 

This check is never reached for compact blocks that yield `Missing` or `Collided` — the attacker's goal is precisely to prevent full reconstruction.

**Height window is the only staleness guard, and it is wide.**

`non_contextual_check` rejects blocks only if `header.number() < tip - epoch_length` (~1000 blocks). An attacker targeting `tip + 1` always passes: [6](#0-5) 

**Unconditional insertion on `Missing` or `Collided` with no capacity check.**

`missing_or_collided_post_process` inserts into the map unconditionally: [7](#0-6) 

**The map is an unbounded `HashMap`.**

`PendingCompactBlockMap` is a plain `HashMap` with no capacity limit: [8](#0-7) 

Initialized as `HashMap::default()`: [9](#0-8) 

**The only eviction path requires full block reconstruction.**

Entries are removed only in the `ReconstructionResult::Block` branch. `shrink_to_fit!` compacts allocation but does not remove entries. There is no TTL sweep, no periodic cleanup, and no per-peer cap: [10](#0-9) 

**Deduplication is per `(block_hash, peer)`, not per peer.**

The pending check only blocks the same hash from the same peer: [11](#0-10) 

Different nonces produce different block hashes, so a single peer sending N headers with distinct nonces inserts N independent entries.

## Impact Explanation

Each `pending_compact_blocks` entry holds a full `CompactBlock` (header + short IDs + prefilled transactions + proposals). With no eviction until a legitimate block is accepted, an attacker accumulates thousands of entries consuming megabytes of heap. Additionally, `pending_compact_blocks` is protected by a `tokio::sync::Mutex`; a bloated map degrades every legitimate compact block relay that must acquire this lock. Sustained attack leads to OOM and node crash.

**Impact: High — Vulnerabilities which could easily crash a CKB node (10001–15000 points).**

## Likelihood Explanation

The attack requires no privileged access and no stake. The only cost is generating headers with `compact_target = 0x207fffff` against the current tip's parent hash — any nonce satisfies PoW at this target, so a single commodity machine produces thousands of valid headers per second. The attacker needs only one known valid parent block hash (the public chain tip) and can vary the nonce to produce arbitrarily many distinct block hashes. The attack is repeatable indefinitely and does not require victim mistakes.

## Recommendation

1. **Add a size cap** on `pending_compact_blocks`: reject insertion when the map exceeds a constant (e.g., 512 entries).
2. **Add a TTL sweep**: periodically evict entries older than a configurable timeout (e.g., 30 seconds) regardless of whether their transactions arrived.
3. **Add a per-peer entry limit**: track how many pending entries each peer has contributed and disconnect or ignore peers that exceed a threshold.
4. **Verify `compact_target` against the epoch** inside `contextual_check`, mirroring the check in `contextual_block_verifier.rs`'s `EpochVerifier`, so headers with an anomalously easy target are rejected before insertion.

## Proof of Concept

```
1. Connect to a CKB node as a relay peer.
2. Obtain the current tip block hash T and its epoch info.
3. For i in 1..1000:
     a. Build a header: parent_hash=T, number=tip+1,
        compact_target=0x207fffff, epoch=successor(tip.epoch),
        timestamp=tip.median_time+1, nonce=i
        (trivially satisfies PowVerifier since block_target is near-maximal).
     b. Wrap in a CompactBlock with one short_id referencing a
        transaction not in the node's mempool (guaranteed Missing).
     c. Send via the relay protocol.
4. Observe len(pending_compact_blocks) == 1000 and growing.
5. Repeat indefinitely; the map never shrinks until a legitimate block
   is accepted and its epoch evicts older entries.
```

impl<'a, DL: HeaderFieldsProvider> Verifier for HeaderVerifier<'a, DL> {
    type Target = HeaderView;
    fn verify(&self, header: &Self::Target) -> Result<(), Error> {
        // POW check first
        PowVerifier::new(header, self.consensus.pow_engine().as_ref()).verify()?;
        let parent_fields = self
            .data_loader
            .get_header_fields(&header.parent_hash())
            .ok_or_else(|| UnknownParentError {
                parent_hash: header.parent_hash(),
            })?;
        NumberVerifier::new(parent_fields.number, header).verify()?;
        EpochVerifier::new(parent_fields.epoch, header).verify()?;
        TimestampVerifier::new(
            self.data_loader,
            header,
            self.consensus.median_time_block_count(),
        )
        .verify()?;
        Ok(())
    }
