### Title
Unbounded Gas Forwarding in Legacy `revealWithCallback` Path Allows Permanent Request Griefing - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function has two execution branches. The modern path (V2, `gasLimit10k != 0`) safely uses `excessivelySafeCall` with a bounded gas limit and catches callback failures. The **legacy path** (`gasLimit10k == 0`, entered via `requestWithCallback`) calls `IEntropyConsumer(callAddress)._entropyCallback(...)` with **no gas cap and no error handling**. A malicious requester can deploy a contract whose `_entropyCallback` consumes all available gas, causing every provider `revealWithCallback` attempt for that request to revert. The request is permanently stuck and can never be cleared.

---

### Finding Description

`revealWithCallback` branches on `req.gasLimit10k`:

```
if (req.gasLimit10k != 0 && req.callbackStatus == CALLBACK_NOT_STARTED) {
    // SAFE: excessivelySafeCall with bounded gas, errors caught
} else {
    // UNSAFE: direct call, no gas cap, no try/catch
    clearRequest(provider, sequenceNumber);          // line 666
    ...
    IEntropyConsumer(callAddress)._entropyCallback(  // line 676
        sequenceNumber, provider, randomNumber
    );
}
``` [1](#0-0) 

The safe branch uses `excessivelySafeCall` with `uint256(req.gasLimit10k) * TEN_THOUSAND` as the gas cap and catches all reverts. [2](#0-1) 

The `else` branch (legacy path) calls `clearRequest` at line 666 before the callback, but if the callback reverts, the entire transaction reverts — including `clearRequest` — leaving the request permanently active. [3](#0-2) 

The `reveal` function cannot be used as a bypass: it checks `req.callbackStatus != CALLBACK_NOT_NECESSARY` and reverts with `InvalidRevealCall` for any request made via `requestWithCallback`. [4](#0-3) 

---

### Impact Explanation

- A malicious requester creates a request via `requestWithCallback` from a contract whose `_entropyCallback` always exhausts gas (e.g., `assert(false)` or an infinite loop).
- Every provider call to `revealWithCallback` for that sequence number reverts unconditionally.
- The request is permanently stuck: it cannot be cleared, fulfilled, or bypassed via `reveal`.
- The provider wastes gas on every fulfillment attempt and their off-chain fulfillment queue may stall if it processes requests sequentially.
- The attacker's fee is locked in the contract, but the cost is a one-time fee to permanently grief a provider's specific sequence slot.

---

### Likelihood Explanation

- Entry path is fully permissionless: any address can call `requestWithCallback` and pay the fee.
- No privileged role, leaked key, or external oracle manipulation is required.
- The attack is cheap: one fee payment creates a permanently unfulfillable request.
- Providers operating automated fulfillment bots are directly impacted if their queue is sequential.

---

### Recommendation

Apply the same `excessivelySafeCall` pattern used in the V2 path to the legacy `else` branch. Wrap the callback in a bounded gas call with error handling, or migrate all legacy requests to require a non-zero `gasLimit10k`. At minimum, wrap the call in a `try/catch` so that a reverting callback does not roll back `clearRequest`:

```solidity
// Instead of:
IEntropyConsumer(callAddress)._entropyCallback(sequenceNumber, provider, randomNumber);

// Use:
try IEntropyConsumer(callAddress)._entropyCallback{gas: LEGACY_GAS_CAP}(
    sequenceNumber, provider, randomNumber
) {} catch {}
``` [5](#0-4) 

---

### Proof of Concept

1. Deploy a malicious contract:
```solidity
contract PoisonRequester is IEntropyConsumer {
    function getEntropy() external pure override returns (address) { return address(0); }
    function _entropyCallback(uint64, address, bytes32) external override {
        // Consume all gas
        assembly { invalid() }
    }
}
```

2. Call `requestWithCallback(provider, userRandomNumber)` from `PoisonRequester`, paying the required fee. This sets `gasLimit10k = 0` on the stored request.

3. Provider calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`. Execution reaches line 676, `_entropyCallback` triggers `invalid()`, consuming all gas. The entire transaction reverts. `clearRequest` at line 666 is also reverted.

4. The request remains active with `callbackStatus == CALLBACK_NOT_STARTED`. `reveal` reverts with `InvalidRevealCall` (line 508–510). The request is permanently stuck. [3](#0-2) [6](#0-5)
