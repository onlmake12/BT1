### Title
`AddBlockFilterHash::migrate` Panics on Startup After Reorg When Fork-Chain Filter Data Exists Without Main-Chain Counterpart — (`util/migrate/src/migrations/add_block_filter_hash.rs`)

---

### Summary

The `AddBlockFilterHash::migrate` function (v0.108+) contains a logic error in its fork-chain walk: it computes `latest_built_filter_data_block_number = K` (the height of the first fork block), then unconditionally calls `.expect("filter data stored")` on the **main-chain** block at height K — which has no filter data when the filter service was interrupted mid-reorg. Any node that experienced a reorg while the filter service was running, then shut down before the filter service rebuilt the main-chain block at height K, will panic unconditionally on upgrade.

---

### Finding Description

**Fork-chain walk returns the wrong upper bound.**

In the fork branch of `AddBlockFilterHash::migrate`:

```rust
// find the fork block number
let mut header = chain_db
    .get_block_header(&block_hash)
    .expect("header stored");
while !chain_db.is_main_chain(&header.parent_hash()) {
    header = chain_db
        .get_block_header(&header.parent_hash())
        .expect("parent header stored");
}
header.number()   // ← returns K, the height of the first fork block
``` [1](#0-0) 

The loop terminates when `header.parent_hash()` is on the main chain, meaning `header` is the **fork block at height K** (the first block that diverged). `header.number()` is therefore K — the fork-point height — not K-1 (the last shared ancestor).

**The loop then reads main-chain filter data at height K.**

```rust
let mut block_number = 0;
loop {
    for _ in 0..10000 {
        if block_number > latest_built_filter_data_block_number { break; }
        let block_hash = chain_db.get_block_hash(block_number).expect("index stored");
        let filter_data = chain_db
            .get_block_filter(&block_hash)
            .expect("filter data stored");   // ← panics at block_number == K
        ...
        block_number += 1;
    }
}
``` [2](#0-1) 

`get_block_hash(K)` returns the **main-chain** block hash at height K. But `COLUMN_BLOCK_FILTER` only contains filter data for the **fork** block at height K (built before the reorg was detected). The main-chain block at K has no filter data yet. `get_block_filter` returns `None`, and `.expect("filter data stored")` panics. [3](#0-2) 

**How the DB reaches this state.**

The `build_filter_data` service in `block-filter/src/filter.rs` uses the identical fork-chain walk logic and starts rebuilding from height K on the new main chain after a reorg: [4](#0-3) 

`insert_block_filter` updates `META_LATEST_BUILT_FILTER_DATA_KEY` atomically per block: [5](#0-4) 

If the node shuts down **after** the reorg is committed to the chain state but **before** `build_filter_data_for_block` writes the new main-chain block at K, then:
- `META_LATEST_BUILT_FILTER_DATA_KEY` still points to the old fork block at height H (H ≥ K)
- `COLUMN_BLOCK_FILTER` has filter data for fork block at K, but **not** for main-chain block at K

On the next startup with v0.108+, `AddBlockFilterHash::migrate` runs and panics.

---

### Impact Explanation

The node process terminates with a panic during the mandatory migration step. The node cannot start at all. Recovery requires either manual DB surgery or a full re-sync. Every node that:
1. Had block filter enabled (v0.105+),
2. Experienced a reorg at height K,
3. Shut down before the filter service rebuilt the main-chain block at K,

will be permanently broken on upgrade to v0.108+.

---

### Likelihood Explanation

Reorgs are routine on any PoW chain. The window between a reorg being committed and the filter service writing the first new main-chain block is small but non-zero — any SIGTERM, OOM kill, power loss, or graceful shutdown during that window produces the vulnerable DB state. No attacker action is required; the scope rules explicitly include "database restart/reorg state" as a valid path.

---

### Recommendation

Change the fork-chain walk upper bound from `header.number()` (the fork block's height, K) to `header.number() - 1` (the last shared ancestor, K-1), so the loop only covers blocks whose main-chain filter data is guaranteed to exist:

```rust
// Before (buggy): returns K — the fork block height
header.number()

// After (correct): returns K-1 — the last common ancestor
header.number() - 1
// (guard against underflow if header.number() == 0)
```

The same off-by-one exists in `block-filter/src/filter.rs` `build_filter_data` and should be audited for consistency, though that path has a `build_filter_data_for_block` guard that skips already-built blocks. [6](#0-5) 

---

### Proof of Concept

1. Create a RocksDB fixture with:
   - `COLUMN_META[META_LATEST_BUILT_FILTER_DATA_KEY]` = hash of fork block at height K
   - `COLUMN_BLOCK_FILTER[fork_block_hash_K]` = valid filter bytes
   - `COLUMN_BLOCK_FILTER` has **no entry** for main-chain block hash at height K
   - `COLUMN_INDEX` maps height K → main-chain block hash (different from fork block hash)
2. Run `AddBlockFilterHash::migrate` against this DB.
3. Observe unconditional panic: `"filter data stored"` at `block_number == K`. [3](#0-2)
