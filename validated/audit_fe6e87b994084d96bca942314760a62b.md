Audit Report

## Title
Missing `BLOCK_INVALID` Early-Return Causes Redundant PoW Re-Verification — (`sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` at lines 301–302 noting that it should return early when a header's status is already `BLOCK_INVALID`, but it does not. Because `BLOCK_INVALID = 1 << 12` does not set the `HEADER_VALID = 1` bit, the `status.contains(BlockStatus::HEADER_VALID)` guard at line 304 evaluates to `false` for `BLOCK_INVALID` headers, causing execution to fall through and redundantly re-run `prev_block_check`, `non_contextual_check` (including Eaglesong PoW via `HeaderVerifier::verify()`), and `version_check` for every header already cached as invalid. The peer is banned after this single message via `StatusCode::HeadersIsInvalid` → `should_ban()`.

## Finding Description
In `HeaderAcceptor::accept()` (lines 295–358 of `sync/src/synchronizer/headers_process.rs`), after fetching block status at line 303, only the `HEADER_VALID` path has an early-return (lines 304–322). The `BLOCK_INVALID` path has no early-return, as explicitly noted by the `FIXME` at lines 301–302. Since `BLOCK_INVALID = 1 << 12 = 4096` and `HEADER_VALID = 1` (confirmed in `shared/src/block_status.rs` lines 11 and 16), the bitwise check `status.contains(BlockStatus::HEADER_VALID)` evaluates to `false` for any `BLOCK_INVALID` header, and execution falls through to lines 324–354 where `prev_block_check`, `non_contextual_check` (calling `self.verifier.verify(self.header)` at line 256, which includes full Eaglesong PoW verification), and `version_check` are all re-executed. These checks ultimately fail and return `ValidationState::Invalid`, causing `execute()` to return `StatusCode::HeadersIsInvalid` (code 415). The `should_ban()` function in `sync/src/status.rs` lines 165–179 returns `Some(BAD_MESSAGE_BAN_TIME)` for all 4xx codes except `GetHeadersMissCommonAncestors`, so the peer is banned after this single message. Sustained attack requires reconnection after each ban, which requires Sybil capability. The "state confusion" scenario is not caused by this bug: after a node restart, `block_status_map` is cleared entirely, so `get_block_status` returns `UNKNOWN` and the missing early-return is never triggered.

## Impact Explanation
The concrete impact is redundant CPU computation — re-running `HeaderVerifier::verify()` including Eaglesong PoW verification for headers already cached as `BLOCK_INVALID` — before the peer is banned. This does not crash a node, does not cause consensus deviation, and does not cause network-wide congestion. The impact is bounded to one redundant PoW verification per peer connection before the ban. This fits **Low (501–2000 points): Any other important performance improvements for CKB**.

## Likelihood Explanation
Any P2P peer can send a `SendHeaders` message containing a known-`BLOCK_INVALID` header. However, the peer is banned after the first such message via `StatusCode::HeadersIsInvalid` → `should_ban()` → `BAD_MESSAGE_BAN_TIME`. Sustained attack requires reconnection after each ban, requiring either many IPs or repeated reconnections (Sybil capability). The redundant computation per message is real but bounded and not catastrophic.

## Recommendation
Add an explicit early-return for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept()`, resolving the `FIXME` at line 301 of `sync/src/synchronizer/headers_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent));
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early-return logic
}
```

## Proof of Concept
1. Connect to a CKB node as a P2P peer.
2. Send a `SendHeaders` message with a header `H` whose hash is in `block_status_map` as `BLOCK_INVALID`.
3. Observe via tracing/logging that `prev_block_check` and `non_contextual_check` (including `HeaderVerifier::verify()`) are executed before the `Invalid` result is returned — rather than returning immediately after the status check at line 303.
4. Confirm the peer is banned after this single message via `StatusCode::HeadersIsInvalid` → `should_ban()` → `BAD_MESSAGE_BAN_TIME`.
5. The `FIXME` at lines 301–302 of `sync/src/synchronizer/headers_process.rs` is the developers' own acknowledgment of the missing path.
