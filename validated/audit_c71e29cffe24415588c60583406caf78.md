Audit Report

## Title
Unbounded Descendant BFS in `remove_entry` Enables O(N) CPU Stall on Block Commit — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry` unconditionally calls `update_descendants_index_key`, which performs an unbounded BFS over all descendants via `calc_descendants`/`calc_relation_ids`. Because `cell_dep` references create parent-child links in `TxLinksMap` with no symmetric `max_descendants_count` enforcement, an attacker can accumulate N children under a single root transaction. When that root is committed in a block, the tx-pool service thread performs O(N) work traversing and updating all descendants, stalling all subsequent pool operations.

## Finding Description

**`remove_entry` calls `update_descendants_index_key` unconditionally:** [1](#0-0) 
Both `update_ancestors_index_key` and `update_descendants_index_key` are called with no guard on descendant set size.

**`update_descendants_index_key` performs an unbounded BFS:** [2](#0-1) 
It calls `self.links.calc_descendants`, which delegates to `calc_relative_ids` → `calc_relation_ids`: [3](#0-2) 
This is an iterative BFS with no depth or count cap. Every descendant then receives a `modify_by_id` call on the multi-index map.

**`cell_dep` references create parent-child links:** [4](#0-3) 
Any in-pool transaction referenced as a `cell_dep` is inserted into `parents`. Then `_record_ancestors` calls `self.links.add_child(parent, short_id)` for each parent: [5](#0-4) 
This makes the root a parent of every such child in `TxLinksMap`.

**No `max_descendants_count` exists anywhere in the codebase** — confirmed by zero matches for `max_descendants`. Only `max_ancestors_count` is enforced: [6](#0-5) 

**`remove_entry_and_descendants` avoids the issue by pre-clearing links:** [7](#0-6) 
The comment explicitly states this pre-clearing is done "so that we won't update_descendants_index_key in remove_entry." Committed transactions cannot use this path because their descendants must remain in the pool, so they go through plain `remove_entry`, which does not pre-clear links.

**Integration test confirms 2,000 cell_dep children are accepted with no enforcement:** [8](#0-7) 

## Impact Explanation
The tx-pool service thread is single-threaded. An O(N) stall during block-commit processing delays block-template generation, transaction relay, and all subsequent pool operations. With the default 180 MB pool and minimal transaction sizes (~100–200 bytes), N can reach tens of thousands. A sustained attack keeps the tx-pool thread near 100% CPU, degrading block propagation latency and potentially causing the node to fall behind the chain tip. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
No privileged access is required — only the ability to submit valid fee-paying transactions via RPC or P2P relay. Each child transaction has `ancestors_count = 2`, well within the 1,000 limit, so no existing check rejects them. The attack is repeatable after every block commit. The existing integration test already demonstrates 2,000 such transactions being accepted in a single run, confirming the precondition is trivially satisfiable.

## Recommendation
1. **Add a `max_descendants_count` limit** symmetric to `max_ancestors_count`. Enforce it in `_record_ancestors` or `record_entry_descendants` when a new child is linked — reject the child if the parent's transitive descendant count would exceed the limit.
2. **Alternatively**, cap `edges.deps[out_point].len()` to bound the number of in-pool transactions that may reference the same `out_point` as a `cell_dep`.
3. **Short-term mitigation**: increase `min_fee_rate` to raise the cost of filling the pool with fan-out descendants.

## Proof of Concept
```
1. Submit tx_root via send_transaction RPC (any valid tx, fee >= min_fee_rate).
2. For i in 1..N:
     Submit tx_i with:
       - inputs: [some unrelated live cell_i]
       - cell_deps: [OutPoint { tx_hash: tx_root.hash(), index: 0 }]
     Each tx_i has ancestors_count = 2, accepted by the pool.
3. Assert tx_root has N children in TxLinksMap (no enforcement prevents this).
4. Mine a block containing tx_root.
5. Node calls remove_entry(tx_root_id):
     → update_descendants_index_key calls calc_descendants → BFS over N entries
     → N modify_by_id calls on the multi-index map
6. Measure tx-pool thread CPU time; observe O(N) scaling.
7. Repeat from step 1 with a new tx_root.

Extend the existing TxPoolLimitAncestorCount integration test
(test/src/specs/tx_pool/limit.rs L93-100, already submits N=2000)
to mine a block containing tx_a and measure remove_entry latency.
```

    fn update_descendants_index_key(&mut self, parent: &TxEntry, op: EntryOp) {
        let descendants: HashSet<ProposalShortId> =
            self.links.calc_descendants(&parent.proposal_short_id());
        for desc_id in &descendants {
            // update child score
            self.entries.modify_by_id(desc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_ancestor_weight(parent),
                    EntryOp::Add => e.inner.add_ancestor_weight(parent),
                };
                e.score = e.inner.as_score_key();
            });
        }
    }

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
    }
