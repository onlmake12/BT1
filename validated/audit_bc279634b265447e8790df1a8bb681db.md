Audit Report

## Title
Stale Descendant-Weight Accounting in `remove_entry_and_descendants` Corrupts Eviction Priority â€” (`File: tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` strips all parent/child links for every entry in the removal set before calling `remove_entry` on each one. Because `remove_entry` relies on those same links to locate ancestors and decrement their `descendants_*` fields via `update_ancestors_index_key`, the pre-removal of links silently skips the update for any ancestor that lies outside the removal set. Those surviving ancestors are left with permanently inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`, which corrupts their `EvictKey` and causes them to be systematically skipped during pool eviction.

## Finding Description
`remove_entry_and_descendants` collects the target entry and all its descendants, then calls `remove_entry_links` on every entry in that set before iterating and calling `remove_entry` on each: [1](#0-0) 

`remove_entry_links` removes the entry from its parents' children lists, removes the entry from its children's parents lists, and then removes the entry from the links map entirely: [2](#0-1) 

When `remove_entry` is subsequently called, it invokes `update_ancestors_index_key`, which resolves ancestors through `self.links.calc_ancestors`: [3](#0-2) 

Because `remove_entry_links` was already called for every entry in `removed_ids`, `calc_ancestors` returns an empty set for each of them. The `sub_descendant_weight` call is therefore never reached for any ancestor outside the removal set: [4](#0-3) 

The comment in `remove_entry_and_descendants` acknowledges the intent ("so that we won't update_descendants_index_key in remove_entry") but the side-effect of also suppressing `update_ancestors_index_key` for surviving ancestors is unintended. The `update_descendants_index_key` path (which updates `ancestors_*` fields of descendants being removed) is correctly skipped since those entries are all being removed anyway, but the `update_ancestors_index_key` path (which updates `descendants_*` fields of surviving ancestors) must not be skipped.

**Concrete exploit flow:**
1. Submit `tx_A` (low fee) â€” enters pool; `tx_A.descendants_fee = fee_A`.
2. Submit `tx_B` (child of `tx_A`, high fee) â€” `tx_A.descendants_fee = fee_A + fee_B`.
3. Submit `tx_C` (spends the same input as `tx_B`) â€” `resolve_conflict` calls `remove_entry_and_descendants(tx_B_id)`.
4. Inside `remove_entry_and_descendants`: `removed_ids = [tx_B]`; `remove_entry_links(tx_B)` strips tx_B from the links map; `remove_entry(tx_B)` â†’ `calc_ancestors(tx_B)` returns âˆ… â†’ `tx_A.sub_descendant_weight(tx_B)` is **never called**.
5. After the call: `tx_A.descendants_fee` remains `fee_A + fee_B` even though `tx_B` no longer exists.

The same stale-state condition is triggered by every call site of `remove_entry_and_descendants`: `resolve_conflict`, `resolve_conflict_header_dep`, `process_rbf`, `limit_size`, and `remove_by_detached_proposal`. [5](#0-4) 

The existing test `test_resolve_conflict_descendants` only asserts that the conflicted entries are absent from the pool; it does not assert that `tx1`'s `descendants_*` fields are correctly decremented, confirming the regression is untested: [6](#0-5) 

## Impact Explanation
The `EvictKey` for each pool entry is derived directly from the (now-stale) descendant fields: [7](#0-6) 

`EvictKey` ordering places the entry with the lowest `fee_rate` first for eviction: [8](#0-7) 

`next_evict_entry` selects the first entry in `iter_by_evict_key()` order, and `limit_size` uses this to decide which entries to remove when the pool is over capacity: [9](#0-8) [10](#0-9) 

A surviving ancestor with an inflated `descendants_fee` reports a falsely high `fee_rate` in its `EvictKey` and is therefore systematically skipped during eviction. An attacker can repeat the three-step pattern across many transactions to fill the pool with low-fee entries that cannot be evicted, preventing legitimate higher-fee transactions from entering the pool. This matches the **High** impact category: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation
The trigger path requires no special privilege. Any user with RPC access can call `send_transaction` to submit `tx_A`, `tx_B`, and `tx_C` in sequence. The conflict resolution path (`resolve_conflict`) is exercised on every transaction submission that spends an already-spent input. The pattern is repeatable indefinitely, and each iteration permanently inflates the `descendants_*` fields of one additional surviving ancestor. The attacker's cost per inflation event is the sum of fees for three transactions, which can be kept near the minimum fee rate for `tx_A` and `tx_C`; only `tx_B` needs to be a child of `tx_A`.

## Recommendation
Before stripping links, explicitly update the `descendants_*` fields of all ancestors that are **not** in the removal set:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();

    // Decrement descendants_* for surviving ancestors BEFORE links are removed.
    for rid in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(rid).map(|e| e.inner.clone()) {
            let ancestors = self.links.calc_ancestors(rid);
            for anc_id in ancestors {
                if !removed_set.contains(&anc_id) {
                    self.entries.modify_by_id(&anc_id, |e| {
                        e.inner.sub_descendant_weight(&entry);
                        e.evict_key = e.inner.as_evict_key();
                    });
                }
            }
        }
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }
    removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
}
```

Alternatively, add a regression test that asserts `descendants_*` fields of surviving ancestors are correctly decremented after `remove_entry_and_descendants` is called.

## Proof of Concept
**Setup**: Pool contains `tx1 â†’ tx3 â†’ tx4` (tx3 spends tx1's output 0, tx4 spends tx3's output 0). After adding all three, `tx1` has `descendants_count=3`, `descendants_fee=fee1+fee3+fee4`.

**Trigger**: Submit `tx2` that spends `tx1`'s output 0 (same as tx3). `resolve_conflict` calls `remove_entry_and_descendants(tx3_id)`.

**Inside `remove_entry_and_descendants`**:
- `removed_ids = [tx3, tx4]`
- `remove_entry_links(tx3)` and `remove_entry_links(tx4)` are called â€” all links gone.
- `remove_entry(tx3)` â†’ `update_ancestors_index_key(tx3, Remove)` â†’ `calc_ancestors(tx3)` returns âˆ… â†’ `tx1.sub_descendant_weight(tx3)` is **never called**.
- `remove_entry(tx4)` â†’ same result.

**After**: `tx1` still reports `descendants_count=3`, `descendants_fee=fee1+fee3+fee4`. Its `EvictKey.fee_rate` is computed from ghost fees of tx3 and tx4. When the pool fills and `limit_size` runs, `tx1` is ranked as if it still has two high-fee descendants and is skipped for eviction.

A minimal unit test can confirm this by asserting `pool.get_entry(tx1_id).descendants_count == 1` and `pool.get_entry(tx1_id).descendants_fee == fee1` after calling `pool.resolve_conflict(&tx2)` â€” both assertions will currently fail.

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }

                }
            }
        }
