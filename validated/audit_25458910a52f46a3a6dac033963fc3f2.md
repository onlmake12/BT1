Audit Report

## Title
Proposed Transactions Shield Pool Space From Fee-Rate Eviction, Enabling Sustained Tx-Pool DoS — (`File: tx-pool/src/pool.rs`)

## Summary

`TxPool::limit_size` enforces a strict status-priority eviction order (Pending → Gap → Proposed) regardless of fee rate. A single shared `total_tx_size` counter covers all statuses. An attacker who fills the pool with minimum-fee transactions and allows miners to propose them can permanently occupy pool space that is immune to eviction, causing every subsequent Pending submission — regardless of fee rate — to be immediately rejected with `Reject::Full`.

## Finding Description

`PoolMap` maintains one `total_tx_size` counter that accumulates the serialized size of every entry regardless of its `Status`: [1](#0-0) 

When `total_tx_size` exceeds `max_tx_pool_size`, `limit_size` runs an eviction loop that unconditionally tries Pending first, then Gap, then Proposed: [2](#0-1) 

Within each status group, `next_evict_entry` selects the entry with the lowest `EvictKey` by filtering by status: [3](#0-2) 

`EvictKey` ordering is fee-rate ascending, so the cheapest transaction in the chosen status group is evicted first: [4](#0-3) 

`limit_size` is called immediately after every successful insertion, with `current_entry_id` pointing to the just-inserted transaction: [5](#0-4) 

**Exploit flow:**

1. Attacker submits N minimum-fee transactions until `total_tx_size` approaches `max_tx_pool_size` (180 MB default). They enter as `Status::Pending`.
2. Miners propose them; they transition to `Status::Gap` / `Status::Proposed` via `set_entry`. No `Status::Pending` entries remain.
3. A legitimate user submits a high-fee transaction. `add_entry` succeeds and increments `total_tx_size` above the limit.
4. `limit_size` is called. The loop finds `total_tx_size > max_tx_pool_size`, calls `next_evict_entry(Status::Pending)`, which returns the just-inserted high-fee transaction (the only Pending entry). It is evicted and `ret = Some(Reject::Full(...))` is set.
5. The Proposed entries — even those with lower fee rates — are never examined. The caller receives `Reject::Full`.

There is no cross-status fee-rate comparison and no separate capacity budget for the Proposed sub-pool.

## Impact Explanation

Once the pool is saturated with Proposed transactions, every `send_transaction` RPC call or P2P relay submission for a new Pending transaction is rejected with `Reject::Full`, regardless of fee rate. Legitimate users cannot enter the pool because the eviction logic will always evict the newly inserted Pending entry before touching any lower-fee Proposed entry. This constitutes a sustained, complete denial-of-service against transaction submission for the duration the Proposed transactions remain uncommitted (up to the 12-hour default expiry). This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

The attack is cheap and fully unprivileged. At the default minimum fee rate of 1,000 shannons/KB and a 180 MB pool, filling the pool costs approximately 1.8 CKB in fees: [6](#0-5) 

The entry path is the public `send_transaction` RPC or any P2P peer relay — no special privileges required. The attacker sustains the DoS by continuously re-submitting new minimum-fee transactions to replace those that eventually get committed, keeping the pool perpetually full of Proposed entries.

## Recommendation

Replace the status-priority eviction order with a globally fee-rate-aware selection: pick the entry with the lowest `EvictKey` across **all** statuses rather than filtering by status first. Alternatively, impose a separate size cap on the Proposed sub-pool (e.g., a configurable `max_proposed_tx_pool_size`) so that Proposed transactions cannot consume more than a bounded fraction of `max_tx_pool_size`, preserving headroom for new Pending submissions regardless of how many transactions have been proposed.

## Proof of Concept

1. Submit minimum-fee transactions via `send_transaction` until `pool_map.total_tx_size` approaches `max_tx_pool_size` (180 MB). All enter as `Status::Pending`.
2. Allow miners to propose them. Observe via RPC that entries transition to `Status::Gap` / `Status::Proposed`. No `Status::Pending` entries remain.
3. Submit a new transaction with a fee rate well above `min_fee_rate`.
4. Observe that `submit_entry` calls `_submit_entry` (succeeds, `total_tx_size` exceeds limit), then calls `limit_size(…, Some(&entry.proposal_short_id()))`.
5. Inside `limit_size`, `next_evict_entry(Status::Pending)` returns the just-inserted high-fee transaction (the only Pending entry). It is evicted and `ret = Some(Reject::Full(...))` is set.
6. The caller receives `Reject::Full`. All Proposed entries with lower fee rates remain untouched.
7. Repeat step 3 to confirm the DoS is sustained: every new Pending submission is immediately rejected.
