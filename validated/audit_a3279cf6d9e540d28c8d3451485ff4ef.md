### Title
`check_tx_fee` Uses Serialized Size Instead of Actual Weight, Allowing Below-`min_fee_rate` Transactions Into the Pool — (`tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the `min_fee_rate` admission gate using only the transaction's serialized byte size as the weight denominator. The actual weight used everywhere else in the pool is `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For any transaction whose cycle consumption dominates its byte size, the admission check uses a reference value that is strictly smaller than the true weight, so the check passes even when the real fee rate is far below `min_fee_rate`. This is structurally identical to the BondingTax bug: the bound is derived from the same dynamic quantity being validated, making the guard trivially satisfiable.

---

### Finding Description

`check_tx_fee` is the sole fee-rate admission gate in the tx-pool processing pipeline:

```rust
// tx-pool/src/util.rs  lines 28-54
pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(...).transaction_fee(rtx)?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
    Ok(fee)
}
``` [1](#0-0) 

The comment acknowledges the approximation but treats it as a "cheap pre-check." The problem is that no accurate weight-based re-check is ever performed after script verification. In `_process_tx`, the flow is:

1. `pre_check` → calls `check_tx_fee` (size-only weight) — **only fee-rate gate**
2. `verify_rtx` → executes scripts, returns `verified.cycles`
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry stored with correct cycles
4. `submit_entry` — admitted to pool [2](#0-1) 

After step 2 the actual cycles are known, but there is no step that re-evaluates `fee >= min_fee_rate × get_transaction_weight(size, cycles)`. The cheap check is the only check.

The correct weight formula is:

```rust
// util/types/src/core/tx_pool.rs  lines 298-303
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. With the default `max_tx_verify_cycles = 70_000_000`, the maximum cycle-derived weight is `≈ 11,940 bytes`. A minimal transaction can be as small as `~100 bytes`. The size-based check therefore underestimates the required fee by up to **~119×**. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

An unprivileged tx-pool submitter (RPC `send_transaction` or P2P relay `RelayTransaction`) can craft a transaction that:

- Has a small serialized size (e.g., 100 bytes)
- Executes a script consuming near-`max_tx_verify_cycles` cycles (e.g., 70 M cycles → weight ≈ 11,940 bytes)
- Pays a fee that satisfies `fee >= min_fee_rate × size` but violates `fee >= min_fee_rate × weight`

Such a transaction is admitted to the pool with an effective fee rate far below `min_fee_rate`. Consequences:

1. **Spam admission**: The pool's anti-spam gate is bypassed; low-fee-rate transactions occupy pool memory and CPU.
2. **Relay propagation**: Admitted transactions are relayed to peers, which run the same flawed check and also admit them, amplifying the effect across the network.
3. **Pool eviction distortion**: Eviction uses the correct `TxEntry::fee_rate()` (which uses `get_transaction_weight`), so these transactions are eventually evicted when the pool is full — but only after consuming resources and displacing legitimate transactions. [6](#0-5) 

---

### Likelihood Explanation

Any script author can write a lock or type script that loops for many cycles while keeping the transaction's serialized byte size small (the script binary lives in a cell dep, not in the transaction itself). The attacker controls both the fee amount and the script cycle count, making this straightforward to exploit deliberately. The entry path is fully unprivileged: `send_transaction` RPC or P2P relay.

---

### Recommendation

After `verify_rtx` returns `verified.cycles`, re-check the fee rate using the actual weight:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

The existing size-only pre-check can remain as an early-exit optimization before the expensive script execution, but the accurate weight-based check must be enforced after cycles are known.

---

### Proof of Concept

**Setup**: `min_fee_rate = 1000 shannons/KW` (default), `DEFAULT_BYTES_PER_CYCLES ≈ 0.000170571`.

**Attacker crafts a transaction**:
- Serialized size: `100 bytes`
- Script cycles consumed: `1,000,000` → weight = `max(100, 1,000,000 × 0.000170571)` = `max(100, 170)` = **170 bytes**
- Fee paid: `100 shannons`

**`check_tx_fee` evaluation** (size-only):
```
min_fee = 1000 × 100 / 1000 = 100 shannons
fee (100) >= min_fee (100)  →  PASSES
```

**Correct weight-based evaluation** (never performed):
```
min_fee = 1000 × 170 / 1000 = 170 shannons
fee (100) < min_fee (170)   →  SHOULD REJECT
```

The transaction is admitted to the pool with an effective fee rate of `100 × 1000 / 170 ≈ 588 shannons/KW`, which is **41% below** the configured `min_fee_rate` of 1000 shannons/KW. At the maximum cycle count of 70 M cycles (weight ≈ 11,940 bytes), the attacker can pay only `100 shannons` instead of the correct `11,940 shannons` — a **~119× underpayment** — and still pass admission.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
