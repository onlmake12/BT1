Audit Report

## Title
`check_purge` `take(len/2)` integer truncation allows single-group peer store flood to block new peer admission — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
When all entries in the peer store share the same `/16` network group, the second-pass eviction in `check_purge` computes `take(1 / 2) == take(0)`, producing an empty candidate list and returning `Err(PeerStoreError::EvictionFailed)`. An attacker with a single active connection can advertise 16,384 addresses from one `/16` subnet via discovery, saturating the peer store and blocking all subsequent `add_addr` calls until the fake addresses age out.

## Finding Description
`add_addr` calls `check_purge()` before inserting any new address once the store reaches `ADDR_COUNT_LIMIT` (16,384). [1](#0-0) [2](#0-1) 

`check_purge`'s first pass evicts addresses where `is_connectable` returns `false`. Fresh attacker-injected addresses have `last_connected_at_ms=0` and `attempts_count=0`; since `0 < ADDR_MAX_RETRIES (3)`, `is_connectable` returns `true` and they survive the first pass. [3](#0-2) 

When the first pass evicts nothing, the second pass groups addresses by `Group` (first two IPv4 octets only) and calls `take(len / 2)`. [4](#0-3) 

`Group::IP4` is keyed on only the first two octets, so all 65,535 IPs in a `/16` map to the same key. [5](#0-4) 

If all 16,384 addresses share one `/16`, `peers_by_network_group.len() == 1`, so `take(1 / 2) == take(0)`. The iterator is empty, `candidate_peers` is empty, and `EvictionFailed` is returned. [6](#0-5) 

There is also a secondary failure mode: even when `len > 1`, the `flat_map` only yields candidates when `addrs.len() > 4`, so an attacker distributing addresses across two groups of ≤ 4 each would also produce an empty `candidate_peers`. [7](#0-6) 

## Impact Explanation
`add_addr` propagates `EvictionFailed` to every subsequent discovery-advertised address. The node's peer store is saturated with attacker-controlled addresses; legitimate peer addresses cannot be stored. Peer discovery is effectively halted until the fake addresses age out through failed feeler connection attempts (`ADDR_MAX_RETRIES=3` failures per address × 16,384 entries at a slow feeler rate). [8](#0-7) 

This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*. An attacker can apply this to multiple nodes simultaneously with minimal effort, degrading the network's ability to route and discover peers.

## Likelihood Explanation
Any peer connected to the victim can send `Nodes` discovery responses. No per-group insertion cap exists at `add_addr` time. Filling 16,384 slots requires only sending addresses in bulk over the standard discovery protocol — only the subnet homogeneity is adversarial. The attacker needs a single active connection to the victim, which is a normal P2P precondition.

## Recommendation
Replace `take(len / 2)` with `take(len.saturating_add(1) / 2)` (ceiling division) so that a single group is still selected when `len == 1`. Additionally, enforce a per-group insertion cap at `add_addr` time to prevent any single `/16` from consuming a disproportionate share of the store. [9](#0-8) 

## Proof of Concept
```rust
let mut store = PeerStore::default();
// Fill store with 16384 addresses all in Group::IP4([1, 2])
for i in 0u32..ADDR_COUNT_LIMIT as u32 {
    let ip = format!("1.2.{}.{}", (i >> 8) & 0xff, i & 0xff);
    let addr: Multiaddr = format!(
        "/ip4/{}/tcp/8115/p2p/QmYyQSo1c1Ym7orWxLYvCrM2EmxFTANf8wXmmE7DWjhx5N", ip
    ).parse().unwrap();
    store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
// Store is now at ADDR_COUNT_LIMIT, all in Group::IP4([1,2])
// First pass: all addresses are connectable (attempts_count=0), nothing evicted
// Second pass: len=1, take(1/2)==take(0), candidate_peers empty -> EvictionFailed
let extra: Multiaddr =
    "/ip4/9.9.9.9/tcp/8115/p2p/QmYyQSo1c1Ym7orWxLYvCrM2EmxFTANf8wXmmE7DWjhx5N"
    .parse().unwrap();
assert!(matches!(
    store.add_addr(extra, Flags::COMPATIBILITY),
    Err(e) if e.to_string().contains("EvictionFailed")
));
```

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
