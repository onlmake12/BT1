### Title
Entropy Provider Can Permanently Lock User Fees With No Refund or Cancel Mechanism - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract credits provider fees to the provider's balance immediately upon a user's randomness request, before the provider has revealed anything. There is no timeout, cancel, or refund function anywhere in the contract. If a provider refuses to reveal (or goes permanently offline), the user's paid native-token fee is locked in the contract forever with no on-chain escape hatch.

---

### Finding Description

In `requestHelper`, at the moment a user submits a request, the fee is split and credited unconditionally:

```solidity
// Entropy.sol lines 233–239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;          // provider credited immediately
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

The provider can immediately call `withdraw()` to extract their credited fee. Completing the request, however, requires the provider to supply their secret hash-chain value `x_i` to `revealWithCallback` or `reveal`. Without it, `revealHelper` always reverts:

```solidity
// Entropy.sol lines 405–408
if (
    keccak256(bytes.concat(userCommitment, providerCommitment)) !=
    req.commitment
) revert EntropyErrors.IncorrectRevelation();
```

The only function that clears a pending request is `clearRequest`, which is called exclusively from `reveal` (line 529) and `revealWithCallback` (lines 620, 666) — both of which require the provider's secret. There is no `cancelRequest`, `refundRequest`, or block-timeout mechanism anywhere in the contract. A search of the entire `entropy/` directory confirms zero occurrences of any refund or cancel path.

The protocol design documentation itself acknowledges the underlying trust dependency:

> *"Providers are trusted to reveal their random number (x_i) regardless of what the final result (r) is. Providers can compute (r) off-chain before they reveal (x_i), which permits a censorship attack."*

Despite this acknowledged risk, no on-chain escape hatch was implemented to protect users when a provider withholds the reveal.

---

### Impact Explanation

A user who calls `requestV2` or `requestWithCallback` pays a fee in native token. That fee is immediately credited to the provider and to the Pyth protocol. If the provider never calls `revealWithCallback`:

- The user's fee is permanently locked — there is no function to reclaim it.
- The provider has already been paid and can withdraw their share at any time via `withdraw()`.
- The user's downstream contract (e.g., an NFT mint, a game) is permanently blocked waiting for a callback that will never arrive.
- The request occupies a slot in the 32-slot `requests` array or the `requestsOverflow` mapping indefinitely.

**Impact class:** Permanent loss of user funds (native token fees) and permanent denial of service to the requesting contract.

---

### Likelihood Explanation

The default provider (`_state.defaultProvider`) is the centralized Fortuna service operated by the Pyth team. Any of the following realistic scenarios triggers the vulnerability for any user who chose that provider:

1. **Provider infrastructure failure** — Fortuna goes offline permanently or for an extended period.
2. **Selective censorship** — Provider computes the random outcome off-chain and withholds the reveal for unfavorable (to the provider) results, as explicitly described in the protocol design doc.
3. **Regulatory action** — Provider is compelled by a government entity to stop serving specific addresses.
4. **Malicious third-party provider** — Any permissionlessly registered provider can accept requests, collect fees via `withdraw()`, and never reveal.

Because `requestV2()` with no arguments silently routes to the default provider, most integrators are exposed without realizing it.

---

### Recommendation

Implement a block-height timeout and refund path:

1. Store `blockNumber` in the request (already stored at line 262).
2. Add a `cancelRequest(address provider, uint64 sequenceNumber)` function that:
   - Requires `block.number > req.blockNumber + TIMEOUT_BLOCKS` (e.g., 256 blocks).
   - Requires `msg.sender == req.requester`.
   - Refunds `msg.value` (or the fee portion) to the requester.
   - Reverses the provider's `accruedFeesInWei` credit (if not yet withdrawn) or charges a slashing bond.
   - Calls `clearRequest`.
3. Alternatively, require providers to post a bond on registration that is slashed if they fail to reveal within the timeout window.

---

### Proof of Concept

```
1. Provider registers via register(...) with feeInWei = 1 ether.

2. User calls:
     entropy.requestV2{value: 1 ether}(provider, userRandomNumber, 0)
   → requestHelper executes:
       providerInfo.accruedFeesInWei += 1 ether   // line 237
       _state.accruedPythFeesInWei += 0            // line 238-239

3. Provider immediately calls:
     entropy.withdraw(1 ether)
   → Provider receives 1 ether. Fee is gone from contract.

4. Provider never calls revealWithCallback.
   → Any attempt by the user to call reveal() reverts with InvalidRevealCall (line 510).
   → Any attempt to call revealWithCallback with a fabricated providerContribution
     reverts with IncorrectRevelation (line 408).

5. No cancelRequest, no refund, no timeout function exists.
   → User's request is permanently stuck. Fee is permanently lost.
   → User's downstream contract (e.g., NFT mint awaiting callback) is permanently blocked.
