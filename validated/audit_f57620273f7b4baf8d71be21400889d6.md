Audit Report

## Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Produces Incorrect Withdrawal Capacity - (File: util/dao/src/lib.rs)

## Summary
`DaoCalculator::calculate_maximum_withdraw` computes an interest-adjusted withdrawal capacity as a u128 intermediate value and then silently truncates it to u64 via `as u64` at line 156. If the intermediate product exceeds `u64::MAX`, the truncated result can be far smaller than the deposited capacity. Downstream callers that assume `maximum_withdraw >= input_capacity` then fail with `safe_sub`, causing `Reject::Malformed` in the tx-pool and block validation failure in `dao_field_with_current_epoch`.

## Finding Description
In `util/dao/src/lib.rs` at lines 152–156:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
```

The `as u64` cast is an unchecked truncation. When `withdraw_counted_capacity > u64::MAX`, the high bits are silently discarded, yielding a value that can be arbitrarily smaller than `counted_capacity`. No `DaoError::Overflow` is returned; the wrong value propagates silently.

This is inconsistent with every other u128→u64 narrowing in the same file. At lines 244–245, `miner_issuance128` uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, and at lines 257–258, `ar_increase128` uses the same checked pattern. The `calculate_maximum_withdraw` function is the sole exception.

The silently truncated result feeds two downstream paths:

1. **tx-pool path** (`tx-pool/src/util.rs` lines 34–41): `check_tx_fee` calls `DaoCalculator::transaction_fee`, which calls `maximum_withdraw.safe_sub(outputs_capacity)`. If `maximum_withdraw` is truncated below the deposited amount, `safe_sub` fails and `check_tx_fee` returns `Reject::Malformed`, permanently banning the transaction from the pool.

2. **Block validation path** (`util/dao/src/lib.rs` lines 312–332): `withdrawed_interests` calls `transaction_maximum_withdraw` (which calls `calculate_maximum_withdraw`) and then `maximum_withdraws.safe_sub(input_capacities)`. A truncated `maximum_withdraws` causes this subtraction to fail. The error propagates through `dao_field_with_current_epoch` at lines 252–254, causing the entire block to be rejected.

The existing test `check_withdraw_calculation_overflows` (tests.rs lines 295–350) only catches the case where `safe_add(occupied_capacity)` overflows *after* the `as u64` cast. It does not cover the case where the cast itself silently truncates to a small value that then passes `safe_add` but breaks the downstream `safe_sub`.

## Impact Explanation
A DAO depositor whose cell triggers the truncation condition cannot withdraw their funds. The tx-pool permanently marks the withdrawal transaction as `Reject::Malformed`, and any block containing it fails `dao_field_with_current_epoch` validation. This constitutes a permanent freeze of deposited CKB with no recovery path through the normal protocol, fitting the "Vulnerabilities which could easily damage CKB economy" impact class.

## Likelihood Explanation
The truncation condition requires `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. The total CKB supply is approximately 3.36 × 10^18 shannons, while u64::MAX ≈ 1.84 × 10^19 shannons. For a maximum-supply single cell, the AR ratio must grow by approximately 5.48× from deposit time before truncation occurs — roughly 40+ years at current secondary issuance rates. The PoC's claim that "artificially constructed cells with capacity near u64::MAX" are reachable today is incorrect: a cell's capacity cannot exceed the total CKB supply, since transaction validation requires inputs to cover outputs. The vulnerability is therefore not immediately exploitable but is a confirmed latent defect that will become naturally triggerable as AR accumulates over decades.

## Recommendation
Replace the silent `as u64` cast with an explicit checked conversion, consistent with the pattern already used for `miner_issuance128` and `ar_increase128` in the same file:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity)
        .map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64)
        .safe_add(occupied_capacity)?;
```

This ensures that any overflow is surfaced as `DaoError::Overflow` rather than silently producing a wrong value.

## Proof of Concept
The existing test `check_withdraw_calculation_overflows` (tests.rs lines 295–350) already demonstrates the overflow boundary using `capacity = 18_446_744_073_709_550_000`. To specifically demonstrate the silent-truncation path (distinct from the `safe_add` overflow path the existing test exercises):

1. Set `deposit_ar` and `withdrawing_ar` such that `counted_capacity × withdrawing_ar / deposit_ar` exceeds u64::MAX but `(result mod 2^64) + occupied_capacity` does not (so `safe_add` succeeds and the wrong value propagates silently).
2. Call `calculate_maximum_withdraw` and assert the returned `Ok` value is less than the original deposited capacity — demonstrating the silent corruption.
3. Pass the result through `transaction_fee` with `outputs_capacity` equal to the original deposited amount and assert `safe_sub` fails with `DaoError::Overflow`.
