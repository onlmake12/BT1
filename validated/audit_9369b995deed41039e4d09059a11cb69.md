### Title
Missing Empty `priceIds` Array Validation in `Echo.requestPriceUpdatesWithCallback` Allows Fee Payment Without Receiving Price Data - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` accepts an empty `priceIds` array without reverting. A caller pays the base protocol fee plus provider base fee plus gas-cost component, but the stored request carries zero price-ID prefixes. When `executeCallback` is eventually invoked, the consumer's `_echoCallback` receives an empty `priceFeeds` array — the caller has permanently lost funds and received nothing useful.

---

### Finding Description

`requestPriceUpdatesWithCallback` validates that `priceIds.length` does not exceed `MAX_PRICE_IDS`, but performs **no lower-bound check**:

```solidity
if (priceIds.length > MAX_PRICE_IDS) {
    revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
}
``` [1](#0-0) 

With `priceIds.length == 0`, `getFee` returns `pythFeeInWei + providerBaseFee + gasEstimate` (no per-feed component), and the caller must send at least that amount:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [2](#0-1) 

The fee is immediately split: `pythFeeInWei` is credited to the protocol and the remainder to the provider:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
...
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [3](#0-2) 

The request is stored with an empty `priceIdPrefixes` array. When `executeCallback` is called, `parsePriceFeedUpdates` is invoked with empty `priceIds`, and the consumer's callback receives an empty `priceFeeds` slice — zero price data delivered. [4](#0-3) 

There is no cancellation or refund path visible in the contract for a stored request.

---

### Impact Explanation

A caller (any unprivileged `msg.sender`) pays `pythFeeInWei + providerBaseFee + gas-cost component` in ETH and receives no price data. The paid ETH is permanently distributed to the protocol and provider. If the provider never fulfills the empty request (rational behavior), the funds are effectively lost with no recourse. This is a direct, permanent loss of user funds triggered entirely by a missing one-line input check.

---

### Likelihood Explanation

**Medium.** The scenario arises from a simple user mistake (passing `[]` instead of a populated array) or from a malicious third party constructing a transaction on behalf of a naive consumer contract. No privileged access is required. The `requestPriceUpdatesWithCallback` function is the primary public entry point for Echo consumers, making accidental misuse realistic.

---

### Recommendation

Add a lower-bound guard immediately after the upper-bound check:

```solidity
if (priceIds.length == 0) revert EmptyPriceIds();
if (priceIds.length > MAX_PRICE_IDS) {
    revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
}
``` [1](#0-0) 

---

### Proof of Concept

1. Alice deploys a consumer contract that implements `IEchoConsumer`.
2. Alice calls `echo.requestPriceUpdatesWithCallback{value: fee}(provider, block.timestamp, new bytes32[](0), gasLimit)`.
3. The call succeeds. `fee` (= `pythFeeInWei + providerBaseFee + gasEstimate`) is deducted from Alice's balance and credited to the protocol/provider.
4. The stored request has `priceIdPrefixes.length == 0`.
5. A provider (or anyone) calls `executeCallback` with empty `updateData` and `priceIds`.
6. `parsePriceFeedUpdates` returns an empty array; Alice's `echoCallback` is invoked with `priceFeeds.length == 0`.
7. Alice has paid real ETH and received zero price data, with no refund mechanism available. [5](#0-4)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
