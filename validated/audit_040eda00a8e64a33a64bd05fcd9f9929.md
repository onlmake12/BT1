### Title
Unbounded `broadcast::Receiver` Accumulation via Repeated `subscribe()` Calls — (`rpc/src/module/subscription.rs`)

### Summary

`SubscriptionRpcImpl::subscribe()` creates a new `broadcast::Receiver` and a pinned `BoxStream` on every call with no per-connection or global subscription count limit. An unauthenticated caller on a TCP or WebSocket connection can call `subscribe()` in a tight loop, accumulating millions of live receiver/stream objects and exhausting process memory.

---

### Finding Description

Every invocation of `subscribe()` unconditionally calls `tx.subscribe()` on the shared `broadcast::Sender`, allocating a new `broadcast::Receiver<PublishMsg<String>>` and wrapping it in a heap-allocated `BoxStream` (a pinned async generator holding the receiver): [1](#0-0) 

There is no guard of any kind — no counter, no cap, no deduplication — before or after that call. The `subscribe()` function signature accepts only a `Topic` enum; the caller identity and existing subscription count are never consulted. [2](#0-1) 

The TCP server spawns an unbounded task per accepted connection and imposes no request-rate or subscription-count limit: [3](#0-2) 

The only batch-size guard in the codebase applies exclusively to HTTP batch requests, not to TCP/WS streams: [4](#0-3) 

---

### Impact Explanation

Each `subscribe()` call allocates:

| Object | Approximate size |
|---|---|
| `broadcast::Receiver` (pos counter + `Arc` ref) | ~32 bytes |
| `BoxStream` async generator state (holds the receiver + loop state) | ~200–400 bytes |
| Entry in the `jsonrpc_utils` per-connection subscription `HashMap` | ~100 bytes |

**Total per call: ~350–550 bytes.** With no limit, an attacker sending N=10,000,000 subscribe requests exhausts ~3.5–5.5 GB of heap — sufficient to OOM-kill a typical node.

One important correction to the question's framing: the `broadcast` ring buffer is **shared** across all receivers via `Arc`; it is **not** duplicated per receiver. The ring buffer stays fixed at `NOTIFY_CHANNEL_SIZE` slots regardless of receiver count. The actual unbounded growth comes from the accumulation of receiver structs and `BoxStream` generator objects, not from ring-buffer duplication. [5](#0-4) 

---

### Likelihood Explanation

- TCP and WebSocket RPC are opt-in but commonly enabled on public-facing nodes.
- No authentication is required to call `subscribe()`.
- A single connection can pipeline millions of JSON-RPC requests; at 1 Gbps the ~70-byte request payload for 10M calls takes under 10 seconds to transmit.
- The server processes each request synchronously within the connection task, so the attacker does not need multiple connections.

---

### Recommendation

1. **Enforce a per-connection subscription cap** (e.g., 128 or 256) inside `subscribe()`, returning a JSON-RPC error when exceeded.
2. **Optionally enforce a global receiver cap** by checking `sender.receiver_count()` before calling `tx.subscribe()`.
3. Consider **deduplicating** subscriptions per topic per connection (one receiver per topic per connection is sufficient).

---

### Proof of Concept

```bash
# Connect via TCP, flood subscribe calls, observe RSS growth
python3 - <<'EOF'
import socket, json, time

s = socket.create_connection(("127.0.0.1", 18114))
req = json.dumps({"id": 1, "jsonrpc": "2.0",
                  "method": "subscribe",
                  "params": ["new_tip_block"]}) + "\n"
req_bytes = req.encode()

for i in range(1_000_000):
    s.sendall(req_bytes)

time.sleep(60)   # keep receivers alive; observe node RSS
EOF
```

Monitor node RSS with `ps -o rss= -p <ckb_pid>` before and after; growth will be proportional to the number of calls with no plateau.

    fn subscribe(&self, topic: Topic) -> Result<Self::S> {
        let tx = match topic {
            Topic::NewTipHeader => self.new_tip_header_sender.clone(),
            Topic::NewTipBlock => self.new_tip_block_sender.clone(),
            Topic::NewTransaction => self.new_transaction_sender.clone(),
            Topic::ProposedTransaction => self.proposed_transaction_sender.clone(),
            Topic::RejectedTransaction => self.new_reject_transaction_sender.clone(),
            Topic::Log => self.log_sender.clone(),
        };
        let mut rx = tx.subscribe();
        Ok(Box::pin(async_stream::stream! {
                loop {
                    match rx.recv().await {
                        Ok(msg) => {
                            yield msg;
                        }
                        Err(RecvError::Lagged(cnt)) => {
                            error!("subscription lagged error: {:?}", cnt);
                        }
                        Err(RecvError::Closed) => {
                            break;
                        }
                    }
                }
        }))
    }
