### Title
Wrong Slot Ordering Condition in TWAP Validation Allows Division-by-Zero Panic - (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary

`validateTwapPriceInfo` in `Pyth.sol` uses a strict `>` comparison to guard the slot ordering, but `calculateTwap` divides by `slotDiff`. When start and end slots are equal, `slotDiff = 0` and the division panics. The correct guard should use `>=` (i.e., reject equal slots), mirroring the Solana receiver which correctly enforces `end_slot > start_slot`.

### Finding Description

In `validateTwapPriceInfo`, the slot ordering check is:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This only rejects the case where `startSlot > endSlot`. It silently passes when `startSlot == endSlot`. [1](#0-0) 

Immediately after validation, `calculateTwap` computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot;
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // panics if slotDiff == 0
uint128 twapConf  = confDiff / uint128(slotDiff);           // panics if slotDiff == 0
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panics if slotDiff == 0
``` [2](#0-1) 

The Solana receiver correctly enforces strict inequality:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The EVM validation should mirror this: revert when `startSlot >= endSlot`. The current `>` check is the wrong condition — it checks the wrong boundary, exactly as M-17 checked the wrong variable.

### Impact Explanation

Any caller of `parseTwapPriceFeedUpdates` who submits two Wormhole-signed TWAP messages sharing the same `publishSlot` will trigger a Solidity division-by-zero panic (EVM panic code `0x12`) instead of a clean `InvalidTwapUpdateDataSet` revert. The entire transaction reverts, the fee is refunded, but the caller receives an opaque panic rather than a meaningful error. Downstream integrators that catch specific error selectors will fail to handle this case, and any protocol that wraps `parseTwapPriceFeedUpdates` in a try/catch keyed on `InvalidTwapUpdateDataSet` will not catch the panic, potentially breaking their error-handling logic. [4](#0-3) 

### Likelihood Explanation

`parseTwapPriceFeedUpdates` is a public payable function callable by any unprivileged user. A caller only needs to supply two valid Wormhole-signed TWAP accumulator messages that happen to carry the same `publishSlot` value. Pythnet can legitimately produce two TWAP messages at the same slot (e.g., a start and end snapshot taken within the same slot). The Wormhole guardians sign whatever Pythnet produces, so no privileged access or key compromise is required. [5](#0-4) 

### Recommendation

Change the slot ordering guard in `validateTwapPriceInfo` from strict `>` to `>=`, so that equal slots are rejected with a proper error before reaching the division:

```solidity
// Before (wrong):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}

// After (correct):
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This aligns the EVM implementation with the Solana receiver's `end_slot > start_slot` requirement and eliminates the division-by-zero path. [6](#0-5) 

### Proof of Concept

1. Obtain two valid Wormhole-signed TWAP accumulator messages for the same price feed where both messages carry `publishSlot = S` (same slot).
2. Call `parseTwapPriceFeedUpdates{value: fee}([startUpdate, endUpdate], [priceId])`.
3. `validateTwapPriceInfo` passes because `S > S` is `false`.
4. `calculateTwap` computes `slotDiff = S - S = 0`.
5. `priceDiff / int128(uint128(0))` triggers EVM panic `0x12` (division by zero), reverting the transaction with an opaque panic instead of `InvalidTwapUpdateDataSet`. [7](#0-6)

        if (
            twapPriceInfoStart.prevPublishTime >= twapPriceInfoStart.publishTime
