### Title
Non-Decreasing Timestamp Validation Enables Keeper Front-Running and Subscription Balance Drain - (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `Scheduler` contract enforces a strict non-decreasing `publishTime` check on every `updatePriceFeeds` call. Because `updatePriceFeeds` is explicitly permissionless, any actor can front-run legitimate keepers by submitting update data with the newest available Pyth `publishTime`, causing all competing keeper transactions with a slightly older `publishTime` to revert with `TimestampOlderThanLastUpdate`. This allows an attacker to monopolize keeper fees, drain subscription balances, and ultimately cause a DoS for price data consumers when the subscription runs out of funds.

### Finding Description
`Scheduler._validateShouldUpdatePrices` computes `updateTimestamp` as the maximum `publishTime` across all price feeds in the submitted update data, then enforces:

```solidity
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(
        updateTimestamp,
        status.priceLastUpdatedAt
    );
}
``` [1](#0-0) 

On every successful update, `status.priceLastUpdatedAt` is set to the winning `updateTimestamp`:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

`updatePriceFeeds` has **no access control** — it is intentionally permissionless so any keeper can call it:

> "Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`." [3](#0-2) 

The function signature confirms no `onlyKeeper` or similar modifier: [4](#0-3) 

Pyth price update data carries a `publishTime` that advances with each Pythnet slot. Multiple keepers fetching data at slightly different moments will obtain update blobs with different `publishTime` values. If Keeper A fetches at slot N and Keeper B fetches at slot N+1, and Keeper B's transaction lands first, Keeper A's transaction reverts — even though Keeper A's data was valid and fresh.

An attacker can exploit this deterministically:
1. Monitor the mempool or the Pyth price service for the absolute latest available `publishTime`.
2. Whenever update conditions are met (heartbeat elapsed or deviation threshold crossed), immediately submit `updatePriceFeeds` with the newest slot's data.
3. All competing keeper transactions with a slightly older `publishTime` revert with `TimestampOlderThanLastUpdate`.
4. The attacker collects the full keeper fee from the subscription balance on every update. [5](#0-4) 

### Impact Explanation
- **Legitimate keepers lose gas** on every reverted transaction with no compensation.
- **Attacker monopolizes keeper fees**, draining the subscription's `balanceInWei` faster than the manager may anticipate.
- Once `balanceInWei` falls below the minimum balance threshold, the subscription becomes underfunded. The next `updatePriceFeeds` call reverts with `InsufficientBalance`, halting all price updates for that subscription. [6](#0-5) 

- **Price data consumers** (`getPricesNoOlderThan`, `getEmaPricesNoOlderThan`) then receive `StalePrice` reverts, causing a DoS for any protocol relying on the subscription. [7](#0-6) 

### Likelihood Explanation
- `updatePriceFeeds` is permissionless by design; no special role or capital is required to call it.
- The Pyth price service (Hermes) is publicly accessible; any actor can fetch the latest price update data at any time.
- The attack is profitable: the attacker earns keeper fees exceeding their gas cost on every successful front-run.
- The attack is repeatable on every heartbeat interval or deviation trigger, making it a sustained griefing vector.

### Recommendation
1. **Relax the strict non-decreasing check**: Instead of reverting when `updateTimestamp <= priceLastUpdatedAt`, only reject updates that are older than a configurable tolerance window (e.g., `updateTimestamp < priceLastUpdatedAt - TOLERANCE`). This mirrors the recommendation in the external report.
2. **Alternatively, use a keeper whitelist or commit-reveal scheme** to prevent front-running of keeper transactions.
3. **At minimum**, document that the keeper fee model assumes competitive but honest keepers, and that subscription managers should monitor their balance actively.

### Proof of Concept
1. A subscription is created with `heartbeatSeconds = 60` and sufficient balance.
2. At `T=60` (heartbeat elapsed), Keeper A fetches Pyth update data with `publishTime = 59`.
3. Attacker monitors the Pyth price service and fetches data with `publishTime = 60` (one slot newer).
4. Both submit `updatePriceFeeds` transactions. Attacker's transaction lands first (or is front-run via MEV).
5. `status.priceLastUpdatedAt` is set to `60`.
6. Keeper A's transaction executes `_validateShouldUpdatePrices`: `59 <= 60` → reverts with `TimestampOlderThanLastUpdate(59, 60)`. [1](#0-0) 

7. Attacker repeats this on every heartbeat interval, collecting all keeper fees and draining the subscription balance until it falls below the minimum, at which point `updatePriceFeeds` reverts for everyone, and price data becomes permanently stale for readers. [8](#0-7)

    function _validateShouldUpdatePrices(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams storage params,
        SchedulerStructs.SubscriptionStatus storage status,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal view returns (uint256) {
        // Use the most recent timestamp, as some asset markets may be closed.
        // Closed markets will have a publishTime from their last trading period.
        // Since we verify all updates share the same Pythnet slot, we still ensure
        // that all price feeds are synchronized from the same update cycle.
        uint256 updateTimestamp = 0;
        for (uint8 i = 0; i < priceFeeds.length; i++) {
            if (priceFeeds[i].price.publishTime > updateTimestamp) {
                updateTimestamp = priceFeeds[i].price.publishTime;
            }
        }

        // Calculate the minimum acceptable timestamp (clamped at 0)
        // The maximum acceptable timestamp is enforced by the parsePriceFeedUpdatesWithSlots call
        uint256 minAllowedTimestamp = (block.timestamp >
            PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            : 0;

        // Validate that the update timestamp is not too old
        if (updateTimestamp < minAllowedTimestamp) {
            revert SchedulerErrors.TimestampTooOld(
                updateTimestamp,
                block.timestamp
            );
        }

        // Reject updates if they're older than the latest stored ones
        if (
            status.priceLastUpdatedAt > 0 &&
            updateTimestamp <= status.priceLastUpdatedAt
        ) {
            revert SchedulerErrors.TimestampOlderThanLastUpdate(
                updateTimestamp,
                status.priceLastUpdatedAt
            );
        }
```

- Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`. The main goal of making this component a permissionless network rather a set of permissioned nodes is to enhance reliability for the feeds -- if one provider fails, others should be available to service the subscriptions. We can improve this reliability by sourcing independent providers, and by making it profitable to push updates, paid out by the users of the feeds.

- Keepers are paid directly by the subscription's funds held in this contract for each successful update they perform. The payment covers gas costs plus a premium, and payment is sent directly to `msg.sender` (the keeper) at the end of `updatePriceFeeds`. The first transaction included in a block that passes checks will succeed and receive the payment. Subsequent attempts for the same update interval will revert since we verify the update criteria on-chain. By only allowing updates when they are needed, we keep costs predictable for the users.
