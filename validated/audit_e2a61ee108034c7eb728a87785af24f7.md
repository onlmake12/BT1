### Title
Immutable `pythFeeInWei` in Echo Contract Causes Provider Underpayment and Potential Fund Lock - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo` contract stores a `pythFeeInWei` value set at initialization with no admin or governance function to update it. Unlike `EntropyGovernance` which exposes `setPythFee()`, Echo has no equivalent. When the Pyth oracle raises its fee via governance, the stale `pythFeeInWei` causes providers (relayers) executing callbacks to be underpaid, and in the worst case causes `executeCallback` to revert with an arithmetic underflow, permanently locking user funds.

---

### Finding Description

At request time, `requestPriceUpdatesWithCallback` collects `_state.pythFeeInWei` from the user and stores the remainder as the provider's fee: [1](#0-0) 

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

At callback time, `executeCallback` queries the **live** Pyth fee and deducts it from the provider's payout: [3](#0-2) 

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
...
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

The `pythFeeInWei` stored in `EchoState` is set once at initialization: [4](#0-3) 

There is **no function to update it**. The contract itself notes governance is not yet implemented: [5](#0-4) 

```solidity
// TODO: move out governance functions into a separate PulseGovernance contract
```

No `EchoGovernance.sol` exists. Compare with `EntropyGovernance`, which has a proper `setPythFee()`: [6](#0-5) 

The `pythFeeInWei` field in `EchoState` is therefore permanently frozen: [7](#0-6) 

---

### Impact Explanation

When the Pyth oracle raises its `singleUpdateFeeInWei` via governance (a routine operation), two consequences follow:

1. **Provider underpayment**: The live `pythFee = pyth.getUpdateFee(updateData)` exceeds the `pythFeeInWei` collected from the user. The excess is silently deducted from the provider's payout, making callback execution unprofitable and discouraging providers from fulfilling requests.

2. **Permanent fund lock**: If `actual_pythFee > req.fee + msg.value`, the subtraction `(req.fee + msg.value) - pythFee` underflows and reverts. Since there is no cancel/refund mechanism for stuck requests, the user's funds are permanently locked in the contract. [8](#0-7) 

---

### Likelihood Explanation

Pyth governance regularly adjusts `singleUpdateFeeInWei` across all deployed chains (evidenced by the `SetFee` governance action and its tests). Any such increase after Echo deployment triggers this condition for all in-flight requests. No attacker action is required — normal protocol operation is sufficient. [9](#0-8) 

---

### Recommendation

Add a `setPythFee` governance function to Echo (mirroring `EntropyGovernance.setPythFee`), restricted to the admin:

```solidity
function setPythFee(uint96 newPythFeeInWei) external {
    require(msg.sender == _state.admin, "Only admin");
    _state.pythFeeInWei = newPythFeeInWei;
}
```

Additionally, consider adding a request cancellation/refund path so that if `executeCallback` cannot be completed due to fee mismatch, users can recover their funds.

---

### Proof of Concept

1. Echo is deployed with `pythFeeInWei = 1 wei`.
2. User calls `requestPriceUpdatesWithCallback` paying `1 wei` (Pyth fee) + provider fee. `req.fee = provider_fee`.
3. Pyth governance raises `singleUpdateFeeInWei` to `100 wei` on the underlying Pyth contract.
4. Provider calls `executeCallback`. `pyth.getUpdateFee(updateData)` returns `100 wei`.
5. The expression `(req.fee + msg.value) - 100` underflows if `req.fee + msg.value < 100`.
6. `executeCallback` reverts. The request remains active but unfulfillable. User funds are permanently locked. [10](#0-9)
