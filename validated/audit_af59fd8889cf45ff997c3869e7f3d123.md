Audit Report

## Title
Peer Store Permanently Blocked via Crafted Identify `listen_addrs` Flooding — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
An unprivileged remote peer can fill the `PeerStore` to `ADDR_COUNT_LIMIT=16384` by opening ~1639 sequential TCP connections and reporting 10 unique globally-routable listen addresses per session, spread across ≥4096 distinct `/16` subnets (≤4 per subnet). This defeats both eviction strategies in `check_purge`, causing it to permanently return `PeerStoreError::EvictionFailed`. All subsequent `add_addr` calls from honest peers are silently dropped, blocking peer discovery indefinitely and progressively isolating the victim node.

## Finding Description

**Entry point — `process_listens` / `add_remote_listen_addrs`:**

`process_listens` accepts up to `MAX_ADDRS=10` listen addresses per identify message from any remote peer. It only checks that the count does not exceed 10 and that addresses are globally reachable IPs. It does not rate-limit, deduplicate across sessions, or restrict which IPs can be reported (`network/src/protocols/identify/mod.rs`, L123–150). These addresses are forwarded to `add_remote_listen_addrs`, which calls `peer_store.add_addr` for each one and only logs errors (`network/src/protocols/identify/mod.rs`, L488–494).

**`add_addr` inserts with zero connection history:**

`add_addr` calls `check_purge()` then inserts via `AddrInfo::new(addr, 0, score, flags.bits())`, setting `last_connected_at_ms=0` and `attempts_count=0` (`network/src/peer_store/peer_store_impl.rs`, L71–80; `network/src/peer_store/types.rs`, L63–76).

**`check_purge` Strategy 1 — evict non-connectable addresses:**

`is_connectable` returns `false` only when `attempts_count >= ADDR_MAX_RETRIES(3)` (never connected) or `attempts_count >= ADDR_MAX_FAILURES(10)` (stale). Fresh attacker addresses have `attempts_count=0`, so all return `true` and Strategy 1 evicts nothing (`network/src/peer_store/types.rs`, L88–105).

**`check_purge` Strategy 2 — evict from large network groups:**

Strategy 2 groups addresses by `/16` subnet (first two octets of IPv4, `network/src/network_group.rs`, L26–29), then only evicts 2 random peers from groups with `> 4` members. If the attacker spreads 16384 addresses across ≥4096 distinct `/16` subnets with exactly 4 per subnet, no group ever has `> 4` members. `candidate_peers` remains empty and `Err(PeerStoreError::EvictionFailed)` is returned (`network/src/peer_store/peer_store_impl.rs`, L357–401).

**Error silently swallowed:**

The caller in `add_remote_listen_addrs` only logs the error — it does not disconnect the session or take corrective action (`network/src/protocols/identify/mod.rs`, L490–492). The `ADDR_COUNT_LIMIT` is 16384 (`network/src/peer_store/mod.rs`, L26).

## Impact Explanation

Once the peer store is full and eviction is permanently blocked, all `add_addr` calls return `Err` and are silently dropped. The victim node can no longer learn about new peers via the identify protocol. If existing connections drop due to churn, restarts, or bans, the node cannot replace them and becomes progressively isolated. A fully isolated node cannot participate in block propagation or transaction relay. This matches the **High** impact category: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* — targeted isolation of multiple nodes degrades overall network connectivity and propagation.

## Likelihood Explanation

The attack requires approximately ⌈16384/10⌉ = 1639 sequential TCP connections, each sending 10 unique globally-routable listen addresses from distinct `/16` subnets. The attacker does not need to own those IPs — they are only reported as listen addresses. A single source IP suffices since the peer store stores reported listen addresses, not the connection source. No privileged role is required. The only friction is connection rate and the need for globally-routable IP addresses in the payload, both trivially satisfied by any attacker with internet access. The attack is repeatable and persistent across node restarts if the peer store is persisted to disk.

## Recommendation

1. **Fix the eviction fallback** — when all addresses are connectable and no group has `> 4` members, fall back to evicting the oldest or lowest-scored addresses rather than returning `EvictionFailed`. Prefer evicting addresses with `last_connected_at_ms == 0` (never successfully connected) before addresses with a connection history.
2. **Rate-limit address contributions per source IP or per session** — cap how many addresses a single peer can contribute to the peer store over a time window.
3. **Bound the number of addresses accepted per identify session** at the peer store level, not just at the protocol level.
4. **Consider disconnecting or penalizing peers** whose `add_addr` calls consistently fail due to a full store, rather than silently logging.

## Proof of Concept

```rust
// Fill peer store with 16384 addresses across 4096 distinct /16 subnets (exactly 4 per subnet)
// Each address: last_connected_at_ms=0, attempts_count=0 → is_connectable=true
let mut peer_store = PeerStore::default();
let mut count = 0u32;
'outer: for a in 1u8..=255 {
    for b in 0u8..=255 {
        for port in 1u16..=4 {
            if count >= 16384 { break 'outer; }
            let addr: Multiaddr = format!(
                "/ip4/{}.{}.1.1/tcp/{}/p2p/{}", a, b, port, PeerId::random().to_base58()
            ).parse().unwrap();
            peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
            count += 1;
        }
    }
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// Now any new add_addr returns EvictionFailed
let honest_addr: Multiaddr = format!(
    "/ip4/8.8.8.1/tcp/9999/p2p/{}", PeerId::random().to_base58()
).parse().unwrap();
let result = peer_store.add_addr(honest_addr, Flags::COMPATIBILITY);
assert!(result.is_err()); // PeerStoreError::EvictionFailed
```

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute

            return true;
        }

        }
        true
    }
