### Title
Deviation Trigger Compares Raw Price Mantissas Without Exponent Normalization, Causing Incorrect Update Conditions - (File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol)

### Summary
The Pulse `Scheduler` contract computes price deviation by comparing raw `int64` mantissa values from two consecutive price updates without normalizing for their respective `expo` fields. If the exponent of a Pyth price feed changes between the stored previous update and the incoming update — a legitimate occurrence in the Pyth protocol — the deviation calculation produces a wildly incorrect result, either triggering spurious updates or suppressing real ones.

### Finding Description
In `_validateUpdateConditions`, the deviation check reads the raw `price` mantissa from both the incoming feed and the stored previous feed, then computes basis-point deviation directly on those integers:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;
int64 previousPrice = previousFeed.price.price;
// ...
uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
```

The actual price represented by a Pyth `Price` struct is `mantissa × 10^expo`. The deviation between two prices is only meaningful when both mantissas are expressed at the same exponent. The code never reads `priceFeeds[i].price.expo` or `previousFeed.price.expo` during this calculation, so it silently assumes the exponent is identical across both updates. [1](#0-0) 

The `PythStructs.Price` struct carries an independent `expo` field per update: [2](#0-1) 

Pyth has historically changed the exponent for feeds (e.g., from `-8` to `-7`). When such a change occurs, the first keeper-submitted update after the change will have a different `expo` than the value stored in `_state.priceUpdates`. The mantissa values are then on different scales and their arithmetic difference is meaningless as a percentage.

### Impact Explanation
**False positive (spurious trigger):** Suppose the previous stored price is `mantissa=100_000_000, expo=-8` (= $1.00) and the new price is `mantissa=10_000_000, expo=-7` (= $1.00, same real price). The deviation calculation sees `|10_000_000 − 100_000_000| / 100_000_000 = 90%`, far exceeding any reasonable `deviationThresholdBps`. The keeper is paid for an unnecessary update, draining the subscription balance.

**False negative (missed trigger):** Suppose the previous stored price is `mantissa=100_000_000, expo=-8` (= $1.00) and the new price is `mantissa=100_000_000, expo=-7` (= $10.00, a 10× real move). The deviation calculation sees `|100_000_000 − 100_000_000| / 100_000_000 = 0%`. No update is triggered even though the real price moved 900%. Consumers reading from the Scheduler receive a stale, incorrect price.

The false-negative case is the more severe impact: protocols relying on the Scheduler for timely price updates (e.g., lending protocols checking collateral value) would operate on a price that is off by an order of magnitude.

### Likelihood Explanation
Pyth has changed feed exponents in the past. Any keeper — a fully permissionless, unprivileged actor — can submit a valid signed Pyth price update that carries the new exponent. The Pyth core contract (`parsePriceFeedUpdatesWithSlots`) verifies the cryptographic integrity of the update but does not enforce exponent stability. The Scheduler stores the update and the bug manifests on the very next deviation check against the old stored value. [3](#0-2) 

### Recommendation
Normalize both prices to the same exponent before computing the deviation. The simplest approach is to use `PythUtils.convertToUint` (already available in the SDK) to convert both the current and previous price to a common fixed-point representation (e.g., 18 decimals) before taking the ratio:

```solidity
uint256 currentNorm  = PythUtils.convertToUint(
    priceFeeds[i].price.price, priceFeeds[i].price.expo, 18);
uint256 previousNorm = PythUtils.convertToUint(
    previousFeed.price.price, previousFeed.price.expo, 18);
uint256 numerator    = currentNorm > previousNorm
    ? currentNorm - previousNorm
    : previousNorm - currentNorm;
uint256 deviationBps = Math.mulDiv(numerator, 10_000, previousNorm);
``` [4](#0-3) 

### Proof of Concept
1. A subscription is created with `updateOnDeviation = true` and `deviationThresholdBps = 100` (1%).
2. A keeper submits the first update: feed `F` with `price = 100_000_000, expo = -8` (= $1.00). This is stored in `_state.priceUpdates`.
3. Pyth legitimately changes feed `F`'s exponent to `-7`. The new price is `price = 10_000_000, expo = -7` (= $1.00, unchanged).
4. A keeper submits this new update. `_validateUpdateConditions` computes:
   - `numerator = |10_000_000 − 100_000_000| = 90_000_000`
   - `denominator = 100_000_000`
   - `deviationBps = 9_000` (90%) ≥ 100 → update is accepted and keeper is paid.
5. The subscription balance is drained by spurious keeper payments even though the real price never moved.

Alternatively, in step 3, if the new price is `price = 100_000_000, expo = -7` (= $10.00, a 10× real move):
- `deviationBps = 0` → update is **rejected**, leaving consumers with a $1.00 price when the real price is $10.00. [5](#0-4)

    function convertToUint(
        int64 price,
        int32 expo,
        uint8 targetDecimals
    ) public pure returns (uint256) {
        if (price < 0) {
            revert PythErrors.NegativeInputPrice();
        }
        if (expo < -255) {
            revert PythErrors.InvalidInputExpo();
        }

        // If targetDecimals is 6, we want to multiply the final price by 10 ** -6
        // So the delta exponent is targetDecimals + currentExpo
        int32 deltaExponent = int32(uint32(targetDecimals)) + expo;

        // Bounds check: prevent overflow/underflow with base 10 exponentiation
        // Calculation: 10 ** n <= (2 ** 256 - 63) - 1
        //              n <= log10((2 ** 193) - 1)
        //              n <= 58.2
        if (deltaExponent > 58 || deltaExponent < -58)
            revert PythErrors.ExponentOverflow();

        // We can safely cast the price to uint256 because the above condition will revert if the price is negative
        uint256 unsignedPrice = uint256(uint64(price));

        if (deltaExponent > 0) {
            (bool success, uint256 result) = Math.tryMul(
                unsignedPrice,
                10 ** uint32(deltaExponent)
            );
            // This condition is unreachable since we validated deltaExponent bounds above.
            // But keeping it here for safety.
            if (!success) {
                revert PythErrors.CombinedPriceOverflow();
            }
            return result;
        } else {
            (bool success, uint256 result) = Math.tryDiv(
                unsignedPrice,
                10 ** uint(Math.abs(deltaExponent))
            );
            // This condition is unreachable since we validated deltaExponent bounds above.
            // But keeping it here for safety.
            if (!success) {
                revert PythErrors.CombinedPriceOverflow();
            }
            return result;
        }
    }
