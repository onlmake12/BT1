Audit Report

## Title
SQL Bind/Placeholder Order Mismatch in `get_cells` and `get_cells_capacity` Causes RPC Crash or Wrong Results When Dead Cells and Filter Params Coexist — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

## Summary
When `index_tx_pool=true` and the tx-pool contains at least one pending transaction (producing non-empty `dead_cells`), and the RPC caller also supplies `filter.script` or `filter.output_data`, the SQL placeholder numbering assigns dead-cell tx_hash positions before filter param positions, but the `.bind()` call sequence binds filter params first and dead-cell tx_hashes second. This mismatch causes wrong values to be bound to wrong `$N` positions, resulting in either a DB type error (RPC crash on PostgreSQL) or silently incorrect filtering (SQLite). The same bug exists identically in `get_cells_capacity`.

## Finding Description
**SQL-building phase (`get_cells.rs`):**

1. `param_index` starts at 1. [1](#0-0) 
2. `build_query_script_sql` consumes $1–$4 (prefix mode). After: `param_index = 5`.
3. The dead-cells loop (lines 124–135) assigns $5, $6, …, $5+N-1 — one placeholder per dead-cell tx_hash. After: `param_index = 5+N`. [2](#0-1) 
4. `build_cell_filter` (lines 145–150) is called next and assigns $5+N, $5+N+1, … to filter params. [3](#0-2) 

**Bind phase (`get_cells.rs`):**
- Lines 167–188: main script params bound to $1–$4. ✓ [4](#0-3) 
- Lines 189–220: **filter params** bound next — these land at $5, $6, … but the SQL expects dead-cell tx_hashes at those positions. ✗ [5](#0-4) 
- Lines 221–225: **dead-cell tx_hashes** bound last — these land at $5+filter_count, … but the SQL expects filter params there. ✗ [6](#0-5) 

**Concrete example** (N=1 dead cell, `filter.script` set, prefix mode):

| Position | SQL expects | Bind provides |
|---|---|---|
| $5 | `dead_cell[0].tx_hash` (32-byte blob) | `filter.script.code_hash` (32-byte blob) |
| $6 | `filter.script.code_hash` | `filter.script.hash_type` (i16) |
| $7 | `filter.script.hash_type` | `filter.script.args` (bytes) |
| $8 | `filter.script.args` (lower) | `upper_boundary(filter.script.args)` |
| $9 | `upper_boundary(filter.script.args)` | `dead_cell[0].tx_hash` |

The identical structural mismatch is confirmed in `get_cells_capacity.rs`: dead-cells loop at lines 88–102 assigns placeholders before `build_cell_filter` at lines 104–109, but the bind phase at lines 142–173 binds filter params before dead-cell tx_hashes at lines 174–178. [7](#0-6) 

No existing guard checks or reorders the bind sequence relative to the SQL placeholder assignment order. [8](#0-7) 

## Impact Explanation
On **PostgreSQL**: binding an `i16` value (`hash_type`) to a `BYTEA` parameter position ($6, which SQL expects as `filter.script.code_hash`) causes a DB type error. This error propagates as `Error::DB(...)` returned to the RPC caller — a local RPC API crash. This matches the allowed impact: **Note (0–500 points): Any local RPC API crash**.

On **SQLite**: type coercion is more permissive, so the query executes but with wrong values in the dead-cell exclusion and filter conditions, producing silently incorrect results (e.g., spent cells returned as live, or filter matching incorrectly). [9](#0-8) 

## Likelihood Explanation
All preconditions are reachable in normal production operation with no special privileges:
- `index_tx_pool=true` is a documented, supported configuration.
- Any pending transaction in the mempool creates at least one dead cell.
- Any unprivileged caller of `ckb_get_cells` or `ckb_get_cells_capacity` can supply `filter.script`.

The bug is deterministically triggered whenever these three conditions coincide, which is routine in a live node with an active mempool. [10](#0-9) 

## Recommendation
In the bind phase of both `get_cells` and `get_cells_capacity`, bind dead-cell tx_hashes **before** filter params, matching the SQL placeholder assignment order:

```rust
// Bind dead-cell tx_hashes FIRST (matching SQL placeholder order)
if !dead_cells.is_empty() {
    for (tx_hash, _) in &dead_cells {
        query = query.bind(tx_hash);
    }
}
// Then bind filter params
if let Some(filter) = search_key.filter.as_ref() {
    // ... existing filter bind logic ...
}
```

Alternatively, restructure `build_cell_filter` to be called before the dead-cells loop in the SQL-building phase so both phases are consistent. The fix must be applied identically to both `get_cells.rs` and `get_cells_capacity.rs`. [11](#0-10) 

## Proof of Concept
**Minimal manual reproduction steps:**
1. Start a CKB node with `index_tx_pool = true` in the rich-indexer config.
2. Submit any transaction to the mempool (creates at least one dead cell).
3. Call `ckb_get_cells` with a `search_key` that includes a non-empty `filter.script` field.
4. **On PostgreSQL**: the RPC returns a DB error due to type mismatch at bind position $6 (i16 bound where BYTEA expected).
5. **On SQLite**: the RPC returns results, but dead cells are not correctly excluded and/or the filter script condition matches incorrectly.

**Unit test plan**: construct a mock `AsyncRichIndexerHandle` with a non-empty pool (one dead cell) and a `search_key` with `filter.script` set; call `get_cells`; assert the generated SQL's `$5` placeholder receives the dead-cell tx_hash bytes, not the filter script code_hash bytes. [6](#0-5)
