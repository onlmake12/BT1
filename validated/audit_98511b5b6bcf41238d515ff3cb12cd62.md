Audit Report

## Title
Single-Element HashMap Sampling in `need_clean` Allows Permanent Orphan Group Retention — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::need_clean` samples exactly one arbitrarily-ordered element from a `HashMap` to decide whether to evict an entire orphan group. An attacker can insert two syntactically valid blocks sharing the same unknown parent — one with an expired epoch and one with a far-future epoch — causing the cleanup logic to permanently retain the group. Because `BlockVerifier` does not check PoW or epoch validity, this requires no mining work and is freely repeatable, leading to unbounded growth of the orphan pool and associated maps.

## Finding Description

`InnerPool` stores orphan blocks grouped by parent hash:

```rust
blocks: HashMap<ParentHash, HashMap<packed::Byte32, LonelyBlockHash>>
```

`need_clean` decides whether to evict a group by inspecting exactly one element via `map.iter().next()`: [1](#0-0) 

`HashMap::iter()` has non-deterministic order. If the sampled element is not expired, the entire group — including genuinely expired siblings — is retained. `clean_expired_blocks` calls `need_clean` for every leader and removes the group only when it returns `true`: [2](#0-1) 

The cleanup timer fires every 60 seconds: [3](#0-2) 

**Critical enabler — no PoW required.** `asynchronous_process_block` calls `non_contextual_verify` before routing to the orphan pool: [4](#0-3) 

`non_contextual_verify` invokes `BlockVerifier::verify`, which runs only five checks: [5](#0-4) 

None of these check PoW or epoch validity. After passing verification, `process_lonely_block` inserts the block into the orphan pool when the parent is unknown: [6](#0-5) 

The pool is initialized with `ORPHAN_BLOCK_SIZE` as a capacity hint only — `HashMap::with_capacity` does not enforce a maximum, so there is no hard upper bound on pool size: [7](#0-6) 

**Exploit flow:**
1. Attacker crafts Block A: `parent = P` (unknown), `epoch = 1`, valid cellbase, correct merkle root.
2. Attacker crafts Block B: `parent = P`, `epoch = 9999`, valid cellbase, correct merkle root.
3. Both pass `non_contextual_verify` and are inserted into `InnerPool::blocks[P]`.
4. Chain advances past epoch 7. `clean_expired_blocks(8)` fires.
5. `need_clean(P, 8)` calls `map.iter().next()`. If it returns Block B: `9999 + 6 < 8` → `false` → group not cleaned.
6. Block A (epoch 1, genuinely expired) remains permanently in `InnerPool::blocks`, `InnerPool::parents`, `block_status_map`, and `header_view`.
7. Repeating with fresh parent hashes `P1, P2, …` grows all four structures without bound.

`EXPIRED_EPOCH` is defined as `6`: [8](#0-7) 

## Impact Explanation

Unbounded growth of `InnerPool::blocks`, `InnerPool::parents`, `block_status_map`, and `header_view` exhausts heap memory, causing an OOM crash. This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**. Applied simultaneously across many nodes, it also maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

Any connected peer can relay blocks. The only gate before orphan pool insertion is `BlockVerifier::verify`, which is confirmed to check only proposals limit, block bytes, cellbase structure, duplicates, and merkle root — none of which involve PoW or epoch values. Crafting two syntactically valid blocks with the same unknown parent and different epoch numbers requires only correct serialization and merkle root computation, both trivially cheap. The attack is indefinitely repeatable with fresh parent hashes.

## Recommendation

Replace the single-element sample with a check over all elements in the group. The conservative fix (only clean if every block in the group is expired):

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            map.values().all(|lonely_block| {
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
```

Alternatively, track the minimum epoch number per group at insertion time to avoid iterating the inner map on every cleanup cycle.

## Proof of Concept

Extend the existing unit test in `chain/src/tests/orphan_block_pool.rs`:

```rust
#[test]
fn test_need_clean_skips_expired_block_when_sibling_is_not_expired() {
    use crate::utils::orphan_block_pool::{EXPIRED_EPOCH, OrphanBlockPool};
    use crate::{LonelyBlock, LonelyBlockHash};
    use ckb_types::core::{BlockBuilder, EpochNumberWithFraction};
    use ckb_systemtime::unix_time_as_millis;
    use std::sync::Arc;

    let pool = OrphanBlockPool::with_capacity(10);
    let parent_hash = ckb_types::packed::Byte32::default();

    // Block A: epoch 1 (expired when tip_epoch > 7)
    let block_a = BlockBuilder::default()
        .parent_hash(parent_hash.clone())
        .number(1u64.pack())
        .epoch(EpochNumberWithFraction::new(1, 0, 1000))
        .timestamp(unix_time_as_millis())
        .build();

    // Block B: epoch 9999 (never expired)
    let block_b = BlockBuilder::default()
        .parent_hash(parent_hash.clone())
        .number(1u64.pack())
        .epoch(EpochNumberWithFraction::new(9999, 0, 1000))
        .timestamp(unix_time_as_millis() + 1)
        .build();

    pool.insert(LonelyBlock { block: Arc::new(block_a), switch: None, verify_callback: None }.into());
    pool.insert(LonelyBlock { block: Arc::new(block_b), switch: None, verify_callback: None }.into());

    let tip_epoch = 1 + EXPIRED_EPOCH + 1; // = 8
    let cleaned = pool.clean_expired_blocks(tip_epoch);

    // With the bug: non-deterministically returns 0 or 2 depending on HashMap order
    // With the fix: deterministically returns 2
    assert_eq!(cleaned.len(), 2);
}
```

Running this test non-deterministically passes or fails with the current `iter().next()` implementation, demonstrating the HashMap ordering dependency. With the `all()`-based fix it passes deterministically.

        let block_number = lonely_block.block().number();
        let block_hash = lonely_block.block().hash();
        // Skip verifying a genesis block if its hash is equal to our genesis hash,
        // otherwise, return error and ban peer.
        if block_number < 1 {
            if self.shared.genesis_hash() != block_hash {
                warn!(
                    "receive 0 number block: 0-{}, expect genesis hash: {}",
                    block_hash,
                    self.shared.genesis_hash()

                self.shared
                    .insert_block_status(lonely_block.block().hash(), BlockStatus::BLOCK_INVALID);
                let error = InternalErrorKind::System
                    .other("Invalid genesis block received")
                    .into();
                lonely_block.execute_callback(Err(error));
            } else {
                warn!("receive 0 number block: 0-{}", block_hash);
                lonely_block.execute_callback(Ok(false));
            }
