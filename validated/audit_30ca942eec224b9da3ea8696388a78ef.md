### Title
DAO Withdrawal `header_dep_index` Integer Width Mismatch Between C VM and Rust Verifier Causes Consensus Split — (`util/dao/src/lib.rs`)

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` reads the `header_dep_index` from the witness as a full `u64`, while the C VM (dao.c) reads it as a `u8` (lowest byte only). A transaction sender can craft a DAO withdrawal transaction with `header_dep_index` = 257, causing C VM and Rust to resolve different block headers for DAO interest calculation. The C VM accepts the transaction (correct deposit block via lowest byte), but Rust's verifier rejects it (wrong block via full `u64`), producing a consensus split between the authoritative on-chain script and the Rust node's verification layer.

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the `header_dep_index` from the witness using `LittleEndian::read_u64`, then uses it to index into `header_deps`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

followed by:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
    .and_then(|hash| header_deps.get(&hash))
    .ok_or(DaoError::InvalidOutPoint)
``` [1](#0-0) 

The C VM (dao.c) reads the same 8-byte little-endian field as a `u8` (lowest byte only). When the witness encodes index `257` (little-endian: `0x01, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00`):

- **C VM** reads: `0x01` → resolves `header_deps[1]` → correct deposit block
- **Rust** reads: `257` → resolves `header_deps[257]` → wrong block (e.g., the withdraw block)

Rust then reaches the block-number consistency check:

```rust
if deposit_header.number() >= withdrawing_header.number() {
    return Err(DaoError::InvalidOutPoint);
}
``` [2](#0-1) 

Because Rust resolved `header_deps[257]` = the withdraw block, `deposit_header.number()` equals `withdrawing_header.number()`, and the check fires — Rust returns `DaoError::InvalidOutPoint` and rejects the transaction. The C VM, using `header_deps[1]` = the actual deposit block, passes the same check and accepts.

This discrepancy is explicitly documented in the test suite:

```rust
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
...
// Rust resolves index 257 → withdraw block (number 200), but cell data
// says deposited at block 100. Block number check catches the mismatch.
assert!(result.is_err(), "expected Err, got {result:?}");
``` [3](#0-2) 

Because `DaoCalculator::transaction_fee` (which calls `transaction_maximum_withdraw`) is invoked from `verification/src/transaction_verifier.rs` as part of block-level transaction verification, a block containing such a crafted transaction will be rejected by every Rust node even though the C VM script execution succeeds.

### Impact Explanation

An unprivileged transaction sender constructs a DAO phase-2 withdrawal transaction with 258 `header_deps`, placing the correct deposit block at index 1 and the withdraw block at index 257, and encodes `257u64` as the `input_type` witness field. The C VM resolves index 1 (deposit block) and accepts the transaction. Rust resolves index 257 (withdraw block), fails the block-number check, and returns `DaoError::InvalidOutPoint`.

1. **Tx-pool censorship**: The transaction is rejected by every node's tx-pool, making it impossible to relay a valid DAO withdrawal through normal P2P channels.
