### Title
Silent Truncation in DAO Withdrawal Capacity Calculation — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` converts a `u128` intermediate result to `u64` using a silent truncating `as u64` cast. Every other analogous `u128 → u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When the intermediate value exceeds `u64::MAX`, the function silently returns a wrong (truncated) withdrawal amount instead of propagating an error, causing incorrect DAO interest payouts and potentially breaking withdrawal transaction verification.

---

### Finding Description

In `calculate_maximum_withdraw` (`util/dao/src/lib.rs`, lines 152–156):

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a **wrapping/truncating** cast in Rust. If `withdraw_counted_capacity > u64::MAX`, the high bits are silently discarded and the function returns a value that is `2^64 * k` less than the correct result (for some integer `k ≥ 1`).

Every other `u128 → u64` narrowing in the same file uses a checked conversion:

```rust
// secondary_block_reward (line 204)
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch (line 244–245)
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);

// dao_field_with_current_epoch (line 258)
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The inconsistency is clear: `calculate_maximum_withdraw` is the only function in this file that omits the overflow check.
