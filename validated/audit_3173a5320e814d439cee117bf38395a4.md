Audit Report

## Title
Unbounded Concurrent Decompression Memory via Per-Frame 8 MB Allocation With No Global Budget — (`network/src/compress.rs`)

## Summary
`LengthDelimitedCodecWithCompress::decode` allocates up to 8 MB per frame based on the snappy-declared uncompressed length before decompression, with no global or per-connection budget limiting concurrent allocations. An unprivileged attacker can open up to 117 inbound connections and send max-amplification snappy frames on each, causing up to 117 × 8 MB ≈ 936 MB of concurrent heap allocation, which can OOM-kill a CKB node on memory-constrained hardware.

## Finding Description
In `network/src/compress.rs`, `LengthDelimitedCodecWithCompress::decode` reads the declared decompressed length from the snappy stream header via `decompress_len`, checks it against `MAX_UNCOMPRESSED_LEN` (8 MB at line 13), and then unconditionally zero-allocates a buffer of that full size before decompression: [1](#0-0) [2](#0-1) 

The per-frame guard only rejects frames whose *declared* decompressed size exceeds 8 MB; it does not limit how many such allocations may be live simultaneously across connections. The `max_frame_length` check enforced by the inner `length_delimited` codec bounds only the **compressed wire frame**. For Sync (2 MB compressed max) and RelayV3 (4 MB compressed max), 8 MB of repetitive data (e.g., zeros) compresses to ~8 KB with snappy — well within those limits — yet decompresses to 8 MB, a 1000× amplification ratio: [3](#0-2) 

The codec is instantiated per-connection per-protocol via the `CKBProtocol` builder: [4](#0-3) 

There is no semaphore, counter, or token bucket anywhere in the codec layer that limits how many 8 MB `BytesMut` allocations may be live simultaneously. CKB uses a multi-threaded tokio runtime, so multiple `decode` calls execute concurrently across worker threads. Each live allocation persists from the time `decode` returns until the application layer drops the `BytesMut`.

## Impact Explanation
The default peer configuration is `max_peers = 125`, `max_outbound_peers = 8`, giving `max_inbound_peers = 117`: [5](#0-4) [6](#0-5) 

Peak concurrent decompression heap usage is bounded only by `max_inbound_peers × MAX_UNCOMPRESSED_LEN = 117 × 8 MB ≈ 936 MB`. On memory-constrained nodes (2–4 GB RAM VPS, common for CKB validators), this alone can exhaust available memory and trigger the OOM killer, crashing the node. This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attack requires only: (1) opening up to 117 TCP connections to the node's public P2P port (port 8115 by default), (2) completing the tentacle protocol handshake (secio key exchange + protocol negotiation), (3) opening the Sync or RelayV3 protocol, and (4) sending a crafted snappy-compressed frame (~8 KB on the wire) on each connection. No proof-of-work, no keys, no privileged role is required. The attack is repeatable and sustainable — the attacker simply re-sends max-amplification frames continuously to maintain heap pressure.

## Recommendation
Introduce a global decompression semaphore or token bucket shared across all codec instances, e.g.:

```rust
static DECOMPRESS_BUDGET: Semaphore = Semaphore::const_new(MAX_CONCURRENT_DECOMPRESS); // e.g. 8
```

Acquire a permit before `BytesMut::zeroed(decompressed_bytes_len)` and release it after decompression completes. Alternatively, lower `MAX_UNCOMPRESSED_LEN` to match the per-protocol `max_frame_length` (so the compressed-frame bound also bounds the decompressed output), or add a node-wide decompression memory counter that rejects frames when aggregate in-flight decompression exceeds a configurable limit (e.g., 64 MB).

## Proof of Concept
1. Start a CKB node with default config (`max_peers = 125`).
2. Record baseline RSS.
3. Open 117 inbound TCP connections to port 8115, complete the tentacle handshake and Sync protocol negotiation on each.
4. On each connection, send a snappy-compressed frame: compress 8 MB of `\x00` bytes (result ~8 KB), prepend the compress flag byte (`0x80`) and the 4-byte length-delimited header.
5. Observe target node RSS delta; assert it is bounded by a reasonable per-node limit rather than `117 × 8 MB ≈ 936 MB`.

A unit test can be written directly against `LengthDelimitedCodecWithCompress::decode` by spawning N tokio tasks each calling `decode` with a crafted max-amplification snappy frame simultaneously and measuring peak allocator usage via `jemalloc` stats or `/proc/self/status VmRSS`.
