### Title
Two-Phase Commit Bypass via `ProposalShortId` Collision — (`verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

CKB's two-phase commit rule requires every committed transaction to have been previously proposed within the proposal window. The `TwoPhaseCommitVerifier` enforces this by matching committed transactions to proposals using `ProposalShortId` — a **10-byte (80-bit) truncation** of the transaction hash. Because two distinct transactions can share the same `ProposalShortId`, a miner can propose a "decoy" transaction and then commit a completely different transaction that was never proposed, bypassing the two-phase commit requirement entirely. All nodes accept the block because they run the same truncated-ID check.

---

### Finding Description

`ProposalShortId` is defined as the first 10 bytes of the BLAKE2b transaction hash:

```rust
// util/gen-types/src/extension/shortcut.rs:29-33
pub fn from_tx_hash(h: &packed::Byte32) -> Self {
    let mut inner = [0u8; 10];
    inner.copy_from_slice(&h.as_slice()[..10]);
    inner.into()
}
``` [1](#0-0) 

The `TwoPhaseCommitVerifier::verify()` collects all proposed short IDs from the proposal window, then checks that every committed transaction's short ID appears in that set:

```rust
// verification/contextual/src/contextual_block_verifier.rs:187-195
let committed_ids: HashSet<_> = self
    .block
    .transactions()
    .iter()
    .skip(1)
    .map(TransactionView::proposal_short_id)   // ← 10-byte truncation
    .collect();

if committed_ids.difference(&proposal_txs_ids).next().is_some() {
    return Err((CommitError::Invalid).into());
}
``` [2](#0-1) 

The check is purely over 10-byte short IDs. If `tx_A` and `tx_B` share the same `ProposalShortId` (i.e., their hashes agree on the first 10 bytes), proposing `tx_A` satisfies the two-phase commit check for `tx_B`. The verifier never compares full 32-byte transaction hashes.

The codebase itself acknowledges that short ID collisions are a known possibility in the compact-block relay path:

```rust
// sync/src/relayer/block_transactions_process.rs:12-22
// Keeping in mind that short_ids are expected to occasionally collide.
// ...
// there may be short_id collision in transaction pool.
``` [3](#0-2) 

The relay layer handles collisions gracefully (falls back to requesting full transactions). The consensus verifier does not — it treats a matching short ID as proof of a valid proposal.

The `RewardCalculator` also uses `ProposalShortId::from_tx_hash` to match committed transactions to proposals for reward distribution, meaning a collision also corrupts proposer reward accounting:

```rust
// util/reward-calculator/src/lib.rs:204-210
let committed_idx_proc = |hash: &Byte32| -> Vec<ProposalShortId> {
    store
        .get_block_txs_hashes(hash)
        .into_iter()
        .skip(1)
        .map(|tx_hash| ProposalShortId::from_tx_hash(&tx_hash))
        .collect()
};
``` [4](#0-3) 

---

### Impact Explanation

A miner who finds a `ProposalShortId` collision between `tx_A` and `tx_B` can:

1. Propose `tx_A` (the decoy) in block N — this is a normal, valid proposal.
2. Mine block N+k (within the proposal window) that commits `tx_B` (the real target) without ever proposing it.
3. `TwoPhaseCommitVerifier` sees `tx_B`'s short ID in the proposal set (from `tx_A`'s proposal) and accepts the block.
4. All other nodes run the same verifier and also accept the block.

**Consequences:**
- The two-phase commit rule — a core CKB consensus invariant — is bypassed for the committed transaction.
- The network never had the required proposal window to observe and validate `tx_B` before it was committed.
- Proposer reward attribution is corrupted: the reward for committing `tx_B` is credited to whoever proposed `tx_A`, not the actual proposer of `tx_B` (which does not exist).
- The attack is network-wide and silent: no node detects the bypass because all nodes use the same truncated-ID check.

---

### Likelihood Explanation

`ProposalShortId` is 80 bits. Finding a collision (two transactions whose hashes share the same first 10 bytes) requires approximately **2^40 hash evaluations** via a birthday attack. This is within reach of a miner with dedicated hardware (comparable to mining-level computation). The attacker controls the transaction content (inputs, outputs, witnesses) and can iterate over witness bytes to search for a collision without needing to break BLAKE2b — only its 80-bit prefix needs to match.

The tx-pool's `check_txid_collision` rejects `tx_B` from the mempool if `tx_A` is already present with the same short ID, but a miner does not need `tx_B` in the mempool — they can include it directly in a block they produce. [5](#0-4) 

---

### Recommendation

The `TwoPhaseCommitVerifier` should verify proposals against the **full 32-byte transaction hash**, not the 10-byte `ProposalShortId`. This requires storing full transaction hashes in the proposal zone rather than short IDs, which is a protocol-level change. As an interim measure, the proposal zone could store both the short ID (for compact relay) and the full hash (for consensus verification). Alternatively, the proposal short ID length should be increased to at least 20 bytes to make collisions computationally infeasible.

---

### Proof of Concept

The codebase's own test infrastructure demonstrates that two transactions can share a `ProposalShortId` while having different full hashes:

```rust
// sync/src/relayer/tests/compact_block_process.rs:585-597
let fake_hash = missing_tx.hash().as_builder()
    .nth31(0u8).nth30(0u8).nth29(0u8).nth28(0u8).build();
// Fake tx with the same ProposalShortId but different hash with missing_tx
let fake_tx = missing_tx.clone().fake_hash(fake_hash);

assert_eq!(missing_tx.proposal_short_id(), fake_tx.proposal_short_id());
assert_ne!(missing_tx.hash(), fake_tx.hash());
``` [6](#0-5) 

**Attack steps:**

1. Craft `tx_A` and `tx_B` such that `tx_A.hash()[..10] == tx_B.hash()[..10]` (requires ~2^40 BLAKE2b evaluations over witness bytes).
2. Submit a block proposing `tx_A`'s short ID in the proposal zone.
3. After `proposal_window.closest()` blocks, mine a block that commits `tx_B` (not `tx_A`) in the transaction zone.
4. `TwoPhaseCommitVerifier::verify()` at line 195 computes `committed_ids` containing `tx_B.proposal_short_id()`, finds it in `proposal_txs_ids` (which contains `tx_A.proposal_short_id()` — identical value), and returns `Ok(())`.
5. The block passes all consensus checks on every node. `tx_B` is committed without ever being proposed. [2](#0-1)
