### Title
Timestamp Underflow in `ema_price_no_older_than` Causes DoS When `publish_time` Exceeds Current Block Time - (`File: target_chains/fuel/contracts/pyth-contract/src/main.sw`)

---

### Summary

The `ema_price_no_older_than` function in the Pyth Fuel contract performs an unchecked unsigned integer subtraction `current_time - price.publish_time`. In Sway, arithmetic on `u64` panics on underflow. If a price update is submitted with a `publish_time` greater than the Fuel blockchain's current `timestamp()`, any call to `ema_price_no_older_than` (and by extension `ema_price`) will revert until the chain's clock catches up. The sibling function `price_no_older_than` already has a saturating-subtraction guard for exactly this reason, but `ema_price_no_older_than` was left unprotected.

---

### Finding Description

In `target_chains/fuel/contracts/pyth-contract/src/main.sw`, the private function `ema_price_no_older_than` computes the age of a stored EMA price as:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,  // ← underflow if publish_time > current_time
        PythError::OutdatedPrice,
    );
    price
}
``` [1](#0-0) 

The developers were already aware of this class of bug and applied a saturating-subtraction guard in the parallel `price_no_older_than` function:

```sway
fn price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
``` [2](#0-1) 

The fix was applied to `price_no_older_than` but not to `ema_price_no_older_than`. Both functions are exposed as public ABI entry points: [3](#0-2) 

The `ema_price` public entry point also calls `ema_price_no_older_than` via `valid_time_period()`, so it is equally affected:

```sway
fn ema_price(price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(valid_time_period(), price_feed_id)
}
```

The `publish_time` stored in `latest_price_feed` originates from VAA/attestation data submitted by any unprivileged caller via `update_price_feeds`. Pythnet timestamps can legitimately run slightly ahead of the Fuel blockchain's clock due to inter-chain clock drift. A price update with `publish_time` even 1 second ahead of the Fuel node's `timestamp()` is sufficient to trigger the underflow. [4](#0-3) 

---

### Impact Explanation

Any call to `ema_price` or `ema_price_no_older_than` reverts with an arithmetic underflow panic for the duration that `price.publish_time > timestamp()`. DeFi protocols on Fuel that rely on EMA prices (e.g., for TWAP-based liquidations or risk calculations) are completely blocked from reading EMA prices during this window. The DoS is temporary (resolves once the Fuel clock catches up) but is repeatable: a new update with a future `publish_time` can be submitted again immediately after the window closes.

---

### Likelihood Explanation

Pythnet and Fuel run independent clocks. Minor clock drift (seconds) between the two chains is a normal operational condition, not an adversarial one. A legitimate price update relayed by any unprivileged updater can carry a `publish_time` that is a few seconds ahead of the Fuel block timestamp. This is not a theoretical edge case — it is the exact scenario the `price_no_older_than` fix was written to handle.

---

### Recommendation

Apply the same saturating-subtraction guard to `ema_price_no_older_than`:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

---

### Proof of Concept

1. Attacker (or any relayer) calls `update_price_feeds` with a valid VAA whose EMA price `publish_time` is set to `current_fuel_timestamp + 5` seconds (within normal Pythnet clock drift).
2. The price is accepted and stored in `latest_price_feed` because `update_price_feeds` does not validate that `publish_time <= current_time`.
3. Any subsequent call to `ema_price(price_feed_id)` or `ema_price_no_older_than(period, price_feed_id)` executes `current_time - price.publish_time` where `price.publish_time > current_time`, causing a `u64` underflow panic in Sway and reverting the transaction.
4. All callers of EMA price functions are DoS'd for up to ~5 seconds. The attacker can repeat step 1 continuously to extend the DoS indefinitely. [1](#0-0) [2](#0-1)
