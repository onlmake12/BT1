### Title
`PythAggregatorV3.latestRoundData()` and `latestAnswer()` Return Stale Price Without Any Staleness Check — (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official production SDK contract that implements the Chainlink `AggregatorV3Interface` and is the documented migration path for Chainlink users. Every price-returning function in the contract calls `pyth.getPriceUnsafe()`, which explicitly makes no recency guarantees. There is no staleness check, no heartbeat validation, and the `answeredInRound == roundId` construction causes any downstream Chainlink-style staleness guard (`require(answeredInRound >= roundId)`) to always pass silently, masking the stale state entirely.

---

### Finding Description

`PythAggregatorV3.latestRoundData()` is implemented as:

```solidity
function latestRoundData() external view returns (
    uint80 roundId, int256 answer,
    uint256 startedAt, uint256 updatedAt, uint80 answeredInRound
) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);  // ← no staleness check
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
``` [1](#0-0) 

`getPriceUnsafe()` is explicitly documented as returning a price from **arbitrarily far in the past** with no recency guarantee: [2](#0-1) 

The same unchecked `getPriceUnsafe()` call is used in `latestAnswer()`, `latestTimestamp()`, `getAnswer()`, `getTimestamp()`, `getRoundData()`, and `decimals()`: [3](#0-2) 

There are two compounding issues:

1. **No staleness check**: `getPriceUnsafe()` returns whatever is cached on-chain regardless of age. Pyth is a pull oracle — if no one calls `updateFeeds()`, the cached price can be hours or days old.

2. **`answeredInRound` is always equal to `roundId`**: Both are set to `uint80(price.publishTime)`. This means the standard Chainlink downstream guard `require(answeredInRound >= roundId, "stale price")` always passes, giving integrators false confidence that the price is fresh. [4](#0-3) 

A secondary issue exists in `decimals()`: the cast `uint8(-1 * int8(price.expo))` performs a narrowing `int32 → int8` truncation. If `price.expo` is outside `[-128, 127]` (valid for `int32`), the cast silently truncates the value, producing a wrong decimal count. If `price.expo` is positive, the negation produces a negative value and `uint8(negative)` reverts in Solidity ≥0.8, causing a DoS on `decimals()`. [5](#0-4) 

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a Chainlink-compatible oracle adapter — which is the **officially documented migration path** — and calls `latestRoundData()` or `latestAnswer()` will silently receive a stale price. Downstream protocols (lending, derivatives, liquidation engines) that rely on this price for collateral valuation, liquidation thresholds, or settlement will operate on incorrect data. This can enable:

- Undercollateralized borrowing (attacker borrows against inflated stale collateral price)
- Blocked liquidations (attacker avoids liquidation because stale price shows healthy collateral)
- Incorrect settlement of derivative positions

The migration guide explicitly instructs users to deploy `PythAggregatorV3` and pass it to their existing Chainlink-consuming app without any additional staleness wrapper: [6](#0-5) 

---

### Likelihood Explanation

**Medium-High.** `PythAggregatorV3` is a production SDK contract published in `@pythnetwork/pyth-sdk-solidity` and is the official Chainlink migration adapter. Pyth is a pull oracle — prices are only updated when `updateFeeds()` is called. During network congestion, market closures, or keeper downtime, the cached price can become arbitrarily stale. Any protocol that migrated from Chainlink using this adapter and did not add its own staleness check is currently vulnerable. The `answeredInRound == roundId` construction actively suppresses the most common downstream staleness guard.

---

### Recommendation

Replace `getPriceUnsafe()` with `getPriceNoOlderThan()` in all price-returning functions, using a configurable `maxAge` parameter set at construction time:

```solidity
uint256 public maxAge;

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (
    uint80 roundId, int256 answer,
    uint256 startedAt, uint256 updatedAt, uint80 answeredInRound
) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

For `decimals()`, replace the unsafe cast with a safe bounds-checked conversion:

```solidity
function decimals() public view virtual returns (uint8) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    require(price.expo <= 0 && price.expo >= -255, "invalid expo");
    return uint8(uint32(-price.expo));
}
```

### Proof of Concept

1. A lending protocol deploys `PythAggregatorV3` for ETH/USD following the official Pyth migration guide.
2. The keeper that calls `updateFeeds()` goes offline for 2 hours. The ETH price drops 20% in that time.
3. An attacker calls `latestRoundData()` on the adapter. It returns the 2-hour-old stale price via `getPriceUnsafe()` with no revert.
4. The downstream lending protocol's staleness guard `require(answeredInRound >= roundId)` passes because `answeredInRound == roundId` always.
5. The attacker borrows against the inflated stale collateral value, extracting funds the protocol would not have permitted at the true current price.

The root cause is entirely within Pyth's own production SDK file `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol` — the use of `getPriceUnsafe()` in all price-returning functions with no staleness enforcement and a `answeredInRound` construction that defeats downstream guards.
