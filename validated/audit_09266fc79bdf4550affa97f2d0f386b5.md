### Title
Permanent Post-Upgrade Deadlock: `governance::verify_vaa` Calls `assert_latest_only` Before `migrate_version` Is Applied, Making `migrate()` Uncallable — (`target_chains/sui/contracts/sources/migrate.move`, `governance/governance.move`)

---

### Summary

After `contract_upgrade::commit_upgrade` is called, the Pyth `State` object's stored version remains at `V__0_1_1` while the newly deployed package's hard-coded `current_version()` is `V__0_1_2`. The only way to advance the stored version is to call `migrate::migrate()`, which requires a `WormholeVAAVerificationReceipt`. The only way to obtain that receipt is via `governance::verify_vaa`, which unconditionally calls `state::assert_latest_only` — a check that fails because the stored version has not yet been advanced. `migrate_version` is only called *inside* `migrate()` itself. This creates an unresolvable circular dependency: the receipt cannot be produced without the version being current, and the version cannot be made current without the receipt.

---

### Finding Description

**Step 1 — `governance::verify_vaa` hard-gates on `assert_latest_only`** [1](#0-0) 

`verify_vaa` is the sole constructor for `WormholeVAAVerificationReceipt`. Its very first action is `state::assert_latest_only(pyth_state)`, which aborts unless the version stored in `State`'s dynamic fields equals `version_control::current_version()`.

**Step 2 — `assert_latest_only` compares stored version against the new package's hard-coded constant** [2](#0-1) [3](#0-2) 

After `commit_upgrade`, the stored version is still `V__0_1_1`. The new package's `current_version()` returns `V__0_1_2`. The assertion fails.

**Step 3 — `migrate()` requires the receipt as an external input** [4](#0-3) 

`migrate()` is `public` but takes `receipt: WormholeVAAVerificationReceipt` as a mandatory argument. There is no alternative entry point.

**Step 4 — `migrate_version` is called *inside* `handle_migrate`, after the receipt is already required** [5](#0-4) 

`state::migrate_version` (line 56) advances the stored version from `V__0_1_1` to `V__0_1_2`. But this only executes after `migrate()` has already been entered with a valid receipt — a receipt that cannot be produced because the version has not yet been advanced.

**Step 5 — `WormholeVAAVerificationReceipt` has no `store` or `key` ability; it cannot be persisted across transactions** [6](#0-5) 

The receipt is a hot-potato-like value. It cannot be created in Transaction 1 (old package, where `assert_latest_only` would pass) and carried into Transaction 2 (new package, where `migrate()` must be called). The receipt must be created in the same PTB as `migrate()`, but in that PTB `verify_vaa` aborts.

**Step 6 — No escape hatch exists**

`contract_upgrade::authorize_upgrade` also requires a `WormholeVAAVerificationReceipt`: [7](#0-6) 

So a second upgrade to "fix" the first is equally blocked. The `UpgradeCap` is embedded inside `State` and is only accessible through `state::authorize_upgrade`, which is gated behind `contract_upgrade::authorize_upgrade`, which is gated behind `verify_vaa`. The contract is permanently frozen.

**Contrast with the Wormhole vendor implementation**

The Wormhole `migrate.move` correctly avoids this by accepting raw VAA bytes and calling `parse_and_verify` / `governance_message::verify_vaa` *inside* `handle_migrate`, *after* `state::migrate_version` has already been applied: [8](#0-7) 

Pyth diverged from this pattern by pre-verifying the VAA externally, introducing the deadlock.

---

### Impact Explanation

Once `commit_upgrade` is executed, the Pyth contract on Sui is permanently non-functional:
- All price-feed update paths call `assert_latest_only` and abort.
- All governance actions (fee changes, data-source changes, further upgrades) are blocked.
- Any SUI fees accumulated in the contract are permanently locked.
- DeFi protocols depending on Pyth price feeds lose access to fresh prices.

The `UpgradeCap` is trapped inside the frozen `State` object with no recovery path.

---

### Likelihood Explanation

This triggers on every legitimate upgrade attempt — it is not a probabilistic or attacker-dependent condition. Any operator following the documented upgrade procedure (`authorize_upgrade` → Sui upgrade → `commit_upgrade` → `migrate`) will permanently brick the contract. No privileged key compromise or malicious actor is required.

---

### Recommendation

Adopt the same pattern used by the Wormhole vendor code: pass the raw VAA bytes (not a pre-verified receipt) into `migrate()`, and call `parse_and_verify` / `verify_vaa` *inside* `handle_migrate` *after* `state::migrate_version` has been applied. Concretely:

```move
public fun migrate(
    pyth_state: &mut State,
    verified_vaa: VAA,          // raw VAA, not a pre-verified receipt
) {
    handle_migrate(pyth_state, verified_vaa);
}

fun handle_migrate(pyth_state: &mut State, verified_vaa: VAA) {
    state::migrate_version(pyth_state);          // advance version FIRST
    let latest_only = state::assert_latest_only(pyth_state);  // now passes
    // verify the VAA here, after version is current
    let receipt = governance::verify_vaa(pyth_state, verified_vaa);
    let digest = contract_upgrade::take_upgrade_digest(receipt);
    state::assert_authorized_digest(&latest_only, pyth_state, digest);
    ...
}
```
