### Title
Deviation-Only Scheduler Subscriptions Become Permanently Stale During Low-Volatility Periods — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler` contract allows subscriptions to be configured with `updateOnDeviation = true` and `updateOnHeartbeat = false`. When the market price does not move by `deviationThresholdBps`, no keeper can push a valid update (every call to `updatePriceFeeds` reverts with `UpdateConditionsNotMet`). After `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` (1 hour) elapses without an update, all calls to `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` revert with `StalePrice`, making the subscription's price data completely unavailable for an unbounded duration.

---

### Finding Description

In `_validateShouldUpdatePrices`, the logic evaluates update criteria in order:

1. **Heartbeat check** (lines 400–410): if `updateOnHeartbeat` is enabled and the heartbeat interval has elapsed, the update is accepted and the function returns.
2. **Deviation check** (lines 413–451): if `updateOnDeviation` is enabled, it iterates all price feeds and checks whether any feed's price has moved by at least `deviationThresholdBps` basis points relative to the stored previous price.
3. **Fallthrough** (line 453): if neither condition triggers, the function reverts with `UpdateConditionsNotMet`. [1](#0-0) 

When a subscription is created with `updateOnHeartbeat = false` and `updateOnDeviation = true`, the heartbeat branch is entirely skipped. If the price stays within the deviation threshold — which is common during low-volatility periods for stablecoins, pegged assets, or any asset in a calm market — every `updatePriceFeeds` call reverts. No update can be stored. [2](#0-1) 

The `_validateSubscriptionParams` function only rejects `deviationThresholdBps == 0`; it places no upper bound on the value. A subscription with `deviationThresholdBps = 10000` (100%) would never update unless the price doubles. [3](#0-2) 

`PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is hardcoded to 1 hour: [4](#0-3) 

Once 1 hour passes without a successful update, `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` both check `distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds` and revert with `StalePrice`: [5](#0-4) [6](#0-5) 

The subscription's price data is then completely inaccessible for as long as the price remains within the deviation band — potentially indefinitely.

---

### Impact Explanation

Any protocol that creates a Scheduler subscription with deviation-only update criteria and relies on `getPricesNoOlderThan` or `getEmaPricesNoOlderThan` will experience a complete DoS on price reads during any period of low volatility. This is directly analogous to the original report: a threshold parameter that is not bounded causes the system to be unavailable for extended periods. The subscription's balance is also drained by failed `updatePriceFeeds` calls (the Pyth fee is deducted before the revert check on update conditions). [7](#0-6) 

---

### Likelihood Explanation

Any user can create a subscription (unprivileged entry path via `createSubscription`). The `UpdateCriteria` struct exposes `updateOnHeartbeat` and `updateOnDeviation` as independent booleans with no enforcement that a heartbeat must accompany deviation-only mode. Stablecoin pairs, pegged assets, and any asset during calm market conditions will routinely stay within a 1–5% deviation band for hours or days. The combination of a deviation-only subscription and a threshold above typical volatility is a realistic and likely configuration. [8](#0-7) 

---

### Recommendation

1. **Require a heartbeat fallback when using deviation-only mode**: In `_validateSubscriptionParams`, if `updateOnDeviation = true` and `updateOnHeartbeat = false`, revert with `InvalidUpdateCriteria`. This ensures there is always a time-based safety net that guarantees eventual updates.
2. **Alternatively, enforce a maximum `deviationThresholdBps`**: Add a constant (e.g., `MAX_DEVIATION_THRESHOLD_BPS = 1000`) and reject subscriptions that exceed it, analogous to the original report's recommendation to constrain `THRESHOLD` to a tight range. [9](#0-8) 

---

### Proof of Concept

1. Alice calls `createSubscription` with:
   - `updateOnHeartbeat = false`
   - `updateOnDeviation = true`
   - `deviationThresholdBps = 500` (5%)
2. Alice's protocol calls `updatePriceFeeds` at T=0. The first update succeeds (no previous price stored, so `previousFeed.id == bytes32(0)` triggers an immediate return).
3. The asset price moves only 2% over the next 2 hours (within the 5% threshold).
4. Every subsequent `updatePriceFeeds` call reverts with `UpdateConditionsNotMet` at line 453.
5. At T=1 hour, `getPricesNoOlderThan(subscriptionId, priceIds, 3600)` reverts with `StalePrice` because `distance(block.timestamp, status.priceLastUpdatedAt) > 3600`.
6. Alice's protocol is unable to read any price from the subscription for the entire duration the price stays within the 5% band — potentially days. [10](#0-9)

        // If updateOnDeviation is enabled, check if any price has deviated enough
        if (params.updateCriteria.updateOnDeviation) {
            for (uint8 i = 0; i < priceFeeds.length; i++) {
                // Get the previous price feed for this price ID using subscriptionId
                PythStructs.PriceFeed storage previousFeed = _state
                    .priceUpdates[subscriptionId][priceFeeds[i].id];

                // If there's no previous price, this is the first update
                if (previousFeed.id == bytes32(0)) {
                    return updateTimestamp;
                }

                // Calculate the deviation percentage
                int64 currentPrice = priceFeeds[i].price.price;
                int64 previousPrice = previousFeed.price.price;

                // Skip if either price is zero to avoid division by zero
                if (previousPrice == 0 || currentPrice == 0) {
                    continue;
                }

                // Calculate absolute deviation basis points (scaled by 1e4)
                uint256 numerator = SignedMath.abs(
                    currentPrice - previousPrice
                );
                uint256 denominator = SignedMath.abs(previousPrice);
                uint256 deviationBps = Math.mulDiv(
                    numerator,
                    10_000,
                    denominator
                );

                // If deviation exceeds threshold, trigger update
                if (
                    deviationBps >= params.updateCriteria.deviationThresholdBps
                ) {
                    return updateTimestamp;
                }
            }
        }

        revert SchedulerErrors.UpdateConditionsNotMet();
