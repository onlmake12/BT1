### Title
Scheduler Allows Multiple Price Updates Per Block, Enabling Keeper Fee Drain — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `Scheduler` contract's `updatePriceFeeds` function has no per-block guard. Because it validates freshness using Pyth's off-chain `publishTime` (not `block.timestamp`), and the validity window is 1 hour, any unprivileged keeper can submit two or more valid Pyth price updates in the same block and collect keeper fees for each call. This drains subscription balances faster than the subscription owner intended.

### Finding Description

`Scheduler.updatePriceFeeds` is callable by any address with no access control. Its only temporal guard is in `_validateShouldUpdatePrices`:

```solidity
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(...);
}
``` [1](#0-0) 

This check uses the Pyth `publishTime` (an off-chain timestamp), not `block.timestamp`. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is 1 hour: [2](#0-1) 

Pyth publishes prices every ~400ms. Within any given block, there exist dozens of valid, Wormhole-attested Pyth price updates with strictly increasing `publishTime` values, all within the 1-hour validity window.

For a **heartbeat-based subscription** with `heartbeatSeconds = 60`, an attacker can:
1. Fetch two valid Pyth price updates: P1 at `publishTime = T` and P2 at `publishTime = T + 60`, both within the last hour.
2. In a single block, call `updatePriceFeeds(subscriptionId, P1)` — passes the heartbeat check (`lastUpdateTime == 0` or `T >= lastUpdateTime + 60`), stores P1, pays keeper fee.
3. Immediately call `updatePriceFeeds(subscriptionId, P2)` — passes the heartbeat check (`T+60 >= T + 60`), stores P2, pays keeper fee again.

After each successful call, `_processFeesAndPayKeeper` pays `msg.sender`: [3](#0-2) 

The subscription's `balanceInWei` is debited twice in the same block. The subscription owner expected one debit per heartbeat interval, but receives two (or more).

For **deviation-based subscriptions**, the same attack applies if the price moved enough between P1 and P2 to satisfy `deviationThresholdBps` — which is possible in volatile markets.

### Impact Explanation

Subscription balances are drained at a multiple of the expected rate. A subscription funded for N heartbeat periods can be exhausted in N/2 (or fewer) periods. Once the balance falls below `minimumBalancePerFeed`, the subscription stops receiving updates, breaking the price feed for all downstream consumers. The attacker profits from the extra keeper fees at the expense of the subscription owner.

### Likelihood Explanation

`updatePriceFeeds` has no access control — any address can act as a keeper. Valid Pyth price data for any timestamp within the last hour is freely available from Hermes. Constructing two valid, Wormhole-attested price updates with `publishTime` values exactly `heartbeatSeconds` apart is trivial. The attack requires no privileged access, no flash loans, and no front-running.

### Recommendation

Add a per-block guard analogous to the fix recommended in the referenced FlatMoney report. Track the last `block.timestamp` at which an update was stored for each subscription, and reject a second update in the same block:

```solidity
// In SchedulerState or SubscriptionStatus:
uint256 public lastBlockUpdated; // per subscription

// In updatePriceFeeds:
if (status.lastBlockUpdated == block.number) {
    revert SchedulerErrors.AlreadyUpdatedThisBlock();
}
status.lastBlockUpdated = block.number;
```

Alternatively, use `block.timestamp` with a minimum spacing (e.g., require `block.timestamp > status.lastBlockTimestampUpdated`).

### Proof of Concept

1. Create a heartbeat subscription with `heartbeatSeconds = 60` and fund it with 1 ETH.
2. Fetch two valid Pyth price updates from Hermes: P1 with `publishTime = now - 120` and P2 with `publishTime = now - 60`. Both are within the 1-hour validity window.
3. Deploy an attacker contract that calls `scheduler.updatePriceFeeds(id, P1)` then `scheduler.updatePriceFeeds(id, P2)` in a single transaction.
4. Observe that both calls succeed (P2's `publishTime` satisfies `>= P1.publishTime + heartbeatSeconds`), the attacker receives two keeper fee payments, and `status.balanceInWei` is debited twice.
5. Repeat until the subscription balance is exhausted, at which point the subscription stops receiving price updates.

The root cause — no `block.timestamp`/`block.number` guard in `updatePriceFeeds` — is directly analogous to the missing `lastOffchainUpdate >= block.timestamp` guard in FlatMoney's `OracleModule.updatePythPrice`. [4](#0-3)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }
