### Title
User Funds Permanently Locked in Echo Contract When `executeCallback` Is Never Called — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, a user's ETH fee is split at request time: the Pyth protocol portion is credited immediately, but the provider fee portion is stored in `req.fee` and only credited to the provider upon `executeCallback`. There is no cancellation or refund path. If no provider ever calls `executeCallback`, the user's ETH is permanently locked in the contract with no recovery mechanism — a direct structural analog to the RocketPool vault issue.

---

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, the fee is split as follows:

```solidity
// Echo.sol L84, L99
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
...
_state.accruedFeesInWei += _state.pythFeeInWei;
```

The Pyth protocol fee (`_state.pythFeeInWei`) is credited to `_state.accruedFeesInWei` immediately and is withdrawable by the admin. However, the remainder — `msg.value - _state.pythFeeInWei` — is stored in `req.fee` inside the `Request` struct and is **only credited to the provider** inside `executeCallback`:

```solidity
// Echo.sol L161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

`clearRequest` is called only after this accounting line. If `executeCallback` is never called, `req.fee` remains in the contract indefinitely.

A search of the entire `echo/` directory confirms there is **no** `cancelRequest`, `refundRequest`, `receive()`, or `fallback()` function. The only way to clear a request is through `executeCallback`. The contract itself contains a TODO acknowledging this:

```solidity
// Echo.sol L155-156
// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
// If executeCallback can revert, then funds can be permanently locked in the contract.
``` [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Any ETH paid as the provider fee portion (`msg.value - pythFeeInWei`) is irrecoverable by the user if no provider ever calls `executeCallback`. The admin's `withdrawFees` function only covers `_state.accruedFeesInWei` (the Pyth protocol portion), not `req.fee`. Provider `withdrawAsFeeManager` only covers `_state.providers[provider].accruedFeesInWei`, which is only populated after a successful `executeCallback`. The locked ETH has no accounting path back to the user. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The scenario is reachable by any unprivileged user interacting with the Echo contract. It becomes likely when:

1. **Provider goes offline** after a user submits a request — no one calls `executeCallback`.
2. **Gas cost exceeds provider fee** — if `callbackGasLimit * gasPrice > req.fee`, executing the callback is unprofitable for any provider, so no provider does it.
3. **Malicious provider** registers, collects requests, and never fulfills them. After the exclusivity period expires, other providers *may* execute, but are not obligated to and have no incentive if the fee is insufficient.

The exclusivity period (`_state.exclusivityPeriodSeconds`) only gates *which* provider can execute; it does not guarantee execution by anyone. [6](#0-5) 

---

### Recommendation

Add a `cancelRequest` function that allows the original requester to reclaim `req.fee` after a timeout (e.g., after the exclusivity period plus a grace window). Example pattern:

```solidity
function cancelRequest(uint64 sequenceNumber) external {
    Request storage req = findActiveRequest(sequenceNumber);
    require(req.requester == msg.sender, "Not requester");
    require(block.timestamp > req.publishTime + _state.exclusivityPeriodSeconds + GRACE_PERIOD, "Too early");
    uint96 refund = req.fee;
    clearRequest(sequenceNumber);
    (bool sent, ) = msg.sender.call{value: refund}("");
    require(sent, "Refund failed");
}
```

This mirrors the fix applied in the RocketPool case: instead of funds going to an unrecoverable destination, they are returned to the rightful owner.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback{value: 1 ether}(provider, publishTime, priceIds, gasLimit)`.
2. `req.fee = 1 ether - pythFeeInWei` is stored; `_state.accruedFeesInWei += pythFeeInWei`.
3. Provider never calls `executeCallback` (offline, unprofitable, or malicious).
4. User has no function to call to recover `req.fee`.
5. `withdrawFees` (admin) only drains `_state.accruedFeesInWei` — the `pythFeeInWei` portion only.
6. `withdrawAsFeeManager` only drains `_state.providers[provider].accruedFeesInWei` — which is 0 because `executeCallback` was never called.
7. `req.fee` = `1 ether - pythFeeInWei` is permanently locked. [7](#0-6) [8](#0-7)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }

    // TODO: does this need to be payable? Any cost paid to Pyth could be taken out of the provider's accrued fees.
    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);

        // TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
        // a better solution would be a doubly-linked list of active requests.
        // After successful callback, update firstUnfulfilledSeq if needed
        while (
            _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
            !isActive(findRequest(_state.firstUnfulfilledSeq))
        ) {
            _state.firstUnfulfilledSeq++;
        }

        try
            IEchoConsumer(req.requester)._echoCallback{
                gas: req.callbackGasLimit
            }(sequenceNumber, priceFeeds)
        {
            // Callback succeeded
            emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
        } catch Error(string memory reason) {
            // Explicit revert/require
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                reason
            );
        } catch {
            // Out of gas or other low-level errors
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                "low-level error (possibly out of gas)"
            );
        }
    }
