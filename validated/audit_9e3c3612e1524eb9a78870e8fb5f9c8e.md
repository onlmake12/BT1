### Title
Callback Delivered to `address(0)` After Storage Deletion for Overflow Requests — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, when a request is stored in the `requestsOverflow` mapping (due to a hash-table collision in the fixed-size `requests[32]` array), `executeCallback` calls `clearRequest` — which issues a full `delete` on the overflow entry — **before** reading `req.requester` and `req.callbackGasLimit` to invoke the user's callback. Because `req` is a storage pointer, the delete zeroes the underlying storage, so the callback is dispatched to `address(0)` with 0 gas. The provider is still paid, but the user's callback is permanently lost.

### Finding Description

`Echo.sol` uses a two-level hash table to store in-flight requests: a fixed `Request[32]` array (primary) and a `mapping(bytes32 => Request) requestsOverflow` (secondary). When a new request's 5-bit `shortKey` collides with an already-active slot, `allocRequest` evicts the prior request into `requestsOverflow`. [1](#0-0) [2](#0-1) 

`executeCallback` then proceeds as follows:

1. Obtains a `storage` pointer `req` via `findActiveRequest`, which may resolve to `_state.requestsOverflow[key]`.
2. Credits the provider fee using `req.fee` (still valid at this point).
3. Calls `clearRequest(sequenceNumber)`. [3](#0-2) 

`clearRequest` has asymmetric behavior:

```solidity
function clearRequest(uint64 sequenceNumber) internal {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    Request storage req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        req.sequenceNumber = 0;          // array path: only zeroes one field
    } else {
        delete _state.requestsOverflow[key]; // overflow path: zeroes ENTIRE struct
    }
}
``` [4](#0-3) 

For the **overflow path**, `delete _state.requestsOverflow[key]` zeroes every field of the struct in storage. Because `req` is a storage pointer to that same slot, subsequent reads of `req.requester` return `address(0)` and `req.callbackGasLimit` returns `0`.

4. The callback is then dispatched:

```solidity
try
    IEchoConsumer(req.requester)._echoCallback{   // address(0)
        gas: req.callbackGasLimit                 // 0
    }(sequenceNumber, priceFeeds)
``` [5](#0-4) 

The call to `address(0)` with 0 gas silently fails, the `catch` branch fires, and `PriceUpdateCallbackFailed` is emitted with `address(0)` as the requester. The actual requester never receives their price feeds.

By contrast, for requests in the primary array, `clearRequest` only zeroes `sequenceNumber`, leaving `requester` and `callbackGasLimit` intact — so those callbacks succeed.

### Impact Explanation

Any user whose request lands in `requestsOverflow` pays the full fee but **never receives their price-update callback**. The provider is paid in full. The user's funds are effectively lost with no recourse, since the request is cleared and cannot be re-executed.

### Likelihood Explanation

The `requests` array has only 32 slots and uses a 5-bit hash (`shortKey = uint8(hash[0] & 0x1f)`). With 32 concurrently active requests, the next request is **guaranteed** to collide (pigeonhole). An unprivileged attacker can deliberately fill all 32 slots with their own requests, then make one more request that collides with a victim's slot, evicting the victim to overflow. The attacker then calls `executeCallback` for the victim's sequence number, triggering the bug. No privileged access is required; `executeCallback` is `external` and callable by anyone. [6](#0-5) 

### Recommendation

Read all necessary fields from `req` into local memory variables **before** calling `clearRequest`, then use those local copies for the callback:

```solidity
address requester = req.requester;
uint32 gasLimit   = req.callbackGasLimit;
clearRequest(sequenceNumber);
try IEchoConsumer(requester)._echoCallback{gas: gasLimit}(sequenceNumber, priceFeeds) {
    ...
}
```

Alternatively, align `clearRequest` so the overflow path also preserves non-sequenceNumber fields (e.g., only zero `sequenceNumber` in both branches), matching the behavior of the primary-array path.

### Proof of Concept

1. Deploy `Echo` with `NUM_REQUESTS = 32`.
2. Attacker submits 32 requests, filling all array slots.
3. Victim submits request with `sequenceNumber = V`; its `shortKey` collides with one of the attacker's slots, so the victim's request is evicted to `requestsOverflow`.
4. Attacker calls `executeCallback(provider, V, updateData, priceIds)`.
5. Inside `executeCallback`:
   - `findActiveRequest(V)` returns a storage pointer into `requestsOverflow`.
   - Provider fee is credited from `req.fee` (correct).
   - `clearRequest(V)` executes `delete _state.requestsOverflow[key]` → all fields zeroed.
   - `req.requester == address(0)`, `req.callbackGasLimit == 0`.
   - `IEchoConsumer(address(0))._echoCallback{gas: 0}(...)` fails silently.
   - `PriceUpdateCallbackFailed` is emitted with `requester = address(0)`.
6. Victim's callback is never delivered; victim's fee is gone; provider is paid. [7](#0-6)
