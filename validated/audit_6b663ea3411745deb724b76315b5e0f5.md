### Title
`ScriptHashTypeVerifier` Omits Type-Script Hash-Type Validation, Allowing Deferred Rejection — (`verification/src/transaction_verifier.rs`)

### Summary

`ScriptHashTypeVerifier::verify` iterates over transaction outputs and validates only the **lock** script's `hash_type` against `ENABLED_SCRIPT_HASH_TYPE`. It never inspects the **type** script's `hash_type`. A transaction whose outputs carry a valid lock `hash_type` but an invalid type `hash_type` (e.g., `6` / `Data3`) passes `NonContextualTransactionVerifier` and is deferred to contextual verification before being rejected.

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is defined as: [1](#0-0) 

```
{ 0 (Data), 1 (Type), 2 (Data1), 4 (Data2) }
```

`ScriptHashTypeVerifier::verify` checks only `output.lock().hash_type()`: [2](#0-1) 

The loop body calls `output.lock().hash_type()` exclusively. There is no corresponding call to `output.type_().hash_type()`. The struct comment at line 70 even documents this limitation explicitly: *"Check whether output lock hash type within enabled range"*. [3](#0-2) 

`NonContextualTransactionVerifier::verify` calls `self.script_hash_type.verify()` as its final step and returns `Ok(())` for such a transaction: [4](#0-3) 

### Impact Explanation

A transaction with `lock.hash_type ∈ {0,1,2,4}` and `type.hash_type = 6` passes the non-contextual gate and enters `ContextualTransactionVerifier::verify`, which runs `time_relative.verify()`, `capacity.verify()`, and then `script.verify(max_cycles)` (the `TransactionScriptsVerifier`). The script verifier must resolve the type script group before it can reject the transaction. This is wasted work per relayed transaction. [5](#0-4) 

### Likelihood Explanation

An unprivileged attacker who controls any live cell can craft such transactions at will. The only barrier is owning live cells (standard for any tx-pool participant). The tx-pool does apply rate limiting, but the check gap means each crafted transaction consumes more node resources than it should before rejection.

### Recommendation

Extend `ScriptHashTypeVerifier::verify` to also validate the type script's `hash_type` when a type script is present:

```rust
// After the existing lock check, inside the output loop:
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }
    } else {
        return Err(TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }.into());
    }
}
```

### Proof of Concept

Build a `TransactionView` with one output whose `lock.hash_type = 0` (Data) and `type.hash_type = 6` (unknown/Data3). Call `ScriptHashTypeVerifier::new(&tx).verify()` — it returns `Ok(())`. Then call `NonContextualTransactionVerifier::new(&tx, &consensus).verify()` — it also returns `Ok(())`, confirming the gap. The transaction would only be rejected inside `ContextualTransactionVerifier::verify` when `TransactionScriptsVerifier` attempts to resolve the type script group. [6](#0-5)

pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}
