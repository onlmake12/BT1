### Title
`ScriptHashTypeVerifier` Validates Only Lock Script `hash_type`, Omitting Type Script Validation — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and checks only the **lock script's** `hash_type` against `ENABLED_SCRIPT_HASH_TYPE`. The **type script's** `hash_type` is never checked. This means a transaction carrying an output whose type script uses an invalid or not-yet-activated `hash_type` (e.g., `Data3 = 6` before the corresponding hardfork, or a raw byte value such as `3` that maps to no valid variant) passes the non-contextual verifier and is admitted to the tx-pool, where it consumes resources before being rejected during the more expensive full script-execution phase.

---

### Finding Description

`NonContextualTransactionVerifier` is the first gate applied to every submitted transaction. It bundles several cheap, context-free checks, one of which is `ScriptHashTypeVerifier`:

```
verification/src/transaction_verifier.rs  lines 796–814
```

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

The loop body calls `output.lock().hash_type()` exclusively. `output.type_()` — the optional type script — is never inspected. The struct comment says *"Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules"*, but only half of each output (the lock) is actually verified.

The analogous gap exists in `CellbaseVerifier` (lines 135–144 of `verification/src/block_verifier.rs`), which also checks only `output.lock().hash_type()`, though cellbase outputs are already required to have no type script, so that instance is harmless.

`ScriptHashType` is defined with a macro-generated set of variants (`Data=0`, `Type=1`, `Data1=2`, `Data2=4`, `DataN = N<<1` for N ≥ 3):

```
util/gen-types/src/core.rs  lines 9–32
```

`ENABLED_SCRIPT_HASH_TYPE` (defined in `util/constant/src/consensus.rs`) contains only the currently activated subset. A type script carrying `hash_type = 6` (`Data3`, not yet activated) or `hash_type = 3` (no valid variant at all) will:

1. **Pass** `ScriptHashTypeVerifier` — the check never touches `output.type_()`.
2. **Reach** `TxInfo::select_version()` / `extract_script_and_dep_index()` during full script execution, where `ScriptHashType::try_from(hash_type)` returns `Err` or the `hash_type =>` arm fires:

3. **Be rejected** only at that late, cycle-consuming stage.

The mismatch is structurally identical to the reference report: one code path (the adapter / the verifier) applies a transformation or check to one field, while a parallel field that should receive the same treatment is silently skipped, causing the two paths to diverge.

---

### Impact Explanation

Any unprivileged RPC caller (`send_transaction`) can craft a transaction whose outputs carry a type script with an arbitrary `hash_type` byte. Such transactions:

- Bypass the cheap non-contextual gate and enter the tx-pool.
- Force the node to perform full dependency resolution and script-execution setup before the error is surfaced.
- Occupy tx-pool slots (bounded by pool limits) and consume CPU/memory during the verification pipeline.

A sustained flood of such transactions constitutes a low-cost resource-exhaustion attack against the tx-pool and the verification worker threads, degrading throughput for legitimate transactions.

---

### Likelihood Explanation

The attack requires no privileges, no keys, and no on-chain funds beyond the minimum to construct a structurally valid transaction skeleton (inputs with valid out-points, correct capacity). The malformed field is a single byte in the type script of any output. The entry point is the public `send_transaction` RPC, reachable by any peer or local caller.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's type script when one is present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        check_hash_type(output.lock().hash_type())?;

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the existing `try_into` / `ENABLED_SCRIPT_HASH_TYPE` logic. This mirrors the completeness already expected by the verifier's own documentation and makes the non-contextual gate consistent for both script fields of every output.

---

### Proof of Concept

1. Construct a transaction with one input (any live cell) and one output whose **type script** has `hash_type = 3` (an odd byte that maps to no `ScriptHashType` variant).
2. Submit via `send_transaction` RPC.
3. `NonContextualTransactionVerifier::verify()` calls `ScriptHashTypeVerifier::verify()`, which iterates outputs and checks only `output.lock().hash_type()` — the type script byte `3` is never read. The verifier returns `Ok(())`.
4. The transaction proceeds into the tx-pool and toward full verification.
5. Only when `TxInfo::extract_script_and_dep_index()` or `select_version()` is reached does `ScriptHashType::try_from(3u8)` return `Err`, producing `ScriptError::InvalidScriptHashType`.
6. Repeat at high rate to exhaust tx-pool slots and verification-thread CPU before each transaction is finally rejected.

/// Basic checks that don't depend on any context
/// Contains:

    pub(crate) size: SizeVerifier<'a>,
    pub(crate) empty: EmptyVerifier<'a>,
    pub(crate) duplicate_deps: DuplicateDepsVerifier<'a>,

            version: VersionVerifier::new(tx, consensus.tx_version()),
            size: SizeVerifier::new(tx, consensus.max_block_bytes()),
            empty: EmptyVerifier::new(tx),
            duplicate_deps: DuplicateDepsVerifier::new(tx),

            script_hash_type: ScriptHashTypeVerifier::new(tx),

        self.empty.verify()?;
        self.duplicate_deps.verify()?;
        self.outputs_data_verifier.verify()?;
        self.script_hash_type.verify()?;
        Ok(())
    }

    /// Specifies how the script `code_hash` is used to match the script code and how to run the code.
    /// The hash type is split into the high 7 bits and the low 1 bit,
    /// when the low 1 bit is 1, it indicates the type,
    /// when the low 1 bit is 0, it indicates the data,
    /// and then it relies on the high 7 bits to indicate
    /// that the data actually corresponds to the version.
     #[derive(Default, Clone, Copy, PartialEq, Eq, Debug, Hash, FromRepr)]
     #[repr(u8)]
    pub enum ScriptHashType {
        /// Type "type" matches script code via cell type script hash.
        Type = 1,
        /// Type "data" matches script code via cell data hash, and run the script code in v0 CKB VM.
        #[default]
        Data = 0,

        Data1 = 2,
