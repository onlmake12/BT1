### Title
`getPricesNoOlderThan` Staleness Check Uses Maximum Subscription Timestamp Instead of Per-Feed `publishTime`, Bypassing Freshness Guarantee — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions in the Pyth Pulse `Scheduler` contract validate staleness against `status.priceLastUpdatedAt`, which is the **maximum** `publishTime` across all feeds in a subscription. Individual feeds with much older `publishTime` values (e.g., closed-market equities) pass this check and are returned to callers who believe they have a freshness guarantee.

---

### Finding Description

In `Scheduler.sol`, `updatePriceFeeds` computes `priceLastUpdatedAt` as the maximum `publishTime` across all feeds in the batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [1](#0-0) 

The past-validity check is also applied only to this maximum timestamp, not to each individual feed:

```solidity
if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(...);
}
``` [2](#0-1) 

The `parsePriceFeedUpdatesWithConfig` call uses `minAllowedPublishTime = 0`, explicitly accepting any per-feed timestamp:

```solidity
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
    ...
);
``` [3](#0-2) 

When a reader calls `getPricesNoOlderThan`, the staleness check compares `block.timestamp` against `status.priceLastUpdatedAt` (the maximum), not against each returned feed's `publishTime`:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [4](#0-3) 

The same flaw exists in `getEmaPricesNoOlderThan`: [5](#0-4) 

The `IScheduler` interface documents `getPricesNoOlderThan` as returning prices "no older than `age` seconds," which is the guarantee that is violated: [6](#0-5) 

---

### Impact Explanation

A protocol integrating with Pyth Pulse and calling `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)` expects a price no older than 60 seconds. Due to this bug, it can receive a price whose `publishTime` is days old (e.g., from a closed equity market's last trading session), while the staleness check passes because a co-subscribed crypto feed had a recent timestamp. This is the same class of impact as M-1: stale prices used in lending/derivative protocols can cause incorrect liquidations or fund theft.

---

### Likelihood Explanation

Any permissionless keeper (unprivileged updater) can call `updatePriceFeeds`. A subscription containing both always-active feeds (crypto) and market-hours feeds (equities, commodities) is a natural and documented use case — the contract's own comments acknowledge closed-market feeds will carry stale timestamps. On weekends or outside trading hours, a keeper can submit a batch where the crypto feed is fresh and the equity feed is days old. The maximum-timestamp check passes, `priceLastUpdatedAt` is set to the crypto feed's recent time, and `getPricesNoOlderThan` returns the stale equity price without reverting.

---

### Recommendation

In `getPricesNoOlderThan` and `getEmaPricesNoOlderThan`, validate staleness against each individual returned feed's `publishTime`, not against the subscription-level `priceLastUpdatedAt`:

```solidity
for (uint i = 0; i < prices.length; i++) {
    if (distance(block.timestamp, prices[i].publishTime) > age_seconds)
        revert PythErrors.StalePrice();
}
```

Alternatively, store a per-feed `lastUpdatedAt` timestamp separately from the feed's `publishTime`, and check that per-feed timestamp in the staleness guard.

---

### Proof of Concept

1. A subscription is created with two price IDs: `BTC_USD` (crypto, always active) and `AAPL_USD` (equity, closed on weekends).
2. On Saturday, a keeper calls `updatePriceFeeds` with update data containing:
   - `BTC_USD`: `publishTime = block.timestamp` (fresh)
   - `AAPL_USD`: `publishTime = block.timestamp - 2 days` (Friday close)
3. In `_validateShouldUpdatePrices`, `updateTimestamp = max(now, now - 2 days) = now`. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` (1 hour) check passes. `priceLastUpdatedAt = now`.
4. A lending protocol calls `getPricesNoOlderThan(subscriptionId, [AAPL_USD_ID], 60)`.
5. The check `distance(block.timestamp, priceLastUpdatedAt) = 0 ≤ 60` passes.
6. `getPricesUnsafe` returns the stored AAPL price with `publishTime = 2 days ago`.
7. The lending protocol uses a 2-day-old AAPL price, enabling forced liquidations or collateral mispricing if AAPL moved significantly since Friday's close. [7](#0-6) [8](#0-7)
