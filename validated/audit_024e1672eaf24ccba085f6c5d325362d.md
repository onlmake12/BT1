### Title
`ProposalShortId` (10-byte tx_hash truncation) Used as Sole Deduplication Key in tx-pool — Witness-Differing Transactions Permanently Blocked from Pool Admission - (File: `tx-pool/src/util.rs`)

---

### Summary

CKB's tx-pool uses `ProposalShortId` — the first 10 bytes of `tx_hash` — as the sole cardinality key for all pool entries. Because `tx_hash` is computed over `RawTransaction` (inputs, outputs, cell_deps, header_deps, version) and **explicitly excludes witnesses**, two structurally distinct transactions sharing identical raw content but carrying different witnesses produce the same `ProposalShortId`. The pool's `check_txid_collision` guard rejects any incoming transaction whose `ProposalShortId` is already present, regardless of whether the full `witness_hash` differs. This mirrors the LineaRollup pattern exactly: a truncated/partial hash is used as the unique cardinality for stored data, while the omitted fields (witnesses in CKB; nonce/signature in Linea) are not constrained to be unique, allowing a first-submitted entry to permanently block a second, legitimately distinct entry.

---

### Finding Description

**`ProposalShortId` construction — 10-byte truncation of `tx_hash`:**

`ProposalShortId::from_tx_hash` copies only the first 10 bytes of the 32-byte blake2b hash:

```rust
// util/gen-types/src/extension/shortcut.rs
pub fn from_tx_hash(h: &packed::Byte32) -> Self {
    let mut inner = [0u8; 10];
    inner.copy_from_slice(&h.as_slice()[..10]);
    inner.into()
}
``` [1](#0-0) 

**`tx_hash` excludes witnesses — `witness_hash` is a separate field:**

The `tx_hash` is computed from `RawTransaction` only; witnesses are hashed separately into `witness_hash`:

```rust
// util/gen-types/src/extension/calc_hash.rs
pub fn calc_tx_hash(&self) -> packed::Byte32 { self.raw().calc_tx_hash() }
pub fn calc_witness_hash(&self) -> packed::Byte32 { self.calc_hash() }
``` [2](#0-1) 

**Pool deduplication uses only `ProposalShortId`:**

`check_txid_collision` rejects any transaction whose `ProposalShortId` is already present in the pool. The comment incorrectly claims this implies identical witnesses:

```rust
// tx-pool/src/util.rs
pub(crate) fn check_txid_collision(tx_pool: &TxPool, tx: &TransactionView) -> Result<(), Reject> {
    let short_id = tx.proposal_short_id();
    if tx_pool.contains_proposal_id(&short_id) {
        return Err(Reject::Duplicated(tx.hash()));
    }
    Ok(())
}
``` [3](#0-2) 

The comment in `pre_check` that calls this function states:

> `// Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.`

This is factually wrong — `tx_hash` (and thus `ProposalShortId`) does **not** cover witnesses. [4](#0-3) 

**`pool_map` is keyed exclusively by `ProposalShortId`:**

```rust

### Impact Explanation

**Scenario A — Same raw content, different witnesses (scripts accepting multiple valid witnesses):**

For lock scripts that accept more than one valid witness (always-success scripts, threshold multisig where any valid key suffices, or any custom script that does not bind the witness to a unique signer), an attacker who observes a victim's pending transaction can:

1. Construct `T_attacker` with identical inputs/outputs/cell_deps but different witnesses.
2. Submit `T_attacker` via `send_transaction` RPC before `T_victim` reaches the pool.
3. `T_victim` is permanently rejected with `PoolRejectedDuplicatedTransaction` — it can never enter the pool as long as `T_attacker` occupies the `ProposalShortId` slot.

The victim's only recourse is to change the raw transaction content (e.g., add a dummy output), which changes the `ProposalShortId` — but this may not always be possible (e.g., the inputs are time-sensitive or the outputs are protocol-constrained).

**Scenario B — Birthday attack on 80-bit `ProposalShortId` space:**

An attacker controlling their own UTXOs can generate ~2^40 candidate transactions (birthday bound on 80 bits) to find one whose `ProposalShortId` collides with a target victim's transaction. Once submitted, the victim's transaction is permanently blocked. This is computationally expensive but within reach of a well-resourced adversary.

**Secondary impact — `committed_txs_hash_cache` corruption:**

The cache `LruCache<ProposalShortId, Byte32>` maps short IDs to committed tx hashes. A `ProposalShortId` collision between two committed transactions in different blocks silently overwrites the cache entry, causing compact block reconstruction to retrieve the wrong transaction for a given short ID. [9](#0-8) 

---

### Likelihood Explanation

- **Scenario A** is realistic wherever non-standard or always-success lock scripts are used (common in testing, layer-2 bridges, and custom dApps). The attacker only needs to observe the victim's transaction (available via P2P relay or mempool RPC) and submit a competing one slightly faster. No cryptographic work is required.
- **Scenario B** requires ~2^40 hash computations, feasible for a state-level attacker targeting a high-value transaction.
- The behavior is confirmed by an existing integration test, meaning it is a reproducible, deterministic outcome — not a race condition in the traditional sense.

---

### Recommendation

1. **Use the full `tx_hash` (or `witness_hash`) as the pool deduplication key**, not the 10-byte `ProposalShortId`. The `ProposalShortId` should remain as the proposal/compact-block identifier, but pool admission should check the full 32-byte hash.
2. **Correct the misleading comment** in `pre_check` that claims `ProposalShortId` equality implies witness equality.
3. **Consider using `witness_hash`** (which covers the full transaction including witnesses) as the pool's primary identity key, so that two transactions with the same raw content but different witnesses are treated as distinct entries.
4. For the `committed_txs_hash_cache`, store the full `tx_hash` as the key rather than the truncated `ProposalShortId`.

fn cousin_txs_with_same_hash_different_witness_hash(node: &Node) -> (TransactionView, TransactionView) {
