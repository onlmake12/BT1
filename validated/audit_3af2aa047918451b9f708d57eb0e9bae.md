### Title
Wrong File Opened During Multi-Segment Crash Recovery in Freezer Repair Loop — (`freezer/src/freezer_files.rs`)

---

### Summary

In `FreezerFilesBuilder::build()`, the crash-recovery repair loop that re-synchronises the head data file with the index file contains a stale-variable bug. When the loop detects it must "slip back" to an earlier data-file segment, it opens the **old** head file (using `head_index.file_id`) instead of the **new** earlier file (using `new_index.file_id`). The node therefore initialises the freezer with the wrong file handle, causing subsequent cold-archive writes to corrupt an already-finalised data segment.

---

### Finding Description

`FreezerFiles` is CKB's cold block archive. It stores finalised blocks across a chain of numbered data files (`blk000000`, `blk000001`, …) plus a single `INDEX` file whose entries each record `(file_id, end_offset)` for every frozen block.

On startup, `FreezerFilesBuilder::build()` runs a repair loop to handle the case where a prior crash left the index and the head data file out of sync:

```
freezer/src/freezer_files.rs  lines 430-465
``` [1](#0-0) 

The relevant branch inside the loop is:

```rust
// truncate the index to matching the head file
if expect_head_size > head_size {
    helper::truncate_file(&mut index, index_size - INDEX_ENTRY_SIZE)?;
    index_size -= INDEX_ENTRY_SIZE;

    index.seek(SeekFrom::Start(index_size - INDEX_ENTRY_SIZE))?;
    index.read_exact(&mut buffer)?;
    let new_index = IndexEntry::decode(&buffer)?;   // ← the new "last" entry

    // slipped back into an earlier head-file
    if new_index.file_id != head_index.file_id {
        let head_file_name = helper::file_name(head_index.file_id); // ← BUG
        let (new_head, size) = self.open_append(self.file_path.join(head_file_name))?;
        head = new_head;
        head_size = size;
    }
    expect_head_size = new_index.offset;
    head_index = new_index;   // ← head_index updated AFTER the open
}
``` [2](#0-1) 

The condition `new_index.file_id != head_index.file_id` fires precisely when the repair must switch to an **earlier** segment file. At that point `new_index.file_id` is the correct earlier file, but the code calls `helper::file_name(head_index.file_id)` — the **old** (current) head file — and opens that instead. `head_index` is only updated to `new_index` on line 464, **after** the wrong file has already been opened.

The identical logic in `FreezerFiles::truncate()` performs the same cross-file switch correctly, using `new_index.file_id`:

```rust
if new_index.file_id != self.head_id {
    self.release(new_index.file_id);
    let (new_head_file, offset) = self.open_append(new_index.file_id)?; // correct
    ...
    self.head_id = new_index.file_id;
    self.head = Head::new(new_head_file, offset);
}
``` [3](#0-2) 

---

### Impact Explanation

After the repair loop exits, `FreezerFiles` is constructed with `head` pointing to the **wrong** (old) data file: [4](#0-3) 

Every subsequent `append()` call writes new frozen blocks into that already-finalised segment file, overwriting existing block data at arbitrary offsets. `retrieve()` then reads back corrupted bytes for any block whose data was overwritten. Because the index still records the correct `(file_id, offset)` pairs for the old blocks, the corruption is silent — no error is returned, but the wrong bytes are served.

Concretely:
- Historical block bodies stored in the cold archive become unreadable or return garbage.
- A node that has been through crash-recovery will silently diverge from the canonical chain data it previously stored.
- Peers syncing from this node receive corrupted block data.

---

### Likelihood Explanation

The trigger condition requires two things to coincide:

1. **A prior crash** that left the freezer index ahead of the head data file (`expect_head_size > head_size`). This is a normal outcome of any unclean shutdown (power loss, OOM kill, SIGKILL) that interrupted a `head.write()` + `write_index()` pair.
2. **The crash occurred on a block that was the first entry in a new data segment** (i.e., the last valid index entry belongs to the previous segment file, so `new_index.file_id != head_index.file_id`). This happens whenever the node was killed immediately after `open_truncated()` created a new segment but before any data was flushed.

Both conditions are realistic in production. Unclean shutdowns are common, and the segment-boundary case occurs once every ~2 GB of frozen data (the default `MAX_FILE_SIZE`).

---

### Recommendation

Replace `head_index.file_id` with `new_index.file_id` on the buggy line:

```rust
// Before (buggy):
let head_file_name = helper::file_name(head_index.file_id);

// After (correct):
let head_file_name = helper::file_name(new_index.file_id);
``` [5](#0-4) 

This mirrors the already-correct pattern used in `truncate()`.

---

### Proof of Concept

1. Run a CKB node until the freezer has accumulated at least two data segments (≥ 2 GB of cold blocks), so `head_id ≥ 1`.
2. Kill the node with `SIGKILL` immediately after it calls `open_truncated()` to create segment `blk000002` but before any block data is written to it. At this point the index has one entry pointing to `file_id=2, offset=N` but `blk000002` is empty (`head_size = 0`), so `expect_head_size (= N) > head_size (= 0)`.
3. Restart the node. The repair loop enters the `expect_head_size > head_size` branch, removes the dangling index entry, and reads the new last entry which belongs to `file_id=1`. The condition `new_index.file_id (1) != head_index.file_id (2)` is true.
4. Due to the bug, `helper::file_name(head_index.file_id)` = `"blk000002"` is opened (the empty new file) instead of `"blk000001"`. `head` now points to `blk000002` with `head_size = 0`.
5. The loop exits (sizes now match). `FreezerFiles` is built with `head → blk000002`.
6. The next `append()` writes block data into `blk000002` starting at offset 0, but the index records it under `file_id=1` offsets — reads for those blocks now return data from the wrong file. [6](#0-5)
