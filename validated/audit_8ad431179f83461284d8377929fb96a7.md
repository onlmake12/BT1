### Title
`setFeeManager()` Replaces Fee Manager Without Withdrawing Accrued Provider Fees — (`Entropy.sol` / `Echo.sol`)

### Summary

Both `Entropy.sol` and `Echo.sol` contain a `setFeeManager()` function that atomically replaces the old fee manager address with a new one. If the provider calls `setFeeManager(newManager)` while `accruedFeesInWei > 0`, the old fee manager immediately loses its only withdrawal path (`withdrawAsFeeManager`), and those fees become inaccessible unless the provider itself withdraws them via `withdraw()`. This is a direct analog to the LendingPool vault-swap bug: a critical address is swapped without first draining the balance that was exclusively accessible through that address.

---

### Finding Description

In `Entropy.sol`, `setFeeManager()` unconditionally overwrites `provider.feeManager`:

```solidity
// Entropy.sol L876-L893
function setFeeManager(address manager) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) {
        revert EntropyErrors.NoSuchProvider();
    }
    address oldFeeManager = provider.feeManager;
    provider.feeManager = manager;          // ← old manager loses access instantly
    emit ProviderFeeManagerUpdated(...);
}
```

The only way for the fee manager to withdraw is `withdrawAsFeeManager`, which hard-checks `providerInfo.feeManager != msg.sender`:

```solidity
// Entropy.sol L187-L188
if (providerInfo.feeManager != msg.sender) {
    revert EntropyErrors.Unauthorized();
}
```

Once `setFeeManager` is called, the old fee manager address can never call `withdrawAsFeeManager` again. The provider itself can still call `withdraw()`, but `withdraw()` sends funds to `msg.sender` (the provider address), not to the fee manager. If the fee manager is a separate smart contract (e.g., a multisig or a fee-splitting contract) that was the intended recipient of those fees, those funds are effectively redirected away from it without its consent.

The same pattern exists verbatim in `Echo.sol`:

```solidity
// Echo.sol L350-L358
function setFeeManager(address manager) external override {
    require(_state.providers[msg.sender].isRegistered, "Provider not registered");
    address oldFeeManager = _state.providers[msg.sender].feeManager;
    _state.providers[msg.sender].feeManager = manager;   // ← no prior withdrawal
    emit FeeManagerUpdated(msg.sender, oldFeeManager, manager);
}
```

The `IEntropy` interface documents this explicitly as a known design choice ("calling this function again with a new value will override the previous value"), but it does not document that the old fee manager loses access to already-accrued fees.

---

### Impact Explanation

- **Stranded / misdirected fees**: Any `accruedFeesInWei` that was earned while the old fee manager was active can no longer be withdrawn by that fee manager. The provider can still call `withdraw()`, but the ETH goes to the provider address, not the fee manager contract.
- **Fee manager contract insolvency**: If the fee manager is a smart contract (e.g., a revenue-sharing contract) that relies on `withdrawAsFeeManager` to distribute funds to downstream beneficiaries, those beneficiaries permanently lose their share of already-accrued fees.
- **Griefing by provider**: A malicious or negligent provider can rotate the fee manager at any time, denying the old fee manager its earned fees. The fee manager has no on-chain recourse.

---

### Likelihood Explanation

- `setFeeManager` is callable by any registered Entropy/Echo provider — no privileged role required beyond being a registered provider.
- The Fortuna keeper (`setup_provider.rs`) calls `sync_fee_manager` automatically on every setup run, meaning fee manager rotation is a routine operational action.
- Any provider that rotates their fee manager (e.g., upgrading a multisig, changing a revenue-split contract) while fees are accrued will trigger this silently.

---

### Recommendation

Before overwriting `provider.feeManager`, the contract should either:
1. Require that `accruedFeesInWei == 0` before allowing rotation, or
2. Automatically transfer `accruedFeesInWei` to the old fee manager before replacing it:

```diff
function setFeeManager(address manager) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();

+   // Drain accrued fees to old fee manager before rotating
+   address oldFeeManager = provider.feeManager;
+   if (oldFeeManager != address(0) && provider.accruedFeesInWei > 0) {
+       uint128 pending = provider.accruedFeesInWei;
+       provider.accruedFeesInWei = 0;
+       (bool sent, ) = oldFeeManager.call{value: pending}("");
+       require(sent, "fee drain to old manager failed");
+   }

-   address oldFeeManager = provider.feeManager;
    provider.feeManager = manager;
    emit ProviderFeeManagerUpdated(msg.sender, oldFeeManager, manager, bytes(""));
}
```

---

### Proof of Concept

1. Provider registers with Entropy. Sets `feeManager = contractA` (a revenue-split contract).
2. Users make randomness requests; `provider.accruedFeesInWei` grows to 10 ETH.
3. Provider calls `setFeeManager(contractB)` — no withdrawal step occurs.
4. `contractA` calls `withdrawAsFeeManager(provider, 10 ether)` → reverts with `Unauthorized` because `provider.feeManager == contractB` now.
5. Provider calls `withdraw(10 ether)` → 10 ETH goes to the provider address, not `contractA`.
6. `contractA`'s downstream beneficiaries receive nothing for the 10 ETH they were owed.
