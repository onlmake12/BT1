### Title
Minimum Fee Rate Admission Check Uses Raw Byte Size Instead of Transaction Weight, Allowing Cycle-Heavy Transactions to Bypass `min_fee_rate` Enforcement — (`tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee by passing the raw serialized byte size (`tx_size`) directly to `FeeRate::fee()`. However, `FeeRate` is defined as **shannons per kilo-weight**, where `weight = max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions, `weight >> tx_size`, so the minimum fee threshold is computed in the wrong unit. An unprivileged tx-pool submitter can craft a transaction with small serialized size but near-maximum cycle consumption, pay a fee that satisfies the size-based check, and be admitted to the pool with an actual fee rate orders of magnitude below `min_fee_rate`.

---

### Finding Description

`FeeRate` is explicitly documented as **shannons per kilo-weight**:

```rust
// util/types/src/core/fee_rate.rs
/// shannons per kilo-weight
pub struct FeeRate(pub u64);

pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
``` [1](#0-0) 

Transaction weight is defined as:

```rust
// util/types/src/core/tx_pool.rs
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

The admission check in `check_tx_fee` passes raw byte size as the weight argument:

```rust
// tx-pool/src/util.rs
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [3](#0-2) 

The comment acknowledges the mismatch but treats it as an acceptable approximation. In practice, `check_tx_fee` is the **only** fee-rate gate in the admission path — `verify_rtx` checks cycle limits but not fee rate. There is no subsequent weight-based fee rate re-check before the transaction is admitted.

**Concrete arithmetic with default parameters:**

| Parameter | Value |
|---|---|
| `min_fee_rate` | 1,000 shannons/KW |
| `DEFAULT_BYTES_PER_CYCLES` | 0.000_170_571_4 |
| `max_tx_verify_cycles` | 70,000,000 |
| Max cycle-equivalent weight | 70,000,000 × 0.000_170_571_4 ≈ **11,940 bytes** |

A transaction with `tx_size = 100 bytes` and `cycles = 69,999,999`:
- `weight = max(100, 11,940) = 11,940`
- `min_fee` (size-based, as coded) = `1,000 × 100 / 1,000 = 100 shannons`
- Actual fee rate = `100 / 11,940 × 1,000 ≈ 8.4 shannons/KW`
- Required minimum = `1,000 shannons/KW`

The transaction is admitted with an actual fee rate **~119× below** `min_fee_rate`. [4](#0-3) 

---

### Impact Explanation

An unprivileged tx-pool submitter can flood the mempool with cycle-heavy, low-fee transactions that pass the admission check. These transactions:

1. Consume pool capacity (`max_tx_pool_size = 180 MB`), displacing legitimate transactions.
2. Are relayed to peers via the relay protocol, propagating the spam network-wide.
3. Consume node CPU during script execution (`verify_rtx`) before the cycle limit rejects them — or, if cycles stay just below `max_tx_verify_cycles`, they are fully admitted and persist in the pool.
4. Degrade miner block assembly quality, since the pool contains transactions with actual fee rates far below the configured floor.

The pool eviction mechanism eventually removes low-fee-rate entries when the pool is full, but the attacker can continuously re-submit, maintaining pool pressure at negligible cost.

---

### Likelihood Explanation

The attack requires only:
- Access to the `send_transaction` RPC (default-enabled, no authentication).
- A script that consumes near-maximum cycles with minimal code size (e.g., a tight computation loop in a lock script). This is straightforward to construct using CKB-VM RISC-V.
- A fee of just `min_fee_rate × tx_size / 1,000` shannons — for a 100-byte transaction at the default rate, that is 100 shannons (≈ 0.000001 CKB), making sustained attacks economically trivial.

No privileged access, no majority hashpower, and no social engineering is required.

---

### Recommendation

Replace the size-based minimum fee computation in `check_tx_fee` with the actual transaction weight. Since cycles are not yet known at the point `check_tx_fee` is called (script execution happens in `verify_rtx`), the declared cycle count from the transaction should be used:

```rust
// Use declared cycles for the weight-based min-fee check
let declared_cycles = tx.cycles().unwrap_or(0);
let weight = get_transaction_weight(tx_size, declared_cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

If declared cycles are unavailable at this stage, the check should be deferred until after `verify_rtx` returns the actual cycle count, and the fee rate should be re-validated against the weight computed from actual cycles. This mirrors the correct pattern used in `TxEntry::fee_rate()` and `FeeRateCollector::statistics()`, both of which compute weight from both size and cycles before deriving a fee rate. [5](#0-4) [6](#0-5) 

---

### Proof of Concept

1. Construct a CKB transaction whose lock script executes a tight RISC-V loop consuming ~69,000,000 cycles. The script binary itself can be stored in a cell dep, keeping the transaction's serialized size small (e.g., ~200 bytes).
2. Set the transaction fee to `ceil(min_fee_rate × tx_size / 1000)` = `ceil(1000 × 200 / 1000)` = 200 shannons.
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons` — the transaction passes.
5. `verify_rtx` executes the script; cycles ≈ 69,000,000 < 70,000,000 limit — the transaction passes.
6. The transaction is admitted with actual fee rate = `200 / max(200, 69,000,000 × 0.000_170_571_4) × 1000 = 200 / 11,769 × 1000 ≈ 17 shannons/KW`, far below the 1,000 shannons/KW floor.
7. Repeat to fill the pool with such transactions at negligible cost. [7](#0-6) [8](#0-7)

use crate::core::Capacity;

/// shannons per kilo-weight
#[derive(Clone, Copy, Default, Debug, PartialEq, Eq, PartialOrd, Ord)]
pub struct FeeRate(pub u64);

const KW: u64 = 1000;

impl FeeRate {
    /// Calculates the fee rate from a total fee and weight.
    pub fn calculate(fee: Capacity, weight: u64) -> Self {
        if weight == 0 {
            return FeeRate::zero();
        }
        FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
    }

    /// Creates a fee rate from shannons per kilo-weight.
    pub const fn from_u64(fee_per_kw: u64) -> Self {
        FeeRate(fee_per_kw)
    }

    /// Returns the fee rate as shannons per kilo-weight.
    pub const fn as_u64(self) -> u64 {
        self.0
    }

    /// Creates a zero fee rate.
    pub const fn zero() -> Self {
        Self::from_u64(0)
    }

    /// Calculates the fee for a given weight.
    pub fn fee(self, weight: u64) -> Capacity {
        let fee = self.0.saturating_mul(weight) / KW;
        Capacity::shannons(fee)
    }

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
