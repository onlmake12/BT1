### Title
`ScriptHashTypeVerifier` Omits Type Script Hash Type Validation, Allowing Non-Permitted Hash Types into Tx-Pool — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

The `ScriptHashTypeVerifier` inside `NonContextualTransactionVerifier` enforces that output **lock** scripts use only hash types permitted by the current consensus (`ENABLED_SCRIPT_HASH_TYPE`), but performs **no equivalent check on output type scripts**. An unprivileged transaction sender can craft a transaction whose output carries a type script with a non-permitted hash type (e.g., `Data2` before the CKB2023 hardfork), pass non-contextual verification entirely, and have the transaction admitted to the tx-pool. The transaction is only rejected later during contextual (script) verification, wasting node resources and polluting the pool — a direct structural analog to the reported pattern of a product passing a whitelist gate while a required sub-component (the collateral) is never validated at that gate.

---

### Finding Description

### Impact Explanation

A transaction with a consensus-disabled type script hash type passes the non-contextual gate and enters the tx-pool. It consumes pool slots, triggers full contextual script verification (CPU-intensive), and is only evicted after that expensive step fails. An attacker who pays the minimum fee can continuously inject such transactions, polluting the pool, displacing legitimate pending transactions, and forcing every verifying worker to burn cycles on provably invalid work. During hardfork transition windows — when new hash types are being introduced — the gap is widest and the attack surface is largest.

---

### Likelihood Explanation

Crafting such a transaction requires only knowledge of the molecule encoding format and the numeric value of a non-enabled hash type. No privileged access, no key material, and no majority hashpower is needed. The attacker submits via the standard `send_transaction` RPC or the P2P relay path. The fee requirement raises the cost but does not close the gate; a modest budget sustains a sustained pool-pollution campaign.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type for every output that carries one:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        let lock_ht = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })?;
        let val: u8 = lock_ht.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_ht = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                })?;
            let val: u8 = type_ht.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Before the CKB2023 hardfork activates, `Data2` (byte value `4`) is a valid molecule enum value but is absent from `ENABLED_SCRIPT_HASH_TYPE`.
2. Construct a transaction whose single output has:
   - a standard secp256k1 lock script (lock hash type = `Type`, permitted),
   - a type script with `hash_type = 0x04` (`Data2`, not yet permitted).
3. Submit via `send_transaction` RPC (or relay over P2P).
4. `non_contextual_verify` → `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()` inspects only the lock script; the type script hash type is never checked → **returns `Ok(())`**.
5. `pre_check` resolves the transaction and checks fees → succeeds.
6. `verify_rtx` (contextual) runs `ScriptVerifier`; the VM encounters the `Data2` type script and rejects it → **returns `Err`**.
7. The transaction occupied a pool slot and consumed a full contextual verification cycle before being evicted.
8. Repeat at scale to exhaust pool capacity and verification worker bandwidth. [7](#0-6) [8](#0-7)

                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
