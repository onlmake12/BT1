### Title
Scheduler's Deviation-Based Update Condition Silently Skips Zero-Price Feeds, Breaking Core Update Functionality - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's internal price-update validation function silently skips the deviation check for any price feed whose current or previous stored price is zero. When a subscription uses deviation-only update criteria (`updateOnDeviation: true`, `updateOnHeartbeat: false`) and all subscribed feeds have a zero price value, the entire deviation loop completes without triggering an update, causing the call to revert with `UpdateConditionsNotMet`. This is structurally identical to the Perennial M-3 bug: a zero-value guard that incorrectly prevents a necessary operation from executing.

---

### Finding Description

In `Scheduler.sol`, the internal function that validates whether a price update should be accepted contains the following logic:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
``` [1](#0-0) 

This `continue` statement is inside the deviation-check loop:

```solidity
if (params.updateCriteria.updateOnDeviation) {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        PythStructs.PriceFeed storage previousFeed = _state
            .priceUpdates[subscriptionId][priceFeeds[i].id];

        if (previousFeed.id == bytes32(0)) {
            return updateTimestamp;  // first update — OK
        }

        int64 currentPrice = priceFeeds[i].price.price;
        int64 previousPrice = previousFeed.price.price;

        if (previousPrice == 0 || currentPrice == 0) {
            continue;  // BUG: skips the feed entirely
        }
        // ... deviation math ...
        if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
            return updateTimestamp;
        }
    }
}

revert SchedulerErrors.UpdateConditionsNotMet();
``` [2](#0-1) 

The `continue` is intended to avoid a division-by-zero when `previousPrice == 0`. However, it silently discards the feed from consideration rather than treating the transition as an infinite deviation. When **all** feeds in the subscription have a zero previous or current price, the loop exhausts without ever returning `updateTimestamp`, and execution falls through to `revert SchedulerErrors.UpdateConditionsNotMet()`.

Two concrete scenarios trigger this:

1. **Previous price stored as zero, new price is non-zero**: A feed's stored price is `0` (e.g., a market that was halted or a price that legitimately reached zero). A new update arrives with a non-zero price — an effectively infinite deviation — but the check is skipped.

2. **Current price is zero, previous price is non-zero**: A price crashes to zero (a catastrophic event). The deviation is infinite, yet the check is skipped and the update is rejected.

The `previousFeed.id == bytes32(0)` guard at line 420 only handles the case where no price has ever been stored. Once a price has been stored (even as `0`), the zero-price skip applies. [3](#0-2) 

---

### Impact Explanation

For any Scheduler subscription configured with `updateOnDeviation: true` and `updateOnHeartbeat: false`:

- If all subscribed price feeds have a zero stored price or a zero incoming price, `updatePriceFeeds` will always revert with `UpdateConditionsNotMet`, regardless of how large the actual price change is.
- The subscription becomes permanently stuck: no update can be pushed through the deviation path, and there is no heartbeat fallback.
- Protocols relying on the Scheduler for price feed freshness will receive stale data indefinitely, potentially enabling mispricing, bad liquidations, or missed risk signals.
- The subscription manager's deposited funds continue to be locked in a non-functional subscription.

This directly breaks the core functionality of the Scheduler for deviation-only subscriptions — an exact analog to M-3's broken `rebalanceGroup`.

---

### Likelihood Explanation

- Pyth price feeds legitimately publish a `price` field of `0` for halted markets, illiquid assets, or assets with no recent trade. This is a documented, reachable state.
- Any unprivileged updater (keeper/pusher) calling `updatePriceFeeds` on an affected subscription triggers the revert — no special access is required.
- The condition is self-sustaining: once stuck, no deviation-based update can escape the loop, and without a heartbeat criterion there is no alternative path.
- Subscriptions that track exotic or low-liquidity assets are at elevated risk.

---

### Recommendation

Replace the `continue` with a treatment of zero-price transitions as an unconditional trigger (infinite deviation), mirroring the Perennial fix pattern:

```solidity
// If either price is zero, treat as infinite deviation — always trigger update
if (previousPrice == 0 || currentPrice == 0) {
    return updateTimestamp;
}
```

This ensures that any transition to or from a zero price is treated as a maximum-deviation event and always triggers an update, which is the correct semantic. Division-by-zero is avoided because the return happens before the division.

---

### Proof of Concept

1. Deploy `Scheduler` and create a subscription with:
   - `updateOnDeviation: true`, `deviationThresholdBps: 100`
   - `updateOnHeartbeat: false`
   - Two price feed IDs, e.g. `feedA` and `feedB`

2. Call `updatePriceFeeds` with both feeds having `price = 0`. This succeeds (first update, `previousFeed.id == bytes32(0)` path), storing `price = 0` for both feeds.

3. Advance time. Call `updatePriceFeeds` again with both feeds now having `price = 1_000_000` (a massive deviation from 0).

4. Observe: the call reverts with `UpdateConditionsNotMet` because both feeds hit the `continue` branch (previousPrice == 0), the loop exhausts, and the function falls through to the revert — despite the actual deviation being infinite. [4](#0-3)
