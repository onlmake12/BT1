### Title
Minimum Fee Rate Check Ignores Cycles, Allowing High-Cycle Transactions to Bypass Fee Rate Enforcement — (File: `tx-pool/src/util.rs`)

---

### Summary

The `check_tx_fee` function computes the minimum required fee using only the transaction's serialized byte size (`tx_size`), completely ignoring the cycles consumed by scripts. Because CKB's actual transaction weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`, a transaction with high cycles but low fee passes the minimum fee rate check and enters the pool paying far less than the enforced minimum.

---

### Finding Description

In `check_tx_fee`, the minimum fee is computed as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

This uses only `tx_size` as the weight. CKB's actual transaction weight formula is `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`: [2](#0-1) 

For a high-cycle transaction, the actual weight can be significantly larger than `tx_size` alone.

`check_tx_fee` is called inside `pre_check` **before** `verify_rtx` is called to determine actual cycles: [3](#0-2) 

After `verify_rtx` returns the actual cycles, a `TxEntry` is created with the correct cycles and submitted to the pool — but **no second fee rate check using the actual weight is ever performed**: [4](#0-3) 

The code comment in `check_tx_fee` explicitly acknowledges the limitation:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [5](#0-4) 

No compensating check is performed after cycles are known. The `TxEntry.fee_rate()` method does compute the correct weight-based fee rate: [6](#0-5) 

But this is only used for sorting and eviction, not for admission enforcement.

---

### Impact Explanation

An unprivileged transaction submitter (RPC caller or P2P relayer) can craft a transaction with:

- `tx_size = 1000 bytes`
- `cycles = 70,000,000` (the `max_tx_verify_cycles` limit per `ckb.toml`)
- `fee = 1000 shannons` (just above `min_fee_rate × tx_size / 1000 = 1000`)

`check_tx_fee` computes `min_fee = 1000 × 1000 / 1000 = 1000 shannons` → **passes**.

The actual weight is `max(1000, 70,000,000 × 0.000170571) = max(1000, 11,940) = 11,940`. The correct minimum fee should be `1000 × 11,940 / 1000 = 11,940 shannons`. The transaction paid only 1,000 shannons — **~12× below the required minimum**.

The effective fee rate is `1000 × 1000 / 11,940 ≈ 84 shannons/KW`, far below the configured minimum of 1,000 shannons/KW. [7](#0-6) 

Consequences:
1. **Pool spam**: High-cycle, low-fee transactions bypass minimum fee rate enforcement and enter the pool.
2. **Verification resource waste**: `verify_rtx` actually executes the scripts (consuming real CPU cycles) before the entry is admitted — the attacker gets script execution for free relative to the fee paid.
3. **Fee rate estimation distortion**: These transactions appear in the pool with artificially low fee rates, skewing `estimate_fee_rate` results for all users.

The transaction is only evicted when the pool is full (via `limit_size`), which uses the correct weight-based fee rate. Until then it occupies pool space and has already consumed verification resources.

---

### Likelihood Explanation

Any unprivileged user can submit transactions via the `send_transaction` RPC or P2P relay. Crafting a transaction with high cycles is straightforward — any script that loops or performs expensive operations can consume up to `max_tx_verify_cycles` (70M by default). The attacker pays `min_fee_rate × tx_size` instead of `min_fee_rate × max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`, which is up to ~12× cheaper. No special privileges, keys, or majority hashpower are required.

---

### Recommendation

After `verify_rtx` returns the actual cycles, perform a second fee rate check using the actual transaction weight before calling `submit_entry`:

```rust
// In _process_tx, after verified.cycles is known:
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((
        Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate,
                               min_fee_by_weight.as_u64(),
                               fee.as_u64())),
        snapshot,
    ));
}
``` [8](#0-7) 

---

### Proof of Concept

1. Craft a transaction: `tx_size = 1000 bytes`, `cycles = 70,000,000`, `fee = 1000 shannons`.
2. Submit via `send_transaction` RPC.
3. `pre_check` → `check_tx_fee`: `min_fee = 1000 × 1000 / 1000 = 1000` → **passes** (fee == min_fee).
4. `verify_rtx` executes scripts, returns `cycles = 70,000,000`.
5. `TxEntry::new(rtx, 70_000_000, Capacity::shannons(1000), 1000)` is created.
6. `entry.fee_rate()` = `FeeRate::calculate(1000, get_transaction_weight(1000, 70_000_000))` = `1000 × 1000 / 11,940 ≈ 84 shannons/KW`.
7. Transaction enters the pool at **84 shannons/KW**, 12× below the 1,000 shannons/KW minimum, with no rejection. [9](#0-8)

/// Equal to MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES, see ckb-chain-spec.
/// The precision is set so that the difference between MAX_BLOCK_CYCLES * DEFAULT_BYTES_PER_CYCLES
/// and MAX_BLOCK_BYTES is less than 1.
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

/// vbytes has been deprecated, renamed to weight to prevent ambiguity
#[deprecated(
    since = "0.107.0",
    note = "Please use the get_transaction_weight instead"
)]
pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

    pub(crate) async fn _process_tx(
        &self,
        tx: TransactionView,
        declared_cycles: Option<Cycle>,
        command_rx: Option<&mut watch::Receiver<ChunkCommand>>,
    ) -> Option<(Result<Completed, Reject>, Arc<Snapshot>)> {
        let wtx_hash = tx.witness_hash();
        let instant = Instant::now();
        let is_sync_process = command_rx.is_none();

        let (ret, snapshot) = self.pre_check(&tx).await;

        let (tip_hash, rtx, status, fee, tx_size) = try_or_return_with_snapshot!(ret, snapshot);

        let verify_cache = self.fetch_tx_verify_cache(&tx).await;
        let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
        let tip_header = snapshot.tip_header();
        let tx_env = Arc::new(status.with_env(tip_header));

        let verified_ret = verify_rtx(
            Arc::clone(&snapshot),
            Arc::clone(&rtx),
            tx_env,
            &verify_cache,
            max_cycles,
            command_rx,
        )
        .await;

        let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

        if let Some(declared) = declared_cycles
            && declared != verified.cycles
        {
            info!(
                "process_tx declared cycles not match verified cycles, declared: {}, verified: {}, tx_hash: {}",
                declared,
                verified.cycles,
                tx.hash()
            );
            return Some((
                Err(Reject::DeclaredWrongCycles(declared, verified.cycles)),
                snapshot,
            ));
        }

        let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);

        let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
        try_or_return_with_snapshot!(ret, submit_snapshot);
