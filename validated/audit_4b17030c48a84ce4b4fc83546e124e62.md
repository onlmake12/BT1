### Title
Unbounded `constructProviderCommitment` Loop Enables Permanent DoS on Entropy Requests When `maxNumHashes = 0` — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `constructProviderCommitment` function in `Entropy.sol` contains an unbounded `while` loop that iterates `numHashes` times. When a provider has `maxNumHashes = 0` (the Solidity default, and the Fortuna keeper default when `max_num_hashes` is unconfigured), there is no cap on `req.numHashes` at request-creation time. An unprivileged user can make many sequential requests to such a provider, causing `numHashes` to grow linearly. If `numHashes` exceeds the block-gas-limit threshold (~1 million iterations at ~30 gas/keccak256 on a 30M-gas chain), the `revealWithCallback` (or `reveal`) transaction for that request will permanently exceed the block gas limit, making the request forever unfulfillable and the user's fee permanently lost.

---

### Finding Description

**Root cause — unbounded loop in `constructProviderCommitment`:** [1](#0-0) 

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
```

`numHashes` is a `uint32` field stored in each request, set at creation time as the gap between the request's assigned sequence number and the provider's `currentCommitmentSequenceNumber`: [2](#0-1) 

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

The guard is **entirely bypassed when `maxNumHashes == 0`**. The `uint32` field `maxNumHashes` in `EntropyStructsV2.ProviderInfo` defaults to `0` in Solidity: [3](#0-2) 

The Fortuna keeper also defaults to `0` when the operator does not configure `max_num_hashes`: [4](#0-3) 

```rust
sync_max_num_hashes(
    &contract,
    &provider_info,
    chain_config.max_num_hashes.unwrap_or(0),  // defaults to 0
)
```

**Call path to the loop:**

`revealWithCallback` → `revealHelper` → `constructProviderCommitment(req.numHashes, ...)`: [5](#0-4) 

`req.numHashes` is fixed at request-creation time and never changes. Once a request is created with a large `numHashes`, no subsequent provider action (including `advanceProviderCommitment`) can reduce the stored value in that request.

---

### Impact Explanation

If an attacker makes N requests to a provider with `maxNumHashes = 0` before the provider's keeper advances `currentCommitmentSequenceNumber`, the N-th request is stored with `numHashes = N`. When the provider (or Fortuna keeper) later calls `revealWithCallback` for that request, `constructProviderCommitment` must execute N keccak256 iterations. At ~30 gas each, N > ~1,000,000 exceeds a 30M-gas block limit. The transaction reverts with out-of-gas on every attempt. The request is **permanently unfulfillable**:

- The user's fee is permanently lost.
- The user's application never receives its random number callback.
- The provider's `sequenceNumber` has already advanced past the stuck request, so it cannot be "skipped" without a new commitment rotation that still cannot retroactively fix the stored `req.numHashes`.

This is a permanent, irreversible DoS on affected Entropy requests.

---

### Likelihood Explanation

- **`maxNumHashes = 0` is the default** for every newly registered provider and for every Fortuna-managed provider that does not explicitly configure `max_num_hashes` in its chain config. This is a broad surface.
- An unprivileged user only needs to call `requestV2` or `requestWithCallback` repeatedly, paying the provider's fee each time. No privileged access is required.
- The economic cost scales with the provider's fee. Providers with low fees (or zero Pyth-protocol fee) make the attack cheaper.
- A keeper outage, network congestion, or a burst of requests in a single block can naturally cause `numHashes` to grow without any attacker involvement, making this a latent correctness hazard even without malicious intent.
- The `advanceProviderCommitment` mitigation only helps for **future** requests; it cannot fix already-created requests with large stored `numHashes`.

---

### Recommendation

1. **Enforce a hard cap on `numHashes` at request creation time regardless of `maxNumHashes`:** Add an absolute upper bound (e.g., `MAX_NUM_HASHES = 1000`) that is always checked, independent of the provider-configurable `maxNumHashes`.
2. **Change the default for `maxNumHashes` from `0` (disabled) to a safe non-zero value** in both the contract and the Fortuna keeper's `unwrap_or` default.
3. **Add a gas-bounded version of `constructProviderCommitment`** that reverts with a descriptive error (e.g., `NumHashesTooLarge`) rather than silently running out of gas, so the failure mode is diagnosable.

---

### Proof of Concept

1. Provider `P` registers with default `maxNumHashes = 0` and `currentCommitmentSequenceNumber = 0`.
2. Attacker calls `requestV2(provider=P, ...)` N times in rapid succession (e.g., across a few blocks before Fortuna responds), paying the fee each time.
3. The N-th request is stored with `req.numHashes = N` (fixed at creation).
4. Fortuna calls `revealWithCallback(P, seqN, userContrib, providerContrib)`.
5. `revealHelper` calls `constructProviderCommitment(N, providerContrib)`.
6. For N ≥ ~1,000,000 (on a 30M-gas chain), the transaction reverts with out-of-gas on every block.
7. The request at sequence number `seqN` is permanently stuck; the user's fee is lost and no callback is ever delivered.
