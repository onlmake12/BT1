### Title
Unprivileged First-Caller Can Permanently Brick Publisher Onboarding by Initializing Config with Arbitrary Authority — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary
`initialize()` has no access control on the payer and accepts a caller-supplied `authority` bytes field. Any funded account can call it before the legitimate operator, storing an arbitrary (e.g., all-zero) authority in the singleton config PDA. Because the PDA can only be initialized once and no authority-update instruction exists, the legitimate operator is permanently locked out of `InitializePublisher`.

### Finding Description
`validate_payer` imposes no identity constraint — it only requires `is_signer` and `is_writable`. [1](#0-0) 

`initialize()` writes the caller-supplied `args.authority` directly into the config PDA without any validation of who the caller is or what authority value is acceptable. [2](#0-1) 

`config::create()` enforces a one-time-write guard (`data.format != 0` → `AlreadyInitialized`), so the config PDA cannot be re-initialized after the first call. [3](#0-2) 

`validate_authority()` in `initialize_publisher` enforces `authority.key.to_bytes() == config.authority` with no fallback or recovery path. [4](#0-3) 

There is no `UpdateAuthority` or equivalent instruction anywhere in the program. [5](#0-4) 

### Impact Explanation
An attacker who calls `initialize()` first — with `args.authority = [0u8; 32]` — permanently sets the config authority to an uncontrolled key. Every subsequent `InitializePublisher` call by the real operator will fail with `MissingRequiredSignature` because `validate_authority` compares the signer's key against the stored (attacker-chosen) authority. The entire publisher onboarding pipeline is permanently disabled with no on-chain recovery path. This matches the scoped impact: **smart contract unable to operate / griefing without profit motive**.

### Likelihood Explanation
The attack requires only a funded keypair (≥1,141,440 lamports for rent) and knowledge of the program ID. No privileged access, leaked keys, or oracle manipulation is needed. The window is the gap between program deployment and the operator's own `initialize()` call — a realistic race on any network where program IDs are observable.

### Recommendation
Restrict `initialize()` to a known privileged signer. The standard pattern is to either:
1. Hard-code the expected authority pubkey as a program constant and verify `payer.key == EXPECTED_AUTHORITY`, or
2. Use an upgrade-authority check: verify the payer matches the program's `BPFLoaderUpgradeable` upgrade authority, which is already a trusted, operator-controlled key.

Additionally, add an `UpdateAuthority` instruction (gated on the current authority's signature) to allow recovery from misconfiguration.
