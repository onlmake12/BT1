Audit Report

## Title
Forward Rate Limiter Bypass via Attacker-Controlled `content.from`/`content.to` Enables Unbounded Forwarding and Memory Growth - (`network/src/protocols/hole_punching/mod.rs`, `connection_request.rs`, `connection_request_delivered.rs`, `connection_sync.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter` is keyed entirely by attacker-supplied `(content.from, content.to, msg_item_id)` values parsed from the message payload. An attacker rotating `content.from` on every message generates a fresh rate-limit bucket each time, bypassing the intended 1 req/sec per-pair cap and allowing all 30 messages/sec (the outer cap) to be forwarded. The `HashMapStateStore` backing the limiter is never cleaned up during a live connection, growing without bound at up to 30 entries/sec.

## Finding Description
`HolePunching` declares two rate limiters in `mod.rs` (L45–46): `rate_limiter` keyed by `(PeerIndex, u32)` (network-assigned, non-forgeable) and `forward_rate_limiter` keyed by `(PeerId, PeerId, u32)` (attacker-controlled message fields).

The outer limiter at `mod.rs` L95–107 correctly caps any single peer at 30 messages/sec using the network-assigned `session_id`. The inner `forward_rate_limiter` is checked in all three message processors using `content.from` and `content.to` parsed directly from the attacker's message bytes:

- `connection_request.rs` L132–143
- `connection_request_delivered.rs` L134–145
- `connection_sync.rs` L85–96

Because the key is entirely attacker-controlled, sending 30 messages/sec each with a distinct `PeerId::random()` as `from` creates 30 independent buckets per second — none of which ever hits the 1 req/sec cap. All 30 messages pass the `forward_rate_limiter` and are forwarded.

`retain_recent()` is called on `forward_rate_limiter` only in `disconnected()` (`mod.rs` L67–68). The periodic `notify()` handler (`mod.rs` L169–175) cleans `pending_delivered` and `inflight_requests` but not `forward_rate_limiter`. For a connection that never disconnects, the map grows at 30 entries/sec indefinitely.

When the target peer is not directly connected, `ConnectionRequestProcess::forward_message()` (`connection_request.rs` L279–305) broadcasts to `sqrt(N)` peers via gossip, each of which repeats the process, compounding the amplification across the P2P network.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single attacker session can sustain 30 forwarded `ConnectionRequest` messages/sec. Each is gossip-broadcast to `sqrt(N)` peers, who forward to `sqrt(N)` more, creating network-wide amplification. The intended 1 req/sec per-pair limit is completely ineffective. Additionally, the unbounded `HashMapStateStore` growth constitutes a memory exhaustion vector: after 24 hours of a single persistent connection, approximately 2.6 million entries accumulate with no cleanup.

## Likelihood Explanation
**High.** Any peer that establishes a single HolePunching protocol session can exploit this immediately. No special privileges, keys, or majority hashpower are required. The attacker simply sends valid `ConnectionRequest` messages with `PeerId::random()` as `from` in each one. The outer 30 req/sec cap is the only real constraint, making this trivially and continuously exploitable for the duration of the connection.

## Recommendation
Key the `forward_rate_limiter` by the actual network peer identity (`session_id` / `PeerIndex`) rather than by attacker-controlled message fields, matching the pattern already used by the outer `rate_limiter`:

```rust
// Replace:
self.forward_rate_limiter.check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))

// With:
self.forward_rate_limiter.check_key(&(self.peer, self.msg_item_id))
```

If the semantic intent is to deduplicate forwarding of the same logical `(from, to)` request across different peers, use a separate bounded LRU cache (e.g., a fixed-capacity `LruCache<(PeerId, PeerId, u32), ()>`) instead of an unbounded `HashMapStateStore` keyed on attacker-supplied data. Additionally, add `forward_rate_limiter.retain_recent()` to the `notify()` periodic handler to bound memory growth regardless of connection lifetime.

## Proof of Concept
1. Attacker connects to a CKB node and negotiates the `HolePunching` protocol.
2. Attacker sends 30 `ConnectionRequest` messages per second (outer cap), each with a freshly generated `PeerId::random()` as `from`, a fixed valid `to` PeerId, and a valid `listen_addrs` list.
3. For each message, `forward_rate_limiter.check_key(&(content.from, content.to, item_id))` sees a brand-new key and passes immediately — the 1 req/sec limit is never triggered for any key.
4. The node forwards all 30 messages/sec to its peers via gossip (`sqrt(N)` peers each), who forward them further, amplifying traffic across the network.
5. After N seconds, the `forward_rate_limiter`'s internal `HashMapStateStore` contains N×30 entries. After 24 hours of a single connection: ~2.6 million entries with no cleanup until disconnect.
6. Unit test confirmation: create a `HolePunching` instance, call `forward_rate_limiter.check_key` 30 times with distinct `(PeerId::random(), fixed_to, item_id)` tuples, and assert all 30 return `Ok(())` — demonstrating the limiter never fires.
