### Title
Peer Store Eviction Algorithm Defeated by Crafted Discovery Addresses — (`network/src/peer_store/peer_store_impl.rs`)

### Summary

The `check_purge` eviction algorithm in `PeerStore` has two sequential strategies and returns `PeerStoreError::EvictionFailed` if both produce no candidates. An unprivileged remote peer can craft discovery `Nodes` messages containing public IP addresses spread across many distinct /24 network groups (≤4 per group) to fill the store to `ADDR_COUNT_LIMIT` in a state where neither eviction strategy can make room, blocking all subsequent `add_addr` calls.

---

### Finding Description

`check_purge` is called by `add_addr` every time a new address is to be stored, once the store is at capacity. [1](#0-0) 

**Strategy 1** collects addresses for which `is_connectable` returns `false`: [2](#0-1) 

For a freshly discovered address, `AddrInfo` is constructed with `last_connected_at_ms = 0`, `attempts_count = 0`, and `last_tried_at_ms = 0`: [3](#0-2) 

`is_connectable` returns `true` for such an address because `attempts_count (0) < ADDR_MAX_RETRIES (3)`: [4](#0-3) 

So Strategy 1 yields zero candidates when the store is filled with fresh discovery addresses.

**Strategy 2** groups addresses by network segment, takes the top half by group size, and only evicts from groups with **strictly more than 4 entries**: [5](#0-4) 

If every group has exactly ≤4 entries, the `addrs.len() > 4` guard is never satisfied, and Strategy 2 also yields zero candidates.

**Result:** both strategies fail, and the function returns the error: [6](#0-5) 

`add_addr` propagates this error: [7](#0-6) 

In `DiscoveryAddressManager::add_new_addrs`, the error is silently swallowed at debug log level — no disconnect, no crash: [8](#0-7) 

**Attack construction:** The discovery protocol allows a non-announce `Nodes` message with up to `MAX_ADDR_TO_SEND` (1000) items, each with up to `MAX_ADDRS` (3) addresses = 3000 addresses per session. [9](#0-8) 

`ADDR_COUNT_LIMIT` is 16384: [10](#0-9) 

An attacker needs only ~6 sessions (6 × 3000 = 18 000 > 16 384), each sending one non-announce `Nodes` message with 1000 items × 3 addresses, all from distinct /24 subnets with ≤4 addresses per subnet. The IPv4 public address space contains ~16 million /24 subnets, so crafting 4096+ distinct subnets is trivial. Each address passes `is_valid_addr` because `is_reachable` only checks that the IP is a public/routable address — it does not verify actual reachability.

---

### Impact Explanation

Once the store is full in this configuration, every subsequent `add_addr` call returns `EvictionFailed`. The node cannot learn new peers via discovery. If existing outbound connections drop, the node cannot replenish them from the peer store. The DoS persists until the node has attempted connections to enough fake addresses to exhaust `ADDR_MAX_RETRIES` (3) per address, at which point `is_connectable` starts returning `false` and Strategy 1 can evict them. However, the attacker can reconnect and refill the store faster than the node exhausts fake addresses, sustaining the isolation. The node does **not** crash — the error is silently dropped — but peer discovery is effectively disabled for the duration of the attack.

---

### Likelihood Explanation

The attack requires only a handful of TCP connections to the victim node (no PoW, no privileged role, no key material). The address crafting is trivial. The only rate-limiting factor is the node's feeler connection rate, which the attacker can outpace by maintaining ~6 simultaneous sessions. This is a low-cost, repeatable attack.

---

### Recommendation

The eviction algorithm must guarantee it can always make room. Concrete fixes:

1. **Remove the `> 4` guard** in Strategy 2, or replace it with a proportional threshold so that even groups of size 1 can be evicted when the store is full.
2. **Add a fallback Strategy 3**: when Strategies 1 and 2 both fail, randomly evict any address that has never been successfully connected to (`last_connected_at_ms == 0`), regardless of group membership.
3. **Rate-limit discovery address ingestion per session** to reduce how quickly a single peer can fill the store.

---

### Proof of Concept

```rust
// Fill AddrManager with 16384 addresses across 4096 /24 groups, 4 per group
let mut peer_store = PeerStore::default();
for subnet in 0u32..4096 {
    for host in 1u32..=4 {
        let ip = Ipv4Addr::from(((10u32 + subnet / 256) << 24) | ((subnet % 256) << 16) | host);
        let addr: Multiaddr = format!("/ip4/{}/tcp/8115", ip).parse().unwrap();
        // Bypass check_purge by calling addr_manager directly for setup
        peer_store.mut_addr_manager().add(AddrInfo::new(addr, 0, 100, Flags::COMPATIBILITY.bits()));
    }
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// Now attempt to add one more address via the normal path
let new_addr: Multiaddr = "/ip4/200.0.0.1/tcp/8115".parse().unwrap();
let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
assert!(matches!(result, Err(e) if e.to_string().contains("EvictionFailed")));
// No recovery path without manual intervention or connection attempts exhausting ADDR_MAX_RETRIES
```

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
