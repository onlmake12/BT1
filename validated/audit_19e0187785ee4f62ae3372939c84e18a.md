### Title
Unconditional `.expect()` on `txs_sizes` in `FeeRateCollector::statistics` panics on legacy `BlockExt` records — (`rpc/src/util/fee_rate.rs`)

---

### Summary

`get_fee_rate_statistics` (and its deprecated alias `get_fee_rate_statics`) unconditionally unwraps `block_ext.txs_sizes` via `.expect()`. The store's compatibility reader for the pre-v0.106 `BlockExt` format always deserialises `txs_sizes` as `None`. Any node whose canonical chain's last ≤101 blocks include at least one legacy-format record will panic the moment an unprivileged caller invokes either RPC endpoint.

---

### Finding Description

**Panic site** — `rpc/src/util/fee_rate.rs:93`:

```rust
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
``` [1](#0-0) 

**Why `txs_sizes` can be `None`** — `store/src/store.rs:247-263` dispatches on `count_extra_fields()`. When a record was written in the legacy 5-field `BlockExt` format (before v0.106), `count_extra_fields()` returns `0` and the reader is converted via `reader.into()`: [2](#0-1) 

That conversion is implemented in `util/types/src/conversion/storage.rs:139-151`, which hard-codes both `cycles` and `txs_sizes` to `None`: [3](#0-2) 

**Why `filter_map` does not protect the caller** — `collect` uses `filter_map` only to skip blocks whose hash or ext is absent from the DB (i.e., `get_block_ext_by_number` returns `None`). A legacy block returns `Some(BlockExt { txs_sizes: None, … })`, which passes through the iterator and reaches the `.expect()` call: [4](#0-3) 

**New blocks are always written as `BlockExtV1`** — `store/src/transaction.rs:241-252` always packs as `BlockExtV1`, so only blocks committed before v0.106 carry the legacy format: [5](#0-4) 

---

### Impact Explanation

A Rust `panic!` in a synchronous RPC handler unwinds the handler thread. Depending on whether the JSON-RPC server wraps handlers in `catch_unwind`, the outcome is either a hard process crash or a hung/dropped connection for that request. Either way the RPC endpoint becomes unavailable for the duration of the panic, and repeated calls keep triggering it.

Scope: **Note — Any local RPC API crash** (0–500 pts).

---

### Likelihood Explanation

The window is the last `min(target, 101)` blocks. On a long-running mainnet node (millions of blocks) all recent blocks are `BlockExtV1`, so the window never reaches legacy records — the exploit is not reachable there. However, the condition is realistic for:

- Testnets / devnets with short chains upgraded from a pre-v0.106 binary.
- Any node whose chain height at upgrade time was ≤ 101 blocks.
- Integration-test environments that replay a short chain from genesis with an old DB snapshot.

The RPC requires no authentication and accepts any `target` value from 1 to 101. [6](#0-5) 

---

### Recommendation

Replace the unconditional `.expect()` with a graceful skip:

```rust
// Before
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");

// After
let Some(txs_sizes) = txs_sizes else { return fee_rates; };
```

This matches the existing guard on `cycles` (line 97) and makes the function safe for any chain state, including nodes upgraded from legacy databases.

---

### Proof of Concept

1. Open a `ChainDB` and insert a genesis block whose `BlockExt` is serialised as the legacy 5-field `packed::BlockExt` (i.e., **not** `BlockExtV1`) — `txs_sizes` field absent.
2. Wrap the store in a `Snapshot` and call `FeeRateCollector::new(snapshot.as_ref()).statistics(Some(1))`.
3. Observe the process panics at `rpc/src/util/fee_rate.rs:93` with message `"expect txs_size's length >= 1"`.

The existing test in `store/src/tests/db.rs:54-68` already constructs a `BlockExt { txs_sizes: None }` and round-trips it through the store — extending that test to call `FeeRateCollector::statistics` is sufficient to reproduce the panic. [7](#0-6)
