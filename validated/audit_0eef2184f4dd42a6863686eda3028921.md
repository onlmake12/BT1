### Title
Unbounded OrphanBlockPool Growth and O(N) Per-Block CPU Cost via Cheap Low-Difficulty Orphan Flooding — (`chain/src/orphan_broker.rs`, `chain/src/utils/orphan_block_pool.rs`)

### Summary
An unprivileged P2P peer can flood the node with syntactically valid blocks that each claim a trivially-low `compact_target` (minimum PoW difficulty), pass `non_contextual_verify`, reference a unique unknown parent hash, and accumulate in the `OrphanBlockPool` without bound. Because `ORPHAN_BLOCK_SIZE` is only the initial `HashMap` capacity and there is no size-based eviction, the pool grows indefinitely. Because `search_orphan_leaders` iterates every leader on every new block arrival, CPU cost scales as O(N²) over the flood.

### Finding Description
In `chain/src/utils/orphan_block_pool.rs`, `InnerPool::with_capacity` uses `ORPHAN_BLOCK_SIZE` (= 8192) only as the initial allocation hint for the `blocks` `HashMap`. [9](#0-8)  There is no maximum-size check anywhere in `InnerPool::insert`. [2](#0-1) 

The only eviction path is `clean_expired_blocks`, which removes entries whose epoch number satisfies `epoch + EXPIRED_EPOCH < tip_epoch`. [10](#0-9)  Blocks crafted with the current epoch number survive for approximately 24 hours (6 epochs × ~4 hours/epoch).

In `chain/src/chain_service.rs`, `asynchronous_process_block` calls `non_contextual_verify` which checks only that the block hash satisfies the `compact_target` embedded in the block header itself — it does not verify that `compact_target` is the correct difficulty for the current epoch. [7](#0-6)  An attacker sets `compact_target` to the minimum representable value, making PoW trivially solvable, and the block passes verification.

After each block is processed, `process_lonely_block` calls `search_orphan_leaders`, which calls `clone_leaders()` and iterates the entire leaders set. [3](#0-2)  With N injected blocks each having a distinct unknown parent, the leaders set has N entries, so each subsequent block triggers O(N) work, yielding O(N²) total CPU for N blocks.

### Impact Explanation
- **Memory**: Each orphan entry occupies `blocks`, `parents`, and `leaders` map entries plus the stored block data in RocksDB. N injected blocks → O(N) unbounded RAM and disk growth.
- **CPU**: `search_orphan_leaders` is called synchronously inside the single-threaded `ChainService` loop. O(N²) total work stalls block processing for all peers.
- **Availability**: Sustained injection causes the node to become unresponsive to legitimate blocks, effectively a remote DoS.

### Likelihood Explanation
The attack requires only a P2P connection and the ability to generate blocks with minimum-difficulty PoW (trivially cheap). No privileged access, no key material, no majority hashpower is needed. The 60-second expiry timer and 6-epoch grace period mean the attacker can sustain the pool at size N by re-injecting at a rate that outpaces expiry. The `process_block_rx` channel capacity of 24 [11](#0-10)  throttles throughput but does not prevent the attack — it only sets the injection rate.

### Recommendation
1. **Hard cap**: Enforce a maximum pool size in `InnerPool::insert`; reject or evict (e.g., oldest leader) when the cap is reached.
2. **Difficulty floor**: In `non_contextual_verify`, reject blocks whose `compact_target` is below a minimum threshold (e.g., genesis difficulty / some factor).
3. **Per-peer orphan accounting**: Track and cap how many orphans each peer may contribute.
4. **Lazy leader scan**: Instead of scanning all leaders on every block, only check the specific parent hash of the newly arrived block.

### Proof of Concept
```
1. Connect to a CKB node as a sync peer.
2. For i in 1..N:
   a. Construct a block with:
      - parent_hash = random_bytes(32)   # unique unknown parent
      - compact_target = 0x207fffff      # minimum difficulty
      - epoch = current_tip_epoch        # avoids 6-epoch expiry
      - valid PoW solution (trivial at min difficulty)
   b. Send via P2P BlockTransactions / SendBlock message.
3. Assert: orphan_blocks_broker.len() == N (no eviction observed).
4. Measure: latency of processing block i scales linearly with i
   (search_orphan_leaders iterates i leaders), confirming O(N²) total.
```

    fn insert(&mut self, lonely_block: LonelyBlockHash) {
        let hash = lonely_block.hash();
        let parent_hash = lonely_block.parent_hash();
        self.blocks
            .entry(parent_hash.clone())
            .or_default()
            .insert(hash.clone(), lonely_block);
        // Out-of-order insertion needs to be deduplicated
        self.leaders.remove(&hash);
        // It is a possible optimization to make the judgment in advance,
        // because the parent of the block must not be equal to its own hash,
        // so we can judge first, which may reduce one arc clone
        if !self.parents.contains_key(&parent_hash) {
            // Block referenced by `parent_hash` is not in the pool,
            // and it has at least one child, the new inserted block, so add it to leaders.
            self.leaders.insert(parent_hash.clone());
        }
        self.parents.insert(hash, parent_hash);
    }

    /// cleanup expired blocks(epoch + EXPIRED_EPOCH < tip_epoch)
    pub fn clean_expired_blocks(&mut self, tip_epoch: EpochNumber) -> Vec<LonelyBlockHash> {
        let mut result = vec![];

        for hash in self.leaders.clone().iter() {
            if self.need_clean(hash, tip_epoch) {
                // remove items in orphan pool and return hash to callee(clean header map)
                let descendants = self.remove_blocks_by_parent(hash);
                result.extend(descendants);
            }
        }
        result
    }

    /// get 1st block belongs to that parent and check if it's expired block
    fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
        self.blocks
            .get(parent_hash)
            .and_then(|map| {
                map.iter().next().map(|(_, lonely_block)| {
                    lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
                })
            })
            .unwrap_or_default()
    }
