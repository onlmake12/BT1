Looking at the code carefully to verify the claim.

### Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Unique Attacker-Controlled PeerId Keys — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

The `forward_rate_limiter` in `HolePunching` is backed by a `HashMapStateStore<(PeerId, PeerId, u32)>`. Its `retain_recent()` cleanup is called **only** in `disconnected()`. The periodic `notify()` timer (every 5 minutes) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter. Because the key includes attacker-controlled `(from, to)` PeerIds parsed from message content — not from the session — a single persistent connection can insert 30 new entries per second indefinitely, growing the HashMap without bound until the process is killed by OOM.

---

### Finding Description

`forward_rate_limiter` is declared as:

```rust
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,
```

backed by `governor::state::keyed::HashMapStateStore`. [1](#0-0) 

`retain_recent()` is invoked in exactly two places in the entire codebase — both inside `disconnected()`:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    ...
}
``` [2](#0-1) 

The `notify()` handler fires every `CHECK_INTERVAL` (5 minutes) and does clean `pending_delivered` and `inflight_requests`, but it never calls `retain_recent()` on either rate limiter:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    ...
    self.pending_delivered.retain(|_, (_, t)| (now - *t) < TIMEOUT);
    self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);
    // forward_rate_limiter.retain_recent() is ABSENT
    ...
}
``` [3](#0-2) 

The `forward_rate_limiter` key is `(content.from, content.to, msg_item_id)`, where `from` and `to` are PeerIds **parsed from the message body**, fully attacker-controlled:

```rust
self.protocol
    .forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
``` [4](#0-3) 

The outer `rate_limiter` is keyed by `(session_id, msg.item_id())`. Since `session_id` is fixed per connection and `item_id` is only 0, 1, or 2, this limiter allows **30 ConnectionRequest messages per second** from one session — each with a fresh `(from, to)` pair, each inserting a new entry into `forward_rate_limiter`'s HashMap. [5](#0-4) 

---

### Impact Explanation

Each `(PeerId, PeerId, u32)` key is approximately 80–100 bytes (two 39-byte multihash PeerIds + u32 + governor state cell). At 30 inserts/sec from a single persistent connection:

- After 1 hour: ~108,000 entries (~10 MB)
- After 10 hours: ~1,080,000 entries (~100 MB)
- After 100 hours: ~10.8M entries (~1 GB)

With multiple simultaneous connections (up to the peer limit), the rate multiplies proportionally. The process eventually hits OOM and is killed by the OS, crashing the node and all dependent peers relying on it for sync.

---

### Likelihood Explanation

The exploit requires only a standard P2P connection — no privileges, no PoW, no key material. The attacker constructs valid `ConnectionRequest` messages with syntactically valid but unique random PeerIds in the `from`/`to` fields. The outer rate limiter does not prevent this because it keys on `(session_id, item_id)`, not on message content. The attack is slow but fully automated and requires no interaction beyond maintaining the TCP connection.

---

### Recommendation

Call `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside the `notify()` handler, which already fires every 5 minutes. This mirrors the pattern used in `disconnected()` and ensures stale entries (whose 1-second quota window has long expired) are evicted on a timer regardless of whether the peer ever disconnects:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    // ... existing cleanup ...
}
``` [3](#0-2)

### Proof of Concept

1. Attacker opens one TCP connection to a CKB node with HolePunching enabled.
2. Attacker sends 30 `ConnectionRequest` messages per second, each with a freshly generated random `from` PeerId and `to` PeerId in the message body.
3. The outer `rate_limiter` passes all 30/sec (quota not exceeded for key `(session_id, 0)`).
4. Each message reaches `forward_rate_limiter.check_key(&(from_i, to_i, 0))` with a unique key, inserting a new `HashMapStateStore` entry.
5. `retain_recent()` is never called because the attacker never disconnects and `notify()` does not invoke it.
6. After N seconds, the HashMap contains 30·N entries. Instrument with `forward_rate_limiter.len()` to confirm linear growth. Assert `len() < threshold` after 60 seconds — assertion fails, confirming the invariant is broken.
