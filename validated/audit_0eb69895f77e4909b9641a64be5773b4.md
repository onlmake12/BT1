### Title
`EntropyUpgradable` Contract on Blast Does Not Configure Gas Yield Mode - (`target_chains/ethereum/contracts/contracts/entropy/EntropyUpgradable.sol`)

### Summary

The Pyth team applied a Blast-specific gas yield patch to `Pyth.sol` but did not apply the equivalent configuration to `EntropyUpgradable.sol`. As a result, all gas fees earned by the Entropy contract deployed on Blast (`0x5744Cbf430D99456a0A8771208b674F27f8EF0Fb`) are permanently unclaimable, representing a continuous value leak for the protocol.

### Finding Description

Pyth is aware of Blast's gas yield mechanism and explicitly patched `Pyth.sol` to call `IBlast(0x4300000000000000000000000000000000000002).configureClaimableGas()` via a chain-specific diff: [1](#0-0) 

However, the `EntropyUpgradable` contract's `initialize()` function — the only initialization entry point — contains no equivalent Blast gas yield configuration: [2](#0-1) 

On Blast, contracts default to a `VOID` gas mode, meaning gas fees are not claimable unless `configureClaimableGas()` is explicitly called. Since `EntropyUpgradable.initialize()` never calls this, all gas spent by users and keepers interacting with the Entropy contract on Blast accrues to the Blast L2 and is permanently lost to Pyth.

The Entropy contract is confirmed deployed on Blast: [3](#0-2) 

The Entropy contract receives substantial gas usage: users call `request()`, `requestV2()`, `requestWithCallback()`, and keepers call `revealWithCallback()` continuously. The `requestHelper` internal function alone performs multiple storage writes per call: [4](#0-3) 

### Impact Explanation

All gas yield accrued on the `EntropyUpgradable` contract on Blast since deployment is permanently lost. Blast's gas yield mechanism allows contracts to reclaim a portion of gas fees paid by callers — but only if `configureClaimableGas()` is called. Without it, the yield defaults to `VOID` mode and is irrecoverable. Given the high transaction volume of the Entropy contract (every random number request and reveal), this represents a meaningful and ongoing loss of value for the Pyth protocol.

### Likelihood Explanation

This is a deployment-time omission that is already in effect on mainnet Blast. Every transaction sent to the Entropy contract on Blast since deployment has been silently losing gas yield. No attacker action is required — the value leak is passive and continuous.

### Recommendation

Add a Blast-specific chain patch for `EntropyUpgradable.sol` analogous to the existing `blast-gas-claim-patch.diff`. The `initialize()` function (or a separate admin-callable function) should invoke:

```solidity
IBlast(0x4300000000000000000000000000000000000002).configureClaimableGas();
```

Additionally, expose an admin function to claim the accumulated gas yield and forward it to the Pyth fee recipient. Also evaluate whether `EchoUpgradeable` is deployed or planned for Blast, as it has the same omission: [5](#0-4) 

### Proof of Concept

1. Observe that `blast-gas-claim-patch.diff` patches only `Pyth.sol` with `configureClaimableGas()`.
2. Observe that `EntropyUpgradable.initialize()` has no call to `IBlast.configureClaimableGas()`.
3. Confirm Entropy is live on Blast at `0x5744Cbf430D99456a0A8771208b674F27f8EF0Fb`.
4. Call `IBlast(0x4300000000000000000000000000000000000002).readGasParams(0x5744Cbf430D99456a0A8771208b674F27f8EF0Fb)` on Blast mainnet — the `gasMode` will return `VOID` (0), confirming gas yield is unconfigured and all accrued gas yield is unclaimable.

        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];
        if (_state.providers[provider].sequenceNumber == 0)
            revert EntropyErrors.NoSuchProvider();

        // Assign a sequence number to the request
        uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
        if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
            revert EntropyErrors.OutOfRandomness();
        providerInfo.sequenceNumber += 1;

        // Check that fees were paid and increment the pyth / provider balances.
        uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
        if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
        uint128 providerFee = getProviderFee(provider, callbackGasLimit);
        providerInfo.accruedFeesInWei += providerFee;
        _state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) -
