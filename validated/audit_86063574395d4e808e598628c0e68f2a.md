Audit Report

## Title
Permanent Spent-Cell State Corruption via `break` in Input Loop — (`util/rich-indexer/src/indexer/mod.rs`)

## Summary

In `AsyncRichIndexer::insert_transaction`, the input-processing loop at line 232 uses `break` instead of `continue` when `spend_cell` returns `false`. When a filter is configured and a transaction's first input references a cell absent from the `output` table, the loop exits early, leaving all subsequent inputs' cells permanently marked `is_spent = 0` even if those cells are present in the database. The indexer's live-cell view becomes permanently corrupted for the affected cells until a full re-index.

## Finding Description

At `util/rich-indexer/src/indexer/mod.rs` lines 228–248, the loop is:

```rust
if tx_index != 0 {
    for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
        let out_point = input.previous_output();
        if !spend_cell(&out_point, tx).await? {
            break;   // ← exits entire loop on first absent cell
        }
        // ... build input rows
    }
}
```

`spend_cell` at `util/rich-indexer/src/indexer/insert.rs` lines 403–427 issues `UPDATE output SET is_spent = 1 WHERE tx_id = ... AND output_index = ...` and returns `false` when `rows_affected == 0` — i.e., the referenced cell is not in the `output` table.

A cell is legitimately absent from `output` in two production-reachable scenarios confirmed by lines 162–168 of `mod.rs`:

1. **`cell_filter` configured**: the producing transaction's output was filtered out at index time.
2. **`block_filter` configured**: the producing block was stored via `bulk_insert_blocks_simple`, which inserts no output rows.

When `input[0]` hits this condition, `break` exits the loop. `input[1]`, `input[2]`, … are never visited. Any of those later inputs that reference cells **present** in `output` are never updated to `is_spent = 1`. Since `insert_transaction` is called exactly once per block during forward sync with no subsequent reconciliation, the corruption is permanent. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation

The rich indexer's `output` table retains `is_spent = 0` for definitively spent cells. Every subsequent `get_cells` / `get_cells_capacity` RPC query returns those cells as live. This is a concrete incorrect implementation of the CKB state storage mechanism — the indexer's authoritative record of cell liveness is permanently wrong for affected cells without a full rollback and re-index.

**Severity: Medium (2001–10000 points) — Suboptimal/incorrect implementation of CKB state storage mechanism.**

The consensus layer is unaffected; the corruption is confined to the indexer's derived state. Clients relying on the indexer for cell selection will receive stale data, but any transaction they construct referencing a spent cell will be rejected by consensus (no fund loss). The indexer cannot self-heal.

## Likelihood Explanation

Requires only that the node operator has configured `block_filter` or `cell_filter` (a documented, supported production feature) and that a committed transaction's `input[0]` references a cell absent from the indexer DB while at least one later input references a cell that is present. No attacker action is required; this triggers naturally during normal chain operation with filters enabled. A miner could also deliberately order inputs to guarantee the trigger.

## Recommendation

Replace `break` with `continue` at line 232 of `util/rich-indexer/src/indexer/mod.rs`:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // skip unindexed input, process remaining inputs
}
```

## Proof of Concept

1. Configure `cell_filter` to exclude cells locked by script `L_filtered`.
2. Index block A containing `tx_A` with `output[0]` locked by `L_filtered` (filtered out, not in `output` table) and `output[1]` locked by `L_indexed` (indexed, `is_spent = 0`).
3. Index block B containing `tx_B` with:
   - `input[0]` = `OutPoint { tx_hash: tx_A.hash, index: 0 }` → `spend_cell` returns `false` → `break` fires
   - `input[1]` = `OutPoint { tx_hash: tx_A.hash, index: 1 }` → never visited
4. Query: `SELECT is_spent FROM output WHERE tx_id = (SELECT id FROM ckb_transaction WHERE tx_hash = <tx_A_hash>) AND output_index = 1`
5. **Observed**: `is_spent = 0`. **Expected**: `is_spent = 1`.
6. Call `get_cells` for `L_indexed` — the spent cell is returned as live.

```rust
pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
}
```
