### Title
Unprivileged Rent Theft via `close_signature_set` — Missing Payer Authority Check — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

### Summary

`close_signature_set` allows any signer to designate themselves as `sol_destination` and receive the rent lamports from a `SignatureSet` account they did not fund, because the instruction enforces no relationship between `sol_destination` and the original payer who created the `SignatureSet`.

### Finding Description

The `CloseSignatureSet` account context has three constraints:

1. `sol_destination` — only requires `Signer`. No constraint ties it to the original payer.
2. `posted_vaa` — must be a valid PDA with `has_one = signature_set`, confirming the VAA references the given `SignatureSet`.
3. `signature_set` — closed with `close = sol_destination`, transferring all lamports to whoever `sol_destination` is. [1](#0-0) 

The `SignatureSet` account is created and funded by a `payer` in `VerifySignatures` via `init_if_needed, payer = payer`, but the `SignatureSet` struct stores only `sig_verify_successes`, `message_hash`, and `guardian_set_index` — **no payer field is recorded**. [2](#0-1) [3](#0-2) 

Because no payer is stored and `close_signature_set` imposes no authority check on `sol_destination`, any third party can call the instruction and redirect the lamports.

### Impact Explanation

An attacker who observes any `PostedVaaV1` account on-chain (all are public PDAs) can immediately call `close_signature_set` with themselves as `sol_destination`, draining the rent from the associated `SignatureSet` to their own wallet. The original payer permanently loses those lamports. This can be applied to every existing `SignatureSet` account on-chain in a single sweep.

### Likelihood Explanation

The preconditions are entirely public and require no privileged access:
- A valid `PostedVaaV1` PDA exists (observable via RPC).
- Its `signature_set` field points to a funded `SignatureSet` account (also observable).
- The attacker only needs to sign a transaction with themselves as `sol_destination`.

No front-running race is even required — the attacker can act at any time after `post_vaa` is called, since the `SignatureSet` persists until explicitly closed.

### Recommendation

Record the original payer's pubkey inside the `SignatureSet` account at creation time, and add a constraint in `CloseSignatureSet` requiring `sol_destination` to match that stored payer:

```rust
// In SignatureSet struct, add:
pub payer: Pubkey,

// In CloseSignatureSet, add:
#[account(
    mut,
    close = sol_destination,
    constraint = signature_set.payer == sol_destination.key()
        @ CoreBridgeError::InvalidFeeRecipient
)]
signature_set: Account<'info, AccountVariant<SignatureSet>>,
```

Alternatively, require `sol_destination` to be a signer **and** match the `payer` stored in the account.

### Proof of Concept

```
1. payer_A calls verify_signatures (legacy), funding a new SignatureSet keypair account.
2. payer_A calls post_vaa (legacy), creating PostedVaaV1 with signature_set = <keypair pubkey>.
3. attacker_B constructs close_signature_set with:
     sol_destination = attacker_B  (attacker_B signs)
     posted_vaa      = <the PostedVaaV1 PDA>
     signature_set   = <the keypair account funded by payer_A>
4. Anchor's `close = sol_destination` transfers all lamports from signature_set → attacker_B.
5. payer_A receives nothing; their rent is permanently lost.
``` [4](#0-3)
