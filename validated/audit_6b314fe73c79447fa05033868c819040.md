### Title
Unchecked ETH Refund Return Value Causes Permanent Fund Loss - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

### Summary
`PythAggregatorV3.updateFeeds()` silently ignores the return value of the ETH refund call to `msg.sender`. If the refund fails, the excess ETH is permanently locked in the contract with no recovery path.

### Finding Description
In `updateFeeds()`, after paying the Pyth update fee, the contract attempts to refund the remaining ETH balance to `msg.sender`:

```solidity
function updateFeeds(bytes[] calldata priceUpdateData) public payable {
    uint fee = pyth.getUpdateFee(priceUpdateData);
    pyth.updatePriceFeeds{value: fee}(priceUpdateData);

    // refund remaining eth
    // solhint-disable-next-line no-unused-vars
    (bool success, ) = payable(msg.sender).call{
        value: address(this).balance
    }("");
}
```

The `success` return value is explicitly suppressed via the `no-unused-vars` lint comment and never checked. If the low-level call fails — for example, because `msg.sender` is a contract whose `receive()` or `fallback()` reverts — the transaction still succeeds, the ETH is not returned, and it becomes permanently stuck in `PythAggregatorV3`. The contract has no admin, no `withdraw` function, and no recovery mechanism of any kind. [1](#0-0) 

### Impact Explanation
Any excess ETH sent to `updateFeeds()` by a contract caller whose `receive()`/`fallback()` reverts is permanently locked in `PythAggregatorV3`. There is no admin withdrawal, no `receive()` on the aggregator itself that could be used to drain it, and no upgrade path. The loss is irreversible.

### Likelihood Explanation
`updateFeeds()` is a `public payable` function callable by any unprivileged address, including smart contract wallets, multisigs, and protocol integrators — all of which commonly have non-trivial `receive()` logic. The pattern of sending slightly more ETH than the exact fee (to avoid underpayment reverts) is standard practice. The combination makes this reachable in normal usage.

### Recommendation
Check the return value and revert if the refund fails:

```solidity
(bool success, ) = payable(msg.sender).call{value: address(this).balance}("");
require(success, "ETH refund failed");
```

Alternatively, use a pull-payment pattern: track the refund amount in a mapping and let callers withdraw it separately.

### Proof of Concept
1. Deploy `PythAggregatorV3` pointing to a live Pyth contract.
2. Deploy an attacker contract whose `receive()` function reverts.
3. From the attacker contract, call `updateFeeds{value: fee + 1 ether}(data)`.
4. `pyth.updatePriceFeeds` succeeds and consumes `fee` wei.
5. The refund call to the attacker contract's `receive()` reverts; `success == false`.
6. Because `success` is never checked, the outer transaction succeeds.
7. `1 ether` (minus `fee`) is now permanently locked in `PythAggregatorV3` with no recovery path. [2](#0-1)
