### Title
Unbounded `numHashes` Loop in `constructProviderCommitment` Enables Permanent DoS of Entropy Reveal — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When an Entropy provider has not set `maxNumHashes` (the default is `0`), an unprivileged user can make a large number of randomness requests without any reveals occurring. Each new request stores a `numHashes` value equal to `assignedSequenceNumber − currentCommitmentSequenceNumber`. The reveal path calls `constructProviderCommitment`, which loops `numHashes` times. If `numHashes` is large enough, the reveal transaction exceeds the block gas limit and can never be executed, permanently locking those requests.

---

### Finding Description

When a request is created in `requestHelper`, `numHashes` is computed and stored:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
``` [1](#0-0) 

The only guard against a large `numHashes` is:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [2](#0-1) 

Because `maxNumHashes` defaults to `0` (the zero-initialized struct field), this guard is **never active** for providers who have not explicitly called `setMaxNumHashes`. The `ProviderInfo` struct confirms the field defaults to zero: [3](#0-2) 

At reveal time, `revealHelper` calls `constructProviderCommitment`, which loops `numHashes` times:

```solidity
while (numHashes > 0) {
    currentHash = keccak256(bytes.concat(currentHash));
    numHashes -= 1;
}
``` [4](#0-3) 

Each `keccak256` on 32 bytes costs ~36 gas. At ~833,000 iterations the loop exceeds Ethereum's ~30 M gas block limit, making the reveal transaction permanently unrevertable.

The `numHashes` value is **frozen at request creation time** and is never updated when the provider reveals earlier requests. So requests created before any reveals carry the full accumulated distance from `currentCommitmentSequenceNumber`.

---

### Impact Explanation

- Any request whose stored `numHashes` exceeds ~833,000 can never be revealed by anyone (the function is permissionless but gas-bounded).
- Users who paid fees for those requests permanently lose their funds and never receive their random numbers.
- Callback-based consumers (e.g., on-chain games, lotteries) are permanently stuck waiting for a callback that can never arrive.
- The provider's entire remaining chain beyond that point is effectively bricked.

---

### Likelihood Explanation

- The default value of `maxNumHashes` is `0`, meaning **every provider that has not explicitly called `setMaxNumHashes` is vulnerable**.
- Any unprivileged address can call `requestV2` or `requestWithCallback` paying only the provider's fee per request.
- The attacker's cost is `N × feeInWei`; for a low-fee provider this is cheap relative to the permanent damage caused.
- The provider cannot retroactively fix already-created requests by setting `maxNumHashes` later — the stored `numHashes` in existing requests is immutable.

---

### Recommendation

1. **Set a non-zero default for `maxNumHashes`** at registration time (e.g., enforce `chainLength` as an upper bound), or require providers to supply a `maxNumHashes > 0` during `register`.
2. **Enforce the `maxNumHashes` check unconditionally** (remove the `!= 0` short-circuit), so that a zero value is treated as "use a safe protocol-level default" rather than "no limit".
3. Document clearly that providers must call `setMaxNumHashes` before accepting requests, and add a warning in the SDK.

---

### Proof of Concept

1. Provider registers with `chainLength = 2_000_000` and does **not** call `setMaxNumHashes` (default `maxNumHashes = 0`).
2. Attacker calls `requestV2(provider, ...)` 900,000 times, paying the provider fee each time. No reveals occur.
3. The 900,000th request is stored with `numHashes = 900,000 − 0 = 900,000`.
4. Anyone calls `revealWithCallback(provider, 900000, ...)`. The `constructProviderCommitment` loop runs 900,000 iterations ≈ 32.4 M gas > block gas limit → transaction always reverts.
5. The request is permanently unresolvable. The user's fee is lost and the callback never fires. [5](#0-4) [6](#0-5) [7](#0-6)
