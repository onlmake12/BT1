Audit Report

## Title
`ScriptHashTypeVerifier` Omits Type Script Hash Type Consensus Check, Enabling Zero-Cost DoS via Forced Contextual Verification — (`File: verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify` enforces the `ENABLED_SCRIPT_HASH_TYPE` consensus allowlist only against `output.lock().hash_type()`, never against `output.type_().hash_type()`. An unprivileged attacker can craft a transaction whose type script carries a structurally valid but consensus-disallowed hash type (e.g., `Data3 = 6`), bypass the cheap non-contextual gate entirely, and force every receiving node to perform expensive contextual script verification before the transaction is finally rejected — at zero fee cost to the attacker.

## Finding Description

**Root cause — lock-only check in `ScriptHashTypeVerifier::verify`:**

The verifier iterates over outputs and validates only the lock script hash type:

```rust
// verification/src/transaction_verifier.rs L796-814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())
}
```

`output.type_()` is never read. [1](#0-0) 

**Why `check_data()` does not compensate:**

`check_data()` covers both scripts but only validates structural validity — that the raw byte is a recognized `ScriptHashType` enum discriminant (even numbers pass `verify_value`). `Data3 = 6` is even, so it passes. The consensus allowlist (`{0, 1, 2, 4}`) is enforced exclusively by `ScriptHashTypeVerifier`, and only for the lock script. [2](#0-1) 

**Allowlist definition:**

```rust
// util/constant/src/consensus.rs L7-12
pub static ENABLED_SCRIPT_HASH_TYPE: Set<u8> = phf_set! {
    0u8, // Data
    1u8, // Type
    2u8, // Data1
    4u8, // Data2
};
```

`Data3 = 6`, `Data4 = 8`, etc. are absent. [3](#0-2) 

**Error message confirms the omission is by design (for lock only):**

```rust
// util/types/src/core/error.rs L223-228
/// The lock Script hash_type field is invalid.
#[error("InvalidScriptHashType: the lock Script hash_type field is invalid")]
InvalidScriptHashType { hash_type: Byte }
``` [4](#0-3) 

**Exploit flow:**

1. Attacker submits a transaction with `lock.hash_type = 0` (Data, allowed) and `type_.hash_type = 6` (Data3, disallowed).
2. `NonContextualTransactionVerifier` calls `ScriptHashTypeVerifier::verify()` → **passes** (only lock is checked).
3. `check_data()` → **passes** (6 is even, structurally valid).
4. Transaction enters `ContextualTransactionVerifier` and `TransactionScriptsVerifier`.
5. `select_version()` is called on the type script → returns `Err(ScriptError::InvalidScriptHashType(...))`. [5](#0-4) 
6. Transaction is rejected — but only after full contextual verification overhead (cycle accounting, script group construction, VM dispatch).
7. No fee is deducted (rejection precedes fee collection). Repeat at high rate.

## Impact Explanation

The non-contextual gate exists precisely to cheaply reject invalid transactions before expensive script execution. By bypassing it, an attacker forces every relaying/validating full node to perform contextual verification — including script group construction and VM dispatch — for transactions that should have been dropped at the first gate. Since the transactions are rejected before fee deduction, the attacker bears zero economic cost. A sustained high-rate stream of such transactions degrades node throughput and can cause network congestion.

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation

The attack requires only the ability to submit transactions via the public RPC (`send_transaction`) or the P2P relay protocol — both reachable by any unprivileged peer with no key material, mining power, or special access. Constructing a transaction with `type_.hash_type = 6` is trivial. The attack is fully repeatable and automatable.

## Recommendation

Extend `ScriptHashTypeVerifier::verify` to also validate the type script hash type for each output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        let lock_ht = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: output.lock().hash_type() })?;
        let val: u8 = lock_ht.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }

        // NEW: type-script check (if present)
        if let Some(type_script) = output.type_().to_opt() {
            let type_ht = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: type_script.hash_type() })?;
            let val: u8 = type_ht.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        }
    }
    Ok(())
}
```

A corresponding new error variant (e.g., `InvalidTypeScriptHashType`) should be added to `util/types/src/core/error.rs` to distinguish type-script rejections from lock-script rejections. [6](#0-5) 

## Proof of Concept

```
1. Build a transaction with one output:
     lock  = Script { hash_type: 0 (Data), code_hash: <any 32 bytes>, args: <any> }
     type_ = Some(Script { hash_type: 6 (Data3), code_hash: <any 32 bytes>, args: <any> })
     capacity = sufficient

2. Submit via JSON-RPC:
     curl -X POST http://localhost:8114 \
       -H 'Content-Type: application/json' \
       -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<tx>,"passthrough"]}'

3. Observe:
   - ScriptHashTypeVerifier passes (only lock hash_type=0 is checked)
   - check_data() passes (hash_type=6 is even, structurally valid)
   - ContextualTransactionVerifier is entered; select_version() rejects with
     ScriptError::InvalidScriptHashType for the type script
   - Node expends contextual-verification resources before rejection
   - No fee is deducted

4. Automate at high rate to exhaust node CPU and tx-pool processing capacity.
```
