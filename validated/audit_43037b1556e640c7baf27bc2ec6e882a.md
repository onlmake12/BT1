### Title
Attacker Can Permanently Destroy Consumer Callback by Supplying Insufficient Gas to `executeCallback` — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` clears the request and credits the provider **before** invoking the consumer callback, but performs **no check** that `gasleft()` is sufficient to actually forward `req.callbackGasLimit` gas to the callback. Due to EVM's 63/64 rule, a caller who supplies gas in the "middle range" — enough to complete all pre-callback storage operations but not enough to forward the full `callbackGasLimit` — will permanently destroy the request while the consumer callback silently fails. There is no retry mechanism.

---

### Finding Description

In `Echo.sol`, `executeCallback` follows this sequence:

1. Validates the request and provider exclusivity.
2. Parses price feeds via Pyth.
3. Credits the provider: `_state.providers[providerToCredit].accruedFeesInWei += ...`
4. **Clears the request**: `clearRequest(sequenceNumber)` — permanently removes it from storage.
5. Advances `firstUnfulfilledSeq`.
6. Calls `IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` inside a `try/catch`.

The `try/catch` silently absorbs any callback failure (including out-of-gas) and emits `PriceUpdateCallbackFailed`. Because the request was already cleared in step 4, there is **no way to retry** the callback.

The IEcho.sol interface comment states:

> "Requires 1.5x the callback gas limit to account for cross-contract call overhead"

But this is **not enforced on-chain**. There is no `require(gasleft() >= req.callbackGasLimit * 3 / 2, ...)` or equivalent guard before the callback.

Due to EVM's 63/64 rule (CALL opcode forwards at most 63/64 of the current context's gas), if `gasleft()` at the point of the `_echoCallback` call is less than `req.callbackGasLimit * 64 / 63`, the callback receives fewer than `req.callbackGasLimit` gas units. If the callback needs the full `req.callbackGasLimit`, it will revert with out-of-gas. The `catch` block absorbs this silently.

Compare this to `Entropy.sol`, which has an explicit guard:

```solidity
uint256 startingGas = gasleft();
(success, ret) = req.requester.excessivelySafeCall(
    uint256(req.gasLimit10k) * TEN_THOUSAND, ...
);
// ...
} else {
    // Calling context did not have enough gas to run the callback
    revert EntropyErrors.InsufficientGas(); // <-- entire tx reverts, request preserved
}
```

`Entropy.sol` also has a `CALLBACK_FAILED` state that allows retries. `Echo.sol` has neither protection.

---

### Impact Explanation

A consumer who calls `requestPriceUpdatesWithCallback` with a `callbackGasLimit` of, say, 1,000,000 gas pays the corresponding fee. If `executeCallback` is called with gas in the range `[overhead_for_pre_callback_logic, overhead + callbackGasLimit * 64/63)`:

- All pre-callback state changes succeed (provider credited, request cleared).
- The callback receives fewer than `callbackGasLimit` gas and fails.
- The `catch` block emits `PriceUpdateCallbackFailed` and returns successfully.
- The request is permanently gone — **the consumer's callback can never be executed**.

The consumer paid for a service they will never receive. Any application state that depends on the callback (e.g., a DeFi protocol waiting for a price update to settle a position) is permanently stuck.

---

### Likelihood Explanation

`executeCallback` is callable by **anyone** (`external`). An attacker can:

1. Monitor the mempool for pending `requestPriceUpdatesWithCallback` transactions.
2. Front-run or race the legitimate provider by calling `executeCallback` with a carefully chosen gas limit.
3. The gas limit is chosen to be above the cost of all pre-callback logic but below `req.callbackGasLimit * 64/63 + overhead`.

The attack requires no special privileges, no leaked keys, and no trusted role. The only cost is the gas for the transaction itself. The attacker does not need to profit — this is a pure griefing attack against any Echo consumer.

---

### Recommendation

Add a gas sufficiency check before the callback, mirroring the pattern in `Entropy.sol`:

```solidity
// Ensure the calling context has enough gas to forward callbackGasLimit to the callback.
// Due to EVM's 63/64 rule, we need at least callbackGasLimit * 64/63 gas available here.
require(
    gasleft() >= uint256(req.callbackGasLimit) * 64 / 63 + GAS_OVERHEAD_FOR_CALLBACK,
    "Insufficient gas to execute callback"
);
```

If the check fails, revert the entire transaction so the request is preserved and can be retried with sufficient gas. Alternatively, adopt the `CALLBACK_FAILED` retry pattern from `Entropy.sol`.

### Proof of Concept

```
1. Provider registers with Echo, sets feePerGasInWei.
2. Consumer calls requestPriceUpdatesWithCallback(..., callbackGasLimit=1_000_000).
   - Consumer pays fee. Request stored with callbackGasLimit=1_000_000.
3. Attacker calls executeCallback{gas: 400_000}(provider, seqNum, updateData, priceIds).
   - 400_000 gas is enough to complete all pre-callback logic (Pyth parse, storage writes, clearRequest).
   - At the point of _echoCallback{gas: 1_000_000}(...), gasleft() ≈ 50_000.
   - EVM forwards min(1_000_000, 63/64 * 50_000) ≈ 49_218 gas to the callback.
   - Callback runs out of gas and reverts.
   - catch block emits PriceUpdateCallbackFailed.
   - Transaction succeeds. Request is permanently cleared.
4. Consumer's callback is permanently lost. No retry possible.
