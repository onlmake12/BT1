Audit Report

## Title
Remote peer triggers node panic via `get_block_uncles().expect()` on freezer-pruned uncle data in `GetBlocksProofProcess` — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary

When the CKB freezer is enabled, `delete_block_body` removes `COLUMN_BLOCK_UNCLE` for ancient blocks while leaving `COLUMN_BLOCK_HEADER` and `COLUMN_INDEX` intact. `GetBlocksProofProcess::execute` calls `get_block_uncles(&block_hash).expect("block uncles must be stored")` directly — without any freezer-aware fallback — after confirming `is_main_chain()`. A remote peer sending a `GetBlocksProof` message containing any frozen block hash causes an unconditional panic and node crash.

## Finding Description

In `get_blocks_proof.rs` lines 88–90, for each hash that passes `is_main_chain()`:

```rust
let uncles = snapshot
    .get_block_uncles(&block_hash)
    .expect("block uncles must be stored");
```

`get_block_uncles()` reads exclusively from `COLUMN_BLOCK_UNCLE` [1](#0-0)  and returns `None` when the key is absent.

`delete_block_body()` — called by `wipe_out_frozen_data` with the comment `// remain header` — explicitly deletes `COLUMN_BLOCK_UNCLE` (line 97) while leaving `COLUMN_BLOCK_HEADER` and `COLUMN_INDEX` intact. [2](#0-1) 

`wipe_out_frozen_data` in `shared/src/shared.rs` calls `batch.delete_block_body()` for every frozen block. [3](#0-2) 

`is_main_chain()` reads from `COLUMN_INDEX`, which is never deleted by the freezer, so it returns `true` for frozen blocks. [4](#0-3) 

`get_block()` in `ChainStore` has an explicit freezer-aware branch (lines 44–54) that retrieves the full block from flat files for frozen blocks, bypassing `get_block_uncles()`. `get_blocks_proof.rs` calls `get_block_uncles()` directly, skipping this path entirely. [5](#0-4) 

The panic site is line 88–90 of `get_blocks_proof.rs`: [6](#0-5) 

This handler is reachable from any connected peer via the light-client protocol message dispatch. [7](#0-6) 

## Impact Explanation

Any remote peer connected to a light-client-protocol-enabled node with the freezer active can crash the node process by sending a single `GetBlocksProof` message. This maps directly to the allowed CKB bounty impact: **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**. The panic unwinds the async task and terminates the process; no recovery is possible without a restart.

## Likelihood Explanation

Both the freezer and the light-client protocol server are non-default but documented, supported production features. Operators running storage-optimized or archival nodes are the target audience for the freezer. Once both features are enabled, every frozen block hash (all blocks older than the freeze threshold, publicly available from any block explorer) is a valid trigger. The attacker requires no credentials, no proof-of-work, and no special knowledge — only a network connection to the light-client protocol port and any old block hash.

## Recommendation

Replace the direct `get_block_uncles().expect()` call in `get_blocks_proof.rs` with a freezer-aware path, mirroring the logic in `get_block()` (`store/src/store.rs` lines 44–54): check if the block number falls below `freezer.number()`, and if so, retrieve the full block from the freezer flat file and extract uncles from it. Alternatively, return a `StatusCode::InternalError` instead of panicking when uncle data is absent for a main-chain block, consistent with how `reply_proof` handles MMR errors. [8](#0-7) 

## Proof of Concept

1. Enable the freezer and the light-client protocol server in the node config; let the chain advance past the freeze threshold so at least one block is frozen (uncle data deleted from `COLUMN_BLOCK_UNCLE`, index and header retained).
2. Connect a peer to the light-client protocol port.
3. Send a `GetBlocksProof` message where `block_hashes` contains any frozen block hash and `last_hash` is the current tip (passes `is_main_chain()`).
4. Execution path: `is_main_chain(frozen_hash)` → `true`; `get_block_header(frozen_hash)` → `Some(...)`; `get_block_uncles(frozen_hash)` → `None`; `.expect("block uncles must be stored")` → **panic** → node process crashes. [9](#0-8)
