### Title
Silently Ignored `add_tx` Return Value Causes Silent Transaction Loss During RBF Recovery - (File: `tx-pool/src/process.rs`)

### Summary
Inside `TxPoolService::submit_entry`, when RBF (Replace-By-Fee) removes conflicting transactions from the pool, the code attempts to re-add the displaced transactions back to the verify queue via `queue.add_tx(tx, false, None)`. The `Result<bool, Reject>` return value is explicitly discarded with `let _ = ...`. If the queue is full or the add fails for any reason, the displaced transactions are silently and permanently dropped from the pool with no error logged and no caller notification.

### Finding Description
In `tx-pool/src/process.rs`, the `submit_entry` method calls `process_rbf` to remove conflicting transactions and collect "recoverable" ones. It then spawns an async task to re-enqueue those transactions:

```rust
if !may_recovered_txs.is_empty() {
    let self_clone = self.clone();
    tokio::spawn(async move {
        let mut queue = self_clone.verify_queue.write().await;
        for tx in may_recovered_txs {
            debug!("recover back: {:?}", tx.proposal_short_id());
            let _ = queue.add_tx(tx, false, None);  // ← return value discarded
        }
    });
}
```

`queue.add_tx` returns `Result<bool, Reject>`. The `Reject` variants include `Reject::Full` (queue capacity exceeded) and others. By using `let _ = ...`, any failure is silently swallowed. No error is logged, no metric is emitted, and no callback is invoked for the dropped transaction.

The `process_rbf` function, called just before, has already removed the conflicting transactions from the pool and called `self.callbacks.call_reject(tx_pool, &old, reject)` on them with `Reject::RBFRejected`. Once removed, the only recovery path is through this `add_tx` call. If it fails silently, those transactions are permanently gone. [1](#0-0) [2](#0-1) 

### Impact Explanation
An unprivileged transaction submitter can exploit this to permanently evict legitimate pending transactions from the tx pool:

1. Attacker floods the verify queue to near-capacity (using many low-fee transactions).
2. Attacker submits a high-fee RBF transaction that conflicts with one or more legitimate transactions already in the pool.
3. `process_rbf` removes the legitimate transactions and marks them as `RBFRejected`.
4. The spawned recovery task calls `add_tx` for each displaced transaction, but the queue is full — `Reject::Full` is returned and silently discarded.
5. The displaced transactions are permanently lost from the pool. Their original submitters receive no notification.

This constitutes incorrect internal accounting: the pool believes the RBF replacement succeeded cleanly, but the displaced transactions that should have been recoverable are gone. A targeted attacker can use this to selectively censor specific transactions from the mempool without any trace in logs. [3](#0-2) 

### Likelihood Explanation
The verify queue has a finite capacity enforced by `VerifyQueue`. An attacker with the ability to submit transactions (any unprivileged RPC caller or P2P peer) can:
- Submit enough transactions to fill the verify queue (bounded by `max_tx_verify_cycles` and queue size limits).
- Then submit an RBF replacement targeting a victim transaction.

The attack requires no special privileges, no majority hashpower, and no trusted role. It is reachable via the standard `send_transaction` RPC or P2P transaction relay path. [4](#0-3) 

### Recommendation
Replace the silent discard with explicit error handling:

```rust
for tx in may_recovered_txs {
    debug!("recover back: {:?}", tx.proposal_short_id());
    if let Err(reject) = queue.add_tx(tx.clone(), false, None) {
        warn!(
            "failed to re-enqueue RBF-recovered tx {}: {}",
            tx.hash(),
            reject
        );
        // Optionally invoke reject callback so callers are notified
    }
}
```

At minimum, log the failure at `warn` or `error` level so operators can detect the condition. Ideally, invoke the reject callback so the original submitter is notified and the transaction is recorded in `recent_reject`.

### Proof of Concept
1. Fill the verify queue to capacity by submitting many transactions via `send_transaction` RPC.
2. Submit a valid RBF replacement transaction that conflicts with a legitimate transaction already in the pool (higher fee, same input).
3. `submit_entry` → `process_rbf` removes the legitimate transaction and calls `call_reject` on it with `RBFRejected`.
4. The spawned task calls `queue.add_tx(recovered_tx, false, None)` → returns `Err(Reject::Full)`.
5. `let _ = ...` discards the error. No log entry is produced.
6. Query the pool: the original legitimate transaction is gone. The attacker's RBF transaction is accepted. The displaced transaction is unrecoverable. [1](#0-0) [5](#0-4)

    pub(crate) async fn submit_entry(
        &self,
        pre_resolve_tip: Byte32,
        entry: TxEntry,
        mut status: TxStatus,
    ) -> (Result<(), Reject>, Arc<Snapshot>) {
        let (ret, snapshot) = self
            .with_tx_pool_write_lock(move |tx_pool, snapshot| {
                // check_rbf must be invoked in `write` lock to avoid concurrent issues.
                let conflicts = if tx_pool.enable_rbf() {
                    tx_pool.check_rbf(&snapshot, &entry)?
                } else {
                    // RBF is disabled but we found conflicts, return error here
                    // after_process will put this tx into conflicts_pool
                    let conflicted_outpoint =
                        tx_pool.pool_map.find_conflict_outpoint(entry.transaction());
                    if let Some(outpoint) = conflicted_outpoint {
                        return Err(Reject::Resolve(OutPointError::Dead(outpoint)));
                    }
                    HashSet::new()
                };

                // if snapshot changed by context switch we need redo time_relative verify
                let tip_hash = snapshot.tip_hash();
                if pre_resolve_tip != tip_hash {
                    debug!(
                        "submit_entry {} context changed. previous:{} now:{}",
                        entry.proposal_short_id(),
                        pre_resolve_tip,
                        tip_hash
                    );

                    // destructuring assignments are not currently supported
                    status = check_rtx(tx_pool, &snapshot, &entry.rtx)?;

                    let tip_header = snapshot.tip_header();
                    let tx_env = status.with_env(tip_header);
                    time_relative_verify(snapshot, Arc::clone(&entry.rtx), tx_env)?;
                }

                let may_recovered_txs = self.process_rbf(tx_pool, &entry, &conflicts);
                let evicted = _submit_entry(tx_pool, status, entry.clone(), &self.callbacks)?;

                // in a corner case, a tx with lower fee rate may be rejected immediately
                // after inserting into pool, return proper reject error here
                for evict in evicted {
                    let reject = Reject::Invalidated(format!(
                        "invalidated by tx {}",
                        evict.transaction().hash()
                    ));
                    self.callbacks.call_reject(tx_pool, &evict, reject);
                }

                tx_pool.remove_conflict(&entry.proposal_short_id());
                tx_pool
                    .limit_size(&self.callbacks, Some(&entry.proposal_short_id()))
                    .map_or(Ok(()), Err)?;

                if !may_recovered_txs.is_empty() {
                    let self_clone = self.clone();
                    tokio::spawn(async move {
                        // push the recovered txs back to verify queue, so that they can be verified and submitted again
                        let mut queue = self_clone.verify_queue.write().await;
                        for tx in may_recovered_txs {
                            debug!("recover back: {:?}", tx.proposal_short_id());
                            let _ = queue.add_tx(tx, false, None);
                        }
                    });
                }
                Ok(())
            })
            .await;

        (ret, snapshot)
    }
