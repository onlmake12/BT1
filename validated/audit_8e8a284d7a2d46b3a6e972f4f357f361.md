### Title
Unsigned Integer Underflow in `ema_price_no_older_than` Causes Revert When `publish_time` Exceeds Current Block Time - (File: `target_chains/fuel/contracts/pyth-contract/src/main.sw`)

---

### Summary

The private `ema_price_no_older_than` function in the Fuel Pyth contract performs an unchecked unsigned integer subtraction `current_time - price.publish_time`. If the stored EMA price's `publish_time` exceeds the current Fuel block timestamp (a realistic condition due to clock drift between Pythnet and the Fuel chain), the subtraction underflows and the transaction panics/reverts. The sibling function `price_no_older_than` already contains the correct saturating guard for this exact scenario, but `ema_price_no_older_than` was never updated to match.

---

### Finding Description

In `target_chains/fuel/contracts/pyth-contract/src/main.sw`, the private helper `ema_price_no_older_than` (lines 311–320) performs a bare unsigned subtraction:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,  // ← underflows if publish_time > current_time
        PythError::OutdatedPrice,
    );
    price
}
``` [1](#0-0) 

The sibling function `price_no_older_than` (lines 330–343) already recognizes and handles this exact underflow risk with an explicit comment and a conditional guard:

```sway
fn price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
``` [2](#0-1) 

The `publish_time` field is stored in TAI64 format (UNIX timestamp + `TAI64_DIFFERENCE = 4611686018427387904`), parsed directly from the Wormhole-attested price feed payload: [3](#0-2) 

The Fuel chain's `timestamp()` also returns TAI64. If Pythnet's clock is even slightly ahead of the Fuel chain's block timestamp at the moment a price update is stored, `price.publish_time > current_time` and the subtraction wraps to a huge `u64` value, causing the `require` to revert.

The two public entry points that route through the vulnerable private function are:

```sway
fn ema_price(price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(valid_time_period(), price_feed_id)
}

fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(time_period, price_feed_id)
}
``` [4](#0-3) 

Both are `#[storage(read)]` functions callable by any unprivileged user.

---

### Impact Explanation

Any consumer contract or user calling `ema_price()` or `ema_price_no_older_than()` on the Fuel Pyth contract will receive a revert/panic whenever the most recently stored EMA price has a `publish_time` that is ahead of the current Fuel block timestamp. This is a denial-of-service on the EMA price query path. Protocols relying on EMA prices for liquidations, collateral checks, or other time-sensitive operations will be unable to read the price during the affected window.

**Impact: Medium** — EMA price queries are completely blocked for the duration of the clock skew window; no funds are directly at risk but dependent protocols are disrupted.

---

### Likelihood Explanation

Clock drift between Pythnet (a Solana-based app-chain) and the Fuel blockchain is a normal operational condition. Pythnet block times and Fuel block times are independent; a price update published at Pythnet time `T` can be relayed and stored on Fuel before the Fuel chain's own clock reaches `T`. This is not a hypothetical edge case — it is the same scenario the developers already anticipated and fixed in `price_no_older_than` but missed in `ema_price_no_older_than`.

**Likelihood: Medium** — Requires only normal cross-chain clock drift, no attacker action needed; any relayer submitting a fresh price update can inadvertently trigger the condition.

---

### Recommendation

Apply the same saturating subtraction guard already present in `price_no_older_than` to `ema_price_no_older_than`:

```diff
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
+   // Mimicking saturating subtraction to avoid underflow
+   let time_difference = if current_time > price.publish_time {
+       current_time - price.publish_time
+   } else {
+       0
+   };
    require(
-       current_time - price.publish_time <= time_period,
+       time_difference <= time_period,
        PythError::OutdatedPrice,
    );
    price
}
```

### Proof of Concept

1. A relayer calls `update_price_feeds` with a valid Wormhole-attested accumulator update. The price feed's `publish_time` (in TAI64) is `T_pyth`, which is 2 seconds ahead of the current Fuel block timestamp `T_fuel` due to normal clock drift.
2. The price is stored successfully in `storage.latest_price_feed`.
3. A user (or a dependent DeFi protocol) calls `ema_price(price_feed_id)`.
4. Inside `ema_price_no_older_than`, `current_time = T_fuel` and `price.publish_time = T_pyth > T_fuel`.
5. `current_time - price.publish_time` underflows: `T_fuel - T_pyth` wraps to `u64::MAX - (T_pyth - T_fuel) + 1`, a value far exceeding any `time_period`.
6. The `require` condition fails and the transaction reverts with `PythError::OutdatedPrice`.
7. The same call to `price_no_older_than` for the spot price succeeds correctly because of its saturating guard. [1](#0-0) [2](#0-1)
