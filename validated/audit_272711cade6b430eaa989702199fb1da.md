### Title
Permissionless Out-of-Order Governance VAA Submission Permanently Blocks Lower-Sequence Actions — (`lazer/contracts/cardano/lib/pyth/governance.ak`)

### Summary
Any unprivileged relayer can submit a higher-sequence governance VAA before a lower-sequence one, permanently preventing the lower-sequence VAA from ever being applied. Because `GovernanceAction` is permissionless and `seen_sequence` is a strict monotone gate, this is a concrete, local-testable griefing path against any pending governance action.

### Finding Description
`parse_and_verify_action` enforces `governance.seen_sequence < seen_sequence` and then sets `seen_sequence` to the new value. [1](#0-0) 

There is no enforcement of submission order beyond this check. Given two valid guardian-signed VAAs:
- **S1 = 50** (`UpgradeWithdrawScript`)
- **S2 = 100** (`UpdateTrustedSigner`)

A relayer submits S2 first. The check `0 < 100` passes; `seen_sequence` becomes 100. Now S1 is submitted: `100 < 50` is `False` → `expect` fails → S1 is permanently unexecutable on-chain.

The `GovernanceAction` branch in `pyth_state.ak` has no owner/privilege gate, so any actor can trigger this. [2](#0-1) 

### Impact Explanation
The blocked VAA is an `UpgradeWithdrawScript` action. The withdraw script is not upgraded; the protocol remains on the old (potentially vulnerable) script. The `deprecated_withdraw_scripts` list is never updated with the new entry. Recovery requires the Wormhole governance body to issue a **new** VAA with sequence > 100 — this is an off-chain governance process that may be slow or contested, and the window of exposure on the old script persists until then. [4](#0-3) 

### Likelihood Explanation
The attacker needs only:
1. Access to two valid, guardian-signed governance VAAs (both are broadcast publicly via Wormhole's gossip network).
2. The ability to submit a Cardano transaction (fully permissionless).

No key compromise, no privileged role, no Sybil attack required. The attack is a single transaction.

### Recommendation
Replace the strict `seen_sequence < new_sequence` gate with an exact `seen_sequence + 1 == new_sequence` check (enforcing strict in-order processing), or maintain a set/bitmap of accepted sequences so that gaps are allowed but already-seen sequences are rejected without blocking future lower-sequence ones. The simplest safe fix is to require sequential processing:
```aiken
expect governance.seen_sequence + 1 == seen_sequence
``` [5](#0-4) 

### Proof of Concept
```

Step 1: Submit VAA(sequence=100, action=UpdateTrustedSigner)
  → check: 0 < 100  ✓
  → seen_sequence := 100
Step 2: Submit VAA(sequence=50, action=UpgradeWithdrawScript)
  → check: 100 < 50  ✗  → expect fails → transaction rejected
Result: UpgradeWithdrawScript is permanently blocked.
