### Title
Pyth Fee Accrual Uses `msg.value - providerFee` Instead of `pythFeeInWei`, Permanently Capturing User Overpayments - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `requestHelper` function credits the Pyth protocol treasury with `msg.value - providerFee` instead of the governance-configured `pythFeeInWei`. When a caller sends more ETH than the exact required fee (overpayment), the entire excess is permanently captured by `accruedPythFeesInWei` rather than being refunded. This is a direct analog to the OmniXMultisender bug: a fee is computed against the total submitted value rather than the intended component.

---

### Finding Description

`requestHelper` in `Entropy.sol` handles fee accounting as follows:

```solidity
// Lines 234–239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

`getFeeV2` is defined as:

```solidity
// Line 764
return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
```

So `requiredFee = providerFee + pythFeeInWei`. The provider is correctly credited with exactly `providerFee`. However, the Pyth treasury is credited with `msg.value - providerFee` — not `pythFeeInWei`. When `msg.value > requiredFee`, the surplus `(msg.value - requiredFee)` flows entirely into `accruedPythFeesInWei` and is never returned to the caller.

The interface documentation at `IEntropyV2.sol` line 96 explicitly states: *"excess value is **not** refunded to the caller."* This confirms the behavior is present in production and not guarded against. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Any ETH sent above the exact `requiredFee` is permanently captured by the Pyth protocol treasury. Users cannot recover it. Concretely:

- A user calls `getFeeV2(provider, gasLimit)` off-chain, gets `X` wei.
- Before their transaction lands, the provider calls `setProviderFee` and lowers their fee.
- The user's transaction executes with `msg.value = X`, but `requiredFee` is now `X - delta`.
- `accruedPythFeesInWei` is incremented by `X - providerFee`, which is `pythFeeInWei + delta` — capturing `delta` wei that belongs to the user.

The same outcome occurs whenever any caller (contract or EOA) sends a buffer above the quoted fee to avoid reverts, a common defensive pattern in fee-paying contracts.

**Impact: HIGH** — direct, unrecoverable loss of user funds (ETH) with no on-chain recourse.

---

### Likelihood Explanation

**Likelihood: MEDIUM**

- The `IEntropyV2` documentation explicitly warns that provider fees change over time and instructs callers to re-query before each call, implying fee changes are routine.
- Integrating contracts (e.g., `EntropyTester.sol` line 136–137) call `getFeeV2` and immediately forward the result, but any latency between the view call and the transaction inclusion creates a window.
- Contracts that add a small ETH buffer to avoid reverts (a standard defensive pattern) will silently overpay on every call.
- No refund path exists; the loss is silent and permanent. [3](#0-2) [4](#0-3) 

---

### Recommendation

Replace the `msg.value`-based Pyth fee accrual with the governance-configured constant, and refund any excess to the caller:

```solidity
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += _state.pythFeeInWei;

uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

This mirrors the fix recommended for OmniXMultisender: charge fees only on the intended component, not on the total submitted value.

---

### Proof of Concept

1. Deploy `Entropy` with `pythFeeInWei = 100 wei` and a provider with `feeInWei = 200 wei`.
2. `getFeeV2(provider, 0)` returns `300 wei`.
3. Provider calls `setProviderFee(150 wei)` — fee drops to `250 wei` total.
4. User submits `requestV2{value: 300}(provider, userRandom, 0)`.
5. Inside `requestHelper`:
   - `requiredFee = 250`, check passes (`300 >= 250`).
   - `providerFee = 150`, provider accrues `150 wei` ✓.
   - `accruedPythFeesInWei += 300 - 150 = 150 wei` — Pyth accrues `150 wei` instead of `100 wei`.
6. User loses `50 wei` permanently; Pyth treasury gains `50 wei` above governance intent. [5](#0-4) [6](#0-5)
