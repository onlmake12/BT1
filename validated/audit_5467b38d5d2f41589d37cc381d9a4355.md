### Title
CPU Amplification DoS via Unbounded CBMT Work in `GetTransactionsProof` Handler — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

### Summary

The `GetTransactionsProofProcess::execute` handler bounds the number of requested tx_hashes to `GET_TRANSACTIONS_PROOF_LIMIT = 1000`, but the actual CPU work per request is `O(limit × max_txs_per_block)` because `CBMT::build_merkle_proof` is called once per distinct block and is fed **all** transactions in that block, not just the requested ones.

### Finding Description

The limit check at lines 37–39 only restricts the count of incoming tx_hashes: [1](#0-0) 

After the check, found tx_hashes are grouped by `block_hash`: [2](#0-1) 

Then, for **each distinct block**, the server fetches the full block and passes **all** of its transactions to `CBMT::build_merkle_proof`: [3](#0-2) 

The leaf input to `CBMT::build_merkle_proof` is `block.transactions().iter().map(|tx| tx.hash()).collect()` — every transaction in the block, regardless of how many were requested. If an attacker sends 1000 tx_hashes each from a different block, the server performs up to 1000 separate CBMT tree constructions, each over the full transaction set of its respective block. Additionally, `block.calc_witnesses_root()` and `snapshot.get_block_uncles()` are called once per block, adding further I/O and hashing work. [4](#0-3) 

There is no rate limiting, per-peer request throttling, or cap on the number of distinct blocks in `lib.rs`: [5](#0-4) 

The constant file confirms the limit is 1000: [6](#0-5) 

### Impact Explanation

Total CBMT work per request: `O(GET_TRANSACTIONS_PROOF_LIMIT × avg_txs_per_block)`. On a busy mainnet where blocks contain hundreds of transactions, a single well-crafted request multiplies the server's CPU work by that factor. An attacker can pipeline many such requests from one or more peers, causing sustained CPU exhaustion on any full node serving the light client protocol.

### Likelihood Explanation

- The attacker is fully unprivileged; any peer that speaks the light client protocol can send `GetTransactionsProof`.
- The required tx_hashes are public chain data, trivially collected by observing the chain.
- The message is structurally valid (not malformed), so the ban-on-bad-message path is never triggered.
- No rate limiting or per-peer quota exists in the handler or the protocol dispatcher.

### Recommendation

1. **Cap the number of distinct blocks** a single `GetTransactionsProof` request may span (e.g., to a small constant like 10 or 20).
2. Alternatively, bound total work by tracking the sum of `block.transactions().len()` across all referenced blocks and rejecting requests that exceed a threshold.
3. Add per-peer rate limiting on `GetTransactionsProof` messages at the protocol dispatcher level.

### Proof of Concept

1. Identify 1000 mainnet blocks each containing many transactions (e.g., 500+ txs).
2. From each block, pick one transaction hash.
3. Construct a `GetTransactionsProof` message with all 1000 tx_hashes and a valid `last_hash`.
4. Send the message to a light-client-serving CKB node.
5. Observe: the server calls `CBMT::build_merkle_proof` 1000 times, each iterating over 500+ transaction hashes — ~500,000 hash operations for a single 1000-element request, versus the ~1000 operations the limit was intended to bound.
6. Repeat at the maximum message rate; CPU usage scales linearly with block fullness, not with the nominal limit.
