Audit Report

## Title
Wrong Row Variable in `script_exists_in_output` Silently Deletes Referenced Type Scripts on PostgreSQL During Reorg — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary
At line 252 of `script_exists_in_output`, the code reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, `row_lock` already holds a `false` boolean (the lock-script check returned false, otherwise the function would have exited early), so `try_get::<bool, _>(0)` succeeds and returns `Ok(false)` unconditionally, causing `rollback_block` to delete every type script from the rolled-back block even when earlier blocks' outputs still reference them. The `script` table becomes permanently inconsistent with the `output` table until a full re-sync.

## Finding Description
`script_exists_in_output` is a two-phase guard called by `rollback_block` to decide whether a script row is safe to delete after its referencing outputs are removed.

**Phase 1 (lines 208–235):** Queries `SELECT EXISTS(SELECT 1 FROM output WHERE lock_script_id = $1)` into `row_lock`. On PostgreSQL, `row_lock.try_get::<bool, _>(0)` succeeds. If the result is `true`, the function returns `Ok(true)` early. If `false`, execution falls through — meaning `row_lock` now holds a confirmed `false` boolean.

**Phase 2 (lines 237–256):** Queries `SELECT EXISTS(SELECT 1 FROM output WHERE type_script_id = $1)` into `row_type`. The result is fetched correctly into `row_type`. However, line 252 reads:

```rust
match row_lock.try_get::<bool, _>(0) {   // BUG: should be row_type
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

On PostgreSQL, `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL `EXISTS` returns `BOOLEAN`) and returns `Ok(false)` — the value already established in Phase 1. `row_type` is fetched but never read. The function always returns `false` for the type-script existence check on PostgreSQL.

On SQLite, `row_lock.try_get::<bool, _>(0)` fails (SQLite stores `EXISTS` as `BIGINT`), so the `Err` branch correctly reads `row_type.get::<i64, _>(0) == 1`. SQLite is unaffected.

`rollback_block` calls this guard for every type script in the rolled-back block:

```rust
if let Some(type_script_id) = type_script_id
    && !script_exists_in_output(type_script_id, tx).await?
{
    script_id_list_to_remove.push(type_script_id);
}
```

Because the guard always returns `false` on PostgreSQL, every type script is pushed into `script_id_list_to_remove` and deleted via `remove_batch_by_blobs("script", "id", &script_id_list_to_remove, tx)`. Outputs in earlier blocks that share the same `type_script_id` now hold dangling foreign-key references to deleted `script` rows. [1](#0-0) [2](#0-1) 

## Impact Explanation
This is a confirmed implementation bug in the CKB rich-indexer's state storage mechanism. The `script` table is a core indexer storage table; permanent inconsistency between `script` and `output` corrupts all subsequent RPC queries that join these tables (e.g., cell queries filtered by type script). This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**. The rich-indexer is the CKB node's optional but official state storage and query layer; its persistent data corruption is a direct implementation defect in that mechanism.

## Likelihood Explanation
Chain reorgs are a normal, unprivileged network event requiring no attacker capability. Any reorg on a PostgreSQL-backed rich-indexer node triggers the bug for every type script that appears in the rolled-back block's outputs and is not simultaneously used as a lock script — the common case for application-specific type scripts (UDT, NFT, DAO). The bug is deterministic and repeatable on every reorg.

## Recommendation
Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

## Proof of Concept
1. Start a CKB rich-indexer backed by PostgreSQL.
2. Index Block A containing output O1 with `type_script_id = S1`, where S1 is not used as any lock script.
3. Index Block B containing output O2 also with `type_script_id = S1`.
4. Trigger a reorg that rolls back Block B (`rollback_block` is called).
5. After rollback, execute `SELECT * FROM script WHERE id = S1`.
6. **Expected:** S1 is present (O1 in Block A still references it).
7. **Actual (PostgreSQL):** S1 is absent — deleted because `script_exists_in_output` returned `false` by reading `row_lock` (which holds `false` from Phase 1) instead of `row_type`.
8. Query outputs for O1 via RPC; the type script data is missing or the join returns no row.
