### Title
Permanent Protocol Fee Lock: All SUI Update Fees Deposited into `PriceInfoObject` Are Irrecoverable — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every call to `update_single_price_feed` deposits SUI fee coins into a `PriceInfoObject`'s dynamic object field. No function exists anywhere in the contract to withdraw these coins. The `fee_recipient_address` in `State` is dead code for fee collection. All protocol fee revenue is permanently frozen.

### Finding Description
`pyth::pyth::update_single_price_feed` charges a `base_update_fee` in SUI and routes it via `price_info::deposit_fee_coins` into the calling `PriceInfoObject`'s dynamic object field keyed by `FEE_STORAGE_KEY`.

```
// pyth.move line 277
price_info::deposit_fee_coins(price_info_object, fee);
```

`deposit_fee_coins` is `public` and only ever accumulates coins via `coin::join`. There is no corresponding `withdraw_fee_coins`, no admin extraction function, and no governance action that touches `FEE_STORAGE_KEY`. The `set_fee_recipient` governance module, which previously routed fees to `fee_recipient_address`, now carries an explicit comment that it is no longer used. The `fee_recipient_address` field in `State` is read-only dead code with respect to actual fee flows.

### Impact Explanation
Every `update_single_price_feed` call permanently locks `base_update_fee` MIST of SUI inside the corresponding `PriceInfoObject`. With N price feeds each updated K times, the total locked amount is `N × K × base_update_fee`. These funds are irrecoverable under the current contract. The protocol collects fees but can never access them, constituting a complete protocol fund freeze that grows monotonically with usage.

### Likelihood Explanation
Certainty, not likelihood — the condition is structural. Every single successful price update call triggers the lock. No special attacker action is required; normal protocol operation causes the freeze. The `set_fee_recipient.move` comment confirms the design regression is intentional (old path removed) but the replacement withdrawal path was never implemented.

### Recommendation
Add a privileged `withdraw_fee_coins` function (gated by `LatestOnly` or a governance action) to `price_info.move` that extracts the `Coin<SUI>` from `FEE_STORAGE_KEY` and transfers it to `state::get_fee_recipient(pyth_state)`. Alternatively, revert to the previous design of transferring fees directly to the recipient address at update time instead of accumulating them in the object.

### Proof of Concept
**Invariant test (pseudocode):**
```
1. Call setup_test with base_update_fee = 50
2. Create price feeds
3. Call update_single_price_feed K times, each with fee = 50 SUI
4. Assert price_info::get_balance(price_info_object) == K * 50  // fees accumulated
5. Exhaustively search all public/public(friend) functions in price_info.move and pyth.move
   for any that call dynamic_object_field::remove or coin::split on FEE_STORAGE_KEY
6. Assert: none exist  // confirmed by grep returning zero matches
7. Assert: fee_recipient address balance unchanged  // fees never routed out
```

The existing test `test_create_and_update_price_feeds_with_batch_attestation_success` already asserts step 4 (`price_info::get_balance(&price_info_object_1)==DEFAULT_BASE_UPDATE_FEE`) but never tests that the balance can be reduced — because it cannot. [5](#0-4)
