### Title
`updateFeeds()` Drains Entire Contract Balance Instead of Refunding Only Excess ETH — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

### Summary

`PythAggregatorV3.updateFeeds()` refunds leftover ETH to `msg.sender` using `address(this).balance` instead of `msg.value - fee`. This is a direct analog of the RouteProcessor4 bug: any ETH that accumulates in the contract (from failed refunds) can be stolen by the next caller.

### Finding Description

In `PythAggregatorV3.sol`, the `updateFeeds()` function computes the required Pyth update fee, pays it, and then attempts to refund the remainder to the caller:

```solidity
function updateFeeds(bytes[] calldata priceUpdateData) public payable {
    uint fee = pyth.getUpdateFee(priceUpdateData);
    pyth.updatePriceFeeds{value: fee}(priceUpdateData);

    // refund remaining eth
    // solhint-disable-next-line no-unused-vars
    (bool success, ) = payable(msg.sender).call{
        value: address(this).balance   // <-- BUG: should be msg.value - fee
    }("");
}
``` [1](#0-0) 

Two compounding flaws exist:

1. **Wrong refund amount**: `address(this).balance` is used instead of `msg.value - fee`. If the contract holds any pre-existing ETH (see below), the entire balance is sent to the current caller.
2. **Unchecked return value**: The `success` boolean is explicitly suppressed with a `no-unused-vars` comment. If the refund call fails (e.g., `msg.sender` is a contract without a `receive()` function), the ETH silently remains in the contract. [2](#0-1) 

### Impact Explanation

ETH accumulates in `PythAggregatorV3` whenever a caller is a contract that rejects ETH (no `receive`/`fallback`). Once any ETH is stranded in the contract, any subsequent unprivileged caller of `updateFeeds()` — even one sending only the exact fee amount — will receive the entire accumulated balance via `address(this).balance`. This constitutes direct theft of funds from the contract.

### Likelihood Explanation

`updateFeeds()` is a `public payable` function callable by any address. Integrators commonly call it from other contracts (e.g., keeper bots, automation contracts) that may not implement `receive()`. A single failed refund is sufficient to leave ETH stranded. An attacker can then immediately drain it in the next transaction with zero privilege required.

### Recommendation

Replace `address(this).balance` with the exact excess amount:

```solidity
- (bool success, ) = payable(msg.sender).call{value: address(this).balance}("");
+ uint256 excess = msg.value - fee;
+ if (excess > 0) {
+     (bool success, ) = payable(msg.sender).call{value: excess}("");
+     require(success, "Refund failed");
+ }
```

### Proof of Concept

1. Victim contract `V` (no `receive()`) calls `updateFeeds{value: 1 ether}(data)` where `fee = 0.001 ether`. After Pyth is paid, the refund of `0.999 ether` to `V` fails silently. `PythAggregatorV3` now holds `0.999 ether`.
2. Attacker EOA calls `updateFeeds{value: 0.001 ether}(data)` with the same or any valid update data. `fee = 0.001 ether` is paid to Pyth. The refund sends `address(this).balance = 0.999 ether` to the attacker.
3. Attacker receives `0.999 ether` at zero cost beyond the fee. [1](#0-0)
