Audit Report

## Title
Per-removal `shrink_to_fit` on `MultiIndexVerifyEntryMap` causes repeated O(N) reallocation during queue drain â€” (`tx-pool/src/component/verify_queue.rs`)

## Summary

`VerifyQueue::remove_tx` unconditionally calls `self.shrink_to_fit()` on every successful removal. The `shrink_to_fit!` macro guard uses `SHRINK_THRESHOLD = 100`, which is structurally too small relative to the post-shrink capacity surplus introduced by the backing store's load factor. For queues above approximately 700 entries, every `pop_front` triggers an O(N) reallocation across the multi-index map's hash-indexed structures while the `verify_queue` write lock is held, degrading tx verification throughput for all peers sharing the node. The attack requires only structurally valid transactions (no UTXO lookup, no real CKB tokens) submitted via the unprivileged P2P relay or `send_transaction` RPC.

## Finding Description

**Root cause â€” `remove_tx` calls `shrink_to_fit` on every removal:**

`VerifyQueue::remove_tx` at line 146 calls `self.shrink_to_fit()` unconditionally inside the `.map()` closure, executing on every successful removal. [1](#0-0) 

**The macro guard:**

`VerifyQueue::shrink_to_fit` invokes the `shrink_to_fit!` macro with `SHRINK_THRESHOLD = 100`. The guard fires when `capacity > len + 100`. [2](#0-1) [3](#0-2) [4](#0-3) 

**Why the guard fails â€” the self-reinforcing cycle:**

`MultiIndexVerifyEntryMap` is generated by the `multi_index_map` crate (v0.15.1) and contains two hashed indices (`id`: `hashed_unique`, `is_large_cycle`: `hashed_non_unique`) backed by hashbrown hash maps via `rustc-hash`. [5](#0-4) 

After `shrink_to_fit()` executes on the hash map indices, hashbrown reallocates to minimum capacity for the current `len`, approximately `capacity_after_shrink â‰ˆ len Ã— 8/7 â‰ˆ 1.143 Ã— len`. On the very next pop: `capacity = 1.143N`, `len = N âˆ’ 1`. Guard check: `1.143N > (N âˆ’ 1) + 100` â†’ `0.143N > 99` â†’ **N > 692**. For any queue above ~700 entries, every single pop triggers another O(N) shrink (rehashing all hash-indexed fields), which resets capacity to `1.143 Ã— (Nâˆ’1)`, which again satisfies the condition on the next pop. The threshold of 100 is structurally insufficient relative to the post-shrink capacity surplus.

**The `pop_front` path used by the verify worker:**

`pop_front` calls `remove_tx`, triggering the shrink on every normal verification cycle. [6](#0-5) 

**The verify worker acquires the write lock during `pop_front`:**

The `verify_queue` write lock is held for the entire duration of `pop_front` (including `shrink_to_fit`), blocking all concurrent `add_tx` calls. [7](#0-6) 

**The attacker's entry path:**

`submit_remote_tx` â†’ `resumeble_process_tx` â†’ `non_contextual_verify` â†’ `enqueue_verify_queue`. The `non_contextual_verify` check requires only structural validity (version, size, empty check, duplicate deps, outputs data, script hash type) â€” no UTXO lookup, no script execution, no real CKB tokens. [8](#0-7) [9](#0-8) [10](#0-9) 

**The queue size limit:**

The queue accepts up to 256 MB of total serialized transaction data â€” far above the ~700-entry threshold even at conservative transaction sizes. [11](#0-10) 

## Impact Explanation

Once the queue exceeds ~700 entries, every call to `pop_front` triggers an O(N) `MultiIndexVerifyEntryMap` reallocation (rehashing all hash-indexed fields). The `verify_queue` write lock is held for the duration, blocking concurrent `add_tx` operations. The total cost to drain a queue of N entries becomes O(NÂ²) instead of O(N). An attacker who fills the queue with structurally valid transactions (no real CKB tokens required) and sustains the submission rate can continuously degrade tx verification throughput for all peers sharing the node.

This maps to: **High (10001â€“15000 points) â€” Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The attack requires only: (1) submitting transactions that pass `non_contextual_verify` (structural validity only â€” inputs may reference nonexistent UTXOs); (2) maintaining the queue above ~700 entries (well below the 256 MB cap); (3) sustaining the submission rate to keep the queue populated. Both the P2P relay path and the `send_transaction` RPC are unprivileged. The attacker spends O(N) bandwidth per unit time; the node pays O(NÂ²) processing cost to drain the queue. The asymmetry is favorable to the attacker.

## Recommendation

Remove the `shrink_to_fit` call from `remove_tx`. Shrinking should be decoupled from the hot removal path entirely. Acceptable alternatives: (a) shrink only when the queue transitions from non-empty to empty (as `clear` already does); (b) shrink periodically every K removals where K is large enough to amortize the cost; (c) if per-removal shrinking is desired, the threshold must exceed `len / 7` to account for hashbrown's post-shrink capacity surplus â€” but this is impractical as a static constant. The correct fix is option (a) or (b). [12](#0-11)

    pub fn remove_tx(&mut self, id: &ProposalShortId) -> Option<Entry> {
        self.inner.remove_by_id(id).map(|e| {
            let tx_size = e.inner.tx.data().serialized_size_in_block();
            if let Some(total_tx_size) = self.total_tx_size.checked_sub(tx_size) {
                self.total_tx_size = total_tx_size;
            } else if let Some(total_tx_size) = self.recompute_total_tx_size() {
                error!(
                    "verify_queue total_tx_size {} underflowed by sub {}, recomputed {}",
                    self.total_tx_size, tx_size, total_tx_size

                self.total_tx_size = total_tx_size;
            } else {
                error!(
                    "verify_queue total_tx_size {} underflowed by sub {}, and recomputing overflowed",
                    self.total_tx_size, tx_size
                );
            }
            self.shrink_to_fit();
            e.inner
        })
    }

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
