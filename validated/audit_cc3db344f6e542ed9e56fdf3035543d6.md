### Title
Default-Disabled JSON-RPC Batch Limit Allows Unbounded Request Count to Exhaust RPC Server Resources - (`rpc/src/server.rs`)

---

### Summary

The CKB JSON-RPC server enforces a batch-request count limit only when the operator explicitly sets `rpc_batch_limit` in the config. By default this limit is `None`, meaning the guard in `handle_jsonrpc` is never reached. Any caller who can reach the RPC endpoint can submit a single HTTP/TCP/WS request containing an arbitrarily large batch of expensive calls, exhausting CPU, memory, and I/O on the node process. This is a direct analog to the Hats Protocol M-17 finding: an unbounded user-controlled field (batch size instead of string length) causes excessive resource consumption when processed.

---

### Finding Description

**Root cause — `rpc/src/server.rs`**

`JSONRPC_BATCH_LIMIT` is a process-global `OnceLock<usize>` that is only populated when the operator sets `rpc_batch_limit` in `ckb.toml`:

```rust
// rpc/src/server.rs  line 34
static JSONRPC_BATCH_LIMIT: OnceLock<usize> = OnceLock::new();

// line 53-55
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

The batch-size guard in `handle_jsonrpc` is conditional on the lock being populated:

```rust
// rpc/src/server.rs  lines 275-282
Request::Batch(calls) => {
    if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
        && calls.len() > *batch_size
    {
        return make_error_response(...);
    }
    // ALL calls are processed when limit is absent
``` [1](#0-0) [2](#0-1) [3](#0-2) 

The default production config ships with the option commented out:

```toml
# By default, there is no limitation on the size of batch request size
# a huge batch request may cost a lot of memory or makes the RPC server slow,
# to avoid this, you may want to add a limit for the batch request size.
# rpc_batch_limit = 2000
``` [4](#0-3) 

The config struct field is `Option<usize>` with no default, so it is `None` unless the operator explicitly sets it: [5](#0-4) 

**Attack path**

The HTTP endpoint is bounded by `max_request_body_size = 10 MiB`. Within 10 MiB an attacker can pack roughly 100 000+ minimal JSON-RPC calls (e.g. `get_tip_block_number` is ~70 bytes each). Each call is dispatched and awaited sequentially inside the batch handler:

```rust
let stream = stream::iter(calls)
    .then(move |call| {
        let io = Arc::clone(&io);
        async move { io.handle_call(call, T::default()).await }
    })
    .filter_map(|response| async move { response });
``` [6](#0-5) 

For heavier methods (`get_block`, `get_transaction`, `send_transaction`) each call performs database reads or full script verification. A batch of thousands of such calls ties up the async runtime, exhausts RocksDB I/O bandwidth, and can spike memory to gigabytes before the response is flushed.

The TCP subscription endpoint (`tcp_listen_address`) uses `LinesCodec::new_with_max_length(2 * 1024 * 1024)` — a 2 MiB per-line limit — but applies the same unguarded batch handler and has no connection-level request-count limit: [7](#0-6) 

---

### Impact Explanation

An attacker who can reach the RPC endpoint can:

1. **Exhaust async worker threads** by flooding the batch handler with CPU-heavy calls (`send_transaction` triggers full CKB-VM script verification up to 70 M cycles each).
2. **Exhaust RocksDB I/O** with thousands of `get_block` / `get_transaction` calls in a single batch, starving the chain service and tx-pool of database access.
3. **Exhaust heap memory** because all batch responses are buffered in a `StreamBodyAs::json_array` stream before being flushed to the client.

The result is a denial-of-service of the RPC server, and potentially the entire node process if memory pressure triggers OOM. The node's block-processing and P2P relay functions share the same process and async runtime.

---

### Likelihood Explanation

- The HTTP RPC binds to `127.0.0.1:8114` by default, limiting the attacker to the local machine. However, many operators expose the RPC publicly (explicitly supported and documented), and any local process (e.g. a malicious dApp, a compromised miner client) can reach it.
- The TCP subscription endpoint (`tcp_listen_address`) is commonly enabled for wallet/dApp subscriptions and is frequently bound to `0.0.0.0`, making it reachable from the network.
- The default config ships with `rpc_batch_limit` commented out and no enforcement in code when the option is absent, so the vast majority of production nodes are unprotected.
- No privileged key or special role is required; any JSON-RPC client can send a batch request.

---

### Recommendation

1. **Set a safe default** for `rpc_batch_limit` in the `Config` struct (e.g. 100–1000) so that protection is active without operator action:

```rust
// util/app-config/src/configs/rpc.rs
pub rpc_batch_limit: Option<usize>,
// change to:
#[serde(default = "default_rpc_batch_limit")]
pub rpc_batch_limit: usize,

fn default_rpc_batch_limit() -> usize { 200 }
```

2. **Enforce the limit unconditionally** in `handle_jsonrpc` rather than behind an `Option` guard.
3. **Apply the same limit to the TCP server**, which currently has no batch-count enforcement at all.

---

### Proof of Concept

```bash
# Build a batch of 50 000 get_tip_block_number calls (~3.5 MB, within 10 MiB limit)
python3 -c "
import json, sys
calls = [{'id': i, 'jsonrpc': '2.0', 'method': 'get_tip_block_number', 'params': []}
         for i in range(50000)]
sys.stdout.write(json.dumps(calls))
" > batch.json

# Send to a node with default config (no rpc_batch_limit set)
curl -s -X POST http://127.0.0.1:8114 \
     -H 'Content-Type: application/json' \
     --data-binary @batch.json > /dev/null &

# Observe: node CPU spikes to 100%, RPC becomes unresponsive for the duration
# Repeat with heavier methods (get_block with large block hashes) to amplify I/O impact
```

The node's RPC handler will process all 50 000 calls sequentially before returning, blocking the async executor and starving other RPC clients and internal services of scheduling time.
