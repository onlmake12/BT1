### Title
Subscription-Level Timestamp in `getPricesNoOlderThan` / `getEmaPricesNoOlderThan` Bypasses Per-Feed Staleness, Returning Arbitrarily Old Prices Without Revert — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` in `Scheduler.sol` validate freshness using `status.priceLastUpdatedAt`, which is the **maximum** `publishTime` across all feeds in the subscription batch. For subscriptions containing closed-market feeds (e.g., equities alongside 24/7 crypto feeds), individual feeds can carry arbitrarily old `publishTime` values while the subscription-level timestamp remains fresh. The staleness check passes, and stale prices are silently returned to callers who believe they received a freshness guarantee.

---

### Finding Description

When `updatePriceFeeds` is called, `_validateShouldUpdatePrices` selects the **maximum** `publishTime` across all feeds in the batch and stores it as `status.priceLastUpdatedAt`:

```solidity
// _validateShouldUpdatePrices — lines 362–370
// Use the most recent timestamp, as some asset markets may be closed.
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

This maximum is then stored:

```solidity
status.priceLastUpdatedAt = latestPublishTime; // line 340
```

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` then check freshness against this single aggregate value:

```solidity
// getPricesNoOlderThan — lines 551–552
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds); // returns individual feed prices
```

The same pattern appears in `getEmaPricesNoOlderThan` at lines 594–597.

The check compares `block.timestamp` against the **subscription-level maximum**, not against the `publishTime` of each individual feed being returned. A subscription containing BTC/USD (updated every few seconds, 24/7) and AAPL/USD (last updated 16+ hours ago when the market closed) will have `priceLastUpdatedAt` driven entirely by BTC/USD. A caller requesting only AAPL/USD via `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)` will pass the staleness check and receive a price that is 16+ hours old — with no revert.

The `SchedulerConstants.sol` explicitly acknowledges this design for closed-market feeds:

```solidity
// SchedulerConstants.sol line 17–21
/// @dev Note: We don't use this when parsing update data from the Pyth contract
/// because don't want to reject update data if it contains a price from a market
/// that closed a few days ago, since it will contain a timestamp from the last
/// trading period.
uint64 public constant PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours;
```

The design intentionally allows closed-market feeds with old timestamps to coexist in a subscription, but the staleness-checking read path does not account for this.

---

### Impact Explanation

Any whitelisted reader of a mixed subscription (active + closed-market feeds) calling `getPricesNoOlderThan` or `getEmaPricesNoOlderThan` receives a false freshness guarantee. The function is documented as "a sanity-checked version of `getPricesUnsafe`" that "reverts if the price wasn't updated sufficiently recently." This contract is broken for closed-market feeds. Downstream protocols relying on this revert-on-stale guarantee for collateral valuation, liquidation triggers, or trade execution will silently consume stale prices, enabling incorrect financial outcomes (e.g., under-collateralized loans not liquidated, trades executed at stale equity prices).

---

### Likelihood Explanation

High. The Scheduler explicitly supports and expects mixed subscriptions with closed-market feeds — this is a documented design choice. Equity markets are closed ~16 hours per weekday and ~48 hours over weekends. Any subscription combining a 24/7 feed (crypto) with an equity feed will exhibit this behavior during every market-close window. No special attacker action is required; the condition arises from normal market operation.

---

### Recommendation

Replace the subscription-level aggregate staleness check with a per-feed check against each returned price's individual `publishTime`. In `getPricesNoOlderThan` and `getEmaPricesNoOlderThan`, after retrieving the feeds via `_getPricesInternal`, iterate over each returned price and verify `distance(block.timestamp, prices[i].publishTime) <= age_seconds` before returning. This mirrors the per-price staleness check used in `AbstractPyth.getPriceNoOlderThan` (lines 56–57 of `AbstractPyth.sol`).

---

### Proof of Concept

1. Create a subscription with two feeds: `BTC_ID` (24/7) and `AAPL_ID` (equity, market hours).
2. Call `updatePriceFeeds` with update data where `BTC publishTime = block.timestamp` and `AAPL publishTime = block.timestamp - 72000` (20 hours ago, market closed).
3. `_validateShouldUpdatePrices` sets `updateTimestamp = max(block.timestamp, block.timestamp - 72000) = block.timestamp`.
4. `status.priceLastUpdatedAt = block.timestamp`.
5. Caller invokes `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)` (60-second age limit).
6. Check: `distance(block.timestamp, block.timestamp) = 0 ≤ 60` → **does not revert**.
7. `getPricesUnsafe` returns AAPL price with `publishTime = block.timestamp - 72000` (20 hours stale).
