**The guard** (lines 100–103):
```rust
require!(
    !ctx.accounts.emitter_authority.executable,
    CoreBridgeError::ExecutableDisallowed
```

**The flaw**: The check only tests the *current* `executable` flag. On Solana, when a program is closed via the BPF Upgradeable Loader's `Close` instruction (discriminator `[5,0,0,0]`), the runtime marks the program account `executable=false`. The attacker still holds the deploy keypair (whose pubkey IS the program address), and `Signer<'info>` only verifies the transaction signature — it does not check account state, lamports, or ownership.

**The attack path is concrete**:
1. Attacker deploys program at address P (permissionless, costs SOL)
2. Program builds downstream trust using `cpi_program_id=Some(P)` (PDA emitter path)
3. Attacker closes the program → account at P becomes `executable=false`
4. Attacker calls `init_message_v1(emitter_authority=K, cpi_program_id=None)` signing with keypair K (where `K.pubkey = P`)
5. `!executable` check passes (false → `!false = true`)
6. `PostedMessageV1.emitter` is set to P

This bypasses the PDA requirement: when active, the program must use the `[b"emitter"]` PDA as authority; after closing, the raw keypair can be used directly to emit with `emitter=P`.

### Title
Closed-program keypair bypasses executable guard in `init_message_v1`, enabling emitter impersonation — (`target_chains/solana/programs/core-bridge/src/processor/post_message/init_message_v1.rs`)

### Summary
The `ExecutableDisallowed` guard in `init_message_v1` checks only the current `executable` flag of `emitter_authority`. After a program is closed via the BPF Upgradeable Loader, its account becomes `executable=false`, causing the guard to pass. An attacker who holds the original deploy keypair can then sign as `emitter_authority` with `cpi_program_id=None` and emit Wormhole messages with `emitter=closed_program_pubkey`, impersonating the formerly deployed program.

### Finding Description
In the `None` branch of `init_message_v1`, the code intends to block programs from being used as legacy emitters: [1](#0-0) 

The guard reads `ctx.accounts.emitter_authority.executable` at call time. The BPF Upgradeable Loader's `Close` instruction sets `executable=false` on the program account. After closing, the deploy keypair still signs valid transactions (Solana's `Signer` constraint is a pure signature check), so the attacker passes both the `Signer` constraint and the `!executable` guard simultaneously.

The `cpi_program_id=Some(program_id)` path enforces a PDA (`[b"emitter"]`) as authority, preventing raw-keypair signing: [2](#0-1) 

Closing the program lets the attacker escape to the `None` branch and use the raw keypair, bypassing that PDA requirement entirely.

### Impact Explanation
The attacker emits valid `PostedMessageV1` messages with `emitter = closed_program_pubkey`. Any downstream Wormhole consumer (cross-chain bridge, oracle relay, governance contract) that whitelists that emitter address by pubkey will accept these forged messages as authentic. The severity depends on what the closed program was previously authorized to emit (price updates, governance actions, etc.).

### Likelihood Explanation
The attack is permissionless: deploying a Solana program requires only SOL and a keypair. The attacker controls the timing of the close and the subsequent `init_message_v1` call. No privileged access, leaked keys, or third-party cooperation is needed beyond what the attacker already owns.

### Recommendation
Replace the runtime `executable` flag check with a check against the BPF Upgradeable Loader's program data account. A program address P has a canonical program-data PDA at `find_program_address(&[P.as_ref()], &bpf_loader_upgradeable::id())`. If that PDA ever existed (even if now closed), the address should be rejected as a legacy emitter. Alternatively, require that the `emitter_authority` account is owned by `SystemProgram` and has never been a loader-owned account, or simply disallow the `None` path entirely and require all programs to use the `cpi_program_id=Some(...)` PDA path.
