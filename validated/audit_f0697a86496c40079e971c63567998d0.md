Audit Report

## Title
Stale Descendant-Weight Accounting in `remove_entry_and_descendants` Corrupts Eviction Priority — (File: tx-pool/src/component/pool_map.rs)

## Summary
`remove_entry_and_descendants` pre-strips all parent/child links for every entry in the removal set before calling `remove_entry` on each one. Because `remove_entry` resolves ancestors through `self.links.calc_ancestors` to decrement `descendants_*` fields, and those links are already gone, surviving ancestors are left with permanently inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`. This corrupts their `EvictKey`, causing them to be systematically skipped during pool eviction and enabling a low-cost pool-congestion attack.

## Finding Description
`remove_entry_and_descendants` collects the target and all its descendants, calls `remove_entry_links` on every entry in the set, then calls `remove_entry` on each: [1](#0-0) 

`remove_entry_links` removes the entry from its parents' children lists, removes the entry from its children's parents lists, and then removes the entry from the links map entirely via `self.links.remove(id)`: [2](#0-1) 

`remove_entry` then calls `update_ancestors_index_key`, which resolves ancestors through `self.links.calc_ancestors`: [3](#0-2) 

`calc_ancestors` calls `calc_relative_ids`, which starts by looking up `self.inner.get(short_id)`. Since `remove_entry_links` already called `self.links.remove(id)` for every entry in `removed_ids`, this lookup returns `None` and the function returns an empty set: [4](#0-3) 

The `sub_descendant_weight` call inside `update_ancestors_index_key` is therefore never reached for any ancestor outside the removal set.

**Concrete exploit flow:**
1. Submit `tx1` (low fee) — `tx1.descendants_count = 1`, `tx1.descendants_fee = fee1`.
2. Submit `tx3` (child of `tx1`) — `update_ancestors_index_key` adds tx3's weight to tx1: `tx1.descendants_count = 2`, `tx1.descendants_fee = fee1 + fee3`.
3. Submit `tx4` (child of `tx3`) — `tx1.descendants_count = 3`, `tx1.descendants_fee = fee1 + fee3 + fee4`.
4. Submit `tx2` (spends same input as `tx3`) — `resolve_conflict` calls `remove_entry_and_descendants(tx3_id)`.
5. Inside: `removed_ids = [tx3, tx4]`; `remove_entry_links(tx3)` strips tx3 from the links map; `remove_entry_links(tx4)` strips tx4; `remove_entry(tx3)` → `calc_ancestors(tx3)` returns ∅ → `tx1.sub_descendant_weight(tx3)` is **never called**; same for tx4.
6. After: `tx1.descendants_count = 3`, `tx1.descendants_fee = fee1 + fee3 + fee4` — permanently stale.

The same stale-state condition is triggered by every call site of `remove_entry_and_descendants`: `resolve_conflict`, `resolve_conflict_header_dep`, `process_rbf`, `limit_size`, and `remove_by_detached_proposal`. [5](#0-4) 

The existing test `test_resolve_conflict_descendants` only asserts that the conflicted entries are absent from the pool; it does not assert that `tx1`'s `descendants_*` fields are correctly decremented: [6](#0-5) 

## Impact Explanation
The `EvictKey` for each pool entry is derived directly from the (now-stale) descendant fields: [7](#0-6) 

`EvictKey` ordering places the entry with the lowest `fee_rate` first for eviction: [8](#0-7) 

`next_evict_entry` selects the first entry in `iter_by_evict_key()` order, and `limit_size` uses this to decide which entries to remove when the pool is over capacity: [9](#0-8) [10](#0-9) 

A surviving ancestor with inflated `descendants_fee` reports a falsely high `fee_rate` in its `EvictKey` and is therefore systematically skipped during eviction. An attacker can repeat the three-step pattern across many transactions to fill the pool with low-fee entries that cannot be evicted, preventing legitimate higher-fee transactions from entering the pool. This matches the **High** impact category: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation
The trigger path requires no special privilege. Any user with RPC access can call `send_transaction` to submit `tx_A`, `tx_B`, and `tx_C` in sequence. The conflict resolution path (`resolve_conflict`) is exercised on every transaction submission that spends an already-spent input. The pattern is repeatable indefinitely, and each iteration permanently inflates the `descendants_*` fields of one additional surviving ancestor. The attacker's cost per inflation event is the sum of fees for three transactions, which can be kept near the minimum fee rate for `tx_A` and `tx_C`; only `tx_B` needs to be a child of `tx_A`.

## Recommendation
Before stripping links, explicitly update the `descendants_*` fields of all ancestors that are **not** in the removal set:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();

    // Decrement descendants_* for surviving ancestors BEFORE links are removed.
    for rid in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(rid).map(|e| e.inner.clone()) {
            let ancestors = self.links.calc_ancestors(rid);
            for anc_id in ancestors {
                if !removed_set.contains(&anc_id) {
                    self.entries.modify_by_id(&anc_id, |e| {
                        e.inner.sub_descendant_weight(&entry);
                        e.evict_key = e.inner.as_evict_key();
                    });
                }
            }
        }
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }
    removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
}
```

## Proof of Concept
**Setup**: Pool contains `tx1 → tx3 → tx4`. After adding all three, `tx1` has `descendants_count = 3`, `descendants_fee = fee1 + fee3 + fee4`.

**Trigger**: Submit `tx2` that spends `tx1`'s output 0 (same as tx3). `resolve_conflict` calls `remove_entry_and_descendants(tx3_id)`.

**Inside `remove_entry_and_descendants`**:
- `removed_ids = [tx3, tx4]`
- `remove_entry_links(tx3)` and `remove_entry_links(tx4)` are called — all links gone.
- `remove_entry(tx3)` → `update_ancestors_index_key(tx3, Remove)` → `calc_ancestors(tx3)` returns ∅ → `tx1.sub_descendant_weight(tx3)` is **never called**.
- `remove_entry(tx4)` → same result.

**After**: `tx1` still reports `descendants_count = 3`, `descendants_fee = fee1 + fee3 + fee4`.

A minimal regression test can confirm this by adding the following assertions after `pool.resolve_conflict(&tx2)` in `test_resolve_conflict_descendants`:

```rust
let tx1_entry = pool.get(&tx1_short_id).unwrap();
assert_eq!(tx1_entry.descendants_count, 1);
assert_eq!(tx1_entry.descendants_fee, MOCK_FEE);
```

Both assertions will currently fail, confirming the regression is live and untested. [1](#0-0)

        &mut self,

        current_entry_id: Option<&ProposalShortId>,

        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
