### Title
Integer Overflow in `Echo.getFee()` Causes `requestPriceUpdatesWithCallback()` to Revert — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.getFee()` computes the gas-cost component of the fee by multiplying `callbackGasLimit` (`uint32`) by `providerFeeInWei` (`uint96`). In Solidity ≥0.8.0, this multiplication is evaluated in the common type `uint96`. When the product exceeds `type(uint96).max`, the built-in overflow check panics and reverts. Because `getFee()` is called unconditionally inside `requestPriceUpdatesWithCallback()`, the revert propagates and permanently blocks users from submitting requests to that provider.

---

### Finding Description

In `Echo.getFee()`:

```solidity
uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei;
uint256 gasFee = callbackGasLimit * providerFeeInWei;   // ← overflow here
```

`callbackGasLimit` is `uint32` and `providerFeeInWei` is `uint96`. Solidity resolves the common type as `uint96`, so the multiplication is performed in `uint96` before the result is widened to `uint256`. In Solidity 0.8.x, overflow in `uint96` arithmetic causes a panic revert.

`type(uint96).max ≈ 7.92 × 10^28`. Overflow occurs when:

```
callbackGasLimit * feePerGasInWei > 2^96
```

Concrete example:
- `callbackGasLimit = type(uint32).max = 4,294,967,295`
- `feePerGasInWei = 2 × 10^19` (20 ETH per gas unit — extreme but settable with no on-chain cap)
- Product ≈ `8.6 × 10^28 > 7.92 × 10^28` → **panic revert**

There is no upper bound enforced on `feePerGasInWei` during `registerProvider()` or `setProviderFee()`.

---

### Impact Explanation

`getFee()` is called on line 75 of `requestPriceUpdatesWithCallback()`:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
```

If `getFee()` reverts, `requestPriceUpdatesWithCallback()` reverts for every caller regardless of `msg.value`. Users cannot submit any price-update requests to the affected provider. The Echo contract's core functionality is completely blocked for that provider.

---

### Likelihood Explanation

Any registered provider can call `setProviderFee()` to set `feePerGasInWei` to an arbitrary `uint96` value. A provider that sets `feePerGasInWei ≥ 2^64 ≈ 1.84 × 10^19` will cause overflow for any user who passes `callbackGasLimit = type(uint32).max`. A provider that sets `feePerGasInWei = type(uint96).max` will cause overflow for any `callbackGasLimit ≥ 2`. Provider registration is permissionless (`registerProvider()` has no access control beyond "not already registered"), so any address can become a provider and trigger this condition. Even without malicious intent, a misconfigured provider could set an extreme fee and inadvertently brick their own service.

---

### Recommendation

Cast `callbackGasLimit` to `uint256` before the multiplication so the arithmetic is performed in 256-bit context:

```solidity
uint256 gasFee = uint256(callbackGasLimit) * providerFeeInWei;
```

Similarly, the `priceIds.length * feePerFeedInWei` multiplication on line 246 is already safe because `priceIds.length` is `uint256`, which promotes `feePerFeedInWei` to `uint256` automatically.

---

### Proof of Concept

1. Deploy `Echo` with a legitimate admin and Pyth address.
2. Register a provider with `feePerGasInWei = type(uint96).max`.
3. Call `getFee(provider, 2, priceIds)` — reverts with arithmetic overflow panic.
4. Call `requestPriceUpdatesWithCallback{value: 1 ether}(provider, block.timestamp, priceIds, 2)` — reverts because `getFee()` panics internally.
5. No user can ever submit a request to this provider regardless of `msg.value`.
