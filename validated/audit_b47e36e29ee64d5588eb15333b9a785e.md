### Title
Unbounded Per-Peer Request Rate for `GetBlocksProof` Enables O(K·log H) MMR DB-Read Amplification — (`util/light-client-protocol-server/src/lib.rs`, `src/components/get_blocks_proof.rs`, `src/constant.rs`)

---

### Summary

`LightClientProtocol` has no per-peer rate limiter. An unprivileged remote peer may send `GetBlocksProof` messages at wire speed, each carrying up to 1 000 valid main-chain block hashes. Every such message forces the server to perform O(K · log H) reads from `COLUMN_CHAIN_ROOT_MMR` (where K ≤ 1 000 and H is the chain height) plus 3 000 block-data reads, with no throttle, no cost accounting, and no ban triggered on a well-formed request.

---

### Finding Description

**Entry point — `GetBlocksProof` message**

`LightClientProtocol::try_process` dispatches `GetBlocksProof` directly to `GetBlocksProofProcess::execute` with no rate check: [1](#0-0) 

The only structural guard is a count ceiling: [2](#0-1) [3](#0-2) 

A request with exactly 1 000 hashes passes this check and is processed in full.

**Per-request work**

For each of the K found hashes, the server reads `COLUMN_BLOCK_HEADER`, `COLUMN_BLOCK_UNCLE`, and `COLUMN_BLOCK_EXTENSION` (3 K DB reads): [4](#0-3) 

Then `reply_proof` calls `mmr.gen_proof(items_positions)` on an MMR of size `leaf_index_to_mmr_size(H-1)`: [5](#0-4) 

Each `gen_proof` node access resolves to a `COLUMN_CHAIN_ROOT_MMR` read via `MMRStore<HeaderDigest> for &Snapshot`: [6](#0-5) [7](#0-6) 

For K = 1 000 positions spread across an MMR of size M = `leaf_index_to_mmr_size(H-1)`, proof generation requires O(K · log M) = O(1 000 · log H) node reads. At H = 1 000 000 that is ≈ 20 000 `COLUMN_CHAIN_ROOT_MMR` reads per request, on top of the 3 000 block-data reads.

**No rate limiting exists in `LightClientProtocol`**

A grep for `rate_limiter`, `RateLimiter`, or `TooManyRequests` in the entire `util/light-client-protocol-server/` tree returns zero results. Compare this with `Relayer`, which explicitly checks a per-peer, per-message-type governor before dispatching any message: [8](#0-7) 

`LightClientProtocol` has no equivalent field or check.

**Peer scoring is disabled**

The behaviour-scoring system that could otherwise penalise expensive peers is explicitly disabled in production: [9](#0-8) 

A well-formed `GetBlocksProof` with 1 000 valid hashes returns `Status::ok()`, so no ban is issued either.

---

### Impact Explanation

A single attacker connection can saturate the RocksDB I/O of the serving full node by sending back-to-back `GetBlocksProof` messages at wire speed. Multiple connections (or multiple coordinated peers) multiply the effect linearly. The resulting I/O pressure delays block verification and chain-tip advancement on the targeted node, degrading its participation in the CKB network and potentially causing it to fall behind the tip, which constitutes CKB network congestion achievable at negligible cost to the attacker.

---

### Likelihood Explanation

The attack requires only a valid P2P connection to a node running the light-client protocol and knowledge of 1 000 main-chain block hashes, both of which are trivially obtained from any public CKB node or block explorer. No PoW, no key material, and no privileged access are needed. The attacker can reuse the same 1 000 hashes in every request.

---

### Recommendation

Add a per-peer, per-message-type rate limiter to `LightClientProtocol` mirroring the one in `Relayer`:

- Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`.
- At the top of `try_process`, call `self.rate_limiter.check_key(&(peer_index, message.item_id()))` and return `StatusCode::TooManyRequests` on failure.
- Tune the quota conservatively for proof-generating messages (`GetBlocksProof`, `GetTransactionsProof`, `GetLastStateProof`), since each carries a much higher per-request DB cost than relay messages.

---

### Proof of Concept

1. Connect to a CKB full node with the light-client protocol enabled.
2. Collect 1 000 evenly-spaced main-chain block hashes (blocks 1, H/1000, 2H/1000, …, H).
3. In a tight loop, send `GetBlocksProof { last_hash: tip_hash, block_hashes: [h1..h1000] }`.
4. Instrument RocksDB with a `COLUMN_CHAIN_ROOT_MMR` read counter; observe it increment by ≈ 20 000 per iteration on a chain of height 1 000 000.
5. Observe the node's block-processing latency increase proportionally to the request rate, with no ban or throttle applied to the sender.

    async fn try_process(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        message: packed::RelayMessageUnionReader<'_>,
    ) -> Status {
        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =
            !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));

        if should_check_rate

                .rate_limiter
