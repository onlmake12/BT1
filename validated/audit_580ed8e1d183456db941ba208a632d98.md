Audit Report

## Title
Unbounded O(N) DB and CPU Amplification via `GetTransactionsProof` with No Rate Limiting — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary
An unprivileged light-client peer can send a single `GetTransactionsProof` message with up to 1000 `tx_hashes`, each from a distinct block, forcing the server to perform ≥4000 DB reads plus CPU-intensive CBMT and MMR proof computations per request. Because valid requests return `Status::ok()` (200) and the ban logic only fires on 4xx responses, the attacker is never penalized. No rate limiting, per-peer quota, or cooldown exists anywhere in the light-client-protocol-server, allowing the attacker to pipeline these requests indefinitely, saturating disk I/O and CPU on the serving node.

## Finding Description
`GET_TRANSACTIONS_PROOF_LIMIT` is set to 1000. The `execute()` function rejects requests with `> 1000` tx_hashes (returning `MalformedProtocolMessage`, a 4xx code that triggers a 5-minute ban), but a request with exactly 1000 tx_hashes is fully processed without any penalty.

For each unique block containing a found tx, the server unconditionally performs:
- `snapshot.get_block(&block_hash)` — full block deserialization from DB
- `CBMT::build_merkle_proof(...)` — CPU-bound proof construction over all block transactions
- `snapshot.get_block_uncles(&block_hash)` — additional DB read
- `snapshot.get_block_extension(&block_hash)` — additional DB read

After the per-block loop, `reply_proof` calls `mmr.gen_proof(items_positions)` with up to 1000 positions — an O(N log N) MMR traversal.

A valid request returns `Status::ok()` (200). The `should_ban()` method only returns a ban duration for 4xx codes, so a successful maximum-size request never triggers a ban. A grep across the entire `util/light-client-protocol-server/` directory for any rate-limiting, token-bucket, per-peer quota, cooldown, or throttle mechanism returns zero matches — confirming no such protection exists.

## Impact Explanation
The light-client protocol server runs inside the CKB node process and shares the same DB and CPU resources as the consensus engine. Continuously pipelining maximum-size `GetTransactionsProof` requests (4000+ DB reads + CBMT/MMR CPU work per request, with no back-pressure) saturates disk I/O and CPU, preventing the node from processing new blocks and syncing the chain. This maps to the **High** bounty impact: *"Vulnerabilities which could easily crash a CKB node"* (10001–15000 points).

## Likelihood Explanation
All required inputs are publicly observable on mainnet: tx_hashes are in block data and `last_hash` is the chain tip. No key material, privileged access, or hashpower is required. Any peer that can open a light-client protocol connection — which is an unprivileged, unauthenticated operation — can execute this attack. The attacker is never banned because each individual request is technically valid (returns 200). The attack is trivially repeatable and can be parallelized across multiple connections.

## Recommendation
1. **Add per-peer rate limiting** on `GetTransactionsProof` (and `GetBlocksProof`) using a token-bucket or sliding-window counter; ban peers that exceed a threshold of requests per second.
2. **Cap the number of distinct blocks** processed per request (not just the raw tx_hash count), since the amplification factor is driven by block diversity.
3. **Consider banning peers** that repeatedly send maximum-size requests within a short window, even if each individual request is technically valid.

## Proof of Concept
```
1. Sync a CKB node with >= 1000 blocks, each containing at least one transaction.
2. Collect 1000 tx_hashes from 1000 distinct blocks (publicly available via RPC/chain scan).
3. Obtain the current chain tip hash as last_hash.
4. Craft a GetTransactionsProof message with all 1000 tx_hashes and the valid last_hash.
5. Connect as a light-client peer and send the message in a tight loop (or from multiple
   parallel connections).
6. Observe: each iteration triggers >=4000 DB reads + CBMT/MMR computation on the server,
   the response is always Status::ok() (200), no ban is applied, and sustained CPU/IO
   exhaustion degrades or crashes the node.
```
