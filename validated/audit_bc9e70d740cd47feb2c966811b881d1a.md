Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Post-Eviction `total_tx_size`/`total_tx_cycles` in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` is called before `check_and_record_ancestors` and captures a pre-eviction snapshot of `self.total_tx_size + entry.size` into a local variable. When `check_and_record_ancestors` evicts transactions via `remove_entry_and_descendants → remove_entry → update_stat_for_remove_tx`, it correctly decrements `self.total_tx_size`. However, the stale local is then unconditionally written back at L218, erasing those decrements and permanently inflating the pool size counter by the total size of evicted transactions.

## Finding Description
The exact code path in `add_entry` (L210–219):

```
L210–211: (total_tx_size, total_tx_cycles) = self.updated_stat_for_add_tx(entry.size, entry.cycles)
          // updated_stat_for_add_tx is &self (pure read, L711): returns self.total_tx_size + tx_size
          // snapshot: total_tx_size = S + new_size, self.total_tx_size still = S

L213:     evicts = self.check_and_record_ancestors(&mut entry)?
          // eviction branch (L603–625): when ancestors_count > max_ancestors_count
          //   but ancestors_count - cell_ref_parents.len() <= max_ancestors_count,
          //   calls self.remove_entry_and_descendants(next_id) (L618)
          //     → remove_entry (L235–249)
          //       → update_stat_for_remove_tx (L247, L733–740)
          //         → self.total_tx_size -= evicted_size   // now S − evicted_size

L218:     self.total_tx_size = total_tx_size   // writes S + new_size  ← BUG
L219:     self.total_tx_cycles = total_tx_cycles
```

`updated_stat_for_add_tx` (L711) has signature `&self` and only reads `self.total_tx_size`, adding `tx_size` with overflow check. It does not mutate any state. `update_stat_for_remove_tx` (L733–740) directly assigns `self.total_tx_size = total_tx_size` after a checked subtraction, so the decrement is real and in-place. The unconditional assignment at L218 then overwrites the post-eviction value with the pre-eviction snapshot, inflating the counter by exactly `evicted_size` rather than the correct `S + new_size − evicted_size`. The eviction branch is only reached when `ancestors_count > max_ancestors_count` and `cell_ref_parents` is non-empty (L598–625), so the bug is silent in the common case.

## Impact Explanation
`limit_size` (pool.rs) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting lowest-fee-rate transactions. An inflated counter causes it to evict valid, fee-paying transactions that should remain in the pool. Subsequent calls to `updated_stat_for_add_tx` start from the inflated baseline, causing valid transactions to be rejected with `Reject::Full` even when real pool capacity exists. An attacker who repeatedly triggers the eviction path accumulates inflation across submissions, progressively degrading mempool capacity until legitimate transactions are systematically rejected. This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* (10001–15000 points). No privileges are required; the attacker only needs the ability to submit transactions via `send_transaction` or the P2P relay.

## Likelihood Explanation
The trigger condition requires a new transaction whose ancestor count (including `cell_ref_parents`) exceeds `max_ancestors_count` (default 25), with at least one `cell_ref_parent` evictable to bring the count within limit. An attacker observing the mempool via `get_raw_tx_pool` can construct this deliberately: build a 24-deep chain, submit a cell-dep consumer `R` (making it a `cell_ref_parent`), then submit `T_new` spending the same output. This is straightforward, requires no special privilege, and can be repeated to accumulate inflation. The condition also arises organically in normal mempool usage.

## Recommendation
Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so the snapshot reflects the post-eviction state of `self.total_tx_size`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, replace the local-variable pattern with direct in-place increments (`self.total_tx_size += entry.size`) applied only after all mutations are complete, handling overflow separately.

## Proof of Concept
1. Fill the pool with a chain `T1 → T2 → … → T24` (each spending the previous output).
2. Submit transaction `R` that takes `T1`'s output as a **cell dep** (`R` becomes a `cell_ref_parent` of any tx spending `T1`'s output).
3. Submit `T_new` spending `T1`'s output. Ancestor count = 24 (chain) + 1 (`R`) = 25, which with `T_new` itself = 26, exceeding `max_ancestors_count` = 25.
4. `check_and_record_ancestors` enters the eviction branch (L603): `ancestors_count − cell_ref_parents.len() = 26 − 1 = 25 ≤ 25`, so it evicts `R` (size `S_R`) via `remove_entry_and_descendants`, decrementing `self.total_tx_size` by `S_R`.
5. L218 overwrites `self.total_tx_size` with the pre-eviction snapshot, inflating it by `S_R`.
6. Call `get_tip_tx_pool_info` and compare `total_tx_size` against the sum of actual entry sizes — the counter exceeds the real sum by `S_R`.
7. Observe that `limit_size` subsequently evicts a valid transaction that fits within `max_tx_pool_size`, confirming spurious eviction.
8. Repeat steps 1–5 multiple times to accumulate inflation and eventually cause `Reject::Full` for transactions that fit within the real pool capacity.
