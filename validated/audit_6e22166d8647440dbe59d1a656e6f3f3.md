### Title
`Scheduler.getPricesNoOlderThan` / `getEmaPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp Instead of Per-Feed Publish Time — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getPricesNoOlderThan` and `getEmaPricesNoOlderThan` perform a single staleness check against `status.priceLastUpdatedAt`, which is set to the **maximum** `publishTime` across all feeds in the subscription. When a subscription contains a mix of open-market feeds (recent timestamps) and closed-market feeds (timestamps from the last trading period, potentially days old), the staleness check passes based on the freshest feed's timestamp, while the returned prices for closed-market feeds can be arbitrarily stale. This is the direct Pyth analog of M-10: the timestamp used for the staleness guard does not correspond to the actual update time of the individual price feeds being returned.

---

### Finding Description

In `_validateShouldUpdatePrices`, the `updateTimestamp` is computed as the **maximum** `publishTime` across all feeds in the batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

The code explicitly acknowledges this design: *"Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period."*

This max timestamp is then stored as the subscription-level staleness marker:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
```

Both `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` perform their staleness guard against this single subscription-level value:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
```

The staleness check is subscription-level (max timestamp), but the guarantee implied by the function name `getPricesNoOlderThan` is per-price-feed. A caller requesting only a closed-market feed (e.g., AAPL/USD with `publishTime` = 2 days ago) will pass the staleness check if any other feed in the subscription (e.g., BTC/USD) has a recent timestamp, because `priceLastUpdatedAt` reflects the max, not the individual feed's actual age.

---

### Impact Explanation

Downstream contracts integrating the Scheduler as a push-oracle and calling `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)` expect that the returned AAPL price is no older than 60 seconds. Instead, they silently receive a price from the last trading session (potentially days old) without any revert. This defeats the entire purpose of the `NoOlderThan` API and can cause:

- Incorrect collateral valuations in lending protocols using equity feeds
- Incorrect settlement prices in derivatives protocols
- Silent acceptance of stale prices in any protocol that relies on the staleness revert as a safety gate

The `publishTime` field in the returned `PythStructs.Price` struct does contain the actual (stale) timestamp, but the contract's own staleness API is supposed to abstract that check away — consumers who call `getPricesNoOlderThan` are explicitly trusting the contract to revert on stale data.

---

### Likelihood Explanation

The Scheduler is explicitly designed to support subscriptions containing mixed asset classes (crypto + equities). The code comment acknowledges closed-market feeds will have old timestamps. Any subscription containing at least one equity feed alongside a crypto feed will exhibit this behavior whenever the equity market is closed. A whitelisted reader of such a subscription can trigger the issue by calling `getPricesNoOlderThan` with a tight `age_seconds` and a closed-market price ID. No privileged access or key compromise is required — only a valid subscription with mixed feeds and a whitelisted reader address.

---

### Recommendation

Replace the subscription-level staleness check with a per-feed check inside `getPricesNoOlderThan` and `getEmaPricesNoOlderThan`. After retrieving the individual price feeds via `_getPricesInternal`, iterate over each returned feed and verify its own `publishTime` satisfies the `age_seconds` constraint:

```solidity
function getPricesNoOlderThan(...) external view ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

The same fix applies to `getEmaPricesNoOlderThan` using `priceFeeds[i].emaPrice.publishTime`.

---

### Proof of Concept

1. Create a subscription with two price IDs: `BTC_ID` (crypto, always open) and `AAPL_ID` (equity, closed on weekends).
2. On a Saturday, a keeper calls `updatePriceFeeds`. Pyth returns:
   - BTC: `publishTime = now` (e.g., `T`)
   - AAPL: `publishTime = T - 172800` (2 days ago, last Friday close)
3. `_validateShouldUpdatePrices` sets `updateTimestamp = T` (the max).
4. `status.priceLastUpdatedAt = T`.
5. A whitelisted reader calls `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)`.
6. The check `distance(block.timestamp, T) = 0 ≤ 60` passes.
7. The function returns the AAPL price with `publishTime = T - 172800` — 2 days stale — without reverting.
