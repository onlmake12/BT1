### Title
Unauthenticated `advanceProviderCommitment` Allows Anyone to Modify Any Provider's Commitment State — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

`advanceProviderCommitment` in `Entropy.sol` accepts an arbitrary `provider` address parameter and performs no `msg.sender == provider` check. Any unprivileged caller who possesses a valid hash-chain preimage (obtainable from the public Fortuna revelation API for already-requested sequence numbers) can advance any registered provider's `currentCommitmentSequenceNumber` and `currentCommitment` without the provider's consent. This is the on-chain analog of the SumSub report's unauthenticated token-issuance endpoint: a sensitive state-mutation function that should be restricted to the identity it acts upon is instead open to any caller.

### Finding Description

`advanceProviderCommitment` is declared `public` and takes `address provider` as a caller-supplied parameter:

```solidity
function advanceProviderCommitment(
    address provider,
    uint64 advancedSequenceNumber,
    bytes32 providerContribution
) public override {
    EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[provider];
    ...
    if (providerCommitment != providerInfo.currentCommitment)
        revert EntropyErrors.IncorrectRevelation();

    providerInfo.currentCommitmentSequenceNumber = advancedSequenceNumber;
    providerInfo.currentCommitment = providerContribution;
    if (providerInfo.currentCommitmentSequenceNumber >= providerInfo.sequenceNumber) {
        providerInfo.sequenceNumber =
            providerInfo.currentCommitmentSequenceNumber + 1;
    }
}
```

There is no `require(msg.sender == provider)` guard anywhere in the function. [1](#0-0) 

The only protection is the cryptographic check at line 467: the caller must supply a `providerContribution` whose hash-chain reconstruction equals `providerInfo.currentCommitment`. [2](#0-1) 

The Fortuna off-chain service exposes a public, unauthenticated REST endpoint `GET /v1/chains/{chain_id}/revelations/{sequence}` that returns the provider's hash-chain value for any sequence number that has an active on-chain request. [3](#0-2) 

**Attack path:**

1. Attacker observes that provider `P` has `N` pending requests (sequence numbers `k` through `k+N-1`).
2. Attacker queries `GET /v1/chains/<chain>/revelations/<k+N-1>` from the public Fortuna API and receives `providerContribution` for that sequence number.
3. Attacker calls `advanceProviderCommitment(P, k+N-1, providerContribution)` from any EOA.
4. The contract updates `P.currentCommitmentSequenceNumber = k+N-1` and `P.currentCommitment = providerContribution` without `P`'s consent.

The test suite itself calls `advanceProviderCommitment` without any `vm.prank(provider1)`, confirming the absence of an authorization check: [4](#0-3) 

The interface documentation confirms the function is intentionally public but does not document the missing authorization: [5](#0-4) 

### Impact Explanation

1. **Unauthorized provider state mutation**: Any attacker can advance any provider's `currentCommitmentSequenceNumber` and `currentCommitment` to the latest publicly available revelation, without the provider's knowledge or consent. This is a direct integrity violation of provider-owned state.

2. **Sequence number skipping / DoS on future requests**: The dangerous branch at lines 472–483 fires when `advancedSequenceNumber >= providerInfo.sequenceNumber`. If an attacker obtains hash-chain values beyond the current `sequenceNumber` (e.g., from a provider whose hash chain is partially observable, or by replaying a provider's own prior `advanceProviderCommitment` transactions on a different chain), the contract bumps `sequenceNumber = advancedSequenceNumber + 1`, permanently skipping those sequence numbers and making them unavailable to users. The code comment explicitly acknowledges this risk: *"someone can use the leaked revelation to derive favorable random numbers."* [6](#0-5) 

3. **Interference with provider's gas-optimization strategy**: Providers use `advanceProviderCommitment` deliberately to reduce `numHashes` for future reveals. An attacker can force this operation at an undesired time, disrupting the provider's operational cadence.

### Likelihood Explanation

- The Fortuna revelation API is public and unauthenticated, providing the exact `providerContribution` values needed. [7](#0-6) 
- No privileged access, leaked keys, or social engineering is required.
- The attacker only needs to be an unprivileged Ethereum transaction sender.
- The function is `public` with no access modifier, callable by any EOA or contract.

### Recommendation

Add an authorization check requiring `msg.sender == provider`:

```solidity
function advanceProviderCommitment(
    address provider,
    uint64 advancedSequenceNumber,
    bytes32 providerContribution
) public override {
    require(msg.sender == provider, "Unauthorized");
    ...
}
```

Alternatively, remove the `provider` parameter entirely and use `msg.sender` as the provider identity, consistent with how `setProviderFee`, `setProviderUri`, `setFeeManager`, and `setMaxNumHashes` are all implemented. [8](#0-7) 

### Proof of Concept

```solidity
// Attacker EOA (no privileged access)
// 1. Read the provider's current revelation from the public Fortuna API:
//    GET https://fortuna.dourolabs.app/v1/chains/ethereum/revelations/<seqNum>
//    => { "value": { "data": "<providerContribution_hex>" } }

// 2. Call advanceProviderCommitment as any address:

entropy.advanceProviderCommitment(
    VICTIM_PROVIDER_ADDRESS,          // any registered provider

    bytes32(providerContribution)     // obtained from public Fortuna API
```

```text
    function advanceProviderCommitment(
        address provider,
        uint64 advancedSequenceNumber,
        bytes32 providerContribution
    ) public override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];
        if (
            advancedSequenceNumber <=
            providerInfo.currentCommitmentSequenceNumber
        ) revert EntropyErrors.UpdateTooOld();
        if (advancedSequenceNumber >= providerInfo.endSequenceNumber)
            revert EntropyErrors.AssertionFailure();

        uint32 numHashes = SafeCast.toUint32(
            advancedSequenceNumber -
                providerInfo.currentCommitmentSequenceNumber
        );
        bytes32 providerCommitment = constructProviderCommitment(
            numHashes,
            providerContribution
        );

        if (providerCommitment != providerInfo.currentCommitment)
            revert EntropyErrors.IncorrectRevelation();

        providerInfo.currentCommitmentSequenceNumber = advancedSequenceNumber;
        providerInfo.currentCommitment = providerContribution;
        if (
            providerInfo.currentCommitmentSequenceNumber >=
            providerInfo.sequenceNumber
        ) {
            // This means the provider called the function with a sequence number that was not yet requested.
            // Providers should never do this and we consider such an implementation flawed.
            // Assuming this is landed on-chain it's better to bump the sequence number and never use that range
            // for future requests. Otherwise, someone can use the leaked revelation to derive favorable random numbers.
            providerInfo.sequenceNumber =
                providerInfo.currentCommitmentSequenceNumber +
                1;
        }
    }
```

```rust
    State(state): State<crate::api::ApiState>,
    Path(RevelationPathParams { chain_id, sequence }): Path<RevelationPathParams>,
    Query(RevelationQueryParams {
        encoding,
        block_number,
    }): Query<RevelationQueryParams>,
) -> Result<Json<GetRandomValueResponse>, RestError> {
    state
        .metrics
        .http_requests
        .get_or_create(&RequestLabel {
            value: "/v1/chains/{chain_id}/revelations/{sequence}".to_string(),
        })
        .inc();

    let state = state
        .chains
        .read()
        .await
        .get(&chain_id)
        .ok_or(RestError::InvalidChainId)?
        .clone();

    let state = match state {
        ApiBlockChainState::Initialized(state) => state,
        ApiBlockChainState::Uninitialized => {
            return Err(RestError::Uninitialized);
        }
    };

    let current_block_number_fut = state
        .contract
        .get_block_number(state.confirmed_block_status);

    match block_number {
        Some(block_number) => {
            let maybe_request_fut = state.contract.get_request_with_callback_events(
                block_number,
                block_number,
                state.provider_address,
            );

            let (maybe_request, current_block_number) =
                try_join!(maybe_request_fut, current_block_number_fut).map_err(|e| {
                    tracing::error!(chain_id = chain_id, "RPC request failed {}", e);
                    RestError::TemporarilyUnavailable
                })?;

            if current_block_number.saturating_sub(state.reveal_delay_blocks) < block_number {
                return Err(RestError::PendingConfirmation);
            }

            maybe_request
                .iter()
                .find(|r| r.sequence_number == sequence)
                .ok_or(RestError::NoPendingRequest)?;
        }
        None => {
            let maybe_request_fut = state
                .contract
                .get_request_v2(state.provider_address, sequence);
            let (maybe_request, current_block_number) =
                try_join!(maybe_request_fut, current_block_number_fut).map_err(|e| {
                    tracing::error!(chain_id = chain_id, "RPC request failed {}", e);
                    RestError::TemporarilyUnavailable
                })?;

            match maybe_request {
                Some(r)
                    if current_block_number.saturating_sub(state.reveal_delay_blocks)
                        >= r.block_number =>
                {
                    Ok(())
                }
                Some(_) => Err(RestError::PendingConfirmation),
                None => Err(RestError::NoPendingRequest),
            }?;
        }
    }

    let value = &state.state.reveal(sequence).map_err(|e| {
        tracing::error!(
            chain_id = chain_id,
            sequence = sequence,
            "Reveal failed {}",
            e
        );
        RestError::Unknown
    })?;
    let encoded_value = Blob::new(encoding.unwrap_or(BinaryEncoding::Hex), *value);

    Ok(Json(GetRandomValueResponse {
        value: encoded_value,
    }))
```
