### Title
`MAX_DEPOSIT_LIMIT` for Permanent Subscriptions Bypassed via Repeated `addFunds` Calls - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `MAX_DEPOSIT_LIMIT` (100 ETH) cap on permanent subscriptions in `Scheduler.sol` is checked against the single-transaction `msg.value` rather than the cumulative `status.balanceInWei`. By calling `addFunds()` repeatedly with amounts ≤ `MAX_DEPOSIT_LIMIT`, any caller can accumulate an unbounded balance in a permanent subscription, rendering the cap entirely ineffective.

---

### Finding Description

In `Scheduler.sol`, permanent subscriptions are subject to a `MAX_DEPOSIT_LIMIT` of 100 ETH, intended to cap the total ETH that can be permanently locked (since permanent subscriptions prohibit withdrawals). However, both `createSubscription` and `addFunds` check only the per-call `msg.value` against the limit, not the resulting cumulative balance:

**`createSubscription` (line 48):**
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**`addFunds` (lines 612–617):**
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

The check is `msg.value > MAX_DEPOSIT_LIMIT`, not `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`. After each call, `status.balanceInWei` grows unchecked. Additionally, `addFunds` has **no access control** — any address can call it for any subscription ID. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

1. **Deposit cap bypass**: The `MAX_DEPOSIT_LIMIT` is completely ineffective. A permanent subscription can accumulate far more than 100 ETH by calling `addFunds` N times with 100 ETH each, resulting in N×100 ETH permanently locked.
2. **Permanent lock of excess ETH**: Since permanent subscriptions explicitly block `withdrawFunds` (line 640–641), any ETH deposited beyond the intended cap is irrecoverably locked in the contract.
3. **Third-party griefing**: Because `addFunds` is permissionless (no `onlyManager` modifier), any external actor can call it to force-deposit ETH into a permanent subscription they do not own, permanently locking their own ETH against someone else's subscription — a griefing vector with no recourse. [3](#0-2) 

---

### Likelihood Explanation

- **Reachable by any unprivileged caller**: `addFunds` is `external payable` with no role check.
- **Trivially exploitable**: Requires only repeated calls with `msg.value == MAX_DEPOSIT_LIMIT` (100 ETH each).
- **No special conditions**: Works on any active permanent subscription. [4](#0-3) 

---

### Recommendation

Replace the per-call `msg.value` check with a cumulative balance check:

```solidity
// In addFunds():
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Apply the same fix to `createSubscription`. Additionally, consider restricting `addFunds` for permanent subscriptions to `onlyManager` to prevent third-party griefing. [5](#0-4) 

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription with exactly MAX_DEPOSIT_LIMIT (100 ETH)
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);
// status.balanceInWei == 100 ETH — passes the check

// 2. Call addFunds repeatedly with MAX_DEPOSIT_LIMIT each time
for (uint i = 0; i < 10; i++) {
    scheduler.addFunds{value: 100 ether}(subId);
    // Each call: msg.value (100 ETH) is NOT > MAX_DEPOSIT_LIMIT (100 ETH) → passes
    // status.balanceInWei grows by 100 ETH each iteration
}
// Final status.balanceInWei == 1100 ETH — 11× the intended cap
// All ETH is permanently locked; withdrawFunds reverts for permanent subscriptions
``` [4](#0-3) [2](#0-1)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
```
