### Title
Permissionless `slash` Execution Allows Any Unprivileged Caller to Slash Stakers' PYTH Tokens - (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

### Summary
The `slash` instruction in the integrity-pool program accepts any generic `signer` with no role constraint. Once a `slash_event` has been created by the authorized `reward_program_authority`, any unprivileged user can call `slash` to apply that slash event to any staker's account that delegated to the relevant publisher, reducing their PYTH token balance without the staker's consent and without requiring the caller to hold any privileged role.

### Finding Description
The integrity-pool program exposes two related instructions:

**`create_slash_event`** (lines 348–424) is properly access-controlled: it requires `reward_program_authority` with `"relations": ["pool_config"]` and `"signer": true`, meaning only the stored authority can create a slash event. [1](#0-0) 

**`slash`** (lines 762–960) has no such constraint. Its only signer field is:

```json
{
  "name": "signer",
  "signer": true
}
```

There is no `"relations"` field tying `signer` to any authority stored in `pool_config` or elsewhere. [2](#0-1) 

The `slash` instruction then CPIs into the staking program's `slash_account`, which requires `pool_authority` with `"relations": ["config"]`. [3](#0-2) 

The integrity-pool program itself is the `pool_authority` for the staking program, so it signs the CPI as its own PDA — satisfying the staking program's check. The staking program's access control is therefore bypassed at the integrity-pool layer: any unprivileged caller can initiate the chain. [4](#0-3) 

The `slash` instruction targets a specific `stake_account_positions` and its associated `delegation_record` (PDA derived from `publisher + stake_account_positions`), meaning it can be applied to any staker who delegated to the publisher named in the slash event. [5](#0-4) 

### Impact Explanation
Any unprivileged user can call `slash` on any staker's account (for any publisher with an existing slash event), causing an immediate reduction of that staker's PYTH token balance. The slashed tokens are transferred to `slash_custody` and ultimately to the DAO wallet. Stakers cannot prevent this execution once a slash event exists.

The most concrete harm is a **front-running attack**: a staker who observes a newly created `slash_event` and attempts to unstake before it is applied can be front-run by an attacker calling `slash` first, forcing the slash to land before the unstake transaction. The slashing rulebook confirms stakers cannot force-unstake post-slash, so the window between `create_slash_event` and `slash` execution is the only opportunity for a staker to exit. [6](#0-5) 

### Likelihood Explanation
Likelihood is medium-high. Slash events are on-chain and publicly visible. Any MEV bot or adversarial actor monitoring the chain can detect a new `slash_event` and immediately call `slash` against targeted stakers before those stakers can react. No privileged access, leaked key, or governance majority is required — only a funded Solana wallet.

### Recommendation
Add a role constraint to the `signer` field of the `slash` instruction, tying it to `reward_program_authority` (or a dedicated executor role) stored in `pool_config`, analogous to how `create_slash_event` is gated:

```json
{
  "name": "reward_program_authority",
  "relations": ["pool_config"],
  "signer": true
}
```

Alternatively, if permissionless execution is intentional (keeper pattern), document it explicitly and implement a time-lock or cooldown window after `create_slash_event` to give stakers a fair opportunity to exit before the slash is executable by anyone.

### Proof of Concept
1. `reward_program_authority` calls `create_slash_event` for publisher `P` with `slash_ratio = R` at index `N`. This creates a `slash_event` PDA at seeds `["slash_event", P, N]`.
2. Staker `S` (who delegated to `P`) observes the on-chain event and submits an `undelegate` transaction to exit before the slash lands.
3. Attacker `A` (any wallet) front-runs `S`'s transaction by calling `slash` with:
   - `signer` = `A` (any keypair)
   - `publisher` = `P`
   - `stake_account_positions` = `S`'s positions account
   - `index` = `N`
4. The integrity-pool program CPIs into `slash_account` on the staking program, signing as its own PDA (the registered `pool_authority`). The staking program accepts the CPI.
5. `S`'s `stake_account_custody` balance is reduced by `slash_ratio`, and the slashed tokens are transferred to `slash_custody`. `S`'s subsequent `undelegate` transaction executes on the already-reduced balance. [7](#0-6) [8](#0-7)
