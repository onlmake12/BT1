### Title
Provider `accruedFeesInWei` Permanently Inaccessible When `feeManager` Is `address(0)` in Echo Contract - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the **only** withdrawal path for a provider's accumulated fees is `withdrawAsFeeManager()`, which requires `msg.sender == _state.providers[provider].feeManager`. Because `feeManager` defaults to `address(0)` on registration and can be explicitly reset to `address(0)`, and because `msg.sender` can never equal `address(0)` in the EVM, any provider whose `feeManager` is `address(0)` has their entire `accruedFeesInWei` balance permanently stranded in the contract.

---

### Finding Description

`Echo.sol` credits provider fees inside `executeCallback()`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

The only function that allows a provider to withdraw those fees is `withdrawAsFeeManager()`:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    ...
}
```

Unlike `Entropy.sol`, which provides a direct `withdraw(uint128 amount)` callable by `msg.sender` as the provider itself, **Echo.sol has no equivalent direct-provider withdrawal function**. The `withdrawFees()` function is restricted to the admin and only drains `_state.accruedFeesInWei` (the Pyth protocol fee pool), not the per-provider `accruedFeesInWei`.

The `ProviderInfo` struct initializes `feeManager` to `address(0)` by default:

```solidity
struct ProviderInfo {
    uint96 baseFeeInWei;
    uint96 feePerFeedInWei;
    uint96 feePerGasInWei;
    uint128 accruedFeesInWei;
    address feeManager;
    bool isRegistered;
}
```

`registerProvider()` does not set `feeManager`, so it remains `address(0)` unless the provider explicitly calls `setFeeManager()`. Furthermore, `setFeeManager()` accepts `address(0)` as a valid argument (consistent with the documented behavior in `IEntropy.sol`: *"Call this function with the all-zero address to disable the fee manager role"*), meaning a provider can also actively reset `feeManager` to `address(0)` after fees have accumulated.

In both cases — default registration without setting a fee manager, or explicit reset to `address(0)` — the check `msg.sender == address(0)` can never pass, so `withdrawAsFeeManager` always reverts, and the provider's fees are permanently locked.

---

### Impact Explanation

Any provider registered in `Echo.sol` who has not explicitly set a non-zero `feeManager` address accumulates fees that can never be withdrawn. The ETH is held in the contract with no recovery path. This is a direct, permanent loss of provider funds. The impact is equivalent to the external report: fees credited to a party become permanently inaccessible due to a blocking state condition.

---

### Likelihood Explanation

The default state for every newly registered provider is `feeManager == address(0)`. A provider who calls `registerProvider()` and then has callbacks executed against them immediately begins accumulating fees they can never claim. No special action is required to trigger the loss — it is the default behavior. Additionally, the `setFeeManager(address(0))` path is explicitly documented as a supported operation, making it a realistic operational scenario.

---

### Recommendation

Add a direct `withdraw(uint128 amount)` function to `Echo.sol` that allows `msg.sender` (as the provider) to withdraw their own `accruedFeesInWei`, mirroring the pattern in `Entropy.sol`:

```solidity
function withdraw(uint128 amount) external {
    ProviderInfo storage providerInfo = _state.providers[msg.sender];
    require(providerInfo.isRegistered, "Provider not registered");
    require(providerInfo.accruedFeesInWei >= amount, "Insufficient balance");
    providerInfo.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "withdrawal failed");
}
```

Alternatively, reject `address(0)` in `setFeeManager()` and require a non-zero `feeManager` to be set before fees can accumulate.

---

### Proof of Concept

1. Provider calls `registerProvider(baseFee, feedFee, gasFee)` — `feeManager` is `address(0)` by default.
2. A user calls `requestPriceUpdatesWithCallback{value: fee}(provider, ...)`.
3. Anyone calls `executeCallback(provider, seqNum, updateData, priceIds)` — this credits `_state.providers[provider].accruedFeesInWei += (req.fee + msg.value - pythFee)`.
4. Provider attempts `withdrawAsFeeManager(provider, amount)` — reverts with `"Only fee manager"` because `msg.sender != address(0)`.
