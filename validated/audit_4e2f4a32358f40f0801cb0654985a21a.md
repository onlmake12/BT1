### Title
Decompression Bomb via Crafted Snappy Header Enables Memory Exhaustion DoS — (File: `network/src/compress.rs`)

---

### Summary

The CKB P2P layer trusts the uncompressed-length field embedded in the snappy stream header to pre-allocate a receive buffer before any decompression is attempted. An unprivileged peer can craft a message whose snappy header claims the maximum allowed decompressed size (8 MB) while the actual compressed payload is only a handful of bytes. Each such message forces the node to zero-initialise an 8 MB buffer. With a modest number of concurrent connections this exhausts process memory and crashes the node.

---

`snap::raw::decompress_len` reads only the varint stored in the first bytes of the snappy stream header; it does **not** inspect or validate the compressed payload. An attacker therefore constructs:

```
[0x80]                    ← COMPRESS_FLAG byte
[0x80 0x80 0x80 0x04]    ← varint(8_388_608) = exactly MAX_UNCOMPRESSED_LEN
[<garbage>]              ← any bytes to satisfy the frame length check
```

Total wire size: **≈ 5 bytes**. The node:
1. Passes the `max_frame_length` check (frame is tiny).
2. Calls `decompress_len` → returns 8,388,608.
3. Evaluates `8_388_608 > 8_388_608` → **false** → proceeds.
4. Executes `BytesMut::zeroed(8_388_608)` — zero-initialises 8 MB, touching every page and forcing the OS to commit physical memory.
5. Calls `SnapDecoder::decompress` → fails on garbage payload.
6. Returns an error and closes the connection.

The amplification ratio is **≈ 1,677,721 : 1** (8 MB output from a 5-byte input).

The `LengthDelimitedCodecWithCompress` codec is installed for every protocol built via `CKBProtocol::new_with_support_protocol` with `compress: true` (the default), covering Sync (2 MB frame limit), RelayV3 (4 MB), LightClient (2 MB), Filter (2 MB), Discovery (512 KB), and HolePunching (512 KB). ### Impact Explanation

Every concurrent connection that delivers one crafted message pins 8 MB of committed physical memory until the async task completes. Because tokio runs multiple connection tasks in parallel, an attacker with *N* simultaneous connections can force *N × 8 MB* of live allocations. At the default peer limit (typically 125 full-relay peers) this is **1 GB** of committed memory from a total of ~625 bytes of network traffic. Exceeding available RAM triggers an OOM kill of the node process, halting block and transaction processing on that node. Repeating the attack prevents the node from staying online.

---

### Likelihood Explanation

The attack requires only an unauthenticated TCP connection to the CKB P2P port (default 8115). No keys, no stake, no prior relationship with the target node are needed. The crafted payload is trivial to construct (a 5-byte snappy header followed by garbage). Any protocol that negotiates compression — Sync, RelayV3, LightClient, Filter, Discovery, HolePunching — is a valid entry point. The attacker does not need to complete any application-level handshake beyond the tentacle identify step before sending the malicious frame.

---

### Recommendation

1. **Enforce a minimum compression ratio before allocating.** Before calling `BytesMut::zeroed` / `vec![0; …]`, verify that the compressed payload is at least a plausible fraction of the claimed decompressed size (e.g., `compressed_len * 256 >= decompressed_bytes_len`). Reject the frame if the ratio is implausible.

2. **Reduce `MAX_UNCOMPRESSED_LEN` to match the largest legitimate protocol frame.** The current ceiling (8 MB) exceeds the largest `max_frame_length` (RelayV3 at 4 MB). Setting `MAX_UNCOMPRESSED_LEN` equal to the per-protocol `max_frame_length` eliminates the gap.

3. **Allocate lazily / stream-decompress.** Use a streaming decompressor that writes into a size-bounded ring buffer rather than pre-allocating the full claimed size up front.

### Proof of Concept

```rust
// Craft a 5-byte snappy "bomb" claiming 8 MB decompressed size.
// Snappy varint encoding of 8_388_608 (= 1 << 23):
//   8388608 = 0x800000
//   varint bytes: 0x80, 0x80, 0x80, 0x04
let mut payload = BytesMut::new();
payload.put_u8(0x80);                          // COMPRESS_FLAG
payload.extend_from_slice(&[0x80, 0x80, 0x80, 0x04]); // varint(8_388_608)
// No valid snappy body — decompress() will fail, but the 8 MB
// allocation at BytesMut::zeroed(8_388_608) happens first.

// Open max_peers connections to the target node, send `payload`
// on each simultaneously via the Sync or RelayV3 protocol stream.
// Each connection triggers one 8 MB zeroed allocation before the
// error path is reached, exhausting ~1 GB of RAM at 125 peers.
```

The `decompress_len` call on the 4-byte varint body returns `8_388_608`, the `> MAX_UNCOMPRESSED_LEN` guard evaluates to `false` (equal, not greater), and `BytesMut::zeroed(8_388_608)` executes unconditionally before any decompression is attempted. [6](#0-5) [7](#0-6)
