### Title
Scheduler Keeper Incentive Mechanism Fails During Gas Price Spikes, Causing Exploitable Price Feed Staleness — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler` contract reimburses keepers for gas costs dynamically using `tx.gasprice`, but the `minimumBalancePerFeed` enforced at subscription creation is a static value that does not scale with gas price. During network congestion (elevated `tx.gasprice`), the computed keeper fee exceeds the subscription balance, causing `updatePriceFeeds` to revert. No rational keeper will submit updates under these conditions, leaving Scheduler-backed price feeds stale. An attacker can then exploit the stale prices in any protocol that reads from the Scheduler.

---

### Finding Description

`Scheduler.updatePriceFeeds` captures gas at entry and, at the end of execution, calls `_processFeesAndPayKeeper`:

```solidity
// Scheduler.sol line 279
uint256 startGas = gasleft();
...
// Scheduler.sol line 845-849
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;

if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
```

`gasCost` is unbounded with respect to `tx.gasprice`. The `GAS_OVERHEAD` constant is fixed at 30,000 gas units:

```solidity
// SchedulerConstants.sol line 29
uint256 public constant GAS_OVERHEAD = 30000;
```

The minimum balance a subscription must hold is computed as a fixed product:

```solidity
// Scheduler.sol line 734-738
function getMinimumBalance(uint8 numPriceFeeds) external view override returns (uint256 minimumBalanceInWei) {
    return uint256(numPriceFeeds) * this.getMinimumBalancePerFeed();
}
```

`minimumBalancePerFeed` is a governance-set constant that does not incorporate any gas price assumption. At normal gas prices (e.g., 10 gwei) a subscription holding the minimum balance is solvent. At 200 gwei — a level Ethereum has sustained for hours during congestion events — the same subscription's `totalKeeperFee` is 20× larger, exceeding the balance and causing every keeper call to revert.

Because the revert happens **after** the Pyth fee has already been deducted from `status.balanceInWei` (line 305) and forwarded to the Pyth contract (line 311), the entire transaction reverts and the balance is restored. However, the keeper has already spent gas with no reimbursement. Rational keepers therefore stop submitting updates entirely until gas prices fall, leaving the subscription's stored prices frozen.

The `getPricesNoOlderThan` read path enforces freshness:

```solidity
// Scheduler.sol line 551-552
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();
```

Once the keeper blackout exceeds `age_seconds`, every consumer call reverts with `StalePrice`, or — if the consumer uses `getPricesUnsafe` — silently returns the frozen price.

---

### Impact Explanation

Any protocol that subscribes to the Scheduler for push-style price feeds and reads via `getPricesNoOlderThan` will either:

- **Revert entirely** (DoS of liquidations, collateral checks, trade execution), or
- **Consume a stale price** if it uses `getPricesUnsafe` without an independent freshness check.

An attacker who observes that a subscription's stored price is stale can exploit the divergence between the frozen on-chain price and the true market price to extract value from any protocol that reads from that subscription (e.g., open an under-collateralised position, avoid a liquidation, or drain a lending pool).

---

### Likelihood Explanation

Ethereum gas prices regularly spike 10–100× above baseline during periods of high market activity — precisely the moments when accurate oracle prices are most critical. Subscriptions funded at the minimum balance (which is calibrated for normal gas prices) will fail to update during every significant congestion event. The attacker requires no special access: they only need to (1) observe that a subscription's `priceLastUpdatedAt` is stale (readable via `getSubscription`) and (2) interact with a protocol that consumes that subscription's prices.

---

### Recommendation

**Short term:** In `_processFeesAndPayKeeper`, cap `tx.gasprice` at a configurable `maxGasPriceForReimbursement` parameter. Keepers that submit at higher gas prices absorb the excess themselves, but the subscription balance is never exhausted by a single spike. Alternatively, add a gas-price-aware pre-flight check so keepers can simulate profitability before submitting.

**Long term:** Make `getMinimumBalance` gas-price-aware by incorporating a recent on-chain gas price estimate (e.g., from a TWAP of `block.basefee`) so that the enforced minimum balance always covers several updates at the prevailing gas price. Emit an event when a subscription's balance falls below a safety threshold so managers can top up proactively.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` with `minimumBalancePerFeed = 0.001 ether` (calibrated for ~10 gwei gas).
2. Create a subscription with 2 price feeds, funding it at exactly `getMinimumBalance(2) = 0.002 ether`.
3. Simulate a gas price spike: `vm.txGasPrice(200 gwei)`.
4. Call `scheduler.updatePriceFeeds(subscriptionId, updateData)` as a keeper.
5. Observe revert with `InsufficientBalance` — the keeper fee at 200 gwei (~0.02 ETH) exceeds the 0.002 ETH balance.
6. Advance time past the subscription's `heartbeatSeconds`.
7. Call `scheduler.getPricesNoOlderThan(subscriptionId, priceIds, heartbeatSeconds)` from a consumer contract — observe revert with `StalePrice`.
8. Call `scheduler.getPricesUnsafe(subscriptionId, priceIds)` — observe the frozen (stale) price returned silently.

The existing test `testUpdatePriceFeedsRevertsInsufficientBalanceForKeeperFee` in `PulseScheduler.t.sol` (lines 1373–1438) already demonstrates step 4–5 at `gasPrice = 0.5 gwei`; scaling `gasPrice` to realistic congestion values reproduces the full scenario.
