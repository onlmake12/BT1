### Title
Mismatched Error Names and Messages in TWAP Validation Cause Misleading On-Chain Diagnostics — (File: `target_chains/solana/programs/pyth-solana-receiver/src/error.rs`)

### Summary
In `error.rs`, three consecutive TWAP-related error variants have their `#[msg(...)]` strings rotated by one position relative to their variant names. The variant name, the human-readable message, and the actual check that triggers each error are all inconsistent with one another. Any unprivileged caller of `post_twap_update` who receives one of these errors will be given a completely wrong description of why their transaction was rejected.

### Finding Description
In `target_chains/solana/programs/pyth-solana-receiver/src/error.rs` lines 18–23, three error variants are declared as follows:

```rust
#[msg("Cannot calculate TWAP, end slot must be greater than start slot")]
FeedIdMismatch,
#[msg("The start and end messages must have the same feed ID")]
ExponentMismatch,
#[msg("The start and end messages must have the same exponent")]
InvalidTwapSlots,
```

The actual usage in `validate_twap_messages` (lib.rs lines 526–543) is:

| Variant raised | Actual condition checked | Message emitted on-chain |
|---|---|---|
| `FeedIdMismatch` | `start_msg.feed_id != end_msg.feed_id` | "Cannot calculate TWAP, end slot must be greater than start slot" |
| `ExponentMismatch` | `start_msg.exponent != end_msg.exponent` | "The start and end messages must have the same feed ID" |
| `InvalidTwapSlots` | `end_msg.publish_slot <= start_msg.publish_slot` | "The start and end messages must have the same exponent" |

Every message is shifted one position from its correct variant. The messages are the externally visible, on-chain-logged strings that Anchor emits in transaction logs and that the JS/JSON IDL (`pyth_solana_receiver.json` lines 120–133, `pyth_solana_receiver.ts` lines 723–737) exposes to all downstream integrators.

### Impact Explanation
Any protocol, keeper, or user that calls `post_twap_update` and receives one of these three errors will be told the wrong reason for rejection. A caller whose transaction fails because the two TWAP messages reference different price feed IDs will receive the message "Cannot calculate TWAP, end slot must be greater than start slot" — the exact opposite of the real problem. This directly misleads integrators and automated systems that parse error messages (rather than numeric error codes) to decide how to retry or fix a failed TWAP submission. The IDL files that ship with the SDK propagate the same wrong strings to every downstream consumer.

### Likelihood Explanation
`post_twap_update` is a publicly callable instruction. Any unprivileged relayer or user submitting TWAP updates will encounter these errors whenever their inputs are malformed. The three affected conditions (feed ID mismatch, exponent mismatch, invalid slot ordering) are all normal validation failures that any integrator will hit during development or in production.

### Recommendation
Align each `#[msg(...)]` string with its variant name and the condition that triggers it in `validate_twap_messages`:

```rust
#[msg("The start and end messages must have the same feed ID")]
FeedIdMismatch,
#[msg("The start and end messages must have the same exponent")]
ExponentMismatch,
#[msg("Cannot calculate TWAP, end slot must be greater than start slot")]
InvalidTwapSlots,
```

Regenerate the IDL JSON and TypeScript files after the fix so that all SDK consumers receive the corrected strings.
