### Title
Cross-Chain Replay of Lazer Price Update Signatures Due to Missing Domain Separation - (`File: lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.verifyUpdate()` computes the signed digest as `keccak256(payload)` with no chain ID, contract address, or any domain separator included. Because the same Pyth Lazer trusted-signer key is registered across all EVM deployments, a valid signed update captured on one chain is cryptographically indistinguishable from a valid update on any other chain. An unprivileged attacker can replay a Lazer price update from chain A onto chain B within the consumer's freshness window.

### Finding Description

In `lazer/contracts/evm/src/PythLazer.sol`, `verifyUpdate()` extracts the payload and computes the recovery digest as:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `hash` is purely `keccak256(payload)`. The Lazer payload structure (as parsed by `PythLazerLib.parsePayloadHeader`) contains only: a 4-byte magic constant, a `uint64` timestamp, a 1-byte channel ID, and feed price data. [2](#0-1) 

None of these fields encode the target chain ID or the `PythLazer` contract address. The magic constant (`2479346549`) is identical across all deployments. [3](#0-2) 

The same issue exists in the Sui implementation, where `secp256k1_ecrecover(signature, payload, 0)` is called over the raw payload with no domain binding: [4](#0-3) 

### Impact Explanation

The Pyth Lazer backend registers the same trusted-signer key across all EVM chains (Ethereum, Arbitrum, Optimism, Base, etc.). Because the signed digest contains no chain ID or contract address, a signed update captured from chain A passes `verifyUpdate()` on chain B without modification. Within the consumer contract's freshness window, an attacker can:

1. Observe a valid signed Lazer update on chain A (e.g., Ethereum mainnet).
2. Submit the identical `update` bytes to a `PythLazer`-integrated DeFi protocol on chain B (e.g., Arbitrum).
3. `verifyUpdate()` succeeds — the signer is trusted, the signature is valid, the timestamp is fresh.
4. The consumer contract on chain B accepts a price that was signed for chain A.

If prices differ between chains at the moment of the attack (due to latency, different market conditions, or deliberate timing), the attacker can inject a stale or off-market price into chain B's DeFi protocol, enabling liquidation manipulation, arbitrage, or collateral mispricing.

### Likelihood Explanation

- No privileged access is required. Any observer of on-chain transactions or the Lazer WebSocket stream can obtain a valid signed update.
- The attack is bounded by the consumer's freshness window (typically seconds to minutes for a low-latency oracle), but within that window it is trivially executable.
- Cross-chain price discrepancies for the same asset are common during high-volatility periods, making the attack economically meaningful.
- The Pyth Lazer deployment model explicitly targets multiple EVM chains with the same signer key, making the precondition always satisfied.

### Recommendation

Include a domain separator in the signed digest that binds the signature to a specific chain and contract. The standard approach is EIP-712 domain separation:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    "\x19\x01",
    domainSeparator,   // keccak256(abi.encode(DOMAIN_TYPEHASH, chainId, address(this)))
    keccak256(payload)
));
```

Alternatively, include `block.chainid` and `address(this)` directly in the signed message structure at the protocol level (i.e., in the Lazer payload format itself), so that the off-chain signer embeds the target chain and contract in every update.

The Sui contract's `verify_le_ecdsa_message` should similarly bind the payload to the Sui object ID of the `State` object.

### Proof of Concept

1. Deploy `PythLazer` on Ethereum (chain 1) and Arbitrum (chain 42161) with the same trusted signer address.
2. Call `pythLazer.verifyUpdate{value: fee}(update)` on Ethereum — succeeds, returns `payload` and `signer`.
3. Submit the **identical** `update` bytes to `pythLazer.verifyUpdate{value: fee}(update)` on Arbitrum.
4. The call succeeds: `keccak256(payload)` is identical, `ECDSA.tryRecover` returns the same trusted signer, `isValidSigner(signer)` returns `true`. [5](#0-4) 

The test in `PythLazer.t.sol` confirms the same `update` bytes can be submitted multiple times on the same chain (no replay guard), and since the digest is chain-agnostic, the same bytes are equally valid on any chain: [6](#0-5)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
