### Title
Unprotected `initialize` Function Allows Attacker to Front-Run Deployment and Permanently Compromise Contract Configuration â€” (`File: target_chains/stylus/contracts/pyth-receiver/src/lib.rs` and `target_chains/stylus/contracts/wormhole/src/lib.rs`)

---

### Summary

Both `PythReceiver::initialize` and `WormholeContract::initialize` in the Stylus (Arbitrum) contracts are publicly callable by any account with no access control on `msg.sender`. The only guard is a boolean `initialized` flag. An unprivileged attacker can front-run the deployer's initialization transaction, permanently setting attacker-controlled parameters (wormhole address, guardian set, data sources, governance emitter) before the legitimate deployer can do so.

---

### Finding Description

`PythReceiver::initialize` in `target_chains/stylus/contracts/pyth-receiver/src/lib.rs` is marked `#[public]` and checks only:

```rust
if self.initialized.get() {
    return Err(PythReceiverError::AlreadyInitialized.into());
}
```

There is no check that `msg.sender` is the deployer or any privileged account. The function then sets:
- `self.wormhole` â€” the address of the Wormhole contract used for VAA verification
- `self.valid_data_sources` / `self.is_valid_data_source` â€” the set of trusted price emitters
- `self.governance_data_source_chain_id` / `self.governance_data_source_emitter_address` â€” the governance authority

Identically, `WormholeContract::initialize` in `target_chains/stylus/contracts/wormhole/src/lib.rs` is `#[public]` and checks only `self.initialized.get()`, with no caller restriction. It sets:
- `self.current_guardian_set_index`
- `self.chain_id`, `self.governance_chain_id`, `self.governance_contract`
- The guardian key set used for all VAA signature verification

On Arbitrum (Stylus), contract deployment and initialization are separate transactions. An attacker monitoring the mempool can observe the deployment transaction, then submit a higher-gas `initialize` call that executes before the deployer's own initialization transaction.

Once `initialized` is set to `true`, the flag is permanent â€” there is no re-initialization path. The contract is irrecoverably configured with attacker-controlled parameters.

---

### Impact Explanation

**Critical â€” permanent compromise of the price oracle trust chain.**

If the attacker front-runs `WormholeContract::initialize`:
- The attacker supplies their own guardian set (e.g., keys they control)
- All subsequent `parse_and_verify_vm` calls verify signatures against attacker-controlled keys
- The attacker can forge any VAA, making `PythReceiver` accept arbitrary price data

If the attacker front-runs `PythReceiver::initialize`:
- `self.wormhole` is set to an attacker-controlled contract that always returns "verified" for any VAA
- `self.is_valid_data_source` is set to attacker-controlled emitters
- `self.governance_data_source_emitter_address` is set to an attacker-controlled governance emitter, granting the attacker full governance control (fee changes, data source changes, etc.)

The result is that all price feeds served by the contract are under attacker control, enabling arbitrary price manipulation for any DeFi protocol consuming these feeds.

---

### Likelihood Explanation

**High.** Stylus contracts on Arbitrum are deployed as standard EVM transactions. The deployment transaction and the `initialize` call are always two separate transactions. Any attacker watching the mempool (or even just watching confirmed blocks and racing with a higher gas price) can call `initialize` before the deployer. No special privileges, leaked keys, or oracle collusion are required â€” only the ability to submit a transaction. The attack is straightforward and requires no complex setup.

---

### Recommendation

Add a deployer/owner check to both `initialize` functions. The simplest correct pattern is to record the deployer address at contract creation time and require it in `initialize`:

```rust
// In storage struct:
pub deployer: StorageAddress,

// In a constructor or first-call check:
// Set deployer = msg::sender() at deploy time (Stylus supports this via vm().msg_sender())

// In initialize():
if self.vm().msg_sender() != self.deployer.get() {
    return Err(PythReceiverError::Unauthorized.into());
}
```

Alternatively, follow the OpenZeppelin `Ownable` + `initializer` pattern used by the EVM Solidity contracts (`PythUpgradable`, `EntropyUpgradable`, `ExecutorUpgradable`) which use `constructor() initializer {}` to lock the implementation and `__Ownable_init()` to gate `initialize` to the deployer.

---

### Proof of Concept

1. Attacker monitors the Arbitrum mempool.
2. Deployer broadcasts the `WormholeContract` deployment transaction.
3. Attacker sees the new contract address (deterministic from deployer nonce).
4. Attacker immediately calls `WormholeContract::initialize(attacker_guardians, 0, chain_id, gov_chain_id, gov_contract)` with higher gas, where `attacker_guardians` is a list of addresses whose private keys the attacker controls.
5. Attacker's transaction is included first; `initialized` is set to `true`.
6. Deployer's `initialize` call reverts with `AlreadyInitialized`.
7. Attacker repeats for `PythReceiver::initialize`, setting `wormhole = attacker_wormhole_contract`.
8. All `update_price_feeds` calls now route through the attacker's Wormhole contract, which accepts any VAA. The attacker can push arbitrary prices for any feed ID.
