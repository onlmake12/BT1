Audit Report

## Title
Unauthenticated `from` Field in `ConnectionRequest`/`ConnectionSync` Enables Attacker-Controlled NAT Traversal — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
`ConnectionRequestProcess::execute` accepts `content.from` verbatim from the wire message without verifying it matches the actual sending session's peer ID. This allows any connected peer to spoof an arbitrary `from` peer ID with attacker-controlled `listen_addrs`, inserting those addresses into `pending_delivered`. A subsequent spoofed `ConnectionSync` then causes the victim to spawn a NAT traversal task and call `raw_session` to an attacker-controlled IP, establishing a new P2P session to an attacker-controlled node.

## Finding Description

**Root cause — `ConnectionRequestProcess::execute` (L145–147):** [1](#0-0) 

When `self_peer_id == &content.to`, the code calls `respond_delivered(content.from, ...)` directly. `content.from` is taken verbatim from the wire message. `self.peer` (the actual session's `PeerIndex`) is never looked up in the peer registry to verify that the associated peer ID equals `content.from`. No such check exists anywhere in the function.

**`respond_delivered` (L161–167, L234–237):** [2](#0-1) [3](#0-2) 

The only guard is a 2-minute cooldown keyed by `from_peer_id` — the spoofed value. After the cooldown check, the attacker's addresses are unconditionally inserted into `pending_delivered[from_peer_id]`. Using a fresh spoofed `target_peer_id` per attempt trivially bypasses the cooldown.

**`ConnectionSyncProcess` — missing sender identity (L51–74, L111–115, L154–160):** [4](#0-3) 

`ConnectionSyncProcess::new` does not accept a `peer: PeerIndex` parameter at all — there is no sender identity available. The process looks up `pending_delivered` using the spoofed `content.from`: [5](#0-4) 

This returns the attacker's IP inserted in step 1. The code then spawns a NAT traversal task and calls `raw_session` with the attacker's IP: [6](#0-5) 

**Rate limiter analysis:**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`. [7](#0-6) 

`ConnectionRequest` has `item_id=0` and `ConnectionSync` has `item_id=2`, confirmed by the generated union: [8](#0-7) 

They occupy separate rate limiter slots, so both can be sent within the same second without triggering the forward rate limit. Using a fresh spoofed `target_peer_id` per pair of messages also bypasses the forward rate limiter entirely, since the key changes each time.

## Impact Explanation

The victim node establishes raw TCP sessions to attacker-controlled IPs via `raw_session`. By repeating the two-message sequence with many different spoofed `target_peer_id` values — each bypassing the 2-minute cooldown — the attacker fills the victim's connection slots with attacker-controlled nodes. This constitutes a targeted eclipse attack: the victim's view of the network is dominated by attacker-controlled peers, enabling selective withholding or manipulation of block/transaction relay. A victim node under eclipse can be made to follow a fork, constituting consensus deviation.

**Severity: High (10001–15000 points)** — Vulnerability that could easily crash/isolate a CKB node. With a sufficiently targeted victim (e.g., a mining node), escalation to consensus deviation is plausible, but the base impact is reliable node isolation.

## Likelihood Explanation

The attack requires only a single legitimate P2P connection to the victim — no special privileges, no key material. The spoofed fields (`from`, `listen_addrs`) are accepted verbatim from the wire message with no cryptographic binding to the sending session. The two-message sequence (one `ConnectionRequest`, one `ConnectionSync`) is sufficient to trigger `raw_session` to an arbitrary attacker-controlled IP. The cooldown bypass (fresh spoofed `target_peer_id` per attempt) makes the attack repeatable indefinitely from a single session.

## Recommendation

In `ConnectionRequestProcess::execute`, before calling `respond_delivered`, verify that `content.from` matches the actual peer ID of the sending session by looking up `self.peer` in the peer registry:

```rust
let actual_from = self.protocol.network_state
    .peer_registry.read()
    .get_peer(self.peer)
    .and_then(|p| p.connected_addr.peer_id());
if actual_from.as_ref() != Some(&content.from) {
    return StatusCode::InvalidFromPeerId.with_context("from does not match sender");
}
```

Similarly, pass the sender's `PeerIndex` into `ConnectionSyncProcess::new` and verify `content.from` against the actual sending peer before consuming `pending_delivered`.

## Proof of Concept

```
1. Attacker connects to victim (legitimate session, session_id=S, attacker_peer_id=A)

2. Attacker sends ConnectionRequest {
       from = target_peer_id  (arbitrary, not A),
       to   = victim_peer_id,
       listen_addrs = [attacker_ip:port],
       max_hops = 1,
       route = []
   }
   → victim: self_peer_id == content.to
   → respond_delivered(target_peer_id, ...) called
   → no check that target_peer_id == A
   → pending_delivered[target_peer_id] = ([attacker_ip:port], now)
   → ConnectionRequestDelivered sent back to session S

3. Attacker sends ConnectionSync {
       from  = target_peer_id,
       to    = victim_peer_id,
       route = []
   }
   → victim: route is empty, self_peer_id == content.to
   → pending_delivered.get(&target_peer_id) = [attacker_ip:port]
   → spawns try_nat_traversal(bind_addr, attacker_ip:port)
   → attacker's listener accepts TCP connection
   → control.raw_session(stream, attacker_ip, RawSessionInfo::inbound(listen_addr))
   → new P2P session established to attacker-controlled node

4. Repeat steps 2–3 with fresh target_peer_id values (bypasses 2-min cooldown).
   Victim's connection slots are progressively filled with attacker-controlled nodes.
```
