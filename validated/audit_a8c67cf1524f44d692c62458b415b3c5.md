Audit Report

## Title
Unbounded heap allocation in `do_estimate` via attacker-controlled fee rate — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary
`do_estimate` derives `max_bucket_index` directly from the highest fee rate in the live tx pool snapshot with no upper bound, then allocates `vec![0u64; max_bucket_index + 1]` twice. A transaction with a sufficiently high fee causes `max_bucket_index` to reach hundreds of millions or more, triggering an OOM allocation that crashes any node calling `estimate_fee_rate` while `WeightUnitsFlow` is active and the attacker's transaction remains in the pool.

## Finding Description

**Root cause 1 — `FeeRate::calculate` saturates without cap:**

`FeeRate::calculate` uses `saturating_mul(1000)` before dividing by weight: [1](#0-0) 

For any transaction where `fee * 1000 > u64::MAX` (fee > ~1.844e16 shannons), the fee rate saturates to `u64::MAX / weight`. With a minimum realistic tx weight of ~100, this yields `fee_rate ≈ 1.844e17`.

**Root cause 2 — `max_bucket_index_by_fee_rate` is unbounded:**

The final branch of `max_bucket_index_by_fee_rate` is a linear formula with no ceiling: [2](#0-1) 

For `fee_rate = 1.25e13` (achievable with fee = 1.25e12 shannons, weight = 100): `max_bucket_index = (1.25e13 + 11_500_000) / 100_000 ≈ 125,000,115`. For saturated `fee_rate = u64::MAX / 100 ≈ 1.844e17`: `max_bucket_index ≈ 1.844e12`.

**Root cause 3 — `do_estimate` allocates without capping:**

`max_fee_rate` is taken directly from the highest-fee-rate entry in the pool snapshot with no sanitization: [3](#0-2) 

Two vectors of size `max_bucket_index + 1` are then allocated unconditionally: [4](#0-3) [5](#0-4) 

**No existing guard:** The tx pool enforces only a `min_fee_rate` floor via `check_tx_fee`; there is no `max_fee_rate` admission check anywhere in the acceptance path: [6](#0-5) 

The `TxPoolConfig` struct confirms only `min_fee_rate` and `min_rbf_rate` exist — no upper bound on fee rate: [7](#0-6) 

**Exploit path:**
1. Attacker submits a valid CKB transaction with a high fee (e.g., 12,500 CKB fee, minimal tx size/cycles).
2. The tx propagates via P2P relay and enters the pool of any node with `WeightUnitsFlow` configured.
3. Any caller of the `estimate_fee_rate` RPC (publicly accessible, no authentication): [8](#0-7) 
   triggers `do_estimate`, which attempts to allocate ~1 GB (or more) on the heap.
4. In Rust, `Vec::with_capacity` / `vec!` macro calls the global allocator; on OOM, `std::alloc` aborts the process. On Linux, the OOM killer terminates the node process. The crash is reproducible on every subsequent `estimate_fee_rate` call until the tx is confirmed or evicted.

## Impact Explanation

Any node operator who has configured `WeightUnitsFlow`: [9](#0-8) 

and exposes the `estimate_fee_rate` RPC can have their node process crashed and kept crashed by a single attacker transaction. This is a concrete, reproducible node crash mapping to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires no special privilege — any CKB holder can submit a high-fee transaction via P2P relay or the `send_transaction` RPC. The economic cost scales with desired impact: ~12,500 CKB causes ~1 GB allocation per call (OOM on memory-constrained nodes); ~1,250,000 CKB causes ~100 GB allocation (OOM on all nodes). The tx remains in the pool until confirmed or evicted, so the crash is reproducible on every `estimate_fee_rate` call during that window. `WeightUnitsFlow` is not the default algorithm (the `algorithm` field defaults to `None`), limiting the affected population to nodes that have explicitly opted in, but it is a fully supported, documented configuration option.

## Recommendation

Cap `max_bucket_index` to a safe constant before allocating in `do_estimate`:

```rust
const MAX_BUCKET_COUNT: usize = 10_001; // supports fee rates up to ~2,000,000 shannons/KW

let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
    .min(MAX_BUCKET_COUNT - 1);
```

Alternatively, cap the fee rate used in the estimator to a reasonable maximum (e.g., 10,000,000 shannons/KW) in `TxStatus::new_from_entry_info`: [10](#0-9) 

Or add a `max_fee_rate` admission guard in `check_tx_fee` in `tx-pool/src/util.rs`.

## Proof of Concept

```rust
#[test]
fn test_oom_via_high_fee_rate() {
    use ckb_types::core::FeeRate;
    // fee = 1.25e12 shannons, weight = 100 → fee_rate = 1.25e13
    let fee_rate = FeeRate::from_u64(1_250_000_000_000u64 * 1000 / 100); // 1.25e13
    let max_bucket_index = Algorithm::max_bucket_index_by_fee_rate(fee_rate);
    // max_bucket_index ≈ 125_000_115
    // vec![0u64; 125_000_116] ≈ 1 GB — triggers OOM
    assert!(max_bucket_index > 100_000_000,
        "index {} would cause ~1 GB allocation", max_bucket_index);

    // For saturated fee_rate (fee > u64::MAX/1000):
    let saturated = FeeRate::from_u64(u64::MAX / 100); // ~1.844e17
    let saturated_index = Algorithm::max_bucket_index_by_fee_rate(saturated);
    // saturated_index ≈ 1.844e12 → ~14.75 TB allocation
    assert!(saturated_index > 1_000_000_000_000usize);
}
```

To trigger the full crash: configure a node with `algorithm = "WeightUnitsFlow"` in `ckb.toml`, submit a transaction with fee ≥ 12,500 CKB and minimal size/cycles, then call `estimate_fee_rate` via RPC. The node process will be killed by OOM.

```rust
    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }
```

```rust
pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
```
