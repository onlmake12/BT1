### Title
Unbounded `callbackGasLimit` with No Minimum `feePerGasInWei` Validation Allows Requesters to Drain Provider Keeper Resources at Minimal Cost - (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, `requestPriceUpdatesWithCallback()` accepts a caller-controlled `callbackGasLimit` (type `uint32`, max ~4.29 billion) with no upper-bound validation. Simultaneously, `registerProvider()` and `setProviderFee()` impose no minimum on `feePerGasInWei`, allowing it to be set to zero. When `feePerGasInWei = 0`, the gas-proportional fee component is always zero regardless of `callbackGasLimit`. A requester can therefore submit a request with `callbackGasLimit = type(uint32).max` while paying only the flat base fees, forcing the provider's keeper to attempt executing a callback that requires billions of gas — far exceeding any EVM block gas limit — permanently locking the request and exhausting keeper resources.

---

### Finding Description

**Root cause 1 — No upper bound on `callbackGasLimit`:**

In `requestPriceUpdatesWithCallback()`, the only input validations are a `publishTime` ceiling and a `MAX_PRICE_IDS` cap. There is no check on `callbackGasLimit`:

```solidity
// Echo.sol lines 52–102
function requestPriceUpdatesWithCallback(
    address provider,
    uint64 publishTime,
    bytes32[] calldata priceIds,
    uint32 callbackGasLimit          // ← no upper-bound check
) external payable override returns (uint64 requestSequenceNumber) {
    require(publishTime <= block.timestamp + 60, "Too far in future");
    if (priceIds.length > MAX_PRICE_IDS) { revert TooManyPriceIds(...); }
    ...
    req.callbackGasLimit = callbackGasLimit;   // stored verbatim
``` [1](#0-0) 

**Root cause 2 — `feePerGasInWei` can be zero:**

`registerProvider()` and `setProviderFee()` accept all three fee parameters without any minimum validation:

```solidity
// Echo.sol lines 381–393
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei          // ← can be 0
) external override {
    ...
    provider.feePerGasInWei = feePerGasInWei;
``` [2](#0-1) 

**Root cause 3 — Fee calculation multiplies by zero:**

```solidity
// Echo.sol lines 248–254
uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei;
uint256 gasFee = callbackGasLimit * providerFeeInWei;  // = 0 when feePerGasInWei = 0
feeAmount = baseFee + providerBaseFee + providerFeedFee + SafeCast.toUint96(gasFee);
``` [3](#0-2) 

**Root cause 4 — `executeCallback` forwards the stored gas limit verbatim:**

```solidity
// Echo.sol lines 176–179
IEchoConsumer(req.requester)._echoCallback{
    gas: req.callbackGasLimit    // ← up to 4,294,967,295 gas
}(sequenceNumber, priceFeeds)
``` [4](#0-3) 

The `callbackGasLimit` field is `uint32` in `EchoState.Request`, giving a maximum of 4,294,967,295 (~4.29 billion gas). [5](#0-4) 

---

### Impact Explanation

**Scenario A — `feePerGasInWei = 0`:**
A requester pays only `pythFeeInWei + baseFeeInWei + feePerFeedInWei * numFeeds` (all flat fees), regardless of `callbackGasLimit`. Setting `callbackGasLimit = type(uint32).max` (~4.29B gas) costs the requester nothing extra. The provider's keeper must submit a transaction forwarding ~4.29B gas to the callback. Since Ethereum mainnet's block gas limit is ~30M, this transaction can never be included in any block. The request is permanently unfulfillable, the requester's funds are locked in the contract, and the keeper wastes compute resources retrying indefinitely.

**Scenario B — `feePerGasInWei = 1 wei` (very small but non-zero):**
`callbackGasLimit = type(uint32).max` costs only `4,294,967,295 wei ≈ 4.29 gwei` in gas fees — negligible on any chain — while still forcing the keeper to attempt a transaction requiring billions of gas.

**Scenario C — `callbackGasLimit` within block gas limit but `feePerGasInWei = 0`:**
Even with a "reasonable" `callbackGasLimit` (e.g., 30M gas), if `feePerGasInWei = 0`, the requester pays zero for the gas component. The provider must execute an expensive callback for free, subsidizing the requester's computation costs entirely.

The concrete impacts are:
- **Provider keeper resource exhaustion** (repeated failed fulfillment attempts)
- **Permanent fund lockup** (requests that can never be fulfilled)
- **Provider financial loss** (executing high-gas callbacks at zero cost to requester)

---

### Likelihood Explanation

The attack requires only that a provider has `feePerGasInWei = 0`. This is a realistic configuration: providers may set it to zero to attract users, or may simply misconfigure it. The `registerProvider()` function imposes no minimum, and the test suite itself uses `DEFAULT_PROVIDER_FEE_PER_GAS = 1 wei` — a value so small that the attack in Scenario B is essentially free. [6](#0-5) 

The attacker entry path is fully permissionless: any address can call `requestPriceUpdatesWithCallback` with an arbitrary `callbackGasLimit`. No privileged role is required.

---

### Recommendation

1. **Enforce a maximum `callbackGasLimit`** in `requestPriceUpdatesWithCallback()`:
   ```solidity
   uint32 public constant MAX_CALLBACK_GAS_LIMIT = 30_000_000; // or chain-appropriate value
   require(callbackGasLimit <= MAX_CALLBACK_GAS_LIMIT, "Gas limit too high");
   ```

2. **Enforce a minimum `feePerGasInWei`** in `registerProvider()` and `setProviderFee()`, or alternatively require that `feePerGasInWei > 0` when `callbackGasLimit > 0`.

3. **Validate in `executeCallback`** that `gasleft() >= callbackGasLimit * 64 / 63` before attempting the callback, reverting early rather than wasting gas on a doomed call.

---

### Proof of Concept

```solidity
// 1. Provider registers with feePerGasInWei = 0
vm.prank(maliciousProvider);
echo.registerProvider(
    1 wei,   // baseFeeInWei
    1 wei,   // feePerFeedInWei
    0        // feePerGasInWei = 0 (no minimum enforced)
);

// 2. Requester submits request with max callbackGasLimit, paying only base fees
bytes32[] memory priceIds = new bytes32[](1);
priceIds[0] = bytes32(uint256(1));

uint96 fee = echo.getFee(maliciousProvider, type(uint32).max, priceIds);
// fee = pythFeeInWei + 1 wei + 1 wei + (type(uint32).max * 0) = tiny flat fee

vm.deal(attacker, fee);
vm.prank(attacker);
uint64 seq = echo.requestPriceUpdatesWithCallback{value: fee}(
    maliciousProvider,
    uint64(block.timestamp),
    priceIds,
    type(uint32).max   // ~4.29 billion gas — exceeds any block gas limit
);

// 3. Provider keeper attempts executeCallback — requires ~4.29B gas, always reverts
// Request is permanently stuck; keeper wastes resources retrying
vm.expectRevert();
echo.executeCallback{gas: 30_000_000}(maliciousProvider, seq, updateData, priceIds);
// Attacker's funds locked, keeper exhausted
```

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
