### Title
Silent `u128 → u64` Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Capacity Accounting - (File: `util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the maximum withdrawable capacity using a `u128` intermediate value and then casts it to `u64` with a bare `as u64` — a silent truncating cast. Every other analogous `u128 → u64` conversion in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern. When the intermediate result exceeds `u64::MAX`, the cast silently wraps to a small value instead of returning `DaoError::Overflow`, producing a grossly incorrect maximum-withdraw figure that propagates into fee calculation and the DAO field update.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

Every other `u128 → u64` narrowing in the same file is checked:

```rust
// line 245
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The truncated result flows into two critical paths:

**Path 1 — DAO field update (`withdrawed_interests`):**

```rust
let maximum_withdraws = rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
    self.transaction_maximum_withdraw(rtx)   // calls calculate_maximum_withdraw
        .and_then(|c| capacities.safe_add(c).map_err(Into::into))
})?;
...
maximum_withdraws.safe_sub(input_capacities).map_err(Into::into)
``` [5](#0-4) 

If `withdraw_counted_capacity` wraps to a small value, `maximum_withdraws` becomes much smaller than `input_capacities`, causing `safe_sub` to fail with an unexpected underflow error rather than `DaoError::Overflow`. This causes `dao_field_with_current_epoch` to return an error, rejecting the block.

**Path 2 — Fee calculation:**

```rust
pub fn transaction_fee(&self, rtx: &ResolvedTransaction) -> Result<Capacity, DaoError> {
    let maximum_withdraw = self.transaction_maximum_withdraw(rtx)?;
    rtx.transaction
        .outputs_capacity()
        .and_then(|y| maximum_withdraw.safe_sub(y))
        .map_err(Into::into)
}
``` [6](#0-5) 

A truncated `maximum_withdraw` smaller than `outputs_capacity` causes `safe_sub` to fail, causing the `ContextualTransactionVerifier` to reject the transaction. [7](#0-6) 

Note that `CapacityVerifier` explicitly **skips** the `OutputsSumOverflow` check for DAO withdraw transactions, delegating enforcement to the DAO type script:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    // ... OutputsSumOverflow check
}
``` [8](#0-7) 

This means the Rust node's only capacity-accounting gate for DAO withdrawals is the fee calculator, which uses the truncated value.

### Impact Explanation

When `withdraw_counted_capacity > u64::MAX`, the silent cast produces a small wrapped value. Two concrete effects follow:

1. **Incorrect DAO field bookkeeping**: `withdrawed_interests` underflows → `current_s` is not decremented correctly → the DAO savings pool (`S`) is inflated by the full withdrawn interest amount. Future depositors earn interest calculated against an inflated `S`, creating a systematic capacity accounting error across the chain.

2. **False rejection of valid DAO withdrawal transactions**: The fee calculator returns an error for a transaction that the on-chain DAO type script would accept, causing the Rust node to reject a valid block. If the on-chain script and the Rust node diverge on validity, this is a consensus-layer accounting failure.

### Likelihood Explanation

The condition `withdraw_counted_capacity > u64::MAX` requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  2^64
```

`counted_capacity` is bounded by the total CKB supply (~3.36 × 10¹⁸ shannons). For overflow, `withdrawing_ar / deposit_ar` must exceed ~5.5×. Starting from the genesis `ar = 10^16`, this ratio requires `ar` to reach ~5.5 × 10^16. At normal secondary issuance rates this takes an extremely long time, making exploitation on mainnet impractical in the near term. However, the defect is a latent time-bomb: the code is structurally inconsistent, the error path is silent rather than checked, and the condition becomes reachable as `ar` accumulates over decades or in test/devnet environments with accelerated issuance.

### Recommendation

Replace the silent cast with the same checked pattern used everywhere else in the file:

```rust
// Before (line 155-156):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After:
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    ).safe_add(occupied_capacity)?;
``` [9](#0-8) 

This makes the overflow explicit and consistent with `miner_issuance128`, `ar_increase128`, and `reward128` conversions in the same file.

### Proof of Concept

Construct a DAO cell with `counted_capacity` near `u64::MAX` and a withdrawing header whose `ar` is sufficiently larger than the deposit header's `ar` such that:

```
counted_capacity × withdrawing_ar  >  deposit_ar × 2^64
```

Call `DaoCalculator::calculate_maximum_withdraw` with these parameters. With the current code, `withdraw_counted_capacity as u64` silently wraps to a small value (e.g., near 0), and `withdraw_capacity` is returned as approximately `occupied_capacity` — far below the true maximum. The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` (lines 296–350) demonstrates that the overflow path is reachable and expected to error, but it catches the error via the subsequent `safe_add` overflow, not via the `as u64` cast itself — confirming the cast is the unguarded step. [10](#0-9)
