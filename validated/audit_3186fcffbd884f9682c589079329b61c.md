### Title
Unauthorized ERC-20 Fee Drain via `caller_address` Misuse in `update_price_feeds_internal` - (File: `target_chains/starknet/contracts/src/pyth.cairo`)

---

### Summary

The Starknet Pyth contract's `update_price_feeds_internal` function collects ERC-20 fees by calling `transferFrom(caller_address, pyth_contract, fee)`, where `caller_address` is the **direct caller** of the Pyth contract — not the original transaction initiator. When an integrating contract (e.g., a DeFi protocol) pre-approves the Pyth contract to spend its fee tokens and exposes a publicly callable function that internally calls `update_price_feeds`, an attacker can trigger that function to drain the integrating contract's entire pre-approved token allowance.

---

### Finding Description

In `update_price_feeds_internal`, the fee payer is resolved as:

```cairo
let execution_info = get_execution_info().unbox();
let caller = execution_info.caller_address;   // direct caller, not tx originator
let contract = execution_info.contract_address;
```

This `caller` is then passed directly to `transfer_fee`, which executes:

```cairo
fee_contract.transferFrom(caller, contract, total_fee)
``` [1](#0-0) [2](#0-1) 

In Starknet's execution model, `caller_address` is the **immediate calling contract**, not the account that originated the transaction. When a DeFi protocol (e.g., a lending market or DEX) integrates Pyth and pre-approves the Pyth contract to spend its STRK/ETH tokens — a common pattern to abstract fee payment from end users — the Pyth contract will call `transferFrom(dexContract, pythContract, fee)` whenever the DEX's public function triggers `update_price_feeds`. An attacker who can invoke that public function controls how much fee is charged (bounded by `num_updates` in the submitted VAA) and can repeatedly drain the DEX's pre-approved balance.

The Starknet documentation in the codebase explicitly notes that callers must pre-approve the Pyth contract before calling `update_price_feeds`: [3](#0-2) 

This design encourages integrating contracts to hold standing allowances, which is precisely the condition the attack exploits.

---

### Impact Explanation

An attacker can drain the entire pre-approved ERC-20 token balance (STRK or ETH) of any integrating contract that:
1. Has granted a standing allowance to the Pyth contract, and
2. Exposes a publicly callable function that internally calls `update_price_feeds`.

The stolen tokens are transferred to the Pyth contract itself (as collected fees), not to the attacker directly — but the integrating contract suffers a permanent loss of its fee reserve. If governance later raises the `single_update_fee`, the per-call damage scales proportionally. The impact class is **direct theft of ERC-20 tokens from third-party contracts**.

---

### Likelihood Explanation

Likelihood is **medium**. The attack requires an integrating contract with a standing allowance and a publicly callable entry point — a realistic and common integration pattern explicitly encouraged by Pyth's own documentation. The attacker needs no special privileges, no leaked keys, and no governance access. Any unprivileged transaction sender can execute the attack once a suitable integrating contract is deployed.

---

### Recommendation

Replace `execution_info.caller_address` with `execution_info.tx_info.account_contract_address` (the originating account of the transaction) as the fee payer, or require the caller to explicitly pass the payer address and validate it matches `get_caller_address()`. This mirrors the Taiko fix: track the actual transaction initiator rather than the intermediate call-stack address.

Alternatively, document clearly that integrating contracts **must not** grant standing allowances to the Pyth contract and must instead approve the exact fee amount atomically within the same transaction.

---

### Proof of Concept

1. Deploy an integrating contract `IntegratorDEX` on Starknet that:
   - Holds 1,000 STRK and has called `strk.approve(pyth_contract, 1000e18)`.
   - Exposes `public fn refreshAndSwap(data: ByteBuffer)` which calls `pyth.update_price_feeds(data)`.

2. Attacker obtains a valid Wormhole VAA containing `N` price updates (e.g., from Hermes).

3. Attacker calls `IntegratorDEX.refreshAndSwap(validVAA)`.

4. Inside `update_price_feeds_internal`:
   - `caller = IntegratorDEX_address` (not the attacker's address).
   - `transfer_fee` checks `allowance(IntegratorDEX, pyth) >= fee` → passes.
   - `transferFrom(IntegratorDEX, pyth, fee)` executes, draining `IntegratorDEX`'s STRK.

5. Attacker repeats until `IntegratorDEX`'s allowance or balance is exhausted.
