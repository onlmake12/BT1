Audit Report

## Title
Out-of-Bounds Index Panic in `remove_unconfirmed_tx` When `tx_age > 1000` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary

`TxConfirmStat::remove_unconfirmed_tx` correctly guards `block_unconfirmed_txs` access for `tx_age >= 1000`, but the immediately following `count_failure` branch unconditionally indexes `confirm_blocks_to_failed_txs[tx_age - 1]` with no corresponding bounds check. When a tracked tx is evicted after more than 1000 blocks with `count_failure = true`, `tx_age - 1 >= 1000` exceeds the Vec's length of 1000, causing a Rust index-out-of-bounds panic that crashes the tx-pool service thread.

## Finding Description

`confirm_blocks_to_failed_txs` is initialized with length `MAX_CONFIRM_BLOCKS = 1000`, giving valid indices `0..=999`: [1](#0-0) 

In `remove_unconfirmed_tx`, the guard at line 208 routes `block_unconfirmed_txs` access for aged-out txs (`tx_age >= 1000`) to the `old_unconfirmed_txs` counter. However, the `count_failure` branch immediately below has **no bounds check** on `tx_age`: [2](#0-1) 

When `tx_age = 1001`, `tx_age - 1 = 1000` is out of bounds for a Vec of length 1000 → **panic**. Note: `tx_age = 1000` is the boundary case where `tx_age - 1 = 999` is still valid; the panic triggers at `tx_age >= 1001`.

`drop_tx` always passes `count_failure = true`: [3](#0-2) 

`drop_tx_inner` passes `self.best_height` as `tip_height`, which is updated on every committed block: [4](#0-3) 

`reject_tx` calls `drop_tx`: [5](#0-4) 

The reject callback registered in `shared_builder.rs` calls `fee_estimator.reject_tx` on every pool eviction: [6](#0-5) 

The pool eviction paths (`remove_expired`, `limit_size`) both call `callbacks.call_reject`: [7](#0-6) [8](#0-7) 

The full call chain is: `remove_expired`/`limit_size` → `callbacks.call_reject` → `fee_estimator.reject_tx` → `drop_tx` → `drop_tx_inner(tx_hash, true)` → `remove_unconfirmed_tx(..., count_failure=true)` → `confirm_blocks_to_failed_txs[tx_age - 1]` → **panic** when `tx_age >= 1001`.

A tx is only tracked when it enters the pool at the current tip height (`height == self.best_height`): [9](#0-8) 

So `tx_age = best_height - entry_height`. For the panic, the chain must advance at least 1001 blocks after the tx was tracked before eviction occurs.

## Impact Explanation

A Rust Vec index-out-of-bounds panics unconditionally at runtime with no recovery. The tx-pool service thread panics, crashing the node's transaction processing. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation

Any unprivileged user can submit a low-fee-rate tx via RPC (`send_transaction`) or P2P relay. The default pool expiry is 12 hours; at ~8 seconds/block, that is ~5400 blocks — far exceeding the 1001-block threshold. Any tracked tx that remains unconfirmed for more than 1000 blocks and is then evicted by pool expiry or size pressure will trigger the panic. No special privileges are required. The attacker only needs to submit a tx with a fee rate low enough that it will not be confirmed within 1000 blocks, then wait for natural eviction.

## Recommendation

Add a bounds check before accessing `confirm_blocks_to_failed_txs` in `remove_unconfirmed_tx`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

Or use `.get_mut(tx_age - 1)` to silently skip out-of-range ages, consistent with how `block_unconfirmed_txs` already handles the overflow case via `old_unconfirmed_txs`.

## Proof of Concept

Minimal unit test (add to the `tests` module in `confirmation_fraction.rs`):

```rust
#[test]
fn test_no_oob_on_old_tx_drop() {
    let buckets = vec![FeeRate::from_u64(1000)];
    let mut stat = TxConfirmStat::new(buckets, 1000, 0.993);
    // tx entered at height 0, now at height 1001 → tx_age = 1001
    // tx_age - 1 = 1000 is out of bounds for Vec of length 1000 → panics without fix
    stat.remove_unconfirmed_tx(0, 1001, 0, true);
}
```

Manual steps:
1. Start a CKB node with the `ConfirmationFraction` fee estimator enabled (non-IBD mode).
2. Submit a tx with a fee rate below the market rate so it will not be confirmed.
3. Advance the chain by 1001+ blocks (e.g., via a private testnet).
4. Allow the pool expiry timer to fire (or trigger `limit_size` eviction by filling the pool).
5. The eviction path calls `remove_unconfirmed_tx` with `tx_age >= 1001` and `count_failure = true`.
6. `confirm_blocks_to_failed_txs[1000]` panics → node tx-pool thread crashes.
