Audit Report

## Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Eviction Snapshot — (`File: tx-pool/src/component/pool_map.rs`)

## Summary

`add_entry` snapshots post-add totals via `updated_stat_for_add_tx` before calling `check_and_record_ancestors`. When `check_and_record_ancestors` evicts entries to satisfy `max_ancestors_count`, each eviction correctly decrements `self.total_tx_size` via `update_stat_for_remove_tx`. The final two lines of `add_entry` then unconditionally overwrite `self.total_tx_size` and `self.total_tx_cycles` with the stale pre-eviction snapshot, permanently inflating the pool-size tracker and causing `limit_size` to expel legitimate transactions.

## Finding Description

`PoolMap` maintains running totals `total_tx_size` and `total_tx_cycles`. [1](#0-0) 

At the top of `add_entry`, before any mutations, `updated_stat_for_add_tx` computes a snapshot of what the totals *would* be after adding the new entry and stores them in locals: [2](#0-1) 

`updated_stat_for_add_tx` is a pure read — it does not modify `self`: [3](#0-2) 

`check_and_record_ancestors` is then called. When `ancestors_count > max_ancestors_count` but reducible by evicting cell-dep parents, it calls `remove_entry_and_descendants` in a loop: [4](#0-3) 

Each `remove_entry_and_descendants` call reaches `remove_entry`, which calls `update_stat_for_remove_tx`, correctly decrementing `self.total_tx_size` and `self.total_tx_cycles` for each evicted entry: [5](#0-4) [6](#0-5) 

After `check_and_record_ancestors` returns, `add_entry` unconditionally overwrites the correctly-updated fields with the stale locals: [7](#0-6) 

The stale locals were computed before any evictions, so they do not reflect the decrements applied during eviction. The result:

```
self.total_tx_size  == old_total + new_entry_size          (stale, inflated)
actual pool size    == old_total + new_entry_size - Σ(evicted_sizes)
```

No existing guard prevents this: `update_stat_for_remove_tx` correctly updates `self.total_tx_size` during eviction, but that update is immediately discarded by the overwrite at lines 218–219.

## Impact Explanation

`total_tx_size` is the sole metric used by `limit_size` to enforce `max_tx_pool_size`: [8](#0-7) 

With an inflated tracker, `limit_size` believes the pool is over-capacity when it is not, and evicts legitimate high-fee-rate transactions. Each such eviction correctly decrements the tracker, so the inflation is consumed — but only after an excess number of valid transactions have been expelled. An attacker can repeat the trigger to keep the tracker perpetually inflated, effectively preventing honest transactions from remaining in the pool. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

## Likelihood Explanation

The trigger is reachable by any unprivileged `send_transaction` caller with no special privileges or victim interaction required. The attacker submits N > `max_ancestors_count` transactions each referencing the same in-pool cell-dep output, then submits one transaction consuming that output. This enters the eviction branch at line 603, removes `(N − max_ancestors_count + 1)` entries, and inflates `total_tx_size` by their combined sizes. The attack is cheap (small transactions suffice), repeatable, and compounds across rounds. The integration test `TxPoolLimitAncestorCount` in `test/src/specs/tx_pool/limit.rs` confirms this exact flow is a supported, exercised code path.

## Recommendation

Move the stat-update commit to **after** `check_and_record_ancestors` completes, incorporating any evictions. Replace the pre-computed snapshot pattern with a post-eviction delta:

```rust
// After check_and_record_ancestors returns `evicts`:
let evicted_size:   usize = evicts.iter().map(|e| e.size).sum();
let evicted_cycles: Cycle = evicts.iter().map(|e| e.cycles).sum();
self.total_tx_size = self.total_tx_size
    .saturating_add(entry.size)
    .saturating_sub(evicted_size);
self.total_tx_cycles = self.total_tx_cycles
    .saturating_add(entry.cycles)
    .saturating_sub(evicted_cycles);
```

Remove the pre-computed `(total_tx_size, total_tx_cycles)` locals and the `updated_stat_for_add_tx` call from the top of `add_entry` (overflow checking can be re-added inline before insertion). Alternatively, call `recompute_total_stat` after `check_and_record_ancestors`, consistent with the fallback already used in `update_stat_for_remove_tx`: [9](#0-8) 

## Proof of Concept

Using `max_ancestors_count = 25`, entry size S:

1. Submit 30 transactions T₁…T₃₀, each referencing cell-dep output `X` (an in-pool UTXO). `self.total_tx_size = 30S`.
2. Submit T_consume spending output `X`.
3. `add_entry(T_consume)` executes:
   - **(A)** `total_tx_size_local = 30S + S_consume` (snapshot, line 210–211)
   - **(B)** `check_and_record_ancestors` finds 31 ancestors > 25; evicts T₁…T₆ via `remove_entry_and_descendants`. Each calls `update_stat_for_remove_tx` → `self.total_tx_size = 24S` (line 247)
   - **(C)** `self.total_tx_size = total_tx_size_local = 30S + S_consume` (overwrite, line 218)
4. Actual pool size = `24S + S_consume`. Tracker = `30S + S_consume`. **Inflation = 6S**.
5. `limit_size` is called; it evicts 6S worth of legitimate transactions that should have remained.
6. Repeat from step 1 to accumulate further inflation.
