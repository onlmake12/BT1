### Title
Missing Per-Peer Rate Limiter in Filter Protocol Enables Amplified Resource Exhaustion DoS — (`sync/src/filter/mod.rs`)

### Summary

The `BlockFilter` protocol handler processes `GetBlockFilters` messages with no per-peer rate limiting. Each such message triggers up to `BATCH_SIZE=1000` database reads and allocates up to 1.8 MB of response data. The `Relayer` and `HolePunching` protocols both carry an explicit `rate_limiter` field and enforce it before dispatch; `BlockFilter` has neither. Any unprivileged connected peer can exploit this asymmetry to exhaust CPU and memory on every full node that has block-filter data built.

---

### Finding Description

**`BlockFilter` struct — no rate limiter field** [1](#0-0) 

The struct holds only `shared: Arc<SyncShared>`. Compare with `Relayer`: [2](#0-1) 

and `HolePunching`: [3](#0-2) 

Both of those protocols enforce the limiter before any message dispatch:

- `Relayer::try_process` checks `self.rate_limiter.check_key(&(peer, message.item_id()))` and returns `TooManyRequests` on failure. [4](#0-3) 
- `HolePunching::received` does the same check immediately after parsing. [5](#0-4) 

**`BlockFilter::received` — no rate check, direct dispatch** [6](#0-5) 

After parsing, the message is handed directly to `self.process(...)` → `self.try_process(...)` → `GetBlockFiltersProcess::execute()` with zero throttling.

**`GetBlockFiltersProcess::execute` — up to 1000 DB reads + 1.8 MB allocation per message** [7](#0-6) [8](#0-7) 

Each `GetBlockFilters(start_number=0)` message causes the handler to loop up to 1000 times, calling `get_block_hash` and `get_block_filter` (two DB reads per iteration), and accumulates up to 1.8 MB of response data before sending.

---

### Impact Explanation

The request/response asymmetry is severe:

| Side | Cost |
|---|---|
| Attacker | 8-byte message per request |
| Victim | ≤2000 DB reads + ≤1.8 MB allocation per message |

With `max_peers` (default 125) concurrent attackers each flooding at even a modest rate (e.g., 50 msg/s), the aggregate load is ~12.5 M DB reads/s and ~11 GB/s of allocation pressure. This can saturate I/O, exhaust memory, and stall the async runtime, causing the node to stop processing legitimate sync and relay messages — effectively taking it off the network.

---

### Likelihood Explanation

- Requires only a standard P2P connection — no key, no privilege, no PoW.
- The attack message is trivially constructable (8 bytes: a `u64` block number).
- The precondition (filter data built) is the normal production state for any node running with `--block-filter-enable`, which is the recommended configuration for light-client support.
- The gap is directly visible by comparing `BlockFilter` to `Relayer` and `HolePunching`, both of which were explicitly hardened with the same `governor`-based rate limiter.

---

### Recommendation

Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`), initialize it in `BlockFilter::new` with the same 30 req/s-per-peer quota, and check it at the top of `try_process` before dispatching to any `GetBlockFilters*` processor. Also call `rate_limiter.retain_recent()` in `disconnected`.

---

### Proof of Concept

```
1. Run a CKB full node with --block-filter-enable (filter data built for N blocks).
2. Open max_peers TCP connections to the node's P2P port, each performing the Filter protocol handshake.
3. From each connection, send GetBlockFilters { start_number: 0 } in a tight loop.
4. Monitor node CPU (DB read threads), RSS memory, and relay/sync message latency.
5. Assert: node becomes unresponsive to legitimate peers within seconds; memory and CPU spike to saturation.
```
