Audit Report

## Title
VerifyQueue Enforces Hardcoded 256 MB Ceiling Independent of Operator-Configured `max_tx_pool_size`, Enabling Unbounded Memory Consumption — (`tx-pool/src/component/verify_queue.rs`)

## Summary
The `VerifyQueue` admits transactions up to a compile-time constant of 256 MB (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`) regardless of the operator's `max_tx_pool_size` setting. The main pool's `limit_size()` eviction loop is blind to verify-queue occupancy, and `TxPoolInfo.total_tx_size` excludes verify-queue bytes. Any unprivileged RPC caller or P2P peer can saturate the verify queue to 256 MB on a node configured with a far smaller memory budget, potentially triggering OOM-kill.

## Finding Description

**Hardcoded ceiling independent of config:**
`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` is a compile-time constant of 256 MB. [1](#0-0) 

**`is_full()` references only the hardcoded constant, never `TxPoolConfig::max_tx_pool_size`:** [2](#0-1) 

**`add_tx()` gates admission solely on `is_full()`:** [3](#0-2) 

**`limit_size()` evicts only from `pool_map`; the verify queue is never consulted:** [4](#0-3) 

**`TxPoolInfo` exposes verify-queue occupancy as a count only; `total_tx_size` is `pool_map`-only:** [5](#0-4) 

**The only pre-queue gate, `non_contextual_verify`, rejects only transactions > 512 KB and cellbase transactions — no fee-rate floor, no rate limit:** [6](#0-5) 

**`TRANSACTION_SIZE_LIMIT` is 512 KB:** [7](#0-6) 

**Exploit flow:**
1. Operator sets `max_tx_pool_size = 10 MB`.
2. Attacker submits ~500 syntactically valid transactions each ≈ 512 KB via `send_transaction` RPC or P2P.
3. Each passes `non_contextual_verify` (size ≤ 512 KB, not cellbase).
4. Each is admitted to `VerifyQueue` because `is_full()` compares against 256 MB, not 10 MB.
5. Script verification is CPU-bound and asynchronous; transactions linger in the queue.
6. Node RSS grows by ≈ 256 MB while `total_tx_size` in `TxPoolInfo` remains near 0 — no operator-visible alarm.
7. On a node with ≤ 512 MB total RAM, OOM-kill terminates the process.

## Impact Explanation

**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

An operator-configured memory budget is bypassed by a factor of up to 25× (256 MB verify queue vs. 10 MB `max_tx_pool_size`). The crash is deterministic once the queue is saturated on a RAM-constrained node. Block production and P2P connectivity are halted for the duration of the OOM event. The operator's monitoring metric (`total_tx_size`) provides no warning because verify-queue bytes are excluded from it.

## Likelihood Explanation

No privilege is required. Any RPC caller or P2P peer can submit transactions. The attacker needs only ≈ 500 syntactically valid transactions of ≈ 512 KB each — no keys, no hashpower, no Sybil capability. The attack is repeatable: after the node restarts, the queue is empty and the same flood can be re-applied. The cost is negligible because transactions are rejected (not mined), so no on-chain fees are paid.

## Recommendation

1. Replace the hardcoded `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` with a value derived from `TxPoolConfig::max_tx_pool_size` (e.g., pass `max_tx_pool_size` into `VerifyQueue::new()` and store it as an instance field), so the verify queue ceiling scales with the operator's memory budget.
2. Include `verify_queue.total_tx_size` in the combined size accounting reported by `TxPoolInfo` and consulted by `limit_size()`, so in-flight transactions are visible to both operators and the eviction loop.
3. Optionally, enforce a per-peer submission rate limit in the verify-queue admission path to bound the rate at which any single source can fill the staging area.

## Proof of Concept

1. Configure a CKB node with `max_tx_pool_size = 10_000_000` (10 MB) in `ckb.toml`.
2. Generate ≈ 500 syntactically valid transactions each close to 512 KB (e.g., large witness data with a valid but slow-to-verify lock script).
3. Submit all transactions via `send_transaction` RPC in rapid succession (no waiting for verification results).
4. Call `tx_pool_info` RPC: observe `verify_queue_size` ≈ 500 while `total_tx_size` ≈ 0.
5. Monitor node RSS: it grows by ≈ 256 MB — 25× the configured `max_tx_pool_size`.
6. On a node with ≤ 512 MB total RAM, the process is OOM-killed.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
