### Title
Stale Epoch Number Used in RFC0044 MMR Extension Activation Check Causes Off-by-One at Epoch Boundary — (`tx-pool/src/block_assembler/mod.rs`)

---

### Summary

In `BlockAssembler::build_extension`, the activation check for RFC0044 (the MMR chain-root block extension required by the light-client protocol) reads the **tip block's epoch number** — the parent of the block being assembled — instead of the **candidate block's epoch number**. The same stale epoch number is used symmetrically in `BlockExtensionVerifier::verify`. Because both sides of the protocol use the parent's epoch number, the first block of the RFC0044 activation epoch is assembled and accepted **without** the mandatory MMR chain-root extension, silently breaking MMR continuity for light clients at the exact activation boundary.

The code itself acknowledges this:

> *"The use of the epoch number of the tip here leads to an off-by-one bug, so be careful, it needs to be preserved for consistency reasons and not fixed directly."*

---

### Finding Description

**Root cause — block assembler (`tx-pool/src/block_assembler/mod.rs`, line 568–570):**

```rust
let mmr_activate = snapshot
    .consensus()
    .rfc0044_active(tip_header.epoch().number());   // ← tip epoch, not candidate epoch
```

`tip_header` is the current chain tip, i.e., the *parent* of the block being assembled. The candidate block's epoch number may differ (specifically at an epoch boundary). The correct value to pass is the candidate block's epoch number. [1](#0-0) 

**Root cause — block verifier (`verification/contextual/src/contextual_block_verifier.rs`, line 540–543):**

```rust
let mmr_active = self
    .context
    .consensus
    .rfc0044_active(self.parent.epoch().number());  // ← parent epoch, not block epoch
```

`self.parent` is the same tip block. Both sides consistently use the parent's epoch number, so they agree with each other — but both are wrong relative to the RFC. [2](#0-1) 

**Activation threshold (`spec/src/consensus.rs`, line 1005–1011):**

```rust
pub fn rfc0044_active(&self, target: EpochNumber) -> bool {
    let rfc0044_active_epoch = ...;   // 8651 mainnet, 5711 testnet
    target >= rfc0044_active_epoch
}
``` [3](#0-2) 

**Concrete scenario at the activation boundary:**

| Block | Parent epoch | Candidate epoch | `rfc0044_active(parent)` | Extension included? | Correct? |
|---|---|---|---|---|---|
| Last block of epoch 8650 | 8649 | 8650 | false | No | ✓ |
| **First block of epoch 8651** | **8650** | **8651** | **false** | **No** | **✗** |
| Second block of epoch 8651 | 8651 | 8651 | true | Yes | ✓ |

The first block of epoch 8651 is assembled without the MMR chain-root extension and accepted by the verifier without it, because both check `rfc0044_active(8650) = false` instead of `rfc0044_active(8651) = true`.

---

### Impact Explanation

RFC0044 (CKB Light Client Protocol) requires every block from the activation epoch onward to embed the MMR chain-root hash in its extension field. The first block of the activation epoch is missing this field. Light clients that walk the MMR chain to verify headers encounter a gap: the MMR root for block `N` (first block of epoch 8651) is absent, breaking the cryptographic chain of custody. A light client cannot trustlessly verify any block at or after that position without special-casing the gap, undermining the security guarantee of the light-client protocol for users who rely on it. [4](#0-3) 

---

### Likelihood Explanation

The bug fires exactly once per activation epoch boundary. On mainnet, RFC0044 activated at epoch 8651 (`RFC0044_ACTIVE_EPOCH = 8651`); on testnet at epoch 5711. Any miner who produced the first block of epoch 8651 on mainnet triggered this path automatically — no special action required. The block assembler's template omits the extension, the verifier accepts the block, and the gap is permanently committed to the chain. The same pattern would recur for any future softfork that uses the same `rfc0044_active`-style epoch-number check. [5](#0-4) 

---

### Recommendation

In `build_extension`, compute the candidate block's epoch number (via `consensus.next_epoch_ext(tip_header, ...)`) and pass that to `rfc0044_active` instead of `tip_header.epoch().number()`. Apply the same correction in `BlockExtensionVerifier::verify` (use the block's own epoch number, not the parent's). Because the comment warns that a direct fix risks a consensus split, any correction must be coordinated as a consensus-layer change (hardfork or carefully staged rollout) to ensure all nodes adopt the corrected check simultaneously.

---

### Proof of Concept

1. Run a CKB node on testnet approaching epoch 5711.
2. Call `get_block_template` RPC when the tip is the last block of epoch 5710.
3. Observe that the returned template has no `extension` field (MMR root absent).
4. Submit the block; it is accepted by the network.
5. A light-client node attempting to verify the MMR chain root for this block finds no extension and cannot complete verification, confirming the gap.
