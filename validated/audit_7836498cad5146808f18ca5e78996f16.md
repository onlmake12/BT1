Audit Report

## Title
Sync Thread Indefinite Block via Saturating `bounded(24)` `process_block_tx` Channel — (`chain/src/init.rs`, `util/channel/src/lib.rs`)

## Summary
`ChainController::asynchronous_process_lonely_block` calls `Request::call_without_response`, which performs a blocking `crossbeam_channel::Sender::send()` on the `bounded(24)` `process_block_tx` channel with no timeout or fallback. If the channel is full, the calling sync thread stalls indefinitely. Because the sync protocol handler calls this path synchronously before returning, a stalled call prevents the node from processing any further P2P messages until the channel drains.

## Finding Description
**Blocking send — `util/channel/src/lib.rs` lines 44–50:**
```rust
pub fn call_without_response(sender: &Sender<Request<A, R>>, arguments: A) {
    let (responder, _response) = oneshot::channel();
    let _ = sender.send(Request { responder, arguments });
}
```
`Sender` is `crossbeam_channel::Sender` (re-exported at line 2). `send()` on a bounded channel blocks until a slot is free. The `let _ =` discards only the `Result`; it does not make the call non-blocking. No `try_send`, `send_timeout`, or deadline exists anywhere in the chain crate (confirmed by grep).

**Channel capacity — `chain/src/init.rs` line 93:**
```rust
let (process_block_tx, process_block_rx) = channel::bounded(24);
```

**Call site in the sync thread — `chain/src/chain_controller.rs` lines 61–63:**
```rust
pub fn asynchronous_process_lonely_block(&self, lonely_block: LonelyBlock) {
    Request::call_without_response(&self.process_block_sender, lonely_block);
}
```
`asynchronous_process_remote_block` (line 52–59) delegates directly to this function. `sync/src/synchronizer/block_process.rs` line 76 calls `asynchronous_process_remote_block` from within `BlockProcess::execute()`, which is invoked synchronously by the sync protocol handler before returning `Status::ignored()` (line 80). If `send()` blocks, `execute()` never returns, and the sync thread cannot service any further peer message.

**Consumer drain rate:** `ChainService::start_process_block` (`chain/src/chain_service.rs` lines 37–70) processes one block per loop iteration: non-contextual verification (fast) + DB insert + orphan broker dispatch. During rapid block arrival or chain reorganization the 24-slot buffer can fill before the consumer catches up.

**Existing guard insufficient:** `shared.new_block_received(&block)` (`block_process.rs` line 43) only deduplicates already-seen blocks; it does not rate-limit or bound the number of unique blocks that can be queued.

## Impact Explanation
A stalled sync thread means the node stops processing all incoming P2P messages (block announcements, headers, transactions, ping/pong). Peers time out and disconnect, effectively isolating the node from the network for the duration of the stall. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
An attacker does not need to mine blocks. They only need to relay 24 unique valid blocks (e.g., sourced from a competing fork, a chain reorganization, or rapid tip succession on the public network) to the victim node faster than `ChainService` can drain the channel. A well-connected relay node can achieve this in a short burst. The attack is repeatable: once the consumer catches up, the attacker can re-saturate the channel immediately.

## Recommendation
Replace the blocking `send()` in `call_without_response` with `try_send()` and drop the message (with a warning log) when the channel is full, or introduce a per-call deadline via `send_timeout`. The sync/relay path must never block on a downstream bounded channel. Alternatively, increase the channel capacity significantly and add backpressure metrics, but a non-blocking send with a drop-and-warn policy is the correct fix for the async path.

## Proof of Concept
1. Instantiate `build_chain_services` with a slow `insert_block` mock (e.g., `thread::sleep(50ms)` per block) so the consumer drains slowly.
2. From a test thread, call `chain_controller.asynchronous_process_lonely_block(block_i)` for 24 distinct blocks in rapid succession — all return immediately (slots available).
3. Spawn a second thread that calls `asynchronous_process_lonely_block` with a 25th distinct block.
4. Assert the second thread does not return within a 500 ms deadline — it will stall indefinitely, confirming the blocking `send()` on the full `bounded(24)` channel.
5. Without the mock slowdown, reproduce on a live node by relaying 24+ valid tip blocks in rapid succession from a high-bandwidth peer.
