Audit Report

## Title
Unauthenticated `content.from` Field Allows Spoofed `ConnectionRequestDelivered` to Consume Victim's `inflight_requests` and Redirect NAT Traversal — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary
In `ConnectionRequestDeliveredProcess::execute`, the branch that identifies the local node as the hole-punch initiator is gated solely on `content.from == self_peer_id`. Because `content.from` is taken directly from the attacker-supplied message and is never validated against the authenticated peer ID of the actual sending session (`self.peer`), any connected peer can set `from = victim_peer_id`, `route = []`, and attacker-controlled `listen_addrs` to unconditionally consume the victim's `inflight_requests` entry and redirect NAT traversal to attacker-controlled addresses. This enables a persistent, low-cost DoS on the victim's entire hole-punching capability.

## Finding Description
**Root cause:** At line 151 of `connection_request_delivered.rs`, `execute()` compares `self_peer_id` (the local peer ID) against `content.from` (a field the remote sender controls freely):

```rust
match content.route.last() {
    Some(next_peer_id) => self.forward_delivered(next_peer_id).await,
    None => {
        let self_peer_id = self.protocol.network_state.local_peer_id();
        if self_peer_id != &content.from {
            self.forward_delivered(&content.from).await
        } else {
            // "I am the initiator" branch
            let request_start = self.protocol.inflight_requests.remove(&content.to);
            ...
            self.try_nat_traversal(ttl, content.listen_addrs);
        }
    }
}
```

There is no check that `content.from` matches the authenticated peer ID of the sending session (`self.peer`). The `self.peer` field (a `PeerIndex`) is available in the struct but is only used in `respond_sync` to send the sync reply back to the actual sender — it is never used to validate `content.from`.

**Exploit path:**

1. Attacker establishes a standard P2P connection to the victim (session ID `attacker_sid`).
2. Victim's `notify()` loop broadcasts a `ConnectionRequest` for `target_peer_id` via `filter_broadcast` and inserts `target_peer_id → timestamp` into `inflight_requests` (lines 224–241 of `mod.rs`). The `to_peer_id` is observable from gossip.
3. Attacker crafts a `ConnectionRequestDelivered` with:
   - `from = victim_local_peer_id` (public, exchanged during identify handshake)
   - `to = target_peer_id` (observed from gossip)
   - `route = []` (empty)
   - `listen_addrs = [attacker_ip:attacker_port]`
4. Message passes all guards:
   - `listen_addrs` non-empty check (line 125) ✓
   - `route` length check (line 130) ✓
   - `forward_rate_limiter` check (lines 134–145): key `(victim_peer_id, target_peer_id, item_id)` is new ✓
5. `content.route.last()` returns `None` (line 147). `self_peer_id == content.from` → `else` branch entered (line 154).
6. `inflight_requests.remove(&content.to)` (line 160) consumes the entry.
7. `respond_sync` sends a `ConnectionSync` back to `self.peer` (the attacker, line 228).
8. `try_nat_traversal(ttl, content.listen_addrs)` (line 171) spawns a task that opens raw TCP connections toward the attacker-controlled address for up to 30 seconds.
9. Any subsequent legitimate `ConnectionRequestDelivered` for the same `target_peer_id` hits the `None` arm at line 175 and is silently ignored.

**Why existing guards fail:**
- The `forward_rate_limiter` (lines 134–145) is keyed on `(content.from, content.to, msg_item_id)`. Since each `inflight_requests` entry is consumed on the first hit, one message per entry is sufficient; the rate limiter does not prevent the attack.
- The `listen_addrs` validation (lines 57–63) only enforces that the embedded peer ID matches `content.to`; the IP and port are fully attacker-controlled.
- There is no check anywhere in `execute()` that `content.from` matches the peer ID of the actual sending session.

## Impact Explanation
**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker with a single standard P2P connection can watch gossip for `ConnectionRequest` broadcasts and immediately cancel every hole-punch the victim initiates, permanently preventing the victim from establishing connections to NAT-ed peers via this mechanism. Applied simultaneously to many nodes (each requiring only one connection), this degrades the reachability of NAT-ed nodes across the network, reduces peer diversity, and can cause network-wide connectivity degradation as nodes fail to maintain sufficient outbound connections. Additionally, each attack causes the victim to open outbound TCP connections toward an attacker-controlled address for up to 30 seconds, wasting file descriptors and connection-pool slots.

## Likelihood Explanation
- Requires only a standard, unprivileged P2P connection to the victim — no special role or key.
- The victim's local peer ID is public (exchanged during the identify handshake).
- The `to_peer_id` values in `inflight_requests` are directly observable from gossip `ConnectionRequest` broadcasts (lines 224–234 of `mod.rs`).
- The attack requires sending one well-formed message per `inflight_requests` entry; it is trivially scriptable and repeatable every time the victim's `notify()` loop fires (every 5 minutes per `CHECK_INTERVAL`).
- The rate limiter does not prevent the attack because each entry is consumed on the first hit.

## Recommendation
Before entering the "I am the initiator" branch, validate that `content.from` matches the authenticated peer ID of the actual sending session. Concretely, in the `None` arm, resolve the peer ID for `self.peer` from the peer registry and reject any message where `content.from == self_peer_id` but the sender's authenticated peer ID does not also equal `self_peer_id`:

```rust
// In the None arm, before the self_peer_id comparison:
let sender_peer_id = self.protocol.network_state
    .peer_registry.read()
    .get_key_by_peer_id_or_session(self.peer); // resolve PeerIndex → PeerId

if content.from == *self_peer_id && sender_peer_id.as_ref() != Some(self_peer_id) {
    return StatusCode::Ignore.with_context("spoofed from field");
}
```

Alternatively, enforce globally that `content.from` always equals the authenticated peer ID of the sending session for the entire message, not just in this branch.

## Proof of Concept
```rust
// Setup: victim has an inflight entry for target_peer_id
protocol.inflight_requests.insert(target_peer_id.clone(), unix_time_as_millis());

// Craft spoofed message:
//   from  = victim's own local_peer_id
//   to    = target_peer_id
//   route = []  (empty)
//   listen_addrs = [attacker_ip:port]
let spoofed_msg = build_connection_request_delivered(
    /*from=*/        victim_local_peer_id.clone(),
    /*to=*/          target_peer_id.clone(),
    /*route=*/       vec![],
    /*listen_addrs=*/vec![attacker_addr_without_peer_id],
);

// Deliver from attacker's session
let status = ConnectionRequestDeliveredProcess::new(
    spoofed_msg.as_reader(), &mut protocol, &control,
    attacker_session_id, bind_addr, item_id,
).execute().await;

// Assert: inflight entry consumed
assert!(protocol.inflight_requests.is_empty());

// Deliver the legitimate ConnectionRequestDelivered for the same target
let legit_status = /* same target_peer_id */ .execute().await;
// Returns Ignore — "the request is not in flight"
assert_eq!(legit_status, StatusCode::Ignore);
```
