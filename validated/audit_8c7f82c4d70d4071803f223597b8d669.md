### Title
Missing Weight-Based Minimum Fee Enforcement in `check_tx_fee` Allows High-Cycles Transactions to Underpay Fees - (File: tx-pool/src/util.rs)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using only the transaction's serialized byte size (`tx_size`), not its actual **weight** (`get_transaction_weight(tx_size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). A transaction with maximum cycles but minimal serialized size passes the fee gate paying a fee that is up to ~119× lower than the minimum fee rate actually requires when measured against the resource it consumes. No second weight-based check is performed after verification when the real cycle count is known.

---

### Finding Description

`check_tx_fee` is the sole fee-rate gate for tx-pool admission:

```rust
// tx-pool/src/util.rs  lines 28-53
pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(...).transaction_fee(rtx)?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);  // ← size only
    if fee < min_fee {
        return Err(Reject::LowFeeRate(...));
    }
    Ok(fee)
}
``` [1](#0-0) 

The actual transaction weight used everywhere else (sorting, eviction, block selection, fee estimation) is:

```rust
// util/types/src/core/tx_pool.rs  lines 298-303
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`, so at `max_tx_verify_cycles = 70_000_000`:

```
weight = max(size, 70_000_000 × 0.000_170_571_4) = max(size, 11_940)
``` [3](#0-2) 

For a transaction with `size = 100 bytes` and `cycles = 70_000_000`:

| Metric | Value |
|---|---|
| `min_fee` (size-based, what the gate checks) | `1000 × 100 / 1000 = 100 shannons` |
| `min_fee` (weight-based, what it should check) | `1000 × 11_940 / 1000 = 11_940 shannons` |
| Ratio | **~119×** |

The transaction is admitted with 100 shannons fee. Its actual fee rate by weight is `100 × 1000 / 11_940 ≈ 8 shannons/KW`, far below the configured `min_fee_rate = 1000 shannons/KW`.

The admission path is `pre_check → check_tx_fee (size-based) → verify_rtx (cycles determined) → TxEntry::new(rtx, verified.cycles, fee, tx_size)`. No weight-based fee check is performed after `verified.cycles` is known. [4](#0-3) 

`TxEntry::fee_rate()` correctly uses weight, but this is only used for sorting/eviction, not for admission gating:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [5](#0-4) 

---

### Impact Explanation

An unprivileged tx-pool submitter can craft transactions with near-maximum cycles but minimal serialized size to:

1. **Force expensive script verification at below-minimum cost.** The node runs `verify_rtx` consuming up to `max_tx_verify_cycles = 70_000_000` CKB-VM cycles per transaction while the submitter pays only the size-based minimum fee (e.g., 100 shannons instead of 11,940 shannons). This is a sustained, cheap CPU-exhaustion vector against any node with `min_fee_rate > 0`.

2. **Admit transactions to the pool at a fee rate far below `min_fee_rate`.** The admitted entry has `fee_rate ≈ 8 shannons/KW` while the operator configured `min_fee_rate = 1000 shannons/KW`. The pool's stated invariant — "txs with lower fee rate than this will not be relayed or be mined" — is violated at admission. [6](#0-5) 

3. **Degrade fee estimation accuracy.** The admitted low-fee-rate entries pollute the pool used by `estimate_fee_rate`, causing the node to recommend artificially low fees to users. [7](#0-6) 

---

### Likelihood Explanation

The entry path is the public `send_transaction` RPC, reachable by any unprivileged actor. The attacker needs only a valid UTXO and a fee of `min_fee_rate × size / 1000` shannons — a trivially small amount. The attack can be repeated continuously. The default `min_fee_rate = 1000` and `max_tx_verify_cycles = 70_000_000` are set in the shipped `ckb.toml`. [8](#0-7) 

---

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee check using the real weight before calling `submit_entry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Alternatively, if a pre-verification check is required, use `declared_cycles` (already accepted by the relay protocol) as a conservative upper bound for the weight calculation in `check_tx_fee`.

---

### Proof of Concept

1. Obtain a spendable cell with capacity `C`.
2. Craft a transaction: `size ≈ 100 bytes`, lock script that consumes `~70_000_000` cycles, output capacity `C - 100` (fee = 100 shannons).
3. Submit via `send_transaction` RPC to a node with default config (`min_fee_rate = 1000`).
4. `check_tx_fee` computes `min_fee = 1000 × 100 / 1000 = 100 shannons`; `fee (100) >= min_fee (100)` → **admitted**.
5. `verify_rtx` runs the lock script consuming `~70_000_000` cycles.
6. `TxEntry` is created with `fee_rate = 100 × 1000 / 11_940 ≈ 8 shannons/KW` — 125× below `min_fee_rate`.
7. Repeat from step 2 with fresh UTXOs to continuously exhaust node verification CPU at ~1% of the intended minimum cost. [9](#0-8) [10](#0-9)

    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )

        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()

                            return Err(Reject::Resolve(OutPointError::Dead(out)));

                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
