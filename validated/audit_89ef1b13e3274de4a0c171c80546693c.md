### Title
Silently Discarded `add_tx` Return Value Causes Permanent Transaction Loss After RBF — (`tx-pool/src/process.rs`)

---

### Summary

Inside `TxPoolService::submit_entry`, when an RBF (Replace-By-Fee) replacement succeeds, the displaced-but-recoverable transactions are re-queued into the verify queue via `queue.add_tx(...)`. The `Result<bool, Reject>` return value is unconditionally discarded with `let _ = ...`. If the verify queue is full at that moment, `add_tx` returns `Err(Reject::Full(...))`, the error is silently swallowed, and the recovered transactions are permanently lost from the mempool with no log, no notification, and no fallback path.

---

### Finding Description

`VerifyQueue::add_tx` returns `Result<bool, Reject>`:

- `Ok(true)` — successfully enqueued
- `Ok(false)` — already present (duplicate)
- `Err(Reject::Full(...))` — queue size limit exceeded; transaction **not** enqueued [1](#0-0) 

Inside `submit_entry`, after a successful RBF replacement, `process_rbf` returns a list of transactions that were displaced but are no longer conflicting and should be re-verified. These are re-queued in a spawned task:

```rust
tokio::spawn(async move {
    let mut queue = self_clone.verify_queue.write().await;
    for tx in may_recovered_txs {
        debug!("recover back: {:?}", tx.proposal_short_id());
        let _ = queue.add_tx(tx, false, None);   // ← return value discarded
    }
});
``` [2](#0-1) 

The `let _ = queue.add_tx(tx, false, None)` pattern is the direct CKB analog of the Teller Finance `transferFrom()` whose boolean return was not checked. In both cases, a fallible operation's failure indicator is explicitly thrown away, and execution proceeds as if the operation succeeded.

The `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` cap is 256 MB: [3](#0-2) 

When the queue is at or near this limit, `is_full` returns true and `add_tx` returns `Err(Reject::Full(...))`: [4](#0-3) 

---

### Impact Explanation

Recovered transactions — those that were legitimately in the pool before being displaced by an RBF replacement — are permanently and silently evicted from the mempool. Their original submitters receive no error, no notification, and no re-broadcast. The transactions are not placed in the orphan pool, not retried, and not persisted. From the user's perspective, a previously-accepted transaction simply vanishes.

This corrupts the mempool's expected invariant: after a successful RBF, non-conflicting displaced transactions should be recoverable. When they are silently dropped, the effective mempool state diverges from what users and downstream systems expect.

---

### Likelihood Explanation

RBF must be enabled (it is a node-operator configuration). An attacker who can submit transactions to the network can:

1. Fill the verify queue to near the 256 MB limit by submitting many large-serialized transactions.
2. Submit an RBF replacement transaction that displaces several existing pool entries.
3. The spawned recovery task calls `add_tx` on each recovered transaction; all calls return `Err(Reject::Full(...))` and are discarded.
4. The displaced transactions are permanently lost.

This is reachable by any unprivileged tx-pool submitter (RPC caller or P2P peer) without any special privilege.

---

### Recommendation

Replace the `let _ = ...` discard with explicit error handling. At minimum, log the failure so operators are aware. Ideally, fall back to the orphan pool or a retry mechanism:

```rust
for tx in may_recovered_txs {
    debug!("recover back: {:?}", tx.proposal_short_id());
    if let Err(e) = queue.add_tx(tx.clone(), false, None) {
        warn!(
            "failed to re-enqueue recovered RBF tx {:#x}: {}",
            tx.hash(), e
        );
        // optionally: add to orphan pool or a retry list
    }
}
``` [2](#0-1) 

---

### Proof of Concept

1. Enable RBF on a CKB node (`ckb.toml`: `[tx_pool] min_rbf_rate = ...`).
2. Submit enough large transactions to bring `verify_queue.total_tx_size` close to 256 MB (can be verified via `tx_pool_info` RPC).
3. Submit transaction `T1` spending outpoint `O`.
4. Submit transaction `T2` (RBF replacement) spending `O` with a higher fee rate. `T2` is accepted; `T1` and any non-conflicting displaced transactions enter `may_recovered_txs`.
5. The spawned task calls `queue.add_tx(recovered_tx, false, None)` → returns `Err(Reject::Full(...))` → discarded.
6. Query `get_transaction` for the recovered tx hash → not found. The transaction has been permanently lost from the mempool with no error surfaced to the submitter. [5](#0-4) [6](#0-5)

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
```
