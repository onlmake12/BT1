### Title
Stale Price Data Persists After Subscription Deactivation and Reactivation — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler` contract stores per-subscription price data in a nested mapping `_state.priceUpdates[subscriptionId][priceId]` and a timestamp `priceLastUpdatedAt`. When a subscription is deactivated via `updateSubscription`, neither the stored price data nor `priceLastUpdatedAt` is cleared. Upon reactivation with the same price IDs, the old stale price data is immediately reused as the authoritative baseline for deviation checks and as the value returned to readers, mirroring the M-15 pattern exactly.

---

### Finding Description

`SchedulerState.sol` defines the state with a nested mapping for price data:

```solidity
mapping(uint256 => mapping(bytes32 => PythStructs.PriceFeed)) priceUpdates;
``` [1](#0-0) 

The `updateSubscription` function handles deactivation as follows:

```solidity
} else if (wasActive && !willBeActive) {
    currentParams.isActive = false;
    _removeFromActiveSubscriptions(subscriptionId);
    emit SubscriptionDeactivated(subscriptionId);
}
``` [2](#0-1) 

After this block, `_clearRemovedPriceUpdates` is called, but it only deletes entries for price IDs that are **removed** from the new params list — not for IDs that remain the same:

```solidity
bool newPriceIdsAdded = _clearRemovedPriceUpdates(
    subscriptionId,
    currentParams.priceIds,
    newParams.priceIds
);
if (newPriceIdsAdded) {
    _state.subscriptionStatuses[subscriptionId].priceLastUpdatedAt = 0;
}
``` [3](#0-2) 

If the subscription is deactivated and later reactivated with the **same** price IDs, `_clearRemovedPriceUpdates` finds every old ID in the new list and deletes nothing. `priceLastUpdatedAt` is also not reset because `newPriceIdsAdded` remains `false`. The old `priceUpdates` entries and the old `priceLastUpdatedAt` are silently reused. [4](#0-3) 

---

### Impact Explanation

**1. Stale prices immediately served to readers after reactivation.**
`getPricesUnsafe` and `getEmaPricesUnsafe` read directly from `_state.priceUpdates`, which still holds the pre-deactivation values. Any whitelisted reader or DeFi protocol consuming these prices receives arbitrarily old data the moment the subscription is reactivated. [5](#0-4) 

**2. Deviation-only subscriptions can be permanently blocked from updating.**
`_validateShouldUpdatePrices` computes deviation against the stored `previousFeed` from `_state.priceUpdates`. If the current market price has not moved beyond `deviationThresholdBps` relative to the stale pre-deactivation price, every keeper call reverts with `UpdateConditionsNotMet`, leaving the subscription serving stale prices indefinitely. [6](#0-5) 

**3. Heartbeat subscriptions have a delayed first update after reactivation.**
Because `priceLastUpdatedAt` is not reset, the heartbeat timer continues from the old timestamp. If the subscription was deactivated for less than `heartbeatSeconds`, the keeper cannot push a fresh update until the full interval elapses from the old timestamp, not from the reactivation time. [7](#0-6) 

---

### Likelihood Explanation

Any subscription manager (an unprivileged transaction sender) can trigger this by:
1. Calling `updateSubscription` to deactivate their subscription (e.g., to pause it or because balance ran low).
2. Later calling `updateSubscription` again to reactivate with the same price IDs.

This is a normal operational pattern. The manager may not realize that old price data persists. The scenario is especially likely for subscriptions that run out of balance, are deactivated, topped up, and reactivated — a common lifecycle. No privileged access, leaked key, or external oracle manipulation is required.

---

### Recommendation

When a subscription is deactivated, explicitly clear `priceLastUpdatedAt` and delete all stored `priceUpdates` entries for the subscription's current price IDs:

```solidity
} else if (wasActive && !willBeActive) {
    currentParams.isActive = false;
    _removeFromActiveSubscriptions(subscriptionId);
+   // Clear stale price data so it cannot be reused on reactivation
+   _state.subscriptionStatuses[subscriptionId].priceLastUpdatedAt = 0;
+   for (uint i = 0; i < currentParams.priceIds.length; i++) {
+       delete _state.priceUpdates[subscriptionId][currentParams.priceIds[i]];
+   }
    emit SubscriptionDeactivated(subscriptionId);
}
``` [8](#0-7) 

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` and create a subscription with price IDs `[A]`, deviation-only update criteria, and a threshold of 100 bps.
2. Call `updatePriceFeeds` to store an initial price of `1000` for feed `A`. `priceLastUpdatedAt` is set and `priceUpdates[subId][A].price = 1000`.
3. Call `updateSubscription` with `isActive = false` (same price IDs). Subscription is deactivated. `priceUpdates[subId][A]` and `priceLastUpdatedAt` are **not** cleared.
4. Call `updateSubscription` with `isActive = true` (same price IDs). Subscription is reactivated. `newPriceIdsAdded = false`, so `priceLastUpdatedAt` is not reset.
5. Attempt `updatePriceFeeds` with a current price of `1005` (0.5% deviation, below 100 bps threshold). The call reverts with `UpdateConditionsNotMet` because the deviation is computed against the stale `1000` baseline.
6. Call `getPricesUnsafe` — it returns the stale price `1000` even though the subscription is active and significant time may have passed. [9](#0-8) [10](#0-9)

    function updateSubscription(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams memory newParams
    ) external payable override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage currentStatus = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage currentParams = _state
            .subscriptionParams[subscriptionId];

        // Add incoming funds to balance
        currentStatus.balanceInWei += msg.value;

        // Updates to permanent subscriptions are not allowed
        if (currentParams.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        // If subscription is inactive and will remain inactive, no need to validate parameters
        bool wasActive = currentParams.isActive;
        bool willBeActive = newParams.isActive;
        if (!wasActive && !willBeActive) {
            // Update subscription parameters
            _state.subscriptionParams[subscriptionId] = newParams;
            emit SubscriptionUpdated(subscriptionId);
            return;
        }
        _validateSubscriptionParams(newParams);

        // Check minimum balance if subscription remains active
        if (willBeActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(newParams.priceIds.length)
            );
            if (currentStatus.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        // Handle activation/deactivation
        if (!wasActive && willBeActive) {
            // Reactivating a subscription - ensure minimum balance
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(newParams.priceIds.length)
            );

            // Check if balance meets minimum requirement
            if (currentStatus.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }

            currentParams.isActive = true;
            _addToActiveSubscriptions(subscriptionId);
            emit SubscriptionActivated(subscriptionId);
        } else if (wasActive && !willBeActive) {
            // Deactivating a subscription
            currentParams.isActive = false;
            _removeFromActiveSubscriptions(subscriptionId);
            emit SubscriptionDeactivated(subscriptionId);
        }

        // Clear price updates for removed price IDs before updating params
        bool newPriceIdsAdded = _clearRemovedPriceUpdates(
            subscriptionId,
            currentParams.priceIds,
            newParams.priceIds
        );

        // Reset priceLastUpdatedAt to 0 if new price IDs were added
        if (newPriceIdsAdded) {
            _state.subscriptionStatuses[subscriptionId].priceLastUpdatedAt = 0;
        }

        // Update subscription parameters
        _state.subscriptionParams[subscriptionId] = newParams;

        emit SubscriptionUpdated(subscriptionId);
    }

    function _clearRemovedPriceUpdates(
        uint256 subscriptionId,
        bytes32[] storage currentPriceIds,
        bytes32[] memory newPriceIds
    ) internal returns (bool newPriceIdsAdded) {
        // Iterate through old price IDs
        for (uint i = 0; i < currentPriceIds.length; i++) {
            bytes32 oldPriceId = currentPriceIds[i];
            bool found = false;

            // Check if the old price ID exists in the new list
            for (uint j = 0; j < newPriceIds.length; j++) {
                if (newPriceIds[j] == oldPriceId) {
                    found = true;
                    break; // Found it, no need to check further
                }
            }

            // If not found in the new list, delete its stored update data
            if (!found) {
                delete _state.priceUpdates[subscriptionId][oldPriceId];
            }
        }

        // Check if any new price IDs were added
        for (uint i = 0; i < newPriceIds.length; i++) {
            bytes32 newPriceId = newPriceIds[i];
            bool found = false;

            // Check if the new price ID exists in the current list
            for (uint j = 0; j < currentPriceIds.length; j++) {
                if (currentPriceIds[j] == newPriceId) {
                    found = true;
                    break;
                }
            }

            // If a new price ID was added, mark as changed
            if (!found) {
                newPriceIdsAdded = true;
                break;
            }
        }

        return newPriceIdsAdded;

                if (previousPrice == 0 || currentPrice == 0) {

                    10_000,
