### Title
Truncating `as u64` Cast in `calculate_maximum_withdraw` Skips Overflow Invariant Check — (`File: util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes an intermediate u128 value `withdraw_counted_capacity` and then narrows it to u64 with a bare `as u64` truncating cast. Every other analogous u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. The missing checked conversion means that if the product overflows u64, the function silently returns a wrong (too-small) capacity instead of propagating a `DaoError::Overflow`, violating the post-condition invariant that the returned withdrawal capacity is mathematically correct.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the DAO withdrawal amount as:

```
withdraw_counted_capacity = counted_capacity * withdrawing_ar / deposit_ar   (u128)
withdraw_capacity          = withdraw_counted_capacity as u64  +  occupied_capacity
``` [1](#0-0) 

The `as u64` cast on line 156 is a **truncating** (wrapping) narrowing. If `withdraw_counted_capacity` exceeds `u64::MAX`, the lower 64 bits are silently kept, producing a value that is far smaller than the true result. The subsequent `safe_add(occupied_capacity)` only catches overflow of the *already-truncated* value; it cannot detect that the truncation already occurred.

Every other u128→u64 narrowing in the same file uses the checked pattern:

```rust
// secondary_block_reward (line 204)
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch (line 245)
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)

// dao_field_with_current_epoch (line 258)
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

`calculate_maximum_withdraw` is the sole exception.

The function is called from two paths:

1. **`withdrawed_interests` → `dao_field_with_current_epoch`**: the result feeds directly into the `current_s` (DAO savings) field written into every block header. [5](#0-4) 

2. **`transaction_fee`**: used to compute the fee charged for a DAO withdrawal transaction. [6](#0-5) 

3. **RPC `calculate_dao_maximum_withdraw`**: informational, but callers rely on the returned value to construct valid withdrawal transactions. [7](#0-6) 

### Impact Explanation

If `withdraw_counted_capacity` silently wraps, `calculate_maximum_withdraw` returns a value that is `2^64` shannons smaller than the correct answer. Downstream effects:

- **`withdrawed_interests`** returns a too-small interest figure, causing `current_s` in the DAO field to be inflated (DAO savings pool appears larger than it is). Because all nodes share the same bug, the inflated `current_s` is accepted by consensus, permanently corrupting the on-chain DAO accounting state.
- **`transaction_fee`** returns a too-large fee (since `maximum_withdraw` is too small, `maximum_withdraw - outputs_capacity` underflows or is wrong), potentially causing valid DAO withdrawal transactions to be incorrectly rejected from the tx-pool.
- **RPC callers** receive a wrong maximum-withdraw figure and may construct transactions that are rejected on-chain. [8](#0-7) 

### Likelihood Explanation

The overflow condition requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

`counted_capacity` is at most `u64::MAX` shannons. `ar` starts at `10_000_000_000_000_000` (10^16) and grows by roughly `ar × g2 / C` per block — on the order of a few thousand units per block. For `withdrawing_ar / deposit_ar` to reach 2 (the minimum ratio that causes overflow when `counted_capacity ≈ u64::MAX`) would require on the order of 10^13 blocks, far beyond any realistic chain lifetime.

Likelihood is therefore **very low** under normal CKB consensus parameters. However, the defect is a real, demonstrable code inconsistency: the invariant `withdraw_counted_capacity ≤ u64::MAX` is never asserted, unlike every analogous computation in the same file. The existing test `check_withdraw_calculation_overflows` only exercises overflow caught by the later `safe_add`, not by the truncating cast itself. [9](#0-8) 

### Recommendation

Replace the truncating cast with the same checked pattern used everywhere else in the file:

```rust
// Before (line 155-156):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After:
let withdraw_counted_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_u64).safe_add(occupied_capacity)?;
``` [10](#0-9) 

Add a test that constructs a scenario where `counted_capacity * withdrawing_ar / deposit_ar` exceeds `u64::MAX` and asserts the function returns `DaoError::Overflow` rather than a silently truncated value.

### Proof of Concept

```
counted_capacity  = u64::MAX = 18_446_744_073_709_551_615  shannons
deposit_ar        = 10_000_000_000_000_000
withdrawing_ar    = 20_000_000_000_000_001   (ar has doubled + 1)

withdraw_counted_capacity (u128)
  = 18_446_744_073_709_551_615 × 20_000_000_000_000_001
    / 10_000_000_000_000_000
  ≈ 36_893_488_147_419_103_232   (> u64::MAX = 18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 36_893_488_147_419_103_232 mod 2^64
  = 36_893_488_147_419_103_232 - 18_446_744_073_709_551_616
  = 18_446_744_073_709_551_616   (wrong, ~half the correct value)

safe_add(occupied_capacity) succeeds silently → wrong capacity returned
```

The function returns approximately half the correct withdrawal amount with no error, instead of `DaoError::Overflow`. [11](#0-10)

#[test]
fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
}
