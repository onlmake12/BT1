### Title
Unauthenticated `remove_transaction` / `clear_tx_pool` RPC Allows Any Caller to Arbitrarily Drain the Mempool — (`File: rpc/src/module/pool.rs`)

---

### Summary

The CKB JSON-RPC server exposes destructive Pool module operations — `remove_transaction` and `clear_tx_pool` — with no caller authentication or authorization check of any kind. The RPC server itself has no authentication middleware. Any entity that can reach the RPC socket (any local process by default; any remote host when the operator binds to `0.0.0.0`) can silently remove individual transactions or wipe the entire mempool without possessing any credential, key, or privileged role.

---

### Finding Description

**Root cause 1 — No authentication in the RPC server**

`rpc/src/server.rs` builds the Axum router with `CorsLayer::permissive()` and no authentication layer. Every incoming HTTP/TCP/WS connection is dispatched directly to the method handler with no token, API key, or IP-allowlist check enforced at the transport level. [1](#0-0) 

The `RpcConfig` struct has no authentication fields at all — there is no provision for a bearer token, HMAC secret, or per-method IP allowlist. [2](#0-1) 

**Root cause 2 — `remove_transaction` performs no authorization check**

`PoolRpcImpl::remove_transaction` immediately forwards the caller-supplied hash to `tx_pool.remove_local_tx` with zero verification of who is calling. [3](#0-2) 

**Root cause 3 — `clear_tx_pool` performs no authorization check**

`PoolRpcImpl::clear_tx_pool` wipes the entire mempool with no authorization check. [4](#0-3) 

The Pool module is enabled by default in the production configuration. [5](#0-4) 

---

### Impact Explanation

An unauthorized caller can:

1. **Silently remove any specific pending transaction** from the mempool by hash via `remove_transaction`, enabling targeted transaction censorship — a submitted transaction disappears from the pool without error to the original submitter.
2. **Wipe the entire mempool** via `clear_tx_pool`, causing all pending transactions to be dropped and forcing every user to resubmit, disrupting normal operation.
3. **Repeatedly drain the pool** to prevent any transaction from ever being confirmed, constituting a sustained denial-of-service against transaction processing.

The `remove_transaction` RPC removes the target transaction *and all descendant transactions that depend on it*, amplifying the impact of a single call. [6](#0-5) 

---

### Likelihood Explanation

**Scenario A (default config — local attacker):** The default bind address is `127.0.0.1:8114`. Any process running on the same host as the CKB node — malware, a compromised co-located service, a malicious script — can call these methods with a plain HTTP POST. No credential is required.

**Scenario B (misconfigured node — remote attacker):** The documentation warns against binding to `0.0.0.0` but does not prevent it. Nodes exposed to the internet (e.g., cloud VMs, misconfigured deployments) are fully reachable by any remote attacker. The warning in the config template is advisory only. [7](#0-6) 

Both scenarios require no privileged key, no social engineering, and no majority hashpower.

---

### Recommendation

1. **Add an authentication layer to the RPC server** — introduce a configurable bearer token or HMAC secret that must be presented in the `Authorization` header for all state-mutating RPC calls. Reject unauthenticated requests at the transport layer before they reach any handler.
2. **Classify RPC methods by privilege tier** — read-only queries (`get_block`, `get_peers`, etc.) may remain open; destructive operations (`remove_transaction`, `clear_tx_pool`, `set_ban`, `add_node`, `remove_node`) must require authentication.
3. **Add an IP allowlist field to `RpcConfig`** — enforce it in the server middleware so that even if a token is leaked, only trusted source IPs can invoke privileged methods.

---

### Proof of Concept

```bash
# Wipe the entire mempool of a CKB node (default local config):
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}'
# Returns: {"jsonrpc":"2.0","result":null,"id":1}  — no credential required

# Remove a specific transaction by hash:
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction",
       "params":["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"],
       "id":2}'
# Returns: {"jsonrpc":"2.0","result":true,"id":2}  — no credential required
```
