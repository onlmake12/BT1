### Title
Silent u128ŌåÆu64 Truncation in `calculate_maximum_withdraw` Produces Wrong DAO Withdrawal Amounts ŌĆö (`File: util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` uses a bare `as u64` cast to narrow a u128 intermediate result, silently truncating instead of returning an error. Every other u128ŌåÆu64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. The inconsistency means that under specific (but theoretically reachable) conditions the function returns a silently wrong, too-small withdrawal capacity, causing the user's DAO withdrawal transaction to be rejected by the on-chain DAO type script with no actionable error, permanently blocking the withdrawal.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ŌåÉ silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast in Rust silently wraps (truncates the high bits) when the value exceeds `u64::MAX`. No `DaoError::Overflow` is ever raised by this path.

Every other u128ŌåÆu64 narrowing in the same file uses the safe pattern:

```rust
// secondary_block_reward (line 204)
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch (lines 245, 258)
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) 

The overflow condition is:

```
counted_capacity ├Ś withdrawing_ar / deposit_ar  >  u64::MAX
```

Since `withdrawing_ar Ōēź deposit_ar` always holds (the accumulation rate only grows), this requires `counted_capacity` to be large enough that the ratio pushes the product past `u64::MAX`. The total CKB supply is ~3.36 ├Ś 10┬╣ŌüĖ shannons; `u64::MAX` is ~1.84 ├Ś 10┬╣Ōü╣. A cell holding a large fraction of the supply combined with a sufficiently grown AR ratio (e.g., after many years of secondary issuance) can satisfy the condition.

When truncation occurs, two outcomes are possible:

- **Silent wrong result (most dangerous):** The truncated value plus `occupied_capacity` does not overflow u64, so `safe_add` succeeds and the function returns a silently incorrect, too-small capacity. The RPC `calculate_dao_maximum_withdraw` returns this wrong value. The user builds a withdrawal transaction with the wrong output capacity. The on-chain DAO type script rejects it. The user has no way to determine the correct amount or why the transaction fails.

- **Misleading overflow error:** The truncated value plus `occupied_capacity` happens to overflow u64, so `safe_add` returns `Err(Overflow)`. The RPC surfaces `CKBInternalError: Mathematics overflow` with no guidance on how to proceed. [4](#0-3) 

The function is called from two attacker-reachable paths:

1. **RPC path:** `ExperimentRpcImpl::calculate_dao_maximum_withdraw` ŌåÆ `DaoCalculator::calculate_maximum_withdraw` [5](#0-4) 

2. **Tx-pool path:** `check_tx_fee` ŌåÆ `DaoCalculator::transaction_fee` ŌåÆ `transaction_maximum_withdraw` ŌåÆ `calculate_maximum_withdraw` [6](#0-5) 

The DAO withdrawal flow in integration tests confirms that `calculate_dao_maximum_withdraw` is the authoritative source for the output capacity placed in the withdrawal transaction: [7](#0-6) 

### Impact Explanation

A DAO depositor with a sufficiently large cell (or after the AR has grown enough) calls `calculate_dao_maximum_withdraw` to determine the correct output capacity for their withdrawal transaction. The RPC returns a silently wrong (too-small) value. The user constructs and submits the withdrawal transaction. The on-chain DAO type script, which independently computes the correct expected capacity, rejects the transaction. The user cannot withdraw their DAO deposit. There is no error message from the RPC indicating the calculation was wrong, and no documented procedure for recovering from this state. The funds remain locked until the user can determine the correct capacity by other means.

### Likelihood Explanation

The condition requires `counted_capacity ├Ś withdrawing_ar / deposit_ar > u64::MAX`. Given the current CKB supply and AR growth rate this is not immediately triggerable on mainnet, but it is a deterministic arithmetic defect ŌĆö not a probabilistic one ŌĆö and the threshold will be approached as the chain ages and secondary issuance accumulates. The inconsistency with the three correct `u64::try_from` calls in the same file confirms this is an unintentional omission rather than a deliberate design choice.

### Recommendation

**Short term:** Replace the silent cast with the same checked conversion used everywhere else in the file:

```rust
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
```

**Long term:** Add a property-based or fuzz test (analogous to the existing `check_withdraw_calculation_overflows` test) that exercises `calculate_maximum_withdraw` with `counted_capacity` values near `u64::MAX` and verifies that the function either returns `DaoError::Overflow` or a correct result ŌĆö never a silently truncated one.

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` already constructs a cell with `capacity = 18_446_744_073_709_550_000` (close to `u64::MAX`) and expects `result.is_err()`. [8](#0-7) 

That test passes today only because the final `safe_add(occupied_capacity)` happens to overflow u64 in that specific case. A slightly different set of parameters ŌĆö where the truncated u64 value plus `occupied_capacity` does not overflow ŌĆö would cause the test to pass with a silently wrong return value. Concretely:

```
deposit_ar      = 10_000_000_000_000_000   (initial AR)
withdrawing_ar  = 20_000_000_000_000_000   (AR doubled after many years)
output_capacity = 10_000_000_000_000_000   (10^16 shannons, ~100M CKB)
occupied_capacity Ōēł 6_100_000_000          (minimal cell)

counted_capacity = 10_000_000_000_000_000 - 6_100_000_000
                 Ōēł 9_999_999_993_900_000_000

withdraw_counted_capacity (u128)
  = 9_999_999_993_900_000_000 ├Ś 20_000_000_000_000_000
    / 10_000_000_000_000_000
  = 19_999_999_987_800_000_000   -- exceeds u64::MAX (18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 19_999_999_987_800_000_000 mod 2^64
  = 1_553_255_914_090_448_384   -- silently truncated, no error

withdraw_capacity = 1_553_255_914_090_448_384 + 6_100_000_000
                  = 1_553_255_920_190_448_384   -- wrong, no overflow, no error
```

The function returns `Ok(Capacity::shannons(1_553_255_920_190_448_384))` instead of `Err(DaoError::Overflow)`. The user's withdrawal transaction is built with this wrong capacity and rejected by the DAO type script.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
