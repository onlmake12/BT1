Looking at the code carefully, I need to trace the exact data flow through the rate limiters and determine whether the `forward_rate_limiter` map can grow without bound.

### Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Attacker-Controlled `(from, to)` PeerId Pairs — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

The `HolePunching` protocol maintains a `forward_rate_limiter` backed by `governor`'s `HashMapStateStore`, keyed by attacker-controlled `(PeerId, PeerId, u32)` tuples extracted from `ConnectionRequest` message payloads. The only cleanup call — `retain_recent()` — is invoked exclusively in `disconnected()`. A single persistent TCP session can insert up to 30 new entries per second into this map indefinitely, causing unbounded heap growth that can OOM-crash the node.

---

### Finding Description

**Data structures involved:**

`HolePunching` holds two rate limiters: [1](#0-0) 

- `rate_limiter`: keyed by `(PeerIndex, u32)` — i.e., `(session_id, msg_item_id)`, quota 30/second. This is the outer guard.
- `forward_rate_limiter`: keyed by `(PeerId, PeerId, u32)` — i.e., `(from, to, msg_item_id)`, quota 1/second. Both `from` and `to` come directly from the attacker's message payload.

**Cleanup is only in `disconnected()`:** [2](#0-1) 

`retain_recent()` is never called during an active session — not in `received()`, not in `notify()`.

**The outer rate limiter does NOT bound the map size.** It checks `(session_id, msg_item_id)`: [3](#0-2) 

This allows 30 `ConnectionRequest` messages per second from one session, regardless of the `from`/`to` content. Each of those 30 messages can carry a fresh unique `(from, to)` pair.

**Each unique `(from, to)` pair inserts a new entry into `forward_rate_limiter`:** [4](#0-3) 

The `governor` crate's `HashMapStateStore` has no automatic eviction. Every `check_key()` call with a previously-unseen key allocates a new map entry. Without `retain_recent()` being called, these entries accumulate for the entire lifetime of the session.

---

### Impact Explanation

- **Growth rate**: 30 entries/second per session (bounded by outer rate limiter).
- **Entry size**: `(PeerId, PeerId, u32)` key ≈ 82 bytes + governor atomic state ≈ 8 bytes ≈ ~100 bytes/entry.
- **24-hour session**: 30 × 86,400 = ~2.6M entries ≈ **~260 MB** from a single connection.
- **Multiple sessions**: growth scales linearly with the number of concurrent attacker-controlled sessions.
- **Result**: unbounded heap growth → OOM crash of the CKB node → network-wide disruption for peers relying on this node.

---

### Likelihood Explanation

The attack requires only a standard P2P TCP connection — no privileges, no PoW, no key material. The attacker sends valid, well-formed `ConnectionRequest` messages with randomly generated `from`/`to` PeerIds (both fields are parsed but not validated against any connected-peer registry before the `forward_rate_limiter` check). The outer rate limiter slows but does not stop the attack. A node left running for days under this attack will exhaust available memory.

---

### Recommendation

1. **Call `retain_recent()` periodically**, not only on disconnect. The existing `notify()` timer fires every 5 minutes (`CHECK_INTERVAL`) and already performs cleanup of `pending_delivered` and `inflight_requests`; add `retain_recent()` calls for both rate limiters there: [5](#0-4) 

2. **Cap the `forward_rate_limiter` map size** by switching to a bounded LRU-keyed structure, or by enforcing a hard limit on the number of tracked `(from, to)` pairs per session.

3. **Validate `from` against the sending session's known PeerId** before inserting into `forward_rate_limiter`, so an attacker cannot use arbitrary fabricated PeerIds as map keys.

---

### Proof of Concept

```rust
// Pseudocode unit test
let mut protocol = HolePunching::new(network_state);
for i in 0..N {
    let from = PeerId::random();
    let to   = PeerId::random();
    let msg  = build_connection_request(from, to, valid_listen_addrs);
    protocol.received(ctx, msg).await;
    // No disconnect between iterations
}
// Assert: forward_rate_limiter internal HashMap has ~N entries
// (retain_recent() was never called; each unique (from,to) pair is a new key)
```

Each iteration passes the outer `rate_limiter` (same `session_id`/`msg_item_id`, 30/s budget) and inserts a fresh entry into `forward_rate_limiter`. After `N` iterations without a `disconnected()` call, the map holds `N` entries, growing proportionally to `N`.
