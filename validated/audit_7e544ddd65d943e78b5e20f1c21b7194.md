### Title
ETH Sent to `Executor` Contract Is Permanently Locked — No Withdrawal Mechanism - (File: `target_chains/ethereum/contracts/contracts/executor/Executor.sol`)

---

### Summary
`Executor.sol` exposes a `receive() external payable {}` function and a `payable` `execute()` entry point, but contains no mechanism to withdraw ETH from the contract. Any ETH deposited — either directly via `receive()` or as excess `msg.value` over `gi.value` in a successful `execute()` call — is permanently locked.

---

### Finding Description
`Executor.sol` accepts ETH in two ways:

1. **Direct ETH transfer via `receive()`** (line 113):
   ```solidity
   receive() external payable {}
   ```
   This is a completely open, unprivileged entry point. Any address can send ETH directly to the contract and it will be silently accepted.

2. **Excess `msg.value` in `execute()`** (lines 66–110):
   ```solidity
   function execute(bytes memory encodedVm) public payable returns (bytes memory response)
   ```
   The function forwards exactly `gi.value` (decoded from the governance VAA) to `callAddress`:
   ```solidity
   (success, response) = address(callAddress).call{value: gi.value}(gi.callData);
   ```
   If a caller sends `msg.value > gi.value`, the difference (`msg.value - gi.value`) remains in the contract with no path to recover it.

Neither `Executor.sol` nor `ExecutorUpgradable.sol` contains any `withdraw`, `rescue`, or ETH-recovery function. A search for `withdraw` in both files returns zero matches.

---

### Impact Explanation
Any ETH sent to the `Executor` contract — whether by a user accidentally sending ETH to the contract address, or by a caller of `execute()` who over-funds the call — is permanently locked. There is no admin, governance, or owner function to recover it. The contract holds no accounting of "free" ETH balance, so even a governance-crafted VAA cannot instruct the contract to send its own balance anywhere (the `execute()` path calls an *external* `callAddress`, not `address(this)`).

**Impact:** Permanent loss of ETH for any sender. Medium severity — matches the reference report's classification.

---

### Likelihood Explanation
The `receive()` function is a completely open, zero-permission entry point reachable by any EOA or contract. Accidental ETH transfers to contract addresses are common (e.g., mistaken direct sends, contract interactions that forward ETH). Additionally, any caller of `execute()` who attaches more ETH than the governance instruction specifies in `gi.value` will silently lose the difference. Both paths require no special role or privilege.

---

### Recommendation
Either:
- **Remove `receive()`** if the contract has no legitimate reason to accept direct ETH transfers, and remove the `payable` modifier from `execute()` if callers should never send more ETH than `gi.value`; or
- **Add an ETH withdrawal function** restricted to the governance/owner, e.g.:
  ```solidity
  function withdrawEth(address payable recipient, uint256 amount) external onlyGovernance {
      (bool sent, ) = recipient.call{value: amount}("");
      require(sent, "ETH transfer failed");
  }
  ```

---

### Proof of Concept

1. Deploy `ExecutorUpgradable` (the production proxy wrapping `Executor`).
2. Call `address(executor).call{value: 1 ether}("")` from any EOA — the `receive()` hook accepts it silently.
3. Observe `address(executor).balance == 1 ether`.
4. Attempt to recover: there is no function in `Executor.sol` or `ExecutorUpgradable.sol` that transfers ETH out of the contract.
5. The 1 ETH is permanently locked.
