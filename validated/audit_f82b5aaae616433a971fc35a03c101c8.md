### Title
Stale `ancestors_size`/`ancestors_cycles` in `TxSelector::txs_to_commit` Causes Valid Transactions to Be Incorrectly Excluded from Block Templates — (`tx-pool/src/component/tx_selector.rs`)

---

### Summary

In `TxSelector::txs_to_commit`, when a transaction package is committed to the block template, `update_modified_entries` is called to propagate the weight reduction to descendants via `sub_ancestor_weight`. However, due to an acknowledged inconsistency in `calc_descendants()`, some descendants may not be added to `modified_entries` and thus never have their `ancestors_size`/`ancestors_cycles` decremented. When those descendants are later evaluated from the `sorted_proposed_iter()` iterator, the limit check uses stale (over-inflated) ancestor weight values, double-counting already-committed ancestors and incorrectly rejecting transactions that would actually fit within the block's size and cycles limits.

---

### Finding Description

The `txs_to_commit` function in `TxSelector` selects transactions for block templates. It maintains two running accumulators — `size` and `cycles` — that track the **individual** sizes/cycles of transactions already committed to the template. Each `TxEntry` also carries `ancestors_size`/`ancestors_cycles`, which represent the **cumulative** size/cycles of the transaction plus all its unconfirmed ancestors.

**Phase 1 — Commit ancestors:** When a package is selected, each new ancestor is added to `fetched_txs` and its individual `entry.size`/`entry.cycles` is added to the running `size`/`cycles` accumulators: [1](#0-0) 

**Phase 2 — Update descendants:** `update_modified_entries` is called to reduce `ancestors_size`/`ancestors_cycles` on all descendants of the just-committed transactions, so that future limit checks remain accurate: [2](#0-1) 

**The bug:** `calc_descendants()` is acknowledged to be potentially inconsistent: [3](#0-2) 

When `calc_descendants()` misses a descendant, that descendant is never added to `modified_entries` and never has `sub_ancestor_weight` called on it. It remains in the `sorted_proposed_iter()` pool with stale `ancestors_size`/`ancestors_cycles` that still include the already-committed ancestor's weight.

**Phase 2 — Limit check (incorrect):** When the stale descendant is later evaluated from the pool iterator, the limit check computes: [4](#0-3) 

Here `size` already includes the committed ancestor's individual size, and `tx_entry.ancestors_size` (stale) also still includes that ancestor's size. The result is double-counting: `next_size = size + ancestors_size = (committed_ancestor.size + ...) + (committed_ancestor.size + descendant.size + ...) = 2 × committed_ancestor.size + ...`. This inflated `next_size` may exceed `size_limit` even when the descendant would actually fit.

Additionally, the pre-filter applied to the pool iterator also uses the stale value: [5](#0-4) 

A descendant with stale `ancestors_size` may be filtered out here entirely, never reaching the limit check at all.

The `TxEntry` ancestor weight fields are initialized to include the transaction itself: [6](#0-5) 

And `sub_ancestor_weight` is the correct mechanism to reduce them when ancestors are committed: [7](#0-6) 

But this is only called for descendants that `calc_descendants()` returns — descendants it misses retain permanently stale values for the duration of the block template assembly loop.

---

### Impact Explanation

Valid proposed transactions are incorrectly excluded from block templates. Miners assemble blocks with fewer transactions than the size/cycles limits would allow, losing fee revenue. Transaction senders whose transactions are affected experience delayed confirmation even though their transactions are valid and proposed. The block assembler calls `package_txs` which delegates to `TxSelector`: [8](#0-7) 

This is called from both `update_full` and `update_transactions` in the block assembler: [9](#0-8) 

---

### Likelihood Explanation

The inconsistency in `calc_descendants()` is explicitly acknowledged in a code comment referencing a specific merged PR (`#3706`), indicating this is a known, reproducible condition — not a theoretical edge case. Any transaction sender can submit a transaction graph (e.g., involving cell-dep relationships between pool transactions) that triggers the inconsistency. When the pool is near its size/cycles limit, the double-counting is most likely to cause incorrect rejection.

---

### Recommendation

1. Fix `calc_descendants()` to be fully consistent, or document and enforce the invariant that it must be complete for correctness of `TxSelector`.
2. As a defensive measure, before applying the limit check in `txs_to_commit`, verify that the ancestor IDs returned by `calc_ancestors` are all either already in `fetched_txs` or will be newly added, and compute `next_size`/`next_cycles` from scratch rather than relying on potentially stale `ancestors_size`/`ancestors_cycles` for entries sourced from the pool iterator (not from `modified_entries`).
3. Alternatively, after committing a package, re-scan all remaining pool entries whose ancestors overlap with the committed set and force-update their ancestor weights.

---

### Proof of Concept

1. Submit tx `A` (parent) and tx `B` (child of `A`) to the proposed pool, where `B`'s `ancestors_size = A.size + B.size`.
2. Arrange the transaction graph so that `calc_descendants(A)` does not return `B` (e.g., via a cell-dep relationship that PR #3706 made inconsistent).
3. Run `package_txs` with a `size_limit` such that `A.size + B.size ≤ size_limit < 2 × A.size + B.size`.
4. Observe: `A` is committed first (`size += A.size`). `update_modified_entries` misses `B`. When `B` is evaluated from the pool iterator, `next_size = A.size + (A.size + B.size) = 2×A.size + B.size > size_limit`, so `B` is rejected — even though `A.size + B.size ≤ size_limit` and `B` would fit.
