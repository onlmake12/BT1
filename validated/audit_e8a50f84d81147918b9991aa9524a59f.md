Audit Report

## Title
Unconditional `.expect()` on `txs_sizes` in `FeeRateCollector::statistics` Panics on Pre-v0.106 `BlockExt` Records — (`rpc/src/util/fee_rate.rs`)

## Summary

`FeeRateCollector::statistics` calls `.expect()` unconditionally on `block_ext.txs_sizes` for every `BlockExt` returned by the fee-rate window iterator. Nodes upgraded in-place from a version prior to v0.106 retain historical blocks stored in the old 5-field `BlockExt` schema, which deserialises with `txs_sizes: None`. Any caller invoking `get_fee_rate_statistics` with a window that overlaps those blocks triggers a guaranteed panic at `rpc/src/util/fee_rate.rs:93`.

## Finding Description

**Root cause — the `.expect()` call:**

In `rpc/src/util/fee_rate.rs`, the closure passed to `collect` destructures every `BlockExt` and immediately panics if `txs_sizes` is absent:

```rust
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
``` [1](#0-0) 

There is no guard, no `if let Some(...)`, and no early-return before this line.

**How `txs_sizes` becomes `None` in storage:**

`get_block_ext` in `store/src/store.rs` reads the raw bytes and branches on the field count. When `count_extra_fields() == 0` (the pre-v0.106 five-field `BlockExt` schema), it converts via `packed::BlockExtReader::into()`: [2](#0-1) 

That conversion path explicitly sets both new fields to `None`: [3](#0-2) 

**Why `filter_map` does not protect against this:**

`collect` uses `filter_map` only to skip blocks whose hash or ext record is entirely absent from storage. A block stored in the old format is still returned as `Some(BlockExt { txs_sizes: None, … })` and is passed directly into the panicking closure: [4](#0-3) 

## Impact Explanation

A panic in the RPC handler crashes the `get_fee_rate_statistics` request. This is a reliable, repeatable denial-of-service against that RPC endpoint on any upgraded mainnet node carrying pre-v0.106 data. The impact is a **local RPC API crash** (Note, 0–500 points).

## Likelihood Explanation

Any mainnet node that was running before v0.106 and upgraded in-place retains old-format `BlockExt` records for all blocks committed before the upgrade. The maximum window is 101 blocks (`MAX_TARGET`), so on a node with even a single pre-migration block in the last 101 blocks the panic is guaranteed. The RPC requires no authentication, making this trivially triggerable by any caller. [5](#0-4) 

## Recommendation

Replace the `.expect()` with a graceful skip:

```rust
let Some(txs_sizes) = txs_sizes else {
    return fee_rates;
};
```

This matches the existing guard `if txs_sizes.len() > 1 && !txs_fees.is_empty()` that already handles the case where there is nothing useful to compute. [6](#0-5) 

## Proof of Concept

1. Start a node that was previously running pre-v0.106 (or manually insert a `BlockExt` record with the old 5-field schema for any block in the last 101 blocks).
2. Call `get_fee_rate_statistics` with `target = 101` (or any value whose window overlaps the old-format block).
3. The node panics with `"expect txs_size's length >= 1"` at `rpc/src/util/fee_rate.rs:93`.
