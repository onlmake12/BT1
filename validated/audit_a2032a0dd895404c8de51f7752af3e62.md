### Title
`PythAggregatorV3.latestRoundData()` Returns Stale or Negative Price Without Reverting - (`target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

### Summary

`PythAggregatorV3` is Pyth's official Chainlink-compatible adapter, explicitly recommended for protocols migrating from Chainlink. Its `latestRoundData()` function calls `getPriceUnsafe()` instead of `getPriceNoOlderThan()`, and performs no validation on the returned price value. This means it can silently return a stale or negative price — the direct analog of Chainlink's minAnswer/maxAnswer circuit breaker issue described in M-16.

### Finding Description

`PythAggregatorV3.latestRoundData()` is implemented as follows: [1](#0-0) 

Two distinct problems exist:

**1. Staleness — `getPriceUnsafe` returns arbitrarily old prices without reverting.**

`IPyth.getPriceUnsafe` is explicitly documented as returning "the most recent price update in this contract without any recency checks" and warns that "the returned price update may be arbitrarily far in the past." [2](#0-1) 

The safe alternative, `getPriceNoOlderThan`, reverts if the price is too old. `PythAggregatorV3` never calls it. [3](#0-2) 

**2. No price positivity check — `price.price` is `int64` and can be negative.**

`PythStructs.Price.price` is typed as `int64`, which can hold negative values. [4](#0-3) 

`latestRoundData()` returns `int256(price.price)` directly with no `<= 0` guard. A negative `answer` returned from `latestRoundData()` is treated by most Chainlink-consuming protocols as a valid price, causing catastrophic mispricing. [5](#0-4) 

By contrast, the Aave-specific `PythPriceOracleGetter` correctly uses `getPriceNoOlderThan` and explicitly reverts on non-positive prices: [6](#0-5) 

`PythAggregatorV3` has no equivalent guards.

### Impact Explanation

Protocols that use `PythAggregatorV3` as a drop-in Chainlink replacement in lending/borrowing contexts (the stated migration use case) will:

- **Stale price scenario**: During a market crash or network disruption, Pyth price updates may lag. `getPriceUnsafe` returns the last stored (inflated) price without reverting. Users can borrow against collateral at an inflated price, creating bad debt — exactly the LUNA/Venus scenario described in M-16.
- **Negative price scenario**: If `price.price` is negative (valid in `int64`), `latestRoundData()` returns a negative `answer`. Protocols that cast this to `uint256` (common in Chainlink integrations) will receive an astronomically large value, allowing unlimited borrowing.

### Likelihood Explanation

The migration guide explicitly recommends deploying `PythAggregatorV3` as a replacement for Chainlink `AggregatorV3Interface` feeds in DeFi protocols: [7](#0-6) 

Any lending protocol that follows this migration path and calls `latestRoundData()` is exposed. The staleness path is reachable whenever Pyth price updates are delayed (network congestion, publisher outages), which is a realistic operational condition.

### Recommendation

Replace `getPriceUnsafe` with `getPriceNoOlderThan` using a configurable `maxAge` parameter set at construction time, and add a positivity check on the returned price:

```solidity
uint public maxAge; // set in constructor

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    require(price.price > 0, "Invalid non-positive price");
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

The same fix should be applied to `latestAnswer()`, `getRoundData()`, `getAnswer()`, and `decimals()`, all of which also call `getPriceUnsafe`. [8](#0-7) 

### Proof of Concept

1. A protocol deploys `PythAggregatorV3` for token T following the Chainlink migration guide.
2. Token T's price crashes from $1.00 to $0.01. Pyth publisher updates are delayed (e.g., 10 minutes of network congestion).
3. The protocol calls `latestRoundData()`. It returns `getPriceUnsafe()` — the last stored price of $1.00 — with no revert.
4. The protocol values token T at $1.00 and allows a user to borrow $0.99 worth of other assets against 1 token T (actually worth $0.01).
5. The loan is immediately underwater; the protocol accumulates bad debt.

Alternatively, if `price.price` is `-1` (a valid `int64`), `latestRoundData()` returns `answer = -1`. A protocol that computes `uint256(answer)` receives `2^256 - 1`, treating the collateral as worth an astronomically large amount.
