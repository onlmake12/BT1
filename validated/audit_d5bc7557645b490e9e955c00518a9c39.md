Audit Report

## Title
Missing Per-Peer Rate Limiter in `BlockFilter` Protocol Enables Unbounded RocksDB Read Amplification - (`sync/src/filter/mod.rs`)

## Summary

The `BlockFilter` P2P protocol handler contains no per-peer rate limiter, unlike the `Relayer` and `HolePunching` protocols which both enforce a 30 req/sec cap keyed by `(peer, message_id)`. Any connected peer can repeatedly send `GetBlockFilterCheckPoints`, `GetBlockFilterHashes`, or `GetBlockFilters` messages, each triggering up to 4,000 unbounded RocksDB reads with no server-side throttle. With up to 125 connected peers sending at high frequency, aggregate RocksDB read pressure is unbounded, degrading block relay and sync throughput.

## Finding Description

`BlockFilter` struct holds only `shared: Arc<SyncShared>` with no rate limiter field: [1](#0-0) 

`try_process` dispatches directly to process handlers with no rate check: [2](#0-1) 

`GetBlockFilterCheckPointsProcess::execute` loops up to `BATCH_SIZE=2000` iterations, each performing two RocksDB reads (`get_block_hash` + `get_block_filter_hash`), stepping by `CHECK_POINT_INTERVAL=2000` per iteration: [3](#0-2) [4](#0-3) 

The same pattern applies to `GetBlockFilterHashes` (BATCH_SIZE=2000, increments by 1) and `GetBlockFilters` (BATCH_SIZE=1000, reads full filter data): [5](#0-4) [6](#0-5) 

By contrast, `Relayer` constructs a `governor::RateLimiter` keyed by `(PeerIndex, message.item_id())` at 30 req/sec and checks it before every dispatch: [7](#0-6) [8](#0-7) 

`HolePunching` applies the identical pattern: [9](#0-8) 

`BlockFilter` is the only message-serving protocol that omits this guard entirely.

## Impact Explanation

Default configuration allows up to 125 inbound peers: [10](#0-9) 

`Filter` is enabled by default in `support_protocols`: [11](#0-10) 

With 125 peers each sending `GetBlockFilterCheckPoints(start_number=0)` at ≥10 Hz, the node sustains ≥5,000,000 RocksDB reads/sec from filter traffic alone, with no throttle. This causes sustained RocksDB block cache eviction (256 MB cache configured at `cache_size = 268435456`), elevated CPU from deserialization, and degraded block relay and sync throughput — fitting **High: bad designs which could cause CKB network congestion with few costs**, since degrading multiple nodes' filter-serving capacity impairs block/transaction propagation across the network.

## Likelihood Explanation

The attack requires only a standard P2P connection negotiating `SupportProtocols::Filter`. No PoW, no keys, no privileged role. `GetBlockFilterCheckPoints` is a 9-byte struct (`start_number: Uint64`), trivially constructable. Any peer that completes the protocol handshake can send it at will at maximum rate.

## Recommendation

Add a `governor::RateLimiter<(PeerIndex, u32), ...>` field to `BlockFilter` in `sync/src/filter/mod.rs`, initialized in `BlockFilter::new` following the same pattern as `Relayer::new`: [7](#0-6) 

Check the rate limiter at the top of `try_process` before dispatching to any of the three process handlers, returning `StatusCode::TooManyRequests` on violation. A cap of 1–2 req/sec per peer per message type is sufficient for legitimate light-client use.

## Proof of Concept

1. Connect 125 peers to a target node, all negotiating `SupportProtocols::Filter`.
2. Each peer sends `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop (≥10 Hz).
3. Monitor RocksDB metrics (`rocksdb.block.cache.miss`, `rocksdb.number.keys.read`) and CPU time attributed to filter reads vs. normal block processing.
4. Observe that RocksDB read throughput scales linearly with peer count and message rate with no server-side throttle, while block relay latency increases measurably.
5. Apply the rate limiter fix and confirm the metrics return to baseline under the same load.

        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,

        match message {
            packed::BlockFilterMessageUnionReader::GetBlockFilters(msg) => {
                GetBlockFiltersProcess::new(msg, self, nc, peer)
                    .execute()
                    .await
            }
            packed::BlockFilterMessageUnionReader::GetBlockFilterHashes(msg) => {
                GetBlockFilterHashesProcess::new(msg, self, nc, peer)
                    .execute()
                    .await
            }
            packed::BlockFilterMessageUnionReader::GetBlockFilterCheckPoints(msg) => {
                GetBlockFilterCheckPointsProcess::new(msg, self, nc, peer)
                    .execute()
                    .await
            }
            packed::BlockFilterMessageUnionReader::BlockFilters(_)
            | packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)
            | packed::BlockFilterMessageUnionReader::BlockFilterCheckPoints(_) => {
                // remote peer should not send block filter to us without asking
                // TODO: ban remote peer
                warn_target!(
                    crate::LOG_TARGET_FILTER,
                    "Received unexpected message from peer: {:?}",
                    peer
                );
                Status::ignored()
            }
        }
    }
