### Title
Legacy `revealWithCallback` Path Allows Malicious Requester to Permanently DOS Provider Fulfillment - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
In `Entropy.sol`, the `revealWithCallback` function contains a legacy execution path (triggered when `req.gasLimit10k == 0`) that calls the requester's `_entropyCallback` directly without any revert-catching mechanism. A malicious requester can deploy a callback contract that always reverts, causing every provider fulfillment attempt to revert, permanently locking the request and the user's paid fees in the contract while continuously wasting the provider keeper's gas.

### Finding Description
`revealWithCallback` branches on `req.gasLimit10k`:

- **New path** (`gasLimit10k != 0`): uses `excessivelySafeCall` to catch reverts safely.
- **Legacy path** (`gasLimit10k == 0`): calls `_entropyCallback` directly with no revert protection.

The legacy path is entered whenever the provider has `defaultGasLimit == 0` (i.e., has not opted into the new callback-failure-state flow). In `requestHelper`, this is set as:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;  // legacy path
}
``` [1](#0-0) 

In the legacy path of `revealWithCallback`, the request is cleared **before** the callback is invoked:

```solidity
clearRequest(provider, sequenceNumber);
// ...
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber, provider, randomNumber
    );
}
``` [2](#0-1) 

If `_entropyCallback` reverts, the entire transaction reverts — including the `clearRequest`. The request is restored to active state. The provider's keeper (Fortuna) will retry indefinitely, wasting gas on every attempt, and the request can never be fulfilled.

### Impact Explanation
1. **Permanent request lock**: The request is never cleared; the user's paid fees are locked in the contract with no recovery path.
2. **Provider keeper DOS**: The Fortuna keeper retries `revealWithCallback` on every unfulfilled request. A malicious requester can flood a legacy provider with requests whose callbacks always revert, causing continuous gas waste and degrading the keeper's ability to serve legitimate users.
3. **No escape hatch**: Unlike the new path (which transitions to `CALLBACK_FAILED` state allowing recovery), the legacy path has no failure state — the request stays `CALLBACK_NOT_STARTED` forever.

### Likelihood Explanation
Any unprivileged user can call `requestWithCallback` or `requestV2(..., 0)` against any provider that has not set `defaultGasLimit`. The attacker only pays the request fee once; the provider pays gas on every failed fulfillment attempt. The attack is cheap to sustain and requires no privileged access. [3](#0-2) 

### Recommendation
Remove the legacy path entirely, or wrap the direct `_entropyCallback` call in a `try/catch` block and transition the request to a `CALLBACK_FAILED` state on revert (mirroring the new path). Providers should be required to set `defaultGasLimit != 0` before accepting new requests.

### Proof of Concept

1. Attacker deploys a malicious consumer contract:
```solidity
contract MaliciousConsumer is IEntropyConsumer {
    function _entropyCallback(uint64, address, bytes32) external override {
        revert("always fails");
    }
    function getEntropy() internal view override returns (address) { ... }
}
```

2. Attacker calls `requestWithCallback(legacyProvider, userContribution)` paying the required fee. Since `legacyProvider.defaultGasLimit == 0`, `req.gasLimit10k` is set to `0`.

3. Provider's Fortuna keeper calls `revealWithCallback(legacyProvider, seqNum, userContrib, providerContrib)`.

4. Execution reaches the legacy `else` branch at line 661. `clearRequest` executes, then `_entropyCallback` is called directly and reverts.

5. The entire transaction reverts. `clearRequest` is undone. The request remains active with `callbackStatus == CALLBACK_NOT_STARTED`.

6. The keeper retries indefinitely. Each retry costs the provider gas. The attacker's fees remain locked with no recovery mechanism. [4](#0-3) [5](#0-4)
