Audit Report

## Title
Eviction Protection Bypass via `as_secs()` Granularity and Unsolicited Ping Manipulation — (`network/src/peer_registry.rs`, `network/src/protocols/ping.rs`)

## Summary
`try_evict_inbound_peer` compares `ping_rtt` values using `.as_secs()`, collapsing all sub-second RTTs to the same score (0), making the lowest-ping protection slot trivially occupiable. The `last_ping_protocol_message_received_at` field is updated by any received Ping — solicited or not — allowing an attacker to permanently hold the most-recent-message protection slot. When `max_inbound ≤ 2 × EVICTION_PROTECT_PEERS` (= 16), both protection rounds exhaust the entire candidate pool, causing `try_evict_inbound_peer` to return `None` and permanently blocking honest inbound connections.

## Finding Description

**Root cause 1 — `.as_secs()` granularity in ping comparator**

`peer_registry.rs` lines 155–164:
```rust
let peer1_ping = peer1.ping_rtt.map(|p| p.as_secs()).unwrap_or_else(|| u64::MAX);
let peer2_ping = peer2.ping_rtt.map(|p| p.as_secs()).unwrap_or_else(|| u64::MAX);
peer2_ping.cmp(&peer1_ping)
```
Any `ping_rtt < 1 s` maps to `0`. All well-connected peers (including attacker peers that respond immediately) receive the same score. The sort among them is arbitrary; the "lowest ping" protection degenerates to a random selection from the full candidate set.

**Root cause 2 — Unsolicited Pings update the protection timestamp**

`ping.rs` lines 215–223 and 62–68:
```rust
PingPayload::Ping(nonce) => {
    self.ping_received(session.id);   // updates last_ping_protocol_message_received_at
    ...
}
fn ping_received(&mut self, id: SessionId) {
    ...
    peer.last_ping_protocol_message_received_at = Some(Instant::now());
}
```
No solicitation check exists. Any remote peer can send a Ping at any time to set this timestamp to `now()`. The "most recent message" protection slot is therefore fully attacker-controlled.

**Root cause 3 — Nonce does not prevent near-zero RTT**

`ping.rs` line 117–119:
```rust
fn nonce(t: &Instant, start_time: Instant) -> u32 {
    t.saturating_duration_since(start_time).as_secs() as u32
}
```
The nonce is embedded in the outgoing Ping message and echoed back by the peer. An attacker simply reflects the received nonce; the check at line 228 (`(true, nonce) == (status.processing, status.nonce())`) is trivially satisfied regardless of response latency.

**Exploit flow for `max_inbound ≤ 16`**

`sort_then_drop` (`peer_registry.rs` lines 55–63) sorts descending and truncates to `len - n`, removing the "best" `n` peers from the candidate list (protecting them). With 16 attacker peers all having `ping_rtt.as_secs() = 0` and `last_ping_protocol_message_received_at = now`:

- Round 1 (ping): 16 candidates → protect 8 → 8 remain
- Round 2 (last message): 8 candidates → protect 8 → **0 remain**
- Round 3 (connection time): `protect_peers = 0 >> 1 = 0` → no-op
- `evict_group.choose(...)` on an empty vec → `None`

`accept_peer` then hits line 120:
```rust
return Err(PeerError::ReachMaxInboundLimit.into());
```
Every subsequent honest connection attempt is permanently rejected.

**Why existing checks fail**

The nonce check validates the Pong reply but does not bound RTT from below. The `sort_then_drop` function has no minimum-pool-size guard before returning. There is no rate limit or solicitation flag on inbound Ping handling.

## Impact Explanation

For nodes configured with `max_inbound ≤ 16` (= `2 × EVICTION_PROTECT_PEERS`), an attacker with 16 TCP connections can permanently prevent any honest peer from obtaining an inbound slot — a complete inbound eclipse. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, since eclipsing multiple nodes disrupts transaction and block propagation across the network. For default configurations (`max_inbound ≈ 117`), the `as_secs()` flaw still renders the ping-based protection slot meaningless for all sub-second-RTT peers, degrading eviction quality across all deployments.

## Likelihood Explanation

- Filling ≤ 16 inbound slots requires only 16 TCP connections from distinct peer IDs; no proof-of-work, no key material, no privileged access.
- Responding to Pings in < 1 s is trivially achievable on any modern host or co-located VPS.
- Sending unsolicited Pings is a single message per session per second.
- The attack is fully reproducible on a single machine in a local test environment.
- The `as_secs()` granularity flaw affects all deployments regardless of `max_inbound`.

## Recommendation

1. **Replace `.as_secs()` with `.as_millis()` or `.as_micros()`** in both comparators inside `try_evict_inbound_peer` (`peer_registry.rs` lines 157 and 175) so sub-second RTT differences are meaningful.
2. **Ignore unsolicited inbound Pings** for the purpose of updating `last_ping_protocol_message_received_at`. Track whether the local node sent a Ping to this session (e.g., via a `pending_ping` flag in `PingStatus`) and only update the timestamp when a Pong is received, or use a separate field updated by application-layer messages (Sync/Relay) that cannot be refreshed by the Ping protocol alone.
3. **Enforce a minimum candidate pool size** before returning from `try_evict_inbound_peer`: if the pool after protection rounds is empty, skip the last protection round(s) or return a fallback candidate, ensuring at least one peer is always evictable.
4. **Rate-limit inbound Ping messages** per session to prevent timestamp refresh flooding.

## Proof of Concept

```
1. Start a CKB node with max_inbound = 16 (max_peers = 24, max_outbound_peers = 8).
2. Connect 16 attacker sessions (distinct peer IDs, any IPs).
3. For each attacker session:
   a. On receiving a Ping from the node, immediately reflect the nonce in a Pong.
      → ping_rtt ≈ 0 µs → as_secs() = 0
   b. Every 500 ms, send an unsolicited Ping to the node.
      → last_ping_protocol_message_received_at = Instant::now() → as_secs() = 0
4. Attempt to connect a 17th (honest) peer.
5. try_evict_inbound_peer:
   - 16 candidates, all ping_rtt.as_secs()=0 → sort arbitrary, protect 8 → 8 remain
   - 8 candidates, all last_message.as_secs()=0 → sort arbitrary, protect 8 → 0 remain
   - protect_peers = 0>>1 = 0 → evict_group empty → returns None
6. accept_peer returns Err(ReachMaxInboundLimit) → honest peer rejected.
7. Repeat step 4 indefinitely: honest peer is permanently blocked.
```

Verifiable with a unit test mirroring `test_accept_inbound_peer_eviction` in `network/src/tests/peer_registry.rs`, using `max_inbound = 16`, setting all 16 peers' `ping_rtt = Some(Duration::from_millis(1))` and `last_ping_protocol_message_received_at = Some(Instant::now())`, then asserting that `accept_peer` for a 17th peer returns `Err(ReachMaxInboundLimit)` with no evicted peer.
