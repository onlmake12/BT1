### Title
Silent u128→u64 Truncating Cast in DAO Withdrawal Capacity Calculation Permanently Blocks Large Depositors - (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` uses a bare `as u64` cast to downcast a u128 intermediate result, silently truncating the value on overflow. Every other u128→u64 conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, but this one path does not. When the product `counted_capacity × withdrawing_ar` exceeds u64::MAX before the division, the truncated result is used as the authoritative "maximum withdraw" capacity. Because the truncated value is far smaller than the true value, any DAO withdrawal transaction whose correct payout exceeds the truncated figure will be permanently rejected by the verifier.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity as:

```rust
// util/dao/src/lib.rs  lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a Rust truncating (wrapping) cast: if `withdraw_counted_capacity > u64::MAX`, the high bits are silently discarded and the result wraps to a small value near zero.

Every other u128→u64 narrowing in the same file is guarded:

```rust
// secondary_block_reward – line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch – line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);

// dao_field_with_current_epoch – line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

`calculate_maximum_withdraw` is the sole exception.

**Overflow condition:**

```
withdraw_counted_capacity = counted_capacity × withdrawing_ar / deposit_ar
```

For this to exceed u64::MAX (≈ 1.84 × 10¹⁹):

| Variable | Bound |
|---|---|
| `counted_capacity` | up to ~3.36 × 10¹⁸ shannons (total CKB supply) |
| `withdrawing_ar / deposit_ar` | must exceed ~5.5× |

`ar` starts at `10_000_000_000_000_000` (10¹⁶) and grows each block by `ar × g2 / C`. Over a sufficiently long chain lifetime the ratio can exceed 5.5×. A single large depositor (e.g., holding a significant fraction of total supply) depositing early and withdrawing late is the concrete trigger.

---

### Impact Explanation

`calculate_maximum_withdraw` is the authoritative source for the permitted withdrawal amount. It is called from:

1. **`transaction_maximum_withdraw`** → **`transaction_fee`** → tx-pool admission and `verification/src/transaction_verifier.rs` [5](#0-4) 
2. **`transaction_maximum_withdraw`** → **`withdrawed_interests`** → **`dao_field_with_current_epoch`** → block assembly and block verification [6](#0-5) 
3. **`calculate_dao_maximum_withdraw` RPC** directly calls `calculate_maximum_withdraw` [7](#0-6) 

When the truncated value is, say, 500 shannons instead of the true 20 billion shannons, the verifier computes `fee = maximum_withdraw − outputs_capacity` and gets a negative result (underflow), returning an error. The withdrawal transaction is rejected at both the tx-pool and block-verification layers. The depositor's funds are permanently locked: no valid withdrawal transaction can ever be constructed because the node's own arithmetic produces the wrong bound.

---

### Likelihood Explanation

The overflow requires `ar` to grow to roughly 5.5× its genesis value. The secondary issuance rate is ~1.344 billion CKB/year against a total supply of ~33.6 billion CKB, giving a per-block `ar` growth of approximately `ar × g2 / C`. This is a slow but monotonically increasing quantity. On a long-lived chain (decades), the ratio will eventually cross the threshold. A depositor who locks a large fraction of the supply early and holds through that crossing is permanently affected. The condition is deterministic and chain-state-driven — no attacker action is required beyond normal chain operation.

---

### Recommendation

Replace the bare `as u64` cast with the same checked conversion used everywhere else in the file:

```rust
// util/dao/src/lib.rs  lines 155-156  — proposed fix
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
```

This makes overflow an explicit, recoverable `DaoError::Overflow` rather than silent data corruption, consistent with all sibling calculations in the same function.

---

### Proof of Concept

Construct a scenario where `counted_capacity × withdrawing_ar` overflows u64:

```
deposit_ar      = 10_000_000_000_000_000   (genesis ar, 10^16)
withdrawing_ar  = 60_000_000_000_000_000   (6× growth, reachable over time)
counted_capacity = 3_200_000_000_00_000_000 shannons  (32 billion CKB, within total supply)

withdraw_counted_capacity (u128) = 3.2×10^18 × 6×10^16 / 10^16
                                 = 3.2×10^18 × 6
                                 = 1.92×10^19

u64::MAX                         ≈ 1.844×10^19

1.92×10^19 > u64::MAX  → overflow

withdraw_counted_capacity as u64 = 1.92×10^19 - 1.844×10^19 ≈ 7.6×10^17  (truncated)
```

The computed maximum withdraw is now ~7.6 × 10¹⁷ shannons instead of the correct ~1.92 × 10¹⁹ shannons. Any withdrawal transaction claiming the correct amount will fail the fee/capacity check. The depositor's DAO cell is permanently unspendable. [8](#0-7)

        let withdrawing_header = self

            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let output_capacity: Capacity = output.capacity().into();

            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
