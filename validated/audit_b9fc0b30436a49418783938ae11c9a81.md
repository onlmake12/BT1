### Title
`check_tx_fee` Uses `tx_size` Instead of `weight` for `min_fee_rate` Enforcement, Allowing High-Cycles Transactions to Bypass the Fee Floor — (File: `tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` computes the minimum required fee using only the serialized byte size of the transaction, while the canonical fee-rate unit in CKB is **shannons per kilo-weight**, where `weight = max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For a transaction whose cycle cost dominates its byte size, `weight >> tx_size`, so the size-only check admits the transaction at a fee far below `min_fee_rate`. No subsequent weight-based fee-rate check is performed before the entry is inserted into the pool.

---

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` enforces the pool's minimum fee rate as follows:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0) 

`FeeRate::fee` computes `(fee_rate × weight) / 1000`:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
``` [2](#0-1) 

When called with `tx_size` instead of `weight`, the minimum fee is computed as `(min_fee_rate × tx_size) / 1000`. However, the true weight of a transaction is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [4](#0-3) 

After `verify_rtx` returns the actual cycles, the `TxEntry` is created with the real cycle count and its `fee_rate()` method correctly uses weight:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [5](#0-4) 

But `submit_entry` performs **no second fee-rate check** after cycles are known — the entry is inserted directly into the pool. [6](#0-5) 

The same pattern appears in `calculate_min_replace_fee`, which also uses `size` instead of weight for the RBF extra-fee floor:

```rust
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
``` [7](#0-6) 

---

### Impact Explanation

**Concrete numbers** (mainnet defaults):

| Parameter | Value |
|---|---|
| `min_fee_rate` | 1 000 shannons / kilo-weight |
| Typical 2-in-2-out `tx_size` | 597 bytes |
| `max_tx_verify_cycles` | 70 000 000 |
| Cycle-equivalent bytes at max cycles | 70 000 000 × 0.000 170 571 4 ≈ 11 940 |
| True `weight` | max(597, 11 940) = **11 940** |

- **Size-based check** requires: `1000 × 597 / 1000 = 597 shannons`
- **Weight-based requirement** would be: `1000 × 11 940 / 1000 = 11 940 shannons`
- **Actual fee rate** of admitted entry: `597 × 1000 / 11 940 ≈ 50 shannons/kilo-weight` — **20× below `min_fee_rate`**

An attacker pays only ~5 % of the intended minimum fee per transaction. They can flood the pool with entries that have a true fee rate far below the configured floor, consuming pool capacity and CPU verification time. Miners will not include these transactions (low fee rate), so they persist in the pool and displace legitimate transactions until evicted by `limit_size`.

---

### Likelihood Explanation

The attacker entry path is fully unprivileged: any RPC caller (`send_transaction`) or network peer (relay) can submit a transaction. Crafting a CKB script that consumes near-maximum cycles while keeping the serialized transaction small is straightforward — a tight loop in a RISC-V lock script suffices. No special key, majority hashpower, or social engineering is required.

---

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee-rate check using the true weight before inserting the entry into the pool:

```rust
// After verify_rtx returns `verified.cycles`:
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

Apply the same fix to `calculate_min_replace_fee`, substituting the entry's true weight for `size`.

---

### Proof of Concept

1. Write a CKB lock script that executes a tight loop consuming ~70 000 000 cycles. Deploy it on-chain.
2. Construct a transaction spending a cell locked by that script. Serialized size ≈ 597 bytes.
3. Set the transaction fee to 597 shannons (= `min_fee_rate × tx_size / 1000`).
4. Submit via `send_transaction` RPC.
5. `check_tx_fee` computes `min_fee = 1000 × 597 / 1000 = 597` — check passes.
6. `verify_rtx` executes the script, consuming ~70 000 000 cycles.
7. `TxEntry` is created with `cycles ≈ 70 000 000`, `fee = 597`, `size = 597`.
8. `entry.fee_rate()` = `597 × 1000 / 11 940 ≈ 50 shannons/kilo-weight` — **20× below `min_fee_rate`**.
9. Entry is inserted into the pool without further fee-rate validation.
10. Repeat to fill the pool with sub-minimum-fee-rate entries at 5 % of the intended cost.
