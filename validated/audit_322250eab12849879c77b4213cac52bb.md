Audit Report

## Title
`consecutive_failed` Never Reset on Successful Transaction Inclusion Allows Premature Block-Building Loop Termination — (`tx-pool/src/component/tx_selector.rs`)

## Summary

In `txs_to_commit`, the `consecutive_failed` counter is initialized once and incremented on two failure paths but is never reset to zero when a transaction is successfully packaged. An attacker who places alternating oversized and small transactions in fee-rate order in the proposed pool can cause `consecutive_failed` to accumulate monotonically across successful inclusions, reach `MAX_CONSECUTIVE_FAILURES = 4000`, and terminate the loop while the block still has physical room for profitable transactions.

## Finding Description

`consecutive_failed` is initialized once at the top of `txs_to_commit`: [1](#0-0) 

The pre-loop filter passes any entry whose `ancestors_size <= size_limit`: [2](#0-1) 

Inside the loop, the first failure path (size/cycles overflow) increments the counter: [3](#0-2) 

The second failure path (ancestor not proposed) also increments it: [4](#0-3) 

The entire success path — collecting ancestors, updating `size`/`cycles`, calling `update_modified_entries` — contains no assignment to `consecutive_failed`: [5](#0-4) 

**Exploit construction:** Place in the proposed pool:
- 4001 "small" txs: `ancestors_size = 1`, fee rates 100, 98, 96, …
- 4001 "large" txs: `ancestors_size ≈ size_limit`, fee rates 99, 97, 95, …

`sorted_proposed_iter()` yields them in strict descending fee-rate order: S(100), L(99), S(98), L(97), …

- S(100): `next_size = 0 + 1 ≤ size_limit` → success; `consecutive_failed` stays 0.
- L(99): `next_size = 1 + size_limit > size_limit` → failure; `consecutive_failed = 1`.
- S(98): success; `consecutive_failed` stays **1** (not reset).
- L(97): failure; `consecutive_failed = 2`.
- …
- After 4001 L-failures interleaved with 4001 S-successes: `consecutive_failed = 4001 > 4000` → `break`.

Remaining small transactions — which fit within the block's remaining size and cycles budget — are abandoned. Note that `failed_txs` is only populated when `using_modified = true` (lines 154–156), so large transactions from the main pool iterator are not cached and do not short-circuit; each consumes one loop iteration and one increment of `consecutive_failed`. [6](#0-5) 

## Impact Explanation

Miners systematically under-fill blocks. In the constructed scenario, profitable small transactions that fit within the block's remaining size and cycles budget are excluded solely because `consecutive_failed` was never reset on success. This reduces miner revenue per block and delays legitimate user transactions. This matches the in-scope **High** impact: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*. [7](#0-6) 

## Likelihood Explanation

**Entry point:** Standard `send_transaction` RPC or P2P relay — no privilege required. **Precondition:** Transactions must reach `Status::Proposed` via CKB's two-phase commit; any miner will propose transactions with sufficient fees. The attacker's small transactions pay their own way. Large transactions require locking CKB capacity proportional to their size, but this cost is amortized across every block-template construction call while the adversarial transactions remain in the proposed pool. The attacker can also reduce individual large-tx size (e.g., `ancestors_size = size_limit / k`) at the expense of needing `k` small-tx bytes accumulated before each large-tx fails, preserving the monotonic accumulation of `consecutive_failed`. **Repeatability:** The attack applies to every block-template construction call. **Detectability:** None — no log or metric exposes premature loop termination.

## Recommendation

Reset `consecutive_failed` to zero immediately after a transaction package is successfully added, restoring the original intent that the counter measures an *unbroken run* of failures:

```rust
// after self.update_modified_entries(&ancestors);
consecutive_failed = 0;
``` [8](#0-7) 

## Proof of Concept

```
Pool state (all Status::Proposed):
  4001 "small" txs: ancestors_size=1,          fee_rate = 100, 98, 96, ..., 100-2*4000
  4001 "large" txs: ancestors_size=size_limit,  fee_rate =  99, 97, 95, ...,  99-2*4000

sorted_proposed_iter order: S(100), L(99), S(98), L(97), ..., S(...), L(...)

Trace:
  iter 1:    S(100) → success,  consecutive_failed = 0
  iter 2:    L(99)  → failure,  consecutive_failed = 1
  iter 3:    S(98)  → success,  consecutive_failed = 1  ← not reset
  iter 4:    L(97)  → failure,  consecutive_failed = 2
  ...
  iter 8001: S(..)  → success,  consecutive_failed = 4000
  iter 8002: L(..)  → failure,  consecutive_failed = 4001 > 4000 → BREAK

Result: remaining small txs abandoned despite block having room.

Comparison:
  pool with 8002 small txs only (no large):  all ~8002 committed
  pool with 4001 small + 4001 large (above): loop breaks at 4001 small committed
  delta caused solely by missing consecutive_failed = 0 on success
```

A unit test can be written against `TxSelector::txs_to_commit` by constructing a `PoolMap` with the above interleaved entries and asserting that the returned `Vec<TxEntry>` contains all 4001 small transactions (currently it will contain only 4001 due to premature termination; after the fix it will contain all of them).
