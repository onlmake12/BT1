### Title
Overpayment Not Refunded in `Echo.requestPriceUpdatesWithCallback()` — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback()` accepts any `msg.value >= requiredFee` but stores the entire `msg.value - pythFeeInWei` as the provider fee (`req.fee`). Any ETH sent beyond the exact required fee is silently credited to the provider rather than refunded to the caller, causing direct financial loss for users who overpay.

---

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback()` computes the exact required fee and enforces only a lower bound:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
```

It then stores the provider fee as the full `msg.value` minus only the Pyth protocol fee:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```

When `executeCallback()` is later called, the provider is credited:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

Because `req.fee` was set to `msg.value_request - pythFeeInWei` rather than `requiredFee - pythFeeInWei`, any overpayment at request time is permanently transferred to the provider's accrued balance. There is no refund path.

This directly contradicts the interface documentation in `IEcho.sol`, which states:

```
* @dev The msg.value must be equal to getFee(callbackGasLimit)
```

The implementation allows `msg.value > requiredFee` but silently absorbs the excess into the provider's balance instead of enforcing equality or refunding the difference.

---

### Impact Explanation

Any user who sends `msg.value > requiredFee` loses the excess ETH permanently to the provider. This is a direct, unrecoverable financial loss. The excess is not trapped in the contract — it is credited to the provider's `accruedFeesInWei` and becomes withdrawable by the provider via `withdrawAsFeeManager()`.

Scenarios where overpayment occurs in practice:
- The fee changes between when the user queries `getFee()` and when the transaction is mined (e.g., `tx.gasprice` fluctuation or provider fee update).
- The user adds a small buffer to ensure the transaction does not revert due to a fee increase.
- The user hardcodes a fee value that is higher than the current fee.

---

### Likelihood Explanation

Moderate. The entry path is any unprivileged user calling `requestPriceUpdatesWithCallback`. Overpayment is a common pattern in fee-bearing transactions where users add a buffer to avoid reverts. The fee computation includes `callbackGasLimit * feePerGasInWei`, which is sensitive to gas price changes, making exact-fee submission difficult in practice.

---

### Recommendation

Enforce exact payment or refund the excess before storing `req.fee`:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();

// Refund any overpayment to the caller
if (msg.value > requiredFee) {
    (bool refunded, ) = msg.sender.call{value: msg.value - requiredFee}("");
    require(refunded, "Refund failed");
}

// Store only the exact provider fee, not the full msg.value
req.fee = SafeCast.toUint96(requiredFee - _state.pythFeeInWei);
```

---

### Proof of Concept

1. Provider registers with `baseFeeInWei = 0`, `feePerFeedInWei = 0`, `feePerGasInWei = 1`.
2. `getFee(provider, 100_000, priceIds)` returns `pythFeeInWei + 100_000` (e.g., `200_000` wei total).
3. User calls `requestPriceUpdatesWithCallback{value: 1 ether}(...)` — overpaying by `~1 ether - 200_000 wei`.
4. `req.fee = 1 ether - pythFeeInWei` is stored.
5. When `executeCallback` is called, provider is credited `req.fee + msg.value_execute - pythFee_execute`, which includes the ~1 ether overpayment.
6. Provider withdraws the excess via `withdrawAsFeeManager()`.
7. User has permanently lost ~1 ether with no recourse. [1](#0-0) [2](#0-1) [3](#0-2)
