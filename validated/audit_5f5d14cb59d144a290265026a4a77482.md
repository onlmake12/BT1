### Title
`Scheduler.updateSubscription` Bypasses `_validateSubscriptionParams` for Inactive-to-Inactive Transitions, Allowing Invalid State to Be Stored Silently — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `updateSubscription` function contains an early return path for inactive-to-inactive subscription transitions that skips `_validateSubscriptionParams`. This allows a subscription manager to store invalid subscription parameters (e.g., empty `priceIds`, duplicate entries, invalid `updateCriteria`) without any revert, while still emitting a `SubscriptionUpdated` event that signals a valid update to external observers — directly analogous to the ERC1155Fuse pattern where `safeTransferFrom` silently succeeds without performing the expected state change.

---

### Finding Description

The `updateSubscription` function checks whether a subscription was inactive and will remain inactive. If so, it stores the new parameters directly and returns early, bypassing the `_validateSubscriptionParams` call entirely: [1](#0-0) 

```solidity
// Lines 94-103
bool wasActive = currentParams.isActive;
bool willBeActive = newParams.isActive;
if (!wasActive && !willBeActive) {
    // Update subscription parameters
    _state.subscriptionParams[subscriptionId] = newParams;
    emit SubscriptionUpdated(subscriptionId);
    return;
}
_validateSubscriptionParams(newParams);
```

`_validateSubscriptionParams` enforces all critical structural invariants on subscription parameters: [2](#0-1) 

Specifically, it enforces:
- `priceIds.length > 0` (no empty subscriptions)
- `priceIds.length <= MAX_PRICE_IDS_PER_SUBSCRIPTION`
- No duplicate `priceIds`
- `readerWhitelist.length <= MAX_READER_WHITELIST_SIZE`
- No duplicate `readerWhitelist` entries
- Valid `updateCriteria` (heartbeat or deviation must be properly configured)

All of these checks are skipped when `!wasActive && !willBeActive`. The function emits `SubscriptionUpdated` regardless, signaling to external observers that a valid update occurred.

Additionally, the early return path also skips the `_clearRemovedPriceUpdates` call that appears later in the normal flow: [3](#0-2) 

This means stale price data for removed `priceIds` persists in `_state.priceUpdates[subscriptionId]` even after the subscription params are updated to no longer include those IDs.

---

### Impact Explanation

A subscription manager can store structurally invalid parameters for an inactive subscription without any revert. The `SubscriptionUpdated` event is emitted unconditionally, which can mislead:

- Off-chain monitoring systems and keeper bots that track subscription state via events
- Any contract or system that assumes `SubscriptionUpdated` implies a validly-structured subscription

The inconsistent on-chain state means the subscription's stored `priceIds` may be empty, contain duplicates, or have an invalid `updateCriteria` configuration. While the subscription cannot be actively used while inactive (keepers check `isActive` at line 286), the stale price data for removed feeds persists in storage, and the misleading event emission violates the contract's invariants. If the manager later attempts to reactivate the subscription with the same invalid params, `_validateSubscriptionParams` will revert, effectively requiring the manager to provide entirely new params — the stored invalid state cannot be directly reactivated. [4](#0-3) 

---

### Likelihood Explanation

Any subscription manager — an unprivileged role available to any user who calls `createSubscription` — can trigger this by calling `updateSubscription` with invalid `newParams` while their subscription is inactive. The `onlyManager` modifier restricts the call to the subscription's own manager, but any user can become a manager by creating a subscription. This can occur accidentally (e.g., a manager clearing their `priceIds` list while deactivated) or intentionally as a griefing vector against their own subscription state. [5](#0-4) 

---

### Recommendation

Remove the early return path and always call `_validateSubscriptionParams` before storing new parameters. The optimization of skipping validation for inactive subscriptions is unsound because the stored params define the subscription's future behavior upon reactivation:

```solidity
// Remove the early return block entirely.
// Always validate before storing:
_validateSubscriptionParams(newParams);

// Then handle the inactive→inactive case without early return,
// ensuring _clearRemovedPriceUpdates is also called.
```

At minimum, `_validateSubscriptionParams(newParams)` must be called before `_state.subscriptionParams[subscriptionId] = newParams` in all code paths.

---

### Proof of Concept

1. Alice calls `createSubscription` with valid params (`priceIds = [feedA, feedB]`, `isActive = true`). Alice is now the manager.
2. Alice calls `updateSubscription(id, {isActive: false, priceIds: [feedA, feedB], ...})` to deactivate. This takes the normal path, calls `_validateSubscriptionParams`, stores valid params.
3. Alice calls `updateSubscription(id, {isActive: false, priceIds: [], updateCriteria: {updateOnHeartbeat: false, updateOnDeviation: false}, ...})` — passing structurally invalid params with `isActive = false`.
4. The condition `!wasActive && !willBeActive` is `true`. The early return path at lines 97–102 executes: invalid params are stored, `SubscriptionUpdated` is emitted, function returns without reverting.
5. On-chain state now has `priceIds = []` and invalid `updateCriteria` for this subscription. `SubscriptionUpdated` was emitted, signaling a valid update to external observers.
6. Stale `priceUpdates` for `feedA` and `feedB` remain in storage because `_clearRemovedPriceUpdates` was never called.
7. If Alice attempts to reactivate by calling `updateSubscription(id, {isActive: true, priceIds: [], ...})`, `_validateSubscriptionParams` reverts with `EmptyPriceIds` — Alice must provide entirely new valid params to recover.

This mirrors the ERC1155Fuse pattern precisely: the function silently succeeds (no revert, event emitted) when the expected validation should have caused a revert, because an early return condition was satisfied — leaving on-chain state inconsistent with what the emitted event implies.

```text
    function _validateSubscriptionParams(
        SchedulerStructs.SubscriptionParams memory params
    ) internal pure {
        // No zero‐feed subscriptions
        if (params.priceIds.length == 0) {
            revert SchedulerErrors.EmptyPriceIds();
        }

        // Price ID limits and uniqueness
        if (
            params.priceIds.length >
            SchedulerConstants.MAX_PRICE_IDS_PER_SUBSCRIPTION
        ) {
            revert SchedulerErrors.TooManyPriceIds(
                params.priceIds.length,
                SchedulerConstants.MAX_PRICE_IDS_PER_SUBSCRIPTION
            );
        }
        for (uint i = 0; i < params.priceIds.length; i++) {
            for (uint j = i + 1; j < params.priceIds.length; j++) {
                if (params.priceIds[i] == params.priceIds[j]) {
                    revert SchedulerErrors.DuplicatePriceId(params.priceIds[i]);
                }
            }
        }

        // Whitelist size limit and uniqueness
        if (params.readerWhitelist.length > MAX_READER_WHITELIST_SIZE) {
            revert SchedulerErrors.TooManyWhitelistedReaders(
                params.readerWhitelist.length,
                MAX_READER_WHITELIST_SIZE
            );
        }
        for (uint i = 0; i < params.readerWhitelist.length; i++) {
            for (uint j = i + 1; j < params.readerWhitelist.length; j++) {
                if (params.readerWhitelist[i] == params.readerWhitelist[j]) {
                    revert SchedulerErrors.DuplicateWhitelistAddress(
                        params.readerWhitelist[i]
                    );
                }
            }
        }

        // Validate update criteria
        if (
            !params.updateCriteria.updateOnHeartbeat &&
            !params.updateCriteria.updateOnDeviation
        ) {
            revert SchedulerErrors.InvalidUpdateCriteria();
        }
        if (
            params.updateCriteria.updateOnHeartbeat &&
            params.updateCriteria.heartbeatSeconds == 0
        ) {
            revert SchedulerErrors.InvalidUpdateCriteria();
        }
        if (
            params.updateCriteria.updateOnDeviation &&
            params.updateCriteria.deviationThresholdBps == 0
        ) {
            revert SchedulerErrors.InvalidUpdateCriteria();
        }
    }
```
