### Title
Off-By-One in Staleness Boundary Check Allows Price Exactly At Age Threshold to Pass — (`File: target_chains/ethereum/sdk/solidity/AbstractPyth.sol`)

---

### Summary

`getPriceNoOlderThan` and `getEmaPriceNoOlderThan` in `AbstractPyth.sol` use a strict `>` comparison instead of `>=` when checking whether a cached price is stale. This means a price whose age equals exactly the caller-specified `age` parameter is accepted as fresh, when the function's own name and semantics require it to be rejected.

---

### Finding Description

In `AbstractPyth.sol`, both staleness-guarded accessors share the same boundary check:

```solidity
// getPriceNoOlderThan (line 56)
if (diff(block.timestamp, price.publishTime) > age)
    revert PythErrors.StalePrice();

// getEmaPriceNoOlderThan (line 75)
if (diff(block.timestamp, price.publishTime) > age)
    revert PythErrors.StalePrice();
``` [1](#0-0) 

The function is named `getPriceNoOlderThan(id, age)`. "No older than `age`" semantically means the price must be **strictly younger** than `age` seconds — i.e., the check should reject when `diff >= age`. The current `diff > age` check accepts a price whose age is exactly equal to `age`, which violates the stated contract.

The Aptos and Sui implementations of the same function correctly use strict `<`:

```move
// Aptos pyth.move line 485
assert!(age < max_age_secs, error::stale_price_update());

// Sui pyth.move line 409
assert!(age < max_age_secs, E_STALE_PRICE_UPDATE);
``` [2](#0-1) [3](#0-2) 

Both of those correctly treat `age == max_age_secs` as stale. The Ethereum implementation diverges at the boundary.

---

### Impact Explanation

Any protocol that calls `getPriceNoOlderThan(id, age)` or `getEmaPriceNoOlderThan(id, age)` on the Ethereum Pyth contract receives a price that is exactly `age` seconds old when it should receive a `StalePrice` revert. For protocols with tight freshness requirements (e.g., derivatives using `age = 10`), this silently extends the effective staleness window by one second. A stale price used for liquidation, collateral valuation, or settlement can cause incorrect financial outcomes for users of those protocols.

---

### Likelihood Explanation

The entry path requires no privilege: any transaction sender or protocol contract can call `getPriceNoOlderThan`. The boundary condition (`diff == age`) is a one-second window, but EVM block timestamps are coarse (one block ≈ 12 s on Ethereum mainnet), making an exact match plausible across any block. The `getPrice()` convenience wrapper also routes through this path via `getPriceNoOlderThan(id, getValidTimePeriod())`, so the exposure is not limited to direct callers. [4](#0-3) 

---

### Recommendation

Change both staleness checks from strict `>` to `>=`:

```solidity
// getPriceNoOlderThan
if (diff(block.timestamp, price.publishTime) >= age)
    revert PythErrors.StalePrice();

// getEmaPriceNoOlderThan
if (diff(block.timestamp, price.publishTime) >= age)
    revert PythErrors.StalePrice();
```

This aligns the Ethereum implementation with the Aptos and Sui implementations and with the function's documented semantics.

---

### Proof of Concept

1. A price feed is updated at `T = 1000`. `publishTime = 1000`.
2. A protocol calls `getPriceNoOlderThan(id, 60)` at `block.timestamp = 1060`.
3. `diff(1060, 1000) = 60`. The check `60 > 60` is **false**, so no revert occurs.
4. The price from `T = 1000` is returned as fresh, even though it is exactly 60 seconds old — the boundary the caller specified as the maximum acceptable age.
5. With the fix (`>= 60`), the check `60 >= 60` is **true** and `StalePrice` is correctly reverted. [5](#0-4)
