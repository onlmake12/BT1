### Title
Cross-Chain Replay of Lazer Price Updates Due to Missing Chain Binding in Signed Payload - (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` computes the signature hash as `keccak256(payload)` where `payload` contains only price data (a shared `PAYLOAD_MAGIC`, timestamp, channel, and feed values). No chain ID, contract address, or EIP-712 domain separator is included in the signed material. Because the same trusted signer key is registered across all EVM deployments of `PythLazer`, a valid price update accepted on one chain is cryptographically valid on every other chain where that signer is trusted.

---

### Finding Description

In `PythLazer.sol`, `verifyUpdate` extracts the payload from bytes `[71 : 71+payload_len]` and computes:

```solidity
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(hash, uint8(update[68]) + 27,
    bytes32(update[4:36]), bytes32(update[36:68]));
``` [1](#0-0) 

The outer EVM-specific wrapper magic (`EVM_FORMAT_MAGIC = 706910618`, bytes 0–3) is checked for format validity but is **not** included in the hash. The signed payload begins with `PAYLOAD_MAGIC = 2479346549`, which is the same constant used in the Sui contract (`PAYLOAD_MAGIC: u32 = 2479346549`) and the Solana/Aptos integrations. [2](#0-1) [3](#0-2) 

The Sui contract's `verify_le_ecdsa_message` recovers the signer from the identical payload bytes:

```move
let pubkey = secp256k1_ecrecover(signature, payload, 0);
``` [4](#0-3) 

Because the payload format and magic are identical across all chains, and because the Pyth Lazer infrastructure registers the same trusted signer key on every deployment, a signature produced for one chain is mathematically valid on every other chain.

---

### Impact Explanation

An unprivileged attacker who observes a valid Lazer update on Chain A (all updates are public on-chain) can submit the identical `update` bytes to `PythLazer.verifyUpdate` on Chain B. The call succeeds because:

1. `keccak256(payload)` is identical (same bytes, no chain binding).
2. The recovered `signer` address is the same trusted Pyth Lazer key registered on Chain B.
3. `isValidSigner(signer)` returns `true`. [5](#0-4) 

Concrete impact scenarios:

- **Stale cross-chain price injection**: An attacker replays a price update from Chain A (e.g., Ethereum) to Chain B (e.g., Arbitrum) seconds after it was originally submitted. Consumer contracts on Chain B that enforce a freshness window (e.g., 10 seconds) accept the replayed update as fresh, even though it was not signed specifically for Chain B.
- **Testnet-to-mainnet replay**: If the same signer key is used on a testnet deployment and mainnet (a common operational pattern for oracle infrastructure), a testnet price update — which may be manipulated or incorrect — can be replayed on mainnet and accepted by any consumer contract.
- **Cross-ecosystem replay**: The `leEcdsa` payload format and `PAYLOAD_MAGIC` are shared between EVM and Sui. A payload verified on Sui can be replayed on EVM chains if the same ECDSA signer key is trusted on both.

Consumer contracts that use Lazer prices for liquidations, collateral valuation, or trade execution are directly affected.

---

### Likelihood Explanation

- **Entry path is fully permissionless**: `verifyUpdate` is a public `payable` function requiring only a small fee (`verification_fee = 1 wei`). Any address can call it with any `bytes calldata update`.
- **No privileged access required**: The attacker only needs to observe a valid update on one chain (trivially available from on-chain events or mempool) and submit it to another.
- **Same signer key across chains**: Pyth Lazer's operational model uses a single signer infrastructure registered on all supported chains, making the cross-chain replay directly exploitable.

---

### Recommendation

Include chain-specific binding in the signed payload. The standard approach is EIP-712 domain separation, which incorporates `chainId` and `verifyingContract` into the hash:

```solidity
bytes32 domainSeparator = keccak256(abi.encode(
    keccak256("EIP712Domain(string name,uint256 chainId,address verifyingContract)"),
    keccak256("PythLazer"),
    block.chainid,
    address(this)
));
bytes32 hash = keccak256(abi.encodePacked("\x19\x01", domainSeparator, keccak256(payload)));
```

Alternatively, the Lazer payload format itself could be extended to include a `chain_id` field that the signer commits to, and the on-chain contract verifies it matches `block.chainid`. This mirrors the fix applied in the referenced Futureswap PR #495, where the contract-specific `Registry` address was added to the signed message.

---

### Proof of Concept

1. Deploy `PythLazer` on two EVM chains (e.g., Ethereum and Arbitrum) with the same trusted signer registered on both.
2. On Ethereum, call `verifyUpdate{value: fee}(update)` with a valid Lazer price update. Record the `update` bytes from the transaction calldata.
3. On Arbitrum, call `verifyUpdate{value: fee}(update)` with the **identical** `update` bytes.
4. Observe that the call succeeds and returns the same `payload` and `signer` — the Arbitrum contract has accepted a price update that was never signed for Arbitrum.

public(package) fun verify_le_ecdsa_message(
    state: &State,
    clock: &Clock,
    signature: &vector<u8>,
    payload: &vector<u8>,
) {
    let current_cap = state.current_cap();

    // 0 stands for keccak256 hash
    let pubkey = secp256k1_ecrecover(signature, payload, 0);

    // Check if the recovered pubkey is in the trusted signers list
    let trusted_signers = state.trusted_signers(&current_cap);
    let mut maybe_idx = trusted_signers.find_index!(|signer|
        signer.public_key() == &pubkey
    );

    assert!(maybe_idx.is_some(), ESignerNotTrusted);
    let idx = maybe_idx.extract();
    let expires_at_ms = trusted_signers[idx].expires_at_ms();
    assert!(clock.timestamp_ms() < expires_at_ms, ESignerExpired);
