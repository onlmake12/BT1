### Title
Unbounded Cell Accumulation in `get_transactions` with `group_by_transaction=true` Violates `limit` Response-Size Invariant — (`util/indexer/src/service.rs`)

---

### Summary

The `group_by_transaction=true` path in `IndexerHandle::get_transactions` contains a break condition that only fires when the iterator crosses a transaction boundary **after** the limit is reached. When the last transaction in the result set has many cells matching the search script, the loop continues accumulating all of them into the final entry with no per-transaction cell cap, making the actual response size a function of the transaction's cell count rather than the caller-supplied `limit`.

---

### Finding Description

In `util/indexer/src/service.rs`, `get_transactions` validates the `limit` parameter: [1](#0-0) 

Then, in the `group_by_transaction` branch, the loop break condition is: [2](#0-1) 

The condition is a logical AND: it only breaks when **both** `tx_with_cells.len() == limit` **and** the incoming cell belongs to a **different** transaction. When `len == limit` but the current cell's `tx_hash` matches the last entry's `tx_hash`, the condition evaluates to `false` and the loop continues. The cell is then unconditionally appended: [3](#0-2) 

This means the last `IndexerTxWithCells` entry in the result can accumulate every cell of that transaction — inputs and outputs — with no upper bound other than the transaction's own cell count and the `TimeoutIterator`.

The `TimeoutIterator` (default 10 s) is the only backstop: [4](#0-3) 

However, iterating thousands of RocksDB key-value entries takes milliseconds, so the timeout does not fire before all cells are accumulated. The timeout only returns an error if the total query time exceeds the limit; it does not cap per-transaction cell count.

`request_limit` (default `usize::MAX`) caps only the `limit` parameter itself, not cells per transaction: [5](#0-4) 

---

### Impact Explanation

A caller submits `get_transactions` with `group_by_transaction=true` and `limit=1`. If the first matching transaction has N outputs all sharing the same lock script, the single returned `TxWithCells` object contains N cell entries. The JSON response size grows linearly with N. CKB's block-size limit bounds N to roughly a few thousand outputs per transaction, but that is still orders of magnitude larger than what `limit=1` implies. Under concurrent load, each such request allocates a proportionally large in-process buffer before serialization, creating memory pressure on the node.

---

### Likelihood Explanation

The attack requires a transaction with many outputs sharing a lock script to be confirmed on-chain. CKB's block size limit constrains the maximum output count per transaction, but a few thousand outputs is achievable within those limits. The RPC endpoint is unauthenticated by default. No privileged access, PoW, or key material is required beyond submitting and mining a normal (if large) transaction.

---

### Recommendation

Add a per-transaction cell cap inside the accumulation loop. When `tx_with_cells.len() == limit` and the current cell still belongs to the last transaction, either:

1. **Break immediately** (accept that the last transaction may be incomplete and rely on cursor-based pagination to resume), or
2. **Cap `last.cells` at a configurable maximum** (e.g., `request_limit`) and break once that cap is reached.

Option 1 is simpler and consistent with how the non-grouped path handles the limit. The cursor (`last_key`) already supports resumption.

---

### Proof of Concept

1. Construct a CKB transaction with ~5,000 outputs, all using the same lock script `S`.
2. Submit and mine the transaction.
3. Call:
   ```json
   {
     "method": "get_transactions",
     "params": [
       {"script": <S>, "script_type": "lock", "group_by_transaction": true},
       "asc",
       "0x1"
     ]
   }
   ```
4. Observe that the response contains exactly 1 `TxWithCells` object whose `cells` array has ~5,000 entries, producing a response of several hundred KB despite `limit=1`.
5. Repeat concurrently from multiple clients to amplify memory pressure on the node.
