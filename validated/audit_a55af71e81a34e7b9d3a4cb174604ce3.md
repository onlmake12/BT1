### Title
Entropy Provider Accrued Fees Permanently Frozen if Provider or Fee Manager is a Non-Payable Contract — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Both `withdraw()` and `withdrawAsFeeManager()` in `Entropy.sol` hardcode the ETH recipient as `msg.sender` with no option to redirect funds to an alternative address. If a registered provider or its designated fee manager is a smart contract that lacks a `receive()` or `fallback()` function, every withdrawal attempt reverts and the provider's entire accrued fee balance is permanently frozen inside the Entropy contract.

---

### Finding Description

**`withdraw()` — provider self-withdrawal:**

```solidity
// Entropy.sol line 150-173
function withdraw(uint128 amount) public override {
    EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[msg.sender];
    require(providerInfo.accruedFeesInWei >= amount, "Insufficient balance");
    providerInfo.accruedFeesInWei -= amount;

    (bool sent, ) = msg.sender.call{value: amount}("");   // ← hardcoded recipient
    require(sent, "withdrawal to msg.sender failed");
    ...
}
``` [1](#0-0) 

**`withdrawAsFeeManager()` — fee-manager withdrawal:**

```solidity
// Entropy.sol line 175-209
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    ...
    providerInfo.accruedFeesInWei -= amount;

    (bool sent, ) = msg.sender.call{value: amount}("");   // ← hardcoded recipient
    require(sent, "withdrawal to msg.sender failed");
    ...
}
``` [2](#0-1) 

Neither function accepts a `recipient` parameter. The ETH is unconditionally pushed to `msg.sender`.

**`setFeeManager()` is only a partial mitigation:**

A provider can designate a fee manager via `setFeeManager()`:

```solidity
// Entropy.sol line 876-893
function setFeeManager(address manager) external override { ... }
``` [3](#0-2) 

However, `withdrawAsFeeManager()` also sends to `msg.sender` (the fee manager), not to the provider. If the fee manager is itself a non-payable contract, the same freeze applies. Furthermore, a provider contract that does not embed a `setFeeManager` call in its own logic cannot use this escape hatch at all.

---

### Impact Explanation

A provider's entire `accruedFeesInWei` balance is permanently locked in the Entropy contract with no recovery path. There is no admin override, no alternative withdrawal route, and no governance mechanism to rescue the funds. The loss is total and irreversible for the affected provider.

---

### Likelihood Explanation

Entropy providers are not required to be EOAs. Any smart contract can call `register()` and become a provider. A contract that acts as a provider but omits a `receive()` / `fallback()` function — a common oversight in minimal proxy patterns, multisig wrappers, or purpose-built keeper contracts — will trigger this freeze the first time it attempts to withdraw. The probability is low but non-zero and grows as the provider ecosystem expands to include more contract-based operators.

---

### Recommendation

Add an explicit `recipient` parameter to both withdrawal functions, guarded so that only the authorized caller can direct funds:

```diff
- function withdraw(uint128 amount) public override {
+ function withdraw(uint128 amount, address payable recipient) public override {
      ...
-     (bool sent, ) = msg.sender.call{value: amount}("");
+     (bool sent, ) = recipient.call{value: amount}("");
      require(sent, "withdrawal failed");
  }

- function withdrawAsFeeManager(address provider, uint128 amount) external override {
+ function withdrawAsFeeManager(address provider, uint128 amount, address payable recipient) external override {
      ...
-     (bool sent, ) = msg.sender.call{value: amount}("");
+     (bool sent, ) = recipient.call{value: amount}("");
      require(sent, "withdrawal failed");
  }
```

The authorization check (`msg.sender == provider` / `msg.sender == feeManager`) is unchanged; only the destination of the ETH becomes caller-specified.

---

### Proof of Concept

1. Deploy a provider contract **without** a `receive()` or `fallback()` function.
2. Call `register(feeInWei, commitment, commitmentMetadata, chainLength, uri)` from that contract — it is now a registered Entropy provider.
3. Users call `requestWithCallback(providerAddress, ...)` paying fees; `providerInfo.accruedFeesInWei` accumulates.
4. The provider contract calls `withdraw(accruedAmount)`.
5. The EVM executes `msg.sender.call{value: accruedAmount}("")` targeting the non-payable contract.
6. The inner call reverts (no receive function); `sent == false`.
7. `require(sent, "withdrawal to msg.sender failed")` reverts the entire transaction.
8. `accruedFeesInWei` is restored (CEI pattern), but the provider has no alternative path to retrieve the ETH — it is permanently frozen. [4](#0-3) [5](#0-4)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer
        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "withdrawal to msg.sender failed");

        emit EntropyEvents.Withdrawal(provider, msg.sender, amount);
        emit EntropyEventsV2.Withdrawal(
            provider,
            msg.sender,
            amount,
            bytes("")
        );
    }
