### Title
`remove_expired` Does Not Cascade-Remove Descendants, Leaving Child Transactions Permanently Stuck in Pool After Parent Expiry — (File: tx-pool/src/pool.rs)

---

### Summary

`remove_expired` in `tx-pool/src/pool.rs` removes each expired transaction individually via `pool_map.remove_entry()`, which does **not** cascade to descendants. When a parent transaction expires, its child transactions remain in the pool in an uncommittable state: their inputs are still registered in `edges.inputs` as consumed, yet the parent outputs they depend on no longer exist in the pool. This is the direct CKB analog of the gauge-removal bug: a resource (parent transaction) is removed while dependent state (child transactions and their consumed-cell registrations) is not cleaned up, leaving users unable to reclaim the on-chain cells committed as inputs to those children.

---

### Finding Description

`remove_expired` collects all entries whose `timestamp + expiry < now_ms` and removes them one by one:

```rust
// tx-pool/src/pool.rs  lines 271-288
pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
    let now_ms = ckb_systemtime::unix_time_as_millis();
    let removed: Vec<_> = self
        .pool_map
        .iter()
        .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
        .map(|entry| entry.inner.clone())
        .collect();

    for entry in removed {
        self.pool_map.remove_entry(&entry.proposal_short_id()); // ← no cascade
        let reject = Reject::Expiry(entry.timestamp);
        callbacks.call_reject(self, &entry, reject);
    }
}
```

`remove_entry` cleans up only the removed transaction's own edges:

```rust
// tx-pool/src/component/pool_map.rs  lines 642-653
fn remove_entry_edges(&mut self, entry: &TxEntry) {
    for i in entry.transaction().input_pts_iter() {
        self.edges.remove_input(&i);          // frees parent's own inputs
    }
    let id = entry.proposal_short_id();
    for d in entry.related_dep_out_points().cloned() {
        self.edges.delete_txid_by_dep(d, &id);
    }
    self.edges.header_deps.remove(&id);
}
```

It does **not** touch the `edges.inputs` entries belonging to the children, which still map `parent_output_X → child_tx_id`. The children remain in `pool_map.entries` with their own `edges.inputs` registrations intact.

By contrast, `limit_size` (size-based eviction) correctly cascades:

```rust
// tx-pool/src/pool.rs  lines 306-307
if let Some(id) = next_evict_entry() {
    let removed = self.pool_map.remove_entry_and_descendants(&id); // ← cascade
```

The existing unit test at `tx-pool/src/component/tests/score_key.rs` lines 157–168 explicitly confirms that after `remove_entry(&tx1_id)`, `tx2` and `tx3` (descendants) remain in the pool:

```rust
map.remove_entry(&tx1_id);
assert!(!map.contains_key(&tx1_id));
assert!(map.contains_key(&tx2_id));   // child still present
assert!(map.contains_key(&tx3_id));   // grandchild still present
```

**Concrete scenario:**

1. Alice submits `parent_tx` at time `T1` (spending on-chain cell `C`).
2. Alice submits `child_tx` at time `T2 = T1 + Δ` (spending `parent_tx`'s output, plus an additional on-chain cell `D`).
3. At `T1 + expiry`, `remove_expired` removes `parent_tx` but **not** `child_tx`.
4. `child_tx` remains in the pool until `T2 + expiry = T1 + Δ + expiry`.
5. During the window `[T1+expiry, T1+Δ+expiry]`:
   - `child_tx` can never be committed (its parent input no longer exists in the pool or on-chain).
   - `edges.inputs` still maps `cell_D → child_tx_id`, so any new transaction trying to spend `D` is rejected as a double-spend.
   - Alice cannot reclaim cell `D` without either waiting for `child_tx` to expire or using the local-only `remove_transaction` RPC.

---

### Impact Explanation

**Cell locking (direct analog):** Any on-chain cell used as an input to a child transaction remains marked as consumed in `edges.inputs` for up to a full additional `expiry` period after the parent expires. A remote tx submitter (relay peer) has no way to unblock those cells; only a local node operator can call `remove_transaction`. This matches the original report's impact: a user's committed resource (on-chain cell capacity) is locked and cannot be reclaimed.

**Pool bloat / DoS:** An attacker submits a cheap parent transaction at time `T1`, then submits many child transactions at time `T1 + expiry - ε`. The parent expires at `T1 + expiry`; the children expire at `T1 + 2·expiry - ε`. The children occupy pool space for nearly a full extra `expiry` period, potentially filling the pool and blocking legitimate transactions from admission.

---

### Likelihood Explanation

- Any unprivileged tx-pool submitter — RPC caller (`send_transaction`) or relay peer — can trigger this by submitting a parent followed by children with a later timestamp.
- `remove_expired` is called on every block via `_update_tx_pool_for_reorg` (line 1110 of `tx-pool/src/process.rs`), making this a regularly exercised code path.
- No special privileges, keys, or majority hashpower are required.

---

### Recommendation

Replace `remove_entry` with `remove_entry_and_descendants` inside `remove_expired`, mirroring the pattern used in `limit_size`:

```rust
// tx-pool/src/pool.rs
for entry in removed {
    let tx_hash = entry.transaction().hash();
    debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
-   self.pool_map.remove_entry(&entry.proposal_short_id());
+   let evicted = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
+   for evicted_entry in evicted {
+       let reject = Reject::Expiry(evicted_entry.timestamp);
+       callbacks.call_reject(self, &evicted_entry, reject);
+   }
-   let reject = Reject::Expiry(entry.timestamp);
-   callbacks.call_reject(self, &entry, reject);
}
```

This ensures that when a parent transaction expires, all of its descendants are also removed and their `edges.inputs` registrations are freed, allowing users to immediately reuse the on-chain cells that were committed as inputs to those descendants.

---

### Proof of Concept

```
1. Submit parent_tx (inputs: [on_chain_cell_C], outputs: [pool_cell_P]) at time T1.
2. Submit child_tx  (inputs: [pool_cell_P, on_chain_cell_D])           at time T2 = T1 + Δ.
3. Advance time to T1 + expiry.
   → remove_expired fires, removes parent_tx via remove_entry (no cascade).
   → child_tx remains in pool_map.entries.
   → edges.inputs still contains: pool_cell_P → child_tx_id
                                   on_chain_cell_D → child_tx_id
4. Attempt to submit new_tx spending on_chain_cell_D:
   → rejected: "Dead" / double-spend (cell_D is still "consumed" by child_tx).
5. child_tx cannot be committed (parent_tx output pool_cell_P does not exist).
6. child_tx remains stuck until T2 + expiry = T1 + Δ + expiry.
   on_chain_cell_D is inaccessible for the entire window [T1+expiry, T1+Δ+expiry].

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
