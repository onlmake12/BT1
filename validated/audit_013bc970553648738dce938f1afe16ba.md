Audit Report

## Title
Unauthenticated Arbitrary Peer Forwarding via Attacker-Controlled `route` in `ConnectionRequestDelivered::execute()` — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary
`ConnectionRequestDeliveredProcess::execute()` forwards the message to whatever `PeerId` appears last in the attacker-supplied `route` field with no verification that this route was established by a prior `ConnectionRequest`. The `forward_rate_limiter` key `(content.from, content.to, msg_item_id)` is entirely attacker-controlled, making it trivially bypassable. Any peer connected to a relay can use it as an unconditional message forwarder to any other peer connected to that relay.

## Finding Description
In `connection_request_delivered.rs`, `execute()` parses `route` directly from the wire at L43–50 with no cross-reference to any prior `ConnectionRequest` state. The only guards applied are: route length ≤ `MAX_HOPS` (L130–132), `listen_addrs` count 1–24 (L125–128), and a `forward_rate_limiter` keyed by `(content.from, content.to, self.msg_item_id)` (L134–145) — all three values are attacker-controlled fields in the message.

At L147–148, `content.route.last()` is used directly as the forwarding target with no check that this peer was ever part of an original `ConnectionRequest` route:

```rust
match content.route.last() {
    Some(next_peer_id) => self.forward_delivered(next_peer_id).await,
```

`forward_delivered()` (L182–212) looks up the attacker-supplied `peer_id` in the live `peer_registry` and unconditionally sends the message to it. The `inflight_requests` map is only consulted in the terminal delivery branch (L160) — when `route` is empty and `self_peer_id == content.from` — never during forwarding.

The `forward_rate_limiter` (defined in `mod.rs` L254–257 at 1 req/sec per `(PeerId, PeerId, u32)`) is fully bypassable because all three key components are attacker-controlled message fields. The session-level limiter (L95–107, 30 req/sec per `(session_id, msg.item_id())`) can be multiplied by opening additional sessions.

The `forward_delivered()` helper in `component/mod.rs` (L223–241) strips the last element from `route` before forwarding, so the forwarded message arrives at the target peer with a shorter route — but the forwarding itself is unconditional.

## Impact Explanation
This is a bad design that allows any unprivileged peer to use a relay node as an unconditional message forwarder to any other peer connected to that relay. With the forward rate limiter fully bypassable via `from`/`to` rotation and the session limiter multiplied by opening additional connections, an attacker can sustain a high-rate flood of `ConnectionRequestDelivered` messages at targeted peers. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — the attacker's cost is only maintaining standard P2P connections, while the relay amplifies delivery to targeted peers and processes forwarding overhead itself.

## Likelihood Explanation
Requires only a standard P2P connection to the relay — no privilege, no key, no PoW. The target peer's `PeerId` is discoverable via the discovery protocol. The `from`/`to` rotation to bypass the forward limiter requires no coordination. The attack is fully local-testable. Repeatability is high.

## Recommendation
Before forwarding a `ConnectionRequestDelivered`, the relay must verify that the sender's session `PeerId` equals `route.last()` (i.e., the message is coming back from the peer that was last in the original forward path). Additionally, a per-relay forwarding state table should be maintained — analogous to `inflight_requests` — keyed by `(from, to)`, recording which `ConnectionRequest` messages this node actually forwarded, and consulted before forwarding any `ConnectionRequestDelivered`. The `forward_rate_limiter` key should incorporate the sender's authenticated session identity rather than attacker-controlled message fields.

## Proof of Concept
1. Attacker A connects to relay R via the HolePunching protocol.
2. A discovers that peer P (`PeerId = P_id`) is connected to R via the discovery protocol.
3. A sends: `ConnectionRequestDelivered { from: random_id_1, to: random_id_2, route: [P_id], sync_route: [], listen_addrs: [valid_multiaddr] }`
4. R's `execute()` hits `content.route.last() == Some(P_id)`, calls `forward_delivered(P_id)`.
5. R looks up `P_id` in `peer_registry`, finds the session, strips `P_id` from route, and sends the message to P.
6. P receives the unsolicited `ConnectionRequestDelivered { route: [], from: random_id_1, to: random_id_2, ... }` and processes it.
7. Repeat with rotated `from`/`to` values to bypass the `forward_rate_limiter`; open additional sessions to multiply the 30 req/sec session cap.
8. To verify: write a unit test constructing a `ConnectionRequestDeliveredProcess` with a mock `HolePunching` state containing no `inflight_requests` entry for `random_id_2`, confirm `forward_delivered` is called with `P_id` and returns `Status::ok()`.
