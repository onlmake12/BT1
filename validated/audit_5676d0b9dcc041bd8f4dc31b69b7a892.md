### Title
No Refund of Gas-Proportional Fee When Callback Fails in Echo - (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, users pay a fee upfront that includes a gas-proportional component (`callbackGasLimit * feePerGasInWei`). When `executeCallback` is called, the full fee is credited to the provider **before** the callback is attempted. If the callback reverts (for any reason, including early revert or out-of-gas), the provider retains the entire fee — including the gas component for gas that was never consumed. The user receives no refund.

---

### Finding Description

`Echo.getFee` computes the total fee as:

```
baseFee + providerBaseFee + providerFeedFee + callbackGasLimit * feePerGasInWei
``` [1](#0-0) 

The `callbackGasLimit * feePerGasInWei` component is a gas-proportional charge meant to compensate the provider for executing the callback up to `callbackGasLimit` gas.

In `executeCallback`, the full `req.fee` is credited to the provider and the request is cleared **before** the callback is invoked:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

clearRequest(sequenceNumber);
``` [2](#0-1) 

Only after this does the callback execute:

```solidity
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{
    emitPriceUpdate(...);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(...);
} catch {
    emit PriceUpdateCallbackFailed(...);
}
``` [3](#0-2) 

If the callback reverts at any point — even after consuming only a tiny fraction of `callbackGasLimit` — the provider keeps the full gas-proportional fee. There is no mechanism to refund the unused gas fee to the requester.

---

### Impact Explanation

A user who sets `callbackGasLimit = 1,000,000` and whose callback reverts after consuming only 10,000 gas still loses `1,000,000 * feePerGasInWei` in fees. The provider is paid for 990,000 gas units that were never consumed. For high gas limits (which are explicitly supported and encouraged for complex callbacks), this loss is non-trivial. [4](#0-3) 

The `req.fee` stored at request time already encodes the full gas-proportional amount, and there is no post-callback adjustment path.

---

### Likelihood Explanation

Callbacks fail regularly in practice: consumer contract bugs, state changes between request and fulfillment, insufficient gas passed by the provider, or deliberate reverts. The `callbackGasLimit` parameter is user-controlled and can be set to large values (the interface explicitly supports up to `uint16.max * 10000` gas). Any callback failure results in a loss proportional to the unused gas. [5](#0-4) 

---

### Recommendation

Track actual gas consumed during the callback and refund the unused gas-fee component to the requester. One approach:

1. Record `gasleft()` before the callback call.
2. After the `try/catch`, compute `gasUsed = gasLimitBefore - gasleft()`.
3. If the callback failed, compute `refundAmount = (callbackGasLimit - gasUsed) * feePerGasInWei` and transfer it back to `req.requester`, reducing the provider's accrued fee accordingly.

Alternatively, credit the provider only the gas-proportional fee for gas actually consumed, and refund the remainder to the requester regardless of success or failure.

---

### Proof of Concept

1. Provider registers with `feePerGasInWei = 10 gwei`.
2. User calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 500_000`. Fee paid includes `500_000 * 10 gwei = 5_000_000 gwei = 0.005 ETH` as the gas component.
3. User's `echoCallback` reverts at line 5 of 1000, consuming only ~3,000 gas.
4. Provider calls `executeCallback`. The full `req.fee` (including the 0.005 ETH gas component) is credited to the provider at line 161–162 of `Echo.sol`.
5. The callback reverts; `PriceUpdateCallbackFailed` is emitted.
6. The user has lost `(500_000 - 3_000) * 10 gwei ≈ 0.00497 ETH` for gas that was never consumed. [6](#0-5) [3](#0-2)
