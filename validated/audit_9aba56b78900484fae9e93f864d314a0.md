Audit Report

## Title
Stale Pre-Eviction Snapshot Overwrites Live `total_tx_size`/`total_tx_cycles` Counters in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, new pool-size totals are captured into local variables before a conditional eviction step runs. When `check_and_record_ancestors` evicts transactions via `remove_entry_and_descendants` (which calls `update_stat_for_remove_tx` to correctly decrement `self.total_tx_size`/`self.total_tx_cycles`), the stale pre-eviction snapshot is then unconditionally written back at lines 218–219, silently cancelling every decrement. Repeated exploitation inflates the counters arbitrarily, causing `limit_size` to continuously evict legitimate transactions and reject all new submissions with `Reject::Full`, effectively disabling the mempool.

## Finding Description
`add_entry` (lines 200–221) executes this sequence:

1. **Lines 210–211**: `updated_stat_for_add_tx` takes `&self` (immutable borrow), computes `self.total_tx_size + new_size` and `self.total_tx_cycles + new_cycles` as local values, and returns them — it does **not** write to `self`. [1](#0-0) 

2. **Line 213**: `check_and_record_ancestors` takes `&mut self`. When `ancestors_count > max_ancestors_count` AND `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (line 603), it enters the eviction branch and calls `remove_entry_and_descendants` in a loop (line 618). This internally calls `update_stat_for_remove_tx`, which **does** write to `self.total_tx_size` and `self.total_tx_cycles`. [2](#0-1) [3](#0-2) 

3. **Lines 218–219**: The stale local snapshot is unconditionally written back, overwriting the correctly-decremented live values. [4](#0-3) 

Concrete arithmetic: if `self.total_tx_size = 100`, a new tx of size 10 is added (snapshot = 110), then an eviction of size 20 correctly sets `self.total_tx_size = 80`, but line 218 writes back 110. The counter is inflated by 20 (the evicted amount) per call. The eviction branch is reachable when `ancestors_count > max_ancestors_count` AND `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, a condition fully controllable by an unprivileged sender. [5](#0-4) 

## Impact Explanation
`total_tx_size` is the sole guard in `limit_size` (pool.rs line 298). [6](#0-5) 

An inflated counter causes `limit_size` to believe the pool is over capacity, triggering spurious evictions of legitimate fee-paying transactions. Repeated exploitation drives the counter to `max_tx_pool_size` while the actual pool is nearly empty, causing the node to continuously reject all new transaction submissions with `Reject::Full` and evict its own contents — effectively disabling the mempool. This matches the **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The trigger is fully under attacker control with no privileged access required. An unprivileged sender can craft transactions where a base transaction's output is used as a cell dep by many pool transactions, then submit a new transaction that pushes the ancestor count just over `max_ancestors_count`. The eviction branch fires, the stale snapshot is written back, and the inflation accumulates. The attack path is the standard `send_transaction` RPC / P2P relay path. The pattern can be repeated with fresh transactions to accumulate unbounded inflation.

## Recommendation
Compute the new totals **after** all evictions have completed. Replace the pre-eviction snapshot pattern with a post-eviction addition:

```rust
// Validate capacity BEFORE eviction (read-only check, no snapshot stored)
self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Apply addition AFTER evictions have already decremented the counters
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

This ensures the addition is always applied on top of the post-eviction state.

## Proof of Concept
**Setup:** `max_ancestors_count = 25`, `max_tx_pool_size = 1_000_000` bytes.

1. Submit base transaction `T0` creating output `O`.
2. Submit 24 transactions `T1…T24`, each referencing `O` as a cell dep (~1 000 bytes each). Pool has 25 entries, `total_tx_size ≈ 25_000`.
3. Submit `Tnew` (1 000 bytes) spending an output of `T0`. Its ancestor set includes `T1…T24` via cell-dep linkage → `ancestors_count = 26 > 25`. `cell_ref_parents = {T1…T24}`, so `26 - 24 = 2 ≤ 25` — eviction branch taken.
4. `check_and_record_ancestors` evicts e.g. `T1`, `T2` (2 × 1 000 bytes). `update_stat_for_remove_tx` correctly sets `self.total_tx_size = 23_000`.
5. Line 218 writes back stale snapshot: `self.total_tx_size = 26_000` (should be `24_000`). Inflation: +2 000 bytes per call.
6. Repeat ~490 times. `total_tx_size` reaches `≈ 1_000_000` while actual pool is nearly empty.
7. `limit_size` fires on every subsequent `add_entry`, evicting legitimate transactions and returning `Reject::Full` to all new submissions — mempool disabled.
