Audit Report

## Title
`remove_entry_and_descendants()` Pre-Removes Links Before `update_ancestors_index_key()` Runs, Leaving Ancestor Entries With Stale Descendant Weights and Eviction Keys - (`tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::remove_entry_and_descendants()` removes all link entries for the entire subtree in a first pass, then calls `remove_entry()` for each. Because `remove_entry()` relies on `self.links.calc_ancestors()` to find which remaining entries need their descendant weights decremented, and those links are already gone, ancestor entries that remain in the pool are never updated. Their `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` fields stay inflated, and their `evict_key` is never refreshed, causing incorrect eviction ordering for the lifetime of those ancestor entries.

## Finding Description

`remove_entry_and_descendants` collects the root and all its descendants, then calls `remove_entry_links` for every entry in that set before calling `remove_entry` on any of them: [1](#0-0) 

`remove_entry_links(B)` removes B's own link entry entirely (`self.links.remove(id)`) and also removes B from A's children set and from C's parents set: [2](#0-1) 

When `remove_entry(B)` is subsequently called, `update_ancestors_index_key` tries to resolve B's ancestors via `self.links.calc_ancestors(&B.proposal_short_id())`: [3](#0-2) 

`calc_ancestors` calls `calc_relative_ids`, which does `self.inner.get(short_id)` on B's ID. Since B's link entry was already removed, this returns `None`, and `unwrap_or_default()` yields an empty set: [4](#0-3) 

The loop over ancestors is therefore empty. `sub_descendant_weight` is never called on A, so A's `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` remain at their pre-removal values. The `evict_key` update (`e.evict_key = e.inner.as_evict_key()`) is also skipped: [5](#0-4) 

The `EvictKey` is built directly from these stale descendant fields: [6](#0-5) 

`next_evict_entry` iterates the pool ordered by `evict_key`, so A's inflated key causes it to rank as more valuable than it actually is: [7](#0-6) 

By contrast, a direct `remove_entry(B)` call (single-entry removal) works correctly because B's link entry is still present when `update_ancestors_index_key` runs, so `calc_ancestors` returns `{A}` and A's weights are properly decremented. The divergence is intentional for the descendants side (the comment on line 256 says "so that we won't update_descendants_index_key in remove_entry"), but it inadvertently also suppresses the ancestor-side update. [8](#0-7) 

## Impact Explanation

This is a suboptimal implementation of the CKB tx-pool state storage mechanism. After `remove_entry_and_descendants(B)` in a chain `A → B → C`, entry A retains inflated `descendants_*` fields and an inflated `evict_key`. The pool's eviction logic (`limit_size` → `next_evict_entry`) will rank A as higher-value than it truly is, causing genuinely low-fee transactions to persist in the pool longer than they should. The stale state is permanent for A until A itself is removed or the pool is cleared. This maps to: **Medium (2001–10000 points) — Suboptimal implementation of CKB state storage mechanism.**

## Likelihood Explanation

Any code path that calls `remove_entry_and_descendants` on a transaction that has an in-pool ancestor triggers this. Reachable paths include `resolve_conflict` (double-spend committed), `resolve_conflict_header_dep`, `check_and_record_ancestors`, and `limit_size`. An unprivileged user can trigger this by submitting `A` (low fee) → `B` (high fee) → `C` (high fee), then submitting a conflicting transaction that causes `B` to be committed or double-spent, triggering `remove_entry_and_descendants(B)`. No special privileges are required. The condition (ancestor still in pool when a descendant subtree is removed) is common in normal pool operation. [9](#0-8) 

## Recommendation

Before removing any links in `remove_entry_and_descendants`, update the ancestors of the root entry while links are still intact. Specifically, call `update_ancestors_index_key(root_entry, EntryOp::Remove)` for the root (and only the root, since its removal accounts for the entire subtree weight via the accumulated `descendants_*` fields) before the link-removal loop. Alternatively, accumulate the total size/cycles/fee of all entries in `removed_ids` and apply a single `sub_descendant_weight`-equivalent update to each ancestor of the root after computing them while links are intact.

## Proof of Concept

1. Add `tx1` (low fee, e.g. 1 shannon/byte) to the pool as a pending entry.
2. Add `tx2` (high fee) spending `tx1`'s output.
3. Add `tx3` (high fee) spending `tx2`'s output.
4. Assert `tx1.descendants_count == 3`, `tx1.descendants_fee == fee1 + fee2 + fee3`.
5. Call `pool_map.remove_entry_and_descendants(&tx2.proposal_short_id())`.
6. Assert `tx1` is still in the pool.
7. Assert `tx1.descendants_count` — it will still be `3` (or `2` if only B+C counted), not `1`.
8. Assert `tx1.evict_key` reflects the inflated `descendants_feerate`.
9. Compare with calling `pool_map.remove_entry(&tx2.proposal_short_id())` directly (with `tx3` already removed): `tx1.descendants_count` will correctly be `1`.

This can be implemented as a unit test in `tx-pool/src/component/tests/score_key.rs` following the pattern of the existing `test_remove_entry` test. [10](#0-9)

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

#[test]
fn test_remove_entry() {
    let mut map = PoolMap::new(DEFAULT_MAX_ANCESTORS_COUNT);
    let tx1 = TxEntry::dummy_resolve(
        TransactionBuilder::default().build(),
        100,
        Capacity::shannons(100),
        100,
    );
    let tx2 = TxEntry::dummy_resolve(
        TransactionBuilder::default()
            .input(
                CellInput::new_builder()
                    .previous_output(
                        OutPoint::new_builder()
                            .tx_hash(tx1.transaction().hash())
                            .index(0u32)
                            .build(),
                    )
                    .build(),
            )
            .witness(Bytes::new())
            .build(),
        200,
        Capacity::shannons(200),
        200,
    );
    let tx3 = TxEntry::dummy_resolve(
        TransactionBuilder::default()
            .input(
                CellInput::new_builder()
                    .previous_output(
                        OutPoint::new_builder()
                            .tx_hash(tx2.transaction().hash())
                            .index(0u32)
                            .build(),
                    )
                    .build(),
            )
            .witness(Bytes::new())
            .build(),
        200,
        Capacity::shannons(200),
        200,
    );
    let tx1_id = tx1.proposal_short_id();
    let tx2_id = tx2.proposal_short_id();
    let tx3_id = tx3.proposal_short_id();
    map.add_proposed(tx1).unwrap();
    map.add_proposed(tx2).unwrap();
    map.add_proposed(tx3).unwrap();
    let descendants_set = map.calc_descendants(&tx1_id);
    assert!(descendants_set.contains(&tx2_id));
    assert!(descendants_set.contains(&tx3_id));

    let tx3_entry = map.get(&tx3_id);
    assert!(tx3_entry.is_some());
    let tx3_entry = tx3_entry.unwrap();
    assert_eq!(tx3_entry.ancestors_count, 3);

    map.remove_entry(&tx1_id);
    assert!(!map.contains_key(&tx1_id));
    assert!(map.contains_key(&tx2_id));
    assert!(map.contains_key(&tx3_id));

    let tx3_entry = map.get(&tx3_id).unwrap();
    assert_eq!(tx3_entry.ancestors_count, 2);
    assert_eq!(
        map.calc_ancestors(&tx3_id),
        vec![tx2_id].into_iter().collect()
    );
}
