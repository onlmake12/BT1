### Title
`MAX_DEPOSIT_LIMIT` Check in `addFunds` Applies to Single Deposit Amount Rather Than Cumulative Balance, Allowing Permanent Subscription Cap Bypass — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `addFunds` function in `Scheduler.sol` enforces `MAX_DEPOSIT_LIMIT` (100 ETH) against `msg.value` (the single deposit amount) rather than against the resulting cumulative balance `status.balanceInWei + msg.value`. Any unprivileged caller can bypass the cap entirely by making multiple deposits each ≤ 100 ETH. Because permanent subscriptions prohibit withdrawals, bypassed funds are permanently locked. A secondary deadlock arises when `minimumBalance > MAX_DEPOSIT_LIMIT`: no single deposit can simultaneously satisfy both the minimum-balance requirement and the deposit cap, making the subscription permanently un-fundable.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
``` [1](#0-0) 

`addFunds` enforces this limit only against the incoming `msg.value`:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [2](#0-1) 

The same per-transaction check appears in `createSubscription`:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [3](#0-2) 

`addFunds` has **no access control** — any address may call it for any subscription ID:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
``` [4](#0-3) 

Withdrawals from permanent subscriptions are unconditionally blocked:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [5](#0-4) 

`getMinimumBalance` returns `numPriceFeeds × minimumBalancePerFeed`:

```solidity
return uint256(numPriceFeeds) * this.getMinimumBalancePerFeed();
``` [6](#0-5) 

`MAX_PRICE_IDS_PER_SUBSCRIPTION = 255`, so with `minimumBalancePerFeed ≥ 0.393 ETH` the minimum balance for a max-feed subscription exceeds 100 ETH, triggering the deadlock:

```solidity
uint8 public constant MAX_PRICE_IDS_PER_SUBSCRIPTION = 255;
``` [7](#0-6) 

---

### Impact Explanation

**Issue 1 — Cap bypass (no privilege required):**  
Any caller can invoke `addFunds` N times with `msg.value = MAX_DEPOSIT_LIMIT` each time, accumulating `N × 100 ETH` in a permanent subscription. Because withdrawals are blocked for permanent subscriptions, all deposited ETH is permanently locked. The intended safety ceiling of 100 ETH is completely ineffective.

**Issue 2 — Permanent subscription deadlock:**  
When `minimumBalance > MAX_DEPOSIT_LIMIT` (e.g., 255 feeds × 0.4 ETH/feed = 102 ETH), the subscription owner faces an impossible constraint:
- Any deposit ≥ `minimumBalance` (102 ETH) exceeds `MAX_DEPOSIT_LIMIT` → `MaxDepositLimitExceeded` revert.
- Any deposit ≤ `MAX_DEPOSIT_LIMIT` (100 ETH) leaves `balanceInWei < minimumBalance` → `InsufficientBalance` revert.

The subscription is permanently stuck: it cannot receive price updates (balance too low) and cannot be topped up (cap blocks the required amount). Since permanent subscriptions cannot be deactivated or updated, there is no recovery path.

---

### Likelihood Explanation

- **Issue 1**: Likelihood is **high**. `addFunds` is permissionless and callable by any EOA or contract. No special conditions are required beyond knowing a permanent subscription ID (which is discoverable via `getActiveSubscriptions`).
- **Issue 2**: Likelihood is **medium**. It requires `minimumBalancePerFeed` to be set such that a large subscription's minimum balance exceeds 100 ETH. This is plausible on high-gas-price chains or after an admin fee adjustment, and the subscription owner has no recourse once it occurs.

---

### Recommendation

In `addFunds`, replace the per-transaction check with a cumulative balance check:

```solidity
// Before (incorrect — checks only the single deposit):
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;

// After (correct — checks the resulting total balance):
status.balanceInWei += msg.value;
if (params.isPermanent && status.balanceInWei > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Additionally, ensure `MAX_DEPOSIT_LIMIT` is always set strictly above the maximum possible `minimumBalance` (i.e., `MAX_PRICE_IDS_PER_SUBSCRIPTION × minimumBalancePerFeed`), or enforce this invariant in the admin setter for `minimumBalancePerFeed`.

---

### Proof of Concept

**Cap bypass (Issue 1):**

```solidity
// Attacker calls addFunds 10 times with 100 ETH each
for (uint i = 0; i < 10; i++) {
    scheduler.addFunds{value: 100 ether}(permanentSubscriptionId);
}
// Result: permanentSubscription.balanceInWei == 1000 ETH
// Withdrawal is blocked → 1000 ETH permanently locked
```

**Deadlock (Issue 2):**

```
Setup:
  minimumBalancePerFeed = 0.4 ETH
  numFeeds = 255
  minimumBalance = 255 × 0.4 = 102 ETH
  MAX_DEPOSIT_LIMIT = 100 ETH

Owner calls addFunds{value: 102 ETH}:
  → 102 ETH > MAX_DEPOSIT_LIMIT → MaxDepositLimitExceeded ✗

Owner calls addFunds{value: 100 ETH}:
  → balanceInWei (0 + 100) < minimumBalance (102) → InsufficientBalance ✗

No valid deposit amount exists. Subscription is permanently inoperable.
```
