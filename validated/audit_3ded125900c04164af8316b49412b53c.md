### Title
Backup Replica Proceeds to Submit `reveal_with_callback` on Any RPC Error from `get_request_v2` — (`apps/fortuna/src/keeper/process_event.rs`)

### Summary

In `process_event_with_backoff`, the backup replica's guard check calls `get_request_v2` to determine whether the primary replica already fulfilled a request. If that RPC call returns **any** error — including transient network errors, rate-limit responses, or provider timeouts — the `Err(e)` branch at line 87 logs a warning and falls through, treating the situation identically to "request still open." The backup replica then unconditionally proceeds to call `reveal_with_callback` on a request that may already have been fulfilled.

### Finding Description

The multi-replica failover logic in `process_event_with_backoff` is designed to prevent duplicate `reveal_with_callback` submissions:

```rust
match chain_state
    .contract
    .get_request_v2(event.provider_address, event.sequence_number)
    .await
{
    Ok(Some(req)) => { /* check callback_status, maybe skip */ }
    Ok(None)      => { return Ok(()); }   // already fulfilled → skip
    Err(e)        => {
        tracing::warn!(...);              // ← falls through, proceeds anyway
    }
}
// backup replica submits reveal_with_callback regardless
``` [1](#0-0) 

The two safe branches (`Ok(None)` and `Ok(Some(req))` with non-`CallbackNotStarted` status) both return early. The `Err` branch does not. Any transient RPC failure — rate-limit (HTTP 429), connection reset, provider timeout — causes the backup replica to behave as if the request is still open and submit a `reveal_with_callback` transaction.

### Impact Explanation

When the primary replica has already fulfilled the request and the backup replica's `get_request_v2` call fails transiently:

1. The backup replica calls `reveal_with_callback` for an already-cleared request.
2. Gas estimation reverts with `NoSuchRequest` (the contract's `findActiveRequest` reverts because the request slot was cleared).
3. The `error_mapper` in `submit_tx_with_backoff` recognises `NoSuchRequest` and schedules slow retries: 5 s → 10 s → 60 s → 60 s → 60 s, then returns a permanent error after 5 retries. [2](#0-1) 

4. The backup replica's async task is occupied for up to ~5 minutes per affected event.
5. The `request_failovers_triggered` metric is incremented spuriously, polluting observability.
6. Under a sustained RPC degradation (e.g., rate-limiting during high load), every event processed by the backup replica during that window triggers this path, multiplying the wasted retry cycles.

The contract itself prevents double-fulfillment (no user funds are at risk), but the provider's keeper service is disrupted: processing capacity is consumed by futile retries, and the backup replica's intended failover purpose is undermined precisely when the network is already stressed. [3](#0-2) 

### Likelihood Explanation

The `backup_delay_seconds` window (default 30 s, recommended 10–30 s) is long enough that transient RPC errors are a realistic occurrence in production. Rate-limiting, brief provider outages, and connection resets are common on public EVM RPC endpoints, especially during periods of high chain activity — exactly when Entropy request volume is also elevated. No attacker action is required; the trigger is ordinary infrastructure noise. [4](#0-3) 

### Recommendation

Treat an `Err` result from `get_request_v2` as an **uncertain** state, not as "request is open." The backup replica should either:

- **Return an error** and let the caller retry the entire `process_event_with_backoff` call after a delay, or
- **Retry `get_request_v2`** with its own backoff before deciding to proceed.

Proceeding to submit a transaction when the existence of the request is unknown conflates two distinct error conditions ("not found" vs. "RPC unavailable"), mirroring the exact class of bug described in the reference report.

### Proof of Concept

1. Deploy Fortuna with `replica_config` (2 replicas, `backup_delay_seconds: 15`).
2. User submits an Entropy request with sequence number `N` where `N % 2 == 0` (assigned to replica 0).
3. Replica 0 (primary) successfully calls `reveal_with_callback`; the contract clears the request.
4. Replica 1 (backup) wakes after 15 s and calls `get_request_v2(provider, N)`.
5. Simulate a transient RPC error at this point (e.g., inject a 429 response from the RPC provider).
6. The `Err(e)` branch is taken; replica 1 proceeds to call `reveal_with_callback(provider, N, ...)`.
7. Gas estimation reverts with `NoSuchRequest`; `submit_tx_with_backoff` retries 5 times over ~195 s before giving up.
8. Replica 1's processing task is blocked for ~5 minutes on a request that was already fulfilled. [5](#0-4) [6](#0-5)

    let error_mapper = |num_retries, e| {
        if let backoff::Error::Transient {
            err: SubmitTxError::GasUsageEstimateError(tx, ContractError::Revert(revert)),
            ..
        } = &e
        {
            if let Ok(PythRandomErrorsErrors::NoSuchRequest(_)) =
                PythRandomErrorsErrors::decode(revert)
            {
                let err = SubmitTxError::GasUsageEstimateError(
                    tx.clone(),
                    ContractError::Revert(revert.clone()),
                );
                // Slow down the retries if the request is not found.
                // This probably means that the request is already fulfilled via another process.
                // After 5 retries, we return the error permanently.
                if num_retries >= 5 {
                    return backoff::Error::Permanent(err);
                }
                let retry_after_seconds = match num_retries {
                    0 => 5,
                    1 => 10,
                    _ => 60,
                };
                return backoff::Error::Transient {
                    err,
                    retry_after: Some(Duration::from_secs(retry_after_seconds)),
                };
            }
        }
        e
    };
