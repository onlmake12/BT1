### Title
Fee Permanently Consumed With No Price State Transition When `cached_timestamp == update_timestamp` — (`target_chains/sui/contracts/sources/pyth.move`)

### Summary
`update_single_price_feed` deposits the caller's SUI fee into the `PriceInfoObject` **before** performing the freshness check. When the incoming VAA carries a price timestamp equal to the already-cached timestamp, `is_fresh_update` returns `false` and the price state is silently left unchanged, but the fee has already been irrecoverably deposited. No privileged access is required; any relayer can trigger this with a valid, correctly-signed, equal-timestamp VAA.

### Finding Description
In `pyth::pyth::update_single_price_feed` (`pyth.move` lines 262–296):

```
// line 274 – fee check passes
assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);

// line 277 – fee is deposited UNCONDITIONALLY
price_info::deposit_fee_coins(price_info_object, fee);

// lines 283-291 – matching entry found, update_cache called
update_cache(latest_only, cur_price_info, price_info_object, clock);
```

Inside `update_cache` (lines 306–323):

```
if (is_fresh_update(update, price_info_object)){   // line 316
    // state write only happens here
    price_info::update_price_info_object(price_info_object, update);
}
// if NOT fresh → silent no-op, fee already gone
```

`is_fresh_update` (line 337) uses strict `>`:

```
update_timestamp > cached_timestamp   // equal timestamps → false
```

`deposit_fee_coins` in `price_info.move` (lines 105–116) is `public` (not `public(friend)`) and performs no freshness pre-check. Once coins are joined into the dynamic field, there is no withdrawal function anywhere in the module.

### Impact Explanation
The caller's SUI fee is permanently locked inside the `PriceInfoObject`. There is no refund, no withdrawal, and no governance path to recover it. Every call to `update_single_price_feed` with an equal-timestamp VAA silently drains the caller's wallet with zero protocol benefit. Over time, accumulated fees become permanently inaccessible.

### Likelihood Explanation
This is trivially reachable in normal operation:
- Two relayers race to submit the same VAA; the second one always hits the equal-timestamp case.
- A relayer retries a failed transaction with the same VAA.
- A malicious actor front-runs a legitimate update with the identical VAA, then the legitimate caller loses their fee.

No privileged role, leaked key, or oracle manipulation is required. The VAA only needs to be valid and correctly signed by Wormhole guardians — which is the normal precondition for any price update.

### Recommendation
Move `deposit_fee_coins` to **after** the freshness check, or refund the fee if `is_fresh_update` returns `false`. The corrected order in `update_single_price_feed` should be:

1. Assert fee sufficiency.
2. Find matching price info.
3. Check `is_fresh_update`; abort or return fee if stale.
4. Only then call `deposit_fee_coins`.

Alternatively, add a refund branch inside `update_cache` that returns the fee to the caller when no state transition occurs.
