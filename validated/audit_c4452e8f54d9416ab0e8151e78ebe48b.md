### Title
Staking Protocol `freeze` Flag Blocks User Withdrawals — (`governance/pyth_staking_sdk/src/idl/staking.json`)

---

### Summary

The Pyth staking program's `GlobalConfig` contains a `freeze: bool` field. When set, it triggers a `Frozen` error (code 6017). The `withdraw_stake` instruction includes `config` as a required account, indicating the freeze check is applied to withdrawals. This is a direct analog to the `sDaiStrategy` pause-blocks-withdrawal vulnerability: a governance-triggered freeze prevents users from withdrawing their staked PYTH tokens.

---

### Finding Description

The `GlobalConfig` struct in the staking program contains a `freeze` boolean field: [1](#0-0) 

The program defines a `Frozen` error: [2](#0-1) 

The `withdraw_stake` instruction requires the `config` PDA as an account: [3](#0-2) 

The `withdrawTokensFromStakeAccountCustody` client method calls `withdraw_stake` with the `config` account, meaning any on-chain freeze check in that instruction will block user withdrawals: [4](#0-3) 

**Limitation:** The Rust source files for the staking program are not indexed in this repository (glob searches for `**/staking/programs/**/*.rs` returned no results). The exact line that checks `config.freeze` in `withdraw_stake` cannot be directly cited. The finding is inferred from: (1) the `freeze` field in `GlobalConfig`, (2) the `Frozen` error existing in the IDL, and (3) `config` being a required account in `withdraw_stake`. If the freeze check is absent from `withdraw_stake`, this finding does not apply.

---

### Impact Explanation

If `freeze = true` is set in `GlobalConfig` by the `governance_authority`, users calling `withdraw_stake` will receive a `Frozen` error and be unable to retrieve their staked PYTH tokens from the custody account. This is identical in class to the `sDaiStrategy` issue: a protocol-level pause/freeze that should only halt new deposits or staking operations instead also blocks user exits. Users with positions in cooldown or with unlocked balances cannot withdraw even though no active staking risk exists for those funds.

**Impact: Medium** — user funds are temporarily locked; not permanently lost, but inaccessible during the freeze period.

---

### Likelihood Explanation

The `governance_authority` can update `GlobalConfig` to set `freeze = true`. This is a legitimate emergency action (e.g., in response to an exploit). The freeze is expected to be transient, but during the freeze window all withdrawals are blocked. This mirrors the `sDaiStrategy` scenario exactly.

**Likelihood: Medium** — freeze is an infrequent but real operational event.

---

### Recommendation

The `withdraw_stake` instruction should not check `config.freeze`. The freeze guard should only apply to deposit and staking operations (`create_position`, `deposit_tokens`), not to withdrawals. Users must always be able to exit, even at reduced value, to service obligations elsewhere.

---

### Proof of Concept

1. Governance authority calls the config-update instruction with `freeze = true` on `GlobalConfig`.
2. User calls `withdraw_stake` (via `withdrawTokensFromStakeAccountCustody`) with their `stakeAccountPositions`.
3. The instruction loads `config`, checks `config.freeze`, and reverts with `Frozen` (error 6017).
4. User cannot retrieve tokens from `stakeAccountCustody` until governance unfreezes the protocol. [1](#0-0) [2](#0-1)
