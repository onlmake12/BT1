### Title
Unbounded `fetch_all` Memory Exhaustion via Default `request_limit=usize::MAX` in Rich-Indexer `get_cells`/`get_transactions` — (`util/rich-indexer/src/service.rs`, `get_cells.rs`)

---

### Summary

When `request_limit` is absent from `IndexerConfig` (the shipped default), `RichIndexerService::new` sets `request_limit = usize::MAX`. The guard in `get_cells` and `get_transactions` compares `limit as usize > self.request_limit`, which is always `false` for any `u32` value on a 64-bit host. An unprivileged RPC caller can therefore pass `limit = u32::MAX`, causing the SQL backend to execute `LIMIT 4294967295` and `SQLXPool::fetch_all` to materialize every matching row into a single heap-allocated `Vec<AnyRow>`, exhausting node memory.

---

### Finding Description

**Step 1 — Default produces `usize::MAX`.**

`IndexerConfig::default()` ships with `request_limit: None`. [1](#0-0) 

`RichIndexerService::new` converts that `None` to `usize::MAX`: [2](#0-1) 

**Step 2 — Guard is permanently bypassed.**

In `get_cells`, the only size check is:

```rust
let limit = limit.value();   // Uint32 → u32, max = 4_294_967_295
if limit as usize > self.request_limit { … }   // usize::MAX on 64-bit
``` [3](#0-2) 

On any 64-bit host `u32::MAX as usize = 4_294_967_295 < usize::MAX = 18_446_744_073_709_551_615`, so the error branch is unreachable. The identical guard exists in `get_transactions`. [4](#0-3) 

**Step 3 — Raw `u32::MAX` is forwarded to SQL.**

```rust
query_builder.limit(limit);   // LIMIT 4294967295
``` [5](#0-4) 

**Step 4 — `fetch_all` materializes the entire result set.**

```rust
let cells = self.store.fetch_all(query).await …
    .collect::<Vec<_>>();
``` [6](#0-5) 

`SQLXPool::fetch_all` calls sqlx's `fetch_all`, which buffers every row in a single `Vec<AnyRow>` before returning. [7](#0-6) 

---

### Impact Explanation

CKB mainnet currently holds tens of millions of live cells. Each `AnyRow` carries raw binary blobs (lock/type scripts, cell data, hashes). A single `get_cells` call with `limit=u32::MAX` and an empty-prefix `args` search key matches every live cell and forces the process to allocate several gigabytes in one shot, crashing the node with OOM. The same path exists in `get_transactions` via `get_tx_with_cell` / `get_tx_with_cells`.

---

### Likelihood Explanation

- `request_limit` is `None` in `IndexerConfig::default()` — no explicit operator action is required to be vulnerable; the out-of-the-box configuration is affected.
- Rich-indexer nodes are routinely exposed publicly for dApp developers.
- The RPC call requires no authentication, no PoW, and no privileged key.
- A single HTTP POST is sufficient; the attack is trivially repeatable.

---

### Recommendation

1. **Change the default**: replace `unwrap_or(usize::MAX)` with a safe constant (e.g., `unwrap_or(DEFAULT_MAX_RESULTS)` where `DEFAULT_MAX_RESULTS = 1000` or similar), matching the existing `ckb-indexer` convention.
2. **Enforce an absolute hard cap**: regardless of operator config, reject any `limit` above a compile-time maximum (e.g., `10_000`).
3. **Use streaming instead of `fetch_all`**: replace `SQLXPool::fetch_all` with the existing `SQLXPool::fetch` (which uses `try_next` row-by-row) so memory is bounded by the page size even if the limit guard is misconfigured.

---

### Proof of Concept

```
# Precondition: rich-indexer enabled, no request_limit in ckb.toml, DB populated
curl -X POST http://<node>:8114 \
  -H 'Content-Type: application/json' \
  -d '{
    "id": 1, "jsonrpc": "2.0", "method": "get_cells",
    "params": [
      {"script": {"code_hash": "0x0000...0000", "hash_type": "data", "args": "0x"},
       "script_type": "lock"},
      "asc",
      "0xffffffff",
      null
    ]
  }'
# Expected: node OOM-kills or returns after exhausting heap
```

The empty `args` prefix with `Prefix` search mode matches every script, so every live cell is returned. With `limit = 0xffffffff` and `request_limit = usize::MAX`, the guard at `get_cells.rs:29` never fires, `LIMIT 4294967295` is sent to SQLite/Postgres, and `fetch_all` allocates the full result set in one `Vec`.
