### Title
Missing Per-Peer Rate Limit on `GetBlockFilters` Enables Bandwidth Amplification — (`sync/src/filter/get_block_filters_process.rs`)

### Summary

The `BlockFilter` protocol handler has no rate limiter. Any unauthenticated peer can send `GetBlockFilters(start_number=0)` at high frequency, causing the node to perform up to 1000 RocksDB reads and send a ~1.8 MB response per message, with no per-peer throttle.

### Finding Description

`GetBlockFiltersProcess::execute` iterates up to `BATCH_SIZE=1000` times, calling `active_chain.get_block_hash` (snapshot read) and `active_chain.get_block_filter` (live `ChainDB` read) per iteration, then serializes and sends a response capped at 1.8 MB. [1](#0-0) [2](#0-1) 

The 1.8 MB cap was introduced in PR #4972 solely to avoid tentacle frame-size disconnects, not to limit request frequency. [3](#0-2) 

The `BlockFilter` protocol handler (`sync/src/filter/mod.rs`) dispatches directly to `GetBlockFiltersProcess::execute` with **no rate-limiting check**: [4](#0-3) 

By contrast, the `Relayer` protocol explicitly instantiates a `RateLimiter<(PeerIndex, u32)>` at 30 req/sec per peer per message type and checks it before every dispatch: [5](#0-4) 

The `BlockFilter` struct has no such field and no equivalent check anywhere in its handler path.

### Impact Explanation

Each `GetBlockFilters` request from a single peer:
- Triggers up to 1000 `ChainDB` reads (two per block: hash lookup + filter lookup)
- Produces a response up to ~1.8 MB

A peer sending this in a tight loop (limited only by TCP round-trip time) can sustain continuous outbound bandwidth consumption and sustained RocksDB read pressure from a single unauthenticated connection. Multiple connections multiply the effect linearly.

### Likelihood Explanation

The `GetBlockFilters` message is a fixed 8-byte struct (`start_number: Uint64`), trivially cheap to generate. The Filter protocol (`/ckb/filter`, protocol ID 121) is registered and open to any connecting peer. [6](#0-5) [7](#0-6) 

The attack requires no authentication, no PoW, no stake, and no special knowledge — just a TCP connection to the Filter protocol port.

### Recommendation

Add a `RateLimiter<(PeerIndex, u32)>` to `BlockFilter` (mirroring the `Relayer` implementation) and check it at the top of `try_process` before dispatching `GetBlockFilters`, `GetBlockFilterHashes`, and `GetBlockFilterCheckPoints`. A quota of 1–5 req/sec per peer per message type is sufficient for legitimate light-client sync while eliminating the amplification window.

### Proof of Concept

1. Run a CKB full node with block filters enabled and ≥ 1000 blocks with filters built.
2. Connect a peer to `SupportProtocols::Filter`.
3. In a tight loop, send `packed::GetBlockFilters { start_number: 0 }` wrapped in `BlockFilterMessage`.
4. Measure bytes received per second on the attacker side — each response is up to ~1.8 MB.
5. Observe that the node's outbound bandwidth is saturated and RocksDB read I/O spikes, with no disconnect or throttle applied to the sending peer.

    pub fn new(chain: ChainController, shared: Arc<SyncShared>) -> Self {
        // setup a rate limiter keyed by peer and message type that lets through 30 requests per second
        // current max rps is 10 (ASK_FOR_TXS_TOKEN / TX_PROPOSAL_TOKEN), 30 is a flexible hard cap with buffer
        let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
        let rate_limiter = RateLimiter::hashmap(quota);

        Relayer {
            chain,
            shared,
            rate_limiter,
        }
    }

    /// Get shared state
    pub fn shared(&self) -> &Arc<SyncShared> {
        &self.shared
    }

    async fn try_process(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        message: packed::RelayMessageUnionReader<'_>,
    ) -> Status {
        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =
            !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));

        if should_check_rate
            && self
                .rate_limiter
                .check_key(&(peer, message.item_id()))
                .is_err()
        {
            return StatusCode::TooManyRequests.with_context(message.item_name());
        }
