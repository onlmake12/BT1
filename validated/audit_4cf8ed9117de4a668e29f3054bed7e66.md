### Title
Unauthenticated `content.from` Spoofing Bypasses `forward_rate_limiter` in Hole Punching Protocol — (`network/src/protocols/hole_punching/component/connection_request.rs`)

### Summary

The `HolePunching` protocol uses two rate limiters: a per-session `rate_limiter` keyed on the actual sending peer (`session_id`), and a `forward_rate_limiter` keyed on `(content.from, content.to, msg_item_id)` extracted from the message payload. Because `content.from` is never verified against the actual sending peer's identity, any connected peer can forge the `from` field to consume the `forward_rate_limiter` quota for an arbitrary `(victim_A, victim_C)` pair, permanently blocking legitimate hole-punching between those two peers.

### Finding Description

The `HolePunching` struct in `network/src/protocols/hole_punching/mod.rs` maintains two rate limiters:

- `rate_limiter: RateLimiter<(PeerIndex, u32)>` — keyed by the **actual** sending session and message type (30 req/s).
- `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` — keyed by `(content.from, content.to, msg_item_id)` (1 req/s). [1](#0-0) 

In `HolePunching::received()`, the per-session `rate_limiter` is checked first using the authenticated `session_id`: [2](#0-1) 

Then `ConnectionRequestProcess::execute()` checks the `forward_rate_limiter` using `content.from` and `content.to`, which are **deserialized directly from the attacker-controlled message payload** with no verification that `content.from` matches the actual sending peer: [3](#0-2) 

The `from` field is parsed from raw bytes in the message: [4](#0-3) 

The actual sending peer identity (`self.peer: PeerIndex`) is available in `ConnectionRequestProcess` but is **never compared** to `content.from`. The same unauthenticated pattern exists in `ConnectionRequestDeliveredProcess` and `ConnectionSyncProcess`: [5](#0-4) [6](#0-5) 

**Exploit path:**

1. Attacker controls peer B, connected to relay node R.
2. Attacker wants to block peer A from hole-punching to peer C.
3. Attacker sends `ConnectionRequest { from=A_peer_id, to=C_peer_id, ... }` from session B.
4. `rate_limiter.check_key(&(B_session_id, 0))` passes — B has not hit its 30/s per-session cap.
5. `forward_rate_limiter.check_key(&(A_peer_id, C_peer_id, 0))` passes and **consumes** the 1/s quota for `(A, C, ConnectionRequest)`.
6. When legitimate peer A sends `ConnectionRequest { from=A, to=C }`, the `forward_rate_limiter` returns `TooManyRequests` and the request is silently dropped.
7. Attacker repeats step 3 once per second to permanently suppress all hole-punching from A to C.

The `forward_rate_limiter` quota is 1 per second per `(from, to, msg_type)` tuple: [7](#0-6) 

The message schema confirms `from` and `to` are plain `Bytes` with no cryptographic binding to the sending session: [8](#0-7) 

### Impact Explanation

An unprivileged peer with a single connection to any relay node can permanently suppress NAT traversal (`ConnectionRequest`, `ConnectionRequestDelivered`, `ConnectionSync`) between any two targeted peers. The victim peers will never receive `TooManyRequests` feedback — the relay node silently drops their legitimate requests. This degrades the P2P network's ability to establish outbound connections for NAT-ed nodes, reducing connectivity and potentially isolating targeted peers from the network.

### Likelihood Explanation

Any peer connected to the CKB P2P network can execute this attack with a single TCP connection. No privileged access, no key material, and no Sybil capability is required. The attacker only needs to know the `PeerId` values of the two target peers (which are publicly advertised via the discovery protocol). The cost is one spoofed message per second per targeted pair.

### Recommendation

Before checking the `forward_rate_limiter`, verify that `content.from` matches the actual sending peer's `PeerId`. The `session_id` can be resolved to a `PeerId` via the peer registry:

```rust
// In ConnectionRequestProcess::execute(), after parsing content:
let actual_peer_id = self.protocol.network_state
    .with_peer_registry(|reg| {
        reg.get_peer(self.peer)
            .and_then(|p| extract_peer_id(&p.connected_addr))
    });

if actual_peer_id.as_ref() != Some(&content.from) {
    return StatusCode::InvalidFromPeerId
        .with_context("from field does not match actual sender");
}
```

This ensures the `forward_rate_limiter` key `(content.from, content.to, msg_item_id)` can only be consumed by the peer that actually owns `content.from`, directly mirroring the fix for the `SettV3` issue (checking the `from` address rather than `msg.sender`).

### Proof of Concept

```
Attacker peer B (session_id=42) connects to relay node R.

Loop every 950ms:
  Send HolePunchingMessage::ConnectionRequest {
    from: <A's PeerId bytes>,   // spoofed — not B's actual PeerId
    to:   <C's PeerId bytes>,
    max_hops: 6,
    route: [],
    listen_addrs: [<any valid TCP multiaddr with A's PeerId>],
  }

Effect on relay node R:
  - rate_limiter[(42, 0)] = 1/30 per second → passes
  - forward_rate_limiter[(A, C, 0)] = 1/1 per second → consumed

When legitimate peer A sends ConnectionRequest{from=A, to=C}:
  - rate_limiter[(A_session, 0)] passes
  - forward_rate_limiter[(A, C, 0)] → TooManyRequests → silently dropped

Peer A never establishes a hole-punched connection to C.
``` [9](#0-8)

    pub(crate) async fn execute(mut self) -> Status {
        let content = match RequestContent::try_from(&self.message) {
            Ok(content) => content,
            Err(status) => return status,
        };
        if content.listen_addrs.len() > ADDRS_COUNT_LIMIT || content.listen_addrs.is_empty() {
            return StatusCode::InvalidListenAddrLen
                .with_context("the listen address count is too large or empty");
        }

        if content.max_hops > MAX_HOPS {
            return StatusCode::InvalidMaxTTL.into();
        }
        if content.route.len() > MAX_HOPS as usize {
            return StatusCode::InvalidRoute.with_context("the route length is too long");
        }

        let self_peer_id = self.protocol.network_state.local_peer_id();
        if content.route.contains(self_peer_id) {
            return StatusCode::Ignore.with_context("the message is passed, ignore it");
        }

        if self
            .protocol
            .forward_rate_limiter
            .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
            .is_err()
        {
            debug!(
                "from: {}, to {}, item_name: {}, rate limit is reached",
                content.from, content.to, "ConnectionRequest",
            );
            return StatusCode::TooManyRequests.with_context("ConnectionRequest");
        }

        if self_peer_id == &content.to {
            self.respond_delivered(content.from, &content.to, content.listen_addrs)
                .await
        } else if content.max_hops == 0u8 {
            StatusCode::ReachedMaxHops.into()
        } else {
            self.forward_message(self_peer_id, &content.to).await
        }
    }
