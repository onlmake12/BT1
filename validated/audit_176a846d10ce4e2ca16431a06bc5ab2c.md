### Title
Executor's `ownerEmitterAddress` Cannot Be Updated After Governance Data Source Transfer, Permanently Bricking Governance Execution — (`target_chains/ethereum/contracts/contracts/executor/Executor.sol`)

---

### Summary

The `Executor.sol` contract stores `ownerEmitterChainId` and `ownerEmitterAddress` at initialization with no mechanism to update them. The Pyth governance system supports changing the governance data source via `AuthorizeGovernanceDataSourceTransfer` in `PythGovernance.sol`. After such a transfer, the Executor's stored emitter reference becomes permanently stale, causing every subsequent call to `execute()` to revert with `UnauthorizedEmitter`. This bricks all governance execution on the affected chain.

---

### Finding Description

`Executor.sol` initializes two immutable-in-practice fields:

```solidity
uint16 private ownerEmitterChainId;
bytes32 private ownerEmitterAddress;
```

These are set once in `_initialize` and never again. The `verifyGovernanceVM` function enforces them on every call:

```solidity
if (
    vm.emitterChainId != ownerEmitterChainId ||
    vm.emitterAddress != ownerEmitterAddress
) revert ExecutorErrors.UnauthorizedEmitter();
```

The codebase itself acknowledges the missing setter with a TODO comment inside the `ExecutorAction` enum:

```solidity
// TODO: add an instruction to change the governance data source.
Execute // 0
```

Meanwhile, `PythGovernance.sol` fully supports rotating the governance data source via `AuthorizeGovernanceDataSourceTransfer`, which updates the emitter chain ID and emitter address stored in the Pyth price feed contract. When this governance action is executed, the Pyth price feed contract is updated, but the `Executor` contract retains the old `ownerEmitterAddress`. All subsequent governance VAAs signed by the new emitter are rejected by the Executor.

---

### Impact Explanation

After a legitimate governance data source transfer:

- Every call to `Executor.execute()` reverts with `UnauthorizedEmitter`.
- No governance instructions can be executed on the affected chain through this Executor.
- All contracts that rely on the Executor for governance (e.g., `EntropyUpgradable`, `SchedulerUpgradeable`) lose their governance path permanently — fee changes, admin transfers, upgrades, and withdrawals all become impossible.
- The Executor itself cannot be self-repaired because the only repair path (`execute()`) is the function that is now bricked.

This is a permanent, chain-level governance DoS.

---

### Likelihood Explanation

The Pyth governance system explicitly supports governance data source rotation (`AuthorizeGovernanceDataSourceTransfer`). Key rotations or governance contract upgrades on Pythnet are realistic operational events. The TODO comment in the Executor source confirms the developers are aware of the gap. Any legitimate governance rotation that updates the Pyth price feed contract's emitter without a corresponding Executor update triggers the issue.

---

### Recommendation

Add a governance-controlled setter to `Executor.sol` that allows updating `ownerEmitterChainId` and `ownerEmitterAddress`, analogous to how `PythGovernance.sol` handles `AuthorizeGovernanceDataSourceTransfer`. The new `ExecutorAction` enum value (already noted in the TODO) should be implemented and the `verifyGovernanceVM` path should be bootstrapped carefully to avoid a chicken-and-egg problem (e.g., by requiring the update VAA to be signed by both the old and new emitter, or by using a two-step transfer pattern).

---

### Proof of Concept

1. Governance emits an `AuthorizeGovernanceDataSourceTransfer` VAA targeting the Pyth price feed contract, rotating the governance emitter from `OLD_EMITTER` to `NEW_EMITTER`.
2. `PythGovernance.executeGovernanceInstruction()` processes the VAA and updates `_state.governanceDataSource` to `NEW_EMITTER`.
3. `Executor._initialize()` was called at deploy time with `ownerEmitterAddress = OLD_EMITTER`. No setter exists.
4. Governance now sends a new instruction VAA signed by `NEW_EMITTER` to `Executor.execute()`.
5. `verifyGovernanceVM` checks `vm.emitterAddress != ownerEmitterAddress` → `NEW_EMITTER != OLD_EMITTER` → reverts `UnauthorizedEmitter`.
6. All governance execution on this chain is permanently bricked.

    function AuthorizeGovernanceDataSourceTransfer(
        AuthorizeGovernanceDataSourceTransferPayload memory payload
    ) internal {
        PythInternalStructs.DataSource
            memory oldGovernanceDatSource = governanceDataSource();

        // Make sure the claimVaa is a valid VAA with RequestGovernanceDataSourceTransfer governance message
        // If it's valid then its emitter can take over the governance from the current emitter.
        // The VAA is checked here to ensure that the new governance data source is valid and can send message
        // through wormhole.
        (IWormhole.VM memory vm, bool valid, ) = wormhole().parseAndVerifyVM(
            payload.claimVaa
        );
        if (!valid) revert PythErrors.InvalidWormholeVaa();

        GovernanceInstruction memory gi = parseGovernanceInstruction(
            vm.payload
        );
        if (gi.targetChainId != chainId() && gi.targetChainId != 0)
            revert PythErrors.InvalidGovernanceTarget();

        if (gi.action != GovernanceAction.RequestGovernanceDataSourceTransfer)
            revert PythErrors.InvalidGovernanceMessage();

        RequestGovernanceDataSourceTransferPayload
            memory claimPayload = parseRequestGovernanceDataSourceTransferPayload(
                gi.payload
            );

        // Governance data source index is used to prevent replay attacks, so a claimVaa cannot be used twice.
        if (
            governanceDataSourceIndex() >=
            claimPayload.governanceDataSourceIndex
        ) revert PythErrors.OldGovernanceMessage();

        setGovernanceDataSourceIndex(claimPayload.governanceDataSourceIndex);
