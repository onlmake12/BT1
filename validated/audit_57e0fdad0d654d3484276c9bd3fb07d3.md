Audit Report

## Title
Unbounded Per-Peer RocksDB Read Amplification in `BlockFilter` Due to Missing Per-Peer Rate Limiter — (`sync/src/filter/mod.rs`)

## Summary
The `BlockFilter` protocol handler contains no per-peer rate limiter, while the structurally equivalent `Relayer` handler has an explicit `governor::RateLimiter<(PeerIndex, u32)>` capped at 30 req/s. Any unprivileged remote peer can send `GetBlockFilterHashes` or `GetBlockFilterCheckPoints` messages without restriction, each triggering up to 4,000 RocksDB reads (2 reads × 2,000 loop iterations). This asymmetry allows a low-cost, parallelizable I/O amplification attack that can degrade block-import and sync performance.

## Finding Description

**Missing rate limiter in `BlockFilter`.** The struct holds only `shared: Arc<SyncShared>` with no `rate_limiter` field: [1](#0-0) 

The `received` handler parses the message and immediately calls `self.process()` → `self.try_process()` with no rate check, dispatching directly to the handler: [2](#0-1) 

**Contrast with `Relayer`**, which declares a `RateLimiter<(PeerIndex, u32)>` at 30 req/s: [3](#0-2) 

And checks it at the top of `try_process` before any handler runs: [4](#0-3) 

**Per-message DB read amplification.** `BATCH_SIZE = 2000` and the loop runs up to 2,000 iterations in `GetBlockFilterHashesProcess::execute`: [5](#0-4) [6](#0-5) 

Each iteration issues two RocksDB reads:
- `active_chain.get_block_hash(block_number)` → `self.get(COLUMN_INDEX, …)`: [7](#0-6) 
- `active_chain.get_block_filter_hash(&block_hash)` → `self.get(COLUMN_BLOCK_FILTER_HASH, …)`: [8](#0-7) 

Total: **up to 4,000 RocksDB reads per message** (plus 2 more for the parent lookup when `start_number > 0`, lines 40–47).

`GetBlockFilterCheckPointsProcess` has the same `BATCH_SIZE = 2000` and the same missing rate limiter, compounding the attack surface: [9](#0-8) [10](#0-9) 

## Impact Explanation
An attacker connecting N peers and sending `GetBlockFilterHashes(start_number=0)` at maximum TCP/yamux rate forces up to 4,000 RocksDB reads per message with no floor. The aggregate I/O and CPU load scales linearly with N, starving the chain-controller thread that processes new blocks and the sync protocol that services other peers. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The attack requires only a standard CKB P2P connection — no PoW, no stake, no privileged key. The `Filter` protocol is advertised in the node's supported-protocols list and any peer can negotiate it. The message payload is tiny (8 bytes for `start_number`). The asymmetry with `Relayer` (rate-limited since the `governor` integration) indicates an oversight rather than an intentional design choice. The attack is repeatable, stateless, and trivially parallelizable.

## Recommendation
Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` mirroring the `Relayer` implementation, and check it at the top of `try_process` before dispatching to any handler. A quota of 1–2 req/s per peer per message type is sufficient for legitimate light-client sync while eliminating the amplification vector. Apply the same fix to `GetBlockFilterCheckPointsProcess` and `GetBlockFiltersProcess`.

## Proof of Concept
1. Start a CKB mainnet node with block filters enabled (≥ 2,000 blocks of filter data built).
2. Connect N peers (e.g. N = 10) each running:
   ```
   loop { send GetBlockFilterHashes { start_number: 0 } }
   ```
3. Monitor RocksDB metrics (`rocksdb.block.cache.miss`, `rocksdb.number.keys.read`) and block-import latency.
4. Assert: read IOPS grows proportionally to N; block-import latency degrades with no floor; no peer is disconnected or rate-limited by the node.

The fix can be verified by adding the same `rate_limiter.check_key(&(peer, message.item_id()))` guard used in `Relayer::try_process` and confirming IOPS plateaus at the configured quota regardless of N.
