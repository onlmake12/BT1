### Title
Witness-Stripping Front-Run Blocks Legitimate Transactions via `ProposalShortId` Collision in Tx-Pool Deduplication — (`tx-pool/src/util.rs`)

### Summary

CKB's tx-pool deduplicates transactions using `ProposalShortId`, which is derived from the first 10 bytes of `tx_hash`. Critically, `tx_hash` is computed **without witnesses**. An unprivileged attacker who observes a pending transaction can strip or replace its witnesses, submit the witness-stripped version to the same node first, and cause the legitimate transaction to be rejected as a duplicate. The attacker's invalid transaction eventually fails contextual (script) verification and is evicted, but the attack can be repeated indefinitely at low cost, constituting a sustained griefing DoS against any transaction sender.

---

### Finding Description

**Root cause — `tx-pool/src/util.rs`, `check_txid_collision`:** [1](#0-0) 

The function checks for duplicates using `tx.proposal_short_id()`, which is the first 10 bytes of `tx_hash`. [2](#0-1) 

`tx_hash` is computed from the transaction body **excluding witnesses**. Two transactions that share the same inputs, outputs, cell deps, and header deps — but carry different witnesses — produce the **identical `tx_hash`** and therefore the **identical `ProposalShortId`**.

This is explicitly demonstrated in the integration test helper: [3](#0-2) 

`tx1` and `tx2` have `tx1.hash() == tx2.hash()` and `tx1.proposal_short_id() == tx2.proposal_short_id()` while `tx1.witness_hash() != tx2.witness_hash()`.

**The misleading comment in `pre_check`:** [4](#0-3) 

The comment states "Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc." — this is incorrect. Same `txid`/`ProposalShortId` does **not** imply same witnesses.

**The same collision check is also applied in the verify-queue path:** [5](#0-4) 

`verify_queue_contains` also keys on `proposal_short_id()`, so a witness-stripped transaction sitting in the async verify queue blocks the legitimate transaction from even entering the queue.

**The pool map enforces uniqueness by `ProposalShortId`:** [6](#0-5) 

`PoolEntry.id` is declared `hashed_unique`, so only one entry per `ProposalShortId` can exist at any time.

**`add_entry` silently returns `Ok((false, …))` on collision:** [7](#0-6) 

---

### Impact Explanation

An attacker can:

1. Observe Alice's pending transaction `tx_A` (valid witnesses, valid lock-script signatures) via the P2P relay or public mempool.
2. Construct `tx_E` = same inputs/outputs/deps as `tx_A`, but with **empty or garbage witnesses**. `tx_E.hash() == tx_A.hash()`, so `tx_E.proposal_short_id() == tx_A.proposal_short_id()`.
3. Submit `tx_E` to the target node via `send_transaction` RPC or P2P relay before `tx_A` arrives.
4. `tx_E` passes `non_contextual_verify` (structure is valid; witnesses are not script-executed at this stage) and enters the verify queue.
5. When `tx_A` arrives, `verify_queue_contains` or `check_txid_collision` returns `true` → `tx_A` is rejected with `PoolRejectedDuplicatedTransaction`.
6. `tx_E` eventually fails contextual script verification (invalid/missing witness) and is evicted.
7. Alice resubmits `tx_A`; Eve repeats from step 2.

The attacker gains nothing except ownership of a failed tx slot, but can sustain the DoS indefinitely at the cost of repeated fee-paying submissions. Alice's transaction is permanently prevented from being confirmed as long as the attacker maintains the attack.

---

### Likelihood Explanation

- **Entry path**: Fully unprivileged. Any peer connected to the target node, or any RPC caller, can submit transactions.
- **Observation**: CKB transactions propagate via P2P relay; an attacker running a node can observe pending transactions before they are confirmed.
- **Timing**: The attacker only needs to reach the target node before the victim's transaction is processed — feasible when the attacker is a direct peer.
- **Cost**: Each attack round costs one transaction fee (for `tx_E`). No cryptographic work is required; the attacker simply copies the victim's inputs/outputs and clears the witnesses.
- **Repeatability**: The attack loop has no natural termination; it can run indefinitely.

Likelihood is **medium** (requires network positioning and timing, but no privileged access or cryptographic capability).

---

### Recommendation

1. **Key the deduplication check on `witness_hash` (or full transaction identity) rather than `tx_hash`/`ProposalShortId`** when deciding whether to reject an incoming transaction as a duplicate. Two transactions with the same `tx_hash` but different witnesses are distinct submissions and should not cause one to block the other.
2. Alternatively, **accept the later submission if it has a valid witness and the earlier one is still in the verify queue awaiting script execution**, replacing the pending-verification slot.
3. Fix the misleading comment at `tx-pool/src/process.rs:280` which incorrectly claims same txid implies same witnesses.

---

### Proof of Concept

The existing integration test already proves the collision is reachable: [8](#0-7) 

Attack steps:

```
1. Alice: tx_A = new_transaction_spend_tip_cellbase()   # valid secp256k1 witness
2. Eve:   tx_E = tx_A.as_advanced_builder()
                    .witness(Bytes::default())           # empty/invalid witness
                    .build()
   # tx_A.hash() == tx_E.hash()  ✓
   # tx_A.proposal_short_id() == tx_E.proposal_short_id()  ✓
   # tx_A.witness_hash() != tx_E.witness_hash()  ✓

3. Eve submits tx_E via send_transaction RPC (or P2P relay) to target node.
   → tx_E passes non_contextual_verify (structure valid, no script exec yet)
   → tx_E enters verify_queue keyed by proposal_short_id

4. Alice submits tx_A to same node.
   → verify_queue_contains(&tx_A) == true  (same proposal_short_id as tx_E)
   → tx_A rejected: PoolRejectedDuplicatedTransaction

5. tx_E fails contextual script verification → evicted from pool.

6. Repeat from step 3.
``` [1](#0-0) [5](#0-4)
