### Title
`ema_price_no_older_than` Arithmetic Underflow Causes Revert When `publish_time > current_time` - (File: `target_chains/fuel/contracts/pyth-contract/src/main.sw`)

### Summary

The Fuel/Sway implementation of `ema_price_no_older_than` performs an unchecked unsigned subtraction `current_time - price.publish_time`. When a stored EMA price has a `publish_time` greater than `current_time` (a future-timestamped update), this underflows and panics/reverts in the FuelVM. The sibling function `price_no_older_than` was already patched with saturating subtraction for exactly this reason, but `ema_price_no_older_than` was not.

### Finding Description

In `target_chains/fuel/contracts/pyth-contract/src/main.sw`, the private function `ema_price_no_older_than` computes staleness as:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,  // ← underflows if publish_time > current_time
        PythError::OutdatedPrice,
    );
    price
}
``` [1](#0-0) 

The sibling function `price_no_older_than` was explicitly fixed with a saturating subtraction guard, and even carries a comment documenting the fix:

```sway
fn price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
``` [2](#0-1) 

The developer was clearly aware of the underflow risk for `price_no_older_than` but did not apply the same fix to `ema_price_no_older_than`. In the FuelVM, unsigned integer underflow causes a transaction revert (panic), not a wraparound.

The public-facing entry points that route through the vulnerable function are:

```sway
fn ema_price(price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(valid_time_period(), price_feed_id)
}

fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(time_period, price_feed_id)
}
``` [3](#0-2) 

### Impact Explanation

Any on-chain consumer on Fuel calling `ema_price` or `ema_price_no_older_than` for a price feed whose stored EMA `publish_time` exceeds `current_time` will receive a revert instead of a price. This is a DoS of the EMA price staleness check for the affected price feed ID. Protocols that depend on EMA prices (e.g., for TWAP-based liquidations or risk calculations) will be unable to read prices until the block timestamp catches up to the stored `publish_time`.

### Likelihood Explanation

Pyth price updates originate from Pythnet validators and are relayed by unprivileged updaters. Clock skew between Pythnet and the Fuel chain, or a deliberate submission of an update with a `publish_time` a few seconds in the future (which `update_price_feeds` does not explicitly reject), is a realistic scenario. The `update_price_feeds` function on Fuel does not enforce an upper bound on `publish_time`. [4](#0-3) 

An unprivileged relayer submitting a valid Wormhole-attested price update with a slightly future `publish_time` is the attacker-controlled entry path. No privileged access is required.

### Recommendation

Apply the same saturating subtraction guard used in `price_no_older_than` to `ema_price_no_older_than`:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

### Proof of Concept

1. An unprivileged relayer calls `update_price_feeds` with a Wormhole-attested batch where the EMA `publish_time` is `current_time + 5` (5 seconds in the future).
2. The update is accepted and stored.
3. Any on-chain contract calls `ema_price(price_feed_id)`.
4. Internally, `ema_price_no_older_than(valid_time_period(), price_feed_id)` is called.
5. `current_time - price.publish_time` underflows (e.g., `T - (T+5)` on `u64`), causing a FuelVM panic/revert.
6. The caller's transaction reverts. All EMA price reads for this feed ID are DoS-ed until the block timestamp exceeds `publish_time`.

The contrast with the already-fixed `price_no_older_than` (which returns `time_difference = 0` in this case and correctly passes the staleness check) confirms this is an inconsistency introduced by an incomplete fix. [1](#0-0) [5](#0-4)
