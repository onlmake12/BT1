Audit Report

## Title
MMR `get_root()` I/O Amplification via Crafted `GetLastStateProof` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary

Any peer connected to the light client protocol can send a crafted `GetLastStateProof` message with `last_n_blocks=499` and `difficulties=[]` that passes the size guard while forcing the server to call `chain_root_mmr(n-1).get_root()` 499 times per request, each traversing O(log H) MMR nodes from RocksDB. No rate limiting exists on `LightClientProtocol`, and every valid request returns `Status::ok()` with no peer penalty, making the attack indefinitely repeatable at zero cost.

## Finding Description

**Step 1 — Limit check passes.**
The guard at `execute()` checks `difficulties.len() + (last_n_blocks as usize) * 2 > GET_LAST_STATE_PROOF_LIMIT`. With `difficulties=[]` and `last_n_blocks=499`: `0 + 998 = 998 ≤ 1000` — passes. [1](#0-0) [2](#0-1) 

**Step 2 — `last_n_numbers` is maximally populated.**
When `difficulty_boundary` equals the tip's total difficulty, `get_first_block_total_difficulty_is_not_less_than` resolves to the tip block. The guard at L313 fires, setting `difficulty_boundary_block_number = last_block_number - 499`, and `last_n_numbers` becomes a 499-entry vector `[tip-499, …, tip-1]`. [3](#0-2) 

**Step 3 — `complete_headers` calls `get_root()` once per block.**
For every entry in `block_numbers`, the loop unconditionally constructs a new `ChainRootMMR` and calls `get_root()`. [4](#0-3) 

**Step 4 — Each `get_root()` issues O(log H) RocksDB reads.**
`chain_root_mmr(n)` creates an MMR of size `leaf_index_to_mmr_size(n)`. The `MMRStore` implementation routes every `get_elem(pos)` call to `self.store.get_header_digest(pos)`, which is a direct RocksDB point read from `COLUMN_CHAIN_ROOT_MMR`. For H ≈ 10⁶, this is ~20 reads per `get_root()` call. [5](#0-4) [6](#0-5) [7](#0-6) 

**Step 5 — No rate limiting on `LightClientProtocol`.**
`LightClientProtocol` has only a `shared: Shared` field — no rate limiter. `try_process` dispatches directly to `GetLastStateProofProcess::execute()` with no throttle check. Contrast with `Relayer`, which has a `governor`-based `rate_limiter: RateLimiter<(PeerIndex, u32)>` checked before every handler dispatch. [8](#0-7) [9](#0-8) [10](#0-9) [11](#0-10) 

**Step 6 — Valid requests return `Status::ok()` with no ban.**
`reply_proof` always returns `Status::ok()` on success; `received()` only bans on `status.should_ban()`, which is never triggered by a well-formed request. [12](#0-11) [13](#0-12) 

**Total I/O per request:** 499 × ~20 = ~9,980 RocksDB point reads for MMR roots alone, plus 499 `get_block` calls and uncle hash computations.

## Impact Explanation

This is a **bad design which could cause CKB network congestion with few costs** (High, 10001–15000 points). An attacker maintaining a persistent connection and sending back-to-back crafted requests sustains continuous RocksDB I/O amplification. Because the light client server shares the same RocksDB instance as the main chain processing pipeline, saturating I/O degrades block validation, peer sync, and all other storage-dependent operations on the targeted node. Multiple attackers targeting multiple light client servers simultaneously can degrade the CKB light client network broadly. The cost to the attacker is zero: no PoW, no stake, no privileged role.

## Likelihood Explanation

Any peer that can open a light-client protocol connection can send this message. The crafted parameters (`last_n_blocks=499`, `difficulties=[]`, `difficulty_boundary=tip_total_difficulty`, `last_hash=tip_hash`) are trivially derived from the tip state freely advertised via `GetLastState`. The attack is repeatable indefinitely with no ban risk, as valid requests always return `Status::ok()`. [14](#0-13) 

## Recommendation

1. **Cache the MMR root per request**: compute `chain_root_mmr(tip-1).get_root()` once before the `complete_headers` loop and reuse it for all sampled headers, since all headers in a single response share the same tip MMR.
2. **Pre-store `parent_chain_root`** alongside block headers at write time (in `reconcile_main_chain`), eliminating per-block MMR traversal at read time entirely.
3. **Add per-peer rate limiting** to `LightClientProtocol::try_process` using the same `governor`-based pattern already used in `Relayer`. [15](#0-14) 

## Proof of Concept

Send the following message to a node running the light client protocol server:

```
GetLastStateProof {
    last_n_blocks:        499,
    difficulties:         [],
    last_hash:            <current tip hash>,
    start_hash:           <genesis hash>,
    start_number:         0,
    difficulty_boundary:  <tip total difficulty>
}
```

Limit check: `0 + 499×2 = 998 ≤ 1000` → passes.
Server builds `last_n_numbers = [tip-499, …, tip-1]` (499 entries).
`complete_headers` calls `chain_root_mmr(n-1).get_root()` 499 times, each issuing O(log H) RocksDB reads.

**Instrumentation test**: add a counter to `Snapshot::get_header_digest`; send one such request and observe ~499 × ⌊log₂(tip_height)⌋ calls. Send at maximum rate; observe RocksDB read IOPS scale linearly with request rate × log(H), with no ban applied and `Status::ok()` returned for every request. [6](#0-5) [16](#0-15)
