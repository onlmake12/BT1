### Title
Unprivileged Signer Can Overwrite Any Publisher's Stake Account in Integrity Pool — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `set_publisher_stake_account` instruction in the Oracle Integrity Staking (OIS) integrity pool program accepts a generic, unconstrained `signer` account with no relationship to the `publisher` whose stake account is being modified. Any unprivileged transaction sender can call this instruction for any registered publisher, replacing that publisher's associated stake account with an arbitrary account they control.

---

### Finding Description

The `set_publisher_stake_account` instruction in the integrity pool program (`pyti8TM4zRVBjmarcgAPmTNNAXYKJv7WVHrkrm6woLN`) is defined in the IDL as follows:

```json
{
  "name": "set_publisher_stake_account",
  "accounts": [
    { "name": "signer",    "signer": true },
    { "name": "publisher", "docs": ["CHECK : The publisher will be checked against data in the pool_data"] },
    { "name": "pool_data", "relations": ["pool_config"], "writable": true },
    { "name": "pool_config", "pda": { "seeds": [{ "kind": "const", "value": [112,111,111,108,...] }] } },
    { "name": "new_stake_account_positions_option",     "optional": true },
    { "name": "current_stake_account_positions_option", "optional": true }
  ]
}
```

In Anchor programs, the IDL is the authoritative representation of account constraints. The `signer` account carries **no `relations` constraint** tying it to `publisher`. The `publisher` account carries only a data-existence check ("checked against data in the pool_data"), not an ownership or signing check. There is no constraint of the form `signer == publisher` anywhere in the IDL.

This means any funded Solana wallet can submit a transaction calling `set_publisher_stake_account`, supplying:
- Their own keypair as `signer`
- Any registered publisher's pubkey as `publisher`
- An arbitrary account as `new_stake_account_positions_option`

The writable `pool_data` account is then mutated to record the attacker-chosen stake account as the publisher's stake account.

Compare this to the `update_reward_program_authority` instruction, which correctly constrains its signer via `"relations": ["pool_config"]`, or `create_slash_event`, which constrains `owner` via `"relations": ["pool_config"]`. `set_publisher_stake_account` has no such guard. [1](#0-0) 

---

### Impact Explanation

The publisher's stake account in `pool_data` is the account from which the integrity pool program reads the publisher's self-delegation and computes reward caps. By replacing it with an attacker-controlled account (or with `None` by omitting `new_stake_account_positions_option`):

1. **Reward misdirection**: Rewards accrued to the publisher's self-delegation position are computed against the wrong stake account, potentially crediting the attacker's account.
2. **Cap manipulation**: Publisher stake caps used by the `advance` instruction to bound delegator rewards are derived from the publisher's stake account. Replacing it with a high-balance account inflates the cap, allowing over-rewarding of delegators staked to that publisher.
3. **Publisher disruption**: Setting the stake account to `None` removes the publisher's self-delegation from reward accounting, effectively zeroing their self-stake contribution and potentially triggering slashing or exclusion logic.

This directly affects the integrity and fairness of the OIS reward distribution system. [2](#0-1) 

---

### Likelihood Explanation

- The instruction is callable by any funded Solana wallet with no prerequisite role, token deposit, or governance approval.
- The only input that must be valid is a registered `publisher` pubkey, which is publicly readable from `pool_data` on-chain.
- The attack requires a single transaction and costs only the Solana transaction fee (~0.000005 SOL).
- There is no time-lock, cooldown, or rate-limit visible in the IDL. [3](#0-2) 

---

### Recommendation

Add an explicit signer constraint requiring the `signer` to equal the `publisher` pubkey. In Anchor Rust source, this would be:

```rust
#[account(constraint = signer.key() == publisher.key() @ IntegrityPoolError::InvalidSigner)]
pub signer: Signer<'info>,
```

Alternatively, make `publisher` itself a `Signer` account, removing the separate `signer` account entirely, consistent with how other publisher-scoped instructions are structured. [1](#0-0)

### Proof of Concept

```typescript
// Attacker wallet: any funded Solana keypair
const attacker = Keypair.generate();

// Target: any registered publisher pubkey (readable from pool_data on-chain)
const victimPublisher = new PublicKey("<registered_publisher_pubkey>");

// Attacker-controlled stake account (pre-funded with large PYTH balance)
const attackerStakeAccount = new PublicKey("<attacker_stake_account>");

await integrityPoolProgram.methods
  .setPublisherStakeAccount()
  .accounts({
    signer:    attacker.publicKey,   // attacker's own keypair — no publisher relation required
    publisher: victimPublisher,      // victim publisher — only existence-checked
    poolData:  poolDataPDA,
    poolConfig: poolConfigPDA,
    newStakeAccountPositionsOption:     attackerStakeAccount,
    currentStakeAccountPositionsOption: null,
  })
  .signers([attacker])
  .rpc();
// pool_data now records attackerStakeAccount as victimPublisher's stake account.
// Subsequent advance() calls compute caps and rewards against the attacker's account.
``` [1](#0-0) [4](#0-3)
