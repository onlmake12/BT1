### Title
Deviation-Only Subscription Permanently DoS'd via Zero-Price Storage Bypassing Deviation Check — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler._validateShouldUpdatePrices`, a zero-price guard silently skips the deviation check for any feed whose stored or incoming price is zero. For subscriptions configured with `updateOnDeviation = true` and `updateOnHeartbeat = false`, once a zero price is stored (achievable on the very first permissionless update), every subsequent update exhausts the deviation loop without triggering, permanently reverting with `UpdateConditionsNotMet`. The subscription is bricked and consumers receive stale prices indefinitely.

---

### Finding Description

`Scheduler.updatePriceFeeds` is a permissionless function — any address may call it with valid Wormhole-signed Pyth update data. On the first update for a subscription, `_validateShouldUpdatePrices` unconditionally accepts it:

```solidity
// If there's no previous price, this is the first update
if (previousFeed.id == bytes32(0)) {
    return updateTimestamp;
}
``` [1](#0-0) 

After the first update, the stored price is whatever was in the submitted Pyth VAA. The Pyth price field is `int64` and the protocol does not enforce non-zero values at the contract level; zero is a valid wire value (e.g., for funding-rate feeds or closed-market snapshots).

On all subsequent updates, the deviation check runs:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
``` [2](#0-1) 

If `previousPrice == 0` for every feed in the subscription, the `continue` fires for every iteration. The loop terminates without ever executing `return updateTimestamp`, and control falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [3](#0-2) 

Because `updateOnHeartbeat = false`, there is no alternative path to return early. The subscription can never be updated again.

The `updatePriceFeeds` entry point is fully permissionless:

```solidity
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {
``` [4](#0-3) 

An attacker who observes a newly created deviation-only subscription can front-run the first legitimate update with a valid Pyth VAA that carries `price = 0` for all subscribed feeds.

---

### Impact Explanation

Once a zero price is stored for all feeds in a deviation-only subscription:

- Every call to `updatePriceFeeds` reverts with `UpdateConditionsNotMet`.
- The subscription's stored prices are permanently frozen at zero.
- Any consumer (DeFi protocol, keeper, etc.) reading prices from the Scheduler via `getPricesUnsafe` / `getPriceNoOlderThan` receives the stale zero price or a stale pre-attack price, depending on what was stored.
- The subscription owner cannot recover without deploying a new subscription; the existing one is bricked.
- Subscription funds (`balanceInWei`) are locked and continue to be drained by keeper gas reimbursements for failed update attempts. [5](#0-4) 

---

### Likelihood Explanation

- `updatePriceFeeds` requires no special role — any EOA or contract can call it.
- The Pyth price wire format uses `int64` for price; zero is a valid value published by the network for certain feed types (funding rates, closed-market carry-forward).
- An attacker only needs to locate a historical or current valid Wormhole-signed Pyth VAA where the target feed's price is zero, then submit it as the first update before the legitimate keeper does.
- The `parsePriceFeedUpdatesWithConfig` call uses `minAllowedPublishTime = 0`, so historical zero-price VAAs are accepted. [6](#0-5) 

---

### Recommendation

**Short term:** Replace the silent `continue` with a treatment that counts a zero price as an infinite deviation (always trigger update), or revert if a zero price is encountered rather than silently skipping:

```solidity
// Treat zero price as maximum deviation — always trigger update
if (previousPrice == 0 || currentPrice == 0) {
    return updateTimestamp;
}
```

**Long term:** Add a non-zero price validation gate in `_storePriceUpdates` (or before it) to reject Pyth updates that carry a zero price for any subscribed feed, analogous to the `InvalidNonPositivePrice` guard already present in `PythPriceOracleGetter`:

```solidity
error InvalidNonPositivePrice();
``` [7](#0-6) 

---

### Proof of Concept

1. Deploy a `Scheduler` subscription with `updateOnDeviation = true`, `updateOnHeartbeat = false`, `deviationThresholdBps = 100`.
2. Obtain a valid Wormhole-signed Pyth VAA for the subscribed feed where `price = 0` (valid for funding-rate or closed-market feeds; `minAllowedPublishTime = 0` means historical VAAs are accepted).
3. Call `scheduler.updatePriceFeeds(subscriptionId, zeroPriceVAA)` — succeeds because `previousFeed.id == bytes32(0)` (first update path).
4. Zero price is now stored: `_state.priceUpdates[subscriptionId][feedId].price.price == 0`.
5. Call `scheduler.updatePriceFeeds(subscriptionId, realPriceVAA)` with any non-zero price.
6. Inside `_validateShouldUpdatePrices`: `previousPrice == 0` → `continue` for every feed → loop exhausts → `revert UpdateConditionsNotMet()`.
7. Repeat step 5 indefinitely — always reverts. Subscription is permanently bricked. [8](#0-7)
