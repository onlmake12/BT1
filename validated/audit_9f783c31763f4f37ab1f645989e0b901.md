### Title
`transfer()` Used for ETH Refund Causes DoS for Smart Contract Callers — (`lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.verifyUpdate()` refunds excess ETH to `msg.sender` using `payable(msg.sender).transfer(...)`, which forwards only 2300 gas. Any smart contract caller whose `receive`/`fallback` function consumes more than 2300 gas will have the entire `verifyUpdate()` call revert, making the function permanently unusable for that caller.

### Finding Description

In `PythLazer.sol`, the `verifyUpdate()` function accepts a fee and refunds any excess to the caller: [1](#0-0) 

```solidity
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
```

`transfer()` is hardcoded to forward exactly 2300 gas. Since EIP-1884 (Istanbul hard fork), many common operations (e.g., `SLOAD`) cost more than 2300 gas. Any smart contract with a non-trivial `receive` or `fallback` function — including multisigs, proxy wallets, or DeFi protocols — will cause this `transfer()` to revert, which reverts the entire `verifyUpdate()` call.

A second instance exists in `Governance.sol` at line 118 (`recipient.transfer(transfer.amount)`), but that recipient is set by a trusted governance action and is therefore out of scope for unprivileged attacker paths.

### Impact Explanation

Smart contract integrators (e.g., DeFi protocols, multisig-controlled bots, proxy-based accounts) that call `verifyUpdate()` with any `msg.value` above `verification_fee` will have every call revert. This is a complete denial-of-service for those callers — they cannot use the Lazer verification service at all unless they send the exact fee amount, which is fragile and may not be possible if `verification_fee` changes between the time they read it and the time their transaction lands.

### Likelihood Explanation

Smart contract callers of `verifyUpdate()` are a realistic and common integration pattern for on-chain DeFi protocols consuming Lazer price data. Sending a small buffer above the fee to avoid underpayment reverts is standard practice. The combination makes this likely to be triggered in production.

### Recommendation

Replace `transfer()` with a low-level `call` and check the return value:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Failed to refund excess fee");
}
```

### Proof of Concept

1. Deploy a contract `Caller` with a `receive()` function that performs one `SLOAD` (costs >2300 gas post-Istanbul).
2. From `Caller`, call `PythLazer.verifyUpdate{value: verification_fee + 1 wei}(validUpdate)`.
3. The `transfer()` at line 76 forwards 2300 gas to `Caller.receive()`, which runs out of gas.
4. The entire `verifyUpdate()` call reverts — the Lazer update is never processed. [2](#0-1)
