### Title
`addFunds()` MAX_DEPOSIT_LIMIT Check Does Not Account for Existing Balance, Allowing Permanent Subscriptions to Accumulate Unbounded ETH — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `addFunds()` function in `Scheduler.sol` validates each individual deposit against `MAX_DEPOSIT_LIMIT` for permanent subscriptions, but does not include the subscription's existing `balanceInWei` in the comparison. This allows the hard cap to be bypassed through repeated deposits, each just under the limit, permanently locking ETH beyond the intended ceiling.

### Finding Description

`MAX_DEPOSIT_LIMIT` is defined as 100 ETH and is documented as the maximum deposit limit for permanent subscriptions. [1](#0-0) 

In `addFunds()`, the guard is:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [2](#0-1) 

The check compares only `msg.value` against the limit, not `status.balanceInWei + msg.value`. Because `addFunds` has no access-control modifier (anyone may call it for any subscription), an actor can call it repeatedly with amounts ≤ `MAX_DEPOSIT_LIMIT`, accumulating an arbitrary total balance in a permanent subscription.

The analogous correct check would be:
```solidity
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

The same flaw does **not** exist in `createSubscription` because the subscription's balance starts at zero there, so `msg.value` alone is the total. [3](#0-2) 

### Impact Explanation

Permanent subscriptions explicitly prohibit withdrawals:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [4](#0-3) 

Any ETH deposited into a permanent subscription is irrecoverable. By bypassing `MAX_DEPOSIT_LIMIT`, an actor can permanently lock an unbounded amount of ETH in the contract beyond the protocol's intended ceiling. This constitutes a direct, irreversible loss of funds for whoever supplies the ETH, and breaks the protocol invariant that permanent subscriptions are capped at 100 ETH.

### Likelihood Explanation

The entry path requires no privilege — `addFunds` is callable by any address for any subscription ID. The bypass requires only multiple sequential transactions, each with `msg.value ≤ MAX_DEPOSIT_LIMIT`. No special knowledge, key, or governance access is needed. Any unprivileged transaction sender can trigger this.

### Recommendation

Replace the per-deposit check with a cumulative balance check in `addFunds()`:

```solidity
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

This mirrors the intent of the limit: the total locked balance in a permanent subscription must never exceed `MAX_DEPOSIT_LIMIT`.

### Proof of Concept

1. Deploy `Scheduler` (or use a live instance).
2. Create a permanent subscription with exactly `MAX_DEPOSIT_LIMIT` (100 ETH). This succeeds.
3. Call `addFunds(subscriptionId)` with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH). The check `100 ETH > 100 ETH` is **false**, so it passes.
4. `status.balanceInWei` is now 200 ETH — double the intended cap.
5. Repeat step 3 indefinitely. Each call passes the guard, and the balance grows without bound.
6. Because the subscription is permanent, `withdrawFunds` always reverts with `CannotUpdatePermanentSubscription`, so all accumulated ETH is permanently locked.
