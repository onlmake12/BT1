### Title
Missing Duplicate Index Validation in `GetBlockTransactionsProcess` Enables Bandwidth Amplification - (File: `sync/src/relayer/get_block_transactions_process.rs`)

---

### Summary

The `GetBlockTransactionsProcess::execute()` handler validates the *count* of requested transaction indexes but never checks for *duplicate* index values within the `indexes` field of a `GetBlockTransactions` message. An unprivileged P2P peer can craft a single message containing up to `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) identical indexes, forcing the responding node to serialize and transmit the same transaction thousands of times in a single `BlockTransactions` reply.

---

### Finding Description

`GetBlockTransactionsProcess::execute()` in `sync/src/relayer/get_block_transactions_process.rs` performs two guard checks on the incoming `GetBlockTransactions` message:

1. `indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH` — rejects if count exceeds 32,767.
2. `uncle_indexes().len() > max_uncles_num()` — rejects if uncle count exceeds consensus limit. [1](#0-0) 

Neither check tests for duplicate values within `indexes`. The handler then unconditionally iterates over every element of `indexes`, fetching the corresponding transaction from the stored block for each entry: [2](#0-1) 

Because `filter_map` returns `Some(tx)` for every valid index — including repeated ones — the resulting `transactions` `Vec` will contain one copy of the same transaction per duplicate index. All of these are packed into a `BlockTransactions` response and sent back to the requesting peer: [3](#0-2) 

By contrast, the analogous `GetBlockProposalProcess` and `GetTransactionsProcess` handlers **do** perform explicit deduplication before processing: [4](#0-3) [5](#0-4) 

The `GetBlockTransactions` handler is the only relay message handler that accepts a variable-length list of identifiers without a uniqueness check.

The `GetBlockTransactions` message schema confirms `indexes` is an unconstrained `Uint32Vec`: [6](#0-5) 

---

### Impact Explanation

A malicious peer sends one `GetBlockTransactions` message with `indexes = [k, k, k, …, k]` (32,767 copies of the same valid transaction index `k`). The responding node:

1. Fetches the same transaction object 32,767 times.
2. Serializes 32,767 copies into a single `BlockTransactions` message.
3. Transmits the oversized message back to the attacker.

If the targeted transaction is 200 bytes, the response is ~6.5 MB from a ~131 KB request — roughly a **50× bandwidth amplification factor**. Repeated at the per-peer rate limit of 30 messages/second, a single attacker peer can force the victim to emit ~195 MB/s of outbound traffic per attacker connection, exhausting network bandwidth and serialization CPU. Multiple attacker peers compound the effect linearly.

---

### Likelihood Explanation

The attack requires only a standard P2P connection. No privileged role, key material, or majority hashpower is needed. The attacker only needs to know any valid `block_hash` stored by the target (trivially obtained from the public chain) and any valid transaction index within that block (index 0, the cellbase, is always present). The `GetBlockTransactions` message type is a normal part of the compact block relay protocol, so sending it raises no suspicion and bypasses no authentication layer.

---

### Recommendation

Add a duplicate-index check in `GetBlockTransactionsProcess::execute()` immediately after the count check, mirroring the pattern already used in `GetBlockProposalProcess` and `GetTransactionsProcess`:

```rust
// After the existing count check:
let indexes_set: HashSet<u32> = self.message.indexes().iter()
    .map(Into::<u32>::into)
    .collect();
if indexes_set.len() != self.message.indexes().len() {
    return StatusCode::RequestDuplicate
        .with_context("Duplicate transaction indexes");
}
```

Apply the same check to `uncle_indexes` for consistency.

---

### Proof of Concept

```
Attacker → Victim (GetBlockTransactions):
  block_hash:    <any valid stored block hash, e.g. tip>
  indexes:       [1, 1, 1, 1, …]  (32,767 entries, all index 1)
  uncle_indexes: []

Victim → Attacker (BlockTransactions):
  block_hash:    <same hash>
  transactions:  [tx_at_index_1, tx_at_index_1, …]  (32,767 copies)
  uncles:        []
```

The responding node executes the loop at lines 61–71 of `get_block_transactions_process.rs` 32,767 times for the same transaction, producing a response up to ~50× larger than the request. No special privileges or chain state manipulation are required. [7](#0-6)

    pub async fn execute(self) -> Status {
        let shared = self.relayer.shared();
        {
            let get_block_transactions = self.message;
            if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "Indexes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    get_block_transactions.indexes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
            if get_block_transactions.uncle_indexes().len() > shared.consensus().max_uncles_num() {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "UncleIndexes count({}) > consensus max_uncles_num({})",
                    get_block_transactions.uncle_indexes().len(),
                    shared.consensus().max_uncles_num(),
                ));
            }
        }

        let block_hash = self.message.block_hash().to_entity();
        debug_target!(
            crate::LOG_TARGET_RELAY,
            "get_block_transactions {}",
            block_hash
        );

        if let Some(block) = shared.store().get_block(&block_hash) {
            let transactions = self
                .message
                .indexes()
                .iter()
                .filter_map(|i| {
                    block
                        .transactions()
                        .get(Into::<u32>::into(i) as usize)
                        .cloned()
                })
                .collect::<Vec<_>>();

            let uncles = self
                .message
                .uncle_indexes()
                .iter()
                .filter_map(|i| block.uncles().get(Into::<u32>::into(i) as usize))
                .collect::<Vec<_>>();

            let content = packed::BlockTransactions::new_builder()
                .block_hash(block_hash)
                .transactions(
                    transactions
                        .into_iter()
                        .map(|tx| tx.data())
                        .collect::<Vec<_>>(),
                )
                .uncles(
                    uncles
                        .into_iter()
                        .map(|uncle| uncle.data())
                        .collect::<Vec<_>>(),
                )
                .build();
            let message = packed::RelayMessage::new_builder().set(content).build();

            return async_send_message_to(&self.nc, self.peer, &message).await;
        }

        Status::ok()
    }
