### Title
Rust `DaoCalculator` reads full `u64` header-dep index while on-chain C script reads only the lowest byte — missing index-range guard causes tx-pool to reject valid DAO withdrawals — (`File: util/dao/src/lib.rs`)

### Summary

`DaoCalculator::transaction_maximum_withdraw` in `util/dao/src/lib.rs` reads the 8-byte little-endian `header_dep_index` from the witness as a full `u64`, then uses it to index `header_deps`. The deployed on-chain `dao.c` script reads only the **lowest byte** of that same 8-byte field. When `header_dep_index > 255`, the two sides resolve to different entries in `header_deps`, causing the Rust tx-pool fee check to reject a transaction that the C script would accept on-chain.

### Finding Description

In `transaction_maximum_withdraw` the witness `input_type` field is decoded as a full `u64`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// …
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // uses full u64
```

There is no guard asserting `header_dep_index <= u8::MAX` before this lookup. The on-chain C script, however, treats the same 8-byte field as a single byte index (reads only `data[0]`). For any witness value whose full `u64` interpretation differs from its lowest byte — i.e., any value `> 255` — the two sides resolve to different `header_deps` entries.

The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this split:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

With witness value `257` (LE bytes `[0x01, 0x01, 0x00, …]`):
- C reads byte `0x01` → `header_deps[1]` = deposit block (number 100) → script passes
- Rust reads `0x0101` = 257 → `header_deps[257]` = withdraw block (number 200) → block-number check at line 105 fails → `DaoError::InvalidOutPoint`

The block-number guard (`deposit_header.number() != deposited_block_number`) catches the mismatch in this specific test, but it is not a complete fix: an attacker can place a different block at position 257 that shares the same block number as the deposit block (e.g., a fork block at the same height), bypassing the guard while still causing Rust to use a different AR value than C.

### Impact Explanation

`DaoCalculator::transaction_fee` is called in two places:

1. **`tx-pool/src/util.rs` `check_tx_fee`** — rejects the transaction from the tx-pool if the computed fee is below the minimum fee rate.
2. **`verification/src/transaction_verifier.rs` `FeeCalculator::transaction_fee`** — used inside `ContextualTransactionVerifier::verify` for block-level fee accounting.

When Rust resolves to a different header than C, the computed `maximum_withdraw` diverges. If Rust resolves to a block with a lower accumulation rate (AR) than the actual deposit block, the computed fee is lower than the real fee, and `check_tx_fee` rejects a valid DAO withdrawal transaction. The transaction is valid on-chain (C accepts it) but cannot enter the tx-pool, permanently blocking the user's withdrawal.

Conversely, if Rust resolves to a block with a higher AR, the fee is over-estimated, the transaction enters the tx-pool with an inflated fee, and the block verifier's fee accounting diverges from the actual on-chain result.

### Likelihood Explanation

A transaction submitter (unprivileged) can trigger this by constructing a DAO withdrawal transaction with more than 256 `header_deps` entries and placing the deposit block hash at a position whose index exceeds 255. This is an unusual but protocol-legal transaction structure. No privileged access, key material, or majority hash power is required. The entry path is the standard `send_transaction` RPC or P2P relay.

### Recommendation

Add a range check immediately after decoding `header_dep_index`:

```rust
let idx = LittleEndian::read_u64(&header_deps_index_data.unwrap());
if idx > u8::MAX as u64 {
    return Err(DaoError::InvalidDaoFormat);
}
```

This aligns Rust's index interpretation with the on-chain C script's single-byte read, eliminating the discrepancy. Alternatively, document and enforce that `header_deps` for DAO withdrawals must contain at most 256 entries and that the deposit index must fit in one byte.

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` already demonstrates the split:

- Witness `input_type` = `257u64` (LE: `[0x01, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`)
- `header_deps[1]` = deposit block (number 100) — what C resolves
- `header_deps[257]` = withdraw block (number 200) — what Rust resolves
- Cell data = `100u64` (deposited block number)

Rust's block-number check (`200 != 100`) fires and returns `DaoError::InvalidOutPoint`, rejecting a transaction the C script would accept. To bypass the block-number guard and cause silent fee miscalculation, replace `header_deps[257]` with any block at height 100 other than the deposit block (e.g., a stale fork block at the same height); Rust then passes the number check but uses a different AR, computing a wrong withdrawal amount. #[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
