Audit Report

## Title
RBF Griefing: Attacker Self-Replacement Raises `min_replace_fee` to Deny Victim's Transaction Replacement - (File: `tx-pool/src/pool.rs`)

## Summary
`calculate_min_replace_fee` computes the replacement fee threshold from the live pool state at the moment the write lock is held. An attacker who controls a pending transaction can self-replace it with a fee equal to the victim's queued replacement fee, causing `min_replace_fee` to increase by one `extra_rbf_fee` increment before the victim's write lock is acquired. This can be repeated at ~300 shannons per round, indefinitely blocking the victim's replacement.

## Finding Description
`calculate_min_replace_fee` in `tx-pool/src/pool.rs` L101–127 derives the threshold entirely from the current fees of conflicting entries in the live pool:

```rust
fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
    let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
    ...
    sum.safe_add(extra_rbf_fee)
}
``` [1](#0-0) 

The rejection check at L664–671 calls this against the live pool state inside the write lock:

```rust
if let Some(min_replace_fee) = self.calculate_min_replace_fee(&all_conflicted, entry.size) {
    if fee < min_replace_fee {
        return Err(Reject::RBFRejected(...));
    }
}
``` [2](#0-1) 

`check_rbf` is invoked inside `with_tx_pool_write_lock` in `submit_entry` (`process.rs` L103–106), with the comment "check_rbf must be invoked in `write` lock to avoid concurrent issues": [3](#0-2) 

The write lock serializes simultaneous `check_rbf` calls but does **not** prevent the attacker's self-replacement from being enqueued and processed **before** the victim's transaction acquires the lock. Because script verification is async, there is a window between the victim submitting `tx_B` and `tx_B` acquiring the write lock. The attacker submits `tx_A'` (fee = victim's fee, a valid self-replacement since `F' >= F + extra_rbf_fee`) into this window. Once `tx_A'` is committed under the write lock, the victim's subsequent lock acquisition sees `min_replace_fee = fee(tx_A') + extra_rbf_fee > fee(tx_B)` and rejects. There is no rate-limiting, cooldown, or frozen-threshold mechanism anywhere in the codebase.

## Impact Explanation
Matches **High**: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* An attacker can sustain the griefing for the entire 2–10 block proposal/commitment window at a total cost of a few thousand shannons (≈ 0.00003 CKB), permanently blocking the victim's replacement and ensuring the attacker's transaction is committed. The victim wastes escalating fees and RPC round-trips without effect.

## Likelihood Explanation
RBF is enabled by default when `min_rbf_rate > min_fee_rate` (1500 vs 1000 shannons/KB per `resource/ckb.toml`). The attacker only needs to poll `get_transaction_with_verbosity` to observe the victim's pending replacement fee, then submit a self-replacement via `send_transaction` — no special privilege required. The timing window is favorable: async script verification creates a gap between victim submission and write-lock acquisition. Cost per round is ~300 shannons; 10 rounds costs ~3000 shannons total. [4](#0-3) 

## Recommendation
1. **Freeze the replacement threshold at first conflict entry**: Record `min_replace_fee` when the conflicting transaction first enters the pool and use that frozen value for all subsequent replacement checks against the same input set.
2. **Rate-limit self-replacements**: Enforce a minimum time gap (e.g., one block interval) between successive replacements of the same input set.
3. **Use the original conflicting tx's fee**: When computing `min_replace_fee`, use the fee of the transaction that was in the pool when the victim's replacement was first submitted, not the fee of whatever transaction currently occupies that input.

## Proof of Concept
1. Attacker submits `tx_A` spending input `I` with fee `F = 100_000_000` shannons.
2. Victim calls `get_transaction_with_verbosity(tx_A, 2)` → `min_replace_fee = F + 300 = 100_000_300`.
3. Victim constructs `tx_B` spending `I` with fee `V = 100_000_300` and submits via RPC.
4. Before `tx_B` acquires the write lock (during async verification), attacker submits `tx_A'` spending `I` with fee `F' = V = 100_000_300` (valid self-replacement: `F' >= F + extra_rbf_fee`).
5. `tx_A'` acquires the write lock first; `tx_A` is evicted, `tx_A'` enters the pool.
6. `tx_B` acquires the write lock next; `check_rbf` computes `min_replace_fee = F' + 300 = 100_000_600 > V` → `RBFRejected`.
7. Victim must now submit `tx_C` with fee ≥ 100_000_600. Attacker repeats step 4 with fee 100_000_600.
8. Reproducible as an integration test in `test/src/specs/tx_pool/replace.rs`: submit two concurrent RPC calls where the attacker's self-replacement is enqueued before the victim's replacement, then assert the victim receives `RBFRejected` despite offering the correct fee.

    fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
        let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
        // don't account for duplicate txs
        let replaced_fees: HashMap<_, _> = conflicts
            .iter()
            .map(|c| (c.id.clone(), c.inner.fee))
            .collect();
        let replaced_sum_fee = replaced_fees
            .values()
            .try_fold(Capacity::zero(), |acc, x| acc.safe_add(*x));
        let res = replaced_sum_fee.map_or(Err(CapacityError::Overflow), |sum| {
            sum.safe_add(extra_rbf_fee)
        });
        if let Ok(res) = res {
            Some(res)
        } else {
            let fees = conflicts.iter().map(|c| c.inner.fee).collect::<Vec<_>>();
            error!(
                "conflicts: {:?} replaced_sum_fee {:?} overflow by add {}",
                conflicts.iter().map(|e| e.id.clone()).collect::<Vec<_>>(),
                fees,
                extra_rbf_fee
            );
            None
        }
    }
