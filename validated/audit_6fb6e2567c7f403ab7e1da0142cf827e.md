### Title
Publisher Config Account Creation DOS via Pre-funded PDA — (`File: target_chains/solana/programs/pyth-price-store/src/processor/initialize_publisher.rs`)

---

### Summary

The `initialize_publisher` processor in `pyth-price-store` uses `system_instruction::create_account` to create the publisher config PDA. The Solana system program's `create_account` instruction unconditionally fails if the target account already holds any lamports. Because the publisher config PDA address is fully deterministic and publicly computable, an unprivileged attacker can permanently block registration of any specific publisher by pre-funding their PDA with even 1 lamport before `InitializePublisher` is called.

---

### Finding Description

In `initialize_publisher.rs`, the publisher config PDA is created via a raw `invoke_signed` call to `system_instruction::create_account`: [1](#0-0) 

The PDA address is derived deterministically from the public seeds `[PUBLISHER_CONFIG_SEED, publisher_pubkey]`: [2](#0-1) 

Where `PUBLISHER_CONFIG_SEED = "PUBLISHER_CONFIG"`: [3](#0-2) 

Because both the seed string and the publisher pubkey are public, any observer can compute the PDA off-chain and send a `transfer` of 1 lamport to it. After that, every subsequent call to `InitializePublisher` for that publisher will fail at the `create_account` CPI with `AccountAlreadyInUse` (or equivalent), since `create_account` requires the target account to have exactly 0 lamports.

The same pattern also affects the global config PDA in `initialize.rs`: [4](#0-3) 

The correct safe pattern — checking existing lamports and falling back to `transfer` + `allocate` + `assign` — is already implemented elsewhere in the same repository: [5](#0-4) 

And in `message_buffer`: [6](#0-5) 

---

### Impact Explanation

The `publisher_config` account is the prerequisite for a publisher to call `SubmitPrices`. Without a successfully initialized `publisher_config`, a publisher can never write price updates to the `pyth-price-store` program. An attacker who pre-funds the PDA for a targeted publisher permanently prevents that publisher from being onboarded, with no recovery path short of a program upgrade. The same attack on the global config PDA (`initialize.rs`) would prevent the entire program from ever being initialized.

---

### Likelihood Explanation

The attack requires only a standard SOL transfer of 1 lamport to a deterministically computable address — no privileged access, no leaked keys, no governance majority. The PDA seeds are public constants (`"PUBLISHER_CONFIG"` + publisher pubkey). Any on-chain observer watching the mempool for `InitializePublisher` transactions can front-run them. The cost to the attacker is negligible (< 1 lamport + transaction fee).

---

### Recommendation

Replace `system_instruction::create_account` in both `initialize_publisher.rs` and `initialize.rs` with the safe three-step pattern already used in `core-bridge/src/utils/cpi.rs` (`create_account_safe`) and `message_buffer/src/instructions/create_buffer.rs`:

1. Check `new_account.lamports()`.
2. If `> 0`, use `transfer` to top up to `minimum_balance`, then `allocate` + `assign`.
3. If `== 0`, use `create_account` as normal. [7](#0-6) 

---

### Proof of Concept

```
1. Observe (or predict) the publisher pubkey P that the authority intends to register.
2. Compute PDA = find_program_address(["PUBLISHER_CONFIG", P], PYTH_PRICE_STORE_PROGRAM_ID).
3. Send a transfer of 1 lamport to PDA from any funded wallet.
4. Authority calls InitializePublisher for publisher P.
5. invoke_signed(...create_account(...)) fails because PDA.lamports() == 1 != 0.
6. Publisher P can never be registered; SubmitPrices is permanently unavailable for P.
```

The attack on the global config follows the same steps using seed `["CONFIG"]` and targeting the one-time `Initialize` instruction. [1](#0-0) [4](#0-3)

```rust
impl<'info> CreateBuffer<'info> {
    /// Manually invoke transfer, allocate & assign ixs to create an account
    /// to handle situation where an account already has lamports
    /// since system_program::create_account will fail in this case
    fn create_account<'a>(
        new_account_info: &AccountInfo<'a>,
        space: usize,
        payer: &Signer<'a>,
        seeds: &[&[&[u8]]],
        system_program: &AccountInfo<'a>,
    ) -> Result<()> {
        let target_rent = Rent::get()?.minimum_balance(space);
        if new_account_info.lamports() < target_rent {
            system_program::transfer(
                CpiContext::new_with_signer(
                    system_program.to_account_info(),
                    Transfer {
                        from: payer.to_account_info(),
                        to: new_account_info.to_account_info(),
                    },
                    seeds,
                ),
                target_rent - new_account_info.lamports(),
            )?;
        };

        system_program::allocate(
            CpiContext::new_with_signer(
                system_program.to_account_info(),
                Allocate {
                    account_to_allocate: new_account_info.to_account_info(),
                },
                seeds,
            ),
            space.try_into().unwrap(),
        )?;

        system_program::assign(
            CpiContext::new_with_signer(
                system_program.to_account_info(),
                Assign {
                    account_to_assign: new_account_info.to_account_info(),
                },
                seeds,
            ),
            &crate::ID,
        )?;

        Ok(())
    }
```
