### Title
Incorrect variable reference in `script_exists_in_output` causes type-script entries to be wrongly deleted from the script table during rollback on PostgreSQL — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary
At line 252 of `remove.rs`, `row_lock.try_get::<bool, _>(0)` is used where `row_type.try_get::<bool, _>(0)` is required. On PostgreSQL, this causes `script_exists_in_output` to always return the lock-query result for the type-script existence check, making it blind to whether the script is still referenced as a `type_script_id` in remaining outputs.

### Finding Description
`rollback_block` removes outputs belonging to the tip block, then calls `script_exists_in_output(type_script_id)` to decide whether to delete the script row. [2](#0-1) 

Inside `script_exists_in_output`, after the outputs are already deleted, the function queries whether the script is still referenced anywhere else. The lock-query result (`row_lock`) is checked first; if false, execution falls through to the type-query. But the final `match` at line 252 reads `row_lock` again: [3](#0-2) 

On PostgreSQL, `row_lock.try_get::<bool, _>(0)` succeeds and returns `false` (the lock query already returned false, which is why we reached this point). The function therefore returns `Ok(false)` — "script does not exist" — even when `row_type` would have returned `true`. The script is then pushed into `script_id_list_to_remove` and deleted. [4](#0-3) 

### Impact Explanation
After a reorg on a PostgreSQL-backed rich-indexer node, any script that is used **only** as a `type_script_id` (and not simultaneously as a `lock_script_id` in any surviving output) is incorrectly deleted from the `script` table. Subsequent `get_cells` / `get_transactions` RPC calls filtered by that type script silently return empty results, corrupting the indexer's view of live cells. This is a persistent data-integrity failure: the deleted rows are not recovered on the next sync step.

### Likelihood Explanation
- Reorgs are a normal, unprivileged network event; any block relayer can cause one.
- Type scripts that differ from lock scripts are the common case (e.g., UDT/xUDT, Nervos DAO, NFT type scripts).
- PostgreSQL is the recommended production backend for the rich-indexer.
- The bug fires on every reorg that touches outputs with a type script, making it reliably reproducible.

### Recommendation
Change line 252 from:

```rust
match row_lock.try_get::<bool, _>(0) {
```

to:

```rust
match row_type.try_get::<bool, _>(0) {
``` [5](#0-4) 

### Proof of Concept
1. Start a rich-indexer backed by PostgreSQL.
2. Append a block containing one output whose `type_script_id ≠ lock_script_id` and whose type script does not appear as a lock script anywhere else.
3. Append a second block that does **not** reference that type script.
4. Trigger a reorg that rolls back the first block.
5. Query `SELECT COUNT(*) FROM script WHERE id = <type_script_id>` — it returns 0, even though the script was still referenced by the surviving second block's outputs (or should have been retained for future use).
6. Call `get_cells` filtered by that type script hash — returns empty, confirming the corruption.

    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }

        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,

    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

}
