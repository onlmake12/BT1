Audit Report

## Title
Missing IBD State Guard in `GetBlocksProcess::execute()` Allows Block Serving During Initial Block Download — (File: sync/src/synchronizer/get_blocks_process.rs)

## Summary
`sync/src/synchronizer/mod.rs` line 4 explicitly documents that both `GetHeaders` and `GetBlocks` must be answered with `packed::InIBD` during IBD mode. `GetHeadersProcess::execute()` correctly enforces this at lines 53–66, but `GetBlocksProcess::execute()` contains no equivalent guard, allowing any connected peer to force RocksDB block-status lookups and full block retrieval on a node that is still in IBD.

## Finding Description
`mod.rs` line 4 states: *"When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` and `GetBlocks` requests."* `GetHeadersProcess::execute()` (`get_headers_process.rs` lines 53–66) calls `active_chain.is_initial_block_download()`, sends `InIBD`, and returns `Status::ignored()`. `GetBlocksProcess::execute()` (`get_blocks_process.rs` lines 33–97) has no such check: after the `MAX_HEADERS_LEN` length guard (line 36) it immediately obtains `active_chain` (line 43), iterates up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` hashes, calls `contains_block_status()` (line 60) and `get_block()` (line 68) for each, and spawns async `SendBlock` tasks (lines 78–83). The dispatch in `mod.rs` lines 407–411 routes `GetBlocks` directly to `GetBlocksProcess` with no pre-dispatch IBD gate. Existing guards (genesis-hash rejection at line 52, `BLOCK_VALID` status check at line 60) do not substitute for an IBD check: the status lookup itself is the unwanted work, and any already-validated early blocks will be served in full.

## Impact Explanation
An attacker-controlled inbound peer can repeatedly send `GetBlocks` messages while the target is in IBD. Each message triggers up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` RocksDB `BlockStatus` lookups and, for any hashes that resolve to `BLOCK_VALID` blocks, full block retrieval and async dispatch — all competing with the node's own IBD download and verification pipeline. This degrades IBD throughput and violates the documented isolation invariant. Impact maps to **Low (501–2000 points): Any other important performance improvements for CKB**.

## Likelihood Explanation
`GetBlocks` is a standard, unauthenticated P2P sync message. Any peer that completes the CKB handshake can send it. CKB nodes accept inbound connections by default; the attacker needs only a TCP connection to the P2P port and a well-formed `GetBlocks` payload. No special privilege, key, or majority hashpower is required. The condition is trivially repeatable for the entire duration of IBD.

## Recommendation
Add an IBD guard at the top of `GetBlocksProcess::execute()` in `sync/src/synchronizer/get_blocks_process.rs`, mirroring `GetHeadersProcess`:

```rust
pub fn execute(self) -> Status {
    let active_chain = self.synchronizer.shared.active_chain();
    if active_chain.is_initial_block_download() {
        // Optionally send InIBD response as documented in mod.rs line 4
        return Status::ignored();
    }
    // ... existing logic
}
```

This aligns the implementation with the documented invariant at `mod.rs` line 4 and with the behavior of `GetHeadersProcess`.

## Proof of Concept
1. Start a CKB node from genesis (tip timestamp far behind wall clock → `is_initial_block_download()` returns `true`).
2. Connect as an inbound peer via the P2P port and complete the handshake.
3. Send `SyncMessage::GetBlocks` containing valid early block hashes (e.g., block 1 hash if already validated).
4. Observe: the node performs `contains_block_status()` RocksDB lookups and, for `BLOCK_VALID` hashes, sends `SendBlock` responses — no `InIBD` is sent.
5. Contrast: send `SyncMessage::GetHeaders` to the same node in IBD — the node correctly responds with `InIBD` (`get_headers_process.rs` line 53–65) and ignores the request.
6. The asymmetry between `GetHeaders` (guarded) and `GetBlocks` (unguarded) during IBD is the confirmed root cause.
