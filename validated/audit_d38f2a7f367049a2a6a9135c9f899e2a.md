Audit Report

## Title
P2P Snappy Decompression Memory Amplification via Attacker-Controlled Varint Header - (File: `network/src/compress.rs`)

## Summary
Both `Message::decompress` and `LengthDelimitedCodecWithCompress::decode` in `network/src/compress.rs` read the uncompressed length from the attacker-controlled snappy varint header and unconditionally allocate a buffer of that size before attempting decompression. The only guard is a cap at `MAX_UNCOMPRESSED_LEN = 1 << 23` (8 MB); there is no check on the ratio between the compressed wire size and the claimed output size. Any unprivileged peer can send a ~10-byte frame that triggers an 8 MB heap allocation, and sustained delivery of such frames can exhaust node memory and crash the node.

## Finding Description
`MAX_UNCOMPRESSED_LEN` is defined at line 13 of `network/src/compress.rs`: [1](#0-0) 

In `Message::decompress`, after `decompress_len` returns the attacker-supplied value, the code checks only `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN`. If the value is ≤ 8 MB (including exactly 8,388,608), it immediately allocates `vec![0; decompressed_bytes_len]` before calling `SnapDecoder::decompress`: [2](#0-1) 

The identical pattern exists in `LengthDelimitedCodecWithCompress::decode`, using `BytesMut::zeroed(decompressed_bytes_len)`: [3](#0-2) 

When `SnapDecoder::decompress` fails on garbage bytes, the error is returned but the 8 MB allocation has already occurred. The `length_delimited` codec's `max_frame_length` check (applied before decompression) limits the *compressed* frame size, not the claimed output size, so it does not prevent the amplification. For Ping (1 KB frame limit), a ~10-byte frame claiming 8 MB yields an ~800,000× amplification ratio.

`LengthDelimitedCodecWithCompress` is installed as the codec for every `CKBProtocol` built via `CKBProtocol::build`: [4](#0-3) 

All protocols are affected, including Sync (2 MB frame limit) and RelayV3 (4 MB frame limit): [5](#0-4) 

## Impact Explanation
An unprivileged peer can continuously send ~10-byte crafted frames, each causing an 8 MB heap allocation before the error path frees it. Under concurrent connections sending at line rate, the allocator is overwhelmed, leading to OOM termination or severe swap pressure that takes the node offline. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation
The attack requires only an inbound or outbound TCP connection — no authentication, no stake, no special role. CKB nodes accept connections from arbitrary peers by default. The crafted frame is trivially constructable from the public snappy specification. The `MAX_UNCOMPRESSED_LEN` guard introduced to fix GHSA-3gjh-29fv-8hr6 caps the per-allocation size but does not prevent the amplification attack, since it only bounds the maximum allocation, not the ratio of allocation to wire bytes. The attack is repeatable at network line rate.

## Recommendation
Before allocating the decompression buffer, validate that the claimed decompressed size is proportional to the compressed input size. Apply the same fix to both `Message::decompress` and `LengthDelimitedCodecWithCompress::decode`:

```rust
const MAX_COMPRESSION_RATIO: usize = 256;
if decompressed_bytes_len > compressed_data.len() * MAX_COMPRESSION_RATIO {
    return Err(io::ErrorKind::InvalidData.into());
}
```

Alternatively, replace the pre-allocation pattern with `SnapDecoder::decompress_vec`, which sizes the output buffer internally without trusting the header length.

## Proof of Concept
```rust
// Craft a ~10-byte "compressed" message claiming 8 MB decompressed size.
// snappy varint(8_388_608) = [0x80, 0x80, 0x80, 0x04]
let crafted_payload: &[u8] = &[
    0x80,                         // CKB compress flag
    0x80, 0x80, 0x80, 0x04,       // snappy varint: 8_388_608 (8 MB)
    0x00, 0x00, 0x00,             // garbage "compressed" bytes
];

use ckb_network::compress::decompress;
use p2p::bytes::BytesMut;

// Each call allocates ~8 MB before returning Err.
for _ in 0..1000 {
    let _ = decompress(BytesMut::from(crafted_payload));
    // ~8 GB of allocations after 1000 iterations
}
```

`decompress_len` returns 8,388,608; the check `8_388_608 > MAX_UNCOMPRESSED_LEN` is false (equal, not greater); `BytesMut::zeroed(8_388_608)` allocates 8 MB; `SnapDecoder::decompress` fails on the garbage payload; the error is returned. Under concurrent peer connections each sending this frame at line rate, the allocator is overwhelmed. [6](#0-5)

    pub(crate) fn decompress(mut self) -> Result<Bytes, io::Error> {
        if self.inner.is_empty() {
            Err(io::ErrorKind::InvalidData.into())
        } else if self.compress_flag() {
            match decompress_len(&self.inner[1..]) {
                Ok(decompressed_bytes_len) => {
                    if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {
                        debug!(
                            "The limit for uncompressed bytes len is exceeded. limit: {}, len: {}",
                            MAX_UNCOMPRESSED_LEN, decompressed_bytes_len
                        );
                        Err(io::ErrorKind::InvalidData.into())
                    } else {
                        let mut buf = vec![0; decompressed_bytes_len];
                        match SnapDecoder::new().decompress(&self.inner[1..], &mut buf) {
                            Ok(_) => Ok(buf.into()),
                            Err(e) => {
                                debug!("snappy decompress error: {:?}", e);
                                Err(io::ErrorKind::InvalidData.into())
                            }
                        }

                    Err(io::ErrorKind::InvalidData.into())
                }
            }

        }
    }
