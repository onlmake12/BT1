### Title
Stale Pre-Eviction Stat Overwrites Post-Eviction `total_tx_size`/`total_tx_cycles` in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `pool_map.rs`, the `add_entry` function pre-computes new `total_tx_size` and `total_tx_cycles` values **before** calling `check_and_record_ancestors`, which can evict existing pool entries and immediately update those same running totals. The pre-computed (stale) values are then unconditionally written back, silently overwriting the eviction-adjusted totals. The result is that `total_tx_size` and `total_tx_cycles` are permanently inflated by the aggregate size/cycles of the evicted transactions, corrupting the pool's accounting state.

---

### Finding Description

In `add_entry`:

```rust
// Step 1: pre-compute new totals BEFORE any evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//  total_tx_size  = self.total_tx_size + entry.size   (snapshot of old state)
//  total_tx_cycles = self.total_tx_cycles + entry.cycles

// Step 2: may evict entries — each eviction calls update_stat_for_remove_tx,
//         which MODIFIES self.total_tx_size and self.total_tx_cycles in-place
evicts = self.check_and_record_ancestors(&mut entry)?;

// ...

// Step 3: unconditionally overwrite with the STALE pre-eviction snapshot
self.total_tx_size = total_tx_size;    // ← discards eviction adjustments
self.total_tx_cycles = total_tx_cycles; // ← discards eviction adjustments
``` [1](#0-0) 

`updated_stat_for_add_tx` captures `self.total_tx_size + entry.size` at the moment of the call. [2](#0-1) 

`check_and_record_ancestors` enters the eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants`, which calls `update_stat_for_remove_tx` for every evicted entry, correctly subtracting their sizes/cycles from `self.total_tx_size`/`self.total_tx_cycles`. [3](#0-2) [4](#0-3) 

After evictions, the correct running total should be:

```
correct_total_tx_size = old_total − Σ(evicted.size) + entry.size
```

But what is stored is:

```
stored_total_tx_size  = old_total + entry.size          ← Σ(evicted.size) too large
```

The error equals the aggregate size of all evicted transactions. This is the direct analog of the GaugeController bug: a cumulative total is updated using a stale snapshot that does not account for an intermediate state-modifying operation.

---

### Impact Explanation

`total_tx_size` and `total_tx_cycles` are the authoritative pool-wide accounting fields exposed via `tx_pool_info` RPC and used internally for pool-full enforcement. [5](#0-4) 

After the bug fires, the pool believes it holds more data than it actually does. Consequences:

1. **False pool-full rejections**: subsequent `updated_stat_for_add_tx` calls see an inflated baseline, causing legitimate transactions to be rejected with `Reject::Full` even when actual pool occupancy is well within limits.
2. **Excess evictions**: any size-based eviction logic that reads `total_tx_size` will over-evict, removing valid pending transactions unnecessarily.
3. **Incorrect RPC reporting**: `tx_pool_info` returns inflated `total_tx_size`/`total_tx_cycles`, misleading operators and fee-estimation clients.

The inflation persists indefinitely (until the pool is cleared or the node restarts) because there is no periodic reconciliation path that corrects the totals under normal operation. The `recompute_total_stat` fallback is only invoked on underflow, not on this overcount path. [6](#0-5) 

---

### Likelihood Explanation

The eviction branch in `check_and_record_ancestors` is reachable by any unprivileged transaction submitter (RPC `send_transaction` / P2P relay). The attacker must arrange for a new transaction to have:

- More than `max_ancestors_count` (default 125) total ancestors in the pool, **and**
- Enough of those ancestors to be cell-dep references so that removing them brings the count within the limit.

Concretely: submit a chain of ~126 transactions where the final transaction also declares several earlier transactions as cell deps. This is a standard transaction graph that requires no special privilege, only sufficient CKB to pay fees. Once triggered, the stat corruption is permanent for the lifetime of the pool, amplifying the effect with each subsequent trigger.

---

### Recommendation

Move the stat pre-computation to **after** `check_and_record_ancestors` completes (and all evictions have been applied to `self.total_tx_size`/`self.total_tx_cycles`), or recompute the delta from the post-eviction baseline:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Compute and apply stat update AFTER evictions have already adjusted the totals
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

This ensures the running totals always reflect the true post-eviction pool state.

---

### Proof of Concept

1. Fill the pool with a chain of 126 transactions `T1 → T2 → … → T126` where each `Ti` spends an output of `T(i-1)`.
2. Craft `T_new` that spends an output of `T126` **and** declares `T1` (or several early chain members) as a cell dep. This makes `T1` a `cell_ref_parent` of `T_new`.
3. Submit `T_new` via `send_transaction`. `ancestors_count` = 127 > 125, but `ancestors_count − cell_ref_parents.len()` ≤ 125, so the eviction branch fires. `T1` (and its descendants) are evicted; `update_stat_for_remove_tx` correctly subtracts their sizes from `self.total_tx_size`. Then `self.total_tx_size = total_tx_size` overwrites with the stale pre-eviction value.
4. Query `tx_pool_info`: `total_tx_size` is now inflated by the aggregate size of the evicted transactions.
5. Repeat step 1–3 to accumulate further inflation. Eventually, `updated_stat_for_add_tx` returns `Reject::Full` for ordinary transactions that would otherwise fit, demonstrating the DoS effect on legitimate submitters. [7](#0-6) [8](#0-7)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    fn updated_stat_for_add_tx(
        &self,

    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(

                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
