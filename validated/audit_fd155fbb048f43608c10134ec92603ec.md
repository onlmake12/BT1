### Title
Missing `commitment` Validation in `Entropy.register()` Allows Provider to Permanently Collect Fees Without Fulfilling Requests — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `register()` function in `Entropy.sol` validates only that `chainLength != 0` but performs no validation that `commitment != bytes32(0)`. A provider (or a misconfigured default provider) can register with a zero commitment, making it cryptographically impossible to ever fulfill randomness requests. Users who pay fees to this provider will never receive their random numbers, and their fees are permanently transferred to the provider's `accruedFeesInWei` balance.

---

### Finding Description

The `register()` function at line 118 contains only one input guard:

```solidity
if (chainLength == 0) revert EntropyErrors.AssertionFailure();
```

There is no validation that `commitment != bytes32(0)`. The function then unconditionally stores the supplied commitment:

```solidity
provider.originalCommitment = commitment;   // line 131
provider.currentCommitment = commitment;    // line 133
```

If `commitment = bytes32(0)`, then `provider.currentCommitment = bytes32(0)`.

When a user subsequently calls `requestWithCallback` or `requestV2`, `requestHelper` computes:

```solidity
req.commitment = keccak256(bytes.concat(userCommitment, providerInfo.currentCommitment));
// = keccak256(bytes.concat(userCommitment, bytes32(0)))
```

and immediately accrues the fee:

```solidity
providerInfo.accruedFeesInWei += providerFee;   // line 237
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);  // line 238-239
```

For the provider to later call `revealWithCallback`, `revealHelper` requires:

```solidity
bytes32 providerCommitment = constructProviderCommitment(req.numHashes, providerContribution);
// must equal bytes32(0) — the stored currentCommitment
```

`constructProviderCommitment` computes `keccak256^numHashes(providerContribution)`. Finding any `providerContribution` such that repeated keccak256 hashing yields `bytes32(0)` is computationally infeasible (preimage resistance). Therefore, every `revealWithCallback` call will revert with `IncorrectRevelation` permanently.

The user's request remains in `CALLBACK_NOT_STARTED` state forever, the fee is already in the provider's `accruedFeesInWei`, and the provider can call `withdraw()` to extract it. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**High.** Users pay fees for randomness requests that can never be fulfilled. The fees are immediately and irrevocably transferred to the provider's `accruedFeesInWei` balance at request time. The provider can then call `withdraw()` to extract those fees. For the default provider, this affects every caller of the no-argument `requestV2()` variant, which routes to `getDefaultProvider()` automatically. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** Any address can call `register()` and become a provider — no privileged role is required. A malicious provider can deliberately register with `commitment = bytes32(0)` and a large `chainLength` to attract requests. Additionally, a legitimate provider (including the Fortuna keeper) could misconfigure and supply a zero commitment, breaking service for all users of that provider. The `register()` function is also used for re-registration (commitment rotation), so an existing provider with accrued users could rotate to a zero commitment mid-operation. [5](#0-4) 

---

### Recommendation

Add a zero-commitment guard in `register()`:

```solidity
if (chainLength == 0) revert EntropyErrors.AssertionFailure();
if (commitment == bytes32(0)) revert EntropyErrors.AssertionFailure();
```

This mirrors the existing `chainLength` guard and prevents registration in a state where reveals are permanently impossible. [6](#0-5) 

---

### Proof of Concept

1. Attacker calls `register(lowFeeInWei, bytes32(0), metadata, 10000, uri)` — succeeds because only `chainLength == 0` is checked.
2. User calls `requestWithCallback{value: fee}(attacker, userRandomNumber)` — succeeds; `providerInfo.accruedFeesInWei += providerFee`.
3. Attacker attempts `revealWithCallback(attacker, seqNum, userRandomNumber, anyContribution)` — always reverts with `IncorrectRevelation` because `constructProviderCommitment(numHashes, anyContribution) != bytes32(0)`.
4. User's request is permanently stuck in `CALLBACK_NOT_STARTED`; the user has no refund path.
5. Attacker calls `withdraw(providerFee)` — succeeds; attacker receives the user's fee. [7](#0-6)

    // and initial commitment. Re-registering the same provider rotates the provider's commitment (and updates

    //
    // chainLength is the number of values in the hash chain *including* the commitment, that is, chainLength >= 1.
    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;
