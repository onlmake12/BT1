### Title
Inaccurate `min_replace_fee` Returned by RPC Due to Existing-vs-New Transaction Size Mismatch — (File: `tx-pool/src/pool.rs`)

### Summary

The `min_replace_fee` value exposed via the `get_transaction_with_verbosity` RPC is computed using the **existing** (to-be-replaced) transaction's serialized size, while the actual RBF admission check in `check_rbf` uses the **new** (replacement) transaction's size. When a user constructs a replacement transaction that is larger than the original and sets its fee to exactly the RPC-returned `min_replace_fee`, the submission is rejected — mirroring the external report's pattern of a "preview" function returning a value that diverges from the actual enforced value due to a secondary calculation step.

### Finding Description

`TxPool::min_replace_fee` in `tx-pool/src/pool.rs` is the function that computes and surfaces the minimum replacement fee to callers (including the RPC layer):

```rust
pub fn min_replace_fee(&self, tx: &TxEntry) -> Option<Capacity> {
    ...
    self.calculate_min_replace_fee(&conflicts, tx.size)  // ← existing tx's size
}
``` [1](#0-0) 

`calculate_min_replace_fee` derives the mandatory extra fee as:

```rust
fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
    let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
    ...
    sum.safe_add(extra_rbf_fee)
}
``` [2](#0-1) 

`FeeRate::fee` performs integer division that rounds down:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
``` [3](#0-2) 

The actual RBF enforcement in `check_rbf` calls the **same** helper but passes `entry.size` — the **new** transaction's size:

```rust
if let Some(min_replace_fee) = self.calculate_min_replace_fee(&all_conflicted, entry.size) {
    if fee < min_replace_fee {
        return Err(Reject::RBFRejected(...));
    }
}
``` [4](#0-3) 

Because `extra_rbf_fee = min_rbf_rate × size / 1000`, whenever the replacement transaction is larger than the original (`new_size > old_size`), the enforced threshold exceeds the RPC-advertised value by `min_rbf_rate × (new_size − old_size) / 1000` shannons. A user who sets the replacement fee to exactly the RPC-returned `min_replace_fee` will be rejected.

The existing integration test confirms the RPC value is derived from the existing tx's size (242 bytes → 363 shannons extra at `min_rbf_rate = 1500`):

```
// min_replace_fee is 363
assert_eq!(ret.min_replace_fee.unwrap().to_string(), "0x5f5e26b");
``` [5](#0-4) 

### Impact Explanation

**Impact: Low.** A user who queries `min_replace_fee` via `get_transaction_with_verbosity` and constructs a replacement transaction that is larger than the original (e.g., adds outputs, witnesses, or cell deps) will have the replacement rejected by the tx-pool even though they paid the advertised minimum. No funds are lost, but the original transaction remains stuck in the pool at its lower fee rate, and the user receives a misleading error. In a congested network this can prevent timely fee-bumping.

### Likelihood Explanation

**Likelihood: High.** RBF replacement transactions are commonly larger than the originals — adding a change output, extending witnesses, or including additional cell deps all increase size. Any wallet or tool that reads `min_replace_fee` from the RPC and uses it as the exact fee target for a larger replacement will trigger this rejection on every such attempt.

### Recommendation

Pass the **new** transaction's size into `calculate_min_replace_fee` when computing the RPC-advertised value, or document clearly that the returned figure is a lower bound valid only when the replacement transaction is no larger than the original. The cleanest fix mirrors the recommendation in the external report — query the actual enforced value rather than a pre-computed approximation:

```diff
- self.calculate_min_replace_fee(&conflicts, tx.size)
+ // caller supplies the anticipated replacement size, or document the approximation
```

Alternatively, expose a separate RPC endpoint that accepts the replacement transaction and returns the exact required fee.

### Proof of Concept

1. Submit `tx1` (size S₁, fee F₁) to the tx-pool with `min_rbf_rate = 1500`.
2. Call `get_transaction_with_verbosity(tx1.hash, 2)` → `min_replace_fee = F₁ + 1500 × S₁ / 1000`.
3. Construct `tx2` that spends the same inputs as `tx1` but is larger (size S₂ > S₁, e.g., adds a change output). Set `tx2.fee = min_replace_fee` from step 2.
4. Submit `tx2` → rejected with `"Tx's current fee is X, expect it to >= Y to replace old txs"` where `Y = F₁ + 1500 × S₂ / 1000 > min_replace_fee`.

The discrepancy is `1500 × (S₂ − S₁) / 1000` shannons — exactly the rounding gap introduced by computing `extra_rbf_fee` against the wrong transaction's size.

    /// min_replace_fee = sum(replaced_txs.fee) + extra_rbf_fee
    fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
        let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
        // don't account for duplicate txs
        let replaced_fees: HashMap<_, _> = conflicts
            .iter()
            .map(|c| (c.id.clone(), c.inner.fee))
            .collect();
        let replaced_sum_fee = replaced_fees
            .values()
            .try_fold(Capacity::zero(), |acc, x| acc.safe_add(*x));
        let res = replaced_sum_fee.map_or(Err(CapacityError::Overflow), |sum| {
            sum.safe_add(extra_rbf_fee)
        });
        if let Ok(res) = res {
            Some(res)
        } else {
            let fees = conflicts.iter().map(|c| c.inner.fee).collect::<Vec<_>>();
            error!(
                "conflicts: {:?} replaced_sum_fee {:?} overflow by add {}",
                conflicts.iter().map(|e| e.id.clone()).collect::<Vec<_>>(),
                fees,
                extra_rbf_fee
            );
            None
        }
    }
