Audit Report

## Title
`RecentReject::put()` silently destroys just-written entry when `shrink()` randomly drops the same shard — (`tx-pool/src/component/recent_reject.rs`)

## Summary
`put()` writes an entry to a deterministically-assigned shard, then calls `shrink()` based on a counter that is never incremented inside `put()`. `shrink()` randomly drops and recreates one of the `shard_num` column families. If the randomly selected shard matches the one just written, the entry is permanently destroyed while `put()` returns `Ok(())`, producing silent data loss.

## Finding Description
**Missing counter update:** In `put()` at line 62, `self.total_keys_num.checked_add(1)` produces a local binding `total_keys_num` used only for the comparison; `self.total_keys_num` is never assigned the result. The field is only refreshed inside `shrink()` at line 111 via `estimate_total_keys_num()`. Consequently, once the DB holds `>= count_limit` entries at startup, the condition `total_keys_num > self.count_limit` (line 63) evaluates to `true` on every subsequent `put()` call, triggering `shrink()` unconditionally.

**Invariant-breaking sequence:**
1. `put()` writes the entry to shard X via `self.db.put(&shard, hash_slice, json_string)` (line 60).
2. The stale `self.total_keys_num` satisfies `>= count_limit`, so `shrink()` is called (lines 62–64).
3. `shrink()` samples a random shard via `thread_rng()` over `Uniform::new(0, self.shard_num)` (line 106).
4. If the random pick equals shard X, `drop_cf(X)` destroys all data in X (line 107), then `create_cf_with_ttl(X, ttl)` recreates it empty (line 108).
5. `put()` returns `Ok(())` — the caller observes success.
6. A subsequent `get()` on the same hash returns `None` because the CF is now empty.

**Shard assignment is deterministic:** `get_shard()` computes `u32::from_le_bytes(hash[0..4]) % shard_num` (lines 115–119), so an attacker can craft a transaction hash that maps to any desired shard.

**Existing checks are insufficient:** The callback in `shared_builder.rs` at lines 580–585 only logs an `error!` on `put()` failure — it cannot detect silent data loss since `put()` returns `Ok(())`. The existing test in `tests/recent_reject.rs` uses `limit = 100` with only 80 puts on a fresh DB, so `self.total_keys_num` (starting at 0, never incremented) never reaches `count_limit`, and `shrink()` is never triggered.

## Impact Explanation
The recent-reject store is the authoritative persistent record of rejected transactions. Silent entry loss causes `get_transaction` RPC to return `TxStatus::Unknown` instead of `TxStatus::Rejected` for affected transactions, and allows previously-rejected transactions to be re-submitted and re-verified without the node recognizing them as previously rejected. This constitutes a **suboptimal implementation of the CKB state storage mechanism** (Medium, 2001–10000 points): a critical tx-pool storage component fails to reliably persist entries due to a missing counter update and an unguarded post-write shrink.

## Likelihood Explanation
The trigger condition (`self.total_keys_num >= count_limit`) is met on any node whose DB already held `>= keep_rejected_tx_hashes_count` rejected transactions at startup — a routine operational state for any active node. Once triggered, the condition fires on every `put()` call (since the counter is never incremented), and the probability of destroying the just-written entry is 1/`shard_num` = 20% with the default of 5 shards. An unprivileged attacker who can submit rejected transactions via P2P relay can repeatedly trigger this condition. With `shard_num = 2`, the probability rises to 50% per attempt, making reliable triggering straightforward within a small number of attempts.

## Recommendation
1. **Increment `self.total_keys_num` in `put()`** after a successful write:
   ```rust
   self.total_keys_num = self.total_keys_num.saturating_add(1);
   ```
2. **Call `shrink()` before writing**, not after, so the just-written entry is never in the shard that may be dropped.
3. Alternatively, **exclude the just-written shard from `shrink()`'s random selection** by passing it as a parameter and retrying if the random pick matches.

## Proof of Concept
```rust
// shard_num=2, count_limit=1, ttl=-1
let tmp = tempfile::tempdir().unwrap();
let mut rr = RecentReject::build(tmp.path(), 2, 1, -1).unwrap();

// Force total_keys_num to count_limit so shrink fires on next put
rr.total_keys_num = rr.count_limit; // simulate DB at limit

// Craft a hash that maps to shard 0 (first 4 bytes LE % 2 == 0)
let hash = Byte32::new([0u8; 32]); // 0 % 2 == 0 → shard "0"

// put() writes to shard 0, then shrink() randomly drops shard 0 or 1
rr.put(&hash, Reject::Malformed("x".into(), Default::default())).unwrap();

// With 50% probability, get() returns None immediately after Ok put
// Run in a loop: reliable within ~3 attempts
assert!(rr.get(&hash).unwrap().is_none()); // invariant violated
```

A targeted test with `shard_num = 2` and `total_keys_num = count_limit` will reproduce the failure within a small number of iterations.
