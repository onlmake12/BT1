Audit Report

## Title
Missing Duplicate-Index Check and Byte-Size Guard in `GetBlockTransactionsProcess::execute` Enables Multi-Peer OOM Amplification — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` enforces only a count bound on indexes but performs no deduplication and no byte-size cap on the assembled response. An attacker with 128 peers can send `GetBlockTransactions` messages containing 32,767 identical indexes, causing the node to clone the same large transaction 32,767 times per call, with up to 3,840 concurrent calls per second, exhausting heap memory and crashing the node.

## Finding Description

**No duplicate index rejection:**

The only structural check in `execute` is a count bound: [1](#0-0) 

After this check, indexes are iterated with `filter_map` and `.cloned()` with no deduplication. If all 32,767 indexes are identical (e.g., all `0`), the resulting `Vec` contains 32,767 independent clones of transaction 0: [2](#0-1) 

The `#[allow(clippy::needless_collect)]` annotation on `execute` confirms the `Vec` is fully materialized in memory before `async_send_message_to` is called: [3](#0-2) 

The full `BlockTransactions` message is then built and serialized in memory before any send occurs: [4](#0-3) 

**No byte-size cap on the outgoing payload:**

The file imports only `MAX_RELAY_TXS_NUM_PER_BATCH`, not `MAX_RELAY_TXS_BYTES_PER_BATCH`: [5](#0-4) 

**Contrast with sibling handlers that have both guards:**

`GetTransactionsProcess::execute` explicitly rejects duplicate hashes via a `HashSet` comparison: [6](#0-5) 

And enforces `MAX_RELAY_TXS_BYTES_PER_BATCH` on the outgoing payload: [7](#0-6) 

`GetBlockProposalProcess::execute` similarly deduplicates via `HashSet`: [8](#0-7) 

And enforces the byte cap: [9](#0-8) 

**Rate limiter is per-peer, not global:**

The rate limiter is keyed by `(PeerIndex, u32)` at 30 req/sec per key: [10](#0-9) 

With `MAX_RELAY_PEERS = 128`, the aggregate ceiling is 128 × 30 = 3,840 concurrent `execute()` calls per second, each independently allowed: [11](#0-10) 

**Inbound frame limit does not protect against this:** A `GetBlockTransactions` message with 32,767 `uint32` indexes is ~128 KB — well within the 4 MB inbound cap for `RelayV3`. The frame limit constrains the inbound request, not the outbound response allocation.

## Impact Explanation

A stored block containing a large transaction (CKB's block byte limit allows transactions of hundreds of KB). With 32,767 duplicate indexes pointing to that transaction, each `execute()` call allocates `32,767 × tx_size` bytes on the heap before the send. At 3,840 concurrent calls per second, aggregate heap pressure can reach tens to hundreds of GB, crashing the node with OOM. This matches the allowed bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires no PoW, no keys, and no privileged access — only 128 TCP connections and knowledge of any stored block hash (public information from the chain). The attacker sends `GetBlockTransactions` with `indexes = [0, 0, ..., 0]` (32,767 copies, which passes the `> MAX_RELAY_TXS_NUM_PER_BATCH` check since 32,767 is not strictly greater than 32,767) at 30 req/sec per connection. This is straightforward to automate and repeatable indefinitely.

## Recommendation

1. **Reject duplicate indexes** before processing, mirroring the guard in `GetTransactionsProcess` and `GetBlockProposalProcess`:
   ```rust
   let deduped: HashSet<u32> = indexes.iter().map(Into::into).collect();
   if deduped.len() != indexes.len() {
       return StatusCode::RequestDuplicate.with_context("duplicate indexes");
   }
   ```
2. **Enforce `MAX_RELAY_TXS_BYTES_PER_BATCH`** on the assembled transaction payload before building the response message, mirroring `GetTransactionsProcess` and `GetBlockProposalProcess`.
3. Consider a **global (cross-peer) rate limiter** or concurrency semaphore on `GetBlockTransactions` processing to bound aggregate memory pressure.

## Proof of Concept

```
1. Attacker opens 128 TCP connections to the target CKB node.
2. Attacker reads any recent block hash H from the public chain (block is stored on the node).
3. Each connection sends, at 30 req/sec, a RelayV3 GetBlockTransactions message:
     block_hash = H
     indexes    = [0, 0, 0, ..., 0]  // 32,767 copies of index 0
4. Each execute() call passes the count check (32,767 <= MAX_RELAY_TXS_NUM_PER_BATCH),
   then clones the coinbase/largest tx 32,767 times into a Vec.
5. The full BlockTransactions message is serialized in memory before async_send_message_to.
6. With 128 peers × 30 req/sec = 3,840 concurrent allocations, heap exhaustion
   causes the node process to be OOM-killed.
```
