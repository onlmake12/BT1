Audit Report

## Title
Stale Pre-Eviction Snapshot Overwrites Correctly-Decremented `total_tx_size` in `PoolMap::add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures `self.total_tx_size + entry.size` into a local variable before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts cell-dep parent transactions via `remove_entry_and_descendants`, each eviction correctly decrements `self.total_tx_size` through `update_stat_for_remove_tx`. The stale pre-eviction snapshot is then unconditionally written back to `self.total_tx_size`, permanently inflating the counter by the aggregate size of all evicted transactions. Repeated exploitation drives `total_tx_size` arbitrarily high while the actual pool remains nearly empty, causing `limit_size` to reject all new submissions with `Reject::Full`.

## Finding Description
The exact sequence in `add_entry` (L210–219):

```rust
// Step 1: snapshot taken BEFORE evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;  // L210-211

// Step 2: evictions happen — remove_entry_and_descendants → update_stat_for_remove_tx
//         correctly decrements self.total_tx_size for each evicted tx
evicts = self.check_and_record_ancestors(&mut entry)?;         // L213

// Step 3: stale snapshot written back unconditionally, erasing all decrements
self.total_tx_size = total_tx_size;                            // L218
self.total_tx_cycles = total_tx_cycles;                        // L219
```

`updated_stat_for_add_tx` (L711–729) is a read-only computation: it returns `self.total_tx_size + entry.size` without modifying `self`. `check_and_record_ancestors` (L603–625) enters the eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, calling `remove_entry_and_descendants` (L618) in a loop. `remove_entry_and_descendants` (L252–265) calls `remove_entry` for each removed entry; `remove_entry` calls `update_stat_for_remove_tx` (L247), which correctly decrements `self.total_tx_size` (L738–740). The unconditional write-back at L218–219 then restores the pre-eviction value, erasing all decrements.

Net effect per invocation where `k` transactions are evicted:

| Counter | Correct value | Actual value written |
|---|---|---|
| `total_tx_size` | `original − Σ(evicted.size) + entry.size` | `original + entry.size` |
| `total_tx_cycles` | `original − Σ(evicted.cycles) + entry.cycles` | `original + entry.cycles` |

The self-healing path in `update_stat_for_remove_tx` (L742–756) only triggers on underflow, never on overflow, so the inflation is never corrected.

## Impact Explanation
`limit_size` uses `self.pool_map.total_tx_size > self.config.max_tx_pool_size` as its sole eviction trigger. An inflated `total_tx_size` causes `limit_size` to continuously evict otherwise-valid pending transactions and reject new submissions with `Reject::Full`. Because the inflation accumulates across repeated invocations, an attacker can drive `total_tx_size` arbitrarily high while the actual pool remains nearly empty, rendering the mempool non-functional for all users. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The trigger condition requires a new transaction whose ancestor count via cell-dep references exceeds `max_ancestors_count` but can be reduced to within limits by evicting low-fee cell-dep parents. This is reachable by any unprivileged `send_transaction` RPC caller or P2P relay peer. No key material, privileged access, or majority hash power is required. The attack is repeatable: each crafted submission inflates the counter further, and the inflation compounds across invocations.

## Recommendation
Move the stat snapshot computation to after `check_and_record_ancestors` completes, so evictions are already reflected in `self.total_tx_size` when the snapshot is taken:

```diff
- let (total_tx_size, total_tx_cycles) =
-     self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
  trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
  evicts = self.check_and_record_ancestors(&mut entry)?;
  self.record_entry_edges(&entry)?;
  self.insert_entry(&entry, status);
  self.record_entry_descendants(&entry);
  self.track_entry_statics(None, Some(status));
+ let (total_tx_size, total_tx_cycles) =
+     self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
  self.total_tx_size = total_tx_size;
  self.total_tx_cycles = total_tx_cycles;
```

The overflow check in `updated_stat_for_add_tx` is preserved; moving it after evictions is safe because evictions only reduce `self.total_tx_size`, making overflow less likely.

## Proof of Concept
1. Configure a node with `max_ancestors_count = N` and a modest `max_tx_pool_size`.
2. Submit transactions `T1 … Tk` where each `Ti` is referenced as a cell dep by `T_{i+1}`, forming a chain of length `N`. All transactions have low fee rates.
3. Submit a new transaction `Tnew` that references `T1` as a cell dep. Its ancestor count is `N+1`, exceeding the limit. Since `cell_ref_parents = {T1}`, the condition at L603 is satisfied: `(N+1) - 1 = N <= max_ancestors_count`.
4. `check_and_record_ancestors` evicts `T1` (and its descendants) via `remove_entry_and_descendants`. `self.total_tx_size` is decremented by `Σ(evicted sizes)`.
5. `add_entry` writes back the pre-eviction snapshot at L218, re-inflating `total_tx_size` by `Σ(evicted sizes)`.
6. Repeat steps 2–5. Each iteration inflates `total_tx_size` by the evicted batch size.
7. After enough iterations, `total_tx_size > max_tx_pool_size` even though the pool is nearly empty. `limit_size` begins evicting all remaining transactions; new submissions are rejected with `Reject::Full`.
