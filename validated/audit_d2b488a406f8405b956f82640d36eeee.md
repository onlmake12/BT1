### Title
No Maximum Fee Protection in Entropy `requestHelper` Allows Silent Overpayment or Unexpected Revert — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Pyth Entropy contract's `requestHelper` function reads the current fee from on-chain state at execution time and silently captures all excess `msg.value` beyond the provider fee into the Pyth protocol treasury. Because providers can update their fee at any time via `setProviderFee()`, and the Fortuna keeper actively adjusts fees based on gas prices, there is a window between when a user queries `getFeeV2()` and when their transaction executes during which the fee can change. There is no `maxFee` parameter to protect users against this slippage.

### Finding Description

In `requestHelper` (Entropy.sol lines 233–239):

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

All `msg.value` beyond the provider fee is credited to `accruedPythFeesInWei` — it is never refunded. The interface explicitly states "excess value is *not* refunded to the caller." [2](#0-1) 

Providers can change their fee at any time without restriction: [3](#0-2) 

The Fortuna keeper (`apps/fortuna/src/keeper/fee.rs`) actively and automatically adjusts provider fees based on current gas prices, making fee changes a routine, frequent occurrence: [4](#0-3) 

None of the `requestV2` / `requestWithCallback` entry points accept a `maxFee` parameter: [5](#0-4) 

**Scenario A — Silent overpayment (fee decreases):**
1. User calls `getFeeV2()` → 100 wei
2. Fortuna decreases provider fee; new total = 80 wei
3. User's transaction executes with `msg.value = 100 wei`
4. `requiredFee = 80 wei` → passes the check
5. Provider gets 50 wei (new `providerFee`); Pyth gets `100 − 50 = 50 wei` (20 wei excess silently captured)
6. User loses 20 wei with no revert, no event, no indication

**Scenario B — Unexpected revert (fee increases):**
1. User calls `getFeeV2()` → 100 wei
2. Fortuna or a provider increases fee to 120 wei
3. User's transaction executes with `msg.value = 100 wei`
4. Reverts with `InsufficientFee`; user loses gas and must retry

### Impact Explanation

**Scenario A** causes a direct, silent loss of native token funds for the user. The excess is permanently captured by the Pyth protocol treasury with no refund path. The loss per transaction is bounded by the fee delta, but given Fortuna's continuous fee adjustments, this can affect every user who queries the fee and submits a transaction in a separate block.

**Scenario B** causes a denial-of-service: the user's transaction fails unexpectedly, wasting gas. A malicious provider can front-run any pending `requestV2` call with a `setProviderFee` increase to reliably cause reverts.

### Likelihood Explanation

The Fortuna keeper (`adjust_fee_if_necessary`) runs continuously and adjusts fees whenever gas prices move outside a tolerance band. On active chains, fee changes happen in nearly every epoch. Any user who queries `getFeeV2()` off-chain and submits a transaction in a subsequent block is exposed. No special attacker capability is required for Scenario A; Scenario B requires only a registered provider (permissionless registration).

### Recommendation

Add a `maxFee` parameter to `requestV2` / `requestWithCallback` that reverts if the current fee exceeds the caller's stated maximum:

```solidity
function requestV2(
    address provider,
    bytes32 userContribution,
    uint32 gasLimit,
    uint128 maxFee          // new: revert if requiredFee > maxFee
) public payable override returns (uint64) { ... }
```

Alternatively, refund excess `msg.value` to the caller instead of silently crediting it to the protocol treasury.

### Proof of Concept

```solidity
// 1. User queries fee off-chain
uint128 fee = entropy.getFeeV2(provider, 0); // e.g., 100 wei

// 2. In the same or next block, Fortuna decreases provider fee
// (setProviderFee called by Fortuna keeper, new total = 80 wei)

// 3. User submits with msg.value = 100 wei (the fee they queried)
entropy.requestV2{value: fee}(provider, userRandom, 0);
// Transaction succeeds, but 20 wei is silently captured by Pyth treasury.
// User has no way to prevent this without a maxFee guard.
```

The root cause is in `requestHelper` at [6](#0-5)  — the fee is read from current state with no upper-bound check against the caller's expectation, and all excess is non-refundably credited to the protocol.
