### Title
Unrestricted `ckb_wait` Syscall Allows Any Spawned VM to Steal Sibling Exit Codes, Causing Parent Script Failure - (File: `script/src/scheduler.rs`)

---

### Summary

The `ckb_wait` syscall in CKB's multi-process VM scheduler performs no ownership or parentage check. Any VM can call `ckb_wait` with an arbitrary target VM ID — including sibling VMs it did not spawn. A malicious spawned child script can consume the exit code of a sibling VM from the scheduler's `terminated_vms` map, causing the legitimate parent VM's subsequent `ckb_wait` call to return `WAIT_FAILURE`. This can cause a lock script using the spawn/wait pattern to fail, preventing legitimate transactions from being validated.

---

### Finding Description

**Root cause — `script/src/syscalls/wait.rs`:**

The `Wait` syscall reads `target_id` directly from register A0 (fully script-controlled) and posts a `Message::Wait` to the shared message box with zero validation: [1](#0-0) 

There is no check that `self.id` (the calling VM) has any relationship to `target_id`.

**Root cause — `script/src/scheduler.rs`, `process_message_box`:**

When the scheduler processes `Message::Wait`, it checks whether the target VM has already terminated and, if so, returns the exit code **and permanently removes it** from `terminated_vms`: [2](#0-1) 

The critical line is: [3](#0-2) 

`terminated_vms.retain` removes the exit code entry for `target_id` from the global map. No check is made that `vm_id` (the waiter) is the parent of `args.target_id`. Any VM — including a sibling — can call this path and consume the exit code.

**Scheduler execution order amplifies the attack:**

The scheduler always runs the VM with the largest ID first: [4](#0-3) 

A malicious child (always assigned a higher ID than the root) is guaranteed to run before the root VM, making the attack deterministic and reliable.

**The `fds` map has an ownership check; `wait` does not:**

For comparison, the `Message::Spawn` handler explicitly checks fd ownership: [5](#0-4) 

And `Message::FdRead`/`FdWrite` check fd ownership: [6](#0-5) 

No equivalent check exists for `Message::Wait`.

**The `terminated_vms` map is the shared mutable state being exploited:** [7](#0-6) 

This is the direct analog of the H-05 adapter storage: shared state that any callee can freely manipulate because the framework does not enforce isolation.

---

### Impact Explanation

A malicious spawned script (provided as a cell dep) can consume the exit code of a sibling VM from `terminated_vms`. The legitimate parent's subsequent `ckb_wait` call finds neither an entry in `terminated_vms` nor an entry in `states` (the sibling has already been removed from `states` upon termination), and receives `WAIT_FAILURE`. If the parent lock script treats `WAIT_FAILURE` as an error (the natural behavior), it fails, and the transaction is rejected.

In lock scripts that use the spawn/wait pattern for modular verification (e.g., spawning a signature verifier or a sub-script), this allows a malicious cell dep to prevent the transaction from being validated — a denial-of-service on specific transactions. If the lock script uses the child's exit code to make authorization decisions, the malicious child can also cause incorrect authorization outcomes.

---

### Likelihood Explanation

The attack requires a lock script that:
1. Spawns multiple child VMs, and
2. Spawns at least one child from a cell dep whose code is not verified by code hash (i.e., the transaction author controls which binary runs).

This is realistic in modular lock script designs where the transaction author provides a "plugin" or "verifier" cell dep. The scheduler's deterministic execution order (largest VM ID runs first) makes the attack reliably reproducible without any timing dependency. The attacker only needs to know sibling PIDs, which are sequential integers starting from 1 and trivially predictable.

---

### Recommendation

Track the parent-child relationship when a VM is spawned. In `boot_vm`, record `parent_vm_id → child_vm_id`. In the `Message::Wait` handler, verify that `vm_id` (the waiter) is the recorded parent of `args.target_id` before allowing the wait to proceed. If the check fails, return `WAIT_FAILURE` immediately. This mirrors the existing fd ownership check pattern already used for `Message::Spawn`, `Message::FdRead`, `Message::FdWrite`, and `Message::Close`.

---

### Proof of Concept

**Setup:** A lock script (root VM, pid=0) spawns two children: a legitimate helper (pid=1) and a malicious helper (pid=2, provided as a crafted cell dep).

**Execution sequence (deterministic due to largest-ID-first scheduling):**

1. Root (pid=0) spawns Child A (pid=1) and Child B (pid=2, malicious). Both are `Runnable`.
2. Child B (pid=2) runs first. It does not call `ckb_wait(1)` yet — it stays `Runnable`.
3. Child A (pid=1) runs and terminates with exit code 0. The scheduler executes:
   - `terminated_vms.insert(1, 0)` — exit code stored.
   - No VM is in `VmState::Wait { target_vm_id: 1 }`, so no immediate wakeup.
   - Child A removed from `states`, `instantiated`, `suspended`.
4. Child B (pid=2) runs again (largest runnable ID). It calls `ckb_wait(1)`:
   - `terminated_vms.get(&1)` → `Some(0)` — exit code found.
   - Exit code written to Child B's memory. Child B marked `Runnable`.
   - **`terminated_vms.retain(|id, _| id != &1)` — exit code for pid=1 permanently removed.**
5. Root (pid=0) runs and calls `ckb_wait(1)`:
   - `terminated_vms.get(&1)` → `None` — entry was consumed by Child B.
   - `states.contains_key(&1)` → `false` — Child A was removed from states.
   - Scheduler returns `WAIT_FAILURE` to root VM.
6. Root VM receives `WAIT_FAILURE`, fails, transaction is rejected.
