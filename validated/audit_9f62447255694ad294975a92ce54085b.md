### Title
Permanent User Fund Lock via Unguarded `parsePriceFeedUpdates` Revert in `Echo.executeCallback()` — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` calls `pyth.parsePriceFeedUpdates` with an exact-timestamp constraint (`req.publishTime` as both `minPublishTime` and `maxPublishTime`). If no Pyth price update exists at exactly that timestamp, the external call reverts. There is no `try/catch` around this call, so the entire `executeCallback` reverts. Because the Echo contract has no cancel/refund mechanism, the user's pre-paid fee is permanently locked. The contract's own TODO comment at line 155–156 explicitly acknowledges this risk.

---

### Finding Description

In `Echo.executeCallback()`, after validating the request and price IDs, the function makes an external call to the Pyth oracle:

```solidity
// Lines 144–153
IPyth pyth = IPyth(_state.pyth);
uint256 pythFee = pyth.getUpdateFee(updateData);
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
    value: pythFee
}(
    updateData,
    priceIds,
    SafeCast.toUint64(req.publishTime),   // minPublishTime = exact timestamp
    SafeCast.toUint64(req.publishTime)    // maxPublishTime = exact timestamp
);
```

Both `minPublishTime` and `maxPublishTime` are set to `req.publishTime`, requiring the submitted `updateData` to contain a price update with **exactly** that timestamp. If no such update exists (e.g., Pyth had a brief outage, the user chose a timestamp between two Pyth publish slots, or the data has been pruned from Hermes), `parsePriceFeedUpdates` reverts with `PriceFeedNotFoundWithinRange`.

Because there is no `try/catch` around this call, the revert propagates and the entire `executeCallback` transaction fails. The request is never cleared (`clearRequest` at line 164 is never reached), and the provider fee accounting at lines 161–162 is never committed. The user's `req.fee` — paid at request time — remains locked in the contract with no recovery path.

The developers themselves flag this exact risk at lines 155–156:

```solidity
// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
// If executeCallback can revert, then funds can be permanently locked in the contract.
```

There is no `cancelRequest`, `refundRequest`, or timeout-based expiry function anywhere in the Echo contract. The only way to clear a request is via a successful `executeCallback`.

Additionally, a permanently stuck request at sequence number `N` prevents `_state.firstUnfulfilledSeq` from advancing past `N`, causing `getFirstActiveRequests` to perpetually surface the stuck request to providers, wasting their gas on repeated failing attempts.

---

### Impact Explanation

A user who calls `requestPriceUpdatesWithCallback` with a `publishTime` for which no Pyth price update exists at that exact second will have their entire `msg.value` (minus the Pyth protocol fee already accrued at line 99) permanently locked in the Echo contract. There is no admin escape hatch, no user-initiated refund, and no timeout expiry. This constitutes a direct, irreversible loss of user funds.

---

### Likelihood Explanation

- Pyth publishes prices at ~400 ms intervals; multiple updates can share the same Unix second. However, if a user sets `publishTime` to a second during a Pyth network gap, maintenance window, or data-source outage, no update will exist at that exact timestamp.
- The `publishTime` is user-controlled (constrained only to `<= block.timestamp + 60`). A user who sets `publishTime = block.timestamp` during a Pyth outage, or who sets it to a past second with no coverage, triggers the lock.
- No privileged access is required. Any unprivileged user making a normal `requestPriceUpdatesWithCallback` call can end up in this state.
- The risk is elevated for any chain where Pyth coverage is intermittent or where the Hermes service has gaps.

---

### Recommendation

1. **Add a `try/catch` around `parsePriceFeedUpdates`** so that a revert from the Pyth oracle does not block the entire `executeCallback`. On failure, emit an event and allow the provider to retry with corrected data without reverting.
2. **Widen the timestamp window**: use `[req.publishTime - delta, req.publishTime + delta]` instead of exact matching, so minor timing mismatches do not cause permanent failures.
3. **Add a request cancellation / expiry mechanism**: allow the original requester to reclaim their fee after a timeout (e.g., if the request is unfulfilled after `N` blocks), analogous to how Entropy's `CALLBACK_FAILED` state allows recovery.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `publishTime = T` (a second during a Pyth outage) and pays `fee = F` wei.
2. Request is stored with `req.publishTime = T`, `req.fee = F - pythFeeInWei`.
3. Provider calls `executeCallback` with any `updateData`. `parsePriceFeedUpdates` is called with `minPublishTime = maxPublishTime = T`.
4. Pyth has no price update at exactly `T`; `parsePriceFeedUpdates` reverts with `PriceFeedNotFoundWithinRange`.
5. The entire `executeCallback` reverts. `clearRequest` is never called. `req.fee` remains in the contract.
6. No `cancelRequest` or `refund` function exists. Funds are permanently locked. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
