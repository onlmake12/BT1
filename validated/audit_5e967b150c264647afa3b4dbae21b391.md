### Title
Uncapped CKB-VM Cycle Execution via Peer-Declared `declared_cycles` in Remote Transaction Verification — (`tx-pool/src/process.rs`)

---

### Summary

A remote P2P peer can force a CKB node to run CKB-VM script execution for up to `max_block_cycles` (the consensus-level maximum, far exceeding the operator-configured `max_tx_verify_cycles`) per relayed transaction. The node uses the peer-supplied `declared_cycles` value directly as the CKB-VM cycle cap in `_process_tx`, with no enforcement that it stays within `max_tx_verify_cycles`. This is a direct analog to the gas-griefing class described in the report: an attacker-controlled input causes the victim node to perform maximum-cost computation with minimal attacker cost.

---

### Finding Description

In `tx-pool/src/process.rs`, the function `_process_tx` determines the CKB-VM cycle limit as follows:

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
```

For remote transactions, `declared_cycles = Some(peer_supplied_value)`. This value is then passed directly to `verify_rtx` as the cycle cap:

```rust
let verified_ret = verify_rtx(
    Arc::clone(&snapshot),
    Arc::clone(&rtx),
    tx_env,
    &verify_cache,
    max_cycles,   // ← peer-controlled
    command_rx,
)
.await;
```

The only upstream guard in `TransactionsProcess::execute()` is:

```rust
if txs.iter().any(|(_, declared_cycles)| declared_cycles > &max_block_cycles) {
    self.nc.ban_peer(...);
    return Status::ok();
}
```

This rejects `declared_cycles > max_block_cycles` but permits `declared_cycles == max_block_cycles`. The operator-configured `max_tx_verify_cycles` (default `TWO_IN_TWO_OUT_CYCLES * 20 ≈ 70 million cycles`) is never enforced as a hard cap on verification; it is only used to route transactions to a "large-cycle" worker queue in `VerifyQueue::add_tx`:

```rust
let is_large_cycle = remote
    .map(|(cycles, _)| cycles > self.large_cycle_threshold)
    .unwrap_or(false);
```

No rejection occurs here. The transaction proceeds to full CKB-VM execution with `max_cycles = declared_cycles`.

---

### Impact Explanation

A single peer can craft a transaction whose lock/type script is designed to consume exactly `max_block_cycles` cycles before failing (non-zero exit). The node runs CKB-VM for the full `max_block_cycles` budget — orders of magnitude more than `max_tx_verify_cycles` — before rejecting the transaction. The peer is then banned, but the attacker can reconnect from a new IP and repeat. The cost to the attacker is only the minimum transaction fee (as low as ~200 shannons with default `min_fee_rate = 1000 shannons/KW`) plus one live cell. The cost to the node is sustained CPU saturation of the verify worker thread(s), which blocks legitimate transaction processing.

---

### Likelihood Explanation

The attack path is fully reachable by any unprivileged P2P peer. The two-step relay protocol (`RelayTransactionHashes` → `GetRelayTransactions` → `RelayTransactions`) is the standard transaction propagation mechanism. The attacker controls the `cycles` field in the `RelayTransaction` molecule struct. The only prerequisite is a small amount of CKB to fund the transaction fee and a script cell that loops for `max_block_cycles` cycles. Such scripts are trivially constructable. The per-attack cost is negligible; the per-attack CPU cost to the victim is maximal.

---

### Recommendation

1. In `_process_tx`, cap `max_cycles` at `self.tx_pool_config.max_tx_verify_cycles` when processing remote transactions:
   ```rust
   let max_cycles = declared_cycles
       .map(|c| c.min(self.tx_pool_config.max_tx_verify_cycles))
       .unwrap_or_else(|| self.consensus.max_block_cycles());
   ```
2. In `TransactionsProcess::execute()`, add an early rejection for `declared_cycles > max_tx_verify_cycles` (not just `> max_block_cycles`) and ban the peer, consistent with the config comment "tx pool rejects txs that cycles greater than max_tx_verify_cycles".
3. Align the config documentation with actual enforcement behavior.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker connects as a P2P peer via `RelayV3`.
2. Sends `RelayTransactionHashes` containing hash `H` of a crafted transaction `T`.
3. Node adds `H` to `unknown_tx_hashes` and sends `GetRelayTransactions` for `H`.
4. Attacker responds with `RelayTransactions` containing `T` with `cycles = max_block_cycles`.
5. `TransactionsProcess::execute()` passes the `declared_cycles <= max_block_cycles` check.

**Node-side execution:**

- `submit_remote_tx(T, max_block_cycles, peer)` → `resumeble_process_tx` → `enqueue_verify_queue`
- Verify worker calls `_process_tx(T, Some(max_block_cycles), ...)`
- `pre_check` passes (fee satisfied, cell live)
- `max_cycles = max_block_cycles` (peer-supplied, uncapped)
- `verify_rtx(..., max_block_cycles, ...)` runs CKB-VM for up to `max_block_cycles` cycles
- Script exits non-zero → `Reject::Verification(ScriptError::ValidationFailure)` → `is_malformed_tx() = true` → peer banned

**Repeat:** Attacker reconnects from new IP, repeats with a new cell. Each iteration saturates one verify worker for the full `max_block_cycles` budget.

```rust
pub(crate) async fn verify_rtx(
    snapshot: Arc<Snapshot>,
    rtx: Arc<ResolvedTransaction>,
    tx_env: Arc<TxVerifyEnv>,
    cache_entry: &Option<CacheEntry>,
    max_tx_verify_cycles: Cycle,
    command_rx: Option<&mut watch::Receiver<ChunkCommand>>,
) -> Result<Completed, Reject> {
    let consensus = snapshot.cloned_consensus();
    let data_loader = snapshot.as_data_loader();

    if let Some(completed) = cache_entry {
        TimeRelativeTransactionVerifier::new(rtx, consensus, data_loader, tx_env)
            .verify()
            .map(|_| *completed)
            .map_err(Reject::Verification)
    } else if let Some(command_rx) = command_rx {
        ContextualTransactionVerifier::new(
            Arc::clone(&rtx),
            consensus,
            data_loader,
            Arc::clone(&tx_env),
        )
        .verify_with_pause(max_tx_verify_cycles, command_rx)
        .await
        .and_then(|result| {
            DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
                .verify()?;
            Ok(result)
        })
        .map_err(Reject::Verification)
    } else {
        block_in_place(|| {
            ContextualTransactionVerifier::new(Arc::clone(&rtx), consensus, data_loader, tx_env)
                .verify(max_tx_verify_cycles, false)
                .and_then(|result| {
                    DaoScriptSizeVerifier::new(
                        rtx,
                        snapshot.cloned_consensus(),
                        snapshot.as_data_loader(),
                    )
                    .verify()?;
                    Ok(result)
                })
                .map_err(Reject::Verification)
        })
    }
```
