### Title
`CapacityVerifier` Skips Entire `OutputsSumOverflow` Check When Any Input Uses DAO Type Script, Allowing Non-DAO Capacity Inflation — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`CapacityVerifier::verify()` unconditionally skips the `OutputsSumOverflow` check for the entire transaction whenever `valid_dao_withdraw_transaction()` returns true. That helper returns true if **any** resolved input carries the DAO type script. An attacker who controls a single DAO cell can therefore include arbitrary non-DAO inputs and outputs in the same transaction, inflating non-DAO output capacity beyond non-DAO input capacity with no enforcement, because the DAO type script only validates DAO-cell withdrawal amounts and never audits the total transaction capacity balance.

---

### Finding Description

**Root cause — `CapacityVerifier::verify()`** [1](#0-0) 

```rust
pub fn verify(&self) -> Result<(), Error> {
    if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
        let inputs_sum = self.resolved_transaction.inputs_capacity()?;
        let outputs_sum = self.resolved_transaction.outputs_capacity()?;
        if inputs_sum < outputs_sum {
            return Err(TransactionError::OutputsSumOverflow { ... }.into());
        }
    }
    // per-output occupied-capacity check still runs, but does NOT
    // prevent outputs from exceeding inputs in aggregate
    ...
}
```

The guard at line 483 is a binary all-or-nothing: if the transaction is classified as a DAO withdraw, the **entire** `OutputsSumOverflow` check is skipped for every cell in the transaction, not just for the DAO cells.

**Bypass trigger — `valid_dao_withdraw_transaction()`** [2](#0-1) 

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
```

`any(...)` returns `true` as soon as a single input carries the DAO type script. One minimal DAO cell is sufficient to suppress the capacity conservation check for the whole transaction.

**Why the DAO type script does not compensate**

`DaoCalculator::transaction_maximum_withdraw()` iterates resolved inputs and only applies the interest formula to cells that `is_dao_output` — all other inputs are passed through at face value: [3](#0-2) 

```rust
if is_dao_output {
    // ... compute interest-adjusted withdrawal capacity
} else {
    Ok(output.capacity().into())   // non-DAO cell: raw capacity, no conservation check
}
```

The on-chain DAO type script (C VM) mirrors this: it only verifies the DAO cell's withdrawal amount against the deposit/withdrawal block AR ratio. It never asserts `total_outputs_capacity ≤ total_inputs_capacity + dao_interest`. The comment in `CapacityVerifier` acknowledges this split responsibility but the implementation leaves a gap: [4](#0-3) 

```
// skip OutputsSumOverflow verification for resolved cellbase and DAO
// withdraw transactions.
// DAO withdraw transaction is verified via the type script of DAO cells
```

The type script verifies DAO cells; nothing verifies non-DAO cells when the bypass fires.

**Exploit path (step-by-step)**

1. Attacker deposits a minimal DAO cell (e.g., 102 CKB minimum occupied capacity) to obtain a live DAO cell.
2. Attacker also holds a regular cell with, say, 1 000 CKB.
3. Attacker constructs a transaction:
   - **Input 0**: DAO cell (102 CKB) — triggers `valid_dao_withdraw_transaction() = true`
   - **Input 1**: Regular cell (1 000 CKB)
   - **Output 0**: DAO withdrawal cell (102 CKB + interest ≈ 103 CKB) — passes DAO type script
   - **Output 1**: Regular cell (2 000 CKB) — 1 000 CKB inflation, **no check**
4. `CapacityVerifier::verify()` enters the `if` branch, sees `valid_dao_withdraw_transaction() = true`, and skips `OutputsSumOverflow`.
5. The per-output `InsufficientCellCapacity` check at lines 496–512 passes because 2 000 CKB ≥ occupied capacity of Output 1.
6. The DAO type script accepts Output 0 as a valid withdrawal.
7. The transaction is committed; the attacker has created 1 000 CKB from nothing.

---

### Impact Explanation

An unprivileged transaction sender who owns any DAO cell can inflate non-DAO output capacity beyond non-DAO input capacity in a single transaction. This violates CKB's fundamental capacity-conservation invariant, allowing arbitrary CKB issuance. Every committed such transaction permanently inflates the total supply, diluting all holders and breaking the economic security model of the chain. Impact is **critical** (arbitrary token minting, consensus-level invariant violation).

---

### Likelihood Explanation

The precondition is owning a live DAO cell, which requires only a prior deposit transaction — a normal, permissionless user action. No privileged key, no majority hashpower, no social engineering. Any user who has ever deposited into Nervos DAO satisfies the precondition. Likelihood is **high** given the low barrier.

---

### Recommendation

Replace the all-or-nothing bypass with a capacity-aware comparison that accounts for legitimate DAO interest:

1. Compute `dao_interest` = `DaoCalculator::transaction_maximum_withdraw(rtx)? - rtx.inputs_capacity()?` for the DAO inputs only.
2. Enforce `inputs_sum + dao_interest >= outputs_sum` unconditionally (even for DAO withdraw transactions).

This preserves the legitimate DAO withdrawal surplus while closing the gap for non-DAO cells. Alternatively, split the check: verify non-DAO inputs vs non-DAO outputs independently of DAO cells, so the DAO type script and `CapacityVerifier` each cover their own domain without overlap or gap.
