### Title
Permanent Locking of User Funds Due to Untracked `req.fee` When `executeCallback` Reverts on Underflow — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, user-paid fees stored in `req.fee` are **not tracked in any withdrawable accounting variable**. If `executeCallback` reverts due to an arithmetic underflow — triggered when the actual Pyth oracle fee (`pythFee`) exceeds `req.fee + msg.value` — the user's funds are permanently locked with no admin rescue path and no cancellation/refund mechanism.

---

### Finding Description

**At request time** in `requestPriceUpdatesWithCallback`: [1](#0-0) 

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);   // L84
...
_state.accruedFeesInWei += _state.pythFeeInWei;                 // L99
```

Only `_state.pythFeeInWei` (the Echo protocol fee) is credited to the withdrawable `_state.accruedFeesInWei`. The remainder — `req.fee = msg.value - _state.pythFeeInWei` — is stored only in the request struct, **not in any withdrawable accounting variable**.

**At execution time** in `executeCallback`: [2](#0-1) 

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
...
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // L161-162
```

If `pythFee > req.fee + msg.value`, the subtraction **underflows and reverts** (Solidity 0.8+ checked arithmetic). The request is never cleared, and `req.fee` remains in the contract with no path to withdrawal.

The code itself acknowledges the general risk at line 155–156: [3](#0-2) 

> "TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract."

However, the fee-underflow revert path is not addressed.

**Withdrawal functions only cover tracked variables:** [4](#0-3) 

`withdrawFees` drains `_state.accruedFeesInWei` (Echo protocol fee only). `withdrawAsFeeManager` drains `provider.accruedFeesInWei`. Neither can reach `req.fee` stored in the request struct. [5](#0-4) 

There is no cancellation or refund function anywhere in the contract.

---

### Impact Explanation

For every request where `pythFee > req.fee + msg.value_exec`:

- `req.fee = msg.value_request - _state.pythFeeInWei` is permanently locked.
- The admin can only recover `_state.pythFeeInWei` (the Echo protocol fee portion).
- The provider portion — which constitutes the majority of the user's payment — is irrecoverable.
- No privileged role (admin, fee manager) has any function to rescue these funds.

---

### Likelihood Explanation

The Pyth oracle fee (`pyth.getUpdateFee(updateData)`) is a live on-chain value that can change via Pyth governance. If the Pyth oracle fee rises after a request is submitted:

1. The stored `req.fee` was computed against the old fee level.
2. Executors calling `executeCallback` with `msg.value = 0` will hit the underflow revert.
3. Executors are not obligated to subsidize the shortfall; if the gap is large enough, no executor will fulfill the request.
4. The user has no recourse — no cancel, no refund, no timeout.

This is directly reachable by any unprivileged user who calls `requestPriceUpdatesWithCallback`.

---

### Recommendation

1. **Add a cancellation/refund path**: Allow users (or the admin) to cancel unfulfilled requests after a timeout and refund `req.fee` to the requester.
2. **Track `req.fee` in a withdrawable variable**: Credit `req.fee` to `_state.accruedFeesInWei` at request time (not just `pythFeeInWei`), so the admin can rescue funds if requests become permanently unfulfillable.
3. **Guard against underflow explicitly**: Before crediting the provider, check `req.fee + msg.value >= pythFee` and handle the deficit gracefully (e.g., revert with a clear error, or allow partial fulfillment with executor subsidy).

---

### Proof of Concept

1. `_state.pythFeeInWei = 100 wei`; provider fee = 900 wei; `requiredFee = 1000 wei`.
2. User calls `requestPriceUpdatesWithCallback{value: 1000}(...)`.
   - `req.fee = 900 wei` stored in request struct.
   - `_state.accruedFeesInWei += 100 wei` (withdrawable by admin).
3. Pyth governance raises the oracle fee. Now `pyth.getUpdateFee(updateData) = 1000 wei`.
4. Executor calls `executeCallback{value: 0}(...)`.
   - `pythFee = 1000`
   - `(req.fee + msg.value) - pythFee = (900 + 0) - 1000` → **underflow → revert**.
5. Request remains active. User's 900 wei is permanently locked.
6. Admin calls `withdrawFees(100)` — recovers only the 100 wei Echo protocol fee.
7. The 900 wei in `req.fee` has no withdrawal path. No function in `Echo.sol` can reach it.
