Audit Report

## Title
Unbounded Inline Script Verification in `process_orphan_tx` Enables CPU-Exhaustion DoS — (File: `tx-pool/src/process.rs`)

## Summary

`process_orphan_tx` performs synchronous BFS-driven CKB-VM script verification for every orphan resolved by a newly accepted parent, with no per-invocation cap on total work. An unprivileged relay peer can fill the 100-entry orphan pool with transactions all spending distinct outputs of one crafted parent, then submit the parent to trigger up to 100 consecutive inline `_process_tx` calls — each running full script verification with `command_rx = None` — causing CPU exhaustion and repeated tx-pool write-lock contention that degrades transaction processing for all peers.

## Finding Description

**Root cause — `process_orphan_tx` (lines 591–671, `tx-pool/src/process.rs`)**

When a transaction is accepted, `after_process` unconditionally calls `process_orphan_tx` at lines 500 and 536. Inside, a BFS loop calls `find_orphan_by_previous` and, for every orphan whose `declared_cycle ≤ max_tx_verify_cycles`, invokes `_process_tx(orphan.tx.clone(), Some(orphan.cycle), None)` at line 626. The `None` argument is `command_rx`, which disables chunked/interruptible execution — `verify_rtx` runs to completion with no pause mechanism.

The orphan pool is hard-capped at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` (orphan.rs line 16), but `limit_size` enforces only the total count with no per-parent-output sub-limit (orphan.rs lines 96–132). `find_by_previous` (orphan.rs lines 161–167) returns every orphan referencing any output of the given transaction, so all 100 slots can be filled with orphans spending distinct outputs of one parent.

**Attack path:**
1. Attacker crafts a parent transaction with ≥ 100 outputs.
2. Attacker relays 100 orphan transactions via RelayV3, each spending a different output of the parent. Each is accepted into the orphan pool (parent unknown).
3. Attacker submits the parent. `after_process` → `process_orphan_tx` → `find_orphan_by_previous` returns all 100 orphans.
4. For each orphan: `_process_tx(..., None)` → `verify_rtx(..., command_rx=None)` runs full CKB-VM verification synchronously, up to `max_tx_verify_cycles` per orphan.
5. Total inline work: up to `100 × max_tx_verify_cycles` CKB-VM cycles in a single call. Each `_process_tx` also acquires and releases the tx-pool write lock via `pre_check` and `submit_entry`, serialising all concurrent submissions.

**Amplification via `remove_orphan_txs_by_attach` (lines 870–876):** During a chain reorg, `process_orphan_tx` is called for every transaction in every attached block, multiplying the total inline verification work by the number of attached transactions if the orphan pool is kept full.

**Existing guards are insufficient:** `limit_size` only prevents the pool from exceeding 100 entries total; it does not limit how many orphans may reference the same parent output. There is no per-peer or per-parent-output admission limit in `add_orphan_tx`.

## Impact Explanation

This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* A single trigger event causes up to 100 consecutive synchronous CKB-VM verifications, each consuming up to `max_tx_verify_cycles` cycles (≈70 M on mainnet), totalling ≈7 billion cycles of CPU work per attack invocation. Concurrent write-lock acquisitions during each `_process_tx` call block all other pending transaction submissions for the duration. The attack can be repeated continuously at low cost (cheap transactions, no special privileges), sustaining degradation of tx-pool throughput across the network.

## Likelihood Explanation

Any peer connected via `SupportProtocols::RelayV3` can execute this attack. No special privileges, keys, or on-chain funds beyond minimal transaction fees are required. Crafting a parent with 100 outputs and 100 child orphans is straightforward. The orphan pool accepts submissions from any peer up to the 100-entry cap with no per-peer or per-parent-output rate limit. The attack is repeatable: once the orphan pool is drained by a trigger, it can be refilled and triggered again.

## Recommendation

1. **Cap inline orphan processing per invocation.** Introduce a maximum number of orphans processed inline per `process_orphan_tx` call (e.g., 10). Defer remaining orphans to the async verify queue.
2. **Limit orphans per parent output.** In `add_orphan_tx`, cap the number of orphan entries that may reference the same `OutPoint` (e.g., 5), preventing a single parent from triggering a full-pool sweep.
3. **Route all orphan verification through the verify queue.** Remove the inline `_process_tx` path in `process_orphan_tx` entirely; always enqueue orphans to the async `VerifyQueue` regardless of declared cycle count, decoupling orphan resolution from the calling task and subjecting it to existing queue size and rate controls.

## Proof of Concept

```
1. Connect to a CKB node via the RelayV3 P2P protocol.

2. Construct parent_tx with 100 outputs, each a minimal always-success cell.

3. For i in 0..100:
     child_i = Transaction {
         inputs:  [OutPoint(parent_tx.hash(), i)],
         outputs: [one output],
         witnesses: [always-success witness],
         declared_cycles: ALWAYS_SUCCESS_SCRIPT_CYCLE  // ≤ max_tx_verify_cycles
     }
     relay child_i → accepted into orphan pool (parent unknown)

4. After all 100 children are in the orphan pool, relay parent_tx.
   → parent_tx accepted into pending pool.
   → after_process (line 500) calls process_orphan_tx(parent_tx).
   → find_orphan_by_previous returns all 100 children.
   → For each child: _process_tx called with command_rx = None (line 626),
     running full CKB-VM verification synchronously.
   → 100 consecutive script verifications execute inline, each acquiring
     and releasing the tx-pool write lock.

5. Concurrently submit legitimate transactions from a second peer and
   observe significantly increased latency or timeouts on tx-pool RPC calls,
   confirming lock contention and CPU exhaustion.

6. Repeat from step 3 to sustain the degradation.
```
