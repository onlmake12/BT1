Audit Report

## Title
Unbounded Growth of `pending_get_block_proposals` Causes Resource Exhaustion in `prune_tx_proposal_request` - (File: sync/src/relayer/mod.rs, sync/src/types/mod.rs)

## Summary
`SyncState::pending_get_block_proposals` is an uncapped `DashMap` that any connected peer can fill by repeatedly sending `GetBlockProposal` relay messages containing non-existent proposal IDs. The periodic `prune_tx_proposal_request` drains and forwards the entire map to the tx-pool service in a single unbounded call, causing proportional memory, CPU, and tx-pool service degradation that can crash or stall a CKB node.

## Finding Description
`pending_get_block_proposals` is declared with no capacity bound: [1](#0-0) 

`insert_get_block_proposals` inserts every peer-supplied ID with no total-size guard: [2](#0-1) 

`GetBlockProposalProcess::execute()` enforces only a per-message count limit of `max_block_proposals_limit * max_uncles_num` (typically 1500 × 2 = 3 000): [3](#0-2) 

Proposals absent from the local tx-pool are then unconditionally inserted into the map with no validation that they correspond to any known compact block: [4](#0-3) 

`drain_get_block_proposals` clones the entire map and clears it — returning the full, unbounded snapshot: [5](#0-4) 

`prune_tx_proposal_request` then forwards all drained IDs to the tx-pool service in a single `fetch_txs` call and iterates every entry: [6](#0-5) 

There is no per-peer rate limit, no global map-size cap, and no check that proposal IDs correspond to any previously received compact block. An attacker with N connections, each sending 3 000 unique random IDs per message, accumulates `N × 3000 × message_rate` entries between timer ticks.

## Impact Explanation
The concrete impact is a node crash or sustained unresponsiveness matching the **High** bounty tier: "Vulnerabilities which could easily crash a CKB node." Memory grows proportionally to injected entries (each a 10-byte key plus a `HashSet<PeerIndex>`). On each timer tick, the node clones the entire map, sends an arbitrarily large `HashSet` to the single-threaded tx-pool service, and iterates all entries — stalling tx admission, block assembly, and all other tx-pool callers. Sustained attack keeps the node perpetually degraded without any privileged access.

## Likelihood Explanation
Any unauthenticated peer can send `GetBlockProposal` messages. No stake, role, or special privilege is required. The per-message limit of ~3 000 IDs is generous, and there is no per-peer rate limit or global cap anywhere in the code path. A single attacker with a handful of connections can continuously saturate the map between timer ticks, making the attack cheap and repeatable.

## Recommendation
1. **Cap the total map size**: enforce a `MAX_INFLIGHT_PROPOSALS` constant in `insert_get_block_proposals` and drop or evict entries when the cap is reached.
2. **Validate against known compact blocks**: only accept proposal IDs that appear in a compact block already stored in `pending_compact_blocks`, rejecting unsolicited IDs.
3. **Chunk the drain**: in `prune_tx_proposal_request`, process the drained map in bounded batches rather than forwarding the entire set to the tx-pool in one `fetch_txs` call.
4. **Per-peer rate limiting**: track per-peer injection counts and disconnect or ban peers exceeding a threshold per interval.

## Proof of Concept
1. Open N connections to the victim CKB node.
2. From each connection, repeatedly send `GetBlockProposal` relay messages containing 3 000 unique, randomly generated 10-byte `ProposalShortId` values.
3. Because none exist in the victim's tx-pool, all are inserted into `pending_get_block_proposals` via `insert_get_block_proposals` with no size check.
4. Between timer ticks the map accumulates `N × 3000 × message_rate` entries.
5. On the next `prune_tx_proposal_request` tick, the node clones the entire map, sends all IDs to the tx-pool service in a single `fetch_txs` call, and iterates all entries.
6. Monitor node memory, CPU, and tx-pool response latency — all grow proportionally to injected entries. Sustained attack renders the node unresponsive.
