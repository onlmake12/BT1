Audit Report

## Title
Unbounded TCP/WebSocket Connection and Subscription Growth Causes Node Resource Exhaustion — (`rpc/src/server.rs`, `rpc/src/module/subscription.rs`)

## Summary

The TCP accept loop in `start_tcp_server` spawns an unlimited number of tokio tasks with no connection cap. `SubscriptionRpcImpl::subscribe()` unconditionally calls `tx.subscribe()` with no per-connection or global receiver count guard. An attacker who can reach `tcp_listen_address` or `ws_listen_address` can open arbitrarily many connections, subscribe to all six topics per connection, and exhaust memory, file descriptors, and the tokio task scheduler — crashing the node.

## Finding Description

**Unbounded TCP accept loop:**

`start_tcp_server` accepts connections in an infinite loop with no semaphore, counter, or backpressure mechanism:

```rust
while let Ok((stream, _)) = listener.accept().await {
    tokio::spawn(async move { ... });
}
``` [1](#0-0) 

Every accepted connection unconditionally spawns a new `tokio::spawn` task. There is no `max_connections` field in `RpcConfig`: [2](#0-1) 

**Uncapped subscription creation:**

`subscribe()` calls `tx.subscribe()` unconditionally, creating a new `broadcast::Receiver` and a persistent async stream task per call. There is no check on `tx.receiver_count()`, no per-connection subscription cap, and no global limit: [3](#0-2) 

Six `broadcast::Sender` instances are created at startup, one per topic: [4](#0-3) 

**Lagged receivers are never dropped:**

When a slow/non-reading client falls behind, `RecvError::Lagged` is logged but the loop continues — the receiver and its driving task remain alive indefinitely: [5](#0-4) 

A grep across all of `rpc/` for `receiver_count`, `max_connections`, `connection_limit`, and `semaphore` returns zero matches, confirming no existing guard exists anywhere in the RPC layer.

**Resource growth mechanics:**

- N connections × 6 topics = **6N live tokio tasks** and **6N broadcast receivers**.
- On every new block, `broadcast::Sender::send()` wakes all N receivers for `new_tip_header` and `new_tip_block` simultaneously — an O(N) burst of task wakeups in the tokio scheduler.
- Each TCP connection consumes one OS file descriptor.

## Impact Explanation

This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

At N=50,000 connections, tens of thousands of tokio task stacks and receiver structs consume gigabytes of RSS. The O(N) wakeup burst on each new block stalls block validation, peer relay, and tx-pool processing tasks sharing the same tokio runtime. FD exhaustion is reached before memory on most Linux defaults (`ulimit -n 1048576` is common but reachable). The combined effect is a sustained degradation leading to node halt.

## Likelihood Explanation

The `tcp_listen_address` and `ws_listen_address` endpoints are opt-in (commented out in the default `resource/ckb.toml`): [6](#0-5) 

However, any node operator using the subscription feature must enable one of these endpoints. Once enabled, the endpoint is network-accessible with no authentication. TCP connections are cheap; an attacker on the same network (or the internet, if the port is exposed) can open tens of thousands of connections from a single machine using standard socket APIs. No credentials, proof-of-work, or special privileges are required. The exploit is linear and deterministic.

## Recommendation

1. **Connection limit**: Add a `max_connections: Option<usize>` field to `RpcConfig` and enforce it in `start_tcp_server` using a `tokio::sync::Semaphore` before calling `tokio::spawn`.
2. **Per-connection subscription limit**: In `subscribe()`, check `tx.receiver_count()` against a configurable maximum and return a JSON-RPC error if exceeded.
3. **Global subscription limit**: Maintain an `Arc<AtomicUsize>` counter across all topics and connections, incremented on subscribe and decremented when the stream terminates.
4. **Idle connection timeout**: Drop connections that do not read within a configurable window (e.g., integrate with `tower_http::timeout` or a per-task `tokio::time::timeout`).
5. **Lagged receiver termination**: Change the `RecvError::Lagged` arm to `break` (or return an error to the client) rather than continuing the loop, so non-reading clients are evicted.

## Proof of Concept

```python
import socket, json

N = 10000
socks = []
for i in range(N):
    s = socket.socket()
    s.connect(("127.0.0.1", 18114))
    for topic in ["new_tip_block", "new_tip_header", "new_transaction",
                  "proposed_transaction", "rejected_transaction", "log"]:
        req = json.dumps({
            "id": i, "jsonrpc": "2.0",
            "method": "subscribe", "params": [topic]
        }) + "\n"
        s.sendall(req.encode())
    socks.append(s)

# Observe: node RSS grows O(N); `tokio` task count reaches ~6N;
# block-relay latency increases measurably on each new block.
input("Hold connections open...")
```

Run `watch -n1 'ps -o rss= -p $(pgrep ckb)'` in a second terminal to observe linear RSS growth. Tokio task count can be confirmed via the metrics endpoint or by attaching a debugger and inspecting the runtime. The `broadcast::Sender::receiver_count()` for each topic sender will equal N after the loop completes.
