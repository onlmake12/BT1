### Title
Governance Sequence Gap Allows Permanent Skip of Critical Actions — (`lazer/contracts/sui/sources/governance.move`)

### Summary

`governance::process_incoming` uses a strictly-less-than check (`seen_sequence < sequence`) instead of enforcing strict sequential ordering (`sequence == seen_sequence + 1`). Any unprivileged relayer holding two valid VAAs can submit the higher-sequence one first, permanently blocking the lower-sequence one from ever being processed.

### Finding Description

The vulnerable check is in `process_incoming`: [1](#0-0) 

```move
assert!(self.seen_sequence < sequence, EOldSequenceNumber);
self.seen_sequence = sequence;
```

This accepts **any** sequence number strictly greater than `seen_sequence`, not just the next one. If `seen_sequence = S` and a relayer submits a VAA with sequence `S+2`, the check `S < S+2` passes and `seen_sequence` is set to `S+2`. A subsequent attempt to submit the VAA with sequence `S+1` then fails because `S+2 < S+1` is false, triggering `EOldSequenceNumber` permanently.

The public entrypoints that any caller can invoke are: [2](#0-1) [3](#0-2) 

Both `upgrade` and `update_trusted_signer` are declared `public fun` (not restricted to a privileged role), so any transaction or Move module can call them with any valid Wormhole VAA.

### Impact Explanation

A governance action encoded in sequence `S+1` (e.g., removing a compromised or expired trusted signer, or authorizing a critical contract upgrade) is permanently unprocessable. The `State`'s `trusted_signers` vector will never reflect that action, leaving a malicious or expired signer active indefinitely, or blocking a required upgrade. [4](#0-3) 

### Likelihood Explanation

Wormhole VAAs are publicly observable once signed by the guardian set. A relayer who obtains VAA `S+2` (e.g., a routine signer expiry update) before or simultaneously with `S+1` (a signer removal) can trivially submit `S+2` first. No key compromise, privileged access, or oracle manipulation is required — only the ability to call a `public fun` with a valid VAA.

### Recommendation

Replace the less-than check with a strict next-sequence check:

```move
// Before (vulnerable):
assert!(self.seen_sequence < sequence, EOldSequenceNumber);

// After (fixed):
assert!(sequence == self.seen_sequence + 1, EOldSequenceNumber);
```

This enforces that every governance message is processed in exact order, making it impossible to skip any sequence number.

### Proof of Concept

1. Deploy the contract with `seen_sequence = 0`.
2. Obtain two valid guardian-signed VAAs: sequence `1` (removes a compromised signer) and sequence `2` (routine update).
3. Call `actions::update_trusted_signer(state, vaa_seq_2)` — passes because `0 < 2`; `seen_sequence` becomes `2`.
4. Call `actions::update_trusted_signer(state, vaa_seq_1)` — aborts with `EOldSequenceNumber` because `2 < 1` is false.
5. Assert that the compromised signer is still present in `state.trusted_signers`, confirming the removal was permanently skipped. [5](#0-4)

    self: &mut Governance,
    vaa: VAA
): vector<u8> {

    let (chain_id, address, payload) = vaa.take_emitter_info_and_payload();
    assert!(self.chain_id == chain_id, EMismatchedEmitterChainID);
    assert!(self.address == address, EMismatchedAddress);
    assert!(self.seen_sequence < sequence, EOldSequenceNumber);
    self.seen_sequence = sequence;
    payload

public fun update_trusted_signer(state: &mut State, vaa: VAA) {
    let current_cap = state.current_cap();
    let (header, mut parser) = state.unwrap_ptgm(&current_cap, vaa);
    assert!(header.is_update_trusted_signer(), ENotUpdateTrustedSigner);

    let public_key = parser.take_bytes(secp256k1_compressed_pubkey_len());
    let expires_at = parser.take_u64_be();
    parser.destroy_empty();
    state.update_trusted_signer(&current_cap, public_key, expires_at);
}
