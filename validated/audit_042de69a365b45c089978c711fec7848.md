Audit Report

## Title
Overcounted `replace_count` in `check_rbf` Due to Non-Deduplicated Shared Descendants Causes False RBF Rejection — (File: `tx-pool/src/pool.rs`)

## Summary
In `TxPool::check_rbf`, `replace_count` is accumulated by summing `descendants.len() + 1` for each conflicting transaction independently. When two or more conflicting transactions share common descendants (possible because CKB transactions support multiple inputs), those shared descendants are counted multiple times. This causes `replace_count` to exceed `MAX_REPLACEMENT_CANDIDATES` (100) even when the actual number of unique transactions to replace is within the limit, resulting in false rejection of legitimate RBF transactions. An attacker with no special privileges can exploit this to permanently block a victim's RBF attempts for targeted UTXOs.

## Finding Description
In `check_rbf` at `tx-pool/src/pool.rs` lines 611–624:

```rust
let mut replace_count: usize = 0;
let mut all_conflicted = conflicts.clone();
let ancestors = self.pool_map.calc_ancestors(&short_id);
for conflict in conflicts.iter() {
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    replace_count += descendants.len() + 1;   // ← no dedup across conflicts
    if replace_count > MAX_REPLACEMENT_CANDIDATES {
        return Err(Reject::RBFRejected(...));
    }
    ...
    all_conflicted.extend(entries);            // ← duplicates accumulate here too
}
``` [1](#0-0) 

`calc_descendants` returns a `HashSet<ProposalShortId>` of all transitive children of a single transaction. [2](#0-1) 

Because CKB transactions support multiple inputs, a single descendant `TX_C` can legitimately spend outputs from two different conflicting transactions `TX_A` and `TX_B` simultaneously. In that case, `calc_descendants(TX_A)` and `calc_descendants(TX_B)` both return `{TX_C}`, and `replace_count` counts it twice.

**Exploit path:**
1. Attacker submits `TX_A` (spends on-chain cell `X`), `TX_B` (spends on-chain cell `Y`), and `TX_C` (spends output of `TX_A` and output of `TX_B` — valid multi-input structure).
2. Repeat with 50 such fan-in descendants shared between `TX_A` and `TX_B`.
3. Victim submits `TX_NEW` spending both `X` and `Y` with sufficient fee (RBF attempt).
4. `find_conflict_tx` returns `{TX_A, TX_B}`. [3](#0-2) 

5. Loop: `calc_descendants(TX_A)` = 50 shared descendants → `replace_count += 51`. `calc_descendants(TX_B)` = same 50 → `replace_count += 51`. Final `replace_count = 102 > 100`, rejected. Unique transactions to replace = 52 ≤ 100.

`MAX_REPLACEMENT_CANDIDATES` is defined as `100` at line 33. [4](#0-3) 

Notably, `calculate_min_replace_fee` correctly deduplicates via a `HashMap` keyed by `id`, so the fee check is unaffected — only the count check is broken. [5](#0-4) 

## Impact Explanation
This is a targeted DoS on the RBF mechanism: an attacker can permanently block a victim's ability to fee-bump or replace specific in-pool transactions. Legitimate RBF transactions are incorrectly rejected with `"Tx conflict with too many txs"` even when the true unique replacement count is within the protocol limit. This maps to **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**: if RBF is blocked for targeted UTXOs, victims cannot fee-bump stuck transactions, contributing to pool and network congestion. The attacker's cost is submitting three valid transactions.

## Likelihood Explanation
The attacker requires no special privilege, key, or majority hashpower. The entry path is the standard `send_transaction` RPC or P2P relay, accessible to any unprivileged submitter. The attacker only needs to know which UTXOs the victim intends to RBF-replace. The multi-input transaction structure used (`TX_C` spending outputs from two pool parents) is standard and valid CKB. The attack is repeatable and persistent as long as the attacker's transactions remain in the pool.

## Recommendation
Collect all descendants across all conflicts into a single `HashSet` before counting, so shared descendants are deduplicated:

```rust
let mut all_descendants: HashSet<ProposalShortId> = HashSet::new();
for conflict in conflicts.iter() {
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    if !descendants.is_disjoint(&ancestors) {
        return Err(Reject::RBFRejected(...));
    }
    // per-conflict input-in-descendants check still applies here
    ...
    all_descendants.extend(descendants);
}
let replace_count = conflicts.len() + all_descendants.len();
if replace_count > MAX_REPLACEMENT_CANDIDATES {
    return Err(Reject::RBFRejected(...));
}
```

This mirrors the deduplication already applied in `calculate_min_replace_fee` via `HashMap`. [6](#0-5) 

## Proof of Concept
**Pool state setup:**
- Submit `TX_A`: input = on-chain cell `X`, output[0] = cell `P`
- Submit `TX_B`: input = on-chain cell `Y`, output[0] = cell `Q`
- Submit `TX_C_1` through `TX_C_50`: each spends one output of `TX_A` and one output of `TX_B` (fan-in, multi-input)

**Trigger:**
- Submit `TX_NEW` via `send_transaction` RPC spending both `X` and `Y` with fee ≥ `min_replace_fee`

**Expected:** `replace_count = 52 ≤ 100`, RBF proceeds.

**Actual:** `replace_count = 2 × 51 = 102 > 100`, rejected with `"Tx conflict with too many txs, conflict txs count: 102, expect <= 100"`. [7](#0-6)
