### Title
Tx-Pool Submission-Order Dependency Permanently Rejects Consensus-Valid Transactions - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

The CKB tx-pool permanently rejects a consensus-valid transaction C (which uses cell A as a `cell_dep`) when transaction B (which spends cell A as an input) has already been submitted to the pool. This is an ordering-dependent admission failure: the block ordering `[C, B]` is consensus-valid, but the tx-pool's cell-resolution overlay marks A as dead the moment B is admitted, causing C to be rejected with `Reject::Resolve(OutPointError::Dead(...))` regardless of how many times it is resubmitted.

---

### Finding Description

The CKB tx-pool uses an `OverlayCellProvider` / `OverlayCellChecker` that layers pending-pool state on top of the chain snapshot. When transaction B is admitted to the pool and spends cell A, the pool's edge map records A's outpoint as consumed. Subsequently, when transaction C is submitted with A listed as a `cell_dep`, the `resolve_tx` call inside `pre_check` resolves A through the overlay and finds it dead (already claimed by B), returning `Reject::Resolve(OutPointError::Dead(...))`.

The `resolve_conflict` function in `pool_map.rs` is the state-mutating step: it registers every input outpoint of an admitted transaction into the pool's edge tracking. This is the exact analog of `restrictMinting` in the Frankencoin report — an intermediate admission step modifies shared state (the edge map) in a way that causes a subsequent, independently valid admission to fail.

The CKB codebase itself documents this as a known limitation in the integration test:

```
// NOTE: It MUST submit C before B. If submit C after B, the proposed pool will reject C as
// it thinks that B has already spent A; A is one of C's cell-deps; hence C is invalid. This
// is current tx-pool implementation limitation but not consensus rule.
```

The root cause is that the pool's cell-resolution layer does not distinguish between "A is dead because B spent it and B will appear before C in the block" versus "A is dead and C cannot coexist with B in any valid block." The consensus rule only forbids the block ordering `[B, C]`; the ordering `[C, B]` is valid. The pool, however, applies a blanket dead-cell rejection that is stricter than consensus. ### Impact Explanation

A consensus-valid transaction C is permanently excluded from the tx-pool and therefore from block assembly whenever B (spending the same cell A that C references as a `cell_dep`) is admitted first. C cannot be re-submitted successfully; the rejection is deterministic and persistent as long as B remains in the pool. This means:

1. **Transaction censorship via ordering**: any party who can get B into the pool before C is submitted can permanently block C from being mined through the normal tx-pool path.
2. **Block template incompleteness**: `get_block_template` will never include C even though a miner could manually construct a valid block `[C, B]`.
3. **User funds locked in scripts**: if C is a time-sensitive withdrawal or unlock transaction whose script depends on a cell-dep that is simultaneously being spent by another pending transaction, the user cannot get C mined via the standard RPC path. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The trigger requires only two transactions sharing a cell in the roles of "input" and "cell-dep" respectively — a pattern that arises naturally when a script library cell is being migrated (spent and replaced) while other transactions still reference it as a dep. Any unprivileged RPC caller or P2P relay peer who submits B first causes the rejection. No special privileges, keys, or majority hashpower are required. The scenario is realistic in production: script upgrade transactions and dependent transactions are routinely broadcast in close succession. [6](#0-5) 

---

### Recommendation

The tx-pool's `resolve_tx` / cell-resolution layer should be extended to recognize the case where a cell is dead-as-input in the pool but the incoming transaction references it only as a `cell_dep`. In that case, the pool should either:

1. Accept C into a separate "cell-dep-pending" bucket that is only promoted to the main pool after B is evicted or committed, mirroring how the orphan pool handles missing inputs; or
2. During `resolve_conflict` (when B is admitted), proactively check whether any existing pool entry uses B's inputs as `cell_dep`s and evict those entries with an `Invalidated` reject rather than silently blocking future submissions of C.

The fix is analogous to the Frankencoin mitigation: reorder the operations so that the state-modifying step (registering A as dead) does not block the admission of C, which is independently valid under the correct block ordering. [7](#0-6) 

---

### Proof of Concept

The CKB integration test suite already contains a direct proof of concept:

1. Commit transaction A (creates cell A).
2. Create transaction B spending cell A as an input.
3. Create transaction C using cell A as a `cell_dep`.
4. Submit B to the pool first (`send_transaction(B)`).
5. Submit C to the pool second (`send_transaction(C)`).
6. C is rejected with `Resolve(Dead(...))`.
7. Manually construct block `[C, B]` and call `submit_block` — it succeeds, proving C is consensus-valid.

The test at `test/src/specs/tx_pool/dead_cell_deps.rs:267-273` explicitly documents that the workaround is to submit C before B, confirming the ordering dependency is a real, reproducible, and acknowledged limitation of the production tx-pool code. [8](#0-7) [9](#0-8) [1](#0-0)
