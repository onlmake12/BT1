### Title
`MAX_DEPOSIT_LIMIT` for Permanent Subscriptions Can Be Arbitrarily Bypassed via Repeated `addFunds` Calls - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `MAX_DEPOSIT_LIMIT` cap enforced on permanent Scheduler subscriptions is checked only against the current transaction's `msg.value`, not against the total accumulated `balanceInWei`. This allows any caller to bypass the limit by calling `addFunds` repeatedly with amounts at or below `MAX_DEPOSIT_LIMIT`, accumulating an unbounded balance in a permanent subscription from which funds can never be withdrawn.

### Finding Description

In `Scheduler.sol`, permanent subscriptions are subject to a deposit cap of `MAX_DEPOSIT_LIMIT = 100 ether` to prevent excessive ETH from being permanently locked (since permanent subscriptions prohibit withdrawals). This cap is enforced in two places:

**`createSubscription` (line 48):**
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**`addFunds` (line 613):**
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Both checks compare only `msg.value` (the current deposit) against `MAX_DEPOSIT_LIMIT`, not the post-deposit total `status.balanceInWei + msg.value`. The `createSubscription` check is correct because it initializes a fresh balance. However, the `addFunds` check is flawed: after the initial deposit of up to 100 ETH, any caller can invoke `addFunds` repeatedly with up to 100 ETH each time, and each call passes the check independently. The balance accumulates without bound.

Critically, `addFunds` has no access control — it is callable by any address, not just the subscription manager:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    // No onlyManager modifier
    ...
    if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }
    status.balanceInWei += msg.value;
    ...
}
``` [1](#0-0) 

Meanwhile, `withdrawFunds` unconditionally reverts for permanent subscriptions:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [2](#0-1) 

The constant is defined as:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
``` [3](#0-2) 

### Impact Explanation

- The `MAX_DEPOSIT_LIMIT` invariant for permanent subscriptions is completely ineffective. An attacker (or any caller) can lock an arbitrarily large amount of ETH permanently in any permanent subscription by calling `addFunds` N times with 100 ETH each.
- Since permanent subscriptions cannot withdraw funds and cannot be deactivated, all excess ETH is irrecoverably locked in the contract.
- A third party can grief the owner of a permanent subscription by donating ETH to it (permanently locking their own ETH), or the subscription creator can self-fund beyond the intended cap, undermining the protocol's risk controls.

### Likelihood Explanation

The entry path requires no privileges: `addFunds` is a public `payable` function with no access control. Any externally owned account can call it on any active permanent subscription. The only cost is the ETH sent. This is directly reachable by an unprivileged transaction sender.

### Recommendation

Change the `addFunds` check to compare the **post-deposit total balance** against `MAX_DEPOSIT_LIMIT`:

```solidity
// Before (flawed):
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// After (correct):
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [4](#0-3) 

### Proof of Concept

1. Deploy `Scheduler` (or use a live instance).
2. Create a permanent subscription with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH). This succeeds.
3. Call `addFunds(subscriptionId)` with `msg.value = 100 ETH`. The check `100 ETH > 100 ETH` is `false`, so it passes. `balanceInWei` is now 200 ETH.
4. Repeat step 3 arbitrarily many times. Each call passes the check. `balanceInWei` grows without bound.
5. Attempt `withdrawFunds(subscriptionId, 1)` — reverts with `CannotUpdatePermanentSubscription`. All deposited ETH is permanently locked.

The test suite at `target_chains/ethereum/contracts/test/PulseScheduler.t.sol` (lines 1085–1094) tests that a single `addFunds` call with `maxDepositLimit + 1` reverts, but does **not** test the case of two sequential calls each with `maxDepositLimit`, which would succeed and accumulate `2 * MAX_DEPOSIT_LIMIT` in the balance.
