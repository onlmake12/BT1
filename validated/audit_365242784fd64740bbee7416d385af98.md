Audit Report

## Title
Full CKB-VM Execution Before Cycle Mismatch Detection Enables Asymmetric CPU DoS via Relay Protocol — (`tx-pool/src/process.rs`)

## Summary

In `_process_tx`, full `verify_rtx` (CKB-VM script execution) completes unconditionally before the `declared != verified.cycles` check. The verify cache is never populated on `DeclaredWrongCycles` rejection, and `remove_from_known_txs` is called on rejection, allowing the same transaction to be re-announced after reconnection. An attacker with one valid UTXO and multiple IPs can force repeated full VM executions at negligible cost.

## Finding Description

The execution order in `_process_tx` is confirmed by the code:

1. `pre_check` (cheap) — line 715
2. `fetch_tx_verify_cache` — line 719
3. `max_cycles = declared_cycles` — line 720 (attacker controls this)
4. `verify_rtx` — **full CKB-VM execution** — lines 724–732
5. `declared != verified.cycles` check — lines 736–748, returns `Err(Reject::DeclaredWrongCycles(...))`
6. Cache update — lines 758–765, **only reached on success**, never on `DeclaredWrongCycles` [1](#0-0) 

With `declared_cycles = actual_cycles + 1`, `max_cycles = actual_cycles + 1` is passed to `verify_rtx`. The VM runs to full completion consuming `actual_cycles`, then the mismatch is detected. Using `declared_cycles = actual_cycles - 1` would cause a cycle-limit-exceeded error before the mismatch check — the `C+1` path is the confirmed working attack vector.

`DeclaredWrongCycles` is confirmed `is_malformed_tx() = true`, triggering IP ban: [2](#0-1) 

`DeclaredWrongCycles` is also confirmed `is_allowed_relay() = true`, triggering `send_result_to_relayer(TxVerificationResult::Reject { tx_hash })`, which calls `remove_from_known_txs`: [3](#0-2) [4](#0-3) 

This removes the tx_hash from the node's known-tx tracking, so after the attacker reconnects from a new IP, the same tx_hash is no longer filtered and the node will accept a new relay announcement for it.

The only pre-VM guard in `TransactionsProcess::execute` is `declared_cycles > max_block_cycles` (immediate ban, no VM). A value of `actual_cycles + 1` passes this guard for any valid transaction: [5](#0-4) 

## Impact Explanation

Each relay submission with `declared_cycles = actual_cycles + 1` forces the victim node to run full `ContextualTransactionVerifier` including secp256k1 CKB-VM execution, occupying the verify queue for the full duration. The attacker spends O(1) CPU (send a P2P message); the victim spends O(secp256k1_cycles) CPU. With N IPs, the attacker sustains N parallel verify-queue occupations. This matches: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

**Preconditions:**
- One valid UTXO to pass `pre_check` (UTXO is never consumed since the tx is always rejected)
- Multiple IPs to bypass the 3-day ban per IP (Tor, VPN, or a botnet suffice)

The attack is repeatable indefinitely because: (1) the UTXO is never spent, (2) the tx_hash is removed from the relay filter on rejection via `remove_from_known_txs`, allowing re-announcement after reconnection, and (3) the cache is never populated for the rejected witness_hash, so every resubmission triggers full VM execution. [6](#0-5) 

## Recommendation

Two complementary fixes:

1. **Populate `txs_verify_cache` even on `DeclaredWrongCycles` rejection** so that subsequent resubmissions of the same witness use the cheap cached path instead of full VM execution.

2. **Do not call `remove_from_known_txs` on `DeclaredWrongCycles`** — or treat it as a non-relayable rejection — to prevent re-announcement of the same tx_hash after reconnection. The `is_allowed_relay()` returning `true` for `DeclaredWrongCycles` is the design choice that enables the relay filter bypass.

## Proof of Concept

```
1. Attacker owns UTXO U, creates valid secp256k1-signed tx T (actual cycles = C)
2. Connect to victim node as relay peer
3. Send RelayTransactionHashes([T.hash])
4. Receive GetRelayTransactions([T.hash])
5. Send RelayTransactions([{cycles: C+1, transaction: T}])
   → victim runs full secp256k1 VM (C cycles of CPU, within limit C+1)
   → victim detects C+1 != C → DeclaredWrongCycles → bans attacker IP
   → victim calls remove_from_known_txs(T.hash) (is_allowed_relay=true)
6. Reconnect from new IP → goto step 3
   (UTXO U is still unspent; T.hash is no longer in known_txs filter)
```

The existing integration test `DeclaredWrongCycles` in `test/src/specs/tx_pool/declared_wrong_cycles.rs` confirms the rejection path is reachable; adapting it to use `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1` (over-declared) confirms the `DeclaredWrongCycles` path specifically and that the cache is not populated on rejection.
