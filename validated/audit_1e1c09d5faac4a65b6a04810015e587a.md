Audit Report

## Title
Unbounded Heap Growth via Repeated `GetBlockProposal` Messages in `pending_get_block_proposals` DashMap — (`sync/src/relayer/get_block_proposal_process.rs`, `sync/src/types/mod.rs`)

## Summary
The `pending_get_block_proposals` DashMap in `SyncState` has no per-peer or global size cap. Any relay peer can send up to 3,000 unique `ProposalShortId` values per `GetBlockProposal` message, all of which are unconditionally inserted into the map. The only drain is a periodic timer (`TX_PROPOSAL_TOKEN`, every 100 ms), so between ticks a sustained multi-peer attack accumulates millions of entries, exhausting heap memory and crashing the node.

## Finding Description

**Per-message limit only — no cumulative guard:**
`GetBlockProposalProcess::execute()` enforces a per-message ceiling of `max_block_proposals_limit * max_uncles_num` (mainnet: 1,500 × 2 = 3,000) but imposes no cross-message or per-peer cumulative limit. [1](#0-0) 

**Intra-message deduplication only:**
The `HashSet` at L47–52 rejects duplicates within a single message; unique IDs across successive messages bypass this check entirely. [2](#0-1) 

**Non-pool IDs unconditionally inserted:**
IDs absent from the tx pool are passed directly to `insert_get_block_proposals` with no size check. [3](#0-2) 

**`insert_get_block_proposals` is unbounded:**
The function iterates every ID and inserts it into the DashMap with no cap or eviction policy. [4](#0-3) 

**`pending_get_block_proposals` is an unbounded DashMap:**
The field is declared with no capacity limit. [5](#0-4) 

**The only drain is the `TX_PROPOSAL_TOKEN` timer (100 ms interval):**
`drain_get_block_proposals` is called exclusively from `prune_tx_proposal_request`, which fires every 100 ms. Between ticks the map grows without bound. [6](#0-5) [7](#0-6) 

**Rate limiter limits message frequency, not entry volume:**
The 30 req/sec per-peer rate limit does not bound the number of unique IDs inserted. At 30 msg/sec × 3,000 IDs/msg = 90,000 entries/sec per peer; with 128 peers the map accumulates ~1.15 M entries per 100 ms drain interval. [8](#0-7) [9](#0-8) 

## Impact Explanation

Each DashMap entry (`ProposalShortId → HashSet<PeerIndex>`) carries heap overhead well beyond the 10-byte key. With 128 peers each sending 3 messages per 100 ms tick (30 rps × 0.1 s), the map accumulates up to 128 × 3 × 3,000 = 1,152,000 entries per tick. Sustained over multiple ticks (if processing lags or the drain is slow), this causes unbounded heap growth leading to OOM and node crash. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node*. [10](#0-9) 

## Likelihood Explanation

The attack requires only a standard relay peer connection — no PoW, no valid transactions, no authentication, and no privileged access. The attacker generates random 10-byte `ProposalShortId` values guaranteed not to be in the tx pool. The 30 rps rate limit is the only barrier, and it is insufficient given the 3,000-ID-per-message payload. The attack is repeatable, automatable, and requires minimal resources from the attacker. [1](#0-0) 

## Recommendation

- Enforce a **global size cap** on `pending_get_block_proposals` (e.g., reject insertions when the map exceeds `max_block_proposals_limit * max_uncles_num` total entries).
- Alternatively, enforce a **per-peer cap** inside `insert_get_block_proposals`, tracking how many pending IDs each peer has queued and rejecting further insertions once the per-peer limit is reached.
- Consider reducing the per-message ID limit or introducing a per-peer per-interval entry budget that accounts for DashMap accumulation rate, not just message frequency. [4](#0-3) 

## Proof of Concept

1. Connect to a CKB node as a relay peer (RelayV3 protocol).
2. In a loop at ≤30 iterations/second:
   - Generate 3,000 random 10-byte `ProposalShortId` values not present in the tx pool.
   - Send `GetBlockProposal { block_hash: tip_hash, proposals: [3000 random IDs] }`.
3. Observe `pending_get_block_proposals.len()` growing by ~3,000 per message.
4. After one 100 ms tick (3 messages): ~9,000 entries from one peer; with 128 peers doing the same: ~1.15 M entries per tick.
5. Sustained attack (multiple ticks with slow drain or concurrent insertions) causes unbounded heap growth, OOM, and node crash. [11](#0-10)
