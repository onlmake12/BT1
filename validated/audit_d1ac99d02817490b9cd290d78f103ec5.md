### Title
Deposit Refund Sent to Contract Itself Instead of Original Caller in `AuthorizeGovernanceDataSourceTransfer` Handler â€” (`target_chains/near/receiver/src/governance.rs`)

---

### Summary

In the NEAR Pyth receiver contract, when a governance VAA containing an `AuthorizeGovernanceDataSourceTransfer` action is processed, the attached NEAR deposit refund is sent to the contract's own account (`env::predecessor_account_id()` inside a `#[private]` callback) rather than to the original transaction submitter. The original caller's account ID is available as the `account_id` parameter but is never forwarded to the inner callback chain.

---

### Finding Description

The entry point `execute_governance_instruction` correctly captures the original caller via `env::predecessor_account_id()` and passes it as `account_id` to the `#[private]` callback `verify_gov_vaa_callback`: [1](#0-0) 

Inside `verify_gov_vaa_callback`, for all governance actions except `AuthorizeGovernanceDataSourceTransfer`, the refund is correctly issued to `account_id`: [2](#0-1) 

However, for the `AuthorizeGovernanceDataSourceTransfer` branch, the code constructs a new inner promise chain and passes `env::predecessor_account_id()` â€” not `account_id` â€” to both `authorize_gov_source_transfer` and `refund_vaa`: [3](#0-2) 

Because `verify_gov_vaa_callback` is annotated `#[private]` and is invoked via `Self::ext(env::current_account_id())`, the value of `env::predecessor_account_id()` inside it is the **contract's own account ID**, not the original external caller. The original caller's identity is only available through the `account_id` parameter, which is silently dropped here.

`authorize_gov_source_transfer` then refunds to whatever `account_id` it receives: [4](#0-3) 

And `refund_vaa` transfers to whatever `account_id` it receives on failure: [5](#0-4) 

Both receive the contract's own address, so the original caller's deposit is permanently retained by the contract.

---

### Impact Explanation

Any NEAR account that submits a valid `AuthorizeGovernanceDataSourceTransfer` governance VAA to `execute_governance_instruction` attaches a NEAR deposit to cover storage costs. That deposit is never returned to them. It is either credited to the contract's own balance (via `refund_storage_usage`) or silently absorbed on failure (via `refund_vaa`). The loss is irreversible because NEAR has no mechanism to reclaim funds already transferred to a contract account without that contract's cooperation.

---

### Likelihood Explanation

`execute_governance_instruction` is a public, permissionless entry point â€” any account can call it with a valid governance VAA. The `AuthorizeGovernanceDataSourceTransfer` action is a standard governance operation used during governance source key rotation. Every such rotation event triggers this code path, making the impact certain for every governance source transfer operation on NEAR.

---

### Recommendation

Replace both occurrences of `env::predecessor_account_id()` inside the `AuthorizeGovernanceDataSourceTransfer` branch of `verify_gov_vaa_callback` with the already-captured `account_id` parameter:

```rust
// Before (incorrect):
.authorize_gov_source_transfer(
    env::predecessor_account_id(),  // contract itself
    claim_vaa,
    storage,
)
.then(
    Self::ext(env::current_account_id())
        ...
        .refund_vaa(env::predecessor_account_id(), env::attached_deposit()),
)

// After (correct):
.authorize_gov_source_transfer(
    account_id.clone(),  // original caller
    claim_vaa,
    storage,
)
.then(
    Self::ext(env::current_account_id())
        ...
        .refund_vaa(account_id, env::attached_deposit()),
)
```

This matches the pattern used correctly in all other governance action branches.

---

### Proof of Concept

1. Alice calls `execute_governance_instruction` with a valid `AuthorizeGovernanceDataSourceTransfer` VAA, attaching 1 NEAR as deposit.
2. The contract captures `env::predecessor_account_id()` = `alice.near` and passes it as `account_id` to `verify_gov_vaa_callback`.
3. Inside `verify_gov_vaa_callback` (a `#[private]` callback called by `Self::ext(env::current_account_id())`), `env::predecessor_account_id()` = `pyth.near` (the contract itself).
4. The inner promise chain calls `authorize_gov_source_transfer(pyth.near, ...)` and `refund_vaa(pyth.near, ...)`.
5. On success, `refund_storage_usage` sends the leftover deposit to `pyth.near`, not `alice.near`.
6. On failure, `refund_vaa` sends the full deposit to `pyth.near`, not `alice.near`.
7. Alice's 1 NEAR is permanently lost. [6](#0-5)
