Audit Report

## Title
Unbounded `vec![0u64; max_bucket_index + 1]` allocation in `do_estimate` via high-fee-rate pool entry causes OOM/node crash — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary

`do_estimate` derives `max_bucket_index` directly from the highest fee_rate in the pending pool using an unbounded linear formula, then allocates two `Vec<u64>` of that size with no cap. A transaction carrying a fee of ~120,000 CKB on a minimum-size transaction is sufficient to cause a multi-gigabyte heap allocation attempt, crashing the node via OOM or panic. Any subsequent call to the `estimate_fee_rate` RPC after such a transaction enters the pool triggers the crash.

## Finding Description

`max_bucket_index_by_fee_rate` maps any `FeeRate` to a bucket index. For `fee_rate > 2_000_000` shannons/KW, the formula is:

```
index = (x + 11_500_000) / 100_000
```

This is unbounded. [1](#0-0) 

`do_estimate` then allocates two `Vec<u64>` of size `max_bucket_index + 1` with no guard: [2](#0-1) [3](#0-2) 

`FeeRate::calculate` uses `saturating_mul(KW)`, so the maximum achievable fee_rate from a 60-byte tx is `u64::MAX / 60 ≈ 3.07e17`, yielding `max_bucket_index ≈ 3.07e12` and ~49 TB of attempted allocation: [4](#0-3) 

The tx-pool admission path (`check_tx_fee`) enforces only a **minimum** fee rate — there is no maximum fee or fee_rate cap anywhere in the admission path: [5](#0-4) 

The `estimate_fee_rate` RPC reads live pool state and passes it directly into `do_estimate` without any sanitization: [6](#0-5) 

**Concrete arithmetic for a 16 GB node OOM:**
- Need `max_bucket_index * 8 > 16e9` → `max_bucket_index > 2e9`
- Requires `fee_rate > 2e14` shannons/KW
- With weight=60: `fee > 2e14 * 60 / 1000 = 1.2e13 shannons = 120,000 CKB`

`TRANSACTION_SIZE_LIMIT` is 512,000 bytes, confirming minimum-size transactions are valid pool entries: [7](#0-6) 

## Impact Explanation

A successful attack crashes the node process via OOM or panic, causing a full denial of service for that node. This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires:
1. The `WeightUnitsFlow` estimator to be configured and `is_ready = true` (post-IBD).
2. The `Experiment` RPC module to be enabled on the target node.
3. The attacker to own ~120,000 CKB to OOM a 16 GB node (a financial cost, not a privilege). This is approximately 0.00036% of total CKB supply and is achievable by a well-funded attacker.

Once the high-fee transaction is in the pool, any call to `estimate_fee_rate` — including by a third party — triggers the crash. The attack is repeatable as long as the attacker has sufficient CKB.

## Recommendation

Cap `max_bucket_index` to a safe constant before allocating the bucket vectors. The maximum realistic fee rate of ~10,000,000 shannons/KW maps to index ~135, so a guard such as:

```rust
let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
    .min(MAX_SAFE_BUCKET_INDEX);
```

with `MAX_SAFE_BUCKET_INDEX = 135` (or a small multiple for headroom) would bound both allocations to a few kilobytes regardless of input fee_rate. [8](#0-7) 

## Proof of Concept

```rust
use ckb_types::core::{Capacity, FeeRate};
use ckb_types::core::tx_pool::{TxEntryInfo, TxPoolEntryInfo};
use weight_units_flow::Algorithm;

let mut algo = Algorithm::new();
algo.update_ibd_state(false);
// Advance current_tip past historical_blocks threshold by calling commit_block enough times.

// Construct a TxEntryInfo with fee that saturates fee*KW:
// fee = u64::MAX / 1000 + 1 shannons → fee * 1000 saturates to u64::MAX
// weight = 60 → fee_rate = u64::MAX / 60 ≈ 3.07e17
// max_bucket_index ≈ 3.07e12 → two Vec<u64> of ~24 TB each → OOM
let info = TxEntryInfo {
    fee: Capacity::shannons(u64::MAX / 1000 + 1),
    size: 60,
    cycles: 0,
    ..Default::default()
};

let mut pending = std::collections::HashMap::new();
pending.insert(Default::default(), info);

// This call attempts to allocate ~49 TB and panics/OOMs:
let _ = algo.estimate_fee_rate(10, TxPoolEntryInfo {
    pending,
    proposed: Default::default(),
    conflicted: vec![],
});
```

    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }
