### Title
`getPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp Instead of Per-Feed Publish Time, Allowing Arbitrarily Stale Prices to Pass Freshness Checks - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler` contract's `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions perform their staleness check against `status.priceLastUpdatedAt`, which is set to the **maximum** `publishTime` across all price feeds in a batch update. When a subscription contains feeds for both active and closed markets, the closed-market feeds can have `publishTime` values that are days old, yet the staleness check passes because it only examines the most recent timestamp in the batch. Consumers receive a false freshness guarantee, analogous to the Redstone bug where combined `cacheStaleness + priceStaleness` exceeds the assumed maximum.

---

### Finding Description

In `Scheduler.updatePriceFeeds`, after parsing the price feeds, `_validateShouldUpdatePrices` computes `updateTimestamp` as the **maximum** `publishTime` across all feeds in the batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

This `updateTimestamp` is then stored as `status.priceLastUpdatedAt`:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
```

The code comment explicitly acknowledges that closed markets will have old timestamps:

> "Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period."

However, `getPricesNoOlderThan` checks staleness only against this subscription-level max timestamp, not against each individual feed's `publishTime`:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
```

`getPricesUnsafe` returns the raw stored price for each requested feed, including closed-market feeds whose `publishTime` may be days old. The staleness check passes because `priceLastUpdatedAt` reflects the active-market feed's recent timestamp, not the closed-market feed's stale timestamp.

The same flaw exists in `getEmaPricesNoOlderThan`.

---

### Impact Explanation

A DeFi protocol (lending, derivatives, liquidation engine) that integrates the Scheduler and calls `getPricesNoOlderThan(subscriptionId, [closedMarketFeedId], 60)` will:

1. Receive a `StalePrice` revert only if `distance(block.timestamp, priceLastUpdatedAt) > 60` — but `priceLastUpdatedAt` is the active-market feed's timestamp, which is recent.
2. Receive the closed-market feed's price, which may be hours or days old, with no revert.
3. Believe the price is no older than 60 seconds when it is actually `block.timestamp − (priceLastUpdatedAt − closedFeed.publishTime + age_seconds)` old — potentially days stale.

Concrete consequences matching the Redstone report's impact class:
- Borrowing more of an underpriced closed-market asset (e.g., equity, commodity)
- Borrowing more value against overpriced closed-market collateral
- Inability to liquidate in a timely manner when the true price has moved

---

### Likelihood Explanation

- Any subscription containing a mix of 24/7 crypto feeds and traditional-market feeds (equities, commodities, FX) is affected.
- Any unprivileged keeper can call `updatePriceFeeds` to trigger the update that sets `priceLastUpdatedAt` to the max timestamp.
- Any whitelisted reader (or anyone, if `whitelistEnabled = false`) can then call `getPricesNoOlderThan` and receive the stale price.
- No privileged access, leaked keys, or governance majority is required.
- The Scheduler is explicitly designed to support mixed-market subscriptions, making this a realistic deployment scenario.

---

### Recommendation

Replace the subscription-level staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed staleness check. Instead of checking `priceLastUpdatedAt` (the max across all feeds), check each requested feed's individual `publishTime`:

```solidity
function getPricesNoOlderThan(...) external view ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

Alternatively, document clearly that `getPricesNoOlderThan` does **not** guarantee per-feed freshness for closed-market assets, and require consumers to check individual `publishTime` fields.

---

### Proof of Concept

1. Create a subscription with two feeds: `ETH/USD` (24/7, active) and `AAPL/USD` (equity, closed on weekends).
2. On a Saturday, call `updatePriceFeeds`. Pyth returns:
   - `ETH/USD`: `publishTime = block.timestamp` (e.g., `T`)
   - `AAPL/USD`: `publishTime = T − 2 days` (last Friday close)
   - Both feeds share the same Pythnet slot (slot check passes).
3. `_validateShouldUpdatePrices` sets `updateTimestamp = T` (max of the two).
4. `status.priceLastUpdatedAt = T`.
5. Call `getPricesNoOlderThan(subscriptionId, [AAPL_USD_ID], 60)`:
   - Check: `distance(block.timestamp, T) = 0 ≤ 60` → **passes**.
   - Returns `AAPL/USD` price with `publishTime = T − 2 days`.
6. A lending protocol accepts this 2-day-old AAPL price as "no older than 60 seconds" and allows borrowing against it at a stale valuation.
