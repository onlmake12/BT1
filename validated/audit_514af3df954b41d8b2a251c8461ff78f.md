### Title
Silent u128→u64 Truncation in DAO Withdrawal Capacity Calculation Produces Wrong Capacity - (File: util/dao/src/lib.rs)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes `withdraw_counted_capacity` as a `u128` intermediate value, then casts it to `u64` with a bare `as u64` — a **silent truncating cast**. When the intermediate value exceeds `u64::MAX`, the high bits are silently discarded, producing a drastically wrong (smaller) withdrawal capacity with no error returned. Every other analogous u128→u64 conversion in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern, making this inconsistency the root cause.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits when `withdraw_counted_capacity > u64::MAX`. The three other u128→u64 conversions in the same file all use the checked form:

- `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?` [2](#0-1) 
- `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?` [3](#0-2) 
- `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?` [4](#0-3) 

The truncated value is then fed into `transaction_fee` via `transaction_maximum_withdraw`:

```rust
pub fn transaction_fee(&self, rtx: &ResolvedTransaction) -> Result<Capacity, DaoError> {
    let maximum_withdraw = self.transaction_maximum_withdraw(rtx)?;
    rtx.transaction
        .outputs_capacity()
        .and_then(|y| maximum_withdraw.safe_sub(y))
        .map_err(Into::into)
}
``` [5](#0-4) 

And into the `withdrawed_interests` function used for block DAO field computation: [6](#0-5)

### Impact Explanation

**Scenario triggering silent truncation:**

- `counted_capacity ≈ u64::MAX` shannons (a large but valid DAO cell)
- `withdrawing_ar = deposit_ar + 1` (minimal interest, e.g., `deposit_ar = 10_000_000_000_000_000`)
- `withdraw_counted_capacity = u64::MAX * (10^16 + 1) / 10^16 ≈ u64::MAX + 1`
- After `as u64`: result wraps to `≈ 0`
- `Capacity::shannons(0).safe_add(occupied_capacity)` succeeds, returning `occupied_capacity` — a value orders of magnitude smaller than the correct withdrawal amount

**Downstream effects:**

1. **`calculate_dao_maximum_withdraw` RPC** returns a silently wrong (tiny) capacity to the caller, causing the user to construct a withdrawal transaction claiming far less than they are owed. [7](#0-6) 

2. **`check_tx_fee` in tx-pool** calls `transaction_fee`, which calls `maximum_withdraw.safe_sub(outputs_capacity)`. With a truncated `maximum_withdraw` that is smaller than `outputs_capacity`, this returns `DaoError::Overflow`, causing the valid DAO withdrawal transaction to be **rejected from the tx-pool** with a misleading error. [8](#0-7) 

3. **Block DAO field corruption**: If a truncated value passes fee checks (truncated value happens to be ≥ outputs_capacity), `withdrawed_interests` uses the wrong value to update `current_s` in the block header's DAO field, corrupting the on-chain NervosDAO accounting state. [9](#0-8) 

---

### Likelihood Explanation

- Any unprivileged transaction sender can submit a DAO withdrawal transaction referencing a large-capacity DAO cell.
- Any RPC caller can invoke `calculate_dao_maximum_withdraw` with attacker-chosen parameters.
- The condition requires a cell with capacity near `u64::MAX` shannons (≈ 184 billion CKB). While this is a large amount, it is not protocol-prohibited, and the NervosDAO is designed to hold long-term deposits of arbitrary size.
- The `ar` ratio grows monotonically over time; any non-zero interest period satisfies the condition given sufficient cell capacity.
- No privileged access, key material, or majority hashpower is required.

---

### Recommendation

Replace the silent `as u64` cast with the same checked conversion pattern used everywhere else in the file:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This makes overflow explicit and consistent with `miner_issuance128`, `reward128`, and `ar_increase128` handling in the same file. [1](#0-0)

### Proof of Concept

```
deposit_ar      = 10_000_000_000_000_000   (initial ar, u64)
withdrawing_ar  = 10_000_000_000_000_001   (one unit of interest accrued)
counted_capacity = 18_446_744_073_709_551_615  (u64::MAX shannons)

withdraw_counted_capacity (u128)
  = 18_446_744_073_709_551_615 * 10_000_000_000_000_001
    / 10_000_000_000_000_000
  = 18_446_744_073_709_551_615 + 1   (≈ u64::MAX + 1)
  = 18_446_744_073_709_551_616       (= 2^64, exceeds u64::MAX)

withdraw_counted_capacity as u64
  = 18_446_744_073_709_551_616 mod 2^64
  = 0                                ← silently wrong

Capacity::shannons(0).safe_add(occupied_capacity)
  → Ok(occupied_capacity)            ← no error, wrong value returned
```

The correct behavior (matching all other conversions in the file) would be to return `Err(DaoError::Overflow)`.

        let maximum_withdraws = rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
            self.transaction_maximum_withdraw(rtx)
                .and_then(|c| capacities.safe_add(c).map_err(Into::into))
        })?;
        let input_capacities = rtxs.try_fold(Capacity::zero(), |capacities, rtx| {

                    let output_capacity: Capacity = cell_meta.cell_output.capacity().into();
                    tx_capacities.safe_add(output_capacity)
                },
            )?;
            capacities.safe_add(tx_input_capacities)
        })?;
        maximum_withdraws
            .safe_sub(input_capacities)
            .map_err(Into::into)
    }
