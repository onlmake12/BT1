### Title
`ScriptHashTypeVerifier` Omits Type Script Hash-Type Validation, Allowing Permanently Unincludable Transactions Into the Tx-Pool — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` checks only the **lock script's** `hash_type` against the set of consensus-permitted values (`ENABLED_SCRIPT_HASH_TYPE`), but never inspects the **type script's** `hash_type`. A transaction sender can craft an output whose type script carries a future/unsupported `ScriptHashType` (e.g. `Data3` = 6). The non-contextual verifier accepts the transaction into the tx-pool; the script engine then rejects it at execution time, so the transaction can never be committed to the chain. The sender's input cells are effectively frozen in the pool until eviction, and the pool accumulates permanently-unincludable entries with no automated cleanup path.

---

### Finding Description

`NonContextualTransactionVerifier` runs `ScriptHashTypeVerifier` as one of its mandatory checks: [1](#0-0) 

`ScriptHashTypeVerifier::verify()` iterates over every output and validates only `output.lock().hash_type()`: [2](#0-1) 

The type script field is never touched. `ENABLED_SCRIPT_HASH_TYPE` (imported from `ckb_constant::consensus`) contains only the currently-activated values (`Data`=0, `Type`=1, `Data1`=2, `Data2`=4). The `ScriptHashType` enum, however, is defined for values up to `Data127`: [3](#0-2) 

The existing test `test_not_enabled_hash_type_output_lock` confirms that `Data3` (value 6) is rejected when it appears in a **lock** script: [4](#0-3) 

No equivalent test or code path exists for a **type** script carrying the same unsupported value. Because `check_data` in the molecule layer only verifies that the raw byte is a structurally valid `ScriptHashType` value (even/odd rule), not that it is consensus-activated: [5](#0-4) 

…a transaction whose output type script has `hash_type = Data3` passes all structural and non-contextual checks and enters the tx-pool. When the block assembler later attempts to execute the type script, `select_version` returns `InvalidScriptHashType`: [6](#0-5) 

The transaction is silently dropped from block candidates but remains in the pool.

---

### Impact Explanation

1. **Tx-pool pollution / resource exhaustion.** Any RPC caller or relay peer can inject an unbounded number of permanently-unincludable transactions. Each occupies pool memory and is counted against pool limits, crowding out legitimate transactions.

2. **Temporary locking of sender's input cells.** While the malformed transaction is live in the pool, its input cells are marked as pending-spent. The sender cannot re-spend those cells without an explicit RBF replacement. Users who are unaware of RBF may believe their funds are lost until the transaction is eventually evicted.

3. **No automated recovery.** There is no mechanism that proactively detects and evicts transactions whose type scripts use unsupported hash types. Recovery depends on the pool's size-based eviction heuristic, which is non-deterministic.

---

### Likelihood Explanation

- **Attacker path:** Any unprivileged RPC caller (`send_transaction` with default `passthrough` validator) or relay peer can submit such a transaction. No special privilege is required.
- **Accidental path:** A wallet or dApp developer targeting a future VM version (e.g. `Data3`) before the corresponding hardfork activates will have their transaction silently accepted by the pool and silently rejected by every miner, with no clear error at submission time.
- **Likelihood:** Medium. The `passthrough` default for `outputs_validator` means the RPC-level guard is also absent: [7](#0-6) 

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` for every output that carries one:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

This mirrors the existing lock-script logic and ensures that transactions with unsupported type script hash types are rejected at the non-contextual verification stage, before they enter the tx-pool.

### Proof of Concept

1. Construct a transaction whose single output has:
   - A valid secp256k1 lock script (`hash_type = Type`, correct args length — passes `ScriptHashTypeVerifier`).
   - A type script with `hash_type = 6` (`Data3`, not yet activated).
2. Submit via `send_transaction` with `outputs_validator = null` (default `passthrough`).
3. Observe: the RPC returns a tx hash (accepted into pool).
4. Mine blocks; observe: the transaction is never committed.
5. Query `get_raw_tx_pool`; observe: the transaction remains in the pool indefinitely.
6. The sender's input cells cannot be re-spent without an RBF replacement transaction.

/// Basic checks that don't depend on any context
/// Contains:

pub struct NonContextualTransactionVerifier<'a> {
    pub(crate) version: VersionVerifier<'a>,
    pub(crate) size: SizeVerifier<'a>,
    pub(crate) empty: EmptyVerifier<'a>,
    pub(crate) duplicate_deps: DuplicateDepsVerifier<'a>,
    pub(crate) outputs_data_verifier: OutputsDataVerifier<'a>,

    /// Creates a new NonContextualTransactionVerifier
    pub fn new(tx: &'a TransactionView, consensus: &'a Consensus) -> Self {
        NonContextualTransactionVerifier {
            version: VersionVerifier::new(tx, consensus.tx_version()),
            size: SizeVerifier::new(tx, consensus.max_block_bytes()),
            empty: EmptyVerifier::new(tx),
            duplicate_deps: DuplicateDepsVerifier::new(tx),
            outputs_data_verifier: OutputsDataVerifier::new(tx),
            script_hash_type: ScriptHashTypeVerifier::new(tx),
        }

        self.version.verify()?;
        self.size.verify()?;
        self.empty.verify()?;
        self.duplicate_deps.verify()?;
        self.outputs_data_verifier.verify()?;
        self.script_hash_type.verify()?;
        Ok(())
    }
