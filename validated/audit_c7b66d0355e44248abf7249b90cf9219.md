### Title
DAO Phase 1 Withdrawal Permanently Blocked via tx-pool Before `rfc0044` Activation — (`tx-pool/src/util.rs`, `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

The tx-pool unconditionally applies `DaoScriptSizeVerifier` to every DAO Phase 1 withdrawal transaction, while the block verifier gates the same check on `rfc0044_active`. Before `rfc0044` activates, a DAO Phase 1 withdrawal transaction that changes the lock script size is **valid per consensus** (block verifier skips the check) but is **permanently rejected by the tx-pool** (which always runs the check). The user has no standard RPC path to submit the transaction, mirroring the "no interface to exit EARNING state" pattern from the external report.

---

### Finding Description

**Root cause — tx-pool (`tx-pool/src/util.rs`, lines 101–131):**

`verify_rtx` always chains `DaoScriptSizeVerifier::verify()` after `ContextualTransactionVerifier`, with no guard on `rfc0044_active`:

```rust
// async path
.and_then(|result| {
    DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
        .verify()?;          // ← unconditional
    Ok(result)
})

// sync path
.and_then(|result| {
    DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
        .verify()?;          // ← unconditional
    Ok(result)
})
```

**Root cause — block verifier (`verification/contextual/src/contextual_block_verifier.rs`, lines 444–453):**

The identical verifier is gated on the soft-fork activation flag:

```rust
.and_then(|result| {
    if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {
        DaoScriptSizeVerifier::new(Arc::clone(tx), ...).verify()?;
    }
    Ok(result)
})
```

**What `DaoScriptSizeVerifier` checks (`verification/src/transaction_verifier.rs`, lines 845–890):**

For every DAO deposit cell (input data all-zeros) committed at or after `starting_block_limiting_dao_withdrawing_lock` (block 10 000 000 on mainnet), the Phase 1 output lock script must be the **same byte size** as the deposit cell's lock script. If sizes differ, it returns `TransactionError::DaoLockSizeMismatch`.

**The divergence:**

| Context | `DaoScriptSizeVerifier` applied? |
|---|---|
| tx-pool (`verify_rtx`) | Always |
| Block verifier | Only when `rfc0044_active` |

Before `rfc0044` activates, a Phase 1 withdrawal that changes lock script size is:
- **Rejected** by the tx-pool → user cannot submit via any RPC endpoint
- **Accepted** by the block verifier → transaction is consensus-valid

The user is left with no standard submission path. Their DAO deposit is frozen in the "deposited" state with no reachable Phase 1 transition through the node's public interface.

---

### Impact Explanation

A transaction sender (RPC caller / tx-pool submitter) who holds a DAO deposit cell committed after block 10 000 000 and needs to change their lock script size during Phase 1 withdrawal has **no standard path** to submit the transaction. The tx-pool is the only public submission interface; it unconditionally rejects the transaction. The deposit cell is permanently inaccessible through the node's RPC layer until `rfc0044` activates — at which point the block verifier also rejects it, making the lock permanent. This is a direct capacity/asset lock on user funds.

---

### Likelihood Explanation

Any user who:
1. Deposited into NervosDAO after block 10 000 000 on mainnet (or block 0 on devnet, where `starting_block_limiting_dao_withdrawing_lock = 0`),
2. Needs to change their lock script size in the Phase 1 withdrawal output (e.g., migrating to a new lock script with different args length), and
3. Submits before `rfc0044` is active on their network

will hit this. No privileged access is required. The trigger is a normal `send_transaction` RPC call. On devnet the threshold is block 0, making every DAO deposit immediately subject to the asymmetry.

---

### Recommendation

Gate `DaoScriptSizeVerifier` in `verify_rtx` on the same `rfc0044_active` condition used by the block verifier, so the tx-pool admission policy is consistent with consensus:

```rust
// in verify_rtx, after ContextualTransactionVerifier
.and_then(|result| {
    let consensus = snapshot.cloned_consensus();
    let tip_header = snapshot.tip_header();
    if consensus.rfc0044_active(tip_header.epoch().number()) {
        DaoScriptSizeVerifier::new(rtx, consensus, snapshot.as_data_loader())
            .verify()?;
    }
    Ok(result)
})
```

---

### Proof of Concept

1. On devnet (`starting_block_limiting_dao_withdrawing_lock = 0`, `rfc0044` not yet active), deposit into NervosDAO with a secp256k1 lock script whose `args` is 20 bytes (total lock size = 53 bytes).
2. Construct a Phase 1 withdrawal transaction whose output lock script has `args` of 32 bytes (total lock size = 65 bytes). The transaction is consensus-valid because `rfc0044_active` is false.
3. Submit via `send_transaction` RPC.
4. The tx-pool calls `verify_rtx` → `DaoScriptSizeVerifier::verify()` → detects `53 != 65` → returns `TransactionError::DaoLockSizeMismatch { index: 0 }` → RPC returns error.
5. Construct a raw block containing the same transaction and submit it directly to the block verifier (e.g., via `generate_block_with_template`). The block verifier skips `DaoScriptSizeVerifier` because `rfc0044_active` is false → block is accepted.
6. The deposit cell is now permanently unreachable through the standard RPC submission path, confirming the withdrawal lock. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
```
