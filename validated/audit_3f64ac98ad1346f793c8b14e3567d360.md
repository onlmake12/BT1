### Title
Incomplete `ScriptHashType` Validation in `ScriptHashTypeVerifier` — Type Script Hash Type Not Checked — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` only validates the `hash_type` field of the **lock script** of each transaction output. It never inspects the **type script** `hash_type`. A separate, more complete validation function (`ScriptHashType::verify_value`, used via `check_data()`) exists and does check both lock and type scripts, but it is wired only into the P2P relay path, not into the non-contextual transaction verification pipeline that gates tx-pool admission for RPC-submitted transactions. As a result, a transaction output carrying a type script with an invalid or not-yet-consensus-enabled `hash_type` byte passes `NonContextualTransactionVerifier` and enters the tx pool, where it consumes resources before being rejected during contextual script execution.

---

### Finding Description

`ScriptHashTypeVerifier::verify()` iterates over `self.transaction.outputs()` and for each output checks only `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  L796-L814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

`output.type_()` is never consulted. A type script with `hash_type = 3` (structurally invalid — not a multiple of 2 and not 1) or `hash_type = 6` (`Data3`, structurally valid but not yet consensus-enabled) passes this verifier silently.

A parallel, more complete validation path does exist. `ScriptHashType::verify_value` checks the structural rule (`v % 2 == 0 || v == 1`):

```rust
// util/gen-types/src/core.rs  L39-L41
pub fn verify_value(v: u8) -> bool {
    v.is_multiple_of(2) || v == 1
}
``` [2](#0-1) 

This is called through `check_data()` on `packed::ScriptReader`, which in turn is called for both lock and type scripts of every `CellOutput`:

```rust
// util/gen-types/src/extension/check_data.rs  L10-L27
impl<'r> packed::ScriptReader<'r> {
    fn check_data(&self) -> bool {
        core::ScriptHashType::verify_value(self.hash_type().into())
    }
}
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()
    }
}
``` [3](#0-2) 

However, `check_data()` on `TransactionReader` is `pub(crate)` and is wired into the P2P relay message handlers (`RelayTransactionsReader`, `SendBlockReader`, `CompactBlockReader`), not into the `NonContextualTransactionVerifier` pipeline that processes RPC-submitted transactions. [4](#0-3) 

`NonContextualTransactionVerifier` calls `self.script_hash_type.verify()` as its only hash-type gate:

```rust
// verification/src/transaction_verifier.rs  L94-L101
pub fn verify(&self) -> Result<(), Error> {
    self.version.verify()?;
    self.size.verify()?;
    self.empty.verify()?;
    self.duplicate_deps.verify()?;
    self.outputs_data_verifier.verify()?;
    self.script_hash_type.verify()?;   // ← only checks lock script
    Ok(())
}
``` [5](#0-4) 

The type script hash type is therefore never validated at the non-contextual stage for RPC-submitted transactions.

---

### Impact Explanation

An unprivileged RPC caller can craft a transaction whose output carries a type script with `hash_type = 3` (structurally invalid) or `hash_type = 6` (`Data3`, not yet enabled). The transaction passes `NonContextualTransactionVerifier` and is admitted to the tx pool. It is only rejected later, during contextual script execution in `select_version()`:

```rust
// script/src/types.rs  L903-L904
let script_hash_type = ScriptHashType::try_from(script.hash_type())
    .map_err(|err| ScriptError::InvalidScriptHashType(err.to_string()))?;
``` [6](#0-5) 

Between admission and rejection the transaction occupies tx-pool slots, triggers resolution of its cell deps, and consumes CPU during contextual verification. Repeated at scale this constitutes targeted tx-pool resource exhaustion: an attacker can keep the pool filled with cheap-to-craft, expensive-to-evict invalid transactions, degrading throughput for legitimate users.

---

### Likelihood Explanation

The attack requires only a valid RPC connection and the ability to construct a transaction — no keys, no stake, no privileged access. Crafting a transaction with an invalid type script `hash_type` byte is trivial (set the byte to 3 or 6 in the raw molecule encoding). The node's RPC endpoint is reachable by any network peer. Likelihood is **high**.

---

### Recommendation

**Short term:** Extend `ScriptHashTypeVerifier::verify()` to also iterate over `output.type_().to_opt()` and apply the same `ENABLED_SCRIPT_HASH_TYPE` check to the type script's `hash_type`. Mirror the completeness already present in `CellOutputReader::check_data()`.

**Long term:** Unify the structural (`verify_value`) and consensus-enabled (`ENABLED_SCRIPT_HASH_TYPE`) checks into a single validation step called consistently for every script field — lock, type, and cell-dep scripts — at the earliest possible point in both the RPC and P2P ingestion paths, eliminating the current divergence between the two code paths.

---

### Proof of Concept

1. Build a raw transaction (molecule encoding) with one output whose type script has `hash_type = 0x03` (byte value 3, structurally invalid — not a multiple of 2 and not 1).
2. Submit via `send_transaction` RPC.
3. Observe: `NonContextualTransactionVerifier` returns `Ok(())` — `ScriptHashTypeVerifier` never inspected the type script.
4. The transaction enters the tx pool.
5. During contextual verification `ScriptHashType::try_from(3u8)` returns `Err`, and the transaction is evicted — but only after consuming pool resources.

Repeat with `hash_type = 0x06` (`Data3`, structurally valid but not yet consensus-enabled) to confirm the same bypass of the non-contextual gate.
