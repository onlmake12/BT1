### Title
Integer Overflow in Guard Check Bypasses Per-Message Work Limit, Enabling O(chain_length) DB Read DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

`GetLastStateProofProcess::execute()` contains an integer overflow in its sole rate-limiting guard. An unprivileged peer can send a crafted `GetLastStateProof` message with `last_n_blocks = 2^63` (fits in `Uint64`) to wrap `(last_n_blocks as usize) * 2` to `0` in Rust release mode, bypassing the `GET_LAST_STATE_PROOF_LIMIT = 1000` check entirely. After the bypass, the same large `last_n_blocks` value causes the server to collect **every block number from `start_block_number` to `last_block_number`** and perform multiple DB reads per block — O(chain_length) work per message.

---

### Finding Description

**Guard check with overflow** — `execute()` reads `last_n_blocks` as `u64` and immediately uses it in an arithmetic guard: [1](#0-0) 

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

On a 64-bit host, `usize` is 64 bits. When `last_n_blocks = 2^63`:

- `last_n_blocks as usize` = `2^63` (lossless cast)
- `2^63 * 2` = `2^64` → wraps to **`0`** in Rust release mode (no overflow check)
- Guard becomes `difficulties.len() + 0 > 1000`, which passes for any `difficulties.len() ≤ 1000`

The `Uint64` wire type accepts any 64-bit value; `2^63` is a valid encoding. [2](#0-1) 

There is no separate cap on `difficulties.len()` alone, and `Uint256Vec` has no built-in molecule size limit. [3](#0-2) 

**Post-bypass amplification** — after the guard, `last_n_blocks` (still `2^63` as `u64`) is compared against the actual chain span: [4](#0-3) 

```rust
if last_block_number - start_block_number <= last_n_blocks  // always true: real chain << 2^63
{
    let sampled_numbers = Vec::new();
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
    ...
}
```

Because `last_n_blocks = 2^63` dwarfs any realistic chain length, the condition is always `true`, and `last_n_numbers` becomes a vector of **every block number** from `start_block_number` to the tip. This vector is then passed to `complete_headers`, which performs multiple DB reads per entry (hash lookup, block_ext, ancestor traversal, MMR root): [5](#0-4)

### Impact Explanation

A single small P2P message (~100 bytes) forces the full node to:
1. Allocate a `Vec<BlockNumber>` of length equal to the entire chain
2. Execute O(chain_length) DB reads inside `complete_headers` (hash, block_ext, ancestor, block, MMR per entry)

On mainnet (millions of blocks), this saturates disk I/O and CPU on the serving node. Any number of peers can send this message concurrently. There is no per-peer rate limit visible in this code path. The impact is network-wide CPU/IO exhaustion of all light-client-serving full nodes.

---

### Likelihood Explanation

- No privilege required — any peer that speaks the `LightClient` protocol can send this message.
- The crafted value (`2^63`) is trivially constructable.
- Rust release builds wrap on overflow by default; this is not a debug-only behavior.
- The attack is repeatable and stateless (no chain state needed beyond knowing a valid `last_hash`).

---

### Recommendation

Replace the unchecked arithmetic with overflow-safe operations before the comparison:

```rust
// Option A: saturating arithmetic
let n_blocks_contribution = (last_n_blocks as usize).saturating_mul(2);
if self.message.difficulties().len().saturating_add(n_blocks_contribution)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option B: reject oversized last_n_blocks early
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, add a standalone cap on `difficulties.len()` independent of `last_n_blocks`, and consider adding a per-peer request rate limit for this message type.

### Proof of Concept

```rust
// Craft a GetLastStateProof message:
let content = packed::GetLastStateProof::new_builder()
    .last_hash(valid_tip_hash)          // any valid main-chain hash
    .start_hash(genesis_hash)
    .start_number(0u64)
    .last_n_blocks(u64::MAX / 2 + 1)   // = 2^63; causes (as usize)*2 == 0
    .difficulty_boundary(some_u256)
    // difficulties: empty or up to 999 entries — guard passes either way
    .build();
// Send via LightClient P2P protocol.
// Server allocates Vec of length == chain_tip_number and does O(chain_length) DB reads.
// Assert: server CPU/IO spikes; response never arrives (OOM or timeout).
```
