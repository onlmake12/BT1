### Title
`PythAggregatorV3.latestRoundData()` Uses `getPriceUnsafe()` Without Staleness Validation, Enabling Stale Price Exploitation - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatible adapter distributed in the Solidity SDK, implements every price-reading function — including `latestRoundData()`, `latestAnswer()`, `getRoundData()`, and `decimals()` — by calling `pyth.getPriceUnsafe()`. This function explicitly performs **no staleness check** and may return a price from arbitrarily far in the past. Additionally, the returned `answeredInRound` field is set equal to `roundId` (both derived from `publishTime`), which causes the standard Chainlink staleness guard (`require(answeredInRound >= roundId)`) to always pass, giving consumers a false sense of freshness.

---

### Finding Description

In `PythAggregatorV3.sol`, every price-reading function delegates to `pyth.getPriceUnsafe(priceId)`:

```solidity
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
``` [1](#0-0) [2](#0-1) 

`getPriceUnsafe()` is explicitly documented as returning a price from arbitrarily far in the past with no recency guarantee:

> "This function returns the most recent price update in this contract without any recency checks. This function is unsafe as the returned price update may be arbitrarily far in the past." [3](#0-2) 

The safe alternative, `getPriceNoOlderThan(id, age)`, reverts if the price is stale: [4](#0-3) 

Furthermore, `latestRoundData()` sets both `roundId` and `answeredInRound` to `uint80(price.publishTime)`. Chainlink-compatible consumers commonly guard against stale rounds with `require(answeredInRound >= roundId, "stale")`. Since both values are identical, this check always passes regardless of how old the price is. [5](#0-4) 

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a Chainlink-compatible price feed adapter (as documented in the official Chainlink migration guide) and calls `latestRoundData()` or `latestAnswer()` will silently receive a stale price with no revert. This directly mirrors the original report's impact: stale prices affect account health calculations, enabling:

- Incorrect liquidations of healthy positions (if the stale price is lower than the real price)
- Under-collateralized borrowing (if the stale price is higher than the real price)
- Incorrect settlement of derivative positions

The `PythAggregatorV3` contract is part of the official Pyth Solidity SDK (`@pythnetwork/pyth-sdk-solidity`) and is actively recommended for Chainlink migrations, making the blast radius large. [6](#0-5) 

---

### Likelihood Explanation

- Pyth is a pull oracle; on-chain prices are only updated when someone calls `updatePriceFeeds()`. If no keeper or user updates the price, the on-chain value can become arbitrarily stale.
- The `PythAggregatorV3` contract is the official migration path for Chainlink users. Protocols following the documented migration guide will deploy this contract and call `latestRoundData()` expecting Chainlink-equivalent freshness semantics.
- An attacker can passively wait for the on-chain price to drift from the real market price (e.g., during low-activity periods or network congestion) and then exploit the stale price.
- No privileged access is required; any unprivileged user can interact with a downstream protocol that uses this adapter.

---

### Recommendation

Replace all `getPriceUnsafe()` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`, where `maxAge` is a configurable constructor parameter (e.g., defaulting to `getValidTimePeriod()`). This mirrors the fix applied in the referenced H2O report: switching from the unchecked method to the validated one.

Additionally, `latestRoundData()` should either revert on stale prices or set `answeredInRound` to a value that correctly signals staleness to downstream Chainlink-compatible consumers.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a BTC/USD price feed.
2. Call `updatePriceFeeds()` once to set an initial on-chain price of $100,000.
3. Wait (or simulate via `vm.warp`) until the price is 2 hours old and the real market price has dropped to $80,000.
4. Call `latestRoundData()` on `PythAggregatorV3`. It returns `answer = 100000e8` (stale) with no revert.
5. The standard Chainlink staleness check `require(answeredInRound >= roundId)` passes because both are `uint80(publishTime)`.
6. A lending protocol using this adapter computes collateral value at $100,000 instead of $80,000, allowing the attacker to borrow more than the real collateral supports. [7](#0-6)

    function getPriceNoOlderThan(
        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
        price = getPriceUnsafe(id);

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();

        return price;
    }
