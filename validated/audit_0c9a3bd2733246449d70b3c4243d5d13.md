Audit Report

## Title
Unbounded JSON-RPC Batch Request Processing Enables RPC Denial of Service by Default — (`File: rpc/src/server.rs`)

## Summary
The CKB JSON-RPC server's batch-size guard relies on a `static OnceLock<usize>` that is only initialized when `rpc_batch_limit` is explicitly set to `Some(n)` in config. The shipped `ckb.toml` leaves `rpc_batch_limit` commented out, so the `OnceLock` is never written and the guard is unconditionally bypassed at runtime. Any caller can POST a single HTTP request containing thousands of JSON-RPC calls, all of which are processed sequentially before a response is returned, exhausting the RPC worker pool.

## Finding Description

**Root cause — `rpc_batch_limit` defaults to `None`:**
`util/app-config/src/configs/rpc.rs` declares the field as `pub rpc_batch_limit: Option<usize>` with no `#[serde(default)]` or explicit default value, so it deserializes as `None` when absent. [1](#0-0) 

The shipped production config explicitly leaves it disabled: [2](#0-1) 

**Guard initialization is conditional:**
In `RpcServer::new`, `JSONRPC_BATCH_LIMIT` is only initialized when `config.rpc_batch_limit` is `Some`. When it is `None`, the `OnceLock` is never written. [3](#0-2) 

**Guard is skipped at request time:**
In `handle_jsonrpc`, the batch-size check is wrapped in `if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()`. When the `OnceLock` was never initialized, `.get()` returns `None` and the entire rejection branch is bypassed. [4](#0-3) 

**All calls dispatched with no bound:**
After the skipped guard, every call in the batch is dispatched sequentially via `stream::iter(calls).then(...)` with no size cap. [5](#0-4) 

**Secondary finding — `max_request_body_size` is never applied:**
`start_server` does not accept `max_request_body_size` as a parameter, and the axum `Router` built inside it applies no `DefaultBodyLimit` or `RequestBodyLimitLayer` (confirmed: no such layer exists anywhere in `rpc/src/`). The configured 10 MiB value is silently ignored; axum's internal 2 MB default applies instead. [6](#0-5) 

Within axum's 2 MB default, a minimal call body (~40 bytes) allows approximately 50,000 calls per batch request. The test harness itself acknowledges the risk by explicitly setting `rpc_batch_limit: Some(1000)` — a guard that production nodes do not get by default. [7](#0-6) 

## Impact Explanation

An attacker who can reach the RPC port can repeatedly POST ~2 MB batches of ~50,000 calls. The server processes every call before returning, holding a worker for the full duration. Concurrent such requests saturate the RPC worker pool, making the RPC endpoint unresponsive to legitimate callers (miners, wallets, indexers). This maps to **Note (0–500 points) — Any local RPC API crash**, since the default binding is `127.0.0.1` and the direct impact is RPC service unavailability rather than node process termination or consensus disruption.

## Likelihood Explanation

The default `127.0.0.1` binding limits exposure to local processes, but the attack requires only a standard HTTP client and knowledge of the JSON-RPC batch format — no authentication, no special privileges. Many production deployments expose the RPC port via reverse proxies or bind to `0.0.0.0`, widening the attacker surface. The attack is trivially repeatable and requires no cryptographic material.

## Recommendation

- **Short term:** Set a safe non-`None` default for `rpc_batch_limit` in `RpcConfig` (e.g., 100–500) so the guard in `handle_jsonrpc` is always active without requiring operator action.
- **Short term:** Pass `max_request_body_size` into `start_server` and apply it via `axum::extract::DefaultBodyLimit::max(config.max_request_body_size)` as a router layer, so the configured value is actually enforced.
- **Long term:** Unsafe defaults (unlimited batch size, unenforced body limit) should require explicit opt-in, not opt-out.

## Proof of Concept

```python
import json, requests

# ~50,000 minimal calls fit within axum's 2 MB default body limit
batch = [{"jsonrpc": "2.0", "method": "get_tip_block_number", "id": i}
         for i in range(50_000)]

r = requests.post(
    "http://127.0.0.1:8114",
    json=batch,
    headers={"Content-Type": "application/json"},
    timeout=300,
)
print(r.status_code, len(r.json()))
```

With `rpc_batch_limit` absent from `ckb.toml` (the default), the server processes all calls per request. Sending this repeatedly from multiple threads will exhaust the RPC worker pool. The fix can be verified by confirming that adding `rpc_batch_limit = 500` to `ckb.toml` causes the server to return an `invalid params` error for batches exceeding the limit.
