**Execution order inside the write lock:**

1. `check_rbf` validates the new tx passes RBF rules (fee > sum(conflicts) + extra_rbf_fee)
2. `process_rbf` — **irreversibly** removes old conflicting txs from `pool_map`, calls `record_conflict` (moves them to `conflicts_cache`), and fires reject callbacks
3. `_submit_entry` — inserts the new tx into `pool_map` (succeeds)
4. `limit_size` — evicts lowest fee-rate txs until pool fits within `max_tx_pool_size`; if the new tx is the lowest fee-rate entry, it is evicted and `limit_size` returns `Some(Reject::Full(...))`
5. `.map_or(Ok(()), Err)?` — converts that `Some(reject)` into `Err`, causing early return from the closure
6. The `may_recovered_txs` spawn at line 154 is **never reached**

The critical asymmetry: `process_rbf` is not rolled back when `limit_size` later evicts the new tx.

**Can the new tx have a higher total fee but lower fee rate than the old txs?**

Yes. `check_rbf` only checks total fee: `new_fee >= sum(conflict_fees) + extra_rbf_fee`. A large new tx (many inputs/outputs) can satisfy this while having a lower `fee/size` ratio than the old txs. `limit_size` evicts by fee rate (lowest first), so the new tx gets evicted while the old high-fee-rate txs are already gone from `pool_map`.

### Title
RBF Replacement Leaves Pool in Inconsistent State When New Tx Is Evicted by `limit_size` — (`tx-pool/src/process.rs`)

### Summary
When RBF is enabled and a new transaction passes `check_rbf`, `process_rbf` unconditionally removes the old conflicting transactions from `pool_map` and moves them to `conflicts_cache`. If `limit_size` subsequently evicts the new transaction (because it has the lowest fee rate in the pool), the closure returns an error without restoring the old transactions, permanently removing valid transactions from the pool without a successful replacement.

### Finding Description

Inside `submit_entry`, the write-locked closure executes in this order: [1](#0-0) 

`process_rbf` at line 136 removes all conflicting entries from `pool_map` and calls `record_conflict` for each: [2](#0-1) 

`record_conflict` places the old tx into the bounded LRU `conflicts_cache`: [3](#0-2) 

After `_submit_entry` succeeds (new tx is in `pool_map`), `limit_size` is called: [4](#0-3) 

`limit_size` evicts the lowest fee-rate entry. If that is the new tx, it returns `Some(Reject::Full(...))`. The `.map_or(Ok(()), Err)?` at line 152 converts this to an `Err`, causing early return. The `may_recovered_txs` spawn at line 154 is never reached: [5](#0-4) 

There is no rollback of `process_rbf`. The old conflicting txs remain only in `conflicts_cache` (LRU, capped at 10,000 entries): [6](#0-5) 

### Impact Explanation

Valid, high-fee-rate transactions are permanently evicted from `pool_map` without a successful replacement. They reside only in the bounded `conflicts_cache` (10,000 entries), where they can be silently displaced by subsequent LRU evictions. The pool's effective capacity shrinks, and those transactions are no longer relayed or included in block templates.

### Likelihood Explanation

Requires: RBF enabled (`min_rbf_rate > min_fee_rate`), pool near `max_tx_pool_size`, and a crafted replacement tx with high total fee (to pass `check_rbf` rule #3/#4) but low fee rate (large serialized size). All three conditions are attainable by a remote peer submitting a transaction via P2P relay or RPC. No privileged access is required.

### Recommendation

Move `limit_size` before `process_rbf`, or check whether the new tx would survive `limit_size` before committing the removal of old conflicting txs. Alternatively, if `limit_size` evicts the new tx, restore the old conflicting txs from `conflicts_cache` back into `pool_map` before returning the error.

### Proof of Concept

1. Fill pool to `max_tx_pool_size` with medium-fee-rate txs.
2. Submit tx A (small, high fee rate) — it enters the pool.
3. Submit tx B that conflicts with A: B has a large serialized size, total fee > A's fee + `extra_rbf_fee` (passes `check_rbf`), but fee rate < A's fee rate and lower than all other pool entries.
4. `process_rbf` removes A from `pool_map`, records it in `conflicts_cache`.
5. `_submit_entry` inserts B into `pool_map`.
6. `limit_size` evicts B (lowest fee rate), returns `Reject::Full`.
7. Closure returns `Err`; `submit_entry` returns error to caller.
8. Assert: A is absent from `pool_map`, B is absent from `pool_map`, A is in `conflicts_cache`, net pool size is reduced by A's size.

    pub(crate) fn record_conflict(&mut self, tx: TransactionView) {
        let short_id = tx.proposal_short_id();
        for inputs in tx.input_pts_iter() {
            self.conflicts_outputs_cache.put(inputs, short_id.clone());
        }
        self.conflicts_cache.put(short_id.clone(), tx);
        debug!(
            "record_conflict {:?} now cache size: {}",
            short_id,
            self.conflicts_cache.len()
        );

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
