### Title
`PythAggregatorV3` Returns Arbitrarily Stale Prices Without Reverting Across All Price-Reading Functions - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink-compatible adapter, explicitly recommended for migrating Chainlink-dependent protocols to Pyth. Every price-reading function in the contract calls `getPriceUnsafe()`, which by definition returns a price from arbitrarily far in the past with no staleness enforcement. During a market crash or feed disruption, the adapter silently returns the last known (stale) price instead of reverting, directly mirroring the Chainlink `minAnswer/maxAnswer` circuit-breaker class of vulnerability.

---

### Finding Description

`PythAggregatorV3` implements the `AggregatorV3Interface` and is the contract Pyth officially ships and recommends for Chainlink migration. All five price-reading functions delegate to `getPriceUnsafe()`:

```solidity
// latestAnswer — line 53-56
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// latestRoundData — line 110
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// getRoundData — line 89
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
```

`getPriceUnsafe()` is documented as returning a price "from arbitrarily far in the past" with no recency guarantee. The safe alternative, `getPriceNoOlderThan()`, is never called anywhere in this contract. [1](#0-0) [2](#0-1) [3](#0-2) 

A second compounding issue: `latestRoundData()` sets both `roundId` and `answeredInRound` to `uint80(price.publishTime)`:

```solidity
roundId = uint80(price.publishTime);
return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
``` [4](#0-3) 

Chainlink-compatible protocols commonly guard against stale data with the check `answeredInRound >= roundId`. Because both values are always set to the same `publishTime` cast, this check **always passes** regardless of how old the price is, silently bypassing the staleness guard that downstream protocols rely on.

The safe `getPriceNoOlderThan()` in `AbstractPyth.sol` reverts with `StalePrice` when the price is too old:

```solidity
if (diff(block.timestamp, price.publishTime) > age)
    revert PythErrors.StalePrice();
``` [5](#0-4) 

This revert path is never reached through `PythAggregatorV3`.

---

### Impact Explanation

Any lending, derivatives, or margin protocol that uses `PythAggregatorV3` as a drop-in Chainlink replacement (its stated purpose) will receive stale prices without any revert signal during:

- A market crash where publishers stop updating (e.g., LUNA-style collapse)
- A network outage causing feed staleness
- Any period where the Pyth on-chain price is not being pushed

The protocol continues operating at the last known price. An attacker can borrow over-collateralized assets at the stale (inflated) price, or avoid liquidation on a position whose collateral has crashed in value. This is the exact impact described in the reference report.

---

### Likelihood Explanation

- `PythAggregatorV3` is the official Pyth-provided Chainlink migration path, documented at `apps/developer-hub/content/docs/price-feeds/core/migrate-an-app-to-pyth/chainlink.mdx`.
- Protocols migrating from Chainlink expect `latestRoundData()` to revert on stale data; the adapter does not.
- The `answeredInRound >= roundId` staleness check that many Chainlink integrations rely on is structurally bypassed.
- Feed staleness events (market stress, network congestion) are not rare. [6](#0-5) 

---

### Recommendation

Replace all `getPriceUnsafe()` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`, where `maxAge` is a configurable constructor parameter (e.g., defaulting to 60 seconds). If the price is stale, the functions should revert, matching the behavior downstream Chainlink-compatible protocols expect.

Additionally, `roundId` and `answeredInRound` should not both be set to `publishTime`; a monotonically increasing counter should be used, or the contract should clearly document that the `answeredInRound >= roundId` staleness pattern does not apply.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a Pyth contract with a price feed for asset X.
2. Update the Pyth on-chain price for X to $100 at time T.
3. Wait 24 hours without pushing any new price update (simulating a feed outage or market crash).
4. Call `latestRoundData()` on `PythAggregatorV3`.
5. Observe: returns `answer = 100`, `updatedAt = T` (24 hours ago), no revert.
6. Observe: `answeredInRound == roundId` (both equal `uint80(T)`), so any downstream `answeredInRound >= roundId` check passes.
7. A lending protocol using this adapter accepts the stale $100 price and allows borrowing against collateral that may now be worth $0. [7](#0-6) [8](#0-7)

    function getPriceNoOlderThan(
        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
        price = getPriceUnsafe(id);

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();

        return price;
    }
