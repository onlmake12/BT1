Audit Report

## Title
Unconditional Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary
`confirm_blocks_to_failed_txs` is allocated with exactly `max_confirm_blocks` (1000) elements. `remove_unconfirmed_tx` correctly guards the `block_unconfirmed_txs` write against `tx_age >= len`, but the `confirm_blocks_to_failed_txs[tx_age - 1]` write at line 215 is unconditional and outside that guard. When a tracked transaction is rejected after 1001+ blocks with `count_failure=true`, `tx_age - 1 = 1000` indexes past the last valid index (999), causing an unconditional Rust index-out-of-bounds panic that crashes the node process.

## Finding Description
`confirm_blocks_to_failed_txs` is initialized with length `max_confirm_blocks` (= `MAX_CONFIRM_BLOCKS` = 1000): [1](#0-0) 

In `remove_unconfirmed_tx`, the `block_unconfirmed_txs` access is correctly guarded by a bounds check: [2](#0-1) 

However, the `count_failure` branch immediately following is **outside** that guard and performs no bounds check before indexing `confirm_blocks_to_failed_txs`: [3](#0-2) 

When `tx_age = 1001`, `tx_age - 1 = 1000` is past the last valid index (999) of the 1000-element `confirm_blocks_to_failed_txs`. Rust panics unconditionally.

The full call chain is confirmed in the codebase:

- `remove_expired` calls `callbacks.call_reject`: [4](#0-3) 

- The registered reject callback calls `reject_tx` on the fee estimator, which calls `drop_tx`: [5](#0-4) 

- `drop_tx` calls `drop_tx_inner` with `count_failure=true`, which calls `remove_unconfirmed_tx`: [6](#0-5) 

`track_tx` only records transactions at the current `best_height`: [7](#0-6) 

So after 1001+ blocks, `tx_age = best_height - entry_height >= 1001`, and the unguarded `confirm_blocks_to_failed_txs[1000]` access panics.

## Impact Explanation
The Rust index-out-of-bounds panic is unrecoverable and propagates up through the tx-pool service thread, crashing the node process. This matches the **High** impact class: *"Vulnerabilities which could easily crash a CKB node"* (10001–15000 points). There is no recovery path.

## Likelihood Explanation
CKB produces blocks roughly every 8–10 seconds; 1001 blocks ≈ 2.2–2.8 hours. The default tx-pool expiry is 12 hours, so a transaction easily survives 1001 blocks within the expiry window. When expiry fires, `remove_expired` → `call_reject` → `reject_tx` → `drop_tx_inner(count_failure=true)` → `remove_unconfirmed_tx` triggers the panic. Alternatively, an attacker can submit a conflicting (double-spend) transaction after 1001 blocks to trigger the same rejection path immediately. No special privileges, PoW, or majority hashpower are required — only the ability to submit a valid transaction to the mempool.

## Recommendation
Add a bounds check before the `confirm_blocks_to_failed_txs` write, mirroring the existing guard on `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This silently drops the failure sample for transactions older than `MAX_CONFIRM_BLOCKS`, consistent with how `block_unconfirmed_txs` already handles the same case.

## Proof of Concept
```rust
// Reproduces with max_confirm_blocks = 10
let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
let mut stat = TxConfirmStat::new(buckets, 10, 0.993);

// Track a tx at height 0
let bucket_index = stat.add_unconfirmed_tx(0, FeeRate::from_u64(1000)).unwrap();

// 11 blocks later, reject it with count_failure=true
// tx_age = 11, confirm_blocks_to_failed_txs[10] → index 10 on a 10-element Vec → PANIC
stat.remove_unconfirmed_tx(0, 11, bucket_index, true);
```

In production: submit a valid tx at height H, wait for `best_height` to reach H+1001 (≈2.2 hours), then either wait for expiry or submit a conflicting tx. The original tx is rejected via `reject_tx` → `drop_tx_inner(count_failure=true)` → `remove_unconfirmed_tx` → `confirm_blocks_to_failed_txs[1000]` → **index out of bounds → node process crash**.
