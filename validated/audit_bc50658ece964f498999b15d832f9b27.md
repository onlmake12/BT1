### Title
Stale Pre-Eviction Accounting Snapshot Overwrites In-Flight `update_stat_for_remove_tx` Decrements in `add_entry`, Inflating `total_tx_size`/`total_tx_cycles` — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::add_entry` captures a pre-eviction snapshot of `total_tx_size`/`total_tx_cycles`, then calls `check_and_record_ancestors`, which may evict transactions and correctly decrement those counters via `update_stat_for_remove_tx`. However, `add_entry` unconditionally overwrites `self.total_tx_size` and `self.total_tx_cycles` with the stale snapshot at the end, discarding the decrements. This inflates the pool's reported size, causing `limit_size` to evict legitimate transactions that should not have been removed.

---

### Finding Description

In `add_entry` (`tx-pool/src/component/pool_map.rs`, lines 200–221):

**Step 1 — Snapshot taken before any eviction (lines 210–211):**
```rust
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
// total_tx_size = self.total_tx_size + entry.size  (local variable)
// total_tx_cycles = self.total_tx_cycles + entry.cycles  (local variable)
```

**Step 2 — Evictions happen inside `check_and_record_ancestors` (line 213):**

When the new transaction's ancestor count exceeds `max_ancestors_count` and `cell_ref_parents` exist (transactions that hold one of the new tx's inputs as a cell dep), `check_and_record_ancestors` calls `remove_entry_and_descendants` (line 618) → `remove_entry` (line 263) → `update_stat_for_remove_tx` (line 247), which **decrements** `self.total_tx_size` and `self.total_tx_cycles` for every evicted entry.

**Step 3 — Stale snapshot overwrites the decremented values (lines 218–219):**
```rust
self.total_tx_size = total_tx_size;   // stale: ignores eviction decrements
self.total_tx_cycles = total_tx_cycles; // stale: ignores eviction decrements
```

The decrements applied by `update_stat_for_remove_tx` during eviction are silently discarded. After `add_entry` returns, `self.total_tx_size` is inflated by exactly the sum of sizes of all evicted transactions.

The `cell_ref_parents` eviction path is reachable by any unprivileged tx-pool submitter: it fires whenever a submitted transaction's input outpoint is already referenced as a cell dep by another in-pool transaction, and the combined ancestor count exceeds `max_ancestors_count`.

---

### Impact Explanation

`limit_size` (`tx-pool/src/pool.rs`, line 298) drives its eviction loop entirely from `self.pool_map.total_tx_size`:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee-rate transactions
}
```

With `total_tx_size` inflated by the ghost sizes of already-removed transactions, `limit_size` believes the pool is over-capacity when it is not, and evicts additional legitimate transactions with `Reject::Full`. Each subsequent trigger of the bug compounds the inflation. An attacker who repeatedly triggers the eviction path can drive `total_tx_size` far above `max_tx_pool_size`, causing the pool to continuously evict all newly submitted and existing transactions, effectively rendering the mempool unusable. The `get_tx_pool_info` / `get_raw_tx_pool` RPCs also report the inflated figures, misleading operators and fee-estimation logic.

---

### Likelihood Explanation

The trigger condition is reachable by any unprivileged tx-pool submitter with no special privileges:

1. Submit a chain of N transactions (N ≥ `max_ancestors_count`, default 25) where each spends the previous output.
2. Submit a separate transaction `B` that uses the Nth chain output as a **cell dep** (not spending it).
3. Submit transaction `C` that **spends** the Nth chain output.

`C`'s ancestor set now includes the full chain (N entries) plus `B` (cell_ref_parent), totalling N+1 > `max_ancestors_count`. `check_and_record_ancestors` evicts `B` to bring the count within limits, triggering the stale-snapshot overwrite. The attacker can repeat this pattern with fresh chains to accumulate unbounded inflation of `total_tx_size`.

---

### Recommendation

Move the final assignment of `self.total_tx_size` and `self.total_tx_cycles` to **after** `check_and_record_ancestors` completes, reading the already-updated `self` fields and adding only the new entry's contribution:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Correct: add new entry's contribution to the post-eviction totals
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

Alternatively, call `updated_stat_for_add_tx` after `check_and_record_ancestors` returns, so the snapshot is taken from the already-decremented `self` fields.

---

### Proof of Concept

Assume `max_ancestors_count = 25`, `max_tx_pool_size = 10_000` bytes, and each transaction is 200 bytes.

1. Submit chain: `tx_A1 → tx_A2 → … → tx_A25` (25 txs, 5 000 bytes total). `total_tx_size = 5 000`.
2. Submit `tx_B` (200 bytes) with `cell_dep = outpoint(tx_A25, 0)`. `total_tx_size = 5 200`.
3. Submit `tx_C` (200 bytes) spending `outpoint(tx_A25, 0)`.
   - `updated_stat_for_add_tx` snapshots `total_tx_size_snap = 5 200 + 200 = 5 400`.
   - `check_and_record_ancestors`: ancestor count = 26 > 25; `tx_B` is a `cell_ref_parent`; `remove_entry_and_descendants(tx_B)` → `update_stat_for_remove_tx(200, …)` → `self.total_tx_size = 5 200 − 200 = 5 000`.
   - `self.total_tx_size = total_tx_size_snap = 5 400` ← **stale overwrite**.
   - Actual pool: 25 + 1 = 26 txs × 200 = 5 200 bytes. Reported: **5 400 bytes** (inflated by 200).
4. Repeat steps 1–3 with fresh chains. Each iteration adds 200 bytes of phantom inflation.
5. After ~25 iterations, `total_tx_size` exceeds `max_tx_pool_size = 10 000` while the real pool holds only ~5 200 bytes. `limit_size` begins evicting all legitimate transactions with `Reject::Full`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    ) -> Result<HashSet<TxEntry>, Reject> {

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
