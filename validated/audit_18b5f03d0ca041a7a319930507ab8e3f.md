### Title
Excess ETH Permanently Lost to Protocol Fee Pool in Entropy Request Functions — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The `request()`, `requestWithCallback()`, and all `requestV2()` overloads in `Entropy.sol` are `payable` functions that accept ETH but never refund any excess above the required fee. Any ETH sent beyond `getFeeV2(provider, callbackGasLimit)` is silently credited to `_state.accruedPythFeesInWei` — permanently transferred from the user to the Pyth protocol treasury — with no refund path.

### Finding Description

In `requestHelper()`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value - providerFee` is credited to `accruedPythFeesInWei`. If `msg.value > requiredFee`, the surplus is silently absorbed into the Pyth fee pool rather than returned to the caller. The NatSpec comments on the public-facing functions confirm this is the current behavior:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

All `requestV2()` overloads (lines 286–310) are also `payable` and route through the same `requestHelper`, inheriting the same behavior: [4](#0-3) 

### Impact Explanation

Any user who sends more ETH than the exact required fee permanently loses the surplus to the Pyth protocol treasury. The ETH is not recoverable by the user — it can only be withdrawn by the Pyth admin via a privileged governance action. From the user's perspective, the excess ETH is irretrievably lost. This is a direct financial loss for any caller who:

- Estimates fees conservatively and overpays
- Uses a wrapper contract that forwards a rounded or pre-computed ETH amount
- Is integrating via a third-party SDK that does not compute the exact fee

### Likelihood Explanation

The likelihood is medium-to-high. Fee estimation is a common source of off-by-one or rounding errors in integrations. The `getFeeV2()` function must be called on-chain immediately before the request to get the exact fee; any cached or approximate value will result in overpayment. Wrapper contracts (e.g., `requestWithCallback` calling `requestV2`) add an additional layer where callers may not know the exact fee. The behavior is not surfaced as a revert — it silently succeeds while consuming the excess ETH.

### Recommendation

Add an excess-ETH refund at the end of `requestHelper()`, analogous to the pattern already used in `PythLazer.sol`:

```solidity
if (msg.value > requiredFee) {
    payable(msg.sender).transfer(msg.value - requiredFee);
}
``` [5](#0-4) 

Alternatively, change the functions to revert on overpayment (`require(msg.value == requiredFee)`), forcing callers to supply the exact fee.

### Proof of Concept

1. Deploy `EntropyUpgradable` on a testnet.
2. Call `getFeeV2(provider, 0)` → returns `X` wei.
3. Call `requestV2{value: X + 1 ether}(provider, 0)` — transaction succeeds.
4. Observe that `getProviderInfo(provider).accruedFeesInWei` increased by `providerFee` and `accruedPythFeesInWei` increased by `X + 1 ether - providerFee` (absorbing the full 1 ETH surplus).
5. The caller has no mechanism to recover the 1 ETH surplus. [6](#0-5)

    function requestHelper(
        address provider,
        bytes32 userCommitment,
        bool useBlockhash,
        bool isRequestWithCallback,
        uint32 callbackGasLimit
    ) internal returns (EntropyStructsV2.Request storage req) {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];
        if (_state.providers[provider].sequenceNumber == 0)
            revert EntropyErrors.NoSuchProvider();

        // Assign a sequence number to the request
        uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
        if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
            revert EntropyErrors.OutOfRandomness();
        providerInfo.sequenceNumber += 1;

        // Check that fees were paid and increment the pyth / provider balances.
        uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
        if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
        uint128 providerFee = getProviderFee(provider, callbackGasLimit);
        providerInfo.accruedFeesInWei += providerFee;
        _state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) -
            providerFee);
```
