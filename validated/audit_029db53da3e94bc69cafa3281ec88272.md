### Title
Provider Can Front-Run `requestV2`/`requestWithCallback` to Drain Excess ETH From Users — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestV2()`, `requestWithCallback()`, and `request()` functions in `Entropy.sol` accept `msg.value` with no upper-bound (`maxFee`) parameter. A registered provider can call `setProviderFee()` to raise their fee between the moment a user reads the fee off-chain and the moment the user's transaction is mined. Because the contract explicitly does **not** refund excess ETH, any ETH the user sent above the new fee is silently consumed by the contract (split between the provider and the Pyth protocol fee pool). This is a direct analog of the ERC721Sale front-running finding.

---

### Finding Description

`requestHelper()` enforces only a lower bound on payment:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

There is no upper-bound check. The public-facing functions explicitly document this:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

A provider can change their fee at any time with no time-lock or delay:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    ...
}
``` [4](#0-3) 

None of the public entry points (`requestV2()`, `requestWithCallback()`, `request()`) accept a `maxFee` argument: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Direct ETH loss for users who send excess ETH.** The official SDK documentation and the developer-hub tutorial both show the canonical integration pattern:

```solidity
uint128 requestFee = entropy.getFeeV2();
if (msg.value < requestFee) revert("not enough fees");
uint64 sequenceNumber = entropy.requestV2{ value: requestFee }(…);
``` [7](#0-6) 

In practice, many integrating contracts add a safety buffer (e.g., `msg.value = requestFee * 2`) to avoid reverts from minor fee fluctuations. A malicious provider can front-run any such transaction and raise `feeInWei` to exactly `msg.value`, causing the user to pay the full buffered amount with no refund. The provider's `accruedFeesInWei` increases by the full new `providerFee`, draining the user's ETH beyond what they consented to.

Even without a buffer, a provider can cause a targeted DoS by front-running with a fee increase that makes `msg.value < requiredFee`, reverting the user's transaction repeatedly at will.

---

### Likelihood Explanation

- Any address can register as an Entropy provider (permissionless).
- `setProviderFee()` has no time-lock, no delay, and no governance check — it takes effect in the same block.
- Mempool visibility of pending `requestV2` transactions is standard on all EVM chains where Entropy is deployed.
- The keeper infrastructure (`apps/fortuna`) already monitors and adjusts fees programmatically, demonstrating that automated fee manipulation is operationally feasible. [8](#0-7) 

---

### Recommendation

Add a `maxFeeInWei` parameter to `requestV2`, `requestWithCallback`, and `request`. Revert if `requiredFee > maxFeeInWei`:

```solidity
function requestV2(
    address provider,
    bytes32 userContribution,
    uint32 gasLimit,
    uint128 maxFeeInWei   // NEW
) public payable returns (uint64) {
    uint128 requiredFee = getFeeV2(provider, gasLimit);
    if (requiredFee > maxFeeInWei) revert EntropyErrors.FeeTooHigh();
    ...
}
```

This mirrors the fix applied to `ERC721Sale` (`expectedPaymentToken` + `maxTotal`).

---

### Proof of Concept

```
Setup:
  - MaliciousProvider registers with Entropy, sets feeInWei = 0.001 ETH.
  - Alice's integrating contract sends msg.value = 0.002 ETH (2× buffer) when calling requestV2().

Attack:
  1. Alice's contract calls getFeeV2(maliciousProvider) → 0.001 ETH.
  2. Alice's contract submits requestV2{value: 0.002 ETH}(maliciousProvider, …).
  3. MaliciousProvider observes the pending tx in the mempool.
  4. MaliciousProvider front-runs with setProviderFee(0.002 ETH).
  5. Alice's tx executes:
       requiredFee = 0.002 ETH  (new fee)
       msg.value   = 0.002 ETH  ✓ (passes the lower-bound check)
       providerInfo.accruedFeesInWei += 0.002 ETH  (provider captures full amount)
       _state.accruedPythFeesInWei  += 0           (nothing left for Pyth)
  6. Alice paid 0.002 ETH instead of the expected 0.001 ETH, with no refund.
     MaliciousProvider gained an extra 0.001 ETH per request.
``` [9](#0-8) [10](#0-9)
