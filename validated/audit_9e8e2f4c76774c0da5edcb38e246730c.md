### Title
Missing Transaction Deadline in `requestPriceUpdatesWithCallback` Allows Stale Price Delivery to Consumer Contracts - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

---

### Summary

The `requestPriceUpdatesWithCallback` function in `Echo.sol` has no transaction deadline/expiry parameter. A user who submits this function with `publishTime` set to the current time (computed off-chain at submission) will have their request stored with a stale `publishTime` if the transaction is delayed in the mempool. The callback will then deliver a stale price to the consumer contract, which may use it for financial decisions.

---

### Finding Description

`requestPriceUpdatesWithCallback` accepts a caller-supplied `publishTime` parameter. The only temporal guard is an upper-bound check:

```solidity
require(publishTime <= block.timestamp + 60, "Too far in future");
``` [1](#0-0) 

There is no lower-bound check on `publishTime` and, critically, **no transaction deadline parameter**. `publishTime` is a value the caller computes off-chain (typically `block.timestamp` at the moment they build the transaction). If the transaction sits in the mempool for minutes or hours before inclusion, `block.timestamp` at execution time will be far ahead of the `publishTime` the user supplied. The request is stored as-is:

```solidity
req.publishTime = publishTime;
``` [2](#0-1) 

When `executeCallback` is later called, it passes `req.publishTime` directly as both `minPublishTime` and `maxPublishTime` to `parsePriceFeedUpdates`:

```solidity
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(
    updateData, priceIds,
    SafeCast.toUint64(req.publishTime),
    SafeCast.toUint64(req.publishTime)
);
``` [3](#0-2) 

The consumer's `echoCallback` then receives price feeds whose `publishTime` equals the stale value the user originally supplied, not the current time. [4](#0-3) 

The interface documentation confirms `publishTime` is user-controlled and is expected to be near `block.timestamp`:

```
@param publishTime The minimum publish time for price updates,
       it should be less than or equal to block.timestamp + 60
``` [5](#0-4) 

---

### Impact Explanation

Consumer contracts that call `requestPriceUpdatesWithCallback` expecting to receive a near-current price will instead receive a price anchored to a stale `publishTime` if the request transaction is delayed. Any financial logic in `echoCallback` (e.g., liquidation thresholds, swap pricing, collateral valuation) will operate on stale data, potentially causing incorrect outcomes — identical in class to the LOB stale-order execution described in the external report.

---

### Likelihood Explanation

Transaction delays are routine during periods of network congestion or when users submit transactions with low gas prices. A user who sets `publishTime = block.timestamp` (the natural and documented usage) and whose transaction is delayed by even a few minutes will silently receive a stale price with no on-chain indication that anything went wrong. No privileged access is required; any unprivileged caller of `requestPriceUpdatesWithCallback` is affected.

---

### Recommendation

Add a `deadline` parameter to `requestPriceUpdatesWithCallback`. Revert at the start of execution if `block.timestamp > deadline`:

```solidity
function requestPriceUpdatesWithCallback(
    address provider,
    uint64 publishTime,
    bytes32[] calldata priceIds,
    uint32 callbackGasLimit,
    uint64 deadline          // <-- new parameter
) external payable override returns (uint64 requestSequenceNumber) {
    require(block.timestamp <= deadline, "Request expired");
    require(publishTime <= block.timestamp + 60, "Too far in future");
    // ...
}
```

This mirrors the pattern used in the Express Relay SVM bid construction, where a `deadline` field is explicitly required: [6](#0-5) 

---

### Proof of Concept

1. At time `T`, user constructs a transaction calling `requestPriceUpdatesWithCallback(provider, publishTime=T, priceIds, gasLimit)` and submits it with a low gas price.
2. The transaction sits in the mempool. At time `T + 3600` (1 hour later), it is included in a block.
3. Inside the function, `block.timestamp = T + 3600`. The check `T <= (T+3600) + 60` passes.
4. The request is stored with `req.publishTime = T`.
5. A keeper calls `executeCallback`. `parsePriceFeedUpdates` is called with `minPublishTime = maxPublishTime = T`, fetching a price that is 1 hour old.
6. `echoCallback` on the consumer contract receives a 1-hour-old price feed.
7. The consumer contract executes financial logic (e.g., a liquidation check) based on the stale price, potentially causing an incorrect outcome. [7](#0-6)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
