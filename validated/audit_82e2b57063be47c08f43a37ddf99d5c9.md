### Title
Collected Verification Fees in `PythLazer.sol` Are Permanently Locked With No Withdrawal Mechanism — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

The `PythLazer.sol` contract collects ETH via its `payable` `verifyUpdate` function but contains no mechanism to withdraw the accumulated `verification_fee` balance. Every call to `verifyUpdate` by any Lazer updater permanently locks the exact fee amount in the contract, with no owner or governance function to recover it.

---

### Finding Description

In `PythLazer.sol`, the `verifyUpdate` function is declared `external payable` and enforces a minimum payment of `verification_fee` wei:

```solidity
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [1](#0-0) 

The excess above `verification_fee` is correctly refunded to the caller. However, the exact `verification_fee` amount is retained in the contract's ETH balance. The complete contract exposes only four external/public functions:

- `updateTrustedSigner` (owner-only, manages signers)
- `isValidSigner` (view)
- `verifyUpdate` (payable, collects fees)
- `version` (pure) [2](#0-1) 

None of these functions transfer ETH out of the contract. There is no `withdrawFee`, `sweep`, or equivalent function. The contract is not shown to use a proxy/upgradeable pattern, and `Ownable` does not provide a fee-withdrawal function. As a result, every `verification_fee` paid by every Lazer updater accumulates in the contract permanently.

This is in direct contrast to the analogous Entropy contract, which has an explicit `withdrawFee` in `EntropyGovernance.sol`:

```solidity
function withdrawFee(address targetAddress, uint128 amount) external {
    _authoriseAdminAction();
    _state.accruedPythFeesInWei -= amount;
    (bool success, ) = targetAddress.call{value: amount}("");
    ...
}
``` [3](#0-2) 

`PythLazer.sol` has no equivalent.

---

### Impact Explanation

All ETH paid as `verification_fee` to `PythLazer.sol` is permanently locked in the contract with no recovery path. As Lazer adoption grows and more integrators call `verifyUpdate`, the locked balance grows monotonically. If the contract is non-upgradeable (no proxy pattern is present), these funds are irrecoverable. Even if the owner rotates signers or adjusts the fee, previously collected ETH cannot be extracted.

---

### Likelihood Explanation

Likelihood is **high**. Every single call to `verifyUpdate` — the primary entry point for all Lazer users — contributes to the locked balance. This is the normal, intended usage path. No special conditions or attacker behavior are required; the locking occurs automatically on every legitimate fee payment.

---

### Recommendation

Add an owner-restricted fee withdrawal function to `PythLazer.sol`, analogous to `EntropyGovernance.withdrawFee`:

```solidity
function withdrawFees(address payable recipient, uint256 amount) external onlyOwner {
    require(address(this).balance >= amount, "Insufficient balance");
    (bool success, ) = recipient.call{value: amount}("");
    require(success, "Transfer failed");
    emit FeesWithdrawn(recipient, amount);
}
```

This mirrors the pattern already used in `EntropyGovernance.sol` and `Echo.sol`. [4](#0-3) 

---

### Proof of Concept

1. Deploy `PythLazer.sol` with `verification_fee = 1000 wei`.
2. Any Lazer updater calls `verifyUpdate{value: 1000}(update)`.
3. The call succeeds; `1000 wei` is now in `address(PythLazer).balance`.
4. Repeat for N callers — balance grows to `N * 1000 wei`.
5. Attempt to call any function to withdraw ETH — no such function exists.
6. The entire accumulated balance is permanently locked.

The attacker-controlled entry path is the unprivileged `verifyUpdate` call, which is the standard usage path for all Lazer integrators. No privileged access is required to trigger the locking; it occurs on every normal fee payment. [5](#0-4)
