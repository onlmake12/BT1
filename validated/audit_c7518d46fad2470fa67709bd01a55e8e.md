### Title
Index-Out-of-Bounds Panic in `remove_unconfirmed_tx` When `count_failure=true` and `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

The `ConfirmationFraction` fee estimator's `remove_unconfirmed_tx` function contains a missing bounds check on the `confirm_blocks_to_failed_txs` array access inside the `count_failure` branch. When a tracked transaction has aged more than `MAX_CONFIRM_BLOCKS = 1000` blocks and is then evicted from the pool (triggering `count_failure=true`), the code accesses `confirm_blocks_to_failed_txs[tx_age - 1]` where `tx_age - 1 >= 1000`, which is out of bounds for a vec of length 1000. In Rust, this causes an immediate panic, crashing the node.

---

### Finding Description

`confirm_blocks_to_failed_txs` is initialized with exactly `MAX_CONFIRM_BLOCKS = 1000` rows: [1](#0-0) [2](#0-1) 

In `remove_unconfirmed_tx`, the guard at line 208 only protects the `block_unconfirmed_txs` access. The `count_failure` branch at line 214–216 has **no corresponding bounds check**: [3](#0-2) 

When `tx_age >= 1001`, `tx_age - 1 >= 1000`, which is out of bounds for a vec of length 1000 (valid indices: `0..=999`). The existing guard at line 208 (`tx_age >= self.block_unconfirmed_txs.len()`) only redirects the unconfirmed-tx accounting to `old_unconfirmed_txs`; it does **not** suppress or bound the `confirm_blocks_to_failed_txs` write below it.

The call chain that reaches this with `count_failure=true`:

- `reject_tx` (line 476) → `drop_tx` (line 429) → `drop_tx_inner(tx_hash, true)` (line 416) → `remove_unconfirmed_tx(..., count_failure=true)` (line 418) [4](#0-3) [5](#0-4) 

`track_tx` records `entry_height = best_height` at the time of submission: [6](#0-5) 

`drop_tx_inner` uses `self.best_height` as `tip_height`: [7](#0-6) 

So `tx_age = self.best_height - entry_height`. After 1001 blocks, `tx_age = 1001`, `tx_age - 1 = 1000` → panic.

---

### Impact Explanation

A Rust index-out-of-bounds access panics unconditionally. If this executes on a thread that owns critical node state (the fee estimator is updated inline during block processing and tx-pool operations), the panic propagates and crashes the node process. **Impact: remote crash / denial of service of any CKB full node running the `ConfirmationFraction` estimator.**

---

### Likelihood Explanation

The preconditions are fully attacker-controllable via standard, unprivileged operations:

1. **Submit a low-fee transaction** via the standard RPC/P2P path so it is tracked by the estimator.
2. **Wait 1001+ blocks** (~7.8 hours at 28 s/block) — no special access needed.
3. **Trigger pool eviction** of that transaction (e.g., flood the pool with higher-fee transactions to cause pool-full eviction, or submit an RBF replacement). Both paths call `reject_tx` → `drop_tx` → `count_failure=true`.

No privileged access, no hashpower, no key material required.

---

### Recommendation

Add a bounds check inside the `count_failure` branch, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure {
    let max_idx = self.confirm_blocks_to_failed_txs.len();
    if tx_age >= 1 && tx_age - 1 < max_idx {
        self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
    }
    // else: tx aged out of the tracking window; silently discard
}
```

This mirrors the semantics of the `old_unconfirmed_txs` fallback already used for the unconfirmed-tx accounting.

### Proof of Concept

```rust
#[test]
fn test_remove_unconfirmed_tx_no_panic_on_old_failure() {
    let mut stat = TxConfirmStat::default();
    // Add a tx at height 0
    let fee_rate = FeeRate::from_u64(1000);
    let bucket_index = stat.add_unconfirmed_tx(0, fee_rate).unwrap();
    // Simulate 1001 blocks passing, then evict with count_failure=true
    // tx_age = 1001, tx_age - 1 = 1000 >= confirm_blocks_to_failed_txs.len() (1000)
    // This panics on the unpatched code:
    stat.remove_unconfirmed_tx(0, 1001, bucket_index, true);
}
```

Running this test against the current code triggers:
```
thread 'test_remove_unconfirmed_tx_no_panic_on_old_failure' panicked at
'index out of bounds: the len is 1000 but the index is 1000'
```
