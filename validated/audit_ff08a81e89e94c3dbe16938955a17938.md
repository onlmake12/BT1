### Title
Orphan Transaction Pool Exhaustion via Fee-Free Relay — (`File: tx-pool/src/component/orphan.rs`)

---

### Summary

The `OrphanPool` in CKB's tx-pool has a hard cap of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries and evicts entries **randomly** when full. Critically, orphan transactions bypass the `min_fee_rate` check entirely because their inputs cannot be resolved at admission time. Any connected P2P peer can flood the orphan pool with 100 zero-fee crafted transactions referencing non-existent parent hashes, continuously displacing legitimate orphan transactions at zero economic cost.

---

### Finding Description

`OrphanPool` is defined in `tx-pool/src/component/orphan.rs` with a hard limit: [1](#0-0) 

When the pool is full, `limit_size()` first expires timed-out entries, then evicts **randomly** by iterating `HashMap` keys: [2](#0-1) 

The admission path in `after_process()` (`tx-pool/src/process.rs`) adds a transaction to the orphan pool whenever its inputs are unresolvable — **without any fee-rate gate**: [3](#0-2) 

The fee check (`check_tx_fee`) in `tx-pool/src/util.rs` is only reachable after successful input resolution in `_process_tx` → `pre_check`. Orphan transactions fail at the resolve step before `check_tx_fee` is ever called: [4](#0-3) 

An attacker crafts transactions whose inputs reference non-existent `OutPoint`s (random tx hashes). These pass `non_contextual_verify` (structurally valid), fail resolution with `OutPointError::Unknown`, satisfy `is_missing_input`, and are unconditionally inserted into the orphan pool. No CKB funds are required. [5](#0-4) 

---

### Impact Explanation

An attacker maintaining a connection to a victim node can keep the 100-slot orphan pool saturated with zero-fee fake orphans. When a legitimate user's child transaction (whose parent is in-flight) is relayed to the victim node, it enters the orphan pool but is randomly evicted by the next attacker-controlled orphan insertion. The evicted tx is reported back to the relayer as `Reject`, causing it to be marked `unknown` in the bloom filter and not re-requested. This disrupts transaction propagation for any user relying on the orphan pool to buffer child-before-parent delivery, degrading mempool liveness on the targeted node.

---

### Likelihood Explanation

The attack requires only a single P2P connection to the victim node (any unprivileged peer). Crafting structurally valid transactions with fake input `OutPoint`s is trivial and costs nothing on-chain. The pool size of 100 is small enough to saturate in a single burst. The random eviction policy means the attacker does not need to predict which slot to target — continuous insertion at the rate of one tx per eviction cycle is sufficient to maintain saturation. The `ORPHAN_TX_EXPIRE_TIME` of `100 * MAX_BLOCK_INTERVAL` is long enough that natural expiry provides no meaningful relief during an active attack. [6](#0-5) 

---

### Recommendation

1. **Apply a minimum fee-rate check at orphan admission time.** Even though inputs cannot be resolved, the declared fee can be estimated from the transaction's output capacity sum vs. a minimum absolute fee floor. Reject orphans that cannot plausibly meet `min_fee_rate` once resolved.
2. **Enforce per-peer orphan quotas.** Track how many orphan slots each `PeerIndex` occupies and cap per-peer contribution (e.g., 10 of the 100 slots), preventing a single peer from monopolizing the pool.
3. **Replace random eviction with fee-rate-ordered eviction.** Use the declared cycle count and transaction size to compute an estimated fee rate and evict the lowest-rate orphan first, analogous to how `limit_size()` in the main pool evicts by `EvictKey`. [7](#0-6) 

---

### Proof of Concept

```
1. Attacker connects to victim CKB node via RelayV3 P2P protocol.

2. Attacker constructs 100 transactions T_1..T_100 where each T_i:
   - Has one input referencing OutPoint { tx_hash: random_bytes_32, index: 0 }
   - Has one output with capacity = input capacity (zero fee)
   - Has a valid witness (always-success lock or empty)
   - Passes NonContextualTransactionVerifier (structurally valid)

3. Attacker relays T_1..T_100 via RelayTransaction messages.

4. For each T_i, the victim node:
   a. Calls _process_tx → pre_check → resolve inputs → OutPointError::Unknown
   b. is_missing_input() returns true
   c. add_orphan(T_i, peer, declared_cycle) is called
   d. OrphanPool::add_orphan_tx inserts T_i; if pool > 100, random eviction occurs

5. OrphanPool is now full with T_1..T_100 (all attacker-controlled).

6. Legitimate user relays child_tx (parent not yet seen by victim):
   a. child_tx enters orphan pool
   b. limit_size() randomly evicts one entry — with 100/101 probability it evicts child_tx
   c. Evicted child_tx hash is sent as TxVerificationResult::Reject to relayer
   d. Relayer marks child_tx as unknown; it is not re-requested

7. Attacker continuously sends new fake orphans to maintain saturation,
   ensuring child_tx cannot persist in the pool.
``` [2](#0-1) [8](#0-7)

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
