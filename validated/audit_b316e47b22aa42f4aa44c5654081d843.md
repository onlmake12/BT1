### Title
Unsafe `u128 as u64` Truncating Cast in DAO Maximum Withdraw Calculation - (File: `util/dao/src/lib.rs`)

### Summary
In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `u128` result `withdraw_counted_capacity` is cast to `u64` using the Rust `as` operator, which silently truncates on overflow. Every other analogous `u128 → u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, making this a clear inconsistency in a security-critical capacity-accounting path reachable by any DAO-withdrawal transaction sender.

### Finding Description
In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← unsafe truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The same file performs three other `u128 → u64` narrowing conversions using the safe `u64::try_from(…).map_err(|_| DaoError::Overflow)?` pattern:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

The `as u64` cast on line 156 is the sole exception. If `withdraw_counted_capacity` exceeds `u64::MAX`, the value is silently truncated to its lower 64 bits, producing an arbitrarily wrong (smaller) withdrawal capacity without any error being raised.

### Impact Explanation
`calculate_maximum_withdraw` is called from `transaction_maximum_withdraw`, which feeds into both `transaction_fee` (used during transaction verification) and `withdrawed_interests` (used during DAO field computation for block assembly). A truncated result would cause:

1. **Incorrect DAO withdrawal capacity** — a depositor's computed maximum withdrawal is silently reduced, causing either a loss of entitled funds or a transaction that fails capacity checks unexpectedly.
2. **Incorrect DAO field in block headers** — `withdrawed_interests` feeds into `dao_field_with_current_epoch`, which computes the `S` (savings) component of the DAO field committed in every block header. A wrong value here corrupts consensus-critical state. [3](#0-2) [4](#0-3) 

### Likelihood Explanation
The overflow requires `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX ≈ 1.844 × 10¹⁹`. Since the total CKB supply is ~3.36 × 10¹⁸ shannons, the AR ratio `withdrawing_ar / deposit_ar` would need to exceed ~5.5×. The accumulation rate starts at `10¹⁶` and grows slowly via secondary issuance, so reaching a 5.5× ratio would take an extremely long time under normal chain operation. **Likelihood is very low in practice**, but the unsafe cast is a genuine code defect — inconsistent with the rest of the file — in a consensus-critical path reachable by any unprivileged DAO-withdrawal transaction sender.

### Recommendation
Replace the unsafe truncating cast with the same safe pattern used elsewhere in the file:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This is consistent with how `miner_issuance128`, `reward128`, and `ar_increase128` are all converted in the same file. [5](#0-4) 

### Proof of Concept
Any DAO withdrawal transaction triggers `calculate_maximum_withdraw`. The attacker-controlled entry path is:

1. Submit a DAO deposit transaction (standard, unprivileged).
2. Wait for the accumulation rate to grow sufficiently (or craft a scenario where `counted_capacity` is maximized and `withdrawing_ar / deposit_ar` is large).
3. Submit a DAO withdrawal transaction referencing the deposit.
4. `DaoCalculator::calculate_maximum_withdraw` is called during both tx-pool admission (`transaction_fee`) and block verification (`DaoHeaderVerifier`).
5. If `withdraw_counted_capacity` overflows `u64`, the `as u64` cast silently truncates, producing a wrong capacity value that propagates into consensus state without any error.

The inconsistency is directly observable by comparing line 156 (`as u64`) against lines 204, 245, and 258 (`u64::try_from(…).map_err(…)?`) in the same source file. [6](#0-5)
