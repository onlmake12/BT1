Audit Report

## Title
`readd_detached_tx` Holds `tx_pool` Write Lock During Full CKB-VM Script Execution — (File: tx-pool/src/process.rs)

## Summary

`TxPoolService::update_tx_pool_for_reorg` acquires the `tx_pool` write lock at line 836 and holds it through the entire `readd_detached_tx` call (lines 849–851). For each detached transaction absent from the pre-fetched cache, `verify_rtx` is invoked with `command_rx = None` (line 901), causing it to execute the `block_in_place` branch in `tx-pool/src/util.rs` (lines 116–131) — full synchronous CKB-VM execution — while the write lock remains held. All concurrent readers and writers on `tx_pool` are blocked for the entire re-verification window.

## Finding Description

The write lock is acquired at line 836 and released only when the enclosing block closes at line 851:

```rust
let mut tx_pool = self.tx_pool.write().await;   // line 836
_update_tx_pool_for_reorg(&mut tx_pool, ...);
self.readd_detached_tx(&mut tx_pool, retain, fetched_cache).await;
// line 851 — lock released
```

Inside `readd_detached_tx` (lines 878–914), for every detached transaction, `verify_rtx` is called with `command_rx = None` (line 901). In `tx-pool/src/util.rs`, `verify_rtx` (lines 85–131) branches as follows: if there is a cache hit, only a cheap time-relative check runs (lines 96–100); if `command_rx` is `Some`, the async pauseable verifier runs (lines 101–115); otherwise — the case here — it falls into `block_in_place` (lines 116–131), running full synchronous `ContextualTransactionVerifier::verify` while the write lock guard is still owned by the calling async task.

The correct pattern is demonstrated in `_process_tx` (lines 705–753): `verify_rtx` is called at lines 724–732 with no lock held, and only the fast `submit_entry` call (line 753), which internally calls `with_tx_pool_write_lock` (line 103), acquires the lock. `readd_detached_tx` inverts this ordering.

The pre-fetch at line 828 (`fetch_txs_verify_cache`) only warms the cache for transactions already known to the node. An attacker who mines a private block containing transactions never broadcast to the network ensures the cache is cold for all detached transactions, forcing every iteration into the `block_in_place` branch.

## Impact Explanation

While the write lock is held during re-verification:
- `with_tx_pool_write_lock` (line 262) blocks — `submit_entry` for incoming transactions cannot proceed.
- `with_tx_pool_read_lock` (line 251) blocks — tokio's `RwLock` write lock excludes all readers; `pre_check` for new transactions stalls.
- Any other component calling `tx_pool.read().await` or `tx_pool.write().await` stalls for the full duration.

Stall duration scales as `N_detached_txs × max_tx_verify_cycles / CPU_speed`. For a block filled with max-cycle transactions (`max_block_cycles` ≈ 3.5 × 10⁹), this produces multi-second stalls per reorg event, during which the node cannot admit new transactions or update block templates. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

The attacker must be a miner capable of producing at least one competing block at the current tip height. They mine a private block containing transactions consuming `max_tx_verify_cycles` cycles each, never broadcast to the network (bypassing `txs_verify_cache`). Submitting this block triggers a 1-block reorg. Any miner with nonzero hashpower can eventually achieve this. The attack is repeatable: each reorg event re-triggers the full lock-hold window with no additional cost beyond block production.

## Recommendation

Move all `verify_rtx` calls in `readd_detached_tx` outside the `tx_pool` write lock, mirroring the `_process_tx` pattern: collect all `(rtx, status, fee, tx_size, verified)` tuples first without holding the lock, then acquire the write lock only for the fast `_submit_entry` calls. Alternatively, release and reacquire the lock between each transaction's verification, or pass a `command_rx` to allow interruption and avoid the `block_in_place` branch entirely.

## Proof of Concept

1. Attacker mines a private block containing N transactions each consuming `max_tx_verify_cycles` cycles, none previously broadcast (victim's `txs_verify_cache` is cold for all).
2. Attacker submits the block to the victim node at the same tip height, triggering a 1-block reorg.
3. Victim node calls `update_tx_pool_for_reorg` → acquires `tx_pool.write()` at line 836 → calls `readd_detached_tx` → calls `verify_rtx(..., None)` for each of the N transactions via `block_in_place` while holding the lock.
4. Concurrently issue a `submit_remote_tx` (which calls `pre_check` → `with_tx_pool_read_lock` → `tx_pool.read().await`) and measure time-to-acquire.
5. Assert that the lock hold time exceeds an acceptable threshold (e.g., > 1 second for a block with many max-cycle transactions).
