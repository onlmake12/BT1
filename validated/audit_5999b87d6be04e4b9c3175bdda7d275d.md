### Title
API Access Token Exposed in Plaintext WebSocket URL Query Parameter — (`apps/hermes/server/src/api/token.rs`, `price_service/client/js/src/PriceServiceConnection.ts`, `apps/hermes/client/rust/src/client.rs`, `apps/insights/src/hooks/pyth-pro-demo/use-datastream.ts`)

---

### Summary

The Hermes server and all its official clients transmit the API access token as a plaintext URL query parameter (`ACCESS_TOKEN=<token>`) for WebSocket connections. While the WebSocket transport itself is TLS-encrypted (WSS), the full request URI — including the token — is recorded in server-side HTTP access logs, browser history, and any intermediate proxy or CDN logs. This is the direct analog of KZG-02/KZG-03: the token is placed in a channel (the URL) that is structurally more exposed than the Authorization header, and it is accepted by the server as a fully valid credential.

---

### Finding Description

The Hermes server's `extract_token_from_headers_and_uri` function explicitly accepts the API token from the URL query string as a first-class authentication path: [1](#0-0) 

This is consumed by both the WebSocket route handler and the HTTP metrics middleware: [2](#0-1) [3](#0-2) 

All official clients append the token to the WebSocket URL before connecting:

**JavaScript client (`PriceServiceConnection`):** [4](#0-3) 

**Hermes JS client:** [5](#0-4) 

**Rust client:** [6](#0-5) 

**Production frontend (`apps/insights`)** — embeds the token directly in the URL string literal: [7](#0-6) 

The justification given in the code is that the browser WebSocket API does not support custom headers. However, the server-side design accepts the query-parameter form as fully equivalent to the `Authorization: Bearer` header, meaning any server that logs the full request URI (the standard behavior for HTTP/WebSocket upgrade requests) will record the complete token in plaintext.

---

### Impact Explanation

An attacker who gains read access to:
- Hermes server HTTP access logs
- Any reverse proxy or CDN access logs (e.g., nginx, Cloudflare)
- A victim's browser history or developer-tools network tab
- A shared network proxy in a corporate environment

can extract the full `ACCESS_TOKEN` value and reuse it indefinitely to:
1. Impersonate the legitimate token holder and consume their rate-limit quota.
2. Access any Hermes API endpoint that requires authentication.
3. Exhaust the victim's rate limits, causing their own price-pusher or integration to be rejected.

The token is long-lived (no expiry mechanism is visible in the server code), so a single log-read event yields persistent unauthorized access.

**Severity: Medium** — token leakage through logs/history; no direct price manipulation, but unauthorized API access and rate-limit exhaustion are concrete impacts.

---

### Likelihood Explanation

**Medium.** The scenario requires access to server logs or browser history, which is not trivially available to an external attacker. However:
- Server log access is a common post-exploitation step.
- Shared corporate proxies or CDN providers routinely log full request URIs.
- Browser history is accessible to any malicious extension or local attacker.
- The `apps/insights` frontend embeds the token in the URL, making it visible in the browser address bar and browser history for every user of that application.

The pattern is present across every official client library, meaning the exposure surface is the entire Hermes user base that uses WebSocket connections.

---

### Recommendation

1. **Server side**: Stop accepting `ACCESS_TOKEN` as a query parameter. Require the `Authorization: Bearer` header exclusively. For browser WebSocket clients that cannot set custom headers, use a short-lived ticket/nonce exchange: the client first calls an authenticated HTTP endpoint to obtain a one-time WebSocket upgrade token, then presents that token in the WebSocket URL. The one-time token expires after a single use or a short TTL.

2. **Client side**: Remove the `ACCESS_TOKEN` query-parameter append from `PriceServiceConnection`, `HermesClient`, and the Rust `HermesClient`. Update `use-datastream.ts` to route WebSocket connections through a server-side proxy (as is already done in `apps/developer-hub/src/app/api/playground/stream/route.ts`) so the real token never reaches the browser.

3. **Immediate mitigation**: Configure the Hermes server (and any reverse proxy in front of it) to redact or strip the `ACCESS_TOKEN` query parameter from access logs.

---

### Proof of Concept

1. A user configures a price-pusher using the official JS SDK with an access token:
   ```
   new PriceServiceConnection("wss://hermes.pyth.network", { accessToken: "SECRET_TOKEN" })
   ```
2. The SDK constructs the WebSocket URL:
   ```
   wss://hermes.pyth.network/ws?ACCESS_TOKEN=SECRET_TOKEN
   ``` [8](#0-7) 
3. The Hermes server logs the HTTP upgrade request. A standard nginx access log entry would contain:
   ```
   GET /ws?ACCESS_TOKEN=SECRET_TOKEN HTTP/1.1
   ```
4. An attacker reads the log and replays the token:
   ```bash
   curl -H "Authorization: Bearer SECRET_TOKEN" https://hermes.pyth.network/v2/updates/price/latest?ids[]=...
   ```
5. The server's `extract_token_from_headers_and_uri` accepts the token from the header and grants full API access. [9](#0-8)
