### Title
`MAX_DEPOSIT_LIMIT` for Permanent Subscriptions Is Not Enforced Cumulatively — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `Scheduler` contract enforces a `MAX_DEPOSIT_LIMIT` of 100 ETH per call in both `createSubscription()` and `addFunds()` for permanent subscriptions. However, neither function checks the cumulative `status.balanceInWei` against the limit. Because `addFunds()` is also permissionless (no access control), the limit can be trivially bypassed by calling `addFunds()` multiple times with amounts ≤ 100 ETH, permanently locking an unbounded amount of ETH in the contract.

### Finding Description

`MAX_DEPOSIT_LIMIT = 100 ether` is declared in `SchedulerConstants.sol` as a cap on how much ETH can be deposited into a permanent subscription in a single call: [1](#0-0) 

In `createSubscription()`, the check is: [2](#0-1) 

In `addFunds()`, the check is identical — only `msg.value` is compared, not the post-deposit cumulative balance: [3](#0-2) 

After the check, `status.balanceInWei += msg.value` unconditionally accumulates the deposit. There is no guard of the form `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`. Permanent subscriptions explicitly block all withdrawals: [4](#0-3) 

Additionally, `addFunds()` has no `onlyManager` modifier — the test suite itself confirms that **anyone** can call it: [5](#0-4) 

### Impact Explanation

Permanent subscriptions are designed to be irrevocable: funds deposited can never be withdrawn, the subscription can never be deactivated, and parameters can never be changed. The `MAX_DEPOSIT_LIMIT` is the only safeguard preventing an unbounded amount of ETH from being permanently locked in a single subscription. By bypassing it via repeated `addFunds()` calls, an attacker (or the manager themselves) can lock an arbitrary amount of ETH permanently in the contract, far exceeding the intended 100 ETH cap. Since withdrawal is permanently blocked, this constitutes an irreversible loss of funds.

### Likelihood Explanation

The bypass requires no privileges, no special knowledge, and no complex setup. `addFunds()` is a public, payable function with no access control. Any externally-owned account can call it on any active permanent subscription. The bypass is a single repeated call pattern, identical to the sNote M-04 pattern.

### Recommendation

Replace the per-call `msg.value` check with a cumulative post-deposit balance check in both `createSubscription()` and `addFunds()`:

```solidity
// In addFunds():
if (params.isPermanent && (status.balanceInWei + msg.value) > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Apply the same fix in `createSubscription()` (though there `status.balanceInWei` is zero at that point, so the existing check is sufficient for creation — the critical gap is in `addFunds()`).

### Proof of Concept

1. Deploy `Scheduler` and create a permanent subscription with exactly `MAX_DEPOSIT_LIMIT` (100 ETH). This passes the creation check.
2. Call `addFunds{value: 100 ether}(subscriptionId)` — passes because `msg.value (100 ETH) <= MAX_DEPOSIT_LIMIT (100 ETH)`.
3. Repeat step 2 N times. After N iterations, `status.balanceInWei = (N+1) * 100 ETH`, all permanently locked with no withdrawal path.
4. The `MAX_DEPOSIT_LIMIT` is completely bypassed.
