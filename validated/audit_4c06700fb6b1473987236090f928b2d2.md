Audit Report

## Title
Unbounded Wait Loop in `read_from_json` Causes Permanent CLI Hang When Importing a Block with `number = u64::MAX` — (`util/instrument/src/import.rs`)

## Summary

`Import::read_from_json` unconditionally updates `largest_block_number` to the maximum block number seen from the channel, then enters a boundless polling loop waiting for `snapshot.get_block_hash(largest_block_number)` to return `Some`. A crafted JSONL file containing a block whose `number` field is `0xffffffffffffffff` sets `largest_block_number` to `u64::MAX`. Because that block's parent cannot exist in the chain, it is permanently parked in the orphan pool (or its callback merely prints an error on verification failure), and `get_block_hash(u64::MAX)` returns `None` forever — the process never exits.

## Finding Description

**Root cause — `largest_block_number` is set before dispatch, with no failure path to reset it.** [1](#0-0) 

For every non-genesis block dequeued from the channel, `largest_block_number` is updated unconditionally at L192 — before the block is handed to the chain service. There is no mechanism to roll this value back if the block is later rejected.

**Block is dispatched fire-and-forget.** [2](#0-1) 

`asynchronous_process_lonely_block` returns immediately; the caller receives no success/failure signal through the return value.

**The verify callback does not unblock the wait loop.** [3](#0-2) 

On verification failure the callback only prints an error to stderr. It does not write to any shared variable that the wait loop checks.

**Chain service: two failure paths, neither satisfies the wait condition.**

*Path A — non-contextual verification fails (without `--skip-all-verify`):* [4](#0-3) 
The callback is invoked with `Err`, the block is never inserted into `COLUMN_INDEX`, and `get_block_hash(u64::MAX)` remains `None`.

*Path B — verification skipped or passes, block inserted into DB, then orphaned:* [5](#0-4) [6](#0-5) 
Because the parent of a `u64::MAX`-numbered block is not stored and not pending-verify, the block is placed in the orphan pool (`orphan_blocks_broker.insert`) and its callback is never invoked with a committed result. `COLUMN_INDEX` is never written for height `u64::MAX`.

**The wait loop has no exit condition for failure.** [7](#0-6) 

`get_block_hash` is a plain `COLUMN_INDEX` lookup: [8](#0-7) 

No block at height `u64::MAX` is ever written to `COLUMN_INDEX`, so the lookup returns `None` on every iteration and the loop spins at 1-second intervals indefinitely.

**Orphan cleanup does not help.** [9](#0-8) [10](#0-9) 

Even after the orphan is evicted by `clean_expired_orphans`, `get_block_hash(u64::MAX)` still returns `None` — the wait condition is never satisfied.

## Impact Explanation

The `ckb import` CLI process hangs permanently, consuming a thread sleeping in a 1-second loop and holding all chain-service threads open. The only recovery is `SIGKILL`. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local command line crash.**

## Likelihood Explanation

The attack requires only the ability to supply a crafted JSONL file to `ckb import`. With `--skip-all-verify`, no PoW solution is needed. Even without that flag, `BlockVerifier` does not check block number directly: [11](#0-10) 
A structurally valid block with `number = u64::MAX` (cellbase `since` field set to `u64::MAX`, valid merkle root) passes non-contextual checks and still ends up in the orphan pool, triggering the hang. The file can be as small as two lines (genesis + the malicious block). No cryptographic material, network access, or elevated privileges are required.

## Recommendation

Replace the unconditional `largest_block_number` update with tracking only blocks that are successfully committed, **or** add a deadline/timeout to the wait loop, **or** feed a shared atomic/channel from the verify callback so the loop can exit on permanent failure. A minimal fix:

```rust
// Option 1: add a timeout
let deadline = std::time::Instant::now() + std::time::Duration::from_secs(MAX_WAIT_SECS);
while self.shared.snapshot().get_block_hash(largest_block_number).is_none() {
    if std::time::Instant::now() > deadline {
        return Err(...);
    }
    std::thread::sleep(std::time::Duration::from_secs(1));
}

// Option 2: track largest_block_number only on successful commit via callback
```
