### Title
`RecentReject` Counter Never Incremented — `shrink()` Guard Never Triggers, Enabling Unbounded Disk Growth - (File: `tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject::put()` computes `self.total_keys_num.checked_add(1)` into a **local variable** but never writes it back to `self.total_keys_num`. The field stays frozen at its startup estimate forever. Because the `count_limit` guard compares against this stale value, `shrink()` is never called, and the rejected-transaction RocksDB database grows without bound. Any unprivileged RPC caller who can cause repeated transaction rejections can exhaust node disk space.

---

### Finding Description

`RecentReject` is a sharded RocksDB-with-TTL store that records recently rejected transactions. It is designed to cap its size at `count_limit` entries by calling `shrink()` (which drops and recreates a random shard) whenever the count exceeds the limit.

The counter `total_keys_num` is initialized once at startup from `estimate_num_keys_cf`: [1](#0-0) 

Every subsequent call to `put()` is supposed to increment this counter and trigger `shrink()` when the limit is exceeded. The actual code is: [2](#0-1) 

The critical flaw is on line 62:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
}
```

`checked_add(1)` returns a new value bound to the **local** `total_keys_num`. `self.total_keys_num` is **never assigned** the result. After every `put()` call, `self.total_keys_num` remains at whatever value it had at startup (typically 0 for a fresh node). The comparison `total_keys_num > self.count_limit` therefore always evaluates against `initial_estimate + 1`, which is almost always below `count_limit`. `shrink()` is never invoked.

The `shrink()` method itself correctly re-estimates and updates `self.total_keys_num`: [3](#0-2) 

But it is unreachable through the normal `put()` path because the guard condition is never satisfied.

The existing unit test passes trivially for the wrong reason — it uses `ttl = -1` (immediate expiration), so the DB estimate stays near 0 regardless, masking the bug: [4](#0-3) 

---

### Impact Explanation

**Impact: Medium.**

The `RecentReject` database grows without bound on disk. A node operator who has configured `keep_rejected_tx_hashes_count` (the `count_limit`) to cap disk usage will find that cap is never enforced. Over time, or under deliberate attack, this exhausts disk space, which can crash the node or prevent it from writing new blocks to the chain store.

---

### Likelihood Explanation

**Likelihood: Medium.**

Any unprivileged RPC caller or P2P peer can submit transactions that will be rejected and recorded. The `put_recent_reject` path is reached whenever `reject.should_recorded()` is true: [5](#0-4) [6](#0-5) 

An attacker needs only to submit a sustained stream of transactions that are rejected with a recordable rejection type (e.g., script verification failures, capacity errors). No special privilege is required. The RocksDB TTL provides a natural rate of expiry, but because `shrink()` is never called, entries accumulate faster than they expire under any meaningful attack rate.

---

### Recommendation

Assign the incremented value back to `self.total_keys_num` before the comparison:

```diff
- if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
-     if total_keys_num > self.count_limit {
-         self.shrink()?;
-     }
- } else {
-     self.shrink()?;
- }
+ match self.total_keys_num.checked_add(1) {
+     Some(n) => {
+         self.total_keys_num = n;
+         if self.total_keys_num > self.count_limit {
+             self.shrink()?;
+         }
+     }
+     None => {
+         self.shrink()?;
+     }
+ }
```

Additionally, update the unit test to use a positive TTL so that entries actually persist and the shrink logic can be meaningfully exercised.

### Proof of Concept

1. Start a CKB node with `keep_rejected_tx_hashes_count = 100` and `recent_reject` path configured.
2. Submit 10 000 transactions via `send_transaction` RPC that fail script verification (e.g., always-fail lock script). Each rejection calls `recent_reject.put()`.
3. Observe that `get_estimate_total_keys_num()` (exposed via the terminal RPC `total_recent_reject_num`) returns a value near 0 throughout, because `self.total_keys_num` is never incremented.
4. Observe that the on-disk size of the `recent_reject` RocksDB directory grows proportionally to the number of submissions, with no shrinking occurring.

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;

use ckb_types::{core::tx_pool::Reject, packed::Byte32};

use crate::component::recent_reject::RecentReject;

#[test]
fn test_basic() {
    let tmp_dir = tempfile::Builder::new().tempdir().unwrap();
    let shard_num = 2;
    let limit = 100;
    let ttl = -1;

    let mut recent_reject = RecentReject::build(tmp_dir.path(), shard_num, limit, ttl).unwrap();

    for i in 0..80u64 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        recent_reject
            .put(&key, Reject::Malformed(i.to_string(), Default::default()))
            .unwrap();
    }

    for i in 0..80u64 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        let reject: ckb_jsonrpc_types::PoolTransactionReject =
            Reject::Malformed(i.to_string(), Default::default()).into();
        assert_eq!(
            recent_reject.get(&key).unwrap().unwrap(),
            serde_json::to_string(&reject).unwrap()
        )
    }

        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        recent_reject
            .put(&key, Reject::Malformed(i.to_string(), Default::default()))
            .unwrap();
    }

    assert!(recent_reject.total_keys_num < 100);
}
