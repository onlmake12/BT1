Audit Report

## Title
`total_tx_size` and `total_tx_cycles` Inflated When Entries Are Evicted During `add_entry` - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry()`, `updated_stat_for_add_tx()` computes a pre-eviction snapshot of `total_tx_size`/`total_tx_cycles` into local variables before `check_and_record_ancestors()` runs. If `check_and_record_ancestors()` evicts entries via `remove_entry_and_descendants()` → `update_stat_for_remove_tx()`, those in-place decrements to `self.total_tx_size`/`self.total_tx_cycles` are immediately overwritten by the stale snapshot at lines 218–219. An attacker can repeat this to inflate `total_tx_size` past `max_tx_pool_size`, causing `limit_size()` to evict honest transactions and all new submissions to receive `Reject::Full`.

## Finding Description
The code in `add_entry()` at lines 210–219 of `tx-pool/src/component/pool_map.rs`:

1. **Lines 210–211**: `updated_stat_for_add_tx(entry.size, entry.cycles)` reads `self.total_tx_size` and `self.total_tx_cycles` and returns `(old_total_size + entry.size, old_total_cycles + entry.cycles)` into local variables `total_tx_size` and `total_tx_cycles`.

2. **Line 213**: `check_and_record_ancestors()` enters the eviction branch (lines 603–625) when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants()` in a loop. `remove_entry_and_descendants()` calls `remove_entry()` (line 263), which calls `update_stat_for_remove_tx()` at line 247, **mutating `self.total_tx_size` and `self.total_tx_cycles` in place**.

3. **Lines 218–219**: `self.total_tx_size = total_tx_size` and `self.total_tx_cycles = total_tx_cycles` **unconditionally overwrite** the already-decremented live fields with the stale pre-eviction snapshot, erasing all subtractions performed in step 2.

The call chain is confirmed:
- `check_and_record_ancestors` → `remove_entry_and_descendants` (line 618) → `remove_entry` (line 263) → `update_stat_for_remove_tx` (line 247), which writes directly to `self.total_tx_size`/`self.total_tx_cycles` (lines 739–740).
- Those writes are then unconditionally clobbered at lines 218–219.

No guard exists between lines 213 and 218–219 to account for evictions that occurred during `check_and_record_ancestors()`.

## Impact Explanation
`total_tx_size` is the sole enforcement variable in `limit_size()` at `tx-pool/src/pool.rs` line 298: `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. When inflated past `max_tx_pool_size` (default 180 MB), `limit_size()` begins evicting honest pending/proposed transactions. Simultaneously, `updated_stat_for_add_tx()` uses the inflated value to gate new admissions, causing premature `Reject::Full` for all new submissions even when real pool occupancy is well below the limit. The attack is repeatable without bound, constituting a tx-pool denial-of-service. This matches the allowed **High** impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
The eviction path is reachable by any unprivileged RPC/relay transaction submitter. The attacker needs only to submit transactions `D1…Dk` referencing a cell `C` as a `cell_dep` (recorded in `edges.deps[C]` as `cell_ref_parents`), then submit a new transaction `T` that consumes `C` as an input and has an ancestor chain exceeding `max_ancestors_count` (default 125) in the pool. Both conditions are achievable with valid, fee-paying transactions. The attack is repeatable across multiple submissions, progressively inflating the counters without bound. No special privileges, leaked keys, or victim mistakes are required.

## Recommendation
Move `updated_stat_for_add_tx()` to **after** `check_and_record_ancestors()` returns, so it reads the already-eviction-adjusted `self.total_tx_size`/`self.total_tx_cycles`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, call `recompute_total_stat()` after all mutations are complete, as already used in the underflow recovery path at lines 743–749.

## Proof of Concept
```
Initial pool state:
  total_tx_size   = 1_000_000
  total_tx_cycles = 500_000
  Pool contains D1 (size=200_000, cycles=100_000) referencing cell C as dep.

Attacker submits T (size=50_000, cycles=25_000):
  - T consumes cell C as input  →  D1 is a cell_ref_parent
  - T has 126 ancestors in pool  →  ancestors_count (127) > max_ancestors_count (125)
  - 127 − 1 (D1) = 126 > 125, so D1 is evicted

Step-by-step in add_entry():
  1. updated_stat_for_add_tx(50_000, 25_000):
       total_tx_size_snapshot   = 1_000_000 + 50_000 = 1_050_000
       total_tx_cycles_snapshot = 500_000 + 25_000   = 525_000

  2. check_and_record_ancestors() evicts D1 via update_stat_for_remove_tx():
       self.total_tx_size   = 1_000_000 − 200_000 = 800_000
       self.total_tx_cycles = 500_000 − 100_000   = 400_000

  3. self.total_tx_size   = 1_050_000  ← snapshot overwrites 800_000
     self.total_tx_cycles = 525_000    ← snapshot overwrites 400_000

Correct values:  total_tx_size=850_000, total_tx_cycles=425_000
Actual values:   total_tx_size=1_050_000, total_tx_cycles=525_000
Inflation:       +200_000 bytes, +100_000 cycles
```

A unit test can be written against `PoolMap::add_entry()` directly: populate the pool with a `cell_dep` transaction D1, submit a new transaction T triggering the eviction branch, and assert that `pool_map.total_tx_size` equals the sum of all remaining entries' sizes (i.e., does not include D1's size). Repeating the attack inflates `total_tx_size` until it exceeds `max_tx_pool_size`, at which point `limit_size()` evicts honest transactions and all new submissions receive `Reject::Full`.
