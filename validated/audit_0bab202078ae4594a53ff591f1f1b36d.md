### Title
Missing Excess ETH Refund in `requestPriceUpdatesWithCallback` Silently Transfers Overpayment to Provider — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `requestPriceUpdatesWithCallback` enforces only a lower-bound fee check (`msg.value < requiredFee → revert`) but performs no upper-bound check and issues no refund. Any ETH above `requiredFee` is silently absorbed: the `pythFeeInWei` portion accrues to the protocol, and the entire remainder is stored in `req.fee` and later credited to the provider's `accruedFeesInWei`. The interface specification (`IEcho.sol`) explicitly states `msg.value must be equal to getFee(...)`, but the implementation accepts any amount ≥ fee without refunding the excess.

---

### Finding Description

In `requestPriceUpdatesWithCallback`:

```solidity
// Only a lower-bound check — no upper-bound, no refund
if (msg.value < requiredFee) revert InsufficientFee();
...
// Entire excess above pythFeeInWei is stored as provider fee
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [1](#0-0) 

The `req.fee` field captures `msg.value - pythFeeInWei`, meaning any overpayment above `pythFeeInWei` is stored as the provider's fee. When `executeCallback` is later called, this full amount is credited to the provider:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The interface contract explicitly documents the invariant that is violated:

```solidity
/// @dev The msg.value must be equal to getFee(callbackGasLimit)
``` [3](#0-2) 

This is the direct analog to the reported vulnerability class: the specification (interface) dictates one behavior (`msg.value == fee`), but the implementation silently accepts and absorbs more, with no refund path.

---

### Impact Explanation

Any ETH sent above `getFee(provider, callbackGasLimit, priceIds)` is permanently transferred to the provider's `accruedFeesInWei` balance. The requester has no mechanism to recover the excess. This constitutes a direct, irreversible loss of user funds for any caller who overpays — whether by accident, by using a stale fee estimate, or because the provider lowered their fee between the user's `getFee()` call and transaction inclusion. [1](#0-0) 

---

### Likelihood Explanation

Medium. Callers commonly add a small ETH buffer to avoid reverts from fee estimation races. Any such buffer is silently lost. Additionally, if a provider reduces their fee between a user's `getFee()` call and transaction mining, the user overpays and the excess is irrecoverably credited to the provider. No privileged access is required; any unprivileged requester is affected. [4](#0-3) 

---

### Recommendation

Add an excess-refund path after the fee check, consistent with the pattern already used in `PythLazer.verifyUpdate`:

```solidity
if (msg.value > requiredFee) {
    payable(msg.sender).transfer(msg.value - requiredFee);
}
``` [5](#0-4) 

Alternatively, enforce strict equality: `if (msg.value != requiredFee) revert IncorrectFee();`, which matches the `IEcho` specification.

---

### Proof of Concept

1. Provider registers with `baseFeeInWei = 1000 wei`, `feePerFeedInWei = 100 wei`, `feePerGasInWei = 1 wei`.
2. User calls `getFee(provider, 100_000, priceIds)` → returns `requiredFee = X`.
3. User calls `requestPriceUpdatesWithCallback{value: X + 1 ether}(...)`.
4. Check passes (`msg.value >= requiredFee`).
5. `req.fee = (X + 1 ether) - pythFeeInWei` — the full 1 ETH overpayment is stored as provider fee.
6. Provider calls `executeCallback`; provider's `accruedFeesInWei` increases by `req.fee + msg.value - pythFee`, which includes the user's 1 ETH overpayment.
7. Provider calls `withdrawAsFeeManager` and withdraws the excess. User has no recourse. [1](#0-0) [2](#0-1)
