Audit Report

## Title
`WeightUnitsFlow` Fee Estimator Permanently Inflates Historical Flow Data via No-Op `reject_tx` — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary

The `WeightUnitsFlow` fee estimator records every accepted transaction's weight in `self.txs` (keyed by block number) but never removes entries when transactions are evicted, because `reject_tx` is explicitly a no-op for this variant. An attacker can repeatedly submit and RBF-evict high-fee-rate transactions, accumulating duplicate weight entries in `self.txs` and inflating `flow_speed_buckets` for up to 256 blocks, causing `estimate_fee_rate` to return artificially elevated fee rate recommendations.

## Finding Description

**Root cause — `reject_tx` is a no-op:**

`util/fee-estimator/src/estimator/mod.rs` lines 84–88 explicitly skips `WeightUnitsFlow` in the reject dispatcher: [1](#0-0) 

**No tx_hash identity in storage — deduplication is structurally impossible:**

`TxStatus` (lines 76–80 of `weight_units_flow.rs`) stores only `weight` and `fee_rate`. `accept_tx` (lines 153–162) pushes a new `TxStatus` into `self.txs: HashMap<BlockNumber, Vec<TxStatus>>` on every accepted transaction with no identity field. Even if `reject_tx` were wired up, there is no key to remove a specific entry. [2](#0-1) [3](#0-2) 

**All eviction paths reach the no-op:**

- Pool-size eviction: `tx-pool/src/pool.rs` line 323 (`limit_size` → `callbacks.call_reject`)
- Expiry eviction: `tx-pool/src/pool.rs` line 286 (`remove_expired` → `callbacks.call_reject`)
- RBF eviction: `resolve_conflict` → `callbacks.call_reject` for displaced entries

The registered reject callback at `shared/src/shared_builder.rs` line 600 calls `fee_estimator.reject_tx(&tx_hash)`, which is a no-op for `WeightUnitsFlow`. [4](#0-3) [5](#0-4) [6](#0-5) 

**Inflation propagates to fee estimate:**

`sorted_flowed` (lines 303–313) collects all `TxStatus` entries from `self.txs` over the historical window without deduplication. `flow_speed_buckets` divides total accumulated weight by `historical_blocks` (lines 268–271). The `passed` check at lines 279–297 uses:

```rust
let added_weight = flow_speed_buckets[bucket_index] * target_blocks;
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
let passed = current_weight + added_weight <= removed_weight;
```

Inflating `flow_speed_buckets` for high-fee-rate buckets causes low-fee-rate buckets to fail `passed`, pushing the returned fee rate upward. [7](#0-6) [8](#0-7) [9](#0-8) 

**`expire()` is insufficient:**

`expire()` (lines 147–151) only removes entries older than 256 blocks. It does not correct for duplicate entries within the window and imposes no cap on weight contributions per block. [10](#0-9) 

## Impact Explanation

Users and wallets relying on `estimate_fee_rate` (publicly accessible RPC at `rpc/src/module/experiment.rs` lines 301–315) receive systematically inflated fee rate recommendations, causing them to overpay miners. [11](#0-10) 

The attack can be renewed continuously, sustaining the inflation indefinitely. This constitutes economic harm to CKB users, matching the allowed impact class: **Vulnerabilities which could easily damage CKB economy**. Severity is bounded by the fact that `WeightUnitsFlow` must be explicitly configured (the default is `Dummy` per `shared/src/shared_builder.rs` lines 406–415) and the fee estimator is advisory, but nodes that enable it are fully exposed with no mitigation. [12](#0-11) 

## Likelihood Explanation

The attack requires only the ability to submit valid transactions — an unprivileged capability available to any RPC caller or P2P peer. The attacker pays only the marginal RBF fee increment per cycle while multiplying the recorded flow weight by an arbitrary factor. The attack is repeatable every block and requires no special knowledge beyond the node's RBF configuration. `ConfirmationFraction` is not affected because it correctly implements `reject_tx` with per-tx-hash tracking. [1](#0-0) 

## Recommendation

Implement a `reject_tx` handler for `WeightUnitsFlow`. Because `self.txs` currently stores `Vec<TxStatus>` without tx_hash identity, the simplest fix is to change the storage structure to `HashMap<BlockNumber, HashMap<Byte32, TxStatus>>` (keyed by tx_hash within each block), mirroring how `ConfirmationFraction` tracks individual transactions. This allows `reject_tx` to remove the specific entry by hash. Alternatively, cap the maximum total weight contribution per block in `self.txs` to bound the inflation any single attacker can cause, though this is a weaker mitigation.

## Proof of Concept

```
Prerequisites: Node configured with WeightUnitsFlow fee estimator, RBF enabled.

1. Attacker calls send_transaction(tx_A, inputs=[X], fee_rate=HIGH)
   → Accepted: accept_tx records TxStatus{weight=W, fee_rate=HIGH} in self.txs[tip]

2. Attacker calls send_transaction(tx_A_rbf, inputs=[X], fee_rate=HIGH+1)
   → tx_A evicted via RBF: reject_tx called → no-op for WeightUnitsFlow
   → tx_A_rbf accepted: accept_tx records TxStatus{weight=W, fee_rate=HIGH+1} in self.txs[tip]
   → self.txs[tip] now contains 2 entries with weight W

3. Repeat step 2 N times.
   → self.txs[tip] contains N+1 entries, total weight (N+1)*W for high fee-rate buckets.

4. flow_speed_buckets[high_bucket] = (N+1)*W / historical_blocks
   → added_weight for high bucket >> removed_weight
   → Low fee-rate buckets fail the `passed` check
   → estimate_fee_rate returns an elevated fee rate

5. Effect persists for 256 blocks (~2 hours). Repeat from step 1 to sustain.

Verification: Unit test comparing estimate_fee_rate output before and after N RBF cycles
on a mock Algorithm instance with controlled self.txs population.
```
