### Title
User-Controlled `callbackGasLimit = 0` Bypasses Provider Gas Fee in Echo Contract - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the `getFee()` function computes the provider's gas-proportional fee as `callbackGasLimit * feePerGasInWei`. Because `callbackGasLimit` is entirely user-supplied with no minimum enforced, any unprivileged caller can pass `callbackGasLimit = 0` to eliminate the gas-proportional fee component entirely, paying only base fees while still creating a valid on-chain request that the provider must service.

---

### Finding Description

`getFee()` computes the total fee as:

```solidity
uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei;
uint256 gasFee = callbackGasLimit * providerFeeInWei;
feeAmount = baseFee + providerBaseFee + providerFeedFee + SafeCast.toUint96(gasFee);
``` [1](#0-0) 

`callbackGasLimit` is passed directly by the caller to `requestPriceUpdatesWithCallback` with no lower-bound check:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [2](#0-1) 

When `callbackGasLimit = 0`, `gasFee = 0`, so the user pays only `pythFeeInWei + providerBaseFee + providerFeedFee`. The stored fee is:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [3](#0-2) 

This means `req.fee` contains zero gas-proportional compensation for the provider. The provider's `feePerGasInWei` field exists precisely to cover the gas cost of executing the callback: [4](#0-3) 

When `executeCallback` is later called, the callback is invoked with the stored gas limit:

```solidity
IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(sequenceNumber, priceFeeds)
``` [5](#0-4) 

With `callbackGasLimit = 0`, the callback always fails (out of gas), yet the provider still pays gas for the outer `executeCallback` transaction and receives no gas-proportional compensation. The test suite explicitly validates that `callbackGasLimit = 0` is accepted and produces a fee with zero gas component: [6](#0-5) 

The comment in `IEcho.sol` even acknowledges that `tx.gasprice` was originally intended to be used for fee computation, confirming the design intent was gas-proportional fees: [7](#0-6) 

---

### Impact Explanation

A user can submit a valid `requestPriceUpdatesWithCallback` with `callbackGasLimit = 0`, paying zero gas-proportional fee. The provider:
1. Receives no gas-fee compensation (`feePerGasInWei * 0 = 0`)
2. Must still call `executeCallback`, paying gas for the outer function
3. Has the callback always fail (0 gas forwarded)

This makes the provider unprofitable on every such request. A malicious user can spam zero-gas-limit requests to grief the provider into abandoning the service or operating at a loss, effectively breaking the fee model that `feePerGasInWei` was designed to enforce.

---

### Likelihood Explanation

Any unprivileged user can call `requestPriceUpdatesWithCallback` with `callbackGasLimit = 0`. No special role, key, or governance access is required. The only cost to the attacker is the base fee (`pythFeeInWei + providerBaseFee + providerFeedFee`), which can be minimal. The attack is trivially repeatable.

---

### Recommendation

- **Short term:** Enforce a minimum `callbackGasLimit` in `requestPriceUpdatesWithCallback`, either as a protocol-level constant or as a per-provider configured minimum (e.g., stored in `ProviderInfo`). Revert if `callbackGasLimit < minCallbackGasLimit`.
- **Long term:** Decouple the gas-proportional fee from the user-supplied `callbackGasLimit` alone. Consider using the provider's `defaultGasLimit` as a floor, or charge a fixed minimum gas fee regardless of the requested limit, so that providers are always compensated for at least the baseline cost of executing a callback.

---

### Proof of Concept

1. Provider registers with `feePerGasInWei = 1000` (1000 wei per gas unit), `baseFeeInWei = 100`, `feePerFeedInWei = 50`.
2. User calls `requestPriceUpdatesWithCallback(provider, block.timestamp, priceIds, 0)` with `callbackGasLimit = 0`.
3. `getFee()` returns `pythFeeInWei + 100 + 50*N + 0` — zero gas component.
4. User pays only base fees; `req.fee = providerBaseFee + providerFeedFee` (no gas compensation).
5. Provider calls `executeCallback`, paying gas for the outer function (~50k–100k gas at prevailing gas price).
6. `_echoCallback{gas: 0}(...)` is invoked and immediately fails out of gas.
7. Provider's `accruedFeesInWei` increases by only `req.fee - pythFee_for_update`, with zero gas-proportional component.
8. Provider is net-negative on every such request. Repeating this attack at scale makes the provider unprofitable.
