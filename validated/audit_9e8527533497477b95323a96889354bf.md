### Title
Entropy Provider Fees Credited at Request Time With No Fulfillment Enforcement — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Pyth Entropy contract credits a provider's fee balance at **request time**, not at **fulfillment time**. There is no on-chain timeout, refund, or penalty mechanism if a provider never calls `revealWithCallback`. A provider can collect fees from many users and then simply never reveal, keeping all fees while users receive no random number.

### Finding Description

In `requestHelper`, the provider's accrued fee balance is incremented unconditionally as soon as a user submits a request:

```solidity
// Entropy.sol line 236-237
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
``` [1](#0-0) 

The provider can then call `withdraw` at any time to extract those fees:

```solidity
// Entropy.sol line 150-163
function withdraw(uint128 amount) public override {
    ...
    providerInfo.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [2](#0-1) 

Fulfillment requires the provider to call `revealWithCallback` with their secret `providerContribution`. Because this value is only known to the provider (it is the pre-image in their committed hash chain), **no third party can fulfill on the provider's behalf**:

```solidity
// Entropy.sol line 542-566
function revealWithCallback(
    address provider,
    uint64 sequenceNumber,
    bytes32 userContribution,
    bytes32 providerContribution   // <-- only provider knows this
) public override { ... }
``` [3](#0-2) 

The contract contains no timeout, no refund path, and no slashing mechanism. The fees documentation explicitly states: *"There are no fees for revealing the random numbers"* — confirming the entire fee is collected at request time with zero on-chain obligation to fulfill. [4](#0-3) 

### Impact Explanation

A malicious or negligent provider can:
1. Register as a provider with a valid hash chain commitment.
2. Accept arbitrarily many `requestV2` / `requestWithCallback` calls, collecting `providerFee` per request.
3. Call `withdraw` to extract all accrued fees.
4. Never call `revealWithCallback`, leaving every user's request permanently stuck.

Users lose their paid fees and never receive a random number. Any application (e.g., NFT mint, lottery, game) that depends on the callback is permanently blocked with no on-chain recourse.

### Likelihood Explanation

Any address can permissionlessly register as a provider via `register()`. A malicious actor can set a competitive fee, attract users, collect fees, and exit. The only deterrent is off-chain reputation, which is insufficient for a permissionless on-chain protocol. The Fortuna service (Pyth's own provider) is honest, but the contract makes no distinction between the default provider and any other registered provider. [5](#0-4) 

### Recommendation

Decouple fee disbursement from request time. Options include:

1. **Escrow fees until fulfillment:** Hold `providerFee` in escrow per request and only release it to `providerInfo.accruedFeesInWei` upon a successful `revealWithCallback`. If a timeout passes without fulfillment, refund the user.
2. **Add a timeout-based refund:** Allow users to reclaim their fee after `N` blocks if the request remains unfulfilled (i.e., `req.callbackStatus == CALLBACK_NOT_STARTED` after a deadline).
3. **Require a provider bond:** Providers must stake collateral that can be slashed if they fail to fulfill within a time window.

### Proof of Concept

1. Attacker calls `register(highFee, commitment, ..., chainLength=1000, uri)` to register as a provider.
2. Users call `requestV2(attackerProvider, ...)` paying `highFee + pythFee` each. At each call, `providerInfo.accruedFeesInWei += highFee` is executed immediately. [6](#0-5) 
3. Attacker calls `withdraw(providerInfo.accruedFeesInWei)` to drain all collected fees. [7](#0-6) 
4. Attacker never calls `revealWithCallback`. All user requests remain in state `CALLBACK_NOT_STARTED` indefinitely. No on-chain path exists to refund users or force fulfillment. [8](#0-7)

    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }

    function revealWithCallback(
        address provider,
        uint64 sequenceNumber,
        bytes32 userContribution,
        bytes32 providerContribution
    ) public override {
        EntropyStructsV2.Request storage req = findActiveRequest(
            provider,
            sequenceNumber
        );

        if (
            !(req.callbackStatus ==
                EntropyStatusConstants.CALLBACK_NOT_STARTED ||
                req.callbackStatus == EntropyStatusConstants.CALLBACK_FAILED)
        ) {
            revert EntropyErrors.InvalidRevealCall();
        }

        bytes32 randomNumber;
        (randomNumber, ) = revealHelper(
            req,
            userContribution,
            providerContribution
        );
```
