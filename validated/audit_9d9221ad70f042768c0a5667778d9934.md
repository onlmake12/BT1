Audit Report

## Title
Unbounded `pending_delivered` HashMap Growth via Spoofed `from_peer_id` in `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `HolePunching` protocol maintains a `pending_delivered: HashMap<PeerId, PendingDeliveredInfo>` with no size cap. An attacker with a standard P2P connection can send `ConnectionRequest` messages addressed to the victim node (`to = victim_peer_id`) with a fresh, unique `from_peer_id` per message. Every such message that passes validation inserts a new entry into the map. The only eviction is a periodic `retain` that fires every 5 minutes, allowing an attacker to accumulate thousands of entries per connection window and exhaust heap memory on the victim node.

## Finding Description
`pending_delivered` is declared as an unbounded `HashMap<PeerId, PendingDeliveredInfo>` where `PendingDeliveredInfo = (Vec<Multiaddr>, u64)`. [1](#0-0) 

The deduplication guard in `respond_delivered` only rejects a `from_peer_id` that was **already seen** in the map within `HOLE_PUNCHING_INTERVAL`. With a unique `from_peer_id` per message, this check always misses and execution continues. [2](#0-1) 

After a successful `send_message_to` back to the attacker's session, the new entry is unconditionally inserted: [3](#0-2) 

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`. Because each message carries a distinct `from`, every message gets its own fresh bucket and the limiter never fires: [4](#0-3) 

The outer `rate_limiter` is keyed by `(session_id, msg.item_id())` and allows 30 req/s per session — it throttles insertion rate but does not bound total map size: [5](#0-4) 

Cleanup only runs every `CHECK_INTERVAL = 5 minutes` and only removes entries older than `TIMEOUT = 5 minutes`, meaning the map grows freely between ticks: [6](#0-5) 

The only prerequisite for insertion is that `remote_listens` contains at least one valid TCP/IPv4 or IPv6 address — trivially satisfied by any routable address: [7](#0-6) 

## Impact Explanation
Each inserted entry costs approximately `PeerId` (~39 B) + `Vec<Multiaddr>` (up to 24 × ~50 B = 1,200 B) + `u64` (8 B) + HashMap overhead (~50 B) ≈ ~1,300 B. At 30 req/s over a 5-minute window, a single attacker session accumulates ~9,000 entries (~11.7 MB). With a typical inbound connection limit of 125 peers, a coordinated attacker can grow the map to ~1.46 GB of heap in a single cleanup interval, causing an OOM crash on resource-constrained nodes. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
- HolePunching is a production feature with no opt-in flag.
- The attacker requires only a standard inbound P2P connection — no authentication, no PoW, no privileged role.
- The attacker must supply at least one valid TCP/IPv4 or IPv6 listen address, which is trivially satisfied.
- The attacker keeps the session alive to ensure `send_message_to` succeeds, which is entirely within attacker control.
- The attack is repeatable across any number of simultaneous connections up to the node's inbound limit.

## Recommendation
1. **Add a hard size cap** on `pending_delivered` (e.g., 1,024 entries). Reject new insertions or evict LRU entries when the cap is reached.
2. **Re-key `forward_rate_limiter` on `(session_id, to)` or `(session_id, msg_item_id)`** instead of `(from, to, msg_item_id)`, so rotating `from_peer_id` cannot bypass it.
3. **Validate that `content.from` matches the authenticated peer ID of the sending session**, preventing spoofed `from` values from being used as map keys entirely.

## Proof of Concept
```rust
// Pseudocode — run against a live victim node
let victim_peer_id = /* victim's known PeerId */;
let session = connect_to_victim(); // standard P2P connection

for i in 0..9_000u64 {
    let unique_from = PeerId::random(); // new key each iteration
    let msg = ConnectionRequest {
        from: unique_from,
        to: victim_peer_id.clone(),
        listen_addrs: vec!["/ip4/1.2.3.4/tcp/1234".parse().unwrap()],
        route: vec![],
        max_hops: MAX_HOPS,
    };
    session.send(msg); // outer rate limiter allows 30/s
    sleep(Duration::from_millis(34)); // stay within 30 req/s
}
// After 5 minutes: pending_delivered.len() ≈ 9,000 per session
// No cleanup until notify() fires; repeat across N sessions for N × 9,000 entries
```

Each iteration inserts a new entry because `from_peer_id` is unique (bypassing the dedup check at L161–167) and the `forward_rate_limiter` key `(from, to, item_id)` is also unique (bypassing the forward rate limit at L132–143). The map grows without bound until the 5-minute `notify` tick.
