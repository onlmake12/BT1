### Title
Absence of Cumulative Balance Check Against `MAX_DEPOSIT_LIMIT` in Permanent Subscriptions — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`MAX_DEPOSIT_LIMIT` is defined as a constant (100 ether) to cap the total ETH that can be permanently locked in a permanent Scheduler subscription. However, the enforcement in `addFunds` checks only the per-transaction `msg.value` against this constant, not the cumulative `status.balanceInWei`. Any caller can bypass the cap by making multiple deposits of exactly `MAX_DEPOSIT_LIMIT`, accumulating an unbounded balance in a permanent subscription from which ETH can never be withdrawn.

---

### Finding Description

`SchedulerConstants` defines:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
``` [1](#0-0) 

In `createSubscription`, the check is correctly scoped to the initial deposit:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [2](#0-1) 

In `addFunds`, the same per-transaction check is applied — but the cumulative balance is never validated:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [3](#0-2) 

The check `msg.value > MAX_DEPOSIT_LIMIT` passes as long as each individual deposit is ≤ 100 ether. There is no check of the form `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`. An attacker can call `addFunds` with exactly 100 ether repeatedly, accumulating 200, 300, 400 ether, etc., in the permanent subscription's balance.

`addFunds` has no access control — it is callable by any address: [4](#0-3) 

Permanent subscriptions prohibit all withdrawals:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [5](#0-4) 

So any ETH deposited beyond `MAX_DEPOSIT_LIMIT` is permanently and irrecoverably locked in the contract.

---

### Impact Explanation

- ETH permanently locked in the contract beyond the protocol-intended maximum, with no recovery path.
- The `MAX_DEPOSIT_LIMIT` invariant — the only protection against unbounded permanent ETH lock-up — is trivially bypassed.
- Locked ETH is slowly drained to keepers via `_payKeeper`, but at keeper-fee rates this could take years for large deposits, effectively constituting permanent loss of user funds.
- Any third party can also grief a permanent subscription owner by depositing on their behalf (since `addFunds` is permissionless), forcing the owner's ETH to be locked beyond the limit they intended.

---

### Likelihood Explanation

- The entry path is a public, payable function (`addFunds`) with no access control.
- No special role, key, or governance action is required.
- The bypass requires only two sequential calls with `msg.value = MAX_DEPOSIT_LIMIT`.
- The existing test suite confirms `addFunds` is intentionally open to anyone: [6](#0-5) 

---

### Recommendation

Replace the per-transaction check in `addFunds` with a cumulative balance check:

```solidity
// Before crediting the balance:
if (params.isPermanent &&
    status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

Apply the same fix to `createSubscription` for consistency (though the initial deposit is a single transaction, the pattern should be uniform).

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription with the maximum allowed deposit
params.isPermanent = true;
uint256 subId = scheduler.createSubscription{value: 100 ether}(params);
// status.balanceInWei == 100 ether ✓

// 2. Call addFunds repeatedly — each call passes the check since msg.value == MAX_DEPOSIT_LIMIT (not >)
scheduler.addFunds{value: 100 ether}(subId);
// status.balanceInWei == 200 ether — exceeds MAX_DEPOSIT_LIMIT, no revert

scheduler.addFunds{value: 100 ether}(subId);
// status.balanceInWei == 300 ether — no revert

// 3. Withdrawal is permanently blocked
scheduler.withdrawFunds(subId, 1 ether);
// reverts: CannotUpdatePermanentSubscription

// Result: 300 ether permanently locked, MAX_DEPOSIT_LIMIT bypassed
``` [4](#0-3) [7](#0-6)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
