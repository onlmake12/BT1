### Title
`prune()` skips Header/TxHash deletion when no `ConsumedOutPoint` entries exist, causing unbounded storage growth — (`util/indexer/src/indexer.rs`)

### Summary

`Indexer::prune()` uses `min_block_number` (initialized to `BlockNumber::MAX`) as the seek key for the Header/TxHash deletion iterator. When no `ConsumedOutPoint` entries exist — which is the normal state when all indexed blocks contain only cellbase transactions — `min_block_number` is never updated, the iterator seeks past all real block entries, and neither `Header` nor `TxHash` records are ever deleted.

### Finding Description

**Root cause — `prune()`, lines 765–805:**

`min_block_number` is initialized to `BlockNumber::MAX`: [1](#0-0) 

It is only updated inside the `ConsumedOutPoint` loop body. If that iterator yields zero items (empty namespace), `min_block_number` stays `u64::MAX`.

The Header/TxHash deletion iterator is then seeked using that sentinel value: [2](#0-1) 

`key_prefix_header` becomes `[0xE0, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF]`. Since block numbers are stored big-endian and real chain heights are orders of magnitude smaller than `u64::MAX`, the RocksDB iterator finds no entries at or after that seek position. The `take_while` guard is never entered, so no `Header` or `TxHash` keys are deleted.

**Why `ConsumedOutPoint` entries are absent for cellbase-only blocks:**

In `append()`, the `ConsumedOutPoint` write is inside `if tx_index > 0`: [3](#0-2) 

Cellbase (tx_index == 0) is explicitly skipped. Meanwhile, cellbase outputs that pass the cell filter do set `tx_matched = true` and cause a `TxHash` entry to be written: [4](#0-3) 

And a `Header` entry is written for every block regardless: [5](#0-4) 

So for every cellbase-only block: one `Header` entry + one `TxHash` entry are written, zero `ConsumedOutPoint` entries are written, and `prune()` deletes none of them.

### Impact Explanation

`KeyPrefix::TxHash` (192) and `KeyPrefix::Header` (224) namespaces grow monotonically. With `keep_num = 100` and `prune_interval = 2000` (typical defaults), after N blocks the store holds O(N) unreclaimed entries. On a chain that has ever had a sustained period of cellbase-only blocks (early chain, low-activity periods, or a miner deliberately omitting transactions), the indexer's RocksDB instance grows without bound, eventually exhausting disk space and degrading all read/write operations.

### Likelihood Explanation

Cellbase-only blocks are routine on CKB mainnet during low-activity periods and are trivially producible by any miner regardless of hashpower share — a miner simply omits all pending transactions from the block template. No majority hashpower, no key material, and no privileged access is required. The bug is also triggered by ordinary chain sync on any historical segment that contained cellbase-only blocks, making it reachable without any adversarial intent.

### Recommendation

Replace the `min_block_number`-based seek with an unconditional seek from the start of the Header namespace, and use `prune_to_block` as the sole upper bound:

```rust
// prune Header => Transactions and TxHash => TransactionInputs
let key_prefix_header = vec![KeyPrefix::Header as u8];
let iter = self
    .store
    .iter(&key_prefix_header, IteratorDirection::Forward)?
    .take_while(|(key, _value)| {
        key.starts_with(&[KeyPrefix::Header as u8])
            && BlockNumber::from_be_bytes(
                key[1..9].try_into().expect("stored block_number"),
            ) <= prune_to_block
    });
```

This removes the dependency on `min_block_number` entirely for the Header/TxHash sweep, ensuring old entries are always reclaimed regardless of whether any `ConsumedOutPoint` entries exist.

### Proof of Concept

```
1. Create an Indexer with keep_num=10, prune_interval=1.
2. append() 200 blocks, each containing only a cellbase transaction
   whose output passes the default cell filter.
3. Call prune() (it is called automatically on each append).
4. Count KeyPrefix::TxHash (192) and KeyPrefix::Header (224) entries.
5. Expected (correct): ≤ keep_num+1 entries each.
6. Actual (buggy): all 200 entries remain — prune() deleted nothing.
```
