### Title
`PythPriceOracleGetter.getAssetPrice()` Ignores Confidence Interval, Enabling Over-Borrowing via Adversarial Price Selection — (`File: target_chains/ethereum/contracts/contracts/aave/PythPriceOracleGetter.sol`)

---

### Summary

`PythPriceOracleGetter.getAssetPrice()` returns the raw Pyth spot price (`price.price`) to Aave for collateral valuation without applying any confidence-interval discount. Pyth's pull oracle model allows any unprivileged user to permissionlessly submit any valid (Wormhole-attested) price update within the staleness window. An attacker can select the most favorable price update within that window, and because no confidence-based discount is applied, collateral is systematically overvalued. This allows an attacker to over-borrow from any Aave deployment that uses this oracle, draining the lending pool.

---

### Finding Description

`PythPriceOracleGetter` is Pyth's own reference implementation of the Aave `IPriceOracleGetter` interface. Its `getAssetPrice()` function fetches the Pyth price and returns it directly:

```solidity
PythStructs.Price memory price = pyth().getPriceNoOlderThan(
    priceId,
    PythAssetRegistry.validTimePeriodSeconds()
);
// ...
uint256 normalizedPrice = uint64(price.price);  // raw spot price only
// price.conf is never read or used
return normalizedPrice;
```

The returned `PythStructs.Price` struct contains both `price.price` (the aggregate midpoint) and `price.conf` (the confidence interval). The function reads only `price.price` and discards `price.conf` entirely.

Pyth's own best-practices documentation explicitly states:

> "For lending protocols, use conservative pricing: Collateral valuation: `price - confidence` (lower bound)"

and

> "A lending protocol valuing a user's collateral can use the lower valuation price `μ−σ`."

The pull oracle model compounds this: because price updates are permissionless, any user can submit any valid Wormhole-attested price update that satisfies the staleness constraint. An attacker can therefore select the highest price within the staleness window and submit it immediately before borrowing, maximising the overvaluation.

---

### Impact Explanation

An Aave deployment using `PythPriceOracleGetter` as its price oracle will value collateral at the raw midpoint price rather than the conservative lower bound `price - conf`. During periods of high volatility or publisher disagreement, the confidence interval can be several percent of the price. An attacker who:

1. Observes a Pyth price update where `price.conf` is large (e.g., 2–5% of `price.price`),
2. Submits that update to the on-chain Pyth contract (permissionless),
3. Immediately calls Aave's `borrow()`,

will have their collateral valued at `price.price` instead of `price.price - price.conf`. The attacker borrows more than the conservative collateral value supports. If the true market price then falls to the lower bound, the position becomes undercollateralised and the lending pool suffers a bad-debt loss. The attack is repeatable and scales with pool TVL.

---

### Likelihood Explanation

- The pull oracle model is permissionless by design: any transaction sender can call `updatePriceFeeds()` with any valid VAA.
- Pyth price feeds update every ~400 ms; within a 60-second staleness window there are ~150 price updates to choose from.
- Wide confidence intervals occur regularly during market volatility, which is precisely when attackers are most incentivised to exploit lending protocols.
- No special role, key, or governance access is required.

---

### Recommendation

Apply the confidence-interval discount recommended by Pyth's own best-practices documentation when valuing collateral:

```solidity
// For collateral valuation, use the conservative lower bound
uint256 conservativePrice = uint64(price.price) - price.conf;
```

Additionally, add a circuit-breaker that reverts when the confidence-to-price ratio exceeds a threshold (e.g., 2%):

```solidity
require(
    price.conf * 100 <= uint64(price.price) * MAX_CONF_RATIO,
    "Price confidence too wide"
);
```

**Root cause — `price.conf` is never read:** [1](#0-0) 

The `PythStructs.Price` struct carries both `price` and `conf` fields, but only `price` is consumed: [2](#0-1) 

**Pyth's own documentation mandates the lower bound for collateral:** [3](#0-2) 

**Pull oracle is permissionless — any user can submit any valid price update:** [4](#0-3) 

**Adversarial price selection is a documented attack vector:** [5](#0-4) 

**Attack scenario:**

1. Attacker monitors Hermes for a price update where `conf/price > 2%` (e.g., ETH/USD: price = $3000, conf = $90).
2. Attacker calls `pyth.updatePriceFeeds(vaa)` — permissionless, costs only the update fee.
3. Attacker calls Aave `borrow()`. Aave calls `getAssetPrice(ETH)`, which returns `$3000` (not `$2910`).
4. Attacker borrows against collateral overvalued by `$90 × collateral_amount`.
5. If ETH price falls to $2910, the position is undercollateralised and the pool absorbs the loss.

The `validTimePeriodSeconds` is set at construction time with no on-chain enforcement of a minimum: [6](#0-5) 

A deployer who sets a generous staleness window (e.g., 60 seconds as used in the test suite) gives the attacker a large selection of price updates to choose from, amplifying the adversarial-selection component of the attack. [7](#0-6)
