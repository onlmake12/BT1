Audit Report

## Title
`prune()` Fails to Delete `Header`/`TxHash` Entries When No `ConsumedOutPoint` Entries Exist — (`util/indexer/src/indexer.rs`)

## Summary
In `prune()`, `min_block_number` is derived solely from `ConsumedOutPoint` entries. When no such entries exist — as is the case for cellbase-only blocks — `min_block_number` remains `u64::MAX`. The subsequent `Header`/`TxHash` pruning iterator seeks to the end of the `Header` key space and immediately terminates, leaving all old `Header => Transactions` and `TxHash => TransactionInputs` entries in RocksDB indefinitely. This causes unbounded storage growth proportional to chain length.

## Finding Description
`min_block_number` is initialized to `BlockNumber::MAX` at line 765 and is only updated inside the `ConsumedOutPoint` loop (lines 766–781) when at least one entry with `block_number < prune_to_block` exists. [1](#0-0) 

When the loop body never executes, `min_block_number` remains `u64::MAX`. At line 785, `key_prefix_header` is extended with `u64::MAX.to_be_bytes()` = `[0xFF; 8]`, producing seek key `[224u8, 0xFF×8]` — the end of the `Header` key space. The `take_while` guard at line 793 (`BlockNumber <= prune_to_block`) immediately terminates because no real block exists at block `u64::MAX`. [2](#0-1) 

The root cause of absent `ConsumedOutPoint` entries for cellbase-only blocks: the input-processing loop is gated on `tx_index > 0` (line 339), so cellbase transactions (always `tx_index == 0`) never produce `ConsumedOutPoint` entries. [3](#0-2) 

However, cellbase outputs ARE processed (line 424 onward). If any output matches the cell filter, `tx_matched = true` (line 438), causing a `TxHash` entry to be written (lines 482–491). [4](#0-3) 

Additionally, `Header` entries are written unconditionally for every block (lines 494–510), regardless of whether any transactions matched. So even without a cell filter match, `Header` entries accumulate every block and are never pruned. [5](#0-4) 

The existing `prune_bound` test (lines 1534–1600) exercises exactly this scenario — 21 cellbase-only blocks — but only asserts that `get_block_hash` works; it does not assert that old `Header`/`TxHash` entries are absent after pruning, confirming the gap. [6](#0-5) 

## Impact Explanation
`TxHash => TransactionInputs` (prefix `192`) and `Header => Transactions` (prefix `224`) entries grow proportionally to chain length with no upper bound. The indexer's RocksDB store is never compacted of these entries, degrading read/scan performance and consuming unbounded disk space. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation
Triggered by any chain segment where non-cellbase transactions are absent — a normal condition during low-activity periods on mainnet. No attacker is required; any miner producing valid cellbase-only blocks (standard behavior) triggers this path. The condition is reachable without any special privilege and is a routine occurrence on the live network.

## Recommendation
Decouple the `Header`/`TxHash` pruning start key from `min_block_number`. When `min_block_number == BlockNumber::MAX` (no `ConsumedOutPoint` entries were pruned), the iterator should start from block `0` rather than `u64::MAX`:

```rust
let start_block = if min_block_number == BlockNumber::MAX {
    0u64
} else {
    min_block_number
};
let mut key_prefix_header = vec![KeyPrefix::Header as u8];
key_prefix_header.extend_from_slice(&start_block.to_be_bytes());
```

## Proof of Concept
1. Create an `Indexer` with `keep_num = 10`, `prune_interval = 1`.
2. Append `keep_num + 2 = 12` blocks, each containing only a cellbase transaction with one output matching the cell filter (no non-cellbase txs).
3. `prune()` is called automatically on each `append` since `prune_interval = 1`.
4. Scan all keys with prefix `KeyPrefix::TxHash` (`192`) and `KeyPrefix::Header` (`224`).
5. Assert that entries for blocks `<= tip - keep_num - 1` are absent — they will **not** be absent, demonstrating the bug.

The `prune_bound` test at line 1534 already sets up this exact scenario but omits the pruning invariant assertion, confirming the gap. [7](#0-6)
