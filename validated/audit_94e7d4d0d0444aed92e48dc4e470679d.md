### Title
TOCTOU Race in `is_pending_verify` Allows Child Blocks of Failed Parents to Enter Verification Pipeline, Causing Panic and Block-Verification DoS — (`chain/src/orphan_broker.rs`)

---

### Summary

`OrphanBroker::process_lonely_block` reads `is_pending_verify` to decide whether to queue a child block for verification. This read and the subsequent enqueue are not atomic. A concurrent `ConsumeUnverifiedBlocks` thread can fail-verify the parent block, delete it from the store, and remove it from `is_pending_verify` between those two steps. The child block is then enqueued in `preload_unverified_tx` with a now-deleted parent. When `PreloadUnverifiedBlocksChannel` later loads the child, it calls `expect("parent header stored")` on the deleted parent header, panicking the thread. Because `PreloadUnverifiedBlocksChannel::start` has no `catch_unwind`, the thread dies, its `unverified_block_tx` sender is dropped, `ConsumeUnverifiedBlocks` exits on the resulting channel error, and block verification halts permanently.

---

### Finding Description

CKB's async block pipeline runs across three threads:

- **ChainService** (`chain/src/chain_service.rs`) — receives blocks, does non-contextual verification, inserts into DB, calls `orphan_broker.process_lonely_block`.
- **PreloadUnverifiedBlocksChannel** (`chain/src/preload_unverified_blocks_channel.rs`) — reads `LonelyBlockHash` from `preload_unverified_tx`, loads the full block and parent header from the store, forwards to `unverified_block_tx`.
- **ConsumeUnverifiedBlocks** (`chain/src/verify.rs`) — reads `UnverifiedBlock` from `unverified_block_rx`, runs contextual verification, removes the block hash from `is_pending_verify`.

`is_pending_verify` is a shared `Arc<DashSet<Byte32>>` that marks blocks currently in the verification pipeline.

### Impact Explanation

A single malicious peer can permanently halt block verification on a target node. The node continues to accept P2P connections and relay messages but never advances its chain tip. This is a complete sync denial-of-service requiring no privileged access and no majority hashpower.

---

### Likelihood Explanation

An attacker needs to:
1. Send block A with valid PoW and valid non-contextual structure, but invalid contextual content (e.g., a transaction spending a non-existent cell, or a script that fails). This passes `non_contextual_verify` and `insert_block` but fails `ContextualBlockVerifier`.
2. Immediately send child block B (also with valid PoW) before A's contextual verification completes.

Contextual verification involves CKB-VM script execution and can take tens to hundreds of milliseconds. The race window is wide enough to be reliably triggered. No special network position is required; any peer that can relay blocks can attempt this.

---

### Recommendation

1. **Atomic state transition**: In `process_lonely_block`, after the `is_pending_verify.contains` check, re-verify the parent's status under a lock or use a compare-and-swap pattern before enqueuing the child. If the parent has transitioned to `BLOCK_INVALID` between the check and the enqueue, call `process_invalid_block` on the child instead.

2. **Remove the `expect` panics**: Replace `expect("parent header stored")` and `expect("block stored")` in `load_full_unverified_block_by_hash` with graceful error handling. If the parent header is missing, log an error and discard the child block rather than panicking.

3. **Add `catch_unwind` to `PreloadUnverifiedBlocksChannel::start`**: Mirror the panic-recovery pattern already used in `ConsumeUnverifiedBlocks::start` so that a single bad block cannot kill the thread.

---

### Proof of Concept

1. Attacker connects to a target CKB node as a sync peer.
2. Attacker crafts block A: valid PoW, valid non-contextual structure, but containing a transaction that spends a dead cell (fails contextual verification).
3. Attacker sends block A via `SendBlock`. The ChainService thread inserts A into the DB and places A in `is_pending_verify`, then forwards A to the `preload_unverified_tx` channel.
4. While `ConsumeUnverifiedBlocks` is running contextual verification on A (CKB-VM execution), the attacker immediately sends child block B (parent = A, valid PoW).
5. The ChainService thread calls `process_lonely_block(B)`, sees `is_pending_verify.contains(&A_hash) == true`, and calls `process_descendant(B)` — enqueuing B in `preload_unverified_tx`.
6. `ConsumeUnverifiedBlocks` finishes verifying A: failure. It calls `delete_unverified_block(A)` (removing A's header from the store), marks A as `BLOCK_INVALID`, and removes A from `is_pending_verify`.
7. `PreloadUnverifiedBlocksChannel` dequeues B and calls `get_block_header(&A_hash).expect("parent header stored")` — A's header is gone → **panic**.
8. The preload thread dies. `unverified_block_tx` sender is dropped. `ConsumeUnverifiedBlocks` exits on channel error. Block verification halts permanently.
