### Title
`update_y` Does Not Advance `last_updated_epoch`, Enabling Retroactive Reward Accrual Over Zero-Rate Epochs — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

In Pyth's Oracle Integrity Staking (OIS) integrity pool program, the `update_y` instruction changes the reward rate `y` stored in `PoolConfig` but does **not** touch `PoolData.last_updated_epoch`. When `y` is set to `0` (effectively pausing rewards) and later restored to a non-zero value, the epoch checkpoint is never advanced during the zero-rate window. The next `advance` call will then process all skipped epochs using the **new, non-zero `y`**, retroactively accruing rewards for the period during which rewards were supposed to be zero.

---

### Finding Description

The integrity pool program exposes two relevant instructions:

**`update_y`** — modifies only `PoolConfig`:

```
accounts: [reward_program_authority (signer), pool_config (writable), system_program]
args: [y: u64]
```

It has **no `pool_data` account**, so it cannot touch `PoolData.last_updated_epoch`. [1](#0-0) 

**`advance`** — reads `pool_config.y` and writes `pool_data` (including `last_updated_epoch`):

```
accounts: [signer, pool_data (writable), publisher_caps, pool_config, pool_reward_custody (writable)]
``` [2](#0-1) 

`PoolData` holds the epoch checkpoint:

```json
{ "name": "last_updated_epoch", "type": "u64" }
``` [3](#0-2) 

Each `advance` call records an `Event` with the **current** `y` from `PoolConfig` for that epoch:

```json
{ "name": "epoch", "type": "u64" },
{ "name": "y",     "type": "u64" }
``` [4](#0-3) 

Because `y` is a single live value in `PoolConfig` (not stored historically per-epoch before `advance` is called), when `advance` catches up over multiple missed epochs it applies the **current** `y` to all of them. There is no per-epoch historical `y` snapshot taken at the time of `update_y`.

**Attack scenario:**

1. Epoch 100: `y = 10`, `last_updated_epoch = 100`, `advance` called normally.
2. Epoch 200: `reward_program_authority` calls `update_y(0)`. `pool_config.y = 0`. `last_updated_epoch` stays at `100` (no `pool_data` account in `update_y`).
3. Epochs 101–200: `advance` is not called (no economic incentive; `y = 0` means zero rewards, so keepers may skip).
4. Epoch 200: `reward_program_authority` calls `update_y(10)`. `pool_config.y = 10`. `last_updated_epoch` still `100`.
5. Epoch 201: Anyone calls `advance`. It catches up from epoch 100 to 201 using `y = 10`, recording 101 epochs of rewards — including the 100 epochs (101–200) when `y` was supposed to be `0`.

Users who call `advance_delegation_record` after step 5 will receive rewards for the zero-rate window. [1](#0-0) [3](#0-2) 

---

### Impact Explanation

Stakers and publishers receive PYTH token rewards for epochs during which `y = 0` was explicitly set. This drains the `pool_reward_custody` token account beyond the intended reward budget. The `claimable_rewards` field in `PoolData` is inflated, and users can claim those inflated rewards via `advance_delegation_record`. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The `reward_program_authority` is a privileged role, but the vulnerability does **not** require a malicious authority — it is a latent accounting flaw that fires automatically whenever:

1. `y` is set to `0` (already done per OP-PIP-103 on mainnet), AND
2. `advance` is not called for one or more epochs during the zero-rate window (plausible since there is no reward incentive to call it), AND
3. `y` is later restored to non-zero.

Step 3 is the planned future state of the protocol. Any unprivileged staker can then call `advance_delegation_record` to claim the inflated rewards — no special access required. [1](#0-0) 

---

### Recommendation

`update_y` should require `pool_data` as a writable account and call the same epoch-advance logic used by `advance` before changing `y`. Concretely:

- Add `pool_data` (writable, related to `pool_config`) to the `update_y` instruction accounts.
- Before writing the new `y`, advance `last_updated_epoch` to the current epoch (recording events with the **old** `y`).
- Only then update `pool_config.y`.

This mirrors the fix pattern from the ZEALInfinityPool report: flush the pending state with the old rate before the rate change takes effect.

---

### Proof of Concept

1. Confirm `update_y` has no `pool_data` account (cannot touch `last_updated_epoch`): [1](#0-0) 

2. Confirm `PoolData.last_updated_epoch` is the sole epoch checkpoint: [7](#0-6) 

3. Confirm `Event.y` is populated at `advance`-call time (not at `update_y` time), meaning there is no historical snapshot of `y` for missed epochs: [8](#0-7) 

4. Confirm `advance_delegation_record` is permissionless and pays out based on recorded events: [9](#0-8) [10](#0-9) 

> **Note on source availability:** The Rust implementation of `advance` (which would confirm the exact catch-up loop behavior) is not present in the indexed files of this repository. The structural proof above is derived entirely from the production IDL. If the `advance` implementation processes only one epoch per call and always requires being called every epoch without gaps, the impact window narrows but the root cause (missing `last_updated_epoch` update in `update_y`) remains. The finding should be verified against the on-chain program source.
