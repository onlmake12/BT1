### Title
Write lock held across unbounded `block_in_place` script execution in `readd_detached_tx` during reorg — (`tx-pool/src/process.rs`)

### Summary
During a chain reorganization, `update_tx_pool_for_reorg` holds the `tx_pool` write lock while calling `readd_detached_tx`, which in turn calls `verify_rtx` (full `ContextualTransactionVerifier` script execution via `block_in_place`) for every detached transaction that is not in the verification cache. Because the write lock is held for the entire duration, all concurrent tx-pool read and write operations are blocked until re-verification of all N detached transactions completes.

### Finding Description
`update_tx_pool_for_reorg` (`tx-pool/src/process.rs:836`) acquires `self.tx_pool.write().await` and, within the same lock scope, calls `self.readd_detached_tx(&mut tx_pool, retain, fetched_cache).await` (`line 849`). Inside `readd_detached_tx`, for each transaction without a cache hit, `verify_rtx` is called with `command_rx = None` (`line 895`). `verify_rtx` (`tx-pool/src/util.rs:117`) then calls `tokio::task::block_in_place(|| ContextualTransactionVerifier::new(...).verify(max_tx_verify_cycles, false))`, which runs the full RISC-V VM execution synchronously. The `tokio::sync::RwLockWriteGuard` is alive across all of these calls. `block_in_place` does not release the lock; it only signals Tokio to move other tasks to other threads, but those tasks cannot acquire the lock until the guard is dropped at line 851.

### Impact Explanation
All tx-pool operations that require the read or write lock — `get_block_template`, `submit_local_tx`, `submit_remote_tx`, `get_tx_pool_info` — are blocked for the entire re-verification window. With N detached transactions each consuming near-`max_tx_verify_cycles` cycles, the freeze duration scales as O(N × cycles). A mining node is effectively unable to produce new block templates or accept new transactions during this window.

### Likelihood Explanation
Any miner (no majority hashpower required) can construct a private chain of depth D containing blocks packed with high-cycle transactions, then release it to a target node. The node will reorg if the private chain has higher total difficulty. The transactions will not be in the victim's `txs_verify_cache` because they were never submitted to its pool. The attack is repeatable and locally testable on a private network.

### Recommendation
Release the write lock before calling `verify_rtx`. The standard pattern already used for normal transaction submission (`_process_tx`) performs script verification outside the lock and only acquires the write lock for the final `submit_entry` step. `readd_detached_tx` should be refactored to follow the same pattern: resolve and verify each transaction without holding the lock, collect results, then re-acquire the write lock only to insert verified entries.

### Proof of Concept
1. Set up a private CKB devnet with a target mining node.
2. Mine a side chain of depth 1 containing 50 transactions, each with a RISC-V script that loops to near `max_tx_verify_cycles`.
3. Release the side chain to the target node (ensure it has higher total difficulty).
4. Immediately after the reorg notification, poll `get_tx_pool_info` in a tight loop.
5. Observe that `get_tx_pool_info` is unresponsive for the entire re-verification window.
6. Measure freeze duration vs. N and cycles per transaction; assert linear scaling.
