### Title
Unbounded `ids[]` Array in Hermes SSE and WebSocket Streaming Endpoints Enables Amplified Computation DoS - (File: `apps/hermes/server/src/api/rest/v2/sse.rs`)

---

### Summary

The Hermes server's `/v2/updates/price/stream` SSE endpoint and the WebSocket subscribe handler accept an unbounded array of price feed IDs per connection. On every Pythnet slot update (~400 ms), the server executes `get_price_feeds_with_update_data` for every subscribed ID, building Merkle proofs for each. There is no server-side cap on the number of IDs a single connection may subscribe to, and `validate_price_ids` performs only existence checks. An unprivileged attacker can open a small number of connections each subscribing to all available price feeds (~500+), causing disproportionate and sustained server-side computation at zero cost.

---

### Finding Description

`StreamPriceUpdatesQueryParams` declares `ids` as an unbounded `Vec<PriceIdInput>`: [1](#0-0) 

`validate_price_ids`, called before the stream is established, only partitions IDs into valid/invalid sets — it contains no maximum-count guard: [2](#0-1) 

Once the SSE connection is open, `handle_aggregation_event` fires on every `AggregationEvent` (every Pythnet slot). It calls `get_price_feeds_with_update_data` for the full subscribed ID list, building Merkle proofs for every requested feed: [3](#0-2) 

The connection is kept alive for up to 24 hours: [4](#0-3) 

The identical pattern exists in the WebSocket handler, where `Subscribe` messages also accept an unbounded `ids` array with no count check before inserting into `price_feeds_with_config`: [5](#0-4) 

---

### Impact Explanation

Each slot update (~2.5/second) triggers Merkle proof construction for every subscribed ID. With 500 IDs per connection and multiple connections open simultaneously, the attacker forces the Hermes server to perform hundreds of proof-generation operations per second continuously for up to 24 hours per connection. This can exhaust CPU and memory on the Hermes node, degrading or denying price-update service to all downstream consumers (dApps, price pushers, on-chain contracts relying on Hermes for VAA data).

---

### Likelihood Explanation

The public Hermes endpoint enforces a rate limit of 10 requests per 10 seconds per IP, but:
1. The rate limit governs connection establishment, not the ongoing per-slot computation of an already-open SSE stream.
2. An attacker using multiple IPs or targeting node-provider instances (which may have no rate limit) can open many such connections.
3. The attack requires only a standard HTTP GET request with a large `ids[]` query string — no credentials, fees, or privileged access.

---

### Recommendation

1. Add a server-side maximum on the number of IDs accepted per SSE/WebSocket connection (e.g., 100), enforced in `validate_price_ids` or at the handler level before the stream is established.
2. Apply the same cap to the REST batch endpoints (`/v2/updates/price/latest`, `/v2/updates/price/{publish_time}`, `/api/latest_price_feeds`, `/api/latest_vaas`), all of which share the same unbounded `ids: Vec<PriceIdInput>` pattern.

---

### Proof of Concept

```bash
# Open a single SSE connection subscribing to all ~500 available price feeds.
# Repeat from multiple IPs to amplify load.
IDS=$(curl -s https://hermes.pyth.network/api/price_feed_ids | jq -r '.[] | "ids[]=" + .' | tr '\n' '&')
curl -N "https://hermes.pyth.network/v2/updates/price/stream?${IDS}" &
# Each Pythnet slot (~400ms) now triggers get_price_feeds_with_update_data
# for all 500 IDs on this connection, sustained for up to 24 hours.
```

The attacker pays nothing. The Hermes server performs O(N_ids × slots_per_second) Merkle proof operations per connection, per second, for the full connection lifetime.

pub async fn validate_price_ids<S>(
    state: &ApiState<S>,
    price_ids: &[PriceIdentifier],
    remove_invalid: bool,
) -> Result<Vec<PriceIdentifier>, RestError>
where
    S: Aggregates,
{
    let state = &*state.state;
    let available_ids = Aggregates::get_price_feed_ids(state).await;

    // Partition into (valid_ids, invalid_ids)
    let (valid_ids, invalid_ids): (Vec<_>, Vec<_>) = price_ids
        .iter()
        .copied()
        .partition(|id| available_ids.contains(id));

    if invalid_ids.is_empty() || remove_invalid {
        // All IDs are valid
        Ok(valid_ids)
    } else {
        // Return error with list of missing IDs
        Err(RestError::PriceIdsNotFound {
            missing_ids: invalid_ids,
        })
    }
}
