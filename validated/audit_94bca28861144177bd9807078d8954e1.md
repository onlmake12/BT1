### Title
No Excess Fee Refund and No Maximum Fee Guard in `requestV2()` — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `requestHelper()` function accepts `msg.value` for a fee payment whose required amount is computed dynamically at execution time. Any `msg.value` exceeding the provider fee is silently credited to `_state.accruedPythFeesInWei` (the Pyth treasury) and is never returned to the caller. There is no `maxFee` parameter allowing callers to bound their payment, creating a direct financial loss vector analogous to the SecuritizeVault slippage issue.

---

### Finding Description

In `requestHelper()`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire surplus `msg.value - providerFee` is added to `accruedPythFeesInWei`. There is no branch that returns excess ETH to `msg.sender`. The interface documentation explicitly acknowledges this: *"excess value is not refunded to the caller"*. [2](#0-1) 

The fee is composed of a dynamic provider fee (`feeInWei`, settable by the provider's fee manager via `setProviderFee`) and a dynamic Pyth protocol fee (`pythFeeInWei`, settable by governance). Both can change between the block in which a user reads `getFeeV2()` and the block in which their `requestV2{value: X}()` transaction is mined. [3](#0-2) 

---

### Impact Explanation

If the fee decreases between a user's off-chain `getFeeV2()` query and on-chain execution, the user overpays. The surplus is permanently captured by the Pyth treasury (`accruedPythFeesInWei`). There is no mechanism for the user to recover it. For integrating smart contracts that pre-fund requests (e.g., `requestV2{value: storedFee}()`), a fee reduction between storage and execution causes a guaranteed, unrecoverable ETH loss on every call. The loss per call equals `msg.value - requiredFee`, which can be non-trivial if fees are actively managed or if the caller over-provisions ETH as a safety margin.

---

### Likelihood Explanation

- The provider fee manager can call `setProviderFee()` at any time, and Pyth governance can adjust `pythFeeInWei` at any time.
- The interface documentation itself warns callers that the fee changes over time, confirming this is an expected operational condition, not a theoretical edge case.
- Any integrating contract that caches the fee (a common pattern) or that sends a small buffer above the queried fee will silently lose ETH on every request where the fee has dropped.
- No privileged access is required by the attacker — the loss is triggered by the normal, unprivileged act of calling `requestV2()`.

---

### Recommendation

1. **Add a `maxFee` parameter** to all `requestV2` variants so callers can bound their payment:
   ```solidity
   function requestV2(uint32 gasLimit, uint128 maxFee)
       external payable returns (uint64 assignedSequenceNumber) {
       require(getFeeV2(gasLimit) <= maxFee, "Fee exceeds maxFee");
       ...
   }
   ```
2. **Refund excess `msg.value`** after the required fee is deducted:
   ```solidity
   uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
   if (excess > 0) {
       (bool ok,) = msg.sender.call{value: excess}("");
       require(ok, "Refund failed");
   }
   _state.accruedPythFeesInWei += (requiredFee - providerFee);
   ```
3. Update the `IEntropyV2` interface NatSpec to reflect the new behavior.

---

### Proof of Concept

1. Provider sets `feeInWei = 80 wei`; Pyth fee = 20 wei → `getFeeV2()` returns 100 wei.
2. Integrating contract stores `cachedFee = 100 wei` and later calls `entropy.requestV2{value: cachedFee}()`.
3. Before the tx is mined, the provider calls `setProviderFee(30 wei)` → `getFeeV2()` is now 50 wei.
4. `requestHelper` executes: `requiredFee = 50`, check passes (`100 >= 50`).
5. `providerFee = 30`, so `accruedPythFeesInWei += 100 - 30 = 70` (instead of the correct `20`).
6. The caller loses 50 wei silently; the Pyth treasury gains 50 wei it is not entitled to.
7. This repeats on every subsequent call until the contract's cached fee is updated, draining ETH from the integrating contract. [4](#0-3)
