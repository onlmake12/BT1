Audit Report

## Title
Tx-Pool Re-verifies All Persisted Transactions on Every Startup Without Caching Verification Results - (File: `tx-pool/src/service.rs`)

## Summary
On every node restart, `TxPoolServiceBuilder::start` loads raw transactions from the persisted file and re-submits each one through the full verification pipeline. The in-memory `TxVerificationCache` is never written to disk and starts empty on each boot, so every persisted transaction unconditionally triggers full CKB-VM script execution. An attacker who fills the pool with cycle-heavy transactions (paying `min_fee_rate` once) causes repeated CPU saturation and delayed block template availability on every subsequent node restart.

## Finding Description
`TxPoolServiceBuilder::start` (service.rs L581) calls `tx_pool.load_from_file()`, which deserializes only raw `TransactionVec` bytes — no cycle counts, no verification results (`persisted.rs` L17–55). `save_into_file` (`persisted.rs` L57–90) likewise persists only raw transaction bytes. `load_persisted_data` (service.rs L433–453) iterates every transaction and calls `self.submit_local_tx(tx)` for each. `submit_local_tx` sends `Message::SubmitLocalTx`, handled by `service.process_tx(tx, None)` (service.rs L809), which calls `_process_tx`. Inside `_process_tx` (process.rs L719), `fetch_tx_verify_cache` is called against `TxVerificationCache`, which is `lru::LruCache<Byte32, CacheEntry>` with `CACHE_SIZE = 30_000` (cache.rs L11–13) — a pure in-memory structure never written to disk. On restart it is always empty, so `verify_cache` is always `None` for all persisted transactions. `verify_rtx` (util.rs L96–131): when `cache_entry` is `None`, falls through to `ContextualTransactionVerifier::verify(max_tx_verify_cycles, false)` — full CKB-VM script execution for every transaction. There is no guard, cap, or fast-path that skips re-execution for previously verified transactions on startup.

## Impact Explanation
**Low (501–2000 points): Any other important performance improvements for CKB.** With `max_tx_pool_size = 180_000_000` bytes and `max_tx_verify_cycles = 70_000_000` cycles per transaction, an attacker can fill the pool with many small but cycle-heavy transactions. On every restart, the node must re-execute all scripts before the tx-pool is fully operational, saturating CPU and delaying block template generation. The node does not crash and eventually recovers, so this does not meet the threshold for High or Critical impacts.

## Likelihood Explanation
The attack requires calling the `send_transaction` RPC. By default the RPC is bound to `127.0.0.1:8114` (localhost only), so the attacker needs local access or an operator who has exposed the RPC externally. The attacker must also pay `min_fee_rate` (default 1000 shannons/KB) for all submitted transactions. Node restarts are routine (upgrades, maintenance, crash recovery), and the persisted pool state survives graceful shutdowns, so the cost is paid once but the re-verification penalty is incurred on every subsequent restart.

## Recommendation
**Short term**: Extend the persisted data format to include verified cycle counts alongside raw transaction bytes. On reload, when a cached cycle count is present and the transaction's inputs are still live in the current chain snapshot, skip full script re-execution and use `TimeRelativeTransactionVerifier` only (the same lightweight path already taken on in-memory cache hits in `verify_rtx` L96–100).

**Long term**: Add a startup benchmark test measuring tx-pool reload time with a pool filled to capacity with max-cycle transactions and enforce a time bound. Consider a configurable `max_pool_reload_cycles` cap to bound worst-case startup cost independently of pool size.

## Proof of Concept
1. Expose or access the CKB node's RPC (default `127.0.0.1:8114`).
2. Submit N transactions via `send_transaction`, each with a lock script that consumes close to `max_tx_verify_cycles` (70M) cycles. Keep total byte size under `max_tx_pool_size` (180 MB).
3. Trigger a graceful shutdown (`ckb stop`). `save_into_file` persists only raw transaction bytes to disk.
4. Restart the node. Observe `TxPoolServiceBuilder::start` → `load_persisted_data` → `submit_local_tx` × N → `_process_tx` → `verify_rtx` with `cache_entry = None` → full `ContextualTransactionVerifier` for every transaction.
5. Measure time until block templates include all pool transactions. Delay scales linearly with N × cycles-per-tx.

    /// Load persisted txs into pool, assume that all txs are sorted
    fn load_persisted_data(&self, txs: Vec<TransactionView>) -> Result<(), AnyError> {
        if !txs.is_empty() {
            info!("Loading persistent tx-pool data, total {} txs", txs.len());
            let mut failed_txs = 0;
            for tx in txs {
                if self.submit_local_tx(tx)?.is_err() {
                    failed_txs += 1;
                }
            }
            if failed_txs == 0 {
                info!("Persistent tx-pool data is loaded");
            } else {
                info!(
                    "Persistent tx-pool data is loaded, {} stale txs are ignored",
                    failed_txs
                );
            }
        }
        Ok(())
    }
