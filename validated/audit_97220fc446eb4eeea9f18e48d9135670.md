Audit Report

## Title
O(n) Linear Scan Over `unknown_tx_hashes` Under Blocking Mutex on Every `RelayTransactionHashes` Message When Queue Is Full — (File: sync/src/types/mod.rs)

## Summary
`SyncState::add_ask_for_txs` holds the `unknown_tx_hashes` mutex for its entire execution. When the queue reaches 50,000 entries, it performs a full O(n) nested scan over all entries and their peer lists before returning. Two unprivileged peers can cooperatively fill the queue to capacity while each staying below the per-peer ban threshold (32,767), then sustain continuous O(n) mutex-held scans on every subsequent relay message, blocking all callers that share the same mutex.

## Finding Description
`add_ask_for_txs` acquires `self.unknown_tx_hashes.lock()` at line 1484 and holds it until the function returns. Hashes are inserted unconditionally first (lines 1486–1504). After insertion, if `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) or the peer-scaled threshold is met (line 1507–1509), the code iterates over every `(hash, priority)` pair in the queue and, for each, iterates over `priority.peers` to count how many entries belong to `peer_index` (lines 1517–1522). The `peers` field is an unbounded `Vec<PeerIndex>` with no cap. If the per-peer counter reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767), the peer is banned; otherwise `Status::ignored()` is returned with no ban.

The exploit path: peer A sends 25,000 distinct fake hashes (queue = 25,000, no threshold); peer B sends 25,000 distinct fake hashes (queue = 50,000, threshold fires, O(50,000) scan runs, peer B counter = 25,000 < 32,767 → `Status::ignored()`, no ban). From this point, every `RelayTransactionHashes` message from any peer re-enters the scan path, running O(50,000) iterations under the mutex. `pop_ask_for_txs`, `mark_as_known_txs`, and `unknown_tx_hashes()` all contend on the same `Mutex`, starving legitimate relay and tx-pool scheduling operations. The attacker can sustain the saturated state indefinitely by periodically re-announcing hashes before natural eviction.

The existing per-peer ban check is insufficient because it fires only after the O(n) scan completes, and the threshold (32,767) is larger than the per-peer contribution needed to fill the queue with two peers (~25,000 each).

## Impact Explanation
This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** Sustained O(50,000) scans under the `unknown_tx_hashes` mutex block all concurrent callers on the victim node, degrading transaction relay throughput and tx-request scheduling. The attacker cost is minimal: two TCP connections and approximately 1.6 MB of fake hash data to saturate the queue, then cheap small relay messages to sustain the condition.

## Likelihood Explanation
Any unauthenticated peer on the `RelayV3` protocol can participate. No special privileges are required. The setup requires only two connections and two batches of 32-byte fake hashes (~800 KB each). The per-peer ban is bypassed by distributing hashes across peers. The saturated state can be maintained indefinitely since the node cannot distinguish fake hashes from real ones at announcement time.

## Recommendation
1. **Maintain a per-peer counter map** (`HashMap<PeerIndex, usize>`) updated incrementally on insert/remove, replacing the O(n) scan with an O(1) lookup.
2. **Apply the per-peer check before inserting** rather than after, so the queue never grows beyond the threshold due to a single peer's contribution.
3. **Bound `UnknownTxHashPriority::peers`** to a small constant (e.g., 8) to cap inner-loop cost even if the outer loop cannot be eliminated immediately.

## Proof of Concept
1. Attacker opens two peer connections (`peer_A`, `peer_B`) to the target node.
2. `peer_A` sends `RelayTransactionHashes` with 25,000 distinct fake tx hashes → queue grows to 25,000; threshold not crossed; no ban.
3. `peer_B` sends `RelayTransactionHashes` with 25,000 distinct fake tx hashes → queue reaches 50,000; threshold fires; O(50,000) scan runs; `peer_B` counter = 25,000 < 32,767 → `Status::ignored()`, not banned.
4. Attacker sends `RelayTransactionHashes` from `peer_A` (or any peer) at high rate → each message acquires the mutex, runs the O(50,000) scan, releases the mutex; `pop_ask_for_txs` and `mark_as_known_txs` callers are blocked waiting for the mutex.
5. Attacker periodically re-announces hashes to replenish evicted entries, sustaining the saturated condition and continuous O(n) scans indefinitely.
