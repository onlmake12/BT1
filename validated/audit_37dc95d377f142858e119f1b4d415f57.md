Audit Report

## Title
Incomplete Pool Size Accounting Omits `verify_queue` Size in `TxPool::limit_size` — (File: tx-pool/src/pool.rs)

## Summary
`TxPool::limit_size` enforces `max_tx_pool_size` by checking only `pool_map.total_tx_size`, while `VerifyQueue` enforces a separate hardcoded 256 MB ceiling independently. The two limits are fully additive, meaning the tx-pool subsystem can consume up to `max_tx_pool_size + 256 MB` of memory — silently violating the operator-configured resource bound. An unprivileged remote submitter can exploit this to cause OOM conditions on memory-constrained nodes.

## Finding Description
The eviction loop in `TxPool::limit_size` checks only `pool_map.total_tx_size`:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
``` [1](#0-0) 

It never reads `verify_queue.total_tx_size`. The `VerifyQueue` enforces its own independent ceiling via a hardcoded constant:

```rust
const DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE: usize = 256_000_000;
``` [2](#0-1) 

The `is_full` guard in `VerifyQueue::add_tx` checks only against this hardcoded constant, with no reference to `max_tx_pool_size` or current `pool_map` occupancy:

```rust
pub fn is_full(&self, add_tx_size: usize) -> bool {
    add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size
}
``` [3](#0-2) 

The entry path into the verify queue is `resumeble_process_tx`, which performs only `non_contextual_verify` (structural/format checks) and duplicate detection before calling `enqueue_verify_queue` — no check against `pool_map` occupancy or `max_tx_pool_size`: [4](#0-3) 

`enqueue_verify_queue` itself simply acquires the write lock and calls `queue.add_tx`: [5](#0-4) 

`TxPoolInfo` further obscures the combined pressure by reporting only `pool_map.total_tx_size` as `total_tx_size` and exposing only the entry count (not bytes) for the verify queue: [6](#0-5) 

The two limits are structurally independent and additive. No existing guard combines them.

## Impact Explanation
The effective memory ceiling for the tx-pool subsystem is `max_tx_pool_size + 256 MB`. With the default `max_tx_pool_size = 180 MB`, the node can consume ~436 MB for the tx-pool alone. An operator who sets `max_tx_pool_size = 10 MB` to conserve memory on a constrained host will find the node can actually consume ~266 MB — 26× the configured limit — with no eviction triggered. Under sustained transaction flooding this can trigger OOM, causing the OS to kill the node process. This maps to **High: Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The entry path is fully unprivileged. Any peer or RPC caller can invoke `send_transaction` (RPC) or the P2P relay path, both of which route into `resumeble_process_tx` → `enqueue_verify_queue`. Transactions need only pass `non_contextual_verify` (structural checks) and duplicate detection to be admitted to the verify queue — script verification happens later. An attacker can continuously submit transactions that will eventually fail script verification but occupy verify queue space while pending. No special privilege, key, or majority hashpower is required. The 256 MB ceiling is not operator-configurable, so the gap cannot be closed by configuration alone. [7](#0-6) 

## Recommendation
Pass the current `verify_queue` byte size into `limit_size` and incorporate it into the eviction condition:

```rust
while self.pool_map.total_tx_size + verify_queue_size > self.config.max_tx_pool_size {
    // evict ...
}
```

Alternatively, derive `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` from `max_tx_pool_size` at construction time rather than using a hardcoded constant, treating both as a shared budget. Additionally, `TxPoolInfo` should expose `verify_queue` byte usage (not just entry count) so operators can observe actual combined memory pressure.

## Proof of Concept
1. Configure a CKB node with `max_tx_pool_size = 10 MB`.
2. Connect as an unprivileged peer or RPC caller.
3. Submit a stream of syntactically valid transactions (e.g., spending non-existent inputs — they pass `non_contextual_verify` and duplicate checks and enter the verify queue before script verification rejects them). At ~500 bytes each, ~512,000 transactions saturate the 256 MB verify queue ceiling.
4. Simultaneously submit valid transactions that enter `pool_map` up to its 10 MB limit.
5. Observe via `tx_pool_info` RPC: `total_tx_size` reports only ~10 MB (pool_map), while actual RSS for the tx-pool subsystem is ~266 MB.
6. On a host with limited RAM, the node process is killed by the OOM killer while `limit_size` never fires, because `pool_map.total_tx_size` (10 MB) never exceeds `max_tx_pool_size` (10 MB).
