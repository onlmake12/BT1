### Title
`DaoScriptSizeVerifier` Bypassed via Input/Output Index Mismatch, Enabling DAO Interest Inflation — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`DaoScriptSizeVerifier::verify()` enforces that a DAO deposit cell and its corresponding withdrawal-marker output use lock scripts of the same byte size. It does so by zipping `resolved_inputs` with `transaction.outputs()` positionally (index-by-index). An unprivileged transaction sender can reorder inputs and outputs so the DAO deposit input sits at index `j` while the DAO withdrawal-marker output sits at index `k ≠ j`. Neither positional pair will satisfy the "both cells carry the DAO type script" guard, so the lock-size check is silently skipped. Because the DAO type script binary itself does not enforce this constraint (the Rust verifier was added explicitly as a "temporary solution till Nervos DAO script can be properly upgraded"), the bypass is complete. The attacker can then shrink the lock script on the withdrawal marker, inflating the "counted capacity" that accrues DAO interest.

---

### Finding Description

**Root cause — wrong entity used for the check**

In `verification/src/transaction_verifier.rs`, `DaoScriptSizeVerifier::verify()` iterates:

```rust
for (i, (input_meta, cell_output)) in self
    .resolved_transaction
    .resolved_inputs
    .iter()
    .zip(self.resolved_transaction.transaction.outputs())   // ← positional zip
    .enumerate()
{
    if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
        && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
    {
        continue;   // ← both must carry DAO type script at the SAME index
    }
    ...
    if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
        return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
    }
}
``` [1](#0-0) 

The guard `cell_uses_dao_type_script(input) && cell_uses_dao_type_script(output)` is evaluated on the **positionally paired** cells, not on the **semantically paired** DAO deposit ↔ withdrawal-marker cells. The DAO protocol does not require that the DAO deposit input and the DAO withdrawal-marker output share the same transaction index.

**Bypass construction**

A sender submits a phase-1 DAO withdrawal transaction structured as:

| Index | Input | Output |
|-------|-------|--------|
| 0 | Normal cell (no DAO type script) | DAO withdrawal marker (DAO type script, **small lock**) |
| 1 | DAO deposit cell (DAO type script, **large lock**, data = `0x00…00`) | Normal cell (no DAO type script) |

The verifier evaluates:
- Pair (0): `input[0]` has no DAO type script → `continue`
- Pair (1): `output[1]` has no DAO type script → `continue`

The lock-size check is never reached. The withdrawal marker is accepted with a lock script smaller than the deposit cell's lock script.

**Why the DAO type script does not close the gap**

The code comment at line 818 states explicitly:

> "It provides a temporary solution till Nervos DAO script can be properly upgraded." [2](#0-1) 

This confirms the compiled DAO type script binary does not itself enforce lock-script-size equality; the Rust verifier is the sole enforcement point.

**Downstream interest inflation**

In phase 2, `DaoCalculator::calculate_maximum_withdraw` computes interest using the withdrawal marker (the phase-1 output, now a phase-2 input):

```rust
let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
let output_capacity: Capacity = output.capacity().into();
let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [3](#0-2) 

`occupied_capacity` is derived from the withdrawal marker's lock script. A smaller lock script → smaller `occupied_capacity` → larger `counted_capacity` → larger `withdraw_counted_capacity` → larger total withdrawal. The attacker earns more interest than the deposit entitles them to, draining the DAO secondary-issuance pool.

The `CapacityVerifier` explicitly skips the `OutputsSumOverflow` check for any transaction that has at least one DAO-type-script input, delegating correctness entirely to the DAO type script and this verifier:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    // ... OutputsSumOverflow check
}
``` [4](#0-3) 

So there is no fallback capacity guard.

---

### Impact Explanation

An unprivileged user can inflate the DAO interest they receive beyond what their deposit entitles them to. The excess interest is drawn from the global DAO secondary-issuance pool (`s` field in the DAO header), reducing the interest available to all other depositors. Repeated exploitation across many deposits compounds the drain. The attack requires no privileged access, no majority hash power, and no social engineering.

---

### Likelihood Explanation

The attack requires only:
1. A standard DAO deposit (phase 0).
2. A phase-1 withdrawal transaction with a non-DAO input placed at index 0 and the DAO deposit input placed at index 1 (or any arrangement where the DAO input and DAO output are at different indices).
3. A phase-2 withdrawal transaction.

All three steps are reachable via the public RPC (`send_transaction`). No special tooling beyond a standard CKB SDK is needed.

---

### Recommendation

Replace the positional `.zip()` pairing in `DaoScriptSizeVerifier::verify()` with explicit semantic matching: for each input that carries the DAO type script and has all-zero cell data (deposit cell), locate the output that also carries the DAO type script and compare lock script sizes. Do not assume that the DAO deposit input and the DAO withdrawal-marker output share the same transaction index.

### Proof of Concept

```
Phase 0 – Deposit
  tx_deposit:
    input[0]:  normal UTXO (always-success lock, large args, e.g. 100 bytes)
    output[0]: DAO deposit cell
                 capacity = C
                 lock     = always-success { args: [0u8; 100] }   ← large lock
                 type     = DAO type script
                 data     = 0x00 00 00 00 00 00 00 00             ← deposit marker

Phase 1 – Prepare (bypass DaoScriptSizeVerifier)
  tx_prepare:
    input[0]:  any normal UTXO (no DAO type script)              ← index 0, no DAO
    input[1]:  tx_deposit.output[0]  (DAO type script, large lock, data=0x00…) ← index 1, DAO
    output[0]: DAO withdrawal marker
                 capacity = C
                 lock     = always-success { args: [] }           ← SMALL lock (0 bytes)
                 type     = DAO type script
                 data     = <deposit block number>                ← index 0, DAO
    output[1]: change cell (no DAO type script)                  ← index 1, no DAO

  DaoScriptSizeVerifier evaluation:
    pair(0): input[0] has no DAO type script → continue (skipped)
    pair(1): output[1] has no DAO type script → continue (skipped)
    → lock-size check never executes

Phase 2 – Withdraw
  tx_withdraw:
    input[0]:  tx_prepare.output[0]  (DAO withdrawal marker, SMALL lock)
    output[0]: normal cell receiving C + interest

  calculate_maximum_withdraw uses the SMALL lock's occupied_capacity:
    occupied_capacity  = 8 + 33 + 1 + 0  (capacity + code_hash + hash_type + args)
                       = 42 bytes  (vs. 142 bytes if large lock were preserved)
    counted_capacity   = C − 42           (inflated vs. C − 142)
    withdraw_capacity  = counted_capacity × (ar_withdraw / ar_deposit) + 42
                       → attacker receives more interest than legitimately earned
```

    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
