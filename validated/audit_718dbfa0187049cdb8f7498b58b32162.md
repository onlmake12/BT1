Audit Report

## Title
Premature `mark_as_known_txs` Before `submit_remote_tx` Enables Transaction Propagation Suppression - (File: sync/src/relayer/transactions_process.rs)

## Summary
In `TransactionsProcess::execute()`, `mark_as_known_txs` is called at line 76 to remove all incoming relay transaction hashes from `unknown_tx_hashes` and insert them into `tx_filter` before any submission result is known. The actual `submit_remote_tx` calls are deferred into a spawned async task. When submission fails, the hash is permanently absent from `unknown_tx_hashes`, preventing the node from proactively re-requesting the transaction from any other peer. An unprivileged P2P peer can exploit this to suppress targeted transactions from a victim node's pool.

## Finding Description
**Confirmed code path in `sync/src/relayer/transactions_process.rs`:**

At line 76, before the async spawn:
```rust
shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));
```
`mark_as_known_txs` (lines 1443–1451 of `sync/src/types/mod.rs`) atomically removes each hash from `unknown_tx_hashes` and inserts it into `tx_filter`. The actual submission happens in a separate spawned task (lines 80–93).

`remove_from_known_txs` (lines 1436–1438 of `sync/src/types/mod.rs`) only removes from `tx_filter` — it never touches `unknown_tx_hashes`. This asymmetry is the root cause.

**Failure path analysis (confirmed against code):**

- `resumeble_process_tx_and_notify_full_reject` (lines 364–366 of `tx-pool/src/process.rs`) only sends `TxVerificationResult::Reject` for `Reject::Full`. For all other rejection types (`DeclaredWrongCycles`, `LowFeeRate`, malformed), no reject notification is sent.
- `send_bulk_of_tx_hashes` (lines 673–675 of `sync/src/relayer/mod.rs`) calls `remove_from_known_txs` only when `TxVerificationResult::Reject` is received — i.e., only for `Reject::Full`.

**Note on a factual error in the submitted claim:** The claim states that for `Reject::DeclaredWrongCycles`, `TxVerificationResult::Reject` is sent and `remove_from_known_txs` is called. This is incorrect per the code — `TxVerificationResult::Reject` is only sent for `Reject::Full`. For `DeclaredWrongCycles`, `remove_from_known_txs` is never called, meaning the hash stays in `tx_filter` until TTL expiry AND is permanently absent from `unknown_tx_hashes`. This makes the `DeclaredWrongCycles` attack scenario more effective than claimed, not less.

**Concrete attack flow using `DeclaredWrongCycles`:**
1. Attacker connects as P2P relay peer and sends `RelayTransactionHashes` advertising hash `T`.
2. Victim adds `T` to `unknown_tx_hashes` and sends `GetRelayTransactions`.
3. Attacker responds with `RelayTransactions` containing `T` but with `declared_cycles` set to an incorrect value.
4. `mark_as_known_txs([T])` fires — `T` is removed from `unknown_tx_hashes` and inserted into `tx_filter`.
5. Verification fails with `DeclaredWrongCycles`; no `TxVerificationResult::Reject` is sent; `remove_from_known_txs` is never called.
6. `T` is absent from `unknown_tx_hashes` and present in `tx_filter` until TTL expiry.
7. Any subsequent `RelayTransactionHashes` announcement of `T` from other peers is filtered out by `tx_filter.contains(&tx.hash())` in the filter step (lines 49–55 of `transactions_process.rs`), preventing re-entry into `unknown_tx_hashes`.
8. The victim node will not proactively re-request `T` from any peer until TTL expiry.

## Impact Explanation
This matches **"High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** and **"Medium: Suboptimal implementation of CKB state storage mechanism."** An attacker with modest resources (multiple IP addresses or Tor exits) can continuously suppress targeted transactions from a victim node's pool by cycling through peer connections after each ban. At scale — targeting many transactions simultaneously across many victim nodes — this degrades transaction propagation across the network, delays block inclusion, and constitutes a low-cost denial-of-service against the mempool propagation layer. The `tx_filter` TTL window (during which the hash is blocked from re-entry) amplifies the suppression window beyond the immediate ban period.

## Likelihood Explanation
Medium. The attacker requires only an inbound P2P connection (CKB accepts public inbound connections by default) and knowledge of the target transaction hash (observable from the public mempool or by monitoring `RelayTransactionHashes` messages). No cryptographic material, privileged access, or victim mistakes are required. Sustaining suppression requires reconnecting after each ban, which is feasible with multiple IPs or Tor. The attack is most effective when the victim has few alternative peers that could re-announce the transaction before TTL expiry.

## Recommendation
Move `mark_as_known_txs` inside the spawned async task, called only after a successful `submit_remote_tx`:

```rust
self.relayer.shared.shared().async_handle().spawn(async move {
    for (tx, declared_cycles) in txs {
        match tx_pool.submit_remote_tx(tx.clone(), declared_cycles, peer).await {
            Ok(_) => {
                shared_state.mark_as_known_tx(tx.hash());
            }
            Err(e) => {
                error!("submit_tx error {}", e);
            }
        }
    }
});
```

Alternatively, on any submission failure, re-insert the hash into `unknown_tx_hashes` so the node can re-request it from other peers. Also fix `remove_from_known_txs` to restore the hash to `unknown_tx_hashes` when appropriate, mirroring the full inverse of `mark_as_known_txs`.

## Proof of Concept
1. Connect to a CKB node as a P2P relay peer.
2. Observe pending transaction hash `T` from the public mempool.
3. Send `RelayTransactionHashes` advertising `T` to the victim node.
4. When the victim sends `GetRelayTransactions` for `T`, respond with a `RelayTransactions` message containing `T` but with `declared_cycles` set to an incorrect value (e.g., `actual_cycles + 1`).
5. Confirm via debug logging that `mark_as_known_txs` fires (line 76 of `transactions_process.rs`), removing `T` from `unknown_tx_hashes`.
6. Confirm verification fails with `DeclaredWrongCycles`; confirm no `TxVerificationResult::Reject` is emitted (only `Reject::Full` triggers this at lines 364–366 of `tx-pool/src/process.rs`).
7. Confirm `T` is absent from `unknown_tx_hashes` and present in `tx_filter`.
8. Send `RelayTransactionHashes` for `T` from a second peer connection; confirm the victim does not add `T` to `unknown_tx_hashes` (blocked by `tx_filter`).
9. Confirm the victim does not proactively request `T` from any connected peer.
10. Repeat from step 3 using a new peer connection after the ban expires to sustain suppression beyond TTL.
