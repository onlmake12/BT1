### Title
Zero-Amount Withdrawal Allowed in Entropy Fee Accounting — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The `withdraw` and `withdrawAsFeeManager` functions in `Entropy.sol` perform no `amount > 0` check. Any address — including unregistered providers — can call `withdraw(0)`, pass the balance check (`0 >= 0` is always true for `uint128`), and trigger a `call{value: 0}("")` to themselves while emitting a spurious `Withdrawal` event. This is a direct analog to the AMM `removeLiquidity(shareAmount == 0)` class of zero-amount operation vulnerabilities.

### Finding Description
`withdraw(uint128 amount)` reads the caller's `ProviderInfo` from storage and checks only that `providerInfo.accruedFeesInWei >= amount`. [1](#0-0) 

Because `amount` is `uint128`, the condition `0 >= 0` is trivially true for any caller — including addresses that have never registered as a provider (whose `accruedFeesInWei` defaults to `0`). The function then executes `msg.sender.call{value: 0}("")` and emits a `Withdrawal` event with `amount = 0`. [2](#0-1) 

`withdrawAsFeeManager` does check `providerInfo.sequenceNumber == 0` to block unregistered providers, but it still lacks an `amount > 0` guard, so a registered fee manager can call it with `amount = 0` and emit a spurious `Withdrawal` event. [3](#0-2) 

Note that `withdraw` itself has **no** `sequenceNumber != 0` guard (unlike `withdrawAsFeeManager`), so the zero-amount path is reachable by any EOA or contract. [4](#0-3) 

### Impact Explanation
- **Event log pollution**: Any address can emit `Withdrawal` events with `amount = 0`, corrupting off-chain monitoring, indexers, and analytics that track provider fee flows.
- **Spurious external call**: `msg.sender.call{value: 0}("")` is executed against the caller's address. For contract callers this triggers their `receive()` or `fallback()` function. Although the checks-effects-interactions pattern prevents reentrancy from draining funds, it enables arbitrary side-effect triggering in the caller's own contract at no cost.
- **No fund loss** occurs directly, but the integrity of the fee-accounting event stream is compromised.

### Likelihood Explanation
Trivially reachable by any unprivileged transaction sender with no preconditions. No special role, key, or governance access is required. The call costs only gas.

### Recommendation
Add an explicit non-zero guard at the top of both functions:

```solidity
function withdraw(uint128 amount) public override {
    if (amount == 0) revert EntropyErrors.AssertionFailure();
    ...
}

function withdrawAsFeeManager(address provider, uint128 amount) external override {
    if (amount == 0) revert EntropyErrors.AssertionFailure();
    ...
}
```

This mirrors the existing `chainLength == 0` guard in `register`. [5](#0-4) 

### Proof of Concept

```solidity
// Any EOA or contract can execute this with no prior registration:
IEntropy entropy = IEntropy(ENTROPY_ADDRESS);

// withdraw(0): passes `0 >= 0`, emits Withdrawal(msg.sender, msg.sender, 0)
entropy.withdraw(0);

// For a registered fee manager:
// withdrawAsFeeManager(provider, 0): passes balance check, emits Withdrawal(provider, msg.sender, 0)
entropy.withdrawAsFeeManager(registeredProvider, 0);
```

Both calls succeed on-chain, emit `Withdrawal` events with `amount = 0`, and execute `call{value: 0}("")` against the caller. [6](#0-5)
