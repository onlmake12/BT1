### Title
Permanent Fee Loss in Legacy `revealWithCallback` Path When Consumer Callback Permanently Reverts — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `revealWithCallback` function in `Entropy.sol` has two execution paths depending on whether the provider has set a `defaultGasLimit`. In the **legacy path** (`req.gasLimit10k == 0`), the consumer's `_entropyCallback` is invoked without any error catching. If the callback always reverts, the entire `revealWithCallback` transaction reverts, the request remains permanently active, and the user's fee — already credited to the provider and Pyth at request time — is irrecoverably lost. There is no refund mechanism.

---

### Finding Description

When a user calls `requestV2` / `requestWithCallback`, fees are immediately credited to the provider and Pyth inside `requestHelper`:

```solidity
// target_chains/ethereum/contracts/contracts/entropy/Entropy.sol, lines 236-239
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

Fee accounting is final at request time, regardless of whether the callback ever succeeds.

`revealWithCallback` then branches on `req.gasLimit10k`:

**New path (gasLimit10k != 0)** — uses `excessivelySafeCall` to catch errors:
```solidity
// lines 582-596
(success, ret) = req.requester.excessivelySafeCall(
    uint256(req.gasLimit10k) * TEN_THOUSAND,
    256,
    abi.encodeWithSelector(IEntropyConsumer._entropyCallback.selector, ...)
);
```
A reverting callback moves the request to `CALLBACK_FAILED` and allows retry.

**Legacy path (gasLimit10k == 0)** — no error catching:
```solidity
// lines 675-681
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    );
}
```
If `_entropyCallback` reverts, the entire `revealWithCallback` transaction reverts (including the `clearRequest` at line 666), leaving the request active. The provider can retry, but if the callback **always** reverts (e.g., consumer contract is paused, has a bug, or implements any blocking guard), the request can never be fulfilled. The fee is permanently lost with no refund path.

The legacy path is active for any provider that has not called `setDefaultGasLimit`, i.e., `provider.defaultGasLimit == 0`:
```solidity
// lines 268-271
if (providerInfo.defaultGasLimit == 0) {
    // Provider doesn't support the new callback failure state flow
    // Set gasLimit10k to 0 to disable.
```

---

### Impact Explanation

A user who:
1. Pays the entropy fee upfront via `requestWithCallback` / `requestV2` to a provider with `defaultGasLimit == 0`, and
2. Has a consumer contract whose `entropyCallback` reverts (e.g., due to a pause guard, an upgrade, a bug, or any blocking condition)

…will permanently lose their fee. The fee is already credited to the provider and Pyth at request time. There is no `refundRequest` or equivalent function in the Entropy contract. The request remains active forever but can never be cleared.

**Impact**: Direct loss of user funds (entropy fees). Severity: Medium–High depending on fee size and frequency.

---

### Likelihood Explanation

- Any provider that has not called `setDefaultGasLimit` triggers the legacy path. This includes newly registered providers and any provider that explicitly sets `defaultGasLimit` back to 0.
- Any consumer contract that implements a pause mechanism (e.g., `whenNotPaused` on `entropyCallback`), gets upgraded to a broken implementation, or has any revert condition in its callback is affected.
- The user has no way to know at request time whether their callback will revert at reveal time (e.g., if the contract is paused between request and reveal).
- No privileged access is required; any Entropy user can trigger this scenario.

---

### Recommendation

1. Apply `excessivelySafeCall` (or equivalent error catching) in the legacy path of `revealWithCallback`, mirroring the new path. A reverting callback should move the request to a `CALLBACK_FAILED` state rather than reverting the entire transaction.
2. Implement a user-accessible refund mechanism (e.g., `refundRequest`) that allows users to reclaim fees for requests that have been in `CALLBACK_FAILED` state beyond a timeout, or that have never been fulfilled.
3. Alternatively, deprecate the legacy path entirely and require all providers to set `defaultGasLimit` before accepting requests.

---

### Proof of Concept

1. Provider registers without calling `setDefaultGasLimit` → `provider.defaultGasLimit == 0`.
2. User deploys a consumer contract with `whenNotPaused` on `entropyCallback` and calls `requestWithCallback{value: fee}(provider, userRandomNumber)`. Fee is credited immediately.
3. Admin pauses the consumer contract.
4. Provider calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`.
5. Execution reaches the legacy branch (line 661). `clearRequest` is called (line 666). Then `_entropyCallback` is called (line 676), which reverts due to `whenNotPaused`.
6. The entire transaction reverts. `clearRequest` is undone. Request is still active.
7. No matter how many times the provider retries, the call always reverts.
8. The user's fee (already in `providerInfo.accruedFeesInWei` and `_state.accruedPythFeesInWei`) is permanently lost. No refund function exists.
