### Title
`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp Instead of Individual Feed Timestamps, Allowing Arbitrarily Stale Prices to Pass Freshness Validation — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions in `Scheduler.sol` validate freshness by comparing `block.timestamp` against `status.priceLastUpdatedAt`, which is set to the **maximum** `publishTime` across all feeds in the last update batch. For subscriptions containing feeds from different markets (e.g., active crypto feeds alongside closed-market equity feeds), this check passes while individual feed prices may be arbitrarily old — days or weeks stale. Any whitelisted reader (or any reader when whitelist is disabled) calling these functions with a strict `age_seconds` parameter receives a false freshness guarantee.

---

### Finding Description

In `_validateShouldUpdatePrices`, the contract deliberately selects the **maximum** publish time across all feeds as the representative update timestamp:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

This value is then stored as `status.priceLastUpdatedAt`:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [1](#0-0) [2](#0-1) 

The individual feed prices (including stale closed-market prices) are stored separately via `_storePriceUpdates`, preserving their original `publishTime` values.

When a reader calls `getPricesNoOlderThan`, the staleness check is:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [3](#0-2) 

The check passes if the subscription was updated within `age_seconds`, but `getPricesUnsafe` returns the raw stored prices — including individual feeds whose `publishTime` may be days old (e.g., an equity feed last updated during the previous trading session). The same flaw exists in `getEmaPricesNoOlderThan`: [4](#0-3) 

The `IScheduler` interface documents `getPricesNoOlderThan` as: *"Returns the price that is no older than `age` seconds of the current time."* This guarantee is not upheld for individual feeds in mixed-market subscriptions. [5](#0-4) 

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` constant only gates the **maximum** timestamp in incoming update data, not individual feed timestamps: [6](#0-5) 

So a keeper can legitimately push a batch where one active feed has `publishTime = now - 5s` and one closed-market feed has `publishTime = 3 days ago`. The max-timestamp check passes (`now - 5s` is within 1 hour), `priceLastUpdatedAt` is set to `now - 5s`, and the 3-day-old price is stored and later returned through `getPricesNoOlderThan` without triggering a revert.

---

### Impact Explanation

**Severity: High**

A protocol integrating Pyth Pulse for a basket of assets (e.g., crypto + tokenized equities) calls:

```solidity
PythStructs.Price[] memory prices = scheduler.getPricesNoOlderThan(
    subscriptionId, priceIds, 60 // expect all prices ≤ 60 seconds old
);
```

The call succeeds. The protocol receives prices for all feeds, including equity feeds whose `publishTime` is from the last trading session (potentially 16+ hours ago over a weekend). The protocol uses these prices for collateral valuation or liquidation decisions, leading to:

- Incorrect collateral valuations based on stale equity prices
- Missed or incorrect liquidations
- Direct financial loss to the protocol or its users

The function name and interface documentation explicitly promise freshness for all returned prices, making this a silent correctness failure rather than an obvious design trade-off to integrators.

---

### Likelihood Explanation

**Likelihood: Medium**

- Pyth Pulse is designed to support mixed-market subscriptions (the `_validateShouldUpdatePrices` comment explicitly mentions closed markets as a use case).
- DeFi protocols tracking baskets of real-world assets (tokenized stocks, commodities) alongside crypto are a primary target audience for Pulse.
- The vulnerability requires no special permissions: any whitelisted reader (or any reader when `whitelistEnabled = false`) can trigger it by calling `getPricesNoOlderThan`.
- The failure is silent — no revert, no warning — making it likely to go undetected in production integrations.

---

### Recommendation

Replace the subscription-level `priceLastUpdatedAt` check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with per-feed timestamp validation:

```solidity
function getPricesNoOlderThan(
    uint256 subscriptionId,
    bytes32[] calldata priceIds,
    uint256 age_seconds
) external view override onlyWhitelistedReader(subscriptionId) returns (PythStructs.Price[] memory prices) {
    prices = this.getPricesUnsafe(subscriptionId, priceIds);
    for (uint i = 0; i < prices.length; i++) {
        if (distance(block.timestamp, prices[i].publishTime) > age_seconds)
            revert PythErrors.StalePrice();
    }
}
```

Alternatively, if the current behavior (checking subscription update time rather than individual feed timestamps) is intentional, the function documentation and interface must be updated to clearly state that the freshness guarantee applies to the subscription update cycle, not to individual feed `publishTime` values, and integrators must be warned to check individual `publishTime` fields manually.

---

### Proof of Concept

```solidity
// Scenario: subscription with one active crypto feed and one closed equity feed
// Keeper pushes a valid batch update where:
//   - cryptoFeed.publishTime = block.timestamp - 5 seconds  (fresh)
//   - equityFeed.publishTime = block.timestamp - 86400 seconds  (1 day old, closed market)
// max(publishTime) = block.timestamp - 5s → passes PAST_TIMESTAMP_MAX_VALIDITY_PERIOD check
// priceLastUpdatedAt = block.timestamp - 5s

// Reader calls with 60-second age requirement:
PythStructs.Price[] memory prices = scheduler.getPricesNoOlderThan(
    subscriptionId,
    priceIds,  // [cryptoFeedId, equityFeedId]
    60
);
// distance(block.timestamp, priceLastUpdatedAt) = 5 ≤ 60 → NO REVERT
// prices[0].publishTime = block.timestamp - 5s   ✓ fresh
// prices[1].publishTime = block.timestamp - 86400s  ✗ 1 day old — returned without error
```

The `getPricesUnsafe` call returns the raw stored prices with their original `publishTime` values, bypassing any per-feed freshness check. [7](#0-6)
