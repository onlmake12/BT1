### Title
`MAX_DEPOSIT_LIMIT` Not Enforced Cumulatively in `addFunds()` for Permanent Subscriptions — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol` defines `MAX_DEPOSIT_LIMIT = 100 ether` as the maximum balance a permanent subscription may hold. While `createSubscription()` and `addFunds()` both contain a guard, the guard in `addFunds()` checks only the single incoming `msg.value` against the cap — not the resulting cumulative balance. Any subscription manager can therefore call `addFunds()` repeatedly with amounts ≤ 100 ETH to push a permanent subscription's balance arbitrarily above the cap.

---

### Finding Description

`SchedulerConstants.sol` declares:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
```

`createSubscription()` enforces it correctly against the initial deposit:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

`addFunds()` contains an identical but incomplete check:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

status.balanceInWei += msg.value;
```

The guard tests `msg.value > MAX_DEPOSIT_LIMIT`, not `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`. After the initial creation with up to 100 ETH, every subsequent call to `addFunds` with any amount ≤ 100 ETH passes the check and increments `status.balanceInWei` without bound.

Because `withdrawFunds()` unconditionally reverts for permanent subscriptions:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

all deposited ETH is irrecoverably locked in the contract.

---

### Impact Explanation

A subscription manager can lock an unbounded amount of ETH permanently in the Scheduler contract, violating the protocol's explicit 100 ETH cap on permanent subscription balances. Because withdrawals from permanent subscriptions are permanently blocked, every wei deposited above the cap is irreversibly locked. This breaks the protocol invariant that permanent subscriptions are bounded to `MAX_DEPOSIT_LIMIT`, and can result in permanent, unrecoverable loss of user funds beyond the intended ceiling.

---

### Likelihood Explanation

Any subscription manager — a fully unprivileged role — can trigger this by calling `addFunds()` multiple times. No special access, governance majority, or leaked key is required. The call path is direct and requires only ETH. Likelihood is **High**.

---

### Recommendation

Change the check in `addFunds()` to validate the post-deposit cumulative balance:

```solidity
if (params.isPermanent &&
    status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

---

### Proof of Concept

1. Call `createSubscription{value: 100 ether}(params)` with `params.isPermanent = true`. Passes: `100 ether > 100 ether` is false.
2. Call `addFunds{value: 100 ether}(subscriptionId)`. Passes: `100 ether > 100 ether` is false. `status.balanceInWei` is now 200 ETH.
3. Repeat step 2 arbitrarily. Each call passes the guard and increments `status.balanceInWei` by 100 ETH.
4. `withdrawFunds` reverts unconditionally for permanent subscriptions — all deposited ETH above 100 ETH is permanently locked.
