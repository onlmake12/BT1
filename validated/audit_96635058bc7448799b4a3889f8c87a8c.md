### Title
User-Controlled `gasLimit` Exceeding Block Gas Limit Permanently Traps Entropy Requests — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The Pyth Entropy contract allows any user to specify a `gasLimit` up to `uint16.max × 10,000 = 655,350,000` gas when calling `requestV2`. Because this ceiling far exceeds the block gas limit of every deployed EVM chain (e.g., Ethereum ~30 M, Arbitrum ~32 M), a user can create a request whose callback the provider's keeper can **never** successfully execute. Every `revealWithCallback` attempt reverts with `InsufficientGas`, leaving the request permanently stuck in `CALLBACK_NOT_STARTED` and causing the keeper to burn gas on every retry.

### Finding Description
In `Entropy.sol`, `revealWithCallback` executes the callback via `excessivelySafeCall` forwarding `req.gasLimit10k * 10_000` gas:

```solidity
(success, ret) = req.requester.excessivelySafeCall(
    uint256(req.gasLimit10k) * TEN_THOUSAND,   // ← user-supplied, up to 655 M
    256,
    abi.encodeWithSelector(...)
);
```

After the call, the contract decides what happened:

```solidity
if (success) {
    clearRequest(...);
} else if (
    (startingGas * 31) / 32 >
    uint256(req.gasLimit10k) * TEN_THOUSAND   // ← "was enough gas available?"
) {
    req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED;
} else {
    revert EntropyErrors.InsufficientGas();   // ← permanent revert path
}
```

The `else` branch (revert) is reached whenever `startingGas * 31/32 ≤ gasLimit10k * 10_000`. Because `startingGas` is bounded by the chain's block gas limit (≤ 30–32 M on mainnet chains), any request with `gasLimit > ~31 M` will always fall into this branch. The status is reset to `CALLBACK_NOT_STARTED` on line 599 before the revert, so the request is never cleared and never transitions to `CALLBACK_FAILED`. The only way to escape is for the callback to succeed — impossible if the requester contract is designed to revert.

The maximum allowed gas limit is enforced only against `uint16.max × 10_000`:

```solidity
if (gasLimit > uint256(type(uint16).max) * TEN_THOUSAND) {
    revert EntropyErrors.MaxGasLimitExceeded();
}
```

There is no check against the actual block gas limit of the deployed chain.

### Impact Explanation
1. **Permanently stuck requests.** Any request with `gasLimit > block.gaslimit` can never be fulfilled. The random number is never delivered to the requester.
2. **Provider keeper griefing.** The keeper repeatedly submits `revealWithCallback` transactions, each reverting with `InsufficientGas` and consuming the keeper's gas budget. On chains with many such requests, the keeper's operational cost grows unboundedly.
3. **Fee-proportional but cheap on low-fee chains.** The fee scales as `feeInWei × gasLimit / defaultGasLimit`. On chains where providers have set a low `feeInWei` (e.g., 1 wei), setting `gasLimit = 655 M` costs only ~6,550 wei — effectively free — while permanently blocking the request and griefing the keeper.

### Likelihood Explanation
Any unprivileged Entropy user can call `requestV2(uint32 gasLimit)` with `gasLimit` up to 655 M. No special role or key is required. On low-fee chains the economic cost to the attacker is negligible. The provider's keeper has no on-chain protection and must rely on off-chain heuristics to detect and skip such requests, which is not guaranteed.

### Recommendation
Add an on-chain upper bound for `gasLimit` that is configurable per-provider (or per-deployment) and reflects the realistic block gas limit of the target chain. For example:

```solidity
// In requestHelper / roundTo10kGas:
if (gasLimit > providerInfo.maxCallbackGasLimit) {
    revert EntropyErrors.GasLimitExceedsChainMax();
}
```

Alternatively, enforce that `gasLimit ≤ block.gaslimit` at request time, or expose a governance-settable `maxGasLimit` per contract deployment.

### Proof of Concept
1. Provider sets `defaultGasLimit = 100_000` and `feeInWei = 1 wei` on a chain with a 30 M block gas limit.
2. Attacker deploys a contract whose `entropyCallback` always reverts.
3. Attacker calls `requestV2(provider, userRandom, 655_350_000)`, paying `655_350_000 / 100_000 × 1 = 6_550 wei`.
4. Provider keeper calls `revealWithCallback(...)` with all available gas (~30 M).
   - `startingGas ≈ 29_900_000`
   - `excessivelySafeCall` forwards `min(655_350_000, 63/64 × 29_900_000) ≈ 29_430_000` gas.
   - Callback reverts (attacker-controlled).
   - Condition: `29_900_000 × 31/32 = 28_965_625 > 655_350_000`? **FALSE**.
   - `revert EntropyErrors.InsufficientGas()` — keeper tx reverts, request stays `CALLBACK_NOT_STARTED`.
5. Keeper retries indefinitely, burning gas each time. Request is never fulfilled. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    // If gas is not evenly divisible by 10k, rounds up.
    function roundTo10kGas(uint32 gas) internal pure returns (uint16) {
        if (gas > MAX_GAS_LIMIT) {
            revert EntropyErrors.MaxGasLimitExceeded();
        }

        uint32 gas10k = gas / TEN_THOUSAND;
        if (gas10k * TEN_THOUSAND < gas) {
            gas10k += 1;
        }
