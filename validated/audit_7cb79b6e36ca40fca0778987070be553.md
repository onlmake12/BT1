### Title
Provider `defaultGasLimit` Not Enforced at Registration Disables Callback Failure Recovery for Users — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `register()` function in `Entropy.sol` does not require providers to set a `defaultGasLimit`. When `defaultGasLimit` is zero (the Solidity default), the callback failure state flow is silently disabled for every request made to that provider. If a user's callback permanently reverts, the request is permanently stuck with no on-chain recovery path, and the user's paid fee is effectively lost.

---

### Finding Description

**Root cause — `register()` does not enforce `defaultGasLimit`:**

`register()` accepts `feeInWei`, `commitment`, `commitmentMetadata`, `chainLength`, and `uri`, but never touches `defaultGasLimit`. Because Solidity zero-initialises storage, every newly registered provider starts with `defaultGasLimit == 0`. [1](#0-0) 

**Gate in `requestHelper()` — zero `defaultGasLimit` disables the failure-state flow:**

When a user calls any `requestV2` / `requestWithCallback` variant, `requestHelper()` checks the provider's `defaultGasLimit`. If it is zero, `req.gasLimit10k` is set to `0`, opting the request out of the callback failure state flow entirely. [2](#0-1) 

**Consequence in `revealWithCallback()` — no error catching, no `CALLBACK_FAILED` state:**

When `req.gasLimit10k == 0`, `revealWithCallback()` takes the `else` branch: it clears the request from storage first, then calls the callback directly with no `excessivelySafeCall` wrapper and no `CALLBACK_FAILED` transition. If the callback reverts, the whole transaction reverts (restoring the cleared request), but the request remains in `CALLBACK_NOT_STARTED` state forever. Every subsequent retry will also revert, leaving the request permanently stuck. [3](#0-2) 

By contrast, when `req.gasLimit10k != 0`, a failing callback is caught via `excessivelySafeCall`, a `CallbackFailed` event is emitted, and the request transitions to `CALLBACK_FAILED`. The requester can then fix their contract and retry successfully. [4](#0-3) 

---

### Impact Explanation

A user who requests randomness from a provider with `defaultGasLimit == 0` and whose `entropyCallback` permanently reverts (e.g., due to a bug in a non-upgradeable contract, an invariant that is always violated, or a logic error introduced after deployment) has **no on-chain recovery path**:

- The request stays in `CALLBACK_NOT_STARTED` state indefinitely.
- Every call to `revealWithCallback` reverts.
- The provider has already accrued the user's fee; there is no refund mechanism.
- The random number is never delivered; any application state depending on it is permanently blocked.

This is a direct analogue to the BridgeFacet "delegate not enforced" issue: just as a missing delegate leaves a cross-chain transfer permanently stuck when slippage conditions are unfavorable, a missing `defaultGasLimit` leaves an entropy request permanently stuck when the callback is broken.

---

### Likelihood Explanation

The `defaultGasLimit` field is **not part of the `register()` signature** and is not mentioned in the registration flow. Any provider that calls `register()` without subsequently calling `setDefaultGasLimit()` will silently operate in the legacy (no-recovery) mode. The Fortuna keeper's `setup_provider` flow does call `sync_default_gas_limit`, but third-party or permissionless providers have no such guardrail. [5](#0-4) 

Users have no way to know at request time whether their chosen provider has opted into the failure-state flow; `getProviderInfoV2` exposes `defaultGasLimit`, but nothing in the request path warns the user or reverts if it is zero.

---

### Recommendation

1. **Enforce `defaultGasLimit` in `register()`**: Add a `defaultGasLimit` parameter to `register()` and require it to be non-zero, or at minimum emit a warning event when it is zero.
2. **Warn users at request time**: In `requestHelper()`, emit an event or revert (optionally behind a flag) when `providerInfo.defaultGasLimit == 0` and `isRequestWithCallback == true`, so callers are aware they are opting out of the recovery flow.
3. **Document the behavior explicitly**: The NatSpec for `register()` and `requestWithCallback()` / `requestV2()` should clearly state that providers with `defaultGasLimit == 0` disable the callback failure recovery mechanism, and that users requesting from such providers have no recourse if their callback permanently fails.

---

### Proof of Concept

```solidity
// 1. Provider registers without setting defaultGasLimit (defaultGasLimit == 0)
entropy.register(feeInWei, commitment, metadata, chainLength, uri);
// provider.defaultGasLimit == 0 (Solidity default)

// 2. User requests randomness from this provider
uint64 seq = entropy.requestWithCallback{value: fee}(provider, userRandom);
// requestHelper sets req.gasLimit10k = 0 because providerInfo.defaultGasLimit == 0

// 3. User's callback always reverts (e.g., non-upgradeable contract with a bug)
// IEntropyConsumer(req.requester)._entropyCallback(...) always reverts

// 4. Every call to revealWithCallback reverts — request is permanently stuck
entropy.revealWithCallback(provider, seq, userRandom, providerRandom); // REVERTS
// req stays in CALLBACK_NOT_STARTED state, no CALLBACK_FAILED transition,
// no recovery path, user fee is lost, random number never delivered
```

The `gasLimit10k == 0` branch in `revealWithCallback` (lines 661–702) has no `excessivelySafeCall`, no `CALLBACK_FAILED` state transition, and no retry-without-callback escape hatch, confirming the permanent stuck condition. [6](#0-5) [7](#0-6)

        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[

        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.

        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));

            req.callbackStatus == EntropyStatusConstants.CALLBACK_NOT_STARTED

            bool success;
            bytes memory ret;
            uint256 startingGas = gasleft();
            (success, ret) = req.requester.excessivelySafeCall(
                // Warning: the provided gas limit below is only an *upper bound* on the gas provided to the call.
                // At most 63/64ths of the current context's gas will be provided to a call, which may be less
                // than the indicated gas limit. (See CALL opcode docs here https://www.evm.codes/?fork=cancun#f1)
                // Consequently, out-of-gas reverts need to be handled carefully to ensure that the callback
                // was truly provided with a sufficient amount of gas.
                uint256(req.gasLimit10k) * TEN_THOUSAND,
                256, // copy at most 256 bytes of the return value into ret.
                abi.encodeWithSelector(
                    IEntropyConsumer._entropyCallback.selector,
                    sequenceNumber,
                    provider,
                    randomNumber
                )
            );
            uint32 gasUsed = SafeCast.toUint32(startingGas - gasleft());
            // Reset status to not started here in case the transaction reverts.
            req.callbackStatus = EntropyStatusConstants.CALLBACK_NOT_STARTED;

            if (success) {
                emit RevealedWithCallback(
                    EntropyStructConverter.toV1Request(req),
                    userContribution,
                    providerContribution,
                    randomNumber
                );
                emit EntropyEventsV2.Revealed(
                    provider,
                    req.requester,
                    req.sequenceNumber,
                    randomNumber,
                    userContribution,
                    providerContribution,
                    false,
                    ret,
                    SafeCast.toUint32(gasUsed),
                    bytes("")
                );
                clearRequest(provider, sequenceNumber);
            } else if (
                (startingGas * 31) / 32 >
                uint256(req.gasLimit10k) * TEN_THOUSAND

                // The callback reverted for some reason.
                // We don't use ret to condition the behavior here (out-of-gas or other revert), as we have found that some user contracts
                // catch out-of-gas errors and revert with a different error.
                // In this case, ensure that the callback was provided with sufficient gas. Technically, 63/64ths of the startingGas is forwarded,
                // but we're using 31/32 to introduce a margin of safety.
                emit CallbackFailed(
                    provider,
                    req.requester,
                    sequenceNumber,
                    userContribution,
                    providerContribution,
                    randomNumber,
                    ret
                );
                emit EntropyEventsV2.Revealed(
                    provider,
                    req.requester,
                    sequenceNumber,
                    randomNumber,
                    userContribution,
                    providerContribution,
                    true,
                    ret,
                    SafeCast.toUint32(gasUsed),
                    bytes("")
                );
