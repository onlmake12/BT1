### Title
Hard-Coded Magic Numbers `96` and `32` in Consensus-Critical Block Extension Validation — (File: `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

`BlockExtensionVerifier::verify()` enforces consensus-critical block extension size limits using raw numeric literals `96` (maximum extension bytes) and `32` (MMR root hash size), with no named constants and no explanatory comments. The literal `32` appears twice — once as a length guard and once as a byte-slice bound — without a shared constant, creating a concrete inconsistency risk if one instance is updated but not the other. A secondary instance exists in `sync/src/synchronizer/headers_process.rs` where the block version is checked against the raw literal `0` instead of the `BLOCK_VERSION` constant that already exists in `util/types/src/constants.rs`.

---

### Impact Explanation

- **Block extension verifier**: The `96` and `32` values are consensus rules enforced on every block submitted by any peer, miner, or block relayer. If the two instances of `32` (lines 563 and 573) diverge — e.g., one is updated to accommodate a new protocol field but the other is not — nodes would either accept an extension too short to contain a valid MMR hash, or attempt to slice beyond the accepted length, producing an inconsistent validation result. Nodes running different versions of the check would disagree on block validity, causing a **chain split**.
- **Version check**: If `BLOCK_VERSION` is incremented in a future protocol upgrade but the hard-coded `0` in the sync layer is not updated, the sync layer would reject valid headers from upgraded peers independently of the consensus layer, causing a **network partition**.
- **Since timestamp**: If the `1000` multiplier diverges from the actual timestamp unit, timestamp-based `since` locks would be validated against the wrong time, allowing transactions to be accepted before their lock expires or rejected after it has.

---

### Likelihood Explanation

The current values are correct and consistent. The risk materializes during future maintenance. The `32` literal appearing twice without a shared constant is a concrete duplication that increases the probability of a one-sided update. The `96` limit has no comment tracing it to RFC0044, making it opaque to future maintainers. The `BLOCK_VERSION` inconsistency is a latent bug that activates on the next block version increment. All three code paths are reachable by any unprivileged peer, block relayer, transaction sender, or miner without any special access.

---

### Recommendation

1. Define `const MAX_BLOCK_EXTENSION_BYTES: usize = 96;` with a comment referencing RFC0044, and `const MMR_HASH_BYTES: usize = 32;` (or reuse `Byte32::BYTES` = 32) in a shared constants module. Replace all raw literals in `BlockExtensionVerifier::verify()`.
2. Replace `!= 0` in `version_check` with `!= BLOCK_VERSION` (importing from `ckb_types::constants`).
3. Export `MILLISECONDS_IN_A_SECOND` from `spec/src/consensus.rs` (or a shared constants crate) and use it in `Since::extract_metric` instead of the raw `1000`.

---

### Proof of Concept

A block relayer submits a block whose extension field is exactly 32 bytes. Currently, both the length guard (`extension.len() < 32` → false for `len == 32`) and the slice (`slice(..32)`) are consistent, so the block passes. If a future patch changes the length guard to `< 33` (e.g., to reserve space for a new field) but leaves the slice as `..32`, the node accepts a 33-byte extension but reads only 32 bytes of MMR hash — silently ignoring the extra byte and potentially accepting a block with an invalid chain root. Conversely, if the slice is changed to `..33` but the guard is not, a 32-byte extension passes the guard but the slice panics or errors. Either scenario is triggered by any block relayer submitting a block with an extension field, with no privileged access required. [6](#0-5)
