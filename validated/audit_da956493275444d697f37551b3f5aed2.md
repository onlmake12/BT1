Audit Report

## Title
TOCTOU Race in `_process_tx` Admits Transactions With Dead Inputs When Tip Is Unchanged — (`tx-pool/src/process.rs`)

## Summary
`TxPoolService::_process_tx` resolves inputs under a read lock, releases it, runs async script verification without any lock, then finalises admission under a write lock. When the chain tip has not advanced, `submit_entry` skips `check_rtx` entirely, meaning pool-state liveness of inputs is never re-verified. A concurrent RBF replacement that evicts a parent transaction during the unlocked verification window causes the child transaction to be admitted with a permanently unresolvable input, corrupting `edges.inputs` and enabling a repeatable tx-pool DoS.

## Finding Description

**Phase 1 — `pre_check` (read lock, then released):**
`pre_check` calls `resolve_tx` via `PoolCell::cell`, which returns `CellStatus::live_cell` for C2 if TX_A is in the pool's output map. The read lock is released and `tip_hash` is captured. [1](#0-0) [2](#0-1) 

**Phase 2 — `verify_rtx` (no lock):**
Script execution is async and unbounded in duration. During this window a second worker can process TX_A' (RBF of TX_A), which calls `process_rbf` → `remove_entry_and_descendants`, evicting TX_A and its output C2 from the pool entirely. [3](#0-2) 

**Phase 3 — `submit_entry` (write lock):**
The critical guard is:

```rust
if pre_resolve_tip != tip_hash {
    status = check_rtx(tx_pool, &snapshot, &entry.rtx)?;
    ...
    time_relative_verify(...)?;
}
```

When no new block arrived, `pre_resolve_tip == tip_hash`, so `check_rtx` — the only call that re-verifies input liveness against the current pool state — is **never executed**. [4](#0-3) 

The RBF-enabled path calls `check_rbf`, which detects double-spend conflicts in `edges.inputs`, but C2 is no longer in `edges.inputs` (TX_A was removed), so no conflict is found. The non-RBF path calls `find_conflict_outpoint`, which also only checks `edges.inputs`. [5](#0-4) 

`_submit_entry` then calls `edges.insert_input` for C2, registering it as consumed by TX_B even though C2 exists nowhere — not on-chain, not in the pool. [6](#0-5) 

**Why existing checks are insufficient:**
- `find_conflict_outpoint` / `check_rbf` check for double-spend conflicts, not for whether the referenced output actually exists.
- `check_rtx` (which calls `check_rtx_from_pool` and verifies liveness) is gated exclusively on the tip-hash comparison.
- `PoolCell::cell` returns `CellStatus::Unknown` for a missing output, which would cause rejection — but this path is only exercised inside `check_rtx`, which is skipped. [7](#0-6) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

1. TX_B is permanently stuck in the pending pool with an unresolvable input. The block assembler cannot include it; it persists until the expiry timer or `limit_size` evicts it.
2. C2 is recorded in `edges.inputs` as consumed by TX_B. Any subsequent legitimate transaction spending C2 (e.g., after a reorg re-introduces TX_A) is rejected as a double-spend, even though TX_B is invalid.
3. By repeating the TX_A → TX_B → TX_A' pattern, an attacker fills the pool with permanently-stuck invalid entries, evicting legitimate transactions via `limit_size`. This constitutes a sustained, low-cost tx-pool DoS against any node with RBF enabled.

## Likelihood Explanation

- Requires only standard RPC access (`send_transaction`). No privileged role, no hashpower.
- The race window equals the full duration of `verify_rtx` for TX_B. An attacker maximises this by crafting TX_B with a script consuming close to `max_tx_verify_cycles`, while TX_A' uses a trivial always-success script processed almost instantly by a concurrent worker.
- `VerifyMgr` spawns multiple concurrent workers, making the interleaving routine rather than exceptional. [8](#0-7) 
- Cost per injected invalid entry: fees for TX_A + TX_A' only. TX_B is never mined, so its fee is never paid.

## Recommendation

In `submit_entry`, unconditionally re-run `check_rtx` against the current pool state before calling `_submit_entry`, regardless of whether the tip hash changed. The tip-hash guard should only gate the *time-relative* re-verification:

```rust
// Always re-verify input liveness against current pool state
status = check_rtx(tx_pool, &snapshot, &entry.rtx)?;

// Only redo time-relative verify if the chain tip advanced
if pre_resolve_tip != tip_hash {
    let tip_header = snapshot.tip_header();
    let tx_env = status.with_env(tip_header);
    time_relative_verify(snapshot, Arc::clone(&entry.rtx), tx_env)?;
}
```

This ensures that any pool-state change during the unlocked `verify_rtx` window is caught before admission.

## Proof of Concept

1. Attacker holds live on-chain cell C1.
2. Submit TX_A (input: C1 → output: C2, trivial always-success script). TX_A enters pool; C2 is live in `pool_map` outputs.
3. Submit TX_B (input: C2, script consuming ~`max_tx_verify_cycles`). `pre_check` resolves C2 as `Live`; read lock released; `verify_rtx` begins and occupies a worker for several seconds.
4. Submit TX_A' (input: C1, higher fee — valid RBF of TX_A, trivial script). Second worker processes TX_A' quickly. `submit_entry` for TX_A': `check_rbf` finds TX_A as conflict → `process_rbf` → `remove_entry_and_descendants(TX_A)`. TX_A and C2 are gone from `pool_map`. TX_B is still in the verify queue.
5. TX_B's `verify_rtx` completes. `submit_entry` for TX_B: `check_rbf` finds no conflict (C2 not in `edges.inputs`). `pre_resolve_tip == tip_hash` (no new block) → `check_rtx` skipped. `_submit_entry` inserts TX_B; C2 is written into `edges.inputs`.
6. TX_B is now in the pending pool with input C2 that does not exist anywhere. Block assembler silently skips TX_B. Any tx spending C2 is rejected as a double-spend. Repeat steps 2–5 to exhaust pool capacity.
