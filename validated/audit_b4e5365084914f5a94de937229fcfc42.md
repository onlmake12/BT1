Audit Report

## Title
`DownloadScheduler::decrease` Uses `task_count` Instead of `timeout_count` as Base, Bypassing Grace Period and Causing Premature Peer Disconnection - (File: `sync/src/types/mod.rs`)

## Summary
In `sync/src/types/mod.rs`, `DownloadScheduler::decrease` at line 522 assigns `self.timeout_count` using `self.task_count` as the base instead of `self.timeout_count`. Because `task_count` is initialized to 32 and `num` is always 1 or 2, the result always exceeds the threshold of 2, causing `task_count` to be decremented on every slow block response rather than every third one. This eliminates the intended grace period entirely, causing peers to be penalized 3× more aggressively and disconnected after ~32 slow responses instead of ~96.

## Finding Description
`DownloadScheduler` is initialized with `task_count = INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32` and `timeout_count = 0` ( [1](#0-0) ). The `decrease` function is intended to accumulate `timeout_count` and only decrement `task_count` after more than 2 slow responses:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: should be self.timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
``` [2](#0-1) 

Since `task_count` starts at 32, `self.task_count.saturating_add(1)` = 33 and `self.task_count.saturating_add(2)` = 34 — both always exceed 2. The `if` branch fires on every invocation, decrementing `task_count` by 1 and resetting `timeout_count` to 0. The grace period counter never accumulates.

`decrease(1)` or `decrease(2)` is called from `remove_by_block` whenever a block response falls in the `NormalToUpper` or `UpperToMax` time quantile and `should_punish` is true (i.e., `download_schedulers.len() > protect_num`): [3](#0-2) 

When `task_count` reaches 0, `prune` inserts the peer into `disconnect_list` and removes it: [4](#0-3) 

## Impact Explanation
This is a concrete sync performance degradation triggerable by any unprivileged sync peer. A peer experiencing transient latency — or one deliberately responding slowly — causes the victim node to exhaust `task_count` in ~32 slow responses instead of the intended ~96, resulting in premature disconnection and reduced block download capacity. This fits the allowed impact: **Low (501–2000 points) — important performance improvement for CKB**, as it directly degrades sync throughput and causes the adaptive scheduler to behave incorrectly under normal network conditions.

## Likelihood Explanation
The `decrease` path is exercised whenever block response times fall in the upper two quantiles of `TimeAnalyzer`, which occurs under any non-ideal network condition. The bug is active on every production node with `adjustment = true` (the default) and more than `protect_num` sync peers. No special privileges are required — any sync peer that responds slowly, whether due to load or deliberate throttling, triggers this path.

## Recommendation
Change line 522 from:
```rust
self.timeout_count = self.task_count.saturating_add(num);
```
to:
```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```
This restores the intended semantics: `timeout_count` accumulates across calls and `task_count` is only decremented after more than 2 slow responses.

## Proof of Concept
1. Initialize `DownloadScheduler::default()`: `task_count = 32`, `timeout_count = 0`.
2. Call `decrease(1)`: `timeout_count = 32 + 1 = 33 > 2` → `task_count = 31`, `timeout_count = 0`.
3. Call `decrease(1)` again: `timeout_count = 31 + 1 = 32 > 2` → `task_count = 30`, `timeout_count = 0`.
4. Repeat 30 more times: `task_count` reaches 0.
5. `prune` disconnects the peer.
6. Under the correct implementation, step 2 would yield `timeout_count = 0 + 1 = 1 ≤ 2` — no decrement. Three calls are required per decrement, so ~96 slow responses are needed to reach `task_count = 0`.

A unit test can confirm this directly by calling `decrease(1)` 32 times on a default `DownloadScheduler` and asserting `task_count == 0`, then repeating with the fix applied and asserting `task_count` remains 32 after fewer than 3 calls.
