### Title
Unbounded O(T) Work Per Request in `GetTransactionsProofProcess::execute` with No Rate Limiting — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

### Summary

An unprivileged remote peer can send repeated `GetTransactionsProof` P2P messages with up to 1000 tx hashes all belonging to a single large block. Each request causes the server to: (1) load the full block from DB (O(T) deserialization), and (2) call `CBMT::build_merkle_proof` with all T transaction hashes as leaves (O(T) computation). The `LightClientProtocol` has **no rate limiter**, unlike the `Relayer` protocol. This creates a CPU/IO amplification DoS path reachable by any peer.

### Finding Description

**Entrypoint**: `LightClientProtocol::received` in `util/light-client-protocol-server/src/lib.rs` dispatches `GetTransactionsProof` messages to `GetTransactionsProofProcess::execute` with no rate limiting. [1](#0-0) 

The `LightClientProtocol` struct contains no rate limiter field. A grep across the entire `util/light-client-protocol-server/` directory returns zero matches for `rate_limit`, `rate_limiter`, or `RateLimiter`. This contrasts with the `Relayer`, which explicitly applies a 30 req/sec per-peer-per-message-type governor: [2](#0-1) 

**The limit is 1000 tx hashes per request**: [3](#0-2) 

**The O(T) work path**: In `execute`, after grouping found tx hashes by block, for each unique block the code calls `snapshot.get_block(&block_hash)` (loading all T transactions from `COLUMN_BLOCK_BODY`) and then passes **all T transaction hashes** as the first argument to `CBMT::build_merkle_proof`: [4](#0-3) 

`CBMT::build_merkle_proof` (from `merkle-cbt`) requires all T leaves to construct the tree before extracting the proof path. This is O(T) regardless of how many indices (K) are requested. [5](#0-4) 

**Attack scenario**: An attacker mines (or observes) a maximally-full block with T transactions. They send `GetTransactionsProof` with 1000 tx hashes all from that block. The server performs:
- 1000 individual `get_transaction_with_info` DB lookups
- 1 full `get_block` load (all T transactions deserialized)
- 1 `CBMT::build_merkle_proof` call over T leaves

With no rate limiting, the attacker can repeat this continuously from a single connection.

### Impact Explanation

Each request causes server work proportional to block size T (not request size K). A maximally-full block near the `MAX_BLOCK_BYTES` limit can contain thousands of transactions. Continuous requests from one or more peers exhaust server CPU (CBMT tree construction, blake2b hashing) and DB I/O (block body reads), causing CKB network congestion and degraded service for legitimate peers.

### Likelihood Explanation

The attack requires only a P2P connection to a node running the light client protocol server. No PoW, no keys, no privileged access. The attacker only needs to know tx hashes from a large block (observable from the public chain). The absence of any rate limiter in `LightClientProtocol` makes sustained attack trivial.

### Recommendation

1. Add a per-peer rate limiter to `LightClientProtocol` analogous to the one in `Relayer` (e.g., using `governor::RateLimiter`).
2. Consider caching the full block's tx hash vector so repeated requests for the same block do not re-deserialize all transactions.
3. Optionally, cap the number of distinct blocks that can be queried in a single `GetTransactionsProof` request (currently all 1000 hashes could span multiple blocks, each triggering a separate full block load).

### Proof of Concept

1. Mine a block near the block size limit with T transactions (e.g., T = 5000 minimal transactions).
2. Connect to a CKB node with light client protocol enabled.
3. Send `GetTransactionsProof` with `tx_hashes` = 1000 hashes from that block, `last_hash` = current tip.
4. Repeat in a tight loop from a single peer.
5. Observe: server CPU spikes due to repeated O(T) CBMT tree construction; DB I/O spikes due to repeated full block deserialization; legitimate light client requests are delayed or dropped. [6](#0-5) [7](#0-6)
