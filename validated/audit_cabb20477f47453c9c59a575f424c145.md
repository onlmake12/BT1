### Title
Missing Maximum Fee Guard in Entropy Request Functions Allows Silent User Overpayment — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `request()`, `requestWithCallback()`, and `requestV2()` entry points in `Entropy.sol` accept any `msg.value ≥ requiredFee` and permanently capture all excess value as protocol revenue. Because the provider fee is actively adjusted by the Fortuna keeper and can decrease between a user's off-chain `getFee()` query and on-chain execution, users routinely overpay with no refund and no way to bound their maximum payment. No `maxFee` guard exists in any of the three public request paths.

---

### Finding Description

`requestHelper()` enforces only a lower bound on the fee:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

Every wei above `providerFee` is silently swept into `accruedPythFeesInWei`. The code comment on all three public functions explicitly acknowledges this: *"Note that excess value is **not** refunded to the caller."* [2](#0-1) 

The provider fee is not static. `setProviderFee()` and `setProviderFeeAsFeeManager()` allow the provider or its designated fee manager to change `feeInWei` at any time with no time-lock: [3](#0-2) 

More importantly, the Fortuna keeper runs a continuous `adjust_fee_wrapper` loop that calls `set_provider_fee_as_fee_manager` whenever the current fee falls outside a target profit band: [4](#0-3) [5](#0-4) 

The standard user flow recommended in the SDK and documentation is:

```solidity
uint256 fee = entropy.getFeeV2();
uint64 sequenceNumber = entropy.requestV2{ value: fee }();
``` [6](#0-5) 

Between the off-chain `getFeeV2()` call and the on-chain `requestV2()` execution, the Fortuna keeper may lower the fee. The user's transaction still succeeds (because `msg.value ≥ new requiredFee`), but the difference is permanently captured as Pyth protocol revenue. There is no `maxFee` parameter on any of the three public request functions (`request`, `requestWithCallback`, `requestV2`) that would let the user bound their payment.

The same pattern exists in `Echo.sol`'s `requestPriceUpdatesWithCallback()`, where excess value above `pythFeeInWei` is credited to the provider's accrued fees rather than refunded: [7](#0-6) 

---

### Impact Explanation

A user who queries `getFeeV2()` and immediately submits a transaction with exactly that value will silently overpay whenever the Fortuna keeper lowers the fee in the same block or in a pending-mempool race. The overpayment is irrecoverable — it accrues to `accruedPythFeesInWei` and can only be withdrawn by the Pyth admin. For high-frequency consumer contracts that batch many requests, the cumulative loss can be material. The user has no on-chain mechanism to express a maximum acceptable fee.

**Impact: Medium** — direct, unrecoverable ETH loss for users; no protocol invariant is broken, but user funds are silently captured.

---

### Likelihood Explanation

The Fortuna keeper is documented to run a continuous fee-adjustment loop keyed to current gas prices. Fee decreases are routine and expected. Any user who follows the recommended pattern of querying the fee before submitting is exposed on every request where the fee drops between query and inclusion. On chains with variable gas prices (e.g., Ethereum mainnet, Arbitrum), this is a common occurrence.

**Likelihood: Medium** — routine automated fee adjustments make the race condition a normal operating condition, not an edge case.

---

### Recommendation

Add a `maxFee` parameter to `request()`, `requestWithCallback()`, and `requestV2()` (and the analogous `requestPriceUpdatesWithCallback()` in Echo) that reverts if `msg.value > maxFee`. Alternatively, refund any excess above `requiredFee` back to `msg.sender` at the end of `requestHelper()`. Either approach gives users the same slippage protection that the original report recommends for ERC4626 redemption functions.

---

### Proof of Concept

1. Fortuna keeper observes that the current provider fee (e.g., 1000 wei) exceeds `target_fee_max` and calls `setProviderFeeAsFeeManager(provider, 800)` in block N.
2. In the same block N (or block N+1 before the user's tx is included), a user's transaction arrives with `msg.value = 1000` (the value returned by `getFeeV2()` queried in block N-1).
3. `requestHelper()` computes `requiredFee = 800`, passes the `msg.value < requiredFee` check, credits `providerInfo.accruedFeesInWei += 800`, and sweeps the remaining `200 wei` into `_state.accruedPythFeesInWei`.
4. The user's request succeeds, but 200 wei is permanently captured. The user has no way to detect or prevent this outcome because no `maxFee` bound exists.
