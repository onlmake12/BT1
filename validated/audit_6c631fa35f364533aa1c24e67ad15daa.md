### Title
Hardcoded `TRANSACTION_SIZE_LIMIT` Creates Permanent Pool-Rejection Gap for Consensus-Valid Transactions - (File: `util/types/src/core/tx_pool.rs`)

### Summary

A hardcoded, immutable constant `TRANSACTION_SIZE_LIMIT` (512 KB) gates tx-pool admission in CKB. Because the consensus block-size ceiling `MAX_BLOCK_BYTES` is 597 KB, any transaction whose serialized size falls in the 512 KB–597 KB range is **consensus-valid** yet **permanently rejected** by every standard node's tx pool. Cells whose only viable spending transaction falls in that range are effectively unspendable through the normal submission path, analogous to Comet's collateral that cannot be sold once `targetReserves` is exceeded.

---

### Finding Description

`TRANSACTION_SIZE_LIMIT` is declared as a compile-time constant and is never read from configuration: [1](#0-0) 

```rust
/// The maximum size of the tx-pool to accept transactions
/// The ckb consensus does not limit the size of a single transaction,
/// but if the size of the transaction is close to the limit of the block,
/// it may cause the transaction to fail to be packed
pub const TRANSACTION_SIZE_LIMIT: u64 = 512 * 1_000;
```

This constant is applied unconditionally in `non_contextual_verify`, the first gate every submitted transaction must pass: [2](#0-1) 

```rust
let tx_size = tx.data().serialized_size_in_block() as u64;
if tx_size > TRANSACTION_SIZE_LIMIT {
    return Err(Reject::ExceededTransactionSizeLimit(
        tx_size,
        TRANSACTION_SIZE_LIMIT,
    ));
}
```

The consensus block-size ceiling is derived from two separate constants: [3](#0-2) 

```
TWO_IN_TWO_OUT_BYTES = 597
TWO_IN_TWO_OUT_COUNT = 1_000
MAX_BLOCK_BYTES       = 597 × 1_000 = 597,000 bytes
```

The resulting gap:

| Constant | Value |
|---|---|
| `TRANSACTION_SIZE_LIMIT` | 512,000 bytes |
| `MAX_BLOCK_BYTES` (consensus) | 597,000 bytes |
| **Unreachable gap** | **85,000 bytes** |

Any transaction whose serialized size is in `(512_000, 597_000]` bytes is:
- **Consensus-valid** — a miner may include it in a block.
- **Permanently pool-rejected** — every standard node returns `ExceededTransactionSizeLimit` and the rejection is recorded in the recent-reject store. [4](#0-3) 

The rejection is surfaced to RPC callers as `PoolRejectedTransactionBySizeLimit`: [5](#0-4) 

Because `TRANSACTION_SIZE_LIMIT` is a `pub const` baked into the binary, no operator configuration can raise it without recompiling. It is structurally immutable at runtime, exactly like Comet's `targetReserves`.

---

### Impact Explanation

A transaction sender (RPC caller, relay peer) who constructs a transaction whose serialized block size exceeds 512 KB but is below the 597 KB consensus ceiling cannot spend the referenced input cells through the standard submission path (`send_transaction` RPC or P2P relay). The cells remain live on-chain but are effectively locked: the only recourse is to find a miner willing to accept the transaction out-of-band (directly into block assembly), which is not a supported or reliable mechanism for ordinary users.

If the spending transaction is inherently large — for example, a multi-party aggregated signature with a large witness, a script with large cell-dep data, or a batch operation — the user cannot restructure it below the limit without changing the protocol semantics. The capacity locked in those input cells is held in an unwanted state, mirroring the Comet scenario where collateral is held indefinitely because `buyCollateral` cannot be called.

Additionally, if `MAX_BLOCK_BYTES` is raised by a future hardfork (it is a consensus parameter derived from `TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT`), the gap widens automatically while `TRANSACTION_SIZE_LIMIT` stays fixed, making the problem progressively worse without any code change.

---

### Likelihood Explanation

Likelihood is **low-to-moderate**:

- Most transactions today are far below 512 KB, so the gap is rarely encountered in practice.
- However, the gap is not zero: scripts with large witnesses (e.g., zk-proof verification, aggregated multi-sig, large data-carrying cells) can legitimately approach or exceed 512 KB while remaining under 597 KB.
- The constant is immutable at runtime; there is no operator escape hatch.
- The condition is deterministic and reproducible: any transaction in the gap is rejected on every standard node, every time.

---

### Recommendation

1. **Tie `TRANSACTION_SIZE_LIMIT` to `MAX_BLOCK_BYTES` dynamically.** Compute the pool admission limit as `consensus.max_block_bytes() - BLOCK_OVERHEAD_BYTES` (where `BLOCK_OVERHEAD_BYTES` accounts for the block header and cellbase), so the limit tracks the consensus ceiling automatically.

2. **Make the limit configurable.** Expose it in `TxPoolConfig` with a sensible default, similar to how `max_tx_verify_cycles` and `max_ancestors_count` are already configurable. [6](#0-5) 

3. **Document the gap explicitly.** If the intentional headroom (to account for block header + cellbase overhead) is desired, document the exact byte budget and derive the constant from `MAX_BLOCK_BYTES` rather than hard-coding an unrelated value.

---

### Proof of Concept

**Entry path:** RPC caller → `send_transaction` → `non_contextual_verify` → `ExceededTransactionSizeLimit`

1. Construct a transaction whose `serialized_size_in_block()` is, e.g., 513,000 bytes (consensus-valid, under `MAX_BLOCK_BYTES = 597,000`).
2. Submit via `send_transaction` RPC.
3. The node executes `non_contextual_verify` in `tx-pool/src/util.rs:68`:
   ```
   513_000 > 512_000  →  Reject::ExceededTransactionSizeLimit(513_000, 512_000)
   ```
4. RPC returns `PoolRejectedTransactionBySizeLimit (-1108)`.
5. The same rejection occurs on every standard node. The input cells cannot be spent through any pool-based path.
6. A miner who assembles a block directly (bypassing the pool) could include the transaction, but this is not accessible to ordinary users and is not a reliable mechanism.

The root cause — the hardcoded `TRANSACTION_SIZE_LIMIT = 512 * 1_000` constant that is never derived from or compared against the consensus `MAX_BLOCK_BYTES = 597 * 1_000` — is the necessary and sufficient condition for this class of permanent rejection. [7](#0-6) [8](#0-7) [9](#0-8)
