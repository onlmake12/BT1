### Title
Attacker-Controlled Snappy Decompressed-Length Causes Up-to-8 MB Heap Allocation Per P2P Message Before Validation — (File: network/src/compress.rs)

### Summary
The CKB P2P network layer uses Snappy compression for protocol messages. When a compressed message is received, the code reads the claimed decompressed length from the Snappy varint header and immediately allocates a heap buffer of that size — up to `MAX_UNCOMPRESSED_LEN` (8 MB) — before any validation of the actual compressed payload. An unprivileged peer can craft a tiny compressed frame whose Snappy header claims the maximum decompressed size, forcing the node to allocate 8 MB per message. Repeated across many connections or messages, this constitutes a memory-exhaustion DOS.

### Finding Description

In `network/src/compress.rs`, two code paths share the same pattern:

**Path 1 — `Message::decompress()`:** [1](#0-0) 

**Path 2 — `LengthDelimitedCodecWithCompress::decode()`:** [2](#0-1) 

In both paths, `decompress_len(&data[1..])` reads the Snappy varint header — a field entirely controlled by the sender — and the result is used directly to allocate a heap buffer (`vec![0; decompressed_bytes_len]` or `BytesMut::zeroed(decompressed_bytes_len)`). The only guard is:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN { ... reject ... }
```

where `MAX_UNCOMPRESSED_LEN = 1 << 23` (8 MB). [3](#0-2) 

Because the Snappy format stores the uncompressed length as a varint at the very start of the compressed stream, an attacker can craft a byte sequence of the form `[varint(8_388_608)] ++ [garbage]` — a payload of only a handful of bytes — that causes `decompress_len` to return 8 MB. The check `8_388_608 > 8_388_608` is `false`, so the guard passes and 8 MB is allocated. The subsequent `SnapDecoder::decompress` call then fails (because the payload is invalid), the error is returned, and the connection is dropped — but the 8 MB allocation already occurred.

The `max_frame_length` for the Relay and Sync protocols allows frames up to 4 MB and 2 MB respectively: [4](#0-3) 

A compressed frame of a few bytes fits well within these limits, so the frame-length guard provides no protection against this attack.

### Impact Explanation

Each crafted message causes an 8 MB heap allocation on the receiving node. An attacker opening many connections (from different IPs or using Tor/proxies) and sending one crafted message per connection can force the node to allocate hundreds of megabytes or gigabytes of heap memory in a short window, leading to OOM and node crash. Even within a single connection, the attacker can send many such messages before the connection is dropped. This is a remote, unauthenticated denial-of-service against any CKB full node that has compression enabled (the default for all protocols built via `CKBProtocol::new_with_support_protocol`). [5](#0-4) 

### Likelihood Explanation

Any unprivileged P2P peer can connect to a CKB node and send compressed messages. No authentication, key material, or special privilege is required. The crafted payload is trivial to construct: a valid Snappy varint encoding `8_388_608` followed by arbitrary bytes. The attack is reachable through the Sync, Relay, LightClient, and Filter protocols, all of which enable compression by default and accept inbound connections from the public internet.

### Recommendation

1. **Validate the claimed decompressed length against the compressed input size before allocating.** Snappy's theoretical maximum compression ratio is approximately 250:1. A compressed frame of `N` bytes cannot decompress to more than `~250 * N` bytes. Reject messages where `decompress_len > compressed_len * MAX_RATIO` before allocating.
2. **Use a streaming/incremental decompressor** that writes into a pre-sized, bounded output buffer rather than allocating a buffer sized to the attacker-supplied varint.
3. **Lower `MAX_UNCOMPRESSED_LEN`** to match the actual maximum legitimate message size for each protocol (e.g., the protocol's `max_frame_length`), rather than using a single global 8 MB cap that is larger than most protocols' frame limits.

### Proof of Concept

An attacker connects to a CKB node on the Relay protocol and sends the following raw bytes as a length-delimited frame:

```
Frame header (4 bytes): 0x00 0x00 0x00 0x06   (frame length = 6)
Compress flag (1 byte): 0x80                   (COMPRESS_FLAG set)
Snappy payload (5 bytes): 0xFE 0xFF 0xFF 0xFF 0x03  
  (Snappy varint encoding of 8_388_608 = 0x800000, followed by garbage)
```

Upon receipt, `LengthDelimitedCodecWithCompress::decode()` calls `decompress_len` on the 5-byte payload, which returns `8_388_608`. The check `8_388_608 > 8_388_608` is false, so `BytesMut::zeroed(8_388_608)` allocates 8 MB. `SnapDecoder::decompress` then fails, the error is returned, and the connection is closed — but 8 MB was transiently allocated. Repeating this across hundreds of connections exhausts node memory. [6](#0-5)
