### Title
Min-Fee-Rate Check Applied to Serialized Size Instead of Actual Weight, Allowing Cycle-Heavy Transactions to Bypass the Fee Floor - (File: tx-pool/src/util.rs)

### Summary

`check_tx_fee` enforces the `min_fee_rate` threshold using only the transaction's serialized byte size as the weight denominator. However, the actual effective weight used for block-packing priority is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions, this weight can be orders of magnitude larger than `tx_size`, meaning the effective fee rate is far below `min_fee_rate`. Because there is no re-check after cycles are verified, an unprivileged transaction sender can bypass the configured fee rate floor.

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

The code itself acknowledges this is an approximation:

```
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
``` [1](#0-0) 

The actual transaction weight used everywhere else (block packing, fee rate display, eviction) is computed by `get_transaction_weight`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`.

The admission flow in `_process_tx` is:

1. `pre_check` → `check_tx_fee(tx_pool, snapshot, &rtx, tx_size)` — fee checked against `tx_size` only, before cycles are known.
2. `verify_rtx` — actual cycles determined.
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles, **no re-check of fee rate against actual weight**. [3](#0-2) 

**Concrete example** with default `min_fee_rate = 1000 shannons/KW`:

| Parameter | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 70,000,000 (max block cycles) |
| `weight` (actual) | `max(200, 70_000_000 × 0.000_170_571_4)` = **11,940** |
| `min_fee` checked | `1000 × 200 / 1000` = **200 shannons** |
| Effective fee rate | `200 × 1000 / 11,940` ≈ **16.7 shannons/KW** |
| Required minimum | **1000 shannons/KW** |

A transaction paying only 200 shannons passes `check_tx_fee` but has an effective fee rate ~60× below the configured minimum.

### Impact Explanation

An attacker can submit transactions that appear to satisfy `min_fee_rate` (based on size) but whose actual effective fee rate (based on weight) is far below the threshold. Such transactions:

- Are admitted to the tx pool and relayed to peers.
- Consume pool resources (memory, CPU for verification) at a fraction of the intended cost.
- Undermine the fee rate floor that node operators configure to protect against spam.
- Degrade the quality of fee estimation data, since admitted transactions have misleadingly low effective fee rates.

The `min_fee_rate` is the primary economic gate for tx-pool admission. Bypassing it for cycle-heavy transactions breaks the intended resource-pricing guarantee.

### Likelihood Explanation

Any unprivileged transaction sender can exploit this via the `send_transaction` RPC or by relaying a transaction to a peer. Crafting a transaction with high cycle consumption but small serialized size is straightforward: a script that runs a tight computation loop produces exactly this profile. No special privileges, keys, or majority hash power are required.

### Recommendation

After `verify_rtx` returns the actual cycle count, re-check the fee rate against the true weight:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

Alternatively, move the fee rate check into `submit_entry` after cycles are known, replacing the size-only pre-check with a weight-based check.

### Proof of Concept

1. Craft a lock script that consumes ~70,000,000 cycles (e.g., a tight hash loop) but whose serialized transaction size is ~200 bytes.
2. Set the output capacity so that `inputs_capacity - outputs_capacity = 200 shannons` (fee = 200 shannons).
3. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000`.
4. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`; the check passes.
5. After script verification, `verified.cycles ≈ 70,000,000`; actual weight ≈ 11,940; actual fee rate ≈ 16.7 shannons/KW — 60× below `min_fee_rate`.
6. The transaction is admitted to the pool. Observe via `get_transaction` that it is in `pending` state despite its effective fee rate being far below the configured minimum. [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
