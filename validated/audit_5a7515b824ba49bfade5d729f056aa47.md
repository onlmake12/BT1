Audit Report

## Title
`ScriptHashTypeVerifier` Omits `hash_type` Validation for Type Scripts on Transaction Outputs — (File: `verification/src/transaction_verifier.rs`)

## Summary
`ScriptHashTypeVerifier::verify()` validates `hash_type` only on the lock script of each output, never on the optional type script. A transaction with an invalid or non-permitted `hash_type` on a type script passes `NonContextualTransactionVerifier`, enters the tx-pool, and is relayed to peers before being rejected at contextual verification. Because the transaction is never mined, the attacker's input cells are never consumed, enabling unbounded low-cost network spam.

## Finding Description
`ScriptHashTypeVerifier::verify()` at lines 796–814 of `verification/src/transaction_verifier.rs` iterates over outputs and calls only `output.lock().hash_type()`: [1](#0-0) 

`output.type_()` is never consulted. The struct's own doc comment at line 70 confirms the narrowed scope: "Check whether output **lock** hash type within enabled range." [2](#0-1) 

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` permits only `{0, 1, 2, 4}`: [3](#0-2) 

`NonContextualTransactionVerifier::verify()` calls `self.script_hash_type.verify()` as its final gate (line 100), with no other non-contextual check covering type script `hash_type`: [4](#0-3) 

At contextual verification, `extract_script_and_dep_index()` in `script/src/types.rs` does call `ScriptHashType::try_from(script.hash_type())` and returns `ScriptError::InvalidScriptHashType` for unknown or non-permitted values — but only after the transaction has already entered the tx-pool and been relayed: [5](#0-4) 

## Impact Explanation
This matches the High-severity allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** Every full node must perform contextual verification work and consume relay bandwidth for each crafted transaction. Because rejected transactions do not consume inputs, the attacker can reuse the same UTXOs indefinitely, making the attack repeatable at near-zero marginal cost.

## Likelihood Explanation
Exploitation requires only submitting a transaction via the `send_transaction` RPC with `outputs_validator = "passthrough"`, or via the P2P relay protocol. No key material, privilege, or hash-power is needed. The crafted transaction simply sets the `hash_type` byte of any output's type script to a value outside `{0, 1, 2, 4}` (e.g., `0x03`). The asymmetry is directly provable: placing `0x03` on the lock script is immediately rejected by `ScriptHashTypeVerifier`; placing it on the type script is not.

## Recommendation
Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's type script when present, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }
        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

## Proof of Concept
1. Obtain any live UTXO on a CKB testnet node.
2. Build a transaction spending that cell with one output whose lock script uses `hash_type = 0x00` (valid `Data`) and whose type script uses `hash_type = 0x03` (not a valid `ScriptHashType` variant, not in `ENABLED_SCRIPT_HASH_TYPE`).
3. Submit via `send_transaction` RPC with `outputs_validator = "passthrough"`.
4. Observe that `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()` returns `Ok(())` and the transaction enters the pool.
5. Confirm the transaction is subsequently rejected at contextual verification with `ScriptError::InvalidScriptHashType`.
6. Resubmit a slightly varied version (e.g., different witness) using the same input cell — confirm it again passes non-contextual verification, demonstrating zero-cost repeatability.
7. For comparison, place `hash_type = 0x03` on the lock script instead: confirm `ScriptHashTypeVerifier` immediately rejects it with `TransactionError::InvalidScriptHashType`, proving the asymmetric gap.
