Audit Report

## Title
Unbounded `GetBlocksProof` Request Rate with O(k·log(N)) MMR RocksDB Reads Enables Low-Cost DoS on Light Client Servers — (`util/light-client-protocol-server/src/lib.rs`, `util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary

`LightClientProtocol` contains no rate limiter, unlike `Relayer` which explicitly gates all non-PoW messages at 30 req/s per peer. Any peer can repeatedly send `GetBlocksProof` messages containing up to 1000 valid main-chain block hashes, each triggering O(k·log(N)) RocksDB reads via `mmr.gen_proof(items_positions)`. At modest request rates this saturates server I/O and degrades or halts normal chain processing.

## Finding Description

**Struct definition — no rate limiter field:**
`LightClientProtocol` holds only `pub shared: Shared` (lib.rs L26-29). No `rate_limiter` field exists.

**Dispatch path — no rate check:**
`received()` (lib.rs L55-92) deserializes the message and immediately calls `self.try_process(...)`. `try_process` (lib.rs L96-125) dispatches `GetBlocksProof` directly to `GetBlocksProofProcess::execute()` with no interposed rate check.

**Per-request limit — 1000 hashes:**
`GET_BLOCKS_PROOF_LIMIT` is 1000 (constant.rs L5). `execute()` rejects messages exceeding this limit (get_blocks_proof.rs L38-40) but imposes no frequency constraint.

**MMR proof generation — O(k·log(N)) RocksDB reads:**
After validating hashes, `execute()` calls `reply_proof` with up to 1000 leaf positions (get_blocks_proof.rs L104-113). Inside `reply_proof`, `mmr.gen_proof(items_positions)` is invoked (lib.rs L210). The `MMRStore` implementation for `&Snapshot` issues one RocksDB read per MMR node accessed (`get_header_digest`, snapshot/src/lib.rs L293-296). Standard MMR proof generation for k leaves in a tree of height N requires O(k·log(N)) node reads — approximately 23,000 reads per max-size request on a chain of ~10M blocks.

**Contrast with `Relayer`:**
`Relayer::new()` (sync/src/relayer/mod.rs L88-98) installs a `governor`-based rate limiter keyed by `(peer, message_type)` at 30 req/s. `try_process` (relayer/mod.rs L116-123) checks this limiter before any message handling and returns `TooManyRequests` on violation. No equivalent guard exists anywhere in `LightClientProtocol`.

## Impact Explanation

A single attacker peer sending `GetBlocksProof` at 10 req/s with 1000 spread-out valid hashes forces ~230,000 RocksDB reads/second. This saturates I/O on a typical server, starving the chain processing pipeline of storage bandwidth and causing the node to become unresponsive or crash. This matches the allowed High impact: **"Vulnerabilities which could easily crash a CKB node"** and **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation

The attack requires only a valid P2P connection (no PoW, no keys, no privilege). The 1000 required main-chain block hashes are trivially obtained from any public block explorer or by syncing headers. The attack is repeatable indefinitely from a single peer and scales linearly with the number of attacker-controlled peers. The cost to the attacker is negligible network bandwidth; the cost to the server grows with log(chain height).

## Recommendation

Add a per-peer rate limiter to `LightClientProtocol` analogous to the one in `Relayer`:

```rust
pub struct LightClientProtocol {
    pub shared: Shared,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

In `try_process`, check the limiter before dispatching any message, returning `StatusCode::TooManyRequests` on violation. Additionally, consider reducing `GET_BLOCKS_PROOF_LIMIT` (and the analogous `GET_TRANSACTIONS_PROOF_LIMIT` and `GET_LAST_STATE_PROOF_LIMIT`) or adding a global async work queue with backpressure for MMR proof generation.

## Proof of Concept

1. Connect to a mainnet light client server via P2P (protocol id 120, `/ckb/lightclient`).
2. Collect 1000 valid main-chain block hashes spread across the full chain height (e.g., blocks 0, 10000, 20000, …, 9,990,000) from a public block explorer.
3. In a tight loop, send `GetBlocksProof { last_hash: <current tip hash>, block_hashes: [1000 hashes] }`.
4. Benchmark: instrument `mmr.gen_proof` call count and wall time on chains of height 100, 10,000, and 1,000,000 to confirm O(k·log(N)) growth.
5. Observe server RocksDB I/O metrics saturating and chain processing latency increasing monotonically with request rate.
