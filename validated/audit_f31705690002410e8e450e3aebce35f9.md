### Title
Unbounded Linear Scan in `executeCallback` Can Cause Permanent DOS - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` function contains an unbounded `while` loop that linearly scans cleared sequence-number slots to advance `_state.firstUnfulfilledSeq`. Each iteration performs cold storage reads. If a sufficiently large gap of cleared requests accumulates between `firstUnfulfilledSeq` and the next active request, the loop hits the block gas limit, causing every subsequent `executeCallback` call to revert permanently — DOSing the entire Echo contract.

---

### Finding Description

In `Echo.sol`, after clearing a fulfilled request, `executeCallback` runs:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a
// storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs two storage reads:

1. `_state.requests[shortKey]` — warm after the first 32 unique slots (~100 gas)
2. `_state.requestsOverflow[key]` — **cold per unique sequence number** (~2,100 gas, EIP-2929) [2](#0-1) 

The overflow mapping key is `keccak256(abi.encodePacked(sequenceNumber))`, so every distinct sequence number is a unique cold storage slot. At ~2,280 gas per iteration and a 30M gas block limit, approximately **13,000+ cleared requests** between `firstUnfulfilledSeq` and the next active request suffice to trigger OOG.

The code comment on line 166 explicitly acknowledges this gas hazard, confirming the root cause is known but unmitigated.

`firstUnfulfilledSeq` is **only ever advanced** inside this while loop. There is no governance escape hatch to reset it. If the loop OOGs, the variable is permanently stuck, and every future `executeCallback` call — for **any** sequence number — will hit the same OOG revert.

---

### Impact Explanation

If the while loop hits the block gas limit:

- The entire `executeCallback` transaction reverts (including `clearRequest` and fee crediting).
- `firstUnfulfilledSeq` remains stuck at the gap's start.
- Every subsequent `executeCallback` call for any sequence number re-enters the same loop and reverts again.
- All pending Echo requests are permanently unresolvable; user funds locked in request fees cannot be recovered via the normal fulfillment path.

This is a complete, permanent DOS of the Echo contract's core fulfillment mechanism.

**Impact: High**

---

### Likelihood Explanation

An attacker needs to create a large gap of cleared requests ahead of an unfulfilled one at `firstUnfulfilledSeq`. The practical attack path:

1. Attacker submits one request (sequence Y = current `firstUnfulfilledSeq`), paying a single request fee.
2. The attacker's consumer contract is designed so its callback always reverts — but `clearRequest` is called **before** the callback, so the provider still collects fees and the request is cleared regardless.
3. However, the provider is incentivized to fulfill the attacker's request to collect fees, so the attacker cannot easily keep their request alive indefinitely.
4. A more realistic path: the contract accumulates organic traffic; if `firstUnfulfilledSeq` lags behind (e.g., due to a request whose provider goes offline), the gap grows naturally over time.

The ~13,000-iteration threshold makes deliberate exploitation expensive, but the acknowledged TODO comment and the gas benchmark test (`testMultipleRequestsOutOfOrderFulfillment`) confirm the protocol designers are aware this is a real gas hazard.

**Likelihood: Low**

---

### Recommendation

Replace the unbounded linear scan with O(1) bookkeeping. The code comment itself suggests the fix:

> "a better solution would be a doubly-linked list of active requests."

Alternatively, cap the number of iterations the while loop may perform per call (e.g., advance at most N steps), and allow a separate permissionless `advanceFirstUnfulfilledSeq(uint64 steps)` function so the pointer can be advanced incrementally without blocking `executeCallback`.

---

### Proof of Concept

1. Deploy Echo with a provider registered.
2. Create request at sequence number 1 (attacker-controlled consumer that always reverts callback — note: request is still cleared since `clearRequest` precedes the callback).
3. Create and fulfill 13,000+ additional requests (sequences 2–13,001) via normal `executeCallback` calls. Each call clears its request and runs the while loop, but since `firstUnfulfilledSeq = 1` and request 1 is still active, the loop terminates immediately after one check.
4. Now fulfill request 1. `clearRequest(1)` executes. The while loop starts at seq 1 (now cleared) and must scan forward through 13,000 cleared slots to find the next active request (or `currentSequenceNumber`). Each iteration costs ~2,280 gas (cold SLOAD on `requestsOverflow`). Total: ~29.6M gas — exceeds the 30M block gas limit.
5. The transaction reverts. `firstUnfulfilledSeq` remains at 1. Every subsequent `executeCallback` for any sequence number hits the same loop and reverts. Echo is permanently DOSed. [3](#0-2) [2](#0-1) [4](#0-3)

        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        } else {

        }
