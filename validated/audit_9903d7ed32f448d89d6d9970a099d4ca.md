Audit Report

## Title
Missing Rate Limiting on `GetBlockFilterCheckPoints` Enables Peer-Triggered DB Read Exhaustion — (`sync/src/filter/get_block_filter_check_points_process.rs`)

## Summary

The `BlockFilter` protocol handler processes `GetBlockFilterCheckPoints` messages with no per-peer rate limiting, allowing any connected peer to repeatedly trigger up to 4,000 synchronous RocksDB reads per request (2 reads × 2,000 loop iterations). The relayer protocol explicitly implements a keyed rate limiter for the same class of message, but this protection was never applied to the filter protocol, leaving it open to low-cost, high-impact DB read flooding that degrades sync for all peers.

## Finding Description

In `sync/src/filter/get_block_filter_check_points_process.rs`, the constants `BATCH_SIZE = 2000` and `CHECK_POINT_INTERVAL = 2000` are set at lines 9–10. The `execute()` method (lines 43–56) loops up to 2,000 times; each iteration calls `get_block_hash(block_number)` followed by `get_block_filter_hash(&block_hash)` — two synchronous DB reads — for a maximum of 4,000 DB reads per single request. The loop only exits early if a hash is missing; on a fully synced node with all filter data built, all 2,000 iterations complete. [1](#0-0) [2](#0-1) 

The `BlockFilter` handler in `sync/src/filter/mod.rs` dispatches directly to `execute()` with no rate check: [3](#0-2) 

The only protection in `received()` is banning peers that send unparseable bytes (lines 128–143). A well-formed `GetBlockFilterCheckPoints { start_number: 0 }` is never banned and never throttled: [4](#0-3) 

By contrast, `sync/src/relayer/mod.rs` defines a `RateLimiter<(PeerIndex, u32)>` keyed by peer and message type (lines 63–67, 81) and checks it before every non-CompactBlock message (lines 116–123): [5](#0-4) [6](#0-5) 

The `sync/src/filter/` directory contains no equivalent rate-limiting mechanism:


## Impact Explanation

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker opening multiple P2P connections and sending `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop from each connection forces the node to execute hundreds of thousands of RocksDB point-lookups per second from this path alone. This saturates the node's DB read capacity, increases latency for block/header sync DB operations, and causes sync degradation visible to all connected peers — constituting network congestion at very low attacker cost (tiny well-formed messages, no PoW, no stake).

## Likelihood Explanation

The attack requires only a valid P2P connection to a node with the filter protocol enabled. The message is a single `BlockNumber` field, costs the attacker essentially nothing to send repeatedly, and is indistinguishable from a legitimate light client request. No privilege escalation, leaked keys, or victim mistakes are required. The filter protocol is enabled on nodes serving light clients, which is a standard deployment configuration on mainnet.

## Recommendation

1. Add a `RateLimiter<(PeerIndex, u32)>` to the `BlockFilter` struct, mirroring the pattern already used in `Relayer` (`sync/src/relayer/mod.rs` lines 63–82), and check it in `try_process()` before dispatching to `GetBlockFilterCheckPointsProcess::execute()`.
2. Alternatively, reduce `BATCH_SIZE` from 2,000 to a smaller value (e.g., 100–200) to reduce per-request DB cost, though this alone does not prevent flooding.
3. Ban peers that exceed a configurable request-per-second threshold for filter messages, consistent with the relayer's `TooManyRequests` status path.

## Proof of Concept

On a node with filter data built for a mature chain:
1. Open 50 TCP connections to the node's filter protocol port.
2. From each connection, send `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop.
3. Monitor RocksDB read IOPS via `rocksdb.stats` or `/proc` metrics.
4. Simultaneously send `GetHeaders`/`GetBlocks` sync messages from a separate legitimate peer and measure response latency.
5. Expected: DB IOPS spike proportional to the number of concurrent senders; sync message response latency increases measurably, confirming DB read saturation from the filter protocol path.
