### Title
Blocking `thread::sleep` in `NetworkController::try_broadcast` Can Starve Tokio Worker Threads — (`network/src/network.rs`)

### Summary

`NetworkController::try_broadcast` uses a synchronous `thread::sleep` loop that can block for up to 6 seconds. This function is called by the public `broadcast()` and `quick_broadcast()` methods, which are invoked from async relay/sync protocol handlers running on tokio worker threads.

### Finding Description

The constants are defined as: [1](#0-0) 

`P2P_SEND_TIMEOUT` = 6 seconds, `P2P_TRY_SEND_INTERVAL` = 100ms.

The `try_broadcast` function at line 1472 enters a loop: on `WouldBlock` from `filter_broadcast`, it calls `thread::sleep(P2P_TRY_SEND_INTERVAL)` and retries until `P2P_SEND_TIMEOUT` elapses: [2](#0-1) 

This is a **blocking** OS sleep — not `tokio::time::sleep`. The public `broadcast()` and `quick_broadcast()` methods call `try_broadcast` directly: [3](#0-2) 

The relay and synchronizer protocol handlers are `async fn received` / `async fn notify` (confirmed in `sync/src/relayer/mod.rs` and `sync/src/synchronizer/mod.rs`), and they call `.broadcast()`. Calling `thread::sleep` inside a tokio async task blocks the worker thread for the entire sleep duration.

A non-blocking async alternative **does exist** — `broadcast_inner` / `broadcast_with_handle` uses `ServiceAsyncControl` with `.await`: [4](#0-3) 

But the blocking `broadcast()` path is still reachable from async contexts.

### Impact Explanation

A peer that stalls its TCP receive window causes the p2p service's internal send channel to fill, making `filter_broadcast` return `WouldBlock`. The tokio worker thread executing the relay/sync `async fn received` handler then sleeps for up to 6 seconds (60 × 100ms iterations). During this time, all other async tasks scheduled on that worker thread are starved — including sync, ping, and other relay tasks. With multiple attacker peers stalling simultaneously, the effect compounds across worker threads.

### Likelihood Explanation

An attacker only needs a connected TCP session and the ability to stop reading from the socket (zero receive window). No authentication, PoW, or privileged access is required. Block/transaction relay is a normal, frequent code path.

### Recommendation

Replace `thread::sleep(P2P_TRY_SEND_INTERVAL)` with `tokio::time::sleep(...).await` inside an async function, or route all broadcast calls through the existing `broadcast_with_handle` / `broadcast_inner` async path. The non-blocking infrastructure already exists at line 1510–1529.

### Proof of Concept

1. Connect a peer that never reads from its socket (zero TCP receive window).
2. Trigger a block relay broadcast to all peers.
3. Measure tokio task scheduling latency on the worker thread handling the relay `received` callback.
4. Observe latency spikes up to 6 seconds; other async tasks (ping, sync headers) time out during the stall window.
