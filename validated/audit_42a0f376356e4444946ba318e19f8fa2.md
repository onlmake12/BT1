I need to carefully trace the code path to verify whether `calc_ancestors` truly returns an empty set after `remove_entry_links` is called, and whether surviving ancestors are indeed skipped.

The code trace confirms the vulnerability. Here is the exact execution path:

1. `remove_entry_and_descendants` calls `remove_entry_links(id)` for every id in the subtree — including the root — which calls `self.links.remove(id)`, erasing the entry from `TxLinksMap::inner`.
2. Then `remove_entry` is called for each id, which calls `update_ancestors_index_key`.
3. `update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`, which calls `calc_relative_ids`, which does `self.inner.get(short_id)` — returning `None` because the entry was already removed — so `direct` is `unwrap_or_default()` (empty), and the entire ancestor walk returns an empty set.
4. No surviving ancestor ever receives `sub_descendant_weight`, leaving `descendants_fee` permanently inflated.

Audit Report

## Title
Stale `descendants_fee` After Subtree Removal Enables Eviction Bypass and Pool-Fill Attack - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-removes all link entries for the entire subtree before calling `remove_entry` on each. This causes `update_ancestors_index_key` to call `calc_ancestors` on an already-unlinked entry, receiving an empty ancestor set and silently skipping the `sub_descendant_weight` update on every surviving ancestor. An unprivileged attacker can exploit this via repeated RBF replacements to permanently inflate a low-fee parent's `descendants_fee`, making it immune to eviction and enabling pool-filling attacks that cause legitimate transactions to be rejected with `Reject::Full`.

## Finding Description

`remove_entry_and_descendants` first strips all link entries for the entire removed subtree, then calls `remove_entry` for each: [1](#0-0) 

`remove_entry_links` calls `self.links.remove(id)`, which removes the entry from `TxLinksMap::inner`: [2](#0-1) 

Inside `remove_entry`, `update_ancestors_index_key` is called with `EntryOp::Remove`: [3](#0-2) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`: [4](#0-3) 

`calc_ancestors` delegates to `calc_relative_ids`, which does `self.inner.get(short_id)`. Because `remove_entry_links` already called `self.links.remove(id)`, this returns `None`, so `direct` is `unwrap_or_default()` (empty `HashSet`), and `calc_relation_ids` returns an empty set: [5](#0-4) 

No ancestor IDs are iterated, so `sub_descendant_weight` is never called on any surviving ancestor. Their `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` remain permanently inflated.

The `sub_descendant_weight` function uses `saturating_sub`, so any future underflow silently zeros the field rather than panicking, and there is no recompute path for `descendants_fee` analogous to the `recompute_total_stat` fallback used for `total_tx_size`: [6](#0-5) 

The stale `descendants_fee` directly feeds `EvictKey` via `descendants_feerate.max(feerate)`: [7](#0-6) 

`resolve_conflict` — the RBF path — calls `remove_entry_and_descendants`, making this reachable by any tx-pool submitter: [8](#0-7) 

The pool's size-limit eviction loop also calls `remove_entry_and_descendants`, so the same stale accounting affects eviction under memory pressure: [9](#0-8) 

## Impact Explanation

A surviving ancestor transaction retains an inflated `descendants_fee`. Its `EvictKey` is computed as `descendants_feerate.max(feerate)`. With an artificially high `descendants_feerate`, the entry ranks as highly valuable and is never selected for eviction by `next_evict_entry`. An attacker can keep a near-zero-fee parent permanently in the pool, repeatedly inflate its apparent value by submitting and RBF-replacing high-fee children, and fill the pool with such entries, causing legitimate transactions to be rejected with `Reject::Full`. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation

Any node's RPC endpoint accepts `send_transaction` from unprivileged callers. RBF is enabled whenever `min_rbf_rate > min_fee_rate`. The attacker pays incrementally higher fees per replacement (RBF requirement), but each replacement permanently inflates the parent's `descendants_fee` by the replaced child's fee. After N replacements the inflation is `sum(child_fees[0..N-1])`, which grows without bound. No privileged access, key material, or majority hashpower is required.

## Recommendation

Before calling `remove_entry_links` for any entry in the subtree, collect the root's surviving ancestors (those not in `removed_ids`) and apply `sub_descendant_weight` to each of them. Concretely, in `remove_entry_and_descendants`, compute `calc_ancestors(id)` for the root **before** any link removal, filter out IDs that are themselves in `removed_ids`, then iterate over those ancestor IDs and call `sub_descendant_weight` with the root entry's weight. This mirrors the logic already present in `update_ancestors_index_key` but executed before links are torn down.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
