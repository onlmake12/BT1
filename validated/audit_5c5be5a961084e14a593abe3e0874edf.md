### Title
Division by Zero in `calculateTwap()` When Start and End Messages Share the Same `publishSlot` - (File: `target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary
`calculateTwap()` in `Pyth.sol` divides by `slotDiff` (the difference between end and start `publishSlot`) without guarding against the case where both slots are equal (`slotDiff == 0`). The upstream validator `validateTwapPriceInfo()` only rejects `startSlot > endSlot` (strict greater-than), so equal slots pass validation and reach the division, causing an EVM panic revert.

### Finding Description
`validateTwapPriceInfo()` enforces:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

The check uses `>` (strictly greater than), not `>=`. When `startSlot == endSlot`, `slotDiff` is zero and execution proceeds to `calculateTwap()`:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // panic: division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // panic: division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panic: division by zero
``` [2](#0-1) 

All three divisions by `slotDiff` will trigger an EVM panic (Solidity 0.8 checked arithmetic) rather than a meaningful revert with a protocol error code.

The Solana receiver (`pyth-solana-receiver`) correctly uses `end_msg.publish_slot > start_msg.publish_slot` (strict), so `slot_diff` is guaranteed non-zero there:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

However, the EVM contract's `validateTwapPriceInfo` does not enforce strict ordering on `publishSlot`, creating the gap.

### Impact Explanation
Any call to `parseTwapPriceFeedUpdates()` that supplies two Wormhole-attested TWAP messages with identical `publishSlot` values will revert with an opaque EVM panic (division by zero) rather than a descriptive protocol error. This breaks the TWAP price feed for that slot pair. Integrators relying on TWAP for time-sensitive operations (e.g., liquidations, option settlements) will receive an unexpected panic revert with no actionable error code, making it impossible to distinguish a protocol-level rejection from a bug. [4](#0-3) 

### Likelihood Explanation
The scenario is reachable by any unprivileged caller who:
1. Obtains two valid Wormhole-attested TWAP messages (from Hermes) that happen to share the same `publishSlot` — a realistic edge case if the price publisher emits two snapshots within the same slot, or
2. Submits the same valid Wormhole VAA as both the start and end update (the individual-message checks `prevPublishTime < publishTime` pass per-message, and the equal-slot pair passes the `>` guard).

No privileged access or key compromise is required. The caller only needs to pay the TWAP update fee. [5](#0-4) 

### Recommendation
Change the `publishSlot` ordering check in `validateTwapPriceInfo()` from `>` to `>=` so that equal slots are explicitly rejected with a meaningful error before reaching the division:

```solidity
// Before (allows equal slots → division by zero in calculateTwap)
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}

// After (rejects equal slots with a clear error)
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This mirrors the protection already present in the Solana receiver (`end_msg.publish_slot > start_msg.publish_slot`). [1](#0-0)

    function validateTwapPriceInfo(
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure {
        // First validate each individual price's uniqueness
        if (
            twapPriceInfoStart.prevPublishTime >= twapPriceInfoStart.publishTime
        ) {
            revert PythErrors.InvalidTwapUpdateData();
        }
        if (twapPriceInfoEnd.prevPublishTime >= twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateData();
        }

        // Then validate the relationship between the two data points
        if (twapPriceInfoStart.expo != twapPriceInfoEnd.expo) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishTime > twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
    }

```text
    function calculateTwap(
        bytes32 priceId,
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure returns (PythStructs.TwapPriceFeed memory twapPriceFeed) {
        twapPriceFeed.id = priceId;
        twapPriceFeed.startTime = twapPriceInfoStart.publishTime;
        twapPriceFeed.endTime = twapPriceInfoEnd.publishTime;

        // Calculate differences between start and end points for slots and cumulative values
        uint64 slotDiff = twapPriceInfoEnd.publishSlot -
            twapPriceInfoStart.publishSlot;
        int128 priceDiff = twapPriceInfoEnd.cumulativePrice -
            twapPriceInfoStart.cumulativePrice;
        uint128 confDiff = twapPriceInfoEnd.cumulativeConf -
            twapPriceInfoStart.cumulativeConf;

        // Calculate time-weighted average price (TWAP) and confidence by dividing
        // the difference in cumulative values by the number of slots between data points
        int128 twapPrice = priceDiff / int128(uint128(slotDiff));
        uint128 twapConf = confDiff / uint128(slotDiff);

        // The conversion from int128 to int64 is safe because:
        // 1. Individual prices fit within int64 by protocol design
        // 2. TWAP is essentially an average price over time (cumulativePrice₂-cumulativePrice₁)/slotDiff
        // 3. This average must be within the range of individual prices that went into the calculation
        // We use int128 only as an intermediate type to safely handle cumulative sums
        twapPriceFeed.twap.price = int64(twapPrice);
        twapPriceFeed.twap.conf = uint64(twapConf);
        twapPriceFeed.twap.expo = twapPriceInfoStart.expo;
        twapPriceFeed.twap.publishTime = twapPriceInfoEnd.publishTime;

        // Calculate downSlotsRatio as a value between 0 and 1,000,000
        // 0 means no slots were missed, 1,000,000 means all slots were missed
        uint64 totalDownSlots = twapPriceInfoEnd.numDownSlots -
            twapPriceInfoStart.numDownSlots;
        uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff;

        // Safely downcast to uint32 (sufficient for value range 0-1,000,000)
        twapPriceFeed.downSlotsRatio = uint32(downSlotsRatio);

        return twapPriceFeed;
    }
