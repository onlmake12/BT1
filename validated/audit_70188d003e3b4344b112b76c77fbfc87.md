Audit Report

## Title
`PoolMap::add_entry()` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Eviction Snapshot — (File: tx-pool/src/component/pool_map.rs)

## Summary

`PoolMap::add_entry()` snapshots `self.total_tx_size + entry.size` into a local variable before calling `check_and_record_ancestors`, which may evict cell-ref parent transactions and correctly decrement `self.total_tx_size` in-place. Lines 218–219 then unconditionally overwrite those fields with the stale pre-eviction locals, permanently re-inflating the counters by the size and cycles of every evicted transaction. An unprivileged attacker can repeatedly trigger this path to make the pool's accounting diverge arbitrarily from reality, causing over-eviction of valid transactions and rejection of legitimate submissions.

## Finding Description

**Root cause — stale snapshot written after in-place mutation:**

- **Lines 210–211**: `updated_stat_for_add_tx` computes `(self.total_tx_size + entry.size, self.total_tx_cycles + entry.cycles)` into locals. At this moment evicted transactions are still in the pool, so the snapshot includes their contribution.
- **Line 213**: `check_and_record_ancestors` is called. When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (lines 603–625), it calls `remove_entry_and_descendants` in a loop. Each call reaches `update_stat_for_remove_tx` (lines 733–758), which **directly writes** decremented values into `self.total_tx_size` and `self.total_tx_cycles`.
- **Lines 218–219**: `self.total_tx_size = total_tx_size; self.total_tx_cycles = total_tx_cycles;` — the stale locals (which still include the evicted entries' sizes/cycles) overwrite the correctly-decremented fields.

Net effect after one eviction cycle: `self.total_tx_size` is inflated by `Σ size(evicted_tx)` and `self.total_tx_cycles` by `Σ cycles(evicted_tx)`. This inflation is permanent and cumulative.

**Why existing guards are insufficient:**

`update_stat_for_remove_tx` has an underflow guard that falls back to `recompute_total_stat()` (lines 742–756), but this guard only fires on underflow. The overwrite at lines 218–219 happens unconditionally after the correct decrement, so the recompute path is never reached for this bug.

## Impact Explanation

The inflated `total_tx_size` is the authoritative counter used in two critical paths:

1. **`limit_size()` over-eviction**: `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` — an inflated counter causes the loop to evict valid, fee-paying transactions even when the real pool is within its configured limit.
2. **Future admission rejection**: Every subsequent `add_entry` call reads the inflated `self.total_tx_size` as its baseline in `updated_stat_for_add_tx`, causing legitimate transactions to be rejected with `Reject::Full` even when the pool has real capacity.

An attacker can repeat the trigger to inflate the counter without bound, effectively making the node's mempool refuse all new transactions and continuously evict existing ones. This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

The eviction branch in `check_and_record_ancestors` requires: (1) a submitted transaction whose `ancestors_count > max_ancestors_count`, and (2) `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. Both conditions are fully attacker-controlled via standard `send_transaction` RPC. The default `max_ancestors_count` on mainnet is 25, so the attacker needs ~27 transactions per inflation cycle. This is cheap, repeatable, and requires no privileged access or key material.

## Recommendation

Do not snapshot totals before `check_and_record_ancestors`. Instead, use `updated_stat_for_add_tx` only as an overflow pre-check (discarding its return value), and after all evictions have completed, add only the new entry's contribution directly to the post-eviction `self.total_tx_size`/`self.total_tx_cycles`:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Overflow pre-check only; discard the returned snapshot.
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Add new entry's contribution on top of post-eviction totals.
    self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

## Proof of Concept

1. Configure a node with `max_ancestors_count = 5` and a small `max_tx_pool_size`.
2. Submit 5 chained transactions A→B→C→D→E (fills ancestor limit).
3. Submit transaction F that uses transaction C as a **cell dep** (C becomes a cell-ref parent of F).
4. Submit transaction G that spends E's output and also depends on F. G's ancestor count = 6 > 5, but cell-ref parent C can be evicted to bring it to 5 — triggering the eviction branch.
5. Assert `pool_map.total_tx_size == pool_map.recompute_total_stat().0` after the `add_entry` call. The invariant will be violated: `total_tx_size` is inflated by the sizes of C and its descendants (D, E, F — all evicted as descendants of C).
6. Repeat steps 2–4 to inflate the counter further. Eventually all new `send_transaction` calls return `Reject::Full` and `limit_size` continuously evicts valid transactions.
