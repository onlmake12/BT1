Audit Report

## Title
Missing `compact_target` Epoch Difficulty Validation in Compact Block Relay Path Enables Zero-Cost HeaderMap and Disk Flooding — (`sync/src/relayer/compact_block_process.rs`, `verification/src/header_verifier.rs`)

## Summary

The compact block relay path's `HeaderVerifier` omits the epoch `compact_target` check that the contextual block verifier enforces. Because rate limiting is explicitly disabled for `CompactBlock` messages and `compact_target = 0x20ffffff` produces a near-maximum PoW target that any nonce satisfies, an unprivileged peer can flood the node's `HeaderMap` (up to 256 MB in memory, then unbounded on disk via sled), pollute `pending_compact_blocks` indefinitely, and trigger `GetBlockTransactions` amplification — all at zero real computational cost.

## Finding Description

**Rate limiting is explicitly bypassed for `CompactBlock`:**

The comment at `sync/src/relayer/mod.rs` L112–114 reads *"CompactBlock will be verified by POW, it's OK to skip rate limit checking."* This assumption is the root of the problem. [1](#0-0) 

**`compact_to_target(0x20ffffff)` produces a near-maximum target that passes PoW for essentially any nonce:**

From `util/types/src/utilities/difficulty.rs` L62–76, `compact_to_target(0x20ffffff)` yields `(0xffffff << 232, false)` — not zero, not overflow. The test suite confirms `target_to_compact(U256::max_value()) == 0x20ffffff` and `compact_to_difficulty(0x20ffffff) == U256::one()`. [2](#0-1) [3](#0-2) 

In `EaglesongPowEngine::verify`, the check `block_target.is_zero() || overflow` is `false`, so execution reaches `eaglesong_output > block_target`. With `block_target = 0xffffff << 232`, the probability of the hash exceeding this value is ~1/2^24 (~1 in 16 million), meaning virtually any nonce passes. [4](#0-3) 

**The relay-path `HeaderVerifier` does not check `compact_target` against the epoch's expected difficulty:**

`contextual_check` in `compact_block_process.rs` constructs a `HeaderVerifier` and calls `verify`. That verifier runs only four sub-checks: `PowVerifier`, `NumberVerifier`, `EpochVerifier` (continuity only), and `TimestampVerifier`. [5](#0-4) 

The relay-path `EpochVerifier::verify` only checks `is_well_formed` and `is_successor_of` — it never validates the `compact_target` value against the epoch's expected target. [6](#0-5) 

**The real `compact_target` check exists only in the contextual block verifier**, which runs during full block acceptance — after `insert_valid_header` has already mutated state: [7](#0-6) 

**State mutation is unconditional once the relay-path check passes:**

`insert_valid_header` is called immediately after `contextual_check` returns `ok`, inserting the attacker's header into `HeaderMap` and updating `shared_best_header`. [8](#0-7) [9](#0-8) 

**`HeaderMap` memory is capped at 256 MB, but the sled backend is unbounded on disk:**

The 5-second `limit_memory` task spills excess entries to sled, which has no size cap. [10](#0-9) [11](#0-10) 

**`pending_compact_blocks` is also polluted without bound:**

When block reconstruction fails (all attacker blocks, since they carry no real transactions), the compact block is inserted into the unbounded `PendingCompactBlockMap`. The only cleanup path (`retain` by epoch number and `remove` on block acceptance) is never triggered because the attacker never submits a full block. [12](#0-11) [13](#0-12) 

**Network amplification:** For each compact block with missing transactions, the node sends a `GetBlockTransactions` message back to the attacker. [14](#0-13) 

## Impact Explanation

This is a **High** severity issue matching: *"Vulnerabilities which could easily crash a CKB node"* and *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

- **Memory exhaustion**: `HeaderMap` consumes up to 256 MB of RAM from zero-cost headers.
- **Unbounded disk growth**: Sled backend receives writes every 5 seconds with no size cap, growing until disk is full and the node crashes or becomes unresponsive.
- **`pending_compact_blocks` leak**: Each failed reconstruction inserts an entry that is never evicted, holding compact blocks in memory indefinitely.
- **CPU overhead**: Each message triggers PoW hash computation, median-time lookup, and skip-list construction in `insert_valid_header`.
- **Network amplification**: Every attacker compact block elicits a `GetBlockTransactions` response, enabling traffic amplification against the node's bandwidth.

## Likelihood Explanation

The attack requires no hashpower, no privileged access, and no special network position — only a standard P2P connection. Building a chain of headers with `compact_target = 0x20ffffff`, valid epoch succession (`is_successor_of` passes trivially), and monotonically increasing timestamps is trivial. With ~1/2^24 failure probability per nonce, an attacker finds a valid nonce in at most a few attempts on average. Thousands of such headers can be generated and relayed per second. The attack is fully repeatable and requires no victim interaction.

## Recommendation

Add a `compact_target` validation step to `contextual_check` in `sync/src/relayer/compact_block_process.rs` (or to `HeaderVerifier::verify` in `verification/src/header_verifier.rs`) that computes the expected epoch target from the parent's epoch extension and rejects headers whose `compact_target` does not match. This mirrors the check already present in `verification/contextual/src/contextual_block_verifier.rs` `EpochVerifier::verify` at lines 500–507. The fix must execute **before** `insert_valid_header` is called. [7](#0-6) 

## Proof of Concept

1. Connect to a mainnet/testnet node as a standard P2P peer.
2. Starting from genesis, build a chain of headers: for each block N, set `compact_target = 0x20ffffff`, `number = N`, `epoch = valid_successor(parent.epoch)` (satisfying `is_successor_of`), `timestamp = parent.timestamp + 1`, `parent_hash = hash(block_{N-1})`, and any nonce (retry on the ~1/2^24 failure probability).
3. Wrap each header as a `CompactBlock` with only a cellbase prefill (so reconstruction always fails with missing transactions).
4. Send each `CompactBlock` over the relay protocol. No rate limit applies.
5. **Observe**: `HeaderMap` memory grows proportionally to messages sent; sled backend receives writes every 5 seconds and grows without bound; `pending_compact_blocks` accumulates entries; `GetBlockTransactions` messages are sent back to the attacker for each block.
6. The `compact_target` mismatch is only caught in the contextual block verifier — which is never reached because the attacker never submits a full block.

pub fn compact_to_target(compact: u32) -> (U256, bool) {
    let exponent = compact >> 24;
    let mut mantissa = U256::from(compact & 0x00ff_ffff);

    let mut ret;
    if exponent <= 3 {
        mantissa >>= 8 * (3 - exponent);
        ret = mantissa.clone();
    } else {
        ret = mantissa.clone();
        ret <<= 8 * (exponent - 3);
    }

    let overflow = !mantissa.is_zero() && (exponent > 32);
    (ret, overflow)

const INTERVAL: Duration = Duration::from_millis(5000);
const ITEM_BYTES_SIZE: usize = size_of::<HeaderIndexView>();
const WARN_THRESHOLD: usize = ITEM_BYTES_SIZE * 100_000;

impl HeaderMap {
    pub fn new<P>(
        tmpdir: Option<P>,
        memory_limit: usize,
        async_handle: &Handle,
        ibd_finished: Arc<AtomicBool>,
    ) -> Self
    where
        P: AsRef<path::Path>,
    {
        if memory_limit < ITEM_BYTES_SIZE {
            panic!("The limit setting is too low");
        }
        if memory_limit < WARN_THRESHOLD {
            ckb_logger::warn!(
                "The low memory limit setting {} will result in inefficient synchronization",
                memory_limit
            );
        }
        let size_limit = memory_limit / ITEM_BYTES_SIZE;
        let inner = Arc::new(HeaderMapKernel::new(tmpdir, size_limit, ibd_finished));
        let map_weak = Arc::downgrade(&inner);
        let stop_rx: CancellationToken = new_tokio_exit_rx();

        async_handle.spawn(async move {
            let mut interval = tokio::time::interval(INTERVAL);
            interval.set_missed_tick_behavior(MissedTickBehavior::Skip);
            loop {
                tokio::select! {
                    _ = interval.tick() => {
                        if let Some(map) = map_weak.upgrade() {
                            map.limit_memory();
                        } else {
                            debug!("HeaderMap inner was dropped, exiting background task");
                            break;
                        }
                    }
                    _ = stop_rx.cancelled() => {
                        info!("HeaderMap limit_memory received exit signal, exit now");
                        break
                    },
                }
            }
        });
