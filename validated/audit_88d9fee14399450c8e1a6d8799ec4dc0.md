Audit Report

## Title
Fee-Rate Admission Check Uses Byte-Size Instead of Weight, Allowing Cycle-Heavy Transactions to Bypass `min_fee_rate` — (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` enforces `min_fee_rate` using only the serialized byte size of the transaction as the weight denominator, while the canonical weight function is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. No second fee-rate check is performed after `verify_rtx` reveals the actual cycle count. An attacker can submit a compact but cycle-saturated transaction that passes the size-based gate while its true weight-adjusted fee rate is up to ~49× below the configured floor, causing the node to execute full script verification and relay the transaction to peers at a fraction of the intended cost.

## Finding Description
`FeeRate` is defined as shannons per kilo-weight (KW = 1000 weight units). [1](#0-0) 

The canonical weight function is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`: [2](#0-1) 

`DEFAULT_BYTES_PER_CYCLES` is `0.000_170_571_4`: [3](#0-2) 

`TxEntry::fee_rate()` — used for eviction, ordering, and fee-rate reporting — correctly uses this full weight: [4](#0-3) 

However, the admission gate in `check_tx_fee` ignores cycles entirely and passes raw `tx_size` as the weight. The code comment explicitly acknowledges this is theoretically incorrect: [5](#0-4) 

`check_tx_fee` is called inside `pre_check`, before script execution, so the actual cycle count is not yet known: [6](#0-5) 

After `verify_rtx` returns the real cycle count, `TxEntry` is constructed with the verified cycles — but **no second fee-rate check is performed**: [7](#0-6) 

The size-based gate is the only fee-rate check in the entire admission pipeline. On the relay path where `declared_cycles` is `None`, `max_cycles` falls back to `self.consensus.max_block_cycles()` (70M), meaning the node will execute up to 70M cycles with no cycle-aware fee guard whatsoever: [8](#0-7) 

## Impact Explanation
With default config (`min_fee_rate = 1000` shannons/KW, `max_tx_verify_cycles = 70_000_000`), a ~242-byte transaction consuming 70M cycles has:
- True weight: `max(242, 70_000_000 × 0.000_170_571_4) = 11,940`
- Admission min-fee (size-based): `1000 × 242 / 1000 = 242` shannons
- Effective fee rate after admission: `242 × 1000 / 11,940 ≈ 20` shannons/KW — **49× below the configured floor**

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** The attacker pays 242 shannons per transaction while forcing the node (and each peer it relays to) to execute 70M RISC-V cycles. The pool fills with transactions carrying a weight-adjusted fee rate far below the configured floor, displacing legitimate transactions. [9](#0-8) 

## Likelihood Explanation
High. No privileged access is required. Any user who can call `send_transaction` over the public JSON-RPC endpoint, or any peer who can relay a transaction via P2P, can trigger this path. Crafting a compact RISC-V binary that runs a tight loop consuming close to `max_tx_verify_cycles` is straightforward. The attacker needs only a valid UTXO to spend and 242 shannons of fee per transaction. The attack is repeatable with negligible per-transaction cost to the attacker. The relay path is particularly dangerous because `declared_cycles` is `None`, so the node uses the full `max_block_cycles()` as the cycle limit with no declared-cycle mismatch check possible. [10](#0-9) 

## Recommendation
Replace the size-only minimum-fee check in `check_tx_fee` with a weight-based check that accounts for declared cycles. Because cycles are not yet verified at pre-check time, use the caller-declared cycle count (already available as `declared_cycles` in `_process_tx`) as a conservative upper bound. If `declared_cycles` is absent (relay path without declaration), fall back to `max_tx_verify_cycles` as the conservative bound:

```rust
pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
    declared_cycles: Cycle,   // add this parameter
) -> Result<Capacity, Reject> {
    let fee = /* ... as before ... */;
    let weight = get_transaction_weight(tx_size, declared_cycles);
    let min_fee = tx_pool.config.min_fee_rate.fee(weight);
    if fee < min_fee {
        return Err(Reject::LowFeeRate(...));
    }
    Ok(fee)
}
```

This closes the gap between the admission check and the post-admission fee-rate accounting without requiring a second verification pass. [11](#0-10) 

## Proof of Concept
**Setup**: Node running with default config (`min_fee_rate = 1000`, `max_tx_verify_cycles = 70_000_000`).

**Step 1 — Craft the script**: Write a CKB-VM RISC-V program that executes a tight loop consuming exactly 69,999,999 cycles (just under the limit). The binary can be as small as ~100 bytes.

**Step 2 — Build the transaction**: Construct a transaction spending a live cell, attaching the loop script as a type script. The serialized transaction size will be ~242 bytes.

**Step 3 — Set fee to 242 shannons**: This satisfies the size-based check: `1000 × 242 / 1000 = 242`.

**Step 4 — Submit**:
```json
{"method": "send_transaction", "params": [<tx>, null]}
```

**Observed behavior**:
- `check_tx_fee` passes: `fee (242) >= min_fee (242)` ✓
- `verify_rtx` executes 70M cycles of RISC-V — full CPU cost paid by the node
- Transaction admitted with effective fee rate `242 × 1000 / 11940 ≈ 20` shannons/KW — 49× below the configured minimum
- Transaction relayed to all connected peers, each repeating the 70M-cycle verification

**Repeat**: Submit N such transactions. Each costs the attacker 242 shannons; each costs the node (and each peer) ~70M cycles of CPU. The pool fills with transactions that carry a weight-adjusted fee rate far below the configured floor. [9](#0-8) [7](#0-6)

```rust
pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
```
