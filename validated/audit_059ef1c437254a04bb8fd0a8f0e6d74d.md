### Title
`getPricesNoOlderThan` Staleness Check Uses Batch-Maximum Timestamp Instead of Per-Feed Timestamps, Allowing Stale Individual Prices to Bypass Freshness Guarantees — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.sol` contract's `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions check freshness against `status.priceLastUpdatedAt`, which is set to the **maximum** `publishTime` across all feeds in the last update batch. Because `updatePriceFeeds` accepts individual feeds with arbitrarily old timestamps (minimum is `0`) as long as the batch-maximum timestamp is within the 1-hour `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`, an unprivileged updater can submit a valid Pyth batch where one feed is current and another is hours old. The batch-level staleness check passes, but individual feed prices returned to consumers can be far older than the `age_seconds` they requested.

---

### Finding Description

**Root cause — `getPricesNoOlderThan` checks batch-level timestamp, not per-feed timestamps:** [1](#0-0) 

The check `distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds` uses `priceLastUpdatedAt`, which is assigned in `updatePriceFeeds` as the **maximum** `publishTime` across all feeds: [2](#0-1) 

**Root cause — `updatePriceFeeds` accepts feeds with any timestamp ≥ 0:**

The call to `parsePriceFeedUpdatesWithConfig` uses `minAllowedPublishTime = 0`, explicitly allowing individual feeds to carry timestamps from arbitrarily far in the past: [3](#0-2) 

**Root cause — `_validateShouldUpdatePrices` only enforces the maximum timestamp:**

The past-validity check (`PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hour`) is applied only to the batch maximum, not to each individual feed: [4](#0-3) 

**Confirmed by test — a feed with a timestamp 1 hour 5 minutes old passes when the batch maximum is valid:** [5](#0-4) 

The test explicitly shows `priceFeeds[0]` with `stalePublishTime` (outside the 1-hour validity window) and `priceFeeds[1]` with `validPublishTime` (30 minutes ago). The update succeeds, and `priceLastUpdatedAt` is set to `validPublishTime`. A consumer calling `getPricesNoOlderThan(..., 1800)` would receive `priceFeeds[0]`'s price, which is over an hour old, without any revert.

**The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` constant:** [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any protocol that integrates with `Scheduler.getPricesNoOlderThan` or `getEmaPricesNoOlderThan` and relies on the `age_seconds` parameter as a safety bound (e.g., a lending protocol requiring prices no older than 60 seconds) will silently receive individual feed prices that are up to 1 hour old. This breaks the freshness guarantee the function is designed to provide. Downstream effects include:

- Incorrect collateral valuations using stale prices
- Incorrect liquidation triggers (either missed or spurious)
- Incorrect margin/health-factor calculations in derivative protocols

This is a direct analog to the external report: the pull oracle allows a user to select a price update where different feeds within the same batch have very different effective ages, and the per-feed staleness is not enforced at read time.

---

### Likelihood Explanation

The attack requires no privileged access. `updatePriceFeeds` on the Scheduler is permissionless — any address can call it: [8](#0-7) 

A realistic trigger is a subscription containing both a 24/7 crypto feed (e.g., BTC/USD) and a traditional-market feed (e.g., AAPL/USD). When the stock market is closed, the AAPL feed's `publishTime` is from the last trading period (potentially many hours ago). An attacker finds a Pythnet slot where BTC/USD has a current timestamp and AAPL/USD has a stale one, submits it, and the batch passes all checks. Pyth's own best-practices documentation explicitly acknowledges this adversarial selection risk for pull oracles: [9](#0-8) 

---

### Recommendation

Replace the batch-level staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed check. After retrieving the price feeds via `_getPricesInternal`, iterate over each returned feed and verify that `distance(block.timestamp, feed.price.publishTime) <= age_seconds` before returning. This ensures the `age_seconds` guarantee holds for every individual price, not just the batch maximum.

---

### Proof of Concept

1. A subscription is created with two price feeds: `CRYPTO_FEED` (BTC/USD, always active) and `EQUITY_FEED` (AAPL/USD, closed market).
2. The attacker finds a valid Pythnet slot where `CRYPTO_FEED.publishTime = T` (current) and `EQUITY_FEED.publishTime = T - 3500s` (just under 1 hour ago, within `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`).
3. The attacker calls `Scheduler.updatePriceFeeds(subscriptionId, updateData)`.
4. Inside `updatePriceFeeds`, `parsePriceFeedUpdatesWithConfig` accepts both feeds (minAllowedPublishTime = 0). The slot check passes (same Pythnet slot). `_validateShouldUpdatePrices` computes `updateTimestamp = max(T, T-3500) = T`, which is within the 1-hour validity window. The update succeeds and `priceLastUpdatedAt = T`.
5. A consumer protocol calls `scheduler.getPricesNoOlderThan(subscriptionId, [CRYPTO_FEED, EQUITY_FEED], 60)`.
6. The check `distance(block.timestamp, T) > 60` is false (T is current), so no revert.
7. The consumer receives `EQUITY_FEED.price` with `publishTime = T - 3500s` — nearly an hour old — despite requesting prices no older than 60 seconds. [10](#0-9) [11](#0-10)

    function getPricesNoOlderThan(
        uint256 subscriptionId,
        bytes32[] calldata priceIds,
        uint256 age_seconds
    )
        external
        view
        override
        onlyWhitelistedReader(subscriptionId)
        returns (PythStructs.Price[] memory prices)
    {
        SchedulerStructs.SubscriptionStatus memory status = _state
            .subscriptionStatuses[subscriptionId];

        // Use distance (absolute difference) since pythnet timestamps
        // may be slightly ahead of this chain.
        if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
            revert PythErrors.StalePrice();

        prices = this.getPricesUnsafe(subscriptionId, priceIds);
    }

## Adversarial selection

Pull updates give users of Pyth Network some ability to select which price to use in a transaction.
This ability is highly circumscribed by various constraints: on-chain prices must move forward in time and cannot be from too far in the past.
However, users can still choose any price update that satisfies these constraints.
This ability is functionally equivalent to latency: it allows users to see the price in the future before using a price from the past.

The simplest way to guard against this attack vector is to incorporate a **staleness check** to ensure that the price used in a transaction is sufficiently recent.

The Pyth SDK provides the [getPriceNoOlderThan()](https://api-reference.pyth.network/price-feeds/evm/getPriceNoOlderThan) method to help users guard against this attack vector. This method returns the most recent price update that is not older than a specified threshold.
Highly latency-sensitive protocols may wish to reduce the threshold to a few seconds to better suit their needs.
Please also see the section below on [latency mitigations](#latency) for additional ideas on how latency-sensitive protocols can minimize the impact of oracle latency.
