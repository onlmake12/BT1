Audit Report

## Title
`script_exists_in_output` Uses Wrong Row Variable on PostgreSQL, Silently Discarding Type-Script Existence Check During Reorg — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary
At line 252 of `script_exists_in_output`, `row_lock.try_get::<bool, _>(0)` is used instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, `EXISTS` returns a native `BOOLEAN`, so this call succeeds and yields `Ok(false)` — the only value possible at that point since `true` already caused an early return. The `row_type` query result is silently discarded, causing `script_exists_in_output` to always return `false` for the type-script path on PostgreSQL, which leads `rollback_block` to unconditionally delete type scripts from the `script` table even when they are still referenced by live outputs.

## Finding Description
`script_exists_in_output` (line 204) issues two `SELECT EXISTS` queries: one for `lock_script_id` stored in `row_lock` (lines 208–220), and one for `type_script_id` stored in `row_type` (lines 237–249). After the lock query, if the result is `true` the function returns early (lines 225–227). Execution only reaches the type query when `row_lock` is `false`.

The final match block at line 252 reads:

```rust
match row_lock.try_get::<bool, _>(0) {   // BUG: should be row_type
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

On PostgreSQL, `EXISTS` returns `BOOLEAN`, so `row_lock.try_get::<bool, _>(0)` succeeds and returns `Ok(false)` — the only value it can have at this point. `row_type` is fetched from the database but never read. On SQLite, `EXISTS` returns `BIGINT`, so `try_get::<bool, _>(0)` fails, the `Err(_)` branch fires, and `row_type.get::<i64, _>(0)` is correctly consulted. This bug is PostgreSQL-only.

`rollback_block` (lines 28–39) calls `script_exists_in_output` for every type script in the rolled-back block's outputs and pushes all scripts for which it returns `false` into `script_id_list_to_remove`, then deletes them. Because the function always returns `false` for the type-script path on PostgreSQL, every type script from the rolled-back block is deleted from the `script` table regardless of whether it is still referenced by outputs in retained blocks.

## Impact Explanation
This is a concrete corruption of the rich-indexer's `script` table: rows that are still semantically live are deleted. All subsequent `get_cells` / `get_transactions` RPC calls filtering by an affected type script return zero results because the script row no longer exists to join against. This matches **Suboptimal (incorrect) implementation of CKB state storage mechanism** — Medium (2001–10000 points).

## Likelihood Explanation
No attacker capability is required. Reorgs are a normal, unprivileged consensus event triggered by any competing fork of sufficient work. Any PostgreSQL-backed rich-indexer node that experiences a reorg where the rolled-back block contains a type script also present in earlier retained outputs will silently corrupt its index. High-value shared type scripts such as the Nervos DAO script or UDT type scripts are realistic targets; short reorgs (1–2 blocks) occur in normal CKB operation.

## Recommendation
Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // was: row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

## Proof of Concept
1. Start a PostgreSQL-backed rich-indexer node.
2. Index block A containing an output with the DAO type script (lock + type).
3. Index block B (tip) containing another output with the same DAO type script.
4. Trigger a reorg that rolls back block B (e.g., by feeding a competing chain of equal or greater work).
5. `rollback_block` removes block B's outputs, then calls `script_exists_in_output(dao_type_script_id, tx)`.
6. The function fetches `row_type` (EXISTS in remaining outputs → `true`), but at line 252 reads `row_lock` instead, returning `Ok(false)`.
7. The DAO type script id is pushed into `script_id_list_to_remove` and deleted.
8. Query `get_cells` with a type-script filter for the DAO script.
9. **Expected**: DAO deposit cell from block A is returned.
10. **Actual**: Zero results — the script row was deleted.
