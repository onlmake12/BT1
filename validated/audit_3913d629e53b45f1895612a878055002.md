### Title
DoS via Deprecated `.transfer()` Refund in `verifyUpdate` — (File: lazer/contracts/evm/src/PythLazer.sol)

### Summary
`PythLazer.verifyUpdate` uses `payable(msg.sender).transfer(...)` to refund excess ETH to callers. The `.transfer()` function forwards only 2300 gas, which is insufficient for contract callers with non-trivial `receive`/`fallback` functions. This causes the entire `verifyUpdate` call to revert, creating a DoS condition for such callers and preventing them from consuming Lazer price updates.

### Finding Description
In `PythLazer.sol` at line 76, the `verifyUpdate` function refunds excess ETH using:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
```

The `.transfer()` function has a hardcoded 2300 gas stipend. Since EIP-1884 (Istanbul hard fork), many common EVM operations (e.g., `SLOAD`) cost more gas, making 2300 gas insufficient for contract receivers that perform any non-trivial logic in their `receive` or `fallback` functions. When the refund fails, `.transfer()` reverts the entire transaction — including the price verification — rather than silently failing.

This is the direct analog to the reported ERC20 unchecked-return-value class: both stem from improper handling of an external value-transfer call. In the ERC20 case the call silently returns `false`; here the call reverts the entire transaction due to a gas stipend limitation, denying service to the caller.

All other ETH-sending operations in the Pyth codebase correctly use the `.call{value: ...}("")` pattern with explicit return-value checks:
- `Entropy.sol` line 163: `(bool sent, ) = msg.sender.call{value: amount}(""); require(sent, ...);`
- `Scheduler.sol` line 860: `(bool sent, ) = msg.sender.call{value: totalKeeperFee}(""); if (!sent) revert ...;`
- `PythGovernance.sol` line 268: `(bool success, ) = payload.targetAddress.call{value: payload.fee}(""); require(success, ...);`
- `EntropyGovernance.sol` line 112: `(bool success, ) = targetAddress.call{value: amount}(""); require(success, ...);`

`PythLazer.sol` is the sole outlier.

### Impact Explanation
Any contract that calls `verifyUpdate` with `msg.value > verification_fee` and whose `receive`/`fallback` function requires more than 2300 gas will have its call permanently reverted. The contract cannot receive the Lazer price verification result, effectively being DoS'd from using the Lazer price feed service. Funds are returned (transaction reverts), but the price update is never processed, breaking any downstream logic that depends on it.

### Likelihood Explanation
Moderate. Many DeFi protocols use patterns like `msg.value >= fee` and send slightly more ETH than required to avoid exact-amount failures. Contracts with proxy patterns, storage-writing `receive` functions, or event-emitting fallbacks commonly exceed 2300 gas. Any such contract caller that overpays is affected. The caller cannot easily detect this failure mode without testing, and the workaround (sending exactly `verification_fee`) is non-obvious.

### Recommendation
Replace `.transfer()` with `.call{value: ...}("")` and check the return value, consistent with every other ETH-sending site in the codebase:

```solidity
if (msg.value > verification_fee) {
    (bool sent, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(sent, "Refund failed");
}
```

### Proof of Concept
1. Deploy a `Consumer` contract whose `receive()` function writes to a storage variable (costs >2300 gas, e.g., `counter++`).
2. `Consumer` calls `PythLazer.verifyUpdate{value: verification_fee + 1 wei}(validUpdate)`.
3. The signature verification logic at lines 79–105 succeeds.
4. The refund of 1 wei via `.transfer()` at line 76 fails because `Consumer.receive()` exceeds the 2300 gas stipend.
5. The entire transaction reverts; `Consumer` never receives `payload` or `signer`.
