### Title
`PythAggregatorV3::latestRoundData` and `latestAnswer` Do Not Check for Stale Prices - (File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink `AggregatorV3Interface`-compatible adapter, explicitly recommended for protocols migrating from Chainlink. Every price-returning function in the contract calls `pyth.getPriceUnsafe(priceId)`, which returns a price from arbitrarily far in the past with no staleness check. Any protocol that uses this adapter as a drop-in Chainlink replacement will silently consume stale prices.

---

### Finding Description

`PythAggregatorV3` implements the Chainlink `AggregatorV3Interface` and is the official migration path for Chainlink integrators. The contract's `latestRoundData()`, `latestAnswer()`, `getRoundData()`, and `decimals()` functions all call `pyth.getPriceUnsafe(priceId)`:

```solidity
// latestRoundData — line 110
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// latestAnswer — line 54
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// getRoundData — line 89
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
```

`getPriceUnsafe` is explicitly documented in `IPyth.sol` as returning "the most recent price update in this contract **without any recency checks**" and that "the returned price update may be arbitrarily far in the past." The safe alternative, `getPriceNoOlderThan(id, age)`, is available in `AbstractPyth.sol` and enforces a `publishTime` age bound, reverting with `StalePrice` if exceeded. `PythAggregatorV3` never calls it.

There is no configurable `maxAge` parameter, no heartbeat threshold, and no revert path for stale data anywhere in the contract.

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a Chainlink-compatible oracle (its stated purpose per the migration guide) will receive prices from arbitrarily far in the past without any on-chain revert or warning. If the Pyth on-chain price becomes stale — due to a keeper outage, a market closure, or a network disruption — the adapter will silently return the last cached price. Downstream protocols (lending, perpetuals, liquidation engines) will execute against incorrect prices, enabling:

- Borrowing against overvalued collateral using a stale high price
- Avoiding liquidation using a stale price that has not reflected a real-world drop
- Liquidating positions at incorrect prices

This is a direct loss-of-funds impact for users of any protocol that integrates `PythAggregatorV3`.

---

### Likelihood Explanation

The contract is the official, actively-promoted migration path for Chainlink integrators, documented at `apps/developer-hub/content/docs/price-feeds/core/migrate-an-app-to-pyth/chainlink.mdx`. Protocols that follow this guide will deploy `PythAggregatorV3` and call `latestRoundData()` — the standard Chainlink pattern — without realizing there is no staleness guard. The staleness condition itself (keeper down, market closed) is a realistic operational scenario explicitly acknowledged in Pyth's own best-practices documentation.

---

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`. Add a configurable `maxAge` parameter to the constructor (or a setter) so deployers can specify the acceptable staleness threshold per feed, analogous to Chainlink's per-feed heartbeat. For example:

```solidity
uint256 public maxAge;

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    ...
}
```

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a price feed ID.
2. Allow the on-chain Pyth price to go stale (e.g., stop calling `updatePriceFeeds` for longer than the feed's heartbeat).
3. Call `latestRoundData()` on the adapter. It returns the stale price with no revert.
4. A lending protocol using this adapter will accept the stale price for collateral valuation, allowing a user to borrow against collateral whose real value has dropped significantly.
