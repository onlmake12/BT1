### Title
Fortuna Entropy Provider `reveal_delay_blocks` Lacks Minimum Validation, Enabling Randomness Manipulation via Immediate Reveal - (File: `apps/fortuna/src/config.rs`)

---

### Summary

The `Config::load()` function in Fortuna does not validate that `reveal_delay_blocks` is set to a safe minimum value per chain. The official sample configuration explicitly ships with `reveal_delay_blocks: 0`, and `confirmed_block_status` defaults to `BlockStatus::Latest` (the least safe option). A provider running with these settings reveals its hash-chain commitment immediately after a request lands on the latest (unfinalized) block, enabling provider-side censorship attacks and block-reorg-based randomness manipulation for users of the Entropy protocol.

---

### Finding Description

**Root cause — no lower-bound validation on `reveal_delay_blocks`:**

`EthereumConfig.reveal_delay_blocks` is typed as `BlockNumber` (a plain `u64`) with no minimum constraint. [1](#0-0) 

`Config::load()` validates profit-percentage ordering and replica settings, but performs **no check** on `reveal_delay_blocks`: [2](#0-1) 

**Root cause — `confirmed_block_status` defaults to `Latest`:**

`BlockStatus` derives `Default` as `Latest`, so omitting the field from the config silently selects the least-safe block finality level: [3](#0-2) 

The field is deserialized with `#[serde(default)]`, meaning any config that omits it gets `Latest`: [4](#0-3) 

**The sample config ships with `reveal_delay_blocks: 0`:**

The canonical example configuration that operators copy to get started explicitly sets the delay to zero: [5](#0-4) 

**How the delay is enforced at reveal time:**

The revelation endpoint gates the provider's response on `reveal_delay_blocks`. With the value at 0, `saturating_sub(0) >= r.block_number` is always true for any confirmed block, so the provider reveals immediately: [6](#0-5) 

The keeper's `get_latest_safe_block` also subtracts `reveal_delay_blocks` from the confirmed block number, returning the same block when the delay is 0: [7](#0-6) 

**Why the delay matters — documented security property:**

The Pyth documentation explicitly states the reveal delay exists to prevent block-reorg manipulation of randomness outcomes: [8](#0-7) 

The protocol design acknowledges that providers can compute the random number `r = hash(x_i, x_U)` off-chain before revealing, enabling a censorship attack if they reveal too early: [9](#0-8) 

---

### Impact Explanation

With `reveal_delay_blocks = 0` and `confirmed_block_status = Latest`:

1. **Provider censorship / selective non-reveal**: The provider knows its entire hash chain in advance. After a user submits their contribution `x_U` on-chain, the provider can immediately compute `r = hash(x_i, x_U)`. If the outcome is unfavorable (e.g., the user wins a lottery), the provider can withhold the reveal, causing the request to stall. This directly harms users of any application built on Entropy (lotteries, NFT minting, gaming, DeFi).

2. **Block-reorg randomness prediction**: With zero delay on `Latest` blocks, the provider reveals before the request block is finalized. If the block is reorganized, the provider has already published its hash-chain value `x_i` for sequence number `S`. In the reorganized chain, sequence number `S` may be assigned to a different user with a different `x_U`. Because `x_i` is now public, anyone can precompute the random number for that reorganized request, making the outcome fully predictable and exploitable.

The on-chain `revealHelper` in the Entropy contract incorporates the provider contribution into the final random number, so a known provider contribution directly determines the output: [10](#0-9) 

---

### Likelihood Explanation

- The official `config.sample.yaml` ships with `reveal_delay_blocks: 0`. Any operator who copies this file and does not consciously change the value runs with zero delay.
- `confirmed_block_status` defaults to `Latest` silently; operators who omit it get the least-safe setting.
- `Config::load()` validates other safety-critical parameters (replica config, profit bounds) but not `reveal_delay_blocks`, so there is no startup-time warning or rejection.
- The Entropy provider role is permissionless — any address can register. A malicious actor can deliberately deploy a provider with these settings to manipulate outcomes for users who choose that provider.

---

### Recommendation

1. **Add a minimum validation in `Config::load()`** for `reveal_delay_blocks` per chain. Refuse to start (or emit a prominent warning) when the value is 0 and `confirmed_block_status` is `Latest`.
2. **Change the sample config** to use a safe non-zero default (e.g., `reveal_delay_blocks: 1` or higher, matching the mainnet values documented in the chainlist).
3. **Change the default for `confirmed_block_status`** to `Safe` or `Finalized` rather than `Latest`, or require it to be explicitly set.
4. **Document per-chain safe minimums** for `reveal_delay_blocks` analogous to how the chainlist page documents mainnet reveal delays.

---

### Proof of Concept

1. Deploy a Fortuna provider using `config.sample.yaml` verbatim (`reveal_delay_blocks: 0`, `confirmed_block_status` omitted → defaults to `Latest`).
2. A user calls `requestWithCallback(provider, x_U)` on the Entropy contract. The request is assigned sequence number `S` and stored with `blockNumber = N`.
3. Fortuna's keeper immediately processes block `N` (no delay). The revelation endpoint returns the provider's hash-chain value `x_i` for sequence `S` without waiting for any confirmations.
4. **Censorship path**: The provider computes `r = hash(x_i, x_U)` off-chain. If `r` is unfavorable (e.g., user wins), the provider's keeper simply does not submit the `revealWithCallback` transaction. The user's request stalls indefinitely.
5. **Reorg path**: On a chain with shallow reorgs (e.g., a PoW chain or a chain with fast finality but `Latest` blocks), block `N` is reorganized. The provider has already published `x_i`. A new request in the reorganized block gets sequence `S` with a different `x_U'`. Because `x_i` is public, the attacker precomputes `r' = hash(x_i, x_U')` and exploits the known outcome in the same block.

        // Open and read the YAML file

pub async fn get_latest_safe_block(chain_state: &BlockchainState) -> BlockNumber {
    loop {
        match chain_state
            .contract
            .get_block_number(chain_state.confirmed_block_status)
            .await
        {
            Ok(latest_confirmed_block) => {
                tracing::info!(
                    "Fetched latest safe block {}",
                    latest_confirmed_block - chain_state.reveal_delay_blocks
                );
                return latest_confirmed_block - chain_state.reveal_delay_blocks;
            }
            Err(e) => {
                tracing::error!("Error while getting block number. error: {:?}", e);
                time::sleep(RETRY_INTERVAL).await;
            }
        }
    }
}

This flow is secure as long as several trust assumptions hold:

- Providers are trusted to reveal their random number $$(x_i)$$ regardless of what the final result $$(r)$$ is. Providers can compute $$(r)$$ off-chain before they reveal $$(x_i)$$, which permits a censorship attack.
- Providers are trusted not to front-run user transactions (via the mempool or colluding with the validator). Providers who observe user transactions can manipulate the result by inserting additional reuests or rotating their commitment.
- Providers are trusted not to keep their hash chain a secret. Anyone with the hash chain can predict the result of a randomness request before it is requested,
  and therefore manipulate the result. This applies both to users of the protocol as well as blockchain validators who can use this information to manipulate the on-chain PRNG or reorder user transactions.

    function revealHelper(
        EntropyStructsV2.Request storage req,
        bytes32 userContribution,
        bytes32 providerContribution
    ) internal returns (bytes32 randomNumber, bytes32 blockHash) {
        bytes32 providerCommitment = constructProviderCommitment(
            req.numHashes,
            providerContribution
        );
        bytes32 userCommitment = constructUserCommitment(userContribution);
        if (
            keccak256(bytes.concat(userCommitment, providerCommitment)) !=
            req.commitment
        ) revert EntropyErrors.IncorrectRevelation();

        blockHash = bytes32(uint256(0));
        if (req.useBlockhash) {
            bytes32 _blockHash = blockhash(req.blockNumber);

            // The `blockhash` function will return zero if the req.blockNumber is equal to the current
            // block number, or if it is not within the 256 most recent blocks. This allows the user to
            // select between two random numbers by executing the reveal function in the same block as the
            // request, or after 256 blocks. This gives each user two chances to get a favorable result on
            // each request.
            // Revert this transaction for when the blockHash is 0;
            if (_blockHash == bytes32(uint256(0)))
                revert EntropyErrors.BlockhashUnavailable();

            blockHash = _blockHash;
        }

        randomNumber = combineRandomValues(
            userContribution,
            providerContribution,
            blockHash
        );

        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            req.provider
        ];
        if (providerInfo.currentCommitmentSequenceNumber < req.sequenceNumber) {
            providerInfo.currentCommitmentSequenceNumber = req.sequenceNumber;
            providerInfo.currentCommitment = providerContribution;
        }
    }
