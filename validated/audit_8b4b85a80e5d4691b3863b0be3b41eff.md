### Title
No Refund of Excess ETH and No Max-Fee Protection in Entropy `requestHelper` — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Pyth Entropy contract's `requestHelper` function absorbs all excess `msg.value` into `accruedPythFeesInWei` without refunding the caller. Combined with the Fortuna keeper's continuous on-chain fee adjustments, users have no mechanism to specify a maximum acceptable fee, and any ETH sent above the exact required fee is permanently lost.

### Finding Description

In `requestHelper`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value - providerFee` is credited to `accruedPythFeesInWei`. If `msg.value` exceeds `requiredFee`, the surplus is permanently absorbed by the Pyth protocol fee pool — it is never returned to the caller. The `IEntropyV2` interface explicitly documents this: *"excess value is not refunded to the caller."* [2](#0-1) 

The Fortuna keeper (`apps/fortuna/src/keeper/fee.rs`) continuously calls `setProviderFeeAsFeeManager` to adjust the on-chain fee based on current gas prices: [3](#0-2) 

This creates a race condition: a user calls `getFeeV2()` to read the current fee, then submits `requestV2{value: fee}()`. If the Fortuna keeper increases the fee in the same block or before the user's transaction executes, the user's transaction reverts with `InsufficientFee`. To avoid this, users and integrating contracts defensively send excess ETH (e.g., `msg.value = getFeeV2() * 1.5`). That excess is permanently lost.

No variant of `requestV2` accepts a `maxFee` parameter: [4](#0-3) 

### Impact Explanation

- **Direct ETH loss**: Any ETH sent above the exact `requiredFee` at execution time is permanently absorbed into `accruedPythFeesInWei` and cannot be recovered by the user.
- **Forced reverts**: Users who send exactly `getFeeV2()` may have their transaction revert if the fee increases between their read and execution, requiring repeated retries.
- **No protection mechanism**: There is no `maxFee` parameter allowing users to bound their acceptable fee, unlike the Deriverse fix pattern.

### Likelihood Explanation

The Fortuna keeper runs in a continuous loop polling for fee adjustments: [5](#0-4) 

On chains with volatile gas prices (e.g., Ethereum mainnet), fee updates are frequent. Integrating contracts that defensively overpay to avoid reverts will silently lose ETH on every request. This is a normal operational condition, not an edge case.

### Recommendation

1. Refund excess `msg.value` to the caller in `requestHelper`:
   ```solidity
   uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
   if (excess > 0) {
       (bool sent,) = msg.sender.call{value: excess}("");
       require(sent, "refund failed");
   }
   ```
2. Alternatively, add a `maxFee` parameter to `requestV2` variants so callers can specify their acceptable fee ceiling and revert cleanly if the fee exceeds it.

### Proof of Concept

1. User calls `getFeeV2()` off-chain → returns `1000 wei`.
2. User submits `requestV2{value: 1500 wei}()` with a 50% buffer to avoid `InsufficientFee` reverts.
3. Transaction executes; `requiredFee` at execution time is still `1000 wei`.
4. `accruedPythFeesInWei += (1500 - providerFee)` — the 500 wei excess is absorbed by the protocol.
5. User permanently loses 500 wei with no recourse.

Alternatively:
1. User calls `getFeeV2()` → `1000 wei`.
2. Fortuna keeper calls `setProviderFeeAsFeeManager(provider, 1200)` in the same block.
3. User's `requestV2{value: 1000 wei}()` reverts with `InsufficientFee`.
4. User must retry, paying gas twice, with no guarantee the fee won't change again. [6](#0-5) [7](#0-6)

    // Set provider fee. It will revert if provider is not registered.
    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
