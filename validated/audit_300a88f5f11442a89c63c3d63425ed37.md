Audit Report

## Title
Stale Tx-Pool Snapshot Allows Dead-Cell Transactions to Bypass Admission Validation — (`tx-pool/src/pool.rs`, `tx-pool/src/process.rs`)

## Summary
The `TxPool` snapshot is updated asynchronously via `try_send` on a bounded channel; if the channel is full, the notification is silently dropped and the snapshot remains stale. During this window, a transaction spending a cell already consumed in a committed block resolves as live against the stale snapshot and is unconditionally admitted to the mempool, because the only re-validation guard in `submit_entry` is conditional on the snapshot tip having changed between `pre_check` and `submit_entry` — a condition that is false when both reads see the same stale snapshot.

## Finding Description

**Stale snapshot root cause:**
`TxPool.snapshot()` is explicitly documented as potentially inconsistent with the chain. [1](#0-0) 

The snapshot is updated only when a reorg notification is processed. That notification is sent via `try_send` on a channel of capacity `DEFAULT_CHANNEL_SIZE = 512`. [2](#0-1) [3](#0-2) 

If the channel is full, `try_send` returns an error (only logged) and the snapshot stays stale across an arbitrary number of blocks.

**Exploit path through `_process_tx`:**

1. `pre_check` acquires a read lock and calls `resolve_tx`, which internally calls `self.snapshot()` — the stale snapshot. A cell spent in a recently committed block appears `Live` to `OverlayCellProvider`. [4](#0-3) 
The `tip_hash` captured here is from the stale snapshot (e.g., height H). [5](#0-4) 

2. The dead-cell guard at `pre_check` lines 292–303 only fires when `resolve_tx` returns `OutPointError::Dead`. Since the cell appears `Live` in the stale snapshot, this branch is never reached. [6](#0-5) 

3. `verify_rtx` runs with the stale snapshot's data loader and stale `tip_header` — passes. [7](#0-6) 

4. `submit_entry` acquires the write lock and reads the snapshot. If the snapshot has **not** been updated (still at H), then `snapshot.tip_hash() == pre_resolve_tip` (both H). The guard at line 120 is **not entered**, `check_rtx` is skipped, and `_submit_entry` inserts the transaction unconditionally. [8](#0-7) 

The `check_rtx_from_pool` function (which uses `OverlayCellChecker` against the current snapshot) is the correct re-validation tool but is only invoked inside the conditional block that is bypassed. [9](#0-8) 

## Impact Explanation

Invalid transactions admitted to the mempool are relayed to connected peers via the P2P relay protocol. Peers running the same code share the same stale-snapshot window and may also admit the transaction, propagating the invalid entry across the network. When the reorg channel is under backpressure (rapid block arrival or high tx throughput), the snapshot can remain stale across many blocks, widening the attack window significantly. This matches the allowed High impact: **Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points).

## Likelihood Explanation

The vulnerability is reachable by any unprivileged RPC caller (`send_transaction`) or P2P transaction sender. Under normal conditions the window is milliseconds, requiring precise timing. However, the window is reliably widened under load: once the 512-slot reorg channel fills, `try_send` silently drops notifications and the snapshot stays stale across multiple blocks, making every such double-spend submission succeed without any special privileges, keys, or majority hashpower.

## Recommendation

1. **Remove the conditional guard in `submit_entry`**: `check_rtx` (using `OverlayCellChecker` against the current snapshot) should run unconditionally, not only when `pre_resolve_tip != tip_hash`. This ensures cell liveness is always re-validated under the write lock against the latest snapshot.
2. **Replace `try_send` with a non-lossy send** for the reorg notification, or update the tx-pool snapshot synchronously before the chain service returns, to eliminate the stale-snapshot window entirely.
3. As defense-in-depth, after `_update_tx_pool_for_reorg` completes, sweep remaining pool entries against the new snapshot to evict any entries whose inputs are now dead.

## Proof of Concept

**Setup:** Node N with tx-pool snapshot stale at block height H; actual chain tip is H+1 where cell C was spent by Tx A.

1. Attacker observes Tx A (spending cell C) committed in block H+1.
2. Before the tx-pool snapshot updates to H+1 (either within the normal millisecond window, or during an extended window caused by reorg channel backpressure), attacker submits Tx B (also spending cell C) via `send_transaction` RPC.
3. `pre_check` runs: `resolve_tx_from_pool` uses the stale snapshot at H → cell C appears `Live` → Tx B resolves successfully. The `OutPointError::Dead` branch is never reached.
4. `verify_rtx` runs with stale snapshot → passes.
5. `submit_entry` runs: `pre_resolve_tip == snapshot.tip_hash()` (both H from the stale snapshot) → the guard at line 120 is **not entered** → `_submit_entry` inserts Tx B unconditionally.
6. Tx B is relayed to peers. Peers with the same stale window also admit it.
7. Eventually `update_tx_pool_for_reorg` runs and evicts Tx B — but the invalid entry was live in the pool and propagated across the network during the window.

**Extended-window variant:** Submit blocks rapidly enough to fill the 512-slot reorg channel, then submit double-spend transactions; the stale snapshot persists until the channel drains, making every such submission succeed reliably.
