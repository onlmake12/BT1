### Title
Missing `rejected_transaction` Notification on `remove_transaction` RPC — (`tx-pool/src/process.rs`, `tx-pool/src/pool.rs`)

---

### Summary

The `remove_transaction` RPC silently removes a transaction (and all its descendants) from the tx-pool without invoking the reject callback chain. This means no `rejected_transaction` notification is emitted to subscribers, leaving off-chain clients — including the built-in indexer — with a permanently stale, incorrect view of the pool.

---

### Finding Description

CKB's notification system (`ckb-notify`) is the direct analog to Solidity event emissions. The `NotifyController` exposes `notify_reject_transaction`, which feeds the `rejected_transaction` subscription topic consumed by RPC subscribers and the indexer's `PoolService`. [1](#0-0) 

The callback wiring that connects pool mutations to notifications is established in `register_tx_pool_callback`: [2](#0-1) 

Every sensitive pool removal path that is supposed to notify subscribers calls `callbacks.call_reject(self, &entry, reject)`:

- `remove_expired` — calls `callbacks.call_reject` [3](#0-2) 
- `limit_size` — calls `callbacks.call_reject` [4](#0-3) 
- `remove_committed_tx` (conflict resolution) — calls `callbacks.call_reject` [5](#0-4) 

However, `TxPool::remove_tx`, which is the backend for the `remove_transaction` RPC, does **not** call any callback: [6](#0-5) 

This is called from `TxPoolService::remove_tx`: [7](#0-6) 

Which is dispatched from the `RemoveLocalTx` message handler in the service loop, triggered by the public `remove_transaction` RPC: [8](#0-7) 

---

### Impact Explanation

1. **RPC subscribers** listening on the `rejected_transaction` topic (via TCP/WebSocket `subscribe`) will never receive a notification when `remove_transaction` is called. A transaction that was previously announced via `new_transaction` silently vanishes from the pool with no corresponding rejection event.

2. **The built-in indexer's `PoolService`** maintains its own in-memory pool view by consuming `new_transaction` and `reject_transaction` notifications: [9](#0-8) 

   When `remove_transaction` is called, the indexer never receives the rejection signal, so its pool view permanently diverges from the actual pool state. Any indexer query about pool membership for the removed transaction will return a stale "pending" result.

3. **Descendant transactions** are also silently removed (`remove_entry_and_descendants` is called), compounding the staleness — multiple transactions disappear with zero notifications. [10](#0-9) 

---

### Likelihood Explanation

The `remove_transaction` RPC is a supported, documented, publicly reachable endpoint callable by any local RPC user — an explicitly in-scope entry point. The integration test suite exercises it directly: [11](#0-10) 

Any operator or local tool that calls `remove_transaction` (e.g., to manually evict a stuck transaction) will silently corrupt the notification stream and the indexer's pool state. This is a routine operational action, not an exotic attack.

---

### Recommendation

`TxPoolService::remove_tx` should call `callbacks.call_reject` for every removed entry, consistent with all other removal paths. A synthetic `Reject` variant (e.g., `Reject::Removed` or reusing `Reject::Expiry`) should be introduced or an existing variant reused to signal the manual removal reason. The same fix applies to `clear_pool`, which also clears all entries without emitting any rejection notifications.

---

### Proof of Concept

1. Subscribe to `rejected_transaction` via the WebSocket RPC.
2. Submit a transaction via `send_transaction` — observe the `new_transaction` notification arrives.
3. Call `remove_transaction` with the same tx hash.
4. Observe: **no `rejected_transaction` notification is received**, despite the transaction being removed from the pool.
5. Query the indexer's pool view — the transaction still appears as pending.

The root cause is the missing `callbacks.call_reject` call in `TxPool::remove_tx` at `tx-pool/src/pool.rs:358-361`, which is the only pool-removal function that bypasses the callback chain.
