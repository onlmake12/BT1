### Title
Duplicate-Index Amplification in `GetBlockTransactionsProcess::execute` Causes OOM Process Abort — (`sync/src/relayer/get_block_transactions_process.rs`)

### Summary
An unprivileged remote P2P peer can send a `GetBlockTransactions` message containing up to 32,767 duplicate transaction indexes for any stored block. The handler performs no deduplication, clones the same transaction up to 32,767 times, and then calls `Vec::with_capacity(expected_length())` inside the molecule builder with a `usize` value that can reach ~19.5 GB, triggering an OOM abort that crashes the node process.

### Finding Description

`MAX_RELAY_TXS_NUM_PER_BATCH` is set to **32,767**: [1](#0-0) 

The guard in `execute` only checks the count against this limit, with no check for duplicate indexes: [2](#0-1) 

The transaction lookup loop uses `filter_map` with no deduplication — if the same index appears 32,767 times, the same `TransactionView` is cloned 32,767 times: [3](#0-2) 

These clones are then passed into `packed::BlockTransactions::new_builder().transactions(...).build()`: [4](#0-3) 

Inside `TransactionVecBuilder`, `expected_length()` sums the serialized sizes of all items as a `usize`: [5](#0-4) 

Then `build()` calls `Vec::with_capacity(self.expected_length())` with that full `usize` value: [6](#0-5) 

The consensus `MAX_BLOCK_BYTES` is ~597,000 bytes: [7](#0-6) 

A single transaction can be up to ~597,000 bytes (enforced by `SizeVerifier`): [8](#0-7) 

**Math**: 32,767 × 597,000 ≈ **19.56 GB** >> 4 GB (2³²). `Vec::with_capacity(19_561_899_000)` on a 64-bit system calls the global allocator for ~19.56 GB. When the allocation fails, Rust's default allocator calls `handle_alloc_error`, which **aborts the process** — crashing the node and its RPC API.

A secondary issue: `write()` casts `total_size as molecule::Number` (u32), which silently truncates if `total_size > u32::MAX`, producing a malformed length field — but the OOM abort occurs first. [9](#0-8) 

### Impact Explanation
The node process aborts, taking down the RPC API and all sync/relay functionality. Any connected peer can trigger this repeatedly, preventing the node from staying online.

### Likelihood Explanation
The attacker only needs to:
1. Connect as a P2P peer (no privilege required).
2. Know the hash of any stored block that contains a transaction of meaningful size (trivially discoverable on mainnet via the public chain).
3. Send a single crafted `GetBlockTransactions` message with 32,767 copies of the same index.

No PoW, no key, no special role is required.

### Recommendation
- **Deduplicate indexes** before the lookup loop (e.g., sort and dedup the index list, or use a `HashSet`).
- Add a secondary guard: after deduplication, re-check that the resulting count is within a safe bound.
- Alternatively, cap the total serialized size of the response before building the molecule buffer.

### Proof of Concept
1. Mine or locate a stored block containing a transaction of size ~597 KB (valid under consensus).
2. Connect to the target node as a P2P peer on the relay protocol.
3. Send a `GetBlockTransactions` message with `block_hash` set to that block's hash and `indexes` set to `[0u32; 32767]` (32,767 copies of index 0).
4. Observe the node process abort with an OOM error; the RPC API becomes unreachable.
