### Title
TOCTOU Race Condition in Tx-Pool Duplicate Detection Allows Double-Enqueue of the Same Transaction - (File: tx-pool/src/process.rs)

### Summary

`TxPoolService::resumeble_process_tx` checks for duplicate transactions in the orphan pool and verify queue using separate, non-atomic read locks, then acquires a separate write lock to enqueue. Between the read checks and the write enqueue, another concurrent task can insert the same transaction, resulting in the same transaction being enqueued twice into the verify queue. This is the CKB analog of the reentrancy vulnerability: state is read under one lock, an external/concurrent operation can mutate that state, and then the original operation proceeds based on stale information.

### Finding Description

In `resumeble_process_tx` (called by `submit_remote_tx` and `notify_tx` for all remote peer-submitted transactions), the duplicate check and the enqueue are performed in three separate, non-atomic async operations:

```
// Step 1: acquire orphan read lock, check, release
if self.orphan_contains(&tx).await { ... }

// Step 2: acquire verify_queue read lock, check, release
if self.verify_queue_contains(&tx).await { ... }

// Step 3: acquire verify_queue write lock, insert
self.enqueue_verify_queue(tx, is_proposal_tx, remote).await
```

Each of these three steps acquires and releases its own lock independently. Between steps 2 and 3, another concurrent async task (e.g., a second peer relaying the same transaction, or the `notify_tx` path triggered by a compact block) can pass its own duplicate check and also reach `enqueue_verify_queue`. Both tasks then acquire the write lock sequentially and both call `queue.add_tx(...)`.

`VerifyQueue::add_tx` does have an internal duplicate check at line 204, but when `is_proposal_tx` is `false` (the normal remote-tx path), it simply returns `Ok(false)` without error — meaning the second insertion silently succeeds as a no-op. However, when `is_proposal_tx` is `true` (the `notify_tx` path), the existing entry is **removed and re-inserted**, resetting its `added_time` and causing it to be re-verified unnecessarily.

More critically, the `process_tx` path (used for local RPC `send_transaction`) performs its duplicate check across two separate lock acquisitions:

```rust
if self.verify_queue_contains(&tx).await || self.orphan_contains(&tx).await {
    return Err(Reject::Duplicated(tx.hash()));
}
// ... then calls _process_tx which calls pre_check (read lock) then submit_entry (write lock)
```

Between the duplicate check and `submit_entry`'s write lock, a concurrent task can insert the same transaction into the pool. The `pre_check` inside `_process_tx` uses a read lock and calls `check_txid_collision`, which checks `tx_pool.contains_proposal_id`. But if the concurrent task has not yet committed to the pool (still in verify queue), `check_txid_collision` will not detect it, and both tasks proceed to `submit_entry`. Inside `submit_entry`'s write lock, `_submit_entry` calls `tx_pool.add_pending` which calls `pool_map.add_entry` — this is the final guard, but the two concurrent verify operations (CPU-intensive script execution via `verify_rtx`) have already been launched in parallel, wasting node resources and potentially causing inconsistent fee/cycle accounting.

The root cause is that the three pools (`tx_pool`, `orphan`, `verify_queue`) are protected by three independent `RwLock`s, and no single atomic operation spans all three. This is structurally analogous to the reentrancy vulnerability: state is read (checked) before an operation completes, and a concurrent actor can modify that state in the window between check and act.

**Exact code path:**

1. Peer A sends tx T → `submit_remote_tx` → `resumeble_process_tx`
2. `orphan_contains` acquires+releases orphan read lock → not found
3. `verify_queue_contains` acquires+releases verify_queue read lock → not found
4. **Context switch** — Peer B sends same tx T → same path, same checks pass
5. Peer A: `enqueue_verify_queue` acquires write lock → inserts T
6. Peer B: `enqueue_verify_queue` acquires write lock → `add_tx` sees duplicate, returns `Ok(false)` (non-proposal) or removes+reinserts (proposal)
7. For `notify_tx` (`is_proposal_tx=true`): T is removed and re-inserted, resetting ordering and triggering redundant re-verification [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation

**Resource exhaustion / tx-pool state corruption:**

- An unprivileged network peer can relay the same transaction twice (or two peers can relay the same tx simultaneously) to trigger concurrent verify-queue insertions.
- For the `notify_tx` / `is_proposal_tx=true` path: the existing entry is evicted and re-inserted, resetting its `added_time`. This disrupts FIFO ordering of the verify queue and causes the transaction to be re-verified from scratch, wasting CPU cycles proportional to the transaction's script complexity.
- For the `process_tx` (local RPC) path: two concurrent `_process_tx` calls can both reach `verify_rtx` for the same transaction, doubling the CPU cost of script verification. Since `verify_rtx` is the most expensive operation (CKB-VM script execution), this can be triggered by any RPC caller to cause sustained CPU exhaustion.
- The `total_tx_size` accounting in `VerifyQueue` is only updated atomically within the write lock, so the double-enqueue via the proposal path can cause `total_tx_size` to underflow when the entry is removed and re-added (the remove subtracts the size, the re-add adds it back, but if the remove happens while another task has already accounted for the size, the counter can drift). ### Likelihood Explanation

- **High likelihood**: Any two peers relaying the same transaction simultaneously (common in a well-connected network) trigger this window. The async await points between the three lock acquisitions are real yield points in Tokio's cooperative scheduler.
- The `notify_tx` path is triggered automatically when a compact block is received, making it reachable without any special attacker capability — a normal block relay event can trigger concurrent `notify_tx` calls for the same transaction from multiple peers.
- The `submit_remote_tx` path is reachable by any unprivileged peer that sends a `RelayTransactionHashes` or `SendRelayTransactions` message. [7](#0-6) [8](#0-7) 

### Recommendation

Combine the duplicate check and enqueue into a single atomic operation under the verify_queue write lock. Specifically, `resumeble_process_tx` should acquire the `verify_queue` write lock once and perform both the `contains_key` check and the `add_tx` insertion within that single lock scope, eliminating the window between check and act. The orphan check can remain separate since orphan insertion is a different code path, but the verify_queue check+insert must be atomic.

Additionally, the `process_tx` path should perform its `check_txid_collision` inside the same write lock as `submit_entry`, or use a per-tx-hash mutex/semaphore to serialize concurrent processing of the same transaction. [9](#0-8) [10](#0-9) 

### Proof of Concept

```
1. Node receives tx T from Peer A (submit_remote_tx) and Peer B (notify_tx) concurrently.

2. Both tasks enter resumeble_process_tx concurrently.

3. Task A: orphan_contains(&T) → acquires orphan read lock → false → releases lock
   Task B: orphan_contains(&T) → acquires orphan read lock → false → releases lock

4. Task A: verify_queue_contains(&T) → acquires vq read lock → false → releases lock
   Task B: verify_queue_contains(&T) → acquires vq read lock → false → releases lock
   [Both see empty queue — no duplicate detected]

5. Task A: enqueue_verify_queue(T, false, ...) → acquires vq write lock → inserts T → releases
   Task B: enqueue_verify_queue(T, true, ...)  → acquires vq write lock
           → add_tx sees T already present, is_proposal_tx=true
           → remove_tx(T) [total_tx_size decremented]
           → re-insert T with new added_time [total_tx_size incremented]
           → releases lock

6. T is now re-queued with reset ordering. The VerifyMgr worker picks T again and
   runs full CKB-VM script verification a second time — wasting CPU proportional
   to T's script complexity.

7. Repeat with N peers to cause sustained CPU exhaustion on the target node.
``` [11](#0-10) [1](#0-0) [12](#0-11)

    /// If the queue did not have this tx present, true is returned.
    /// If the queue did have this tx present, false is returned.
    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }
