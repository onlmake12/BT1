Audit Report

## Title
`GetBlocksProcess` Missing IBD Guard Allows Resource Exhaustion of Syncing Nodes — (File: sync/src/synchronizer/get_blocks_process.rs)

## Summary
The module-level documentation in `sync/src/synchronizer/mod.rs` explicitly contracts that an IBD-mode node will respond with `InIBD` to both `GetHeaders` and `GetBlocks` requests. `GetHeadersProcess::execute()` enforces this contract, but `GetBlocksProcess::execute()` contains no IBD check whatsoever. Any unprivileged P2P peer can therefore send `GetBlocks` to an IBD node and force it to perform RocksDB reads, block serialization, and network transmission — consuming resources the node needs for its own block download.

## Finding Description
The module documentation at `sync/src/synchronizer/mod.rs` L4 states:
> "When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` and `GetBlocks` requests"

`GetHeadersProcess::execute()` at `sync/src/synchronizer/get_headers_process.rs` L53–65 faithfully implements this: it calls `active_chain.is_initial_block_download()`, sends `InIBD`, and returns `Status::ignored()`.

`GetBlocksProcess::execute()` at `sync/src/synchronizer/get_blocks_process.rs` L33–97 has no such check. After validating only message size (L36), genesis-block requests (L52), duplicates (L56), and `BlockStatus::BLOCK_VALID` (L60), it immediately reads blocks from `active_chain.get_block()` (L68) and spawns async tasks to transmit `SendBlock` messages (L83). There is no call to `is_initial_block_download()` anywhere in the function.

The `Synchronizer::received()` dispatcher at `sync/src/synchronizer/mod.rs` L890–970 performs no IBD gate before calling `self.process(nc, peer_index, msg).await` at L963, so `GetBlocksProcess` is reached unconditionally. The IBD guard in `sync/src/relayer/mod.rs` L815–818 covers only relay-protocol messages and does not protect the sync protocol's `GetBlocks` path.

## Impact Explanation
An attacker with a single P2P connection can send repeated `GetBlocks` messages, each containing up to `MAX_HEADERS_LEN` block hashes. For every valid hash the IBD node performs a RocksDB read, serializes a full block, and transmits it. Targeting many IBD nodes simultaneously (e.g., all newly bootstrapping nodes) with minimal cost constitutes a bad design that can cause CKB network congestion and prevent new nodes from completing IBD. This matches the **High** impact category: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
IBD is the normal startup state for any new or long-offline CKB node. Establishing a P2P connection requires no privilege, key, or stake. The attack requires only sending valid `GetBlocks` messages in a loop, which is trivially automatable. No special knowledge beyond knowing a few early block hashes (publicly available) is needed.

## Recommendation
Add an IBD guard at the top of `GetBlocksProcess::execute()`, mirroring `GetHeadersProcess`:

```rust
let active_chain = self.synchronizer.shared.active_chain();
if active_chain.is_initial_block_download() {
    self.send_in_ibd();
    return Status::ignored();
}
```

`GetBlocksProcess` would also need a `send_in_ibd()` helper identical to the one in `GetHeadersProcess` (L101–115). Alternatively, add a single IBD gate in `Synchronizer::process()` covering both `GetHeaders` and `GetBlocks` message types to eliminate per-handler duplication.

## Proof of Concept
1. Start a CKB node from genesis; confirm `is_initial_block_download() == true`.
2. Connect a custom P2P client as an inbound peer.
3. Send `SyncMessage::GetBlocks` containing hashes of known-valid early blocks (e.g., blocks 1–16).
4. Observe the node responds with `SendBlock` messages — no `InIBD` is sent.
5. Repeat in a tight loop from multiple connections; measure disk I/O and outbound bandwidth increasing while the node's own block-download rate decreases.
6. Contrast with `GetHeaders`: sending `GetHeaders` to the same IBD node returns `InIBD` and no headers are served, confirming the asymmetry.
