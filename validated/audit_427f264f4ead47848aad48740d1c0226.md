Audit Report

## Title
Concurrent `HeaderMapKernel::get()` Calls Race on Destructive `backend.remove()`, Causing Unconditional Panic on `HEADER_VALID` Header — (`shared/src/types/header_map/kernel_lru.rs`)

## Summary

`HeaderMapKernel::get()` promotes sled-backend entries to memory via a destructive `backend.remove()` with no lock spanning the memory-miss check and the backend removal. Two concurrent callers for the same hash that exists only in sled will race: one wins the remove and gets `Some(view)`, the other gets `None` and propagates it up. When the losing thread reaches `HeaderAcceptor::accept()` in `headers_process.rs`, the subsequent `unwrap_or_else(|| panic!(...))` fires unconditionally because the header has `HEADER_VALID` status but is no longer findable in either the header map or the chain store, crashing the node process.

## Finding Description

**Root cause — `kernel_lru.rs` lines 109–142:**

`get()` calls `self.memory.get_refresh(hash)`, which acquires and releases a write lock on the `MemoryMap` `RwLock`. On a miss, it calls `self.backend.remove(hash)` — a destructive sled operation. There is no lock or coordination spanning both operations. Two threads that both miss the memory check will both call `backend.remove()` for the same key. The non-destructive `backend.get()` method exists in `SledBackend` (line 56) but is not used in the read path.

**Sled `remove()` — `backend_sled.rs` lines 90–100:**

Sled's `remove` is atomic per-key, so exactly one thread wins and gets `Some(view)`; the other gets `None`. The losing thread returns `None` from `HeaderMapKernel::get()`. Thread A's subsequent `memory.insert(view)` at line 137 is too late — Thread B already passed the memory check and will not retry.

**Race interleaving:**

| Step | Thread A | Thread B |
|------|----------|----------|
| 1 | `memory.get_refresh(H)` → `None` (lock released) | |
| 2 | | `memory.get_refresh(H)` → `None` (lock released) |
| 3 | `backend.remove(H)` → `Some(view)` | |
| 4 | | `backend.remove(H)` → `None` ← loses |
| 5 | `memory.insert(view)` | |
| 6 | returns `Some(view)` | returns `None` ← propagates up |

**How entries reach sled-only state — `kernel_lru.rs` lines 168–181:**

`limit_memory()` runs every 5 seconds via a background task, calling `backend.insert_batch()` followed by `memory.remove_batch()`. During IBD the memory limit is routinely exceeded, making sled-only entries common.

**Panic site — `headers_process.rs` lines 304–316:**

`HeaderAcceptor::accept()` checks `BlockStatus::HEADER_VALID` and then calls `sync_shared.get_header_index_view()` with `store_first = status.contains(BlockStatus::BLOCK_STORED)`. For a header that is `HEADER_VALID` but not yet `BLOCK_STORED`, `store_first` is `false`.

**`get_header_index_view()` — `sync/src/types/mod.rs` lines 1158–1166:**

With `store_first = false`, it calls `header_map().get(hash)` first. Thread B's call returns `None` (race). The fallback to the chain store also returns `None` because the header is not yet stored on-chain. The entire function returns `None`, and the `unwrap_or_else(|| panic!(...))` fires unconditionally.

**Existing guards are insufficient:** The `MemoryMap` `RwLock` only serializes memory-layer operations; it does not extend to cover the backend remove. The `SledBackend` has a non-destructive `get()` method available but is not used in the read path.

## Impact Explanation

The panic terminates the CKB node process. This matches the allowed bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.** The panic is unconditional (`unwrap_or_else(|| panic!(...))`) and requires no special privileges — only two standard P2P peer connections sending valid headers.

## Likelihood Explanation

During IBD or rapid header sync, the node processes `SendHeaders` messages from multiple peers concurrently in separate tasks/threads. The memory limit is routinely exceeded, pushing entries to sled. An attacker with two or more peer connections can send the same valid header batch simultaneously, reliably triggering concurrent `get()` calls for the same hash. No PoW forgery or privileged access is required. The race window (between the memory miss and the sled remove) is narrow but reproducible under load, and the attacker can widen it by sending large header batches that saturate the memory limit.

## Recommendation

Replace the destructive `backend.remove()` in the read path of `HeaderMapKernel::get()` with the non-destructive `backend.get()`. The `SledBackend` already implements a `get()` method. Promotion to memory (and lazy eviction from sled) should be handled separately and idempotently — for example, during the next `limit_memory()` sweep — or the entire get-and-promote sequence must be serialized per key. The minimal safe fix:

```rust
// In kernel_lru.rs get(), replace:
if let Some(view) = self.backend.remove(hash) {
    self.memory.insert(view.clone());
    Some(view)
}
// With:
if let Some(view) = self.backend.get(hash) {
    self.memory.insert(view.clone());
    Some(view)
}
```

This makes `get()` idempotent and eliminates the destructive race. Eviction from sled can remain in `limit_memory()`.
