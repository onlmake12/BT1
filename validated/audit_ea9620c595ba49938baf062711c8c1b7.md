### Title
`MAX_DEPOSIT_LIMIT` Bypass for Permanent Subscriptions via Cumulative `addFunds` Calls — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract enforces a `MAX_DEPOSIT_LIMIT` (100 ETH) on permanent subscriptions to cap how much ETH can be permanently locked per subscription. However, both `createSubscription` and `addFunds` validate only the **current transaction's `msg.value`** against the cap, not the **cumulative balance**. An unprivileged user can bypass the cap entirely by calling `addFunds` repeatedly with amounts at or below the limit.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
``` [1](#0-0) 

In `createSubscription`, the guard is:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [2](#0-1) 

In `addFunds`, the guard is identical in structure:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [3](#0-2) 

Neither check accounts for the already-accumulated `status.balanceInWei`. The check is `msg.value > MAX_DEPOSIT_LIMIT`, not `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`.

---

### Impact Explanation

Permanent subscriptions explicitly prohibit withdrawals:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [4](#0-3) 

Any ETH deposited into a permanent subscription is **irrecoverably locked** in the contract. The `MAX_DEPOSIT_LIMIT` exists precisely to bound this risk per subscription. By bypassing it, an attacker (or a user acting against their own interest) can lock an **unbounded amount of ETH** permanently in a single subscription — far exceeding the intended 100 ETH ceiling. This undermines the protocol's safety invariant and could result in large-scale permanent ETH loss.

---

### Likelihood Explanation

The entry point is `addFunds`, which is `external payable` with **no access control** — any address can call it for any active subscription. The bypass requires only repeated calls with `msg.value ≤ MAX_DEPOSIT_LIMIT`. No privileged role, leaked key, or external oracle manipulation is needed. The attack is trivially executable by any transaction sender.

---

### Recommendation

Change both checks to validate the **post-deposit cumulative balance**:

In `createSubscription`:
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
```
should become:
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) { // already correct at creation since balanceInWei starts at 0
```
(This one is actually fine at creation since `balanceInWei` starts at 0, but for consistency and future-proofing, use the cumulative form.)

In `addFunds`:
```solidity
// Replace:
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {

// With:
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
``` [5](#0-4) 

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription with exactly MAX_DEPOSIT_LIMIT (100 ETH)
//    → passes: msg.value (100e18) is NOT > MAX_DEPOSIT_LIMIT (100e18)
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);

// 2. Call addFunds repeatedly with 100 ETH each time
//    → each call passes: msg.value (100e18) is NOT > MAX_DEPOSIT_LIMIT (100e18)
//    → status.balanceInWei grows: 200 ETH, 300 ETH, 400 ETH, ...
scheduler.addFunds{value: 100 ether}(subId);
scheduler.addFunds{value: 100 ether}(subId);
// ... repeat N times → total locked = (N+1) * 100 ETH, all permanently irrecoverable
```

The cumulative balance is never checked against `MAX_DEPOSIT_LIMIT` in `addFunds`, so the cap is fully bypassed through multiple transactions — a direct analog to the reported `depositCap` bypass. [6](#0-5)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
