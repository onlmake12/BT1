### Title
Minimum Balance Validation Applied to Gross Deposit But Not Re-Checked After Fee Deductions in `updatePriceFeeds` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol`'s `updatePriceFeeds` function deducts both a Pyth protocol fee and a keeper fee from a subscription's balance, but never re-validates that the remaining balance stays at or above the protocol-mandated minimum balance. The minimum balance check is only enforced at creation/deposit time (gross), not after fee deductions (net), allowing active subscriptions to be drained below the minimum balance through normal keeper-triggered updates.

---

### Finding Description

`getMinimumBalance` returns `numPriceFeeds * minimumBalancePerFeed` and is enforced as a hard floor in every balance-mutating path **except** `updatePriceFeeds`:

- `createSubscription` (L38–44): reverts if `msg.value < minimumBalance`
- `updateSubscription` (L106–112): reverts if `currentStatus.balanceInWei < minimumBalance`
- `addFunds` (L620–626): reverts if post-deposit balance < minimumBalance
- `withdrawFunds` (L649–655): reverts if post-withdrawal balance < minimumBalance

In `updatePriceFeeds` (L275–348), the only balance check is:

```solidity
// L294-297
if (status.balanceInWei < pythFee) {
    revert SchedulerErrors.InsufficientBalance();
}
```

This checks only that the gross balance covers the Pyth fee. Immediately after, the Pyth fee is deducted (L305), and then `_processFeesAndPayKeeper` (L345, L840–863) deducts an additional keeper fee (`gasCost + keeperSpecificFee`). Neither deduction is followed by a check that `status.balanceInWei >= getMinimumBalance(numPriceFeeds)`. [1](#0-0) [2](#0-1) 

The minimum balance is explicitly documented as "the minimum balance an active subscription of a given size needs to hold," and is enforced in all other mutating paths. [3](#0-2) 

---

### Impact Explanation

A subscription created with exactly `getMinimumBalance(N)` (the minimum required deposit) will have its balance drained below the minimum after the first keeper-triggered `updatePriceFeeds` call, because the combined `pythFee + keeperFee` is deducted without any floor check. The subscription remains active on-chain but now holds less than the minimum balance the protocol guarantees for active subscriptions.

This violates the protocol invariant that active subscriptions always maintain a minimum balance. Downstream consequences include:

1. The subscription manager cannot call `addFunds` with a small top-up — `addFunds` enforces the minimum balance post-deposit, so the manager must supply enough to restore the full minimum in one transaction, which may be unexpected.
2. Keepers may attempt further updates on a subscription whose balance is below the minimum, wasting gas and potentially causing failed updates.
3. The protocol's stated guarantee — that active subscriptions hold at least `minimumBalance` — is silently broken, misleading integrators who rely on this invariant for off-chain monitoring or on-chain logic. [4](#0-3) 

---

### Likelihood Explanation

**High.** Any unprivileged keeper (the `pusher` role) can call `updatePriceFeeds` on any active subscription. A subscription funded to exactly the minimum balance — the natural starting state, as `createSubscription` requires at least `minimumBalance` — will have its balance fall below the minimum after the very first update. No special conditions, privileged access, or external oracle manipulation is required. [5](#0-4) 

---

### Recommendation

After `_processFeesAndPayKeeper` completes in `updatePriceFeeds`, add a check that the remaining balance is at or above the minimum balance:

```solidity
_processFeesAndPayKeeper(status, startGas, params.priceIds.length);

// Re-validate minimum balance after all fee deductions
uint256 minimumBalance = this.getMinimumBalance(uint8(params.priceIds.length));
if (status.balanceInWei < minimumBalance) {
    revert SchedulerErrors.InsufficientBalance();
}
```

Alternatively, the minimum balance should be designed to account for at least one full update cycle (pythFee + keeperFee), so that a subscription funded to the minimum can always complete one update without falling below the floor.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` with `minimumBalancePerFeed = 1 ether` and 1 price feed.
2. Call `createSubscription{value: 1 ether}(params)` — passes because `msg.value == minimumBalance`.
3. A keeper calls `updatePriceFeeds(subscriptionId, updateData)`.
   - `pythFee` (e.g., 0.001 ether) is deducted at L305.
   - `keeperFee` (gas cost + `singleUpdateKeeperFeeInWei`) is deducted at L856.
   - `status.balanceInWei` is now `< 1 ether` (below `minimumBalance`).
   - No revert occurs.
4. Verify: `getSubscription(subscriptionId).status.balanceInWei < getMinimumBalance(1)` — the invariant is broken.
5. Attempt `addFunds{value: 0.0001 ether}(subscriptionId)` — reverts with `InsufficientBalance` because the post-deposit balance is still below minimum, trapping the manager. [6](#0-5) [7](#0-6) [8](#0-7)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
