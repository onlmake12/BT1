### Title
`set_ban` RPC Silently Inserts Immediately-Expired Ban Entry When `absolute=true` and `ban_time` Is Null/Zero — (File: `rpc/src/module/net.rs`)

---

### Summary

The `set_ban` RPC handler performs no validation on the `ban_time` parameter when `absolute=true`. If a node operator omits `ban_time` (passes `null`) or passes `0x0`, the missing value is silently defaulted to `Timestamp(0)`. This produces a `ban_until` of `0`, which is stored as `now_ms + 0 = now_ms` — a timestamp already in the past. The targeted peer is disconnected but can immediately reconnect, because every subsequent ban-check evaluates `ban_until.gt(&now_ms)` as `false`. The RPC returns `null` (success) with no error, leaving the operator with a false belief that the peer is banned.

---

### Finding Description

**Root cause — `rpc/src/module/net.rs`, lines 706–717:** [1](#0-0) 

```rust
"insert" => {
    let ban_until = if absolute.unwrap_or(false) {
        ban_time.unwrap_or_default().into()   // ← Timestamp::default() == 0 when ban_time is null
    } else {
        unix_time_as_millis()
            + ban_time
                .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
                .value()
    };
    self.network_controller
        .ban(ip_network, ban_until, reason.unwrap_or_default());
```

When `absolute = true` and `ban_time = null`, `ban_time.unwrap_or_default()` yields `Timestamp(0)`, so `ban_until = 0`.

`NetworkController::ban` receives `ban_until = 0` and forwards it verbatim to `PeerStore::ban_network` as the `timeout_ms` argument.

```rust
        address: network,

```

`ban_until` is stored as `now_ms + 0 = now_ms`.

**Ban check — `network/src/peer_store/ban_list.rs`, lines 48–58:** [4](#0-3) 

```rust

    ...

```

Because `now_ms_check >= now_ms_at_insert`, the strict `gt` check is always `false`. The peer is never considered banned.

The `disconnect_peers_in_ip_range` call inside `NetworkController::ban` does disconnect the peer at the moment of the RPC call, but since the ban entry is immediately expired, the peer can reconnect on the next dial attempt.

---

### Impact Explanation

A node operator who calls `set_ban` with `absolute=true` and omits `ban_time` (a natural mistake when using JSON-RPC tooling that omits optional fields) silently creates a no-op ban. The targeted peer — which may be sending invalid blocks, spam transactions, or otherwise abusing the node — is disconnected for one connection cycle and then reconnects freely. The operator sees a success response and a ban entry in `get_banned_addresses` with a `ban_until` value that looks like a plausible timestamp (`now_ms`), giving no indication that the ban is already expired. The node's only peer-level defense mechanism against a misbehaving peer is rendered ineffective.

---

### Likelihood Explanation

The RPC documentation states that `ban_time = null` means "use the default time of 24h," but this default only applies in the `absolute=false` branch. In the `absolute=true` branch, `null` silently becomes `0` with no fallback to the default duration and no error. Any operator or script that sets `absolute=true` but omits `ban_time` — a common pattern when adapting examples or using partial JSON payloads — will trigger this silently. The RPC returns HTTP 200 with `"result": null`, providing no feedback that the ban is ineffective.

---

### Recommendation

Add an explicit guard in the `absolute=true` branch to reject a zero or missing `ban_time`:

```rust
"insert" => {
    let ban_until = if absolute.unwrap_or(false) {
        let t: u64 = ban_time.unwrap_or_default().into();
        if t == 0 {
            return Err(RPCError::invalid_params(
                "Expected `params[2]` (ban_time) to be a non-zero absolute \
                 timestamp in milliseconds when `absolute` is true"
            ));
        }
        t
    } else {
        unix_time_as_millis()
            + ban_time
                .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
                .value()
    };
```

This mirrors the pattern recommended in the original report: a `require`-equivalent guard that reverts on a zero/null critical parameter, while still allowing intentional use of any valid non-zero timestamp.

### Proof of Concept

**Request (null ban_time with absolute=true):**
```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "set_ban",
  "params": ["192.168.0.2", "insert", null, true, "block malicious peer"]
}
```

**Response:** `{"id":1,"jsonrpc":"2.0","result":null}` — success, no error.

**Equivalent explicit zero:**
```json
["192.168.0.2", "insert", "0x0", true, "block malicious peer"]
```

**Observable effect:**
1. The peer at `192.168.0.2` is disconnected immediately.
2. `get_banned_addresses` returns an entry with `ban_until` equal to approximately the current Unix time in ms — visually indistinguishable from a valid near-future timestamp without careful inspection.
3. The ban check `ban_until.gt(&now_ms)` evaluates to `false` on every subsequent connection attempt.
4. The peer reconnects on its next dial, bypassing the intended ban entirely. [5](#0-4) [3](#0-2) [4](#0-3)
