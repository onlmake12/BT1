Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Allows Peer-Driven Memory Exhaustion — (`File: sync/src/types/mod.rs`)

## Summary
`SyncState::pending_get_block_proposals` is a `DashMap` with no size cap, no per-peer quota, and no eviction policy. Any peer that completes the CKB P2P handshake can send `GetBlockProposal` relay messages containing fabricated `ProposalShortId` values not present in the local tx-pool; each such ID is unconditionally inserted into the map. The map is only cleared by a periodic timer drain, so between drain ticks it grows without bound, enabling a single peer to drive the node to OOM.

## Finding Description
`SyncState` declares the map at `sync/src/types/mod.rs:1330`:
```rust
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>,
```
`insert_get_block_proposals` (lines 1594–1601) iterates every supplied ID and calls `entry(id).or_default().insert(pi)` with no size guard. The call site in `GetBlockProposalProcess::execute()` (`get_block_proposal_process.rs:73–77`) performs no additional check before writing. The only validation in `execute()` is a per-message count ceiling (`max_block_proposals_limit * max_uncles_num = 3000` on mainnet) and a within-message duplicate check (lines 35–52). IDs absent from the tx-pool are all forwarded to `insert_get_block_proposals`.

The sole drain path is `drain_get_block_proposals()` (lines 1586–1592), called from `prune_tx_proposal_request()` on a periodic timer. Between timer ticks the map accumulates every entry written by every peer with no upper bound.

A `RateLimiter<(PeerIndex, u32)>` exists in the `Relayer` struct (mod.rs line 81), capped at 30 requests/second per peer per message type. This limits insertion rate to `30 × 3000 = 90,000 IDs/second` per peer, but it does **not** cap the total map size. If the drain interval is on the order of seconds, a single peer can accumulate hundreds of thousands of entries per drain cycle; with the maximum peer count (128 peers, `MAX_RELAY_PEERS = 128`) the aggregate rate is `128 × 90,000 = ~11.5 million entries/second`, making OOM reachable in seconds.

## Impact Explanation
Unbounded heap growth from attacker-controlled P2P input leads to process termination by the OS OOM killer. This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.** A crashed node is fully offline until manually restarted; the attack can be resumed immediately after restart.

## Likelihood Explanation
Any peer that completes the standard CKB P2P handshake can send `GetBlockProposal` relay messages. No stake, fee, or special privilege is required. The rate limiter (30 req/s) slows but does not stop the attack; a single persistent peer can sustain it indefinitely. Multiple coordinated peers amplify the rate proportionally.

## Recommendation
- **Short term**: Add a hard cap inside `insert_get_block_proposals` on the total entry count of `pending_get_block_proposals` (e.g., `max_block_proposals_limit * max_uncles_num * MAX_RELAY_PEERS`). Reject insertions and consider banning the peer when the cap is reached.
- **Short term**: Track per-peer insertion counts and enforce a per-peer quota to prevent monopolization.
- **Long term**: Replace the unbounded `DashMap` with a capacity-bounded structure (LRU or FIFO), consistent with how `OrphanTxPool` uses `DEFAULT_MAX_ORPHAN_TRANSACTIONS` and `limit_size()`.

## Proof of Concept
1. Attacker establishes a peer connection and completes the CKB P2P handshake.
2. Attacker constructs `GetBlockProposal` relay messages each containing 3000 unique, randomly generated `ProposalShortId` values (10 bytes each) that do not exist in the victim's tx-pool.
3. Attacker sends messages at the rate-limiter ceiling (~30/s), inserting ~90,000 fake IDs/second into `pending_get_block_proposals`.
4. Between periodic `drain_get_block_proposals()` calls the map accumulates entries; with a drain interval of several seconds the map holds hundreds of thousands of entries consuming hundreds of MB.
5. With multiple coordinated peers the aggregate rate exceeds 10 million entries/second, exhausting gigabytes of memory within seconds.
