### Title
Unbounded `while` Loop in `constructProviderCommitment` Enables DoS of Provider Reveal When `maxNumHashes == 0` - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`constructProviderCommitment` contains a `while` loop that iterates `numHashes` times. When a provider's `maxNumHashes` is `0` (the default), there is no on-chain upper bound on `numHashes`. An unprivileged user can make a large number of requests to such a provider, causing the provider's `reveal`/`revealWithCallback` transactions to consume O(N) gas and eventually exceed the block gas limit, permanently bricking fulfillment of in-flight requests.

---

### Finding Description

`constructProviderCommitment` iterates `numHashes` times, computing a `keccak256` on each step:

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
``` [1](#0-0) 

`numHashes` for a request is set at request time as:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
``` [2](#0-1) 

The only guard against a large `numHashes` is:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [3](#0-2) 

The check is **skipped entirely when `maxNumHashes == 0`**, which is the default value for every newly registered provider. The test `testZeroMaxNumHashesDisableChecks` explicitly confirms this behavior — setting `maxNumHashes = 0` disables the guard. [4](#0-3) 

When `revealHelper` is called during `reveal` or `revealWithCallback`, it passes `req.numHashes` directly into `constructProviderCommitment`: [5](#0-4) 

---

### Impact Explanation

An attacker makes N requests to a provider whose `maxNumHashes == 0`. Each request increments `assignedSequenceNumber` by 1 while `currentCommitmentSequenceNumber` stays fixed (no reveals have happened). The N-th request stores `numHashes = N`. When the provider later calls `reveal` or `revealWithCallback` for that request, `constructProviderCommitment` loops N times. At ~36 gas per `keccak256` iteration, approximately 833,000 requests suffice to exceed Ethereum's 30M gas block limit. On low-fee L2 chains, the cost to reach this threshold is negligible.

The result: the provider's reveal transaction permanently reverts with out-of-gas. All in-flight requests with large `numHashes` values become permanently unresolvable. Users who paid fees and are waiting for randomness callbacks are permanently stuck — their `CALLBACK_NOT_STARTED` requests can never be fulfilled.

---

### Likelihood Explanation

- `requestV2` / `request` are permissionless; any address can call them.
- The attacker pays provider fees per request, but on low-fee chains (Polygon, Arbitrum, Base) the cost to make hundreds of thousands of requests is low.
- Providers that have not explicitly called `setMaxNumHashes` are vulnerable by default, since `maxNumHashes` initializes to `0`.
- The `advanceProviderCommitment` function (which could reduce future `numHashes`) requires a valid hash-chain proof and cannot be called by the attacker.

---

### Recommendation

1. **Change the default behavior**: Treat `maxNumHashes == 0` as "use a protocol-level default cap" rather than "no cap." Introduce a `DEFAULT_MAX_NUM_HASHES` constant and apply it when `providerInfo.maxNumHashes == 0`.
2. **Enforce a hard cap**: Add an absolute upper bound on `numHashes` in `requestHelper` regardless of `maxNumHashes`, sized to fit within a safe gas budget (e.g., `require(req.numHashes <= ABSOLUTE_MAX_HASHES)`).
3. **Document the risk**: At minimum, clearly warn providers in `register` and `setMaxNumHashes` that leaving `maxNumHashes == 0` exposes them to this DoS vector.

---

### Proof of Concept

1. Deploy Entropy on a low-fee chain. Register a provider without calling `setMaxNumHashes` (so `maxNumHashes == 0`).
2. As an attacker, call `requestV2(provider, ...)` 900,000 times, paying the provider fee each time. No reveals occur.
3. The 900,000th request is stored with `numHashes = 900000`.
4. The provider calls `revealWithCallback(provider, 900000, ...)`. Inside `revealHelper`, `constructProviderCommitment(900000, x)` loops 900,000 times (~32.4M gas), exceeding the block gas limit.
5. The transaction reverts with out-of-gas. The request is permanently stuck in `CALLBACK_NOT_STARTED` state and can never be cleared. [6](#0-5)
