### Title
Off-by-One Staleness Boundary Discrepancy in Aptos `check_price_is_fresh` vs All Other Chain Implementations - (File: `target_chains/aptos/contracts/sources/pyth.move`)

---

### Summary

The Aptos on-chain staleness check uses a strict `<` comparison (`age < max_age_secs`), while every other Pyth chain implementation (EVM Solidity, Fuel, Near, Starknet) and the JS SDK all use an inclusive `<=` comparison (`diff > age` / `time_difference <= time_period` / `actual_age <= age`). At the exact boundary where `age == max_age_secs`, the Aptos contract rejects a price as stale while all other implementations accept it as fresh. This is a direct analog to the reported off-by-one discrepancy between on-chain and off-chain timer checks.

---

### Finding Description

**Aptos** — `target_chains/aptos/contracts/sources/pyth.move`, `check_price_is_fresh`:

```move
fun check_price_is_fresh(price: &Price, max_age_secs: u64) {
    let age = abs_diff(timestamp::now_seconds(), price::get_timestamp(price));
    assert!(age < max_age_secs, error::stale_price_update());  // STRICT <
}
``` [1](#0-0) 

This asserts `age < max_age_secs`. When `age == max_age_secs` (exactly at the threshold), the assertion **fails** and the price is rejected as stale.

**EVM Solidity** — `target_chains/ethereum/sdk/solidity/AbstractPyth.sol`:

```solidity
if (diff(block.timestamp, price.publishTime) > age)
    revert PythErrors.StalePrice();
``` [2](#0-1) 

Uses `> age` — when `diff == age`, the condition is **false**, so the price is **accepted**.

**Fuel** — `target_chains/fuel/contracts/pyth-contract/src/main.sw`:

```sway
require(time_difference <= time_period, PythError::OutdatedPrice);
``` [3](#0-2) 

Uses `<=` — accepts at boundary.

**Near** — `target_chains/near/receiver/src/lib.rs`:

```rust
if u64::abs_diff(block_timestamp, price_timestamp.try_into().unwrap()) > age {
    return None;
}
``` [4](#0-3) 

Uses `> age` — accepts at boundary.

**Starknet** — `target_chains/starknet/contracts/src/pyth.cairo`:

```cairo
actual_age <= age
``` [5](#0-4) 

Uses `<=` — accepts at boundary.

**JS SDK** — `price_service/sdk/js/src/index.ts`:

```typescript
if (Math.abs(currentTime - price.publishTime) > age) {
    return undefined;
}
``` [6](#0-5) 

Uses `> age` — accepts at boundary.

**Summary table:**

| Implementation | Condition | At boundary (`age == max_age`) |
|---|---|---|
| Aptos | `age < max_age_secs` | **REJECTS** (stale) |
| EVM Solidity | `diff > age` | **ACCEPTS** (fresh) |
| Fuel | `time_difference <= time_period` | **ACCEPTS** (fresh) |
| Near | `abs_diff > age` | **ACCEPTS** (fresh) |
| Starknet | `actual_age <= age` | **ACCEPTS** (fresh) |
| JS SDK | `abs > age` | **ACCEPTS** (fresh) |

### Impact Explanation

Any caller of `get_price` or `get_price_no_older_than` on Aptos with a price that is exactly `max_age_secs` old will receive an unexpected `stale_price_update` abort. This is inconsistent with the behavior on every other supported chain and with the off-chain JS SDK. An integrator who validates freshness off-chain using the JS SDK (which accepts at boundary) before submitting a transaction to the Aptos contract will experience a failed transaction and wasted gas — the same failure mode described in the reference report. The Aptos contract is one second stricter than the protocol's intended semantics as implemented everywhere else.

---

### Likelihood Explanation

The exact boundary case (`age == max_age_secs`) occurs in normal operation whenever a price update is submitted at the last valid second. It is more likely to be hit by integrators who use tight staleness thresholds (e.g., 10–30 seconds for derivatives). The JS SDK and other off-chain tooling will signal the price as valid, causing the integrator to submit a transaction that reverts on Aptos. Likelihood is low-to-medium given the 1-second window, but the inconsistency is systematic and reproducible.

---

### Recommendation

Update `check_price_is_fresh` in `target_chains/aptos/contracts/sources/pyth.move` to use `<=` instead of `<`, matching all other chain implementations:

```move
fun check_price_is_fresh(price: &Price, max_age_secs: u64) {
    let age = abs_diff(timestamp::now_seconds(), price::get_timestamp(price));
    assert!(age <= max_age_secs, error::stale_price_update());
}
```

Additionally, conduct a cross-chain audit of all staleness boundary comparisons to ensure they are consistent across Aptos, EVM, Fuel, Near, Starknet, and Sui.

---

### Proof of Concept

1. Deploy the Aptos Pyth contract with `stale_price_threshold = 60`.
2. Submit a price update with `publish_time = T`.
3. Advance the Aptos clock to `T + 60` (exactly at the threshold).
4. Call `get_price_no_older_than(price_id, 60)`.
5. **Observed**: transaction aborts with `stale_price_update` (error code 524292).
6. Call the same function with `age = 61` — it succeeds.
7. On EVM with the same parameters (`age = 60`, `diff = 60`): `diff > age` → `60 > 60` → false → **succeeds**.
8. The Aptos test suite itself confirms this boundary behavior at line 1255–1259:

```move
timestamp::update_global_time_for_test_secs(current_timestamp + stale_price_threshold);
// Check that we can access the price if we increase the threshold by 1
assert!(pyth::get_price_no_older_than(
    price_identifier, pyth::get_stale_price_threshold_secs() + 1) == price, 1);
``` [7](#0-6) 

The test itself documents that at exactly `stale_price_threshold` seconds, the price is rejected and `threshold + 1` is required — confirming the off-by-one relative to all other chains.
