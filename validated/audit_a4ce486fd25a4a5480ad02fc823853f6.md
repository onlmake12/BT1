### Title
`getFee()` Returns Non-Zero Fee for Exhausted Entropy Provider While All Request Functions Revert — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `Entropy` contract exposes `getFee(provider)` / `getFeeV2(provider, gasLimit)` as the canonical query functions for determining the cost of a randomness request. However, these functions do not check whether the provider has remaining capacity (`sequenceNumber < endSequenceNumber`). When a provider is exhausted, every request entry-point (`request`, `requestWithCallback`, `requestV2`) reverts with `OutOfRandomness`, yet `getFee()` continues to return a non-zero fee — a direct state/interface mismatch analogous to the ERC4626 `maxDeposit`/`deposit` discrepancy.

---

### Finding Description

In `requestHelper`, the contract enforces a hard capacity check before accepting any randomness request:

```solidity
// Entropy.sol line 228-230
uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
    revert EntropyErrors.OutOfRandomness();
``` [1](#0-0) 

Every public request entry-point (`request`, `requestWithCallback`, `requestV2`) routes through `requestHelper`, so all of them revert when the provider is exhausted. [2](#0-1) 

Meanwhile, `getFee` and `getFeeV2` compute the fee purely from the provider's configured `feeInWei` and the Pyth protocol fee, with no check on remaining capacity:

```solidity
// Entropy.sol lines 744-764
function getFee(address provider) public view override returns (uint128 feeAmount) {
    return getFeeV2(provider, 0);
}

function getFeeV2(address provider, uint32 gasLimit)
    public view override returns (uint128 feeAmount)
{
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
``` [3](#0-2) 

`getProviderFee` likewise only reads `feeInWei` and `defaultGasLimit` — it never inspects `sequenceNumber` vs `endSequenceNumber`. [4](#0-3) 

The interface documentation explicitly instructs callers to use `getFee(provider)` to determine the required payment before calling `request`:

> "This method will revert unless the caller provides a sufficient fee (at least `getFee(provider)`) as msg.value." [5](#0-4) 

There is no `isProviderAvailable()`, `getMaxRequests()`, or equivalent function. The `ProviderInfo` struct does expose `sequenceNumber` and `endSequenceNumber` via `getProviderInfoV2`, but this requires callers to manually compute the exhaustion condition — it is not surfaced by the fee query functions that the interface directs integrators to use. [6](#0-5) 

---

### Impact Explanation

Any smart contract or off-chain integration that follows the documented pattern of:
1. Calling `getFee(provider)` to determine the required payment
2. Calling `request(provider, ...)` / `requestV2(provider, ...)` with that value

will have its transaction revert with `OutOfRandomness` when the provider is exhausted, even though `getFee()` returned a valid non-zero fee. The ETH sent is returned on revert, so there is no direct fund loss, but:

- **Gas is wasted** on every failed attempt.
- **Smart contract integrators** that do not explicitly handle the `OutOfRandomness` revert will propagate the failure to their callers, potentially breaking downstream protocol logic (e.g., a DeFi protocol that depends on Entropy for randomness in a critical path).
- **Off-chain bots/keepers** that use `getFee()` as a liveness signal will be misled into submitting transactions that always revert during the exhaustion window.

The Fortuna keeper service re-registers providers when they exhaust their chain, but there is a window between exhaustion and re-registration during which this mismatch is live. [7](#0-6) 

---

### Likelihood Explanation

Providers commit to a finite hash chain (`chainLength`) at registration time. When `sequenceNumber` reaches `endSequenceNumber`, the provider is exhausted and must re-register. This is a normal, expected operational event — not a hypothetical edge case. Any integrator that does not independently check `getProviderInfoV2().sequenceNumber < getProviderInfoV2().endSequenceNumber` before calling `request()` is exposed to this mismatch during every exhaustion window. The default Pyth provider (Fortuna) is the most commonly used provider, so exhaustion of the default provider affects the widest set of integrators. [8](#0-7) 

---

### Recommendation

Add a capacity check inside `getFeeV2(address provider, uint32 gasLimit)` (or introduce a dedicated `isProviderAvailable(address provider) external view returns (bool)` function) so that the fee query surface accurately reflects whether a request would succeed:

```solidity
function getFeeV2(address provider, uint32 gasLimit)
    public view override returns (uint128 feeAmount)
{
    EntropyStructsV2.ProviderInfo storage info = _state.providers[provider];
    if (info.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    if (info.sequenceNumber >= info.endSequenceNumber)
        revert EntropyErrors.OutOfRandomness();
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
```

Alternatively, document clearly that `getFee()` returning a non-zero value does **not** guarantee that a request will succeed, and direct integrators to check `getProviderInfoV2()` for capacity before calling any request function.

---

### Proof of Concept

1. Provider registers with `chainLength = 1` (minimum), so `endSequenceNumber = sequenceNumber + 1 = 1`.
2. One user calls `request(provider, ...)` — succeeds, `sequenceNumber` is now `1 == endSequenceNumber`.
3. A second user (or smart contract) calls `getFee(provider)` — returns a non-zero fee (e.g., `1000 wei`).
4. The second user calls `request(provider, commitment, false){value: 1000 wei}` — **reverts with `OutOfRandomness`**.
5. Despite the revert, gas is consumed and any smart contract that did not wrap the call in a try/catch propagates the failure.

The mismatch is confirmed by the code path: `getFee` → `getFeeV2` → `getProviderFee` (reads only `feeInWei`, never `sequenceNumber`/`endSequenceNumber`) vs. `request` → `requestHelper` → hard revert at line 229–230. [9](#0-8)

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }
    }
```
