### Title
Delegators Can Escape Slashing by Calling `undelegate` Before `slash` Is Applied — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`, `governance/pyth_staking_sdk/src/types/integrity-pool.ts`)

---

### Summary

The OIS integrity pool's `undelegate` instruction does not require the delegator's `DelegationRecord` to be current (i.e., all pending slash events applied) before allowing a position to be reduced or closed. Because `slash` is a separate, per-delegator instruction that applies slash events sequentially via `DelegationRecord.next_slash_event_index`, a delegator who observes a newly created `SlashEvent` on-chain can front-run the slash by calling `undelegate` first, exiting their position before the slash is applied and escaping the penalty entirely.

---

### Finding Description

The integrity pool program exposes two relevant instructions:

**`undelegate` (position exit):** Takes `position_index` and `amount` as arguments. Its account list includes `owner`, `poolData`, `poolConfig`, `publisher`, `configAccount`, `stakeAccountPositions`, `stakeAccountMetadata`, `stakeAccountCustody`, `stakingProgram`, and `systemProgram`. Critically, it does **not** include `delegationRecord`. [1](#0-0) 

**`slash` (slash application):** Takes a `SlashEvent` PDA and the delegator's `delegationRecord` (writable). It applies the slash to `stakeAccountPositions` and increments `delegationRecord.next_slash_event_index` to mark the event as processed. [2](#0-1) 

The `DelegationRecord` struct tracks two fields: `last_epoch` (for reward settlement) and `next_slash_event_index` (for sequential slash application). [3](#0-2) 

Because `undelegate` never loads or validates `delegationRecord`, it cannot enforce that all pending slash events (those with index ≥ `next_slash_event_index`) have been applied before the position is reduced. A delegator can therefore close their position while unapplied slash events exist for their publisher.

The TypeScript SDK's `unstakeFromPublisher` and `unstakeFromAllPublishers` methods confirm this: they build `undelegate` instructions with no delegation record account and no pre-flight check for pending slashes. [4](#0-3) 

---

### Impact Explanation

A delegator staked to a publisher that is subsequently slashed can observe the `SlashEvent` PDA creation on-chain and immediately call `undelegate` to remove their entire position before `slash` is invoked on their account. When `slash` is later called for that delegator, the position is already gone (or reduced), so the slash applies to zero tokens. The delegator suffers no penalty despite having been staked during the misbehavior epoch.

This directly undermines the OIS slashing guarantee: "Both self-stakers and delegators are slashed proportionally to their staked amount in the impacted pools." The protocol's security model — that staked capital is at risk for inaccurate data — is broken for any delegator who acts before the slash is applied. [5](#0-4) 

---

### Likelihood Explanation

- Slash events are created by the Pythian Council as on-chain PDAs (`SlashEvent` accounts), visible to all participants immediately upon creation.
- The slash application is a separate per-delegator transaction that must be submitted for each staker individually. This creates a non-trivial window between event creation and full application across all delegators.
- Any delegator (unprivileged staking user) can monitor the chain for new `SlashEvent` PDAs and submit `undelegate` in the same or next block.
- The `undelegate` instruction requires only the position owner's signature — no privileged role is needed.
- The SDK's `unstakeFromAllPublishers` makes this trivially scriptable. [6](#0-5) 

---

### Recommendation

The `undelegate` instruction should require the delegator's `DelegationRecord` as an account and enforce that `next_slash_event_index` equals `pool_data.num_slash_events[publisher_index]` before allowing any position reduction. Equivalently, the program should require that `advance_delegation_record` (which processes all pending slashes) has been called for the relevant publisher in the current epoch before `undelegate` is permitted. The existing `OutdatedDelegatorAccounting` error code (6008) suggests this guard was intended but is not enforced in `undelegate`. [7](#0-6) 

---

### Proof of Concept

1. Delegator D stakes 1,000,000 PYTH to publisher P's pool (position enters `LOCKED` state after warmup).
2. The Pythian Council creates a `SlashEvent` for publisher P at index `i` with slash rate `w = 5%`.
3. Delegator D observes the `SlashEvent` PDA on-chain.
4. Delegator D calls `undelegate(position_index, 1_000_000)` — no `delegationRecord` account required, transaction succeeds, position is closed.
5. Anyone later calls `slash(index=i)` for delegator D's `stakeAccountPositions`. The position is already zero; the slash applies to 0 tokens.
6. Delegator D retains all 1,000,000 PYTH, having escaped the 50,000 PYTH slash they should have incurred.

The `undelegate` IDL confirms no `delegation_record` account is present, making step 4 unconditionally executable regardless of pending slash state. [8](#0-7)

  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }

  public async unstakeFromAllPublishers(
    stakeAccountPositions: PublicKey,
    positionStates: (PositionState.LOCKED | PositionState.LOCKING)[],
  ) {
    const [stakeAccountPositionsData, currentEpoch] = await Promise.all([
      this.getStakeAccountPositions(stakeAccountPositions),
      getCurrentEpoch(this.connection),
    ]);

    const instructions = await Promise.all(
      stakeAccountPositionsData.data.positions
        .map((position, index) => {
          const publisher =
            position.targetWithParameters.integrityPool?.publisher;
          return publisher === undefined
            ? undefined
            : { index, position, publisher };
        })
        // By separating this filter from the next, typescript can narrow the
        // type and automatically infer that there will be no `undefined` values
        // in the array after this line.  If we combine those filters,
        // typescript won't narrow properly.
        .filter((positionInfo) => positionInfo !== undefined)
        .filter(({ position }) =>
          (positionStates as PositionState[]).includes(
            getPositionState(position, currentEpoch),
          ),
        )
        .reverse()
        .map(({ position, index, publisher }) =>
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({ publisher, stakeAccountPositions })
            .instruction(),
        ),
    );

    return sendTransaction(instructions, this.connection, this.wallet);
  }
