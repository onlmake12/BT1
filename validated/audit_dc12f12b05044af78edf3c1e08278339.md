### Title
Governance VAA Out-of-Order Submission Permanently Blocks Lower-Sequence Actions — (`lazer/contracts/sui/sources/governance.move`)

### Summary

`governance::process_incoming` uses a monotonic high-watermark (`seen_sequence`) with a strict `<` guard. Any unprivileged party who possesses two valid Wormhole-signed governance VAAs can submit the higher-sequence one first, permanently making the lower-sequence VAA unprocessable. If the skipped VAA was an `UpdateTrustedSigner` action, the new signer can never be added; when the existing signer expires, all price-update verification fails and downstream user funds are frozen.

### Finding Description

`governance::process_incoming` enforces replay protection with a single monotonic counter:

```move
assert!(self.seen_sequence < sequence, EOldSequenceNumber);
self.seen_sequence = sequence;
``` [1](#0-0) 

Once `seen_sequence` is set to `N+1`, any attempt to process a VAA with sequence `N` evaluates `N+1 < N` → `false` → permanent abort with `EOldSequenceNumber`. There is no "pending" queue, no gap-fill mechanism, and no way to reset `seen_sequence`.

The two public entrypoints that call through to `process_incoming` via `state::unwrap_ptgm`:

```move
public fun update_trusted_signer(state: &mut State, vaa: VAA) { ... }
public fun upgrade(state: &mut State, vaa: VAA): UpgradeTicket { ... }
``` [2](#0-1) 

are `public` (not `entry`), callable by any Sui Programmable Transaction Block. Wormhole VAAs are publicly observable on-chain; any party who sees two pending governance VAAs can choose their submission order.

`unwrap_ptgm` adds no additional ordering enforcement — it simply delegates to `process_incoming`:

```move
let payload = self.governance.process_incoming(vaa);
``` [3](#0-2) 

### Impact Explanation

If the skipped VAA is an `UpdateTrustedSigner` action (action byte `1`): [4](#0-3) 

the new trusted signer is never added. `TrustedSignerInfo` entries carry an `expires_at` field: [5](#0-4) 

Once the existing signer's `expires_at` passes, all Lazer price-update verification fails, freezing any downstream protocol that depends on verified price feeds. This matches the stated scope: *locking of price-update capability, freezing downstream user funds*.

### Likelihood Explanation

- Governance VAAs are broadcast publicly on Wormhole and can be observed by any party before on-chain submission.
- The two public functions require no special capability or role — any Sui address can call them.
- A single malicious (or even accidentally mis-ordered) relayer transaction is sufficient to trigger the permanent block.
- The governance system has no recovery path: `seen_sequence` can only increase, and there is no admin reset function.

### Recommendation

Replace the high-watermark with a set-based replay-protection store (e.g., a `Table<u64, bool>` of processed sequences). Alternatively, if strict ordering is intentional, use a `>=` check that rejects-but-does-not-permanently-block out-of-order submissions (i.e., return a retriable error rather than permanently advancing the watermark past unprocessed sequences). The simplest safe fix:

```move
// Instead of:
assert!(self.seen_sequence < sequence, EOldSequenceNumber);
self.seen_sequence = sequence;

// Use a processed-set:
assert!(!self.processed_sequences.contains(sequence), EAlreadyProcessed);
self.processed_sequences.add(sequence, true);
```

### Proof of Concept

State-level test (no privileged access needed):

1. Initialize `Governance` with `seen_sequence = 0`.
2. Call `update_trusted_signer` with a VAA of sequence **5** (N+1). Succeeds; `seen_sequence = 5`.
3. Call `update_trusted_signer` with a VAA of sequence **4** (N, the "add new signer" VAA). Aborts with `EOldSequenceNumber` because `5 < 4` is false.
4. Assert that the new signer's public key is absent from `state.trusted_signers`.
5. Advance the clock past the existing signer's `expires_at`; assert that price-update verification now fails.

The existing test infrastructure in `actions.move` already provides `TEST_UPDATE_TRUSTED_SIGNER_ADD` (sequence 2) and `TEST_UPDATE_TRUSTED_SIGNER_REMOVE` (sequence 4) VAAs — processing sequence 4 before sequence 2 reproduces the exact failure. [6](#0-5)

public fun update_trusted_signer(state: &mut State, vaa: VAA) {
    let current_cap = state.current_cap();
    let (header, mut parser) = state.unwrap_ptgm(&current_cap, vaa);
    assert!(header.is_update_trusted_signer(), ENotUpdateTrustedSigner);

    let public_key = parser.take_bytes(secp256k1_compressed_pubkey_len());
    let expires_at = parser.take_u64_be();
    parser.destroy_empty();
    state.update_trusted_signer(&current_cap, public_key, expires_at);
}
