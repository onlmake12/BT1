Audit Report

## Title
Insufficient RBF Fee Validation Before Expensive Script Execution Enables CPU Exhaustion - (File: `tx-pool/src/process.rs`)

## Summary

In `tx-pool/src/process.rs`, the `_process_tx` function runs full CKB-VM script execution (`verify_rtx`) after `pre_check` passes only a `min_fee_rate` check for RBF-candidate transactions. The stricter `min_replace_fee` threshold (Rule #3/#4), unconfirmed-input constraint (Rule #2), and replacement-count limit (Rule #5) are only enforced inside `submit_entry` under write lock, after script execution has already completed. An unprivileged attacker can exploit this ordering to force repeated full VM execution on transactions guaranteed to be rejected, causing CPU exhaustion on any node with RBF enabled.

## Finding Description

The `_process_tx` function in `tx-pool/src/process.rs` follows this sequence:

**Step 1 ŌĆö `pre_check` (read lock, lines 715ŌĆō717):** When a transaction's input resolves as `Dead`, the RBF path is taken at line 292. It calls `check_tx_fee` (line 294), which only verifies `fee >= min_fee_rate` (implemented in `tx-pool/src/util.rs` lines 45ŌĆō52). It then calls `find_conflict_outpoint` (line 295) to confirm a conflict exists, and returns `Ok`. The code comment at line 305 explicitly acknowledges this: *"we also return Ok here, so that the entry will be continue to be verified before submit"*.

**Step 2 ŌĆö `verify_rtx` (lines 724ŌĆō732):** Full `ContextualTransactionVerifier` execution, including CKB-VM lock/type script execution, runs unconditionally on the result of `pre_check`.

**Step 3 ŌĆö `submit_entry` (lines 753ŌĆō754, write lock):** Only here does `check_rbf` run (line 106 of `process.rs`, implemented in `tx-pool/src/pool.rs` lines 574ŌĆō676), enforcing Rule #2 (no new unconfirmed inputs, line 602ŌĆō609), Rule #3/#4 (`fee >= min_replace_fee = sum(replaced_fees) + extra_rbf_fee`, lines 664ŌĆō671), and Rule #5 (Ōēż100 replacement candidates, lines 619ŌĆō623).

The critical gap: `check_tx_fee` checks `fee >= min_fee_rate` (e.g., 1000 shannons/KB per `util/app-config/src/legacy/tx_pool.rs` line 10), while `check_rbf` Rule #3/#4 requires `fee >= min_replace_fee`, which equals the sum of all replaced transactions' fees plus an additional RBF increment (`calculate_min_replace_fee` in `tx-pool/src/pool.rs` lines 102ŌĆō127). These thresholds are entirely different. A transaction paying just above `min_fee_rate` but far below `min_replace_fee` passes `pre_check`, triggers full VM execution, and is only rejected in `submit_entry`.

The existing `check_txid_collision` guard (line 282) only blocks exact `proposal_short_id` duplicates. Since `proposal_short_id` is derived from `tx.hash()` (which excludes witnesses but includes outputs), the attacker can vary output capacities by 1 shannon per attempt to generate fresh txids, bypassing both `check_txid_collision` and `verify_queue_contains` (line 349) on every submission.

## Impact Explanation

**High ŌĆö Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker can saturate the tx-pool's async verification workers by continuously submitting RBF-candidate transactions that pass `pre_check` but fail `check_rbf`. Each submission forces full CKB-VM script execution (potentially millions of cycles per transaction) at zero cost to the attacker beyond the RPC/P2P message overhead. The verify queue workers will be occupied processing these doomed transactions, degrading or blocking legitimate transaction processing on the targeted node.

## Likelihood Explanation

- RBF must be enabled (`min_rbf_rate > min_fee_rate`), which is the default configuration (`min_fee_rate = 1000`, `min_rbf_rate = 1500` per `resource/ckb.toml` lines 212ŌĆō214).
- The attacker only needs to identify one transaction currently in the pool, trivially observable via `get_raw_tx_pool` RPC or P2P relay observation.
- No privileged access is required ŌĆö any RPC caller or P2P peer can submit transactions.
- The `check_txid_collision` guard only blocks exact `proposal_short_id` duplicates; the attacker varies output capacities by 1 shannon per attempt to generate fresh txids.
- The `verify_queue_contains` check is bypassed the same way.
- The attack is fully repeatable and cheap: the attacker pays only network/RPC overhead; the node pays full VM execution cost per attempt.

## Recommendation

Move a partial RBF fee adequacy check into `pre_check` under the read lock, before the transaction is enqueued for script verification. Specifically:

- Compute a conservative `min_replace_fee` using the directly conflicting transaction's fee (readable under the read lock via `find_conflict_outpoint` + pool entry lookup) and reject early if `fee < min_replace_fee`.
- Optionally, also check Rule #2 (no new unconfirmed inputs) in `pre_check`, as this requires only a snapshot read.

The full atomicity-sensitive conflict resolution (Rule #5, descendant graph traversal) can remain in `submit_entry` under the write lock. The fee-adequacy and input-validity checks do not require write-lock exclusivity and can be evaluated cheaply upfront to prevent wasted VM cycles.

## Proof of Concept

1. Submit a high-fee transaction `tx_A` spending `cell_X` to the pool (enters pending state). Observe its fee, e.g., 10 CKB.
2. Craft `tx_B` spending the same `cell_X`, with fee = `min_fee_rate * size` (e.g., 363 shannons ŌĆö far below `min_replace_fee` of ~10 CKB + extra RBF increment). Set output capacity to `input_capacity - 363`.
3. Submit `tx_B` via `send_transaction` RPC.
4. **`pre_check`**: `resolve_tx(..., rbf=true)` succeeds; `check_tx_fee` passes (363 Ōēź min_fee); `find_conflict_outpoint` finds `tx_A` ŌåÆ returns `Ok`.
5. **`verify_rtx`**: Full CKB-VM script execution runs on `tx_B` (potentially millions of cycles).
6. **`submit_entry` ŌåÆ `check_rbf`**: Rule #3/#4: `363 < 10_CKB + extra` ŌåÆ `Err(RBFRejected)`.
7. `tx_B` is recorded in the conflicts pool; node has wasted full VM cycles.
8. Repeat steps 2ŌĆō7 with fresh `tx_B` variants (change output capacity by 1 shannon each time to change txid, bypassing `check_txid_collision`).
9. With sufficient submission rate, the verify queue workers are saturated, blocking legitimate transaction processing.

This is confirmed by the existing integration test `RbfSameInputwithLessFee` in `test/src/specs/tx_pool/replace.rs` lines 218ŌĆō265, which demonstrates that a transaction with insufficient fee to replace is accepted through `pre_check` and only rejected at `submit_entry`, with the rejected tx recorded in the conflicts pool ŌĆö proving the full VM execution path is traversed before rejection.
