### Title
Malicious Entropy Requester Can Permanently Block `revealWithCallback` Fulfillment via Reverting `_entropyCallback` in the Legacy Path — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, `revealWithCallback` has two execution branches. When a provider has not set a `defaultGasLimit` (i.e., `providerInfo.defaultGasLimit == 0`), every request is stored with `req.gasLimit10k = 0`. This causes `revealWithCallback` to take the **legacy path**, which makes a **direct, uncaught external call** to `IEntropyConsumer(callAddress)._entropyCallback(...)`. A malicious requester contract can implement `_entropyCallback` to always revert, causing the entire `revealWithCallback` transaction to revert and permanently preventing the provider's keeper (Fortuna) from fulfilling the request.

---

### Finding Description

`revealWithCallback` branches on `req.gasLimit10k`:

```
if (req.gasLimit10k != 0 && req.callbackStatus == CALLBACK_NOT_STARTED) {
    // NEW SAFE PATH: uses excessivelySafeCall — catches reverts, emits CallbackFailed
} else {
    // LEGACY PATH: direct call — revert propagates to caller
    clearRequest(provider, sequenceNumber);
    if (len != 0) {
        IEntropyConsumer(callAddress)._entropyCallback(...);  // ← uncaught
    }
}
```

`req.gasLimit10k` is set to `0` in `requestHelper` whenever `providerInfo.defaultGasLimit == 0`:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;  // legacy path, no safe-call protection
}
```

`defaultGasLimit` defaults to `0` for every provider that has not explicitly called `setDefaultGasLimit`. This means **all legacy providers** (and any new provider that has not opted into the new flow) produce requests that land in the unprotected branch.

In the legacy branch, `clearRequest` is called before the external call (checks-effects-interactions). However, if `_entropyCallback` reverts, the **entire transaction reverts**, undoing `clearRequest`. The request remains in `CALLBACK_NOT_STARTED` state and can never be cleared, because every subsequent `revealWithCallback` attempt will also revert.

**Attack path:**
1. Attacker deploys a contract whose `_entropyCallback` unconditionally reverts.
2. Attacker calls `requestWithCallback` against any provider with `defaultGasLimit == 0`.
3. Provider's Fortuna keeper calls `revealWithCallback` — always reverts.
4. The request is permanently stuck; the keeper wastes gas on every retry.

---

### Impact Explanation

- The provider's Fortuna keeper service enters an infinite retry loop for the stuck request, wasting gas on every attempt.
- The request can never be cleared from contract storage, permanently consuming a slot.
- If many such requests are created, the keeper's operational costs increase significantly (griefing at scale).
- The user's application never receives its random number, breaking any downstream logic that depends on the callback.

This is analogous to the Moonwell M-01 finding: just as an ERC777 `tokensReceived` hook allows a borrower to revert the liquidation transaction, a reverting `_entropyCallback` allows a requester to revert the fulfillment transaction, permanently blocking a critical protocol operation.

---

### Likelihood Explanation

- Any unprivileged user can trigger this by deploying a contract with a reverting `_entropyCallback` and calling `requestWithCallback`.
- The default state for all providers is `defaultGasLimit == 0`, so the vulnerable legacy path is the **default** for the majority of deployed providers.
- The attacker pays the request fee, but this is a one-time cost per stuck slot; the provider bears ongoing gas costs.
- No special privileges, leaked keys, or governance access are required.

---

### Recommendation

1. **Deprecate the legacy path**: Require all providers to set a non-zero `defaultGasLimit` before accepting `requestWithCallback` calls. Revert if `providerInfo.defaultGasLimit == 0` and a callback is requested.
2. **Alternatively**, apply `excessivelySafeCall` unconditionally in `revealWithCallback` regardless of `gasLimit10k`, using a reasonable fallback gas limit when `gasLimit10k == 0`.
3. **At minimum**, document clearly that providers must call `setDefaultGasLimit` to protect their keepers from this griefing vector.

---

### Proof of Concept

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` (line 999 of `Entropy.t.sol`) already demonstrates this behavior:

```solidity
// Provider has defaultGasLimit == 0 (legacy path)
EntropyConsumer consumer = new EntropyConsumer(address(random), true); // true = revert on callback
vm.deal(address(consumer), fee);
vm.startPrank(address(consumer));
uint64 assignedSequenceNumber = random.requestWithCallback{value: fee}(
    provider1,
    userRandomNumber
);

vm.expectRevert(); // ← revealWithCallback always reverts
random.revealWithCallback(
    provider1,
    assignedSequenceNumber,
    userRandomNumber,
    provider1Proofs[assignedSequenceNumber]
);
```

The request remains permanently active after this revert, as confirmed by the test — the provider cannot fulfill it.
