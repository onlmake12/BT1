### Title
Entropy `requestsOverflow` Unbounded Storage Bloat via Zero-Fee Requests — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `Entropy` contract allows any address to permissionlessly register as a randomness provider with `feeInWei = 0`, and the protocol-level `pythFeeInWei` can be initialized to zero or set to a negligibly small value with no enforced minimum. An attacker can register their own provider at zero cost, make unlimited requests that are never revealed, and cause unrevealed requests to permanently accumulate in the unbounded `requestsOverflow` mapping. On low-gas-cost chains (Polygon, BNB Chain, Arbitrum), this bloat attack is feasible at near-zero economic cost.

---

### Finding Description

**Root cause 1 — No minimum provider fee:**

The `register()` function accepts `feeInWei = 0` with no lower-bound check:

```solidity
function register(
    uint128 feeInWei,   // can be 0 — no minimum enforced
    bytes32 commitment,
    bytes calldata commitmentMetadata,
    uint64 chainLength,
    bytes calldata uri
) public override {
    if (chainLength == 0) revert EntropyErrors.AssertionFailure();
    // ...
    provider.feeInWei = feeInWei;
``` [1](#0-0) 

**Root cause 2 — No minimum protocol fee:**

`pythFeeInWei` is accepted as-is during initialization and can be set to 0 by the admin with no lower-bound guard:

```solidity
_state.pythFeeInWei = pythFeeInWei;   // no minimum enforced
``` [2](#0-1) 

```solidity
function setPythFee(uint128 newPythFee) external {
    _authoriseAdminAction();
    uint oldPythFee = _state.pythFeeInWei;
    _state.pythFeeInWei = newPythFee;   // can be set to 0
``` [3](#0-2) 

The total required fee is `providerFee + pythFeeInWei`. When both are 0, `requestHelper` passes the fee check with `msg.value = 0`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [4](#0-3) 

**Root cause 3 — Unrevealed requests accumulate permanently in `requestsOverflow`:**

The `requestsOverflow` mapping is unbounded. Entries are only removed when `clearRequest` is called during `reveal` or `revealWithCallback`. An attacker who never reveals their requests causes permanent accumulation:

```solidity
mapping(bytes32 => EntropyStructsV2.Request) requestsOverflow;
``` [5](#0-4) 

The `allocRequest` function explicitly acknowledges this risk:

```solidity
// There is a chance that some requests never get revealed and remain active forever.
``` [6](#0-5) 

**Root cause 4 — Unlimited chain length:**

The attacker can set `chainLength = type(uint64).max` to obtain essentially unlimited sequence numbers per registration, and re-registration is also free:

```solidity
provider.endSequenceNumber = provider.sequenceNumber + chainLength;
``` [7](#0-6) 

---

### Impact Explanation

**Storage bloat:** The `requestsOverflow` mapping grows without bound. Each `EntropyStructsV2.Request` occupies 3 EVM storage slots (~96 bytes). Millions of unrevealed requests permanently bloat the contract's on-chain storage.

**Gas griefing:** The fixed-size `requests[32]` array has only 32 slots. When a new request hashes to an occupied slot, `allocRequest` evicts the existing request to `requestsOverflow` at a cost of ~20,000 gas (cold SSTORE), paid by the transaction sender of the new request. If an attacker fills all 32 slots with unrevealed requests, every subsequent legitimate user request triggers this expensive eviction path. [8](#0-7) 

---

### Likelihood Explanation

The attack requires `pythFeeInWei` to be 0 or negligibly small. The test suite initializes it to `7` wei:

```solidity
uint128 pythFeeInWei = 7;
``` [9](#0-8) 

On chains like Polygon (gas ~30 gwei, MATIC ~$0.50), 7 wei is economically negligible. The attacker registers their own provider with `feeInWei = 0` — a permissionless, free operation — and then makes requests paying only gas. On Polygon, each request costs roughly $0.001 in gas. For $100, an attacker can submit ~100,000 unrevealed requests, permanently bloating `requestsOverflow` and forcing all 32 array slots to be perpetually occupied.

---

### Recommendation

1. **Enforce a meaningful minimum `pythFeeInWei`** that makes the attack economically infeasible on all supported chains. The fee should be calibrated per-chain to represent a non-trivial cost (analogous to the Polkadot Existential Deposit recommendation of at least `1 CENTS`).

2. **Enforce a minimum provider `feeInWei`** in `register()` to prevent zero-cost provider registrations from being used as attack vehicles.

3. **Consider a maximum `chainLength`** per registration to limit the number of sequence numbers an attacker can pre-allocate in a single call.

---

### Proof of Concept

```solidity
// Attacker steps (on a chain where pythFeeInWei = 0 or negligible):

// 1. Register attacker-controlled provider with feeInWei = 0 and max chain length
entropy.register(
    0,                          // feeInWei = 0
    attackerCommitment,
    "",
    type(uint64).max,           // unlimited sequence numbers
    ""
);

// 2. Make ~112 requests (birthday bound for 32 slots) — never reveal them
for (uint i = 0; i < 112; i++) {
    entropy.request{value: 0}(attackerProvider, userCommitment, false);
}
// All 32 array slots are now occupied by attacker's unrevealed requests.

// 3. Every legitimate user request now triggers an expensive eviction:
//    allocRequest() writes attacker's request to requestsOverflow (20k gas)
//    paid by the legitimate user's transaction.

// 4. Attacker repeats step 2 to refill slots, keeping the attack active.
//    requestsOverflow grows unboundedly with permanently unrevealed entries.
``` [10](#0-9) [5](#0-4)

```text
    function allocRequest(
        address provider,
        uint64 sequenceNumber
    ) internal returns (EntropyStructsV2.Request storage req) {
        (, uint8 shortKey) = requestKey(provider, sequenceNumber);

        req = _state.requests[shortKey];
        if (isActive(req)) {
            // There's already a prior active request in the storage slot we want to use.
            // Overflow the prior request to the requestsOverflow mapping.
            // It is important that this code overflows the *prior* request to the mapping, and not the new request.
            // There is a chance that some requests never get revealed and remain active forever. We do not want such
            // requests to fill up all of the space in the array and cause all new requests to incur the higher gas cost
            // of the mapping.
            //
            // This operation is expensive, but should be rare. If overflow happens frequently, increase
            // the size of the requests array to support more concurrent active requests.
            (bytes32 reqKey, ) = requestKey(req.provider, req.sequenceNumber);
            _state.requestsOverflow[reqKey] = req;
        }
    }
