### Title
Missing Per-Method Role-Based Access Control in RPC Pool and Net Modules Allows Any Caller to Execute Privileged Node Operations — (File: `rpc/src/module/pool.rs`, `rpc/src/module/net.rs`)

---

### Summary

The CKB RPC server implements access control exclusively at the module-enable level (e.g., `Pool`, `Net`). Once a module is enabled, every method within it — including destructive, privileged operations — is equally callable by any reachable RPC client with no per-method authorization check. This directly mirrors the reported vulnerability class: a single flat "owner" role (here: "can reach the RPC") controls all operations, with no finer-grained role separation. Any caller who can reach the RPC endpoint can invoke `clear_tx_pool`, `clear_tx_verify_queue`, `remove_transaction`, `set_ban`, `clear_banned_addresses`, `set_network_active`, `add_node`, and `remove_node` without any credential or role check.

---

### Finding Description

**Root cause — Pool module:**

`clear_tx_pool()` and `clear_tx_verify_queue()` in `rpc/src/module/pool.rs` perform no caller identity or authorization check before wiping the entire transaction pool or verification queue:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
``` [1](#0-0) 

`remove_transaction()` similarly removes any specific transaction from the pool with no authorization: [2](#0-1) 

**Root cause — Net module:**

`clear_banned_addresses()` and `set_ban()` in `rpc/src/module/net.rs` perform no caller authorization check. `set_ban` only validates IP format and the `insert`/`delete` command string — no identity check: [3](#0-2) 

`set_network_active`, `add_node`, and `remove_node` are similarly unguarded in the same file.

**Root cause — Service builder (module-only gating):**

The `ServiceBuilder` in `rpc/src/service_builder.rs` is the sole access control mechanism. It gates entire modules on/off via config flags (`pool_enable()`, `net_enable()`, etc.), but applies no per-method authorization within a module: [4](#0-3) 

Once `Pool` or `Net` is enabled, all methods — read-only and destructive alike — are equally reachable by any caller. There is no concept of a "read-only" vs. "admin" caller role.

**The module list in the default config confirms `Pool` and `Net` are enabled by default:** [5](#0-4) 

---

### Impact Explanation

**`clear_tx_pool` / `clear_tx_verify_queue`:** Any reachable RPC caller can atomically wipe all pending and queued transactions from the node's pool. All user-submitted transactions are silently dropped and must be resubmitted. This is a targeted, zero-cost service disruption attack against the node's mempool — directly analogous to the "privileged admin wipes state" impact in the reference report.

**`remove_transaction`:** Any caller can selectively evict a specific transaction (e.g., a high-value pending transfer) from the pool, forcing resubmission and potentially enabling fee-sniping or MEV-adjacent manipulation.

**`set_ban` / `clear_banned_addresses`:** Any caller can ban legitimate peers (isolating the node from the network) or unban previously-banned malicious peers (re-exposing the node to known-bad actors). This undermines the node's network security posture.

**`set_network_active` / `add_node` / `remove_node`:** Any caller can disable networking entirely or manipulate peer connections.

---

### Likelihood Explanation

The RPC binds to `127.0.0.1:8114` by default, which limits exposure to local processes. However:

1. Many node operators expose the RPC to external networks to serve wallets, dApps, or monitoring tools — this is a documented and common deployment pattern.
2. Any process running on the same host (e.g., a compromised web server, a malicious npm package in a co-located service) can reach `127.0.0.1:8114` without any credential.
3. There is no authentication mechanism (no token, no API key, no TLS client cert) in the RPC server at all — the only protection is network-level reachability.
4. The `Pool` and `Net` modules are enabled in the default production config, so the attack surface is present in all default deployments.

Likelihood is **moderate**: requires RPC reachability (either local process or exposed endpoint), but no credentials or special knowledge beyond the method name.

---

### Recommendation

Implement per-method role-based access control within the RPC server. Specifically:

1. Introduce a caller role model (e.g., `ReadOnly`, `User`, `Admin`) enforced at the method handler level, not just at the module-enable level.
2. Gate destructive operations (`clear_tx_pool`, `clear_tx_verify_queue`, `remove_transaction`, `set_ban`, `clear_banned_addresses`, `set_network_active`) behind an `Admin` role requiring a configurable bearer token or IP allowlist.
3. Allow node operators to expose read-only and user-facing methods (`send_transaction`, `get_peers`, `tx_pool_info`) without simultaneously exposing admin-destructive methods.
4. At minimum, add a separate `admin_token` config field and require it as a header for state-mutating privileged methods.

---

### Proof of Concept

**Preconditions:**
- CKB node running with default config (`Pool` and `Net` modules enabled).
- Attacker can reach `127.0.0.1:8114` (local process) or the operator has exposed the RPC to the network.

**Step 1 — Confirm pending transactions exist:**
```json
{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":1}
```
Response shows `pending > 0`.

**Step 2 — Wipe the entire transaction pool (no credentials required):**
```json
{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":2}
```
Response: `{"result": null}` — success, no authorization challenge.

**Step 3 — Confirm pool is empty:**
```json
{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":3}
```
Response shows `pending: 0` — all user transactions have been silently dropped.

**Step 4 — Isolate the node from the network (no credentials required):**
```json
{"jsonrpc":"2.0","method":"set_ban","params":["0.0.0.0/0","insert","0x174876E800",true,"attacker"],"id":4}
```
Bans the entire IPv4 address space, disconnecting all peers.

The handler at `rpc/src/module/net.rs:691` executes `self.network_controller.ban(...)` directly with no authorization gate. [6](#0-5)

    fn set_ban(
        &self,
        address: String,
        command: String,
        ban_time: Option<Timestamp>,
        absolute: Option<bool>,
        reason: Option<String>,
    ) -> Result<()> {
        let ip_network = address.parse().map_err(|_| {
            RPCError::invalid_params(format!(
                "Expected `params[0]` to be a valid IP address, got {address}"
            ))
        })?;

        match command.as_ref() {
            "insert" => {
                let ban_until = if absolute.unwrap_or(false) {
                    ban_time.unwrap_or_default().into()
                } else {
                    unix_time_as_millis()
                        + ban_time
                            .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
                            .value()
                };
                self.network_controller
                    .ban(ip_network, ban_until, reason.unwrap_or_default());
                Ok(())
            }
            "delete" => {
                self.network_controller.unban(&ip_network);
                Ok(())
            }
            _ => Err(RPCError::invalid_params(format!(
                "Expected `params[1]` to be in the list [insert, delete], got {address}"
            ))),
        }
    }
