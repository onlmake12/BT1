Audit Report

## Title
Unbounded `runtime::spawn` via Unauthenticated `ConnectionSync` Against Attacker-Populated `pending_delivered` — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

## Summary

`ConnectionSyncProcess::execute()` unconditionally calls `runtime::spawn` for every `ConnectionSync` message whose `content.from` matches an entry in `pending_delivered`, without consuming or guarding that entry. An attacker can pre-populate `pending_delivered` with up to 9,000 entries by sending `ConnectionRequest` messages with unique `from_peer_id` values, then sustain 30 `runtime::spawn` calls per second indefinitely by cycling through those entries with `ConnectionSync` messages, leading to file descriptor exhaustion and async runtime saturation that crashes the victim node.

## Finding Description

**Phase 1 — Populating `pending_delivered`**

`respond_delivered()` in `connection_request.rs` stores attacker-supplied `listen_addrs` into `pending_delivered` keyed by `from_peer_id`:

```rust
// connection_request.rs L234-237
let now = unix_time_as_millis();
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
```

The only re-insertion guard is a per-`from_peer_id` cooldown of `HOLE_PUNCHING_INTERVAL = 2 minutes` (L161-166). With unique `from_peer_ids`, each gets its own independent entry. The `forward_rate_limiter` is keyed on `(content.from, content.to, msg_item_id)` at 1/sec per tuple (L132-143); with unique `from` values, each gets its own bucket. The binding constraint is the outer `rate_limiter` at 30/sec per `(session_id, msg_item_id)` (mod.rs L95-107). Over the 5-minute `TIMEOUT` window, a single attacker session accumulates up to `30 × 300 = 9,000` entries, each holding up to `ADDRS_COUNT_LIMIT = 24` TCP addresses.

The `from_peer_id` field is fully attacker-controlled — arbitrary bytes parsed from the message with no verification that the peer is actually connected. The only filtering applied to `listen_addrs` is that they must be TCP/IPv4 or TCP/IPv6 (L196-215), which the attacker trivially satisfies.

**Phase 2 — Triggering unbounded spawns via `ConnectionSync`**

When a `ConnectionSync` arrives with `content.to == self_peer_id` and `content.from` present in `pending_delivered`, `ConnectionSyncProcess::execute()` unconditionally spawns a task:

```rust
// connection_sync.rs L111-115
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)       // .get(), NOT .remove()
    .map(|info| info.0.clone());
```

```rust
// connection_sync.rs L145-162
runtime::spawn(async move {
    if let Ok(((stream, addr), _)) = select_ok(tasks).await {
        ...
    }
});
```

The `pending_delivered` entry is read with `.get()`, not consumed with `.remove()`. The same entry can trigger unlimited spawns. Contrast with `ConnectionRequestDeliveredProcess::execute()`, which gates NAT traversal on `inflight_requests.remove(&content.to)` (connection_request_delivered.rs L160), consuming the entry on first use and returning `StatusCode::Ignore` on subsequent attempts (L175). `ConnectionSyncProcess` has no such guard.

The `forward_rate_limiter` for `ConnectionSync` is keyed on `(content.from, content.to, msg_item_id)` at 1/sec (connection_sync.rs L85-96). With M unique `from_peer_ids` pre-populated in `pending_delivered`, M independent rate-limit buckets exist. The effective spawn rate is `min(M, 30)` per second, capped by the outer 30/sec limiter.

**Phase 3 — Resource cost per spawned task**

Each `try_nat_traversal` future loops for up to 30 seconds with ~200ms retry intervals (component/mod.rs L65-111), creating a new `TcpSocket` per iteration. `select_ok(tasks)` runs all 24 futures concurrently and only terminates when the first succeeds or all fail. Since the attacker controls the target addresses and does not respond, all 24 futures run for the full 30 seconds. Each spawned task holds 24 concurrent async futures × ~150 socket operations = ~3,600 socket operations over its lifetime.

## Impact Explanation

From a single attacker session at 30 `ConnectionSync` messages/sec, over the 30-second task lifetime: 30 × 30 = 900 concurrent spawned tasks, each with 24 concurrent async futures = **21,600 concurrent async futures** making TCP connection attempts to attacker-controlled addresses. This causes file descriptor exhaustion (each `TcpSocket::new_v4()`/`new_v6()` consumes an fd), async runtime saturation (21,600 futures polling on a shared executor), and outbound TCP SYN floods. The victim node becomes unresponsive to legitimate peers.

This matches **High (10001–15000 points)**: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation

The attack requires only a standard P2P connection — no special privileges, no PoW, no key material. The `from_peer_id` and `listen_addrs` fields in `ConnectionRequest` are fully attacker-controlled. The two-phase setup is straightforward: populate `pending_delivered` over ~300 seconds at 30 req/sec, then sustain `ConnectionSync` spam indefinitely. The rate limits slow but do not prevent the attack. Multiple attacker sessions scale the impact linearly. The victim's peer ID is publicly discoverable on the P2P network.

## Recommendation

1. **Add an `inflight_requests` guard in `ConnectionSyncProcess::execute()`** — mirror the check in `ConnectionRequestDeliveredProcess`: only proceed if `inflight_requests` contains `content.from`, and remove the entry on use to prevent repeated spawns.

2. **Consume `pending_delivered` entries on use** — change `.get(&content.from)` to `.remove(&content.from)` in `ConnectionSyncProcess::execute()` so each entry triggers at most one spawn.

3. **Bound `pending_delivered` map size** — cap the total number of entries (e.g., 256) to prevent unbounded memory growth regardless of rate limiting.

4. **Add a global cap on concurrent NAT traversal tasks** — use a semaphore or atomic counter to limit total in-flight `runtime::spawn` NAT traversal tasks across both `ConnectionSync` and `ConnectionRequestDelivered` paths.

## Proof of Concept

```
1. Attacker connects to victim as a normal P2P peer.

2. Attacker sends 30 ConnectionRequest messages/sec for 300 seconds, each with:
   - content.to = victim_peer_id
   - content.from = unique_peer_id_N  (N = 1..9000, arbitrary bytes)
   - content.listen_addrs = [24 valid TCP/IPv4 addresses under attacker control]
   → victim stores 9,000 entries in pending_delivered[peer_id_N] = ([24 addrs], now)
     (each unique from_peer_id bypasses the HOLE_PUNCHING_INTERVAL cooldown)

3. Attacker sends 30 ConnectionSync messages/sec indefinitely, each with:
   - content.to = victim_peer_id
   - content.from = peer_id_N  (cycling through pending_delivered entries)
   - content.route = []
   → victim calls runtime::spawn(select_ok(24 try_nat_traversal tasks)) for each
     (pending_delivered entry is NOT consumed; same entry triggers unlimited spawns)

4. After 30 seconds: 900 spawned tasks × 24 futures = 21,600 concurrent async futures
   making TCP connection attempts to attacker-controlled addresses that never respond.

5. Observable result: victim node's file descriptor table exhausts, async runtime
   stalls, node becomes unresponsive to legitimate peers.

Verification: add logging to runtime::spawn in connection_sync.rs and observe
spawn count growing unboundedly; monitor /proc/<pid>/fd count on Linux.
```
