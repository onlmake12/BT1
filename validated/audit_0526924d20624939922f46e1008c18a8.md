Audit Report

## Title
TCP RPC Server Bypasses `JSONRPC_BATCH_LIMIT`, Enabling Unbounded Batch Execution — (File: `rpc/src/server.rs`)

## Summary
`JSONRPC_BATCH_LIMIT` is enforced only inside the `handle_jsonrpc` axum HTTP handler. The TCP server path calls `serve_stream_sink` directly, never routing through `handle_jsonrpc`, so the batch limit is silently bypassed for all TCP connections. A single 2 MB newline-terminated JSON batch can carry thousands of `get_fee_rate_statistics` calls, each performing up to 202 synchronous RocksDB reads, saturating I/O and degrading block processing on the targeted node.

## Finding Description

`JSONRPC_BATCH_LIMIT` is a process-wide `OnceLock<usize>` initialized from `config.rpc_batch_limit`: [1](#0-0) [2](#0-1) 

The batch-size check is enforced exclusively inside `handle_jsonrpc` (the axum HTTP handler): [3](#0-2) 

`start_tcp_server` never calls `handle_jsonrpc`. It calls `serve_stream_sink` directly with the raw `MetaIoHandler`, which natively handles JSON-RPC batch arrays with no batch-size interception: [4](#0-3) 

The `LinesCodec` is configured with a 2 MB maximum frame length, allowing a single line to carry ~29,000 minimal calls: [5](#0-4) 

`pipeline_size=4` limits concurrent *messages* (lines), not calls within a single batch message — one 2 MB line is one message: [6](#0-5) 

Each `get_fee_rate_statistics` call iterates up to `MAX_TARGET = 101` blocks, performing `get_block_hash` + `get_block_ext` per block (202 synchronous RocksDB reads per call): [7](#0-6) [8](#0-7) 

The test setup confirms the bypass: `rpc_batch_limit: Some(1000)` is set, but the TCP listener at `tcp_listen_address: Some("127.0.0.1:0")` receives no such enforcement: [9](#0-8) 

## Impact Explanation

A single TCP connection sending one newline-terminated 2 MB JSON batch (e.g., 200 calls × 202 RocksDB reads = 40,400 synchronous reads; at 29,000 calls it reaches ~5.86 million reads) saturates RocksDB I/O and the Tokio thread pool. Repeated connections amplify the effect. The node falls behind the chain tip and can be crashed through resource exhaustion. This matches **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

TCP RPC is opt-in (`tcp_listen_address` is commented out by default), but TCP is the *only* transport supporting the Subscription RPC module, making it a required configuration for mining pools and infrastructure operators. Any unauthenticated attacker with TCP access to the port can trigger this. No keys, PoW, or victim interaction are required. The attack is trivially repeatable from a single connection.

## Recommendation

Move the batch-limit check out of `handle_jsonrpc` and into a shared enforcement layer. Concretely, wrap the TCP `serve_stream_sink` call with a pre-processing step that deserializes the incoming line, enforces `JSONRPC_BATCH_LIMIT`, and rejects oversized batches before dispatching to `MetaIoHandler`. Alternatively, implement the check as middleware on the `MetaIoHandler` dispatch path so HTTP, WS, and TCP all share the same enforcement point.

## Proof of Concept

Prerequisites: node started with `tcp_listen_address = "0.0.0.0:18114"` in `ckb.toml`. Even with `rpc_batch_limit = 200` set, the TCP path ignores it and executes all calls.

```python
import socket, json

calls = [
    {"jsonrpc": "2.0", "method": "get_fee_rate_statistics", "params": [], "id": i}
    for i in range(200)
]
payload = json.dumps(calls) + "\n"

s = socket.create_connection(("127.0.0.1", 18114))
s.sendall(payload.encode())
# 200 calls × 202 RocksDB reads = 40,400 synchronous reads from one TCP frame.
# Loop to exhaust I/O budget and degrade block processing latency.
```
