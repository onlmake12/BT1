### Title
Missing Governance Action to Replace Fee Token Addresses Permanently Locks Starknet Price Feed Updates - (File: target_chains/starknet/contracts/src/pyth.cairo)

### Summary

The Starknet Pyth contract stores exactly two fee token addresses (`fee_token_address1`, `fee_token_address2`) set immutably at construction. The governance system provides `SetFeeInToken` to update the fee *amount* for an already-registered token, but no governance action exists to replace or remove a fee token *address*. If either token becomes unusable (e.g., a Starknet token migration, ERC20 contract pause, or protocol deprecation), price feed updates on Starknet become permanently blocked until a full contract upgrade is executed.

### Finding Description

The Starknet Pyth contract stores two fee token slots in its storage:

```cairo
fee_token_address1: ContractAddress,
fee_token_address2: ContractAddress,
single_update_fee1: u256,
single_update_fee2: u256,
``` [1](#0-0) 

These are written once in the constructor and never modified thereafter:

```cairo
self.fee_token_address1.write(fee_token_address1);
self.single_update_fee1.write(single_update_fee1);
self.fee_token_address2.write(fee_token_address2);
self.single_update_fee2.write(single_update_fee2);
``` [2](#0-1) 

The `SetFeeInToken` governance action only updates the fee *amount* for a token that already matches one of the two stored addresses. If the supplied token address does not match either slot, it panics:

```cairo
fn set_fee(ref self: ContractState, value: u64, expo: u64, token: ContractAddress) {
    ...
    } else {
        panic_with_felt252(GovernanceActionError::InvalidGovernanceMessage.into())
    };
``` [3](#0-2) 

The full set of governance actions defined in `governance.cairo` contains no action to replace a fee token address:

```cairo
pub enum GovernanceAction {
    UpgradeContract,
    AuthorizeGovernanceDataSourceTransfer,
    SetDataSources,
    SetFee,
    SetValidPeriod,
    RequestGovernanceDataSourceTransfer,
    SetWormholeAddress,
    SetFeeInToken,
}
``` [4](#0-3) 

During `update_price_feeds`, the contract attempts to collect fees from `fee_token_address1` first, then falls back to `fee_token_address2`. If both fail, the call reverts:

```cairo
if !fee1_transfered {
    let fee2_transfered = transfer_fee(..., self.fee_token_address2.read(), ...);
    if !fee2_transfered {
        panic_with_felt252(UpdatePriceFeedsError::InsufficientFeeAllowance.into());
    }
}
``` [5](#0-4) 

On mainnet, the two tokens are STRK (`0x04718f5a0fc34cc1af16a1cdee98ffb20c31f5cd61d6ab07201858f4287c938d`) and ETH (`0x049d36570d4e46f48e99674bd3fcc84644ddd6b96f7c741b1562b82f9e004dc7`): [6](#0-5) 

### Impact Explanation

If either fee token contract is paused, deprecated, or migrated (a realistic scenario on Starknet, which has already undergone token introductions), **all price feed updates on Starknet are blocked**. Every call to `update_price_feeds`, `parse_price_feed_updates`, or `parse_unique_price_feed_updates` will revert because fee collection from both slots fails. Protocols relying on Pyth prices on Starknet would receive stale data, potentially enabling liquidation manipulation, oracle bypass, or other downstream financial exploits. The only recovery path is a full contract upgrade, which requires governance coordination and introduces upgrade risk.

### Likelihood Explanation

Starknet is an actively evolving L2. Token contract upgrades, fee mechanism changes, and token migrations are plausible operational events. STRK itself was introduced relatively recently. The absence of a governance escape hatch means any such event immediately and completely disables Pyth price feed updates on Starknet with no fast remediation path.

### Recommendation

Add a `SetFeeToken` governance action (analogous to `SetFeeInToken`) that allows governance to replace `fee_token_address1` or `fee_token_address2` with a new address. This mirrors the fix recommended in the original report: provide an admin mechanism to remove/replace entries in the supported token set, and ensure the new token address is validated before being written.

### Proof of Concept

1. Governance deploys Pyth on Starknet with `fee_token_address1 = STRK`, `fee_token_address2 = ETH`.
2. Starknet undergoes a token migration; the STRK ERC20 contract is paused or replaced.
3. Any user calls `update_price_feeds(data)`.
4. The contract attempts `transfer_fee(..., fee_token_address1=STRK, ...)` — this fails because STRK is paused.
5. The contract falls back to `transfer_fee(..., fee_token_address2=ETH, ...)` — this also fails if the user only holds STRK or if ETH is similarly affected.
6. The call reverts with `InsufficientFeeAllowance`.
7. Governance has no `SetFeeToken` action to point `fee_token_address1` at the new STRK contract.
8. All Starknet price feed updates are permanently blocked until a contract upgrade is executed.
