### Title
Unauthenticated Block Template Injection via Miner Notify HTTP Server Allows Mining Reward Theft — (`miner/src/client.rs`)

### Summary

When the CKB miner is configured in "notify mode," it starts an HTTP server to receive block template push notifications from the CKB node. The `handle` function that processes incoming HTTP requests performs no authentication or source verification whatsoever. Any attacker who can reach the miner's listen port can inject a crafted `BlockTemplate` JSON payload — including one with an attacker-controlled coinbase lock script — causing the miner to mine on the attacker's template and redirect block rewards to the attacker's address.

### Finding Description

The miner's notify mode is activated when `listen` is set in `ckb-miner.toml`. The `spawn_background` function in `miner/src/client.rs` starts a TCP listener and serves incoming HTTP connections via `listen_block_template_notify`. [1](#0-0) 

Every incoming HTTP request is dispatched to the `handle` function: [2](#0-1) 

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }

    Ok(Response::new(Empty::new()))
}
```

There is **no check on the source IP, no authentication header validation, no shared secret, and no HMAC**. The body is deserialized directly into a `BlockTemplate` and passed to `update_block_template`, which immediately dispatches it to the mining workers: [3](#0-2) 

The `work_id` deduplication check inside `update_block_template` is not a security control — an attacker simply supplies a `work_id` value different from the current one to bypass it.

Note that `parse_authorization` exists only for the miner's **outgoing** RPC calls to the CKB node: [4](#0-3) 

There is no symmetric mechanism protecting the **incoming** notify server.

The `submit_block` RPC on the node side verifies the PoW and parent hash but does **not** verify that the coinbase lock script matches the node's configured `block_assembler`: [5](#0-4) 

Therefore, a block mined on an attacker-injected template with a different coinbase lock script will be accepted by the network.

### Impact Explanation

An attacker who can reach the miner's notify HTTP port can:

1. Craft a `BlockTemplate` JSON with an attacker-controlled `lock` script in the coinbase output.
2. POST it to the miner's listen address (e.g., `http://miner-ip:8888/`).
3. The miner replaces its current work with the attacker's template and begins mining on it.
4. If the miner finds a valid PoW solution, it calls `submit_block` on the CKB node with the attacker's block.
5. The block is accepted by the network; the coinbase reward (newly issued CKB) is credited to the attacker's address, not the legitimate miner's address.

This is **direct theft of mining rewards** — a complete financial loss for the miner operator for every block found while the attacker's template is active. The attacker can also inject templates that censor specific transactions or include attacker-chosen transactions, affecting transaction ordering and fee revenue.

### Likelihood Explanation

The `listen` option is an explicitly supported, documented configuration feature intended for mining pool deployments where the CKB node and miner run on separate machines: [6](#0-5) 

In any multi-machine mining setup, the listen address must be bound to a network interface reachable by the CKB node — making it reachable by any attacker on the same network segment. The attack requires only the ability to send an HTTP POST request to the miner's port. No credentials, keys, or privileged access are needed. The attacker does not need to break any cryptographic primitive.

### Recommendation

Add authentication to the miner's notify HTTP server. The simplest correct fix is a shared secret (token) configured in `ckb-miner.toml` that the CKB node includes as an `Authorization` header in its notify requests, and that the `handle` function validates before processing the body. The CKB node's block assembler `notify` configuration already supports arbitrary URLs, so the secret can be embedded in the URL or passed as a header. Reject any request that does not present the correct token with HTTP 401.

### Proof of Concept

**Preconditions:** Miner is running in notify mode with `listen = "0.0.0.0:8888"` (or any network-reachable address). Attacker has network access to port 8888.

**Steps:**

1. Attacker obtains a valid `BlockTemplate` by calling `get_block_template` on any public CKB node (this RPC is unauthenticated and publicly documented).
2. Attacker replaces the `lock` field in the `cellbase.data.outputs[0]` with their own lock script (attacker's public key hash).
3. Attacker sets `work_id` to a value different from the miner's current `work_id` (e.g., `"0x0"` on first injection, then `"0x1"`, etc.).
4. Attacker sends:
   ```
   POST http://miner-ip:8888/ HTTP/1.1
   Content-Type: application/json

   { <crafted BlockTemplate JSON with attacker coinbase> }
   ```
5. The miner's `handle` function deserializes the payload and calls `update_block_template` with no validation.
6. Mining workers receive the new `Work` derived from the attacker's template and begin hashing.
7. Upon finding a valid nonce, the miner calls `submit_block` on the CKB node. The node verifies the PoW and accepts the block.
8. The coinbase reward is paid to the attacker's address.

**Expected outcome:** Every block found by the miner while the attacker's template is active transfers the block reward to the attacker. The legitimate miner operator receives nothing.

    fn update_block_template(&self, block_template: BlockTemplate) {
        let work_id = block_template.work_id.into();
        let updated = |id| {
            if id != work_id || id == 0 {
                Some(work_id)
            } else {
                None
            }

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}
