Audit Report

## Title
Missing Count Sanity Check in `TransactionsProcess::execute()` Allows Unbounded Deserialization Before Filtering — (`sync/src/relayer/transactions_process.rs`)

## Summary

`TransactionsProcess::execute()` performs full molecule deserialization (`to_entity().into_view()`) on every transaction in a `RelayTransactions` message before the filter predicate runs, with no upper-bound check on transaction count. Every sibling relay handler (`TransactionHashesProcess`, `GetTransactionsProcess`, `GetBlockTransactionsProcess`) enforces `MAX_RELAY_TXS_NUM_PER_BATCH` before doing any work. Any connected peer can exploit this to force O(n) full deserialization — including hash computation — per message, at the rate-limiter ceiling of 30 messages/second, with no short-circuit.

## Finding Description

In `sync/src/relayer/mod.rs`, the dispatcher routes `RelayTransactions` to `TransactionsProcess` after only a molecule structural check (`check_data()`): [1](#0-0) 

Inside `TransactionsProcess::execute()`, the iterator chain applies `.map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))` — full deserialization including hash computation — before the `.filter()` predicate. In Rust's lazy iterator model, for each element `map` is evaluated before `filter`, so every transaction is fully deserialized regardless of whether it passes the filter: [2](#0-1) 

No count guard exists anywhere in this path. `MAX_RELAY_TXS_NUM_PER_BATCH` is not imported in `transactions_process.rs` at all. Contrast with `TransactionHashesProcess`: [3](#0-2) 

`GetTransactionsProcess` also guards: [4](#0-3) 

`GetBlockTransactionsProcess` also guards: [5](#0-4) 

The constant that should bound this handler is defined but unused in it: [6](#0-5) 

The rate limiter at 30 req/s per peer provides partial mitigation but does not eliminate the attack surface: [7](#0-6) 

## Impact Explanation

A connected peer sends molecule-valid `RelayTransactions` messages containing the maximum number of minimal-size transactions that fit within the P2P frame. At 30 messages/second (rate-limiter ceiling), the target node performs O(n) full deserialization — including SHA3 hash computation via `into_view()` — for every transaction in every message, with no count guard to short-circuit. All transactions will be discarded by the filter (which requires the hash to be in `unknown_tx_hashes` with the correct requesting peer), but the CPU cost is already paid. With multiple coordinated peers this scales linearly. The sustained CPU pressure degrades the node's ability to process legitimate blocks and transactions. This matches: **High — Vulnerabilities which could easily crash a CKB node** (or at minimum **Low — important performance improvement**).

## Likelihood Explanation

Any peer that completes the standard P2P handshake can send `RelayTransactions` messages. No privileged key, majority hashpower, or social engineering is required. The attack is trivially scriptable: construct a molecule-valid `RelayTransactions` message with the maximum number of minimal-size `RelayTransaction` entries and send at the rate-limiter ceiling. Multiple peers multiply the effect linearly.

## Recommendation

Add a count check at the top of `TransactionsProcess::execute()`, consistent with all sibling handlers:

```rust
pub fn execute(self) -> Status {
    let tx_count = self.message.transactions().len();
    if tx_count > MAX_RELAY_TXS_NUM_PER_BATCH {
        return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
            "Transactions count({tx_count}) > MAX_RELAY_TXS_NUM_PER_BATCH({MAX_RELAY_TXS_NUM_PER_BATCH})"
        ));
    }
    // ... existing logic
}
```

Additionally, consider reordering the iterator chain to perform the cheap hash-based filter before the expensive `to_entity().into_view()` deserialization where the molecule API permits it.

## Proof of Concept

1. Complete the CKB P2P handshake with a target node to become a connected peer.
2. Construct a molecule-valid `RelayTransactions` message containing the maximum number of minimal-size `RelayTransaction` entries that fit within the P2P frame limit (each entry ~88 bytes).
3. Send this message at 30 messages/second (the rate-limiter ceiling for this message type per peer).
4. The target node calls `check_data()` (structural validation) and then `TransactionsProcess::execute()` (O(n) full deserialization via `.to_entity().into_view()`) for every message, with no count guard to short-circuit. All transactions are discarded by the filter, but the deserialization cost is already paid. Observe elevated CPU usage proportional to the number of transactions per message. Scale with additional peers to amplify the effect.
