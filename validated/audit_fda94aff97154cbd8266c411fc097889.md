### Title
`Scheduler.getPricesNoOlderThan` Staleness Check Uses Batch-Level Max Timestamp Instead of Per-Feed `publishTime`, Allowing Stale Prices to Pass as Fresh - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getPricesNoOlderThan` and `Scheduler.getEmaPricesNoOlderThan` validate freshness by checking the subscription-level `priceLastUpdatedAt` field (which equals the **maximum** `publishTime` across all feeds in the last update batch) against the caller-supplied `age_seconds`. However, the prices actually returned are fetched per-feed from storage and can individually carry `publishTime` values that are arbitrarily older than `age_seconds` — for example, feeds for closed markets whose last trading period was days ago. A consumer protocol that calls `getPricesNoOlderThan` expecting all returned prices to be within `age_seconds` will silently receive stale prices for closed-market assets.

---

### Finding Description

`updatePriceFeeds` computes `latestPublishTime` as the **maximum** `publishTime` across all feeds in the batch and stores it in `status.priceLastUpdatedAt`:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
...
status.priceLastUpdatedAt = latestPublishTime;
``` [1](#0-0) [2](#0-1) 

This is intentional: the Scheduler explicitly allows closed-market feeds to carry old `publishTime` values from their last trading period, using the max timestamp to represent "when the last update cycle ran."

`getPricesNoOlderThan` then checks only this batch-level max timestamp:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [3](#0-2) 

`getPricesUnsafe` returns each feed's stored price directly from `_state.priceUpdates[subscriptionId][priceId]` with no per-feed timestamp check: [4](#0-3) 

The same pattern applies to `getEmaPricesNoOlderThan`: [5](#0-4) 

The `IScheduler` interface documents `getPricesNoOlderThan` as: *"Returns the price that is no older than `age` seconds of the current time. Reverts if the price wasn't updated sufficiently recently."* [6](#0-5) 

This contract-level guarantee is violated for any subscription that mixes active-market feeds (recent `publishTime`) with closed-market feeds (stale `publishTime`). The staleness gate passes because the active-market feed's timestamp satisfies `age_seconds`, but the closed-market feed's price — potentially days old — is returned without any per-feed check.

---

### Impact Explanation

A consumer protocol that calls `getPricesNoOlderThan(subscriptionId, priceIds, 300)` (5-minute freshness) on a subscription containing both BTC/USD (active, `publishTime = now - 2min`) and AAPL/USD (closed market, `publishTime = 3 days ago`) will:

1. Pass the staleness gate because `distance(block.timestamp, priceLastUpdatedAt) = 2min < 300s`
2. Receive AAPL/USD with `publishTime = 3 days ago` — a price that is 3 days stale

Any DeFi protocol (lending, derivatives, structured products) that relies on `getPricesNoOlderThan` to guarantee price freshness before executing financial operations (collateral valuation, liquidation, settlement) will use a stale price while believing it is fresh. This is the direct analog to the WrappedIbbtcEth bug: a cached price variable is used in financial operations without an effective per-asset staleness check, enabling risk-free profit by exploiting the gap between the stale on-chain price and the actual market price.

---

### Likelihood Explanation

The Scheduler is explicitly designed to support subscriptions with mixed active and closed-market feeds. The comment in `_validateShouldUpdatePrices` confirms this is an intended use case:

> *"Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period."* [7](#0-6) 

Any subscription containing at least one closed-market feed alongside active feeds will trigger this condition on every update cycle. The keeper runs regularly, keeping `priceLastUpdatedAt` fresh, while closed-market feeds remain stale. Consumer protocols following the SDK's recommended usage pattern (`getPricesNoOlderThan` with a tight age) are silently exposed.

---

### Recommendation

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` should validate freshness per individual feed rather than against the batch-level `priceLastUpdatedAt`. Replace the single batch check with a per-feed check:

```solidity
function getPricesNoOlderThan(...) external view ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

Alternatively, if the batch-level check is intentional for closed-market support, the function's NatSpec and interface documentation must be updated to clearly state that the age guarantee applies to the update cycle timestamp, not to individual feed `publishTime` values, so consumer protocols are not misled.

---

### Proof of Concept

1. Create a Scheduler subscription with two feeds: `BTC/USD` (active market) and `AAPL/USD` (equity, closed market).
2. Keeper calls `updatePriceFeeds`. Pyth returns `BTC/USD` with `publishTime = T` and `AAPL/USD` with `publishTime = T - 3 days` (last trading period). Both share the same Pythnet slot, so `PriceSlotMismatch` is not triggered.
3. `_validateShouldUpdatePrices` sets `updateTimestamp = T` (max). `status.priceLastUpdatedAt = T`.
4. Consumer calls `getPricesNoOlderThan(subscriptionId, [BTC_ID, AAPL_ID], 300)` at time `T + 60s`.
5. Check: `distance(T+60, T) = 60 < 300` → passes. No revert.
6. `getPricesUnsafe` returns `AAPL/USD` with `publishTime = T - 3 days`.
7. Consumer protocol uses the 3-day-old AAPL price believing it is within 5 minutes of freshness. [8](#0-7) [9](#0-8)
