### Title
No Request Expiry in Echo Contract Allows Permanent Fund Locking and Stale Price Delivery - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

---

### Summary

The `Echo.sol` contract implements a request-callback flow for price updates (`requestPriceUpdatesWithCallback` / `executeCallback`) with no expiry deadline on pending requests and no refund mechanism. Once a user submits a fee and a request, the funds are locked indefinitely until `executeCallback` is called. After the exclusivity period (15 seconds by default), any unprivileged actor can call `executeCallback` with price data that was valid at the stored `publishTime` but is arbitrarily stale at execution time, delivering outdated prices to the consumer's callback.

---

### Finding Description

In `Echo.sol::requestPriceUpdatesWithCallback`, a user submits a fee and a request containing `publishTime`, `priceIds`, `callbackGasLimit`, `requester`, and `provider`. The fee is split between the Pyth protocol and the provider and stored in the contract. [1](#0-0) 

The only way to clear a request and deliver price data is through `executeCallback`. There is no `cancelRequest` function, no expiry timestamp on the stored request, and no refund path for the user. [2](#0-1) 

The exclusivity period restricts *who* can call `executeCallback` (only the assigned provider during the first N seconds), but imposes no upper bound on *when* it must be called: [3](#0-2) 

After the exclusivity period expires, any address can call `executeCallback` with any `providerToCredit`. The only constraint on the supplied price data is that it satisfies `parsePriceFeedUpdates` with `req.publishTime` as the minimum publish time — there is no maximum publish time enforced by Echo itself. Price data from exactly `publishTime` (which could be up to 60 seconds before the request was made, plus however long the request has been pending) is valid. [4](#0-3) 

The `IEcho` interface documents no expiry or cancellation mechanism: [5](#0-4) 

---

### Impact Explanation

**Impact 1 — Permanent fund locking**: If the assigned provider goes offline or is unresponsive, the user's fee is locked in the contract with no on-chain recourse. There is no `cancelRequest`, no timeout-triggered refund, and no governance path to recover individual user funds. This is a direct financial loss for users.

**Impact 2 — Stale price delivery by unprivileged actor**: After the 15-second exclusivity period, any unprivileged actor can call `executeCallback` with Pyth price data fetched from Hermes at exactly `req.publishTime`. If the request has been pending for minutes or hours, the delivered prices are stale relative to the current market. Consumer contracts that do not independently validate `publishTime` against `block.timestamp` will act on outdated data, potentially enabling incorrect liquidations, mispriced trades, or other financial harm.

---

### Likelihood Explanation

- Provider liveness failures are realistic in production (network issues, key rotation, service outages).
- The 15-second exclusivity window is very short; after it expires, the attack surface is open to any EOA.
- Hermes provides historical price data publicly, so an attacker can trivially obtain valid price data for any past `publishTime`.
- No on-chain mechanism prevents indefinite deferral of `executeCallback`.

---

### Recommendation

1. **Add a request expiry timestamp**: Store a `deadline = block.timestamp + MAX_FULFILLMENT_WINDOW` in each request. In `executeCallback`, revert if `block.timestamp > req.deadline`.
2. **Add a `cancelRequest` / refund function**: Allow the requester to reclaim their fee after the deadline passes without fulfillment.
3. **Enforce a maximum publish time in `executeCallback`**: Pass `maxPublishTime = req.publishTime + ALLOWED_STALENESS` to `parsePriceFeedUpdates` so that price data cannot be arbitrarily old relative to the request.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` at time `T` with `publishTime = T`, paying fee `F`. Request is stored with sequence number `S`.
2. The assigned provider never calls `executeCallback` (offline or malicious). User's fee `F` is locked indefinitely — no refund path exists.
3. Alternatively, at time `T + 1 hour`, an attacker fetches Pyth price data from Hermes for timestamp `T` (valid, verifiable VAA). The exclusivity period expired at `T + 15s`.
4. Attacker calls `executeCallback(anyAddress, S, oldUpdateData, priceIds)`. `parsePriceFeedUpdates` accepts the data because `publishTime == req.publishTime`. The consumer's `echoCallback` is invoked with 1-hour-old prices.
5. If the consumer contract uses these prices for financial logic without checking `publishTime`, it acts on stale data. [1](#0-0) [6](#0-5)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
