### Title
Stale `CALLBACK_FAILED` Entropy Requests Persist Without Cancellation, Enabling Known-Randomness Delivery Timing Attack — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When an Entropy callback fails (e.g., due to gas exhaustion or consumer logic revert), the request is not cleared from storage. Instead, `callbackStatus` is set to `CALLBACK_FAILED` and the request persists indefinitely. The random number is already publicly revealed via the `CallbackFailed` event. Because `revealWithCallback` is permissionless and accepts `CALLBACK_FAILED` requests, any unprivileged caller can re-trigger the callback at an arbitrary future time — delivering a known, pre-revealed random number to the consumer contract at a moment of the attacker's choosing.

---

### Finding Description

In `Entropy.sol`, `revealWithCallback` handles the `CALLBACK_FAILED` state as follows:

1. `revealHelper` is called, which validates the commitment and re-derives `randomNumber`. It also updates `providerInfo.currentCommitment` as a side effect.
2. The code enters the `else` branch (since `callbackStatus == CALLBACK_FAILED` does not satisfy `callbackStatus == CALLBACK_NOT_STARTED`):

```solidity
} else {
    address callAddress = req.requester;
    EntropyStructs.Request memory reqV1 = EntropyStructConverter.toV1Request(req);
    clearRequest(provider, sequenceNumber);
    // ...
    if (len != 0) {
        IEntropyConsumer(callAddress)._entropyCallback(
            sequenceNumber, provider, randomNumber
        );
    }
}
```

The request is cleared and the callback is invoked with **no gas cap and no error catching**. Critically:

- There is **no `cancelRequest` function** in `Entropy.sol`. A consumer contract has no on-chain mechanism to cancel a `CALLBACK_FAILED` request.
- The `randomNumber` was already emitted in the `CallbackFailed` event during the first failed attempt, making it **publicly known**.
- `revealWithCallback` is **callable by anyone** — not just the original requester or the Fortuna keeper.

This means a `CALLBACK_FAILED` request is a permanently live, publicly-known random number that any external actor can deliver to the consumer at any time.

The `isActive` check only tests `sequenceNumber != 0`, so `CALLBACK_FAILED` requests remain active and retrievable via `findActiveRequest` indefinitely.

---

### Impact Explanation

A consumer contract (e.g., a lottery, NFT mint, game) that uses Entropy for randomness assumes the random number is unknown until the callback fires. With `CALLBACK_FAILED`:

1. The random number is revealed and publicly known (emitted in `CallbackFailed`).
2. The consumer's state may advance — a new lottery round starts, a new game begins, a new NFT batch opens.
3. An attacker observes the known `randomNumber` from the event log.
4. The attacker waits until the consumer's current state makes that known `randomNumber` favorable (e.g., it would select the attacker as winner, or mint a rare NFT to the attacker's address).
5. The attacker calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)` with the known parameters.
6. The consumer's `_entropyCallback` fires with the pre-known `randomNumber` at the attacker-chosen moment.

If the consumer contract does not validate that the incoming `sequenceNumber` matches its current active request (a common omission in simple consumer implementations), the stale callback is processed as if it were the current round's result. This is the direct analog to the GMX "Phantom Decrease" — a stale pending action (the `CALLBACK_FAILED` request) unexpectedly executes against a new state (the consumer's new round/game).

**Impact:** Manipulation of randomness-dependent outcomes in consumer contracts. Loss of funds or unfair advantage in any protocol built on Pyth Entropy that does not explicitly guard against stale sequence number delivery.

---

### Likelihood Explanation

- **Entry path is fully permissionless**: `revealWithCallback` has no access control. Any EOA can call it.
- **Parameters are fully public**: `userContribution` and `providerContribution` are emitted in `RequestedWithCallback` and `CallbackFailed` events respectively.
- **No expiry or cancellation**: `CALLBACK_FAILED` requests have no TTL. They persist until successfully recovered.
- **Consumer contracts commonly omit sequence validation**: The Pyth documentation and SDK examples do not mandate that `_entropyCallback` validate the sequence number against a stored "current request" variable.
- **Fortuna keeper behavior**: The Fortuna keeper service retries `CALLBACK_FAILED` requests, but an attacker can front-run the keeper's retry transaction once the consumer's state is favorable.

Likelihood is **medium-high** for consumer contracts that handle high-value randomness outcomes and do not implement strict sequence number validation.

---

### Recommendation

1. **Add a `cancelRequest` function** to `Entropy.sol` callable by the original `req.requester`, which clears a `CALLBACK_FAILED` request and optionally refunds the fee. This mirrors the GMX fix of cancelling stale orders.
2. **Restrict `revealWithCallback` recovery** for `CALLBACK_FAILED` requests to only the original `req.requester` (or a designated recovery address), preventing third-party timing attacks.
3. **Document the security implication** clearly: once a request enters `CALLBACK_FAILED`, the random number is public. Consumer contracts must treat any `CALLBACK_FAILED` sequence number as compromised and issue a new request rather than relying on recovery.

---

### Proof of Concept

**Step 1:** Provider sets `defaultGasLimit`. Consumer contract `C` requests randomness → sequence number `N` assigned.

**Step 2:** Fortuna calls `revealWithCallback(provider, N, userContribution, providerContribution)`. Consumer's `_entropyCallback` reverts (gas or logic error). Contract emits:
```
CallbackFailed(provider, C, N, userContribution, providerContribution, randomNumber, revertReason)
```
`randomNumber` is now publicly known. Request remains active with `callbackStatus = CALLBACK_FAILED`.

**Step 3:** Consumer contract `C` starts a new lottery round. It calls `requestV2` → sequence number `M` assigned. Round 2 is now active.

**Step 4:** Attacker computes: "if `randomNumber` (from step 2) is delivered to `C` now, does it select me as winner of Round 2?" If yes:

**Step 5:** Attacker calls:
```solidity
entropy.revealWithCallback(provider, N, userContribution, providerContribution);
```
`findActiveRequest` succeeds (request is still active). `revealHelper` re-derives the same `randomNumber`. The `else` branch fires: `clearRequest` then `C._entropyCallback(N, provider, randomNumber)`.

    function allocRequest(
        address provider,
        uint64 sequenceNumber
    ) internal returns (EntropyStructsV2.Request storage req) {
        (, uint8 shortKey) = requestKey(provider, sequenceNumber);

        req = _state.requests[shortKey];
        if (isActive(req)) {
            // There's already a prior active request in the storage slot we want to use.
            // Overflow the prior request to the requestsOverflow mapping.
            // It is important that this code overflows the *prior* request to the mapping, and not the new request.
            // There is a chance that some requests never get revealed and remain active forever. We do not want such
            // requests to fill up all of the space in the array and cause all new requests to incur the higher gas cost
            // of the mapping.
            //
            // This operation is expensive, but should be rare. If overflow happens frequently, increase
            // the size of the requests array to support more concurrent active requests.
            (bytes32 reqKey, ) = requestKey(req.provider, req.sequenceNumber);
            _state.requestsOverflow[reqKey] = req;
        }
    }

    function isActive(

        // a randomness request with sequence number 0.
        return req.sequenceNumber != 0;
