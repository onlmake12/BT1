Audit Report

## Title
Off-by-one in spawn guard allows 17 concurrent VMs instead of the defined limit of 16 — (`script/src/scheduler.rs`)

## Summary
`MAX_VMS_COUNT` is defined as `16`, but the spawn guard at line 532 uses strict `>` instead of `>=`. When exactly 16 VMs are already alive (`suspended + instantiated == 16`), the condition `16 > 16` evaluates to `false`, the guard body is skipped, and `boot_vm` is called, creating a 17th VM in violation of the defined invariant.

## Finding Description
`MAX_VMS_COUNT` is declared at line 34 as `pub const MAX_VMS_COUNT: u64 = 16`. The spawn guard at line 532 reads:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
```

When the root VM plus 15 children are alive (`suspended + instantiated == 16`), `16 > 16` is `false`, the guard body is skipped, and execution falls through to `boot_vm` at line 539, creating a 17th VM. The guard only fires at 17 existing VMs (`17 > 16 == true`), meaning the effective limit enforced is 17, not 16. This is the only guard before `boot_vm`; no secondary limit exists.

## Impact Explanation
This is an incorrect implementation of CKB-VM's spawn subsystem, matching the allowed High impact class: "Incorrect implementation or behavior of CKB-VM or system scripts." `MAX_VMS_COUNT` is a defined resource invariant of the VM scheduler. The off-by-one silently violates it for every script execution that reaches the limit, allowing one extra VM (with its full machine state and `Snapshot2` allocation) per `Scheduler` instance.

## Likelihood Explanation
The path is fully unprivileged. Any user can submit a transaction via RPC or P2P relay containing a V2 ELF script that calls `spawn` 16 times. No proof-of-work, key material, or operator access is required. The condition is deterministic and locally reproducible on any node running the affected code.

## Recommendation
Change the guard from strict greater-than to greater-than-or-equal:

```rust
// Before (off-by-one):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

## Proof of Concept
1. Write a V2 RISC-V ELF that calls `spawn` in a loop 16 times; each child immediately calls `wait` on a non-existent pid so it remains alive in the scheduler.
2. The root VM calls `spawn` a 17th time.
3. With the current code (`>`), the 17th `spawn` returns `SUCCESS` and `Scheduler` holds 17 live VM entries.
4. Assert `scheduler.suspended.len() + scheduler.instantiated.len() == 17` to confirm the invariant is broken.
5. With the fix (`>=`), the 17th `spawn` returns `MAX_VMS_SPAWNED` and the count stays at 16.
