### Title
Missing Duplicate `priceIds` Validation in `Echo.requestPriceUpdatesWithCallback` Causes Permanent Fund Lock - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` accepts a `priceIds` array without checking for duplicate entries. If a user submits duplicate price IDs, the stored request can never be fulfilled: `executeCallback` internally calls `pyth.parsePriceFeedUpdates` with the duplicate `priceIds`, which only fills the **first** occurrence of each price ID (leaving the second slot at `id == 0`) and then reverts with `PriceFeedNotFoundWithinRange`. Because `clearRequest` is called **after** `parsePriceFeedUpdates`, the revert rolls back the entire transaction, the request is never cleared, and the user's funds are permanently locked.

---

### Finding Description

**Root cause — `parsePriceFeedUpdates` is first-match-only for duplicate `priceIds`**

In `Pyth.sol`, `parsePriceFeedUpdatesWithConfig` allocates `context.priceFeeds` with length `priceIds.length` (all `id == 0`). For every price-feed message parsed from `updateData`, it calls `findIndexOfPriceId`, which returns the **first** index where `priceIds[k] == targetPriceId` and breaks: [1](#0-0) 

If `priceIds = [A, A]` and the update data contains one message for feed A, `findIndexOfPriceId` returns `0`, fills `context.priceFeeds[0]`, and `context.priceFeeds[1]` remains `id == 0`. The final validation loop then reverts: [2](#0-1) 

**Missing validation in Echo**

`requestPriceUpdatesWithCallback` stores only the first 8-byte prefix of each price ID and performs no uniqueness check: [3](#0-2) 

**`clearRequest` is called after `parsePriceFeedUpdates`**

In `executeCallback`, the Pyth call precedes `clearRequest`. A revert inside `parsePriceFeedUpdates` rolls back the entire transaction: [4](#0-3) 

Because the provider must supply `priceIds` whose first-8-byte prefixes match the stored `priceIdPrefixes` (verified at lines 124–141), and those stored prefixes contain duplicates, the provider is forced to pass duplicate `priceIds`, guaranteeing the revert on every attempt.

**Contrast with Scheduler**

The `Scheduler` contract explicitly rejects duplicate price IDs at subscription creation/update time: [5](#0-4) 

Echo has no equivalent guard.

---

### Impact Explanation

A user who submits duplicate `priceIds` to `requestPriceUpdatesWithCallback` pays the full fee (`priceIds.length × feePerFeedInWei + baseFee + gasFee`) but receives a request that can never be executed. The fee stored in `req.fee` is never credited to the provider and never refunded to the user — it is permanently locked in the Echo contract. There is no cancellation or refund mechanism for active requests.

---

### Likelihood Explanation

Low. A user must pass duplicate entries in the `priceIds` array. This can occur through a buggy integration, a copy-paste error in calldata construction, or a malicious contract that deliberately passes duplicates on behalf of a victim. No privileged access is required; any unprivileged caller of `requestPriceUpdatesWithCallback` can trigger this condition.

---

### Recommendation

Add a uniqueness check on `priceIds` inside `requestPriceUpdatesWithCallback`, mirroring the pattern already used in `Scheduler._validateSubscriptionParams`:

```solidity
for (uint i = 0; i < priceIds.length; i++) {
    for (uint j = i + 1; j < priceIds.length; j++) {
        require(priceIds[i] != priceIds[j], "Duplicate priceId");
    }
}
```

Alternatively, enforce uniqueness via a `mapping(bytes32 => bool)` scratch check for O(n) cost.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback(provider, t, [A, A], gasLimit)` with sufficient `msg.value`.
2. Contract stores `req.priceIdPrefixes = [prefix(A), prefix(A)]` and `req.fee = msg.value - pythFeeInWei`.
3. Provider calls `executeCallback(provider, seq, updateData, [A, A])`.
4. Prefix check passes (both entries match `prefix(A)`).
5. `pyth.parsePriceFeedUpdates(updateData, [A, A], t, t)` is called.
6. Inside `parsePriceFeedUpdatesWithConfig`: `findIndexOfPriceId([A,A], A)` returns `0`; `priceFeeds[0]` is filled; `priceFeeds[1].id` remains `0`.
7. Final loop: `priceFeeds[1].id == 0` → revert `PriceFeedNotFoundWithinRange`.
8. Entire `executeCallback` transaction reverts; `clearRequest` is never reached.
9. Request remains active indefinitely; user's funds are permanently locked.
