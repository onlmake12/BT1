Audit Report

## Title
Tx-Pool Admission Denial via ProposalShortId Slot Pre-Occupation — (`tx-pool/src/process.rs`, `tx-pool/src/component/verify_queue.rs`, `tx-pool/src/util.rs`)

## Summary
`ProposalShortId` is derived solely from `tx_hash`, which excludes witnesses. The `VerifyQueue` enforces uniqueness on `ProposalShortId`. The P2P relay admission path (`resumeble_process_tx` → `submit_remote_tx`) enqueues transactions after only `non_contextual_verify`, which performs no witness validation. An attacker can pre-occupy a `ProposalShortId` slot with a cousin transaction (identical inputs/outputs, garbage witnesses) before the victim's valid transaction, causing the victim's RPC submission to be rejected with `PoolRejectedDuplicatedTransaction`. With multiple Sybil peers rotating after each ban, the attack can be sustained indefinitely.

## Finding Description

**Root cause — `ProposalShortId` is witness-blind**

`ProposalShortId::from_tx_hash` takes the first 10 bytes of `tx_hash`, which is computed from `RawTransaction` and structurally excludes witnesses. Two transactions with identical inputs, outputs, cell-deps, and header-deps but different witnesses share the same `tx_hash` and `ProposalShortId`. The integration test explicitly confirms this:

```rust
// test/src/specs/tx_pool/collision.rs L214-223
let tx1 = node.new_transaction_spend_tip_cellbase();
let tx2 = tx1.as_advanced_builder().witness(Bytes::default()).build();
assert_eq!(tx1.hash(), tx2.hash());
assert_eq!(tx1.proposal_short_id(), tx2.proposal_short_id());
assert_ne!(tx1.witness_hash(), tx2.witness_hash());
```

**VerifyQueue enforces uniqueness on `ProposalShortId`**

`VerifyEntry` uses `#[multi_index(hashed_unique)]` on `id: ProposalShortId`, making it the sole deduplication key. Both `verify_queue_contains` and `orphan_contains` key on `proposal_short_id()`.

**Relay admission path bypasses witness/script validation**

`submit_remote_tx` calls `resumeble_process_tx_and_notify_full_reject`, which calls `resumeble_process_tx`. This function runs only `non_contextual_verify` before enqueuing. `NonContextualTransactionVerifier` checks only version, size, empty inputs/outputs, duplicate deps, outputs data, and script hash type — no witness validation. A cousin transaction with `Bytes::default()` as witness passes all these checks and is enqueued.

**Victim's transaction rejected as Duplicated**

`process_tx` (RPC path) checks `verify_queue_contains` before proceeding. If the attacker's cousin already occupies the slot, the victim's transaction is immediately rejected with `Reject::Duplicated(tx.hash())`.

**Self-correction is delayed and bypassable**

When the cousin eventually fails script verification in the verify_queue worker, `after_process` calls `ban_malformed`, which bans the peer and removes their transactions from the queue. However, this only occurs after the verify_queue worker processes the cousin — a window that grows with queue depth (up to 256 MB). With multiple Sybil peers each submitting cousin transactions, the attacker can rotate peers to sustain the denial indefinitely, as each peer is only banned after their transactions are individually processed.

**Misleading comment**

The comment in `pre_check` at L280 incorrectly states "Same txid means exactly the same transaction, including inputs, outputs, witnesses" — witnesses are excluded from `tx_hash`.

## Impact Explanation

**High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker with multiple Sybil peers can continuously fill the 256 MB verify_queue with cousin transactions (one per observed victim tx hash), causing:
1. Targeted denial of specific transactions — a victim's valid transaction is rejected until the cousin is dequeued and the peer banned.
2. Sustained network congestion — by rotating Sybil peers, the attacker keeps the verify_queue saturated, delaying all legitimate transaction processing across the node.

The cost is low: creating Sybil peers on a permissionless P2P network requires only bandwidth and IP diversity. No privileged access is required.

## Likelihood Explanation

Any connected P2P peer can send `RelayTransactions` messages. The relay filter requires the node to have previously requested the tx hash via `GetRelayTransactions`, but this is easily triggered by the attacker first announcing the hash via `RelayTransactionHashes`. The victim's transaction structure (inputs/outputs) is observable once broadcast, or predictable for known cell-spending patterns. A single-peer attacker is eventually banned after the cousin fails script verification. However, with multiple Sybil peers (low cost on a permissionless network), the attack is sustained indefinitely. The 256 MB verify_queue limit means the attacker can queue a large volume of cousin transactions before any are processed, extending the denial window significantly.

## Recommendation

1. **Key the verify_queue and orphan pool on `witness_hash` instead of `ProposalShortId`** for deduplication. `witness_hash` covers the full transaction including witnesses, making cousin transactions distinguishable.
2. **Alternatively, perform a lightweight witness structural check** in `resumeble_process_tx` before enqueuing — e.g., verify that the witness count matches the input count — so that obviously invalid cousins are rejected before occupying the slot.
3. **Fix the misleading comment** in `pre_check` at L280 that claims `tx_hash` covers witnesses.

## Proof of Concept

The existing integration test already demonstrates the collision and rejection. The attack is the reverse of the test's submission order:

```rust
// test/src/specs/tx_pool/collision.rs
let tx1 = node.new_transaction_spend_tip_cellbase(); // valid witnesses
let tx2 = tx1.as_advanced_builder().witness(Bytes::default()).build(); // garbage witness
assert_eq!(tx1.hash(), tx2.hash());
assert_eq!(tx1.proposal_short_id(), tx2.proposal_short_id());
assert_ne!(tx1.witness_hash(), tx2.witness_hash());

// Attack: submit tx2 (garbage witnesses) first via P2P relay
// Attacker: RelayTransactionHashes(tx2.hash()) → node requests → attacker sends tx2
// tx2 passes non_contextual_verify, enqueued in verify_queue

// Victim submits tx1 (valid witnesses) via RPC → rejected
let result = node.rpc_client().send_transaction_result(tx1.data().into());
assert!(result.err().unwrap().to_string().contains("PoolRejectedDuplicatedTransaction"));
```

The attacker entry path is `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue`, bypassing all script checks. The victim's entry path is `process_tx` → `verify_queue_contains` → `Reject::Duplicated`.
