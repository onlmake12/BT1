Audit Report

## Title
Tx-Pool Minimum Fee Rate Gate Uses Serialized Size Instead of Weight, Allowing High-Cycle Transactions to Bypass the Pre-Check and Force Expensive Script Verification - (File: tx-pool/src/util.rs)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces `min_fee_rate` using only the transaction's serialized byte size, while the correct weight metric (`max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`) is used everywhere else in the pool. Because cycles are unknown at pre-check time and no secondary fee rate check is performed after verification, an attacker with valid UTXOs can craft a small-serialized, high-cycle transaction that passes the cheap gate and forces the node to perform full, expensive script verification at up to ~60× below the intended minimum fee rate.

## Finding Description

In `check_tx_fee` (`tx-pool/src/util.rs`, lines 42–45), the minimum fee is computed using raw serialized size:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code's own comment acknowledges the approximation. The correct weight formula, defined in `util/types/src/core/tx_pool.rs` (lines 298–303) and used by `TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs`, lines 115–117), is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

`check_tx_fee` is called in `pre_check` (`tx-pool/src/process.rs`, line 289) before script execution, so actual cycles are unavailable. After verification, the entry is created at line 751 with real cycles and admitted to the pool — but no secondary `min_fee_rate` check against the weight-based fee rate is ever performed. The pool's eviction and scoring correctly use weight, but the gate that was supposed to prevent expensive work has already been bypassed.

## Impact Explanation

With `max_tx_verify_cycles = 70,000,000` and `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`, the maximum cycle-derived weight is ≈11,940 bytes. A transaction with serialized size ~200 bytes and ~70M cycles pays only `min_fee_rate × 200 / 1000 = 200 shannons` to pass `check_tx_fee`, while the correct minimum should be `min_fee_rate × 11,940 / 1000 = 11,940 shannons` — a ~60× discount. The node then performs full script verification consuming up to 70M CPU cycles per transaction. Repeated submissions with multiple UTXOs constitute a resource exhaustion / DoS vector, matching the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

## Likelihood Explanation

The attack is reachable by any unprivileged actor via the `send_transaction` RPC or P2P relay. The attacker must hold valid UTXOs (real CKB), which imposes a cost floor, but the ~60× fee discount makes sustained attacks far cheaper than the protocol intends. The discrepancy is deterministic and straightforward to engineer under default consensus parameters.

## Recommendation

Since cycles are unknown at pre-check time, use a conservative upper bound in `check_tx_fee`:

```rust
let max_possible_weight = get_transaction_weight(tx_size, tx_pool.config.max_tx_verify_cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(max_possible_weight);
```

Alternatively, add a secondary fee rate check after verification (when actual cycles are known) and reject the transaction if its effective weight-based fee rate falls below `min_fee_rate`. The first approach is preferable as it preserves the pre-check gate's purpose of avoiding expensive work for under-priced transactions.

## Proof of Concept

1. Obtain a UTXO with capacity C.
2. Construct a transaction with:
   - One input spending that UTXO
   - One output with capacity `C - 200` shannons (fee = 200 shannons)
   - A lock/type script consuming ~70,000,000 cycles
   - Serialized size ~200 bytes
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`; the transaction passes.
5. The node performs full script verification consuming ~70M cycles.
6. The transaction is admitted to the pool with effective fee rate ≈ 17 shannons/KW, far below the configured `min_fee_rate` of 1000 shannons/KW.
7. Repeat with additional UTXOs to continuously exhaust node CPU at ~60× below the intended minimum cost.
