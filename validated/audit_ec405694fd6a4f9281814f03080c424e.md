### Title
Unbounded `NOT IN` SQL Clause from tx-pool Dead Cells Causes Indexer RPC DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`, `get_cells_capacity.rs`)

### Summary

When `index_tx_pool: true` is configured, `get_cells` and `get_cells_capacity` collect **all** dead cells from the indexer's `Pool` without any cap and embed them as SQL parameters in a `NOT IN (...)` clause. An unprivileged attacker who floods the tx-pool with transactions spending many distinct UTXOs can grow this clause to thousands of parameters, causing extreme query latency or hard query failure for every concurrent indexer RPC caller.

### Finding Description

In `get_cells.rs` lines 110–135, the code reads the entire `dead_cells` `HashSet` from the pool and builds one SQL placeholder per entry: [1](#0-0) 

The identical pattern appears in `get_cells_capacity.rs` lines 74–102: [2](#0-1) 

The indexer-side `Pool` struct stores dead cells in an unbounded `HashSet<OutPoint>` with no size cap: [3](#0-2) 

The only guard in `get_cells` is a `request_limit` check on the **result count**, not on the number of dead cells used to build the query: [4](#0-3) 

The tx-pool enforces a `max_tx_pool_size` byte limit: [5](#0-4) 

However, this limits total transaction bytes, not the number of inputs. With a default pool size of ~180 MB and a minimal single-input CKB transaction of ~200–300 bytes, an attacker can pack tens of thousands of transactions, each contributing one dead cell, yielding tens of thousands of SQL parameters — well above SQLite's default variable limit (999 in older builds, 32 766 in newer) and approaching PostgreSQL's 65 535 limit.

### Impact Explanation

- **SQLite backend**: query fails with "too many SQL variables" for every `get_cells`/`get_cells_capacity` call while the pool is flooded, making the indexer RPC completely unresponsive.
- **PostgreSQL backend**: extreme query planning and execution latency as the planner processes a `NOT IN` list with tens of thousands of literals, degrading all concurrent RPC callers.
- Core node operation (sync, block production) is unaffected; only the indexer RPC is impacted.

### Likelihood Explanation

- Transaction submission is fully unprivileged (RPC `send_transaction` or P2P relay).
- The attack requires `index_tx_pool: true`, which **defaults to `false`**: [6](#0-5) 

This is an opt-in operator configuration, which significantly limits the realistic attack surface to nodes that have explicitly enabled tx-pool indexing. Nodes running with the default configuration are not affected.

### Recommendation

Cap the number of dead cells included in the `NOT IN` clause (e.g., skip the filter entirely if `dead_cells.len()` exceeds a safe threshold such as 500), or replace the `NOT IN (...)` approach with a temporary table / batch join that does not grow the query string linearly with pool size.

### Proof of Concept

1. Enable `index_tx_pool = true` in the node config.
2. Submit 5 000 transactions via `send_transaction`, each spending a distinct confirmed UTXO.
3. Call `get_cells` with any valid `search_key`.
4. Observe: on SQLite the call returns an error ("too many SQL variables"); on PostgreSQL, latency spikes proportionally to pool size, blocking all concurrent callers.

---

The vulnerability is **real and reachable** under the stated precondition (`index_tx_pool: true`), but its scope is limited to the indexer RPC subsystem on nodes that have opted into tx-pool indexing.
