Audit Report

## Title
DB Read Amplification via Crafted `difficulties` Array in `GetLastStateProof` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
`GET_LAST_STATE_PROOF_LIMIT = 1000` caps the number of difficulty entries in a `GetLastStateProof` message, but each entry triggers an independent binary-search over the block range via `get_first_block_total_difficulty_is_not_less_than`, costing O(log(end − start)) DB reads per entry. An unprivileged remote peer can craft 1000 strictly-increasing difficulty values that each resolve to a consecutive block near the start of the chain, forcing ~1000 × log₂(N) DB reads per request. All three in-code mitigations are individually bypassable.

## Finding Description
**Root cause — guard counts entries, not reads:**
`GET_LAST_STATE_PROOF_LIMIT = 1000` at `constant.rs:6` caps the array length checked at `get_last_state_proof.rs:201–204`. It does not bound the number of DB operations each entry generates.

**Per-entry cost — binary search:**
`get_first_block_total_difficulty_is_not_less_than` (L29–70) reads `start_block_number`, `end_block_number - 1`, then loops performing a binary search — O(log(end − start)) reads total per call.

**Bypass of `current_difficulty` deduplication (L82–84):**
After finding block `i` with `total_difficulty(i)`, `current_difficulty` is set to `total_difficulty(i)`. The next crafted entry `total_difficulty(i+1)` is strictly greater (enforced by the monotonicity check at L254), so the `continue` is never triggered. The deduplication provides no protection against a well-crafted strictly-increasing sequence.

**Bypass of `start_block_number` advancement (L90–91):**
After finding block `num`, `start_block_number = num - 1`. If crafted difficulties map to consecutive blocks 2, 3, 4, …, 1001, the search range for iteration `i` is `[i-1, N)`. The midpoint of this range is ~N/2, far from block `i+1`, so each binary search still takes O(log N) steps.

**Bypass of `take_while` filter (L325–328):**
The filter removes difficulties above `total_difficulty(difficulty_boundary_block_number - 1)`. An attacker who sets `difficulty_boundary` just above `total_difficulty(1001)` ensures `difficulty_boundary_block_number ≈ 1001`, and all 1000 crafted entries (each equal to `total_difficulty(i)` for i = 1…1000) satisfy `d <= total_difficulty(1000)` and survive the filter.

## Impact Explanation
For the current CKB chain (≈13M blocks), log₂(13,000,000) ≈ 23. A single malicious request causes ~23,000 DB reads instead of ~1,000. With no per-peer rate limiting in the handler, an attacker can sustain a stream of such requests from a single peer, amplifying disk I/O on the light-client server node by ~23×. This maps to **Low (501–2000 points): important performance improvement for CKB** — the amplification is real, attacker-controlled, and repeatable, but a single request does not crash the node or cause network-wide congestion.

## Likelihood Explanation
- The entrypoint is the standard unauthenticated P2P `GetLastStateProof` message, reachable by any peer.
- Required knowledge (per-block total difficulty) is public and queryable from any synced full node.
- No rate limiting or per-peer request quota is present in the handler.
- The attack is fully repeatable in a tight loop from a single peer with no special privileges.

## Recommendation
Bound the total number of DB reads, not just the number of difficulty entries. Concretely: introduce a per-request read counter inside `get_block_numbers_via_difficulties` or inside `get_first_block_total_difficulty_is_not_less_than` and abort once it exceeds `GET_LAST_STATE_PROOF_LIMIT`. Alternatively, cap the number of difficulties that actually enter the binary-search path (i.e., those that pass the `current_difficulty` check at L82) to a constant well below 1000, independent of the raw array length.

## Proof of Concept
On a chain of N blocks (e.g., N = 1,000,000):

1. Query the node to obtain `total_difficulty(i)` for i = 1 … 1000.
2. Build `difficulties = [total_difficulty(1), total_difficulty(2), …, total_difficulty(1000)]` — strictly increasing, each equal to the actual total difficulty of block i.
3. Set `start_number = 0`, `last_n_blocks = 0`, `difficulty_boundary = total_difficulty(1001)`, `last_hash` = current tip.
4. Send the `GetLastStateProof` P2P message.

**Expected behavior:** `get_block_numbers_via_difficulties` calls `get_first_block_total_difficulty_is_not_less_than` 1000 times. Iteration i searches range `[i-1, N)`, performing ~log₂(N) reads each. Total DB reads ≈ 1000 × log₂(N) ≈ 20,000 for N = 1M, versus the ~1000 the guard was designed to allow. Repeating this in a loop amplifies node I/O proportionally.
