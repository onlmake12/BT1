### Title
Unprivileged User Can Exhaust `maxNumHashes` Limit to DoS Entropy Randomness Requests - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Pyth Entropy's `requestHelper()` computes `numHashes` as the gap between the current sequence number and the provider's `currentCommitmentSequenceNumber`. When a provider sets `maxNumHashes`, any request whose `numHashes` exceeds that limit reverts with `LastRevealedTooOld`. An unprivileged attacker can submit many requests without ever revealing them, causing `numHashes` to grow until it exceeds `maxNumHashes`, at which point **all** new randomness requests from any user revert — a global DoS of the Entropy service for that provider.

---

### Finding Description

In `requestHelper()`, the `numHashes` field is computed as:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [1](#0-0) 

`currentCommitmentSequenceNumber` only advances when `revealHelper` is called (i.e., when a request is fulfilled). It does **not** advance automatically when new requests are submitted. [2](#0-1) 

`providerInfo.sequenceNumber` increments on every new request regardless of whether prior requests are ever revealed: [3](#0-2) 

**Attack path:**

1. Attacker calls `requestV2()` (or `request()`) `maxNumHashes` times, paying the fee each time, but never calls `reveal()` / `revealWithCallback()`.
2. `providerInfo.sequenceNumber` grows to `originalSequenceNumber + maxNumHashes`.
3. `currentCommitmentSequenceNumber` remains at its original value (no reveals happened).
4. The next legitimate user's request computes `numHashes = sequenceNumber - currentCommitmentSequenceNumber > maxNumHashes` and reverts with `LastRevealedTooOld`.
5. All new requests are blocked until the Fortuna keeper calls `advanceProviderCommitment()`.
6. The attacker can immediately re-exhaust the limit after each keeper intervention.

The Fortuna keeper does monitor this and triggers `advanceProviderCommitment` at 95% of `maxNumHashes`: [4](#0-3) 

However, the keeper response is not instantaneous (it polls every 30 seconds), and the attacker can re-exhaust the limit immediately after each keeper call, creating a sustained DoS at the cost of `maxNumHashes * fee` per cycle.

The `maxNumHashes` guard is explicitly opt-in and is used by the Pyth default provider: [5](#0-4) 

The error is documented as a known operational concern: [6](#0-5) 

---

### Impact Explanation

All new randomness requests to the targeted provider revert with `LastRevealedTooOld` for the duration of the attack. Any application relying on Pyth Entropy (gaming, NFT minting, DeFi randomness) is unable to request new random numbers. The DoS is temporary per cycle but can be sustained indefinitely by a motivated attacker. This matches the external report's impact class: **temporary unavailability of the protocol's core function**.

---

### Likelihood Explanation

- **Entry point is fully permissionless**: any address can call `requestV2()`.
- **Cost**: the attacker must pay `maxNumHashes × fee` per DoS cycle. On low-gas L2s (Arbitrum, Base, Optimism — all supported Entropy chains), this can be very cheap.
- **No refund**: unlike the Axelar case where the attacker recovers tokens, here fees are lost. This raises the cost but does not eliminate the attack on cheap chains.
- **Keeper mitigation has a 30-second polling gap** and requires the attacker to spend only one additional cycle to re-exhaust after each keeper response.
- The Fortuna keeper itself explicitly logs this as a possible DDOS attack scenario, confirming the threat model is known but not fully mitigated on-chain. [7](#0-6) 

---

### Recommendation

1. **On-chain rate limiting per requester**: Track per-address request counts within a sliding window to prevent a single address from exhausting the global `numHashes` budget.
2. **Increase `maxNumHashes` relative to expected concurrent requests**: A larger limit makes the attack more expensive.
3. **Reduce keeper polling interval**: The 30-second `UPDATE_COMMITMENTS_INTERVAL` creates a window; reducing it limits the DoS duration per cycle.
4. **Fee escalation for unrevealed requests**: Charge a higher fee for requests that have been pending for a long time, making sustained non-revelation economically unattractive.

### Proof of Concept

```solidity
// Attacker contract
contract EntropyDoS {
    IEntropy entropy;
    address provider;
    uint32 maxNumHashes; // read from provider info

    function attack() external payable {
        uint128 fee = entropy.getFee(provider);
        // Submit maxNumHashes requests, never reveal them
        for (uint32 i = 0; i < maxNumHashes; i++) {
            bytes32 userCommitment = entropy.constructUserCommitment(bytes32(uint256(i)));
            entropy.request{value: fee}(provider, userCommitment, false);
        }
        // Now all new requests from any user revert with LastRevealedTooOld
    }
}
```

After `attack()` completes, any call to `entropy.request(provider, ...)` or `entropy.requestV2(...)` targeting the same provider reverts with `EntropyErrors.LastRevealedTooOld` until the Fortuna keeper calls `advanceProviderCommitment`. The attacker can call `attack()` again immediately after the keeper responds, sustaining the DoS.
