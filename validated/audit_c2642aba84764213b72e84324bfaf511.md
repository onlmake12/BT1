### Title
Unbounded `pending_compact_blocks` Growth via Rate-Limiter-Exempt CompactBlock Flood with Minimum-Difficulty PoW — (`sync/src/relayer/compact_block_process.rs`)

---

### Summary

An unprivileged remote peer can exhaust the memory of any CKB node by flooding it with syntactically valid compact blocks that carry a trivially-solvable PoW (minimum `compact_target`), each containing one unknown `short_id`. Because the relay path's `HeaderVerifier` does not validate that `compact_target` matches the epoch-expected difficulty, and because CompactBlock messages are explicitly exempt from the rate limiter, every such block passes all checks and is inserted into the unbounded `pending_compact_blocks` `HashMap`. No eviction or size cap exists on that map in the `Missing` path.

---

### Finding Description

**1. Rate-limiter exemption (confirmed)**

In `sync/src/relayer/mod.rs`, `try_process` explicitly skips rate-limit checking for every `CompactBlock` message:

```rust
let should_check_rate =
    !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));
``` [1](#0-0) 

The comment reads *"CompactBlock will be verified by POW, it's OK to skip rate limit checking."* This is the root assumption that the rest of the analysis invalidates.

**2. `HeaderVerifier` does not check `compact_target` against epoch-expected difficulty**

`contextual_check` runs `HeaderVerifier::verify()`: [2](#0-1) 

`HeaderVerifier::verify()` performs only four checks: [3](#0-2) 

- `PowVerifier` — verifies the nonce against the header's **own** `compact_target`, not the epoch-expected one.
- `NumberVerifier` — block number = parent + 1.
- `EpochVerifier` — epoch is well-formed and is a successor of the parent epoch.
- `TimestampVerifier` — timestamp is within bounds.

There is **no check** that the `compact_target` field in the header matches the difficulty the consensus algorithm expects for that epoch. That check only occurs during full contextual block verification, which is never reached for a `Missing` compact block. An attacker can therefore set `compact_target = 0x20ffffff` (difficulty = 1, the minimum representable) and solve the Eaglesong PoW in microseconds on commodity hardware. [4](#0-3) 

**3. `non_contextual_check` adds no difficulty guard**

The only non-contextual checks are uncle count, proposal count, and a staleness window (tip − epoch_length). None constrain `compact_target`. [5](#0-4) 

**4. `CompactBlockVerifier::verify` adds no difficulty guard**

It only checks prefilled-transaction ordering and short-ID uniqueness. [6](#0-5) 

**5. `missing_or_collided_post_process` inserts into an unbounded map**

When reconstruction yields `ReconstructionResult::Missing`, the compact block is stored unconditionally: [7](#0-6) 

`PendingCompactBlockMap` is a plain `HashMap` with no capacity limit: [8](#0-7) 

Initialized as `HashMap::default()` with no bound: [9](#0-8) 

**6. The only cleanup path is never triggered**

`pending_compact_blocks.retain(...)` and `shrink_to_fit!` are called only inside `ReconstructionResult::Block`, i.e., only when a block is fully reconstructed and accepted: [10](#0-9) 

When every incoming compact block has one unknown `short_id`, reconstruction always returns `Missing`, so this cleanup branch is never reached.

**7. Per-peer deduplication is insufficient**

The existing guard only prevents the same `(peer, block_hash)` pair from being inserted twice: [11](#0-10) 

A single peer can send an unlimited number of distinct block hashes (different nonces → different hashes, all building on the same known parent), each of which passes the guard and is inserted as a new map entry.

---

### Impact Explanation

Each entry in `pending_compact_blocks` stores a full `packed::CompactBlock` (header + short IDs + prefilled transactions) plus a `HashMap<PeerIndex, (Vec<u32>, Vec<u32>)>`. A compact block with one short ID is on the order of hundreds of bytes. At 10,000 entries the map consumes tens of megabytes; at 10^6 entries it consumes gigabytes. Because the relay thread holds the `tokio::sync::Mutex` on `pending_compact_blocks` during every insert, a sustained flood also causes relay-thread starvation: legitimate compact blocks from honest peers queue behind the mutex, delaying block propagation across the network.

---

### Likelihood Explanation

The attack requires only a single P2P connection and the ability to mine blocks at difficulty 1 (trivial on any CPU). No privileged access, no key material, no majority hashpower, and no Sybil attack is needed. The attacker needs only to know the current tip hash (publicly available) to construct valid parents. The attack is fully local-testable.

---

### Recommendation

1. **Enforce a size cap on `pending_compact_blocks`**: Reject insertion (or evict the oldest entry) when the map exceeds a fixed limit (e.g., 512 or 1024 entries).
2. **Validate `compact_target` against the epoch-expected difficulty in the relay path**: Before inserting into `pending_compact_blocks`, verify that the header's `compact_target` matches the difficulty the consensus expects for that epoch, consistent with what full block verification already enforces.
3. **Apply a per-peer rate limit to CompactBlock messages**: The current exemption assumes PoW is a sufficient cost barrier, but at minimum difficulty that assumption fails. A per-peer limit of, e.g., 10 CompactBlock messages per second would bound the insertion rate without affecting honest peers.

### Proof of Concept

```
1. Spin up a private CKB node (or connect to a testnet node).
2. Obtain the current tip block hash H and its epoch E.
3. Construct compact blocks B_1 … B_N where each B_i has:
     - parent_hash = H
     - epoch = successor of E (well-formed)
     - compact_target = 0x20ffffff  (difficulty 1)
     - nonce = i  (distinct hash per block)
     - short_ids = [one unknown short_id]
     - prefilled_transactions = [cellbase]
4. For each B_i, solve Eaglesong PoW against target 0xffffff…ff (trivial).
5. Send all N compact blocks over the RelayV3 protocol to the victim node.
6. Assert: pending_compact_blocks.len() == N (grows linearly).
7. Assert: RSS of the victim process grows linearly with N.
```
