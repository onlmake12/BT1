### Title
Excess ETH Not Refunded in `updatePriceFeeds` and Related Payable Functions — (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary

`updatePriceFeeds`, `parsePriceFeedUpdatesWithConfig`, and `parseTwapPriceFeedUpdates` in `Pyth.sol` are `payable` and enforce a minimum fee via `msg.value >= requiredFee`, but they never refund any excess ETH sent above the required fee. Any overpayment is silently absorbed into the contract's balance and is irrecoverable by the original sender.

### Finding Description

`updatePriceFeeds` checks that `msg.value` meets the minimum required fee but performs no refund of any surplus:

```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
// No refund of msg.value - requiredFee
``` [1](#0-0) 

`parsePriceFeedUpdatesWithConfig` (called by `parsePriceFeedUpdates` and `parsePriceFeedUpdatesUnique`) has the same pattern:

```solidity
if (msg.value < getUpdateFee(updateData))
    revert PythErrors.InsufficientFee();
// No refund
``` [2](#0-1) 

`parseTwapPriceFeedUpdates` is identical:

```solidity
uint requiredFee = getTwapUpdateFee(updateData);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
// No refund
``` [3](#0-2) 

Unlike the Entropy contract, which explicitly tracks `accruedPythFeesInWei` and exposes a `withdrawFee` admin function, the core Pyth price-feed contract has no analogous per-sender accounting. Excess ETH accumulates in the contract's raw balance. [4](#0-3) 

`PythGovernance.sol` does contain withdraw/transfer-fee helpers (confirmed by grep), so governance can drain the balance — but the original overpaying caller has no path to recover their own funds.

### Impact Explanation

Any caller of `updatePriceFeeds`, `parsePriceFeedUpdates`, `parsePriceFeedUpdatesUnique`, or `parseTwapPriceFeedUpdates` who sends more ETH than the exact required fee permanently loses the surplus. The surplus is not credited to any per-user balance and cannot be reclaimed by the sender. At best it is recoverable only by a privileged governance action directed to an arbitrary address, not to the original payer. This constitutes a direct, quantifiable loss of user funds on every overpaying call.

### Likelihood Explanation

The required fee depends on the number of price-feed updates packed inside the VAA blob, which callers must parse off-chain before calling `getUpdateFee`. In practice:

- Integrators frequently add a small ETH buffer to guarantee the transaction does not revert due to fee estimation errors or last-minute VAA changes.
- Wallets and SDK wrappers that call `updatePriceFeeds` on behalf of users commonly pass `msg.value` slightly above the estimated fee.
- The `getUpdateFee(uint updateDataSize)` overload (deprecated but still present) over-estimates by up to 255× per blob, making callers who use it systematically overpay. [5](#0-4) 

These conditions make overpayment a routine occurrence, not a theoretical edge case.

### Recommendation

Add a refund of any surplus after the fee check in each affected function:

```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
if (msg.value > requiredFee) {
    (bool ok, ) = msg.sender.call{value: msg.value - requiredFee}("");
    require(ok, "refund failed");
}
```

Apply the same pattern to `parsePriceFeedUpdatesWithConfig` and `parseTwapPriceFeedUpdates`. This is the same approach already used in `PythLazer.sol`:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [6](#0-5) 

### Proof of Concept

1. Deploy the Pyth contract on a local fork.
2. Determine `requiredFee` for a single-update VAA via `getUpdateFee(updateData)`.
3. Call `updatePriceFeeds{value: requiredFee + 1 ether}(updateData)`.
4. Observe that the call succeeds, the price feed is updated, and the contract's ETH balance increased by `requiredFee + 1 ether`.
5. Confirm there is no function the original caller can invoke to recover the `1 ether` surplus — only a governance action can move it, and to an arbitrary address, not back to the caller.
