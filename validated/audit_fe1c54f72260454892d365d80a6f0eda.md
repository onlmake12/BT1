### Title
Wrong Circular-Buffer Index in `estimate_median` Causes Permanently Inflated `extra_count` in Fee Rate Estimation — (`File: util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

The `ConfirmationFraction` fee estimator maintains a circular buffer `block_unconfirmed_txs` keyed by `entry_height % MAX_CONFIRM_BLOCKS`. When a transaction is added, its count is stored at slot `entry_height % 1000`. However, `estimate_median` reads this buffer using `confirm_blocks - 1` as the index — a completely different key — causing `extra_count` (the pending-tx penalty in the confirmation-rate denominator) to reflect an arbitrary, unrelated slot of the circular buffer rather than the actual count of long-pending transactions. This is a direct analog to the external report's "state added to one window slot, removed/read from a different slot" vulnerability class.

---

### Finding Description

`TxConfirmStat` uses `block_unconfirmed_txs: Vec<Vec<usize>>` as a circular buffer of size `MAX_CONFIRM_BLOCKS = 1000`. The first dimension is a circular slot index, the second is a fee-rate bucket index.

**Add path** — `add_unconfirmed_tx` stores the count at `entry_height % 1000`:

```rust
fn add_unconfirmed_tx(&mut self, entry_height: u64, fee_rate: FeeRate) -> Option<usize> {
    let bucket_index = self.bucket_index_by_fee_rate(fee_rate)?;
    let block_index = (entry_height % (self.block_unconfirmed_txs.len() as u64)) as usize;
    self.block_unconfirmed_txs[block_index][bucket_index] += 1;
    Some(bucket_index)
}
``` [1](#0-0) 

**Read path** — `estimate_median` reads the buffer using `confirm_blocks - 1` as the index:

```rust
extra_count += &self.block_unconfirmed_txs[confirm_blocks - 1][bucket_index];
``` [2](#0-1) 

`confirm_blocks - 1` is a confirmation-target offset (ranging from 2 to 999 for valid inputs), not a circular-buffer slot keyed by block height. The slot `block_unconfirmed_txs[confirm_blocks - 1]` actually holds the count of transactions that entered the pool at heights `≡ (confirm_blocks - 1) (mod 1000)` — a completely unrelated set of transactions. The two keys are semantically incompatible.

The same wrong index is used again in the inner sliding-window subtraction loop:

```rust
extra_count -= &self.block_unconfirmed_txs[confirm_blocks - 1][start_bucket_index];
``` [3](#0-2) 

The `move_track_window` and `remove_unconfirmed_tx` functions correctly use `entry_height % 1000` as the slot key, so the circular buffer is maintained correctly — but `estimate_median` reads it with the wrong key, producing a permanently mismatched `extra_count`. [4](#0-3) 

---

### Impact Explanation

`extra_count` appears in the confirmation-rate denominator:

```rust
let confirm_rate = confirmed_txs / (txs_count + failure_count + extra_count as f64);
``` [5](#0-4) 

When `extra_count` is inflated (the misread slot happens to contain many entries), `confirm_rate` is artificially depressed, causing `estimate_median` to fail to find a satisfying bucket at the correct fee rate and return a higher fee rate than warranted. When the slot is near-zero, `extra_count` is deflated and the estimator returns a lower fee rate than warranted. Because the slot read is determined by the fixed `confirm_blocks - 1` value for each priority mode (e.g., `DEFAULT_TARGET = 128` always reads slot 127), the error is systematic and persistent, not random noise.

The `estimate_fee_rate` RPC is exposed to any caller: [6](#0-5) 

---

### Likelihood Explanation

**High.** Any tx-pool submitter can submit transactions at a block height `H` where `H % 1000 == confirm_blocks - 1` for the target priority mode. For `NoPriority` (`DEFAULT_TARGET = 128`), the attacker targets slot 127, meaning any block at height `127, 1127, 2127, …`. These heights occur naturally every 1000 blocks (~7.8 hours at 28 s/block), so no special timing is required. The attacker simply submits transactions to the pool at those heights and lets them sit unconfirmed. The bug is also triggered passively by ordinary network traffic at those heights, meaning it fires without any deliberate attacker.

---

### Recommendation

Replace the wrong index `confirm_blocks - 1` in `estimate_median` with the correct circular-buffer slot key. The `extra_count` should represent the count of transactions that have been pending for at least `confirm_blocks` blocks. One correct approach is to sum over all circular-buffer slots whose corresponding heights are older than `confirm_blocks` blocks relative to `best_height`, or to maintain a separate age-indexed structure. Alternatively, follow Bitcoin Core's approach and use `old_unconfirmed_txs` (already correctly maintained) as the proxy for long-pending transactions, and remove the `block_unconfirmed_txs` read from `estimate_median` entirely, replacing it with a properly age-gated count.

---

### Proof of Concept

1. Node is at `best_height = 1127`. Attacker submits 100 transactions at height 1127 (`1127 % 1000 = 127`). Each is tracked: `block_unconfirmed_txs[127][bucket] += 1` for each tx.

2. Attacker calls `estimate_fee_rate` with `NoPriority` → `target_blocks = DEFAULT_TARGET = 128`.

3. `estimate_median(128, ...)` executes: `extra_count += block_unconfirmed_txs[128 - 1][bucket]` = `block_unconfirmed_txs[127][bucket]` = 100 (the attacker's txs).

4. The confirmation rate becomes: `confirmed_txs / (txs_count + failure_count + 100)`. With 100 phantom "extra" pending txs inflating the denominator, `confirm_rate` drops below `DEFAULT_MIN_CONFIRM_RATE = 0.85`, causing the estimator to skip valid low-fee-rate buckets and return an inflated fee rate.

5. The attacker's transactions never need to be confirmed or dropped — they persist in the pool, keeping `block_unconfirmed_txs[127]` elevated indefinitely, permanently biasing the `NoPriority` fee estimate upward. [7](#0-6) [8](#0-7) [9](#0-8)
