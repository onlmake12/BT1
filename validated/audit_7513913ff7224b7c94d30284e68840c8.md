Audit Report

## Title
Missing `MAX_RELAY_TXS_NUM_PER_BATCH` Count Check in `TransactionsProcess::execute()` Enables CPU Exhaustion via Oversized `RelayTransactions` Message — (`sync/src/relayer/transactions_process.rs`)

## Summary
`TransactionsProcess::execute()` performs no count guard on the number of transactions in an incoming `RelayTransactions` message before entering an iterator chain that calls `tx.transaction().to_entity().into_view()` — a full molecule deserialization — on every entry. Both sibling handlers (`TransactionHashesProcess` and `GetTransactionsProcess`) enforce `MAX_RELAY_TXS_NUM_PER_BATCH = 32767` at their entry points. An unprivileged relay peer can send a single oversized `RelayTransactions` message to force unbounded deserialization work on the receiving node's relay processing thread.

## Finding Description
The asymmetry is confirmed directly in the source:

`transaction_hashes_process.rs` line 29 guards with:
```rust
if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
    return StatusCode::ProtocolMessageIsMalformed.with_context(...);
}
``` [1](#0-0) 

`get_transactions_process.rs` line 35 guards with:
```rust
if message_len > MAX_RELAY_TXS_NUM_PER_BATCH {
    return StatusCode::ProtocolMessageIsMalformed.with_context(...);
}
``` [2](#0-1) 

`transactions_process.rs` has no such guard. `execute()` proceeds immediately to:
```rust
self.message
    .transactions()
    .iter()
    .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))
    .filter(|(tx, _)| {
        !tx_filter.contains(&tx.hash())
            && unknown_tx_hashes
                .get_priority(&tx.hash())
                .map(|priority| priority.requesting_peer() == Some(self.peer))
                .unwrap_or_default()
    })
    .collect()
``` [3](#0-2) 

In Rust's lazy iterator model, `.map()` executes per-element before `.filter()`. This means `to_entity().into_view()` — a full molecule deserialization into an owned `TransactionView` — is called for **every** transaction in the message, regardless of whether it passes the filter. The `unknown_tx_hashes` filter bounds the output `Vec` size (capped at `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767`), so allocated memory in the result is bounded, but CPU cost is proportional to the total number of transactions in the incoming message, which has no enforced upper bound on the receiving side. [4](#0-3) 

The rate limiter in `mod.rs` is keyed by `(peer, message.item_id())` at 30 req/s — it limits message frequency, not message size, so one large message per rate window is sufficient to trigger the issue. [5](#0-4) 

No receiving-side byte cap is applied to `RelayTransactions` in the relay handler. The outbound byte cap `MAX_RELAY_TXS_BYTES_PER_BATCH = 1024 * 1024` is enforced only on the sending side in `GetTransactionsProcess` and `prune_tx_proposal_request`. [6](#0-5) 

## Impact Explanation
An attacker can force the victim node to perform unbounded deserialization work (CPU) on a single `RelayTransactions` message. A message containing hundreds of thousands of fabricated transactions causes proportional CPU consumption on the relay processing thread. Sustained or repeated attacks within the rate-limit window can degrade or crash a CKB node. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attack requires only a standard relay peer connection — no keys, no hash power, no privilege. The attacker fully controls the content of `RelayTransactions` messages. Pre-staging via `RelayTransactionHashes` is not required: even with fabricated hashes absent from `unknown_tx_hashes`, the `.map()` deserialization still executes for all entries before `.filter()` discards them. The rate limiter (30/s) does not prevent a single large message from being processed. The attack is repeatable and requires minimal attacker resources.

## Recommendation
Add the same count guard at the top of `TransactionsProcess::execute()` that exists in the other two handlers:

```rust
pub fn execute(self) -> Status {
+   let tx_count = self.message.transactions().len();
+   if tx_count > MAX_RELAY_TXS_NUM_PER_BATCH {
+       return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
+           "RelayTransactions count({tx_count}) > MAX_RELAY_TXS_NUM_PER_BATCH({MAX_RELAY_TXS_NUM_PER_BATCH})",
+       ));
+   }
    let shared_state = self.relayer.shared().state();
```

Also import `MAX_RELAY_TXS_NUM_PER_BATCH` and `StatusCode` in `transactions_process.rs`, mirroring the pattern in `transaction_hashes_process.rs`. [7](#0-6) 

## Proof of Concept
1. Connect to a CKB node as a relay peer.
2. Construct a `packed::RelayTransactions` message containing `N >> MAX_RELAY_TXS_NUM_PER_BATCH` entries (e.g., 500,000 minimal transactions with fabricated hashes). Send it.
3. `TransactionsProcess::execute()` enters the iterator chain at line 45 with no count guard. `.map()` calls `to_entity().into_view()` for all 500,000 entries before `.filter()` discards those not in `unknown_tx_hashes`.
4. Observe sustained CPU spike on the node's relay processing thread. Repeat once per rate-limit window to maintain pressure.

A unit test can be written by constructing a `packed::RelayTransactions` with `N > MAX_RELAY_TXS_NUM_PER_BATCH` entries, calling `TransactionsProcess::new(...).execute()`, and asserting that the call returns `StatusCode::ProtocolMessageIsMalformed` — currently it does not; it returns `Status::ok()` after iterating all entries. [8](#0-7)
