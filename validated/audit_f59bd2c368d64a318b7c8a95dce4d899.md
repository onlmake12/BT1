Audit Report

## Title
Tx-Pool Admission Uses Size-Only Fee Check While Post-Verification Entry Uses Weight-Based Fee Rate — (`tx-pool/src/util.rs`, `tx-pool/src/process.rs`)

## Summary
`check_tx_fee` gates pool admission using only `tx_size` to compute the minimum required fee, while `TxEntry::fee_rate()`, `EvictKey`, and `AncestorsScoreSortKey` all use `get_transaction_weight(tx_size, cycles)` — the `max` of size and cycle-equivalent bytes. After `verify_rtx` returns the actual cycle count, no second fee-rate check is performed. An attacker can craft a small transaction with near-maximum cycles, pay only the size-based minimum fee, force full script verification, and be admitted with a weight-based fee rate up to ~60× below `min_fee_rate`.

## Finding Description
In `tx-pool/src/process.rs`, `_process_tx` calls `pre_check` (which invokes `check_tx_fee` with `tx_size` only), then calls `verify_rtx` to obtain the actual cycle count, and unconditionally constructs a `TxEntry` and calls `submit_entry` with no post-verification fee-rate re-check: [1](#0-0) 

`check_tx_fee` in `tx-pool/src/util.rs` explicitly acknowledges the approximation in its comment and uses only `tx_size`: [2](#0-1) 

Meanwhile, `TxEntry::fee_rate()` uses the weight-based formula: [3](#0-2) 

Which calls: [4](#0-3) 

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` and `max_tx_verify_cycles = 70_000_000`:
- A 200-byte transaction consuming 70 M cycles has weight ≈ 11,940 bytes
- Size-based `min_fee = 1000 × 200 / 1000 = 200 shannons` → passes admission
- Weight-based fee rate = `200 / 11,940 × 1000 ≈ 16.7 shannons/KW` → ~60× below `min_fee_rate`

The code comment at `tx-pool/src/util.rs:42–44` explicitly acknowledges this is a deliberate approximation ("cheap check"), confirming the structural inconsistency is known but unguarded.

## Impact Explanation
Any unprivileged `send_transaction` RPC caller or P2P peer can force the node to execute up to 70 M cycles of script verification per transaction while paying only the size-based minimum fee (200 shannons). The transaction is admitted, immediately becomes the top eviction candidate by weight-based fee rate, and is evicted — but the CPU cost has already been paid. Repeated at scale, this constitutes a sustained CPU-exhaustion attack on individual nodes and can cause CKB network congestion with minimal attacker cost. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (10001–15000 points)**.

## Likelihood Explanation
No special privilege is required. Any caller with access to `send_transaction` RPC or P2P relay can trigger this. Crafting a CKB-VM script that runs a tight loop consuming ~70 M cycles is straightforward. The attacker cost is 200 shannons per transaction (the size-based minimum fee), while the node pays 70 M cycles of verification. The inconsistency is structural and present in every node running the default configuration. The attack is indefinitely repeatable as long as the attacker holds minimal CKB.

## Recommendation
After `verify_rtx` returns the actual cycle count in `_process_tx`, re-check the fee rate using the true weight before constructing the `TxEntry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(
        tx_pool.config.min_fee_rate,
        min_fee_by_weight.as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

This ensures the admission gate uses the same cost metric as eviction and scoring. Alternatively, derive a conservative upper-bound cycle estimate from transaction structure before full execution to enforce a cycles-proportional fee floor at the pre-check stage.

## Proof of Concept
1. Construct a CKB transaction with a single input/output (serialized size ≈ 200 bytes) and a lock script that executes a tight loop consuming ≈ 70,000,000 cycles.
2. Set fee = `ceil(min_fee_rate × tx_size / 1000)` = 200 shannons (passes size-based check in `check_tx_fee`).
3. Submit via `send_transaction` RPC to a node with default config (`min_fee_rate = 1000`).
4. Observe: the node accepts the transaction (no `LowFeeRate` rejection), runs full script verification consuming ~70 M cycles, admits the entry with effective weight-based fee rate ≈ 16.7 shannons/KW, then immediately marks it as the lowest-priority eviction candidate.
5. Repeat in a loop. Each iteration costs the attacker 200 shannons and forces the node to spend ~70 M cycles of verification work.
