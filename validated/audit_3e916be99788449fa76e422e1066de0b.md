### Title
Unbounded `tx.gasprice` in `_processFeesAndPayKeeper` Allows Keeper to Drain Subscription Balance and Extract Profit via `GAS_OVERHEAD` Multiplier — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `_processFeesAndPayKeeper` internal function computes the keeper reimbursement as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
```

There is no cap on `tx.gasprice`. Because `GAS_OVERHEAD` (a fixed constant) is **added** to the measured gas before multiplying by `tx.gasprice`, a malicious keeper can set an arbitrarily high gas price to:

1. Drain a subscription's entire balance in a single `updatePriceFeeds` call.
2. Extract a net profit of `GAS_OVERHEAD × tx.gasprice` per call, since the keeper pays only `actualGasUsed × tx.gasprice` to the validator but receives `(actualGasUsed + GAS_OVERHEAD) × tx.gasprice + keeperSpecificFee` from the subscription.

The subscription manager has no mechanism to specify a maximum acceptable fee per update — there is no slippage protection equivalent.

---

### Finding Description

`updatePriceFeeds` is a permissionless function callable by any address. At the end of a successful call it invokes `_processFeesAndPayKeeper`:

```solidity
function _processFeesAndPayKeeper(
    SchedulerStructs.SubscriptionStatus storage status,
    uint256 startGas,
    uint256 numPriceIds
) internal {
    uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
    uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
        numPriceIds;
    uint256 totalKeeperFee = gasCost + keeperSpecificFee;

    if (status.balanceInWei < totalKeeperFee) {
        revert SchedulerErrors.InsufficientBalance();
    }

    status.balanceInWei -= totalKeeperFee;
    status.totalSpent += totalKeeperFee;

    (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
    ...
}
``` [1](#0-0) 

The `GAS_OVERHEAD` constant is added to the actual gas consumed before multiplying by `tx.gasprice`. This overhead is intended to compensate for gas costs not captured between `startGas` and `gasleft()`. However, because it is multiplied by the caller-controlled `tx.gasprice`, it becomes a direct profit lever for the keeper.

**Keeper profit per call:**
```
profit = GAS_OVERHEAD × tx.gasprice + keeperSpecificFee
```

The keeper pays `actualGasUsed × tx.gasprice` to the block proposer and receives `(actualGasUsed + GAS_OVERHEAD) × tx.gasprice + keeperSpecificFee` from the subscription. The net gain scales linearly with `tx.gasprice`.

The subscription manager sets no `maxGasPrice` or `maxFeePerUpdate` in `SubscriptionParams` — the struct contains only `priceIds`, `updateCriteria`, `readerWhitelist`, `isActive`, and `isPermanent`. [2](#0-1) 

---

### Impact Explanation

A malicious keeper calling `updatePriceFeeds` with `tx.gasprice = P`:

- Drains `(actualGas + GAS_OVERHEAD) × P + keeperSpecificFee` from the subscription in one transaction.
- Profits `GAS_OVERHEAD × P + keeperSpecificFee` net.
- At `P = 10,000 gwei` and `GAS_OVERHEAD = 50,000 gas`: profit ≈ **0.5 ETH per call**.
- The subscription balance can be fully drained in a single update, permanently disrupting the push-feed service for all downstream readers.

The subscription manager suffers direct loss of deposited ETH far exceeding the cost of a legitimate update. [3](#0-2) 

---

### Likelihood Explanation

- `updatePriceFeeds` requires no registration, whitelist, or privileged role — any EOA can call it.
- Setting a high `tx.gasprice` (or `maxPriorityFeePerGas` on EIP-1559 chains) is trivially achievable by any transaction sender.
- The attack is profitable for the keeper whenever `GAS_OVERHEAD × tx.gasprice > gas cost of the inflated tip`, which is always true since the keeper receives the overhead reimbursement but only pays actual gas.
- The only constraint is that `totalKeeperFee ≤ status.balanceInWei`; the keeper can observe the subscription balance on-chain and choose a `tx.gasprice` that extracts the maximum amount without reverting. [4](#0-3) 

---

### Recommendation

1. **Add a `maxGasPriceInWei` field to `SubscriptionParams`** that the subscription manager sets at creation/update time. In `_processFeesAndPayKeeper`, cap the effective gas price:
   ```solidity
   uint256 effectiveGasPrice = Math.min(tx.gasprice, params.maxGasPriceInWei);
   uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
   ```
2. **Alternatively, cap `GAS_OVERHEAD` reimbursement separately** from the variable gas cost so it is not multiplied by an unbounded `tx.gasprice`.
3. **Emit the actual fee paid** in the `PricesUpdated` event so subscription managers can monitor for anomalous drain rates.

---

### Proof of Concept

1. Subscription manager calls `createSubscription` with 2 ETH balance, 2 price feeds, heartbeat = 60 s.
2. Attacker (keeper) waits 60 s, then calls `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = 50,000 gwei`.
3. Inside `_processFeesAndPayKeeper`:
   - `actualGasUsed ≈ 250,000`
   - `gasCost = (250,000 + GAS_OVERHEAD) × 50,000 gwei`
   - If `GAS_OVERHEAD = 50,000`: `gasCost = 300,000 × 50,000 gwei = 15 ETH` → reverts (balance only 2 ETH).
   - Attacker reads balance (2 ETH), solves for `P`: `(300,000) × P = 2 ETH` → `P ≈ 6,667 gwei`.
   - At `P = 6,667 gwei`: `gasCost ≈ 2 ETH`, subscription drained in one call.
   - Keeper net profit: `GAS_OVERHEAD × 6,667 gwei = 50,000 × 6,667 gwei ≈ 0.33 ETH`.
4. Subscription balance is zero; all future `updatePriceFeeds` calls revert with `InsufficientBalance`, breaking the push feed. [5](#0-4)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
