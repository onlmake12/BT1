### Title
`PythAggregatorV3` Returns Stale Prices Without Staleness Validation Across All Price-Returning Functions — (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatibility adapter contract, uses `getPriceUnsafe()` in every price-returning function — including `latestAnswer()`, `latestRoundData()`, `getRoundData()`, and `decimals()` — with no staleness check. Additionally, `answeredInRound` is always set equal to `roundId`, defeating the standard Chainlink staleness guard pattern. Any protocol that migrates from Chainlink to Pyth using this adapter will silently consume arbitrarily stale prices.

---

### Finding Description

`PythAggregatorV3.sol` is the official Pyth SDK adapter that implements Chainlink's `AggregatorV3Interface` on top of Pyth price feeds. It is actively recommended in Pyth's Chainlink migration guide.

Every price-returning function in the contract calls `pyth.getPriceUnsafe(priceId)`:

```solidity
// latestAnswer — line 53-56
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// latestRoundData — line 99-119
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}

// getRoundData — line 76-97
function getRoundData(uint80 _roundId) external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return (_roundId, int256(price.price), price.publishTime, price.publishTime, _roundId);
}
```

`getPriceUnsafe()` is explicitly documented as returning a price "from arbitrarily far in the past" with no recency guarantee. The safe alternative, `getPriceNoOlderThan()`, is never used.

A second compounding issue: `answeredInRound` is always set to the same value as `roundId` (both derived from `price.publishTime`). Protocols that apply the standard Chainlink staleness guard `require(answeredInRound >= roundId, "stale")` will always pass, even when the price is hours or days old. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Impact: High**

Protocols that use `PythAggregatorV3` as a drop-in Chainlink replacement will silently receive stale prices from both `latestAnswer()` and `latestRoundData()`. This leads to:

- Incorrect collateral valuations in lending protocols (e.g., borrowing against overvalued collateral using a stale high price)
- Incorrect liquidation thresholds (e.g., blocking valid liquidations or triggering invalid ones)
- Incorrect share price / NAV calculations in vault protocols

The `updatedAt` field returned by `latestRoundData()` is `price.publishTime`, which a downstream protocol could check — but the adapter itself never enforces any threshold, and the `answeredInRound == roundId` invariant defeats the most common Chainlink staleness guard. [4](#0-3) 

---

### Likelihood Explanation

**Likelihood: Medium**

Pyth's own documentation explicitly recommends `PythAggregatorV3` for Chainlink migrations. Protocols migrating from Chainlink expect `latestRoundData()` to behave safely (as it does on Chainlink, where the feed contract itself enforces freshness). The Pyth adapter silently breaks this assumption. Since Pyth is a pull oracle, price feeds are only updated when someone calls `updatePriceFeeds()`. If no keeper is running, the on-chain price can become arbitrarily stale, and the adapter will return it without any revert. [5](#0-4) 

---

### Recommendation

Replace all `getPriceUnsafe()` calls in `PythAggregatorV3` with `getPriceNoOlderThan()` using a configurable staleness threshold. Add a constructor parameter (e.g., `maxStaleness`) and use it in every price-returning function:

```solidity
uint256 public maxStaleness; // e.g., 3600 seconds

constructor(address _pyth, bytes32 _priceId, uint256 _maxStaleness) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxStaleness = _maxStaleness;
}

function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxStaleness);
    return int256(price.price);
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxStaleness);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

`getPriceNoOlderThan()` reverts with `StalePrice()` if the price is older than the threshold, matching the behavior downstream Chainlink-compatible protocols expect. [6](#0-5) 

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a Pyth contract where the target feed was last updated 2 hours ago (no keeper running).
2. Call `latestRoundData()` on the adapter.
3. Observe: the call succeeds and returns a 2-hour-old price. `updatedAt` is 2 hours in the past. `answeredInRound == roundId`, so `require(answeredInRound >= roundId)` passes.
4. A lending protocol consuming this adapter accepts the stale price as current, allowing a user to borrow against collateral at an incorrect valuation.

The `getPriceUnsafe()` documentation confirms this behavior explicitly:

> "This function returns the most recent price update in this contract without any recency checks. This function is unsafe as the returned price update may be arbitrarily far in the past." [7](#0-6) [8](#0-7)

        bytes32 id,
        uint age

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();
