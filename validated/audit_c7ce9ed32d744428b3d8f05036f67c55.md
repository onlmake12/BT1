### Title
`get_fee_rate_statistics` Returns Inflated Fee Rates During `assume_valid_target` IBD Due to Zero-Cycle `BlockExt` Entries — (File: `rpc/src/util/fee_rate.rs`)

---

### Summary

`FeeRateCollector::statistics()` reads `BlockExt.cycles` directly from the database without checking whether the node was in `assume_valid_target` mode when those blocks were processed. During `assume_valid_target` IBD — the **default** for all mainnet and testnet nodes — blocks are verified with `Switch::DISABLE_SCRIPT`, causing every transaction's cycle count to be stored as `0`. The fee rate calculation then uses these zero-cycle values, producing systematically inflated fee rate statistics that are returned to any RPC caller.

---

### Finding Description

**The "paused" state — `assume_valid_target` disables script execution**

When `assume_valid_target` is active, `chain/src/verify.rs` selects `Switch::DISABLE_SCRIPT` for each block:

```rust
// chain/src/verify.rs ~line 233
Switch::DISABLE_SCRIPT
``` [1](#0-0) 

Because `switch.disable_all()` is false, the normal verification path runs and `insert_ok_ext` is called with `Some(&cache_entries)`. However, inside `BlockTxsVerifier::verify()`, `skip_script_verify = true` forces every transaction's cycle count to `0`:

```rust
// verification/contextual/src/contextual_block_verifier.rs
let cycles = if skip_script_verify { 0 } else { self.script.verify(max_cycles)? };
``` [2](#0-1) 

The resulting `Completed { cycles: 0, fee: actual_fee }` entries are stored via `insert_ok_ext`, which writes `ext.cycles = Some(vec![0, 0, ...])` and `ext.txs_fees = actual_fees` into the database: [3](#0-2) 

**The accounting function that ignores the paused state**

`FeeRateCollector::statistics()` iterates over recent `BlockExt` records and computes fee rates without any check on `BlockExt.verified` or whether the node was in `assume_valid_target` mode:

```rust
// rpc/src/util/fee_rate.rs
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
if txs_sizes.len() > 1 && !txs_fees.is_empty() {
    if let Some(cycles) = cycles {
        for (fee, cycles, size) in itertools::izip!(txs_fees, cycles, txs_sizes.iter().skip(1)) {
            let weight = get_transaction_weight(*size as usize, cycles);
            if weight > 0 {
                fee_rates.push(FeeRate::calculate(fee, weight).as_u64());
            }
        }
    }
}
``` [4](#0-3) 

With `cycles = 0`, `get_transaction_weight(size, 0)` returns a weight based solely on byte size, omitting the cycle contribution. Since `fee_rate = fee / weight`, a lower weight produces a **higher (inflated) fee rate**. The `BlockExt` struct confirms that `cycles` is the field being misread: [5](#0-4) 

**Contrast with the correctly guarded fee estimator**

The more sophisticated `FeeEstimator` (used by `estimate_fee_rate`) correctly handles this state by tracking `is_ready` and returning `Error::NotReady` during IBD:

```rust
// util/fee-estimator/src/estimator/weight_units_flow.rs
pub fn estimate_fee_rate(...) -> Result<FeeRate, Error> {
    if !self.is_ready {
        return Err(Error::NotReady);
    }
    ...
}
``` [6](#0-5) 

`get_fee_rate_statistics` has no equivalent guard.

**Secondary issue — panic when `txs_sizes` is `None`**

When blocks are processed with `Switch::DISABLE_ALL` (e.g., `ckb import --skip-all-verify`), `insert_ok_ext` is called with `txs_sizes = None`: [7](#0-6) 

Any subsequent call to `get_fee_rate_statistics` hits the unconditional `expect` on line 93 of `rpc/src/util/fee_rate.rs` and panics, crashing the RPC task.

---

### Impact Explanation

Any RPC caller invoking `get_fee_rate_statistics` or `get_fee_rate_statics` on a node that processed blocks under `assume_valid_target` (the default for mainnet and testnet) receives inflated fee rate statistics. Wallets and tooling that rely on this endpoint to set transaction fees will systematically overpay during and for up to 101 blocks after IBD completes. The `FeeRateCollector` reads from the canonical chain snapshot, so the stale zero-cycle data persists in the returned statistics until enough post-IBD blocks displace the window.

---

### Likelihood Explanation

`assume_valid_target` is enabled by default for all mainnet and testnet nodes. Every node that performs IBD using the default configuration will store zero-cycle `BlockExt` entries for the entire IBD range. The `get_fee_rate_statistics` RPC is a public, unauthenticated endpoint callable by any local or remote RPC client. No special privileges or attacker-controlled block content are required; the incorrect state is already present in the database from normal node operation.

---

### Recommendation

`FeeRateCollector::statistics()` should skip `BlockExt` entries where `cycles` contains all-zero values (indicating `DISABLE_SCRIPT` processing) or where `BlockExt.verified` is not `Some(true)`. Concretely:

1. In `rpc/src/util/fee_rate.rs`, filter out blocks whose `cycles` vector is all zeros before computing fee rates, or expose an IBD-awareness flag analogous to `FeeEstimator::is_ready`.
2. Replace the unconditional `txs_sizes.expect(...)` with a graceful `if let Some(txs_sizes) = txs_sizes` guard to prevent the panic when `txs_sizes` is `None`.
3. Consider returning `null` from `get_fee_rate_statistics` when the node is in IBD, consistent with the behavior of `estimate_fee_rate`.

---

### Proof of Concept

1. Start a fresh mainnet or testnet node (default config, `assume_valid_target` active).
2. Allow IBD to proceed for at least 21 blocks.
3. Call `get_fee_rate_statistics` via RPC.
4. Observe that the returned `mean` and `median` fee rates are inflated relative to the actual network fee rates, because all cycle counts in the sampled `BlockExt` records are `0`.
5. After IBD completes and 101 new blocks are produced with full script verification, the statistics normalize — confirming the root cause is the zero-cycle `BlockExt` entries written during `assume_valid_target` processing.
