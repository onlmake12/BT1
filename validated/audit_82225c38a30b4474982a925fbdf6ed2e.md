### Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Duplicate `RelayTransactionHashes` — (File: `sync/src/types/mod.rs`)

### Summary
`UnknownTxHashPriority::push_peer` appends a `PeerIndex` to a `Vec<PeerIndex>` without any duplicate check. An unprivileged relay peer can repeatedly announce the same unknown transaction hash via `RelayTransactionHashes` messages, causing the `peers` Vec for that single entry to grow without bound, exhausting node memory.

### Finding Description
`UnknownTxHashPriority` tracks which peers announced a given unknown transaction hash and is stored in `SyncState::unknown_tx_hashes` (a `KeyedPriorityQueue`). [1](#0-0) 

The `push_peer` method unconditionally appends to the `peers` Vec: [2](#0-1) 

In `add_ask_for_txs`, when a tx_hash is already present in the queue (the `Occupied` branch), `push_peer` is called with no guard against duplicate peer entries: [3](#0-2) 

The global overflow check that follows is only triggered when `unknown_tx_hashes.len()` exceeds `MAX_UNKNOWN_TX_HASHES_SIZE` or a per-peer threshold: [4](#0-3) 

If the attacker uses only **one** tx_hash, `unknown_tx_hashes.len()` stays at `1` and this guard is **never reached**, so the `peers` Vec for that single entry grows without bound.

The entry point is `TransactionHashesProcess::execute`, which filters hashes only against `tx_filter` (hashes already fully processed/known). A hash that is pending in `unknown_tx_hashes` but not yet resolved is **not** in `tx_filter`, so it passes the filter on every repeated message: [5](#0-4) 

### Impact Explanation
Each repeated `RelayTransactionHashes` message containing the same unknown tx_hash causes one additional `PeerIndex` (8 bytes) to be pushed into the `peers` Vec of the corresponding `UnknownTxHashPriority` entry. Because the global size guard is bypassed when only one tx_hash is used, the `peers` Vec grows without any upper bound, leading to unbounded heap memory consumption on the victim node. Sustained over a long-lived connection this constitutes a remote memory-exhaustion denial-of-service.

### Likelihood Explanation
The relay protocol is open to any connected peer. The rate limiter caps throughput at 30 messages/second per `(PeerIndex, message_type)` pair: [6](#0-5) 

At 30 req/s a single peer adds ~108,000 duplicate `PeerIndex` entries per hour (~864 KB/hr). Multiple peers or a higher-bandwidth connection amplify this. No authentication or privilege is required.

### Recommendation
In `push_peer`, check for an existing entry before appending:

```rust
pub fn push_peer(&mut self, peer_index: PeerIndex) {
    if !self.peers.contains(&peer_index) {
        self.peers.push(peer_index);
    }
}
```

Alternatively, replace `Vec<PeerIndex>` with a `HashSet<PeerIndex>` or `LinkedHashSet<PeerIndex>` to make deduplication O(1) and structurally enforced.

### Proof of Concept
1. Connect to a CKB node as a relay peer (standard `RelayV3` protocol).
2. Craft a `RelayTransactionHashes` message containing a single tx_hash that the node does not know (e.g., a random 32-byte value that will never resolve).
3. Send this message in a tight loop (up to 30/s due to the rate limiter).
4. Because the hash is never resolved into a known transaction, it stays in `unknown_tx_hashes` and never enters `tx_filter`.
5. Each message hits the `Occupied` branch in `add_ask_for_txs` and calls `push_peer`, appending the same `PeerIndex` to the `peers` Vec.
6. The global size guard at line 1507 is never triggered (`unknown_tx_hashes.len() == 1`).
7. Observe monotonically increasing heap usage on the victim node. [7](#0-6)

        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {

                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }
