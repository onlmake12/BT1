Audit Report

## Title
`check_tx_fee` Uses Serialized Size Instead of Weight for Minimum Fee Enforcement, Allowing Cycle-Heavy Transactions to Bypass `min_fee_rate` — (`tx-pool/src/util.rs`)

## Summary

`check_tx_fee` computes the minimum required fee using only `tx_size` (serialized bytes) rather than `weight = max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. Since `FeeRate` is defined as shannons per kilo-weight, this produces a systematic underestimate of the required fee for cycle-heavy transactions. A transaction with a small serialized size but near-maximum cycle consumption can pass admission while paying a true weight-based fee rate up to ~59× below the configured `min_fee_rate`, enabling cheap CPU exhaustion of the node's script-verification pipeline.

## Finding Description

`FeeRate` is explicitly defined as shannons per kilo-weight, and `fee()` multiplies by weight: [1](#0-0) [2](#0-1) 

`weight` is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`: [3](#0-2) 

`check_tx_fee` passes `tx_size as u64` directly to `min_fee_rate.fee()`, with a comment explicitly acknowledging the approximation: [4](#0-3) 

`check_tx_fee` is called inside `pre_check`, **before** script verification runs: [5](#0-4) 

After `verify_rtx` returns the actual `verified.cycles`, there is **no second fee-rate check** using weight. The entry is created and submitted directly: [6](#0-5) 

The only post-verification check is a `declared_cycles` equality check (lines 736–749), which does not involve fee rate at all. For local RPC submissions, `declared_cycles` is `None`, so even that check is skipped. For P2P relay, the attacker simply declares the correct cycle count.

By contrast, all internal pool operations (sorting, eviction) correctly use `get_transaction_weight`: [7](#0-6) 

The same mismatch exists in `calculate_min_replace_fee` for RBF: [8](#0-7) 

The `readd_detached_tx` path also calls `check_tx_fee` with the same size-only logic, meaning re-org recovery is equally affected: [9](#0-8) 

**Exploit flow**: An attacker crafts a transaction with ~200 bytes serialized size and a RISC-V lock script consuming ~70,000,000 cycles (the default `max_tx_verify_cycles`). With `min_fee_rate = 1000` shannons/KW:

- Size-based min fee (what is checked): `1000 × 200 / 1000 = 200 shannons`
- Weight-based min fee (what should be checked): `1000 × max(200, 70_000_000 × 0.000_170_571_4) / 1000 = 1000 × 11,940 / 1000 = 11,940 shannons`

A transaction paying 201 shannons passes `check_tx_fee` and proceeds to `verify_rtx`, which executes the full 70,000,000-cycle script. The attacker pays ~59× less than the intended economic floor per unit of CPU consumed. By flooding the `verify_queue` with such transactions, the attacker saturates the node's script-verification pipeline at minimal cost.

## Impact Explanation

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** The verify queue is a sequential processing pipeline; filling it with cycle-heavy transactions at 1/59th the intended fee cost delays or starves legitimate transaction processing. The `readd_detached_tx` path also calls `check_tx_fee` with the same size-only logic, meaning re-org recovery is equally affected.

## Likelihood Explanation

The entry path is fully unprivileged via `send_transaction` JSON-RPC or P2P relay. No special privileges, leaked keys, or victim mistakes are required. The attacker only needs to deploy a loop-heavy RISC-V script (standard CKB capability) and keep the serialized transaction small. The attack is repeatable and cheap: each submission costs 201 shannons (~0.000000201 CKB) and forces the node to execute up to 70,000,000 cycles of script verification. The code comment at `util.rs:42–44` confirms the developers were aware of the approximation, establishing that the behavior is exactly as described.

## Recommendation

After `verify_rtx` returns the actual cycle count in `_process_tx`, add a post-verification fee-rate check using `get_transaction_weight`:

```rust
// tx-pool/src/process.rs, after line 734
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

The existing size-based check in `pre_check` can remain as a cheap early filter. Apply the same fix to `calculate_min_replace_fee` in `tx-pool/src/pool.rs`, passing the replacement transaction's weight (requiring cycles to be available at RBF check time).

## Proof of Concept

1. Start a CKB node with default config (`min_fee_rate = 1000` shannons/KW, `max_tx_verify_cycles = 70_000_000`).
2. Deploy a cell whose lock script is a RISC-V loop consuming ~70,000,000 cycles.
3. Construct a transaction spending that cell: serialized size ~200 bytes, fee = 201 shannons.
4. Submit via `send_transaction` RPC. **Expected (correct)**: rejected with `LowFeeRate` (weight-based min fee = 11,940 shannons). **Actual**: accepted into the pool; node executes 70,000,000 cycles of script verification.
5. Repeat in a tight loop. Monitor node CPU and `verify_queue` depth; legitimate transactions submitted concurrently will experience significant processing delays.

/// The precision is set so that the difference between MAX_BLOCK_CYCLES * DEFAULT_BYTES_PER_CYCLES
/// and MAX_BLOCK_BYTES is less than 1.
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

/// vbytes has been deprecated, renamed to weight to prevent ambiguity
#[deprecated(
    since = "0.107.0",
    note = "Please use the get_transaction_weight instead"
)]
pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
