### Title
Permanent Subscription `addFunds` Silently Locks Third-Party ETH Forever With No Warning or Revert - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.addFunds()` accepts ETH from **any caller** with no access control. When called against a subscription where `isPermanent == true`, the deposited ETH is permanently and irrecoverably locked in the contract. `withdrawFunds()` unconditionally reverts for permanent subscriptions. `addFunds()` contains no guard, revert, or event warning the caller that their funds will be permanently locked. The subscription IDs of permanent subscriptions are publicly discoverable via `getActiveSubscriptions()`.

---

### Finding Description

`addFunds` is declared with no access modifier — any EOA or contract may call it for any active subscription ID:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    ...
    if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }
    status.balanceInWei += msg.value;   // ETH credited, no refund path
    ...
}
``` [1](#0-0) 

The only permanent-subscription check in `addFunds` is the deposit cap (`MAX_DEPOSIT_LIMIT`). There is no check that reverts or warns when `isPermanent == true` and the caller is not the subscription manager.

`withdrawFunds` unconditionally reverts for permanent subscriptions regardless of who deposited:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [2](#0-1) 

The test suite itself documents this as intentional behavior — "Anyone can add funds (not just manager)" — immediately followed by a confirmed revert on withdrawal: [3](#0-2) 

The `IScheduler` NatDoc for `addFunds` contains no mention of permanent subscriptions or the irrecoverability of deposited funds: [4](#0-3) 

---

### Impact Explanation

Any unprivileged caller who sends ETH via `addFunds` to a permanent subscription permanently loses that ETH. There is no admin escape hatch, no refund function, and no upgrade path that returns the funds to the depositor. The loss is bounded by `MAX_DEPOSIT_LIMIT` per call but is repeatable across multiple calls up to that cap each time. [5](#0-4) 

---

### Likelihood Explanation

- Permanent subscription IDs are publicly enumerable via `getActiveSubscriptions()` with no access control.
- Integrators building on top of the Scheduler (e.g., a contract that auto-tops-up a subscription) may not inspect `isPermanent` before calling `addFunds`.
- A user who previously funded a non-permanent subscription and later the manager upgrades it to permanent (a one-way transition) will lose any subsequent top-up ETH.
- The `IScheduler` interface and NatDoc give no indication that `addFunds` behaves differently for permanent subscriptions. [6](#0-5) 

---

### Recommendation

**Short term:** Add an explicit revert in `addFunds` when `params.isPermanent == true` and `msg.sender` is not the subscription manager, or revert unconditionally for permanent subscriptions since their balance is self-sustaining by design:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

**Long term:** Update the `IScheduler` NatDoc for `addFunds` to explicitly state that calling this function on a permanent subscription results in irrecoverable fund loss. Mirror the same warning in `createSubscription` when `isPermanent == true`.

---

### Proof of Concept

1. Deploy `Scheduler` (or use a live instance).
2. Manager calls `createSubscription` with `isPermanent = true` and funds it to `minimumBalance`.
3. Victim (any address) calls `addFunds{value: 1 ether}(subscriptionId)` — succeeds, ETH credited.
4. Victim calls `withdrawFunds(subscriptionId, 1 ether)` — reverts with `CannotUpdatePermanentSubscription`.
5. Manager calls `withdrawFunds` — also reverts with `CannotUpdatePermanentSubscription`.
6. Victim's 1 ETH is permanently locked with no recovery path. [1](#0-0) [7](#0-6)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
```
