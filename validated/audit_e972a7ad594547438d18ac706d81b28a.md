### Title
Out-of-Order Governance VAA Submission Permanently Blocks Lower-Sequence VAAs — (`lazer/contracts/sui/sources/governance.move`)

---

### Summary

`governance::process_incoming` enforces only `seen_sequence < sequence` (not `seen_sequence + 1 == sequence`). Because `actions::update_trusted_signer` and `actions::upgrade` are unrestricted `public` functions, any observer can submit a higher-sequence governance VAA before a lower one, permanently making the lower sequence unprocessable and potentially preventing a new trusted signer from ever being registered.

---

### Finding Description

The sequence guard in `process_incoming` is:

```move
assert!(self.seen_sequence < sequence, EOldSequenceNumber);
self.seen_sequence = sequence;
``` [1](#0-0) 

This is a strictly-greater-than check, not a consecutive-sequence check. Once `seen_sequence` is set to any value S, every VAA with sequence ≤ S is permanently rejected.

Both governance action entrypoints are declared `public` with no caller restriction:

```move
public fun update_trusted_signer(state: &mut State, vaa: VAA) { ... }
public fun upgrade(state: &mut State, vaa: VAA): UpgradeTicket { ... }
``` [2](#0-1) [3](#0-2) 

Wormhole VAAs are publicly observable once emitted. An attacker who sees two pending governance VAAs — sequence N (e.g., add new signer) and sequence N+1 (any action) — can submit N+1 first. After that call, `seen_sequence = N+1`, and any subsequent attempt to process sequence N hits `assert!(N+1 < N)` → aborts with `EOldSequenceNumber`. [4](#0-3) 

---

### Impact Explanation

If the blocked VAA was adding a replacement trusted signer before the existing signer's `expires_at` deadline passes, no new signer can be registered via that VAA. The governance team must emit a fresh VAA with sequence > N+1. A persistent attacker can repeat the attack on every new VAA, indefinitely blocking signer rotation. Once all `TrustedSignerInfo` entries expire, price-update verification fails for every downstream consumer, effectively freezing any protocol that gates actions on a valid Lazer price. [5](#0-4) 

---

### Likelihood Explanation

- **No privilege required.** The attacker is any Wormhole observer; both `update_trusted_signer` and `upgrade` accept calls from any address.
- **VAAs are public.** Wormhole guardian signatures are broadcast; the attacker needs only to observe two pending VAAs and submit the higher-sequence one first.
- **Sui has no traditional mempool**, but the race is not mempool-based — it is simply "who submits first." Because both VAAs are available simultaneously on Wormhole, the attacker can act before the official relayer.
- **Recovery is possible but attackable.** The governance team can emit a new VAA with sequence > N+1, but a persistent attacker can keep front-running each new emission.

---

### Recommendation

Replace the strictly-greater-than check with a consecutive-sequence check:

```move
assert!(self.seen_sequence + 1 == sequence, EOldSequenceNumber);
```

This ensures VAAs must be processed in strict order, making it impossible to skip a sequence. Alternatively, use a `VecSet<u64>` of processed sequences (replay map) and allow any sequence to be processed exactly once regardless of order, which is the pattern used by the main Pyth Wormhole contracts. [1](#0-0) 

---

### Proof of Concept

```move
#[test, expected_failure(abort_code = governance::EOldSequenceNumber)]
public fun test_out_of_order_blocks_lower_sequence() {
    // VAA_SEQ_5 = governance VAA with sequence=5 (add new signer)
    // VAA_SEQ_6 = governance VAA with sequence=6 (unrelated action)
    // Attacker submits sequence 6 first
    update_trusted_signer(&mut state, VAA_SEQ_6); // seen_sequence → 6
    // Now sequence 5 is permanently unprocessable
    update_trusted_signer(&mut state, VAA_SEQ_5); // aborts: assert!(6 < 5)
}
```

State after the first call: `seen_sequence = 6`. [1](#0-0) 
Second call aborts at the same assert, confirming the invariant is broken.
