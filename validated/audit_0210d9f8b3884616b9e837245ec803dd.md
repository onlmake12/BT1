### Title
`TransactionTemplate.depends` and `TransactionTemplate.required` Fields Documented but Never Populated in `get_block_template` RPC — (File: `tx-pool/src/block_assembler/mod.rs`)

---

### Summary

The `get_block_template` RPC returns a `BlockTemplate` containing `TransactionTemplate` entries. Two fields — `depends` and `required` — are fully documented in the type definition and the RPC README as critical hints for miners selecting transaction subsets. Both fields are hardcoded to stub values (`None` and `false`) with explicit `// unimplemented` comments in the production code path. The tx-pool internally tracks full parent-child dependency graphs but never exposes them through the template. Miners who rely on the `depends` field to safely select transaction subsets will always receive `null`, causing them to unknowingly assemble invalid blocks when they omit a parent transaction while including its child.

---

### Finding Description

In `tx-pool/src/block_assembler/mod.rs`, the function `tx_entry_to_template` converts a pooled transaction entry into the JSON-RPC `TransactionTemplate` type:

```rust
pub(crate) fn tx_entry_to_template(entry: &TxEntry) -> TransactionTemplate {
    TransactionTemplate {
        hash: entry.transaction().hash().into(),
        required: false, // unimplemented
        cycles: Some(entry.cycles.into()),
        depends: None, // unimplemented
        data: entry.transaction().data().into(),
    }
}
``` [1](#0-0) 

The `TransactionTemplate` type in `util/jsonrpc-types/src/block_template.rs` documents `depends` as:

> "Transaction dependencies. This is a hint to help miners selecting transactions. This transaction can only be committed if its dependencies are also committed in the new block. This field is a list of indices into the array `transactions` in the block template." [2](#0-1) 

And `required` as:

> "Whether miner must include this transaction in the new block." [3](#0-2) 

The `BlockTemplate` RPC documentation reinforces this contract:

> "Miners must include the transactions marked as `required` in the assembled new block." [4](#0-3) 

Critically, the tx-pool **does** maintain full parent-child dependency graphs. `TxLinksMap` tracks `parents` and `children` per transaction: [5](#0-4) 

And `TxSelector.txs_to_commit` explicitly packages ancestors in topological order before adding them to the template: [6](#0-5) 

The dependency information exists internally but is discarded at the serialization boundary in `tx_entry_to_template`. Every miner calling `get_block_template` receives `"depends": null` for every transaction, regardless of actual dependency structure.

---

### Impact Explanation

The `get_block_template` RPC is explicitly designed to allow miners to **select a subset** of the provided transactions:

> "Miners optional pick transactions and then assemble the final block." [7](#0-6) 

When a miner selects a subset — for example, to stay within a custom byte or cycle budget — they have no way to know which transactions carry intra-block dependencies. A miner who includes a child transaction (spending an output created by another transaction in the same template) without including its parent will produce a block that fails contextual verification at `submit_block`. The PoW work for that block is entirely wasted. An adversary who floods the mempool with long parent-child chains can systematically maximize the probability that any subset selection by a miner produces an invalid block, amplifying wasted hashrate across the mining pool.

---

### Likelihood Explanation

The `get_block_template` RPC is the standard interface for all external miners and mining pools. Any miner that implements custom transaction selection logic (fee filtering, size trimming, cycle-budget enforcement) will encounter this gap. The `depends` field is part of the published API contract and miners implementing compliant clients per the RPC documentation will trust it. The tx-pool regularly contains parent-child transaction chains (CPFP is a documented and tested feature), so the dependency scenario is not hypothetical.

---

### Recommendation

In `tx_entry_to_template`, compute the `depends` field by:
1. Looking up the transaction's ancestors in the pool's `TxLinksMap`.
2. Intersecting those ancestors with the set of transactions already added to the current template.
3. Emitting the indices of matching entries in the `transactions` array.

Similarly, evaluate whether any transactions should be marked `required: true` (e.g., transactions whose omission would invalidate the DAO field or other template-level commitments) and populate that field accordingly.

---

### Proof of Concept

1. Submit two transactions to the tx-pool: `tx_parent` (spends a live cell) and `tx_child` (spends an output of `tx_parent`). Both enter the proposed pool.
2. Call `get_block_template`. Both appear in `template.transactions`. Observe that `tx_child.depends` is `null`.
3. Construct a block using only `tx_child` (omitting `tx_parent`), using the template's `cellbase`, `dao`, `compact_target`, etc.
4. Solve PoW and call `submit_block`.
5. The node rejects the block with a dead-cell or resolve error because `tx_child`'s input references an output that does not exist in the chain and is not provided by any other transaction in the block.
6. The miner's PoW work is wasted. With `depends` correctly populated, step 3 would have shown `tx_child.depends = [index_of_tx_parent]`, allowing the miner to either include both or exclude both.
