### Title
`RecentReject::put()` Never Increments `self.total_keys_num`, Disabling `shrink()` and Allowing Unbounded RocksDB Growth — (`tx-pool/src/component/recent_reject.rs`)

### Summary
`RecentReject::put()` computes a local `total_keys_num` but never writes it back to `self.total_keys_num`. As a result, the shrink threshold is never reached for any `count_limit >= 1`, and the `DBWithTTL` backing store grows without bound as rejected transactions are inserted.

### Finding Description
In `put()` (lines 62–69), the pattern is:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // ← self.total_keys_num = total_keys_num; is MISSING here
}
```

`self.total_keys_num` is initialized once in `build()` from a RocksDB estimate (line 44–51). After that, every call to `put()` re-computes `self.total_keys_num + 1` into a local variable but never stores it. `self.total_keys_num` is only ever updated inside `shrink()` (line 111), which is unreachable unless the stale counter already exceeds `count_limit`. For a freshly started node with an empty DB, `self.total_keys_num` stays at `0` forever, so the guard `1 > count_limit` is false for any `count_limit >= 1`, and `shrink()` is never invoked. [3](#0-2) 

### Impact Explanation
Every rejected transaction (any reject reason: `LowFeeRate`, `Full`, `Verification`, `Resolve`, etc.) is written to the `DBWithTTL` shard via `self.db.put()` at line 60. With `shrink()` permanently disabled, the on-disk RocksDB store grows proportionally to the number of rejected transactions received. An attacker who can continuously submit transactions that are rejected (e.g., low-fee-rate transactions via P2P relay or the `send_transaction` RPC) can exhaust disk space and crash the node. [4](#0-3) 

### Likelihood Explanation
The attack path is straightforward and requires no privilege: submit transactions that will be rejected (e.g., below minimum fee rate) via the public `send_transaction` RPC or P2P relay. The `recent_reject` DB is enabled by default when a non-empty path is configured. The bug is deterministic — it triggers on every single `put()` call regardless of `count_limit`.

### Recommendation
Add `self.total_keys_num = total_keys_num;` inside the `if let Some` branch of `put()`, after the `checked_add` succeeds and before the `shrink` threshold check:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;  // ← missing assignment
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```
