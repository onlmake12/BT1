### Title
Unsorted Peer-Supplied Block Locator Hashes Violate Ordering Assumption in `insert_peer_unknown_header_list`, Causing IBD Sync Stall — (`sync/src/types/mod.rs`)

---

### Summary

During Initial Block Download (IBD), when a CKB node receives a `GetHeaders` message from an outbound/protected peer, it calls `insert_peer_unknown_header_list` to populate the peer's `unknown_header_list`. This function and the downstream `BlockFetcher` loop both rely on the invariant that the peer-supplied block locator hashes are sorted from **highest to lowest** block number, and both use an **early-break** when a known hash is found. A malicious outbound peer can send a `GetHeaders` message with block locator hashes in a deliberately wrong order, violating this invariant. The early-break then fires prematurely, causing the node to set `best_known_header` for that peer to an artificially low block number and never insert the peer's actual unknown (higher) hashes. As a result, the node stops fetching blocks from that peer for the remainder of IBD.

---

### Finding Description

**Root cause — `insert_peer_unknown_header_list` in `sync/src/types/mod.rs`:**

```rust
pub fn insert_peer_unknown_header_list(&self, pi: PeerIndex, header_list: Vec<Byte32>) {
    if self.state().peers.unknown_header_list_is_empty(pi) {
        // header list is an ordered list, sorted from highest to lowest,
        // so here you discard and exit early
        for hash in header_list {
            if let Some(header) = self.shared().header_map().get(&hash) {
                self.state()
                    .peers
                    .may_set_best_known_header(pi, header.as_header_index());
                break;                          // ← early-break on first known hash
            } else {
                self.state().peers.insert_unknown_header_hash(pi, hash)
            }
        }
    }
}
```

The comment explicitly states the invariant: *"header list is an ordered list, sorted from highest to lowest, so here you discard and exit early."* The `break` is only correct when the list is sorted, because then every hash after the first known one is guaranteed to be at a lower (already-known) block number. [1](#0-0) 

**Caller — `get_headers_process.rs`:**

The `header_list` passed to this function is taken verbatim from the peer's `GetHeaders` wire message (`block_locator_hashes`). No ordering validation is performed before the call. [2](#0-1) 

**Second early-break — `block_fetcher.rs`:**

The downstream consumer of `unknown_header_list` also relies on the same sorted invariant:

```rust
// unknown list is an ordered list, sorted from highest to lowest,
// when header hash unknown, break loop is ok
while let Some(hash) = state.peers().take_unknown_last(self.peer) {
    if let Some(header) = self.sync_shared.get_header_index_view(&hash, false) {
        state.peers().may_set_best_known_header(self.peer, header.as_header_index());
    } else {
        state.peers().insert_unknown_header_hash(self.peer, hash);
        break;   // ← early-break: assumes all hashes above this are also unknown
    }
}
``` [3](#0-2) 

**Attack scenario:**

A malicious outbound peer sends a `GetHeaders` message where `block_locator_hashes` is deliberately **not** sorted from highest to lowest — for example, it places a hash the victim already knows (e.g., genesis or an old checkpoint) at position 0, followed by hashes for blocks at much higher numbers that the victim does not yet have.

`insert_peer_unknown_header_list` iterates:
1. Sees the known hash at position 0 → calls `may_set_best_known_header` with the old/low block → **breaks immediately**.
2. All subsequent hashes (the peer's actual high-number unknown blocks) are **never inserted** into `unknown_header_list`.

Consequence: `best_known_header` for this peer is pinned to the old low block. `BlockFetcher` sees no useful unknown hashes and no better `best_known_header`, so it fetches nothing from this peer for the rest of IBD.

---

### Impact Explanation

- The victim node stops downloading blocks from the affected peer connection for the duration of IBD.
- If the attacker controls multiple outbound slots (or the victim has few outbound peers), this can significantly delay or effectively stall IBD.
- No consensus state is corrupted; the node can still sync from honest peers on other connections. However, in a targeted attack where the attacker occupies all or most outbound slots, the node's IBD can be indefinitely stalled.

---

### Likelihood Explanation

- The attack requires the adversary to be an **outbound peer** of the victim (or whitelist/protected), which is the condition checked at the call site.
- Establishing outbound connections to a target node is straightforward for a motivated attacker (run a reachable CKB node and wait for the victim to connect, or advertise via DNS seeds).
- The malicious `GetHeaders` message is a single, cheap-to-craft P2P message.
- The `unknown_header_list_is_empty` guard means the attack only needs to succeed **once per peer connection** — after the first malformed message is processed, the list is considered populated and subsequent calls are ignored, locking in the corrupted state for the lifetime of the connection. [4](#0-3) 

---

### Recommendation

Before calling `insert_peer_unknown_header_list`, validate that the supplied `block_locator_hashes` are monotonically decreasing in block number (or at minimum, reject the message if any hash appears out of order). Alternatively, sort the list by block number (descending) inside `insert_peer_unknown_header_list` before iterating, so the early-break invariant is always satisfied regardless of peer behavior. A simpler mitigation is to remove the early-break and always iterate the full list, accepting a small performance cost in exchange for correctness under adversarial input.

---

### Proof of Concept

1. Run a CKB node (`attacker`) that accepts inbound connections.
2. Wait for a victim node in IBD to establish an outbound connection to the attacker.
3. When the victim sends a `GetHeaders` request, the attacker responds with a crafted `GetHeaders` message (or triggers the victim to process one) where `block_locator_hashes = [genesis_hash, tip_hash_unknown_to_victim, ...]` — genesis first (known), high-number unknown hashes after.
4. `insert_peer_unknown_header_list` fires, finds `genesis_hash` known at index 0, sets `best_known_header` to genesis, breaks.
5. `unknown_header_list` for this peer remains empty (or contains only the genesis entry).
6. `BlockFetcher` finds no useful unknown hashes and no better `best_known_header`; it fetches zero blocks from this peer.
7. Repeat across all outbound slots to stall the victim's IBD entirely.
