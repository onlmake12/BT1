### Title
`DaoScriptSizeVerifier` Pairs DAO Inputs with Outputs by Positional Index, Allowing Lock-Size Check Bypass and Capacity Inflation — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`DaoScriptSizeVerifier::verify()` enforces that a DAO deposit cell (input) and its corresponding DAO withdrawing cell (output) use lock scripts of the same size. This is the **only** node-level guard against a known capacity-inflation attack on the NervosDAO. However, the verifier pairs inputs with outputs strictly by positional index via `.zip()`. An unprivileged transaction sender can craft a DAO phase-1 (deposit→prepare) transaction where the DAO deposit input and the DAO withdrawing output are at **different** positions, causing every pair in the zip to be silently skipped. Because DAO transactions are simultaneously exempt from the `OutputsSumOverflow` check in `CapacityVerifier`, the attacker can create a withdrawing cell with a larger lock script and inflated capacity, then redeem more CKB in phase 2 than was originally deposited.

---

### Finding Description

`DaoScriptSizeVerifier::verify()` iterates over `(resolved_inputs[i], outputs[i])` pairs:

```rust
for (i, (input_meta, cell_output)) in self
    .resolved_transaction
    .resolved_inputs
    .iter()
    .zip(self.resolved_transaction.transaction.outputs())   // ← positional zip
    .enumerate()
{
    if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
        && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
    {
        continue;   // ← silently skipped if either side is not DAO
    }
    ...
    if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
        return Err(...DaoLockSizeMismatch...);
    }
}
``` [1](#0-0) 

The guard only fires when **both** `resolved_inputs[i]` and `outputs[i]` carry the DAO type script. An attacker constructs a phase-1 transaction like:

| Index | Inputs | Outputs |
|-------|--------|---------|
| 0 | non-DAO cell | DAO withdrawing cell (large lock) |
| 1 | DAO deposit cell (small lock) | non-DAO cell |

The zip produces pairs `(non-DAO, DAO)` and `(DAO, non-DAO)`. Neither pair satisfies the "both must be DAO" guard, so both are `continue`d and the lock-size check is never reached.

Simultaneously, `CapacityVerifier::verify()` skips the `OutputsSumOverflow` check for any transaction that has a DAO input:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    ...
    if inputs_sum < outputs_sum {
        return Err(TransactionError::OutputsSumOverflow { ... });
    }
}
``` [2](#0-1) 

`valid_dao_withdraw_transaction()` returns `true` whenever any input uses the DAO type script, which is exactly the case in phase 1. [3](#0-2) 

The on-chain DAO type script does not enforce lock-script-size equality; the code comment explicitly acknowledges this: *"It provides a temporary solution till Nervos DAO script can be properly upgraded."* [4](#0-3) 

---

### Impact Explanation

**Capacity theft from the NervosDAO pool.**

Attack steps:
1. Deposit `D` CKB with a small lock script (e.g., 20-byte args, occupied capacity ≈ 61 bytes).
2. Submit a phase-1 (prepare) transaction with the DAO deposit input at index 1 and the DAO withdrawing output at index 0. The withdrawing output uses a large lock script (e.g., 100-byte args) and carries capacity `D + Δ` (where `Δ` is the extra capacity needed to satisfy `InsufficientCellCapacity` for the larger lock). Because the `OutputsSumOverflow` check is skipped for DAO transactions, the node accepts `outputs_sum > inputs_sum`.
3. In phase 2, `DaoCalculator::calculate_maximum_withdraw` computes the withdrawal based on the inflated `output_capacity` of the withdrawing cell, returning `D + Δ + interest`. The attacker redeems more CKB than was deposited; the surplus `Δ` is drawn from the DAO secondary-issuance pool. [5](#0-4) 

The `DaoScriptSizeVerifier` is also invoked during block verification in `contextual_block_verifier.rs`, so the same bypass applies when a miner includes such a transaction in a block. [6](#0-5) 

---

### Likelihood Explanation

**High.** The attack requires only:
- Constructing a standard CKB transaction with inputs and outputs in a specific order — a capability available to any unprivileged user via `send_transaction` RPC or P2P relay.
- No special keys, no majority hashpower, no social engineering.

The entry path is fully reachable by any tx-pool submitter or RPC caller. The `DaoScriptSizeVerifier` is the sole Rust-level guard; bypassing it requires only reordering transaction fields.

---

### Recommendation

Replace the positional `.zip()` with a scan that correctly pairs each DAO deposit input with the DAO withdrawing output that corresponds to it. The correct pairing is: for each input at index `i` that is a DAO deposit cell (data = all zeros), find the output at index `i` that is also a DAO cell. If no such output exists at the same index, the transaction should be rejected (a DAO deposit input must produce a DAO withdrawing output at the same index, as required by the DAO protocol). Alternatively, iterate over all DAO deposit inputs and independently search for a matching DAO output at the same index, rejecting the transaction if the output at that index is not a DAO cell or has a different lock script size.

---

### Proof of Concept

Construct a `ResolvedTransaction` with:

```
resolved_inputs[0] = non-DAO cell (any)
resolved_inputs[1] = DAO deposit cell, data = [0u8; 8], lock = Script { args: Bytes::new() }  // small lock
outputs[0]         = DAO withdrawing cell, lock = Script { args: Bytes::from(vec![1u8; 100]) } // large lock
outputs[1]         = non-DAO cell (any)
```

Pass this to `DaoScriptSizeVerifier::new(...).verify()`. The verifier returns `Ok(())` despite the lock-size mismatch, because:

- Pair `(inputs[0]=non-DAO, outputs[0]=DAO)` → `continue` (input is not DAO)
- Pair `(inputs[1]=DAO, outputs[1]=non-DAO)` → `continue` (output is not DAO)

The lock-size check at line 885 is never reached. [7](#0-6) 

The existing test suite confirms the verifier only catches mismatches when the DAO input and DAO output are at the **same** index (e.g., both at index 1 in `test_dao_disables_different_lock_script_size`), and has no test for the misaligned-position case. [8](#0-7)

    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
