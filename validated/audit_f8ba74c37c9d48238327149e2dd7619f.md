### Title
Missing `BLOCK_INVALID` Guard in `HeadersProcess::accept` Enables Repeated Re-processing of Invalid Blocks — (File: `sync/src/synchronizer/headers_process.rs`)

---

### Summary

The `HeadersProcess::accept` function does not return early when a block's status is `BLOCK_INVALID`. A malicious block/header relayer peer can repeatedly re-send headers for blocks already rejected by full verification, causing the node to overwrite the `BLOCK_INVALID` status with `HEADER_VALID` and re-enter the download-and-verify cycle indefinitely. This is compounded by a second partial-cleanup bug: when full verification fails, `remove_header_view` is never called, leaving a stale `header_map` entry that persists alongside the `BLOCK_INVALID` flag.

---

### Finding Description

**Root cause 1 — Missing guard in `HeadersProcess::accept`**

In `sync/src/synchronizer/headers_process.rs`, the `accept` function reads the block status and only short-circuits when the status already contains `HEADER_VALID`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    ...
    return result;
}
``` [1](#0-0) 

`BLOCK_INVALID` is `1 << 12` (bit 12 only). `HEADER_VALID` is `1` (bit 0 only). The bitflag `contains` check therefore evaluates to `false` for `BLOCK_INVALID`, so the early return is **never triggered** for an already-rejected block. The FIXME comment is a developer acknowledgment that this guard is missing. [2](#0-1) 

Execution then falls through to `prev_block_check`, `non_contextual_check`, `version_check`, and finally:

```rust
sync_shared.insert_valid_header(self.peer, self.header);
``` [3](#0-2) 

`insert_valid_header` is documented to update `block_status_map`:

```rust
/// Sync a new valid header, try insert to sync state
// Update the header_map
// Update the block_status_map
// Update the shared_best_header if need
// Update the peer's best_known_header
pub fn insert_valid_header(&self, peer: PeerIndex, header: &core::HeaderView) {
``` [4](#0-3) 

Because `insert_block_status` unconditionally overwrites any existing entry in the `DashMap`:

```rust
pub fn insert_block_status(&self, block_hash: Byte32, status: BlockStatus) {
    self.block_status_map.insert(block_hash, status);
}
``` [5](#0-4) 

…calling `insert_valid_header` for a `BLOCK_INVALID` block overwrites the rejection flag with `HEADER_VALID`, resetting the block into the sync pipeline.

---

**Root cause 2 — Partial cleanup on verification failure**

In `chain/src/verify.rs`, `consume_unverified_blocks` performs asymmetric cleanup:

```rust
Ok(_) => {
    self.shared.remove_block_status(&block_hash);   // ✓ removed
    self.shared.remove_header_view(&block_hash);    // ✓ removed
}
Err(err) => {
    ...
    self.shared.insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
    // remove_header_view is NEVER called here
}
``` [6](#0-5) 

On failure, the `header_map` retains a stale entry for the rejected block. `get_block_status` checks `block_status_map` first, so the stale `header_map` entry does not immediately cause a wrong status return — but it means the entry is never reclaimed unless the block happens to be an expired orphan cleaned up by `clean_expired_orphans`. [7](#0-6) 

The two bugs interact: root cause 1 allows `insert_valid_header` to overwrite `BLOCK_INVALID`; root cause 2 ensures the stale `header_map` entry is already present, so `get_block_status` will return `HEADER_VALID` (from `header_map`) once the `block_status_map` entry is overwritten.

---

### Impact Explanation

A malicious peer acting as a block/header relayer can:

1. Craft a block with a syntactically valid header but semantically invalid body (e.g., invalid script execution, bad capacity accounting).
2. The node downloads, stores, and fully verifies the block → `BLOCK_INVALID` is written to `block_status_map`; `header_map` retains a stale entry.
3. The peer re-sends the header via the `SendHeaders` message.
4. `HeadersProcess::accept` does not return early (FIXME guard absent); `insert_valid_header` overwrites `BLOCK_INVALID` with `HEADER_VALID`.
5. The block fetcher sees `HEADER_VALID` and re-requests the block from the peer.
6. Steps 2–5 repeat indefinitely.

The result is unbounded CPU (re-verification via CKB-VM), bandwidth (repeated block download), and disk I/O (repeated DB writes/deletes) consumed by a single malicious peer with no special privileges. The node's sync progress stalls while it loops on the attacker-controlled block.

---

### Likelihood Explanation

The attack requires only a standard P2P connection. Any unprivileged peer that can relay blocks and headers can trigger it. The FIXME comment confirms the developers are aware the guard is absent. No cryptographic material, majority hashpower, or operator access is needed.

---

### Recommendation

1. **Add the missing guard** in `HeadersProcess::accept` immediately after reading the block status:

   ```rust
   if status.contains(BlockStatus::BLOCK_INVALID) {
       return ValidationResult::with_error(ValidationError::InvalidBlock);
   }
   ``` [1](#0-0) 

2. **Call `remove_header_view`** in the failure branch of `consume_unverified_blocks`, mirroring the success branch:

   ```rust
   Err(err) => {
       ...
       self.shared.insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
       self.shared.remove_header_view(&block_hash); // add this
   }
   ``` [8](#0-7)

### Proof of Concept

```
Attacker peer                          CKB Node
     |                                     |
     |--- SendHeaders(H) ---------------->|  // H has valid header, invalid body
     |                                     |  // headers_process: insert_valid_header → HEADER_VALID
     |<-- GetBlocks(H) -------------------|  // block fetcher requests block
     |--- SendBlock(B) ------------------>|  // node stores B, runs CKB-VM
     |                                     |  // verify.rs: BLOCK_INVALID set; header_map NOT cleared
     |--- SendHeaders(H) [repeat] ------->|  // FIXME guard absent → insert_valid_header overwrites BLOCK_INVALID
     |<-- GetBlocks(H) [repeat] ---------|  // node re-requests block
     |--- SendBlock(B) [repeat] -------->|  // node re-verifies, fails again
     |         ... loop forever ...       |
```
