### Title
Stale `proposedAdmin` Not Cleared on Ownership Transfer Allows Unauthorized Admin Takeover — (`target_chains/ethereum/contracts/contracts/entropy/EntropyGovernance.sol`)

---

### Summary

`EntropyUpgradable` and `SchedulerUpgradeable` both implement a two-step admin transfer via `proposeAdmin`/`acceptAdmin`. The pending `proposedAdmin` value is stored in contract state but is never cleared when contract ownership is transferred via OZ `Ownable2StepUpgradeable`. This allows a previously proposed (but not yet accepted) admin address to call `acceptAdmin()` and seize the admin role after a new owner takes over — without the new owner's knowledge or consent.

---

### Finding Description

Both `EntropyGovernance.sol` and `SchedulerGovernance.sol` implement a two-step admin handoff:

- `proposeAdmin(address newAdmin)` — callable by the current **owner or admin** — writes `_state.proposedAdmin = newAdmin`.
- `acceptAdmin()` — callable by anyone whose address matches `_state.proposedAdmin` — promotes that address to admin and resets `_state.proposedAdmin` to `address(0)`. [1](#0-0) [2](#0-1) 

The `proposedAdmin` field lives in `EntropyInternalStructs.State`: [3](#0-2) 

`EntropyUpgradable` inherits `Ownable2StepUpgradeable` and never overrides `_transferOwnership` to clear `_state.proposedAdmin`: [4](#0-3) 

The same pattern exists identically in `SchedulerGovernance.sol` / `SchedulerUpgradeable.sol`: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

An address that was proposed as admin by the **previous** owner or admin can call `acceptAdmin()` at any time after a new owner takes over, gaining full admin privileges without the new owner's approval. Admin privileges include:

- Setting the Pyth protocol fee (`setPythFee`) [7](#0-6) 
- Setting the default randomness provider (`setDefaultProvider`) [8](#0-7) 
- Withdrawing accrued Pyth fees (`withdrawFee`) [9](#0-8) 
- For Scheduler: setting keeper fees and minimum balance per feed [10](#0-9) 

This constitutes unauthorized privilege escalation over live production contracts.

---

### Likelihood Explanation

The scenario is realistic in normal governance operations:

1. The current admin (or owner) calls `proposeAdmin(X)` to initiate an admin rotation.
2. Before `X` calls `acceptAdmin()`, ownership of the contract is transferred to a new governance contract or multisig (e.g., via the documented `entropy-accept-admin-and-ownership` script). [11](#0-10) 
3. The new owner is unaware of the pending `proposedAdmin`.
4. `X` (or a malicious address previously proposed by a compromised admin) calls `acceptAdmin()` permissionlessly and becomes admin.

The `proposeAdmin` function is callable by the **admin** (not just the owner), meaning a compromised or malicious admin can plant a `proposedAdmin` entry just before ownership is transferred, without the owner noticing. [12](#0-11) 

---

### Recommendation

Override `_transferOwnership` in both `EntropyUpgradable` and `SchedulerUpgradeable` to clear the pending `proposedAdmin` whenever ownership changes:

```solidity
function _transferOwnership(address newOwner) internal override {
    _state.proposedAdmin = address(0);
    super._transferOwnership(newOwner);
}
```

This mirrors the fix recommended in the referenced OwnableSmartWallet report: invalidate all pending approvals/proposals when the controlling party changes.

---

### Proof of Concept

```
// EntropyUpgradable scenario
1. Owner A (or Admin A) calls proposeAdmin(attacker)
   → _state.proposedAdmin = attacker

2. Owner A calls transferOwnership(ownerB)
   ownerB calls acceptOwnership()
   → owner() == ownerB, but _state.proposedAdmin still == attacker

3. attacker calls acceptAdmin()
   → passes: msg.sender == _state.proposedAdmin
   → _state.admin = attacker
   → attacker now controls setPythFee, setDefaultProvider, withdrawFee
   → ownerB never approved this
```

The same sequence applies to `SchedulerUpgradeable` with `SchedulerGovernance`. [2](#0-1) [13](#0-12)
