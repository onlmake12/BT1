### Title
Predictable Subscription ID Assignment Enables Reorg-Based Fund Theft in Scheduler — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract assigns subscription IDs using a monotonically incrementing storage counter (`_state.subscriptionNumber`). During a chain reorganization, an attacker can frontrun a victim's `createSubscription` call to claim the same subscription ID. When the victim's `addFunds` call — submitted with the original subscription ID from the pre-reorg block — is replayed, it deposits funds into the attacker-controlled subscription. The attacker then drains those funds via `withdrawFunds`.

---

### Finding Description

**Root cause — predictable ID assignment:**

`createSubscription` assigns IDs sequentially from a shared counter: [1](#0-0) 

```solidity
subscriptionId = _state.subscriptionNumber++;
...
_state.subscriptionManager[subscriptionId] = msg.sender;
```

The next subscription ID is fully predictable by any observer of the chain state.

**Root cause — `addFunds` has no caller restriction:**

`addFunds` accepts ETH for any active subscription without verifying the caller is the subscription manager: [2](#0-1) 

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    ...
    if (!params.isActive) { revert SchedulerErrors.InactiveSubscription(); }
    ...
    status.balanceInWei += msg.value;
}
```

**Root cause — `withdrawFunds` is manager-only:**

Only the subscription manager can withdraw: [3](#0-2) 

```solidity
function withdrawFunds(
    uint256 subscriptionId,
    uint256 amount
) external override onlyManager(subscriptionId) {
```

**Attack path (reorg scenario):**

1. Victim calls `createSubscription` in block B, receiving subscription ID N. Victim observes the ID from the receipt and submits `addFunds(N)` to the mempool.
2. A chain reorganization invalidates block B. Both the victim's `createSubscription` and `addFunds(N)` transactions are back in the mempool.
3. Attacker frontruns the victim's `createSubscription` in the new block, consuming counter value N and becoming the manager of subscription N.
4. Victim's `createSubscription` is included next, receiving ID N+1 instead.
5. Victim's `addFunds(N)` — still in the mempool with the original ID — is included and deposits ETH into the attacker's subscription N.
6. Attacker calls `withdrawFunds(N, amount)` and drains the victim's deposited funds.

The attacker must create a valid, active subscription (satisfying `minimumBalance`) to pass the `isActive` check in `addFunds`. This is a minor cost that the attacker recovers when withdrawing.

---

### Impact Explanation

Direct theft of user ETH. The victim's `addFunds` payment is irrecoverably credited to the attacker's subscription, and the attacker withdraws it in full. The victim's actual subscription (ID N+1) receives no funds. Impact is **high**.

---

### Likelihood Explanation

Requires a chain reorganization deep enough to include both the victim's `createSubscription` and `addFunds` transactions. Reorgs are rare on Ethereum mainnet post-merge but occur with meaningful frequency on L2s and alt-L1s (Polygon has experienced 100+ block reorgs). The victim must also submit `addFunds` as a separate transaction after `createSubscription` — a normal usage pattern for subscription top-ups. Likelihood is **low**, yielding overall **medium** severity.

---

### Recommendation

1. **Bind subscription IDs to the creator.** Derive the subscription ID as `keccak256(abi.encode(msg.sender, _state.subscriptionNumber++))` so an attacker cannot claim the same ID by frontrunning.
2. **Alternatively, restrict `addFunds` to the manager.** Add `onlyManager(subscriptionId)` to `addFunds`. This prevents third-party top-ups but eliminates the theft vector entirely.
3. **Combine both** for defense in depth.

---

### Proof of Concept

```
State before reorg:
  _state.subscriptionNumber = N

Block B (original, later reorged):
  Tx1: victim.createSubscription{value: minBalance}(params)
       → subscriptionId = N, manager[N] = victim

Mempool after Tx1 confirmed:
  Tx2: victim.addFunds{value: X}(N)

Reorg occurs — block B is invalidated.

New block B':
  Tx0 (attacker): attacker.createSubscription{value: minBalance}(params)
       → subscriptionId = N, manager[N] = attacker   ← attacker claims N
  Tx1 (victim):   victim.createSubscription{value: minBalance}(params)
       → subscriptionId = N+1, manager[N+1] = victim  ← victim gets N+1
  Tx2 (victim):   victim.addFunds{value: X}(N)
       → status[N].balanceInWei += X                  ← funds go to attacker's sub

Post-reorg:
  attacker.withdrawFunds(N, X)  → attacker receives X ETH
``` [4](#0-3) [2](#0-1) [5](#0-4)

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
