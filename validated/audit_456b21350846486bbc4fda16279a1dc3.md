### Title
Out-of-Bounds Panic in `remove_unconfirmed_tx` Causes Fee-Estimator / Tx-Pool DoS - (File: `util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

`TxConfirmStat::remove_unconfirmed_tx` in the `ConfirmationFraction` fee-estimator algorithm accesses `confirm_blocks_to_failed_txs[tx_age - 1]` without bounding `tx_age` to the array's length (`MAX_CONFIRM_BLOCKS = 1000`). When a tracked transaction stays in the pool for more than 1 000 blocks and is then evicted, `tx_age - 1 ≥ 1000` produces an out-of-bounds index, causing a Rust panic in the tx-pool service thread.

---

### Finding Description

`TxConfirmStat` maintains a fixed-size circular buffer `confirm_blocks_to_failed_txs` of length `MAX_CONFIRM_BLOCKS = 1000`. [1](#0-0) [2](#0-1) 

In `remove_unconfirmed_tx`, the guard on line 208 only protects the counter-decrement branch; it does **not** bound the subsequent array access on line 215:

```rust
// line 204-216
let tx_age = tip_height.saturating_sub(entry_height) as usize;
if tx_age < 1 { return; }
if tx_age >= self.block_unconfirmed_txs.len() {          // guards -= 1 only
    self.bucket_stats[bucket_index].old_unconfirmed_txs -= 1;
} else {
    let block_index = (entry_height % self.block_unconfirmed_txs.len() as u64) as usize;
    self.block_unconfirmed_txs[block_index][bucket_index] -= 1;
}
if count_failure {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64; // ← OOB
}
``` [3](#0-2) 

`self.block_unconfirmed_txs.len() == MAX_CONFIRM_BLOCKS == 1000`. When `tx_age = 1001`, `tx_age - 1 = 1000`, which is past the last valid index (999) of `confirm_blocks_to_failed_txs`. Rust panics with an index-out-of-bounds in debug **and** release builds.

`count_failure = true` is set when a tx is **dropped** (evicted) rather than confirmed, via `drop_tx` → `drop_tx_inner(hash, true)`: [4](#0-3) [5](#0-4) 

The public entry point is `Algorithm::reject_tx`, exposed through `FeeEstimator::reject_tx` (active only for the `ConfirmationFraction` variant): [6](#0-5) 

---

### Impact Explanation

A panic in the tx-pool service thread crashes or hangs the fee-estimation subsystem. Because the fee estimator is held inside the tx-pool service (see `tx-pool/src/service.rs` and `tx-pool/src/process.rs`), a panic propagates to the tx-pool worker, disrupting transaction admission and fee-rate estimation for all subsequent callers. This is a **remote DoS** against any node running the `ConfirmationFraction` estimator.

---

### Likelihood Explanation

The trigger requires:
1. A transaction submitted to the pool (any unprivileged RPC caller via `send_transaction`).
2. The tx remains unconfirmed for more than 1 000 blocks (~2.2 hours on mainnet at 8 s/block).
3. The tx is then evicted (e.g., pool-size eviction when higher-fee transactions arrive, or explicit removal).

All three conditions are realistic in normal operation. A low-fee transaction submitted during a fee spike will naturally stay unconfirmed for hundreds of blocks and then be evicted when the pool fills. No special privilege is required.

---

### Recommendation

Add a bounds check before accessing `confirm_blocks_to_failed_txs`:

```rust
if count_failure {
    let capped_age = tx_age.min(self.confirm_blocks_to_failed_txs.len());
    self.confirm_blocks_to_failed_txs[capped_age - 1][bucket_index] += 1f64;
}
```

Additionally, the `old_unconfirmed_txs -= 1` on line 209 should use `saturating_sub(1)` to guard against a symmetric underflow if the circular-buffer slot was already zeroed by `move_track_window` before `tx_age` crossed the threshold. [7](#0-6) 

---

### Proof of Concept

1. Node is configured with the `ConfirmationFraction` fee estimator.
2. Attacker (or any user) submits a transaction with a fee rate just above the minimum at block height H. The estimator calls `add_unconfirmed_tx(H, fee_rate)`, recording `block_unconfirmed_txs[H % 1000][bucket] = 1`.
3. 1 001 blocks pass. `process_block` is called for each, advancing `best_height` to `H + 1001`. `move_track_window(H + 1000)` zeroes the slot and increments `old_unconfirmed_txs`.
4. A burst of higher-fee transactions fills the pool, triggering eviction of the low-fee tx. The tx-pool calls `FeeEstimator::reject_tx(&tx_hash)`.
5. Inside `drop_tx_inner(hash, true)` → `remove_unconfirmed_tx(H, H+1001, bucket, true)`:
   - `tx_age = 1001`
   - `tx_age >= 1000` → `old_unconfirmed_txs -= 1` (OK)
   - `count_failure = true` → `confirm_blocks_to_failed_txs[1000][bucket] += 1f64`
   - **Panic: index out of bounds: the len is 1000 but the index is 1000**
6. The tx-pool service thread panics, causing a DoS. [8](#0-7) [9](#0-8)

    fn remove_unconfirmed_tx(
        &mut self,
        entry_height: u64,
        tip_height: u64,
        bucket_index: usize,
        count_failure: bool,
    ) {
        let tx_age = tip_height.saturating_sub(entry_height) as usize;
        if tx_age < 1 {
            return;
        }
        if tx_age >= self.block_unconfirmed_txs.len() {
            self.bucket_stats[bucket_index].old_unconfirmed_txs -= 1;
        } else {
            let block_index = (entry_height % self.block_unconfirmed_txs.len() as u64) as usize;
            self.block_unconfirmed_txs[block_index][bucket_index] -= 1;
        }
        if count_failure {
            self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
        }
    }
