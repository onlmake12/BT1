Audit Report

## Title
Unbounded Synchronous VM Execution in `process_orphan_tx` BFS Loop — (`tx-pool/src/process.rs`)

## Summary

`process_orphan_tx` performs a BFS over all orphans resolved by a newly accepted transaction, calling `_process_tx` synchronously (via `block_in_place`) for every orphan whose `declared_cycle ≤ max_tx_verify_cycles`. There is no per-invocation work budget, orphan count cap, or cycle-budget limit on the loop. An attacker who pre-loads the orphan pool (hard-capped at 100) with transactions carrying scripts consuming exactly `declared_cycle` cycles, then submits the resolving parent, triggers up to 100 sequential full-VM executions inside a single `process_orphan_tx` call, stalling the tx-pool service for the entire duration.

## Finding Description

**Step 1 — Orphan admission:**
`add_orphan_tx` accepts any transaction whose inputs are unknown, stores the peer-supplied `declared_cycle` verbatim, and enforces only a pool-size cap of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100`. [1](#0-0) [2](#0-1) 

**Step 2 — BFS loop with no work budget:**
When the resolving parent is accepted, `after_process` calls `process_orphan_tx`. The loop pops each resolved orphan and branches on `orphan.cycle`:
- `cycle > max_tx_verify_cycles` → non-blocking enqueue to the async verify queue
- `cycle ≤ max_tx_verify_cycles` → **synchronous** `_process_tx(..., Some(orphan.cycle), None).await`

There is no counter, timer, or cycle-budget check limiting how many orphans are processed in one invocation. [3](#0-2) [4](#0-3) 

**Step 3 — `_process_tx` runs full CKB-VM synchronously:**
`_process_tx` is called with `command_rx = None`. Inside `verify_rtx`, the absence of both a cache hit and a `command_rx` takes the `else` branch, which wraps `ContextualTransactionVerifier::verify` in `block_in_place` — a fully synchronous, blocking CKB-VM execution. [5](#0-4) [6](#0-5) 

**Step 4 — `DeclaredWrongCycles` requires exact cycle match:**
After VM execution, `_process_tx` rejects the transaction if `declared != verified.cycles`. The attacker must craft scripts consuming exactly `declared_cycle` cycles — a known-feasible technique using a RISC-V loop with fixed per-iteration cost. [7](#0-6) 

**Step 5 — Reachable from P2P relay:**
The full path is `submit_remote_tx` → `after_process` → `process_orphan_tx`, reachable by any unprivileged relay peer. [8](#0-7) 

## Impact Explanation

With 100 orphans each carrying a script consuming `max_tx_verify_cycles − 1` cycles, a single parent submission triggers 100 sequential `block_in_place` VM executions inside one `process_orphan_tx` call. The tx-pool service is an actor that processes requests sequentially; while `block_in_place` does not block the entire tokio runtime, the tx-pool actor cannot process any new requests (new transactions, block assembly, RPC) until the loop completes. This is a **100× amplification** of the worst-case single-transaction verification cost, achievable by any unprivileged P2P relay peer. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation

The attack requires: (1) submitting 100 orphan transactions via standard P2P relay — no privilege needed; (2) crafting scripts with exact cycle counts — technically feasible, CKB test infrastructure already does this; (3) paying minimum fees for 100 transactions — a real but low cost barrier; (4) submitting the resolving parent — one additional relay message. The attack is repeatable and the cost per amplification window is bounded only by minimum transaction fees.

## Recommendation

- Add a per-invocation work budget to `process_orphan_tx` (e.g., a maximum number of orphans processed synchronously per call, or a total cycle budget across all orphans in one BFS pass).
- Alternatively, route **all** orphan resolution through the async verify queue regardless of `declared_cycle`, eliminating the synchronous fast-path entirely. The `cycle > max_tx_verify_cycles` branch already demonstrates the correct pattern (non-blocking enqueue); the `≤` branch should follow the same model. [9](#0-8) 

## Proof of Concept

```
1. Connect to a CKB node as a relay peer.
2. Craft 100 transactions T_1..T_100, each:
   - spending an output of a not-yet-submitted parent P
   - containing a RISC-V lock script that consumes exactly
     (max_tx_verify_cycles - 1) cycles
   - declaring cycle = max_tx_verify_cycles - 1
3. Send T_1..T_100 via RelayTransactions; each is admitted to
   the orphan pool (missing parent P).
4. Send P via RelayTransactions; P is accepted into the tx-pool.
5. after_process calls process_orphan_tx(&P).
6. The BFS loop finds all 100 orphans; each has
   cycle = max_tx_verify_cycles - 1 ≤ max_tx_verify_cycles,
   so _process_tx is called 100 times sequentially.
7. Each call runs block_in_place(ContextualTransactionVerifier::verify(
   max_tx_verify_cycles - 1, false)).
8. Assert: total wall-clock time ≈ 100 × single-tx-max-verify-time.
   The tx-pool service is unresponsive for this entire window.
```

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }

        let mut orphan_queue: VecDeque<TransactionView> = VecDeque::new();
        orphan_queue.push_back(tx.clone());

        while let Some(previous) = orphan_queue.pop_front() {
            let orphans = self.find_orphan_by_previous(&previous).await;
            for orphan in orphans.into_iter() {
                if orphan.cycle > self.tx_pool_config.max_tx_verify_cycles {
                    debug!(
                        "process_orphan {} added to verify queue; find previous from {}",
                        orphan.tx.hash(),
                        tx.hash(),
                    );
                    let orphan_id = orphan.tx.proposal_short_id();
                    match self
                        .enqueue_verify_queue(
                            orphan.tx.clone(),
                            false,
                            Some((orphan.cycle, orphan.peer)),
                        )
                        .await
                    {
                        Ok(_) => {
                            self.remove_orphan_tx(&orphan_id).await;
                        }
                        Err(reject) => {
                            warn!(
                                "process_orphan {} failed to enqueue verify queue: {}; keep orphan from {}",
                                orphan.tx.hash(),
                                reject,
                                tx.hash(),
                            );
                        }
                    }
