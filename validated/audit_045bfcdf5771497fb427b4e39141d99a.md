### Title
Provider Fee Front-Running Allows Capture of User's Excess ETH in Entropy Requests — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` function in `Entropy.sol` reads the provider's `feeInWei` from mutable on-chain state at execution time, and no `maxFee` parameter is accepted from the caller. Because excess `msg.value` is explicitly never refunded, a malicious provider can front-run a pending `requestV2` transaction — after observing it in the public mempool — by calling `setProviderFee()` to raise their fee to exactly `msg.value − pythFee`, capturing the entire ETH buffer the caller sent.

---

### Finding Description

Inside `requestHelper`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);   // reads live feeInWei
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

`getFeeV2` and `getProviderFee` both read `provider.feeInWei` from storage at the moment the transaction executes. [2](#0-1) 

A provider can update `feeInWei` at any time with no delay or cap:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
``` [3](#0-2) 

The same capability is available to a designated fee manager via `setProviderFeeAsFeeManager`. [4](#0-3) 

The `IEntropyV2` interface explicitly documents that excess value is never returned:

> "Further note that excess value is *not* refunded to the caller." [5](#0-4) 

None of the four `requestV2` overloads accept a `maxFee` parameter, so callers have no on-chain mechanism to bound what they are willing to pay. [6](#0-5) 

---

### Impact Explanation

**Attack arithmetic** (let `P` = original provider fee, `F` = Pyth protocol fee, `B` = caller's buffer):

| Step | Value |
|---|---|
| Caller sends | `msg.value = P + F + B` |
| Provider front-runs: `setProviderFee(P + B)` | — |
| `requiredFee` at execution | `P + B + F = msg.value` → check passes |
| `providerInfo.accruedFeesInWei +=` | `P + B` (provider gains `B`) |
| `accruedPythFeesInWei +=` | `F` (Pyth unchanged) |
| Caller refund | **0** |

The provider captures the entire buffer `B`. Integrating contracts that add a percentage buffer (e.g., 10–20 %) to guard against fee fluctuations are the primary victims. The loss per request is bounded by the buffer, not the caller's full balance, but the attack is repeatable across every request to that provider.

---

### Likelihood Explanation

1. **Mempool visibility**: On all public EVM chains the pending transaction is visible before inclusion; the provider can read `msg.value` and compute the exact fee to set.
2. **Realistic caller behaviour**: The official SDK documentation and example code (e.g., `EntropyTester.sol`) call `getFeeV2` immediately before `requestV2` in the same transaction, sending the exact fee. However, integrating contracts that separate fee estimation from submission — or that add a safety buffer — are common in production and are the vulnerable pattern. [7](#0-6) 

3. **No time-lock on fee changes**: `setProviderFee` takes effect immediately with no delay, making the front-run window as wide as the mempool propagation time.

---

### Recommendation

Add a `maxFee` parameter to the `requestV2` family and enforce it inside `requestHelper`:

```solidity
// Caller-supplied upper bound on acceptable fee
if (maxFee != 0 && requiredFee > maxFee) revert EntropyErrors.FeeTooHigh();
```

This mirrors the fix applied to MechMarketplace (`requesterMaxDeliveryRate >= mechMaxDeliveryRate`) and gives callers a trustless guarantee that the fee has not been raised between their off-chain estimation and on-chain execution.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

interface IEntropy {
    function getFeeV2(address provider, uint32 gasLimit) external view returns (uint128);
    function requestV2(address provider, bytes32 userRandom, uint32 gasLimit) external payable returns (uint64);
    function setProviderFee(uint128 newFeeInWei) external;
    function getPythFee() external view returns (uint128);
}

contract PoC {
    IEntropy entropy;
    address provider; // attacker-controlled provider

    constructor(address _entropy, address _provider) {
        entropy = IEntropy(_entropy);
        provider = _provider;
    }

    // Step 1: victim contract calls this, sending a 10% buffer
    function victimRequest() external payable {
        uint128 fee = entropy.getFeeV2(provider, 0);
        uint128 buffer = fee / 10;
        // victim sends fee + 10% buffer to be safe
        entropy.requestV2{value: fee + buffer}(provider, bytes32(0), 0);
    }

    // Step 2: provider front-runs victimRequest() by calling this first
    // Provider observes msg.value = fee + buffer in the mempool,
    // then sets feeInWei = (fee + buffer) - pythFee to consume the buffer.
    function providerFrontRun(uint128 newFee) external {
        // called by the provider account before victimRequest lands
        entropy.setProviderFee(newFee);
    }
    // Result: provider.accruedFeesInWei gains the full buffer; victim is not refunded.
}
```

The `setProviderFee` call is submitted with a higher gas price than the victim's `requestV2`, ensuring it is mined first. After both transactions confirm, `providerInfo.accruedFeesInWei` reflects the inflated fee and the victim's buffer ETH is permanently captured.

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }
