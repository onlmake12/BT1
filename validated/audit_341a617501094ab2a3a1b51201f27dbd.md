### Title
Missing HTTP Request Timeout in Miner RPC Client Enables Unbounded Task Accumulation and Resource Exhaustion — (File: miner/src/client.rs)

---

### Summary

The `Rpc` struct in `miner/src/client.rs` wraps all outbound JSON-RPC calls from the miner to the CKB node. It constructs an `HttpClient` with no timeout and spawns an unbounded number of Tokio tasks — one per in-flight request — with no per-task deadline. Separately, when the miner runs in notify/listen mode, the inbound HTTP handler (`handle`) collects the full request body with no timeout and no body-size cap, and the listener loop imposes no connection limit. Both paths allow an attacker-reachable slow peer to hold miner resources open indefinitely.

---

### Finding Description

**Root cause 1 — outbound `Rpc` client, no timeout**

`Rpc::new` (lines 60–107) builds the HTTP client with no timeout:

```rust
let client = HttpClient::builder(TokioExecutor::new())
    .build::<_, Full<Bytes>>(https);   // no connect/read/write timeout
```

For every queued `RpcRequest` it immediately spawns a detached task:

```rust
loop_handle.spawn(async move {
    let request = match client
        .request(req)
        .await          // ← hangs forever if the server stalls
        ...
```

The channel feeding this loop is sized at 65 535 (`mpsc::channel(65_535)`). If the upstream RPC endpoint stalls, up to 65 535 tasks can accumulate simultaneously, each holding a live TCP connection and associated allocations, with no eviction path.

`ClientConfig` (util/app-config/src/configs/miner.rs lines 17–30) has no timeout field, so there is no operator-level escape valve.

**Root cause 2 — inbound notify listener, no timeout and no connection cap**

When `listen` is set, `listen_block_template_notify` (lines 234–271) accepts connections in a bare loop and spawns one task per connection with no limit:

```rust
tokio::spawn(async move {
    if let Err(err) = conn.await { ... }   // no timeout
});
```

The `handle` function (lines 358–369) collects the full body before doing anything:

```rust
let body = BodyExt::collect(req).await?.aggregate();  // no timeout, no size cap
```

An attacker who can reach the listen socket can open many connections, send HTTP headers but trickle or withhold the body, and hold each task open indefinitely — a classic slow-loris / slow-body pattern.

---

### Impact Explanation

- **Outbound path**: A CKB node whose RPC endpoint is made artificially slow (e.g., by flooding it with expensive RPC calls or exploiting a node-side stall bug) causes the miner to accumulate up to 65 535 hung Tokio tasks. Memory grows proportionally; the miner process eventually OOMs or becomes unresponsive, stopping block submission entirely. Mining revenue is lost for the duration.
- **Inbound path (listen mode)**: An attacker reachable to the listen socket (which may be bound to a non-loopback address) can exhaust the miner's file-descriptor and memory budget with O(connections) cost, achieving the same DoS with no per-connection work on the attacker side.

---

### Likelihood Explanation

- **Outbound**: The CKB node RPC is typically localhost, so a remote attacker must first degrade the node (e.g., via a tx-pool spam campaign that saturates the node's RPC worker threads). This is a two-step attack but both steps are reachable without privilege.
- **Inbound (listen mode)**: Requires the operator to have set `listen = "<addr>"` in `ckb-miner.toml`. The feature is documented and supported; if the address is non-loopback, any network peer can exploit it directly with zero privilege.

Overall likelihood: **Low–Medium** (outbound requires node degradation first; inbound is direct but gated on a non-default config).

---

### Recommendation

1. Wrap every `client.request(req).await` call with `tokio::time::timeout(DEADLINE, ...)` in `Rpc::new`.
2. Add a `rpc_timeout_ms` field to `ClientConfig` (util/app-config/src/configs/miner.rs) so operators can tune it.
3. In `listen_block_template_notify`, wrap `conn.await` with a per-connection timeout and enforce a maximum concurrent-connection count (e.g., via a semaphore).
4. In `handle`, wrap `BodyExt::collect(req).await` with a timeout and enforce a maximum body size before collecting.

---

### Proof of Concept

**Outbound (no timeout):**
1. Point `rpc_url` at a TCP server that accepts connections but never sends a response.
2. Start `ckb miner`. The miner calls `get_block_template` on every `poll_interval` tick.
3. Each call spawns a new Tokio task that blocks on `client.request(req).await` forever.
4. After `65_535 / poll_interval_seconds` seconds the channel fills; the miner's dispatch loop stalls; no further blocks are submitted.

**Inbound (listen mode, no timeout, no connection cap):**
1. Configure `listen = "0.0.0.0:8888"` in `ckb-miner.toml`.
2. From a remote host, open thousands of TCP connections to port 8888, send a valid HTTP `POST /` with `Transfer-Encoding: chunked` but never send the terminal chunk.
3. Each connection holds one Tokio task in `BodyExt::collect(req).await` indefinitely.

    async fn listen_block_template_notify(&self, addr: SocketAddr) {
        let listener = TcpListener::bind(addr).await.unwrap();
        let server = auto::Builder::new(TokioExecutor::new());
        let graceful = GracefulShutdown::new();
        let stop_rx: CancellationToken = new_tokio_exit_rx();

        loop {
            let client = self.clone();
            let handle = service_fn(move |req| handle(client.clone(), req));
            tokio::select! {
                conn = listener.accept() => {
                    let (stream, _) = match conn {
                        Ok(conn) => conn,
                        Err(e) => {
                            info!("accept error: {}", e);
                            tokio::time::sleep(std::time::Duration::from_secs(1)).await;
                            continue;
                        }
                    };
                    let stream = hyper_util::rt::TokioIo::new(Box::pin(stream));
                    let conn = server.serve_connection_with_upgrades(stream, handle);

                    let conn = graceful.watch(conn.into_owned());
                    tokio::spawn(async move {
                        if let Err(err) = conn.await {
                            info!("connection error: {}", err);
                        }
                    });
                },
                _ = stop_rx.cancelled() => {
                    info!("Miner client received exit signal. Exit now");
                    break;
                }
            }
        }
        drop(listener);
        graceful.shutdown().await;
