### Title
Attacker-Controlled `listen_port` in GetNodes Poisons Peer Store and Propagates to Up to 3 Peers Per Announce Cycle — (`network/src/protocols/discovery/state.rs`, `network/src/protocols/discovery/mod.rs`)

---

### Summary

An unprivileged inbound peer can send a single `GetNodes` message with an arbitrary `listen_port` value. The receiving node unconditionally replaces the TCP port in the stored address with the attacker-supplied value, transitions the address to `RemoteAddress::Listen`, adds it to the peer store, and subsequently broadcasts it to up to 3 other peers per announce cycle. There is no validation that the supplied port is the peer's actual listen port.

---

### Finding Description

**Inbound session initialization** (`state.rs` line 81):

When a peer connects inbound, `remote_addr` is set to `RemoteAddress::Init(context.session.address.clone())` — the actual ephemeral source address of the TCP connection. [1](#0-0) 

**GetNodes handler** (`mod.rs` lines 124–139):

When the attacker sends `GetNodes { version: 1, listen_port: Some(P) }`:

1. `update_port(P)` is called unconditionally on the `Init` address. It replaces the TCP port component with the attacker-supplied `P` and immediately transitions the state to `RemoteAddress::Listen(crafted_addr)`. [2](#0-1) 

2. The crafted address is inserted into `addr_known` and added to the peer store via `add_new_addr`. [3](#0-2) 

3. The subsequent `change_to_listen()` call (guarded by `version >= REUSE_PORT_VERSION`) is a **no-op** because `update_port` already transitioned the state to `Listen`. [4](#0-3) [5](#0-4) 

**Announce cycle** (`mod.rs` lines 224–262):

`notify()` calls `check_timer()`, which returns the `Listen` address. The only filter is `is_valid_addr()`, which checks only IP reachability — not port validity. The crafted address is then queued for broadcast to up to 3 randomly selected peers. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

- The attacker's IP is used as-is (from the actual TCP connection), but the port is fully attacker-controlled.
- The victim node stores `attacker_ip:crafted_port` in its peer store and broadcasts it to up to 3 peers per announce cycle (every 24 hours, `ANNOUNCE_INTERVAL`).
- Receiving peers store the crafted address and may re-broadcast it further, causing network-wide peer store pollution.
- Nodes attempting to connect to `attacker_ip:crafted_port` will fail (if the port is wrong) or be redirected to an attacker-controlled service (if the port is intentional).
- The `received_get_nodes` flag prevents replay within a session, but a new connection resets the state. [8](#0-7) 

---

### Likelihood Explanation

- Requires only an inbound TCP connection to any CKB node — no authentication, no PoW, no privileged role.
- A single well-formed `GetNodes` message (valid molecule encoding) is sufficient.
- The `DuplicateGetNodes` misbehavior check only prevents a second message on the same session; a new connection resets the state entirely.
- The attack is repeatable at low cost.

---

### Recommendation

Validate the `listen_port` against the actual source IP of the session before accepting it. At minimum, verify that the port is non-zero and within the valid range (1–65535). Stronger: cross-check the reported listen port against the identify protocol's advertised addresses, or require that the port match the source port of an outbound connection attempt from the peer. Do not allow an inbound peer to unilaterally promote its `Init` address to `Listen` with an arbitrary port.
