### Title
`PythLazer` Proxy Initialization Requires Multi-Step Ownership Transfer, Leaving Deployer EOA in Control of Trusted Signer Set - (File: `lazer/contracts/evm/script/PythLazerDeploy.s.sol`)

---

### Summary

`PythLazer` is deployed with a hardcoded deployer EOA (`0x78357316239040e19fC823372cC179ca75e64b81`) as the initial proxy owner. A separate, manually-run script (`PythLazerChangeOwnership.s.sol`) is required to transfer ownership to the intended governance executor. During the window between deployment and ownership transfer — or permanently if the transfer is never executed — the deployer EOA holds full `onlyOwner` control over the Lazer oracle's trusted signer set and upgrade path. Additionally, `PythLazer` uses single-step `OwnableUpgradeable` (not `Ownable2StepUpgradeable`), so a mistaken `transferOwnership` call permanently and irrecoverably transfers control to the wrong address.

---

### Finding Description

In `PythLazerDeploy.s.sol`, the `deployProxy` function explicitly sets the initial owner to the deployer EOA with the comment "for the time being":

```solidity
// Set the top authority to the deployer for the time being
address topAuthority = deployer;
``` [1](#0-0) 

The proxy is then initialized with this deployer as owner:

```solidity
abi.encodeWithSignature("initialize(address)", topAuthority)
``` [2](#0-1) 

Finalization of initialization requires running a completely separate script, `PythLazerChangeOwnership.s.sol`, which calls `lazer.transferOwnership(NEW_OWNER)` to hand control to the governance executor: [3](#0-2) 

`PythLazer` inherits `OwnableUpgradeable` (single-step), not `Ownable2StepUpgradeable`. This means `transferOwnership` takes effect immediately with no confirmation step: [4](#0-3) 

The `onlyOwner` modifier gates both the trusted signer management and the upgrade path: [5](#0-4) 

---

### Impact Explanation

During the window between `PythLazerDeploy.s.sol` and `PythLazerChangeOwnership.s.sol` execution — or permanently if the second script is never run — the deployer EOA can:

1. Call `updateTrustedSigner(maliciousSigner, farFutureExpiry)` to register an attacker-controlled signer.
2. Submit fake price updates via `verifyUpdate()` signed by that malicious signer.
3. Any on-chain protocol consuming Lazer price feeds accepts attacker-controlled prices.
4. Call `_authorizeUpgrade` to replace the proxy implementation entirely.

Additionally, if `PythLazerChangeOwnership.s.sol` is run with an incorrect `NEW_OWNER` (e.g., a typo, wrong chain address, or undeployed contract), ownership is permanently and irrecoverably transferred to the wrong address — there is no two-step confirmation to catch the mistake. [6](#0-5) 

---

### Likelihood Explanation

- The deployer address is hardcoded and publicly visible in the script (`0x78357316239040e19fC823372cC179ca75e64b81`), making it a known target.
- Every new chain deployment creates a fresh window of exposure.
- Operational failure (forgetting to run the second script, running it with a wrong address) is a realistic scenario given the manual, multi-step process.
- The `PythLazerChangeOwnership.s.sol` script loads `NEW_OWNER` from an environment variable with no on-chain validation that it is a deployed executor contract. [7](#0-6) 

---

### Recommendation

Apply the same atomic initialization pattern used by other Pyth upgradeable contracts. The intended final owner (the executor contract address) should be passed directly into `deployProxy` and set as the owner inside the same `deployCreate3` call, eliminating the post-deployment ownership transfer step entirely:

```solidity
// Pass the executor address directly — no separate ownership transfer script needed
address topAuthority = executorAddress; // final owner, not deployer
```

Additionally, upgrade `PythLazer` to inherit `Ownable2StepUpgradeable` instead of `OwnableUpgradeable`, so any future ownership transfer requires explicit acceptance by the new owner, preventing irrecoverable misdirection. [8](#0-7) 

---

### Proof of Concept

1. Deploy `PythLazer` proxy via `PythLazerDeploy.s.sol` on a new chain. Proxy owner is now `0x78357316239040e19fC823372cC179ca75e64b81`.
2. Operator does not immediately run `PythLazerChangeOwnership.s.sol` (or the deployer key is compromised during the window).
3. Attacker (or compromised deployer) calls:
   ```solidity
   PythLazer(proxy).updateTrustedSigner(attackerAddress, type(uint256).max);
   ```
4. Attacker signs a crafted price payload with `attackerAddress`'s private key.
5. Any caller invokes `verifyUpdate(craftedUpdate)` — `isValidSigner(attackerAddress)` returns `true`, the call succeeds, and the attacker-controlled payload is returned as a valid Lazer price update.
6. Downstream protocols consuming the Lazer feed accept the manipulated price. [9](#0-8) [10](#0-9)
