## Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Computed Values After In-Flight Evictions ‚Äî (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`add_entry` snapshots `(self.total_tx_size + entry.size, self.total_tx_cycles + entry.cycles)` into local variables before calling `check_and_record_ancestors`, which can evict pool entries and directly decrement `self.total_tx_size` via `update_stat_for_remove_tx`. Lines 218‚Äì219 then unconditionally overwrite the live, already-decremented fields with the stale pre-eviction snapshot, permanently overcounting `total_tx_size` by the sum of evicted transaction sizes. This causes `limit_size` to evict valid transactions with `Reject::Full` even when the pool has physical capacity.

## Finding Description

**Root cause ‚Äî `add_entry` (lines 200‚Äì221):**

`updated_stat_for_add_tx` is declared `&self` (immutable borrow, lines 711‚Äì728). It computes `self.total_tx_size + tx_size` and `self.total_tx_cycles + cycles` and returns them as a tuple without touching the struct fields. The returned values are bound to local variables `total_tx_size` / `total_tx_cycles` at lines 210‚Äì211.

`check_and_record_ancestors` (line 213) is then called. When `ancestors_count > self.max_ancestors_count` and `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count` (line 603), it iterates `evict_candidates` and calls `self.remove_entry_and_descendants(next_id)` (line 618). `remove_entry_and_descendants` (lines 252‚Äì264) calls `self.remove_entry(id)` for each removed entry. `remove_entry` (lines 235‚Äì250) calls `self.update_stat_for_remove_tx(entry.inner.size, entry.inner.cycles)` at line 247, which **directly writes** the decremented values back to `self.total_tx_size` and `self.total_tx_cycles` (lines 738‚Äì740).

After all evictions complete, lines 218‚Äì219 execute:
```rust

self.total_tx_cycles = total_tx_cycles; // stale: original + entry.cycles
```
This overwrites the correctly-decremented live fields with the pre-eviction snapshot, producing:
```

correct  total_tx_size = original ‚àí evicted_sizes + entry.size
overcount              = evicted_sizes
```

**Trigger path ‚Äî `get_tx_ancenstors` (lines 517‚Äì554):**

`cell_ref_parents` is populated at lines 531‚Äì533 when any input's `previous_output` matches a key in `self.edges.deps` ‚Äî i.e., when existing pool transactions declared that same outpoint as a `cell_dep`. This is the precise condition that activates the eviction branch in `check_and_record_ancestors`.

**Downstream effect ‚Äî `limit_size` (pool.rs line 298):**

```rust

```
With an inflated `total_tx_size`, this loop evicts additional valid entries and issues `Reject::Full` for transactions that should have been accepted. The overcount persists across subsequent calls because no code path resets `total_tx_size` to a recomputed ground truth under normal operation.

## Impact Explanation

Valid, fee-paying transactions are silently dropped from the pool with `Reject::Full` even when the pool has physical capacity. The `tx_pool_info` RPC returns an inflated `total_tx_size`, misleading node operators. The overcount is persistent and additive: each successful trigger increases the phantom size by `sum(evicted_sizes)`, meaning repeated invocations can accumulate enough phantom size to cause the pool to reject all incoming transactions despite being physically empty. This matches **Low (501‚Äì2000 points): Any other important performance improvements for CKB** ‚Äî it is a correctness defect in pool size-accounting that degrades transaction throughput without crashing the node or causing consensus deviation.

## Likelihood Explanation

The trigger requires: (a) existing pool transactions that declare a specific unspent output as a `cell_dep`, and (b) a new transaction whose input spends that same output, with `ancestors_count` exceeding `max_ancestors_count` while `cell_ref_parents` is large enough to bring it back within limits. This is reachable via an unprivileged `send_transaction` RPC call with no special privileges. The scenario requires a chain depth near `max_ancestors_count` but is fully within reach of any external actor. The attack is repeatable: each round costs only the fees for the evicted transactions, and the phantom overcount accumulates across rounds.

## Recommendation

Remove the pre-computation of `(total_tx_size, total_tx_cycles)` before `check_and_record_ancestors`. After all mutations are complete, update the live fields directly using the already-correct `self.total_tx_size` (which already reflects any evictions):

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// self.total_tx_size already reflects evictions; add only the new entry.
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .ok_or_else(|| Reject::Full(format!(
        "tx-pool total_tx_size {} overflows by add {}", self.total_tx_size, entry.size
    )))?;
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .ok_or_else(|| Reject::Full(format!(
        "tx-pool total_tx_cycles {} overflows by add {}", self.total_tx_cycles, entry.cycles
    )))?;
```

Optionally add a `debug_assert_eq!(self.total_tx_size, self.recompute_total_stat().unwrap().0)` post-insertion invariant check to catch future drift in tests.
