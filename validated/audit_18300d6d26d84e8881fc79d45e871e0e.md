### Title
`check_tx_fee` Minimum Fee Computed Using Only Serialized Size, Ignoring Cycle Weight — (`tx-pool/src/util.rs`)

### Summary
The tx-pool admission gate `check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, while the actual block-packing weight of a transaction is a function of **both** size and cycles (`get_transaction_weight(size, cycles)`). A transaction sender can craft a small-serialized-size but high-cycle transaction that passes the fee gate with a fee far below what its true block weight warrants, admitting it to the pool while miners will not include it, enabling a sustained tx-pool resource exhaustion attack.

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

`FeeRate::fee` performs integer division `fee_rate * weight / 1000`:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
``` [2](#0-1) 

The code itself acknowledges the mismatch with a comment:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [3](#0-2) 

However, once a transaction is admitted, its actual fee rate inside the pool is computed using `get_transaction_weight(size, cycles)`, which is always ≥ size alone:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [4](#0-3) 

The same weight formula is used for block assembly fee-rate ranking and eviction: [5](#0-4) 

The result is a structural gap: the admission threshold is calibrated to `size` only, but the actual cost a transaction imposes on the block is `f(size, cycles)`. A transaction with a tiny serialized body but cycles near `max_tx_verify_cycles` (default 70,000,000) passes the cheap size-only gate while its true weight-adjusted fee rate is far below `min_fee_rate`.

### Impact Explanation

An unprivileged tx-pool submitter can flood the mempool with transactions that:
1. Are small enough (e.g., ~250 bytes) to pass `check_tx_fee` with a fee of just 1 shannon above the size-based minimum.
2. Consume near-maximum cycles, making their actual weight-adjusted fee rate a tiny fraction of `min_fee_rate`.
3. Are never selected by miners (actual fee rate too low), so they persist until expiry (default 12 hours).

The tx-pool has a 180 MB cap (`max_tx_pool_size = 180_000_000`). An attacker can fill it with these underpriced-by-weight transactions at a cost proportional to `fee_rate * size` rather than `fee_rate * weight`, effectively paying far less than the protocol intends for the pool space consumed. Legitimate high-fee transactions may be evicted or delayed. [6](#0-5) [7](#0-6) 

### Likelihood Explanation

**Medium.** The attack requires no special privilege — any RPC caller or P2P relay peer can submit transactions. The attacker must pay a small fee (the size-based minimum) and must craft scripts that consume high cycles without large serialized size (e.g., a tight loop in a lock script). The `max_tx_verify_cycles` limit bounds the per-transaction cycle budget, but the ratio of cycles-to-size can be made very large. The 12-hour expiry and pool eviction provide partial mitigation but do not prevent sustained pressure.

### Recommendation

Replace the size-only minimum fee check with a weight-aware check. After cycle verification completes and the actual cycle count is known, recompute `min_fee` using `get_transaction_weight(tx_size, cycles)` instead of `tx_size` alone:

```rust
let weight = get_transaction_weight(tx_size as usize, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

This mirrors how `TxEntry::fee_rate()` and the block assembler already compute effective fee rates, closing the gap between admission policy and actual resource cost. [8](#0-7) [9](#0-8) 

### Proof of Concept

1. Craft a CKB transaction with a lock script that executes a tight computational loop consuming ~69,000,000 cycles (just under `max_tx_verify_cycles = 70,000,000`).
2. Keep the serialized transaction size small (~250 bytes — a minimal input/output structure).
3. With `min_fee_rate = 1000` shannons/KW:
   - Size-based `min_fee = 1000 * 250 / 1000 = 250` shannons.
   - True weight-based `min_fee = 1000 * get_transaction_weight(250, 69_000_000) / 1000` — orders of magnitude larger.
4. Submit the transaction with fee = 251 shannons via `send_transaction` RPC.
5. `check_tx_fee` passes (251 > 250). The transaction enters the pool.
6. Miners compute the actual fee rate using `get_transaction_weight` and find it far below `min_fee_rate`; the transaction is never mined.
7. Repeat until the 180 MB pool is saturated with expiry-pending, miner-rejected transactions. [10](#0-9) [2](#0-1) [11](#0-10)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
```
