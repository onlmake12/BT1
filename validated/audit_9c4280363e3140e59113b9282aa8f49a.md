### Title
Rate Limiter Counts Messages Not Hashes, Allowing Per-Peer Tx-Pool Lookup Amplification — (`sync/src/relayer/get_transactions_process.rs`)

### Summary

The `Relayer` rate limiter is keyed by `(PeerIndex, message_item_id)` and enforces 30 messages/second per peer per message type. However, each `GetRelayTransactions` message may contain up to `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) hashes. The rate limiter passes after counting one token per message regardless of hash count, so a single unprivileged peer can drive up to 30 × 32,767 ≈ **983,010 tx-pool lookups per second** against the node.

### Finding Description

**Rate limiter setup** — 30 req/s per `(peer, item_id)`: [1](#0-0) 

**Rate limiter key** — counts one token per message, not per hash: [2](#0-1) 

**Hash-count ceiling** — up to 32,767 hashes allowed per message: [3](#0-2) [4](#0-3) 

**Tx-pool lookup** — all hashes forwarded in a single batch call after the rate check passes: [5](#0-4) 

The `MAX_RELAY_TXS_BYTES_PER_BATCH` limit applies only to the **outgoing** response, not to the incoming hash list. [6](#0-5) 

### Impact Explanation

The tx-pool controller processes `fetch_txs_with_cycles` requests sequentially. Flooding it with 30 batches/s of 32,767 hashes each saturates the tx-pool service's internal queue, delaying or blocking legitimate transaction admission, proposal processing, and block template construction for all peers — not just the attacker. This is a resource exhaustion path with measurable, scoped impact on node liveness.

### Likelihood Explanation

Any unprivileged P2P peer can send `GetRelayTransactions` messages. No authentication, PoW, or stake is required. The attacker needs only a single TCP connection and the ability to craft valid relay messages at 30/s with 32,767 hashes each (~1 MB/message × 30/s = ~30 MB/s inbound, which is within reach of a well-connected adversary).

### Recommendation

Enforce a **per-peer hash budget** in addition to the per-message rate limit. Two complementary approaches:

1. **Hash-count token consumption**: Consume `tx_hashes.len()` tokens from the rate limiter instead of 1, so the 30/s cap applies to hashes, not messages.
2. **Hard cap on hashes per peer per second**: Track a rolling per-peer hash counter and reject or drop messages that exceed a threshold (e.g., 1,000 hashes/s).

Either change should be applied before `execute()` is dispatched, inside `try_process()`. [7](#0-6) 

### Proof of Concept

1. Connect a single peer to a CKB node.
2. Send 30 `GetRelayTransactions` messages per second, each containing 32,767 distinct (non-duplicate) tx hashes.
3. Monitor the tx-pool service queue depth and `fetch_txs_with_cycles` latency.
4. Assert that the tx-pool lookup rate reaches ~983K/s and that legitimate tx-pool operations (e.g., `send_transaction` RPC) experience measurable latency increase.

The rate limiter at `check_key(&(peer, message.item_id()))` will pass all 30 messages/s without throttling, and `execute()` will forward all 32,767 hashes per message to the tx pool. [8](#0-7)
