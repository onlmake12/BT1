### Title
Unbounded `numHashes` Loop in `constructProviderCommitment` Bricks Provider Reveals When `maxNumHashes == 0` — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When a provider's `maxNumHashes` is `0` (the default/unset state), an unprivileged attacker can submit enough `requestV2` calls to grow the stored `req.numHashes` on a target request to an arbitrarily large value. When the provider later calls `revealWithCallback`, the `while (numHashes > 0)` loop in `constructProviderCommitment` iterates that many times and exceeds the block gas limit, permanently preventing reveal of all affected in-flight requests.

---

### Finding Description

**The guard is disabled when `maxNumHashes == 0`**

In `requestHelper`, `req.numHashes` is computed as the distance from `currentCommitmentSequenceNumber` to the newly assigned sequence number: [1](#0-0) 

The cap check on line 252 is gated by `providerInfo.maxNumHashes != 0`. When `maxNumHashes` is `0` (the sentinel for "no limit"), the check is entirely skipped and `req.numHashes` is stored with no upper bound.

**`constructProviderCommitment` loops `numHashes` times unconditionally** [2](#0-1) 

Each iteration costs ~36–42 gas (one `keccak256` over 32 bytes). At a 30 M gas block limit, roughly 700 k–800 k iterations exhaust the block. The attacker only needs to submit that many fee-paid requests before any reveal occurs.

**`req.numHashes` is frozen at request creation time**

`revealHelper` uses the stored `req.numHashes` directly: [3](#0-2) 

Even if the provider reveals earlier requests first (advancing `currentCommitmentSequenceNumber`), the stored `numHashes` on later requests is not recalculated — it permanently reflects the gap at the time of creation. The provider cannot escape the large loop.

**`advanceProviderCommitment` offers no escape**

`advanceProviderCommitment` also calls `constructProviderCommitment` with the same large `numHashes`: [4](#0-3) 

So the provider cannot use this path to skip past the poisoned requests either.

**`maxNumHashes == 0` is the default deployment state**

The Fortuna off-chain keeper defaults to `0` when `max_num_hashes` is not configured: [5](#0-4) 

This means any provider that has not explicitly called `setMaxNumHashes` with a nonzero value is vulnerable.

---

### Impact Explanation

All in-flight `requestV2` requests whose stored `numHashes` exceeds the feasible per-block hash count become permanently unrevealable. The provider's `revealWithCallback` (and `advanceProviderCommitment`) will always OOG-revert for those sequence numbers. Fees paid by users for those requests are locked in the contract. The provider's entire callback pipeline is bricked for the affected range.

---

### Likelihood Explanation

The precondition (`maxNumHashes == 0`) is the **default** state for any provider that has not explicitly configured the field. The attacker only needs to pay the provider's per-request fee for each poisoning request — no privileged access, no key compromise, no governance majority. The cost scales linearly with the number of requests needed to reach the OOG threshold, but the attacker can choose any target request in the future queue and only needs to push `numHashes` high enough for that one request.

---

### Recommendation

Remove the `maxNumHashes != 0` bypass. Treat `maxNumHashes == 0` as "not yet configured" and either:

1. **Require providers to set a nonzero `maxNumHashes` before accepting requests**, or
2. **Apply a protocol-level hard cap** (e.g., `MAX_NUM_HASHES`) unconditionally in `requestHelper`, independent of the provider-configured value.

The check at lines 251–256 should become:

```solidity
uint32 hardCap = providerInfo.maxNumHashes != 0
    ? providerInfo.maxNumHashes
    : DEFAULT_MAX_NUM_HASHES; // e.g., 1000
if (req.numHashes > hardCap) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

---

### Proof of Concept

```
1. Provider registers: register(fee, commitment, meta, chainLength=2^32-1, uri)
   → providerInfo.maxNumHashes == 0 (default, never set)
   → providerInfo.defaultGasLimit != 0

2. Attacker loop (N = ~800_000 iterations):
   requestV2(provider, userContribution, gasLimit)
   → each call stores req.numHashes = i (growing by 1 each time)
   → no revert because maxNumHashes == 0 disables the cap

3. Provider calls:
   revealWithCallback(provider, seqN, userContribution, providerContribution)
   → revealHelper → constructProviderCommitment(numHashes=N, ...)
   → while(N > 0) { keccak256(...); N--; }  // ~800k iterations × ~37 gas ≈ 30M gas
   → OUT OF GAS revert

4. All requests with sequence numbers ≥ seqN are permanently unrevealable.
   Provider fees for those requests are locked.
```

        uint64 numHashes,
