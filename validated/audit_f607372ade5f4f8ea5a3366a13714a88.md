### Title
Cross-Chain Replay of Lazer Price Updates Due to Missing Domain Separator — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.sol`'s `verifyUpdate` function signs only `keccak256(payload)` with no chain ID, contract address, or any other domain-separating context. Because the Sui Lazer contract uses the identical signature scheme (`secp256k1_ecrecover` over `keccak256(payload)`), and because the payload format contains no chain-specific field, a malicious relayer can take a valid signed Lazer update from one deployment and replay it verbatim on any other deployment — EVM-to-EVM or EVM-to-Sui — and the contract will accept it as authentic.

---

### Finding Description

In `verifyUpdate`, the only data that is cryptographically bound to the signature is the raw payload bytes:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);          // ← no chain id, no address
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `EVM_FORMAT_MAGIC` (bytes 0–3 of `update`) is checked before the signature, but it is **not** included in the signed data — the signature covers only `payload`, which starts at byte 71.

The payload itself contains only:

| Field | Size |
|---|---|
| `PAYLOAD_FORMAT_MAGIC` | 4 bytes |
| `timestamp` | 8 bytes |
| `channel` | 1 byte |
| `feedsLen` | 1 byte |
| feed data | variable | [2](#0-1) 

There is no `chainId`, no contract address, and no network-specific prefix anywhere in the signed bytes.

The Sui Lazer contract uses the identical primitive:

```move
let pubkey = secp256k1_ecrecover(signature, payload, 0);  // 0 = keccak256
``` [3](#0-2) 

Both chains use secp256k1 ECDSA over `keccak256(payload)`. A signature produced by a Pyth Lazer trusted signer for one chain is therefore cryptographically valid on every other chain that shares the same signer key — which is the expected operational model for a centralized oracle.

The Sui contract does additionally check `PAYLOAD_MAGIC` inside the payload, but the EVM contract does not even perform that check, making it strictly more permissive. [4](#0-3) 

---

### Impact Explanation

A malicious relayer who observes a valid signed Lazer update on chain A can submit the identical bytes to the `verifyUpdate` call on chain B. The EVM contract will:

1. Accept the `EVM_FORMAT_MAGIC` check (same magic on all EVM deployments).
2. Recover the same signer address (same payload → same hash → same ECDSA recovery).
3. Pass `isValidSigner` (same trusted signer registered on both chains).
4. Return the payload to the consumer contract as verified.

The consumer contract receives price data whose `timestamp` is from chain A's update, not a fresh update for chain B. In time-sensitive protocols (liquidations, options settlement, perpetuals funding), even a few seconds of staleness introduced by a targeted replay can be exploited to front-run liquidations or avoid margin calls. Additionally, if the attacker replays a deliberately old (but still within the signer's validity window) update, they can pin the on-chain price to a value of their choosing for the duration of the replay window.

---

### Likelihood Explanation

- Pyth Lazer is a centralized, subscription-based service. The same trusted signer key is operationally expected to be registered on all supported chains (EVM mainnet, Arbitrum, Base, Optimism, Sui, etc.).
- Any subscriber who receives the raw signed update bytes from the Lazer WebSocket can act as a relayer and submit those bytes to any chain.
- No privileged access is required; the attacker only needs to be a normal Lazer subscriber and a transaction sender on the target chain.
- The `verifyUpdate` function is `external payable` with no access control beyond the fee check. [5](#0-4) 

---

### Recommendation

Include the EVM chain ID and the contract's own address in the signed digest, analogous to EIP-712 domain separation:

```solidity
bytes32 hash = keccak256(
    abi.encodePacked(block.chainid, address(this), payload)
);
```

This makes every signature chain- and contract-specific, so a replay on a different chain or a different contract instance will produce a different hash and fail signature recovery. The Lazer router must include these fields when constructing the message that the trusted signer signs.

---

### Proof of Concept

1. Subscribe to the Pyth Lazer WebSocket and receive a valid `leEcdsa` update for Ethereum mainnet. The raw bytes are `[EVM_FORMAT_MAGIC (4B)] [r (32B)] [s (32B)] [v (1B)] [payload_len (2B)] [payload ...]`.
2. Submit the identical byte string to `PythLazer.verifyUpdate{value: fee}(update)` on the Arbitrum deployment of `PythLazer`.
3. The function recovers the same signer address (because `keccak256(payload)` is chain-agnostic), passes `isValidSigner`, and returns `(payload, signer)` to the caller.
4. A consumer contract on Arbitrum that calls `verifyUpdate` and does not independently validate `block.chainid` against a value embedded in the payload will accept the Ethereum mainnet update as a valid Arbitrum price update, with the original Ethereum timestamp. [6](#0-5) [7](#0-6)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }

public(package) fun verify_le_ecdsa_message(
    state: &State,
    clock: &Clock,
    signature: &vector<u8>,
    payload: &vector<u8>,
) {
    let current_cap = state.current_cap();

    // 0 stands for keccak256 hash
    let pubkey = secp256k1_ecrecover(signature, payload, 0);

    // Check if the recovered pubkey is in the trusted signers list
    let trusted_signers = state.trusted_signers(&current_cap);
    let mut maybe_idx = trusted_signers.find_index!(|signer|
        signer.public_key() == &pubkey
    );

    assert!(maybe_idx.is_some(), ESignerNotTrusted);
    let idx = maybe_idx.extract();
    let expires_at_ms = trusted_signers[idx].expires_at_ms();
    assert!(clock.timestamp_ms() < expires_at_ms, ESignerExpired);
