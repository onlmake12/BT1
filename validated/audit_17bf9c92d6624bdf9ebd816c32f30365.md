### Title
`forward_rate_limiter` Bypass via Spoofed `from` PeerId in `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

### Summary
The `forward_rate_limiter` in the hole-punching protocol is keyed on `(from, to, item_id)`, where `from` is taken directly from the attacker-controlled message payload without any validation that it matches the actual sender's session peer ID. An attacker can send up to 30 `ConnectionRequest` messages per second (the outer `rate_limiter` cap), each with a distinct spoofed `from` PeerId but the same `to`, causing every message to occupy a unique bucket in the `forward_rate_limiter` and all 30 to pass through to `forward_message()` or `respond_delivered()`.

### Finding Description

**Two-layer rate limiting design:**

The outer `rate_limiter` is keyed by `(session_id, msg.item_id())` at 30 req/sec: [1](#0-0) 

The inner `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` at 1 req/sec: [2](#0-1) 

**The `from` field is never validated against the actual sender's session peer ID.** In `RequestContent::try_from`, `from` is only checked to be a syntactically valid `PeerId`: [3](#0-2) 

There is no check anywhere in `execute()` that `content.from` equals the peer ID of the actual connected session. The `peer: PeerIndex` field is available in the struct but is never used to cross-validate `content.from`: [4](#0-3) 

**Bypass path:**
- Attacker connects to a relay node (one P2P session).
- Attacker sends 30 `ConnectionRequest` messages/sec, each with a freshly generated random `from` PeerId and the same `to` PeerId.
- The outer `rate_limiter` passes all 30 (quota is exactly 30/sec per session).
- Each message hits `forward_rate_limiter.check_key(&(unique_from, same_to, 0))` — each is a brand-new key with a fresh bucket, so all 30 pass.
- All 30 reach `forward_message()` (or `respond_delivered()` if the relay is the target): [5](#0-4) 

### Impact Explanation

The `forward_rate_limiter` comment states the invariant explicitly:

> "the same group of from/to should not be received by the same node more than 1 times within one second" [6](#0-5) 

This invariant is fully bypassed. A single attacker peer can drive 30 forwarded `ConnectionRequest` messages per second toward any chosen `to` peer. When `forward_message` does not find the target directly, it broadcasts to `sqrt(total_peers)` additional nodes, creating a fan-out amplification: [7](#0-6) 

The target peer's own outer `rate_limiter` (30/sec per session) will absorb the direct relay case, but the broadcast fan-out path multiplies the load across the network proportional to `sqrt(N)` connected peers.

### Likelihood Explanation

- Requires only a standard P2P connection — no privileges, no keys, no PoW.
- The `from` field is a free-form `Bytes` field in the molecule schema with no protocol-level binding to the session: [8](#0-7) 
- Generating 30 distinct random `PeerId` values per second is trivially cheap.
- The attack is locally reproducible with a unit test as described in the question.

### Recommendation

Validate that `content.from` matches the actual session's peer ID before using it as a rate-limit key. The `peer: PeerIndex` is already available in `ConnectionRequestProcess`; the network state can resolve it to a `PeerId` for comparison. Alternatively, key the `forward_rate_limiter` on `(actual_session_peer_id, to, item_id)` instead of `(message.from, to, item_id)`, which removes the attacker's ability to vary the key.

### Proof of Concept

```
1. Connect one P2P session to a relay node.
2. In a loop at 30 iterations/sec:
   a. Generate a fresh random PeerId as `from`.
   b. Build a ConnectionRequest with from=<random>, to=<fixed_target>, max_hops=6, valid listen_addrs.
   c. Send it over the single session.
3. Assert: all 30 calls per second pass forward_rate_limiter and invoke forward_message().
   (The outer rate_limiter allows exactly 30/sec; the forward_rate_limiter sees 30 distinct keys.)
4. Observe: the target peer or intermediate relay nodes receive 30 forwarded messages/sec
   instead of the intended 1/sec cap.
```
