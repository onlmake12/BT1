### Title
Unrestricted `providerToCredit` in `Echo::executeCallback` Enables Fee Theft After Exclusivity Period — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol::executeCallback` has no access control after the exclusivity window expires, and the `providerToCredit` parameter is fully attacker-controlled. Any unprivileged actor can call the function after the exclusivity period and redirect the stored request fee to their own address, stealing the legitimate provider's earned fee.

---

### Finding Description

`executeCallback` is declared `external payable` with no caller restriction: [1](#0-0) 

During the exclusivity window the contract enforces `providerToCredit == req.provider`. Once that window closes, the check is skipped entirely and any address may be supplied: [2](#0-1) 

The fee credit is then applied unconditionally to the caller-supplied address: [3](#0-2) 

`req.fee` was set at request time as the user's payment minus the Pyth protocol fee: [4](#0-3) 

`registerProvider` is permissionless, so any EOA can become a registered provider: [5](#0-4) 

`withdrawAsFeeManager` allows a provider to drain their own `accruedFeesInWei` once a fee manager is set: [6](#0-5) 

---

### Impact Explanation

A fully unprivileged attacker can steal the provider fee from every pending Echo request after the exclusivity period expires. The legitimate provider fulfills their off-chain obligation (the callback is executed regardless of who calls `executeCallback`) but receives zero compensation. At scale this makes the Echo provider role economically unviable and breaks the incentive model of the protocol.

---

### Likelihood Explanation

- `registerProvider` is permissionless — no barrier to entry.
- Valid Pyth price update data is publicly available from the Pyth price service; the attacker only needs to supply it and cover the small Pyth fee (`msg.value = pythFee`).
- Pending requests are visible on-chain via `getFirstActiveRequests`.
- The attacker simply waits for `block.timestamp >= req.publishTime + exclusivityPeriodSeconds` and then front-runs or replaces the legitimate provider's `executeCallback` transaction.

---

### Recommendation

Restrict `providerToCredit` to `req.provider` unconditionally, or implement the penalty/redistribution mechanism already noted in the code's TODO comments so that only the original provider (or a designated replacement with a provable penalty deducted from the original provider) can be credited.

---

### Proof of Concept

```solidity
// Attacker setup (one-time)
echo.registerProvider(0, 0, 0);                  // permissionless
echo.setFeeManager(attackerAddress);             // set self as fee manager

// Per-request attack (after exclusivity period)
uint256 pythFee = pyth.getUpdateFee(updateData);
echo.executeCallback{value: pythFee}(
    attackerAddress,   // providerToCredit — attacker-controlled
    victimSequenceNumber,
    updateData,
    priceIds
);
// req.fee is now credited to attackerAddress

echo.withdrawAsFeeManager(attackerAddress, stolenAmount);
// Attacker receives the legitimate provider's fee
```

        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
    }
