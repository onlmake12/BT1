Audit Report

## Title
Stale `descendants_*` Accounting in `remove_entry_and_descendants` Leaves Surviving Ancestors with Inflated Eviction Keys — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link graph entries for the entire removal batch before invoking `remove_entry` on any of them. Because `remove_entry` relies on the live link graph to locate surviving ancestors and call `sub_descendant_weight` + refresh `evict_key`, those updates never occur. Surviving ancestors permanently carry inflated `descendants_fee`/`descendants_size`/`descendants_cycles`/`descendants_count` and a stale `evict_key`, corrupting the pool's eviction ordering for the lifetime of those entries.

## Finding Description

**Root cause — pre-removal of links breaks ancestor accounting:**

`remove_entry_and_descendants` (lines 252–265) first calls `remove_entry_links` for every entry in the removal batch, then calls `remove_entry` on each:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ALL links torn down here
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // remove_entry now sees empty link graph
        .collect()
}
```

`remove_entry_links` (lines 418–430) removes the entry from `self.links.inner` entirely via `self.links.remove(id)`, and also removes it from its parents' children sets and its children's parents sets.

`remove_entry` (lines 235–250) then calls `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` at line 242, which calls `self.links.calc_ancestors(&child.proposal_short_id())` at line 434. `calc_ancestors` delegates to `calc_relative_ids` (links.rs lines 37–50), which does `self.inner.get(short_id)` — but since `remove_entry_links` already called `self.links.remove(id)`, this returns `None`, and `direct` becomes an empty set via `.unwrap_or_default()`. `calc_relation_ids` then returns an empty `HashSet`. No surviving ancestor ever has `sub_descendant_weight` called on it, and no surviving ancestor's `evict_key` is refreshed.

**Contrast with the single-entry path:**

In the single-entry `remove_entry` path (lines 235–250), `remove_entry_links` is called at line 245, *after* `update_ancestors_index_key` at line 242, so the link graph is intact when the ancestor walk happens. The batch path inverts this ordering for the entire batch.

**Fields left stale on surviving ancestors:**

`sub_descendant_weight` (entry.rs lines 133–142) decrements `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee`. None of these are decremented for surviving ancestors. The `evict_key` (lines 442–443) is recomputed from those fields and is therefore also stale.

**`EvictKey` computation:**

`EvictKey` (entry.rs lines 234–247) sets `fee_rate = max(descendants_feerate, feerate)`. With inflated `descendants_fee`/`descendants_size`/`descendants_cycles`, `descendants_feerate` is computed over a phantom package larger than what actually exists, making the ancestor appear to have a higher fee rate than it does.

**`next_evict_entry` ordering:**

`next_evict_entry` (pool_map.rs lines 380–385) iterates `iter_by_evict_key()` and returns the entry with the lowest `evict_key`. `EvictKey::cmp` (sort_key.rs lines 92–103) orders by `fee_rate` ascending. A surviving ancestor with an inflated `fee_rate` in its `evict_key` is ranked higher (less likely to be evicted) than it should be.

## Impact Explanation

**High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker can exploit this to permanently anchor low-fee-rate transactions in the pool:

1. Submit parent A (low fee rate). Submit child B (high fee rate, spends A's output). A's `evict_key` is inflated to reflect B's fee.
2. Replace B via RBF. `remove_entry_and_descendants(B)` is called. A's `descendants_*` fields are never decremented. A's `evict_key` remains inflated.
3. The replacement B′ (which must also spend A's output) is added, further inflating A's `descendants_*` via `add_descendant_weight`.
4. Repeating steps 2–3 accumulates stale fee contributions in A's `descendants_*`, making A's apparent `descendants_feerate` grow without bound.
5. When the pool is full, `limit_size` calls `next_evict_entry`, which skips A because its apparent fee rate is inflated. Legitimate transactions at A's true fee rate are rejected while A occupies pool space indefinitely.

Because all CKB nodes share this code path, the attack affects the mempool of every node simultaneously. The bug also fires on non-attacker paths: `resolve_conflict` (block relay), `resolve_conflict_header_dep` (header relay), and `remove_expired` (expiry timer) all call `remove_entry_and_descendants`, meaning the pool's eviction ordering degrades silently under normal operation.

## Likelihood Explanation

Any unprivileged transaction sender can trigger this. The preconditions are:
- Submit a parent transaction A and a child transaction B (standard mempool operations).
- Submit a replacement B′ that satisfies RBF rules (higher fee rate than B).

No special privileges, leaked keys, or victim mistakes are required. The trigger is a standard RBF submission, which is a documented and supported protocol behavior. The bug fires on every invocation of `remove_entry_and_descendants`, including automatic pool maintenance (expiry, size-limit eviction), so it is also triggered without any attacker involvement under normal network load.

## Recommendation

Move the ancestor-index update before tearing down the link graph. For each entry in `removed_ids`:

1. While links are still intact, call `update_ancestors_index_key(entry, EntryOp::Remove)`, restricting updates to ancestors **not** in `removed_ids` (to avoid updating entries that are themselves being removed).
2. Only then call `remove_entry_links` and `remove_entry` for that entry.

Alternatively, replicate the single-entry `remove_entry` ordering (links removed last) and guard against double-updates within the batch by checking whether each ancestor ID is in `removed_ids` before applying `sub_descendant_weight`.

## Proof of Concept

**Minimal unit test plan** against `PoolMap` in `tx-pool/src/component/tests/`:

1. Build chain A → B (A is parent, B is child). Add both to the pool.
2. Record A's `descendants_count`, `descendants_fee`, `descendants_size`, `descendants_cycles`, and `evict_key` before removal.
3. Call `pool_map.remove_entry_and_descendants(&B.proposal_short_id())`.
4. Fetch A's entry from the pool. Assert:
   - `A.descendants_count == 1` (should equal A's own count only).
   - `A.descendants_fee == A.fee` (B's fee should have been subtracted).
   - `A.evict_key.fee_rate` equals `FeeRate::calculate(A.fee, get_transaction_weight(A.size, A.cycles))`.
5. With the current code, all three assertions fail: `descendants_count` still includes B, `descendants_fee` still includes B's fee, and `evict_key` is stale.

**Manual RBF PoC:**

1. Submit A (low fee rate, e.g., 1 shannon/byte) and B (child of A, high fee rate, e.g., 1000 shannons/byte).
2. Submit B′ (RBF replacement for B, fee rate > B's fee rate).
3. Inspect A via `get_pool_tx_verbose` RPC: `descendants_fee` and `descendants_cycles` still include B's values.
4. Fill the pool to capacity with medium-fee-rate transactions.
5. Observe that `limit_size` does not evict A, even though A's true fee rate is below the eviction threshold.
