### Title
DAO Withdrawal Broken for `header_dep_index >= 256` Due to Index Width Mismatch Between Rust Node and C VM — (`File: util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` function reads the DAO deposit `header_dep_index` from the witness as a full `u64`, while the on-chain C VM `dao.c` script reads only the **lowest byte** of that same 8-byte field. When a user constructs a valid DAO withdrawal transaction with `header_dep_index >= 256`, the Rust node resolves a completely different `header_dep` entry than the C VM script does. The Rust node then rejects the transaction as invalid, even though the C VM script would accept it. This breaks DAO withdrawal functionality for any transaction where the deposit block's position in `header_deps` is 256 or greater.

---

### Finding Description

The DAO withdrawal protocol (RFC-0023) requires the withdrawing transaction to include a witness `input_type` field containing an 8-byte little-endian `u64` index into `header_deps`, pointing to the original deposit block header.

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` reads this index as a full `u64` and uses it directly:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // uses full u64
```

However, the C VM `dao.c` script (referenced at `test/src/specs/dao/dao_user.rs:14`) reads only the **lowest byte** of the same 8-byte field when indexing into `header_deps`. This is explicitly documented in the test added to the codebase:

```rust
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
let mut header_deps = vec![dummy; 258];
header_deps[1] = deposit_block.hash();
header_deps[257] = withdraw_block.hash();

// input_type = 257, lowest byte = 1
let witness = WitnessArgs::new_builder()
    .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
    .build();
```

With `index = 257` (`0x0000000000000101` in little-endian):
- **C VM** reads lowest byte → `0x01` = 1 → resolves `header_deps[1]` = correct deposit block ✓
- **Rust node** reads full u64 → `257` → resolves `header_deps[257]` = wrong block ✗

The Rust node then compares the resolved header's block number against the cell data's `deposited_block_number` and returns `DaoError::InvalidOutPoint` because they do not match. The transaction is rejected.

---

### Impact Explanation

A user constructing a legitimate DAO withdrawal transaction where the deposit block hash happens to be placed at position ≥ 256 in `header_deps` (e.g., because many other header deps are also required) will have their transaction permanently rejected by the Rust node at both the tx-pool admission stage (`tx-pool/src/util.rs`) and the consensus verification stage (`verification/src/transaction_verifier.rs`). The C VM script would accept the same transaction. This constitutes broken DAO withdrawal functionality and a consensus split between the Rust node and any implementation that faithfully mirrors the C VM's byte-width behavior.

---

### Likelihood Explanation

A transaction with 256+ `header_deps` is unusual but not prohibited by the protocol. Each `header_dep` is a 32-byte hash; 256 entries = 8 192 bytes, well within `max_block_bytes`. A user with many DAO cells deposited across many blocks, or a user who deliberately constructs such a transaction, can trigger this path. The discrepancy is reachable by any unprivileged transaction sender via the standard `send_transaction` RPC.

---

### Recommendation

In `transaction_maximum_withdraw`, after reading the full `u64` index from the witness, validate that it fits within a `u8` (i.e., `header_dep_index <= 255`) and return `DaoError::InvalidDaoFormat` if it does not, to match the C VM's actual behavior. Alternatively, if the C VM is intended to support full `u64` indices, the C VM `dao.c` script must be updated to read all 8 bytes rather than only the lowest byte. The two implementations must agree on the same index width.

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
