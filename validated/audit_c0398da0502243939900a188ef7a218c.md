### Title
Cross-Status Linear Scan in `score_sorted_iter_by_status` / `next_evict_entry` Enables Block-Assembly and Pool-Eviction DoS via Adversarial Transaction Flooding — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

The CKB tx-pool stores Pending, Gap, and Proposed transactions in a **single unified** `MultiIndexPoolEntryMap` with shared ordered indices (`score` and `evict_key`). Two functions — `score_sorted_iter_by_status` (used during block assembly) and `next_evict_entry` (used during pool eviction) — perform **O(N\_total) linear scans** through all entries of every status to find entries of one specific status. An unprivileged transaction submitter can flood the pool with adversarially crafted Pending transactions to force these scans to traverse a large number of irrelevant entries, degrading block-template generation and pool-eviction performance in a manner directly analogous to the gas-griefing DoS described in the reference report.

---

### Finding Description

**Unified pool with no per-status index**

`PoolEntry` is declared with a single `score` ordered index and a single `evict_key` ordered index, both shared across all three statuses:

```rust
// tx-pool/src/component/pool_map.rs  lines 46-58
#[derive(MultiIndexMap, Clone)]
pub struct PoolEntry {
    #[multi_index(hashed_unique)]
    pub id: ProposalShortId,
    #[multi_index(ordered_non_unique)]
    pub score: AncestorsScoreSortKey,   // shared across Pending/Gap/Proposed
    #[multi_index(hashed_non_unique)]
    pub status: Status,                 // only a hash index, not an ordered one
    #[multi_index(ordered_non_unique)]
    pub evict_key: EvictKey,            // shared across Pending/Gap/Proposed
    pub inner: TxEntry,
}

**Vulnerable path 1 — block assembly (`score_sorted_iter_by_status`)**

`sorted_proposed_iter()` is the entry point used by `TxSelector::txs_to_commit` during every `get_block_template` call:

```rust

// pool_map.rs lines 398-406
pub(crate) fn score_sorted_iter_by_status(
    &self,
    status: Status,
) -> impl Iterator<Item = &TxEntry> {
    self.entries
        .iter_by_score()   // walks ALL N entries in score order
        .rev()
        .filter_map(move |entry| (entry.status == status).then_some(&entry.inner))
}
``` [2](#0-1) [3](#0-2) 

`iter_by_score().rev()` yields entries from highest score to lowest across **all** statuses. The `.filter_map` discards every non-Proposed entry but still visits it. If an attacker fills the pool with N\_pending Pending transactions whose `AncestorsScoreSortKey` exceeds that of all Proposed transactions, the iterator must traverse all N\_pending entries before yielding the first Proposed one.

This iterator is consumed inside `TxSelector::txs_to_commit`:

Each `iter.peek()` / `iter.next()` call advances the underlying filtered iterator, scanning through adversarial Pending entries until the next Proposed entry is found.

**Vulnerable path 2 — pool eviction (`next_evict_entry` in `limit_size`)**

`limit_size` is called after every successful transaction insertion:

`next_evict_entry` performs a linear scan:

```rust
// pool_map.rs lines 380-385

    self.entries
        .iter_by_evict_key()   // walks ALL N entries in evict_key order
        .find(move |entry| entry.status == status)
        .map(|entry| entry.id.clone())

``` [6](#0-5) 

`EvictKey` sorts by `fee_rate` ascending. If an attacker (or a miner-colluding attacker) populates the pool with many Gap/Proposed entries at the minimum fee rate, those entries occupy the front of `iter_by_evict_key()`. Every call to `next_evict_entry(Status::Pending)` must scan through all of them before finding a Pending entry, making each eviction step O(N\_gap + N\_proposed). [7](#0-6) 

---

### Impact Explanation

**Block assembly DoS (path 1):** `get_block_template` is called by mining software on a tight loop (every few seconds). Each call invokes `TxSelector::txs_to_commit`, which calls `sorted_proposed_iter()`. With N\_pending adversarial Pending entries whose scores exceed all Proposed entries, each block-template generation requires O(N\_pending) extra work just to locate the first Proposed transaction. At the default pool limit of 180 MB and minimum transaction size (~100 bytes), N can reach ~1.8 million entries. This can make `get_block_template` prohibitively slow, effectively preventing the node from producing competitive block templates and degrading mining revenue or halting block production entirely.

**Pool eviction DoS (path 2):** Every transaction submission that pushes the pool over `max_tx_pool_size` triggers the `limit_size` while loop. Each iteration calls `next_evict_entry` up to three times, each an O(N) scan. An attacker who keeps the pool near capacity forces O(N) work per submission, amplifying CPU cost for all pool participants.

---

### Likelihood Explanation

Path 1 (block assembly) is reachable by **any unprivileged RPC caller** via `send_transaction`. The attacker only needs to submit Pending transactions with fee rates slightly above the current Proposed transactions. At the default minimum fee rate of 1000 shannons/KB, filling a 180 MB pool costs approximately 1.8 CKB — a negligible economic barrier. The attack is sustained as long as the attacker keeps re-submitting transactions to replace evicted ones. Path 2 additionally requires getting transactions into Gap/Proposed status, which requires miner cooperation, making it less likely in isolation but still feasible for a miner-attacker.

---

### Recommendation

Maintain **separate per-status ordered indices** for `score` and `evict_key`, analogous to the reference report's fix of separating the order book into two lists. Concretely:

1. Replace the single shared `#[multi_index(ordered_non_unique)] pub score` with per-status score indices, or maintain three separate `BTreeMap`s (one per status) for score-sorted iteration.
2. Replace `score_sorted_iter_by_status(status)` with a direct iteration over the status-specific index, eliminating the cross-status filter scan.
3. Replace `next_evict_entry(status)` with a direct O(log N) lookup into a per-status evict-key index.

This mirrors the fix applied in the reference report: separating the unified structure into per-direction (per-status) structures so that iteration over one status does not require scanning entries of another.

### Proof of Concept

```
1. Node is running with default config (max_tx_pool_size = 180 MB, min_fee_rate = 1000 shannons/KB).

2. Assume the pool contains K Proposed transactions with AncestorsScoreSortKey = S_proposed.

3. Attacker submits N Pending transactions, each with:
   - Minimum valid size (~100 bytes)
   - Fee rate slightly above S_proposed (e.g., S_proposed + 1 shannon/byte)
   - Total cost: N * 100 shannons ≈ negligible

4. These N Pending entries now occupy the top N positions in iter_by_score().rev().

5. Miner calls get_block_template → TxPool::get_block_template → TxSelector::txs_to_commit
   → sorted_proposed_iter() → score_sorted_iter_by_status(Status::Proposed)
   → iter_by_score().rev().filter_map(|e| e.status == Proposed)

6. The iterator scans through all N adversarial Pending entries before yielding the first
   Proposed entry. For each of the K Proposed entries, the iterator must traverse O(N)
   Pending entries in the worst case.

7. Total extra work per get_block_template call: O(N * K) filter evaluations.
   With N = 100,000 and K = 1,000: 100 million filter evaluations per block template.

8. Miner's block assembly stalls; competing miners without the attack produce blocks faster.
```

The entry path is: unprivileged peer → `send_transaction` RPC → `TxPool::submit_entry` → `PoolMap::add_entry` → pool is populated → `get_block_template` → `score_sorted_iter_by_status` scans adversarial entries. [1](#0-0) [3](#0-2) [6](#0-5) [5](#0-4) [8](#0-7)
