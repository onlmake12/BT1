### Title
Stale Storage Reference After `clearRequest` in `Echo.executeCallback` Delivers Callback to `address(0)` for Overflow Requests — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`'s `executeCallback`, the request is cleared via `clearRequest` **before** the callback is dispatched. For requests stored in the overflow mapping (`requestsOverflow`), `clearRequest` issues a Solidity `delete`, which zeros every field in the struct — including `requester` and `callbackGasLimit`. Because `req` is a **storage reference** (not a memory copy), the subsequent callback is dispatched to `address(0)` with 0 gas. The try/catch silently swallows the failure, the provider is credited fees, and the requester's callback is permanently lost.

---

### Finding Description

`executeCallback` in `Echo.sol` executes in this order:

```
1. findActiveRequest(sequenceNumber)          → req (storage ref)
2. accruedFeesInWei += (req.fee + msg.value) - pythFee
3. clearRequest(sequenceNumber)               ← CLEARS STORAGE
4. firstUnfulfilledSeq advance loop
5. IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)
                  ^^^^^^^^^^^^ stale storage ref
```

`clearRequest` has two branches:

```solidity
// Echo.sol lines 323-332
function clearRequest(uint64 sequenceNumber) internal {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    Request storage req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        req.sequenceNumber = 0;          // primary slot: only seqNum zeroed
    } else {
        delete _state.requestsOverflow[key]; // overflow slot: ALL fields zeroed
    }
}
```

For **overflow requests** (stored in `requestsOverflow`), `delete` zeros every field. Since `req` in `executeCallback` is a storage reference pointing to the same slot, `req.requester` becomes `address(0)` and `req.callbackGasLimit` becomes `0` by the time the callback fires.

The callback line:

```solidity
// Echo.sol lines 176-201
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
```

calls `address(0)` with 0 gas. The `catch` block emits `PriceUpdateCallbackFailed` and execution continues. The provider's `accruedFeesInWei` has already been incremented; the requester's callback is permanently undeliverable.

Compare with the **correct pattern** used in `Entropy.sol`'s legacy path, which explicitly saves fields to memory before clearing:

```solidity
// Entropy.sol lines 663-667
address callAddress = req.requester;
EntropyStructs.Request memory reqV1 = EntropyStructConverter.toV1Request(req);
clearRequest(provider, sequenceNumber);
// WARNING: DO NOT USE req BELOW HERE AS ITS CONTENTS HAS BEEN CLEARED
```

`Echo.sol` omits this safeguard entirely.

A request lands in the overflow slot whenever its `shortKey = uint8(keccak256(abi.encodePacked(sequenceNumber))[0] & 0x1f)` collides with an already-active primary-slot request. With only 32 primary slots (`NUM_REQUESTS = 32`) and any non-trivial concurrent load, overflow is routine.

---

### Impact Explanation

- **Requester pays fees but receives no callback.** The price-update data is parsed and verified on-chain, fees are credited to the provider, but `_echoCallback` is never invoked on the requester's contract.
- **Provider accounting is permanently inflated.** `accruedFeesInWei` is incremented for work that was not delivered.
- **No recovery path.** The request is cleared; there is no retry mechanism for a failed overflow callback.
- **Severity: High** — direct, permanent loss of paid service for any requester whose request lands in the overflow slot.

---

### Likelihood Explanation

- With 32 primary slots and sequential sequence numbers, the probability of a collision is non-trivial under any real load.
- An adversary can **force** a victim's request into the overflow slot: observe the victim's pending sequence number, compute its `shortKey`, and submit a concurrent request that occupies that primary slot before the victim's request is fulfilled.
- No privileged access is required; any unprivileged transaction sender can trigger this.

---

### Recommendation

Save `req.requester` and `req.callbackGasLimit` to **memory** before calling `clearRequest`, mirroring the pattern already used in `Entropy.sol`:

```solidity
// Echo.sol executeCallback — fix
address callAddress = req.requester;
uint32 gasLimit = req.callbackGasLimit;
uint64 seqNum = req.sequenceNumber;

clearRequest(sequenceNumber);
// DO NOT USE req BELOW THIS LINE

try
    IEchoConsumer(callAddress)._echoCallback{gas: gasLimit}(seqNum, priceFeeds)
{ ... } catch { ... }
```

---

### Proof of Concept

1. Alice submits `requestPriceUpdatesWithCallback(...)` → assigned sequence number **N**. `shortKey(N) = S`. Primary slot `S` is free → stored in primary slot.
2. Bob submits `requestPriceUpdatesWithCallback(...)` → assigned sequence number **M** where `shortKey(M) = S`. Primary slot `S` is occupied by Alice → Bob's request is stored in `requestsOverflow[keccak256(M)]`.
3. Provider calls `executeCallback(providerToCredit, M, updateData, priceIds)`.
4. `findActiveRequest(M)` returns a storage reference `req` pointing to `_state.requestsOverflow[keccak256(M)]`.
5. `_state.providers[providerToCredit].accruedFeesInWei += (req.fee + msg.value) - pythFee` — provider credited.
6. `clearRequest(M)` → `delete _state.requestsOverflow[keccak256(M)]` → `req.requester = address(0)`, `req.callbackGasLimit = 0`.
7. `IEchoConsumer(address(0))._echoCallback{gas: 0}(M, priceFeeds)` → silently fails, caught by `catch`.
8. `PriceUpdateCallbackFailed` event emitted. Bob's contract never receives the price update. Bob's fees are gone.
