### Title
State Mutation Before Push Causes Permanent Alert Delivery Failure to All Connected Peers - (`File: util/network-alert/src/alert_relayer.rs`)

### Summary

In `AlertRelayer::received`, all connected peers are permanently marked as having received an alert **before** the broadcast is attempted. If `quick_filter_broadcast` fails (e.g., `WouldBlock` on a full channel), the alert is silently dropped for every currently-connected peer with no retry, because the `known_lists` state has already been mutated to exclude them from future broadcasts.

### Finding Description

In `util/network-alert/src/alert_relayer.rs`, the `received` handler collects the set of peers to broadcast to by calling `mark_as_known` as a **side-effect inside the `filter` closure**:

```rust
let selected_peers: Vec<PeerIndex> = nc
    .connected_peers()
    .into_iter()
    .filter(|peer| self.mark_as_known(*peer, alert_id))  // ← mutates state here
    .collect();
if let Err(err) = nc.quick_filter_broadcast(
    TargetSession::Multi(Box::new(selected_peers.into_iter())),
    data,
) {
    debug!("alert broadcast error: {:?}", err);  // ← only debug-logged, no retry
}
``` [1](#0-0) 

`mark_as_known` inserts the `alert_id` into the peer's entry in the `LruCache<PeerIndex, HashSet<u32>>`: [2](#0-1) 

The synchronous `quick_filter_broadcast` implementation in `DefaultCKBProtocolContext` is a single non-retrying call to `ServiceControl::quick_filter_broadcast`: [3](#0-2) 

This call returns `WouldBlock` if the underlying channel is full. Unlike `NetworkController::try_broadcast`, which has a spin-retry loop with `thread::sleep`: [4](#0-3) 

...the `alert_relayer` makes no such retry. The error is only logged at `debug` level and execution continues. By this point, every peer in `selected_peers` is already permanently recorded in `known_lists` as having received the alert. The `KNOWN_LIST_SIZE` cap of 64 entries means eviction is the only recovery path. [5](#0-4) 

The `connected()` handler does re-send stored alerts to **newly** connecting peers: [6](#0-5) 

However, peers that were **already connected** at the time of the failed broadcast are permanently excluded from receiving the alert from this node, because they are in `known_lists` and `mark_as_known` returns `false` for them on any subsequent relay attempt.

### Impact Explanation

Network alerts carry critical security messages (e.g., emergency upgrade notices). If the broadcast fails at the moment of relay, all currently-connected peers on that node miss the alert permanently. In a scenario where many nodes experience the same channel-full condition simultaneously (e.g., during a network-wide traffic spike or targeted flooding), the alert may fail to propagate across a significant portion of the network, defeating the purpose of the alert system.

### Likelihood Explanation

The `quick_filter_broadcast` path goes through a bounded channel (`ServiceControl`). Under high peer count or message load — conditions reachable by any connected peer sending a burst of valid protocol messages — the channel can return `WouldBlock`. An unprivileged peer does not need to forge an alert; they only need to saturate the quick-queue at the moment a legitimate alert is relayed. The alert itself requires 2-of-4 developer signatures, but the delivery failure is triggered purely by channel pressure, which any connected peer can influence.

### Recommendation

Separate the state mutation from the broadcast attempt. Collect the candidate peers without marking them, attempt the broadcast, and only mark peers as known on success:

```rust
// 1. Collect without mutating state
let selected_peers: Vec<PeerIndex> = nc
    .connected_peers()
    .into_iter()
    .filter(|peer| !self.is_known(*peer, alert_id))
    .collect();

// 2. Attempt broadcast
if nc.quick_filter_broadcast(
    TargetSession::Multi(Box::new(selected_peers.iter().copied())),
    data,
).is_ok() {
    // 3. Only mark as known on success
    for peer in &selected_peers {
        self.mark_as_known(*peer, alert_id);
    }
}
```

Additionally, elevate the broadcast error log from `debug` to `warn` or `error`, and consider a retry loop analogous to `NetworkController::try_broadcast`.

### Proof of Concept

1. Connect N peers to a CKB node.
2. Flood the node's quick-queue channel from one or more connected peers (send a burst of valid relay/sync messages) until the channel is at capacity.
3. Relay a valid signed network alert to the node.
4. `AlertRelayer::received` is called: lines 166–170 mark all N peers as known in `known_lists`.
5. `quick_filter_broadcast` at line 171 returns `Err(WouldBlock)`; the error is logged at `debug` only.
6. Observe that none of the N peers receive the alert message.
7. Send the same alert again from another peer: the node's `has_received` check at line 147 now returns `false` (alert not yet in notifier), but `mark_as_known` at line 169 returns `false` for all peers (already marked), so `selected_peers` is empty and no broadcast occurs.
8. The alert is added to `notifier` at line 178 only after the failed broadcast, so only future **newly connecting** peers will receive it — all currently-connected peers are permanently excluded. [7](#0-6)

    fn mark_as_known(&mut self, peer: PeerIndex, alert_id: u32) -> bool {
        match self.known_lists.get_mut(&peer) {
            Some(alert_ids) => alert_ids.insert(alert_id),
            None => {
                let mut alert_ids = HashSet::new();
                alert_ids.insert(alert_id);
                self.known_lists.put(peer, alert_ids);
                true
            }
        }
    }
