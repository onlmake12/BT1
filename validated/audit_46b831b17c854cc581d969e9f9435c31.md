### Title
Inbound Peer Can Inject Arbitrary Listen Port via `GetNodes.listen_port`, Bypassing Identify-Protocol Verification, Causing Announced Address Poisoning — (`network/src/protocols/discovery/mod.rs`, `network/src/protocols/discovery/state.rs`)

### Summary

An unprivileged inbound peer can send `GetNodes{version:1, listen_port:P}` to make the victim node store and subsequently announce `attacker_real_IP:P` (with an attacker-chosen port) to up to 3 other peers per `notify()` tick, violating the invariant that only identify-protocol-verified addresses should be propagated as listen addresses.

### Finding Description

**Step 1 — Inbound session initialization**

When an inbound peer connects, `SessionState::new()` sets:

```rust
RemoteAddress::Init(context.session.address.clone())
``` [1](#0-0) 

The address is `Init`, meaning it is not yet eligible for announcement.

**Step 2 — `GetNodes` handler calls `update_port(P)`**

When the attacker sends `GetNodes{version:1, listen_port:Some(P)}`, the handler at: [2](#0-1) 

calls `state.remote_addr.update_port(port)`. `update_port` replaces only the TCP port component of the existing address (which carries the real attacker IP from the TCP connection) with the attacker-supplied value, and **immediately transitions the variant from `Init` to `Listen`**: [3](#0-2) 

The resulting `Listen(attacker_IP:P)` address is then immediately added to the peer store via `add_new_addr()`: [4](#0-3) 

The subsequent `change_to_listen()` call at line 138 is a no-op because `update_port` already set the variant to `Listen`.

**Step 3 — `notify()` picks up the poisoned address**

In `notify()`, `check_timer()` returns `Some(&addr)` whenever `remote_addr` is the `Listen` variant: [5](#0-4) 

The address then passes two filters:

```rust
.filter(|addr| self.addr_mgr.is_valid_addr(addr))   // public IP check
&& let Some(flags) = self.addr_mgr.node_flags(*id)  // identify_info check
``` [6](#0-5) 

- `is_valid_addr` checks `is_reachable(ip)` — passes for any public IP, which the attacker's real IP satisfies.
- `node_flags` returns `Some` only if `identify_info` is set (populated by the identify protocol). Since the identify protocol completes within seconds of connection and `notify()` fires every 60 seconds, this is set for any normal inbound connection by the time the first tick fires.

**Step 4 — Propagation to 3 peers**

The poisoned address is pushed into `announce_list` and then distributed to up to 3 randomly chosen peer sessions: [7](#0-6) 

On the next `send_messages()` call those sessions emit `Nodes(announce=true)` containing `attacker_IP:P`: [8](#0-7) 

The design intent is acknowledged but unimplemented — the `FIXME` comment in `SessionState` reads:

```
// FIXME: Remote listen address, resolved by id protocol
pub(crate) remote_addr: RemoteAddress,
``` [9](#0-8) 

### Impact Explanation

- The attacker controls the port (any `u16`); the IP is their real public IP from the TCP connection — arbitrary IPs cannot be injected.
- `attacker_IP:P` is stored in the victim's peer store and re-announced to 3 peers per 60-second tick, from which it propagates further.
- Receiving peers will attempt connections to `attacker_IP:P`; if nothing listens there, connections fail. If the attacker controls a different service on port P, connections are redirected.
- This degrades peer-discovery quality network-wide and can be used to exhaust peer-store slots with unreachable addresses.

### Likelihood Explanation

- Requires only one inbound TCP connection from a public IP — no privilege, no PoW, no key material.
- The attacker must complete the identify protocol normally (trivial; it is automatic).
- The `received_get_nodes` guard prevents duplicate `GetNodes` per session, but one message per connection is sufficient.
- The 60-second `ANNOUNCE_CHECK_INTERVAL` means the poisoned address propagates within one minute of connection.

### Recommendation

- Do not accept `listen_port` from the discovery `GetNodes` message as authoritative. The listen address should be sourced exclusively from the identify protocol's verified exchange.
- If `update_port` must remain for NAT traversal, gate it on `identify_info` already being set and the port matching what the identify protocol reported.
- Alternatively, remove `update_port` entirely and rely solely on `change_to_listen` (which uses the connection address without port substitution) for the reuse-port path.

### Proof of Concept

1. Start a victim CKB node with a public IP.
2. Connect inbound; complete the identify handshake normally.
3. Send `DiscoveryMessage::GetNodes { version: 1, count: 100, listen_port: Some(9999), required_flags: 0 }`.
4. Wait 60 seconds for `notify()` to fire.
5. Observe outbound `Nodes(announce=true)` messages to other connected peers containing `attacker_IP:9999`.
6. Confirm `attacker_IP:9999` appears in those peers' peer stores.

    pub(crate) fn check_timer(&mut self, now: Instant, interval: Duration) -> Option<&Multiaddr> {
        if self
            .last_announce
            .map(|time| now.saturating_duration_since(time) > interval)
            .unwrap_or(true)
        {
            self.last_announce = Some(now);
            if let RemoteAddress::Listen(addr) = &self.remote_addr {
                Some(addr)
            } else {
                None
            }
        } else {
            None
        }
    }
