### Title
Unbounded `uri` and `commitmentMetadata` in Entropy Provider Registration Causes Gas Exhaustion in `getProviderFee` / `getProviderInfo` — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract's `register()` and `setProviderUri()` functions accept unbounded `bytes` fields (`uri`, `commitmentMetadata`) with no length validation. These fields are stored in `ProviderInfo` and are subsequently loaded in their entirety into memory by `getProviderFee()` (called by `getFeeV2()`) and `getProviderInfo()`. Any on-chain caller of these functions against a provider with a very large `uri`/`commitmentMetadata` will pay gas proportional to the stored data size, enabling a registered provider to DoS or inflate gas costs for integrating contracts.

---

### Finding Description

`register()` stores caller-supplied `commitmentMetadata` and `uri` directly into `ProviderInfo` storage with no length check: [1](#0-0) 

`setProviderUri()` similarly stores an unbounded `newUri` with no length check: [2](#0-1) 

The `ProviderInfo` struct holds both fields as dynamic `bytes`: [3](#0-2) 

`getProviderFee()` — called by every variant of `getFeeV2()` — loads the **entire** `ProviderInfo` struct into memory: [4](#0-3) 

`getProviderInfo()` calls `EntropyStructConverter.toV1ProviderInfo()`, which copies both `commitmentMetadata` and `uri` into a new in-memory struct: [5](#0-4) [6](#0-5) 

Additionally, `register()` itself emits an event by calling `toV1ProviderInfo(provider)` after writing the large fields to storage, meaning re-registration with a large payload is also expensive: [7](#0-6) 

---

### Impact Explanation

Any smart contract that integrates with Entropy and calls `getFeeV2(attackerProvider, gasLimit)` or `getProviderInfo(attackerProvider)` on-chain will pay gas proportional to the size of the attacker's stored `uri`/`commitmentMetadata`. On low-cost EVM chains (Polygon, BNB Chain, etc.) where block gas limits are 30M+, a provider can store data large enough to make these calls consume millions of gas, effectively DoS-ing or making prohibitively expensive any on-chain fee-check or provider-info-read against that provider. If a contract allows user-specified provider addresses, a malicious provider can grief all users of that contract.

---

### Likelihood Explanation

Registration is **permissionless** — anyone can call `register()` and become a provider. The attacker only needs to pay the one-time gas cost to store a large payload (which can be done incrementally via `setProviderUri()` across multiple transactions on low-cost chains). The cost to the attacker is bounded by the block gas limit per transaction, but the damage to on-chain callers is unbounded per call.

---

### Recommendation

Introduce a maximum length check on `uri` and `commitmentMetadata` in both `register()` and `setProviderUri()`, analogous to the fix applied in the Hats Protocol (capped at ~7000 bytes):

```solidity
uint256 public constant MAX_URI_LENGTH = 7000;
uint256 public constant MAX_METADATA_LENGTH = 7000;

function register(..., bytes calldata commitmentMetadata, ..., bytes calldata uri) public override {
    if (uri.length > MAX_URI_LENGTH) revert EntropyErrors.UriTooLong();
    if (commitmentMetadata.length > MAX_METADATA_LENGTH) revert EntropyErrors.MetadataTooLong();
    ...
}
```

Apply the same check in `setProviderUri()`.

---

### Proof of Concept

1. Attacker calls `register(0, commitment, largeMetadata, 1, largeUri)` where `largeUri` and `largeMetadata` are each ~30KB (feasible within a single block on Polygon at ~30M gas limit).
2. Attacker calls `setProviderUri(largeUri2)` repeatedly to grow the stored URI further across multiple transactions.
3. A victim contract calls `getFeeV2(attackerProvider, gasLimit)` on-chain. Internally, `getProviderFee()` executes `EntropyStructsV2.ProviderInfo memory provider = _state.providers[attackerProvider]`, loading all storage slots for the large `uri`/`commitmentMetadata` into memory. Each 32-byte cold storage slot costs 2100 gas (EIP-2929). For 30KB of data: ~937 slots × 2100 = ~1.97M gas just for the SLOAD operations, plus memory expansion costs.
4. The victim contract's on-chain fee-check reverts out-of-gas or costs far more than expected, effectively DoS-ing the integration.

        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,

        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0

        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;
