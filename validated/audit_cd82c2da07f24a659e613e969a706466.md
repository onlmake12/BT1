Audit Report

## Title
Peer Store Permanent Lock via Single-Group Address Flooding — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
`check_purge` in `PeerStore` uses integer division `len / 2` to select eviction candidates from network groups, where `len` is the number of distinct `/16` network groups. When all 16,384 stored addresses belong to a single `/16` subnet, `len = 1` and `1 / 2 = 0` in Rust integer division, causing `take(0)` to yield an empty iterator. `candidate_peers` is empty, `check_purge` returns `Err(PeerStoreError::EvictionFailed)`, and `add_addr` propagates this error unconditionally via `?`. The peer store becomes permanently locked until node restart.

## Finding Description

**Root cause — `check_purge` step 2 (`peer_store_impl.rs` L366–401):**

`check_purge` is entered whenever `addr_manager.count() >= ADDR_COUNT_LIMIT` (16,384):

```rust
if self.addr_manager.count() < ADDR_COUNT_LIMIT {
    return Ok(());
}
``` [1](#0-0) 

Step 1 collects addresses where `is_connectable(now_ms)` returns `false`. For freshly injected addresses, `AddrInfo::new` initialises `last_tried_at_ms = 0`, `attempts_count = 0`, `last_connected_at_ms = 0`: [2](#0-1) 

Walking `is_connectable`:
- `tried_in_last_minute`: `0 >= now_ms − 60_000` → false (now_ms ≈ 1.7 × 10¹²)
- `last_connected_at_ms == 0 && attempts_count >= ADDR_MAX_RETRIES (3)`: `0 >= 3` → false
- `now_ms.saturating_sub(0) > ADDR_TIMEOUT_MS && attempts_count >= ADDR_MAX_FAILURES (10)`: second clause `0 >= 10` → false

All freshly injected addresses return `true` from `is_connectable`, so step 1 evicts nothing: [3](#0-2) 

Step 2 groups addresses by network group. IPv4 addresses are bucketed by the first two octets only: [4](#0-3) 

Every address in `1.2.0.0/16` maps to `Group::IP4([1, 2])` — a single group, so `len = 1`. The eviction iterator then computes:

```rust
let len = peers_by_network_group.len();  // = 1
peers.into_iter()
    .take(len / 2)                       // take(0) — empty iterator
    ...
    .collect()                           // candidate_peers = []
``` [5](#0-4) 

The empty-check then returns the error: [6](#0-5) 

`add_addr` propagates this error unconditionally via `?`: [7](#0-6) 

## Impact Explanation

Once the store is at capacity with a single network group, every subsequent `add_addr` call returns `Err(EvictionFailed)`. The node cannot record any new peer address from any network group. As existing connections drop, the node cannot replenish its peer set and becomes progressively isolated from the CKB network. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node* — network isolation renders the node unable to participate in the network, which is functionally equivalent to a crash.

## Likelihood Explanation

The attack requires only that an adversary delivers ~16,384 discovery `SendAddr` messages carrying addresses from a single `/16` subnet (`ADDR_COUNT_LIMIT = 16384`): [8](#0-7) 

A `/16` subnet contains 65,536 addresses, making 16,384 distinct addresses trivially available. The attack requires no PoW, no key material, and no privileged access — a single controlled peer node suffices. The outcome is deterministic and repeatable after each node restart.

## Recommendation

Replace the `len / 2` integer division with a minimum-of-one guard in `check_purge`:

```rust
let take_count = (len / 2).max(1);
peers.into_iter().take(take_count) …
```

Additionally, add a per-group cap during insertion in `add_addr` or `AddrManager::add` to prevent a single `/16` from monopolising the store:

```rust
const MAX_ADDRS_PER_GROUP: usize = 256;
if group_count >= MAX_ADDRS_PER_GROUP { return Ok(()); }
```

## Proof of Concept

```rust
let mut peer_store = PeerStore::default();
// Fill store with 16384 addresses from a single /16
for i in 0u16..=255 {
    for j in 0u16..=63 {
        let addr: Multiaddr = format!("/ip4/1.2.{i}.{j}/tcp/8115").parse().unwrap();
        peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    }
}
// Store is now at ADDR_COUNT_LIMIT with exactly 1 network group
// Any subsequent add_addr — even from a completely different /16 — must fail
let new_addr: Multiaddr = "/ip4/9.9.9.9/tcp/8115".parse().unwrap();
assert!(peer_store.add_addr(new_addr, Flags::COMPATIBILITY).is_err());
// check_purge: count=16384 >= ADDR_COUNT_LIMIT, step1 evicts nothing (all connectable),
// step2: len=1, take(0), candidate_peers=[], returns Err(EvictionFailed)
```
