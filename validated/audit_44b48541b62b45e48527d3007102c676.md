### Title
Publisher Stake Account Reassignment Without Prior Delegation Record Advancement Causes Pending Reward Loss - (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`, `governance/pyth_staking_sdk/src/pyth-staking-client.ts`)

---

### Summary

The `set_publisher_stake_account` instruction in the Pyth OIS (Oracle Integrity Staking) integrity pool program does not require `advance_delegation_record` to be called first for the publisher's current stake account before reassigning it. This is structurally identical to the M-19 bug: a state-changing operation that invalidates the current staking position proceeds without first settling pending rewards, causing those rewards to be permanently unclaimable.

---

### Finding Description

In the OIS system, rewards for a publisher's self-staked position are tracked via a `DelegationRecord` PDA keyed by `(publisher, stake_account_positions)`. The `advance_delegation_record` instruction settles all pending epoch rewards into the stake account custody up to the current epoch, updating `delegation_record.last_epoch`.

The `set_publisher_stake_account` instruction (IDL discriminator `[99, 46, 72, 132, 100, 235, 211, 117]`) accepts only five accounts:

```
signer, publisher, pool_data, pool_config,
new_stake_account_positions_option (optional),
current_stake_account_positions_option (optional)
``` [1](#0-0) 

Critically, `delegation_record`, `stake_account_custody`, and `pool_reward_custody` are **absent** from this instruction's account list. The instruction therefore cannot advance the delegation record as part of its execution. It simply overwrites the publisher's registered stake account in `pool_data`.

The SDK's `setPublisherStakeAccount` and `reassignPublisherStakeAccount` methods call this instruction directly with no preceding `advance_delegation_record` call:

```typescript
async setPublisherStakeAccount(publisher, stakeAccountPositions, newStakeAccountPositions) {
  const instruction = await this.integrityPoolProgram.methods
    .setPublisherStakeAccount()
    .accounts({ ... })
    .instruction();
  await sendTransaction([instruction], this.connection, this.wallet);
}
``` [2](#0-1) [3](#0-2) 

After `set_publisher_stake_account` executes, `pool_data` records the new stake account as the publisher's canonical stake account. When `advance_delegation_record` is subsequently called for the old publisher stake account, the `publisher_stake_account_positions` passed (the old account) no longer matches what `pool_data` records. The `publisher_stake_account_positions` field is optional in `advance_delegation_record`: [4](#0-3) 

If the program enforces that the optional `publisher_stake_account_positions` must match the publisher's registered account in `pool_data` when provided, the old delegation record becomes permanently un-advanceable after reassignment. All epochs between `delegation_record.last_epoch` and the reassignment epoch are lost. If the field is simply ignored when it doesn't match, the publisher's self-staking reward component is silently skipped, producing an understated reward payout.

The `DelegationRecord` struct confirms the epoch-tracking mechanism: [5](#0-4) 

---

### Impact Explanation

A publisher who has accrued OIS rewards across one or more epochs (i.e., `delegation_record.last_epoch < current_epoch`) and whose stake account is reassigned — either by themselves via `reassignPublisherStakeAccount` or by governance via the multisig CLI — without a prior `advance_delegation_record` call will permanently lose all pending rewards for the epochs between `last_epoch` and the reassignment epoch. These rewards are locked in `pool_reward_custody` but can never be credited to the old stake account custody after the reassignment invalidates the delegation record's advanceability. [6](#0-5) 

---

### Likelihood Explanation

The `reassignPublisherStakeAccount` flow is a documented, user-facing operation available both in the staking SDK and the xc_admin CLI. Publishers reassign stake accounts during operational key rotations or account migrations. The SDK provides no guard, warning, or automatic pre-flight `advance_delegation_record` call before executing the reassignment. Any publisher performing this operation without manually constructing and prepending the advance instruction will silently lose pending rewards. The likelihood is **medium**: the operation is infrequent but entirely normal, and the loss is silent with no on-chain revert.

---

### Recommendation

The `setPublisherStakeAccount` method in `governance/pyth_staking_sdk/src/pyth-staking-client.ts` should be modified to automatically prepend `advance_delegation_record` instructions for the current publisher stake account before submitting the `set_publisher_stake_account` instruction. Specifically, `getAdvanceDelegationRecordInstructions` should be called for the publisher's current stake account and the resulting instructions included in the same transaction. Alternatively, the on-chain `set_publisher_stake_account` instruction should enforce via a constraint that `delegation_record.last_epoch == current_epoch` for the current stake account before allowing reassignment. [2](#0-1) 

---

### Proof of Concept

1. Publisher P has stake account `SA_old` registered in `pool_data`. `delegation_record(P, SA_old).last_epoch = E`.
2. Current epoch advances to `E+3`. Publisher P has accrued 3 epochs of self-staking rewards, not yet settled.
3. Publisher P (or governance) calls `set_publisher_stake_account(publisher=P, current=SA_old, new=SA_new)` directly, without first calling `advance_delegation_record` for `SA_old`.
4. `pool_data` now records `SA_new` as P's stake account. `delegation_record(P, SA_old).last_epoch` remains `E`.
5. Any subsequent call to `advance_delegation_record(publisher=P, stake_account_positions=SA_old, publisher_stake_account_positions=SA_old)` either reverts (if the program validates the publisher stake account against `pool_data`) or silently skips the publisher's self-staking reward component (if the mismatch is ignored).
6. The 3 epochs of rewards for `SA_old` are permanently unclaimable. `SA_new` starts with `last_epoch = current_epoch`, inheriting no historical rewards. [7](#0-6) [1](#0-0)

  async getAdvanceDelegationRecordInstructions(
    stakeAccountPositions: PublicKey,
    payer?: PublicKey,
  ) {
    const poolData = await this.getPoolDataAccount();
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const allPublishers = extractPublisherData(poolData);
    const publishers = allPublishers
      .map((publisher) => {
        const positionsWithPublisher =
          stakeAccountPositionsData.data.positions.filter(
            ({ targetWithParameters }) =>
              targetWithParameters.integrityPool?.publisher.equals(
                publisher.pubkey,
              ),
          );

        let lowestEpoch;
        for (const position of positionsWithPublisher) {
          lowestEpoch = bigintMin(position.activationEpoch, lowestEpoch);
        }

        return {
          ...publisher,
          lowestEpoch,
        };
      })
      .filter(({ lowestEpoch }) => lowestEpoch !== undefined);

    const delegationRecords = await Promise.all(
      publishers.map(({ pubkey }) =>
        this.getDelegationRecord(stakeAccountPositions, pubkey),
      ),
    );

    let lowestEpoch: bigint | undefined;
    for (const [index, publisher] of publishers.entries()) {
      const maximum = bigintMax(
        publisher.lowestEpoch,
        delegationRecords[index]?.lastEpoch,
      );
      lowestEpoch = bigintMin(lowestEpoch, maximum);
    }

    const currentEpoch = await getCurrentEpoch(this.connection);

    // Filter out delegationRecord that are up to date
    const filteredPublishers = publishers.filter((_, index) => {
      return !(delegationRecords[index]?.lastEpoch === currentEpoch);
    });

    // anchor does not calculate the correct pda for other programs
    // therefore we need to manually calculate the pdas
    const advanceDelegationRecordInstructions = await Promise.all(
      filteredPublishers.map(({ pubkey, stakeAccount }) =>
        this.integrityPoolProgram.methods
          .advanceDelegationRecord()
          .accountsPartial({
            payer: payer ?? this.wallet.publicKey,
            publisher: pubkey,
            publisherStakeAccountCustody: stakeAccount
              ? getStakeAccountCustodyAddress(stakeAccount)
              : null, // eslint-disable-line unicorn/no-null
            publisherStakeAccountPositions: stakeAccount,
            stakeAccountCustody: getStakeAccountCustodyAddress(
              stakeAccountPositions,
            ),
            stakeAccountPositions,
          })
          .instruction(),
      ),
    );
