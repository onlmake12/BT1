### Title
Fee Permanently Locked in `PriceInfoObject` on Stale Update — (`target_chains/sui/contracts/sources/pyth.move`)

### Summary

In `update_single_price_feed`, the update fee is unconditionally deposited into the `PriceInfoObject` **before** the freshness check is performed. When `is_fresh_update` returns `false` (stale update), the price is silently not updated, but the fee is already irrecoverably locked inside the `PriceInfoObject`. No refund path and no fee-recipient withdrawal mechanism exist for funds stored in `PriceInfoObject`.

---

### Finding Description

The execution order in `update_single_price_feed` is:

1. **Line 277** — `price_info::deposit_fee_coins(price_info_object, fee)` — fee deposited unconditionally.
2. **Lines 283–291** — loop finds the matching `price_identifier`.
3. **Line 287** — `update_cache(...)` is called.
4. Inside `update_cache`, **line 316** — `if (is_fresh_update(update, price_info_object))` — price is only written when the incoming timestamp strictly exceeds the cached one.
5. If `is_fresh_update` returns `false`, `update_cache` returns silently with no error and no refund. [1](#0-0) 

`deposit_fee_coins` merges the caller's `Coin<SUI>` into a dynamic object field on the `PriceInfoObject`: [2](#0-1) 

There is no `withdraw_fee_coins`, no refund function, and no mechanism for the fee recipient to pull funds out of a `PriceInfoObject`. The `fee_recipient_address` stored in `State` is used only for governance-level operations; it has no access to the per-object fee balance. [3](#0-2) 

`is_fresh_update` requires strictly greater timestamp to return `true`: [4](#0-3) 

---

### Impact Explanation

Every call to `update_single_price_feed` with a stale-but-cryptographically-valid accumulator message permanently destroys the caller's fee. The funds accumulate in the `PriceInfoObject` with no withdrawal path for either the payer or the fee recipient. This directly matches the scope criterion: **locking of user funds in-motion**.

---

### Likelihood Explanation

Two realistic paths exist:

1. **Accidental staleness** — A user fetches a fresh accumulator message, but by the time their Sui transaction is sequenced, another updater has already applied the same (or a newer) message. The user's fee is lost.
2. **Front-run / griefing** — An attacker observes a pending `update_single_price_feed` transaction in the mempool, submits the identical accumulator message first, and causes the victim's subsequent transaction to deposit a fee without updating the price. Repeating this drains the victim's wallet.

Both paths require only a public, permissionless call with a valid (but stale) accumulator message — no privileged access needed.

---

### Recommendation

Move `deposit_fee_coins` **after** the freshness check, or abort (refund via transaction revert) when `is_fresh_update` returns `false`. The simplest fix:

```move
// inside update_cache, before the silent return on stale:
if (!is_fresh_update(update, price_info_object)) {
    abort E_STALE_PRICE_UPDATE   // causes full tx revert, coins returned to caller
};
```

Alternatively, deposit the fee only on the branch where `update_price_info_object` is actually called.

---

    public fun update_single_price_feed(
        pyth_state: &PythState,
        price_updates: HotPotatoVector<PriceInfo>,
        price_info_object: &mut PriceInfoObject,
        fee: Coin<SUI>,
        clock: &Clock
    ): HotPotatoVector<PriceInfo> {
        let latest_only = state::assert_latest_only(pyth_state);

        // On Sui, users get to choose which price feeds to update. They specify a single price feed to
        // update at a time. We therefore charge the base fee for each such individual update.
        // This is a departure from Eth, where users don't get to necessarily choose.
        assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);

        // store fee coins within price info object
        price_info::deposit_fee_coins(price_info_object, fee);

        // Find price update corresponding to PriceInfoObject within the array of price_updates
        // and use it to update PriceInfoObject.
        let i = 0;
        let found = false;
        while (i < hot_potato_vector::length<PriceInfo>(&price_updates)){
            let cur_price_info = hot_potato_vector::borrow<PriceInfo>(&price_updates, i);
            if (has_same_price_identifier(cur_price_info, price_info_object)){
                found = true;
                update_cache(latest_only, cur_price_info, price_info_object, clock);
                break
            };
            i = i + 1;
        };
        if (found==false){
            abort E_PRICE_UPDATE_NOT_FOUND_FOR_PRICE_INFO_OBJECT
        };
        price_updates
    }

    fun is_fresh_update(update: &PriceInfo, price_info_object: &PriceInfoObject): bool {
        // Get the timestamp of the update's current price
        let price_feed = price_info::get_price_feed(update);
        let update_timestamp = price::get_timestamp(&price_feed::get_price(price_feed));

        // Get the timestamp of the cached data for the price identifier
        let cached_price_info = price_info::get_price_info_from_price_info_object(price_info_object);
        let cached_price_feed =  price_info::get_price_feed(&cached_price_info);
        let cached_timestamp = price::get_timestamp(&price_feed::get_price(cached_price_feed));

        update_timestamp > cached_timestamp
    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
