### Title
`HeaderAcceptor::version_check` Ignores RFC0048 Hardfork Switch, Permanently Rejects Valid Post-CKB2023 Headers — (`sync/src/synchronizer/headers_process.rs`)

### Summary

`HeaderAcceptor::version_check()` unconditionally rejects any header with `version != 0`, with no consultation of the RFC0048 hardfork switch (`is_remove_header_version_reservation_rule_enabled`). That switch function is defined but never called anywhere in the production codebase. After RFC0048 activates at `CKB2023_START_EPOCH = 12_293`, any legitimately mined block with a non-zero version field will be permanently marked `BLOCK_INVALID` by the sync layer, stalling chain progress.

### Finding Description

`HeaderAcceptor::version_check()` is a hard-coded, epoch-blind check: [1](#0-0) 

It is called unconditionally inside `HeaderAcceptor::accept()`, and on failure the block hash is written into the shared block-status map as `BLOCK_INVALID`: [2](#0-1) 

The hardfork infrastructure for RFC0048 is fully wired: `CKB2023` carries an `rfc_0048` epoch field, and the `define_methods!` macro generates `is_remove_header_version_reservation_rule_enabled(epoch) -> bool`: [3](#0-2) 

The mainnet activation epoch is set to `12_293`: [4](#0-3) 

And `CKB2023::new_mirana()` wires it into the consensus object used by every mainnet node: [5](#0-4) 

A grep across the entire Rust source tree for `is_remove_header_version_reservation_rule_enabled` returns exactly **one hit** — the macro definition itself. The function is never called in any production path. `version_check()` therefore has no awareness of the hardfork boundary.

Neither `BlockVerifier` (non-contextual block verification) nor `HeaderVerifier` (context-dependent header verification) performs a version check: [6](#0-5) [7](#0-6) 

So `HeaderAcceptor::version_check()` is the **sole** enforcement point for the header version rule in the sync layer, and it is permanently frozen at "version must be 0."

### Impact Explanation

After epoch 12293 activates RFC0048, the protocol permits non-zero header versions (e.g., for future versionbits signaling via the header version field). Any peer that relays such a header — even one backed by valid PoW — will cause the receiving node to:

1. Call `version_check()`, which returns `Err(())`.
2. Write `BlockStatus::BLOCK_INVALID` for that block hash into the shared status map.
3. Refuse to fetch or process the full block.
4. Propagate the `BLOCK_INVALID` status to all descendant headers via `prev_block_check`.

If the canonical chain advances through blocks with non-zero version, affected nodes permanently stall at the last zero-version block. The node does not crash, but it cannot follow the canonical chain.

### Likelihood Explanation

RFC0048 is a scheduled mainnet hardfork at epoch 12293. The versionbits signaling mechanism currently uses the cellbase witness rather than the header version field (the header-version-based `condition()` is commented out in `spec/src/versionbits/mod.rs`), so miners do not currently produce non-zero version blocks. However, RFC0048 explicitly removes the reservation rule to enable future use of the version field. Any future softfork or protocol upgrade that uses the header version field for signaling would immediately trigger this bug on all unpatched nodes. The exploit requires no privileged access — relaying a legitimately mined block is sufficient.

### Recommendation

`version_check()` must consult the hardfork switch. The fix is to pass the consensus/hardfork context into `HeaderAcceptor` and gate the check:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    let epoch_number = self.header.epoch().number();
    let hardfork = &self.consensus.hardfork_switch.ckb2023;
    if !hardfork.is_remove_header_version_reservation_rule_enabled(epoch_number)
        && self.header.version() != 0
    {
        state.invalid(Some(ValidationError::Version));
        return Err(());
    }
    Ok(())
}
```

`HeaderAcceptor` already holds a reference to `HeaderVerifier` which carries `consensus`, so the context is available without structural changes.

### Proof of Concept

Build a `HeaderView` with `version = 1` and `epoch = 12293`, run it through `HeaderAcceptor::version_check()` with a mainnet consensus object (`CKB2023::new_mirana()`), and assert it incorrectly returns `ValidationState::Invalid`. The same header passed through `HeaderVerifier::verify()` (PoW, number, epoch, timestamp) would pass all checks, confirming the version rejection is the sole cause of the stall.
