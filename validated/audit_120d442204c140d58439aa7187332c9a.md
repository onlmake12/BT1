### Title
Single-Miner Median Timestamp Manipulation Enables `since` Time-Lock Bypass — (`traits/src/header_provider.rs`)

---

### Summary

CKB computes a median block timestamp from the last 37 blocks to enforce both block header validity and transaction `since` time-lock conditions. Because 37 is odd, a single miner who mines one block can shift the median timestamp to any value within a bounded range by choosing their block's timestamp strategically. This is a direct structural analog to the Apollo Oracle single-oracle median manipulation: one "data source" (block) controls the center of the sorted set.

---

### Finding Description

`block_median_time` in `traits/src/header_provider.rs` collects timestamps from the last `median_block_count` (37) ancestor blocks, sorts them, and returns the element at index `timestamps.len() >> 1` — i.e., index 18, the exact center of the 37-element sorted array. [1](#0-0) 

The constant is fixed at 37 (odd): [2](#0-1) 

`TimestampVerifier::verify` enforces that a new block's timestamp must be **strictly greater** than this median and at most `now + ALLOWED_FUTURE_BLOCKTIME` (15 000 ms): [3](#0-2) [4](#0-3) 

This means a miner who mines block B can freely choose `B.timestamp` anywhere in the open interval `(current_median, now + 15_000]`. Because B's timestamp is one of the 37 inputs to the next median computation, the miner can push the resulting median to any value between the 18th and 20th elements (0-indexed: 17 and 19) of the sorted timestamps of the other 36 blocks — exactly the oracle-median-skew scenario from the external report.

`SinceVerifier` then uses this manipulable median directly for both absolute and relative timestamp time-locks: [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A miner who mines a single block can inflate the median timestamp by up to the gap between the 18th and 20th sorted timestamps of the surrounding 36 blocks (typically a few seconds to tens of seconds under normal block intervals). This allows the miner to:

1. **Prematurely satisfy an absolute `since` timestamp lock** — include a transaction whose `since` target falls within the manipulable range, even though honest median time has not yet reached that target.
2. **Prematurely satisfy a relative `since` timestamp lock** — reduce the apparent elapsed time required before a cell can be spent.

The attacker can front-run or selectively unlock time-locked outputs (e.g., payment channels, escrow, DAO withdrawal windows) within the manipulation window, gaining an ordering or timing advantage over other participants.

---

### Likelihood Explanation

Any miner who successfully mines even a single block can execute this. Mining is permissionless in CKB; no key compromise, trusted role, or majority hashpower is required. The manipulation window is bounded (seconds to tens of seconds) but is non-zero and deterministically computable by the miner before they commit to a timestamp. Miners already have strong economic incentives to include high-fee transactions; a miner who also holds a position in a time-locked contract has a direct financial incentive to exploit this.

---

### Recommendation

**Short term:** Monitor on-chain `since`-locked transactions for inclusion patterns that correlate with anomalous block timestamps near the upper bound of `now + ALLOWED_FUTURE_BLOCKTIME`. Alert when a block's timestamp is significantly higher than its parent's and a time-locked transaction is included in the same block.

**Long term:** Consider increasing `MEDIAN_TIME_BLOCK_COUNT` to reduce the per-block influence on the median, or adopt a stricter timestamp bound (e.g., require the new block timestamp to be within a tighter window of the previous block's timestamp rather than `now + 15s`). Alternatively, use a trimmed-mean or require a supermajority of recent timestamps to exceed a threshold before a time-lock is considered satisfied, making single-block manipulation insufficient.

---

### Proof of Concept

**Setup:** 37 consecutive blocks with timestamps (ms) approximately:

```
T-360000, T-350000, ..., T-20000, T-10000   (36 honest blocks)
```

Sorted, the 18th element (index 17) ≈ `T-200000` and the 20th element (index 19) ≈ `T-180000`. The honest median (index 18) ≈ `T-190000`.

**Attack:** Eve mines block 37 and sets its timestamp to `now + 14_999` ms (just under the `ALLOWED_FUTURE_BLOCKTIME` cap). After sorting all 37 timestamps, Eve's inflated value displaces the center element, pushing the median up to the value previously at index 19 ≈ `T-180000`.

**Result:** A transaction with `since` absolute timestamp `T-185000` (which honest median `T-190000` would reject as immature) now passes `SinceVerifier::verify_absolute_lock` because `tip_timestamp (T-180000) >= timestamp (T-185000)`. [6](#0-5) 

Eve includes this transaction in her block, bypassing the time-lock by approximately 5–10 seconds — a window that is small but deterministically controllable and sufficient to front-run time-sensitive protocol interactions.

    fn block_median_time(&self, block_hash: &Byte32, median_block_count: usize) -> u64 {
        let mut timestamps: Vec<u64> = Vec::with_capacity(median_block_count);
        let mut block_hash = block_hash.clone();
        for _ in 0..median_block_count {
            let header_fields = self
                .get_header_fields(&block_hash)
                .expect("parent header exist");
            timestamps.push(header_fields.timestamp);
            block_hash = header_fields.parent_hash;

            if header_fields.number == 0 {
                break;
            }
        }

        // return greater one if count is even.
        timestamps.sort_unstable();
        timestamps[timestamps.len() >> 1]
    }

    pub fn verify(&self) -> Result<(), Error> {
        // skip genesis block
        if self.header.is_genesis() {
            return Ok(());
        }

        let min = self.data_loader.block_median_time(
            &self.header.data().raw().parent_hash(),
            self.median_block_count,
        );
        if self.header.timestamp() <= min {
            return Err(TimestampError::BlockTimeTooOld {
                min,
                actual: self.header.timestamp(),
            }
            .into());
        }
        let max = self.now + ALLOWED_FUTURE_BLOCKTIME;
        if self.header.timestamp() > max {
            return Err(TimestampError::BlockTimeTooNew {
                max,
                actual: self.header.timestamp(),
            }
            .into());
        }
        Ok(())
    }
