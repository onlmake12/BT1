### Title
Missing Fee Withdrawal Path ŌĆö All Update Fees Permanently Locked in `PriceInfoObject` - (`target_chains/sui/contracts/sources/price_info.move`)

---

### Summary

Every call to `pyth::pyth::update_single_price_feed` deposits the caller's SUI fee into the target `PriceInfoObject` via `price_info::deposit_fee_coins`. No function anywhere in the production codebase removes or transfers those coins out. The `fee_recipient_address` stored in `State` is never consulted during fee deposit and has no associated withdrawal path. All accumulated fees are permanently locked.

---

### Finding Description

**Deposit path (confirmed):**

`update_single_price_feed` in `pyth.move` unconditionally calls:

```move
price_info::deposit_fee_coins(price_info_object, fee);
``` [1](#0-0) 

`deposit_fee_coins` in `price_info.move` stores the coin under the `FEE_STORAGE_KEY` dynamic object field of the `PriceInfoObject`, either by adding a new field or joining with the existing `Coin<SUI>`:

```move
const FEE_STORAGE_KEY: vector<u8> = b"fee_storage";
...
dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
// or
coin::join(current_fee, fee_coins);
``` [2](#0-1) [3](#0-2) 

**Withdrawal path (absent):**

A grep across all production `.move` files under `target_chains/sui/contracts/sources/` for `dynamic_object_field::remove`, `withdraw.*fee`, `fee.*withdraw`, `transfer.*fee`, and `FEE_STORAGE` returns **only** the deposit and balance-read usages in `price_info.move`. No removal or transfer of the stored `Coin<SUI>` exists anywhere.

`get_balance` can read the accumulated amount but provides no extraction: [4](#0-3) 

**`fee_recipient_address` is a dead field:**

`State` stores a `fee_recipient_address`: [5](#0-4) 

Governance can update it via `set_fee_recipient`: [6](#0-5) 

But this address is **never read** during fee deposit or at any other point to route coins. The `set_fee_recipient.move` module itself contains the admission:

> "The previous version of the contract sent the fees to a recipient address but this state is not used anymore. This module is kept for backward compatibility." [7](#0-6) 

**Governance actions enumerated ŌĆö none withdraw fees:**

The full set of governance actions is `CONTRACT_UPGRADE`, `SET_GOVERNANCE_DATA_SOURCE`, `SET_DATA_SOURCES`, `SET_UPDATE_FEE`, `SET_STALE_PRICE_THRESHOLD`, `SET_FEE_RECIPIENT`. None of these touch `FEE_STORAGE_KEY`. [8](#0-7) [9](#0-8) 

---

### Impact Explanation

Every SUI fee paid by any caller of `update_single_price_feed` is irrecoverably locked inside the corresponding `PriceInfoObject`. Because `PriceInfoObject` is a shared object with no destructor exposed to production code, and because no `dynamic_object_field::remove` call on `FEE_STORAGE_KEY` exists in any module, the locked balance can only grow. The protocol can never claim its fee revenue, and users' fee payments are permanently destroyed from an economic standpoint.

---

### Likelihood Explanation

This triggers on every single price feed update. It is not conditional on any attacker action ŌĆö the locking is the normal, intended code path. The only question is whether the missing withdrawal was intentional (a design decision to burn fees) or an oversight from the migration away from the old fee-recipient model. The comment in `set_fee_recipient.move` strongly suggests the latter: the old transfer-to-recipient mechanism was removed but no replacement withdrawal path was added.

---

### Recommendation

Add a privileged withdrawal function (gated on `LatestOnly` or a governance action) to `price_info.move` that calls `dynamic_object_field::remove<vector<u8>, Coin<SUI>>(&mut price_info_object.id, FEE_STORAGE_KEY)` and transfers the resulting coin to `state::get_fee_recipient(pyth_state)`. Wire this into a new governance action or an admin-only entry function so the fee recipient can periodically sweep accumulated fees.

---

### Proof of Concept

Static analysis proof (no exploit transaction needed ŌĆö the absence of code is the bug):

1. Search all `.move` files under `target_chains/sui/contracts/sources/` for `dynamic_object_field::remove` ŌåÆ **zero results**.
2. Search for any function that reads `FEE_STORAGE_KEY` and produces a `Coin<SUI>` output ŌåÆ **zero results** (only `get_balance` reads it as an immutable borrow).
3. Confirm `fee_recipient_address` is written by `set_fee_recipient` but never read in any fee-routing context ŌåÆ **confirmed**.
4. Confirm `deposit_fee_coins` is called on every `update_single_price_feed` invocation ŌåÆ **confirmed** at line 277 of `pyth.move`.

Conclusion: the invariant "fees paid by users must be claimable by the fee recipient" is broken. All fees are permanently locked.

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
