Audit Report

## Title
Non-Atomic Destructive Backend Read in `HeaderMapKernel::get` Causes Unconditional Panic in `HeaderAcceptor::accept` — (`shared/src/types/header_map/kernel_lru.rs`)

## Summary

`HeaderMapKernel::get` promotes entries from the sled backend to memory via a destructive `backend.remove` call that is not protected by any lock covering the full `get` body. Two concurrent callers that both miss the memory layer will both reach `backend.remove`; sled's atomic remove guarantees exactly one winner and one `None` return. When the loser's `None` propagates through `get_header_index_view` for a header whose status is `HEADER_VALID` but not yet `BLOCK_STORED`, the unconditional `panic!` in `HeaderAcceptor::accept` fires, killing the sync task and dropping the peer connection. Repeated triggering by an attacker controlling two peers can prevent a node from completing IBD.

## Finding Description

**Non-atomic destructive read in `HeaderMapKernel::get`**

`HeaderMapKernel` is wrapped in `Arc` and shared across threads. Its `get` method takes `&self`:

```rust
// kernel_lru.rs L109-141
pub(crate) fn get(&self, hash: &Byte32) -> Option<HeaderIndexView> {
    if let Some(view) = self.memory.get_refresh(hash) { // write-locks MemoryMap, then releases
        return Some(view);
    }
    if self.backend.is_empty() { return None; }
    if let Some(view) = self.backend.remove(hash) {  // destructive, no outer lock
        self.memory.insert(view.clone());
        Some(view)
    } else {
        None  // ← race loser returns None here
    }
}
```

`memory.get_refresh` acquires and releases the `RwLock` inside `MemoryMap` before `backend.remove` is ever called. There is no mutex covering the gap between the memory miss and the backend remove. Two concurrent callers that both miss memory will both call `backend.remove`; sled's per-key atomicity means exactly one gets `Some` and the other gets `None`.

**`limit_memory` creates the precondition**

The background task fires every 5 seconds:

```rust
// kernel_lru.rs L168-181
if let Some(values) = self.memory.front_n(self.memory_limit) {
    tokio::task::block_in_place(|| {
        self.backend.insert_batch(&values);   // H now in sled AND memory
    });
    self.memory.remove_batch(...);            // H now ONLY in sled
}
```

After `remove_batch` completes, any concurrent `get(H)` misses memory and enters the destructive-read path.

**Unconditional panic in `HeaderAcceptor::accept`**

When a peer re-sends header H that is `HEADER_VALID` but not `BLOCK_STORED`:

```rust
// headers_process.rs L303-316
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    let header_index = sync_shared
        .get_header_index_view(&self.header.hash(),
            status.contains(BlockStatus::BLOCK_STORED))  // false
        .unwrap_or_else(|| panic!(
            "header {}-{} with HEADER_VALID should exist", ...
        ))
```

`get_header_index_view` with `store_first=false` calls `header_map().get(hash)` first, then falls back to the chain store. If `get` returns `None` (race loser) and H is not yet in the chain store (not `BLOCK_STORED`), the entire call returns `None` and the `unwrap_or_else` panics unconditionally — there is no error path or graceful recovery.

**Existing guards are insufficient**

`backend.is_empty()` only short-circuits when the backend has zero entries; it does not prevent two concurrent callers from racing on the same key. The `MemoryMap`'s `RwLock` only protects the memory layer; it is released before the backend is touched. The `SledBackend` already exposes a non-destructive `get` method (`backend.rs` L19) that would avoid the race entirely.

## Impact Explanation

A panic in a tokio task aborts that task; the node process continues. The concrete effects are: the sync task for the losing peer is killed, the peer connection is dropped, and the node's header sync is disrupted. An attacker controlling two peers can repeatedly trigger this during IBD — when evictions are frequent and the same headers are sent by multiple peers — preventing the node from completing IBD. This maps to **High: Vulnerabilities which could easily crash a CKB node** (the node's sync subsystem is rendered non-functional under sustained attack, effectively halting the node's ability to participate in the network).

## Likelihood Explanation

During IBD the header map routinely exceeds the 256 MB default memory limit, making `limit_memory` evictions frequent (every 5 seconds). The CKB sync protocol explicitly sends the same headers to multiple peers, so two peers concurrently re-sending a known-accepted header is normal behavior, not an adversarial precondition. An attacker controlling two peers can deliberately time re-sends of a known-accepted header (one whose block status is `HEADER_VALID` but not `BLOCK_STORED`) to maximize collision probability. The race window is narrow but repeatable at IBD throughput rates, and the attack requires no special privileges — any two connectable peers suffice.

## Recommendation

Replace the destructive `backend.remove` in `HeaderMapKernel::get` with the already-available non-destructive `backend.get`, and schedule a separate cleanup pass to remove the entry from the backend after it has been safely promoted to memory:

```rust
// Instead of:
if let Some(view) = self.backend.remove(hash) {
    self.memory.insert(view.clone());
    Some(view)
} else {
    None
}

// Use:
if let Some(view) = self.backend.get(hash) {
    self.memory.insert(view.clone());
    // schedule async or deferred removal from backend
    Some(view)
} else {
    None
}
```

Alternatively, protect the entire memory-miss → backend-read → memory-insert sequence with a per-key lock or a coarse `Mutex` to make the promotion atomic.

## Proof of Concept

```
1. Node accepts header H from peer A.
   → H inserted into header_map memory; block status = HEADER_VALID (not BLOCK_STORED).

2. limit_memory fires (memory > limit):
   backend.insert_batch([H])   // H now in sled AND memory
   memory.remove_batch([H])    // H now ONLY in sled

3. Peers B and C concurrently send H again.
   Both enter HeaderAcceptor::accept:
     get_block_status(H) → HEADER_VALID (not BLOCK_STORED)
     get_header_index_view(H, store_first=false)
       → header_map().get(H)
           → memory.get_refresh(H) → None  (H evicted)
           → backend.remove(H):
               Peer B wins → Some(H), re-inserts into memory, returns Some
               Peer C loses → None, returns None
       → chain store fallback → None (not BLOCK_STORED)
     Peer C: .unwrap_or_else(|| panic!("header H with HEADER_VALID should exist"))
     → PANIC: sync task for peer C is killed, connection dropped

4. Repeat step 3 continuously to prevent IBD completion.
