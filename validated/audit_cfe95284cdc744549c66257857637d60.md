Audit Report

## Title
`HeaderAcceptor::version_check` Ignores RFC-0048 Hardfork Gate, Permanently Rejecting Valid Post-Hardfork Headers — (`sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::version_check` unconditionally rejects any block header whose `version` field is non-zero, with no reference to the RFC-0048 (`remove_header_version_reservation_rule`) hardfork gate that was specifically introduced to lift this restriction. The gate function `is_remove_header_version_reservation_rule_enabled` is defined but never called anywhere in the codebase. After RFC-0048 activation (epoch ≥ 12,293 on mainnet), any miner producing a block with a non-zero header version will have that block permanently marked `BLOCK_INVALID` by all current CKB nodes via the sync layer, causing a consensus split.

## Finding Description
`HeaderAcceptor::version_check` in `sync/src/synchronizer/headers_process.rs` (L286–293) hardcodes the version-zero requirement:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    if self.header.version() != 0 {
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
```

When this returns `Err(())`, `accept()` (L346–354) immediately writes `BlockStatus::BLOCK_INVALID` into the shared block-status map for that header hash and returns — permanently. There is no mechanism to reconsider the block.

The hardfork infrastructure to gate this check correctly already exists: `CKB2023::is_remove_header_version_reservation_rule_enabled(epoch)` is generated by the `define_methods!` macro in `util/types/src/core/hardfork/ckb2023.rs` (L66–73). A grep across the entire repository returns exactly one hit for this symbol — the macro definition site — confirming it is never invoked.

The consensus-layer verifiers (`HeaderVerifier::verify` in `verification/src/header_verifier.rs` L30–50, and `BlockVerifier::verify` in `verification/src/block_verifier.rs` L36–48) contain no header version check whatsoever. This means the sync layer is the sole enforcer of the version-zero rule, and it enforces it unconditionally even after RFC-0048 has lifted it.

The exploit path requires valid PoW because `non_contextual_check` (which calls `HeaderVerifier::verify`, including `PowVerifier`) runs before `version_check` in `accept()` (L334–354). A header that fails PoW is rejected at the earlier step and never reaches `version_check`. Therefore the triggering actor must be a miner (or an entity with mining capability).

## Impact Explanation
This is a **consensus deviation** vulnerability. After RFC-0048 activation (epoch ≥ 12,293, already live on mainnet), a miner is permitted by consensus to produce a block with a non-zero header `version`. Any CKB node receiving such a block via the P2P sync `SendHeaders` message will permanently mark it `BLOCK_INVALID` and never reconsider it. Nodes that correctly implement RFC-0048 (or future clients) will accept the block. The affected node silently forks onto a stale chain with no operator-visible error beyond a debug log line. This matches the **Critical** impact class: "Vulnerabilities which could easily cause consensus deviation."

## Likelihood Explanation
RFC-0048 is already active on mainnet. No current miner software produces non-zero-version headers, so the bug is latent today. The required conditions are: (1) a miner updates block-assembly software to set a non-zero header version field (a natural use of the now-unreserved bits), and (2) that block is relayed to a node running the current code. Condition (1) is realistic as the ecosystem evolves post-RFC-0048. A malicious miner with even modest hash rate could deliberately trigger the split by producing a single valid block with `version = 1`, causing all current nodes to permanently reject the canonical chain tip.

## Recommendation
In `HeaderAcceptor::version_check`, consult the hardfork switch before enforcing the version-zero rule. The header's epoch number is available via `self.header.epoch().number()`, and the consensus object is reachable through `self.active_chain`:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    let epoch = self.header.epoch().number();
    let hardfork = self.active_chain.sync_shared()
        .shared()
        .consensus()
        .hardfork_switch();
    if !hardfork.ckb2023.is_remove_header_version_reservation_rule_enabled(epoch)
        && self.header.version() != 0
    {
        state.invalid(Some(ValidationError::Version));
        return Err(());
    }
    Ok(())
}
```

This mirrors the existing pattern used for VM-version and syscall gating in `script/src/types.rs` (L887–897).

## Proof of Concept
1. Confirm RFC-0048 is active: call `get_consensus` RPC and verify `hardfork_features` contains `{"rfc":"0048","epoch_number":"0x3005"}`.
2. Using a test harness (e.g., a modified `ckb-test` integration test), configure a chain that has passed epoch 12,293.
3. Produce a block header with `version = 1` and a valid PoW solution for the current tip.
4. Relay the header to a node under test via a `SendHeaders` P2P message.
5. Observe that `HeaderAcceptor::version_check` fires at `sync/src/synchronizer/headers_process.rs:287`, `BLOCK_INVALID` is written for that hash, and the node never syncs the block — even though `HeaderVerifier` and `BlockVerifier` contain no version check and would accept it at the consensus layer.
6. Confirm the node is stuck on a stale fork while a node with the patched `version_check` continues to follow the canonical chain.
