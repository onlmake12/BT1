### Title
Division by Zero in `deriveCrossRate` When `price2 == 0` — (`File: target_chains/ethereum/sdk/solidity/PythUtils.sol`)

---

### Summary

`PythUtils.deriveCrossRate` guards against negative prices but does not guard against `price2 == 0`. When `price2` is zero, the function passes `uint64(0)` as the denominator to `Math.mulDiv`, which triggers an unrecoverable panic revert instead of a clean, descriptive error.

---

### Finding Description

`deriveCrossRate` computes the cross-rate `price1 / price2` scaled by exponents. Its only input validation for `price2` is:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
``` [1](#0-0) 

`price2 == 0` passes this check. The function then casts `price2` to `uint64(0)` and uses it as the denominator in `Math.mulDiv`:

```solidity
// deltaExponent > 0 branch:
result = Math.mulDiv(
    uint64(price1),
    10 ** uint64(deltaExponent),
    uint64(price2)          // <-- denominator = 0
);

// deltaExponent <= 0 branch:
result = Math.mulDiv(
    uint64(price1),
    1,
    10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // <-- denominator = 0
);
``` [2](#0-1) 

Inside `Math.mulDiv`, when `denominator == 0`:

- Fast path (`high == 0`): `return low / denominator` — Solidity reverts with a division-by-zero panic.
- Slow path (`high != 0`): explicitly calls `panic(DIVISION_BY_ZERO)`. [3](#0-2) 

Both paths produce an opaque panic revert (`0x4e487b71` with code `0x12`) rather than a `PythErrors` custom error. The NatSpec for `deriveCrossRate` states it "will work for full range of int64 prices," but `price2 = 0` is within `int64` range and is not handled. [4](#0-3) 

---

### Impact Explanation

Any on-chain integrator that calls `deriveCrossRate` with a Pyth price feed whose current price is `0` will receive an unexpected panic revert. Because the revert is a low-level panic rather than a named `PythErrors` custom error, integrating contracts that catch specific errors (e.g., `try/catch` on `PythErrors.NegativeInputPrice`) will not catch it, causing the entire calling transaction to revert. This is a denial-of-service for any protocol that relies on cross-rate computation during periods when a denominator asset's Pyth price is reported as zero (e.g., newly listed assets, halted feeds, or extreme market events).

---

### Likelihood Explanation

Pyth price feeds can legitimately return `price == 0` for newly listed assets before sufficient publisher data is aggregated, or during feed initialization. A user or keeper that calls `deriveCrossRate` with live Pyth data without pre-filtering zero prices will trigger this revert. The function is `public pure`, so it is directly callable by any external account or contract without any privileged access. [5](#0-4) 

---

### Recommendation

Add an explicit zero-price guard for `price2` alongside the existing negativity check:

```solidity
if (price1 < 0 || price2 <= 0) {
    revert PythErrors.NegativeInputPrice(); // or introduce ZeroInputPrice error
}
```

Alternatively, introduce a dedicated `PythErrors.ZeroInputPrice` error for clarity, since `price2 == 0` is semantically distinct from a negative price. [6](#0-5)

            int32(-6),         // expo2

        // Reverts with panic 0x12 (division by zero) instead of a PythErrors custom error
    }
