Audit Report

## Title
Missing Duplicate Index Check in `GetBlockTransactionsProcess::execute()` Enables Bandwidth Amplification — (File: `sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute()` validates only the count of requested transaction indexes but never checks for duplicates within the list. A peer can craft a `GetBlockTransactions` message with `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) copies of the same index, forcing the node to serialize and transmit the same transaction data 32,767 times in a single `BlockTransactions` reply. The sibling handler `GetTransactionsProcess` already contains an explicit duplicate-rejection guard, confirming this is an unintentional omission.

## Finding Description
In `sync/src/relayer/get_block_transactions_process.rs`, `execute()` performs two length-only guards: [1](#0-0) 

After those guards pass, the handler iterates the raw `indexes` list with no deduplication: [2](#0-1) 

Every element of `indexes` — including duplicates — produces one entry in the response `Vec`, which is then serialized and sent back to the peer: [3](#0-2) 

The length guard uses `>` (strictly greater than), so a list of exactly 32,767 identical indexes passes. By contrast, `GetTransactionsProcess::execute()` explicitly detects duplicates via a `HashSet` and returns `StatusCode::RequestDuplicate` before doing any work: [4](#0-3) 

`MAX_RELAY_TXS_NUM_PER_BATCH` is defined as 32,767: [5](#0-4) 

## Impact Explanation
The amplification ratio is `MAX_RELAY_TXS_NUM_PER_BATCH` × (serialized size of the targeted transaction). For a transaction of even 500 bytes, a single crafted request produces a ~16 MB outbound response while the attacker sends only ~65 KB (~246× amplification). Repeated from multiple simultaneous connections, this constitutes targeted outbound bandwidth exhaustion against a specific node. This matches the allowed High impact: **Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points).

## Likelihood Explanation
The entry path requires only a completed P2P handshake — no special privilege, key, or hash-power. The crafted message is syntactically valid and passes all existing guards. The attack is trivially reproducible with a single crafted packet and is repeatable at will from any number of connections simultaneously.

## Recommendation
Add a duplicate-index check immediately after the existing length guards, mirroring the pattern already used in `GetTransactionsProcess`:

```rust
use std::collections::HashSet;

let indexes = get_block_transactions.indexes();
let mut seen: HashSet<u32> = HashSet::new();
for idx in indexes.iter() {
    if !seen.insert(Into::<u32>::into(idx)) {
        return StatusCode::ProtocolMessageIsMalformed
            .with_context("Duplicate index in GetBlockTransactions");
    }
}

let uncle_indexes = get_block_transactions.uncle_indexes();
let mut seen_uncle: HashSet<u32> = HashSet::new();
for idx in uncle_indexes.iter() {
    if !seen_uncle.insert(Into::<u32>::into(idx)) {
        return StatusCode::ProtocolMessageIsMalformed
            .with_context("Duplicate uncle index in GetBlockTransactions");
    }
}
```

## Proof of Concept
1. Connect to a CKB node as a peer and complete the identify/sync handshake.
2. Obtain any block hash `H` the node has stored.
3. Craft a `GetBlockTransactions` message: `block_hash = H`, `indexes = [0; 32767]` (all pointing to the always-present coinbase transaction).
4. Send the message over the relay protocol.
5. Observe the node replies with a `BlockTransactions` message containing 32,767 serialized copies of the coinbase transaction. No ban or rejection is triggered because the list length equals `MAX_RELAY_TXS_NUM_PER_BATCH`, not exceeds it.

A unit test analogous to `test_duplicate` in `sync/src/relayer/tests/get_transactions_process.rs` can confirm the missing rejection: [6](#0-5) 

Construct a `GetBlockTransactions` with repeated index `0`, call `execute()`, and assert it currently returns `Status::ok()` instead of `StatusCode::ProtocolMessageIsMalformed`.
