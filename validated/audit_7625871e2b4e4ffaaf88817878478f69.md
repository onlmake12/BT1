### Title
Governance Sequence Gap Allows Permanent Skip of Critical Signer-Removal Messages — (`lazer/contracts/sui/sources/governance.move`)

### Summary

`governance::process_incoming` enforces only a **greater-than** check on the incoming VAA sequence number, not a **consecutive** check. An unprivileged relayer can submit a higher-sequence VAA (e.g., an upgrade at seq=10) while withholding lower-sequence VAAs (seq=6..9), permanently making those intermediate messages unprocessable. If the skipped messages include signer-removal governance actions, compromised signers remain active and can produce malicious Lazer price updates consumed by downstream protocols.

---

### Finding Description

The guard in `process_incoming` is:

```move
assert!(self.seen_sequence < sequence, EOldSequenceNumber);
self.seen_sequence = sequence;
``` [1](#0-0) 

The condition `self.seen_sequence < sequence` accepts **any** future sequence number, not just the immediately next one. After `seen_sequence` is advanced to `10`, any VAA with sequence ≤ 10 will fail the `< sequence` assertion and can never be applied.

Both `upgrade` and `update_trusted_signer` in `actions.move` are declared `public fun` (not `entry fun`), so any caller — including an unprivileged relayer — can invoke them with a valid Wormhole VAA: [2](#0-1) [3](#0-2) 

The full call chain is: `upgrade(state, vaa)` → `state.unwrap_ptgm(...)` → `self.governance.process_incoming(vaa)`. [4](#0-3) 

---

### Impact Explanation

If governance emits (in order):
- seq=6..9: `UpdateTrustedSigner` messages removing a compromised signer
- seq=10: `UpgradeSuiLazerContract`

A malicious relayer relays seq=10 first. `seen_sequence` advances to 10. The signer-removal messages at seq=6..9 are now permanently blocked. The compromised signer remains in `State.trusted_signers`. [5](#0-4) 

Any Lazer price update signed by the compromised key will pass signature verification, enabling fabricated prices. Protocols that consume Lazer prices and act on them (e.g., lending, derivatives) can be drained.

---

### Likelihood Explanation

All Wormhole VAAs are publicly observable. A relayer only needs to selectively submit a higher-sequence VAA before the intermediate ones are processed. No privileged access, leaked keys, or governance compromise is required — only the ability to call a `public fun` with a valid VAA that already exists on-chain.

---

### Recommendation

Replace the `<` check with a strict consecutive-sequence check:

```move
assert!(self.seen_sequence + 1 == sequence, ESequenceNotConsecutive);
self.seen_sequence = sequence;
``` [1](#0-0) 

This enforces that every governance message is processed in exact order, with no gaps.

---

### Proof of Concept

1. Initialize state with `seen_sequence = 5`.
2. Construct (or obtain from Wormhole) a valid `UpdateTrustedSigner` VAA at seq=6 that removes signer `S`, and a valid `UpgradeSuiLazerContract` VAA at seq=10.
3. Call `actions::upgrade(state, vaa_seq10)` — passes because `5 < 10`.
4. `seen_sequence` is now `10`.
5. Attempt `actions::update_trusted_signer(state, vaa_seq6)` — aborts with `EOldSequenceNumber` because `10 < 6` is false.
6. Assert `state.trusted_signers` still contains signer `S`.
7. Contrast with processing seq=6..10 in order: `S` is absent from `trusted_signers`.

The differential confirms the invariant break: skipped signer-removal messages are not reflected in the final trusted-signer set.
