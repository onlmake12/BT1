### Title
Incorrect `ancestors_count` Accounting in `check_and_record_ancestors` Causes Over-Eviction of Transactions — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `pool_map.rs`, the function `check_and_record_ancestors` attempts to evict "cell-ref parent" transactions to bring a new transaction's ancestor count within the `max_ancestors_count` limit. The eviction loop decrements the `ancestors_count` counter by exactly 1 per evicted entry, but the underlying removal call `remove_entry_and_descendants` removes the target transaction **and all its descendants**. When some of those descendants are also ancestors of the incoming transaction, the actual ancestor count drops by more than 1, but the loop counter does not reflect this. The loop therefore continues evicting additional cell-ref parents that are no longer necessary to remove, causing legitimate transactions to be unnecessarily expelled from the pool.

---

### Finding Description

The relevant code is in `check_and_record_ancestors`:

```rust
// pool_map.rs ~L595
let mut ancestors_count = ancestors.len() + 1;

// ...
while ancestors_count > self.max_ancestors_count {
    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id);
        ancestors_count = ancestors_count.saturating_sub(1); // always -1
        parents.remove(next_id);
        evicted.extend(removed);
    } else {
        break;
    }
}
```

`remove_entry_and_descendants(next_id)` removes `next_id` **plus every descendant** of `next_id` currently in the pool. If any of those descendants are also ancestors of the incoming transaction, the true remaining ancestor count falls by more than 1. However, `ancestors_count` is only decremented by 1 per loop iteration. The loop termination condition `ancestors_count > self.max_ancestors_count` therefore remains `true` for extra iterations, causing the loop to evict additional cell-ref parents that are no longer needed.

The post-loop `assert!(ancestors.len() < self.max_ancestors_count)` (line 636) recalculates from scratch and will pass, but by that point more transactions have already been evicted than the protocol required. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

A transaction sender can craft a transaction whose ancestor graph contains cell-ref parents that themselves have descendants also in the ancestor chain. When the pool processes this transaction, the eviction loop removes more pool entries than the protocol requires. Legitimate, fee-paying transactions belonging to other users are silently expelled from the pool with a `Reject::Full` or `Reject::Invalidated` reason, even though the pool had sufficient capacity to accommodate them after the minimum necessary evictions. This constitutes a targeted, low-cost denial-of-service against pending transactions. [3](#0-2) 

---

### Likelihood Explanation

The scenario requires that at least one cell-ref parent in the ancestor graph has one or more descendants that are also ancestors of the incoming transaction. This is a realistic pool state: a chain of transactions where an intermediate transaction is both a cell-dep consumer (making it a cell-ref parent) and an ancestor of the new transaction. Any unprivileged transaction sender can submit a carefully ordered sequence of transactions to construct this state before submitting the triggering transaction. [4](#0-3) 

---

### Recommendation

After calling `remove_entry_and_descendants(next_id)`, decrement `ancestors_count` by the actual number of removed entries that were ancestors of the incoming transaction, not by a fixed 1. One approach is to intersect the `removed` set with the original `ancestors` set and subtract the intersection size:

```rust
let removed = self.remove_entry_and_descendants(next_id);
let removed_ancestor_count = removed.iter()
    .filter(|e| ancestors.contains(&e.proposal_short_id()))
    .count();
ancestors_count = ancestors_count
    .saturating_sub(removed_ancestor_count.max(1));
```

Alternatively, recompute `ancestors_count` from scratch after each eviction using `calc_relation_ids`, which is already done after the loop. [5](#0-4) 

---

### Proof of Concept

**Setup:** Suppose `max_ancestors_count = 3`.

1. Submit tx A (no ancestors).
2. Submit tx B, child of A, which is also a cell-dep consumer of some cell C (making B a cell-ref parent).
3. Submit tx D, child of B.
4. Submit tx E, child of D, which also consumes cell C as a dep.

Now submit tx F, child of E. F's ancestors are {A, B, D, E}, so `ancestors_count = 5`. `cell_ref_parents = {B}` (B is a cell-ref parent). `ancestors_count - cell_ref_parents.len() = 4 > 3`, so... actually let me adjust.

**Adjusted Setup:** `max_ancestors_count = 25`. Suppose the new tx has 27 ancestors, 3 of which are cell-ref parents (C1, C2, C3). C1 has a descendant D1 that is also an ancestor of the new tx.

- `ancestors_count = 28`, limit = 25, need to reduce by 3.
- Iteration 1: evict C1 + D1 (both ancestors). Actual ancestor count drops to 26. But `ancestors_count` becomes 27.
- Iteration 2: evict C2. Actual drops to 25. `ancestors_count` becomes 26.
- Iteration 3: evict C3 (unnecessary — actual count is already 25 ≤ limit). `ancestors_count` becomes 25, loop exits.

C3 was evicted unnecessarily. Any transactions depending on C3 are also cascade-evicted via `remove_entry_and_descendants`. [6](#0-5)
