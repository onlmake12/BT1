Audit Report

## Title
Stale `total_tx_size`/`total_tx_cycles` Overwrite After In-Flight Eviction in `add_entry` Inflates Pool Accounting — (File: tx-pool/src/component/pool_map.rs)

## Summary
`PoolMap::add_entry` snapshots `self.total_tx_size + entry.size` into a local variable before calling `check_and_record_ancestors`, which can internally evict transactions via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, correctly decrementing the live counters. After eviction, `add_entry` unconditionally overwrites the live counters with the stale pre-eviction snapshot, permanently inflating them by the aggregate size/cycles of every evicted transaction. The inflated counters cause `limit_size` to over-evict legitimate transactions with `Reject::Full` even when the pool has physical capacity.

## Finding Description
In `add_entry` (lines 200–221 of `tx-pool/src/component/pool_map.rs`):

1. **Lines 210–211**: `updated_stat_for_add_tx` is a pure read (`&self`) that returns `self.total_tx_size + entry.size` and `self.total_tx_cycles + entry.cycles` as a local snapshot. No state is mutated.

2. **Line 213**: `check_and_record_ancestors` is called. When `ancestors_count > max_ancestors_count` but `ancestors_count.saturating_sub(cell_ref_parents.len()) <= max_ancestors_count` (lines 603–625), it enters the eviction branch and calls `self.remove_entry_and_descendants(next_id)` in a loop. `remove_entry_and_descendants` calls `remove_entry` for each removed tx, which calls `self.update_stat_for_remove_tx(entry.inner.size, entry.inner.cycles)` (line 247), correctly decrementing `self.total_tx_size` and `self.total_tx_cycles` in place.

3. **Lines 218–219**: `add_entry` unconditionally assigns `self.total_tx_size = total_tx_size` and `self.total_tx_cycles = total_tx_cycles`, overwriting the correctly-decremented live values with the stale pre-eviction snapshot. The net effect: `self.total_tx_size` becomes `pre_eviction_total + entry.size` instead of the correct `pre_eviction_total - evicted_sizes + entry.size`, inflating it by exactly the sum of evicted transaction sizes.

4. `limit_size` (pool.rs line 298) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, so the inflated counter causes it to evict additional legitimate pending/proposed transactions with `Reject::Full`.

Existing guards do not prevent this: `updated_stat_for_add_tx` only checks for integer overflow on the pre-eviction total; it does not account for post-eviction state. `update_stat_for_remove_tx` correctly handles its own decrement but its result is immediately discarded by the overwrite at lines 218–219.

## Impact Explanation
An attacker can repeatedly trigger the eviction branch in `check_and_record_ancestors` to progressively inflate `total_tx_size`. Each trigger causes `limit_size` to over-evict legitimate pending/proposed transactions. Over repeated triggering, the effective pool capacity shrinks toward zero, preventing legitimate transactions from entering the mempool and degrading transaction throughput across the network. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The eviction branch requires a transaction whose ancestor count exceeds `max_ancestors_count` (default 1,000) but whose excess ancestors are all cell-dep-referencing parents. An unprivileged user reachable via `send_transaction` RPC or P2P relay can craft such a chain. Building a 1,000-tx chain has a one-time fee cost, but each successful trigger permanently inflates the counters, so the attacker's investment compounds over time. The bug also fires in non-adversarial conditions whenever organic chains hit this path.

## Recommendation
Remove the pre-computed snapshot and instead apply the addition directly to `self.total_tx_size`/`self.total_tx_cycles` after `check_and_record_ancestors` completes, so the increment is applied to the post-eviction state:

```rust
// Remove lines 210-211 (the pre-snapshot)
// Retain updated_stat_for_add_tx as a pre-flight overflow check only, discarding its return value
self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
// After all mutations, replace lines 218-219 with:
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

This ensures the increment is applied to the post-eviction state rather than the pre-eviction snapshot.

## Proof of Concept
1. Build a chain of 999 transactions `T1 → T2 → … → T999` where each `Ti` is a cell-dep parent of `T1000`, making `T1000`'s ancestor count = 1,000 via cell-dep references.
2. Submit `T1000`. `check_and_record_ancestors` detects `ancestors_count = 1,001 > max_ancestors_count = 1,000` but `1,001 - 999 = 2 ≤ 1,000`, so it evicts cell-dep parents (e.g., `T999`, size S) via `remove_entry_and_descendants`, correctly decrementing `self.total_tx_size` by S.
3. `add_entry` lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot, inflating it by S.
4. `limit_size` is called immediately after; it sees `total_tx_size > max_tx_pool_size` and evicts one additional legitimate transaction.
5. Repeat: each submission that triggers the eviction branch inflates `total_tx_size` by the evicted size, causing `limit_size` to cascade-evict additional legitimate transactions until the pool is effectively unusable.
