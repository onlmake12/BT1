Audit Report

## Title
Unvalidated `from` Field Bypasses `forward_rate_limiter`, Enabling Gossip Amplification — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `forward_rate_limiter` in the HolePunching protocol is keyed on `(content.from, content.to, item_id)`, where `content.from` is parsed directly from the attacker-controlled message body and never validated against the actual session's peer ID. An attacker with a single TCP connection can send 30 `ConnectionRequest` messages per second (the outer rate limit), each with a distinct random `from` PeerID, causing every message to hit a fresh rate-limiter bucket and pass. When `to` is unknown, each message triggers a `filter_broadcast` to `sqrt(N)` downstream peers, which recursively repeat the same logic up to `MAX_HOPS = 6` levels deep, producing unbounded network amplification from a single connection.

## Finding Description

**Outer rate limiter** is correctly keyed on `(session_id, msg.item_id())` with a quota of 30/s: [1](#0-0) [2](#0-1) 

This caps the attacker to 30 `ConnectionRequest` messages per second per session.

**Inner `forward_rate_limiter`** is keyed on `(content.from, content.to, self.msg_item_id)` with a quota of 1/s: [3](#0-2) [4](#0-3) 

The intent is to allow at most one forward per `(from, to)` pair per second. However, `content.from` is parsed directly from the message body: [5](#0-4) 

There is no assertion anywhere in `execute()` that `content.from` matches the actual session's peer ID: [6](#0-5) 

An attacker who sends 30 messages per second, each with a unique random `from` PeerID, causes each message to hit a brand-new `(from_i, to, item_id)` bucket in the `forward_rate_limiter`, so all 30 pass the inner limiter.

**Gossip broadcast path** — when `to` is not found in the peer registry, the node broadcasts to `sqrt(N)` peers: [7](#0-6) 

**`forward_request`** adds the current node to the route and decrements `max_hops` by 1: [8](#0-7) 

The route check at L128 (`content.route.contains(self_peer_id)`) only prevents a single node from appearing twice in the forwarding chain of one specific message. Since each of the 30 attacker messages has a distinct `from`, each is treated as an independent message at every intermediate node — the route check provides no cross-message deduplication. Each intermediate node's own `forward_rate_limiter` is also bypassed by the same unique-`from` trick, so the amplification cascades up to `MAX_HOPS = 6` hops. [9](#0-8) [10](#0-9) 

**Memory pressure** — `retain_recent()` is only called on disconnect, so the `forward_rate_limiter` `HashMapStateStore` accumulates one entry per unique `(from, to, item_id)` tuple indefinitely during a long-lived session. The `notify` timer does not call `retain_recent()` on either rate limiter: [11](#0-10) [12](#0-11) 

## Impact Explanation

A single attacker connection generates 30 × sqrt(N) downstream messages per second at hop 1. Each of those sqrt(N) intermediate nodes receives 30 messages with distinct `from` values, bypasses its own `forward_rate_limiter`, and re-broadcasts to its own sqrt(N) peers. This cascades up to `MAX_HOPS = 6` levels. Even at 2–3 hops with a modest N=100 (sqrt=10), a single attacker connection produces 30 × 10 × 10 = 3,000 messages/second across the network. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The attack requires only a standard P2P connection — no special privileges, no PoW, no cryptographic keys. Generating random PeerIDs is trivial (as demonstrated by `PeerId::random()` used in the codebase's own tests). The exploit path is entirely within production code, locally reproducible with a single modified CKB peer, and repeatable indefinitely.

## Recommendation

1. **Validate `from` against the session's actual peer ID** before the `forward_rate_limiter` check. Retrieve the session's peer ID from the peer registry and reject any message where `content.from != session_peer_id`. This collapses all messages from one session to a single `from` value, making the outer `rate_limiter` the sole effective cap.

2. **Re-key `forward_rate_limiter` on `(session_id, to, item_id)`** instead of `(from, to, item_id)`, since `session_id` (a `PeerIndex`) is not attacker-controlled.

3. **Call `retain_recent()` periodically** (e.g., in the `notify` timer at `CHECK_INTERVAL = 5 minutes`) rather than only on disconnect, to bound `forward_rate_limiter` map growth.

## Proof of Concept

```rust
// Run against a local CKB devnet with N peers connected
let session = connect_to_victim_node(); // standard P2P handshake
loop {
    for _ in 0..30 {
        let from = PeerId::random();   // unique per message — bypasses forward_rate_limiter
        let to   = PeerId::random();   // unknown to the network — triggers filter_broadcast
        let msg  = build_connection_request(from, to, valid_tcp_listen_addr, max_hops=6);
        session.send(msg);
    }
    sleep(Duration::from_secs(1));
}
// Instrument: count HolePunching messages received by all other N peers.
// Expected: total received > 30 * sqrt(N) per second at hop 1,
//           cascading further at each intermediate node up to max_hops=6.
```

Verification steps:
1. Start a local devnet with N ≥ 9 nodes fully meshed.
2. Run the PoC from a single external connection to one node.
3. Add counters to `HolePunching::received` on all other nodes.
4. Assert that the total message count across all nodes exceeds `30 * sqrt(N)` per second, confirming O(sqrt(N)) amplification per attacker message and cascading across hops.

    pub(crate) async fn execute(mut self) -> Status {
        let content = match RequestContent::try_from(&self.message) {
            Ok(content) => content,
            Err(status) => return status,
        };
        if content.listen_addrs.len() > ADDRS_COUNT_LIMIT || content.listen_addrs.is_empty() {
            return StatusCode::InvalidListenAddrLen
                .with_context("the listen address count is too large or empty");
        }

        if content.max_hops > MAX_HOPS {
            return StatusCode::InvalidMaxTTL.into();
        }
        if content.route.len() > MAX_HOPS as usize {
            return StatusCode::InvalidRoute.with_context("the route length is too long");
        }

        let self_peer_id = self.protocol.network_state.local_peer_id();
        if content.route.contains(self_peer_id) {
            return StatusCode::Ignore.with_context("the message is passed, ignore it");
        }

        if self
            .protocol
            .forward_rate_limiter
            .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
            .is_err()
        {
            debug!(
                "from: {}, to {}, item_name: {}, rate limit is reached",
                content.from, content.to, "ConnectionRequest",
            );
            return StatusCode::TooManyRequests.with_context("ConnectionRequest");
        }

        if self_peer_id == &content.to {
            self.respond_delivered(content.from, &content.to, content.listen_addrs)
                .await
        } else if content.max_hops == 0u8 {
            StatusCode::ReachedMaxHops.into()
        } else {
            self.forward_message(self_peer_id, &content.to).await
        }
    }

pub(crate) fn forward_request(
    request: packed::ConnectionRequestReader<'_>,
    current_id: &PeerId,
) -> packed::ConnectionRequest {
    let max_hops: u8 = request.max_hops().into();
    let message = request.to_entity();
    let new_route = message
        .route()
        .as_builder()
        .push(current_id.as_bytes())
        .build();
    message
        .as_builder()
        .max_hops(max_hops.saturating_sub(1))
        .route(new_route)
        .build()
}
