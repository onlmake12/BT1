### Title
Unenforced `max_request_body_size` on HTTP/WS RPC Server and Unbounded Default Batch Request Size Allow RPC-Caller DoS - (File: `rpc/src/server.rs`)

### Summary
The CKB RPC server has two compounding input-size control failures. First, the `max_request_body_size` configuration value is read from `ckb.toml` but is **never wired into the axum HTTP/WebSocket router** as a body-limit middleware layer, so the configured limit is silently ignored. Second, `rpc_batch_limit` defaults to `None` (no limit), which is explicitly documented in the default config. Together, an RPC caller can submit an arbitrarily large batch of JSON-RPC calls over HTTP, causing memory exhaustion and CPU saturation on the node.

### Finding Description

**Root cause 1 — `max_request_body_size` not applied to HTTP/WS server**

`RpcServer::new()` receives the full `RpcConfig` (which includes `max_request_body_size`), but when it calls `Self::start_server()`, the body-size value is never passed and never applied: [1](#0-0) 

`start_server()` builds an axum `Router` with only `CorsLayer`, `TimeoutLayer`, and `Extension` layers — no `DefaultBodyLimit` or `RequestBodyLimitLayer` is added: [2](#0-1) 

The config field exists and is documented: [3](#0-2) 

But a search of all `rpc/src/` files confirms `max_request_body_size` is referenced only in the test setup, never in the production server code. The TCP server independently hardcodes 2 MB via `LinesCodec::new_with_max_length(2 * 1024 * 1024)`, also ignoring the config value: [4](#0-3) 

**Root cause 2 — `rpc_batch_limit` defaults to `None` (unlimited)**

The default production config explicitly documents no default limit: [5](#0-4) 

`rpc_batch_limit` is `Option<usize>`: [6](#0-5) 

In `RpcServer::new()`, `JSONRPC_BATCH_LIMIT` is only initialized when the option is `Some`: [7](#0-6) 

In `handle_jsonrpc`, the batch-size guard is only evaluated when `JSONRPC_BATCH_LIMIT.get()` returns `Some` — if the option was never set, the guard is entirely skipped and the batch is processed unconditionally: [8](#0-7) 

Each call in the batch is dispatched through `io.handle_call()` sequentially via an async stream, holding all in-flight state in memory simultaneously: [9](#0-8) 

### Impact Explanation

An RPC caller (the attacker role is explicitly in scope) who can reach the RPC port — either locally or on a node where the operator has exposed the port — can:

1. Send a single HTTP POST with a JSON array containing tens of thousands of JSON-RPC calls. Because no body-size limit is enforced on the HTTP server, the request body is unbounded.
2. Because `rpc_batch_limit` is `None` by default, all calls are accepted and dispatched.
3. Each call allocates memory for its request and response. Expensive calls (e.g., `get_block`, `send_transaction`) also consume CPU.
4. Result: memory exhaustion, CPU saturation, or effective denial of service for all other RPC clients and internal node operations that share the same runtime.

### Likelihood Explanation

- The RPC port defaults to `127.0.0.1:8114` (localhost only), but the config explicitly warns operators may expose it, and many do for dApp/wallet integrations.
- No authentication is required on the RPC endpoint.
- The `rpc_batch_limit` being `None` by default is the shipping default for all nodes that do not explicitly set it — the config comment confirms this is the intended default behavior, making it a systemic issue rather than a misconfiguration.
- Exploitation requires only a single HTTP POST with a crafted JSON array — trivial to automate.

### Recommendation

1. **Enforce `max_request_body_size` in `start_server()`**: Pass the value from `RpcConfig` into `start_server()` and add `.layer(DefaultBodyLimit::max(config.max_request_body_size))` to the axum router before serving.
2. **Set a safe default for `rpc_batch_limit`**: Change `rpc_batch_limit: Option<usize>` to have a hardcoded safe default (e.g., 2000) that is applied even when the operator does not configure it explicitly. The current behavior of "no limit unless configured" is unsafe.
3. **Apply the config body limit to the TCP server**: Replace the hardcoded `2 * 1024 * 1024` in `LinesCodec::new_with_max_length` with the configured `max_request_body_size`.

### Proof of Concept

```bash
# Generate a batch of 50,000 cheap-but-valid JSON-RPC calls
python3 -c "
import json, sys
batch = [{'jsonrpc':'2.0','id':i,'method':'get_tip_block_number','params':[]} for i in range(50000)]
sys.stdout.write(json.dumps(batch))
" > big_batch.json

# Send to a node with default config (rpc_batch_limit not set)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  --data-binary @big_batch.json | wc -c
```

With default config (`rpc_batch_limit` not set, `max_request_body_size` not enforced on the HTTP layer), the node accepts and processes all 50,000 calls, allocating memory for every response and blocking the RPC worker pool for the duration. Repeating this in a loop from a local process (or a remote process if the port is exposed) exhausts node memory and degrades or halts RPC service.

    fn start_server(
        rpc: &Arc<MetaIoHandler<Option<Session>>>,
        address: String,
        handler: Handle,
        enable_websocket: bool,
    ) -> Result<SocketAddr, AnyError> {
        let stream_config = StreamServerConfig::default()
            .with_keep_alive(true)
            .with_pipeline_size(4)
            .with_shutdown(async move {
                new_tokio_exit_rx().cancelled().await;
            });

        // HTTP and WS server.
        let post_router = post(handle_jsonrpc::<Option<Session>>);
        let get_router = if enable_websocket {
            get(handle_jsonrpc_ws::<Option<Session>>)
        } else {
            get(get_error_handler)
        };
        let method_router = post_router.merge(get_router);

        let app = Router::new()
            .route("/", method_router.clone())
            .route("/{*path}", method_router)
            .route("/ping", get(ping_handler))
            .layer(Extension(Arc::clone(rpc)))
            .layer(CorsLayer::permissive())
            .layer(TimeoutLayer::with_status_code(
                StatusCode::REQUEST_TIMEOUT,
                Duration::from_secs(30),
            ))
            .layer(Extension(stream_config));

        let (tx_addr, rx_addr) = tokio::sync::oneshot::channel::<SocketAddr>();

        handler.spawn(async move {
            let listener = tokio::net::TcpListener::bind(
                &address
                    .to_socket_addrs()
                    .expect("config listen_address parsed")
                    .next()
                    .expect("config listen_address parsed"),
            )
            .await
            .unwrap();
            let server = axum::serve(listener, app.into_make_service());

            let _ = tx_addr.send(server.local_addr().unwrap());
            let graceful = server.with_graceful_shutdown(async move {
                new_tokio_exit_rx().cancelled().await;
            });
            drop(graceful.await);
        });

        let rx_addr = handler.block_on(rx_addr)?;
        Ok(rx_addr)
    }
