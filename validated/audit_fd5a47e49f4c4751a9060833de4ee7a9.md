### Title
Missing Deduplication of Indexes in `GetBlockTransactions` Handler Enables Bandwidth Amplification - (File: `sync/src/relayer/get_block_transactions_process.rs`)

### Summary

`GetBlockTransactionsProcess::execute` iterates over peer-supplied `indexes` and `uncle_indexes` without checking for duplicates. An unprivileged peer can send a `GetBlockTransactions` message whose `indexes` list contains the same transaction index repeated up to `MAX_RELAY_TXS_NUM_PER_BATCH` times, causing the responding node to serialize and transmit the same transaction data that many times in a single `BlockTransactions` reply.

### Finding Description

When a peer sends a `GetBlockTransactions` relay message, `GetBlockTransactionsProcess::execute` performs two bound checks: [1](#0-0) 

Neither check rejects duplicate values in the lists. The handler then iterates directly over the raw `indexes` vector: [2](#0-1) 

Because `filter_map` is applied to the unsanitised iterator, every occurrence of a repeated index produces another copy of the same `TransactionView` in the `transactions` vector. The same pattern applies to `uncle_indexes`. The resulting `BlockTransactions` message is then sent back to the requesting peer: [3](#0-2) 

The `GetBlockTransactions` wire format explicitly allows arbitrary `Uint32Vec` values for both fields: [4](#0-3) 

By contrast, the `GetBlockProposal` handler **does** reject duplicate proposal IDs, demonstrating that the pattern is known and applied inconsistently across relay message handlers.

### Impact Explanation

An attacker who knows any valid `(block_hash, transaction_index)` pair — both of which are public once a compact block is relayed — can craft a `GetBlockTransactions` message with `MAX_RELAY_TXS_NUM_PER_BATCH` copies of the same index. The node will:

1. Fetch and clone the same `TransactionView` up to `MAX_RELAY_TXS_NUM_PER_BATCH` times.
2. Serialize all copies into a single outbound `BlockTransactions` message.
3. Transmit the inflated message to the attacker.

This is a **bandwidth amplification** attack: a small inbound request triggers a response whose size is proportional to `MAX_RELAY_TXS_NUM_PER_BATCH × tx_size`. For large transactions (e.g., those carrying witness data), the amplification factor is significant. Repeated from multiple connections, this can exhaust a node's outbound bandwidth and CPU serialization budget, degrading relay performance for legitimate peers.

The same amplification applies to `uncle_indexes`, bounded by `max_uncles_num`.

### Likelihood Explanation

The attack requires only:
- A TCP connection to the target node (no authentication).
- Knowledge of a recent block hash (broadcast publicly via compact blocks).
- Knowledge of at least one valid transaction index within that block (derivable from the compact block's `short_ids` field).

All three are trivially available to any peer on the network. No special privilege, key material, or majority hash power is needed.

### Recommendation

Deduplicate `indexes` and `uncle_indexes` before iterating, analogous to how `GetBlockProposalProcess` rejects duplicate proposal IDs. For example:

```rust
// In GetBlockTransactionsProcess::execute, after the length checks:
let mut seen_indexes = std::collections::HashSet::new();
for i in get_block_transactions.indexes().iter() {
    if !seen_indexes.insert(Into::<u32>::into(i)) {
        return StatusCode::ProtocolMessageIsMalformed
            .with_context("Duplicate transaction index in GetBlockTransactions");
    }
}
let mut seen_uncle_indexes = std::collections::HashSet::new();
for i in get_block_transactions.uncle_indexes().iter() {
    if !seen_uncle_indexes.insert(Into::<u32>::into(i)) {
        return StatusCode::ProtocolMessageIsMalformed
            .with_context("Duplicate uncle index in GetBlockTransactions");
    }
}
```

### Proof of Concept

1. Connect to a CKB node as a relay peer.
2. Wait to receive a `CompactBlock` message; record its `block_hash` and note that transaction index `1` exists (any non-cellbase index).
3. Craft a `GetBlockTransactions` message:
   ```
   block_hash: <observed block hash>
   indexes:    [1, 1, 1, ..., 1]   // MAX_RELAY_TXS_NUM_PER_BATCH entries, all identical
   uncle_indexes: []
   ```
4. Send the message to the node.
5. Observe that the node replies with a `BlockTransactions` message containing `MAX_RELAY_TXS_NUM_PER_BATCH` identical copies of transaction 1, with a response body far larger than the request.

The root cause is in `sync/src/relayer/get_block_transactions_process.rs` lines 61–78, where the `indexes` iterator is consumed without a prior deduplication step, directly analogous to the missing `seen_assets` check in the Blend `create_interest_auction_data` function. [5](#0-4)

    pub async fn execute(self) -> Status {
        let shared = self.relayer.shared();
        {
            let get_block_transactions = self.message;
            if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "Indexes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    get_block_transactions.indexes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
            if get_block_transactions.uncle_indexes().len() > shared.consensus().max_uncles_num() {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "UncleIndexes count({}) > consensus max_uncles_num({})",
                    get_block_transactions.uncle_indexes().len(),
                    shared.consensus().max_uncles_num(),
                ));
            }
        }

        let block_hash = self.message.block_hash().to_entity();
        debug_target!(
            crate::LOG_TARGET_RELAY,
            "get_block_transactions {}",
            block_hash
        );

        if let Some(block) = shared.store().get_block(&block_hash) {
            let transactions = self
                .message
                .indexes()
                .iter()
                .filter_map(|i| {
                    block
                        .transactions()
                        .get(Into::<u32>::into(i) as usize)
                        .cloned()
                })
                .collect::<Vec<_>>();

            let uncles = self
                .message
                .uncle_indexes()
                .iter()
                .filter_map(|i| block.uncles().get(Into::<u32>::into(i) as usize))
                .collect::<Vec<_>>();

            let content = packed::BlockTransactions::new_builder()
                .block_hash(block_hash)
                .transactions(
                    transactions
                        .into_iter()
                        .map(|tx| tx.data())
                        .collect::<Vec<_>>(),
                )
                .uncles(
                    uncles
                        .into_iter()
                        .map(|uncle| uncle.data())
                        .collect::<Vec<_>>(),
                )
                .build();
            let message = packed::RelayMessage::new_builder().set(content).build();

            return async_send_message_to(&self.nc, self.peer, &message).await;
        }

        Status::ok()
    }
