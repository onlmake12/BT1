### Title
No Maximum Fee Guard in Entropy Request Functions Causes Permanent Loss of Excess ETH - (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract's `requestHelper` function (called by all `request*` and `requestV2*` variants) explicitly does not refund excess `msg.value` to the caller. Because provider fees can change at any time via `setProviderFee` / `setProviderFeeAsFeeManager`, and the Fortuna keeper actively adjusts fees based on gas prices, there is a window between a user's off-chain fee query and on-chain execution where the fee can change. Without a `maxFee` slippage guard, users who send a buffer above the quoted fee to avoid reverts permanently lose that excess ETH to the Pyth protocol fee pool.

---

### Finding Description

In `Entropy.sol`, `requestHelper` is the internal function that processes all randomness requests:

```solidity
// Check that fees were paid and increment the pyth / provider balances.
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) -
    providerFee);
``` [1](#0-0) 

The entire `msg.value - providerFee` is credited to `accruedPythFeesInWei`. There is no refund path. This is documented in the interface:

> "Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2(provider, gasLimit)` prior to each invocation... Further note that excess value is *not* refunded to the caller." [2](#0-1) 

Provider fees are mutable at any time via `setProviderFee` (callable by the provider) and `setProviderFeeAsFeeManager` (callable by the fee manager): [3](#0-2) 

The Fortuna keeper (`adjust_fee_if_necessary`) actively and continuously adjusts provider fees based on prevailing gas prices: [4](#0-3) 

This means fees change regularly in production. A user who queries `getFeeV2()` off-chain and then submits a transaction with a small buffer (to avoid a revert if the fee increases slightly) will permanently lose the buffer amount to the Pyth protocol fee pool. There is no `maxFee` parameter on any of the public request functions (`request`, `requestWithCallback`, `requestV2`) that would allow the user to revert if the actual required fee exceeds their acceptable maximum. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Any user (or integrating contract) that sends `msg.value` greater than the exact required fee permanently loses the excess ETH. The excess is credited to `_state.accruedPythFeesInWei` and is irrecoverable by the user. In practice, users and integrators are advised to call `getFeeV2()` before each request, but the fee can change in the same block (e.g., via a Fortuna keeper transaction or a provider front-running the user). Users who defensively over-pay to avoid reverts are penalized with a direct ETH loss. Integrating contracts that hardcode a fee buffer are particularly vulnerable.

**Impact:** Direct loss of user ETH (excess `msg.value` permanently transferred to Pyth protocol fees).

---

### Likelihood Explanation

- The Fortuna keeper runs continuously and adjusts fees based on gas price fluctuations, meaning fee changes are frequent in production.
- Any user who sends a buffer above the quoted fee (a common defensive pattern) loses that buffer.
- A provider or their fee manager can also front-run a pending user transaction by calling `setProviderFee` to a value just below `msg.value`, extracting the maximum possible fee from the user while still allowing the transaction to succeed.
- No special privileges are required to trigger this — any user submitting a request with `msg.value > requiredFee` is affected.

---

### Recommendation

Add a `maxFee` parameter to all public request functions. Revert if `requiredFee > maxFee`:

```solidity
function requestV2(
    address provider,
    bytes32 userContribution,
    uint32 gasLimit,
    uint128 maxFee  // NEW: slippage guard
) public payable override returns (uint64) {
    uint128 requiredFee = getFeeV2(provider, gasLimit);
    if (requiredFee > maxFee) revert EntropyErrors.FeeTooHigh();
    // ... existing logic
}
```

Alternatively, refund excess `msg.value` back to the caller after deducting `requiredFee`.

---

### Proof of Concept

1. Provider fee is `F = 1000 wei`, Pyth fee is `100 wei`. Total quoted fee = `1100 wei`.
2. User calls `getFeeV2(provider, 0)` off-chain → receives `1100 wei`.
3. User submits `requestV2{value: 1600 wei}(provider, userRandom, 0)` with a 500 wei buffer to avoid a revert if the fee increases.
4. Fortuna keeper (or provider) front-runs with `setProviderFee(1400)` → new total fee = `1500 wei`.
5. User's transaction executes: `requiredFee = 1500`, `msg.value = 1600 ≥ 1500` → no revert.
6. `providerFee = 1400`, `accruedPythFeesInWei += 1600 - 1400 = 200`.
7. User paid `1600 wei` but expected to pay `1100 wei`. The extra `500 wei` is split: `400 wei` extra to provider, `100 wei` extra to Pyth protocol. **User has no recourse.**

Without a `maxFee` guard, the user cannot express "I am willing to pay at most X wei; revert otherwise." The only options are: (a) send exactly the quoted fee and risk a revert, or (b) send a buffer and risk losing it — neither is acceptable for a production-quality protocol.
