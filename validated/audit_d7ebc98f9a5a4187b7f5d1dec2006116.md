### Title
Unbounded `while` Loop in `constructProviderCommitment` Enables Provider-Controlled Gas Exhaustion DoS — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `constructProviderCommitment` function contains a `while` loop that iterates `numHashes` times (a `uint32`, max ~4.29 billion). When a provider does not advance their commitment, `numHashes` grows unboundedly for each new user request. Because the `maxNumHashes` guard is only enforced when the provider explicitly sets it to a non-zero value, a permissionlessly registered malicious provider can create requests whose `numHashes` is large enough to exhaust the block gas limit during any subsequent `reveal` or `revealWithCallback` call, permanently bricking those requests.

---

### Finding Description

`constructProviderCommitment` hashes `revelation` exactly `numHashes` times in a tight loop:

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
``` [1](#0-0) 

`numHashes` is stored in the request at creation time as a `uint32`:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
``` [2](#0-1) 

The only guard against a large `numHashes` is:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [3](#0-2) 

`maxNumHashes` defaults to `0` in `ProviderInfo`: [4](#0-3) 

When `maxNumHashes == 0` the guard is entirely skipped, so `numHashes` is unbounded. `constructProviderCommitment` is called from `revealHelper`, which is invoked by both `reveal` (called by the requester) and `revealWithCallback` (callable by anyone): [5](#0-4) 

---

### Impact Explanation

Each `keccak256` costs ~30 gas. At `numHashes = 1,000,000` the loop alone consumes ~30 M gas, exceeding Ethereum's 30 M block gas limit. Any `reveal` or `revealWithCallback` transaction for such a request will revert with out-of-gas, permanently locking the user's in-flight request and the fees they paid. Because `clearRequest` is only called after a successful reveal, the request can never be cleared, and the user's funds are stuck.

---

### Likelihood Explanation

Any address can permissionlessly call `register` to become a provider. A malicious provider simply omits calling `setMaxNumHashes` (leaving `maxNumHashes = 0`) and never calls `advanceProviderCommitment`. As users submit requests, `assignedSequenceNumber` grows while `currentCommitmentSequenceNumber` stays fixed, so `numHashes` for each new request equals the total number of requests made since the last commitment advancement. After ~33,000 requests (≈ 1 M gas / 30 gas per hash) the loop already approaches the gas limit. The provider earns fees from every request before the DoS manifests.

---

### Recommendation

1. **Enforce a hard cap on `numHashes` unconditionally.** Remove the `maxNumHashes != 0` short-circuit and always revert when `req.numHashes` exceeds a protocol-defined maximum (e.g., 10,000).
2. **Alternatively**, compute `constructProviderCommitment` iteratively with an explicit gas check inside the loop, reverting cleanly rather than running out of gas.
3. Require providers to set a non-zero `maxNumHashes` during `register`, preventing the default-zero bypass.

---

### Proof of Concept

1. Deploy Entropy on a testnet.
2. Call `register` with `chainLength = 2^32`, leaving `maxNumHashes` at its default of `0`.
3. Have 50,000 different users call `requestWithCallback`, paying the required fee each time. Because the provider never calls `advanceProviderCommitment`, `currentCommitmentSequenceNumber` stays at `0`, so request #50,000 is stored with `numHashes = 50,000`.
4. Provider computes the correct `providerContribution` for request #50,000 and calls `revealWithCallback`. The transaction reverts out-of-gas: 50,000 × ~30 gas/hash = 1.5 M gas just for the loop, and with realistic request #500,000 the loop alone exceeds 15 M gas.
5. The request is permanently stuck; the user cannot recover their fee. [6](#0-5)
