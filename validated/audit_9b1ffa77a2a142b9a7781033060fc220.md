Audit Report

## Title
Stale Pre-Eviction Snapshot Overwrites Correctly-Decremented `total_tx_size`/`total_tx_cycles` in `add_entry` - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` computes new running totals into local variables before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts `cell_ref_parent` transactions, it correctly decrements `self.total_tx_size` and `self.total_tx_cycles` via `update_stat_for_remove_tx`. However, `add_entry` then unconditionally overwrites those fields with the stale pre-eviction locals, permanently inflating the pool's reported size and cycle count by the sum of all evicted transactions per triggering insertion.

## Finding Description
In `add_entry` (`tx-pool/src/component/pool_map.rs`, L200‚Äì221):

1. **L210‚Äì211**: `updated_stat_for_add_tx(entry.size, entry.cycles)` computes `total_tx_size = self.total_tx_size + entry.size` and stores it in a **local variable**. `self.total_tx_size` is not yet modified.
2. **L213**: `check_and_record_ancestors` is called. When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (L603), it iterates `evict_candidates` and calls `remove_entry_and_descendants` (L618). This calls `remove_entry` (L263), which at L247 calls `update_stat_for_remove_tx`, **directly decrementing `self.total_tx_size`** for each evicted transaction.
3. **L218‚Äì219**: `self.total_tx_size = total_tx_size` and `self.total_tx_cycles = total_tx_cycles` write back the stale snapshot, overwriting the correctly-decremented values.

After this sequence, `self.total_tx_size = old_total + new_entry_size` instead of the correct `old_total - evicted_sizes + new_entry_size`. The inflation of `sum(evicted_tx.size)` accumulates permanently with each triggering insertion.

Existing guards do not prevent this: `updated_stat_for_add_tx` only checks for integer overflow (L716‚Äì721), not for the stale-overwrite race. `update_stat_for_remove_tx` has an underflow fallback that recomputes from scratch (L743‚Äì749), but this path is never reached because the overwrite at L218‚Äì219 happens after the correct decrement, not before.

## Impact Explanation
`total_tx_size` is the sole guard in `limit_size()` (`tx-pool/src/pool.rs`, L298): `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated value causes `limit_size()` to evict legitimate pending transactions even when real pool capacity remains. Additionally, `updated_stat_for_add_tx` uses the inflated `total_tx_size` to issue `Reject::Full` for new submissions (L716‚Äì721), falsely signaling a full pool. The inflation accumulates with each triggering insertion, progressively shrinking the effective pool capacity toward zero. An attacker can exploit this to cause a targeted CKB node to reject and evict legitimate transactions at scale, degrading its ability to relay transactions and contributing to **CKB network congestion with few costs** ‚Äî matching the High (10001‚Äì15000 points) impact class.

## Likelihood Explanation
The eviction path at L603‚Äì625 is reachable by any unprivileged user via RPC `send_transaction` or P2P relay. The attacker must submit a transaction that: (a) references a cell dep output already consumed by an existing pool transaction (`cell_ref_parents` non-empty), and (b) has an ancestor count that exceeds `max_ancestors_count` before eviction but satisfies the limit after eviction. Both conditions can be deliberately engineered by observing the mempool state and crafting transactions accordingly. No special privilege, key, or admin access is required. The attack is repeatable: each triggering insertion inflates `total_tx_size` further, and the inflation persists until node restart.

## Recommendation
Remove the pre-eviction snapshot approach. After `check_and_record_ancestors` returns, `self.total_tx_size` already reflects the correct post-eviction value. Apply the new entry's contribution additively:

```rust
// Replace lines 210-211 and 218-219 with:
// (keep the overflow check, but apply it after evictions)
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Now self.total_tx_size is correct post-eviction; just add the new entry
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

The overflow check from `updated_stat_for_add_tx` should be moved to after evictions complete, operating on the already-correct `self.total_tx_size`.

## Proof of Concept
1. Configure a node with a small `max_ancestors_count` (e.g., 25) and `max_tx_pool_size`.
2. Submit a chain of 24 transactions where the last transaction (`T_ref`) consumes a cell dep output.
3. Submit a new transaction `T_new` that also references the same cell dep output as an ancestor, pushing `ancestors_count` to 25 (equal to `max_ancestors_count`), satisfying the eviction condition at L603.
4. `check_and_record_ancestors` evicts `T_ref` (and its descendants) via `remove_entry_and_descendants`, correctly decrementing `self.total_tx_size` by `T_ref.size`.
5. `add_entry` then overwrites `self.total_tx_size` with the stale pre-eviction value.
6. Query `tx_pool_info` via RPC: `total_tx_size` will exceed the sum of actual pool entries' sizes by `T_ref.size`.
7. Repeat steps 2‚Äì6 with fresh transactions: each iteration inflates `total_tx_size` further.
8. After sufficient iterations, `total_tx_size > max_tx_pool_size` while the pool is nearly empty; `limit_size()` evicts legitimate transactions and `updated_stat_for_add_tx` rejects new submissions as `Reject::Full`.
