### Title
`max_tx_verify_cycles` Cycle Budget Not Enforced in `_process_tx` for Remote Relay Transactions — (`tx-pool/src/process.rs`)

### Summary

`_process_tx` uses the attacker-supplied `declared_cycles` directly as the `max_cycles` budget passed to `verify_rtx` / the CKB-VM, rather than capping it at `max_tx_verify_cycles`. The only upstream guard rejects `declared_cycles > max_block_cycles` (strict), so any value in `(max_tx_verify_cycles, max_block_cycles]` passes unchecked. An unprivileged relay peer who can craft a transaction whose scripts genuinely consume N cycles (where `max_tx_verify_cycles < N ≤ max_block_cycles`) and declares exactly N can force the victim node to run the CKB-VM for N cycles per transaction instead of the configured 70 M-cycle cap — a factor-of-50 amplification.

### Impact Explanation

An attacker who controls valid UTXOs can craft a transaction containing a script loop that consumes exactly N cycles (`max_tx_verify_cycles < N ≤ max_block_cycles`). By declaring `declared_cycles = N` in the RelayV3 message, the victim node runs the CKB-VM for N cycles (up to 3.5 B) instead of the configured 70 M-cycle cap — a **50× amplification** of CPU cost per transaction. Because `declared == verified`, the transaction is accepted into the pool, so the attacker does not lose the UTXO until the transaction is mined. Repeated submissions across multiple connections cause sustained CPU exhaustion and degrade block-template assembly and peer-relay throughput.

---

### Likelihood Explanation

The path is reachable by any unprivileged P2P peer with no special privileges. The attacker needs valid UTXOs and a script that runs for a known, exact cycle count — both are straightforward to construct on CKB (a simple counted loop in a lock script). The `declared_cycles == max_block_cycles` edge case passes the only guard. No PoW, no majority hashpower, no leaked keys are required.

---

### Recommendation

In `_process_tx`, cap `max_cycles` at `max_tx_verify_cycles` when a `declared_cycles` value is present and the transaction originates from a remote peer:

```rust
let max_cycles = declared_cycles
    .map(|d| d.min(self.tx_pool_config.max_tx_verify_cycles))
    .unwrap_or_else(|| self.consensus.max_block_cycles());
```

Alternatively, reject the transaction before verification if `declared_cycles > max_tx_verify_cycles`, returning `Reject::ExceededMaximumAncestorsCount` or a dedicated `Reject::ExceededCyclesLimit` variant, and ban the peer.

### Proof of Concept

1. Craft a CKB lock script that executes a tight loop consuming exactly `N = max_block_cycles` (3 500 000 000) cycles.
2. Build a valid transaction spending a UTXO locked by that script.
3. Connect to the victim node as a RelayV3 peer.
4. Send a `RelayTransactions` message with `cycles = N` for that transaction.
5. Observe via `perf`/`top` that the victim node's tx-pool worker thread saturates a CPU core for the full 3.5 B-cycle verification window instead of the expected 70 M-cycle window.
6. Confirm the transaction enters the pending pool (since `declared == verified`).
7. Repeat with fresh UTXOs to sustain CPU exhaustion.

The wall-clock verification time will be proportional to 3.5 B cycles, not 70 M cycles, falsifying the invariant that `max_tx_verify_cycles` bounds per-transaction CKB-VM cost in the pool.
