### Title
Missing Authority Check on `sol_destination` in `close_signature_set` Allows Rent Theft ‚Äî (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

---

### Summary

The `CloseSignatureSet` instruction closes a `signature_set` account and transfers its rent lamports to `sol_destination`. There is no constraint requiring `sol_destination` to be the original payer who funded the `signature_set`. Any unprivileged signer can pass themselves as `sol_destination` against any valid `posted_vaa`/`signature_set` pair and receive the rent.

---

### Finding Description

The `CloseSignatureSet` account context is: [1](#0-0) 

`sol_destination` is validated only as `Signer<'info>` ‚Äî any keypair that signs the transaction qualifies. [2](#0-1) 

The `posted_vaa` constraint only enforces `has_one = signature_set`, meaning the provided `signature_set` pubkey must match the one stored in the `posted_vaa` account. There is no check tying `sol_destination` to the original payer. [3](#0-2) 

The `close = sol_destination` Anchor macro unconditionally transfers all lamports from `signature_set` to whoever `sol_destination` is. [4](#0-3) 

The `SignatureSet` struct stores no payer field ‚Äî only `sig_verify_successes`, `message_hash`, and `guardian_set_index` ‚Äî so there is no on-chain record of who funded the account. [5](#0-4) 

The `VerifySignatures` instruction creates the `signature_set` with `payer = payer` and `init_if_needed`, but never writes the payer's pubkey into the account data. [6](#0-5) 

---

### Impact Explanation

A relayer (Alice) calls `verify_signatures` + `post_vaa`, paying rent for the `signature_set` keypair account (~0.001 SOL for a 19-guardian set). An attacker (Bob) observes the on-chain `posted_vaa` PDA, then calls `close_signature_set` with `sol_destination = Bob`. The Anchor `close` macro transfers all lamports from `signature_set` to Bob. Alice's rent is permanently stolen. Bob's only cost is the transaction fee (~0.000005 SOL), making this profitable at any scale.

---

### Likelihood Explanation

The attack requires no privileges, no leaked keys, and no governance access. Any on-chain observer can automate this by watching for new `PostedVaaV1` accounts and front-running or racing the legitimate relayer's cleanup call. The `posted_vaa` PDA address is deterministic from the message hash, so the attacker can compute the target accounts without any special information.

---

### Recommendation

Store the original payer's pubkey in the `SignatureSet` account at creation time (in `verify_signatures`), then add a constraint in `CloseSignatureSet` requiring `sol_destination.key() == signature_set.payer`. Alternatively, add a `has_one = payer` constraint on `signature_set` and record the payer in the struct. This is the standard Anchor pattern for rent-reclaim authority.

---

### Proof of Concept

1. Alice calls `verify_signatures` (with her keypair as payer and a fresh `signature_set` keypair), then calls `post_vaa` ‚Äî creating a `PostedVaaV1` PDA with `info.signature_set = signature_set.pubkey()`.
2. Bob (a different keypair) constructs a `close_signature_set` transaction with:
   - `sol_destination` = Bob (Bob signs)
   - `posted_vaa` = the PDA from step 1
   - `signature_set` = Alice's signature_set account
3. The `has_one = signature_set` check passes (the pubkeys match).
4. Anchor's `close = sol_destination` transfers all lamports from `signature_set` to Bob.
5. Assert: Bob's balance increased by `signature_set` rent; Alice's `signature_set` account is gone.
