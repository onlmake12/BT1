### Title
DAO Withdrawal Witness Index Interpreted Differently by Rust Fee Calculator vs. On-Chain C VM Script — (`util/dao/src/lib.rs`)

### Summary
The `DaoCalculator::transaction_maximum_withdraw` in Rust reads the full `u64` from `WitnessArgs.input_type` as the `header_deps` index to locate the deposit block header. The on-chain `dao.c` C VM script reads only the **lowest byte** (`u8`) of that same field. When a transaction sender supplies an index value above 255, the two layers resolve to different entries in `header_deps`, creating a discrepancy between the Rust fee/DAO-field calculation and what the C VM actually executes — a direct analog to the Socket "black box payload" class.

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the deposit-block pointer from the witness:

```rust
// dao contract stores header deps index as u64 in the input_type field of WitnessArgs
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // full u64 used as index
``` [1](#0-0) 

The on-chain `dao.c` script (referenced in the test comment at `test/src/specs/dao/dao_user.rs:14`) reads only the lowest byte of the same 8-byte little-endian field as the index. The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this split:

```
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
``` [2](#0-1) 

The block-number cross-check at line 105 catches the specific case where the full-u64 index resolves to the **withdraw** block (whose number does not match cell data):

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
``` [3](#0-2) 

However, this guard does **not** cover the reverse scenario: an attacker places a valid deposit block at position `full_u64_index` (so Rust's block-number check passes) while the lowest-byte index points to a different block that the C VM will actually use. In that case Rust and the C VM silently disagree on which deposit header governs the interest calculation.

### Impact Explanation

Two concrete consequences:

1. **Tx-pool pollution / false fee acceptance.** `check_tx_fee` in `tx-pool/src/util.rs` calls `DaoCalculator::transaction_fee`, which calls `transaction_maximum_withdraw`. [4](#0-3)  If Rust resolves a deposit header with a higher AR ratio than the one the C VM will use, Rust computes a larger maximum withdrawal, accepts the transaction as fee-paying, and admits it to the pool. The C VM then rejects it on-chain (output capacity exceeds the C-VM-computed maximum), wasting pool resources.

2. **Incorrect DAO field in mined blocks.** `calc_dao` in `tx-pool/src/block_assembler/mod.rs` calls `DaoCalculator::dao_field_with_current_epoch`, which internally calls `withdrawed_interests → transaction_maximum_withdraw`. [5](#0-4)  If a crafted DAO withdrawal is included in a block template, the miner's Rust code computes `withdrawed_interests` using the wrong deposit header, producing an incorrect `dao` field in the block header. Other nodes, whose C VM verifies against the correct deposit header, reject the block, causing a consensus split for that miner.

### Likelihood Explanation

The entry path is fully unprivileged: any user can submit a DAO withdrawal transaction via the `send_transaction` RPC with a crafted `WitnessArgs.input_type` value whose `u64` representation exceeds 255. The transaction structure is valid (passes molecule deserialization and the 8-byte length check), and the block-number guard only catches the narrow case where the full-u64 index resolves to the withdraw block. Constructing the exploit requires only knowledge of the `header_deps` ordering in the transaction, which the attacker controls entirely.

### Recommendation

Enforce that the `header_dep_index` stored in `WitnessArgs.input_type` fits in a single byte (i.e., `<= 0xFF`) before using it, matching the range the C VM script actually supports. Alternatively, document and enforce a maximum `header_deps` length of 256 for DAO withdrawal transactions. Either bound eliminates the Rust/C-VM index divergence.

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` already demonstrates the divergence:

- `header_deps[1]` = deposit block (number 100) — what C VM resolves via lowest byte of `257`
- `header_deps[257]` = withdraw block (number 200) — what Rust resolves via full `u64` of `257`
- Witness `input_type` = `257u64` (little-endian bytes `[1, 1, 0, 0, 0, 0, 0, 0]`) [6](#0-5) 

The test asserts `result.is_err()` because Rust resolves index 257 → withdraw block → block-number mismatch. The inverse — where Rust resolves a valid deposit block at the full-u64 index while the C VM resolves a different block at the lowest-byte index — is not tested and is not caught by the block-number guard, leaving the discrepancy exploitable for tx-pool pollution and incorrect DAO field assembly.
