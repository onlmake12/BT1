### Title
Tx-Pool Admission Denial via ProposalShortId Slot Pre-Occupation — (`tx-pool/src/process.rs`, `tx-pool/src/component/verify_queue.rs`, `tx-pool/src/util.rs`)

---

### Summary

An unprivileged P2P peer can permanently deny a victim's valid transaction from entering the CKB tx-pool by submitting a "cousin" transaction — one with identical inputs/outputs (same `tx_hash` / `ProposalShortId`) but invalid witnesses — before the victim. The cousin occupies the unique `ProposalShortId` slot in the `verify_queue`, causing the victim's submission to be rejected with `PoolRejectedDuplicatedTransaction`. This is a direct analog to M-17's "frontrun the uniqueness slot" pattern.

---

### Finding Description

**Root cause — `ProposalShortId` is witness-blind**

`ProposalShortId` is derived from the first 10 bytes of `tx_hash`, and `tx_hash` is computed from `RawTransaction` which **excludes witnesses**: [1](#0-0) [2](#0-1) 

Two transactions with identical inputs, outputs, cell-deps, and header-deps but **different witnesses** share the same `tx_hash` and the same `ProposalShortId`. The integration test explicitly confirms this: [3](#0-2) 

**Deduplication is keyed on `ProposalShortId` across all three pool structures**

`verify_queue_contains` and `orphan_contains` both use `proposal_short_id()`: [4](#0-3) 

`check_txid_collision` (called inside `pre_check` for the main pool) also uses `proposal_short_id()`: [5](#0-4) 

The comment in `pre_check` incorrectly states "Same txid means exactly the same transaction, including inputs, outputs, witnesses" — this is false; witnesses are excluded from `tx_hash`: [6](#0-5) 

**The P2P relay admission path enqueues without script verification**

`resumeble_process_tx` (used by `submit_remote_tx` for P2P relay) runs only `non_contextual_verify` — which checks version, size, empty, duplicate deps, outputs data, and script hash type, but **not witness/script validity** — then immediately enqueues the transaction: [7](#0-6) 

`non_contextual_verify` does not check witnesses: [8](#0-7) 

**Victim's transaction is rejected as Duplicated**

Both `process_tx` (RPC path) and `resumeble_process_tx` (relay path) check `verify_queue_contains` before proceeding. If the attacker's cousin is already in the queue, the victim's transaction is immediately rejected: [9](#0-8) 

The `VerifyQueue` uses `ProposalShortId` as its unique index: [10](#0-9) 

The integration test confirms the rejection: [11](#0-10) 

---

### Impact Explanation

A victim who submits a valid transaction T (spending their own cells with valid signatures) can be denied pool admission. An attacker who observes T's structure (inputs/outputs are public once broadcast or can be predicted) constructs T′ with the same `tx_hash` but garbage witnesses, submits T′ via P2P relay first, and T′ occupies the `ProposalShortId` slot in the `verify_queue`. The victim's T is then rejected with `PoolRejectedDuplicatedTransaction (-1107)`. The victim cannot get their transaction into the pool until T′ is dequeued and fails script verification — a window that grows with verify_queue depth. In time-sensitive scenarios (e.g., a transaction that must be confirmed before a `since`-locked deadline), this denial has direct financial impact.

---

### Likelihood Explanation

- Any connected P2P peer can send `RelayTransactions` messages containing a cousin transaction with no fee and garbage witnesses; `non_contextual_verify` will pass.
- The victim's transaction structure (inputs/outputs) is observable once it is broadcast to the network, or can be predicted for known cell-spending patterns.
- A single-peer attacker is eventually banned after the cousin fails script verification in the verify_queue worker (via `ban_malformed`). However, the denial window (queue processing latency) is real and exploitable. With multiple peers (low Sybil cost on a permissionless P2P network), the attack can be sustained indefinitely.
- The `VerifyQueue` has a 256 MB size limit, so the attacker can also fill it with many cousin transactions to extend the denial window. [12](#0-11) 

---

### Recommendation

1. **Key the verify_queue and orphan pool on `witness_hash` instead of `ProposalShortId`** for the purpose of deduplication. `witness_hash` covers the full transaction including witnesses, making cousin transactions distinguishable.
2. **Alternatively, perform a lightweight witness non-emptiness / structural check** before enqueuing in `resumeble_process_tx`, so that a transaction with obviously invalid witnesses (e.g., wrong witness count relative to inputs) is rejected before occupying the slot.
3. **Fix the misleading comment** in `pre_check` that claims `tx_hash` covers witnesses — it does not.

---

### Proof of Concept

The existing integration test already demonstrates the collision:

```rust
// test/src/specs/tx_pool/collision.rs
let tx1 = node.new_transaction_spend_tip_cellbase(); // valid witnesses
let tx2 = tx1.as_advanced_builder().witness(Bytes::default()).build(); // garbage witness
assert_eq!(tx1.hash(), tx2.hash());           // same tx_hash
assert_eq!(tx1.proposal_short_id(), tx2.proposal_short_id()); // same slot key
assert_ne!(tx1.witness_hash(), tx2.witness_hash()); // different full identity

// Attack: submit tx2 (garbage witnesses) first via P2P relay
node.submit_transaction(&tx2); // attacker's cousin occupies ProposalShortId slot

// Victim submits tx1 (valid witnesses) — rejected
let result = node.rpc_client().send_transaction_result(tx1.data().into());
assert!(result.err().unwrap().to_string().contains("PoolRejectedDuplicatedTransaction"));
``` [11](#0-10) 

The attacker entry path is `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue`, bypassing all script checks. The victim's entry path is `process_tx` → `verify_queue_contains` → `Reject::Duplicated`. [13](#0-12) [14](#0-13)

            return Err(Reject::Duplicated(tx.hash()));
        }
        self.enqueue_verify_queue(tx, is_proposal_tx, remote).await
    }

```rust
pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
