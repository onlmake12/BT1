Audit Report

## Title
Unverified `from` Peer Identity in Hole-Punching `ConnectionRequest` Enables NAT-Traversal Hijacking and Rate-Limiter Bypass — (File: network/src/protocols/hole_punching/component/connection_request.rs)

## Summary
`ConnectionRequestProcess::execute()` decodes the `from` field exclusively from the attacker-controlled message payload and never cross-checks it against the actual session identity (`self.peer`). This unverified `from` is used as the key for both the `forward_rate_limiter` and the `pending_delivered` cache. An attacker can poison `pending_delivered[victim_peer_id]` with attacker-controlled listen addresses and bypass the forward rate limiter by cycling through arbitrary `from` values, enabling NAT-traversal hijacking and gossip amplification across the network.

## Finding Description

**Root cause — unverified `from` field:**

In `connection_request.rs` lines 36–38, `from` is decoded entirely from the raw message bytes with no comparison against the actual session's `PeerIndex`/`PeerId`:

```rust
let from = PeerId::from_bytes(value.from().raw_data().to_vec()).map_err(|_| {
    StatusCode::InvalidFromPeerId.with_context("the from peer id is invalid")
})?;
```

`ConnectionRequestProcess` holds `self.peer: PeerIndex` (line 88) but it is never used to resolve and verify the sender's `PeerId` against `content.from`.

**Attack path 1 — `pending_delivered` cache poisoning and NAT-traversal hijacking:**

When `self_peer_id == &content.to` (line 145), `respond_delivered(content.from, ...)` is called. Inside `respond_delivered` (lines 234–237), the attacker-controlled `from_peer_id` is inserted as the cache key:

```rust
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
```

The only guard (lines 161–166) is a 2-minute deduplication check on the same `from_peer_id` — it does not prevent the initial insertion with a spoofed identity.

`ConnectionSyncProcess::execute()` (lines 111–115 of `connection_sync.rs`) looks up the cache using the equally unverified `content.from` from the sync message:

```rust
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
```

`ConnectionSyncProcess::new()` does not accept a `peer: PeerIndex` parameter at all (lines 60–74 of `connection_sync.rs`), so there is no mechanism to verify the sender of the sync message.

Exploit flow:
1. Attacker `A` connects to victim node `V` over the HolePunching protocol.
2. `A` sends a `ConnectionRequest` with `from = peer_id_of_B`, `to = V.peer_id`, `listen_addrs = [A_tcp_addr]`, empty `route`.
3. `V` receives it: `self_peer_id == content.to`, so `respond_delivered(peer_id_of_B, [A_tcp_addr])` is called. `pending_delivered[peer_id_of_B] = ([A_tcp_addr], now)` is inserted.
4. `A` immediately sends a `ConnectionSync` with `from = peer_id_of_B`, `to = V.peer_id`, empty `route`.
5. `V` receives it: route is empty, `self_peer_id == content.to`, looks up `pending_delivered[peer_id_of_B]`, finds `[A_tcp_addr]`, spawns `try_nat_traversal(bind_addr, A_tcp_addr)` — connecting to `A` instead of `B`.

**Attack path 2 — forward rate-limiter bypass:**

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` (lines 132–143 of `connection_request.rs`) with a quota of 1 per second per key (line 256 of `mod.rs`). The outer `rate_limiter` is keyed by `(session_id, msg.item_id())` with a quota of 30 per second (line 251 of `mod.rs`).

An attacker sending 30 `ConnectionRequest` messages/second with 30 distinct fake `from` values saturates the outer rate limiter but produces 30 unique keys in the `forward_rate_limiter`, each passing the 1/s check. Every message that passes is broadcast to `sqrt(total_peers)` nodes (lines 281–305 of `connection_request.rs`) for up to `MAX_HOPS = 6` hops. With N=100 peers and sqrt(100)=10 fanout, this yields up to 30 × 10^6 message deliveries per second at the deepest hop.

The route deduplication check (lines 128–130) only prevents a node from re-processing a message whose route already contains its own ID. With 30 distinct `from` values, each message is a distinct "conversation" with its own route, so the same relay node processes all 30 messages and forwards each to sqrt(N) peers.

**Attack path 3 — `pending_delivered` memory exhaustion:**

`pending_delivered` is an unbounded `HashMap` (line 44 of `mod.rs`) pruned only every 5 minutes (`TIMEOUT = 5 * 60 * 1000 ms`, `CHECK_INTERVAL = 5 * 60s`, lines 25–28 of `mod.rs`). At 30 messages/second with distinct fake `from` values, an attacker can insert up to 9,000 entries per eviction window, each holding a `PeerId` and up to 24 `Multiaddr` values (`ADDRS_COUNT_LIMIT = 24`, line 27 of `mod.rs`).

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The forward rate-limiter bypass is the primary network-level impact. A single connected peer sending 30 `ConnectionRequest` messages/second with distinct fake `from` values bypasses the `forward_rate_limiter` at every relay node. Each message is broadcast to `sqrt(N)` peers and forwarded up to 6 hops. With N=100 peers, this yields up to 30 × 10^6 message deliveries per second at the deepest hop, saturating the gossip layer. The NAT-traversal hijacking additionally causes victim nodes to spawn 30-second TCP connection tasks to attacker-controlled endpoints, wasting file descriptors and CPU.

## Likelihood Explanation

Any peer connected to a CKB node over the HolePunching protocol can trigger this. No special privilege, key material, or hashpower is required. The attacker only needs one active session. The outer `rate_limiter` (30 msg/s per session) is the only throttle, and it is insufficient: 30 distinct fake `from` values per second is enough to sustain both the cache poisoning and the rate-limiter bypass indefinitely. The attack is repeatable and requires no victim interaction.

## Recommendation

In `ConnectionRequestProcess::execute()`, resolve the actual `PeerId` of the sending session from the peer registry using `self.peer` (the `PeerIndex`) before accepting `content.from`. Assert that the resolved `PeerId` equals `content.from`; if they differ, return an error status and optionally ban the session. Pass `peer: PeerIndex` to `ConnectionSyncProcess` and apply the same check for `content.from`. Additionally, cap the size of `pending_delivered` (e.g., with an LRU eviction policy) to bound memory consumption independent of the rate limiter.

## Proof of Concept

**Cache poisoning + NAT hijacking:**
1. Attacker peer `A` connects to victim node `V` over the HolePunching protocol.
2. `A` sends a `ConnectionRequest` with `from = peer_id_of_B` (any known peer), `to = V.peer_id`, `listen_addrs = [A_tcp_addr]`, `max_hops = 6`, empty `route`.
3. `V` receives it: `self_peer_id == content.to`, so `respond_delivered(peer_id_of_B, ...)` is called. `pending_delivered[peer_id_of_B] = ([A_tcp_addr], now)` is inserted.
4. `A` immediately sends a `ConnectionSync` with `from = peer_id_of_B`, `to = V.peer_id`, empty `route`.
5. `V` receives it: `self_peer_id == content.to`, route is empty, looks up `pending_delivered[peer_id_of_B]`, finds `[A_tcp_addr]`, spawns `try_nat_traversal(bind_addr, A_tcp_addr)` — connecting to `A` instead of `B`.

**Forward rate-limiter bypass:**
1. `A` connects to `V`.
2. `A` sends 30 `ConnectionRequest` messages/second, each with a distinct randomly-generated `from` PeerId, `to = any_peer_id`, valid `listen_addrs`, `max_hops = 6`.
3. Each message passes the outer `rate_limiter` (30/s per session) and the `forward_rate_limiter` (unique key per message), and is broadcast to `sqrt(N)` peers for up to 6 hops, amplifying traffic across the gossip network.
