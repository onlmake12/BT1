### Title
`MAX_DEPOSIT_LIMIT` Bypass via Cumulative `addFunds` Calls on Permanent Subscriptions — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `addFunds` function in `Scheduler.sol` checks only the per-transaction deposit amount (`msg.value`) against `MAX_DEPOSIT_LIMIT`, not the cumulative subscription balance. Because permanent subscriptions prohibit withdrawals, a user can call `addFunds` repeatedly with amounts just at or below `MAX_DEPOSIT_LIMIT` to accumulate a balance far exceeding the intended cap, permanently locking ETH in the contract beyond the protocol's design limit.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
/// Maximum deposit limit for permanent subscriptions in wei
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
```

Both `createSubscription` and `addFunds` enforce this limit only against `msg.value`:

```solidity
// createSubscription (line 48)
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// addFunds (line 613)
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Neither check accounts for the **existing balance** (`status.balanceInWei`) already held by the subscription. The correct guard should be:

```solidity
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Because `withdrawFunds` unconditionally reverts for permanent subscriptions:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

any ETH deposited beyond `MAX_DEPOSIT_LIMIT` is irrecoverably locked.

---

### Impact Explanation

A user can permanently lock an arbitrarily large amount of ETH in a single permanent subscription by calling `addFunds` N times with `msg.value = MAX_DEPOSIT_LIMIT` each time, accumulating `N × 100 ETH` in a non-withdrawable subscription. This:

1. Causes permanent, irrecoverable loss of the depositor's ETH beyond the intended cap.
2. Violates the protocol's explicit design invariant that permanent subscriptions are bounded by `MAX_DEPOSIT_LIMIT`.
3. Could be used by any unprivileged user (including a third party, since `addFunds` has no access control) to grief a permanent subscription owner by locking their subscription's balance far beyond the limit.

---

### Likelihood Explanation

The entry path requires no special privileges. `addFunds` is callable by anyone (`external payable` with no `onlyManager` modifier), so even a third party can trigger this on any active permanent subscription. The only cost is the ETH deposited. The bypass is straightforward and requires no complex setup.

---

### Recommendation

Change both `createSubscription` and `addFunds` to check the **post-deposit cumulative balance** against `MAX_DEPOSIT_LIMIT`:

```solidity
// In addFunds:
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// In createSubscription (msg.value is the initial balance, so this is already correct
// as long as addFunds is fixed; but for consistency):
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

---

### Proof of Concept

1. Deploy `Scheduler` with default parameters.
2. Create a permanent subscription with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH). This succeeds.
3. Call `addFunds{value: MAX_DEPOSIT_LIMIT}(subscriptionId)` — this also succeeds because `msg.value == MAX_DEPOSIT_LIMIT` (not `>`).
4. Repeat step 3 N times. The subscription balance is now `(N+1) × 100 ETH`.
5. Attempt `withdrawFunds` — it reverts with `CannotUpdatePermanentSubscription`.
6. All deposited ETH beyond 100 ETH is permanently locked.
