### Title
Fee Rate Check Bypassed at Verify Queue Admission in `resumeble_process_tx` Allows Zero-Fee Spam to Exhaust Verify Queue and Block Legitimate Transactions — (`tx-pool/src/process.rs`)

---

### Summary

The `resumeble_process_tx` function, which handles all remote P2P-relayed transactions, adds transactions to the verify queue without enforcing the configured `min_fee_rate`. The fee rate check (`check_tx_fee`) is deferred to the verify worker that runs after dequeuing. An unprivileged P2P peer with valid UTXOs can continuously submit zero-fee transactions that pass structural validation, fill the bounded verify queue, and cause all subsequent legitimate transactions to be rejected with `Reject::Full` while verify workers waste CPU cycles processing and discarding the spam.

---

### Finding Description

**Root Cause — Missing Fee Rate Gate at Verify Queue Admission**

`resumeble_process_tx` is the entry point for every transaction submitted via P2P relay (`submit_remote_tx`):

```rust
// tx-pool/src/process.rs
pub(crate) async fn resumeble_process_tx(
    &self,
    tx: TransactionView,
    is_proposal_tx: bool,
    remote: Option<(Cycle, PeerIndex)>,
) -> Result<bool, Reject> {
    self.non_contextual_verify(&tx, remote).await?;   // structural only
    if self.orphan_contains(&tx).await { ... }
    if self.verify_queue_contains(&tx).await { ... }
    self.enqueue_verify_queue(tx, is_proposal_tx, remote).await  // NO fee check
}
``` [1](#0-0) 

`non_contextual_verify` checks only structural validity (version, size, cellbase-like). It does not check fee rate. [2](#0-1) 

`enqueue_verify_queue` checks only for duplicates and the total-size cap; it never calls `check_tx_fee`:

```rust
// tx-pool/src/component/verify_queue.rs
pub fn add_tx(...) -> Result<bool, Reject> {
    ...
    if self.is_full(tx_size) {
        return Err(Reject::Full(...));
    }
    ...
}
``` [3](#0-2) 

The fee rate check only executes inside `_process_tx` → `pre_check` → `check_tx_fee`, which runs in the verify worker **after** the transaction has already been dequeued:

```rust
// tx-pool/src/util.rs
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [4](#0-3) 

By contrast, the local-submission path (`process_tx` → `_process_tx`) calls `pre_check` synchronously and rejects low-fee transactions before they ever touch the verify queue: [5](#0-4) 

**The Verify Queue Is Bounded But Exhaustible**

The verify queue enforces a total-size cap (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`, ~256 MB based on test fixtures). Each transaction is capped at `TRANSACTION_SIZE_LIMIT = 512 KB`: [6](#0-5) 

An attacker needs only ~512 structurally valid zero-fee transactions to saturate the queue. Once full, every subsequent `enqueue_verify_queue` call returns `Reject::Full`, which is propagated to the caller and reported to the relayer as a rejection.

**The Configured Default Enforces a Non-Zero Minimum**

The default `min_fee_rate` is 1 000 shannons/KB: [7](#0-6) [8](#0-7) 

Zero-fee transactions (outputs capacity = inputs capacity) will always fail this check — but only after they have already occupied the verify queue.

---

### Impact Explanation

1. An attacker pre-funds ~512+ UTXOs on-chain (no privileged access required).
2. The attacker crafts zero-fee transactions (outputs capacity = inputs capacity) and relays them via the P2P protocol.
3. Each transaction passes `non_contextual_verify` and enters the verify queue without a fee check.
4. The verify queue reaches its size cap; all subsequent `enqueue_verify_queue` calls return `Reject::Full`.
5. Legitimate users' transactions submitted via `send_transaction` RPC or P2P relay are rejected with `PoolIsFull (-1106)`.
6. Verify workers continuously dequeue and reject the zero-fee spam, consuming CPU. As slots free up, the attacker refills them.
7. The node's block assembly is degraded: the proposal list and committed transaction set shrink because the verify queue is perpetually occupied by unverifiable spam.

The attack is self-sustaining: zero-fee transactions return their full capacity to the attacker's outputs, so the attacker can recycle UTXOs across waves (subject to `max_ancestors_count = 25` chaining limit, requiring multiple independent UTXO sets). [9](#0-8) 

---

### Likelihood Explanation

- **Attacker profile:** Any unprivileged P2P peer. No keys, no operator access, no mining power required.
- **Entry point:** `submit_remote_tx` → `resumeble_process_tx`, reachable from any connected peer via the Relay protocol.
- **Cost:** Requires on-chain UTXOs, but zero-fee transactions preserve capacity, so the attacker's funds are not consumed — only temporarily locked in outputs.
- **Detectability:** Low. The spam transactions are structurally valid and indistinguishable from legitimate transactions at admission time. [10](#0-9) 

---

### Recommendation

Apply the fee rate check at verify queue admission, before `enqueue_verify_queue`, inside `resumeble_process_tx`. Specifically, perform a lightweight read-lock check equivalent to `check_tx_fee` (resolve the transaction and compare fee against `min_fee_rate`) before calling `enqueue_verify_queue`. This mirrors the protection already present in the synchronous `process_tx` path and closes the gap between local and remote transaction admission.

Alternatively, add a pre-admission fee rate check directly inside `enqueue_verify_queue` or as a guard in `resumeble_process_tx` using a read-only snapshot, consistent with how `check_txid_collision` is applied in `pre_check`. [11](#0-10) 

---

### Proof of Concept

```
1. Attacker controls N UTXOs (e.g., N = 600) with capacity C each.
2. For each UTXO_i, attacker builds tx_i:
     inputs:  [UTXO_i]
     outputs: [Cell(capacity=C, lock=attacker_lock)]
     fee:     0  (outputs_capacity == inputs_capacity)
3. Attacker connects to a CKB node as a P2P peer.
4. Attacker sends RelayTransactions messages containing tx_1 … tx_600.
5. Each tx_i passes non_contextual_verify (valid structure, size < 512KB).
6. Each tx_i is admitted to the verify queue via enqueue_verify_queue.
7. After ~512 transactions, verify queue total_tx_size ≥ DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE.
8. All subsequent enqueue_verify_queue calls return Reject::Full.
9. Legitimate users calling send_transaction RPC receive PoolIsFull (-1106).
10. Verify workers process and reject tx_1…tx_N for LowFeeRate; attacker resubmits.
11. Node's block template proposals and committed transactions are starved.
``` [12](#0-11) [13](#0-12)

        &self,
        tx: &TransactionView,

        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {

    pub fn add_tx(

        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }

        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {

                tx.hash()

        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
