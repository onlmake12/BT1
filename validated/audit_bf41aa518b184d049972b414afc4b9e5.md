Audit Report

## Title
Broken `timeout_count` Accumulation in `DownloadScheduler::decrease` Causes Unconditional `task_count` Decrement on Every Slow Response — (File: sync/src/types/mod.rs)

## Summary
`DownloadScheduler::decrease` at line 522 writes `self.timeout_count = self.task_count.saturating_add(num)` instead of `self.timeout_count.saturating_add(num)`. Because `task_count` initialises to 32, the expression always evaluates to ≥ 33, which always exceeds the guard threshold of 2, so `task_count` is decremented and `timeout_count` is reset on every single slow block response. The intended 3-response grace window is never applied, causing peers to be disconnected after 32 slow responses instead of the intended ~96+.

## Finding Description
In `sync/src/types/mod.rs` lines 521–527:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num); // BUG: should be self.timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

The intended design: `timeout_count` accumulates slow-response penalties; only when it exceeds 2 (i.e., after 3+ slow responses) is `task_count` decremented and `timeout_count` reset. The bug overwrites `timeout_count` with `task_count + num` on every call. Since `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32`, the value is always ≥ 33 > 2, so the guard fires unconditionally. `timeout_count` is never accumulated — it is always overwritten and immediately reset to 0.

The call path is: `remove_by_block` (line 785) evaluates `should_punish = self.download_schedulers.len() > self.protect_num` (default `protect_num = 4`). When a block response falls in `NormalToUpper` or `UpperToMax` time quantile and `should_punish` is true, `set.decrease(1)` or `set.decrease(2)` is called. Each such call unconditionally decrements `task_count`. When `task_count` reaches 0, `prune` (lines 692–700) inserts the peer into `disconnect_list`.

Existing guards are insufficient: `should_punish` only requires more than 4 download peers (standard mainnet), and `adjustment` defaults to `true`. There is no other check preventing the unconditional decrement.

## Impact Explanation
This is a correctness degradation of CKB's sync peer management mechanism. Peers are penalised 3× more aggressively than the protocol intends: disconnected after 32 slow responses instead of ~96+. Legitimately slow peers (high-latency links, loaded CPUs) are prematurely evicted, degrading the node's peer set quality and sync throughput. This fits **Low (501–2000 points) — Any other important performance improvements for CKB**, with a secondary path toward sync stalls on nodes with limited peer counts.

## Likelihood Explanation
- Any unprivileged sync peer can trigger this by responding to `GetBlocks` with valid blocks in the `NormalToUpper` or `UpperToMax` time quantile (above `normal_time`, below `BLOCK_DOWNLOAD_TIMEOUT = 30s`).
- `should_punish` is satisfied on any node with more than 4 download peers — standard mainnet conditions.
- `adjustment` defaults to `true`.
- No special keys, operator access, or majority hashpower required.
- The condition fires on every qualifying slow response, making it trivially repeatable.

## Recommendation
Change line 522 in `sync/src/types/mod.rs` from:

```rust
self.timeout_count = self.task_count.saturating_add(num);
```

to:

```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

This restores the intended accumulator semantics: `task_count` is only decremented after the accumulated slow-response count exceeds 2.

## Proof of Concept
Unit test plan:
1. Construct a `DownloadScheduler` with default `task_count = 32` and `timeout_count = 0`.
2. Call `decrease(1)` once.
3. Assert `task_count == 31` — demonstrates unconditional decrement (bug present).
4. Assert `timeout_count == 0` — demonstrates accumulator is never used.
5. With the fix applied, assert `task_count == 32` (no decrement on first call) and `timeout_count == 1` (accumulator incremented).
6. Call `decrease(1)` two more times; assert `task_count` decrements to 31 only on the third call (after `timeout_count` exceeds 2).
