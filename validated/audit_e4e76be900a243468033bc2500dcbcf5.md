### Title
`PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` Hardcoded to 1 Hour Causes Permanent Update Failure for Non-Crypto (Equity/Commodity) Price Feed Subscriptions - (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler` contract enforces a hardcoded `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` against the maximum `publishTime` found in any submitted price update batch. Any subscription composed entirely of non-crypto price feeds (equities, commodities, FX) — whose Pyth `publishTime` reflects the last trading period — will permanently revert with `TimestampTooOld` for the entire duration the underlying market is closed, which can be 16+ consecutive hours per day.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint64 public constant PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours;
``` [1](#0-0) 

In `_validateShouldUpdatePrices`, the contract computes `updateTimestamp` as the **maximum** `publishTime` across all price feeds in the submitted batch, then rejects the entire update if that maximum is older than `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`:

```solidity
uint256 minAllowedTimestamp = (block.timestamp > PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    : 0;

if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(updateTimestamp, block.timestamp);
}
``` [2](#0-1) 

The developers were aware that closed-market feeds carry stale `publishTime` values — they deliberately set `minPublishTime = 0` when calling `parsePriceFeedUpdatesWithConfig` to avoid rejecting such data at the Pyth layer:

```solidity
// Note: We don't want to reject update data if it contains a price
// from a market that closed a few days ago...
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0, // We enforce the past max validity ourselves
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
    ...
);
``` [3](#0-2) 

However, the subsequent self-enforced check using `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hour` contradicts this intent. When **all** feeds in a subscription are from a closed market (e.g., a subscription tracking only AAPL/USD and TSLA/USD), the maximum `publishTime` across all feeds equals the last market-close timestamp. Once more than 1 hour has elapsed since market close, every `updatePriceFeeds` call reverts.

`_validateSubscriptionParams` imposes **no upper bound** on `heartbeatSeconds`:

```solidity
if (
    params.updateCriteria.updateOnHeartbeat &&
    params.updateCriteria.heartbeatSeconds == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [4](#0-3) 

A user legitimately setting `heartbeatSeconds = 86400` for a daily equity subscription will find the subscription completely non-functional for ~16.5 hours every weekday and all weekend.

---

### Impact Explanation

- Any active subscription whose price feed set consists entirely of non-crypto assets (equities, commodities, FX with market hours) becomes **permanently unable to update** for the entire off-hours window — up to 16.5 hours on weekdays and 65+ hours over weekends.
- The subscription's balance is locked and cannot be productively used during this window.
- Downstream consumers reading prices via `getPricesNoOlderThan` will receive `StalePrice` reverts for the entire off-hours period, breaking any protocol that depends on those prices.
- The subscription manager paid to create and fund the subscription but receives no service for the majority of each day.

---

### Likelihood Explanation

- Pyth explicitly supports equity, commodity, and FX price feeds. Subscriptions for these asset classes are a primary use case of the Scheduler/Pulse product.
- US equity markets are closed ~16.5 hours/day on weekdays and all weekend. Any subscription with only equity feeds hits this condition daily.
- No privileged access is required: any user calling `createSubscription` with equity feed IDs and any `heartbeatSeconds > 0` triggers this path.
- The keeper (Argus) fetching fresh Pyth data from Hermes for a closed market will receive the last-known price with `publishTime = last trade time`, which is exactly the stale timestamp that triggers the revert.

---

### Recommendation

1. **Remove or raise `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`** for the `TimestampTooOld` check, or make it configurable per subscription so that subscriptions with non-crypto feeds can specify a longer validity window matching the asset's actual market hours.
2. Alternatively, **skip the `TimestampTooOld` check entirely** (since `minPublishTime = 0` is already passed to the Pyth parser) and rely solely on the `TimestampOlderThanLastUpdate` check to prevent replay. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` guard was intended to prevent very stale updates, but it conflicts with the closed-market design.
3. At minimum, **add a validation in `_validateSubscriptionParams`** that rejects `heartbeatSeconds > PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` with a clear error, so users are not silently given a broken subscription.

---

### Proof of Concept

1. Deploy `Scheduler` (inherits `SchedulerConstants` with `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours`).
2. Call `createSubscription` with:
   - `priceIds = [AAPL_USD_FEED_ID]` (an equity feed)
   - `updateCriteria.updateOnHeartbeat = true`, `heartbeatSeconds = 86400`
3. At `t = 0`, keeper calls `updatePriceFeeds` with Pyth data where `publishTime = block.timestamp`. Succeeds.
4. Advance `block.timestamp` by 3601 seconds (1 hour + 1 second). The underlying equity market is now closed; the latest Pyth data for AAPL/USD still has `publishTime = T0` (market close time).
5. Keeper calls `updatePriceFeeds` with the latest available Pyth data (`publishTime = T0`).
6. Inside `_validateShouldUpdatePrices`:
   - `updateTimestamp = T0`
   - `minAllowedTimestamp = block.timestamp - 3600 = T0 + 1`
   - `updateTimestamp (T0) < minAllowedTimestamp (T0 + 1)` → **`TimestampTooOld` revert**
7. This revert persists for every subsequent call until the equity market reopens (~16 hours later), even though the heartbeat condition (`86400s`) has not yet been met and the subscription is fully funded. [5](#0-4) [6](#0-5)
