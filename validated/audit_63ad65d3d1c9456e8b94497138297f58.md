### Title
Inconsistent `script_len_range` Upper-Bound Semantics Between `get_cells` and `get_cells_capacity` — (`File: util/indexer/src/service.rs`)

---

### Summary

The `get_cells` and `get_cells_capacity` RPC handlers in the CKB indexer apply the `script_len_range` filter with different boundary semantics. `get_cells` correctly treats the upper bound as **exclusive** (`script_len >= r1` → exclude), while `get_cells_capacity` incorrectly treats it as **inclusive** (`script_len > r1` → exclude). This means `get_cells_capacity` silently includes cells whose script length equals exactly `r1`, cells that `get_cells` would exclude. Any RPC caller relying on both endpoints with the same filter will observe inconsistent results.

---

### Finding Description

The public documentation for `script_len_range` states the semantics are `[inclusive, exclusive]`:

> `script_len_range: [u64; 2], filter cells by script len range, [inclusive, exclusive]`

In `get_cells`, the filter is applied correctly:

```rust
// util/indexer/src/service.rs  ~line 303
if script_len < r0 || script_len >= r1 {   // exclusive upper bound ✓
    return None;
}
```

In `get_cells_capacity`, the same filter uses a different operator:

```rust
// util/indexer/src/service.rs  ~line 780
if script_len < r0 || script_len > r1 {    // inclusive upper bound ✗
    return None;
}
```

The `>` vs `>=` discrepancy means that for any query with `script_len_range = [a, b]`:

- `get_cells` returns cells where `a ≤ script_len < b`
- `get_cells_capacity` sums capacity for cells where `a ≤ script_len ≤ b`

The extra cells (those with `script_len == b`) are counted in the capacity total but are invisible to `get_cells`. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Any application or wallet that:
1. Calls `get_cells_capacity` with a `script_len_range` filter to determine available capacity, then
2. Calls `get_cells` with the same filter to enumerate the actual cells

will observe a capacity figure that is **higher** than the sum of capacities of the returned cells. The discrepancy equals the total capacity of all live cells whose script length equals exactly `r1`. This can cause:

- Incorrect balance displays (overstated available capacity)
- Applications permitting cell-creation operations that will fail on-chain because the actual spendable capacity is less than reported
- Incorrect fee estimation or capacity reservation logic in dApps

The `get_cells_capacity` endpoint is a publicly accessible RPC callable by any unprivileged RPC user. [3](#0-2) 

---

### Likelihood Explanation

The `script_len_range` filter is a documented, supported parameter of the indexer RPC. Any application that uses `get_cells_capacity` with this filter to compute balances or capacity budgets is affected. The bug is silent — no error is returned, only a subtly wrong number — so it is unlikely to be caught without explicit cross-validation against `get_cells`.

---

### Recommendation

Change the comparison in `get_cells_capacity` to match `get_cells` and the documented `[inclusive, exclusive]` semantics:

```rust
// util/indexer/src/service.rs  ~line 780 and ~line 790
// Change:
if script_len < r0 || script_len > r1 {
// To:
if script_len < r0 || script_len >= r1 {
```

Apply the same fix to both the `Lock` and `Type` branches inside `get_cells_capacity`. [2](#0-1) 

---

### Proof of Concept

1. Index a chain that contains at least one live cell whose lock script serialized length is exactly `N` bytes.
2. Call `get_cells` with `script_len_range: [0, N]` — the cell is **excluded** (correct, upper bound exclusive).
3. Call `get_cells_capacity` with the same `script_len_range: [0, N]` — the cell's capacity **is included** (incorrect, upper bound treated as inclusive).
4. Observe that `get_cells_capacity.capacity > sum(get_cells.objects[*].output.capacity)`.

The discrepancy is exactly the capacity of all live cells with `script_len == N`, demonstrating the inconsistent boundary semantics between the two handlers. [4](#0-3) [5](#0-4)
