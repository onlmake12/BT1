### Title
Malicious Keeper Can Drain Subscription Balance via Unbounded `tx.gasprice` in `updatePriceFeeds` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.updatePriceFeeds` is callable by any unprivileged address with no access control. The keeper reimbursement is computed as `(gasUsed + GAS_OVERHEAD) * tx.gasprice`, where `tx.gasprice` is entirely controlled by the caller. A malicious keeper can set an artificially inflated `tx.gasprice` to drain a subscription's entire deposited balance in a single valid update, causing the subscription to immediately run out of funds and stop receiving price updates.

---

### Finding Description

`updatePriceFeeds` has no caller restriction:

```solidity
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {          // ← no onlyKeeper / access control
``` [1](#0-0) 

Inside `_processFeesAndPayKeeper`, the keeper fee is:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [2](#0-1) 

There is no cap on `tx.gasprice`. The only guard is:

```solidity
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [3](#0-2) 

A malicious keeper can:

1. Observe the subscription's `balanceInWei` on-chain (public storage).
2. Simulate `updatePriceFeeds` to estimate `gasUsed`.
3. Solve for the exact `tx.gasprice` that makes `totalKeeperFee ≈ balanceInWei`:
   `tx.gasprice = (balanceInWei − keeperSpecificFee) / (gasUsed + GAS_OVERHEAD)`
4. Submit the transaction at that gas price once update conditions are met.

The keeper pays `tx.gasprice × gasUsed` in actual gas but receives `(gasUsed + GAS_OVERHEAD) × tx.gasprice + keeperSpecificFee` from the subscription. Net profit = `GAS_OVERHEAD × tx.gasprice + keeperSpecificFee`. By inflating `tx.gasprice`, the attacker profits from the overhead multiplier while draining the subscription in one shot.

The subscription balance is credited by the manager at `createSubscription` / `addFunds`:

```solidity
status.balanceInWei = msg.value;
``` [4](#0-3) 

and the keeper is paid directly from it:

```solidity
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [5](#0-4) 

The manager's deposited funds are used as the source of payment for an attacker-controlled amount — the direct analog of `safeTransferFrom(_loan.borrower, …)` in the reference report.

---

### Impact Explanation

A malicious keeper drains the subscription's entire ETH balance in a single transaction the moment update conditions are satisfied. The subscription immediately falls below the minimum balance and stops receiving price updates. Any DeFi protocol or application relying on that subscription's push feed loses access to fresh prices, potentially enabling stale-price exploits downstream. The manager loses all deposited funds with no recourse.

---

### Likelihood Explanation

Medium. The attack requires:
- Update conditions to be met at least once (heartbeat or deviation threshold — both are routine).
- The attacker to estimate gas usage (trivially done via `eth_estimateGas` or a local simulation).
- No special privilege, registration, or token holding.

Subscription balances and parameters are fully public on-chain. Any keeper node operator can execute this attack profitably on any subscription.

---

### Recommendation

Cap the effective gas price used in the fee calculation to a governance-controlled maximum:

```solidity
uint256 effectiveGasPrice = Math.min(tx.gasprice, _state.maxKeeperGasPriceInWei);
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

`maxKeeperGasPriceInWei` should be set by the admin to a value that covers legitimate high-gas-price conditions (e.g., 500 gwei) but prevents artificial inflation. Alternatively, reimburse keepers at `block.basefee + maxPriorityFeePerGas` capped at a reasonable premium rather than the raw `tx.gasprice`.

---

### Proof of Concept

```solidity
// Attacker is a keeper with no special privileges.
// Subscription has 1 ETH balance, heartbeat = 60s, 2 price feeds.
// GAS_OVERHEAD = 50_000 (example), singleUpdateKeeperFeeInWei = 0.001 ETH.

uint256 subscriptionBalance = 1 ether;
uint256 keeperSpecificFee   = 0.001 ether * 2; // 2 feeds
uint256 estimatedGasUsed    = 200_000;          // from eth_estimateGas
uint256 gasOverhead         = 50_000;

// Solve for draining gas price:
uint256 drainGasPrice = (subscriptionBalance - keeperSpecificFee)
                        / (estimatedGasUsed + gasOverhead);
// ≈ 3_996 gwei

// After 60 seconds, attacker calls:
vm.txGasPrice(drainGasPrice);
scheduler.updatePriceFeeds(subscriptionId, validUpdateData);

// Result:
// - subscription.balanceInWei ≈ 0  (drained)
// - attacker received ≈ 1 ETH from subscription
// - attacker paid ≈ 0.799 ETH in actual gas (200k × 3996 gwei)
// - attacker net profit ≈ 0.201 ETH (GAS_OVERHEAD × drainGasPrice + keeperSpecificFee)
// - subscription deactivated / starved of funds
```
