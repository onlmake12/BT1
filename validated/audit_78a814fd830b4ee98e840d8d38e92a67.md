Audit Report

## Title
Stale `descendants_*` Accounting in `remove_entry_and_descendants` Leaves Ancestor Entries with Inflated Eviction Keys — (File: tx-pool/src/component/pool_map.rs)

## Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link entries from `self.links` in Phase 1 before calling `remove_entry` in Phase 2. Because `update_ancestors_index_key` resolves ancestors via `self.links.calc_ancestors`, and those links are already gone by Phase 2, the ancestor set is always empty. Surviving ancestor entries never receive the `sub_descendant_weight` call, leaving their `descendants_fee`, `descendants_size`, `descendants_cycles`, `descendants_count`, and `evict_key` permanently inflated. This corrupts eviction ordering in `limit_size`, allowing zombie low-fee ancestors to accumulate and causing legitimate transactions to be rejected with `Reject::Full`.

## Finding Description

**Phase 1 destroys the link graph before Phase 2 can use it.**

`remove_entry_and_descendants` (pool_map.rs L252–265) first iterates all entries in `removed_ids` and calls `remove_entry_links` on each: [1](#0-0) 

`remove_entry_links` (L418–430) calls `self.links.remove(id)` at L429, which deletes the entry's record from `TxLinksMap::inner` entirely: [2](#0-1) 

Phase 2 then calls `remove_entry` for each id (L261–264). Inside `remove_entry` (L235–250), the first call is `update_ancestors_index_key`: [3](#0-2) 

`update_ancestors_index_key` (L432–445) resolves ancestors via `self.links.calc_ancestors`: [4](#0-3) 

`calc_ancestors` (links.rs L78–80) calls `calc_relative_ids`, which does `self.inner.get(short_id)`: [5](#0-4) 

Because the entry was removed from `self.links` in Phase 1, `self.inner.get(short_id)` returns `None`, `direct` is an empty set, and `calc_ancestors` returns an empty `HashSet`. The loop body in `update_ancestors_index_key` that calls `e.inner.sub_descendant_weight(child)` and updates `e.evict_key` is never reached for any external ancestor.

**Consequence — stale fields on surviving ancestors.**

`sub_descendant_weight` (entry.rs L133–142) decrements `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee`. None of these decrements happen: [6](#0-5) 

`EvictKey` (entry.rs L234–247) is computed from those stale fields: [7](#0-6) 

The comment in the code (`// update links state for remove, so that we won't update_descendants_index_key in remove_entry`) documents the intent of Phase 1 — to suppress `update_descendants_index_key` for entries being removed. However, it inadvertently also suppresses `update_ancestors_index_key` for entries that **remain** in the pool. There is no compensating update for external ancestors anywhere in the function.

The existing test `test_remove_entry_and_descendants` (score_key.rs L171–230) only asserts that removed entries are gone and that the descendants set is empty — it never checks that surviving ancestors have correct `descendants_count` or `descendants_fee`: [8](#0-7) 

## Impact Explanation

`limit_size` (pool.rs L292–329) evicts entries by calling `next_evict_entry`, which iterates `entries` ordered by `evict_key` ascending (lowest fee-rate first): [9](#0-8) 

`next_evict_entry` (pool_map.rs L380–385) finds the first entry with the given status ordered by `evict_key`: [10](#0-9) 

`EvictKey` ordering compares `fee_rate` first (sort_key.rs L92–103): [11](#0-10) 

An ancestor whose high-fee descendants were removed still carries their fee contribution in `descendants_fee`, so its `evict_key.fee_rate` is inflated. It sorts as more valuable than it actually is and is skipped during eviction. The pool accumulates zombie low-fee ancestors that cannot be evicted, causing legitimate high-fee transactions to be rejected with `Reject::Full`.

This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The bug is reachable by an unprivileged user via at least two paths:

1. **`limit_size` → `remove_entry_and_descendants`** (pool.rs L307): Any tx submitter who fills the pool triggers eviction. If the evicted entry has a surviving parent, that parent's `evict_key` becomes stale. No mining power required.
2. **`resolve_conflict` → `remove_entry_and_descendants`** (pool_map.rs L285, L310, L321): Any committed block that spends an input already in the pool triggers this path.

The attack is repeatable: each cycle of submit-parent + submit-child + trigger-removal-of-child leaves one more zombie ancestor in the pool. Cost is proportional to the number of transactions submitted, which is low relative to pool capacity.

## Recommendation

Before Phase 1 pre-removes links, collect the set of **external ancestors** — ancestors of the root entry that are not themselves in `removed_ids` — and for each entry being removed, call `sub_descendant_weight` and update `evict_key` on those external ancestors directly. Alternatively, restructure the function to call `update_ancestors_index_key` on the root entry only (before its links are removed), since descendants' ancestor fields need not be updated as they are being removed. The key invariant to restore: every entry that remains in the pool after `remove_entry_and_descendants` must have accurate `descendants_*` fields and a correct `evict_key`.

## Proof of Concept

Add the following test to `tx-pool/src/component/tests/score_key.rs` following the pattern of `test_remove_entry`:

```rust
#[test]
fn test_remove_entry_and_descendants_stale_ancestor() {
    let mut map = PoolMap::new(DEFAULT_MAX_ANCESTORS_COUNT);
    // tx_A: low fee parent
    let tx_a = TxEntry::dummy_resolve(
        TransactionBuilder::default().build(),
        100,
        Capacity::shannons(1),   // fee = 1 shannon
        100,
    );
    // tx_B: high fee child of tx_A
    let tx_b = TxEntry::dummy_resolve(
        TransactionBuilder::default()
            .input(CellInput::new_builder()
                .previous_output(OutPoint::new_builder()
                    .tx_hash(tx_a.transaction().hash())
                    .index(0u32.pack())
                    .build())
                .build())
            .witness(Bytes::new())
            .build(),
        200,
        Capacity::shannons(1000), // fee = 1000 shannons
        100,
    );
    let tx_a_id = tx_a.proposal_short_id();
    let tx_b_id = tx_b.proposal_short_id();

    map.add_proposed(tx_a).unwrap();
    map.add_proposed(tx_b).unwrap();

    // After adding tx_B, tx_A should have descendants_count=2, descendants_fee=1001
    let a = map.get(&tx_a_id).unwrap();
    assert_eq!(a.descendants_count, 2);
    assert_eq!(a.descendants_fee, Capacity::shannons(1001));

    // Remove tx_B (and its descendants, none here)
    map.remove_entry_and_descendants(&tx_b_id);
    assert!(!map.contains_key(&tx_b_id));
    assert!(map.contains_key(&tx_a_id));

    // tx_A should now reflect only itself: descendants_count=1, descendants_fee=1
    let a = map.get(&tx_a_id).unwrap();
    assert_eq!(a.descendants_count, 1);  // FAILS on current code (returns 2)
    assert_eq!(a.descendants_fee, Capacity::shannons(1)); // FAILS on current code (returns 1001)
}
```

This test will fail on the current code, confirming the bug. The stale `evict_key` on tx_A means it will be skipped by `next_evict_entry` during pool eviction, allowing zombie low-fee ancestors to accumulate and causing `Reject::Full` for legitimate transactions.

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
