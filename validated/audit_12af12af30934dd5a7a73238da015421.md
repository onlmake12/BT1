### Title
`Status::Gap` Silently Collapsed to `Pending` in `get_transaction` RPC, Breaking Expected Transaction State Machine â€” (File: `tx-pool/src/service.rs`)

---

### Summary

The CKB tx-pool internally tracks **three** transaction states (`Pending`, `Gap`, `Proposed`), but the `get_transaction` RPC only exposes **two** (`Pending`, `Proposed`). Transactions in the `Gap` state â€” which have already been proposed in a block's proposal zone but are not yet in the committable window â€” are silently reported as `Pending` to all external callers. This means a transaction can appear to jump from `Pending` directly to `Committed` without ever showing `Proposed`, breaking the state machine that integrating protocols depend on. Additionally, the `tx_pool_info` RPC omits the gap count entirely, even though it is tracked internally.

---

### Finding Description

**Internal three-state model vs. two-state RPC exposure**

`PoolMap` in `tx-pool/src/component/pool_map.rs` defines three distinct statuses:

```rust
pub enum Status {
    Pending,
    Gap,
    Proposed,
}
``` [1](#0-0) 

`Gap` represents a transaction whose proposal short ID has appeared in a block's proposal zone, but the proposal window (`w_close`) has not yet elapsed, so the transaction is not yet committable. The pool tracks this count separately: [2](#0-1) 

**Both RPC handlers collapse `Gap` â†’ `Pending`**

In `tx-pool/src/service.rs`, the `GetTxStatus` message handler (used by `get_transaction` verbosity=1):

```rust
let status = if status == &Status::Proposed {
    TxStatus::Proposed
} else {
    TxStatus::Pending   // Gap silently becomes Pending
};
``` [3](#0-2) 

The `GetTransactionWithStatus` handler (verbosity=2) has the same collapse:

```rust
let (tx_status, min_replace_fee) = if status == &Status::Proposed {
    (TxStatus::Proposed, None)
} else {
    (TxStatus::Pending, tx_pool.min_replace_fee(entry))  // Gap silently becomes Pending
};
``` [4](#0-3) 

**Missing gap count in `tx_pool_info`**

The `info()` method that backs the `tx_pool_info` RPC exposes `pending_size` and `proposed_size` but has no `gap_size` field, even though `gap_count` is maintained internally: [5](#0-4) 

The `TxStatus` enum exposed to JSON-RPC callers has no `Gap` variant at all: [6](#0-5) 

---

### Impact Explanation

CKB uses a two-phase commit: a transaction must first be **proposed** (appear in a block's proposal zone), then after `w_close` blocks it enters the committable window and can be **committed**. The observable state machine that integrating protocols (wallets, exchanges, bridges) rely on is:

```
Pending â†’ Proposed â†’ Committed
```

Because `Gap` is collapsed to `Pending`, the actual observable sequence for any transaction that passes through the gap window is:

```
Pending (truly pending) â†’ Pending (actually Gap/proposed) â†’ Committed
```

The `Proposed` status is **never observed** by external callers for transactions that are committed before the caller polls again after the gap window closes. This means:

1. A protocol that waits for `Proposed` status before releasing funds or taking an action may never see it, causing stuck workflows or incorrect error handling.
2. A monitoring system cannot distinguish a transaction that has been proposed (and will soon be committed) from one that has not been proposed at all â€” both appear as `Pending`.
3. The missing `gap_size` in `tx_pool_info` means operators and integrators have no visibility into how many transactions are in this intermediate state.

This is directly analogous to the ChainlinkAtlasWrapper issue: the same external identifier (`Pending`) maps to two distinct internal states, and a key piece of interface information (`gap_size`, analogous to `decimals()`/`getRoundData()`) is entirely absent from the exposed API.

---

### Likelihood Explanation

The gap window is a normal, mandatory part of every CKB transaction lifecycle. Any transaction that is proposed and then committed will pass through `Gap` state. Any RPC caller â€” wallet, exchange, bridge, or script author â€” that queries `get_transaction` during this window receives misleading status. No special privileges or network position are required; a standard RPC call suffices.

---

### Recommendation

1. **Expose `Gap` as a distinct status** in the `get_transaction` RPC response (add a `gap` variant to `TxStatus` in `util/types/src/core/tx_pool.rs` and propagate it through `util/jsonrpc-types/src/blockchain.rs`).
2. **Add `gap_size` to `tx_pool_info`** so operators and integrators can observe the gap pool size.
3. **Document the three-phase lifecycle** (`Pending` â†’ `Gap` â†’ `Proposed` â†’ `Committed`) clearly in the RPC documentation so integrators know to handle the gap state.

---

### Proof of Concept

1. Submit a transaction `tx` to the node via `send_transaction`.
2. Mine a block that includes `tx`'s proposal short ID in its proposal zone (but does not commit `tx`). At this point `tx` enters `Gap` state internally.
3. Call `get_transaction(tx.hash)` â€” the response shows `status: "pending"`, even though `tx` has already been proposed.
4. Mine `w_close` more blocks. `tx` transitions internally to `Proposed`, then is committed.
5. Call `get_transaction(tx.hash)` again â€” the response now shows `status: "committed"`.
6. The `status: "proposed"` state was **never visible** to the external caller, breaking the expected `Pending â†’ Proposed â†’ Committed` state machine.

The `GetTxStatus` handler at `tx-pool/src/service.rs:879â€“883` is the root cause: it maps any non-`Proposed` pool status (including `Gap`) to `TxStatus::Pending`, making the gap state invisible to all RPC consumers. [7](#0-6)
