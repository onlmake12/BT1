### Title
L2 Address Aliasing Not Considered in `Entropy::reveal()` Blocks Contract Requesters During Sequencer Downtime - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol::reveal()` enforces `req.requester == msg.sender` to prevent front-running. When a contract calls `request()` on an L2 rollup (Arbitrum, Optimism — both confirmed deployment targets), `req.requester` is stored as the contract's normal address. If the L2 sequencer goes down and the contract attempts to call `reveal()` via L1 forced inclusion, the EVM aliases `msg.sender` by adding `0x1111000000000000000000000000000000001111`. The stored `req.requester` no longer matches, causing an `Unauthorized` revert and permanently blocking the contract from fulfilling its in-flight randomness request until the sequencer recovers.

---

### Finding Description

In `requestHelper`, the requester is stored as the raw `msg.sender` at request time: [1](#0-0) 

In `reveal()`, the caller is validated against that stored address: [2](#0-1) 

The comment on `reveal()` explicitly states this check exists to prevent front-running: [3](#0-2) 

On Arbitrum and Optimism, when a contract submits a transaction via L1 forced inclusion (bypassing the sequencer), the EVM applies address aliasing: `msg.sender = original_address + 0x1111000000000000000000000000000000001111`. The stored `req.requester` holds the un-aliased address, so the equality check fails.

Critically, there is no escape hatch: `revealWithCallback()` explicitly rejects non-callback requests: [4](#0-3) 

So a contract using the non-callback `request()`/`reveal()` flow is completely blocked from fulfilling its request via forced inclusion.

The Entropy contract is confirmed deployed on both affected L2 chains: [5](#0-4) 

---

### Impact Explanation

A contract that called `request()` on Arbitrum or Optimism has its in-flight randomness request permanently frozen for the duration of sequencer downtime. The contract cannot call `reveal()` via forced inclusion because the aliased `msg.sender` does not match `req.requester`. Since `revealWithCallback()` is gated to callback-mode requests only, there is no alternative fulfillment path. Any protocol logic depending on the random number (e.g., lottery resolution, NFT reveal, game outcome) is blocked. The paid fee is also locked in the contract for the duration.

---

### Likelihood Explanation

Entropy is live on Arbitrum One and Optimism mainnet. Both chains have experienced sequencer outages historically. Any contract integrator using the non-callback `request()`/`reveal()` flow (which is a documented, supported interface in `IEntropy`) is affected. The forced-inclusion mechanism is a standard L2 liveness guarantee that integrators may reasonably rely on for time-sensitive operations.

---

### Recommendation

In `reveal()`, additionally accept the aliased version of `req.requester` as a valid caller:

```solidity
address aliasedRequester = address(
    uint160(req.requester) + uint160(0x1111000000000000000000000000000000001111)
);
if (req.requester != msg.sender && aliasedRequester != msg.sender) {
    revert EntropyErrors.Unauthorized();
}
```

Alternatively, extract the aliasing logic into a shared library and apply it consistently across all `msg.sender`-vs-stored-address checks in the codebase (including `Scheduler.sol::onlyManager` and `onlyWhitelistedReader`). [6](#0-5) 

---

### Proof of Concept

1. Protocol X deploys a contract `C` on Arbitrum that calls `entropy.request(provider, userRandomNumber)` — `req.requester` is stored as `address(C)`.
2. The Arbitrum sequencer goes down before the provider reveals.
3. Protocol X attempts to call `entropy.reveal(provider, seqNum, userContrib, providerContrib)` from contract `C` via L1 forced inclusion.
4. The EVM sets `msg.sender = address(C) + 0x1111000000000000000000000000000000001111`.
5. The check `req.requester != msg.sender` evaluates to `true` → `revert EntropyErrors.Unauthorized()`.
6. The randomness request remains permanently blocked until the sequencer recovers. Any time-sensitive protocol action depending on this random number (e.g., a lottery that must resolve before a deadline) is DoS'd for the full duration of the outage.
