Audit Report

## Title
`check_tx_fee` Minimum Fee Rate Admission Uses Only Serialized Size, Allowing Cycle-Heavy Transactions to Bypass the Threshold — (`tx-pool/src/util.rs`)

## Summary

`check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, while the true transaction weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` returns the actual cycle count, no second fee-rate gate is applied. A cycle-heavy, byte-small transaction can therefore enter the pool while paying an actual fee rate up to ~119× below the configured `min_fee_rate`, forcing full script verification (up to 70M cycles) on every receiving and relaying node at a fraction of the intended cost.

## Finding Description

**Root cause — size-only admission gate:**

`check_tx_fee` in `tx-pool/src/util.rs` (lines 42–45) explicitly uses `tx_size` as the weight denominator, with a comment acknowledging the limitation:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The true weight formula is defined in `util/types/src/core/tx_pool.rs` (lines 298–303):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

**No post-verification re-check:**

In `_process_tx` (`tx-pool/src/process.rs`), the flow is:
1. `pre_check` → calls `check_tx_fee` with only `tx_size` (lines 289, 294).
2. `verify_rtx` → returns `Completed` with actual `verified.cycles` (line 734).
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is created at line 751 — **no fee-rate re-check with actual weight occurs**.

`TxEntry::fee_rate()` (entry.rs lines 114–118) does correctly compute the weight-aware fee rate using `get_transaction_weight(self.size, self.cycles)`, but this is only used for eviction ordering and block assembly — never as an admission gate.

**Exploit path:**

An attacker with valid UTXOs and a deployed high-cycle script cell:
1. Crafts a transaction: serialized size ≈ 200 bytes, type script consuming ~70M cycles.
2. Submits via `send_transaction` RPC (no `declared_cycles` required; `max_cycles` defaults to `max_block_cycles()` = 70M).
3. `check_tx_fee` computes `min_fee = fee(200) = 200 shannons`. Fee ≥ min_fee → **accepted**.
4. `verify_rtx` executes ~70M cycles. Actual weight ≈ 11,940. Actual fee rate ≈ `200 × 1,000 / 11,940 ≈ 16 shannons/KW` — ~62× below the 1,000 shannons/KW threshold.
5. Transaction enters the pool. Repeat with fresh UTXOs.

**Existing guards are insufficient:**

- The 180 MB pool size cap (`max_tx_pool_size`) bounds memory but does not prevent sustained CPU exhaustion.
- The `max_tx_verify_cycles = 70_000_000` limit bounds per-transaction CPU cost but is precisely what the attacker exploits to maximize the size/weight discrepancy.
- The `declared_cycles` mismatch check (lines 736–748) only rejects transactions where the caller explicitly declares a wrong cycle count; it does not enforce a weight-based fee floor.

## Impact Explanation

**High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker submitting cycle-heavy, byte-small transactions:
1. Passes `check_tx_fee` by paying only the size-based minimum (e.g., 200 shannons).
2. Forces every receiving node to execute up to 70M cycles of script verification per transaction — significant CPU load.
3. Because `check_tx_fee` is also the admission gate on relay peers, these transactions propagate across the network, multiplying the CPU cost.
4. Fills the pool with entries whose actual fee rate is far below `min_fee_rate`, degrading pool quality for miners.

## Likelihood Explanation

The attack requires only a valid RPC connection or relay peer path — no privileged access. The attacker needs valid UTXOs and a deployed script cell that consumes near-maximum cycles. The fee cost is ~62–125× lower than the configured threshold, making sustained attack economically viable. The `max_tx_verify_cycles` limit (70M) is the only natural bound on the per-transaction discrepancy.

## Recommendation

After `verify_rtx` returns the actual cycle count in `_process_tx`, add a weight-aware fee rate check before creating the `TxEntry`. In `tx-pool/src/process.rs`, between lines 734 and 751:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_min_fee = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, actual_min_fee.as_u64(), fee.as_u64())), snapshot));
}
```

This ensures the admission gate enforces the same weight formula used for eviction and block assembly.

## Proof of Concept

1. Deploy a type script cell that loops to consume ~70M cycles.
2. Craft a transaction: serialized size ≈ 200 bytes, one input, one output, the type script as cell dep. Fee = `1,000 × 200 / 1,000 = 200 shannons`.
3. Submit via `send_transaction` RPC (no `declared_cycles` required).
4. `check_tx_fee` computes `min_fee = fee(200) = 200 shannons`. Fee ≥ min_fee → **accepted**.
5. `verify_rtx` executes ~70M cycles. Actual weight ≈ 11,940. Actual fee rate ≈ `200 × 1,000 / 11,940 ≈ 16 shannons/KW` — ~62× below the 1,000 shannons/KW threshold.
6. Transaction enters the pool. Repeat with fresh UTXOs to fill the pool with sub-threshold entries and impose sustained CPU load on all relay nodes.
