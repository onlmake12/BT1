### Title
`MAX_DEPOSIT_LIMIT` Bypassed for Permanent Subscriptions via Repeated `addFunds()` Calls — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.sol` contract enforces `MAX_DEPOSIT_LIMIT` (100 ETH) on permanent subscriptions by checking only the **incoming `msg.value`** of each individual call, not the **resulting cumulative balance**. Any caller can invoke `addFunds()` repeatedly with amounts ≤ `MAX_DEPOSIT_LIMIT` to push a permanent subscription's balance arbitrarily above the cap. Because permanent subscriptions prohibit withdrawals, all excess ETH is permanently locked in the contract.

---

### Finding Description

`SchedulerConstants.sol` declares the cap:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
``` [1](#0-0) 

`createSubscription()` enforces it correctly at creation time:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [2](#0-1) 

`addFunds()` applies the **same per-call check** — but never validates the post-addition cumulative balance:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

status.balanceInWei += msg.value;   // ← cumulative balance never capped
``` [3](#0-2) 

The missing check should be:

```solidity
if (params.isPermanent &&
    status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Because `withdrawFunds()` unconditionally reverts for permanent subscriptions:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [4](#0-3) 

every wei deposited above `MAX_DEPOSIT_LIMIT` is irrecoverably locked.

---

### Impact Explanation

1. **Invariant broken**: `balanceInWei ≤ MAX_DEPOSIT_LIMIT` for permanent subscriptions is violated.
2. **Permanent fund loss**: Any ETH deposited beyond the cap into a permanent subscription is unrecoverable — there is no withdrawal path.
3. **Griefing / accidental loss**: A user who calls `addFunds()` a second time on their own permanent subscription (e.g., to top it up) will silently exceed the cap and permanently lock the excess ETH with no error.

---

### Likelihood Explanation

The entry path is fully permissionless — `addFunds()` has no access control and accepts ETH from any caller:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
``` [5](#0-4) 

Any subscription manager (or even a third party, since `addFunds` is open to anyone) can trigger this. The scenario of a manager topping up their own permanent subscription is a normal, expected use-case, making accidental exploitation highly likely.

---

### Recommendation

Replace the per-call amount check in `addFunds()` with a post-addition cumulative balance check:

```solidity
if (params.isPermanent) {
    if (status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }
}
status.balanceInWei += msg.value;
```

Apply the same fix to `createSubscription()` for consistency (though it is correct there since `balanceInWei` starts at zero).

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription at exactly the cap
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);
// status.balanceInWei == 100 ether  ✓ (at cap)

// 2. Add funds again — msg.value (100 ether) == MAX_DEPOSIT_LIMIT, check passes
scheduler.addFunds{value: 100 ether}(subId);
// status.balanceInWei == 200 ether  ✗ (cap exceeded)

// 3. Attempt withdrawal — reverts unconditionally for permanent subscriptions
scheduler.withdrawFunds(subId, 100 ether);
// → CannotUpdatePermanentSubscription  (200 ether permanently locked)
```

The `MAX_DEPOSIT_LIMIT` invariant is broken after step 2, and the 200 ETH is permanently locked with no recovery path.
