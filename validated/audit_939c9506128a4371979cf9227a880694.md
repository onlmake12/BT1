### Title
Fee Accounting Mismatch Between Request and Callback Causes Permanent Fund Lock - (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, the fee stored at request time uses a fixed Echo protocol fee (`_state.pythFeeInWei`) as the Pyth fee estimate, but `executeCallback` pays the actual dynamic Pyth fee (`pyth.getUpdateFee(updateData)`) at callback time. If the actual Pyth fee exceeds the provider's portion of the stored fee, Solidity 0.8+ checked arithmetic causes an underflow revert in `executeCallback`, permanently locking user funds with no recovery path.

### Finding Description

**At request time** (`requestPriceUpdatesWithCallback`), the stored provider fee is computed as:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
```

`_state.pythFeeInWei` is Echo's own fixed protocol fee â€” it is **not** the Pyth contract's dynamic update fee. [1](#0-0) 

**At callback time** (`executeCallback`), the actual Pyth fee is fetched dynamically and paid:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(...);
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The provider credit computation is `(req.fee + msg.value_callback) - pythFee_actual`. Expanding `req.fee`:

```
(msg.value_request - _state.pythFeeInWei + msg.value_callback) - pythFee_actual
```

If `pythFee_actual > (msg.value_request - _state.pythFeeInWei + msg.value_callback)`, this **underflows and reverts** under Solidity 0.8+ checked arithmetic.

The `getFee()` function acknowledges this design tension with a comment:

> "Note: The provider needs to set its fees to include the fee charged by the Pyth contract." [3](#0-2) 

However, the Pyth contract's `singleUpdateFeeInWei` is governance-controlled and can change between the time a request is submitted and when the callback is executed. If it increases, all in-flight requests whose provider fees no longer cover the new Pyth fee become permanently unexecutable.

There is no cancellation, timeout, or refund mechanism in `Echo.sol` â€” `executeCallback` is the **only** path to clear a request. [4](#0-3) 

### Impact Explanation

User funds paid via `requestPriceUpdatesWithCallback` are permanently locked in the Echo contract. The provider cannot execute the callback (it reverts), the user cannot cancel or reclaim their fee, and no recovery function exists. This constitutes a direct loss of user funds for all in-flight requests at the time of a Pyth fee increase.

### Likelihood Explanation

Pyth governance can increase `singleUpdateFeeInWei` on the Pyth EVM contract at any time. Any pending Echo request submitted before the fee increase and fulfilled after it is vulnerable. The trigger is a normal, expected governance action â€” not an exotic attack. The more requests are in-flight, the larger the aggregate impact.

### Recommendation

At callback time, use the Pyth fee that was in effect at request time (store it in `req`) rather than re-querying it dynamically. Alternatively, store the actual Pyth fee at request time by calling `pyth.getUpdateFee` during `requestPriceUpdatesWithCallback` and saving it in the `Request` struct. This ensures the fee paid to Pyth at callback time matches what was collected from the user, eliminating the mismatch.

Additionally, implement a request cancellation/refund mechanism so users can recover funds if a callback becomes permanently unexecutable.

### Proof of Concept

1. Pyth governance sets `singleUpdateFeeInWei = 1 wei`. Echo admin sets `_state.pythFeeInWei = 1 wei`. Provider sets fees to cover 1 wei Pyth fee.
2. User calls `requestPriceUpdatesWithCallback{value: getFee(...)}`. `req.fee = msg.value - 1` is stored.
3. Pyth governance increases `singleUpdateFeeInWei` to `1000 wei`.
4. Provider calls `executeCallback` with no extra ETH (`msg.value = 0`).
5. `pythFee = pyth.getUpdateFee(updateData)` returns `1000 wei`.
6. `(req.fee + 0) - 1000` underflows â†’ transaction reverts.
7. The request remains active forever; user's ETH is permanently locked. [5](#0-4) [6](#0-5)
