Audit Report

## Title
Stale Descendant Weight After `remove_entry_and_descendants` Causes Incorrect Pool Eviction Ordering — (File: tx-pool/src/component/pool_map.rs)

## Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link-graph records before invoking `remove_entry` on each batch member. Because `update_ancestors_index_key` resolves ancestors through `calc_ancestors`, which reads the entry's own link record, the pre-deletion causes it to return an empty set. Surviving pool ancestors of the removed subtree therefore never receive a `sub_descendant_weight` call, leaving their `descendants_fee`, `descendants_size`, `descendants_cycles`, `descendants_count`, and derived `EvictKey.fee_rate` permanently inflated. This distorts pool eviction ordering, allowing low-fee transactions to survive while legitimate higher-fee transactions are dropped.

## Finding Description

**Root cause — link pre-removal before ancestor update:**

`remove_entry_and_descendants` collects the removal batch, strips all link records first, then calls `remove_entry` on each:

```rust

    self.remove_entry_links(id);   // deletes id's own HashMap record
}
```

`remove_entry_links` ends with `self.links.remove(id)` (line 429), which deletes the entry's own record from `TxLinksMap::inner`.

`remove_entry` then calls `update_ancestors_index_key` (line 242):


`calc_ancestors` → `calc_relative_ids` reads the entry's own link record:

```rust

    .map(|link| link.get_direct_ids(relation))
```

The empty set propagates through `calc_relation_ids`, the loop in `update_ancestors_index_key` never executes, and no surviving ancestor ever receives `sub_descendant_weight`.

**Concrete scenario (A → B → C, B replaced by B′ via RBF):**

1. A inserted: `descendants_fee(A) = fee_A`.
2. B inserted as child of A: `descendants_fee(A) += fee_B`.
3. C inserted as child of B: `descendants_fee(A) += fee_C`.
4. B′ conflicts with B; `process_rbf` calls `remove_entry_and_descendants(B)`.
   - `remove_entry_links(B)` and `remove_entry_links(C)` run first, deleting both records.
   - `remove_entry(B)` → `calc_ancestors(B)` → empty → A not updated.
   - `remove_entry(C)` → `calc_ancestors(C)` → empty → A not updated.
   - A's `descendants_fee` still equals `fee_A + fee_B + fee_C`.
5. B′ inserted as child of A: `update_ancestors_index_key(B′, Add)` increments A further.
   - A's `descendants_fee` becomes `fee_A + fee_B + fee_C + fee_B′` (correct: `fee_A + fee_B′`).
6. `EvictKey` for A is computed as `max(descendants_feerate, feerate)` (entry.rs lines 234-247). The inflated `descendants_fee` inflates `descendants_feerate`, inflating `EvictKey.fee_rate`.
7. `next_evict_entry` iterates `iter_by_evict_key()` ascending (lines 380-385); A's inflated `fee_rate` places it later in eviction order, so accurate-fee transactions are evicted first.

**Why the existing guard does not prevent this:**

The comment at line 256 (`// update links state for remove, so that we won't update_descendants_index_key in remove_entry`) confirms the pre-removal is intentional to suppress redundant descendant updates within the batch. However, it silently also suppresses the necessary ancestor updates for entries *outside* the batch. There is no compensating step to update surviving ancestors.

## Impact Explanation

The bug permanently inflates `EvictKey.fee_rate` for any pool entry that is an ancestor of a removed subtree. `limit_size` selects eviction victims by iterating `iter_by_evict_key()` ascending; an entry with an overstated fee-rate is skipped in favour of entries with accurate (lower) fee-rates. A low-fee transaction can therefore survive pool eviction while legitimate higher-fee transactions from other users are dropped. This constitutes a **suboptimal implementation of the CKB transaction-pool state storage mechanism**, matching **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation

- RBF is enabled by default (`min_rbf_rate = 1_500 > min_fee_rate = 1_000`).
- Any unprivileged submitter can craft the required A → B → C chain and submit a conflicting B′ with sufficient fee to pass RBF rules.
- The tracking corruption occurs unconditionally on every `remove_entry_and_descendants` call where the removed root has surviving pool ancestors; no special privileges, keys, or hash power are required.
- The eviction impact materialises when the pool approaches capacity, but the metric corruption itself is unconditional and compounds with each RBF cycle.

## Recommendation

Capture surviving ancestors of the root entry *before* any links are removed, then explicitly decrement their descendant weights for each removed entry after the batch removal:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Capture surviving ancestors BEFORE any links are removed
    let surviving_ancestors = self.links.calc_ancestors(id);

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    let removed: Vec<TxEntry> = removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect();

    // Decrement descendant weights on surviving ancestors for each removed entry
    for removed_entry in &removed {
        for anc_id in &surviving_ancestors {
            self.entries.modify_by_id(anc_id, |e| {
                e.inner.sub_descendant_weight(removed_entry);
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

    removed
}
```
