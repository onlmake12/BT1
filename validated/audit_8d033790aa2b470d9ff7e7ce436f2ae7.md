### Title
Unbounded `while` Loop in `executeCallback` Enables Gas-Exhaustion DoS, Permanently Locking Requester Funds — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` function contains an unbounded `while` loop that linearly scans every sequence number from `firstUnfulfilledSeq` to `currentSequenceNumber` after each fulfilled request. An unprivileged attacker can create a large number of requests and fulfill them out of order, forcing the loop to iterate over thousands of already-cleared slots when the lowest-sequence request is finally fulfilled. The resulting out-of-gas revert rolls back the entire transaction — including the `clearRequest` and provider-credit effects — permanently locking the victim requester's funds with no recovery path.

---

### Finding Description

In `executeCallback`, after clearing the fulfilled request and crediting the provider, the contract runs:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a
// storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs two storage reads: one into the fixed-size `_state.requests[shortKey]` array and, when the slot does not match, one into the `_state.requestsOverflow[key]` mapping. [2](#0-1) 

The loop has no upper-bound guard. Its termination depends entirely on how many consecutive inactive (already-fulfilled) sequence numbers exist starting from `firstUnfulfilledSeq`. Because `firstUnfulfilledSeq` only advances when the *lowest* outstanding sequence number is fulfilled, fulfilling all higher-numbered requests first keeps `firstUnfulfilledSeq` pinned at the lowest sequence number. When that lowest request is finally fulfilled, the loop must scan every previously fulfilled sequence number in one transaction.

The developer explicitly acknowledged both the gas risk and the fund-locking consequence in adjacent TODO comments:

```
// TODO: if this effect occurs here, we need to guarantee that executeCallback
// can never revert. If executeCallback can revert, then funds can be
// permanently locked in the contract.
``` [3](#0-2) 

`clearRequest` and the provider credit both execute *before* the while loop. If the loop exhausts gas, the EVM reverts the entire transaction, undoing those effects. The request remains active but is now permanently unfulfillable because every subsequent attempt to call `executeCallback` for it will re-enter the same loop and run out of gas again. [4](#0-3) 

---

### Impact Explanation

- **Permanent fund lock**: The requester's fee (stored in `req.fee`) is locked inside the contract with no withdrawal or cancellation function available to the requester.
- **Provider DoS**: The provider can never collect the fee for the targeted request.
- **Protocol integrity**: Any request whose sequence number is the current `firstUnfulfilledSeq` can be rendered permanently unfulfillable once enough higher-sequence requests have been fulfilled ahead of it.

The `EchoState` struct stores requests in a fixed 32-slot array plus an overflow mapping; there is no cap on `currentSequenceNumber`. [5](#0-4) 

---

### Likelihood Explanation

`requestPriceUpdatesWithCallback` and `executeCallback` are both permissionless. Any address can create requests and fulfill them. The attacker pays fees for N−1 requests and gas for N−1 `executeCallback` calls. Each while-loop iteration costs roughly 2,100–2,300 gas (one cold SLOAD into the overflow mapping per unique sequence number). Against a 30 M gas block limit, approximately 5,000–13,000 pre-fulfilled requests are sufficient to exhaust gas on the victim's `executeCallback`. If the provider's per-feed and per-gas fees are set low (or zero), the attack cost is dominated by Ethereum transaction gas alone, making it economically feasible for a motivated attacker.

---

### Recommendation

Replace the unbounded while loop with a bounded alternative. Two standard approaches:

1. **Doubly-linked list of active requests** (already suggested in the developer TODO): maintain `prev`/`next` pointers so `firstUnfulfilledSeq` can be updated in O(1).
2. **Cap the scan per call**: advance `firstUnfulfilledSeq` by at most a fixed number of steps (e.g., 64) per `executeCallback` invocation, accepting that the pointer may lag behind temporarily but never blocking the transaction.
3. **Remove the `firstUnfulfilledSeq` advancement entirely** from `executeCallback` and move it to a separate, gas-bounded maintenance function callable by keepers.

---

### Proof of Concept

```solidity
// Attacker steps (pseudocode / Foundry test outline):

// 1. Attacker creates N requests (seq 1 … N), paying the minimum fee each time.
for (uint i = 0; i < N; i++) {
    echo.requestPriceUpdatesWithCallback{value: fee}(
        provider, publishTime, priceIds, callbackGasLimit
    );
}
// After this: currentSequenceNumber = N+1, firstUnfulfilledSeq = 0 (or 1).

// 2. Attacker fulfills requests 2 … N via executeCallback (out of order).
//    Each call's while loop stops immediately because seq 1 is still active,
//    so firstUnfulfilledSeq stays pinned at 1.
for (uint64 seq = 2; seq <= N; seq++) {
    echo.executeCallback(provider, seq, updateData, priceIds);
}
// After this: firstUnfulfilledSeq = 1, requests 2…N are cleared.

// 3. Victim (or anyone) tries to fulfill request 1.
//    The while loop must now scan seq 1 (just cleared) → seq 2 → … → seq N:
//    N iterations × ~2,300 gas ≈ N × 2,300 gas.
//    With N = 10,000 → ~23,000,000 gas → exceeds block gas limit → OOG revert.
echo.executeCallback{gas: 30_000_000}(provider, 1, updateData, priceIds);
// ↑ reverts with out-of-gas; req.fee is permanently locked.
```

The `EchoGasBenchmark` test suite already documents that out-of-order fulfillment causes a linear scan and measures its cost, confirming the gas-scaling behaviour described above. [6](#0-5)
