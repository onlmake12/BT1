### Title
`uint16` Position Counter Overflow in Unchecked Assembly Reads Enables Silent Incorrect Price Parsing - (File: `lazer/contracts/evm/src/PythLazerLib.sol`)

### Summary

The `parseUpdateFromPayload` function in `PythLazerLib.sol` parses attacker-supplied bytes using assembly-based read helpers (`_readBytes1/2/4/8`) that perform **no bounds checking**. The position counter `pos` is typed as `uint16`, which overflows silently when a crafted payload encodes large `feedsLen` (up to 255) and `numProperties` (up to 255) values. An overflow can cause `pos` to wrap back to a value that equals `payload.length`, bypassing the only integrity check (`require(pos == payload.length)`), while all intermediate reads return data from wrong offsets or from zero-initialized EVM memory beyond the payload.

### Finding Description

`parseUpdateFromPayload` is declared `public pure` and accepts arbitrary `bytes memory payload` with no caller restriction. [1](#0-0) 

The four assembly read helpers load a 32-byte EVM word at `data + 0x20 + pos` with no check that `pos + read_size <= data.length`: [2](#0-1) 

The position counter is `uint16` (max 65 535). The outer loop iterates up to `feedsLen` (uint8, max 255) feeds; the inner loop iterates up to `numProperties` (uint8, max 255) properties per feed. The heaviest property type (`FundingRate`, `FundingTimestamp`, `FundingRateInterval`, `FeedUpdateTimestamp`) consumes 10 bytes per iteration (1 property-ID byte + 1 exists byte + 8 value bytes). Maximum total advance:

An attacker can tune `feedsLen`, `numProperties`, and property types so that `pos` wraps to exactly `payload.length`, satisfying the only post-parse guard: [3](#0-2) 

All intermediate reads that land past the real payload boundary silently return 0 (EVM memory is zero-initialized beyond the allocated region). Reads that land inside the payload due to the wrap return bytes from the wrong offset, producing attacker-influenced price values.

### Impact Explanation

Any consumer contract that calls `parseUpdateFromPayload` on bytes it did not first authenticate through `PythLazer.verifyUpdate` receives a structurally valid-looking `Update` struct whose field values are wrong. Because the function is `public pure`, it is callable by any EOA or contract with arbitrary input. A Lazer consumer that passes user-supplied update bytes directly to the library (a common integration pattern) can have its on-chain price state set to attacker-controlled values, enabling price manipulation, incorrect liquidations, or mispriced derivatives.

### Likelihood Explanation

`parseUpdateFromPayload` is the documented consumer-facing parsing entry point for the Pyth Lazer EVM SDK. Consumer contracts are expected to call it after `verifyUpdate`, but nothing in the library enforces this ordering. The overflow requires only crafting a byte string with specific `feedsLen`/`numProperties` counts â€” no privileged key or governance access is needed. Any unprivileged Lazer relayer or end-user interacting with a consumer contract that exposes this path can trigger it.

### Recommendation

1. Replace the `uint16 pos` counter with `uint256` to eliminate overflow.
2. Add explicit bounds checks in each `_readBytes*` helper before the `mload`, analogous to the safe `BytesLib` used elsewhere in the codebase: [4](#0-3) 

3. Validate `feedsLen` and `numProperties` against the remaining payload length at the start of each loop iteration before advancing `pos`.
4. Consider making `parseUpdateFromPayload` `internal` or adding a require that the payload was produced by `verifyUpdate`.
