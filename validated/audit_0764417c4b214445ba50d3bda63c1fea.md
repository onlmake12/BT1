### Title
Loss of Requester Funds When Callback Fails in Echo — (`Echo.sol`)

### Summary

In `Echo.sol`, the provider is credited the full fee (including the gas-based component `callbackGasLimit * feePerGasInWei`) and the request is cleared **before** the callback is attempted. When the callback reverts, the provider retains the entire fee regardless of how little gas was actually consumed, and there is no cancellation or refund path for the requester.

---

### Finding Description

In `Echo.requestPriceUpdatesWithCallback`, the user pays a fee that includes a gas-proportional component:

```
feeAmount = pythFeeInWei + baseFeeInWei + feePerFeedInWei * priceIds.length
          + feePerGasInWei * callbackGasLimit
``` [1](#0-0) 

The stored `req.fee` captures everything above `pythFeeInWei`:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

In `executeCallback`, the provider is credited and the request is cleared **before** the callback is invoked:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // ← full fee credited

clearRequest(sequenceNumber);                       // ← request deleted

try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{ ... }
catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(...);            // ← no refund issued
}
catch {
    emit PriceUpdateCallbackFailed(...);            // ← no refund issued
}
``` [3](#0-2) 

When the callback reverts (for any reason), the provider has already been credited `req.fee` in full — including `feePerGasInWei * callbackGasLimit` — even if the callback consumed only a tiny fraction of `callbackGasLimit` before reverting. Because the request has been cleared, the requester has no retry path and no mechanism to recover the overpaid gas-based fee.

The developers themselves flag this ordering risk in a TODO comment:

> "if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract." [4](#0-3) 

---

### Impact Explanation

A requester who sets a large `callbackGasLimit` (e.g., 1 000 000 gas) and whose callback reverts after consuming only a small amount of gas (e.g., 5 000 gas) loses the entire gas-proportional fee:

```
lost ≈ (callbackGasLimit - actualGasUsed) * feePerGasInWei
```

For `callbackGasLimit = 1 000 000` and `feePerGasInWei = 1 gwei`, the requester loses ≈ 0.001 ETH per failed request. There is no cancellation function and no refund path; the funds are permanently transferred to the provider.

This is directly analogous to the zkSync M-03 finding: in both cases the user pre-pays for gas capacity, the execution fails early, and the unspent portion is not returned.

---

### Likelihood Explanation

Callbacks fail in practice for well-known reasons:

1. The consumer contract contains a bug that causes a revert.
2. The consumer sets `callbackGasLimit` slightly too low, causing an out-of-gas revert inside the callback.
3. The consumer contract is upgraded between request and fulfillment, changing its gas profile.
4. A malicious or griefing provider deliberately calls `executeCallback` with a `callbackGasLimit` that is just enough to enter the callback but not enough to complete it, collecting the full fee while the callback fails.

Scenario 4 is particularly concerning: because `callbackGasLimit` is set by the **requester** at request time and stored on-chain, the provider can observe it and craft a transaction that passes the gas check in `executeCallback` while ensuring the callback runs out of gas, collecting the full fee.

---

### Recommendation

1. **Credit the provider only after a successful callback.** Move the fee-credit line inside the `try` block so that a failed callback does not enrich the provider.
2. **Refund the gas-proportional portion on failure.** Measure `gasleft()` before and after the callback call and refund `(callbackGasLimit - gasUsed) * feePerGasInWei` to `req.requester` on a catch path.
3. **Restore the request on failure** (or add a separate cancellation function) so the requester can reclaim their funds if no provider ever successfully executes the callback.

---

### Proof of Concept

```
1. Alice calls requestPriceUpdatesWithCallback(provider, t, priceIds, 1_000_000)
   paying fee = pythFee + baseFee + feedFee + 1_000_000 * feePerGasInWei.

2. Provider calls executeCallback(...).
   - Line 161-162: provider.accruedFeesInWei += req.fee  (full 1_000_000 * feePerGasInWei credited)
   - Line 164: clearRequest(sequenceNumber)              (request deleted)
   - Line 176-201: callback reverts after 5_000 gas.
   - PriceUpdateCallbackFailed emitted; no refund.

3. Alice has lost (1_000_000 - 5_000) * feePerGasInWei in excess fees.
   She cannot cancel, retry, or recover the funds.
``` [5](#0-4) [6](#0-5)

    function getFee(
        address provider,
        uint32 callbackGasLimit,
        bytes32[] calldata priceIds
    ) public view override returns (uint96 feeAmount) {
        uint96 baseFee = _state.pythFeeInWei; // Fixed fee to Pyth
        // Note: The provider needs to set its fees to include the fee charged by the Pyth contract.
        // Ideally, we would be able to automatically compute the pyth fees from the priceIds, but the
        // fee computation on IPyth assumes it has the full updated data.
        uint96 providerBaseFee = _state.providers[provider].baseFeeInWei;
        uint96 providerFeedFee = SafeCast.toUint96(
            priceIds.length * _state.providers[provider].feePerFeedInWei
        );
        uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei; // Provider's per-gas rate
        uint256 gasFee = callbackGasLimit * providerFeeInWei; // Total provider fee based on gas
        feeAmount =
            baseFee +
            providerBaseFee +
            providerFeedFee +
            SafeCast.toUint96(gasFee); // Total fee user needs to pay
