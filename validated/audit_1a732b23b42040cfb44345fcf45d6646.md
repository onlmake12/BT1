Audit Report

## Title
Rate Limiter Keyed by Ephemeral `SessionId` Allows Rate Limit Bypass via Reconnect Cycling ‚Äî (`network/src/protocols/hole_punching/mod.rs`)

## Summary
The `HolePunching` protocol's primary `rate_limiter` is keyed by `(PeerIndex, u32)` where `PeerIndex` is a type alias for `SessionId`, an ephemeral integer reassigned on every TCP connection. An attacker can bypass the 30 msg/sec cap by disconnecting and reconnecting to obtain a fresh `SessionId` and a clean rate limit window. Combined with the attacker-controlled `from`/`to` payload fields that bypass the secondary `forward_rate_limiter`, each `ConnectionRequest` can trigger a `sqrt(N)` gossip broadcast, enabling low-cost amplified network congestion.

## Finding Description
The `HolePunching` struct declares its primary rate limiter keyed by the ephemeral session ID at `network/src/protocols/hole_punching/mod.rs` line 45:

```rust
rate_limiter: RateLimiter<(PeerIndex, u32)>,
```

`PeerIndex` is confirmed as a type alias for `SessionId` at `network/src/protocols/mod.rs` line 33:

```rust
pub type PeerIndex = SessionId;
```

`SessionId` is a monotonically increasing integer assigned per TCP connection. In `received()`, the rate check uses this ephemeral value (`mod.rs` lines 95‚Äì107):

```rust
if self.rate_limiter.check_key(&(session_id, msg.item_id())).is_err() { ... return; }
```

The `disconnected()` handler (`mod.rs` lines 66‚Äì70) only calls `retain_recent()` (a GC operation) and does **not** remove the disconnected session's rate limit entry, nor does it ban the peer:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
}
```

On reconnect, the peer receives a fresh `SessionId` (e.g., `N+1`), so the key `(N+1, item_id)` has no prior rate limit history and the 30 msg/sec window starts fresh.

The secondary `forward_rate_limiter` is keyed by `(content.from, content.to, item_id)` at `connection_request.rs` lines 132‚Äì143. However, `content.from` and `content.to` are parsed directly from the attacker-controlled message payload (`connection_request.rs` lines 36‚Äì40) with no verification that `content.from` matches the actual authenticated session peer ID. An attacker can trivially vary these fields across messages to bypass the 1 msg/sec secondary limit entirely.

The amplification path is confirmed in `forward_message()` at `connection_request.rs` lines 280‚Äì305: when the target peer is not directly connected (the common case), the node performs a `filter_broadcast` to `sqrt(total_peers)` other nodes.

## Impact Explanation
This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs.*

An attacker cycling connections at rate `R` reconnects/sec delivers `30 √ó R` `ConnectionRequest` messages per second to the victim node. Each message that reaches `forward_message()` triggers a `filter_broadcast` to `sqrt(N)` peers. With 125 connected peers, each attacker message fans out to ~11 additional nodes. Both the primary rate limiter (bypassed via reconnect) and the secondary rate limiter (bypassed via varying payload `from`/`to` fields) are defeated, leaving no effective cost barrier.

## Likelihood Explanation
The attack requires only a standard inbound P2P connection ‚Äî no special privileges, no leaked keys, no majority hashpower. The attacker implements a simple loop: connect ‚Üí send 30 `ConnectionRequest` messages with varying `from`/`to` ‚Üí disconnect ‚Üí reconnect. The secio handshake adds per-reconnect latency but does not prevent the attack at meaningful scale. Both rate limiters are bypassable with trivial attacker effort.

## Recommendation
Key the `rate_limiter` by `(PeerId, u32)` instead of `(PeerIndex, u32)`. The `PeerId` is stable across reconnects and is available via `extract_peer_id(&context.session.address)` at the point of the `received()` call. This matches the design already used by `forward_rate_limiter`'s key type. Additionally, verify that `content.from` in `ConnectionRequestProcess::execute()` matches the authenticated session's `PeerId` to prevent payload-based bypass of the `forward_rate_limiter`. Consider banning peers that repeatedly trigger the rate limiter rather than silently dropping messages.

## Proof of Concept
```
Attacker controls a node with a valid keypair (stable PeerId P).

loop:
  1. Connect to victim node ‚Üí assigned new SessionId N
  2. Send 30 ConnectionRequest messages with arbitrary from/to PeerIds
     (vary from/to each message to bypass forward_rate_limiter)
     ‚Üí all pass rate_limiter check for key (N, item_id) [fresh window]
     ‚Üí each triggers forward_message() ‚Üí filter_broadcast to sqrt(peers) nodes
  3. Disconnect (SessionId N retired; no ban issued)
  4. Reconnect ‚Üí assigned new SessionId N+1
  5. Send 30 more ConnectionRequest messages
     ‚Üí all pass rate_limiter check for key (N+1, item_id) [fresh window again]
  ...

Total messages processed per second = 30 √ó reconnect_frequency.
Each message fans out to sqrt(N) additional peers via gossip broadcast.
Network amplification factor = sqrt(connected_peers).
```
