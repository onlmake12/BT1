Audit Report

## Title
Off-by-one epoch gate in `BlockTxsVerifier` silently skips `DaoScriptSizeVerifier` for the first block of RFC0044 activation epoch — (`verification/contextual/src/contextual_block_verifier.rs`)

## Summary

`BlockTxsVerifier::verify` gates `DaoScriptSizeVerifier` using `self.parent.epoch().number()` instead of the current block's epoch number. Because `rfc0044_active` uses a `>=` comparison, the first block of epoch `RFC0044_ACTIVE_EPOCH` is evaluated against `RFC0044_ACTIVE_EPOCH - 1`, returning `false` and silently skipping the verifier. A miner who produces that one block can include a DAO withdrawal transaction where the withdrawal cell's lock script has a different serialized size than the deposit cell's, violating the RFC0044 invariant. `BlockExtensionVerifier` has the identical off-by-one, allowing the first block of the activation epoch to omit the required MMR extension.

## Finding Description

`BlockTxsVerifier` holds both `header: HeaderView` (the current block) and `parent: &'b HeaderView` as struct fields. [1](#0-0) 

The epoch gate at line 445 reads `self.parent.epoch().number()` rather than `self.header.epoch().number()`: [2](#0-1) 

`rfc0044_active` is defined as `target >= rfc0044_active_epoch`: [3](#0-2) 

When the block being verified is the first block of epoch `N` (= `RFC0044_ACTIVE_EPOCH`), its parent is the last block of epoch `N-1`. So `self.parent.epoch().number()` = `N-1`, and `rfc0044_active(N-1)` = `N-1 >= N` = `false`. `DaoScriptSizeVerifier` is never called.

`DaoScriptSizeVerifier` has no independent epoch gate — its only internal guard is a block-number check on the deposit cell's commit height (`starting_block_limiting_dao_withdrawing_lock`), not on the current block's epoch: [4](#0-3) 

The sole epoch gate is the one in `BlockTxsVerifier`, and it is off by one.

`BlockExtensionVerifier::verify` has the identical pattern at line 540–543, meaning the MMR extension check (`NoBlockExtension` / `InvalidChainRoot`) is also skipped for the first block of the activation epoch: [5](#0-4) 

RFC0044 activation epochs are hardcoded: mainnet epoch `8651`, testnet epoch `5711`: [6](#0-5) [7](#0-6) 

## Impact Explanation

RFC0044 is enforced exclusively at the node verification layer; the DAO script in CKB-VM does not independently enforce the lock-script-size invariant. Because all nodes share the same buggy gate, the malformed block is accepted by the entire network — there is no fork, but the RFC0044 invariant is permanently violated on-chain for that block. An attacker can create a DAO withdrawal cell whose lock script differs in serialized size from the deposit cell, bypassing the protection RFC0044 was designed to provide. This constitutes damage to the CKB DAO economy: the invariant that deposit and withdrawal lock scripts must be equally sized — which prevents certain lock-script-substitution attacks on DAO funds — is broken for any transaction included in that one block. This maps to **Critical (15001–25000 points): Vulnerabilities which could easily damage CKB economy**.

## Likelihood Explanation

Any miner who wins the PoW lottery for the first block of epoch `RFC0044_ACTIVE_EPOCH` can exploit this. No special privilege beyond valid PoW is required. The activation epoch is publicly known and deterministic (mainnet: 8651, testnet: 5711). The attacker pre-creates the deposit cell and the mismatched withdrawal transaction before the epoch boundary, then includes it the moment they mine that block. The window is exactly one block but is fully predictable and requires no victim mistakes.

## Recommendation

In `BlockTxsVerifier::verify`, change the gate to use the current block's epoch number:

```rust
// Before (buggy):
if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {

// After (correct):
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
```

Apply the same fix to `BlockExtensionVerifier::verify` at line 540–543. Since `BlockExtensionVerifier` receives the current `block: &BlockView` as a parameter to `verify`, replace `self.parent.epoch().number()` with `block.header().epoch().number()`.

## Proof of Concept

1. Set `RFC0044_ACTIVE_EPOCH = N` (e.g., testnet epoch 5711).
2. Create a DAO deposit cell committed at or after `starting_block_limiting_dao_withdrawing_lock`.
3. Construct a DAO withdrawal transaction where the withdrawal cell's lock script serialized size differs from the deposit cell's.
4. Build a block whose parent is the last block of epoch `N-1` (i.e., the first block of epoch `N`).
5. Call `BlockTxsVerifier::verify` — returns `Ok(...)` because `rfc0044_active(N-1)` is `false`; `DaoScriptSizeVerifier` is never invoked.
6. Build an identical block whose parent is the first block of epoch `N` (i.e., the second block of epoch `N`).
7. Call `BlockTxsVerifier::verify` — returns `Err(DaoLockSizeMismatch)` because `rfc0044_active(N)` is `true`.

Step 5 demonstrates the bypass; step 7 confirms the check works correctly one block later.
