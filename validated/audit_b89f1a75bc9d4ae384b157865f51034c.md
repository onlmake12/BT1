Audit Report

## Title
Silent u128→u64 Truncation in DAO Withdrawal Capacity Calculation — (File: util/dao/src/lib.rs)

## Summary
In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `withdraw_counted_capacity` (a `u128`) is cast to `u64` with a bare `as u64` at line 156, silently discarding the upper 64 bits on overflow. Every other u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. If `withdraw_counted_capacity` exceeds `u64::MAX`, the result wraps to a near-zero value with no error, corrupting both the on-chain DAO accumulator `S` (stored in every block header) and the value returned to callers of the `calculate_dao_maximum_withdraw` RPC.

## Finding Description
In `calculate_maximum_withdraw` (util/dao/src/lib.rs, lines 152–156):

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)  // ← silent truncation
        .safe_add(occupied_capacity)?;
```

The multiplication is correctly widened to u128, but the final division result is narrowed back to u64 with `as u64`, which wraps silently. The correct pattern — used at lines 244–245 and 258 — is `u64::try_from(...).map_err(|_| DaoError::Overflow)?`.

`calculate_maximum_withdraw` is called from two paths:
1. `transaction_maximum_withdraw` → `withdrawed_interests` → `dao_field_with_current_epoch`: a truncated result causes `withdrawed_interests` to be near-zero, so `current_s` (the NervosDAO secondary-issuance accumulator written into every block header) is inflated by the full interest amount that should have been subtracted, corrupting consensus state for all subsequent blocks.
2. `calculate_dao_maximum_withdraw` RPC: returns a silently wrong (drastically underestimated) capacity to wallets and users.

Existing guards (`safe_add`, `safe_sub`) do not detect the truncation because the wrapped value is a valid `u64`; no error is propagated.

## Impact Explanation
- **Consensus deviation (Critical, 15001–25000 pts)**: A silently wrong `withdrawed_interests` corrupts `current_s` in `dao_field_with_current_epoch`. Because `current_s` feeds into every subsequent block's DAO field computation, the corruption propagates indefinitely, causing nodes that process the offending withdrawal to diverge from nodes that do not, splitting consensus.
- **Economic damage (Critical, 15001–25000 pts)**: The depositor's entitled withdrawal is silently underestimated by up to ~1,845 million CKB in the PoC scenario, with no error raised and no recourse.

## Likelihood Explanation
The overflow condition is:
```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX  (~1.84 × 10¹⁹)
```
With the total CKB supply at ~3.36 × 10¹⁸ shannons and `ar` starting at 10¹⁶, a near-maximum single-cell deposit requires `ar` to grow to ~5.5 × 10¹⁶ (a ~5.5× increase). At ~4% annual secondary issuance growth this takes roughly 40+ years. The likelihood is low in the near term but non-zero over the protocol's indefinite lifetime. No attacker action is required beyond making a large DAO deposit early and withdrawing after sufficient `ar` growth — a normal user operation. The condition is deterministic and reproducible given the right deposit size and epoch gap.

## Recommendation
Replace the unchecked cast with the same checked conversion used at lines 244–245 and 258:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with lines 244–245 and 258):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
```

## Proof of Concept
Construct inputs where `withdraw_counted_capacity` exceeds `u64::MAX`:

- `counted_capacity` = 3,360,000,000,000,000,000 shannons (realistic near-max single cell, within total supply)
- `deposit_ar` = 10,000,000,000,000,000 (genesis value)
- `withdrawing_ar` = 55,000,000,000,000,000 (5.5× growth, ~40 years at 4%/yr)

```
withdraw_counted_capacity = 3,360,000,000,000,000,000 × 55,000,000,000,000,000
                            / 10,000,000,000,000,000
                          = 18,480,000,000,000,000,000
```

`18,480,000,000,000,000,000 > u64::MAX (18,446,744,073,709,551,615)`

With `as u64`: result wraps to `33,255,926,290,448,384` — roughly 1% of the correct value — with no error raised. A unit test can be written in `util/dao/src/tests` by mocking `deposit_header` and `withdrawing_header` with the above `ar` values and asserting that `calculate_maximum_withdraw` returns `Err(DaoError::Overflow)` after the fix, and currently (incorrectly) returns `Ok(...)` with a truncated value.
