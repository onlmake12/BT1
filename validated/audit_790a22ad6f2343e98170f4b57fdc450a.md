Audit Report

## Title
Unbounded `forward_rate_limiter` HashMap Growth via Attacker-Controlled `(from, to)` PeerIds Causes OOM — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

`HolePunching::forward_rate_limiter` is a `governor::RateLimiter` backed by `HashMapStateStore<(PeerId, PeerId, u32)>`. All three message handlers insert new entries keyed by attacker-supplied `from`/`to` fields. `retain_recent()` is called only in `disconnected()`, never in the periodic `notify()` handler. A single long-lived session sending messages with unique `(from, to)` pairs causes the HashMap to grow without bound until the node is OOM-killed.

## Finding Description

`forward_rate_limiter` is declared as `RateLimiter<(PeerId, PeerId, u32)>` backed by `HashMapStateStore` (`mod.rs` L31–46). The outer `rate_limiter`, keyed by `(session_id, msg.item_id())`, throttles to 30 msg/sec per `(session, type)` — but `session_id` is fixed and `item_id` has only 3 values, so this limiter is bounded to at most 3 entries per session and does not constrain unique-key growth in `forward_rate_limiter` (`mod.rs` L95–107).

All three message processors call `forward_rate_limiter.check_key()` with attacker-controlled `content.from` and `content.to` (`connection_request.rs` L132–143, `connection_request_delivered.rs` L134–145, `connection_sync.rs` L85–96). `PeerId::from_bytes()` only validates encoding — it does not restrict `from`/`to` to known or connected peers. Each unique `(from, to, item_id)` tuple causes `HashMapStateStore` to insert a new entry with a fresh bucket, so the per-key rate limit (1/sec) is never triggered for novel pairs, meaning every message both passes and inserts.

`retain_recent()` is called on both rate limiters in `disconnected()` (`mod.rs` L66–70). The `notify()` handler (fires every `CHECK_INTERVAL = 5 minutes`) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter (`mod.rs` L169–175). As long as the attacker's session remains open, the HashMap accumulates entries indefinitely with no cleanup path.

## Impact Explanation

**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.** The attack causes unbounded RSS growth in the node process. At 30 entries/sec × 3 message types × ~150–200 bytes/entry, the node accumulates ~48 MB/hour from a single attacker session. Sustained over hours to days, this exhausts available memory and triggers an OOM kill, fully crashing the node. The impact is concrete and measurable, not hypothetical.

## Likelihood Explanation

The attacker requires only a single unprivileged P2P connection with the HolePunching protocol negotiated — no proof-of-work, no keys, no special role. The attack is trivially automatable: send `ConnectionRequest`, `ConnectionRequestDelivered`, or `ConnectionSync` messages at ≤30/sec with random 32-byte `from`/`to` fields while keeping the session open. The outer rate limiter throttles throughput but does not bound total unique-key count. The attack is repeatable and requires no victim interaction.

## Recommendation

Add `retain_recent()` calls for both rate limiters inside the `notify()` handler, which already fires every 5 minutes:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();           // add
    self.forward_rate_limiter.retain_recent();   // add
    // ... existing cleanup ...
}
```

This bounds both HashMaps to entries active within the last rate-limit window, regardless of session duration, matching the existing pattern already used in `disconnected()`.

## Proof of Concept

```python
# Single TCP session, send N ConnectionRequest messages
# each with a unique random (from_peer_id, to_peer_id)
import os, time
session = connect_to_ckb_node_hole_punching_protocol()
for _ in range(10_000_000):
    from_id = os.urandom(32)
    to_id   = os.urandom(32)
    msg = build_connection_request(from_id, to_id, max_hops=6, listen_addrs=[...])
    session.send(msg)
    time.sleep(1/30)  # stay within outer rate limit
# Monitor: node RSS grows ~150-200 bytes per iteration
# Verify: forward_rate_limiter HashMap size increases monotonically
# Expected: node OOM-killed after sustained run; no cleanup until disconnect
```

A unit test can confirm the bug by calling `received()` in a loop with unique `(from, to)` pairs on a single session without calling `disconnected()`, then asserting that `forward_rate_limiter` internal state size grows proportionally to the number of messages sent.
