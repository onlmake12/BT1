### Title
`ScriptHashTypeVerifier` Omits Type Script Hash-Type Check, Allowing Unsupported Hash Types to Bypass Non-Contextual Validation — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates only the **lock script's** `hash_type` against `ENABLED_SCRIPT_HASH_TYPE`. It never inspects the **type script's** `hash_type`. A transaction output carrying a type script with a not-yet-activated hash type (e.g., `Data3`) silently passes this non-contextual gate and is only rejected later—at the more expensive script-execution stage—with a different error kind. This is the direct CKB analog of the zkSync `DefaultAccount#fallback` missing `payable`: a missing attribute/check on one half of a two-part structure causes a behavioral divergence from the intended protocol invariant.

---

### Finding Description

`ScriptHashTypeVerifier::verify()` is registered inside `NonContextualTransactionVerifier` with the stated purpose of enforcing that every output's script hash types are within the consensus-permitted range:

```rust
// verification/src/transaction_verifier.rs  line 796-814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())  // ← lock only
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }
    }
    Ok(())
}
```

The loop body calls `output.lock().hash_type()` exclusively. `output.type_()` is never consulted. [1](#0-0) 

The comment on `NonContextualTransactionVerifier` itself confirms the gap — it says "Check whether output **lock** hash type within enabled range" and says nothing about type scripts: [2](#0-1) 

By contrast, the low-level molecule `check_data` helper correctly validates **both** lock and type scripts on a `CellOutput`:

```rust
// util/gen-types/src/extension/check_data.rs  line 24-28
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()  // both checked
    }
}
``` [3](#0-2) 

When a transaction with an output whose type script carries an unsupported hash type (e.g., `Data3 = 6`) reaches the script executor, `select_version` catches it:

```rust
// script/src/types.rs  line 930-935
hash_type => {
    return Err(ScriptError::InvalidScriptHashType(format!(
        "The ScriptHashType/{:?} has not been activated…",
        hash_type
    )));
}
``` [4](#0-3) 

So the transaction is ultimately rejected, but:
- at the **wrong layer** (script execution instead of non-contextual),
- with the **wrong error kind** (`ScriptError` instead of `TransactionError::ScriptHashTypeNotPermitted`),
- after **more work** has been done (script group resolution, VM environment setup).

The existing unit test `test_not_enabled_hash_type_output_lock` demonstrates the check works for lock scripts but has no counterpart for type scripts: [5](#0-4) 

---

### Impact Explanation

1. **Tx-pool DoS amplification.** An unprivileged RPC caller can submit a stream of transactions whose outputs have type scripts with unsupported hash types. Each transaction passes `NonContextualTransactionVerifier` and is queued for full script verification before being discarded. Script execution is significantly more expensive than the non-contextual check; the missing gate removes the cheap early-exit that should protect the node.

2. **Error-kind inconsistency / tooling breakage.** Downstream tooling, light clients, and dApps that distinguish `TransactionError::ScriptHashTypeNotPermitted` from `ScriptError` to decide whether to retry or blacklist a transaction will misclassify these rejections.

3. **Consensus-layer asymmetry risk.** If a future code path (e.g., a relay or sync shortcut) skips script execution but trusts the non-contextual result, a cell with an invalid type-script hash type could be persisted, permanently locking the capacity it holds.

---

### Likelihood Explanation

The attack requires only a valid RPC endpoint (`send_transaction`). No key material, privileged access, or majority hash power is needed. Crafting a transaction with an output whose type script has `hash_type = Data3` (byte value `6`) is trivial. The attacker pays only the minimum fee to submit; the node bears the cost of script-group resolution and VM environment setup before rejection.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's hash type for each output, mirroring the existing lock-script logic:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check …
        check_hash_type(output.lock().hash_type())?;

        // add: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

This mirrors the `check_data` helper that already validates both scripts at the molecule layer and aligns with the stated intent of `NonContextualTransactionVerifier`.

---

### Proof of Concept

A transaction constructed as follows passes `ScriptHashTypeVerifier` today but should be rejected at the non-contextual stage:

```rust
let tx = TransactionBuilder::default()
    .output(
        CellOutput::new_builder()
            .lock(Script::default()
                .as_builder()
                .hash_type(ScriptHashType::Type)   // valid lock hash type
                .build())
            .type_(Some(
                Script::default()
                    .as_builder()
                    .hash_type(ScriptHashType::Data3) // unsupported type-script hash type
                    .build()
            ))
            .build(),
    )
    .build();

let verifier = ScriptHashTypeVerifier::new(&tx);
// Currently returns Ok(()), should return Err(ScriptHashTypeNotPermitted)
assert!(verifier.verify().is_ok());
```

The transaction then proceeds to script execution, where `select_version` returns `ScriptError::InvalidScriptHashType` — a different error kind, at a higher cost, than the intended early rejection.
