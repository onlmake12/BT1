### Title
Unvalidated `observed_addr` Injection Allows Arbitrary Address Poisoning in Advertised Listen Addresses — (`network/src/protocols/identify/mod.rs`)

---

### Summary

An unprivileged inbound peer can send a crafted `observed_addr` in an `IdentifyMessage` pointing to any arbitrary IP:port. The victim node stores this address without verifying it against the actual session's remote IP, and subsequently re-advertises it as its own listen address to every new peer that connects, for as long as the attacker's session remains open.

---

### Finding Description

The exploit path is fully traceable through production code:

**Step 1 — Attacker sends crafted `observed_addr`.**
When the victim receives an `IdentifyMessage`, `IdentifyProtocol::received()` calls `process_observed()` unconditionally after duplicate and identify checks pass: [1](#0-0) 

`process_observed()` calls `self.callback.add_observed_addr(observed, info.session.id)` with the raw, attacker-supplied `Multiaddr`: [2](#0-1) 

**Step 2 — No IP validation in `IdentifyCallback::add_observed_addr`.**
The only operation performed is appending the local node's peer ID if absent. There is no check that the IP in `observed_addr` matches the actual session's remote address: [3](#0-2) 

**Step 3 — Raw insert into `observed_addrs`.**
`NetworkState::add_observed_addr` performs a plain `HashMap::insert` keyed by `session_id` with no validation: [4](#0-3) 

**Step 4 — Poisoned address included in advertised listen addresses.**
`local_listen_addrs()` fills remaining slots (up to `MAX_RETURN_LISTEN_ADDRS = 10`) with `observed_addrs` whenever the node has fewer than 10 configured public addresses — a common condition for nodes without explicit `public_addresses` config: [5](#0-4) 

**Step 5 — Broadcast to every new peer.**
In `IdentifyProtocol::connected()`, `local_listen_addrs()` is called and the result is sent to every newly connecting peer. The only filter applied is `is_reachable()`, which returns `true` for any globally routable public IP — exactly what the attacker supplies: [6](#0-5) 

**Step 6 — Cleanup only on disconnect.**
The `observed_addrs` entry is removed only when the attacker's session closes: [7](#0-6) 

So the attacker simply stays connected to keep the poisoned address active.

---

### Impact Explanation

While the attacker's session is open, every new peer that connects to the victim receives an `IdentifyMessage` listing the attacker-controlled IP:port as one of the victim's own listen addresses. Those peers store it in their peer stores and may further propagate it via the Discovery protocol. This enables:

- **Address poisoning at network scale**: the victim node's identity is associated with an arbitrary third-party IP:port across the entire P2P network.
- **DDoS amplification**: all peers that receive the poisoned address will attempt to dial the third-party target, directing connection floods at it.

The `observed_addrs` map is keyed by `SessionId`, so a single attacker connection injects exactly one poisoned address. Multiple attacker connections can inject multiple distinct addresses simultaneously.

---

### Likelihood Explanation

- Requires only an inbound TCP connection — no privilege, no key, no PoW.
- The precondition (fewer than 10 configured public addresses) is the default for most nodes that do not set `public_addresses` in their config.
- The attacker-supplied address must be a globally routable IP to pass the `is_reachable()` filter, which is trivially satisfied by any real public IP.
- The attack is persistent for the duration of the session and requires no repeated action.

---

### Recommendation

In `IdentifyCallback::add_observed_addr`, validate that the IP component of the supplied `observed_addr` matches the actual remote IP of the session (available from `SessionContext::address`). Reject or ignore any `observed_addr` whose IP does not match the session's remote IP. This is the standard mitigation used by libp2p's identify protocol.

Additionally, apply an `is_reachable()` check to `observed_addr` at the point of insertion (mirroring the check already applied to `listen_addrs` in `process_listens`). [8](#0-7) 

---

### Proof of Concept

```
1. Attacker connects to victim as inbound peer (standard TCP dial).
2. Attacker sends a valid IdentifyMessage with:
     observed_addr = /ip4/1.2.3.4/tcp/8115
   (where 1.2.3.4 is the third-party target; the attacker's actual IP is irrelevant)
3. Victim stores /ip4/1.2.3.4/tcp/8115/p2p/<victim_peer_id> in observed_addrs[attacker_session_id].
4. A second peer connects to the victim.
5. Victim calls local_listen_addrs(), which (since public_addrs < 10) appends observed_addrs.
6. Victim sends IdentifyMessage to the second peer with listen_addrs containing /ip4/1.2.3.4/tcp/8115/p2p/<victim_peer_id>.
7. Assert: the second peer's received listen_addrs includes the attacker-injected address.
```
