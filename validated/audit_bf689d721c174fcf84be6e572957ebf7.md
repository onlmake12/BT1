Audit Report

## Title
Snappy Decompression Bomb via Attacker-Controlled Header Enables Per-Connection 8MB Heap Amplification — (`network/src/compress.rs`)

## Summary

`LengthDelimitedCodecWithCompress::decode` allocates up to 8MB of zeroed memory based solely on the attacker-controlled snappy varint header, before any decompression is performed. Because snappy can compress 8MB of zeros into ~80KB, an attacker can send a small compressed frame per connection that triggers an 8MB heap allocation. With many concurrent connections, simultaneous live allocations cause unbounded heap growth and OOM crash of the target node.

## Finding Description

In `LengthDelimitedCodecWithCompress::decode` at `network/src/compress.rs` lines 232–244: [1](#0-0) 

The guard at line 235 uses strict `>`, so a frame whose snappy header claims exactly `MAX_UNCOMPRESSED_LEN` (8,388,608 bytes) passes the check and immediately triggers `BytesMut::zeroed(8_388_608)` at line 242 — a full 8MB zero-fill — before any decompression work is done. [2](#0-1) 

The allocation is driven entirely by the attacker-controlled varint in the snappy stream header. An attacker can write a varint claiming 8MB followed by minimal or invalid compressed data; `decompress_len` returns 8MB, the allocation occurs, and only then does `SnapDecoder::decompress` fail and return an error. During the window between allocation and failure, the memory is live.

The inner `length_delimited` codec only enforces that the **compressed** frame fits within `max_frame_length`. For `RelayV3`, this is 4MB: [3](#0-2) 

Compression is enabled by default (`compress: true`) for all protocols built via `CKBProtocol::new_with_support_protocol` and `CKBProtocol::new`: [4](#0-3) 

`LengthDelimitedCodecWithCompress` is installed as the codec for every such protocol: [5](#0-4) 

No compression-ratio check exists anywhere in the decode path. The only guard is the absolute size ceiling, which itself allows the full 8MB allocation.

**Exploit path:**
1. Attacker crafts a snappy stream: varint claiming 8MB uncompressed length, followed by minimal/invalid compressed data (~a few bytes). Total frame: well under 4MB `max_frame_length`.
2. Attacker opens N TCP connections to the victim's P2P port and sends one such frame per connection.
3. Each connection triggers `BytesMut::zeroed(8_388_608)` in `decode()`.
4. Because tokio processes connections concurrently, all N allocations are live simultaneously: N × 8MB total heap.
5. After `decompress` fails, the memory is freed — but the attacker can immediately re-send, keeping allocations continuously live.

## Impact Explanation

This matches the **High** impact class: *"Vulnerabilities which could easily crash a CKB node."* With a realistic number of concurrent connections (e.g., 125 connections × 8MB = 1GB; 500 connections = 4GB), a typical CKB node will be OOM-killed. The crash stops the node from participating in block propagation and consensus. The attack is repeatable: after the node restarts, the attacker can immediately re-execute it.

## Likelihood Explanation

Requirements are minimal: (1) ability to open TCP connections to the victim's P2P port — no authentication required, (2) ability to produce a crafted snappy varint header — trivially done with any snappy library or by hand-writing the varint bytes, (3) minimal outbound bandwidth (~a few KB per connection). No privileged role, leaked key, or majority hashpower is needed. Any unprivileged internet peer can execute this.

## Recommendation

Add a compression-ratio check before allocating, rejecting frames where the claimed decompressed size is disproportionate to the actual compressed payload size:

```rust
let compressed_len = data[1..].len().max(1);
if decompressed_bytes_len / compressed_len > MAX_COMPRESSION_RATIO
    || decompressed_bytes_len > MAX_UNCOMPRESSED_LEN
{
    return Err(io::ErrorKind::InvalidData.into());
}
```

A reasonable `MAX_COMPRESSION_RATIO` for legitimate CKB messages is 8–16. Additionally, consider reducing `MAX_UNCOMPRESSED_LEN` per-protocol to match realistic maximum message sizes rather than using a single global 8MB ceiling, and changing the guard from strict `>` to `>=` to close the off-by-one.

## Proof of Concept

```rust
// Craft a snappy bomb frame: varint claiming 8MB, minimal payload
// Snappy varint for 8_388_608 (0x800000): encoded as multi-byte varint
let mut snappy_stream = vec![];
// Write varint for 8MB
let mut val = 8_388_608usize;
loop {
    let mut byte = (val & 0x7F) as u8;
    val >>= 7;
    if val != 0 { byte |= 0x80; }
    snappy_stream.push(byte);
    if val == 0 { break; }
}
// Append minimal/invalid compressed chunk data (triggers decompress failure after allocation)
snappy_stream.extend_from_slice(&[0x00]); // invalid chunk, decompress will error

// Frame: 4-byte big-endian length | 0x80 (COMPRESS_FLAG) | snappy_stream
let frame_len = (snappy_stream.len() + 1) as u32;
let mut frame = frame_len.to_be_bytes().to_vec();
frame.push(0x80);
frame.extend_from_slice(&snappy_stream);

// Open N TCP connections to victim's P2P port, send `frame` on each.
// Each triggers BytesMut::zeroed(8_388_608) in decode() before decompress fails.
// N=500 → 4GB simultaneous heap → OOM on typical node.
```
