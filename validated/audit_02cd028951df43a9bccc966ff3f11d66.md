### Title
Unauthenticated `get_cells_capacity` with `script_search_mode=Partial` Exhausts SQLXPool via Unbounded Full-Table Scan — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`)

---

### Summary

`get_cells_capacity` accepts `script_search_mode=Partial` from any unauthenticated HTTP caller, generates an `instr(args, $1) > 0` predicate that forces a full-table scan on the SQLite `script` table, holds an explicit DB transaction open for two sequential queries with no execution timeout, and has no concurrency guard. Ten concurrent calls saturate the 10-connection `AnyPool`, causing all subsequent indexer RPC calls to block for the full 60-second `acquire_timeout` before failing.

---

### Finding Description

**1. Unauthenticated RPC entry point**

`get_cells_capacity` is exposed directly as an async RPC method with no authentication layer: [1](#0-0) 

**2. `Partial` mode generates an unindexable full-table scan on SQLite**

`build_query_script_id_sql` (called by `get_cells_capacity`) emits `instr(args, $1) > 0` for SQLite, which cannot use the B-tree index on `args` and forces a sequential scan of the entire `script` table: [2](#0-1) 

**3. `get_cells_capacity` holds a DB transaction open for two sequential queries**

Unlike `get_cells` (which calls `fetch_all` directly on the pool), `get_cells_capacity` opens an explicit transaction that holds a connection for the duration of both the capacity aggregation query and the tip-block lookup: [3](#0-2) 

**4. No query execution timeout**

The pool is configured with only a connection-acquisition timeout (60 s), not a statement execution timeout. A slow `instr()` scan runs unbounded: [4](#0-3) 

**5. Pool is capped at 10 connections** [5](#0-4) 

**6. `get_cells_capacity` has no concurrency or complexity guard**

`get_cells` and `get_transactions` both enforce `self.request_limit` on their pagination `limit` parameter. `get_cells_capacity` has no analogous guard — it accepts any `search_key` and immediately begins executing: [6](#0-5) [7](#0-6) 

**7. `request_limit` defaults to `usize::MAX`**

Even the pagination guard in `get_cells`/`get_transactions` is effectively disabled by default: [8](#0-7) 

---

### Impact Explanation

Ten concurrent `get_cells_capacity` calls with `script_search_mode=Partial` and a 1-byte `args` pattern (e.g., `0x00`) each acquire one of the 10 pool connections and hold it open for the duration of two sequential slow queries. Once all 10 connections are held, every other indexer RPC call (`get_cells`, `get_transactions`, `get_indexer_tip`) blocks for up to 60 seconds waiting for a connection, then fails. The attacker can sustain this state with a trivial loop of cheap unauthenticated HTTP requests, causing continuous RPC unavailability for the indexer module.

---

### Likelihood Explanation

- The rich-indexer is an opt-in module, but it is the recommended indexer for production CKB nodes and is widely deployed.
- The RPC endpoint defaults to localhost but is routinely exposed by node operators for dApp use.
- No authentication, no rate limiting, no query timeout, and no concurrency cap exist on this path.
- The attack requires only a standard HTTP client and knowledge of the JSON-RPC schema (publicly documented).
- Cost to the attacker: negligible (10 HTTP requests per 60-second window).

---

### Recommendation

1. **Add a query execution timeout** for SQLite (e.g., `PRAGMA busy_timeout`) and for Postgres (`statement_timeout`) so that slow `instr()` scans are killed after a bounded interval.
2. **Add a per-IP or global concurrency limit** on `get_cells_capacity` at the RPC layer.
3. **Reject or warn on `Partial` mode with very short args** (e.g., fewer than N bytes) since a 1-byte pattern matches nearly every row.
4. **Consider adding a dedicated connection pool** for write/indexing operations separate from the read/RPC pool so that RPC exhaustion cannot block block ingestion.

### Proof of Concept

```python
import asyncio, aiohttp, json

PAYLOAD = {
    "id": 1, "jsonrpc": "2.0", "method": "get_cells_capacity",
    "params": [{
        "script": {
            "code_hash": "0x" + "00" * 32,
            "hash_type": "type",
            "args": "0x00"          # 1-byte pattern → instr() matches almost everything
        },
        "script_type": "lock",
        "script_search_mode": "partial"
    }]
}

async def call(session):
    async with session.post("http://localhost:8114", json=PAYLOAD) as r:
        return await r.json()

async def main():
    async with aiohttp.ClientSession() as s:
        # Saturate all 10 pool connections
        tasks = [asyncio.create_task(call(s)) for _ in range(10)]
        # 11th call will block for acquire_timeout (60 s) then fail
        tasks.append(asyncio.create_task(call(s)))
        results = await asyncio.gather(*tasks, return_exceptions=True)
        print(results)

asyncio.run(main())
```

Assert: the 11th call returns an error after ~60 seconds; during that window, `get_indexer_tip` and all other indexer RPC calls also block.
