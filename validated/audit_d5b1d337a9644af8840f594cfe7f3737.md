### Title
Unchecked Integer Overflow in `get_block_economic_state` RPC Handler - (File: rpc/src/module/chain.rs)

### Summary
The `get_block_economic_state` RPC handler performs a plain `u64` addition `block_number + delay_length` without overflow protection. In Rust release builds, this silently wraps around on overflow, producing an incorrect `finalized_at_number`. The same arithmetic performed elsewhere in the codebase (in the reward calculator) correctly uses `checked_add`.

### Finding Description
In `rpc/src/module/chain.rs`, the `get_block_economic_state` function computes:

```rust
let delay_length = snapshot.consensus().finalization_delay_length();
let finalized_at_number = block_number + delay_length;
if block_number == 0 || snapshot.tip_number() < finalized_at_number {
    return Ok(None);
}
``` [1](#0-0) 

Both `block_number` (a `u64` block number from the chain) and `delay_length` (a `u64` consensus parameter) are added with the plain `+` operator. In Rust release builds, integer overflow wraps silently. If `block_number + delay_length > u64::MAX`, the result wraps to a small number, causing `finalized_at_number` to be far smaller than the actual finalization block number.

By contrast, the reward calculator performs the **identical** arithmetic using `checked_add`:

```rust
let finalization_parent_number = target
    .number()
    .checked_add(self.consensus.finalization_delay_length())
    .and_then(|number| number.checked_sub(1))
    .ok_or(DaoError::Overflow)?;
``` [2](#0-1) 

And the block assembler also uses `checked_add` for block number arithmetic:

```rust
let candidate_number = tip
    .number()
    .checked_add(1)
    .ok_or(BlockAssemblerError::Overflow)?;
``` [3](#0-2) 

The inconsistency is clear: the same `block_number + finalization_delay_length` pattern is guarded with `checked_add` in consensus-critical paths but left unchecked in the RPC handler.

### Impact Explanation
If `block_number + delay_length` wraps, `finalized_at_number` becomes a small number. The guard `snapshot.tip_number() < finalized_at_number` then incorrectly passes (tip is larger than the wrapped small value), and `snapshot.get_block_hash(finalized_at_number)` fetches the hash of a completely different (early) block. The function then returns economic state data for the wrong block — or fabricated/mismatched issuance, miner reward, and fee data — to the RPC caller. This is an incorrect data integrity result from a public RPC endpoint. [4](#0-3) 

### Likelihood Explanation
The likelihood of a natural overflow is extremely low: `block_number` would need to be near `u64::MAX` (~1.8 × 10¹⁹), which is practically unreachable on any real chain. However, the vulnerability class is real and the code is inconsistent with the rest of the codebase. An RPC caller can trigger this code path by supplying any valid block hash via `get_block_economic_state`. The impact is limited to incorrect RPC response data with no effect on consensus or fund safety.

### Recommendation
Replace the unchecked addition with `checked_add`, consistent with how the same computation is handled in `util/reward-calculator/src/lib.rs`:

```rust
let finalized_at_number = block_number
    .checked_add(delay_length)
    .ok_or_else(|| RPCError::invalid_params("block_number overflow"))?;
``` [5](#0-4) 

### Proof of Concept
1. Call `get_block_economic_state` with a block hash whose on-chain block number is `u64::MAX - delay_length + 1` (or any value where `block_number + delay_length > u64::MAX`).
2. In a release build, `finalized_at_number` wraps to a small value (e.g., `0` or `1`).
3. The guard `snapshot.tip_number() < finalized_at_number` evaluates to `false` (tip is much larger than the wrapped value), so execution continues.
4. `snapshot.get_block_hash(finalized_at_number)` returns the hash of block `0` or `1` instead of the correct finalization block.
5. The function returns economic state data belonging to a completely different block, silently producing incorrect output.
