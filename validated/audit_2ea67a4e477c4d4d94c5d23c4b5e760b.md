Audit Report

## Title
`MaturityVerifier::verify()` Omits Cellbase Maturity Check on `resolved_dep_groups` — (File: verification/src/transaction_verifier.rs)

## Summary
`MaturityVerifier::verify()` enforces the cellbase maturity rule on `resolved_inputs` and `resolved_cell_deps` but never iterates `resolved_dep_groups`. When a `CellDep` with `dep_type = DepGroup` is resolved, the container cell is placed in `resolved_dep_groups`, not `resolved_cell_deps`. An immature cellbase used as a dep-group container therefore bypasses the maturity gate entirely, allowing it to be referenced as a live dependency before the maturity period elapses.

## Finding Description
`ResolvedTransaction` holds three distinct `Vec<CellMeta>` collections: `resolved_inputs`, `resolved_cell_deps`, and `resolved_dep_groups`. In `resolve_transaction_dep`, when `cell_dep.dep_type() == DepType::DepGroup`, the container cell is pushed to `resolved_dep_groups` and its sub-cells are pushed to `resolved_cell_deps`. The non-DepGroup path pushes directly to `resolved_cell_deps`. `MaturityVerifier::verify()` defines the `cellbase_immature` closure and applies it to `resolved_inputs` and `resolved_cell_deps`, then returns `Ok(())` — `resolved_dep_groups` is never iterated. A cellbase output whose `data` field encodes a valid molecule `OutPointVec` submitted as a dep-group container therefore passes maturity verification regardless of epoch. Using `dep_type = code` on the same cellbase routes it to `resolved_cell_deps`, where the check fires and correctly rejects it, confirming the gap is specific to the dep-group path.

## Impact Explanation
The cellbase maturity rule is a consensus-level invariant. Because all nodes run the same flawed `MaturityVerifier`, they all accept the transaction silently — no fork occurs, but the protocol rule is violated chain-wide. A miner can immediately make their block reward cell usable as a dep-group dependency, circumventing the economic delay the maturity rule is designed to enforce. This matches **Critical (15001–25000 points): Vulnerabilities which could easily damage CKB economy**.

## Likelihood Explanation
Preconditions are minimal: any miner can produce a cellbase whose `data` field encodes a valid `OutPointVec` pointing to mature live cells. The transaction sender is any unprivileged RPC caller. No privileged keys, insider access, or victim mistakes are required. The gap has existed since dep-group support was introduced and is not gated by any hardfork switch.

## Recommendation
Add a third maturity check in `MaturityVerifier::verify()` for `resolved_dep_groups`, mirroring the existing checks, inserted between the `resolved_cell_deps` check and the final `Ok(())`:

```rust
if let Some(index) = self
    .transaction
    .resolved_dep_groups
    .iter()
    .position(cellbase_immature)
{
    return Err(TransactionError::CellbaseImmaturity {
        inner: TransactionErrorSource::CellDeps,
        index,
    }
    .into());
}
```

## Proof of Concept
1. Mine block N; craft the cellbase output so its `data` field is a valid molecule `OutPointVec` pointing to two mature live cells (e.g., genesis system cells).
2. Before `cellbase_maturity` epochs elapse, submit via RPC: `cell_deps = [{ out_point: <cellbase_outpoint>, dep_type: "dep_group" }]`.
3. Observe: the transaction is accepted into the tx-pool and committed. `MaturityVerifier` does not return `CellbaseImmaturity` because `resolved_dep_groups` is never iterated.
4. Confirm: replacing `dep_type` with `code` causes the same cellbase to be placed in `resolved_cell_deps`, where the check fires and the transaction is correctly rejected.
5. A unit test in `verification/src/tests/transaction_verifier.rs` mirroring existing `MaturityVerifier` test cases but populating `resolved_dep_groups` with an immature cellbase `CellMeta` will demonstrate that `verify()` currently returns `Ok(())` instead of `Err(CellbaseImmaturity)`.
