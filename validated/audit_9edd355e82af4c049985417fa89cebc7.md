### Title
Stale Price Returned Without Staleness Validation in `PythAggregatorV3` — (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatible adapter contract, calls `pyth.getPriceUnsafe(priceId)` in every price-returning function without any staleness check or positive-price validity check. This is the direct Pyth analog of the Chainlink min/max answer circuit breaker issue: both result in the oracle returning a price that does not reflect current market conditions, enabling protocols to operate on wrong prices.

---

### Finding Description

`PythAggregatorV3` is the official Pyth SDK contract for protocols migrating from Chainlink. It implements `AggregatorV3Interface` and is documented as a drop-in replacement for Chainlink price feeds. However, every price-returning function — `latestAnswer`, `latestRoundData`, `getRoundData`, `latestTimestamp`, and `decimals` — calls `pyth.getPriceUnsafe(priceId)`, which explicitly bypasses all staleness checks and returns the last stored price regardless of age.

```solidity
// PythAggregatorV3.sol line 53-56
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);  // no staleness check, no positive-price check
}

// PythAggregatorV3.sol line 99-119
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);  // unsafe
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

`IPyth.sol` explicitly documents `getPriceUnsafe` as: *"This function returns the most recent price update in this contract without any recency checks. This function is unsafe as the returned price update may be arbitrarily far in the past."*

Additionally, `latestAnswer()` and `latestRoundData()` return `int256(price.price)` where `price.price` is `int64` — no check that the value is positive. The `PythPriceOracleGetter` (Aave integration) correctly adds `if (price.price <= 0) revert InvalidNonPositivePrice()`, but `PythAggregatorV3` does not.

---

### Impact Explanation

Protocols using `PythAggregatorV3` as a Chainlink replacement (as documented in Pyth's official migration guide at `apps/developer-hub/content/docs/price-feeds/core/migrate-an-app-to-pyth/chainlink.mdx`) will receive stale prices when:

1. The Pyth price feed has not been updated recently (network issues, no keeper calling `updateFeeds`)
2. An asset's price has moved significantly since the last on-chain update

This is functionally identical to the Venus/LUNA scenario: if an asset crashes and the Pyth feed is stale, `PythAggregatorV3.latestRoundData()` returns the last stored (pre-crash) price without reverting. A lending protocol using this adapter would allow borrowing against the asset at the wrong (inflated) price, enabling undercollateralized positions and protocol insolvency.

---

### Likelihood Explanation

Medium-High. The official Pyth migration guide explicitly recommends deploying `PythAggregatorV3` as a drop-in Chainlink replacement without mentioning the need for additional staleness checks by the consuming protocol. Protocols that follow the guide verbatim inherit the vulnerability. The attack is permissionless: any user can interact with the lending/derivative protocol during a period of price feed staleness.

---

### Recommendation

Replace `getPriceUnsafe` with `getPriceNoOlderThan` in all price-returning functions, parameterized by a configurable `maxAge`:

```solidity
uint256 public maxPriceAge; // set in constructor

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxPriceAge);
    require(price.price > 0, "Invalid non-positive price");
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

The same fix applies to `latestAnswer`, `getRoundData`, `latestTimestamp`, and `decimals`.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a valid Pyth price feed (e.g., ETH/USD).
2. Call `updateFeeds` to push a fresh price on-chain (e.g., ETH = $3000).
3. Wait for the price to become stale (no further `updateFeeds` calls; ETH crashes to $100 off-chain).
4. Call `latestRoundData()` — it returns `answer = 3000e8` (the stale pre-crash price) without reverting.
5. A lending protocol using this adapter accepts ETH as collateral at $3000, allowing a user to borrow far more than the collateral is worth at the actual $100 market price. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
