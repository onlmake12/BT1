### Title
Missing Registration Check for `providerToCredit` in `executeCallback` Allows Fee Theft - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
`Echo.sol`'s `executeCallback` function does not verify that the `providerToCredit` address is a registered provider before crediting fees to it. After the exclusivity period expires, any unprivileged caller can supply an arbitrary address as `providerToCredit` and redirect the request's fee payment away from the legitimate provider.

### Finding Description
`executeCallback` enforces provider exclusivity only during the window `block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds`. Once that window closes, the function accepts any `providerToCredit` address without checking `_state.providers[providerToCredit].isRegistered`.

```solidity
function executeCallback(
    address providerToCredit,
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override {
    Request storage req = findActiveRequest(sequenceNumber);

    if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
        require(
            providerToCredit == req.provider,
            "Only assigned provider during exclusivity period"
        );
    }
    // ... price verification ...
    _state.providers[providerToCredit].accruedFeesInWei +=
        SafeCast.toUint128((req.fee + msg.value) - pythFee);
``` [1](#0-0) 

The `ProviderInfo` struct tracks `isRegistered` as the eligibility flag, and `requestPriceUpdatesWithCallback` correctly gates on it. However, `executeCallback` omits the equivalent gate for `providerToCredit`. [2](#0-1) [3](#0-2) 

### Impact Explanation
An attacker who pre-registers as a provider (permissionless via `registerProvider`) and sets themselves as their own fee manager can front-run the legitimate provider's `executeCallback` call after the exclusivity period. The fees for the fulfilled request are credited to the attacker's `accruedFeesInWei` slot. The attacker then calls `withdrawAsFeeManager` to extract those funds. The legitimate provider receives nothing for fulfilling the request, and the user's callback is still executed (so the user is unharmed), but the economic incentive for honest providers is undermined. [4](#0-3) 

### Likelihood Explanation
Registration is permissionless and cheap. The exclusivity period is a fixed, observable on-chain value. Any MEV bot or attacker monitoring the mempool can detect pending `executeCallback` transactions and front-run them with a higher gas price. The attack requires no privileged access, no leaked keys, and no governance majority. [5](#0-4) 

### Recommendation
Add a registration check for `providerToCredit` in `executeCallback`, mirroring the check already present in `requestPriceUpdatesWithCallback`:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit not registered"
);
```

Alternatively, restrict `providerToCredit` to be the request's originally assigned `req.provider` at all times, removing the open-competition model after exclusivity.

### Proof of Concept
1. Attacker calls `registerProvider(baseFee, feePerFeed, feePerGas)` — permissionless.
2. Attacker calls `setFeeManager(attackerAddress)` to set themselves as their own fee manager.
3. A user submits a request via `requestPriceUpdatesWithCallback(defaultProvider, ...)`.
4. Attacker waits for `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`.
5. Attacker calls `executeCallback(attackerAddress, sequenceNumber, updateData, priceIds)` with a higher gas price than the legitimate provider's pending transaction.
6. `_state.providers[attackerAddress].accruedFeesInWei` is incremented by the full fee.
7. Attacker calls `withdrawAsFeeManager(attackerAddress, amount)` to drain the stolen fees. [6](#0-5) [7](#0-6)
