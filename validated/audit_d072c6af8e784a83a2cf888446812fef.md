### Title
Missing Rate Limiter in `BlockFilter` Protocol Handler Enables Unbounded RocksDB Read Amplification — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

### Summary

The `BlockFilter` P2P protocol handler has no per-peer rate limiter, unlike `Relayer` and `HolePunching` which both enforce 30 req/sec per `(PeerIndex, message_type)`. Any unprivileged peer can send `GetBlockFilters` messages in a tight loop, each triggering up to 2,000 RocksDB reads (1,000 × `get_block_hash` + 1,000 × `get_block_filter`). With up to 125 inbound peers (default `max_peers`), this yields up to ~250,000 RocksDB reads per processing round with no throttle.

### Impact Explanation

Each `GetBlockFilters` message is 8 bytes (`start_number` only). With 125 peers each sending in a tight loop:
- Up to 125 × 2,000 = **250,000 RocksDB reads per processing round** with zero throttle
- `get_block_filter` reads from `COLUMN_BLOCK_FILTER` are not covered by the snapshot in-memory index; they hit the RocksDB block cache or disk
- Sustained I/O pressure degrades block validation and storage writes, which share the same RocksDB instance
- Degraded block write throughput directly delays compact block relay and block propagation

The "consensus deviation" claim (node falls behind tip) is plausible under sustained multi-peer flood but not guaranteed; the primary confirmed impact is **I/O saturation and degraded block propagation**.

### Likelihood Explanation

- Attack requires only a valid P2P connection (no PoW, no stake, no key)
- The `GetBlockFilters` message is trivially small and valid
- The filter protocol is on by default in the shipped `ckb.toml`
- The contrast with `Relayer` and `HolePunching` confirms this is an unintentional omission, not a deliberate design choice

### Recommendation

Add a `governor`-based `RateLimiter<(PeerIndex, u32)>` to `BlockFilter`, mirroring the pattern in `Relayer::new()` (30 req/sec per peer per message type). Check it at the top of `try_process()` before dispatching to any of the three process handlers. Also apply the same fix to `GetBlockFilterHashesProcess` and `GetBlockFilterCheckPointsProcess`.

### Proof of Concept

1. Sync a CKB node to a chain with ≥1,000 blocks with filter data built.
2. Open N TCP connections (e.g., N=50) to the node's P2P port, each negotiating the `/ckb/filter` protocol.
3. From each connection, send `GetBlockFilters { start_number: 0 }` in a tight loop.
4. Monitor RocksDB read IOPS (via `ckb_metrics` or `iostat`) and compact-block relay latency.
5. Assert: RocksDB read rate scales linearly with N and block relay latency exceeds acceptable bounds, while the same test against the `Relayer` protocol is throttled at 30 req/sec per peer.
