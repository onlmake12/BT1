### Title
Stale Tx-Pool Snapshot Makes Staleness Guard Ineffective in `submit_entry` — (`tx-pool/src/process.rs`)

---

### Summary

The CKB transaction pool (`TxPool`) holds its own `Arc<Snapshot>` that is updated **asynchronously** after a new block is committed. The staleness guard in `submit_entry` is supposed to detect when the chain tip has advanced between `pre_check` and final admission, but it compares two values both derived from the **same stale tx-pool snapshot**, making the guard a no-op during the window between chain tip update and tx-pool snapshot update.

---

### Finding Description

When a new block is committed in `chain/src/verify.rs`, the chain's shared snapshot is updated immediately:

```rust
self.shared.store_snapshot(Arc::clone(&new_snapshot));  // chain tip updated
// ...
tx_pool_controller.update_tx_pool_for_reorg(...)        // async notification
``` [1](#0-0) 

The tx-pool snapshot is only updated **inside** `_update_tx_pool_for_reorg` when it acquires the write lock:

```rust
tx_pool.snapshot = Arc::clone(&snapshot);
``` [2](#0-1) 

The `TxPool` struct itself documents this gap explicitly:

```rust
/// Tx-pool owned snapshot, it may not consistent with chain cause tx-pool update snapshot asynchronously
pub(crate) fn snapshot(&self) -> &Snapshot {
    &self.snapshot
}
``` [3](#0-2) 

The transaction admission path is:

1. **`pre_check`** acquires a read lock, calls `tx_pool.cloned_snapshot()` (stale), resolves cells and determines `TxStatus` (`Fresh`/`Gap`/`Proposed`) via `get_tx_status`:

```rust
fn get_tx_status(snapshot: &Snapshot, short_id: &ProposalShortId) -> TxStatus {
    if snapshot.proposals().contains_proposed(short_id) {
        TxStatus::Proposed
    } else if snapshot.proposals().contains_gap(short_id) {
        TxStatus::Gap
    } else {
        TxStatus::Fresh
    }
}
``` [4](#0-3) 

2. **`submit_entry`** acquires a write lock, again calls `tx_pool.cloned_snapshot()` (still the same stale snapshot), and checks:

```rust
let tip_hash = snapshot.tip_hash();
if pre_resolve_tip != tip_hash {
    // redo time_relative verify
    status = check_rtx(tx_pool, &snapshot, &entry.rtx)?;
    ...
}
``` [5](#0-4) 

**The flaw**: `pre_resolve_tip` is the tip hash from `pre_check`'s snapshot (`tx_pool.cloned_snapshot()`), and `tip_hash` in `submit_entry` is also from `tx_pool.cloned_snapshot()`. If the tx-pool snapshot has not yet been updated (i.e., `update_tx_pool_for_reorg` has not yet run), **both values are the same stale hash**, so `pre_resolve_tip != tip_hash` is always `false` and the re-check is never triggered.

The correct comparison should be against `shared.snapshot().tip_hash()` — the chain's authoritative current tip — not the tx-pool's own potentially-stale snapshot.

Both `with_tx_pool_read_lock` and `with_tx_pool_write_lock` source their snapshot from the tx-pool's internal field:

```rust
pub(crate) async fn with_tx_pool_read_lock<U, F: FnMut(&TxPool, Arc<Snapshot>) -> U>(
    &self, mut f: F,
) -> (U, Arc<Snapshot>) {
    let tx_pool = self.tx_pool.read().await;
    let snapshot = tx_pool.cloned_snapshot();   // <-- stale tx-pool snapshot
    ...
}
``` [6](#0-5) 

---

### Impact Explanation

During the window between `shared.store_snapshot(new_snapshot)` and the completion of `update_tx_pool_for_reorg`:

1. **`TxStatus` misclassification**: A transaction whose `ProposalShortId` was included in the new block's proposal zone will be classified as `TxStatus::Fresh` (pending) instead of `TxStatus::Proposed`. It is added to the pending pool via `add_pending` instead of `add_proposed`, causing it to miss its commit window in the immediately following block. [7](#0-6) 

2. **Stale cell resolution**: A cell spent in the newly committed block still appears live in the stale snapshot. A transaction spending that cell passes `resolve_tx_from_pool` and `check_rtx_from_pool` (both use the stale snapshot), gets admitted to the pool with a dead input, and is only cleaned up when `remove_committed_txs` runs asynchronously. [8](#0-7) 

The first impact (misclassification) is the direct analog to the Notional bug: just as `checkRebalance()` uses stale `PrimeCashFactors` causing the bot to miss rebalancing, CKB's tx-pool uses a stale snapshot causing transactions to miss their proposal-to-commit promotion window.

---

### Likelihood Explanation

This window opens on **every block commit**. Any transaction submitted via RPC (`send_transaction`) or P2P relay (`RelayTransaction`) during this window is affected. Under normal mainnet conditions with ~10-second block intervals and a busy mempool, the window is small but non-zero. Under load (many transactions, slow reorg processing), the window widens. The entry path requires no special privileges — any unprivileged RPC caller or P2P transaction relayer can trigger this.

---

### Recommendation

In `submit_entry`, compare `pre_resolve_tip` against the **chain's authoritative snapshot** rather than the tx-pool's own snapshot:

```rust
// Instead of:
let tip_hash = snapshot.tip_hash();  // tx-pool's own stale snapshot

// Use:
let tip_hash = self.shared.snapshot().tip_hash();  // chain's current tip
```

This mirrors the pattern already used in `test/src/node.rs` where the test harness explicitly waits for the tx-pool tip hash to match the chain tip hash before proceeding, acknowledging the asynchronous gap:

```rust
/// The states of chain and txpool are updated asynchronously. Which means that the chain has
/// updated to the newest tip but txpool not.
``` [9](#0-8) 

---

### Proof of Concept

**Sequence of events:**

1. Block N is committed. `shared.store_snapshot(snapshot_N)` is called. `tx_pool.snapshot` still holds `snapshot_{N-1}`.
2. Block N's proposal zone contains `tx_A`'s `ProposalShortId`.
3. A peer relays `tx_A` to the node. `resumeble_process_tx` → `pre_check` is called.
4. `pre_check` calls `tx_pool.cloned_snapshot()` → gets `snapshot_{N-1}`. `get_tx_status` checks `snapshot_{N-1}.proposals()` → `tx_A` is NOT in proposals → returns `TxStatus::Fresh`. `pre_resolve_tip = snapshot_{N-1}.tip_hash()`.
5. `submit_entry` is called. It calls `tx_pool.cloned_snapshot()` → still `snapshot_{N-1}` (reorg not yet processed). `tip_hash = snapshot_{N-1}.tip_hash()`. `pre_resolve_tip == tip_hash` → **staleness guard does not fire**.
6. `_submit_entry` is called with `TxStatus::Fresh` → `tx_pool.add_pending(entry)`. `tx_A` is placed in the pending pool.
7. `update_tx_pool_for_reorg` finally runs, updates `tx_pool.snapshot` to `snapshot_N`. `tx_A` is now in the pending pool but should have been in the proposed pool. It will not be committed in block N+1 unless re-promoted, potentially missing its commit window entirely. [10](#0-9) [11](#0-10)

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
        (ret, snapshot)
    }

fn _submit_entry(
    tx_pool: &mut TxPool,
    status: TxStatus,
    entry: TxEntry,
    callbacks: &Callbacks,
) -> Result<HashSet<TxEntry>, Reject> {
    let tx_hash = entry.transaction().hash();
    debug!("submit_entry {:?} {}", status, tx_hash);
    let (succ, evicts) = match status {
        TxStatus::Fresh => tx_pool.add_pending(entry.clone())?,
        TxStatus::Gap => tx_pool.add_gap(entry.clone())?,
        TxStatus::Proposed => tx_pool.add_proposed(entry.clone())?,
    };
    if succ {
        match status {
            TxStatus::Fresh => callbacks.call_pending(&entry),
            TxStatus::Gap => callbacks.call_pending(&entry),
            TxStatus::Proposed => callbacks.call_proposed(&entry),
        }
    }
    Ok(evicts)
}

    pub fn get_tip_tx_pool_info(&self) -> TxPoolInfo {

        let instant = Instant::now();
