### Title
Unbounded `while` Loop in `constructProviderCommitment` Enables Gas-Exhaustion DoS on Entropy Reveals When `maxNumHashes == 0` — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `constructProviderCommitment` function in `Entropy.sol` contains an unbounded `while (numHashes > 0)` loop. The `numHashes` value stored per request is computed as `assignedSequenceNumber - currentCommitmentSequenceNumber` at request time, and is only bounded by a guard that is **silently skipped when `maxNumHashes == 0`** (the default for any provider that has not explicitly configured it). An unprivileged attacker can make many cheap requests against such a provider to inflate `numHashes` for subsequent legitimate user requests, causing the provider's `revealWithCallback` transaction to exceed the block gas limit. The user's entropy request can never be fulfilled and their paid fee is permanently locked in the contract.

---

### Finding Description

**Root cause 1 — Unbounded loop:** [1](#0-0) 

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
```

Each iteration executes one `keccak256`, costing ~30 gas. With `numHashes = 1,000,000` the loop alone consumes ~30 M gas, which meets or exceeds the block gas limit on most EVM chains.

**Root cause 2 — Guard is silently bypassed when `maxNumHashes == 0`:** [2](#0-1) 

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

The condition `providerInfo.maxNumHashes != 0` means the entire bound check is **skipped** when a provider has not set `maxNumHashes`. The field defaults to `0` in the `ProviderInfo` struct: [3](#0-2) 

Any provider that registered without explicitly calling a setter for `maxNumHashes` is therefore unprotected.

**How `numHashes` is inflated:**

`numHashes` is the gap between the new request's sequence number and the provider's `currentCommitmentSequenceNumber` at the moment of the request. An attacker who submits N requests before a victim's request causes the victim's stored `numHashes ≈ N`. Because `numHashes` is written once at request time and never updated, the provider cannot reduce it by revealing earlier requests.

---

### Impact Explanation

1. The provider's `revealWithCallback` call invokes `constructProviderCommitment(req.numHashes, providerContribution)`.
2. With a sufficiently large `numHashes`, the transaction reverts with out-of-gas.
3. The request remains active in the contract; there is no cancellation or refund path visible in the contract.
4. The victim user's paid fee is permanently locked, and their application never receives the random number callback. [4](#0-3) 

---

### Likelihood Explanation

- Any provider that registered without setting `maxNumHashes` is vulnerable (the field defaults to `0`).
- The attack cost scales with the per-request fee on the target chain. On low-fee EVM chains (Polygon, BSC, Arbitrum, etc.) where fees can be fractions of a cent, making ~1 M requests to push `numHashes` past the gas limit is economically feasible for a motivated attacker.
- The attacker does not need any privileged role; `requestV2` is a public payable function callable by anyone. [5](#0-4) 

---

### Recommendation

1. **Remove the `!= 0` bypass**: Change the guard to always enforce a maximum, e.g., require `maxNumHashes > 0` at provider registration time and remove the `!= 0` short-circuit:
   ```solidity
   if (req.numHashes > providerInfo.maxNumHashes) {
       revert EntropyErrors.LastRevealedTooOld();
   }
   ```
2. **Enforce `maxNumHashes` at registration**: Require providers to set a non-zero `maxNumHashes` during `register()`, preventing the unprotected default.
3. **Cap `numHashes` unconditionally**: Add a protocol-level hard cap (e.g., `MAX_NUM_HASHES`) independent of provider configuration so that no single reveal can ever exceed a safe gas budget.

---

### Proof of Concept

```
Setup: Provider P registers with maxNumHashes = 0 (default).
       Provider fee = 0.001 USD/request (e.g., Polygon).

Step 1: Attacker calls requestV2() × 1,000,000 times, paying fees.
        After each call, providerInfo.sequenceNumber increments.
        providerInfo.currentCommitmentSequenceNumber stays at 0
        (no reveals have happened yet).

Step 2: Victim calls requestV2().
        req.numHashes = 1,000,001 - 0 = 1,000,001.
        Guard check: maxNumHashes == 0 → condition false → no revert.
        Request stored with numHashes = 1,000,001.

Step 3: Provider (Fortuna) calls revealWithCallback(P, victimSeqNum, ...).
        constructProviderCommitment(1_000_001, revelation) executes
        1,000,001 keccak256 iterations ≈ 30,000,030 gas → exceeds block gas limit.
        Transaction reverts with out-of-gas.

Result: Victim's request is permanently stuck. Fee is locked.
        No refund mechanism exists in the contract.
``` [1](#0-0) [2](#0-1)
