### Title
`HeaderDigest::verify()` Does Not Check for Invalid `compact_to_difficulty` Return Value, Allowing Forged MMR Proofs — (`util/types/src/utilities/merkle_mountain_range.rs`)

---

### Summary

`compact_to_difficulty` returns `U256::zero()` as a sentinel for invalid or overflowing compact targets (the CKB analog of `isInvalid = true`). `HeaderDigest::verify()` calls `compact_to_difficulty` to reconstruct the expected total difficulty of a block range, but never checks whether the returned value is zero before using it. A malicious light-client server can craft a `HeaderDigest` message with an overflowing `compact_target` and `total_difficulty = 0`; the verification arithmetic collapses to `0 == 0` and the forged digest passes.

---

### Finding Description

`compact_to_target` returns a `(U256, bool)` tuple where the `bool` is an overflow flag:

```rust
// util/types/src/utilities/difficulty.rs
pub fn compact_to_target(compact: u32) -> (U256, bool) {
    ...
    let overflow = !mantissa.is_zero() && (exponent > 32);
    (ret, overflow)
}
```

`compact_to_difficulty` wraps this and silently maps any invalid input to `U256::zero()`:

```rust
pub fn compact_to_difficulty(compact: u32) -> U256 {
    let (target, overflow) = compact_to_target(compact);
    if target.is_zero() || overflow {
        return U256::zero();   // ← sentinel for "invalid"
    }
    target_to_difficulty(&target)
}
```

The PoW engines (`EaglesongPowEngine`, `EaglesongBlake2bPowEngine`) correctly call `compact_to_target` directly and reject blocks when `overflow` is true:

```rust
// pow/src/eaglesong.rs
let (block_target, overflow) = compact_to_target(header.raw().compact_target().into());
if block_target.is_zero() || overflow {
    return false;
}
```

However, `HeaderDigest::verify()` — used in the light-client MMR proof path — calls `compact_to_difficulty` and uses the result without any zero-check:

```rust
// util/types/src/utilities/merkle_mountain_range.rs  lines 108-117
let blocks_count = end_number - start_number + 1;
let block_difficulty = compact_to_difficulty(start_compact_target); // may return 0 for invalid target
let total_difficulty_calculated = block_difficulty * blocks_count;  // 0 * N = 0
if total_difficulty != total_difficulty_calculated {
    return Err(...)
}
```

If `start_compact_target` is an overflowing value (e.g., `0xff123456`), `compact_to_difficulty` returns `U256::zero()`, `total_difficulty_calculated` becomes `0`, and the check `total_difficulty != 0` passes trivially when the attacker also sets `total_difficulty = 0` in the digest.

Similarly, `VerifiableHeader::total_difficulty()` calls `compact_to_difficulty` without checking for zero:

```rust
// util/types/src/utilities/merkle_mountain_range.rs  lines 264-268
pub fn total_difficulty(&self) -> U256 {
    let parent_total_difficulty: U256 = self.parent_chain_root.total_difficulty().into();
    let block_difficulty = compact_to_difficulty(self.header.compact_target());
    parent_total_difficulty + block_difficulty  // block_difficulty silently 0 if target invalid
}
```

---

### Impact Explanation

A malicious full-node acting as a light-client server can craft `HeaderDigest` messages with:
- `start_compact_target` = any overflowing value (e.g., `0xff123456`)
- `total_difficulty` = `U256::zero()`

`HeaderDigest::verify()` will accept such a digest because the difficulty cross-check degenerates to `0 == 0`. The light client is then presented with a chain whose cumulative difficulty is falsely reported as zero (or any attacker-chosen value), undermining the light client's fork-choice logic and its ability to distinguish the heaviest chain. This is the direct CKB analog of the Polynomial finding: a validity sentinel returned by a lower-level function is silently ignored by a higher-level consumer, allowing invalid data to flow through security-critical checks.

---

### Likelihood Explanation

Any unprivileged peer that the light client connects to can send crafted `HeaderDigest` / `SendLastStateProof` messages. No special privilege, key, or majority hash power is required. The attacker only needs to be a reachable peer of the light client.

---

### Recommendation

In `HeaderDigest::verify()`, after calling `compact_to_difficulty`, explicitly reject a zero result as an invalid compact target:

```rust
let block_difficulty = compact_to_difficulty(start_compact_target);
if block_difficulty.is_zero() {
    return Err("invalid compact_target: results in zero difficulty".to_string());
}
let total_difficulty_calculated = block_difficulty * blocks_count;
```

Alternatively, call `compact_to_target` directly and check the `overflow` flag, mirroring the pattern already used in the PoW engines. Apply the same guard in `VerifiableHeader::total_difficulty()`.

---

### Proof of Concept

1. A malicious full-node constructs a `HeaderDigest` with:
   - `start_compact_target = end_compact_target = 0xff123456` (overflow; `compact_to_difficulty` → `0`)
   - `total_difficulty = U256::zero()`
   - `start_epoch_number == end_epoch_number` (same-epoch branch is taken)
   - Any valid `start_number ≤ end_number`

2. The light client calls `HeaderDigest::verify()` on this digest.

3. `compact_to_difficulty(0xff123456)` returns `U256::zero()`.

4. `total_difficulty_calculated = U256::zero() * blocks_count = U256::zero()`.

5. The check `total_difficulty (0) != total_difficulty_calculated (0)` is `false`; no error is returned.

6. The forged digest passes verification, and the light client accepts a chain segment with fabricated (zero) total difficulty. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    let exponent = compact >> 24;
