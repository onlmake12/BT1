### Title
Indexer Pool Dead-Cell Tracking Clears Cell Lock on Any Single Rejection, Ignoring Remaining Pool References - (File: `util/indexer-sync/src/pool.rs`)

---

### Summary

The indexer's `Pool` struct tracks which cells are "dead" (being spent by pending transactions) using a plain `HashSet<OutPoint>`. When any transaction referencing a cell is rejected — including via RBF replacement — the cell is unconditionally removed from `dead_cells`, even if another transaction currently in the pool still spends that same cell. This causes the indexer to transiently report a cell as live when it is actually still locked by a pending pool transaction.

---

### Finding Description

`util/indexer-sync/src/pool.rs` defines `Pool` with:

```rust
pub struct Pool {
    dead_cells: HashSet<OutPoint>,
}
```

The three mutating methods are:

```rust
pub fn new_transaction(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        self.dead_cells.insert(input.previous_output());
    }
}

pub fn transaction_rejected(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        self.dead_cells.remove(&input.previous_output());   // ← unconditional remove
    }
}

pub fn transaction_committed(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        self.dead_cells.remove(&input.previous_output());   // ← unconditional remove
    }
}
``` [1](#0-0) 

With RBF enabled, the tx-pool's `process_rbf` function (in `tx-pool/src/process.rs`) removes the old conflicting transaction and fires `call_reject` for it **before** `_submit_entry` inserts the replacement transaction and fires the `pending` callback: [2](#0-1) 

Both `notify_reject_transaction` and `notify_new_transaction` are dispatched asynchronously via channels: [3](#0-2) 

The indexer's `PoolService` processes these events in a `tokio::select!` loop, one at a time: [4](#0-3) 

**Vulnerable sequence (RBF path):**

1. `tx1` spending `cell_A` enters pool → `new_transaction(tx1)` → `dead_cells = {cell_A}`
2. `tx2` (RBF replacement) spending the same `cell_A` is submitted
3. `tx1` is rejected → `transaction_rejected(tx1)` → `dead_cells.remove(cell_A)` → `dead_cells = {}` ← **cell_A incorrectly appears live**
4. `tx2` is accepted → `new_transaction(tx2)` → `dead_cells = {cell_A}` ← eventually corrected

Between steps 3 and 4, `cell_A` is absent from `dead_cells` even though `tx2` (already in the pool) still spends it. Any indexer query during this window returns `cell_A` as a live cell.

The rich indexer's `get_cells` handler reads `dead_cells` to exclude pool-spent outputs from results: [5](#0-4) 

The same pattern applies in `get_cells_capacity`. The root cause is structural: `HashSet` carries no reference count, so a single removal unconditionally clears the lock regardless of how many pool transactions still reference the cell — the exact same class of bug as the Forta slashing freeze issue.

---

### Impact Explanation

During the race window between the reject notification for the old transaction and the new-transaction notification for the RBF replacement, the indexer incorrectly exposes the replaced cell as live. Applications or wallets querying `get_cells` or `get_cells_capacity` via the indexer RPC will receive stale/incorrect data: a cell that is actively being spent by a pending transaction appears available. This can cause downstream applications to attempt to re-spend the cell, construct invalid transactions, or make incorrect balance calculations.

---

### Likelihood Explanation

RBF is a supported, documented feature of the CKB tx-pool (enabled when `min_rbf_rate > min_fee_rate`). Any unprivileged RPC caller can trigger this by submitting a transaction and then submitting an RBF replacement. The race window exists on every RBF replacement. The window is narrow (bounded by async channel scheduling latency) but is structurally guaranteed to exist on every RBF event, making it reliably triggerable by any tx-pool submitter who uses the indexer.

---

### Recommendation

Replace `HashSet<OutPoint>` with a reference-counted map `HashMap<OutPoint, usize>`. Increment the counter in `new_transaction` and only remove the entry in `transaction_rejected` / `transaction_committed` when the counter reaches zero:

```rust
pub struct Pool {
    dead_cells: HashMap<OutPoint, usize>,
}

pub fn new_transaction(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        *self.dead_cells.entry(input.previous_output()).or_insert(0) += 1;
    }
}

pub fn transaction_rejected(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        let out_point = input.previous_output();
        if let Some(count) = self.dead_cells.get_mut(&out_point) {
            *count -= 1;
            if *count == 0 {
                self.dead_cells.remove(&out_point);
            }
        }
    }
}
```

Apply the same pattern to `transaction_committed`.

---

### Proof of Concept

1. Enable RBF on a node (`min_rbf_rate > min_fee_rate`) with the rich indexer enabled (`index_tx_pool = true`).
2. Submit `tx1` spending `cell_A` via `send_transaction` RPC.
3. Immediately submit `tx2` (RBF replacement) spending the same `cell_A` with a higher fee.
4. Between the `reject_transaction` notification for `tx1` and the `new_transaction` notification for `tx2`, call `get_cells` on the indexer for `cell_A`.
5. Observe that `cell_A` is returned as a live cell, even though `tx2` (already accepted into the pool) is spending it.

The window can be widened for testing by inserting a deliberate `tokio::time::sleep` in the `new_transaction` branch of `PoolService::index_tx_pool` to simulate slow channel delivery.
