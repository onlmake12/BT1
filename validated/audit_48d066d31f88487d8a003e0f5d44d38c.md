### Title
Missing Zero-Address Validation for `wormhole` Parameter in `PythUpgradable.initialize` — (File: `target_chains/ethereum/contracts/contracts/pyth/PythUpgradable.sol`)

---

### Summary

`PythUpgradable.initialize` accepts a `wormhole` address parameter and stores it without any zero-address check. If initialized with `address(0)`, the contract is permanently bricked: all price update submissions and all governance instructions fail, with no recovery path. Sibling contracts in the same codebase (`ExecutorUpgradable`, `Executor._initialize`) explicitly guard against this with `require(wormhole != address(0), ...)`, making the omission in `PythUpgradable` an inconsistent and dangerous oversight.

---

### Finding Description

`PythUpgradable.initialize` passes the `wormhole` argument directly into `Pyth._initialize`, which calls `setWormhole(wormhole)` with no zero-address guard:

```solidity
// PythUpgradable.sol L23-L48
function initialize(
    address wormhole,
    ...
) public initializer {
    __Ownable_init();
    __UUPSUpgradeable_init();
    Pyth._initialize(wormhole, ...);   // no zero-check before this
    renounceOwnership();
}
``` [1](#0-0) 

`Pyth._initialize` immediately stores the value:

```solidity
// Pyth.sol L30
setWormhole(wormhole);
``` [2](#0-1) 

By contrast, `ExecutorUpgradable.initialize` and `Executor._initialize` both guard the same parameter:

```solidity
require(wormhole != address(0), "wormhole is zero address");
``` [3](#0-2) [4](#0-3) 

`EntropyUpgradable.initialize` and `SchedulerUpgradeable.initialize` similarly validate all address parameters before use. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

If `wormhole = address(0)` is passed at initialization:

1. **All price update submissions fail permanently.** Every call to `updatePriceFeeds` → `updatePriceInfosFromAccumulatorUpdate` → `wormhole().parseAndVerifyVM(...)` calls into `address(0)`, returning empty bytes that cannot be decoded as a valid `IWormhole.VM`, causing a revert on every update attempt.

2. **All governance instructions fail permanently.** `executeGovernanceInstruction` calls `verifyGovernanceVM` → `wormhole().parseAndVerifyVM(...)` — same failure path. This means the governance action `SetWormholeAddress` (the only recovery mechanism) is also unreachable, permanently locking the contract in a broken state.

3. **No recovery path exists.** `renounceOwnership()` is called at the end of `initialize`, so there is no owner who could call any privileged function even if one existed.

---

### Likelihood Explanation

The `initialize` function is `public initializer` and is called once at proxy deployment. A deployer who accidentally passes `address(0)` (e.g., due to a misconfigured deployment script, a missing environment variable resolving to zero, or a copy-paste error) would permanently brick the deployed Pyth price feed contract. The inconsistency with `ExecutorUpgradable` — which guards the identical parameter — increases the probability of this being an unintentional omission rather than a deliberate design choice. Deployment scripts for `PythUpgradable` pass the wormhole address from external configuration, making misconfiguration realistic. [7](#0-6) 

---

### Recommendation

Add a zero-address check for `wormhole` in `PythUpgradable.initialize`, consistent with `ExecutorUpgradable`:

```solidity
function initialize(
    address wormhole,
    ...
) public initializer {
    require(wormhole != address(0), "wormhole is zero address");
    __Ownable_init();
    __UUPSUpgradeable_init();
    Pyth._initialize(wormhole, ...);
    renounceOwnership();
}
```

Alternatively, add the check inside `Pyth._initialize` itself, so all callers benefit.

---

### Proof of Concept

1. Deploy `PythUpgradable` implementation.
2. Deploy an `ERC1967Proxy` pointing to the implementation, calling `initialize(address(0), ...)` with any valid non-empty `dataSourceEmitterChainIds`/`dataSourceEmitterAddresses` arrays.
3. Initialization succeeds — no revert.
4. Call `updatePriceFeeds(validUpdateData)` with a properly formed accumulator update. The call reverts because `wormhole().parseAndVerifyVM(...)` is a call to `address(0)`.
5. Call `executeGovernanceInstruction(validGovernanceVAA)`. Same revert — governance is also inaccessible.
6. Confirm `owner()` returns `address(0)` (ownership was renounced), so no privileged recovery is possible. [1](#0-0) [8](#0-7)
