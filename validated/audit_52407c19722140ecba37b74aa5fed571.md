Looking at the exact code paths in question:

### Title
Missing Peer Ban for `OutPointError::InvalidDepGroup` Enables Unbounded Relay Spam — (`util/types/src/core/tx_pool.rs`, `tx-pool/src/process.rs`)

### Summary

`Reject::Resolve(OutPointError::InvalidDepGroup)` is classified as `is_malformed_tx() == false`, which prevents peer banning in `after_process`. Because `is_allowed_relay()` simultaneously returns `true` for this variant, an attacker who deploys a cell with malformed `OutPointVec` data and relays transactions referencing it as a `DepGroup` dep can spam a node indefinitely from the same peer connection without ever being banned.

---

### Finding Description

**`is_malformed_tx()` classification gap**

In `util/types/src/core/tx_pool.rs`, the `is_malformed_tx()` method explicitly handles only one `Resolve` variant:

```rust
pub fn is_malformed_tx(&self) -> bool {
    match self {
        Reject::Malformed(_, _) => true,
        Reject::DeclaredWrongCycles(..) => true,
        Reject::Verification(err) => is_malformed_from_verification(err),
        Reject::Resolve(OutPointError::OverMaxDepExpansionLimit) => true,
        _ => false,   // <-- InvalidDepGroup falls here
    }
}
``` [1](#0-0) 

`Reject::Resolve(OutPointError::InvalidDepGroup)` falls into the `_ => false` arm. Consequently `is_allowed_relay()` returns `true` for it:

```rust
pub fn is_allowed_relay(&self) -> bool {
    matches!(self, Reject::DeclaredWrongCycles(..))
        || (!matches!(self, Reject::LowFeeRate(..)) && !self.is_malformed_tx())
}
``` [2](#0-1) 

**Ban gate in `after_process`**

The only place a remote peer is banned for a bad transaction is:

```rust
if reject.is_malformed_tx() {
    self.ban_malformed(peer, format!("reject {reject}")).await;
}
``` [3](#0-2) 

Because `is_malformed_tx()` is `false` for `InvalidDepGroup`, `ban_malformed` is never called, and the peer remains connected.

**How `InvalidDepGroup` is produced**

In `resolve_transaction_dep`, when a `DepGroup`-typed cell dep is encountered, the cell data is parsed as `OutPointVec`. If parsing fails, the error is:

```rust
let sub_out_points =
    parse_dep_group_data(data).map_err(|_| OutPointError::InvalidDepGroup(outpoint))?;
``` [4](#0-3) 

This is wrapped into `Reject::Resolve(OutPointError::InvalidDepGroup)` by the tx-pool's `resolve_tx` helper and surfaces in `after_process`.

**Inconsistency with `OverMaxDepExpansionLimit`**

`OutPointError::OverMaxDepExpansionLimit` — another dep-resolution error — is explicitly treated as malformed: [5](#0-4) 

`InvalidDepGroup` is structurally equivalent in permanence: a cell whose data is not a valid `OutPointVec` will never become a valid dep group while it remains live. The omission is an inconsistency, not a design choice.

---

### Impact Explanation

An attacker who has deployed one malformed-data cell on-chain can craft an unlimited stream of distinct transactions (varying outputs to produce different tx hashes) that all reference that cell as a `DepGroup` dep. Each transaction:

1. Passes `non_contextual_verify` (structurally valid transaction).
2. Enters the verify queue.
3. Fails in `pre_check → resolve_tx → resolve_transaction_dep → parse_dep_group_data`.
4. Returns `Reject::Resolve(InvalidDepGroup)` → no ban → peer stays connected.

The attacker can reuse the same live input cell across all crafted transactions because each tx is rejected before the input cell is consumed. The node performs a DB lookup and data parse for every transaction, with no rate-limiting mechanism tied to this specific rejection path and no peer eviction.

---

### Likelihood Explanation

The preconditions are achievable by any unprivileged actor:
- Deploy one cell with arbitrary (malformed) data: one-time on-chain cost (CKB capacity + fee).
- Possess one live cell to use as a reusable input.
- Establish a P2P connection to the target node (standard peer connection).

After setup, the marginal cost per spam transaction is near-zero (craft a transaction, vary one output byte, relay). The node has no mechanism to detect or throttle this pattern.

---

### Recommendation

Add `OutPointError::InvalidDepGroup` to the `is_malformed_tx()` match arm, parallel to `OverMaxDepExpansionLimit`:

```rust
Reject::Resolve(OutPointError::OverMaxDepExpansionLimit) => true,
Reject::Resolve(OutPointError::InvalidDepGroup(_)) => true,  // add this
``` [6](#0-5) 

A transaction referencing a live cell whose data is not a valid `OutPointVec` as a `DepGroup` dep is permanently invalid (as long as the cell is live) and is structurally indistinguishable from a malformed transaction from the node's perspective. Peers that relay such transactions should be banned.

---

### Proof of Concept

1. Deploy a cell on-chain whose `outputs_data` is 20 bytes of random data (not a valid `OutPointVec`). Record its `OutPoint` as `BAD_OP`.
2. Establish a P2P connection to the target node.
3. In a loop, craft transaction `T_i` with:
   - One live input cell (reused across all iterations).
   - One `CellDep` with `out_point = BAD_OP`, `dep_type = DepGroup`.
   - One output with `data = [i as u8]` (varies the tx hash each iteration).
4. Relay `T_i` to the target node via the `SendTransactions` P2P message.
5. Assert: after 1 000 iterations, the peer connection is still open and the node has not issued a ban.

Each iteration triggers `parse_dep_group_data` failure → `OutPointError::InvalidDepGroup` → `Reject::Resolve(InvalidDepGroup)` → `is_malformed_tx() == false` → no `ban_malformed` call. [7](#0-6) [8](#0-7)

    async fn ban_malformed(&self, peer: PeerIndex, reason: String) {
        const DEFAULT_BAN_TIME: Duration = Duration::from_secs(3600 * 24 * 3);

        #[cfg(feature = "with_sentry")]
        use sentry::{Level, capture_message, with_scope};

        #[cfg(feature = "with_sentry")]
        with_scope(
            |scope| scope.set_fingerprint(Some(&["ckb-tx-pool", "receive-invalid-remote-tx"])),
            || {
                capture_message(
                    &format!(
                        "Ban peer {} for {} seconds, reason: \
                        {}",
                        peer,
                        DEFAULT_BAN_TIME.as_secs(),
                        reason
                    ),
                    Level::Info,
                )
            },
        );
        self.network.ban_peer(peer, DEFAULT_BAN_TIME, reason);
        self.verify_queue.write().await.remove_txs_by_peer(&peer);
```

        match OutPointVec::from_slice(slice) {
            Ok(v) => {
                if v.is_empty() {
                    Err("dep group is empty".to_owned())
                } else {
                    Ok(v)
                }
            }
            Err(err) => Err(err.to_string()),
        }
    }
