### Title
Unbounded `while` Loop in `executeCallback` Causes Unbounded Gas Consumption, Permanently Locking Funds - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that linearly scans through all fulfilled (inactive) sequence numbers to advance `_state.firstUnfulfilledSeq`. An unprivileged actor can create many requests and fulfill them out of order, causing the final fulfillment of the earliest sequence number to iterate through an arbitrarily large number of storage slots, consuming unbounded gas and potentially reverting, permanently locking the fee paid for that request.

---

### Finding Description

In `Echo.executeCallback`, after a successful callback, the contract advances `_state.firstUnfulfilledSeq` with the following loop:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
// After successful callback, update firstUnfulfilledSeq if needed
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest()`, which performs a storage lookup:

```solidity
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];
    }
}
``` [2](#0-1) 

The loop has no upper bound on the number of iterations — it runs until it finds the next active request or exhausts all sequence numbers. The developer's own TODO comment acknowledges this: *"I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. a better solution would be a doubly-linked list of active requests."* [3](#0-2) 

The `requestPriceUpdatesWithCallback` function is callable by any user, and `executeCallback` is callable by any provider (or anyone after the exclusivity period): [4](#0-3) 

The gas benchmark test explicitly documents the worst-case out-of-order fulfillment scenario:

> *"The last fulfillment will be the most expensive since it needs to linearly scan through all the fulfilled requests in storage in order to update `_state.lastUnfulfilledReq`"* [5](#0-4) 

---

### Impact Explanation

When `executeCallback` is called for sequence number `S` (the current `firstUnfulfilledSeq`), and all sequence numbers `S+1` through `S+N` have already been fulfilled, the while loop must iterate `N` times — each time performing a cold storage read via `findRequest`. With a sufficiently large `N`, the gas cost exceeds the block gas limit, causing the transaction to revert. The fee paid by the requester for sequence `S` is permanently locked in the contract, since `executeCallback` for that sequence number can never succeed.

---

### Likelihood Explanation

Any unprivileged user can call `requestPriceUpdatesWithCallback` to create requests. After the exclusivity period, any address can call `executeCallback`. An attacker can:

1. Create `N` requests (sequence numbers `S`, `S+1`, ..., `S+N`).
2. Fulfill requests `S+1` through `S+N` out of order (advancing `currentSequenceNumber` but leaving `firstUnfulfilledSeq = S`).
3. When request `S` is finally fulfilled, the while loop iterates `N` times through cold storage.

The cost scales linearly with `N`. With ~2,100 gas per cold `SLOAD` on EVM, approximately 14,000 out-of-order fulfillments would exhaust a 30M gas block limit. This is a realistic scenario for a live contract with active usage, and the attack requires no special privileges.

---

### Recommendation

Replace the linear scan with a bounded or O(1) mechanism:

1. **Cap the scan**: Add a maximum iteration count to the while loop (e.g., `MAX_SCAN = 256`) so `executeCallback` never iterates more than a fixed number of times.
2. **Doubly-linked list**: As the developer's own TODO suggests, maintain a doubly-linked list of active requests so that `firstUnfulfilledSeq` can be updated in O(1) when a request is cleared.
3. **Lazy update**: Remove the `firstUnfulfilledSeq` advancement from `executeCallback` entirely and compute it lazily or off-chain.

---

### Proof of Concept

1. Deploy `Echo` with a registered provider.
2. Create `N = 15,000` requests via `requestPriceUpdatesWithCallback` (sequence numbers 1 through 15,000).
3. Call `executeCallback` for sequence numbers 2 through 15,000 (out of order, skipping sequence 1). Each call succeeds cheaply since `firstUnfulfilledSeq = 1` and the loop does not advance.
4. Call `executeCallback` for sequence number 1. The while loop now iterates 14,999 times, each performing a cold storage read via `findRequest`. The transaction reverts with out-of-gas.
5. Sequence number 1's fee is permanently locked in the contract; no future call to `executeCallback` for it can succeed. [6](#0-5) [7](#0-6)
