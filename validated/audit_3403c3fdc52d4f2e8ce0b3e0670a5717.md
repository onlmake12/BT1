Audit Report

## Title
`n_sync_started` Counter Permanently Inflated via TOCTOU Race in `start_sync_headers` ‚Äî (`sync/src/synchronizer/mod.rs`, `sync/src/types/mod.rs`)

## Summary

In `start_sync_headers`, `n_sync_started` is atomically incremented via `fetch_update` before `peer_state.start_sync()` is called. If a peer disconnects between these two operations, `disconnected()` observes `sync_started() == false` and skips the decrement, leaving `n_sync_started` permanently inflated. In IBD mode, a counter value of 1 with zero actual syncing peers causes every subsequent call to `start_sync_headers` to break immediately, permanently preventing the node from advancing its chain tip until restarted.

## Finding Description

**Step 1 ‚Äî unconditional increment before peer state mutation:**

In `start_sync_headers`, the `fetch_update` at lines 672‚Äì682 increments `n_sync_started` subject only to the IBD guard (`if ibd && x != 0 { None }`), with no verification that the peer still exists in the map: [1](#0-0) 

**Step 2 ‚Äî `get_mut` can return `None` after the increment:**

Immediately after, `get_mut` is called on the `DashMap`. If `disconnected()` ran concurrently and already removed the peer, `get_mut` returns `None` and `start_sync()` is never called, leaving `headers_sync_state` as `SyncProtocolConnected` (not `Started`): [2](#0-1) 

**Step 3 ‚Äî `disconnected()` only decrements when `sync_started()` is true:**

`disconnected()` removes the peer from `state` and checks `sync_started()`. Because `start_sync()` was never called, `headers_sync_state` is still `SyncProtocolConnected`, `started()` returns `false`, and the decrement is skipped: [3](#0-2) 

**`sync_started()` requires `HeadersSyncState::Started` exactly:** [4](#0-3) [5](#0-4) 

**`start_sync()` is the only path that transitions to `Started`:** [6](#0-5) [7](#0-6) 

**`n_sync_started` is a plain `AtomicUsize` with no rollback path:** [8](#0-7) 

The `DashMap` provides per-shard locking, not a global lock covering both the `fetch_update` and the `get_mut`. There is no rollback of the increment anywhere in `start_sync_headers` when `get_mut` returns `None`.

## Impact Explanation

In IBD mode, `if ibd && x != 0 { None }` causes `fetch_update` to return `Err` whenever `n_sync_started >= 1`. Once the counter is inflated to 1 with zero actual syncing peers, every subsequent iteration of the peer loop immediately breaks. The node cannot initiate header sync with any peer. Since exiting IBD requires completing header sync, the node is permanently stuck in IBD ‚Äî it cannot advance its chain tip until restarted. This matches **High: Vulnerabilities which could easily crash a CKB node**, as a node permanently unable to participate in chain synchronization is functionally equivalent to a crashed node from a network standpoint.

## Likelihood Explanation

The race window is between two consecutive operations in the same function: the `fetch_update` and the `get_mut`. Peer disconnection events are processed by the network layer on separate threads concurrently with the sync loop. Any inbound or outbound peer that connects, passes the `can_start_sync` filter, and then immediately drops the TCP connection can trigger this ‚Äî no special privileges are required. With repeated connection/disconnection attempts, the probability of hitting the window accumulates. Once triggered, the impact is persistent until node restart, making even a low per-attempt probability operationally significant.

## Recommendation

Restructure `start_sync_headers` so the increment and the `start_sync()` call are atomic with respect to peer removal. Acquire the peer state entry first, verify the peer exists, and only then increment `n_sync_started`:

```rust
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    if self.shared().state().n_sync_started()
        .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
            if ibd && x != 0 { None } else { Some(x + 1) }
        })
        .is_err()
    {
        break;
    }
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
} else {
    continue; // peer gone, no increment needed
}
```

Alternatively, add a rollback path: if `get_mut` returns `None` after the increment, immediately call `fetch_sub(1, Ordering::AcqRel)` to restore the counter.

## Proof of Concept

1. Node A is in IBD mode (`is_initial_block_download() == true`).
2. Remote peer P connects and completes the sync handshake; P appears in `peers().state` with `can_start_sync() == true`.
3. `start_sync_headers` runs its peer loop for P:
   - `n_sync_started.fetch_update(...)` succeeds ‚Üí `n_sync_started = 1`.
   - Concurrently, P drops the TCP connection; `disconnected(P)` runs on the network thread, removes P from `state`, observes `sync_started() == false`, does **not** decrement `n_sync_started`.
   - `peers().state.get_mut(&P)` returns `None`; `start_sync()` is never called.
4. `n_sync_started == 1` with zero actual syncing peers.
5. Every subsequent call to `start_sync_headers` hits `if ibd && x != 0 { None }` ‚Üí `fetch_update` returns `Err` ‚Üí `break` ‚Üí no peer ever starts syncing.
6. Node A is permanently stuck in IBD until restarted.

A targeted test can reproduce this by spawning a thread that calls `peers.disconnected(peer)` between the `fetch_update` and `get_mut` calls (using a condvar or sleep to widen the window), then asserting `n_sync_started.load() == 0` after the race ‚Äî the assertion will fail, confirming the inflation.
