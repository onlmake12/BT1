Audit Report

## Title
VerifyQueue Lacks Per-Peer Fairness Scheduling, Enabling Single-Peer DoS of Transaction Verification Pipeline — (File: `tx-pool/src/component/verify_queue.rs`)

## Summary

`VerifyQueue` maintains only a single global `total_tx_size` counter with no per-peer byte accounting. A single P2P peer can flood the queue via the relay protocol at up to 30 MiB/s, saturating the 256 MB cap in under 10 seconds and causing every subsequent `add_tx()` call — from any peer or local RPC user — to return `Reject::Full`. The FIFO dequeue order means legitimate transactions that did enter the queue are processed only after all attacker transactions ahead of them complete CKB-VM verification.

## Finding Description

**Root cause — global-only cap, no per-peer accounting.**

`VerifyQueue` tracks only a single `total_tx_size: usize` field with no `HashMap<PeerIndex, usize>` or equivalent per-peer counter. [1](#0-0) 

`is_full()` compares only against the global remaining budget: [2](#0-1) 

`add_tx()` rejects only when the global total would be exceeded; there is no per-peer byte counter anywhere in the function: [3](#0-2) 

**FIFO dequeue — no fairness.**

`peek()` iterates via `iter_by_added_time()` — pure wall-clock insertion order — with no per-peer interleaving: [4](#0-3) 

**Relay injection path.**

The relay rate limiter allows 30 messages/s per `(PeerIndex, message_type)` key, and each `RelayTransactions` message may carry up to `MAX_RELAY_TXS_BYTES_PER_BATCH = 1 MiB`: [5](#0-4) 

`TransactionsProcess::execute()` spawns an async task that calls `submit_remote_tx()` for every accepted transaction with no per-peer queue-occupancy check: [6](#0-5) 

`submit_remote_tx()` calls `resumeble_process_tx_and_notify_full_reject()`: [7](#0-6) 

Which calls `resumeble_process_tx()` → `enqueue_verify_queue()` → `add_tx()` on the shared `VerifyQueue`: [8](#0-7) 

**Existing guards are insufficient.** The only pre-enqueue check is `non_contextual_verify()`, which performs structural validation only — no script execution. Transactions with invalid scripts still occupy queue space and worker time until they fail inside the worker. The `remove_txs_by_peer()` helper exists but is only invoked reactively (e.g., on peer disconnect), not proactively during admission: [9](#0-8) 

The `unknown_tx_hashes` filter in `TransactionsProcess::execute()` (requiring the node to have previously requested the hash from that peer) provides a minor friction but does not prevent the attack — the attacker controls the announcement flow via `RelayTransactionHashes` and can ensure all hashes are requested before sending. [10](#0-9) 

## Impact Explanation

Once the 256 MB queue is saturated by one peer, every subsequent `add_tx()` call returns `Reject::Full`, surfaced to RPC callers as `PoolIsFull (-1106)`. Legitimate transactions from all other peers and local RPC users are rejected outright. Transactions that did enter the queue before saturation are processed only after all earlier attacker transactions complete CKB-VM verification (up to `max_tx_verify_cycles = 70,000,000` cycles each), monopolizing the fixed `max_tx_verify_workers` thread pool.

This matches the allowed bounty impact: **High (10001–15000 points) — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** The entire transaction relay pipeline of a targeted node is rendered non-functional for the duration of the attack with no on-chain cost to the attacker.

## Likelihood Explanation

- **No privilege required**: Any peer completing the standard P2P handshake can send `RelayTransactionHashes` → receive `GetRelayTransactions` → reply with `RelayTransactions`.
- **Low cost**: Transactions need only pass `non_contextual_verify()` (structural checks, no script execution) to be enqueued. Invalid scripts still consume queue space and worker cycles.
- **Rate limiter is insufficient**: 30 `RelayTransactions` messages/s × 1 MiB/message = 30 MiB/s injection rate fills the 256 MB queue in approximately 8–9 seconds.
- **Repeatable**: The attacker can maintain the connection and continuously re-fill the queue as workers drain it, sustaining the DoS indefinitely.

## Recommendation

**Short term**: Track per-peer byte occupancy inside `VerifyQueue`. Add a `HashMap<PeerIndex, usize>` field alongside `total_tx_size`. In `add_tx()`, reject a new entry if the submitting peer already accounts for more than `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE / N` bytes (where N is a configurable peer cap). Update the per-peer counter in both `add_tx()` and `remove_tx()`.

**Long term**: Replace the FIFO dequeue in `peek()` with a weighted round-robin or fair-queuing discipline over peer buckets, ensuring no single peer can delay transactions from all other peers regardless of submission rate.

## Proof of Concept

1. Connect a custom peer to a CKB node via the RelayV3 protocol and complete the P2P handshake.
2. Generate N structurally valid transactions (passing `non_contextual_verify`) with unique hashes, e.g., by varying cell deps or outputs. Scripts may be invalid — they will still be enqueued.
3. Send `RelayTransactionHashes` messages (up to 32,767 hashes each, 30/s) to announce all N hashes.
4. When the node replies with `GetRelayTransactions`, respond with `RelayTransactions` messages (up to 1 MiB each, 30/s).
5. `TransactionsProcess::execute()` calls `submit_remote_tx()` for each transaction, which calls `VerifyQueue::add_tx()`.
6. After approximately 8–9 seconds, `VerifyQueue::total_tx_size` reaches 256,000,000 bytes; `is_full()` returns `true`.
7. Any subsequent `send_transaction` RPC call from a legitimate user returns `PoolIsFull (-1106)`.
8. Confirm by observing that `add_tx()` returns `Err(Reject::Full(...))` for all new submissions while the attacker's connection remains open.

    pub fn peek(&self, only_small_cycle: bool) -> Option<ProposalShortId> {
        let mut iter = self.inner.iter_by_added_time();

        if let Some(proposal_entry) = iter.find(|e| e.is_proposal_tx) {
            return Some(proposal_entry.inner.tx.proposal_short_id());

        let entry = if only_small_cycle {
            self.inner.iter_by_added_time().find(|e| !e.is_large_cycle)

            self.inner.iter_by_added_time().next()
        };

        entry.map(|e| e.inner.tx.proposal_short_id())
    }

        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
