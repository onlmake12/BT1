### Title
Insufficient Minimum Fee Rate Check Due to Integer Division Precision Loss in `FeeRate::fee()` - (File: `util/types/src/core/fee_rate.rs`)

---

### Summary

`FeeRate::fee()` computes the minimum required fee using integer (floor) division. This underestimates the true minimum fee, allowing a transaction submitter to pass the `check_tx_fee` gate with a fee rate strictly below the configured `min_fee_rate`. The same rounding flaw affects the RBF extra-fee requirement in `calculate_min_replace_fee`.

---

### Finding Description

`FeeRate::fee()` is defined as:

```rust
// util/types/src/core/fee_rate.rs, line 34-37
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
```

This computes `⌊fee_rate × weight / 1000⌋`, which rounds down whenever `fee_rate × weight` is not a multiple of 1000.

The result is used as the enforcement threshold in `check_tx_fee`:

```rust
// tx-pool/src/util.rs, lines 45-52
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    let reject = Reject::LowFeeRate(...);
    return Err(reject);
}
```

Because `min_fee` is rounded down, the condition `fee < min_fee` is weaker than the intended `fee < ⌈fee_rate × weight / 1000⌉`. A transaction whose fee equals `⌊fee_rate × weight / 1000⌋` passes the check even though its effective fee rate is strictly less than `min_fee_rate`.

The same `FeeRate::fee()` call appears in the RBF path:

```rust
// tx-pool/src/pool.rs, line 103
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
```

So the extra fee required to replace a transaction is also underestimated by up to 1 shannon.

---

### Impact Explanation

An unprivileged tx-pool submitter can craft a transaction whose size `s` satisfies `min_fee_rate × s mod 1000 ≠ 0`, then submit it with fee `= ⌊min_fee_rate × s / 1000⌋`. The node accepts it despite its effective fee rate being below `min_fee_rate`. The maximum underestimation per transaction is `(KW − 1) / KW = 999/1000`, i.e., at most 1 shannon below the true threshold. For RBF, the same rounding allows a replacement transaction to pay 1 shannon less than the required extra fee.

While the per-transaction monetary gap is tiny (≤ 1 shannon ≈ 10⁻⁸ CKB), the check is the primary admission gate for the tx pool. Transactions that should be rejected as below-minimum-fee-rate are silently accepted, undermining the operator-configured fee floor and the RBF bump requirement.

---

### Likelihood Explanation

Any transaction sender can trigger this with no special privileges. The condition is met whenever `min_fee_rate × tx_size` is not divisible by 1000, which is the common case for arbitrary transaction sizes and non-round fee rates. The default `min_fee_rate` is 1000 shannons/KW (exactly `KW`), so with the default setting the rounding is zero; however, any non-default or non-round fee rate (e.g., 1001, 999, 1500 shannons/KW) triggers the underestimation for most transaction sizes.

---

### Recommendation

Replace the division-based threshold with a cross-multiplication comparison to eliminate rounding entirely, mirroring the pattern already used in `AncestorsScoreSortKey::min_fee_and_weight()` and `Ord for AncestorsScoreSortKey`:

```rust
// tx-pool/src/util.rs
// Instead of:
//   let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
//   if fee < min_fee { ... }
// Use:
if (fee.as_u64() as u128) * (KW as u128)
    < (tx_pool.config.min_fee_rate.as_u64() as u128) * (tx_size as u128)
{
    // reject
}
```

Alternatively, change `FeeRate::fee()` to use ceiling division:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = (self.0.saturating_mul(weight) + KW - 1) / KW;
    Capacity::shannons(fee)
}
```

Apply the same fix to the `extra_rbf_fee` calculation in `calculate_min_replace_fee`.

---

### Proof of Concept

**Parameters:**
- `min_fee_rate = 1001` shannons/KW (a non-default but valid operator setting)
- `tx_size = 999` bytes

**Current behaviour:**
```
min_fee = ⌊1001 × 999 / 1000⌋ = ⌊999.999⌋ = 999 shannons
```
A transaction with `fee = 999` satisfies `fee < min_fee` → `999 < 999` → **false** → accepted.

**Effective fee rate of accepted transaction:**
```
999 × 1000 / 999 = 1000 shannons/KW  <  1001 shannons/KW (the configured minimum)
```

The transaction is admitted with a fee rate 1 shannons/KW below the operator-configured floor.

**True check (cross-multiplication):**
```
999 × 1000 = 999 000  <  1001 × 999 = 999 999  → true → correctly rejected
