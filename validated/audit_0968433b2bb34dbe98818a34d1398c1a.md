### Title
Missing Per-Peer Rate Limiter in BlockFilter Protocol Handler Enables RocksDB I/O Saturation DoS — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

### Summary
The `BlockFilter` P2P protocol handler processes `GetBlockFilters`, `GetBlockFilterHashes`, and `GetBlockFilterCheckPoints` messages from any remote peer with no per-peer rate limiting. Each message triggers up to 2000 RocksDB point-reads. An unprivileged attacker can flood the handler at TCP line rate, saturating the node's RocksDB I/O and starving block validation and sync threads.

### Finding Description
`BlockFilter::received` in `sync/src/filter/mod.rs` (lines 122–160) dispatches every inbound message directly to `try_process` without any rate check. The `BlockFilter` struct carries no `rate_limiter` field. By contrast, `Relayer::try_process` (`sync/src/relayer/mod.rs`, lines 116–123) gates every non-PoW message through a `governor`-based `RateLimiter<(PeerIndex, u32)>` at 30 req/s per (peer, message-type) key before doing any work.

`GetBlockFiltersProcess::execute` (lines 38–72 of `get_block_filters_process.rs`) enters a loop of up to `BATCH_SIZE = 1000` iterations, calling `active_chain.get_block_hash(block_number)` and `active_chain.get_block_filter(&block_hash)` on each iteration — both are synchronous RocksDB reads via `ChainDB`. The guard condition `if latest >= start_number` is trivially satisfied with `start_number=0` because `get_latest_built_filter_block_number` returns `unwrap_or_default()` = 0 when no filter data has been built yet, making `0 >= 0` true.

`GetBlockFilterHashes` (BATCH_SIZE=2000) and `GetBlockFilterCheckPoints` (BATCH_SIZE=2000) are identically unguarded.

### Impact Explanation
A single attacker peer sending `GetBlockFilters(start_number=0)` at maximum TCP rate forces up to 2000 RocksDB reads per message. With N concurrent attacker connections (each below the general peer limit), the aggregate read IOPS scales as `N × 2000 × msg_rate`. RocksDB uses a shared thread pool and block cache; I/O saturation delays `get_block_hash`/`get_block_filter` calls from block validation and sync threads, degrading block processing latency proportionally to N. The node remains reachable but its chain-tip advancement stalls.

### Likelihood Explanation
The Filter protocol is enabled by default on mainnet nodes that have built filter data. Any peer that can establish a TCP connection can send `GetBlockFilters` messages — no authentication, no PoW, no stake. The attack message is minimal (8 bytes for `start_number`). The attacker can open multiple connections up to the node's peer limit and send at line rate. No special knowledge of the chain state is required.

### Recommendation
Add a `governor`-based `RateLimiter<(PeerIndex, u32)>` to the `BlockFilter` struct, mirroring the pattern in `Relayer::new` (`sync/src/relayer/mod.rs`, lines 88–98). Check the limiter at the top of `BlockFilter::try_process` before dispatching to any `*Process::execute`, and return `StatusCode::TooManyRequests` (without banning) on excess. A quota of 30 req/s per (peer, message-type) is consistent with the existing Relayer policy. Call `rate_limiter.retain_recent()` in `BlockFilter::disconnected` to bound memory growth.

### Proof of Concept
```
1. Start a CKB node with block filters enabled (filter data built for at least block 0).
2. Open N TCP connections to the node's Filter protocol port.
3. On each connection, send GetBlockFilters { start_number: 0 } in a tight loop.
4. Monitor RocksDB read IOPS (via metrics or perf) and block processing latency.
5. Assert: without a rate limiter, block processing latency degrades linearly with N;
   with a 30 req/s rate limiter per peer, latency remains bounded.
```
