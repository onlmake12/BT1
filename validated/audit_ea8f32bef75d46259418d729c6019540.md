### Title
Advertised `DISCOVERY` Capability Flag Not Cleared When Discovery Protocol Is Disabled - (`util/launcher/src/lib.rs`)

### Summary

CKB nodes advertise their capabilities to peers via the Identify protocol using a `Flags` bitfield. When certain protocols are disabled via configuration, the corresponding flags are removed before broadcasting. However, when the `Discovery` protocol is disabled, the `Flags::DISCOVERY` bit is **never removed** from the advertised flags. This causes a node to falsely claim it can serve peer-discovery requests, misleading connecting peers and degrading network topology.

### Finding Description

In `util/launcher/src/lib.rs`, the node initializes its capability flags with `Flags::all()` and then conditionally removes flags for disabled protocols:

```rust
// util/launcher/src/lib.rs:429
let mut flags = Flags::all();

if support_protocols.contains(&SupportProtocol::Relay) { ... }
else { flags.remove(Flags::RELAY); }          // correctly cleared

if support_protocols.contains(&SupportProtocol::Filter) { ... }
else { flags.remove(Flags::BLOCK_FILTER); }   // correctly cleared

if support_protocols.contains(&SupportProtocol::LightClient) { ... }
else { flags.remove(Flags::LIGHT_CLIENT); }   // correctly cleared
``` [1](#0-0) 

However, the `Discovery` protocol is conditionally registered in `NetworkService::new()`:

```rust
// network/src/network.rs:897-914
if config.support_protocols.contains(&SupportProtocol::Discovery) {
    // ... register DiscoveryProtocol
    protocol_metas.push(disc_meta);
}
// No corresponding: else { flags.remove(Flags::DISCOVERY); }
``` [2](#0-1) 

The `Flags::DISCOVERY` bit is defined as:

```rust
/// Discovery protocol, which can provide peers data service
const DISCOVERY = 0b10;
``` [3](#0-2) 

These flags are encoded into the Identify message and broadcast to every connecting peer:

```rust
// network/src/protocols/identify/mod.rs:525-534
fn new(name: String, flags: Flags, client_version: String) -> Self {
    Identify {
        encode_data: packed::Identify::new_builder()
            .name(name.as_str())
            .flag(flags.bits())
            ...
``` [4](#0-3) 

When a remote peer receives this Identify message, it stores the flags and uses them to decide which protocols to open and which addresses to propagate:

```rust
// network/src/protocols/identify/mod.rs:434-443
} else if required_flags_filter(required_flags, flags) {
    // The remote end can support all local protocols.
    let _ = context.open_protocols(
        context.session.id,
        TargetProtocol::Filter(Box::new(move |id| {
            id != &SupportProtocols::Feeler.protocol_id()
        })),
    ).await;
``` [5](#0-4) 

The peer store also propagates addresses tagged with the falsely-advertised `DISCOVERY` flag to the rest of the network:

```rust
// network/src/protocols/identify/mod.rs:421-423
self.network_state.with_peer_store_mut(|peer_store| {
    peer_store.add_outbound_addr(context.session.address.clone(), flags);
});
``` [6](#0-5) 

The `required_flags_filter` function used by the peer store to select peers for outbound connections also trusts these flags:

```rust
// network/src/peer_store/peer_store_impl.rs:407-412
pub(crate) fn required_flags_filter(required: Flags, t: Flags) -> bool {
    if required == Flags::RELAY | Flags::DISCOVERY | Flags::SYNC {
        t.contains(required) || t.contains(Flags::COMPATIBILITY)
    } else {
        t.contains(required)
    }
}
``` [7](#0-6) 

### Impact Explanation

A node with `Discovery` removed from `support_protocols` will:

1. **Falsely advertise `DISCOVERY` capability** to every peer via the Identify protocol.
2. **Cause connecting peers to attempt opening the Discovery protocol** with it — a protocol that is not registered and will be rejected at the transport layer.
3. **Pollute the peer store** of connecting peers: the node's address is stored with the `DISCOVERY` flag set and then re-broadcast to the wider network via the Discovery protocol's `GetNodes`/`Nodes` exchange. Other nodes will preferentially select this address when looking for discovery peers, wasting connection slots.
4. **Degrade network topology**: nodes seeking peer addresses via Discovery will fail to get them from this node despite its advertisement, slowing peer discovery for the network.

### Likelihood Explanation

The `support_protocols` list is a documented, user-facing configuration option. The default config explicitly lists all protocols, but the comment in `ckb.toml` states "only `Sync` and `Identify` are mandatory, others are optional," directly inviting operators to remove `Discovery`. [8](#0-7) 

Any operator running a specialized node (e.g., a relay-only node, a light-client server, or a private network node) who removes `Discovery` from `support_protocols` will silently trigger this mismatch. No attacker capability is required — the bug is triggered by ordinary, documented configuration.

### Recommendation

In `util/launcher/src/lib.rs`, add a flag-removal branch for `Discovery` analogous to the existing branches for `Relay`, `Filter`, and `LightClient`:

```rust
if support_protocols.contains(&SupportProtocol::Discovery) {
    // ... register Discovery in NetworkService::new (already done there)
} else {
    flags.remove(Flags::DISCOVERY);  // <-- add this
}
``` [9](#0-8) 

Alternatively, move the Discovery protocol registration into the launcher (alongside Relay, Filter, LightClient) so that the flag and the protocol registration are always co-located and cannot diverge.

### Proof of Concept

1. Configure a CKB node with `support_protocols = ["Ping", "Identify", "Feeler", "DisconnectMessage", "Sync", "Relay", "Time", "Alert"]` (Discovery omitted).
2. Start the node. The `Flags::all()` initialization at `util/launcher/src/lib.rs:429` sets `DISCOVERY = 0b10`.
3. No `flags.remove(Flags::DISCOVERY)` is executed because there is no such branch.
4. The flags (including `DISCOVERY`) are passed to `NetworkService::new` and encoded into the Identify message.
5. Connect a second node. It receives the Identify message, sees `DISCOVERY` set, stores the first node's address with `DISCOVERY` in its peer store, and attempts to open the Discovery protocol.
6. The Discovery protocol is not registered on the first node; the open attempt fails silently.
7. The second node's peer store now contains the first node's address tagged `DISCOVERY`, and will re-advertise it to other peers as a discovery-capable node.
