### Title
Unvalidated `providerToCredit` in `Echo.executeCallback` Allows Permanent Fee Locking — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` accepts a caller-supplied `providerToCredit` address and unconditionally credits it with the request's provider fee. There is no check that `providerToCredit` is a registered provider. After the exclusivity period expires, any unprivileged actor can call `executeCallback` with an arbitrary unregistered address as `providerToCredit`, permanently locking the provider fee inside the contract and depriving the legitimate provider of their earned fee.

---

### Finding Description

`Echo.executeCallback` is a permissionless function. Its only caller restriction is a time-gated exclusivity check:

```solidity
// Echo.sol lines 113–121
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

Once the exclusivity window closes, `providerToCredit` is completely unconstrained. The function then unconditionally credits that address:

```solidity
// Echo.sol lines 161–162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

There is no check that `providerToCredit` is a registered provider (i.e., `_state.providers[providerToCredit].isRegistered == true`). If `providerToCredit` is an unregistered address, its `feeManager` field is `address(0)`. The only withdrawal path for provider fees is `withdrawAsFeeManager`, which requires `msg.sender == _state.providers[provider].feeManager`. Since `address(0)` can never sign a transaction, the credited fees are permanently irrecoverable.

Additionally, `clearRequest(sequenceNumber)` is called unconditionally before the callback, so the legitimate provider can never re-fulfill the same request.

---

### Impact Explanation

An attacker who calls `executeCallback` after the exclusivity period with `providerToCredit = address(0xdead)` (or any unregistered address):

1. Permanently locks the entire provider fee (`req.fee`) inside the contract.
2. Prevents the legitimate provider from ever recovering that fee.
3. The consumer's callback still executes, so the consumer is unaffected — making the attack silent from the consumer's perspective.

Repeated across many requests, this constitutes a systematic griefing attack that drains provider revenue without any financial gain to the attacker (pure destruction). Alternatively, an attacker who first registers as a provider can pass their own address as `providerToCredit` after exclusivity expires, redirecting the legitimate provider's fee to themselves.

---

### Likelihood Explanation

- Valid price update data (`updateData`) is freely and publicly available from the Pyth Hermes API — no privileged access is required.
- The attacker only needs to pay gas.
- The exclusivity period is a finite, configurable window; any request that the legitimate provider fails to fulfill within that window becomes exploitable.
- No special role, leaked key, or governance majority is required.

---

### Recommendation

Add a registration check before crediting `providerToCredit`:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit is not a registered provider"
);
```

Optionally, also enforce `msg.sender == providerToCredit` to ensure only the fulfilling provider can claim the fee, preventing fee redirection between providers.

---

### Proof of Concept

1. Alice (legitimate provider) is assigned to fulfill request `#42` with `publishTime = T`.
2. The exclusivity period is 15 seconds; Alice fails to call `executeCallback` before `T + 15`.
3. Attacker fetches valid `updateData` for the requested `priceIds` from Hermes.
4. Attacker calls:
   ```solidity
   echo.executeCallback(
       address(0xdead),   // unregistered, feeManager == address(0)
       42,
       validUpdateData,
       priceIds
   );
   ```
5. `_state.providers[address(0xdead)].accruedFeesInWei` is incremented by `req.fee`.
6. `clearRequest(42)` removes the request — Alice can never fulfill it.
7. `address(0xdead)` has no fee manager; `withdrawAsFeeManager` will always revert for it.
8. Alice's fee is permanently locked. The consumer's `_echoCallback` fires normally.

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
    }

    function registerProvider(
        uint96 baseFeeInWei,
        uint96 feePerFeedInWei,
        uint96 feePerGasInWei
    ) external override {
        ProviderInfo storage provider = _state.providers[msg.sender];
        require(!provider.isRegistered, "Provider already registered");
        provider.baseFeeInWei = baseFeeInWei;
        provider.feePerFeedInWei = feePerFeedInWei;
        provider.feePerGasInWei = feePerGasInWei;
        provider.isRegistered = true;
        emit ProviderRegistered(msg.sender, feePerGasInWei);
    }
