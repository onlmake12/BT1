Audit Report

## Title
Off-by-One in Spawn VM Count Guard Allows Creating 17 VMs When Limit Is 16 - (File: script/src/scheduler.rs)

## Summary
`Scheduler::process_message_box` in `script/src/scheduler.rs` uses `>` instead of `>=` when checking the alive VM count against `MAX_VMS_COUNT` (16) before permitting a `spawn` syscall. When exactly 16 VMs are alive, `16 > 16` evaluates to `false`, causing `boot_vm` to be called and a 17th VM to be created, silently violating the protocol-defined resource limit.

## Finding Description
`MAX_VMS_COUNT` is defined as `16` at line 34: [1](#0-0) 

The `Message::Spawn` handler at line 532 guards with a strict `>`: [2](#0-1) 

When `suspended.len() + instantiated.len() == 16`, the condition `16 > 16` is `false`, so execution falls through to `boot_vm` at line 539: [3](#0-2) 

`boot_vm` performs no count check of its own; it unconditionally inserts into `self.instantiated` and `self.states`: [4](#0-3) 

The analogous `MAX_FDS` check at line 595 correctly uses `>=`, confirming the inconsistency is a defect in the spawn path: [5](#0-4) 

## Impact Explanation
This is an incorrect implementation of the CKB-VM spawn syscall resource limit, matching the allowed bounty impact: **"Incorrect implementation or behavior of CKB-VM or system scripts" — High (10001–15000 points)**. The protocol invariant `alive_vms ≤ MAX_VMS_COUNT` is universally unenforced at the exact boundary value. Because all nodes run the same code, every node will accept a transaction whose script creates 17 concurrent VMs, meaning the resource cap is effectively raised by one for any transaction that reaches the boundary.

## Likelihood Explanation
Any unprivileged transaction sender can trigger this. A script that spawns children until `suspended.len() + instantiated.len() == 16` (root VM + 15 spawned children all still alive) and then issues one additional `ckb_spawn` call will receive `SUCCESS` instead of `MAX_VMS_SPAWNED`. No special privileges, leaked keys, or victim interaction are required. The entry path is a standard lock or type script submission.

## Recommendation
Change the comparison operator from `>` to `>=` at line 532:

```diff
- if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
+ if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

This makes the `MAX_VMS_COUNT` guard consistent with the `MAX_FDS` guard at line 595 and correctly rejects a spawn when the alive VM count is already at the protocol limit.

## Proof of Concept
1. Write a CKB lock or type script that uses `ckb_spawn` to create children in a chain, with each child waiting (keeping itself alive), until `suspended.len() + instantiated.len() == 16`.
2. From any alive VM, issue one additional `ckb_spawn` call.
3. **Current behavior (`>`):** `16 > 16` is `false` → spawn proceeds → `boot_vm` inserts VM 17 → syscall returns `SUCCESS`.
4. **Expected behavior (`>=`):** `16 >= 16` is `true` → spawn is rejected → syscall returns `MAX_VMS_SPAWNED`.

The existing test `check_spawn_max_vms_count` at line 103 delegates to test case `[10]` in `testdata/spawn_cases`: [6](#0-5) 

This test does not probe the exact boundary of 16 simultaneously alive VMs, leaving the off-by-one undetected. A targeted unit test that holds 16 VMs alive concurrently and asserts the next spawn returns `MAX_VMS_SPAWNED` would reproduce the failure under the current code.
