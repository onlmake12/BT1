Audit Report

## Title
Minimum Fee Rate Check Uses Only Serialized Size, Ignoring Cycle-Based Weight — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the transaction's serialized byte size, while the true transaction weight used for pool ordering and eviction is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. An unprivileged sender can craft a compact high-cycle transaction that passes the size-only fee gate while its weight-based effective fee rate is ~40× below the configured minimum, enabling cheap saturation of VM verification workers and pool capacity.

## Finding Description
In `tx-pool/src/util.rs` (L42–52), `check_tx_fee` computes the minimum required fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

The code comment explicitly acknowledges this as a known design gap. This check runs inside `pre_check` (`process.rs` L269–316) before VM execution; cycles are not yet known. After `pre_check` passes, `verify_rtx` runs and returns `verified.cycles`, which are stored in the `TxEntry` at `process.rs` L751:

```rust
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

From that point, all pool ordering and eviction uses the true weight via `entry.fee_rate()` (`entry.rs` L114–118):

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
```

where `get_transaction_weight` (`tx_pool.rs` L298–303) is `max(tx_size, cycles * 0.000_170_571_4)`.

With default config (`min_fee_rate = 1_000` shannons/KB, `max_tx_verify_cycles = 70_000_000`), a 300-byte transaction consuming 69,000,000 cycles has weight = `max(300, 11,769)` = 11,769. The size-only gate requires only 300 shannons, but the effective weight-based fee rate is `300 × 1,000 / 11,769` ≈ 25 shannons/KB — ~40× below the minimum. The transaction is admitted.

Pool capacity is tracked in `pool_map.total_tx_size` (serialized bytes, `pool_map.rs` L69), not weight. A 300-byte transaction contributes only 300 bytes to this counter, allowing ~600,000 such transactions to fill a 180 MB pool, each triggering full VM execution of ~70 M cycles.

## Impact Explanation
This matches the High impact class: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** Flooding the pool with 600,000 high-cycle transactions saturates VM verification worker threads and the verify queue, degrading transaction relay and block assembly across the network. The total fee cost is ~1.8 CKB (600,000 × 300 shannons), far below what the `min_fee_rate` is designed to impose for the actual computational burden.

## Likelihood Explanation
No privileged access is required. The attacker deploys a compact RISC-V loop script on-chain (one-time cost), then submits transactions referencing it via the standard `send_transaction` RPC or P2P relay. The attack is fully automatable and repeatable. The code comment at the root cause site confirms this is a known design gap, not an oversight.

## Recommendation
After `verify_rtx` returns `verified.cycles`, perform a second fee check using the true weight before admitting the entry to the pool:

```rust
// In _process_tx, after verify_rtx:
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee_by_weight, fee.as_u64())), snapshot));
}
```

Alternatively, apply a conservative upper-bound estimate at `pre_check` time using `max_tx_verify_cycles` as a stricter admission gate, then refine after verification.

## Proof of Concept
1. Compile a minimal RISC-V loop script executing ~69,000,000 cycles; binary is ~150–300 bytes.
2. Deploy the script cell on-chain.
3. Construct a transaction with this script as a lock; `tx_size ≈ 300 bytes`.
4. Set `fee = 300 shannons` (= `min_fee_rate × tx_size / 1000 = 1,000 × 300 / 1,000`).
5. Submit via `send_transaction` RPC.
6. `check_tx_fee` passes: `300 ≥ 300`.
7. After VM execution: `verified.cycles ≈ 69,000,000`; `weight = max(300, 11,769) = 11,769`; effective fee rate ≈ 25 shannons/KB — ~40× below the 1,000 shannons/KB minimum.
8. Transaction is admitted. Repeat in a loop to fill the pool and saturate verification workers.
