Audit Report

## Title
Unauthenticated `from` PeerId in Hole-Punching Protocol Enables Attacker-Directed Outbound TCP Connection Storms — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
An attacker with a single standard P2P connection can forge the `from` field in `ConnectionRequest` and `ConnectionSync` messages to inject arbitrary TCP addresses into the victim's `pending_delivered` map and trigger passive NAT traversal tasks that make repeated outbound TCP connections to attacker-controlled addresses. The `forward_rate_limiter` is trivially bypassed by rotating `from` PeerIds, and the only real constraint — the per-session `rate_limiter` at 30 req/sec — allows up to 720 outbound TCP connection attempts per second from the victim, with each spawned task looping for 30 seconds.

## Finding Description

**Root cause — no binding of `from` to the actual session sender.**

In `connection_request.rs`, when `self_peer_id == &content.to`, `respond_delivered()` is called with `content.from` taken directly from the message with no check that it equals the PeerId of the actual connected session: [1](#0-0) 

The deduplication guard inside `respond_delivered` is keyed by this attacker-supplied `from_peer_id`: [2](#0-1) 

A fresh random `from` PeerId is never present in `pending_delivered`, so the guard is never triggered. After filtering to TCP/IP addresses (which any attacker-controlled `/ip4/.../tcp/...` address passes), the forged `from_peer_id` and attacker addresses are inserted unconditionally: [3](#0-2) 

**Step 2 — `ConnectionSync` triggers NAT traversal using the poisoned entry.**

In `connection_sync.rs`, when `route` is empty and `self_peer_id == &content.to`, the handler looks up `pending_delivered` using `content.from` — again with no check that `content.from` matches the actual session's authenticated PeerId: [4](#0-3) 

For each stored address it creates tasks calling `try_nat_traversal`: [5](#0-4) 

These are spawned as a `runtime::spawn` task, conditioned only on the victim having a listen address configured (a normal condition for any running node): [6](#0-5) 

**`try_nat_traversal` loops for 30 seconds per spawned task.**

Each spawned task retries TCP connections every ~200ms for up to 30 seconds, meaning each task makes ~150 connection attempts to the attacker-controlled address: [7](#0-6) 

**Rate-limit bypass.**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` with a quota of 1 req/sec per key: [8](#0-7) 

A fresh random `from` PeerId creates a new key with a fresh bucket, providing zero protection. The only real constraint is the per-session `rate_limiter` keyed by `(session_id, msg.item_id())` at 30 req/sec: [9](#0-8) 

**Unbounded `pending_delivered` growth.**

`pending_delivered` is an unbounded `HashMap` with cleanup only in `notify` every 5 minutes, retaining entries newer than `TIMEOUT` (5 minutes): [10](#0-9) 

At 30 insertions/sec with unique `from` PeerIds, the map accumulates up to 9,000 entries per 5-minute window before any cleanup, consuming unbounded memory.

## Impact Explanation

Each `ConnectionRequest`/`ConnectionSync` pair causes the victim to spawn one async task that makes repeated TCP connection attempts (up to ~150 per task over 30 seconds) to up to `ADDRS_COUNT_LIMIT` (24) attacker-controlled addresses. At the per-session cap of 30 pairs/sec, a single attacker session drives up to **720 outbound TCP connection attempts per second** to arbitrary third-party targets, with the number of concurrent tasks growing to ~900 after 30 seconds. This exhausts the victim node's file descriptors, OS connection-tracking state, and network bandwidth, crashing the node or rendering it unable to maintain legitimate peer connections. The victim's IP is the apparent source of the flood, enabling third-party amplification abuse.

This matches: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node** and **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The attacker requires only a single standard, unprivileged P2P connection — the normal peer connection any node accepts. No key material, PoW, or special role is needed. The `HolePunching` protocol is enabled by default. The two-message sequence is trivially constructable from the public `packed::ConnectionRequest` / `packed::ConnectionSync` schema. The attack works on all platforms since `try_nat_traversal` creates a socket and connects even when `bind_addr` is `None`. The attack is fully repeatable and sustainable for the duration of the session.

## Recommendation

1. **Authenticate `from` in `respond_delivered`**: In `ConnectionRequestProcess`, resolve the PeerId for `self.peer` from the peer registry and reject any message where `content.from` does not match the actual session's authenticated PeerId.
2. **Authenticate `from` in `ConnectionSyncProcess`**: Before looking up `pending_delivered`, verify `content.from` against the actual session's authenticated PeerId.
3. **Cap `pending_delivered` size**: Enforce a maximum entry count (e.g., equal to `max_outbound`) to prevent memory exhaustion from forged `from` PeerIds even if authentication is added.
4. **Bound spawned NAT traversal tasks**: Track and limit the number of concurrently active `try_nat_traversal` tasks to prevent unbounded resource consumption.

## Proof of Concept

```rust
// Step 1: send ConnectionRequest to victim with forged `from` and attacker TCP addr
let forged_from = PeerId::random();
let attacker_addr: Multiaddr = "/ip4/1.2.3.4/tcp/9999".parse().unwrap();
let req = packed::ConnectionRequest::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())          // victim's own PeerId
    .listen_addrs(vec![attacker_addr.to_vec()].pack())
    .max_hops(6u8.pack())
    .route(vec![].pack())               // empty route → victim is `to`
    .build();
attacker_session.send(req);

// Step 2: send ConnectionSync with same forged `from`
let sync = packed::ConnectionSync::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())
    .route(vec![].pack())               // empty route → victim is `to`
    .build();
attacker_session.send(sync);

// Victim spawns runtime::spawn task calling try_nat_traversal(bind_addr, "/ip4/1.2.3.4/tcp/9999")
// try_nat_traversal loops for 30s making TCP connection attempts every ~200ms
// Repeat with fresh forged_from PeerIds at 30/sec → 720 TCP connection attempts/sec to 1.2.3.4:9999
// After 30s: ~900 concurrent tasks each making ~5 attempts/sec
```

An integration test can assert that after the two-message sequence, the victim node makes an outbound TCP connection attempt to `1.2.3.4:9999`, and that repeating with fresh `forged_from` values at 30/sec is not blocked by any existing guard.
