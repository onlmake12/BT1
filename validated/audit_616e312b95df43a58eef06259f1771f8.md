### Title
Copy-paste bug in `script_exists_in_output` causes incorrect script deletion on PostgreSQL during rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary
`script_exists_in_output` always reads `row_lock` in its final `match` expression instead of `row_type`, causing the type-script existence check to be silently skipped on PostgreSQL. Scripts referenced only as `type_script_id` are incorrectly deleted from the `script` table during any chain rollback.

### Finding Description
At line 252, the code reads:
```rust
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```
It should read `row_type.try_get::<bool, _>(0)`. Because `row_lock` was already evaluated as `false` before reaching this point (the early-return at lines 224–227 handles the `true` case), this expression always returns `Ok(false)` on PostgreSQL, regardless of what `row_type` contains. [3](#0-2) 

### Impact Explanation
On PostgreSQL-backed rich-indexer deployments, any chain reorganization that rolls back a block containing outputs with a type script (whose `script_id` is not shared with any surviving output's `lock_script_id`) will permanently delete that script row. Subsequent `get_cells`/`get_transactions` RPC calls for cells with that type script will return NULL or missing script fields, silently corrupting indexer state. The indexer cannot self-heal without a full re-sync.

### Likelihood Explanation
Chain reorganizations are a normal, unprivileged blockchain event — any valid competing chain of sufficient work triggers one. PostgreSQL is an explicitly supported and documented deployment option. The condition (output with type script, no lock script sharing the same `script_id`) is common in practice (e.g., UDT/NFT outputs).

### Recommendation
Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

### Proof of Concept
1. Start a CKB node with `--rich-indexer` backed by PostgreSQL.
2. Submit a block containing one transaction output with only a `type_script` (lock script has a distinct `script_id`).
3. Trigger a rollback of that block (e.g., via a competing chain of equal or greater work).
4. Query the `script` table: the `type_script` row is absent despite no surviving output referencing it as `lock_script_id` — the type-script check was never evaluated.
5. Call `get_cells` with that type script: returns empty or NULL script data.
