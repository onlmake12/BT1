### Title
Links Removed Before Ancestor-Weight Update in `remove_entry_and_descendants` Leaves Stale Descendant Stats on Surviving Ancestors — (`tx-pool/src/component/pool_map.rs`)

### Summary

`remove_entry_and_descendants` strips all link records **before** calling `remove_entry` on each removed transaction. Because `update_ancestors_index_key` relies on those same link records to discover which surviving ancestors need their `descendants_fee / descendants_size / descendants_cycles` decremented, the ancestor-weight update is silently skipped. Surviving ancestors permanently carry inflated descendant-fee stats, which distorts the eviction-key (`EvictKey`) used to decide which transaction to drop when the pool is full.

### Finding Description

`remove_entry_and_descendants` first collects the target and all its descendants, then calls `remove_entry_links` for **every** id in that set before any entry is actually removed:

```rust
// tx-pool/src/component/pool_map.rs  lines 252-265
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← ALL links torn down first
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← ancestor update runs here
        .collect()
}
```

`remove_entry` then calls `update_ancestors_index_key`, which walks `self.links.calc_ancestors(…)` to find surviving parents:

```rust
// lines 432-444
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id()); // ← empty: links already gone
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
```

Because the link for `child` was already removed, `calc_ancestors` returns an empty set. The loop body never executes. Every surviving ancestor of the removed subtree keeps its old `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` values — all of which are now too high.

The `EvictKey` for those ancestors is computed from the stale `descendants_fee` / `descendants_size`:

```rust
// entry.rs  lines 234-247
let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
let feerate = FeeRate::calculate(entry.fee, weight);
EvictKey { fee_rate: descendants_feerate.max(feerate), ... }
```

`next_evict_entry` picks the entry with the **lowest** `EvictKey`. An ancestor whose `descendants_fee` is inflated appears to have a higher fee rate than it actually does, so it is skipped during eviction.

### Impact Explanation

An unprivileged tx-pool submitter can keep a low-fee-rate transaction alive in the pool indefinitely:

1. Submit **tx1** (low fee rate, e.g. 1 shannon/byte).
2. Submit **tx2** (descendant of tx1, high fee rate, e.g. 1 000 shannons/byte). tx1's `descendants_fee` now includes tx2's large fee; tx1's `EvictKey` is high.
3. Trigger removal of tx2 via any of the reachable code paths (RBF replacement, header-dep invalidation via block relay, or ancestor-limit eviction when adding a new transaction). `remove_entry_and_descendants(tx2_id)` is called.
4. tx1's `descendants_fee` is **not** decremented. tx1 still appears to have a high fee rate.
5. When the pool is full, `limit_size` calls `next_evict_entry`, which skips tx1 and evicts a legitimate higher-fee-rate transaction instead.

The result is incorrect eviction ordering: low-fee-rate transactions survive pool pressure while higher-fee-rate transactions are dropped, degrading the economic guarantees of the tx-pool and potentially allowing an attacker to grief honest users.

### Likelihood Explanation

All triggering code paths are reachable by an unprivileged peer:

- **RBF replacement** (`check_rbf` → `remove_entry_and_descendants`) — any submitter can send a conflicting transaction with a higher fee.
- **Header-dep invalidation** (`resolve_conflict_header_dep` → `remove_entry_and_descendants`) — any block relayer can supply a block that invalidates a header dep.
- **Ancestor-limit eviction** (`check_and_record_ancestors` → `remove_entry_and_descendants`) — any submitter can push a transaction that exceeds the ancestor limit.
- **Size-limit eviction** (`limit_size` → `remove_entry_and_descendants`) — any submitter can flood the pool to trigger size-limit enforcement.

No privileged access, key material, or majority hash power is required.

### Recommendation

Update ancestor stats **before** tearing down links. One correct approach: for the root entry only, call `update_ancestors_index_key` while links are still intact, then proceed with link removal and entry deletion for the full subtree.

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Update surviving ancestors' descendant stats BEFORE removing links
    if let Some(root_entry) = self.entries.get_by_id(id).map(|e| e.inner.clone()) {
        self.update_ancestors_index_key(&root_entry, EntryOp::Remove);
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

### Proof of Concept

**Setup:** pool with tx1 → tx2 (tx2 spends tx1's output).

| Step | Action | tx1 `descendants_fee` | tx1 `EvictKey` |
|---|---|---|---|
| 1 | Add tx1 (fee=1, size=100) | 1 shannon | low |
| 2 | Add tx2 (fee=1000, size=100) | 1001 shannons | high |
| 3 | `remove_entry_and_descendants(tx2_id)` | **still 1001** (bug) | **still high** (bug) |
| 4 | Pool full; `next_evict_entry` | tx1 skipped | tx3 (fee=500) evicted instead |

After step 3, tx1 has `descendants_fee = 1001` but its only descendant (tx2) is gone. The correct value is `1`. Because `EvictKey` uses `descendants_feerate.max(feerate)`, tx1's eviction key remains inflated, and a transaction with a genuinely higher fee rate (tx3, fee=500) is evicted in its place.
