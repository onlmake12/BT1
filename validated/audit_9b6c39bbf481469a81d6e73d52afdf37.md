### Title
Stale Descendant Statistics in `PoolMap` Due to Pre-Removed Links Before Ancestor Weight Update — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all transaction links are pre-removed for the entire batch before `remove_entry` is called on each member. Inside `remove_entry`, `update_ancestors_index_key` attempts to walk the link graph to find ancestors and decrement their `descendants_*` statistics — but the links are already gone, so `calc_ancestors` returns an empty set and no ancestor weight update occurs. Ancestors that remain in the pool are left with permanently inflated `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee`, corrupting the pool's eviction-ordering state.

---

### Finding Description

`remove_entry_and_descendants` first collects the target transaction and all its descendants, then calls `remove_entry_links` for every id in the batch: [1](#0-0) 

The comment on line 256 acknowledges the intent: pre-removing links prevents `update_descendants_index_key` from running inside `remove_entry` for entries that are themselves being removed. However, the same pre-removal also silences `update_ancestors_index_key`: [2](#0-1) 

`update_ancestors_index_key` calls `self.links.calc_ancestors`, which traverses `TxLinksMap::inner`: [3](#0-2) 

Because `remove_entry_links` already deleted the target's record from `TxLinksMap::inner`, `calc_ancestors` returns an empty `HashSet`. The `for anc_id in &ancestors` loop body never executes, so no ancestor's `descendants_*` fields are decremented.

`remove_entry_links` itself removes the entry from its parents' children sets and from its own record: [4](#0-3) 

After the batch removal, any transaction that was a parent of the removed subtree root still carries the old `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` values contributed by the now-removed children.

---

### Impact Explanation

The stale `descendants_*` fields corrupt two derived keys that govern pool behaviour:

**Eviction key** — computed in `EvictKey::from`: [5](#0-4) 

`descendants_feerate` and `descendants_count` are both inflated. A transaction whose real descendants have been removed will appear to have a higher-fee-rate descendant cluster than it actually does, shifting its position in the eviction order. Under pool pressure, the wrong transactions are evicted: a low-fee ancestor may be protected from eviction because its stale `descendants_feerate` is artificially high, while a legitimately higher-value transaction is displaced.

**Score key** — `AncestorsScoreSortKey` uses `ancestors_*` fields, not `descendants_*`, so mining-priority ordering is not directly corrupted by this path. However, the eviction of the wrong transactions indirectly changes which transactions are available for block assembly.

The pool's internal consistency invariant — that every entry's `descendants_*` fields equal the sum of the corresponding fields over its actual in-pool descendants — is violated for every ancestor of a batch-removed subtree.

---

### Likelihood Explanation

The bug is reachable by any unprivileged tx-pool submitter through `check_and_record_ancestors`, which calls `remove_entry_and_descendants` when a newly submitted transaction's ancestor count exceeds `max_ancestors_count` and cell-reference parents must be evicted: [6](#0-5) 

**Concrete trigger path:**
1. Attacker submits `tx_A → tx_B → tx_C` (a three-hop chain) to the pool. `tx_B` is used as a cell dep by a future transaction.
2. Attacker submits `tx_X` that lists `tx_B` as a cell dep and has enough ancestors to push the count over `max_ancestors_count`.
3. `check_and_record_ancestors` identifies `tx_B` as a `cell_ref_parent` candidate and calls `remove_entry_and_descendants(&tx_B_id)`.
4. `removed_ids = [tx_B, tx_C]`. Links for both are pre-removed.
5. `remove_entry(tx_B)` runs; `update_ancestors_index_key` finds no ancestors via the now-empty link graph; `tx_A`'s `descendants_*` fields are never decremented.
6. `tx_A` remains in the pool with permanently inflated descendant statistics.

No special privilege, hashpower, or key material is required. The attacker needs only the ability to submit transactions to the RPC or P2P relay interface.

---

### Recommendation

Do not pre-remove links before calling `remove_entry`. Instead, collect the ancestors of the subtree root *before* any removal, then pass that set into the per-entry removal so that ancestor weight updates can be applied correctly:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Collect the set of ancestors that will REMAIN in the pool after removal.
    // These need their descendants_* stats decremented.
    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();

    // Remove links first so that remove_entry does not double-update descendants
    // that are themselves being removed.
    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    // Now remove each entry; pass the remaining-ancestor set so that
    // update_ancestors_index_key can still find and update them.
    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry_with_ancestor_set(id, &removed_set))
        .collect()
}
```

Alternatively, snapshot each entry's ancestor set (via `calc_ancestors`) before any links are removed, and apply the weight decrements to those ancestors after the batch removal.

### Proof of Concept

**State before:**
- Pool contains `tx_A` (root), `tx_B` (child of `tx_A`), `tx_C` (child of `tx_B`).
- `tx_A.descendants_count == 3`, `tx_A.descendants_size == size_A + size_B + size_C`.

**Trigger:** call `pool_map.remove_entry_and_descendants(&tx_B_id)`.

**Execution trace:**
1. `removed_ids = [tx_B, tx_C]` — `calc_descendants(tx_B)` returns `{tx_C}`.
2. `remove_entry_links(tx_B)`: removes `tx_B` from `tx_A.children`, removes `tx_C` from `tx_B.parents`, deletes `links.inner[tx_B]`.
3. `remove_entry_links(tx_C)`: removes `tx_C` from `tx_B.children` (already gone), deletes `links.inner[tx_C]`.
4. `remove_entry(tx_B)`:
   - `update_ancestors_index_key(tx_B, Remove)` → `calc_ancestors(tx_B)` → `links.inner.get(&tx_B_id)` returns `None` → empty set → **no update to `tx_A`**.
5. `remove_entry(tx_C)`: same — no ancestors found.

**State after:**
- `tx_A` remains in pool.
- `tx_A.descendants_count` is still `3` (should be `1`).
- `tx_A.descendants_size` still includes `size_B + size_C` (should be `size_A` only).
- `tx_A.evict_key` is computed from the stale inflated `descendants_feerate` and `descendants_count`, corrupting eviction ordering. [7](#0-6) [8](#0-7)
