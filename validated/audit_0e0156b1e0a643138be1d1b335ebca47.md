Audit Report

## Title
Unbounded `async_handle().spawn()` Task Accumulation via Unauthenticated `GetHeaders` Flood — (`sync/src/synchronizer/get_headers_process.rs`)

## Summary
Any unprivileged peer can flood a post-IBD CKB node with `GetHeaders` messages using a trivially valid two-entry locator (`[<random_hash>, <genesis_hash>]`). Each message unconditionally builds a `SendHeaders` payload of up to 2,000 headers and spawns a fire-and-forget Tokio task with no rate limit, backpressure, or task cap. Tasks accumulate unboundedly, leading to memory exhaustion and starvation of block processing, relay, and RPC subsystems.

## Finding Description

**Root cause — no rate limit on `GetHeaders` in `Synchronizer`:**

`GetHeadersProcess::execute()` performs only two cheap checks before spawning a task:
1. Locator size ≤ `MAX_LOCATOR_SIZE` (101) — `sync/src/synchronizer/get_headers_process.rs` lines 47–51
2. Node is not in IBD — lines 53–66

If both pass, it unconditionally builds a full `SendHeaders` message and spawns a fire-and-forget task:

```rust
// sync/src/synchronizer/get_headers_process.rs, lines 84–93
let content = packed::SendHeaders::new_builder()
    .headers(headers.into_iter().map(|x| x.data()).collect::<Vec<_>>())
    .build();
let message = packed::SyncMessage::new_builder().set(content).build();
let nc = Arc::clone(self.nc);
self.synchronizer
    .shared()
    .shared()
    .async_handle()
    .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
```

**Trivially bypassable `locate_latest_common_block` gate:**

`locate_latest_common_block` (`sync/src/types/mod.rs`, lines 1857–1902) only requires the last locator entry to equal the genesis hash. With locator `[<random_hash>, <genesis_hash>]`:
- The last entry is the genesis hash → passes the check at line 1867
- The iterator finds the genesis hash at index 1 (the random hash is unknown to the node)
- `latest_common == Some(0)` → returns `Some(0)` at line 1879–1880

This causes `get_locator_response(0, &hash_stop)` to return up to `MAX_HEADERS_LEN` = 2,000 headers.

**`Handle::spawn()` is unbounded:**

`util/runtime/src/native.rs` lines 47–60 show `Handle::spawn()` is a thin wrapper over `tokio::Handle::spawn()` with no semaphore, queue depth limit, or backpressure.

**`Synchronizer` struct has no rate limiter field:**

`sync/src/synchronizer/mod.rs` lines 357–362 show the `Synchronizer` struct contains only `chain`, `shared`, and `fetch_channel` — no rate limiter. The `try_process()` match arm for `GetHeaders` (lines 397–401) has no rate-limiting guard.

**Contrast with `Relayer`:**

`sync/src/relayer/mod.rs` lines 63–92 show the `Relayer` struct carries a `governor`-based `RateLimiter<(PeerIndex, u32)>` capped at 30 req/s per peer per message type, applied at lines 113–123 before any processing. No equivalent protection exists in `Synchronizer`.

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Each spawned task holds a serialized `SendHeaders` message of up to 2,000 × ~192 bytes ≈ 384 KB, allocated synchronously before the spawn. At 10,000 queued tasks, heap pressure from message payloads alone reaches ~3.84 GB. The shared Tokio runtime services block processing, relay propagation, and RPC; unbounded task accumulation starves all these subsystems, causing OOM kill or livelock of the node. The attack is asymmetric: the attacker sends ~200-byte messages while the victim allocates ~384 KB per message.

## Likelihood Explanation

All preconditions are reachable in normal mainnet operation with no privilege:
1. Establish a standard P2P connection (no key, no role, no PoW required).
2. Wait for the target node to exit IBD (normal state for any synced node).
3. Send `GetHeaders` in a tight loop with locator `[<random_32_byte_hash>, <genesis_hash>]`.

A single peer connection is sufficient. The attack is repeatable and requires no special knowledge of the target chain state.

## Recommendation

Apply a per-peer rate limit to `GetHeaders` processing in the `Synchronizer`, mirroring the existing `governor`-based `RateLimiter` in the `Relayer` (`sync/src/relayer/mod.rs`, lines 63–92). A limit of 1–5 `GetHeaders` per second per peer is sufficient for legitimate sync behavior. Additionally, consider bounding total outstanding `SendHeaders` spawn tasks via a `tokio::sync::Semaphore` to enforce a hard cap on queued work regardless of peer count.

## Proof of Concept

1. Connect to a post-IBD CKB node via the Sync P2P protocol.
2. In a tight loop, send `SyncMessage::GetHeaders` with `block_locator_hashes = [<random_32_byte_hash>, <genesis_hash>]` and `hash_stop = Byte32::zero()`.
3. Monitor Tokio task count (via metrics or `/proc/<pid>/status` VmRSS) and heap allocation.
4. Assert: after N messages, task queue depth grows proportionally to N and heap grows proportionally to `N × MAX_HEADERS_LEN × header_size`.
5. Observe RPC latency and block relay degradation as the runtime is saturated, eventually leading to OOM kill or node unresponsiveness.
