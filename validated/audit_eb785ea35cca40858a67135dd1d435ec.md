### Title
Active Subscription Balance Can Fall Below `minimumBalance` After Keeper Fee Deductions Without Deactivation — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

In `Scheduler.sol`, the `minimumBalance` invariant is enforced at deposit and withdrawal time (`createSubscription`, `updateSubscription`, `addFunds`, `withdrawFunds`), but **not** after fee deductions inside `updatePriceFeeds`. A keeper calling `updatePriceFeeds` deducts both the Pyth fee and the keeper fee from `status.balanceInWei` with no post-deduction check against `minimumBalance`. The subscription remains marked `isActive = true` even when its balance has been drained below the minimum, violating the protocol's own invariant.

### Finding Description

`updatePriceFeeds` performs two sequential balance deductions:

1. **Pyth fee deduction** — checks only `status.balanceInWei >= pythFee`, then deducts.
2. **Keeper fee deduction** — checks only `status.balanceInWei >= totalKeeperFee`, then deducts.

Neither step checks whether the post-deduction balance remains at or above `minimumBalance`. The subscription is never deactivated by this path.

```
// Step 1 — only checks pythFee, not minimumBalance
if (status.balanceInWei < pythFee) revert InsufficientBalance();
status.balanceInWei -= pythFee;                          // line 305

// Step 2 — only checks keeperFee, not minimumBalance
if (status.balanceInWei < totalKeeperFee) revert InsufficientBalance();
status.balanceInWei -= totalKeeperFee;                   // line 856
// ← no check: status.balanceInWei >= minimumBalance
``` [1](#0-0) [2](#0-1) 

By contrast, every user-controlled path enforces the minimum:

- `createSubscription`: `if (msg.value < minimumBalance) revert InsufficientBalance()`
- `updateSubscription`: `if (currentStatus.balanceInWei < minimumBalance) revert InsufficientBalance()`
- `addFunds`: `if (status.balanceInWei < minimumBalance) revert InsufficientBalance()`
- `withdrawFunds`: `if (status.balanceInWei - amount < minimumBalance) revert InsufficientBalance()` [3](#0-2) [4](#0-3) [5](#0-4) 

The keeper-controlled path (`updatePriceFeeds`) is the only path that modifies `balanceInWei` without enforcing the minimum, creating an asymmetry identical in structure to the external report.

The existing test `testAddFundsEnforcesMinimumBalance` explicitly confirms that repeated `updatePriceFeeds` calls drain the balance below `minimumBalance` while the subscription remains active: [6](#0-5) 

### Impact Explanation

1. **Invariant violation**: An active subscription can hold a balance below `minimumBalance`. The protocol's own invariant — "active subscriptions must hold at least `minimumBalance`" — is silently broken by normal keeper operations.

2. **Manager locked out of incremental top-ups**: `addFunds` enforces that the post-deposit balance must be `>= minimumBalance`. If the balance has been drained below minimum by fee deductions, the manager cannot add a small amount to sustain the subscription — they must deposit enough to clear the full minimum in one transaction. This is a direct operational consequence for subscription managers.

3. **Subscription drains to zero while remaining active**: Keepers continue to discover the subscription via `getActiveSubscriptions` and call `updatePriceFeeds`, further draining the balance until it can no longer cover even the Pyth fee. The subscription stays in the active list throughout, wasting keeper gas and misleading downstream readers who assume active subscriptions are funded. [7](#0-6) 

### Likelihood Explanation

This is not an edge case — it is the normal lifecycle of any subscription funded to exactly `minimumBalance`. The very first successful `updatePriceFeeds` call deducts `pythFee + keeperFee`, which is always positive, immediately pushing the balance below `minimumBalance`. Any subscription funded at or near the minimum will trigger this condition on its first update. The entry path is fully permissionless: any keeper can call `updatePriceFeeds`. [8](#0-7) 

### Recommendation

After all fee deductions in `updatePriceFeeds` (i.e., after `_processFeesAndPayKeeper` returns), add a check:

```solidity
uint256 minimumBalance = this.getMinimumBalance(uint8(params.priceIds.length));
if (status.balanceInWei < minimumBalance) {
    params.isActive = false;
    _removeFromActiveSubscriptions(subscriptionId);
    emit SubscriptionDeactivated(subscriptionId);
}
```

This mirrors the pattern used in `withdrawFunds` and `addFunds`, and ensures the invariant is maintained after every balance-modifying operation. Alternatively, the minimum balance check could be enforced at the start of `updatePriceFeeds` (before deductions) to prevent updates that would push the balance below minimum. [9](#0-8) 

### Proof of Concept

1. Admin deploys `SchedulerUpgradeable` with `minimumBalancePerFeed = 1 ether` and `singleUpdateKeeperFeeInWei = 0.6 ether`.
2. Manager calls `createSubscription` with 1 price feed, sending exactly `1 ether` (the minimum).
3. Keeper calls `updatePriceFeeds`. The Pyth fee (e.g., 0.01 ether) plus keeper fee (gas + 0.6 ether) totals ~0.61 ether. Both checks pass (`1 ether >= 0.01 ether`, `0.99 ether >= 0.61 ether`). After deductions, `balanceInWei ≈ 0.39 ether < 1 ether = minimumBalance`.
4. Subscription is still `isActive = true` and still in `activeSubscriptionIds`.
5. Manager calls `addFunds{value: 0.5 ether}`. This reverts with `InsufficientBalance` because `0.39 + 0.5 = 0.89 ether < 1 ether`. Manager must send at least `0.61 ether` in a single call to top up.
6. Keeper calls `updatePriceFeeds` again, draining the balance further. This repeats until the balance is too low to cover even the Pyth fee. [10](#0-9) [11](#0-10)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
```

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;
```
