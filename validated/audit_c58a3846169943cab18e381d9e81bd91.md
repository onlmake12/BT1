Audit Report

## Title
Missing Per-Peer Rate Limiting on `BlockFilter` Protocol Handler Allows Resource Exhaustion — (File: `sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

## Summary
The `BlockFilter` P2P protocol handler carries no `rate_limiter` field and performs zero rate-limit checks across its entire call chain (`received` → `process` → `try_process` → `GetBlockFiltersProcess::execute`). Any peer holding a valid P2P connection can send `GetBlockFilters` messages in a tight loop, triggering up to 1,000 DB reads and up to ~1.8 MB of heap allocation per request with no throttling. With `max_peers = 125`, this saturates RocksDB I/O, the async thread pool, and the heap allocator, potentially crashing the node or rendering it unresponsive to legitimate peers.

## Finding Description
**Root cause:** `BlockFilter` (defined in `sync/src/filter/mod.rs`) has only a single field:

```rust
pub struct BlockFilter {
    shared: Arc<SyncShared>,
}
```

No `rate_limiter` field exists. The full dispatch chain is:

1. `BlockFilter::received` (`sync/src/filter/mod.rs:122`) — parses the message, then calls `self.process(...)` with no rate check.
2. `BlockFilter::process` (`mod.rs:70`) — calls `self.try_process(...)` with no rate check.
3. `BlockFilter::try_process` (`mod.rs:33`) — dispatches directly to `GetBlockFiltersProcess::new(...).execute().await` with no rate check.
4. `GetBlockFiltersProcess::execute` (`sync/src/filter/get_block_filters_process.rs:33`) — iterates up to `BATCH_SIZE = 1000` blocks, performing two DB lookups per block (`get_block_hash` + `get_block_filter`), accumulating results into `block_hashes` and `filters` Vecs, and serializing them into a molecule message capped at 1.8 MB before calling `async_send_message_to`.

The 1.8 MB cap (`get_block_filters_process.rs:53`) exists solely to prevent tentacle frame-size disconnects, not as a security measure against flooding.

**Contrast with protected handlers:**
- `Relayer` (`sync/src/relayer/mod.rs:81`) carries `rate_limiter: RateLimiter<(PeerIndex, u32)>` and checks it at the very top of `try_process` (`mod.rs:116–123`), returning `StatusCode::TooManyRequests` before any processing.
- `HolePunching` (`network/src/protocols/hole_punching/mod.rs:45`) carries `rate_limiter: RateLimiter<(PeerIndex, u32)>` and checks it at the top of `received` (`mod.rs:95–107`), returning early before any processing.

`BlockFilter` is the only major P2P protocol handler with no equivalent guard. Additionally, `GetBlockFilterHashesProcess` (`sync/src/filter/get_block_filter_hashes_process.rs:8`) uses `BATCH_SIZE = 2000` and is equally unprotected under the same `BlockFilter` handler.

## Impact Explanation
With `max_peers = 125`, up to 125 inbound peers can each send `GetBlockFilters(start_number=0)` in a tight loop. Each request triggers up to 1,000 DB reads (two per block: `get_block_hash` + `get_block_filter`) and up to ~1.8 MB of heap allocation. Concurrently, this saturates the async thread pool, RocksDB I/O bandwidth, and the heap allocator. The node becomes unresponsive to legitimate sync peers and may OOM-crash on memory-constrained deployments.

This matches two allowed High-severity CKB bounty impacts:
- **"Vulnerabilities which could easily crash a CKB node"** (High, 10001–15000 points)
- **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (High, 10001–15000 points)

## Likelihood Explanation
The attack requires only a valid P2P connection — no PoW, no keys, no privileged role. The `Filter` protocol is listed in the default `support_protocols` and is open to any peer on mainnet. A single attacker controlling even a handful of peers can sustain the flood indefinitely at negligible cost. The message is a fixed 8-byte struct (`GetBlockFilters { start_number: Uint64 }`), so the attacker's bandwidth cost is trivial while the victim's DB I/O and memory cost per message is orders of magnitude higher.

## Recommendation
Add a `governor`-based `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter`, mirroring the pattern in `Relayer::new` (`sync/src/relayer/mod.rs:88–98`) which sets 30 req/sec per peer per message type. Check it at the top of `BlockFilter::received` (or `try_process`) before dispatching to any process handler, returning early (and optionally banning the peer) on rate-limit violation. Call `rate_limiter.retain_recent()` in `BlockFilter::disconnected` to prevent unbounded memory growth in the limiter's state store.

## Proof of Concept
1. Connect N peers (up to `max_peers = 125`) to a node that has block filters built for a long chain (e.g., mainnet or a local chain with >1000 blocks).
2. Each peer sends `BlockFilterMessage { GetBlockFilters { start_number: 0 } }` in a tight loop over the `Filter` protocol (SupportProtocols::Filter).
3. Monitor node CPU, heap allocation, and RocksDB I/O — all climb without bound.
4. Observe that legitimate sync requests time out and the node becomes unresponsive.
5. On a memory-constrained deployment, the node OOM-crashes.

The call path with no rate check anywhere: `BlockFilter::received` (`sync/src/filter/mod.rs:152`) → `BlockFilter::process` (`mod.rs:78`) → `BlockFilter::try_process` (`mod.rs:41–43`) → `GetBlockFiltersProcess::execute` (`sync/src/filter/get_block_filters_process.rs:33`, up to 1,000 DB reads + ~1.8 MB alloc) → `async_send_message_to`.
