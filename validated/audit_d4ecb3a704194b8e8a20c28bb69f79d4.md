Audit Report

## Title
Unbounded `runtime::spawn` of 30-second NAT traversal futures via spoofed `ConnectionSync` messages causes Tokio runtime exhaustion — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

## Summary
An unprivileged remote peer can exhaust the Tokio runtime by first poisoning `pending_delivered` with entries keyed by spoofed `from` peer IDs via `ConnectionRequest` messages (bypassing per-`from` rate limits using distinct fake peer IDs), then sending `ConnectionSync` messages that each trigger an uncapped `runtime::spawn` of a `select_ok` task over up to 24 concurrent 30-second `try_nat_traversal` futures. At steady state this produces ~900 concurrent tasks holding ~21,600 live futures, exhausting file descriptors and starving the shared Tokio runtime, causing the local RPC API to become unresponsive.

## Finding Description

**Phase 1 — Unbounded `pending_delivered` poisoning**

`pending_delivered` is a plain `HashMap<PeerId, PendingDeliveredInfo>` with no size cap. [1](#0-0) 

When a `ConnectionRequest` arrives with `content.to == local_peer_id`, `respond_delivered` is called with the attacker-supplied `content.from`, which is never validated against the actual session peer ID. [2](#0-1) 

The only per-`from` guard is `HOLE_PUNCHING_INTERVAL` (2 minutes), which only blocks re-insertion for the **same** `from_peer_id`. [3](#0-2) 

After passing the address filter (TCP/IPv4 or IPv6, trivially satisfied by the attacker), the entry is inserted unconditionally with no map size limit. [4](#0-3) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1/sec, so using N distinct fake `from` peer IDs creates N independent rate-limit buckets, bypassing it entirely. [5](#0-4) 

The binding throttle is the top-level `rate_limiter` at 30 messages/second per `(session_id, msg_item_id)`, allowing 30 new `pending_delivered` entries per second. [6](#0-5) 

Entries expire after 5 minutes (`TIMEOUT`), so up to 30 × 300 = 9,000 entries can accumulate. [7](#0-6) 

**Phase 2 — Uncapped `runtime::spawn` per `ConnectionSync`**

When a `ConnectionSync` arrives with `content.to == local_peer_id`, an empty `route`, and `content.from` matching a `pending_delivered` entry, `execute` creates one `try_nat_traversal` future per stored address and calls `runtime::spawn` with no concurrency cap or in-flight counter. [8](#0-7) 

The `forward_rate_limiter` for `ConnectionSync` is also keyed by `(from, to, msg_item_id)` at 1/sec, so using 30 distinct fake `from` values allows 30 spawns/second, matching the top-level limiter ceiling. [9](#0-8) 

Each spawned task runs `select_ok` over up to 24 `try_nat_traversal` futures, each looping for a hard-coded 30-second `timeout_duration`, creating a TCP socket every ~200–400 ms. [10](#0-9) [11](#0-10) 

**Accumulation math:**
- Spawn rate: 30 tasks/second (top-level limiter ceiling)
- Task lifetime: 30 seconds (`timeout_duration`)
- Steady-state concurrent tasks: 30 × 30 = **900 tasks**
- Concurrent futures: 900 × 24 = **21,600 futures**
- Socket operations: up to ~54,000 open/close per second

## Impact Explanation
The Tokio runtime is shared between the P2P layer and the RPC server. Accumulation of 21,600 live futures each repeatedly creating TCP sockets exhausts the process file descriptor limit, saturates the Tokio task queue, and starves RPC handler tasks. The local RPC API becomes unresponsive or crashes. This matches the in-scope bounty impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation
The attack requires only a single P2P connection with no special privileges, no PoW, and no key material. The target node's peer ID is publicly discoverable via the P2P protocol. The `from` field in `ConnectionRequest` is not authenticated against the actual session peer ID. The attack is fully scriptable: connect, send 30 `ConnectionRequest`/second for ~10 seconds to build `pending_delivered`, then send 30 `ConnectionSync`/second indefinitely. No Sybil attack or majority hashpower is needed.

## Recommendation
1. **Cap `pending_delivered` size**: enforce a hard maximum (e.g., 256 entries) and evict oldest entries on overflow.
2. **Bound concurrent NAT traversal tasks per entry**: track an in-flight counter per `from` peer ID; skip `runtime::spawn` if a task is already running for that entry.
3. **Validate `content.from` against the actual session peer ID** in `ConnectionRequest` processing to prevent spoofed-`from` poisoning.
4. **Reduce `try_nat_traversal` timeout** or add a global cap on total concurrent NAT traversal tasks.

## Proof of Concept
```
1. Connect to target node (single TCP connection); obtain local_peer_id via peer discovery.
2. For i in 0..300 (at 30/second):
     Send ConnectionRequest {
       from: random_peer_id_i,   // distinct fake peer ID each time
       to:   local_peer_id,
       listen_addrs: [1.2.3.4:1000 .. 1.2.3.4:1023]  // 24 TCP/IPv4 addrs
     }
   // pending_delivered accumulates 300 entries over 10 seconds
3. Loop forever at 30/second:
     For each fake_peer_id_i in pending_delivered:
       Send ConnectionSync {
         from: fake_peer_id_i,
         to:   local_peer_id,
         route: []
       }
   // Each message triggers runtime::spawn(select_ok(24 × try_nat_traversal(30s)))
4. After 30 seconds: ~900 concurrent tasks, ~21,600 live futures, ~54,000 socket ops/sec.
5. Assert: RPC endpoint (e.g., HTTP JSON-RPC) stops responding.
```
