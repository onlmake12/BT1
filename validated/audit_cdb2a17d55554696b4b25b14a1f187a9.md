### Title
Single Subscription-Level Staleness Timestamp Masks Per-Feed Staleness in `getPricesNoOlderThan` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler.sol` contract stores a single `priceLastUpdatedAt` timestamp per subscription, computed as the **maximum** `publishTime` across all feeds in the batch. The `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` read functions validate freshness against this single subscription-level timestamp rather than per-feed timestamps. When a subscription contains feeds with different update frequencies (e.g., a 24/7 crypto feed alongside an equity feed that only updates during market hours), the staleness check passes based on the crypto feed's recent timestamp while the equity feed's price may be many hours old. Readers relying on `getPricesNoOlderThan` receive a false freshness guarantee.

---

### Finding Description

In `_validateShouldUpdatePrices`, the contract deliberately selects the **maximum** publish time across all feeds as the representative update timestamp:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

This is then stored as the subscription-level timestamp:

```solidity
status.priceLastUpdatedAt = latestPublishTime; // = max publishTime across all feeds
```

The `parsePriceFeedUpdatesWithConfig` call uses `minAllowedPublishTime = 0`, explicitly allowing arbitrarily old individual feed timestamps to pass parsing, with the comment:

> "We don't want to reject update data if it contains a price from a market that closed a few days ago."

However, the read-side staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` uses only this single max-based timestamp:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
```

The function name `getPricesNoOlderThan` and the `age_seconds` parameter imply that **all** returned prices satisfy the age constraint. This guarantee is not upheld: individual feeds whose `publishTime` is far older than `age_seconds` are returned without any per-feed age check.

---

### Impact Explanation

A protocol integrating with the Scheduler that calls `getPricesNoOlderThan(subscriptionId, priceIds, 60)` (expecting all prices to be at most 60 seconds old) will receive the call without revert even when some feeds in the subscription have prices that are hours or days old (e.g., an equity feed during off-hours or a weekend). The protocol's downstream logic — liquidations, collateral valuation, trade settlement — operates on stale prices it believes are fresh, potentially causing incorrect liquidations, mispriced collateral, or exploitable arbitrage.

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` constant (1 hour) only bounds the **maximum** timestamp in the batch, not each individual feed's timestamp. A subscription mixing ETH/USD (updates every second) with AAPL/USD (last updated 16 hours ago at market close) will have `priceLastUpdatedAt` set to ETH/USD's recent time, while AAPL/USD's 16-hour-old price is silently served as "fresh."

---

### Likelihood Explanation

The Scheduler is explicitly designed to support mixed-asset subscriptions including equity feeds with market hours. The README and inline comments acknowledge that "closed markets will have a publishTime from their last trading period." Any subscription manager who combines crypto and equity feeds — a natural and documented use case — creates the conditions for this issue. Any whitelisted reader (or any reader if `whitelistEnabled = false`) can trigger the misleading staleness check by calling `getPricesNoOlderThan`. No privileged access is required.

---

### Recommendation

Replace the subscription-level staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed check against each feed's individual `publishTime`:

```solidity
function getPricesNoOlderThan(
    uint256 subscriptionId,
    bytes32[] calldata priceIds,
    uint256 age_seconds
) external view override onlyWhitelistedReader(subscriptionId) returns (PythStructs.Price[] memory prices) {
    prices = this.getPricesUnsafe(subscriptionId, priceIds);
    for (uint i = 0; i < prices.length; i++) {
        if (distance(block.timestamp, prices[i].publishTime) > age_seconds)
            revert PythErrors.StalePrice();
    }
}
```

Alternatively, document clearly that `getPricesNoOlderThan` checks the subscription update time (not per-feed freshness) and provide a separate `getPricesWithPerFeedAgeCheck` function for protocols that require per-feed freshness guarantees.

---

### Proof of Concept

1. Subscription manager creates a subscription with two feeds: `ETH/USD` (crypto, updates every second) and `AAPL/USD` (equity, last updated 16 hours ago at market close). `whitelistEnabled = false`.

2. A keeper calls `updatePriceFeeds(subscriptionId, updateData)` with a batch where:
   - `ETH/USD.publishTime = block.timestamp - 30` (30 seconds ago, within `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 3600`)
   - `AAPL/USD.publishTime = block.timestamp - 57600` (16 hours ago, market closed)
   - Both feeds share the same Pythnet slot (slot check passes)

3. In `_validateShouldUpdatePrices`:
   - `updateTimestamp = max(block.timestamp - 30, block.timestamp - 57600) = block.timestamp - 30`
   - `minAllowedTimestamp = block.timestamp - 3600`
   - `updateTimestamp (block.timestamp - 30) >= minAllowedTimestamp` → passes
   - `status.priceLastUpdatedAt = block.timestamp - 30`

4. A reader (any address) calls `getPricesNoOlderThan(subscriptionId, [ETH_ID, AAPL_ID], 60)`:
   - `distance(block.timestamp, priceLastUpdatedAt) = 30 < 60` → **no revert**
   - Returns `ETH/USD` price (30s old, fresh) and `AAPL/USD` price (16h old, stale)

5. The reader's protocol uses the returned `AAPL/USD` price believing it is at most 60 seconds old, leading to incorrect financial decisions.
