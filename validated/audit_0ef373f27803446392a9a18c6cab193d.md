The bug described is real and present in the code. Here is the analysis:

### Title
Pool Size Accounting Corruption via Eviction During `add_entry` — (`tx-pool/src/component/pool_map.rs`)

### Summary

In `add_entry`, the new totals are computed into local variables **before** `check_and_record_ancestors` runs. That function can evict existing entries (via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`), which correctly decrements `self.total_tx_size`/`self.total_tx_cycles` in-place. However, lines 218–219 then **overwrite** those fields with the stale pre-eviction local values, permanently overstating both counters by the sum of the evicted entries' sizes and cycles.

### Finding Description

The execution order in `add_entry` is:

```
line 210-211: local (total_tx_size, total_tx_cycles) = self.total_tx_size + entry.size,
                                                        self.total_tx_cycles + entry.cycles
line 213:     check_and_record_ancestors()
                → remove_entry_and_descendants()
                  → remove_entry()
                    → update_stat_for_remove_tx()   ← mutates self.total_tx_size/cycles
line 218-219: self.total_tx_size  = total_tx_size   ← OVERWRITES post-eviction value
              self.total_tx_cycles = total_tx_cycles ← OVERWRITES post-eviction value
``` [1](#0-0) 

The eviction path inside `check_and_record_ancestors` is triggered when:
- `ancestors_count > max_ancestors_count`, AND
- `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`

meaning the new tx consumes a cell that existing pool transactions reference as a cell dep, and the ancestor count is just over the limit. [2](#0-1) 

`remove_entry` calls `update_stat_for_remove_tx`, which correctly subtracts the evicted entry's size and cycles from `self.total_tx_size`/`self.total_tx_cycles`: [3](#0-2) 

But lines 218–219 then blindly overwrite those fields with the pre-eviction snapshot, discarding the subtraction entirely.

**Correct final value:** `(S − evicted_size) + entry.size`
**Actual value written:** `S + entry.size`

The overstatement equals the total size/cycles of all entries evicted during that `add_entry` call. It is permanent — there is no self-correction path unless the pool is fully cleared.

### Impact Explanation

`total_tx_size` and `total_tx_cycles` are the authoritative counters used by `updated_stat_for_add_tx` to gate future admissions. An overstated counter causes the pool to reject subsequent valid transactions with `Reject::Full` even when actual pool occupancy is below the configured limit. The error accumulates across multiple such eviction events, making the pool progressively more restrictive than intended. [4](#0-3) 

### Likelihood Explanation

The trigger condition — a new transaction consuming an output that existing pool transactions reference as a cell dep, with ancestor count just over `max_ancestors_count` — is a supported and documented code path (the `cell_ref_parents` eviction logic exists precisely for this case). An unprivileged submitter can craft this scenario via the standard `send_transaction` RPC or P2P transaction relay. No privileged access, key material, or majority hashpower is required.

### Recommendation

Compute the new totals **after** `check_and_record_ancestors` completes (so any eviction-driven decrements are already reflected in `self.total_tx_size`/`self.total_tx_cycles`), then add only the new entry's contribution:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute AFTER evictions have already updated self.total_tx_size/cycles
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

The pre-flight overflow check in `updated_stat_for_add_tx` can still run early (it only reads, does not write), but the assignment must happen after all mutations.

### Proof of Concept

1. Fill the pool to near `max_ancestors_count` with a chain: `tx0 → tx1 → … → txN` where `tx0`'s output is used as a cell dep by a side transaction `txDep` also in the pool.
2. Submit a new transaction `txNew` that spends `tx0`'s output as an **input** (not cell dep) and has `txN` as an ancestor, pushing `ancestors_count` just above `max_ancestors_count`.
3. `check_and_record_ancestors` evicts `txDep` (a `cell_ref_parent`), calling `update_stat_for_remove_tx(txDep.size, txDep.cycles)` — correctly decrementing `self.total_tx_size`.
4. Lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot, re-adding `txDep.size` to the total even though `txDep` is no longer in the pool.
5. Assert: `pool_map.total_tx_size != pool_map.entries.iter().map(|e| e.inner.size).sum()` — the invariant is broken.
6. Subsequent `send_transaction` calls are rejected with `Reject::Full` despite the pool having free capacity equal to `txDep.size`. [5](#0-4)

                self.total_tx_size, tx_size

                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
