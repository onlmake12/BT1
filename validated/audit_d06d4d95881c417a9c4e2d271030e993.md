### Title
ETH Permanently Locked in `Executor` Contract via Empty `receive()` — (File: `target_chains/ethereum/contracts/contracts/executor/Executor.sol`)

### Summary
The `Executor` contract defines an empty `receive() external payable {}` function that silently accepts ETH from any sender, with no corresponding withdrawal mechanism anywhere in the contract or its upgradeable wrapper. Any ETH sent directly to the contract is permanently locked.

### Finding Description
`Executor.sol` at line 113 declares:

```solidity
receive() external payable {}
```

This function accepts any ETH transfer with empty calldata without restriction. The `execute()` function at line 68 is also `payable`, meaning a caller who sends `msg.value > gi.value` will have the excess ETH trapped in the contract as well (since only `gi.value` is forwarded to `callAddress`).

Neither `Executor.sol` nor `ExecutorUpgradable.sol` contains any function to withdraw or recover ETH held by the contract. The only ETH-outflow path is the `call{value: gi.value}` inside `execute()`, which requires a valid governance VAA — an entirely separate privileged flow that cannot be used to recover accidentally deposited ETH. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
**High.** Any ETH sent to the `ExecutorUpgradable` proxy (which delegates to `Executor`) via a plain ETH transfer is permanently unrecoverable. There is no admin, owner, or governance path to drain the ETH balance. The contract owns itself (`_transferOwnership(address(this))`), so even the owner role cannot be used to add a rescue function post-deployment without a full upgrade cycle.

### Likelihood Explanation
**Low.** A user would have to mistakenly send ETH directly to the `Executor` contract address. This is an error scenario, not a normal interaction path. However, the `execute()` function being `payable` also creates a path where a governance executor accidentally includes `msg.value > gi.value`, trapping the difference.

### Recommendation
Remove the `receive()` function entirely, since the `Executor` contract has no legitimate use for holding ETH beyond what is immediately forwarded in `execute()`. Alternatively, replace it with a reverting body:

```solidity
receive() external payable {
    revert("Executor does not accept ETH");
}
```

This mirrors the pattern already used in `ReceiverImplementation.sol` and `Implementation.sol` (Wormhole contracts in the same repo), which both revert in their `receive()` and `fallback()` functions. [5](#0-4) 

### Proof of Concept
1. Deploy `ExecutorUpgradable` (or interact with an existing deployment).
2. Send 1 ETH to the contract address with empty calldata: `(bool ok,) = address(executor).call{value: 1 ether}("")`.
3. The `receive()` function silently accepts the ETH.
4. Attempt to recover the ETH — no function exists to do so.
5. The 1 ETH is permanently locked in the contract.
