### Title
Scripts Using `hash_type = Type` Silently Break When VM Version Upgrades Due to `Exec` Syscall Replacement by `ExecV2` — (File: `script/src/syscalls/generator.rs`)

---

### Summary

The `Exec` syscall (syscall number `2043`) is registered **only for `ScriptVersion::V1`** and is replaced by `ExecV2` in `ScriptVersion::V2`. Both handlers respond to the same syscall number but have fundamentally different error-handling semantics. Scripts using `hash_type = Type` automatically run in the latest activated VM version, so they silently switch from `Exec` to `ExecV2` behavior when CKB2023 activates. This is the direct CKB analog of EIP-4758: a protocol upgrade replaces a feature that existing deployed scripts rely on, potentially voiding their security guarantees without any on-chain signal.

---

### Finding Description

In `script/src/syscalls/generator.rs`, the syscall dispatch is version-gated:

```rust
if script_version == &ScriptVersion::V1 {
    syscalls.push(Box::new(Exec::new(sg_data)));
}
if script_version >= &ScriptVersion::V2 {
    syscalls.append(&mut vec![
        Box::new(ExecV2::new(vm_id, vm_context)),
        ...
    ]);
}
```

Both `Exec` (`script/src/syscalls/exec.rs`) and `ExecV2` (`script/src/syscalls/exec_v2.rs`) match on the same constant `EXEC = 2043` (`script/src/syscalls/mod.rs` line 78). However, their error-handling semantics diverge critically:

**`Exec` (V1) — `exec.rs` lines 104–105:**
```rust
let source = Source::parse_from_u64(machine.registers()[A1].to_u64())?;
let place  = Place::parse_from_u64(machine.registers()[A2].to_u64())?;
```
`Source::parse_from_u64` and `Place::parse_from_u64` return `Err(VMError::External(...))` for any unrecognized value. The `?` operator propagates this as a **fatal VM error**, immediately terminating the script with a hard failure.

**`ExecV2` (V2) — `exec_v2.rs` lines 39–44:**
```rust
let data_piece_id = match DataPieceId::try_from((source, index, place)) {
    Ok(id) => id,
    Err(_) => {
        machine.set_register(A0, Mac::REG::from_u8(INDEX_OUT_OF_BOUND));
        return Ok(true);
    }
};
```
On any invalid `source`/`place` combination, `ExecV2` sets `A0 = 1` (`INDEX_OUT_OF_BOUND`) and returns `Ok(true)` — a **recoverable error** that leaves the script running.

The VM version for a script is selected in `TxInfo::select_version()` (`script/src/types.rs` lines 900–936). Scripts with `hash_type = Type` receive the **latest enabled version**: V0 before CKB2021, V1 before CKB2023, and V2 after CKB2023 activates. There is no opt-out mechanism.

The behavioral divergence is explicitly confirmed by the integration test `exec_from_witness_place_error` (`script/src/verify/tests/ckb_latest/features_since_v2021.rs` lines 1773–1782):

```rust
let from = ExecFrom::OutOfBound(0, 1, 3, 0);  // place = 3, invalid
let res = if script_version <= ScriptVersion::V1 {
    Err("Place parse_from_u64".to_string())    // fatal VM error in V1
} else {
    Err("error code 1".to_string())            // A0 = INDEX_OUT_OF_BOUND in V2
};
```

---

### Impact Explanation

A lock script deployed with `hash_type = Type` that uses syscall `2043` as a **tail call** — expecting that an invalid `place` or `source` argument causes a fatal abort — will instead **continue executing** after CKB2023 activation. If the script does not explicitly check `A0` after the `exec` call (a reasonable assumption for a tail call), execution proceeds past the intended termination point. Depending on the script's subsequent logic, this can allow an attacker to spend cells that the script was designed to protect, constituting an unauthorized fund transfer.

Additionally, `ExecV2` uses `Err(VMError::Yield)` to suspend the VM and processes the exec asynchronously via a message box, whereas `Exec` resets the machine synchronously. Scripts that relied on the synchronous reset of machine state (e.g., register values, cycle counters) will observe different post-call state in V2, further invalidating assumptions baked into V1-era scripts.

---

### Likelihood Explanation

`hash_type = Type` is the standard "upgradeable script" pattern in CKB and is used by system scripts and many application-layer contracts. Any such script that:
1. calls syscall `2043` with a potentially invalid `source` or `place` argument, and
2. does not check `A0` after the call (treating `exec` as a non-returning tail call),

is silently broken by the CKB2023 activation. The pattern is realistic because `exec` is semantically a process-replacement primitive — callers naturally assume it does not return on success, and may not guard against the case where it returns with an error code. The behavioral change is invisible at the protocol level: no transaction is rejected, no error is surfaced to the caller.

---

### Recommendation

1. **Script authors:** Scripts using `hash_type = Type` that call syscall `2043` must explicitly check `A0` after every call and treat any non-zero value as a hard failure by returning a non-zero exit code.
2. **Protocol documentation:** The CKB RFC and developer documentation should prominently warn that `hash_type = Type` scripts are subject to automatic VM version promotion and that syscall semantics may change across versions.
3. **Future hardfork design:** Future VM version upgrades should either (a) preserve error-handling semantics for existing syscall numbers, or (b) assign new syscall numbers for changed behavior, so that `hash_type = Type` scripts are not silently affected.

---

### Proof of Concept

**Setup:** Deploy a lock script `S` with `hash_type = Type` containing logic equivalent to:

```c
// Intended: exec into callee; if exec fails for any reason, abort.
// Bug: in V2, exec returns INDEX_OUT_OF_BOUND in A0 instead of aborting.
int ret = syscall(2043, index, source, /*place=*/3, bounds, argc, argv);
// Script author assumes this line is unreachable on error.
// In V1: unreachable (fatal VM error). In V2: reachable (A0 = 1).
return 0;  // Script succeeds — cell is spendable by anyone.
```

**Step 1 (V1 / pre-CKB2023):** Submit a transaction spending a cell locked by `S`. The node runs `S` in V1. `Place::parse_from_u64(3)` returns `Err(VMError::External("Place parse_from_u64 3"))`. The script fails. Transaction rejected.

**Step 2 (V2 / post-CKB2023):** Submit the same transaction. The node now runs `S` in V2 (same `hash_type = Type`, same code hash). `ExecV2` sets `A0 = INDEX_OUT_OF_BOUND` and returns. The script falls through to `return 0`. Script succeeds. Transaction accepted. Cell spent without authorization.
