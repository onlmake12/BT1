### Title
Deviation-Only Subscription Permanently Frozen When Stored Price Is Zero — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
In `Scheduler._validateShouldUpdatePrices`, when `previousPrice == 0 || currentPrice == 0`, the deviation check is silently skipped via `continue`. For a subscription configured with `updateOnDeviation: true` and `updateOnHeartbeat: false`, if all stored prices are zero (set during the first update), no subsequent keeper call can ever satisfy the deviation condition. The loop exhausts all feeds without returning, falls through to `revert SchedulerErrors.UpdateConditionsNotMet()`, and the subscription is permanently frozen.

### Finding Description
`_validateShouldUpdatePrices` in `Scheduler.sol` contains the following guard at lines 428–431:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
```

This `continue` silently skips the deviation check for any feed whose stored (`previousPrice`) or incoming (`currentPrice`) price is zero. The first update to any subscription always succeeds unconditionally:

```solidity
// If there's no previous price, this is the first update
if (previousFeed.id == bytes32(0)) {
    return updateTimestamp;
}
```

So if the first update carries `price == 0` for all feeds, those zero values are stored via `_storePriceUpdates`. Every subsequent update then hits the `continue` branch for every feed, the loop completes without returning `updateTimestamp`, and the function reverts with `UpdateConditionsNotMet`. For a deviation-only subscription (no heartbeat fallback), this is permanent: no keeper can ever push a new price, regardless of how far the market has moved.

The Pyth core contract (`parsePriceFeedUpdatesWithConfig`) does not reject `price == 0`; the `price` field is `int64` and zero is a valid encoding. The `PythPriceOracleGetter` integration contract does check `price.price <= 0`, but that check is absent from the Scheduler's update path.

### Impact Explanation
Any downstream protocol reading prices from a frozen deviation-only Scheduler subscription will receive indefinitely stale data. If the subscription covers a collateral asset, a lending protocol could allow under-collateralised borrowing or block legitimate liquidations. The subscription owner cannot recover without deploying a new subscription, losing all pre-funded balance in the frozen one.

### Likelihood Explanation
The Pyth oracle publishes `int64` prices; zero is technically valid and can appear for newly bootstrapped feeds, feeds in a halted state, or assets whose publishers all report zero during an extreme event. Any unprivileged keeper who calls `updatePriceFeeds` first with such a VAA triggers the freeze. The attacker does not need to forge Pyth data — they only need to be the first to relay a legitimately published zero-price VAA to the Scheduler.

### Recommendation
Replace the silent `continue` with one of the following:
1. **Treat zero-previous-price as infinite deviation** — always return `updateTimestamp` when `previousPrice == 0`, so the subscription can always be updated out of a zero state.
2. **Reject zero prices at storage time** — add a check in `_storePriceUpdates` that reverts if `price.price == 0`, preventing the degenerate state from being stored in the first place.
3. **Require heartbeat alongside deviation** — document and enforce that deviation-only subscriptions are unsupported; require `updateOnHeartbeat: true` as a safety net.

### Proof of Concept
1. Deploy `Scheduler` and create a subscription with `updateOnDeviation: true`, `updateOnHeartbeat: false`, `deviationThresholdBps: 100`, for a single price feed `F`.
2. Obtain a valid Pyth VAA for feed `F` with `price.price == 0` (e.g., from a newly bootstrapped feed).
3. Call `scheduler.updatePriceFeeds(subscriptionId, updateData)`. This succeeds because `previousFeed.id == bytes32(0)` triggers the first-update fast-path. `_storePriceUpdates` stores `price.price = 0`.
4. Obtain a second valid Pyth VAA for feed `F` with `price.price == 10_000` (a 100 % move, far above the threshold).
5. Call `scheduler.updatePriceFeeds(subscriptionId, updateData2)`. Inside `_validateShouldUpdatePrices`: `previousPrice = 0` → `continue` → loop ends → `revert UpdateConditionsNotMet()`.
6. Repeat step 5 with any non-zero price: the result is always `UpdateConditionsNotMet`. The subscription is permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2)
