### Title
Permanent Locking of All Protocol Update Fees — No On-Chain Withdrawal Path for `Coin<SUI>` Stored in `PriceInfoObject` - (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every SUI fee paid via `update_single_price_feed` is deposited into the corresponding `PriceInfoObject` under `FEE_STORAGE_KEY` and can never be extracted. No function in `price_info.move`, `pyth.move`, or any governance module calls `dynamic_object_field::remove` or any equivalent split/withdraw on that key. The `fee_recipient_address` stored in `State` is a dead field — it is never used in any transfer.

### Finding Description
`price_info::deposit_fee_coins` (public, line 105) uses `dynamic_object_field::add` / `coin::join` to accumulate `Coin<SUI>` inside each `PriceInfoObject`. The matching removal path — `dynamic_object_field::remove<vector<u8>, Coin<SUI>>(&mut price_info_object.id, FEE_STORAGE_KEY)` — does not exist anywhere in the production codebase. The governance module `set_fee_recipient` only writes an `address` into `State`; it performs no coin movement. There is no privileged or unprivileged transaction sequence that can reduce the balance returned by `get_balance`.

### Impact Explanation
All SUI paid as update fees by every caller of `update_single_price_feed` is permanently locked inside the respective `PriceInfoObject` shared objects. The protocol can never collect its own revenue. The `fee_recipient_address` governance mechanism is entirely non-functional with respect to actual fee collection. The invariant `sum(get_balance(pio) for all pio)` is strictly monotonically increasing and irreversible.

### Likelihood Explanation
This triggers on every single call to `update_single_price_feed` — the primary production entry point for price feed consumers. No special attacker action is required; the normal protocol flow causes the permanent lock. The impact accumulates with every price update across every price feed.

### Recommendation
Add a privileged withdrawal function in `price_info.move` (friend of `pyth::pyth` or a new governance module) that calls `dynamic_object_field::remove<vector<u8>, Coin<SUI>>(&mut price_info_object.id, FEE_STORAGE_KEY)` and transfers the extracted coin to `state::get_fee_recipient(pyth_state)`. Gate it behind `LatestOnly` and require the caller to pass the `PythState` so the recipient address is authoritative. Alternatively, route fees directly to the `fee_recipient_address` at deposit time inside `update_single_price_feed` using `transfer::public_transfer`, bypassing storage in `PriceInfoObject` entirely.

module pyth::set_fee_recipient {

    use wormhole::external_address::{Self};

    use pyth::state::{Self, State, LatestOnly};

    friend pyth::governance;

    struct PythFeeRecipient {
        recipient: address
    }

    public(friend) fun execute(latest_only: &LatestOnly, state: &mut State, payload: vector<u8>) {
        let PythFeeRecipient { recipient } = from_byte_vec(payload);
        state::set_fee_recipient(latest_only, state, recipient);
    }

    fun from_byte_vec(payload: vector<u8>): PythFeeRecipient {
        let cur = cursor::new(payload);

        // Recipient must be non-zero address.
        let recipient = external_address::take_nonzero(&mut cur);

        cursor::destroy_empty(cur);

        PythFeeRecipient {
            recipient: external_address::to_address(recipient)
        }
    }
