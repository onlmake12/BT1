The vulnerability is **real and concrete**:

- In `PostUpdate`, `payer` and `write_authority` are **separate signers** — the `payer` funds the rent-exempt deposit, while `write_authority` is stored in `price_update_account.write_authority`. These are explicitly allowed to be different parties.
- In `ReclaimRent`, the constraint forces `payer.key() == price_update_account.write_authority`, meaning the signer must be the `write_authority`. The `close = payer` directive then sends **all lamports** (including the original rent deposit) to this signer.
- There is **no check** that the `ReclaimRent` `payer` is the same account that originally funded the `init_if_needed` in `PostUpdate`. The original depositor's address is never stored in the account.

The exploit path is direct and requires no privilege:

1. `payer_A` calls `post_update` with `write_authority = write_auth_B`, paying `PriceUpdateV2::LEN` rent.
2. `write_auth_B` calls `reclaim_rent` passing itself as `payer`.
3. Constraint passes (`write_auth_B == price_update_account.write_authority`), account closes, all lamports go to `write_auth_B`.
4. `payer_A` loses their rent deposit with no recourse.

### Title
Write authority can drain original payer's rent deposit via `reclaim_rent` — (`target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

### Summary
The `ReclaimRent` and `ReclaimTwapRent` account structs use `close = payer` where `payer` is constrained to be the `write_authority`. Because `PostUpdate` allows a separate `payer` to fund account creation and a separate `write_authority` to be stored in the account, the `write_authority` can unilaterally close the account and receive the original payer's rent deposit.

### Finding Description
In `PostUpdate`, the `payer` field funds the `init_if_needed` account creation (paying `PriceUpdateV2::LEN` lamports as rent), while `write_authority` is a separate signer whose pubkey is stored in `price_update_account.write_authority`. [1](#0-0) 

In `ReclaimRent`, the constraint `price_update_account.write_authority == payer.key()` forces the signer to be the `write_authority`, and `close = payer` sends all lamports to that same signer. The original depositor's pubkey is never recorded in the account, so there is no way to enforce that rent is returned to the correct party. [2](#0-1) 

The same flaw exists in `ReclaimTwapRent`. [3](#0-2) 

### Impact Explanation
Any `write_authority` that did not pay for account creation can steal the original payer's rent deposit (the full `PriceUpdateV2::LEN`-sized rent-exempt balance) by calling `reclaim_rent`. This is a direct, permanent transfer of lamports from the original payer to the `write_authority` without consent. The original payer has no mechanism to recover their funds.

### Likelihood Explanation
The scenario where `payer != write_authority` is an explicitly supported and documented use case (the comment on line 341–342 describes the `write_authority` constraint without any assumption that payer and write_authority are the same). Any integrator or protocol that pays for account creation on behalf of users while setting a user-controlled `write_authority` is immediately vulnerable.

### Recommendation
Store the original `payer`'s pubkey inside `PriceUpdateV2` at initialization time (e.g., a `payer` field). In `ReclaimRent`, change the `close` target and the constraint to require that the signer is the stored original payer, not the `write_authority`. Alternatively, restrict `PostUpdate` so that `payer` and `write_authority` must always be the same account.

### Proof of Concept
```
1. payer_A calls post_update(write_authority=write_auth_B):
   - price_update_account is created, funded with ~2000 lamports by payer_A
   - price_update_account.write_authority = write_auth_B

2. write_auth_B calls reclaim_rent(payer=write_auth_B, price_update_account=<same account>):
   - Constraint: price_update_account.write_authority == write_auth_B.key() ✓
   - close = payer: all lamports transferred to write_auth_B ✓

3. Result:
   - payer_A loses ~2000 lamports (rent deposit)
   - write_auth_B gains ~2000 lamports
   - No consent from payer_A required
```
