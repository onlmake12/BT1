### Title
`getPricesNoOlderThan` Staleness Check Uses Subscription-Level Maximum Timestamp Instead of Per-Feed Timestamp — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol`'s `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions validate freshness using `status.priceLastUpdatedAt`, which is the **maximum** `publishTime` across all feeds in the last update batch — not the individual feed's own `publishTime`. A subscription containing a mix of active-market and closed-market feeds will have its subscription-level timestamp driven by the fresh active-market feed, while the closed-market feed's stale price passes the check unchallenged. Consumers calling `getPricesNoOlderThan` receive stale per-feed prices while believing all returned prices satisfy the requested age bound.

---

### Finding Description

**Root cause — how `priceLastUpdatedAt` is set:**

In `updatePriceFeeds`, the call to `parsePriceFeedUpdatesWithConfig` explicitly passes `minAllowedPublishTime = 0`, permitting arbitrarily old per-feed timestamps:

```solidity
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0,   // ← no lower bound on individual feed timestamps
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
    ...
);
``` [1](#0-0) 

`_validateShouldUpdatePrices` then computes `updateTimestamp` as the **maximum** `publishTime` across all feeds in the batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [2](#0-1) 

This maximum is stored as the subscription-level staleness anchor:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [3](#0-2) 

**Root cause — how `getPricesNoOlderThan` checks staleness:**

`getPricesNoOlderThan` checks only the subscription-level anchor, then returns all individual prices without any per-feed timestamp validation:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [4](#0-3) 

`getEmaPricesNoOlderThan` has the identical pattern: [5](#0-4) 

`getPricesUnsafe` returns the stored `PriceFeed.price` struct directly with no recency check: [6](#0-5) 

The design comment in `_validateShouldUpdatePrices` acknowledges that closed markets will carry old timestamps and that the slot check is relied upon for synchronization: [7](#0-6) 

However, the slot check only guarantees that all feeds came from the **same Pythnet update cycle** — it does not bound how old any individual feed's `publishTime` is. A closed-market feed's `publishTime` can be days old even when it shares a slot with a fresh crypto feed.

---

### Impact Explanation

A consumer protocol that calls `getPricesNoOlderThan(subscriptionId, [EQUITY_PRICE_ID], 60)` expects every returned price to be at most 60 seconds old. Because the staleness gate is subscription-level (driven by the freshest feed in the batch), the call succeeds and returns the equity price whose `publishTime` may be 48+ hours old (last trading session before a weekend or holiday). Any financial logic downstream — collateral valuation, liquidation threshold checks, borrow limit calculations — operates on a materially stale price while the protocol believes it is fresh. This is the same class of harm as M-17: incorrect credit/liquidation decisions caused by undetected price staleness.

---

### Likelihood Explanation

Subscriptions mixing crypto feeds (24/7) with traditional-market feeds (equities, commodities, FX) are a natural and documented use case for the Scheduler. Every weekend, every market holiday, and every overnight session produces exactly this condition: the crypto feed is fresh, the traditional-market feed is stale, and `priceLastUpdatedAt` is set to the crypto feed's timestamp. Any unprivileged keeper calling `updatePriceFeeds` with such a batch triggers the condition. No special access or key compromise is required.

---

### Recommendation

Replace the subscription-level staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed check against each returned price's own `publishTime`:

```solidity
function getPricesNoOlderThan(
    uint256 subscriptionId,
    bytes32[] calldata priceIds,
    uint256 age_seconds
) external view override onlyWhitelistedReader(subscriptionId)
  returns (PythStructs.Price[] memory prices)
{
    prices = this.getPricesUnsafe(subscriptionId, priceIds);
    for (uint i = 0; i < prices.length; i++) {
        if (distance(block.timestamp, prices[i].publishTime) > age_seconds)
            revert PythErrors.StalePrice();
    }
}
```

If the intentional design is to allow closed-market feeds to carry old timestamps, the function should be renamed (e.g., `getPricesNoNewerBatchThan`) and its documentation must clearly state that individual feed timestamps are **not** bounded by `age_seconds`, so consumers are not misled.

---

### Proof of Concept

1. A subscription is created with two price IDs: `BTC_USD` (crypto, 24/7) and `AAPL_USD` (equity, closed on weekends).
2. On Saturday, a keeper calls `updatePriceFeeds` with a Pyth update blob. `BTC_USD` has `publishTime = now` (Saturday 10:00 UTC). `AAPL_USD` has `publishTime = Friday 20:00 UTC` (last close, ~14 hours ago). Both share the same Pythnet slot, so the slot check passes. `_validateShouldUpdatePrices` sets `updateTimestamp = now` (BTC's timestamp). `status.priceLastUpdatedAt = now`.
3. A consumer DeFi protocol calls `getPricesNoOlderThan(subscriptionId, [AAPL_USD], 60)` to price AAPL collateral.
4. `distance(block.timestamp, status.priceLastUpdatedAt)` ≈ 0 < 60 → check passes, no revert.
5. The returned `AAPL_USD` price has `publishTime = Friday 20:00 UTC`, i.e., ~14 hours stale.
6. The protocol uses this stale AAPL price to compute collateral value, potentially allowing under-collateralized borrows or blocking legitimate liquidations — the same harm described in M-17.
