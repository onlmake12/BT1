The `Initialize` struct and function are the key focus. The analysis is straightforward from the code:

**`initialize` function** (line 48–56): accepts any `Signer` as `payer` with no authority check.

**`Initialize` accounts struct** (lines 298–304): `payer` is only constrained to be a `Signer` — there is no constraint tying it to a deployer, upgrade authority, or any pre-authorized key. The `config` account uses `init` (line 301), which prevents double-initialization but does **not** prevent a race to be first.

### Title
Permissionless `initialize` allows any caller to seize governance authority — (`target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

### Summary
The `initialize` instruction has no access control. Any unprivileged account can call it before the legitimate deployer and supply an attacker-controlled `governance_authority`, permanently seizing governance over the receiver program.

### Finding Description
`initialize` accepts an arbitrary `initial_config: Config` and writes it directly to the config PDA with no check that the caller is the program's upgrade authority or any other trusted key. [1](#0-0) 

The `Initialize` accounts struct requires only a generic `Signer` as `payer`: [2](#0-1) 

Anchor's `init` constraint on `config` (line 301) enforces that the PDA is created exactly once, but it provides no protection against *who* creates it first. The first caller wins unconditionally.

### Impact Explanation
`governance_authority` controls every privileged instruction: `set_data_sources`, `set_wormhole_address`, `set_fee`, `set_minimum_signatures`, and `request_governance_authority_transfer`. [3](#0-2) 

An attacker who front-runs `initialize` with their own pubkey as `governance_authority` gains complete, irrevocable control over all of these. The legitimate deployer cannot re-initialize (Anchor's `init` will return `AccountAlreadyInitialized`), and cannot reclaim governance without the attacker's cooperation.

### Likelihood Explanation
Solana does not have a public mempool, but the program deployment transaction and the `initialize` call are necessarily separate transactions. Any on-chain observer can detect the program deployment (via program account creation) and immediately submit their own `initialize` in the next block. No special privileges, leaked keys, or off-chain coordination are required — only a funded wallet and knowledge of the program ID.

### Recommendation
Bind `initialize` to the program's upgrade authority. Anchor exposes this via the `BpfUpgradeableLoader` account:

```rust
#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(mut, constraint = payer.key() == program_data.upgrade_authority_address.ok_or(ReceiverError::NoUpgradeAuthority)? @ ReceiverError::UnauthorizedInitializer)]
    pub payer: Signer<'info>,
    #[account(seeds = [program_id.key().as_ref()], bump, seeds::program = bpf_loader_upgradeable::ID)]
    pub program_data: Account<'info, ProgramData>,
    // ...
}
```

This ensures only the current upgrade authority can call `initialize`, matching the intended deployment invariant.

### Proof of Concept
1. Deploy `pyth-solana-receiver` (program account created, `initialize` not yet called).
2. Attacker submits `initialize(Config { governance_authority: attacker_pubkey, valid_data_sources: [...], wormhole: <real_wormhole>, minimum_signatures: 1, single_update_fee_in_lamports: 0 })` — any valid `Config` suffices.
3. Config PDA is created with `governance_authority = attacker_pubkey`.
4. Legitimate deployer's `initialize` call returns `AccountAlreadyInitialized`.
5. All subsequent `Governance`-gated calls (`set_data_sources`, `set_wormhole_address`, etc.) require `payer.key() == config.governance_authority` — which is now the attacker. [4](#0-3)
