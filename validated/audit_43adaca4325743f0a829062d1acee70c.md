### Title
Transaction Suppression via Premature `mark_as_known_txs` Before Verification — (`sync/src/relayer/transactions_process.rs`)

### Summary
An unprivileged remote peer that is the `requesting_peer` for a tx hash can send a structurally valid but semantically invalid `RelayTransactions` message. The node marks the hash as known (removing all peer announcements from `unknown_tx_hashes`) before verification completes. On rejection, only `tx_filter` is cleaned up — the hash is never re-queued for fetching from alternative announcing peers.

### Finding Description

In `TransactionsProcess::execute`, line 76 calls `mark_as_known_txs` synchronously before spawning the async `submit_remote_tx` task: [1](#0-0) 

`mark_as_known_txs` holds both locks simultaneously and performs two operations atomically: [2](#0-1) 

This removes the hash from `unknown_tx_hashes` — destroying the full list of announcing peers (including peer B) — and inserts it into `tx_filter`.

When `submit_remote_tx` rejects the tx, `TxVerificationResult::Reject` is processed in `send_bulk_of_tx_hashes`: [3](#0-2) 

`remove_from_known_txs` only removes from `tx_filter`: [4](#0-3) 

After this, the hash is absent from both `tx_filter` (cleaned up) and `unknown_tx_hashes` (never restored). Peer B's announcement — which was stored in the `peers` vec of the `UnknownTxHashPriority` entry — is permanently discarded. [5](#0-4) 

The `requesting_peer` check in the filter confirms the attacker must be `peers.first()` with `requested == true`: [6](#0-5) [7](#0-6) 

### Impact Explanation

After the attacker's invalid tx is rejected:
- The hash is not in `tx_filter` → the node does not consider it "known"
- The hash is not in `unknown_tx_hashes` → no re-fetch is scheduled for peer B
- Peer B will not re-announce (standard relay behavior: announce once, don't repeat)
- The victim node never fetches the valid tx from peer B

The suppression persists until peer B independently re-announces the hash (not guaranteed) or a third peer announces it. For a targeted transaction (e.g., a time-sensitive contract call or a miner's own pending tx), this can cause indefinite exclusion from the victim's mempool.

### Likelihood Explanation

The preconditions are achievable by any unprivileged peer:
1. Connect to the victim node
2. Announce the target tx hash via `RelayTransactionHashes` before other peers (or be selected as `requesting_peer` via `pop_ask_for_txs`)
3. When the node sends `GetRelayTransactions`, respond with a tx body that has the correct hash but fails semantic validation (e.g., capacity overflow, invalid witness, bad script)

No special privileges, keys, or majority hashpower are required.

### Recommendation

Move `mark_as_known_txs` to **after** successful verification, not before. On `TxVerificationResult::Reject`, restore the remaining peer announcements to `unknown_tx_hashes` so the node can re-fetch from alternative peers. Alternatively, on rejection, re-insert the hash into `unknown_tx_hashes` with the remaining peers (excluding the rejecting peer) rather than simply removing from `tx_filter`.

### Proof of Concept

1. Peer A (attacker) connects to victim node and announces `tx_hash` via `RelayTransactionHashes`
2. Peer B (honest) also announces `tx_hash` — both are now in `unknown_tx_hashes.peers` for that hash
3. Victim's `pop_ask_for_txs` selects peer A as `requesting_peer` (first announcer), sends `GetRelayTransactions`
4. Peer A responds with a `RelayTransactions` message: correct `tx_hash`, but tx body has e.g. output capacity > input capacity
5. Victim calls `mark_as_known_txs` → hash removed from `unknown_tx_hashes` (peer B's slot gone), inserted into `tx_filter`
6. `submit_remote_tx` rejects → `TxVerificationResult::Reject` → `remove_from_known_txs` removes from `tx_filter`
7. Observe: victim never sends `GetRelayTransactions` to peer B; peer B is never queried for the tx
