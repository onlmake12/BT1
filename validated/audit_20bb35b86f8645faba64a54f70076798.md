### Title
Non-Atomic Read-Then-Withdraw Race Condition in Fortuna Keeper Fee Withdrawal — (File: `apps/fortuna/src/keeper/fee.rs`, `apps/fortuna/src/command/withdraw_fees.rs`)

### Summary
The Fortuna keeper's fee withdrawal logic reads `accrued_fees_in_wei` from the Entropy contract as a stale off-chain snapshot, computes an exact withdrawal amount from that snapshot, and then submits a transaction for that exact amount. In multi-replica deployments (explicitly documented and supported), two keeper instances can concurrently read the same stale balance, compute overlapping withdrawal amounts, and race to submit — causing one transaction to revert with "Insufficient balance" and leaving keeper wallets underfunded.

### Finding Description

In `withdraw_fees_if_necessary` (`apps/fortuna/src/keeper/fee.rs`):

```rust
// Step 1: stale read of on-chain balance
let provider_info = contract_as_fee_manager
    .get_provider_info_v2(provider_address)
    .call()
    .await?;
let available_fees = U256::from(provider_info.accrued_fees_in_wei);  // snapshot

// Step 2: compute withdrawal from stale snapshot
let withdrawal_amount = calculate_fair_fee_withdrawal_amount(
    Arc::new(provider.clone()),
    keeper_address,
    &other_keeper_addresses,
    available_fees,   // stale
).await?;

// Step 3: submit exact amount — no re-check of current balance
let contract_call = contract_as_fee_manager
    .withdraw_as_fee_manager(provider_address, withdrawal_amount.as_u128());
send_and_confirm(contract_call).await?;
``` [1](#0-0) 

The same pattern exists in `withdraw_fees_for_chain`:

```rust
let fees = provider_info.accrued_fees_in_wei;  // stale read
let withdrawal_amount_wei = fees.saturating_sub(retained_balance);
// ... submits exact withdrawal_amount_wei without re-checking
``` [2](#0-1) 

The multi-replica setup is explicitly documented and supported. The README shows both replicas configured with the same `fee_manager_private_key`, and `keeper.rs` spawns a `withdraw_fees_wrapper` loop per replica: [3](#0-2) 

The `calculate_fair_fee_withdrawal_amount` function compounds the staleness by also reading all keeper wallet balances at different points in time, making the computed `withdrawal_amount` doubly stale by the time the transaction is submitted: [4](#0-3) 

The on-chain `withdrawAsFeeManager` enforces `accruedFeesInWei >= amount` and reverts if not satisfied: [5](#0-4) 

### Impact Explanation

When two Fortuna replicas concurrently execute `withdraw_fees_if_necessary`:

1. Both read `accrued_fees_in_wei = F` at approximately the same time.
2. Both compute `withdrawal_amount = F/2` (fair share for 2 keepers).
3. Replica A's transaction lands first, reducing on-chain balance to `F/2`.
4. Replica B's transaction attempts to withdraw `F/2` from a balance of `F/2` — this succeeds, draining the full balance. Both keepers withdraw their fair share, but the total withdrawn is `F`, which is correct.

However, in the `withdraw_fees_for_chain` CLI command (which withdraws `fees - retained_balance`, i.e., nearly all fees):

1. Both instances read `fees = F`.
2. Both compute `withdrawal_amount = F - retained`.
3. Replica A withdraws `F - retained`, leaving `retained` in the contract.
4. Replica B attempts to withdraw `F - retained` from a balance of `retained` → **revert**.

The revert is caught and logged but not retried immediately: [6](#0-5) 

Repeated reverts waste gas and leave keeper wallets underfunded. An underfunded keeper cannot pay gas for entropy fulfillment callbacks, causing Entropy requests to go unfulfilled — a liveness failure for the Entropy service.

### Likelihood Explanation

**Medium.** The multi-replica setup is explicitly documented in the README as the production high-availability configuration:



The sample config shows both replicas with `fee_manager_private_key` configured. The `withdraw_fees_wrapper` loop runs on a fixed `WITHDRAW_INTERVAL`, making concurrent execution likely when two replicas are running. Additionally, the `withdraw_fees` CLI command can be run manually at any time, racing with the background keeper loop.

### Recommendation

1. **Use `saturating_sub` or re-read balance before withdrawal**: After computing `withdrawal_amount`, re-read `accrued_fees_in_wei` immediately before submitting and use `min(withdrawal_amount, current_balance - retained)` to avoid over-withdrawal.
2. **Catch and handle "Insufficient balance" reverts gracefully**: Distinguish between a revert due to race condition (retry with fresh balance) vs. a genuine error.
3. **Ensure only one replica runs fee management**: The README note "Only enable fee management threads on the instance using the fee manager wallet" should be enforced programmatically, not just by documentation.
4. **Use `withdraw_as_fee_manager` with a max-available pattern**: Instead of withdrawing an exact stale amount, withdraw `min(desired, current_accrued)` by re-reading the balance atomically just before the call.

### Proof of Concept

**Setup**: Two Fortuna replicas (Replica 0, Replica 1) both configured with the same `fee_manager_private_key`. Contract has `accrued_fees_in_wei = 1 ETH`. Both replicas' `withdraw_fees_wrapper` loops fire simultaneously.

**Execution**:
```
Replica 0: get_provider_info_v2() → accrued_fees = 1 ETH
Replica 1: get_provider_info_v2() → accrued_fees = 1 ETH  (same stale read)

Replica 0: calculate_fair_fee_withdrawal_amount() → withdrawal = 0.5 ETH
Replica 1: calculate_fair_fee_withdrawal_amount() → withdrawal = 0.5 ETH

Replica 0: withdraw_as_fee_manager(provider, 0.5 ETH) → SUCCESS, balance = 0.5 ETH
Replica 1: withdraw_as_fee_manager(provider, 0.5 ETH) → SUCCESS (0.5 ETH available)
```

In the `withdraw_fees_for_chain` CLI scenario with `retained_balance = 0`:
```
CLI instance A: fees = 1 ETH, withdrawal = 1 ETH
CLI instance B: fees = 1 ETH, withdrawal = 1 ETH

Instance A: withdraw_as_fee_manager(provider, 1 ETH) → SUCCESS, balance = 0
Instance B: withdraw_as_fee_manager(provider, 1 ETH) → REVERT "Insufficient balance"
```

Instance B's error is logged at `tracing::error!` level and the loop continues, but the keeper wallet for Instance B is not topped up. If this pattern repeats, Instance B's keeper wallet is depleted and it can no longer pay gas for entropy callbacks.
