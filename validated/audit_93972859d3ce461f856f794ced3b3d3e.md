### Title
Excess `msg.value` Not Refunded to Caller in Entropy Fee Accounting â€” (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` internal function in `Entropy.sol` uses a non-strict `>=` check when validating the fee paid by the caller. Any `msg.value` exceeding the required fee is silently absorbed into `_state.accruedPythFeesInWei` rather than being returned to the caller. This is an exact structural analog to the Chainport `_depositTokens()` / `burnTokens()` pattern described in the external report.

---

### Finding Description

In `Entropy.sol`, `requestHelper` is the shared fee-accounting path for all public entry points (`request`, `requestWithCallback`, `requestV2`):

```solidity
// Entropy.sol lines 234â€“239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The fee split is:
- Provider receives exactly `providerFee`.
- Pyth receives `msg.value - providerFee` â€” which includes **all excess** beyond `requiredFee`.

If a caller sends `requiredFee + X`, the surplus `X` is permanently credited to `accruedPythFeesInWei`. There is no refund path. The `IEntropy` and `IEntropyV2` interfaces document this explicitly:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

The same pattern exists in `Echo.sol`'s `requestPriceUpdatesWithCallback`:

```solidity
// Echo.sol lines 75â€“84
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
...
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [4](#0-3) 

Here, any excess beyond `pythFeeInWei` is stored in `req.fee` and later credited entirely to the provider in `executeCallback`, not returned to the requester. [5](#0-4) 

---

### Impact Explanation

Any caller who sends `msg.value > getFee(provider)` (or `getFeeV2`) permanently loses the surplus ETH. The excess is irrecoverably credited to Pyth's or the provider's accrued fee balance. There is no mechanism for the original sender to reclaim it. This constitutes a direct, permanent loss of user funds for every overpaying transaction.

---

### Likelihood Explanation

Realistic overpayment scenarios include:

1. **Fee increase between query and submission**: A caller queries `getFee()` at block N, the provider raises their fee at block N+1, and the caller submits with the old (now-insufficient) value plus a safety buffer â€” the buffer is lost.
2. **Defensive buffering**: Callers or integrators who add a small ETH buffer to avoid `InsufficientFee` reverts (a common pattern) silently donate the buffer to Pyth.
3. **Hardcoded fee values**: Any integration that hardcodes a fee value larger than the exact current fee will lose the difference on every call.

The `IEntropyV2` NatSpec explicitly warns callers to recompute the fee before each invocation precisely because fees can change, making scenario 1 a realistic operational condition. [6](#0-5) 

---

### Recommendation

Either:
1. **Strict equality check**: Replace `msg.value < requiredFee` with `msg.value != requiredFee` to revert on overpayment, or
2. **Refund excess**: After crediting fees, return `msg.value - requiredFee` to `msg.sender`:
   ```solidity
   if (msg.value > requiredFee) {
       payable(msg.sender).transfer(msg.value - requiredFee);
   }
   ```
   (as already implemented correctly in `PythLazer.sol`'s `verifyUpdate`) [7](#0-6) 

---

### Proof of Concept

```solidity
// Attacker/user calls request() with excess fee
uint128 fee = entropy.getFee(provider);          // e.g., 1000 wei
entropy.request{value: fee + 500}(provider, commitment, false);
// Result: provider gets providerFee, Pyth gets pythFee + 500
// User loses 500 wei permanently â€” no refund issued
// Confirmed by: accruedPythFeesInWei += (msg.value - providerFee)
``` [8](#0-7) 

The test suite itself confirms this: `testFees()` in `Entropy.t.sol` explicitly calls `requestWithFee(user2, pythFeeInWei + provider2FeeInWei + 10000, ...)` and then asserts `random.getAccruedPythFees() == pythFeeInWei * 5 + 10000`, confirming the 10000 wei surplus is absorbed into Pyth fees rather than returned. [9](#0-8)
