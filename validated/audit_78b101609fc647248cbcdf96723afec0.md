### Title
Unbounded Linear Scan Over `trusted_signers` in `verify_message` Blocks All Lazer Price Verification on Aptos - (File: `lazer/contracts/aptos/sources/pyth_lazer.move`)

---

### Summary

The Aptos Pyth Lazer contract stores trusted signers in an unbounded `vector<TrustedSignerInfo>` and performs a full linear scan over it inside `verify_message`, which is the critical path for every Lazer price update verification. There is no cap on the number of entries that can be added. If the vector grows large enough, the loop exceeds Aptos transaction gas limits and permanently DoS-es all Lazer price verification on Aptos.

---

### Finding Description

In `lazer/contracts/aptos/sources/pyth_lazer.move`, the `Storage` struct holds an unbounded vector:

```move
struct Storage has key {
    treasury: address,
    single_update_fee: u64,
    trusted_signers: vector<TrustedSignerInfo>   // no cap
}
``` [1](#0-0) 

The public function `verify_message` (and its entry-function wrapper `verify_message_with_funder`) performs a full linear scan over this vector on every call:

```move
let i = 0;
let valid = false;
while (i < storage.trusted_signers.length()) {
    let signer_info = vector::borrow(&storage.trusted_signers, (i as u64));
    if (&signer_info.pubkey == &trusted_signer
        && signer_info.expires_at > timestamp::now_seconds()) {
        valid = true;
        break
    };
    i = i + 1;
};
assert!(valid, EINVALID_SIGNER);
``` [2](#0-1) 

The `update_trusted_signer` admin function appends new entries with `vector::push_back` and has no upper-bound check:

```move
vector::push_back(
    &mut storage.trusted_signers,
    TrustedSignerInfo { pubkey: trusted_signer, expires_at }
);
``` [3](#0-2) 

This is in direct contrast to the EVM Lazer contract, which uses a fixed-size array `TrustedSignerInfo[100]` combined with an O(1) mapping lookup (`trustedSignerToExpiresAtMapping`) in `verifyUpdate`, making the EVM version immune to this class of issue: [4](#0-3) [5](#0-4) 

The Sui Lazer contract has the same unbounded-vector pattern via `find_index!` in `verify_le_ecdsa_message`, also with no cap enforced in `state::update_trusted_signer`: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

If `trusted_signers` grows to a size where the linear scan in `verify_message` exceeds the Aptos per-transaction gas limit, every call to `verify_message` and `verify_message_with_funder` will abort. Since these are the sole verification entry points for Lazer price updates on Aptos, all downstream consumers of Lazer prices on Aptos are permanently blocked until the contract is upgraded or signers are removed. This is a complete DoS of the Lazer price feed on Aptos.

---

### Likelihood Explanation

`update_trusted_signer` requires `AdminCapability`: [8](#0-7) 

The admin is the necessary actor. However, the risk is realistic through legitimate operational use: key rotation cycles (add new signer, then remove old) that are not perfectly cleaned up, or accumulation of expired-but-not-removed entries over time. Because there is no enforced cap, the contract has no protection against gradual growth. The EVM contract's design (fixed array of 100 + O(1) mapping) shows the Pyth team is aware of the need to bound this structure on other chains, making the omission on Aptos a concrete gap.

---

### Recommendation

1. Enforce a small hard cap (e.g., `MAX_TRUSTED_SIGNERS = 10`) inside `update_trusted_signer` before calling `vector::push_back`, aborting if the cap would be exceeded.
2. Mirror the EVM design: replace the linear scan in `verify_message` with a `Table` (Aptos equivalent of a mapping) keyed by pubkey for O(1) lookup, keeping the vector only for enumeration.
3. Add a cleanup path that removes expired signers to prevent unbounded accumulation.

---

### Proof of Concept

1. Admin calls `update_trusted_signer` N times with distinct pubkeys and far-future `expires_at` values, accumulating N entries in `storage.trusted_signers`.
2. Any user calls `verify_message_with_funder` with a valid signer pubkey that happens to be at position N-1 in the vector (worst case: the valid signer was added last).
3. The `while` loop in `verify_message` iterates N times before finding the match.
4. At sufficiently large N (dependent on Aptos gas schedule per `vector::borrow` and comparison), the transaction aborts with an out-of-gas error.
5. All subsequent calls to `verify_message` / `verify_message_with_funder` also abort, permanently blocking Lazer price verification on Aptos. [9](#0-8)

    public fun verify_message(
        message: vector<u8>,
        signature: vector<u8>,
        trusted_signer: vector<u8>,
        fee: coin::Coin<AptosCoin>
    ) acquires Storage {
        let storage = borrow_global<Storage>(@pyth_lazer);

        // Verify fee amount
        assert!(coin::value(&fee) >= storage.single_update_fee, EINSUFFICIENT_FEE);

        // Transfer fee to treasury
        coin::deposit(storage.treasury, fee);

        // Verify signer is trusted and not expired
        let i = 0;
        let valid = false;
        while (i < storage.trusted_signers.length()) {
            let signer_info = vector::borrow(&storage.trusted_signers, (i as u64));
            if (&signer_info.pubkey == &trusted_signer
                && signer_info.expires_at > timestamp::now_seconds()) {
                valid = true;
                break
            };
            i = i + 1;
        };
        assert!(valid, EINVALID_SIGNER);

        // Verify signature
        let sig = ed25519::new_signature_from_bytes(signature);
        let pk = ed25519::new_unvalidated_public_key_from_bytes(trusted_signer);
        assert!(
            ed25519::signature_verify_strict(&sig, &pk, message),
            EINVALID_SIGNATURE
        );
    }
