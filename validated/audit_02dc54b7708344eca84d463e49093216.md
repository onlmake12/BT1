### Title
Incorrect Provider Address in `PriceUpdateExecuted` Event Emission — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, the `emitPriceUpdate` internal function emits `msg.sender` as the `provider` field of the `PriceUpdateExecuted` event, while the actual provider being credited with fees is `providerToCredit` — a distinct, caller-supplied parameter. Any relayer or bot can call `executeCallback` with an arbitrary `providerToCredit`, making `msg.sender ≠ providerToCredit` the normal case. Off-chain systems indexing this event will record the wrong provider for every fulfilled request.

### Finding Description
`executeCallback` accepts `providerToCredit` as an explicit parameter and credits that address's `accruedFeesInWei`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [1](#0-0) 

It then calls `emitPriceUpdate`, which emits `msg.sender` — the transaction sender — as the `provider` field:

```solidity
emit PriceUpdateExecuted(
    sequenceNumber,
    msg.sender,   // ← should be providerToCredit
    priceIds, prices, conf, expos, publishTimes
);
``` [2](#0-1) 

The event definition declares this second field as `address indexed provider`: [3](#0-2) 

`msg.sender` is whoever submits the transaction (a relayer, keeper, or any third party). `providerToCredit` is the registered provider whose balance is actually incremented. These are structurally different roles and will routinely differ.

For comparison, the analogous `withdrawAsFeeManager` in `Echo.sol` has a related but distinct issue: it emits `FeesWithdrawn(msg.sender, amount)` without including the `provider` whose fees were drained, making it impossible to attribute the withdrawal to the correct provider from logs alone: [4](#0-3) 

The Entropy contract correctly handles the same pattern by emitting both `provider` and `msg.sender` in its `Withdrawal` event: [5](#0-4) 

### Impact Explanation
Off-chain systems (indexers, dashboards, the Fortuna keeper) that listen to `PriceUpdateExecuted` to determine which provider fulfilled a request will record the wrong provider address. Provider reputation tracking, fee attribution audits, and SLA monitoring will all be corrupted. The `provider` field is `indexed`, meaning it is used as a filter key in event queries — queries for `PriceUpdateExecuted` filtered by a specific provider address will return incorrect or missing results.

### Likelihood Explanation
This is triggered on every `executeCallback` invocation where the caller (`msg.sender`) differs from `providerToCredit`. Because `executeCallback` is a permissionless function callable by any relayer or keeper, and because the exclusivity period allows any party to call it after expiry, `msg.sender ≠ providerToCredit` is the expected common case in production. No special privileges or attack setup are required. [6](#0-5) 

### Recommendation
Pass `providerToCredit` instead of `msg.sender` to `emitPriceUpdate`, or add `providerToCredit` as a parameter to `emitPriceUpdate` and use it in the event:

```solidity
// In executeCallback, replace:
emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
// With:
emitPriceUpdate(sequenceNumber, providerToCredit, priceIds, priceFeeds);
```

And update `emitPriceUpdate` to accept and emit `providerToCredit` as the `provider` argument.

For `withdrawAsFeeManager`, update the `FeesWithdrawn` event to include the `provider` field, or emit a separate event that captures both the provider whose fees were withdrawn and the recipient.

### Proof of Concept
1. Provider A registers and accrues fees.
2. Provider A sets Provider B as its fee manager via `setFeeManager`.
3. An independent relayer (address `R`) calls `executeCallback(providerA, seqNum, updateData, priceIds)`.
4. `providerA.accruedFeesInWei` is incremented — correct on-chain state.
5. `PriceUpdateExecuted` is emitted with `provider = R` (the relayer's address) instead of `providerA`.
6. Any off-chain system querying `PriceUpdateExecuted` events filtered by `provider == providerA` will miss this fulfillment entirely, while queries filtered by `provider == R` will incorrectly attribute the fulfillment to the relayer. [7](#0-6)

    function emitPriceUpdate(
        uint64 sequenceNumber,
        bytes32[] memory priceIds,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal {
        int64[] memory prices = new int64[](priceFeeds.length);
        uint64[] memory conf = new uint64[](priceFeeds.length);
        int32[] memory expos = new int32[](priceFeeds.length);
        uint64[] memory publishTimes = new uint64[](priceFeeds.length);

        for (uint i = 0; i < priceFeeds.length; i++) {
            prices[i] = priceFeeds[i].price.price;
            conf[i] = priceFeeds[i].price.conf;
            expos[i] = priceFeeds[i].price.expo;
            // Safe cast because this is a unix timestamp in seconds.
            publishTimes[i] = SafeCast.toUint64(
                priceFeeds[i].price.publishTime
            );
        }

        emit PriceUpdateExecuted(
            sequenceNumber,
            msg.sender,
            priceIds,
            prices,
            conf,
            expos,
            publishTimes
        );
    }
```
