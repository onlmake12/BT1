### Title
Unbounded `vec![0u64; max_bucket_index + 1]` allocation in `do_estimate` via high-fee-rate pool entry causes OOM/node crash — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

### Summary

`do_estimate` derives `max_bucket_index` directly from the highest fee_rate in the pending pool with no upper bound, then allocates two `Vec<u64>` of that size. A transaction with a sufficiently high fee_rate (achievable by burning CKB as fee on a minimum-size tx) causes a petabyte-scale heap allocation, crashing the node.

### Finding Description

`max_bucket_index_by_fee_rate` maps any `FeeRate` to a bucket index using the formula `(x + 11_500_000) / 100_000` for `x > 2_000_000`. This is unbounded: for `x ≈ u64::MAX - 11_500_001`, the result is ~1.84e14. [1](#0-0) 

`do_estimate` then allocates two `Vec<u64>` of size `max_bucket_index + 1` with no cap: [2](#0-1) [3](#0-2) 

`FeeRate::calculate` uses `saturating_mul`, so the maximum achievable fee_rate from a 60-byte tx is `u64::MAX / 60 ≈ 3.07e17`, yielding `max_bucket_index ≈ 3.07e12` and ~49 TB of attempted allocation: [4](#0-3) 

The tx-pool admission path (`check_tx_fee`) enforces only a **minimum** fee rate — there is no maximum fee rate or maximum fee cap: [5](#0-4) 

The `estimate_fee_rate` RPC reads live pool state and passes it directly into `do_estimate`: [6](#0-5) 

### Impact Explanation

Any caller of the `estimate_fee_rate` RPC (no authentication required) can crash the node process via OOM or panic after a high-fee-rate transaction is in the pool. The crash terminates the node, causing a full denial of service.

### Likelihood Explanation

The attack requires:
1. The `WeightUnitsFlow` estimator to be configured and ready (post-IBD).
2. The `Experiment` RPC module to be enabled.
3. The attacker to own enough CKB to create a high-fee transaction (~120,000 CKB to OOM a 16 GB node). This is a financial cost, not a privilege.

Once the transaction is in the pool, any subsequent call to `estimate_fee_rate` triggers the crash — including by a different, unprivileged caller.

### Recommendation

Cap `max_bucket_index` to a safe constant (e.g., the maximum index corresponding to a realistic fee rate ceiling like 10,000,000 shannons/KW, which maps to index ~135) before allocating the bucket vectors. Alternatively, add a `min(max_bucket_index, MAX_BUCKET_INDEX)` guard in `do_estimate`.
