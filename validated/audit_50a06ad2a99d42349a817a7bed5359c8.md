### Title
Malicious Entropy Provider Can Front-Run `requestV2` to Drain User's Excess ETH or Cause Persistent DoS - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

A malicious Entropy provider can observe a pending `requestV2` transaction in the mempool and front-run it by calling `setProviderFee` with an inflated fee. Because `requestHelper` reads the provider's current `feeInWei` at execution time and there is no user-specified maximum fee cap, the provider can either (a) cause the user's transaction to revert (DoS) or (b) drain any excess ETH the user sent, since excess value is explicitly not refunded.

### Finding Description

The `requestHelper` function in `Entropy.sol` computes the required fee at execution time by reading the provider's current `feeInWei` from storage:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

Any registered provider can update their fee at any time via `setProviderFee`:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
``` [2](#0-1) 

The interface explicitly documents that excess ETH is **not** refunded:

> "Note that excess value is *not* refunded to the caller." [3](#0-2) 

There is no mechanism for the user to specify a `maxFeeInWei` cap at request time. The fee used is always the live provider state at the moment of execution.

### Impact Explanation

**Attack vector A — DoS (transaction revert):**
1. User calls `getFeeV2(provider, gasLimit)` off-chain → receives fee `X`.
2. User submits `requestV2{value: X}(provider, gasLimit)`.
3. Provider observes the pending transaction in the mempool and front-runs with `setProviderFee(X - pythFee + 1)`.
4. At execution: `requiredFee = X + 1 > msg.value = X` → `revert InsufficientFee`.
5. User's ETH is returned, but the request fails. Provider can repeat this indefinitely, permanently blocking a specific user or contract from obtaining randomness.

**Attack vector B — ETH drain (excess value):**
1. User calls `getFeeV2()` → `X`, then submits `requestV2{value: X + buffer}()` (common defensive pattern).
2. Provider sees `msg.value = X + buffer` in the mempool.
3. Provider front-runs with `setProviderFee(X + buffer - pythFee)`.
4. At execution: `requiredFee = X + buffer = msg.value` → transaction succeeds.
5. Provider accrues `X + buffer - pythFee` (the full user payment minus Pyth's flat cut). The user overpays by `buffer` with no recourse.

### Likelihood Explanation

- Any permissionlessly registered provider is a potential attacker; registration requires no trust or vetting.
- `setProviderFee` is a standard, unrestricted provider operation with no timelock or delay.
- Mempool visibility of `msg.value` is trivial on all public EVM chains.
- The DoS variant requires zero economic cost to the attacker (they simply deny themselves fees temporarily, then restore them).
- The ETH drain variant is profitable for the attacker and requires only that the victim sends any amount above the quoted fee — a common defensive pattern in fee-paying contracts.

### Recommendation

Add a `maxFeeInWei` parameter to `requestV2` (and `requestWithCallback`) that the caller specifies at submission time. Inside `requestHelper`, revert if the live `requiredFee` exceeds the caller's stated maximum:

```solidity
if (requiredFee > maxFeeInWei) revert EntropyErrors.FeeTooHigh();
```

This mirrors the fix applied to the RoyaltyModule (`maxRts` parameter) and is the standard slippage-protection pattern for any protocol where a third-party-controlled value is consumed at execution time.

### Proof of Concept

```solidity
// Attacker is a registered Entropy provider.
// Step 1: User submits (pending in mempool):
//   entropy.requestV2{value: 1000 wei}(attackerProvider, 0);
//   (getFeeV2 returned 1000 wei)

// Step 2: Attacker front-runs (higher gas):
entropy.setProviderFee(1000 wei - pythFee);  // raises provider fee to consume all user ETH

// Step 3: User tx executes:
//   requiredFee = 1000 wei (now matches msg.value exactly)
//   providerInfo.accruedFeesInWei += (1000 - pythFee)  ← attacker drains user's buffer
//   accruedPythFeesInWei += pythFee

// Result: User paid 1000 wei but expected to pay only the original quoted fee.
// Attacker accrues (1000 - pythFee) wei instead of the original lower amount.
// For the DoS variant: set fee to 1001 wei → user tx reverts, ETH returned, request blocked.
``` [4](#0-3) [5](#0-4)

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
