Audit Report

## Title
`total_tx_size` Counter Permanently Inflated by Pre-Eviction Snapshot Overwrite in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` snapshots `self.total_tx_size + entry.size` into a local before ancestor-eviction runs. `check_and_record_ancestors` may call `remove_entry_and_descendants`, which correctly decrements `self.total_tx_size` via `update_stat_for_remove_tx`. The final assignment at L218–219 then overwrites `self.total_tx_size` with the stale pre-eviction snapshot, permanently discarding every decrement. The inflation is cumulative across eviction events and persists until node restart, eventually causing `limit_size` to evict legitimate transactions even when the pool is not actually full.

## Finding Description
The exact sequence in `add_entry` (L210–219):

```rust
// L210-211: snapshot BEFORE eviction — self is immutable here
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
// total_tx_size == self.total_tx_size + entry.size  (pre-eviction value)

// L213: may evict cell_ref_parents via remove_entry_and_descendants,
//        each of which calls update_stat_for_remove_tx and decrements
//        self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

// L218-219: OVERWRITES the decremented self.total_tx_size with the
//            pre-eviction snapshot — discards all decrements
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (L711–728) takes `&self` and only computes `self.total_tx_size.checked_add(tx_size)`, returning a local — it never mutates `self`. Inside `check_and_record_ancestors` (L603–625), when `ancestors_count > max_ancestors_count` but reducible by removing `cell_ref_parents`, `remove_entry_and_descendants` is called in a loop (L616–621). Each call chains to `remove_entry` (L252–265 → L235–250), which at L247 calls `update_stat_for_remove_tx`, correctly decrementing `self.total_tx_size`. The assignment at L218–219 then overwrites `self.total_tx_size` with the stale value, discarding every decrement.

**Concrete example:**
- Pool: `total_tx_size = 1000`; tx A (size=200) is a `cell_ref_parent`.
- New tx B (size=100) submitted.
- L210: local `total_tx_size = 1100`.
- L213: tx A evicted → `self.total_tx_size` decremented to `800`.
- L218: `self.total_tx_size = 1100` (overwrites 800).
- Correct value: 900. Actual value: 1100. **Inflation: +200 per eviction event.**

The inflation accumulates across repeated eviction events and is never corrected until node restart (unless an underflow triggers the `recompute_total_stat` fallback path in `update_stat_for_remove_tx`, which only fires on underflow, not overcount).

## Impact Explanation
`limit_size` uses `self.pool_map.total_tx_size > self.config.max_tx_pool_size` as its sole eviction guard. An inflated counter causes it to evict legitimate pending/proposed transactions even when the pool is not actually full. An attacker can repeatedly trigger the vulnerable path at low cost to keep `total_tx_size` permanently above `max_tx_pool_size`, causing the pool to continuously reject valid transactions with `Reject::Full`. This constitutes **High** impact: a vulnerability that can cause CKB network congestion (degraded transaction throughput, effective pool DoS) with few costs to the attacker, matching the allowed impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The path is reachable by any unprivileged P2P relay participant or RPC `send_transaction` caller. The attacker needs only to: (1) submit a transaction T1 used as a cell dep (creating a `cell_ref_parent` relationship), (2) submit a chain of transactions referencing T1 as a cell dep to push ancestor count near `max_ancestors_count`, and (3) submit a transaction consuming T1's output, triggering the eviction path in `check_and_record_ancestors`. This is a realistic, repeatable scenario requiring no special privileges. Each iteration inflates the counter by the size of the evicted transaction(s). No victim mistakes or external dependencies are required.

## Recommendation
Move the `updated_stat_for_add_tx` call to **after** `check_and_record_ancestors` completes, so `self.total_tx_size` already reflects all evictions before the new entry's size is added:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute AFTER evictions so self.total_tx_size is already decremented
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

This preserves the overflow check in `updated_stat_for_add_tx` while ensuring eviction decrements are not discarded.

## Proof of Concept
1. Configure a node with a small `max_tx_pool_size` and `max_ancestors_count = N`.
2. Submit tx T1 (size S1) that is used as a cell dep by subsequent transactions.
3. Submit a chain of N−1 transactions each referencing T1 as a cell dep, filling the ancestor count to the limit.
4. Submit tx T2 that consumes T1's output cell. This triggers `check_and_record_ancestors` to evict T1 (and descendants) via `remove_entry_and_descendants`.
5. Query `tx_pool_info` via RPC. Assert `total_tx_size` equals the actual sum of sizes of remaining entries. **Observation: `total_tx_size` is inflated by S1.**
6. Repeat steps 2–4 multiple times. Each iteration inflates `total_tx_size` by S1. Eventually `total_tx_size > max_tx_pool_size` even though the pool holds far fewer bytes, causing `limit_size` to evict legitimate transactions on every subsequent submission.
