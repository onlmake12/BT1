Audit Report

## Title
Unbounded `peers` Vec Growth via Repeated `RelayTransactionHashes` — (`sync/src/types/mod.rs`)

## Summary
`UnknownTxHashPriority::push_peer` appends a `PeerIndex` to an internal `Vec` with no length cap and no deduplication. The overflow guard in `add_ask_for_txs` only triggers when the number of **distinct hash keys** in `unknown_tx_hashes` reaches 50,000, not when the cumulative peer entries within a single key's `Vec` grow without bound. An unprivileged peer can exhaust node memory by repeatedly sending `RelayTransactionHashes` messages containing 32,767 copies of one unknown hash, growing that entry's `peers` Vec by ~32,766 per message with no effective limit.

## Finding Description

**`push_peer` — no cap, no dedup** (`sync/src/types/mod.rs`, L1291–1293):
`push_peer` unconditionally calls `self.peers.push(peer_index)`. There is no length check and no check for a duplicate `PeerIndex`.

**`add_ask_for_txs` — guard counts distinct hash keys, not peer entries** (`sync/src/types/mod.rs`, L1507–1529):
The condition `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) tests the number of distinct keys in the `KeyedPriorityQueue`. With a single tx hash entry the map length is 1, so this branch is never entered and the per-peer counter check at L1524 is never reached. The second condition `>= self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` is similarly bypassed when only one key exists.

**`add_ask_for_txs` — Occupied branch calls `push_peer` for every duplicate in the input** (`sync/src/types/mod.rs`, L1486–1504):
`.take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)` limits the number of hashes processed per call to 32,767, but does not deduplicate. If the attacker includes hash H 32,767 times in one message, the first occurrence hits `Vacant` (creating the entry), and the remaining 32,766 occurrences each hit `Occupied` and call `push_peer`, growing that entry's `peers` Vec by 32,766 in a single invocation.

**`TransactionHashesProcess::execute` — no dedup before forwarding** (`sync/src/relayer/transaction_hashes_process.rs`, L38–49):
The only filter applied is `tx_filter.contains`, which removes hashes already marked as known. An unknown hash passes through every time, regardless of how many times the same peer has announced it. The message-level size check at L29 only enforces `len() <= MAX_RELAY_TXS_NUM_PER_BATCH` (32,767), not uniqueness.

**Constants confirm the scale** (`util/constant/src/sync.rs`, L68–72):
`MAX_RELAY_TXS_NUM_PER_BATCH = MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32,767` and `MAX_UNKNOWN_TX_HASHES_SIZE = 50,000`.

## Impact Explanation

`unknown_tx_hashes` is a process-wide `Mutex<KeyedPriorityQueue<Byte32, UnknownTxHashPriority>>`. Each `UnknownTxHashPriority` holds a `Vec<PeerIndex>`. At 8 bytes per `PeerIndex` (usize on 64-bit), 1,000 messages each containing 32,767 copies of one hash → ~262 MB for a single map entry. The mutex is held for the entire `add_ask_for_txs` loop, so unbounded Vec growth also causes increasing lock contention on every concurrent operation touching `unknown_tx_hashes` (`pop_ask_for_txs`, `mark_as_known_txs`). Continued growth leads to OOM and node crash.

**Impact: High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires only a standard P2P connection. `RelayTransactionHashes` is an ordinary relay message accepted from any connected peer with no PoW, no key material, and no privileged role. The attacker needs only one tx hash that is not yet in `tx_filter` — a freshly fabricated hash that will never be resolved keeps the entry alive indefinitely. The attack is repeatable at the rate the network accepts messages from a single peer.

## Recommendation

1. **Cap `peers` Vec length in `push_peer`**: reject the push if `self.peers.len()` already exceeds a small constant (e.g., the number of connected peers or a fixed limit such as 256).
2. **Deduplicate in `push_peer`**: skip the push if `peer_index` is already present.
3. **Deduplicate hashes within a single `RelayTransactionHashes` message** before passing them to `add_ask_for_txs` (e.g., collect into a `HashSet` first).
4. **Move the per-peer counter check before insertion**, not after, so it is enforced regardless of the total map size.

## Proof of Concept

```
1. Attacker peer A connects to victim node.
2. A sends RelayTransactionHashes{ tx_hashes: [H; 32767] }
   → TransactionHashesProcess::execute: H not in tx_filter → all 32767 copies forwarded.
   → add_ask_for_txs: first copy hits Vacant → entry created with peers: [A].
     copies 2–32767 hit Occupied → push_peer called 32766 times.
   → unknown_tx_hashes.len() == 1 → guard at L1507 never triggers.
3. Repeat step 2 indefinitely.
4. After K messages: peers.len() for H ≈ 1 + 32766 * K.
   At K=1000: ~262 MB for a single entry; node OOM-crashes.
5. Every pop_ask_for_txs / mark_as_known_txs call must acquire the same mutex,
   causing increasing lock contention as the Vec grows.

Unit assertion: after calling add_ask_for_txs(peer_A, vec![H; 32767]) twice
(H already Occupied after first call), peers.len() for H == 1 + 32766 + 32767 = 65534,
confirming no cap is enforced.
```
