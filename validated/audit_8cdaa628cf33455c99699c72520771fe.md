### Title
Expired Orphan Blocks Silently Drop `verify_callback` Without Execution, Permanently Losing Peer-Penalty and Relay-Broadcast State — (`chain/src/orphan_broker.rs`)

---

### Summary

`clean_expired_orphans()` in `chain/src/orphan_broker.rs` iterates over expired `LonelyBlockHash` entries and deletes them, but never calls `execute_callback()` on them. Each `LonelyBlockHash` carries a `verify_callback: Option<VerifyCallback>` closure injected by the sync/relayer layer. When the orphan expires, that closure is silently dropped — the caller is never notified, the peer is never penalized, and any pending relay broadcast is permanently lost. This is structurally identical to the VUSD pattern: an item is consumed from a processing queue, the intended side-effect is never executed, and there is no retry path.

---

### Finding Description

**Entry path — how a `verify_callback` gets attached**

Both the synchronizer and the relayer create a `RemoteBlock` with a non-`None` `verify_callback` before handing the block to the chain service:

Both callbacks call `post_sync_process` → `nc.ban_peer()` on `Err`, and the relayer additionally calls `build_and_broadcast_compact_block` on `Ok(true)`.

---

### Impact Explanation

**1. Peer-penalty bypass (security)**
A remote peer that deliberately sends blocks whose parents are never relayed causes those blocks to accumulate in the orphan pool. When the epoch window closes and `clean_expired_orphans` runs, every such block is evicted without the callback firing. The peer is never banned. Because the orphan pool is bounded, a sustained stream of crafted orphan blocks can evict legitimate orphans while the attacker faces zero protocol-level consequence.

**2. Relay broadcast permanently lost (liveness)**
For blocks received via the relayer, the `Ok(true)` branch of the callback calls `build_and_broadcast_compact_block`. If a block enters the orphan pool and its parent arrives *after* the orphan has already been evicted and deleted, the compact-block broadcast is permanently lost. No retry path exists; the block will not propagate to other peers through this node.

Both effects are permanent: there is no mechanism to re-queue the callback or re-trigger the broadcast.

---

### Likelihood Explanation

Any unprivileged P2P peer can trigger this path by sending a `SendBlock` (sync) or compact-block-derived full block (relayer) whose parent hash is unknown and whose parent is never subsequently sent. This is a zero-cost, zero-privilege operation. The orphan expiry window is epoch-based, so the attacker only needs to wait one epoch per batch of orphans. No special hashpower, key material, or Sybil capability is required.

---

### Recommendation

Before dropping each `expired_orphan`, call `execute_callback` with an appropriate error so the sync/relayer layer can take action:

```rust
for expired_orphan in expired_orphans {
    self.delete_block(&expired_orphan);
    self.shared.remove_header_view(&expired_orphan.hash());
    self.shared.remove_block_status(&expired_orphan.hash());
    info!("cleaned expired orphan: {}-{}", ...);

    // NEW: notify caller so peer-penalty and relay logic can run
    let err = InternalErrorKind::Other
        .other(format!(
            "orphan block {}-{} expired without parent",
            expired_orphan.number(),
            expired_orphan.hash()
        ))
        .into();
    expired_orphan.execute_callback(Err(err));
}
```

The sync and relayer callbacks already guard against `is_internal_db_error`, so a dedicated `OrphanExpired` error kind (or a flag on the existing error) can be used to distinguish expiry from genuine invalidity, allowing the callbacks to skip peer-banning for the expiry case while still releasing any held resources.

---

### Proof of Concept

1. Connect a peer to a CKB node.
2. Send a `SendBlock` message containing a syntactically valid block whose `parent_hash` refers to a block that does not exist and will never be sent.
3. The block passes `non_contextual_verify`, is inserted into the DB, and is placed in the orphan pool via `orphan_blocks_broker.insert(lonely_block)`.
4. Wait for the node's tip epoch to advance past the orphan's epoch number.
5. `clean_expired_orphans` fires (called from `chain_service.rs` on each new block), removes the orphan, and drops the `verify_callback` without calling it.
6. Observe: `post_sync_process` / `nc.ban_peer` is never called for the sending peer; the peer remains connected and un-penalized indefinitely.
