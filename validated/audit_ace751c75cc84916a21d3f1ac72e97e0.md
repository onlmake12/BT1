### Title
Signed Lazer Price Update Replay — No Timestamp Validation in `verifyUpdate` - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.sol::verifyUpdate` verifies the ECDSA signature and checks that the signer has not expired, but it never validates the `timestamp` embedded in the signed payload against any stored on-chain state. Because the contract is entirely stateless with respect to payload freshness, any previously-seen, validly-signed Lazer update can be replayed indefinitely by an unprivileged transaction sender until the signer's `expiresAt` passes.

### Finding Description

The `verifyUpdate` function performs three checks:

1. Fee payment
2. EVM magic bytes
3. ECDSA signature recovery + signer validity (`isValidSigner`) [1](#0-0) 

The signed payload contains a `timestamp` field (parsed by `PythLazerLib::parsePayloadHeader`): [2](#0-1) 

This timestamp is part of the data that the trusted signer signs — it is the only field that distinguishes one price update from another at the same price level. However, `verifyUpdate` never reads this timestamp, never compares it to a stored "last accepted timestamp," and never reverts on a stale or duplicate payload. The contract holds no per-signer or per-feed sequence/timestamp state at all.

The analog to the external report is exact:

| Astaria | Pyth Lazer |
|---|---|
| Strategist nonce is included in the signed commitment | Timestamp is included in the signed payload |
| `validateCommitment` never checks the nonce against stored state | `verifyUpdate` never checks the timestamp against stored state |
| Commitment can be reused up to 5 times | Signed update can be replayed unlimited times |

The test suite confirms the same `update` bytes can be submitted twice in the same block with no revert: [3](#0-2) 

### Impact Explanation

An attacker who has observed any valid signed Lazer update (e.g., from a public mempool, an event log, or a WebSocket stream) can replay it to any consumer contract that calls `verifyUpdate`. Consumer contracts that do not independently enforce `_timestamp > lastTimestamp` will accept the stale price as current. For DeFi protocols using Lazer prices for collateral valuation, liquidation triggers, or AMM pricing, this enables:

- Forcing a stale (artificially low or high) price to trigger or prevent liquidations
- Replaying a favorable price snapshot to extract value from a protocol

The Pyth documentation example shows the timestamp guard as optional consumer-side logic, not a contract-enforced invariant: [4](#0-3) 

### Likelihood Explanation

Likelihood is high. Every signed Lazer update is broadcast over WebSocket to all subscribers. Any subscriber can capture and replay any update. No privileged access, leaked key, or governance majority is required. The attacker only needs to have received a single valid signed update, which is the normal operating condition for any Lazer consumer.

### Recommendation

`PythLazer.sol` should maintain a `mapping(address signer => uint64 lastTimestamp)` (or a global `uint64 lastTimestamp` if a single monotonic stream is assumed) and revert in `verifyUpdate` if the payload timestamp is not strictly greater than the stored value:

```solidity
mapping(address => uint64) public lastAcceptedTimestamp;

// inside verifyUpdate, after recovering signer:
uint64 payloadTimestamp = uint64(bytes8(payload[4:12]));
require(payloadTimestamp > lastAcceptedTimestamp[signer], "stale update");
lastAcceptedTimestamp[signer] = payloadTimestamp;
```

Alternatively, the function should be clearly renamed and documented as a pure signature verifier with no freshness guarantee, and the SDK should enforce timestamp checks at the library level before returning to the consumer.

### Proof of Concept

1. Alice subscribes to Lazer WebSocket and receives a signed EVM update `U` at timestamp `T` with price `P_old`.
2. The real price moves to `P_new` (e.g., asset crashes).
3. Bob (attacker) replays `U` to a victim DeFi contract that calls `pythLazer.verifyUpdate{value: fee}(U)`.
4. `verifyUpdate` succeeds — signature is valid, signer has not expired — and returns the payload with `P_old`.
5. The victim contract, which does not independently check `_timestamp > lastTimestamp`, updates its price to `P_old`.
6. Bob exploits the stale price (e.g., borrows against over-valued collateral or avoids a liquidation).

The replay succeeds because `verifyUpdate` contains no state tracking and no timestamp validation: [1](#0-0)
