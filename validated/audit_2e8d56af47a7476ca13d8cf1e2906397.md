Audit Report

## Title
Tx-Pool `total_tx_size`/`total_tx_cycles` Permanently Inflated by Stale Snapshot Overwrite After Eviction in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, `updated_stat_for_add_tx` snapshots `self.total_tx_size + entry.size` into a local variable before `check_and_record_ancestors` runs. When that call evicts entries via `remove_entry_and_descendants` → `update_stat_for_remove_tx`, it decrements `self.total_tx_size` in-place. The stale pre-eviction local is then unconditionally written back to `self.total_tx_size`, silently discarding all decrements. Each such eviction round permanently inflates `total_tx_size` by the aggregate size of all evicted entries, eventually causing `limit_size` to reject valid transactions with `Reject::Full` even when the pool is nearly empty.

## Finding Description

**Root cause — stale snapshot overwrite:**

In `add_entry` (lines 200–221 of `tx-pool/src/component/pool_map.rs`):

```
Line 210–211: total_tx_size = self.total_tx_size + entry.size   ← snapshot taken
Line 213:     check_and_record_ancestors(...)                    ← may evict, decrementing self.total_tx_size in-place
Lines 218–219: self.total_tx_size = total_tx_size               ← stale snapshot written back
```

`updated_stat_for_add_tx` (lines 711–728) reads `self.total_tx_size` and returns `self.total_tx_size + tx_size` as a plain `usize` local — it does not hold a reference, so any subsequent in-place mutations to `self.total_tx_size` are invisible to it.

`check_and_record_ancestors` (lines 588–640) enters the eviction branch when `ancestors_count > max_ancestors_count` and `cell_ref_parents` is non-empty (line 603). It calls `remove_entry_and_descendants` in a loop (line 618). That function calls `update_stat_for_remove_tx` (lines 733–758), which directly writes `self.total_tx_size = self.total_tx_size - tx_size` for each removed entry.

After `check_and_record_ancestors` returns, lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot, undoing every decrement. The net inflation per `add_entry` call equals the sum of sizes of all entries evicted during that call. The corruption accumulates monotonically across repeated invocations.

The underflow guard in `update_stat_for_remove_tx` (lines 742–756) is irrelevant here: the decrements succeed normally; the corruption is introduced by the subsequent overwrite, not by underflow.

**Why `limit_size` is affected:**

`limit_size` (pool.rs lines 292–329) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting entries and issuing `Reject::Full`. An inflated `total_tx_size` causes this loop to fire and evict legitimate transactions, and eventually causes the newly submitted transaction itself to be rejected with `Reject::Full` even when actual pool occupancy is far below the configured limit.

## Impact Explanation

An unprivileged attacker can drive `total_tx_size` above `max_tx_pool_size` with a small number of crafted transactions, after which every subsequent `send_transaction` call (local RPC or relay) is rejected with `Reject::Full`. This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*. The attacker does not need to fill the pool with real data — the counter is inflated by phantom bytes that no longer correspond to any pooled transaction.

## Likelihood Explanation

The eviction branch in `check_and_record_ancestors` is reachable by any unprivileged caller of `send_transaction` (RPC) or via the `RelayV3` protocol. No key material, admin access, or majority hashpower is required. The attacker needs only: (1) a live cell usable as a `cell_dep`, (2) the ability to submit transactions referencing it, and (3) a transaction that spends it as an input to trigger the eviction loop. The attack is repeatable at low cost per iteration and requires no victim interaction.

## Recommendation

Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so it reads the post-eviction `self.total_tx_size`:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    // Read post-eviction totals before adding new entry's contribution
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

This guarantees `self.total_tx_size` always equals `(post-eviction pool total) + (new entry size)`.

## Proof of Concept

1. Mine enough blocks to obtain a mature cellbase output `C`.
2. Submit `tx_root` spending `C`, producing output `cell_dep_out`.
3. Submit `N > max_ancestors_count` transactions `ref_tx_1…ref_tx_N`, each spending an independent live cell and including `cell_dep_out` as a `cell_dep`. These become `cell_ref_parents` in the pool graph.
4. Submit `tx_consume` spending `cell_dep_out` as an input. Its ancestor count is `N+1 > max_ancestors_count`, so `check_and_record_ancestors` enters the eviction branch (line 603), removes `N − max_ancestors_count + 1` of the `ref_tx_*` entries via `remove_entry_and_descendants` (line 618), each decrementing `self.total_tx_size` in-place — but the pre-computed `total_tx_size` local is then written back at line 218, discarding those decrements.
5. Call `tx_pool_info`. Observe `total_tx_size` exceeds the true sum of sizes of all currently-pooled transactions by `sum(size(ref_tx_i) for evicted i)`.
6. Repeat steps 3–4 to accumulate inflation until `total_tx_size > max_tx_pool_size`.
7. Submit any valid transaction; observe it is rejected with `Reject::Full` despite the pool being nearly empty.

A unit test can confirm this by asserting `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.inner.size).sum()` after each `add_entry` call that triggers the eviction branch.
