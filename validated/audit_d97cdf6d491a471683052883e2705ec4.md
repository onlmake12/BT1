### Title
`Entropy::getProviderFee` Rounds Additional Gas Fee Down in Favor of the User, Causing Continuous Provider Fee Leakage — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol::getProviderFee` computes the additional fee for requests with a custom gas limit above the provider's default using integer division that truncates (rounds down). This means every such request pays slightly less than the proportional fee, leaking value from the provider. The in-code comment acknowledges the rounding direction but incorrectly claims the error is "< 1 wei"; the actual maximum rounding error per transaction is `provider.defaultGasLimit − 1` wei.

---

### Finding Description

When a user calls `requestV2` with a `gasLimit` above the provider's `defaultGasLimit`, `getProviderFee` computes an `additionalFee` using:

```solidity
uint128 additionalFee = ((roundedGasLimit -
    provider.defaultGasLimit) * provider.feeInWei) /
    provider.defaultGasLimit;
```

Solidity integer division truncates toward zero. The remainder — up to `provider.defaultGasLimit − 1` wei — is silently discarded. The user pays `provider.feeInWei + additionalFee` (rounded down), while the provider receives exactly that rounded-down amount. The code comment at line 785 reads:

> "This calculation rounds down the fee, which means that users can get some gas in the callback for free. However, the value of the free gas is < 1 wei, which is insignificant."

This claim is incorrect. The rounding error is bounded by `provider.defaultGasLimit − 1` wei per transaction. For a provider with `defaultGasLimit = 200,000`, the maximum rounding error is **199,999 wei per request**. Over many requests this accumulates into a meaningful loss for the provider.

The fee split in `requestHelper` confirms the provider receives only the rounded-down value:

```solidity
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

Pyth receives `msg.value − providerFee`, which equals exactly `pythFeeInWei` when the user pays the exact required fee. The rounding loss falls entirely on the provider.

---

### Impact Explanation

Any unprivileged user calling `requestV2(gasLimit)` with a `gasLimit` above the provider's default triggers the rounding shortfall. The provider's `accruedFeesInWei` is credited less than the proportional fee for the extra gas. The maximum shortfall per transaction is `provider.defaultGasLimit − 1` wei. For a provider with `defaultGasLimit = 200,000` and `feeInWei = 1,000,000 wei`, the rounding error can reach ~190,000 wei per transaction. Across high-volume usage, this constitutes a slow, continuous, and compounding drain on provider revenue — directly analogous to the EIP-4626 rounding-in-favor-of-user pattern described in the reference report.

---

### Likelihood Explanation

The trigger path is fully permissionless: any caller of `requestV2(uint32 gasLimit)` or `requestV2(address, bytes32, uint32)` with a `gasLimit` above the provider's default exercises this code path. No privileged role, leaked key, or external oracle manipulation is required. The rounding error is deterministic and occurs on every such request.

---

### Recommendation

Round the `additionalFee` **up** (ceiling division) to favor the provider, consistent with EIP-4626 security guidance and standard fee-accounting practice:

```solidity
// Ceiling division: (a * b + c - 1) / c
uint128 additionalFee = uint128(
    (uint256(roundedGasLimit - provider.defaultGasLimit) * provider.feeInWei
        + provider.defaultGasLimit - 1)
    / provider.defaultGasLimit
);
```

This ensures the provider is never underpaid for extra gas, and the rounding error (at most 1 wei) is borne by the user rather than the provider.

### Proof of Concept

**Setup**: Provider registers with `defaultGasLimit = 30,000` and `feeInWei = 100,007 wei`.

**User request**: `gasLimit = 60,000` → `roundedGasLimit = 60,000` (already a multiple of 10k).

**Current behavior**:
```
additionalFee = (30,000 × 100,007) / 30,000
             = 3,000,210,000 / 30,000
             = 100,007  (exact — no error here)
```

**User request**: `gasLimit = 50,001` → `roundedGasLimit = 60,000`.

```
additionalFee = (30,000 × 100,007) / 30,000 = 100,007  (exact)
```

**User request**: `gasLimit = 40,001` → `roundedGasLimit = 50,000`.

```
additionalFee = (20,000 × 100,007) / 30,000
             = 2,000,140,000 / 30,000
             = 66,671  (truncated from 66,671.333...)
rounding error = 2,000,140,000 % 30,000 = 10,000 wei lost by provider per request
```
