### Title
Static `feePerGasInWei` Creates Incentive Gap for Callback Execution During Gas Price Spikes - (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

The Echo contract's provider fee for callback execution is calculated using a static, provider-configured `feePerGasInWei` rate that does not reflect actual on-chain gas prices at execution time. The fee is locked into the request at submission time. When gas prices spike between request submission and callback execution, the provider's execution cost can exceed the locked-in fee, eliminating any economic incentive to call `executeCallback`. This is a direct structural analog to the PoolTogether M-09 finding: a fee cap bounded by a stale/wrong reference value rather than the actual cost of fulfillment.

---

### Finding Description

In `Echo.sol`, the fee a user pays is computed in `getFee()` as:

```
feeAmount = pythFeeInWei + baseFeeInWei + (priceIds.length * feePerFeedInWei) + (callbackGasLimit * feePerGasInWei)
``` [1](#0-0) 

`feePerGasInWei` is a static value set by the provider at registration or via `setProviderFee`. It is **not** `tx.gasprice` — the code even contains a developer comment acknowledging this: `// FIXME: this comment is wrong. (we're not using tx.gasprice)`. [2](#0-1) 

At request time, the entire fee paid by the user is locked into the stored request:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [3](#0-2) 

When `executeCallback` is called, the provider receives exactly `req.fee + msg.value - pythFee` — the fee that was locked at request time, not a fee reflecting current gas prices. [4](#0-3) 

The `ProviderInfo` struct confirms `feePerGasInWei` is a plain stored value with no dynamic gas-price linkage: [5](#0-4) 

The Fortuna off-chain keeper does have a `adjust_fee_if_necessary` mechanism that updates the on-chain `feePerGasInWei` dynamically: [6](#0-5) 

However, this only affects **future** requests. Existing in-flight requests have their fee already locked. During a gas price spike, the keeper will raise `feePerGasInWei` for new requests, but all pending requests remain at the old, insufficient rate.

---

### Impact Explanation

When network gas prices spike significantly above the provider's configured `feePerGasInWei`:

1. The cost to submit `executeCallback` (which must supply `callbackGasLimit` gas to the consumer callback) exceeds the fee locked in the request.
2. The provider/executor has no economic incentive to execute the callback.
3. The request sits unfulfilled until gas prices drop back below the locked fee rate.
4. Consumer contracts depending on the price update callback (e.g., for liquidations, settlements, or randomness-dependent logic) are blocked.

There is no on-chain penalty mechanism for the provider failing to execute — the code even has a TODO acknowledging this gap: [7](#0-6) 

**Impact**: Denial of service on price update callbacks during gas price spikes. Consumer contracts that rely on timely price updates may miss critical windows (e.g., liquidation opportunities, time-sensitive settlements).

---

### Likelihood Explanation

Gas price spikes are a recurring, well-documented phenomenon on EVM chains (e.g., during NFT mints, token launches, or network congestion events). The `feePerGasInWei` is a static value that providers update reactively, not proactively. The window between a gas spike and a fee update is the attack/failure surface. Any user can trigger this condition simply by submitting a request when gas is cheap — the request will be unprofitable to fulfill if gas spikes before the callback is executed.

---

### Recommendation

1. **Dynamic fee at execution time**: Compute the provider's gas compensation using `tx.gasprice` (or `block.basefee + priority_fee`) at `executeCallback` time, not at request time. The user would pre-pay a maximum fee, and the actual gas cost would be deducted at execution.

2. **Alternatively, store a minimum acceptable gas price in the request**: At request time, record the current gas price. At execution time, require that the executor is compensated at least at that rate, or allow the executor to top up the fee.

3. **On-chain penalty/escalation**: Implement the TODO penalty mechanism so that if the assigned provider does not execute within the exclusivity period, their accrued fees are slashed and redistributed to whoever executes the callback, creating a self-correcting incentive.

---

### Proof of Concept

1. Gas price is 10 gwei. Provider sets `feePerGasInWei = 10 gwei`.
2. User calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 1,000,000`. Fee paid: `1,000,000 * 10 gwei = 0.01 ETH`. This is locked as `req.fee`.
3. Gas price spikes to 200 gwei (e.g., during a network event).
4. Provider's cost to call `executeCallback` with 1M gas: `1,000,000 * 200 gwei = 0.2 ETH`.
5. Provider would lose `0.2 - 0.01 = 0.19 ETH` per callback execution.
6. Provider skips all pending callbacks. Consumer contracts receive no price updates.
7. Fortuna keeper raises `feePerGasInWei` to 200 gwei for new requests, but existing locked requests remain at 10 gwei and are never executed until gas drops. [8](#0-7) [4](#0-3)
