### Title
Legacy `revealWithCallback` Path Lacks Callback Failure Protection, Permanently Blocking Entropy Fulfillment — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

`Entropy.sol::revealWithCallback()` contains two execution paths. The modern path (when `req.gasLimit10k != 0`) uses `excessivelySafeCall` to catch callback reverts and transitions the request to a recoverable `CALLBACK_FAILED` state. The legacy path (when `req.gasLimit10k == 0`) calls the consumer's `_entropyCallback` directly with no try/catch. If the consumer's callback permanently reverts — due to any external condition the user cannot control — the legacy path's `revealWithCallback` will always revert, permanently blocking fulfillment of the request with no recovery mechanism.

### Finding Description

In `revealWithCallback`, the branching logic is:

```solidity
if (
    req.gasLimit10k != 0 &&
    req.callbackStatus == EntropyStatusConstants.CALLBACK_NOT_STARTED
) {
    // Modern path: excessivelySafeCall catches reverts → CALLBACK_FAILED state
    (success, ret) = req.requester.excessivelySafeCall(...);
    ...
    req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED; // recoverable
} else {
    // Legacy path: direct call, NO try/catch
    clearRequest(provider, sequenceNumber);
    if (len != 0) {
        IEntropyConsumer(callAddress)._entropyCallback(
            sequenceNumber, provider, randomNumber
        );
    }
    // If callback reverts here, entire tx reverts (clearRequest is undone)
}
``` [1](#0-0) [2](#0-1) 

A request lands in the legacy path when `req.gasLimit10k == 0`. This happens when the provider has never called `setDefaultGasLimit` (i.e., `providerInfo.defaultGasLimit == 0`):

```solidity
if (providerInfo.defaultGasLimit == 0) {
    // Provider doesn't support the new callback failure state flow
    req.gasLimit10k = 0;
}
``` [3](#0-2) 

In the legacy path, `clearRequest` is called before the callback. If the callback reverts, the entire transaction reverts (including `clearRequest`), so the request remains active. However, since the callback is called with no error isolation, if the consumer's callback **permanently** reverts (e.g., the consumer contract calls a downstream contract that becomes paused/killed), every subsequent `revealWithCallback` call will also revert. The request is stuck in an unfulfillable state with no recovery path — unlike the modern path which provides `CALLBACK_FAILED` state and a retry mechanism.

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` confirms that for a legacy provider (no `defaultGasLimit` set), a reverting callback causes `revealWithCallback` to revert entirely: [4](#0-3) 

### Impact Explanation

- The randomness request is permanently unfulfillable: the user never receives their random number.
- The provider's accrued fee for that request is locked — they cannot earn it because fulfillment always reverts.
- The user's paid fee is effectively lost (it was credited to the provider's `accruedFeesInWei` at request time, but the provider cannot deliver the service).
- There is no on-chain escape hatch: unlike the modern path's `CALLBACK_FAILED` state, the legacy path provides no way to mark the request as failed and move on.

### Likelihood Explanation

- Legacy providers (those that have never called `setDefaultGasLimit`) still exist in the codebase and are explicitly supported.
- Any consumer contract that calls a downstream protocol (e.g., a DeFi protocol, a token contract) in its `entropyCallback` is vulnerable if that downstream protocol becomes paused or enters a broken state — an external condition the user cannot control.
- The `requestWithCallback` (V1) function is still callable and routes through the same legacy path for providers with `defaultGasLimit == 0`. [5](#0-4) 

### Recommendation

Apply the same `excessivelySafeCall` / callback-failure-state protection to the legacy path, or require all providers to set a non-zero `defaultGasLimit` before accepting new requests. At minimum, the legacy path should use a try/catch so that a permanently reverting callback does not permanently block fulfillment.

### Proof of Concept

1. Provider `P` has `defaultGasLimit == 0` (legacy, never called `setDefaultGasLimit`).
2. User `U` deploys a consumer contract `C` whose `entropyCallback` calls an external protocol `X`.
3. `U` calls `requestWithCallback(P, userRandomness)` — request is stored with `gasLimit10k = 0`.
4. Protocol `X` becomes paused/killed; `C.entropyCallback` now always reverts.
5. Provider `P` calls `revealWithCallback(P, seqNum, userContrib, providerContrib)`.
6. Execution enters the legacy `else` branch; `clearRequest` runs, then `_entropyCallback` is called and reverts.
7. The entire transaction reverts; the request is restored to active state.
8. Steps 5–7 repeat indefinitely. The request can never be fulfilled. `U` never receives randomness. `P` can never earn the fee. [6](#0-5)
