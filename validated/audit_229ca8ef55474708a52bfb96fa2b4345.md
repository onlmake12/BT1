### Title
Fee Rate Unit Mismatch in `check_tx_fee` Allows Cycle-Heavy Transactions to Bypass Min Fee Rate Enforcement - (File: `tx-pool/src/util.rs`)

### Summary

The `check_tx_fee` function enforces the `min_fee_rate` threshold using raw serialized byte size (`tx_size`) as the denominator, but `FeeRate` is defined and configured in units of **shannons per kilo-weight** (shannons/KW), where weight = `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions, weight can be orders of magnitude larger than size. Because cycles are only known after script execution and no subsequent weight-based fee check exists, a transaction submitter can craft a cycle-heavy transaction that passes the size-based admission check with an effective fee rate far below the configured `min_fee_rate`.

### Finding Description

**Root cause — `tx-pool/src/util.rs`, `check_tx_fee`, line 45:**

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

`FeeRate::fee(weight)` is defined in `util/types/src/core/fee_rate.rs` as:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;  // KW = 1000
    Capacity::shannons(fee)
}
```

The function expects `weight` in weight-units (where weight = `max(size, cycles × 0.000_170_571_4)`), but `tx_size` (raw serialized bytes) is passed instead. The code comment acknowledges this explicitly:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly"

However, `check_tx_fee` is the **only** fee rate admission check. There is no subsequent weight-based fee check after script execution (when cycles become known). The "cheap check" is the sole gate.

**The unit mismatch:**

| Context | Unit used | Meaning |
|---|---|---|
| `FeeRate` type definition | shannons/KW | shannons per 1000 weight-units |
| `check_tx_fee` enforcement | shannons/KB | shannons per 1000 bytes |
| Config comment (`ckb.toml`) | "shannons/KB" | Matches the enforcement, not the type |

For a cycle-heavy transaction:
- `size` = 242 bytes (minimal tx)
- `cycles` = 70,000,000 (max, `max_tx_verify_cycles`)
- `weight` = max(242, 70,000,000 × 0.000_170_571_4) ≈ **11,940**
- `min_fee` enforced = `1000 × 242 / 1000` = **242 shannons** (size-based)
- `min_fee` correct = `1000 × 11,940 / 1000` = **11,940 shannons** (weight-based)
- Effective fee rate admitted = `242 / 11,940 × 1000` ≈ **20 shannons/KW** (vs. configured 1000)

### Impact Explanation

Any RPC caller (`send_transaction`) or P2P relayer can submit a cycle-heavy transaction with a fee of only `min_fee_rate × size / 1000` shannons — approximately **2% of the true weight-based minimum** — and have it admitted to the tx-pool. This allows:

1. **Mempool spam**: Filling the pool with transactions whose effective fee rate is ~49× below the configured floor, displacing legitimate transactions.
2. **Fee rate floor bypass**: The `min_fee_rate` spam-prevention mechanism is rendered ineffective for cycle-heavy scripts (e.g., complex lock scripts, type scripts with heavy computation).

The maximum weight inflation factor is bounded by `max_tx_verify_cycles = 70,000,000`, giving a worst-case ratio of ~49× for a minimal-size transaction.

### Likelihood Explanation

**Medium.** The entry path is fully unprivileged — any node's RPC `send_transaction` endpoint or the P2P relay protocol accepts transactions from external callers. Crafting a cycle-heavy transaction requires writing a script that consumes many cycles (e.g., a lock script with a tight computation loop), which is within the capability of any script author. The `max_tx_verify_cycles` cap limits the maximum exploitation ratio but does not prevent the bypass.

### Recommendation

After script execution (when `cycles` are known), add a weight-based fee check using `get_transaction_weight(tx_size, cycles)`:

```rust
// After verify_rtx returns cycles:
let weight = get_transaction_weight(tx_size, cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

The existing size-based check in `check_tx_fee` can remain as a fast pre-filter before script execution, but must be followed by a weight-based check post-execution.

### Proof of Concept

1. Author a CKB lock script that performs a tight computation loop consuming ~70,000,000 cycles.
2. Create a transaction spending a cell locked by this script. Transaction serialized size ≈ 242 bytes.
3. Set the transaction fee to `1000 × 242 / 1000 = 242 shannons` (just above the size-based threshold).
4. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000`.
5. **Observed**: Transaction is admitted to the pool. Effective fee rate = `242 / 11940 × 1000 ≈ 20 shannons/KW`, which is ~50× below the configured minimum.
6. **Expected**: Transaction should be rejected with `LowFeeRate` because the weight-based minimum fee is 11,940 shannons.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

use crate::core::Capacity;

/// shannons per kilo-weight
#[derive(Clone, Copy, Default, Debug, PartialEq, Eq, PartialOrd, Ord)]
pub struct FeeRate(pub u64);

const KW: u64 = 1000;

impl FeeRate {
    /// Calculates the fee rate from a total fee and weight.
    pub fn calculate(fee: Capacity, weight: u64) -> Self {
        if weight == 0 {
            return FeeRate::zero();
        }
        FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
    }

    /// Creates a fee rate from shannons per kilo-weight.
    pub const fn from_u64(fee_per_kw: u64) -> Self {
        FeeRate(fee_per_kw)
    }

    /// Returns the fee rate as shannons per kilo-weight.
    pub const fn as_u64(self) -> u64 {
        self.0
    }

    /// Creates a zero fee rate.
    pub const fn zero() -> Self {
        Self::from_u64(0)
    }

    /// Calculates the fee for a given weight.
    pub fn fee(self, weight: u64) -> Capacity {
        let fee = self.0.saturating_mul(weight) / KW;
        Capacity::shannons(fee)
    }

#[deprecated(

pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {

    )

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
