### Title
Miner Can Selectively Exclude Uncle Blocks to Game the Epoch Difficulty Adjustment — (`spec/src/consensus.rs`)

### Summary

CKB's epoch difficulty adjustment uses `epoch_uncles_count` as a direct input to compute the next epoch's difficulty and length. Because uncle inclusion in a mined block is **optional** — the `BlockTemplate` marks all uncle candidates as `required: false` — any miner can strategically withhold eligible uncle blocks from their own mined blocks. This artificially deflates the observed uncle rate at epoch boundary, causing `next_epoch_ext` to compute a lower difficulty (and longer epoch) for the next epoch. The attack is analogous to the VRF re-roll: instead of re-rolling randomness, the miner re-rolls the difficulty-adjustment inputs by choosing which uncle evidence to commit.

---

### Finding Description

**Step 1 — Uncle inclusion is optional by protocol design.**

The `BlockTemplate` returned by `get_block_template` exposes uncle candidates with `required: false`:

```rust
// util/jsonrpc-types/src/block_template.rs
pub struct UncleTemplate {
    pub hash: H256,
    pub required: bool,   // always false in practice
    ...
}
``` [1](#0-0) 

The RPC documentation explicitly states: *"Miners must include the uncles marked as `required` in the assembled new block."* Since no uncle is ever marked `required: true`, miners may freely omit any or all suggested uncles. [2](#0-1) 

**Step 2 — `total_uncles_count` is accumulated from actual uncle inclusions.**

In `chain/src/verify.rs`, the `BlockExt` for every new block accumulates the uncle count from the block's actual uncle list:

```rust
let ext = BlockExt {
    total_uncles_count: parent_ext.total_uncles_count
        + block.data().uncles().len() as u64,
    ...
};
``` [3](#0-2) 

If a miner omits uncles, `block.data().uncles().len()` is zero, and the running total is not incremented.

**Step 3 — `epoch_uncles_count` is derived from the accumulated total at epoch boundary.**

In `traits/src/epoch_provider.rs`, when the tail block of an epoch is processed, `epoch_uncles_count` is computed as:

```rust
let epoch_uncles_count = self
    .get_block_ext(&header.hash())
    .expect("stored block ext")
    .total_uncles_count
    - self
        .get_block_ext(&last_block_hash_in_previous_epoch)
        .expect("stored block ext")
        .total_uncles_count;
``` [4](#0-3) 

Any uncle blocks that miners chose not to include are invisible here — they never contributed to `total_uncles_count`.

**Step 4 — `epoch_uncles_count` directly drives the next epoch's difficulty and length.**

In `spec/src/consensus.rs`, `next_epoch_ext` uses `epoch_uncles_count` in every branch of the difficulty calculation:

```rust
let last_epoch_hash_rate = last_difficulty
    * (epoch.length() + epoch_uncles_count)
    / &last_epoch_duration;
```

and

```rust
let last_orphan_rate = RationalU256::new(
    U256::from(epoch_uncles_count),
    last_epoch_length_u256.clone(),
);
``` [5](#0-4) 

A lower `epoch_uncles_count` → lower estimated hash rate → lower next-epoch difficulty and/or longer epoch length.

**Step 5 — The `UnclesVerifier` imposes no floor on uncle count.**

`verification/contextual/src/uncles_verifier.rs` only enforces an *upper* bound (`max_uncles_num`) and validity of included uncles. There is no rule requiring miners to include all eligible uncles. [6](#0-5) 

---

### Impact Explanation

A miner who consistently withholds eligible uncle blocks from their own mined blocks artificially suppresses `epoch_uncles_count`. At the epoch tail block, `next_epoch_ext` observes a lower-than-real orphan rate and computes:

- A **lower difficulty** for the next epoch (easier mining for everyone, but the attacker mines the same fraction of blocks at lower cost).
- A **longer epoch length** (more blocks before the next adjustment, extending the window of reduced difficulty).

Over successive epochs, a miner can keep the difficulty below the honest equilibrium, collecting a disproportionate share of block rewards relative to their true hash contribution. The dampening filter (`TAU = 2`) limits the per-epoch swing to 2×, but a sustained strategy across many epochs compounds the effect. The attack requires no majority hashpower — even a 5–10% miner controls enough blocks to measurably shift `epoch_uncles_count`. [7](#0-6) 

---

### Likelihood Explanation

The entry path is the standard `get_block_template` / `submit_block` RPC flow available to every miner. No special privilege, key, or majority hashpower is required. The miner simply omits the uncle entries from the assembled block before submitting. The `CandidateUncles::prepare_uncles` function in `tx-pool/src/block_assembler/candidate_uncles.rs` returns a suggestion list; the miner is free to ignore it entirely. [8](#0-7) 

The attack is low-cost, undetectable on-chain (an empty uncle list is valid), and repeatable every epoch.

---

### Recommendation

1. **Require inclusion of all eligible uncles up to `max_uncles_num`**: Add a consensus rule that a block must include every uncle that `prepare_uncles` would return (up to the limit), making selective omission a consensus violation.
2. **Decouple difficulty adjustment from miner-controlled uncle inclusion**: Use a separate, non-gameable signal (e.g., a sliding window of observed block intervals) as the primary difficulty input, with uncle rate as a secondary, bounded correction factor.
3. **Add a confirmation delay analogous to the VRF mitigation**: Compute `epoch_uncles_count` over a window that ends several blocks before the epoch tail, so no single miner can influence the final tally by withholding uncles in the last few blocks.

---

### Proof of Concept

```
Epoch N (length = 1000 blocks, target uncle rate = 1/40 ≈ 25 uncles):

Honest scenario:
  epoch_uncles_count = 25
  → next_epoch_diff computed at honest equilibrium

Attacker (10% hashpower, mines ~100 blocks):
  - Attacker mines 100 blocks, each time omitting all suggested uncles
  - Attacker suppresses ~10% of uncle inclusions
  - epoch_uncles_count ≈ 22 instead of 25

  next_epoch_ext sees:
    last_orphan_rate = 22/1000 = 0.022  (vs honest 0.025)
    → lower hash-rate estimate
    → lower compact_target (easier difficulty) for epoch N+1
    → longer epoch length for epoch N+1

  Attacker repeats every epoch → difficulty drifts below honest equilibrium
  → attacker earns more CKB per unit of real hash work
```

    // Apply the dampening filter on hash_rate estimation calculate
    fn bounding_hash_rate(
        &self,
        last_epoch_hash_rate: U256,
        last_epoch_previous_hash_rate: U256,
    ) -> U256 {
        if last_epoch_previous_hash_rate == U256::zero() {
            return last_epoch_hash_rate;
        }

        let lower_bound = &last_epoch_previous_hash_rate / TAU;
        if last_epoch_hash_rate < lower_bound {
            return lower_bound;
        }

        let upper_bound = &last_epoch_previous_hash_rate * TAU;
        if last_epoch_hash_rate > upper_bound {
            return upper_bound;
        }
        last_epoch_hash_rate
    }

    // Apply the dampening filter on epoch_length calculate
    fn bounding_epoch_length(
        &self,
        length: BlockNumber,
        last_epoch_length: BlockNumber,
    ) -> (BlockNumber, bool) {
        let max_length = cmp::min(self.max_epoch_length(), last_epoch_length * TAU);
        let min_length = cmp::max(self.min_epoch_length(), last_epoch_length / TAU);
        if length > max_length {
            (max_length, true)
        } else if length < min_length {
            (min_length, true)
        } else {
            (length, false)
        }
    }

    pub fn verify(&self) -> Result<(), Error> {
        let uncles_count = self.block.data().uncles().len() as u32;

        // if self.block.uncles is empty, return
        if uncles_count == 0 {
            return Ok(());
        }

        // if block is genesis, which is expected with zero uncles, return error
        if self.block.is_genesis() {
            return Err(UnclesError::OverCount {
                max: 0,
                actual: uncles_count,
            }
            .into());
        }

        // verify uncles length =< max_uncles_num
        let max_uncles_num = self.provider.consensus().max_uncles_num() as u32;
        if uncles_count > max_uncles_num {
            return Err(UnclesError::OverCount {
                max: max_uncles_num,
                actual: uncles_count,
            }
            .into());

```rust
    pub fn prepare_uncles(
        &mut self,
        snapshot: &Snapshot,
        current_epoch_ext: &EpochExt,
    ) -> Vec<UncleBlockView> {
        let candidate_number = snapshot.tip_number() + 1;
        let epoch_number = current_epoch_ext.number();
        let max_uncles_num = snapshot.consensus().max_uncles_num();
        let mut uncles: Vec<UncleBlockView> = Vec::with_capacity(max_uncles_num);
        let mut removed = Vec::new();

        for uncle in self.values() {
            if uncles.len() == max_uncles_num {
                break;
            }
            let parent_hash = uncle.header().parent_hash();
            // we should keep candidate util next epoch
            if uncle.compact_target() != current_epoch_ext.compact_target()
                || uncle.epoch().number() != epoch_number
            {
                removed.push(uncle.clone());
            } else if !snapshot.is_main_chain(&uncle.hash())
                && !snapshot.is_uncle(&uncle.hash())
                && uncle.number() < candidate_number
                && (uncles.iter().any(|u| u.hash() == parent_hash)
                    || snapshot.is_main_chain(&parent_hash)
                    || snapshot.is_uncle(&parent_hash))
            {
                uncles.push(uncle.clone());
            }
        }

        for r in removed {
            self.remove_by_number(&r);
        }
        uncles
    }
