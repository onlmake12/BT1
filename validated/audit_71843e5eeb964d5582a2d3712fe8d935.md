Audit Report

## Title
Missing `MAX_RELAY_TXS_NUM_PER_BATCH` Count Check in `TransactionsProcess::execute()` Enables CPU Exhaustion via Oversized Relay Batch — (File: `sync/src/relayer/transactions_process.rs`)

## Summary

`TransactionsProcess::execute()` deserializes every transaction in a received `RelayTransactions` message before any count guard is applied. Both sibling handlers (`TransactionHashesProcess` and `GetTransactionsProcess`) reject messages exceeding `MAX_RELAY_TXS_NUM_PER_BATCH` (32 767) at entry. An unprivileged relay peer can send a single `RelayTransactions` message containing arbitrarily many transactions, forcing the node to perform O(N) deserialization work before the per-hash filter discards the unsolicited entries.

## Finding Description

**Missing check confirmed.** `TransactionsProcess::execute()` at lines 37–57 of `sync/src/relayer/transactions_process.rs` begins iterating immediately with no count guard:

```rust
self.message
    .transactions()
    .iter()
    .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))  // line 48: deserialize
    .filter(|(tx, _)| {                                                          // line 49: filter AFTER
        !tx_filter.contains(&tx.hash())
            && unknown_tx_hashes
                .get_priority(&tx.hash())
                .map(|priority| priority.requesting_peer() == Some(self.peer))
                .unwrap_or_default()
    })
    .collect()
``` [1](#0-0) 

Because Rust's iterator adapters are lazy and fused element-by-element, `.map()` (which calls `to_entity().into_view()`, fully deserializing the transaction) executes for each element *before* `.filter()` decides whether to keep it. The deserialization cost is O(N) in the total number of transactions in the message, regardless of how many survive the filter.

**Check confirmed present in sibling handlers.**

`TransactionHashesProcess::execute()` rejects at lines 29–35:
```rust
if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
    return StatusCode::ProtocolMessageIsMalformed.with_context(...);
}
``` [2](#0-1) 

`GetTransactionsProcess::execute()` rejects at lines 35–39:
```rust
if message_len > MAX_RELAY_TXS_NUM_PER_BATCH {
    return StatusCode::ProtocolMessageIsMalformed.with_context(...);
}
``` [3](#0-2) 

**Why the rate limiter does not mitigate.** The rate limiter in `Relayer::new()` is configured at 30 requests per second per `(peer, message_type)` pair: [4](#0-3) 

It caps the number of `RelayTransactions` *messages*, not the number of transactions *within* each message. At 30 messages/second, each padded to the network frame limit, the per-second deserialization work is unbounded.

**Why outbound truncation does not mitigate.** `ask_for_txs()` truncates outbound `GetRelayTransactions` to `MAX_RELAY_TXS_NUM_PER_BATCH`: [5](#0-4) 

This limits what the node *requests*, but the attacker controls the response and can include any number of additional transactions beyond what was requested. The pre-staging step is not even required — an oversized `RelayTransactions` can be sent at any time and the deserialization cost is incurred regardless.

`MAX_RELAY_TXS_NUM_PER_BATCH` is defined as `32767`: [6](#0-5) 

## Impact Explanation

The attack forces the victim node to perform unbounded deserialization work on every oversized `RelayTransactions` message received. Sustained at 30 messages/second, each carrying transactions up to the network frame size limit, this constitutes a CPU-exhaustion denial-of-service reachable from any connected relay peer with no authentication. This matches **High — Vulnerabilities which could easily crash a CKB node** and/or **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

The attacker requires only a standard relay peer connection — no keys, no stake, no mining power. The fabricated transactions need only be well-formed molecule-encoded structures that pass `check_data()` (verified at `mod.rs` line 136); they do not need to be valid CKB transactions. The attack is repeatable indefinitely (reconnect after a ban, or use multiple peers). The pre-staging step (sending `RelayTransactionHashes` to trigger requests) is optional. [7](#0-6) 

## Recommendation

Add the same count guard at the top of `TransactionsProcess::execute()` that already exists in the two sibling handlers:

```rust
pub fn execute(self) -> Status {
+   let tx_count = self.message.transactions().len();
+   if tx_count > MAX_RELAY_TXS_NUM_PER_BATCH {
+       return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
+           "RelayTransactions count({tx_count}) > MAX_RELAY_TXS_NUM_PER_BATCH({MAX_RELAY_TXS_NUM_PER_BATCH})",
+       ));
+   }
    let shared_state = self.relayer.shared().state();
    ...
```

`MAX_RELAY_TXS_NUM_PER_BATCH` must be imported from `crate::relayer` (already imported in both sibling files). The check must be placed before any iteration or deserialization begins.

## Proof of Concept

1. Connect to a CKB node as a relay peer using the `RelayV3` protocol.
2. *(Optional)* Send `RelayTransactionHashes` messages to cause the node to issue `GetRelayTransactions` requests back to the attacker.
3. Construct a `RelayTransactions` molecule message containing `K × MAX_RELAY_TXS_NUM_PER_BATCH` entries (K ≥ 2), where each entry is a syntactically valid `RelayTransaction` that passes `check_data()`. The entries beyond what was requested will be filtered out, but all will be deserialized.
4. Send the message. Observe that `TransactionsProcess::execute()` returns `Status::ok()` and that the node's CPU usage spikes proportionally to K.
5. Repeat at 30 messages/second to sustain the load.
6. Unit test: in `sync/src/relayer/tests/`, construct a `RelayTransactionsReader` with `2 × MAX_RELAY_TXS_NUM_PER_BATCH` entries, call `TransactionsProcess::new(...).execute()`, and assert the result is `StatusCode::ProtocolMessageIsMalformed` — this assertion currently fails, confirming the missing guard.
