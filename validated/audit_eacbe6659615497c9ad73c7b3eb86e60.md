### Title
Osmosis Fee Validation Assumes 1:1 Exchange Rate Between All Valid Fee Tokens — (`target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

### Summary

The Osmosis-specific `is_fee_sufficient` function in the CosmWasm Pyth contract accepts any Osmosis-allowed fee token at a 1:1 raw-amount ratio with the base fee denom (`uosmo`), without converting using actual spot prices. An unprivileged caller can pay `update_price_feeds` fees in a token worth less than `uosmo` and still have the fee accepted as sufficient.

### Finding Description

In `target_chains/cosmwasm/contracts/pyth/src/contract.rs`, the `#[cfg(feature = "osmosis")]` variant of `is_fee_sufficient` (lines 163–198) sums the raw `u128` amounts of all coins sent by the caller and compares that sum against the base fee amount denominated in `uosmo`:

```rust
let mut total_amount = 0u128;
for coin in &info.funds {
    if coin.denom != state.fee.denom && !is_allowed_tx_fees_denom(deps, &coin.denom) {
        return Err(PythContractError::InvalidFeeDenom { ... })?;
    }
    total_amount = total_amount.checked_add(coin.amount.u128())...;
}
let base_denom_fee = get_update_fee(deps, data)?;
// NOTE: ... we are keeping the base fee amount same for each valid denom -> 1
// but this logic will be updated to use spot price for different valid tokens in future
Ok(base_denom_fee.amount.u128() <= total_amount)
``` [1](#0-0) 

The companion function `get_update_fee_for_denom` (lines 721–736) exhibits the same pattern — it returns the same raw amount for any valid denom without any exchange-rate conversion:

```rust
// for other valid denoms too we are using the base amount as 1
// this will be change later on to add custom logic using spot price for valid tokens
Ok(coin(get_update_fee_amount(deps, vaas)?, denom))
``` [2](#0-1) 

The code itself acknowledges the problem with a `// FIXME` and a `// NOTE` comment, and the `is_allowed_tx_fees_denom` helper only checks whether the Osmosis `txfees` module has a pool for the denom — it does not retrieve or apply the pool's spot price. [3](#0-2) 

### Impact Explanation

If the governance-controlled fee is ever raised to a non-trivial amount (currently 1 `uosmo`, acknowledged as "almost negligible"), any caller can satisfy the fee check by sending the same raw integer count of a cheaper Osmosis-allowed token (e.g., `uion` or an IBC asset whose spot price is a fraction of `uosmo`). The protocol would accept the price update while collecting less economic value than intended. This is the direct analog of the external report: a token is assumed to be worth the same as the base unit without querying an actual price feed.

### Likelihood Explanation

The entry path is fully permissionless — any account can call `update_price_feeds` on the Osmosis deployment and supply funds in a non-base denom. No privileged role is required. The only precondition is that the chosen denom is registered in Osmosis's `txfees` module, which is a public, on-chain registry. The current fee of 1 `uosmo` makes the practical economic loss negligible today, but the structural flaw is live and reachable.

### Recommendation

Before comparing `total_amount` against `base_denom_fee.amount`, convert each non-base coin to its `uosmo` equivalent using the Osmosis `SpotPrice` query (available via `TxfeesQuerier` or the `PoolManagerQuerier`). Only the converted sum should be compared against the required fee. The `get_update_fee_for_denom` function should similarly return the fee amount denominated in the requested token at the current spot price, not the raw `uosmo` amount.

### Proof of Concept

1. Osmosis deployment has fee set to `N uosmo` (e.g., `N = 1000` after a governance update).
2. Attacker observes that `uion` trades at 0.01 `uosmo` on Osmosis.
3. Attacker calls `update_price_feeds` sending `N uion` (worth `0.01 * N uosmo`).
4. `is_allowed_tx_fees_denom` returns `true` (uion has a pool in the txfees module).
5. `total_amount = N`; `base_denom_fee.amount = N`; check `N <= N` passes.
6. Price update is accepted; attacker paid 1% of the intended fee value. [4](#0-3)

fn is_allowed_tx_fees_denom(deps: &Deps, denom: &String) -> bool {
    // TxFeesQuerier uses stargate queries which we can't mock as of now.
    // The capability has not been implemented in `cosmwasm-std` yet.
    // Hence, we are hacking it with a feature flag to be able to write tests.
    // FIXME
    #[cfg(test)]
    if denom == "uion"
        || denom == "ibc/FF3065989E34457F342D4EFB8692406D49D4E2B5C70F725F127862E22CE6BDCD"
    {
        return true;
    }

    let querier = TxfeesQuerier::new(&deps.querier);
    match querier.denom_pool_id(denom.to_string()) {
        Ok(_) => true,
        Err(_) => false,
    }
}

pub fn get_update_fee_for_denom(deps: &Deps, vaas: &[Binary], denom: String) -> StdResult<Coin> {
    let config = config_read(deps.storage).load()?;

    // if the denom is not a base denom it should be an allowed one
    if denom != config.fee.denom && !is_allowed_tx_fees_denom(deps, &denom) {
        return Err(PythContractError::InvalidFeeDenom { denom })?;
    }

    // the base fee is set to -> denom = base denom of a chain, amount = 1
    // which is very minimal
    // for other valid denoms too we are using the base amount as 1
    // base amount is multiplied to number of vaas to get the total amount

    // this will be change later on to add custom logic using spot price for valid tokens
    Ok(coin(get_update_fee_amount(deps, vaas)?, denom))
}
