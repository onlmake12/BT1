Audit Report

## Title
Unbounded response amplification via duplicate indexes in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` accepts up to 32,767 transaction indexes from an untrusted peer but performs no deduplication and no byte-cap check on the assembled response. An attacker sending 32,767 copies of index `0` forces the node to clone and serialize the same transaction (e.g., the cellbase) 32,767 times into a single `BlockTransactions` message. The sibling handlers `GetTransactionsProcess` and `GetBlockProposalProcess` both implement deduplication and byte-cap protections; `GetBlockTransactionsProcess` does not.

## Finding Description
**Root cause:** `GetBlockTransactionsProcess::execute` in `sync/src/relayer/get_block_transactions_process.rs` only checks that the number of indexes does not exceed `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767). It never deduplicates the index list and never caps the total serialized size of the response.

**Code path:**

1. The only guard (L37–43) rejects messages with more than 32,767 indexes but allows 32,767 identical indexes:
```rust
if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH { ... }
```

2. Every index, including duplicates, is resolved to a cloned `TransactionView` (L61–71):
```rust
let transactions = self.message.indexes().iter()
    .filter_map(|i| block.transactions().get(Into::<u32>::into(i) as usize).cloned())
    .collect::<Vec<_>>();
```

3. All clones are serialized into a single `BlockTransactions` message and sent with no size limit (L80–97). No `MAX_RELAY_TXS_BYTES_PER_BATCH` check exists anywhere in this handler.

**Why existing checks fail:**
- The count check at L37 is satisfied by 32,767 copies of index `0` (count == 32,767 ≤ 32,767).
- The rate limiter (keyed by `(PeerIndex, message_type)`, 30 req/s) bounds message *count*, not message *size*.
- `MAX_RELAY_TXS_BYTES_PER_BATCH` is not imported or referenced in this file at all.

**Contrast with sibling handlers:**
- `GetTransactionsProcess` (L54–61) builds a `HashSet` from tx hashes and rejects if `message_len != set.len()`, then enforces `MAX_RELAY_TXS_BYTES_PER_BATCH` (L90).
- `GetBlockProposalProcess` (L47–52) collects proposals into a `HashSet` and rejects duplicates, then enforces `MAX_RELAY_TXS_BYTES_PER_BATCH` (L84).

## Impact Explanation
**High — "Vulnerabilities which could easily crash a CKB node"** and **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

A CKB mainnet cellbase transaction with a full witness is several kilobytes (~4 KB). Sending 32,767 copies of index `0` causes the node to allocate a `Vec<TransactionView>` of ~128 MB before serialization, then serialize and send the full payload. The rate limiter permits 30 such requests per second per peer. With `MAX_RELAY_PEERS = 128` peers each sending 30 requests/second, the node faces:
- **Memory:** Each in-flight response allocates ~128 MB of cloned transaction data before serialization, causing rapid OOM.
- **Bandwidth:** 128 peers × 30 req/s × ~128 MB = hundreds of GB/s outbound — far beyond any node's capacity.

Both OOM crash and network congestion are concrete, not hypothetical.

## Likelihood Explanation
- Requires only a valid P2P connection — no key, no PoW, no stake.
- The attacker needs only one stored block hash, trivially obtained from any public block explorer or by passively observing the network.
- The exploit is deterministic and repeatable; a single malicious peer can sustain it indefinitely within the rate limit.
- The rate limiter does not bound response size, only request count, so it provides no meaningful mitigation.

## Recommendation
1. **Deduplicate indexes** before lookup, mirroring `GetTransactionsProcess`:
```rust
let indexes_set: HashSet<u32> = self.message.indexes().iter()
    .map(Into::<u32>::into).collect();
if indexes_set.len() != self.message.indexes().len() {
    return StatusCode::RequestDuplicate.with_context("Duplicate indexes");
}
```
2. **Import and enforce `MAX_RELAY_TXS_BYTES_PER_BATCH`** on the assembled response, aborting or splitting once the total serialized size exceeds 1 MiB, consistent with `GetTransactionsProcess` and `GetBlockProposalProcess`.

## Proof of Concept
```
1. Connect to a CKB node over P2P (RelayV3 protocol).
2. Obtain any stored block hash H (e.g., genesis block hash, publicly known).
3. Craft a GetBlockTransactions message:
     block_hash    = H
     indexes       = [0] * 32767   # 32767 copies of index 0
     uncle_indexes = []
4. Send the message.
5. Observe: the node allocates and serializes 32767 copies of the cellbase
   transaction and sends a single BlockTransactions response.
6. Assert: response_bytes >> MAX_RELAY_TXS_BYTES_PER_BATCH (1 MiB).
7. Repeat at 30 req/s per peer connection to sustain bandwidth/memory exhaustion.
8. Open 128 peer connections to multiply the effect by MAX_RELAY_PEERS.
```
