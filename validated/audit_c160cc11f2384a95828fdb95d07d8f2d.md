### Title
Unbounded MMR Root Computation on Off-Chain `last_hash` in `GetBlocksProof`/`GetTransactionsProof` — (`util/light-client-protocol-server/src/lib.rs`, `get_blocks_proof.rs`, `get_transactions_proof.rs`)

---

### Summary

An unprivileged remote peer can repeatedly send `GetBlocksProof` or `GetTransactionsProof` messages with a `last_hash` that is not on the current main chain. The server unconditionally calls `reply_tip_state`, which internally calls `get_verifiable_tip_header` → `chain_root_mmr(tip_number - 1).get_root()` (O(log H) DB reads) on every such message. No rate limiter exists in `LightClientProtocol`, and the handler returns `Status::ok()`, so no ban is ever issued. The attacker's cost is a single valid off-chain block hash (any uncle or reorged block) plus network bandwidth.

---

### Finding Description

**Entry point — `GetBlocksProofProcess::execute`:**

In `util/light-client-protocol-server/src/components/get_blocks_proof.rs`, lines 44–49:

```rust
let last_block_hash = self.message.last_hash().to_entity();
if !snapshot.is_main_chain(&last_block_hash) {
    return self
        .protocol
        .reply_tip_state::<packed::SendBlocksProof>(self.peer, self.nc)
        .await;
}
```

When `last_hash` is off-chain, execution immediately branches to `reply_tip_state`. The identical pattern exists in `GetTransactionsProofProcess::execute` (lines 43–48 of `get_transactions_proof.rs`).

**Expensive work inside `reply_tip_state` → `get_verifiable_tip_header`:**

In `util/light-client-protocol-server/src/lib.rs`, lines 127–155:

```rust
let mmr = snapshot.chain_root_mmr(tip_block.number() - 1);
match mmr.get_root() {
    Ok(root) => root,
    ...
}
```

`chain_root_mmr(tip_number - 1).get_root()` performs O(log H) RocksDB reads to reconstruct the MMR root for the entire chain history. On a mature mainnet chain this is non-trivial I/O.

**`reply_tip_state` always returns `Status::ok()`** (line 178 of `lib.rs`), so the ban-check gate in `received` (lines 81–91) never fires.

**No rate limiter exists in `LightClientProtocol`:**

The struct definition (`lib.rs`, lines 26–29) contains only `shared: Shared`. Compare with `sync/src/relayer/mod.rs` line 81, where `Relayer` carries `rate_limiter: RateLimiter<(PeerIndex, u32)>`. The `constant.rs` file for the light-client server defines only message-count caps (`GET_BLOCKS_PROOF_LIMIT = 1000`) and a ban duration for malformed bytes — no per-peer request rate limit of any kind.

---

### Impact Explanation

Each crafted message forces the server to:
1. Acquire a snapshot.
2. Compute `chain_root_mmr(tip_number - 1).get_root()` — O(log H) DB reads.
3. Build a full `VerifiableHeader`.
4. Serialize and send a `SendBlocksProof` response.

Because `received` takes `&mut self`, the protocol handler processes one message at a time; a flood of these messages queues behind each other, starving legitimate light-client peers of service. Sustained flooding from even a single connection degrades the node's I/O and CPU, slowing block propagation and sync on the same process, contributing to network-level congestion at low attacker cost.

---

### Likelihood Explanation

- **Precondition**: attacker needs one valid off-chain block hash. Uncle hashes are broadcast in every block; reorged block hashes are observable from any chain explorer or by briefly mining a fork. This is trivially satisfied.
- **No authentication or PoW required**: any peer that completes the p2p handshake can send light-client messages.
- **No self-limiting mechanism**: the server never bans, never throttles, and the message is well-formed (passes the `from_slice` check), so the only guard that exists — the malformed-message ban — does not apply.

---

### Recommendation

1. Add a per-peer rate limiter to `LightClientProtocol` (mirroring the `RateLimiter` already present in `Relayer`), keyed on `(PeerIndex, message_type)`.
2. Cache the result of `get_verifiable_tip_header` with a short TTL (e.g., one block interval) so repeated calls within the same block epoch are served from memory rather than recomputing the MMR root.
3. Consider returning a non-OK status (e.g., `StatusCode::TooManyRequests`) when the rate limit is exceeded, which would trigger the existing ban logic.

---

### Proof of Concept

```
1. Connect to a CKB node with light-client protocol enabled.
2. Observe any uncle block hash from a recent block announcement (broadcast in normal sync).
3. In a tight loop, send:
     GetBlocksProof {
         last_hash:    <uncle_block_hash>,   // not on main chain
         block_hashes: [<any_valid_hash>],
     }
4. Observe: server never bans the peer; each message triggers
   chain_root_mmr(tip_number-1).get_root() in get_verifiable_tip_header();
   node I/O and CPU climb proportionally to message rate.
```
