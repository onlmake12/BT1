### Title
Acknowledged Off-by-One in RFC0044 MMR Activation Epoch Check Allows First Activation-Epoch Block to Omit MMR Extension — (`tx-pool/src/block_assembler/mod.rs` and `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

Both the block assembler (`build_extension`) and the block verifier (`BlockExtensionVerifier`) determine whether RFC0044 (the MMR/light-client extension) is active by passing the **parent block's** epoch number to `rfc0044_active`, rather than the epoch number of the block being built or verified. This creates a one-block gap at the exact activation epoch boundary: the first block of the activation epoch is neither required to include, nor assembled with, the MMR chain-root extension. The code itself contains an explicit acknowledgment that this is a bug, preserved only for internal consistency.

---

### Finding Description

`rfc0044_active` in `spec/src/consensus.rs` checks:

```rust
target >= rfc0044_active_epoch
``` [1](#0-0) 

The block assembler calls it with the **tip** (parent) epoch number:

```rust
// The use of the epoch number of the tip here leads to an off-by-one bug,
// so be careful, it needs to be preserved for consistency reasons and not fixed directly.
let mmr_activate = snapshot
    .consensus()
    .rfc0044_active(tip_header.epoch().number());
``` [2](#0-1) 

The block verifier (`BlockExtensionVerifier::verify`) makes the identical call with the parent's epoch number:

```rust
let mmr_active = self
    .context
    .consensus
    .rfc0044_active(self.parent.epoch().number());
``` [3](#0-2) 

Let `N` = `RFC0044_ACTIVE_EPOCH` (8651 on mainnet, 5711 on testnet). For the **first block of epoch N**, the parent is the last block of epoch N−1, so `parent.epoch().number() = N−1`. The check `N−1 >= N` is **false**, so:

- The assembler does **not** include the MMR extension in that block.
- The verifier does **not** require the MMR extension in that block.

The first block of the activation epoch is therefore accepted on-chain **without** a committed MMR chain root, creating a one-block gap in the MMR chain.

---

### Impact Explanation

RFC0044 (CKB light client protocol) relies on the MMR chain root committed in each block's extension field to allow light clients to verify block inclusion without downloading the full chain. The one-block gap at the activation epoch boundary means:

- Light clients **cannot** use the MMR to verify the first block of the activation epoch.
- A malicious full node can serve a fabricated version of that specific block to a light client, and the light client has no MMR-based means to detect the forgery for that block.
- All subsequent blocks in epoch N and beyond do commit the MMR root (since their parent's epoch number is N, satisfying `N >= N`), so the gap is exactly one block wide but is permanent and unfixable after the fact.

---

### Likelihood Explanation

The activation epoch boundary is a fixed, predictable point. A miner who mines the first block of the activation epoch will naturally produce a block without the MMR extension (because the assembler does not include it). No special action is required; the off-by-one fires automatically. The verifier accepts the block. The gap is therefore **guaranteed** to occur at activation, not merely possible.

For mainnet (epoch 8651) and testnet (epoch 5711), this has already occurred. For any future network using this codebase with a configured `RFC0044_ACTIVE_EPOCH`, the gap will occur at that epoch's first block.

---

### Recommendation

The fix requires changing **both** the assembler and the verifier simultaneously to use the **current block's** epoch number rather than the parent's:

- In `build_extension`: derive the new block's epoch from the snapshot's next-epoch calculation rather than `tip_header.epoch().number()`.
- In `BlockExtensionVerifier::verify`: use `block.header().epoch().number()` instead of `self.parent.epoch().number()`.

The code comment acknowledges the bug but warns it "needs to be preserved for consistency reasons and not fixed directly" — meaning a coordinated, atomic change to both sites is required to avoid producing blocks that the verifier rejects.

---

### Proof of Concept

1. Configure a devnet with `RFC0044_ACTIVE_EPOCH = E` for some future epoch `E`.
2. Mine until the last block of epoch `E−1`.
3. Call `get_block_template` RPC. Observe that the returned template has **no** `extension` field (the assembler omits it because `tip.epoch().number() = E−1 < E`).
4. Submit the block. The verifier accepts it (`parent.epoch().number() = E−1`, `mmr_active = false`, `extra_fields_count = 0` → OK).
5. Mine the second block of epoch `E`. Its template **does** include the MMR extension (parent is now in epoch `E`).
6. The MMR chain root in block 2 of epoch `E` covers all blocks including the first block of epoch `E`, but the first block of epoch `E` itself carries no MMR commitment — the gap is confirmed. [4](#0-3) [5](#0-4) [6](#0-5)

    pub fn rfc0044_active(&self, target: EpochNumber) -> bool {
        let rfc0044_active_epoch = match self.id.as_str() {
            mainnet::CHAIN_SPEC_NAME => softfork::mainnet::RFC0044_ACTIVE_EPOCH,
            testnet::CHAIN_SPEC_NAME => softfork::testnet::RFC0044_ACTIVE_EPOCH,
            _ => 0,
        };

    }

    pub(crate) fn build_extension(snapshot: &Snapshot) -> Result<Option<packed::Bytes>, AnyError> {
        let tip_header = snapshot.tip_header();
        // The use of the epoch number of the tip here leads to an off-by-one bug,
        // so be careful, it needs to be preserved for consistency reasons and not fixed directly.
        let mmr_activate = snapshot
            .consensus()
            .rfc0044_active(tip_header.epoch().number());
        if mmr_activate {
            let chain_root = snapshot
                .chain_root_mmr(tip_header.number())
                .get_root()
                .map_err(|e| InternalErrorKind::MMR.other(e))?;
            let bytes = chain_root.calc_mmr_hash().as_bytes().into();
            Ok(Some(bytes))
        } else {
            Ok(None)
        }
    }
