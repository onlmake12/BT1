Audit Report

## Title
`CapacityVerifier::valid_dao_withdraw_transaction()` Incorrectly Bypasses `OutputsSumOverflow` for Deposit-Phase DAO Inputs, Enabling Arbitrary CKB Inflation — (`File: verification/src/transaction_verifier.rs`)

## Summary

`CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` for any transaction whose inputs contain a DAO-typed cell, including deposit-phase cells (`cell_data == 0x0000000000000000`). This causes `CapacityVerifier::verify()` to skip the `OutputsSumOverflow` guard for the entire transaction. Because the DAO type script (dao.c) only validates DAO-typed cells and not non-DAO outputs, a deposit→prepare transaction can include non-DAO outputs with arbitrarily inflated capacity that no verifier checks, allowing any user to mint CKB from nothing.

## Finding Description

`CapacityVerifier::valid_dao_withdraw_transaction()` checks only whether any input uses the DAO type script, with no inspection of cell data:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
```

This causes `CapacityVerifier::verify()` to skip the overflow guard for all DAO-input transactions, including deposit→prepare (phase 1) transactions where `deposited_block_number == 0`:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    // skipped for deposit→prepare transactions
    if inputs_sum < outputs_sum {
        return Err(TransactionError::OutputsSumOverflow { ... }.into());
    }
}
```

The bypass is only semantically correct for phase 2 (prepare→withdraw) transactions, where the DAO script legitimately permits outputs to exceed inputs by the accrued interest. For phase 1 (deposit→prepare), the DAO script enforces only that `prepare_output.capacity == deposit_output.capacity` for the DAO-typed cells themselves — explicitly documented in the codebase: `// NOTE: dao.c uses deposit_header to ensure the prepare_output.capacity == deposit_output.capacity`. Non-DAO outputs in the same transaction are not validated by the DAO script and, with the `OutputsSumOverflow` check skipped, are not validated by any other verifier either.

`DaoCalculator::transaction_maximum_withdraw()` correctly distinguishes the two phases by reading cell data and checking `deposited_block_number > 0` before allowing capacity above face value. `CapacityVerifier` makes no such distinction.

## Impact Explanation

This is a consensus-level economic vulnerability matching **"Vulnerabilities which could easily damage CKB economy" — Critical (15001–25000 points)**. An attacker can craft a deposit→prepare transaction that mints an unbounded amount of CKB into non-DAO outputs. Every node that accepts such a transaction inflates the total CKB supply, breaking the fixed-supply economic model and the security guarantees of all CKB holders. The inflation is permanent and consensus-visible.

## Likelihood Explanation

The attack requires only a DAO deposit cell (trivially created by any user via a standard deposit transaction) and a crafted deposit→prepare transaction submitted via the standard `send_transaction` RPC. No privileged access, no majority hashpower, no social engineering, and no victim interaction is required. The attack is deterministic, repeatable, and executable by any unprivileged network participant.

## Recommendation

`valid_dao_withdraw_transaction()` must restrict its `true` return to phase 2 (prepare→withdraw) inputs only — i.e., inputs where `deposited_block_number > 0`. `CapacityVerifier` must be given access to a `CellDataProvider` (as `DaoScriptSizeVerifier` already has) to load and inspect cell data:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| {
            cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash)
                && self.data_loader.load_cell_data(cell_meta)
                    .map(|data| data.len() == 8 && LittleEndian::read_u64(&data) > 0)
                    .unwrap_or(false)
        })
}
```

This mirrors the existing phase distinction in `DaoCalculator::transaction_maximum_withdraw()` and ensures the bypass is applied only when the DAO script is actually permitted to produce outputs exceeding inputs.

## Proof of Concept

**Step 1 — Create a DAO deposit cell** by sending a standard deposit transaction (any user can do this).

**Step 2 — Craft a deposit→prepare transaction** with:
- Input 0: the DAO deposit cell (e.g., 100 CKB, `cell_data = 0x0000000000000000`)
- Input 1: a regular cell (e.g., 1 CKB)
- Output 0: DAO prepare cell (100 CKB, `cell_data = deposit_block_number`) — passes DAO script validation
- Output 1: regular cell with inflated capacity (e.g., 1,000,000 CKB) — unchecked

**Step 3 — Submit via `send_transaction` RPC.**

**Expected result:** `CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` because Input 0 has the DAO type script. The `OutputsSumOverflow` check is skipped. The DAO script validates only Output 0. Output 1 (1,000,000 CKB) is accepted with no capacity conservation check. The attacker receives 999,999 CKB minted from nothing.

A unit test can be written directly against `CapacityVerifier` by constructing a `ResolvedTransaction` with a DAO deposit cell input (`cell_data` all zeros) alongside a regular input, and a regular output whose capacity exceeds the regular input's capacity, then asserting that `verifier.verify()` incorrectly returns `Ok(())`.
