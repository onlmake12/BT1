## Analysis

### Race Condition in `HeaderMapKernel::get()`

The `get()` function has **no outer lock** protecting the entire read-promote-from-backend operation: [1](#0-0) 

The sequence is:
1. `memory.get_refresh(hash)` — acquires/releases `RwLock` write lock, returns `None`
2. `backend.remove(hash)` — sled atomic remove, **no lock held**
3. `memory.insert(view.clone())` — acquires/releases `RwLock` write lock

Two threads can both pass step 1 with `None`, then race on step 2. Sled's `remove` is atomic at the key level, so exactly one thread gets `Some(view)` and the other gets `None`. The losing thread returns `None` from `get()`. [2](#0-1) 

### Propagation to the Panic

`get_header_index_view()` with `store_first = false` (i.e., `BLOCK_STORED` is not set): [3](#0-2) 

It calls `header_map().get(hash)` first. If the race causes that to return `None`, and the block is not in the store (because `BLOCK_STORED` is false), the entire function returns `None`.

That `None` hits the `unwrap_or_else(|| panic!(...))` directly in `HeaderAcceptor::accept()`: [4](#0-3) 

### Correction to the Question's Path

The question describes the panic as being triggered **via `get_ancestor()` traversal**. This is inaccurate. The panic in `headers_process.rs` is triggered **directly** by `get_header_index_view()` returning `None` — `get_ancestor()` is not in this call chain. There is a separate `unwrap_or_else(|| panic!(...))` in `get_locator()` that does use `get_ancestor()`: [5](#0-4) 

But that is a different code path.

### Race Window Assessment

The race window is the interval between Thread B's `memory.get_refresh()` returning `None` and Thread A's `memory.insert()` completing — a window that spans Thread A's entire sled `remove()` call. Sled operations involve disk I/O and are not instantaneous, making this window non-trivial and reproducible under load.

### Attack Feasibility

- **Entry point**: Two attacker-controlled peers send the same valid header simultaneously via P2P sync
- **Precondition**: Node is in IBD with the header map's memory limit exceeded, causing headers to be evicted to the sled backend — a normal operational condition
- **Trigger**: Both threads enter `HeaderAcceptor::accept()` for the same `HEADER_VALID` hash, both call `HeaderMapKernel::get()`, one loses the sled race and gets `None`
- **Impact**: `unwrap_or_else(|| panic!(...))` fires, crashing the node

The timing requirement is tight but achievable with repeated attempts, especially under IBD load when sled I/O latency widens the race window.

### Title
Concurrent `HeaderMapKernel::get()` Race on Sled Backend Causes Node Crash via `unwrap_or_else(panic)` — (`shared/src/types/header_map/kernel_lru.rs`)

### Summary
`HeaderMapKernel::get()` promotes entries from the sled backend to memory without holding any lock across the check-then-remove-then-insert sequence. Two concurrent sync threads processing the same `HEADER_VALID` header can both miss the memory map, race on `backend.remove()`, and one thread receives `None`. This `None` propagates through `get_header_index_view()` to an `unwrap_or_else(|| panic!(...))` in `HeaderAcceptor::accept()`, crashing the node.

### Finding Description
`HeaderMapKernel::get()` performs a non-atomic read-promote sequence:
1. `memory.get_refresh(hash)` (write-locked, released)
2. `backend.remove(hash)` (no lock held — sled atomic remove)
3. `memory.insert(view.clone())` (write-locked, released)

Two threads can both observe `None` from step 1, then race on step 2. Sled's `remove` is key-level atomic, so one thread wins `Some(view)` and the other gets `None`. The losing thread returns `None` from `get()`.

`get_header_index_view()` with `store_first = false` calls `header_map().get(hash)` and, if that returns `None`, falls back to the chain store. For a header not yet stored on-chain (`BLOCK_STORED` unset), the store fallback also returns `None`, so `get_header_index_view()` returns `None`.

`HeaderAcceptor::accept()` calls `get_header_index_view()` under the `HEADER_VALID` branch and immediately panics on `None` with no recovery path.

### Impact Explanation
The node process terminates via `panic!`. Any peer can trigger this by connecting two sockets and sending the same syntactically valid header from both simultaneously while the node is in IBD with headers evicted to sled.

### Likelihood Explanation
During IBD the memory limit is routinely exceeded, pushing headers to sled. The race window spans a sled disk I/O operation, making it non-trivially wide. An attacker with two peer connections can attempt the race repeatedly at negligible cost. Reliable exploitation requires timing precision but is achievable under load.

### Recommendation
Protect the entire check-remove-insert sequence in `HeaderMapKernel::get()` with a per-key or global mutex, or replace the destructive `backend.remove()` promotion with a non-destructive `backend.get()` followed by a deferred eviction pass, so a concurrent `None` return is impossible for a key that exists.

// Assert one returns None:
assert!(r1.is_none() || r2.is_none(),

// The None-returning thread's get_header_index_view() returns None,
// hitting the unwrap_or_else(|| panic!(...)) in HeaderAcceptor::accept()
