### Title
Provider Accrued Fees Permanently Locked When Provider Is a Smart Contract Without ETH Receive Capability - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `withdraw` function in `Entropy.sol` hardcodes `msg.sender` as the ETH recipient. If a registered provider is a smart contract that lacks a `receive()` or `fallback()` function, the native-token transfer will always revert and the provider's accrued fees will be permanently locked inside the Entropy contract with no alternative recovery path.

---

### Finding Description

When a user calls any `requestV2` / `requestWithCallback` variant, `requestHelper` distributes the entire `msg.value` between the provider and Pyth:

```solidity
// Entropy.sol L236-239
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The provider's share accumulates in `providerInfo.accruedFeesInWei`. The only way for a provider to claim these fees is through `withdraw`:

```solidity
// Entropy.sol L150-164
function withdraw(uint128 amount) public override {
    EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[msg.sender];
    require(providerInfo.accruedFeesInWei >= amount, "Insufficient balance");
    providerInfo.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "withdrawal to msg.sender failed");
    ...
}
``` [2](#0-1) 

The recipient is unconditionally `msg.sender`. If the provider is a smart contract without a `receive()` or `fallback()` function, the low-level `call{value: amount}("")` returns `success = false`, the `require` reverts the entire transaction (rolling back the balance decrement), and the fees remain trapped in `accruedFeesInWei` indefinitely.

There is no `withdrawTo(address recipient, uint128 amount)` overload. The only escape hatch is `withdrawAsFeeManager`, but that requires the provider to have previously called `setFeeManager` to designate a rescue address — a step that a misconfigured or immutable provider contract may never have taken. [3](#0-2) 

---

### Impact Explanation

A provider that is a smart contract without ETH receive capability accumulates fees that can never be withdrawn. Every user request that names this provider increases the locked balance. The funds are not lost from the contract's accounting, but they are permanently inaccessible to the provider, constituting a loss of provider funds. This is directly analogous to the AmarokFacet pattern: a hardcoded `msg.sender` recipient that cannot receive native tokens causes funds to be locked.

---

### Likelihood Explanation

Likelihood is **low**. It requires:
1. A provider to be a smart contract (not an EOA).
2. That contract to lack a `receive()` / `fallback()` function.
3. No fee manager to have been set via `setFeeManager`.

All three conditions are plausible for automated on-chain keeper contracts or factory-deployed provider contracts that were not designed to receive ETH directly.

---

### Recommendation

Add a `withdrawTo` function that accepts an explicit recipient address, mirroring the fix applied to AmarokFacet:

```solidity
function withdrawTo(address payable recipient, uint128 amount) public {
    EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[msg.sender];
    require(providerInfo.accruedFeesInWei >= amount, "Insufficient balance");
    providerInfo.accruedFeesInWei -= amount;
    (bool sent, ) = recipient.call{value: amount}("");
    require(sent, "withdrawal failed");
    emit EntropyEventsV2.Withdrawal(msg.sender, recipient, amount, bytes(""));
}
```

This allows a provider smart contract to redirect fees to any address it controls (e.g., a treasury EOA), eliminating the lock-up risk.

---

### Proof of Concept

1. Deploy `ProviderContract` — a Solidity contract with no `receive()` or `fallback()` function.
2. From `ProviderContract`, call `Entropy.register(...)` to register as a provider.
3. Have users call `Entropy.requestV2{value: fee}(address(ProviderContract), ...)`. Each call increments `_state.providers[address(ProviderContract)].accruedFeesInWei`.
4. From `ProviderContract`, call `Entropy.withdraw(accruedAmount)`.
5. Inside `withdraw`, `msg.sender.call{value: accruedAmount}("")` targets `ProviderContract`, which has no `receive()` — the call returns `(false, "")`.
6. `require(sent, "withdrawal to msg.sender failed")` reverts the transaction.
7. `accruedFeesInWei` is unchanged; the fees are permanently locked with no recovery path (assuming `setFeeManager` was never called). [4](#0-3)
