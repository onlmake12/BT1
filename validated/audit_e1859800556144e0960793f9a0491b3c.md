### Title
Unbounded `tx.gasprice` in Keeper Fee Calculation Allows Subscription Balance Drain — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `_processFeesAndPayKeeper` function in `Scheduler.sol` computes the keeper reimbursement as `(gasUsed + GAS_OVERHEAD) * tx.gasprice`. Because `tx.gasprice` is fully attacker-controlled and no on-chain maximum is enforced, any unprivileged keeper can inflate their reimbursement by submitting `updatePriceFeeds` with an artificially high gas price, draining the subscription balance far beyond the actual network cost.

### Finding Description

`_processFeesAndPayKeeper` calculates the total keeper fee as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [1](#0-0) 

`tx.gasprice` is the gas price of the incoming transaction, which is set entirely by the caller. There is no on-chain cap or governance-enforced maximum gas price checked before this multiplication. The keeper's net profit from a single call is:

```
profit = GAS_OVERHEAD * tx.gasprice + singleUpdateKeeperFeeInWei * numPriceIds
```

Because the keeper is reimbursed for `GAS_OVERHEAD` gas units at their chosen `tx.gasprice`, setting an arbitrarily high gas price directly scales the profit extracted from the subscription, independent of actual network conditions.

`updatePriceFeeds` is permissionless — any address can call it: [2](#0-1) 

The only on-chain protection is an `InsufficientBalance` revert if the subscription cannot cover the inflated fee, which means the attack succeeds until the subscription is fully drained. [3](#0-2) 

### Impact Explanation

A malicious keeper can drain any active subscription's balance in a single transaction by submitting `updatePriceFeeds` with `tx.gasprice` set to an extreme value (e.g., `type(uint128).max / GAS_OVERHEAD`). The subscription owner — who pre-funded the contract expecting normal gas prices — loses their entire deposited balance. The keeper pays the inflated gas cost to the network but is fully reimbursed by the contract, netting `GAS_OVERHEAD * tx.gasprice` per call. On EIP-1559 chains, `tx.gasprice = min(maxFeePerGas, baseFee + maxPriorityFeePerGas)`, so the keeper can set `maxFeePerGas` to any value and the contract will honor it.

### Likelihood Explanation

`updatePriceFeeds` is callable by any address with no access control. The attack requires only a valid price update (obtainable from Hermes) and a transaction with an inflated gas price. The attacker's only cost is the actual gas consumed at the inflated price, which is immediately returned by the contract. This is a straightforward, low-cost, single-transaction exploit available to any Entropy/Scheduler user.

### Recommendation

Enforce an on-chain maximum gas price cap in `_processFeesAndPayKeeper`. For example:

```solidity
uint256 effectiveGasPrice = tx.gasprice < _state.maxGasPriceInWei
    ? tx.gasprice
    : _state.maxGasPriceInWei;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

`maxGasPriceInWei` should be a governance-settable parameter (analogous to `singleUpdateKeeperFeeInWei` in `SchedulerGovernance.sol`). [4](#0-3) 

Alternatively, cap the reimbursable gas price at a multiple of the current `block.basefee` (e.g., `2 * block.basefee`) to bound the keeper's profit while still covering legitimate high-priority submissions.

### Proof of Concept

1. Subscriber creates a subscription with 1 price feed and deposits 1 ETH.
2. Attacker fetches a valid price update from Hermes for the subscribed feed.
3. Attacker calls `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = 10_000 gwei`.
4. Inside `_processFeesAndPayKeeper`:
   - `gasUsed ≈ 200_000` (actual execution cost)
   - `GAS_OVERHEAD` (a fixed constant, e.g., `100_000`)
   - `gasCost = (200_000 + 100_000) * 10_000 gwei = 3_000_000_000 gwei = 3 ETH`
5. The contract attempts to pay `3 ETH` to the attacker. If the subscription balance is ≥ 3 ETH, the attacker receives it. If not, the call reverts — but the attacker can tune `tx.gasprice` to exactly drain the available balance.
6. Attacker's actual gas cost: `200_000 * 10_000 gwei = 2 ETH` (paid to the network). Net profit: `1 ETH` extracted from the subscription. [5](#0-4)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
