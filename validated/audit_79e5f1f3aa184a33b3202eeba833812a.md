Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Attacker-Controlled `(from, to)` PeerIds — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter`, a `governor::RateLimiter` backed by `HashMapStateStore<(PeerId, PeerId, u32)>`, grows without bound when an attacker sends `ConnectionRequest` messages with unique `(from, to)` pairs over a persistent session. The `retain_recent()` cleanup is called only in `disconnected()`, never in the periodic `notify()` handler, so the HashMap is never pruned while the attacker remains connected. This causes unbounded heap growth leading to OOM crash of the node.

## Finding Description
`forward_rate_limiter` is declared as `RateLimiter<(PeerId, PeerId, u32)>` where `RateLimiter<T>` is `governor::RateLimiter<T, HashMapStateStore<T>, DefaultClock>` (mod.rs lines 31–35, 46). Each call to `check_key(&(content.from, content.to, item_id))` with a previously unseen triple inserts a new entry into the backing HashMap (connection_request.rs lines 132–135). The `from` and `to` fields are parsed directly from the attacker-supplied molecule message body with no constraint on their values beyond being valid PeerIds, which the attacker generates freely.

The only cleanup call, `self.forward_rate_limiter.retain_recent()`, appears in `disconnected()` (mod.rs lines 67–68). The `notify()` handler, which fires every 5 minutes via `CHECK_INTERVAL`, cleans only `pending_delivered` and `inflight_requests` (mod.rs lines 173–175) and never calls `retain_recent()` on either rate limiter.

The outer rate limiter (mod.rs lines 95–107) allows 30 messages/sec per `(session_id, item_id)`. Since `item_id` is a fixed discriminant for `ConnectionRequest`, this sets the HashMap insertion rate at 30 entries/sec per session — it does not cap total map size.

The route self-check (connection_request.rs lines 127–130) is bypassed by sending an empty `route` field: `content.route.contains(self_peer_id)` returns false for an empty vector, which is valid per the molecule schema. The only other precondition is a non-empty `listen_addrs` (line 115), trivially satisfied with one valid multiaddr.

An attacker who connects, never disconnects, and sends 30 `ConnectionRequest` messages/sec with fresh random `(from, to)` pairs inserts 30 new HashMap entries per second. `retain_recent()` is never triggered. The HashMap grows at 30 entries/sec × ~200 bytes/entry ≈ 6 KB/sec per session. With multiple concurrent attacker sessions (scaling to the node's max peer count), the rate scales linearly.

## Impact Explanation
**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.** A single persistent attacker session causes ~500 MB of heap growth over ~24 hours. With multiple concurrent sessions at the node's max peer count (e.g., 125 peers), the growth rate reaches ~750 KB/sec, exhausting typical node RAM in hours. The node process OOMs and crashes. This is a single-node crash, not a network-wide event, placing it at High rather than Critical.

## Likelihood Explanation
The attacker requires only a standard P2P connection to a node with `HolePunching` enabled — no privileges, no proof-of-work, no cryptographic keys beyond generating valid PeerIds (trivial). The attack requires only that the attacker not disconnect, which is trivially maintained. The outer rate limiter does not prevent the attack; it merely sets the growth rate. The route self-check is trivially bypassed with an empty route field. The attack is repeatable and requires no victim interaction or mistakes.

## Recommendation
Add `self.forward_rate_limiter.retain_recent()` and `self.rate_limiter.retain_recent()` calls inside the `notify()` handler (mod.rs lines 169–175), which already fires on the 5-minute `CHECK_INTERVAL`. This bounds the map to entries active within the last rate-limit window regardless of whether peers disconnect, mirroring the existing cleanup pattern in `disconnected()`.

## Proof of Concept
1. Connect to a target CKB node with `HolePunching` enabled using a standard P2P client.
2. In a loop at 30 messages/sec:
   - Generate a fresh random valid `PeerId` for `from` and a fresh random valid `PeerId` for `to`.
   - Construct a valid `ConnectionRequest` molecule message with `route = []` (empty, bypassing the self-peer-id check), `max_hops = 1`, and one valid `listen_addrs` entry containing a TCP multiaddr with the `from` peer id.
   - Send the message over the `HolePunching` protocol stream.
3. Never disconnect.
4. After `T` seconds, the `forward_rate_limiter` `HashMapStateStore` contains `30*T` entries. Assert map size grows linearly with `T` and is not bounded by the number of active sessions.
5. Monitor node RSS memory; observe linear growth. After sufficient time (hours to days depending on available RAM), the node process OOMs and crashes.
6. To accelerate: open multiple concurrent sessions (up to the node's max peer count), each inserting at 30 entries/sec.
