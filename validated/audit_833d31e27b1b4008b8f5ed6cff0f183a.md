### Title
Missing `publishSlot` Equality Check in TWAP Validation Causes Division-by-Zero Revert — (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

`validateTwapPriceInfo` in `Pyth.sol` uses a strict `>` comparison for `publishSlot`, allowing start and end TWAP updates with **equal slots** to pass validation. `calculateTwap` then unconditionally divides by `slotDiff`, which is zero in that case, causing a Solidity division-by-zero panic (revert) on every division. This is the direct analog of the `setTokenPricingBounds` issue where `floorPrice == ceilingPrice` passes the only sanity check but produces a broken state.

---

### Finding Description

In `validateTwapPriceInfo` the slot ordering check is:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This only rejects `start > end`. Equal slots (`start == end`) silently pass. [1](#0-0) 

`calculateTwap` then computes three unchecked divisions by `slotDiff`:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot;
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // panic if slotDiff == 0
uint128 twapConf  = confDiff / uint128(slotDiff);           // panic if slotDiff == 0
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panic if slotDiff == 0
``` [2](#0-1) 

The Solana reference implementation correctly uses a **strict** `>` requirement (rejecting equal slots):

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

Additionally, Solana's `calculate_twap` uses `checked_div` which returns a graceful error instead of panicking. [4](#0-3) 

The EVM implementation is therefore inconsistent with the Solana implementation and lacks the same protection.

---

### Impact Explanation

Any call to `parseTwapPriceFeedUpdates` where the two supplied update blobs carry the same `publishSlot` will revert with a division-by-zero panic inside `calculateTwap`. This completely blocks TWAP price retrieval for that time window. Downstream protocols relying on `parseTwapPriceFeedUpdates` for settlement, liquidation, or pricing will be unable to obtain a TWAP result for that window. [5](#0-4) 

---

### Likelihood Explanation

Pythnet publishes one TWAP accumulator message per slot. If a user (or Hermes) fetches two TWAP updates that happen to reference the same `publishSlot` — an edge case that can arise during slot skips, reorgs, or when the same slot is reused as both the start and end of a window — and submits them to `parseTwapPriceFeedUpdates`, the call reverts. No forged signatures are required; the attacker only needs valid Wormhole-signed data with equal slots, which Hermes can legitimately serve. The likelihood is low-to-medium given it is an edge case, but the missing check is a clear correctness gap relative to the Solana implementation.

---

### Recommendation

Change the slot comparison in `validateTwapPriceInfo` from strict `>` to `>=`, mirroring the Solana implementation:

```solidity
// Before (allows equal slots → division by zero downstream)
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After (rejects equal slots)
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

Optionally, add a zero-guard in `calculateTwap` as a defence-in-depth measure, consistent with the Solana `checked_div` pattern.

---

### Proof of Concept

1. Obtain two valid Wormhole-signed TWAP accumulator blobs for the same price feed where both blobs carry `publishSlot = S` (e.g., by querying Hermes for a window where start and end fall in the same Pythnet slot).
2. Call:
   ```solidity
   pyth.parseTwapPriceFeedUpdates{value: fee}(updateData, priceIds);
   ```
3. `validateTwapPriceInfo` passes because `S > S` is `false`.
4. `calculateTwap` computes `slotDiff = S - S = 0`.
5. `priceDiff / int128(uint128(0))` triggers a Solidity division-by-zero panic → transaction reverts. [6](#0-5)

        if (startPriceIds.length != endPriceIds.length) {

        // This allows us to assume startPriceIds[i] == endPriceIds[i] for efficiency
        for (uint i = 0; i < startPriceIds.length; i++) {

                revert PythErrors.InvalidTwapUpdateDataSet();

        // Initialize the output array
        twapPriceFeeds = new PythStructs.TwapPriceFeed[](priceIds.length);

            // Find the index of this price ID in the startPriceIds array
            // (which is the same as the endPriceIds array based on our validation above)
            for (uint j = 0; j < startPriceIds.length; j++) {
                if (startPriceIds[j] == requestedPriceId) {
                    startIdx = int(j);
                    break;
                }
            }

            // If we found the price ID
            if (startIdx >= 0) {
                uint idx = uint(startIdx);
                // Validate the pair of price infos
                validateTwapPriceInfo(
                    startTwapPriceInfos[idx],
                    endTwapPriceInfos[idx]
                );

                // Calculate TWAP from these data points
                twapPriceFeeds[i] = calculateTwap(
                    requestedPriceId,
                    startTwapPriceInfos[idx],
                    endTwapPriceInfos[idx]
                );
            }
        }

        // Ensure all requested price IDs were found
        for (uint k = 0; k < priceIds.length; k++) {
            if (twapPriceFeeds[k].id == 0) {
                revert PythErrors.PriceFeedNotFoundWithinRange();
            }
        }
