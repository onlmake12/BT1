### Title
Cross-Chain and Cross-Proxy Signature Replay in `PythLazer.verifyUpdate` Due to Missing Domain Binding — (`File: lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` computes the signed hash as `keccak256(payload)` with no chain ID, no contract address, and no domain separator bound into the hash. Any valid Lazer update accepted on one chain (or one proxy instance) is cryptographically identical on every other chain and every other proxy deployment that shares the same trusted signer, enabling cross-chain and cross-proxy signature replay by an unprivileged caller.

---

### Finding Description

In `PythLazer.verifyUpdate`, the signature verification hash is computed as:

```solidity
bytes32 hash = keccak256(payload);          // line 93
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` bytes contain only a format magic, timestamp, channel, and feed data — no chain ID and no contract address. [2](#0-1) 

Because the hash is purely over the payload content, the same `(r, s, v, payload)` tuple is valid on every EVM chain where the same trusted signer is registered, and on every `PythLazer` proxy instance on the same chain. The contract is a UUPS upgradeable proxy; multiple independent proxy instances can exist with the same implementation and the same trusted signer set. [3](#0-2) 

The analog to the MechMarketplace bug is exact:

| MechMarketplace | PythLazer |
|---|---|
| `domainSeparator` set in constructor → same for all proxies | `keccak256(payload)` with no domain → same hash on all chains/proxies |
| Signature replay across proxies on same chain | Signature replay across chains and across proxy instances |

---

### Impact Explanation

An unprivileged attacker who observes a valid Lazer price update on chain A can submit the identical `update` bytes to `PythLazer.verifyUpdate` on chain B (or a second proxy on chain A). The function will return `(payload, signer)` as if the update were freshly signed for that chain. Any consuming contract that relies on `verifyUpdate` returning a trusted payload will accept the replayed data. If the attacker times the replay to coincide with a price discrepancy between chains (e.g., due to update lag), they can feed a stale or cross-chain price to a DeFi protocol consuming Lazer on the target chain, enabling price manipulation.

---

### Likelihood Explanation

Pyth Lazer is deployed on multiple EVM chains (Ethereum, Arbitrum, BNB Chain, etc.) with the same trusted signer keys. The attacker entry path requires only: (1) subscribing to the public Lazer WebSocket stream to obtain a valid signed update, and (2) submitting it to `verifyUpdate` on a different chain. No privileged access, leaked key, or governance majority is required. The fee is minimal (`verification_fee = 1 wei`). [4](#0-3) 

---

### Recommendation

Bind the signed hash to the chain and contract address using EIP-712 domain separation, or include `block.chainid` and `address(this)` in the hashed message. For example:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    payload
));
```

This ensures a signature produced for one chain/proxy is cryptographically invalid on any other.

---

### Proof of Concept

1. Deploy two `PythLazer` proxy instances (`proxy1`, `proxy2`) on the same chain, both initialized with the same trusted signer.
2. Call `proxy1.verifyUpdate{value: fee}(update)` with a valid signed update — succeeds.
3. Call `proxy2.verifyUpdate{value: fee}(update)` with the **identical** `update` bytes — also succeeds, returning the same `(payload, signer)`.
4. Repeat step 3 on a second EVM chain where the same trusted signer is registered — also succeeds.

The root cause is that `keccak256(payload)` at line 93 contains no chain or contract binding, so the recovered `signer` is identical regardless of which chain or proxy processes the update. [5](#0-4)
