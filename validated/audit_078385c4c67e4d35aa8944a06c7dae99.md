Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Post-Eviction Decrements in `add_entry`, Permanently Inflating Pool Size Accounting — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures `self.total_tx_size + new_size` into local variables before `check_and_record_ancestors` runs. That inner call can evict transactions via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` in place. However, the stale pre-eviction locals are then unconditionally written back to `self.total_tx_size` and `self.total_tx_cycles`, erasing those decrements. The pool permanently believes it holds more data than it actually does, causing spurious evictions and premature rejection of valid transactions.

## Finding Description
The exact sequence in `add_entry` (lines 210–219):

```rust
// Line 210-211: captures S + new_size into locals BEFORE any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Line 213: may evict; each eviction calls update_stat_for_remove_tx
//           → self.total_tx_size -= evicted_size  (in-place, correct)
evicts = self.check_and_record_ancestors(&mut entry)?;

// Lines 218-219: overwrites with stale pre-eviction value
self.total_tx_size  = total_tx_size;   // S + new_size  ← BUG
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) is a pure read: it adds the new entry's size to `self.total_tx_size` and returns the result as a local — it does not mutate state. [1](#0-0) 

`check_and_record_ancestors` (lines 615–625) calls `remove_entry_and_descendants` when ancestor count exceeds `max_ancestors_count`, which chains into `remove_entry` (line 247), which calls `update_stat_for_remove_tx` — directly decrementing `self.total_tx_size` in place. [2](#0-1) [3](#0-2) 

After eviction, `self.total_tx_size = S − evicted_size`. The write-back at line 218 then sets it to `S + new_size`, inflating it by exactly `evicted_size`. The correct value should be `S + new_size − evicted_size`. [4](#0-3) 

## Impact Explanation
`total_tx_size` is the sole counter driving `limit_size` (pool.rs lines 298–328). An inflated counter causes two concrete effects:

1. **Spurious further evictions**: `limit_size` is called after `add_entry` returns. It sees `total_tx_size > max_tx_pool_size` and evicts additional valid, fee-paying transactions that should have remained in the pool.
2. **Premature rejection of future submissions**: subsequent calls to `updated_stat_for_add_tx` start from an already-inflated baseline, causing valid transactions to be rejected with `Reject::Full` even when the pool has real capacity. [5](#0-4) 

An attacker can repeatedly trigger this to permanently degrade mempool throughput and selectively starve the pool of legitimate transactions with minimal cost. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation
The eviction path inside `check_and_record_ancestors` is triggered when a new transaction's cell dep is already consumed by an in-pool transaction (`cell_ref_parents` is non-empty) and the total ancestor count exceeds `max_ancestors_count` (default 25). Both conditions are reachable by any unprivileged caller via the `send_transaction` RPC or relay peer. An attacker who observes mempool state via `get_raw_tx_pool` can deliberately engineer the required chain structure. The inflation is cumulative and permanent across multiple such submissions, making the attack repeatable at low cost.

## Recommendation
Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so eviction-driven decrements are already reflected in `self.total_tx_size` before the new entry's contribution is added:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Now read post-eviction totals and add only the new entry's contribution
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, drop the local variables entirely and apply the increment in-place after all mutations complete (e.g., `self.total_tx_size = self.total_tx_size.checked_add(entry.size)?`).

## Proof of Concept
1. Fill the pool with a chain of 24 transactions `T1 → T2 → … → T24` where each spends the previous output.
2. Submit a separate transaction `R` that takes `T1`'s output as a **cell dep** (`R` becomes a `cell_ref_parent` of the chain).
3. Submit a new transaction `T_new` that spends `T1`'s output. `T_new` now has 24 ancestors + `R` as a cell-ref parent, exceeding `max_ancestors_count = 25`.
4. `check_and_record_ancestors` evicts `R` (size `S_R`) via `remove_entry_and_descendants`, correctly decrementing `self.total_tx_size` by `S_R`.
5. The write-back at line 218 overwrites `self.total_tx_size` with the pre-eviction value, inflating it by `S_R`.
6. Verify via `get_tip_tx_pool_info` that `total_tx_size` exceeds the sum of actual entry sizes, and observe that `limit_size` subsequently evicts a valid transaction that fits within `max_tx_pool_size`.
7. Repeat steps 1–6 to show the inflation is cumulative across multiple eviction events.

        })?;

    }
