### Title
Minimum Fee Rate Check Uses `tx_size` Instead of Actual Transaction Weight, Allowing Cycle-Heavy Transactions to Bypass the Fee Floor — (File: `tx-pool/src/util.rs`)

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the serialized byte size of the transaction (`tx_size`), while the canonical transaction weight used everywhere else in the system is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. A transaction with a small serialized size but a script that consumes many cycles will pass the fee floor check with a fee far below what the actual weight-based minimum fee rate requires, allowing it to be admitted to the tx pool.

### Finding Description

CKB's fee model defines transaction weight as:

```rust
// util/types/src/core/tx_pool.rs
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [1](#0-0) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. This weight is used for fee rate calculation in `TxEntry::fee_rate()`, block assembly scoring, and eviction:

```rust
// tx-pool/src/component/entry.rs
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [2](#0-1) 

However, the admission gate `check_tx_fee` — the only fee rate check before a transaction enters the pool — uses only `tx_size` as the weight denominator:

```rust
// tx-pool/src/util.rs
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [3](#0-2) 

This function is called in `pre_check`, which runs before `verify_rtx` (the step that determines actual cycles): [4](#0-3) 

Because cycles are not yet known at the `pre_check` stage, the fee check cannot use the real weight. The code acknowledges this with the comment "Theoretically we cannot use size as weight directly." The result is a structural discrepancy: the admission check uses `tx_size`, but the rest of the system uses `weight = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`.

### Impact Explanation

An attacker crafts a transaction with:
- A small serialized size (e.g., 200 bytes)
- A lock/type script that consumes a large number of cycles (e.g., near `max_block_cycles`)

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` and `max_block_cycles ≈ 3,500,000,000`:

- Cycle-equivalent weight: `3,500,000,000 × 0.000_170_571_4 ≈ 597,000` bytes
- Actual weight: `max(200, 597,000) = 597,000`
- Fee check requires: `min_fee_rate × 200 / 1000` shannons
- Correct minimum fee: `min_fee_rate × 597,000 / 1000` shannons
- **Discrepancy factor: ~2985×** [5](#0-4) 

The transaction is admitted to the pool with a fee ~3000× below the intended minimum. Once inside, its `fee_rate()` (computed with real weight) is far below the minimum, so miners will not include it, but it still occupies pool memory. An attacker can flood the pool with such transactions at a tiny fraction of the intended cost, displacing legitimate transactions and degrading pool performance.

### Likelihood Explanation

The attack is reachable by any unprivileged RPC caller (`send_transaction`) or P2P transaction relay sender. No special privilege is required. The attacker only needs to submit a transaction whose script performs many iterations of cheap computation (e.g., a tight loop in CKB-VM). The script must actually execute and pass `verify_rtx`, but this is straightforward to construct. The discrepancy is largest when `min_fee_rate` is set to its default non-zero value and the transaction's cycle consumption is near the per-transaction cycle cap. [6](#0-5) 

### Recommendation

After `verify_rtx` returns the actual `verified.cycles`, re-check the fee rate using the real weight before calling `submit_entry`:

```rust
// In _process_tx, after verify_rtx succeeds:
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
``` [7](#0-6) 

This mirrors how `TxEntry::fee_rate()` and `AncestorsScoreSortKey` already compute weight, making the admission gate consistent with the rest of the fee accounting system. [8](#0-7) 

### Proof of Concept

1. Construct a CKB transaction with a lock script that runs a tight RISC-V loop consuming ~3,500,000,000 cycles. The serialized transaction size is ~200 bytes.
2. Set the transaction fee to `min_fee_rate × 200 / 1000` shannons (e.g., with `min_fee_rate = 1000` shannons/KW, fee = 0.2 shannons, rounded up to 1 shannon).
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200` shannons and the fee passes.
5. `verify_rtx` executes the script, confirming ~3,500,000,000 cycles consumed.
6. `TxEntry` is created with `size=200, cycles=3_500_000_000, fee=200 shannons`.
7. `entry.fee_rate()` = `FeeRate::calculate(200, max(200, 597_000))` = `200 × 1000 / 597_000 ≈ 0.33` shannons/KW — far below the 1000 shannons/KW minimum.
8. The transaction occupies pool memory. Repeat to exhaust pool capacity at ~3000× lower cost than intended. [9](#0-8) [10](#0-9)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
