### Title
Unbounded `pending_delivered` HashMap Growth via Unique Spoofed `from_peer_id` in `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

A single connected peer can flood the local node's `pending_delivered` HashMap with up to ~9,000 entries per 5-minute window by sending `ConnectionRequest` messages that each carry a distinct, attacker-controlled `from_peer_id` while targeting the local node as `to_peer_id`. The map is only pruned every 5 minutes, and no guard bounds it by unique `from_peer_id` count.

---

### Finding Description

**Entrypoint:** Any peer connected over P2P that has the `HolePunching` protocol open.

**Rate limiter — does NOT prevent this:**

The primary rate limiter is keyed by `(session_id, msg.item_id())`: [1](#0-0) 

This allows 30 messages/sec from one session regardless of `from_peer_id` content.

The forward rate limiter is keyed by `(from_peer_id, to_peer_id, msg_item_id)`: [2](#0-1) 

Because the attacker uses a **distinct** `from_peer_id` per message, each message is a new key — the forward rate limiter does not throttle this pattern.

**Deduplication check — bypassed by unique keys:** [3](#0-2) 

This only skips re-insertion for the **same** `from_peer_id`. A unique `from_peer_id` per message bypasses it entirely.

**Unconditional insert after successful send:** [4](#0-3) 

Every message with a unique `from_peer_id` that passes the `send_message_to` call inserts a new entry.

**Pruning only every 5 minutes:** [5](#0-4) 

`retain()` runs only on `CHECK_INTERVAL = 5 minutes`. Between pruning cycles, the map accumulates all injected entries.

**Secondary growth — `forward_rate_limiter`:**

The `forward_rate_limiter` also accumulates one entry per unique `(from, to, item_id)` tuple and is only cleaned up on disconnect via `retain_recent()`: [6](#0-5) 

---

### Impact Explanation

- **`pending_delivered` max entries:** 30 msg/sec × 300 sec = **9,000 entries**
- **Per entry size:** `PeerId` key (~39 bytes) + `Vec<Multiaddr>` (up to 24 addrs × ~9 bytes) + `u64` timestamp ≈ ~263 bytes
- **Total heap growth:** ~9,000 × 263 bytes ≈ **~2.4 MB** per attacker session
- **`forward_rate_limiter` additional growth:** ~9,000 × ~82 bytes ≈ **~738 KB**
- **Combined:** ~3.1 MB per connected attacker session

This is bounded, low-severity heap growth. A single attacker session cannot cause OOM on a modern node, but multiple concurrent attacker sessions multiply the effect linearly.

---

### Likelihood Explanation

The path requires only a standard P2P connection with the `HolePunching` protocol. No authentication, no PoW, no privileged role. The attacker only needs to:
1. Connect to the victim node
2. Send `ConnectionRequest` messages with unique `from_peer_id` bytes, `to_peer_id` = victim's peer ID, and at least one valid TCP `listen_addr`

This is trivially scriptable. The `listen_addrs` validity check (must be non-empty TCP/IP addresses) is easily satisfied with any real or fake IP:port. [7](#0-6) 

---

### Recommendation

1. **Cap `pending_delivered` size:** Add a `MAX_PENDING_DELIVERED` constant (e.g., 256) and reject insertion when the map is at capacity.
2. **Rate-limit insertions by session:** Track how many `pending_delivered` entries originated from each session and evict on disconnect.
3. **Validate `from_peer_id` against connected peers:** Only accept `ConnectionRequest` where `from_peer_id` corresponds to a known connected peer, rejecting arbitrary spoofed IDs.

---

### Proof of Concept

```
1. Connect to victim node with HolePunching protocol.
2. In a loop for 300 seconds at 30 msg/sec:
   a. Generate a fresh random PeerId as `from_peer_id`.
   b. Craft a ConnectionRequest with:
      - from = fresh_random_peer_id
      - to   = victim's local_peer_id
      - listen_addrs = [/ip4/1.2.3.4/tcp/1234]
      - max_hops = 6
   c. Send the message.
3. After 300 seconds, assert pending_delivered.len() ≈ 9000.
4. Observe ~3 MB heap growth attributable to pending_delivered and forward_rate_limiter.
```
