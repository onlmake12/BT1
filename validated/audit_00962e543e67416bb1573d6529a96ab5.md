Audit Report

## Title
`DaoScriptSizeVerifier` Skipped for First Block of RFC0044 Activation Epoch Due to Off-by-One Epoch Parameter — (`File: verification/contextual/src/contextual_block_verifier.rs`)

## Summary
In `BlockTxsVerifier::verify`, the hardfork activation guard for `DaoScriptSizeVerifier` is evaluated using `self.parent.epoch().number()` (the parent block's epoch) instead of `self.header.epoch().number()` (the current block's epoch). For the first block of the RFC0044 activation epoch `N`, the parent is in epoch `N-1`, so `rfc0044_active(N-1)` returns `false` and `DaoScriptSizeVerifier` is silently skipped for every non-cellbase transaction in that block. A miner can exploit this to include a transaction with an oversized DAO script, causing a consensus split between patched and unpatched nodes.

## Finding Description
`BlockTxsVerifier` stores both `header: HeaderView` (current block) and `parent: &'b HeaderView` (parent block) as distinct fields. [1](#0-0) 

Inside `verify`, the guard at line 445 reads:

```rust
if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {
``` [2](#0-1) 

When the current block is the first block of activation epoch `N`, its parent is the last block of epoch `N-1`. `rfc0044_active(N-1)` returns `false`, so `DaoScriptSizeVerifier` is never invoked for any transaction in that block. For all subsequent blocks of epoch `N`, the parent is also in epoch `N`, so the guard evaluates correctly. The bug is confined to exactly one block per activation epoch.

The tx-pool applies `DaoScriptSizeVerifier` unconditionally (no `rfc0044_active` guard), so an oversized DAO script transaction is rejected by normal `send_transaction` submission: [3](#0-2) 

The attacker must bypass the tx-pool entirely by assembling the block manually and submitting it via the `submit_block` RPC, which does not re-run tx-pool validation.

## Impact Explanation
A miner who submits a block containing an oversized DAO script transaction at the first block of the RFC0044 activation epoch will have that block accepted by all nodes running this code. Any node running a corrected build evaluates `rfc0044_active(self.header.epoch().number())` = `rfc0044_active(N)` = `true`, runs `DaoScriptSizeVerifier`, and rejects the block. This produces a **consensus split** — a Critical impact matching "Vulnerabilities which could easily cause consensus deviation."

## Likelihood Explanation
The attack window is exactly one block per RFC0044 activation epoch, but it is fully deterministic and predictable from the chain spec. A miner who knows the activation epoch can prepare the crafted transaction in advance. The `submit_block` RPC is a standard, documented interface available to any miner operating a node. No additional privileges beyond operating a mining node are required. Likelihood is low in practice (requires a deliberately malicious miner), but the preconditions are minimal and the exploit is mechanically straightforward.

## Recommendation
Replace `self.parent.epoch().number()` with `self.header.epoch().number()` in the `rfc0044_active` guard:

```rust
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
``` [4](#0-3) 

Additionally, add a unit test that verifies `DaoScriptSizeVerifier` is enforced for the **first** block of the RFC0044 activation epoch (parent in epoch `N-1`, current block in epoch `N`), not only for subsequent blocks.

## Proof of Concept
1. Determine the RFC0044 activation epoch `N` from the chain spec.
2. Craft a transaction `T` whose DAO-type output script exceeds the RFC0044 size limit. Submitting `T` via `send_transaction` RPC is rejected by the tx-pool's unconditional `DaoScriptSizeVerifier`.
3. Manually assemble a block at the first height `H` of epoch `N` (parent = last block of epoch `N-1`).
4. Include `T` directly in the block's transaction list, bypassing the tx-pool.
5. Submit the block via `submit_block` RPC.
6. The unpatched node accepts the block: `rfc0044_active(self.parent.epoch().number())` = `rfc0044_active(N-1)` = `false`, so `DaoScriptSizeVerifier` is never called.
7. A patched node evaluates `rfc0044_active(self.header.epoch().number())` = `rfc0044_active(N)` = `true`, runs `DaoScriptSizeVerifier`, and rejects the block — producing a consensus split.
