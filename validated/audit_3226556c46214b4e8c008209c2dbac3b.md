### Title
Size-Only Fee Gate Allows High-Cycle Transactions to Force Expensive Script Execution With Insufficient Compensation — (`tx-pool/src/util.rs`)

---

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` enforces the minimum fee rate using **only serialized transaction size**, ignoring cycle consumption. Because the correct transaction weight is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`, a transaction with a tiny serialized body but a near-maximum-cycle script passes the fee gate with a trivially small fee, then forces every receiving node to execute expensive script verification — with no mechanism to opt out and no adequate compensation.

---

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee as:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The code itself acknowledges the theoretical incorrectness. The proper weight formula, used everywhere else in the codebase (fee estimation, pool scoring, eviction), is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

The `_process_tx` pipeline in `tx-pool/src/process.rs` runs `pre_check` (which calls `check_tx_fee`) **before** `verify_rtx` (which executes scripts):

```rust
let (ret, snapshot) = self.pre_check(&tx).await;
// ...
let verified_ret = verify_rtx(..., max_cycles, command_rx).await;
``` [4](#0-3) 

For local transactions submitted via RPC, `declared_cycles` is `None`, so `max_cycles` defaults to `consensus.max_block_cycles()` — the full block cycle limit, not the pool's `max_tx_verify_cycles`:

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
``` [5](#0-4) 

**Concrete numbers (mainnet defaults):**

| Parameter | Value |
|---|---|
| `min_fee_rate` | 1,000 shannons/KB |
| `max_tx_verify_cycles` | 70,000,000 |
| `DEFAULT_BYTES_PER_CYCLES` | 0.000_170_571_4 |

A transaction with serialized size = 200 bytes and 70,000,000 cycles:

- **Size-based min fee** (what `check_tx_fee` enforces): `1000 × 200 / 1000 = 200 shannons`
- **Weight-based min fee** (what is fair): `max(200, 70_000_000 × 0.000_170_571_4) = 11,940` → `1000 × 11,940 / 1000 = 11,940 shannons`

The attacker pays **200 shannons** but forces **~60× more computational work** than the fee justifies.

The `TxEntry::fee_rate()` method correctly uses `get_transaction_weight` after admission, meaning the pool internally tracks the true fee rate — but the admission gate never enforces it:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [6](#0-5) 

---

### Impact Explanation

1. **CPU exhaustion at tx-pool admission**: Every full node that receives such a transaction via P2P relay or RPC must execute scripts up to `max_block_cycles` before the transaction can be rejected or accepted. An attacker can flood the network with small-size, high-cycle transactions, each paying only the size-based minimum fee, forcing all nodes to burn CPU on script execution.

2. **Miner under-compensation**: Miners who include such transactions in blocks receive far less fee than the computational work justifies. The block template assembler selects transactions by true weight-based fee rate (via `fee_rate()` on `TxEntry`), but transactions that passed the size-only gate at admission may have a true fee rate far below `min_fee_rate`.

3. **All validating nodes forced to execute expensive scripts**: When a block containing such transactions is committed, every full node must re-execute the scripts during block validation with no compensation at all.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender via the `send_transaction` RPC or P2P relay. No special privileges are required. The attacker only needs to:
1. Deploy a script that loops near the cycle limit (trivially done with a custom lock script)
2. Craft a transaction with a small serialized body referencing that script
3. Pay only the size-based minimum fee (e.g., 200 shannons)
4. Submit via `send_transaction` RPC or relay to the P2P network

The `max_tx_verify_cycles` config (default 70M) provides a per-transaction cap, but the fee check does not account for it. The gap between size-based and weight-based minimum fees grows linearly with cycle consumption.

---

### Recommendation

Replace the size-only fee check in `check_tx_fee` with a weight-based check. For the pre-execution "cheap check", use the declared cycles (for remote transactions) or a conservative upper bound (for local transactions) to compute the weight:

```rust
// Use declared_cycles if available; otherwise use max_tx_verify_cycles as upper bound
let weight = get_transaction_weight(tx_size, declared_cycles.unwrap_or(max_tx_verify_cycles));
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

After script execution, re-validate the fee against the actual verified cycles. This mirrors the Holograph mitigation: allow the node to "opt out" (reject) a transaction whose true computational cost exceeds what the fee covers, before committing to full execution. [7](#0-6) 

---

### Proof of Concept

1. Deploy a lock script that executes a tight loop consuming ~70,000,000 cycles (near `max_tx_verify_cycles`).
2. Create a cell locked by this script. The transaction body is small (e.g., 200 bytes serialized).
3. Craft a spending transaction with fee = `min_fee_rate × tx_size / 1000 = 1000 × 200 / 1000 = 200 shannons`.
4. Submit via `send_transaction` RPC.
5. `check_tx_fee` passes: `fee (200) >= min_fee (200)`. ✓
6. `verify_rtx` executes the script for ~70,000,000 cycles, consuming significant CPU.
7. The true weight-based fee rate is `200 × 1000 / 11940 ≈ 16.7 shannons/KW` — far below the 1,000 shannons/KW `min_fee_rate`.
8. Repeat with many such transactions to exhaust node CPU resources at minimal cost.

The `check_tx_fee` size-only gate at line 45 of `tx-pool/src/util.rs` is the necessary vulnerable step; without it, the transaction would be rejected before script execution. [1](#0-0) [8](#0-7) [9](#0-8)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
