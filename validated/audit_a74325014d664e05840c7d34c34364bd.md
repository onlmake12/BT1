Audit Report

## Title
IPv4-Mapped IPv6 Address Bypasses Peer Ban — (`network/src/peer_store/ban_list.rs`)

## Summary
`ip_to_network` in `network/src/peer_store/types.rs` stores and checks ban entries using the raw IP family without normalizing IPv4-mapped IPv6 addresses (`::ffff:x.x.x.x`). A peer banned via its IPv4 address is stored as `IpNetwork::V4(x.x.x.x/32)`; if it reconnects via the IPv4-mapped IPv6 form, the check produces `IpNetwork::V6(::ffff:x.x.x.x/128)`, which matches nothing in the ban list. The ban is fully bypassed, and the bypass is bidirectional.

## Finding Description

**Root cause — `ip_to_network` (`network/src/peer_store/types.rs` L152-157):**

```rust
pub fn ip_to_network(ip: IpAddr) -> IpNetwork {
    match ip {
        IpAddr::V4(ipv4) => IpNetwork::V4(ipv4.into()),
        IpAddr::V6(ipv6) => IpNetwork::V6(ipv6.into()),   // no to_ipv4_mapped() call
    }
}
``` [1](#0-0) 

**Ban storage path (`peer_store_impl.rs` L286-292):**
`ban_addr` calls `multiaddr_to_socketaddr`, then `ip_to_network(addr.ip())`. If the connection arrived as IPv4, the stored key is `IpNetwork::V4(1.2.3.4/32)`. [2](#0-1) 

**Ban check path (`ban_list.rs` L48-58):**
When the same peer reconnects via `::ffff:1.2.3.4`:
1. `ip_to_network(...)` → `IpNetwork::V6(::ffff:1.2.3.4/128)`
2. `self.inner.get(&IpNetwork::V6(...))` → **None** (stored key is V4)
3. Linear scan: `IpNetwork::V4(1.2.3.4/32).contains(IpAddr::V6(::ffff:1.2.3.4))` → **false** (`ipnetwork` never matches cross-family)
4. Returns `false` → **ban bypassed** [3](#0-2) 

**Gatekeeper (`peer_registry.rs` L109-111):**
`accept_peer` calls `peer_store.is_addr_banned(&remote_addr)` before admitting any inbound connection. Because the check returns `false`, the banned peer is admitted. [4](#0-3) 

**Production evidence:** The RPC documentation in `util/jsonrpc-types/src/net.rs` L93-95 shows actual production peers advertising `/ip6/::ffff:18.185.102.19/tcp/8115/p2p/...`, confirming the tentacle library presents dual-stack connections as IPv4-mapped IPv6 multiaddresses. [5](#0-4) 

**Inconsistency within the same codebase:** `network/src/network_group.rs` L31-35 already calls `ipv6.to_ipv4()` to normalize IPv4-mapped IPv6 to IPv4 for peer grouping. The ban list has no equivalent normalization. [6](#0-5) 

**Ban trigger confirmed:** `test/src/specs/p2p/malformed_message.rs` L33-45 demonstrates that sending two malformed sync messages causes the node to disconnect and ban the peer, confirming the ban mechanism is the primary defense against misbehaving peers. [7](#0-6) 

## Impact Explanation

The ban mechanism is the sole automated defense against misbehaving peers. With this bypass, a banned peer can immediately reconnect by switching between IPv4 and IPv4-mapped IPv6 representations. This enables: (1) indefinite connection slot consumption — the attacker reconnects faster than the node can re-ban, exhausting `max_inbound` slots and preventing legitimate peers from connecting; (2) repeated malformed message floods with no effective rate limiting. This maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

On Linux (the dominant CKB node platform), dual-stack TCP sockets are the default. The tentacle library presents such connections as `/ip6/::ffff:x.x.x.x/...` multiaddresses, as confirmed by production RPC output. No special tooling is required — any standard TCP client connecting to an IPv6 socket using an IPv4-mapped address triggers the bypass automatically. The attacker needs only a single IP address and the ability to open TCP connections. The bypass is repeatable without limit.

## Recommendation

Normalize IPv4-mapped IPv6 addresses to their IPv4 equivalents in `ip_to_network` (`network/src/peer_store/types.rs`):

```rust
pub fn ip_to_network(ip: IpAddr) -> IpNetwork {
    let ip = match ip {
        IpAddr::V6(v6) => v6.to_ipv4_mapped()
            .map(IpAddr::V4)
            .unwrap_or(IpAddr::V6(v6)),
        v4 => v4,
    };
    match ip {
        IpAddr::V4(ipv4) => IpNetwork::V4(ipv4.into()),
        IpAddr::V6(ipv6) => IpNetwork::V6(ipv6.into()),
    }
}
```

This mirrors the normalization already present in `network_group.rs` and ensures that `ban_addr` and `is_ip_banned_until` operate on a canonical IP representation regardless of how the peer connected.

## Proof of Concept

```rust
use std::net::{IpAddr, Ipv4Addr, Ipv6Addr};
use ckb_network::peer_store::{PeerStore, types::BannedAddr};
use ipnetwork::IpNetwork;

let mut peer_store = PeerStore::default();
let now_ms = ckb_systemtime::unix_time_as_millis();

// Ban the IPv4 address 1.2.3.4
let v4_network: IpNetwork = "1.2.3.4/32".parse().unwrap();
peer_store.mut_ban_list().ban(BannedAddr {
    address: v4_network,
    ban_until: now_ms + 100_000,
    created_at: now_ms,
    ban_reason: "test".into(),
});

// Sanity check: IPv4 is banned
let v4_ip = IpAddr::V4(Ipv4Addr::new(1, 2, 3, 4));
assert!(peer_store.ban_list().is_ip_banned(&v4_ip));

// Bypass: reconnect via IPv4-mapped IPv6 ::ffff:1.2.3.4
let mapped_v6 = IpAddr::V6(Ipv6Addr::new(0, 0, 0, 0, 0, 0xffff, 0x0102, 0x0304));
// Returns false — ban bypassed
assert!(!peer_store.ban_list().is_ip_banned(&mapped_v6));
```

The existing test infrastructure in `network/src/tests/peer_store.rs` already exercises `ban_addr` and `is_addr_banned`; adding a test case with an IPv4-mapped IPv6 address against a V4 ban entry will reproduce the failure deterministically. [8](#0-7)
