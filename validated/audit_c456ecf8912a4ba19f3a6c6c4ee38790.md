Audit Report

## Title
`get_cellbase` Missing Freezer Fallback Causes Unconditional Panic via `get_block_economic_state` RPC — (`store/src/store.rs`, `util/reward-calculator/src/lib.rs`)

## Summary

`get_cellbase` in `store/src/store.rs` reads exclusively from `COLUMN_BLOCK_BODY` with no freezer fallback, unlike `get_block` and `get_transaction_with_info` which both check `freezer.number()` and retrieve from the freezer. After the freezer background thread wipes a block's body via `delete_block_body`, `get_cellbase` returns `None` for that block. `block_reward_internal` in `util/reward-calculator/src/lib.rs` calls `.expect("target cellbase exist")` on that `None`, causing an unconditional process panic reachable via the `get_block_economic_state` RPC method.

## Finding Description

**`get_cellbase` has no freezer fallback.**

`get_cellbase` at `store/src/store.rs:469–477` reads only from `COLUMN_BLOCK_BODY`:

```rust
fn get_cellbase(&self, hash: &packed::Byte32) -> Option<TransactionView> {
    let key = packed::TransactionKey::new_builder()
        .block_hash(hash.to_owned())
        .build();
    self.get(COLUMN_BLOCK_BODY, key.as_slice()).map(|slice| { ... })
}
```

By contrast, `get_block` at `store/src/store.rs:44–54` and `get_transaction_with_info` at `store/src/store.rs:321–336` both guard with `if let Some(freezer) = self.freezer() && header.number() > 0 && header.number() < freezer.number()` and call `freezer.retrieve(...)` before falling back to the kv store.

**Freezer wipes the cellbase from the kv store.**

`wipe_out_frozen_data` in `shared/src/shared.rs:222–226` calls `batch.delete_block_body(*number, hash, *txs)` for every frozen block, removing all `COLUMN_BLOCK_BODY` entries including the cellbase at index 0. `compact_block_body` at `shared/src/shared.rs:282–299` then calls `compact_range` on `COLUMN_BLOCK_BODY`, consolidating the tombstones. After this, `get_cellbase` for any frozen block hash returns `None`.

**Hard panic in `block_reward_internal`.**

`block_reward_internal` at `util/reward-calculator/src/lib.rs:90–94` calls:

```rust
self.store
    .get_cellbase(&target.hash())
    .expect("target cellbase exist")
```

There is no `?`-propagation or graceful error path. The `expect` panics unconditionally when `get_cellbase` returns `None`.

**RPC entrypoint.**

`block_reward_for_target` at `util/reward-calculator/src/lib.rs:63–78` calls `block_reward_internal`, and is itself called by the `get_block_economic_state` RPC handler in `rpc/src/module/chain.rs`. No authentication is required.

## Impact Explanation

By default the RPC is bound to `127.0.0.1`, making this a **Note (0–500 points) — Any local RPC API crash**. If the operator exposes the RPC port externally (a supported configuration), any remote caller can trigger the panic, escalating to **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The freezer activates automatically after epoch 2 on mainnet (`THRESHOLD_EPOCH = 2` in `shared/src/shared.rs:37`). On any fully-synced mainnet node, the vast majority of blocks are frozen and their bodies wiped. The trigger requires no special privileges — only the ability to call the standard `get_block_economic_state` RPC with any frozen block number. The condition is permanently and trivially satisfied on any synced node with the freezer enabled.

## Recommendation

Fix `get_cellbase` in `store/src/store.rs` to include the same freezer fallback pattern used by `get_block` and `get_transaction_with_info`: check `freezer.number()`, call `freezer.retrieve(header.number())`, and extract the transaction at index 0 from the frozen block data. Additionally, replace the `.expect("target cellbase exist")` in `block_reward_internal` at `util/reward-calculator/src/lib.rs:94` with a proper `?`-propagated `DaoError` return so that even if the lookup fails, the RPC returns a graceful JSON error instead of panicking.

## Proof of Concept

1. Start a CKB mainnet node with the freezer enabled (default configuration).
2. Sync past epoch 2 and wait for the freezer background thread (`spawn_freeze` in `shared/src/shared.rs:120`) to freeze and wipe blocks.
3. Identify any block number below the current freezer threshold (e.g., block 1).
4. Call `get_block_economic_state` via RPC:
   ```
   curl -X POST -H "Content-Type: application/json" \
     -d '{"jsonrpc":"2.0","method":"get_block_economic_state","params":["0x1"],"id":1}' \
     http://127.0.0.1:8114
   ```
5. Observe the node panic with message `"target cellbase exist"` rather than returning a JSON error response.
