### Title
Deviation Check in `_validateShouldUpdatePrices` Ignores Exponent, Causing Incorrect Update Triggering - (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler._validateShouldUpdatePrices()` computes price deviation by comparing raw `int64` mantissa values without normalizing for the `expo` field. Because Pyth prices are fixed-point numbers (`actual_price = mantissa × 10^expo`), any change in exponent between the stored previous price and the incoming current price makes the deviation calculation completely wrong, causing either spurious updates (fund drain) or missed updates (stale prices delivered to subscribers).

---

### Finding Description

In `_validateShouldUpdatePrices`, the deviation-based update trigger is implemented as:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;
int64 previousPrice = previousFeed.price.price;

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
``` [1](#0-0) 

Only the raw `int64` mantissa is used. The `expo` field from `PythStructs.Price` is never read here. [2](#0-1) 

Pyth prices are fixed-point: `actual_price = price × 10^expo`. The exponent is not a constant — Pyth feeds can and do change their exponent between updates. The TWAP validation code in `Pyth.sol` explicitly guards against exponent mismatches between start and end TWAP snapshots, confirming that exponent changes are a known, real occurrence:

```solidity
if (twapPriceInfoStart.expo != twapPriceInfoEnd.expo) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [3](#0-2) 

The Scheduler deviation check has no analogous guard and no normalization.

---

### Impact Explanation

Two distinct failure modes arise when `currentPrice.expo != previousPrice.expo`:

**Scenario A — False negative (missed update, stale prices):**
- Stored previous: `price = 100_000_000`, `expo = -8` → actual $1.00
- Incoming current: `price = 100_000_000`, `expo = -6` → actual $100.00 (100× increase)
- Raw deviation: `|100_000_000 − 100_000_000| / 100_000_000 = 0 bps`
- Result: update is **not triggered** even though the real price moved 100×. Subscribers receive the old stored price ($1.00) instead of $100.00. Any downstream protocol reading from the Scheduler gets a catastrophically wrong price.

**Scenario B — False positive (spurious update, fund drain):**
- Stored previous: `price = 100_000_000`, `expo = -8` → actual $1.00
- Incoming current: `price = 1_000_000`, `expo = -6` → actual $1.00 (unchanged)
- Raw deviation: `|1_000_000 − 100_000_000| / 100_000_000 = 9900 bps`
- Result: update is **spuriously triggered** even though the real price is identical. Subscription ETH balance is consumed for a needless update.

Scenario A is the more severe impact: subscribers that rely on deviation-based updates (e.g., lending protocols, perps) would act on a stale price, enabling liquidation manipulation or mispriced trades.

---

### Likelihood Explanation

The entry path is fully unprivileged: `updatePriceFeeds()` is callable by any keeper/relayer with valid Pyth update data. [4](#0-3) 

Pyth feeds do change their exponent in practice (the TWAP code's explicit check proves this is a known protocol-level event). When an exponent change occurs, the very next `updatePriceFeeds` call will hit the broken deviation path. No special attacker capability is required — a normal keeper submitting a legitimate price update is sufficient to trigger either failure mode.

---

### Recommendation

Before computing deviation, normalize both prices to the same exponent. The existing `PythUtils.convertToUint` utility already handles this correctly:

```solidity
uint256 currentNorm  = PythUtils.convertToUint(
    priceFeeds[i].price.price, priceFeeds[i].price.expo, 18);
uint256 previousNorm = PythUtils.convertToUint(
    previousFeed.price.price,  previousFeed.price.expo,  18);

uint256 numerator    = currentNorm > previousNorm
    ? currentNorm - previousNorm
    : previousNorm - currentNorm;
uint256 deviationBps = Math.mulDiv(numerator, 10_000, previousNorm);
``` [5](#0-4) 

---

### Proof of Concept

1. A subscription is created with `updateOnDeviation = true` and `deviationThresholdBps = 100` (1%).
2. First `updatePriceFeeds` call stores: `price = 100_000_000`, `expo = -8` (actual $1.00).
3. Pyth changes the exponent for this feed from `-8` to `-6`.
4. Second `updatePriceFeeds` call arrives with: `price = 100_000_000`, `expo = -6` (actual $100.00 — a 9900 bps real move).
5. The Scheduler computes: `numerator = |100_000_000 − 100_000_000| = 0`, `deviationBps = 0`.
6. `0 < 100` → `revert UpdateConditionsNotMet()`.
7. The subscription's stored price remains $1.00 while the true market price is $100.00. Any protocol reading `getPriceUnsafe` or `getPrice` from this subscription now receives a price that is 100× wrong. [6](#0-5)

    function convertToUint(
        int64 price,
        int32 expo,
        uint8 targetDecimals
    ) public pure returns (uint256) {
        if (price < 0) {
            revert PythErrors.NegativeInputPrice();
        }
        if (expo < -255) {
            revert PythErrors.InvalidInputExpo();
        }

        // If targetDecimals is 6, we want to multiply the final price by 10 ** -6
        // So the delta exponent is targetDecimals + currentExpo
        int32 deltaExponent = int32(uint32(targetDecimals)) + expo;

        // Bounds check: prevent overflow/underflow with base 10 exponentiation
        // Calculation: 10 ** n <= (2 ** 256 - 63) - 1
        //              n <= log10((2 ** 193) - 1)
        //              n <= 58.2
        if (deltaExponent > 58 || deltaExponent < -58)
            revert PythErrors.ExponentOverflow();

        // We can safely cast the price to uint256 because the above condition will revert if the price is negative
        uint256 unsignedPrice = uint256(uint64(price));

        if (deltaExponent > 0) {
            (bool success, uint256 result) = Math.tryMul(
                unsignedPrice,
                10 ** uint32(deltaExponent)
            );
            // This condition is unreachable since we validated deltaExponent bounds above.
            // But keeping it here for safety.
            if (!success) {
                revert PythErrors.CombinedPriceOverflow();
            }
            return result;
        } else {
            (bool success, uint256 result) = Math.tryDiv(
                unsignedPrice,
                10 ** uint(Math.abs(deltaExponent))
            );
            // This condition is unreachable since we validated deltaExponent bounds above.
            // But keeping it here for safety.
            if (!success) {
                revert PythErrors.CombinedPriceOverflow();
            }
            return result;
        }
    }
