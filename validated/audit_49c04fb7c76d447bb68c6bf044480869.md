Audit Report

## Title
Off-by-one in spawn guard allows 17 VMs when `MAX_VMS_COUNT` is 16 - (File: script/src/scheduler.rs)

## Summary
`MAX_VMS_COUNT` is defined as `16`, representing the maximum number of VMs that can exist simultaneously. The spawn guard at line 532 uses strict `>` instead of `>=`, so when exactly 16 VMs are alive the guard does not fire and `boot_vm` is called, creating a 17th VM. This is an incorrect implementation of the CKB-VM spawn resource limit.

## Finding Description
`MAX_VMS_COUNT: u64 = 16` is declared at line 34. The `Message::Spawn` arm at line 532 checks:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
```

`instantiated` always contains the root VM from scheduler initialization. After 15 child VMs have been spawned, `suspended.len() + instantiated.len() == 16`. The expression `16 > 16` evaluates to `false`, the guard does not fire, and `boot_vm` at line 539 creates a 16th child (17th VM total). The guard only fires at count 17, meaning the enforced ceiling is 17, not 16. The correct operator is `>=`.

## Impact Explanation
This is an incorrect implementation of the CKB-VM spawn resource limit, matching the allowed High impact class: "Incorrect implementation or behavior of CKB-VM or system scripts." Any V2 (Meepo hardfork) script can exceed the defined VM count ceiling by exactly one, bypassing the resource constraint as specified.

## Likelihood Explanation
The path is fully attacker-controlled with no privileges required. A script author submits a V2 ELF transaction that calls `spawn` 17 times, keeping all child VMs alive (e.g., blocked on a pipe read). No PoW, no social engineering, no victim interaction is needed. The bug is reachable on any node that has activated the Meepo hardfork. Because all nodes share the same binary, there is no consensus divergence — the bypass is uniform across the network.

## Recommendation
Change `>` to `>=` at line 532:

```rust
// Before (buggy):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

## Proof of Concept
1. Write a V2 ELF script that calls `spawn` in a loop, passing a pipe fd to each child so children block on `read` and remain alive.
2. Boot a scheduler with this script.
3. Spawn exactly `MAX_VMS_COUNT` (16) child VMs; verify all are alive in `suspended + instantiated`.
4. Issue one additional `spawn` syscall.
5. With the current `>` check: the call succeeds (returns `SUCCESS`) and a 17th VM is instantiated.
6. With the corrected `>=` check: the call returns `MAX_VMS_SPAWNED` (error code 8), correctly enforcing the limit.
