Audit Report

## Title
Unbounded `pending_get_block_proposals` Map Growth via Relay Peer Messages Causes Memory/CPU Exhaustion - (File: sync/src/types/mod.rs)

## Summary
Any peer completing the `RelayV3` handshake can send `GetBlockProposal` messages containing up to `max_block_proposals_limit * max_uncles_num` (≈3,000 on mainnet) unique fabricated `ProposalShortId` values per message. IDs absent from the tx-pool are inserted unconditionally into the unbounded `pending_get_block_proposals` `DashMap` in `SyncState`. On each relay timer tick, `drain_get_block_proposals` performs an O(n) clone of the entire map, which can exhaust memory and stall the relay subsystem. A rate limiter of 30 req/s per peer exists but does not prevent the attack when multiple peers cooperate.

## Finding Description
`SyncState` holds an unbounded `DashMap`:

```rust
// sync/src/types/mod.rs:1330
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>,
```

`insert_get_block_proposals` inserts every supplied ID with no cumulative size guard:

```rust
// sync/src/types/mod.rs:1594-1601
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
```

`GetBlockProposalProcess::execute` enforces only a per-message limit (`max_block_proposals_limit * max_uncles_num`, ≈3,000) with no cumulative cap. IDs absent from the tx-pool are forwarded directly to `insert_get_block_proposals` (lines 68–77 of `get_block_proposal_process.rs`).

On the relay timer, `drain_get_block_proposals` clones the entire map before clearing it:

```rust
// sync/src/types/mod.rs:1586-1592
pub fn drain_get_block_proposals(
    &self,
) -> DashMap<packed::ProposalShortId, HashSet<PeerIndex>> {
    let ret = self.pending_get_block_proposals.clone();
    self.pending_get_block_proposals.clear();
    ret
}
```

A rate limiter keyed by `(PeerIndex, message_item_id)` at 30 req/s per peer does exist in `sync/src/relayer/mod.rs` (lines 81–92, 116–123) and applies to `GetBlockProposal`. However, this only limits a single peer to 30 × 3,000 = 90,000 entries/second. With `MAX_RELAY_PEERS = 128` cooperating peers, the insertion rate reaches ~11.5 million entries/second. The `disconnected()` handler (lines 1607–1616) does **not** clean up `pending_get_block_proposals` entries for the disconnected peer, so entries from banned/disconnected peers persist until the next drain tick.

The O(n) clone in `drain_get_block_proposals` acquires locks on every internal DashMap shard, stalling all concurrent readers/writers for the duration of the clone.

## Impact Explanation
With enough cooperating peers (feasible via Sybil), the `pending_get_block_proposals` map can be inflated to tens of millions of entries between drain ticks. The O(n) clone at each tick causes proportional memory allocation and CPU stall under shard locks, stalling block-proposal and transaction propagation. Sustained flooding can crash the node process via OOM. This matches the **High** impact: *Vulnerabilities which could easily crash a CKB node* (10,001–15,000 points).

## Likelihood Explanation
`GetBlockProposal` is part of the standard unauthenticated `RelayV3` protocol. Fabricating unique 10-byte `ProposalShortId` values is trivial. The per-peer rate limiter (30 req/s) reduces single-peer impact but does not prevent multi-peer amplification. `MAX_RELAY_PEERS = 128` is a public constant. No cumulative map-size cap or per-peer quota exists. The CHANGELOG confirms analogous unbounded-growth bugs in `inflight_proposals` (#3093) and `inflight_blocks` (#3094) were previously fixed, but `pending_get_block_proposals` was not addressed.

## Recommendation
1. **Cap the map size**: in `insert_get_block_proposals`, refuse new insertions once `pending_get_block_proposals.len()` exceeds a configurable bound (e.g., `max_block_proposals_limit * max_uncles_num * MAX_RELAY_PEERS`).
2. **Per-peer quota**: track each peer's contribution count and reject further insertions (and optionally ban the peer) once a per-peer limit is reached.
3. **Avoid full clone on drain**: replace the clone+clear pattern with `std::mem::replace(&mut *map, DashMap::new())` or equivalent swap, eliminating the O(n) allocation under shard locks.
4. **Clean up on disconnect**: in `disconnected()`, remove all `pending_get_block_proposals` entries associated with the disconnecting peer.

## Proof of Concept
```
# Requires N cooperating attacker peers (N ≤ MAX_RELAY_PEERS = 128)
# Each peer is rate-limited to 30 GetBlockProposal msgs/sec

for each attacker_peer in range(N):
    loop forever (at 30 req/s):
        ids = [random_10_byte_id() for _ in range(3000)]  # unique, non-existent
        msg = GetBlockProposal { block_hash: tip_hash, proposals: ids }
        send(msg)
        # passes per-message limit (3000 ≤ limit) and rate limit (30/s)
        # victim inserts all 3000 IDs into pending_get_block_proposals

# Insertion rate: N * 30 * 3000 = N * 90,000 entries/sec
# With N=128: ~11.5M entries/sec

# On next relay timer tick:
#   drain_get_block_proposals() clones the inflated DashMap
#   → O(entries) memory allocation + CPU under all shard locks
#   → relay timer stalls; node OOM-crashes or relay service becomes unresponsive
```

Entry point: `GetBlockProposalProcess::execute` in `sync/src/relayer/get_block_proposal_process.rs`, reachable from any unauthenticated relay peer via the `RelayV3` protocol handler.
