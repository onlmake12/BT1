### Title
EIP-7702 EOA Requester Bypasses `extcodesize` Contract Check in `revealWithCallback`, Enabling Unlimited-Gas Callback and Permanent Request DoS — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`'s `revealWithCallback` function (the legacy/Path-2 branch, active when `req.gasLimit10k == 0`), the contract uses `extcodesize` to decide whether to invoke `_entropyCallback` on the requester. Under EIP-7702 (live since the Pectra hardfork), an EOA can have delegated code, making `extcodesize` return non-zero. This causes the contract to call `_entropyCallback` on the EIP-7702 EOA with **unlimited gas** and **no revert-catching**, enabling either a permanent DoS on the request or arbitrary reentrancy into the Entropy contract.

---

### Finding Description

`revealWithCallback` has two execution branches:

**Path 1** (`req.gasLimit10k != 0`, modern providers): uses `excessivelySafeCall` with a bounded gas limit, catches reverts, and has a `CALLBACK_IN_PROGRESS` reentrancy guard.

**Path 2** (`req.gasLimit10k == 0`, legacy providers): clears the request first (CEI), then checks `extcodesize` to decide whether to call `_entropyCallback`. The call is made with **unlimited gas** and **no try/catch**. [1](#0-0) 

The `extcodesize` check at line 672 is the root cause:

```solidity
uint len;
assembly {
    len := extcodesize(callAddress)
}
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    );
}
```

Under EIP-7702, an EOA that has applied a delegation designation will have `extcodesize > 0`. This means:

1. An EOA that called `requestWithCallback` as a plain EOA (expecting no callback) will receive an unexpected `_entropyCallback` invocation if EIP-7702 delegation is applied between request and reveal.
2. The call is made with **unlimited gas** — unlike Path 1 which uses `excessivelySafeCall` with `uint256(req.gasLimit10k) * TEN_THOUSAND`.
3. There is **no revert-catching** — if the EIP-7702 delegated code reverts inside `_entropyCallback`, the entire `revealWithCallback` transaction reverts, including the `clearRequest` at line 666. The request remains active but every subsequent reveal attempt also reverts, permanently blocking fulfillment. [2](#0-1) 

Path 1 (for comparison) uses `excessivelySafeCall` with a gas cap and catches reverts: [3](#0-2) 

The `CALLBACK_IN_PROGRESS` status constant is explicitly documented as a reentry guard, confirming the design intent: [4](#0-3) 

Path 2 has no equivalent guard and no gas cap.

---

### Impact Explanation

**DoS on request fulfillment**: If an EOA requester applies EIP-7702 delegation (for any reason — e.g., adopting a smart wallet) with code that does not implement `_entropyCallback` or that reverts inside it, every call to `revealWithCallback` for that request will revert. The request is permanently stuck until the delegation is removed. The requester loses the fee paid for the request and the provider's sequence number for that slot is consumed but never resolved.

**Unlimited-gas reentrancy**: If the EIP-7702 delegated code implements `_entropyCallback` with malicious logic, it executes with unlimited gas. While the current request is cleared before the callback (CEI pattern at line 666), the callback can still:
- Call `requestWithCallback` to consume additional provider sequence numbers.
- Call `withdraw` or `withdrawAsFeeManager` (both use CEI, so direct double-withdrawal is blocked, but interaction with other in-flight state is possible).
- Call `revealWithCallback` for other active requests. [5](#0-4) 

---

### Likelihood Explanation

EIP-7702 is live on Ethereum mainnet since the Pectra upgrade. Users increasingly adopt EIP-7702 for smart-wallet features (batching, gas sponsorship). A user who has an active Entropy request and then applies EIP-7702 delegation — without knowing it affects their pending request — will find their request permanently stuck. This is a realistic, unprivileged user action with no malicious intent required to trigger the DoS.

The attacker-controlled entry path is:
1. Unprivileged user calls `requestWithCallback` / `requestV2` as a plain EOA.
2. User applies EIP-7702 delegation (their own key, no privileged access needed).
3. Provider (or anyone) calls `revealWithCallback`.
4. `extcodesize` returns non-zero; `_entropyCallback` is called on the EIP-7702 EOA.
5. If the delegated code reverts → permanent DoS; if it executes → unlimited-gas callback.

---

### Recommendation

1. **Wrap the Path-2 callback in a try/catch** (matching the spirit of Path 1's `excessivelySafeCall`):
   ```solidity
   if (len != 0) {
       try IEntropyConsumer(callAddress)._entropyCallback(
           sequenceNumber, provider, randomNumber
       ) {} catch {}
   }
   ```
2. **Apply a gas stipend** to the Path-2 callback, consistent with Path 1's bounded approach.
3. **Alternatively**, treat any address whose `extcodesize` is non-zero only due to EIP-7702 delegation (detectable via the `0xef01` prefix in the code) as an EOA and skip the callback.

### Proof of Concept

```
1. Provider registers with defaultGasLimit = 0 (legacy provider, triggers Path 2).
2. EOA (user1) calls requestWithCallback{value: fee}(provider, userCommitment).
   → req.gasLimit10k = 0, req.requester = user1 (plain EOA, extcodesize = 0 at this point).
3. user1 applies EIP-7702 delegation to a contract that implements _entropyCallback
   to revert unconditionally.
   → extcodesize(user1) is now > 0.
4. Provider calls revealWithCallback(provider, seqNum, userContrib, providerContrib).
   → Path 2 branch taken (gasLimit10k == 0).
   → clearRequest(provider, seqNum) executes (line 666).
   → extcodesize(user1) = non-zero → enters the if(len != 0) branch.
   → IEntropyConsumer(user1)._entropyCallback(...) reverts.
   → Entire revealWithCallback reverts; clearRequest is rolled back.
5. Request remains active. Every subsequent revealWithCallback call also reverts.
   → Request is permanently stuck; user1's fee is lost.
``` [6](#0-5) [7](#0-6)
