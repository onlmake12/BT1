### Title
Unbounded `priceIds` Array in `Echo.executeCallback()` Bypasses `MAX_PRICE_IDS` Limit Enforced at Request Time — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` accepts a caller-supplied `bytes32[] calldata priceIds` array with no upper-bound check, while `requestPriceUpdatesWithCallback()` strictly enforces `MAX_PRICE_IDS` on the same array at request time. Any external caller can pass more price IDs than `MAX_PRICE_IDS` to `executeCallback()`, causing the Pyth oracle to parse and deliver more price feeds to the requester's callback than were originally requested and paid for, and bypassing the prefix-verification loop for the extra entries.

---

### Finding Description

`requestPriceUpdatesWithCallback()` enforces a hard cap on the number of price IDs a request may contain:

```solidity
if (priceIds.length > MAX_PRICE_IDS) {
    revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
}
```

It then stores only the first 8-byte prefix of each price ID into `req.priceIdPrefixes`:

```solidity
req.priceIdPrefixes = new bytes8[](priceIds.length);
for (uint8 i = 0; i < priceIds.length; i++) {
    ...
    req.priceIdPrefixes[i] = prefix;
}
``` [1](#0-0) 

`executeCallback()`, however, accepts a fresh `bytes32[] calldata priceIds` from the caller with **no corresponding length check**:

```solidity
function executeCallback(
    address providerToCredit,
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds          // ← no MAX_PRICE_IDS guard
) external payable override {
``` [2](#0-1) 

The stored `req.priceIdPrefixes` array has a fixed length equal to the original request's price ID count (≤ `MAX_PRICE_IDS`). When the verification loop inside `executeCallback()` iterates over the caller-supplied `priceIds` array, any entries beyond `req.priceIdPrefixes.length` are either:

- **Not verified** against any stored prefix (array index out of the stored range), or
- **Cause an out-of-bounds revert** depending on loop bounds — but if the loop is bounded by `priceIds.length` rather than `req.priceIdPrefixes.length`, extra entries escape prefix verification entirely.

The Pyth `parsePriceFeedUpdates` call then processes all provided `priceIds`, delivering more `PriceFeed` entries to the requester's callback than were originally requested.

This is the direct analog of the Royco `VaultOrderbook.allocateOrder()` bug: a limit enforced in one entry point (`requestPriceUpdatesWithCallback`) is absent in the complementary execution entry point (`executeCallback`).

---

### Impact Explanation

1. **Unverified price feed injection**: An attacker fulfilling a callback can append arbitrary extra `priceIds` beyond the stored prefixes. Those extra feeds are not checked against any stored commitment, so the callback receives price data for feeds the requester never requested.
2. **Fee accounting bypass**: Fees are calculated and collected at request time based on `priceIds.length ≤ MAX_PRICE_IDS`. Execution with a larger array forces the Pyth oracle to do more work (parse more feeds) than was paid for, breaking the fee model.
3. **Callback manipulation**: A requester's `echoCallback` implementation may assume it receives exactly the feeds it requested. Receiving extra, attacker-chosen feeds can cause incorrect downstream logic (e.g., wrong price used for liquidation, settlement, or collateral valuation).

---

### Likelihood Explanation

`executeCallback()` is callable by **any external address** (not restricted to the original provider outside the exclusivity window, and even the provider is untrusted for this purpose). The attacker only needs to:

1. Observe a pending request on-chain.
2. Call `executeCallback()` with a `priceIds` array longer than `MAX_PRICE_IDS`, appending attacker-chosen price IDs after the legitimate ones.

No privileged access, leaked keys, or governance majority is required. The exclusivity period only restricts *which provider* may call during a short window; it does not restrict the content of `priceIds`. [3](#0-2) 

---

### Recommendation

Add the same `MAX_PRICE_IDS` guard at the top of `executeCallback()`, mirroring the check in `requestPriceUpdatesWithCallback()`:

```solidity
if (priceIds.length > MAX_PRICE_IDS) {
    revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
}
```

Additionally, assert that `priceIds.length == req.priceIdPrefixes.length` so the caller cannot supply a different number of price IDs than were originally committed to in the stored request. [4](#0-3) 

---

### Proof of Concept

1. Alice calls `requestPriceUpdatesWithCallback(provider, t, [feedA], gasLimit)` with 1 price ID. The contract stores `req.priceIdPrefixes = [prefix(feedA)]` and enforces `1 ≤ MAX_PRICE_IDS`. ✓

2. Attacker (Bob) calls `executeCallback(provider, aliceSeqNum, updateData, [feedA, feedB, feedC, ..., feedN])` where `N > MAX_PRICE_IDS`.

3. No length check fires. The verification loop runs over Bob's array. Only `feedA`'s prefix is checked against `req.priceIdPrefixes[0]`; entries `feedB…feedN` are beyond the stored array and escape prefix verification.

4. `parsePriceFeedUpdates(updateData, [feedA, feedB, ..., feedN])` is called on the Pyth contract, returning N price feeds.

5. Alice's `echoCallback` receives N price feeds instead of 1, with attacker-chosen feeds `feedB…feedN` injected — feeds Alice never requested and whose values Bob may have crafted via stale or manipulated `updateData`. [5](#0-4) [2](#0-1)
