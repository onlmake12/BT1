### Title
Tx-Pool Admission Uses Size-Only Fee Rate Check While Actual Weight Includes Cycles, Allowing Below-`min_fee_rate` Transactions to Fill the Pool - (`tx-pool/src/util.rs`)

---

### Summary

CKB's tx-pool admission gate checks fee rate using only the serialized transaction size, but the actual fee rate used for eviction, sorting, and block selection uses `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. An attacker can craft transactions with high cycles and small size that pass the admission check but carry an effective fee rate far below `min_fee_rate`. These transactions occupy pool space, and because the eviction key also uses the weight-based fee rate, they are the first to be evicted — but the attacker can continuously re-submit them to maintain pool pressure and cause legitimate transactions to receive `Reject::Full`.

---

### Finding Description

**Admission-time check** (`tx-pool/src/util.rs`, `check_tx_fee`):

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
```

The gate is: `fee ≥ min_fee_rate × size`. [1](#0-0) 

**Actual weight function** (`util/types/src/core/tx_pool.rs`, `get_transaction_weight`):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [2](#0-1) 

**Actual fee rate** used for sorting and eviction (`tx-pool/src/component/entry.rs`, `TxEntry::fee_rate()` and `EvictKey`):

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [3](#0-2) [4](#0-3) 

**Concrete mismatch example:**

| Parameter | Value |
|---|---|
| `tx_size` | 1,000 bytes |
| `cycles` | 70,000,000 (= `max_tx_verify_cycles`) |
| `weight` | max(1,000, 70,000,000 × 0.000_170_571_4) = **11,940 bytes** |
| Admission fee required | 1,000 shannons (1,000 shannons/KB × 1 KB) |
| Effective fee rate | 1,000 / 11,940 × 1,000 ≈ **83.7 shannons/KB** |
| `min_fee_rate` | **1,000 shannons/KB** |

The transaction passes admission at 1,000 shannons/KB but its true fee rate is ~12× below `min_fee_rate`.

**Pool eviction** (`tx-pool/src/pool.rs`, `limit_size`) evicts by `EvictKey` (weight-based fee rate), so attacker transactions are evicted first when the pool is full. However, the attacker can continuously re-submit to maintain pool pressure. [5](#0-4) 

---

### Impact Explanation

1. An attacker submits transactions with high cycles and small size. Each passes `check_tx_fee` (size-based gate) but carries an effective fee rate far below `min_fee_rate`.
2. These transactions accumulate in the pending pool (up to `max_tx_pool_size = 180 MB`).
3. When the pool is full, legitimate transactions receive `Reject::Full`.
4. The eviction mechanism removes the attacker's transactions first (lowest weight-based fee rate), but the attacker can immediately re-submit to refill the pool.
5. Legitimate transactions experience repeated rejection windows. Miners assembling blocks via `package_proposals` → `package_txs` → `TxSelector::txs_to_commit` will still select the best-fee-rate transactions, but the pending pool churn degrades throughput and causes relay failures. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

- Entry path: any unprivileged tx-pool submitter via `send_transaction` RPC or P2P relay.
- The attacker needs CKB to pay fees (locked in UTXOs for up to 12 hours until expiry). For 1,000-byte transactions at 1,000 shannons each, filling 180 MB requires ~180,000 transactions. The capital requirement is non-trivial but not prohibitive for a motivated attacker.
- The attack is continuous: the attacker must keep re-submitting as eviction clears space. This raises the sustained cost but does not eliminate the threat.
- The comment in the code (`"here min fee rate is used as a cheap check"`) confirms the developers are aware of the approximation, but no compensating control (e.g., weight-based eviction floor, re-check on eviction) prevents the admission bypass. [8](#0-7) 

---

### Recommendation

Replace the size-only admission check in `check_tx_fee` with the full weight-based check:

```rust
// Use weight (max of size and cycle-equivalent bytes) for the fee rate gate,
// consistent with how fee_rate() is computed on TxEntry.
let weight = get_transaction_weight(tx_size, /* cycles not yet known here */);
```

Because cycles are not known until after script execution, the practical fix is to perform a **post-verification re-check**: after `verify_rtx` returns the actual cycle count, recompute the weight-based fee rate and reject the entry if it falls below `min_fee_rate` before calling `submit_entry`. This mirrors the approach used in `TxEntry::fee_rate()` and `EvictKey`.

Alternatively, enforce a minimum fee based on weight at the point where `TxEntry` is constructed (line 751 of `process.rs`), before the entry is inserted into the pool. [9](#0-8) 

---

### Proof of Concept

1. Craft a CKB transaction with:
   - Serialized size ≈ 1,000 bytes
   - A lock/type script that consumes ≈ 70,000,000 cycles (the `max_tx_verify_cycles` default)
   - Fee = 1,001 shannons (just above `min_fee_rate × size = 1,000 shannons`)
2. Submit via `send_transaction` RPC. The node calls `check_tx_fee` → passes (1,001 ≥ 1,000).
3. After verification, `TxEntry::fee_rate()` computes weight = max(1,000, 11,940) = 11,940 → effective fee rate ≈ 83.8 shannons/KB, far below `min_fee_rate = 1,000`.
4. Repeat with ~180,000 such transactions (each from a distinct UTXO) to fill the 180 MB pool.
5. Submit a legitimate transaction. It receives `Reject::Full`.
6. The eviction removes the attacker's transactions (lowest fee rate), but the attacker immediately re-submits to refill, sustaining the rejection window.

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
