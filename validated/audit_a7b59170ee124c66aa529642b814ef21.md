Looking at the actual code to verify each claim in the report.

**Verified facts:**

1. `check_tx_fee` at [1](#0-0)  uses `tx_size as u64` directly as the weight argument, with the comment explicitly acknowledging this is theoretically incorrect.

2. `check_tx_fee` is called in `pre_check` at [2](#0-1)  — before `verify_rtx` runs, so cycles are not yet known.

3. In `_process_tx`, after `verify_rtx` returns actual cycles at [3](#0-2) , there is **no subsequent weight-based fee rate re-check** before `submit_entry` is called.

4. `fee_rate()` using `get_transaction_weight` at [4](#0-3)  is only used for pool ordering/eviction, not admission.

5. `DEFAULT_MAX_TX_VERIFY_CYCLES = TWO_IN_TWO_OUT_CYCLES * 20` at [5](#0-4)  — confirming ~70M cycles is the per-tx cap.

6. `get_transaction_weight` at [6](#0-5)  confirms weight = `max(size, cycles × 0.000_170_571_4)`.

**One inaccuracy in the report:** Impact point 3 claims attacker transactions are "evicted last." The `EvictKey` at [7](#0-6)  uses actual weight-based fee rate, so attacker transactions (lowest actual fee rate) are evicted **first**, not last. However, this does not invalidate the core finding — the verification cost (70M cycles) is already paid by the node before eviction.

The core vulnerability is confirmed: the admission gate uses size-only weight, cycles are verified afterward but never re-checked against `min_fee_rate`, and no guard exists between `verify_rtx` and `submit_entry`.

Audit Report

## Title
Min-Fee-Rate Invariant Broken by Size-Only Weight in `check_tx_fee` — (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` computes the minimum required fee using only the serialized byte size of a transaction as the weight, rather than `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. Cycles are determined by `verify_rtx` after the fee gate, but no weight-based fee rate re-check occurs before pool admission. An unprivileged caller can submit cycle-heavy, byte-small transactions that pass the fee gate at fee rates orders of magnitude below the configured `min_fee_rate`, forcing the node to execute up to 70M cycles of script verification per transaction at near-zero cost.

## Finding Description
`check_tx_fee` is called in `pre_check` before script execution, using only `tx_size` as the weight:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The admission flow in `_process_tx` is:
1. `pre_check` → `check_tx_fee(tx_size)` — fee gate passes on byte size alone
2. `verify_rtx` → actual cycles returned (up to 70M)
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with actual cycles
4. `submit_entry` — admitted to pool

No step between 2 and 4 re-evaluates the fee rate using `get_transaction_weight(tx_size, verified.cycles)`. The weight-based `fee_rate()` in `TxEntry` is used only for pool ordering and eviction, not admission.

With `min_fee_rate = 1000 shannons/KW` and a 100-byte transaction consuming 70M cycles:
- Actual weight: `max(100, 70,000,000 × 0.000_170_571_4) = 11,940 bytes`
- `check_tx_fee` requires: `1000 × 100 / 1000 = 100 shannons`
- Actual fee rate if 100 shannons paid: `100 × 1000 / 11,940 ≈ 8 shannons/KW` — 125× below minimum

The code comment at the call site explicitly acknowledges the theoretical incorrectness but treats it as acceptable for a "cheap check," with no compensating post-verification check anywhere in the admission path.

## Impact Explanation
This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

An attacker can repeatedly submit cycle-heavy transactions via `send_transaction` RPC, each forcing the node to execute up to 70M cycles of CKB-VM script verification while paying only ~0.8% of the required fee. The `min_fee_rate` anti-spam invariant — the primary economic defense against pool flooding — is bypassed. Sustained submission fills the verification queue and consumes node CPU at a cost 125× below the intended floor, constituting network congestion achievable with minimal economic cost.

## Likelihood Explanation
The entry point is the `send_transaction` RPC, reachable by any unprivileged caller with no keys or special roles. The attacker writes a CKB-VM lock script (e.g., a tight loop) consuming ~70M cycles, stores it in a cell dep to keep the transaction body small (~100–200 bytes serialized), and submits transactions spending cells locked by this script. The transaction is consensus-valid; it simply pays less than the weight-based minimum. The attack is repeatable with no per-attempt barrier beyond the nominal fee (100 shannons ≈ negligible).

## Recommendation
After `verify_rtx` returns actual cycles, perform a weight-based fee rate check before `submit_entry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

Alternatively, restructure `check_tx_fee` to accept an optional `cycles` parameter: use size-only for the pre-check (as now), and pass actual cycles for a post-verification check. The pre-check can remain as a cheap early rejection for obviously underpriced transactions; the post-check enforces the true invariant.

## Proof of Concept
1. Write a CKB-VM lock script that executes a tight loop consuming ~70,000,000 cycles. Deploy it in a cell.
2. Construct a transaction spending a cell locked by this script, with the script stored as a cell dep (keeping serialized tx size ~100 bytes). Set `fee = 100 shannons` (`outputs_capacity = inputs_capacity - 100`).
3. Submit via `send_transaction` RPC to a node with default `min_fee_rate = 1000`.
4. `check_tx_fee` computes `min_fee = 1000 × 100 / 1000 = 100 shannons`. Fee equals min_fee → admitted past the gate.
5. `verify_rtx` executes the script, consuming 70M cycles. No post-verification fee rate check occurs.
6. Transaction enters the pool with actual fee rate ≈ 8 shannons/KW — 125× below `min_fee_rate`.
7. Repeat to continuously force 70M-cycle verifications at negligible cost, congesting the node's verification pipeline.
