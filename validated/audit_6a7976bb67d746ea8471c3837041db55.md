### Title
Keeper Can Inflate Subscription Fee via Uncapped `tx.gasprice` in `_processFeesAndPayKeeper` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `_processFeesAndPayKeeper` function in `Scheduler.sol` calculates the keeper reimbursement using `tx.gasprice`, which is fully controlled by the keeper (the transaction sender). Since there is no cap on `tx.gasprice`, any unprivileged keeper can set an arbitrarily high gas price to inflate the `GAS_OVERHEAD` component of the fee, draining the subscription manager's deposited balance faster than expected and extracting profit at the subscription manager's expense.

### Finding Description
In `Scheduler._processFeesAndPayKeeper()`, the keeper fee is calculated as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [1](#0-0) 

`GAS_OVERHEAD = 30000` is a fixed constant added to the actual gas used in the function. [2](#0-1) 

Since `tx.gasprice` is set by the keeper (the `msg.sender` calling `updatePriceFeeds`), a malicious keeper can set it to an arbitrarily high value. This inflates the `GAS_OVERHEAD * tx.gasprice` component, which is pure profit for the keeper extracted from the subscription balance. The keeper's net profit per call equals `GAS_OVERHEAD * tx.gasprice + keeperSpecificFee`. By setting a higher `tx.gasprice`, the keeper extracts more from the subscription balance than they actually spent on gas overhead.

There is no cap on `tx.gasprice` anywhere in the fee calculation path. The subscription manager has no mechanism to limit the gas price that keepers use when calling `updatePriceFeeds`. [3](#0-2) 

The `updatePriceFeeds` function is permissionless — anyone can call it as a keeper. [4](#0-3) 

### Impact Explanation
A malicious keeper can drain a subscription's balance faster than expected by setting an artificially high `tx.gasprice`. The subscription manager loses more ETH than they should for each update. Concrete example with `tx.gasprice = 1000 gwei` and `actual_gas_used = 200,000`:

- Keeper pays to network: `200,000 × 1000 gwei = 0.2 ETH`
- Keeper receives from contract: `(200,000 + 30,000) × 1000 gwei = 0.23 ETH`
- Net profit: `0.03 ETH` per call, extracted from the subscription balance

For a 1 ETH subscription, the balance is drained in approximately 4–5 calls. The subscription manager's deposited funds are consumed at a rate far exceeding what legitimate gas costs would justify. This is a direct financial loss to the subscription manager (user), analogous to the external report's taker receiving fewer tokens than expected.

### Likelihood Explanation
The attack path is fully permissionless. Any address can call `updatePriceFeeds()` as a keeper — no registration or privileged role is required. The attacker only needs to:
1. Identify a subscription with a meaningful balance (visible on-chain via `getActiveSubscriptions`)
2. Wait for the update conditions to be met (heartbeat interval or price deviation threshold)
3. Submit `updatePriceFeeds()` with valid Hermes update data and an inflated `tx.gasprice`

Valid update data is freely obtainable from the public Hermes API. The attack is economically rational whenever the subscription balance is large enough to cover the keeper's inflated gas cost and yield a profit from the `GAS_OVERHEAD` component. Subscriptions with large balances (up to `MAX_DEPOSIT_LIMIT = 100 ETH` for permanent subscriptions) are attractive targets. [5](#0-4) 

### Recommendation
Add a configurable `maxGasPrice` parameter to `SubscriptionParams` that the subscription manager sets at creation time. In `_processFeesAndPayKeeper`, cap the gas price used in the fee calculation:

```solidity
uint256 effectiveGasPrice = tx.gasprice < params.maxGasPrice
    ? tx.gasprice
    : params.maxGasPrice;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, use a protocol-level `maxGasPrice` cap set by the admin, or replace `tx.gasprice` with a TWAP of recent on-chain gas prices to prevent single-transaction manipulation.

### Proof of Concept
1. Subscription manager calls `createSubscription` with 1 ETH balance and a 60-second heartbeat.
2. Malicious keeper waits 60 seconds for the heartbeat condition to be satisfied.
3. Keeper fetches valid Pyth update data from the public Hermes API.
4. Keeper submits `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = 1000 gwei`.
5. Inside `_processFeesAndPayKeeper`: `gasCost = (200,000 + 30,000) × 1000 gwei = 0.23 ETH`.
6. `status.balanceInWei -= 0.23 ETH + pythFee`; keeper receives `0.23 ETH`.
7. Keeper's actual network cost: `200,000 × 1000 gwei = 0.2 ETH`. Net profit: `0.03 ETH`.
8. Keeper repeats every 60 seconds. After ~4 calls the 1 ETH subscription is drained.
9. The subscription manager's balance is consumed at 23× the rate expected at a normal 1 gwei gas price, with the excess flowing to the malicious keeper. [6](#0-5)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
