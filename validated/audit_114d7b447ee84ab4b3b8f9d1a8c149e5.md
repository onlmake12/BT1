Audit Report

## Title
`CapacityVerifier::valid_dao_withdraw_transaction()` Skips Overflow Check for DAO Deposit Cells, Enabling Non-DAO Capacity Inflation — (`File: verification/src/transaction_verifier.rs`)

## Summary

`CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` for any transaction whose inputs include a DAO-typed cell, including deposit-phase cells (`cell_data == 0x0000000000000000`). This causes the `OutputsSumOverflow` guard to be skipped for the entire transaction. Because the DAO type script only validates the DAO cell pair and not non-DAO outputs, a transaction sender can inflate non-DAO output capacity arbitrarily in a deposit→prepare transaction, creating CKB from nothing.

## Finding Description

`valid_dao_withdraw_transaction()` checks only whether any input uses the DAO type script, without distinguishing between deposit cells (`deposited_block_number == 0`) and prepare cells (`deposited_block_number > 0`): [1](#0-0) 

When this returns `true`, `CapacityVerifier::verify()` skips the `OutputsSumOverflow` check for the **entire** transaction, including all non-DAO outputs: [2](#0-1) 

`CapacityVerifier` holds no `data_loader` and cannot read cell data to distinguish phases: [3](#0-2) 

The DAO C script, invoked via `ScriptVerifier`, validates only the DAO cell pair. For a deposit→prepare transaction, it enforces `prepare_output.capacity == deposit_input.capacity` for the DAO cell, but does not inspect non-DAO outputs: [4](#0-3) 

By contrast, `DaoCalculator::transaction_maximum_withdraw()` correctly distinguishes phases by reading cell data: [5](#0-4) 

`DaoScriptSizeVerifier` already holds a `data_loader` and reads cell data to distinguish deposit from prepare cells; `CapacityVerifier` does not: [6](#0-5) 

**Exploit flow:**
1. Attacker owns a DAO deposit cell (Input A: 100 CKB, `cell_data = 0x0000000000000000`) and a regular cell (Input B: 1 CKB).
2. Attacker crafts a deposit→prepare transaction: Output A is a DAO prepare cell (100 CKB, `cell_data = block_number`); Output B is a regular cell with 1,000,000 CKB.
3. `valid_dao_withdraw_transaction()` returns `true` because Input A has the DAO type script → `OutputsSumOverflow` check is skipped.
4. The DAO C script validates Output A (100 CKB == Input A 100 CKB) and passes.
5. No verifier checks Output B (1,000,000 CKB) against Input B (1 CKB). The attacker mints 999,999 CKB from nothing.

## Impact Explanation

**Critical** — matches the impact class: *"Vulnerabilities which could easily damage CKB economy."* Any unprivileged user can mint arbitrary CKB by submitting a crafted deposit→prepare transaction. Nodes accepting such a transaction inflate the total CKB supply, breaking the economic model and the security guarantees of all CKB holders. The inflation is consensus-level: any node that processes the transaction will accept the inflated supply, causing a chain-wide accounting violation.

## Likelihood Explanation

The attack requires only owning a DAO deposit cell (trivially created by any user via a standard deposit transaction) and submitting a crafted deposit→prepare transaction via the standard `send_transaction` RPC. No privileged access, no majority hashpower, and no social engineering is required. The attack is deterministic and repeatable by any unprivileged transaction sender.

## Recommendation

`valid_dao_withdraw_transaction()` must restrict its `true` return to phase 2 (prepare→withdraw) transactions only — i.e., inputs where `deposited_block_number > 0`. `CapacityVerifier` should be given a `CellDataProvider` (as `DaoScriptSizeVerifier` already has) to load cell data and check the deposit block number:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| {
            cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash)
                && self.data_loader.load_cell_data(cell_meta)
                    .map(|data| data.len() == 8 && LittleEndian::read_u64(&data) > 0)
                    .unwrap_or(false)
        })
}
```

This mirrors the existing phase distinction in `DaoCalculator::transaction_maximum_withdraw()` and ensures the `OutputsSumOverflow` bypass is only applied when the DAO script is actually permitted to produce outputs exceeding inputs (phase 2 interest accrual).

## Proof of Concept

The existing integration test `WithdrawDAOWithOverflowCapacity` only tests phase 2 (prepare→withdraw) overflow and does not cover the deposit→prepare case with mixed non-DAO outputs: [7](#0-6) 

**Minimal unit test plan:**

1. Build a `ResolvedTransaction` with:
   - `resolved_inputs[0]`: DAO-typed cell, `cell_data = [0u8; 8]` (deposit cell, `deposited_block_number == 0`), capacity = 100 CKB
   - `resolved_inputs[1]`: regular cell, capacity = 1 CKB
   - `transaction.outputs[0]`: DAO-typed cell, capacity = 100 CKB
   - `transaction.outputs[1]`: regular cell, capacity = 1,000,000 CKB
2. Construct `CapacityVerifier::new(rtx, dao_type_hash)` and call `verify()`.
3. **Expected (correct) result:** `Err(OutputsSumOverflow { inputs_sum: 101 CKB, outputs_sum: 1_000_100 CKB })`.
4. **Actual (buggy) result:** `Ok(())` — because `valid_dao_withdraw_transaction()` returns `true` for the deposit cell input, skipping the overflow check entirely.
