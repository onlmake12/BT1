### Title
Silent `int128ŌåÆint64` Truncation in TWAP Price Calculation Produces Wrong Price ŌĆö (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

The EVM `calculateTwap` function in `Pyth.sol` silently truncates the computed `int128 twapPrice` to `int64` without any overflow check. An attacker-controlled TWAP update pair with crafted `cumulativePrice` values can cause the returned `twap.price` to be a completely wrong (wrapped) value, while the call succeeds and emits no error. Downstream DeFi protocols that consume `parseTwapPriceFeedUpdates` will receive a silently corrupted price.

---

### Finding Description

In `calculateTwap`, after dividing `priceDiff` (an `int128`) by `slotDiff`, the result is cast directly to `int64` with no bounds check:

```solidity
// target_chains/ethereum/contracts/contracts/pyth/Pyth.sol  line 739
twapPriceFeed.twap.price = int64(twapPrice);   // silent truncation
```

The comment at lines 734ŌĆō738 claims this is safe because "individual prices fit within int64 by protocol design." However, `cumulativePrice` is an `int128` field that accumulates `price ├Ś 1` per slot. Nothing in the on-chain validation path enforces that the *difference* `(endCumulativePrice ŌłÆ startCumulativePrice) / slotDiff` fits in `int64`. The validation in `validateTwapPriceInfo` only checks:

- `prevPublishTime < publishTime` (uniqueness)
- `expo` equality
- `publishSlot` ordering (`start Ōēż end`, not strict `<`)
- `publishTime` ordering

It does **not** verify that the cumulative price values are bounded, nor that the resulting quotient fits in `int64`.

The analogous Solana implementation in `pyth-solana-receiver/src/lib.rs` uses `i64::try_from(...)` with an explicit error return, correctly catching overflow:

```rust
// target_chains/solana/programs/pyth-solana-receiver/src/lib.rs  line 576
let price = i64::try_from(price_diff / i128::from(slot_diff))
    .map_err(|_| ReceiverError::TwapCalculationOverflow)?;
```

The EVM version has no equivalent guard. Solidity `^0.8.0` does **not** revert on signed integer narrowing casts ŌĆö `int64(x)` silently truncates the high bits when `x` is outside `[type(int64).min, type(int64).max]`.

Additionally, `validateTwapPriceInfo` uses `>` (not `>=`) for the slot check:

```solidity
// Pyth.sol line 604
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This allows `startSlot == endSlot`, making `slotDiff = 0`, which causes a division-by-zero panic at line 731 (`priceDiff / int128(uint128(slotDiff))`). This is a separate but related DoS vector.

---

### Impact Explanation

**Primary (silent wrong price):** Any DeFi protocol calling `parseTwapPriceFeedUpdates` and trusting the returned `twap.price` will receive a silently corrupted value. This can be used to manipulate liquidations, collateral valuations, or any price-sensitive logic that relies on the TWAP output. The call succeeds with no revert and no event indicating corruption.

**Secondary (DoS via zero slotDiff):** Submitting a valid TWAP pair where `startSlot == endSlot` causes a division-by-zero panic, reverting the call. This blocks any user from obtaining a TWAP for that feed in that transaction.

---

### Likelihood Explanation

The TWAP feature is externally callable by any address paying the fee. The `cumulativePrice` field is parsed directly from attacker-supplied calldata (via `parseTwapPriceFeedMessage` inside an `unchecked` block) and is only authenticated by the Wormhole Merkle proof ŌĆö meaning the values must come from a valid Pythnet-signed message. However, Pythnet `cumulativePrice` values are `i128` and can legitimately grow large over long windows, making the overflow reachable without any key compromise. The zero-slotDiff DoS requires only that both VAAs reference the same slot, which is a valid on-chain state.

---

### Recommendation

1. **Add an overflow check before the cast**, mirroring the Solana implementation:
   ```solidity
   require(twapPrice >= type(int64).min && twapPrice <= type(int64).max,
       "TWAP price overflow");
   twapPriceFeed.twap.price = int64(twapPrice);
   ```
2. **Change the slot validation to strict inequality** (`>=` instead of `>`):
   ```solidity
   if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
       revert PythErrors.InvalidTwapUpdateDataSet();
   }
   ```
