### Title
Excess `msg.value` Permanently Credited to Pyth Fee Pool When Provider Fee Decreases Between Query and Execution - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `requestHelper()` function does not refund excess `msg.value` to the caller. When a provider reduces their fee via `setProviderFee()` between the moment a user queries `getFeeV2()` and the moment their transaction executes, the user overpays and the excess ETH is permanently credited to `_state.accruedPythFeesInWei` rather than returned to the user.

---

### Finding Description

The `requestHelper()` function in `Entropy.sol` enforces a minimum fee check but does not enforce an exact-match or refund-excess check:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value - providerFee` is credited to `_state.accruedPythFeesInWei`. If `msg.value` exceeds `requiredFee` (e.g., because the provider lowered their fee after the user's off-chain `getFeeV2()` call), the surplus `msg.value - requiredFee` is silently absorbed into Pyth's fee pool instead of being returned to the caller.

The provider fee is mutable at any time by the provider via `setProviderFee()`:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
``` [2](#0-1) 

`getFeeV2(provider, gasLimit)` returns `getProviderFee(provider, gasLimit) + _state.pythFeeInWei`, so any reduction in `provider.feeInWei` directly reduces `requiredFee` at execution time while the user's `msg.value` was computed against the pre-reduction fee. [3](#0-2) 

The `IEntropyV2` interface documents this behavior as a warning ("excess value is *not* refunded to the caller"), but this does not eliminate the user fund loss — it merely shifts responsibility to the caller without providing a safe mechanism to avoid the race condition. [4](#0-3) 

---

### Impact Explanation

A user who calls `getFeeV2()` off-chain and sends the returned amount as `msg.value` can permanently lose ETH if the provider fee decreases before the transaction is mined. The excess is credited to `_state.accruedPythFeesInWei` and is only recoverable by Pyth governance — not by the user. For high-fee providers or large gas-limit requests, the overpayment can be significant.

---

### Likelihood Explanation

Any registered provider can call `setProviderFee()` at any time without restriction. A provider can front-run a pending user transaction (visible in the mempool) by submitting a `setProviderFee(0)` transaction with a higher gas price, causing the user to overpay the maximum possible amount. Even without malicious intent, fee changes during normal operations create a natural race condition for every user request. The `requestV2()` family of functions is the primary entry point for all Entropy users, making this path widely reachable. [5](#0-4) 

---

### Recommendation

After computing `requiredFee`, refund any excess `msg.value` to the caller before crediting balances:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee);
```

---

### Proof of Concept

1. Provider registers with `feeInWei = 1 ether`. Pyth fee is `0.01 ether`. `getFeeV2(provider, 0)` returns `1.01 ether`.
2. User calls `getFeeV2(provider, 0)` off-chain → `1.01 ether`. User submits `requestV2(provider, userRandom, 0)` with `msg.value = 1.01 ether`.
3. Before the user's transaction is mined, the provider front-runs with `setProviderFee(0)`. Now `getFeeV2(provider, 0) = 0.01 ether`.
4. User's transaction executes: `requiredFee = 0.01 ether`, check passes. `providerFee = 0`. `accruedPythFeesInWei += 1.01 ether` (instead of `0.01 ether`).
5. User loses `1 ether` to Pyth's fee pool with no recovery path available to them. [6](#0-5)
