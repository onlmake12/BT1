Audit Report

## Title
Unauthenticated Inbound Peer Can Poison `observed_addrs` and Cause Local Node to Broadcast Arbitrary IP:Port as Its Own Listen Address ‚Äî (`network/src/protocols/identify/mod.rs`)

## Summary
`process_observed` accepts the raw `observed_addr` field from any peer's `IdentifyMessage` and stores it without verifying that the IP matches the session's actual transport-layer source address. The stored address is subsequently included in `local_listen_addrs()` and broadcast to every new connecting peer as one of the local node's own listen addresses, enabling any unprivileged inbound peer to redirect connection attempts from the wider network toward an arbitrary victim IP:port.

## Finding Description
**Step 1 ‚Äî No IP validation in `process_observed`.**
`process_observed` (L152‚Äì169) receives the raw `observed` multiaddr from the wire and immediately calls `self.callback.add_observed_addr(observed, info.session.id)`. There is no comparison between the IP component of `observed` and `context.session.address` (the actual remote address of the session).

**Step 2 ‚Äî `add_observed_addr` appends local peer_id and stores unconditionally.**
`IdentifyCallback::add_observed_addr` (L497‚Äì507) only checks whether a `/p2p/` component is present. If absent, it appends the **local node's own peer_id**, making the attacker-supplied IP:port appear to be a legitimate self-address. It then calls `NetworkState::add_observed_addr` (L513‚Äì516), which is a blind `insert` into the `observed_addrs` map keyed by `SessionId`.

**Step 3 ‚Äî Poisoned address fills `local_listen_addrs()`.**
`local_listen_addrs` (L458‚Äì470) first collects `public_addrs` via `listen_addrs()`. If fewer than `MAX_RETURN_LISTEN_ADDRS` (10) entries are present ‚Äî the common case for NAT nodes with no configured `public_addresses` ‚Äî it fills the remaining slots from `observed_addrs`. The poisoned address is included here.

**Step 4 ‚Äî Poisoned address passes `is_reachable` and is sent to subsequent peers.**
In `connected()` (L211‚Äì232), `local_listen_addrs()` is filtered by `is_reachable`. A public victim IP passes this filter by definition. The resulting list is encoded into the `IdentifyMessage` sent to every new connecting peer.

**Missing guard:** nowhere in this path is there a check that the IP in `observed_addr` matches the actual remote IP of the session that sent it.

## Impact Explanation
Every peer that subsequently connects to the poisoned node receives the victim's IP:port (decorated with the local node's peer_id) as a purported listen address. Those peers store it in their peer stores and will attempt outbound connections to the victim. The TLS/secio handshake fails (peer_id mismatch), but the TCP connection attempt still reaches the victim. With the attacker maintaining a single persistent inbound connection, the poisoned address remains in `observed_addrs` indefinitely, causing a continuous stream of unsolicited connection attempts to the victim from every peer that connects to the poisoned node. This constitutes a reflected, amplified connection flood achievable at near-zero cost, matching the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" ‚Äî High (10001‚Äì15000 points)**.

## Likelihood Explanation
The attack requires only a standard inbound TCP connection speaking the tentacle/secio handshake ‚Äî no special privileges, no proof-of-work, no key material. The precondition (`public_addrs.len() < MAX_RETURN_LISTEN_ADDRS`) is satisfied by any node that has not explicitly configured 10 or more `public_addresses`, which is the default for most deployments. The `IdentifyMessage` is the first message exchanged on every new session, so the injection window is always open. The attacker can maintain the poison indefinitely by staying connected, or re-inject it by reconnecting. The attack is locally reproducible and requires no Sybil capability.

## Recommendation
In `IdentifyCallback::add_observed_addr`, add a `session_address: &Multiaddr` parameter (already available from `SessionContext::address` at the call site in `process_observed`). Before storing, extract the IP component from both `addr` and `session_address` and reject (return `MisbehaveResult::Disconnect`) any `observed_addr` whose IP does not match the session's transport-layer source IP. This is the standard mitigation used by libp2p's identify protocol.

## Proof of Concept
1. Start a CKB node N with no configured `public_addresses` (so `public_addrs` is empty).
2. Connect as inbound peer A using any TCP client that completes the tentacle/secio handshake.
3. After the handshake, send an `IdentifyMessage` with `observed_addr` set to `/ip4/<victim_ip>/tcp/<victim_port>` (the `/p2p/` component may be omitted; `add_observed_addr` will append N's peer_id automatically).
4. Keep peer A connected.
5. Connect as a second inbound peer B.
6. Capture the `IdentifyMessage` that N sends to B.
7. Assert that `listen_addrs` in that message contains `/ip4/<victim_ip>/tcp/<victim_port>/p2p/<N_peer_id>`.
8. Observe that B (and all subsequent peers) will attempt to connect to `<victim_ip>:<victim_port>`, constituting a reflected connection flood against the victim.
