### Title
Excess ETH Sent to `Entropy::requestHelper()` Is Permanently Absorbed Into Protocol Fees Instead of Being Refunded - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

`Entropy.requestHelper()` checks that `msg.value >= requiredFee` but then credits the **entire** `msg.value` (not just `requiredFee`) into protocol fee balances. Any ETH sent above the required fee is permanently transferred to Pyth's accrued fee treasury rather than being refunded to the caller. This is the direct analog of the Curves M-09 finding.

### Finding Description

In `requestHelper()`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

`requiredFee = providerFee + pythFee`. When `msg.value = requiredFee + excess`:

- `providerFee` → `providerInfo.accruedFeesInWei` (correct)
- `pythFee + excess` → `_state.accruedPythFeesInWei` (excess silently absorbed)
- Caller receives **zero refund**

This affects all public entry points that route through `requestHelper`: `request()`, `requestWithCallback()`, and all `requestV2()` overloads. [2](#0-1) [3](#0-2) [4](#0-3) 

The same pattern exists in `Pyth.updatePriceFeeds()`: it checks `msg.value < requiredFee` but never refunds excess, which stays in the contract balance. [5](#0-4) 

The `Echo.requestPriceUpdatesWithCallback()` has the same root cause: excess above `requiredFee` is stored in `req.fee` and later credited entirely to the provider's accrued fees. [6](#0-5) 

The interface NatDoc explicitly acknowledges this behavior:

> "Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2()` prior to each invocation... Further note that excess value is *not* refunded to the caller." [7](#0-6) 

Documentation of the behavior does not eliminate the financial loss — it confirms the root cause is present and intentional.

The existing test suite explicitly demonstrates overpayment being absorbed:

```solidity
// this call overpays for the random number
requestWithFee(user2, pythFeeInWei + provider2FeeInWei + 10000, provider2, 42, false);
// ...
assertEq(random.getAccruedPythFees(), pythFeeInWei * 5 + 10000); // 10000 excess absorbed
``` [8](#0-7) 

### Impact Explanation

Any user who sends `msg.value > getFeeV2(provider, gasLimit)` — which can happen when:
- The provider raises their fee between the off-chain `getFeeV2()` call and the on-chain transaction landing
- A caller uses a hardcoded or stale fee estimate
- A contract wrapper adds a buffer to avoid `InsufficientFee` reverts

…permanently loses the excess ETH to Pyth's treasury. There is no recovery path for the caller. The loss is proportional to the overpayment and scales with usage volume.

### Likelihood Explanation

Fee changes are a normal operational event: providers call `setFee()` to adjust their rates, and the Pyth protocol fee can be updated via governance. Any user whose transaction is submitted with a fee estimate that becomes stale before inclusion (e.g., during mempool congestion or fee bumping) will silently overpay. Contracts that add a safety buffer (e.g., `value: fee * 110 / 100`) to avoid reverts will consistently lose the buffer on every call.

### Recommendation

Refund the excess after crediting the required fee:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee); // use requiredFee, not msg.value

uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

Apply the same fix to `Pyth.updatePriceFeeds()` and `Echo.requestPriceUpdatesWithCallback()`.

### Proof of Concept

1. Provider fee is `0.001 ETH`, Pyth fee is `0.0001 ETH`. `requiredFee = 0.0011 ETH`.
2. User calls `getFeeV2(provider, gasLimit)` off-chain → gets `0.0011 ETH`.
3. Provider calls `setFee()` raising their fee to `0.002 ETH`. `requiredFee` is now `0.0021 ETH`.
4. User's transaction lands with `msg.value = 0.0011 ETH` → reverts with `InsufficientFee`.
5. User retries with `msg.value = 0.003 ETH` (generous buffer to avoid another revert).
6. Transaction succeeds. `providerFee = 0.002 ETH` credited to provider. `0.001 ETH` credited to `accruedPythFeesInWei` (= `pythFee + 0.0009 ETH` excess). User loses `0.0009 ETH` with no recourse. [9](#0-8)

            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.

            true,
            gasLimit

            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,

        );

            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            uint32(req.gasLimit10k) * TEN_THOUSAND,
            bytes("")
        );
