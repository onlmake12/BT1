Audit Report

## Title
`check_tx_fee` Uses Serialized Size Instead of Weight for Minimum Fee Enforcement, Allowing Cycle-Heavy Transactions to Bypass `min_fee_rate` — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` computes the pool-admission fee floor using only `tx_size` bytes, not `get_transaction_weight(tx_size, cycles)`. Because cycles are unknown before script verification, and no weight-based fee check is performed after `verify_rtx` returns actual cycles, a transaction with small serialized size but maximum cycle consumption passes admission while paying a true weight-based fee rate up to ~59× below the configured `min_fee_rate`. This allows an unprivileged attacker to force full script verification on every receiving node at negligible fee cost, constituting a sustained CPU-exhaustion DoS against the CKB network.

## Finding Description
`check_tx_fee` in `tx-pool/src/util.rs` at lines 42–52 explicitly uses `tx_size as u64` as the weight argument to `min_fee_rate.fee()`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The comment acknowledges this is an approximation. `FeeRate::fee` in `util/types/src/core/fee_rate.rs` at line 34 computes `rate * weight / KW`, so passing `tx_size` instead of `get_transaction_weight(tx_size, cycles)` underestimates the required fee whenever cycles dominate.

The call flow in `_process_tx` (`tx-pool/src/process.rs`) is:
1. **L715–717**: `pre_check` → calls `check_tx_fee` with `tx_size` only (cycles unknown)
2. **L724–734**: `verify_rtx` → produces `verified.cycles`
3. **L751**: `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — no weight-based fee check between steps 2 and 3

`get_transaction_weight` in `util/types/src/core/tx_pool.rs` at lines 298–303 computes `max(tx_size, cycles * 0.000_170_571_4)`. For a 200-byte transaction consuming 70,000,000 cycles, weight ≈ 11,940 — 59.7× the raw size. A transaction paying 201 shannons passes the size-based check (`200 shannons` minimum) but has a true weight-based fee rate of ≈16.8 shannons/KW against a configured floor of 1,000 shannons/KW.

Post-admission code (`TxEntry::fee_rate()` at `entry.rs` L114–118 and `EvictKey` at L234–247) correctly uses `get_transaction_weight`, but this only governs eviction priority — it does not prevent admission.

The same mismatch exists in `calculate_min_replace_fee` at `pool.rs` line 103: `self.config.min_rbf_rate.fee(size as u64)` uses raw size for the RBF extra-fee floor.

## Impact Explanation
This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* The attacker pays O(tx_size) fees but imposes O(cycles) CPU cost on every node that processes the transaction via JSON-RPC or P2P relay. With a 59× weight multiplier at `max_tx_verify_cycles = 70,000,000`, the cost-to-impact ratio is highly asymmetric. Pool eviction (weight-based) will remove these entries, but continuous resubmission sustains the CPU load across the network at minimal fee cost.

## Likelihood Explanation
The entry path is fully unprivileged: any actor can call `send_transaction` via JSON-RPC or relay a transaction over P2P. Crafting a cycle-heavy transaction requires only a RISC-V script with a tight loop — no special privileges or victim cooperation. The `max_tx_verify_cycles` default of 70,000,000 gives a weight multiplier of up to ~59× over size, making the bypass significant and repeatable. The code comment at `util.rs` lines 42–44 explicitly acknowledges the approximation, confirming this is a known limitation with real consequences.

## Recommendation
After `verify_rtx` returns `verified.cycles` in `_process_tx` (after line 734 in `process.rs`), perform a second weight-based fee check before calling `submit_entry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, min_fee.as_u64(), fee.as_u64())), snapshot));
}
```

The existing size-based check in `pre_check` can remain as a cheap early filter. Apply the same fix to `calculate_min_replace_fee` in `pool.rs` line 103, passing the replacement transaction's weight (requiring cycles alongside size) instead of raw size.

## Proof of Concept
**Setup**: CKB node with default config (`min_fee_rate = 1000` shannons/KW, `max_tx_verify_cycles = 70,000,000`).

1. Deploy a lock script containing a RISC-V loop consuming ~70,000,000 cycles.
2. Construct a transaction spending a cell locked by that script, ~200 bytes serialized.
3. Set fee to 201 shannons (just above `1000 × 200 / 1000 = 200` shannons size-based minimum).
4. Submit via `send_transaction` RPC.

**Expected**: Rejected with `LowFeeRate` — weight-based min fee = `1000 × 11,940 / 1000 = 11,940 shannons > 201`.

**Actual**: Accepted. True fee rate ≈ 16.8 shannons/KW, 59× below the configured floor.
