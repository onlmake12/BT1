### Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` — (`sync/src/types/mod.rs`, `sync/src/relayer/transaction_hashes_process.rs`)

### Summary

A remote peer can cause unbounded memory growth in the `unknown_tx_hashes` priority queue by repeatedly announcing the same set of transaction hashes. Because `push_peer()` appends unconditionally and the overflow guard only checks the number of **unique hash keys** (not the size of individual `peers` Vecs), the `peers` Vec inside each `UnknownTxHashPriority` entry grows without limit, leading to OOM on the victim node.

---

### Finding Description

**Entry point** — `TransactionHashesProcess::execute()`: [1](#0-0) 

The handler filters out already-known hashes via `tx_filter`, then calls `state.add_ask_for_txs(self.peer, tx_hashes)`. Hashes for transactions that were never actually delivered are **never added to `tx_filter`**, so the same hashes pass the filter on every repeated announcement.

**`add_ask_for_txs`** — the vulnerable insertion path: [2](#0-1) 

For a hash already present in `unknown_tx_hashes`, the code clones the existing `UnknownTxHashPriority`, calls `push_peer(peer_index)`, and writes it back. `push_peer` is: [3](#0-2) 

No deduplication, no cap.

**The overflow guard is bypassed** — it only fires when the number of **unique keys** in the queue exceeds `MAX_UNKNOWN_TX_HASHES_SIZE` (50 000): [4](#0-3) 

If the attacker sends the same N hashes repeatedly, `unknown_tx_hashes.len()` stays at N. The guard never triggers, and the `peers` Vec for each entry grows by 1 on every repetition.

**Rate limiter** — 30 messages/second per `(PeerIndex, message_type)`: [5](#0-4) 

`MAX_RELAY_TXS_NUM_PER_BATCH` = 32 767 hashes per message: [6](#0-5) 

So the attacker can inject up to 30 × 32 767 ≈ 983 000 `PeerIndex` entries per second into the `peers` Vecs.

**Drain rate is negligible** — `pop_ask_for_txs` runs every 100 ms and removes at most one entry per hash per `RETRY_ASK_TX_TIMEOUT_INCREASE` (30 s): [7](#0-6) [8](#0-7) 

Net growth rate is effectively the full injection rate.

---

### Impact Explanation

Each `PeerIndex` is a `usize` (8 bytes on 64-bit). At ~983 K entries/second across 100 hashes:

- After 60 s → ~470 MB
- After 10 min → ~4.7 GB
- After 1 hour → ~28 GB

The node runs out of memory and is killed by the OS (OOM), causing a complete denial of service. No block production, no transaction relay, no RPC responses.

---

### Likelihood Explanation

- Requires only a single connected peer — no special privileges, no PoW, no keys.
- The attacker announces phantom tx hashes (hashes for transactions that don't exist). The victim node never fetches the actual transactions, so the hashes are never added to `tx_filter` and the attack sustains indefinitely.
- The rate limiter (30 req/s) slows but does not stop the attack; a single peer can exhaust gigabytes of RAM within minutes.

---

### Recommendation

1. **Deduplicate in `push_peer`**: Before appending, check `self.peers.contains(&peer_index)`.
2. **Cap `peers` Vec length**: Enforce a hard maximum (e.g., equal to the number of connected peers or a small constant like 8).
3. **Alternatively, use a `HashSet<PeerIndex>`** instead of `Vec<PeerIndex>` to get O(1) dedup automatically.
4. **Add a per-peer rate limit on repeated announcements of the same hash**: track a per-peer bloom filter or LRU set of recently announced hashes and drop duplicates before calling `add_ask_for_txs`.

---

### Proof of Concept

```
1. Connect to a CKB node as a relay peer.
2. Generate 100 random 32-byte values as fake tx hashes.
3. In a loop (≤30 iterations/second to stay within the rate limit):
   a. Send a RelayTransactionHashes message containing all 100 hashes.
4. After 1000 iterations, lock unknown_tx_hashes (via debug RPC or instrumentation)
   and assert peers.len() == 1000 for each hash.
5. Observe RSS of the ckb process growing proportionally.
6. Continue until OOM kill.
```

The `peers.len()` grows by exactly 1 per repetition per hash because `push_peer` appends unconditionally and the overflow guard never fires (queue length stays at 100 unique keys, well below `MAX_UNKNOWN_TX_HASHES_SIZE` = 50 000).

    pub fn execute(self) -> Status {

        {

        let tx_hashes: Vec<_> = {

                .collect()
        };

        state.add_ask_for_txs(self.peer, tx_hashes)
    }
