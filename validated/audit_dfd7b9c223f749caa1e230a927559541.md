Audit Report

## Title
Unwrap Panic in `process_range_block` Causes CLI Crash on Block Verification Failure — (File: ckb-bin/src/subcommand/replay.rs)

## Summary
`process_range_block` calls `.unwrap()` directly on the `VerifyResult` returned by `blocking_process_block_with_switch`. If any block in the replayed range fails contextual verification, the process panics with exit code 101 and a raw backtrace instead of a structured error message. The `sanity_check` function in the same file already demonstrates the correct error-handling pattern that `process_range_block` does not follow.

## Finding Description
In `ckb-bin/src/subcommand/replay.rs`, `process_range_block` (lines 101–119) iterates over a block range and submits each block with `Switch::NONE` (full verification). The result is immediately unwrapped at line 116:

```rust
chain_controller
    .blocking_process_block_with_switch(Arc::new(block), Switch::NONE)
    .unwrap();   // line 116 — panics on Err
```

`blocking_process_block_with_switch` in `chain/src/chain_controller.rs` (lines 71–77) returns `VerifyResult`, which is `Result<bool, Error>`. Any block whose contextual verification fails returns `Err(...)`, and `.unwrap()` on that value triggers a Rust panic — a process abort with exit code 101 and a raw backtrace, not a clean error message.

The `sanity_check` function (lines 141–154) in the same file already handles this correctly with `if let Err(e) = chain_controller.blocking_process_block_with_switch(...)`. `process_range_block` is called twice from `profile` (lines 73 and 76): once for the warm-up pass (`1..from`) and once for the measured pass (`from..=to`). Neither call site has any error handling around the return value.

## Impact Explanation
This is a local command line crash (0–500 points, Note). When `ckb replay --profile` is run against a database containing a block whose verification fails under `Switch::NONE`, the process terminates with a Rust panic (exit code 101) rather than a structured error message, violating the expected CLI error-handling contract demonstrated elsewhere in the same file.

## Likelihood Explanation
Triggering the panic requires local access to a CKB data directory and running `ckb replay --profile`. The condition is met by any database containing a block that fails full re-verification — for example, a database migrated from a different consensus configuration, partially corrupted storage, or one where blocks were originally accepted under a looser `Switch`. No network access, cryptographic material, or elevated privileges are required beyond normal filesystem ownership of the CKB data directory.

## Recommendation
Replace the `.unwrap()` in `process_range_block` with explicit error handling mirroring `sanity_check`. Change the function signature to return `Result<usize, ckb_error::Error>` and propagate errors with `?`, or handle them inline with `if let Err(e)`. The callers in `profile` should print the error and return early, consistent with the rest of the CLI subcommand error-handling style.

## Proof of Concept
1. Obtain a valid CKB data directory (mainnet or testnet snapshot).
2. Use a low-level RocksDB tool (e.g., `ldb`) to corrupt one transaction in a stored block so that its script verification fails under `Switch::NONE`.
3. Run: `ckb replay --profile --tmp-target /tmp/replay_tmp --config <path>`
4. **Expected (correct):** clean error message printed to stderr, exit code 1.
5. **Actual:** `thread 'main' panicked at 'called \`Result::unwrap()\` on an \`Err\` value: ...'`, exit code 101.

The panic is deterministic: every block in the range is processed with `Switch::NONE`, so the first block whose verification returns `Err` immediately triggers the unwrap at line 116.
