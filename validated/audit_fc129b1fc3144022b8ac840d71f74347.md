### Title
Gameable Phase-2 Eviction in `check_purge` Allows Attacker to Eclipse Peer Store by Concentrating Addresses in Small Groups — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

The `check_purge` phase-2 eviction algorithm in `PeerStore` is gameable by an unprivileged remote peer. By distributing attacker-controlled addresses across many network groups of ≤4 peers each (per /16 subnet), while honest peers are concentrated in a few large groups, the attacker ensures that every `check_purge` invocation evicts only honest peers and never attacker peers. Over repeated `add_addr` calls, the attacker can fully eclipse the node's peer store.

---

### Finding Description

`check_purge` is called inside `add_addr` before inserting a new address, whenever the store is at `ADDR_COUNT_LIMIT` (16384). [1](#0-0) 

**Phase 1** evicts addresses that are not connectable (`is_connectable` returns false). A freshly added address has `last_connected_at_ms = 0`, `attempts_count = 0`, `last_tried_at_ms = 0`, which makes `is_connectable` return `true`. [2](#0-1) 

So phase 1 produces an empty candidate set for fresh attacker addresses, and phase 2 is entered.

**Phase 2** groups all addresses by network group (first 2 octets of IPv4, i.e., /16 subnet): [3](#0-2) 

It then sorts groups by size descending, takes the **top half** (`take(len / 2)`), and from those, evicts 2 random peers **only if the group has more than 4 peers**: [4](#0-3) 

The two flaws that combine to create the vulnerability:
1. `take(len / 2)` selects the largest groups — which are the honest peer groups if the attacker keeps their groups small.
2. `if addrs.len() > 4` skips eviction for small groups — attacker groups with ≤4 peers are **never evicted**, even when they appear in the top half.

---

### Impact Explanation

An attacker who can call `add_addr` (via the P2P discovery protocol) with addresses spread across many /16 subnets (≤4 per subnet) can ensure:
- Their addresses are never selected for eviction (group size ≤ 4 → `None` branch).
- Honest peers concentrated in large groups are always selected for eviction (group size > 4 → evict 2).

Each `check_purge` invocation removes 2 honest peers and adds 1 attacker address. Over ~8000 iterations, the attacker can fully replace the peer store with attacker-controlled addresses, enabling a complete eclipse attack: the victim node only connects to attacker-controlled peers, losing access to the honest network for block/transaction relay.

The entrypoint is the unauthenticated P2P discovery protocol: [5](#0-4) 

which calls `add_addr` with remote-supplied addresses: [6](#0-5) 

---

### Likelihood Explanation

The attacker needs addresses in many distinct /16 subnets (not /24 as the question states — the grouping is by first 2 octets). This is achievable via cloud providers, VPS services, or botnets. No PoW, no privileged access, and no key material is required. The attack is purely additive: the attacker only needs to send `GetAddr`/`Nodes` discovery messages with crafted address lists.

---

### Recommendation

1. **Remove the `> 4` guard** or apply it symmetrically: if a group is too small to evict from, it should not be immune — consider evicting 1 peer from any group in the top half regardless of size.
2. **Evict from the bottom half** (smallest groups) instead of the top half, or use a random group selection weighted inversely by size.
3. **Cap per-group contributions** at insertion time (e.g., reject `add_addr` if the group already has ≥ N addresses), similar to Bitcoin Core's `addrman` bucketing.
4. **Rate-limit `add_addr`** per peer session to reduce the attacker's throughput.

---

### Proof of Concept

Construct a store with 1 honest group of 16380 peers (all in `192.168.x.x`) and 4 attacker groups of 1 peer each (in `1.0.x.x`, `2.0.x.x`, `3.0.x.x`, `4.0.x.x`). Total = 16384 = `ADDR_COUNT_LIMIT`. [7](#0-6) 

When `check_purge` runs:
- `len = 5` groups
- `take(5 / 2) = take(2)` → top 2 groups: `[192.168 (16380), 1.0 (1)]`
- `192.168` group: 16380 > 4 → evict 2 honest peers
- `1.0` group: 1 ≤ 4 → `None`, skip
- Groups `2.0`, `3.0`, `4.0`: not taken at all

Result: 2 honest peers evicted, 0 attacker peers evicted. The attacker's new address is then inserted. Repeating this process drains the honest peer group while attacker addresses accumulate. [8](#0-7)
