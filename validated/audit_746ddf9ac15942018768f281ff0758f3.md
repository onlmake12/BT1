### Title
Unbounded Full-Pool Iteration in `get_raw_tx_pool` RPC Blocks Tx-Pool Service — (`File: rpc/src/module/pool.rs`)

### Summary
The `get_raw_tx_pool` RPC endpoint iterates over every transaction in the tx-pool without any pagination or limit. When called with `verbose=true`, it serializes the full entry metadata for every pending, gap, and proposed transaction. Because the tx-pool service processes this request while holding the pool read lock, a large pool causes the service to stall for an extended period, delaying all concurrent tx-pool operations including transaction submission, block assembly, and reorg processing.

### Finding Description
The `get_raw_tx_pool` RPC handler in `rpc/src/module/pool.rs` dispatches to either `get_all_entry_info()` (verbose) or `get_all_ids()` (non-verbose), both of which unconditionally iterate over the entire pool: [1](#0-0) 

`get_all_entry_info()` in `tx-pool/src/pool.rs` collects every pending, gap, proposed, and conflicted entry into a single allocation with no bound on count: [2](#0-1) 

`get_ids()` does the same for the non-verbose path: [3](#0-2) 

Both paths call `score_sorted_iter_by_statuses` and `sorted_proposed_iter`, which traverse the entire multi-index map: [4](#0-3) 

The tx-pool service message loop processes the `GetAllEntryInfo` message by acquiring the async read lock and running the full iteration synchronously before releasing it: [5](#0-4) 

No `limit`, `offset`, or cursor parameter exists on `get_raw_tx_pool`. The RPC signature accepts only `verbose: Option<bool>`: [6](#0-5) 

A secondary instance of the same pattern exists in `get_pool_tx_detail_info`: `get_tx_detail` calls `get_ids()` (full O(N) scan) and then performs a second linear scan of all pending IDs to compute `rank_in_pending`, plus unbounded `calc_descendants` and `calc_ancestors` traversals: [7](#0-6) 

### Impact Explanation
The tx-pool is bounded by `max_tx_pool_size` (bytes), not by transaction count. With the minimum viable transaction size (~100–200 bytes), the pool can hold hundreds of thousands of entries. Iterating and JSON-serializing all of them in a single synchronous call:

1. **Blocks the tx-pool service**: While `GetAllEntryInfo` is being processed, all other queued messages (transaction submission, block template assembly, reorg updates) are delayed.
2. **Consumes unbounded memory**: The full result set is materialized into a `HashMap` / `Vec` before being returned.
3. **Amplifies with repeated calls**: An RPC caller can issue back-to-back `get_raw_tx_pool(verbose=true)` requests, keeping the service continuously busy.

The `get_max_update_time` helper, called on every `tx_pool_info` RPC, also iterates all entries: [8](#0-7) 

### Likelihood Explanation
The `get_raw_tx_pool` RPC is a standard, documented endpoint reachable by any local or network-exposed RPC caller. No authentication is required beyond network access to the RPC port. Operators commonly expose the RPC port for wallet and tooling integrations. An attacker who can submit transactions (to grow the pool) and call the RPC can trigger this condition. The pool can be grown by any transaction sender paying the minimum fee rate.

### Recommendation
1. Add `limit` and `offset` (or a cursor) parameters to `get_raw_tx_pool` so callers can paginate results.
2. Add a server-side cap on the maximum number of entries returned in a single call.
3. Consider moving the full-pool serialization off the tx-pool service's main message loop (e.g., snapshot the IDs under the lock, then serialize outside it).
4. For `get_pool_tx_detail_info`, avoid calling `get_ids()` (full O(N) scan) just to compute `rank_in_pending`; maintain a rank index or document that this endpoint is for debugging only and should not be called under load.

### Proof of Concept
1. Fill the tx-pool to near capacity by submitting many low-fee transactions via `send_transaction` RPC.
2. Repeatedly call `get_raw_tx_pool(verbose=true)` in a tight loop from any RPC client.
3. Observe that concurrent `send_transaction` and `get_block_template` RPC calls experience significantly increased latency, because the tx-pool service is occupied processing the unbounded iteration for each `get_raw_tx_pool` call.

    fn get_raw_tx_pool(&self, verbose: Option<bool>) -> Result<RawTxPool> {
        let tx_pool = self.shared.tx_pool_controller();

        let raw = if verbose.unwrap_or(false) {
            let info = tx_pool
                .get_all_entry_info()
                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Verbose(info.into())
        } else {
            let ids = tx_pool
                .get_all_ids()
                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Ids(ids.into())
        };
        Ok(raw)
    }

    pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
        let pending = self
            .pool_map
            .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let proposed = self
            .pool_map
            .sorted_proposed_iter()
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let conflicted = self
            .conflicts_cache
            .iter()
            .map(|(_id, tx)| tx.hash())
            .collect();
        TxPoolEntryInfo {
            pending,
            proposed,
            conflicted,
        }
    }

    pub(crate) fn get_tx_detail(&self, id: &ProposalShortId) -> Option<PoolTxDetailInfo> {
        if let Some(entry) = self.pool_map.get_by_id(id) {
            let ids = self.get_ids();
            let rank_in_pending = if entry.status == Status::Proposed {
                0
            } else {
                let tx_hash = entry.inner.transaction().hash();
                ids.pending
                    .iter()
                    .enumerate()
                    .find(|(_, hash)| &tx_hash == *hash)
                    .map(|r| r.0)
                    .unwrap_or_default()
                    + 1
            };
            let res = PoolTxDetailInfo {
                timestamp: entry.inner.timestamp,
                entry_status: entry.status.to_string(),
                pending_count: self.pool_map.pending_size(),
                rank_in_pending,
                proposed_count: ids.proposed.len(),
                descendants_count: self.pool_map.calc_descendants(id).len(),
                ancestors_count: self.pool_map.calc_ancestors(id).len(),
                score_sortkey: entry.inner.as_score_key().into(),
            };
            Some(res)
        } else {
            None
        }
    }
