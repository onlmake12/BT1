### Title
TOCTOU Race in `start_sync_headers` Permanently Inflates `n_sync_started`, Blocking IBD Sync — (`sync/src/synchronizer/mod.rs`)

### Summary

`start_sync_headers` reads peer eligibility state, then atomically increments `n_sync_started`, then writes the peer's sync state — as three separate, non-atomic operations. A peer that disconnects between the atomic increment and the subsequent `peer_state.start_sync()` call causes `n_sync_started` to be permanently inflated by 1 with no corresponding decrement. In IBD mode this permanently prevents the node from selecting any new sync peer, stalling initial block download indefinitely. The test suite itself acknowledges the race with the comment "There may be competition between header sync and eviction, it will case assert panic."

### Finding Description

`start_sync_headers` in `sync/src/synchronizer/mod.rs` executes three logically related but non-atomic steps:

**Step 1 — Read peer eligibility (lock released immediately):**
```rust
let peers: Vec<PeerIndex> = self
    .peers()
    .state
    .iter()
    .filter(|kv_pair| kv_pair.value().can_start_sync(now, ibd))
    .map(|kv_pair| *kv_pair.key())
    .collect();
``` [1](#0-0) 

The DashMap read-reference is dropped after the filter closure returns. Peer X is in `SyncProtocolConnected` state and is collected.

**Step 2 — Atomically increment `n_sync_started` (no peer-state lock held):**
```rust
if self.shared().state().n_sync_started()
    .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
        if ibd && x != 0 { None } else { Some(x + 1) }
    })
    .is_err()
{ break; }
``` [2](#0-1) 

`n_sync_started` is now 1. No peer-state lock is held at this point.

**Step 3 — Write peer state (separate lock acquisition):**
```rust
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
}
``` [3](#0-2) 

**The race window** exists between Steps 2 and 3. If peer X disconnects during this window, `Peers::disconnected` runs:

```rust
pub fn disconnected(&self, peer: PeerIndex) {
    if let Some(peer_state) = self.state.remove(&peer).map(|(_, peer_state)| peer_state) {
        if peer_state.sync_started() {  // returns false — state is still SyncProtocolConnected
            assert_ne!(
                self.n_sync_started.fetch_sub(1, Ordering::AcqRel), 0, ...
            );
        }
    }
}
``` [4](#0-3) 

Because `sync_started()` checks `headers_sync_state == Started` and the state was never advanced to `Started` (Step 3 never ran), `n_sync_started` is **not decremented**. Back in Step 3, `get_mut(&peer)` returns `None` (peer is gone), so `start_sync` is never called. `n_sync_started` is now permanently 1 with no peer actually syncing.

### Impact Explanation

In IBD mode, `fetch_update` returns `None` (error) when `n_sync_started != 0`, causing `start_sync_headers` to break immediately without selecting any peer:

```rust
if ibd && x != 0 { None } else { Some(x + 1) }
``` [8](#0-7) 

With `n_sync_started` stuck at 1, no new sync peer is ever selected in IBD mode. The node cannot download headers, cannot advance its chain, and remains permanently stalled at its current tip — effectively a **complete denial of IBD sync**. The node must be restarted to recover. Since no peer is actually syncing, no `tip_synced` or `suspend_sync` call will ever decrement the counter.

A secondary impact: if the race produces a counter value of 0 when `suspend_sync` or `tip_synced` tries to decrement, the `assert_ne!(..., 0)` guard panics and crashes the node process: [9](#0-8) 

### Likelihood Explanation

The race window (between `fetch_update` and `get_mut`) is narrow but realistic under normal network conditions. Any peer — including a malicious one — can trigger it by connecting and immediately disconnecting. The `SEND_GET_HEADERS_TOKEN` notify fires periodically, and peer disconnect callbacks fire asynchronously from the network layer, making concurrent execution routine. No privileged access is required; any unprivileged P2P peer can connect and disconnect at will. The attack is repeatable and requires no special knowledge of the node's internal state.

### Recommendation

The increment of `n_sync_started` and the write to `peer_state` (via `start_sync`) must be performed atomically under the same DashMap entry lock. One approach: acquire the DashMap write entry for the peer first, verify the peer is still eligible, then increment `n_sync_started` and call `start_sync` while holding the entry lock — mirroring the pattern used in `Peers::disconnected` which holds the entry while checking `sync_started()`. Alternatively, use a compare-and-swap on the peer's `headers_sync_state` field before incrementing `n_sync_started`, and roll back the increment if the peer entry is absent in Step 3.

### Proof of Concept

1. Node A starts IBD (no blocks synced yet, `is_initial_block_download()` = true).
2. Malicious peer B connects → `Peers::relay_connected` sets state to `SyncProtocolConnected`.
3. `SEND_GET_HEADERS_TOKEN` fires → `start_sync_headers` collects peer B as eligible.
4. `start_sync_headers` calls `fetch_update` → `n_sync_started` = 1.
5. Peer B immediately disconnects → `Peers::disconnected` removes B, checks `sync_started()` = false, does **not** decrement `n_sync_started`.
6. `start_sync_headers` calls `self.peers().state.get_mut(&B)` → `None`, `start_sync` never called.
7. `n_sync_started` = 1 permanently.
8. All subsequent `SEND_GET_HEADERS_TOKEN` fires: `fetch_update` returns `Err` (IBD + `n_sync_started != 0`) → `break` → no peer selected.
9. Node A is permanently stuck in IBD, unable to sync headers from any peer, until restarted.
