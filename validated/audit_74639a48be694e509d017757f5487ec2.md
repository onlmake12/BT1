### Title
`Echo.executeCallback()` Lacks `gasleft()` Guard, Enabling Silent Callback OOG with Permanent Request Loss — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` performs substantial state-mutating work (fee crediting, request clearing, `firstUnfulfilledSeq` scan) **before** forwarding exactly `req.callbackGasLimit` gas to the consumer callback. There is no `gasleft()` check to verify the calling context holds enough gas to actually deliver `callbackGasLimit` to the inner call. A provider (or any caller after the exclusivity window) can submit `executeCallback` with just enough gas to complete the pre-callback work but not enough to forward the promised gas to `_echoCallback`. The callback silently fails, the request is permanently cleared, and the provider is already credited — the consumer's callback is irreversibly lost.

---

### Finding Description

In `Echo.executeCallback()`, the execution order is:

1. **Line 161–162**: Provider fees are credited (`accruedFeesInWei += ...`).
2. **Line 164**: `clearRequest(sequenceNumber)` — request is permanently deleted from storage.
3. **Lines 169–174**: A `while` loop scans forward to update `_state.firstUnfulfilledSeq`, consuming unbounded gas proportional to the number of already-fulfilled requests.
4. **Lines 176–179**: The callback is attempted with `{gas: req.callbackGasLimit}`. [1](#0-0) 

Due to EIP-150's 63/64 rule, the actual gas forwarded to the inner call is `min(req.callbackGasLimit, (63/64) * gasleft())`. If `gasleft()` at the point of the `try` is less than `req.callbackGasLimit * 64/63`, the callback receives less gas than promised. There is **no `gasleft()` guard** anywhere in `executeCallback` to prevent this.

The `IEcho.sol` NatSpec acknowledges this overhead ("Requires 1.5x the callback gas limit") but provides no on-chain enforcement: [2](#0-1) 

By contrast, `Entropy.sol` explicitly checks `(startingGas * 31) / 32 > uint256(req.gasLimit10k) * TEN_THOUSAND` and reverts with `InsufficientGas()` if the calling context was under-gassed, preserving the request for retry: [3](#0-2) 

`Echo.sol` has no equivalent protection.

---

### Impact Explanation

When `executeCallback` is called with insufficient total gas:

- The callback silently fails and is caught by the `catch` block, emitting `PriceUpdateCallbackFailed`.
- The request has already been cleared (`clearRequest` at line 164) — **no retry is possible**.
- The provider has already been credited the fee — **no refund to the consumer**.
- The consumer's `_echoCallback` never executes despite having paid for it.

This constitutes a **permanent, irreversible loss of callback execution** for the consumer, with no recovery path. [4](#0-3) 

---

### Likelihood Explanation

**High.** Any registered provider (or any address after the exclusivity period expires) can call `executeCallback`. A provider may accidentally under-gas the transaction due to the undocumented overhead of `parsePriceFeedUpdates`, the `firstUnfulfilledSeq` scan loop, and storage writes — all of which consume gas before the callback. A malicious provider can also do this deliberately to collect fees while denying the consumer their callback. The existing test `testExecuteCallbackWithInsufficientGas` confirms the revert path exists but does not test the silent-failure path where the outer transaction succeeds but the callback OOGs. [5](#0-4) 

---

### Recommendation

Add a `gasleft()` guard at the start of the callback dispatch, mirroring the pattern in `Entropy.sol`:

```solidity
// Before the try block at line 176:
if (gasleft() < uint256(req.callbackGasLimit) * 64 / 63) {
    revert InsufficientGas();
}
```

This ensures the transaction reverts (preserving the request and allowing retry) rather than silently clearing the request with an under-gassed callback. Alternatively, move `clearRequest` and fee crediting to **after** a successful callback, or implement a retry/recovery mechanism analogous to `Entropy.sol`'s `CALLBACK_FAILED` state.

---

### Proof of Concept

```solidity
function test_echoCallbackOOG() public {
    // Consumer requests a callback with 500_000 gas limit
    uint32 callbackGasLimit = 500_000;
    bytes32[] memory priceIds = createPriceIds();
    uint96 fee = echo.getFee(defaultProvider, callbackGasLimit, priceIds);
    vm.deal(address(consumer), fee);
    vm.prank(address(consumer));
    uint64 seq = echo.requestPriceUpdatesWithCallback{value: fee}(
        defaultProvider, uint64(block.timestamp), priceIds, callbackGasLimit
    );

    // Provider calls executeCallback with just enough gas for pre-callback work
    // but not enough to forward callbackGasLimit to _echoCallback.
    // The pre-callback work (parsePriceFeedUpdates + storage writes + while loop)
    // consumes ~100k-200k gas, leaving < callbackGasLimit for the inner call.
    PythStructs.PriceFeed[] memory feeds = createMockPriceFeeds(block.timestamp);
    mockParsePriceFeedUpdates(pyth, feeds);
    bytes[] memory updateData = createMockUpdateData(feeds);

    vm.prank(defaultProvider);
    // Submit with total gas = callbackGasLimit + small overhead (not 1.5x as required)
    echo.executeCallback{gas: callbackGasLimit + 80_000}(
        defaultProvider, seq, updateData, priceIds
    );

    // Request is permanently cleared — no retry possible
    vm.expectRevert(NoSuchRequest.selector);
    echo.executeCallback(defaultProvider, seq, updateData, priceIds);

    // Consumer callback was never executed
    assertEq(consumer.lastSequenceNumber(), 0);
}
```

The request is cleared and the provider is paid, but `consumer.lastSequenceNumber()` remains `0`, proving the callback never executed.
