Audit Report

## Title
Tx-Pool Write Lock Held Across CKB-VM `block_in_place` Execution in `readd_detached_tx` During Reorg — (File: tx-pool/src/process.rs)

## Summary
In `update_tx_pool_for_reorg`, the `tx_pool` write lock is acquired at L836 and held for the entire duration of `readd_detached_tx` (L849–850). For each cache-missing detached transaction, `readd_detached_tx` calls `verify_rtx` with `command_rx = None` (L901), which routes to `block_in_place` in `util.rs` (L117–131) for synchronous CKB-VM script execution — all while the write guard remains owned by the async task. This blocks every concurrent caller of `self.tx_pool.write().await` (new tx submission, block template generation, graceful shutdown) for the full duration of script execution per detached transaction.

## Finding Description
`update_tx_pool_for_reorg` (`process.rs` L836–851) acquires the write lock at L836 and does not release it until after `readd_detached_tx` returns:

```rust
let mut tx_pool = self.tx_pool.write().await;   // L836: write lock acquired
_update_tx_pool_for_reorg(...);
self.readd_detached_tx(&mut tx_pool, retain, fetched_cache).await;  // L849: lock held
```

Inside `readd_detached_tx` (L878–914), for each transaction, `verify_rtx` is called with `command_rx = None` at L901:

```rust
if let Ok(verified) = verify_rtx(
    snapshot, Arc::clone(&rtx), tx_env, &verify_cache, max_cycles,
    None,   // L901
).await
```

In `util.rs` L116–131, the `None` branch executes `block_in_place`:

```rust
} else {
    block_in_place(|| {
        ContextualTransactionVerifier::new(...).verify(max_tx_verify_cycles, false)...
    })
}
```

`block_in_place` moves the current OS thread to a blocking pool so the Tokio runtime can schedule other async tasks — but the `tokio::sync::RwLock` write guard is still owned by the current async task. Any other async task calling `self.tx_pool.write().await` queues behind this guard for the entire CKB-VM execution duration.

The `fetched_cache` is pre-populated at L828 before the lock is acquired, but only contains entries for transactions previously processed through `_process_tx` on this node. Transactions mined without passing through the victim node's mempool (e.g., via compact block relay or direct miner RPC submission) will always be cache misses, forcing the full `block_in_place` path. Cache updates are spawned as async tasks (L761–764) that may not have committed before a reorg, and LRU eviction can also clear entries.

By contrast, `_process_tx` (L705–777) correctly separates concerns: `verify_rtx` is called at L724–732 before any write lock is acquired, and `submit_entry` (which acquires the write lock) is called only after verification completes at L753. `readd_detached_tx` does not follow this pattern.

## Impact Explanation
While the write lock is held during `block_in_place` execution, `submit_entry` for new transactions is blocked (tx-pool cannot admit any new transactions), block template generation (`update_full`) is blocked (miner nodes cannot produce new block templates), and `save_pool` (graceful shutdown) is blocked. For N detached transactions each running up to `max_tx_verify_cycles` cycles (default 70,000,000), the total lock hold time is O(N × max script execution time). An attacker can sustain this by repeatedly triggering 1-block reorgs, causing persistent tx-pool unavailability and stalling block template production on targeted nodes. This matches: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points).

## Likelihood Explanation
A 1-block reorg requires no majority hashpower — it occurs naturally when two miners find blocks at the same height. An attacker with any mining capacity can deliberately produce a competing block at the same height. The attacker does not need to control the victim node or have any special privileges. The heavy-script transaction can be submitted directly to the miner's RPC (bypassing the victim's mempool), guaranteeing a cache miss on the victim node. The attack is repeatable: each new competing block at the same height triggers another reorg cycle.

## Recommendation
Move script verification outside the write lock scope, mirroring the pattern already used in `_process_tx`. In `update_tx_pool_for_reorg`, resolve and verify all detached transactions before acquiring the write lock, then acquire the lock only for pool mutation via `_submit_entry`:

```rust
// Phase 1: resolve and verify outside the lock (no lock held)
let mut verified_entries = Vec::new();
for tx in retain {
    if let Ok((rtx, status)) = resolve_tx_readonly(&snapshot, tx) {
        if let Ok(verified) = verify_rtx(snapshot, rtx, tx_env, &verify_cache, max_cycles, None).await {
            verified_entries.push((rtx, status, verified, fee, tx_size));
        }
    }
}

// Phase 2: acquire write lock only for pool mutation
{
    let mut tx_pool = self.tx_pool.write().await;
    _update_tx_pool_for_reorg(&mut tx_pool, ...);
    for (rtx, status, verified, fee, tx_size) in verified_entries {
        let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
        let _ = _submit_entry(&mut tx_pool, status, entry, &self.callbacks);
    }
}
```

Note that `resolve_tx` currently takes `&mut TxPool`, so a read-only resolution path may need to be introduced, or resolution can be deferred to inside the lock while keeping verification outside.

## Proof of Concept
1. Craft `TX_heavy`: a transaction whose lock script loops for `max_tx_verify_cycles` cycles.
2. Submit `TX_heavy` directly to Miner M's RPC (not through the victim node), so the victim's `txs_verify_cache` has no entry for it.
3. Miner M mines `Block_N` containing `TX_heavy`. Victim node receives and stores `Block_N`.
4. Attacker mines `Block_N'` at the same height (any hashpower, or wait for a natural fork).
5. Victim node receives `Block_N'`, triggering a reorg: `update_tx_pool_for_reorg` is called with `retain = [TX_heavy]`, `fetched_cache = {}`.
6. At `process.rs` L836, write lock is acquired. `readd_detached_tx` is called. Cache miss → `verify_rtx(..., None)` → `block_in_place` runs full CKB-VM for `TX_heavy`. Write lock held for entire execution.
7. During step 6, any concurrent call to `self.tx_pool.write().await` (new tx submission, block template update, shutdown) blocks until CKB-VM finishes.
8. Attacker repeats step 4 to sustain the DoS.
