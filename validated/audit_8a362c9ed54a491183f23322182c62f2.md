Audit Report

## Title
Unbounded Memory Allocation via `get_cells` with Default `request_limit=usize::MAX` and `with_data=None` — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

## Summary
The rich-indexer `get_cells` RPC defaults `request_limit` to `usize::MAX`, rendering the only guard against oversized requests inert for any valid `Uint32` input. With `with_data` defaulting to `true`, an unauthenticated caller can issue a single RPC call with `limit = 0xFFFFFFFF` that causes `fetch_all` to materialize all matching cell data into a single heap-allocated `Vec<AnyRow>`, potentially OOM-killing the node process.

## Finding Description

**`request_limit` defaults to `usize::MAX`:**

`RichIndexerService::new` at `util/rich-indexer/src/service.rs` line 51 sets:
```rust
request_limit: config.request_limit.unwrap_or(usize::MAX),
``` [1](#0-0) 

The `resource/ckb.toml` config acknowledges this is dangerous and recommends `request_limit = 400`, but the line is commented out — the unsafe default ships as-is: [2](#0-1) 

**Guard check is ineffective at default:**

The only enforcement in `get_cells` is:
```rust
if limit as usize > self.request_limit {
    return Err(...);
}
``` [3](#0-2) 

`limit` is `Uint32` (max ~4.29 billion); `self.request_limit` defaults to `usize::MAX` (~18.4 quintillion on 64-bit). This check never triggers for any valid `Uint32` input.

**`with_data` defaults to `true`, selecting full blob data:** [4](#0-3) 

**`fetch_all` materializes the entire result set into memory at once:** [5](#0-4) 

`fetch_all` is implemented as `query.fetch_all(pool)` — sqlx's bulk-load API that returns a `Vec<AnyRow>` with all rows loaded before returning: [6](#0-5) 

**No timeout guard exists in the rich-indexer path:**

The RocksDB-backed `IndexerService` wraps its iterator in a `TimeoutIterator` with a 10-second default: [7](#0-6) [8](#0-7) 

`AsyncRichIndexerHandle` has no equivalent timeout mechanism — the SQL query and `fetch_all` run unbounded.

## Impact Explanation

An attacker sends a single unauthenticated RPC call with `limit = 0xFFFFFFFF` and `with_data` omitted (defaults `true`), targeting a script that matches many cells with large `output_data`. The SQL query runs `SELECT ... output.data ... LIMIT 4294967295`. `fetch_all` allocates all N×D bytes of cell data into a single `Vec<AnyRow>` in process memory simultaneously. The config comment notes serde JSON serialization adds another ~10× overhead. With 1,000 cells × 1 MB each, this is ~10 GB of process memory — sufficient to OOM-kill the node process.

This matches: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

- The rich-indexer RPC is publicly accessible to any caller who can reach the RPC port (no authentication, no PoW, no consensus check).
- `request_limit` is `None` by default in `IndexerConfig`; operators must affirmatively opt in to the safe value. The safe value is only mentioned in a comment.
- Mainnet/testnet cells routinely carry large `output_data` (on-chain scripts, NFT metadata, Spore cells).
- A single HTTP POST is sufficient to trigger the condition — no sustained traffic or special privileges required.
- The attack is repeatable and requires no prior state setup beyond the existence of matching cells in the indexed chain.

## Recommendation

1. **Change the default** of `request_limit` from `usize::MAX` to a safe value (e.g., 400, as the comment recommends) in `RichIndexerService::new` at `util/rich-indexer/src/service.rs` line 51.
2. **Add a per-response byte budget**: track cumulative `output_data` size during row iteration and abort if it exceeds a configurable threshold.
3. **Use streaming** (`fetch` with `try_next`) instead of `fetch_all` so memory is bounded to one row at a time, with early termination on budget exhaustion. Note that `SQLXPool::fetch` already exists as a streaming alternative at `util/rich-indexer/src/store.rs` lines 151–162. [9](#0-8) 
4. **Add a timeout guard** analogous to `TimeoutIterator` used in the RocksDB indexer path, applied to the SQL query execution.

## Proof of Concept

1. Start a CKB node with rich-indexer enabled and no `request_limit` configured (the default).
2. Index transactions containing cells under the same lock script, each with large `output_data` (e.g., 1 MB each).
3. Send:
   ```bash
   curl -X POST http://localhost:8114 \
     -H 'Content-Type: application/json' \
     -d '{"jsonrpc":"2.0","method":"get_cells","params":[{"script":{"code_hash":"<hash>","hash_type":"type","args":"0x"},"script_type":"lock"},"asc","0xFFFFFFFF",null],"id":1}'
   ```
   (`with_data` omitted → defaults `true`; `limit = u32::MAX`)
4. Observe node RSS grow proportionally to N×D bytes (plus ~10× serde overhead) and OOM-kill.

    pub async fn fetch<'a, T>(&self, query: Query<'a, Any, T>) -> Result<Vec<AnyRow>>
    where
        T: Send + IntoArguments<'a, Any> + 'a,
    {
        let pool = self.get_pool()?;
        let mut res = vec![];
        let mut rows = query.fetch(pool);
        while let Some(row) = rows.try_next().await? {
            res.push(row)
        }

    }
