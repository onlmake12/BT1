### Title
`getPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp Instead of Per-Feed `publishTime` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` in `Scheduler.sol` validate freshness against a single subscription-level timestamp (`status.priceLastUpdatedAt`), which is the **maximum** `publishTime` across all feeds in the last update batch — not each individual feed's own `publishTime`. An unprivileged keeper can submit a batch containing one recent feed and one arbitrarily stale feed (e.g., a closed-market feed). The update passes all on-chain checks, the stale feed is stored, and a consumer calling `getPricesNoOlderThan(..., 60)` receives the stale feed without any revert, despite its `publishTime` being hours old.

---

### Finding Description

In `_validateShouldUpdatePrices`, the staleness gate is computed as the **maximum** `publishTime` across all feeds in the batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [1](#0-0) 

This max value is then stored as the subscription-level `status.priceLastUpdatedAt`:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` check (1 hour) is applied only to this max value, not to individual feeds:

```solidity
if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(...);
}
``` [3](#0-2) 

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` constant is 1 hour: [4](#0-3) 

When a consumer calls `getPricesNoOlderThan`, the freshness check compares `block.timestamp` against this single subscription-level max, not against each returned feed's own `publishTime`:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [5](#0-4) 

The same flaw exists in `getEmaPricesNoOlderThan`: [6](#0-5) 

The design explicitly accommodates stale individual feeds. The code comment states:

> "Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period." [7](#0-6) 

The existing test `testUpdatePriceFeedsSucceedsWithStaleFeedIfLatestIsValid` directly confirms that a feed with `publishTime` more than 1 hour in the past is accepted and stored when paired with a recent feed: [8](#0-7) 

---

### Impact Explanation

A consumer protocol calling `getPricesNoOlderThan(subscriptionId, priceIds, 60)` expects a guarantee that **every** returned price is at most 60 seconds old. Instead, the guarantee is only that the **most recent feed** in the batch is within 60 seconds. Individual feeds for closed markets can have `publishTime` values hours or days in the past. A lending protocol, derivatives platform, or any financial application relying on this function for per-feed freshness guarantees will silently consume stale prices, potentially leading to incorrect collateral valuations, mispriced positions, or exploitable arbitrage.

---

### Likelihood Explanation

The scenario is not an edge case — it is the explicitly documented and tested behavior of the Scheduler. Any subscription containing a mix of active-market and closed-market feeds (e.g., equities alongside crypto) will exhibit this behavior on every update. Any unprivileged keeper can call `updatePriceFeeds` with such a batch. No privileged access, key compromise, or external oracle manipulation is required.

---

### Recommendation

In `getPricesNoOlderThan` and `getEmaPricesNoOlderThan`, validate staleness against each individual feed's `publishTime` rather than the subscription-level `priceLastUpdatedAt`:

```solidity
function getPricesNoOlderThan(...) external view ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

Alternatively, if the subscription-level semantics are intentional, rename the function (e.g., `getPricesWithBatchNoOlderThan`) and prominently document that the age bound applies to the most recent feed in the batch, not to each individual feed.

---

### Proof of Concept

1. Create a subscription with two price feeds: `feedA` (crypto, active) and `feedB` (equity, closed market).
2. An unprivileged keeper calls `updatePriceFeeds(subscriptionId, updateData)` where:
   - `feedA.publishTime = block.timestamp` (current)
   - `feedB.publishTime = block.timestamp - 7200` (2 hours ago, same Pythnet slot)
3. `_validateShouldUpdatePrices` computes `updateTimestamp = max(block.timestamp, block.timestamp - 7200) = block.timestamp`. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` check passes. `status.priceLastUpdatedAt = block.timestamp`.
4. Both feeds are stored via `_storePriceUpdates`.
5. A whitelisted consumer calls `getPricesNoOlderThan(subscriptionId, [feedA_id, feedB_id], 60)`.
6. Check: `distance(block.timestamp, block.timestamp) = 0 ≤ 60` → **passes**.
7. Both prices are returned. `feedB.publishTime = block.timestamp - 7200` — **2 hours stale** — is returned without any revert.
8. The consumer believes both prices are at most 60 seconds old. The false freshness guarantee leads to decisions based on a 2-hour-old price.
