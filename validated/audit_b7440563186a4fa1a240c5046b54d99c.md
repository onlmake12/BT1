### Title
`setMaxNumHashes` Lacks Upper Bound Validation, Enabling Permanent Reveal OOG or Request DOS - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `setMaxNumHashes` accepts any `uint32` value with no upper bound check, while the analogous `setDefaultGasLimit` setter explicitly validates against `MAX_GAS_LIMIT`. Because `maxNumHashes` directly controls the maximum `numHashes` stored in a request, and `numHashes` drives an unbounded `while` loop in `constructProviderCommitment`, an improperly large value allows requests to be created whose reveal will always OOG, permanently locking user funds. An improperly small value causes all new requests to revert with `LastRevealedTooOld`, DOS-ing the provider's service.

---

### Finding Description

`setMaxNumHashes` at [1](#0-0)  accepts any `uint32` value and stores it directly with no validation:

```solidity
function setMaxNumHashes(uint32 maxNumHashes) external override {
    ...
    provider.maxNumHashes = maxNumHashes;   // no bounds check
    ...
}
```

By contrast, `setDefaultGasLimit` at [2](#0-1)  explicitly calls `roundTo10kGas(gasLimit)` which reverts if `gasLimit > MAX_GAS_LIMIT`:

```solidity
function setDefaultGasLimit(uint32 gasLimit) external override {
    ...
    roundTo10kGas(gasLimit);   // reverts if > MAX_GAS_LIMIT
    ...
}
```

The `maxNumHashes` value is used in `requestHelper` to cap the `numHashes` field stored in each request: [3](#0-2) 

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber - providerInfo.currentCommitmentSequenceNumber
);
if (providerInfo.maxNumHashes != 0 && req.numHashes > providerInfo.maxNumHashes) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

That stored `numHashes` is then consumed by `constructProviderCommitment` inside `revealHelper`, which runs an unbounded `while` loop: [4](#0-3) 

```solidity
function constructProviderCommitment(uint64 numHashes, bytes32 revelation)
    internal pure returns (bytes32 currentHash)
{
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
```

`revealHelper` is called from both `reveal` and `revealWithCallback`. [5](#0-4) 

---

### Impact Explanation

**Too-high `maxNumHashes`:** A provider sets `maxNumHashes` to a large value (e.g., `500_000` or higher). Users make requests that accumulate a large `numHashes` gap. When `revealWithCallback` is called (by anyone), `constructProviderCommitment` iterates `numHashes` times. At ~30 gas per `keccak256`, `numHashes = 500_000` costs ~15M gas, and `numHashes = 1_000_000` costs ~30M gas — exceeding Ethereum's block gas limit. The reveal transaction always reverts OOG. User funds paid as fees are permanently locked in the contract with no recovery path, since the request can never be fulfilled.

**Too-low `maxNumHashes`:** A provider sets `maxNumHashes = 1`. After the very first request, every subsequent request reverts with `LastRevealedTooOld` until the provider advances their commitment. This effectively DOS-es the provider's service for all users.

Neither bound is validated on-chain.

---

### Likelihood Explanation

An Entropy provider is an explicitly in-scope actor ("Entropy user/provider"). A misconfigured or malicious provider can call `setMaxNumHashes` with any value at any time. The `setMaxNumHashes` function is callable by any registered provider with no governance delay or timelock. The Fortuna keeper service does monitor `maxNumHashes` to trigger commitment advances, but this is an off-chain heuristic — it does not prevent a provider from setting an arbitrarily large value that allows OOG-inducing requests to be created before the keeper reacts. [6](#0-5) 

---

### Recommendation

Add an on-chain upper bound constant (e.g., `MAX_NUM_HASHES`) to `Entropy.sol`, analogous to `MAX_GAS_LIMIT`, and enforce it in `setMaxNumHashes`:

```solidity
uint32 public constant MAX_NUM_HASHES = 100_000; // calibrate to safe gas budget

function setMaxNumHashes(uint32 maxNumHashes) external override {
    ...
    if (maxNumHashes > MAX_NUM_HASHES) revert EntropyErrors.MaxNumHashesExceeded();
    provider.maxNumHashes = maxNumHashes;
    ...
}
```

The constant should be derived from the block gas limit of the target chain divided by the per-iteration gas cost of `keccak256`, with a safety margin. A minimum non-zero floor should also be considered to prevent the `maxNumHashes = 1` DOS scenario.

---

### Proof of Concept

1. Provider calls `setMaxNumHashes(800_000)` — accepted with no revert.
2. Users submit 800,000 requests to the provider without the provider calling `advanceProviderCommitment`. Each request is accepted because `numHashes ≤ maxNumHashes`.
3. The 800,000th request stores `req.numHashes = 800_000`.
4. Anyone calls `revealWithCallback(provider, 800_000th_seq, ...)`.
5. `revealHelper` → `constructProviderCommitment(800_000, ...)` → `while` loop runs 800,000 keccak256 iterations → ~24M gas consumed → OOG revert on most EVM chains.
6. The request can never be fulfilled. The user's fee is permanently locked.
