Audit Report

## Title
Minimum Fee Rate Admission Check Uses Serialized Size Instead of True Transaction Weight — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` gates pool admission using only the transaction's serialized byte size to compute the minimum required fee, while the pool's actual ordering, eviction, and fee-rate accounting all use `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. An attacker can craft a compact transaction that executes a high-cycle script, pass the size-only admission gate with a tiny fee, and occupy pool weight far exceeding what the fee would normally permit. The code comment at the root cause site explicitly acknowledges this is a known theoretical gap.

## Finding Description
In `tx-pool/src/util.rs` L42–52, `check_tx_fee` computes the minimum fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

This check runs inside `pre_check` (called from `_process_tx` at L715), before `verify_rtx` executes the scripts and determines actual cycles. After `verify_rtx` returns `verified.cycles`, the code at L751 constructs `TxEntry::new(rtx, verified.cycles, fee, tx_size)` and submits it — with no second fee-rate check against the true weight.

The true weight used everywhere else in the pool is defined in `util/types/src/core/tx_pool.rs` L298–303:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. This weight drives:
- Pool ordering (`AncestorsScoreSortKey` via `entry.rs` L221–231)
- Eviction priority (`EvictKey` via `entry.rs` L234–247)
- Fee rate display (`TxEntry::fee_rate()` at `entry.rs` L114–118)
- Pool size eviction (`limit_size` at `pool.rs` L292–328, which evicts by `EvictKey`)

The exploit path is:
1. Attacker deploys a compact RISC-V loop script (~200–300 bytes serialized).
2. Constructs a transaction referencing this script; `tx_size ≈ 300`, `cycles ≈ 69,000,000`.
3. Sets `fee = min_fee_rate × tx_size / 1000 = 1000 × 300 / 1000 = 300 shannons`.
4. `check_tx_fee` passes: `fee (300) >= min_fee (300)`.
5. After VM execution, `weight = max(300, 69_000_000 × 0.000_170_571_4) ≈ 11,769`.
6. Effective fee rate = `300 × 1000 / 11,769 ≈ 25 shannons/KB` — ~40× below the 1,000 shannons/KB minimum.
7. Transaction is admitted. The `limit_size` eviction loop at `pool.rs` L298 only evicts when `total_tx_size > max_tx_pool_size`, using `total_tx_size` which tracks serialized byte size, not weight — so the attacker's small-byte-size transactions do not trigger size-based eviction easily, but they occupy disproportionate weight-based block capacity.

No post-verification fee-rate check against the true weight exists anywhere in `_process_tx`, `submit_entry`, or `readd_detached_tx`.

## Impact Explanation
This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

An unprivileged attacker can flood the mempool with transactions paying ~1/40th to ~1/60th of the true minimum fee rate. Each such transaction consumes `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)` weight units of effective block capacity while paying only for its byte size. With `max_tx_pool_size = 180 MB` and a 60× fee discount, the attacker can occupy pool capacity that would normally require 60× more fee expenditure. Legitimate transactions paying the true minimum fee rate may be evicted or delayed, degrading mempool quality and miner revenue, and causing effective network congestion at a fraction of the intended cost.

## Likelihood Explanation
The attack requires only:
1. A one-time deployment of a compact loop script on-chain (trivial cost).
2. Repeated `send_transaction` RPC or P2P relay calls — no privileged access.
3. No victim mistakes or external dependencies.

The code comment at the root cause site (`// Theoretically we cannot use size as weight directly`) confirms this gap is known and unmitigated. The attack is fully repeatable and automatable.

## Recommendation
After `verify_rtx` returns `verified.cycles`, perform a second fee-rate check using the true weight before constructing the `TxEntry`:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
// Post-verification: check fee rate against true weight
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

Alternatively, at admission time in `check_tx_fee`, apply a conservative upper-bound using `max_tx_verify_cycles` as a stricter gate to reject transactions whose fee would be insufficient even at maximum cycle consumption.

## Proof of Concept
1. Write a minimal RISC-V loop script that executes ~69,000,000 cycles. Compiled binary is ~150–300 bytes.
2. Deploy the script cell on-chain (one-time cost).
3. Construct a transaction with `tx_size ≈ 300` bytes using this script as a lock.
4. Set `fee = 300 shannons` (exactly `min_fee_rate × tx_size / 1000`).
5. Submit via `send_transaction` RPC.
6. Observe: `check_tx_fee` passes (`300 >= 300`); after VM execution `verified.cycles ≈ 69,000,000`; `weight = max(300, 11,769) = 11,769`; effective fee rate ≈ 25 shannons/KB (40× below minimum).
7. Transaction is admitted to the pool with no further fee-rate rejection.
8. Repeat with many UTXOs to fill the pool at ~1/40th the intended cost, displacing legitimate transactions.
