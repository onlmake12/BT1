Audit Report

## Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Returns Incorrect Withdrawal Capacity — (File: `util/dao/src/lib.rs`)

## Summary
`DaoCalculator::calculate_maximum_withdraw` computes a `u128` intermediate value `withdraw_counted_capacity` and narrows it to `u64` with a bare `as u64` cast at line 156. When the intermediate result exceeds `u64::MAX`, the upper 64 bits are silently discarded and the function returns a drastically incorrect (too-small) capacity with no error. Every other `u128→u64` narrowing in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern, making this omission an inconsistency and a latent defect.

## Finding Description
In `util/dao/src/lib.rs` lines 152–156:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits if `withdraw_counted_capacity > u64::MAX`. No error is propagated; the function proceeds with a wrong value and returns `Ok(wrong_capacity)`.

All three other `u128→u64` narrowings in the same file use the safe pattern:
- Line 204: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- Lines 244–245: `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?`
- Line 258: `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?` [2](#0-1) [3](#0-2) [4](#0-3) 

The existing overflow test `check_withdraw_calculation_overflows` (lines 296–350) uses `withdrawing_ar/deposit_ar ≈ 1 + 10^-10`. With that ratio, `withdraw_counted_capacity` exceeds `u64::MAX` by only a tiny margin, and the resulting truncated value plus `occupied_capacity` overflows `u64` in `safe_add`, which returns `Err`. The test passes — but it does so because `safe_add` catches the overflow, **not** because the cast is safe. The silent-truncation path — where the cast wraps to a small value that `safe_add` then accepts — is not exercised. [5](#0-4) 

## Impact Explanation
`calculate_maximum_withdraw` feeds three downstream paths:

1. **`withdrawed_interests` → `dao_field_with_current_epoch`**: `current_s` is computed as `parent_s + nervosdao_issuance - withdrawed_interests`. A silently truncated (too-small) `maximum_withdraw` makes `withdrawed_interests` too small, inflating `current_s`. This value is embedded in every block header and is consensus-critical. Any future node with the corrected cast would compute a different `current_s` and diverge — **consensus deviation**.

2. **`transaction_fee`**: Fee = `maximum_withdraw - outputs_capacity`. A silently truncated `maximum_withdraw` smaller than `outputs_capacity` causes `safe_sub` to return `DaoError`, causing valid DAO withdrawal transactions to be incorrectly rejected — **economic damage / network disruption**.

3. **RPC `calculate_dao_maximum_withdraw`**: Users receive a silently wrong withdrawal amount and construct transactions with incorrect output capacities — **economic damage**.

This maps to: **Critical — Vulnerabilities which could easily cause consensus deviation** and **Critical — Vulnerabilities which could easily damage CKB economy**.

## Likelihood Explanation
The truncation requires `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. Since `ar` starts at `10^16` and grows monotonically, the ratio `withdrawing_ar/deposit_ar` must exceed approximately 5× for a cell holding a near-maximum CKB balance. On mainnet this is a long-horizon condition (decades at current issuance rates). However, the condition is immediately and trivially reachable in devnet/testnet by setting `ar` values directly in the DAO field of block headers, exactly as the existing test infrastructure already does. The defect is demonstrably present and inconsistent with the defensive pattern used everywhere else in the file. There is no defense-in-depth: no checked cast, no assertion, no comment acknowledging the assumption.

## Recommendation
Replace the bare `as u64` cast with the checked conversion already used everywhere else in the file:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

Add a test case where `withdraw_counted_capacity` itself (before `safe_add`) exceeds `u64::MAX` and verify that `DaoError::Overflow` is returned rather than a silently wrong value.

## Proof of Concept
Using the existing test scaffolding in `util/dao/src/tests.rs`:

```
deposit_ar     = 10_000_000_000_000_000   (10^16, genesis value)
withdrawing_ar = 55_000_000_000_000_000   (5.5× — directly settable via pack_dao_data in tests)
counted_capacity = 18_000_000_000_000_000_000  (near u64::MAX)

withdraw_counted_capacity (u128) = 18_000_000_000_000_000_000 × 55_000_000_000_000_000
                                   / 10_000_000_000_000_000
                                 = 99_000_000_000_000_000_000

u64::MAX = 18_446_744_073_709_551_615

99_000_000_000_000_000_000 > u64::MAX  →  truncation occurs

withdraw_counted_capacity as u64 = 99_000_000_000_000_000_000 − 5 × 2^64
                                 ≈ 6_766_279_631_452_241_920

safe_add(occupied_capacity) succeeds → returns Ok(~6.77×10^18 shannons)
Correct answer: ~9.9×10^19 shannons
Silent error: >10× wrong, no Err returned
```

A minimal unit test: set `deposit_ar = 10_000_000_000_000_000`, `withdrawing_ar = 55_000_000_000_000_000`, `output capacity = 18_000_000_000_000_000_000` shannons, call `calculate_maximum_withdraw`, and assert `result.is_err()`. With the current code the assertion fails (returns `Ok` with a wrong value); with the fix it passes. [6](#0-5)

fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
}
