Audit Report

## Title
Unguarded `set_unverified_tip` in `send_unverified_block` allows trivial IBD sync halt via crafted block — (`chain/src/orphan_broker.rs`)

## Summary
In `send_unverified_block`, `unverified_tip` is updated whenever `block_number > verified_tip`, with no comparison against the current `unverified_tip`. An attacker can send a single P2P `SendBlock` message containing a structurally valid but contextually invalid block with an arbitrarily high `block_number`, inflating `unverified_tip` past `tip + BLOCK_DOWNLOAD_WINDOW * 9` and causing `block_fetcher.fetch()` to return `None` for all peers. Because the contextual verifier only resets `unverified_tip` after processing the fake block, the attacker can continuously re-send crafted blocks to keep any IBD node permanently unable to download blocks.

## Finding Description

**Root cause — wrong comparand in `send_unverified_block` (`orphan_broker.rs:180`):**

```rust
if block_number > self.shared.snapshot().tip_number() {
    self.shared.set_unverified_tip(ckb_shared::HeaderIndex::new(
        block_number,
        block_hash.clone(),
        U256::from(0u64),
    ));
```

The guard compares `block_number` against the *verified* tip (`snapshot().tip_number()`), not the current `unverified_tip`. Any block with `block_number > verified_tip` unconditionally overwrites `unverified_tip`, regardless of how far ahead it is.

**`set_unverified_tip` has no monotonicity enforcement (`shared/src/shared.rs:407-409`):**

```rust
pub fn set_unverified_tip(&self, header: crate::HeaderIndex) {
    self.unverified_tip.store(Arc::new(header));
}
```

A bare `ArcSwap::store` with no CAS loop or bounds check.

**`block_fetcher.fetch()` is gated on `unverified_tip` (`block_fetcher.rs:111-129`):**

```rust
if self.sync_shared.shared().get_unverified_tip().number() >= unverified_tip_limit {
    return None;
}
```

Where `unverified_tip_limit = tip + BLOCK_DOWNLOAD_WINDOW * 9 = tip + 73728`. Once `unverified_tip` is set to `tip + 73729`, every call to `fetch()` returns `None` for every peer.

**Non-contextual verification does not check PoW or block-number consistency (`block_verifier.rs:39-47`):**

`BlockVerifier` only runs `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier`. There is no PoW check and no `NumberVerifier` (which requires the parent's number from the DB and is contextual). The attacker-controlled `block_number` passes unchecked.

**`CellbaseVerifier` checks that the cellbase `since` field equals `block_number`, but the attacker controls both fields.** The block header's `number` field and the cellbase input's `since` field are both attacker-supplied, so setting both to `tip + 73729` satisfies the check.

**Attack path:**
1. Attacker connects via P2P and sends a `SendBlock` message.
2. Block is crafted: `parent_hash = current_tip_hash`, `block_number = tip + 73729`, cellbase `since = tip + 73729`, valid merkle root.
3. `non_contextual_verify` passes — no PoW check, no number check.
4. `insert_block` stores the block in the DB.
5. `process_lonely_block` routes to `process_descendant` because `parent_hash` matches the stored tip (`BLOCK_STORED`).
6. `send_unverified_block` calls `set_unverified_tip(tip + 73729)`.
7. Every subsequent `block_fetcher.fetch()` call hits the `>= unverified_tip_limit` guard and returns `None`.
8. The contextual verifier (`ConsumeUnverifiedBlockProcessor`) eventually processes the fake block, detects the invalid parent number, and resets `unverified_tip` back to the verified tip.
9. The attacker immediately re-sends another crafted block (different hash, e.g., by varying a nonce or timestamp), re-inflating `unverified_tip`. The cycle repeats indefinitely.

**Why existing checks fail:**
- The `block_number < 1` guard in `asynchronous_process_block` only rejects genesis-level blocks.
- `CellbaseVerifier` is satisfied because the attacker controls both `since` and `block_number`.
- `NumberVerifier` (checks `block_number == parent_number + 1`) is contextual and not invoked in the non-contextual path.
- The contextual verifier does reset `unverified_tip` on failure, but only bounds the disruption window per message — it does not prevent continuous repetition.
- No peer-banning mechanism prevents the attacker from re-sending a new crafted block (different hash) after each reset cycle.

## Impact Explanation

During IBD, `block_fetcher.fetch()` is the sole mechanism by which the node requests blocks from peers. With `unverified_tip` inflated past `tip + BLOCK_DOWNLOAD_WINDOW * 9`, `fetch()` returns `None` for every peer on every poll cycle. The verified tip cannot advance. By continuously re-sending crafted blocks (each costing only a P2P message with a trivially constructed payload — no PoW mining required), an attacker can keep any target node permanently unable to complete IBD.

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. Applied at scale against multiple nodes simultaneously, it can prevent the broader network from syncing new nodes.

## Likelihood Explanation

Trivially exploitable: no hashpower, no privileged access, and no victim interaction required. The attacker needs only a standard P2P connection and the ability to craft a structurally valid block (correct merkle root, valid cellbase structure with matching `since` field). The attack is repeatable at negligible cost after each contextual-verifier reset cycle. Any node in IBD is a valid target.

## Recommendation

In `send_unverified_block` (`orphan_broker.rs:180`), replace the guard:

```rust
// Before (wrong comparand):
if block_number > self.shared.snapshot().tip_number() {

// After (correct comparand):
if block_number > self.shared.get_unverified_tip().number() {
```

This ensures `unverified_tip` advances monotonically and cannot be jumped arbitrarily far ahead by a single block.

Additionally, add a monotonicity guard in `set_unverified_tip` (`shared/src/shared.rs:407`) using a CAS loop on the `ArcSwap` to reject any update that would set a lower or equal value, providing defense-in-depth.

## Proof of Concept

1. Start a CKB node in IBD mode. Record `tip = shared.snapshot().tip_number()` and `tip_hash`.
2. Craft a block: `parent_hash = tip_hash`, `block_number = tip + 73729`, one cellbase transaction with `since = tip + 73729`, empty transaction list otherwise, valid merkle root computed from the cellbase.
3. Send the block via P2P `SendBlock` message.
4. Assert: `shared.get_unverified_tip().number() == tip + 73729`.
5. Assert: `BlockFetcher::fetch()` returns `None` for all connected peers (hits `>= unverified_tip_limit` guard).
6. Assert: no further blocks are downloaded; verified tip does not advance.
7. Wait for the contextual verifier to process and reject the block (resetting `unverified_tip` to `tip`). Craft a new block with a different hash (e.g., vary a header field) and repeat from step 3 to confirm continuous disruption.
