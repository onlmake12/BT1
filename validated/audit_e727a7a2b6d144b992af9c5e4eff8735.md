### Title
Unprotected `initialize` Function Allows Any Caller to Hijack Guardian Set Before Deployment Completes — (`File: target_chains/stylus/contracts/wormhole/src/lib.rs`)

---

### Summary

The Stylus `WormholeContract`'s `initialize` function is publicly callable by any address and has no caller restriction. The only guard is a boolean `initialized` flag. Because deployment and initialization are two separate transactions in the Stylus deployment flow, an attacker can front-run the legitimate initialization call and install an attacker-controlled guardian set, permanently compromising VAA verification for the Pyth receiver.

---

### Finding Description

The `initialize` function in `WormholeContract` is declared inside the `#[public]` impl block, making it an externally callable entry point on Arbitrum Stylus:

```rust
#[public]
impl WormholeContract {
    pub fn initialize(
        &mut self,
        initial_guardians: Vec<Address>,
        initial_guardian_set_index: u32,
        chain_id: u16,
        governance_chain_id: u16,
        governance_contract: Address,
    ) -> Result<(), Vec<u8>> {
        if self.initialized.get() {
            return Err(WormholeError::AlreadyInitialized.into());
        }
        // ... sets guardian set, chain_id, governance_contract ...
        self.initialized.set(true);
        Ok(())
    }
``` [1](#0-0) 

There is no `msg.sender` check, no deployer-only guard, and no equivalent of OpenZeppelin's `Ownable` or `_disableInitializers`. The deployment script (`initialize_contract.sh`) confirms that initialization is performed in a **separate transaction** after contract deployment:

```bash
cast send "$CONTRACT_ADDRESS" "initialize(address[],uint16,uint16,address)" ...
``` [2](#0-1) 

This creates a window between deployment and initialization during which any address can call `initialize` first with attacker-controlled guardian addresses.

The same pattern exists in the Stylus `PythReceiver`:

```rust
pub fn initialize(&mut self, ...) -> Result<(), PythReceiverError> {
    if self.initialized.get() {
        return Err(PythReceiverError::AlreadyInitialized.into());
    }
    // ... sets wormhole address, data sources, governance ...
    self.initialized.set(true);
    Ok(())
}
``` [3](#0-2) 

By contrast, the Solidity contracts correctly use OpenZeppelin's `initializer` modifier (which is atomic with proxy deployment via `initData` in the constructor) and `_disableInitializers()` in the implementation constructor:

```solidity
constructor() initializer {}
``` [4](#0-3) 

The Stylus contracts have no equivalent protection.

---

### Impact Explanation

If an attacker front-runs the `initialize` call on `WormholeContract`:

1. The attacker sets `initial_guardians` to addresses they control.
2. `initialized` is set to `true`, permanently blocking the legitimate deployer from calling `initialize` again.
3. The attacker can now produce ECDSA signatures from their controlled guardian keys and construct valid-looking VAAs.
4. `parse_and_verify_vm` will accept these forged VAAs because it verifies against the attacker-installed guardian set.
5. The `PythReceiver` calls `parse_and_verify_vm` to validate price update VAAs — forged VAAs will pass, allowing the attacker to inject arbitrary price data for any price feed.

This is a complete compromise of the Pyth price oracle on any chain where the Stylus Wormhole contract is deployed. [5](#0-4) 

---

### Likelihood Explanation

The deployment and initialization are explicitly two separate steps (a deployment transaction followed by a `cast send` call). Any attacker monitoring the mempool on Arbitrum Stylus can observe the deployment transaction, compute the deployed contract address, and submit an `initialize` call with higher gas priority before the deployer's initialization transaction is included. This is a standard front-running attack with no special prerequisites — no privileged role, no leaked key, no oracle manipulation required.

---

### Recommendation

1. **Add a deployer-only guard**: Store `msg::sender()` at construction time (or use a constructor argument) and require it in `initialize`.
2. **Alternatively, call `initialize` atomically in the deployment script** in a single transaction or use a factory contract that deploys and initializes atomically.
3. **Mirror the Solidity pattern**: Use a `_disableInitializers()` equivalent so the bare implementation cannot be initialized, and pass `initData` to the proxy constructor.

---

    pub fn parse_and_verify_vm(&self, encoded_vaa: Vec<u8>) -> Result<Vec<u8>, Vec<u8>> {
        if !self.initialized.get() {
            return Err(WormholeError::NotInitialized.into());
        }

        if encoded_vaa.is_empty() {
            return Err(WormholeError::InvalidVAAFormat.into());
        }

        let vaa = self.parse_vm(&encoded_vaa)?;

        let _verified = self.verify_vm(&vaa)?;

        Ok(vaa.payload)
    }
