Audit Report

## Title
Unbounded JSON-RPC Batch Request Causes RPC Server Resource Exhaustion — (File: rpc/src/server.rs)

## Summary
The CKB JSON-RPC server's batch-size guard is opt-in and disabled by default. `rpc_batch_limit` is declared as `Option<usize>` with no default, and the `JSONRPC_BATCH_LIMIT` `OnceLock` is only populated when the operator explicitly sets the field. When absent, the guard is never entered and arbitrarily large batch arrays are processed unconditionally, making the RPC server unresponsive for the duration of the batch.

## Finding Description
`rpc_batch_limit` is declared as `Option<usize>` with no `#[serde(default)]` and no default value, so it deserializes to `None` when absent from `ckb.toml`. [1](#0-0) 

In `RpcServer::new`, `JSONRPC_BATCH_LIMIT` is only initialized inside an `if let Some(...)` branch. When `rpc_batch_limit` is `None` (the default), the `OnceLock` is never written. [2](#0-1) 

In `handle_jsonrpc`, the batch-size guard is `if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()`. When the `OnceLock` is unset, `.get()` returns `None`, the arm is never entered, and the full batch is dispatched unconditionally. [3](#0-2) 

Each call is then dispatched sequentially via `stream::iter(calls).then(...)` with no concurrency cap and no per-batch memory ceiling. The only protection is the global 30-second `TimeoutLayer`, which an attacker can exploit by sending repeated requests to keep the server continuously saturated. [4](#0-3) 

The default `resource/ckb.toml` ships with `rpc_batch_limit` commented out and explicitly documents that there is no limit by default, confirming this is the out-of-the-box behavior. [5](#0-4) 

## Impact Explanation
This matches **Note (0–500 points): Any local RPC API crash**. The RPC server becomes unresponsive to all concurrent clients for the duration of each oversized batch. The claim's assertion of a "High" impact (network congestion or node crash) is not proven by the submitted evidence — the code does not demonstrate that the RPC Tokio handle is shared with the node's P2P or consensus runtime in a way that would propagate the saturation beyond the RPC layer. The confirmed, concrete impact is RPC server unresponsiveness, which fits the Note tier.

## Likelihood Explanation
Any unprivileged RPC caller can trigger this with a single HTTP POST. No authentication, no special privilege, and no prior knowledge beyond the public JSON-RPC API is required. The default `ckb.toml` ships without `rpc_batch_limit` set, so all operators using the generated config without explicitly adding the option are affected. The attack is repeatable and requires no victim interaction.

## Recommendation
Set a safe non-`None` default for `rpc_batch_limit` in `Config` (e.g., `2000`, matching the commented-out example in `ckb.toml`) so the guard is active out of the box without operator action. [1](#0-0) 

The `OnceLock` initialization in `RpcServer::new` should then unconditionally populate `JSONRPC_BATCH_LIMIT` rather than only doing so when the field is `Some`. [2](#0-1) 

## Proof of Concept
```python
import json, requests

ENDPOINT = "http://127.0.0.1:8114"

# Build a batch of 200,000 cheap calls — fits within the default 10 MiB body limit
batch = [{"jsonrpc": "2.0", "method": "ping", "id": i} for i in range(200_000)]

# Single POST; server processes all calls sequentially with no limit
resp = requests.post(ENDPOINT, json=batch, timeout=120)
# RPC server is now saturated; concurrent legitimate requests time out
```
Run against a node started with the default generated `ckb.toml` (no `rpc_batch_limit` set). Concurrent RPC calls issued during the batch will time out, confirming the server is unresponsive for the batch duration. Replacing `"ping"` with `"get_block"` (verbosity=2) amplifies per-call cost and reduces the batch size needed to exhaust the 30-second timeout window.
