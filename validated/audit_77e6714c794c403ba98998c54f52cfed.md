Based on my investigation of the codebase, here is my assessment:

---

### Title
Sync-Layer `version_check` Hardcodes `version == 0` Without Epoch Awareness, Permanently Caching Post-Fork Blocks as `BLOCK_INVALID` — (`sync/src/synchronizer/headers_process.rs`)

### Summary

`HeaderAcceptor::version_check` unconditionally rejects any header with `version != 0` and permanently marks the block hash as `BLOCK_INVALID`. It has no awareness of `rfc_0048` / `is_remove_header_version_reservation_rule_enabled`. After rfc_0048 activates and miners begin producing `version=1` blocks, every node running this code will permanently blacklist those valid post-fork block hashes at the sync layer, making recovery impossible without a full resync.

### Finding Description

**Sync-layer `version_check` — hardcoded, epoch-blind:** [1](#0-0) 

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    if self.header.version() != 0 {
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
```

There is no call to `is_remove_header_version_reservation_rule_enabled` here. The check is a static `!= 0` with no epoch parameter.

**On failure, `BLOCK_INVALID` is permanently written:** [2](#0-1) 

```rust
if self.version_check(&mut result).is_err() {
    shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
    return result;
}
```

**The hardfork flag exists but is never consulted by the sync layer:** [3](#0-2) 

`is_remove_header_version_reservation_rule_enabled` is defined for `CKB2023` and is wired to `rfc_0048`, but `HeaderAcceptor::version_check` never calls it.

**The full-block verifier (`BlockVerifier`) does NOT check block version at all:** [4](#0-3) 

`BlockVerifier::verify` calls `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier` — no version check. Similarly, `HeaderVerifier::verify` only checks PoW, number, epoch, and timestamp: [5](#0-4) 

This creates a direct mismatch: the sync layer is more restrictive than the full-block verifier, and the sync layer's rejection is permanent.

### Impact Explanation

After rfc_0048 activates and miners produce `version=1` blocks:

1. Any peer relaying a `SendHeaders` message containing a `version=1` header triggers `HeaderAcceptor::accept()`.
2. `version_check` rejects it unconditionally.
3. `shared.insert_block_status(hash, BlockStatus::BLOCK_INVALID)` is called — the hash is permanently blacklisted.
4. The node can never accept that block or any descendant (via `prev_block_check`).
5. The node is stuck, repeatedly issuing `GetHeaders` requests, flooding the network.

This matches the "network congestion" scope: partitioned nodes cannot advance their chain tip and generate repeated sync traffic.

### Likelihood Explanation

- No attacker required: any legitimate miner producing a post-fork `version=1` block triggers this.
- The `BLOCK_INVALID` status is written to shared state and is not automatically cleared on restart.
- All nodes running this binary version are equally affected — the partition is between nodes running this code and nodes running a corrected version.
- The precondition (rfc_0048 activation epoch reached) is a scheduled, deterministic event.

### Recommendation

Gate `version_check` on the hardfork flag, passing the header's epoch:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    let consensus = self.active_chain.sync_shared().consensus();
    let hardforks = consensus.hardfork_switch();
    let epoch = self.header.epoch().number();
    let max_version = if hardforks.ckb2023().is_remove_header_version_reservation_rule_enabled(epoch) {
        consensus.block_version() // e.g., 1
    } else {
        0
    };
    if self.header.version() > max_version {
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
```

This mirrors how the full-block contextual verifier should use `is_remove_header_version_reservation_rule_enabled`.

### Proof of Concept

Differential test: construct a `version=1` header at the rfc_0048 activation epoch. Run it through:
1. `HeaderAcceptor::version_check` → asserts `Err(())` (rejects)
2. The contextual block verifier at the same epoch → asserts `Ok(())` (accepts)

The divergence is the bug. Both should accept or both should reject for the same input at the same epoch. [6](#0-5) [7](#0-6)
