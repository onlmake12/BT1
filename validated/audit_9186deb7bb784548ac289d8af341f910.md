Audit Report

## Title
Full CKB-VM Execution Before Cycle Mismatch Detection Enables Asymmetric CPU DoS via Relay Protocol — (`tx-pool/src/process.rs`)

## Summary
In `_process_tx`, full `verify_rtx` (CKB-VM script execution) runs unconditionally before the `declared != verified.cycles` check. On `DeclaredWrongCycles` rejection the verify cache is never populated, and the tx_hash is removed from the relay filter, allowing the same transaction to be re-announced after reconnection from a new IP. An attacker with one valid UTXO and multiple IPs can force repeated full VM executions at negligible cost.

## Finding Description
The execution order in `_process_tx` is:

1. `pre_check` (cheap) — line 715
2. `fetch_tx_verify_cache` — line 719
3. `verify_rtx` — **full CKB-VM execution** — lines 724–732, with `max_cycles = declared_cycles` when provided (line 720)
4. `declared != verified.cycles` check — lines 736–748, returns `Err(Reject::DeclaredWrongCycles(...))`
5. Cache update — lines 758–765, **only reached on success**, never on `DeclaredWrongCycles`

The attack uses `declared_cycles = actual_cycles + 1`. With this value, `max_cycles = actual_cycles + 1` is passed to `verify_rtx`; the VM runs to full completion consuming `actual_cycles` cycles (within the limit), and the mismatch is only detected afterward. Using `declared_cycles = actual_cycles - 1` would instead cause a cycle-limit-exceeded error at line 734 before the mismatch check — the `C+1` path is the confirmed working attack vector.

`DeclaredWrongCycles` is `is_malformed_tx() = true`, triggering `ban_malformed` (3-day IP ban per peer index). It is also `is_allowed_relay() = true`, triggering `send_result_to_relayer(TxVerificationResult::Reject { tx_hash })`, which causes `remove_from_known_txs` in the relayer, clearing the tx_hash from the relay filter. Because the cache is never populated for the rejected `witness_hash`, every resubmission from a new IP triggers full VM execution again.

## Impact Explanation
Each relay submission with `declared_cycles = actual_cycles + 1` forces the victim node to run full `ContextualTransactionVerifier` including secp256k1 CKB-VM execution, occupying the verify queue for the full duration. The attacker spends O(1) CPU (send a P2P message); the victim spends O(secp256k1_cycles) CPU. With N IPs the attacker sustains N parallel verify-queue occupations.

**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
**Preconditions:**
- One valid UTXO to pass `pre_check` (never consumed since the tx is always rejected)
- Multiple IPs to bypass the 3-day ban per IP (Tor, VPN, or botnet)

The attack is indefinitely repeatable: (1) the UTXO is never spent, (2) the tx_hash is removed from the relay filter on rejection allowing re-announcement after reconnection, (3) the cache is never populated for the rejected witness_hash so every resubmission triggers full VM execution.

## Recommendation
Two complementary fixes:

1. **Populate `txs_verify_cache` on `DeclaredWrongCycles` rejection** so that subsequent resubmissions of the same witness use the cached `TimeRelativeTransactionVerifier` path instead of full VM execution.

2. **Do not call `remove_from_known_txs` on `DeclaredWrongCycles`** — retaining the tx_hash in the relay filter prevents re-announcement of the same transaction after reconnection, eliminating the repeatability of the attack.

## Proof of Concept
```
1. Attacker owns UTXO U, creates valid secp256k1-signed tx T (actual cycles = C)
2. Connect to victim node as relay peer
3. Send RelayTransactionHashes([T.hash])
4. Receive GetRelayTransactions([T.hash])
5. Send RelayTransactions([{cycles: C+1, transaction: T}])
   → victim runs full secp256k1 VM (C cycles of CPU, within limit C+1)
   → victim detects C+1 != C → DeclaredWrongCycles → bans attacker IP
   → victim removes T.hash from tx_filter (is_allowed_relay=true)
6. Reconnect from new IP → goto step 3
   (UTXO U is still unspent; T.hash is no longer filtered; cache unpopulated)
```

The existing integration test `DeclaredWrongCycles` in `test/src/specs/tx_pool/declared_wrong_cycles.rs` confirms the rejection path is reachable. Adapting it to use `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1` (over-declared) confirms the `DeclaredWrongCycles` path specifically and that the cache remains empty after rejection.

    pub fn is_malformed_tx(&self) -> bool {
        match self {
            Reject::Malformed(_, _) => true,
            Reject::DeclaredWrongCycles(..) => true,
            Reject::Verification(err) => is_malformed_from_verification(err),
            Reject::Resolve(OutPointError::OverMaxDepExpansionLimit) => true,
            _ => false,
        }
    }

    /// Returns true if the reject should be recorded.
    pub fn should_recorded(&self) -> bool {
        !matches!(self, Reject::Duplicated(..))
    }

    /// Returns true if tx can be resubmitted, allowing relay
    /// * Declared wrong cycles should allow relay with the correct cycles
    /// * Reject but is not malformed and the fee rate reached the threshold,

    ///   and expired clearing
    pub fn is_allowed_relay(&self) -> bool {
        matches!(self, Reject::DeclaredWrongCycles(..))
            || (!matches!(self, Reject::LowFeeRate(..)) && !self.is_malformed_tx())
