### Title
`total_tx_size` and `total_tx_cycles` Over-Counted When Cell-Dep Eviction Occurs in `add_entry()` — (File: tx-pool/src/component/pool_map.rs)

---

### Summary

In `PoolMap::add_entry()`, the pool's aggregate `total_tx_size` and `total_tx_cycles` statistics are pre-computed **before** `check_and_record_ancestors()` runs. When that function evicts cell-dep-referencing transactions via `remove_entry_and_descendants()` → `remove_entry()` → `update_stat_for_remove_tx()`, those removals correctly decrement `self.total_tx_size` and `self.total_tx_cycles`. However, the pre-computed values are then unconditionally written back, silently overwriting the correctly-decremented values. The result is that `total_tx_size` and `total_tx_cycles` are permanently inflated by the aggregate size/cycles of every evicted transaction, causing the pool to believe it is fuller than it actually is.

---

### Finding Description

In `add_entry()`:

```rust
// Line 210-211: totals pre-computed BEFORE any evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Line 213: may call remove_entry_and_descendants() → remove_entry()
//           → update_stat_for_remove_tx(), which DECREMENTS self.total_tx_size
evicts = self.check_and_record_ancestors(&mut entry)?;

// ...

// Lines 218-219: OVERWRITES the correctly-decremented self.total_tx_size
//                with the stale pre-computed value
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

The pre-computed `total_tx_size` equals `self.total_tx_size + entry.size` at the moment of the call. If `check_and_record_ancestors()` subsequently evicts entries totalling size `S_evict`, `update_stat_for_remove_tx()` correctly sets `self.total_tx_size = T - S_evict`. But lines 218–219 then unconditionally restore it to `T + entry.size`, discarding the eviction adjustment. The correct post-operation value should be `T - S_evict + entry.size`. [2](#0-1) 

The eviction path inside `check_and_record_ancestors()` is triggered when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, i.e., when a new transaction consumes a cell dep that is referenced by many in-pool transactions. [3](#0-2) 

`update_stat_for_remove_tx()` is the authoritative decrement path for both counters: [4](#0-3) 

---

### Impact Explanation

After the overwrite, `total_tx_size` (and `total_tx_cycles`) are permanently inflated by the aggregate size (cycles) of every transaction evicted during the cell-dep eviction path. This has three concrete effects:

1. **False pool-full rejections.** `limit_size()` compares `self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated counter causes the pool to evict additional legitimate transactions even though actual occupied space is below the limit. [5](#0-4) 

2. **Incorrect admission gating.** `updated_stat_for_add_tx()` uses `self.total_tx_size` as the base for overflow/limit checks. Subsequent honest transactions are rejected with `Reject::Full` even when real pool occupancy is well below `max_tx_pool_size`. [6](#0-5) 

3. **Incorrect RPC-reported pool state.** `TxPoolInfo.total_tx_size` is read directly from `pool_map.total_tx_size`, so the `get_pool_info` RPC returns a misleading value to all callers. [7](#0-6) 

---

### Likelihood Explanation

The trigger condition is reachable by any unprivileged tx-pool submitter. The existing integration test `TxPoolLimitAncestorCount` demonstrates the exact scenario: submit 2 000 transactions that each reference the same cell dep (`tx_a`'s output), then submit a transaction that **consumes** `tx_a`'s output. This causes `check_and_record_ancestors()` to evict 1 002 cell-dep-referencing transactions, which is precisely the code path that leaves `total_tx_size` inflated. [8](#0-7) 

No privileged access, key material, or majority hash power is required. The attacker only needs to be able to submit transactions to the pool.

---

### Recommendation

Move the `updated_stat_for_add_tx()` call to **after** `check_and_record_ancestors()` completes, so it uses the already-decremented `self.total_tx_size` as its base:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // *** MOVED: compute totals AFTER evictions have already decremented self.total_tx_size ***
    evicts = self.check_and_record_ancestors(&mut entry)?;
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

This mirrors the fix pattern from M-22: ensure the accounting update reflects **all** operations that modify the underlying resource, not just the primary one.

---

### Proof of Concept

1. Fill the pool with 2 000 transactions each referencing `tx_a`'s output as a cell dep (as in `TxPoolLimitAncestorCount`). Record `total_tx_size = T`.
2. Submit a transaction that **spends** `tx_a`'s output. `check_and_record_ancestors()` evicts 1 002 cell-dep-referencing transactions (total evicted size `S_evict`).
3. `update_stat_for_remove_tx()` is called 1 002 times, correctly setting `self.total_tx_size = T - S_evict`.
4. Lines 218–219 then overwrite: `self.total_tx_size = T + entry.size`.
5. The pool now reports `total_tx_size ≈ T + entry.size` instead of the correct `T - S_evict + entry.size`.
6. Any subsequent transaction submission that would have been accepted under the real occupancy is now rejected with `Reject::Full`, and `limit_size()` begins evicting additional honest transactions unnecessarily. [9](#0-8)

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
