### Title
Tx-Pool Minimum Fee Admission Check Uses Serialized Size Instead of Transaction Weight, Allowing High-Cycles Transactions to Bypass Fee Rate Enforcement — (File: tx-pool/src/util.rs)

---

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` computes the minimum required fee using raw serialized byte size (`tx_size`) as the weight argument to `min_fee_rate.fee()`. However, the canonical transaction weight used everywhere else in the codebase is `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For transactions whose cycle consumption dominates, the actual weight is far larger than `tx_size`, so the size-based admission threshold is far lower than the intended fee-rate floor. No subsequent weight-based fee check exists in the admission path, so the size-based check is the only gate.

---

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { … }
```

`FeeRate::fee` is defined as:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
```

The canonical weight function used for mining prioritisation, fee-rate statistics, and RBF scoring is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
// DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4
```

`check_tx_fee` passes `tx_size` where `get_transaction_weight(tx_size, cycles)` should be used. The code comment acknowledges the discrepancy ("Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check") but no corrective weight-based check is performed after cycles become known (post-verification).

Concrete numbers with default config (`min_fee_rate = 1 000 shannons/KW`, `max_tx_verify_cycles = 70 000 000`):

| Parameter | Value |
|---|---|
| tx_size | 200 bytes |
| cycles | 70 000 000 |
| Actual weight | max(200, 70 000 000 × 0.000 170 571 4) = **11 940** |
| min_fee (size-based, what the check uses) | 1 000 × 200 / 1 000 = **200 shannons** |
| min_fee (weight-based, what it should be) | 1 000 × 11 940 / 1 000 = **11 940 shannons** |
| Under-enforcement ratio | **~60×** |

The same unit mismatch appears in `calculate_min_replace_fee` for RBF:

```rust
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
```

---

### Impact Explanation

A transaction sender can craft a transaction that:
1. Has high cycle consumption (up to `max_tx_verify_cycles = 70 000 000`) and small serialized size.
2. Pays a fee just above the size-based floor (e.g., 201 shannons) — far below the weight-based floor (11 940 shannons).
3. Passes `check_tx_fee` and is admitted to the pool.

Once admitted, the entry's `fee_rate()` (computed with actual weight) is extremely low. The pool can be flooded with such entries at a fraction of the intended cost, degrading pool quality and potentially displacing legitimate transactions. The same bypass applies to RBF replacement: a high-cycles replacement transaction can satisfy `calculate_min_replace_fee` with a fee ~60× lower than the weight-based minimum, undermining the economic guarantee of RBF.

---

### Likelihood Explanation

Any unprivileged transaction sender reachable via the `send_transaction` JSON-RPC or the P2P relay protocol can trigger this. Crafting a high-cycles, small-size transaction requires only a valid CKB-VM script that loops; no special privilege or key is needed. The discrepancy is largest when cycles approach `max_tx_verify_cycles`, which is a publicly known and configurable bound.

---

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee check using the true weight:

let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight { return Err(Reject::LowFeeRate(…));
