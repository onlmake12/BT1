Audit Report

## Title
Unbounded `vec![0u64; max_bucket_index + 1]` allocation via attacker-controlled fee rate causes OOM node crash — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary
`WeightUnitsFlow::do_estimate` derives `max_bucket_index` directly from the highest fee rate in the mempool via `max_bucket_index_by_fee_rate`, which applies no upper-bound cap. An attacker submitting a single transaction with an extremely high fee causes every subsequent `estimate_fee_rate` RPC call to attempt multi-gigabyte heap allocations, aborting the node process via Rust's `handle_alloc_error`.

## Finding Description

**Unbounded bucket index computation:**

`max_bucket_index_by_fee_rate` maps any `u64` fee rate to a `usize` with no ceiling. The final match arm at line 357:

```rust
x => (x + t * 11_500) / (100 * t),   // t = 1000
``` [1](#0-0) 

For `fee_rate = 5×10^13` (achievable with a ~200-byte tx paying ~100,000 CKB as fee), this yields `max_bucket_index ≈ 5×10^8`.

**Uncapped allocation in `do_estimate`:**

Two separate `Vec<u64>` are allocated unconditionally at this index size:

```rust
let mut buckets = vec![0u64; max_bucket_index + 1];  // line 219
// ...
let mut buckets = vec![0u64; max_bucket_index + 1];  // line 248
``` [2](#0-1) [3](#0-2) 

At `max_bucket_index = 5×10^8`, each allocation is `5×10^8 × 8 = 4 GB`. Rust's global allocator calls `handle_alloc_error` on failure, which **aborts the process**.

**`max_fee_rate` is derived directly from pool state with no cap:**

`do_estimate` takes `max_fee_rate` from the first (highest) element of `sorted_current_txs`, built from all pending and proposed pool entries: [4](#0-3) 

**`FeeRate::calculate` has no saturation cap beyond `u64::MAX`:**

For a minimal tx (weight ≈ 200) paying 100,000 CKB (10^13 shannons): `fee_rate = 10^13 * 1000 / 200 = 5×10^13`. [5](#0-4) 

**No max fee rate guard in tx pool admission:**

`check_tx_fee` only enforces a minimum fee rate floor; there is no upper bound: [6](#0-5) 

**`estimate_fee_rate` is a public, unauthenticated RPC endpoint:** [7](#0-6) 

## Impact Explanation

Every call to the public `estimate_fee_rate` RPC triggers `do_estimate` → two `vec![0u64; max_bucket_index + 1]` allocations sized by the highest fee rate in the pool. With a sufficiently high-fee transaction present, the allocator either aborts the process (OOM via `handle_alloc_error`) or stalls for seconds on overcommit systems. This constitutes **node crash / full DoS**, matching the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

- **Precondition**: `WeightUnitsFlow` must be explicitly configured. It is non-default but is a documented, supported production option per `resource/ckb.toml`.
- **Economic cost**: The attacker must burn CKB as fee. ~100,000 CKB achieves a 4 GB allocation (sufficient for OOM on most nodes). This is a real but not prohibitive cost for a targeted attack.
- **No cryptographic or consensus barrier**: The transaction is fully valid and passes all admission checks.
- **Repeatability**: The attacker can re-submit after each block to maintain the condition indefinitely.

## Recommendation

1. **Cap `max_bucket_index`** in `max_bucket_index_by_fee_rate` to a hard constant (e.g., 10,000), matching the practical fee rate range the algorithm is designed for.
2. **Clamp `max_fee_rate`** in `do_estimate` before computing the bucket index: if `max_fee_rate` exceeds a reasonable threshold, clamp it to the cap value.
3. **Optionally**, add a `max_fee_rate` admission check in `check_tx_fee` to reject transactions with implausibly high fee rates.

## Proof of Concept

```rust
// Compute bucket index for fee_rate = 5×10^13
let fee_rate = FeeRate::from_u64(50_000_000_000_000u64);
let max_bucket_index = Algorithm::max_bucket_index_by_fee_rate(fee_rate);
// max_bucket_index ≈ 500_000_000
// vec![0u64; 500_000_001] → 4 GB allocation → OOM abort

// Attacker transaction parameters:
// - size: ~200 bytes (minimal tx)
// - fee: 100_000 CKB = 10^13 shannons
// - weight: ~200
// - fee_rate = 10^13 * 1000 / 200 = 5×10^13
// - max_bucket_index ≈ 5×10^8 → 4 GB allocation
```

Steps:
1. Configure node with `algorithm = "WeightUnitsFlow"` in `[fee_estimator]`.
2. Submit a ~200-byte transaction paying ~100,000 CKB as fee via `send_transaction` RPC.
3. Call `estimate_fee_rate` RPC — `do_estimate` attempts two ~4 GB allocations → node aborts.
4. Re-submit after each block to maintain the condition.

    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }
