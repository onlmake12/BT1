### Title
Network Alert Signature Replay Across Networks — Missing Chain/Network Domain Separator in `calc_alert_hash` (`File: util/network-alert/src/verifier.rs`)

### Summary

The CKB network alert system signs and verifies `RawAlert` messages using a multi-signature scheme. The signed message is computed as `blake2b_256(raw_alert.as_slice())` — a hash of only the alert payload fields — with no chain identifier, genesis hash, or network domain separator included. This means a valid alert signature produced for one CKB network (e.g., mainnet) is cryptographically valid on any other network sharing the same alert signing keys (e.g., testnet, or any fork), enabling cross-network signature replay.

### Finding Description

**Root cause:** `calc_alert_hash` in `util/gen-types/src/extension/calc_hash.rs` delegates to `self.calc_hash()`, which is `blake2b_256(self.as_slice())` — a plain hash of the molecule-serialized `RawAlert` bytes. [1](#0-0) 

The `RawAlert` molecule schema contains only: `notice_until`, `id`, `cancel`, `priority`, `message`, `min_version`, `max_version`. [2](#0-1) 

No chain ID, genesis hash, or network name is included. The `Verifier::verify_signatures` function computes the message directly from this hash and checks it against the hard-coded public key set: [3](#0-2) 

The hard-coded public keys are the same for all networks by default (loaded from `alert_signature.toml`): [4](#0-3) [5](#0-4) 

**Exploit path:**

1. An attacker observes a valid alert broadcast on CKB testnet (or any other CKB-derived network using the same signing keys). The alert is relayed over P2P and is publicly visible.
2. The attacker extracts the `Alert` message bytes (including the valid signatures) from the P2P stream or via the `get_blockchain_info` RPC.
3. The attacker submits the identical `Alert` (same `id`, `message`, `signatures`) to a CKB mainnet node via the `send_alert` RPC or directly over the P2P alert protocol.
4. `Verifier::verify_signatures` recomputes `blake2b_256(raw_alert.as_slice())` — which is identical on both networks for the same payload — and the signatures verify successfully.
5. The alert is accepted, stored, and relayed across the entire mainnet P2P network.

The `AlertRelayer::received` handler accepts any alert that passes signature verification and has not been seen before by alert ID: [6](#0-5) 

### Impact Explanation

An attacker who can observe a legitimately signed alert on any CKB-derived network (testnet, devnet, or a fork) sharing the same alert signing keys can replay it verbatim on mainnet. The replayed alert will:

- Pass signature verification and be accepted by every mainnet node.
- Be broadcast across the entire P2P network via `quick_filter_broadcast`.
- Appear in `get_blockchain_info().alerts` on every node, potentially causing user confusion, false urgency, or panic (e.g., a testnet "critical bug" alert appearing on mainnet).
- Potentially be used to replay a `cancel` alert, suppressing a legitimate active mainnet alert by replaying a testnet cancellation with the same `cancel` field value.

The alert system is explicitly described as a mechanism for urgent, trusted communication from CKB core developers to all nodes. Unauthorized injection of alerts undermines this trust channel.

### Likelihood Explanation

- Alert messages are publicly broadcast over P2P and visible to any connected peer.
- The `send_alert` RPC endpoint is reachable by any local or remote caller with RPC access.
- No privileged access is required beyond observing the P2P network or an RPC endpoint.
- The same four hard-coded public keys are used as the default for all networks.
- The only precondition is that a legitimately signed alert has been issued on any network sharing those keys — which has already occurred (alert `20230001` is hardcoded in the test suite with real signatures).

### Recommendation

Include a network-specific domain separator in the signed data. The simplest fix is to mix the genesis block hash (or the consensus `identify_name` string such as `"ckb"` vs `"ckb_testnet"`) into the alert hash computation inside `calc_alert_hash`, or to add a `chain_id` field to the `RawAlert` molecule schema. This ensures a signature produced for one network is cryptographically invalid on any other.

    pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
        trace!("Verifying alert {:?}", alert);
        let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
        let signatures: Vec<Signature> = alert
            .signatures()
            .into_iter()
            .filter_map(
                |sig_data| match Signature::from_slice(sig_data.as_reader().raw_data()) {
                    Ok(sig) => {
                        if sig.is_valid() {
                            Some(sig)
                        } else {
                            debug!("invalid signature: {:?}", sig);
                            None
                        }
                    }
                    Err(err) => {
                        debug!("signature error: {}", err);
                        None
                    }
                },
            )
            .collect();
        verify_m_of_n(
            &message,
            self.config.signatures_threshold,
            &signatures,
            &self.pubkeys,
        )
        .map_err(|err| err.kind())?;
        Ok(())
    }
