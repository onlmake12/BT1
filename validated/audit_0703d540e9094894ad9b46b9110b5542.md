Audit Report

## Title
Silent `u64` Truncation in `calculate_maximum_withdraw` Returns Wrong NervosDAO Withdrawal Amount - (File: util/dao/src/lib.rs)

## Summary
In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `u128` result `withdraw_counted_capacity` is cast to `u64` via `as u64` at line 156, silently truncating if the value exceeds `u64::MAX`. Every other analogous `u128`→`u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When truncation occurs, the function returns a drastically wrong (much smaller) withdrawal amount, the public RPC surfaces this wrong value, and any block containing a withdrawal transaction built from it is rejected at the `dao_field_with_current_epoch` step due to underflow in `withdrawed_interests`.

## Finding Description
**Root cause — line 156, `util/dao/src/lib.rs`:**
```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
```
The `as u64` cast keeps only the low 64 bits of `withdraw_counted_capacity`. This is structurally inconsistent with the three other `u128`→`u64` conversions in the same file, all of which use checked conversion:
- Line 204: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- Lines 244–245: `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?`
- Line 258: `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?`

**Overflow condition:**
`counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`

With `deposit_ar = 10^10` (initial) and `counted_capacity` up to ~3.36×10^18 shannons (total CKB supply), overflow occurs when `withdrawing_ar > ~5.36×10^10`, i.e., AR has grown by ~5.36× from deposit time.

**Exploit flow:**
1. A depositor deposits a large amount of CKB into NervosDAO when AR is at its initial value.
2. After AR has grown sufficiently (see Likelihood), the depositor calls the RPC `calculate_dao_maximum_withdraw` (`rpc/src/module/experiment.rs`, lines 259–267), which calls `calculate_maximum_withdraw` and returns the truncated (wrong, much smaller) value.
3. The depositor constructs a withdrawal transaction with `outputs_capacity` equal to the truncated value. Fee validation passes because `transaction_fee = maximum_withdraw - outputs_capacity` computes to zero using the same truncated `maximum_withdraw`.
4. The transaction enters the mempool and is included in a block.
5. During block processing, `dao_field_with_current_epoch` calls `withdrawed_interests` (lines 312–333), which computes `maximum_withdraws - input_capacities`. Because `maximum_withdraws` is the truncated value (far smaller than `input_capacities`), `safe_sub` returns `DaoError::Overflow`, and the block is rejected.

**Why existing checks fail:**
The existing test `check_withdraw_calculation_overflows` (lines 296–349, `util/dao/src/tests.rs`) only exercises the `safe_add` overflow path — where `withdraw_counted_capacity` is just above `u64::MAX` so the truncated value plus `occupied_capacity` still overflows. It does not test the silent truncation path where the truncated value is small enough that `safe_add` succeeds with a wrong result.

## Impact Explanation
A NervosDAO depositor who deposited a large amount of CKB and holds until AR has grown by ~5.36× will find:
- The RPC `calculate_dao_maximum_withdraw` reports a drastically wrong (much smaller) withdrawal amount.
- Any withdrawal transaction constructed from that amount is accepted into the mempool but causes the containing block to be rejected at the `dao_field_with_current_epoch` step.
- The depositor is permanently unable to withdraw their funds via the normal path.
- A miner who includes such a transaction loses their block reward.

This maps to **"Vulnerabilities which could easily damage CKB economy"** (Critical, 15001–25000 points): large depositors lose access to their funds and miners lose block rewards when processing such transactions.

## Likelihood Explanation
On mainnet, the secondary epoch reward is ~1.344 billion CKB/year against a total capacity of ~33.6 billion CKB, giving an AR growth rate of ~4%/year. A 5.36× increase requires approximately 42 years of chain operation — a long-term scenario on mainnet. However:
- On testnets or devnets with higher secondary reward ratios or lower total capacity, the condition is reachable much sooner.
- The bug is structurally present and inconsistent with the rest of the file today.
- Any future parameter change that increases secondary issuance or reduces total capacity accelerates the timeline.
- No unprivileged action is required to trigger it — a depositor simply holds their deposit until the AR threshold is crossed.

## Recommendation
Replace the silent `as u64` cast with the same checked conversion pattern used everywhere else in the file:
```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```
This makes the function consistent with `secondary_block_reward`, `dao_field_with_current_epoch`, and all other `u128`→`u64` conversions in the file, and ensures overflow is surfaced as a `DaoError::Overflow` rather than silently producing a wrong result.

## Proof of Concept
Add the following unit test to `util/dao/src/tests.rs`:
```rust
#[test]
fn check_withdraw_calculation_silent_truncation() {
    // counted_capacity ≈ total CKB supply in shannons
    // withdrawing_ar / deposit_ar = 6 → result = 6 × counted_capacity > u64::MAX
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(3_360_000_000_000_000_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000,  // deposit_ar = 10^10
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            60_000_000_000,  // withdrawing_ar = 6 × 10^10 → 6× growth
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    // ... (store setup as in check_withdraw_calculation_overflows) ...

    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    // With the bug: result is Ok(truncated wrong value) instead of Err(Overflow)
    // With the fix: result is Err(DaoError::Overflow)
    assert!(result.is_err(), "expected Overflow error, got {:?}", result);
}
```
With the current code, `result` is `Ok(Capacity::shannons(1_713_255_926_290_448_384))` — a silently truncated wrong value — instead of `Err(DaoError::Overflow)`. With the recommended fix, it correctly returns `Err`.
