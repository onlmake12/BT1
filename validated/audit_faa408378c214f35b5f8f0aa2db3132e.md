### Title
EIP-150 63/64 Gas Rule Not Enforced in `executeCallback` Allows Malicious Caller to Permanently Brick User Callbacks - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` forwards exactly `req.callbackGasLimit` gas to the user's `_echoCallback` via a `try` block, but performs **no on-chain check** that `gasleft()` is sufficient to actually forward that amount. Because the EVM silently caps forwarded gas at 63/64 of the calling context's `gasleft()` (EIP-150), a malicious caller can supply a precise gas amount that passes all pre-callback logic but causes the callback to receive less gas than `callbackGasLimit`. The request is **cleared before the callback executes**, so the failure is permanent and the user cannot retry.

---

### Finding Description

In `Echo.executeCallback`, the sequence of operations is:

1. Validate provider exclusivity and price IDs
2. Parse price feeds via Pyth
3. **Credit provider fees** (`_state.providers[providerToCredit].accruedFeesInWei += ...`)
4. **`clearRequest(sequenceNumber)`** — request is permanently deleted
5. Update `firstUnfulfilledSeq`
6. `try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` — callback executed [1](#0-0) 

There is no `gasleft()` check before step 6. The EVM's 63/64 rule (EIP-150) means that if `gasleft()` at the point of the `try` call is less than `req.callbackGasLimit * 64 / 63`, the callback receives fewer gas units than `callbackGasLimit`. The `catch` block silently emits `PriceUpdateCallbackFailed` and the function returns successfully. [2](#0-1) 

Because `clearRequest` runs at line 164 **before** the callback, the request is gone regardless of callback outcome. There is no replay or retry mechanism. [3](#0-2) 

Compare this to `Entropy.revealWithCallback`, which explicitly checks `(startingGas * 31) / 32 > uint256(req.gasLimit10k) * TEN_THOUSAND` and reverts with `InsufficientGas` when the calling context lacked sufficient gas — protecting the request from being permanently consumed. [4](#0-3) 

`Echo.sol` has no equivalent protection.

---

### Impact Explanation

An attacker calls `executeCallback` with a gas amount carefully chosen so that:
- Enough gas remains to complete all pre-callback logic (price parsing, fee crediting, `clearRequest`, `firstUnfulfilledSeq` loop)
- At the point of `_echoCallback{gas: req.callbackGasLimit}`, `gasleft()` is slightly above `callbackGasLimit` but below `callbackGasLimit * 64/63`
- The EVM forwards only 63/64 of `gasleft()` < `callbackGasLimit` to the callback
- The callback runs out of gas and reverts
- The `catch` block fires, emitting `PriceUpdateCallbackFailed`
- The request is already cleared; the user's callback is permanently lost

The user paid a fee for a price-update callback that will never execute. The provider is already credited. There is no on-chain recovery path.

For any `callbackGasLimit` ≥ ~952,192 (where `callbackGasLimit / 63` exceeds the gas consumed between the `try` statement and the actual `CALL` opcode), the attack window is reliably exploitable. The `callbackGasLimit` is a `uint32`, allowing values up to ~4.29 billion, so the vulnerable range covers the vast majority of real-world usage. [5](#0-4) 

---

### Likelihood Explanation

- `executeCallback` is permissionless — any address can call it.
- The attacker needs only to identify a pending request with a high `callbackGasLimit` and call `executeCallback` with a precisely computed gas value.
- No privileged access, no leaked keys, no oracle manipulation required.
- The attack is cheap: the attacker pays only the gas for the transaction, and the victim's fee is already locked in the contract. [6](#0-5) 

---

### Recommendation

Add a `gasleft()` check before the callback, accounting for the 63/64 rule and the gas consumed by pre-callback operations. A conservative check:

```solidity
require(
    gasleft() >= (uint256(req.callbackGasLimit) * 64) / 63 + GAS_BUFFER,
    "Echo: insufficient gas for callback"
);
```

Alternatively, mirror the pattern used in `Entropy.revealWithCallback`: record `startingGas = gasleft()` before the call, and after a failed callback, check whether `(startingGas * 63) / 64 >= req.callbackGasLimit`. If not, **revert** (do not clear the request), so the request remains retryable. [7](#0-6) 

---

### Proof of Concept

```solidity
// Attacker contract
contract EchoGriefAttack {
    IEcho echo;

    constructor(address _echo) { echo = IEcho(_echo); }

    // Victim has a pending request with sequenceNumber N and callbackGasLimit = 1_000_000
    // Attacker calls this with gas = X (computed below)
    function attack(
        address provider,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external {
        // Supply gas such that at the point of _echoCallback{gas: 1_000_000},
        // gasleft() is in range (1_000_000, 1_000_000 * 64/63 = ~1_015_873).
        // EVM forwards only 63/64 * gasleft() < 1_000_000 to the callback.
        // Callback OOGs, catch fires, request is already cleared.
        echo.executeCallback{gas: COMPUTED_GAS}(
            provider, sequenceNumber, updateData, priceIds
        );
        // Request is now permanently gone. Victim's callback never ran.
    }
}
```

The `COMPUTED_GAS` value is: gas consumed by pre-callback logic + `callbackGasLimit` + small delta (< `callbackGasLimit / 63`). This passes all pre-callback operations but leaves `gasleft()` just above `callbackGasLimit` at the `try` site, triggering the 63/64 truncation.
