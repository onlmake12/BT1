### Title
Unbounded `pending_delivered` Growth via Spoofed `from` PeerIds in `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

A single connected peer can cause the `pending_delivered` HashMap in `HolePunching` to accumulate O(rate_limit × window) entries — far exceeding the number of connected peers — by sending `ConnectionRequest` messages with distinct, attacker-controlled `from` PeerIds, each targeting the victim node as `to`. The only guard is a per-session rate limiter (30 req/s), allowing ~9,000 insertions per connected peer per 5-minute cleanup window.

---

### Finding Description

`HolePunching::pending_delivered` is a `HashMap<PeerId, (Vec<Multiaddr>, u64)>` keyed by the `from` field of incoming `ConnectionRequest` messages. [1](#0-0) 

When the victim node is the `to` target, `respond_delivered` is called: [2](#0-1) 

Inside `respond_delivered`, the only dedup guard checks whether the **same** `from_peer_id` was seen within `HOLE_PUNCHING_INTERVAL` (2 minutes): [3](#0-2) 

If the check passes, the entry is unconditionally inserted: [4](#0-3) 

The `from` field is **not verified** to match the actual sender's peer ID — it is arbitrary attacker-supplied bytes parsed as a `PeerId`: [5](#0-4) 

The only rate guard is the per-session `rate_limiter` keyed by `(session_id, item_id)`, capped at 30/second: [6](#0-5) 

Cleanup runs every 5 minutes (`CHECK_INTERVAL`) and evicts entries older than 5 minutes (`TIMEOUT`): [7](#0-6) [8](#0-7) 

**Attack arithmetic (single connected peer):**
- Rate limit: 30 `ConnectionRequest`/second
- Window: 300 seconds (5 minutes)
- Max insertions: 30 × 300 = **9,000 entries**
- Each entry stores up to `ADDRS_COUNT_LIMIT` = 24 `Multiaddr` objects
- Heap cost: ~9,000 × 24 × ~80 bytes ≈ **~17 MB per connected peer per window**

With multiple connected peers this scales linearly.

---

### Impact Explanation

The `pending_delivered` map grows proportional to unique spoofed `from` PeerIds, not to the number of connected peers. This violates the intended invariant. The result is sustained heap pressure and HashMap rehashing overhead, causing performance degradation. This is not a crash or data breach — impact is **Low (501–2000)**: meaningful but not catastrophic degradation.

---

### Likelihood Explanation

The attack requires only a single valid P2P connection. The attacker needs to know the victim's peer ID (publicly advertised). Crafting messages with distinct `from` PeerIds is trivial. No PoW, no privileged role, no Sybil attack required.

---

### Recommendation

1. **Cap `pending_delivered` size**: Enforce a hard maximum (e.g., 1,000 entries) and reject new insertions when the cap is reached.
2. **Verify `from` matches the actual sender**: Before inserting, check that `from_peer_id` corresponds to the peer session that sent the message (i.e., the connected peer's actual peer ID).
3. **Alternatively**: Key the dedup check on `session_id` rather than (or in addition to) `from_peer_id`, so a single session cannot insert more than one entry regardless of the `from` field.

---
