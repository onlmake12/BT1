### Title
Indexer `Pool` Dead-Cell Set Misses Inputs of Directly-Proposed Transactions - (File: `util/indexer-sync/src/pool.rs`)

### Summary

The `PoolService` in `util/indexer-sync/src/pool.rs` tracks which cells are consumed by pending transactions by maintaining a `dead_cells: HashSet<OutPoint>`. It subscribes only to `new_transaction` and `reject_transaction` events. However, when a transaction is submitted to the tx-pool and its proposal short ID is already in the current proposal window, it enters the pool directly as `TxStatus::Proposed`, triggering only `call_proposed` → `notify_proposed_transaction`. The indexer's `PoolService` does not subscribe to `proposed_transaction`, so those inputs are never added to `dead_cells`. As a result, the indexer incorrectly reports cells consumed by directly-proposed transactions as live (unspent).

### Finding Description

In `shared/src/shared_builder.rs`, three callbacks are registered on the tx-pool:

- `register_pending` → calls `notify_new_transaction` (for `TxStatus::Fresh` and `TxStatus::Gap`)
- `register_proposed` → calls `notify_proposed_transaction` (for `TxStatus::Proposed`)
- `register_reject` → calls `notify_reject_transaction` [1](#0-0) 

In `tx-pool/src/process.rs`, `_submit_entry` routes the callback based on the resolved status:

```rust
TxStatus::Fresh   => callbacks.call_pending(&entry),
TxStatus::Gap     => callbacks.call_pending(&entry),
TxStatus::Proposed => callbacks.call_proposed(&entry),
``` [2](#0-1) 

`get_tx_status` assigns `TxStatus::Proposed` when the transaction's short ID is already in the current proposal window: [3](#0-2) 

The indexer's `PoolService::index_tx_pool()` subscribes only to `new_transaction` and `reject_transaction`: [4](#0-3) 

There is no subscription to `proposed_transaction`. Consequently, when a transaction enters the pool directly as `TxStatus::Proposed`, `Pool::new_transaction` is never called, and the transaction's inputs are never inserted into `dead_cells`: [5](#0-4) 

The `dead_cells` set is then used by both the basic indexer and the rich indexer to filter out pool-consumed cells from live-cell query results: [6](#0-5) [7](#0-6) 

Because the inputs of directly-proposed transactions are absent from `dead_cells`, those cells pass the filter and are returned as live by `get_cells` and `get_cells_capacity`.

### Impact Explanation

Any RPC caller or application using the indexer's `get_cells` / `get_cells_capacity` APIs (with `index_tx_pool = true`) receives incorrect data: cells consumed by a directly-proposed transaction appear as live and unspent. Wallets and DApps relying on the indexer for cell availability checks may:

1. Attempt to spend cells that are already consumed, resulting in failed transactions.
2. Accept payments or authorizations based on cells that will never be spendable, enabling a form of indexer-level double-spend deception against off-chain logic.

The stale state persists from the moment the transaction enters the proposed state until the block containing it is committed and `transactions_committed` clears the entries.

### Likelihood Explanation

This is triggered by a normal CKB two-phase commit scenario: a transaction is submitted via `send_transaction` RPC while its proposal short ID is already in the current proposal window (i.e., it was proposed in a recent block). Any RPC caller can deliberately craft this sequence. Nodes with `index_tx_pool = true` are affected.

### Recommendation

In `PoolService::index_tx_pool()`, also subscribe to `proposed_transaction` events and call `pool.new_transaction()` for transactions entering the pool directly as proposed (those not previously tracked as pending):

```rust
let mut proposed_transaction_receiver = notify_controller
    .subscribe_proposed_transaction(SUBSCRIBER_NAME.to_string())
    .await;

// In the select! loop:
Some(tx_entry) = proposed_transaction_receiver.recv() => {
    if let Some(pool) = service.pool.as_ref() {
        pool.write().expect("acquire lock").new_transaction(&tx_entry.transaction);
    }
}
```

Alternatively, track which transactions have already been added to `dead_cells` (e.g., by their tx hash) to avoid double-insertion when a tx transitions from pending to proposed.

### Proof of Concept

1. Node is configured with `index_tx_pool = true`.
2. A block is mined that includes the proposal short ID of transaction `tx_A` (but does not commit it).
3. An RPC caller submits `tx_A` via `send_transaction`. Because its short ID is in the proposal window, `get_tx_status` returns `TxStatus::Proposed`; `_submit_entry` calls `callbacks.call_proposed(&entry)` → `notify_proposed_transaction`.
4. The indexer's `PoolService` does not handle `proposed_transaction`; `Pool::new_transaction` is never called; `tx_A`'s inputs are absent from `dead_cells`.
5. A second RPC caller queries `get_cells` for the cells consumed by `tx_A`. The indexer returns them as live because they are not in `dead_cells`.
6. The second caller constructs a transaction spending those cells and submits it; the tx-pool rejects it with `Dead` because the cells are correctly tracked as consumed in the pool's edge map — but the indexer had already reported them as available.
