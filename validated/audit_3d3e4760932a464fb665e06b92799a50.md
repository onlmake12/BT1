Looking at the actual eviction logic in `try_evict_inbound_peer()` and the ping metric update paths in `ping.rs`.

### Title
Eviction Protection Metrics Are Fully Attacker-Controllable, Enabling Inbound Eclipse Attack — (`network/src/peer_registry.rs`)

---

### Summary

The `try_evict_inbound_peer()` function in `network/src/peer_registry.rs` protects peers from eviction using two metrics explicitly labeled as "characteristics that an attacker [is] hard to simulate or manipulate": `ping_rtt` and `last_ping_protocol_message_received_at`. Both metrics are in fact trivially and fully controllable by any inbound peer. An attacker holding `EVICTION_PROTECT_PEERS + 1` (= 9) inbound connections can guarantee all their connections survive every eviction round, causing only honest peers to be evicted until the attacker controls all inbound slots.

---

### Finding Description

**`sort_then_drop` semantics — what "protect" means:** [1](#0-0) 

`sort_then_drop` sorts the list by the given comparator and then calls `truncate(len - n)`, which **removes the last `n` elements**. Because the comparators sort in descending order (highest value first), the last `n` elements are the ones with the **lowest** values — i.e., the best peers. Those are dropped from the candidate list, meaning they are **protected from eviction**.

**Step 1 — Protect 8 peers with lowest `ping_rtt`:** [2](#0-1) 

The comparison key is `p.as_secs()` — truncated to **whole seconds**. Any RTT under 1 second produces `0`. On any modern network, virtually every peer has sub-second RTT, so this criterion degenerates: all fast peers are indistinguishable at 0 seconds. An attacker who responds to pings immediately (which is trivially done) gets `ping_rtt = 0 seconds`, identical to any honest nearby peer.

`ping_rtt` is set in `pong_received()`: [3](#0-2) 

The attacker controls when they send the Pong, so they control the measured RTT.

**Step 2 — Protect 8 peers with most recent `last_ping_protocol_message_received_at`:** [4](#0-3) 

`last_ping_protocol_message_received_at` is updated whenever the victim **receives a Ping or Pong** from the remote peer: [5](#0-4) 

The attacker can send a Ping message to the victim at any time, immediately refreshing this timestamp to `Instant::now()`. This is entirely attacker-controlled.

**`EVICTION_PROTECT_PEERS` constant:** [6](#0-5) 

**The attack math (with default `max_inbound` >> 17):**

- Attacker holds 9 inbound connections, all with `ping_rtt = 0s` and `last_ping_protocol_message_received_at = Instant::now()`.
- A new inbound connection arrives; `try_evict_inbound_peer()` is called on the full candidate pool.
- **Step 1**: 8 attacker peers are removed from candidates (protected by lowest ping).
- **Step 2**: The 9th attacker peer is removed from candidates (protected by most recent message).
- **Result**: Zero attacker peers remain as eviction candidates. Only honest peers can be evicted.
- Repeat until all inbound slots are attacker-controlled.

The code comment at line 149 explicitly states these are "characteristics that an attacker [is] hard to simulate or manipulate" — this assumption is incorrect for both metrics.

---

### Impact Explanation

A successful eclipse attack gives the attacker full control over all inbound connections to the victim node. The victim's view of the blockchain is entirely mediated by attacker-controlled peers, enabling:
- **Consensus deviation**: feeding the victim a fake chain or withholding blocks
- **Transaction censorship**: suppressing transaction relay to/from the victim
- **Double-spend facilitation**: isolating the victim from the honest network during an attack window

---

### Likelihood Explanation

The attack requires only:
1. The ability to open 9 TCP connections to the victim (standard P2P, no privilege required)
2. Responding to pings quickly (trivial)
3. Sending Ping messages periodically (trivial)

No hashpower, no leaked keys, no social engineering. The attacker needs IPs from at least 2 different `/16` subnets to survive the network-group grouping step, but this is easily achieved with commodity VPS providers. The attack is repeatable and deterministic once the attacker holds 9 slots.

---

### Recommendation

1. **Fix `as_secs()` granularity**: Use `as_millis()` or the full `Duration` for `ping_rtt` comparison so sub-second differences are meaningful.
2. **Decouple eviction metrics from attacker-controlled messages**: `last_ping_protocol_message_received_at` should not be updated by inbound Ping messages from the remote peer — only by Pong responses to locally-initiated Pings, and even then with rate limiting.
3. **Add a connection-age floor**: Require a minimum connection age before a peer qualifies for ping-based protection, preventing freshly-connected attackers from immediately gaining protection.
4. **Introduce randomness into protection**: Mix in a random per-session salt so an attacker cannot deterministically guarantee protection across all 9 slots simultaneously.

---

### Proof of Concept

```
1. Victim node: max_inbound = 125, currently at capacity with honest peers.
2. Attacker opens 9 inbound TCP connections to victim.
3. For each attacker connection:
   a. When victim sends SEND_PING_TOKEN ping, attacker immediately replies with Pong
      → ping_rtt = ~0ms → as_secs() = 0 → qualifies for Step 1 protection.
   b. Attacker sends a Ping message to victim every few seconds
      → last_ping_protocol_message_received_at = Instant::now() → qualifies for Step 2 protection.
4. Attacker opens a 10th connection to trigger eviction:
   → try_evict_inbound_peer() runs:
      Step 1: 8 attacker peers (ping=0) protected, removed from candidates.
      Step 2: 9th attacker peer (freshest message) protected, removed from candidates.
      Remaining candidates: only honest peers.
      → An honest peer is evicted.
5. Attacker's 10th connection fills the freed slot.
6. Repeat steps 3–5 until all 125 inbound slots are attacker-controlled.

Differential assertion: run eviction with attacker peers present vs. absent.
With attacker peers: P(honest peer evicted) = 1.0.
Without attacker peers: eviction is random among honest peers.
```

```rust
fn sort_then_drop<T, F>(list: &mut Vec<T>, n: usize, compare: F)
where
    F: FnMut(&T, &T) -> std::cmp::Ordering,
{
    list.sort_by(compare);
    if list.len() > n {
        list.truncate(list.len() - n);
    }
}
```
