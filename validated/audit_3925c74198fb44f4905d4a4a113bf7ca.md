### Title
Unbounded O(log N) DB-read amplification per `GetLastStateProof` request via crafted difficulties array — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
The size guard on `GetLastStateProof` limits the number of difficulties to ≤ 1000 but does not account for the O(log N) binary-search cost per difficulty in `get_block_numbers_via_difficulties`. An unprivileged peer can trigger O(1000 × log N) DB reads per request, violating the O(GET_LAST_STATE_PROOF_LIMIT) invariant.

### Finding Description
`execute()` checks `difficulties.len() + last_n_blocks * 2 > 1000` and rejects if exceeded. With `last_n_blocks = 0`, up to 1000 difficulties are accepted. After the `take_while` filter (which the attacker bypasses by keeping all difficulties ≤ `total_difficulty(boundary - 1)`), `get_block_numbers_via_difficulties` is called with all 1000 entries. For each entry, `get_first_block_total_difficulty_is_not_less_than` performs a binary search over `[start_block_number, difficulty_boundary_block_number)`, doing 2 DB reads per step. The `current_difficulty` guard only skips entries whose target block was already found; by mapping `D_i = total_difficulty(start + i)`, each entry finds a distinct block and no skipping occurs. The range-narrowing (`start_block_number = num - 1`) advances by only 1 per iteration when blocks are near the range start, so each binary search still spans O(N) blocks.

### Impact Explanation
Total DB reads per request: O(1000 × log N × 2). For a 1M-block chain (log₂(1M) ≈ 20): ~40,000 DB reads vs. the ~2,000 implied by the limit. An attacker sending a sustained stream of such messages amplifies I/O load by ~20×, causing CPU and disk saturation on the light-client protocol server, degrading or denying service to legitimate peers.

### Likelihood Explanation
The attacker needs only a P2P connection and knowledge of block total difficulties, which are publicly derivable from the chain. No special privileges, keys, or hashpower are required. The message is well-formed and passes all existing validation checks.

### Recommendation
Replace the flat count guard with a guard that accounts for binary-search cost. One approach: limit `difficulties.len()` to `GET_LAST_STATE_PROOF_LIMIT / ceil(log2(chain_length))`, or replace the per-difficulty binary search with a single linear scan over the sorted difficulty array and block range (O(N + K) total instead of O(K log N)).

### Proof of Concept
On a node with 1,000,000 blocks, craft a `GetLastStateProof` with:
- `last_n_blocks = 0`
- `difficulties = [total_difficulty(start+0), total_difficulty(start+1), ..., total_difficulty(start+999)]`
- `difficulty_boundary` = `total_difficulty(start+1000) + 1`

All 1000 difficulties pass the size guard, the monotonicity check, the `take_while` filter, and the `current_difficulty` guard. Each triggers a binary search over ~1,000,000 blocks: 1000 × 20 × 2 = **40,000 DB reads** for the sampling phase alone, vs. the ~2,000 the limit implies. Assert: `db_read_count > 2 * GET_LAST_STATE_PROOF_LIMIT * log2(chain_length)`.
