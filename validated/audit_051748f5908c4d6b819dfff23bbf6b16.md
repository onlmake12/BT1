## Title
Premature `mark_as_known_txs` Before `submit_remote_tx` Enables Transaction Propagation Suppression - (File: sync/src/relayer/transactions_process.rs)

## Summary
In `TransactionsProcess::execute()`, `mark_as_known_txs` is called at line 76 to remove all incoming relay transaction hashes from `unknown_tx_hashes` and insert them into `tx_filter` before any submission result is known. The actual `submit_remote_tx` calls are deferred into a spawned async task. When submission fails (e.g., `DeclaredWrongCycles`), the hash is permanently absent from `unknown_tx_hashes` and remains in `tx_filter` until TTL expiry, preventing the node from re-requesting the transaction from any other peer.

## Finding Description
**Confirmed code path:**

At `sync/src/relayer/transactions_process.rs` line 76, before the async spawn at lines 80–93:
```rust
shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));
```

`mark_as_known_txs` (lines 1443–1451 of `sync/src/types/mod.rs`) atomically removes each hash from `unknown_tx_hashes` and inserts it into `tx_filter`:
```rust
for hash in hashes {
    unknown_tx_hashes.remove(&hash);
    tx_filter.insert(hash);
}
```

`remove_from_known_txs` (lines 1436–1438 of `sync/src/types/mod.rs`) only removes from `tx_filter` — it never restores the hash to `unknown_tx_hashes`:
```rust
pub fn remove_from_known_txs(&self, hash: &Byte32) {
    self.tx_filter.lock().remove(hash);
}
```

This asymmetry is the root cause.

**Failure path confirmed:**

`resumeble_process_tx_and_notify_full_reject` (lines 364–366 of `tx-pool/src/process.rs`) only sends `TxVerificationResult::Reject` for `Reject::Full`:
```rust
if matches!(ret, Err(Reject::Full(_))) {
    self.send_result_to_relayer(TxVerificationResult::Reject { tx_hash });
}
```

`send_bulk_of_tx_hashes` (lines 673–675 of `sync/src/relayer/mod.rs`) calls `remove_from_known_txs` only on `TxVerificationResult::Reject`:
```rust
TxVerificationResult::Reject { tx_hash } => {
    self.shared.state().remove_from_known_txs(&tx_hash);
}
```

For `DeclaredWrongCycles`, `LowFeeRate`, or any rejection other than `Reject::Full`: no `TxVerificationResult::Reject` is emitted, `remove_from_known_txs` is never called, and the hash remains in `tx_filter` until TTL expiry while being permanently absent from `unknown_tx_hashes`.

**Concrete attack flow using `DeclaredWrongCycles`:**
1. Attacker connects as a P2P relay peer and sends `RelayTransactionHashes` advertising hash `T`.
2. Victim adds `T` to `unknown_tx_hashes` and sends `GetRelayTransactions`.
3. Attacker responds with `RelayTransactions` containing `T` with `declared_cycles` set to an incorrect value.
4. `mark_as_known_txs([T])` fires at line 76 — `T` is removed from `unknown_tx_hashes` and inserted into `tx_filter`.
5. Verification fails with `DeclaredWrongCycles`; no `TxVerificationResult::Reject` is sent; `remove_from_known_txs` is never called.
6. `T` is absent from `unknown_tx_hashes` and present in `tx_filter` until TTL expiry.
7. Any subsequent `RelayTransactionHashes` announcement of `T` from other peers is filtered out by `tx_filter.contains(&tx.hash())` at lines 49–55 of `transactions_process.rs`, preventing re-entry into `unknown_tx_hashes`.
8. The victim node will not proactively re-request `T` from any peer until TTL expiry.

**Existing guards are insufficient:** The filter at lines 49–55 checks `tx_filter` and `unknown_tx_hashes` to prevent duplicate requests, but once `mark_as_known_txs` fires prematurely, these same guards become the mechanism of suppression. The ban applied to the attacker peer does not undo the `tx_filter` insertion.

## Impact Explanation
This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** An attacker with modest resources (multiple IPs or Tor exits) can continuously suppress targeted transactions from a victim node's mempool by cycling through peer connections after each ban. At scale — targeting many transactions simultaneously across many victim nodes — this degrades transaction propagation across the network, delays block inclusion, and constitutes a low-cost denial-of-service against the mempool propagation layer. The `tx_filter` TTL window amplifies the suppression window beyond the immediate ban period.

## Likelihood Explanation
Medium. The attacker requires only an inbound P2P connection (CKB accepts public inbound connections by default) and knowledge of the target transaction hash (observable from the public mempool or by monitoring `RelayTransactionHashes` messages). No cryptographic material, privileged access, or victim mistakes are required. Sustaining suppression requires reconnecting after each ban, which is feasible with multiple IPs or Tor. The attack is most effective when the victim has few alternative peers that could re-announce the transaction before TTL expiry.

## Recommendation
Move `mark_as_known_txs` inside the spawned async task, called only after a successful `submit_remote_tx`:

```rust
self.relayer.shared.shared().async_handle().spawn(async move {
    for (tx, declared_cycles) in txs {
        match tx_pool.submit_remote_tx(tx.clone(), declared_cycles, peer).await {
            Ok(_) => {
                shared_state.mark_as_known_tx(tx.hash());
            }
            Err(e) => {
                error!("submit_tx error {}", e);
            }
        }
    }
});
```

Additionally, fix `remove_from_known_txs` to mirror the full inverse of `mark_as_known_txs` by also re-inserting the hash into `unknown_tx_hashes` when appropriate, so that on any submission failure the node can re-request the transaction from other peers.
