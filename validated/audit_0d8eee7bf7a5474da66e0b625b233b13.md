### Title
Missing Authority Check in `close_signature_set` Allows Anyone to Steal Rent from `SignatureSet` Accounts — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

---

### Summary

The `close_signature_set` instruction closes a `SignatureSet` account and sends its lamports to `sol_destination`. There is no check that `sol_destination` is the original payer/creator of the `SignatureSet`. Any unprivileged signer can call this instruction against any valid `PostedVaaV1`/`SignatureSet` pair and steal the rent from the original creator.

---

### Finding Description

In `CloseSignatureSet`, the account constraints are: [1](#0-0) 

- `sol_destination` is only required to be a `Signer` — no ownership or authority relationship to the `signature_set` or `posted_vaa` is enforced.
- `posted_vaa` is validated via PDA seeds and `has_one = signature_set`, confirming the pair is legitimate.
- `signature_set` is closed with `close = sol_destination`, sending all lamports to the caller.

The `SignatureSet` account is created during the legacy `verify_signatures` flow, where a `payer` funds the account: [2](#0-1) 

The `payer` pubkey is never stored in the `SignatureSet` account: [3](#0-2) 

And the legacy `post_vaa` stores the `signature_set` pubkey into `PostedVaaV1.info.signature_set`: [4](#0-3) 

This means the `has_one = signature_set` constraint only verifies the pair is consistent — it does not restrict *who* may close the account.

---

### Impact Explanation

An attacker can call `close_signature_set` for any `PostedVaaV1` account whose `signature_set` field is non-zero (i.e., created via the legacy `post_vaa` path), passing themselves as `sol_destination`. The `SignatureSet` account is closed and its rent lamports (~0.001–0.002 SOL per account, depending on guardian set size) are transferred to the attacker rather than the original payer. The original creator permanently loses their rent.

---

### Likelihood Explanation

The attack is trivially executable by any on-chain observer. The attacker only needs to:
1. Scan for `PostedVaaV1` accounts with a non-zero `signature_set` pubkey.
2. Submit a `close_signature_set` transaction with themselves as `sol_destination`.

No privileged access, leaked keys, or governance majority is required. The instruction is fully public.

---

### Recommendation

Store the original `payer` pubkey in the `SignatureSet` account at creation time, and add a constraint in `CloseSignatureSet` requiring `sol_destination` to match that stored payer:

```rust
// In SignatureSet state:
pub payer: Pubkey,

// In CloseSignatureSet constraints:
#[account(
    mut,
    close = sol_destination,
    constraint = signature_set.inner().payer == sol_destination.key()
        @ CoreBridgeError::InvalidFeeRecipient
)]
signature_set: Account<'info, AccountVariant<SignatureSet>>,
```

Alternatively, if permissionless closure is intentional, the lamports should be sent to the original payer stored in the account, not to the arbitrary `sol_destination` signer.

---

### Proof of Concept

```
1. Alice calls verify_signatures + post_vaa (legacy), paying rent for a SignatureSet account S.
   PostedVaaV1 P is created with P.signature_set = S.pubkey().

2. Attacker Bob observes P on-chain.

3. Bob calls close_signature_set {
       sol_destination: Bob,   // Bob is the signer
       posted_vaa: P,          // valid PDA, has_one check passes
       signature_set: S,       // matches P.signature_set
   }

4. Anchor's `close = sol_destination` transfers all lamports from S to Bob.
   Alice loses her rent (~0.001–0.002 SOL). Bob gains it.
``` [5](#0-4)
