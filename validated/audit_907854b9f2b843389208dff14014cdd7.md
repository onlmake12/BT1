### Title
Missing Duplicate `tx_hashes` Validation Causes Potential Panic/DoS in Light Client Server — (File: `util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

---

### Summary

`GetTransactionsProofProcess::execute()` does not validate that the `tx_hashes` list in an incoming `GetTransactionsProof` message is free of duplicates. The analogous handler `GetBlocksProofProcess::execute()` explicitly performs this check. When duplicate tx hashes that resolve to the same block are submitted, `CBMT::build_merkle_proof` is called with duplicate leaf indices and the subsequent `.expect()` panics, crashing the node process.

---

### Finding Description

In `get_blocks_proof.rs`, the server explicitly guards against duplicate block hashes before processing: [1](#0-0) 

```rust
let mut uniq = HashSet::new();
if !block_hashes.iter().chain([last_block_hash].iter()).all(|hash| uniq.insert(hash)) {
    return StatusCode::MalformedProtocolMessage.with_context("duplicate block hash exists");
}
```

No equivalent guard exists in `get_transactions_proof.rs`: [2](#0-1) 

When duplicate tx hashes are submitted, both copies pass the `partition` step into `found`, and both are pushed into `txs_and_tx_indices` for the same block with the same `tx_info.index`: [3](#0-2) 

This results in `CBMT::build_merkle_proof` being called with a duplicate index list: [4](#0-3) 

The `merkle_cbt` crate's `build_merkle_proof` returns `None` when indices are not strictly unique and sorted. The `.expect("build proof with verified inputs should be OK")` then panics, terminating the node process.

---

### Impact Explanation

A malicious peer connected to a CKB node running the light client server can crash the node by sending a single `GetTransactionsProof` message containing any valid on-chain tx hash repeated twice. The `.expect()` panic is unrecoverable and terminates the process, constituting a remotely-triggered denial of service.

---

### Likelihood Explanation

The light client protocol is reachable by any unprivileged peer. The `GetTransactionsProof` message is a standard protocol message handled by `LightClientProtocol::try_process`: [6](#0-5) 

No authentication or privilege is required. Any peer that knows a single valid on-chain transaction hash can trigger the crash by sending it twice in the `tx_hashes` field. The attack is trivially constructible and requires no special knowledge beyond a single valid tx hash, which is publicly observable on-chain.

---

### Recommendation

Add a duplicate-check guard at the start of `GetTransactionsProofProcess::execute()`, mirroring the pattern already used in `GetBlocksProofProcess::execute()`:

```rust
let tx_hashes: Vec<_> = self.message.tx_hashes().to_entity().into_iter().collect();
let mut uniq = HashSet::new();
if !tx_hashes.iter().all(|hash| uniq.insert(hash)) {
    return StatusCode::MalformedProtocolMessage.with_context("duplicate tx hash exists");
}
```

This is consistent with how `get_blocks_proof.rs` already handles the analogous case. [7](#0-6) 

---

### Proof of Concept

1. Identify any valid on-chain transaction hash `H` (publicly visible in any block).
2. Connect to a CKB node with the light client server enabled.
3. Send a `GetTransactionsProof` message with `tx_hashes = [H, H]` and a valid `last_hash`.
4. The server resolves both copies of `H` to the same block and same index, builds `txs_and_tx_indices = [(tx, idx), (tx, idx)]`.
5. `CBMT::build_merkle_proof(leaves, &[idx, idx])` returns `None`.
6. `.expect("build proof with verified inputs should be OK")` panics → node process terminates. [4](#0-3)
