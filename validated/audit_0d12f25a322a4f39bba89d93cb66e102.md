### Title
Truncated Price ID Verification (8-byte prefix only) Allows Wrong Feed Substitution in `executeCallback` - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol` stores only the first 8 bytes of each 32-byte price ID at request time and verifies only those 8 bytes at callback execution time. Any caller can pass a `priceIds` array whose elements share the same 8-byte prefix as the originally requested IDs but differ in the remaining 24 bytes, bypassing the identity check and delivering wrong price data to the consumer.

### Finding Description

In `requestPriceUpdatesWithCallback`, the contract truncates each 32-byte price ID to its first 8 bytes and stores those as `bytes8[] priceIdPrefixes`:

```solidity
// Copy only the first 8 bytes of each price ID to storage
for (uint8 i = 0; i < priceIds.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly {
        prefix := priceId   // truncates to first 8 bytes
    }
    req.priceIdPrefixes[i] = prefix;
}
``` [1](#0-0) 

In `executeCallback`, the verification re-extracts only the first 8 bytes of the caller-supplied `priceIds` and compares them to the stored prefixes:

```solidity
for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly {
        prefix := priceId   // again only first 8 bytes
    }
    if (prefix != req.priceIdPrefixes[i]) {
        revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
    }
}
``` [2](#0-1) 

After this weak check passes, the full caller-supplied `priceIds` are forwarded verbatim to `parsePriceFeedUpdates`:

```solidity
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(
    updateData,
    priceIds,          // attacker-controlled, only 8 bytes were verified
    SafeCast.toUint64(req.publishTime),
    SafeCast.toUint64(req.publishTime)
);
``` [3](#0-2) 

The `Request` struct explicitly documents this design choice as a gas optimization, but does not address the security implication: [4](#0-3) 

### Impact Explanation

A malicious provider (or any caller after the exclusivity period expires) can call `executeCallback` with:
- A `priceIds` array where each element shares the same first 8 bytes as the originally requested price IDs but differs in the remaining 24 bytes (i.e., a different real Pyth price feed whose ID happens to share the same 8-byte prefix).
- Valid Wormhole-signed `updateData` for those substitute price IDs.

The prefix check passes, `parsePriceFeedUpdates` succeeds with the substitute IDs, and the consumer's `_echoCallback` receives price data for the **wrong feeds**. A consumer contract that requested BTC/USD could receive price data for a different, potentially manipulable asset, leading to incorrect financial decisions (e.g., wrong collateral valuation, incorrect liquidation thresholds, mispriced derivatives).

### Likelihood Explanation

- `executeCallback` is `external` with no caller restriction beyond the exclusivity window. After `exclusivityPeriodSeconds` elapses, any address can call it. [5](#0-4) 
- Pyth currently publishes hundreds of price feeds. With only 8 bytes (64 bits) of prefix checked, the probability of a natural prefix collision among existing feeds is low but non-zero, and grows as the feed catalog expands.
- An attacker does not need to forge Wormhole signatures — they only need to find one real Pyth price feed whose ID shares the same first 8 bytes as a targeted feed, then supply valid signed update data for it.
- The attack is permissionless and requires no privileged access.

### Recommendation

Store and verify the **full 32-byte price ID** for each request rather than an 8-byte prefix. If gas cost is a concern, use a single `keccak256` hash of the entire `priceIds` array instead of per-element prefix storage:

```solidity
// At request time:
req.priceIdsHash = keccak256(abi.encodePacked(priceIds));

// At callback time:
require(keccak256(abi.encodePacked(priceIds)) == req.priceIdsHash, "Price IDs mismatch");
```

This provides full 32-byte identity verification at the cost of a single hash per call, using less storage than the current `bytes8[]` array.

### Proof of Concept

1. Alice calls `requestPriceUpdatesWithCallback` requesting price feed `0xABCDEF1234567890<24 more bytes for BTC/USD>`. The contract stores prefix `0xABCDEF1234567890`.
2. Attacker finds Pyth feed `0xABCDEF1234567890<different 24 bytes for a low-liquidity asset>` — same 8-byte prefix.
3. After the exclusivity period, attacker calls `executeCallback(sequenceNumber, validUpdateDataForSubstituteId, [substituteId])`.
4. Prefix check: `bytes8(substituteId) == 0xABCDEF1234567890` ✓ — passes.
5. `parsePriceFeedUpdates` is called with `substituteId` and valid signed data — succeeds.
6. Consumer's `_echoCallback` receives price data for the substitute (wrong) feed. [6](#0-5)
