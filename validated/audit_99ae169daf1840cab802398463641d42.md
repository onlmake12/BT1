Audit Report

## Title
Unbounded Memory Allocation via Duplicate Indexes and Missing Byte-Size Cap in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` accepts up to 32767 peer-supplied transaction indexes with no deduplication and no byte-size cap on the assembled response. An unprivileged peer can send all 32767 indexes pointing at the same transaction, forcing the node to clone it 32767 times into a single `Vec` before any send occurs. Combined with the rate limiter's per-request (not per-byte) granularity, this enables sustained memory exhaustion sufficient to crash the node.

## Finding Description

**Defect 1 — Off-by-one in the count guard**

The guard at line 37 uses strict `>`: [1](#0-0) 

`MAX_RELAY_TXS_NUM_PER_BATCH = 32767`, so a message with exactly 32767 entries passes (`32767 > 32767 == false`). Note that `GetTransactionsProcess` uses the same `>` pattern, so this is a consistent (if permissive) design choice rather than an isolated typo. [2](#0-1) 

**Defect 2 — No deduplication of indexes (primary defect)**

Every entry in the peer-supplied index list is looked up and cloned unconditionally: [3](#0-2) 

Sending `indexes = [0u32; 32767]` produces 32767 `.cloned()` copies of `block.transactions()[0]` (the coinbase) in a single `Vec`. By contrast, `GetTransactionsProcess` explicitly rejects duplicates via a `HashSet` comparison: [4](#0-3) 

And `GetBlockProposalProcess` deduplicates via a `HashSet` and rejects if counts differ: [5](#0-4) 

**Defect 3 — No `MAX_RELAY_TXS_BYTES_PER_BATCH` cap on the response (primary defect)**

The entire `transactions` vec is packed into one message and sent with no byte-size check: [6](#0-5) 

Both analogous handlers enforce a 1 MiB batch cap. `GetTransactionsProcess`: [7](#0-6) 

`GetBlockProposalProcess`: [8](#0-7) 

**Exploit path:**
1. Attacker establishes a TCP connection to the relay protocol (no PoW, no key required).
2. Attacker obtains any stored block hash (publicly available from header sync or a block explorer).
3. Attacker sends `GetBlockTransactions { block_hash: <stored>, indexes: [0u32; 32767], uncle_indexes: [] }`.
4. The count guard passes (32767 is not `> 32767`).
5. `execute()` clones the coinbase transaction 32767 times into a `Vec` with no byte-size check.
6. The entire vec is serialized into one `BlockTransactions` message and sent.
7. The rate limiter (30 req/s per peer, keyed by `(PeerIndex, message_type)`) does not bound per-request allocation; the attacker repeats at full rate. [9](#0-8) 

## Impact Explanation

Peak allocation per request: `32767 × coinbase_tx_size`. With a typical mainnet coinbase of ~500 bytes: ~16 MB per request. At 30 req/s from a single peer: ~480 MB/s sustained allocation. With up to `MAX_RELAY_PEERS = 128` peers, the aggregate allocation rate is sufficient to exhaust available memory and crash the node. [10](#0-9) 

In the worst case (large coinbase approaching `max_block_bytes`), a single request can allocate tens of gigabytes, causing an immediate OOM crash. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires only a TCP connection to the relay protocol. No proof-of-work, no cryptographic key, and no privileged role is needed. The attacker only needs a valid stored block hash, which is publicly available. The malformed message passes all existing structural and count checks. The rate limiter provides partial mitigation but does not bound per-request memory. The attack is repeatable and can be issued simultaneously from multiple peers.

## Recommendation

1. Change the count guard to `>=` to enforce the limit inclusively:
   ```rust
   if get_block_transactions.indexes().len() >= MAX_RELAY_TXS_NUM_PER_BATCH {
   ```
2. Deduplicate indexes before lookup, and reject the message if duplicates are detected (matching the pattern in `GetTransactionsProcess` and `GetBlockProposalProcess`):
   ```rust
   let indexes: HashSet<u32> = self.message.indexes().iter().map(Into::into).collect();
   if indexes.len() != self.message.indexes().len() {
       return StatusCode::RequestDuplicate.with_context("Request duplicate index");
   }
   ```
3. Add a `MAX_RELAY_TXS_BYTES_PER_BATCH` byte-size accumulator and split or truncate the response, matching the batching pattern in `GetTransactionsProcess::execute` and `GetBlockProposalProcess::execute`.

## Proof of Concept

```rust
// Unit test pseudocode
let stored_block = /* any block stored by the node, e.g., genesis */;
let block_hash = stored_block.hash();
// coinbase is always at index 0
let indexes = vec![0u32; 32767]; // 32767 duplicates, passes the > guard

let msg = GetBlockTransactions { block_hash, indexes, uncle_indexes: vec![] };
// node.execute(msg) allocates 32767 * coinbase_size bytes before returning
// With a 500-byte coinbase: ~16 MB per call
// With a 597 KB coinbase (max_block_bytes): ~19.5 GB per call → OOM crash
// assert: no truncation occurs, full vec is serialized into one message
```

Repeating at 30 req/s (rate limit) with a 500-byte coinbase sustains ~480 MB/s allocation from a single peer, exhausting node memory within seconds.
