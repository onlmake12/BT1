### Title
Missing Payload Timestamp Freshness Check in `verifyUpdate()` Allows Stale Price Replay - (File: lazer/contracts/evm/src/PythLazer.sol)

---

### Summary
`PythLazer.verifyUpdate()` verifies the ECDSA signature of a Lazer price update and checks that the signer key has not expired, but it performs **no check that the price timestamp embedded in the payload is sufficiently recent relative to `block.timestamp`**. An unprivileged transaction sender can replay an arbitrarily old but validly-signed Lazer update, causing consuming DeFi contracts to act on stale prices.

---

### Finding Description

`verifyUpdate()` in `PythLazer.sol` performs exactly four checks:

1. Fee sufficiency
2. EVM format magic (`706910618`)
3. ECDSA signature recovery
4. Signer expiry: `block.timestamp < trustedSignerToExpiresAtMapping[signer]` [1](#0-0) 

After these checks pass, the raw `payload` bytes are returned to the caller. The payload's internal structure (as parsed by `PythLazerLib.parsePayloadHeader`) begins with a `uint64 timestamp` field (in microseconds) immediately after the `PAYLOAD_FORMAT_MAGIC`: [2](#0-1) 

`verifyUpdate()` never reads or validates this timestamp against `block.timestamp`. There is no `maxAge` parameter, no `verifyUpdateNoOlderThan()` variant, and no internal staleness guard. The signer-expiry check (`block.timestamp < expiresAt`) only confirms the signing key is currently authorized — it says nothing about when the price data was generated.

The Cardano integration docs explicitly acknowledge this gap: *"It does not enforce freshness of the update, nor does it disallow verifying the same update multiple times."* [3](#0-2) 

The EVM integration guide contains **no equivalent warning**, and its example code only checks monotonicity (`_timestamp > timestamp`), not freshness against `block.timestamp`: [4](#0-3) 

This is structurally opposite to Pyth Core, which provides both an unsafe variant and a freshness-enforced variant (`getPriceNoOlderThan`) as first-class API: [5](#0-4) 

Pyth Lazer's EVM contract exposes only the unsafe path with no safe alternative.

---

### Impact Explanation

Any EVM DeFi protocol (lending, perps, derivatives) that calls `verifyUpdate()` and does not independently enforce a freshness window on the returned payload timestamp is vulnerable to **adversarial price selection**:

1. An attacker observes a historical Lazer update signed by a currently-valid trusted signer at a price favorable to them (e.g., BTC = $100,000 from one hour ago, before a crash to $90,000).
2. The attacker submits this old update to the protocol's `updatePrice()` function.
3. `verifyUpdate()` accepts it: fee ✓, magic ✓, signature ✓, signer not expired ✓.
4. The protocol uses the stale $100,000 price for collateral valuation, loan origination, or liquidation decisions.
5. The attacker borrows against inflated collateral or avoids a liquidation that should have triggered.

This is the direct EVM analog of the external report's findings #2 (missing timestamp freshness check) and #3 (oracle not re-checked at action time): the oracle verification contract accepts stale data, and consuming protocols have no on-chain safety net from the Pyth layer.

---

### Likelihood Explanation

**Medium-High.** Pyth Lazer is explicitly marketed for high-frequency DeFi use cases (HFT, MEV, perps, lending). The EVM integration guide does not warn that `verifyUpdate()` provides no freshness guarantee. The example consumer code only checks monotonicity. Most integrators will assume that a function named `verifyUpdate` provides complete validity guarantees — including freshness — analogous to how Pyth Core's `getPriceNoOlderThan` works. The attack requires no privileged access: any transaction sender can submit a historical signed update.

---

### Recommendation

Add a freshness-enforced variant to `PythLazer.sol`, analogous to Pyth Core's `getPriceNoOlderThan`:

```solidity
function verifyUpdateNoOlderThan(
    bytes calldata update,
    uint64 maxAgeSeconds
) external payable returns (bytes calldata payload, address signer) {
    (payload, signer) = this.verifyUpdate{value: msg.value}(update);
    // Parse timestamp from payload (bytes 4–12 after PAYLOAD_FORMAT_MAGIC)
    uint64 payloadTimestampUs = uint64(bytes8(payload[4:12]));
    uint64 payloadTimestampSecs = payloadTimestampUs / 1_000_000;
    require(
        block.timestamp <= payloadTimestampSecs + maxAgeSeconds,
        "stale price update"
    );
}
```

Additionally, update the EVM integration guide to include the same freshness warning present in the Cardano guide, and recommend that consumers always validate `_timestamp` against `block.timestamp` with an explicit maximum age.

### Proof of Concept

```
T=0:      Lazer distributor signs payload: {timestamp: T, BTC/USD: 100_000, ...}
          Signer key valid until T+30days.

T+1hr:    BTC/USD market price drops to 90_000.
          Attacker holds the old signed update from T=0.

T+1hr:    Attacker calls lendingProtocol.borrow(oldSignedUpdate, collateral=1BTC, amount=95_000_USDC)
          Inside borrow():
            pythLazer.verifyUpdate{value: fee}(oldSignedUpdate)
            → fee ✓, magic ✓, sig ✓, signer not expired ✓  → returns payload
            parsePayloadHeader(payload) → timestamp=T, price=100_000
            // No freshness check: 100_000 > 90_000 actual price
            require(amount <= price * ltv / 100)  // 95_000 <= 100_000 * 0.97 ✓
          Attacker receives 95_000 USDC against collateral worth only 90_000.
          Protocol is immediately undercollateralized.
```

The root cause is entirely within `lazer/contracts/evm/src/PythLazer.sol` lines 70–106: `verifyUpdate()` never reads or bounds-checks the payload timestamp against `block.timestamp`. [1](#0-0)
