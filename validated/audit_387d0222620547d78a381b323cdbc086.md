Audit Report

## Title
Minimum Fee Rate Admission Check Uses `tx_size` Instead of Transaction Weight — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` gates pool admission using only the serialized byte size (`tx_size`) as the fee-rate denominator. After script verification, every `TxEntry` computes its fee rate using `get_transaction_weight(size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For compute-heavy transactions where `cycles * DEFAULT_BYTES_PER_CYCLES > tx_size`, the admission threshold is evaluated against a strictly smaller denominator than the one used for all subsequent pool accounting (sorting, eviction, pool-full rejection), allowing transactions with a true fee rate materially below `min_fee_rate` to enter and persist in the pool.

## Finding Description
**Root cause:** `check_tx_fee` (`tx-pool/src/util.rs`, lines 42–52) explicitly acknowledges the discrepancy in a comment and computes:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
```

This is the **sole admission gate**. It is called during `pre_check` (`tx-pool/src/process.rs`, line 289), before script verification runs and before actual cycles are known.

After verification completes, `TxEntry::new` is constructed at `tx-pool/src/process.rs` line 751 with `verified.cycles`. From that point, all fee-rate comparisons use the weight-based `fee_rate()` method (`tx-pool/src/component/entry.rs`, lines 114–118):

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
```

`get_transaction_weight` (`util/types/src/core/tx_pool.rs`, lines 298–303) is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

There is **no second fee-rate gate** between line 751 (`TxEntry::new`) and line 753 (`submit_entry`). Eviction via `EvictKey` and pool-full rejection via `limit_size` (`tx-pool/src/pool.rs`, lines 290–328) both call `entry.fee_rate()` (weight-based), but these operate on already-admitted entries and do not prevent admission.

**Exploit flow:**
1. Attacker deploys a compute-heavy script (permissionless on CKB).
2. Attacker creates cells locked by this script and submits transactions via `send_transaction` RPC.
3. Each transaction pays fee = `min_fee_rate * tx_size / 1000` shannons — just enough to pass `check_tx_fee`.
4. Script verification runs, consuming many cycles; `verified.cycles` is large.
5. `TxEntry` is constructed with the real cycles; its `fee_rate()` (weight-based) is far below `min_fee_rate`.
6. The entry is admitted with no further fee-rate check.
7. Repeated submissions fill the pool with below-minimum-fee-rate entries, displacing legitimate transactions.

**Why existing checks fail:** `limit_size` evicts by lowest weight-based fee rate, but the attacker's transactions are the ones with the lowest fee rate — they will evict legitimate transactions first if the pool fills. The `check_tx_fee` pre-check is the only admission gate and it uses the wrong denominator.

## Impact Explanation
This matches the allowed CKB bounty impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. The tx-pool (default 180 MB) can be saturated with transactions whose true fee rate is far below `min_fee_rate`, displacing legitimate traffic. The economic spam-protection threshold is silently bypassed. The cost to the attacker scales only with `tx_size`, not with the actual weight, making the attack cheap to sustain at scale.

## Likelihood Explanation
The exploit path is the standard `send_transaction` RPC, reachable by any unprivileged peer or local caller. No special role, key, or majority hashpower is required. Deploying a compute-heavy script is permissionless on CKB. The bypass is deterministic: for any transaction where `cycles * DEFAULT_BYTES_PER_CYCLES > tx_size`, the gap exists. The attacker can scale cycles up to `max_tx_verify_cycles = 70,000,000` to maximize the ratio `weight / tx_size`. The attack is repeatable across many transactions.

## Recommendation
After script verification completes and `verified.cycles` is known (at `tx-pool/src/process.rs`, between lines 751 and 753), add a second fee-rate check using the actual weight:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

The existing `check_tx_fee` call can remain as a cheap pre-verification filter (using `tx_size`) to avoid running expensive script verification on obviously-underpaying transactions. The weight-based check should be added as a mandatory post-verification gate before `submit_entry`.

## Proof of Concept
**Parameters:**
- `min_fee_rate` = 1,000 shannons/KW (default)
- `DEFAULT_BYTES_PER_CYCLES` = 0.000_170_571_4
- Transaction serialized size = 100 bytes
- Transaction cycles = 1,000,000

**Weight:** `max(100, 1_000_000 × 0.000_170_571_4) = max(100, 170) = 170`

**Admission check (uses tx_size = 100):**
`min_fee = 1000 × 100 / 1000 = 100 shannons` → attacker pays exactly 100 shannons → **passes** `check_tx_fee`

**True fee rate (uses weight = 170):**
`actual_fee_rate = 100 × 1000 / 170 ≈ 588 shannons/KW` — **41% below** `min_fee_rate`

Scaling to `cycles = 10,000,000` → `weight = 1,705`:
`actual_fee_rate = 100 × 1000 / 1705 ≈ 59 shannons/KW` — **94% below** `min_fee_rate`, while still passing admission.

**Reproducible steps:**
1. Deploy a script that loops for ~1,000,000 cycles on CKB devnet.
2. Create cells locked by this script.
3. Submit transactions spending those cells via `send_transaction` RPC, paying fee = `min_fee_rate × tx_size / 1000` shannons.
4. Observe transactions enter the pool; verify via `get_pool_tx_detail_info` that the entry's fee rate (weight-based) is below `min_fee_rate`.
