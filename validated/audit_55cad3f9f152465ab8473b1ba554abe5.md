Audit Report

## Title
Missing Type Script `hash_type` Validation in `ScriptHashTypeVerifier` — (File: verification/src/transaction_verifier.rs)

## Summary
`ScriptHashTypeVerifier::verify()` iterates over every transaction output but validates only the lock script's `hash_type`, leaving type script `hash_type` bytes entirely unchecked. A transaction with a valid lock `hash_type` but a disallowed type script `hash_type` passes `NonContextualTransactionVerifier`, enters contextual verification, fails with `TransactionError::Compatible` (which is not `is_malformed_tx()`), and the submitting peer is never banned — allowing indefinite free resubmission against the same unspent UTXOs.

## Finding Description
`ScriptHashTypeVerifier::verify()` at lines 796–814 of `verification/src/transaction_verifier.rs` loops over outputs and calls only `output.lock().hash_type()`:

```rust
for output in self.transaction.outputs() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: output.lock().hash_type(),
        }).into());
    }
}
```

`output.type_().to_opt()` is never consulted. The error variant doc comments confirm the scope: `ScriptHashTypeNotPermitted` reads "The **lock** script hash type is not permitted" and `InvalidScriptHashType` reads "the **lock** Script hash_type field is invalid" — type scripts are explicitly absent from both.

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` contains only `{0, 1, 2, 4}`. A type script carrying byte `0x03` or `0x05` passes `ScriptHashTypeVerifier` unchallenged.

`NonContextualTransactionVerifier::verify()` (lines 94–102) calls `self.script_hash_type.verify()` as its final gate. `NonContextualBlockTxsVerifier` propagates the same gap to block processing.

When such a transaction reaches contextual verification, the script verifier encounters the unknown hash_type and returns `TransactionError::Compatible { feature: "..." }` — defined at lines 194–201 of `util/types/src/core/error.rs` as "the feature X is used in current transaction, but not enabled in current chain." `Compatible` is explicitly excluded from `is_malformed_tx()` (line 260), while `ScriptHashTypeNotPermitted` and `InvalidScriptHashType` are included (lines 253–254) — but those are only reachable for lock scripts.

In `tx-pool/src/process.rs` lines 514–515, `ban_malformed` fires only when `reject.is_malformed_tx()` is true. The same guard at lines 323–329 covers the `non_contextual_verify` path. Because `Compatible` is not malformed, no ban is issued.

## Impact Explanation
**High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker holding any valid UTXO can craft transactions that pass all non-contextual checks, consume the full `pre_check` + UTXO resolution + `verify_rtx` pipeline on each submission, and are rejected without banning the source. Because the transaction is rejected rather than committed, the input UTXOs are never consumed. The attacker can resubmit the identical transaction in a tight loop, saturating the verify queue and delaying legitimate transaction processing at essentially zero cost.

## Likelihood Explanation
No special privilege, key material, or hash power is required. Any caller with RPC access (`send_transaction`) or P2P relay access can set a single byte field in a type script to a value absent from `ENABLED_SCRIPT_HASH_TYPE` (e.g., `0x03` or `0x05`). The attacker's CKB is never spent on rejection. The node cannot distinguish this traffic from legitimate submissions and cannot ban the source. The attack is trivially automatable.

## Recommendation
Extend `ScriptHashTypeVerifier::verify()` to also validate `output.type_().to_opt()` when a type script is present, applying the same two-branch check already applied to lock scripts:

```rust
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }
    } else {
        return Err(TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }.into());
    }
}
```

No new error variants are needed. Both `ScriptHashTypeNotPermitted` and `InvalidScriptHashType` are already `is_malformed_tx() = true`, ensuring the peer is banned on detection. Update the struct comment at line 70 and the `NonContextualTransactionVerifier` doc comment to reflect that both lock and type script hash types are validated.

## Proof of Concept
1. Obtain any valid UTXO (any amount of CKB).
2. Construct a `TransactionView` spending that UTXO with one output whose lock script uses `hash_type = 0x01` (Type, enabled) and whose type script uses `hash_type = 0x03` (absent from `ENABLED_SCRIPT_HASH_TYPE`).
3. Submit via `send_transaction` RPC or P2P relay.
4. Observe: `NonContextualTransactionVerifier::verify()` returns `Ok(())` — `ScriptHashTypeVerifier` only checked the lock script.
5. Observe: contextual verification returns `TransactionError::Compatible`; `is_malformed_tx()` returns `false`; no ban is issued.
6. Resubmit the identical transaction. The UTXO is still unspent; the node re-executes the full pipeline on each submission.
7. Automate submissions in a loop to saturate the verify queue and delay processing of legitimate transactions.
