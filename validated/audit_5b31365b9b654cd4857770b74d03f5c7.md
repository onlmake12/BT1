Audit Report

## Title
Off-by-one in `check_purge()` eviction condition allows a single remote peer to permanently freeze peer discovery — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
`check_purge()` uses `addrs.len() > 4` in Pass 2 eviction. An attacker with one P2P connection can advertise exactly 4 addresses per `/16` subnet across 4096 distinct subnets, filling `ADDR_COUNT_LIMIT=16384` with entries that are all connectable (Pass 1 yields nothing) and all in groups of exactly 4 (Pass 2 yields nothing). Every subsequent `add_addr()` call returns `Err(EvictionFailed)`, silently dropped at `debug!` level, permanently freezing peer discovery.

## Finding Description
`check_purge()` triggers when `addr_manager.count() >= ADDR_COUNT_LIMIT (16384)`.

**Pass 1** (`peer_store_impl.rs:341-355`): Collects addresses where `!is_connectable(now_ms)`. Addresses inserted via `add_addr()` are created with `last_connected_at_ms=0` and `attempts_count=0`. In `is_connectable()` (`types.rs:89-105`), the two non-connectable conditions require `attempts_count >= 3` or `attempts_count >= 10` — neither applies to fresh addresses. Pass 1 yields zero candidates, nothing is removed.

**Pass 2** (`peer_store_impl.rs:358-393`): Groups addresses by `/16` IPv4 prefix (`Group::IP4([bits[0], bits[1]])`), sorts by descending group size, takes the top `len/2` groups, and for each applies:
```rust
if addrs.len() > 4 { Some(...evict 2...) } else { None }
```
With exactly 4 addresses per group across 4096 groups: `4 > 4` is `false` for every group → zero candidates → `return Err(PeerStoreError::EvictionFailed.into())`.

The error propagates via `check_purge()?` in `add_addr()` and is caught in `add_new_addrs()` with only a `debug!` log — no ban, no disconnect, no alert. The peer store is permanently frozen.

## Impact Explanation
Peer discovery is permanently frozen for the targeted node. The node cannot add any new peer addresses to its store. After existing connections drop, the node loses the ability to find replacement peers and becomes isolated from the network. This constitutes a remotely-triggerable degradation of network connectivity matching the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
Requires only one accepted inbound or outbound P2P connection. The discovery protocol allows up to `MAX_ADDR_TO_SEND=1000` items × `MAX_ADDRS=3` addresses per `Nodes` message = 3000 addresses per message. Approximately 6 messages suffice to deliver 4096 × 4 = 16384 addresses. Addresses only need to pass `is_reachable()` (a local IP-range check), not actual connectivity. No PoW, no privilege, no Sybil attack required. The attack is repeatable after node restart.

## Recommendation
Change the eviction condition at `peer_store_impl.rs:378` from:
```rust
if addrs.len() > 4 {
```
to:
```rust
if addrs.len() >= 4 {
```
Additionally, consider per-session and per-source-subnet rate limiting in `add_new_addrs()` to bound how many addresses a single peer can contribute to the store.

## Proof of Concept
```rust
// Fill store with 4096 groups × 4 addresses each (all connectable)
for subnet in 0u16..4096 {
    for host in 1u8..=4 {
        let ip = format!("{}.{}.1.{}", subnet >> 8, subnet & 0xff, host);
        let addr = format!("/ip4/{}/tcp/8115/p2p/Qm...", ip).parse().unwrap();
        peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    }
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// 16385th add_addr must fail with EvictionFailed
let extra = "/ip4/200.0.1.1/tcp/8115/p2p/Qm...".parse().unwrap();
let result = peer_store.add_addr(extra, Flags::COMPATIBILITY);
assert!(matches!(result, Err(Error::PeerStore(PeerStoreError::EvictionFailed))));
assert_eq!(peer_store.addr_manager().count(), 16384); // store frozen
```

        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;

            return false;
        }
        true
    }
