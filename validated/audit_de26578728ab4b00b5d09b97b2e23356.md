### Title
`total_tx_size`/`total_tx_cycles` Inflated When Cell-Dep Ancestor Eviction Occurs Inside `add_entry` — (`File: tx-pool/src/component/pool_map.rs`)

### Summary

In `PoolMap::add_entry`, the new aggregate accounting values (`total_tx_size`, `total_tx_cycles`) are computed **before** any in-pool evictions that may occur inside `check_and_record_ancestors`. When that helper evicts cell-dep-referencing ancestor transactions, it correctly decrements the running totals via `update_stat_for_remove_tx`. However, `add_entry` then **overwrites** those correctly-decremented totals with the stale pre-eviction snapshot, permanently inflating `total_tx_size` and `total_tx_cycles` by the aggregate size/cycles of every evicted entry.

### Finding Description

`PoolMap::add_entry` (pool_map.rs lines 200–221) follows this sequence:

1. **Pre-compute** the post-add totals:
   ```rust
   let (total_tx_size, total_tx_cycles) =
       self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
   // total_tx_size = self.total_tx_size + entry.size  (snapshot taken here)
   ```
2. **Potentially evict** cell-dep-referencing ancestors inside `check_and_record_ancestors` (lines 588–640). When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, the function calls `remove_entry_and_descendants` for each eviction candidate, which internally calls `update_stat_for_remove_tx`, correctly decrementing `self.total_tx_size` and `self.total_tx_cycles`.
3. **Overwrite** the now-correct totals with the stale snapshot:
   ```rust
   self.total_tx_size = total_tx_size;   // stale: original + entry.size
   self.total_tx_cycles = total_tx_cycles; // stale: original + entry.cycles
   ```

If evictions occurred, the correct post-operation value is `(original − evicted_size) + entry.size`, but the stored value is `original + entry.size`. The pool's accounting is inflated by `evicted_size` (and analogously for cycles) for every such eviction event. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

`total_tx_size` drives two critical pool-admission decisions:

1. **`limit_size` eviction loop** (pool.rs:298): `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated counter causes the loop to evict additional legitimate pending/proposed transactions that would otherwise remain in the pool.
2. **`updated_stat_for_add_tx` overflow guard** (pool_map.rs:716): new submissions are rejected with `Reject::Full` if the inflated counter would overflow. An attacker can push the counter high enough to block all subsequent submissions.

Each successful trigger of the eviction path inflates the counter by the aggregate byte-size of the evicted entries. Because `limit_size` is called immediately after `_submit_entry` in `submit_entry` (process.rs:151), the inflated counter causes a cascade of additional evictions of other users' transactions in the same call. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

The eviction path inside `check_and_record_ancestors` is reachable by any unprivileged `send_transaction` RPC caller. The attacker must craft a transaction whose in-pool ancestor count exceeds `max_ancestors_count` (default: 25) but where enough of those ancestors are cell-dep-referencing parents (`cell_ref_parents`) that evicting them brings the count back within the limit. This requires:

1. Seeding the pool with a set of transactions that share a common cell dep (a live on-chain cell used as `cell_deps` by multiple pool transactions).
2. Submitting a new transaction whose ancestor graph spans those cell-dep-referencing transactions, pushing the ancestor count just over the limit.

This is a deliberate, repeatable, fee-paying action available to any node peer or RPC caller. No privileged access, key material, or majority hashpower is required. [6](#0-5) 

### Recommendation

Move the accounting snapshot **after** all evictions complete, so it reflects the true post-eviction state before adding the new entry's contribution:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Perform evictions first, THEN compute the new totals.
    evicts = self.check_and_record_ancestors(&mut entry)?;
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, recompute the totals from scratch after evictions using the existing `recompute_total_stat` helper, then add the new entry's contribution.

### Proof of Concept

```
Initial state: total_tx_size = 1000, max_tx_pool_size = 1200

Step 1: Pool contains 10 transactions each of size 100, some sharing a cell dep.
        total_tx_size = 1000 (correct).

Step 2: Attacker submits tx_new (size=50) whose ancestor count = max_ancestors_count+1,
        with 1 cell_ref_parent of size=200 eligible for eviction.

Step 3: add_entry executes:
        - updated_stat_for_add_tx(50, ...) → snapshot: total_tx_size_snap = 1050
        - check_and_record_ancestors evicts the cell_ref_parent (size=200):
            update_stat_for_remove_tx(200, ...) → self.total_tx_size = 800  (correct)
        - insert_entry(tx_new)
        - self.total_tx_size = 1050  ← stale snapshot overwrites correct value

Step 4: Actual pool contents have total size = 800 + 50 = 850.
        Reported total_tx_size = 1050.
        Inflation = 200 (size of evicted entry).

Step 5: limit_size sees 1050 > 1200? No, but repeating the attack accumulates inflation.
        After N iterations, total_tx_size can be pushed above max_tx_pool_size,
        triggering eviction of legitimate transactions or rejection of new submissions.
``` [7](#0-6) [8](#0-7)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
