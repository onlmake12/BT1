### Title
Discovery GetNodes Bandwidth Amplification via Unbounded Connection Cycling — (`network/src/protocols/discovery/mod.rs`)

---

### Summary

A remote unprivileged peer can repeatedly connect, send one `GetNodes` message, receive a ~50 KB `Nodes` response (up to 1000 addresses), disconnect, and immediately reconnect — with no IP-based rate limiting or banning — causing sustained outbound bandwidth exhaustion on the victim node.

---

### Finding Description

The `received_get_nodes` guard at line 110 prevents a second `GetNodes` from the **same session**, but the `misbehave()` implementation unconditionally returns `MisbehaveResult::Disconnect` without ever banning the connecting IP: [1](#0-0) [2](#0-1) 

This means the attacker can immediately reconnect after disconnection. Each new session starts with `received_get_nodes: false`: [3](#0-2) 

On the first `GetNodes` per session, the node calls `get_random(2500, required_flags)` and then caps the response at `min(MAX_ADDR_TO_SEND, count as usize)`. With `count = u32::MAX`, this is always 1000 addresses: [4](#0-3) 

`MAX_ADDR_TO_SEND = 1000` and `MAX_ADDRS = 3` are defined at the top of the file: [5](#0-4) 

The `fetch_random_addrs` implementation in the peer store performs a random scan over all stored addresses on every call, with no caching or rate limiting: [6](#0-5) 

The `accept_peer` function enforces a concurrent session count limit (`max_inbound`) but imposes no per-IP connection rate limit: [7](#0-6) 

---

### Impact Explanation

- **Amplification ratio**: ~50-byte `GetNodes` → ~50 KB `Nodes` response ≈ 1000× per round-trip.
- **Attack loop**: connect → `GetNodes` → receive 1000 addresses → disconnect → repeat. No cooldown, no IP ban.
- **Victim effect**: Sustained outbound bandwidth consumption. With rapid cycling, the victim's uplink can be saturated, degrading block and transaction relay to legitimate peers.
- **Limitation**: This is not a true reflection attack (TCP prevents IP spoofing); the attacker receives the amplified data. The practical amplification is therefore self-limiting — the attacker must absorb the bandwidth they generate. This reduces the severity from "Critical" to **High**.

---

### Likelihood Explanation

The attack path requires only a TCP connection to the victim's P2P port (default 8115), which is publicly reachable on all mainnet nodes. No authentication, PoW, or privileged role is required. The cycling loop is trivially scriptable. The absence of any IP-level rate limiting or banning in `misbehave` makes this continuously exploitable.

---

### Recommendation

1. **Ban the IP** (not just disconnect) after `DuplicateGetNodes` or after N `GetNodes` requests within a time window. The `misbehave` function should call `ban_addr` for repeated offenders.
2. **Add per-IP connection rate limiting** in `accept_peer` — e.g., track connection timestamps per IP and reject connections that arrive faster than a configurable threshold.
3. **Cache or throttle `get_random` responses** per IP/session to avoid repeated peer-store scans.

---

### Proof of Concept

```
for i in 0..200:
    conn = tcp_connect(victim:8115)
    perform_p2p_handshake(conn)
    send_discovery_GetNodes(conn, count=0xFFFFFFFF)
    nodes_response = recv(conn)          # ~50 KB
    assert len(nodes_response) > 40_000
    conn.close()
    # no sleep needed — no rate limit exists
```

Each iteration: attacker sends ~100 bytes, victim sends ~50 KB. At 200 iterations/second the victim emits ~10 MB/s of discovery responses to a single attacker IP, crowding out legitimate sync traffic.
