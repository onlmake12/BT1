### Title
Osmosis Fee Sufficiency Check Compares Raw Token Amounts Across Different Denominations Without Price Conversion - (`target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

### Summary

The Osmosis-specific `is_fee_sufficient` function in the Pyth CosmWasm contract accepts multiple token denominations as fee payment but compares their raw amounts directly against the `uosmo`-denominated required fee, without any price conversion. This is a direct analog to the MIMO `MIMOManagedRebalance` bug: a fee denominated in one asset is validated against payment in a different asset at a 1:1 raw-amount ratio, ignoring actual market exchange rates.

### Finding Description

In `target_chains/cosmwasm/contracts/pyth/src/contract.rs`, the Osmosis feature-gated `is_fee_sufficient` function (lines 163–198) iterates over all coins sent by the caller and sums their raw `u128` amounts into a single `total_amount`, regardless of denomination:

```rust
let mut total_amount = 0u128;
for coin in &info.funds {
    if coin.denom != state.fee.denom && !is_allowed_tx_fees_denom(deps, &coin.denom) {
        return Err(...)?;
    }
    total_amount = total_amount.checked_add(coin.amount.u128())...;
}
let base_denom_fee = get_update_fee(deps, data)?;
Ok(base_denom_fee.amount.u128() <= total_amount)
```

The `base_denom_fee` is denominated in `uosmo` (the base denom). However, `total_amount` is the raw sum of coins that may include `uion`, IBC-wrapped tokens, or any other denom whitelisted by Osmosis's `txfee` module. The function treats 1 `uion` as equivalent to 1 `uosmo` with no spot-price conversion. The code's own comment acknowledges this:

> "NOTE: the base fee denom right now is = denom: 'uosmo', amount: 1, which is almost negligible. It's not important to convert the price right now. For now we are keeping the base fee amount same for each valid denom -> 1 but this logic will be updated to use spot price for different valid tokens in future"

The companion query function `get_update_fee_for_denom` (lines 721–736) has the same flaw: it returns the same raw `uosmo` amount for any requested denom, with no conversion. [1](#0-0) [2](#0-1) 

### Impact Explanation

If governance raises the `uosmo` fee to a non-negligible amount (e.g., 1000 `uosmo`), any caller can satisfy the check by sending 1000 `uion` instead — regardless of the actual `uion/uosmo` exchange rate. If `uion` is worth less than `uosmo`, the protocol receives less economic value than intended. Conversely, if `uion` is worth more, callers overpay. The fee is currently set to 1 (negligible), but the code path is live and the fee is governance-adjustable. The root cause is structurally identical to the MIMO bug: a fee amount in one denomination is compared against payment in another denomination without conversion. [3](#0-2) 

### Likelihood Explanation

The entry path is fully unprivileged: any caller of `update_price_feeds` on the Osmosis deployment can send any whitelisted denom. The `is_allowed_tx_fees_denom` check only verifies that the denom is registered in Osmosis's `txfee` module — it does not enforce any value equivalence. The exploit requires no special role, no leaked key, and no external oracle manipulation. Likelihood increases if governance raises the fee above its current negligible level. [4](#0-3) [5](#0-4) 

### Recommendation

Replace the raw-amount summation with a spot-price-converted comparison. The code's own comment already describes the correct approach:

> "for given coins verify they are allowed in txfee module, convert each of them to the base token that is 'uosmo', combine all the converted token, check with `has_coins`"

Use Osmosis's `SpotPrice` or `ArithmeticTwapToNow` query to convert each non-base denom amount to its `uosmo` equivalent before summing, then compare against `base_denom_fee.amount`. This mirrors the fix applied to `MIMOManagedRebalance`: denominate the fee check in the same unit as the required fee. [6](#0-5) 

### Proof of Concept

1. Governance sets the Osmosis Pyth contract fee to `1000 uosmo`.
2. Caller constructs a `update_price_feeds` transaction sending `info.funds = [Coin { denom: "uion", amount: 1000 }]`.
3. `is_allowed_tx_fees_denom` returns `true` for `"uion"` (it is registered in Osmosis's txfee module).
4. `total_amount = 1000`.
5. `base_denom_fee.amount = 1000` (in `uosmo`).
6. `Ok(1000 <= 1000)` → fee check passes.
7. If `uion` trades at 0.1 `uosmo`, the caller paid the equivalent of 100 `uosmo` but the check accepted it as 1000 `uosmo`. The protocol receives 90% less fee value than required. [1](#0-0)
