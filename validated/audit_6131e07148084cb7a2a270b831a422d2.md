1. `confirm_blocks_to_failed_txs` is allocated with length `MAX_CONFIRM_BLOCKS = 1000` (lines 22, 140).

Audit Report

## Title
Missing Bounds Check on `confirm_blocks_to_failed_txs` Index in `remove_unconfirmed_tx` Causes Node Panic — (`File: util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary
In `TxConfirmStat::remove_unconfirmed_tx`, the code correctly guards `block_unconfirmed_txs` against out-of-bounds access when `tx_age >= 1000`, but then unconditionally indexes `confirm_blocks_to_failed_txs[tx_age - 1]` with no equivalent guard. When a tracked transaction is rejected from the tx-pool after more than 1000 blocks, `tx_age - 1 >= 1000` exceeds the array's valid index range (0–999), triggering a Rust index-out-of-bounds panic that terminates the node process.

## Finding Description
`TxConfirmStat` allocates `confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` both with length `MAX_CONFIRM_BLOCKS = 1000`: [1](#0-0) [2](#0-1) 

In `remove_unconfirmed_tx`, the guard at line 208 correctly routes to `old_unconfirmed_txs` when `tx_age >= self.block_unconfirmed_txs.len()` (i.e., `tx_age >= 1000`). However, the `count_failure` branch immediately below has no such guard: [3](#0-2) 

When `tx_age = 1001`, `tx_age - 1 = 1000` is out of bounds for `confirm_blocks_to_failed_txs` (length 1000, valid indices 0–999), causing a Rust panic.

The `count_failure = true` path is reached via `reject_tx` → `drop_tx` → `drop_tx_inner(tx_hash, true)` → `remove_unconfirmed_tx(..., count_failure=true)`: [4](#0-3) [5](#0-4) 

Every tx-pool rejection is wired to `fee_estimator.reject_tx` via the reject callback in `shared_builder.rs`: [6](#0-5) 

The `limit_size` eviction path also calls `callbacks.call_reject` unconditionally: [7](#0-6) 

## Impact Explanation
When the `ConfirmationFraction` fee estimator is enabled and a tracked transaction is rejected from the tx-pool after more than 1000 blocks, the node panics and terminates. This is a **remote denial-of-service** matching the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**. Any unprivileged tx-pool submitter can crash a node with this estimator configured, with no recovery until the operator manually restarts the process.

## Likelihood Explanation
The `ConfirmationFraction` algorithm is a documented, supported configuration option. An attacker needs only to: (1) submit a low-fee transaction accepted into the pool and tracked at height H; (2) wait 1001+ blocks (~2.8 hours at 10 s/block), or accelerate eviction by flooding the pool with higher-fee transactions to trigger `Reject::Full`; (3) submit a conflicting double-spend with a higher fee, causing the original tracked tx to be rejected via `Reject::RBFRejected` or `Reject::Invalidated`. Step 3 is fully within the attacker's control since they own the private key for the UTXO spent in step 1. The `limit_size` eviction path also triggers the same callback unconditionally, providing an alternative trigger path.

## Recommendation
Add a bounds check before indexing `confirm_blocks_to_failed_txs`, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure {
    if let Some(slot) = self.confirm_blocks_to_failed_txs.get_mut(tx_age - 1) {
        slot[bucket_index] += 1f64;
    }
}
``` [8](#0-7) 

## Proof of Concept
1. Configure the node with `fee_estimator.algorithm = "ConfirmationFraction"`.
2. Submit transaction `tx1` spending a UTXO you control. The estimator tracks it at `entry_height = H` (only if `height == best_height` at submission time).
3. Mine or wait for 1001 blocks so `best_height = H + 1001`.
4. Submit `tx2` double-spending the same UTXO with a higher fee rate. The tx-pool accepts `tx2` and rejects `tx1` via `Reject::RBFRejected` (or flood the pool to trigger `Reject::Full`).
5. The reject callback fires: `fee_estimator.reject_tx(&tx1_hash)` → `drop_tx(&tx1_hash)` → `drop_tx_inner(&tx1_hash, true)` → `remove_unconfirmed_tx(H, H+1001, bucket_index, true)`.
6. `tx_age = 1001 >= 1000` → enters the `old_unconfirmed_txs` branch correctly, then executes `self.confirm_blocks_to_failed_txs[1000][bucket_index]` — **index out of bounds, node panics**. [9](#0-8)
