### Title
Stale `descendants_*` Weight on Ancestor Entries After `remove_entry_and_descendants` Enables Unbounded Eviction-Key Inflation - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

`remove_entry_and_descendants` strips all link records from every entry in the removal set **before** calling `remove_entry` on each one. Because `update_ancestors_index_key` resolves ancestors through those same link records, it finds an empty ancestor set and never decrements the `descendants_count`, `descendants_fee`, `descendants_size`, or `descendants_cycles` fields of transactions that remain in the pool. An unprivileged tx-pool submitter can exploit this by repeatedly adding and RBF-replacing high-fee children of a low-fee parent, inflating the parent's `descendants_count` without bound and permanently biasing the eviction key so the parent is never evicted.

---

### Finding Description

**Root cause — `remove_entry_and_descendants` clears links before updating ancestor weights** [1](#0-0) 

```
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← ALL links torn down here
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← remove_entry called after links are gone
        .collect()
}
```

`remove_entry` then calls `update_ancestors_index_key(entry, Remove)`: [2](#0-1) 

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id()); // returns ∅ — links already gone
    for anc_id in &ancestors {                                 // loop body never executes
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
```

`calc_ancestors` walks the `TxLinksMap`: [3](#0-2) 

Because `remove_entry_links` already deleted the entry for the removed transaction, `calc_ancestors` returns an empty set. The ancestor transaction that **remains** in the pool never has `sub_descendant_weight` called on it, so its `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` are permanently inflated.

**How `descendants_count` drives eviction ordering** [4](#0-3) 

`EvictKey` uses `descendants_count` as the second sort dimension. `next_evict_entry` iterates from the smallest key upward — a transaction with an artificially large `descendants_count` is sorted toward the end of the eviction queue and is therefore evicted last (or never, if the pool stabilises before reaching it). [5](#0-4) 

**Accumulation is unbounded**

`sub_descendant_weight` uses `saturating_sub`: [6](#0-5) 

Because the decrement never fires, each RBF cycle adds one more unit to `descendants_count` (and the corresponding fee/size/cycles) with no corresponding subtraction. The value grows monotonically with each iteration.

---

### Impact Explanation

An attacker who controls a low-fee parent transaction can keep it in the pool indefinitely regardless of pool pressure:

1. The parent's `descendants_count` grows without bound, pushing its `EvictKey` to the top of the ordering.
2. `limit_size` evicts from the bottom of the eviction queue first; the inflated parent is never reached.
3. The pool fills with the attacker's low-fee transaction, displacing legitimate higher-fee transactions that would otherwise be admitted.

This is a **tx-pool denial-of-service**: the attacker occupies pool space at near-zero net cost (RBF replacements require only a marginal fee increment per round) while preventing honest transactions from entering.

---

### Likelihood Explanation

- Requires only the ability to submit transactions to the RPC — no privileged access, no key compromise, no majority hashpower.
- RBF is a standard, documented feature; the attacker pays only the incremental RBF fee per cycle, which can be set to the minimum (`min_rbf_rate`).
- The pool size limit (`max_tx_pool_size`) is finite and well-known, so the attacker can calculate exactly how many cycles are needed to lock out a target transaction.

---

### Recommendation

Before clearing links in `remove_entry_and_descendants`, walk the **external** ancestors of the removal set (i.e., ancestors that are **not** in `removed_ids`) and call `sub_descendant_weight` on each of them for every entry being removed. Concretely:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();

    // Fix: update external ancestors BEFORE tearing down links
    for rid in &removed_ids {
        let ancestors = self.links.calc_ancestors(rid);
        for anc_id in ancestors.difference(&removed_set) {
            if let Some(removed_entry) = self.entries.get_by_id(rid) {
                let inner = removed_entry.inner.clone();
                self.entries.modify_by_id(anc_id, |e| {
                    e.inner.sub_descendant_weight(&inner);
                    e.evict_key = e.inner.as_evict_key();
                });
            }
        }
    }

    for rid in &removed_ids {
        self.remove_entry_links(rid);
    }
    removed_ids.iter().filter_map(|rid| self.remove_entry(rid)).collect()
}
```

Alternatively, restructure `remove_entry` so that it accepts a flag indicating whether ancestor-weight updates should be skipped (only for the internal-chain case), and call the full update path for the root entry of the removal set.

### Proof of Concept

**Setup:** pool with `max_tx_pool_size` = N bytes, RBF enabled.

1. Attacker submits **Tx A** (low fee rate, size S). A enters the pool; `A.descendants_count = 1`.
2. Attacker submits **Tx B** (child of A, high fee rate, size S). `A.descendants_count` becomes 2 via `add_descendant_weight`.
3. Attacker submits **Tx B'** (same inputs as B, fee > B + min\_rbf\_fee). RBF triggers `resolve_conflict` → `remove_entry_and_descendants(B)`. Due to the bug, `A.descendants_count` remains 2 instead of reverting to 1.
4. Repeat steps 2–3 for k rounds. After k rounds, `A.descendants_count = k + 1` even though A has no live descendants.
5. Pool fills to capacity. `limit_size` calls `next_evict_entry`, which iterates `iter_by_evict_key()` from smallest to largest. A's `EvictKey` has `descendants_count = k + 1`; all honest transactions with `descendants_count = 1` sort below A and are evicted first.
6. A remains in the pool indefinitely; honest transactions are continuously displaced. [1](#0-0) [2](#0-1) [4](#0-3) [7](#0-6)

impl Ord for EvictKey {
    fn cmp(&self, other: &Self) -> Ordering {
        if self.fee_rate == other.fee_rate {
            if self.descendants_count == other.descendants_count {
                self.timestamp.cmp(&other.timestamp)
            } else {
                self.descendants_count.cmp(&other.descendants_count)
            }
        } else {
            self.fee_rate.cmp(&other.fee_rate)
        }
