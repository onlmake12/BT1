Audit Report

## Title
Write Lock Held Across Unbounded Synchronous Script Execution During Reorg Re-verification — (`tx-pool/src/process.rs`)

## Summary
`update_tx_pool_for_reorg` acquires the `tx_pool` write lock at L836 and holds it through the entire `readd_detached_tx` call at L849. Inside that function, every detached transaction is re-verified via `verify_rtx(..., None)`, which unconditionally takes the `block_in_place` synchronous path in `util.rs`. The write lock is never released between transactions, blocking all concurrent pool operations for the full duration of re-verification across all detached transactions.

## Finding Description
**Lock acquisition and scope** (`tx-pool/src/process.rs`, L836–851): The write guard is acquired at L836 and lives until the closing brace at L851. The call to `readd_detached_tx` at L849 passes `&mut tx_pool`, keeping the guard alive inside the callee for the entire loop. [1](#0-0) 

**`readd_detached_tx` calls `verify_rtx` with `command_rx = None`** (`tx-pool/src/process.rs`, L895–903): For every detached transaction, `verify_rtx` is called with `None` as the last argument while the write guard is still in scope. [2](#0-1) 

**`verify_rtx` with `command_rx = None` takes the `block_in_place` branch** (`tx-pool/src/util.rs`, L116–131): When `cache_entry` is `None` and `command_rx` is `None`, the code falls through to `block_in_place(|| ...)`, executing full synchronous script verification up to `max_tx_verify_cycles`. `block_in_place` moves other tasks off the current thread but does **not** release the Tokio `RwLock` write guard held in the enclosing scope. [3](#0-2) 

**Async interruptible path bypassed**: The `command_rx = Some(...)` branch at L101–115 uses `verify_with_pause`, which yields to the Tokio scheduler. `readd_detached_tx` passes `None`, bypassing this path entirely. [4](#0-3) 

**Single-consumer reorg loop** (`tx-pool/src/service.rs`, L698–731): The reorg handler is a single `tokio::select!` loop; a second reorg cannot be processed while the first is stalled, and the `.await` on `update_tx_pool_for_reorg` blocks the loop for the full stall duration. [5](#0-4) 

## Impact Explanation
While the write lock is held, `submit_entry` (all tx submissions), `get_block_template` (miner block production), and all RPC calls that acquire the pool write lock are blocked. Stall duration is `N_detached_txs × T_verify_per_tx`. With `max_tx_verify_cycles` at its default (70,000,000 cycles), a block full of max-cycle transactions produces a multi-second stall per detached block. Because all honest nodes receive the same reorg block simultaneously, the stall is network-wide. This matches: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
A 1-block reorg requires no majority hashpower. Any participating miner can privately mine a single valid block at the current tip height and release it after the honest network produces a competing block at the same height. The attacker fills that block with valid transactions consuming `max_tx_verify_cycles` cycles each. The only cost is the sacrificed block reward for the withheld block. No special privilege, no leaked key, and no victim mistake is required. The attack is repeatable at every block interval.

## Recommendation
1. **Release the lock before verification**: Collect the set of transactions to re-add, drop the write guard, verify each transaction independently (without holding any pool lock), then re-acquire the write lock only to insert verified entries.
2. **Pass a `command_rx` to `verify_rtx`**: The async, interruptible `verify_with_pause` path already exists at `util.rs` L101–115; `readd_detached_tx` should supply a `command_rx` so verification yields to the Tokio scheduler and can be cancelled.
3. **Cap the number of transactions re-verified per reorg event** to bound worst-case stall time regardless of block contents.

## Proof of Concept
```
1. Craft transaction T with a lock script consuming exactly max_tx_verify_cycles cycles.
2. Mine block B1 (privately) at height H containing K copies of T (distinct inputs).
3. Wait for the honest network to produce block B1' at height H.
4. Release B1 to a victim node that already has B1' as tip → 1-block reorg.
5. Victim calls update_tx_pool_for_reorg(detached=[B1], attached=[B1']).
6. readd_detached_tx iterates K transactions, calling verify_rtx(..., None) → block_in_place for each.
7. tx_pool write lock is held for K × T_verify_per_tx.
8. During this window, issue get_block_template or send_transaction RPC and observe it hangs until re-verification completes.
```
