### Title
Missing `start <= end` Validation on `IndexerRange` Enables Full-Scan DoS via Inverted Block/Capacity/Data Range — (`File: util/jsonrpc-types/src/indexer.rs`)

---

### Summary

The `IndexerRange` type used in all indexer filter parameters (`block_range`, `output_capacity_range`, `script_len_range`, `output_data_len_range`) accepts any two `Uint64` values without validating that `start <= end`. When an inverted range is supplied (e.g., `block_range = ["0xc8", "0x64"]` meaning `[200, 100)`), the per-entry filter condition becomes universally true, causing the basic indexer's `TimeoutIterator` to scan every indexed entry for the full 10-second timeout before returning an error. Any unauthenticated RPC caller can trigger this.

---

### Finding Description

`IndexerRange` is defined in `util/jsonrpc-types/src/indexer.rs` as a transparent two-element array. Neither the constructor nor the `serde` deserialization path enforces `start <= end`:

```rust
// util/jsonrpc-types/src/indexer.rs lines 102-128
#[derive(Deserialize, Default, JsonSchema)]
#[serde(transparent)]
pub struct IndexerRange {
    inner: [Uint64; 2],
}

impl IndexerRange {
    pub fn new<U>(start: U, end: U) -> Self
    where U: Into<Uint64> {
        IndexerRange { inner: [start.into(), end.into()] }
        // ^^^ no check that start <= end
    }
    pub fn start(&self) -> Uint64 { self.inner[0] }
    pub fn end(&self)   -> Uint64 { self.inner[1] }
}
``` [1](#0-0) 

The `TryInto<FilterOptions>` conversion in `util/indexer/src/service.rs` blindly maps the range into a `[BlockNumber; 2]` array with no ordering check:

```rust
// util/indexer/src/service.rs lines 962-964
let block_range = filter
    .block_range
    .map(|r| [r.start().into(), r.end().into()]);
// ^^^ no require!(r.start() <= r.end())
``` [2](#0-1) 

The filter is then applied per-entry inside the `TimeoutIterator` loop in `get_cells`, `get_transactions`, and `get_cells_capacity`:

```rust
// util/indexer/src/service.rs lines 351-355
if let Some([r0, r1]) = filter_options.block_range
    && (block_number < r0 || block_number >= r1)
{
    return None;
}
``` [3](#0-2) 

When `r0 > r1` (inverted range), the condition `block_number < r0 || block_number >= r1` is **always true** for every possible `block_number`, because any integer is either less than the larger bound or greater-than-or-equal-to the smaller bound. Every entry is rejected, but the iterator is not short-circuited — it continues scanning until the timeout fires.

The `TimeoutIterator` defaults to a 10-second wall-clock timeout:

```rust
// util/indexer/src/service.rs lines 98-99
timeout_limit: Duration::from_secs(config.timeout_limit.unwrap_or(10)),
``` [4](#0-3) 

When the timeout fires, the function returns an **error** (`"Indexer request timeout"`), not an empty result. The `.take(limit)` combinator is applied after the filter and never terminates early because the filter never passes any item:

```rust
// util/indexer/src/service.rs lines 249-372 (get_cells)
let cells = iter
    .by_ref()
    .take_while(|(key, _value)| key.starts_with(&prefix))
    .filter_map(|(key, value)| { /* ... filter_options.block_range check ... */ })
    .take(limit)   // never reached when all entries are filtered
    .collect::<Vec<_>>();
``` [5](#0-4) 

The same pattern applies to `get_transactions` (lines 547–551, 659–663) and `get_cells_capacity` (lines 828–832). [6](#0-5) [7](#0-6) 

The same missing check exists for `output_capacity_range`, `script_len_range`, and `output_data_len_range` — all share the same `IndexerRange` type with no ordering invariant.

---

### Impact Explanation

**Impact: Medium**

Any RPC caller can force the basic indexer to perform a full RocksDB scan (bounded only by the 10-second timeout) per request by supplying an inverted range such as `"block_range": ["0xffffffffffffffff", "0x0"]`. The scan consumes CPU and I/O for the full timeout duration before returning an error. Sending concurrent requests with inverted ranges saturates the indexer thread pool, making the indexer unresponsive to legitimate queries for the duration of the attack. The indexer is a production component used by wallets, dApps, and exchanges to query live cells and transactions; its unavailability disrupts downstream services.

---

### Likelihood Explanation

**Likelihood: High**

The indexer RPC is publicly exposed on any node that enables it. No authentication is required. The attack requires only a single malformed JSON-RPC call with an inverted two-element array in the `filter.block_range` field. The `IndexerRange` type is deserialized directly from user-supplied JSON with no server-side validation.

---

### Recommendation

Add an ordering invariant check in `IndexerRange::new` and in the `TryInto<FilterOptions>` conversion:

```rust
// In IndexerRange::new or in TryInto<FilterOptions>
if r.start() > r.end() {
    return Err(Error::invalid_params(
        "search_key.filter range start must be <= end"
    ));
}
```

Apply this check to all four range fields: `block_range`, `output_capacity_range`, `script_len_range`, and `output_data_len_range`. [8](#0-7) 

---

### Proof of Concept

Send the following JSON-RPC request to a node with the indexer enabled. The `block_range` is inverted (`end < start`):

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells",
  "params": [
    {
      "script": {
        "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
        "hash_type": "type",
        "args": "0x"
      },
      "script_type": "lock",
      "filter": {
        "block_range": ["0xffffffffffffffff", "0x0"]
      }
    },
    "asc",
    "0x64",
    null
  ]
}
```

The indexer will scan all indexed entries for 10 seconds (the default `timeout_limit`) and then return `{"error": {"code": -32602, "message": "Indexer request timeout"}}`. Sending this request in a tight loop from multiple clients exhausts the indexer's capacity.

The root cause is confirmed at `util/jsonrpc-types/src/indexer.rs` lines 110–116 (no `start <= end` check in `IndexerRange::new`) and `util/indexer/src/service.rs` lines 962–964 (no validation in `TryInto<FilterOptions>`). [9](#0-8) [2](#0-1)
