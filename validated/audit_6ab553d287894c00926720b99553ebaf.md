### Title
Stale `priceUpdates` and `priceLastUpdatedAt` Persist After Subscription Deactivation/Reactivation, Causing Stale Prices and Blocked Keeper Updates - (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In the Pulse `Scheduler` contract, when a subscription is deactivated and later reactivated with the same price IDs, neither the stored `priceUpdates` mapping nor `priceLastUpdatedAt` is cleared. This is the direct analog of the HydraDX oracle bug: stale state persists across a removal/re-addition lifecycle, causing incorrect baseline comparisons and stale prices being served to readers after reactivation.

---

### Finding Description

`updateSubscription()` calls `_clearRemovedPriceUpdates()` to delete stored `PriceFeed` data only for price IDs that are **removed** from the subscription. If the same price IDs are retained across a deactivation/reactivation cycle, no data is cleared:

```solidity
// Clear price updates for removed price IDs before updating params
bool newPriceIdsAdded = _clearRemovedPriceUpdates(
    subscriptionId,
    currentParams.priceIds,
    newParams.priceIds
);

// Reset priceLastUpdatedAt to 0 if new price IDs were added
if (newPriceIdsAdded) {
    _state.subscriptionStatuses[subscriptionId].priceLastUpdatedAt = 0;
}
``` [1](#0-0) 

`_clearRemovedPriceUpdates` iterates old IDs and deletes only those absent from the new list. When the same IDs are present in both lists, it returns `newPriceIdsAdded = false`, so `priceLastUpdatedAt` is never reset:

```solidity
if (!found) {
    delete _state.priceUpdates[subscriptionId][oldPriceId];
}
``` [2](#0-1) 

After reactivation with the same price IDs:
- `_state.priceUpdates[subscriptionId][priceId]` still holds the pre-deactivation price data.
- `status.priceLastUpdatedAt` still holds the pre-deactivation timestamp.

**Two concrete failure modes arise in `_validateShouldUpdatePrices`:**

**Failure Mode 1 — Heartbeat subscriptions blocked after reactivation:**

```solidity
if (params.updateCriteria.updateOnHeartbeat) {
    uint256 lastUpdateTime = status.priceLastUpdatedAt;
    if (
        lastUpdateTime == 0 ||
        updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds
    ) {
        return updateTimestamp;
    }
}
``` [3](#0-2) 

If the subscription was deactivated for less than `heartbeatSeconds`, `priceLastUpdatedAt` is non-zero and the heartbeat condition is not met. Keepers cannot push any update until the full heartbeat interval elapses from the pre-deactivation timestamp.

**Failure Mode 2 — Deviation-only subscriptions use stale baseline:**

```solidity
PythStructs.PriceFeed storage previousFeed = _state
    .priceUpdates[subscriptionId][priceFeeds[i].id];

// If there's no previous price, this is the first update
if (previousFeed.id == bytes32(0)) {
    return updateTimestamp;
}
``` [4](#0-3) 

After reactivation, `previousFeed.id != bytes32(0)` because the old data persists. The deviation check compares the new price against the stale pre-deactivation price. If the price moved less than `deviationThresholdBps` from the old baseline (even though the price is now stale), the update is rejected with `UpdateConditionsNotMet`.

The stored state is defined in `SchedulerState.sol`:

```solidity
/// Sub ID -> price ID -> latest parsed price update for the subscribed feed
mapping(uint256 => mapping(bytes32 => PythStructs.PriceFeed)) priceUpdates;
``` [5](#0-4) 

---

### Impact Explanation

1. **Stale prices served to readers**: `getPricesUnsafe` returns the pre-deactivation price with no staleness check. Integrators relying on this function receive incorrect data after reactivation. [6](#0-5) 

2. **Keeper DoS on heartbeat subscriptions**: If a subscription is deactivated and reactivated within one heartbeat window, no keeper can push a fresh update until the full heartbeat interval elapses from the pre-deactivation timestamp. During this window, all readers receive stale prices.

3. **Keeper DoS on deviation-only subscriptions**: If the price moved less than `deviationThresholdBps` from the stale baseline during the deactivation period, no keeper can push an update at all, even though the stored price is arbitrarily old.

---

### Likelihood Explanation

Subscription managers routinely deactivate subscriptions to update parameters (e.g., change price IDs, adjust criteria, pause during maintenance). Reactivating with the same price IDs is the common case. The deactivation window can be short (seconds to minutes), well within a typical heartbeat interval (e.g., 60–3600 seconds). No privileged access beyond the subscription manager role is required, and the manager is an unprivileged user of the Scheduler contract.

---

### Recommendation

When a subscription is deactivated (transition from `isActive=true` to `isActive=false`), clear all stored `priceUpdates` for the subscription's current price IDs and reset `priceLastUpdatedAt` to 0. This mirrors the fix recommended in the HydraDX report: delete oracle entries on removal so that re-addition starts from a clean state.

```solidity
} else if (wasActive && !willBeActive) {
    // Deactivating a subscription — clear stale price data
    for (uint i = 0; i < currentParams.priceIds.length; i++) {
        delete _state.priceUpdates[subscriptionId][currentParams.priceIds[i]];
    }
    _state.subscriptionStatuses[subscriptionId].priceLastUpdatedAt = 0;
    currentParams.isActive = false;
    _removeFromActiveSubscriptions(subscriptionId);
    emit SubscriptionDeactivated(subscriptionId);
}
```

### Proof of Concept

1. Create a heartbeat subscription (heartbeat = 3600s) with price IDs `[A, B]`.
2. Keeper updates prices at `T=0`; `priceLastUpdatedAt = T=0`, `priceUpdates[A].price = $100`.
3. Manager deactivates subscription at `T=1800` (30 min later). `priceUpdates` and `priceLastUpdatedAt` are unchanged.
4. Price of A moves to `$200` on Pythnet during deactivation.
5. Manager reactivates subscription at `T=3000` (50 min after last update). `priceLastUpdatedAt` is still `T=0`; `priceUpdates[A].price` is still `$100`.
6. Keeper attempts to push fresh update at `T=3001` with `publishTime=3001`:
   - Heartbeat check: `3001 < 0 + 3600` → condition NOT met.
   - No other criteria → `revert UpdateConditionsNotMet`.
7. Readers calling `getPricesUnsafe` receive `$100` for asset A, even though the true price is `$200`.
8. This DoS persists until `T=3600`, at which point the heartbeat fires and the keeper can finally push the correct price — 60 minutes of stale data served.
