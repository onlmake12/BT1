Audit Report

## Title
Unsolicited Block Filter Response Messages Never Trigger Peer Ban — (`sync/src/filter/mod.rs`)

## Summary
`BlockFilter::try_process()` returns `Status::ignored()` (StatusCode 101) for unsolicited `BlockFilters`, `BlockFilterHashes`, and `BlockFilterCheckPoints` messages, despite a `// TODO: ban remote peer` comment. Because `should_ban()` only returns `Some` for 4xx codes, `process()` never calls `nc.ban_peer()` for this message class. A peer can send an unbounded stream of these messages indefinitely with no consequence.

## Finding Description
In `try_process()`, the three unsolicited response message types are matched and return `Status::ignored()` (StatusCode 101):

```rust
packed::BlockFilterMessageUnionReader::BlockFilters(_)
| packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)
| packed::BlockFilterMessageUnionReader::BlockFilterCheckPoints(_) => {
    // remote peer should not send block filter to us without asking
    // TODO: ban remote peer
    warn_target!(...);
    Status::ignored()   // StatusCode::Ignored = 101
}
```

The `process()` dispatcher evaluates the returned status:

```rust
if let Some(ban_time) = status.should_ban() {
    nc.ban_peer(peer, ban_time, status.to_string());
}
```

`should_ban()` in `status.rs` gates on the 400–499 range:

```rust
pub fn should_ban(&self) -> Option<Duration> {
    if !(400..500).contains(&(self.code as u16)) {
        return None;
    }
    ...
}
```

`StatusCode::Ignored = 101` is outside this range, so `should_ban()` always returns `None`, and `nc.ban_peer()` is never reached. The ban infrastructure is fully functional — malformed messages in the same `received()` handler correctly call `nc.ban_peer()` directly. The only missing piece is the unimplemented TODO.

## Impact Explanation
An unprivileged external peer can send an unbounded stream of valid-but-unsolicited filter response messages. Each message is parsed, dispatched through `process()` → `try_process()`, logged, and metrics-recorded, with no cost to the sender. Because the peer is never banned or disconnected, the attacker can sustain the flood indefinitely from a single connection. Scaled across many target nodes simultaneously, this constitutes a low-cost availability attack matching: **High — bad designs which could cause CKB network congestion with few costs** (the attacker expends minimal resources while each victim node continuously processes, logs, and records metrics for every message).

## Likelihood Explanation
Any peer that can establish a connection on `SupportProtocols::Filter` can trigger this. No special privileges, keys, or hashpower are required. The attacker simply connects and repeatedly sends valid, well-formed filter response messages. The attack is trivially scriptable, requires no prior state, and can be sustained indefinitely since the peer is never banned.

## Recommendation
Replace `Status::ignored()` in the unsolicited-filter-response arm with a ban-triggering 4xx status, completing the documented TODO:

```rust
packed::BlockFilterMessageUnionReader::BlockFilters(_)
| packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)
| packed::BlockFilterMessageUnionReader::BlockFilterCheckPoints(_) => {
    // remote peer should not send block filter to us without asking
    StatusCode::ProtocolMessageIsMalformed
        .with_context("received unsolicited block filter response")
}
```

This causes `should_ban()` to return `Some(BAD_MESSAGE_BAN_TIME)`, and the existing `process()` dispatcher will call `nc.ban_peer()` automatically, consistent with how every other misbehaving-peer case is handled in the codebase.

## Proof of Concept
1. Attacker establishes a P2P connection to a CKB full node with the block filter protocol enabled.
2. Attacker sends a continuous stream of `BlockFilters` (or `BlockFilterHashes` / `BlockFilterCheckPoints`) messages without any prior `GetBlockFilters` request from the node.
3. Each message enters `BlockFilter::received()` → `process()` → `try_process()`, matches the unsolicited arm, logs a warning, and returns `Status::ignored()`.
4. `process()` calls `status.should_ban()` → returns `None` (code 101 is not in 400–499) → `nc.ban_peer()` is never called.
5. The attacker repeats indefinitely. The node logs warnings and records metrics for every message but never disconnects or bans the sender.
6. To confirm the fix: after replacing `Status::ignored()` with `StatusCode::ProtocolMessageIsMalformed.with_context(...)`, a unit test can assert that `status.should_ban()` returns `Some(BAD_MESSAGE_BAN_TIME)` for the returned status, and an integration test can verify that a peer sending unsolicited filter messages is banned after the first such message.
