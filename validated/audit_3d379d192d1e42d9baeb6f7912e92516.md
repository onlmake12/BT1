### Title
Parameter Binding Order Inversion in `get_cells` When Dead-Cells and `filter.script` Are Both Present — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

### Summary

When `index_tx_pool=true`, at least one pending transaction exists (producing N dead cells), and the RPC caller supplies a non-`None` `filter.script`, the SQL placeholder numbering and the `sqlx` bind sequence are inverted for those two groups. The SQL receives wrong values for both the dead-cell exclusion filter and the script filter, silently returning an incorrect cell set.

### Finding Description

**SQL placeholder generation order** (Prefix/None mode, N dead cells):

| Step | Code location | Placeholders consumed | `param_index` after |
|---|---|---|---|
| `build_query_script_sql` | `get_cells.rs:39-43` | `$1`–`$4` (code_hash, hash_type, args lo/hi) | 5 |
| Dead-cells loop | `get_cells.rs:124-135` | `$5`–`$(4+N)` (one tx_hash per dead cell) | 5+N |
| `build_cell_filter` | `get_cells.rs:145-150` + `mod.rs:163-198` | `$(5+N)`–`$(8+N)` (filter.script code_hash, hash_type, args lo/hi) | 9+N |

So the SQL WHERE clause expects:
- `$5 … $(4+N)` → dead-cell `tx_hash` values
- `$(5+N) … $(8+N)` → `filter.script` fields

**Bind phase order** (`get_cells.rs:165-224`):

```
$1  ← search_key.script.code_hash          ✓
$2  ← search_key.script.hash_type          ✓
$3  ← search_key.script.args lower         ✓
$4  ← search_key.script.args upper         ✓
$5  ← filter.script.code_hash              ✗  (SQL expects dead_cell[0].tx_hash)
$6  ← filter.script.hash_type              ✗  (SQL expects dead_cell[1].tx_hash or tx_hash[0])
$7  ← filter.script.args lower             ✗
$8  ← filter.script.args upper             ✗
$9  ← dead_cell[0].tx_hash                 ✗  (SQL expects filter.script.code_hash)
$10 ← dead_cell[1].tx_hash                 ✗
…
```

The dead-cells loop in the SQL builder advances `param_index` N times before `build_cell_filter` is called: [1](#0-0) 

Then `build_cell_filter` appends filter.script placeholders at the now-advanced `param_index`: [2](#0-1) [3](#0-2) 

But the bind phase binds `filter.script` params first (lines 189–198), then dead-cell tx_hashes last (lines 221–224): [4](#0-3) [5](#0-4) 

### Impact Explanation

Both filters receive wrong values simultaneously:

1. **Dead-cell exclusion** (`NOT IN (($5, idx), …)`): receives `filter.script` byte fields (code_hash, hash_type, args) as tx_hashes. These will never match any real tx_hash, so **all dead cells pass through** — outputs already being spent in pending transactions are returned as live cells.

2. **`filter.script` comparison** (`type_script.code_hash = $(5+N)`, etc.): receives dead-cell tx_hash bytes as a script code_hash. This will match nothing or match wrong rows, so **the script filter is silently bypassed or produces garbage results**.

Neither SQLite nor PostgreSQL will raise a type error here — both columns are byte arrays (`BLOB`/`bytea`). The query executes successfully and returns a silently incorrect cell set.

A wallet or application using `get_cells` with `index_tx_pool=true` and a `filter.script` to enumerate spendable UTXOs will see dead cells (already-spent outputs) in the result, potentially leading to double-spend construction.

### Likelihood Explanation

- `index_tx_pool=true` is a supported, documented configuration for nodes that want real-time UTXO visibility.
- Any unprivileged RPC caller can trigger this by submitting any transaction (creating dead cells) and then calling `get_cells` with a `filter.script`.
- No special privileges, keys, or majority hashpower are required.
- The condition is trivially reproducible in a local test environment.

### Recommendation

Reorder the bind phase in `get_cells.rs` to match the SQL placeholder generation order: bind dead-cell tx_hashes immediately after the main script params (before `filter.script` params), mirroring the order in which `param_index` was advanced during SQL construction. Alternatively, restructure the SQL builder to append dead-cell placeholders after `build_cell_filter` and bind them in the same trailing position.

### Proof of Concept

With an in-memory SQLite instance:
1. Insert one block containing one live cell with a known lock script.
2. Add one dead cell to the pool (any out-point).
3. Call `get_cells` with `filter.script` set to any script value.
4. Observe: the dead cell appears in the result (dead-cell exclusion is broken) and/or cells that should match `filter.script` are absent (script filter receives wrong bytes).
5. Compare against a reference query built with the correct binding order to confirm the discrepancy.
