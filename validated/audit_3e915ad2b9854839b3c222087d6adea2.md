### Title
`check_purge` Group-Eviction `take(len/2)` Integer-Division Deadlock When All Addresses Map to `Group::None` — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

`PeerStore::check_purge` contains an integer-division bug in its group-eviction branch: when all stored addresses map to the same network group (`Group::None`), `len = 1` and `take(len / 2) = take(0)` yields an empty iterator, so no peer is evicted and `Err(EvictionFailed)` is returned. An unprivileged remote peer can trigger this by advertising 16 384 non-IP multiaddrs (e.g., Onion3) via the identify protocol, permanently filling the peer store and blocking new peer discovery.

---

### Finding Description

**Root cause — integer division in `take(len / 2)`:** [1](#0-0) 

```rust
let len = peers_by_network_group.len();   // = 1 when all addrs are Group::None
peers
    .into_iter()
    .take(len / 2)   // 1 / 2 = 0  →  takes nothing
    .flat_map(...)
    .collect()       // always empty
```

When `len == 1`, `len / 2 == 0` (Rust integer division), so the iterator is exhausted before any group is visited. The inner `addrs.len() > 4` guard is never reached, `candidate_peers` stays empty, and the function falls through to: [2](#0-1) 

**Non-IP addresses all hash to `Group::None`:** [3](#0-2) 

Any multiaddr for which `multiaddr_to_socketaddr` returns `None` (Onion3, DNS, `/p2p/…`-only, etc.) falls through to `Group::None`. All such addresses share a single bucket.

**Freshly added addresses are always connectable:** [4](#0-3) 

`ADDR_MAX_RETRIES = 3`, `ADDR_MAX_FAILURES = 10`. `add_addr` stores every new address with `attempts_count = 0`: [5](#0-4) 

With `attempts_count = 0 < 3`, `is_connectable` returns `true`: [6](#0-5) 

So step 1 of `check_purge` (evict non-connectable peers) finds nothing to remove, and execution always reaches the broken group-eviction branch.

**Attacker entry point — identify protocol:** [7](#0-6) 

`add_remote_listen_addrs` iterates every address advertised by a remote peer and calls `add_addr` with no address-type filter. There is no restriction preventing non-IP multiaddrs from being stored.

---

### Impact Explanation

Once the peer store reaches `ADDR_COUNT_LIMIT` (16 384) with all-`Group::None` connectable entries, every subsequent `add_addr` call returns `Err(EvictionFailed)`. The error is only logged at the identify-protocol layer; the node continues running but can no longer record newly discovered peers. Because the store is persisted to disk on shutdown and reloaded on startup, a node restart does not clear the condition. Non-IP addresses are never dialed by a standard CKB node, so `attempts_count` never increments and the entries never become non-connectable on their own. The peer store remains permanently full, blocking all future peer discovery. [8](#0-7) 

---

### Likelihood Explanation

The attacker must establish at least one inbound or outbound P2P connection to the victim and advertise non-IP listen addresses via the identify protocol. Generating 16 384 distinct Onion3 multiaddrs is trivial (each requires a unique 35-byte public key). A single well-connected adversarial node relaying addresses through the discovery protocol, or a small number of colluding peers, can fill the store without any privileged access. No PoW, no key material, and no operator interaction is required.

---

### Recommendation

Replace `take(len / 2)` with `take((len + 1) / 2)` (ceiling division) so that a single-group store still yields one candidate group for eviction:

```rust
// Before (broken for len == 1):
.take(len / 2)

// After (correct):
.take(len.saturating_add(1) / 2)
```

Additionally, consider rejecting or rate-limiting non-IP addresses in `add_addr` (or in `add_remote_listen_addrs`) to prevent a single peer from flooding the store with ungroupable addresses. [1](#0-0) 

---

### Proof of Concept

```rust
// Fill peer store with 16_384 Onion3 addresses (all connectable, all Group::None)
let mut peer_store = PeerStore::default();
for i in 0u64..16_384 {
    // Onion3: 35-byte host derived from i, port 9001
    let mut host = [0u8; 35];
    host[..8].copy_from_slice(&i.to_le_bytes());
    let onion = p2p::multiaddr::Protocol::Onion3(
        p2p::multiaddr::Onion3Addr::from((host, 9001))
    );
    let addr: Multiaddr = std::iter::once(onion).collect();
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
// Store is now at ADDR_COUNT_LIMIT; all entries are Group::None and connectable.
// Adding one more address must NOT return Err(EvictionFailed).
let extra: Multiaddr = "/onion3/aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa:9002"
    .parse().unwrap();
let result = peer_store.add_addr(extra, Flags::COMPATIBILITY);
assert!(result.is_ok(), "peer store permanently blocked: {:?}", result);
// This assertion FAILS on the current code, confirming the vulnerability.
```
