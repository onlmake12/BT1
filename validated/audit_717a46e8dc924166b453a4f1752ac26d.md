### Title
`Echo.executeCallback` Caller Can Skip Consumer Callback by Sending Insufficient Gas — (`Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` wraps the consumer's `_echoCallback` in a `try/catch` with an explicit gas cap (`req.callbackGasLimit`), but performs **no pre-check** that the outer calling context holds enough gas to actually forward that amount. Due to the EVM's 63/64 rule, any caller can supply just enough gas to execute the outer function while ensuring the callback receives fewer gas units than `callbackGasLimit`. The callback silently fails, the catch block emits `PriceUpdateCallbackFailed`, and — because the request is cleared and the provider credited **before** the callback — there is no recovery path.

---

### Finding Description

`Echo.executeCallback` is callable by anyone (only an exclusivity-period check restricts it to the assigned provider during a short window; after that window any address may call it):

```solidity
// Echo.sol lines 176-201
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{
    emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(..., reason);
} catch {
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
``` [1](#0-0) 

The `callbackGasLimit` is set by the requester at request time and stored in the `Request` struct: [2](#0-1) 

Critically, **both the fee credit and the request clearance happen before the callback**:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += ...;  // line 161
clearRequest(sequenceNumber);                                  // line 164
// ... then the try/catch callback at line 176
``` [3](#0-2) 

There is **no guard** equivalent to `Entropy.sol`'s `(startingGas * 31) / 32 > gasLimit` check, which Entropy uses to detect and revert when the outer context cannot forward the required gas: [4](#0-3) 

Because the EVM forwards at most 63/64 of available gas to a sub-call, a caller who provides `X` gas to `executeCallback` will forward at most `X * 63/64` to `_echoCallback`. If `X * 63/64 < callbackGasLimit`, the callback runs out of gas, reverts, and the catch block fires — but the outer transaction succeeds because the remaining `X/64` gas is enough to emit the event.

---

### Impact Explanation

- The consumer's `_echoCallback` is permanently skipped. Because `clearRequest` runs before the callback, the request no longer exists and cannot be retried (unlike `Entropy.sol`, which has a `CALLBACK_FAILED` state and a recovery flow).
- The provider has already been credited their fee.
- Any application logic inside `_echoCallback` (e.g., updating on-chain state based on a price feed) is silently lost.
- The consumer paid a fee that includes a `feePerGasInWei * callbackGasLimit` component, yet their callback was never executed. [5](#0-4) 

---

### Likelihood Explanation

`executeCallback` is `external` with no caller authentication beyond the exclusivity period: [6](#0-5) 

- **During the exclusivity period**: the assigned provider is the only permitted caller. A malicious provider can deliberately call with insufficient gas to deny the consumer their callback while still collecting the fee.
- **After the exclusivity period**: any unprivileged address can call `executeCallback` and trigger the same attack.

The gas calculation required to exploit this is straightforward: choose `X` such that `X * 63/64 < callbackGasLimit` but `X` is sufficient for the outer function's logic (a few thousand gas for storage writes and event emission).

---

### Recommendation

Add a pre-call gas sufficiency check mirroring `Entropy.sol`'s pattern:

```solidity
uint256 startingGas = gasleft();
// Revert if the outer context cannot forward callbackGasLimit to the sub-call.
// Using 31/32 instead of 63/64 introduces a safety margin.
require(
    (startingGas * 31) / 32 >= req.callbackGasLimit,
    "Insufficient gas to execute callback"
);
try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...) {
    ...
}
```

Additionally, consider moving `clearRequest` to **after** a successful callback and implementing a `CALLBACK_FAILED` retry state (as `Entropy.sol` does) so that a failed callback is not permanently lost.

---

### Proof of Concept

1. Consumer calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 500_000`. The request is stored with `req.callbackGasLimit = 500_000`.
2. After the exclusivity period, attacker calls:
   ```solidity
   echo.executeCallback{gas: G}(provider, sequenceNumber, updateData, priceIds);
   ```
   where `G` is chosen so that `G * 63/64 < 500_000` (e.g., `G ≈ 508_000`) but `G` is large enough for the outer function's storage writes and event emission (a few thousand gas).
3. Inside `executeCallback`, `_echoCallback` is invoked with `{gas: 500_000}` but only receives `≈ G * 63/64 < 500_000` gas due to the 63/64 rule. It runs out of gas and reverts.
4. The `catch` block emits `PriceUpdateCallbackFailed`.
5. `clearRequest` already ran at line 164 — the request is gone. The provider was already credited at line 161.
6. The consumer's callback logic is permanently skipped with no retry possible. [1](#0-0)
