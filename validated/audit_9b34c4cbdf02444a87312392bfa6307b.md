### Title
`migrate_transaction_info()` Traverses Wrong Database Column (`COLUMN_UNCLES` Instead of `COLUMN_TRANSACTION_INFO`) — (`File: util/migrate/src/migrations/table_to_struct.rs`)

---

### Summary

In `util/migrate/src/migrations/table_to_struct.rs`, the `migrate_transaction_info()` function is supposed to reformat all entries in `COLUMN_TRANSACTION_INFO` (column `"5"`) from the old Molecule table layout to the new struct layout. Due to a copy-paste error, it instead iterates over `COLUMN_UNCLES` (column `"11"`). The write-back callback correctly targets `COLUMN_TRANSACTION_INFO`, so the net effect is: uncle data is read, sliced with the wrong offset, and written into `COLUMN_TRANSACTION_INFO` under uncle-hash keys — while the actual transaction info entries are never migrated and remain in the old, unreadable format.

---

### Finding Description

Every migration function in `ChangeMoleculeTableToStruct` follows the same pattern: traverse the correct column, strip the Molecule table header bytes, and write the result back to the same column. Three of the four functions do this correctly:

- `migrate_header`: traverses `COLUMN_BLOCK_HEADER`, writes to `COLUMN_BLOCK_HEADER` ✓
- `migrate_uncles`: traverses `COLUMN_UNCLES`, writes to `COLUMN_UNCLES` ✓
- `migrate_epoch_ext`: traverses `COLUMN_EPOCH`, writes to `COLUMN_EPOCH` ✓

`migrate_transaction_info` breaks the pattern:

```rust
// util/migrate/src/migrations/table_to_struct.rs  lines 77-102
fn migrate_transaction_info(&self, db: &RocksDB) -> Result<()> {
    const TRANSACTION_INFO_SIZE: usize = 52;
    ...
    let mut transaction_info_migration = |key: &[u8], value: &[u8]| -> Result<()> {
        if value.len() != TRANSACTION_INFO_SIZE {
            wb.put(COLUMN_TRANSACTION_INFO, key, &value[16..])?;  // writes to correct column
        }
        Ok(())
    };
    ...
    let (_count, nk) =
        db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;  // ← reads WRONG column
    ...
}
``` [1](#0-0) 

The `traverse` call at line 93 passes `COLUMN_UNCLES` (`"11"`) instead of `COLUMN_TRANSACTION_INFO` (`"5"`). [2](#0-1) 

The two columns store completely different data:

- `COLUMN_TRANSACTION_INFO` (`"5"`): keyed by tx hash → `packed::TransactionInfo` (block hash, block number, block epoch, tx index). Old format has a 16-byte Molecule table header; migrated size is 52 bytes.
- `COLUMN_UNCLES` (`"11"`): keyed by uncle hash → `packed::HeaderView` (full block header). Size is ~240 bytes. [3](#0-2) 

Because uncle entries are ~240 bytes, the guard `value.len() != TRANSACTION_INFO_SIZE` (52) is always true for them, so every uncle entry is processed. The callback slices `value[16..]` (designed for the 16-byte TransactionInfo table header) and writes the result into `COLUMN_TRANSACTION_INFO` under the uncle's hash key.

Simultaneously, the real `COLUMN_TRANSACTION_INFO` entries are never visited, so they remain in the old Molecule-table format with the 16-byte prefix intact. [4](#0-3) 

---

### Impact Explanation

After this migration runs, `COLUMN_TRANSACTION_INFO` is in a corrupt state:

1. **Real tx-hash keys** → old-format entries (16-byte table header still present). When the post-migration code calls `packed::TransactionInfoReader::from_slice_should_be_ok`, it parses the wrong bytes, producing garbage `block_hash`, `block_number`, `block_epoch`, and `index` values.

2. **Uncle-hash keys** → spurious entries containing sliced uncle header data, polluting the transaction index with phantom "transactions."

All store functions that depend on `COLUMN_TRANSACTION_INFO` are affected:

- `transaction_exists()` — returns `true` for uncle hashes, `false` for real tx hashes that were never found (or returns stale/wrong data).
- `get_transaction_info()` — returns corrupted `TransactionInfo` for any tx hash.
- `get_transaction_with_info()` / `get_transaction()` — uses the corrupted `block_hash` and `index` to fetch the wrong transaction body from `COLUMN_BLOCK_BODY`. [5](#0-4) 

Downstream consumers of these functions include the RPC (`get_transaction`), the tx-pool (double-spend detection), and chain verification (transaction existence checks). Corrupted transaction lookup results can cause a node to accept double-spends it should reject, serve wrong transaction data over RPC, or fail to locate committed transactions.

---

### Likelihood Explanation

This migration (`version = "20200703124523"`) is registered in the active migration list and is executed whenever a node upgrades from a database version older than July 2020. [6](#0-5) 

Any operator who:
- restores a pre-2020 database backup and runs `ckb migrate`, or
- upgrades a long-running archival node that skipped intermediate versions,

will silently corrupt their transaction index. The migration reports success (no error is returned), so the operator has no indication that anything went wrong. The node then starts with a permanently corrupted `COLUMN_TRANSACTION_INFO`.

---

### Recommendation

Change line 93 to traverse `COLUMN_TRANSACTION_INFO` instead of `COLUMN_UNCLES`:

```rust
// Before (wrong):
let (_count, nk) =
    db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;

// After (correct):
let (_count, nk) =
    db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
``` [7](#0-6) 

Additionally, consider adding a post-migration integrity check that samples a few entries from `COLUMN_TRANSACTION_INFO` and verifies they parse correctly as `packed::TransactionInfo`.

---

### Proof of Concept

1. Start a CKB node with a database at a version older than `20200703124523` (or construct one by reverting the migration version key).
2. Run `ckb migrate`.
3. After migration completes successfully, query any committed transaction via RPC `get_transaction <tx_hash>`.
4. Observe that the RPC either returns wrong transaction data (wrong block hash / index) or fails to find the transaction — because `COLUMN_TRANSACTION_INFO` was never correctly migrated and the tx-hash key still holds old-format bytes.
5. Query an uncle hash via `transaction_exists` (internal) and observe it returns `true`, confirming uncle data was written into the transaction index. [1](#0-0) [8](#0-7)

    fn migrate_uncles(&self, db: &RocksDB) -> Result<()> {
        const HEADER_SIZE: usize = 240;
        let mut next_key = vec![0];
        while !next_key.is_empty() {
            let mut wb = db.new_write_batch();
            let mut uncles_migration = |key: &[u8], value: &[u8]| -> Result<()> {
                // (1 total size field + 2 fields) * 4 byte per field
                if value.len() != HEADER_SIZE {
                    wb.put(COLUMN_UNCLES, key, &value[12..])?;
                }
                Ok(())
            };

            let mode = self.mode(&next_key);
            let (_count, nk) = db.traverse(COLUMN_UNCLES, &mut uncles_migration, mode, LIMIT)?;
            next_key = nk;

            if !wb.is_empty() {
                db.write(&wb)?;
                wb.clear()?;
            }
        }
        Ok(())
    }
