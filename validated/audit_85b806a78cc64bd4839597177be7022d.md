### Title
Block Status Cache Poisoning via Compact Block with Fake Extension Field — (`sync/src/relayer/mod.rs`, `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

An unprivileged relay peer can send a `CompactBlock` whose header is PoW-valid (with `extra_hash = uncles_hash`, the no-extension encoding) but whose compact block body carries an attacker-controlled non-empty `extension` field. After `reconstruct_block()` assembles a `BlockV1` with that fake extension, the contextual `BlockExtensionVerifier` computes `blake2b(uncles_hash || extension_hash) ≠ header.extra_hash` and returns `InvalidExtraHash`. The chain service then permanently writes `BLOCK_INVALID` into the block status cache keyed by the legitimate block's hash. Every subsequent relay of the real block is rejected immediately at the `contextual_check` gate, making the poisoning permanent for the lifetime of the node process.

---

### Finding Description

**Step 1 — Entry point: `CompactBlockProcess::execute()`**

`non_contextual_check()` only validates uncle count, proposal count, and block height. `contextual_check()` runs `HeaderVerifier` which checks PoW, timestamp, and version — none of these touch the extension field. `CompactBlockVerifier::verify()` only checks prefilled-transaction ordering and short-ID deduplication. [1](#0-0) [2](#0-1) 

**Step 2 — `reconstruct_block()` blindly trusts the extension**

When `compact_block.extension()` is `Some`, the function builds a `BlockV1` using the original header verbatim (not reset) and the attacker-supplied extension. The block hash therefore equals the legitimate block's hash. The only post-reconstruction check is the `transactions_root`, which the attacker satisfies by prefilling all transactions. [3](#0-2) 

**Step 3 — Non-contextual verification does not check `extra_hash`**

`BlockVerifier` runs `MerkleRootVerifier` (checks `transactions_root` and `proposals_hash` only), `CellbaseVerifier`, `DuplicateVerifier`, and `BlockBytesVerifier`. None of them inspect the `extra_hash` field against the block body. [4](#0-3) [5](#0-4) 

**Step 4 — Contextual `BlockExtensionVerifier` fails and poisons the cache**

`BlockExtensionVerifier::verify()` computes `block.calc_extra_hash().extra_hash()` from the block's actual content (uncles + extension). Because the extension is present, this yields `blake2b(uncles_hash || extension_hash)`, which does not equal `block.extra_hash()` (= `uncles_hash` from the original header). The error is not an internal DB error, so `insert_block_status(block_hash, BLOCK_INVALID)` is called unconditionally. [6](#0-5) [7](#0-6) 

**Step 5 — Legitimate block permanently rejected**

`contextual_check()` checks the block status before any reconstruction. Once `BLOCK_INVALID` is set, every future relay of the legitimate block returns `StatusCode::BlockIsInvalid` immediately. [8](#0-7)

### Impact Explanation

Any node that receives the malicious compact block before the legitimate one will permanently cache `BLOCK_INVALID` for that block hash. The node will never accept the real block from any peer, causing it to fall behind the canonical chain. An attacker targeting the current chain tip can stall victim nodes indefinitely. Because the `block_status_map` is an in-memory `DashMap` that persists for the node's lifetime, the only recovery is a node restart (which clears the in-memory cache). [11](#0-10) [12](#0-11) 

---

### Likelihood Explanation

The attacker needs only to:
1. Be a connected relay peer (no privileges required).
2. Know a valid block hash and its transactions (both are public on the P2P network).
3. Win a race against the legitimate relay — or simply connect to a node before the block is announced and send the poisoned compact block first.

The attack is repeatable at negligible cost (one P2P message per victim node per target block) and requires no hashpower, no key material, and no Sybil capability.

---

### Recommendation

Before passing the reconstructed block to `accept_block`, verify that the compact block's extension field is consistent with the header's `extra_hash`. Specifically, in `reconstruct_block()` (or immediately after), compute:

```rust
let expected_extra_hash = ExtraHashView::new(
    block.calc_uncles_hash(),
    block.calc_extension_hash(),
).extra_hash();
if expected_extra_hash != block.extra_hash() {
    return ReconstructionResult::Error(
        StatusCode::CompactBlockHasInvalidExtension.with_context(...)
    );
}
```

This check must happen **before** `accept_block` is called, so that a mismatch results in a peer ban (or silent drop) rather than a `BLOCK_INVALID` cache write keyed by the legitimate block hash. [13](#0-12)

### Proof of Concept

```
1. Observe a valid block B at height N with hash H, extra_hash = uncles_hash (no extension).
2. Craft a CompactBlockV1 message:
   - header: identical to B's header (same H, same extra_hash = uncles_hash)
   - prefilled_transactions: all of B's transactions
   - extension: any non-empty bytes (e.g., 32 zero bytes)
3. Connect to victim node as a relay peer and send the crafted CompactBlock.
4. reconstruct_block() builds BlockV1{header=B.header, extension=fake_ext}.
5. non_contextual_verify passes (MerkleRootVerifier only checks tx_root/proposals_hash).
6. BlockExtensionVerifier computes blake2b(uncles_hash || hash(fake_ext)) ≠ uncles_hash → InvalidExtraHash.
7. insert_block_status(H, BLOCK_INVALID) is written.
8. Send the legitimate compact block for B to the same node → immediately rejected with BlockIsInvalid.
```
