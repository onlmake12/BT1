### Title
Verify Queue Hard-Coded Size Limit Enables Griefing DOS on Transaction Submission — (`File: tx-pool/src/component/verify_queue.rs`)

---

### Summary

The `VerifyQueue` in CKB's tx-pool uses a hard-coded, non-configurable 256 MB size cap with **no eviction mechanism**. Any unprivileged peer or RPC caller can fill the queue with valid-looking transactions, causing all subsequent `send_transaction` RPC calls and relayed transactions from peers to be rejected with `Reject::Full` until the queue drains. This is the direct CKB analog of the M-01 deposit-function DOS: a permissionless, capacity-bounded staging area with a strict reject-on-overflow policy and no way to displace low-value entries.

---

### Finding Description

`VerifyQueue` is the pre-verification staging buffer that every incoming transaction must pass through before script execution. Its size limit is a hard-coded constant: [1](#0-0) 

The fullness check is: [2](#0-1) 

When `is_full` returns `true`, `add_tx` immediately returns `Err(Reject::Full(...))` with **no eviction**: [3](#0-2) 

This is structurally different from the main tx-pool (`pool.rs`), which **does** evict the lowest-fee-rate entries when its limit is exceeded: [4](#0-3) 

The verify queue has no such eviction path. Entries are only removed when a worker pops and processes them. There is no fee-rate ordering in the queue — entries are served FIFO by submission time: [5](#0-4) 

Both the RPC path (`notify_tx`) and the P2P relay path (`submit_remote_tx`) funnel through the same `resumeble_process_tx_and_notify_full_reject`, which propagates the `Reject::Full` error directly to the caller: [6](#0-5) 

The `Reject::Full` variant is then surfaced to the RPC caller as `PoolIsFull (-1106)`: [7](#0-6) 

---

### Impact Explanation

**Impact: Medium.** While the verify queue is full, every call to `send_transaction` (RPC) and every relayed transaction from a P2P peer returns `Reject::Full`. Legitimate users cannot submit transactions. The main tx-pool is unaffected (already-verified transactions remain), but no new transactions can enter the verification pipeline. The effect is temporary — it lasts only as long as the attacker keeps the queue saturated — but it can be sustained cheaply and continuously.

---

### Likelihood Explanation

**Likelihood: Medium.** The attack is permissionless and cheap:

- The `TRANSACTION_SIZE_LIMIT` is 512 KB per transaction. [8](#0-7) 
- At 256 MB / 512 KB ≈ 512 transactions are sufficient to saturate the queue.
- Non-contextual verification (the only check before queue admission) does not validate that inputs exist on-chain. An attacker can submit transactions referencing non-existent or already-spent inputs; they pass non-contextual checks, enter the queue, and only fail during script execution — at which point they are removed, but the attacker can immediately resubmit.
- There is no fee-rate gate on the verify queue, so the attacker pays only the cost of constructing and sending the transactions, not a meaningful fee premium.
- The only natural mitigation is that workers drain the queue concurrently, but the attacker can outpace drainage by submitting faster than the CPU-bound script verifiers can process.

---

### Recommendation

1. **Add fee-rate–based eviction to the verify queue**, mirroring the eviction logic in `pool.rs::limit_size`. When the queue is full, evict the entry with the lowest fee rate rather than rejecting the incoming transaction outright.
2. **Make `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` configurable** via `TxPoolConfig` so operators can tune it.
3. **Rate-limit per-peer queue admission**: cap the number of bytes a single peer can occupy in the verify queue at any one time, preventing a single attacker from monopolizing the buffer.

---

### Proof of Concept

1. Attacker connects to a CKB node (or uses the RPC endpoint directly).
2. Attacker constructs ~512 transactions, each close to 512 KB in serialized size, referencing inputs that may or may not exist (non-contextual verification does not check input existence).
3. Attacker submits all transactions via `send_transaction` RPC or via P2P relay.
4. `VerifyQueue::add_tx` accepts each until `total_tx_size >= 256_000_000`.
5. Any subsequent `send_transaction` call from a legitimate user returns `PoolIsFull (-1106)`.
6. Attacker monitors the queue size (via `get_pool_info` → `verify_queue_size`) and resubmits new transactions as workers drain old ones, sustaining the DOS.

The hard-coded constant and the absence of eviction in `add_tx` are the necessary vulnerable steps: [9](#0-8)

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
