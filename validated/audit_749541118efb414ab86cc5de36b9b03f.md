Audit Report

## Title
Stale Inflated Descendant-Weight Accounting After `remove_entry_and_descendants` - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::remove_entry_and_descendants`, all transaction links are torn down via `remove_entry_links` for every removed ID before `remove_entry` is called. Because `update_ancestors_index_key` inside `remove_entry` relies on `self.links.calc_ancestors` to find surviving pool entries needing their `descendants_*` fields decremented, pre-removing the links causes `calc_ancestors` to return an empty set. Any ancestor of the removed chain that remains in the pool retains permanently inflated `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` values, corrupting `EvictKey` computation for the rest of its lifetime in the pool.

## Finding Description
`remove_entry_and_descendants` ( [1](#0-0) ) first iterates all removed IDs and calls `remove_entry_links` on each ( [2](#0-1) ), which removes the entry from `self.links` entirely via `self.links.remove(id)` ( [3](#0-2) ). It then calls `remove_entry` for each ID ( [4](#0-3) ).

Inside `remove_entry`, `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` is called at L242 ( [5](#0-4) ). This function calls `self.links.calc_ancestors(&child.proposal_short_id())` ( [6](#0-5) ). In `calc_relative_ids`, the lookup `self.inner.get(short_id)` returns `None` because `remove_entry_links` already called `self.links.remove(id)` — so `direct` is an empty `HashSet`, `calc_relation_ids` returns an empty set, and the loop body never executes ( [7](#0-6) ). `sub_descendant_weight` is never called on any surviving ancestor.

The existing test `test_remove_entry_and_descendants` only asserts that removed entries are absent and that the links set is empty; it does not assert that tx1's `descendants_count`, `descendants_size`, `descendants_cycles`, or `descendants_fee` are decremented after the call ( [8](#0-7) ), so the bug is not caught.

## Impact Explanation
The stale `descendants_*` fields directly corrupt `EvictKey` computation: `EvictKey::fee_rate` is set to `max(descendants_feerate, own_feerate)`, where `descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight)` ( [9](#0-8) ). An ancestor with artificially inflated `descendants_fee` and `descendants_size`/`descendants_cycles` will show a higher `descendants_feerate` than it actually has, making it appear more valuable and less likely to be evicted when `limit_size` fires. This constitutes a **Suboptimal implementation of CKB state storage mechanism**, matching the **Medium (2001–10000 points)** impact class.

## Likelihood Explanation
`remove_entry_and_descendants` is called on every common pool-management event: `resolve_conflict` (every committed block with a conflicting tx), `resolve_conflict_header_dep` (every block invalidating a header dep), `limit_size` (pool overflow), and `remove_by_detached_proposal` (every reorg). Any unprivileged user can trigger the bug: submit parent P and child C, then submit a double-spend of C's input. When the conflict is resolved, `remove_entry_and_descendants(C)` is called, P's `descendants_*` fields are never decremented, and P retains inflated eviction resistance. The attack is cheap, repeatable, and requires no special privileges.

## Recommendation
Move the ancestor-weight update before the links are torn down. Collect surviving ancestors for each removed entry first, then remove links, then call a variant of `remove_entry` that skips the ancestor update (since it was already done):

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Update surviving ancestors BEFORE removing links
    for rid in &removed_ids {
        if let Some(entry) = self.get(rid).cloned() {
            self.update_ancestors_index_key(&entry, EntryOp::Remove);
        }
    }

    // Now safe to remove links
    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry_skip_ancestor_update(id))
        .collect()
}
```

Alternatively, add a boolean flag to `remove_entry` to skip `update_ancestors_index_key` when called from a bulk-removal context, while performing the ancestor update in the pre-pass above.

## Proof of Concept
Using the existing test infrastructure in `tx-pool/src/component/tests/score_key.rs` ( [10](#0-9) ):

1. Build chain: tx1 (root) → tx2 (child of tx1) → tx3 (child of tx2), all with distinct fees.
2. `add_proposed` all three. Verify tx1's `descendants_count == 3`.
3. Call `map.remove_entry_and_descendants(&tx2_id)`.
4. Assert `map.contains_key(&tx1_id)` is true (tx1 survives).
5. Assert `map.get(&tx1_id).unwrap().descendants_count == 1` — **this assertion will fail**, returning `3` instead of `1`.
6. Assert `map.get(&tx1_id).unwrap().descendants_fee == tx1.fee` — **this assertion will also fail**, retaining the sum of all three fees.

The existing test at L224–229 does not check these fields, confirming the bug is untested and present.

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =

            // update parent score
