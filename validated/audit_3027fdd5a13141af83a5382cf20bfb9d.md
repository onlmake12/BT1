### Title
Zero-Price Feeds Silently Bypass Deviation Check, Blocking Critical Price Updates — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `_validateShouldUpdatePrices` function silently skips the deviation check for any price feed whose current or previous stored price is zero. When a subscription is configured with only deviation-based update criteria (`updateOnDeviation = true`, `updateOnHeartbeat = false`), a valid Pyth update carrying a zero price is unconditionally rejected. Downstream consumers reading from the Scheduler continue to receive the stale pre-crash price indefinitely.

---

### Finding Description

Inside `_validateShouldUpdatePrices`, the deviation loop contains an explicit guard:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
``` [1](#0-0) 

If every feed in the subscription hits this guard (e.g., all current prices are zero after a market crash, or all stored prices are zero after a first-update anomaly), the loop exhausts without ever returning `updateTimestamp`, and execution falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [2](#0-1) 

The Pyth core contract itself does not reject zero prices — `parsePriceFeedUpdatesWithConfig` accepts any `int64` price value, including zero. [3](#0-2) 

The result is that a zero price — which represents a **maximum possible deviation** from any non-zero previous price — is treated as if no deviation occurred at all.

---

### Impact Explanation

Any DeFi protocol (lending, derivatives, liquidation engine) that reads prices from the Scheduler via `getPricesUnsafe` or `getPricesNoOlderThan` will continue to receive the last non-zero stored price even after the underlying asset's Pyth feed has reported zero. [4](#0-3) 

For a lending protocol, this means collateral continues to be valued at the pre-crash price, preventing correct liquidations. For a derivatives protocol, positions are settled at a fabricated price. The stale price persists until either (a) a heartbeat interval fires (if `updateOnHeartbeat` is also enabled) or (b) the price recovers to a non-zero value.

---

### Likelihood Explanation

- Any subscription configured with `updateOnDeviation = true` and `updateOnHeartbeat = false` is affected.
- The Pyth network legitimately publishes zero prices for assets during market halts, circuit breakers, or when publisher quorum is insufficient.
- Any unprivileged keeper calling `updatePriceFeeds` with a valid Pyth update containing a zero price will trigger the revert path — no special access is required.
- The `SchedulerConstants` define a `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`, meaning the stale price window can persist for the full validity period before any other mechanism intervenes. [5](#0-4) 

---

### Recommendation

Replace the `continue` with a positive trigger: a zero current price relative to a non-zero previous price (or vice versa) represents an infinite deviation and should unconditionally satisfy the deviation threshold:

```solidity
// A zero price represents maximum deviation — always trigger update
if (previousPrice == 0 || currentPrice == 0) {
    if (previousPrice != currentPrice) {
        return updateTimestamp;
    }
    continue; // both zero: no change, skip
}
```

Alternatively, treat `currentPrice == 0` as an always-trigger condition regardless of `previousPrice`.

---

### Proof of Concept

1. Deploy `Scheduler` with a subscription configured as:
   - `updateOnHeartbeat = false`
   - `updateOnDeviation = true`, `deviationThresholdBps = 100` (1%)
2. Submit a first update with `price = 1_000_000` for feed `F`. This stores the price and sets `priceLastUpdatedAt`.
3. Advance time by 1 second (heartbeat not applicable).
4. Submit a second update with `price = 0` for feed `F` (valid Pyth accumulator update, signed by Wormhole guardians).
5. Observe: `updatePriceFeeds` reverts with `UpdateConditionsNotMet`.
6. Call `getPricesUnsafe` — returns `price = 1_000_000` (stale pre-crash value).
7. A downstream lending protocol using this Scheduler continues to value the asset at `1_000_000` instead of `0`, blocking correct liquidations. [6](#0-5)
