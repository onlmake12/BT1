Audit Report

## Title
`max_tx_verify_cycles` Ignored for Local RPC Submissions, Allowing Full `max_block_cycles` CPU Budget — (`tx-pool/src/process.rs`)

## Summary
In `TxPoolService::_process_tx`, when a transaction is submitted via the local RPC `send_transaction` endpoint, `declared_cycles` is `None`, causing the cycle ceiling to fall back to `consensus.max_block_cycles()` (~3.5 B on mainnet) instead of the operator-configured `TxPoolConfig.max_tx_verify_cycles` (default 70 M). The configured limit is never read in this code path, allowing any local process to force the verification thread to execute scripts for up to ~50× longer than the operator intended.

## Finding Description
In `_process_tx` (`tx-pool/src/process.rs:720`):

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
```

For local RPC submissions, `declared_cycles` is `None`, so `max_cycles` becomes `max_block_cycles()`. This value is forwarded directly to `verify_rtx` (`tx-pool/src/process.rs:724–732`), which passes it as the hard ceiling to `ContextualTransactionVerifier::verify` and `verify_with_pause` (`tx-pool/src/util.rs:108, 119`).

The field `self.tx_pool_config.max_tx_verify_cycles` is never read in this code path. By contrast, the async remote-tx path correctly initializes `VerifyQueue` with `self.tx_pool_config.max_tx_verify_cycles` (`tx-pool/src/service.rs:576–578`), creating an inconsistency between the two submission paths.

## Impact Explanation
**Low (501–2000 points): Important performance improvement for CKB.**

Any local process with RPC access can submit transactions whose scripts consume up to 3.5 B cycles. The node executes those scripts in full before rejecting or accepting them, consuming ~50× more CPU per submission than the operator configured. Repeated submissions can saturate the verification thread pool and delay block assembly. This does not crash the node, cause consensus deviation, or affect the broader network.

## Likelihood Explanation
The RPC is bound to `127.0.0.1:8114` by default, so only local processes can reach this path. No credentials are required. Any script, compromised co-located service, or user with shell access on the same host can trigger this repeatedly with no rate limiting beyond the verification thread pool itself.

## Recommendation
Replace the fallback in `_process_tx` with the configured limit:

```diff
- let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
+ let max_cycles = declared_cycles.unwrap_or(self.tx_pool_config.max_tx_verify_cycles);
```

This mirrors the pattern already used for the async remote-tx path via `VerifyQueue` and ensures consistent enforcement regardless of submission origin.

## Proof of Concept
1. Start a CKB node with default config (`max_tx_verify_cycles = 70_000_000`).
2. Craft a transaction whose lock script loops consuming close to `max_block_cycles` (3.5 B) cycles before returning success.
3. Submit via `send_transaction` RPC: `curl -X POST http://127.0.0.1:8114 -d '{"method":"send_transaction",...}'`.
4. Observe via CPU profiling or timing that the verification thread runs for the full `max_block_cycles` budget, not the 70 M limit.
5. Repeat in a loop to saturate the verification thread pool and measure block assembly delay.

Root cause confirmed at `tx-pool/src/process.rs:720`; cycle ceiling applied without modification in `tx-pool/src/util.rs:108` and `119`.
