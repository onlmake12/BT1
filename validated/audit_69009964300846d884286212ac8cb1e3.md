Audit Report

## Title
Unguarded `tip_synced()` Transition Allows `n_sync_started` Underflow, Permanently Stalling IBD Header Sync — (`sync/src/synchronizer/headers_process.rs`, `sync/src/types/mod.rs`)

## Summary
The `HeadersSyncState` machine has no enforced transition guards on the `tip_synced()` path. Any connected peer can send an empty `SendHeaders` message before `start_sync_headers` selects it, causing `tip_synced()` to be called when the peer is still in `SyncProtocolConnected` state. Because `n_sync_started` was never incremented for this peer, the symmetric decrement inside `SyncState::tip_synced()` wraps the `AtomicUsize` to `usize::MAX`. In IBD mode, the guard `if ibd && x != 0 { None }` then permanently prevents any peer from being selected for header sync, stalling the node indefinitely.

## Finding Description
`HeadersSyncState` defines five states with an intended progression: `Initialized → SyncProtocolConnected → Started → TipSynced/Suspend`. The transition methods on `ChainSyncState` are unconditional — they overwrite the current state with no check of what state the machine is in.

In `HeadersProcess::execute()`, when a peer sends a `SendHeaders` message with zero headers, `tip_synced()` is called on the peer state with no guard on the current `HeadersSyncState`:

```rust
if headers.is_empty() {
    if let Some(mut state) = self.synchronizer.peers().state.get_mut(&self.peer) {
        self.synchronizer.shared().state().tip_synced(state.value_mut());
    }
    return Status::ok();
}
```

The same unconditional call occurs when a non-full batch of headers is received (fewer than `MAX_HEADERS_LEN`).

Compare this to the guarded `suspend_sync` call in `eviction()`, which correctly checks `sync_started()` before transitioning — no equivalent guard exists for `tip_synced()`.

`n_sync_started` is incremented in `start_sync_headers` before `start_sync()` is called on the peer. In `Peers::disconnected()`, the counter is decremented only when `sync_started()` returns true (state is `Started`). Since `tip_synced()` moves state to `TipSynced`, `disconnected()` will not decrement the counter for a normally-completing peer. Therefore `SyncState::tip_synced()` must itself decrement `n_sync_started` as the symmetric operation. Calling it for a peer that was never in `Started` state (counter never incremented for it) causes `AtomicUsize` wrapping underflow to `usize::MAX`.

With `n_sync_started == usize::MAX`, the `fetch_update` closure in `start_sync_headers` evaluates `if ibd && x != 0 { None }` as always-`None`, causing `is_err()` to be true and `break` to fire for every candidate peer. No peer can ever be selected for header sync again. Since the node cannot exit IBD without syncing headers, the stall is permanent.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

A single attacker-controlled peer that connects and immediately sends one empty `SendHeaders` message permanently stalls the victim node in IBD. The node can no longer sync headers with any peer, cannot advance its chain tip, and cannot exit IBD. This is equivalent to a remote denial-of-service that permanently disables chain synchronization for the targeted node.

## Likelihood Explanation
The attack requires only a standard P2P connection to the target node and sending a single `SendHeaders` message with an empty payload. This is achievable by any unprivileged network peer with no special keys, hashpower, or privileges. The `received` handler dispatches `SendHeaders` to `HeadersProcess::execute()` for any connected peer without pre-checking the peer's current sync state. The attack is repeatable and requires no victim interaction or mistake.

## Recommendation
Add an explicit state guard in `HeadersProcess::execute()` before calling `tip_synced()`, mirroring the existing guard in `eviction()`:

```rust
if let Some(mut state) = self.synchronizer.peers().state.get_mut(&self.peer) {
    if state.sync_started() {
        self.synchronizer.shared().state().tip_synced(state.value_mut());
    }
}
```

Apply the same guard to the non-full-batch `tip_synced()` call at the end of `execute()`. This ensures `tip_synced()` is only called when the peer is in `Started` state (i.e., `n_sync_started` was actually incremented for it), preventing underflow.

## Proof of Concept
1. Attacker establishes a P2P connection to a CKB node running in IBD mode.
2. The node calls `on_connected` → `sync_connected()` → peer state is `SyncProtocolConnected`. `n_sync_started` is still 0 for this peer.
3. Before `start_sync_headers` selects this peer (which would set state to `Started` and increment `n_sync_started`), the attacker sends a `SendHeaders` message with 0 headers.
4. `HeadersProcess::execute()` receives the empty message, enters the `headers.is_empty()` branch, and calls `self.synchronizer.shared().state().tip_synced(state.value_mut())` unconditionally.
5. The peer state transitions `SyncProtocolConnected → TipSynced` without ever passing through `Started`. `n_sync_started` was never incremented for this peer.
6. `SyncState::tip_synced()` decrements `n_sync_started` (the symmetric counterpart to the increment in `start_sync_headers`). The counter underflows from 0 to `usize::MAX`.
7. In `start_sync_headers`, the IBD guard `if ibd && x != 0 { None }` now always fires for every peer, and no peer can ever be selected for header sync again — the node is permanently stalled in IBD.
