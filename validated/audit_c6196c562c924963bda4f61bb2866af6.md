### Title
Collected Update Fees Are Permanently Locked in `PriceInfoObject` With No Protocol-Accessible Withdrawal Path — (`target_chains/sui/contracts/sources/price_info.move`, `pyth.move`)

---

### Summary

Every call to `update_single_price_feed` deposits `Coin<SUI>` into a `PriceInfoObject` via `price_info::deposit_fee_coins`. No function exists anywhere in the Sui contract codebase to withdraw those coins. The `fee_recipient_address` stored in `State` is explicitly annotated as vestigial ("not used anymore"). All collected fees are permanently and irrecoverably locked inside shared `PriceInfoObject` objects.

---

### Finding Description

**Step 1 — Fee deposit is unconditional and permanent.**

`update_single_price_feed` deposits the fee coin at line 277, before the freshness check inside `update_cache`: [1](#0-0) 

The fee is deposited regardless of whether the price data is fresh. If the update is stale, `update_cache` silently skips the write but the fee is already committed to the object.

**Step 2 — `price_info.move` has no withdrawal function.**

The complete public surface of `price_info.move` exposes only `deposit_fee_coins` (write) and `get_balance` (read) for the fee balance field. There is no `withdraw_fee_coins`, `transfer_fees`, or equivalent: [2](#0-1) 

**Step 3 — The governance system has no fee-sweep action.**

The complete set of governance actions is enumerated in `governance_action.move` (actions 0–5: `CONTRACT_UPGRADE`, `SET_GOVERNANCE_DATA_SOURCE`, `SET_DATA_SOURCES`, `SET_UPDATE_FEE`, `SET_STALE_PRICE_THRESHOLD`, `SET_FEE_RECIPIENT`): [3](#0-2) 

None of these actions touch the `Coin<SUI>` balance inside any `PriceInfoObject`. The `execute_governance_instruction` dispatcher in `governance.move` exhausts all possible actions with no fee-sweep branch: [4](#0-3) 

**Step 4 — `fee_recipient_address` is explicitly dead code.**

The `set_fee_recipient` governance module carries a comment confirming the mechanism was removed: [5](#0-4) 

`state::set_fee_recipient` only updates the address field in `State`; it never transfers any balance out of any `PriceInfoObject`: [6](#0-5) 

---

### Impact Explanation

Every `update_single_price_feed` call permanently locks `base_update_fee` SUI (or more, since the check is `<=`) inside the target `PriceInfoObject`. Over the lifetime of the protocol, across all price feeds and all relayers, the total locked SUI grows without bound. Neither the protocol, nor governance, nor the fee recipient can ever recover these funds. The fee model is economically broken: fees are collected but irrecoverable.

---

### Likelihood Explanation

This triggers on every legitimate price update call — it is not a corner case. The Sui SDK (`js/src/client.ts`) calls `update_single_price_feed` for every feed in every update batch: [7](#0-6) 

The condition is already met in production with every relay transaction.

---

### Recommendation

1. Add a `withdraw_fee_coins` function (gated to `LatestOnly` or a governance-controlled capability) to `price_info.move` that extracts the `Coin<SUI>` dynamic object field and transfers it to `fee_recipient_address`.
2. Add a corresponding governance action (e.g., `SWEEP_FEES`) that calls this function for a given `PriceInfoObject`.
3. Alternatively, redirect fees at deposit time to a central treasury object rather than into each individual `PriceInfoObject`.

---

### Proof of Concept

Invariant test (pseudocode):

```move
// After N calls to update_single_price_feed:
let total_paid = N * base_update_fee;
assert!(price_info::get_balance(&price_info_object) == total_paid);

// Attempt to reduce balance to zero via any public or governance path:
// -> No such function exists in price_info.move
// -> No governance action sweeps PriceInfoObject balances
// -> fee_recipient_address has no claim mechanism over PriceInfoObject
assert!(price_info::get_balance(&price_info_object) == total_paid); // unchanged
```

The test at line 817 of `pyth.move` already confirms the accumulation half of this invariant: [8](#0-7) 

No corresponding test or production function exists to reduce that balance.

    public fun get_balance(price_info_object: &PriceInfoObject): u64 {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            return 0
        };
        let fee = dynamic_object_field::borrow<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY);
        coin::value(fee)
    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
