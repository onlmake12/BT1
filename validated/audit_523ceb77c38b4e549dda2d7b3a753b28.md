### Title
DAO Withdrawal Witness Index Truncation Causes Consensus Split Between C VM and Rust Verifier — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw` reads the header-dep index from the witness as a full `u64`, while the on-chain C DAO script (`dao.c`) reads only the lowest byte of that same field. An attacker who holds any DAO deposit can craft a withdrawal transaction with ≥ 258 `header_deps` and a witness index whose lowest byte points to the correct deposit block but whose full `u64` value points to a different block. The C VM accepts the transaction; Rust's `FeeCalculator` rejects it with `DaoError::InvalidOutPoint`. Because `FeeCalculator::transaction_fee` is called inside `ContextualTransactionVerifier::verify` during block verification, any block containing this transaction is rejected by all Rust nodes even though the on-chain script passed — a consensus split.

---

### Finding Description

**Root cause — index width mismatch**

`DaoCalculator::transaction_maximum_withdraw` in `util/dao/src/lib.rs` reads the 8-byte little-endian witness field and uses the full `u64` value to index into `header_deps()`: [1](#0-0) 

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// …
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // full u64 used here
```

The on-chain C DAO script, however, reads the same field using only its lowest byte (documented in the test comment at `util/dao/src/tests.rs:490`):

> "Position 1: correct deposit block (what C VM resolves via lowest byte). Position 257: withdraw block (wrong — Rust resolves this with full u64)." [2](#0-1) 

**Block verification path**

`FeeCalculator::transaction_fee` calls `DaoCalculator::transaction_fee` → `transaction_maximum_withdraw`: [3](#0-2) 

This is called unconditionally (after script execution) inside `ContextualTransactionVerifier::verify`: [4](#0-3) 

`BlockTxsVerifier::verify` calls `ContextualTransactionVerifier::verify` for every non-cellbase transaction in a block: [5](#0-4) 

Any error propagated via `?` causes the entire block to be rejected.

**Exploit construction**

1. Attacker holds a DAO deposit at block `N`.
2. Attacker builds a withdrawal transaction with 258 `header_deps`:
   - `header_deps[1]` = deposit block hash (block `N`)
   - `header_deps[257]` = withdraw block hash (block `M`, `M > N`)
   - All other slots = dummy hashes
3. Witness `input_type` = `257u64` in little-endian (8 bytes).
4. Cell data = `N` (deposit block number).

**C VM path:** reads lowest byte of `257` → `1` → `header_deps[1]` = deposit block → block number `N` matches cell data → script passes.

**Rust path:** reads full `u64` `257` → `header_deps[257]` = withdraw block → block number `M ≠ N` → `DaoError::InvalidOutPoint` returned at line 106: [6](#0-5) 

The test `check_dao_withdraw_header_dep_index_exceeds_u8` confirms Rust returns an error for exactly this construction: [7](#0-6) 

---

### Impact Explanation

A block containing this transaction is accepted by the C VM (on-chain script passes) but rejected by every Rust node during `ContextualTransactionVerifier::verify`. This is a **consensus split**: Rust nodes and any non-Rust implementation that faithfully implements the C DAO script's byte-truncation behavior diverge on chain tip. A sustained split enables double-spend attacks against exchanges and services that rely on Rust node confirmations.

---

### Likelihood Explanation

The attacker only needs:
- Any DAO deposit (no minimum amount).
- Ability to submit a transaction to the network (standard RPC: `send_transaction`).
- A miner willing to include the transaction (the transaction passes C VM script verification, so a miner running a non-Rust client or a patched client would include it).

The 258-entry `header_deps` array is unusual but not blocked by any non-contextual check. The `check_tx_fee` path in the tx-pool (`tx-pool/src/util.rs:28-54`) also calls `DaoCalculator::transaction_fee`, so Rust nodes would reject the transaction at the mempool level — but a miner bypassing the mempool (e.g., via `generate_block_with_template` or a non-Rust miner) can still include it. [8](#0-7) 

---

### Recommendation

1. **Align the Rust index reader with the C VM**: if `dao.c` reads only the lowest byte of the 8-byte witness field, `DaoCalculator::transaction_maximum_withdraw` must do the same — replace `LittleEndian::read_u64(...)` with a byte-masked read (`& 0xFF`) or document and enforce that the index must fit in a `u8`.
2. **Add a non-contextual validation** that rejects any DAO withdrawal transaction whose witness index ≥ 256 (or whatever the C VM's actual limit is), so the discrepancy is caught before block inclusion.
3. **Add a consensus test** that submits this crafted transaction through the full block-verification pipeline (not just `DaoCalculator` in isolation) to catch future regressions.

---

### Proof of Concept

The existing unit test at `util/dao/src/tests.rs:475-537` already demonstrates the Rust-side rejection. To confirm the consensus split, extend it to:

1. Build the crafted transaction (258 `header_deps`, witness index `257`).
2. Run `ContextualTransactionVerifier::verify` with `skip_script_verify = false` against a mock data loader that returns the C VM's accepted result for the script step.
3. Observe that `fee_calculator.transaction_fee()` returns `Err(DaoError::InvalidOutPoint)` even though the script step passed — confirming the block would be rejected by Rust nodes. [9](#0-8) [10](#0-9) [4](#0-3)

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
