### Title
`MAX_DEPOSIT_LIMIT` Checked Against Per-Transaction `msg.value` Instead of Cumulative `balanceInWei` in Permanent Subscriptions — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract enforces a `MAX_DEPOSIT_LIMIT` (100 ETH) on permanent subscriptions to prevent excessive ETH from being permanently locked. However, both `createSubscription` and `addFunds` check this limit against the single-transaction `msg.value` rather than the cumulative `status.balanceInWei`. As a result, any caller can bypass the cap by making multiple deposits, each just at or below the limit, causing the permanent subscription's balance to grow unboundedly beyond `MAX_DEPOSIT_LIMIT`. Since permanent subscriptions prohibit withdrawals, all excess ETH is permanently locked.

---

### Finding Description

In `Scheduler.sol`, the deposit limit for permanent subscriptions is enforced as follows in `createSubscription`:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

And in `addFunds`:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

The check compares only the current transaction's `msg.value` against `MAX_DEPOSIT_LIMIT` (100 ETH), not the resulting cumulative balance (`status.balanceInWei + msg.value`). Because `addFunds` has **no access control** (no `onlyManager` modifier), any external caller can invoke it for any active permanent subscription.

A caller can therefore:
1. Call `addFunds` with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH) — passes the check.
2. Call `addFunds` again with `msg.value = MAX_DEPOSIT_LIMIT` — passes again.
3. Repeat indefinitely.

After N calls, `status.balanceInWei = N × MAX_DEPOSIT_LIMIT`, far exceeding the intended cap. Since `withdrawFunds` unconditionally reverts for permanent subscriptions:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

all deposited ETH beyond the intended limit is permanently locked in the contract.

---

### Impact Explanation

- The `MAX_DEPOSIT_LIMIT` protection for permanent subscriptions is entirely ineffective; it can be bypassed by splitting deposits across multiple transactions.
- Any third party (not just the subscription manager) can call `addFunds` on a permanent subscription, permanently locking their own ETH in it. While the attacker bears the cost, this is a reachable, unprivileged path that causes irreversible fund loss.
- The subscription manager themselves can also bypass the limit intentionally or accidentally, permanently locking more ETH than intended with no recourse.
- The invariant that permanent subscriptions are capped at `MAX_DEPOSIT_LIMIT` is broken, undermining the safety guarantee the constant is designed to provide.

---

### Likelihood Explanation

- `addFunds` is a public, payable function with no access control — any EOA or contract can call it for any active permanent subscription.
- The bypass requires only multiple standard ETH transfers, each ≤ 100 ETH. No special privileges, keys, or oracle manipulation are needed.
- A subscription manager who wants to deposit more than 100 ETH into their own permanent subscription can trivially do so by splitting the deposit.

---

### Recommendation

Change the deposit limit check in both `createSubscription` and `addFunds` to compare against the **resulting cumulative balance** rather than the per-transaction amount:

```solidity
// In addFunds (after incrementing):
status.balanceInWei += msg.value;
if (params.isPermanent && status.balanceInWei > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Or equivalently, check before incrementing:

```solidity
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

Additionally, consider adding an `onlyManager` modifier to `addFunds` for permanent subscriptions to prevent third parties from depositing into subscriptions they do not own.

---

### Proof of Concept

```solidity
// Assume MAX_DEPOSIT_LIMIT = 100 ether
// Attacker or manager calls addFunds twice, each with 100 ETH

scheduler.addFunds{value: 100 ether}(permanentSubscriptionId);
// passes: msg.value (100 ETH) == MAX_DEPOSIT_LIMIT, not strictly greater

scheduler.addFunds{value: 100 ether}(permanentSubscriptionId);
// passes again: msg.value (100 ETH) == MAX_DEPOSIT_LIMIT

// status.balanceInWei is now 200 ETH, double the intended cap
// withdrawFunds reverts for permanent subscriptions → 200 ETH permanently locked
```
