### Title
Orphan Transactions with Unknown Cell Deps Are Never Promoted — (`tx-pool/src/component/orphan.rs`, `tx-pool/src/util.rs`)

---

### Summary

The CKB tx-pool's `is_missing_input` function treats any `OutPointError::Unknown` as a "missing input," including unknown **cell deps**. However, the `OrphanPool` only indexes orphan transactions by their **input** out-points. When a transaction with an unknown cell dep is added to the orphan pool, it is never indexed by the cell dep's out-point, so `find_by_previous` can never locate it when the cell dep's parent transaction arrives. The orphan is permanently stuck until it expires, wasting orphan pool slots and silently dropping the transaction.

---

### Finding Description

**Root cause — `is_missing_input` is too broad:**

`is_missing_input` in `tx-pool/src/util.rs` returns `true` for any `Reject::Resolve(OutPointError::Unknown(_))`: [1](#0-0) 

`resolve_transaction` in `util/types/src/core/cell.rs` emits `OutPointError::Unknown` for **both** unknown inputs and unknown cell deps, because the same `CellStatus::Unknown` branch is hit in both cases: [2](#0-1) 

**Root cause — `add_orphan_tx` only indexes by inputs:**

When `is_missing_input` returns `true`, `after_process` calls `add_orphan(tx, peer, declared_cycle)`, which calls `add_orphan_tx`. That function indexes the orphan **only by input out-points**: [3](#0-2) 

Cell dep out-points are never inserted into `by_out_point`.

**Root cause — `find_by_previous` only searches by input out-points:**

When a new transaction is accepted, `process_orphan_tx` calls `find_by_previous`, which looks up orphans by the new transaction's **output** out-points matched against `by_out_point`: [4](#0-3) 

Because the orphan with an unknown cell dep was never indexed by the cell dep's out-point, `find_by_previous` returns nothing when the cell dep's parent transaction arrives. The orphan is never promoted.

**The full flow in `after_process`:** [5](#0-4) 

---

### Impact Explanation

1. **Orphan pool slot exhaustion (DoS):** A remote peer can relay transactions whose cell deps reference non-existent cells. Each such transaction occupies one of the 100 available orphan slots (`DEFAULT_MAX_ORPHAN_TRANSACTIONS`). When the pool is full, legitimate orphan transactions are evicted randomly. [6](#0-5) 

2. **Silent transaction loss:** A legitimate user who submits a transaction with a cell dep that is in-flight (not yet confirmed) will have their transaction silently stuck in the orphan pool and eventually dropped after `ORPHAN_TX_EXPIRE_TIME` (100 × `MAX_BLOCK_INTERVAL`), even though the cell dep later becomes available. [7](#0-6) 

---

### Likelihood Explanation

The attacker-controlled entry path is the relay protocol: any unprivileged P2P peer can send a `RelayTransaction` message containing a transaction whose `cell_deps` reference a non-existent out-point. The node will accept it into the orphan pool (because `is_missing_input` returns `true`) and never evict it via the normal promotion path. This requires no special privilege, no key material, and no majority hashpower.

---

### Recommendation

1. **Distinguish unknown inputs from unknown cell deps.** Introduce a separate error variant (e.g., `OutPointError::UnknownDep`) for cell dep resolution failures, and do not treat it as a missing input.

2. **Index orphans by cell dep out-points as well.** If the design intent is to support orphans waiting for cell deps, `add_orphan_tx` must also insert entries into `by_out_point` for each cell dep out-point, and `find_by_previous` must search those entries too.

3. **Reject transactions with unknown cell deps immediately.** The simpler fix is to not add transactions with unknown cell deps to the orphan pool at all — only unknown inputs should trigger orphan behavior.

---

### Proof of Concept

1. Attacker constructs a transaction `T` with a valid input (a live cell the attacker owns) and a `cell_dep` pointing to a non-existent out-point `X`.
2. Attacker relays `T` to a CKB node via the relay protocol.
3. `resolve_transaction` fails with `OutPointError::Unknown(X)` on the cell dep.
4. `is_missing_input` returns `true` → `T` is added to the orphan pool via `add_orphan_tx`.
5. `add_orphan_tx` indexes `T` only by its input out-point, not by `X`.
6. Attacker repeats with 100 different transactions, filling the orphan pool.
7. Legitimate orphan transactions from honest peers are evicted randomly.
8. Even if the cell at out-point `X` is later created on-chain, `find_by_previous` never finds `T` because `X` is not in `by_out_point`.
9. `T` expires after `ORPHAN_TX_EXPIRE_TIME` and is silently dropped.
