Audit Report

## Title
Off-by-One in `MAX_VMS_COUNT` Guard Allows 17th VM to Boot — (`script/src/scheduler.rs`)

## Summary

The `Message::Spawn` guard in `process_message_box` uses strict `>` instead of `>=` when comparing the live VM count against `MAX_VMS_COUNT = 16`. When exactly 16 VMs are live, `16 > 16` evaluates to `false`, the guard does not fire, and `boot_vm` is called unconditionally, inserting a 17th VM. The stated invariant "total live VMs ≤ 16" is violated by exactly one.

## Finding Description

`MAX_VMS_COUNT` is defined as `16` at `script/src/scheduler.rs:34`. In `process_message_box`, the spawn guard at line 532 is:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
```

When the combined count of `suspended` + `instantiated` equals exactly 16, `16 > 16` is `false`, so the guard does not return `MAX_VMS_SPAWNED`. Execution falls through to `boot_vm` at line 539, which unconditionally calls `self.instantiated.insert(id, ...)` and `self.states.insert(id, VmState::Runnable)` at lines 1035–1036, creating a 17th VM. Only on the *next* spawn attempt does `17 > 16` evaluate to `true` and correctly block. The scheduler holds exclusive mutable access to `self` throughout; there is no concurrent modification. This is a plain off-by-one.

**Exploit path:**
1. Root VM (1 live VM) issues 15 `Message::Spawn` syscalls → 16 live VMs total.
2. One more `Message::Spawn` is issued. Guard: `16 > 16` = `false` → `boot_vm` called → 17th VM created.
3. Subsequent spawn: `17 > 16` = `true` → correctly blocked.

## Impact Explanation

**High — Incorrect implementation or behavior of CKB-VM or system scripts.** The enforced invariant that at most 16 VMs may exist simultaneously is concretely violated. A script running under `ScriptVersion::V2` can exceed the documented VM count limit by one. While all nodes share the same off-by-one (preventing consensus deviation), the CKB-VM specification is broken: the limit is not enforced at the boundary it claims to enforce.

## Likelihood Explanation

Any `ScriptVersion::V2` script submitted in a transaction can trigger this. No privileged access, leaked keys, or majority hashpower is required. The path is fully deterministic, locally reproducible, and requires only 16 sequential `spawn` syscalls from a root VM — a trivially constructable script.

## Recommendation

Change the guard operator from `>` to `>=` at `script/src/scheduler.rs:532`:

```rust
// Before (off-by-one):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

## Proof of Concept

```rust
// In scheduler test suite:
// Root VM = 1 live VM. Spawn 15 children → 16 total.
for _ in 0..15 {
    assert_eq!(spawn_child(), SUCCESS);
}
// 16th spawn: guard 16 > 16 = false → boot_vm called → 17th VM inserted
assert_eq!(spawn_child(), MAX_VMS_SPAWNED); // FAILS: returns SUCCESS instead
// Invariant check:
assert!(
    scheduler.suspended.len() + scheduler.instantiated.len() <= MAX_VMS_COUNT as usize
); // FAILS: actual count is 17
```

A proptest/fuzzer asserting `spawned_count <= MAX_VMS_COUNT` after each successful spawn would catch this immediately on the 16th successful spawn from the root VM.
