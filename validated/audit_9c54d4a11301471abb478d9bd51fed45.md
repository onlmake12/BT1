Audit Report

## Title
`Scheduler::resume()` Unconditionally Zeroes `iteration_cycles`, Discarding IO-Charged Cycles Not Yet Committed to `total_cycles` — (File: `script/src/scheduler.rs`)

## Summary

`iterate_outer` resets `iteration_cycles` to zero, then calls `process_io()` which can write IO-operation cycles back into `iteration_cycles`. When a `Pause` error propagates immediately after, those IO cycles are in `iteration_cycles` but not in `total_cycles`. `suspend()` faithfully saves the non-zero `iteration_cycles`, but `resume()` unconditionally zeroes it at line 242, permanently discarding those cycles from all accounting. This allows a script to consume more cycles than the enforced limit across each suspend/resume boundary.

## Finding Description

In `iterate_outer` (lines 417–456):

1. `iterate_inner` runs the VM; result stored in `iterate_return`.
2. Line 423: `self.consume_cycles(self.iteration_cycles)` — commits VM-execution cycles into `total_cycles`.
3. Line 428: `self.iteration_cycles = 0` — resets the counter.
4. Line 453: `self.process_io()` — can write IO-operation cycles back into `iteration_cycles`.
5. Line 454: `let id = iterate_return?` — if `Pause`, propagates immediately.

After step 3, `total_cycles` is current and `iteration_cycles` is zero. After step 4, `process_io()` may have written N IO cycles into `iteration_cycles`. When `Pause` propagates at step 5, those N cycles are in `iteration_cycles` and absent from `total_cycles`.

`suspend()` at line 267 faithfully saves `iteration_cycles: self.iteration_cycles` (non-zero). The `FullSuspendedState` struct comment at `types.rs` lines 498–501 explicitly acknowledges this: *"Due to an implementation bug in Meepo hardfork, this value will not always be zero at visible execution boundaries. We will have to preserve this value."*

`resume()` at line 216 correctly restores `iteration_cycles: full.iteration_cycles`, but line 242 unconditionally zeroes it: `scheduler.iteration_cycles = 0`. The comment at lines 239–241 states the intent is to avoid charging cycles for `ensure_vms_instantiated` overhead, but the reset also silently discards the pre-suspension IO cycles.

`consumed_cycles()` at line 156 reads only `total_cycles`. The N IO cycles are permanently lost from all accounting.

In `chunk_run` (lines 481–491), the consumed-cycles delta is `scheduler.consumed_cycles() - previous_cycles`, which reads only `total_cycles`. The discarded N cycles are invisible to the caller.

## Impact Explanation

After resume, `total_cycles` is understated by N cycles. The next chunk runs with the full `limit_cycles` budget, unaware of the missing cycles. Across M suspend/resume boundaries, M × N cycles of budget are silently granted. Transactions that should be rejected for exceeding the cycle limit pass tx-pool chunked verification. If a miner includes such a transaction, the consensus-layer non-resumable verifier computes the correct (higher) cycle count and rejects the block, wasting mining effort and causing chain-tip instability. This matches **Incorrect implementation or behavior of CKB-VM or system scripts** (High, 10001–15000 points).

## Likelihood Explanation

No privileged access is required. Any transaction submitter can craft a script using `spawn` + `pipe` syscalls to ensure pending IO operations are in flight when the chunk limit triggers `Pause`. The tx-pool's chunked verification (`RunMode::LimitCycles`) is the standard code path. The bypass is repeatable across every suspend/resume cycle with no additional attacker capability.

## Recommendation

In `resume()`, save `full.iteration_cycles` before calling `ensure_vms_instantiated`, then restore it afterward. Only cycles added by `ensure_vms_instantiated` itself should be discarded:

```rust
let saved_iteration_cycles = full.iteration_cycles;
// ... build scheduler with iteration_cycles: full.iteration_cycles ...
scheduler.ensure_vms_instantiated(&full.instantiated_ids).unwrap();
// Discard only resume-overhead cycles; restore pre-suspension IO cycles.
scheduler.iteration_cycles = saved_iteration_cycles;
```

Alternatively, `suspend()` should flush `iteration_cycles` into `total_cycles` before saving state, so `iteration_cycles` is always zero at suspension boundaries and the reset in `resume()` is safe.

## Proof of Concept

1. Submit a transaction with a script that uses `spawn` + `pipe` to create pending IO between two VMs.
2. Run the script under tx-pool chunked verification (`RunMode::LimitCycles`).
3. The script runs until the chunk limit triggers `Pause` inside `iterate_inner`.
4. `iterate_outer` executes: commits VM cycles to `total_cycles`, resets `iteration_cycles = 0`, calls `process_io()` which charges N IO cycles into `iteration_cycles`, then `Pause` propagates.
5. `chunk_run` catches `Pause`, calls `scheduler.suspend()` — saves `total_cycles = T`, `iteration_cycles = N`.
6. Next chunk: `resume()` restores `total_cycles = T`, sets `iteration_cycles = 0` — N cycles are lost.
7. Scheduler believes only T cycles consumed; actual is T + N.
8. Repeat across M chunks: M × N cycles of bypass accumulate.
9. Verify by asserting `scheduler.consumed_cycles()` after M chunks equals the true cycle count including IO cycles; the assertion will fail by M × N.
