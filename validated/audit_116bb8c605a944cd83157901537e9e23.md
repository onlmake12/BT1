Audit Report

## Title
Unbounded Outbound Amplification via `ConnectionRequest` with Rotating `to` PeerIds — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `forward_rate_limiter` in the HolePunching protocol is keyed on `(from, to, item_id)`, meaning an attacker who sends 30 `ConnectionRequest` messages per second (the outer per-session cap) with a distinct random `to` PeerId on each message bypasses the inner limiter entirely. Each message with an unknown `to` triggers a `filter_broadcast` to `floor(sqrt(N))` peers, yielding `30 * floor(sqrt(N))` outbound messages per second from a single attacker session. The `forward_rate_limiter` HashMap also grows without bound for the duration of the session.

## Finding Description
**Outer rate limiter** is keyed on `(session_id, msg.item_id())` with a quota of 30/sec. [1](#0-0) [2](#0-1) 

**Inner `forward_rate_limiter`** is keyed on `(content.from, content.to, self.msg_item_id)` with a quota of 1/sec. [3](#0-2) [4](#0-3) 

Because the key includes `content.to`, each message with a fresh random `to` PeerId creates a new rate-limiter bucket that starts at zero — the 1/sec limit is never reached for any single key. The outer limiter allows all 30 messages through.

**Unknown `to` triggers fan-out**: When `get_key_by_peer_id(to_peer_id)` returns `None` (which it always will for random PeerIds), `forward_message` calls `filter_broadcast` to `peers().len().isqrt()` peers. [5](#0-4) 

**Unbounded HashMap growth**: `retain_recent()` on `forward_rate_limiter` is only called in `disconnected()`, so the HashMap accumulates 30 new entries per second for the entire duration of the attacker's session. [6](#0-5) 

**Complete call chain:**
```
received()
  → rate_limiter.check_key((session_id, 0))              ← passes, ≤30/sec
  → ConnectionRequestProcess::execute()
      → forward_rate_limiter.check_key((from, to_N, 0))  ← always passes (new key each time)
      → forward_message()
          → peer_registry.get_key_by_peer_id(to_N) → None
          → filter_broadcast(sqrt(N) peers)               ← fan-out
```

The only other early-exit checks — `route.contains(self_peer_id)` (line 128), `self_peer_id == &content.to` (line 145), and `max_hops == 0` (line 148) — are all trivially avoided by the attacker by omitting the victim from the route, using a random `to` ≠ victim, and setting `max_hops` to any value 1–6. [7](#0-6) 

## Impact Explanation
With `N` connected peers and one attacker session: outbound messages/sec = `30 * floor(sqrt(N))`. At N=10,000 this is 3,000 outbound hole-punching messages per second from a single session. Multiple attacker sessions multiply the effect linearly. This congests the victim node's outbound send queue and imposes forwarding load on `sqrt(N)` downstream peers per message. This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The attack requires only a standard P2P connection — no authentication, no stake, no proof-of-work. The attacker connects as a normal peer and sends well-formed `ConnectionRequest` messages with random but valid 32-byte `to` PeerIds at 30/sec. `PeerId::from_bytes` imposes no cryptographic constraint on the `to` field. The attack is trivially scriptable, repeatable, and amplifiable by opening multiple sessions simultaneously.

## Recommendation
1. **Add a per-session forwarding budget**: Key a separate rate limiter on `(session_id, msg_item_id)` to cap the total number of `filter_broadcast` calls per session per second, independently of the `to` PeerId. A quota of 1–5 forwards/sec per session bounds the fan-out to `5 * sqrt(N)`.
2. **Validate `to` before forwarding**: Drop or strictly rate-limit `ConnectionRequest` messages whose `to` PeerId is not known to the local peer store. Unknown `to` PeerIds are the only case that triggers the expensive `filter_broadcast` path.
3. **Bound the `forward_rate_limiter` HashMap**: Cap the number of entries (e.g., via an LRU eviction policy) or call `retain_recent()` periodically (e.g., in the `notify` timer) rather than only on disconnect.

## Proof of Concept
```rust
// Minimal unit test sketch
let victim = MockNode::new(n_peers = 10_000);
let attacker_session = victim.connect_peer();
let attacker_peer_id = PeerId::random();

for i in 0..30 {
    let unique_to = PeerId::random(); // never in registry
    attacker_session.send(ConnectionRequest {
        from: attacker_peer_id.clone(),
        to: unique_to,
        max_hops: 6,
        route: vec![],                // does not contain victim's peer_id
        listen_addrs: vec![valid_tcp_addr],
    });
}
// After 1 second:
// outer rate_limiter: passes all 30 (keyed on (session_id, 0))
// forward_rate_limiter: passes all 30 (each has unique (from, to_i, 0) key)
// each triggers filter_broadcast to floor(sqrt(10_000)) = 100 peers
assert_eq!(victim.outbound_message_count(), 3_000);
assert!(victim.forward_rate_limiter.len() >= 30); // unbounded growth
```
