### Title
RBF + `limit_size` Race Causes Valid Txs to Be Permanently Evicted Without Successful Replacement — (`tx-pool/src/process.rs`)

### Summary
When RBF is enabled, `submit_entry` removes conflicting txs from `pool_map` into `conflicts_cache` before confirming the replacement tx survives `limit_size`. If `limit_size` subsequently evicts the new tx, the closure returns early via `?`, skipping the recovery spawn. The old txs are permanently displaced from the active pool without a successful replacement.

### Finding Description

In `submit_entry`, `process_rbf` is called at line 136 and unconditionally removes all conflicting txs from `pool_map`, placing them in `conflicts_cache` via `record_conflict`. [1](#0-0) 

`limit_size` is then called with `current_entry_id = Some(&entry.proposal_short_id())`. If the new tx has the lowest fee rate in the pool, it is evicted and `limit_size` returns `Some(Reject::Full(...))`. [2](#0-1) 

The `.map_or(Ok(()), Err)?` propagates this as an `Err`, causing the closure to return before the recovery spawn at lines 154–163. [3](#0-2) 

`limit_size` itself only returns `Some(reject)` when the current entry (the new tx) is the one evicted: [4](#0-3) 

`conflicts_cache` is a bounded LRU of 10,000 entries. Displaced txs are never automatically re-submitted: [5](#0-4) [6](#0-5) 

### Impact Explanation
Valid, previously-accepted txs are permanently removed from `pool_map` and placed in the bounded `conflicts_cache` without a successful replacement. Repeated attacks fill `conflicts_cache` (cap: 10,000), causing LRU eviction of earlier displaced txs entirely. Affected txs are not re-broadcast; their authors must re-submit manually. Pool effective capacity is reduced.

### Likelihood Explanation
RBF is an operator-configurable feature (`min_rbf_rate > min_fee_rate`). When enabled, any remote peer can submit a crafted transaction via P2P relay (`submit_remote_tx` → `_process_tx` → `submit_entry`). The attacker needs only to construct a large tx (e.g., many witnesses) with absolute fee above the RBF threshold but per-byte fee rate below existing pool txs. No privileged access, key material, or hashpower is required.

### Recommendation
Move `process_rbf` (the removal of conflicting txs) to **after** `limit_size` confirms the new tx survives. Alternatively, if `limit_size` evicts the new tx, restore the removed conflicting txs back into `pool_map` before returning the error. The recovery spawn at lines 154–163 should also execute unconditionally (or be moved before the `limit_size` early-return path).

### Proof of Concept
1. Configure node with RBF enabled (`min_rbf_rate > min_fee_rate`) and a small `max_tx_pool_size`.
2. Fill the pool to capacity with txs at fee rate R (high).
3. Submit a replacement tx T_new that spends the same input as an existing pool tx T_old:
   - `fee(T_new) >= fee(T_old) + extra_rbf_fee` (passes `check_rbf`)
   - `fee_rate(T_new) < R` but `>= min_fee_rate` (passes `check_tx_fee`, fails `limit_size`)
   - Achieved by making T_new very large in bytes.
4. Observe: T_old is absent from `pool_map`, present in `conflicts_cache`; T_new is also absent from `pool_map`. Net pool size is reduced by one valid tx with no replacement.
