### Title
Missing RPC Authorization Allows Any Local Caller to Invoke Privileged Node Operations — (`rpc/src/module/pool.rs`, `rpc/src/module/net.rs`)

### Summary
The CKB JSON-RPC server exposes highly privileged node management operations — including `clear_tx_pool`, `remove_transaction`, `clear_tx_verify_queue`, and `set_network_active` — with zero authentication or authorization enforcement. Any process or user that can reach the RPC endpoint (the explicitly in-scope "RPC caller" / "supported local CLI/RPC user" attacker profile) can invoke these operations unconditionally. This is a direct analog to the reported smart contract pattern where a single `authorizedAccounts` mapping controls all privileged functions with no role separation: in CKB's RPC layer, there is no `authorizedAccounts` concept at all — the privileged surface is fully open to any caller.

### Finding Description

**Root cause:** The `ServiceBuilder` in `rpc/src/service_builder.rs` mounts all RPC module methods into a single flat `IoHandler` with no per-method or per-caller authentication middleware. The Pool and Net modules, enabled by default in production (`resource/ckb.toml` line 190), expose destructive administrative operations alongside read-only queries under identical access conditions.

**Privileged operations with no auth check:**

1. `clear_tx_pool()` — `rpc/src/module/pool.rs` lines 684–692: wipes the entire mempool unconditionally.
2. `remove_transaction(tx_hash)` — `rpc/src/module/pool.rs` lines 662–669: removes any specific pending transaction.
3. `clear_tx_verify_queue()` — `rpc/src/module/pool.rs` lines 694–701: clears all transactions awaiting script verification.
4. `set_network_active(false)` — `rpc/src/module/net.rs` lines 772–775: disables all P2P networking.
5. `set_ban` / `clear_banned_addresses` — `rpc/src/module/net.rs` lines 686–726: manipulates the IP ban list.

None of these handlers perform any caller identity check. The implementation of `clear_tx_pool` is:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
```

No token, no session, no IP allowlist check — the call succeeds for any caller.

**Entry path:** The RPC server binds to `127.0.0.1:8114` by default. Any local process (another user on the same host, a compromised application, a malicious script running in the same environment) can issue a plain HTTP POST with no credentials:

```
POST http://127.0.0.1:8114/ HTTP/1.1
Content-Type: application/json

{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}
```

The `ServiceBuilder` mounts the Pool module unconditionally when `pool_enable()` returns true, which it does for the default production config.

### Impact Explanation

- **Transaction censorship:** `remove_transaction(tx_hash)` lets any local caller selectively evict a specific user's pending transaction from the mempool, preventing it from being mined without the user's knowledge.
- **Mempool wipe:** `clear_tx_pool()` destroys all pending transactions, causing every waiting user's transaction to be silently dropped. Users must resubmit; high-fee transactions are lost from the node's view.
- **Network isolation:** `set_network_active(false)` severs all P2P connections, isolating the node from the network. The node stops receiving new blocks and transactions, and stops relaying.
- **Ban list manipulation:** `clear_banned_addresses()` removes all IP bans, re-admitting previously banned malicious peers.

These impacts are permanent state changes (not merely observability noise) and directly affect the node's ability to participate in consensus and serve users.

### Likelihood Explanation

The RPC is localhost-only by default, so the attacker must have local process execution on the same host. This is realistic in shared hosting, container escape scenarios, or when a node operator runs untrusted software alongside CKB. The "supported local CLI/RPC user" is explicitly listed as an in-scope attacker profile. No credentials, keys, or special privileges are required beyond local network access to port 8114.

### Recommendation

Implement per-method or per-module authorization at the RPC layer. Options include:

1. **Token-based auth:** Require a configurable bearer token (set in `ckb.toml`) for all state-mutating RPC methods. The `miner` module already supports HTTP Basic Auth (`parse_authorization` in `miner/src/client.rs`); extend this pattern to the server side.
2. **Method-level allowlist:** Introduce a `privileged_methods` config list; calls to unlisted methods from non-localhost origins (or without a token) are rejected.
3. **Separate privileged port:** Bind destructive operations (`clear_tx_pool`, `set_network_active`, etc.) to a separate listen address that operators can firewall independently.

### Proof of Concept

```bash
# Wipe the entire mempool of a running CKB node — no credentials required
curl -s -X POST http://127.0.0.1:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}'
# Expected: {"jsonrpc":"2.0","result":null,"id":1}

# Disable all P2P networking
curl -s -X POST http://127.0.0.1:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":2}'
# Expected: {"jsonrpc":"2.0","result":null,"id":2}

# Remove a specific pending transaction (replace hash with target tx)
curl -s -X POST http://127.0.0.1:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"],"id":3}'
# Expected: {"jsonrpc":"2.0","result":true,"id":3}
```
