### Title
Incorrect `min_replace_fee` Calculation Uses Existing Tx Size Instead of Replacement Tx Size - (`tx-pool/src/pool.rs`)

### Summary

`TxPool::min_replace_fee` computes the `extra_rbf_fee` component using the **existing** pool transaction's byte size, while `TxPool::check_rbf` (the actual enforcement path) computes the same component using the **new replacement** transaction's byte size. Because these two sizes can differ, the `min_replace_fee` value exposed via the `get_transaction` RPC is incorrect, causing replacement transactions to be rejected even when the user paid exactly the amount the node advertised, or causing users to overpay.

### Finding Description

`TxPool::calculate_min_replace_fee` computes:

```
min_replace_fee = sum(replaced_txs.fee) + extra_rbf_fee
extra_rbf_fee   = min_rbf_rate.fee(size)
```

The `size` argument is the critical parameter. It is supplied from two different call sites with two different values:

**Call site 1 — `min_replace_fee` (RPC query path):** [1](#0-0) 

Here `tx` is the **existing** pool entry being queried. `tx.size` is the serialized byte size of the transaction already in the pool. The result is returned to the caller as the advertised minimum fee a replacement must pay.

**Call site 2 — `check_rbf` (enforcement path):** [2](#0-1) 

Here `entry` is the **incoming replacement** transaction. `entry.size` is the serialized byte size of the new transaction. This is the value actually used to decide whether to accept or reject the replacement.

The shared computation: [3](#0-2) 

Because the two call sites pass different `size` values, the `extra_rbf_fee` component is computed differently in the query path versus the enforcement path. The `min_replace_fee` field returned by the `get_transaction` RPC is therefore incorrect whenever the replacement transaction has a different serialized size than the transaction it replaces.

### Impact Explanation

The `min_replace_fee` field is surfaced through the `get_transaction` RPC: [4](#0-3) 

A user who queries `get_transaction` for an existing pool transaction of size `S_old` receives:

```
min_replace_fee_advertised = fee_old + min_rbf_rate * S_old
```

If the user constructs a replacement transaction of size `S_new > S_old` and sets its fee exactly to `min_replace_fee_advertised`, the enforcement check in `check_rbf` requires:

```
min_replace_fee_actual = fee_old + min_rbf_rate * S_new
```

Since `S_new > S_old`, `min_replace_fee_actual > min_replace_fee_advertised`, and the replacement transaction is rejected with an `RBFRejected` error even though the user paid the exact amount the node told them to pay. Conversely, if `S_new < S_old`, the user overpays unnecessarily.

The `extra_rbf_fee` component — the fee component analogous to the omitted maker fee in the Rubicon report — is the part that is computed incorrectly: it uses the wrong transaction's size.

### Likelihood Explanation

This is reachable by any unprivileged RPC caller. The standard workflow for RBF is:
1. Call `get_transaction` to read `min_replace_fee`.
2. Construct a replacement transaction paying at least that amount.
3. Call `send_transaction`.

Step 3 will fail whenever the replacement transaction's serialized size differs from the original's, which is the common case (e.g., adding or removing outputs, changing witness data, or changing the lock script). No special privileges, keys, or majority hashpower are required.

### Recommendation

In `TxPool::min_replace_fee`, the `size` passed to `calculate_min_replace_fee` should represent the **replacement** transaction's size, not the existing transaction's size. Since the replacement transaction is not yet known at query time, the function should either:

1. Accept the expected replacement size as a parameter from the caller (e.g., from the RPC request), or
2. Document clearly that the returned value is only valid when the replacement transaction has the same serialized size as the original, and update the RPC documentation accordingly.

The enforcement path in `check_rbf` is correct and should not be changed.

### Proof of Concept

Given:
- Existing pool tx A: size = 200 bytes, fee = 1,000,000 shannons
- `min_rbf_rate` = 1,500 shannons/KB

`get_transaction(tx_A_hash).min_replace_fee` returns:
```
1,000,000 + 1,500 * 200 / 1000 = 1,000,300 shannons
```

User constructs replacement tx B: size = 400 bytes, fee = 1,000,300 shannons (exactly as advertised).

`check_rbf` enforces:
```
min_replace_fee_actual = 1,000,000 + 1,500 * 400 / 1000 = 1,000,600 shannons
```

Tx B is rejected: `"Tx's current fee is 1000300, expect it to >= 1000600 to replace old txs"`.

The user paid the exact amount the node advertised but the transaction was still rejected, because the `extra_rbf_fee` component was computed with the wrong size in the query path. [1](#0-0) [3](#0-2) [2](#0-1)

    /// min_replace_fee = sum(replaced_txs.fee) + extra_rbf_fee
    fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
        let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
        // don't account for duplicate txs
        let replaced_fees: HashMap<_, _> = conflicts
            .iter()
            .map(|c| (c.id.clone(), c.inner.fee))
            .collect();
        let replaced_sum_fee = replaced_fees
            .values()
            .try_fold(Capacity::zero(), |acc, x| acc.safe_add(*x));
        let res = replaced_sum_fee.map_or(Err(CapacityError::Overflow), |sum| {
            sum.safe_add(extra_rbf_fee)
        });
        if let Ok(res) = res {
            Some(res)
        } else {
            let fees = conflicts.iter().map(|c| c.inner.fee).collect::<Vec<_>>();
            error!(
                "conflicts: {:?} replaced_sum_fee {:?} overflow by add {}",
                conflicts.iter().map(|e| e.id.clone()).collect::<Vec<_>>(),
                fees,
                extra_rbf_fee
            );
            None
        }
    }
