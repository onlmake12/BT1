### Title
Unauthenticated `set_network_active` RPC Allows Any Caller to Fully Isolate a CKB Node from the P2P Network — (`File: rpc/src/module/net.rs`)

---

### Summary

The `set_network_active` method in CKB's `Net` RPC module accepts a boolean parameter and immediately disables or enables all P2P network activity on the node. There is no authentication layer, no caller identity check, and no permission gate anywhere in the RPC server. The `Net` module is enabled by default in the production configuration. Any process or user that can reach the RPC port — including any local process on the same machine — can call `set_network_active(false)` and permanently isolate the node from the P2P network until the operator manually intervenes.

---

### Finding Description

**Root cause — no authentication in the RPC server:**

A grep for `authentication`, `auth`, `token`, `api_key`, `password`, and `secret` across all of `rpc/src/` returns zero matches. The RPC server is built in `rpc/src/service_builder.rs` and mounts modules directly with no middleware for caller identity or privilege checks. [1](#0-0) 

**The vulnerable method:**

`set_network_active` is declared in the `NetRpc` trait and implemented in `NetRpcImpl`:

```rust
// rpc/src/module/net.rs:772-775
fn set_network_active(&self, state: bool) -> Result<()> {
    self.network_controller.set_active(state);
    Ok(())
}
``` [2](#0-1) 

The trait declaration confirms no guard is present:

```rust
// rpc/src/module/net.rs:419-420
#[rpc(name = "set_network_active")]
fn set_network_active(&self, state: bool) -> Result<()>;
``` [3](#0-2) 

**The `Net` module is enabled by default in production:**

``` [4](#0-3) 

**Secondary unauthenticated privileged methods in the same module:**

- `clear_banned_addresses()` — wipes the entire ban list, allowing all previously banned malicious peers to reconnect immediately.
- `set_ban(address, "delete", ...)` — removes a specific ban entry.

Both are implemented with no access control:

```rust

### Impact Explanation

**`set_network_active(false)`:**
- Immediately halts all P2P activity: block sync, transaction relay, peer discovery, and new inbound/outbound connections.
- The node becomes fully isolated from the network. It stops receiving new blocks and its locally mined blocks are never propagated.
- For a mining node, this causes the miner to work on a stale chain tip, wasting hashpower and potentially producing orphaned blocks.
- The node silently diverges from the canonical chain with no automatic recovery — the operator must manually call `set_network_active(true)` to restore connectivity.
- This is a targeted, persistent denial-of-service against a specific node with a single unauthenticated HTTP request.

**`clear_banned_addresses()`:**
- Removes all IP bans, allowing previously banned peers (e.g., peers banned for sending invalid blocks or malformed messages) to immediately reconnect and resume their attack.
- Undermines the node's self-defense mechanism.

---

### Likelihood Explanation

The RPC binds to `127.0.0.1:8114` by default, which limits exposure to the local machine. However:

1. **Operators routinely expose the RPC port.** The documentation explicitly warns: *"Allowing arbitrary machines to access the JSON-RPC port is dangerous and strongly discouraged"* — this warning exists precisely because it is a common misconfiguration. [8](#0-7) 

2. **Any local process can exploit this.** Malware, a compromised co-located application, or any unprivileged local user can send a single HTTP POST to `127.0.0.1:8114` with no credentials required.

3. **No rate limiting or authentication exists.** The call is a single JSON-RPC request requiring no prior knowledge, no key, and no session.

4. **The `Net` module is on by default.** No operator action is required to expose the vulnerable methods.

---

### Recommendation

1. **Add an authentication layer to the RPC server.** Implement token-based or HTTP Basic authentication (configurable via `ckb.toml`) that must be presented with every request. This is the standard mitigation for unauthenticated local RPC servers (e.g., Bitcoin Core's `rpcuser`/`rpcpassword`).

2. **Separate privileged methods into a distinct admin module.** Methods that mutate node state (`set_network_active`, `clear_banned_addresses`, `set_ban`, `clear_tx_pool`, `remove_transaction`) should be gated behind a separate module that is disabled by default and requires explicit opt-in with authentication.

3. **At minimum, document `set_network_active` as a dangerous admin-only operation** and add a runtime guard that rejects calls unless a configured admin token is present in the HTTP header.

---

### Proof of Concept

**Preconditions:** A CKB node is running with the default configuration (`Net` module enabled, RPC on `127.0.0.1:8114`). The attacker has any form of local access (local shell, co-located process, or an operator who has exposed the RPC port externally).

**Step 1 — Disable all P2P networking:**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":1}'
```

Expected response:
```json
{"jsonrpc":"2.0","result":null,"id":1}
```

**Step 2 — Confirm isolation:**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_peers","params":[],"id":2}'
```

The node will return an empty peers list and will stop syncing blocks. The `sync_state` RPC will show the tip number frozen while the rest of the network advances.

**Step 3 — Clear ban list (secondary exploit):**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_banned_addresses","params":[],"id":3}'
```

All previously banned peers are now permitted to reconnect. No credentials, no signature, no privileged access required at any step.
