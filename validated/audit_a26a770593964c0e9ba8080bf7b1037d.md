### Title
Unvalidated `providerToCredit` in `executeCallback` Permanently Locks Provider Fees - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, the `executeCallback` function accepts a caller-controlled `providerToCredit` address and unconditionally credits fees to `_state.providers[providerToCredit].accruedFeesInWei`. After the exclusivity period expires, there is no check that `providerToCredit` is a registered provider. If an attacker passes an unregistered address, the credited fees have no withdrawal path and are permanently locked in the contract.

### Finding Description
`executeCallback` enforces `providerToCredit == req.provider` only during the exclusivity window:

```solidity
// Echo.sol lines 114–121
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

After that window, `providerToCredit` is fully attacker-controlled. The fee is then credited unconditionally:

```solidity
// Echo.sol line 161–162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

The only withdrawal path for provider fees is `withdrawAsFeeManager`:

```solidity
// Echo.sol lines 364–365
require(
    msg.sender == _state.providers[provider].feeManager,
    "Only fee manager"
);
```

For any unregistered address, `feeManager` is `address(0)` (the zero-value of the `ProviderInfo` struct). No caller can satisfy `msg.sender == address(0)`, so fees credited to an unregistered `providerToCredit` are irrecoverable. There is no direct provider `withdraw` function — only `withdrawAsFeeManager` and the admin-only `withdrawFees` (which drains `_state.accruedFeesInWei`, a separate accounting slot for Pyth protocol fees, not provider fees). ### Impact Explanation
Any fee paid by a user via `requestPriceUpdatesWithCallback` and stored in `req.fee` is permanently lost when `executeCallback` is called with an unregistered `providerToCredit`. The ETH is held by the contract but is unaccounted in any withdrawable slot. The legitimate provider who fulfilled the request receives nothing. At scale, an attacker can grief every provider by front-running or racing `executeCallback` calls after the exclusivity period on every pending request, redirecting all provider fees to a dead address. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
The exclusivity period is configurable and currently defaults to 15 seconds. After it elapses, `executeCallback` is callable by any EOA or contract with no privilege requirement. A griefing attacker needs only to monitor the mempool for pending requests, wait for the exclusivity window to expire, and submit `executeCallback` with `providerToCredit = address(0xdead)` (or any unregistered address). No special access, leaked key, or governance majority is required. [6](#0-5) 

### Recommendation
Add a registration check for `providerToCredit` before crediting fees:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit is not a registered provider"
);
```

This mirrors the pattern already used in `requestPriceUpdatesWithCallback` and `setProviderFee`, which both gate on `isRegistered`. [7](#0-6) [8](#0-7) 

### Proof of Concept

```
1. Deploy Echo with defaultProvider = Alice, exclusivityPeriodSeconds = 15.
2. Alice calls registerProvider(fee, fee, fee).
3. User calls requestPriceUpdatesWithCallback{value: F}(Alice, block.timestamp, priceIds, gasLimit).
   → req.fee = F - pythFee is stored; req.provider = Alice.
4. Wait 16 seconds (exclusivity period expires).
5. Attacker calls executeCallback(address(0xdead), sequenceNumber, updateData, priceIds).
   → No revert: providerToCredit is not checked for isRegistered.
   → _state.providers[address(0xdead)].accruedFeesInWei += (req.fee + msg.value - pythFee).
   → _state.providers[address(0xdead)].feeManager == address(0).
6. Alice calls withdrawAsFeeManager(Alice, amount) → reverts: "Only fee manager" (Alice's own feeManager is address(0) unless she set one).
   Even if Alice set a fee manager, the fees went to address(0xdead), not to Alice.
7. No one can call withdrawAsFeeManager(address(0xdead), amount) because feeManager for 0xdead is address(0).
   Fees are permanently locked.
``` [9](#0-8) [10](#0-9)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
