Audit Report

## Title
Peer Store Permanently Blocked by `Group::None` Collapse via Onion3 Address Flooding — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

The `check_purge` phase-2 eviction path in `PeerStore` uses `take(len / 2)` where `len` is the count of distinct network groups. Because all Onion3 addresses resolve to `Group::None`, flooding 16384 distinct Onion3 addresses collapses `len` to 1, making `1 / 2 = 0` in Rust integer division. `take(0)` yields no eviction candidates, `EvictionFailed` is returned permanently, and every subsequent `add_addr` call fails. The node can no longer learn about new peers from discovery, DNS seeding, or the identify protocol until it is restarted.

## Finding Description

**Step 1 — Onion3 addresses pass `base_addr` and `add_addr` unchanged.**

`base_addr` in `network/src/peer_store/mod.rs` filters only `Ws`, `Wss`, `Memory`, and `Tls` protocols; Onion3 is not listed and passes through. [1](#0-0) 

`add_addr` accepts any non-banned address after `check_purge` succeeds. [2](#0-1) 

**Step 2 — Freshly added Onion3 addresses survive phase-1 eviction.**

`is_connectable` returns `true` for a new address with `last_connected_at_ms = 0`, `attempts_count = 0`, `last_tried_at_ms = 0`: the `tried_in_last_minute` guard is false (0 is not ≥ now−60s), the `attempts_count >= ADDR_MAX_RETRIES` guard is false (0 < 3), and the `attempts_count >= ADDR_MAX_FAILURES` guard is false (0 < 10). All three failure conditions miss, so the address is considered connectable and is NOT removed in phase-1. [3](#0-2) 

**Step 3 — All Onion3 addresses map to `Group::None`.**

`Group::from(&Multiaddr)` calls `multiaddr_to_socketaddr`, which only succeeds for IP4/IP6 addresses. Onion3 returns `None`, falling through to the final `Group::None` arm. [4](#0-3) 

**Step 4 — `take(len / 2)` integer-division bug triggers `EvictionFailed`.**

In phase-2, `peers_by_network_group` has exactly one key (`Group::None`) when all 16384 entries are Onion3. `len = 1`. Rust integer division: `1 / 2 = 0`. `take(0)` produces an empty iterator. The `addrs.len() > 4` inner guard is never reached. `candidate_peers` is empty, and `Err(PeerStoreError::EvictionFailed)` is returned. [5](#0-4) 

**Step 5 — Discovery protocol is the attacker's entry point.**

`DiscoveryAddressManager::add_new_addrs` calls `peer_store.add_addr` for every received address. `is_valid_addr` returns `true` for Onion3 (the `multiaddr_to_socketaddr` branch returns `None`, which maps to `true`). There is no per-session rate limit on total addresses accepted. [6](#0-5) 

`MAX_ADDR_TO_SEND = 1000` caps a single `Nodes` message, but each `Node` item may carry up to `MAX_ADDRS = 3` addresses, so one message can deliver up to 3000 addresses. Filling 16384 slots requires as few as 6 such messages. [7](#0-6) 

## Impact Explanation

Once the peer store is saturated with 16384 `Group::None` entries, every call to `add_addr` returns `Err(PeerStoreError::EvictionFailed)`. The node permanently loses the ability to learn about new peers via discovery, DNS seeding, or the identify protocol. Existing sessions are unaffected, but the node cannot recover from peer churn or network partitions without a restart. This constitutes a **Medium** impact: a broken/suboptimal implementation of the CKB peer-store eviction mechanism that an unprivileged remote attacker can exploit to permanently impair a node's peer-discovery capability (2001–10000 points).

## Likelihood Explanation

- No special configuration is required; Onion3 addresses are accepted by default via `is_valid_addr`.
- A single TCP connection to the victim is sufficient to send discovery `Nodes` messages.
- 16384 distinct Onion3 addresses (trivially generated by varying the 35-byte host) can be delivered in ~6 messages at 3 addresses per item × 1000 items per message.
- The condition is permanent until node restart; there is no self-healing path in the eviction logic.
- The attack is repeatable: after a restart, the attacker can re-flood immediately.

## Recommendation

Replace `take(len / 2)` with a minimum-of-1 guard so that at least one group is always considered for eviction:

```rust
let take_count = (len / 2).max(1);
peers.into_iter().take(take_count)
```

Additionally, add a dedicated `Group::Onion3([u8; N])` variant keyed on the first N bytes of the onion host so that Onion3 addresses are distributed across multiple groups and subject to normal per-group eviction pressure, consistent with how IP4/IP6 addresses are handled. [8](#0-7) 

## Proof of Concept

```rust
#[test]
fn test_onion3_group_none_eviction_deadlock() {
    use crate::{Flags, peer_store::{PeerStore, ADDR_COUNT_LIMIT}};
    let mut peer_store = PeerStore::default();
    for i in 0u64..ADDR_COUNT_LIMIT as u64 {
        let mut host = [0u8; 35];
        host[..8].copy_from_slice(&i.to_le_bytes());
        let encoded = base32::encode(
            base32::Alphabet::RFC4648 { padding: false }, &host
        );
        let onion_addr: p2p::multiaddr::Multiaddr = format!(
            "/onion3/{}:1234/p2p/{}",
            encoded,
            p2p::PeerId::random().to_base58()
        ).parse().unwrap();
        // All 16384 insertions must succeed
        peer_store.add_addr(onion_addr, Flags::COMPATIBILITY).unwrap();
    }
    // Store is now full; any new address must fail with EvictionFailed
    let new_addr: p2p::multiaddr::Multiaddr =
        "/ip4/1.2.3.4/tcp/8114/p2p/QmFoo".parse().unwrap();
    assert!(peer_store.add_addr(new_addr, Flags::COMPATIBILITY).is_err());
}
```

The test confirms: (1) 16384 distinct Onion3 addresses are accepted without error, (2) the 16385th address (any type) returns `Err(PeerStoreError::EvictionFailed)`, and (3) the condition is permanent for the lifetime of the `PeerStore` instance.

            if ip_addr.is_loopback() {

                let bits = ipv4.octets();
                return Group::IP4([bits[0], bits[1]]);
            }
            // IPv6 NetworkGroup
            if let IpAddr::V6(ipv6) = ip_addr {
                if let Some(ipv4) = ipv6.to_ipv4() {

                    return Group::IP4([bits[0], bits[1]]);

            }
        }

        Group::None
    }
