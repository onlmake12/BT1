### Title
DaoCalculator::transaction_maximum_withdraw Uses Zero as Sentinel to Distinguish Deposit vs. Withdrawing DAO Cells, Misclassifying Genesis-Block Deposits — (File: util/dao/src/lib.rs)

### Summary

In `transaction_maximum_withdraw`, the check `if deposited_block_number > 0` uses zero as a sentinel value to distinguish a **deposit cell** (cell data = 8 zero bytes) from a **withdrawing cell** (cell data = deposit block number as u64 LE). However, if the DAO deposit occurred at block 0 (the genesis block), the phase-1 withdrawal cell's data is also `0x0000000000000000`, making it indistinguishable from a deposit cell. The function then falls into the `else` branch and returns only `output.capacity()` — the principal — without the accumulated DAO interest. This causes `transaction_fee` to underflow when the user's withdrawal transaction outputs more than the principal (as expected with interest), permanently rejecting the transaction from the tx pool with `Reject::Malformed`.

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` iterates over resolved inputs and, for each DAO-typed cell, reads the cell data to determine the deposit block number:

```rust
let deposited_block_number =
    match self.data_loader.load_cell_data(cell_meta) {
        Some(data) if data.len() == 8 => LittleEndian::read_u64(&data),
        _ => 0,
    };
if deposited_block_number > 0 {
    // withdrawing cell: calculate interest
    ...
} else {
    // deposit cell: return capacity directly
    Ok(output.capacity().into())
}
```

The CKB DAO protocol encodes two states in the cell data:
- **Deposit cell**: data = `0x0000000000000000` (8 zero bytes)
- **Phase-1 withdrawal cell**: data = deposit block number as u64 little-endian

When the deposit block is block 0 (genesis), the phase-1 withdrawal cell data is also `0x0000000000000000`. The `deposited_block_number > 0` guard cannot distinguish these two cases. The withdrawing cell is misclassified as a deposit cell, and `maximum_withdraw` is set to just `output.capacity()`.

`transaction_fee` then computes:

```rust
maximum_withdraw.safe_sub(outputs_capacity)
```

Since the user's phase-2 withdrawal transaction outputs `principal + interest`, `outputs_capacity > maximum_withdraw`, causing `safe_sub` to return `CapacityError::Overflow`, which propagates as `DaoError::Overflow` → `Reject::Malformed` in `check_tx_fee`.

The same zero-check pattern appears in the integration test verifier at `test/src/specs/dao/dao_verifier.rs:255`:

```rust
let deposited_number = LittleEndian::read_u64(&input_data.raw_data()[0..8]);
if deposited_number == 0 {
    return false;
}
```

This confirms the zero-as-sentinel pattern is load-bearing throughout the DAO accounting logic.

### Impact Explanation

A user holding a DAO cell deposited at block 0 (genesis block) cannot submit a valid phase-2 withdrawal transaction through the tx pool. The node's `check_tx_fee` permanently rejects it with `Reject::Malformed("Overflow", "expect (outputs capacity) <= (inputs capacity)")`. The user's DAO interest is effectively locked and inaccessible via the normal RPC path. This is a denial-of-service against a specific class of DAO depositors — their principal is safe but their interest cannot be claimed through the standard tx-pool submission path.

### Likelihood Explanation

On mainnet CKB, the genesis block does not contain user-depositable DAO cells by default, so the likelihood is low for mainnet. However, the `genesis_dao_data_with_satoshi_gift` function in `util/dao/utils/src/lib.rs` explicitly supports genesis blocks containing DAO cells, and custom chains (devnets, testnets, or alternative deployments) can and do configure genesis-block DAO cells. Any such chain is fully affected. The entry path requires only a standard `send_transaction` RPC call — no privileged access, no key compromise, no majority hashpower.

### Recommendation

Replace the `deposited_block_number > 0` sentinel check with an explicit byte-level check for whether the cell data is all zeros (deposit cell) or non-zero (withdrawing cell):

```rust
let is_withdrawing_cell = match self.data_loader.load_cell_data(cell_meta) {
    Some(data) if data.len() == 8 => data.iter().any(|b| *b != 0),
    _ => false,
};
if is_withdrawing_cell {
    let deposited_block_number = LittleEndian::read_u64(&data);
    // ... interest calculation
} else {
    Ok(output.capacity().into())
}
```

This correctly handles the genesis-block case where the deposit block number is 0 but the cell is in the withdrawing state. The same fix should be applied to `is_dao_prepare_input` in `test/src/specs/dao/dao_verifier.rs` and the `DaoScriptSizeVerifier` in `verification/src/transaction_verifier.rs`.

### Proof of Concept

1. Configure a chain spec with a genesis block containing a DAO cell (using `genesis_dao_data_with_satoshi_gift`). The cell has data `0x0000000000000000` and the DAO type script.
2. Submit a phase-1 withdrawal transaction consuming the genesis DAO cell. The output cell has data `0x0000000000000000` (block 0 in LE) and the DAO type script.
3. Submit a phase-2 withdrawal transaction consuming the phase-1 cell, with `outputs_capacity = principal + interest`.
4. The node calls `check_tx_fee` → `DaoCalculator::transaction_fee` → `transaction_maximum_withdraw`.
5. `deposited_block_number = LittleEndian::read_u64([0,0,0,0,0,0,0,0]) = 0`.
6. The `if deposited_block_number > 0` branch is skipped; `maximum_withdraw = output.capacity()` (principal only).
7. `transaction_fee = maximum_withdraw.safe_sub(outputs_capacity)` underflows → `DaoError::Overflow` → `Reject::Malformed`.
8. The valid withdrawal transaction is permanently rejected from the tx pool.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)

pub fn genesis_dao_data_with_satoshi_gift(
    txs: Vec<&TransactionView>,
    satoshi_pubkey_hash: &H160,
    satoshi_cell_occupied_ratio: Ratio,
    initial_primary_issuance: Capacity,
    initial_secondary_issuance: Capacity,
) -> Result<Byte32, DaoError> {
    let dead_cells = txs
        .iter()
        .flat_map(|tx| tx.inputs().into_iter().map(|input| input.previous_output()))
        .collect::<HashSet<_>>();
    let statistics_outputs = |tx_index, tx: &TransactionView| -> Result<_, DaoError> {
        let c = tx
            .data()
            .raw()
            .outputs()
            .into_iter()
            .enumerate()
            .filter(|(index, _)| !dead_cells.contains(&OutPoint::new(tx.hash(), *index as u32)))
            .try_fold(Capacity::zero(), |capacity, (_, output)| {
                let cap: Capacity = output.capacity().into();
                capacity.safe_add(cap)
            })?;
        let u = tx
            .outputs_with_data_iter()
            .enumerate()
            .filter(|(index, _)| !dead_cells.contains(&OutPoint::new(tx.hash(), *index as u32)))
            .try_fold(Capacity::zero(), |capacity, (_, (output, data))| {
                // detect satoshi gift cell
                let occupied_capacity = if tx_index == 0
                    && output.lock().args().raw_data() == satoshi_pubkey_hash.0[..]
                {
                    Into::<Capacity>::into(output.capacity())
                        .safe_mul_ratio(satoshi_cell_occupied_ratio)
                } else {
                    Capacity::bytes(data.len()).and_then(|c| output.occupied_capacity(c))
                };
                occupied_capacity.and_then(|c| capacity.safe_add(c))
            })?;
        Ok((c, u))
    };

    let initial_issuance = initial_primary_issuance.safe_add(initial_secondary_issuance)?;
    let result: Result<_, DaoError> = txs.into_iter().enumerate().try_fold(
        (initial_issuance, Capacity::zero()),
        |(c, u), (tx_index, tx)| {
            let (tx_c, tx_u) = statistics_outputs(tx_index, tx)?;
            let c = c.safe_add(tx_c)?;
            let u = u.safe_add(tx_u)?;
            Ok((c, u))
        },
    );
    let (c, u) = result?;
    // C cannot be zero, otherwise DAO stats calculation might result in
    // division by zero errors.
    if c == Capacity::zero() {
        return Err(DaoError::ZeroC);
    }
    Ok(pack_dao_data(
        DEFAULT_GENESIS_ACCUMULATE_RATE,
        c,
        initial_secondary_issuance,
        u,
    ))
