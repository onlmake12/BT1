### Title
Permissionless `revealWithCallback` on `CALLBACK_FAILED` Requests Enables Selective Acceptance of Random Numbers - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `revealWithCallback` function in `Entropy.sol` is callable by **anyone without any fee**. When a callback fails, the contract transitions the request to `CALLBACK_FAILED` state and publicly emits the final `randomNumber` in the `CallbackFailed` event. Because re-invocation is permissionless and free, a malicious requester can intentionally design their callback to fail on the first call, observe the revealed random number from the event log, and then selectively accept or permanently abandon the result — effectively choosing their random outcome. This directly undermines the security guarantee of the commit-reveal protocol.

---

### Finding Description

`revealWithCallback` is declared `public` with no access control and no fee requirement: [1](#0-0) 

The gate check at the top of the function explicitly allows re-entry when `callbackStatus == CALLBACK_FAILED`: [2](#0-1) 

On the **first** invocation (when `gasLimit10k != 0` and `callbackStatus == CALLBACK_NOT_STARTED`), the callback is executed via `excessivelySafeCall` (catching reverts). If the callback reverts, the contract emits `CallbackFailed` — which **publicly broadcasts the final `randomNumber`** — and sets `callbackStatus = CALLBACK_FAILED`: [3](#0-2) 

On **subsequent** invocations (when `callbackStatus == CALLBACK_FAILED`), the code falls into the `else` branch. Here `clearRequest` is called **before** the callback, and the callback is invoked **without error catching**: [4](#0-3) 

If the callback reverts, the entire transaction reverts (including `clearRequest`), leaving the request permanently in `CALLBACK_FAILED`. The attacker can repeat this indefinitely at zero protocol cost.

The `CallbackFailed` event definition confirms the random number is fully exposed: [5](#0-4) 

The `EntropyStatusConstants` confirm the state machine values involved: [6](#0-5) 

---

### Impact Explanation

A malicious requester can:

1. Deploy a callback contract that always reverts.
2. Call `requestV2` (paying the required fee once).
3. Wait for the provider's keeper to call `revealWithCallback`. The callback fails; `CallbackFailed` is emitted with the plaintext `randomNumber`.
4. Read the `randomNumber` from the event log.
5. If the outcome is **favorable** (e.g., they win a lottery, get a rare NFT trait), upgrade the callback to succeed and call `revealWithCallback` again — for free.
6. If the outcome is **unfavorable**, abandon the request (it stays in `CALLBACK_FAILED` forever) and submit a new `requestV2` with a fresh fee.

This breaks the core security property of the commit-reveal protocol: the requester can now observe the random number before deciding to accept it. Applications using Pyth Entropy for lotteries, NFT minting, gaming, or DeFi randomness are directly affected. The cost to the attacker is only the fee for each abandoned request, which can be negligible relative to the value of a favorable outcome.

Additionally, any third party can grief a legitimate requester stuck in `CALLBACK_FAILED` by repeatedly calling `revealWithCallback`, forcing the callback to execute (and revert) at arbitrary times, potentially causing unintended side effects in stateful callback contracts.

---

### Likelihood Explanation

- The attack requires no privileged access — any unprivileged Entropy user can exploit this.
- The `revealWithCallback` function is explicitly documented as callable by anyone.
- The `CallbackFailed` event is indexed and easily observable by any on-chain monitor or MEV bot.
- The only cost is the per-request fee, which is small relative to the value of high-stakes applications (lotteries, NFT drops).
- The attack is realistic for any application where the random outcome has significant monetary value.

---

### Recommendation

1. **Store the random number in the request struct on first reveal.** Once `revealWithCallback` is called and the random number is computed, persist it in `req.randomNumber`. Subsequent re-invocations should use the stored value rather than recomputing it. This does not prevent selective acceptance but removes the ability to abandon and re-request.

2. **Restrict re-invocation of `CALLBACK_FAILED` to the original requester only.** The recovery path is intended for the requester to fix their own callback. Limiting it to `req.requester == msg.sender` prevents third-party griefing while preserving the recovery use case.

3. **Add a deadline for `CALLBACK_FAILED` recovery.** After a timeout, allow anyone to force-clear the request (without invoking the callback), freeing the slot and preventing indefinite abandonment.

4. **Document the selective acceptance risk** prominently in the integration guide, warning that callback contracts must not be upgradeable or conditionally reverting based on the random number value.

### Proof of Concept

```solidity
// Attacker's callback contract
contract SelectiveAcceptanceAttack is IEntropyConsumer {
    IEntropyV2 entropy;
    bool public acceptResult = false; // starts false → always reverts

    constructor(address _entropy) { entropy = IEntropyV2(_entropy); }

    function requestRandomness() external payable returns (uint64) {
        uint256 fee = entropy.getFeeV2();
        return entropy.requestV2{value: fee}();
    }

    function entropyCallback(
        uint64, address, bytes32 randomNumber
    ) internal override {
        // Revert unless attacker has decided to accept
        require(acceptResult, "Not accepting yet");
        // Use randomNumber here (e.g., claim lottery prize)
    }

    function getEntropy() internal view override returns (address) {
        return address(entropy);
    }

    // Called by attacker after observing randomNumber from CallbackFailed event
    function acceptAndReveal(
        address provider,
        uint64 sequenceNumber,
        bytes32 userContribution,
        bytes32 providerContribution
    ) external {
        acceptResult = true;
        // revealWithCallback is free and permissionless
        entropy.revealWithCallback(provider, sequenceNumber, userContribution, providerContribution);
    }
}
```

**Steps:**
1. Deploy `SelectiveAcceptanceAttack` with `acceptResult = false`.
2. Call `requestRandomness()` — pays the fee once.
3. Provider's keeper calls `revealWithCallback` → callback reverts → `CallbackFailed` event emitted with `randomNumber`.
4. Attacker reads `randomNumber` from the event.
5. If favorable: call `acceptAndReveal(...)` — sets `acceptResult = true`, calls `revealWithCallback` for free, callback succeeds.
6. If unfavorable: ignore this request (it stays in `CALLBACK_FAILED`), deploy a new instance, repeat from step 2.
