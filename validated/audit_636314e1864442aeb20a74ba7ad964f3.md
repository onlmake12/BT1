Audit Report

## Title
Tx-Pool Promotes Transactions to `Proposed` Status Without Re-Verifying `since` Constraints After Reorg — (`tx-pool/src/pool.rs`, `tx-pool/src/process.rs`)

## Summary
During chain reorganization, `_update_tx_pool_for_reorg` promotes pending/gap transactions to `Proposed` status via `proposed_rtx`, which performs only a status label flip with no re-verification of `since` time-lock constraints against the new chain snapshot. A transaction whose `since` threshold is no longer satisfiable under the post-reorg tip is silently promoted to `Proposed` and fed to the block assembler, which also skips `since` re-verification. The resulting block template contains an invalid transaction that the full contextual block verifier rejects, causing the miner to forfeit the block reward. The condition is self-sustaining until the transaction is manually evicted.

## Finding Description

**`proposed_rtx` is a pure label flip** (`tx-pool/src/pool.rs:405-422`): it checks only for duplicate status, then calls `set_entry_proposed` with no fee, capacity, or `TimeRelativeTransactionVerifier` call. [1](#0-0) 

**`set_entry` only mutates the `status` field** (`tx-pool/src/component/pool_map.rs:223-233`): no verification of any kind is performed. [2](#0-1) 

**`_update_tx_pool_for_reorg` calls `proposed_rtx` directly** (`tx-pool/src/process.rs:1082-1094`): the new post-reorg `snapshot` is available but never passed to any verifier before the status flip. [3](#0-2) 

**`remove_expired` does not cover `since` violations** (`tx-pool/src/pool.rs:271-288`): it filters only on wall-clock age (`expiry + entry.timestamp < now_ms`), not on `since` constraints. [4](#0-3) 

**Contrast with `readd_detached_tx`** (`tx-pool/src/process.rs:888-912`): transactions from detached blocks go through the full pipeline — `resolve_tx` → `check_tx_fee` → `verify_rtx` (which calls `TimeRelativeTransactionVerifier`). Transactions already in the pool that are merely promoted during the same reorg receive none of this re-verification. [5](#0-4) 

**Block assembler does not re-verify `since`** (`tx-pool/src/block_assembler/mod.rs:640-668`): `calc_dao` only calls `entry.rtx.check(...)`, which is a cell-resolution (double-spend/dead-cell) check. No `TimeRelativeTransactionVerifier` is invoked. [6](#0-5) 

**Full block verifier catches the violation — but only after block submission** (`verification/contextual/src/contextual_block_verifier.rs:400-424`): `TxVerifyEnv::new_commit` is created and `TimeRelativeTransactionVerifier` is run, so the assembled block is rejected by the network. [7](#0-6) 

**Concrete exploit scenario:**
- `since = AbsoluteEpoch(E)` (DAO withdrawal), proposal window closest = 2.
- T is proposed at block H; at tip H+1, T enters `Proposed` (commit window [H+2, ...]). At tip H+1, block H+2 satisfies `since` → valid.
- Reorg: new tip drops to H-3. `remove_by_detached_proposal` moves T back to `Pending`.
- Miner mines block H-2, re-proposing T. At tip H-1, `_update_tx_pool_for_reorg` promotes T to `Proposed` — **no `since` check**.
- Block assembler packages T into block H. Full verifier: `TxVerifyEnv::new_commit` at block H; epoch at H < E → `Immature` error → block rejected.
- T remains in `Proposed`; every subsequent block template includes T; every block is rejected until T is manually evicted via `remove_transaction` RPC or the epoch naturally reaches E.

The `remove_by_detached_proposal` path confirms T is moved back to `Pending` without re-verification: [8](#0-7) 

## Impact Explanation

A miner operating in `mine_mode` with a `since`-constrained transaction (particularly a DAO withdrawal with an epoch-based `since`) in the pool will repeatedly produce block templates containing the invalid transaction. Each submitted block is rejected by the network's contextual block verifier, causing the miner to forfeit the block reward for every round until the invalid transaction is manually removed or the `since` threshold is naturally reached. DAO withdrawal transactions, which are among the most common time-locked transactions on CKB mainnet, use epoch-based `since` fields where the gap between the post-reorg epoch and the `since` threshold can span many blocks, causing sustained economic damage. This matches the allowed impact class: **Vulnerabilities which could easily damage CKB economy (Critical, 15001–25000 points)**.

## Likelihood Explanation

- A 2-block reorg is sufficient to trigger the condition; reorgs of this depth occur naturally on mainnet without any attacker involvement.
- DAO withdrawal transactions with epoch-based `since` constraints are among the most common time-locked transactions on CKB mainnet.
- No special privileges or majority hashpower are required; a natural network reorg is the sole entry point.
- Once triggered, the condition is self-sustaining: the invalid transaction stays in `Proposed` and is included in every subsequent block template until manually removed via `remove_transaction` RPC, or until the epoch naturally advances past the `since` threshold.

## Recommendation

In `_update_tx_pool_for_reorg` (`tx-pool/src/process.rs`), before calling `proposed_rtx` or `gap_rtx`, run `time_relative_verify` against the new snapshot for each candidate transaction. If verification fails, call `callbacks.call_reject` and remove the entry instead of promoting it. This mirrors the pattern already used in `readd_detached_tx` (`process.rs:888-912`).

The `time_relative_verify` helper is already available: [9](#0-8) 

## Proof of Concept

1. Configure a CKB node in mine mode (`block_assembler` enabled).
2. Submit a DAO withdrawal transaction T with `since = AbsoluteEpoch(E)` where E is a few epochs in the future (so it is accepted into the pool).
3. Mine blocks until T is proposed and enters `Proposed` status. Confirm via `get_raw_tx_pool`.
4. Inject a competing fork rooted 2 blocks back (tip drops). T returns to `Pending`.
5. Mine a block on the new fork that re-proposes T. Observe via `get_raw_tx_pool` that T is back in `Proposed` status even though the current epoch < E.
6. Call `get_block_template` — T appears in `transactions`.
7. Submit the assembled block; the network rejects it with `TransactionFailedToVerify: Immature`.
8. Repeat steps 6–7 to confirm the condition persists until T is manually removed via `remove_transaction` RPC.

        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {

            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());

            callbacks.call_reject(self, &entry, reject);
        }
    }

                    continue;

        }

        match self.get_pool_entry(short_id) {
            Some(entry) => {
                let tx_hash = entry.inner.transaction().hash();
                if entry.status == Status::Proposed {
                    Err(Reject::Duplicated(tx_hash))
                } else {
                    debug!("proposed_rtx: {:?} => {:?}", tx_hash, short_id);
                    self.set_entry_proposed(short_id);
                    Ok(())
                }
            }
            None => Err(Reject::Malformed(
                String::from("invalid short_id"),
                Default::default(),
            )),
        }
    }

pub(crate) fn time_relative_verify(
    snapshot: Arc<Snapshot>,
    rtx: Arc<ResolvedTransaction>,
    tx_env: TxVerifyEnv,
) -> Result<(), Reject> {
    let consensus = snapshot.cloned_consensus();
    TimeRelativeTransactionVerifier::new(
        rtx,
        consensus,
        snapshot.as_data_loader(),
        Arc::new(tx_env),
    )
    .verify()
    .map_err(Reject::Verification)
}
