### Title
Unrolled cell_ref_parent eviction on failed `record_entry_edges` enables free targeted tx-pool eviction — (`tx-pool/src/component/pool_map.rs`)

### Summary
In `PoolMap::add_entry`, `check_and_record_ancestors` permanently evicts cell_ref_parent transactions before `record_entry_edges` validates inputs for conflicts. If `record_entry_edges` subsequently fails (e.g., double-spend with an existing pool tx), the evictions are not rolled back, violating the invariant that pool evictions must only occur when the evicting transaction is successfully admitted.

### Finding Description
`add_entry` executes in this order:

1. `check_and_record_ancestors` — may evict N cell_ref_parent txs via `remove_entry_and_descendants` (permanent, immediate)
2. `record_entry_edges` — may return `Err` if any input is already claimed in `edges.inputs`
3. On `Err` from step 2, `add_entry` returns the error — but the evictions from step 1 are not undone

An attacker can exploit this by:
- Submitting N cell_ref_parent txs (each with a cell dep on an attacker-controlled outpoint) so that `N + 1 > max_ancestors_count`
- Ensuring tx_B (victim or attacker-placed) already claims outpoint Y in `edges.inputs`
- Submitting tx_C with inputs `[attacker_outpoints..., Y]`

`check_and_record_ancestors` sees the N cell_ref_parents as eviction candidates (because tx_C's inputs match their cell deps via `edges.deps`), evicts them, then `record_entry_edges` hits the conflict on Y and returns `Err`. tx_C is rejected; the N cell_ref_parents are gone. [1](#0-0) 

The `get_tx_ancestors` function identifies cell_ref_parents by checking `self.edges.deps.get(&input_pt)` for each input of the incoming tx: [2](#0-1) 

The eviction loop in `check_and_record_ancestors` calls `remove_entry_and_descendants` with no staging: [3](#0-2) 

The double-spend detection in `record_entry_edges` that triggers the failure: [4](#0-3) 

The existing FIXME comment acknowledges the general problem but incorrectly concludes it is safe for the non-RBF cell_ref_parent path: [5](#0-4) 

### Impact Explanation
An unprivileged attacker can cheaply and repeatedly evict arbitrary in-pool transactions — including those using system scripts as cell deps — without ever having a transaction committed to a block. The attacker only needs to:
- Control N UTXOs (to be used as cell deps by the cell_ref_parent txs)
- Know of one outpoint Y already claimed in the pool (trivially observable via RPC)
- Pay the cost of submitting tx_C (which is rejected and never mined)

After each attack round, the attacker re-submits the cell_ref_parent txs and repeats. The cost per eviction round is effectively zero on-chain cost.

### Likelihood Explanation
The attack is straightforward to construct, requires no privileged access, and is reachable via the standard P2P transaction submission path. The `max_ancestors_count` default of 25 sets the minimum N, which is easily achievable. The attacker controls all preconditions.

### Recommendation
Validate all inputs for conflicts **before** performing any evictions. Specifically, move the double-spend check (or a pre-check equivalent) ahead of `check_and_record_ancestors` in `add_entry`, or implement a rollback mechanism that re-inserts evicted entries if any subsequent step fails. The FIXME comment at line 583 already identifies the correct principle: "it's still safer to report an error before any writing kind of operation."

### Proof of Concept
1. Build a pool with `max_ancestors_count = 25`
2. Submit 25 cell_ref_parent txs, each with a cell dep on outpoint `A_i` (attacker-controlled)
3. Submit tx_B spending outpoint Y (now `edges.inputs[Y] = tx_B_id`)
4. Submit tx_C with inputs `[A_0, A_1, ..., A_24, Y]`
5. `check_and_record_ancestors` sees 25 cell_ref_parents, evicts them all
6. `record_entry_edges` hits `edges.inputs[Y]` occupied → returns `Err`
7. Assert: pool contains tx_B, does not contain any cell_ref_parent, does not contain tx_C
8. Repeat from step 2 — each round costs zero on-chain fees [6](#0-5)
