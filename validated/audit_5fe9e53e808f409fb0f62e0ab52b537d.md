### Title
Excess `AptosCoin` Fee Not Refunded to Caller in `update_price_feeds` — (`File: target_chains/aptos/contracts/sources/pyth.move`)

---

### Summary

The `update_price_feeds` function in the Aptos Pyth contract accepts a `Coin<AptosCoin>` fee object from the caller, computes the exact required fee internally, but then deposits the **entire** coin — including any excess — into the Pyth treasury. Any overpayment is permanently lost to the caller.

---

### Finding Description

In `target_chains/aptos/contracts/sources/pyth.move`, the public function `update_price_feeds` accepts a caller-supplied `Coin<AptosCoin>` as payment:

```move
public fun update_price_feeds(vaas: vector<vector<u8>>, fee: Coin<AptosCoin>) {
    let total_updates = 0;
    while (!vector::is_empty(&vaas)) {
        total_updates = total_updates + update_price_feed_from_single_vaa(vector::pop_back(&mut vaas));
    };
    let update_fee = state::get_base_update_fee() * total_updates;
    assert!(update_fee <= coin::value(&fee), error::insufficient_fee());
    coin::deposit(@pyth, fee);   // <-- entire coin deposited, not just update_fee
}
``` [1](#0-0) 

The assertion on line 178 only checks that the coin's value is **at least** `update_fee`. However, `coin::deposit(@pyth, fee)` on line 179 deposits the **entire** coin object — not just the required `update_fee` amount. In Move, there is no implicit splitting; the whole coin is consumed. Any excess above `update_fee` is permanently credited to the `@pyth` address and is unrecoverable by the caller. [2](#0-1) 

The correct pattern would be to split the coin: extract exactly `update_fee` worth of coins, deposit only that portion, and return the remainder to the caller.

---

### Impact Explanation

Any caller who passes a `Coin<AptosCoin>` with a value greater than the exact required fee permanently loses the excess to the Pyth treasury. This is a direct value leak from users. The excess is credited to `@pyth` and can only be recovered by Pyth governance — not by the original caller.

Concrete scenarios where overpayment occurs:
1. The caller queries `get_update_fee` off-chain, but the base fee is updated by governance between the query and the transaction execution.
2. The caller passes a round-number coin (e.g., 1 APT) to avoid computing the exact fee.
3. Integrating contracts that pass a pre-funded coin object without exact-fee splitting.

**Impact**: Medium — direct, permanent loss of user funds (AptosCoin) with no recovery path for the caller.

---

### Likelihood Explanation

The function is `public`, callable by any unprivileged user or integrating contract. The fee is computed dynamically from `state::get_base_update_fee() * total_updates`, which can change via governance. Any caller who does not pass the exact fee amount — a common and reasonable pattern — will silently lose the excess. The likelihood is **medium**: it requires overpayment, but overpayment is a natural consequence of fee volatility or imprecise coin construction.

---

### Recommendation

Split the coin before depositing. Extract exactly `update_fee` from the provided coin, deposit only that amount, and return the remainder to the caller:

```move
public fun update_price_feeds(vaas: vector<vector<u8>>, fee: Coin<AptosCoin>): Coin<AptosCoin> {
    let total_updates = 0;
    while (!vector::is_empty(&vaas)) {
        total_updates = total_updates + update_price_feed_from_single_vaa(vector::pop_back(&mut vaas));
    };
    let update_fee = state::get_base_update_fee() * total_updates;
    assert!(update_fee <= coin::value(&fee), error::insufficient_fee());
    let exact_fee = coin::extract(&mut fee, update_fee);
    coin::deposit(@pyth, exact_fee);
    fee  // return remainder to caller
}
```

---

### Proof of Concept

1. Governance sets `base_update_fee = 100` (octas).
2. Caller wants to update 1 price feed. Required fee = 100 octas.
3. Caller constructs `fee = coin::withdraw(account, 200)` (passes 200 octas, a round number).
4. `update_price_feeds` is called. `update_fee = 100`. Assertion passes (`100 <= 200`).
5. `coin::deposit(@pyth, fee)` deposits all 200 octas to `@pyth`.
6. Caller loses 100 octas of excess with no recourse.

The entire excess is silently absorbed into `_state.accruedPythFeesInWei` (EVM analog) or `@pyth` balance (Aptos), matching the exact vulnerability class of the reference report: leftover payment tokens are not returned to the user. [1](#0-0)
