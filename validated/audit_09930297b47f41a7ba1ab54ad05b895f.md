### Title
Missing Per-Peer Rate Limiting in `LightClientProtocol` Allows Single Peer to Trigger Maximum I/O Per Request — (`util/light-client-protocol-server/src/lib.rs`, `src/components/get_blocks_proof.rs`)

### Summary

The `LightClientProtocol` handler has no rate limiter, unlike every comparable protocol handler in the codebase. A single unprivileged peer can repeatedly send `GetBlocksProof` messages with exactly 1000 valid main-chain hashes, causing the server to perform 3000 storage reads plus MMR proof generation for 1000 positions per message, with no throttling.

### Finding Description

`LightClientProtocol` in `lib.rs` defines no rate-limiter field and performs no rate-limit check before dispatching messages: [1](#0-0) 

The `received` handler goes directly to `try_process` with no quota check: [2](#0-1) 

Contrast this with `Relayer`, which has an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` and checks it before every non-PoW message: [3](#0-2) 

And `HolePunching`, which also has a per-peer rate limiter checked on every `received` call: [4](#0-3) 

`GET_BLOCKS_PROOF_LIMIT` is 1000: [5](#0-4) 

When a `GetBlocksProof` message with 1000 valid, unique, main-chain hashes arrives, `execute()` iterates all 1000 and calls three storage reads per hash: [6](#0-5) 

Then `reply_proof` calls `mmr.gen_proof(items_positions)` for all 1000 positions: [7](#0-6) 

The only guards present are:
- Empty check (line 34–36)
- Count > 1000 check (line 38–40)
- `last_hash` must be on main chain (line 45–50)
- Duplicate check (line 62–70)

None of these limit the *rate* at which a single peer can submit maximum-size requests. [8](#0-7) 

### Impact Explanation

Each maximum-size `GetBlocksProof` request causes:
- 3 × 1000 = 3000 RocksDB reads (`get_block_header`, `get_block_uncles`, `get_block_extension`)
- MMR proof generation over 1000 leaf positions (logarithmic in chain height but still significant)
- Allocation and serialization of the full response

A single peer sending these in a tight loop can saturate the I/O subsystem and CPU of the full node serving light clients, degrading or denying service to all other peers.

### Likelihood Explanation

Any peer that can open a light-client protocol connection (no authentication required) can exploit this. The message is structurally valid and passes all existing checks. The attacker only needs to know 1000 valid main-chain block hashes, which are trivially obtained from any block explorer or by syncing a few blocks.

### Recommendation

Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol` (mirroring `Relayer` and `HolePunching`) and check it at the top of `try_process` before dispatching any message. A quota of ~10 requests/second per peer per message type would match the existing patterns in the codebase.

### Proof of Concept

1. Connect to a full node's light-client protocol endpoint.
2. Obtain 1000 valid main-chain block hashes (e.g., blocks 1–1000).
3. Choose any valid main-chain `last_hash` not in the set.
4. Send `GetBlocksProof { block_hashes: [h1..h1000], last_hash }` in a tight loop.
5. Observe: each message triggers 3000 storage reads + MMR proof generation with no server-side throttling; node I/O and CPU spike proportionally to request rate.

impl Relayer {
    /// Init relay protocol handle
    ///
    /// This is a runtime relay protocol shared state, and any relay messages will be processed and forwarded by it
    pub fn new(chain: ChainController, shared: Arc<SyncShared>) -> Self {
        // setup a rate limiter keyed by peer and message type that lets through 30 requests per second
        // current max rps is 10 (ASK_FOR_TXS_TOKEN / TX_PROPOSAL_TOKEN), 30 is a flexible hard cap with buffer
        let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
        let rate_limiter = RateLimiter::hashmap(quota);

        Relayer {
            chain,
            shared,
            rate_limiter,
        }
    }

    /// Get shared state
    pub fn shared(&self) -> &Arc<SyncShared> {
        &self.shared
    }

    async fn try_process(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        message: packed::RelayMessageUnionReader<'_>,
    ) -> Status {
        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =
            !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));

        if should_check_rate
            && self
                .rate_limiter
                .check_key(&(peer, message.item_id()))
                .is_err()
        {
            return StatusCode::TooManyRequests.with_context(message.item_name());
        }
