### Title
Unbounded UNION ALL Sub-query in `get_tx_with_cells` Forces O(total_matches) DB Materialization Regardless of Requested Limit — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

---

### Summary

When `get_transactions` is called with `group_by_transaction=true`, the function `get_tx_with_cells` constructs a three-level SQL query where the innermost UNION ALL sub-query carries no `LIMIT`. Because the outer query applies `GROUP BY` before `LIMIT`, the database engine must fully materialize every matching row from both the outputs and inputs sub-queries before it can return even a single result. An unprivileged RPC caller can exploit this to force O(total_matches) memory and I/O on the indexer DB with a single `limit=1` request.

---

### Finding Description

`build_tx_with_cell_union_sub_query` constructs two unbounded sub-queries — one over `output` and one over `input` — and combines them with `UNION ALL`: [1](#0-0) 

Neither the individual sub-queries nor the resulting UNION ALL carries any `LIMIT` clause. The result is used as the innermost sub-query in `get_tx_with_cells`: [2](#0-1) 

This intermediate result is then fed into an outer `SELECT` that applies `GROUP BY` followed by `LIMIT`: [3](#0-2) 

Both SQLite and PostgreSQL must fully evaluate a `GROUP BY` aggregation before they can apply `LIMIT`. There is no way for the query planner to push the `LIMIT` through the `GROUP BY` boundary. The effective SQL shape is:

```sql
SELECT tx_id, ..., GROUP_CONCAT(io_type || ',' || io_index) AS io_pairs
FROM (
  SELECT ... FROM (
    SELECT tx_id, 1 AS io_type, output_index AS io_index FROM output JOIN query_script ...
    UNION ALL
    SELECT consumed_tx_id, 0 AS io_type, input_index AS io_index FROM input JOIN output JOIN query_script ...
    -- ^^^ NO LIMIT HERE
  ) AS res_union
  JOIN ckb_transaction ... JOIN block ...
) AS res
WHERE tx_id > ?
GROUP BY tx_id, block_number, tx_index, tx_hash   -- forces full scan
ORDER BY tx_id
LIMIT 1                                            -- applied AFTER GROUP BY
```

The only guard present is a `request_limit` check on the `limit` parameter value: [4](#0-3) 

This check controls how many rows are *returned*, not how much DB work is performed. The default value is `usize::MAX`: [5](#0-4) 

Critically, the `timeout_limit` feature that exists in the regular indexer (`util/indexer`) is **not implemented** in the rich-indexer path (`util/rich-indexer`). A grep for `timeout_limit` in `util/rich-indexer/**/*.rs` returns zero matches, confirming there is no query timeout protection here.

---

### Impact Explanation

A script that appears in N outputs and M inputs causes the UNION ALL to produce N+M rows. All N+M rows must be read from disk, joined against `ckb_transaction` and `block`, sorted, and aggregated by `GROUP BY` before the `LIMIT` is applied. For a popular lock script on mainnet (e.g., the default secp256k1 lock), N+M can be in the millions. A single `limit=1` RPC call forces the DB to do millions of row reads, sort/hash operations, and potentially spill to temp files — consuming CPU, I/O, and memory proportional to the total match count, not the requested page size. Repeated calls cause sustained resource exhaustion on the indexer node.

---

### Likelihood Explanation

The RPC endpoint is unauthenticated and publicly reachable. The attacker needs only to know (or guess) a script that is widely used — information that is publicly visible on-chain. No special privileges, keys, or hashpower are required. The attack is repeatable with minimal cost to the caller.

---

### Recommendation

Add a `LIMIT` clause to each of the two sub-queries inside `build_tx_with_cell_union_sub_query` when called from the `group_by_transaction=true` path. A safe upper bound is `(after_cursor + limit) * some_multiplier` or a hard cap. Alternatively, restructure the query to use a window function or a two-phase approach: first select the top-N distinct `tx_id` values with `LIMIT`, then fetch their associated `io_type`/`io_index` pairs in a second query. Also apply the `timeout_limit` mechanism (already present in the regular indexer) to the rich-indexer RPC handlers.

---

### Proof of Concept

1. Run a CKB node with the rich-indexer enabled (SQLite or Postgres).
2. Index a chain segment where one lock script appears in 500,000 outputs (e.g., a coinbase script reused across blocks).
3. Call:
   ```json
   {
     "method": "get_transactions",
     "params": [{
       "script": { "code_hash": "0x...", "hash_type": "type", "args": "0x..." },
       "script_type": "lock",
       "group_by_transaction": true
     }, "asc", "0x1", null]
   }
   ```
4. Monitor SQLite WAL / Postgres `pg_stat_activity` temp file usage and query duration. The DB will perform a full scan and sort of all 500,000+ rows before returning the single requested result, demonstrating O(total_matches) work for a `limit=1` request.
