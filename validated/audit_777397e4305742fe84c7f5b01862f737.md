### Title
Unbounded OrphanBlockPool Growth via Capacity-Hint-Only Enforcement — (`chain/src/utils/orphan_block_pool.rs`, `chain/src/init.rs`)

### Summary

`ORPHAN_BLOCK_SIZE` (= `BLOCK_DOWNLOAD_WINDOW` = 8192) is passed to `OrphanBlockPool::with_capacity` and then to `HashMap::with_capacity`, which is a **pre-allocation hint only** — it does not cap the map's size. `OrphanBlockPool::insert` performs no size check and no eviction. An attacker relaying blocks with unknown parent hashes can grow the pool without bound.

### Finding Description

`ORPHAN_BLOCK_SIZE` is defined and used as follows: [1](#0-0) 

It is passed as a capacity hint: [2](#0-1) 

Inside `InnerPool::with_capacity`, it is forwarded only to `HashMap::with_capacity`: [3](#0-2) 

`OrphanBlockPool::insert` unconditionally delegates to `InnerPool::insert` with no size guard: [4](#0-3) 

`InnerPool::insert` unconditionally inserts into `blocks`, `parents`, and `leaders` with no eviction: [5](#0-4) 

The only cleanup mechanism is `clean_expired_blocks`, triggered on a 60-second timer: [6](#0-5) 

It only removes blocks where `epoch_number + EXPIRED_EPOCH (6) < tip_epoch` — roughly 24 hours on mainnet. An attacker sets the epoch field to the current epoch, so blocks are never cleaned up within the attack window: [7](#0-6) 

The code comment explicitly documents the intentional absence of LRU eviction: [8](#0-7) 

No `max_orphan`, per-peer orphan limit, or pool-size guard exists anywhere in the chain layer. A grep for `orphan.*limit`, `max_orphan`, `orphan.*count`, and `orphan.*cap` returns zero matches.

Additionally, `asynchronous_process_block` calls `insert_block` (a DB write) **before** the orphan pool insert, meaning each attacker-supplied orphan block also consumes disk: [9](#0-8) 

### Impact Explanation

An attacker who can relay P2P blocks (unprivileged) can send an unbounded number of structurally valid blocks referencing unknown parent hashes. Each block:
- Passes `non_contextual_verify` (structure/format checks, no chain-state dependency)
- Is written to the node's database
- Is inserted into `OrphanBlockPool` with no cap

This causes unbounded RAM growth (all orphan entries held in `HashMap`) and unbounded disk growth (each block persisted). Sustained attack leads to OOM or disk exhaustion, crashing or severely degrading the node. Impact is scoped as **Low (501–2000): important performance degradation / memory growth**.

### Likelihood Explanation

The attack requires only a P2P connection. No PoW solution is needed to pass non-contextual verification (PoW validity against the declared compact target is contextual). The `process_block_rx` channel is bounded at 24 but is just a processing queue — it throttles throughput, not total orphan accumulation. `MAX_BLOCKS_IN_TRANSIT_PER_PEER = 128` limits concurrent in-flight requests per peer but does not bound total orphan pool size across time or peers.

### Recommendation

Enforce a hard cap inside `OrphanBlockPool::insert`: if `self.inner.read().parents.len() >= capacity`, reject or evict the oldest entry before inserting. The `capacity` field should be stored in `OrphanBlockPool` and checked on every insert. Alternatively, replace the inner `HashMap` with a bounded structure (e.g., a size-limited LRU that also updates `block_status_map` atomically, addressing the comment's stated concern about consistency).

```rust
    fn insert(&mut self, lonely_block: LonelyBlockHash) {
        let hash = lonely_block.hash();
        let parent_hash = lonely_block.parent_hash();
        self.blocks
            .entry(parent_hash.clone())
            .or_default()
            .insert(hash.clone(), lonely_block);
        // Out-of-order insertion needs to be deduplicated
        self.leaders.remove(&hash);
        // It is a possible optimization to make the judgment in advance,
        // because the parent of the block must not be equal to its own hash,
        // so we can judge first, which may reduce one arc clone
        if !self.parents.contains_key(&parent_hash) {
            // Block referenced by `parent_hash` is not in the pool,
            // and it has at least one child, the new inserted block, so add it to leaders.
            self.leaders.insert(parent_hash.clone());
        }
        self.parents.insert(hash, parent_hash);
```
