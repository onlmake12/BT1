### Title
Unbounded `pending_compact_blocks` Growth via Rate-Limit-Exempt CompactBlock Relay Spam — (`File: sync/src/relayer/mod.rs`, `sync/src/relayer/compact_block_process.rs`)

---

### Summary

The `Relayer` protocol handler explicitly exempts `CompactBlock` messages from the per-peer rate limiter. When a received compact block cannot be reconstructed (missing transactions), it is inserted into the shared `pending_compact_blocks` map with no size cap. A miner can continuously relay valid-PoW compact blocks whose transactions are absent from the victim's mempool, causing the map to grow without bound within an epoch, exhausting node memory.

---

### Finding Description

**Root cause 1 — Rate limiter bypass for CompactBlock:**

In `sync/src/relayer/mod.rs`, the `Relayer` struct holds a `RateLimiter<(PeerIndex, u32)>` keyed by `(peer, message_type)` capped at 30 requests per second per key. [1](#0-0) 

However, the rate-limit check is explicitly skipped for every `CompactBlock` message: [2](#0-1) 

The inline comment reads: *"CompactBlock will be verified by POW, it's OK to skip rate limit checking."* This means a peer can send `CompactBlock` messages at an unlimited rate; the only admission gate is PoW header validity.

**Root cause 2 — Unbounded insertion into `pending_compact_blocks`:**

When a compact block cannot be fully reconstructed (missing transactions or uncles), `missing_or_collided_post_process` is called. It inserts the compact block into the shared `pending_compact_blocks` map with no size limit: [3](#0-2) 

The only cleanup of this map occurs inside the `ReconstructionResult::Block` branch — i.e., only when a block is *successfully* reconstructed and accepted: [4](#0-3) 

The `retain` call removes entries from epochs older than the accepted block's epoch. Entries within the **current epoch** are never evicted unless a block from that epoch is accepted. If the attacker's blocks are never accepted (e.g., they are uncle blocks or stale forks), the map accumulates indefinitely for the duration of the epoch (~4 hours, ~1800 blocks).

---

### Impact Explanation

An attacker who is a miner (or controls mining hardware) can:

1. Mine valid-PoW blocks (uncle blocks, fork blocks) whose transaction sets are not present in the victim node's mempool.
2. Relay each as a `CompactBlock` message to the victim — bypassing the rate limiter entirely.
3. Each message triggers `missing_or_collided_post_process`, inserting the full compact block (up to the consensus block size limit) plus a peer map into `pending_compact_blocks`.
4. Because these blocks are never accepted (they are uncle/stale blocks), the cleanup branch never fires for them.
5. The map grows without bound for the entire epoch, consuming memory proportional to `(number_of_crafted_blocks) × (compact_block_size)`.

At scale, this causes memory exhaustion and node crash (OOM), constituting a remote denial-of-service against any reachable CKB full node.

---

### Likelihood Explanation

The attack requires valid PoW, which is a meaningful economic barrier. However:

- A miner with even a small fraction of hashpower (1–5%) produces multiple uncle-eligible blocks per epoch naturally.
- Uncle blocks are valid PoW and pass all header checks (`non_contextual_check`, `contextual_check`, `HeaderVerifier`), so they are accepted into `pending_compact_blocks`.
- The attacker does not need any of the crafted blocks to be accepted on the main chain — only one needs to trigger the missing-transaction path, which is trivially achieved by including a single transaction not in the victim's mempool.
- No privileged access, leaked keys, or social engineering is required — only a P2P connection and mining capability.

Likelihood: **Medium** (requires mining capability; no other privilege needed; attack is sustained over an epoch).

---

### Recommendation

1. **Apply the rate limiter to `CompactBlock` messages.** The PoW check is not a substitute for rate limiting; it only validates authenticity, not submission frequency. Remove the exemption or apply a separate, lower quota (e.g., 2–5 per second per peer) for `CompactBlock`. [5](#0-4) 

2. **Enforce a maximum size on `pending_compact_blocks`.** Before inserting in `missing_or_collided_post_process`, check the map size against a configurable cap (e.g., 32 or 64 entries). Evict the oldest entry (by timestamp, stored as the third tuple element) when the cap is reached. [3](#0-2) 

3. **Add epoch-boundary cleanup proactively.** Do not rely solely on the successful-block path to prune stale entries; run a periodic cleanup task that evicts entries older than the current epoch.

---

### Proof of Concept

1. Attacker Eve controls a miner with modest hashpower.
2. Eve connects to victim node V as a normal P2P peer.
3. Eve mines uncle blocks (valid PoW, parent = current tip or recent ancestor) and for each block includes one transaction `T` that is not in V's mempool.
4. Eve sends `CompactBlock` for each mined block to V via the relay protocol.
5. V's `Relayer::try_process` skips the rate-limit check for `CompactBlock` (line 112–114 of `mod.rs`).
6. V's `CompactBlockProcess::execute` passes `non_contextual_check` and `contextual_check` (valid PoW header).
7. Block reconstruction fails because `T` is missing from V's tx-pool → `ReconstructionResult::Missing`.
8. `missing_or_collided_post_process` inserts the compact block into `pending_compact_blocks` (line 354–361 of `compact_block_process.rs`).
9. Because Eve's blocks are uncle blocks, they are never accepted on the main chain; the cleanup branch (line 106–117) never fires for them.
10. Repeat for the duration of the epoch. `pending_compact_blocks` grows without bound, exhausting V's memory.
