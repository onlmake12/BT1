### Title
Unbounded `numHashes` Growth Allows Attacker to Permanently Block Entropy Provider Fulfillment via Gas Exhaustion - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When an Entropy provider has `maxNumHashes == 0` (the default), any unprivileged user can create a large number of requests without the provider revealing them. Because `numHashes` for each new request equals `assignedSequenceNumber - currentCommitmentSequenceNumber`, the value grows linearly with the backlog. The `constructProviderCommitment` function runs an unbounded `while` loop proportional to `numHashes`. Once the backlog is large enough, the provider's `revealWithCallback` and `advanceProviderCommitment` transactions exceed the block gas limit, permanently blocking fulfillment of those requests.

---

### Finding Description

**Root cause — `constructProviderCommitment` loop is O(numHashes):** [1](#0-0) 

Every call to `revealWithCallback` or `advanceProviderCommitment` invokes this loop with `req.numHashes` iterations. Each `keccak256` costs ~30–200 gas; at a 30 M gas block limit, ~150 k–1 M iterations exhaust the block.

**`numHashes` is set at request creation time and grows with the backlog:** [2](#0-1) 

`numHashes = assignedSequenceNumber − currentCommitmentSequenceNumber`. If the provider has not revealed any requests, every new request increments this value by 1.

**The guard is disabled when `maxNumHashes == 0`:** [3](#0-2) 

The only on-chain cap is skipped entirely when `maxNumHashes == 0`. There is no global floor or default value enforced at registration.

**The Fortuna keeper also skips commitment advancement when `maxNumHashes == 0`:** [4](#0-3) 

The off-chain mitigation (`advanceProviderCommitment`) is never triggered for providers in this state, so the backlog is never cleared automatically.

**`advanceProviderCommitment` itself calls the same unbounded loop:** [5](#0-4) 

Even if the provider tries to manually advance the commitment, the call computes `numHashes = advancedSequenceNumber − currentCommitmentSequenceNumber` hashes. If the gap is large, this transaction also runs out of gas, closing the recovery path.

**Attack entry point — `requestV2` is permissionless:** [6](#0-5) 

Any address can call `requestV2` by paying the provider fee. No registration, whitelist, or rate limit exists.

---

### Impact Explanation

Once the attacker's backlog pushes `numHashes` past the gas-per-block threshold (~150 k–300 k requests depending on gas price), the provider can no longer call `revealWithCallback` for any request with a high sequence number, nor can it call `advanceProviderCommitment` to skip the gap. Requests already in flight with large `numHashes` are permanently stuck: their `callbackStatus` remains `CALLBACK_NOT_STARTED` and the consumer contract never receives its random number. Applications depending on Entropy (lotteries, NFT mints, on-chain games) are permanently frozen for those sequence numbers. The provider's accrued fees remain locked in the contract and cannot be withdrawn without first clearing the stuck requests.

---

### Likelihood Explanation

The attack is reachable by any unprivileged user who can pay the provider fee. The cost scales with the fee: at 0.0001 ETH/request, ~300 k requests costs ~30 ETH. The attack is most effective against providers that have not set `maxNumHashes` (the default state) and against chains with low gas prices where the block gas limit is reached at a higher iteration count. The Fortuna keeper's own warning message acknowledges this scenario explicitly: [7](#0-6) 

---

### Recommendation

1. **Enforce a non-zero `maxNumHashes` at registration.** Add a `require(maxNumHashes > 0)` check in `register` or `requestHelper` so that no provider can accept requests without a gas-bounded hash limit.
2. **Apply the `maxNumHashes` guard unconditionally.** Remove the `providerInfo.maxNumHashes != 0 &&` short-circuit so the check always fires with a protocol-level default when the provider has not set one.
3. **Enable keeper commitment advancement for all providers.** Remove the early-return in `update_commitments_if_necessary` when `max_num_hashes == 0`, or set a safe default in the keeper configuration.

---

### Proof of Concept

1. Provider `P` registers with `maxNumHashes == 0` (default) and `feeInWei = F`.
2. Attacker calls `requestV2(P, randomBytes, 0)` N times in rapid succession, paying `N × F` wei total. No reveals occur between requests.
3. After N requests, `providerInfo.sequenceNumber = N + 1`, `providerInfo.currentCommitmentSequenceNumber = 0`. Request N has `numHashes = N`.
4. Provider's Fortuna keeper attempts `revealWithCallback(P, N, ...)`. This calls `constructProviderCommitment(N, x_N)`, which loops N times. For N ≈ 300 k, gas consumed ≈ 30 M → transaction reverts with out-of-gas.
5. Provider attempts `advanceProviderCommitment(P, N, x_N)`. Same loop, same out-of-gas revert.
6. Provider must reveal requests in strict ascending order (1, 2, 3, …). Each request i still requires i hashes (stored at creation time). Eventually request i requires more gas than the block limit; all requests with sequence number ≥ i are permanently unresolvable.
7. All consumer contracts that requested randomness at sequence numbers ≥ i never receive their `entropyCallback`. Their state is permanently frozen.

    function constructProviderCommitment(
        uint64 numHashes,
        bytes32 revelation
    ) internal pure returns (bytes32 currentHash) {
        currentHash = revelation;
        while (numHashes > 0) {
            currentHash = keccak256(bytes.concat(currentHash));
            numHashes -= 1;
        }
    }
