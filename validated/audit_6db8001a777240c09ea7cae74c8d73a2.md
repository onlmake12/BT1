### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Any Pending Transaction from the Pool — (`rpc/src/module/pool.rs`)

### Summary

The `remove_transaction` JSON-RPC method removes any pending transaction from the tx pool by hash, with no check that the caller is the original submitter or has any authorization over that transaction. The RPC server itself has no authentication middleware. Any caller who can reach the RPC endpoint — including any process on the same host or any remote client on nodes that expose the RPC publicly — can silently evict any other user's pending transaction. This is a direct analog to the `flashRepayFromColl` missing-ownership-check class: a state-mutating operation that should be restricted to the resource owner is callable by anyone.

### Finding Description

**Root cause — no ownership check in `remove_transaction`:**

`rpc/src/module/pool.rs` line 662–669 implements `remove_transaction`:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

The only parameter is `tx_hash`. There is no check that the caller submitted the transaction, owns any of its input cells, or holds any credential. The call goes directly to `TxPoolController::remove_local_tx` → `Message::RemoveLocalTx` → `service.remove_tx(tx_hash)`, which unconditionally removes the transaction and all its dependents from the pool.

**No authentication on the RPC server:**

`rpc/src/server.rs` builds the Axum router with `CorsLayer::permissive()` and no authentication middleware. Every incoming POST request is dispatched directly to `io.handle_call(call, T::default())`. There is no token, session, or IP-based auth enforced at the server level.

**Attack path:**

1. Victim submits a transaction via `send_transaction`; the tx hash is now public via `get_raw_tx_pool`.
2. Attacker calls `remove_transaction` with the victim's tx hash.
3. `remove_local_tx` removes the transaction and all its descendants from the pool unconditionally.
4. The victim's transaction is silently gone; the victim must resubmit and pay fees again, and may miss time-sensitive windows (e.g., `since`-locked cells, DAO withdrawal deadlines, competitive block inclusion).

**Scope of the `remove_transaction` method:**

The `Pool` module is enabled by default in `resource/ckb.toml` (`modules = ["Net", "Pool", "Miner", "Chain", ...]`). The method is part of the standard production API surface, not a debug or integration-test-only endpoint.

### Impact Explanation

- **Transaction censorship**: An attacker can continuously evict a target's transactions from the pool, preventing confirmation indefinitely.
- **Missed time windows**: Transactions using `since` (relative/absolute time locks) or DAO withdrawal transactions that are removed from the pool may miss their valid confirmation window, causing permanent loss of the time-sensitive opportunity.
- **Fee waste**: Every eviction forces the victim to resubmit and pay fees again.
- **Cascading removal**: The implementation removes the target transaction *and all descendants*, so a single call can evict an entire chain of dependent transactions built by the victim.

### Likelihood Explanation

The RPC default bind address is `127.0.0.1:8114` (localhost). However:
- Any process running on the same host (e.g., a co-located malicious script, another tenant on a shared server, or a compromised dependency) can reach it without any credential.
- Many infrastructure operators (exchanges, dApps, block explorers) expose the RPC publicly by changing `listen_address` — a supported and documented configuration option.
- The `get_raw_tx_pool` method (also unauthenticated) provides the attacker with all pending tx hashes needed to target specific victims.
- The attack requires only a single HTTP POST with a known tx hash; it is trivially scriptable and repeatable.

### Recommendation

Add caller-identity verification to `remove_transaction`. Since CKB's RPC has no session concept, the practical fix is one of:
1. **Remove the method from the public API** or move it to a separate privileged/admin-only RPC module that is disabled by default.
2. **Add an API token / HTTP Basic Auth layer** to the RPC server (analogous to how Bitcoin Core's RPC requires `rpcuser`/`rpcpassword`) so that only the node operator can call destructive pool-management methods (`remove_transaction`, `clear_tx_pool`, `clear_tx_verify_queue`).
3. At minimum, document that `remove_transaction` is an operator-only method and gate it behind a separate module flag (e.g., `"PoolAdmin"`) that is off by default.

### Proof of Concept

```
# Step 1: Victim submits a transaction
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<victim_tx>, "passthrough"]}'
# Returns: {"result": "0xVICTIM_TX_HASH"}

# Step 2: Attacker enumerates pool (no auth required)
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}'
# Returns all pending tx hashes including 0xVICTIM_TX_HASH

# Step 3: Attacker removes victim's transaction (no auth required)
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":3,"jsonrpc":"2.0","method":"remove_transaction","params":["0xVICTIM_TX_HASH"]}'
# Returns: {"result": true}
# Victim's transaction and all its descendants are now gone from the pool.
```
