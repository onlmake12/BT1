### Title
Unbounded Cell Scan in `get_cells_capacity` Indexer RPC Causes Unbounded CPU/Memory Consumption — (`rpc/src/module/indexer.rs`, `util/indexer/src/service.rs`)

---

### Summary

The `get_cells_capacity` RPC endpoint in the CKB built-in indexer accepts a `search_key` with no upper bound on the number of cells it will scan. An attacker who has deployed many on-chain cells matching a chosen script prefix can call this endpoint to force the node to iterate over an unbounded number of cells, exhausting CPU and memory. Unlike `get_cells` (which accepts a caller-supplied `limit`) and the rich indexer's `get_cells` (which enforces `self.request_limit`), `get_cells_capacity` has no pagination or result-count guard at all.

---

### Finding Description

`get_cells_capacity` in the basic indexer RPC module accepts only a `search_key` and returns the summed capacity of all matching live cells. There is no `limit` parameter and no internal cap on how many cells the function will scan. [1](#0-0) 

The implementation in the indexer service iterates over every cell that matches the search key, applying per-cell filters including a `memmem::find` partial-match scan over raw cell output data: [2](#0-1) 

There is no early exit, no page limit, and no cycle budget. The loop runs until every matching cell in the entire indexed chain history has been visited.

By contrast, the rich indexer's `get_cells` enforces `self.request_limit`: [3](#0-2) 

The basic indexer's `get_cells` accepts a caller-supplied `Uint32` limit but passes it directly to the handle with no upper-bound enforcement: [4](#0-3) 

`get_cells_capacity` does not even have a limit parameter: [1](#0-0) 

The RPC batch limit is also disabled by default, meaning an attacker with RPC access can issue many concurrent `get_cells_capacity` calls in a single batch request: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

An attacker who has deployed a large number of cells sharing a common lock/type script prefix (or any prefix that matches many existing cells) can call `get_cells_capacity` with that prefix. The indexer will iterate over every matching cell, performing per-cell filtering (including `memmem::find` partial scans over cell output data). With enough matching cells, this:

- Blocks the RPC worker thread for an extended period (CPU exhaustion).
- Allocates unbounded intermediate state proportional to the number of matching cells (memory exhaustion).
- Can be amplified by sending a large JSON-RPC batch (no default batch limit) of `get_cells_capacity` calls in a single 10 MiB HTTP request.

The result is denial of service of the node's RPC service for the duration of the scan.

---

### Likelihood Explanation

The setup cost is real: the attacker must deploy many cells on-chain, which requires spending CKB tokens. However, this is analogous to the Hats Protocol report where the attacker must spend gas to incrementally grow the string. On CKB, cells with minimal data and capacity can be created cheaply in bulk. Once deployed, the cells are permanent (unless the attacker spends more to destroy them), so the DoS condition persists indefinitely. Any subsequent caller of `get_cells_capacity` with the matching prefix will trigger the expensive scan.

The RPC is bound to `127.0.0.1` by default, limiting the attacker to local access. However:
- Many operators expose the RPC port publicly (explicitly warned against but common in practice).
- A malicious local process on the same host qualifies as an "unprivileged RPC caller" per the RESEARCHER.md attacker profiles.
- The `rpc_batch_limit` is disabled by default, amplifying the attack. [7](#0-6) 

---

### Recommendation

1. **Add a hard upper bound to `get_cells_capacity`**: Introduce a `max_cells_capacity_scan_limit` configuration option (analogous to `request_limit` in the rich indexer) and reject or truncate scans that would exceed it.

2. **Enforce an upper bound on the `limit` parameter of the basic indexer's `get_cells`**: Mirror the rich indexer's check (`if limit as usize > self.request_limit`) in the basic indexer handle.

3. **Enable `rpc_batch_limit` by default**: Change the default configuration from `None` (unlimited) to a reasonable value (e.g., 2000) so that batch amplification is bounded out of the box. [8](#0-7) 

---

### Proof of Concept

**Setup (attacker, one-time cost):**
1. Deploy N cells (e.g., N = 100,000) each with the same lock script `code_hash` and a common `args` prefix `0xdeadbeef`.
2. Each cell requires only the minimum capacity (~61 CKB). At current prices this is a finite, bounded cost.

**Attack (repeatable, zero marginal cost after setup):**
```json
POST http://<node-rpc>/
Content-Type: application/json

[
  {"jsonrpc":"2.0","id":1,"method":"get_cells_capacity","params":[{"script":{"code_hash":"0x<attacker_code_hash>","hash_type":"type","args":"0xdeadbeef"},"script_type":"lock","script_search_mode":"prefix"}]},
  ... (repeated 2000+ times in one batch, no default batch limit)
]
```

Each `get_cells_capacity` call forces the indexer to scan all 100,000 matching cells. With no batch limit, a single 10 MiB HTTP request can contain thousands of such calls, each triggering the full scan, saturating the RPC thread pool. [9](#0-8) [1](#0-0) [6](#0-5)
