### Title
Fortuna Keeper Never Retries `CALLBACK_FAILED` Entropy Requests After Restart — (`apps/fortuna/src/keeper/block.rs`)

---

### Summary

The Fortuna keeper (Entropy fulfillment service) discovers pending requests exclusively by scanning for `Requested2` on-chain events within a fixed backlog window. It has no mechanism to discover requests that are already on-chain in the `CALLBACK_FAILED` state. After a keeper restart, any request whose original `Requested2` event falls outside the backlog window and whose callback previously failed will be permanently abandoned, even though the Entropy contract explicitly permits and expects `revealWithCallback` to be called again on such requests.

---

### Finding Description

The Entropy contract (`Entropy.sol`) introduced a `CALLBACK_FAILED` state (`EntropyStatusConstants.CALLBACK_FAILED = 3`). When a callback reverts or runs out of gas, the request is **not cleared** — it remains on-chain with `callbackStatus = CALLBACK_FAILED`, and `revealWithCallback` can be called again to retry delivery. [1](#0-0) 

The contract's `revealWithCallback` explicitly accepts both `CALLBACK_NOT_STARTED` and `CALLBACK_FAILED` as valid entry states: [2](#0-1) 

The Fortuna keeper discovers requests to fulfill by scanning block ranges for `Requested2` events via `get_request_with_callback_events`: [3](#0-2) 

On startup, `run_keeper_threads` processes only the last `backlog_range` blocks for `Requested2` events: [4](#0-3) 

The `fulfilled_requests_cache` is an in-memory `HashSet` that is empty on every restart: [5](#0-4) 

**There is no code path that scans for requests currently in `CALLBACK_FAILED` state.** The keeper only ever learns about a request from its original `Requested2` event. If that event is outside the backlog window at restart time, the `CALLBACK_FAILED` request is silently dropped.

A compounding issue exists in the backup-replica path: `process_event_with_backoff` explicitly skips any request whose on-chain `callback_status` is not `CallbackNotStarted`, which includes `CALLBACK_FAILED`: [6](#0-5) 

This means even if a backup replica re-encounters the original `Requested2` event, it will skip the request once it sees `CALLBACK_FAILED` on-chain, preventing any retry.

---

### Impact Explanation

An Entropy user whose callback contract reverts or exceeds its gas limit will have their request stuck in `CALLBACK_FAILED` state. If the keeper restarts after the original request block falls outside the `backlog_range` window, the keeper will never retry the callback. The user's `entropyCallback` is never delivered, permanently breaking the randomness flow for that request. The request remains locked on-chain consuming storage, and the user has no recourse other than calling `revealWithCallback` themselves.

---

### Likelihood Explanation

`CALLBACK_FAILED` is a normal operational state — any callback that reverts or runs out of gas reaches it. Keeper restarts are routine (deployments, crashes, chain RPC failures). The `backlog_range` is a finite configured window. Any request older than `backlog_range` blocks at restart time that is in `CALLBACK_FAILED` state will be permanently missed. This is a realistic, non-adversarial scenario for any Entropy user with a buggy or gas-constrained callback.

---

### Recommendation

**Short term:** On keeper startup, after processing the backlog of `Requested2` events, additionally query the contract for all requests currently in `CALLBACK_FAILED` state (e.g., by iterating over known sequence numbers or by listening for `CallbackFailed` events in the same block range) and enqueue them for retry.

**Long term:** Decouple request discovery from the original `Requested2` event. Maintain a persistent store of in-flight and failed requests so that a restart does not lose state. Fix the backup-replica check to treat `CALLBACK_FAILED` as a retryable state rather than a terminal one.

---

### Proof of Concept

1. User submits an Entropy request with a gas-limited callback contract that reverts.
2. Keeper calls `revealWithCallback`; the callback reverts; the contract sets `callbackStatus = CALLBACK_FAILED`. The request remains on-chain.
3. Keeper restarts (e.g., due to a deployment). `fulfilled_requests_cache` is reset to empty.
4. `process_backlog` scans only the last `backlog_range` blocks for `Requested2` events.
5. If the original request was made more than `backlog_range` blocks ago, no `Requested2` event is found for it.
6. No code path queries `getRequestV2` for requests in `CALLBACK_FAILED` state.
7. The request is permanently abandoned. The user's callback is never retried. [7](#0-6) [8](#0-7)
