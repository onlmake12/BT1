Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Unique Attacker-Controlled `(PeerId, PeerId)` Keys — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

`HolePunching.forward_rate_limiter` is backed by a `governor::state::keyed::HashMapStateStore<(PeerId, PeerId, u32)>` whose only eviction path is `retain_recent()` inside `disconnected()`. An attacker holding a single persistent session can send `ConnectionRequest` messages at the per-session cap (30/sec), each carrying freshly generated attacker-controlled `from`/`to` PeerIds, inserting one new key per message with zero eviction for the session lifetime. Memory grows linearly and without bound, eventually causing an OOM crash of the node.

## Finding Description

`HolePunching` declares the rate limiter type as `HashMapStateStore`-backed:

```rust
// network/src/protocols/hole_punching/mod.rs L31-35
type RateLimiter<T> = governor::RateLimiter<
    T,
    governor::state::keyed::HashMapStateStore<T>,
    governor::clock::DefaultClock,
>;
```

The two fields are:

```rust
// L45-46
rate_limiter: RateLimiter<(PeerIndex, u32)>,
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,
```

`retain_recent()` is called on both only inside `disconnected()` (L67-68), never inside `notify()` (L169-175). The `notify()` handler fires every 5 minutes and cleans `pending_delivered` and `inflight_requests` but leaves both rate limiters untouched.

In `received()`, the outer guard `rate_limiter` is keyed by `(session_id, msg.item_id())` at 30 req/sec — a bounded key space (one entry per session per message type). Every message that passes then reaches `ConnectionRequestProcess::execute()`, which calls:

```rust
// connection_request.rs L132-135
self.protocol
    .forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
```

`content.from` and `content.to` are parsed directly from the attacker-supplied message body:

```rust
// connection_request.rs L36-40
let from = PeerId::from_bytes(value.from().raw_data().to_vec())...;
let to   = PeerId::from_bytes(value.to().raw_data().to_vec())...;
```

`governor`'s `check_key` on a `HashMapStateStore` inserts a new entry for every previously unseen key. Because the attacker supplies unique `(from, to)` pairs, every message inserts a new entry. The `forward_rate_limiter` quota is 1 req/sec per key, so entries become stale after ~1 second, but `retain_recent()` — the sole eviction mechanism — is never invoked during an active session. Stale entries accumulate indefinitely.

Existing guards that are insufficient:
- The outer `rate_limiter` cap (30/sec) limits insertion rate but does not bound total entries.
- The `notify()` cleanup of `pending_delivered` and `inflight_requests` does not touch either rate limiter.
- There is no per-session entry cap or maximum map size on `forward_rate_limiter`.

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Growth rate: 30 entries/sec × ~150 bytes/entry ≈ 4.3 KB/sec. On a 4 GB node this reaches OOM in approximately 11 days of continuous flooding from a single session. With multiple concurrent sessions the timeline scales down proportionally. The crash is deterministic, requires no special privileges, and no victim interaction. The result is a full local node crash (OOM), which matches the allowed High impact class.

## Likelihood Explanation

- HolePunching must be enabled via `support_protocols` config — an opt-in but supported production feature.
- Attacker needs exactly one persistent P2P connection; no elevated privileges required.
- Generating unique PeerIds is trivial (random 32-byte multihash values).
- The attack is slow (days per single session) but fully deterministic and requires no victim interaction.
- Multiple sessions reduce the timeline proportionally.
- The `notify()` timer at 5-minute intervals is the natural fix point but is unused for this cleanup, confirming the invariant is definitively broken.

## Recommendation

Add `retain_recent()` calls inside `notify()` alongside the existing cleanup:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    // ... existing pending_delivered / inflight_requests cleanup
```

This ensures stale entries are evicted every 5 minutes regardless of whether any peer disconnects, bounding the map size to at most `30 entries/sec × 300 sec = 9,000 entries` between notify ticks.

## Proof of Concept

1. Enable HolePunching on a test node (`support_protocols` includes `HolePunching`).
2. Connect one session and send `ConnectionRequest` messages at 30/sec, each with a freshly generated random `from` and `to` PeerId (valid 32-byte multihash values).
3. After N seconds, inspect the `forward_rate_limiter` internal state — key count equals N × 30 with zero eviction.
4. Confirm `disconnected()` is never triggered (session stays alive); confirm `notify()` fires every 5 minutes but does not reduce key count.
5. Memory grows linearly without bound for the session lifetime, eventually causing OOM.
