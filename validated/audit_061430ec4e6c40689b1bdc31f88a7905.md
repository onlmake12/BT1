### Title
Off-by-one in `Scheduler::process_message_box` allows 17 concurrent VMs when limit is 16 — (`script/src/scheduler.rs`)

### Summary
The spawn guard at line 532 uses `>` instead of `>=` against `MAX_VMS_COUNT`. A script author can spawn exactly 16 VMs and then issue one additional spawn syscall that bypasses the limit, creating a 17th VM in violation of the stated invariant.

### Finding Description
`MAX_VMS_COUNT` is defined as `16` at line 34. The guard at line 532 reads:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
```

When 16 VMs are alive (the spawning VM itself is in `instantiated`, plus 15 others), the expression evaluates to `16 > 16 == false`, so the early-return with `MAX_VMS_SPAWNED` is skipped and `boot_vm` is called, producing a 17th VM. The correct operator is `>=`.

### Impact Explanation
- The resource invariant "at most `MAX_VMS_COUNT` concurrent VMs per script group" is violated by one.
- Each extra VM carries its own memory allocation and cycle accounting. While the overshoot is bounded to exactly one extra VM per script group execution, it is attacker-controlled and deterministic.
- Because `MAX_VMS_COUNT` is a consensus-critical constant, any node that enforces the limit correctly (e.g., after a patch) would reject a block accepted by an unpatched node, creating a consensus split.

### Likelihood Explanation
The exploit path is straightforward: submit a transaction whose lock/type script (CKB-VM V2) spawns 16 child VMs and then issues one more `spawn` syscall. No privileged access, no PoW, no key material required. It is locally reproducible with a unit test.

### Recommendation
Change `>` to `>=` at line 532:

```rust
// Before
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
// After
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

### Proof of Concept
1. Write a CKB-VM V2 script that calls `spawn` in a loop 17 times, recording each return code.
2. The first 15 spawns succeed (return `SUCCESS`); the 16th should return `MAX_VMS_SPAWNED` but instead returns `SUCCESS` due to the off-by-one.
3. The 17th spawn returns `MAX_VMS_SPAWNED`.
4. Assert that the 16th spawn returns `MAX_VMS_SPAWNED` — the test will fail on the unpatched code, confirming the bug.
