### Title
Ineffective Minimum Balance Enforcement in `updatePriceFeeds` Allows Active Subscriptions to Be Drained Below Minimum - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `Scheduler` contract enforces a `minimumBalance` invariant at subscription creation, update, fund addition, and withdrawal. However, `updatePriceFeeds` deducts both the Pyth fee and the keeper fee from the subscription balance without any post-deduction check against `minimumBalance`. Any permissionless keeper can repeatedly call `updatePriceFeeds` to drain an active subscription's balance below the minimum, leaving the subscription in a state that violates the protocol's own invariant.

### Finding Description

The `Scheduler` contract defines a `minimumBalancePerFeed` parameter that is enforced across all user-facing balance operations:

- `createSubscription`: requires `msg.value >= minimumBalance` before creating. [1](#0-0) 
- `updateSubscription`: requires `currentStatus.balanceInWei >= minimumBalance` when the subscription is or will be active. [2](#0-1) 
- `addFunds`: checks `status.balanceInWei >= minimumBalance` after adding. [3](#0-2) 
- `withdrawFunds`: checks `status.balanceInWei - amount >= minimumBalance` before withdrawing. [4](#0-3) 

However, `updatePriceFeeds` only checks that the balance covers the immediate Pyth fee before deducting it, and then `_processFeesAndPayKeeper` only checks that the balance covers the keeper fee before deducting it. There is **no post-deduction check** that the remaining balance is still at or above `minimumBalance`:

```solidity
// Only checks balance >= pythFee, not >= minimumBalance
if (status.balanceInWei < pythFee) {
    revert SchedulerErrors.InsufficientBalance();
}
status.balanceInWei -= pythFee;
// ... parse, validate, store ...
_processFeesAndPayKeeper(status, startGas, params.priceIds.length);
// ^ only checks balance >= keeperFee, not >= minimumBalance
emit PricesUpdated(subscriptionId, latestPublishTime);
``` [5](#0-4) 

The `_processFeesAndPayKeeper` function similarly only checks `status.balanceInWei < totalKeeperFee` before deducting, with no minimum balance guard: [6](#0-5) 

The existing test suite **explicitly confirms** this behavior — after multiple `updatePriceFeeds` calls, the balance drops below `minimumBalance` and the subscription remains active: [7](#0-6) 

### Impact Explanation

1. **Invariant violation**: Active subscriptions can hold a balance below `minimumBalance`, directly violating the protocol's stated invariant: *"A minimum balance must be maintained for active subscriptions."* [8](#0-7) 

2. **Subscription remains active in an underfunded state**: The subscription stays in the active list and continues to be discoverable by keepers via `getActiveSubscriptions`, even though it no longer meets the minimum balance requirement. Keepers will continue to attempt updates, further draining the balance.

3. **Manager locked into a large top-up**: Because `addFunds` enforces the minimum balance post-addition, the manager cannot add a small amount to cover the next update — they must add enough to bring the total back above `minimumBalance` in a single call. If the balance has been drained to near zero, this could require a large deposit.

4. **Minimum balance protection is entirely ineffective against fee drainage**: The security parameter `minimumBalancePerFeed` is bypassed by the most frequent operation in the protocol.

### Likelihood Explanation

`updatePriceFeeds` is permissionless — any address can call it as a keeper. [9](#0-8)  A subscription funded at exactly the minimum balance will drop below minimum after the very first successful `updatePriceFeeds` call, since fees are deducted from the balance. This is not a corner case; it is the normal operating path for any subscription that is not continuously topped up.

### Recommendation

After all fee deductions in `updatePriceFeeds`, add a post-condition check that the remaining balance is at or above `minimumBalance`. If the balance falls below minimum, either revert the update or automatically deactivate the subscription:

```solidity
// After _processFeesAndPayKeeper:
uint256 minimumBalance = this.getMinimumBalance(uint8(params.priceIds.length));
if (params.isActive && status.balanceInWei < minimumBalance) {
    params.isActive = false;
    _removeFromActiveSubscriptions(subscriptionId);
    emit SubscriptionDeactivated(subscriptionId);
}
```

This mirrors the pattern used in `withdrawFunds` and `addFunds`, ensuring the minimum balance invariant is maintained across all state-changing operations.

### Proof of Concept

1. Manager creates a subscription with exactly `minimumBalance` (e.g., `getMinimumBalance(2)` for 2 feeds).
2. Keeper calls `updatePriceFeeds`. The function deducts `pythFee` + `keeperFee` from `status.balanceInWei`. No minimum balance check is performed after deduction.
3. After the call, `status.balanceInWei < minimumBalance`. The subscription is still active.
4. The existing test `testAddFundsEnforcesMinimumBalance` in `PulseScheduler.t.sol` (lines 727–758) already demonstrates this: after 5 `updatePriceFeeds` calls, `statusAfterUpdates.balanceInWei < minimumBalance` is asserted as `true`, confirming the invariant is broken by normal keeper operation. - Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`. The main goal of making this component a permissionless network rather a set of permissioned nodes is to enhance reliability for the feeds -- if one provider fails, others should be available to service the subscriptions. We can improve this reliability by sourcing independent providers, and by making it profitable to push updates, paid out by the users of the feeds.

- Keepers are paid directly by the subscription's funds held in this contract for each successful update they perform. The payment covers gas costs plus a premium, and payment is sent directly to `msg.sender` (the keeper) at the end of `updatePriceFeeds`. The first transaction included in a block that passes checks will succeed and receive the payment. Subsequent attempts for the same update interval will revert since we verify the update criteria on-chain. By only allowing updates when they are needed, we keep costs predictable for the users.
