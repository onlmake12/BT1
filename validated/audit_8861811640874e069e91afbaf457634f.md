### Title
Unbounded `forward_rate_limiter` heap growth via unique `(from, to)` pairs in long-lived HolePunching sessions â€” (`network/src/protocols/hole_punching/mod.rs`)

### Summary
The `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` in `HolePunching` uses a `HashMapStateStore` that accumulates one entry per unique `(from, to, item_id)` triple. `retain_recent()` is called only in `disconnected()`, never in `notify()`. An unprivileged peer can send messages with unique attacker-crafted `from`/`to` PeerIds at the outer rate limit (30/s per message type), causing the map to grow without bound for the lifetime of the session.

### Finding Description
In `network/src/protocols/hole_punching/mod.rs`, `HolePunching::disconnected()` calls `self.forward_rate_limiter.retain_recent()` to evict stale entries. However, `notify()` â€” which fires on a 5-minute timer â€” only prunes `pending_delivered` and `inflight_requests`, leaving `forward_rate_limiter` untouched. The `governor` crate's `HashMapStateStore` inserts a new entry on every `check_key()` call for a previously-unseen key. Because `from` and `to` are parsed from the message body (not from the session identity), an attacker can supply an unbounded number of unique `(from, to)` pairs. The outer per-session rate limiter (`rate_limiter` keyed by `(session_id, item_id)`) limits throughput to 30 messages/second per message type but does not cap the total number of distinct keys inserted into `forward_rate_limiter`.

### Impact Explanation
A single malicious peer maintaining a long-lived connection can grow the node's heap by ~90 entries/second Ã— ~200 bytes/entry â‰ˆ 18 KB/s. Over 24 hours this reaches ~1.5 GB. Multiple simultaneous attackers scale linearly. This causes progressive memory pressure, potential OOM kills, and degraded node performance â€” matching the "Low (501-2000): important performance degradation via unbounded heap growth" scope.

### Likelihood Explanation
Any peer that can establish a HolePunching protocol session (standard P2P connection, no privilege required) can trigger this. The attacker only needs to maintain the connection and send valid-format messages with unique `from`/`to` PeerIds. No PoW, no key material, no special role is needed.

### Recommendation
Call `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside `notify()` on every timer tick, alongside the existing `pending_delivered` and `inflight_requests` pruning. This ensures stale entries are evicted every 5 minutes regardless of whether peers disconnect. [6](#0-5)
