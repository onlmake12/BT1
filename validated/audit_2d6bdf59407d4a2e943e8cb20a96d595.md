Audit Report

## Title
Unbounded `tx_hashes` Array in `get_tx_indices` Enables RPC DoS - (`rpc/src/module/chain.rs`)

## Summary
`get_tx_indices` accepts an arbitrarily large `tx_hashes` vector and performs one synchronous RocksDB read per entry with no upper-bound guard. Both `get_transaction_proof` and `get_transaction_and_witness_proof` pass their input directly to this helper. The light-client P2P path enforces a hard cap of 1,000 hashes; the RPC path does not. Repeated requests with large valid hash arrays can saturate the RPC thread pool and RocksDB read path, degrading or halting RPC service for all callers.

## Finding Description
`get_tx_indices` at [1](#0-0)  checks only for an empty input, then unconditionally iterates over every hash calling `snapshot.get_transaction_info` (a synchronous RocksDB read) for each one. [2](#0-1) 

Both public entry points pass `tx_hashes: Vec<H256>` directly to this helper with no pre-validation: `get_transaction_proof` [3](#0-2)  and `get_transaction_and_witness_proof`. [4](#0-3) 

After `get_tx_indices` returns, `get_transaction_proof` additionally calls `CBMT::build_merkle_proof` over all block transactions. [5](#0-4) 

By contrast, the light-client P2P handler enforces a hard cap of 1,000 at: [6](#0-5)  defined in: [7](#0-6) 

The only existing guard is the 10 MiB request body limit, which still permits approximately 151,967 hashes per request. [8](#0-7) 

## Impact Explanation
This matches **Note (0–500 points): Any local RPC API crash**. An attacker with access to the RPC endpoint can flood it with requests containing tens of thousands of valid transaction hashes, forcing proportional RocksDB reads and Merkle proof construction per request. This degrades or halts RPC service for all other callers. The node's consensus and P2P layers are unaffected; only the RPC service is impacted.

## Likelihood Explanation
The attack requires only: (1) access to the RPC port, and (2) valid transaction hashes from a single block, freely available from any block explorer or via `get_block`. No authentication is required. The default listen address is local-only (`127.0.0.1:8114`), but many operators expose the RPC on internal networks or via reverse proxy. The attack is trivially automatable and costs only network bandwidth.

## Recommendation
Add an upper-bound check at the start of `get_tx_indices`, consistent with the light-client protocol limit:

```rust
const MAX_TX_HASHES_PER_PROOF_REQUEST: usize = 1000;

fn get_tx_indices(&self, tx_hashes: Vec<H256>, ...) -> Result<...> {
    if tx_hashes.is_empty() {
        return Err(RPCError::invalid_params("Empty transaction hashes"));
    }
    if tx_hashes.len() > MAX_TX_HASHES_PER_PROOF_REQUEST {
        return Err(RPCError::invalid_params(format!(
            "Too many transaction hashes, limit is {MAX_TX_HASHES_PER_PROOF_REQUEST}"
        )));
    }
    ...
}
```

The same limit should be applied to the `proof.indices` and `proof.lemmas` arrays in `verify_transaction_proof` and `verify_transaction_and_witness_proof`, which are also unbounded and fed directly into `CBMT::retrieve_leaves`. [9](#0-8) 

## Proof of Concept
1. Query `get_block` on a block with many transactions to collect all tx hashes.
2. Send the following in a tight loop:
```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_transaction_proof",
  "params": [["0x<tx_hash_1>", ..., "0x<tx_hash_N>"], null]
}
```
3. Each request forces N synchronous RocksDB reads in `get_tx_indices` plus `CBMT::build_merkle_proof` over all block transactions. Flooding the endpoint with such requests exhausts the RPC server's capacity to serve legitimate callers.
