### Title
Arithmetic Underflow in `ema_price_no_older_than` Causes Panic Instead of `OutdatedPrice` Error — (`target_chains/fuel/contracts/pyth-contract/src/main.sw`)

---

### Summary

The private `ema_price_no_older_than` function in the Fuel Pyth contract performs an unchecked unsigned subtraction `current_time - price.publish_time` without first verifying that `current_time >= price.publish_time`. In Sway, unsigned integer subtraction panics on underflow. The sibling function `price_no_older_than` explicitly guards against this with a saturating-subtraction pattern and a comment documenting the intent, but `ema_price_no_older_than` was never given the same fix. When a price update is stored with a `publish_time` ahead of the Fuel block timestamp, every call to the public `ema_price` or `ema_price_no_older_than` entry points panics with an arithmetic underflow instead of returning the expected `PythError::OutdatedPrice`.

---

### Finding Description

Two private helper functions implement the "no older than" staleness check:

**`price_no_older_than`** (lines 331–343) — correctly guarded:
```sway
// Mimicking saturating subtraction to avoid underflow
let time_difference = if current_time > price.publish_time {
    current_time - price.publish_time
} else {
    0
};
require(time_difference <= time_period, PythError::OutdatedPrice);
``` [1](#0-0) 

**`ema_price_no_older_than`** (lines 311–320) — missing the guard:
```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,  // ← underflows if publish_time > current_time
        PythError::OutdatedPrice,
    );
    price
}
``` [2](#0-1) 

Both public entry points delegate to the vulnerable private function:

```sway
fn ema_price(price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(valid_time_period(), price_feed_id)
}

fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(time_period, price_feed_id)
}
``` [3](#0-2) 

---

### Impact Explanation

When `price.publish_time > current_time` (Fuel block timestamp), the expression `current_time - price.publish_time` wraps below zero for a `u64`, triggering a Sway VM arithmetic panic. The transaction reverts with a generic VM error rather than `PythError::OutdatedPrice`. Every consumer on Fuel that calls `ema_price` or `ema_price_no_older_than` is denied service for the duration that the stored EMA publish time exceeds the block timestamp. This is a denial-of-service on EMA price reads for all Fuel-based protocols relying on Pyth EMA prices.

---

### Likelihood Explanation

Pyth price feeds are published from Pythnet, whose clock can run slightly ahead of the Fuel block timestamp. A price updater (an unprivileged role — anyone can call `update_price_feeds`) submits a valid Wormhole-signed accumulator update whose EMA `publish_time` is a few seconds in the future relative to the Fuel block. This is a normal operating condition, not an attack: Pyth's own documentation acknowledges clock skew between source and target chains. The `price_no_older_than` function was already patched for exactly this reason (the comment "Mimicking saturating subtraction to avoid underflow" confirms the developers are aware of the scenario), but the EMA counterpart was missed.

---

### Recommendation

Apply the same saturating-subtraction guard to `ema_price_no_older_than`:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

This mirrors the fix already present in `price_no_older_than`. [4](#0-3) 

---

### Proof of Concept

1. A price updater calls `update_price_feeds` with a valid Wormhole-signed accumulator update where the EMA price's `publish_time` is `T+5` (5 seconds ahead of the current Fuel block timestamp `T`). This is accepted because Pyth does not reject future-timestamped updates.
2. Any user (or protocol contract) calls `ema_price(price_feed_id)`.
3. Inside `ema_price_no_older_than`: `current_time = T`, `price.publish_time = T+5`.
4. `current_time - price.publish_time` = `T - (T+5)` = underflow on `u64` → VM arithmetic panic.
5. The call reverts with a generic panic, not `PythError::OutdatedPrice`.
6. All EMA price consumers on Fuel are blocked until the Fuel block timestamp catches up past `T+5`. [2](#0-1)
