### Title
Unbounded `callbackGasLimit` in `requestPriceUpdatesWithCallback` Permanently Locks User Funds — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo.sol` contract accepts an unbounded `callbackGasLimit` parameter from any unprivileged user in `requestPriceUpdatesWithCallback`. When this value exceeds the EVM block gas limit, the corresponding `executeCallback` can never be successfully executed, and there is no refund mechanism, permanently locking the user's ETH in the contract.

---

### Finding Description

In `Echo.requestPriceUpdatesWithCallback`, the `callbackGasLimit` parameter is a `uint32` with no upper bound validation: [1](#0-0) 

The stored `callbackGasLimit` is later forwarded directly to the consumer callback in `executeCallback`: [2](#0-1) 

`type(uint32).max` is `4,294,967,295` gas — roughly **143× the current Ethereum block gas limit (~30M)**. If a user submits a request with `callbackGasLimit = type(uint32).max`, the provider can never call `executeCallback` successfully, because the outer transaction would need to supply at least `1.5 × callbackGasLimit` gas (as documented in the interface), which is physically impossible within a single block. [3](#0-2) 

The fee calculation in `getFee` does not prevent this: when `feePerGasInWei` is small (e.g., 1 wei), `callbackGasLimit * feePerGasInWei` remains within `uint96` range, so the request is accepted and ETH is transferred into the contract: [4](#0-3) 

The provider fee portion of `msg.value` is stored in `req.fee` and only credited to the provider upon a successful `executeCallback` call: [5](#0-4) [6](#0-5) 

There is **no refund function** in the contract. The existing TODO comment in the code explicitly acknowledges this risk: [7](#0-6) 

---

### Impact Explanation

A user who submits a request with `callbackGasLimit` exceeding the block gas limit loses their ETH permanently. The provider fee portion of `msg.value` is stored in `req.fee` and can never be released — there is no `refundRequest`, no timeout-based withdrawal, and no admin recovery path for individual stuck requests. The Pyth protocol fee portion (`pythFeeInWei`) is immediately credited to `_state.accruedFeesInWei` and is withdrawable by the admin, but the provider fee is irrecoverably locked.

Additionally, stuck requests cause `_state.firstUnfulfilledSeq` to stall, degrading the `getFirstActiveRequests` view function for off-chain keepers. [8](#0-7) 

---

### Likelihood Explanation

Any unprivileged user can trigger this by calling `requestPriceUpdatesWithCallback` with `callbackGasLimit = type(uint32).max`. When `feePerGasInWei` is small (e.g., 1 wei), the required fee is negligible (~4.3 gwei), making this cheap to exploit. No special role, key, or governance access is required. [9](#0-8) 

---

### Recommendation

Add an upper bound on `callbackGasLimit` in `requestPriceUpdatesWithCallback`, analogous to how Entropy enforces `MAX_GAS_LIMIT` on its `requestV2` path. A reasonable cap would be the expected maximum block gas limit for the target chain (e.g., 30,000,000 for Ethereum mainnet). Additionally, implement a timeout-based refund mechanism so users can reclaim their funds if a request is not fulfilled within a bounded window.

```solidity
uint32 public constant MAX_CALLBACK_GAS_LIMIT = 30_000_000;

require(callbackGasLimit <= MAX_CALLBACK_GAS_LIMIT, "Gas limit too high");
```

---

### Proof of Concept

1. Provider registers with `feePerGasInWei = 1 wei`.
2. Attacker calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = type(uint32).max` and pays the minimal fee (`baseFee + providerBaseFee + providerFeedFee + 4294967295 wei`).
3. The request is stored with `req.callbackGasLimit = 4294967295`.
4. Any attempt to call `executeCallback` for this sequence number requires forwarding `4,294,967,295` gas to the inner callback call — exceeding the Ethereum block gas limit — causing the outer transaction to run out of gas and revert.
5. The user's ETH (provider fee portion) is permanently locked in the contract with no recovery path. [10](#0-9) [2](#0-1)

```text
    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```
