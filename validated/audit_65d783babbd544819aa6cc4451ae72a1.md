Audit Report

## Title
Unbounded Descendant BFS in `remove_entry` Holds tx-pool Write Lock — (`tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry` unconditionally calls `update_descendants_index_key`, which performs an unbounded BFS over all descendants and issues a `modify_by_id` mutation for each one while the exclusive tx-pool write lock is held. The `remove_committed_tx` path does not pre-remove links before calling `remove_entry`, unlike `remove_entry_and_descendants`, so the full live descendant graph is traversed on every block commit. Because no `max_descendants_count` limit exists anywhere in the codebase, an unprivileged submitter can construct a root transaction with an arbitrarily wide set of children, causing O(N) index mutations under the write lock when the root is committed.

## Finding Description
`remove_entry` at `pool_map.rs:243` unconditionally calls `update_descendants_index_key`. That function (`pool_map.rs:447–460`) calls `self.links.calc_descendants`, which delegates to `calc_relative_ids` → `calc_relation_ids` (`links.rs:52–72`), an unbounded BFS following `children` links until the graph is exhausted. For every entry in the resulting set, `entries.modify_by_id` is called, mutating the multi-index map.

`remove_entry_and_descendants` (`pool_map.rs:252–265`) avoids this cost by pre-removing all links before iterating, as its comment explicitly states: *"update links state for remove, so that we won't update_descendants_index_key in remove_entry."* `remove_committed_tx` (`pool.rs:253–268`) has no equivalent protection — it calls `pool_map.remove_entry` directly at line 255 without pre-removing links.

`remove_committed_tx` is called from `remove_committed_txs` (`pool.rs:223–241`), which is called from `_update_tx_pool_for_reorg` (`process.rs:1039–1056`), which runs entirely inside the exclusive write lock acquired at `process.rs:836`. All concurrent tx-pool operations (submission, RPC, block assembly) are blocked for the duration.

The ancestor guard (`pool_map.rs:595–601`) checks only `ancestors.len() + 1 <= max_ancestors_count`. A root transaction T0 with K outputs can have K children T1…TK, each with exactly one ancestor (T0), so each passes the check individually. A grep search for `max_descendants_count` returns zero matches across the entire codebase — no descendant fan-out limit exists at admission time.

## Impact Explanation
While the write lock is held, all concurrent tx-pool operations stall: new transaction submissions, RPC queries, block assembly (`get_block_template`), and orphan resolution. With a default pool size of tens of megabytes and minimum transaction sizes of ~100–200 bytes, N can reach tens of thousands. Repeated across multiple crafted root transactions committed in successive blocks, this produces sustained write-lock stalls. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, as the attacker's fee cost scales with N but the lock stall duration also scales with N, and the attack is repeatable across blocks.

## Likelihood Explanation
The attacker is an unprivileged tx-pool submitter. No miner cooperation is required — any miner committing T0 in a normal block triggers the traversal. The attacker pays fees proportional to N, but the attack is repeatable: after T0 is committed, a new root with a fresh descendant fan-out can be submitted immediately. The RBF path is separately guarded by `MAX_REPLACEMENT_CANDIDATES = 100`, but the block-commit path has no equivalent descendant count check.

## Recommendation
1. **Apply the link-pre-removal pattern from `remove_entry_and_descendants` to `remove_committed_tx`.** Before calling `pool_map.remove_entry` for a committed transaction, call `remove_entry_links` on it so that `update_descendants_index_key` returns an empty set. Descendants remain in the pool with their ancestor-weight state corrected via a separate bounded pass.
2. **Alternatively, enforce a `max_descendants_count` at admission time**, symmetric to `max_ancestors_count`, rejecting any transaction whose admission would cause an existing pool entry to exceed the descendant limit.
3. **Decouple descendant weight updates from the hot removal path**, batching or deferring `modify_by_id` calls so they do not block the write lock.

## Proof of Concept
1. Submit root transaction T0 with 500 outputs to the tx-pool.
2. Submit T1, T2, …, T500, each spending one distinct output of T0. Each Ti has exactly 1 ancestor (T0), so all pass `max_ancestors_count`.
3. Wait for any miner to commit T0 in a block (normal operation; no miner cooperation required).
4. When the node processes the block:
   - `process.rs:836`: tx-pool write lock acquired.
   - `process.rs:1055`: `remove_committed_txs` called.
   - `pool.rs:255`: `pool_map.remove_entry(T0)` called.
   - `pool_map.rs:243`: `update_descendants_index_key(T0, Remove)` called.
   - `links.rs:59–71`: BFS returns {T1, …, T500}.
   - `pool_map.rs:452`: `entries.modify_by_id` called 500 times under the write lock.
5. Scale to N=10,000 descendants (achievable within a 20 MB pool with ~200-byte transactions) across multiple root transactions committed in successive blocks to produce sustained stalls.
