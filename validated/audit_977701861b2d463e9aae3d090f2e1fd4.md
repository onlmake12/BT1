### Title
Unbounded Storage-Scan Loop in `executeCallback` Can Exceed Gas Limit, Permanently Locking Funds - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that performs a cold storage lookup (`SLOAD`) for every fulfilled sequence number between `_state.firstUnfulfilledSeq` and the next active request. An attacker who creates many requests and fulfills all but the earliest one can force this loop to iterate over thousands of fulfilled slots, causing the transaction to exceed the block gas limit and permanently locking the unfulfilled requester's funds.

---

### Finding Description

Inside `executeCallback`, after clearing the just-fulfilled request, the contract advances `_state.firstUnfulfilledSeq` with an unbounded `while` loop:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which first reads `_state.requests[shortKey]` (a fixed 32-slot array) and, when the sequence number does not match the slot's current occupant, reads `_state.requestsOverflow[key]` — a mapping keyed by `keccak256(sequenceNumber)`. [2](#0-1) 

Because every fulfilled overflow entry has a unique mapping key, each such lookup is a **cold SLOAD** costing 2,100 gas. With `NUM_REQUESTS = 32` slots in the main array, any system with more than 32 concurrent requests will push older requests into the overflow mapping. [3](#0-2) 

The `clearRequest` function deletes the overflow entry but does not advance `firstUnfulfilledSeq`; that is left entirely to the while loop in `executeCallback`. [4](#0-3) 

The code's own TODO comment explicitly acknowledges this design flaw and suggests a doubly-linked list as the correct fix.

---

### Impact Explanation

If `executeCallback` reverts due to out-of-gas, the entire transaction is rolled back. The `clearRequest` call is also reverted, so the target request remains active but can never be fulfilled — every subsequent attempt hits the same unbounded loop. The requester's fee (paid at request time) is permanently locked in the contract with no refund path. [5](#0-4) 

---

### Likelihood Explanation

The attack requires only the ability to call `requestPriceUpdatesWithCallback` (unprivileged, fee-paying) and `executeCallback` (unprivileged). The attacker:

1. Submits N requests, paying the required fee each time.
2. Fulfills requests 2 through N (leaving sequence number 1 unfulfilled). `firstUnfulfilledSeq` stays at 1.
3. When anyone attempts to fulfill request 1, the while loop must scan N−1 fulfilled slots.

At 2,100 gas per cold overflow SLOAD, ~5,000 overflow entries consume ≈10.5 M gas in the loop alone — within reach of typical EVM block gas limits (15–30 M gas). The gas benchmark test file already documents the out-of-order fulfillment scenario and notes the linear gas scaling: [6](#0-5) 

Even without a deliberate attacker, a naturally busy system where the first request is slow to fulfill (e.g., the provider is offline) while thousands of later requests are fulfilled normally will trigger the same condition.

---

### Recommendation

Replace the linear scan with a doubly-linked list of active sequence numbers (as the TODO comment already suggests), so that `firstUnfulfilledSeq` can be advanced in O(1) without touching fulfilled slots. Alternatively, cap the number of iterations the while loop may perform per transaction and require a separate `advanceFirstUnfulfilledSeq` call to progress the pointer in batches.

---

### Proof of Concept

```solidity
// In a Foundry test:
// 1. Create N = 6000 requests (attacker pays fees)
uint256 N = 6000;
for (uint256 i = 0; i < N; i++) {
    vm.prank(attacker);
    echo.requestPriceUpdatesWithCallback{value: fee}(
        provider, block.timestamp, priceIds, callbackGasLimit
    );
}
// sequenceNumbers: 1 .. 6000, firstUnfulfilledSeq = 1

// 2. Fulfill requests 2 .. 6000 (skip seq 1)
for (uint64 seq = 2; seq <= N; seq++) {
    echo.executeCallback(provider, seq, updateData, priceIds);
}
// firstUnfulfilledSeq is still 1

// 3. Attempt to fulfill seq 1 — while loop must scan 5999 fulfilled slots
uint256 gasBefore = gasleft();
echo.executeCallback(provider, 1, updateData, priceIds);
// Expect revert OOG, or gas consumed >> block gas limit
assert(gasBefore - gasleft() > 15_000_000);
```

The `testMultipleRequestsOutOfOrderFulfillment` benchmark in the test suite already demonstrates that the last fulfillment in a reverse-order sequence is the most expensive, confirming the linear scaling of the while loop. [7](#0-6)

    function findRequest(
        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];
        }
    }

    function clearRequest(uint64 sequenceNumber) internal {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        Request storage req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            req.sequenceNumber = 0;
        } else {
            delete _state.requestsOverflow[key];
        }
    }
