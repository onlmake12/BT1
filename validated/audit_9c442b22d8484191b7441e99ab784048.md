Audit Report

## Title
Unbounded O(N) Pool Scan on Every Block in `_update_tx_pool_for_reorg` — (File: tx-pool/src/process.rs)

## Summary
`_update_tx_pool_for_reorg` performs two full linear scans over all `Gap` and `Pending` pool entries on every block when `mine_mode` is active, with no per-invocation element count guard. An attacker who fills the pool with minimum-fee transactions forces these scans to dominate the block-processing critical path, holding the tx-pool write lock for an extended period on every block and degrading mining throughput and RPC responsiveness.

## Finding Description
In `tx-pool/src/process.rs` lines 1065–1080, `_update_tx_pool_for_reorg` executes two unbounded loops when `mine_mode` is active:

```rust
for entry in tx_pool.pool_map.entries.get_by_status(&Status::Gap) { ... }
for entry in tx_pool.pool_map.entries.get_by_status(&Status::Pending) { ... }
```

Neither loop has an early-exit condition or a per-call element limit. The pool is bounded only by `DEFAULT_MAX_TX_POOL_SIZE` (180,000,000 bytes, `util/app-config/src/legacy/tx_pool.rs` line 20) — a byte limit, not an entry count limit. With a minimum CKB transaction size of ~200 bytes, the pool can hold ~900,000 entries. Every block triggers both full scans while the tx-pool write lock is held, blocking all concurrent RPC calls and relay handlers.

A secondary surface exists in `tx-pool/src/component/links.rs` lines 52–72: `calc_relation_ids` performs an unbounded BFS over the ancestor/descendant graph. It is called by `update_ancestors_index_key` and `update_descendants_index_key` in `pool_map.rs` lines 432–460 on every entry add or remove. While `DEFAULT_MAX_ANCESTORS_COUNT` (1,000, line 16) caps ancestor chain depth at insertion time, it does not cap descendant sub-tree width. A transaction with many outputs can have many direct children, producing a wide descendant graph bounded only by pool capacity, and `calc_relation_ids` has no count or depth limit of its own.

Existing guards are insufficient: `max_tx_pool_size` limits total bytes but not entry count; `max_ancestors_count` applies only at insertion time for ancestor depth, not to the BFS traversal width or to the per-block reorg scan.

## Impact Explanation
This matches the allowed CKB bounty impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The tx-pool write lock is held for the duration of both full scans on every block. Mining nodes experience degraded block assembly latency and increased RPC response times for all pool-touching calls. If a mining node falls behind tip due to sustained lock contention, it produces stale blocks, reducing effective hash rate and degrading network block production. The attack is cheap and repeatable on every ~10 s block interval.

## Likelihood Explanation
The attack requires only the ability to submit transactions via the public `send_transaction` RPC or the P2P relay protocol — both available to any unprivileged peer. Filling the pool to 180 MB at the default minimum fee rate of 1,000 shannons/KB costs approximately 1.8 CKB. No special privilege, key material, or majority hashpower is required. `mine_mode` is active on any node running a block assembler, which includes all mining nodes. The attack is repeatable: once the pool is filled, each subsequent block triggers the full scan until the attacker's transactions expire or are evicted.

## Recommendation
1. **Cap the per-block promotion scan**: in `_update_tx_pool_for_reorg`, introduce a configurable `MAX_ENTRIES_PER_REORG_SCAN` constant and break out of the `Gap` and `Pending` loops once that limit is reached, deferring remaining entries to the next block.
2. **Bound descendant/ancestor traversal**: in `calc_relation_ids` (`tx-pool/src/component/links.rs`, lines 52–72), accept a `max_count` parameter and return a truncated result or error when the limit is exceeded, mirroring the existing `max_ancestors_count` guard applied at insertion time.
3. **Enforce a maximum entry count** (not just a byte-size limit) in the pool so that the number of iterations is predictable regardless of transaction size distribution.

## Proof of Concept
1. Connect to a CKB node with a block assembler configured (`mine_mode` active).
2. Submit approximately 900,000 minimum-size, independent transactions via `send_transaction` until `total_tx_size` approaches `max_tx_pool_size` (180 MB).
3. Trigger block arrival (or wait for the next block on mainnet/testnet).
4. Observe via node metrics or logs that `_update_tx_pool_for_reorg` iterates over all ~900,000 entries, holding the tx-pool write lock for an extended period.
5. Confirm that RPC latency for pool-touching calls (`get_transaction`, `send_transaction`) spikes on every block arrival and that block assembly latency increases proportionally to pool size.

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
    }
