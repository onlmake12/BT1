Audit Report

## Title
Unbounded Per-Peer MMR Proof Computation in LightClient `GetBlocksProof` Handler — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary
The `LightClientProtocol` handler for `GetBlocksProof` performs up to ~3000 synchronous RocksDB reads and a full MMR proof generation over up to 1000 leaf positions per request, with no per-peer rate limiting, no concurrency cap, and no request queue depth limit. Any unprivileged peer with a TCP connection can drive this path repeatedly and in parallel, saturating RocksDB I/O shared with block import and sync, and growing the async task queue unboundedly.

## Finding Description
`LightClientProtocol::received` dispatches directly to `GetBlocksProofProcess::execute` with no rate-limiting gate. [1](#0-0) 

The only guards in `execute` are: empty hash list, duplicate hashes, and a size check against `GET_BLOCKS_PROOF_LIMIT = 1000`. [2](#0-1) [3](#0-2) 

After those checks, the handler iterates all found hashes performing three separate DB reads per hash (`get_block_header`, `get_block_uncles`, `get_block_extension`): [4](#0-3) 

Then `reply_proof` is called, which constructs a `chain_root_mmr` and calls `mmr.gen_proof(positions)` over all positions. Each `get_elem` call in the MMR store is a live RocksDB read: [5](#0-4) [6](#0-5) 

The `LightClientProtocol` struct holds only `shared: Shared` — no per-peer state, no request counter, no semaphore: [7](#0-6) 

The `connected` handler does nothing but log: [8](#0-7) 

A grep across the entire `util/light-client-protocol-server` directory for `rate_limit`, `per_peer`, `peer_limit`, `request_limit`, `flood`, `throttle`, `semaphore` returns zero matches. The network layer has `max_inbound`/`max_outbound` connection limits but no per-protocol, per-peer message rate limiting.

The launcher registers `LightClientProtocol` as a plain `CKBProtocol` with no additional middleware: [9](#0-8) 

`BAD_MESSAGE_BAN_TIME` only triggers on structurally malformed messages. A well-formed request with exactly 1000 valid main-chain hashes is never penalized: [10](#0-9) 

The same unbounded pattern applies identically to `GetTransactionsProof` and `GetLastStateProof`, all sharing the same `GET_*_LIMIT = 1000` constants and the same `reply_proof` path with no rate limiting. [11](#0-10) 

## Impact Explanation
Each max-size `GetBlocksProof` request causes approximately 3000 synchronous RocksDB point reads (header + uncles + extension per hash) plus one `chain_root_mmr` construction and `gen_proof` call over 1000 positions, requiring O(1000 × log N) additional MMR node reads from RocksDB. With N concurrent peers each sending requests in a tight loop, the node experiences RocksDB read I/O saturation shared across chain processing, tx-pool, and sync; async task queue depth growth delaying block relay and sync responses; and potential starvation of the chain's critical path (block import, compact block relay). This matches the **High** impact class: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
LightClient is opt-in (`SupportProtocol::LightClient` in config), but when enabled it is reachable by any peer without authentication. The attack requires no PoW, no stake, no key material — only a P2P connection. The `last_hash` needed is trivially obtained from any public node or block explorer. The cost to the attacker is negligible (sending a ~32 KB message per request); the cost to the server is thousands of DB reads and an MMR proof computation. Well-formed max-size requests are never penalized, so the attacker is never banned.

## Recommendation
1. **Per-peer request rate limiting:** Track the last request timestamp per peer in `LightClientProtocol` and enforce a minimum interval (e.g., 1 request per second per peer) for expensive message types (`GetBlocksProof`, `GetTransactionsProof`, `GetLastStateProof`).
2. **Global concurrency cap:** Use a `tokio::sync::Semaphore` to bound the number of simultaneously executing proof-generation tasks.
3. **Reduce the limit or add cost scaling:** Lower `GET_BLOCKS_PROOF_LIMIT` or charge a sliding cost (e.g., ban peers that send max-size requests more than K times per minute).
4. **Peer scoring:** Penalize peers that repeatedly send large requests without consuming the responses.

## Proof of Concept
1. Enable LightClient protocol on a CKB full node (`SupportProtocol::LightClient` in config).
2. Obtain the current tip hash via any public RPC or peer.
3. From 50 concurrent TCP connections, each send in a tight loop:
   ```
   GetBlocksProof {
     last_hash: <tip_hash>,
     block_hashes: [h1, h2, ..., h1000]  // 1000 valid main-chain hashes
   }
   ```
4. Observe on the server:
   - RocksDB read IOPS spike to saturation
   - Async task queue depth grows unboundedly
   - Block relay latency increases; sync stalls
   - No peer is banned (all messages are well-formed and within the 1000-hash limit)

        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
