### Title
Missing Zero-Address Check in `PythGovernance.withdrawFee` Allows Permanent Loss of Protocol Fees - (File: `target_chains/ethereum/contracts/contracts/pyth/PythGovernance.sol`)

### Summary
`PythGovernance.sol`'s `withdrawFee` function transfers accumulated ETH fees to `payload.targetAddress` without validating that the address is non-zero. If a governance VAA is crafted or mistakenly signed with `targetAddress = address(0)`, the entire fee amount is permanently burned. The sibling contract `EntropyGovernance.sol` already has this check, confirming the Pyth team recognizes the risk — but it was not applied consistently.

### Finding Description
`PythGovernance.withdrawFee` is invoked via `executeGovernanceInstruction` when a valid Wormhole VAA with action `WithdrawFee` (action 9) is submitted. The payload is parsed by `parseWithdrawFeePayload`, which extracts `targetAddress` from the raw bytes with no validation:

```solidity
wf.targetAddress = address(encodedPayload.toAddress(index));
```

The `withdrawFee` function then immediately transfers ETH to that address:

```solidity
function withdrawFee(WithdrawFeePayload memory payload) internal {
    if (payload.fee > address(this).balance)
        revert PythErrors.InsufficientFee();

    (bool success, ) = payload.targetAddress.call{value: payload.fee}("");
    require(success, "Failed to withdraw fees");

    emit FeeWithdrawn(payload.targetAddress, payload.fee);
}
```

There is no `require(payload.targetAddress != address(0))` guard anywhere in the call chain.

By contrast, `EntropyGovernance.withdrawFee` explicitly guards against this:

```solidity
function withdrawFee(address targetAddress, uint128 amount) external {
    require(targetAddress != address(0), "targetAddress is zero address");
    ...
}
```

The same gap exists in the Stylus implementation (`target_chains/stylus/contracts/pyth-receiver/src/governance.rs`), where `withdraw_fee` calls `self.vm().transfer_eth(target_address, fee_to_withdraw)` with no zero-address check on `target_address`.

### Impact Explanation
If a governance VAA encoding `targetAddress = 0x0000...0000` is submitted to `executeGovernanceInstruction`, the full `fee` amount of ETH is sent to the zero address via a low-level `.call`. On EVM chains, a `.call` to `address(0)` succeeds (returns `true`) and the ETH is permanently destroyed. The `accruedPythFees` balance is not tracked separately in `PythGovernance` — the contract's entire ETH balance up to `payload.fee` is at risk. This constitutes a direct, irreversible loss of protocol funds.

### Likelihood Explanation
The governance VAA must be signed by the Wormhole guardian set. However:
- The governance tooling (`WithdrawFee.ts`) encodes `targetAddress` as a raw 20-byte hex buffer with no zero-address validation before encoding.
- A scripting or encoding mistake (e.g., uninitialized buffer, wrong offset) could produce an all-zero `targetAddress` in the VAA payload.
- Once the VAA is signed and submitted, the contract has no on-chain safety net to reject it.
- The `EntropyGovernance` contract already has this guard, meaning the Pyth team has acknowledged the risk in one context but not applied it uniformly.

Likelihood is low but non-negligible given the operational complexity of multi-chain governance.

### Recommendation
Add a zero-address guard in `PythGovernance.withdrawFee`:

```solidity
function withdrawFee(WithdrawFeePayload memory payload) internal {
    require(payload.targetAddress != address(0), "targetAddress is zero address");
    if (payload.fee > address(this).balance)
        revert PythErrors.InsufficientFee();
    (bool success, ) = payload.targetAddress.call{value: payload.fee}("");
    require(success, "Failed to withdraw fees");
    emit FeeWithdrawn(payload.targetAddress, payload.fee);
}
```

Apply the same fix to the Stylus `withdraw_fee` in `governance.rs`. Optionally, add the check in `parseWithdrawFeePayload` as a defense-in-depth measure.

### Proof of Concept

1. Governance tooling constructs a `WithdrawFee` VAA with `targetAddress = bytes(20)` (all zeros) — e.g., due to an uninitialized buffer in the off-chain script.
2. The VAA is signed by the Wormhole guardian set and submitted to `executeGovernanceInstruction`.
3. `parseWithdrawFeePayload` decodes `targetAddress = address(0)` with no error.
4. `withdrawFee` checks only `payload.fee > address(this).balance` — passes.
5. `address(0).call{value: payload.fee}("")` executes, returns `(true, "")`.
6. `require(success, ...)` passes.
7. `payload.fee` wei is permanently destroyed; `FeeWithdrawn(address(0), fee)` is emitted.

```text
    /// @dev Parse a WithdrawFeePayload (action 9) with minimal validation
    function parseWithdrawFeePayload(
        bytes memory encodedPayload
    ) public pure returns (WithdrawFeePayload memory wf) {
        uint index = 0;

        wf.targetAddress = address(encodedPayload.toAddress(index));
        index += 20;

        uint64 val = encodedPayload.toUint64(index);
        index += 8;

        uint64 expo = encodedPayload.toUint64(index);
        index += 8;

        wf.fee = uint256(val) * uint256(10) ** uint256(expo);

        if (encodedPayload.length != index)
            revert PythErrors.InvalidGovernanceMessage();
    }
```

```rust
    fn withdraw_fee(
        &mut self,
        value: u64,
        expo: u64,
        target_address: Address,
    ) -> Result<(), PythReceiverError> {
        let fee_to_withdraw =
            U256::from(value).saturating_mul(U256::from(10).pow(U256::from(expo)));
        let current_balance = self.vm().balance(self.vm().contract_address());

        if current_balance < fee_to_withdraw {
            return Err(PythReceiverError::InsufficientFee);
        }

        self.vm()
            .transfer_eth(target_address, fee_to_withdraw)
            .map_err(|_| PythReceiverError::InsufficientFee)?;

        log(
            self.vm(),
            crate::FeeWithdrawn {
                target_address,
                fee_amount: fee_to_withdraw,
            },
        );

        Ok(())
    }
```
