Audit Report

## Title
Copy-Paste Bug in `script_exists_in_output` Reads `row_lock` Instead of `row_type` on PostgreSQL, Causing Incorrect Script Deletion During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary

At line 252 of `script_exists_in_output`, the second `match` block reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, where `EXISTS` returns `BOOLEAN`, this always resolves to `false` (since `row_lock` is already known to be `false` at that point), completely bypassing the type-script existence check. During `rollback_block`, this causes any script referenced only as a `type_script_id` in surviving outputs to be incorrectly deleted from the `script` table, permanently corrupting the rich-indexer's state for those cells on PostgreSQL deployments.

## Finding Description

`script_exists_in_output` issues two SQL `EXISTS` queries: one against `lock_script_id` (result stored in `row_lock`) and one against `type_script_id` (result stored in `row_type`). The first match block (lines 222–235) correctly short-circuits with `Ok(true)` if `row_lock` is true. Execution only reaches line 252 when `row_lock` is `false`.

The bug is at line 252:

```rust
// line 252 — BUG: reads row_lock again, not row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

On **PostgreSQL**, `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL `EXISTS` returns `BOOLEAN`), and since `row_lock` is already `false` at this point (the early return at line 226 would have fired otherwise), this always returns `Ok(false)` — `row_type` is never consulted.

On **SQLite**, `row_lock.try_get::<bool, _>(0)` fails (SQLite `EXISTS` returns `BIGINT`), so the `Err(_)` branch executes `Ok(row_type.get::<i64, _>(0) == 1)`, which correctly reads `row_type`. SQLite is unaffected.

The caller in `rollback_block` (lines 33–37) uses the return value to decide whether to add `type_script_id` to the deletion list: [2](#0-1) 

On PostgreSQL, `script_exists_in_output(type_script_id)` always returns `false` when the script is not also a `lock_script_id` in any surviving output, so the script row is unconditionally added to `script_id_list_to_remove` and deleted from the `script` table even when surviving outputs still reference it via `type_script_id`. [3](#0-2) 

## Impact Explanation

This is a **Medium** severity finding: **Suboptimal implementation of CKB state storage mechanism (2001–10000 points)**. The rich-indexer is CKB's indexed state storage layer. After the incorrect deletion, the `script` table permanently loses the row for the shared type-script. Any subsequent `get_cells` or `get_transactions` RPC call joining against `script` via `type_script_id` will return NULL joins or missing script data for all surviving cells from prior blocks that reference that type-script. The rich-indexer's view of those cells is permanently broken until a full re-sync from scratch. This does not affect consensus or the core node, but it permanently corrupts the indexed state on all PostgreSQL-backed rich-indexer deployments that experience chain reorganizations.

## Likelihood Explanation

Chain reorganizations are a normal part of CKB network operation and require no special privileges to trigger — any peer can relay a valid competing chain. The only structural requirement is that the rolled-back block contains an output whose `type_script_id` is shared with a surviving output in a prior block but whose `lock_script_id` is distinct. This is a routine scenario (e.g., UDT token cells with a shared type-script but different owner lock-scripts across blocks). Any PostgreSQL-backed rich-indexer deployment that experiences a reorg under these conditions will be silently corrupted.

## Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // was: row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

## Proof of Concept

1. Run a PostgreSQL-backed rich-indexer node.
2. Append block 0 containing output A with `type_script = T`, `lock_script = L1`.
3. Append block 1 containing output B with `type_script = T`, `lock_script = L2` (same T, different lock).
4. Trigger a reorg that rolls back block 1 (`rollback_block` executes).
5. After rollback, query the `script` table: the row for T is gone (incorrectly deleted).
6. Call `get_cells` with `script = T`: returns empty or broken results despite output A from block 0 surviving.
7. Expected: T's script row is preserved; `get_cells` returns output A correctly.
