Audit Report

## Title
`declared_cycles` Used as Script Execution Cap Causes Permanent Relay Suppression via Malformed-Tx Classification - (File: tx-pool/src/process.rs)

## Summary
In `tx-pool/src/process.rs`, `_process_tx` uses the peer-supplied `declared_cycles` as the hard execution-cycle cap passed to `verify_rtx`. When `declared_cycles` is below the transaction's actual cycle cost, the script verifier terminates with `ScriptError::ExceededMaximumCycles`, which is classified as a malformed-transaction error. This causes `is_allowed_relay` to return `false`, so `TxVerificationResult::Reject` is never sent to the relayer, and the transaction hash — already marked as known before pool submission — is never removed from the known-tx filter. An unprivileged peer can permanently suppress relay of a valid transaction on a target node for the filter's TTL by first announcing the hash and then relaying it with `declared_cycles = 1`.

## Finding Description

**Root cause — `tx-pool/src/process.rs`, `_process_tx` (line 720):**

`declared_cycles` is used as the execution cap passed to `verify_rtx`:

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
// ...
let verified_ret = verify_rtx(..., max_cycles, command_rx).await;
```

`verify_rtx` passes `max_cycles` directly to `ContextualTransactionVerifier::verify` or `verify_with_pause` as a hard cap (`tx-pool/src/util.rs` lines 108, 119).

When `declared_cycles < actual_cycles`, the VM hits the cap and returns `VMInternalError::CyclesExceeded`, which is mapped to `ScriptError::ExceededMaximumCycles` (`script/src/verify.rs` line 568). This surfaces as `Reject::Verification(ErrorKind::Script)`.

`is_malformed_from_verification` classifies all `ErrorKind::Script` errors as malformed unless the error string contains `ARGV_TOO_LONG_TEXT` (`util/types/src/core/tx_pool.rs` line 75). `ExceededMaximumCycles` does not contain that string, so `is_malformed_tx()` returns `true`.

`is_allowed_relay` returns `false` for this case — only `DeclaredWrongCycles` gets the explicit carve-out (`util/types/src/core/tx_pool.rs` lines 110–113).

In `after_process`, when `is_allowed_relay()` is false, `send_result_to_relayer(TxVerificationResult::Reject)` is never called (`tx-pool/src/process.rs` lines 517–521).

The tx hash was already marked as known **before** pool submission in `TransactionsProcess::execute` (`sync/src/relayer/transactions_process.rs` line 76), before the async spawn that calls `submit_remote_tx`.

`remove_from_known_txs` is only triggered by `TxVerificationResult::Reject`, which is never sent (`sync/src/relayer/mod.rs` lines 673–675). The hash remains in the known-tx filter for its full TTL.

**Asymmetry with `_test_accept_tx`:** That function correctly uses `self.consensus.max_block_cycles()` as the cap, never `declared_cycles` (`tx-pool/src/process.rs` line 787).

**Contrast with `declared > actual` path:** When `declared_cycles > actual_cycles`, the script completes, `verified.cycles` is produced, the exact-match check fires, and `Reject::DeclaredWrongCycles` is returned. `is_allowed_relay` returns `true` for `DeclaredWrongCycles`, so `TxVerificationResult::Reject` is sent and the hash is removed from the filter. The `DeclaredWrongCyclesAndRelayAgain` integration test confirms this recovery path works correctly (`test/src/specs/tx_pool/declared_wrong_cycles.rs` lines 69–117).

**No lower-bound guard:** The only existing check in `TransactionsProcess::execute` rejects `declared_cycles > max_block_cycles`. There is no check preventing `declared_cycles = 1` (`sync/src/relayer/transactions_process.rs` lines 63–74).

**Prerequisite:** The filter at `transactions_process.rs` lines 49–55 requires the tx was previously requested from the relaying peer. The attacker must first send `RelayTransactionHashes` to cause the node to issue a `GetRelayTransactions` request, then respond with the tx and `declared_cycles = 1`. This is a single additional message and does not materially raise the attack cost.

## Impact Explanation

An unprivileged peer can suppress relay of any valid transaction on a target node for the duration of the known-tx filter's TTL. The transaction is marked as known, rejected as malformed (script cycles exceeded), and never re-requestable from other peers. If the attacker targets multiple nodes simultaneously using multiple IPs or Sybil connections, valid transactions can be prevented from propagating across a significant portion of the network, delaying or blocking confirmation. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation

The attack requires only two standard P2P relay messages: `RelayTransactionHashes` (to get the node to request the tx) and `RelayTransactions` with `cycles = 1`. No privileged access, no key material, and no majority hash power is required. The target transaction hash is observable from the public mempool or pending proposals. The only existing guard checks for `declared_cycles > max_block_cycles`; there is no lower-bound check. The attacker is banned after the attempt but the suppression persists. The attack is trivially repeatable across multiple nodes using multiple IP addresses.

## Recommendation

Use `max_block_cycles` (or `max_tx_verify_cycles`) as the execution cap in `_process_tx`, independent of `declared_cycles`. Reserve `declared_cycles` solely for the post-verification exact-match check, matching the behavior of `_test_accept_tx`:

```rust
// Use consensus limit for execution, not the peer-declared value
let max_cycles = self.consensus.max_block_cycles();

let verified_ret = verify_rtx(..., max_cycles, command_rx).await;
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Only now compare against declared_cycles
if let Some(declared) = declared_cycles
    && declared != verified.cycles
{
    return Some((Err(Reject::DeclaredWrongCycles(declared, verified.cycles)), snapshot));
}
```

This ensures that a peer declaring too-low cycles receives `DeclaredWrongCycles` (re-relay allowed, hash removed from filter) rather than a script error (re-relay suppressed, hash permanently stuck in filter).

## Proof of Concept

1. Node A has `max_block_cycles = 3_500_000_000`.
2. A valid transaction `T` uses the `always_success` script requiring `537` cycles.
3. Attacker peer connects to Node A and sends `RelayTransactionHashes` containing `T`'s hash.
4. Node A adds `T` to `unknown_tx_hashes` and sends `GetRelayTransactions` to the attacker.
5. Attacker sends `RelayTransactions` with `T` and `cycles = 1`.
6. `TransactionsProcess::execute`: `1 <= max_block_cycles` → no ban; tx hash marked as known; `submit_remote_tx(T, declared_cycles=1, peer)` called.
7. `_process_tx`: `max_cycles = 1`; `verify_rtx` runs script with cap `1`; script aborts with `ExceededMaximumCycles(1)` → `Reject::Verification(ErrorKind::Script)`.
8. `after_process`: `is_malformed_tx() = true` → attacker peer banned; `is_allowed_relay() = false` → `TxVerificationResult::Reject` NOT sent.
9. `T`'s hash remains in Node A's known-tx filter; Node A will not request `T` from any other peer for the filter's TTL.
10. Legitimate peers relaying `T` with correct cycles `537` are ignored by Node A (tx already "known").

To reproduce: adapt the existing `DeclaredWrongCyclesAndRelayAgain` integration test (`test/src/specs/tx_pool/declared_wrong_cycles.rs` lines 69–117) by changing the attacker's declared cycles from `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1` to `1`. The test should show that after the attacker is banned, Node A still does **not** accept the transaction from Node 1 with correct cycles — confirming the hash is stuck in the filter and `remove_from_known_txs` was never called.
