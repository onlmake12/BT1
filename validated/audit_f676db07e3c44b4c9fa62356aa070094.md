### Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Capacity - (File: `util/dao/src/lib.rs`)

### Summary
`DaoCalculator::calculate_maximum_withdraw` computes the scaled withdrawal capacity as a `u128` intermediate value and then silently casts it to `u64` with `as u64`. If the intermediate value exceeds `u64::MAX`, the cast silently wraps/truncates, producing a drastically incorrect (too-small) withdrawal capacity. Every other analogous u128→u64 conversion in the same file uses the checked `u64::try_from(…).map_err(|_| DaoError::Overflow)?` pattern, making this a clear inconsistency.

### Finding Description
In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a Rust truncating (wrapping) cast. If `withdraw_counted_capacity > u64::MAX`, the high bits are silently discarded, yielding a value that can be arbitrarily smaller than the true result.

Every other u128→u64 narrowing in the same file uses the safe pattern:

```rust
// dao_field_with_current_epoch, line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)

// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;

// secondary_block_reward, line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The analog to the report is direct: the report assigns a computed amount that exceeds the available balance instead of capping at the balance. Here, a computed capacity that exceeds `u64::MAX` is silently truncated instead of being rejected, producing an incorrect final amount.

### Impact Explanation
`calculate_maximum_withdraw` is called from two reachable paths:

1. **Tx-pool admission** – `check_tx_fee` → `DaoCalculator::transaction_fee` → `transaction_maximum_withdraw` → `calculate_maximum_withdraw`. If the truncated value is smaller than the transaction's `outputs_capacity`, `safe_sub` returns an underflow error and the withdrawal transaction is permanently rejected, locking the depositor's funds. [5](#0-4) 

2. **RPC `calculate_dao_maximum_withdraw`** – returns a silently wrong (too-small) capacity to the caller, causing off-chain tooling to construct invalid or under-valued withdrawal transactions. [6](#0-5) 

### Likelihood Explanation
The overflow condition requires `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`. Since `counted_capacity ≤ u64::MAX` and the accumulate rate ratio `withdrawing_ar / deposit_ar` grows very slowly (fractions of a percent per year under normal issuance), reaching a ratio large enough to overflow u64 would take an extremely long time under mainnet parameters. The likelihood is therefore very low in practice, though the defect is real and the inconsistency with every other u128→u64 conversion in the same file is clear.

### Recommendation
Replace the silent cast with the same checked conversion used everywhere else in the file:

```rust
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
``` [7](#0-6) 

### Proof of Concept
Construct a DAO deposit cell with `counted_capacity` near `u64::MAX` (e.g., `u64::MAX - occupied_capacity`) and reference a withdrawing header whose `withdrawing_ar / deposit_ar` ratio is `> 1`. The product `counted_capacity * withdrawing_ar / deposit_ar` exceeds `u64::MAX`. The `as u64` cast silently wraps the result to a small value. `calculate_maximum_withdraw` returns `small_value + occupied_capacity` instead of the correct (larger) amount. When the withdrawal transaction sets `outputs_capacity` to the correct entitled amount, `transaction_fee` calls `maximum_withdraw.safe_sub(outputs_capacity)`, which underflows and rejects the transaction with a `DaoError`. [8](#0-7)

    /// Calculate maximum withdraw capacity of a deposited dao output
    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }
