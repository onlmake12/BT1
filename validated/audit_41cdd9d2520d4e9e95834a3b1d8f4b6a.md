### Title
Zero-Amount `withdraw()` Passes All Checks, Emits Misleading Events Without State Change — (`Entropy.sol`)

---

### Summary

The `withdraw(uint128 amount)` function in `Entropy.sol` lacks a zero-amount guard. Any address — including completely unregistered providers — can call `withdraw(0)`, which passes every check, performs no state change, but emits two `Withdrawal` events with `withdrawnAmount = 0`. This is a direct analog to the `seizeInternal` zero-value issue: a no-op path that consumes gas and emits misleading events.

---

### Finding Description

`withdraw()` at line 150 of `Entropy.sol` contains no `amount > 0` guard and, critically, no check that `msg.sender` is a registered provider (unlike `withdrawAsFeeManager`, which checks `providerInfo.sequenceNumber == 0`).

```solidity
function withdraw(uint128 amount) public override {
    EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
        msg.sender
    ];

    // ❌ No check: amount > 0
    // ❌ No check: msg.sender is a registered provider

    require(
        providerInfo.accruedFeesInWei >= amount,   // 0 >= 0 → always true
        "Insufficient balance"
    );
    providerInfo.accruedFeesInWei -= amount;        // 0 -= 0 → no-op

    (bool sent, ) = msg.sender.call{value: amount}(""); // call{value:0} → succeeds
    require(sent, "withdrawal to msg.sender failed");

    emit EntropyEvents.Withdrawal(msg.sender, msg.sender, amount);      // emits amount=0
    emit EntropyEventsV2.Withdrawal(msg.sender, msg.sender, amount, bytes("")); // emits amount=0
}
``` [1](#0-0) 

When `amount == 0`:

1. `providerInfo.accruedFeesInWei >= 0` is always `true`, even for an address that has never registered as a provider (default storage value is 0).
2. `accruedFeesInWei -= 0` is a no-op; no state is mutated.
3. `msg.sender.call{value: 0}("")` is a zero-value call that succeeds unconditionally.
4. Both `EntropyEvents.Withdrawal` and `EntropyEventsV2.Withdrawal` are emitted with `withdrawnAmount = 0`.

The same gap exists in `withdrawAsFeeManager`, though that function does enforce provider registration and fee-manager authorization before reaching the zero-amount path. [2](#0-1)

---

### Impact Explanation

- **Any unprivileged address** can emit `Withdrawal` events without being a registered Entropy provider and without any ETH actually moving.
- Off-chain indexers, accounting dashboards, and monitoring systems that consume `Withdrawal` events (e.g., Fortuna's fee-tracking logic) will observe spurious zero-amount withdrawal records from arbitrary addresses, polluting provider accounting and potentially triggering false alerts.
- The `Withdrawal` event carries `provider` and `recipient` fields indexed by off-chain consumers; emitting it for unregistered addresses misrepresents the on-chain state.
- Gas is consumed for a fully no-op execution path.

---

### Likelihood Explanation

Trivially reachable. No registration, no fee payment, no privileged role required. Any EOA or contract can call `random.withdraw(0)` at any time. The entry path is a direct public function call by an unprivileged transaction sender, which is explicitly within the Immunefi scope.

---

### Recommendation

Add a zero-amount guard at the top of both `withdraw()` and `withdrawAsFeeManager()`:

```solidity
function withdraw(uint128 amount) public override {
    if (amount == 0) revert EntropyErrors.AssertionFailure();
    // ... rest of function
}

function withdrawAsFeeManager(address provider, uint128 amount) external override {
    if (amount == 0) revert EntropyErrors.AssertionFailure();
    // ... rest of function
}
```

This mirrors the recommendation in the reference report: quit the function early after checks when the operative value is zero.

### Proof of Concept

```solidity
// Any address — registered or not — can execute this:
address attacker = address(0xdead);
vm.prank(attacker);

// Succeeds: emits Withdrawal(attacker, attacker, 0) twice
random.withdraw(0);

// Verify: no state changed, but events were emitted
assertEq(random.getProviderInfoV2(attacker).accruedFeesInWei, 0);
// Off-chain: two Withdrawal events with withdrawnAmount=0 are now in the log
```

The `withdraw(0)` call from an unregistered address passes the `accruedFeesInWei >= 0` check (always true for default storage), performs no state mutation, and emits two `Withdrawal` events — a direct analog to the zero `seizeInternal` pattern in the reference report.
