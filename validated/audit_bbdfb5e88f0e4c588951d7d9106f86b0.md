### Title
Entropy Provider Can Front-Run User `requestV2` Calls by Immediately Raising `feeInWei` Without a Grace Period — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract allows any registered provider (or their designated fee manager) to call `setProviderFee` / `setProviderFeeAsFeeManager` and have the new fee take effect **immediately** with no delay or grace period. Because excess `msg.value` is explicitly **not refunded** to callers of `requestV2`, a malicious provider can front-run a pending user request to raise their fee, either capturing a larger share of the user's payment or forcing the user's transaction to revert.

---

### Finding Description

`setProviderFee` and `setProviderFeeAsFeeManager` write the new fee directly to storage with no time-lock or pending-state mechanism:

```solidity
// Entropy.sol line 810-827
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;          // ← immediate, no grace period
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
    ...
}
``` [1](#0-0) 

The same pattern exists for `setProviderFeeAsFeeManager`: [2](#0-1) 

In `requestHelper`, the fee is computed at call time from the live `feeInWei` value, and the provider accrues exactly `getProviderFee()` while Pyth accrues the remainder of `msg.value`:

```solidity
// Entropy.sol lines 234-239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [3](#0-2) 

The `IEntropyV2` interface explicitly documents that excess value is **not refunded**:

> "Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2()` prior to each invocation… Further note that excess value is *not* refunded to the caller." [4](#0-3) 

Because of this documented behavior, many integrating contracts send a small buffer above `getFeeV2()` to avoid reverts from fee fluctuations. This buffer is the attack surface.

---

### Impact Explanation

**Scenario A — Fee extraction from user's buffer:**
1. User contract calls `getFeeV2(provider)` → receives `F`.
2. User submits `requestV2{value: F + buffer}` to avoid revert risk.
3. Provider front-runs with `setProviderFee(feeInWei + buffer)`.
4. User's transaction passes (`msg.value >= new requiredFee`).
5. Provider accrues `feeInWei + buffer` instead of `feeInWei`; Pyth accrues correspondingly less.

The provider extracts the user's safety buffer, which was intended to go to the Pyth protocol fee pool.

**Scenario B — Targeted DoS:**
1. User submits `requestV2{value: F}` (exact fee, no buffer).
2. Provider front-runs with `setProviderFee(feeInWei + 1)`.
3. User's transaction reverts with `InsufficientFee`.

The provider can selectively deny service to specific users or contracts without permanently changing their fee (they can lower it back after the target tx reverts).

---

### Likelihood Explanation

- Any registered Entropy provider can call `setProviderFee` permissionlessly; registration is open to anyone.
- The Fortuna keeper (`apps/fortuna/src/keeper/fee.rs`) already monitors the mempool and adjusts fees programmatically, demonstrating that automated fee-change tooling exists in the ecosystem. [5](#0-4) 

- On chains with a public mempool (Ethereum mainnet, most EVM L2s), a provider can observe pending `requestV2` transactions and front-run them within the same block.
- The no-refund policy means users have a rational incentive to send a buffer, making Scenario A reliably exploitable.

---

### Recommendation

Add a two-step fee update mechanism with a mandatory delay (grace period) before the new fee takes effect, analogous to a timelock:

```solidity
uint128 public pendingFeeInWei;
uint256 public feeUpdateEffectiveAt;
uint256 public constant FEE_UPDATE_DELAY = 1 days;

function setProviderFee(uint128 newFeeInWei) external override {
    // ... provider check ...
    pendingFeeInWei = newFeeInWei;
    feeUpdateEffectiveAt = block.timestamp + FEE_UPDATE_DELAY;
    emit ProviderFeeUpdateScheduled(msg.sender, newFeeInWei, feeUpdateEffectiveAt);
}

function applyProviderFee() external override {
    require(block.timestamp >= feeUpdateEffectiveAt, "Grace period not elapsed");
    provider.feeInWei = pendingFeeInWei;
    emit ProviderFeeUpdated(...);
}
```

Alternatively, cap the maximum fee increase per update (e.g., no more than 2× the current fee in a single transaction).

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

interface IEntropyMinimal {
    function setProviderFee(uint128 newFeeInWei) external;
    function getFeeV2(address provider, uint32 gasLimit) external view returns (uint128);
    function requestV2(address provider, bytes32 userRandomNumber, uint32 gasLimit)
        external payable returns (uint64);
}

contract FrontRunPoC {
    IEntropyMinimal entropy;
    address provider;

    constructor(address _entropy, address _provider) {
        entropy = IEntropyMinimal(_entropy);
        provider = _provider;
    }

    // Step 1: User calls this, sending fee + 0.001 ether buffer
    function userRequest(bytes32 userRandom) external payable {
        // msg.value = getFeeV2() + 0.001 ether buffer
        entropy.requestV2{value: msg.value}(provider, userRandom, 0);
    }

    // Step 2: Provider (attacker) front-runs userRequest by calling this first
    // in the same block with higher gas price
    function providerFrontRun(uint128 newHigherFee) external {
        // Provider raises fee to consume the user's buffer
        entropy.setProviderFee(newHigherFee);
        // After this tx mines before userRequest:
        // - If user sent enough: provider accrues newHigherFee (captures buffer)
        // - If user sent exactly old fee: userRequest reverts
    }
}
```

The provider submits `providerFrontRun` with a higher gas price than the pending `userRequest`, ensuring it mines first. The user either loses their buffer to the provider or their transaction reverts.

            provider
