### Title
Unbounded `while` Loop in `executeCallback` Advances `firstUnfulfilledSeq` with Unbounded Storage Reads — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that scans forward through all inactive (cleared) sequence numbers to advance `_state.firstUnfulfilledSeq`. An attacker who creates many requests and ensures the lowest-numbered one is fulfilled last can force this loop to perform O(N) cold `SLOAD` operations in a single transaction, potentially exceeding the block gas limit and permanently bricking callback execution for that call.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs: [1](#0-0) 

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

The developer's own TODO comment acknowledges the gas problem. Each iteration calls `findRequest`, which performs at least one `SLOAD` on `_state.requests[shortKey]` and potentially a second on `_state.requestsOverflow[key]`: [2](#0-1) 

There is no cap on `currentSequenceNumber` — it increments with every call to `requestPriceUpdatesWithCallback`: [3](#0-2) 

`NUM_REQUESTS = 32` is only the size of the fixed-size active-request ring buffer, not a limit on total historical requests: [4](#0-3) 

---

### Impact Explanation

**Scenario:**
1. Attacker creates N requests (sequence numbers 1 … N). Each call to `requestPriceUpdatesWithCallback` is permissionless and only requires paying the fee.
2. Requests 2 … N are fulfilled (by the provider or anyone after the exclusivity period). For each of these, the while loop does not advance `firstUnfulfilledSeq` because request 1 is still active — the loop body never executes.
3. When request 1 is finally fulfilled, `clearRequest(1)` marks it inactive. The while loop then scans: seq 1 (inactive) → seq 2 (inactive) → … → seq N (inactive) → seq N+1 (= `currentSequenceNumber`, loop exits). This is **N cold SLOADs** in a single transaction.

At ~2,100 gas per cold SLOAD, ~4,750 requests exhaust a 10M-gas block. On chains with lower block gas limits (e.g., some L2s), the threshold is even lower.

The result is that the `executeCallback` call for request 1 **reverts out-of-gas**, permanently locking the requester's funds in the contract (since there is no alternative withdrawal path for the requester once a request is active).

---

### Likelihood Explanation

- `requestPriceUpdatesWithCallback` is fully permissionless; any address can create requests.
- The exclusivity period only restricts *which provider* can fulfill a request, not *when* request 1 is fulfilled relative to others.
- On low-fee chains (Arbitrum, Base, etc.), the cost to create thousands of requests is negligible.
- The developer comment at line 166 explicitly flags this as a known gas hazard, confirming the issue is real and unmitigated.

---

### Recommendation

Replace the unbounded while loop with one of:

1. **Doubly-linked list of active requests** (as the TODO suggests): O(1) removal and O(1) advancement of `firstUnfulfilledSeq`.
2. **Cap the loop iterations** to a small constant (e.g., 32) per `executeCallback` call, accepting that `firstUnfulfilledSeq` may lag.
3. **Remove `firstUnfulfilledSeq` tracking entirely** from `executeCallback` and advance it lazily in `getFirstActiveRequests` (a view function), since `firstUnfulfilledSeq` is only used there.

---

### Proof of Concept

```
1. Deploy Echo on a chain with 10M block gas limit.
2. Call requestPriceUpdatesWithCallback(...) 5,000 times → sequence numbers 1..5000.
3. Call executeCallback for sequences 2..5000 (in any order).
   - Each call: while loop checks seq=1, which is still active → 0 iterations. Cheap.
4. Call executeCallback for sequence 1.
   - clearRequest(1) marks it inactive.
   - while loop: seq 1 inactive → seq 2 inactive → ... → seq 5000 inactive → seq 5001.
   - 5,000 iterations × ~2,100 gas/SLOAD ≈ 10,500,000 gas → exceeds block gas limit → revert.
5. Request 1's callback is never executed; requester's funds are locked.
``` [5](#0-4) [6](#0-5) [7](#0-6)

    function findRequest(
        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];
        }
    }
