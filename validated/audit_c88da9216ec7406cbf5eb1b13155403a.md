Audit Report

## Title
Unprivileged Child VM Can Consume Sibling's Exit Code via `ckb_wait` Without Parent Ownership Check — (`script/src/syscalls/wait.rs`, `script/src/scheduler.rs`)

## Summary

`Wait::ecall` forwards any caller-supplied `target_id` to the scheduler's message box with no parent-child relationship check. The `Message::Wait` handler in `process_message_box` destructively removes the target's entry from `terminated_vms` upon a successful match, regardless of whether the caller is the legitimate parent. A malicious child VM can therefore consume a sibling's exit code, causing the true parent's subsequent `ckb_wait` on that sibling to return `WAIT_FAILURE` and corrupt the parent script's control flow.

## Finding Description

**`Wait::ecall` — no ownership check:**
`target_id` is read directly from register A0 and enqueued in the message box with only the caller's own `self.id` attached. No parent-child relationship is verified at this point. [1](#0-0) 

**`process_message_box` — destructive single-consumer semantics:**
The `Message::Wait` arm checks only (1) whether `args.target_id` is in `terminated_vms` and (2) whether it is still in `states`. There is no check that `vm_id` (the caller) is the recorded parent of `args.target_id`. On a successful match the entry is permanently removed via `retain`. [2](#0-1) 

**No parent tracking exists in the scheduler:**
The `Scheduler` struct tracks `fds`, `inherited_fd`, `states`, `instantiated`, `suspended`, and `terminated_vms`, but contains no field mapping a child VM ID to its spawning parent. No `parent_vm` field or equivalent exists anywhere in `scheduler.rs`. [3](#0-2) 

**Sequential, predictable VM IDs:**
`next_vm_id` is incremented monotonically at each spawn, making sibling PIDs trivially predictable. [4](#0-3) 

**Termination path inserts into `terminated_vms` unconditionally:**
When a non-root VM exits, its exit code is inserted into `terminated_vms` before any joining VMs are resolved. If no VM is already in `VmState::Wait` for that target at termination time, the entry persists in `terminated_vms` and is available for any VM to consume. [5](#0-4) 

**Exploit flow:**
1. Root spawns child A (PID=1), then child B (PID=2).
2. B exits → `terminated_vms.insert(2, code)`, B removed from `states`.
3. A calls `ckb_wait(2)` — the scheduler finds B in `terminated_vms`, writes the exit code into A's memory, marks A `Runnable`, and calls `terminated_vms.retain(|id,_| id != &2)`, permanently removing B's entry.
4. Root calls `ckb_wait(2)` — B is absent from both `terminated_vms` and `states`, so the scheduler sets A0 to `WAIT_FAILURE`. [6](#0-5) 

## Impact Explanation

This is an incorrect implementation of CKB-VM's spawn/wait mechanism, matching the allowed impact class **"Incorrect implementation or behavior of CKB-VM or system scripts" — High (10001–15000 points)**. A root script that spawns an attacker-controlled child alongside a trusted verification child (a pattern enabled by CKB's cell-based code loading) will receive `WAIT_FAILURE` for the trusted child instead of its real exit code. If the root script gates transaction validity on that exit code (e.g., asserting the child returned 0), the attacker can cause a legitimately valid transaction to be rejected, or — depending on the script's branching logic — cause an invalid transaction to pass validation, enabling unauthorized cell spending.

## Likelihood Explanation

The attack is fully reachable by any unprivileged script author. No special privileges, leaked keys, or majority hashpower are required. The attacker deploys a lock or type script that spawns a malicious child alongside a trusted child. PID prediction is trivial because IDs are sequential integers starting from `FIRST_VM_ID`; alternatively, the root can pass the sibling's PID to the malicious child via a pipe, making the attack fully deterministic. The attack is locally reproducible with standard CKB-VM tooling.

## Recommendation

Track the spawning parent at spawn time by adding a `parent_vm: BTreeMap<VmId, VmId>` field to `Scheduler`. In `Message::Spawn` handling, record `parent_vm.insert(spawned_vm_id, vm_id)`. In the `Message::Wait` handler inside `process_message_box`, before processing the wait, verify:

```rust
if self.parent_vm.get(&args.target_id) != Some(&vm_id) {
    machine.inner_mut().set_register(A0, Self::u8_to_reg(WAIT_FAILURE));
    continue;
}
```

This mirrors POSIX `waitpid` semantics where only the direct parent may reap a child. The `parent_vm` entry should be removed alongside the `terminated_vms` entry after a successful wait.

## Proof of Concept

A minimal unit test written against `Scheduler` directly:
1. Spawn two child VMs from root (A = PID 1, B = PID 2).
2. Let B terminate with exit code 0.
3. Have A call `ckb_wait(2)` — assert it returns `SUCCESS` and exit code 0.
4. Have root call `ckb_wait(2)` — assert it returns `WAIT_FAILURE` (B's entry was consumed by A).

```c
// Root VM (trusted script)
uint64_t pid_a, pid_b;
ckb_spawn(...malicious_code..., &pid_a);   // pid_a = 1
ckb_spawn(...trusted_code...,   &pid_b);   // pid_b = 2
// trusted child B exits with code 0

int8_t exit_code;
int rc = ckb_wait(pid_b, &exit_code);
// Expected: rc == SUCCESS, exit_code == 0
// Actual:   rc == WAIT_FAILURE  (B's entry was consumed by A)
assert(rc == SUCCESS);  // fails → transaction rejected or logic corrupted

// Child A (malicious, runs concurrently)
uint64_t sibling_pid = my_pid + 1;  // sequential: trivially known
int8_t stolen_code;
ckb_wait(sibling_pid, &stolen_code);
// Succeeds: steals B's exit code, removes B from terminated_vms
```
