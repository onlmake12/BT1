Audit Report

## Title
Unauthenticated `from` PeerId in Hole-Punching Protocol Enables Attacker-Directed Outbound TCP Connection Storms — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
The hole-punching protocol accepts an arbitrary `from` PeerId from message content in both `ConnectionRequest` and `ConnectionSync` handlers without verifying it matches the authenticated session sender. An attacker with a single P2P connection can forge `from` to inject attacker-controlled TCP addresses into the victim's `pending_delivered` map, then trigger unbounded outbound TCP connection attempts via a follow-up `ConnectionSync`. The per-session rate limit of 30 req/sec is the only binding constraint, enabling sustained outbound TCP connection storms toward arbitrary third-party targets and exhausting the victim node's file descriptors.

## Finding Description

**Root cause — no `from` authentication in `respond_delivered`:**

`ConnectionRequestProcess::execute` calls `respond_delivered` with `content.from` taken directly from the message when `self_peer_id == &content.to`. The `self.peer` field (the authenticated `PeerIndex`) is never used to verify the claimed `from` identity: [1](#0-0) 

The deduplication guard in `respond_delivered` is keyed by `from_peer_id` from the message content. A fresh random `from` PeerId per request trivially bypasses it: [2](#0-1) 

After filtering to TCP/IP addresses, the forged `from_peer_id` and attacker-controlled `remote_listens` are inserted unconditionally into `pending_delivered`: [3](#0-2) 

**Root cause — no `from` authentication in `ConnectionSyncProcess`:**

When `route` is empty and `self_peer_id == &content.to`, the handler looks up `pending_delivered` using `content.from` from the message with no check that it matches the actual session sender: [4](#0-3) 

For every stored address, a `runtime::spawn` task is created that calls `try_nat_traversal` via `select_ok`, making real outbound TCP connection attempts: [5](#0-4) 

**`try_nat_traversal` makes real, long-lived TCP connections:**

`try_nat_traversal` creates a TCP socket and retries `connect()` in a loop for up to 30 seconds with ~200ms intervals, making approximately 150 TCP connection attempts per task invocation: [6](#0-5) 

**Rate-limit bypass:**

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`: [7](#0-6) 

It is configured at 1 req/sec per `(from, to, msg_item_id)` tuple: [8](#0-7) 

Since the attacker controls `from`, using a fresh random PeerId per message creates a new key and bypasses this limiter entirely. The only binding constraint is the per-session `rate_limiter` at 30 req/sec keyed by `(session_id, msg.item_id())`: [9](#0-8) 

**Unbounded `pending_delivered` growth:**

The cleanup in `notify()` runs every 5 minutes (`CHECK_INTERVAL = 5 * 60s`, `TIMEOUT = 5 * 60 * 1000ms`). At 30 req/sec, an attacker inserts up to 9,000 entries per cleanup window, each holding up to `ADDRS_COUNT_LIMIT` (24) `Multiaddr` values: [10](#0-9) 

**Exploit flow:**
1. Attacker establishes a standard P2P connection to the victim.
2. Attacker sends `ConnectionRequest` with `from = PeerId::random()`, `to = victim_peer_id`, `listen_addrs = [attacker_tcp_addr × 24]`, `route = []`. The victim's `self_peer_id == &content.to` check passes; `respond_delivered` inserts the forged entry into `pending_delivered`.
3. Attacker immediately sends `ConnectionSync` with the same forged `from`, `to = victim_peer_id`, `route = []`. The victim looks up `pending_delivered[forged_from]`, finds 24 attacker addresses, and spawns a `runtime::spawn` task running `select_ok` over 24 concurrent `try_nat_traversal` futures.
4. Each `try_nat_traversal` future retries TCP `connect()` for 30 seconds. At 30 pairs/sec, after 30 seconds there are 900 concurrent spawned tasks, each holding up to 24 concurrent TCP sockets = up to 21,600 concurrent outbound TCP connections.
5. File descriptor exhaustion causes the node process to crash or become unable to accept/initiate legitimate connections.

## Impact Explanation

**High — Vulnerability which could easily crash a CKB node.**

At 30 message pairs/sec (the per-session cap), each pair spawns a `runtime::spawn` task that attempts outbound TCP connections to up to 24 attacker-controlled addresses for 30 seconds each. Sustained over 30 seconds, this creates up to 900 concurrent long-lived tasks holding up to 21,600 concurrent TCP sockets, exhausting the victim node's file descriptors and OS connection-tracking state. The victim's IP also becomes the apparent source of a TCP SYN flood toward arbitrary third-party targets.

## Likelihood Explanation

The attacker requires only a single standard P2P connection — no key material, PoW, or privileged role. The victim's PeerId is publicly observable from the P2P network. The `HolePunching` protocol is enabled by default. The two-message sequence is straightforward to construct from the public `packed::ConnectionRequest` / `packed::ConnectionSync` schema. The attack is repeatable and sustainable for the duration of the session.

## Recommendation

1. **Authenticate `from` in `respond_delivered`**: Resolve the session's PeerId from `self.peer` via `network_state.peer_registry` and reject any `ConnectionRequest` where `content.from` does not match the authenticated session PeerId.
2. **Authenticate `from` in `ConnectionSyncProcess`**: Pass the session's PeerId into `ConnectionSyncProcess` and verify `content.from` against it before looking up `pending_delivered`.
3. **Cap `pending_delivered` size**: Enforce a maximum entry count (e.g., equal to `max_outbound`) to bound memory growth from any remaining unauthenticated paths.

## Proof of Concept

```rust
// Step 1: send ConnectionRequest to victim with forged `from` and attacker TCP addr
// Repeat with fresh forged_from PeerIds at 30/sec to bypass HOLE_PUNCHING_INTERVAL and forward_rate_limiter
let forged_from = PeerId::random();
let attacker_addr: Multiaddr = "/ip4/1.2.3.4/tcp/9999".parse().unwrap();
let req = packed::ConnectionRequest::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())   // victim's known PeerId
    .listen_addrs(vec![attacker_addr.to_vec(); 24].pack())
    .max_hops(6u8.pack())
    .route(vec![].pack())        // empty route → triggers respond_delivered
    .build();
attacker_session.send(req);

// Step 2: send ConnectionSync with same forged `from`
let sync = packed::ConnectionSync::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())
    .route(vec![].pack())        // empty route → triggers pending_delivered lookup
    .build();
attacker_session.send(sync);

// Result: victim spawns runtime::spawn task → select_ok over 24 try_nat_traversal futures
// Each try_nat_traversal retries TCP connect() for 30 seconds (~150 attempts per future)
// Repeat at 30 pairs/sec with fresh forged_from each time
// After 30 seconds: 900 concurrent tasks × 24 sockets = ~21,600 concurrent TCP connections
// File descriptor exhaustion → node crash
```

pub(crate) async fn try_nat_traversal(
    bind_addr: Option<SocketAddr>,
    addr: Multiaddr,
) -> Result<(TcpStream, Multiaddr), std::io::Error> {
    let net_addr = match multiaddr_to_socketaddr(&addr) {
        Some(addr) => addr,
        None => {
            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }
    };

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }

    debug!("Failed to connect to NAT after {} retries", retry_count);
    Err(std::io::ErrorKind::TimedOut.into())
}
