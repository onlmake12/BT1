### Title
Unbounded Per-Message DB Read Cost in `GetBlockFilterCheckPointsProcess::execute` with No Per-Peer Rate Limit — (`sync/src/filter/get_block_filter_check_points_process.rs`)

### Summary

Any unprivileged remote peer can repeatedly send `GetBlockFilterCheckPoints(start_number=0)` over the Filter P2P protocol. Each message unconditionally triggers up to `BATCH_SIZE=2000` loop iterations, each performing two synchronous DB reads (`get_block_hash` + `get_block_filter_hash`). No rate limiting, per-peer throttle, or repeated-request penalty exists anywhere in the filter protocol handler.

### Finding Description

In `GetBlockFilterCheckPointsProcess::execute`, the handler reads `start_number` from the peer message and immediately enters a loop bounded only by `BATCH_SIZE`: [1](#0-0) [2](#0-1) 

Each iteration calls `get_block_hash(block_number)` followed by `get_block_filter_hash(&block_hash)` — two DB reads — stepping by `CHECK_POINT_INTERVAL=2000` each time. On a node with ≥4,000,000 blocks, all 2000 iterations complete, yielding 4,000 DB reads per message.

The `BlockFilter::received` handler in `sync/src/filter/mod.rs` dispatches directly to `try_process` → `execute` with no rate-limiting layer: [3](#0-2) 

The only ban path in `process` is triggered by `status.should_ban()`, which is never returned for a well-formed `GetBlockFilterCheckPoints` message — it returns `Status::ok()` or `Status::ignored()`: [4](#0-3) 

Unlike the relayer (which has rate-limiting logic), the `BlockFilter` protocol handler has no equivalent mechanism. There is no per-peer message counter, no cooldown, and no deduplication of identical requests.

### Impact Explanation

A single attacker peer sending `GetBlockFilterCheckPoints(start_number=0)` in a tight loop forces the server to perform up to 4,000 RocksDB reads per message with no throttle. Multiple coordinated peers (within the node's inbound peer limit) multiply this linearly. The result is sustained I/O saturation and CPU pressure on the DB layer, causing measurable performance degradation for all node operations sharing the same DB (block sync, tx relay, RPC). This matches the stated scope: **Low (501–2000) — important performance degradation**.

### Likelihood Explanation

The attack requires only a valid P2P connection and a single valid message type. No authentication, no PoW, no special state. The Filter protocol is a standard supported protocol registered in production: [5](#0-4) 

### Recommendation

1. Add a per-peer rate limit (e.g., token bucket or fixed window counter) for `GetBlockFilterCheckPoints` messages in `BlockFilter::received` or `try_process`.
2. Deduplicate or cache responses for identical `(peer, start_number)` requests within a short window.
3. Consider reducing `BATCH_SIZE` or requiring a minimum interval between checkpoint requests from the same peer.

### Proof of Concept

    send GetBlockFilterCheckPoints { start_number: 0 }
# Observe: DB read count per message = O(BATCH_SIZE=2000 * 2) = 4000 reads

    pub async fn execute(self) -> Status {
        let active_chain = self.filter.shared.active_chain();
        let start_number: BlockNumber = self.message.to_entity().start_number().into();
        let latest: BlockNumber = active_chain.get_latest_built_filter_block_number();

        let mut block_filter_hashes = Vec::new();

        if latest >= start_number {
            let mut block_number = start_number;
            for _ in 0..BATCH_SIZE {
                if let Some(block_filter_hash) = active_chain
                    .get_block_hash(block_number)
                    .and_then(|block_hash| active_chain.get_block_filter_hash(&block_hash))
                {
                    block_filter_hashes.push(block_filter_hash);
                } else {
                    break;
                }

                    break;

                .set(content)
                .build();
            async_send_message_to(&self.nc, self.peer, &message).await
        } else {
            Status::ignored()
        }
    }
