Audit Report

## Title
Fee-Rate Admission Uses Size-Only Metric While Pool Stores Weight-Based Rate, Allowing Sub-Minimum-Fee-Rate Transactions Into the Pool - (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` enforces `min_fee_rate` using raw serialized byte size before script execution, when cycles are unknown. After `verify_rtx` returns actual cycles, `TxEntry` is constructed with those cycles and passed directly to `submit_entry` with no second fee-rate check using the weight-based metric (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). A cycle-heavy transaction can pass admission paying only the size-based minimum while its effective pool fee rate is up to ~12× below `min_fee_rate` at the default `max_tx_verify_cycles` limit.

## Finding Description
**Admission gate** (`tx-pool/src/util.rs`, L42–52): The comment explicitly acknowledges the mismatch — "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly." The gate computes `min_fee = min_fee_rate.fee(tx_size)` and rejects only if `fee < min_fee`. [1](#0-0) 

**Post-verification path** (`tx-pool/src/process.rs`, L715–753): `pre_check` (which calls `check_tx_fee`) runs before `verify_rtx`. After `verify_rtx` returns `verified.cycles`, `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is constructed and passed directly to `submit_entry`. A grep for `min_fee_rate` in `process.rs` returns zero matches, confirming no weight-based re-check exists anywhere in this path. [2](#0-1) 

**Pool entry fee rate** (`tx-pool/src/component/entry.rs`, L114–118): `TxEntry::fee_rate()` computes `weight = get_transaction_weight(self.size, self.cycles)` — i.e., `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)` — and returns `FeeRate::calculate(self.fee, weight)`. For a 1000-byte transaction consuming 70,000,000 cycles, weight = `max(1000, 11940) = 11940`, so the stored fee rate is `1000 × 1000 / 11940 ≈ 83 shannons/KW` against a `min_fee_rate` of 1000 shannons/KW. [3](#0-2) 

**`get_transaction_weight` formula** (`util/types/src/core/tx_pool.rs`, L298–303): confirmed as `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. [4](#0-3) 

**Existing guards are insufficient**: The `DeclaredWrongCycles` check (L736–748) only fires when a remote peer declares cycles differing from verified cycles; it does not re-evaluate fee rate. The eviction path uses weight-based fee rate but only removes transactions after they are already in the pool. [5](#0-4) 

## Impact Explanation
This is a **bad design which could cause CKB network congestion with few costs** (High, 10001–15000 points). An attacker can submit cycle-heavy transactions at ~12× reduced cost relative to what the weight-based policy intends. By continuously submitting such transactions (each paying only the size-based minimum), the attacker can occupy pool slots at a fraction of the intended cost. When evicted due to low weight-based fee rate, they can be resubmitted immediately, sustaining pool occupation. The fee estimator (`weight_units_flow`) ingests admitted entries and computes weight-based fee rates from them, polluting historical fee-rate data and causing the estimator to recommend lower rates to users, compounding the congestion effect.

## Likelihood Explanation
Any unprivileged `send_transaction` RPC caller can trigger this. The attacker needs only a deployed lock script that runs a tight loop consuming near `max_tx_verify_cycles` (achievable with standard CKB-VM opcodes) and a cell locked by it. No special privileges, no victim interaction, and no external dependencies are required. The default `max_tx_verify_cycles` of 70,000,000 provides a reliable upper bound on the achievable mismatch ratio (~12×), reachable on any node with default configuration.

## Recommendation
After `verify_rtx` returns and actual cycles are known, perform a second fee-rate check using the weight-based metric before calling `submit_entry`:

```rust
let weight = get_transaction_weight(tx_size as usize, verified.cycles);
let actual_fee_rate = FeeRate::calculate(fee, weight);
if actual_fee_rate < tx_pool_config.min_fee_rate {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, ...)), snapshot));
}
```

This check belongs in `_process_tx` immediately after the `verified` result is obtained (after L734) and before `TxEntry::new` is constructed (L751). The pre-check size-only gate can remain as a fast-path rejection before the expensive script execution.

## Proof of Concept
1. Configure a CKB node with default `min_fee_rate = 1000` shannons/KW.
2. Deploy a lock script that executes a tight loop consuming ~70,000,000 cycles (within `max_tx_verify_cycles`).
3. Construct a transaction spending a cell locked by that script; serialized size ≈ 1,000 bytes.
4. Set transaction fee to exactly 1,000 shannons (satisfies `min_fee_rate × size / 1000 = 1000`).
5. Submit via `send_transaction` RPC.
6. Observe the transaction is admitted. Its actual weight = `max(1000, 70_000_000 × 0.000_170_571_4) ≈ 11,940`.
7. Actual fee rate = `1000 × 1000 / 11940 ≈ 83 shannons/KW` — 12× below `min_fee_rate`.
8. Confirm via `get_raw_tx_pool` (verbose=true): the transaction appears in the pool; the `weight` field in `AncestorsScoreSortKey` reflects the true weight, confirming the mismatch with the admission threshold.
9. Repeat continuously to sustain pool occupation at 12× reduced cost.
