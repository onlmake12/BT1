Audit Report

## Title
Minimum Fee Rate Admission Check Uses `tx_size` Instead of Weight, Allowing Cycle-Heavy Transactions to Bypass Fee Rate Enforcement - (File: `tx-pool/src/util.rs`)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using only the raw serialized byte size (`tx_size`), not the canonical weight `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. After script verification completes in `_process_tx` and actual cycles are known, no second weight-based fee-rate check is performed before the transaction is admitted to the pool. A transaction with small byte size and very high cycles passes the admission gate while its true fee rate is orders of magnitude below `min_fee_rate`.

## Finding Description
The canonical weight formula is defined in `util/types/src/core/tx_pool.rs`:

```rust
// util/types/src/core/tx_pool.rs L298-303
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [1](#0-0) 

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` is defined at line 279. [2](#0-1) 

The admission gate `check_tx_fee` deviates from this formula, using only `tx_size`:

```rust
// tx-pool/src/util.rs L42-45
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [3](#0-2) 

`check_tx_fee` is called from `pre_check` at lines 289 and 294, before script verification, so cycles are not yet known. [4](#0-3) 

After verification completes in `_process_tx`, `verified.cycles` is available at line 734, but the code proceeds directly to `TxEntry::new` and `submit_entry` with no weight-based fee-rate check:

```rust
// tx-pool/src/process.rs L734-753
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
// ... DeclaredWrongCycles check (only fires when declared_cycles is Some) ...
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
``` [5](#0-4) 

The `DeclaredWrongCycles` check at lines 736–749 only fires when `declared_cycles` is `Some`. For local RPC (`send_transaction`), `declared_cycles` is `None`, so `max_cycles = self.consensus.max_block_cycles()` and the check is entirely skipped: [6](#0-5) 

`TxEntry::fee_rate()` correctly uses `get_transaction_weight`, but this is only used for sorting and eviction — not for admission: [7](#0-6) 

**Exploit path:**
1. Craft a transaction: serialized size ≈ 200 bytes, actual cycles ≈ 70,000,000 (default `max_tx_verify_cycles`).
2. Set fee = 201 shannons.
3. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`. Fee 201 > 200 → **admitted**.
4. Actual weight = `max(200, 70_000_000 × 0.000_170_571_4)` ≈ 11,940.
5. True fee rate = `201 × 1000 / 11_940` ≈ **16.8 shannons/KW** — ~60× below `min_fee_rate = 1000`.
6. Transaction enters pool. If `send_transaction` uses the direct `_process_tx` path with `max_block_cycles = 3,500,000,000`, the cycle multiplier is ~3000×.

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. An attacker can flood the tx-pool with transactions that each consume a large fraction of the block cycle budget while paying only ~200 shannons per transaction. Each such transaction, once proposed and committed, prevents other transactions from being included in that block. The pool size is 180 MB (`max_tx_pool_size = 180_000_000`); with ~200-byte transactions, thousands of such entries can be submitted. Even a small number saturates the cycle budget for consecutive blocks, causing severe throughput degradation for all users at negligible cost. [8](#0-7) 

## Likelihood Explanation
The attack is reachable via the standard `send_transaction` RPC, which by default binds to `127.0.0.1:8114`. No special privilege, key, or majority hashpower is required beyond RPC access. The attacker only needs to know `min_fee_rate` (visible via `tx_pool_info` RPC) and craft a transaction with a computationally heavy CKB-VM script. The code comment at `util.rs` line 42–44 explicitly acknowledges the approximation, confirming this is a known design gap. The attack is repeatable and cheap.

## Recommendation
After script verification completes in `_process_tx` (where `verified.cycles` is available), perform a second, accurate fee-rate check using the full weight before calling `submit_entry`:

```rust
// After verified.cycles is known, before submit_entry:
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64())), snapshot));
}
```

The existing `check_tx_fee` call in `pre_check` can remain as a cheap lower-bound guard, but a definitive weight-based check must be added post-verification in `_process_tx`.

## Proof of Concept
1. Configure node with default `min_fee_rate = 1000` shannons/KW (`resource/ckb.toml` line 212).
2. Write a CKB-VM script that performs a tight computation loop consuming ~69,999,000 cycles (just under `max_tx_verify_cycles = 70_000_000`).
3. Build a transaction using that script: serialized size ≈ 200 bytes, fee = 201 shannons.
4. Submit via `send_transaction` RPC (no declared cycles needed).
5. `check_tx_fee` passes: `min_fee = 1000 × 200 / 1000 = 200`, fee 201 > 200.
6. `verify_rtx` runs with `max_cycles = max_block_cycles`, script executes successfully consuming ~70M cycles.
7. No `declared_cycles` → `DeclaredWrongCycles` check is skipped entirely.
8. Transaction enters pool with `fee_rate() ≈ 16.8 shannons/KW`, ~60× below `min_fee_rate`.
9. Repeat with chained UTXOs to fill the pool with cycle-saturating, near-zero-fee-rate transactions.
10. Each block can include at most a handful of such transactions (cycle budget exhausted), blocking all legitimate transactions.
