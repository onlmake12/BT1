Audit Report

## Title
Unauthenticated `from` field in `ConnectionRequest`/`ConnectionSync` enables unbounded `try_nat_traversal` task spawning and eclipse attack — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
`ConnectionRequestProcess::execute()` stores wire-controlled `content.from` into `pending_delivered` without verifying it matches the authenticated session peer. `ConnectionSyncProcess` carries no session identity at all, so a subsequent `ConnectionSync` with any matching synthetic `from` causes the victim to spawn unbounded `try_nat_traversal` async tasks making repeated TCP connections to attacker-controlled addresses. The `forward_rate_limiter` is fully bypassed by using N distinct synthetic peer IDs, leaving only the 30 req/s per-session cap as a throttle.

## Finding Description
`ConnectionRequestProcess` holds `peer: PeerIndex` representing the authenticated session, but `execute()` never checks that `content.from` matches the peer ID authenticated for that session. [1](#0-0) 

When `self_peer_id == &content.to`, `respond_delivered` is called with the fully wire-controlled `content.from` and `content.listen_addrs`: [2](#0-1) 

After filtering for TCP/IP4/IP6 addresses, the addresses are stored in `pending_delivered` keyed by the wire-level `from_peer_id`: [3](#0-2) 

`ConnectionSyncProcess` has no `peer` field — it is constructed without a session identity and cannot reference the sender's authenticated peer: [4](#0-3) [5](#0-4) 

When `self_peer_id == &content.to` and `content.route` is empty, the code looks up the wire-controlled `content.from` in `pending_delivered`: [6](#0-5) 

Every stored address is passed to `try_nat_traversal` in a spawned async task: [7](#0-6) [8](#0-7) 

`try_nat_traversal` retries TCP `connect()` every ~200ms for up to 30 seconds: [9](#0-8) 

On success, `raw_session` promotes the stream to a full P2P session: [10](#0-9) 

**Why existing guards fail:**

The `rate_limiter` is keyed by `(session_id, msg.item_id())` at 30 req/s — this is the only binding to the real session and the only effective throttle: [11](#0-10) 

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` at 1 req/s — using N distinct synthetic `from` peer IDs creates N distinct keys, fully bypassing per-pair throttling: [12](#0-11) 

The `HOLE_PUNCHING_INTERVAL` check in `respond_delivered` only deduplicates the same `from_peer_id` within 2 minutes — distinct synthetic IDs each get their own entry: [13](#0-12) 

`pending_delivered` is only pruned every 5 minutes with a 5-minute `TIMEOUT`, giving the attacker a large accumulation window: [14](#0-13) [15](#0-14) 

## Impact Explanation
From a single connected session, the attacker sends 30 `ConnectionRequest` + 30 `ConnectionSync` messages per second (the per-session cap), each with a distinct synthetic `from` peer ID. Each `ConnectionSync` spawns an async task running `try_nat_traversal` for up to 30 seconds, making TCP `connect()` attempts every ~200ms to attacker-controlled addresses. After 30 seconds of sustained attack, ~900 concurrent tasks are active. If the attacker's servers speak the CKB P2P protocol, `raw_session` promotes them to peers, filling peer slots with attacker-controlled nodes. This constitutes **High** impact: network congestion with few costs (a single unprivileged connection suffices), and potentially **Critical** if the resulting eclipse causes consensus deviation.

## Likelihood Explanation
The attacker requires only one normal, unprivileged P2P connection to the victim. The victim's peer ID is public. The two-message sequence (`ConnectionRequest` + `ConnectionSync`) is trivially crafted with arbitrary wire-level `from` fields. No PoW, key material, or privileged role is required. The `forward_rate_limiter` is fully bypassable with synthetic peer IDs. The attack is repeatable and automatable from a single session.

## Recommendation
1. In `ConnectionRequestProcess::respond_delivered`, resolve the authenticated peer ID for `self.peer` from the peer registry and reject the message if `from_peer_id` does not match it.
2. Add a `peer: PeerIndex` field to `ConnectionSyncProcess` (pass `context.session.id` at construction in `mod.rs`), and before looking up `pending_delivered`, verify that `content.from` matches the authenticated peer ID of that session.
3. Alternatively, key `pending_delivered` by `(SessionId, PeerId)` and require the `ConnectionSync` to arrive on the same session as the original `ConnectionRequest`.

## Proof of Concept
```
1. Attacker (session S, authenticated peer A) connects to victim V.
2. For i in 1..N (up to 30/s, limited by rate_limiter):
     A sends ConnectionRequest {
       from = synthetic_peer_id_i,   // arbitrary, not A
       to   = V_peer_id,
       listen_addrs = [/ip4/1.2.3.4/tcp/9999],
       route = [], max_hops = 6
     }
     → V: pending_delivered[synthetic_peer_id_i] = ([/ip4/1.2.3.4/tcp/9999], now)
3. For i in 1..N (up to 30/s):
     A sends ConnectionSync {
       from = synthetic_peer_id_i,
       to   = V_peer_id,
       route = []
     }
     → V: looks up pending_delivered[synthetic_peer_id_i]
     → V: runtime::spawn(try_nat_traversal(bind_addr, /ip4/1.2.3.4/tcp/9999))
     → V: retries TCP connect() to 1.2.3.4:9999 every ~200ms for 30 seconds
4. Attacker's server at 1.2.3.4:9999 accepts; raw_session() promotes it to a peer.
5. After 30 seconds at 30 req/s: ~900 concurrent try_nat_traversal tasks active.
   Peer slots fill with attacker-controlled nodes → eclipse attack.
```
