Audit Report

## Title
`RecentReject::put()` silently destroys just-written entry when `shrink()` randomly drops the same shard — (`tx-pool/src/component/recent_reject.rs`)

## Summary
`RecentReject::put()` writes an entry to a deterministically-assigned shard, then conditionally calls `shrink()`, which randomly drops and recreates one of the `shard_num` column families. If `shrink()` selects the same shard that was just written, the entry is permanently destroyed while `put()` returns `Ok(())`. The root cause is that `self.total_keys_num` is never incremented inside `put()`, causing the shrink trigger to misfire based on a stale estimate, and no guard excludes the just-written shard from the random drop.

## Finding Description

**Missing counter update:** In `put()` at lines 62–69, `self.total_keys_num.checked_add(1)` produces a local binding `total_keys_num` that is used only for the comparison; `self.total_keys_num` itself is never updated. The field is only refreshed inside `shrink()` (line 111) via `estimate_total_keys_num()`. This means the shrink condition `total_keys_num > self.count_limit` evaluates against the DB estimate captured at startup (or after the last shrink), not the true running count.

**Invariant-breaking sequence:**
1. `put()` writes the entry to shard X via `self.db.put(&shard, ...)` (line 60).
2. The stale `self.total_keys_num` satisfies `>= count_limit`, so `shrink()` is called (lines 62–64).
3. `shrink()` samples a random shard via `thread_rng()` over `Uniform::new(0, self.shard_num)` (line 106).
4. If the random pick equals shard X, `drop_cf(X)` destroys all data in X (line 107), then `create_cf_with_ttl(X, ttl)` recreates it empty (line 108).
5. `put()` returns `Ok(())` — the caller observes success.
6. A subsequent `get()` on the same hash returns `None` because the CF is now empty.

**Shard assignment is deterministic and attacker-observable:** `get_shard()` computes `u32::from_le_bytes(hash[0..4]) % shard_num` (lines 115–119), so an attacker can craft a transaction hash that maps to any desired shard. With `DEFAULT_SHARDS = 5`, the probability that `shrink()` drops that exact shard is 1/5 per triggering `put()`.

**Existing checks are insufficient:** `put()` has no mechanism to detect that `shrink()` destroyed the entry it just wrote. The `shared_builder.rs` callback at lines 580–585 only logs an `error!` on `put()` failure — it cannot detect silent data loss since `put()` returns `Ok(())`.

## Impact Explanation

The recent-reject store is the authoritative record of rejected transactions. Silent entry loss causes:
- `get_transaction` RPC to return `TxStatus::Unknown` instead of `TxStatus::Rejected` for the affected transaction.
- The recent-reject guard (`put_recent_reject` / `after_process`) to be silently bypassed, allowing a previously-rejected transaction to be re-submitted and re-verified without the node recognizing it as previously rejected.

This constitutes a **suboptimal implementation of the CKB state storage mechanism** (Medium, 2001–10000 points): the recent-reject store, which is a critical component of the tx-pool's rejection tracking, fails to reliably persist entries due to a missing counter update and an unguarded post-write shrink.

## Likelihood Explanation

The trigger condition (`self.total_keys_num >= count_limit`) is met on any node whose DB already held `>= keep_rejected_tx_hashes_count` rejected transactions at startup — a routine operational state for any active node. Once triggered, the probability of destroying the just-written entry is 1/`shard_num` = 20% with the default of 5 shards. An attacker who can submit rejected transactions via P2P relay (no special privilege required) can repeatedly trigger this condition. With `shard_num = 2` (as in the PoC), the probability rises to 50% per attempt, making reliable triggering straightforward.

## Recommendation

1. **Increment `self.total_keys_num` in `put()`** after a successful write so the counter tracks actual inserts:
   ```rust
   self.total_keys_num = self.total_keys_num.saturating_add(1);
   ```
2. **Call `shrink()` before writing**, not after, so the just-written entry is never in the shard that may be dropped.
3. Alternatively, **exclude the just-written shard from `shrink()`'s random selection** by passing it as a parameter and retrying if the random pick matches.

## Proof of Concept

```rust
// shard_num=2, count_limit=1, ttl=-1
let tmp = tempfile::tempdir().unwrap();
let mut rr = RecentReject::build(tmp.path(), 2, 1, -1).unwrap();

// Force total_keys_num to count_limit so shrink fires on next put
rr.total_keys_num = rr.count_limit; // simulate DB at limit

// Craft a hash that maps to shard 0 (first 4 bytes LE % 2 == 0)
let hash = Byte32::new([0u8; 32]); // 0 % 2 == 0 → shard "0"

// put() writes to shard 0, then shrink() randomly drops shard 0 or 1
rr.put(&hash, Reject::Malformed("x".into(), Default::default())).unwrap();

// With 50% probability, get() returns None immediately after Ok put
// Run in a loop: reliable within ~3 attempts
assert!(rr.get(&hash).unwrap().is_none()); // invariant violated
```

The existing test in `tx-pool/src/component/tests/recent_reject.rs` already exercises `put()` and `get()` but does not set `total_keys_num` to `count_limit` before the second batch of puts, so it does not catch this race. A targeted test with `shard_num = 2` and `total_keys_num = count_limit` will reproduce the failure within a small number of iterations.
