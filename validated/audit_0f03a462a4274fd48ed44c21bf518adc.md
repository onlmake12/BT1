### Title
Missing Saturating Subtraction Guard in `ema_price_no_older_than` Causes DoS via u64 Underflow on Future Timestamps — (File: `target_chains/fuel/contracts/pyth-contract/src/main.sw`)

---

### Summary

The private function `ema_price_no_older_than` in the Fuel Pyth contract performs a raw u64 subtraction `current_time - price.publish_time` without guarding against underflow. The sibling function `price_no_older_than` in the same file explicitly applies a saturating subtraction guard with a comment explaining the underflow risk. The missing guard in `ema_price_no_older_than` means that if a stored EMA price's `publish_time` is even one second ahead of the current Fuel block timestamp, the subtraction wraps to a near-`u64::MAX` value, causing the staleness `require` to always revert. This permanently DoS-es `ema_price()` and `ema_price_no_older_than()` for the affected price feed.

---

### Finding Description

In `target_chains/fuel/contracts/pyth-contract/src/main.sw`, two private staleness-check functions exist side by side:

**Vulnerable — `ema_price_no_older_than` (lines 311–319):**
```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,  // raw u64 subtraction
        PythError::OutdatedPrice,
    );
    price
}
``` [1](#0-0) 

**Fixed — `price_no_older_than` (lines 331–343):**
```sway
fn price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
``` [2](#0-1) 

The developer comment `// Mimicking saturating subtraction to avoid underflow` confirms that Sway u64 arithmetic wraps on underflow in release builds (no panic). The guard was applied to `price_no_older_than` but was never applied to `ema_price_no_older_than`.

Both `publish_time` values are stored in TAI64 format after conversion from UNIX: [3](#0-2) [4](#0-3) 

The Fuel `timestamp()` builtin also returns TAI64. If the Pythnet EMA `publish_time` (TAI64) is even 1 second ahead of the Fuel block's TAI64 timestamp, `current_time - price.publish_time` wraps to approximately `u64::MAX`, which is far greater than any `time_period`, causing the `require` to revert.

### Impact Explanation

Any DeFi protocol on Fuel that calls `ema_price()` or `ema_price_no_older_than()` for a price feed whose stored EMA `publish_time` is ahead of the current Fuel block timestamp will receive a permanent revert. The price feed is effectively bricked for EMA reads until the Fuel block timestamp catches up. Because `update_price_feeds` only stores a new price if `price_feed.price.publish_time > latest_publish_time`: [7](#0-6) 

…once a future-timestamped EMA is stored, it cannot be overwritten by an older (current-time) update, extending the DoS window. Protocols relying on EMA prices for liquidations, collateral valuation, or TWAP-based logic would be unable to function.

---

### Likelihood Explanation

Pythnet and Fuel operate on independent block clocks. Clock drift of even a few seconds between Pythnet's timestamp and the Fuel block timestamp is realistic and has been observed across cross-chain deployments. Any relayer submitting a legitimate, guardian-signed VAA at a moment when the Pythnet EMA timestamp is slightly ahead of the Fuel block timestamp will trigger this condition. No key compromise, governance attack, or forgery is required — only a standard `update_price_feeds` call with a valid VAA.

---

### Recommendation

Apply the same saturating subtraction guard used in `price_no_older_than` to `ema_price_no_older_than`:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

### Proof of Concept

1. A relayer calls `update_price_feeds(update_data)` with a valid Wormhole VAA where the EMA price's `publish_time` (TAI64) is `T_pythnet`, and the current Fuel block timestamp is `T_fuel` where `T_pythnet > T_fuel` by even 1 second.
2. `accumulator_update.update_price_feeds` stores the price feed because `price_feed.price.publish_time > latest_publish_time`.
3. Any subsequent call to `ema_price(price_feed_id)` invokes `ema_price_no_older_than(valid_time_period, price_feed_id)`.
4. Inside, `current_time - price.publish_time` = `T_fuel - T_pythnet` underflows, wrapping to `u64::MAX - (T_pythnet - T_fuel) + 1` ≈ `18446744073709551615`.
5. `require(18446744073709551615 <= valid_time_period_seconds, PythError::OutdatedPrice)` → reverts.
6. All callers of `ema_price()` on this feed are permanently DoS'd until `T_fuel` surpasses `T_pythnet`, and no newer update can overwrite the stored future-timestamped EMA.
