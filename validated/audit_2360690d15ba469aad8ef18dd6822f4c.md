### Title
Tx-Pool Size Accounting Does Not Include Pending `verify_queue` Transactions, Allowing Legitimate Transactions to Be Evicted — (`tx-pool/src/pool.rs`)

---

### Summary

The CKB tx-pool enforces a `max_tx_pool_size` limit (default 180 MB) by checking `pool_map.total_tx_size` in `limit_size()`. However, this accounting snapshot does not include transactions currently sitting in the `verify_queue` awaiting script verification. An unprivileged submitter can flood the verify_queue with transactions that collectively exceed the pool size limit; when they all complete verification and are inserted into `pool_map`, `limit_size()` evicts existing legitimate transactions to restore the limit.

---

### Finding Description

The tx-pool is split into two stages:

1. **`verify_queue`** — a staging area where transactions wait for CKB-VM script verification. It has its own independent size cap (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256 MB`).
2. **`pool_map`** — the main pool of verified transactions. Its size is tracked in `pool_map.total_tx_size` and capped at `config.max_tx_pool_size` (default 180 MB). [1](#0-0) [2](#0-1) 

When a new transaction is submitted, `resumeble_process_tx` runs `pre_check` (which reads `pool_map` under a read lock to check fees and resolve inputs) and then adds the transaction to `verify_queue`. Crucially, **no check is made against `max_tx_pool_size` at this admission point**. The pool-size enforcement only fires in `limit_size()`, which is called *after* a transaction has already been inserted into `pool_map`: [3](#0-2) 

The `limit_size()` loop compares `pool_map.total_tx_size` against `config.max_tx_pool_size`. This total is computed exclusively from entries already in `pool_map`; it has no visibility into the `verify_queue`: [4](#0-3) 

The `TxPoolInfo` struct itself exposes `total_tx_size` (pool_map only) and `verify_queue_size` (count only, no bytes) as separate, unrelated fields, confirming the two accounting domains are never combined for limit enforcement. [5](#0-4) 

---

### Impact Explanation

**Medium.** An attacker who submits a batch of transactions simultaneously can cause the following sequence:

1. Pool is near capacity (e.g., 170 MB of legitimate transactions).
2. Attacker submits many transactions totaling, say, 50 MB. Each individually passes the `verify_queue` admission check (256 MB limit) and `pre_check` (pool_map still reads 170 MB < 180 MB).
3. All 50 MB complete verification concurrently and are inserted into `pool_map`, temporarily bringing it to 220 MB.
4. `limit_size()` fires and evicts 40 MB of transactions — selecting by lowest fee rate first.
5. If the attacker's transactions carry a higher fee rate than some legitimate transactions already in the pool, those legitimate transactions are evicted.

This allows an unprivileged submitter to displace legitimate pending transactions from the pool, delaying or preventing their confirmation. [6](#0-5) 

---

### Likelihood Explanation

**High.** Any node whose pool is near `max_tx_pool_size` and whose `verify_queue` is non-empty is susceptible. Because CKB-VM script verification is CPU-intensive and asynchronous (handled by `VerifyMgr` workers), the verify_queue naturally accumulates multiple in-flight transactions under normal load. An attacker needs only to submit a burst of transactions via the standard `send_transaction` RPC or P2P relay — no privileged access is required. [7](#0-6) 

---

### Recommendation

Before adding a transaction to `verify_queue`, check whether `pool_map.total_tx_size + verify_queue.total_tx_size + new_tx_size > max_tx_pool_size` and reject with `Reject::Full` if so. This mirrors how `verify_queue.add_tx` already guards its own `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` limit: [8](#0-7) 

The combined size of both stages should be the authoritative figure for pool-fullness decisions, so that the `max_tx_pool_size` limit is enforced at admission time rather than only after insertion.

---

### Proof of Concept

1. Configure a node with `max_tx_pool_size = 2000` bytes (as in the existing `SizeLimit` integration test).
2. Fill the pool to ~1800 bytes with legitimate transactions.
3. Simultaneously submit a burst of transactions totaling ~500 bytes via `send_transaction` RPC (each individually < 256 MB verify_queue limit, and pool_map still reads 1800 < 2000 at pre_check time).
4. All burst transactions complete verification and are inserted; `pool_map.total_tx_size` reaches ~2300 bytes.
5. `limit_size()` evicts ~300 bytes of existing transactions (lowest fee rate first).
6. Observe that legitimate transactions with lower fee rates have been evicted despite the pool not being full when they were originally admitted. [9](#0-8) [10](#0-9)

impl VerifyQueue {
    /// Create a new VerifyQueue
    pub(crate) fn new(large_cycle_threshold: u64) -> Self {
        VerifyQueue {
            inner: MultiIndexVerifyEntryMap::default(),
            ready_rx: Arc::new(Notify::new()),
            total_tx_size: 0,
            large_cycle_threshold,
        }
    }

    fn recompute_total_tx_size(&self) -> Option<usize> {
        self.inner.iter().try_fold(0usize, |total, (_, entry)| {
            total.checked_add(entry.inner.tx.data().serialized_size_in_block())
        })
    }

    /// Returns true if the queue contains no txs.
    pub fn is_empty(&self) -> bool {
        self.inner.is_empty()
    }

    pub fn len(&self) -> usize {
        self.inner.len()
    }

    #[cfg(test)]

        self.total_tx_size
    }

        self.total_tx_size = total_tx_size;
    }

    pub fn is_full(&self, add_tx_size: usize) -> bool {
        add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size

    // Remove transactions from the pool until total size <= size_limit.
    // Return a `Reject` for current inserting entry if it's removed
    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }

pub struct SizeLimit;

const MAX_MEM_SIZE_FOR_SIZE_LIMIT: usize = 2000;

impl Spec for SizeLimit {
    fn run(&self, nodes: &mut Vec<Node>) {
        let node = &nodes[0];

        info!("Generate DEFAULT_TX_PROPOSAL_WINDOW block on node");
        node.mine_until_out_bootstrap_period();

        info!("Generate 1 tx on node");
        let mut txs_hash = Vec::new();
        let tx = node.new_transaction_spend_tip_cellbase();
        let mut hash = node.submit_transaction(&tx);
        txs_hash.push(hash.clone());

        let tx_pool_info = node.get_tip_tx_pool_info();
        let one_tx_size = tx_pool_info.total_tx_size.value();
        let one_tx_cycles = tx_pool_info.total_tx_cycles.value();

        info!(
            "one_tx_cycles: {}, one_tx_size: {}",
            one_tx_cycles, one_tx_size
        );

        assert!(MAX_MEM_SIZE_FOR_SIZE_LIMIT as u64 > one_tx_size * 2);

        let max_tx_num = (MAX_MEM_SIZE_FOR_SIZE_LIMIT as u64) / one_tx_size;

        info!("Generate as much as possible txs on : {}", max_tx_num);
        (0..(max_tx_num - 1)).for_each(|_| {
            let tx = node.new_transaction(hash.clone());
            hash = node.rpc_client().send_transaction(tx.data().into());
            txs_hash.push(hash.clone());
            sleep(Duration::from_millis(10));
        });

        info!("The next tx reach size limit");
        let _tx = node.new_transaction(hash);
        node.assert_tx_pool_serialized_size((max_tx_num) * one_tx_size);
        let last =
            node.mine_with_blocking(|template| template.proposals.len() != max_tx_num as usize);
        node.assert_tx_pool_serialized_size(max_tx_num * one_tx_size);
        node.mine_with_blocking(|template| template.number.value() != (last + 1));
        node.mine_with_blocking(|template| template.transactions.len() != max_tx_num as usize);
        node.assert_tx_pool_serialized_size(0);
    }

    fn modify_app_config(&self, config: &mut ckb_app_config::CKBAppConfig) {
        config.tx_pool.max_tx_pool_size = MAX_MEM_SIZE_FOR_SIZE_LIMIT;
        config.tx_pool.min_fee_rate = FeeRate::zero();
    }
