Audit Report

## Title
Unauthenticated `content.from` Field Allows Spoofed `ConnectionRequestDelivered` to Consume Victim's `inflight_requests` and Redirect NAT Traversal — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary
In `ConnectionRequestDeliveredProcess::execute`, the branch that identifies the local node as the hole-punch initiator is gated solely on `content.from == self_peer_id`. Because `content.from` is taken directly from the attacker-supplied message and is never cross-checked against the authenticated peer ID of the sending session (`self.peer`), any connected peer can spoof `from = victim_peer_id`, `route = []`, and attacker-controlled `listen_addrs` to unconditionally drain the victim's `inflight_requests` entry and redirect NAT traversal to attacker-controlled addresses. This enables a persistent, low-cost DoS on the victim's hole-punching capability.

## Finding Description
**Root cause:** At line 151 of `connection_request_delivered.rs`, `execute()` compares `self_peer_id` (the local node's own peer ID) against `content.from` (a field freely set by the remote sender). There is no check that `content.from` equals the authenticated peer ID associated with `self.peer` (the actual sending session).

**Exploit path:**
1. Attacker establishes a standard P2P connection to the victim (session `attacker_sid`).
2. Victim's `notify()` loop broadcasts a `ConnectionRequest` for `target_peer_id` via `filter_broadcast` (lines 225–234 of `mod.rs`) and inserts `target_peer_id → timestamp` into `inflight_requests` (lines 239–242 of `mod.rs`).
3. Attacker crafts a `ConnectionRequestDelivered` with `from = victim_local_peer_id`, `to = target_peer_id`, `route = []`, `listen_addrs = [attacker_ip:port]`.
4. Message passes all guards: `listen_addrs` non-empty check (line 125), `route` length check (line 130), and `forward_rate_limiter` check (lines 134–145) — the key `(victim_peer_id, target_peer_id, item_id)` is new.
5. `content.route.last()` returns `None` (line 147). `self_peer_id == content.from` → `else` branch entered (line 154).
6. `inflight_requests.remove(&content.to)` (line 160) consumes the entry.
7. `respond_sync` sends a sync message back to `self.peer` (the attacker's session, line 228).
8. `try_nat_traversal(ttl, content.listen_addrs)` (line 171) spawns a task that opens raw TCP connections toward the attacker-controlled address for up to 30 seconds.
9. Any subsequent legitimate `ConnectionRequestDelivered` for the same `target_peer_id` hits the `None` arm at line 175 and is silently ignored.

**Why existing guards fail:**
- The `forward_rate_limiter` (lines 134–145) is keyed on `(content.from, content.to, msg_item_id)`. Since the spoofed `(victim_peer_id, target_peer_id)` tuple is new, the rate limiter does not block the first message — which is all that is needed to consume the entry.
- The `listen_addrs` validation (lines 57–63) only enforces that any embedded peer ID matches `content.to`; the IP and port are fully attacker-controlled.
- There is no check anywhere in `execute()` that `content.from` matches the peer ID of the actual sending session. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation
**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single attacker with one standard P2P connection can cancel every hole-punch the victim initiates by sending one spoofed message per `inflight_requests` entry per 2-minute `HOLE_PUNCHING_INTERVAL`. Applied simultaneously to many nodes (each requiring only one connection), this prevents NAT-ed nodes from establishing outbound connections via the hole-punching mechanism, reduces peer diversity across the network, and degrades overall network connectivity. Additionally, each attack causes the victim to waste file descriptors and connection-pool slots on outbound TCP connections to attacker-controlled addresses for up to 30 seconds per attempt. [4](#0-3) [5](#0-4) 

## Likelihood Explanation
- Requires only a standard, unprivileged P2P connection to the victim — no special role or key.
- The victim's local peer ID is public, exchanged during the identify handshake.
- The `to_peer_id` values in `inflight_requests` are directly observable from gossip `ConnectionRequest` broadcasts (lines 225–234 of `mod.rs`).
- The attack requires sending one well-formed message per `inflight_requests` entry; it is trivially scriptable and repeatable every 2 minutes when the victim's `notify()` loop fires.
- The rate limiter does not prevent the attack because each entry is consumed on the first hit, and the rate limiter key is per `(from, to, item_id)` tuple. [6](#0-5) 

## Recommendation
Before entering the "I am the initiator" branch, validate that `content.from` matches the authenticated peer ID of the actual sending session. In the `None` arm, resolve the peer ID for `self.peer` from the peer registry and reject any message where `content.from == self_peer_id` but the sender's authenticated peer ID does not also equal `self_peer_id`:

```rust
// In the None arm, before the self_peer_id comparison:
let sender_peer_id = self.protocol.network_state
    .peer_registry.read()
    .get_key_by_peer_id_or_session(self.peer); // resolve session → peer ID

if content.from == *self_peer_id {
    if sender_peer_id.as_ref() != Some(self_peer_id) {
        return StatusCode::Ignore.with_context("spoofed from field");
    }
    // ... proceed with initiator branch
}
```

Alternatively, enforce globally that `content.from` always equals the authenticated peer ID of the sending session for the entire message, rejecting any `ConnectionRequestDelivered` where `content.from` does not match the sender's authenticated peer ID. [7](#0-6)

        let tasks = remote_addrs

            .filter_map(|listen_addr| match find_type(&listen_addr) {
                TransportType::Tcp => {
                    if listen_addr
                        .iter()
                        .any(|p| matches!(p, Protocol::Ip4(_) | Protocol::Ip6(_)))
                    {
                        Some(Box::pin(try_nat_traversal(self.bind_addr, listen_addr)))
                    } else {
                        None
                    }
                }
                TransportType::Memory
                | TransportType::Onion
                | TransportType::Ws
                | TransportType::Wss
                | TransportType::Tls => None,
            })
            .collect::<Vec<_>>();

        if tasks.is_empty() {
            return;

        debug!("start NAT traversal");

                    .raw_session(
                        stream,
                        addr,
                        RawSessionInfo::outbound(TargetProtocol::Single(
                            SupportProtocols::Identify.protocol_id(),
                        )),
                    )
                    .await;
            }
        });
