Audit Report

## Title
Peer Store Permanently Blocked via Uniform /16-Subnet Address Flooding — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
`check_purge` in `PeerStore` uses a strict `> 4` threshold when selecting eviction candidates in its second eviction pass. An attacker who fills the addr store with exactly 4 connectable addresses per /16 subnet across 4096 distinct subnets (4096 × 4 = 16384 = `ADDR_COUNT_LIMIT`) causes every group to fail the threshold check, leaving `candidate_peers` empty and returning `Err(EvictionFailed)`. Every subsequent `add_addr` call then permanently fails, blocking the node from learning about any new peers via discovery.

## Finding Description
`check_purge` is gated by `self.addr_manager.count() < ADDR_COUNT_LIMIT` at line 328 — it only runs when the store is full. [1](#0-0) 

The first eviction pass removes addresses where `is_connectable` returns `false`. Fresh addresses injected via `add_addr` are created with `last_connected_at_ms=0` and `attempts_count=0`, so neither the `>= ADDR_MAX_RETRIES` (3) nor `>= ADDR_MAX_FAILURES` (10) condition is met — all injected addresses survive the first pass. [2](#0-1) 

The second eviction pass groups addresses by /16 network group (first two IPv4 octets), sorts by descending group size, takes the top `len/2` groups, and evicts 2 random peers **only if `addrs.len() > 4`**: [3](#0-2) 

With exactly 4 addresses per group, `4 > 4` is `false` for every group. `flat_map` yields `None` for all 2048 examined groups, `candidate_peers` is empty, and `Err(PeerStoreError::EvictionFailed)` is returned. Since `add_addr` propagates this error via `self.check_purge()?`, every future `add_addr` call fails permanently. [4](#0-3) 

The /16 grouping is confirmed: [5](#0-4) 

`AddrManager.add` deduplicates only by exact `Multiaddr`, so 4096 × 4 = 16384 distinct addresses across 4096 /16 subnets are all accepted as unique entries, filling the store to `ADDR_COUNT_LIMIT`. [6](#0-5) 

## Impact Explanation
Once the store is poisoned, the node can no longer learn about or connect to new peers via discovery. As existing connections drop through normal peer churn, they cannot be replenished. The node progressively loses all network connectivity and becomes isolated — effectively non-functional as a network participant. This matches the **High** impact class: a vulnerability that can easily crash/isolate a CKB node, achievable with minimal attacker resources.

## Likelihood Explanation
A single malicious peer connected via the discovery protocol can execute this attack. The attacker sends ~17 `Nodes` messages (each up to `MAX_ADDR_TO_SEND = 1000` addresses) containing synthetic IPv4 addresses spread across 4096 /16 subnets. IPv4 has 65536 possible /16 subnets; 4096 is only 6.25% of the space. No address reachability verification is performed before insertion. No PoW, no privileged access, no key material is required. The attack is repeatable and deterministic. [7](#0-6) 

## Recommendation
Replace the strict `> 4` threshold with `>= 4` at `peer_store_impl.rs` line 378 so groups of exactly 4 are also eligible for eviction:
```rust
if addrs.len() >= 4 {   // was: > 4
```
Additionally:
- Enforce a per-/16-subnet insertion cap in `add_addr` (e.g., max 4 per group) to prevent uniform-distribution flooding at insertion time.
- Consider returning `Ok(())` (silently dropping) instead of propagating `EvictionFailed`, so a full store degrades gracefully rather than hard-erroring all callers.

## Proof of Concept
```rust
#[test]
fn test_eviction_failed_with_uniform_groups() {
    let mut peer_store = PeerStore::default();
    // Fill 4096 /16 subnets × 4 addresses = 16384 = ADDR_COUNT_LIMIT
    for subnet in 0u16..4096 {
        let a = (subnet >> 8) as u8;
        let b = (subnet & 0xff) as u8;
        for host in 1u8..=4 {
            let addr: Multiaddr = format!("/ip4/{}.{}.1.{}/tcp/8115", a, b, host)
                .parse().unwrap();
            peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
        }
    }
    // Store is now at ADDR_COUNT_LIMIT with 4096 groups of exactly 4
    // Next add_addr must trigger check_purge → EvictionFailed
    let new_addr: Multiaddr = "/ip4/200.200.200.200/tcp/8115".parse().unwrap();
    let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
    assert!(result.is_err(), "Expected EvictionFailed but got Ok");
}
```
