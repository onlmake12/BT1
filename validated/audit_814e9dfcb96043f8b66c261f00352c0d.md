Audit Report

## Title
Unverified Message-Body Peer IDs Allow Spoofed `forward_rate_limiter` Token Exhaustion, Causing Hole-Punching Relay DoS — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

The `HolePunching` protocol's `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` values parsed directly from the wire message, with no check that the actual TCP session sender matches `content.from`. Any connected peer can craft `ConnectionRequest` or `ConnectionRequestDelivered` messages with arbitrary `from`/`to` peer IDs, consuming the 1 req/sec forwarding token for any peer pair. Legitimate hole-punching requests between those peers are then silently dropped with `TooManyRequests`, causing targeted NAT traversal failures.

## Finding Description

Two distinct rate limiters exist in `HolePunching`:

- `rate_limiter: RateLimiter<(PeerIndex, u32)>` — keyed by the transport-layer `session_id`. Correctly bounds per-sender throughput.
- `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` — keyed by `(content.from, content.to, msg_item_id)` from the **message body**. Vulnerable.

The outer check in `received()` uses the authenticated `session_id`:

```rust
// mod.rs L95-107
if self.rate_limiter.check_key(&(session_id, msg.item_id())).is_err() { ... }
```

This correctly limits an attacker to 30 messages/second per session. However, inside `ConnectionRequestProcess::execute()`, the `forward_rate_limiter` is checked using unverified payload fields:

```rust
// connection_request.rs L132-143
if self.protocol.forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
    .is_err() { ... }
```

The same pattern appears in `ConnectionRequestDeliveredProcess::execute()`:

```rust
// connection_request_delivered.rs L134-145
if self.protocol.forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
    .is_err() { ... }
```

The `forward_rate_limiter` is configured at 1 req/sec per key:

```rust
// mod.rs L254-257
let quota = governor::Quota::per_second(std::num::NonZeroU32::new(1).unwrap());
let forward_rate_limiter = RateLimiter::hashmap(quota);
```

Because `content.from` and `content.to` are never cross-checked against the actual session's peer identity, an attacker sends `ConnectionRequest { from=peer_A, to=peer_B }` to consume the forwarding token for `(peer_A, peer_B, ...)`. When peer A legitimately sends the same message, the relay node returns `TooManyRequests` and drops it silently.

A secondary consequence exists in `ConnectionRequestDeliveredProcess::execute()`: when `content.route` is empty and `content.from` equals the local node's own peer ID (which is publicly advertised), the code unconditionally calls:

```rust
// connection_request_delivered.rs L160
let request_start = self.protocol.inflight_requests.remove(&content.to);
```

An attacker who sets `from=local_peer_id` and `to=victim_peer_id` destroys the in-flight request record, causing the legitimate response to be silently ignored as `"the request is not in flight"`.

Additionally, `HashMapStateStore` backing `forward_rate_limiter` grows with each unique `(from, to)` pair introduced. `retain_recent()` is only called on peer disconnect, so a long-lived attacker session continuously introducing new spoofed pairs causes unbounded memory growth.

## Impact Explanation

An attacker with a single connected session can simultaneously block hole-punching relay for up to 30 distinct `(peer_A, peer_B)` pairs (bounded by the outer 30 req/sec limiter). Peer IDs are publicly advertised via the discovery protocol. Affected nodes behind NAT silently fail to establish connections through the victim relay node. At scale — targeting multiple relay nodes — this degrades NAT traversal network-wide, preventing NAT-ed nodes from joining the CKB P2P network. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** The cost to the attacker is maintaining 1 spoofed message per second per targeted pair from a single connection.

## Likelihood Explanation

Any unprivileged peer that can establish a P2P connection to a CKB node can trigger this. No keys, hashpower, or special privileges are required. Peer IDs are public. The attacker needs only to maintain 1 spoofed message/second per targeted pair, well within the 30 req/sec outer budget. The hole-punching protocol is active on nodes with `reuse_port_on_linux` enabled or NAT traversal configured.

## Recommendation

Before dispatching to `ConnectionRequestProcess` or `ConnectionRequestDeliveredProcess`, resolve the actual peer ID of the session from the peer registry and pass it alongside `session_id`. Inside both `execute()` methods, reject the message if `content.from` does not match the authenticated session peer ID. Alternatively, key `forward_rate_limiter` on `(actual_session_peer_id, content.to, msg_item_id)` instead of the unverified `content.from`, consistent with how `rate_limiter` uses `session_id`. For the secondary issue, guard the `inflight_requests.remove` path by verifying the message arrived from a session whose peer ID matches `content.to` (the relay node that delivered the response), not just by comparing `content.from` to the local peer ID.

## Proof of Concept

```
Setup: Relay node R has hole-punching active. Peer A (behind NAT) is attempting to reach peer B through R.

1. Attacker E connects to R (establishes a valid P2P session).

2. Attacker sends: ConnectionRequest { from=peer_A, to=peer_B, listen_addrs=[valid], max_hops=6, route=[] }
   → R's outer rate_limiter passes (session E, 1 of 30 budget).
   → R's forward_rate_limiter consumes the 1 req/sec token for (peer_A, peer_B, ConnectionRequest_id).

3. Peer A sends its legitimate: ConnectionRequest { from=peer_A, to=peer_B, ... } to R.
   → R's forward_rate_limiter.check_key((peer_A, peer_B, ...)) returns Err (rate exceeded).
   → R returns TooManyRequests, drops the message silently.
   → Peer B never receives the hole-punching request; NAT traversal fails.

4. Attacker repeats step 2 once per second to maintain the DoS indefinitely.
   With 30 req/sec budget, attacker simultaneously blocks 30 distinct (from, to) pairs.

Secondary PoC (inflight_requests destruction):
5. Attacker sends: ConnectionRequestDelivered { from=R's_peer_id, to=peer_A, route=[], ... }
   → forward_rate_limiter passes (new key).
   → content.route is empty, content.from == local_peer_id → enters else branch.
   → inflight_requests.remove(&peer_A) destroys R's legitimate in-flight record for peer_A.
   → When real response arrives, it is dropped as "the request is not in flight".
