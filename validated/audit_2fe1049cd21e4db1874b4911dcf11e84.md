### Title
Stale `parents` set after `remove_entry_and_descendants` causes `get_by_id_checked` panic in `_record_ancestors` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `check_and_record_ancestors`, when a cell-dep-conflict eviction removes tx P **and its descendant D** via `remove_entry_and_descendants(P)`, only `parents.remove(next_id)` (line 620) is called — removing P but leaving D in `parents`. The subsequent call to `_record_ancestors` with this stale `parents` set causes `get_by_id_checked(D)` to panic with `"inconsistent pool"`, crashing the node.

---

### Finding Description

**`get_tx_ancenstors`** (lines 517–554) builds three sets:

- `parents`: all in-pool txs directly referenced by the new tx's inputs or cell_deps
- `cell_ref_parents`: subset of `parents` — in-pool txs whose cell_dep outpoints are being consumed by the new tx's inputs
- `ancestors`: transitive closure of `parents` [1](#0-0) 

Cell-dep txs (lines 541–547) are added to `parents` but **not** to `cell_ref_parents`. Only txs found via `edges.deps` (lines 531–533) enter `cell_ref_parents`.

**`check_and_record_ancestors`** (lines 588–640) enters the eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It iterates `evict_candidates` (filtered from `cell_ref_parents`) and calls `remove_entry_and_descendants(next_id)`: [2](#0-1) 

`remove_entry_and_descendants(P)` removes P **and all of P's descendants** (including D) from both `entries` and `links`: [3](#0-2) 

But line 620 only calls `parents.remove(next_id)` — removing P, not D. D remains in `parents` as a ghost reference.

After the loop, `parents` is passed to `calc_relation_ids` to recompute `ancestors`: [4](#0-3) 

Inside `calc_relation_ids`, D is in `stage`. Since D was removed from `links.inner`, `self.inner.get(&D)` returns `None` — no traversal occurs, but D is still inserted into `relation_ids` (i.e., `ancestors`): [5](#0-4) 

Then `_record_ancestors` iterates `ancestors` and calls `get_by_id_checked` on every entry — including D: [6](#0-5) 

Since D is no longer in `entries`, `get_by_id(D)` returns `None`, and the `.expect("inconsistent pool")` fires: [7](#0-6) 

---

### Impact Explanation

A Rust `expect` panic in the tx-pool service thread crashes the CKB node process. The attacker achieves **remote node crash** with no privileges beyond the ability to submit transactions via RPC or P2P.

---

### Likelihood Explanation

The attacker needs to submit exactly 3 crafted transactions in sequence:

1. **P**: a tx that uses some confirmed on-chain outpoint X as a cell_dep.
2. **D**: a tx that spends P's output (making D a descendant of P in the pool).
3. **N** (the trigger tx): a tx that:
   - Has an input consuming X (the same outpoint P uses as cell_dep → P enters `cell_ref_parents`)
   - Has a cell_dep referencing D's output (→ D enters `parents` but not `cell_ref_parents`)
   - Has enough ancestors to exceed `max_ancestors_count` by exactly 1 (so the eviction branch is taken and one eviction of P suffices)

All three transactions are valid CKB transactions. No PoW, no privileged access, no leaked keys are required. The pool's `max_ancestors_count` is a known constant, making the threshold trivially calculable.

---

### Recommendation

In the eviction loop, after `remove_entry_and_descendants(next_id)`, also remove all descendants of `next_id` from `parents`. The `removed` vec returned by `remove_entry_and_descendants` already contains all evicted entries:

```rust
let removed = self.remove_entry_and_descendants(next_id);
// Remove ALL evicted IDs from parents, not just next_id
for evicted_entry in &removed {
    parents.remove(&evicted_entry.proposal_short_id());
}
ancestors_count = ancestors_count.saturating_sub(removed.len());
evicted.extend(removed);
```

Note also that `ancestors_count = ancestors_count.saturating_sub(1)` (line 619) is itself incorrect — it should subtract the number of actually removed entries, not just 1. [8](#0-7)
