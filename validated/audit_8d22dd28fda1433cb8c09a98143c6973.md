### Title
Stale `descendants_*` Accounting in `remove_entry_and_descendants` Leaves Ancestor Eviction Keys Inflated - (File: tx-pool/src/component/pool_map.rs)

### Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link entries for the entire removed cluster before calling `remove_entry` on each member. Because `update_ancestors_index_key` relies on the live link graph to find which pool entries to update, it silently finds zero ancestors and skips all `sub_descendant_weight` calls. Ancestors of the removed cluster that remain in the pool are left with permanently inflated `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` fields. These stale fields feed directly into each ancestor's `EvictKey`, causing the pool's eviction logic to treat those ancestors as more valuable than they actually are, so they survive pool-full eviction rounds they should lose.

### Finding Description

`PoolMap` maintains per-entry aggregate fields for the entire descendant sub-tree: [1](#0-0) 

These fields are used to compute the `EvictKey` for each entry: [2](#0-1) 

When a single entry is removed via `remove_entry`, the update order is correct — `update_ancestors_index_key` is called **before** `remove_entry_links`, so `calc_ancestors` can still traverse the live graph and call `sub_descendant_weight` on every surviving ancestor: [3](#0-2) 

`update_ancestors_index_key` works by calling `self.links.calc_ancestors(&child.proposal_short_id())` and then modifying each ancestor's `descendants_*` fields and recomputing its `evict_key`: [4](#0-3) 

However, `remove_entry_and_descendants` breaks this invariant. It first strips **all** link entries for the entire removed cluster (root + every descendant), and only then calls `remove_entry` on each member: [5](#0-4) 

The comment acknowledges the intent is to suppress `update_descendants_index_key` (since all descendants are being removed anyway), but the same pre-removal of links also suppresses `update_ancestors_index_key`. When `remove_entry` is called for the root transaction, `calc_ancestors(root_tx.id)` returns an empty set because the root's own link entry was already deleted. No ancestor in the pool receives `sub_descendant_weight`, so their `descendants_*` fields remain permanently inflated.

The stale `descendants_*` values propagate into each ancestor's `EvictKey` via `as_evict_key()`: [6](#0-5) 

`limit_size` uses `next_evict_entry` to pick the lowest-`EvictKey` entry for eviction when `total_tx_size > max_tx_pool_size`: [7](#0-6) 

An ancestor with an inflated `EvictKey` will not be selected for eviction when it should be, causing a legitimately higher-fee incoming transaction to be rejected instead.

### Impact Explanation

When `remove_entry_and_descendants` is called (pool-full eviction, header-dep invalidation, RBF replacement, explicit `remove_transaction` RPC), every surviving ancestor of the removed cluster retains stale `descendants_*` fields. The `EvictKey` for those ancestors is computed from the inflated `descendants_feerate`, making them appear more valuable than they are. Subsequent pool-full eviction rounds will skip those ancestors and instead evict legitimate transactions, causing valid high-fee transactions submitted by honest users to be rejected with `Reject::Full`. The stale state persists until the ancestor itself is removed or the pool is cleared.

### Likelihood Explanation

`remove_entry_and_descendants` is called in multiple common code paths: pool-size enforcement (`limit_size`), header-dep conflict resolution (`resolve_conflict_header_dep`), ancestor-count enforcement during admission (`check_and_record_ancestors`), detached-proposal cleanup (`remove_by_detached_proposal`), and the `remove_transaction` RPC. Any unprivileged tx-pool submitter can trigger the stale-state condition by:

1. Submitting a parent transaction `tx_A` (low fee) and a child `tx_B` (high fee) via the relay protocol.
2. Submitting a conflicting replacement for `tx_B` (RBF), which triggers `remove_entry_and_descendants` on `tx_B`.
3. `tx_A`'s `descendants_*` remain inflated; `tx_A` now survives pool-full eviction rounds it should lose.
4. Legitimate high-fee transactions are rejected with `Reject::Full`.

No privileged access, key material, or majority hash power is required.

### Recommendation

In `remove_entry_and_descendants`, update the ancestors of the root entry **before** stripping any links. Specifically, call `update_ancestors_index_key(&root_entry, EntryOp::Remove)` while the link graph is still intact, then proceed with the bulk link removal and per-entry `remove_entry` calls. Alternatively, collect the set of surviving ancestors before removing links and apply `sub_descendant_weight` to each of them explicitly after the cluster is removed.

### Proof of Concept

```
1. Pool contains tx_A (fee=100, size=200) with no ancestors/descendants.
2. Submit tx_B (fee=1000, size=200) spending tx_A's output.
   → tx_A.descendants_count=2, tx_A.descendants_fee=1100, tx_A.evict_key reflects high descendants_feerate.
3. Submit tx_C conflicting with tx_B (higher fee, RBF).
   → remove_entry_and_descendants(tx_B.id) is called.
   → remove_entry_links called for {tx_B.id} first.
   → remove_entry(tx_B.id) calls update_ancestors_index_key(tx_B, Remove).
   → calc_ancestors(tx_B.id) returns {} because tx_B's link entry is gone.
   → tx_A.descendants_count remains 2, tx_A.descendants_fee remains 1100. (BUG)
4. Pool fills up. limit_size calls next_evict_entry.
   → tx_A.evict_key is inflated; tx_A is not selected for eviction.
   → A legitimate high-fee tx_D is rejected with Reject::Full instead.
``` [5](#0-4) [4](#0-3) [7](#0-6)

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {

                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp

                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()

                        ret = Some(reject.clone());

                }
            }
        }
