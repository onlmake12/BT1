### Title
Unauthorized Rent Theft via `close_signature_set` ‚Äî Missing Funder Authority Check - (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

### Summary
`close_signature_set` allows **any** signer to drain the rent lamports from a `SignatureSet` account they did not fund, by passing a victim's `PostedVaaV1` and its associated `SignatureSet` as inputs while naming themselves as `sol_destination`.

### Finding Description
The `CloseSignatureSet` account context enforces only two constraints on the `signature_set` account:

1. `has_one = signature_set` ‚Äî verifies `posted_vaa.info.signature_set == signature_set.key()` [1](#0-0) 
2. `close = sol_destination` ‚Äî transfers all lamports from `signature_set` to `sol_destination` [2](#0-1) 

`sol_destination` is constrained only to be a `Signer` ‚Äî there is no check that it is the original funder/creator of the `signature_set` account. [3](#0-2) 

In the legacy flow, a relayer creates a `SignatureSet` using a fresh keypair (paying rent), then calls `post_vaa` which stores that keypair's pubkey in `posted_vaa.info.signature_set`. [4](#0-3) 

Because `PostedVaaV1` accounts are PDAs (publicly discoverable on-chain), an attacker can enumerate them, read the `signature_set` field, and call `close_signature_set` with themselves as `sol_destination` to steal the rent.

### Impact Explanation
Every relayer who used the legacy `verify_signatures` + `post_vaa` flow deposited rent into a `SignatureSet` keypair account. That rent is permanently lost to any attacker who calls `close_signature_set` first with `sol_destination = attacker`. The `SignatureSet` account is closed and the lamports are transferred to the attacker. The original relayer has no recourse.

### Likelihood Explanation
The attack requires no privileges, no leaked keys, and no governance access. All inputs (`posted_vaa`, `signature_set`) are publicly readable on-chain. The attacker only needs to sign a transaction with their own wallet. Any `PostedVaaV1` account whose `signature_set` field is non-zero (i.e., created via the legacy flow, not `post_vaa_v1`) is a valid target. This is trivially automatable.

### Recommendation
Add an authority/funder field to `SignatureSet` (or record the payer in `PostedVaaV1`) and enforce `sol_destination == original_funder` in `CloseSignatureSet`. Alternatively, require `sol_destination` to be the same signer who funded the `posted_vaa` account (the `payer` in `PostVaa`). [5](#0-4) 

### Proof of Concept
```
1. Relayer R calls verify_signatures (creates SignatureSet at pubkey S, paying ~0.002 SOL rent)
2. Relayer R calls post_vaa ‚Üí PostedVaaV1 PDA P is created with P.signature_set = S
3. Attacker A scans chain, finds P, reads P.signature_set = S
4. Attacker A calls close_signature_set {
       sol_destination: A,   // A signs the tx ‚Äî only requirement
       posted_vaa: P,        // has_one = signature_set passes: P.signature_set == S ‚úì
       signature_set: S      // close = sol_destination ‚Üí S's lamports ‚Üí A
   }
5. S is closed; A receives ~0.002 SOL; R's rent is permanently stolen
```

The invariant "only the account that originally funded `signature_set` can recover its lamports" is broken because `close_signature_set` never checks who funded `signature_set`. [6](#0-5)
