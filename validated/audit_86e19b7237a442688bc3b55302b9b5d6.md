### Title
Permissionless Keeper Can Inflate Gas Price to Drain Subscription Balances — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler` contract pays keepers based on `tx.gasprice`, which is fully controlled by the keeper (the `msg.sender`). Because anyone can call `updatePriceFeeds` without registration, a malicious keeper can set an artificially high gas price to inflate the `GAS_OVERHEAD * tx.gasprice` profit component, draining subscription balances far faster than intended and at the direct expense of subscription managers.

---

### Finding Description

The `_processFeesAndPayKeeper` function computes the keeper's reimbursement as:

```
gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice
keeperSpecificFee = singleUpdateKeeperFeeInWei * numPriceIds
totalKeeperFee = gasCost + keeperSpecificFee
``` [1](#0-0) 

`GAS_OVERHEAD` is a fixed constant of `30000` gas units: [2](#0-1) 

The keeper's **net profit** from a single update is:

```
profit = GAS_OVERHEAD * tx.gasprice + keeperSpecificFee
```

Because `tx.gasprice` is set by the transaction sender (the keeper), a malicious keeper can set an arbitrarily high gas price. The keeper pays `actualGasUsed * tx.gasprice` in real ETH, but receives `(actualGasUsed + GAS_OVERHEAD) * tx.gasprice + keeperSpecificFee` from the subscription balance. The overhead component `GAS_OVERHEAD * tx.gasprice` is pure profit that scales linearly with the chosen gas price.

There is no cap on `tx.gasprice` in the fee calculation, and no minimum-balance guard prevents a single update from extracting an outsized fee: [3](#0-2) 

The keeper role is explicitly permissionless — no registration is required: [4](#0-3) 

---

### Impact Explanation

A malicious keeper sets `tx.gasprice` to, e.g., 10,000 gwei instead of a normal 1 gwei. With `GAS_OVERHEAD = 30,000`:

- **Normal profit per update:** `30,000 × 1 gwei = 0.00003 ETH`
- **Malicious profit per update:** `30,000 × 10,000 gwei = 0.3 ETH`

The keeper pays the inflated gas cost for actual gas used, but the overhead profit is extracted entirely from the subscription's deposited balance. A subscription with 1 ETH can be drained in ~3 updates. The subscription becomes inactive prematurely, causing downstream protocols that rely on it to consume stale prices — enabling the same keeper (or a colluding party) to profit from liquidations in those downstream protocols, exactly mirroring the Hubble Exchange attack pattern. [5](#0-4) 

---

### Likelihood Explanation

- **Entry barrier:** Zero. No stake, no registration, no governance approval. Any EOA can call `updatePriceFeeds`.
- **Trigger condition:** The keeper only needs to wait for a valid heartbeat or deviation trigger, which is a normal operational event.
- **Cost to attacker:** The attacker pays `actualGasUsed × tx.gasprice` in real ETH, but recovers it from the subscription plus the overhead profit. The attack is net-profitable as long as the subscription balance exceeds the gas cost of the actual execution.
- **Detection difficulty:** High. The transaction appears as a normal keeper update on-chain; only the gas price is anomalous.

---

### Recommendation

1. **Cap `tx.gasprice` in the fee calculation** to a governance-configurable `maxGasPrice` parameter, e.g.:
   ```solidity
   uint256 effectiveGasPrice = Math.min(tx.gasprice, _state.maxGasPrice);
   uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
   ```
2. **Alternatively**, decouple the overhead reimbursement from `tx.gasprice` by using a fixed overhead fee in wei rather than `GAS_OVERHEAD * tx.gasprice`.
3. Consider requiring keepers to register and post a bond, enabling slashing for abusive gas-price submissions.

---

### Proof of Concept

1. Deploy `Scheduler` on a testnet. Create a subscription with `updateOnHeartbeat = true`, `heartbeatSeconds = 60`, funded with 1 ETH.
2. Wait 60 seconds for the heartbeat trigger to become valid.
3. Fetch a valid signed price update from Hermes.
4. Call `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = 10_000 gwei` (set via `vm.txGasPrice(10_000e9)` in Foundry).
5. Observe that the subscription balance decreases by `actualGasUsed * 10_000 gwei + GAS_OVERHEAD * 10_000 gwei + keeperSpecificFee`, where the `GAS_OVERHEAD * 10_000 gwei = 0.3 ETH` component is pure profit to the keeper.
6. Repeat 3 times to fully drain a 1 ETH subscription, deactivating it and causing downstream consumers to read stale prices. [6](#0-5) [2](#0-1)

```text
    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
```

- Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`. The main goal of making this component a permissionless network rather a set of permissioned nodes is to enhance reliability for the feeds -- if one provider fails, others should be available to service the subscriptions. Subsequent attempts for the same update interval will revert since we verify the update criteria on-chain. By only allowing updates when they are needed, we keep costs predictable for the users.
