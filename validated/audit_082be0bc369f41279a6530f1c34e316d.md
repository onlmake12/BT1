### Title
`DaoScriptSizeVerifier` Called Unconditionally in Tx-Pool Without `rfc0044_active` Guard — (`tx-pool/src/util.rs`)

### Summary

In `tx-pool/src/util.rs`, `DaoScriptSizeVerifier` is invoked unconditionally (no `rfc0044_active` epoch guard) inside `verify_rtx`. In contrast, the canonical block verifier in `verification/contextual/src/contextual_block_verifier.rs` gates the same verifier behind `consensus.rfc0044_active(self.parent.epoch().number())`. This asymmetry means the tx-pool enforces a stricter rule than the block verifier, causing valid DAO withdrawal transactions to be permanently rejected from the mempool even though they would be accepted in a block.

### Finding Description

**Tx-pool path (`tx-pool/src/util.rs`, lines 101–131):**

```rust

            .and_then(|result| {
```

**Block verifier path (`verification/contextual/src/contextual_block_verifier.rs`, lines 444–453):**

```rust
.and_then(|result| {
    if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {
        DaoScriptSizeVerifier::new(...).verify()?;   // ← correctly gated
    }
    Ok(result)
})
```

`DaoScriptSizeVerifier::verify()` itself contains a secondary block-number guard (`starting_block_limiting_dao_withdrawing_lock()`), which is a different activation axis (block number) from `rfc0044_active` (epoch number). During any window where `starting_block_limiting_dao_withdrawing_lock()` has been passed but `rfc0044_active` is still false, the tx-pool will reject a DAO deposit-to-withdrawal transaction with mismatched lock script sizes with `DaoLockSizeMismatch`, while the block verifier would accept the same transaction.

### Impact Explanation

A user who holds a Nervos DAO deposit cell and attempts to submit a withdrawal transaction (phase 1 → phase 2) during the activation window described above will have their transaction permanently rejected by every honest node's tx-pool. The transaction cannot be broadcast or mined through normal mempool flow. The user's DAO funds are effectively locked: they cannot complete the withdrawal until either the epoch-based `rfc0044_active` activates (at which point the block verifier also enforces the rule and the transaction would be invalid on-chain too) or they find an out-of-band path to get the transaction mined. This is a direct loss-of-access to funds for DAO depositors.

### Likelihood Explanation

The scenario requires:
1. The chain has passed `starting_block_limiting_dao_withdrawing_lock()` (block-number threshold).
2. `rfc0044_active` is not yet true (epoch threshold not yet reached).
3. A user submits a DAO withdrawal with a lock script whose size differs from the deposit cell's lock script.

Conditions 1 and 2 define a real, finite activation window that exists whenever the two thresholds are not perfectly co-incident. Condition 3 is a normal user action (e.g., using a multisig or custom lock for withdrawal). Any transaction sender (unprivileged RPC caller via `send_transaction`) can trigger this rejection path.

### Recommendation

Mirror the block verifier's guard in `verify_rtx`. Wrap the `DaoScriptSizeVerifier` call with the same `rfc0044_active` epoch check:

```rust
.and_then(|result| {
    if snapshot.cloned_consensus().rfc0044_active(/* current epoch */) {
        DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
            .verify()?;
    }
    Ok(result)
})
```

This makes the tx-pool admission rule consistent with the block validation rule, preventing valid transactions from being incorrectly rejected.

### Proof of Concept

1. Configure a chain where `starting_block_limiting_dao_withdrawing_lock` = block 1000 and `rfc0044_active` activates at epoch 10.
2. Mine to block 1001 (past the block threshold) while still in epoch 9 (before the epoch threshold).
3. Construct a DAO withdrawal transaction (phase 1 → phase 2) where the output lock script has a different size than the deposit cell's lock script.
4. Submit via `send_transaction` RPC.
5. **Observed:** tx-pool rejects with `DaoLockSizeMismatch`.
6. **Expected:** tx-pool accepts the transaction (block verifier would also accept it since `rfc0044_active` is false).

**Root cause:** `verify_rtx` in `tx-pool/src/util.rs` calls `DaoScriptSizeVerifier::verify()` unconditionally, while `BlockTxsVerifier::verify` in `contextual_block_verifier.rs` correctly gates it behind `rfc0044_active`. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
```
