### Title
Funds Sent to Permanent Subscription via `addFunds` Are Permanently Locked — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary
`Scheduler.addFunds()` accepts ETH for any active subscription without checking whether the subscription is permanent. Because `withdrawFunds()` unconditionally reverts for permanent subscriptions, any ETH deposited into a permanent subscription via `addFunds` is permanently locked with no recovery path.

---

### Finding Description
`addFunds` performs only two guards before crediting `msg.value`:

1. `if (!params.isActive)` — permanent subscriptions are always active (deactivation is blocked), so this check always passes.
2. `if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT)` — only rejects oversized single deposits; it does **not** reject the deposit itself. [1](#0-0) 

After crediting the balance, `withdrawFunds` unconditionally reverts for permanent subscriptions: [2](#0-1) 

There is no admin escape hatch, no `emergencyWithdraw`, and no governance path to recover funds from a permanent subscription. The `addFunds` function has **no access control** — any address may call it on any subscription ID. [3](#0-2) 

The interface NatDoc for `addFunds` contains no warning that deposits to permanent subscriptions are irrecoverable, and the `withdrawFunds` NatDoc only mentions the minimum-balance restriction, not the permanent-subscription restriction. [4](#0-3) 

---

### Impact Explanation
Any ETH sent to a permanent subscription via `addFunds` is permanently locked in the contract. The caller — who may be a third party unaware that the subscription is permanent — loses their ETH with no recourse. The `balanceInWei` counter increases, but neither the depositor nor the subscription manager can ever retrieve those funds.

---

### Likelihood Explanation
`addFunds` is a permissionless, payable function callable by any EOA or contract. A subscription manager who makes a subscription permanent and later (or a third party who) calls `addFunds` on it will permanently lose the deposited ETH. The scenario is realistic: integrators or automation scripts that top up subscriptions by ID without re-checking the `isPermanent` flag on every call will silently lock funds. The existing test suite explicitly demonstrates that "anyone can add funds" to a permanent subscription and that withdrawal is blocked. [5](#0-4) 

---

### Recommendation
Add a permanent-subscription guard at the top of `addFunds`, mirroring the guard already present in `withdrawFunds` and `updateSubscription`:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    SchedulerStructs.SubscriptionParams storage params = _state
        .subscriptionParams[subscriptionId];

    if (params.isPermanent) {
        revert SchedulerErrors.CannotUpdatePermanentSubscription();
    }
    if (!params.isActive) {
        revert SchedulerErrors.InactiveSubscription();
    }
    // ... rest of function
}
```

This mirrors the existing pattern in `withdrawFunds` and `updateSubscription` and prevents ETH from being irreversibly deposited into a subscription from which it can never be withdrawn.

---

### Proof of Concept

```solidity
// 1. Manager creates a subscription and funds it to minimum balance.
uint256 subId = scheduler.createSubscription{value: minBalance}(params);

// 2. Manager makes the subscription permanent (irreversible).
params.isPermanent = true;
scheduler.updateSubscription(subId, params);

// 3. A third party (or the manager themselves) adds funds — succeeds with no warning.
address thirdParty = address(0xBEEF);
vm.deal(thirdParty, 1 ether);
vm.prank(thirdParty);
scheduler.addFunds{value: 1 ether}(subId);  // accepted, no revert

// 4. Third party tries to recover their ETH — permanently blocked.
vm.prank(thirdParty);
scheduler.withdrawFunds(subId, 1 ether);
// Reverts: CannotUpdatePermanentSubscription
// ETH is permanently locked.

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
