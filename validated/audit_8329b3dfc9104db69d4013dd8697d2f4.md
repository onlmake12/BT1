Audit Report

## Title
Unbounded Orphan BFS Traversal in `process_orphan_tx` Occupies Verify Worker Without Iteration Cap — (`tx-pool/src/process.rs`)

## Summary

`process_orphan_tx` in `tx-pool/src/process.rs` performs an unbounded BFS over the orphan pool with no per-call iteration limit. For each orphan whose declared cycle count is at or below `max_tx_verify_cycles`, it calls `_process_tx` with `command_rx: None` (the non-pausable synchronous path), running full CKB-VM script verification inline. Because the verify worker calls `after_process` (which calls `process_orphan_tx`) synchronously before returning to its main loop, a single root transaction submission can cause one verify worker to process up to 100 orphans sequentially without yielding back to the queue, delaying all other transactions pending in the verify queue for that worker.

## Finding Description

The BFS loop in `process_orphan_tx` has no iteration counter or per-call bound:

```rust
while let Some(previous) = orphan_queue.pop_front() {
    let orphans = self.find_orphan_by_previous(&previous).await;
    for orphan in orphans.into_iter() {
        if orphan.cycle > self.tx_pool_config.max_tx_verify_cycles {
            // enqueue to verify queue (async path)
        } else if let Some((ret, _snapshot)) = self
            ._process_tx(orphan.tx.clone(), Some(orphan.cycle), None)  // synchronous, no pause
            .await
        { ... orphan_queue.push_back(orphan.tx); }
    }
}

The branch condition at line 598 routes orphans with `declared_cycle > max_tx_verify_cycles` to the async verify queue. Orphans at or below the threshold are processed inline via `_process_tx(..., None)`, where `None` means no `command_rx` is passed, disabling the pause/suspend mechanism used by the normal verify worker path. [2](#0-1) 

`_process_tx` with `command_rx: None` still calls `verify_rtx` and runs full CKB-VM script verification to completion with no interruption point: [3](#0-2) 

`after_process` calls `process_orphan_tx` synchronously (awaits it) on every successful transaction, both for remote and local submissions: [4](#0-3) [5](#0-4) 

The verify worker's `process_inner` loop calls `after_process` and awaits it before returning to pick the next entry from the verify queue: [6](#0-5) 

The orphan pool is capped at 100 entries, which is the maximum number of orphans that can be drained in a single `process_orphan_tx` invocation: [7](#0-6) [8](#0-7) 

The declared cycle count is validated against actual verified cycles after `verify_rtx` completes, so the attacker must declare correct cycles — but they control the transaction scripts and can craft transactions with any cycle count up to `max_tx_verify_cycles` (70,000,000 by default): [9](#0-8) 

## Impact Explanation

One verify worker is occupied processing up to 100 orphan transactions sequentially, each requiring a full CKB-VM script verification run of up to 70,000,000 cycles, before it can return to the verify queue. The default worker count is `max(num_cpus * 3/4, 1)`: [10](#0-9) 

On a typical 4-core node this yields 3 workers. A single attack occupies one worker for the entire orphan chain traversal. The attack is repeatable: after the chain is consumed, the attacker can re-fill the orphan pool and submit another root transaction. Sustained repetition degrades effective verify throughput, constituting **CKB network congestion with few costs** — matching the High impact class (10001–15000 points).

## Likelihood Explanation

The attacker needs only a P2P peer connection and a small amount of CKB to own live cells. The attack steps are:
1. Craft a linear chain of 100 transactions where `tx[i]` spends output 0 of `tx[i-1]`, each with declared cycles ≤ `max_tx_verify_cycles`.
2. Submit `tx[1]` through `tx[99]` via P2P relay; each is accepted into the orphan pool (inputs not yet live). The pool fills to 100 entries.
3. Submit `tx[0]` (the root, spending a live cell). It enters the verify queue normally.
4. The verify worker verifies `tx[0]`, calls `after_process` → `process_orphan_tx`, which cascades through all 100 orphans synchronously. [11](#0-10) 

No special privilege or majority hashpower is required. The attack is repeatable with the same funds (outputs of the chain become new inputs for the next round), making the marginal cost per repetition negligible.

## Recommendation

Introduce a per-call iteration limit to `process_orphan_tx`. Process at most `N` orphans per invocation (e.g., `N = 10`) and re-schedule remaining work asynchronously (e.g., by re-enqueuing unprocessed orphans into the verify queue or posting a deferred task). This prevents a single invocation from monopolizing a verify worker. The high-cycle branch already demonstrates the correct pattern: orphans above the cycle threshold are enqueued to the verify queue rather than processed inline. [12](#0-11) 

## Proof of Concept

1. Connect to a CKB node as a P2P peer with access to at least one live cell.
2. Construct a chain: `tx[0]` spends the live cell; `tx[i]` spends output 0 of `tx[i-1]` for `i = 1..99`. Each transaction's lock script executes exactly `K` cycles where `K ≤ max_tx_verify_cycles`; declare `K` as the cycle count.
3. Submit `tx[1]` through `tx[99]` via `send_transaction` P2P relay. Confirm each is stored in the orphan pool (node logs `add_orphan_tx`). Pool reaches 100 entries.
4. Submit `tx[0]`. Observe via node metrics (`ckb_tx_pool_sync_process`) or timing that the verify worker is occupied for the full orphan chain traversal before returning to the verify queue.
5. Simultaneously submit an unrelated transaction and measure its time-to-acceptance; it will be delayed until the orphan loop completes.
6. Repeat from step 2 to demonstrate sustained throughput degradation.

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
