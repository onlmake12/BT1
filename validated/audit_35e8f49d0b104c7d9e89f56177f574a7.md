### Title
Cross-chain Replay of Lazer Price Update Payloads Lacking Chain ID Binding — (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` verifies an ECDSA signature over a payload that contains no chain ID. Because the same trusted signers are registered on every EVM deployment of `PythLazer` and the payload format is identical across all EVM chains, a valid signed price update captured on one EVM chain can be submitted verbatim to `verifyUpdate` on any other EVM chain and will pass all on-chain checks.

---

### Finding Description

`verifyUpdate` in `PythLazer.sol` recovers the signer from the raw ECDSA signature over `keccak256(payload)`:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The payload is structured as `PAYLOAD_FORMAT_MAGIC (4B) | timestamp (8B) | channel (1B) | feedsLen (1B) | feed data`. There is no chain ID anywhere in this structure:

```solidity
uint32 FORMAT_MAGIC = 2479346549;
// ...
timestamp = _readBytes8(update, pos);
pos += 8;
channel = PythLazerStructs.Channel(_readBytes1(update, pos));
pos += 1;
feedsLen = uint8(_readBytes1(update, pos));
``` [2](#0-1) 

The outer EVM wrapper magic is also a single hardcoded constant shared across all EVM chains:

```solidity
uint32 EVM_FORMAT_MAGIC = 706910618;
``` [3](#0-2) 

The deployment script itself explicitly acknowledges the absence of cross-chain replay protection in the salt:

```solidity
salt = bytes32(
    abi.encodePacked(
        deployer,   // Deployment protection by our deployer
        uint8(0),   // No crosschain replay protection
        seed
    )
);
``` [4](#0-3) 

This means `PythLazer` is intentionally deployed to the **same deterministic address** on every EVM chain, and the same trusted signers are registered on all of them. A signed update blob is therefore structurally identical and cryptographically valid on every EVM chain simultaneously.

---

### Impact Explanation

An attacker who captures a valid signed Lazer price update from chain A (e.g., Ethereum) can submit it to `verifyUpdate` on chain B (e.g., Arbitrum, BNB Chain, Base). The call will succeed because:

1. The EVM format magic matches on all chains.
2. The payload hash is identical.
3. The recovered signer is the same trusted Pyth Lazer key, which is registered on all chains.

`verifyUpdate` performs no timestamp freshness check and no chain ID check. It returns `(payload, signer)` to the consumer. Consumer contracts that do not independently validate the payload timestamp will accept a stale price as current. This enables an attacker to:

- Feed a stale price (e.g., pre-crash BTC/USD) to a lending protocol on chain B, enabling under-collateralised borrows or preventing liquidations.
- Arbitrage price-sensitive DeFi contracts across chains using a single captured update.

The `verifyUpdate` function is the sole on-chain trust boundary advertised to integrators; the README and developer hub documentation present it as the complete verification step. [5](#0-4) 

---

### Likelihood Explanation

`PythLazer` is already deployed on multiple EVM mainnets. The same trusted signer keys are active on all of them. Any user or bot that calls `verifyUpdate` on one chain receives a valid `(payload, signer)` pair that is immediately replayable on every other chain. No privileged access, key compromise, or oracle manipulation is required — only the ability to observe a transaction on one chain and submit it on another.

---

### Recommendation

Bind the signed payload to the target chain by including `block.chainid` (or the Wormhole chain ID) in the data that is hashed and signed. For example, the Pyth Lazer server should sign:

```
keccak256(abi.encodePacked(chainId, payload))
```

and `verifyUpdate` should reconstruct the same hash before recovering the signer. Alternatively, adopt EIP-712 domain separation, which includes `chainId` and `verifyingContract` by design.

---

### Proof of Concept

```
1. On chain A (Ethereum), call PythLazer.verifyUpdate{value: fee}(updateBlob).
   → Returns (payload, signer). Transaction is public on-chain.

2. Copy the identical `updateBlob` bytes from the Ethereum calldata.

3. On chain B (Arbitrum), call PythLazer.verifyUpdate{value: fee}(updateBlob)
   with the same bytes.
   → verifyUpdate recovers the same signer address.
   → isValidSigner(signer) returns true (same key registered on Arbitrum).
   → Returns (payload, signer) — verification passes.

4. Any consumer contract on chain B that calls verifyUpdate and does not
   check payload timestamp now processes the replayed (potentially stale)
   price data as if it were freshly signed for chain B.
```

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
