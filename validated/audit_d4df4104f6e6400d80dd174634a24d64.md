Audit Report

## Title
VerifyQueue Enforces Hardcoded 256 MB Ceiling Independent of Operator-Configured `max_tx_pool_size`, Enabling Memory Exhaustion Beyond Intended Budget — (`tx-pool/src/component/verify_queue.rs`)

## Summary
The CKB tx-pool two-stage admission pipeline places remote P2P relay transactions into a `VerifyQueue` staging area before full script verification. This queue enforces a hardcoded 256 MB ceiling (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`) that is entirely independent of the operator-configured `max_tx_pool_size`. The main pool's `limit_size()` eviction logic has no awareness of the verify queue, and the queue's byte consumption is not reflected in `TxPoolInfo.total_tx_size`. Any unprivileged P2P peer can flood the verify queue up to 256 MB regardless of the operator's configured memory budget, causing actual node memory consumption to reach `max_tx_pool_size + 256 MB`.

## Finding Description

**Hardcoded verify-queue ceiling:**

`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` is a compile-time constant of 256 MB. The comment in the source explicitly acknowledges it exceeds the default `max_tx_pool_size` of 180 MB: [1](#0-0) 

The `is_full` guard checks only against this hardcoded constant, never against the operator-configured `max_tx_pool_size`: [2](#0-1) 

**Reachable exploit path via P2P relay:**

`submit_remote_tx` calls `resumeble_process_tx_and_notify_full_reject`, which calls `resumeble_process_tx`. This runs only `non_contextual_verify` (size ≤ 512 KB, not cellbase, structurally valid) before calling `enqueue_verify_queue`. No UTXO resolution or fee check occurs before the transaction enters the queue: [3](#0-2) [4](#0-3) 

The local RPC `send_transaction` path (`process_tx` → `_process_tx`) does **not** go through the verify queue — it performs synchronous verification. The verify queue is populated exclusively by P2P relay (`submit_remote_tx`, `notify_tx`) and orphan promotion.

**`limit_size()` never drains the verify queue:**

The main pool's eviction loop only checks `pool_map.total_tx_size` against `config.max_tx_pool_size` and only iterates over `pool_map` entries with statuses `Pending`, `Gap`, and `Proposed`. It has no awareness of the verify queue: [5](#0-4) 

**`total_tx_size` excludes verify-queue entries:**

`TxPoolInfo` reports `total_tx_size` from `pool_map.total_tx_size` only. The verify queue's byte consumption is reported separately as `verify_queue_size` (a count via `.len()`, not bytes) and never drives eviction: [6](#0-5) 

**Consequence — total memory budget is `max_tx_pool_size + 256 MB`:**

If an operator sets `max_tx_pool_size = 10 MB` on a memory-constrained node, the verify queue alone can still consume 256 MB — 25× the intended budget — while `total_tx_size` in monitoring metrics shows near zero, providing no alarm signal.

## Impact Explanation

**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

On memory-constrained nodes (e.g., embedded validators, nodes configured with low `max_tx_pool_size`), the verify queue's 256 MB ceiling causes actual RSS to far exceed the operator's configured limit. This can trigger OOM-kill, halting block production and P2P connectivity. Even on well-provisioned nodes, the verify queue acts as a free 256 MB staging buffer that any P2P peer can fill, degrading verification throughput for legitimate transactions and constituting a sustained resource exhaustion attack.

## Likelihood Explanation

The entry path requires no privilege beyond being a P2P peer. Transactions only need to pass `non_contextual_verify` (structural validity, size ≤ 512 KB, not cellbase) before entering the verify queue — no UTXO resolution or fee check occurs at admission time. An attacker can craft ~512 transactions of maximum size (512 KB each) with large witness data and submit them via P2P relay in rapid succession to saturate the 256 MB queue. The verify manager processes transactions sequentially (one per worker), so a burst of 512 large transactions will occupy the queue for a meaningful window. The attack requires no keys, no hashpower, and no Sybil capability.

## Recommendation

1. Replace the hardcoded `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` in `tx-pool/src/component/verify_queue.rs` with a value derived from `TxPoolConfig::max_tx_pool_size` (e.g., `max_tx_pool_size` or `max_tx_pool_size * 1.5`), so the verify queue respects the operator's memory budget.
2. Include `verify_queue.total_tx_size` in the combined size check so that `limit_size()` can account for in-flight transactions when deciding whether to accept new ones.
3. Enforce a per-peer rate limit or per-peer byte quota on verify-queue submissions to bound the rate at which any single P2P source can fill the staging area.
4. Expose `verify_queue_total_tx_size` (bytes, not count) in `TxPoolInfo` so operators can monitor actual memory pressure from the queue.

## Proof of Concept

1. Configure a CKB node with `max_tx_pool_size = 10_000_000` (10 MB) in `ckb.toml`.
2. Connect a malicious P2P peer to the node.
3. Generate ~512 structurally valid transactions each close to 512 KB (e.g., large witness data; inputs may reference non-existent UTXOs — they only need to pass `non_contextual_verify` to enter the queue).
4. Submit all 512 transactions via P2P relay (`RelayTransactions` message) in rapid succession.
5. Observe via `tx_pool_info` RPC that `verify_queue_size` grows to ~512 while `total_tx_size` remains near 0 (transactions are still queued, not yet processed).
6. The node's RSS memory grows by ~256 MB — 25× the configured `max_tx_pool_size` — while the operator-visible `total_tx_size` metric shows no alarm signal.
7. On a node with ≤300 MB available memory, this triggers OOM-kill.

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
