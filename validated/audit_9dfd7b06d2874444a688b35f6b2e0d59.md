Audit Report

## Title
IBD Stall via TOCTOU Race Between `n_sync_started` Increment and Peer State Update in `start_sync_headers` — (File: sync/src/synchronizer/mod.rs)

## Summary
In `start_sync_headers`, `n_sync_started` is atomically incremented before the selected peer's `HeadersSyncState` is transitioned to `Started`. If a peer disconnects in the narrow window between these two operations, `Peers::disconnected` removes the peer while it is still in `SyncProtocolConnected` state, skips the counter decrement (because `sync_started()` returns `false`), and `start_sync` is never called because `get_mut` returns `None`. The counter is permanently stuck at 1 with no peer in `Started` state, and the IBD guard prevents any future header sync from starting.

## Finding Description
**Step 1 — counter increment** (`sync/src/synchronizer/mod.rs` lines 672–682):
```rust
if self.shared().state().n_sync_started()
    .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
        if ibd && x != 0 { None } else { Some(x + 1) }
    })
    .is_err()
{ break; }
``` [1](#0-0) 

**Step 2 — peer state update** (`sync/src/synchronizer/mod.rs` lines 683–687):
```rust
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
}
``` [2](#0-1) 

No lock spans both operations. Between Step 1 and Step 2, `Peers::disconnected` can run concurrently. It removes the peer from `self.state` and only decrements `n_sync_started` if `peer_state.sync_started()` returns `true`: [3](#0-2) 

`sync_started()` delegates to `chain_sync.started()`, which only returns `true` for `HeadersSyncState::Started`: [4](#0-3) [5](#0-4) 

Because the peer is still in `SyncProtocolConnected` at disconnect time, the decrement is skipped. Back in `start_sync_headers`, `get_mut(&peer)` returns `None` (peer already removed), so `start_sync` is never called and the peer never reaches `HeadersSyncState::Started`: [6](#0-5) 

`n_sync_started` is now 1 with zero peers in `Started` state. All three decrement sites (`disconnected`, `suspend_sync`, `tip_synced`) guard behind `sync_started()`: [7](#0-6) 

Since no peer ever reaches `Started`, none of these paths can recover the counter. The IBD guard then permanently blocks all future sync — every subsequent `start_sync_headers` call evaluates `ibd && 1 != 0 → None → is_err() → break`: [8](#0-7) 

## Impact Explanation
A single successful race permanently stalls the node in IBD: it cannot start header sync with any peer, cannot download blocks, and cannot advance its chain tip. The node is functionally non-operational until restarted. This matches **High** impact (10001–15000 points): "Vulnerabilities which could easily crash a CKB node" — a node permanently unable to sync is operationally equivalent to a crash from the operator's perspective.

## Likelihood Explanation
The race window is narrow (a few instructions between `fetch_update` and `get_mut`), but CKB's multi-threaded tokio runtime runs the network disconnect handler and the sync timer on genuinely concurrent threads. A malicious peer requires only an unprivileged TCP connection — no keys, stake, or special role. By repeatedly connecting and disconnecting at the moment the sync timer fires, an attacker increases the probability of hitting the window. A single successful race is sufficient for permanent stall; the attack is repeatable at negligible cost.

## Recommendation
Add an `else` branch to decrement the counter when `get_mut` returns `None`:

```rust
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
} else {
    // Peer disconnected between counter increment and state update; undo the increment.
    self.shared().state().n_sync_started().fetch_sub(1, Ordering::AcqRel);
    continue;
}
```

Alternatively, hold the DashMap entry lock across both the counter increment and the state update so that `disconnected` cannot interleave.

## Proof of Concept
1. Connect to a CKB node in IBD (`is_initial_block_download() == true`).
2. Complete the sync protocol handshake so the peer is inserted into `Peers::state` with `HeadersSyncState::SyncProtocolConnected`.
3. Wait for the sync timer to fire `start_sync_headers`. The peer satisfies `can_start_sync`; `n_sync_started.fetch_update` succeeds, incrementing from 0 to 1.
4. Immediately disconnect. `Peers::disconnected` removes the entry; `sync_started()` is `false`; `n_sync_started` is **not** decremented.
5. `get_mut(&peer)` in `start_sync_headers` returns `None`; `start_sync` is never called.
6. `n_sync_started` is now 1 with no peer in `Started` state.
7. Every subsequent `start_sync_headers` call evaluates `ibd && 1 != 0 → None → is_err() → break` and exits without starting sync.
8. The node remains in IBD indefinitely until the process is restarted.
