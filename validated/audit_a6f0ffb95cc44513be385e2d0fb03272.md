### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Arbitrary Pending Transactions from the Pool — (`rpc/src/module/pool.rs`)

### Summary

The `remove_transaction` JSON-RPC endpoint accepts a caller-supplied `tx_hash` and unconditionally removes the matching transaction (and all its descendants) from the transaction pool. There is no ownership check, no caller identity, and no authentication layer on the RPC server. Any process that can reach the RPC port — including any local process on the same machine, or any remote client if the operator has exposed the port — can enumerate all pending transactions via `get_raw_tx_pool` and then silently evict transactions submitted by other users.

### Finding Description

**Root cause — no ownership check in `remove_transaction`:**

`PoolRpcImpl::remove_transaction` at `rpc/src/module/pool.rs:662-669` takes a caller-supplied `tx_hash: H256` and forwards it directly to `tx_pool.remove_local_tx(tx_hash.into())` with zero authorization logic:

```rust

    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)

}

At no point in this chain is the caller's identity checked against the transaction's submitter or the input cell owners.

### Impact Explanation

Any caller that can reach the RPC port can:

1. Call `get_raw_tx_pool` to enumerate all pending transaction hashes.
2. Call `remove_transaction(tx_hash)` for any hash to silently evict that transaction and all its descendants from the pool.

Concrete consequences:
- **Transaction censorship**: a competing application or malicious co-tenant on the same machine can continuously evict a victim's transactions, preventing them from ever being confirmed.
- **Financial loss in time-sensitive operations**: arbitrage, liquidation, or DAO withdrawal transactions that are evicted at a critical moment cause the user to miss the window, resulting in direct financial loss.
- **Cascade eviction**: because `remove_entry_and_descendants` is called, a single call targeting a parent transaction removes all child transactions that depend on it, amplifying the damage.

The `Pool` module is enabled by default in the production config (`resource/ckb.toml:190`), so `remove_transaction` is always reachable. [8](#0-7) 

### Likelihood Explanation

- **Local attacker**: any process running as the same OS user (or any user on the machine if the socket is world-readable) can call the RPC with a plain HTTP POST. No privilege is required.
- **Remote attacker**: node operators commonly expose the RPC port for dApp backends, exchanges, and wallets. The documentation warns against this but does not enforce it. `CorsLayer::permissive()` means browser-based cross-origin requests are also accepted.
- **No secret required**: the tx_hash is a public identifier obtainable from `get_raw_tx_pool` or from P2P relay observation.
- **Repeatable**: the attacker can loop continuously, re-evicting any resubmitted transaction.

### Recommendation

1. **Require caller proof of relationship**: before removing a transaction, verify that the caller can demonstrate ownership of at least one input cell (e.g., by requiring a signed challenge), or restrict `remove_transaction` to a separate privileged RPC module not enabled by default.
2. **Add an authentication layer to the RPC server**: introduce an optional bearer-token or HTTP Basic Auth middleware in `RpcServer::start_server` so that operators can gate destructive pool-mutation methods.
3. **Separate read-only and mutating modules**: move `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` into a dedicated `Admin` or `Debug` module that is disabled by default, analogous to how `IntegrationTest` and `Debug` modules are already gated.

### Proof of Concept

```bash
# Step 1: enumerate all pending tx hashes
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}' \
  | jq '.result[]'
# returns e.g. "0xabc123..."

# Step 2: evict a victim's transaction (no auth required)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"remove_transaction","params":["0xabc123..."]}'
# returns {"result": true}
# Transaction is now gone from the pool; victim must resubmit.
```

The attacker needs no credentials, no private key, and no relationship to the transaction. The call succeeds for any valid `tx_hash` present in the pool.
