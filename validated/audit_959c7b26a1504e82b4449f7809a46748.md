### Title
Missing Index Consistency Check Between `transactions_proof` and `witnesses_proof` in `verify_transaction_and_witness_proof` - (File: `rpc/src/module/chain.rs`)

---

### Summary

`verify_transaction_and_witness_proof` accepts a `TransactionAndWitnessProof` containing two independent CBMT proofs — `transactions_proof` and `witnesses_proof` — each with their own `indices`. There is no check that both proofs cover the **same set of transaction positions**. An RPC caller can supply mismatched indices, causing the function to return transaction hashes that were never actually paired with their witnesses during verification.

---

### Finding Description

`TransactionAndWitnessProof` carries two independent `MerkleProof` fields:

```rust
pub struct TransactionAndWitnessProof {
    pub block_hash: H256,
    pub transactions_proof: MerkleProof,   // indices into tx hash CBMT
    pub witnesses_proof: MerkleProof,      // indices into witness hash CBMT
}
``` [1](#0-0) 

The verification logic in `verify_transaction_and_witness_proof` independently verifies each proof against the block's full CBMT roots:

1. `witnesses_merkle_proof` is applied to `block.tx_witness_hashes()` to compute `witnesses_proof_root`.
2. `transactions_merkle_proof` is applied to `block.tx_hashes()` to compute `raw_transactions_root`.
3. The combined root is checked: `block.transactions_root() == merkle_root([raw_transactions_root, witnesses_proof_root])`.
4. The function returns `tx_hashes` — the leaves retrieved by `transactions_merkle_proof`. [2](#0-1) 

At no point is there a check that `transactions_proof.indices` and `witnesses_proof.indices` are the same set. The two proofs are verified independently against the block's global CBMT roots, not against each other.

When `get_transaction_and_witness_proof` **generates** a proof, it correctly uses the same `leaf_indices` for both trees:

```rust
witnesses_proof: CBMT::build_merkle_proof(block.tx_witness_hashes(), &leaf_indices)
``` [3](#0-2) 

But `verify_transaction_and_witness_proof` accepts arbitrary caller-supplied indices and never enforces this invariant.

---

### Impact Explanation

The semantic contract of `verify_transaction_and_witness_proof` is: *"these transaction hashes are in the block, and their witnesses are also in the block at the corresponding positions."* Without the index consistency check, the function only proves:

- Some transactions at positions `transactions_proof.indices` are in the block.
- Some witnesses at positions `witnesses_proof.indices` are in the block.

These two sets can be completely disjoint. The returned transaction hashes are presented as "witness-verified" but the witnesses that were actually checked belong to different transactions.

Any downstream consumer (e.g., a light client, bridge, or application) that calls this RPC to confirm that a specific transaction has a specific witness (e.g., a particular signature or lock script witness) can be deceived. An attacker can prove transaction T1 is in the block while substituting the witness proof of a different transaction T2, causing the consumer to incorrectly conclude that T1's witness was verified.

---

### Likelihood Explanation

The attack requires:
1. A block with at least two transactions (trivially satisfied on mainnet).
2. An RPC caller who crafts a `TransactionAndWitnessProof` with mismatched `transactions_proof.indices` and `witnesses_proof.indices`.
3. A downstream consumer that trusts the returned transaction hashes as having been witness-verified.

The RPC endpoint is publicly accessible to any node operator or application. The crafted proof is trivially constructable from any valid block. Likelihood is **medium** — the impact depends on how consumers interpret the return value, but the missing check is a clear correctness flaw in the proof API.

---

### Recommendation

Add an explicit check at the start of `verify_transaction_and_witness_proof` that the sorted indices of `transactions_proof` and `witnesses_proof` are identical before proceeding with verification:

```rust
let mut tx_indices: Vec<u32> = tx_proof.transactions_proof.indices
    .iter().map(|i| i.value()).collect();
let mut wit_indices: Vec<u32> = tx_proof.witnesses_proof.indices
    .iter().map(|i| i.value()).collect();
tx_indices.sort();
wit_indices.sort();
if tx_indices != wit_indices {
    return Err(RPCError::invalid_params(
        "transactions_proof and witnesses_proof must cover the same transaction indices"
    ));
}
```

This mirrors the invariant enforced during proof generation in `get_transaction_and_witness_proof`, where both proofs are built from the same `leaf_indices`. [4](#0-3) 

---

### Proof of Concept

Consider a block with two transactions at indices 0 and 1. An attacker calls:

```json
{
  "method": "verify_transaction_and_witness_proof",
  "params": [{
    "block_hash": "<valid_block_hash>",
    "transactions_proof": {
      "indices": ["0x1"],
      "lemmas": ["<valid_lemma_for_tx_at_index_1>"]
    },
    "witnesses_proof": {
      "indices": ["0x0"],
      "lemmas": ["<valid_lemma_for_witness_at_index_0>"]
    }
  }]
}
```

Both proofs are individually valid against the block's CBMT roots. The function passes the root check at line 2052–2056 and returns the hash of transaction at index 1. However, the witness that was verified belongs to the transaction at index 0. A consumer relying on this RPC to confirm that transaction-at-index-1 has a specific witness is deceived. [5](#0-4)

        &self,
        tx_proof: TransactionAndWitnessProof,

        snapshot

                        .lemmas

                        .map(|lemma| lemma.into())
                        .collect(),
                );
                let witnesses_merkle_proof = MerkleProof::new(
                    tx_proof

                        .indices
                        .into_iter()
                        .map(|index| index.value())
                        .collect(),
                    tx_proof
                        .witnesses_proof
                        .lemmas

                                        }
                                    },
                                )
                            })
                    })
                    .ok_or_else(|| {
                        RPCError::invalid_params("Invalid transaction_and_witness proof")
                    })
