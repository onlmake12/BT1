### Title
Permanent User Fund Lock in Echo Contract Due to Missing Request Cancellation Mechanism — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
The `Echo` contract's `executeCallback` function credits the provider fee and clears the request **before** executing the consumer callback. If `executeCallback` reverts mid-execution (e.g., arithmetic underflow when `pythFee > req.fee + msg.value`), or if the provider permanently fails to call `executeCallback`, user funds paid as request fees are permanently locked in the contract with no rescue or cancellation path available to the user.

### Finding Description

In `Echo.executeCallback`, the fee accounting and request clearing occur before the callback invocation:

```solidity
// Line 161-164
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // ← can underflow/revert

clearRequest(sequenceNumber);                        // ← request cleared before callback

// Lines 176-201: callback executed with try/catch
try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)
```

The code itself acknowledges this danger with a TODO comment at lines 155–156:

> `// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.`
> `// If executeCallback can revert, then funds can be permanently locked in the contract.`

Two concrete locking scenarios exist:

**Scenario A — Arithmetic underflow causing permanent revert:**
- At request time, `req.fee = msg.value - _state.pythFeeInWei` is stored.
- At execution time, `pythFee = pyth.getUpdateFee(updateData)` is fetched live from the Pyth oracle.
- If the Pyth oracle's fee has increased since the request was made, `(req.fee + msg.value) - pythFee` underflows and reverts.
- The request remains active but the user has no way to cancel it and recover their funds.

**Scenario B — Provider never calls `executeCallback`:**
- User pays fees via `requestPriceUpdatesWithCallback`.
- Provider goes offline, becomes malicious, or finds the request unprofitable (e.g., Pyth fee increased beyond what the user paid).
- No `cancelRequest`, `refundRequest`, or timeout-based rescue function exists anywhere in the contract.
- User funds (`req.fee`) are permanently locked in the Echo contract.

There is no function in `Echo.sol` or `IEcho.sol` that allows a user to cancel a pending request and recover their funds.

### Impact Explanation

User funds paid as fees to the Echo contract are permanently locked with no recovery path. The amount locked equals the full fee paid per request (`req.fee = msg.value - pythFeeInWei`). With no cancellation or rescue mechanism, affected users suffer a complete and irrecoverable loss of the ETH they paid.

### Likelihood Explanation

- **Scenario A** is triggered whenever the Pyth oracle's fee increases between request and execution time. Pyth fees are governance-controlled and can change. Any provider calling `executeCallback` with `msg.value = 0` after a fee increase will trigger the underflow.
- **Scenario B** is triggered whenever a provider fails to fulfill a request. Providers are permissionlessly registered (`registerProvider`) with no bonding or slashing mechanism, making abandonment low-cost.
- Both scenarios are reachable by any unprivileged user who calls `requestPriceUpdatesWithCallback`.

### Recommendation

1. **Add a request cancellation function** that allows the original requester to cancel their request and recover `req.fee` after a configurable timeout period (e.g., after `publishTime + exclusivityPeriod + buffer`).
2. **Guard against underflow** in the fee accounting line by using `saturatingSub` or by requiring `msg.value` to cover any Pyth fee shortfall before proceeding.
3. **Add an admin rescue function** for ERC-20 or native tokens that become stranded in the contract due to edge-case execution failures.

### Proof of Concept

**Scenario A (arithmetic underflow):**
1. Pyth oracle fee is 100 wei at request time; `_state.pythFeeInWei = 100`.
2. User calls `requestPriceUpdatesWithCallback{value: 200}(...)`. `req.fee = 200 - 100 = 100`.
3. Pyth oracle fee increases to 150 wei.
4. Provider calls `executeCallback{value: 0}(...)`.
5. `pythFee = pyth.getUpdateFee(updateData) = 150`.
6. `(req.fee + msg.value) - pythFee = (100 + 0) - 150` → underflow → revert.
7. Request remains active; user cannot cancel; funds locked.

**Scenario B (provider abandonment):**
1. User calls `requestPriceUpdatesWithCallback{value: totalFee}(...)`.
2. Provider goes offline or finds the request unprofitable.
3. No timeout, no `cancelRequest`, no refund path exists.
