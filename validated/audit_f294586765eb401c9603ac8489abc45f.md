### Title
Missing `Content-Security-Policy` / `X-Frame-Options` Enables Clickjacking on Staking and Governance Web Apps — (Files: `apps/staking/next.config.js`, `governance/xc_admin/packages/xc_admin_frontend/next.config.js`)

---

### Summary

Every Pyth Network web application in this repository is missing both a `Content-Security-Policy` header (specifically the `frame-ancestors` directive) and an `X-Frame-Options` header. No frame-busting JavaScript is present anywhere in the codebase. This allows any attacker to embed the Pyth Staking app or the xc_admin governance frontend inside a cross-origin `<iframe>` and use transparent overlay layers to trick users into signing unintended wallet transactions.

---

### Finding Description

A repository-wide grep for `Content-Security-Policy`, `frame-ancestors`, `X-Frame-Options`, and `frame-src` returns **zero matches**. A separate grep for frame-busting patterns (`top.location`, `self == top`, `window.top`) also returns **zero matches**.

The staking app's `next.config.js` defines five security headers for all routes:

- `X-XSS-Protection`
- `Referrer-Policy`
- `Strict-Transport-Security`
- `X-Content-Type-Options`
- `Permissions-Policy` [1](#0-0) 

None of these prevent framing. The `Content-Security-Policy: frame-ancestors 'none'` directive and the `X-Frame-Options: DENY` header are both absent.

The same omission exists in `apps/insights/next.config.js`, `apps/entropy-explorer/next.config.js`, `apps/api-reference/next.config.js`, and `apps/developer-hub/next.config.js` — all share the identical header set with no framing protection. [2](#0-1) [3](#0-2) 

The governance frontend (`xc_admin_frontend`) is the most exposed: its `next.config.js` defines **no HTTP security headers whatsoever** — not even the five present in the other apps. [4](#0-3) 

No `_document.tsx` with frame-busting script exists for the staking app (it uses the App Router with `apps/staking/src/app/layout.tsx` which only re-exports `Root` and `metadata`). [5](#0-4) 

The xc_admin frontend does have a `pages/_document.tsx`, but since no frame-busting code appears anywhere in the repository, it does not contain the mitigation either.



---

### Impact Explanation

**Staking app (primary target — unprivileged staking user):**
An attacker embeds `https://staking.pyth.network` in a transparent `<iframe>` on a malicious page. By positioning a fake UI element (e.g., "Claim Airdrop") directly over the staking app's "Stake", "Unstake", or "Claim Rewards" button, the attacker tricks the victim into clicking the underlying button. The victim's connected Solana wallet (Phantom, Solflare, etc.) then presents a legitimate transaction prompt. Because the prompt appears to originate from a real Pyth domain interaction, the user is likely to approve it, resulting in unauthorized staking or unstaking of PYTH tokens.

**xc_admin governance frontend (secondary target — governance participant):**
The same technique applied to the governance frontend could trick a multisig participant into submitting or approving a malicious governance proposal, potentially affecting cross-chain contract configuration across all supported chains.

---

### Likelihood Explanation

The staking app is publicly accessible and wallet-connected. Clickjacking attacks against DeFi staking UIs are well-documented and actively exploited. The absence of both `frame-ancestors` and `X-Frame-Options` means any modern browser will allow the framing without warning. No secondary control (frame-busting JS) exists as a fallback. Likelihood is **high** for the staking app given the low attacker effort required and the direct financial impact on staking users.

---

### Recommendation

1. Add `Content-Security-Policy: frame-ancestors 'none'` and `X-Frame-Options: DENY` to the `headers()` array in every `next.config.js`:

```js
{ key: "Content-Security-Policy", value: "frame-ancestors 'none'" },
{ key: "X-Frame-Options",         value: "DENY" },
```

2. For the `xc_admin_frontend`, add a complete security headers block (it currently has none). [6](#0-5) 

3. Add frame-busting JavaScript to the root layout / `_document.tsx` of each app as a defense-in-depth fallback:

```html
<script>
  if (self === top) {
    document.documentElement.style.display = 'block';
  } else {
    top.location = self.location;
  }
</script>
```

---

### Proof of Concept

1. Host the following HTML on any attacker-controlled domain:

```html
<!DOCTYPE html>
<html>
<head>
  <style>
    iframe { position: absolute; top: 0; left: 0; width: 100%; height: 100%; opacity: 0.0001; z-index: 2; }
    #decoy { position: absolute; top: 300px; left: 200px; z-index: 1; font-size: 24px; cursor: pointer; }
  </style>
</head>
<body>
  <div id="decoy">🎁 Claim your PYTH airdrop!</div>
  <!-- Staking app loads without any framing restriction -->
  <iframe src="https://staking.pyth.network"></iframe>
</body>
</html>
```

2. Position the decoy element over the staking app's "Stake" or "Unstake" button (coordinates determined by inspecting the live app).
3. When the victim clicks the decoy, the click is captured by the invisible iframe, triggering the staking action.
4. The victim's wallet extension presents a legitimate Solana transaction prompt; the user approves it believing it is part of the "airdrop" flow.
5. PYTH tokens are staked or unstaked without the user's informed consent.

The attack requires no smart contract exploit, no privileged access, and no on-chain vulnerability — the missing HTTP header is the sole necessary condition, and it is absent from every Pyth web application in the repository.
