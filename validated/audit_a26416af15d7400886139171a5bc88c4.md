Audit Report

## Title
Unvalidated Remote-Controlled `observed_addr` Stored and Re-Broadcast as Own Listen Address — (`network/src/protocols/identify/mod.rs`, `network/src/network.rs`, `network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
`process_observed()` accepts any attacker-supplied `Multiaddr` from an incoming Identify message and stores it without any reachability check. The stored address is subsequently merged into the node's own advertised listen address list and broadcast to every peer it connects to. Because globally-routable attacker-controlled IPs pass the `is_reachable()` filter applied at send time, and because HolePunching paths apply no filter at all, a single connected peer can inject arbitrary addresses into the network-wide peer store and Discovery propagation chain.

## Finding Description

**Root cause — `process_observed()` performs no validation:**

`process_observed()` passes the raw attacker-supplied `Multiaddr` directly to `IdentifyCallback::add_observed_addr()` with no `is_reachable()` guard: [1](#0-0) 

`add_observed_addr()` only appends the local peer ID if absent, then stores the address unconditionally via `NetworkState::add_observed_addr()`: [2](#0-1) 

`NetworkState::add_observed_addr()` inserts into the `observed_addrs` map with no further checks: [3](#0-2) 

**Contrast with `process_listens()` — the guard that is missing from `process_observed()`:**

The listen-address path correctly applies `is_reachable()` before storing remote peers' addresses: [4](#0-3) 

No equivalent guard exists for the observed-address path.

**Propagation path 1 — Identify `connected()`:**

`local_listen_addrs()` merges raw `observed_addrs` into the advertised list whenever fewer than `MAX_RETURN_LISTEN_ADDRS` (10) public addresses are known: [5](#0-4) 

`connected()` applies `is_reachable()` before sending, which blocks loopback and RFC1918 addresses. However, any valid globally-routable IP (e.g., `/ip4/1.2.3.4/tcp/8114`) passes this filter and is broadcast verbatim to every subsequent peer: [6](#0-5) 

Receiving peers store these addresses in their peer stores via `add_remote_listen_addrs()`, from which the Discovery protocol re-broadcasts them further: [7](#0-6) 

**Propagation path 2 — HolePunching `respond_delivered()` (completely unfiltered):**

`respond_delivered()` in `connection_request.rs` directly chains `observed_addrs` into the HolePunching response with no `is_reachable()` filter at all, leaking even RFC1918/loopback addresses to hole-punching participants: [8](#0-7) 

The same unfiltered pattern appears in `HolePunchingProtocol::notify()`: [9](#0-8) 

**Why existing checks are insufficient:**

The `is_reachable()` filter in `connected()` is applied *after* the poisoned address has already been merged into `local_listen_addrs()`. It only prevents loopback/private addresses from being forwarded via Identify; it does not prevent globally-routable attacker-controlled addresses from being forwarded. The HolePunching paths have no filter at all. `add_observed_addr()` always returns `MisbehaveResult::Continue`, so there is no ban or disconnect. [10](#0-9) 

## Impact Explanation

**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker who connects to a set of CKB nodes and sends crafted `observed_addr` values causes each victim to advertise the attacker's IP as its own listen address. Receiving peers store these addresses and re-broadcast them via the Discovery protocol. The result is network-wide address table pollution: nodes across the network accumulate invalid peer addresses, waste outbound connection slots attempting to reach them, and degrade peer discovery efficiency. At scale, this degrades the connectivity fabric of the CKB P2P network, causing congestion in peer management and slowing block/transaction propagation indirectly through reduced effective peer counts.

## Likelihood Explanation

- Requires only a standard P2P connection — no special privileges, no PoW, no key material.
- The Identify protocol runs on every session open; a single connected peer can poison the victim immediately and silently via the first (and only required) Identify message.
- The `observed_addrs` map is keyed by `SessionId`, so each connected peer injects one address; an attacker maintaining connections to many nodes can poison many simultaneously.
- The attack is silent: no ban, no disconnect, no log warning specific to this path.
- The poisoned address propagates transitively via Discovery, amplifying the effect beyond the directly connected victims.

## Recommendation

1. In `IdentifyCallback::add_observed_addr()` (`network/src/protocols/identify/mod.rs`, line 497), apply `is_reachable()` (and optionally `multiaddr_to_socketaddr()` existence check) before calling `NetworkState::add_observed_addr()`, mirroring the guard already present in `process_listens()`.
2. In `NetworkState::observed_addrs()` (`network/src/network.rs`, line 519), filter out non-reachable addresses before returning them, so all consumers (Identify and HolePunching) are protected uniformly.
3. In `connection_request.rs` `respond_delivered()` and `HolePunchingProtocol::notify()`, apply the same `is_reachable()` filter to `observed_addrs` before including them in HolePunching messages.

## Proof of Concept

**Minimal unit test sketch (no network needed):**

```rust
// 1. Build a NetworkState with no configured public addresses.
let network_state = build_test_network_state(); // public_addrs empty

// 2. Simulate a remote peer sending a crafted observed_addr.
let session_id = SessionId::new(1);
let poison: Multiaddr = "/ip4/1.2.3.4/tcp/8114".parse().unwrap();
network_state.add_observed_addr(session_id, poison.clone());

// 3. Call local_listen_addrs() — observed_addrs are merged because listens.len() < 10.
let mut callback = build_test_identify_callback(network_state.clone());
let addrs = callback.local_listen_addrs();

// 4. The poisoned address IS present in the advertised list.
assert!(addrs.iter().any(|a| a.to_string().contains("1.2.3.4")));

// 5. Simulate connected() filter — is_reachable("1.2.3.4") returns true,
//    so the address passes and is included in the Identify message sent to the next peer.
let filtered: Vec<_> = addrs.iter().filter(|addr| {
    match multiaddr_to_socketaddr(addr) {
        Some(s) => is_reachable(s.ip()),
        None => false,
    }
}).collect();
assert!(filtered.iter().any(|a| a.to_string().contains("1.2.3.4")));
```

**Manual steps:**
1. Connect to any CKB node.
2. Send the initial Identify message with `observed_addr = /ip4/<attacker-ip>/tcp/8114`.
3. Observe that the victim node subsequently advertises `/ip4/<attacker-ip>/tcp/8114` in its own Identify messages to newly connected peers.
4. Confirm those peers add the address to their peer stores and re-broadcast it via Discovery.
