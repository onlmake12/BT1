### Title
Missing Maximum Fee Protection in Entropy Request Functions Allows Provider Front-Running to Extract User Funds - (File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol)

---

### Summary

The Pyth Entropy contract's request functions (`request`, `requestWithCallback`, `requestV2`) accept `msg.value` as payment but expose no `maxFee` parameter for callers to cap the fee they are willing to pay. Because providers can change their fee at any time via `setProviderFee`, and because excess `msg.value` is explicitly **not refunded**, a malicious provider can front-run a user's pending request transaction to inflate their fee and extract the user's entire payment buffer.

---

### Finding Description

In `requestHelper`, the fee check is a one-sided minimum bound only:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The `providerFee` is read at execution time from `providerInfo.feeInWei`, which any registered provider can change at any time:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
``` [2](#0-1) 

The `IEntropyV2` interface documentation explicitly acknowledges both the fee variability and the no-refund policy:

> "Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2()` prior to each invocation (as opposed to hardcoding a value). Further note that **excess value is *not* refunded to the caller**." [3](#0-2) 

This is documented for every variant of `requestV2`: [4](#0-3) [5](#0-4) [6](#0-5) 

None of the four `requestV2` overloads, `request`, or `requestWithCallback` accept a `maxFee` argument: [7](#0-6) 

The same structural gap exists in `Echo.sol`'s `requestPriceUpdatesWithCallback`, where `msg.value - pythFeeInWei` is stored directly as `req.fee` and later credited in full to the provider: [8](#0-7) [9](#0-8) 

The Fortuna keeper service is explicitly designed to **dynamically adjust provider fees** based on prevailing gas prices: [10](#0-9) 

This means fees legitimately fluctuate between blocks, giving integrating contracts a strong practical reason to send a buffer above the quoted minimum fee to avoid reverts — exactly the funds a front-running provider can extract.

---

### Impact Explanation

A malicious registered provider can:

1. Monitor the mempool for pending `requestWithCallback` / `requestV2` transactions targeting their provider address.
2. Read the `msg.value` of the pending transaction.
3. Front-run with `setProviderFee(msg.value - pythFee)`, setting their fee to consume the user's entire payment.
4. The user's transaction executes successfully (since `msg.value == requiredFee`), but the provider accrues `msg.value - pythFee` instead of the originally quoted fee.
5. The user's entire payment buffer is extracted by the provider with no recourse.

The user loses the difference between the fee they expected to pay (from their `getFeeV2()` query) and the inflated fee the provider set. For integrating contracts that routinely send 1.5×–2× the quoted fee as a safety buffer, this can represent a significant and repeatable loss per request.

---

### Likelihood Explanation

- Any registered provider can execute this attack; provider registration is permissionless.
- The attack requires only basic mempool monitoring and a single `setProviderFee` transaction.
- The Fortuna keeper's documented dynamic fee adjustment means users have a legitimate, well-known reason to send buffers — the attack surface is not hypothetical.
- The documentation explicitly warns about fee changes without providing any on-chain mitigation, meaning integrating contracts are expected to handle this risk themselves, but no mechanism is provided to do so.

---

### Recommendation

Add a `maxFee` parameter to all request entry points and revert if the current fee exceeds it:

```solidity
function requestV2(
    address provider,
    bytes32 userRandomNumber,
    uint32 gasLimit,
    uint128 maxFee          // NEW: caller-specified fee cap
) external payable returns (uint64 assignedSequenceNumber) {
    uint128 currentFee = getFeeV2(provider, gasLimit);
    if (currentFee > maxFee) revert EntropyErrors.FeeTooHigh();
    // ... existing logic
}
```

This is the direct analog of a DEX `minAmountOut` / slippage guard: the caller specifies the worst-case fee they accept, and the transaction reverts atomically if the provider has raised their fee beyond that threshold, rather than silently consuming the user's buffer.

---

### Proof of Concept

```
Setup:
  pythFee = 100 wei
  Provider registers with feeInWei = 1000 wei
  getFeeV2(provider) = 1100 wei

Step 1: User (integrating contract) calls getFeeV2() → 1100 wei
Step 2: User submits requestWithCallback{value: 2200}(provider, userRandom)
        (2× buffer to survive fee fluctuations)

Step 3: Provider sees pending tx with msg.value = 2200 in mempool
Step 4: Provider front-runs: setProviderFee(2100)
        → new requiredFee = 2100 + 100 = 2200 = msg.value ✓

Step 5: User's tx executes:
        providerFee = 2100
        providerInfo.accruedFeesInWei += 2100   ← provider gets 2100
        accruedPythFeesInWei += 2200 - 2100 = 100

Result:
  User paid 2200 wei, expected to pay 1100 wei
  Provider extracted 1100 wei extra (the user's safety buffer)
  No revert, no refund, no on-chain protection available
```
