### Title
Scheduler `minimumBalance` and Operational Keeper Funds Share a Single Pool, Allowing Keeper Fees to Drain the Anti-Sybil Deposit — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract holds ETH in a single `status.balanceInWei` field that serves two conceptually distinct purposes: (1) a **minimum balance deposit** enforced as an anti-sybil/quality-of-service floor, and (2) **operational funds** consumed by keeper fees and Pyth update fees. These two purposes are never tracked separately. The minimum balance floor is enforced at deposit and withdrawal time, but is **not** enforced during keeper fee deductions in `_processFeesAndPayKeeper`. As a result, repeated keeper calls to `updatePriceFeeds` can drain a subscription's balance below — and ultimately to zero — while the subscription remains in the active list, making the minimum balance guarantee illusory.

---

### Finding Description

`createSubscription` requires `msg.value >= minimumBalance` and `withdrawFunds` enforces `balanceInWei - amount >= minimumBalance` (when active). However, `_processFeesAndPayKeeper` only checks that `status.balanceInWei >= totalKeeperFee` before deducting:

```solidity
// Scheduler.sol _processFeesAndPayKeeper
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
status.balanceInWei -= totalKeeperFee;
``` [1](#0-0) 

There is no check that `status.balanceInWei - totalKeeperFee >= minimumBalance`. Similarly, the Pyth fee deduction in `updatePriceFeeds` also has no minimum balance floor:

```solidity
status.balanceInWei -= pythFee;
``` [2](#0-1) 

The `SchedulerState` stores both purposes in a single field with no separate locked-deposit tracking: [3](#0-2) 

The test suite explicitly confirms this behavior — balance can be drained below minimum through keeper updates: [4](#0-3) 

This is the direct analog to the ERC-4337 `StakeManager` issue: the "locked deposit" (minimum balance) and the "prepayment" (operational keeper funds) are stored in the same pool, making the boundary unenforceable during the operational spend path.

---

### Impact Explanation

1. **Minimum balance guarantee is illusory**: A subscriber who deposits exactly `minimumBalance` (the required floor) will have that entire deposit consumed by keeper fees over time. The minimum balance does not protect against the subscription running out of funds — it only prevents the subscriber from withdrawing it themselves while active.

2. **Active subscriptions list pollution**: Once `balanceInWei` reaches zero, `updatePriceFeeds` reverts with `InsufficientBalance`, but the subscription remains in `activeSubscriptionIds`. Keepers waste gas iterating over and attempting to update these non-functional subscriptions.

3. **Subscriber fund loss**: A subscriber locked into an active subscription (cannot withdraw below minimum) cannot prevent a keeper from draining their minimum balance deposit through legitimate update calls. When they deactivate and withdraw, the deposit is already gone.

---

### Likelihood Explanation

This occurs naturally in normal protocol operation. Any subscription that is not continuously topped up will eventually have its minimum balance consumed by keeper fees. A subscription created with exactly the minimum balance (the documented requirement) will become non-functional after a finite number of updates. No special attacker capability is required — any keeper calling `updatePriceFeeds` triggers this path.

---

### Recommendation

Track the minimum balance as a **separate locked deposit** distinct from the operational balance. Introduce two fields in `SubscriptionStatus`:

- `lockedDepositInWei` — the anti-sybil deposit, never spendable by keeper fees
- `operationalBalanceInWei` — the spendable operational funds

Enforce that `_processFeesAndPayKeeper` and the Pyth fee deduction only draw from `operationalBalanceInWei`. The `lockedDepositInWei` should only be released when the subscription is deactivated and the subscriber withdraws. This mirrors the fix applied in ERC-4337 PR #76: separating stake from prepayment at the accounting level.

---

### Proof of Concept

1. Subscriber calls `createSubscription{value: minimumBalance}(params)` — deposits exactly the minimum required.
2. Keeper calls `updatePriceFeeds(subscriptionId, updateData)` repeatedly (within valid heartbeat/deviation criteria).
3. Each call deducts `pythFee + gasCost + keeperSpecificFee` from `balanceInWei` with no minimum floor check.
4. After N updates, `balanceInWei < minimumBalance` (confirmed by the existing test at line 755).
5. After further updates, `balanceInWei == 0`.
6. Subscription remains in `activeSubscriptionIds` but all future `updatePriceFeeds` calls revert.
7. Subscriber deactivates and calls `withdrawFunds` — receives 0 ETH. Their minimum balance deposit was entirely consumed by keeper fees. [5](#0-4) [6](#0-5)

        SchedulerStructs.SubscriptionParams storage params = _state

        // Prevent withdrawals from permanent subscriptions

            }
        }

```text
    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
