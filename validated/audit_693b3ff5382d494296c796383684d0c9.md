Audit Report

## Title
`ScriptHashTypeVerifier` Skips Type Script `hash_type` Validation, Enabling Peer-Ban Bypass and Unbounded Verify-Queue Resource Consumption — (File: `verification/src/transaction_verifier.rs`)

## Summary
`ScriptHashTypeVerifier::verify()` checks only the lock script `hash_type` on each transaction output, leaving the optional type script `hash_type` entirely unchecked. A P2P relay peer can craft a transaction whose output carries a type script with a structurally valid but consensus-disabled `hash_type` (e.g., `Data3 = 6`), bypass non-contextual verification and the associated peer-ban gate, enter the verify queue, and consume contextual-verifier CPU cycles with no penalty — whereas an identical malformation on the lock script would result in immediate peer banning.

## Finding Description
`ScriptHashTypeVerifier::verify()` iterates over outputs and performs the enabled-range check exclusively on `output.lock().hash_type()`: [1](#0-0) 

`output.type_()` is never examined inside this loop. `ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}`: [2](#0-1) 

A type script with `hash_type = 6` (`Data3`) passes `check_data`'s `verify_value` check (even-number structural validity) and is also never caught by `ScriptHashTypeVerifier`, so `NonContextualTransactionVerifier` returns `Ok(())`: [3](#0-2) 

The transaction enters the contextual verify queue. `select_version()` is the first code that actually inspects the type script `hash_type`, and it returns `ScriptError::InvalidScriptHashType`: [4](#0-3) [5](#0-4) 

`is_malformed_tx()` only matches `TransactionError` variants, not `ScriptError`: [6](#0-5) 

Therefore the `ban_malformed` call in `non_contextual_verify` is never reached for this rejection: [7](#0-6) 

By contrast, a lock-script-invalid transaction (same `hash_type = 6` on the lock) is caught by `ScriptHashTypeVerifier`, produces `TransactionError::ScriptHashTypeNotPermitted`, `is_malformed_tx()` returns `true`, and the peer is immediately banned.

## Impact Explanation
**Peer-ban bypass with repeated resource consumption.** Because the input cell is not consumed when the transaction is rejected, the attacker can resubmit the same (or rotated) transaction indefinitely from the same peer connection without triggering a ban. Each submission enters the verify queue and forces the contextual verifier to run before rejection. This asymmetry — lock-script-invalid peers are banned, type-script-invalid peers are not — allows a sustained, low-cost attack that occupies verify-queue slots and CPU on targeted nodes.

This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points).**

## Likelihood Explanation
Any unprivileged actor who holds a single live cell (the minimum to construct a syntactically valid transaction) can trigger this. No key compromise, miner privilege, or majority hashpower is required. The attacker constructs a transaction with a valid lock script and a type script whose `hash_type` byte is `6`. The transaction passes `check_data` (structural validity) and `ScriptHashTypeVerifier` (enabled-range check), enters the verify queue, and is rejected only by `select_version()`. The peer is never banned. The attack is repeatable at negligible marginal cost.

## Recommendation
Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script immediately after the lock-script check:

```rust
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err(TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }.into());
    }
}
```

Update the struct doc-comment and the `NonContextualTransactionVerifier` doc-comment to reflect that both lock and type scripts are checked.

## Proof of Concept
1. Acquire any live cell (any amount of CKB).
2. Construct a `Transaction` with that cell as input, and one output whose lock script is valid (`hash_type = 1`, `Type`) and whose type script has `hash_type = 6` (`Data3` — even, passes `verify_value`, absent from `ENABLED_SCRIPT_HASH_TYPE`).
3. Submit via RPC `send_transaction` or relay via P2P.
4. Observe `ScriptHashTypeVerifier::verify()` returns `Ok(())` — non-contextual verification passes.
5. Observe the transaction enters the verify queue and is rejected by `select_version()` with `ScriptError::InvalidScriptHashType`.
6. Confirm no `ban_malformed` is called (peer remains connected and unbanned).
7. Repeat step 3 from the same peer indefinitely; confirm the peer is never banned, whereas replacing the type-script `hash_type = 6` with a lock-script `hash_type = 6` causes immediate peer banning after step 3.
