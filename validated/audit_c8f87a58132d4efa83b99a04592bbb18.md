### Title
Provider Fee Frontrunning Allows Malicious Provider to Extract Excess Fees from Users - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract allows any registered provider to update their `feeInWei` immediately and without a timelock via `setProviderFee` / `setProviderFeeAsFeeManager`. Because `requestHelper` reads the current on-chain fee at execution time and explicitly does **not** refund excess `msg.value`, a malicious provider can frontrun a user's `requestV2` transaction by raising their fee, causing the user to pay more than they expected.

---

### Finding Description

**Root cause — no timelock on fee changes:**

`setProviderFee` takes effect in the same block it is called:

```solidity
// Entropy.sol line 810-827
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;   // immediate, no delay
    emit ProviderFeeUpdated(...);
}
``` [1](#0-0) 

The same is true for the fee-manager path:

```solidity
// Entropy.sol line 829-855
function setProviderFeeAsFeeManager(address provider, uint128 newFeeInWei) external override {
    ...
    providerInfo.feeInWei = newFeeInWei;   // immediate
}
``` [2](#0-1) 

**Root cause — fee is read at execution time, excess is not refunded:**

Inside `requestHelper`, the required fee is computed from the **current** on-chain state at the moment the transaction executes, not from any value the user committed to:

```solidity
// Entropy.sol line 233-239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [3](#0-2) 

The provider receives `getProviderFee(provider, callbackGasLimit)` — the **current** fee — not the fee the user queried before submitting. Excess `msg.value` above `providerFee` flows to Pyth's fee pool; it is never returned to the caller. This is explicitly documented in the interface:

> "Note that excess value is *not* refunded to the caller." [4](#0-3) 

**No `maxFee` guard exists in any `requestV2` variant:**

None of the four `requestV2` overloads accept a caller-specified maximum fee: [5](#0-4) 

---

### Impact Explanation

A malicious provider can frontrun any `requestV2` call where the user sent `msg.value` slightly above the fee they queried (a common defensive pattern to avoid reverts from minor fee fluctuations). The provider raises their fee to a value between the user's queried fee and the user's `msg.value`. The transaction succeeds, but the provider accrues the higher fee. The user pays more than they expected with no recourse.

The Fortuna keeper's own dynamic fee-adjustment loop (`adjust_fee_if_necessary`) demonstrates that provider fees are expected to change frequently in response to gas prices, making it entirely normal for users to add a buffer: [6](#0-5) 

---

### Likelihood Explanation

- Any address can register as a provider (`register()` is permissionless).
- A malicious provider can monitor the mempool for pending `requestV2` calls and insert a `setProviderFee` transaction with a higher gas price.
- Users and integrating contracts routinely send a small buffer above the quoted fee to avoid reverts from fee fluctuations; this is the exact condition the attack exploits.
- The Fortuna keeper's automated fee-adjustment mechanism means fee changes are a normal, expected event, making the attack indistinguishable from legitimate behavior.

---

### Recommendation

1. **Add a `maxFee` parameter** to all `requestV2` variants. Revert if `getFeeV2(provider, gasLimit) > maxFee` at execution time.
2. **Alternatively, add a timelock** for fee increases (e.g., a pending-fee mechanism where a new fee only takes effect after N blocks), analogous to the fix recommended in Teller Finance issue #497.
3. **Refund excess `msg.value`** above the actual required fee back to the caller, eliminating the economic incentive for the frontrun.

---

### Proof of Concept

1. Provider registers with `feeInWei = 1 ETH`.
2. User calls `getFeeV2(provider, 0)` → returns `1 ETH + pythFee`.
3. User submits `requestV2{value: 1.1 ETH + pythFee}(provider, userRandom, 0)` (10 % buffer).
4. Provider observes the pending transaction and frontruns with `setProviderFee(1.05 ETH)` at a higher gas price.
5. User's transaction executes: `requiredFee = 1.05 ETH + pythFee ≤ msg.value` → no revert.
6. `providerInfo.accruedFeesInWei += 1.05 ETH` — provider receives **0.05 ETH more** than the user expected to pay.
7. `_state.accruedPythFeesInWei += 0.05 ETH` — the remaining excess goes to Pyth, not back to the user.

The user paid `1.1 ETH + pythFee` but expected to pay `1 ETH + pythFee`. The provider extracted an extra `0.05 ETH` through a single frontrun transaction with no protocol-level protection available to the user.

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;
        providerInfo.feeInWei = newFeeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            provider,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
