Audit Report

## Title
Copy-paste bug in `script_exists_in_output` returns lock-script existence instead of type-script existence on PostgreSQL during rollback — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary
At line 252 of `remove.rs`, the second `match` block re-reads `row_lock` instead of `row_type`, causing `script_exists_in_output` to return the lock-script existence boolean when evaluating type-script existence on PostgreSQL. During `rollback_block`, this causes type scripts still referenced by surviving live outputs to be incorrectly deleted from the `script` table, permanently corrupting the rich-indexer's script-to-cell mapping on PostgreSQL deployments.

## Finding Description
`script_exists_in_output` performs two `SELECT EXISTS` queries: one against `lock_script_id` (result in `row_lock`) and one against `type_script_id` (result in `row_type`). The first match block at line 223 correctly short-circuits if the lock check is true. After fetching `row_type` at line 237, the second match block at line 252 is supposed to evaluate `row_type` but instead evaluates `row_lock` again:

```rust
// line 252 — BUG: should be row_type.try_get::<bool, _>(0)
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),                              // returns lock result, not type result
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1) // SQLite fallback is correct
}
``` [1](#0-0) 

On PostgreSQL, `row_lock.try_get::<bool, _>(0)` always succeeds (PostgreSQL `EXISTS` returns `BOOLEAN`), so `r` is the lock-script existence boolean. The `Err(_)` branch — which correctly reads `row_type` — is never reached on PostgreSQL. On SQLite, `EXISTS` returns `BIGINT`, so `try_get::<bool, _>(0)` fails and falls through to the correct `row_type.get::<i64, _>(0) == 1` path.

Concrete failure path (PostgreSQL only):
1. Block N is appended with a cell using unique type script `T` (not used as any lock script).
2. Block N+1 is appended spending that cell.
3. A reorg triggers `rollback_block` for block N+1.
4. `rollback_block` removes block N+1's outputs from the `output` table, then calls `script_exists_in_output(type_script_id_of_T, tx)`.
5. `row_lock` query: `T` is not a lock script → `false`.
6. First match: `Ok(false)` → does not short-circuit.
7. `row_type` query: `T` is still a type script on block N's live cell → `true`.
8. Second match (line 252): reads `row_lock` again → `Ok(false)` → returns `false`.
9. `T` is pushed to `script_id_list_to_remove` and hard-deleted from the `script` table.
10. Block N's live cell still references `T` via `type_script_id`, but the script row no longer exists. [2](#0-1) 

All existing rollback tests use SQLite exclusively via `connect_sqlite(MEMORY_DB)`, so this PostgreSQL-specific divergence is not covered by any test. [3](#0-2) [4](#0-3) [5](#0-4) 

## Impact Explanation
The impact matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**. The rich-indexer is the CKB state storage mechanism for cell/script queries. The bug causes permanent, unrecoverable corruption of the script table on PostgreSQL after any reorg involving a cell whose type script is not also used as a lock script. After corruption, `get_cells` RPC queries using `script_type: Type` silently return empty results for valid live cells. Recovery requires a full re-index. The corruption does not affect the CKB consensus layer or node operation, placing it below High severity.

## Likelihood Explanation
Chain reorgs are a normal, unprivileged network event requiring no special keys or majority hashpower. The bug fires on any rollback involving a cell with a type script that is not simultaneously used as a lock script in any other remaining output — a common pattern for TYPE_ID cells, UDT cells, and NFT cells. Only PostgreSQL deployments are affected; SQLite deployments are not. Any operator running the rich-indexer against PostgreSQL on a chain that experiences reorgs (including normal short reorgs) is at risk.

## Recommendation
Change line 252 from:
```rust
match row_lock.try_get::<bool, _>(0) {
```
to:
```rust
match row_type.try_get::<bool, _>(0) {
```
Additionally, add a PostgreSQL-backed integration test for `rollback_block` that verifies script rows referenced only as `type_script_id` by surviving outputs are not deleted. [6](#0-5) 

## Proof of Concept
1. Start the rich-indexer with a PostgreSQL backend.
2. Append block N containing a transaction output with a unique type script `T` (not used as any lock script anywhere).
3. Append block N+1 spending that output.
4. Trigger rollback of block N+1 (e.g., via a competing fork or direct `rollback()` call in a test).
5. Query: `SELECT * FROM script WHERE id = <T_id>` → row is absent (incorrectly deleted).
6. Call `get_cells` RPC with `script_type: Type` matching `T` → returns empty, despite block N's live cell still existing in the `output` table with `type_script_id = <T_id>`. [7](#0-6)

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
