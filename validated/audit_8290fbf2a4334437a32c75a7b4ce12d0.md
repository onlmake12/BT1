Audit Report

## Title
Unauthenticated `from` Field in `ConnectionRequest`/`ConnectionSync` Enables Attacker-Controlled NAT Traversal — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
`ConnectionRequestProcess::execute` accepts `content.from` verbatim from the wire message and passes it directly to `respond_delivered` without verifying it matches the actual sending peer (`self.peer`). A subsequent spoofed `ConnectionSync` with the same fabricated `from` value causes the victim to look up the attacker-inserted entry in `pending_delivered` and spawn a `try_nat_traversal` task that calls `raw_session` to an attacker-controlled IP. Repeating with fresh spoofed peer IDs bypasses all cooldown and rate-limit guards, enabling the attacker to fill the victim's connection table with attacker-controlled nodes.

## Finding Description

**Root cause:** `content.from` in both `ConnectionRequest` and `ConnectionSync` is accepted verbatim with no cryptographic binding to the actual sending session.

**Step 1 — Spoofed `ConnectionRequest`:**

`ConnectionRequestProcess` holds a `peer: PeerIndex` field identifying the actual sender, but it is never compared to `content.from`. When `self_peer_id == &content.to`, `respond_delivered(content.from, ...)` is called directly: [1](#0-0) 

Inside `respond_delivered`, the only guard is a 2-minute cooldown keyed by `from_peer_id` (the spoofed value): [2](#0-1) 

This is trivially bypassed by using a different spoofed `from` value each time. The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` — also bypassed by rotating spoofed `from` values: [3](#0-2) 

The `listen_addrs` validation only checks that any embedded peer ID in the address matches `content.from` (the spoofed value), not the actual sender: [4](#0-3) 

After passing these checks, the attacker's IP is unconditionally inserted into `pending_delivered` under the spoofed key: [5](#0-4) 

**Step 2 — Spoofed `ConnectionSync`:**

`ConnectionSyncProcess` has no `peer` field at all — it cannot identify the actual sender: [6](#0-5) 

When `route` is empty and `self_peer_id == content.to`, it looks up `pending_delivered` using the spoofed `content.from`: [7](#0-6) 

This returns the attacker's IP inserted in Step 1. The code then spawns a NAT traversal task and calls `raw_session` with the attacker's IP: [8](#0-7) 

**`try_nat_traversal` works regardless of `bind_addr`:** When `bind_addr` is `None`, `create_socket` still creates a TCP socket and connects to the target IP — `bind_addr` only controls port reuse, not whether the connection is attempted: [9](#0-8) 

**Rate limiter analysis:** The per-session `rate_limiter` is keyed by `(session_id, msg.item_id())` and allows 30 req/sec: [10](#0-9) [11](#0-10) 

This is the only throttle that cannot be bypassed by rotating spoofed `from` values, and 30 req/sec is far more than needed for this two-message attack sequence.

## Impact Explanation

The victim node establishes a raw TCP P2P session to an attacker-controlled IP via `raw_session`. By repeating the two-message sequence with fresh spoofed peer IDs — each bypassing the 2-minute cooldown and the `(from, to)` keyed `forward_rate_limiter` — the attacker can fill the victim's connection table with attacker-controlled nodes. This constitutes a targeted eclipse attack: the victim's view of the network is dominated by attacker-controlled peers, enabling selective withholding or manipulation of block/transaction relay and causing **consensus deviation**. This matches the Critical impact class: "Vulnerabilities which could easily cause consensus deviation."

## Likelihood Explanation

The attack requires only a single legitimate P2P connection to the victim — no special privileges, no key material, no majority hashpower. The spoofed fields (`from`, `listen_addrs`) are accepted verbatim from the wire message with no cryptographic binding to the sending session. The two-message sequence is sufficient to trigger `raw_session` to an arbitrary attacker-controlled IP. The attack is repeatable at up to 30 iterations/sec (bounded only by the per-session rate limiter) with different spoofed peer IDs to fill all connection slots.

## Recommendation

In `ConnectionRequestProcess::execute`, before calling `respond_delivered`, verify that `content.from` matches the actual peer ID of the sending session by looking up `self.peer` in the peer registry:

```rust
let actual_from = self.protocol.network_state
    .peer_registry.read()
    .get_key_by_peer_id(&content.from);
if actual_from != Some(self.peer) {
    return StatusCode::InvalidFromPeerId.with_context("from does not match sender");
}
```

Similarly, `ConnectionSyncProcess` should be extended with a `peer: PeerIndex` field (passed from `context.session.id` in `mod.rs`) and verify that `content.from` matches the actual sending peer before consuming `pending_delivered`. The `pending_delivered` map should also be keyed by `(from_peer_id, session_id)` or validated against the session at lookup time.

## Proof of Concept

```
Setup:
  - Attacker connects to victim (legitimate session, session_id=S, attacker_peer_id=A)
  - Attacker controls a listener at attacker_ip:port

Step 1: Attacker sends ConnectionRequest {
    from         = T  (arbitrary peer ID, T != A),
    to           = victim_peer_id,
    listen_addrs = [/ip4/attacker_ip/tcp/port/p2p/T],
    max_hops     = 1,
    route        = []
}
→ Victim: self_peer_id == content.to → calls respond_delivered(T, ...)
  [no check that T == self.peer's peer_id]
→ Victim: pending_delivered[T] = ([/ip4/attacker_ip/tcp/port/p2p/T], now)
→ Victim: sends ConnectionRequestDelivered back to session S (attacker ignores it)

Step 2: Attacker sends ConnectionSync {
    from  = T,
    to    = victim_peer_id,
    route = []
}
→ Victim: route is empty, self_peer_id == content.to
→ Victim: listens_info = pending_delivered[T] = [/ip4/attacker_ip/tcp/port/p2p/T]
→ Victim: spawns try_nat_traversal(bind_addr, /ip4/attacker_ip/tcp/port/p2p/T)
→ Attacker's listener accepts the TCP connection
→ Victim: calls raw_session(stream, attacker_ip, RawSessionInfo::inbound(listen_addr))
→ New P2P session established to attacker-controlled node

Step 3: Repeat steps 1-2 with fresh T values (T1, T2, ...) at up to 30/sec
        to fill victim's connection table with attacker-controlled nodes.
        Each fresh T bypasses the HOLE_PUNCHING_INTERVAL cooldown and the
        forward_rate_limiter (both keyed by the spoofed from value).
```
