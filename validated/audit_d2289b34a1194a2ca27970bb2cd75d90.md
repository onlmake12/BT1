### Title
Malicious Entropy Provider Can Front-Run User Requests to Increase Fee, Causing Asset Loss or DoS - (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

A registered Entropy provider (or their designated fee manager) can call `setProviderFee` / `setProviderFeeAsFeeManager` at any time with no delay, cap, or timelock. Because excess `msg.value` is explicitly **not refunded** to callers, a malicious provider can front-run a user's pending `requestV2` / `requestWithCallback` transaction to either (a) capture the user's fee buffer as extra provider revenue, or (b) force the user's transaction to revert entirely.

---

### Finding Description

`setProviderFee` and `setProviderFeeAsFeeManager` update `provider.feeInWei` immediately and unconditionally: [1](#0-0) 

Inside `requestHelper`, the fee check uses the **live** on-chain value at execution time, not the value the user observed when constructing the transaction: [2](#0-1) 

The fee accounting splits `msg.value` as follows: the provider receives exactly `providerFee` (the current `feeInWei`), and Pyth receives the remainder (`msg.value − providerFee`). Crucially, **no ETH is ever returned to the caller**: [3](#0-2) 

This creates a TOCTOU window between when the user reads the fee and when their transaction executes.

---

### Impact Explanation

**Asset-loss path (primary):**

1. User calls `getFeeV2(provider)` → observes fee = `P + F` (provider fee `P`, Pyth fee `F`).
2. User submits `requestV2{value: P + F + buffer}(provider, ...)` with a small buffer to guard against minor fee fluctuations — a common and reasonable defensive pattern explicitly encouraged by the documentation warning that fees "can change over time."
3. Provider (or fee manager) front-runs with `setProviderFee(P + buffer)`.
4. User's transaction executes:
   - `requiredFee = (P + buffer) + F = P + F + buffer`
   - `msg.value ≥ requiredFee` → succeeds
   - Provider accrues `P + buffer` instead of `P` — capturing the user's entire buffer
   - Pyth accrues only `F`
5. User paid `buffer` more than intended with no recourse.

**DoS path (secondary):**

If the user sends exactly the observed fee with no buffer, the provider front-runs with an arbitrarily large fee increase, causing the user's transaction to revert with `InsufficientFee`. The user loses gas and cannot use the service. The provider can repeat this indefinitely. [4](#0-3) 

---

### Likelihood Explanation

- Any registered Entropy provider can execute this attack — registration is permissionless.
- The fee manager role (`setFeeManager`) is also provider-controlled and can be set to any address, including an automated MEV bot.
- On EVM chains with a public mempool (Ethereum mainnet, most L2s), front-running is straightforward using standard MEV infrastructure.
- The Fortuna keeper itself runs an automated `adjust_fee_if_necessary` loop that updates fees based on gas prices, meaning fee changes are a normal, expected on-chain event — making malicious fee changes indistinguishable from legitimate ones. [5](#0-4) 

---

### Recommendation

1. **Introduce a fee-change timelock or delay**: Record a `pendingFee` and `feeEffectiveAt` timestamp; only apply the new fee after a minimum delay (e.g., 1 hour). This eliminates the front-running window.
2. **Alternatively, cap single-step fee increases**: Limit how much a provider can increase their fee in a single transaction (e.g., no more than 2× the current fee per update).
3. **Refund excess `msg.value`**: Return `msg.value − requiredFee` to the caller. This removes the asset-loss path entirely, reducing the worst case to a harmless revert.

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);

            oldFeeInWei,
