Audit Report

## Title
Missing `BLOCK_INVALID` Early-Return Guard in `HeaderAcceptor::accept()` Enables Repeated PoW Re-verification and Peer State Corruption — (File: `sync/src/synchronizer/headers_process.rs`)

## Summary

`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` noting the absence of an early-return for `BLOCK_INVALID` status. Because `BlockStatus::BLOCK_INVALID` (`1 << 12`) shares no bits with `BlockStatus::HEADER_VALID` (`1`), the existing guard does not fire for invalid blocks. A malicious peer can repeatedly relay headers for blocks already marked `BLOCK_INVALID` (e.g., after body-level rejection), causing the node to re-run PoW verification on every delivery without penalizing the peer, and incorrectly updating the peer's `best_known_header` to a known-invalid block.

## Finding Description

**Root cause — bit-flag disjointness:**

`BlockStatus` is defined as a bitflag set where `BLOCK_INVALID = 1 << 12` is completely disjoint from the `HEADER_VALID` chain (`1`, `3`, `7`, `15`):

```
HEADER_VALID  = 0x0001
BLOCK_INVALID = 0x1000
```

`BLOCK_INVALID.contains(HEADER_VALID)` is always `false`. The test suite in `sync/src/tests/block_status.rs` explicitly confirms `BLOCK_INVALID` is only contained in itself.

**The missing guard:**

In `accept()`, the developer left a `FIXME` comment acknowledging the gap:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {   // never fires for BLOCK_INVALID
    ...
    return result;
}
// Falls through for BLOCK_INVALID
```

**Exploit path:**

1. Block `B` is received, its header passes `HeaderAcceptor`, `insert_valid_header` is called, header enters `header_map`.
2. The full block is later processed by `ConsumeUnverifiedBlocks`; script execution or capacity verification fails; `BLOCK_INVALID` is written to `block_status_map` (confirmed in `docs/ckb_async_block_sync.mermaid` and `chain/src/orphan_broker.rs`).
3. `get_block_status` now returns `BLOCK_INVALID` because `block_status_map` is checked first.
4. Malicious peer sends `B`'s header again via `SendHeaders`.
5. `accept()` reads `BLOCK_INVALID`, the `HEADER_VALID` guard does not fire, execution falls through.
6. `prev_block_check` passes — it checks the **parent's** status, not `B`'s own status.
7. `non_contextual_check` runs `HeaderVerifier::verify()` including full PoW (Eaglesong) verification.
8. `version_check` passes.
9. `insert_valid_header` is called: header is re-inserted into `header_map`, `may_set_best_known_header` updates the peer's best-known to `B`, `may_set_shared_best_header` may update the global best header.
10. `accept()` returns `ValidationState::Valid` (the default). The peer is **not penalized**.
11. Steps 4–10 repeat indefinitely.

**Why existing checks are insufficient:**

- `prev_block_check` only checks `parent_hash` for `BLOCK_INVALID`, not the header's own hash.
- `non_contextual_check` only verifies PoW/timestamp/epoch fields — none of which caused the original rejection.
- `version_check` only checks the version field.
- None of the three checks consult the header's own stored `BLOCK_INVALID` status.

**`insert_valid_header` does not write to `block_status_map`** — it only inserts into `header_map` and updates peer/shared state — so `get_block_status` continues returning `BLOCK_INVALID` on subsequent calls, meaning the loop repeats on every new delivery.

**Peer state corruption:**

`asynchronous_process_remote_block` in `sync/src/synchronizer/mod.rs` gates block download on `status.contains(HEADER_VALID)`, so the block body is not re-downloaded. However, `may_set_best_known_header` is called with the invalid block's `HeaderIndex`, skewing `BlockFetcher`'s peer selection toward a peer advertising a chain the local node has already determined is invalid.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single unprivileged peer can send up to `MAX_HEADERS_LEN = 2000` headers per `SendHeaders` message, all pointing to previously-rejected blocks. Each header triggers a full `HeaderVerifier::verify()` call (Eaglesong PoW hash computation). Because the peer is never penalized (result is `Valid`), this can be repeated at the network message rate limit indefinitely. Sustained delivery forces the target node to spend unbounded CPU on PoW re-verification for blocks it already knows are invalid, degrading sync performance and potentially causing the node to fall behind the chain tip.

## Likelihood Explanation

Any unauthenticated P2P peer can send `SendHeaders` messages. The attacker only needs the hash of a block that was previously rejected by the target node — observable by monitoring relay traffic or by sending a crafted block body that passes header checks but fails script/capacity verification. The attack is low-cost, requires no special privilege, and is indefinitely repeatable without triggering peer banning.

## Recommendation

Add an explicit `BLOCK_INVALID` guard at the top of `accept()`, before any validation work:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

if status.contains(BlockStatus::BLOCK_INVALID) {
    state.invalid(Some(ValidationError::InvalidParent)); // reuse or add dedicated variant
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // existing early-return logic
    ...
    return result;
}
```

This resolves both the CPU exhaustion vector and the peer state corruption, and directly addresses the developer's own `FIXME`.

## Proof of Concept

1. Connect a malicious peer to a target CKB node.
2. Craft block `B` with a valid header (valid PoW, timestamp, epoch, version=0, valid parent) but an invalid body (e.g., a transaction with a script that returns non-zero exit code).
3. Send the full block to the target node via `SendBlock`. The node processes it: header passes `HeaderAcceptor`, body fails script verification, `BLOCK_INVALID` is written to `block_status_map`.
4. In a loop, send a `SendHeaders` message containing `B`'s header.
5. Observe via CPU profiling that `HeaderVerifier::verify()` (Eaglesong) is called on every iteration.
6. Observe that the peer is never disconnected or penalized (no `HeadersIsInvalid` status returned to the peer handler).
7. Observe that the peer's `best_known_header` is updated to `B` after each iteration.

Unit test plan: in `sync/src/synchronizer/tests/`, set `BLOCK_INVALID` for a header hash in `block_status_map`, call `HeaderAcceptor::accept()`, assert the returned `ValidationResult` has `state == ValidationState::Invalid` and that `insert_valid_header` was not called.
