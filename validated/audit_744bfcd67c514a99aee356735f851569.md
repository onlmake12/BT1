Looking at the exact code path in detail:

### Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` — (`sync/src/types/mod.rs`)

---

### Summary

`UnknownTxHashPriority::push_peer` appends a `PeerIndex` to an uncapped `Vec` with no deduplication or length guard. The overflow check in `SyncState::add_ask_for_txs` only measures the number of **distinct tx hashes** in the priority queue — not the total number of peer-index entries accumulated across all `peers` Vecs. Because unknown hashes are never added to `tx_filter`, a single unprivileged peer can repeatedly send the same 32,767 unknown tx hashes, causing each queue entry's `peers` Vec to grow without bound, exhausting process memory.

---

### Finding Description

**Entry point:** Any connected P2P peer sends `RelayTransactionHashes` messages.

`TransactionHashesProcess::execute` filters hashes against `tx_filter` (which only contains *known/processed* hashes) and forwards the remainder to `add_ask_for_txs`: [1](#0-0) 

Inside `add_ask_for_txs`, for every hash already present in the queue (the `Occupied` branch), `push_peer` is called unconditionally: [2](#0-1) 

`push_peer` has no cap and no deduplication: [3](#0-2) 

The only overflow guard checks `unknown_tx_hashes.len()` — the count of **distinct hashes** in the queue: [4](#0-3) 

**Why the guard never fires for this attack:**

- `MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000; `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = 32,767.
- After the first send of 32,767 hashes, `unknown_tx_hashes.len()` = 32,767 — below both thresholds.
- On every subsequent send of the **same** hashes, `unknown_tx_hashes.len()` stays at 32,767 (no new distinct entries are added), so neither condition is ever satisfied.
- Each repeated send pushes the attacker's `PeerIndex` into every entry's `peers` Vec again. [5](#0-4) 

The `peers` field of `UnknownTxHashPriority` is a plain `Vec<PeerIndex>` with no capacity bound: [6](#0-5) 

---

### Impact Explanation

A single peer at the rate-limiter ceiling of 30 `RelayTransactionHashes` messages/second, each carrying 32,767 hashes, produces:

```
32,767 entries × 30 pushes/sec × 8 bytes (PeerIndex = usize) ≈ 7.8 MB/sec
```

After ~60 seconds from one peer: ~470 MB of heap growth. With `max_peers = 125` peers all sending simultaneously, the rate scales to ~975 MB/sec. The entries are never evicted because the hashes are for non-existent transactions that will never be marked known via `mark_as_known_txs`. This leads to OOM-kill or severe memory pressure, degrading or crashing the node. [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

The attack requires only a standard P2P connection — no PoW, no keys, no privileged role. The attacker fabricates 32,767 random (non-existent) tx hashes and sends them in a loop. The rate limiter slows but does not stop the attack. The existing `TooManyUnknownTransactions` ban path is never reached because the distinct-hash count stays below the threshold.

---

### Recommendation

1. **Cap `peers` Vec in `push_peer`**: Reject or ignore the push when `self.peers.len()` exceeds a small constant (e.g., `MAX_PEERS` or a dedicated `MAX_PEERS_PER_TX_HASH` constant).
2. **Deduplicate peer indices**: Check `self.peers.contains(&peer_index)` before pushing (or use a `HashSet`).
3. **Fix the overflow guard**: Track total peer-index entries (sum of all `peers.len()`) rather than only distinct hash count, and enforce a global cap.
4. **Evict stale entries**: Periodically remove `unknown_tx_hashes` entries whose `request_time` has aged past a TTL without the tx being resolved.

---

### Proof of Concept

```rust
// Pseudocode: single-peer memory exhaustion
let hashes: Vec<Byte32> = (0..32767).map(|_| random_byte32()).collect();
// First send: populates 32767 entries, each peers=[attacker]
send_relay_tx_hashes(attacker_peer, &hashes);

// Subsequent sends: unknown_tx_hashes.len() stays 32767 < 50000,
// guard never fires, each entry's peers Vec grows by 1 per send
loop {
    send_relay_tx_hashes(attacker_peer, &hashes); // 30/sec via rate limiter
    // After 60s: each of 32767 entries has peers.len() == 1800
    // Heap: 32767 * 1800 * 8 bytes ≈ 472 MB
}
```

The assert from the question holds: `unknown_tx_hashes` memory grows linearly with send count, and no cap is enforced on `peers.len()` in `UnknownTxHashPriority`. [9](#0-8)

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }
