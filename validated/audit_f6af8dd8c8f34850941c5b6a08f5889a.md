### Title
Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

### Summary

A missing bounds check in `remove_unconfirmed_tx` allows `tx_age - 1` to exceed the length of `confirm_blocks_to_failed_txs` (1000), causing a Rust index-out-of-bounds panic and crashing the node process. This is reachable by any unprivileged actor who can submit a transaction to a node configured with the `ConfirmationFraction` fee estimator.

---

### Finding Description

`TxConfirmStat::remove_unconfirmed_tx` computes `tx_age = tip_height - entry_height` and uses it to index into `confirm_blocks_to_failed_txs`, which has a fixed length of `MAX_CONFIRM_BLOCKS = 1000`: [1](#0-0) 

The guard at line 208 (`if tx_age >= self.block_unconfirmed_txs.len()`) correctly handles the `block_unconfirmed_txs` slot, but the `count_failure` branch at line 215 has **no corresponding bounds check**:

```rust
if count_failure {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
    //  ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ panics when tx_age > 1000
}
```

`confirm_blocks_to_failed_txs` is initialized with exactly `MAX_CONFIRM_BLOCKS = 1000` rows: [2](#0-1) 

When `tx_age = 1001`, the index `tx_age - 1 = 1000` is out of bounds (valid range: 0–999), causing an unconditional Rust panic.

---

### Impact Explanation

The panic propagates uncaught and crashes the CKB node process. Any node operator running with `ConfirmationFraction` configured is vulnerable. The `reject_tx` callback is registered unconditionally for all pool evictions: [3](#0-2) 

`reject_tx` on the `ConfirmationFraction` variant calls `algo.write().reject_tx(tx_hash)` → `drop_tx_inner(hash, true)` → `remove_unconfirmed_tx(..., count_failure=true)`: [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

**Precondition:** The node must be configured with `fee_estimator.algorithm = "ConfirmationFraction"`. This is **not the default** — the default is `Dummy`: [6](#0-5) 

When that precondition is met, the exploit is trivial:
1. Submit a valid transaction with a fee rate low enough that miners never select it (unprivileged RPC or P2P relay).
2. Wait for 1001 blocks to pass (≈ ~4 hours on mainnet at 8s/block).
3. The tx is evicted from the pool by size-limit or expiry eviction, triggering `reject_tx` with `count_failure=true`.
4. `tx_age = best_height - entry_height ≥ 1001` → OOB panic → node crash.

No PoW, no keys, no special privileges required beyond transaction submission access.

---

### Recommendation

Add a bounds check before the `confirm_blocks_to_failed_txs` index in `remove_unconfirmed_tx`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This mirrors the existing guard pattern used for `block_unconfirmed_txs` at line 208. [7](#0-6) 

---

### Proof of Concept

```rust
#[test]
fn test_oob_panic_remove_unconfirmed_tx() {
    let mut stat = TxConfirmStat::default(); // MAX_CONFIRM_BLOCKS = 1000
    let fee_rate = FeeRate::from_u64(1000);
    // Track a tx at height 0
    let bucket_index = stat.add_unconfirmed_tx(0, fee_rate).unwrap();
    // Advance the window 1001 times (simulating 1001 blocks)
    for h in 1..=1001u64 {
        stat.move_track_window(h);
    }
    // Evict with count_failure=true: tx_age = 1001, index = 1000 → OOB panic
    stat.remove_unconfirmed_tx(0, 1001, bucket_index, true);
}
```

This test panics with `index out of bounds: the len is 1000 but the index is 1000`.
