Audit Report

## Title
Peer Store Permanently Blocked via Crafted Group-of-4 Address Distribution — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

`check_purge` contains a logical dead zone: when all network groups contain exactly 4 peers, the strict `addrs.len() > 4` threshold combined with `take(len / 2)` integer division ensures zero eviction candidates are ever produced. `add_addr` propagates the resulting `EvictionFailed` error unconditionally, permanently blocking new peer address insertion and leaving the node unable to discover peers.

## Finding Description

`ADDR_COUNT_LIMIT` is 16384. An attacker fills the store with exactly 4096 groups of 4 peers each (4096 × 4 = 16384). The network group for IPv4 is derived from the first two octets (`Group::IP4([bits[0], bits[1]])`), so 4096 distinct `/16` blocks suffice.

When `check_purge` is triggered:

**First eviction pass** (non-connectable peers): All injected addresses have `last_connected_at_ms=0` and `attempts_count=0` (hardcoded in `add_addr` at line 78 and `AddrInfo::new` at types.rs:72). `is_connectable` only returns false when `last_connected_at_ms==0 && attempts_count >= ADDR_MAX_RETRIES(3)`. With `attempts_count=0`, `0 >= 3` is false — every address is connectable. First pass yields zero candidates.

**Second eviction pass** (group-based):
```rust
let len = peers_by_network_group.len();   // 4096
peers.sort_unstable_by_key(|k| Reverse(k.len()));
peers
    .into_iter()
    .take(len / 2)          // take(2048)
    .flat_map(move |addrs| {
        if addrs.len() > 4  // 4 > 4 == false for every group
        { Some(...) } else { None }
    })
    .flatten()
    .collect()              // always empty
```

All 2048 considered groups have exactly 4 peers; none satisfy `> 4`. `candidate_peers` is empty, reaching:

```rust
if candidate_peers.is_empty() {
    return Err(PeerStoreError::EvictionFailed.into());
}
```

`add_addr` propagates this unconditionally via `self.check_purge()?;`. Every subsequent call to `add_addr` returns `Err(EvictionFailed)`. The store is permanently locked.

The `take(len / 2)` integer division is a secondary flaw: with an odd number of groups, the largest group is skipped entirely, but the primary trigger is the strict `> 4` threshold.

## Impact Explanation

Once locked, the node cannot record any newly discovered peer addresses. As existing connections age out or disconnect, the node has no pool from which to establish new outbound connections, leading to complete network isolation. This matches **High** impact: vulnerabilities which could easily crash a CKB node (10001–15000 points), as a fully isolated node is effectively removed from the network.

## Likelihood Explanation

Any unprivileged P2P participant can advertise arbitrary addresses via the CKB discovery protocol. The attacker needs to inject 16384 addresses spread across 4096 distinct `/16` IPv4 blocks (65536 possible groups exist), 4 addresses per block. All injected addresses use default values (`last_connected_at_ms=0`, `attempts_count=0`) which are set automatically by `add_addr`. No special privileges, victim mistakes, or external dependencies are required. The attack is repeatable: if the store is ever purged by other means, the attacker re-floods it.

## Recommendation

1. Replace `addrs.len() > 4` with `addrs.len() >= 4` so groups of exactly 4 qualify for eviction.
2. Replace `take(len / 2)` with `take((len + 1) / 2)` (ceiling division) to ensure the largest group is always considered when `len` is odd.
3. Add a fallback: if `candidate_peers` is still empty after the group pass, unconditionally evict one address from the largest group to guarantee forward progress regardless of distribution.
