Audit Report

## Title
Unbounded SQL OFFSET in `get_tx_with_cell` Enables Attacker-Controlled O(n) DB Scan via Crafted Cursor — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
The ungrouped `get_transactions` RPC path decodes the `after` cursor into a `(last_id: i64, offset: i32)` pair and passes the `offset` field directly to a SQL `OFFSET` clause with no upper-bound validation. An unprivileged caller can craft a cursor encoding `offset = i32::MAX` (~2.1 billion), forcing the database engine to materialize and discard that many rows on every request, making the indexer RPC effectively unresponsive.

## Finding Description

**Cursor parsing — no range check on offset:**

In `get_transactions` (L44–53), the 12-byte `after` cursor is split: the first 8 bytes become `last_id` (i64) and the last 4 bytes become `offset` (i32). The `decode_i32` helper (mod.rs L286–294) only validates that the slice is exactly 4 bytes; it performs no range check on the resulting value. Any i32 value, including `i32::MAX` (2,147,483,647), is accepted without error.

**Offset injected directly into SQL:**

`get_tx_with_cell` (L179–182) passes the decoded offset straight to `query_builder.offset(offset)`, producing:
```sql
SELECT ... ORDER BY tx_id ASC LIMIT <limit> OFFSET <attacker_value>
```
SQL `OFFSET` semantics require the engine to scan and discard `offset` rows before returning results — O(offset) work regardless of the actual result set size.

**Contrast with grouped mode:**

`get_tx_with_cells` (L320–325) uses a keyset cursor (`WHERE tx_id > after`) with no `OFFSET` clause, making it O(1) per page. The ungrouped path was not given the same treatment, creating an exploitable asymmetry.

**Why existing checks fail:**

The only guard on the cursor is the length check (`after.len() != 12`). There is no cap on the offset value, no rate limiting specific to this path, and no query timeout enforced at the application layer.

## Impact Explanation

An attacker can make the indexer RPC unresponsive by repeatedly submitting crafted `get_transactions` requests with `group_by_transaction: false` and a cursor encoding a near-maximum offset. This saturates the database connection pool, blocking all other indexer queries. This matches the allowed impact: **Note (0–500 points) — Any local RPC API crash** (functional unresponsiveness), and potentially **Low (501–2000 points) — Any other important performance improvements for CKB**, given the attack requires zero cost and is repeatable indefinitely.

## Likelihood Explanation

The RPC is exposed over HTTP with no authentication by default. The cursor format (12 bytes, last 4 = offset LE) is directly observable from any legitimate `last_cursor` response and trivially reversible. The attack requires a single crafted JSON-RPC call and can be repeated at will. No special privileges, PoW, or prior knowledge beyond the cursor format is required.

## Recommendation

Replace the SQL `OFFSET`-based pagination in `get_tx_with_cell` with a keyset approach mirroring `get_tx_with_cells`. The cursor should encode `(last_tx_id, last_io_type, last_io_index)` as a composite key, and the query should use `WHERE (tx_id, io_type, io_index) > (last_tx_id, last_io_type, last_io_index)` instead of `OFFSET`. If `OFFSET` must be retained as an interim fix, cap it at `limit` (the only legitimate value is at most `limit - 1` to re-anchor within a tie group), rejecting any cursor whose offset field exceeds `limit`.

## Proof of Concept

1. Enable rich-indexer on a CKB node with a script appearing in many transactions.
2. Call `get_transactions` with `group_by_transaction: false`, `limit: 1`, no `after` → receive a valid 12-byte `last_cursor`.
3. Decode the cursor; replace the last 4 bytes with `\xff\xff\xff\x7f` (i32::MAX in LE).
4. Re-encode and submit as `after` in a new `get_transactions` call.
5. Observe query latency grows proportionally to the crafted offset, confirming O(offset) DB scan cost. Repeat in a loop to sustain DoS.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
