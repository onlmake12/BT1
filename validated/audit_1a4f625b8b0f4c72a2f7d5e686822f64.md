### Title
Double-Charged `update_fee` in NEAR Receiver Merkle Callback — (File: `target_chains/near/receiver/src/lib.rs`)

---

### Summary

The `verify_wormhole_merkle_callback` function artificially lowers the "before" storage baseline by subtracting `update_fee / storage_byte_cost` bytes, then **also** passes `Some(self.update_fee)` as an `additional_fee` to `refund_storage_usage`. This causes the protocol fee to be charged twice for every price update submitted via the accumulator/Merkle path — the primary update mechanism on NEAR.

---

### Finding Description

The NEAR receiver has two divergent callback implementations for processing price updates:

**`verify_wormhole_batch_callback` (line 243) — correct:**
```rust
let storage = env::storage_usage();   // plain snapshot
// ...
Self::refund_storage_usage(account_id, storage, env::storage_usage(),
    env::attached_deposit(), Some(self.update_fee))
```

**`verify_wormhole_merkle_callback` (lines 317–325) — divergent:**
```rust
let storage = (env::storage_usage() as u128)
    .checked_sub(
        self.update_fee
            .as_yoctonear()
            .checked_div(env::storage_byte_cost().as_yoctonear())
            .ok_or(Error::ArithmeticOverflow)?,
    )
    .ok_or(Error::InsufficientDeposit)
    .and_then(|s| u64::try_from(s).map_err(|_| Error::ArithmeticOverflow))?;
// ...
Self::refund_storage_usage(account_id, storage, env::storage_usage(),
    env::attached_deposit(), Some(self.update_fee))   // fee passed again
```

Both callbacks share the same comment block (lines 239–242 and 313–316) claiming the subtraction "forces the caller to add the required fee to the deposit." The batch callback's code is correct — it does not subtract anything. The merkle callback, however, implements the comment literally by lowering the `before` baseline, while still passing `Some(self.update_fee)` as an additional fee.

`refund_storage_usage` (lines 671–675) computes cost as:

```
cost = storage_byte_cost × (after − before) + additional_fee
```

With the artificially lowered `before`:

```
diff  = (actual_after − actual_before) + floor(update_fee / storage_byte_cost)
cost  = storage_byte_cost × actual_diff
      + storage_byte_cost × floor(update_fee / storage_byte_cost)
      + update_fee
      ≈ storage_byte_cost × actual_diff + 2 × update_fee
```

The fee is charged once through the inflated `diff` and once through `additional_fee`.

---

### Impact Explanation

Every caller of `update_price_feeds` with accumulator/Merkle data (the current primary update path, line 174–198) is overcharged by approximately `update_fee`. A user who attaches exactly `storage_cost + update_fee` — the correct documented amount — will receive `Error::InsufficientDeposit` because the contract demands `storage_cost + 2 × update_fee`. Excess funds are locked in the contract and not recoverable by the user.

---

### Likelihood Explanation

The accumulator/Merkle path is the only supported update format for current Pyth price data on NEAR (the legacy batch path handles an older format). Every unprivileged user submitting a price update triggers this code path unconditionally. No special role or condition is required.

---

### Recommendation

Remove the fee-based subtraction from the storage baseline in `verify_wormhole_merkle_callback` and use the same pattern as `verify_wormhole_batch_callback`:

```rust
// Before (buggy):
let storage = (env::storage_usage() as u128)
    .checked_sub(self.update_fee.as_yoctonear()
        .checked_div(env::storage_byte_cost().as_yoctonear())
        ...)?;

// After (correct):
let storage = env::storage_usage();
```

The `update_fee` is already charged correctly through the `additional_fee` parameter in `refund_storage_usage`.

---

### Proof of Concept

1. User calls `update_price_feeds` with accumulator update data, attaching exactly `storage_cost + update_fee` yoctoNEAR.
2. `update_price_feeds` (line 166) forwards the deposit to `verify_wormhole_merkle_callback` via `.with_attached_deposit(env::attached_deposit())` (line 188).
3. Inside the callback, `before = storage_usage − floor(update_fee / storage_byte_cost)`.
4. After price feed storage writes, `after = storage_usage + actual_diff`.
5. `refund_storage_usage` computes `cost = storage_byte_cost × (actual_diff + floor(update_fee / storage_byte_cost)) + update_fee ≈ storage_byte_cost × actual_diff + 2 × update_fee`.
6. `deposit.checked_sub(cost)` returns `None` → `Err(Error::InsufficientDeposit)`.
