The inverted acquisition order is:

accept_peer:  peer_store (289) → peer_registry.write (292)
ban_session:  peer_registry.write (262) → peer_store (264)

However, the critical question for a **bounty-grade** finding is whether these two paths can actually execute **concurrently on separate OS threads** in production. That depends on whether the CKB network event loop uses a multi-threaded executor and whether `accept_peer` and `ban_session` can be dispatched to different threads simultaneously. Both `peer_store.lock()` and `peer_registry.write()` appear to be synchronous blocking locks (not `tokio::sync` async-aware locks), so if they run on separate tokio worker threads, the classic ABBA deadlock is reachable.

### Title
Lock-Order Inversion Deadlock Between `peer_store` and `peer_registry` in `NetworkState` — (`network/src/network.rs`)

### Summary
`accept_peer` acquires `peer_store` then `peer_registry.write()`, while `ban_session` acquires `peer_registry.write()` then `peer_store`. Under concurrent execution on a multi-threaded runtime, this ABBA ordering produces a deadlock that permanently hangs the network event loop.

### Finding Description
`NetworkState::accept_peer` holds `peer_store` (line 289) while waiting for `peer_registry.write()` (line 292). Concurrently, `NetworkState::ban_session` holds `peer_registry.write()` (line 262) while waiting for `peer_store.lock()` (line 264). The code itself acknowledges the risk with a comment at line 287–288. Both locks are synchronous (blocking) primitives. On a multi-threaded tokio runtime, two tasks dispatched to separate worker threads can enter this cycle and deadlock indefinitely. [4](#0-3) [5](#0-4) 

### Impact Explanation
A deadlock in the network event loop causes the node to stop processing all P2P messages — no new connections, no block/tx relay, no sync. From the network's perspective this is equivalent to a crash. The node is silently dropped from the P2P graph without restarting.

### Likelihood Explanation
An attacker can deliberately trigger both paths simultaneously: flood the node with inbound connections (triggering `accept_peer`) while sending protocol-violating messages from existing peers (triggering `ban_session`). Both are reachable from unauthenticated P2P. High-rate inbound churn combined with concurrent ban events is a realistic and low-cost attack scenario.

### Recommendation
Establish and enforce a **global lock acquisition order**: always acquire `peer_store` before `peer_registry` (or vice versa) on every code path. In `ban_session`, restructure so `peer_registry.write()` is released before `peer_store.lock()` is acquired — e.g., extract the peer's address inside the registry lock, drop the registry lock, then call `peer_store.lock().ban_addr(...)`.

### Proof of Concept
```
Thread A: accept_peer()
  → peer_store.lock()          // acquired ✓
  → peer_registry.write()      // BLOCKS — Thread B holds it

Thread B: ban_session()
  → peer_registry.write()      // acquired ✓
  → peer_store.lock()          // BLOCKS — Thread A holds it

→ DEADLOCK
```

                .filter(|peer| !peer.is_whitelist)
                .map(|peer| peer.connected_addr.clone())

            info!(

                addr,
                duration.as_secs(),
                reason
            );

                metrics.ckb_network_ban_peer.inc();
            }
            if let Some(peer) = self.with_peer_registry_mut(|reg| reg.remove_peer(session_id)) {
                let message = format!("Ban for {} seconds, reason: {}", duration.as_secs(), reason);
                self.peer_store.lock().ban_addr(
                    &peer.connected_addr,
                    duration.as_millis() as u64,
                    reason,
                );

                {
                    debug!("Disconnect failed {:?}, error: {:?}", peer.session_id, err);
                }
            }
        } else {
            debug!(
                "Ban session({}) failed: not found in peer registry or it is on the whitelist",
                session_id
            );
        }
    }
