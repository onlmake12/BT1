### Title
`RevealedWithCallback` and `EntropyEventsV2.Revealed` Emitted Unconditionally Even When No Callback Is Executed (EOA Requester) - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`'s `revealWithCallback` legacy code path (`else` branch, lines 661–702), `RevealedWithCallback` and `EntropyEventsV2.Revealed` (with `callbackFailed = false`) are emitted unconditionally — even when the requester is an EOA and no callback was executed at all. Off-chain scripts monitoring these events are misled into believing a callback was successfully delivered.

---

### Finding Description

`revealWithCallback` contains two code paths gated on whether the provider has set a gas limit:

```solidity
if (
    req.gasLimit10k != 0 &&
    req.callbackStatus == EntropyStatusConstants.CALLBACK_NOT_STARTED
) {
    // NEW FLOW: excessivelySafeCall, events gated on success/failure
} else {
    // LEGACY FLOW: direct call, events emitted unconditionally
}
```

In the **new flow** (lines 574–660), events are correctly gated: `RevealedWithCallback` is only emitted on `success == true`, and `CallbackFailed` is emitted on `success == false`. [1](#0-0) 

In the **legacy flow** (lines 661–702), the code checks `extcodesize(callAddress)` and conditionally skips the callback for EOA requesters — but then emits both events **unconditionally** regardless of whether the callback was executed:

```solidity
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(...);
}
// ← No else branch, no guard

emit RevealedWithCallback(reqV1, userContribution, providerContribution, randomNumber);
emit EntropyEventsV2.Revealed(
    provider, callAddress, sequenceNumber, randomNumber,
    userContribution, providerContribution,
    false,      // callbackFailed = false even when NO callback was made
    bytes(""),
    gasUsed,
    bytes("")
);
``` [2](#0-1) 

When `len == 0` (EOA requester):
- The `_entropyCallback` call is **skipped entirely**
- `RevealedWithCallback` is still emitted — its name implies a callback occurred
- `EntropyEventsV2.Revealed` is emitted with `callbackFailed = false` — explicitly asserting the callback did not fail, when in fact it was never attempted

---

### Impact Explanation

Off-chain scripts (relayers, indexers, dashboards, keeper bots) that monitor `RevealedWithCallback` or `EntropyEventsV2.Revealed` with `callbackFailed = false` will incorrectly conclude that a callback was successfully delivered to the requester. In reality, no callback was made. This is the direct analog of the original report: an off-chain observer is deceived into treating a non-executed action as a successful one. Concretely:

- A keeper bot that tracks fulfilled requests by watching `RevealedWithCallback` will mark an EOA request as "callback delivered" when it was not.
- A monitoring system using `EntropyEventsV2.Revealed` with `callbackFailed = false` as a success signal will produce false positives for every EOA-originated request on legacy providers.

---

### Likelihood Explanation

The trigger path is fully permissionless:

1. Any EOA can call `requestWithCallback` (or `requestV2`) on a **legacy provider** — one that has never called `setDefaultGasLimit`, leaving `provider.defaultGasLimit == 0`. This causes `req.gasLimit10k` to be stored as `0`, routing all reveals through the legacy `else` branch.
2. Anyone (including the provider) can then call `revealWithCallback`. Since `req.gasLimit10k == 0`, the `else` branch is entered.
3. Since the requester is an EOA, `extcodesize == 0`, the callback is skipped, and both misleading events are emitted.

Legacy providers exist on mainnet today (any provider that registered before `setDefaultGasLimit` was introduced and has not re-registered). [3](#0-2) 

---

### Recommendation

In the legacy `else` branch, gate the event emission on whether a callback was actually executed:

```solidity
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber, provider, randomNumber
    );
    emit RevealedWithCallback(reqV1, userContribution, providerContribution, randomNumber);
    emit EntropyEventsV2.Revealed(
        provider, callAddress, sequenceNumber, randomNumber,
        userContribution, providerContribution,
        false, bytes(""), gasUsed, bytes("")
    );
} else {
    // EOA requester: emit a distinct event or include a `noCallbackMade` flag
    emit RevealedNoCallback(reqV1, userContribution, providerContribution, randomNumber);
}
```

Alternatively, add a boolean `callbackExecuted` field to `EntropyEventsV2.Revealed` so off-chain consumers can distinguish the two cases without relying on a separate event.

---

### Proof of Concept

1. Identify (or deploy) a legacy Entropy provider with `defaultGasLimit == 0` (i.e., has never called `setDefaultGasLimit`).
2. As an EOA (`msg.sender` has no contract code), call:
   ```solidity
   entropy.requestV2{value: fee}(legacyProvider, 0);
   // or: entropy.requestWithCallback{value: fee}(legacyProvider, userContribution)
   ```
   This stores `req.gasLimit10k = 0` and `req.requester = EOA_address`.
3. The provider calls:
   ```solidity
   entropy.revealWithCallback(legacyProvider, sequenceNumber, userContribution, providerContribution);
   ```
4. Since `req.gasLimit10k == 0`, the `else` branch is entered. Since `extcodesize(EOA_address) == 0`, the `_entropyCallback` call is skipped.
5. Observe that `RevealedWithCallback` and `EntropyEventsV2.Revealed` (with `callbackFailed = false`) are emitted despite no callback having been executed.
6. An off-chain script filtering for `RevealedWithCallback` events records this as a successful callback delivery — which is false. [4](#0-3)
