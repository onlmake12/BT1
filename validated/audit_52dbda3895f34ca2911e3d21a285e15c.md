### Title
Deprecated `.transfer()` for ETH Refund Causes DoS for Smart Contract Callers of `verifyUpdate()` - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` uses the deprecated `.transfer()` method to refund excess ETH to `msg.sender`. After the Istanbul EVM upgrade (EIP-1884), the 2300-gas stipend forwarded by `.transfer()` is insufficient for any contract caller whose `receive()` or `fallback()` function performs more than a trivial operation. This causes a permanent, silent denial-of-service for all smart contract integrators of the Lazer on-chain verification API that overpay the fee.

### Finding Description
In `PythLazer.sol`, the `verifyUpdate()` function is the sole on-chain entry point for Lazer price update verification. It accepts a fee and refunds any excess:

```solidity
// lazer/contracts/evm/src/PythLazer.sol, lines 73-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [1](#0-0) 

`.transfer()` hard-caps the gas forwarded to the recipient at 2300. Since EIP-1884 (Istanbul), `SLOAD` costs 800 gas, `LOG` costs 375+ gas, and any non-trivial `receive()` or `fallback()` in the calling contract will exceed this stipend. The call reverts, and because the revert propagates up, the entire `verifyUpdate()` transaction fails — the caller loses their gas but receives no refund and no price verification.

The official Pyth Lazer integration guide explicitly shows smart contracts as the primary consumer of `verifyUpdate()`: [2](#0-1) 

### Impact Explanation
Any smart contract integrating PythLazer that:
1. Calls `verifyUpdate()` with `msg.value > verification_fee` (overpays — common when using a fixed ETH amount or when the owner changes the fee), **and**
2. Has a `receive()` or `fallback()` function that uses more than 2300 gas (e.g., emits an event, reads a storage slot, or calls another contract)

...will have every `verifyUpdate()` call permanently revert. The contract cannot receive the refund, so it cannot complete verification. This is a complete denial-of-service of the Lazer on-chain verification path for affected integrators, with no workaround available to the caller short of sending exactly `verification_fee` wei — which is fragile since the owner can change `verification_fee` at any time.

### Likelihood Explanation
- `verifyUpdate()` is `external payable` with no access control — reachable by any unprivileged Lazer updater or relayer contract.
- The Pyth Lazer developer documentation and SDK examples show smart contracts as the canonical consumer of `verifyUpdate()`.
- Overpayment is the natural pattern: callers query `verification_fee()` off-chain, but the fee can be updated by the owner between the query and the transaction, making exact payment unreliable.
- Any integrating contract with a non-trivial `receive()` (e.g., emitting an event for accounting) will trigger the bug.

### Recommendation
Replace `.transfer()` with a low-level `.call{value: ...}("")` and check the return value, consistent with the pattern already used elsewhere in the Pyth codebase (e.g., `Echo.sol` line 295):

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Refund failed");
}
```

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.13;

import {PythLazer} from "path/to/PythLazer.sol";

contract VulnerableIntegrator {
    PythLazer public pythLazer;
    uint256 public lastRefund;

    constructor(address _pythLazer) {
        pythLazer = PythLazer(_pythLazer);
    }

    function updatePrice(bytes calldata update) external payable {
        // Overpays intentionally (common pattern: send 0.01 ETH, fee is 1 wei)
        pythLazer.verifyUpdate{value: msg.value}(update);
    }

    // Non-trivial receive: emits an event (uses >2300 gas)
    receive() external payable {
        emit Received(msg.value); // SSTORE + LOG > 2300 gas → transfer() reverts
    }

    event Received(uint256 amount);
}
```

**Result**: Every call to `VulnerableIntegrator.updatePrice()` with `msg.value > 1 wei` reverts at the `.transfer()` line inside `verifyUpdate()`. The integrator contract is permanently unable to use Lazer on-chain verification unless it sends exactly `verification_fee` wei — a value that can be changed by the owner at any time.
