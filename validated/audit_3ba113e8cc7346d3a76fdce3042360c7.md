### Title
Backdated `publishTime = 0` in `requestPriceUpdatesWithCallback` Permanently Locks User Fees - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` accepts any `publishTime` value as low as `0` — there is no lower-bound validation. Because `executeCallback` calls `parsePriceFeedUpdates` with both `minAllowedPublishTime` and `maxAllowedPublishTime` set to `req.publishTime`, a request with `publishTime = 0` (or any timestamp for which no Pyth historical data exists) can never be fulfilled. Since the Echo contract has no cancel or refund mechanism, the user's fee payment is permanently locked.

---

### Finding Description

In `requestPriceUpdatesWithCallback`, the only timestamp guard is an upper-bound check:

```solidity
require(publishTime <= block.timestamp + 60, "Too far in future");
``` [1](#0-0) 

There is no corresponding lower-bound check (e.g., `publishTime > 0` or `publishTime >= block.timestamp - someWindow`). A caller may pass `publishTime = 0` or any arbitrarily old Unix timestamp.

When `executeCallback` is later called, the Pyth oracle is invoked as:

```solidity
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
    value: pythFee
}(
    updateData,
    priceIds,
    SafeCast.toUint64(req.publishTime),   // minAllowedPublishTime
    SafeCast.toUint64(req.publishTime)    // maxAllowedPublishTime
);
``` [2](#0-1) 

Both bounds are set to the exact stored `publishTime`. The Pyth accumulator contract accepts a price only when `publishTime >= minAllowedPublishTime && publishTime <= maxAllowedPublishTime`. [3](#0-2) 

This means the provider must supply price data whose `publishTime` is **exactly** equal to `req.publishTime`. For `publishTime = 0` (or any epoch for which Pyth has no stored data), no valid `updateData` can ever satisfy this constraint, so `parsePriceFeedUpdates` will return an empty array or revert, making the request permanently unfulfillable.

The user's fee is stored in `req.fee`:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [4](#0-3) 

This fee is only ever released inside `executeCallback` when the request is successfully fulfilled. There is no `cancelRequest`, refund, or recovery function anywhere in the Echo contract. A grep for `cancel`, `refund`, or `recover` in the echo directory returns no matches. The fee is permanently locked.

---

### Impact Explanation

A user who submits `publishTime = 0` (or any timestamp predating Pyth's historical data) loses the entire fee they paid. The fee is held in the contract's request storage and can never be extracted — not by the user, not by the provider, and not by the admin (the admin's `withdrawFees` only covers `_state.accruedFeesInWei`, which is the protocol's cut, not `req.fee`). [5](#0-4) 

This is a direct loss of user funds with no recovery path, matching the high-severity classification of the reference report.

---

### Likelihood Explanation

The trigger is a single unprivileged call to `requestPriceUpdatesWithCallback` with `publishTime = 0`. No special role or key is required. A UI bug, copy-paste error, or a default-zero value in a frontend form is sufficient to trigger this. The contract itself imposes no barrier. The likelihood is realistic and the scenario is analogous to the PoolTogether UI-bug scenario cited by the judge.

---

### Recommendation

Add a lower-bound check in `requestPriceUpdatesWithCallback` to ensure `publishTime` is not unreasonably old:

```solidity
require(
    publishTime >= block.timestamp - MAX_PAST_PUBLISH_TIME,
    "publishTime too far in the past"
);
require(publishTime <= block.timestamp + 60, "Too far in future");
```

Additionally, implement a `cancelRequest` function that allows the original requester to reclaim their fee after a timeout period (e.g., if the request has not been fulfilled within a configurable window), mirroring the defense-in-depth approach.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback(provider, 0, priceIds, gasLimit)` with sufficient `msg.value`.
2. The check `0 <= block.timestamp + 60` passes. The request is stored with `req.publishTime = 0` and `req.fee = msg.value - pythFeeInWei`.
3. Provider attempts `executeCallback`. Inside, `parsePriceFeedUpdates` is called with `minAllowedPublishTime = 0` and `maxAllowedPublishTime = 0`. No real Pyth price data has `publishTime == 0`; the call reverts or returns an empty result.
4. The request remains active indefinitely. No cancel or refund path exists.
5. User's fee is permanently locked in the Echo contract. [6](#0-5) [7](#0-6)

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where

        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.

        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
