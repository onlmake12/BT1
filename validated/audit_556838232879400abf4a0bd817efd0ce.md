### Title
Bounced `OP_PARSE_PRICE_FEED_UPDATES` Response Permanently Traps User Funds in Pyth TON Contract — (File: `target_chains/ton/contracts/contracts/Main.fc`, `target_chains/ton/contracts/contracts/Pyth.fc`)

---

### Summary

The Pyth TON contract's `recv_internal` handler unconditionally discards all bounced messages. When a `parse_price_feed_updates` or `parse_unique_price_feed_updates` response is sent to a `target_address` that bounces it, the user's excess TON (everything above the reserved update fee) is permanently trapped in the Pyth contract with no refund path.

---

### Finding Description

In `Main.fc`, the internal message handler checks the bounce flag and silently returns for every bounced message:

```
if (flags & 1) { ;; ignore all bounced messages
    return ();
}
``` [1](#0-0) 

In `Pyth.fc`, `send_price_feeds_response` uses `raw_reserve` with `RESERVE_MODE::INCREASE_BY_ORIGINAL_BALANCE` (mode 4) to retain `original_balance + update_fee` in the contract, then forwards all remaining balance to `target_address` using `SENDMODE::CARRY_ALL_BALANCE` (mode 128) with the bounce flag set (`0x18`):

```
raw_reserve(update_fee, RESERVE_MODE::INCREASE_BY_ORIGINAL_BALANCE);
...
send_raw_message(begin_cell()
    .store_uint(0x18, 6)          ;; bounce flag set
    .store_slice(target_address)
    .store_coins(0)
    .store_uint(1, MSG_SERIALIZE_BITS)
    .store_ref(response.end_cell())
    .end_cell(),
    SENDMODE::CARRY_ALL_BALANCE);
``` [2](#0-1) 

The contract itself acknowledges this gap in a comment:

> "Note: This message is bounceable. If the target contract rejects the message, the excess TON will remain in this contract and won't be automatically refunded to the original sender." [3](#0-2) 

When the target contract bounces the response (e.g., due to gas exhaustion, malformed `custom_payload`, or an upgrade incompatibility), the bounced TON returns to the Pyth contract. The bounce handler in `Main.fc` drops it silently. There is no `on_bounce` handler, no refund to `sender_address`, and no recovery mechanism. The excess TON accumulates in the contract permanently.

The `emit_error` path (used when Pyth itself fails to parse the update data) does send a response back to `sender_address`, but it uses `send_raw_message(msg, 64)` (carry remaining gas only) and does not refund the user's TON deposit: [4](#0-3) 

So neither the success-then-bounce path nor the internal-error path returns the user's TON.

---

### Impact Explanation

Any user who calls `OP_PARSE_PRICE_FEED_UPDATES` or `OP_PARSE_UNIQUE_PRICE_FEED_UPDATES` and whose `target_address` bounces the response loses all TON sent beyond the update fee. The funds accumulate in the Pyth contract with no on-chain recovery path. This is a direct, irreversible loss of user funds triggered by a normal protocol interaction. [5](#0-4) 

---

### Likelihood Explanation

The `target_address` is a caller-supplied parameter. In the documented integration pattern (`User → Pyth → Protocol`), the target is a third-party DeFi contract. Bounces occur routinely in TON when:
- The target contract runs out of gas processing the response.
- The `custom_payload` is malformed or triggers an exception in the target.
- The target contract has been upgraded and its message handler no longer accepts the old format.

These are realistic, non-adversarial conditions. Any integrator whose target contract fails for any reason silently loses their excess TON. The README itself warns about this behavior, confirming it is a known, unresolved issue: [6](#0-5) 

---

### Recommendation

1. **Add a bounce handler** in `Main.fc` that, on receiving a bounced `OP_PARSE_PRICE_FEED_UPDATES` or `OP_PARSE_UNIQUE_PRICE_FEED_UPDATES` response, reads the embedded `sender_address` from the bounced body and refunds the excess TON to that address.

2. **Alternatively**, store the `sender_address` and the refundable amount in contract state before sending the response, and clear it only upon confirmed delivery. On bounce, use the stored state to issue the refund.

3. **At minimum**, change the outbound message flag from `0x18` (bounceable) to `0x10` (non-bounceable) and ensure the update fee is reserved correctly, so that a failed delivery does not silently trap funds. This trades bounce-based recovery for explicit error handling in the target.

---

### Proof of Concept

1. User calls `OP_PARSE_PRICE_FEED_UPDATES` sending 1 TON, with `target_address` set to a contract that always throws on receive (or is temporarily broken due to an upgrade).
2. Pyth validates the update data, calls `send_price_feeds_response`.
3. `raw_reserve(update_fee, 4)` retains `original_balance + update_fee` in the Pyth contract.
4. `send_raw_message(..., 128)` forwards ~0.99 TON (minus gas) to `target_address` with bounce flag.
5. `target_address` throws; TON bounces back to Pyth.
6. `Main.fc` line 35: `if (flags & 1) { return (); }` — bounce is silently dropped.
7. The ~0.99 TON is now permanently held in the Pyth contract. The user has no on-chain mechanism to recover it. [1](#0-0) [7](#0-6)
