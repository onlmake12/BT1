Audit Report

## Title
Minimum Fee Check Uses Raw Byte Size Instead of Transaction Weight, Allowing Below-Minimum Effective Fee Rate Transactions — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces `min_fee_rate` using only the serialized byte size of a transaction. The true admission weight — `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)` — is never checked at admission time. A transaction with near-maximum cycle consumption but minimal byte size passes the fee gate while its actual effective fee rate (as computed by `TxEntry::fee_rate()` for ordering and eviction) can be orders of magnitude below the configured minimum. No second fee check is performed after `verify_rtx` determines actual cycles, and the transaction is relayed to all peers.

## Finding Description

`get_transaction_weight` defines the canonical weight used for pool ordering and eviction: [1](#0-0) 

`TxEntry::fee_rate()` uses this weight-based calculation: [2](#0-1) 

However, the admission gate `check_tx_fee` uses only `tx_size`, with a comment explicitly acknowledging the discrepancy as intentional: [3](#0-2) 

In `_process_tx`, `pre_check` (which calls `check_tx_fee`) runs before `verify_rtx` determines actual cycles. After verification, `TxEntry` is created with the real cycles and the already-approved fee — no second fee check is performed: [4](#0-3) 

The same pattern appears in `readd_detached_tx` — `check_tx_fee` passes with size-only check, `verify_rtx` runs, and `TxEntry` is created without a weight-based re-check: [5](#0-4) 

The code comment at `tx-pool/src/util.rs` L42–44 confirms the gap is known and no guard was added after cycle determination.

## Impact Explanation

This matches the **High** bounty impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

With default configuration (`min_fee_rate = 1000 shannons/KW`, `max_tx_verify_cycles = 70,000,000`):

| Parameter | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 70,000,000 |
| `weight` | `max(200, 70_000_000 × 0.000_170_571_4)` = **11,940** |
| Fee check threshold | `1000 × 200 / 1000` = **200 shannons** |
| Actual effective fee rate | `200 × 1000 / 11,940` ≈ **16 shannons/KW** |
| Configured minimum | **1000 shannons/KW** |

Each admitted transaction forces every receiving node to execute up to 70,000,000 script cycles of verification before admitting and relaying it further. The fee-to-CPU-cost ratio is ~62× worse than the minimum the protocol intends to enforce. An attacker with a set of pre-funded UTXOs can submit a batch of such transactions, causing sustained high CPU load across the network at minimal cost.

## Likelihood Explanation

The attack is reachable by any unprivileged user via `send_transaction` RPC or P2P relay. Required preconditions: (1) a deployed cell containing a tight-loop script consuming near-maximum cycles, (2) UTXOs locked by that script, (3) fees just above `min_fee_rate.fee(tx_size)`. The code comment itself confirms the gap is known and intentional as a "cheap check," meaning no guard was added after cycle determination. The attack is repeatable for each UTXO the attacker controls.

## Recommendation

After `verify_rtx` returns `verified.cycles`, perform a second fee check using the true weight before creating the `TxEntry`:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Re-check fee rate with actual weight now that cycles are known
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64())), snapshot));
}

let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

The size-only pre-check in `check_tx_fee` can remain as an early rejection for obviously under-fee transactions, but must not be the sole gate. The same fix must be applied in `readd_detached_tx`.

## Proof of Concept

1. Deploy a lock script binary (~100 bytes) that runs a tight loop consuming ~70,000,000 cycles. Store it in a cell dep.
2. Create UTXOs locked by that script.
3. Construct a transaction spending one such UTXO:
   - `tx_size` ≈ 200 bytes (minimal inputs/outputs/witnesses)
   - `cycles` ≈ 70,000,000 (from the loop script)
   - `fee` = `min_fee_rate.fee(200)` = 200 shannons
4. Submit via `send_transaction` RPC.
5. Observe: `check_tx_fee` passes (`200 >= 200`); `verify_rtx` runs 70M cycles; `TxEntry` is created and submitted; transaction is relayed to peers.
6. Verify: `entry.fee_rate()` = `FeeRate::calculate(200, get_transaction_weight(200, 70_000_000))` ≈ 16 shannons/KW — 62× below `min_fee_rate`.
7. Repeat for each pre-funded UTXO to sustain CPU load across the network.
