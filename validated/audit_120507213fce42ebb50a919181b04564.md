### Title
`PythAggregatorV3` Returns Stale Prices via `getPriceUnsafe` With No Sequencer or Staleness Check — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

### Summary
`PythAggregatorV3.sol`, Pyth's official Chainlink-compatible adapter contract, calls `pyth.getPriceUnsafe()` in every price-reading function (`latestRoundData`, `latestAnswer`, `getRoundData`, `decimals`, `latestTimestamp`). There is no staleness check and no L2 sequencer uptime check. On L2 chains (Arbitrum, Optimism, Base), when the sequencer goes down, no price updates can be submitted to the Pyth contract. When the sequencer resumes, all these functions silently return the stale cached price as if it were current, with no revert or warning.

### Finding Description
Every public price-reading function in `PythAggregatorV3` delegates to `getPriceUnsafe`:

```solidity
// latestAnswer — line 53-56
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// latestRoundData — line 99-119
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    ...
}

// getRoundData — line 76-97
function getRoundData(uint80 _roundId) external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    ...
}
```

`getPriceUnsafe` is documented to return the price regardless of age. There is no call to `getPriceNoOlderThan`, no check of `publishTime` against `block.timestamp`, and no L2 sequencer feed check. The contract is explicitly marketed as a drop-in Chainlink replacement for protocols migrating to Pyth on EVM chains including L2s. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Protocols that integrate `PythAggregatorV3` as a Chainlink-compatible oracle on L2 chains (Arbitrum, Optimism, Base) are exposed to stale price exploitation:

1. L2 sequencer goes offline — no `updateFeeds` transactions can land.
2. Sequencer resumes — the Pyth on-chain price is now arbitrarily stale (hours old).
3. An attacker calls `latestRoundData()` / `latestAnswer()` through the integrating protocol (e.g., a lending market) and receives the stale price with no revert.
4. The attacker exploits the price discrepancy: e.g., borrows against overvalued collateral or avoids liquidation on undervalued debt.

The `updatedAt` field returned by `latestRoundData` is set to `price.publishTime`, which is the Pyth publish timestamp — not `block.timestamp`. A consuming protocol that checks `updatedAt` against `block.timestamp` will correctly detect staleness, but protocols that trust the Chainlink interface without additional checks (the common case for Chainlink migrations) will not. [4](#0-3) 

### Likelihood Explanation
- `PythAggregatorV3` is the official Pyth SDK adapter for Chainlink migrations, actively recommended in Pyth documentation and the migration guide.
- L2 sequencer outages are a known, recurring event on Arbitrum and Optimism.
- Protocols migrating from Chainlink are likely to replicate their existing integration pattern, which does not include a staleness check on `updatedAt` because Chainlink's push oracle keeps prices fresh automatically.
- An unprivileged attacker only needs to submit a transaction at the right moment after sequencer recovery — no special access required. [5](#0-4) 

### Recommendation
Replace `getPriceUnsafe` with `getPriceNoOlderThan` in all read functions, using a configurable `maxAge` parameter set at construction time:

```solidity
uint256 public maxAge; // e.g., 60 seconds

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    ...
}
```

Additionally, for L2 deployments, add an optional Chainlink sequencer uptime feed check (as documented by Chainlink for L2 sequencer feeds) that reverts if the sequencer has been down or has recently restarted within a grace period.

### Proof of Concept
1. Deploy `PythAggregatorV3` on Arbitrum pointing to a Pyth price feed.
2. Simulate sequencer downtime: no `updateFeeds` calls land for 2 hours.
3. Sequencer resumes. The Pyth on-chain price is now 2 hours stale.
4. Call `latestRoundData()` — it returns the 2-hour-old price with no revert.
5. Call `latestAnswer()` — same stale price returned.
6. A lending protocol using this adapter will compute collateral values based on the stale price, enabling undercollateralized borrowing or blocking valid liquidations. [6](#0-5)
