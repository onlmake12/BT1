### Title
Onion/Tor Peer Connections Bypass IP-Based Ban Enforcement — (File: `network/src/peer_store/ban_list.rs`)

---

### Summary

CKB's peer ban mechanism uses IP addresses as the sole identifier for ban enforcement. When a peer connects via an onion (Tor) address, `multiaddr_to_socketaddr` returns `None`, causing `is_addr_banned` to silently return `false` via `unwrap_or_default()`. Symmetrically, `ban_addr` silently skips creating a ban entry for onion addresses. A misbehaving peer connecting via Tor can never be effectively banned and can reconnect immediately after every forced disconnect.

---

### Finding Description

**Root cause — `ban_list.rs` `is_addr_banned`:**

```rust
pub fn is_addr_banned(&self, addr: &Multiaddr) -> bool {
    multiaddr_to_socketaddr(addr)
        .map(|socket_addr| self.is_ip_banned(&socket_addr.ip()))
        .unwrap_or_default()   // ← returns false for any non-IP address
}
```

`multiaddr_to_socketaddr` returns `None` for onion multiaddrs (e.g., `/onion3/…`). The `unwrap_or_default()` silently returns `false`, meaning every onion peer passes the ban check unconditionally. [1](#0-0) 

**Root cause — `peer_store_impl.rs` `ban_addr`:**

```rust
pub(crate) fn ban_addr(&mut self, addr: &Multiaddr, timeout_ms: u64, ban_reason: String) {
    if let Some(addr) = multiaddr_to_socketaddr(addr) {   // ← None for onion
        let network = ip_to_network(addr.ip());
        self.ban_network(network, timeout_ms, ban_reason)  // ← never reached
    }
    self.addr_manager.remove(addr);   // address removed, but no ban entry created
}
```

For an onion address the `if let Some` arm is never entered, so no `BannedAddr` record is written. The peer is removed from the addr manager but is not banned. [2](#0-1) 

**Call chain that exposes the gap:**

`ban_session` (triggered by any protocol misbehaviour) calls `ban_addr` with `peer.connected_addr`: [3](#0-2) 

`accept_peer` calls `peer_store.is_addr_banned` before admitting a new connection: [4](#0-3) 

Because `is_addr_banned` always returns `false` for onion addresses, the reconnecting peer is admitted every time.

**`set_ban` RPC also IP-only:**

The `set_ban` RPC parses the address as an `IpNetwork` and rejects anything else, so operators have no manual path to block a misbehaving onion peer either. [5](#0-4) 

**Analogy to the reported vulnerability:**

| Original report | CKB analog |
|---|---|
| `hostname` used as permission key | IP address used as ban key |
| `http://example.com` and `https://example.com` share the same hostname | TCP peer at `1.2.3.4` and an onion peer (same operator) share no ban-list entry |
| Fix: block HTTP or use full origin | Fix: track onion addresses in ban list or block onion reconnects |

---

### Impact Explanation

A misbehaving peer connecting via Tor can:

1. Trigger `ban_session` (e.g., by sending an identify message for the wrong network, or sending invalid sync data), be disconnected, but have no ban entry written.
2. Immediately reconnect via the same onion address — `is_addr_banned` returns `false` — and repeat the cycle indefinitely.
3. Continuously waste node CPU (block/header verification), bandwidth, and connection slots.
4. Cannot be stopped by the operator via `set_ban` because the RPC only accepts IP/subnet strings.

The identify protocol explicitly calls `ban_session` for peers on the wrong network: [6](#0-5) 

Every such ban is silently ineffective for onion peers.

---

### Likelihood Explanation

Onion support (`listen_on_onion = true`) is a documented, supported configuration option: [7](#0-6) 

It is not enabled by default, but nodes that enable it for privacy (a stated use-case) are fully exposed. An attacker needs only a Tor client and knowledge that the target node accepts onion connections — both are trivially available. No privileged access, key material, or majority hashpower is required.

---

### Recommendation

1. **Track onion addresses in `BanList`**: Add a `HashSet<Multiaddr>` (or similar) for non-IP addresses alongside the existing `HashMap<IpNetwork, BannedAddr>`. Update `ban_addr` and `is_addr_banned` to consult this set when `multiaddr_to_socketaddr` returns `None`.
2. **Update `set_ban` RPC**: Accept onion multiaddr strings in addition to IP/subnet strings so operators can manually ban misbehaving onion peers.
3. **Fail closed**: Change `unwrap_or_default()` in `is_addr_banned` to `unwrap_or(true)` as a conservative interim measure — treat addresses that cannot be resolved to a socket address as banned unless explicitly allowed.

---

### Proof of Concept

```
1. Run a CKB node with `listen_on_onion = true`.
2. Attacker connects via Tor to the node's onion address.
3. Attacker sends an IdentifyMessage with a mismatched network flag.
4. Node calls ban_session(session_id, BAN_ON_NOT_SAME_NET, "not same network").
5. ban_session → ban_addr(onion_multiaddr, …):
     multiaddr_to_socketaddr(onion_multiaddr) == None
     → if let Some block skipped
     → no BannedAddr entry written
6. Peer is disconnected.
7. Attacker immediately reconnects via the same onion address.
8. accept_peer → is_addr_banned(onion_multiaddr):
     multiaddr_to_socketaddr(onion_multiaddr) == None
     → unwrap_or_default() == false
     → peer admitted
9. Goto step 3 — loop indefinitely, exhausting node resources.
```
