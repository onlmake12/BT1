### Title
Inflated `total_tx_size` / `total_tx_cycles` Due to Overwrite After Ancestor-Eviction in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new-entry statistics (`total_tx_size`, `total_tx_cycles`) are computed **before** the ancestor-eviction step, but the final assignment that writes those values back happens **after** the eviction step has already correctly decremented the running totals. The final assignment silently overwrites the correct post-eviction totals with the stale pre-eviction snapshot, permanently inflating both counters by the sizes/cycles of every evicted entry.

---

### Finding Description

`add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```
1. (total_tx_size, total_tx_cycles) = self.total_tx_size + entry.size   // snapshot BEFORE eviction
2. evicts = self.check_and_record_ancestors(&mut entry)                  // may call remove_entry_and_descendants
                                                                         //   → remove_entry
                                                                         //   → update_stat_for_remove_tx
                                                                         //   which CORRECTLY decrements self.total_tx_size
3. self.total_tx_size  = total_tx_size   // OVERWRITES the decremented value
   self.total_tx_cycles = total_tx_cycles
``` [1](#0-0) 

`updated_stat_for_add_tx` captures a snapshot of the current totals plus the new entry's contribution: [2](#0-1) 

`check_and_record_ancestors` can evict entries via `remove_entry_and_descendants` when the ancestor count exceeds `max_ancestors_count` due to `cell_ref_parents`: [3](#0-2) 

Each evicted entry correctly decrements `self.total_tx_size` inside `remove_entry` → `update_stat_for_remove_tx`: [4](#0-3) [5](#0-4) 

But the final two lines of `add_entry` unconditionally overwrite `self.total_tx_size` and `self.total_tx_cycles` with the stale pre-eviction snapshot: [6](#0-5) 

**Net result:** after one eviction event that removes entries totalling `E` bytes and `C` cycles, the pool counters read `T + entry.size` instead of the correct `T - E + entry.size`. The error accumulates with every subsequent eviction event.

The eviction path is triggered when:
1. A new transaction's ancestor count exceeds `max_ancestors_count`, **and**
2. The excess is attributable to `cell_ref_parents` — pool transactions that reference (as a cell dep) an output that the new transaction is spending as an input. [7](#0-6) 

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict transactions from the pool: [8](#0-7) 

An inflated `total_tx_size` causes `limit_size` to believe the pool is over capacity when it is not, triggering spurious evictions of legitimate pending transactions. Repeated triggering compounds the inflation, progressively degrading pool capacity. The same inflated value is exposed to operators and tooling via the `tx_pool_info` RPC: [9](#0-8) 

Concrete effects:
- **Spurious `Reject::Full` rejections** for valid transactions submitted by honest users.
- **Premature eviction** of legitimate pending/proposed transactions, reducing effective pool throughput.
- **Incorrect pool statistics** reported via `tx_pool_info`, misleading operators and fee-estimation logic.
- The inflation is **permanent** (until the pool is cleared or the node restarts) and **cumulative** across multiple eviction events.

---

### Likelihood Explanation

The trigger condition requires a transaction whose ancestor count exceeds `max_ancestors_count` (default 25) specifically because of `cell_ref_parents`. An unprivileged tx-pool submitter can engineer this:

1. Submit a chain of 25 transactions `T1 → T2 → … → T25` (each spending the previous output).
2. Submit transaction `D` that uses `T25`'s output as a **cell dep** (not an input), causing `edges.deps[T25.output]` to record `D`.
3. Submit a new transaction `N` that **spends** `T25`'s output as an input. `N`'s ancestor count is 26 (T1…T25 + D via `cell_ref_parents`), exceeding the limit of 25. The eviction branch fires, removes `D`, and leaves `total_tx_size` inflated by `D.size`.

This requires no privileged access, no key material, and no majority hash power — only the ability to submit transactions to the mempool, which is available to any RPC caller or P2P peer.

---

### Recommendation

Compute the new totals **after** `check_and_record_ancestors` completes (so any eviction-driven decrements are already reflected in `self.total_tx_size`), or re-read `self.total_tx_size` at the point of the final assignment:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }

    // Validate that adding this entry won't overflow, but do NOT capture the snapshot yet.
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Re-compute AFTER evictions have already decremented the counters.
    self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

This ensures that any `update_stat_for_remove_tx` calls made during ancestor eviction are not silently discarded.

### Proof of Concept

Scenario (unit-test level, no fork required):

```
1. Fill the pool with a chain: T1 → T2 → … → T25
   (each Ti spends Ti-1's output; 25 = max_ancestors_count)

2. Submit D: spends some unrelated UTXO, but declares T25's output as a cell dep.
   → edges.deps[T25.output] = {D.short_id}
   → total_tx_size = S_chain + S_D

3. Submit N: spends T25's output as an input.
   → get_tx_ancenstors(N):
       ancestors = {T1…T25}  (26 entries including N itself)
       cell_ref_parents = {D}  (D uses T25.output as dep)
   → ancestors_count = 26 > max_ancestors_count (25)
   → ancestors_count - cell_ref_parents.len() = 25 ≤ 25  → eviction branch
   → remove_entry_and_descendants(D):
       update_stat_for_remove_tx(D.size, D.cycles)
       self.total_tx_size = S_chain  (correctly decremented)
   → add_entry final lines:
       self.total_tx_size = total_tx_size  // = S_chain + S_D + S_N  (stale snapshot!)

4. Assert: pool_map.total_tx_size == S_chain + S_N  (correct)
           pool_map.total_tx_size == S_chain + S_D + S_N  (actual — inflated by S_D)
```

The inflation of `S_D` bytes persists in `total_tx_size` for the lifetime of the pool, causing `limit_size` to evict legitimate transactions prematurely.

    fn get_tx_ancenstors(
        &self,
        entry: &TransactionView,
    ) -> (
        HashSet<ProposalShortId>,
        HashSet<ProposalShortId>,
        HashSet<ProposalShortId>,
    ) {
        let mut parents: HashSet<ProposalShortId> =
            HashSet::with_capacity(entry.inputs().len() + entry.cell_deps().len());
        let mut cell_ref_parents: HashSet<ProposalShortId> = Default::default();

        for input in entry.inputs() {
            let input_pt = input.previous_output();
            if let Some(deps) = self.edges.deps.get(&input_pt) {
                cell_ref_parents.extend(deps.iter().cloned());
                parents.extend(deps.iter().cloned());
            }

            let id = ProposalShortId::from_tx_hash(&input_pt.tx_hash());
            if self.links.inner.contains_key(&id) {
                parents.insert(id);
            }
        }
        for cell_dep in entry.cell_deps() {
            let dep_pt = cell_dep.out_point();
            let id = ProposalShortId::from_tx_hash(&dep_pt.tx_hash());
            if self.links.inner.contains_key(&id) {
                parents.insert(id);
            }
        }

        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        (ancestors, parents, cell_ref_parents)
    }

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
