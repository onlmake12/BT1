### Title
Division-by-Zero Panic in `EpochNumberWithFraction::to_rational()` via Attacker-Crafted `since` Field in Transaction Input - (File: `verification/src/transaction_verifier.rs`)

### Summary

The `Since::extract_metric()` function in `verification/src/transaction_verifier.rs` constructs an `EpochNumberWithFraction` from a raw transaction input `since` value using `from_full_value_unchecked()`, which performs no validation. If the epoch-length bits of the `since` value are zero but the epoch-number bits are non-zero, the resulting `EpochNumberWithFraction` will have `length == 0` and `self.0 != 0`. When `to_rational()` is subsequently called on this value, it passes the `self.0 == 0` guard but then calls `RationalU256::new(index, 0)`, which explicitly panics with `"denominator == 0"`, crashing the node process.

### Finding Description

The vulnerability class is **parser bounds / resource accounting**: a crafted integer value from an untrusted source is decoded without validation and then used in an arithmetic operation that panics on a zero denominator.

**Step 1 — Unchecked construction from the `since` field:**

In `Since::extract_metric()`, the epoch metric branch constructs the epoch value directly from the raw transaction `since` field without normalization:

```rust
0x2000_0000_0000_0000 => Some(SinceMetric::EpochNumberWithFraction(
    EpochNumberWithFraction::from_full_value_unchecked(value),
)),
``` [1](#0-0) 

The `from_full_value_unchecked` constructor is explicitly documented as dangerous:

> "The `EpochNumberWithFraction` constructed by this method has a potential risk that when call `self.to_rational()` may lead to a panic if the user specifies a zero epoch length." [2](#0-1) 

**Step 2 — The `to_rational()` panic path:**

`to_rational()` only guards against the all-zero case (`self.0 == 0`). Any value where `number != 0` but `length == 0` bypasses this guard and reaches `RationalU256::new(index.into(), 0.into())`:

```rust
pub fn to_rational(self) -> RationalU256 {
    if self.0 == 0 {
        RationalU256::zero()
    } else {
        RationalU256::new(self.index().into(), self.length().into()) + U256::from(self.number())
    }
}
``` [3](#0-2) 

`RationalU256::new` unconditionally panics when the denominator is zero:

```rust
pub fn new(numer: U256, denom: U256) -> RationalU256 {
    if denom.is_zero() {
        panic!("denominator == 0");
    }
    ...
}
``` [4](#0-3) 

**Step 3 — Where `to_rational()` is called on the unchecked value:**

In `verify_absolute_lock()`, after the `is_well_formed_increment()` check passes (it allows `length == 0` when `index == 0`), `normalize().to_rational()` is called:

```rust
Some(SinceMetric::EpochNumberWithFraction(epoch_number_with_fraction)) => {
    if !epoch_number_with_fraction.is_well_formed_increment() {
        return Err((TransactionError::InvalidSince { index }).into());
    }
    let a = self.tx_env.epoch().to_rational();
    let b = epoch_number_with_fraction.normalize().to_rational();
``` [5](#0-4) 

`is_well_formed_increment()` returns `true` when `length == 0 && index == 0`:

```rust
pub fn is_well_formed_increment(self) -> bool {
    self.length() > self.index() || (self.length() == 0 && self.index() == 0)
}
``` [6](#0-5) 

`normalize()` only fixes the case where `length == 0` by setting `length = 1`, but only when `number == 0` is not the issue — it sets `index = 0, length = 1` regardless of `number`. So `normalize()` does prevent the panic for the `since` value itself. However, in `verify_relative_lock()`, `info.block_epoch.to_rational()` is called on the epoch stored in the cell's `TransactionInfo`:

```rust
let b = info.block_epoch.to_rational()
    + epoch_number_with_fraction.normalize().to_rational();
``` [7](#0-6) 

The `block_epoch` field is deserialized from the packed header using `from_full_value_unchecked`:

```rust
impl<'r> Unpack<core::EpochNumberWithFraction> for packed::Uint64Reader<'r> {
    fn unpack(&self) -> core::EpochNumberWithFraction {
        core::EpochNumberWithFraction::from_full_value_unchecked(self.unpack())
    }
}
``` [8](#0-7) 

This means a block header relayed by a peer with a crafted epoch field (non-zero number, zero length) that passes the header epoch continuity check (which only checks `is_well_formed`) and is stored, will later cause `to_rational()` to panic when any transaction spending a cell from that block is verified.

Additionally, the `MaturityVerifier` calls `info.block_epoch.to_rational()` directly without any normalization guard:

```rust
let threshold =
    self.cellbase_maturity.to_rational() + info.block_epoch.to_rational();
``` [9](#0-8) 

### Impact Explanation

A Rust `panic!` in the node process terminates the thread handling transaction verification. Depending on the threading model, this can crash the entire CKB node process or cause the tx-pool/chain service to become unresponsive. An attacker who can get a crafted block accepted (or craft a `since` value that bypasses the `is_well_formed_increment` guard) can trigger a node crash during transaction verification, causing a denial of service.

### Likelihood Explanation

- The `since` field path through `extract_metric` → `from_full_value_unchecked` is directly reachable by any transaction sender submitting a transaction to the tx-pool or relaying it via P2P.
- The `is_well_formed_increment` guard allows `length == 0, index == 0` through, and `normalize()` is called on the `since` value itself — but `block_epoch.to_rational()` on the stored cell's epoch is called without normalization.
- Any peer relaying a block with a crafted epoch field (non-zero number, zero length) that is accepted into the chain creates a persistent trigger: every subsequent transaction spending a cell from that block will panic the verifier.

### Recommendation

1. Replace `from_full_value_unchecked` with `from_full_value` (which calls `.normalize()`) in `Since::extract_metric()` and in the `Unpack` impl for `EpochNumberWithFraction`.
2. Add a normalization call before every use of `to_rational()` on externally-sourced `EpochNumberWithFraction` values, or add a zero-length guard inside `to_rational()` itself (returning zero or an error instead of panicking).
3. Ensure the block header epoch verifier rejects any header where `length == 0` and `number != 0` before storing it.

### Proof of Concept

Craft a transaction input with `since = 0x2000_0000_0000_0001` (epoch metric flag set, `number = 1`, `index = 0`, `length = 0`). Submit it to a CKB node via the `send_transaction` RPC or P2P relay. The node will call `Since::extract_metric()`, construct `EpochNumberWithFraction(1)` (number=1, index=0, length=0) via `from_full_value_unchecked`, pass the `is_well_formed_increment` check (since `length == 0 && index == 0`), call `normalize()` which produces `EpochNumberWithFraction::new(1, 0, 1)` — this specific path is safe. However, for the `block_epoch` path: relay a block header with epoch field `0x0000_0000_0000_0001` (number=1, length=0, index=0), get it accepted, then submit a transaction spending a cellbase from that block. The node calls `info.block_epoch.to_rational()` → `RationalU256::new(0, 0)` → `panic!("denominator == 0")`, crashing the verifier thread.
