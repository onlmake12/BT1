### Title
Unbounded `feeInWei` in `setProviderFee` / `register` Enables Arithmetic Overflow DoS and Instant Fee Griefing — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract allows any registered provider to set their `feeInWei` to an arbitrary `uint128` value with no upper bound. This is the direct analog of the StarBaseDCA `_FEE_RATE_` issue: the same three entry points (`register`, `setProviderFee`, `setProviderFeeAsFeeManager`) accept an unbounded fee parameter and write it to storage without validation. Setting `feeInWei` to `type(uint128).max` causes `getFeeV2` to overflow and revert in Solidity ≥0.8, permanently DoS-ing all requests to that provider. There is also no timelock, so a provider can front-run users by raising the fee between a user's fee-query and their request transaction.

---

### Finding Description

Three functions in `Entropy.sol` write `feeInWei` to provider storage without any upper-bound check:

**1. `register` (line 129):**
```solidity
provider.feeInWei = feeInWei;   // no bounds check
``` [1](#0-0) 

**2. `setProviderFee` (lines 818–819):**
```solidity
uint128 oldFeeInWei = provider.feeInWei;
provider.feeInWei = newFeeInWei;   // no bounds check
``` [2](#0-1) 

**3. `setProviderFeeAsFeeManager` (lines 845–846):**
```solidity
uint128 oldFeeInWei = providerInfo.feeInWei;
providerInfo.feeInWei = newFeeInWei;   // no bounds check
``` [3](#0-2) 

`getFeeV2` computes the total fee as:
```solidity
return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
``` [4](#0-3) 

If `feeInWei = type(uint128).max` and `pythFeeInWei > 0` (which is the normal production state), this addition overflows and reverts under Solidity 0.8 checked arithmetic. `requestHelper` calls `getFeeV2` internally:
```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [5](#0-4) 

So every `request` / `requestV2` call to that provider reverts. There is no timelock on any of the three fee-setting paths; `EntropyGovernance.setPythFee` similarly has no timelock or cap. [6](#0-5) 

---

### Impact Explanation

- **Overflow DoS**: A provider who sets `feeInWei = type(uint128).max` causes `getFeeV2` to revert for every caller, making the provider's service permanently unusable until the provider lowers the fee. Any on-chain integrator contract that calls `getFeeV2` before `requestV2` will also revert, bricking those integrations.
- **Front-running / fee griefing**: Because there is no timelock, a provider can atomically raise their fee between a user's off-chain fee query and their on-chain `request` transaction. The user's transaction reverts with `InsufficientFee`, and the IEntropyV2 interface explicitly warns that excess `msg.value` is **not refunded**, so users who over-send to avoid this lose the excess to the protocol. [7](#0-6) 

---

### Likelihood Explanation

Provider registration is **permissionless** — any address can call `register`. A malicious actor can register as a provider, attract users or be set as the default provider, and then call `setProviderFee(type(uint128).max)` in a single transaction. No privileged access beyond being a registered provider is required. The `feeManager` role (also permissionlessly assignable by the provider) has the same capability via `setProviderFeeAsFeeManager`.

---

### Recommendation

1. Add an upper bound constant and enforce it in all three fee-setting paths:
   ```solidity
   uint128 public constant MAX_PROVIDER_FEE_IN_WEI = <reasonable_cap>;
   require(newFeeInWei <= MAX_PROVIDER_FEE_IN_WEI, "Fee too high");
   ```
2. Apply the same cap in `register`, `setProviderFee`, and `setProviderFeeAsFeeManager`.
3. Consider a timelock or minimum notice period before fee increases take effect, so existing users can cancel pending integrations.

---

### Proof of Concept

1. Attacker calls `register(type(uint128).max, commitment, ..., chainLength, uri)` — sets `feeInWei = type(uint128).max` with no revert.
2. Any user (or contract) calls `getFeeV2(attacker, 0)`:
   - `getProviderFee` returns `type(uint128).max`
   - `type(uint128).max + pythFeeInWei` (where `pythFeeInWei > 0`) overflows → **revert**
3. Any call to `request(attacker, ...)` or `requestV2(attacker, ...)` also reverts at line 234 for the same reason.
4. If the attacker is the default provider (or becomes it), `getFeeV2()` (no-arg) also reverts, breaking all integrations that use the default provider. [1](#0-0) [4](#0-3)

        address provider,

    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;

            oldFeeInWei,
