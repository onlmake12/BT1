### Title
Subscription Manager Without `receive()` Permanently Locks Funds in `Scheduler.withdrawFunds` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol` sends native tokens to `msg.sender` (the subscription manager) via `.call{value: amount}("")` in `withdrawFunds`. If the subscription manager is a contract that lacks a `receive` or payable `fallback` function, every withdrawal attempt reverts. Because there is no mechanism to transfer the manager role to a different address, the subscription balance is permanently locked.

---

### Finding Description

`createSubscription` records `msg.sender` as the immutable manager of the new subscription: [1](#0-0) 

`withdrawFunds` is gated by `onlyManager` and unconditionally pushes ETH to `msg.sender`: [2](#0-1) 

The critical transfer line: [3](#0-2) 

No `transferManager`, `setManager`, or equivalent function exists anywhere in `Scheduler.sol` or `SchedulerState.sol`. The `onlyManager` modifier confirms the manager mapping is read-only after creation: [4](#0-3) 

If the manager address is a contract without a `receive` function (e.g., a multisig, a DAO treasury, or any integration contract that omits ETH receipt), the low-level call returns `false`, the `require` reverts, and the balance can never be extracted. There is no admin escape hatch for per-subscription balances either.

A secondary instance of the same pattern exists in `_processFeesAndPayKeeper`, where keeper payment failure causes the entire `updatePriceFeeds` call to revert: [5](#0-4) 

---

### Impact Explanation

Any ETH deposited into a subscription whose manager is a contract without `receive` is permanently locked. The subscription balance cannot be withdrawn, and there is no admin recovery path. This constitutes a direct, irreversible loss of user funds.

---

### Likelihood Explanation

Moderate. DeFi protocols, DAOs, and multisig-controlled integrations routinely create subscriptions programmatically from contracts. Many such contracts (e.g., Gnosis Safe with no ETH fallback, proxy contracts, governance executors) do not implement `receive`. The pattern of "contract creates subscription → contract cannot withdraw" is a realistic integration scenario.

---

### Recommendation

1. **Allow manager transfer**: Add a `transferManager(uint256 subscriptionId, address newManager)` function callable only by the current manager, so a contract that cannot receive ETH can delegate the manager role to an EOA or a contract that can.
2. **Allow withdrawal to an explicit recipient**: Change `withdrawFunds` to accept a `recipient` address parameter (validated against the manager), rather than always pushing to `msg.sender`.
3. **Optionally check receivability**: Before accepting a contract address as manager, verify it can receive ETH (e.g., via a small test transfer or ERC-165 check), though option 1 or 2 is simpler and more robust.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IScheduler {
    function createSubscription(...) external payable returns (uint256);
    function withdrawFunds(uint256 subscriptionId, uint256 amount) external;
}

// Contract with NO receive() function
contract NoReceiveManager {
    IScheduler scheduler;

    constructor(address _scheduler) { scheduler = IScheduler(_scheduler); }

    function createSub(...) external payable returns (uint256 id) {
        // msg.sender == address(this), so this contract becomes the manager
        id = scheduler.createSubscription{value: msg.value}(...);
    }

    function tryWithdraw(uint256 id, uint256 amount) external {
        // Always reverts: scheduler tries to push ETH to address(this),
        // but address(this) has no receive(), so call returns false,
        // and require(sent, "Failed to send funds") reverts.
        scheduler.withdrawFunds(id, amount);
    }
    // No receive() — ETH push from scheduler always fails
}
```

Steps:
1. Deploy `NoReceiveManager` pointing at the live `Scheduler` proxy.
2. Call `createSub` with sufficient ETH — subscription is created, `NoReceiveManager` is the manager.
3. Call `tryWithdraw` — transaction reverts with `"Failed to send funds"` every time.
4. No other address can call `withdrawFunds` (blocked by `onlyManager`).
5. Funds are permanently locked.

        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;
