### Title
Caller-Controlled `providerToCredit` in `Echo.executeCallback` Allows Fee Theft After Exclusivity Period - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
In `Echo.executeCallback`, the `providerToCredit` parameter is caller-supplied and only validated against `req.provider` during the exclusivity window. After that window expires, any unprivileged caller can pass an arbitrary address as `providerToCredit`, redirecting the requester's stored fee (`req.fee`) away from the legitimate provider to any address — including the attacker's own.

### Finding Description
When a user calls `requestPriceUpdatesWithCallback`, they pay a fee that is split: the Pyth portion is immediately accrued, and the remainder is stored in the request as `req.fee`:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [1](#0-0) 

This stored `req.fee` is intended to compensate `req.provider` — the provider assigned at request time. In `executeCallback`, the fee is credited to the caller-supplied `providerToCredit`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The only guard enforcing `providerToCredit == req.provider` is the exclusivity-period check:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
``` [3](#0-2) 

Once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, this guard is skipped entirely. Any caller may then pass their own address as `providerToCredit`, causing `req.fee` (paid by the original requester) to be credited to the attacker rather than to `req.provider`. The legitimate provider receives nothing for the work they were supposed to be compensated for.

This is a direct receiver-mismatch: the fee payer (`req.requester`) intended to compensate `req.provider`, but the actual recipient is an arbitrary attacker-controlled address.

### Impact Explanation
- The legitimate provider (`req.provider`) loses the fee they are owed for fulfilling the callback.
- An attacker gains `req.fee` (minus the Pyth update fee they must supply as `msg.value`) by calling `executeCallback` with their own address as `providerToCredit` after the exclusivity period.
- Because `req.fee` is set at request time and can be large (it scales with `callbackGasLimit` and `feePerGasInWei`), the stolen amount can be significant.
- The requester's funds are permanently redirected to the wrong recipient; there is no recovery path.

### Likelihood Explanation
- No privileged access is required. Any externally-owned account or contract can call `executeCallback` after the exclusivity period.
- The attacker only needs to monitor the chain for `PriceUpdateRequested` events, wait for `req.publishTime + exclusivityPeriodSeconds` to pass, and supply valid Pyth `updateData` for the requested `priceIds` at `req.publishTime` (this data is publicly available from Hermes).
- The attacker pays the Pyth update fee (`pythFee`) as `msg.value` but profits by `req.fee - pythFee`, which is positive whenever the provider fee exceeds the Pyth update fee — the normal case.

### Recommendation
After the exclusivity period, `providerToCredit` should still be constrained to `req.provider`, or the fee should always be credited directly to `req.provider` regardless of who calls `executeCallback`:

```solidity
// Always credit the assigned provider, not a caller-supplied address
_state.providers[req.provider].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

If the intent is to allow a different relayer to be credited after the exclusivity period, the address should at minimum be validated as a registered provider, and the original `req.provider` should receive a penalty share rather than nothing.

### Proof of Concept
1. Alice calls `requestPriceUpdatesWithCallback(legitimateProvider, publishTime, priceIds, gasLimit)` sending `fee` wei. `req.fee = fee - pythFeeInWei`, `req.provider = legitimateProvider`.
2. The exclusivity period (`publishTime + exclusivityPeriodSeconds`) elapses.
3. Attacker calls `executeCallback(attackerAddress, sequenceNumber, validUpdateData, priceIds)` with `msg.value = pythFee`.
4. The guard at line 114–121 is skipped because the exclusivity period has passed.
5. `_state.providers[attackerAddress].accruedFeesInWei += req.fee + pythFee - pythFee = req.fee`.
6. Attacker calls `withdrawAsFeeManager` or registers as a provider and calls `withdraw` to extract `req.fee` wei.
7. `legitimateProvider` receives zero compensation; Alice's fee is stolen.
