### Title
Unbounded `constructProviderCommitment` Loop Causes OOG-Permanent Request Freeze When `maxNumHashes == 0` — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When a provider's `maxNumHashes` is `0` (the **default, uninitialized state**), the guard that caps `req.numHashes` at request time is disabled. An attacker can accumulate a large number of unrevealed requests against such a provider, causing any subsequent reveal to execute an unbounded `keccak256` loop in `constructProviderCommitment`, exhausting the block gas limit and permanently freezing the request along with the user's paid fee.

---

### Finding Description

**`constructProviderCommitment` — unbounded loop** [1](#0-0) 

The function iterates exactly `numHashes` times with no gas guard. Each iteration executes `keccak256(bytes.concat(currentHash))`, costing roughly 50–100 gas. At ~30 M block gas limit, only ~300 k–600 k iterations are needed to OOG the transaction.

**The guard is disabled when `maxNumHashes == 0`** [2](#0-1) 

The condition `providerInfo.maxNumHashes != 0 && ...` means the entire check is skipped when `maxNumHashes` is zero. Because `maxNumHashes` is a field in `ProviderInfo` stored in a Solidity mapping, it **defaults to 0** for every provider that has not explicitly called `setMaxNumHashes`. This is the common case for any provider that registered before this field existed or simply never set it.

**`numHashes` grows unboundedly with unrevealed requests** [3](#0-2) 

`req.numHashes` is the distance between the current `assignedSequenceNumber` and `currentCommitmentSequenceNumber`. Each request that is not revealed increments `assignedSequenceNumber` while `currentCommitmentSequenceNumber` stays fixed, so `numHashes` grows by 1 per unrevealed request.

**No hard cap on concurrent requests** [4](#0-3) 

The 32-slot array is supplemented by an unbounded `requestsOverflow` mapping, so there is no protocol-level ceiling on the number of concurrent in-flight requests.

**`revealHelper` passes the stored `numHashes` directly into the loop** [5](#0-4) 

Once a request is stored with a large `numHashes`, every future reveal attempt for that request will re-execute the full loop. There is no recovery path: if the loop OOGs, the transaction reverts, the request remains active, and the user's fee is permanently locked.

---

### Impact Explanation

Any request whose stored `req.numHashes` exceeds the gas-per-block threshold (~300 k–600 k at current Ethereum block gas limits) can **never be revealed**. The user's paid fee is permanently frozen in the contract with no refund mechanism. The attacker does not need to be the requester — they only need to inflate `numHashes` for a target provider before a victim's request is made.

---

### Likelihood Explanation

- `maxNumHashes == 0` is the **default state** for all providers; it requires no special action by the provider.
- The `requestsOverflow` mapping imposes no ceiling on concurrent requests.
- An attacker with sufficient capital (fees × N requests) can engineer the condition. With a low-fee provider (e.g., 0.0001 ETH/request), reaching 300 k unrevealed requests costs ~30 ETH — expensive but within reach of a motivated attacker targeting high-value users.
- Organic accumulation (a provider that is slow to reveal) can also reach dangerous `numHashes` values without any attacker involvement.

---

### Recommendation

1. **Remove the `== 0` bypass**: enforce a hard protocol-level cap on `numHashes` regardless of the provider's `maxNumHashes` setting. A reasonable default (e.g., 1000) should be chosen based on gas benchmarks.
2. **Add a gas check inside `constructProviderCommitment`**: revert with a descriptive error if `numHashes` exceeds a safe constant, rather than silently OOGing.
3. **Require providers to set `maxNumHashes` during `register`**: make it a mandatory non-zero parameter so no provider can accidentally operate with the guard disabled.

---

### Proof of Concept

```
Preconditions:
  - Provider registered with maxNumHashes = 0 (default, never called setMaxNumHashes)
  - chainLength set large enough (e.g., 2^20)

Steps:
  1. Attacker calls requestWithCallback() N times (e.g., N = 400,000)
     without any intervening reveals or advanceProviderCommitment calls.
     After N requests: providerInfo.currentCommitmentSequenceNumber is unchanged,
     providerInfo.sequenceNumber = initial + N.

  2. The N-th request is stored with req.numHashes = N.

  3. Attacker (or anyone) calls revealWithCallback(provider, seqN, userContrib, providerContrib).
     -> revealHelper -> constructProviderCommitment(400_000, providerContrib)
     -> 400,000 × keccak256 iterations ≈ 20M–40M gas -> OOG revert.

  4. The request remains active forever. The user's fee is permanently locked.

Fuzz assertion:
  assert gasCost(constructProviderCommitment(N, x)) > block.gaslimit  for N >= threshold
```

    function constructProviderCommitment(
        uint64 numHashes,
        bytes32 revelation
    ) internal pure returns (bytes32 currentHash) {
        currentHash = revelation;
        while (numHashes > 0) {
            currentHash = keccak256(bytes.concat(currentHash));
            numHashes -= 1;
        }
    }
