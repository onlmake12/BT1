### Title
Entropy `revealWithCallback` Legacy Code Path Calls `_entropyCallback` Without Try/Catch, Permanently Locking Requests When Callback Reverts — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`'s `revealWithCallback`, the legacy code path (active when `req.gasLimit10k == 0`, which is the default for any provider whose `defaultGasLimit` is unset) calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly without any try/catch or safe-call wrapper. If the requester contract's callback reverts for any reason — including a wrong `getEntropy()` return value, an unimplemented interface, or a logic revert — the entire `revealWithCallback` transaction reverts. Because there is no cancellation or recovery path for callback-style requests, the request is permanently stuck and the user's already-credited fees are irrecoverable.

---

### Finding Description

`revealWithCallback` contains two execution branches:

**New flow** (lines 574–660): Activated when `req.gasLimit10k != 0`. Uses `excessivelySafeCall` to invoke the callback, catches reverts, and transitions the request to `CALLBACK_FAILED` state, enabling recovery.

**Old/legacy flow** (lines 661–702): Activated when `req.gasLimit10k == 0`. This is the default for every provider whose `providerInfo.defaultGasLimit == 0` (the zero-initialized default). The flow:

1. Calls `clearRequest(provider, sequenceNumber)` **before** the callback.
2. Checks `extcodesize(callAddress) != 0` (verifies the requester is a contract, but does **not** verify it implements `IEntropyConsumer`).
3. Calls `IEntropyConsumer(callAddress)._entropyCallback(sequenceNumber, provider, randomNumber)` with **no try/catch**.

If the callback reverts, Solidity unwinds all state changes including `clearRequest`, leaving the request active. Because the callback will deterministically revert on every retry (e.g., `getEntropy()` returns the wrong address → `"Only Entropy can call this function"`), the request can never be cleared.

The `extcodesize` check is the exact analog of ERC721's `transferFrom`: it confirms the target is a contract but does not confirm it implements the expected interface. The new flow's `excessivelySafeCall` is the analog of `safeTransferFrom`: it handles receiver non-compliance gracefully.

Fees are credited to the provider and Pyth at **request time** inside `requestHelper`:

```solidity
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

There is no `cancelRequest` or refund path for callback-style requests. The `reveal` (non-callback) path explicitly rejects callback requests:

```solidity
if (req.callbackStatus != EntropyStatusConstants.CALLBACK_NOT_NECESSARY) {
    revert EntropyErrors.InvalidRevealCall();
}
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

### Impact Explanation

- **Loss of user funds**: The user pays the full fee at request time. If the callback permanently reverts, the user never receives the random number but the fees are already distributed to the provider and Pyth with no refund mechanism.
- **Permanent DoS on the specific request**: The request slot is occupied indefinitely. Every `revealWithCallback` attempt by the provider reverts, wasting provider gas on each retry.
- **No recovery path**: Unlike the new flow (which moves to `CALLBACK_FAILED` and allows re-invocation), the old flow has no state transition for failure; the request stays in `CALLBACK_NOT_STARTED` forever. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The old flow is the **default** for all providers whose `defaultGasLimit` field is zero (the zero-initialized default). Any user integrating with such a provider and making any of the following common mistakes triggers the stuck-request condition:

- `getEntropy()` returns `address(0)` or a stale/wrong Entropy address (fails `require(msg.sender == entropy)`).
- The requester contract does not implement `_entropyCallback` at all (call reverts with no matching selector, unless a fallback exists).
- The user's `entropyCallback` implementation reverts under the conditions present at reveal time.

These are realistic integration mistakes, especially for new integrators. The `IEntropyConsumer` abstract contract's `_entropyCallback` enforces a strict `msg.sender == entropy` guard that is easy to misconfigure. [7](#0-6) [8](#0-7) 

---

### Recommendation

Wrap the callback invocation in the old flow with a try/catch (or replace it with `excessivelySafeCall`, already imported and used in the new flow). On callback failure, emit a `CallbackFailed` event and transition `callbackStatus` to `CALLBACK_FAILED` so the request can be retried or administratively cleared. Alternatively, add a `cancelRequest` function that allows the original requester to reclaim their fee if the request has been stuck for a configurable timeout period.

---

### Proof of Concept

1. Deploy `MaliciousConsumer` that calls `requestWithCallback(provider, userContribution)` but whose `getEntropy()` returns `address(0x1234)` (wrong address).
2. `requestHelper` credits fees to provider and Pyth; request stored with `callbackStatus = CALLBACK_NOT_STARTED`, `gasLimit10k = 0` (provider has default `defaultGasLimit == 0`).
3. Provider calls `revealWithCallback(provider, seqNum, userContrib, providerContrib)`.
4. Old flow executes: `clearRequest` is called, then `IEntropyConsumer(callAddress)._entropyCallback(...)` is invoked.
5. Inside `_entropyCallback`: `require(msg.sender == entropy)` fails → revert.
6. Entire transaction reverts; `clearRequest` is undone; request remains active.
7. Provider retries → same revert every time.
8. User has paid fees (already in `accruedFeesInWei`), will never receive a random number, and has no mechanism to cancel or recover funds. [9](#0-8) [10](#0-9)

        uint64 sequence,
        address provider,
        bytes32 randomNumber

        address entropy = getEntropy();
