### Title
`CyclesEstimator` Bypasses Cell Liveness Check, Allowing Script Execution Against Dead Cells - (File: `rpc/src/module/chain.rs`)

### Summary

The `CyclesEstimator` struct used by the `estimate_cycles` (and deprecated `dry_run_transaction`) RPC methods explicitly skips the cell liveness check, treating already-spent (dead) cells as live. This is the direct CKB analog of the ERC777 incomplete-state-accounting bug: just as `send`/`transfer`/`transferFrom` in the Pod contract operated on a balance that did not include the pending (unconsolidated) portion, `estimate_cycles` operates on cell state that does not respect the live/dead distinction — allowing any unprivileged RPC caller to execute arbitrary scripts against dead cells without paying fees.

---

### Finding Description

In `rpc/src/module/chain.rs`, the `CyclesEstimator` struct implements `CellProvider` as follows:

```rust
impl<'a> CellProvider for CyclesEstimator<'a> {
    fn cell(&self, out_point: &packed::OutPoint, eager_load: bool) -> CellStatus {
        let snapshot = self.shared.snapshot();
        snapshot
            .get_cell(out_point)
            .map(|mut cell_meta| {
                // ...
                CellStatus::live_cell(cell_meta)
            })  // treat as live cell, regardless of live or dead
            .unwrap_or(CellStatus::Unknown)
    }
}
``` [1](#0-0) 

The comment is explicit: **"treat as live cell, regardless of live or dead"**. The method calls `snapshot.get_cell()` — which returns cell metadata from the store for any cell that has ever existed — and unconditionally wraps the result in `CellStatus::live_cell(...)`. This bypasses the liveness gate that the normal `Snapshot::CellProvider` implementation enforces (returning `CellStatus::Unknown` for cells not in the live-cell set).

By contrast, the correct cell provider used during actual transaction submission (`resolve_tx_from_pool`) uses an `OverlayCellProvider` that properly distinguishes live, dead, and unknown cells:

```rust
pub(crate) fn resolve_tx_from_pool(...) -> Result<Arc<ResolvedTransaction>, Reject> {
    let snapshot = self.snapshot();
    let pool_cell = PoolCell::new(&self.pool_map, rbf);
    let provider = OverlayCellProvider::new(&pool_cell, snapshot);
    ...
    resolve_transaction(tx, &mut seen_inputs, &provider, snapshot)
``` [2](#0-1) 

The `CyclesEstimator::run()` then passes the resolved transaction (which may contain dead-cell inputs) directly into `ScriptVerifier::new(...).verify(max_cycles)`: [3](#0-2) 

The `max_cycles` used is `consensus.max_block_cycles` — the full block cycle budget. [4](#0-3) 

The same `CyclesEstimator` is reused by the deprecated `dry_run_transaction` endpoint: [5](#0-4) 

---

### Impact Explanation

An unprivileged RPC caller can:

1. Identify any dead cell (already spent on-chain) whose lock or type script is computationally expensive.
2. Craft a transaction referencing that dead cell as an input.
3. Repeatedly call `estimate_cycles` (or `dry_run_transaction`) with this transaction.
4. Each call causes the node to run the full `ScriptVerifier` against the dead cell's data, consuming up to `max_block_cycles` CPU cycles per request, with **zero fee payment**.

Because the attacker does not need to own or control the referenced cell (they only need its `OutPoint`), they can exploit any historically expensive script on-chain — including scripts they cannot unlock — to drive sustained CPU exhaustion on the node. The cycle estimate returned is also incorrect/misleading: the same transaction would fail with `TransactionFailedToResolve` if submitted to the actual tx-pool, because the dead cell would not resolve there.

---

### Likelihood Explanation

The `estimate_cycles` RPC is a standard, publicly documented endpoint reachable by any RPC caller without authentication. Dead cells are abundant on any live chain. No special privileges, keys, or majority hashpower are required. The attack requires only knowledge of an `OutPoint` for a dead cell with an expensive script, which is trivially discoverable from chain history.

---

### Recommendation

The `CyclesEstimator::cell()` implementation should respect cell liveness. The simplest fix is to delegate to the snapshot's standard `CellProvider` implementation (which already enforces liveness) rather than calling `get_cell()` directly:

```rust
impl<'a> CellProvider for CyclesEstimator<'a> {
    fn cell(&self, out_point: &packed::OutPoint, eager_load: bool) -> CellStatus {
        // Use the snapshot's CellProvider, which enforces liveness
        self.shared.snapshot().cell(out_point, eager_load)
    }
}
```

Alternatively, if estimation against dead cells is intentionally supported (e.g., for offline script debugging), the RPC should be rate-limited or restricted to authenticated/local callers only, and the documentation should clearly warn that results are not representative of actual submission behavior.

    pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();
                let tx_env = TxVerifyEnv::new_submit(tip_header);
                match ScriptVerifier::new(
                    Arc::new(resolved),
                    snapshot.as_data_loader(),
                    consensus,
                    Arc::new(tx_env),
                )
                .verify(max_cycles)
                {
                    Ok(cycles) => Ok(EstimateCycles {
                        cycles: cycles.into(),
                    }),

                        err,
                    )),
                }
            }
            Err(err) => Err(RPCError::custom_with_error(
