### Title
Permissionless `updatePriceFeeds` Enables Front-Running to Steal Keeper Fees and Cause Keeper DoS — (`Scheduler.sol`)

---

### Summary

The `updatePriceFeeds` function in the Pulse Scheduler is callable by any address with no access control. An attacker can monitor the mempool for a keeper's pending `updatePriceFeeds` transaction, copy the exact same `updateData`, and front-run the keeper with a higher gas price. The attacker's transaction succeeds and receives the keeper fee; the keeper's transaction then reverts with `TimestampOlderThanLastUpdate` because `status.priceLastUpdatedAt` was already set to the same timestamp.

---

### Finding Description

`updatePriceFeeds` is a fully permissionless `external` function. It:

1. Parses price feeds from `updateData` via Pyth.
2. Validates that the update timestamp is strictly greater than `status.priceLastUpdatedAt`.
3. Validates that update conditions (heartbeat or deviation) are met.
4. Writes `status.priceLastUpdatedAt = latestPublishTime`.
5. Calls `_processFeesAndPayKeeper(status, startGas, params.priceIds.length)`, which pays `msg.sender` the keeper fee. [1](#0-0) 

The shared mutable state is `status.priceLastUpdatedAt`. The staleness guard is:

```solidity
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(
        updateTimestamp,
        status.priceLastUpdatedAt
    );
}
``` [2](#0-1) 

Because `updatePriceFeeds` is permissionless and pays `msg.sender`, an attacker can:

1. Observe a keeper's pending `updatePriceFeeds(subscriptionId, updateData)` in the mempool.
2. Copy the identical `updateData` (which already satisfies the update conditions, since the keeper verified it).
3. Submit the same call with a higher gas price to land first in the same block.
4. The attacker's transaction succeeds: `priceLastUpdatedAt` is set to `T`, and the attacker receives the keeper fee.
5. The keeper's transaction reverts: `updateTimestamp (T) <= priceLastUpdatedAt (T)` → `TimestampOlderThanLastUpdate`.

This is structurally identical to the reported M-2 pattern: two callers of the same permissionless function share a monotonic timestamp state variable; the first writer wins and the second reverts.

---

### Impact Explanation

- **Keeper fee theft**: The attacker receives the full keeper fee on every update cycle by front-running with zero additional effort (just copying `updateData`).
- **Keeper DoS**: Legitimate keepers' transactions always revert, causing them to lose gas on every attempt. Rational keepers will stop submitting updates.
- **Oracle staleness risk**: If all legitimate keepers are driven out, only the attacker updates the oracle. The attacker can selectively withhold updates (e.g., during adverse price moves) to benefit from stale prices in downstream protocols.

---

### Likelihood Explanation

- Pyth Scheduler is deployed on EVM chains with public mempools (Ethereum, Arbitrum, etc.).
- `updateData` is fully visible in the pending transaction.
- No special knowledge or privileged access is required — the attacker only needs to copy calldata and pay a higher gas price.
- The economic incentive is clear: the attacker earns the keeper fee on every front-run.

---

### Recommendation

Add a caller-binding mechanism so that only the address that first committed to an update can claim the fee, or restrict `updatePriceFeeds` to a registered keeper whitelist. A simpler mitigation is to record the `msg.sender` of the first valid submission within a block and reject duplicate submissions in the same block:

```diff
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {
+   if (status.lastUpdaterBlock == block.number)
+       revert SchedulerErrors.AlreadyUpdatedThisBlock();
+   status.lastUpdaterBlock = block.number;
    ...
}
```

Alternatively, restrict `updatePriceFeeds` to a whitelist of registered keepers, similar to how Chainlink Automation restricts `performUpkeep`.

---

### Proof of Concept

**Setup:**
- Subscription configured with `heartbeatSeconds = 60`, `deviationThresholdBps = 100`.
- `status.priceLastUpdatedAt = T0`.
- At time `T0 + 60`, the heartbeat condition is met.

**Attack:**
1. Keeper constructs `updateData` containing a valid Pyth price update with `publishTime = T1 > T0`.
2. Keeper broadcasts `updatePriceFeeds(subscriptionId, updateData)` to the mempool.
3. Attacker observes the pending transaction, copies `updateData` verbatim.
4. Attacker submits `updatePriceFeeds(subscriptionId, updateData)` with higher gas price.
5. Attacker's tx lands first:
   - `_validateShouldUpdatePrices` passes (heartbeat elapsed, `T1 > T0`).
   - `status.priceLastUpdatedAt = T1`.
   - `_processFeesAndPayKeeper` pays the attacker.
6. Keeper's tx executes next:
   - `updateTimestamp = T1`, `status.priceLastUpdatedAt = T1`.
   - `T1 <= T1` → revert `TimestampOlderThanLastUpdate(T1, T1)`.
   - Keeper loses gas, receives nothing. [3](#0-2) [4](#0-3)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _validateShouldUpdatePrices(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams storage params,
        SchedulerStructs.SubscriptionStatus storage status,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal view returns (uint256) {
        // Use the most recent timestamp, as some asset markets may be closed.
        // Closed markets will have a publishTime from their last trading period.
        // Since we verify all updates share the same Pythnet slot, we still ensure
        // that all price feeds are synchronized from the same update cycle.
        uint256 updateTimestamp = 0;
        for (uint8 i = 0; i < priceFeeds.length; i++) {
            if (priceFeeds[i].price.publishTime > updateTimestamp) {
                updateTimestamp = priceFeeds[i].price.publishTime;
            }
        }

        // Calculate the minimum acceptable timestamp (clamped at 0)
        // The maximum acceptable timestamp is enforced by the parsePriceFeedUpdatesWithSlots call
        uint256 minAllowedTimestamp = (block.timestamp >
            PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            : 0;

        // Validate that the update timestamp is not too old
        if (updateTimestamp < minAllowedTimestamp) {
            revert SchedulerErrors.TimestampTooOld(
                updateTimestamp,
                block.timestamp
            );
        }

        // Reject updates if they're older than the latest stored ones
        if (
            status.priceLastUpdatedAt > 0 &&
            updateTimestamp <= status.priceLastUpdatedAt
        ) {
            revert SchedulerErrors.TimestampOlderThanLastUpdate(
                updateTimestamp,
                status.priceLastUpdatedAt
            );
        }
