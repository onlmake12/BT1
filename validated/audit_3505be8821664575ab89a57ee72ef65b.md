Audit Report

## Title
Check-After-Insert in `add_ask_for_txs` Allows Unbounded `unknown_tx_hashes` Growth with O(N) Mutex-Held Scan — (`sync/src/types/mod.rs`)

## Summary

`SyncState::add_ask_for_txs` unconditionally inserts all peer-supplied hashes into `unknown_tx_hashes` before evaluating the global size cap. When the global limit is exceeded but the per-peer count remains below `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, the function returns `Status::ignored()` without removing the just-inserted entries, allowing the queue to grow to ~84× the intended 50,000-entry cap. The per-peer counting loop also performs an O(N) full-queue scan while holding the mutex, causing severe lock contention as the queue grows.

## Finding Description

**Check-after-insert root cause:**

`add_ask_for_txs` acquires the mutex at line 1484 and unconditionally inserts all hashes (up to 32,767 per call via `.take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)`) at lines 1486–1504. [1](#0-0) 

The global limit check only follows at lines 1507–1509, after insertion is complete. The comment at line 1506 explicitly acknowledges this ordering: `"Check unknown_tx_hashes's length after inserting the arrival tx_hashes"`. [2](#0-1) 

When `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) but the per-peer counter is below `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767), the function returns `Status::ignored()` at line 1528 with no rollback of the inserted entries. [3](#0-2) 

**O(N) scan while holding the mutex:**

Every time the global limit is exceeded, lines 1516–1523 iterate over every entry in the queue while the mutex from line 1484 is still held. All concurrent callers block for the full duration of this scan. [4](#0-3) 

**Constants driving the math:**

- `MAX_RELAY_TXS_NUM_PER_BATCH` = 32,767 (max hashes per message) [5](#0-4) 
- `MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000 (intended global cap)
- `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = 32,767
- `MAX_RELAY_PEERS` = 128 [6](#0-5) 

Maximum reachable queue size: 128 × 32,766 = **4,194,048 entries ≈ 84× the intended limit**.

**Call chain:**

`TransactionHashesProcess::execute()` at line 49 calls `state.add_ask_for_txs(self.peer, tx_hashes)` after filtering only through `tx_filter`. An attacker bypasses this filter by using unique, previously-unseen hashes. [7](#0-6) 

**Rate limiter — insufficient mitigation:**

The rate limiter caps at 30 req/sec per (peer, message-type). However, a single message carries up to 32,767 hashes, so one message per peer is sufficient to inject the full per-peer quota. [8](#0-7) 

## Impact Explanation

**High — CKB network congestion with low attacker cost.**

1. **Memory bloat:** Each `UnknownTxHashPriority` entry holds a `Vec<PeerIndex>`, `Instant`, and `bool` alongside a 32-byte `Byte32` key. At 4.2M entries, heap usage grows by several hundred MB above the intended baseline, potentially causing OOM on constrained nodes.

2. **Relay stall via lock contention:** The O(N) scan at lines 1516–1523 holds the `unknown_tx_hashes` mutex while iterating up to 4.2M entries. With 128 peers sending at 30 req/sec (3,840 messages/sec once the queue exceeds 50,000), relay threads queue up waiting for the mutex. `pop_ask_for_txs` (which also acquires `unknown_tx_hashes`) is blocked, stalling transaction-request scheduling and degrading transaction propagation across the network.

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation

- Requires only establishing up to 128 peer connections to the target node — no PoW, no cryptographic keys, no stake.
- Each peer sends one `RelayTransactionHashes` message with 32,766 unique random `Byte32` hashes (random bytes, no validity requirement).
- The `tx_filter` pre-check is bypassed by using hashes not previously seen by the node.
- `Status::ignored()` does not disconnect or ban the peer, allowing sustained exploitation.
- The rate limiter (30 req/sec) is irrelevant since a single message per peer is sufficient to inject the full quota.

## Recommendation

1. **Check before insert:** Move the global and per-peer limit checks to before the insertion loop. Reject or truncate the batch if either limit would be exceeded.
2. **Maintain a per-peer counter separately:** Replace the O(N) full-queue scan with a `HashMap<PeerIndex, usize>` counter updated incrementally on insert/remove, making the per-peer check O(1).
3. **Return a ban status on global-limit breach:** When the global limit is reached, return `TooManyUnknownTransactions` (ban) rather than `Status::ignored()` to prevent repeated exploitation from the same peer.

## Proof of Concept

```
1. Attacker establishes 128 peer connections to the target CKB node.
2. Each peer generates 32,766 unique Byte32 values (random bytes, no PoW).
3. Peer 1 sends one RelayTransactionHashes message with 32,766 hashes.
   → queue = 32,766 (< 50,000) → Status::ok(), hashes retained.
4. Peer 2 sends one RelayTransactionHashes message with 32,766 unique hashes.
   → add_ask_for_txs inserts all 32,766 before checking.
   → queue = 65,532 (> 50,000) → limit check fires.
   → peer 2 per-peer count = 32,766 < 32,767 → Status::ignored(), hashes retained.
5. Repeat for peers 3–128.
   → Final queue ≈ 4,194,048 entries.
6. Any subsequent message from any peer (queue > 50,000) triggers the O(4.2M) scan
   while holding the mutex, blocking all concurrent relay threads.
7. Observable: unknown_tx_hashes mutex wait time spikes; pop_ask_for_txs latency
   increases proportionally; transaction relay throughput drops measurably.
```

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }
