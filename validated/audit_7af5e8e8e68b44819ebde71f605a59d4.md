### Title
Missing `txs_fees` Population in `BlockExt` Under `Switch::DISABLE_ALL` Causes Permanent Node Isolation After `ckb import --skip-all-verify` ‚Äî (File: `chain/src/verify.rs`)

---

### Summary

When blocks are processed with `Switch::DISABLE_ALL` (triggered by the supported `ckb import --skip-all-verify` CLI flag), the `txs_fees` field in `BlockExt` is never populated ‚Äî it is stored as an empty `vec![]`. Later, when the node processes new blocks that finalize those imported blocks, `RewardCalculator` reads the empty `txs_fees` and computes zero transaction fees for the target block. The `RewardVerifier` then expects a lower cellbase reward than what honest miners produce, causing the node to permanently reject all valid blocks that finalize any imported block containing fee-paying transactions.

---

### Finding Description

In `chain/src/verify.rs`, `reconcile_main_chain()` has two branches based on `switch.disable_all()`:

**Full-verification path** (lines 645‚Äì714): resolves transactions, runs `ContextualBlockVerifier`, collects `cache_entries` (which contain per-transaction fees), and calls `insert_ok_ext(..., Some(&cache_entries), ...)`.

**`DISABLE_ALL` path** (lines 718‚Äì724): skips all verification and calls `insert_ok_ext(..., None, None)`. [1](#0-0) 

Inside `insert_ok_ext`, when `cache_entries` is `None`, the `txs_fees` field of `BlockExt` is never written ‚Äî it stays as the empty `vec![]` set during initial `BlockExt` construction: [2](#0-1) 

The initial `BlockExt` is constructed with `txs_fees: vec![]`: [3](#0-2) 

This empty `txs_fees` is then persisted to the store via `txn.insert_block_ext(hash, &ext)`.

The `Switch::DISABLE_ALL` flag is a production-reachable path via `ckb import --skip-all-verify`: [4](#0-3) 

Later, when the node processes a new block B at height H, `RewardVerifier` calls `RewardCalculator::block_reward_to_finalize()` to compute the expected reward for the target block T at height `H - finalization_delay_length`. This reads `T`'s stored `txs_fees`: [5](#0-4) 

And also reads `txs_fees` from blocks in the proposal window for proposal rewards: [6](#0-5) 

If T was imported with `--skip-all-verify`, both return zero. The total expected reward is `primary + secondary + 0 + 0`. An honest miner's cellbase includes the real tx fees, so `RewardVerifier` rejects the block. [7](#0-6) 

Note: `Switch::DISABLE_SCRIPT` (used by `assume_valid_targets`) does **not** trigger this bug ‚Äî it still runs `ContextualBlockVerifier` and collects `cache_entries` with correct fees. Only `Switch::DISABLE_ALL` skips fee collection entirely. [8](#0-7) 

---

### Impact Explanation

A node that imports blocks using `ckb import --skip-all-verify` stores empty `txs_fees` for every imported block. Once the node attempts to sync new blocks from the network, it will reject every valid block whose finalization target was an imported block containing fee-paying transactions. The node becomes permanently isolated from the main chain for those block heights, unable to advance its tip. This is a **node isolation / consensus split** impact for any operator using the supported import workflow.

---

### Likelihood Explanation

`ckb import --skip-all-verify` is a documented, supported CLI subcommand intended for fast bootstrapping. Any operator who uses it to import a chain snapshot containing fee-paying transactions (which is the normal case on mainnet) and then attempts to continue syncing will trigger this issue deterministically. The finalization delay on CKB mainnet is 11 blocks, so the failure manifests within 11 blocks of resuming normal sync.

---

### Recommendation

In `insert_ok_ext` (or in the `DISABLE_ALL` branch of `reconcile_main_chain`), when `cache_entries` is `None`, compute `txs_fees` directly from the block's transactions: for each non-cellbase transaction, calculate `inputs_capacity - outputs_capacity` using the already-attached cell data. This is a pure accounting step, not a verification step, and must be performed regardless of the verification switch. The fix is analogous to the external report's patch: record the fee into the accumulator before the state transition is committed, even when verification is skipped.

---

### Proof of Concept

1. Build a chain with blocks containing fee-paying transactions (inputs capacity > outputs capacity).
2. Export those blocks to a file and import them on a fresh node using `ckb import --skip-all-verify`.
3. Inspect the stored `BlockExt` for any imported block with transactions: `txs_fees` will be `[]`.
4. Resume normal sync. When the node processes block at height `H = imported_block_height + finalization_delay_length`, `RewardVerifier` computes expected reward using `txs_fees = []` (zero fees), while the honest miner's cellbase includes the real fees.
5. The node emits a reward verification error and rejects the block, stalling permanently at that height. [1](#0-0) [9](#0-8) [10](#0-9)

    fn insert_ok_ext(
        &self,
        txn: &StoreTransaction,
        hash: &Byte32,
        mut ext: BlockExt,
        cache_entries: Option<&[Completed]>,
        txs_sizes: Option<Vec<u64>>,
    ) -> Result<(), Error> {
        ext.verified = Some(true);
        if let Some(entries) = cache_entries {
            let (txs_fees, cycles) = entries
                .iter()
                .map(|entry| (entry.fee, entry.cycles))
                .unzip();
            ext.txs_fees = txs_fees;
            ext.cycles = Some(cycles);
        }
        ext.txs_sizes = txs_sizes;
        txn.insert_block_ext(hash, &ext)
    }

    fn txs_fees(&self, target: &HeaderView) -> CapacityResult<Capacity> {
        let consensus = self.consensus;
        let target_ext = self
            .store
            .get_block_ext(&target.hash())
            .expect("block body stored");

        target_ext
            .txs_fees
            .iter()
            .try_fold(Capacity::zero(), |acc, tx_fee| {
                tx_fee
                    .safe_mul_ratio(consensus.proposer_reward_ratio())
                    .and_then(|proposer| {
                        tx_fee
                            .safe_sub(proposer)
                            .and_then(|miner| acc.safe_add(miner))
                    })
            })
    }
