### Title
Update Fees Are Permanently Locked in PriceInfoObjects With No Withdrawal Path — (`target_chains/sui/contracts/sources/price_info.move`)

---

### Summary

Every SUI fee paid via `update_single_price_feed` is deposited into the calling `PriceInfoObject`'s dynamic object field and can never be withdrawn. The `fee_recipient_address` stored in `State` is explicitly noted as unused, and no withdrawal function exists anywhere in the contract.

---

### Finding Description

`update_single_price_feed` enforces a fee check and then deposits the coin: [1](#0-0) 

`deposit_fee_coins` stores the coin as a dynamic object field under `FEE_STORAGE_KEY = b"fee_storage"` on the `PriceInfoObject`: [2](#0-1) 

The module exposes `get_balance` (read-only) but **no `withdraw_fee_coins` or equivalent function**: [3](#0-2) 

`State` holds a `fee_recipient_address` field: [4](#0-3) 

But this address is **never used** to transfer fees. The governance module `set_fee_recipient.move` opens with an explicit admission: [5](#0-4) 

`set_fee_recipient` only updates the stored address field; it does not trigger any coin transfer: [6](#0-5) 

A complete search of all `.move` files under `target_chains/sui/contracts/sources/` finds zero occurrences of `withdraw`, `collect_fee`, or any function that removes a `Coin<SUI>` from a `PriceInfoObject`. The only write path is `deposit_fee_coins`.

---

### Impact Explanation

Every SUI coin paid as an update fee since contract deployment is permanently locked across all `PriceInfoObject` shared objects. `PriceInfoObject` has the `key` ability and is shared — it cannot be destroyed or transferred. The `Coin<SUI>` stored as its dynamic object field is inaccessible without a module-level withdrawal function, which does not exist. Protocol revenue is 100% unrecoverable under the current code.

---

### Likelihood Explanation

This triggers on every single call to `update_single_price_feed` when `base_update_fee > 0`. It requires no special privileges, no key compromise, and no governance manipulation. Any user updating any price feed contributes to the locked balance. It is already occurring on every deployed instance where fees are non-zero.

---

### Recommendation

Add a privileged `withdraw_fee_coins` function to `price_info.move` gated behind `LatestOnly` (or a governance capability), and wire it to actually transfer the accumulated `Coin<SUI>` to `state::get_fee_recipient(pyth_state)`. The `set_fee_recipient` governance action already maintains the recipient address — it just needs to be connected to an actual transfer path.

---

### Proof of Concept

1. Deploy with `base_update_fee = 50`.
2. Call `update_single_price_feed` passing `Coin<SUI>` of value 50.
3. Assert `price_info::get_balance(&price_info_object) == 50`. ✓
4. Attempt any transaction sequence to reduce that balance — no public or friend function exists to do so.
5. Assert `price_info::get_balance(&price_info_object) == 50` still holds after all possible calls. ✓

The test in `pyth_tests` already inadvertently confirms step 3: [7](#0-6) 

No corresponding assertion or test for a withdrawal exists anywhere in the test suite, confirming the withdrawal path is absent by design omission.

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
