### Title
Unbounded `constructProviderCommitment` Loop Causes Permanent OOG Griefing of Requests When `maxNumHashes == 0` — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When a provider's `maxNumHashes` is `0` (the default, zero-initialized state), there is no cap on `req.numHashes` stored at request time. An attacker who makes N sequential requests against such a provider causes the Nth request to store `numHashes = N`. Because `req.numHashes` is immutable after request creation and `constructProviderCommitment` loops exactly `numHashes` times with no gas guard, any request whose stored `numHashes` exceeds the block gas budget is permanently unfulfillable. The Fortuna keeper explicitly skips proactive commitment advancement for providers with `maxNumHashes == 0`, compounding the exposure.

---

### Finding Description

**Root cause — unbounded loop:**

`constructProviderCommitment` performs one `keccak256` per iteration with no upper bound: [1](#0-0) 

**Root cause — `numHashes` is fixed at request time and uncapped when `maxNumHashes == 0`:**

In `requestHelper`, `req.numHashes` is set to the current gap and is only rejected when `maxNumHashes != 0`: [2](#0-1) 

`maxNumHashes` defaults to `0` for every newly registered provider (zero-initialized struct). Setting it to `0` explicitly via `setMaxNumHashes(0)` also disables the cap.

**Root cause — `req.numHashes` is immutable and used verbatim at reveal time:**

`revealHelper` passes the stored `req.numHashes` directly into the loop: [3](#0-2) 

No post-hoc mechanism exists to reduce the stored `numHashes` of an already-created request. Even if the provider later calls `advanceProviderCommitment`, the stored value in the request struct is unchanged.

**Root cause — Fortuna keeper skips commitment advancement for `maxNumHashes == 0` providers:** [4](#0-3) 

This means providers in the default configuration receive no automatic protection from gap growth.

**Attack sequence:**

1. Provider registers with default `maxNumHashes == 0`.
2. Attacker calls `requestV2` (or `requestWithCallback`) N times, paying the provider fee each time.
3. Each call increments `providerInfo.sequenceNumber` while `currentCommitmentSequenceNumber` stays fixed (provider never reveals, keeper skips advancement).
4. The Nth request stores `req.numHashes = N`.
5. Provider (or keeper) calls `revealWithCallback` for the Nth request → `constructProviderCommitment(N, ...)` loops N times → OOG revert.
6. The request is permanently stuck: `req.numHashes` cannot be reduced, and no alternative reveal path exists.

---

### Impact Explanation

Any request whose stored `numHashes` exceeds the gas threshold (~833 k iterations at 36 gas/keccak256 on a 30 M gas block) is permanently unfulfillable. The provider's keeper will retry indefinitely and always revert with OOG. Users whose requests fall in this range never receive their random number. The provider cannot recover those specific requests even by calling `advanceProviderCommitment` (which itself calls `constructProviderCommitment` and would also OOG for large gaps, though it can be called in smaller increments — this does not help because `req.numHashes` is already stored).

Scoped impact: **contract unable to fulfill randomness requests (permanent griefing of provider keeper)** — matching the stated scope.

---

### Likelihood Explanation

- `maxNumHashes == 0` is the **default** for every provider; no action is required to be in the vulnerable state.
- The Fortuna keeper explicitly opts out of commitment advancement for such providers.
- The attack cost scales with the provider's per-request fee and the target chain's gas price. On low-fee chains (Polygon, Arbitrum, BSC), reaching the OOG threshold (~833 k requests) is economically feasible for a motivated attacker.
- The attacker does not need any privileged role; `requestV2` is a public payable function.

---

### Recommendation

1. **Enforce a hard cap on `numHashes` unconditionally.** Remove the `maxNumHashes != 0` guard so that a default of `0` does not mean "uncapped":

   ```solidity
   // In requestHelper, replace:
   if (providerInfo.maxNumHashes != 0 && req.numHashes > providerInfo.maxNumHashes) {
       revert EntropyErrors.LastRevealedTooOld();
   }
   // With a protocol-level hard cap (e.g., 500_000) applied regardless of maxNumHashes.
   ```

2. **Add a gas guard inside `constructProviderCommitment`** or cap `numHashes` to a safe maximum before entering the loop.

3. **Update the Fortuna keeper** to proactively advance commitments for all providers, not only those with `maxNumHashes != 0`.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

// Precondition: provider registered with maxNumHashes == 0 (default).
// N = 900_000 (exceeds ~833k keccak256 iterations at 30M gas block limit)

contract AttackPoC {
    IEntropy entropy;
    address provider;

    constructor(address _entropy, address _provider) payable {
        entropy = IEntropy(_entropy);
        provider = _provider;
    }

    function attack(uint256 N) external payable {
        uint256 fee = entropy.getFee(provider);
        for (uint256 i = 0; i < N; i++) {
            // Each call increments provider.sequenceNumber by 1.
            // currentCommitmentSequenceNumber stays fixed (provider never reveals).
            // The Nth request stores numHashes = N.
            entropy.requestV2{value: fee}(provider, bytes32(i), 0);
        }
        // Provider's revealWithCallback for the Nth request now OOGs:
        // constructProviderCommitment(N, ...) loops N times → block gas exhausted.
    }
}
```

Gas measurement: `constructProviderCommitment(N, bytes32(0))` costs approximately `36 * N` gas. At N = 833,334 and a 30 M gas block, the call reverts with OOG. The stored `req.numHashes` for that request is permanently `N` and cannot be reduced.
