Audit Report

## Title
`DaoScriptSizeVerifier` Lock-Script-Size Check Bypassed via Index Misalignment — (`File: verification/src/transaction_verifier.rs`)

## Summary
`DaoScriptSizeVerifier::verify()` enforces RFC 0044 by zipping `resolved_inputs[i]` with `outputs[i]` positionally and only performing the lock-script-size comparison when both slots carry a DAO type script. Because CKB imposes no rule requiring the DAO deposit input and the DAO withdrawal output to share the same transaction index, any sender can place them at different indices, causing every pair to fail the "both are DAO" guard and the size check to be silently skipped. This allows lock-script substitution during DAO withdrawal in violation of a consensus rule enforced on both the block-verification and tx-pool admission paths.

## Finding Description
The root cause is in `verification/src/transaction_verifier.rs` at lines 847–858:

```rust
for (i, (input_meta, cell_output)) in self
    .resolved_transaction
    .resolved_inputs
    .iter()
    .zip(self.resolved_transaction.transaction.outputs())  // positional pairing
    .enumerate()
{
    if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
        && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
    {
        continue;   // silently skips misaligned pairs
    }
    ...
    if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
        return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
    }
}
``` [1](#0-0) 

The guard requires **both** the input and output at the same index to carry a DAO type script. If the DAO deposit input is at `inputs[1]` and the DAO withdrawal output is at `outputs[0]` (e.g., because a standard fee cell occupies `inputs[0]`), the zip produces only "mixed" pairs:

- `(inputs[0], outputs[0])`: `inputs[0]` is not DAO → `continue`
- `(inputs[1], outputs[1])`: `outputs[1]` is not DAO → `continue`

The loop exhausts without ever reaching the size comparison at line 885, and `verify()` returns `Ok(())`. [2](#0-1) 

The verifier is invoked unconditionally (when `rfc0044_active`) in both the block-level verifier and the tx-pool admission path: [3](#0-2) [4](#0-3) 

## Impact Explanation
RFC 0044 / `DaoScriptSizeVerifier` is an explicit consensus rule whose sole purpose is to prevent lock-script substitution during DAO withdrawal ("a temporary solution till Nervos DAO script can be properly upgraded"). Bypassing it allows a depositor to commit a cell with a large lock script (e.g., 200-byte multisig) and withdraw into a cell with a much smaller lock script (e.g., 20-byte secp256k1), violating the invariant the rule was designed to close. Because the check runs on both the block verifier and the tx-pool, a crafted transaction is accepted network-wide under the current code; once a corrected implementation is deployed to a subset of nodes, those nodes will reject transactions that unpatched nodes accept, producing a **consensus deviation** — a chain-split / finality risk. This maps to the Critical allowed impact: *Vulnerabilities which could easily cause consensus deviation*.

## Likelihood Explanation
No privilege is required. Any CKB transaction sender can trigger the bypass by including a single non-DAO input (e.g., a fee cell) before the DAO deposit input — a layout that is standard practice. The bypass is deterministic, repeatable, and requires no brute-force or timing. It affects all DAO deposits committed after `starting_block_limiting_dao_withdrawing_lock`; older deposits are unaffected. The DAO on-chain script does not independently enforce lock-script size equality, so there is no secondary defence. [5](#0-4) 

## Recommendation
Replace the index-coupled `.zip()` loop with a two-pass approach that independently collects all qualifying DAO deposit inputs and all DAO withdrawal outputs, then enforces size equality for each matched pair regardless of their transaction indices:

```rust
let dao_deposit_lock_sizes: Vec<usize> = self.resolved_transaction.resolved_inputs
    .iter()
    .filter(|meta| cell_uses_dao_type_script(&meta.cell_output, &dao_type_hash))
    .filter(|meta| /* data all zeros and block_number >= threshold */)
    .map(|meta| meta.cell_output.lock().total_size())
    .collect();

let dao_withdrawal_lock_sizes: Vec<usize> = self.resolved_transaction.transaction.outputs()
    .into_iter()
    .filter(|output| cell_uses_dao_type_script(output, &dao_type_hash))
    .map(|output| output.lock().total_size())
    .collect();

for (i, (deposit_size, withdrawal_size)) in
    dao_deposit_lock_sizes.iter().zip(dao_withdrawal_lock_sizes.iter()).enumerate()
{
    if deposit_size != withdrawal_size {
        return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
    }
}
```

The exact pairing strategy should follow the DAO protocol's deposit-to-withdrawal semantics, but any index-independent strategy is strictly correct where the current positional zip is not.

## Proof of Concept
Construct a transaction with the following layout:

```
inputs:
  [0]  fee_cell          lock=secp256k1(20B)   type=None   data=0x
  [1]  dao_deposit_cell  lock=multisig(200B)   type=DAO    data=0x0000000000000000

outputs:
  [0]  capacity=C+interest  lock=secp256k1(20B)  type=DAO   data=<deposit_block_number>
  [1]  change_cell          lock=secp256k1(20B)  type=None  data=0x
```

Verifier trace:
1. `.zip()` produces pairs `(inputs[0], outputs[0])` and `(inputs[1], outputs[1])`.
2. Pair 0: `inputs[0]` has no DAO type → `continue`.
3. Pair 1: `outputs[1]` has no DAO type → `continue`.
4. Loop ends; `verify()` returns `Ok(())`.
5. Lock script changed from 200 B to 20 B with no error raised.

Expected correct behaviour: `DaoLockSizeMismatch { index: 0 }` error blocking the transaction.

A unit test mirroring the existing `DaoScriptSizeVerifier` tests in `verification/src/tests/transaction_verifier.rs` with this index-misaligned layout will reproduce the silent bypass deterministically. [6](#0-5)
