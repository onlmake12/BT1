### Title
RBF-Replaced Proposed Transaction Remains in Stale Block Assembler Template and Can Be Committed - (File: tx-pool/src/block_assembler/mod.rs)

---

### Summary

When a proposed transaction (`tx1`) is replaced via RBF by a new transaction (`tx2`), the `BlockAssembler`'s cached `CurrentTemplate` is **not updated** to remove `tx1` from its `transactions` list. A miner calling `get_block_template` receives the stale template still containing `tx1`, and can commit `tx1` even though it has been marked `Rejected` in the tx-pool. The user's RBF replacement (`tx2`) is then rejected because its inputs are already spent. This is explicitly confirmed by a test comment in the codebase.

---

### Finding Description

**Root cause — block assembler template not refreshed on RBF eviction**

The `BlockAssembler` maintains a `CurrentTemplate` struct that caches proposed transactions: [1](#0-0) 

The template's `transactions` field is only refreshed when `BlockAssemblerMessage::Proposed` is received, which triggers `update_transactions`: [2](#0-1) 

`update_transactions` re-reads the tx-pool's current proposed set and rebuilds the transaction list: [3](#0-2) 

**The gap:** When `process_rbf` removes `tx1` from the pool and adds `tx2` as a fresh pending transaction, `notify_block_assembler` is called with `TxStatus::Fresh`, which maps only to `BlockAssemblerMessage::Pending`: [4](#0-3) 

`BlockAssemblerMessage::Pending` triggers only `update_proposals` — **not** `update_transactions`: [5](#0-4) 

So after RBF eviction of a proposed transaction, the `CurrentTemplate.transactions` list is **never refreshed** unless an unrelated proposed transaction happens to arrive later. The stale `tx1` entry persists in the template.

**Miner path:** `get_block_template` returns the cached template directly without triggering any refresh: [6](#0-5) 

**Confirmed by test:** The integration test `RbfReplaceProposedFailed` explicitly documents this behavior: [7](#0-6) 

The comment reads: *"since old tx is already in BlockAssembler, tx1 will be committed, even it is not in tx_pool and with `Rejected` status now"* — confirming the stale template is the root cause.

---

### Impact Explanation

A user who submits `tx2` as an RBF replacement for a proposed `tx1` (e.g., to cancel a payment or redirect funds) finds that `tx1` is committed anyway from the stale block assembler template. `tx2` is then rejected with `TransactionFailedToResolve: Resolve failed Unknown` because `tx1`'s inputs are already spent. The user's RBF fee premium is wasted and the intended replacement never takes effect. If `tx1` was a payment the user wanted to cancel, the funds go to the unintended recipient with no recourse.

**Impact: Medium**

---

### Likelihood Explanation

This requires `tx1` to be in the `Proposed` state (i.e., already packaged into the block assembler template). Transactions enter the proposed state within one proposal window (typically 2 blocks). The vulnerable window is between the moment `tx2` replaces `tx1` and the next time `update_transactions` is triggered by an unrelated proposed transaction. In a busy mempool this window is short; in a quiet mempool it can persist until the next block. The scenario is reachable by any RPC caller using `send_transaction` with RBF enabled.

**Likelihood: Medium**

---

### Recommendation

In `process_rbf`, after removing a proposed transaction from the pool, explicitly send `BlockAssemblerMessage::Proposed` (in addition to the existing `Pending` notification for `tx2`) so that `update_transactions` is triggered and the stale entry is evicted from the cached template. Alternatively, `get_block_template` should re-validate each transaction in the cached template against the current tx-pool state before returning it to the miner.

---

### Proof of Concept

1. Submit `tx1` (low fee) via `send_transaction` RPC.
2. Mine one proposal block so `tx1` enters `Proposed` state and is packaged into the `BlockAssembler` template.
3. Submit `tx2` via `send_transaction` RPC — same inputs as `tx1`, higher fee satisfying RBF rules. `tx1` is evicted from the tx-pool and marked `Rejected`.
4. Call `get_block_template` — the returned template still contains `tx1` in its `transactions` list.
5. Submit the block built from this template. `tx1` is committed on-chain.
6. Observe `tx2` status transitions to `Rejected` with reason `TransactionFailedToResolve`.

This exact sequence is exercised by the existing test `RbfReplaceProposedFailed`: [8](#0-7)

                TxStatus::Fresh => Some(BlockAssemblerMessage::Pending),
