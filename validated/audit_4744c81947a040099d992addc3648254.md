Audit Report

## Title
Unconditional `.expect()` on `txs_sizes` in `FeeRateCollector::statistics` Panics on Nodes with Pre-v0.106 `BlockExt` Records — (`rpc/src/util/fee_rate.rs`)

## Summary
`FeeRateCollector::statistics` calls `.expect()` unconditionally on `block_ext.txs_sizes` for every `BlockExt` yielded by the fee-rate window iterator. Nodes upgraded in-place from a version prior to v0.106 retain historical blocks stored in the old 5-field `BlockExt` schema, which deserialises with `txs_sizes: None`. Any caller invoking `get_fee_rate_statistics` with a window that overlaps those blocks triggers a guaranteed panic, crashing the RPC handler.

## Finding Description
In `rpc/src/util/fee_rate.rs:93`, the closure passed to `collect` destructures every `BlockExt` and immediately panics if `txs_sizes` is `None`:

```rust
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
```

There is no guard, no `if let Some(...)`, and no early-return before this line.

`get_block_ext` in `store/src/store.rs:247–263` reads raw bytes and branches on the field count. When `count_extra_fields() == 0` (the pre-v0.106 five-field `BlockExt` schema), it converts via `packed::BlockExtReader::into()`. That conversion path — implemented in both the `Unpack` trait (`util/types/src/conversion/storage.rs:139–151`) and the `From` impl (`storage.rs:154–166`) — explicitly sets `txs_sizes: None`.

The `filter_map` in `collect` (`fee_rate.rs:45–47`) only skips blocks whose hash or ext record is entirely absent from storage. A block stored in the old format is still returned as `Some(BlockExt { txs_sizes: None, … })` and is passed directly into the panicking closure.

A second production path exists: in `chain/src/verify.rs:718–723`, when `switch.disable_all()` is true, `insert_ok_ext` is called with `txs_sizes: None`, writing a `BlockExtV1` record whose `txs_sizes` option is empty. This record also triggers the panic.

All new writes use `BlockExtV1` (`store/src/transaction.rs:241–252`), so a fresh node syncing from genesis is unaffected. The vulnerability is specific to nodes carrying pre-v0.106 data or blocks committed under `DISABLE_ALL`.

## Impact Explanation
A panic in the RPC handler crashes the `get_fee_rate_statistics` request. This is a reliable, repeatable denial-of-service against that endpoint on any upgraded mainnet node retaining pre-v0.106 `BlockExt` records. Impact is scoped to **local RPC API crash (0–500 points)**.

## Likelihood Explanation
Any mainnet node that was running before v0.106 and upgraded in-place retains old-format `BlockExt` records for all blocks committed before the upgrade. The maximum window is 101 blocks (`MAX_TARGET`), so on a node with even a single pre-migration block in the last 101 blocks the panic is guaranteed. The RPC requires no authentication and is reachable by any local caller.

## Recommendation
Replace the unconditional `.expect()` with a graceful skip:

```rust
let Some(txs_sizes) = txs_sizes else {
    return fee_rates; // skip blocks without size data
};
```

This matches the existing guard `if txs_sizes.len() > 1 && !txs_fees.is_empty()` that already handles the case where there is nothing useful to compute.

## Proof of Concept
1. Start a node that was previously running pre-v0.106 (or manually insert a `BlockExt` record with the old 5-field schema for any block in the last 101 blocks).
2. Call `get_fee_rate_statistics` with `target = 101` (or any value whose window overlaps the old-format block).
3. The node panics with `"expect txs_size's length >= 1"` at `rpc/src/util/fee_rate.rs:93`.
