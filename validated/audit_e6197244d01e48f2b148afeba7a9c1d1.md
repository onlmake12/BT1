Looking at the exact code in `remove_unconfirmed_tx` and how it's called:

### Title
Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` via Long-Lived Pool Eviction — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

### Summary

A missing bounds check in `remove_unconfirmed_tx` allows `tx_age - 1` to exceed the length of `confirm_blocks_to_failed_txs` (1000), causing an index-out-of-bounds panic and crashing the node process. Any unprivileged user who submits a valid transaction that remains unconfirmed for more than 1000 blocks and is then evicted from the pool can trigger this.

---

### Finding Description

`confirm_blocks_to_failed_txs` is initialized with exactly `MAX_CONFIRM_BLOCKS = 1000` rows: [1](#0-0) 

In `remove_unconfirmed_tx`, the guard at line 208 only protects the `block_unconfirmed_txs` access. The `count_failure` write at line 215 is **outside** that if-else and uses `tx_age - 1` with no upper-bound check: [2](#0-1) 

When `tx_age >= 1001`, `tx_age - 1 >= 1000`, which is out of bounds for a Vec of length 1000. Rust panics with `index out of bounds: the len is 1000 but the index is 1000`.

---

### Impact Explanation

The panic propagates up through the async task that owns the fee estimator write lock, crashing the CKB node process. This is a **node crash** (DoS), matching the High scope (10001–15000).

---

### Likelihood Explanation

The production call chain is direct and unprivileged:

1. Attacker submits a valid tx via P2P or RPC with a fee rate inside the tracked bucket range but low enough to never be mined.
2. `accept_tx` → `track_tx` records `entry_height = best_height` at submission time. [3](#0-2) 

3. After 1001+ blocks, the pool evicts the tx (expiry or size-limit). `reject_tx` is called: [4](#0-3) 

4. `reject_tx` → `drop_tx` → `drop_tx_inner(tx_hash, true)` → `remove_unconfirmed_tx(..., count_failure=true)`: [5](#0-4) 

5. `tx_age = best_height - entry_height >= 1001` → panic at line 215.

CKB's default tx-pool expiry is 12 hours (~4320 blocks at 10 s/block), so a tx can trivially remain in the pool for 1001+ blocks. The `FeeEstimator::reject_tx` dispatch confirms only `ConfirmationFraction` is affected (not `Dummy` or `WeightUnitsFlow`): [6](#0-5) 

---

### Recommendation

Add a bounds check before the `count_failure` write:

```rust
if count_failure {
    if let Some(row) = self.confirm_blocks_to_failed_txs.get_mut(tx_age - 1) {
        row[bucket_index] += 1f64;
    }
    // silently drop samples older than MAX_CONFIRM_BLOCKS
}
```

#[test]
fn test_oob_panic_on_old_eviction() {
    let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
    let max_confirm_blocks = 1000;
    let decay = 0.993f64;
    let mut stat = TxConfirmStat::new(buckets, max_confirm_blocks, decay);

    // Track a tx at height 0
    let bucket_index = stat.add_unconfirmed_tx(0, FeeRate::from_u64(1500)).unwrap();

    // Advance the window past MAX_CONFIRM_BLOCKS
    for h in 1..=1001u64 {
        stat.move_track_window(h);
    }

    // Evict with count_failure=true; tx_age = 1001 - 0 = 1001
    // confirm_blocks_to_failed_txs[1000] → OOB panic (len is 1000)
    stat.remove_unconfirmed_tx(0, 1001, bucket_index, true);
}
