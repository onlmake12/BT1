### Title
Stepped (Non-Linear) Fee Curve in Entropy Provider Fee Calculation Due to `roundTo10kGas` Rounding â€” (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The `getProviderFee` function in `Entropy.sol` is documented to scale fees **proportionally** with gas limit above the provider's `defaultGasLimit`. However, because the gas limit is first rounded **up** to the nearest 10,000 via `roundTo10kGas`, the resulting fee curve is a **staircase** of discrete steps rather than a linear ramp. Any Entropy user requesting a `gasLimit` just one unit above a 10k boundary is charged as if they requested the full next 10k step, producing a fee jump of up to `(10000 Ã— feeInWei) / defaultGasLimit` for a single extra gas unit.

|---|---|---|
| â‰¤ `defaultGasLimit` | `defaultGasLimit` | `feeInWei` |

### Impact Explanation

For a provider with `defaultGasLimit = 100,000` and `feeInWei = 1e15` (0.001 ETH):

- A user requesting `gasLimit = 100,001` pays `1.1e15` wei instead of the proportionally correct `â‰ˆ1.00001e15` wei â€” an **overcharge of ~0.0001 ETH** for a single extra gas unit.
- The overcharge per step is `(10,000 Ã— feeInWei) / defaultGasLimit`. With a small `defaultGasLimit` (e.g., 10,000), the step equals 100% of `feeInWei`, doubling the fee for one extra gas unit.
- Excess fees accrue to the provider's `accruedFeesInWei`, not to the protocol, so the provider is the direct beneficiary of the overcharge. [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged Entropy user calling `requestV2(provider, gasLimit)` with a `gasLimit` that is not an exact multiple of 10,000 and exceeds `defaultGasLimit` will trigger the overcharge. This is a common scenario: developers choosing a custom gas limit (e.g., 150,000 for a moderately complex callback) will land in the middle of a step and be overcharged. The entry path requires no special privileges. [5](#0-4) 

---

### Recommendation

Replace the step-based rounding with a direct linear calculation on the raw `gasLimit`:

```solidity
if (provider.defaultGasLimit > 0 && gasLimit > provider.defaultGasLimit) {
    uint128 additionalFee = (uint128(gasLimit - provider.defaultGasLimit)
        * provider.feeInWei) / provider.defaultGasLimit;
    return provider.feeInWei + additionalFee;
}
```

If 10k-granularity is required for the stored `gasLimit10k` field (for callback execution), decouple the fee calculation from the rounding: compute the fee on the raw `gasLimit` but store the rounded value for execution.

### Proof of Concept

```solidity
// Provider: defaultGasLimit = 100_000, feeInWei = 1e15
// pythFeeInWei = 0 for clarity

// roundedGasLimit = 100_000 (== defaultGasLimit), fee = 1e15

uint128 fee2 = entropy.getFeeV2(provider, 100_001);

// additionalFee = (10_000 * 1e15) / 100_000 = 1e14
// fee = 1e15 + 1e14 = 1.1e15

// fee2 - fee1 = 1e14 wei = 0.0001 ETH overcharge for 1 extra gas unit
// Linear expectation: fee2 â‰ˆ 1e15 + (1 * 1e15 / 100_000) = 1.00001e15

uint128 fee3 = entropy.getFeeV2(provider, 110_000);
// roundTo10kGas(110_000) = 11, roundedGasLimit = 110_000
// fee = 1.1e15  (same as fee2 â€” user requesting 9999 more gas pays nothing extra)
```

Users at `gasLimit = 100_001` and `gasLimit = 110_000` pay identical fees, while a user at `gasLimit = 100_001` is overcharged by ~10% of the base fee relative to the linear expectation. The fee curve is a staircase, not a line. [6](#0-5)

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }
    }
