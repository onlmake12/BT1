### Title
Excess `msg.value` Silently Credited to Provider Instead of Refunded to User — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.requestPriceUpdatesWithCallback()`, the stored `req.fee` is computed as `msg.value - pythFeeInWei` rather than `requiredFee - pythFeeInWei`. Any ETH sent above the required fee is silently absorbed into the provider's accrued balance when `executeCallback()` is later called, instead of being refunded to the user.

---

### Finding Description

`requestPriceUpdatesWithCallback()` correctly computes the required fee:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
// getFee = pythFeeInWei + providerBaseFee + providerFeedFee + gasFee
if (msg.value < requiredFee) revert InsufficientFee();
``` [1](#0-0) 

However, the fee stored in the request struct uses the raw `msg.value` rather than the required amount:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

If a user sends `msg.value = requiredFee + X` (overpays by `X`), then:
- `req.fee` is set to `providerFee + X` instead of just `providerFee`
- `_state.accruedFeesInWei += _state.pythFeeInWei` (correct, only Pyth's portion)
- No refund of `X` to the user [3](#0-2) 

Later, in `executeCallback()`, the provider is credited with the full stored `req.fee`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [4](#0-3) 

The provider receives `providerFee + X + msg.value_callback - pythFee_callback`, where `X` is the user's overpayment. The user's excess ETH is permanently transferred to the provider's `accruedFeesInWei` with no mechanism for recovery.

---

### Impact Explanation

Any user who sends `msg.value > getFee(provider, callbackGasLimit, priceIds)` permanently loses the excess ETH to the provider. This is incorrect accounting: the provider is credited more than their configured fee, and the user receives no refund. Over time, this silently drains user funds into provider balances. The `ProviderInfo.accruedFeesInWei` balance is inflated beyond what the provider legitimately earned. [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged user calling `requestPriceUpdatesWithCallback()` is affected if they send more than the exact required fee. This is a realistic scenario: users commonly add a small buffer to avoid `InsufficientFee` reverts due to fee changes between estimation and submission. The entry path requires no special privileges — it is a standard payable external function callable by any EOA or contract. [6](#0-5) 

---

### Recommendation

Replace the raw `msg.value` in the `req.fee` assignment with the computed `requiredFee`, and refund any excess to the caller:

```solidity
req.fee = SafeCast.toUint96(requiredFee - _state.pythFeeInWei);

uint256 excess = msg.value - requiredFee;
if (excess > 0) {
    (bool refunded, ) = msg.sender.call{value: excess}("");
    require(refunded, "Refund failed");
}
```

This ensures the provider is credited only their configured fee, and the user recovers any overpayment.

---

### Proof of Concept

1. Provider registers with `baseFeeInWei = 0.001 ether`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`.
2. `pythFeeInWei = 0.0001 ether`. So `getFee(provider, 0, [priceId]) = 0.0011 ether`.
3. User calls `requestPriceUpdatesWithCallback{value: 0.002 ether}(provider, ...)` — overpays by `0.0009 ether`.
4. `req.fee = 0.002 ether - 0.0001 ether = 0.0019 ether` (should be `0.001 ether`).
5. Provider calls `executeCallback(...)`. Provider's `accruedFeesInWei` increases by `0.0019 ether - pythFee_callback`, absorbing the user's `0.0009 ether` overpayment.
6. User has no way to recover the `0.0009 ether`. [7](#0-6) [4](#0-3)
