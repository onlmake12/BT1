### Title
Missing `Flags::DISCOVERY` Removal When Discovery Protocol Is Disabled Causes False Capability Advertisement - (File: `util/launcher/src/lib.rs`)

### Summary

CKB's Identify protocol broadcasts a `Flags` bitmask to all connecting peers to declare which protocols the local node supports. In `start_network_and_rpc()`, the launcher starts with `Flags::all()` and removes individual flags when their corresponding protocols are disabled. However, when `SupportProtocol::Discovery` is absent from `support_protocols`, the `Flags::DISCOVERY` bit is **never removed** from the advertised flags, even though the Discovery protocol handler is not registered. This is the direct analog of the ERC165 omission: a component implements (or in this case, omits) a capability but the capability advertisement does not match the actual runtime state.

### Finding Description

In `util/launcher/src/lib.rs`, `start_network_and_rpc()` constructs the `flags` bitmask passed to `NetworkService::new()` as the `identify_announce` parameter:

```rust
let mut flags = Flags::all();

if support_protocols.contains(&SupportProtocol::Relay) { ... }
else { flags.remove(Flags::RELAY); }

if support_protocols.contains(&SupportProtocol::Filter) { ... }
else { flags.remove(Flags::BLOCK_FILTER); }

if support_protocols.contains(&SupportProtocol::LightClient) { ... }
else { flags.remove(Flags::LIGHT_CLIENT); }

// SupportProtocol::Discovery is NEVER checked here — no flags.remove(Flags::DISCOVERY)
``` [1](#0-0) 

Meanwhile, the Discovery protocol handler is conditionally registered inside `NetworkService::new()` based on the same config:

```rust
if config.support_protocols.contains(&SupportProtocol::Discovery) {
    // register DiscoveryProtocol handler
}
``` [2](#0-1) 

The `flags` bitmask is encoded into the Identify message and broadcast to every connecting peer:

```rust
Identify::new(name, flags, client_version)
// → packed::Identify::new_builder().flag(flags.bits())...
``` [3](#0-2) 

The `Flags` enum defines `DISCOVERY = 0b10`, and `Flags::all()` includes it: [4](#0-3) 

When a peer receives the Identify message and the `required_flags_filter` passes, it opens **all** protocols (except Feeler) with the remote node, including Discovery:

```rust
TargetProtocol::Filter(Box::new(move |id| {
    id != &SupportProtocols::Feeler.protocol_id()
}))
``` [5](#0-4) 

If the remote node has Discovery disabled, it has no handler registered for protocol ID `1` (`/ckb/discovery`), so the protocol open fails silently. The peer's peer store has already recorded this node's address with `Flags::DISCOVERY` set, and that address is propagated to other nodes via the Discovery `Nodes` messages. [6](#0-5) 

### Impact Explanation

1. **False peer store entries propagate network-wide.** The connecting peer stores the node's address with `Flags::DISCOVERY` set via `peer_store.add_outbound_addr(...)`. These addresses are then broadcast to other nodes in `DiscoveryMessage::Nodes` responses, causing the entire reachable network to believe this node provides discovery services.

2. **Wasted connection slots.** Nodes that specifically need discovery peers (e.g., nodes bootstrapping or with few peers) will preferentially dial this node, open a session, fail to open the Discovery sub-protocol, and either disconnect or hold a useless connection slot.

3. **`required_flags_filter` bypass of intent.** The filter at `peer_store_impl.rs` uses the advertised flags to decide whether to connect to a peer. A node with Discovery disabled but `DISCOVERY` flag set will pass filters it should not pass, distorting peer selection across the network. [7](#0-6) 

### Likelihood Explanation

A node operator must explicitly set `support_protocols` in the network config to exclude `Discovery` (the default `default_support_all_protocols()` includes it). This is a non-default but documented configuration path. Any peer that connects to such a misconfigured node is affected without any action on the peer's part — the false advertisement is sent automatically during the Identify handshake. [8](#0-7) 

### Recommendation

Add the missing flag removal in `start_network_and_rpc()` in `util/launcher/src/lib.rs`, mirroring the pattern used for `Relay`, `Filter`, and `LightClient`:

```rust
if support_protocols.contains(&SupportProtocol::Discovery) {
    // Discovery handler is registered inside NetworkService::new()
} else {
    flags.remove(Flags::DISCOVERY);
}
```

This ensures the advertised `Flags` bitmask always accurately reflects the protocols the node actually runs, consistent with how `Flags::RELAY`, `Flags::BLOCK_FILTER`, and `Flags::LIGHT_CLIENT` are already handled. [9](#0-8) 

### Proof of Concept

1. Configure a CKB node with `support_protocols` that excludes `Discovery` (remove it from the list in `ckb.toml`).
2. Start the node. `NetworkService::new()` will skip registering the Discovery handler (protocol ID `1`).
3. Connect a standard CKB peer to this node. The peer receives an Identify message with `Flags::DISCOVERY` bit set (because `flags = Flags::all()` and `flags.remove(Flags::DISCOVERY)` is never called).
4. The peer calls `peer_store.add_outbound_addr(addr, flags)` with `flags` containing `DISCOVERY`, and then attempts `open_protocols(..., TargetProtocol::Filter(...))` which includes protocol ID `1`.
5. The misconfigured node has no handler for protocol ID `1`; the open fails. The peer's peer store now permanently records this node as a discovery-capable peer and propagates the address to other nodes.

pub(crate) fn required_flags_filter(required: Flags, t: Flags) -> bool {
    if required == Flags::RELAY | Flags::DISCOVERY | Flags::SYNC {
        t.contains(required) || t.contains(Flags::COMPATIBILITY)
    } else {
        t.contains(required)
    }
}
