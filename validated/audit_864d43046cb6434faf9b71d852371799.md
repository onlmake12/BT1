Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Stale-Snapshot Overwrite in `add_entry` Enables Phantom Pool Inflation — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::add_entry` snapshots prospective new totals via `updated_stat_for_add_tx` (a pure read) before calling `check_and_record_ancestors`, which may evict entries and correctly decrement `self.total_tx_size` via `update_stat_for_remove_tx`. The stale pre-eviction snapshot is then unconditionally written back at lines 218–219, permanently inflating `total_tx_size` by the size of every evicted entry. `limit_size` consumes the inflated total to drive eviction of legitimate pool entries, enabling a repeatable local tx-pool denial-of-service.

## Finding Description

In `add_entry` (`pool_map.rs` lines 210–219):

```rust
// Lines 210–211: snapshot taken BEFORE evictions (pure read, &self)
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Line 213: may call remove_entry_and_descendants → remove_entry
//           → update_stat_for_remove_tx → self.total_tx_size -= evicted.size
evicts = self.check_and_record_ancestors(&mut entry)?;

// Lines 218–219: stale snapshot OVERWRITES the post-eviction value
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) takes `&self` and computes `self.total_tx_size + entry.size` without mutating state. When `check_and_record_ancestors` (lines 588–640) finds `ancestors_count > max_ancestors_count` but the excess is entirely from `cell_ref_parents`, it evicts those entries via `remove_entry_and_descendants` (line 618), which calls `remove_entry` (line 247) → `update_stat_for_remove_tx` (lines 733–741), correctly decrementing `self.total_tx_size`. The write-back at lines 218–219 then replaces the decremented value with the pre-eviction snapshot.

The `recompute_total_stat` fallback inside `update_stat_for_remove_tx` (lines 742–755) is only triggered on `checked_sub` underflow and provides no mitigation here, since the subtraction succeeds before being overwritten.

Concrete arithmetic for one trigger cycle:
- Before: `self.total_tx_size = T`
- Snapshot: `total_tx_size = T + size(new_tx)`
- Eviction inside `check_and_record_ancestors`: `self.total_tx_size = T − size(evicted)`
- Write-back: `self.total_tx_size = T + size(new_tx)` ← ignores the subtraction
- Actual pool size: `T − size(evicted) + size(new_tx)`
- **Inflation per cycle: `+size(evicted)`**

`limit_size` (`pool.rs` lines 292–329) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting real entries until the phantom total drops below the limit.

## Impact Explanation

An attacker who repeatedly triggers the ancestor-eviction path accumulates unbounded phantom inflation in `total_tx_size`. Once the phantom total exceeds `max_tx_pool_size`, `limit_size` expels all pending/gap/proposed transactions from the node's pool. Honest users' transactions are rejected with `Reject::Full` and cannot be re-admitted while the attacker continues re-triggering. This constitutes a persistent local tx-pool denial-of-service: the affected node stops propagating or mining transactions.

**Impact: High** — matches "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation

The eviction path in `check_and_record_ancestors` requires `ancestors_count > max_ancestors_count` (default 25) where the excess is entirely from `cell_ref_parents`. The attacker needs:
1. A chain of 24 in-pool transactions (`tx_0`…`tx_23`).
2. One `tx_dep` using `tx_23`'s output as a cell dep (making it a `cell_ref_parent`).
3. A new transaction spending `tx_23`'s output as an input, pushing `ancestors_count` to 26.

All steps use standard `send_transaction` RPC calls with no privileged access. Each cycle inflates `total_tx_size` by `size(tx_dep)`. With `max_tx_pool_size` defaulting to 20 MB, repeated cycles eventually cause `limit_size` to evict all legitimate transactions while the real pool is nearly empty.

**Likelihood: Medium** — deliberate construction required, but no privilege and low cost per cycle.

## Recommendation

Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` so it reads the post-eviction `self.total_tx_size`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// self.total_tx_size now reflects any evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

As a defense-in-depth measure, add a debug-build invariant assertion after `add_entry` that `total_tx_size == entries.iter().map(|e| e.size).sum()`.

## Proof of Concept

1. Configure a node with `max_ancestors_count = 25`, `max_tx_pool_size = 20 MB`.
2. Submit `tx_0` creating output `cell_0`.
3. Submit `tx_1`…`tx_23`, each spending the previous output (chain of 24).
4. Submit `tx_dep` using `tx_23`'s output as a **cell dep** (not consuming it). `tx_dep` becomes a `cell_ref_parent` for any future tx spending `tx_23`'s output.
5. Submit `tx_25` spending `tx_23`'s output as an **input**.
   - `ancestors_count = 25 (chain) + 1 (tx_dep as cell_ref_parent) = 26 > 25`
   - `check_and_record_ancestors` evicts `tx_dep` → `self.total_tx_size -= size(tx_dep)`
   - `add_entry` writes back stale snapshot → `self.total_tx_size += size(tx_dep)` (net inflation)
6. Assert `pool_map.total_tx_size > actual sum of entry sizes` — invariant is broken after step 5.
7. Repeat steps 2–5 with fresh outputs. After N cycles, `total_tx_size` exceeds `max_tx_pool_size` while the real pool is nearly empty; `limit_size` then evicts all remaining legitimate transactions.
