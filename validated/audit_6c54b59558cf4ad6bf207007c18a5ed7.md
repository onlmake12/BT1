### Title
Stale Tx-Pool Snapshot Allows Double-Spend Transactions to Bypass Cell-Liveness Checks During Post-Block Window — (`chain/src/verify.rs`, `tx-pool/src/process.rs`)

---

### Summary

After a new best block is committed, the shared `Snapshot` is atomically swapped to the new chain tip, but the tx-pool's own internal snapshot is updated asynchronously via a separate channel. During the resulting window, any unprivileged RPC caller or relayer can submit a transaction that spends a cell already consumed by the newly committed block. Because the tx-pool resolves cells against its own stale snapshot, the dead cell appears live, all checks pass, the transaction is accepted and relayed to peers, and the `submit_entry` re-verification guard silently no-ops — because both the pre-resolve tip hash and the tx-pool's current tip hash are still the same old value.

---

### Finding Description

In `chain/src/verify.rs`, after a new best block is committed and written to the database, the following two-phase sequence occurs:

**Phase 1 — shared snapshot swapped (atomic):**
```rust
self.shared.store_snapshot(Arc::clone(&new_snapshot));   // line 383
```

**Phase 2 — tx-pool notified (fire-and-forget, async channel):**
```rust
tx_pool_controller.update_tx_pool_for_reorg(             // lines 387-394
    fork.detached_blocks().clone(),
    fork.attached_blocks().clone(),
    fork.detached_proposal_id().clone(),
    new_snapshot,
)
```

`update_tx_pool_for_reorg` sends a `Notify` message via `reorg_sender.try_send(notify)` to a dedicated async task that runs on a separate Tokio loop. The tx-pool's internal snapshot field (`TxPool::snapshot`) is only updated inside `_update_tx_pool_for_reorg` when that task eventually acquires the write lock:

```rust
tx_pool.snapshot = Arc::clone(&snapshot);   // process.rs line 1048
```

The tx-pool itself documents this gap:

```rust
/// Tx-pool owned snapshot, it may not consistent with chain cause tx-pool update snapshot asynchronously
pub(crate) fn snapshot(&self) -> &Snapshot { &self.snapshot }
```

During the window between Phase 1 and Phase 2, every tx-pool operation that reads cell liveness uses the stale snapshot. The full transaction admission path is:

1. **`pre_check`** acquires a read lock and calls `resolve_tx(tx_pool, &snapshot, ...)` where `snapshot = tx_pool.cloned_snapshot()` — the stale one. A cell spent in the new block appears live.

2. **`verify_rtx`** is called with the same stale snapshot. Script execution sees the cell as live and passes.

3. **`submit_entry`** acquires a write lock and runs the re-verification guard:

```rust
let tip_hash = snapshot.tip_hash();          // tx_pool's snapshot — still old
if pre_resolve_tip != tip_hash {             // pre_resolve_tip is also old → NEVER fires
    status = check_rtx(tx_pool, &snapshot, &entry.rtx)?;
    ...
    time_relative_verify(...)?;
}
```

Because `with_tx_pool_write_lock` always clones the snapshot from the tx-pool itself (`tx_pool.cloned_snapshot()`), both `pre_resolve_tip` and `tip_hash` are the same stale value. The guard is designed to catch a snapshot change that happens *between* `pre_check` and `submit_entry`, but it cannot catch the case where the tx-pool snapshot has not yet been updated at all. The transaction is inserted into the pool and the `call_pending` callback fires, relaying it to peers.

---

### Impact Explanation

- **Double-spend transactions are temporarily accepted** into the tx-pool. A cell already consumed by the committed block appears live, so the transaction passes cell-resolution, script verification, fee checks, and the submit guard.
- **The accepted transaction is relayed to peers** via the `call_pending` callback before the reorg update is processed.
- **Valid transactions can be evicted.** When the tx-pool is at capacity, `limit_size` is called inside `submit_entry`. An attacker who floods the pool with stale-snapshot-accepted double-spends during the window can evict legitimate pending transactions. Those transactions are not automatically re-added; they must be resubmitted by their originators.
- The double-spend transaction itself is eventually removed by `remove_committed_txs` when the reorg task runs, so no on-chain double-spend occurs. The impact is tx-pool pollution, relay bandwidth waste, and targeted eviction of valid transactions (transaction censorship).

---

### Likelihood Explanation

The window is short (sub-second, bounded by Tokio task scheduling latency), but it is deterministic and observable: any node watching the chain via `get_tip_block_number` or the subscription RPC will see the new tip immediately after `store_snapshot` completes, before the tx-pool has processed the reorg. An automated bot can:

1. Subscribe to new-block notifications.
2. On each new block, immediately submit transactions spending cells consumed by that block.
3. Repeat at high frequency to maximize eviction of legitimate transactions.

No privileged access, no key material, and no majority hashpower is required. The entry point is the standard `send_transaction` RPC, available to any unprivileged caller.

---

### Recommendation

The re-verification guard in `submit_entry` should compare `pre_resolve_tip` against the **shared** snapshot's tip hash (obtained from `self.shared.snapshot()` or equivalent), not the tx-pool's own stale snapshot. Alternatively, `submit_entry` should re-resolve and re-verify the full transaction whenever `pre_resolve_tip` differs from the shared tip, regardless of whether the tx-pool's internal snapshot has caught up. A simpler mitigation is to hold the reorg channel as a blocking call (synchronous send) so the tx-pool snapshot is updated before `store_snapshot` returns, eliminating the window entirely — consistent with the earlier changelog entry "Update tx-pool for reorg synchronously" (#2746).

---

### Proof of Concept

**Step 1.** Alice submits transaction `T_alice` spending live cell `C` to the tx-pool. `T_alice` is pending.

**Step 2.** Block N is mined and committed. Block N contains transaction `T0` that spends cell `C`. `store_snapshot` atomically advances the shared snapshot to block N. The reorg message is queued but not yet processed.

**Step 3.** Attacker immediately calls `send_transaction` with transaction `T_attacker` that also spends cell `C` (double-spend), with a fee high enough to trigger `limit_size` eviction.

**Step 4.** Inside `pre_check`, `resolve_tx` is called with the tx-pool's stale snapshot (still at block N-1). Cell `C` appears live. Resolution succeeds.

**Step 5.** `verify_rtx` runs against the stale snapshot. Scripts pass.

**Step 6.** Inside `submit_entry`, `snapshot.tip_hash()` returns the block N-1 hash (tx-pool's own snapshot). `pre_resolve_tip` is also block N-1. The guard `if pre_resolve_tip != tip_hash` evaluates to `false`. No re-verification. `T_attacker` is inserted.

**Step 7.** `limit_size` evicts `T_alice` (or another valid transaction) to make room.

**Step 8.** The reorg task eventually runs, calls `remove_committed_txs`, and removes `T_attacker`. But `T_alice` is already gone and must be resubmitted.

    ) -> (U, Arc<Snapshot>) {

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
        (ret, snapshot)
