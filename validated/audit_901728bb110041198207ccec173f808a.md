### Title
Silent u128→u64 Truncation in `DaoCalculator::calculate_maximum_withdraw` Bypasses Overflow Guard - (File: `util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` uses a bare `as u64` truncating cast to narrow a `u128` intermediate result, while every other analogous u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When the intermediate value exceeds `u64::MAX`, the truncation silently produces a drastically wrong (wrapped) capacity instead of returning `DaoError::Overflow`, causing the function to return `Ok(wrong_capacity)`.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the DAO withdrawal amount as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

Every other u128→u64 narrowing in the same file uses a checked conversion:

- `secondary_block_reward`: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- `dao_field_with_current_epoch` (miner issuance): `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?`
- `dao_field_with_current_epoch` (ar increase): `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?` [2](#0-1) [3](#0-2) [4](#0-3) 

When `withdraw_counted_capacity > u64::MAX`, `as u64` wraps the value modulo 2^64, producing a tiny positive number. If that wrapped value plus `occupied_capacity` does not itself overflow u64, `safe_add` succeeds and the function returns `Ok(drastically_underestimated_capacity)` — no error is raised.

The existing test `check_withdraw_calculation_overflows` does not catch this: it passes only because the wrapped value plus `occupied_capacity` happens to overflow u64 in the chosen test parameters, causing `safe_add` to return `Err`. A different set of parameters where the wrapped value is small enough avoids the `safe_add` overflow, and the bug goes undetected. [5](#0-4) 

### Impact Explanation

`calculate_maximum_withdraw` is called from two paths:

1. **RPC path** — `ExperimentRpcImpl::calculate_dao_maximum_withdraw` calls it directly and returns the result to any RPC caller. A wrong return value causes the caller to be told an incorrect (far too small) maximum withdrawal amount. [6](#0-5) 

2. **Block validation path** — `transaction_maximum_withdraw` → `withdrawed_interests` → `dao_field_with_current_epoch`. If the computed maximum withdraw is silently wrong, `withdrawed_interests` is underestimated, `current_s` (the DAO secondary issuance accumulator) is inflated, and the DAO field written into the block header is incorrect. This corrupts the on-chain DAO accounting state. [7](#0-6) [8](#0-7) 

### Likelihood Explanation

The overflow requires `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`. Since `counted_capacity ≤ u64::MAX` and `ar` grows slowly from its genesis value of `10^10`, this requires the `ar` ratio to have grown by more than ~0.000000022% relative to the deposit block. On mainnet with its fixed issuance schedule this is extremely unlikely within the foreseeable lifetime of the chain. However, on testnets or chains with custom consensus parameters (higher secondary issuance, lower total capacity), the threshold is reachable. The RPC path is reachable by any unprivileged caller with a valid DAO cell out-point.

### Recommendation

Replace the truncating cast with a checked conversion, consistent with every other u128→u64 narrowing in the file:

```rust
// Before (silent truncation):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (checked, returns DaoError::Overflow on overflow):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [9](#0-8) 

### Proof of Concept

The following values demonstrate the silent-truncation path (no error returned):

- `output.capacity` = `u64::MAX` shannons (maximum representable)
- `occupied_capacity` = 4,100,000,000 shannons (41-byte minimal cell)
- `counted_capacity` = `u64::MAX - 4_100_000_000` = `18_446_744_069_609_551_615`
- `deposit_ar` = `10_000_000_000_000_000`
- `withdrawing_ar` = `10_000_000_000_000_000 + 223` (ar has grown by 223 units)

Then:
```
withdraw_counted_capacity
  = 18_446_744_069_609_551_615 * 10_000_000_000_000_223 / 10_000_000_000_000_000
  = 18_446_744_069_609_551_615 + floor(18_446_744_069_609_551_615 * 223 / 10^16)
  = 18_446_744_069_609_551_615 + 411_562_192_942
  = 18_446_744_070_021_113_807   -- still < u64::MAX, no overflow here
```

Increase `withdrawing_ar` further so the product exceeds `u64::MAX`:
- `withdrawing_ar` = `deposit_ar + 2_300` (ar grown by 2300 units)

```
withdraw_counted_capacity
  ≈ 18_446_744_069_609_551_615 + 4_244_750_000_000
  = 18_446_744_073_854_301_615   -- exceeds u64::MAX (18_446_744_073_709_551_615) by 144_750_000

as u64 → 144_749_999  (wrapped)

Capacity::shannons(144_749_999).safe_add(4_100_000_000)
  = Ok(Capacity::shannons(4_244_749_999))   -- no error, wrong result returned
```

The function returns `Ok(~42 CKB)` for a cell that should yield an overflow error, silently underreporting the withdrawal amount by ~184 billion CKB. [10](#0-9)

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
            DaoWithdrawingCalculationKind::WithdrawingOutPoint(withdrawing_out_point) => {
                let (_tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                let withdrawing_out_point: packed::OutPoint = withdrawing_out_point.into();
                let (withdrawing_tx, withdrawing_header_hash) = snapshot
                    .get_transaction(&withdrawing_out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid withdrawing_out_point"))?;

                    .outputs()

                    .ok_or_else(|| RPCError::invalid_params("invalid withdrawing_out_point"))?;

                ) {

                }
