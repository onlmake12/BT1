Audit Report

## Title
`DaoScriptSizeVerifier::verify()` Lock-Script-Size Check Bypassed via Positional Index Mismatch — (File: verification/src/transaction_verifier.rs)

## Summary
`DaoScriptSizeVerifier::verify()` pairs resolved inputs with transaction outputs strictly by positional index using `.zip()`. Because CKB imposes no requirement that `inputs[i]` corresponds to `outputs[i]`, an attacker can place a DAO deposit input and its corresponding DAO withdrawal output at non-matching indices, causing every pair examined by the verifier to fail the "both cells use DAO type script" guard and be silently skipped via `continue`. The verifier returns `Ok(())` unconditionally, allowing a Phase-1 withdrawal transaction with a prepare cell whose lock script is arbitrarily larger than the deposit cell's lock script to land on-chain.

## Finding Description
The root cause is at `verification/src/transaction_verifier.rs` lines 847–858:

```rust
for (i, (input_meta, cell_output)) in self
    .resolved_transaction
    .resolved_inputs
    .iter()
    .zip(self.resolved_transaction.transaction.outputs())
    .enumerate()
{
    if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
        && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
    {
        continue;
    }
```

The guard at lines 855–858 requires **both** `inputs[i]` and `outputs[i]` to carry the DAO type script. If either does not, the pair is skipped. An attacker constructs a Phase-1 DAO withdrawal transaction with deliberately misaligned indices:

| Slot | Content |
|------|---------|
| `inputs[0]` | regular fee cell — no DAO type script |
| `inputs[1]` | DAO deposit cell — DAO type script, data = `[0u8; 8]` |
| `outputs[0]` | DAO prepare cell — DAO type script, **enlarged lock script** |
| `outputs[1]` | change cell — no DAO type script |

The verifier examines:
- Pair `(inputs[0], outputs[0])`: `inputs[0]` has no DAO type → `continue`
- Pair `(inputs[1], outputs[1])`: `outputs[1]` has no DAO type → `continue`

The actual DAO deposit/withdrawal pair `(inputs[1], outputs[0])` is never examined. The size-mismatch check at line 885 is never reached, and the verifier returns `Ok(())`.

The struct's doc comment at lines 817–818 explicitly states this verifier is a temporary sole enforcement layer until the DAO script can be upgraded, confirming the on-chain DAO type script does not independently enforce lock-script-size equality.

Both enforcement call sites are reachable by any unprivileged user:
- tx-pool admission path: `tx-pool/src/util.rs` lines 111–113
- block-commit path: `verification/contextual/src/contextual_block_verifier.rs` lines 444–453

## Impact Explanation
The verifier exists because the on-chain DAO type script does not handle the case where the prepare cell's lock script is larger than the deposit cell's lock script. Bypassing the verifier allows a Phase-1 transaction to land on-chain with a prepare cell whose lock script is arbitrarily larger than the deposit cell's lock script. In Phase 2, the DAO type script computes claimable interest against the deposit cell's smaller occupied capacity while the prepare cell's larger occupied capacity is not deducted, allowing the attacker to claim more CKB than entitled — extracting value from the DAO reserve at the expense of other depositors. This constitutes a concrete financial impact matching **Incorrect implementation or behavior of system scripts (High, 10001–15000 points)** and potentially **Vulnerabilities which could easily damage CKB economy (Critical, 15001–25000 points)** depending on scale.

## Likelihood Explanation
Any user who can submit a transaction via RPC `send_transaction` or P2P relay can construct the misaligned transaction. The construction requires only knowledge of the CKB transaction format and the DAO protocol, both of which are fully public. No key material, operator access, or majority hashpower is required. The bypass is deterministic and reproducible by any developer familiar with the CKB SDK.

## Recommendation
Replace the positional `.zip()` pairing with logic that explicitly matches each DAO deposit input against every DAO output in the transaction, regardless of index:

```rust
for input_meta in &self.resolved_transaction.resolved_inputs {
    if !cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash) { continue; }
    for (j, cell_output) in self.resolved_transaction.transaction.outputs().enumerate() {
        if !cell_uses_dao_type_script(&cell_output, &dao_type_hash) { continue; }
        if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
            return Err((TransactionError::DaoLockSizeMismatch { index: j }).into());
        }
    }
}
```

Alternatively, enforce that for every DAO deposit input there exists at most one DAO output and that its lock script size matches, using the witness field to establish the canonical deposit-to-withdrawal correspondence.

## Proof of Concept
Construct a transaction with the following layout using the CKB Rust SDK:

```rust
let tx = TransactionBuilder::default()
    // inputs[0]: regular cell (fee source, no DAO type)
    .input(CellInput::new(regular_out_point, 0))
    // inputs[1]: DAO deposit cell (data = [0u8;8], small lock script)
    .input(CellInput::new(dao_deposit_out_point, since_value))
    // outputs[0]: DAO prepare cell — ENLARGED lock script (e.g. 40-byte args)
    .output(
        CellOutputBuilder::default()
            .capacity(deposit_capacity)
            .lock(Script::new_builder().args(Bytes::from(vec![0u8; 40])).build())
            .type_(Some(dao_type_script.clone()))
            .build()
    )
    .output_data(Bytes::from(deposit_block_number_le_bytes))
    // outputs[1]: change cell (no DAO type)
    .output(change_cell)
    .output_data(Bytes::new())
    .build();
```

Submit via `send_transaction`. `DaoScriptSizeVerifier::verify()` examines pairs `(inputs[0], outputs[0])` and `(inputs[1], outputs[1])`, skips both via `continue`, and returns `Ok(())`. The transaction is accepted by both the tx-pool and block verifier. The prepare cell at `outputs[0]` carries a 40-byte-args lock script while the deposit cell had a 0-byte-args lock script — a size mismatch the verifier was designed to prevent but failed to detect.
