### Title
Permanent Subscription Funds Permanently Locked with No Escape Path ŌĆö (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract allows any subscription manager to mark their subscription as `isPermanent = true`. Once set, this flag is irreversible and unconditionally blocks the only fund-recovery function (`withdrawFunds`). There is no admin override, governance escape hatch, or alternative withdrawal path. Any ETH deposited into a permanent subscription is permanently locked in the contract with no mechanism for recovery.

---

### Finding Description

In `Scheduler.sol`, the `withdrawFunds` function contains an unconditional guard at the top:

```solidity
// Prevent withdrawals from permanent subscriptions
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

The `isPermanent` flag is a one-way door. The `updateSubscription` function also unconditionally reverts for any permanent subscription:

```solidity
// Updates to permanent subscriptions are not allowed
if (currentParams.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

This means a permanent subscription can never be deactivated, never have its `isPermanent` flag cleared, and its funds can never be withdrawn. The `addFunds` function, however, has no such restriction for permanent subscriptions ŌĆö it only checks `isActive` (which is always `true` for permanent subscriptions) and a deposit cap. This means anyone can continue to add ETH into a permanent subscription, compounding the locked balance.

There is no admin emergency withdrawal, no governance-controlled recovery function, and no time-locked escape mechanism anywhere in the contract.

The analog to M-09 is direct: just as `DefaultAccount::executeTransactionFromOutside` was empty and left users with no alternative path to access their L2 ETH when the operator was malicious, `Scheduler::withdrawFunds` is completely blocked for permanent subscriptions and leaves the manager with no alternative path to recover their ETH under any circumstance.

---

### Impact Explanation

Any ETH deposited into a permanent subscription is permanently and irrecoverably locked in the `Scheduler` contract. The subscription manager ŌĆö who deposited the funds ŌĆö has no mechanism to retrieve them. Additionally, because `addFunds` is permissionless (no access control beyond `isActive`), third parties can inadvertently or maliciously add more ETH to a permanent subscription, increasing the locked amount. The locked ETH will be slowly drained by keeper fees until the balance reaches zero, but any excess above what keepers consume is unrecoverable.

---

### Likelihood Explanation

The `isPermanent` flag is directly user-controlled. Any subscription manager who sets `isPermanent = true` ŌĆö either at creation via `createSubscription` or later via `updateSubscription` ŌĆö will trigger this condition. The feature is documented and exposed in the public interface (`SchedulerStructs.SubscriptionParams.isPermanent`). A protocol integrator who wants to guarantee price feed availability (a reasonable use case for `isPermanent`) will permanently lose access to their deposited ETH. No malicious actor is required; the user's own action is sufficient to trigger the fund lock.

---

### Recommendation

Add an emergency withdrawal path that is not blocked by `isPermanent`. Options include:

1. **Admin/governance override**: Allow the contract admin to withdraw funds from permanent subscriptions to the subscription manager's address in an emergency.
2. **Deactivation-then-withdraw**: Allow the manager to deactivate a permanent subscription (removing the `isPermanent` guarantee) after a time-lock period, then withdraw remaining funds.
3. **Excess-balance withdrawal**: Allow withdrawal of any balance above the minimum required to keep the subscription running, even for permanent subscriptions.

---

### Proof of Concept

1. Manager calls `createSubscription` with `isPermanent = true` and deposits 10 ETH.
2. Manager later calls `withdrawFunds(subscriptionId, 5 ether)`.
3. Transaction reverts with `CannotUpdatePermanentSubscription`.
4. Manager calls `updateSubscription(subscriptionId, params_with_isPermanent_false)`.
5. Transaction reverts with `CannotUpdatePermanentSubscription`.
6. No other function in the contract allows fund recovery.
7. The 10 ETH (minus keeper fees) is permanently locked.

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
