### Title
Missing Duplicate `tx_hash` Validation in `GetTransactionsProofProcess` Allows Malformed Proof Generation and Wasted Server Computation — (`File: util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

---

### Summary

`GetTransactionsProofProcess::execute()` accepts a peer-supplied list of `tx_hashes` without checking for duplicates. A duplicate tx_hash causes the same transaction index to appear twice in the `txs_and_tx_indices` vector fed to `CBMT::build_merkle_proof`, which is called with an `.expect()` that panics if the library returns `None` for invalid (duplicate) indices. At minimum, the response is malformed; at worst, the full node panics.

---

### Finding Description

In `GetTransactionsProofProcess::execute()`, the handler validates only that the `tx_hashes` list is non-empty and within the count limit: [1](#0-0) 

No uniqueness check is performed. The hashes are then partitioned into `found`/`missing`, and for each found hash the `(tx, tx_info.index)` pair is pushed into a `HashMap<block_hash, Vec<(tx, index)>>`: [2](#0-1) 

If the same tx_hash appears twice, both iterations push the same `(tx, tx_info.index)` into the same block's vector. The resulting duplicate indices are then passed directly to `CBMT::build_merkle_proof` with an `.expect()`: [3](#0-2) 

The comment `"build proof with verified inputs should be OK"` is incorrect — the inputs are **not** verified for duplicates. If `build_merkle_proof` returns `None` for duplicate indices, the `.expect()` panics and crashes the node.

**Contrast with `GetBlocksProofProcess`**, which correctly applies a `HashSet` deduplication guard before any processing: [4](#0-3) 

**Contrast with the relay `GetTransactionsProcess`**, which also applies a `HashSet` guard: [5](#0-4) 

The `GetTransactionsProofProcess` is the only handler in the light-client protocol server that is missing this guard.

---

### Impact Explanation

An unprivileged light-client peer sends a `GetTransactionsProof` message whose `tx_hashes` field contains the same on-chain transaction hash repeated N times (up to `GET_TRANSACTIONS_PROOF_LIMIT`). The server:

1. Performs N redundant database lookups for the same transaction.
2. Builds a `txs_and_tx_indices` vector with N copies of the same `(tx, index)` pair.
3. Calls `CBMT::build_merkle_proof` with N duplicate indices. If the library returns `None`, the `.expect()` **panics and crashes the full node process**.
4. If the library does not return `None`, the `filtered_block` contains N copies of the same transaction while the proof covers only one unique index — producing a structurally malformed response that will fail light-client verification.

The message handler is reachable from any peer that speaks the light-client protocol: [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged peer connected to a CKB full node with the light-client protocol enabled can send a crafted `GetTransactionsProof` message. No special privileges, keys, or majority hashpower are required. The attacker only needs to know one valid on-chain transaction hash (trivially obtained from any block explorer or by querying the node's RPC). The attack is repeatable and requires minimal bandwidth.

---

### Recommendation

Add a `HashSet` deduplication check for `tx_hashes` immediately after the count-limit check, mirroring the pattern already used in `GetBlocksProofProcess`:

```rust
let tx_hashes: Vec<_> = self.message.tx_hashes().to_entity().into_iter().collect();

let mut uniq = HashSet::new();
if !tx_hashes.iter().all(|hash| uniq.insert(hash)) {
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate tx hash exists");
}
```

This is consistent with the existing fix in:
- `get_blocks_proof.rs` lines 62–70 [4](#0-3) 
- `sync/src/relayer/get_transactions_process.rs` lines 54–61 [5](#0-4) 

---

### Proof of Concept

A malicious light-client peer constructs and sends:

```rust
let tx_hash = /* any known on-chain tx hash */;
let content = packed::GetTransactionsProof::new_builder()
    .last_hash(tip_hash)
    .tx_hashes(vec![tx_hash.clone(); GET_TRANSACTIONS_PROOF_LIMIT])
    .build();
let message = packed::LightClientMessage::new_builder().set(content).build();
nc.send_message_to(peer, message.as_bytes());
```

The server receives this, finds the transaction `GET_TRANSACTIONS_PROOF_LIMIT` times, builds a `txs_and_tx_indices` vector with `GET_TRANSACTIONS_PROOF_LIMIT` identical `(tx, index)` entries for the same block, and calls:

```rust
CBMT::build_merkle_proof(&block_tx_hashes, &[index; GET_TRANSACTIONS_PROOF_LIMIT])
    .expect("build proof with verified inputs should be OK");
// ^ panics if build_merkle_proof returns None for duplicate indices
``` [3](#0-2)
