Audit Report

## Title
Uncharged O(N) `header_deps` Linear Scan in `LoadHeader::load_header` Enables Cycle-Budget Bypass DoS — (`script/src/syscalls/load_header.rs`)

## Summary
`LoadHeader::load_header` performs a full linear scan over all `header_deps` on every syscall invocation for `Source::Transaction(Input)`, `Source::Transaction(CellDep)`, and `Source::Group(Input)` paths, but charges cycles only for bytes written to VM memory. Because the scan work is never metered, a script looping on `ckb_load_header_by_field` with a transaction carrying the maximum number of `header_deps` can consume orders of magnitude more wall-clock CPU time than the cycle budget implies, breaking the invariant that `max_block_cycles` bounds verification time and enabling effective DoS against any node that processes the transaction.

## Finding Description
In `script/src/syscalls/load_header.rs`, `load_header` (lines 56–70) performs a full linear scan over the molecule `Byte32Vec` returned by `header_deps()` before fetching the header:

```rust
// load_header.rs L61-64
if self
    .header_deps()
    .into_iter()
    .any(|hash| &hash == block_hash)
``` [1](#0-0) 

This scan is re-executed on every syscall invocation. The `header_deps()` helper (lines 42–44) re-materializes the molecule `Byte32Vec` from the transaction on every call, adding further overhead: [2](#0-1) 

After the scan, the only cycle charge in `ecall` is:

```rust
// load_header.rs L175
machine.add_cycles_no_checking(transferred_byte_cycles(len))?;
``` [3](#0-2) 

`transferred_byte_cycles` charges `bytes.div_ceil(4)` cycles — for `LOAD_HEADER_BY_FIELD` returning an 8-byte field, this is exactly **2 cycles**: [4](#0-3) 

The O(N) scan work over `header_deps` is **never charged**. There is no `MAX_HEADER_DEPS` constant anywhere in the codebase. The only bound is `MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT = 597 × 1,000 = 597,000 bytes`: [5](#0-4) 

This permits up to `597,000 / 32 ≈ 18,656` distinct 32-byte header dep hashes per transaction. `DuplicateDepsVerifier` prevents duplicate entries but imposes no count limit: [6](#0-5) 

The `Source::Transaction(SourceEntry::HeaderDep)` path (lines 85–95) is **not** affected — it uses direct index access `header_deps().get(index)` with no scan. The vulnerability is confined to the `Input`, `CellDep`, and `Group(Input)` source paths that call `load_header(cell_meta)`: [7](#0-6) 

## Impact Explanation
This breaks the fundamental invariant that `max_block_cycles` bounds script verification wall time. `MAX_BLOCK_CYCLES = 3,500,000 × 1,000 = 3,500,000,000`: [5](#0-4) 

With ~20–25 cycles per syscall invocation (2 charged + ~18–23 RISC-V instruction cycles for the loop), an attacker achieves ~140–175 million syscall invocations within the cycle budget. Each invocation triggers an 18,656-element × 32-byte hash scan, yielding on the order of **2.5–3 trillion hash comparisons** per transaction verification. At realistic hardware speeds (~10¹⁰ comparisons/sec), this translates to ~250 seconds of wall time against a cycle-model-predicted sub-second budget — a factor of ~200× amplification.

Both tx-pool admission and block verification execute this path, so every full node processing the transaction or a block containing it is affected. This maps to the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The attack is fully unprivileged. The attacker needs only: (1) N distinct canonical-chain block hashes, trivially enumerable from any block explorer or node RPC; (2) a minimal RISC-V script (~30 bytes) that loops calling `ckb_load_header_by_field` against a fixed input index; (3) one live UTXO to spend. Because the script hits the cycle limit and the transaction is rejected, the UTXO is not consumed — the attacker can resubmit the same transaction repeatedly, each time triggering ~250 seconds of CPU time on every node that processes it. No special keys, hashpower, or privileged access are required.

## Recommendation
Charge cycles proportional to the number of `header_deps` scanned on each invocation. Before or during the scan in `load_header`, add a base cost of `header_deps().len()` cycles (one cycle per 32-byte element compared) via `machine.add_cycles_no_checking(header_deps_len as u64)?`. Alternatively — and preferably — pre-build a `HashSet<Byte32>` from `header_deps` once per script group execution (e.g., in `LoadHeader::new` or at the start of `ecall`) and replace the O(N) scan with an O(1) lookup, eliminating the unbounded work entirely. The latter approach also removes the per-call molecule re-materialization overhead at line 43. [2](#0-1) 

## Proof of Concept
```
Transaction layout:
  header_deps: [h1, h2, ..., h_N]   (N = ~18,000 distinct mainnet block hashes)
  inputs: [one live UTXO whose block_hash == h1]
  cell_deps: [script_cell]
  witnesses: [...]

Script (RISC-V, ~30 bytes):
  loop:
    li a3, 0          // index = 0 (first input)
    li a4, 1          // source = CKB_SOURCE_INPUT (triggers load_header scan)
    li a5, 0          // field = EpochNumber
    li a7, 2082       // LOAD_HEADER_BY_FIELD_SYSCALL_NUMBER
    ecall             // O(N) scan, charges 2 cycles
    j loop            // repeat until cycle limit

Expected result:
  - Cycles consumed: 3,500,000,000 (hits max_block_cycles)
  - Syscall invocations: ~140,000,000
  - Scan iterations per call: ~18,000
  - Total hash comparisons: ~2.5 × 10^12
  - Actual wall time: ~250 seconds
  - Cycle-model-predicted wall time: <1 second
```

To reproduce: construct the transaction with `ckb-sdk`, submit via `send_transaction` RPC, and measure node CPU time during tx-pool admission. The wall time will be orders of magnitude beyond what the cycle budget predicts.

    pub fn verify(&self) -> Result<(), Error> {
        let transaction = self.transaction;
        let mut seen_cells = HashSet::with_capacity(self.transaction.cell_deps().len());
        let mut seen_headers = HashSet::with_capacity(self.transaction.header_deps().len());

        if let Some(dep) = transaction
            .cell_deps_iter()
            .find_map(|dep| seen_cells.replace(dep))
        {
            return Err(TransactionError::DuplicateCellDeps {
                out_point: dep.out_point(),
            }
            .into());
        }
        if let Some(hash) = transaction
            .header_deps_iter()
            .find_map(|hash| seen_headers.replace(hash))
        {
            return Err(TransactionError::DuplicateHeaderDeps { hash }.into());
        }
        Ok(())
    }
