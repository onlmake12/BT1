### Title
Unbounded `while` Loop in `executeCallback` Enables Out-of-Order Fulfillment DoS — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` function contains an unbounded `while` loop that linearly scans forward through all sequence numbers from `_state.firstUnfulfilledSeq` to find the next active request. An unprivileged attacker can create a large number of requests and fulfill them in reverse order, forcing the final `executeCallback` call to iterate over an arbitrarily large range of already-cleared sequence numbers. This can exhaust the block gas limit, causing the transaction to revert and permanently preventing the earliest request from ever being fulfilled.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` advances `_state.firstUnfulfilledSeq` with this loop:

```solidity
// Echo.sol lines 169–174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs at least one cold `SLOAD` (2,100 gas) against `_state.requests[shortKey]`, and potentially a second cold `SLOAD` against `_state.requestsOverflow[key]` for overflow entries. [2](#0-1) 

There is no cap on the number of iterations. The loop runs for as many steps as there are cleared (inactive) sequence numbers between `firstUnfulfilledSeq` and the sequence number just fulfilled.

The `requestsOverflow` mapping is unbounded — when the 32 fixed slots (`NUM_REQUESTS = 32`) are all occupied, new requests spill into `requestsOverflow`, allowing an attacker to accumulate an arbitrary number of outstanding requests. [3](#0-2) 

The codebase's own benchmark test explicitly documents this worst-case scenario:

> *"The last fulfillment will be the most expensive since it needs to linearly scan through all the fulfilled requests in storage in order to update `_state.lastUnfulfilledReq`"* [4](#0-3) 

---

### Impact Explanation

If the `while` loop exceeds the block gas limit, the entire `executeCallback` transaction reverts. Because `clearRequest` is called **before** the loop, the revert undoes the clearing — the request remains active but can never be successfully fulfilled: every subsequent attempt to fulfill it will re-enter the same loop and hit the same OOG condition. The requester's locked fee (`req.fee`) is permanently trapped in the contract with no recovery path. [5](#0-4) 

---

### Likelihood Explanation

`requestPriceUpdatesWithCallback` is permissionless and requires only a small fee per request. `executeCallback` has no access control after the exclusivity period expires — any address may call it. [6](#0-5) [7](#0-6) 

An attacker pays fees proportional to the number of requests created, but the cost is bounded by the attacker's budget, not by any protocol-enforced limit. With a 30 M gas block limit and ~2,100 gas per cold `SLOAD`, roughly 14,000 iterations suffice to cause OOG — achievable by creating and reverse-fulfilling that many requests.

---

### Recommendation

Replace the unbounded `while` loop with a bounded scan, or eliminate the linear scan entirely by using a doubly-linked list of active requests (as noted in the inline TODO comment at line 167). A minimal fix is to cap the number of iterations per call:

```solidity
uint256 maxAdvance = 1000; // or a governance-configurable constant
uint256 steps = 0;
while (
    steps < maxAdvance &&
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
    steps++;
}
``` [8](#0-7) 

---

### Proof of Concept

1. Attacker calls `requestPriceUpdatesWithCallback` N times (e.g., N = 15,000), paying the minimum fee each time. Sequence numbers 1 … N are created.
2. After the exclusivity period, the attacker calls `executeCallback` for sequences N, N−1, …, 2 (in reverse order). Each call succeeds; `firstUnfulfilledSeq` remains at 1 because request 1 is still active.
3. Attacker calls `executeCallback` for sequence 1. The `while` loop must now advance `firstUnfulfilledSeq` from 1 through N−1 cleared entries. Each iteration performs at least one cold `SLOAD` (~2,100 gas). Total loop gas ≈ 15,000 × 2,100 = 31,500,000 gas — exceeding the 30 M block gas limit.
4. The transaction reverts. Request 1 is never cleared. The requester's fee is permanently locked. [9](#0-8)
