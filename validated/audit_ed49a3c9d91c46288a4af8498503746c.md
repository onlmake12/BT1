### Title
Entropy Contract on Blast Never Configures Gas Yield to Claimable — All Gas Yield Permanently Lost - (File: `target_chains/ethereum/contracts/contracts/entropy/EntropyUpgradable.sol`)

---

### Summary

The `EntropyUpgradable` contract is deployed on Blast (address `0x5744Cbf430D99456a0A8771208b674F27f8EF0Fb`). On Blast, both ETH and gas yield modes default to `VOID`, meaning yield does not accrue unless `IBlast.configureClaimableGas()` is explicitly called. The Entropy contract never calls this function — neither in its `initialize()` nor anywhere in `EntropyGovernance` — and provides no mechanism to do so. All gas yield from the Entropy contract on Blast is permanently lost.

---

### Finding Description

On Blast, the precompile at `0x4300000000000000000000000000000000000002` controls gas and ETH yield modes. The default gas yield mode is `VOID`, which means gas fees do not accrue as yield unless the contract explicitly calls `IBlast.configureClaimableGas()`.

The Pyth team was already aware of this requirement for the oracle contract: a chain-specific patch exists at `target_chains/ethereum/contracts/chain_patches/blast-gas-claim-patch.diff` that adds a `configureClaimableGas()` function to `Pyth.sol`:

```diff
+interface IBlast {
+    function configureClaimableGas() external;
+}
+    function configureClaimableGas() external {
+        IBlast(0x4300000000000000000000000000000000000002)
+            .configureClaimableGas();
+    }
```

However, no equivalent patch or mechanism exists for the Entropy contract. The `EntropyUpgradable.initialize()` function sets up admin, fees, and provider state, but never calls `configureClaimableGas()`:

```solidity
function initialize(
    address owner,
    address admin,
    uint128 pythFeeInWei,
    address defaultProvider,
    bool prefillRequestStorage
) public initializer {
    // ... no IBlast.configureClaimableGas() call
    Entropy._initialize(admin, pythFeeInWei, defaultProvider, prefillRequestStorage);
    _transferOwnership(owner);
}
```

`EntropyGovernance` provides `withdrawFee()` to claim accrued protocol fees from user payments, but this is entirely separate from Blast gas yield — it only withdraws `_state.accruedPythFeesInWei`, not Blast-native gas yield. There is no function in `EntropyGovernance` or `EntropyUpgradable` to call `IBlast.configureClaimableGas()`.

A grep across all `.sol` files confirms zero occurrences of `configureClaimableGas`, `IBlast`, or `configureGovernor` in any production Solidity contract — only in the patch diff file.

---

### Impact Explanation

All gas consumed by transactions interacting with the Entropy contract on Blast (randomness requests, reveals, callbacks, registrations) generates gas yield that could be claimed by the Pyth protocol. Because gas yield mode is permanently `VOID`, this yield never accrues and can never be claimed. The loss is proportional to total gas usage of the Entropy contract on Blast over its lifetime — a permanent, irrecoverable loss of protocol revenue.

---

### Likelihood Explanation

Certainty: the gas yield mode is `VOID` by default on Blast and there is no code path that changes it. Every transaction on the Entropy Blast deployment silently discards gas yield. The Pyth team's own patch for `Pyth.sol` confirms they understood this requirement for the oracle contract, making the omission for Entropy a clear oversight.

---

### Recommendation

Apply the same pattern used in `blast-gas-claim-patch.diff` to `EntropyUpgradable`. The most robust fix is to call `configureClaimableGas()` inside `initialize()`:

```solidity
interface IBlast {
    function configureClaimableGas() external;
}

function initialize(...) public initializer {
    // existing setup ...
    // Configure Blast gas yield to claimable (Blast-specific deployment only)
    IBlast(0x4300000000000000000000000000000000000002).configureClaimableGas();
}
```

Alternatively, expose a one-time governance function callable by the admin/owner to invoke `configureClaimableGas()` post-deployment.
