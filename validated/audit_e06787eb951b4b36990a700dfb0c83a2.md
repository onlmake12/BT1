### Title
Permanent Subscription Deposit Cap Bypass via Cumulative `addFunds` Calls — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `Scheduler` contract enforces a `MAX_DEPOSIT_LIMIT` (100 ETH) on permanent subscriptions to prevent accidentally locking excessive ETH irrecoverably. However, the check is applied only to the **per-transaction** `msg.value`, not to the **cumulative** `status.balanceInWei`. Any caller can bypass the limit entirely by making multiple `addFunds` calls each just at or below 100 ETH, permanently locking an unbounded amount of ETH in the contract with no withdrawal path.

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
``` [1](#0-0) 

Both `createSubscription` and `addFunds` enforce this limit only against the single-call `msg.value`:

```solidity
// createSubscription
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [2](#0-1) 

```solidity
// addFunds
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [3](#0-2) 

Neither check compares `status.balanceInWei + msg.value` against `MAX_DEPOSIT_LIMIT`. The cumulative balance is never bounded.

Permanent subscriptions explicitly block all withdrawals:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [4](#0-3) 

`addFunds` has no `onlyManager` modifier — it is callable by **any address** on any active subscription:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
``` [5](#0-4) 

### Impact Explanation

ETH deposited into a permanent subscription is permanently locked — it can only leave the contract via keeper fee payments over time. The `MAX_DEPOSIT_LIMIT` was specifically introduced to prevent users from accidentally locking more ETH than intended. Because the check is per-transaction rather than cumulative, the protection is completely ineffective. A user who calls `addFunds` N times with 100 ETH each will lock N×100 ETH with no recourse. Additionally, since `addFunds` is permissionless, a third party can grief a permanent subscription by depositing their own ETH into it, permanently locking those funds in a subscription they do not control.

### Likelihood Explanation

Medium. The `addFunds` function is a normal operational call that subscription managers are expected to use repeatedly to top up balances. A manager who misunderstands the per-call vs. cumulative semantics of `MAX_DEPOSIT_LIMIT` will silently exceed the intended cap. The permissionless nature of `addFunds` also opens a griefing vector requiring no privileged access.

### Recommendation

Replace the per-transaction check with a cumulative check against the post-deposit balance:

```solidity
if (params.isPermanent &&
    status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Apply the same fix to `createSubscription`.

### Proof of Concept

1. Deploy `SchedulerUpgradeable` with `MAX_DEPOSIT_LIMIT = 100 ether`.
2. Create a permanent subscription with `msg.value = 100 ether` — succeeds.
3. Call `addFunds{value: 100 ether}(subscriptionId)` — succeeds (per-tx check passes).
4. Repeat step 3 ten times — 1000 ETH is now locked in the permanent subscription.
5. Call `withdrawFunds(subscriptionId, 1 ether)` — reverts with `CannotUpdatePermanentSubscription`.
6. The 1000 ETH is permanently locked, far exceeding the intended 100 ETH cap.
