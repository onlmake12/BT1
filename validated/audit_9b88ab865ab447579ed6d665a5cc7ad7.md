Audit Report

## Title
TOCTOU Race in Tx-Pool Admission Allows Redundant Script Verification DoS — (File: `tx-pool/src/process.rs`)

## Summary
The `_process_tx` function performs its duplicate-transaction check (`check_txid_collision`) under a read lock in `pre_check()`, releases that lock, then runs full CKB-VM script verification with no lock held. `submit_entry()` acquires a write lock but never re-checks for txid collision. Concurrent submissions of the same transaction therefore each independently complete full script verification before any deduplication occurs, enabling an attacker to force N × `max_block_cycles` of wasted CPU work per attack round.

## Finding Description
The three-phase pipeline in `_process_tx` is confirmed by the code:

**Phase 1 — Check (read lock, then released):**
`pre_check()` at `process.rs:276–314` acquires `with_tx_pool_read_lock`, which holds `self.tx_pool.read().await` for the duration of the closure only (`process.rs:251–255`). Inside the closure, `check_txid_collision` at `process.rs:282` calls `tx_pool.contains_proposal_id(&short_id)` (`util.rs:20–26`). The read lock is dropped when the closure returns at line 313. No record of the in-flight transaction is written anywhere.

**Phase 2 — Expensive work (no lock held):**
`verify_rtx()` at `process.rs:724–732` runs with no lock held. For the synchronous RPC path (`command_rx` is `None`), `util.rs:117` uses `block_in_place` to run `ContextualTransactionVerifier::verify(max_tx_verify_cycles, false)`, which executes full CKB-VM script verification up to `max_block_cycles` cycles.

**Phase 3 — State update (write lock, no re-check):**
`submit_entry()` at `process.rs:96–170` acquires `with_tx_pool_write_lock` (`process.rs:103`). Critically, it does **not** call `check_txid_collision`. It only calls `check_rbf` (if RBF enabled, `process.rs:106`) or `find_conflict_outpoint` (if RBF disabled, `process.rs:111`). The `find_conflict_outpoint` path will eventually reject the duplicate via `Reject::Resolve(OutPointError::Dead(outpoint))` at `process.rs:113`, but only after both Phase 2 executions have already completed.

The pre-flight guards in `process_tx` at `process.rs:409` (`verify_queue_contains` and `orphan_contains`) do not close this window. Both are separate async calls with separate lock acquisitions (`process.rs:237–245`); they check the verify queue and orphan pool, not the main tx pool. Two concurrent `process_tx` calls can both pass line 409 before either enters `_process_tx`, and then both pass `pre_check()` before either reaches `submit_entry()`.

The `txs_verify_cache` check in `verify_rtx` at `util.rs:96–100` provides no protection: all N concurrent submissions of a new transaction are cache misses simultaneously, so all N take the expensive `block_in_place` path.

## Impact Explanation
An unprivileged attacker submitting the same valid transaction N times concurrently forces N independent executions of `ContextualTransactionVerifier::verify(max_block_cycles)`. Script verification is the most CPU-intensive operation in the node — it runs the CKB-VM for up to `max_block_cycles` cycles per call. With N concurrent submissions, the node performs N × `max_block_cycles` of wasted CPU work before `submit_entry()` rejects N−1 of them. This can saturate the `block_in_place` thread pool, delaying or blocking legitimate transaction processing. This matches: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The attack requires no privileged access, no keys, and no special network position — only a valid transaction and access to the node's RPC port. The attacker controls both N (number of concurrent submissions) and the script complexity of the transaction, making the CPU cost arbitrarily large relative to the cost of the attack. The race window spans the entire duration of script verification (potentially hundreds of milliseconds for complex scripts), making it wide and reliable. The attack is trivially repeatable with standard HTTP clients.

## Recommendation
Re-check for txid collision inside `submit_entry()` under the write lock, before any other work, and return `Reject::Duplicated` immediately if the tx is already present:

```rust
pub(crate) async fn submit_entry(...) {
    self.with_tx_pool_write_lock(move |tx_pool, snapshot| {
        // Authoritative re-check under write lock
        check_txid_collision(tx_pool, entry.transaction())?;
        // ... existing RBF / conflict checks ...
    }).await
}
```

Additionally, maintain an in-flight set (a `HashSet<ProposalShortId>` protected by its own lock) populated atomically at the end of `pre_check()` and cleared at the end of `submit_entry()`, so that concurrent duplicate submissions are rejected before entering the expensive verification phase entirely.

## Proof of Concept
1. Obtain any valid transaction `tx` with non-trivial script verification cost (e.g., near `max_block_cycles`).
2. Open N concurrent HTTP connections to the node's RPC endpoint.
3. Send `send_transaction(tx)` on all N connections simultaneously.
4. Observe via node metrics or CPU profiling that `verify_rtx` / `block_in_place` is invoked N times for the same transaction hash, consuming N × script-verification CPU time.
5. Confirm that only one submission succeeds; N−1 are rejected with a conflict error from `submit_entry` (via `find_conflict_outpoint` returning `Reject::Resolve(OutPointError::Dead(outpoint))`), but only after all N verifications have completed. pub(crate) fn check_txid_collision(tx_pool: &TxPool, tx: &TransactionView) -> Result<(), Reject> {
    let short_id = tx.proposal_short_id();
    if tx_pool.contains_proposal_id(&short_id) {
        return Err(Reject::Duplicated(tx.hash()));
    }
    Ok(())
}
