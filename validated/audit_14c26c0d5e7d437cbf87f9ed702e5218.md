Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Spoofed `from` PeerId in ConnectionRequest — (`network/src/protocols/hole_punching/mod.rs`, `network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter` is keyed by `(content.from, content.to, item_id)`, where `content.from` is taken directly from the attacker-controlled message body with no validation against the actual sending peer's identity. The outer rate limiter allows 30 `ConnectionRequest` messages per second per session, each carrying a distinct spoofed `from`, inserting 30 new `HashMapStateStore` entries per second. Because `retain_recent()` is never called during the connection lifetime — only on `disconnected()` — a persistent attacker who never disconnects causes unbounded memory growth, leading to OOM and node crash.

## Finding Description
`forward_rate_limiter` is a single `RateLimiter<(PeerId, PeerId, u32)>` backed by `governor::state::keyed::HashMapStateStore<(PeerId, PeerId, u32)>`, shared across all sessions on the `HolePunching` struct.

`from` is parsed from raw message bytes via `PeerId::from_bytes(value.from().raw_data().to_vec())` with no check that it matches the actual sending session's peer ID. Any syntactically valid multihash bytes pass this check.

`forward_rate_limiter.check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))` is then called using this unvalidated `content.from`. Each unique `(from, to, item_id)` triple that has not been seen before creates a new entry in the `HashMapStateStore`. Since the attacker supplies a fresh `from` in each of the 30 allowed messages per second, all 30 pass the forward rate limit check (quota = 1/sec per key, and each key is new), and all 30 create new store entries.

`retain_recent()` is called on both rate limiters only inside `disconnected()`. The `notify()` callback, which fires every `CHECK_INTERVAL = 5 minutes`, only prunes `pending_delivered` and `inflight_requests` — it never calls `retain_recent()` on either rate limiter. Entries eligible for eviction at `T+1s` remain resident until the session disconnects. A persistent attacker simply never disconnects.

## Impact Explanation
**High: Vulnerabilities which could easily crash a CKB node.** At 30 entries/sec per session, a single attacker session accumulates ~9,000 entries in 5 minutes. With 50 concurrent attacker sessions (within typical inbound connection limits), the store reaches ~450,000 entries in 5 minutes and continues growing until OOM. `forward_rate_limiter` is process-global state on the `HolePunching` struct, so the crash affects the entire node process.

## Likelihood Explanation
No special privileges are required — any peer that can establish a standard P2P connection can trigger this. CKB nodes accept inbound connections from the public internet. Generating 30 syntactically valid `PeerId` values per second is trivial. The attack is repeatable, requires no victim interaction, and is not mitigated by any existing check in the `received()` or `notify()` paths.

## Recommendation
1. **Call `retain_recent()` periodically inside `notify()`** (every `CHECK_INTERVAL`) on both `rate_limiter` and `forward_rate_limiter`, not only on disconnect.
2. **Validate `from` against the actual sender**: reject messages where `content.from` does not match the peer ID of the sending session. This eliminates the ability to create arbitrary new keys.
3. **Cap the `forward_rate_limiter` store size**: add a maximum entry count and evict oldest entries when the cap is reached.

## Proof of Concept
```rust
// Setup: connect one attacker session
let mut protocol = HolePunching::new(network_state);
let session_id = PeerIndex::new(1);

// Attack: send 30 ConnectionRequest/sec for 300 seconds, each with unique `from`
for second in 0..300 {
    for i in 0..30 {
        let from = generate_unique_peer_id(second * 30 + i); // distinct each time
        let to   = generate_peer_id(0);
        let msg  = build_connection_request(from, to, max_hops=1);
        protocol.received(ctx_for(session_id), msg).await;
    }
    advance_clock_by(1_second);
}

// Assert: forward_rate_limiter has NOT grown unboundedly
// FAILS: actual ~9000 entries, retain_recent() was never called
assert!(
    forward_rate_limiter_len(&protocol) <= 30,
    "invariant violated: forward_rate_limiter grew unboundedly"
);
```

The outer limiter passes all 30 messages/sec (quota = 30 per `(session_id, item_id)`). Each unique `from` creates a new `HashMapStateStore` entry and passes the forward rate limit (quota = 1/sec per `(from, to, item_id)`, and each key is new). After 300 seconds with no disconnect, `retain_recent()` has never been called, so all ~9,000 entries remain resident. With 50 concurrent attacker sessions the store reaches ~450,000 entries and continues growing until OOM.
