Audit Report

## Title
Synchronous Blocking VM Execution in `estimate_cycles` Saturates RPC Tokio Runtime — (`rpc/src/module/chain.rs`)

## Summary

`CyclesEstimator::run()` calls `ScriptVerifier::verify(max_block_cycles)` synchronously on a tokio worker thread with no offloading and no cycle cap below the consensus maximum. The RPC server runs on axum-on-tokio; blocking the worker thread prevents the tokio runtime from scheduling other tasks on that thread. Concurrent `estimate_cycles` requests with a max-cycle script saturate all RPC worker threads, rendering all RPC methods unresponsive.

## Finding Description

`CyclesEstimator::run()` sets `max_cycles = consensus.max_block_cycles` and calls `ScriptVerifier::verify(max_cycles)` as a fully synchronous, blocking operation:

```rust
// rpc/src/module/chain.rs L2380, L2383-2389
let max_cycles = consensus.max_block_cycles;
...
ScriptVerifier::new(
    Arc::new(resolved),
    snapshot.as_data_loader(),
    consensus,
    Arc::new(tx_env),
)
.verify(max_cycles)
``` [1](#0-0) 

`estimate_cycles` dispatches directly to this with no interposition:

```rust
// rpc/src/module/chain.rs L2119-2122
fn estimate_cycles(&self, tx: Transaction) -> Result<EstimateCycles> {
    let tx: packed::Transaction = tx.into();
    CyclesEstimator::new(&self.shared).run(tx)
}
``` [2](#0-1) 

The RPC server is confirmed to be axum-on-tokio. `handle_jsonrpc` is an `async fn` that calls `io.handle_call(call, T::default()).await`; when the underlying synchronous handler runs, it blocks the tokio worker thread for the full duration of VM execution: [3](#0-2) 

A `TimeoutLayer::with_status_code(..., Duration::from_secs(30))` is applied: [4](#0-3) 

However, this timeout is implemented as an async future race. A blocking synchronous call that never yields to the tokio runtime cannot be preempted by an async timeout — the timeout future cannot be polled while the worker thread is occupied executing synchronous code.

A search for `spawn_blocking`, `block_in_place`, `rayon`, or `tokio::task` across all of `rpc/src/**` returns **no results** — the VM execution is never offloaded to a blocking thread pool. [5](#0-4) 

## Impact Explanation

Sending N concurrent `estimate_cycles` requests (N ≥ tokio worker thread count for the RPC runtime) with a max-cycle looping script blocks all worker threads. All RPC methods — including `get_block_template` — become unresponsive for the duration of the attack. This matches the allowed CKB bounty impact: **"Any local RPC API crash" (Note, 0–500 points)**. The node's p2p/sync runtime is separate and unaffected.

## Likelihood Explanation

Requires only HTTP access to the RPC port and a live cell referencing a max-cycle looping script. No authentication, no PoW, no privilege escalation. The Chain module is enabled by default. The default `127.0.0.1` listen address limits exposure to local attackers; nodes with publicly exposed RPC (wallet/dApp infrastructure) are directly vulnerable. The attack is repeatable and requires no special knowledge beyond the public RPC API.

## Recommendation

- Offload `ScriptVerifier::verify()` to `tokio::task::spawn_blocking` within `CyclesEstimator::run()` so tokio worker threads are not blocked
- Add a configurable `max_cycles_for_estimate` cap in `RpcConfig`, defaulting to a fraction of `max_block_cycles`
- Add a per-IP or global concurrency semaphore for `estimate_cycles` to bound simultaneous VM executions

## Proof of Concept

1. Deploy a CKB script that executes a tight loop consuming exactly `max_block_cycles` cycles and publish it as a live cell on the target network
2. Determine the tokio worker thread count for the RPC runtime (defaults to number of CPU cores)
3. Send `thread_count + 1` concurrent `estimate_cycles` JSON-RPC POST requests referencing that cell to the RPC endpoint
4. While those requests are in-flight, issue a `get_block_template` call
5. Observe that `get_block_template` does not respond until at least one VM execution completes, confirming full RPC runtime saturation

    pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();
                let tx_env = TxVerifyEnv::new_submit(tip_header);
                match ScriptVerifier::new(
                    Arc::new(resolved),
                    snapshot.as_data_loader(),
                    consensus,
                    Arc::new(tx_env),
                )
                .verify(max_cycles)
                {
                    Ok(cycles) => Ok(EstimateCycles {
                        cycles: cycles.into(),
                    }),
                    Err(err) => Err(RPCError::custom_with_error(
                        RPCError::TransactionFailedToVerify,
                        err,
                    )),
                }
            }
            Err(err) => Err(RPCError::custom_with_error(
                RPCError::TransactionFailedToResolve,
                err,
            )),
        }
