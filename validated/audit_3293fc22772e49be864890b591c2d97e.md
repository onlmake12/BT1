### Title
Unbounded WebSocket Subscription with Post-Serialization Rate Check Bypass Enables Memory Exhaustion DoS — (`apps/hermes/server/src/api/ws.rs`)

---

### Summary

The Hermes WebSocket handler imposes no cap on the number of price feed IDs a single client may subscribe to, and its per-IP rate limiter is applied **after** full data materialization and serialization. Critically, the rate limiter is **completely bypassed** when the client's IP address cannot be resolved from the configured header (e.g., direct connection without a proxy). An unauthenticated attacker can subscribe to all available price feeds (~2000+), causing the server to fetch and serialize the entire feed set on every aggregation event with no throttling, exhausting memory and CPU.

---

### Finding Description

**Root cause 1 — No cap on subscribed feed count:**

In `handle_client_message`, when a `Subscribe` message arrives, the handler inserts every valid price ID into `self.price_feeds_with_config` with no upper bound: [1](#0-0) 

There is no check like `if self.price_feeds_with_config.len() >= MAX_FEEDS { return error; }`. A single client can subscribe to every feed Hermes knows about.

**Root cause 2 — Full materialization before rate check:**

On every `AggregationEvent`, `handle_price_feeds_update` collects all subscribed IDs and calls `get_price_feeds_with_update_data` for the entire set in one shot: [2](#0-1) 

It then iterates through every returned update and serializes each one to a JSON string: [3](#0-2) 

Only **after** this serialization does the code consult the rate limiter: [4](#0-3) 

This mirrors the Monad pattern exactly: the expensive allocation happens first; the guard fires too late.

**Root cause 3 — Rate limiter entirely skipped when IP is `None`:**

The requester IP is extracted from a configurable header (`X-Forwarded-For` by default): [5](#0-4) 

If the header is absent, malformed, or the value fails `parse()`, `requester_ip` is `None`. The rate-limit block is guarded by `if let Some(ip_addr) = self.ip_addr`, so when `ip_addr` is `None` the entire rate-limiting path is skipped: [6](#0-5) 

A direct TCP connection to Hermes (bypassing any reverse proxy) will always have `ip_addr = None`, meaning **zero rate limiting** applies regardless of how many feeds are subscribed or how fast updates arrive.

---

### Impact Explanation

Pyth currently publishes ~2000+ price feeds. Each serialized `RpcPriceFeed` with `binary: true` includes a base64-encoded accumulator proof and VAA, typically several KB each. On every aggregation slot (~400 ms on Pythnet):

- Server allocates a `Vec` of all subscribed `PriceFeedWithUpdateData` structs
- Serializes each to a JSON string (heap allocation per feed)
- Feeds all messages into the write buffer before flushing

With N=2000 feeds × ~5 KB each = ~10 MB of heap per update cycle, at ~2.5 updates/second = ~25 MB/s of allocation pressure per attacking connection. Multiple concurrent connections multiply this linearly. The process will either be OOM-killed or thrash the allocator, denying service to legitimate consumers.

---

### Likelihood Explanation

- No authentication is required to open a WebSocket connection to Hermes.
- The public endpoint `hermes.pyth.network/ws` is reachable by any internet client.
- Subscribing to all feeds requires a single JSON message with all known feed IDs (the full list is publicly available via `/v2/price_feeds`).
- Direct connections (no `X-Forwarded-For` header) completely bypass the rate limiter.
- The attack is deterministic and requires no brute force, Sybil attack, or privileged access.

---

### Recommendation

1. **Cap subscribed feeds per connection**: Enforce a hard limit (e.g., 100–500) on `price_feeds_with_config.len()` before inserting new IDs in `handle_client_message`.

2. **Move rate check before materialization**: Check the rate budget against an estimated byte cost (or a fixed per-feed token) before calling `get_price_feeds_with_update_data`, not after serialization.

3. **Require IP for rate limiting**: If `requester_ip` is `None`, either reject the connection or apply a conservative global rate limit rather than skipping all throttling.

4. **Cap concurrent WebSocket connections**: Add a global or per-IP connection counter with a hard ceiling.

---

### Proof of Concept

```
# 1. Fetch all known feed IDs (public, unauthenticated)
curl https://hermes.pyth.network/v2/price_feeds | jq '[.[].id]' > all_ids.json

# 2. Connect directly (no X-Forwarded-For header → ip_addr = None → no rate limit)
wscat -c wss://hermes.pyth.network/ws

# 3. Send subscribe message with all ~2000 feed IDs and binary=true
{"type":"subscribe","ids":[<all_ids>],"binary":true,"verbose":true}

# Result: on every ~400ms aggregation event, the server fetches and serializes
# ~2000 × ~5KB = ~10MB of data with zero rate limiting applied.
# Repeat with N concurrent connections to exhaust server memory.
```

The `binary: true` flag maximizes per-feed payload size by including base64-encoded accumulator update data, amplifying the memory pressure per update cycle.
