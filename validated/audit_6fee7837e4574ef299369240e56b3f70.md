### Title
Rust `DaoCalculator` Reads DAO Witness Header-Dep Index as Full `u64` While On-Chain C Script Uses Only the Lowest Byte, Enabling Consensus Split — (`File: util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` function reads the header-dep index from the DAO withdrawal witness as a full 8-byte `u64`, but the on-chain C DAO script resolves the same field using only its **lowest byte**. A transaction sender can craft a DAO withdrawal with a witness index > 255 that the C VM accepts (resolving to the correct deposit header via the low byte) but that Rust rejects (resolving to a wrong block via the full `u64`). Because `FeeCalculator` calls `DaoCalculator` inside the consensus-critical `ContextualTransactionVerifier::verify()`, Rust nodes will reject any block containing such a transaction, causing a chain split.

---

### Finding Description

In `util/dao/src/lib.rs`, `DaoCalculator::transaction_maximum_withdraw` extracts the deposit-header index from the witness `input_type` field and uses it to look up the deposit block hash:

```rust
// line 91
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
// line 94-96
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // full u64 used as index
```

The on-chain C DAO script (`ckb-system-scripts/c/dao.c`) reads the same 8-byte field but resolves the index using only its **lowest byte** (i.e., `index & 0xFF`). This is documented explicitly in the test added to the codebase:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

The test `check_dao_withdraw_header_dep_index_exceeds_u8` constructs a transaction with witness index `257` (lowest byte = `1`), places the deposit block at `header_deps[1]`, and asserts that Rust's `DaoCalculator` returns an error — confirming the divergence is present and unmitigated in the current code.

The `DaoCalculator` is invoked through `FeeCalculator::transaction_fee()` inside `ContextualTransactionVerifier::verify()`, which is called for every non-cellbase transaction during block verification in `BlockTxsVerifier::verify()`. A `DaoError` propagated from this path causes the entire block to be rejected.

---

### Impact Explanation

**Impact: High**

A DAO withdrawal transaction crafted with a witness index > 255 will:
1. Pass the on-chain C DAO script (C VM resolves `index & 0xFF` → correct deposit header → valid withdrawal).
2. Fail Rust's `DaoCalculator` (Rust resolves the full `u64` → wrong block → block-number mismatch → `DaoError::InvalidOutPoint`).

Because `FeeCalculator::transaction_fee()` propagates this error via `?` inside `ContextualTransactionVerifier::verify()`, Rust nodes reject any block containing such a transaction. This causes a **consensus split**: the C-VM-based chain accepts the block; Rust nodes do not. Funds locked in the DAO are effectively unwithdrawable via this path on Rust nodes, and the chain forks.

---

### Likelihood Explanation

**Likelihood: Medium**

Any DAO depositor can construct a withdrawal transaction. The only requirement is setting the 8-byte `input_type` witness field to a value > 255 (e.g., `257`) and padding `header_deps` to at least 258 entries with the deposit block hash at position `index & 0xFF`. This requires no privileged access, no key leakage, and no majority hashpower — only knowledge of the DAO withdrawal protocol and the ability to submit a transaction. A miner willing to include the transaction (or the depositor who is also a miner) can trigger the split.

---

### Recommendation

In `DaoCalculator::transaction_maximum_withdraw`, mask the extracted index to its lowest byte before using it, matching the C VM's behavior:

```rust
// Before (line 91-96 of util/dao/src/lib.rs):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.get(header_dep_index as usize)

// After:
let raw_index = LittleEndian::read_u64(&header_deps_index_data.unwrap());
let header_dep_index = (raw_index & 0xFF) as usize;  // match C VM lowest-byte semantics
// ...
.get(header_dep_index)
```

Alternatively, add an explicit rejection of any index value that does not fit in a `u8`, so that such transactions are uniformly rejected by both Rust and the C VM before they can reach a miner.

---

### Proof of Concept

The existing test in `util/dao/src/tests.rs` directly demonstrates the divergence:

```
header_deps[1]   = deposit_block.hash()   // C VM resolves 257 & 0xFF = 1 → deposit block ✓
header_deps[257] = withdraw_block.hash()  // Rust resolves 257 → withdraw block ✗

witness input_type = 257u64 (little-endian bytes)
```

**C VM path**: resolves index `1` → deposit block (number 100) → matches cell data (100) → accepts withdrawal.

**Rust path** (`DaoCalculator`): resolves index `257` → withdraw block (number 200) → block-number check `200 != 100` → `DaoError::InvalidOutPoint` → `FeeCalculator::transaction_fee()` returns `Err` → `ContextualTransactionVerifier::verify()` propagates `?` → `BlockTxsVerifier::verify()` rejects the block.
