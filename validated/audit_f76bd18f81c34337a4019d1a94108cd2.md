Audit Report

## Title
Dead Cell Liveness Bypass in `CyclesEstimator` Enables CPU Exhaustion DoS via `estimate_cycles` RPC - (File: `rpc/src/module/chain.rs`)

## Summary

`CyclesEstimator` implements `CellProvider` by unconditionally promoting any cell found in the snapshot to `CellStatus::live_cell`, including spent (dead) cells. This allows `resolve_transaction` to succeed for transactions referencing dead inputs, after which `ScriptVerifier::verify` executes scripts up to the full `consensus.max_block_cycles` limit. An unprivileged caller can flood `estimate_cycles` with dead-cell transactions carrying maximally expensive scripts to exhaust CPU and degrade node responsiveness.

## Finding Description

`CyclesEstimator::cell` at `rpc/src/module/chain.rs` lines 2347–2361 calls `snapshot.get_cell(out_point)` and maps any result — live or dead — to `CellStatus::live_cell(cell_meta)`, with the explicit comment `// treat as live cell, regardless of live or dead`:

```rust
CellStatus::live_cell(cell_meta)
// treat as live cell, regardless of live or dead
```

`CyclesEstimator::run` at line 2378 passes `self` as the `CellProvider` to `resolve_transaction`. Because dead cells are returned as live, resolution succeeds for transactions whose inputs are entirely spent. The resolved transaction is then passed to `ScriptVerifier::new(...).verify(max_cycles)` at lines 2383–2389, where `max_cycles = consensus.max_block_cycles` — the consensus-wide maximum, with no per-call cap.

By contrast, `PoolCell` at `tx-pool/src/pool_cell.rs` lines 34–43 implements `CellChecker::is_live` returning `Some(false)` for spent cells, causing rejection before any script execution. `CyclesEstimator` implements no equivalent liveness gate. No rate limiting exists anywhere in `rpc/src/**` (confirmed: zero matches for rate-limiting patterns).

Exploit path:
1. `CyclesEstimator::cell` returns `CellStatus::live_cell` for a dead out-point.
2. `resolve_transaction` succeeds, producing a `ResolvedTransaction`.
3. `ScriptVerifier::verify(max_block_cycles)` executes all lock/type scripts up to the full block cycle budget.
4. No liveness gate, no per-call cycle cap, no rate limit prevents this.

## Impact Explanation

An attacker can sustain CPU exhaustion on a targeted node by flooding `estimate_cycles` with dead-cell transactions carrying maximally expensive scripts. Each call consumes up to `max_block_cycles` CPU cycles on the RPC handler thread. Saturating the RPC thread pool and CPU prevents legitimate RPC responses and degrades block processing and P2P message handling. This matches the High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** The attacker cost is negligible — only standard RPC calls and publicly observable chain data are required.

## Likelihood Explanation

- `estimate_cycles` is publicly documented and reachable by any client with RPC access; node operators commonly expose it for tooling.
- Dead out-points are trivially enumerable from chain history with no privileged knowledge.
- No authentication, rate limiting, or per-call cycle cap exists beyond the global `max_block_cycles` constant.
- The attack is fully repeatable and parallelizable from multiple connections.

## Recommendation

In `CyclesEstimator::cell`, check the cell's liveness status from the snapshot before returning `CellStatus::live_cell`. Use the snapshot's authoritative liveness check and return `CellStatus::Dead` for spent cells, causing `resolve_transaction` to fail before any script execution. Additionally, enforce a per-call cycle cap for `estimate_cycles` substantially lower than `max_block_cycles`, and apply rate limiting on the endpoint.

## Proof of Concept

1. Submit a valid transaction `T` to the chain and wait for confirmation. Its output at index 0 (`(T.hash, 0)`) is now a dead cell.
2. Deploy a lock script that loops until cycle exhaustion (up to `max_block_cycles`).
3. Construct transaction `T2` with `inputs[0].previous_output = (T.hash, 0)` and the exhausting lock script.
4. Call `estimate_cycles` with `T2` via JSON-RPC.
5. Observe the node executes scripts up to `max_block_cycles` cycles despite the dead input.
6. Repeat in a tight loop from multiple connections to sustain CPU exhaustion and degrade node responsiveness.
