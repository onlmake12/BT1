### Title
Publisher/Delegator Can Escape OIS Slashing by Undelegating and Withdrawing Before Per-Account Slash Execution — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The Oracle Integrity Staking (OIS) slashing mechanism is split into two separate on-chain steps: `create_slash_event` (creates a global slash record) and `slash` (applies the slash to each individual staker's account). There is no withdrawal lock placed on stake accounts between these two steps. A publisher or delegator who observes a `SlashEvent` being created can call `undelegate` to close their OIS position and subsequently call `withdrawStake` to remove tokens from custody before the `slash` instruction is ever executed against their account, thereby escaping the penalty entirely.

---

### Finding Description

**Step 1 — Slash event creation is global, not per-account.**

The `create_slash_event` instruction records a `SlashEvent` PDA keyed by `(publisher, index)` containing `epoch`, `slash_ratio`, and `slash_custody`. [1](#0-0) 

This instruction is called once by the `reward_program_authority` and does not touch any individual staker's account.

**Step 2 — Slash application is lazy and per-account.**

The `slash` instruction must be called separately for every staker's `stake_account_positions`. It reads the staker's `delegation_record` (which tracks `next_slash_event_index`) and calls `slash_account` in the staking program via CPI, which reduces the `stake_account_custody` balance. [2](#0-1) [3](#0-2) 

**Step 3 — No withdrawal lock exists.**

The `DelegationRecord` struct contains only `last_epoch` and `next_slash_event_index`. There is no `is_locked`, `pending_slash`, or equivalent flag that would block `undelegate` or `withdrawStake` when a `SlashEvent` is outstanding. [4](#0-3) 

The `withdrawStake` instruction in the staking program has no account constraint referencing the integrity pool or any pending slash state. [5](#0-4) 

**Step 4 — `undelegate` and `withdrawStake` are independently callable.**

The client SDK shows `unstakeFromPublisher` calls `undelegate` directly without first calling `advance_delegation_record` (which would process pending slashes). There is no enforced ordering. [6](#0-5) 

**Step 5 — Multi-epoch investigation window creates the escape opportunity.**

The slashing rulebook and FAQ confirm the Pythian Council has until the end of epoch T+1 to create the slash event for an incident in epoch T. The cooldown period for OIS is 1–2 epochs. This means a staker who undelegates immediately after a misprint in epoch T can complete cooldown and withdraw before the `slash` instruction is ever called on their account. [7](#0-6) 

---

### Impact Explanation

A publisher or delegator responsible for a data misprint can front-run the slash by:
1. Observing the `create_slash_event` transaction on-chain.
2. Calling `undelegate` to close their OIS position.
3. Waiting out the cooldown (1–2 epochs).
4. Calling `withdrawStake` to remove tokens from `stake_account_custody`.

Once tokens leave the custody account, the `slash_account` CPI has nothing to slash. The slashing mechanism — the primary economic deterrent for data quality — is rendered ineffective for any staker who monitors the chain. This directly undermines the security guarantee of OIS. [8](#0-7) 

---

### Likelihood Explanation

- The attack requires no privileged access — any staker can call `undelegate` and `withdrawStake` permissionlessly.
- `create_slash_event` is a public on-chain transaction, immediately visible to all participants.
- The multi-epoch investigation window (up to 14 days) is longer than the cooldown period, giving stakers ample time to exit.
- Publishers who publish bad data are the most motivated actors and are also the most likely to be monitoring their own pool's slash events. [9](#0-8) 

---

### Recommendation

1. **Add a withdrawal lock flag to `DelegationRecord`**: When `create_slash_event` is called for a publisher, mark all `delegation_record` accounts associated with that publisher as locked, preventing `undelegate` and `withdrawStake` until all pending slash events (up to `pool_config.num_slash_events`) have been applied via `slash` or `advance_delegation_record`.

2. **Alternatively, enforce `advance_delegation_record` as a prerequisite for `undelegate`**: Require that `delegation_record.next_slash_event_index` equals the current total slash event count before allowing undelegation, ensuring all pending slashes are settled first.

3. **Apply slashes to historical epoch balances**: Store the staked amount at the slash epoch inside `SlashEvent` or `DelegationRecord` so that the slash amount is computed from the balance at epoch T, not the current balance, making withdrawal-before-slash ineffective.

---

### Proof of Concept

```
Epoch T:   Publisher submits bad price data. Misprint confirmed.
Epoch T+1: Pythian Council calls create_slash_event(publisher, index=0, slash_ratio=500).
           SlashEvent PDA is created on-chain — publicly visible.

           Staker observes the SlashEvent transaction.
           Staker calls undelegate(index, amount) → closes OIS position.
           Staker's tokens enter PREUNLOCKING (cooldown) state.

Epoch T+2: Cooldown ends. Staker calls withdrawStake(amount).
           Tokens leave stake_account_custody.

           Pythian Council (or anyone) calls slash(index=0) on staker's account.
           slash_account CPI executes: custody balance = 0 → slash amount = 0.
           Staker has escaped the full slash.
``` [10](#0-9) [11](#0-10)

  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }
