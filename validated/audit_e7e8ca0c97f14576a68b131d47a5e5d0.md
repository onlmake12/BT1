### Title
`break` Instead of `continue` in Input Loop Causes Indexed Cells to Remain Marked as Live After Being Spent — (`util/rich-indexer/src/indexer/mod.rs`)

### Summary
In `insert_transaction`, when iterating over a transaction's inputs, `spend_cell` returns `false` if the referenced output is not present in the indexer. The loop responds with `break`, terminating input processing entirely. Any inputs after the first unindexed input are never processed, so their corresponding indexed outputs are never marked `is_spent = 1`. Those cells remain visible as live/unspent in all subsequent `get_cells` queries.

### Finding Description

In `mod.rs`, `insert_transaction` iterates over inputs: [1](#0-0) 

`spend_cell` issues an `UPDATE output SET is_spent = 1 WHERE ...` and returns `true` only if a row was actually updated (i.e., the cell exists in the indexer): [2](#0-1) 

Returning `false` is the normal, expected outcome for any input whose originating output was filtered out by the cell filter or predates the indexer's start height. It is not an error condition — it simply means "this cell is not tracked." The correct response is `continue` (skip to the next input). Instead, `break` exits the loop entirely, silently skipping all remaining inputs.

**Concrete scenario:**
- Transaction T has inputs `[I0, I1, I2]`
- I0's output is not in the indexer (filtered or pre-indexer)
- I1 and I2's outputs **are** in the indexer
- On `append_block`: `spend_cell(I0)` → `false` → `break`; `spend_cell(I1)` and `spend_cell(I2)` are never called; I1 and I2 remain `is_spent = 0`

On `rollback_block`, `reset_spent_cells` resets `is_spent` to `0` for all outputs referenced in the `input` table for the rolled-back transactions: [3](#0-2) 

Because I1 and I2 were never inserted into the `input` table (the loop broke before `build_input_rows` was called for them), rollback also has no effect on them — they were never marked spent, so they stay `is_spent = 0` permanently, even after the block is confirmed.

### Impact Explanation
Any query against the `output` table filtering on `is_spent = 0` (i.e., `get_cells`, live-cell queries) will return I1 and I2 as unspent cells even though they were consumed by a committed, consensus-valid transaction. Wallets and applications relying on the rich indexer will compute inflated balances and may attempt to build transactions spending already-consumed cells.

### Likelihood Explanation
The cell filter is a standard production feature. Any transaction that spends at least one cell not tracked by the indexer (due to filter mismatch or pre-indexer origin) followed by one or more tracked cells triggers this bug. No privileged access is required — a normal block relay carrying such a transaction is sufficient.

### Recommendation
Replace `break` with `continue` on line 232 of `mod.rs`:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;  // cell not in indexer; skip, do not abort the loop
}
``` [4](#0-3) 

### Proof of Concept
1. Enable cell filter (e.g., filter by a specific lock script).
2. Create cell A (matches filter, will be indexed) and cell B (does not match filter, not indexed).
3. Build transaction T with inputs `[B, A]` (unindexed first, indexed second).
4. Append a block containing T.
5. Query `get_cells` for cell A's lock script — it is returned as live (`is_spent = 0`).
6. Expected: cell A should be `is_spent = 1`.

pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
}

async fn reset_spent_cells(tx_id_list: &[i64], tx: &mut Transaction<'_, Any>) -> Result<(), Error> {
    let query = SqlBuilder::update_table("output")
        .set("is_spent", 0)
        .and_where_in_query(
            "id",
            SqlBuilder::select_from("input")
                .field("output_id")
                .and_where_in("consumed_tx_id", tx_id_list)
                .query()
                .map_err(|err| Error::DB(err.to_string()))?,
        )
        .sql()
        .map_err(|err| Error::DB(err.to_string()))?;

    sqlx::query(&query)
        .execute(tx.as_mut())
        .await
        .map_err(|err| Error::DB(err.to_string()))?;

    Ok(())
}
