Audit Report

## Title
Verify Queue Flooding via Structurally Valid Transactions with Fake Inputs Enables Persistent DoS - (File: `tx-pool/src/component/verify_queue.rs`, `tx-pool/src/process.rs`)

## Summary

The `VerifyQueue` in CKB's tx-pool admits transactions after only a structural (`non_contextual_verify`) check, deferring fee validity and input existence checks to a background worker. An unprivileged remote peer can flood the 256 MB queue with structurally valid transactions referencing non-existent inputs at zero on-chain cost, causing legitimate transactions to be rejected with `Reject::Full` and wasting node CPU. The attacker is never banned because missing-input rejections do not trigger `ban_malformed`.

## Finding Description

**Admission gate — only structural checks:**

`resumeble_process_tx` in `tx-pool/src/process.rs` (L335–353) calls `non_contextual_verify` then immediately calls `enqueue_verify_queue`. The `non_contextual_verify` function (via `NonContextualTransactionVerifier`) checks only: version, size, non-empty inputs/outputs, duplicate deps, outputs-data match, and script hash type. It performs no fee check and no input existence check.

**Queue size limit — only total byte count:**

`VerifyQueue::add_tx` in `tx-pool/src/component/verify_queue.rs` (L215–219) rejects only when `total_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB). There is no per-peer slot limit, no minimum fee rate, and no input plausibility check.

**Fee and input checks deferred to worker:**

`_process_tx` (L705–717) calls `pre_check` — which performs snapshot reads, cell resolution (`resolve_tx`), and fee validation — only after the transaction has already been dequeued by the background worker. By this point the attacker's transaction has already occupied queue space.

**No ban for missing-input rejections:**

In `after_process` (L507–512), when `is_missing_input(reject)` is true, the transaction is routed to the orphan pool and `ban_malformed` is never called. `ban_malformed` (L679–703) — the only path that calls `remove_txs_by_peer` — is only triggered for `reject.is_malformed_tx()`, which structurally valid transactions with fake inputs never produce.

**Exploit flow:**
1. Attacker connects as a peer and crafts transactions with random non-existent `OutPoint`s. These pass `non_contextual_verify`.
2. Attacker relays them via `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue`. Each is admitted until `total_tx_size` reaches 256 MB.
3. Legitimate transactions receive `Reject::Full`.
4. Workers drain attacker transactions in FIFO (`added_time`) order, each triggering a full `pre_check` cycle before rejection.
5. Rejected attacker transactions are routed to the orphan pool, compounding resource pressure.
6. After draining, the attacker immediately re-floods. No cooldown, no ban.

## Impact Explanation

This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

A single attacker peer can continuously prevent legitimate transactions from entering the verify queue, effectively making the tx-pool unavailable for normal users. The attack is sustained and repeatable: after each drain cycle the attacker re-floods with a fresh batch. CPU is wasted on `pre_check` for every attacker transaction. The orphan pool is also pressured as a secondary effect.

## Likelihood Explanation

- **Attacker capability**: Any unprivileged remote peer reachable via the standard relay protocol. No key material, no on-chain funds, no majority hashpower required.
- **Cost**: Only network bandwidth. Crafting structurally valid transactions with fake inputs is trivial.
- **Repeatability**: Unlimited. After the worker drains the queue, the attacker re-floods immediately. There is no cooldown, no per-peer admission throttle, and no ban triggered by missing-input rejections.
- **Detection/mitigation gap**: `remove_txs_by_peer` exists but is only reachable via `ban_malformed`, which is never triggered for this attack class.

## Recommendation

1. **Enforce a minimum fee rate at queue admission**: Before `enqueue_verify_queue`, reject transactions whose declared output capacity minus declared input capacity (for locally resolvable inputs) falls below a configurable minimum byte-fee floor.
2. **Per-peer queue byte limit**: Track per-peer bytes in `VerifyQueue` and reject new submissions from a peer once their share exceeds a threshold (e.g., `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE / max_peers`).
3. **Priority eviction**: When the queue is full and a higher-fee-rate transaction arrives, evict the lowest-fee-rate entry rather than rejecting the new arrival outright.
4. **Ban or throttle on repeated missing-input rejections**: Track per-peer missing-input rejection counts and apply a ban or rate limit after a threshold is exceeded.

## Proof of Concept

**Manual steps:**

1. Connect to a CKB node as a peer via the relay protocol.
2. Craft transactions each with one input referencing a randomly generated `OutPoint` (tx_hash: random 32 bytes, index: 0), one output with valid capacity, and a valid lock script structure. These pass `NonContextualTransactionVerifier`.
3. Relay transactions via `submit_remote_tx` in a tight loop. Each reaches `resumeble_process_tx` → `non_contextual_verify` (passes) → `enqueue_verify_queue` (admitted, `total_tx_size` incremented).
4. After ~1.28 million 200-byte transactions, `total_tx_size` reaches 256 MB.
5. Submit a legitimate transaction from a separate peer or RPC. Observe `Reject::Full`.
6. Monitor logs: each attacker transaction triggers `pre_check` in the worker, fails at `resolve_tx` with `OutPointError::Unknown`, is routed to `add_orphan`, and the attacker peer is never banned.
7. After the queue drains, repeat from step 3 to confirm indefinite repeatability.

**Key invariant to test (fuzz/integration):** After any sequence of remote `submit_remote_tx` calls from a single peer containing only transactions with non-existent inputs, a subsequent `submit_remote_tx` from a different peer with a valid transaction must not return `Reject::Full`.
