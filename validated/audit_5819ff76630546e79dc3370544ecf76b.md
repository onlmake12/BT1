### Title
`check_tx_fee` Enforces Minimum Fee Against Byte Size Only, Not Cycle-Adjusted Weight — (`tx-pool/src/util.rs`)

### Summary

The tx-pool admission gate (`check_tx_fee`) computes the minimum required fee using only the transaction's serialized byte size, while the actual fee rate used for pool sorting and eviction is computed against `get_transaction_weight(size, cycles)` — the maximum of byte size and cycle-adjusted size. A tx-pool submitter can craft a cycle-heavy, byte-small transaction that passes the admission check while paying a fee rate orders of magnitude below the configured `min_fee_rate`, causing the node to perform expensive script verification without adequate compensation.

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum fee as:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

This uses `tx_size` (raw serialized bytes) as the weight denominator. However, the actual fee rate for every transaction in the pool is computed via `get_transaction_weight`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES ≈ 0.000_170_571_4`: [3](#0-2) 

The consensus constants that define this ratio are:

```
MAX_BLOCK_BYTES  = TWO_IN_TWO_OUT_BYTES  * 1_000 = 597_000
MAX_BLOCK_CYCLES = TWO_IN_TWO_OUT_CYCLES * 1_000 = 3_500_000_000
``` [4](#0-3) 

The `check_tx_fee` call happens inside `pre_check`, **before** `verify_rtx` executes the scripts and determines actual cycles: [5](#0-4) 

The `TxEntry::fee_rate()` method — used for pool sorting and eviction — correctly uses `get_transaction_weight`: [6](#0-5) 

The same `min_fee_rate` constant is thus applied against two different denominators:
- **Admission gate**: `min_fee_rate * tx_size / 1000` (bytes only)
- **Actual fee rate**: `fee / get_transaction_weight(tx_size, cycles) * 1000` (max of bytes and cycle-adjusted bytes)

### Impact Explanation

Consider a transaction with:
- `tx_size` = 500 bytes (small)
- `cycles` = 70,000,000 (the configured `max_tx_verify_cycles`)
- `min_fee_rate` = 1,000 shannons/KB (mainnet default)

**Admission check** (`check_tx_fee`):
- `min_fee = 1000 * 500 / 1000 = 500 shannons` → attacker pays 500 shannons, passes

**Actual weight** (`get_transaction_weight`):
- `cycle_weight = 70_000_000 * 0.000_170_571_4 ≈ 11,940`
- `weight = max(500, 11_940) = 11,940`

**Effective fee rate**:
- `500 / 11,940 * 1000 ≈ 41.9 shannons/KB`

The attacker pays a fee rate of ~42 shannons/KB while the minimum is 1,000 shannons/KB — a **~24× undercharge**. The node must run full script verification (up to 70M cycles of CKB-VM execution) before discovering the transaction's true fee rate. The transaction is then admitted to the pool at the bottom of the fee-rate ordering and eventually evicted, but the verification CPU cost has already been paid by the node.

An attacker can repeatedly submit such transactions to exhaust node CPU resources at minimal cost, constituting a resource-exhaustion DoS against the tx-pool verification pipeline.

### Likelihood Explanation

Any unprivileged RPC caller with access to `send_transaction` can trigger this. Crafting a transaction with high cycle consumption but small serialized size is straightforward — a script that runs a tight loop or performs many hash operations will consume cycles without adding bytes. The attacker only needs to pay the byte-size-based fee (a few hundred shannons), making the attack economically viable. The `max_tx_verify_cycles` limit (70M cycles) bounds the per-transaction cost to the node but does not prevent repeated submissions. [7](#0-6) 

### Recommendation

The pre-check in `check_tx_fee` should incorporate a declared-cycles value from the transaction submitter (as CKB already supports via the `cycles` field in `send_transaction`), or the minimum fee should be computed against a conservative upper-bound weight that accounts for cycles. Specifically:

1. If the caller declares cycles (already supported in the RPC), use `get_transaction_weight(tx_size, declared_cycles)` in `check_tx_fee` instead of `tx_size` alone.
2. If no cycles are declared, apply a conservative cycle-based weight floor (e.g., `max_tx_verify_cycles * DEFAULT_BYTES_PER_CYCLES`) to ensure the fee covers worst-case verification cost.

### Proof of Concept

A tx-pool submitter calls `send_transaction` with a transaction that:
- Has a lock script executing a tight loop consuming ~70M cycles
- Has a serialized size of ~500 bytes
- Pays a fee of 500 shannons (passes `check_tx_fee`: `1000 * 500 / 1000 = 500`)

The node runs `verify_rtx` → `ContextualTransactionVerifier::verify(max_tx_verify_cycles=70_000_000)`, spending ~70M CKB-VM cycles. The transaction is admitted with an effective fee rate of ~42 shannons/KB (well below the 1,000 shannons/KB minimum). Repeating this in a loop exhausts the node's verification worker threads at a cost of ~500 shannons per 70M cycles of node CPU. [8](#0-7) [2](#0-1)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
        (ret, snapshot)
