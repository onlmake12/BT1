### Title
Osmosis Multi-Denom Fee Accepts Depegged Tokens at Face Value Without Market Price Adjustment — (`target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

---

### Summary

The Osmosis-specific `get_update_fee_for_denom` function in the CosmWasm Pyth contract returns the same nominal fee amount regardless of which allowed token the caller chooses to pay with. There is no spot-price conversion applied. A user paying with a depegged or low-value allowed token pays the same number of token units as a user paying with the base denom, but at a fraction of the real economic value.

---

### Finding Description

`get_update_fee_for_denom` is compiled only under `#[cfg(feature = "osmosis")]` and is the entry point for callers who want to pay the price-update fee in any token accepted by Osmosis's `txFee` module.

    // base amount is multiplied to number of vaas to get the total amount

    // this will be change later on to add custom logic using spot price for valid tokens

`get_update_fee_amount` computes `config.fee.amount * num_vaas` — a fixed amount denominated in the base denom's smallest unit. [2](#0-1) 

When a caller passes a non-base denom, the function substitutes that denom into the same numeric amount with **no spot-price conversion**. The developer comment on line 734 explicitly acknowledges the missing logic: *"this will be change later on to add custom logic using spot price for valid tokens."* [3](#0-2) 

The base fee is also queryable via `get_update_fee`: [4](#0-3)

### Impact Explanation

If the base fee is `N uosmo` per VAA and OSMO trades at $0.50, the intended fee is `N × $0.0000005`. A caller who pays with an Osmosis-allowed token that has depegged to $0.001 per unit pays `N` units of that token = `N × $0.000001` — potentially orders of magnitude less real value than the base-denom fee. The protocol accrues fees in a token worth far less than intended, and the economic cost of spamming price updates is reduced for an attacker holding depegged tokens.

---

### Likelihood Explanation

- Osmosis's `txFee` module maintains a whitelist of accepted fee tokens; any token on that list is eligible.
- Stablecoin depeg events (e.g., UST, USDN) have occurred on Cosmos/Osmosis chains historically.
- The entry path is fully unprivileged: any transaction sender calling `UpdatePriceFeed` on the Osmosis deployment can choose the fee denom.
- The developer comment confirms the gap is known and unresolved.

---

### Recommendation

Apply a spot-price conversion inside `get_update_fee_for_denom` before returning the fee coin. Specifically:

1. Query the Osmosis `SpotPrice` or `TWAP` for `(denom, base_denom)`.
2. Scale `get_update_fee_amount` by `price(base_denom) / price(denom)` so the caller always pays the equivalent real-value amount regardless of which token they choose.
3. Until the spot-price logic is implemented, restrict `get_update_fee_for_denom` to return fees only in the base denom, removing the multi-denom path.

---

### Proof of Concept

**Setup (Osmosis mainnet):**
- Base fee: `1 uosmo` per VAA; OSMO = $0.50 → fee ≈ $0.0000005 per VAA.
- Osmosis txFee whitelist includes `ibc/USDC` (normally $1.00) and hypothetically a depegged token `ibc/DEPEGGED` now worth $0.001.

**Attack:**
1. Attacker holds `ibc/DEPEGGED` tokens.
2. Attacker calls `UpdatePriceFeed` with 100 VAAs, requesting fee quote via `get_update_fee_for_denom(vaas, "ibc/DEPEGGED")`.
3. Contract returns `coin(100, "ibc/DEPEGGED")` — 100 units of the depegged token.
4. Attacker pays 100 × $0.001 = $0.0001 instead of the intended 100 × $0.0000005 = $0.00005 in OSMO terms.

In the inverse scenario where the depegged token is worth far less than the base denom, the attacker pays a tiny fraction of the intended fee, making high-frequency price-update spam economically viable and draining the fee-revenue stream of real value.
