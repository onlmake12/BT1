Audit Report

## Title
O(FILTER_SIZE) Linear Scan Under Shared Mutex on Every RelayTransactionHashes Message — (`sync/src/relayer/transaction_hashes_process.rs`)

## Summary
`TransactionHashesProcess::execute()` acquires the shared `tx_filter` mutex and unconditionally calls `TtlFilter::remove_expired()` on every incoming `RelayTransactionHashes` P2P message. `remove_expired()` iterates all entries of a 50,000-capacity `LruCache` with no cooldown or early-exit. The per-peer rate limiter (30 msg/s × 128 peers) allows up to 3,840 full-filter scans per second, all serializing on the shared mutex and degrading relay throughput.

## Finding Description
`TtlFilter::remove_expired()` at `sync/src/types/mod.rs` lines 359–377 iterates the entire `inner: LruCache<T, u64>` unconditionally: it collects all expired keys into a `Vec` via a full `.iter()` pass, then calls `self.remove()` for each. There is no timestamp-ordered side-index, no cooldown guard, and no early-exit — the scan is always O(FILTER_SIZE) regardless of filter occupancy or time elapsed since the last expiry run.

`TransactionHashesProcess::execute()` at `sync/src/relayer/transaction_hashes_process.rs` lines 38–47 acquires the `tx_filter` mutex (`state.tx_filter()`), immediately calls `tx_filter.remove_expired()`, and holds the mutex for the entire O(50,000) scan before filtering the incoming hashes. The identical pattern appears in the `TxVerificationResult::UnknownParents` branch of `send_bulk_of_tx_hashes()` at `sync/src/relayer/mod.rs` lines 677–684.

The rate limiter at `sync/src/relayer/mod.rs` lines 89–92 and 116–123 is keyed by `(PeerIndex, message.item_id())` — per-peer, not global. With `MAX_RELAY_PEERS = 128` (line 59) and a quota of 30 msg/s per peer, the aggregate ceiling is 3,840 `RelayTransactionHashes` messages/second, each triggering a full O(50,000) LruCache iteration under the shared mutex. The `lru` crate's `LruCache` is backed by a doubly-linked list, making iteration pointer-chasing across ~1.6 MB of `Byte32` data — cache-unfriendly and slow.

The `MAX_RELAY_TXS_NUM_PER_BATCH` guard at lines 29–35 only limits hash count per message, not the frequency of `remove_expired()` invocations, and does not mitigate the issue.

## Impact Explanation
At 3,840 forced scans/second, if each O(50,000) LruCache iteration takes 100–500 µs (realistic for pointer-chasing through a linked list), the shared mutex is held for 384–1,920 ms per second of wall time. This serializes all concurrent relay message processing threads on the shared mutex, causing increased latency for transaction propagation and degraded relay throughput across the node. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, as the attacker cost is simply connecting as a peer and sending messages at the allowed rate.

## Likelihood Explanation
Any unprivileged peer can send `RelayTransactionHashes` messages up to the per-peer rate limit (30/s) with no PoW, no key, and no special role. The `tx_filter` fills to capacity through normal network operation (legitimate tx hash announcements over the TTL window), so the precondition of a near-full filter is met organically. The attack is trivially reproducible with a modified CKB client or a raw P2P message sender. Multiple attacker-controlled peers compound the effect linearly up to the 128-peer ceiling.

## Recommendation
1. **Decouple expiry from message handling**: Move `remove_expired()` to a periodic background timer (e.g., the existing `TX_HASHES_TOKEN` notify at 300 ms intervals in `sync/src/relayer/mod.rs`) rather than calling it on every incoming message.
2. **Add a cooldown guard**: Track the last expiry timestamp inside `TtlFilter` and skip `remove_expired()` if called within a minimum interval (e.g., 60 seconds).
3. **Use a time-ordered structure**: Replace the full-scan approach with a `BTreeMap<expiry_time, key>` side-index so expiry is O(expired_count) not O(FILTER_SIZE).
4. **Add a global rate limit** on `remove_expired()` invocations per second, independent of per-peer limits.
