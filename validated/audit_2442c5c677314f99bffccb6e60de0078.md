Audit Report

## Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Returns Wrong Withdrawal Capacity — (`util/dao/src/lib.rs`)

## Summary
`DaoCalculator::calculate_maximum_withdraw` computes `withdraw_counted_capacity` as a u128 and narrows it to u64 via a silent `as u64` cast at line 156. When the u128 value exceeds `u64::MAX`, Rust wraps it modulo 2^64 with no error, and the function returns `Ok(truncated_capacity)` — a value far below what the depositor is owed. Every other u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, making this the sole silent truncation path.

## Finding Description
In `util/dao/src/lib.rs` lines 152–156:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
```

The `as u64` cast silently truncates to `value % 2^64` when `withdraw_counted_capacity > u64::MAX`. The function then returns `Ok(truncated_capacity)` with no error signal. The two other u128→u64 narrowings in the same file (`miner_issuance128` at line 245 and `ar_increase128` at line 258) both use `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, establishing the correct pattern that this site deviates from.

The overflow condition is `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. `ar` is a u64 stored in the DAO field, initialized at genesis to `10_000_000_000_000_000` and growing monotonically with every block's secondary issuance. The existing test `check_withdraw_calculation_overflows` only exercises the case where the truncated value is large enough that the subsequent `safe_add(occupied_capacity)` overflows — it does not cover the case where truncation produces a small value that passes `safe_add`, causing the function to silently return an incorrect result.

## Impact Explanation
This matches **Critical: Vulnerabilities which could easily damage CKB economy**. NervosDAO is a core protocol incentive mechanism. When truncation produces a small value that passes `safe_add`, the node enforces a silently-wrong maximum withdrawal ceiling. A depositor who constructs a withdrawal transaction claiming the correct (larger) amount has it rejected by block validation. A depositor who constructs a transaction claiming only the truncated amount has it accepted, permanently burning the difference — the interest is irrecoverably lost with no on-chain recovery mechanism.

## Likelihood Explanation
The overflow threshold requires `counted_capacity × ar_ratio > u64::MAX ≈ 1.84 × 10^19`. With `ar` starting at `10^16` and growing monotonically, a cell holding the full genesis allocation (~3.36 × 10^18 shannons) overflows when `ar` reaches ~5.5× its genesis value. No special privilege is required — any DAO depositor triggers this path by submitting a withdrawal transaction. The condition is not immediately reachable on mainnet today, but the protocol is designed to run indefinitely and `ar` grows with every block, making this threshold reachable on a long-running chain.

## Recommendation
Replace the silent cast with a checked conversion, consistent with the rest of the file:

```rust
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    ).safe_add(occupied_capacity)?;
```

This returns `DaoError::Overflow` instead of silently truncating, matching the behavior of the `miner_issuance128` and `ar_increase128` conversions in the same file.

## Proof of Concept
Add the following unit test to `util/dao/src/tests.rs`. It uses parameters where truncation produces a small value that passes `safe_add`, so the current code returns `Ok(wrong_value)` instead of `Err(Overflow)`:

```
deposit_ar       = 10_000_000_000_000_000
withdrawing_ar   = 55_000_000_000_000_000
counted_capacity = 3_360_000_000_000_000_000

withdraw_counted_capacity (u128) =
    3_360_000_000_000_000_000 × 55_000_000_000_000_000
    / 10_000_000_000_000_000
  = 18_480_000_000_000_000_000   (> u64::MAX = 18_446_744_073_709_551_615)

withdraw_counted_capacity as u64 =
    18_480_000_000_000_000_000 % 2^64
  = 33_255_926_290_448_384       (small, passes safe_add)
```

A test asserting `result.is_err()` will fail with the current code (it returns `Ok`) and pass after the fix. The existing `check_withdraw_calculation_overflows` test does not cover this case because it only exercises the `safe_add` overflow path, not the silent-truncation-then-success path.

fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),

    assert!(result.is_err());
}
