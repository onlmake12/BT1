### Title
`undelegate` Instruction Does Not Advance `delegation_record` Checkpoint Before Closing Position — (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `undelegate` instruction in the OIS integrity pool program closes a delegator's staking position without first advancing (or requiring advancement of) the `delegation_record` checkpoint. This is the direct analog of the reported `withdraw`-not-triggering-`SharePriceCheckpoint` pattern. Any staking user who calls `undelegate` without first calling `advance_delegation_record` permanently loses all unclaimed OIS rewards accrued since their last checkpoint epoch.

---

### Finding Description

The OIS integrity pool uses a per-delegator-per-publisher `delegation_record` account as its reward checkpoint. The `advance_delegation_record` instruction iterates over epochs from `delegation_record.last_epoch` up to the current epoch, computes rewards, transfers them to the user's custody account, and updates `last_epoch` to the current epoch.

Comparing the account lists of the three key instructions in `governance/pyth_staking_sdk/src/idl/integrity-pool.json`:

| Instruction | `delegation_record` present? | Writable? |
|---|---|---|
| `advance_delegation_record` | ✅ Yes | ✅ Yes (updated) |
| `delegate` | ✅ Yes | ❌ No (read-only check) |
| `undelegate` | ❌ **Absent** | — |

`advance_delegation_record` (lines 210–346) includes `delegation_record` as a **writable** PDA and is the only instruction that advances `last_epoch` and pays out rewards.

`delegate` (lines 608–716) includes `delegation_record` as a **read-only** account — the program reads it to enforce the `outdatedDelegatorAccounting` (error 6008) guard before allowing new delegation.

`undelegate` (lines 961–1081) includes **no `delegation_record` account at all**. It modifies `pool_data`, `stake_account_positions`, and `stake_account_metadata` to remove the position, but the checkpoint is never consulted or updated.

Because `undelegate` does not include `delegation_record`, the on-chain program cannot:
1. Check whether the delegator's accounting is stale (the `outdatedDelegatorAccounting` guard that exists for `delegate` is absent here).
2. Advance the checkpoint to the current epoch before closing the position.

After `undelegate` executes, the position no longer exists in `stake_account_positions`. A subsequent call to `advance_delegation_record` for the same publisher/delegator pair will find no active positions and compute zero rewards for all epochs between `delegation_record.last_epoch` and the current epoch. Those rewards are permanently unclaimable.

The client-side SDK (`governance/pyth_staking_sdk/src/pyth-staking-client.ts`, `unstakeFromPublisher`, lines 346–401) calls `undelegate` directly without prepending `advance_delegation_record` instructions, confirming the missing checkpoint advancement is not compensated at the SDK layer either.

---

### Impact Explanation

Any OIS delegator who calls `undelegate` (directly on-chain or via the SDK) without first calling `advance_delegation_record` loses all rewards accrued since their last checkpoint epoch. Rewards are denominated in PYTH tokens transferred from `pool_reward_custody`. The loss is proportional to the staked amount, the yield rate `y`, and the number of epochs elapsed since the last `advance_delegation_record` call. This constitutes a direct, permanent loss of user funds with no recovery path.

---

### Likelihood Explanation

The vulnerability is reachable by any unprivileged staking user. The on-chain program imposes no ordering constraint between `advance_delegation_record` and `undelegate`. A user who unstakes after multiple epochs without claiming rewards (a common pattern) will silently lose those rewards. The SDK's `unstakeFromPublisher` and `unstakeFromAllPublishers` do not prepend `advance_delegation_record`, so the default user flow triggers the loss automatically.

---

### Recommendation

The `undelegate` instruction should include `delegation_record` as a writable account and invoke the same reward-advancement logic used by `advance_delegation_record` before closing the position, or alternatively enforce that `delegation_record.last_epoch == current_epoch` (returning `outdatedDelegatorAccounting` if not) so the user is forced to advance first. The SDK methods `unstakeFromPublisher` and `unstakeFromAllPublishers` should prepend `advance_delegation_record` instructions for each affected publisher before the `undelegate` instructions.

---

### Proof of Concept

1. Delegator stakes to publisher P at epoch E via `delegate`.
2. Epochs E, E+1, E+2 pass; rewards accumulate in `pool_data` but `delegation_record.last_epoch` remains at E (no `advance_delegation_record` called).
3. Delegator calls `undelegate` at epoch E+3 directly (or via SDK `unstakeFromPublisher`).
4. `undelegate` succeeds: position is removed from `stake_account_positions`, `pool_data` delegation totals are decremented. `delegation_record.last_epoch` is still E.
5. Delegator calls `advance_delegation_record` for publisher P. The instruction finds no active positions for this delegator/publisher pair in `stake_account_positions` and computes zero rewards for epochs E through E+3.
6. Rewards for 3 epochs are permanently lost.

  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }
