### Title
Wrong Variable in `MAX_DEPOSIT_LIMIT` Check Allows Unlimited Accumulation in Permanent Subscriptions ŌĆö (`Scheduler.sol`)

---

### Summary

The `addFunds` function in `Scheduler.sol` checks `msg.value > MAX_DEPOSIT_LIMIT` for permanent subscriptions, but this compares only the **individual deposit amount** against the limit rather than the **resulting total balance** (`status.balanceInWei + msg.value`). This mirrors the exact vulnerability class from the reference report: a guard uses the wrong variable, making the limit check semantically meaningless.

---

### Finding Description

In `Scheduler.sol`, permanent subscriptions are subject to a deposit cap (`MAX_DEPOSIT_LIMIT`) because they cannot be updated or have funds withdrawn ŌĆö any ETH deposited is locked forever. The cap is enforced in two places:

**`createSubscription` (line 48):**
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**`addFunds` (line 613):**
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Both checks validate only `msg.value` ŌĆö the size of the **current transaction's deposit** ŌĆö not the **post-deposit balance** (`status.balanceInWei + msg.value`). Because `addFunds` is callable by anyone (no access control beyond the subscription existing), a manager of a permanent subscription can call `addFunds` repeatedly with amounts just at or below `MAX_DEPOSIT_LIMIT`. Each individual call passes the check, but the cumulative balance grows without bound, far exceeding the intended cap. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Permanent subscriptions explicitly prohibit withdrawals and updates:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [3](#0-2) 

Any ETH deposited into a permanent subscription is **irrecoverably locked** in the contract. The `MAX_DEPOSIT_LIMIT` exists precisely to bound this risk. Because the check uses `msg.value` instead of the post-deposit balance, the limit is entirely bypassable: a manager can lock an arbitrary amount of ETH by splitting deposits into chunks Ōēż `MAX_DEPOSIT_LIMIT`. This results in permanent, irreversible loss of user funds beyond the intended cap.

---

### Likelihood Explanation

The entry path requires only that the caller is the manager of a permanent subscription ŌĆö a role they created themselves. No privileged access, leaked key, or governance majority is needed. The `addFunds` function has no access control:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
``` [4](#0-3) 

Any user who creates a permanent subscription can immediately exploit this. The exploit requires only multiple sequential transactions, each with `msg.value <= MAX_DEPOSIT_LIMIT`.

---

### Recommendation

Replace the `msg.value`-only check in `addFunds` with a check on the **post-deposit balance**:

```solidity
// Before adding funds to storage:
if (params.isPermanent && (status.balanceInWei + msg.value) > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

Apply the same fix to `createSubscription` for consistency (though `createSubscription` is a single call and the current check there is correct for the initial deposit).

---

### Proof of Concept

1. Manager creates a permanent subscription with `msg.value = MAX_DEPOSIT_LIMIT` ŌĆö passes the check in `createSubscription`.
2. Manager calls `addFunds{value: MAX_DEPOSIT_LIMIT}(subscriptionId)` ŌĆö `msg.value == MAX_DEPOSIT_LIMIT`, so `msg.value > MAX_DEPOSIT_LIMIT` is **false**, check passes.
3. `status.balanceInWei` is now `2 * MAX_DEPOSIT_LIMIT`.
4. Repeat step 2 N times. Total locked balance = `(N+1) * MAX_DEPOSIT_LIMIT`, all permanently irrecoverable.

The existing test `testPermanentSubscriptionDepositLimit` only validates that a **single** deposit exceeding `MAX_DEPOSIT_LIMIT` reverts; it never tests the cumulative case, confirming the bypass is untested. [5](#0-4)

        }
