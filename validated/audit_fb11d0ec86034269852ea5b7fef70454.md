### Title
Double-Charging of `update_fee` in `verify_wormhole_merkle_callback` Causes Users to Overpay — (File: `target_chains/near/receiver/src/lib.rs`)

---

### Summary

In the NEAR Pyth receiver contract, `verify_wormhole_merkle_callback` pre-adjusts the storage baseline by subtracting `update_fee / storage_byte_cost` bytes **and** passes `additional_fee = Some(self.update_fee)` to `refund_storage_usage`. This causes the fee to be charged twice, so callers using the accumulator (Merkle) update path receive approximately half the refund they are entitled to — a direct analog to the VodkaVault pattern where a guard uses one value but the deduction uses a different, larger value.

---

### Finding Description

`verify_wormhole_batch_callback` (the legacy batch path) captures the storage baseline as the raw current usage and charges the fee solely through `additional_fee`:

```rust
// verify_wormhole_batch_callback  (lines 243–297)
let storage = env::storage_usage();          // ← no pre-adjustment
// ...
Self::refund_storage_usage(
    account_id, storage, env::storage_usage(),
    env::attached_deposit(),
    Some(self.update_fee),                   // ← fee charged once here
)
```

`verify_wormhole_merkle_callback` (the accumulator path) pre-subtracts `fee / storage_byte_cost` from the baseline **and** still passes `additional_fee`:

```rust
// verify_wormhole_merkle_callback  (lines 317–325)
let storage = (env::storage_usage() as u128)
    .checked_sub(
        self.update_fee
            .as_yoctonear()
            .checked_div(env::storage_byte_cost().as_yoctonear())  // ← fee baked into baseline
            .ok_or(Error::ArithmeticOverflow)?,
    )
    ...;
// ...
Self::refund_storage_usage(
    account_id, storage, env::storage_usage(),
    env::attached_deposit(),
    Some(self.update_fee),                   // ← fee charged AGAIN
)
```

Inside `refund_storage_usage` (lines 671–675), when storage increases:

```rust
let diff = after - before;   // = actual_diff + fee_in_bytes  (inflated baseline)
let cost = storage_byte_cost * diff + fee;
         // = storage_byte_cost * actual_diff + fee + fee   ← 2× fee
```

The arithmetic:

| Variable | Value |
|---|---|
| `before` | `actual_before − fee/storage_byte_cost` |
| `after` | `actual_before + actual_diff` |
| `diff` | `actual_diff + fee/storage_byte_cost` |
| `cost` | `storage_byte_cost × actual_diff + 2 × update_fee` |

The correct cost should be `storage_byte_cost × actual_diff + update_fee`.

---

### Impact Explanation

Every caller of `update_price_feeds` using the accumulator (Merkle) format — the current production format — is overcharged by approximately `update_fee` yoctoNEAR per call. The excess is silently retained by the contract rather than refunded. Over time this drains user deposits and constitutes a systematic financial loss for all price-feed updaters on NEAR.

---

### Likelihood Explanation

The accumulator path is the **default and recommended** update format for Pyth on NEAR. Any unprivileged account calling `update_price_feeds` with a `PYTHNET_ACCUMULATOR_UPDATE_MAGIC`-prefixed payload triggers this path. No special role or key is required. The bug fires on every successful Merkle update.

---

### Recommendation

Remove the pre-adjustment of the storage baseline in `verify_wormhole_merkle_callback` so it matches `verify_wormhole_batch_callback`:

```rust
// Before (buggy):
let storage = (env::storage_usage() as u128)
    .checked_sub(self.update_fee.as_yoctonear()
        .checked_div(env::storage_byte_cost().as_yoctonear())
        .ok_or(Error::ArithmeticOverflow)?)
    ...;

// After (correct):
let storage = env::storage_usage();
```

The fee is already charged correctly through `additional_fee = Some(self.update_fee)` in `refund_storage_usage`. The pre-subtraction is redundant and causes double-charging.

---

### Proof of Concept

1. Deploy the NEAR Pyth receiver with `update_fee = 1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR).
2. Call `update_price_feeds` with a valid accumulator-format payload (Merkle proof). Attach exactly `get_update_fee_estimate(data)` yoctoNEAR.

5. The caller receives a refund that is ~1 NEAR less than expected — the second fee deduction is silently absorbed by the contract.

### Title
Mismatched Fee Values in `executeCallback()` Cause Provider Underpayment and Potential Revert — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the fee collected from the user at request time uses a fixed admin-set value (`pythFeeInWei`), but the fee deducted from the provider's payout at callback time uses a dynamic value (`pythFee = pyth.getUpdateFee(updateData)`). When these two values diverge — which any unprivileged relayer can force by supplying bloated `updateData` — the provider receives less than owed, and if the dynamic fee exceeds the stored provider portion, the subtraction underflows and the entire `executeCallback` reverts, permanently blocking callback execution for that request.

---

### Finding Description

**At request time** (`requestPriceUpdatesWithCallback`, line 84):

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```

The user pays `msg.value ≥ requiredFee = pythFeeInWei + providerFees`. Echo stores the provider's portion as `req.fee = msg.value − pythFeeInWei` and credits `_state.accruedFeesInWei += pythFeeInWei`.

**At callback time** (`executeCallback`, lines 145 and 161–162):

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);          // dynamic, caller-influenced
...
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - pythFee);  // subtraction uses pythFee, not pythFeeInWei
```

`pythFee` is computed from the `updateData` argument supplied by the relayer. `pyth.getUpdateFee` charges per update present in `updateData`, regardless of how many match `priceIds`. The `priceIds` array is validated against `req.priceIdPrefixes`, but `updateData` itself is not constrained in size. A relayer can therefore include arbitrarily many extra price-feed updates in `updateData`, inflating `pythFee` without bound.

The structural mismatch:

| Step | Value used | Who controls it |
|---|---|---|
| Sufficiency check (request) | `pythFeeInWei` (fixed) | Admin |
| Actual deduction (callback) | `pythFee` (dynamic) | Relayer-supplied `updateData` |

This is the direct analog of the VodkaVaultV2 pattern: the guard condition uses one quantity, but the arithmetic uses a different, potentially larger quantity.

---

### Impact Explanation

**Case 1 — Provider underpayment (always when `pythFee > pythFeeInWei`):**
Provider accrues `req.fee + msg.value_callback − pythFee` instead of the expected `req.fee + msg.value_callback − pythFeeInWei`. The difference is silently taken from the provider's earnings with no revert.

**Case 2 — Callback DoS via underflow (when `pythFee > req.fee + msg.value_callback`):**
Solidity 0.8 reverts on unsigned underflow. Because `clearRequest` is called *after* the fee update (line 164), a revert leaves the request active but the callback permanently unexecutable via this code path. User funds remain locked in the contract and the consumer contract never receives its price data.

---

### Likelihood Explanation

Two realistic triggers exist:

1. **Organic fee drift**: `pythFeeInWei` in Echo is an admin-set estimate. The actual Pyth contract fee can be updated independently. Any period where the Pyth fee exceeds Echo's stored estimate causes underpayment; a large enough divergence causes revert.

2. **Relayer-supplied bloated `updateData`**: Any unprivileged address may call `executeCallback`. By including extra price-feed updates in `updateData` (beyond those matching `priceIds`), a caller inflates `pythFee` at zero cost to themselves. If `pythFee > req.fee + msg.value`, the call reverts. This is a permissionless, low-cost DoS on any in-flight request.

---

### Recommendation

Replace the dynamic `pythFee` deduction with the fixed `pythFeeInWei` that was already collected from the user, and separately enforce that the relayer supplies sufficient `msg.value` to cover the actual Pyth fee:

```solidity
require(msg.value >= pythFee, "Insufficient msg.value for Pyth fee");
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - pythFeeInWei);
```

Or, alternatively, validate that `pythFee <= pythFeeInWei` before proceeding, and revert with a descriptive error if the Pyth fee has drifted above what was collected.

---

### Proof of Concept

1. Alice calls `requestPriceUpdatesWithCallback` with 2 `priceIds`, paying `pythFeeInWei + providerFees`. `req.fee = providerFees` is stored.

2. Mallory (unprivileged relayer) constructs `updateData` containing 1000 price-feed updates (only 2 match `priceIds`). `pyth.getUpdateFee(updateData)` returns `1000 * unitFee`, far exceeding `req.fee + msg.value_callback`.

3. Mallory calls `executeCallback(provider, sequenceNumber, updateData, priceIds)` with `msg.value = 0`.

4. Line 145: `pythFee = 1000 * unitFee`.
   Line 162: `(req.fee + 0) - pythFee` underflows → revert.

5. `clearRequest` was never reached; the request remains active. No other relayer can succeed without also supplying bloated data (since the legitimate `updateData` with 2 feeds would have `pythFee = 2 * unitFee ≤ req.fee` and succeed — but Mallory can front-run every legitimate attempt with the bloated version).

```text
    function getFee(
        address provider,
        uint32 callbackGasLimit,
        bytes32[] calldata priceIds
    ) public view override returns (uint96 feeAmount) {
        uint96 baseFee = _state.pythFeeInWei; // Fixed fee to Pyth
        // Note: The provider needs to set its fees to include the fee charged by the Pyth contract.
        // Ideally, we would be able to automatically compute the pyth fees from the priceIds, but the
        // fee computation on IPyth assumes it has the full updated data.
        uint96 providerBaseFee = _state.providers[provider].baseFeeInWei;
        uint96 providerFeedFee = SafeCast.toUint96(
            priceIds.length * _state.providers[provider].feePerFeedInWei
        );
        uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei; // Provider's per-gas rate
        uint256 gasFee = callbackGasLimit * providerFeeInWei; // Total provider fee based on gas
        feeAmount =
            baseFee +
            providerBaseFee +
            providerFeedFee +
            SafeCast.toUint96(gasFee); // Total fee user needs to pay
```
