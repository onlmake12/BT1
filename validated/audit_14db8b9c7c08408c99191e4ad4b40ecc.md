### Title
Silent Callback Failure Due to Missing `gasleft()` Check Before Gas-Limited `_echoCallback` Call — (`Echo.sol`)

### Summary

`Echo.executeCallback` clears the request and credits provider fees **before** invoking the consumer's callback with a fixed gas cap (`{gas: req.callbackGasLimit}`). There is no `gasleft()` guard to ensure the outer call context holds enough gas to actually forward `callbackGasLimit` to the sub-call. Due to EIP-150 (63/64 rule), a caller who supplies just enough gas to complete the pre-callback state mutations but not enough to forward the full `callbackGasLimit` will cause the callback to silently fail via `try/catch`, while the request is permanently cleared and the provider is already paid.

### Finding Description

In `Echo.executeCallback`, the execution order is:

1. Validate request and price IDs
2. Parse price feeds via Pyth
3. **Credit provider fees** (`_state.providers[providerToCredit].accruedFeesInWei += ...`)
4. **Clear the request** (`clearRequest(sequenceNumber)`)
5. **Advance `firstUnfulfilledSeq`** (unbounded loop over storage)
6. Call `IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` inside a `try/catch` [1](#0-0) 

Steps 3–5 are irreversible state mutations. Step 6 is the only step that delivers value to the consumer. The `try/catch` silently absorbs any revert, including OOG. [2](#0-1) 

The NatSpec on `IEcho.executeCallback` acknowledges the 1.5× gas requirement in a comment but **does not enforce it in code**: [3](#0-2) 

There is no `require(gasleft() >= uint256(req.callbackGasLimit) * 64 / 63)` (or equivalent) anywhere in `Echo.sol`. [4](#0-3) 

Because of EIP-150, the actual gas forwarded to `_echoCallback` is `min(req.callbackGasLimit, gasleft() * 63 / 64)`. If the caller supplies a transaction gas limit that is just sufficient to complete steps 1–5 but leaves fewer than `callbackGasLimit * 64 / 63` gas at step 6, the callback receives less than `callbackGasLimit` gas and reverts with OOG. The `catch` block emits `PriceUpdateCallbackFailed` and returns successfully. [5](#0-4) 

The request is already cleared at this point and **cannot be retried**. The consumer's funds are gone and their application state is never updated.

### Impact Explanation

A consumer who paid fees for a price-update callback permanently loses that callback with no recourse:

- The request is cleared (`clearRequest`) before the callback fires, so there is no retry path.
- The provider is credited their fee regardless of callback success.
- The consumer's on-chain application (e.g., a DeFi protocol relying on the price update to settle a trade or update an oracle) is left in a stale state.

For any consumer whose `echoCallback` performs critical state transitions (position settlement, liquidation triggers, oracle updates), the silent failure causes a permanent, unrecoverable inconsistency between Echo's accounting and the consumer's application state — directly analogous to the gauge accounting inconsistency in the reference report.

### Likelihood Explanation

- **Malicious provider**: A registered provider can intentionally call `executeCallback` with a crafted gas limit — enough to clear the request and collect fees, but not enough to forward `callbackGasLimit` to the consumer. This is a direct, low-cost griefing/theft vector.
- **Accidental**: An honest provider that miscalculates the required gas overhead (the 1.5× rule is only documented, not enforced) will silently fail to deliver the callback.
- The attack requires no special privileges beyond being a registered provider (or any caller after the exclusivity period elapses). [6](#0-5) 

### Recommendation

Add a `gasleft()` guard immediately before the callback invocation, mirroring the pattern used in `Entropy.sol`:

```solidity
// Ensure enough gas remains to forward callbackGasLimit to the sub-call (EIP-150: 63/64 rule)
require(
    gasleft() >= (uint256(req.callbackGasLimit) * 64) / 63,
    "InsufficientGas"
);
try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(
    sequenceNumber, priceFeeds
) { ... } catch { ... }
```

This mirrors the `InsufficientGas` revert path in `Entropy.revealWithCallback`: [7](#0-6) 

Alternatively, move `clearRequest` and fee crediting **after** a successful callback (checks-effects-interactions), so a failed callback leaves the request retryable.

### Proof of Concept

1. Consumer calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 500_000`, paying the required fee.
2. After the exclusivity period, attacker (or provider) calls:
   ```solidity
   echo.executeCallback{gas: G}(provider, sequenceNumber, updateData, priceIds);
   ```
   where `G` is chosen so that after steps 1–5 (storage reads/writes, Pyth parsing, `clearRequest`, `firstUnfulfilledSeq` loop), `gasleft()` at the `try` statement is, say, `480_000` — less than `500_000 * 64/63 ≈ 507_936`.
3. The EVM forwards only `480_000 * 63/64 = 472_500` gas to `_echoCallback`, which is less than `callbackGasLimit = 500_000`.
4. `_echoCallback` runs out of gas and reverts.
5. The `catch` block emits `PriceUpdateCallbackFailed("low-level error (possibly out of gas)")`.
6. `executeCallback` returns successfully. The request is cleared. The provider is paid. The consumer's callback was never executed. [8](#0-7)
