### Title
Permissionless Keeper Can Submit Old Pyth Price Updates to Poison `priceLastUpdatedAt`, Causing `getPricesNoOlderThan` DoS and Subscription Balance Drain — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.sol` contract's `updatePriceFeeds` function is fully permissionless and intentionally accepts Pyth price updates with timestamps up to `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hour` in the past. The `priceLastUpdatedAt` field is set to the maximum `publishTime` found in the submitted update. The `getPricesNoOlderThan` staleness check compares `block.timestamp` against this stored `priceLastUpdatedAt`. An unprivileged keeper can submit a valid but old Pyth VAA (e.g., 59 minutes old) as the first update for any subscription, poisoning `priceLastUpdatedAt` to an old value, causing `getPricesNoOlderThan` to revert for any caller using a staleness threshold shorter than the injected age, while simultaneously collecting the keeper fee from the subscription's balance.

---

### Finding Description

**Root cause — `updatePriceFeeds` accepts old timestamps and stores them verbatim:**

`updatePriceFeeds` calls `parsePriceFeedUpdatesWithConfig` with `minPublishTime = 0`:

```solidity
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0,  // minPublishTime — accepts any past timestamp
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
    false, true, false
);
``` [1](#0-0) 

`_validateShouldUpdatePrices` then computes `updateTimestamp` as the **maximum** `publishTime` across all returned feeds and enforces only that it is within `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hour`:

```solidity
uint256 minAllowedTimestamp = (block.timestamp > PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    : 0;
if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(...);
}
``` [2](#0-1) 

For the **first update** (`priceLastUpdatedAt == 0`), the heartbeat check short-circuits unconditionally:

```solidity
if (lastUpdateTime == 0 || updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds) {
    return updateTimestamp;
}
``` [3](#0-2) 

The returned `updateTimestamp` (the old timestamp) is then stored directly:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [4](#0-3) 

**Staleness check uses `priceLastUpdatedAt` verbatim:**

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();
``` [5](#0-4) 

`PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is 1 hour: [6](#0-5) 

**Attack path:**

1. Attacker queries the Hermes historical API for a valid signed Pyth VAA published ~3599 seconds ago (within the 1-hour window).
2. Attacker calls `updatePriceFeeds(subscriptionId, oldUpdateData)` on a subscription whose `priceLastUpdatedAt == 0` (new subscription or one that had its price IDs reset).
3. `parsePriceFeedUpdatesWithConfig` with `minPublishTime = 0` accepts the old VAA and returns feeds with `publishTime = block.timestamp - 3599`.
4. `_validateShouldUpdatePrices` passes: timestamp is within 1 hour, `priceLastUpdatedAt == 0` so heartbeat check short-circuits.
5. `priceLastUpdatedAt` is set to `block.timestamp - 3599`.
6. Any subsequent call to `getPricesNoOlderThan(subscriptionId, priceIds, age)` with `age < 3599` reverts with `StalePrice`.
7. Attacker collects the keeper fee (`gasCost + keeperSpecificFee`) from the subscription's balance.

**Sustained attack for short-heartbeat subscriptions:**

For subscriptions with `heartbeatSeconds = 60`, the attacker can chain 60 successive old updates (each 60 seconds newer than the last, all within the 1-hour window), collecting the keeper premium on each call and keeping `priceLastUpdatedAt` artificially old for up to ~1 hour. Each call drains the subscription balance.

---

### Impact Explanation

- **DoS on `getPricesNoOlderThan` / `getEmaPricesNoOlderThan`:** Any protocol reading prices from the Scheduler with a staleness threshold shorter than the injected age (e.g., 60 seconds) will receive `StalePrice` reverts, blocking liquidations, deposits, withdrawals, or any price-dependent operation.
- **Subscription balance drain:** The attacker is paid the keeper fee (gas reimbursement + per-feed premium) from the victim subscription's balance on each successful call. For short-heartbeat subscriptions, this can be repeated many times within the 1-hour window.
- **Permanent subscriptions are especially at risk:** `isPermanent` subscriptions cannot be deactivated or have their parameters changed by the manager, so the manager cannot easily reset `priceLastUpdatedAt` to recover.

---

### Likelihood Explanation

- `updatePriceFeeds` has **no access control** — any address can call it.
- Valid historical Pyth VAAs are freely available from the public Hermes API (`https://hermes.pyth.network`).
- The attacker **profits** from each call (keeper premium exceeds gas cost by design), making the attack economically self-sustaining.
- The attack targets the first update of any subscription (or after a price-ID reset), which is a predictable and observable on-chain event.
- No special privileges, leaked keys, or third-party oracle misbehavior are required.

---

### Recommendation

1. **Enforce a minimum `publishTime` relative to `block.timestamp` when storing `priceLastUpdatedAt`.** After `parsePriceFeedUpdatesWithConfig`, before storing, assert that `updateTimestamp >= block.timestamp - MAX_ACCEPTABLE_STALENESS_FOR_STORAGE` (e.g., 60–300 seconds), not just `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`.
2. **Decouple the timestamp used for heartbeat gating from the timestamp stored as `priceLastUpdatedAt`.** The heartbeat check can use the old timestamp to decide whether an update is due, but `priceLastUpdatedAt` should only be set to a timestamp that is genuinely recent (close to `block.timestamp`).
3. **Add access control or a registration requirement for keepers**, or at minimum require that the submitted `updateTimestamp` is within a tighter window (e.g., 60 seconds of `block.timestamp`) before paying the keeper fee.
