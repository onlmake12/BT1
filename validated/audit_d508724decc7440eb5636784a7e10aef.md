### Title
Permanent Subscription `MAX_DEPOSIT_LIMIT` Bypass via Repeated `addFunds` Calls - (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.sol` contract enforces a `MAX_DEPOSIT_LIMIT` (100 ether) on permanent subscriptions to cap the total ETH that can be locked in them. However, both `createSubscription` and `addFunds` check only `msg.value` (the per-transaction deposit) against the cap, not the cumulative `status.balanceInWei`. This allows any user to bypass the cap entirely by calling `addFunds` repeatedly with amounts at or below `MAX_DEPOSIT_LIMIT`, accumulating an unbounded balance in a permanent subscription from which withdrawals are permanently prohibited.

---

### Finding Description

`MAX_DEPOSIT_LIMIT` is defined as 100 ether in `SchedulerConstants.sol`: [1](#0-0) 

In `createSubscription`, the guard is:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [2](#0-1) 

In `addFunds`, the guard is identical in structure:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [3](#0-2) 

Both checks validate only `msg.value` — the amount sent in the current transaction — not the resulting cumulative `status.balanceInWei`. After the check passes, `status.balanceInWei += msg.value` unconditionally accumulates the deposit.

The correct check should be against the post-deposit cumulative balance:
```solidity
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Permanent subscriptions additionally prohibit all withdrawals: [4](#0-3) 

This means any ETH deposited into a permanent subscription is irreversibly locked.

---

### Impact Explanation

A user can lock an arbitrarily large amount of ETH into a permanent subscription, far exceeding the intended 100 ether cap. Because permanent subscriptions cannot be deactivated, updated, or withdrawn from, the excess ETH is permanently trapped in the contract. The `MAX_DEPOSIT_LIMIT` is the only protection against this; bypassing it removes the only safeguard. This is a direct financial loss to the depositor and a violation of the protocol's stated invariant.

---

### Likelihood Explanation

The entry path requires no special privileges — `addFunds` is callable by anyone for any active subscription: [5](#0-4) 

A subscription manager who creates a permanent subscription and then calls `addFunds` multiple times (each with `msg.value ≤ MAX_DEPOSIT_LIMIT`) will trigger this silently. No malicious intent is required; a user who simply wants to top up their permanent subscription balance multiple times will inadvertently bypass the cap.

---

### Recommendation

Replace the per-transaction `msg.value` check with a cumulative post-deposit balance check in both `createSubscription` and `addFunds`:

```solidity
// In addFunds:
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

```solidity
// In createSubscription:
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
// (createSubscription starts from 0, so msg.value == post-deposit balance; this check is already correct for the initial deposit only)
```

For `createSubscription` the check is incidentally correct (initial balance is 0, so `msg.value == post-deposit balance`), but `addFunds` must be fixed to check `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`.

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription with exactly MAX_DEPOSIT_LIMIT (100 ether) — passes
scheduler.createSubscription{value: 100 ether}(permanentParams);
// status.balanceInWei == 100 ether

// 2. Call addFunds with MAX_DEPOSIT_LIMIT again — msg.value (100 ether) is NOT > MAX_DEPOSIT_LIMIT, passes
scheduler.addFunds{value: 100 ether}(subscriptionId);
// status.balanceInWei == 200 ether  ← cap bypassed

// 3. Repeat indefinitely
scheduler.addFunds{value: 100 ether}(subscriptionId);
// status.balanceInWei == 300 ether

// All ETH is permanently locked; withdrawals revert with CannotUpdatePermanentSubscription
```

The existing test `testPermanentSubscriptionDepositLimit` (Test 3) only verifies that a single `addFunds` call with `msg.value > MAX_DEPOSIT_LIMIT` reverts — it does not test the cumulative bypass scenario described above. [6](#0-5)
