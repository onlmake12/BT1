### Title
Unrestricted Access to `remove_transaction` RPC Allows Any Caller to Evict Arbitrary Pending Transactions from the Pool — (File: `rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` RPC method in the default-enabled `Pool` module performs no authentication and no ownership check before evicting a transaction and all its descendants from the tx-pool. Any process that can reach the RPC endpoint — by default any local process on the node host — can silently remove any pending transaction submitted by any user. This is a direct structural analog to the `OmoRouter.transfer()` pattern: a privileged, state-mutating operation is exposed without any caller-identity restriction.

---

### Finding Description

`remove_transaction` is declared in the `PoolRpc` trait and implemented in `PoolRpcImpl` with no guard whatsoever:

```rust
// rpc/src/module/pool.rs  lines 662-669
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
``` [1](#0-0) 

The trait declaration confirms there is no `#[restricted]` annotation, no caller-identity parameter, and no module-level auth middleware: [2](#0-1) 

A grep across the entire `rpc/src/` tree for `authentication`, `auth`, `token`, `bearer`, `password`, or `secret` returns zero matches — the RPC server has no authentication layer at all.

The `Pool` module is listed in the production default module set:

```toml
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
``` [3](#0-2) 

`remove_local_tx` propagates directly to `TxPool::remove_tx`, which removes the target entry **and all descendants** unconditionally: [4](#0-3) 

The service-layer message path confirms no ownership metadata is threaded through the call: [5](#0-4)

### Impact Explanation

Any process on the node host (or any remote host if the operator widens `listen_address` beyond `127.0.0.1`) can:

1. Enumerate pending transactions via `get_raw_tx_pool`.
2. Call `remove_transaction(<tx_hash>)` to evict a target transaction and its entire descendant chain from the pool.
3. The victim's transaction disappears silently; the victim must detect the absence and resubmit.

In time-sensitive scenarios — DAO withdrawal unlock windows, HTLC/timelock expiry, liquidation races — forced eviction of a pending transaction before it is committed causes irreversible financial loss to the transaction submitter. The attacker gains nothing directly (unlike `OmoRouter.transfer()`), but the victim loses the economic opportunity or the locked asset.

**Impact: Medium** — targeted disruption of specific transactions with potential for financial loss in time-sensitive contexts; not a direct theft but a denial of a committed economic action.

---

### Likelihood Explanation

- The RPC binds to `127.0.0.1:8114` by default, so the attacker must be a local process on the node host (or the operator must have exposed the port).
- No credential, key, or privilege is required beyond network reachability.
- Transaction hashes are public (visible via `get_raw_tx_pool` or the P2P relay network), so the attacker can identify targets without any privileged knowledge.
- The `Pool` module is enabled by default; no special configuration is needed.

**Likelihood: Medium** — local-process access is a real precondition, but it is not a privileged-key requirement; any co-resident service, script, or compromised dependency satisfies it.

---

### Recommendation

1. **Add caller authentication** to the RPC server (e.g., a shared secret token in the HTTP header) so that only the node operator can invoke state-mutating Pool methods.
2. Alternatively, **move `remove_transaction` and `clear_tx_pool` to a restricted module** (e.g., a new `Admin` module disabled by default), separate from the read-heavy `Pool` module that operators may expose more broadly.
3. At minimum, document that `remove_transaction` is an operator-only administrative function and gate it behind the same access controls as `clear_tx_pool`.

---

### Proof of Concept

On any host with access to the RPC endpoint (default: the node host itself):

```bash
# Step 1 – discover a pending transaction hash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false],"id":1}'
# Returns: {"result":{"pending":["0xabc..."],...}}

# Step 2 – evict the transaction with no credentials
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xabc..."],"id":2}'
# Returns: {"result":true}
# The transaction and all its descendants are now gone from the pool.
```

No key, no token, no privilege — any caller that can reach port 8114 can evict any pending transaction submitted by any user.
