### Title
Cross-Chain Replay of Lazer Price Updates Due to Missing Chain ID Binding in `verifyUpdate` - (File: lazer/contracts/evm/src/PythLazer.sol)

### Summary
The `verifyUpdate` function in `PythLazer.sol` computes the ECDSA signature hash as `keccak256(payload)` with no chain ID, contract address, or EIP-712 domain separator included in the signed data. Because the same trusted signer keys are registered across all EVM Lazer deployments and the payload contains no chain-specific context, a valid signed price update from one EVM chain is cryptographically valid on every other EVM chain where Pyth Lazer is deployed. An unprivileged attacker can replay a valid update from chain A onto chain B, injecting stale prices before the legitimate pusher delivers the current update.

### Finding Description
In `lazer/contracts/evm/src/PythLazer.sol`, `verifyUpdate` (lines 70–106) performs the following verification:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
if (signer == address(0)) { revert("invalid signature"); }
if (!isValidSigner(signer)) { revert("invalid signer"); }
```

The payload structure (confirmed by the test helper `buildPayload` in `lazer/contracts/evm/test/PythLazer.t.sol` lines 78–95) is:

```
PAYLOAD_FORMAT_MAGIC (uint32) | timestamp (uint64) | channel (uint8) | feedsLen (uint8) | feed data
```

There is no `block.chainid`, no contract address, and no EIP-712 domain separator anywhere in the signed data. The Pyth Lazer server signs the same payload content for all EVM chains (same asset prices, same timestamp). The `updateTrustedSigner` function (lines 31–64) registers the same Pyth-controlled signer address on every chain.

**Attack path:**
1. At time T, the Lazer server signs a payload with price P₁ and broadcasts it to all chains.
2. The price moves to P₂ at time T+1; the server signs a new payload.
3. The legitimate Lazer pusher submits the P₂ update to chain A first.
4. Before the P₂ update reaches chain B, the attacker submits the T-epoch payload (price P₁) to chain B.
5. `verifyUpdate` on chain B succeeds: the signature is valid (same trusted signer, same payload bytes), and the signer is not expired.
6. Consumer contracts on chain B now hold stale price P₁ instead of P₂.

The attacker needs only to observe the mempool or on-chain events on chain A and submit the older payload to chain B — no privileged keys required.

### Impact Explanation
Consumer DeFi contracts (lending protocols, perpetual DEXs, options vaults) that call `verifyUpdate` and then use the returned `payload` for pricing decisions receive a price that is valid by signature but stale by chain context. Concrete impacts:

- **Lending protocol manipulation**: Borrow against collateral priced at P₁ when the true price is P₂ < P₁, extracting undercollateralized loans.
- **Perpetual DEX manipulation**: Open or close positions against a stale mark price, extracting value from the protocol's insurance fund or LPs.
- **Options/structured product manipulation**: Exercise options or trigger liquidations at incorrect prices.

The severity is amplified because Lazer is explicitly marketed for high-frequency, latency-sensitive use cases where even sub-second price staleness is material.

### Likelihood Explanation
The attacker requires:
- A node connection to two EVM chains where Pyth Lazer is deployed (publicly available).
- The ability to observe a valid signed update on chain A (visible in the mempool or on-chain).
- The ability to submit a transaction to chain B before the legitimate pusher does (feasible with standard MEV infrastructure).

No privileged access, no leaked keys, and no governance majority are required. The attack is repeatable on every block where a price movement occurs.

### Recommendation
Include chain-specific context in the signed data. The minimal fix is to incorporate `block.chainid` and the contract address into the hash:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    payload
));
```

Alternatively, adopt EIP-712 structured signing with a domain separator that encodes `chainId` and `verifyingContract`. The Lazer server must then produce per-chain signatures, which eliminates cross-chain replay entirely.

### Proof of Concept
1. Deploy `PythLazer` on two local forks of Ethereum (chain 1) and Arbitrum (chain 42161), registering the same `trustedSigner`.
2. Construct a valid `update` blob signed by `trustedSigner` with timestamp T and price P₁.
3. Call `pythLazer.verifyUpdate{value: fee}(update)` on the Ethereum fork — succeeds.
4. Call `pythLazer.verifyUpdate{value: fee}(update)` on the Arbitrum fork with the **identical bytes** — also succeeds, returning the same `payload`.
5. Construct a second update with timestamp T+1 and price P₂ ≠ P₁; submit it only to the Ethereum fork.
6. The Arbitrum fork still holds the T/P₁ payload as the last verified update, demonstrating stale-price injection with no signature forgery.
