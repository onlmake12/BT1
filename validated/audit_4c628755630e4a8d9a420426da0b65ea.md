### Title
Truncating `as u64` Cast in `calculate_maximum_withdraw` Produces Wrong DAO Withdrawal Capacity — (File: `util/dao/src/lib.rs`)

### Summary
`DaoCalculator::calculate_maximum_withdraw` computes an intermediate `u128` result and then silently truncates it to `u64` via an unchecked `as u64` cast. Every other analogous `u128`→`u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. The inconsistency means that when the intermediate product exceeds `u64::MAX`, the withdrawal capacity is silently corrupted rather than returning an error, producing a wrong (arbitrarily small) capacity value that propagates into fee accounting and tx-pool admission.

### Finding Description
In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the maximum withdrawable capacity for a NervosDAO cell:

```rust
// lines 152–156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The expression `withdraw_counted_capacity as u64` is a Rust **truncating cast**: if the `u128` value exceeds `u64::MAX` (2^64 − 1), the high 64 bits are silently discarded and the result wraps to an arbitrary value in `[0, u64::MAX]`. This is the exact same class of defect as the reported "unchecked arithmetic" bug — arithmetic performed without overflow protection, allowing the result to silently become wrong.

The rest of the same file consistently uses the safe pattern for identical `u128`→`u64` conversions:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

`calculate_maximum_withdraw` is the only site that deviates from this pattern.

### Impact Explanation
`calculate_maximum_withdraw` is called from two paths:

1. **Tx-pool admission / fee verification** — `transaction_maximum_withdraw` → `transaction_fee` → `check_tx_fee` in `tx-pool/src/util.rs`. The fee is computed as `maximum_withdraw − outputs_capacity`. If `maximum_withdraw` is silently truncated to a small value, the computed fee becomes wrong (potentially appearing negative, causing a valid DAO withdrawal to be rejected, or appearing artificially large/small, corrupting fee-rate ordering). [5](#0-4) [6](#0-5) 

2. **RPC exposure** — `calculate_maximum_withdraw` is called directly from `rpc/src/module/experiment.rs`, meaning any RPC caller can trigger the path with attacker-controlled header hashes and cell outputs. [7](#0-6) 

The corrupted capacity value is used as the authoritative withdrawal amount. A truncation that produces a value smaller than `outputs_capacity` causes `safe_sub` to return an error, incorrectly rejecting a legitimate DAO withdrawal. A truncation that produces a value larger than the true amount could allow a transaction to pass fee checks with an inflated apparent fee, corrupting tx-pool fee-rate ordering.

### Likelihood Explanation
The overflow condition requires `counted_capacity * withdrawing_ar > u64::MAX * deposit_ar`. Since `counted_capacity ≤ u64::MAX` shannons and `withdrawing_ar ≥ deposit_ar`, the intermediate product `counted_capacity * withdrawing_ar` can exceed `u64::MAX` when `counted_capacity` is near `u64::MAX` (≈ 1.84 × 10^10 CKB) and `withdrawing_ar` is meaningfully larger than `deposit_ar`. A transaction sender constructing a DAO withdrawal with a very large cell capacity (near the u64 maximum) and referencing headers with a sufficiently grown AR ratio can trigger this path. The cell capacity bound is enforced only by the cell's own declared capacity field, which is attacker-controlled at deposit time.

### Recommendation
Replace the truncating cast with the same checked conversion used everywhere else in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with rest of file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [8](#0-7) 

### Proof of Concept
1. Craft a DAO deposit cell with capacity near `u64::MAX` shannons (e.g., `18_446_744_073_000_000_000`).
2. Wait (or construct headers) such that `withdrawing_ar / deposit_ar` is large enough that `counted_capacity * withdrawing_ar > u64::MAX * deposit_ar`.
3. Submit a Phase-2 DAO withdrawal transaction spending this cell.
4. `calculate_maximum_withdraw` computes `withdraw_counted_capacity` as a `u128` exceeding `u64::MAX`.
5. The `as u64` cast silently truncates the value, producing an arbitrary small number.
6. `transaction_fee` computes `maximum_withdraw − outputs_capacity`; with the truncated `maximum_withdraw` smaller than `outputs_capacity`, `safe_sub` returns `DaoError::Overflow`, and the valid withdrawal is rejected by the tx-pool — or, depending on truncation result, the fee is computed incorrectly, corrupting fee-rate ordering.

The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` already demonstrates awareness of overflow in this function but does not cover the `as u64` truncation path specifically. [9](#0-8)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)

#[test]
fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
