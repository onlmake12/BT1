### Title
Missing Zero-Check on DAO `C` Divisor in `secondary_block_reward` — (`util/dao/src/lib.rs`)

---

### Summary

In `DaoCalculator::secondary_block_reward`, the function guards against the genesis block by checking `target.number() == 0`, but the actual divisor used in the reward calculation is `target_parent_c` — the `C` (total capacity) field extracted from the parent block's DAO data. No zero-check is performed on `target_parent_c` before dividing by it. This is a direct analog to M-17: the guard checks the wrong variable (block number) instead of the actual divisor (`target_parent_c`).

---

### Finding Description

In `util/dao/src/lib.rs`, `secondary_block_reward` performs the following:

```rust
pub fn secondary_block_reward(&self, target: &HeaderView) -> Result<Capacity, DaoError> {
    if target.number() == 0 {          // ← guards on block number
        return Ok(Capacity::zero());
    }
    // ...
    let (_, target_parent_c, _, target_parent_u) = extract_dao_data(target_parent.dao());
    let reward128 = u128::from(target_g2.as_u64()) * u128::from(target_parent_u.as_u64())
        / u128::from(target_parent_c.as_u64());  // ← divides by target_parent_c, no zero-check
``` [1](#0-0) 

The guard at line 185 checks `target.number() == 0` (the block number), but the divisor at line 203 is `target_parent_c` — the `C` field from the parent block's DAO header field. These are two independent variables. The guard does not protect against `target_parent_c == 0`.

The same pattern appears in `dao_field_with_current_epoch`, which also divides by `parent_c` without a zero-check:

```rust
let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
    / u128::from(parent_c.as_u64());
// ...
let ar_increase128 =
    u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
``` [2](#0-1) 

And in `calculate_maximum_withdraw`, which divides by `deposit_ar` without a zero-check:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
``` [3](#0-2) 

---

### Impact Explanation

In Rust, integer division by zero causes a thread panic (not undefined behavior). If `target_parent_c` is zero when `secondary_block_reward` is called, the node process panics and crashes. This is a remote denial-of-service: a block relayer or miner who submits a block whose parent header carries a crafted DAO field with `C = 0` can trigger the panic during block reward verification, crashing the node.

The same crash path exists in `dao_field_with_current_epoch` (called during block assembly and verification) and `calculate_maximum_withdraw` (called during DAO withdrawal transaction verification).

---

### Likelihood Explanation

Under normal chain operation, `C` starts at a non-zero genesis value and only increases, so `target_parent_c` is never zero on a fully-validated chain. However:

1. The DAO field in a block header is set by the miner/block producer. A malicious block producer can craft a block with an arbitrary DAO field value, including `C = 0`.
2. `secondary_block_reward` and `dao_field_with_current_epoch` are called **during** block verification — before the DAO field of the submitted block is fully rejected. If the parent block's DAO field was accepted with `C = 0` (e.g., due to a separate validation gap or ordering issue), the panic fires before the rejection path is reached.
3. The functions are `pub` and reachable from RPC paths (e.g., `get_block_economic_state`) and block processing pipelines, widening the attack surface.

---

### Recommendation

Replace the block-number guard with a guard on the actual divisor. For `secondary_block_reward`:

```rust
let (_, target_parent_c, _, target_parent_u) = extract_dao_data(target_parent.dao());
if target_parent_c.as_u64() == 0 {
    return Err(DaoError::InvalidDaoFormat);
}
let reward128 = u128::from(target_g2.as_u64()) * u128::from(target_parent_u.as_u64())
    / u128::from(target_parent_c.as_u64());
```

Apply the same pattern in `dao_field_with_current_epoch` (guard `parent_c > 0` before both divisions) and in `calculate_maximum_withdraw` (guard `deposit_ar > 0` before dividing by it). [4](#0-3) 

---

### Proof of Concept

1. Craft a block `B_parent` at height `H` with a DAO field encoding `C = 0` (all other fields arbitrary).
2. Submit a child block `B_child` at height `H+1` referencing `B_parent` as its parent to the node via the P2P relay or RPC block submission path.
3. The node calls `secondary_block_reward(&B_child_header)` during block verification.
4. `target.number()` is `H+1 > 0`, so the guard passes.
5. `extract_dao_data(B_parent.dao())` returns `target_parent_c = 0`.
6. Line 203 executes `... / u128::from(0u64)` → integer division by zero → thread panic → node crash. [5](#0-4)

        if target.number() == 0 {
            return Ok(Capacity::zero());

        let target_parent = self
            .data_loader
            .get_header(&target_parent_hash)

        let target_epoch = self
            .data_loader
            .get_epoch_ext(target)
            .ok_or(DaoError::InvalidHeader)?;

        let target_g2 = target_epoch
            .secondary_block_issuance(target.number(), self.consensus.secondary_epoch_reward())?;
        let (_, target_parent_c, _, target_parent_u) = extract_dao_data(target_parent.dao());
        let reward128 = u128::from(target_g2.as_u64()) * u128::from(target_parent_u.as_u64())
            / u128::from(target_parent_c.as_u64());
        let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
        Ok(Capacity::shannons(reward))
    }
