Audit Report

## Title
Missing `BLOCK_INVALID` Early-Return in `HeaderAcceptor::accept()` Allows Re-insertion of Invalid Headers and Redundant Block Downloads - (File: sync/src/synchronizer/headers_process.rs)

## Summary

`HeaderAcceptor::accept()` contains an acknowledged FIXME that omits an early-return guard for `BLOCK_INVALID` status. Because `BLOCK_INVALID` (`1 << 12`) and `HEADER_VALID` (`1`) are disjoint bit flags, a header whose block was already rejected by full verification falls through all three lightweight checks and reaches `insert_valid_header`, which re-inserts the header into `header_map` and advances `shared_best_header` to the invalid block. The block fetcher in `block_fetcher.rs` has no `BLOCK_INVALID` guard in its inner fetch loop, so it will subsequently re-request the block body from the peer, creating a repeatable redundant-download cycle.

## Finding Description

**Root cause — acknowledged FIXME with disjoint bit flags:**

`HeaderAcceptor::accept()` at line 301–304 of `sync/src/synchronizer/headers_process.rs` contains an explicit FIXME comment acknowledging the missing guard. The status check uses `status.contains(BlockStatus::HEADER_VALID)`. Since `BLOCK_INVALID = 1 << 12 = 4096` and `HEADER_VALID = 1` share no bits, a block carrying `BLOCK_INVALID` status never satisfies this branch and falls through to the three lightweight checks. [1](#0-0) [2](#0-1) 

**Three lightweight checks do not catch body-invalid blocks:**

1. `prev_block_check` (lines 244–253) only checks whether the *parent* carries `BLOCK_INVALID`; it does not check the header itself. [3](#0-2) 

2. `non_contextual_check` runs `HeaderVerifier`. A block marked invalid due to script-execution failure in `ConsumeUnverifiedBlockProcessor` has a structurally valid header and passes this check. [4](#0-3) 

3. `version_check` only inspects the version field. [5](#0-4) 

**`insert_valid_header` is reached and corrupts sync state:**

If all three pass, `sync_shared.insert_valid_header(self.peer, self.header)` is called at line 356, inserting the header into `header_map` and calling `may_set_shared_best_header`, advancing `shared_best_header` to the invalid block. [6](#0-5) [7](#0-6) 

**Block fetcher has no `BLOCK_INVALID` guard — redundant downloads follow:**

The inner loop in `block_fetcher.rs` lines 257–284 only skips blocks with `BLOCK_STORED` or `BLOCK_RECEIVED` status. A block with `BLOCK_INVALID` status (disjoint from both) falls into the else-if branch and is added to the fetch list. Since `best_known` now points to the invalid block (set by `insert_valid_header` via `may_set_best_known_header`), the fetcher traverses the chain from that tip and re-requests the block body. [8](#0-7) 

**`BLOCK_INVALID` is set after full contextual verification failure:** [9](#0-8) 

**Contrast — `CompactBlockProcess` correctly rejects `BLOCK_INVALID` at lines 259–261:** [10](#0-9) 

**Test confirms `CompactBlockProcess` behavior:** [11](#0-10) 

## Impact Explanation

The concrete impact is **CKB network congestion with few costs (High)**. Once an attacker possesses a block with a valid header but an invalid body (e.g., a failing script), they can repeatedly send the header to any node via `SendHeaders`. Each delivery causes `insert_valid_header` to advance `shared_best_header` to the invalid block, and the block fetcher — lacking a `BLOCK_INVALID` guard — re-requests the full block body. The block is rejected again by the chain service, the cycle repeats, and the attacker wastes the victim node's bandwidth and CPU with a trivially cheap message stream. Across many nodes, this constitutes network-wide congestion. Additionally, `shared_best_header` pointing to an invalid high-difficulty chain can mislead the synchronizer's chain-selection logic, delaying sync with the legitimate chain.

## Likelihood Explanation

The attack requires one-time PoW work to produce a block with a valid header and invalid body (e.g., a script that fails contextual verification). After that, the attacker sends `SendHeaders` messages — a standard, rate-unlimited sync protocol message accepted from all peers — at negligible cost. No special privileges, keys, or majority hashpower are required beyond the initial block construction. The attack is repeatable and can be directed at any number of connected peers simultaneously.

## Recommendation

Add an explicit early-return for `BLOCK_INVALID` immediately after reading the status in `HeaderAcceptor::accept()`, resolving the acknowledged FIXME:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing logic
}
```

Additionally, add a `BLOCK_INVALID` guard in the `block_fetcher.rs` inner loop to prevent re-fetching already-rejected blocks:

```rust
if status.contains(BlockStatus::BLOCK_STORED) {
    // ...
} else if status.contains(BlockStatus::BLOCK_RECEIVED) {
    // Do not download repeatedly
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    // Do not re-fetch known-invalid blocks
} else if ...
```

## Proof of Concept

1. Connect an attacker peer to a victim node.
2. Craft a block `B` with a valid PoW header and an invalid body (e.g., a transaction whose lock script always fails).
3. Submit `B` via `SendBlock`; the chain service marks `B` as `BLOCK_INVALID` after contextual verification.
4. Immediately send a `SendHeaders` message containing `B`'s header.
5. `HeaderAcceptor::accept()` reads `BLOCK_INVALID` status, skips the `HEADER_VALID` branch, passes `prev_block_check` (parent is valid), passes `non_contextual_check` (header is structurally valid), passes `version_check`, and calls `insert_valid_header`.
6. `shared_best_header` now points to `B`; the block fetcher adds `B` to the in-flight list and re-requests the block body.
7. The chain service rejects `B` again; repeat from step 4 indefinitely.

A unit test can be written analogously to `test_in_block_status_map` in `sync/src/relayer/tests/compact_block_process.rs`, but targeting `HeaderAcceptor::accept()` with a pre-seeded `BLOCK_INVALID` status, asserting that the result state is `ValidationState::Invalid` rather than `ValidationState::Valid`.
