### Title
Provider Can Frontrun `requestPriceUpdatesWithCallback` to Extract Excess Fees — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` stores the entire `msg.value - pythFeeInWei` as `req.fee`, which is later paid out to the provider in `executeCallback`. Because `setProviderFee` has no timelock and takes effect immediately, a provider can frontrun a user's request transaction to increase their fee. If the user sent more than the old minimum (a common defensive practice), the transaction succeeds but the provider captures the full surplus. There is no `maxFee` parameter to bound what the user is willing to spend.

---

### Finding Description

`getFee` computes the required payment as a sum of four components controlled by the provider: [1](#0-0) 

`setProviderFee` updates all three provider-controlled components atomically and immediately, with no delay or timelock: [2](#0-1) 

Inside `requestPriceUpdatesWithCallback`, the only protection is a minimum-fee check. After that check passes, the **entire** `msg.value - pythFeeInWei` is stored as `req.fee` regardless of what the provider's fee actually was: [3](#0-2) 

When `executeCallback` is later called, the provider receives `req.fee` in full: [4](#0-3) 

**Attack path (overpay variant):**

1. User calls `getFee(provider, gasLimit, priceIds)` off-chain and observes fee = X.
2. User submits `requestPriceUpdatesWithCallback{value: X + buffer}` — sending a buffer is standard practice to avoid reverts from minor fee fluctuations.
3. Provider (or its `feeManager`) observes the pending transaction in the mempool and calls `setProviderFee` with a higher fee Y, where `X < Y ≤ X + buffer`, paying a higher gas tip to land first.
4. User's transaction executes: `getFee` now returns Y ≤ X + buffer, so the minimum-fee check passes.
5. `req.fee = (X + buffer) - pythFeeInWei` is stored — the full buffer is credited to the provider.
6. Provider calls `executeCallback` and collects `req.fee`, extracting `buffer` more ETH than the user intended to pay.

**Attack path (revert variant):**

If the user sends exactly X and the provider raises the fee above X before the transaction lands, the call reverts at line 76 with `InsufficientFee`, wasting the user's gas and forcing a retry at the higher fee.

This is structurally identical to the reference report: the provider (analogous to the account being liquidated) can modify state after the user has committed to a transaction, and the user has no on-chain bound on how much they are willing to spend.

---

### Impact Explanation

A malicious or opportunistic provider can systematically extract ETH beyond the fee the user intended to pay. Because `req.fee` captures all of `msg.value - pythFeeInWei`, any buffer the user includes as a safety margin becomes direct profit for the provider. Over many requests, this constitutes a reliable fee-extraction mechanism with no protocol-level cap. Users who send exactly the quoted fee face forced reverts and gas loss.

---

### Likelihood Explanation

Providers are financially incentivized to maximize `req.fee`. The `setProviderFee` call is cheap and requires no special infrastructure beyond mempool monitoring. The exclusivity period in `executeCallback` guarantees the assigned provider is the one who collects `req.fee`, so the provider bears no execution risk. The attack is repeatable on every request and requires no coordination beyond the provider's own key. [5](#0-4) 

---

### Recommendation

Add a `maxFee` parameter to `requestPriceUpdatesWithCallback`. Revert if `msg.value > maxFee` before the minimum-fee check, so users can bound their spend:

```solidity
function requestPriceUpdatesWithCallback(
    address provider,
    uint64 publishTime,
    bytes32[] calldata priceIds,
    uint32 callbackGasLimit,
    uint96 maxFee          // <-- new parameter
) external payable override returns (uint64 requestSequenceNumber) {
    if (msg.value > maxFee) revert ExceedsMaxFee();
    uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
    if (msg.value < requiredFee) revert InsufficientFee();
    ...
}
```

This mirrors the `maxCash` fix applied in the reference report and gives callers a hard upper bound on what they can be charged.

### Proof of Concept

```solidity
// 1. Provider registers with baseFee = 1 ether
echo.registerProvider(1 ether, 0, 0);

// 2. User queries fee: getFee returns 1 ether + pythFee
uint96 fee = echo.getFee(provider, gasLimit, priceIds);
// fee == 1 ether + pythFeeInWei

// 3. User prepares tx with 10% buffer
uint96 userValue = fee + 0.1 ether;

// 4. Provider frontruns: raises baseFee to 1.05 ether (still < userValue)
echo.setProviderFee(provider, 1.05 ether, 0, 0);

// 5. User's tx lands:
//    getFee now returns 1.05 ether + pythFee <= userValue → passes
//    req.fee = userValue - pythFeeInWei = 1.1 ether
//    Provider collects 1.1 ether instead of 1 ether
echo.requestPriceUpdatesWithCallback{value: userValue}(
    provider, publishTime, priceIds, gasLimit
);
// Provider gained an extra 0.1 ether the user did not intend to pay

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

    function setProviderFee(
        address provider,
        uint96 newBaseFeeInWei,
        uint96 newFeePerFeedInWei,
        uint96 newFeePerGasInWei
    ) external override {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );
        require(
            msg.sender == provider ||
                msg.sender == _state.providers[provider].feeManager,
            "Only provider or fee manager can invoke this method"
        );

        uint96 oldBaseFee = _state.providers[provider].baseFeeInWei;

        _state.providers[provider].feePerFeedInWei = newFeePerFeedInWei;
        _state.providers[provider].feePerGasInWei = newFeePerGasInWei;

            provider,
            oldBaseFee,
            oldFeePerFeed,
