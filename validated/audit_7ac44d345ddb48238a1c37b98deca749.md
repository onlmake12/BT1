### Title
EMA Price Staleness Check Panics on Future `publish_time` Due to Missing Underflow Guard - (File: `target_chains/fuel/contracts/pyth-contract/src/main.sw`)

---

### Summary

The Fuel Pyth contract's `ema_price_no_older_than` function performs an unchecked unsigned subtraction `current_time - price.publish_time`. If a stored EMA price has a `publish_time` greater than the current Fuel block timestamp (which is possible due to clock drift between Pythnet and Fuel), the subtraction underflows and the transaction panics, permanently blocking all callers of `ema_price()` and `ema_price_no_older_than()` until the Fuel clock catches up.

The sibling function `price_no_older_than` was already patched with an explicit saturating-subtraction guard and a comment explaining the fix, but `ema_price_no_older_than` was not updated consistently.

---

### Finding Description

In `target_chains/fuel/contracts/pyth-contract/src/main.sw`, the two staleness-check helpers are implemented asymmetrically:

**Vulnerable — `ema_price_no_older_than` (lines 311–320):**
```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,  // ← panics if publish_time > current_time
        PythError::OutdatedPrice,
    );
    price
}
```

**Fixed — `price_no_older_than` (lines 331–343):**
```sway
fn price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

The comment in `price_no_older_than` explicitly acknowledges the underflow risk and guards against it. The identical guard is absent in `ema_price_no_older_than`. [1](#0-0) [2](#0-1) 

The public ABI entry points `ema_price()` and `ema_price_no_older_than()` both delegate directly to the vulnerable private function:

```sway
fn ema_price(price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(valid_time_period(), price_feed_id)
}
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    ema_price_no_older_than(time_period, price_feed_id)
}
``` [3](#0-2) 

The `update_price_feeds` function stores any price update whose `publish_time` is strictly greater than the currently cached value, with no upper bound check against the current block timestamp: [4](#0-3) 

---

### Impact Explanation

Any protocol on Fuel that calls `ema_price()` or `ema_price_no_older_than()` will have its transaction revert for the entire window during which `publish_time > block.timestamp`. Because Pythnet (a Solana-based chain) and Fuel maintain independent clocks, a small positive clock skew on Pythnet's side is a normal operating condition, not an attack. The stored future `publish_time` blocks all EMA price consumers until Fuel's clock catches up, which can be tens of seconds to minutes depending on the skew.

---

### Likelihood Explanation

This is a normal operating condition, not an adversarial scenario. Pythnet timestamps are derived from Solana's clock, which can run slightly ahead of Fuel's block timestamp. Any relayer (unprivileged) submitting a fresh Wormhole-attested price update with a `publish_time` even 1 second ahead of Fuel's current block timestamp will trigger the panic. The condition is self-resolving once Fuel's clock advances, but the window of DoS is real and repeatable on every such update.

---

### Recommendation

Apply the same saturating-subtraction guard already present in `price_no_older_than` to `ema_price_no_older_than`:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

---

### Proof of Concept

1. Pythnet's clock is 30 seconds ahead of Fuel's block timestamp. `block.timestamp = T`.
2. A relayer calls `update_price_feeds` with a valid Wormhole-attested accumulator update where `publish_time = T + 30`.
3. The update passes all checks (no upper-bound timestamp validation in `update_price_feeds`) and is stored because `T + 30 > cached_publish_time`.
4. Any contract calls `ema_price(price_feed_id)` → `ema_price_no_older_than(valid_time_period, price_feed_id)` → `current_time - price.publish_time` = `T - (T+30)` = underflow → **panic/revert**.
5. All EMA price consumers are blocked for ~30 seconds until Fuel's clock reaches `T + 30`.
6. The next Pythnet update repeats the cycle. [1](#0-0) [5](#0-4)
