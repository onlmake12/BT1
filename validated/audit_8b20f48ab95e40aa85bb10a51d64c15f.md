### Title
Incorrect Ring-Buffer Index Assumption in `TxConfirmStat::estimate_median()` Corrupts `extra_count`, Producing Wrong Fee Rate Estimates — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

`TxConfirmStat::estimate_median()` reads `block_unconfirmed_txs[confirm_blocks - 1]` to obtain the "extra" unconfirmed-transaction count for a given confirmation target. However, `block_unconfirmed_txs` is a ring buffer whose first dimension is keyed by `entry_height % MAX_CONFIRM_BLOCKS`, not by confirmation-block count. The function therefore reads an arbitrary, semantically unrelated ring-buffer slot, producing a systematically wrong `extra_count` and, consequently, a wrong `confirm_rate` and wrong fee-rate estimate.

---

### Finding Description

`TxConfirmStat` maintains three parallel arrays, all of length `MAX_CONFIRM_BLOCKS = 1000`:

| Array | First-dimension key |
|---|---|
| `confirm_blocks_to_confirmed_txs` | confirmation-block count (0-indexed) |
| `confirm_blocks_to_failed_txs` | confirmation-block count (0-indexed) |
| `block_unconfirmed_txs` | `entry_height % MAX_CONFIRM_BLOCKS` (ring-buffer slot) | [1](#0-0) 

Transactions are inserted into the ring buffer using the modulo of their entry height: [2](#0-1) 

`move_track_window` also uses the modulo of the current block height to advance the ring buffer: [3](#0-2) 

Inside `estimate_median`, the first two arrays are correctly indexed by `confirm_blocks - 1` (a confirmation count). But `block_unconfirmed_txs` is indexed with the same value: [4](#0-3) 

`confirm_blocks - 1` is a confirmation-count integer (e.g., 2, 9, 14 for the 3-, 10-, 15-block targets used in tests). It is not a ring-buffer slot. The correct slot for "transactions that entered the mempool `confirm_blocks` blocks ago" would be `(current_height - confirm_blocks) % MAX_CONFIRM_BLOCKS`, but `estimate_median` has no access to `current_height` and never computes this. The result is that `extra_count` accumulates counts from whichever ring-buffer slot happens to share the same numeric value as `confirm_blocks - 1`, which is semantically unrelated to the confirmation target being evaluated.

The same wrong index is used when subtracting `extra_count` during the sliding-window shrink step: [5](#0-4) 

---

### Impact Explanation

`extra_count` feeds directly into the confirmation-rate denominator: [6](#0-5) 

- If the wrong slot is **inflated** (many unconfirmed txs happen to sit at that ring-buffer position), `confirm_rate` is artificially depressed, causing the estimator to recommend a **higher fee rate than necessary** — users overpay.
- If the wrong slot is **deflated** (near zero), `confirm_rate` is artificially elevated, causing the estimator to recommend a **lower fee rate than warranted** — transactions may be stuck.

The corrupted estimate is returned directly to any RPC caller of `estimate_fee_rate`: [7](#0-6) 

---

### Likelihood Explanation

The `ConfirmationFraction` algorithm must be explicitly configured by the node operator (the default is `Dummy`): [8](#0-7) 

Once configured, the bug is **always active** — every call to `estimate_fee_rate` with this algorithm goes through the broken `estimate_median` path. No special attacker action is required; the wrong index is structural. A transaction submitter who observes that the node uses this estimator can also deliberately submit transactions at block heights where `height % 1000 == target_confirm_blocks - 1` to inflate `extra_count` for a specific target, biasing the estimate upward for other users.

---

### Recommendation

Pass `current_height` (i.e., `best_height`) into `estimate_median` and compute the correct ring-buffer slot:

```rust
let ring_slot = (current_height.saturating_sub(confirm_blocks as u64)
    % self.block_unconfirmed_txs.len() as u64) as usize;
extra_count += &self.block_unconfirmed_txs[ring_slot][bucket_index];
```

Alternatively, restructure `block_unconfirmed_txs` to be indexed by confirmation count (matching the other two arrays), updating it during `move_track_window` by shifting counts rather than using a modulo ring buffer.

---

### Proof of Concept

1. Configure the CKB node with `fee_estimator_algo = ConfirmationFraction`.
2. Let the chain advance to height H. Submit 100 transactions with fee rate F at height H where `H % 1000 == 9` (i.e., `confirm_blocks - 1` for the 10-block target). Leave them unconfirmed.
3. Call `estimate_fee_rate` with `estimate_mode = "low_priority"` (10-block target).
4. `estimate_median(10, ...)` reads `block_unconfirmed_txs[9][bucket_index]`, which contains the 100 transactions from step 2.
5. `extra_count` is inflated by 100, depressing `confirm_rate` below `DEFAULT_MIN_CONFIRM_RATE = 0.85`, causing the estimator to walk to a higher fee-rate bucket and return an inflated fee rate.
6. Repeat at height H+1000 (same ring-buffer slot) to sustain the manipulation indefinitely. [9](#0-8)

                confirm_blocks,

        let mut txs_count = 0f64;

        let mut extra_count = 0;

        let mut best_bucket_end = 0;
        let mut start_bucket_index = 0;

        // try find enough sample data from buckets
        for (bucket_index, stat) in self.bucket_stats.iter().enumerate() {
            confirmed_txs += self.confirm_blocks_to_confirmed_txs[confirm_blocks - 1][bucket_index];

            extra_count += &self.block_unconfirmed_txs[confirm_blocks - 1][bucket_index];
