### Title
No Payload Timestamp Validation in `verifyUpdate` Allows Replay of Arbitrarily Stale Lazer Price Updates - (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` verifies only the ECDSA signature and signer key expiry. It does **not** validate the timestamp embedded inside the signed payload. Any caller can submit a legitimately signed but arbitrarily old Lazer price update and receive a "verified" payload back, with no on-chain enforcement that the price data is recent. This is the direct analog of reading `slot0` without a TWAP: the contract returns a spot-verified but potentially stale price with no manipulation or staleness resistance.

---

### Finding Description

`PythLazer.verifyUpdate()` performs three checks:

1. EVM format magic bytes match.
2. ECDSA signature recovers to a non-zero address.
3. The recovered signer address has not yet expired (`block.timestamp < trustedSignerToExpiresAtMapping[signer]`). [1](#0-0) 

The payload returned by `verifyUpdate` contains a `timestamp` field (in microseconds) that is parsed by `PythLazerLib.parsePayloadHeader`: [2](#0-1) 

However, `verifyUpdate` never reads or validates this timestamp against `block.timestamp`. A signed Lazer update from hours or days ago — with a signer key that is still within its `expiresAt` window — passes all checks identically to a fresh update. The contract returns the stale payload as if it were current.

The official integration documentation shows the timestamp is available after parsing:

```solidity
(uint64 _timestamp, Channel channel, uint8 feedsLen, uint16 pos) = parsePayloadHeader(payload);
``` [3](#0-2) 

But the documentation example only checks `_timestamp > timestamp` (monotonicity against a stored value), not `_timestamp` against `block.timestamp`. There is no on-chain enforcement of freshness at the `verifyUpdate` layer.

By contrast, the Pyth Core oracle's `getPriceNoOlderThan` enforces freshness at the contract level: [4](#0-3) 

And the Scheduler's `getPricesNoOlderThan` enforces freshness via `priceLastUpdatedAt`: [5](#0-4) 

`PythLazer.verifyUpdate` provides no equivalent protection.

---

### Impact Explanation

An unprivileged attacker who has observed any past Lazer signed update (e.g., from a public mempool, a historical archive, or their own prior transaction) can replay it at any future time while the signer key remains valid. The `verifyUpdate` call succeeds and returns the stale payload. Any consuming protocol that relies on `verifyUpdate` as its security boundary — without independently enforcing a maximum age on `_timestamp` — will consume a stale price.

Concrete attack scenario:
- BTC/USD Lazer price was $50,000 at time T₀. Attacker saves the signed update.
- At time T₁ (hours later), BTC/USD is $60,000.
- Attacker calls `verifyUpdate` with the T₀ update. It passes.
- Attacker uses the returned payload (price = $50,000) to open a leveraged long position in a consuming DeFi protocol at a $10,000 discount.
- Attacker immediately closes the position at the real price, extracting value from the protocol.

The signer key `expiresAt` is set far in the future (e.g., `3000000000000000` in the test), meaning the replay window is effectively unbounded. [6](#0-5) 

---

### Likelihood Explanation

- **Entry path is fully unprivileged**: any EOA can call `verifyUpdate` with any bytes and pay the `verification_fee` (currently 1 wei).
- **Signed updates are publicly observable**: Lazer updates are submitted on-chain by relayers; historical calldata is trivially retrievable.
- **Signer keys have long validity windows**: the `expiresAt` mechanism is designed for key rotation, not per-update freshness.
- **No documentation warning**: the integration guide does not clearly instruct consumers to enforce `_timestamp` against `block.timestamp`, making accidental misuse highly likely.

---

### Recommendation

Add a `maxAge` parameter (or a hardcoded constant) to `verifyUpdate` that rejects payloads whose embedded timestamp is older than `block.timestamp - maxAge`. The timestamp is already present in the payload bytes; it must be parsed and validated before returning the payload to the caller.

```solidity
// After extracting payload:
uint64 payloadTimestamp = uint64(bytes8(payload[4:12])); // microseconds
require(
    block.timestamp * 1_000_000 <= payloadTimestamp + MAX_AGE_MICROSECONDS,
    "stale price update"
);
```

Alternatively, expose a `verifyUpdateNoOlderThan(bytes calldata update, uint64 maxAgeSeconds)` variant that enforces freshness, mirroring the pattern used by `getPriceNoOlderThan` in the Core oracle. [7](#0-6) 

---

### Proof of Concept

1. Deploy `PythLazer` and register a trusted signer with `expiresAt = block.timestamp + 365 days`.
2. At `t = 0`, obtain a valid signed Lazer update with payload timestamp `t₀` (price = P₀).
3. Advance time by 1 hour (`vm.warp(block.timestamp + 3600)`).
4. Call `pythLazer.verifyUpdate{value: fee}(oldUpdate)`.
5. Observe: call succeeds, returns payload with timestamp `t₀` and price `P₀`.
6. Parse `_timestamp` from the returned payload — it is 1 hour in the past.
7. No revert occurs; the consuming contract receives a stale price as if it were current.

The existing test `test_verify` already demonstrates that a fixed historical update passes `verifyUpdate` without any timestamp check: [8](#0-7)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
