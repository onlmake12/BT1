### Title
Silently Ignored `GetBlockTransactions` Send Failure Causes Stalled Compact Block Reconstruction — (File: `sync/src/relayer/block_transactions_process.rs`)

### Summary
In `BlockTransactionsProcess::execute()`, when compact block reconstruction fails due to missing transactions or a short-ID collision, the node sends a `GetBlockTransactions` request back to the peer. The return value of `async_send_message_to` is explicitly discarded with `let _ignore = ...`, and the node's pending state (`expected_transaction_indexes`, `expected_uncle_indexes`) is unconditionally updated immediately after — regardless of whether the message was actually delivered. This is the direct CKB analog of the ERC-20 unchecked-return-value pattern: a critical operation's failure is silently swallowed while dependent state is mutated as if the operation succeeded.

The `async_send_message_to` helper (defined in `sync/src/utils.rs`) returns a `Status` value that encodes success or a `StatusCode::Network` error: [2](#0-1) 

The caller at line 174 names the binding `_ignore`, explicitly opting out of any error handling. Immediately after, the two `mem::replace` calls overwrite the peer's tracked `expected_transaction_indexes` and `expected_uncle_indexes` with the new missing-transaction lists. These are the lists the node will use to validate the *next* `BlockTransactions` response from this peer.

If the send fails (e.g., the underlying P2P channel is congested or broken), the peer never receives the `GetBlockTransactions` request, but the node has already advanced its expected-index state. The node is now waiting for a response to a request that was never delivered.

Compare with the analogous path in `compact_block_process.rs`, which at least logs a warning when the send fails: [3](#0-2) 

The `block_transactions_process.rs` path does not even log.

### Impact Explanation

When the send silently fails:

1. The node has updated `expected_transaction_indexes` / `expected_uncle_indexes` for this peer to the new missing set.
2. The peer never receives the `GetBlockTransactions` request.
3. The pending compact block entry for this peer is now permanently stalled — the peer will not spontaneously re-send the transactions, and the node will not re-request them because it believes a request is already in flight.
4. The compact block reconstruction for this peer is stuck until the pending entry is evicted by a timeout or a competing peer delivers the full block.

This is a **block propagation liveness issue**: a relay peer that experiences a transient send failure during compact block reconstruction will be unable to contribute to block delivery for the duration of the pending-block timeout. Under adversarial network conditions (e.g., a peer that deliberately causes send failures at this exact point), this can be used to slow block propagation to a targeted node.

### Likelihood Explanation

The code path is reachable by any unprivileged relay peer: send a compact block whose short IDs do not match the receiver's mempool, triggering `ReconstructionResult::Missing` or `ReconstructionResult::Collided`. Network send errors are uncommon under normal conditions but are realistic under congestion, connection churn, or deliberate manipulation. The missing error-check is not guarded by any retry or fallback.

### Recommendation

Check the return value of `async_send_message_to` before updating the expected-index state. If the send fails, either:
- Do **not** replace `expected_transaction_indexes` / `expected_uncle_indexes` (keep the old state so a retry can use the correct indexes), or
- Log the error and remove the peer's entry from `pending_compact_blocks` so the node does not wait indefinitely for a response that will never arrive.

The pattern used in `compact_block_process.rs` (log a warning on failure) is a minimal improvement, but the correct fix is to avoid mutating state when the triggering operation failed.

### Proof of Concept

1. Attacker peer connects and sends a `CompactBlock` message whose short IDs do not match the receiver's mempool.
2. Receiver enters `BlockTransactionsProcess::execute()`, reconstruction returns `ReconstructionResult::Missing`.
3. Receiver calls `async_send_message_to` to send `GetBlockTransactions` — attacker's P2P channel is closed/congested at this moment, causing the send to return `StatusCode::Network`.
4. Return value is discarded (`let _ignore = ...`).
5. `expected_transaction_indexes` is replaced with the new missing list.
6. Attacker never receives the request; receiver waits indefinitely for a `BlockTransactions` reply that will never come.
7. The compact block reconstruction for this peer stalls until the pending-block timeout fires. [4](#0-3)

pub(crate) async fn async_send_message<Message: Entity>(
    protocol_id: ProtocolId,
    nc: &Arc<dyn CKBProtocolContext + Sync>,
    peer_index: PeerIndex,
    message: &Message,
) -> Status {
    // ignore Error return, only happens on shutdown case
    if let Err(err) = nc
        .async_send_message(protocol_id, peer_index, message.as_bytes())
        .await
    {
        let name = message_name(protocol_id, message);
        let error_message = format!("nc.send_message {name}, error: {err:?}");
        ckb_logger::error!("{}", error_message);
        return StatusCode::Network.with_context(error_message);
    }

    let bytes = message.as_bytes().len() as u64;
    let item_name = item_name(protocol_id, message);
    let protocol_name = protocol_name(protocol_id);
    metric_ckb_message_bytes(
        MetricDirection::Out,
        &protocol_name,
        &item_name,
        None,
        bytes,
    );

    Status::ok()
}
