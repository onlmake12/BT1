Audit Report

## Title
Incorrect `timeout_count` Accumulation in `DownloadScheduler::decrease` Causes Premature Peer Disconnection — (File: `sync/src/types/mod.rs`)

## Summary
`DownloadScheduler::decrease` overwrites `self.timeout_count` with `self.task_count.saturating_add(num)` instead of `self.timeout_count.saturating_add(num)`. Because `task_count` is initialized to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (a value well above 2), the guard `timeout_count > 2` is always satisfied, causing `task_count` to be decremented on every slow block response. This leads to premature peer disconnection and potential IBD stall.

## Finding Description
In `sync/src/types/mod.rs` at line 522, the `decrease` function reads:

```rust
self.timeout_count = self.task_count.saturating_add(num);
```

The intended accumulator is `self.timeout_count`, not `self.task_count`. Because `task_count` is initialized to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (confirmed at line 493, a constant larger than 2), the expression always produces a value greater than 2. The guard at line 523 (`if self.timeout_count > 2`) is therefore always true, `task_count` is decremented unconditionally at line 524, and `timeout_count` is reset to 0 at line 525. The accumulator never accumulates — it is overwritten with the wrong variable on every invocation.

The call chain is:
1. A peer delivers a block → `remove_by_block` (line 785)
2. `time_analyzer.push_time(elapsed)` (line 798) returns `NormalToUpper` or `UpperToMax`
3. `set.decrease(1)` or `set.decrease(2)` is called (lines 803, 808), gated by `should_punish` (line 786: `download_schedulers.len() > protect_num`)
4. Due to the bug, `task_count` is decremented on every such call
5. When `task_count` reaches 0, `prune()` at lines 692–700 inserts the peer into `disconnect_list` and removes it

The `Default` impl (lines 489–496) confirms `timeout_count` starts at 0, making the intended design a two-strike accumulator before penalizing — a design that is entirely bypassed by the bug.

## Impact Explanation
A node running IBD with `adjustment = true` and peer count exceeding `protect_num` will have legitimate peers' `task_count` decremented on every moderately slow block response. Once `task_count` reaches 0, the peer is forcibly disconnected. If all connected peers exhibit moderate latency (common in geographically distributed networks), the node disconnects all sync peers and stalls block download entirely. This constitutes a condition where a CKB node can be rendered unable to sync, matching the **High** impact class: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation
The `NormalToUpper` and `UpperToMax` quantiles cover the majority of real-world network conditions under load. The `should_punish` guard requires only that the peer count exceed `protect_num`, which is the normal operating state for a well-connected node. No privileged access is required — any peer that consistently delivers blocks in the 1250–1500 ms range (cross-continental latency is typical) will trigger this path. The bug is deterministic and repeatable on every qualifying block delivery.

## Recommendation
Fix `decrease` to accumulate `timeout_count` against itself:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // was: self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

This restores the intended two-strike tolerance before penalizing a peer's download quota.

## Proof of Concept
1. Connect to a CKB node as an unprivileged sync peer, ensuring total peer count exceeds `protect_num`.
2. Respond to `GetBlocks` requests with block delivery times consistently in the `NormalToUpper` range (1250–1500 ms).
3. Each delivered block triggers `remove_by_block` → `time_analyzer.push_time` → `TimeQuantile::NormalToUpper` → `set.decrease(1)`.
4. Observe that `task_count` decrements on every response (not every two) due to the bug.
5. After `INIT_BLOCKS_IN_TRANSIT_PER_PEER` such responses, `task_count` reaches 0.
6. The next `prune()` call disconnects this peer.
7. Repeat with all connected peers exhibiting similar latency to stall IBD on the victim node.

A unit test can directly instantiate `DownloadScheduler::default()`, call `decrease(1)` twice, and assert that `task_count` has only decremented once (the correct behavior) — the bug causes it to decrement twice. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
