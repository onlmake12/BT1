### Title
Push-Based ETH Refund in `verifyUpdate` Causes Permanent DoS for Contract Callers - (`File: lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.verifyUpdate` uses `payable(msg.sender).transfer(...)` to refund excess ETH to the caller. If `msg.sender` is a smart contract without a `receive()` or `fallback()` function (or one whose fallback consumes more than 2300 gas), the `transfer()` call reverts, causing the entire `verifyUpdate` call to fail. This permanently blocks any such contract from consuming Pyth Lazer price feeds unless it sends exactly the right fee amount.

### Finding Description

In `lazer/contracts/evm/src/PythLazer.sol`, the `verifyUpdate` function collects a `verification_fee` and refunds any excess ETH to the caller using the deprecated `.transfer()` pattern:

```solidity
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    // Require fee and refund excess
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
        payable(msg.sender).transfer(msg.value - verification_fee);  // <-- push refund
    }
    ...
}
```

`address.transfer()` forwards only 2300 gas. If `msg.sender` is a contract that:
- Has no `receive()` or `fallback()` function, or
- Has a `receive()`/`fallback()` that consumes more than 2300 gas (e.g., emits an event, writes storage, or calls another contract),

then the `transfer()` call reverts, and the entire `verifyUpdate` call reverts. The caller's ETH is returned but the price update is not verified.

The `verification_fee` is initialized to `1 wei`. Any caller sending `2 wei` or more triggers the refund path. In practice, DeFi integrators routinely add a small buffer to fee computations (e.g., `fee * 110 / 100`) to guard against fee changes, making this a common code path.

### Impact Explanation

Any smart contract that:
1. Calls `verifyUpdate` with `msg.value > verification_fee`, AND
2. Does not implement a `receive()` or `fallback()` function accepting ETH

will have every `verifyUpdate` call revert. The contract is permanently unable to consume Pyth Lazer price feeds via `verifyUpdate` unless it computes the exact fee on-chain before every call. This is a denial-of-service against Lazer consumers.

### Likelihood Explanation

- The `verification_fee` starts at `1 wei`, so any contract sending `≥ 2 wei` hits the refund branch.
- Many DeFi contracts (e.g., pure logic contracts, proxy contracts, multisigs) do not implement `receive()`.
- Integrators following the documented pattern `verifyUpdate{ value: verification_fee }(update)` are safe only if they query `verification_fee` on-chain each call; any hardcoded or buffered value triggers the issue.
- The fee can be changed by the owner, meaning a contract that worked before a fee change may break after one.

### Recommendation

Replace the push-based `transfer()` with a pull-based approach, or use `call{value: ...}("")` with a success check and proper reentrancy guard:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Refund failed");
}
```

Alternatively, accumulate excess fees in a per-caller mapping and expose a `withdrawExcess()` function (pull pattern), which is the fix analogous to what Plaza Finance applied.
