### Title
Malicious Relay Peer Can Permanently Stall Block Reconstruction with Zero Cost via Unanswered `GetBlockTransactions` — No Timeout Eviction of `pending_compact_blocks` — (File: `sync/src/relayer/compact_block_process.rs`)

---

### Summary

When a CKB node receives a `CompactBlock` message it cannot fully reconstruct (missing transactions), it stores the compact block in the shared `pending_compact_blocks` map and sends a `GetBlockTransactions` request back to the announcing peer. If that peer never responds, the entry remains in `pending_compact_blocks` with no timeout-based eviction. The only cleanup paths are (a) successful block acceptance and (b) a `retain` call that removes entries from earlier epochs. A malicious relay peer can exploit this asymmetry — committing zero resources after the initial compact block send — to permanently occupy a slot in the victim node's `pending_compact_blocks` map and delay the sync protocol's fallback download path by at least 2 seconds per affected block hash.

---

### Finding Description

**Root cause — no timeout eviction of `pending_compact_blocks`**

`PendingCompactBlockMap` is defined as a plain `HashMap` keyed by block hash, storing the compact block, a per-peer missing-index map, and an insertion timestamp: [1](#0-0) 

Entries are inserted in `missing_or_collided_post_process` when reconstruction fails: [2](#0-1) 

The only removal paths are:

1. **Successful block acceptance** — `pending_compact_blocks.remove(&block_hash)` followed by a `retain` that drops entries from epochs strictly older than the accepted block: [3](#0-2) 

2. **Successful reconstruction from `BlockTransactions`** — `pending.remove()`: [4](#0-3) 

There is **no periodic timer** that walks `pending_compact_blocks` and evicts entries whose insertion timestamp has aged past a threshold. The timestamp field stored in each entry is only read by `compare_with_pending_compact`, which gates the sync fallback: [5](#0-4) 

This function returns `true` (allowing sync to proceed) only after 2 000 ms have elapsed since the compact block was inserted. Until that window expires, the sync protocol's `BlockFetcher` is suppressed for that block hash. [6](#0-5) 

**Attack flow**

1. Attacker connects as a relay peer and sends a valid `CompactBlock` for a real (or fork) block whose transactions are not in the victim's tx-pool.
2. Victim cannot reconstruct the block → calls `missing_or_collided_post_process` → inserts entry into `pending_compact_blocks` → sends `GetBlockTransactions` to the attacker.
3. Attacker silently drops the `GetBlockTransactions` message. Cost to attacker: one compact block message sent; no further action required.
4. Victim's entry stays in `pending_compact_blocks` indefinitely. The `compare_with_pending_compact` gate suppresses the sync fallback for 2 seconds. After 2 seconds the sync path unblocks, but the stale entry is never evicted — it persists until a new block at the same or later epoch is accepted and the `retain` sweep runs.
5. For fork-tip blocks (blocks that are never accepted on the canonical chain), the entry **never** gets cleaned up by the `retain` sweep, because that sweep only fires on canonical block acceptance and only removes entries from strictly earlier epochs.

---

### Impact Explanation

- **Sync delay**: Every block hash for which a malicious peer holds a pending compact block entry causes a mandatory 2-second suppression of the sync download path for that hash. With multiple concurrent malicious peers or multiple block hashes, this compounds.
- **Memory bloat**: Each stale `pending_compact_blocks` entry holds a full `CompactBlock` struct (header + short IDs + prefilled transactions, potentially hundreds of KB). Fork-tip entries are never reclaimed. An attacker with modest mining power (enough to produce valid headers) can accumulate entries across the current epoch without bound until the next canonical block acceptance triggers the `retain` sweep.
- **Block propagation stall**: The primary relay path for new blocks is compact block relay. Stalling reconstruction for the canonical tip block delays the node's view of the chain tip, affecting mining and transaction confirmation latency.

---

### Likelihood Explanation

The attacker must be a connected P2P relay peer and must supply a PoW-valid block header (the `HeaderVerifier` is called in `contextual_check`): [7](#0-6) 

This requires some mining capability. However:
- The attacker can reuse any valid block header they have already mined or observed; they do not need to mine a new block per attack.
- A miner with even a small fraction of hash rate produces valid headers continuously.
- The attacker's ongoing cost after sending the compact block is **zero** — they simply do not respond to `GetBlockTransactions`. This is the direct analog of the external report's "router responds with amount=0 or simply doesn't respond."
- No ban or penalty is applied to a peer that fails to answer `GetBlockTransactions`; the node only disconnects peers that time out on `inflight_blocks` (the sync download scheduler), not on the relay compact-block path.

---

### Recommendation

1. **Add a periodic eviction timer** for `pending_compact_blocks`. Walk the map on a fixed interval (e.g., every 30 seconds) and remove any entry whose stored timestamp is older than `BLOCK_DOWNLOAD_TIMEOUT` (currently 30 s): [8](#0-7) 

2. **Apply a per-peer penalty** (bandwidth reduction or disconnect) when a peer that has a pending compact block entry fails to respond to `GetBlockTransactions` within the timeout window, analogous to how `InflightBlocks::prune` penalises slow block-download peers: [9](#0-8) 

3. **Cap the size of `pending_compact_blocks`** with an LRU or bounded map to prevent unbounded memory growth independent of the epoch-based `retain` sweep.

---

    pub fn prune(&mut self, tip: BlockNumber) -> HashSet<PeerIndex> {
        let now = unix_time_as_millis();
        let mut disconnect_list = HashSet::new();
        // Since statistics are currently disturbed by the processing block time, when the number
        // of transactions increases, the node will be accidentally evicted.
        //
        // Especially on machines with poor CPU performance, the node connection will be frequently
        // disconnected due to statistics.
        //
        // In order to protect the decentralization of the network and ensure the survival of low-performance
        // nodes, the penalty mechanism will be closed when the number of download nodes is less than the number of protected nodes
        let should_punish = self.download_schedulers.len() > self.protect_num;
        let adjustment = self.adjustment;

        let trace = &mut self.trace_number;
        let download_schedulers = &mut self.download_schedulers;
        let states = &mut self.inflight_states;

        let mut remove_key = Vec::new();
        // Since this is a btreemap, with the data already sorted,
        // we don't have to worry about missing points, and we don't need to
        // iterate through all the data each time, just check within tip + 20,
        // with the checkpoint marking possible blocking points, it's enough
        let end = tip + 20;
        for (key, value) in states.iter() {
            if key.number > end {
                break;
            }
            if value.timestamp + BLOCK_DOWNLOAD_TIMEOUT < now {
                if let Some(set) = download_schedulers.get_mut(&value.peer) {
                    set.hashes.remove(key);
                    if should_punish && adjustment {
                        set.punish(2);
                    }
                };
                if !trace.is_empty() {
                    trace.remove(key);
                }
                remove_key.push(key.clone());
                debug!(
                    "prune: remove InflightState: remove {}-{} from {}",
                    key.number, key.hash, value.peer
                );

                if let Some(metrics) = ckb_metrics::handle() {
                    metrics.ckb_inflight_timeout_count.inc();
                }
            }
        }

        for key in remove_key {
            states.remove(&key);
        }

        download_schedulers.retain(|k, v| {
            // task number zero means this peer's response is very slow
            if v.task_count == 0 {
                disconnect_list.insert(*k);
                false
            } else {
                true
            }
        });
