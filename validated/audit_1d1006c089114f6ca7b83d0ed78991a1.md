### Title
Unvalidated `providerToCredit` Address in `executeCallback` Allows Permanent Locking of Provider Fees - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, the `executeCallback` function accepts a caller-supplied `providerToCredit` address with no zero-address or registration check. After the exclusivity period expires, any unprivileged caller can invoke `executeCallback` with `providerToCredit = address(0)`, crediting provider fees to `_state.providers[address(0)]`. Because `Echo` has no direct `withdraw()` for providers and `withdrawAsFeeManager` requires `msg.sender == feeManager`, fees credited to `address(0)` are permanently locked.

### Finding Description

`Echo.sol`'s `executeCallback` enforces `providerToCredit == req.provider` only during the exclusivity window: [1](#0-0) 

After `req.publishTime + _state.exclusivityPeriodSeconds` (default 15 seconds per tests), the check is skipped entirely and any caller may pass any address as `providerToCredit`: [2](#0-1) 

Fees are unconditionally credited to `_state.providers[providerToCredit].accruedFeesInWei`. There is no `require(providerToCredit != address(0))` and no check that `providerToCredit` is a registered provider.

The only withdrawal path for provider-accrued fees in `Echo` is `withdrawAsFeeManager`: [3](#0-2) 

This requires `msg.sender == _state.providers[provider].feeManager`. For `address(0)`, `feeManager` is `address(0)` (default zero-initialized per `EchoState`): [4](#0-3) 

Since `msg.sender` can never equal `address(0)`, fees credited to `address(0)` are permanently irrecoverable. There is no `withdraw()` function for providers in `Echo` (unlike `Entropy.sol`): [5](#0-4) 

### Impact Explanation

An attacker can permanently lock provider fees for any pending request. After the 15-second exclusivity period, the attacker front-runs the legitimate provider's `executeCallback` call (using the same publicly visible `updateData` from the mempool) with `providerToCredit = address(0)`. The request is cleared, the provider cannot retry, and the fees credited to `address(0)` are permanently locked in the contract. This is a direct loss of provider funds with no recovery path.

### Likelihood Explanation

The exclusivity period is only 15 seconds. On chains with public mempools (Ethereum mainnet, most L2s), an attacker can observe the provider's pending `executeCallback` transaction and front-run it after the exclusivity window. The attacker needs valid `updateData` — which is already present in the provider's pending transaction. The cost to the attacker is only gas. This is a realistic, low-cost griefing attack.

### Recommendation

Add a zero-address and registration check on `providerToCredit` in `executeCallback`:

```solidity
require(providerToCredit != address(0), "providerToCredit is zero address");
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit not registered"
);
```

Additionally, consider adding a direct `withdraw()` function for providers (analogous to `Entropy.sol`) so that fees are never permanently locked even if credited to an unregistered address.

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback`, paying fees. Request is stored with `req.provider = legitimateProvider`.
2. `legitimateProvider` prepares `executeCallback(legitimateProvider, seqNum, updateData, priceIds)`.
3. Attacker observes the pending transaction. After `req.publishTime + 15 seconds`, the exclusivity check is skipped.
4. Attacker submits `executeCallback(address(0), seqNum, updateData, priceIds)` with higher gas.
5. `_state.providers[address(0)].accruedFeesInWei` is incremented. Request is cleared.
6. `legitimateProvider` transaction reverts with `NoSuchRequest`.
7. Attacker (or anyone) calls `withdrawAsFeeManager(address(0), amount)` → reverts: `msg.sender != address(0)`.
8. Fees are permanently locked. [6](#0-5)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
