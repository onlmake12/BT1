Audit Report

## Title
Unbounded Per-Peer Orphan Insertion Allows Cheap DoS Eviction of Legitimate Orphan Transactions - (File: tx-pool/src/component/orphan.rs)

## Summary

`OrphanPool` enforces a global cap of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` with no per-peer contribution limit. A single malicious relay peer can fill the pool with 100 syntactically valid transactions referencing non-existent `OutPoint`s — no signatures required, as input resolution fails before script execution — causing legitimate orphan transactions to be randomly evicted via `limit_size()`. The attacker can sustain saturation indefinitely at negligible cost, permanently preventing automatic orphan resolution for legitimate users.

## Finding Description

**Root cause:** `add_orphan_tx()` in `tx-pool/src/component/orphan.rs` (lines 134–159) performs only a duplicate-key check before inserting any orphan from any peer. There is no per-peer quota or rate limit.

```rust
pub fn add_orphan_tx(&mut self, tx: TransactionView, peer: PeerIndex, declared_cycle: Cycle) -> Vec<Byte32> {
    if self.entries.contains_key(&tx.proposal_short_id()) {
        return vec![];
    }
    // inserts unconditionally, then calls limit_size()
```

`limit_size()` (lines 119–125) evicts by calling `self.entries.keys().next()` on a `HashMap`, which is effectively random due to Rust's hash randomization:

```rust
while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
    let id = self.entries.keys().next().cloned().expect("bound checked");
    if let Some(entry) = self.remove_orphan_tx(&id) { ... }
}
```

**Exploit path:**

1. Attacker connects as a relay peer and sends 100 transactions with valid molecule structure but inputs referencing random non-existent `OutPoint`s.
2. Each transaction passes `non_contextual_verify()` (`tx-pool/src/util.rs` lines 56–83), which only checks structure, size, and cellbase status — not input existence.
3. Input resolution fails with `OutPointError::Unknown`, which `is_missing_input()` (`util.rs` line 150–152) matches: `matches!(reject, Reject::Resolve(out_point_err) if out_point_err.is_unknown())`.
4. `after_process()` (`process.rs` lines 507–512) calls `add_orphan(tx, peer, declared_cycle)` — **no banning occurs** because `ban_malformed` is only triggered by `reject.is_malformed_tx()` (line 514), a distinct error type.
5. Pool reaches 100 entries. A legitimate orphan arrives → pool = 101 → `limit_size()` randomly evicts one entry.
6. Attacker immediately re-relays any evicted malicious orphan. Pool stays saturated at 100 attacker entries.
7. `process_orphan_tx()` (`process.rs` lines 591–596) calls `find_orphan_by_previous()` → `find_by_previous()` (`orphan.rs` lines 161–167) → legitimate child not found → automatic resolution chain broken.

**Why existing checks fail:** The only guard is the global cap enforced by `limit_size()`, which evicts randomly without regard to peer origin. The `Entry` struct stores `peer: PeerIndex` (line 23) but this field is never consulted during insertion or eviction decisions.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The attack disrupts orphan transaction propagation at negligible cost. Orphan resolution is the mechanism by which child transactions are automatically submitted when their parents arrive. Sustained saturation of the orphan pool means legitimate child transactions are silently dropped and never resolved automatically. Applied concurrently to multiple nodes, this degrades network-wide transaction propagation, as orphan chains cannot be resolved across the network without manual user resubmission.

## Likelihood Explanation

**High.** Requirements: one P2P connection (any unprivileged peer), 100 molecule-encoded `Transaction` structs with arbitrary non-existent `OutPoint` inputs (no CKB ownership, no valid signatures, no PoW), and a trivially automated loop to re-relay evicted transactions. The pool limit of 100 makes saturation achievable in a single message batch given `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`. The attacker is never banned because missing-input errors do not trigger `ban_malformed`.

## Recommendation

Track per-peer orphan contribution counts and reject insertions from peers exceeding a quota. Since `Entry` already stores `peer: PeerIndex`, the fix requires only a count check before insertion in `add_orphan_tx`:

```rust
// Before inserting, enforce per-peer quota:
let peer_count = self.entries.values().filter(|e| e.peer == peer).count();
if peer_count >= MAX_ORPHAN_PER_PEER {  // e.g., DEFAULT_MAX_ORPHAN_TRANSACTIONS / max_peers
    return vec![];
}
```

Additionally, consider replacing random eviction with a policy that preferentially evicts entries from the peer with the highest contribution count.
