Audit Report

## Title
Spoofed `from` Field Bypasses `forward_rate_limiter` and Enables `pending_delivered` Poisoning with Unbounded `try_nat_traversal` Resource Exhaustion — (`File: network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
In CKB's hole-punching protocol, the `from` field in `ConnectionRequest`, `ConnectionRequestDelivered`, and `ConnectionSync` messages is deserialized directly from the attacker-controlled message payload with no binding to the actual transport-layer session identity. The `forward_rate_limiter` is keyed on this attacker-controlled `content.from`, making it trivially bypassable by rotating fresh random peer IDs. Additionally, a two-step attack using `ConnectionRequest` followed by `ConnectionSync` with a matching spoofed `from` causes the victim to spawn unbounded `try_nat_traversal` tasks that initiate real outbound TCP connections to arbitrary attacker-specified addresses, enabling resource exhaustion and node crash.

## Finding Description

**Root cause — `from` parsed from payload with no session binding:**

In `connection_request.rs`, `from` is deserialized from the message body:

```rust
let from = PeerId::from_bytes(value.from().raw_data().to_vec()).map_err(|_| {
    StatusCode::InvalidFromPeerId.with_context("the from peer id is invalid")
})?;
``` [1](#0-0) 

The actual sending session's `PeerIndex` (`self.peer`) is available and used for the per-connection `rate_limiter`, but is never compared against `content.from`. There is no assertion that `content.from` equals the peer ID of the session that delivered the message. [2](#0-1) 

**Impact 1 — Forward rate-limiter bypass:**

The `forward_rate_limiter` is configured at 1 req/sec per unique key: [3](#0-2) 

It is keyed on `(content.from, content.to, self.msg_item_id)`: [4](#0-3) 

Because `content.from` is fully attacker-controlled, an attacker rotates a fresh random `from` value in every message, producing a unique key each time and making the `forward_rate_limiter` completely ineffective. The same bypass applies identically in `ConnectionRequestDeliveredProcess`: [5](#0-4) 

And in `ConnectionSyncProcess`: [6](#0-5) 

The per-connection `rate_limiter` (keyed on `(session_id, msg.item_id())`, 30 req/sec) still applies: [7](#0-6) 

But the `forward_rate_limiter` — the only throttle on relay amplification — is fully neutralized. With sqrt(N) gossip fanout per forwarded message, a single attacker connection drives 30 unique-keyed `ConnectionRequest` messages per second, each forwarded to sqrt(N) relay peers without rate limiting. [8](#0-7) 

**Impact 2 — `pending_delivered` poisoning / resource exhaustion:**

When the victim node is the `to` target of a `ConnectionRequest`, `respond_delivered` stores the message's `listen_addrs` (attacker-controlled, filtered only to TCP/IPv4/IPv6) under the spoofed `from_peer_id`: [9](#0-8) 

The existing guard that checks for a recent entry under `from_peer_id` is trivially bypassed by using a fresh random `from` for each message: [10](#0-9) 

When a `ConnectionSync` subsequently arrives with `from` set to the same spoofed peer ID, the victim retrieves those stored addresses and spawns `try_nat_traversal` tasks: [11](#0-10) 

`try_nat_traversal` retries outbound TCP connections in a loop for up to 30 seconds (~150 socket create/connect cycles per task): [12](#0-11) 

The `pending_delivered` map is only cleaned up every 5 minutes via the `notify` timer, so poisoned entries persist: [13](#0-12) 

**Two-step exploit flow:**
1. Attacker sends `ConnectionRequest { from: random_A, to: victim_peer_id, listen_addrs: [192.168.1.1:8080] }` directly to victim → victim stores `pending_delivered[random_A] = ([192.168.1.1:8080], now)`.
2. Attacker sends `ConnectionSync { from: random_A, to: victim_peer_id, route: [] }` directly to victim → victim retrieves stored addresses and spawns `try_nat_traversal(192.168.1.1:8080)`, initiating real outbound TCP connections.

Repeating with fresh `random_B`, `random_C`, ... at 30 pairs/second accumulates 900 concurrent `try_nat_traversal` tasks after 30 seconds, each with up to 24 (ADDRS_COUNT_LIMIT) concurrent TCP socket futures via `select_ok`, totaling up to 21,600 concurrent TCP sockets.

## Impact Explanation

**Node crash (High, 10001–15000 points):** Accumulating hundreds of concurrent `try_nat_traversal` tasks — each holding open TCP sockets for up to 30 seconds — exhausts file descriptors and connection resources on the victim node. This matches: *"Vulnerabilities which could easily crash a CKB node."*

**Network congestion (High, 10001–15000 points):** The `forward_rate_limiter` is the sole throttle on relay amplification. With `from` spoofing, a single attacker connection drives 30 unique-keyed `ConnectionRequest` messages per second, each forwarded to sqrt(N) relay peers without rate limiting. This matches: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

Any peer that establishes a single P2P connection to a CKB node can send `HolePunching` protocol messages. No special privilege, key, or majority hashpower is required. The hole-punching protocol is enabled by default. Crafting a valid molecule-encoded `ConnectionRequest` with a spoofed `from` field is trivial. The victim's peer ID is publicly discoverable via the P2P network. The attack is repeatable and stateless from the attacker's perspective.

## Recommendation

1. **Verify `from` against the actual session identity:** In `ConnectionRequestProcess::execute()`, look up the peer ID of `self.peer` from the peer registry and assert it equals `content.from`. Reject messages where they do not match.
2. **Key `forward_rate_limiter` on the actual sending `PeerIndex`** (`self.peer`) rather than on `content.from`, mirroring the existing per-connection `rate_limiter` design.
3. **Validate `listen_addrs` provenance before storing in `pending_delivered`:** Only store entries where the request was self-originated, or bind the stored entry to the actual session that delivered the message.
4. **Bound the number of concurrent `try_nat_traversal` tasks** to prevent resource exhaustion regardless of message rate.

## Proof of Concept

```
Attacker (single connected peer) → Victim node (victim_peer_id known):

# Rate-limiter bypass (repeat with fresh from each time):
for i in 0..30_per_second:
    send HolePunchingMessage::ConnectionRequest {
        from: PeerId::random(),   // fresh random each iteration
        to:   <victim_peer_id>,
        listen_addrs: [/ip4/127.0.0.1/tcp/9999],
        max_hops: 6,
        route: [],
    }
# Each message passes forward_rate_limiter with a unique key → forwarded to sqrt(N) peers

# Resource exhaustion:
loop with fresh random_X each iteration (30/sec):
    Step 1: send ConnectionRequest {
        from: <random_X>,
        to:   <victim_peer_id>,
        listen_addrs: [/ip4/192.168.1.1/tcp/8080, ... up to 24 addrs],
        max_hops: 6, route: [],
    }
    # Victim: respond_delivered() → pending_delivered[random_X] = ([192.168.1.1:8080, ...], now)

    Step 2: send ConnectionSync {
        from: <random_X>,
        to:   <victim_peer_id>,
        route: [],
    }
    # Victim: retrieves pending_delivered[random_X], spawns runtime::spawn(select_ok(try_nat_traversal × 24))
    # → real outbound TCP SYN to 192.168.1.1:8080, retried for 30 seconds

# After 30s: ~900 spawned tasks × 24 concurrent TCP futures = ~21,600 TCP sockets → file descriptor exhaustion → node crash
```

pub(crate) async fn try_nat_traversal(
    bind_addr: Option<SocketAddr>,
    addr: Multiaddr,
) -> Result<(TcpStream, Multiaddr), std::io::Error> {
    let net_addr = match multiaddr_to_socketaddr(&addr) {
        Some(addr) => addr,
        None => {
            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }
    };

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }

    debug!("Failed to connect to NAT after {} retries", retry_count);
    Err(std::io::ErrorKind::TimedOut.into())
}
