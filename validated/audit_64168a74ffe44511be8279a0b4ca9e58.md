### Title
Fee Charged Unconditionally Before Freshness Check Causes Permanent Fee Lock on Stale Updates — (`target_chains/sui/contracts/sources/pyth.move`)

### Summary
In `update_single_price_feed`, `deposit_fee_coins` is called unconditionally at line 277 before `update_cache` evaluates `is_fresh_update` at line 316. If the submitted price update is not fresher than the cached one, the fee is permanently deposited into the `PriceInfoObject` with no corresponding price update and no withdrawal path.

### Finding Description

The execution order in `update_single_price_feed` is:

1. Fee sufficiency check (line 274)
2. **`price_info::deposit_fee_coins(price_info_object, fee)`** — fee deposited unconditionally (line 277)
3. `update_cache(...)` called (line 287)
4. Inside `update_cache`: **`if (is_fresh_update(update, price_info_object))`** — freshness evaluated only now (line 316)
5. If `update_timestamp <= cached_timestamp`, the branch is skipped — price unchanged, fee already gone [1](#0-0) [2](#0-1) 

`is_fresh_update` uses strict `>` comparison, so an equal-timestamp update also fails the check and loses the fee: [3](#0-2) 

The deposited fee is stored as a `Coin<SUI>` dynamic object field under `FEE_STORAGE_KEY` inside the `PriceInfoObject`. Critically, `price_info.move` exposes only `deposit_fee_coins` and `get_balance` — **there is no `withdraw_fee_coins` or any governance withdrawal function** for fees stored in `PriceInfoObject`: [4](#0-3) 

The entire Sui contracts directory contains zero matches for any fee withdrawal from `FEE_STORAGE_KEY`. Fees deposited on stale updates are permanently frozen.

### Impact Explanation

Any caller of `update_single_price_feed` who submits a price update with `publish_time <= cached_timestamp` loses `base_update_fee` SUI permanently. The funds are locked in the `PriceInfoObject` with no recovery path. This directly matches **permanent freezing of user funds** with no service rendered.

### Likelihood Explanation

Two realistic paths exist:

**Path 1 — Race condition (no attacker needed):** Two independent relayers submit valid VAAs for the same feed near-simultaneously. The second transaction's update is stale relative to the first; its fee is lost. This is a routine operational scenario on a live network.

**Path 2 — Front-running (active attacker):** An attacker observes a user's pending `update_single_price_feed` transaction in the mempool. The attacker submits a valid, slightly-newer VAA (obtained from the Pyth price service) ahead of the user. The user's transaction executes, `is_fresh_update` returns false, and the user's fee is permanently locked. The attacker needs no privileged access — only a valid authenticated VAA, which is publicly available.

### Recommendation

Move the freshness check **before** the fee deposit. Only charge the fee when the update will actually be applied:

```move
public fun update_single_price_feed(...) {
    ...
    assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);

    // Find matching update first
    let i = 0;
    let found = false;
    while (i < hot_potato_vector::length<PriceInfo>(&price_updates)) {
        let cur_price_info = hot_potato_vector::borrow<PriceInfo>(&price_updates, i);
        if (has_same_price_identifier(cur_price_info, price_info_object)) {
            found = true;
            // Only deposit fee if the update is actually fresh
            if (is_fresh_update(cur_price_info, price_info_object)) {
                price_info::deposit_fee_coins(price_info_object, fee);
                update_cache(latest_only, cur_price_info, price_info_object, clock);
            } else {
                // Return fee to caller or abort
                transfer::public_transfer(fee, tx_context::sender(ctx));
            };
            break
        };
        i = i + 1;
    };
    ...
}
```

Alternatively, abort the transaction if the update is stale (similar to `update_price_feeds_if_fresh` on Aptos), so the fee is never consumed.
