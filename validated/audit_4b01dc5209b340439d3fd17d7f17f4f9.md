### Title
Unconstrained `spill` Account in `upgrade_contract` Allows Any Relayer to Front-Run Buffer Lamport Refund — (`target_chains/solana/programs/core-bridge/src/legacy/processor/governance/upgrade_contract.rs`)

---

### Summary

The `spill` account in `UpgradeContract` has no address constraint. Any unprivileged relayer who submits a valid governance upgrade VAA first can set `spill` to their own address, redirecting all buffer lamports to themselves instead of the governance proposer.

---

### Finding Description

The `UpgradeContract` account struct declares `spill` as:

```rust
/// Spill account to collect excess lamports.
///
/// CHECK: This account receives any lamports after the result of the upgrade.
#[account(mut)]
spill: UncheckedAccount<'info>,
``` [1](#0-0) 

The only constraint is `mut`. There is no `address = ...` check binding `spill` to any governance-designated recipient.

The `constraints()` function validates the VAA's guardian set, emitter, and that `buffer` matches the implementation pubkey encoded in the VAA: [2](#0-1) 

But `spill` is never validated. The governance VAA payload encodes only the `implementation` (buffer) pubkey — it does not encode a designated spill recipient.

The `invoke_signed` call then passes `ctx.accounts.spill.key()` directly to the BPF Loader Upgradeable `upgrade` instruction: [3](#0-2) 

The BPF Loader Upgradeable program transfers all remaining lamports from the buffer account to the `spill` address after the upgrade completes. Since `spill` is caller-controlled, whoever submits the transaction first controls where those lamports go.

Replay protection via `claim_vaa` (keyed on `[emitter_address, emitter_chain, sequence]`) ensures each VAA can only be executed once: [4](#0-3) 

This means the first submitter wins — and they can set `spill=attacker_address`.

---

### Impact Explanation

When a governance upgrade VAA is posted on-chain (as a `PostedVaaV1` or `EncodedVaa` account), it becomes publicly visible. Any relayer can construct and submit the `upgrade_contract` instruction with `spill` set to an attacker-controlled address before the legitimate governance executor does. The BPF Loader Upgradeable program transfers all buffer lamports to that address. For a large program (e.g., 1–2 MB), the buffer's rent-exempt balance can be several SOL. The program upgrade itself still succeeds; only the lamport refund is stolen.

---

### Likelihood Explanation

- Governance upgrade VAAs are rare but high-value events.
- The VAA is public once posted on-chain; any observer can front-run.
- No special privilege is required — only the ability to submit a Solana transaction.
- The attack is a single transaction and requires no setup beyond monitoring the mempool/chain for a posted governance VAA.
- Pyth's governance execution is typically automated, but the window exists between VAA posting and execution.

---

### Recommendation

Add an `address` constraint on `spill` binding it to a governance-designated recipient. The simplest fix is to encode the spill recipient in the governance VAA payload itself and verify it in `constraints()`. Alternatively, hard-code `spill` to a known treasury PDA or require `spill == payer` (the governance executor). At minimum, add:

```rust
#[account(
    mut,
    address = <designated_spill_pubkey>
)]
spill: UncheckedAccount<'info>,
```

or extend the governance VAA payload to include the spill address and validate it in `constraints()`.

---

### Proof of Concept

1. Governance proposes an upgrade; a VAA is signed by the guardian set and posted on-chain as a `PostedVaaV1` account.
2. Attacker observes the posted VAA account.
3. Attacker submits `upgrade_contract` with:
   - `vaa` = the posted VAA account
   - `buffer` = the implementation pubkey encoded in the VAA (required by `constraints()`)
   - `spill` = attacker's own address (no constraint prevents this)
   - `claim` = correct PDA derived from `[emitter_address, emitter_chain, sequence]`
4. `constraints()` passes (VAA is valid, buffer matches).
5. `claim_vaa()` creates the claim account (replay protection consumed).
6. `invoke_signed(upgrade, ..., spill=attacker)` executes; BPF Loader transfers buffer lamports to attacker.
7. The legitimate governance executor's subsequent attempt fails because the claim account already exists.

Accounting invariant violated: `buffer_lamports_before` ≠ `governance_proposer_lamports_after`; instead `buffer_lamports_before` == `attacker_lamports_after`.

    fn constraints(ctx: &Context<Self>) -> Result<()> {
        let vaa = VaaAccount::load(&ctx.accounts.vaa)?;
        let gov_payload = super::require_valid_governance_vaa(&ctx.accounts.config, &vaa)?;

        let decree = gov_payload
            .contract_upgrade()
            .ok_or(error!(CoreBridgeError::InvalidGovernanceAction))?;

        // Make sure that the contract upgrade is intended for this network.
        require_eq!(
            decree.chain(),
            SOLANA_CHAIN,
            CoreBridgeError::GovernanceForAnotherChain
        );

        // Read the implementation pubkey and check against the buffer in our account context.
        require_keys_eq!(
            Pubkey::from(decree.implementation()),
            ctx.accounts.buffer.key(),
            CoreBridgeError::ImplementationMismatch
        );

        // Done.
        Ok(())
    }
