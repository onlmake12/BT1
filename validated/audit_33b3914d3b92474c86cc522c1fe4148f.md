Audit Report

## Title
Tx-Pool DoS via Unbounded O(n) Descendant Graph Traversal on Transaction Removal — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry` unconditionally calls `update_descendants_index_key` before stripping links, triggering a full BFS over all descendants while the tx-pool write lock is held. Both `remove_committed_tx` and `remove_expired` call `remove_entry` directly with links intact, unlike `remove_entry_and_descendants` which pre-strips links to make the BFS a no-op. Because no `max_descendants_count` guard exists, an attacker can construct an arbitrarily wide dependency tree and cause O(n) CPU work per committed or expired root transaction, stalling all concurrent pool operations.

## Finding Description
`remove_entry` at L235–250 calls `update_descendants_index_key` at L243, then `remove_entry_links` at L245. [1](#0-0) 

`update_descendants_index_key` at L447–460 calls `self.links.calc_descendants`, which delegates to `calc_relative_ids` → `calc_relation_ids` (L52–72 in `links.rs`) — an iterative BFS that visits every reachable child node and inserts each into a `HashSet`, then calls `modify_by_id` on every descendant. [2](#0-1) [3](#0-2) 

The safe path, `remove_entry_and_descendants` at L252–265, pre-strips links for every node in the set *before* calling `remove_entry`, so when `update_descendants_index_key` runs, `calc_descendants` finds an empty child set and returns immediately. The comment at L256 confirms this is intentional. [4](#0-3) 

`remove_committed_tx` (L253–268 in `pool.rs`) calls `self.pool_map.remove_entry(&short_id)` directly at L255 — not `remove_entry_and_descendants` — so links are intact when the BFS runs. [5](#0-4) 

`remove_expired` (L271–288 in `pool.rs`) likewise calls `self.pool_map.remove_entry(&entry.proposal_short_id())` at L284 with links intact. [6](#0-5) 

`max_ancestors_count` (L67 in `pool_map.rs`) limits ancestor depth but no `max_descendants_count` exists anywhere in the codebase (confirmed by exhaustive search). A transaction with K outputs can have K direct children; the tree can be up to 25 levels deep, yielding up to K^25 descendants bounded only by the 180 MB pool size limit. [7](#0-6) 

## Impact Explanation
The tx-pool write lock serializes all pool operations: `send_transaction` RPC, P2P relay submissions, and block assembly. While the BFS traverses n descendants and issues n hash-map writes, every concurrent operation is blocked. With the 180 MB pool limit and minimum transaction sizes, n can approach millions of entries. An attacker who pre-fills the pool and triggers repeated root-transaction commits can sustain this stall, degrading or denying service to legitimate users across the network. This matches: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The attack entry point is the standard `send_transaction` RPC or P2P relay — no special privilege is required. The minimum fee rate is 1,000 shannons/KB; filling the pool is achievable at low cost. Having the root transaction committed requires only that a miner includes it — standard behavior for any fee-paying transaction. The attack is repeatable: after T0 is committed, subsequent levels of the tree become new roots and can be committed in later blocks, each triggering another O(n) traversal.

## Recommendation
1. **Strip links before calling `remove_entry` in `remove_committed_tx` and `remove_expired`**, mirroring the pattern in `remove_entry_and_descendants`. This ensures `calc_descendants` returns an empty set and reduces per-removal cost to O(1).
2. **Cap descendants per transaction** analogously to `max_ancestors_count`. During transaction submission, reject any transaction that would push any ancestor's descendant count above a configurable limit (e.g., 25).
3. **Lazy descendant-weight updates**: store a generation counter on each entry and recompute ancestor weights on demand rather than eagerly propagating on every removal.

## Proof of Concept
```
1. Submit T0 with K outputs via send_transaction RPC.
2. Submit K transactions T1_1..T1_K, each spending one output of T0.
3. Submit K² transactions T2_1..T2_K², each spending one output of a T1_i.
4. Continue for up to 25 levels until pool approaches 180 MB.
5. T0 is included in a block by any miner.
6. Node calls _update_tx_pool_for_reorg
     → remove_committed_txs
       → remove_committed_tx(T0)
         → pool_map.remove_entry(T0)           [links still intact]
           → update_descendants_index_key(T0)
             → links.calc_descendants(T0)      [BFS over ~K^d nodes]
             → modify_by_id(desc_id) for each descendant
   All executed under the tx-pool write lock.
7. All concurrent send_transaction calls stall for the duration.
8. Repeat as subsequent levels of the tree are committed in later blocks.
```

A unit test in `tx-pool/src/component/tests/` that constructs a pool with a wide dependency tree (e.g., depth 5, fan-out 10 = 100,000 descendants), calls `remove_entry` on the root, and measures wall-clock time relative to a shallow tree would confirm super-linear growth.

    fn update_descendants_index_key(&mut self, parent: &TxEntry, op: EntryOp) {
        let descendants: HashSet<ProposalShortId> =
            self.links.calc_descendants(&parent.proposal_short_id());
        for desc_id in &descendants {
            // update child score
            self.entries.modify_by_id(desc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_ancestor_weight(parent),
                    EntryOp::Add => e.inner.add_ancestor_weight(parent),
                };
                e.score = e.inner.as_score_key();
            });
        }
    }

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
    }

    pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }
