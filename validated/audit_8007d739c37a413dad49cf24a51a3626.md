Audit Report

## Title
Unbounded OFFSET in `get_tx_with_cell` Enables Crafted Cursor to Hang Indexer RPC Queries — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
The `get_transactions` RPC (ungrouped mode) decodes the last 4 bytes of the `after` cursor into an `i32` offset and passes it directly to the SQL query builder with no upper-bound validation. An unprivileged caller can supply an `after` value encoding `i32::MAX` (2,147,483,647), causing the database to execute `OFFSET 2147483647` — a full sequential scan that must materialize and discard billions of rows — hanging the indexer DB connection and blocking all subsequent indexer RPC queries.

## Finding Description
**Root cause — `decode_i32` validates only byte length, not numeric value:**

`decode_i32` at `util/rich-indexer/src/indexer_handle/async_indexer_handle/mod.rs` L286–294 reinterprets 4 bytes as a little-endian `i32` with no range check on the resulting integer. [1](#0-0) 

**Cursor parsing — only structural length is checked:**

In `get_transactions`, the `after` cursor is split into an 8-byte `tx_id` and a 4-byte `offset`. The only guard is that the total length equals 12 bytes. No maximum is enforced on the decoded offset value. [2](#0-1) 

**Offset injected verbatim into SQL:**

The decoded offset is forwarded without any bound check to `query_builder.offset(offset)` inside `get_tx_with_cell`, producing a query of the form `SELECT ... LIMIT <limit> OFFSET 2147483647`. [3](#0-2) 

**Contrast with `limit` — which IS bounded:**

The `limit` parameter is validated against `self.request_limit` before any query is built. No equivalent guard exists for the cursor-derived offset. [4](#0-3) 

**Exploit path:** Attacker calls `get_transactions` with a valid `script` parameter and an `after` value whose last 4 bytes encode `i32::MAX` in little-endian (`0x7fffffff`). The 12-byte length check passes. `decode_i32` returns `2147483647`. `get_tx_with_cell` builds and executes a SQL query with `OFFSET 2147483647`. Both SQLite and PostgreSQL must scan and discard up to 2 billion rows before returning results, saturating the indexer's DB connection for the duration of the query.

## Impact Explanation
The impact is a hang/unresponsiveness of the rich-indexer RPC. While the core CKB node (P2P, consensus, block sync) is not directly affected, the indexer DB connection is blocked for the duration of the malicious query, causing all subsequent `get_transactions`, `get_cells`, and related indexer RPC calls to queue or time out. This matches the allowed bounty impact: **Note (0–500 points) — Any local RPC API crash/hang**. The claim that the entire node process becomes unresponsive is overstated; only the indexer RPC subsystem is affected.

## Likelihood Explanation
The rich-indexer RPC is exposed on the standard CKB RPC port (default 8114) with no authentication. Any network-reachable client can call `get_transactions` with a crafted `after` value. The 12-byte cursor format is documented and derivable from any prior `get_transactions` response. The attack is deterministic, requires no special privileges, and is trivially repeatable.

## Recommendation
Add an explicit upper-bound check on the decoded offset immediately after `decode_i32`, before constructing `last_cursor`. The offset only ever needs to represent the number of rows sharing the same `tx_id` within a single page, so it should be bounded by at most `self.request_limit`:

```rust
let offset = decode_i32(offset)?;
if offset < 0 || offset as usize > self.request_limit {
    return Err(Error::Params("cursor offset out of range".to_string()));
}
last_cursor = Some((last, offset));
```

## Proof of Concept
```python
import struct

# 8-byte tx_id (all zeros) + 4-byte offset = i32::MAX little-endian
tx_id  = b'\x00' * 8
offset = struct.pack('<i', 2147483647)   # bytes: ff ff ff 7f
after  = "0x" + (tx_id + offset).hex()  # "0x000000000000000ffffffff7f"

payload = {
    "id": 1, "jsonrpc": "2.0",
    "method": "get_transactions",
    "params": [{
        "script": {
            "code_hash": "0x" + "00"*32,
            "hash_type": "type",
            "args": "0x"
        },
        "script_type": "lock"
    }, "asc", "0x1", after]
}
# Send to http://localhost:8114 — response will hang for the duration of the DB scan
```

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
