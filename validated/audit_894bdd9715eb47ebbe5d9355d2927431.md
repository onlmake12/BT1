### Title
O(n) Linear Scan Over All Pool Entries on Every `get_tx_pool_info` RPC Call - (File: `tx-pool/src/component/pool_map.rs`)

### Summary
`get_max_update_time()` in `pool_map.rs` performs an unbounded O(n) linear scan over every entry in the tx-pool to compute the maximum timestamp. This function is invoked on every `get_tx_pool_info` RPC call — an unauthenticated, rate-unlimited endpoint. An unprivileged attacker who fills the pool with many small transactions can then hammer this RPC to cause sustained CPU consumption and write-lock starvation on the tx-pool, degrading transaction admission and block processing.

### Finding Description

**Root cause — `get_max_update_time` in `pool_map.rs`:**

```rust
pub(crate) fn get_max_update_time(&self) -> u64 {
    self.entries
        .iter()
        .map(|(_, entry)| entry.inner.timestamp)
        .max()
        .unwrap_or(0)
}
```

This iterates over every `PoolEntry` in `MultiIndexPoolEntryMap` to find the maximum `timestamp`. The result is O(n) where n is the current pool entry count. [1](#0-0) 

**Call site — `TxPoolService::info()` in `service.rs`:**

```rust
last_txs_updated_at: tx_pool.pool_map.get_max_update_time(),
```

`info()` is called while holding the `tx_pool` **read lock**. The read lock is held for the entire duration of the O(n) scan. [2](#0-1) 

**RPC dispatch — `GetTxPoolInfo` message handler in `service.rs`:**

```rust
Message::GetTxPoolInfo(Request { responder, .. }) => {
    let info = service.info().await;
    ...
}
```

This handler is reachable by any RPC caller with no authentication and no rate limiting. [3](#0-2) 

**Pool size bound — `TxPoolConfig`:**

The pool is bounded at `max_tx_pool_size = 180 MB` (default). With minimal CKB transactions (~200–500 bytes each), the pool can hold hundreds of thousands of entries. [4](#0-3) 

**Attacker-controlled entry path:**

1. Attacker submits many small, valid transactions via `send_transaction` RPC (requires fees, but the minimum fee rate is 1000 shannons/KB — filling 180 MB costs ~180 000 000 shannons ≈ 1.8 CKB, a negligible cost).
2. Attacker repeatedly calls `get_tx_pool_info` RPC in a tight loop.
3. Each call acquires the `tx_pool` read lock and iterates over all pool entries.
4. While the read lock is held, any concurrent write (transaction submission, block commit, pool eviction) is blocked.

### Impact Explanation

- **CPU exhaustion**: Each `get_tx_pool_info` call triggers an O(n) scan. With hundreds of thousands of entries and a high call rate, this consumes significant CPU on the node's async runtime.
- **Write-lock starvation**: The read lock held during the scan blocks all concurrent writers (`submit_entry`, `remove_committed_txs`, `limit_size`). Sustained read-lock pressure delays transaction admission and block processing, degrading consensus participation.
- **No permanent state corruption**, but sustained degradation can cause the node to fall behind on block relay and transaction propagation, effectively a liveness DoS.

### Likelihood Explanation

- `get_tx_pool_info` is an unauthenticated RPC endpoint reachable by any local or remote caller (depending on node RPC bind configuration).
- No rate limiting is applied to this endpoint in the codebase.
- Filling the pool to maximize n requires only ~1.8 CKB in fees at default rates — a realistic cost for an attacker.
- The attack is repeatable and does not require any privileged role.

### Recommendation

Track the maximum timestamp **incrementally**: update a stored `max_timestamp` field on every `insert_entry` and `remove_entry` call. This makes `get_max_update_time()` O(1) instead of O(n), eliminating the scan entirely. This is the direct analog of the TangleStaking recommendation to use an enumerable set or a different removal strategy to avoid the O(n) storage loop.

### Proof of Concept

1. Connect to a CKB node with RPC exposed.
2. Submit enough valid transactions to fill the pool (e.g., using a funded wallet with many UTXOs).
3. In a loop, call `get_tx_pool_info` at maximum rate (e.g., 1000 req/s).
4. Observe: node CPU spikes; concurrent `send_transaction` calls experience increased latency; block template generation (`get_block_template`) is delayed because it also acquires the write lock on the pool.

The O(n) scan at `pool_map.rs:130–136`, called unconditionally from `service.rs:1093` on every unauthenticated `GetTxPoolInfo` RPC message, is the necessary vulnerable step — without it, the attacker's repeated RPC calls would be O(1) and harmless.
