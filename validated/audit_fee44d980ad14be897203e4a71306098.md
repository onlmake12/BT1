Audit Report

## Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` — (`sync/src/types/mod.rs`, `sync/src/relayer/transaction_hashes_process.rs`)

## Summary
`push_peer()` unconditionally appends a `PeerIndex` to `UnknownTxHashPriority::peers` with no deduplication or length cap. The overflow guard in `add_ask_for_txs` only checks the count of unique hash keys in `unknown_tx_hashes`, not the size of individual `peers` Vecs. A single remote peer can repeatedly announce phantom transaction hashes, causing each entry's `peers` Vec to grow without bound and ultimately exhaust the node's memory.

## Finding Description

**Root cause — unconditional append:** `push_peer` at `sync/src/types/mod.rs:1291-1293` is a bare `Vec::push` with no deduplication or length check. [1](#0-0) 

**Phantom hashes bypass `tx_filter`:** `TransactionHashesProcess::execute()` filters hashes through `tx_filter`, which only contains hashes of transactions that were actually received and verified via `mark_as_known_txs`. Phantom hashes (for non-existent transactions) are never inserted into `tx_filter`, so they pass the filter on every repeated announcement and are forwarded to `add_ask_for_txs` unconditionally. [2](#0-1) [3](#0-2) 

**Occupied-entry path appends without limit:** For a hash already present in `unknown_tx_hashes`, `add_ask_for_txs` clones the existing `UnknownTxHashPriority`, calls `push_peer(peer_index)`, and writes it back — growing the `peers` Vec by 1 on every repetition. [4](#0-3) 

**Overflow guard is bypassed:** The guard fires only when `unknown_tx_hashes.len()` (count of unique hash keys) reaches `MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) or the per-peer threshold. With 100 phantom hashes, the key count stays at 100 — the guard never triggers, and the per-peer counter check inside it is never reached. [5](#0-4) 

**Drain rate is negligible:** `pop_ask_for_txs` calls `next_request_peer()`, which removes at most one `PeerIndex` per hash per `RETRY_ASK_TX_TIMEOUT_INCREASE` (30 s). With 100 hashes, drain is ~3.3 entries/second. [6](#0-5) [7](#0-6) 

**Injection rate dominates:** The rate limiter allows 30 messages/second per `(PeerIndex, message_type)`. With 100 phantom hashes per message, the attacker injects 3,000 `PeerIndex` entries/second, net growth ~2,997 entries/second. [8](#0-7) [9](#0-8) 

## Impact Explanation

Each `PeerIndex` is a `usize` (8 bytes on 64-bit). At ~3,000 entries/second across 100 hashes, memory grows at ~24 KB/s. Sustained over minutes to hours, the process is OOM-killed by the OS, resulting in complete denial of service of the victim CKB node: no block production, no transaction relay, no RPC responses. This matches **High — Vulnerabilities which could easily crash a CKB node**. [10](#0-9) 

## Likelihood Explanation

- Requires only a single connected relay peer — no special privileges, no PoW, no keys.
- Phantom hashes (for non-existent transactions) are never added to `tx_filter`, so the attack sustains indefinitely without the attacker needing to produce real transactions.
- The rate limiter (30 req/s) slows but does not stop the attack; a single peer can exhaust gigabytes of RAM within minutes to hours.
- Fully repeatable and requires no coordination beyond a single TCP connection. [11](#0-10) 

## Recommendation

1. **Deduplicate in `push_peer`**: Before appending, check `self.peers.contains(&peer_index)` and skip if already present.
2. **Cap `peers` Vec length**: Enforce a hard maximum equal to the number of connected peers or a small constant (e.g., 8).
3. **Use `HashSet<PeerIndex>`** instead of `Vec<PeerIndex>` in `UnknownTxHashPriority` to get O(1) deduplication automatically.
4. **Per-peer hash announcement tracking**: Maintain a per-peer LRU set of recently announced hashes and drop duplicates before calling `add_ask_for_txs`. [1](#0-0) 

## Proof of Concept

```
1. Connect to a CKB node as a relay peer.
2. Generate 100 random 32-byte values as fake tx hashes (hashes for non-existent transactions).
3. In a loop at ≤30 iterations/second (within the rate limit):
   a. Send a RelayTransactionHashes message containing all 100 hashes.
4. After 1000 iterations:
   - Each entry in unknown_tx_hashes will have peers.len() == 1000.
   - Total PeerIndex entries: 100 × 1000 = 100,000 (800 KB).
5. Continue until RSS of the ckb process grows to available RAM and the process is OOM-killed.
```

The `peers.len()` grows by exactly 1 per repetition per hash because `push_peer` appends unconditionally and the overflow guard never fires (queue length stays at 100 unique keys, well below `MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000). [1](#0-0) [12](#0-11)
