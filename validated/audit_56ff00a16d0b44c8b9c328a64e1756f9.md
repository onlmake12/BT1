### Title
Provider Fee Sandwich Attack Causes User Overpayment With No Refund of Excess `msg.value` - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

In `Entropy.sol`'s `requestHelper()`, excess `msg.value` beyond the required fee is never refunded to the caller. Because a registered provider can call `setProviderFee()` with no timelock or delay, a malicious provider can front-run a user's entropy request by raising their fee to just below the user's `msg.value`, capturing the user's buffer as extra accrued fees, then lowering the fee back. This is the direct analog of the LP join sandwich: a user sends value based on a pre-execution quote, the quote changes adversarially in the same block, and the user overpays with no slippage protection.

### Finding Description

In `requestHelper()`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The fee read at line 234 (`getFeeV2`) is the **current on-chain fee at execution time**, not the fee the user observed when constructing the transaction. The interface explicitly documents that excess value is not refunded:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) 

A provider can change their fee at any time with no delay via `setProviderFee()`:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
``` [3](#0-2) 

There is no timelock, minimum notice period, or maximum fee-change magnitude enforced.

### Impact Explanation

A malicious registered provider executes the following sandwich:

1. User queries `getFeeV2(provider, gasLimit)` off-chain → gets `F` wei.
2. User submits `requestV2{value: F + buffer}(...)` (buffer added to avoid revert from fee fluctuation — standard practice).
3. **Front-run**: Provider calls `setProviderFee(F + buffer - 1)` in the same block, raising the fee to just below the user's `msg.value`.
4. User's tx executes: `providerInfo.accruedFeesInWei += (F + buffer - 1)`. Provider captures the full buffer minus 1 wei.
5. **Back-run**: Provider calls `setProviderFee(F)` to restore the original fee.

Net result: the user paid `F + buffer` instead of `F`. The provider extracted `buffer - 1` wei from the user. With fees in the range of `1_500_000_000_000_000` wei (~0.0015 ETH) per the keeper config, a 10% buffer is ~0.00015 ETH (~$0.30–$0.50 per request). At scale across many users, this is material. [4](#0-3) 

### Likelihood Explanation

- Any address can permissionlessly register as an Entropy provider via `register()`.
- `setProviderFee()` requires only that `msg.sender` is the registered provider — no governance, no timelock.
- Users and smart contracts routinely send fee buffers to avoid `InsufficientFee` reverts, especially since the docs warn fees change dynamically.
- A malicious provider who is the **default provider** (set by Pyth governance) would have the largest attack surface, but even a non-default provider can target users who explicitly select them.
- The attack requires mempool visibility and two additional transactions, but MEV bots routinely perform this.

### Recommendation

1. **Add a fee-change delay**: Require that fee changes only take effect after N blocks (e.g., 1 epoch), giving users time to observe the new fee before submitting.
2. **Refund excess `msg.value`**: Return `msg.value - requiredFee` to `msg.sender` after the fee check, analogous to how Uniswap V2 router refunds excess ETH.
3. **Allow users to specify a maximum acceptable fee**: Add a `maxFee` parameter to `requestV2` that reverts if `getFeeV2(provider, gasLimit) > maxFee` at execution time, giving users slippage protection equivalent to the LP `minAmountOut` pattern.

### Proof of Concept

```solidity
// Attacker is a registered Entropy provider
// User contract sends a 10% buffer to avoid InsufficientFee reverts

// Step 1: User observes fee = 1_500_000_000_000_000 wei
uint128 observedFee = entropy.getFeeV2(attackerProvider, 200_000);
uint128 buffer = observedFee / 10; // 10% buffer
uint128 userSends = observedFee + buffer;

// Step 2: Attacker front-runs — raises fee to just below userSends
// (in same block, higher gas price)
entropy.setProviderFee(userSends - 1); // called by attacker/provider

// Step 3: User's tx executes
// requiredFee = userSends - 1 (new fee)
// msg.value = userSends >= requiredFee → succeeds
// providerInfo.accruedFeesInWei += (userSends - 1)  ← attacker captures buffer
// accruedPythFeesInWei += 1

// Step 4: Attacker back-runs — restores original fee
entropy.setProviderFee(observedFee);

// Result: user paid `buffer - 1` extra wei to the attacker provider
// At scale: 150_000_000_000_000 wei (~$0.30) per victimized request
``` [5](#0-4) [6](#0-5)

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
