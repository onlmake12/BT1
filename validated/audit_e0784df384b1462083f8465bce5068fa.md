### Title
`check_tx_fee` Uses Raw Byte Size Instead of Transaction Weight for Minimum Fee Enforcement — (`File: tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee by passing raw serialized byte size (`tx_size`) directly to `FeeRate::fee()`. However, `FeeRate` is defined as **shannons per kilo-weight**, where *weight* is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)` — not raw bytes. For cycle-heavy, small-size transactions the two values diverge by up to ~119×, allowing an unprivileged submitter to have cycle-heavy transactions admitted to the pool while paying a fee that is orders of magnitude below what the weight-based rate would require.

---

### Finding Description

`FeeRate` is explicitly typed as "shannons per kilo-weight": [1](#0-0) 

The canonical weight of a transaction is:

```
weight = max(tx_size_bytes, cycles × DEFAULT_BYTES_PER_CYCLES)
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`: [2](#0-1) 

`TxEntry::fee_rate()` correctly uses this weight: [3](#0-2) 

But the admission gate `check_tx_fee` passes raw `tx_size` (bytes) to `FeeRate::fee()` instead of the proper weight: [4](#0-3) 

The code comment on lines 42–44 explicitly acknowledges the mismatch: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly."*

---

### Impact Explanation

Consider a transaction crafted to be small in serialized size but consume the maximum allowed cycles:

| Parameter | Value |
|---|---|
| `tx_size` | 100 bytes |
| `cycles` | 70,000,000 (= `max_tx_verify_cycles`) |
| Correct `weight` | `max(100, 70,000,000 × 0.000_170_571_4)` ≈ **11,940** |
| `min_fee_rate` (default) | 1,000 shannons/KW |

- **Admission check requires:** `1000 × 100 / 1000` = **100 shannons**
- **Weight-based minimum should be:** `1000 × 11,940 / 1000` = **11,940 shannons**

The attacker pays ~**119× less** than the weight-based rate demands. Such transactions:

1. Pass `check_tx_fee` and are admitted to the pool.
2. Are relayed to peers (propagating the underpriced load).
3. Consume the block's cycle budget when committed, without paying proportional fees.
4. Occupy pool space, potentially evicting legitimately-priced transactions.

This is a tx-pool spam / resource-exhaustion vector: an attacker can flood the pool and the network with cycle-heavy transactions at a fraction of the intended cost.

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction via the standard `send_transaction` RPC or P2P relay — no privileged access, no key material, no majority hashpower. The crafted transaction must simply have a small serialized body (e.g., a single input/output with a compact lock script) while executing a cycle-intensive script. This is straightforward to construct. The default `max_tx_verify_cycles = 70,000,000` and `min_fee_rate = 1,000` are live mainnet values, making the discrepancy immediately exploitable.

---

### Recommendation

Replace the size-only weight in `check_tx_fee` with the proper `get_transaction_weight` call. The cycles value is available after `verify_rtx` completes; the fee check should be deferred to that point, or the function should accept cycles as a parameter:

```rust
// After verify_rtx returns the Completed entry with actual cycles:
let weight = get_transaction_weight(tx_size, completed.cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

This aligns the admission gate with the weight-based fee rate that `TxEntry::fee_rate()` and the block-selection logic already use correctly.

---

### Proof of Concept

1. Craft a transaction with a lock script that loops for ~70,000,000 cycles but whose serialized size is ≤ 200 bytes.
2. Set the fee to `1000 × 200 / 1000 = 200 shannons` (satisfies the size-based check).
3. Submit via `send_transaction` RPC.
4. Observe the transaction is accepted into the pool despite its correct weight-based minimum fee being `1000 × 11,940 / 1000 = 11,940 shannons`.
5. Repeat with many such transactions; each consumes ~70M cycles of block budget while paying only ~1.7% of the weight-based fee.

/// The precision is set so that the difference between MAX_BLOCK_CYCLES * DEFAULT_BYTES_PER_CYCLES

pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

/// vbytes has been deprecated, renamed to weight to prevent ambiguity
#[deprecated(
    since = "0.107.0",
    note = "Please use the get_transaction_weight instead"

pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
