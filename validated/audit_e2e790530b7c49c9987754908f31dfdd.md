Audit Report

## Title
Unbounded `pending_delivered` HashMap Growth via Spoofed `from` PeerIds in `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
An attacker with a standard inbound P2P connection can grow `HolePunching::pending_delivered` without bound by sending `ConnectionRequest` messages that each carry a unique, attacker-controlled `from` PeerId targeting the victim node. The two guards that should prevent repeated insertion — the `forward_rate_limiter` and the dedup check in `respond_delivered()` — are both keyed on `from`, so each unique value bypasses them entirely. Only the per-session 30 req/sec `rate_limiter` limits throughput, allowing up to 9,000 map entries (~8 MB) per session per 5-minute prune window.

## Finding Description
**Root cause:** `pending_delivered` is initialized as `HashMap::new()` with no capacity bound (mod.rs L44, L279). Pruning via `retain()` runs only in `notify()` every `CHECK_INTERVAL` (5 minutes, mod.rs L25, L173–174).

**Rate-limiter bypass chain:**

1. `rate_limiter` (mod.rs L95–107) is keyed by `(session_id, msg.item_id())` at 30 req/sec per session. This is the only effective throughput bound.

2. `forward_rate_limiter` (connection_request.rs L132–143) is keyed by `(content.from, content.to, msg_item_id)`. Each unique `from` PeerId creates a fresh bucket, so the 1 req/sec limit is never triggered for any single key.

3. The dedup guard in `respond_delivered()` (connection_request.rs L161–167) checks `pending_delivered.get(&from_peer_id)`. With a unique `from` per message, this lookup always misses.

4. After `send_message_to` succeeds (L226–232), the unconditional insert fires (L234–237):
   ```rust
   self.protocol.pending_delivered.insert(from_peer_id, (remote_listens, now));
   ```

**Preconditions satisfied trivially:**
- `content.to` = victim's peer ID (publicly observable via identify protocol)
- `content.from` = fresh random PeerId per message (attacker-controlled bytes)
- `content.listen_addrs` contains at least one TCP/IPv4 or IPv6 address (e.g., `127.0.0.1:1234`), required to pass the filter at L196–219
- Attacker keeps the session open so `send_message_to` succeeds

**Accumulation:** 30 msg/sec × 300 sec = 9,000 entries per session before `notify()` prunes. The `forward_rate_limiter`'s internal `HashMapStateStore` also accumulates one entry per unique `(from, to, item_id)` key; `retain_recent()` is only called on `disconnected()` (mod.rs L67–69), compounding growth.

## Impact Explanation
**Medium — Suboptimal implementation of CKB state storage mechanism.**

Per-session memory growth before the 5-minute prune:
- `pending_delivered`: 9,000 entries × (~38 B PeerId + 24 × ~30 B Multiaddr + 8 B timestamp) ≈ 6.9 MB
- `forward_rate_limiter` internal state: 9,000 entries × ~138 B ≈ 1.2 MB
- Total per session: ~8.1 MB

With a typical `max_inbound_peers` of ~117, a coordinated attack across all inbound slots yields ~117 × 8.1 MB ≈ 950 MB of unbounded map growth before the next prune cycle. This violates the invariant that protocol state maps must be bounded in size and constitutes a suboptimal implementation of the node's P2P state storage mechanism. On resource-constrained deployments this creates OOM pressure.

## Likelihood Explanation
- Requires only a standard inbound P2P connection — no privilege escalation needed.
- Victim's peer ID is publicly observable via the identify protocol.
- Attacker generates random PeerIds in software; no cryptographic work required.
- Attack is repeatable across sessions and scales linearly with concurrent attacker-controlled connections.
- The only throughput mitigation (30 req/sec rate limiter) is intentionally permissive and does not bound map size.

## Recommendation
Add a capacity cap in `respond_delivered()` before the insert:
```rust
const MAX_PENDING_DELIVERED: usize = 1024;

if self.protocol.pending_delivered.len() >= MAX_PENDING_DELIVERED {
    return StatusCode::TooManyRequests.with_context("pending_delivered capacity exceeded");
}
```
Additionally, re-key `forward_rate_limiter` on `(session_id, to_peer_id)` rather than `(from, to, item_id)` so that a single session cannot create unbounded unique rate-limiter keys. Both fixes should be applied together.

## Proof of Concept
```rust
// Minimal state test sketch
let mut hp = HolePunching::new(network_state_configured_as_victim());
for _ in 0..10_000 {
    let from = PeerId::random();
    let msg = build_connection_request(
        from,
        victim_peer_id,
        vec!["/ip4/127.0.0.1/tcp/1234".parse().unwrap()],
    );
    hp.received(/* context with open session */, msg).await;
}
// Before notify() fires (within 5-minute window):
assert!(hp.pending_delivered.len() >= 9_000);
// forward_rate_limiter internal map also has ~9,000 entries
```
A fuzz test varying `content.from` while holding `content.to = local_peer_id` and a valid TCP listen address will reliably reproduce unbounded growth up to the rate-limiter ceiling of 9,000 entries per session per 5-minute window.
