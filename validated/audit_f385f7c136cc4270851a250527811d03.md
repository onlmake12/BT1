### Title
Stale Pre-Computed `total_tx_size`/`total_tx_cycles` Overwrites Eviction-Updated Accounting in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the pool's aggregate size and cycle counters (`total_tx_size`, `total_tx_cycles`) are computed **before** a call to `check_and_record_ancestors`, which may internally evict existing pool entries and correctly subtract their sizes/cycles from those same counters. The stale pre-computed values are then unconditionally written back, overwriting the correct post-eviction state. This inflates the pool's reported size and cycle usage by the sum of all evicted transactions, causing the tx-pool to appear permanently fuller than it actually is.

---

### Finding Description

In `tx-pool/src/component/pool_map.rs`, the `add_entry` function follows this sequence:

```rust
// Step 1 — stale snapshot computed BEFORE any evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211

// Step 2 — may evict entries, calling remove_entry → update_stat_for_remove_tx
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Step 3 — stale snapshot written back, OVERWRITING the eviction corrections
self.total_tx_size = total_tx_size;                             // line 218
self.total_tx_cycles = total_tx_cycles;                         // line 219
``` [1](#0-0) 

`updated_stat_for_add_tx` simply returns `self.total_tx_size + entry.size` and `self.total_tx_cycles + entry.cycles` — a snapshot of the counters at the moment of the call, before any evictions have occurred. [2](#0-1) 

`check_and_record_ancestors` contains an eviction path: when `ancestors_count > max_ancestors_count` but the excess can be resolved by removing `cell_ref_parents`, it calls `remove_entry_and_descendants` in a loop. [3](#0-2) 

`remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` correctly subtracts the evicted entries' sizes and cycles from `self.total_tx_size` and `self.total_tx_cycles`. [4](#0-3) [5](#0-4) 

After `check_and_record_ancestors` returns, `self.total_tx_size` and `self.total_tx_cycles` correctly reflect the post-eviction state. But lines 218–219 then unconditionally overwrite them with the stale pre-eviction snapshot, re-adding the sizes/cycles of all evicted transactions.

The net effect after one such `add_entry` call that evicts N entries:

```
self.total_tx_size  = (correct_post_eviction_value) + sum(evicted[i].size)
self.total_tx_cycles = (correct_post_eviction_value) + sum(evicted[i].cycles)
```

The counters are permanently inflated until the pool is cleared or restarted.

---

### Impact Explanation

`total_tx_size` and `total_tx_cycles` are the authoritative pool-fullness counters. They are checked in `updated_stat_for_add_tx` to enforce pool limits: [6](#0-5) 

Inflated counters cause:

1. **Spurious `Reject::Full` rejections** — subsequent valid transactions are rejected as "pool full" even when actual pool usage is below the limit. Any unprivileged RPC caller submitting transactions is affected.
2. **Suboptimal block assembly** — the block assembler reads pool stats to determine how many transactions to package; inflated counters may cause it to under-fill blocks, reducing miner revenue.
3. **Incorrect `tx_pool_info` RPC output** — the `total_tx_size` and `total_tx_cycles` fields reported to external callers are wrong, breaking any tooling that relies on them.

The inflation is **permanent** within a node's session: each eviction-triggering `add_entry` call compounds the error. The pool never self-corrects unless `clear()` is called (e.g., on a chain reorganization that resets the pool).

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is triggered when:

1. A new transaction has `ancestors_count > max_ancestors_count` (default 25), AND
2. The excess ancestors include `cell_ref_parents` — transactions that use one of the new transaction's input cells as a **cell dep** rather than a consumed input. [7](#0-6) 

An unprivileged transaction submitter can deliberately construct this scenario:

- Submit a chain of 25+ pending transactions (tx1 → tx2 → … → tx26).
- Submit a separate transaction (txDep) that uses one of the intermediate outputs as a **cell dep**.
- Submit a new transaction spending tx26's output; its ancestor count exceeds the limit, but evicting txDep reduces it below the threshold, triggering the eviction path.

This requires no special privileges — only the ability to submit transactions via the standard RPC (`send_transaction`). The attack can be repeated to compound the inflation.

---

### Recommendation

Move the `updated_stat_for_add_tx` call to **after** `check_and_record_ancestors` returns, so the snapshot is taken from the already-correct post-eviction state of `self.total_tx_size` and `self.total_tx_cycles`:

Moving it after `check_and_record_ancestors` means the limit check happens on the post-eviction totals, which is semantically correct — evictions free space that the new entry should be allowed to use.

### Proof of Concept

**Setup:**
- `max_ancestors_count = 2` (for simplicity)
- Pool contains: `txA` (size=100, cycles=1000) and `txB` (size=200, cycles=2000), where `txB` uses an output of `txA` as a **cell dep** (not a consumed input).
- `total_tx_size = 300`, `total_tx_cycles = 3000` (correct).

**Attack step:** Submit `txC` (size=50, cycles=500) that spends `txA`'s output. `txC`'s ancestors = {txA, txB} → count = 3 > max (2). `cell_ref_parents = {txB}`. Since `3 - 1 = 2 <= max`, eviction proceeds: `txB` is removed.

**Execution trace in `add_entry`:**

| Step | Action | `self.total_tx_size` | `self.total_tx_cycles` |
|------|--------|----------------------|------------------------|
| Start | — | 300 | 3000 |
| Line 210-211 | `updated_stat_for_add_tx(50, 500)` → snapshot `(350, 3500)` | 300 | 3000 |
| Line 213 | `check_and_record_ancestors` evicts `txB` via `update_stat_for_remove_tx(200, 2000)` | **100** | **1000** |
| Line 218-219 | Write back stale snapshot `(350, 3500)` | **350** ❌ | **3500** ❌ |

**Correct expected values** after inserting `txC` and evicting `txB`:
- `total_tx_size = 100 (txA) + 50 (txC) = 150`
- `total_tx_cycles = 1000 (txA) + 500 (txC) = 1500`

**Actual values:** `350` and `3500` — inflated by `txB`'s size (200) and cycles (2000), which were evicted but whose removal was overwritten by the stale snapshot.

        &self,

        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {

                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
