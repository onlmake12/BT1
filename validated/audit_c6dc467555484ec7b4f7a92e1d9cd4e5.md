Audit Report

## Title
Unsolicited `SendBlock` Accepted Without Sender Verification, Permanently Stalling Sync â€” (File: `sync/src/synchronizer/block_process.rs`)

## Summary
`BlockProcess::execute` calls `shared.new_block_received(&block)` without verifying that `self.peer` matches the peer recorded in `inflight_states` for that block hash. Any connected peer can race-send a `SendBlock` whose header hash matches an in-flight request, consuming the inflight slot and inserting `BLOCK_RECEIVED` into `block_status_map`. After async verification fails the status becomes `BLOCK_INVALID`; subsequent calls to `new_block_received` for the same hash fail the `HEADER_VALID` equality check and return `false`, so the block is never processed and sync stalls permanently at that height.

## Finding Description

**No peer check before consuming the inflight slot:**

`BlockProcess::execute` passes the block directly to `new_block_received` with no comparison between `self.peer` and the peer stored in `inflight_states`:

```rust
// sync/src/synchronizer/block_process.rs L43
if shared.new_block_received(&block) {
```

`new_block_received` calls `remove_by_block` keyed only on `(block.number(), block.hash())`:

```rust
// sync/src/types/mod.rs L1201-1207
if !self.state().write_inflight_blocks()
    .remove_by_block((block.number(), block.hash()).into())
{ return false; }
```

`remove_by_block` removes the `inflight_states` entry and applies the timing credit to `state.peer` (the originally assigned peer), not the actual sender:

```rust
// sync/src/types/mod.rs L791-796
self.inflight_states.remove(&block).map(|state| {
    let elapsed = unix_time_as_millis().saturating_sub(state.timestamp);
    if let Some(set) = download_schedulers.get_mut(&state.peer) {
        set.hashes.remove(&block);
```

After `remove_by_block` returns `true`, `new_block_received` checks `BlockStatus::HEADER_VALID.eq(&status)` (passes for a valid header hash), then inserts `BLOCK_RECEIVED` via a `Vacant`-only guard:

```rust
// sync/src/types/mod.rs L1220-1225

    self.shared().block_status_map().entry(block.hash())
{
    status.insert(BlockStatus::BLOCK_RECEIVED);
    return true;
}
```

**Why the stall is permanent:**

`BLOCK_INVALID = 1 << 12` is disjoint from `BLOCK_RECEIVED = 0b11`, so `BlockFetcher` at line 269 does not skip a `BLOCK_INVALID` block â€” it falls through and re-inserts an inflight entry. When the legitimate block then arrives, `new_block_received` reaches the `HEADER_VALID` equality check:

```rust
// sync/src/types/mod.rs L1216
if !BlockStatus::HEADER_VALID.eq(&status) { return false; }
```

Status is now `BLOCK_INVALID`, so this returns `false`. The block is never forwarded for processing. No code path clears `BLOCK_INVALID` from `block_status_map`. The fetcher loops indefinitely requesting the block; `new_block_received` rejects it every time.

**The lookup API to perform the peer check already exists but is unused here:**

```rust
// sync/src/types/mod.rs L624-626
pub fn inflight_state_by_block(&self, block: &BlockNumberAndHash) -> Option<&InflightState> {
    self.inflight_states.get(block)
}
```

**Contrast with the relay path**, which correctly gates processing on the sending peer being present in `peers_map` before touching shared state (`sync/src/relayer/block_transactions_process.rs` L65-72).

## Impact Explanation

A connected peer can permanently stall a victim node at any specific block height. The sync loop stops advancing past that height because `new_block_received` permanently rejects the legitimate block once `BLOCK_INVALID` is set. The attack is repeatable across multiple heights simultaneously (one attacker message per in-flight hash) and can be applied to many nodes in parallel with negligible cost (only a P2P handshake required). This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation

- Any peer completing the CKB P2P handshake can send `SendBlock` messages; no key material or hashpower is required.
- In-flight block hashes are trivially observable: the attacker watches `GetBlocks` messages from the victim or infers the sync frontier from public chain data.
- The race window is the victim's round-trip time to the legitimate peer â€” easily won by a co-located or low-latency attacker.
- After the attacker peer is banned, a fresh peer identity can repeat the attack on the next in-flight hash. The cost per stalled height is one TCP connection.

## Recommendation

Before calling `new_block_received` (or as a guard inside it), verify the sender:

1. Call `state().read_inflight_blocks().inflight_state_by_block(&(block.number(), block.hash()).into())` to retrieve the `InflightState`.
2. Compare `state.peer` with `self.peer` (the incoming `SendBlock` sender).
3. If they do not match, return early without removing the inflight entry or marking `BLOCK_RECEIVED`. Apply a mild penalty (not a ban) to the unsolicited sender, since unsolicited blocks can arrive legitimately during reorgs.

This mirrors the existing correct pattern in `BlockTransactionsProcess` (`sync/src/relayer/block_transactions_process.rs` L65-72), which gates all state mutation on the sending peer being present in `peers_map`.

## Proof of Concept

1. Connect malicious peer M to victim node V (standard P2P handshake).
2. Observe V sending `GetBlocks` containing hash `H` (block at the sync frontier).
3. Before the legitimate assigned peer responds, M sends a `SendBlock` message: header is identical to block `H` (valid hash), but the transaction list does not match `transactions_root`.
4. V's `BlockProcess::execute` calls `new_block_received`: `remove_by_block` succeeds (hash `H` is in-flight), status is `HEADER_VALID`, entry is `Vacant` â†’ returns `true`. Inflight slot for `H` is consumed.
5. Async verification fails (`transactions_root` mismatch). M is banned. Block status transitions to `BLOCK_INVALID`.
6. On the next `BlockFetcher` cycle, `get_block_status(H)` returns `BLOCK_INVALID` (bit 12). `status.contains(BLOCK_RECEIVED)` is `false` (bits 0-1 are clear), so the fetcher re-inserts `H` into inflight and re-requests it.
7. The legitimate peer responds with the valid block. `new_block_received` is called: `remove_by_block` succeeds, but `BlockStatus::HEADER_VALID.eq(&BLOCK_INVALID)` is `false` â†’ returns `false`. Block is discarded.
8. Steps 6-7 repeat indefinitely. V's sync is permanently stalled at block `H`.
9. Repeat for additional in-flight hashes using fresh peer identities to stall V at multiple heights simultaneously.
