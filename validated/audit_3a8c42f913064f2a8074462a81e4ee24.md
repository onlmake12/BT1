### Title
Unauthenticated `from` Field in Hole Punching `ConnectionRequest` Enables Peer Impersonation and Targeted DoS — (File: `network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

The `ConnectionRequest` handler in the hole punching protocol accepts an arbitrary `from` peer ID without verifying it matches the actual sender's authenticated peer ID. Any unprivileged peer can forge the `from` field to impersonate a victim peer, causing target nodes to store poisoned `pending_delivered` state that silently blocks all legitimate hole-punching attempts from the victim for a renewable 2-minute window.

---

### Finding Description

In `ConnectionRequestProcess::execute()`, the `content.from` field is parsed directly from the attacker-controlled message body and used without any check that it equals the peer ID of the session that actually delivered the message. [1](#0-0) 

The only validation performed on `content.from` is that any peer ID embedded inside a listen address must match it: [2](#0-1) 

This check is circular: it only ensures internal consistency of the attacker-supplied message, not that `content.from` corresponds to the actual authenticated sender (`self.peer`). There is no call equivalent to `_checkSender` — no lookup of the session's real peer ID from the peer registry to compare against `content.from`.

When the target peer (`content.to`) is reached, `respond_delivered` is called with the forged `from_peer_id`: [3](#0-2) 

The target stores the attacker's addresses under the victim's peer ID in `pending_delivered`: [4](#0-3) 

Subsequently, when the real victim sends a legitimate `ConnectionRequest` to the same target, the target checks `pending_delivered.get(&from_peer_id)` and silently ignores the request if the stored timestamp is within `HOLE_PUNCHING_INTERVAL` (2 minutes): [5](#0-4) 

The `HOLE_PUNCHING_INTERVAL` constant is defined as: [6](#0-5) 

The `ConnectionRequest` message schema confirms `from` is a plain `Bytes` field with no protocol-level binding to the sending session: [7](#0-6)

### Impact Explanation

An attacker can permanently suppress hole-punching connectivity between any victim peer V and any target peer T:

1. The attacker sends `ConnectionRequest { from: V, to: T, listen_addrs: [attacker_ip:port] }`. Because listen addresses without an embedded peer ID are silently augmented with `content.from` (line 53), the attacker's plain IP:port is stored as `attacker_ip:port/V` — attributed to the victim.
2. T stores `(V, ([attacker_ip:port/V], now))` in `pending_delivered`.
3. When V sends a legitimate `ConnectionRequest` to T, T finds the poisoned entry and returns `StatusCode::Ignore`, blocking V's NAT traversal attempt.
4. The attacker refreshes the entry once every 2 minutes (well within the `forward_rate_limiter`'s 1 req/sec budget per `(from, to, item_id)` key) to maintain the block indefinitely.

NAT-ed peers that depend on hole punching for connectivity can be selectively isolated from specific peers or, with sufficient message volume, from the broader network — a prerequisite for eclipse-style attacks against those peers.

---

### Likelihood Explanation

- Peer IDs are public, distributed via the discovery protocol, so the attacker needs no privileged information.
- The attacker needs only a single live P2P connection to the CKB network; the `ConnectionRequest` is forwarded by intermediate nodes.
- The `forward_rate_limiter` (1 req/sec per `(from, to, item_id)`) does not prevent the attack; refreshing a 2-minute entry requires only one message every 120

    pub(crate) async fn execute(mut self) -> Status {
        let content = match RequestContent::try_from(&self.message) {
            Ok(content) => content,
            Err(status) => return status,
        };
        if content.listen_addrs.len() > ADDRS_COUNT_LIMIT || content.listen_addrs.is_empty() {
            return StatusCode::InvalidListenAddrLen
                .with_context("the listen address count is too large or empty");
        }

        if content.max_hops > MAX_HOPS {
            return StatusCode::InvalidMaxTTL.into();
        }
        if content.route.len() > MAX_HOPS as usize {
            return StatusCode::InvalidRoute.with_context("the route length is too long");
        }

        let self_peer_id = self.protocol.network_state.local_peer_id();
        if content.route.contains(self_peer_id) {
            return StatusCode::Ignore.with_context("the message is passed, ignore it");
        }

        if self
            .protocol
            .forward_rate_limiter
            .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
            .is_err()
        {
            debug!(
                "from: {}, to {}, item_name: {}, rate limit is reached",
                content.from, content.to, "ConnectionRequest",
            );
            return StatusCode::TooManyRequests.with_context("ConnectionRequest");
        }

        if self_peer_id == &content.to {
            self.respond_delivered(content.from, &content.to, content.listen_addrs)
                .await
        } else if content.max_hops == 0u8 {
            StatusCode::ReachedMaxHops.into()
        } else {
            self.forward_message(self_peer_id, &content.to).await
        }
    }
