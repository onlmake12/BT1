Audit Report

## Title
Missing Per-Peer Rate Limiter on `LightClientProtocol` Allows Unbounded CPU/IO Exhaustion — (`util/light-client-protocol-server/src/lib.rs`, `util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`LightClientProtocol` contains no rate-limiting field and its `try_process` dispatch path applies zero throttling before invoking any handler. A single connected peer can pipeline maximum-size `GetTransactionsProof` requests (1000 tx hashes each, all in distinct blocks), forcing the server to perform up to 1000 RocksDB reads, 1000 CBMT Merkle proof computations, 1000 uncle reads, and one MMR `gen_proof` call over 1000 positions per request — unboundedly. This directly degrades the node's I/O and CPU, reducing responsiveness for all other peers.

## Finding Description

`GetTransactionsProofProcess::execute` enforces only two input guards: an empty-list rejection and a count-exceeds-1000 rejection. Both are bypassed by a valid 1000-tx request. The handler then iterates over every unique block, calling `snapshot.get_block()`, `CBMT::build_merkle_proof()` (hashing all transactions in the block), and `snapshot.get_block_uncles()` per block. It then calls `reply_proof`, which invokes `mmr.gen_proof(items_positions)` over all collected MMR leaf positions.

`LightClientProtocol` is defined with only a single `shared: Shared` field — no rate limiter exists. The `received` → `try_process` path dispatches directly to the handler with no interposed check. This is in direct contrast to `Relayer`, which gates every non-PoW message through a per-peer, per-message-type `governor::RateLimiter` (30 req/s cap) before any processing. The same `governor` crate is already a dependency (used in `HolePunching` and `Relayer`), confirming the pattern is established and intentionally applied elsewhere.

## Impact Explanation

A single unprivileged peer can continuously saturate the server's RocksDB I/O and CPU by pipelining maximum-size `GetTransactionsProof` requests. Each request is O(N) in DB reads and O(N log H) in MMR proof generation (N=1000, H=chain height). Without rate limiting, the server processes these as fast as the network delivers them, degrading responsiveness for all other peers and the node's ability to participate in block/transaction relay. This constitutes an important performance vulnerability with a clear, established fix pattern already present in the codebase. Impact: **Low (501–2000 points) — important performance improvement for CKB node operation**.

## Likelihood Explanation

Any peer that has synced enough chain data to know 1000 committed tx hashes (all public on-chain data) can execute this. No special privilege, key, or hashpower is required. The attacker needs only a single P2P connection to the light-client protocol endpoint. The attack is trivially repeatable and requires no victim interaction.

## Recommendation

Add a per-peer rate limiter to `LightClientProtocol` mirroring the pattern in `Relayer` (`sync/src/relayer/mod.rs`) and `HolePunching` (`network/src/protocols/hole_punching/mod.rs`):

- Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`.
- In `try_process`, check `rate_limiter.check_key(&(peer_index, message.item_id()))` before dispatching, returning `StatusCode::TooManyRequests` on failure.
- Apply a conservative quota (e.g., 5–10 heavy proof requests per second per peer) given the per-request cost relative to the 30 req/s used for lighter relay messages.
- Call `rate_limiter.retain_recent()` in the `disconnected` handler to avoid unbounded memory growth.

## Proof of Concept

1. Connect a peer to the server's light-client protocol port.
2. Collect 1000 committed tx hashes from distinct blocks (public chain data).
3. In a tight loop, send `GetTransactionsProof { last_hash: <tip>, tx_hashes: [1000 hashes] }`.
4. Measure server CPU utilization, RocksDB read latency, and response latency to other peers — all will degrade proportionally to request rate.
5. Confirm: the existing test harness in `util/light-client-protocol-server/src/tests/components/get_transactions_proof.rs` already exercises the handler path; extend it with a loop of N rapid requests and assert that response latency to a concurrent benign peer remains bounded.
