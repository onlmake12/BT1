### Title
Unauthenticated `ConnectionRequest`/`ConnectionSync` Pair Weaponizes Victim Node as TCP Port Scanner — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

---

### Summary

An unprivileged remote peer can cause a CKB node to make repeated outbound TCP connection attempts to arbitrary third-party IP:port pairs for 30 seconds by sending two crafted hole-punching protocol messages. No authentication, PoW, or special privilege is required beyond a standard P2P connection.

---

### Finding Description

The hole-punching protocol has two message handlers that together form an exploitable chain:

**Step 1 — Populate `pending_delivered` via `ConnectionRequest`**

In `ConnectionRequestProcess::execute()`, when the victim node's own peer ID matches `content.to`, it calls `respond_delivered`: [1](#0-0) 

Inside `respond_delivered`, the attacker-supplied `listen_addrs` are filtered to TCP+IPv4/IPv6 only (a weak filter that passes any public IP), then stored verbatim: [2](#0-1) 

**Critically, there is no check that `content.from` matches the actual peer session sending the message.** The attacker can set `from` to any valid peer ID and `listen_addrs` to any TCP address.

**Step 2 — Trigger NAT traversal via `ConnectionSync`**

In `ConnectionSyncProcess::execute()`, when the node is the `to` target and `route` is empty, it looks up `pending_delivered` using the attacker-controlled `content.from`: [3](#0-2) 

It then spawns an async task that calls `try_nat_traversal` for each stored address: [4](#0-3) [5](#0-4) 

Again, **no check that the `ConnectionSync` sender is the peer identified by `content.from`**.

**Step 3 — `try_nat_traversal` loops for 30 seconds**

The function retries TCP `connect()` every ~200ms for a hard-coded 30-second window: [6](#0-5) [7](#0-6) 

With `ADDRS_COUNT_LIMIT = 24`, up to 24 concurrent 30-second scanning tasks are spawned per trigger. [8](#0-7) 

---

### Impact Explanation

The victim CKB node is weaponized as a TCP port scanner or connection amplifier against arbitrary third-party hosts. The victim's IP appears as the source of the SYN packets, not the attacker's. This violates the invariant that the node must not be used against external parties. It can also exhaust the victim's outbound connection resources (file descriptors, async task budget) if triggered repeatedly.

---

### Likelihood Explanation

The attack requires only a standard P2P connection to the victim — no keys, no PoW, no privileged role. The `forward_rate_limiter` (1 req/sec per `(from, to, item_id)` tuple) is trivially bypassed by rotating the `from` peer ID, since peer IDs are not bound to sessions. The `pending_delivered` TTL is 5 minutes, giving ample time to send the follow-up `ConnectionSync`. [9](#0-8) [10](#0-9) 

---

### Recommendation

1. **Bind `from` to the actual session**: In `ConnectionRequestProcess`, verify that `content.from` matches the peer ID of the session that sent the message (available via `peer_registry`). Reject messages where they differ.
2. **Bind `ConnectionSync` to the session**: In `ConnectionSyncProcess`, verify that the sender's session peer ID matches `content.from` before using `pending_delivered`.
3. **Alternatively**: Remove the `from`-field lookup entirely and key `pending_delivered` on the actual session ID, not the attacker-supplied peer ID field.

---

### Proof of Concept

```
1. Attacker A connects to victim V as a normal P2P peer.
2. A sends: ConnectionRequest { from=A_peer_id, to=V_peer_id, listen_addrs=[target_ip:target_port], max_hops=6, route=[] }
   → V stores: pending_delivered[A_peer_id] = ([target_ip:target_port], now)
3. A sends: ConnectionSync { from=A_peer_id, to=V_peer_id, route=[] }
   → V looks up pending_delivered[A_peer_id], finds [target_ip:target_port]
   → V spawns runtime::spawn(try_nat_traversal(bind_addr, target_ip:target_port))
   → V makes TCP SYN packets to target_ip:target_port every ~200ms for 30 seconds
4. Observe outbound TCP SYN packets from V's IP to target_ip:target_port in a packet capture.
5. Repeat step 2-3 with different A_peer_id values to bypass rate limiting and scan multiple targets.
```
