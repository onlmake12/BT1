### Title
Attacker Can Steal Provider Fees via Unvalidated `providerToCredit` in `Echo::executeCallback` - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo::executeCallback` accepts a caller-controlled `providerToCredit` address with no validation that it matches the original request's provider after the exclusivity period expires. Any unprivileged caller can redirect the accumulated request fees to an arbitrary address they control.

### Finding Description

`executeCallback` is a permissionless function. During the exclusivity window it enforces `providerToCredit == req.provider`, but once that window passes the check is skipped entirely. The fee credit line:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

unconditionally credits whatever address the caller supplies. There is no subsequent check that `providerToCredit` is the original provider, or even a registered provider. [1](#0-0) 

The fee is then permanently credited to the attacker-controlled address in `_state.providers[providerToCredit].accruedFeesInWei`. [2](#0-1) 

`registerProvider` is open to anyone and has no access control beyond requiring the caller not already be registered. [3](#0-2) 

`withdrawAsFeeManager` has no `isRegistered` guard, so a self-registered attacker who sets themselves as their own fee manager can drain the credited balance. [4](#0-3) 

### Impact Explanation

The legitimate provider who was assigned to fulfill a request loses the entire fee (`req.fee`) paid by the user. The attacker receives those funds. Since `req.fee` is set to `msg.value - pythFeeInWei` at request time, this equals the full provider portion of the user's payment. [5](#0-4) 

This is a direct theft of provider revenue, not merely a DoS. The legitimate provider's `accruedFeesInWei` receives nothing for work they are expected to perform.

### Likelihood Explanation

- `executeCallback` is fully permissionless after the exclusivity period.
- `registerProvider` is open to any address.
- `setFeeManager` is callable by any registered provider for their own address.
- The exclusivity period is a configurable `uint32` in seconds; once it elapses (which is the normal operating condition for any request the assigned provider is slow to fulfill), the attack window is open.
- No special privileges, leaked keys, or oracle manipulation are required. [6](#0-5) 

### Recommendation

After the exclusivity period, restrict `providerToCredit` to registered providers only, or remove the caller-supplied parameter entirely and always credit `req.provider`. At minimum, add:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit not registered"
);
```

A stronger fix credits `req.provider` unconditionally (or a whitelisted set of providers), removing the attacker-controlled input from the fee accounting path entirely.

### Proof of Concept

1. Attacker calls `registerProvider(0, 0, 0)` — attacker is now a registered provider at `attackerAddr`.
2. Attacker calls `setFeeManager(attackerAddr)` — attacker is their own fee manager.
3. Victim calls `requestPriceUpdatesWithCallback{value: fee}(defaultProvider, publishTime, priceIds, gasLimit)` — `req.fee = fee - pythFeeInWei` is stored.
4. Attacker waits for `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`.
5. Attacker calls `executeCallback(attackerAddr, victimSeqNum, validUpdateData, priceIds)` — the exclusivity check is skipped; `_state.providers[attackerAddr].accruedFeesInWei += req.fee`.
6. Attacker calls `withdrawAsFeeManager(attackerAddr, stolenAmount)` — ETH is transferred to attacker. [7](#0-6) [8](#0-7)

        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        (bool sent, ) = msg.sender.call{value: amount}("");

    }
