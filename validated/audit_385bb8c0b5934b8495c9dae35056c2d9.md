### Title
Network Alert Signature Lacks Chain-ID Domain Separation — Alert Replay Across CKB Networks (`util/network-alert/src/verifier.rs`)

### Summary

The CKB network alert system signs and verifies alerts using only the hash of the `RawAlert` molecule struct. That struct contains no chain identifier, genesis hash, or network name. Because the same four Nervos Foundation signing keys are hard-coded as the default `NetworkAlertConfig` for every CKB network (mainnet, testnet, and any CKB-based fork using the default config), a valid alert signed for one network is cryptographically valid on every other network. An unprivileged attacker who observes a valid alert on any CKB network can replay it on any other CKB network via the public `send_alert` RPC endpoint.

### Finding Description

**Root cause — no domain separator in the signed message**

`calc_alert_hash()` is defined as a plain `blake2b_256` of the raw `RawAlert` molecule bytes:

```
RawAlert { notice_until, id, cancel, priority, message, min_version, max_version }
``` [1](#0-0) [2](#0-1) 

None of those fields carry a chain ID, genesis hash, or network name. The hash is therefore identical for the same alert content regardless of which CKB network it was created for.

**Verification never checks chain context**

`Verifier::verify_signatures()` derives the message to verify from `alert.calc_alert_hash()` and checks it against the hard-coded public key set. No chain-specific data is mixed in at any point: [3](#0-2) 

**Same signing keys on every network**

`NetworkAlertConfig::default()` embeds the same four public keys for all networks: [4](#0-3) [5](#0-4) 

The launcher uses `unwrap_or_default()`, so any node that does not override `alert_signature` in its config uses these keys: [6](#0-5) 

**Attacker-reachable entry point**

The `send_alert` RPC is the public injection point. Any RPC caller can submit an alert: [7](#0-6) 

After passing `verify_signatures`, the alert is immediately broadcast to all connected peers via the P2P alert protocol: [8](#0-7) 

### Impact Explanation

**Scenario A — Cross-network alert injection (misleading warnings)**
An attacker observes a valid testnet alert (e.g., "upgrade from v0.105.x") from the testnet P2P network or via testnet RPC. They submit the identical signed bytes to a mainnet node's `send_alert` RPC. The mainnet node accepts it, stores it, and relays it to all its peers. Every mainnet node operator sees a testnet-specific version warning on their mainnet node, potentially triggering unnecessary upgrades or downgrades.

**Scenario B — Cancel-alert replay to suppress a live mainnet warning**
Alert IDs are small integers (`Uint32`). If a cancel alert is ever issued on testnet with `cancel = X`, and mainnet currently has an active alert with the same ID `X`, the attacker replays the testnet cancel alert on mainnet. The mainnet alert with ID `X` is silently removed from all nodes, suppressing a potentially critical security warning. [1](#0-0) 

Severity: **Medium** — no direct asset theft, but the alert system is the only in-band mechanism for the Nervos Foundation to warn node operators of critical bugs. Suppressing or spoofing alerts undermines the integrity of that trust channel.

### Likelihood Explanation

- The attacker needs no privileged access: only the ability to call `send_alert` on any RPC-exposed mainnet node (default RPC port 8114) and to observe a valid alert on any other CKB network.
- Valid alerts are publicly observable on the testnet P2P network and via testnet RPC.
- Alert IDs are small sequential integers, making ID collisions between mainnet and testnet highly probable over time.
- The attack is fully passive until the replay step; no cryptographic forgery is required.

### Recommendation

Include a chain-specific domain separator in the signed message. The simplest approach is to mix the genesis block hash (already available as `consensus.genesis_hash`) into the data that is hashed before signing:

```
alert_message = blake2b_256(genesis_hash || raw_alert_bytes)
```

This makes a signature valid only for the network whose genesis hash was used, eliminating cross-network replay entirely. The `Verifier` would need to receive the genesis hash at construction time and prepend it before calling `calc_hash`.

### Proof of Concept

1. Run a CKB testnet node and a CKB mainnet node, both with the default `alert_signature` config (same four Nervos Foundation keys).
2. On testnet, issue a valid alert via `send_alert` (requires the Foundation keys — but any *existing* historical alert already on the testnet P2P network suffices; capture it via `get_blockchain_info().alerts` on any testnet node).
3. Submit the captured alert bytes verbatim to the mainnet node's `send_alert` RPC:
   ```json
   {"jsonrpc":"2.0","method":"send_alert","params":[<testnet_alert_json>],"id":1}
   ```
4. Observe that the mainnet node accepts the alert (`null` result), stores it, and relays it to all connected mainnet peers — confirmed via `get_blockchain_info().alerts` on any mainnet peer.

The acceptance is guaranteed because `verify_signatures` hashes only the `RawAlert` content with no chain context, and the same public keys are trusted on both networks. [3](#0-2) [2](#0-1) [5](#0-4)

    pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
        trace!("Verifying alert {:?}", alert);
        let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
        let signatures: Vec<Signature> = alert
            .signatures()
            .into_iter()
            .filter_map(
                |sig_data| match Signature::from_slice(sig_data.as_reader().raw_data()) {
                    Ok(sig) => {
                        if sig.is_valid() {
                            Some(sig)
                        } else {
                            debug!("invalid signature: {:?}", sig);
                            None
                        }
                    }
                    Err(err) => {
                        debug!("signature error: {}", err);
                        None
                    }
                },
            )
            .collect();
        verify_m_of_n(
            &message,
            self.config.signatures_threshold,
            &signatures,
            &self.pubkeys,
        )
        .map_err(|err| err.kind())?;
        Ok(())
    }
