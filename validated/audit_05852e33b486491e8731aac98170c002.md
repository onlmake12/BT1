Audit Report

## Title
Unbounded `forward_rate_limiter` HashMap Growth via Persistent Sessions with Unique `(from, to)` PeerIds — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

The `HolePunching` protocol maintains a `forward_rate_limiter` backed by an unbounded `HashMapStateStore` keyed on `(PeerId, PeerId, u32)`. Its only cleanup path, `retain_recent()`, is called exclusively in `disconnected()`. The `notify()` callback, which fires every 5 minutes, performs no cleanup of either rate limiter. An unprivileged remote peer that holds persistent sessions and sends `ConnectionRequest` messages with unique attacker-controlled `(from, to)` PeerId pairs causes the HashMap to grow without bound, exhausting node memory and crashing the process.

## Finding Description

**Rate limiter type and storage:**
`forward_rate_limiter` is declared as `RateLimiter<(PeerId, PeerId, u32)>` backed by `governor::state::keyed::HashMapStateStore`, which is an unbounded `HashMap`. [1](#0-0) 

**Cleanup only on disconnect:**
`retain_recent()` is called on both rate limiters only inside `disconnected()`. If an attacker never disconnects, this cleanup path is never reached. [2](#0-1) 

**`notify()` performs no rate limiter cleanup:**
The periodic `notify()` callback (every `CHECK_INTERVAL = 5 minutes`) only retains `pending_delivered` and `inflight_requests`. It never calls `retain_recent()` on either rate limiter. [3](#0-2) 

**Key derived entirely from unvalidated message payload:**
Inside `ConnectionRequestProcess::execute()`, the `forward_rate_limiter` is checked with a key `(content.from, content.to, self.msg_item_id)`. The `from` and `to` fields are parsed from raw message bytes and validated only for multihash format — they are never checked against the peer registry. [4](#0-3) [5](#0-4) 

**Outer rate limiter does not bound unique keys:**
The outer `rate_limiter` allows 30 `ConnectionRequest` messages per second per session. Each of those 30 messages may carry a distinct `(from, to)` pair, causing `governor`'s `check_key()` to insert a new HashMap entry for each previously-unseen key — regardless of whether the check returns `Ok` or `Err`. [6](#0-5) 

**Exploit flow:**
1. Attacker opens N persistent inbound sessions (up to the node's connection limit, typically ~125).
2. Each session sends 30 `ConnectionRequest` messages/sec, each with a freshly generated random but valid-multihash `(from, to)` pair.
3. Each unique pair inserts a new entry into `forward_rate_limiter`.
4. Sessions are never closed, so `disconnected()` and its `retain_recent()` call are never triggered.
5. `notify()` fires every 5 minutes but does not clean up the rate limiter.
6. The HashMap grows monotonically at N×30 entries/sec with no upper bound.

## Impact Explanation

Each `HashMapStateStore` entry for `(PeerId, PeerId, u32)` holds two ~39-byte multihash PeerIds, a `u32`, governor's internal `AtomicU64` state, and HashMap bucket overhead (~150–200 bytes per entry). With 125 sessions at 30 entries/sec/session:

- Growth rate: ~3,750 entries/sec ≈ ~562 KB/sec
- After 10 minutes: ~2.25M entries ≈ ~337 MB
- After 1 hour: ~13.5M entries ≈ ~2 GB

This causes OOM on typical node hardware, crashing the process. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires no cryptographic work, no PoW, and no privileged access. Any unprivileged peer can open inbound connections to the node's P2P port, negotiate the HolePunching protocol, and send `ConnectionRequest` messages with randomly generated but valid-multihash `(from, to)` bytes at the maximum allowed rate. The `from`/`to` fields are only checked for valid multihash format, not for peer registry membership. The exploit is locally reproducible with a simple script and is repeatable indefinitely.

## Recommendation

1. **Add periodic cleanup in `notify()`:** Call `self.rate_limiter.retain_recent()` and `self.forward_rate_limiter.retain_recent()` inside `notify()`, which already fires every 5 minutes.
2. **Cap the `forward_rate_limiter` map size:** Switch to a bounded LRU-backed keyed store, or enforce a hard cap on the number of tracked keys (e.g., reject new keys once the map exceeds `MAX_SESSIONS × SOME_CONSTANT`).
3. **Validate `from`/`to` against connected peers:** Reject `ConnectionRequest` messages where `from` does not correspond to a known connected peer, eliminating the ability to inject arbitrary keys into the rate limiter.

## Proof of Concept

```
1. Start a CKB node with default config.
2. Open 100 TCP sessions to the node's P2P port, each negotiating the HolePunching protocol.
3. From each session, send 30 ConnectionRequest messages/sec, each with a freshly generated
   random PeerId (valid multihash format) in both `from` and `to` fields.
4. Do NOT disconnect any session.
5. After 60 seconds: forward_rate_limiter map contains 100 × 30 × 60 = 180,000 entries.
6. After 10 minutes: ~2.25M entries, ~337 MB consumed by the map alone.
7. Continue until OOM kill.
```

The `notify()` timer fires every 5 minutes but does not call `retain_recent()`, so no cleanup occurs. The map grows monotonically until the process is killed by the OS.
