### Title
Unbounded Full-Pool Scan in `get_max_update_time()` Without Cached Value Causes Repeated O(n) Iteration on Every `tx_pool_info` RPC Call — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::get_max_update_time()` iterates over every entry in the transaction pool to compute the maximum `timestamp` value. This result is **not cached**; it is recomputed from scratch on every call. The function is invoked unconditionally inside `TxPoolService::info()`, which backs the public `tx_pool_info` JSON-RPC endpoint. Every other pool-wide aggregate (`total_tx_size`, `total_tx_cycles`, `pending_count`, `gap_count`, `proposed_count`) is maintained as an incrementally-updated storage variable, but `last_txs_updated_at` is the sole exception. An unprivileged attacker who fills the pool with many small transactions can force every subsequent `tx_pool_info` call to perform an O(n) scan over the entire pool.

---

### Finding Description

`PoolMap` maintains several pool-wide statistics as cached fields that are updated incrementally on each insert or remove:

```rust
pub struct PoolMap {
    pub(crate) total_tx_size: usize,
    pub(crate) total_tx_cycles: Cycle,
    pub(crate) pending_count: usize,
    pub(crate) gap_count: usize,
    pub(crate) proposed_count: usize,
    // ...
}
``` [1](#0-0) 

`last_txs_updated_at` is the sole exception. It is computed by a full linear scan every time it is needed:

```rust
pub(crate) fn get_max_update_time(&self) -> u64 {
    self.entries
        .iter()
        .map(|(_, entry)| entry.inner.timestamp)
        .max()
        .unwrap_or(0)
}
``` [2](#0-1) 

This function is called unconditionally inside `TxPoolService::info()`, which is the implementation of the `tx_pool_info` RPC:

```rust
last_txs_updated_at: tx_pool.pool_map.get_max_update_time(),
``` [3](#0-2) 

There is no rate-limiting, caching, or early-exit mechanism. Every `tx_pool_info` call triggers a full iteration over all pool entries regardless of pool size.

The pool's size is bounded by `max_tx_pool_size` (bytes), not by entry count. With the default of 180 MB and a minimum transaction serialized size of roughly 100–200 bytes, the pool can hold on the order of hundreds of thousands to over a million entries. Iterating all of them on every RPC call is a direct analog to the Rigor `projectCost()` pattern: an unbounded loop over a collection that has no per-call stop mechanism and whose result is never cached.

---

### Impact Explanation

Every call to the `tx_pool_info` RPC endpoint causes a full O(n) scan of the entire pool. When the pool is large (e.g., filled by an attacker or during high-traffic periods), this scan becomes expensive. Miners, wallets, and monitoring tools that poll `tx_pool_info` frequently will experience increased latency. In extreme cases, a pool filled to its maximum size could make the RPC handler slow enough to delay mining-related operations that depend on pool state queries.

---

### Likelihood Explanation

The `tx_pool_info` RPC is a public, unauthenticated endpoint reachable by any RPC caller. Filling the pool requires submitting transactions with valid fees, which imposes a cost on the attacker, but the pool can hold a very large number of transactions. Once the pool is large (which can also happen organically during network congestion), any caller — including legitimate users — triggers the expensive scan. The attacker does not need special privileges; they only need RPC access and enough CKB to pay fees for pool-filling transactions.

---

### Recommendation

Cache `last_txs_updated_at` as a storage field in `PoolMap`, exactly as `total_tx_size` and `total_tx_cycles` are handled. Update the cached value whenever an entry is inserted (set to the new entry's timestamp if it is greater) or removed (if the removed entry held the maximum, recompute once at removal time or use a secondary ordered structure). This eliminates the O(n) scan entirely and makes `get_max_update_time()` an O(1) field read, consistent with every other pool-wide statistic.

---

### Proof of Concept

1. **Attacker fills the pool**: Submit a large number of independent transactions (each spending a distinct UTXO) with the minimum acceptable fee rate until `total_tx_size` approaches `max_tx_pool_size`. With a 180 MB pool limit and ~200-byte transactions, this is ~900,000 entries.

2. **Attacker (or any caller) repeatedly calls `tx_pool_info`**: Each call invokes `TxPoolService::info()`, which calls `tx_pool.pool_map.get_max_update_time()`, which iterates all ~900,000 entries. [4](#0-3) 

3. **Result**: The RPC handler performs a full O(n) scan on every call. Under sustained polling (e.g., a miner's block-assembly loop or a monitoring script), this degrades node RPC responsiveness proportionally to pool size.

The structural mismatch is clear: `total_tx_size` and `total_tx_cycles` are updated incrementally in `add_entry` and `update_stat_for_remove_tx`, making their reads O(1): [5](#0-4) [6](#0-5) 

But `last_txs_updated_at` has no such incremental update path and is always recomputed by full scan: [2](#0-1)

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
