### Title
Missing Maximum Fee Guard in Entropy Request Functions Allows Provider Front-Running and Silent User Overpayment — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` function in `Entropy.sol` reads the current provider fee at execution time and silently absorbs any `msg.value` above the provider's portion into Pyth's treasury without refunding the caller. Because no `maxFee` parameter exists in any of the public request entry-points (`request`, `requestWithCallback`, `requestV2`), a provider (or its fee manager) can change `feeInWei` between a user's off-chain `getFeeV2()` call and the inclusion of their transaction. This produces two distinct harms: (1) a fee increase causes an `InsufficientFee` revert, wasting the user's gas; (2) a fee decrease causes the user to silently overpay, with the excess permanently captured by `_state.accruedPythFeesInWei` and never returned.

---

### Finding Description

Users are instructed to call `getFeeV2()` before each request and pass the result as `msg.value`. The interface documentation explicitly warns: *"Note that the fee can change over time"* and *"excess value is not refunded to the caller."*

Inside `requestHelper`, the fee is re-evaluated at execution time:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire surplus `msg.value - providerFee` is credited to Pyth's accrued fees — not returned to the caller. There is no `require(msg.value == requiredFee)` and no `maxFee` guard.

The provider fee is mutable with no timelock:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
``` [2](#0-1) 

The fee manager (a separate address authorized by the provider) can also change the fee via `setProviderFeeAsFeeManager`. [3](#0-2) 

None of the public request entry-points accept a `maxFee` parameter: [4](#0-3) 

The Fortuna keeper (`apps/fortuna`) automatically syncs the provider fee to match current gas prices, meaning fee changes are routine and not purely adversarial.

---

### Impact Explanation

**Scenario A — Fee decrease (financial loss):**
User queries `getFeeV2()` → `X` wei. User submits `requestV2{value: X}()`. Provider (or Fortuna automation) decreases fee to `Y < X` before the user's tx is mined. The user's tx succeeds, but `X − Y` wei is permanently captured by `_state.accruedPythFeesInWei` and never refunded. The user paid more than the service cost with no recourse.

**Scenario B — Fee increase (DoS / gas loss):**
User queries `getFeeV2()` → `X` wei. Provider increases fee to `X + δ`. User's tx reverts with `InsufficientFee`, consuming gas. The user must retry with a new fee query, and the cycle can repeat.

Both scenarios are worsened by the fact that Fortuna adjusts fees dynamically based on gas prices, making accidental collisions a realistic, non-adversarial occurrence.

---

### Likelihood Explanation

- Provider fee changes are a normal operational action with no timelock or delay.
- The Fortuna reference implementation (`apps/fortuna/src/command/setup_provider.rs`) calls `set_provider_fee` automatically whenever the configured fee diverges from the on-chain value, meaning fee changes happen in production without any human intent to harm users.
- On chains with mempool visibility (Ethereum mainnet, most L2s), a malicious provider or fee manager can observe a pending user request and front-run it with a fee change.
- The "excess not refunded" behavior is load-bearing: it means even accidental ordering (no front-running intent) causes financial loss.

---

### Recommendation

Add a `maxFee` parameter to all public request entry-points and revert if the current fee exceeds it:

```solidity
function requestV2(
    address provider,
    bytes32 userRandomNumber,
    uint32 gasLimit,
    uint128 maxFee          // NEW
) external payable returns (uint64) {
    require(getFeeV2(provider, gasLimit) <= maxFee, "Fee exceeds maxFee");
    ...
}
```

Alternatively, refund excess `msg.value` to the caller at the end of `requestHelper` instead of crediting it to Pyth's treasury. Either change eliminates the financial loss vector.

---

### Proof of Concept

**Financial loss (fee decrease):**

1. User calls `entropy.getFeeV2(provider, 0)` → returns `1000 wei`.
2. User submits `entropy.requestWithCallback{value: 1000}(provider, userRandomNumber)`.
3. Provider calls `entropy.setProviderFee(0)` (fee drops to 0; `pythFeeInWei` = 100 wei).
4. Provider's tx is mined first.
5. User's tx executes: `requiredFee = 100 wei`, `msg.value = 1000 wei` → no revert.
6. `providerInfo.accruedFeesInWei += 0`; `_state.accruedPythFeesInWei += 1000 − 0 = 1000 wei`.
7. User paid 1000 wei; only 100 wei was the actual fee. 900 wei is permanently lost to Pyth treasury.

**DoS (fee increase):**

1. User calls `entropy.getFeeV2(provider, 0)` → returns `1000 wei`.
2. User submits `entropy.requestWithCallback{value: 1000}(provider, userRandomNumber)`.
3. Provider calls `entropy.setProviderFee(2000)`.
4. Provider's tx is mined first.
5. User's tx executes: `requiredFee = 2100 wei`, `msg.value = 1000 wei` → reverts with `InsufficientFee`.
6. User loses gas; must retry.

        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();

        uint128 oldFeeInWei = providerInfo.feeInWei;
