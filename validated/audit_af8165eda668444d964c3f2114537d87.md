Audit Report

## Title
Blocking crossbeam `send` on `bounded(24)` channel inside async `SendBlock` handler stalls tokio worker threads — (`sync/src/synchronizer/mod.rs`, `chain/src/chain_controller.rs`)

## Summary

`Synchronizer::try_process` handles `SendBlock` messages in an async context without `tokio::task::block_in_place`, unlike every other sync message type. The `SendBlock` path ultimately calls `Request::call_without_response`, which performs a blocking `crossbeam_channel::Sender::send()` on a `bounded(24)` channel. When that channel is full, the tokio worker thread executing the async task blocks indefinitely, starving the runtime and halting all P2P message processing.

## Finding Description

**Channel capacity** — `process_block_tx`/`process_block_rx` is created with capacity 24: [1](#0-0) 

**Blocking send** — `call_without_response` calls `sender.send()`, the crossbeam blocking variant. The `let _ =` discards the `Result` but does not prevent the thread from blocking until a slot is available: [2](#0-1) 

**Call site** — `asynchronous_process_lonely_block` invokes this directly with no async yield point: [3](#0-2) 

**Missing `block_in_place`** — In `try_process`, `GetHeaders`, `SendHeaders`, and `GetBlocks` are all wrapped in `tokio::task::block_in_place`. `SendBlock` is not: [4](#0-3) 

**Sequential consumer** — `ChainService` processes one block at a time in a single-threaded loop, making it easy to saturate the channel under any I/O pressure: [5](#0-4) 

Full call chain for a `SendBlock` P2P message:
```
Synchronizer::try_process (async, no block_in_place)
  └─ BlockProcess::execute()
       └─ Synchronizer::asynchronous_process_remote_block()
            └─ SyncShared::accept_remote_block()
                 └─ ChainController::asynchronous_process_remote_block()
                      └─ ChainController::asynchronous_process_lonely_block()
                           └─ Request::call_without_response()
                                └─ crossbeam Sender::send()  ← BLOCKS if channel full
```

The guard at `asynchronous_process_remote_block` only passes blocks with `HEADER_VALID` status to `accept_remote_block`, but this status is routinely set during IBD via prior `SendHeaders` exchanges, making the precondition trivially satisfiable: [6](#0-5) 

## Impact Explanation

When the `bounded(24)` channel is full, any tokio worker thread executing a `SendBlock` handler blocks at `sender.send()`. An attacker who saturates the channel with 24 + N concurrent `SendBlock` messages (where N equals the tokio worker thread count, typically the CPU core count) holds all worker threads blocked simultaneously. This stalls the entire P2P message processing loop — no `GetHeaders`, `SendHeaders`, `GetBlocks`, or any other protocol message can be processed for the duration of the attack, effectively partitioning the node from the network.

**Impact: High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points). The node does not crash in the traditional sense but becomes fully unresponsive to all P2P traffic, which is operationally equivalent to a crash from the network's perspective.

## Likelihood Explanation

During IBD, blocks whose headers have already been validated carry `HEADER_VALID` status in `block_status_map`. An attacker controlling even a single peer with a pre-built chain segment of 30+ blocks can send 24+ distinct `SendBlock` messages for `HEADER_VALID` blocks faster than `ChainService` drains them (each drain requires a full `asynchronous_process_block` call including a DB write). Under any disk I/O pressure the channel stays full long enough to block all worker threads. The attack is cheap, repeatable, and requires no special privileges — only a standard P2P connection. [7](#0-6) 

## Recommendation

1. **Immediate**: Wrap the `SendBlock` arm in `tokio::task::block_in_place` to match the other handlers, preventing tokio worker thread starvation even when the send blocks:
   ```rust
   packed::SyncMessageUnionReader::SendBlock(reader) => {
       tokio::task::block_in_place(|| {
           if reader.check_data() {
               BlockProcess::new(reader, self, peer, nc).execute()
           } else {
               StatusCode::ProtocolMessageIsMalformed.with_context("SendBlock is invalid")
           }
       })
   }
   ```
2. **Better**: Replace `sender.send()` in `call_without_response` with `sender.try_send()` and drop or log the block when the channel is full, making the function truly non-blocking.
3. **Structural**: Apply per-peer rate limiting on `SendBlock` messages at the P2P ingress layer so the channel cannot be saturated by a single attacker.

## Proof of Concept

1. Establish a header chain of 30 blocks so their hashes carry `HEADER_VALID` status in the node's `block_status_map` (achievable via a normal `SendHeaders` exchange).
2. Create a mock `ChainService` that never reads from `process_block_rx` (simulating slow or stalled DB).
3. Spawn 30 concurrent tasks, each sending one `SendBlock` message through `BlockProcess::execute()`.
4. Assert that the 25th call does not return within a 1-second timeout — it will not, because `sender.send()` blocks with the channel at capacity 24.
5. Confirm that no other sync messages (e.g., `GetHeaders`) can be processed on any tokio worker thread during the stall, demonstrating full runtime starvation. [1](#0-0) [2](#0-1)
