Audit Report

## Title
Unconditional Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary
`TxConfirmStat::remove_unconfirmed_tx` correctly guards the `block_unconfirmed_txs` access against `tx_age >= 1000`, but the subsequent `confirm_blocks_to_failed_txs[tx_age - 1]` write at line 215 has no equivalent guard. When a tracked transaction is rejected after 1001+ blocks, `tx_age - 1 >= 1000` indexes past the end of the 1000-element vec, causing an unconditional Rust index-out-of-bounds panic. This crashes the node process and is remotely triggerable by any unprivileged peer.

## Finding Description
All three parallel vecs in `TxConfirmStat` are initialized with `max_confirm_blocks = MAX_CONFIRM_BLOCKS = 1000` elements (lines 139–141). In `remove_unconfirmed_tx` (lines 197–217), the guard at line 208 (`if tx_age >= self.block_unconfirmed_txs.len()`) correctly handles the `block_unconfirmed_txs` access for `tx_age >= 1000`. However, the `count_failure` branch at lines 214–216 is entirely outside that guard and has no bounds check of its own:

```rust
if count_failure {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

When `tx_age = 1001`, `tx_age - 1 = 1000`, which is past the last valid index (999) of `confirm_blocks_to_failed_txs`. Rust panics unconditionally with an index-out-of-bounds error.

The reachable call chain with `count_failure = true`:
- `reject_tx` (line 475) calls `drop_tx` (line 476)
- `drop_tx` (line 428–430) calls `drop_tx_inner(tx_hash, true)`
- `drop_tx_inner` (lines 416–425) calls `remove_unconfirmed_tx` with `self.best_height` as `tip_height` and `count_failure = true`

A transaction is tracked at `self.current_tip` at the time of `accept_tx`. If `best_height` later advances to `entry_height + 1001` or more before the tx is rejected, the panic is triggered.

## Impact Explanation
**High severity** — matches the allowed CKB bounty impact: *"Vulnerabilities which could easily crash a CKB node."* The Rust index-out-of-bounds panic propagates up through the fee-estimator service and crashes the node process. It is remotely triggerable by any unprivileged peer with no special privileges, no PoW, and no majority hashpower required.

## Likelihood Explanation
CKB produces blocks roughly every 8–10 seconds; 1001 blocks ≈ 2.2–2.8 hours. The tx-pool expiry is time-based (default ~12 hours), so a transaction easily survives 1001 blocks within the expiry window. When expiry fires, `remove_expired` calls `callbacks.call_reject`, which chains to `reject_tx` → `drop_tx` → `drop_tx_inner(count_failure=true)` → `remove_unconfirmed_tx` → panic. Alternatively, an attacker can submit a conflicting (double-spend) transaction after 1001 blocks to trigger the same rejection path immediately. The attack is repeatable and requires only the ability to submit a valid transaction to the mempool.

## Recommendation
Add a bounds check before the `confirm_blocks_to_failed_txs` write, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This silently drops the failure sample for transactions older than `MAX_CONFIRM_BLOCKS`, consistent with how `block_unconfirmed_txs` already handles the same case.

## Proof of Concept
```rust
// Reproduces with max_confirm_blocks = 10
let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
let mut stat = TxConfirmStat::new(buckets, 10, 0.993);

// Track a tx at height 0
let bucket_index = stat.add_unconfirmed_tx(0, FeeRate::from_u64(1000)).unwrap();

// 11 blocks later, reject it with count_failure=true
// tx_age = 11, confirm_blocks_to_failed_txs[10] → index 10 on a 10-element Vec → PANIC
stat.remove_unconfirmed_tx(0, 11, bucket_index, true);
```

In production: submit a valid tx at height H, wait for `best_height` to reach H+1001 (≈2.2 hours at 8s/block), then either wait for expiry or submit a conflicting tx. The original tx is rejected via `reject_tx` → `drop_tx_inner(count_failure=true)` → `remove_unconfirmed_tx` → `confirm_blocks_to_failed_txs[1000]` → **index out of bounds → node process crash**.
