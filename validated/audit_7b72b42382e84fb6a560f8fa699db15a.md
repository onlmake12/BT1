Audit Report

## Title
Infinite loop in `kill_process` when process ignores SIGTERM — (`ckb-bin/src/subcommand/daemon.rs`)

## Summary
The `kill_process` function sends SIGTERM and polls `check_process` in a loop, decrementing `wait_time` and intending to escalate to SIGKILL after 60 seconds. Due to a misplaced match arm, the `_ if wait_time <= 0` guard is only reachable when `check_process` returns `Err` (process already gone), not when it returns `Ok` (process still alive). If the target process never terminates after SIGTERM, the loop runs forever and `ckb daemon --stop` hangs indefinitely.

## Finding Description
In `kill_process` (`ckb-bin/src/subcommand/daemon.rs`, lines 61–86), after sending SIGTERM, the code enters a loop:

```rust
loop {
    let res = check_process(pid_file);
    match res {
        Ok(_) => {                      // arm 1: fires when process is alive
            wait_time -= 1;
            ...sleep(1s)...
        }
        _ if wait_time <= 0 => {        // arm 2: only reached when res is Err
            kill(...SIGKILL...)?;
            break;
        }
        _ => { break; }                 // arm 3: only reached when res is Err
    }
}
```

Rust evaluates match arms in order. `Ok(_)` matches any `Ok` value unconditionally. The wildcard arms at lines 70 and 82 are only reached when `res` is `Err`. When the process is alive, `check_process` returns `Ok`, arm 1 always fires, `wait_time` decrements from 60 to 0 and below, but the guard `wait_time <= 0` in arm 2 is never evaluated. The SIGKILL escalation path is dead code for the scenario it was designed for. The loop runs indefinitely (in release mode `wait_time` wraps as i32; in debug mode it eventually panics on underflow after ~68 years of wall time, which is not a practical bound).

`check_process` uses `kill(pid, None)` (signal 0) to probe liveness — it returns `Ok` as long as the process exists, regardless of whether it is handling signals. A process blocked in a long I/O flush, a custom signal handler that defers SIGTERM, or any process that simply ignores SIGTERM will keep `check_process` returning `Ok` indefinitely.

## Impact Explanation
`ckb daemon --stop` hangs indefinitely. The operator loses the ability to stop the node through the normal CLI path and must resort to manual `kill -9`. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local command line crash/hang**.

## Likelihood Explanation
No external attacker is required. Any CKB process that defers or ignores SIGTERM triggers this path — for example, a node under heavy I/O load during a flush, or any scenario where the OS does not deliver SIGTERM promptly. This is a normal operational scenario, not a contrived one. The operator running `ckb daemon --stop` is the only actor needed.

## Recommendation
Move the timeout check and SIGKILL escalation into the `Ok` arm:

```rust
Ok(_) => {
    wait_time -= 1;
    if wait_time <= 0 {
        kill(Pid::from_raw(pid), Some(Signal::SIGKILL)).map_err(|_| ExitCode::Failure)?;
        break;
    }
    eprint!("{}", ".".yellow());
    let _ = io::stderr().flush();
    std::thread::sleep(std::time::Duration::from_secs(1));
}
_ => { break; }
```

This ensures the SIGKILL escalation is evaluated on every iteration where the process is still alive, and the loop exits cleanly when the process has already terminated.

## Proof of Concept
1. Write a unit test that replaces `check_process` with a stub always returning `Ok(12345)`.
2. Call `kill_process` with a dummy pid file pointing to that pid.
3. Assert the function returns within 61 iterations (≤61 seconds) and that SIGKILL was sent.
4. With the current code, the assertion fails — the loop never exits because `wait_time` goes negative while `Ok(_)` continues to match unconditionally and the SIGKILL arm is never reached.
