### Title
`work_id=0` Tautology Bypass in Unauthenticated Notify Endpoint Allows Unconditional Template Injection — (`miner/src/client.rs`)

---

### Summary

A logic error in `update_block_template` causes the update guard to evaluate as a tautology when an incoming `BlockTemplate` carries `work_id=0`. Combined with a completely unauthenticated HTTP notify endpoint, any process that can reach the miner's listen address can permanently redirect all hashpower to attacker-controlled block templates.

---

### Finding Description

**The logic bug** is in `update_block_template`: [1](#0-0) 

```rust
let updated = |id| {          // id  = current_work_id (atomic)
    if id != work_id || id == 0 {   // work_id = incoming template's work_id
        Some(work_id)
    } else {
        None
    }
};
```

When the incoming template has `work_id = 0`, the condition expands to:

```
id != 0  ||  id == 0
```

This is a **tautology** — it is `true` for every possible value of `id` (`u64`). `fetch_update` therefore always succeeds, and `Works::New` is always dispatched, regardless of what `current_work_id` currently holds (0, 5, `u64::MAX` — all identical outcome).

**The unauthenticated entry point** is the `handle` function: [2](#0-1) 

There is zero authentication, zero IP allowlist, and zero HMAC/token check. Any HTTP POST containing a syntactically valid `BlockTemplate` JSON body is accepted and forwarded directly to `update_block_template`.

**The listener** is bound to whatever `SocketAddr` the operator configures: [3](#0-2) 

The code does not enforce `127.0.0.1`-only binding. A `0.0.0.0` or LAN-facing address is a normal production deployment choice.

**Initialization** sets `current_work_id` to `0`: [4](#0-3) 

So the miner starts in the exact state where the tautology fires even without the `work_id=0` trick — but the `work_id=0` injection makes it permanent and repeatable at any time, overriding any legitimate work already in progress.

---

### Impact Explanation

Once the attacker injects a template with `work_id=0`, the miner's `current_work_id` is set to `0`. Every subsequent injection with `work_id=0` again satisfies the tautology and overwrites the active work. The miner is locked into mining attacker-supplied templates indefinitely. Any PoW solution found is submitted for a block the attacker constructed (wrong coinbase, wrong transactions, or a stale parent), which the CKB node will reject. The miner produces zero valid blocks for the duration of the attack — complete hashpower waste.

---

### Likelihood Explanation

- Notify mode is a documented, supported production configuration.
- The listen address is operator-chosen; `0.0.0.0` is common for miners running in containers or behind a reverse proxy.
- No credentials, tokens, or signatures are required to POST to the endpoint.
- Crafting a minimal valid `BlockTemplate` JSON with `"work_id": "0x0"` requires no special knowledge beyond the public RPC schema.
- The attack is repeatable: re-injecting every few seconds keeps the miner permanently on attacker work.

---

### Recommendation

**Fix the tautology.** The `id == 0` branch was intended to handle the initial state (no work yet received), but it checks the *current* ID, not the incoming one. The correct guard should check whether the *current* work ID is the sentinel "uninitialized" value, not whether the incoming one is zero. One correct formulation:

```rust
let updated = |id: u64| {
    // Accept if: no work yet (current==0) OR incoming is strictly newer
    if id == 0 || id != work_id {
        Some(work_id)
    } else {
        None
    }
};
```

But this still accepts `work_id=0` from the node on first boot. A cleaner fix is to use a separate `initialized` flag or to treat `work_id=0` as invalid at the HTTP boundary and reject it before calling `update_block_template`.

**Add authentication to the notify endpoint.** Even a shared-secret token in the `Authorization` header (matching the value already used for the outbound RPC connection) would eliminate the unauthenticated injection surface entirely.
