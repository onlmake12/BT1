**Tracing the path:**

1. **Limit check** — `difficulties.len() + last_n_blocks * 2 > GET_LAST_STATE_PROOF_LIMIT` where `GET_LAST_STATE_PROOF_LIMIT = 1000`. With `last_n_blocks = 0` and `difficulties.len() = 499`: `499 + 0 = 499 ≤ 1000` → passes. [1](#0-0) [2](#0-1) 

2. **Monotonicity + boundary checks** — attacker provides 499 strictly increasing values all < `difficulty_boundary`. Both checks pass. [3](#0-2) 

3. **`take_while` filter** — difficulties are filtered to those `<= total_difficulty` of `difficulty_boundary_block_number - 1`. The attacker crafts all 499 values to satisfy this condition, so all 499 survive the filter. [4](#0-3) 

4. **`get_block_numbers_via_difficulties`** — iterates over all 499 difficulties. The `current_difficulty >= *difficulty` skip (line 82) only fires when the found block's actual total difficulty already covers the next requested value — the attacker avoids this by spacing difficulties to match individual block boundaries. For each difficulty, `get_first_block_total_difficulty_is_not_less_than` performs a binary search over `[start_block_number, difficulty_boundary_block_number)`. [5](#0-4) 

5. **`start_block_number` shrinkage** — after finding block `num`, `start_block_number = num - 1` (line 90-91). This shrinks the range by ~1 per iteration, so the total binary search work is `Σ log2(N - k)` for k=0..498, which is still O(499 × log N). [6](#0-5) 

6. **Each DB read** — `get_block_total_difficulty` does 2 DB reads: `get_block_hash` + `get_block_ext`. For a 1M-block chain: 499 × 20 × 2 ≈ **19,960 DB reads per message**. [7](#0-6) 

7. **No rate limiting** — no per-peer message rate limiting or throttling is present in the light client protocol server for this message type. The `received()` handler dispatches directly to `execute()`. [8](#0-7) 

**Assessment of the invariant:** The `GET_LAST_STATE_PROOF_LIMIT` is intended to bound work, but it bounds the number of *output* headers, not the number of DB reads during the binary search phase. The binary search phase is O(limit × log(chain_height)), which is unbounded relative to chain growth.

### Title
Unbounded O(N × log(chain_height)) DB Reads per `GetLastStateProof` Message Enables Light-Client-Triggered CPU/IO Exhaustion — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
An unprivileged light-client peer can craft a `GetLastStateProof` message with up to 999 monotonically increasing `difficulties` values (with `last_n_blocks = 0`) that all pass every input validation check and the `take_while` filter, causing `get_block_numbers_via_difficulties` to perform O(difficulties.len() × log(chain_height)) binary search iterations, each doing 2 RocksDB reads, with no per-peer rate limiting.

### Finding Description
`GetLastStateProofProcess::execute` enforces `difficulties.len() + last_n_blocks * 2 ≤ GET_LAST_STATE_PROOF_LIMIT (1000)` to bound output size, but this does not bound the number of DB reads during the binary search phase. After the `take_while` filter at line 325–328 (which the attacker defeats by crafting all difficulties ≤ `total_difficulty` of `difficulty_boundary_block_number - 1`), `get_block_numbers_via_difficulties` is called with up to 999 difficulties. For each difficulty, `get_first_block_total_difficulty_is_not_less_than` performs a binary search over the block range, calling `get_block_total_difficulty` (2 DB reads per call) O(log N) times. The `start_block_number = num - 1` update shrinks the range by only 1 per iteration, so total DB reads per message scale as O(999 × log(chain_height)).

### Impact Explanation
For a chain with 1M blocks: ~999 × 20 × 2 ≈ 40,000 RocksDB reads per message. A single peer sending messages at network speed can saturate the node's I/O and CPU, degrading or halting block sync and transaction processing for all peers. Impact is local node CPU/IO exhaustion (DoS).

### Likelihood Explanation
Any peer that has completed the light-client handshake can send this message. No PoW, key, or privileged role is required. The crafted message is valid by all current checks. The attack is trivially scriptable.

### Recommendation
Bound the total number of `get_block_total_difficulty` calls within a single `execute()` invocation (e.g., cap at `GET_LAST_STATE_PROOF_LIMIT × C` for a small constant C), or add per-peer request rate limiting for `GetLastStateProof` messages. Alternatively, track a call counter inside `get_block_numbers_via_difficulties` and abort early if it exceeds the limit.

### Proof of Concept
1. Start a CKB node with the light-client protocol enabled and mine ≥ 1000 blocks.
2. Connect as a light-client peer.
3. Send `GetLastStateProof` with:
   - `last_n_blocks = 0`
   - `difficulty_boundary` = total difficulty of tip + 1
   - `difficulties` = 499 values `[d1, d2, ..., d499]` where each `di` is the total difficulty of block `i` (strictly increasing, all < `difficulty_boundary`, all ≤ total difficulty of `difficulty_boundary_block_number - 1`)
4. Instrument `get_block_total_difficulty` call count; observe ~4990 calls per message.
5. Send messages in a tight loop; observe node I/O and CPU saturation.
