Looking at the vulnerability class: **accumulated value (reward/fee) with no complete retrieval mechanism** — a contract collects a type of value but lacks the function to claim/return it.

I'll map this to the Echo contract in the Pyth codebase.

---

### Title
Missing Request Cancellation Mechanism Permanently Locks User Fees in Unfulfilled Requests - (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.sol` has no `cancelRequest()` or refund function. When a user calls `requestPriceUpdatesWithCallback`, their payment (minus the Echo protocol fee) is stored in `req.fee`. If the assigned provider never calls `executeCallback`, that ETH is permanently locked in the contract with no retrieval path — analogous to how `CurveGaugeMarket` accumulates CRV rewards it can never claim.

### Finding Description

In `requestPriceUpdatesWithCallback`, the user's payment is split: [1](#0-0) 

- `_state.accruedFeesInWei += _state.pythFeeInWei` — Echo protocol fee, withdrawable by admin ✓
- `req.fee = msg.value - _state.pythFeeInWei` — provider's portion, stored in the request struct

The provider's portion is only ever credited in `executeCallback`: [2](#0-1) 

If `executeCallback` is never called, `req.fee` is never credited to anyone. The `IEcho` interface exposes no `cancelRequest`, `refundRequest`, or timeout-based reclaim function: [3](#0-2) 

The `EchoState.Request` struct stores `req.fee` as a `uint96` accounting entry, not a separate escrow — the ETH sits in the contract balance with no path out for the user: [4](#0-3) 

The admin's `withdrawFees` only covers `_state.accruedFeesInWei` (the Echo protocol fee), not `req.fee`: [5](#0-4) 

### Impact Explanation
Any ETH paid by a user for a request that is never fulfilled is permanently locked. The locked amount per request is `msg.value - _state.pythFeeInWei`. With multiple unfulfilled requests, this compounds. There is no admin escape hatch, no timeout, and no user-initiated refund path.

**Impact: Medium** — direct, irreversible loss of user funds proportional to the number of unfulfilled requests.

### Likelihood Explanation
Any registered provider can stop fulfilling requests (going offline, becoming unresponsive, or acting adversarially after the exclusivity period). The contract itself acknowledges this gap with a TODO comment: [6](#0-5) 

The entry path is fully permissionless: any user calling `requestPriceUpdatesWithCallback` with any registered provider is exposed.

**Likelihood: Medium** — providers are incentivized to fulfill but there is no on-chain enforcement, and operational failures are realistic.

### Recommendation
Add a `cancelRequest(uint64 sequenceNumber)` function that:
1. Requires a configurable timeout (e.g., `block.timestamp > req.publishTime + cancelDeadline`) to have elapsed.
2. Refunds `req.fee` to `req.requester`.
3. Clears the request from storage.

This mirrors the fix for the original report: just as `CurveGaugeMarket` needed to add a call to `Minter::mint()` to complete its reward collection, Echo needs to add a cancellation path to complete its fee lifecycle.

### Proof of Concept
1. User calls `requestPriceUpdatesWithCallback{value: 1 ether}(provider, ...)`.
2. Contract stores `req.fee = 1 ether - pythFeeInWei`, increments `_state.accruedFeesInWei += pythFeeInWei`.
3. Provider goes offline; `executeCallback` is never called.
4. User calls every function in `IEcho` — none allow reclaiming `req.fee`.
5. `1 ether - pythFeeInWei` is permanently locked in the contract.
