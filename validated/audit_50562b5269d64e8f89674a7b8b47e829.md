### Title
Governance VAA Sequence Check Allows Permanent Skipping of Intermediate Instructions - (File: governance/remote_executor/programs/remote-executor/src/lib.rs)

### Summary
The `execute_posted_vaa` instruction in the Pyth remote executor enforces only a **greater-than** check on the VAA sequence number (`posted_vaa.sequence > claim_record.sequence`) rather than a strict **equals-previous-plus-one** check. Because `execute_posted_vaa` is permissionless, any unprivileged actor can execute a higher-sequence governance VAA before a lower-sequence one has been processed, permanently and irrecoverably skipping all intermediate governance instructions.

### Finding Description
In `governance/remote_executor/programs/remote-executor/src/lib.rs`, the `ExecutePostedVaa` account constraint is:

```rust
constraint = posted_vaa.sequence > claim_record.sequence
    @ ExecutorError::NonIncreasingSequence
``` [1](#0-0) 

After execution, the claim record is updated unconditionally to the executed VAA's sequence:

```rust
claim_record.sequence = posted_vaa.sequence;
``` [2](#0-1) 

The `ClaimRecord` stores a single `u64 sequence` per emitter address: [3](#0-2) 

The claim record PDA is seeded only by `[CLAIM_RECORD_SEED, emitter_address]`, meaning all governance VAAs from the same emitter share one monotonic counter: [4](#0-3) 

Because the check is `> claim_record.sequence` and not `== claim_record.sequence + 1`, executing VAA with sequence N+K (K > 1) before VAA N+1 through N+K-1 have been executed will advance `claim_record.sequence` to N+K. All VAAs with sequences between the old value and N+K are then permanently unexecutable — they will always fail the `NonIncreasingSequence` constraint.

The existing test suite actually demonstrates this behavior: `vaa_account_transfer2` (sequence 3) is executed before `vaa_account_transfer1` (sequence 2), advancing the claim record to 3, after which sequence 2 is permanently rejected: [5](#0-4) 

### Impact Explanation
An unprivileged actor who calls `execute_posted_vaa` with a higher-sequence governance VAA before lower-sequence VAAs have been processed will permanently prevent those lower-sequence governance instructions from ever executing. Skipped instructions could include critical governance actions such as:
- Data source updates (e.g., revoking a compromised price feed source)
- Fee changes
- Contract upgrades or security patches applied via the remote executor

The governance relayer (`crank_pythnet_relayer`) executes VAAs in order, but because `execute_posted_vaa` is fully permissionless, any external actor can race the relayer and execute a higher-sequence VAA first whenever two or more governance VAAs are simultaneously posted to Solana.

### Likelihood Explanation
The attack requires two conditions: (1) multiple governance VAAs from the same emitter are posted to Solana but not yet executed, and (2) the attacker executes a higher-sequence VAA before the relayer processes the lower-sequence ones. Condition (1) occurs naturally during governance batches or relayer downtime. Condition (2) is trivially achievable since `execute_posted_vaa` is permissionless and the attacker only needs to submit a standard Solana transaction. No privileged access, leaked keys, or oracle manipulation is required.

### Recommendation
Replace the non-strict greater-than check with a strict consecutive-sequence check:

```rust
// Instead of:
constraint = posted_vaa.sequence > claim_record.sequence

// Use:
constraint = posted_vaa.sequence == claim_record.sequence + 1
```

This ensures governance VAAs are always executed in strict order, preventing any intermediate instructions from being permanently skipped. If out-of-order execution is intentionally desired for some VAAs, a separate mechanism (e.g., a set of consumed sequence numbers rather than a single high-water mark) should be used.

### Proof of Concept
1. Governance emitter posts VAA sequence 5 (e.g., "revoke compromised data source") and VAA sequence 6 (e.g., "update fee") to Solana.
2. Attacker calls `execute_posted_vaa` with the sequence-6 VAA before the relayer processes sequence 5.
3. `claim_record.sequence` is set to 6.
4. Any subsequent attempt to execute the sequence-5 VAA fails with `NonIncreasingSequence` (5 is not > 6).
5. The "revoke compromised data source" governance action is permanently skipped. [6](#0-5)

    pub fn execute_posted_vaa(ctx: Context<ExecutePostedVaa>) -> Result<()> {
        let posted_vaa = &ctx.accounts.posted_vaa;
        let claim_record = &mut ctx.accounts.claim_record;
        claim_record.sequence = posted_vaa.sequence;

        let payload = ExecutorPayload::try_from_slice(&posted_vaa.payload)?;
        payload.check_header()?;

        let (_, bump) = Pubkey::find_program_address(
            &[EXECUTOR_KEY_SEED.as_bytes(), &posted_vaa.emitter_address],
            &id(),
        );

        for instruction in payload.instructions.iter().map(Instruction::from) {
            // TO DO: We currently pass `remaining_accounts` down to the CPIs, is there a more efficient way to do it?
            invoke_signed(
                &instruction,
                ctx.remaining_accounts,
                &[&[
                    EXECUTOR_KEY_SEED.as_bytes(),
                    &posted_vaa.emitter_address,
                    &[bump],
                ]],
            )?;
        }
        Ok(())
    }
