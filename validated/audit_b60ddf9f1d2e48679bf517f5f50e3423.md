Audit Report

## Title
Unbounded Per-Session Announce Flooding Allows Peer Store Saturation — (`network/src/protocols/discovery/mod.rs`)

## Summary
An unprivileged remote peer can send unlimited `Nodes(announce=true)` messages per session because the `received_nodes` one-time guard is only applied to `announce=false` messages and no per-session rate limit or address quota exists for announce messages. An attacker can repeatedly drive `add_new_addrs`, filling the peer store to its hard cap of 16,384 entries with attacker-controlled addresses. By spreading ≤4 addresses across each of 4,096 distinct `/16` subnets, the eviction logic returns `EvictionFailed` and all subsequent legitimate peer addresses are silently dropped.

## Finding Description

**Step 1 — No rate limit on `announce=true` messages.**

`verify_nodes_message` (lines 266–299) enforces only a per-message item count (`≤ ANNOUNCE_THRESHOLD` items, `≤ MAX_ADDRS` addresses each). There is no counter, timestamp, or token-bucket guard on how many `announce=true` messages a single session may send. [2](#0-1) 

**Step 2 — `received_nodes` guard is bypassed for `announce=true`.**

The duplicate-message guard at line 181 only fires when `!nodes.announce && state.received_nodes`. For `announce=true`, `state.received_nodes` is never set (it is only set at line 203 when `!nodes.announce`), so the `else` branch — which calls `add_new_addrs` — executes on every single message without limit. [1](#0-0) 

**Step 3 — Every passing message writes directly to the peer store.**

`add_new_addrs` iterates every address and calls `peer_store.add_addr` with no session-level quota. Errors are only logged at `debug` level and silently discarded. [3](#0-2) 

**Step 4 — Peer store hard cap is 16,384 entries.** [4](#0-3) 

**Step 5 — Eviction can be defeated.**

`check_purge` first evicts non-connectable peers (none, since attacker addresses are freshly added with zero connection attempts and thus pass the `is_connectable` check). It then groups by `/16` network segment and only evicts from groups with `> 4` peers. An attacker who spreads exactly ≤4 addresses across each of 4,096 distinct `/16` subnets fills all 16,384 slots with no group exceeding 4 entries. Neither eviction branch finds a candidate, `EvictionFailed` is returned, and every subsequent `add_addr` call fails silently. [6](#0-5) 

## Impact Explanation

Once the peer store is saturated with attacker-controlled addresses, the victim node's feeler and outbound connection pools draw exclusively from attacker-controlled entries, degrading or eliminating organic peer discovery. Applied at scale across multiple nodes, this degrades the CKB network's ability to propagate blocks and transactions. This matches the allowed **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* A single persistent TCP session is sufficient to saturate one node; a small botnet can target the entire network simultaneously.

## Likelihood Explanation

- Requires only a standard P2P TCP connection — no privilege, no key, no proof-of-work.
- ~547 messages of 10 items × 3 addresses each saturates the store (16,384 / 30 ≈ 547). Each message is a few hundred bytes; the entire attack fits in a single TCP session.
- `is_valid_addr` filters non-routable IPs, but the attacker only needs syntactically valid routable addresses — they do not need to own them for the saturation step.
- The `addr_known` bloom filter is per-session and only prevents re-sending the same address back to the attacker; it does not prevent the attacker from using fresh addresses each message.

## Recommendation

1. **Add a per-session rate limit on `announce=true` messages** — add a counter in `SessionState` that disconnects or ignores the peer after N announce messages per time window.
2. **Cap the total number of addresses accepted per session** across all announce messages (e.g., ≤ `ANNOUNCE_THRESHOLD` addresses total per session lifetime, not per message).
3. **Improve eviction diversity** — consider source-session attribution so that addresses from a single session can be bulk-evicted if the session is later identified as malicious.

## Proof of Concept

```
1. Connect to victim node on the discovery protocol.
2. In a loop, send Nodes { announce: true, items: [10 × Node { addresses: [addr_i, addr_i+1, addr_i+2] }] }
   where each addr_i is a unique routable IP from a distinct /16 subnet.
3. After ~547 iterations the peer store reaches ADDR_COUNT_LIMIT=16384.
4. Assert: victim's peer store contains only attacker-supplied addresses.
5. Assert: subsequent honest Nodes messages from other peers trigger EvictionFailed and are silently dropped.
6. Assert: victim's outbound feeler connections go exclusively to attacker-controlled IPs.
```

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {

                            None
                        }
                    })
                    .flatten()
                    .collect()

                self.addr_manager.remove(key);
