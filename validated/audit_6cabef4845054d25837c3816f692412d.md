Audit Report

## Title
CPU Amplification via Unbounded CBMT Work in `GetTransactionsProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary
The `GetTransactionsProof` handler enforces a limit of 1000 tx_hashes per request, but the actual server-side work is not bounded by that constant. For each distinct block containing a requested transaction, the server loads the full block and hashes **all** of its transactions to build a CBMT merkle tree. An unprivileged remote peer can craft a single well-formed request with 1000 tx_hashes from 1000 different blocks, causing O(1000 × txs_per_block) blake2b operations and O(1000) full block reads per message, with no throttling or rejection.

## Finding Description
The limit check at [1](#0-0)  only validates the count of requested tx_hashes, not the number of distinct blocks or total transactions to be processed.

After passing this check, found hashes are grouped by block hash at [2](#0-1) , which can produce up to 1000 distinct block entries when each tx_hash comes from a different block.

For each distinct block, the server fetches the full block and iterates over **all** of its transactions to build the merkle tree at [3](#0-2) . The first argument to `CBMT::build_merkle_proof` is `block.transactions().iter().map(|tx| tx.hash()).collect()` — every transaction in the block, not just the requested one.

`CBMT` uses blake2b for every merge operation, as confirmed at [4](#0-3) .

The `received` handler in [5](#0-4)  only bans peers for malformed messages. A well-formed request with 1000 valid tx_hashes from 1000 different blocks is never rejected, throttled, or rate-limited. There is no cap on distinct blocks, total transactions processed, or per-peer message frequency anywhere in the handler.

The constant `GET_TRANSACTIONS_PROOF_LIMIT = 1000` at [6](#0-5)  bounds only the input size, not the work performed.

Additionally, `calc_witnesses_root()` at [7](#0-6)  also iterates over all transactions in each block, compounding the per-block work.

## Impact Explanation
A single well-formed P2P message causes the server to perform O(1000 × N) blake2b hash operations and O(1000) full block reads from storage, where N is the number of transactions per block. Repeated at high frequency from one or more peers, this constitutes CPU and I/O exhaustion capable of crashing or severely degrading a CKB node running the light client protocol server. This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attacker requires only: (1) a P2P connection to a node running the light client protocol server, and (2) knowledge of valid tx_hashes from different blocks — both trivially obtained from any public block explorer or by syncing. No proof-of-work, no keys, and no privileged access are required. The attack is repeatable at will with no ban or throttle consequence for well-formed messages.

## Recommendation
Bound the total work, not just the input count:
- Add a cap on the number of distinct blocks: `if txs_in_blocks.len() > SOME_BLOCK_LIMIT { return error; }` after the grouping step.
- Alternatively, cap the total number of transactions across all blocks being processed.
- Add per-peer rate limiting on `GetTransactionsProof` messages at the protocol handler level.

## Proof of Concept
1. Identify (via block explorer or local sync) 1000 blocks each containing multiple transactions.
2. Collect one tx_hash from each distinct block.
3. Send a single `GetTransactionsProof` P2P message containing all 1000 hashes with a valid `last_hash`.
4. Observe the server executing `CBMT::build_merkle_proof` 1000 times, each iterating over all transactions in its respective block — total blake2b work scales as `1000 × txs_per_block`, not `1000`.
5. Repeat in a tight loop; no ban is triggered because the message is well-formed and valid.

        &mut self,

                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
