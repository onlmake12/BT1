### Title
No Minimum TWAP Window Enforced Allows Near-Spot Price to Be Presented as TWAP - (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol` and `target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

Both the EVM `validateTwapPriceInfo` and the Solana `validate_twap_messages` functions fail to enforce any minimum slot-difference between the start and end TWAP messages. An unprivileged caller can freely select two Wormhole-attested TWAP messages from Hermes that are only 1 slot apart (~400 ms on Pythnet), causing `parseTwapPriceFeedUpdates` (EVM) or `post_twap_update` (Solana) to return/store a near-spot price disguised as a time-weighted average. Any downstream protocol that consumes this output without independently enforcing a minimum window size is exposed to the same class of manipulation described in the reference report.

---

### Finding Description

**EVM path — `Pyth.sol` `validateTwapPriceInfo`**

The only slot-ordering check is:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This allows `endSlot - startSlot = 1` (the minimum non-zero value; `slotDiff = 0` would cause a division-by-zero panic in `calculateTwap`). No lower bound on the window is enforced. [1](#0-0) 

`calculateTwap` then divides cumulative differences by `slotDiff`:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot;
int128 twapPrice = priceDiff / int128(uint128(slotDiff));
```

With `slotDiff = 1`, the result is simply the price increment over a single Pythnet slot — a spot price, not a time-weighted average. [2](#0-1) 

**Solana path — `lib.rs` `validate_twap_messages`**

The Solana validator requires strictly `end_slot > start_slot`, so `slotDiff >= 1`, but again imposes no minimum:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

`calculate_twap` then divides by `slot_diff`, which can be 1:

```rust
let slot_diff = end_msg.publish_slot.checked_sub(start_msg.publish_slot)...;
let price = i64::try_from(price_diff / i128::from(slot_diff))...;
``` [4](#0-3) 

**Caller-controlled window selection**

`parseTwapPriceFeedUpdates` accepts arbitrary `updateData` from any caller:

```solidity
function parseTwapPriceFeedUpdates(
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override returns (PythStructs.TwapPriceFeed[] memory twapPriceFeeds)
``` [5](#0-4) 

Similarly, `post_twap_update` on Solana accepts any two Wormhole-verified VAAs chosen by the caller: [6](#0-5) 

Hermes continuously publishes TWAP accumulator messages. An attacker can query Hermes, find two consecutive messages (1 slot apart) where the Pythnet aggregate price was at a desired level, and submit them. The contract will accept them as a valid TWAP pair.

**SDK warning acknowledges the gap**

The Solana SDK's `get_twap_unchecked` explicitly warns that it does not check window size:

```rust
/// # Warning
/// This function does not check :
/// - How recent the price is
/// - If the TWAP's window size is expected
``` [7](#0-6) 

`get_twap_no_older_than` does enforce an exact window match, but only in the off-chain SDK — not in the on-chain contract itself: [8](#0-7) 

The on-chain contract is the necessary vulnerable step: it produces and stores a `TwapPriceFeed`/`TwapUpdate` with an arbitrarily short window without any on-chain rejection.

---

### Impact Explanation

Any downstream protocol that calls `parseTwapPriceFeedUpdates` with attacker-supplied `updateData`, or reads a `TwapUpdate` account posted by an attacker, and does not independently enforce a minimum window size, will receive a near-spot price. This defeats the manipulation-resistance property that TWAP is intended to provide. A protocol using this price for collateral valuation, liquidation thresholds, or AMM pricing could be exploited in the same manner as the Inverse Finance attack described in the reference report — the attacker selects a 1-slot window that coincides with a brief price spike or dip on Pythnet.

---

### Likelihood Explanation

The attack requires:
1. An attacker to monitor Hermes for a 1-slot window where the Pythnet aggregate price is favorable (low barrier — Hermes is public).
2. A downstream protocol that passes user-supplied `updateData` to `parseTwapPriceFeedUpdates` without enforcing a minimum window, or reads a `TwapUpdate` account without using `get_twap_no_older_than`.

Pythnet's median-based aggregation makes it harder to engineer a specific price spike compared to an AMM, but natural price volatility over a single slot is sufficient. The barrier is moderate, making this a realistic attack against protocols that do not implement the SDK's window-size check.

---

### Recommendation

Add a minimum slot-difference check in both validation functions. For example, in `validateTwapPriceInfo` (EVM):

```solidity
uint64 MIN_TWAP_SLOTS = 150; // ~1 minute on Pythnet (~400ms/slot)
if (twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot < MIN_TWAP_SLOTS) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

And in `validate_twap_messages` (Solana):

```rust
const MIN_TWAP_SLOTS: u64 = 150;
require!(
    end_msg.publish_slot.saturating_sub(start_msg.publish_slot) >= MIN_TWAP_SLOTS,
    ReceiverError::InvalidTwapSlots
);
```

This mirrors the recommendation in the reference report: enforce a meaningful minimum observation window at the contract level, not just in off-chain SDK helpers.

---

### Proof of Concept

1. Query Hermes for historical TWAP accumulator messages for a target price feed.
2. Identify two consecutive messages (`slot N` and `slot N+1`) where the Pythnet aggregate price was at a desired manipulated level (e.g., during a brief spike).
3. On EVM: call `parseTwapPriceFeedUpdates{value: fee}([msg_N, msg_N+1], [priceId])`. The contract accepts the pair (slotDiff = 1 passes `validateTwapPriceInfo`), and returns a `TwapPriceFeed` whose `twap.price` equals `(cumulativePrice[N+1] - cumulativePrice[N]) / 1` — the spot price at slot N.
4. On Solana: post two Wormhole-verified VAAs for slot N and N+1 via `post_twap_update`. The `TwapUpdate` account is written with `start_time = publish_time[N]`, `end_time = publish_time[N+1]`, and `price = spot_price_at_N`.
5. A downstream protocol reading this result without enforcing a minimum window (e.g., using `get_twap_unchecked` instead of `get_twap_no_older_than`) receives a spot price and can be exploited accordingly.

    function validateTwapPriceInfo(
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure {
        // First validate each individual price's uniqueness
        if (
            twapPriceInfoStart.prevPublishTime >= twapPriceInfoStart.publishTime
        ) {
            revert PythErrors.InvalidTwapUpdateData();
        }
        if (twapPriceInfoEnd.prevPublishTime >= twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateData();
        }

        // Then validate the relationship between the two data points
        if (twapPriceInfoStart.expo != twapPriceInfoEnd.expo) {
            revert PythErrors.InvalidTwapUpdateDataSet();

            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishTime > twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }

```rust
    pub fn get_twap_no_older_than(
        &self,
        clock: &Clock,
        maximum_age: u64,
        window_seconds: u64,
        feed_id: &FeedId,
    ) -> std::result::Result<TwapPrice, GetPriceError> {
        // Ensure the update isn't outdated
        let twap_price = self.get_twap_unchecked(feed_id)?;
        check!(
            twap_price
                .end_time
                .saturating_add(maximum_age.try_into().unwrap())
                >= clock.unix_timestamp,
            GetPriceError::PriceTooOld
        );

        // Ensure the twap window size is as expected
        let actual_window = twap_price.end_time.saturating_sub(twap_price.start_time);
        check!(
            actual_window == i64::try_from(window_seconds).unwrap(),
            GetPriceError::InvalidWindowSize
        );

        Ok(twap_price)
    }
