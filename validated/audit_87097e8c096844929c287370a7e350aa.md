### Title
Unbounded `forward_rate_limiter` HashMap Growth via Unique `(from, to)` PeerId Pairs in `ConnectionRequestDelivered` — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

The `forward_rate_limiter` in the `HolePunching` protocol uses a `HashMapStateStore` keyed by `(PeerId, PeerId, u32)`. Its only cleanup call, `retain_recent()`, is invoked exclusively in `disconnected()`. An unprivileged remote peer can maintain a persistent connection and continuously send `ConnectionRequestDelivered` messages with unique `(from, to)` PeerId pairs, causing the HashMap to grow without bound for the entire lifetime of the connection.

---

### Finding Description

`HolePunching` declares two rate limiters:

- `rate_limiter: RateLimiter<(PeerIndex, u32)>` — keyed by session ID + message type, bounded by the finite number of connections.
- `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` — keyed by arbitrary attacker-controlled `(from, to, item_id)` tuples. [1](#0-0) 

Both rate limiters call `retain_recent()` only inside `disconnected()`: [2](#0-1) 

The `notify()` callback fires every 5 minutes and cleans up `pending_delivered` and `inflight_requests`, but never calls `retain_recent()` on either rate limiter: [3](#0-2) 

In `ConnectionRequestDeliveredProcess::execute()`, every unique `(from, to, item_id)` tuple that passes the outer rate limiter check causes `check_key()` to insert a new bucket into the `forward_rate_limiter`'s `HashMapStateStore`: [4](#0-3) 

The outer `rate_limiter` is keyed by `(session_id, item_id)` and allows 30 `ConnectionRequestDelivered` messages per second per session. This bounds the *insertion rate* into `forward_rate_limiter` to 30 entries/second per connection, but places no cap on the *total number of entries*. Since `retain_recent()` is never called during the connection, all inserted entries — including those that became stale after 1 second of inactivity — accumulate indefinitely.

---

### Impact Explanation

At 30 entries/second, each entry consuming roughly 150–200 bytes (two 39-byte PeerIds + u32 key + governor state + HashMap overhead):

| Duration | Entries | Approximate Memory |
|---|---|---|
| 1 hour | 108,000 | ~21 MB |
| 24 hours | 2.6 M | ~520 MB |
| 1 week | 18 M | ~3.6 GB |

A single attacker connection is sufficient. Multiple attacker connections compound the effect linearly. The `forward_rate_limiter` is a single shared instance on the `HolePunching` struct, so all sessions write into the same HashMap.

---

### Likelihood Explanation

The attacker needs only a standard P2P connection to the victim node — no special privileges, no PoW, no keys. The `ConnectionRequestDelivered` message is a valid protocol message. Generating unique PeerId values is trivial (random 39-byte multihash). The attack is sustainable indefinitely as long as the TCP connection remains open.

---

### Recommendation

1. **Periodic cleanup**: Call `self.forward_rate_limiter.retain_recent()` inside `notify()` (which already fires every 5 minutes) alongside the existing `pending_delivered` and `inflight_requests` cleanup.
2. **Absolute size cap**: Before inserting into `forward_rate_limiter`, check the HashMap size and reject or evict entries if it exceeds a configured maximum (e.g., 10,000 entries).
3. **Consider a `DashMap`-backed or LRU-bounded keyed rate limiter** that evicts the least-recently-used entries when a capacity limit is reached.

---

### Proof of Concept

```
1. Connect to victim node via P2P (standard unprivileged connection).
2. In a loop at 30 msg/sec:
     from_i = random PeerId
     to_i   = random PeerId
     send ConnectionRequestDelivered(from=from_i, to=to_i, route=[], listen_addrs=[valid_addr], sync_route=[])
3. Do NOT disconnect.
4. After 1 hour, observe victim process RSS has grown by ~21 MB from forward_rate_limiter alone.
5. After 24 hours, RSS growth ~520 MB; after several days, OOM kill or severe memory pressure.
```

The `forward_rate_limiter` HashMap size can be confirmed to grow linearly with no eviction until the connection is closed.
