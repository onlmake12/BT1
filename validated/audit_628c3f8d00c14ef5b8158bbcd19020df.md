### Title
Gameable Phase-2 Eviction in `check_purge` Allows Attacker to Eclipse Peer Store by Preserving Attacker Addresses While Evicting Honest Peers — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

`check_purge` phase-2 sorts network groups by size descending, takes the top half, and only evicts from groups with `> 4` peers. An attacker who floods `add_addr` (via the P2P discovery protocol) with addresses spread across many small groups (≤4 per group) ensures their addresses are immune to eviction, while honest peers concentrated in large groups are continuously evicted. This allows the attacker to gradually eclipse the peer store.

---

### Finding Description

`check_purge` is called on every `add_addr` invocation once the store reaches `ADDR_COUNT_LIMIT = 16384`. [1](#0-0) 

Phase 1 evicts only addresses that fail `is_connectable` (i.e., have been tried too many times recently). Fresh attacker-supplied addresses always pass this check, so phase 1 finds nothing and phase 2 runs: [2](#0-1) 

Phase 2 groups all addresses by network segment, sorts groups by size descending, takes the top `len/2` groups, and evicts 2 random peers **only from groups with `> 4` members**: [3](#0-2) 

The two critical lines are:

```
peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));  // largest groups first
peers.into_iter().take(len / 2)                               // top half only
    .flat_map(|addrs| {
        if addrs.len() > 4 { evict 2 } else { None }         // skip small groups
    })
```

An attacker who controls many `/24` subnets with ≤4 addresses each ensures:
- Their groups are small (≤4) → immune to the `> 4` eviction guard
- Their groups may appear in the top half (sorted by size), but still return `None`
- Honest peers concentrated in a few large groups are always in the top half and always evicted

**Concrete proof-of-concept state:**
- 1 honest group: 16,380 addresses (e.g., `225.0.0.x`)
- 4 attacker groups: 1 address each (4 different `/24` subnets)
- Total: 16,384 = `ADDR_COUNT_LIMIT`

Phase 2 execution:
- `len = 5`, `take(5/2) = take(2)` → selects `[16380, 1]`
- Group of 16,380: `> 4` → 2 honest peers evicted
- Group of 1: `≤ 4` → skipped
- Net result: attacker addresses untouched, honest peers removed

---

### Impact Explanation

The attacker can continuously drive honest peers out of the peer store. Once the store is dominated by attacker-controlled addresses, the victim node's outbound connection attempts (`fetch_addrs_to_attempt`, `fetch_addrs_to_feeler`) will predominantly return attacker addresses. This enables a full eclipse attack: the node is isolated from the honest network, making it susceptible to transaction relay manipulation, double-spend facilitation, and chain-tip spoofing.

---

### Likelihood Explanation

The entry point is the P2P discovery protocol. `DiscoveryAddressManager::add_new_addrs` calls `peer_store.add_addr` for every address received in a `DiscoveryMessage::Nodes` P2P message: [4](#0-3) 

The only guard is `is_valid_addr`, which checks `is_reachable(ip)` — a public IP range check, not actual connectivity. The attacker only needs to supply public IP addresses from many different `/24` subnets; they do not need to be actually listening. A single connected attacker peer can send up to 1,000 addresses per `Nodes` message: [5](#0-4) 

This requires no special privilege — any peer that completes a TCP connection can send `Nodes` messages. The attacker needs control of (or the ability to enumerate) many public IP prefixes, which is achievable with cloud infrastructure or BGP prefix announcements.

---

### Recommendation

1. **Remove the `> 4` guard from phase-2 eviction**, or apply it uniformly so small groups are also candidates for eviction proportionally.
2. **Cap per-group contribution**: limit the number of addresses accepted per network group during `add_addr`, not just during eviction.
3. **Evict from the largest group unconditionally**: instead of taking the top half and skipping small groups, always evict from the single largest group, regardless of size threshold.
4. **Rate-limit `add_addr` per session**: bound how many new addresses a single discovery session can inject.

### Proof of Concept

```rust
// Construct store: 1 honest group of 16380 + 4 attacker groups of 1
let mut peer_store = PeerStore::default();

// Fill honest group (225.0.0.x)
for i in 0..16380u32 {
    let addr: Multiaddr = format!(
        "/ip4/225.0.{}.{}/tcp/8115/p2p/{}",
        (i >> 8) & 0xff, i & 0xff,
        PeerId::random().to_base58()
    ).parse().unwrap();
    peer_store.addr_manager_mut().add(AddrInfo::new(addr, 0, 100, 0));
}

// 4 attacker groups (different /24 subnets), 1 address each
for subnet in [10u8, 11, 12, 13] {
    let addr: Multiaddr = format!(
        "/ip4/1.{}.0.1/tcp/8115/p2p/{}",
        subnet, PeerId::random().to_base58()
    ).parse().unwrap();
    peer_store.addr_manager_mut().add(AddrInfo::new(addr, 0, 100, 0));
}
// store is now at ADDR_COUNT_LIMIT (16384)

// Snapshot attacker addresses
let attacker_addrs: Vec<_> = (10u8..14).map(|s| {
    format!("/ip4/1.{}.0.1/tcp/8115/...", s)
}).collect();

// Trigger check_purge via add_addr
let new_addr: Multiaddr = "/ip4/8.8.8.8/tcp/8115/p2p/...".parse().unwrap();
peer_store.add_addr(new_addr, Flags::COMPATIBILITY).unwrap();

// Assert: all attacker addresses still present
for addr in &attacker_addrs {
    assert!(peer_store.addr_manager().get(addr).is_some(),
        "Attacker address was evicted — should not happen");
}
// Assert: honest group shrank by 2
let honest_count = peer_store.addr_manager().addrs_iter()
    .filter(|a| a.addr.to_string().contains("225.0."))
    .count();
assert_eq!(honest_count, 16378, "Honest peers were evicted");
```
