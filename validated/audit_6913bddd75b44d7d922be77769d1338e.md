### Title
Uncapped `tx.gasprice` in `_processFeesAndPayKeeper` Allows Keeper to Drain Subscription Balance - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `_processFeesAndPayKeeper` function in the Pulse Scheduler contract computes keeper reimbursement as `(gasUsed + GAS_OVERHEAD) * tx.gasprice` with no upper bound on `tx.gasprice`. Any unprivileged keeper can submit a valid `updatePriceFeeds` transaction with an artificially inflated gas price, draining the subscription's entire balance in a single call while netting a profit equal to approximately `GAS_OVERHEAD * tx.gasprice`.

---

### Finding Description

In `_processFeesAndPayKeeper`:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;

if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
status.balanceInWei -= totalKeeperFee;
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [1](#0-0) 

`GAS_OVERHEAD` is a fixed constant of 30,000 gas units, intended to cover transaction overhead: [2](#0-1) 

The keeper fee is bounded only by `status.balanceInWei` — there is no cap on `tx.gasprice`. The keeper is reimbursed for `gasUsed_measured + GAS_OVERHEAD` gas units at whatever price they set, but only pays the network for `gasUsed_total` gas units. Since `gasUsed_measured` is captured before the fee-calculation and payment code runs, `gasUsed_total > gasUsed_measured`, but the delta is small (a few thousand gas). The keeper therefore always nets a profit of approximately `(GAS_OVERHEAD - post_measurement_gas) * tx.gasprice`, which grows linearly with the chosen gas price.

The entry point is `updatePriceFeeds`, which is permissionless — any address can call it when update conditions are met: [3](#0-2) 

---

### Impact Explanation

A malicious keeper can drain a subscription's entire balance in a single `updatePriceFeeds` call. The keeper selects a `tx.gasprice` such that `totalKeeperFee ≈ status.balanceInWei`, pays the network `gasUsed_total * tx.gasprice`, and receives `totalKeeperFee` from the subscription. The net profit is `GAS_OVERHEAD * tx.gasprice + keeperSpecificFee - post_measurement_gas * tx.gasprice`.

**Concrete example** (subscription balance = 10 ETH, `gasUsed_total ≈ 200,000 gas`, `GAS_OVERHEAD = 30,000`):

- Target `totalKeeperFee = 10 ETH`: solve `(200,000 + 30,000) * P = 10 ETH` → `P ≈ 43,478 gwei`
- Keeper pays network: `200,000 * 43,478 gwei ≈ 8.70 ETH`
- Keeper receives from subscription: `10 ETH`
- **Net profit: ~1.30 ETH**

The subscription manager loses their entire deposited balance in one update cycle instead of the many cycles they funded for.

---

### Likelihood Explanation

- `updatePriceFeeds` is permissionless; any address qualifies as a keeper.
- The keeper only needs to wait for a heartbeat or deviation trigger to be satisfied — a routine event.
- The attack requires upfront capital (`gasUsed * inflated_gasprice`), but the profit is guaranteed as long as the subscription balance covers `totalKeeperFee`.
- Subscriptions with large balances (especially permanent ones, which cannot be withdrawn from) are high-value targets.

Likelihood: **Medium** — economically rational for any keeper with sufficient capital when a well-funded subscription exists.

---

### Recommendation

Cap the effective gas price used in keeper fee calculation to a protocol-defined maximum:

```solidity
uint256 effectiveGasPrice = tx.gasprice < MAX_GAS_PRICE ? tx.gasprice : MAX_GAS_PRICE;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, cap `totalKeeperFee` to a configurable maximum per update (e.g., a multiple of `singleUpdateKeeperFeeInWei * numPriceIds`). This mirrors the recommendation in the external report: set a maximum spend threshold and enforce it automatically.

---

### Proof of Concept

1. Deploy or identify a Scheduler subscription with a large `balanceInWei` (e.g., 10 ETH).
2. Wait for the heartbeat or deviation trigger condition to be satisfied.
3. Compute target gas price: `P = balanceInWei / (estimatedGasUsed + GAS_OVERHEAD)`.
4. Call `scheduler.updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = P`.
5. The contract computes `totalKeeperFee ≈ balanceInWei`, passes the balance check, deducts the full balance, and transfers it to `msg.sender`.
6. The keeper's net profit is `GAS_OVERHEAD * P + keeperSpecificFee - post_measurement_gas * P > 0`. [4](#0-3)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
