### Title
Unbounded `while` Loop in `executeCallback` Can Cause OOG, Permanently Locking Requester Funds — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` by scanning through every fulfilled (inactive) sequence number. An unprivileged caller can manufacture a large gap of fulfilled requests ahead of one still-active request, causing any subsequent `executeCallback` for that active request to run out of gas and revert, permanently locking the requester's funds.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs at least one cold `SLOAD` (2 100 gas) against `_state.requests[shortKey]` and, for overflow-mapped entries, a second cold `SLOAD` (2 100 gas) against `_state.requestsOverflow[key]`, plus an `SSTORE` (5 000 gas) to increment `firstUnfulfilledSeq`. The loop has no upper-bound guard; it runs until it either finds the next active request or exhausts `currentSequenceNumber`.

The code itself acknowledges the problem with a `TODO` comment immediately above the loop:

> "I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number." [2](#0-1) 

`executeCallback` is `external` with no access-control modifier; after the configurable exclusivity period any address may call it: [3](#0-2) 

The sequence counter is unbounded — `currentSequenceNumber` increments on every `requestPriceUpdatesWithCallback` call: [4](#0-3) 

---

### Impact Explanation

If the while loop requires more gas than the block gas limit (~30 M gas on Ethereum mainnet), the entire `executeCallback` transaction reverts. Because `clearRequest` is called before the loop, the revert also un-clears the request, leaving it perpetually active but unfulfillable. The requester's pre-paid fee (stored in `req.fee`) is permanently locked inside the contract with no recovery path.

At ~7 200 gas per iteration (warm `SLOAD` + `SSTORE`), the loop can exhaust a 30 M gas block after roughly **4 000 iterations**. For overflow-mapped entries (cold `SLOAD`), the threshold drops to roughly **3 200 iterations**.

---

### Likelihood Explanation

**Medium.** The attacker must:

1. Call `requestPriceUpdatesWithCallback` N times, paying the protocol fee each time. Fees are set by the provider and can be low.
2. Obtain valid Wormhole-attested Pyth price-update data for the chosen `publishTime` values (freely available from the public Hermes API).
3. After the exclusivity period, call `executeCallback` for requests 2 … N, leaving request 1 unfulfilled.
4. Request 1 is now permanently unfulfillable.

The financial cost scales with N and the fee level, but the attack is fully permissionless and requires no privileged access.

---

### Recommendation

Replace the unbounded scan with a doubly-linked list of active requests (as the in-code `TODO` suggests), or cap the number of iterations per call and allow `firstUnfulfilledSeq` to be advanced in a separate, paginated transaction. This mirrors the fix recommended in the original report: allow partial processing in a single transaction rather than requiring the entire scan to complete atomically.

---

### Proof of Concept

```
1. Deploy Echo with a low pythFeeInWei and a provider with low fees.
2. Attacker calls requestPriceUpdatesWithCallback(...) 4000 times,
   each with publishTime = T (a timestamp for which Hermes data exists).
   → currentSequenceNumber = 4001, firstUnfulfilledSeq = 1.
3. After the exclusivity period, attacker calls executeCallback for
   sequenceNumbers 2 … 4000 (providing valid Hermes updateData for T).
   → Each call succeeds; firstUnfulfilledSeq stays at 1 because
     request 1 is still active.
4. Legitimate provider (or anyone) calls executeCallback for sequenceNumber 1.
   → clearRequest(1) executes.
   → The while loop begins scanning from seq 1 upward.
   → After ~4000 iterations (each costing ~7200 gas), the transaction
     runs out of gas and reverts.
   → clearRequest(1) is also reverted; request 1 is still "active".
5. Every subsequent attempt to fulfill request 1 also OOGs.
   → req.fee is permanently locked in the Echo contract.
```
