### Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Yields Silently Wrong Withdrawal Capacity — (File: `util/dao/src/lib.rs`)

### Summary
`DaoCalculator::calculate_maximum_withdraw` computes the DAO withdrawal capacity using u128 arithmetic to avoid intermediate overflow, but then casts the final result back to u64 with a silent truncating `as u64` cast instead of a checked conversion. If the u128 result exceeds `u64::MAX`, the returned capacity is silently truncated to a fraction of the correct value — the exact analog of the reported "missing minimum value protection" class.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a **silent truncating cast**: if `withdraw_counted_capacity > u64::MAX`, the high bits are discarded and the function returns `Capacity::shannons(withdraw_counted_capacity % 2^64)` — a value that can be arbitrarily smaller than the correct withdrawal amount — without returning any error.

Compare this to the **correct** pattern used in the same file for `secondary_block_reward`:

```rust
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

That function propagates an explicit `DaoError::Overflow`. `calculate_maximum_withdraw` does not.

The existing overflow test (`check_withdraw_calculation_overflows`) only catches the case where the truncated value is still large enough that the subsequent `safe_add(occupied_capacity)` overflows. It does **not** cover the case where truncation wraps to a small value and `safe_add` succeeds silently. [3](#0-2) 

### Impact Explanation

`calculate_maximum_withdraw` is called from two paths:

1. **`transaction_maximum_withdraw` → `transaction_fee`** — used by the tx-pool's `check_tx_fee` to compute the fee for a DAO withdrawal transaction. A silently truncated `maximum_withdraw` causes `maximum_withdraw.safe_sub(outputs_capacity)` to underflow, producing a `DaoError` that causes the tx-pool to **incorrectly reject a valid DAO withdrawal transaction** (denial of service for the depositor). [4](#0-3) [5](#0-4) 

2. **`calculate_dao_maximum_withdraw` RPC** — returns a silently wrong (far too small) capacity to the caller, misleading wallets and tooling about the actual withdrawal amount. [6](#0-5) 

### Likelihood Explanation

**Low.** The truncation requires `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX ≈ 1.84×10¹⁹`. The genesis `ar` is `10^10`; for the ratio `withdrawing_ar / deposit_ar` to reach ~5.5× (needed to overflow with the full ~3.36×10¹⁸ shannon supply), the secondary issuance accumulation rate would need to grow for an extremely long time. However, the defect is a latent correctness bug that is inconsistent with the safe conversion pattern used elsewhere in the same file, and it is reachable by any transaction sender submitting a DAO withdrawal transaction.

### Recommendation

Replace the silent cast with a checked conversion, consistent with the pattern already used in `secondary_block_reward`:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
```

### Proof of Concept

```
deposit_ar       = 10_000_000_000_000_000   (10^16, reachable after long accumulation)
withdrawing_ar   = 20_000_000_000_000_000   (2× deposit_ar)
output_capacity  = 10_000_000_000_000_000   (10^16 shannons, ~10^7 CKB)
occupied_capacity = 0

withdraw_counted_capacity (u128) = 10^16 * 2×10^16 / 10^16 = 2×10^16
```

In this example the result fits in u64 and is correct. To trigger the truncation, scale `output_capacity` to near the total supply and use a sufficiently grown `ar` ratio so that `counted_capacity * withdrawing_ar / deposit_ar > 2^64`. The `as u64` cast then wraps to a small value; `safe_add(occupied_capacity)` succeeds; the function returns a silently wrong small capacity. Any subsequent call to `transaction_fee` that subtracts `outputs_capacity` from this small value underflows and the withdrawal transaction is rejected by the tx-pool. [7](#0-6)

    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())

            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,

    // reject txs which fee lower than min fee rate

            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
