### Title
Unbounded `Nodes{announce:true}` Flooding Enables Peer Store Poisoning — (`network/src/protocols/discovery/mod.rs`)

### Summary

There is no per-session rate limit on inbound `Nodes{announce:true}` messages. `verify_nodes_message` enforces only a per-message item cap (≤10 items), but an attacker can send an unlimited number of such messages. Each one unconditionally calls `add_new_addrs()`, which writes attacker-controlled addresses into the peer store. An attacker operating any reachable node — especially a seed node — can fill the 16,384-entry peer store with fake addresses, displacing legitimate peers and enabling eclipse or network-partition attacks.

---

### Finding Description

**`SessionState::new()` — outbound path** (`state.rs` lines 45–79):

For outbound sessions the victim sends `GetNodes` automatically and sets `remote_addr = RemoteAddress::Listen(...)`. The `received_get_nodes` and `received_nodes` flags are both initialised to `false`. [1](#0-0) 

**`received` handler — `Nodes` branch** (`mod.rs` lines 170–208):

- `verify_nodes_message` rejects a single message only if it carries **>10 items** (announce) or **>1000 items** (non-announce), or **>3 addresses per item**.
- For `announce=true` the `received_nodes` flag is **never set**, so the duplicate-first-nodes guard never fires.
- Every passing message unconditionally calls `state.addr_known.extend(addrs.iter())` and `self.addr_mgr.add_new_addrs(session.id, addrs)`.
- **There is no per-session counter, token bucket, or any other rate limit on how many announce messages may be processed.** [2](#0-1) [3](#0-2) 

**`add_new_addrs` → `peer_store.add_addr()`** (`mod.rs` lines 347–363, `peer_store_impl.rs` lines 71–80):

Errors from `add_addr` (including `EvictionFailed`) are only `debug!`-logged; the loop continues inserting the remaining addresses. [4](#0-3) [5](#0-4) 

**`check_purge` eviction** (`peer_store_impl.rs` lines 327–404):

The peer store is capped at `ADDR_COUNT_LIMIT = 16384`. When full it first evicts non-connectable peers, then evicts 2 peers from each network group that has **>4** entries. An attacker who spreads fake addresses across many distinct /16 groups (e.g. one address per group) defeats both eviction passes and causes `EvictionFailed`, which is silently ignored by the caller. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

An attacker who fills the peer store with 16,384 fake addresses from diverse network groups can prevent the victim from discovering or reconnecting to honest peers after a restart, enabling an eclipse attack. Because seed nodes are connected to outbound by many nodes simultaneously, a single malicious seed node can poison the peer store of every node that connects to it.

---

### Likelihood Explanation

The attack requires only that the victim initiate (or accept) a single TCP connection to the attacker. No PoW, no key material, no privileged role is needed. The attacker sends crafted `Nodes{announce:true}` messages in a tight loop; each message is ≤10 items × 3 addresses = 30 addresses, well within the per-message limit, so no misbehavior is ever triggered. The entire peer store can be filled in seconds over a normal connection.

---

### Recommendation

1. **Add a per-session announce-message counter.** Disconnect (and score as misbehaving) any session that sends more than a small threshold (e.g. `ANNOUNCE_THRESHOLD = 10`) of `announce=true` Nodes messages within a sliding window.
2. **Treat `EvictionFailed` as a hard stop** in `add_new_addrs` rather than silently continuing.
3. **Apply source-IP diversity limits at insertion time** so that a single peer cannot contribute more than a fixed fraction of the peer store regardless of how many messages it sends.

---

### Proof of Concept

```
1. Attacker starts a CKB-compatible node.
2. Victim connects outbound (or attacker connects inbound — both paths are vulnerable).
3. SessionState::new() sends GetNodes for outbound; attacker ignores it.
4. Attacker sends in a loop:
       Nodes { announce: true, items: [Node{addr:1.0.0.1:8115}, …×10] }
   using a fresh /16 subnet per batch to defeat group-based eviction.
5. After ceil(16384/30) ≈ 547 messages the peer store is full of attacker addresses.
6. Victim restarts; its only known peers are attacker-controlled nodes → eclipse.
```

No integration test currently asserts that more than `ANNOUNCE_THRESHOLD` announce messages from a single session triggers misbehavior scoring. [8](#0-7) [9](#0-8)

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {
                            Some(
                                addrs
                                    .iter()
                                    .choose_multiple(&mut rand::thread_rng(), 2)
                                    .into_iter()
                                    .map(|addr| addr.addr.clone())
                                    .collect::<Vec<Multiaddr>>(),
                            )
                        } else {
                            None
                        }
                    })
                    .flatten()
                    .collect()
            };

            for key in candidate_peers.iter() {
                self.addr_manager.remove(key);
            }

            if candidate_peers.is_empty() {
                return Err(PeerStoreError::EvictionFailed.into());
            }
        }
        Ok(())
    }
