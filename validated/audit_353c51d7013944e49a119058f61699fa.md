Audit Report

## Title
Missing IBD Phase Guard in `GetBlocksProcess::execute` Allows Inbound Peers to Drain Resources During Initial Block Download — (File: sync/src/synchronizer/get_blocks_process.rs)

## Summary
The module-level documentation in `sync/src/synchronizer/mod.rs` explicitly contracts that during IBD the node will respond `packed::InIBD` to both `GetHeaders` and `GetBlocks` requests. `GetHeadersProcess::execute` enforces this contract, but `GetBlocksProcess::execute` contains no equivalent guard. Any inbound peer can send `GetBlocks` messages during IBD and receive full serialized block bodies for every hash that resolves to a `BLOCK_VALID` block, wasting the IBD node's I/O, CPU, and bandwidth.

## Finding Description
The module comment at `sync/src/synchronizer/mod.rs` L4 states the invariant:

> "When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` and `GetBlocks` requests"

`GetHeadersProcess::execute` enforces this at `sync/src/synchronizer/get_headers_process.rs` L53–65 by calling `active_chain.is_initial_block_download()`, sending `InIBD`, and returning `Status::ignored()`.

`GetBlocksProcess::execute` at `sync/src/synchronizer/get_blocks_process.rs` L33–97 performs no such check. It immediately processes up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` hashes from the message (bounded by `MAX_HEADERS_LEN` = 2048 at L36), checks `BLOCK_VALID` status for each, and for every matching block spawns an async task that serializes and sends the full block body to the requesting peer (L78–83). Both handlers are dispatched from the same `try_process` match arm at `sync/src/synchronizer/mod.rs` L396–422 with no IBD pre-filter.

The `is_initial_block_download` check at `shared/src/shared.rs` L382–394 is a cheap atomic load once IBD has ended (one-way latch), so adding the guard has negligible overhead.

## Impact Explanation
During IBD the node is supposed to dedicate its outbound bandwidth to catching up with the chain tip. Without the guard, any inbound peer can force the node to serialize and transmit full block bodies for up to 2048 hashes per message. Each matching block spawns an independent async task. An attacker with multiple inbound connections can continuously flood the node with `GetBlocks` messages, consuming significant I/O, CPU, and bandwidth that should be reserved for IBD progress. This maps to **Low (501–2000 points): Any other important performance improvements for CKB** — the concrete impact is degraded IBD performance and wasted node resources, not a crash or network-wide congestion.

## Likelihood Explanation
No special privileges are required. Any peer that establishes a TCP connection during IBD can send a valid molecule-encoded `SyncMessage::GetBlocks` message. The attack is low-cost, repeatable, and requires no knowledge of the node's internal state beyond knowing it is in IBD (which is publicly observable from the `InIBD` responses to `GetHeaders`).

## Recommendation
Add an IBD guard at the top of `GetBlocksProcess::execute`, mirroring `GetHeadersProcess::execute`:

```rust
pub fn execute(self) -> Status {
    let active_chain = self.synchronizer.shared.active_chain();

    if active_chain.is_initial_block_download() {
        // Mirror GetHeadersProcess: send InIBD and ignore the request.
        self.send_in_ibd(); // optional, for well-behaved peer signaling
        return Status::ignored();
    }

    let block_hashes = self.message.block_hashes();
    // ... rest of existing logic
}
```

Optionally expose a `send_in_ibd` helper on `GetBlocksProcess` (as exists on `GetHeadersProcess` at `sync/src/synchronizer/get_headers_process.rs` L101–115) to notify the requesting peer of the node's IBD state.

## Proof of Concept
1. Start a CKB node from genesis so it enters IBD (`is_initial_block_download()` returns `true`).
2. Connect an inbound peer over the Sync protocol.
3. Send a `SyncMessage::GetBlocks` message containing up to 2048 block hashes known to be `BLOCK_VALID` on the target node.
4. Observe that the node responds with `SendBlock` messages for each valid hash rather than `InIBD`, confirming the guard is absent.
5. Repeat in a loop from multiple connections; measure the target node's bandwidth and CPU consumption vs. IBD progress rate.

The missing guard is confirmed by direct code inspection: `sync/src/synchronizer/get_blocks_process.rs` L33–97 contains no call to `is_initial_block_download()`, while the analogous `sync/src/synchronizer/get_headers_process.rs` L53–65 does.
