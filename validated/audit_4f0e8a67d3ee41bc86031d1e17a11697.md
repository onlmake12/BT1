Audit Report

## Title
Missing Per-Peer Rate Limit on `GetBlocksProof` Enables Amplified DB/MMR DoS — (`util/light-client-protocol-server/src/lib.rs`, `util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary

`LightClientProtocol` contains no rate limiter at any layer. Any connected peer can send `GetBlocksProof` messages at line rate, each containing up to 1,000 valid block hashes, forcing up to 3,000 synchronous RocksDB reads plus one `mmr.gen_proof(1000 positions)` call per message with zero throttling. Because `LightClientProtocol` shares the same `Shared` (and therefore the same RocksDB instance) as the main chain processing loop, sustained high-rate requests from a single peer can saturate DB I/O and CPU, degrading or crashing the node.

## Finding Description

**Root cause — no rate limiter in `LightClientProtocol`:**

`LightClientProtocol` holds only `pub shared: Shared` with no rate-limiter field. [1](#0-0) 

`try_process` dispatches directly to handlers with no rate-limit gate before any handler is invoked. [2](#0-1) 

**Contrast — `Relayer` has an explicit 30 req/s keyed rate limiter applied before dispatch:** [3](#0-2) 

**Contrast — `HolePunching` has two keyed rate limiters checked before any handler:** [4](#0-3) 

**Amplification factor — `GET_BLOCKS_PROOF_LIMIT = 1000`:** [5](#0-4) 

**Per-message DB cost — 3 reads per found hash (`get_block_header` + `get_block_uncles` + `get_block_extension`):** [6](#0-5) 

**Per-message MMR cost — `mmr.gen_proof(items_positions)` over up to 1,000 positions:** [7](#0-6) 

**`LightClient` is in `default_support_all_protocols()` and therefore enabled by default:** [8](#0-7) 

**`LightClient` is registered in the launcher unconditionally when present in config:** [9](#0-8) 

**Max frame length is 2 MB — easily accommodates 1,000 × 32-byte hashes:** [10](#0-9) 

## Impact Explanation

Each `GetBlocksProof` message with 1,000 valid block hashes triggers up to 3,000 synchronous RocksDB reads and one `mmr.gen_proof(1000)` call. `LightClientProtocol` shares the same `Shared` state and RocksDB instance as the main chain processing loop. A single peer sending these messages at line rate saturates DB I/O and CPU, directly degrading block validation throughput and potentially crashing the node.

**Impact: High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

- The protocol is on by default in `ckb.toml` and `default_support_all_protocols()`.
- Any peer that can establish a TCP connection can open the `/ckb/lightclient` sub-protocol (protocol ID 120) and send well-formed `GetBlocksProof` messages.
- No PoW, stake, or privileged role is required.
- 1,000 valid canonical block hashes are trivially obtained by syncing headers via the `Sync` protocol first.
- The attack is repeatable and local-testable.

## Recommendation

Add a per-peer, per-message-type keyed rate limiter to `LightClientProtocol`, mirroring the pattern already used in `Relayer` and `HolePunching`:

```rust
pub struct LightClientProtocol {
    pub shared: Shared,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

Apply the check at the top of `try_process`, before dispatching to any handler. A quota of 1–5 req/s per peer per message type is sufficient for legitimate light-client usage while eliminating the amplification window.

## Proof of Concept

1. Sync a full node to height H, recording 1,000 canonical block hashes `h_1..h_1000`.
2. Connect a crafted peer that opens `SupportProtocols::LightClient` (protocol ID 120).
3. In a tight loop, send `GetBlocksProof { last_hash: tip_hash, block_hashes: [h_1..h_1000] }`.
4. Concurrently measure the full node's block validation throughput (blocks/s) and RocksDB read latency.
5. Assert that throughput does not degrade below an acceptable threshold — it will, because each message forces 3,000 DB reads + `mmr.gen_proof(1000)` with no throttle.
