### Title
Unbounded Growth of `pending_compact_blocks` Map via Malicious Relay Peers — (`File: sync/src/relayer/compact_block_process.rs`)

### Summary
The `pending_compact_blocks` map in the relay protocol accumulates entries whenever a compact block cannot be fully reconstructed (missing transactions or short-ID collision). Cleanup of this map only occurs on the **success path** when a block is accepted. An unprivileged relay peer can drive unbounded growth of this map by repeatedly sending compact blocks that intentionally fail reconstruction, consuming memory proportional to the number of distinct block hashes sent within the current epoch window.

### Finding Description

In `sync/src/relayer/compact_block_process.rs`, when a compact block arrives and reconstruction fails (either `ReconstructionResult::Missing` or `ReconstructionResult::Collided`), `missing_or_collided_post_process` is called: [1](#0-0) 

This inserts the full `CompactBlock` struct plus peer metadata into the shared `pending_compact_blocks` map. The map is defined as: [2](#0-1) 

The **only** cleanup path is inside the success branch of `CompactBlockProcess::execute`, which runs `retain` to drop entries from old epochs and calls `shrink_to_fit!`: [3](#0-2) 

There is no periodic timer-based eviction, no maximum size cap, and no rate limit on insertions. The `or_insert_with` pattern means each unique block hash occupies one slot, but an attacker can supply many distinct block hashes within the valid epoch window.

The "already pending" guard only prevents the **same peer** from re-adding the **same block hash**: [4](#0-3) 

Different block hashes, or the same hash from different peers, bypass this guard.

The height window check in `non_contextual_check` limits valid block numbers to within one epoch of the tip: [5](#0-4) 

An epoch is approximately 1800 blocks (~4 hours). Within this window, an attacker can craft up to ~1800 distinct compact block headers (each with a valid PoW-free header structure that passes `non_contextual_check` and `contextual_check`), each inserted into `pending_compact_blocks` with a full `CompactBlock` payload.

### Impact Explanation

Each `pending_compact_blocks` entry holds a `CompactBlock` (which includes short IDs, prefilled transactions, proposals, and uncles) plus a `HashMap<PeerIndex, (Vec<u32>, Vec<u32>)>`. With many peers each sending many distinct compact blocks that fail reconstruction, the map grows without bound until a legitimate block is accepted and triggers the epoch-based `retain` cleanup. During periods of low block acceptance (e.g., during IBD or network partition), memory usage can grow significantly, potentially causing OOM or severe performance degradation on the CKB node.

### Likelihood Explanation

Any peer connected via the relay protocol (`SupportProtocols::RelayV3`) can send `CompactBlock` messages. No authentication or stake is required. The attacker only needs to craft compact block headers that pass the non-contextual and contextual checks (valid structure, within epoch window, unknown parent or valid parent) and then ensure reconstruction fails by omitting transactions. This is straightforward for a motivated attacker with a P2P connection to the target node.

### Recommendation

Add a maximum size cap to `pending_compact_blocks` (e.g., reject new insertions when the map exceeds a configurable limit). Additionally, introduce a periodic timer-based eviction that removes entries older than a configurable TTL (e.g., 30 seconds), independent of whether a block is successfully accepted. This mirrors the fix pattern described in CHANGELOG entry `#3110` and ensures the map cannot grow unboundedly even when no blocks are being accepted.

### Proof of Concept

1. Attacker connects to a CKB node as a relay peer.
2. Attacker constructs N compact block messages, each with a distinct block hash within the current epoch window, each with `short_ids` referencing transactions not present in the target node's tx-pool.
3. Each message passes `non_contextual_check` (valid structure, within epoch) and `contextual_check` (unknown or valid parent, not already stored).
4. Each triggers `ReconstructionResult::Missing`, calling `missing_or_collided_post_process`, inserting into `pending_compact_blocks`.
5. Since no legitimate block is accepted, the `retain`-based cleanup never runs.
6. The map grows to N entries, each holding a full `CompactBlock` struct, consuming memory proportional to N × (compact block size).
7. Repeating across multiple peers multiplies the effect. [6](#0-5) [7](#0-6)

        .state()
