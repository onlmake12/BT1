Audit Report

## Title
Unauthenticated `observed_addr` Injection Poisons Victim's Advertised Listen Addresses — (`network/src/protocols/identify/mod.rs`)

## Summary
A remote peer can send an `IdentifyMessage` containing an arbitrary `observed_addr`. The victim stores this address without validating it against the actual TCP source of the session and subsequently re-broadcasts it to every subsequent peer as part of its own listen addresses. The bogus address persists in `observed_addrs` even after the attacker disconnects, since no cleanup is performed on session teardown.

## Finding Description

**Root cause — no source validation in `process_observed`:**
`process_observed` passes the attacker-supplied `observed` address directly to `add_observed_addr` with no check that the IP matches `info.session.address` (the actual remote address of the connection). It unconditionally returns `MisbehaveResult::Continue`. [1](#0-0) 

**Unconditional storage in `NetworkState`:**
`IdentifyCallback::add_observed_addr` only appends the local peer ID if absent, then writes the attacker-controlled address into `observed_addrs` keyed by `session_id`. No cross-check against the session's actual remote address occurs. [2](#0-1) 

**Poisoned address included in outbound Identify messages:**
`local_listen_addrs` extends the victim's public address list with `observed_addrs` whenever fewer than `MAX_RETURN_LISTEN_ADDRS` (10) configured public addresses exist — the common case for most nodes. [3](#0-2) 

**Broadcast to every subsequent peer:**
On each new connection, the victim sends an `IdentifyMessage` whose `listen_addrs` is built from `local_listen_addrs()`, filtered only by `is_reachable`. Any globally routable IP (e.g., `1.2.3.4`) passes this filter and is included in the outbound message. [4](#0-3) 

**No cleanup on disconnect:**
`disconnected` removes the `RemoteInfo` entry and calls `unregister`, but `unregister` only updates the peer store for outbound sessions. The bogus entry in `observed_addrs` is never removed, so the poisoned address persists indefinitely. [5](#0-4) 

**Storage structure:**
`observed_addrs` is a `HashMap<PeerIndex, Multiaddr>` with one slot per session. Multiple attacker connections fill multiple slots with distinct bogus addresses, all of which are eligible for inclusion in outbound Identify messages. [6](#0-5) 

`observed_addrs()` deduplicates by value but selects randomly up to `count` entries, so multiple distinct bogus addresses from multiple attacker sessions can all be returned and broadcast. [7](#0-6) 

## Impact Explanation

This matches **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs (10001–15000 points)**. An attacker with only a standard P2P connection can inject bogus addresses that propagate to every peer the victim subsequently connects to. Third-party nodes receiving the poisoned Identify message will attempt to dial the bogus address, consuming connection slots and dial resources. With multiple attacker sessions (or repeated reconnections), the victim's entire `observed_addrs` budget can be filled with bogus entries, maximizing the propagation of bad addresses across the network. Because the entries are never cleaned up on disconnect, the effect is persistent and cumulative across the victim's uptime.

## Likelihood Explanation

- Requires only a standard P2P connection — no keys, no privileges, no hashpower.
- The Identify protocol is the first protocol opened on every connection.
- Most production nodes have fewer than 10 configured `public_addrs`, making the `observed_addrs` extension path active by default.
- The attack is trivially reproducible with a minimal custom P2P client that sends a crafted `IdentifyMessage`.
- Persistence after disconnect means a brief connection is sufficient to leave a lasting effect.

## Recommendation

In `process_observed` (or in `IdentifyCallback::add_observed_addr`), validate that the IP portion of the received `observed_addr` matches the actual remote IP of the session (`info.session.address`). Reject and disconnect on mismatch. Additionally:
1. Remove the session's entry from `observed_addrs` in the `disconnected` handler.
2. Consider requiring a minimum quorum of independent peers reporting the same observed address before accepting it (consensus-based NAT detection, as used by Bitcoin Core and libp2p).

## Proof of Concept

1. Connect to the victim node and complete the Identify handshake normally (send a valid `identify` field to pass `received_identify`).
2. In the same `IdentifyMessage`, set `observed_addr` to `/ip4/1.2.3.4/tcp/9999` (any globally routable address).
3. Maintain the connection (or disconnect — the entry persists either way).
4. Have a third node connect to the victim.
5. Observe that the victim's Identify message to the third node includes `/ip4/1.2.3.4/tcp/9999/p2p/<victim_peer_id>` in its `listen_addrs`.
6. Repeat with multiple attacker sessions using distinct bogus addresses to fill the victim's `observed_addrs` map and maximize propagation.

    pub(crate) fn observed_addrs(&self, count: usize) -> Vec<Multiaddr> {
        let observed_addrs = self
            .observed_addrs
            .read()
            .values()
            .cloned()
            .collect::<HashSet<_>>();
        if observed_addrs.len() <= count {
            return observed_addrs.into_iter().collect();
        } else {
            observed_addrs
                .into_iter()
                .choose_multiple(&mut rand::thread_rng(), count)
        }
    }
