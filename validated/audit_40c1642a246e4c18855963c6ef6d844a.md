### Title
Stale Delegation Record After `set_publisher_stake_account` Allows Reward Accounting Bypass — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `set_publisher_stake_account` instruction in the OIS integrity pool program changes the publisher's self-stake account without requiring the `delegation_record` (the per-delegator checkpoint) to be advanced first. This leaves the old delegation record in a permanently stale state, causing incorrect reward and slash accounting for any delegator whose record references the old publisher stake account.

---

### Finding Description

The `DelegationRecord` struct — with fields `last_epoch` and `next_slash_event_index` — is the on-chain checkpoint that tracks reward and slash accounting per `(publisher, stake_account_positions)` pair. [1](#0-0) 

The `advance_delegation_record` instruction is the only mechanism that advances `last_epoch` and distributes rewards. It is permissionless and must be called before any state-changing operation that alters the delegation relationship. [2](#0-1) 

The `set_publisher_stake_account` instruction updates `pool_data.publisherStakeAccounts[index]` to point to a new stake account. Critically, its account list contains **no `delegation_record`** and imposes **no requirement** that `advance_delegation_record` be called first: [3](#0-2) 

After the switch, the old publisher stake account's delegation record retains a stale `last_epoch`. When `advance_delegation_record` is subsequently called for any delegator whose record references the old publisher stake account, the instruction fetches `publisher_stake_account_positions` from `pool_data` — which now returns the **new** account — and uses its custody balance to compute the publisher's self-stake ratio. This produces incorrect reward amounts for all delegators who staked to the publisher via the old account.

The protocol itself acknowledges the danger of stale records via the `outdatedDelegatorAccounting` error (code 6008), but this guard is not enforced in `set_publisher_stake_account`. [4](#0-3) 

The SDK's `setPublisherStakeAccount` wrapper sends only the single `set_publisher_stake_account` instruction with no preceding `advance_delegation_record` call: [5](#0-4) 

---

### Impact Explanation

- **Delegator reward loss**: When the reward rate `y` is non-zero, delegators who staked to the publisher via the old stake account will have their rewards computed against the wrong publisher custody balance, resulting in under- or over-payment.
- **Slash accounting corruption**: The `slash` instruction updates `delegation_record.next_slash_event_index`. If the delegation record was never advanced through intermediate epochs before the publisher switched accounts, slash events for those epochs are silently skipped or misapplied.
- **Permanent record staleness**: Because `advance_delegation_record` uses `pool_data` to look up the publisher stake account, and `pool_data` now points to the new account, it is impossible to correctly advance the old delegation record after the switch.

Currently `y = 0` per OP-PIP-103, so no rewards are distributed. However, the slashing mechanism remains active, and the vulnerability will have full financial impact if `y` is set to a non-zero value in the future. [6](#0-5) 

---

### Likelihood Explanation

- Publishers are expected to switch stake accounts as part of normal operations (e.g., key rotation, account migration).
- The SDK's `reassignPublisherStakeAccount` helper makes this a one-step operation with no warning about advancing the delegation record first.
- The `set_publisher_stake_account` instruction is permissionless for the publisher (they sign it), making this reachable by any staking user who is a publisher. [7](#0-6) 

---

### Recommendation

Before updating `pool_data.publisherStakeAccounts[index]`, the `set_publisher_stake_account` instruction should require that the delegation record for the current publisher stake account is up-to-date (i.e., `delegation_record.last_epoch == current_epoch`). This can be enforced by:

1. Adding `delegation_record` (for the current publisher stake account) as a required account in `set_publisher_stake_account`, and
2. Asserting `delegation_record.last_epoch == current_epoch` before proceeding, returning `OutdatedDelegatorAccounting` otherwise.

The SDK wrapper `setPublisherStakeAccount` should also be updated to prepend the necessary `advance_delegation_record` instructions before calling `set_publisher_stake_account`.

---

### Proof of Concept

1. Publisher P has self-stake account `old_acct` with `delegation_record.last_epoch = E` (several epochs behind current epoch `E+N`).
2. Publisher P calls `set_publisher_stake_account(publisher=P, current=old_acct, new=new_acct)`. No `advance_delegation_record` is called. `pool_data.publisherStakeAccounts[P_index]` is now `new_acct`.
3. Delegator D has staked to publisher P. D calls `advance_delegation_record(publisher=P, stake_account=D_acct)`.
4. The instruction fetches `publisher_stake_account_positions = new_acct` from `pool_data`, but the delegation record being advanced is for `(P, D_acct)` which was created when `old_acct` was the publisher stake account.
5. Reward computation uses `new_acct`'s custody balance (potentially 0 or a different amount) instead of `old_acct`'s balance, producing incorrect rewards for D for epochs `E` through `E+N`.
6. The old delegation record for `(P, old_acct)` can never be correctly advanced, permanently locking any accrued rewards or slash obligations in an unresolvable stale state. [8](#0-7)

  async setPublisherStakeAccount(
    publisher: PublicKey,
    stakeAccountPositions: PublicKey,
    newStakeAccountPositions: PublicKey | undefined,
  ) {
    const instruction = await this.integrityPoolProgram.methods
      .setPublisherStakeAccount()
      .accounts({
        currentStakeAccountPositionsOption: stakeAccountPositions,
        // eslint-disable-next-line unicorn/no-null
        newStakeAccountPositionsOption: newStakeAccountPositions ?? null,
        publisher,
      })
      .instruction();

    await sendTransaction([instruction], this.connection, this.wallet);
    return;
  }
