### Title
No Replay Protection or Timestamp Validation in `verifyUpdate()` Allows Stale Price Replay — (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.sol::verifyUpdate()` performs only ECDSA signature verification and signer-expiry checks. It stores no consumed-update state, enforces no timestamp freshness, and imposes no sequence ordering. Any previously-seen valid signed Lazer update can be replayed by an unprivileged caller indefinitely (until the signer key expires), causing consumer contracts that rely solely on `verifyUpdate()` to accept arbitrarily stale prices.

### Finding Description
The `verifyUpdate()` function in `PythLazer.sol` is the sole on-chain verification primitive for the Lazer (Pyth Pro) price-feed system on EVM chains. Its full logic is:

1. Check `msg.value >= verification_fee`.
2. Validate the EVM format magic number.
3. Recover the ECDSA signer from the embedded signature.
4. Assert `isValidSigner(signer)` — i.e., `block.timestamp < trustedSignerToExpiresAtMapping[signer]`.
5. Return `(payload, signer)`. [1](#0-0) 

There is no:
- Consumed-update registry (no hash set, no nonce map).
- Timestamp extraction or comparison against `block.timestamp`.
- Sequence-number monotonicity check.

The payload contains a `uint64 timestamp` field (parsed by `PythLazerLib.parsePayloadHeader`), but `verifyUpdate()` never reads it. [2](#0-1) 

The Cardano integration documentation explicitly acknowledges this gap: *"It does not enforce freshness of the update, nor does it disallow verifying the same update multiple times."* [3](#0-2) 

By contrast, every other Pyth price-update path enforces monotonic ordering. The standard Pyth EVM contract rejects updates whose `publishTime` is not newer than the cached value: [4](#0-3) 

The Sui Lazer governance module enforces strictly-increasing sequence numbers: [5](#0-4) 

The Solana remote-executor enforces `NonIncreasingSequence` on VAA replay: [6](#0-5) 

`PythLazer.sol` is the only production verification contract that provides none of these guarantees.

### Impact Explanation
An unprivileged attacker who has observed any valid signed Lazer update (freely available from the public WebSocket stream) can:

1. Capture update `U_old` at time `T_old` when the price of asset X is `P_old`.
2. Wait until the live price moves to `P_new` (favorable for the attacker's position in a consumer DeFi protocol).
3. Call `verifyUpdate(U_old)` — it passes because the signer has not expired.
4. The consumer contract receives `(payload_old, signer)` and, if it does not independently validate `payload.timestamp > lastAcceptedTimestamp`, stores `P_old` as the current price.
5. The attacker exploits the stale price (e.g., opens/closes a leveraged position, borrows against inflated collateral, or liquidates a healthy position).

This is the direct analog of the Clique double-spend: in Clique, all forks have equal difficulty so syncing nodes cannot determine the canonical chain; here, all signed updates pass `verifyUpdate()` regardless of age, so consumer contracts cannot determine the canonical (latest) price without out-of-band logic.

The official EVM integration guide shows example consumer code that does guard with `if (feedId == 2 && _timestamp > timestamp)`, but this guard is in sample code only — it is not enforced by the contract. [7](#0-6) 

Any consumer that omits or mis-implements this check is silently vulnerable.

### Likelihood Explanation
- The Lazer WebSocket stream is publicly accessible; capturing valid signed updates requires no privilege.
- Signer keys have long expiry windows (the test fixture uses `3000000000000000` as `expiresAt`), so replayed updates remain valid for extended periods.
- The integration guide's sample code is the primary reference for new integrators; omitting the timestamp guard is a natural mistake.
- No on-chain mechanism alerts integrators that they received a replayed update. [8](#0-7) 

### Recommendation
Add replay protection and timestamp freshness enforcement directly in `verifyUpdate()`:

1. **Replay protection**: Maintain a `mapping(bytes32 => bool) consumedUpdates` keyed on `keccak256(update)` (or on the payload hash already computed at line 93). Revert if the hash has been seen before.
2. **Timestamp freshness**: Extract the `uint64 timestamp` from the payload header inside `verifyUpdate()` and revert if `timestamp < block.timestamp - MAX_STALENESS` (e.g., 60 seconds for real-time channels).
3. **Return timestamp**: Include the parsed timestamp in the return value so consumer contracts can enforce monotonicity without re-parsing.

These mitigations mirror the protections already present in the standard Pyth accumulator path and in the Sui/Solana Lazer governance modules.

interface IPythLazer {
    function verifyUpdate(bytes calldata update)
        external payable returns (bytes calldata payload, address signer);
    function verification_fee() external view returns (uint256);
}

contract ReplayAttack {
    IPythLazer public pythLazer;
    bytes public capturedUpdate; // captured from the public Lazer stream at T_old

    constructor(address _pythLazer, bytes memory _capturedUpdate) {
        pythLazer = IPythLazer(_pythLazer);
        capturedUpdate = _capturedUpdate;
    }

    // Called at T_now >> T_old; verifyUpdate succeeds because:
    // 1. Signature is still valid (signer not expired).
    // 2. No consumed-update registry exists.
    // 3. No timestamp check is performed.
    function replayStalePrice() external payable {
        uint256 fee = pythLazer.verification_fee();
        (bytes calldata payload, address signer) =
            pythLazer.verifyUpdate{value: fee}(capturedUpdate);
        // payload contains P_old; consumer contract accepts it as current price.
        // Attacker exploits the stale price in the consumer protocol.
    }
}

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }

            revert("invalid signature");
