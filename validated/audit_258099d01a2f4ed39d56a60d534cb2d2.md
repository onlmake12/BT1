### Title
`CapacityVerifier::valid_dao_withdraw_transaction` Identifies DAO Transactions by Partial Type-Script Check Only, Allowing Capacity Inflation via Mixed-Input Transactions — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`CapacityVerifier::valid_dao_withdraw_transaction()` classifies any transaction that has **at least one input whose type script matches the DAO code hash** as a "DAO withdraw transaction" and unconditionally skips the global `OutputsSumOverflow` capacity check for the **entire** transaction. It does not verify that the matching input is actually a phase-2 withdrawing cell (non-zero 8-byte data). Because the DAO type script only enforces capacity rules for the cells it governs, non-DAO cells bundled in the same transaction are left completely unchecked, allowing an attacker to mint CKB capacity out of thin air.

---

### Finding Description

`CapacityVerifier::verify()` contains the following guard:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    // ... OutputsSumOverflow check ...
}
```

The helper that gates this bypass:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
```

`cell_uses_dao_type_script` checks only two fields:

```rust
fn cell_uses_dao_type_script(cell_output: &CellOutput, dao_type_hash: &Byte32) -> bool {
    cell_output.type_().to_opt()
        .map(|t| {
            Into::<u8>::into(t.hash_type()) == Into::<u8>::into(ScriptHashType::Type)
                && &t.code_hash() == dao_type_hash
        })
        .unwrap_or(false)
}
```

It checks `hash_type == Type` and `code_hash == dao_type_hash`. It does **not** inspect the cell's data to distinguish:

- **Deposit cell** (data = 8 zero bytes) — phase 1 input; capacity must be conserved.
- **Withdrawing cell** (data = non-zero 8-byte deposit block number) — phase 2 input; capacity legitimately increases by accrued interest.

The capacity bypass was designed for phase-2 withdrawals, where the DAO script is trusted to enforce the interest arithmetic. But the DAO type script only runs for cells that carry it; it has no visibility into non-DAO cells in the same transaction. When a phase-1 deposit cell is present as an input, the DAO script enforces capacity conservation only for that DAO cell pair. The global `inputs_sum < outputs_sum` check — the only guard on non-DAO cells — is skipped.

---

### Impact Explanation

An attacker who controls a DAO deposit cell (trivially obtained by depositing any amount of CKB with an attacker-controlled lock script) can construct a transaction:

| Slot | Cell | Capacity | Data |
|------|------|----------|------|
| Input 0 | DAO deposit cell (attacker's lock) | 100 CKB | `[0; 8]` |
| Input 1 | Normal cell (attacker's lock) | 10 CKB | — |
| Output 0 | DAO withdrawing cell | 100 CKB | deposit block number |
| Output 1 | Normal cell | **20 CKB** | — |

- `valid_dao_withdraw_transaction()` returns `true` (input 0 has DAO type script) → `OutputsSumOverflow` check skipped.
- DAO type script runs for input 0 / output 0 only; verifies output 0 capacity = input 0 capacity (100 CKB). ✓
- No check covers input 1 (10 CKB) vs output 1 (20 CKB).
- **10 CKB is created from nothing.**

Repeating this attack inflates the total CKB supply, violating the hard cap, undermining the economic security model, and enabling the attacker to acquire unbounded native capacity.

---

### Likelihood Explanation

The entry path is fully unprivileged: any transaction sender can submit such a transaction. The attacker only needs to:
1. Deposit a small amount of CKB into the DAO (standard operation).
2. Craft a phase-1 withdrawal transaction that bundles extra non-DAO inputs/outputs with inflated output capacity.
3. Submit via the standard `send_transaction` RPC.

No special keys, majority hashpower, or social engineering are required. The attack is deterministic and repeatable.

---

### Recommendation

`valid_dao_withdraw_transaction()` must additionally verify that the matching DAO input is a **phase-2 withdrawing cell** (cell data is exactly 8 bytes with a non-zero value), not a deposit cell:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| {
            if !cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash) {
                return false;
            }
            // Only phase-2 withdrawing cells (non-zero 8-byte data) legitimately
            // produce more output capacity than input capacity.
            matches!(
                self.data_loader.load_cell_data(cell_meta),
                Some(data) if data.len() == 8 && data.iter().any(|b| *b != 0)
            )
        })
}
```

Alternatively, perform the capacity overflow check on non-DAO cells unconditionally and only exempt the DAO cell pairs from the check.

### Proof of Concept

**Root cause location:**

`CapacityVerifier::valid_dao_withdraw_transaction` — [1](#0-0) 

`cell_uses_dao_type_script` partial check (type script hash only, no data check) — [2](#0-1) 

The bypass gate that skips `OutputsSumOverflow` for the entire transaction — [3](#0-2) 

**Step-by-step exploit:**

1. **Attacker deposits CKB into the DAO** using any lock script they control. This produces a live DAO deposit cell: `type = DAO`, `data = [0u8; 8]`, `capacity = D`. [4](#0-3) 

2. **Attacker also holds a normal cell** with capacity `N` (e.g., 10 CKB).

3. **Attacker constructs a phase-1 withdrawal transaction:**
   - `inputs[0]` = DAO deposit cell (capacity `D`, data all-zeros)
   - `inputs[1]` = normal cell (capacity `N`)
   - `outputs[0]` = DAO withdrawing cell (capacity `D`, data = deposit block number)
   - `outputs[1]` = normal cell (capacity `N + X` for any `X > 0`)

4. **`valid_dao_withdraw_transaction()` returns `true`** because `inputs[0]` has DAO type script. The `OutputsSumOverflow` check (`inputs_sum < outputs_sum`) is skipped for the whole transaction. [5](#0-4) 

5. **DAO type script executes** for `inputs[0]` / `outputs[0]` only. It verifies `outputs[0].capacity == inputs[0].capacity` and `outputs[0].data == deposit_block_number`. It has no visibility into `inputs[1]` / `outputs[1]`. [6](#0-5) 

6. **`inputs[1]` (N CKB) vs `outputs[1]` (N+X CKB) is never checked.** The transaction is accepted. `X` CKB has been minted from nothing.
