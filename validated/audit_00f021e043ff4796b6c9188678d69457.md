### Title
Intermediate Denominator Overflow in `deriveCrossRate` Causes Unexpected Revert for Valid Inputs - (File: `target_chains/ethereum/sdk/solidity/PythUtils.sol`)

### Summary

`PythUtils.deriveCrossRate` computes a cross-rate as `price1 / (10^|deltaExponent| * price2)` when `deltaExponent <= 0`. The denominator is formed by a plain checked multiplication `10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)` outside any `unchecked` block. When `|deltaExponent|` is near its allowed maximum of 58 and `price2` is a large-but-valid `uint64`, this intermediate product overflows `uint256`, causing an unexpected panic revert even though the final result would be a small, representable number.

### Finding Description

In `deriveCrossRate`, after the `deltaExponent` bounds check (`-58 ≤ deltaExponent ≤ 58`), the `else` branch executes:

```solidity
result = Math.mulDiv(
    uint64(price1),
    1,
    10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // ← overflow here
);
``` [1](#0-0) 

The bounds check only validates `deltaExponent` in `[-58, 58]`, ensuring `10 ** 58` alone fits in `uint256` (~10^58 ≈ 2^193). However, it does **not** validate that `10^|deltaExponent| * price2` fits in `uint256`. [2](#0-1) 

**Overflow condition:**

- `2^256 ≈ 1.1579 × 10^77`
- `10^58 * price2_max = 10^58 * (2^64 - 1) ≈ 10^58 * 1.844 × 10^19 = 1.844 × 10^77 > 2^256`

The overflow threshold for `price2` when `deltaExponent = -58` is:

```
price2 > floor(2^256 / 10^58) ≈ 11,579,208,923,731,619,542
```

Since `uint64` max is `18,446,744,073,709,551,615`, there is a large range of valid `uint64` price values for `price2` that trigger the overflow. The multiplication is in checked arithmetic (not `unchecked`), so Solidity panics with `UNDER_OVERFLOW` (0x11), reverting the call. [3](#0-2) 

The `Math.mulDiv` function itself handles 512-bit intermediate products safely, but the overflow occurs **before** `mulDiv` is even called — in the argument expression passed as the denominator. [4](#0-3) 

### Impact Explanation

Any downstream EVM contract that calls `deriveCrossRate` with a `price2` value above ~1.16 × 10^19 and a `deltaExponent` at or near -58 will receive an unexpected revert. The final mathematical result (`price1 / (10^58 * price2)`) would be a tiny but representable value (likely 0 after truncation), so the revert is spurious. Contracts relying on this function for on-chain price conversion, cross-rate arbitrage, or collateral valuation will be silently broken for certain price regimes.

### Likelihood Explanation

A Pyth price updater or relayer (an unprivileged role that can submit signed price attestations) can craft or select a price update where `price2` exceeds the overflow threshold. Pyth prices are `int64` with exponents; a price of, e.g., `12,000,000,000,000,000,000` (1.2 × 10^19, a valid `int64` ≈ `int64` max / ~0.77) with `expo2 = 0` and `targetExponent = 0` and `expo1 = -58` yields `deltaExponent = -58` and triggers the overflow. A front-running attacker who can submit a price update just before a victim's transaction can cause the victim's `deriveCrossRate` call to revert.

### Recommendation

Replace the plain multiplication in the denominator with a checked or 512-bit-safe computation. The simplest fix is to pass `price2` as a separate argument to `mulDiv` using its three-argument form, which handles 512-bit intermediate products:

```solidity
// Instead of:
10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)

// Use mulDiv with price2 as a separate denominator factor:
result = Math.mulDiv(
    uint64(price1),
    1,
    Math.mulDiv(10 ** uint64(Math.abs(deltaExponent)), uint64(price2), 1)
);
// Or restructure to avoid the intermediate product entirely.
```

Alternatively, widen the intermediate computation to use a 512-bit multiply before dividing, or add an explicit overflow check on the denominator product before passing it to `mulDiv`.

### Proof of Concept

```solidity
// deltaExponent = expo1 - (expo2 + targetExponent) = 0 - (0 + (-58)) = -58
// price2 = 12_000_000_000_000_000_000 (1.2e19, valid uint64, < int64.max)
// 10^58 * 1.2e19 = 1.2e77 > 2^256 ≈ 1.157e77 → overflow → revert

PythUtils.deriveCrossRate(
    1,          // price1
    0,          // expo1
    12_000_000_000_000_000_000,  // price2 (valid int64 ≈ 1.2e19)
    0,          // expo2
    -58         // targetExponent → deltaExponent = 0-(0+(-58)) = -58
);
// Expected: returns 0 (price1 / huge_denominator truncates to 0)
// Actual:   reverts with arithmetic overflow panic

    function mulDiv(
        uint256 x,
        uint256 y,
        uint256 denominator
    ) internal pure returns (uint256 result) {
        unchecked {
            (uint256 high, uint256 low) = mul512(x, y);

            // Handle non-overflow cases, 256 by 256 division.
            if (high == 0) {
                // Solidity will revert if denominator == 0, unlike the div opcode on its own.
                // The surrounding unchecked block does not change this fact.
                // See https://docs.soliditylang.org/en/latest/control-structures.html#checked-or-unchecked-arithmetic.
                return low / denominator;
            }
