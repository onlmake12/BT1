### Title
Fee Underpayment via `BatchAttestation` Branch Hardcodes Update Count to 1 — (File: `target_chains/fuel/contracts/pyth-contract/src/main.sw`)

### Summary
In the Fuel Pyth contract's `update_price_feeds` function, the `BatchAttestation` branch unconditionally adds `1` to `total_number_of_updates` regardless of how many price feeds are actually contained in the batch. The `Accumulator` branch correctly uses the actual returned count. This asymmetry allows a caller to update an arbitrary number of price feeds while paying only the fee for a single update.

### Finding Description
The `update_price_feeds` function in the Fuel contract iterates over submitted update data, dispatches to one of two branches based on update type, and then checks the fee after all updates have been applied: [1](#0-0) 

```
match UpdateType::determine_type(data) {
    UpdateType::Accumulator(accumulator_update) => {
        let (number_of_updates, _updated_ids) = accumulator_update.update_price_feeds(...);
        total_number_of_updates += number_of_updates;   // ✓ actual count
    },
    UpdateType::BatchAttestation(batch_attestation_update) => {
        let _updated_ids = batch_attestation_update.update_price_feeds(...);
        total_number_of_updates += 1;                   // ✗ always 1
    },
}
let required_fee = total_fee(total_number_of_updates, storage.single_update_fee);
require(msg_amount() >= required_fee, PythError::InsufficientFee);
```

The `Accumulator` branch captures the actual number of price-feed messages parsed from the blob and accumulates them correctly. The `BatchAttestation` branch discards the returned updated-IDs vector and always increments the counter by exactly `1`, no matter how many attestations the batch contains.

This is structurally identical to the borgCore `blacklist` bug: one branch of a mode-based check omits the same validation (here, counting actual updates) that the other branch performs correctly.

### Impact Explanation
An attacker submits a `BatchAttestation` blob containing `N` price-feed attestations and pays only `total_fee(1, single_update_fee)` — the fee for a single update. All `N` price feeds are written to storage, but the protocol collects only `1/N` of the intended fee revenue. For large `N`, this represents near-total fee bypass. Because the fee check occurs after state writes, the transaction succeeds as long as the attacker pays the single-update fee.

### Likelihood Explanation
The `BatchAttestation` update type is still accepted by the contract's `UpdateType::determine_type` dispatch. Any caller can craft or replay a multi-attestation batch. No privileged role is required; the entry point is the public `update_price_feeds` function callable by any EOA or contract.

### Recommendation
Replace the hardcoded `+= 1` with the actual count returned by `batch_attestation_update.update_price_feeds(...)`, mirroring the `Accumulator` branch:

```sway
UpdateType::BatchAttestation(batch_attestation_update) => {
    let updated_ids = batch_attestation_update.update_price_feeds(...);
    total_number_of_updates += updated_ids.len();   // use actual count
},
```

### Proof of Concept
1. Construct a valid `BatchAttestation` VAA containing `N = 50` price-feed attestations (all signed by a valid guardian set).
2. Call `update_price_feeds` on the Fuel Pyth contract, passing `msg_amount = total_fee(1, single_update_fee)`.
3. Observe that all 50 price feeds are updated on-chain and the transaction succeeds.
4. Compare with the expected fee `total_fee(50, single_update_fee)` — the protocol received only 2 % of the owed fee. [2](#0-1)
