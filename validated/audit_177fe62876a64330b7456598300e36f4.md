### Title
Truncated Price ID Prefix Storage Allows Wrong Price Feed Data Delivery to Consumer Contracts - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
`Echo.sol` stores only the first 8 bytes of each requested price ID as a `bytes8` prefix. The `executeCallback` verification only checks these 8 bytes against caller-supplied price IDs. If two valid Pyth price IDs share the same first 8 bytes, any unprivileged caller (after the exclusivity period) can fulfill a callback with data for the wrong price feed, causing consumer contracts to receive incorrect price data.

### Finding Description
In `requestPriceUpdatesWithCallback`, only the first 8 bytes of each price ID are stored:

```solidity
// Store only first 8 bytes of each price ID to save gas
bytes8[] priceIdPrefixes;
``` [1](#0-0) 

```solidity
bytes8 prefix;
assembly { prefix := priceId }
req.priceIdPrefixes[i] = prefix;
``` [2](#0-1) 

In `executeCallback`, the verification only checks these 8 bytes against the caller-supplied price IDs:

```solidity
bytes8 prefix;
assembly { prefix := priceId }
if (prefix != req.priceIdPrefixes[i]) {
    revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
}
``` [3](#0-2) 

After the check passes, the **full caller-supplied price IDs** (not the originally requested ones) are forwarded directly to `parsePriceFeedUpdates`:

```solidity
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(
    updateData,
    priceIds,   // attacker-controlled
    ...
);
``` [4](#0-3) 

The returned price feeds are then delivered to the consumer's callback:

```solidity
IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(sequenceNumber, priceFeeds)
``` [5](#0-4) 

The root cause is directly analogous to the reported vulnerability: a key is truncated (32 bytes → 8 bytes), causing two distinct entities (two different price IDs) to be treated as equivalent. In the original report, truncating the `sessionKeyId` caused multiple session keys to share the same `SessionKeyData` storage slot. Here, truncating the price ID causes different price feeds to be treated as the same feed during callback fulfillment.

After the exclusivity period, `executeCallback` is callable by **any address** with any `providerToCredit`:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(providerToCredit == req.provider, "Only assigned provider during exclusivity period");
}
``` [6](#0-5) 

There is no post-exclusivity check on who calls `executeCallback` or what price IDs they supply beyond the 8-byte prefix match.

### Impact Explanation
Consumer contracts (DeFi protocols, derivatives, lending platforms) receive price data for the wrong price feed. A consumer that requested BTC/USD prices could receive ETH/USD prices if the two price IDs share the same first 8 bytes. This can cause incorrect liquidations, mispriced collateral, or incorrect settlement values, leading to direct financial loss for users of those consumer contracts.

### Likelihood Explanation
Requires two valid Pyth price IDs to share the same first 8 bytes. Pyth price IDs are keccak256-derived 32-byte values. With ~500 current feeds, the birthday-bound collision probability over a 64-bit prefix space is approximately 7×10⁻¹⁵ — negligible today. However, this is a permanent design flaw: as Pyth expands to thousands of feeds, the probability grows quadratically. Additionally, the attacker must possess valid Wormhole-attested update data for the colliding price ID, which is publicly available for any listed Pyth feed. The entry path (calling `executeCallback` after the exclusivity period) requires no privilege.

### Recommendation
Store and verify the full 32-byte price IDs instead of truncated 8-byte prefixes. If gas savings are required, store a single `keccak256` commitment over the entire `priceIds` array rather than per-element truncated prefixes:

```solidity
bytes32 priceIdsCommitment; // keccak256(abi.encodePacked(priceIds))
```

Verify in `executeCallback`:
```solidity
require(
    keccak256(abi.encodePacked(priceIds)) == req.priceIdsCommitment,
    "Price IDs mismatch"
);
```

This costs one `keccak256` per call regardless of the number of price IDs, and provides full 256-bit collision resistance.

### Proof of Concept
1. User calls `requestPriceUpdatesWithCallback` for price ID `P1 = 0xAABBCCDD11223344<24 more bytes>` (stored prefix: `0xAABBCCDD11223344`).
2. Attacker identifies a valid Pyth price ID `P2 = 0xAABBCCDD11223344<different 24 bytes>` with the same first 8 bytes.
3. Attacker obtains valid Wormhole-attested update data for `P2` from the Pyth price service.
4. After the exclusivity period, attacker calls `executeCallback(attacker, seqNum, updateDataForP2, [P2])`.
5. The prefix check passes: `bytes8(P2) == bytes8(P1) == 0xAABBCCDD11223344`.
6. `parsePriceFeedUpdates` succeeds and returns price data for `P2`.
7. The consumer's `_echoCallback` receives price data for `P2` instead of `P1`, silently corrupting the consumer's state.
