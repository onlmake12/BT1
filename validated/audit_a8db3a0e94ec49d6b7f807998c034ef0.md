Audit Report

## Title
`RecentReject::put` never writes back `total_keys_num`, allowing unbounded RocksDB growth — (File: tx-pool/src/component/recent_reject.rs)

## Summary
`RecentReject::put` computes a local `total_keys_num = self.total_keys_num.checked_add(1)` but never assigns it back to `self.total_keys_num`. The in-memory counter is permanently stuck at its startup-estimated value (0 for a fresh node), so the `count_limit` guard never triggers, and the `recent_reject` RocksDB column families grow without bound as rejected transactions accumulate.

## Finding Description
In `RecentReject::put` (lines 62–69 of `tx-pool/src/component/recent_reject.rs`):

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // BUG: self.total_keys_num is never updated here
} else {
    self.shrink()?;
}
```

`self.total_keys_num` is initialized at startup via `estimate_num_keys_cf`. For a fresh node this returns 0. Every subsequent call to `put()` evaluates `0 + 1 = 1`, which is never `> count_limit` (default: `keep_rejected_tx_hashes_count`, typically in the hundreds or thousands), so `shrink()` is never called and `self.total_keys_num` stays at 0 forever. The only place `self.total_keys_num` is ever written is inside `shrink()` at line 111, which is unreachable under this condition. Existing guards are therefore entirely bypassed: the `count_limit` field exists and is checked, but the counter it compares against is never incremented.

## Impact Explanation
An unprivileged remote peer can submit an unlimited number of unique transactions rejected for any recordable reason (`LowFeeRate`, `Full`, `Verification`, etc.). Each rejection calls `put_recent_reject` → `RecentReject::put`, which writes a new key to RocksDB but never enforces `count_limit`. The `recent_reject` database grows without bound, consuming unbounded disk space and I/O, eventually exhausting node storage and causing a crash or severe degradation. This matches the allowed CKB bounty impact: **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation
The attack requires only the ability to submit transactions over P2P — no keys, no hashpower, no privileged access. Generating unique transactions that fail fee-rate checks is trivial (vary any field to produce a unique hash, set fee below minimum). The bug is deterministic and reproducible on any fresh node with default configuration. It is continuously exploitable with no rate limit imposed by the buggy code path.

## Recommendation
Add `self.total_keys_num = total_keys_num;` in the non-shrink branch of `put()`:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num; // FIX: persist the increment
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

## Proof of Concept
Using the existing test harness in `tx-pool/src/component/tests/recent_reject.rs`:

1. Build a `RecentReject` with `shard_num=2`, `count_limit=10`, `ttl=-1` on a fresh (empty) directory.
2. Insert 10,000 unique rejected transactions (unique `Byte32` keys).
3. After all inserts, assert `recent_reject.total_keys_num == 0` — it will be, confirming the counter was never incremented.
4. Query `db.estimate_num_keys_cf` across all shards — it will report ~10,000 keys, confirming the DB grew without bound while `count_limit=10` was never enforced.

The existing test at line 39 (`assert!(recent_reject.total_keys_num < 100)`) already passes for the wrong reason: `total_keys_num` is stuck at 0, not because `shrink()` correctly enforced the limit.
