### Title
`DaoScriptSizeVerifier` Lock-Script-Size Check Bypassed via Input/Output Position Mismatch — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`DaoScriptSizeVerifier::verify()` enforces that a Nervos DAO deposit cell (input) and its corresponding prepare cell (output) carry lock scripts of identical byte size. The check is the **sole** enforcement layer for this invariant — the on-chain DAO type script does not enforce it (the verifier's own doc comment calls it "a temporary solution till Nervos DAO script can be properly upgraded"). The verifier pairs inputs with outputs strictly by array position using `.zip()`. An unprivileged transaction sender can place the DAO deposit cell at input index `i` and the DAO prepare cell at output index `j ≠ i`, causing the zip to pair each DAO cell with a non-DAO counterpart, silently skipping the size check for both. This lets the sender shrink the lock script on the prepare cell, inflating the "counted capacity" that earns DAO interest and allowing withdrawal of more CKB than was deposited.

---

### Finding Description

**Root cause — positional zip in `DaoScriptSizeVerifier::verify()`**

```rust
// verification/src/transaction_verifier.rs  lines 847-888
for (i, (input_meta, cell_output)) in self
    .resolved_transaction
    .resolved_inputs
    .iter()
    .zip(self.resolved_transaction.transaction.outputs())   // ← positional pairing
    .enumerate()
{
    if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
        && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
    {
        continue;   // ← skipped when either side is not a DAO cell
    }
    ...
    if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
        return Err(...DaoLockSizeMismatch...);
    }
}
```

The guard requires **both** the input at position `i` **and** the output at position `i` to carry the DAO type script. If the attacker places:

| slot | input | output |
|------|-------|--------|
| 0 | DAO deposit cell (lock size 20) | non-DAO change cell |
| 1 | non-DAO fee cell | DAO prepare cell (lock size 10) |

then:
- Pair (0): input is DAO, output is **not** DAO → `continue`
- Pair (1): input is **not** DAO, output is DAO → `continue`

The size check is never reached. The prepare cell with a 10-byte lock script is accepted.

**Why the bypass matters**

The `DaoScriptSizeVerifier` is explicitly described as the only guard:

> "It provides a temporary solution till Nervos DAO script can be properly upgraded."

The DAO interest calculation in `DaoCalculator::calculate_maximum_withdraw()` computes:

```rust
// util/dao/src/lib.rs  lines 149-156
let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
let output_capacity: Capacity = output.capacity().into();
let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
```

`output` here is the **prepare cell** (the phase-2 input). A smaller lock script → smaller `occupied_capacity` → larger `counted_capacity` → larger `withdraw_capacity`. The extra interest is drawn from the shared DAO pool, harming all other depositors.

**Activation gate**

The verifier is invoked only when `rfc0044_active` is true:

```rust
// verification/contextual/src/contextual_block_verifier.rs  lines 444-451
.and_then(|result| {
    if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {
        DaoScriptSizeVerifier::new(...).verify()?;
    }
    Ok(result)
})
```

Once RFC-0044 is active on mainnet the bypass is reachable by any transaction sender.

---

### Impact Explanation

An unprivileged DAO depositor can craft a phase-1 (prepare) transaction that passes all verification, including script execution, capacity conservation (`CapacityVerifier` / `FeeCalculator`), and the `DaoScriptSizeVerifier`, while silently reducing the lock script size on the prepare cell. During phase-2 withdrawal the inflated `counted_capacity` yields a withdrawal amount larger than the depositor's fair share of DAO interest. The excess is drawn from the shared DAO interest pool, constituting direct theft of other depositors' accrued interest. The magnitude scales with deposit size and the lock-script-size reduction.

---

### Likelihood Explanation

The attack requires only a valid DAO deposit and the ability to submit a crafted transaction — no privileged keys, no majority hash power, no social engineering. The transaction structure (reordering inputs/outputs) is trivially constructable with any CKB SDK. The bypass is deterministic and reproducible. Likelihood is **high** once RFC-0044 is active.

---

### Recommendation

Replace the positional `.zip()` with a content-aware pairing: iterate over all inputs, find every DAO deposit cell, then search the outputs for the corresponding DAO prepare cell (matched by type-script hash and, if needed, by the cell's out-point referenced in the witness). Apply the lock-script-size check to each matched pair regardless of array index. Alternatively, enforce the invariant inside the DAO type script itself so the off-chain verifier becomes redundant.

---

### Proof of Concept

Craft a phase-1 DAO transaction with the following layout:

```
inputs:
  [0]  DAO deposit cell  — lock script 20 bytes, data = 0x0000000000000000
  [1]  non-DAO fee cell  — any lock script

outputs:
  [0]  non-DAO change cell
  [1]  DAO prepare cell  — lock script 10 bytes (smaller than deposit)
```

`DaoScriptSizeVerifier::verify()` iterates:
- pair (0): input[0] is DAO, output[0] is **not** DAO → `continue` (no check)
- pair (1): input[1] is **not** DAO, output[1] is DAO → `continue` (no check)

The verifier returns `Ok(())`. The block verifier accepts the transaction. The prepare cell now carries a 10-byte lock script. In phase 2, `DaoCalculator::calculate_maximum_withdraw()` computes a larger `counted_capacity` for the prepare cell, yielding a withdrawal amount that exceeds the depositor's legitimate entitlement, draining excess interest from the shared DAO pool.     pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
