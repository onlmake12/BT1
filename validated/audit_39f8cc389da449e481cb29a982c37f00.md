### Title
Surviving ancestors of a removed subtree retain permanently inflated `descendants_count`/`descendants_fee`/`descendants_cycles`, corrupting `EvictKey` ordering â€” (`tx-pool/src/component/pool_map.rs`)

### Summary

`remove_entry_and_descendants` pre-clears all link entries for the entire subtree before calling `remove_entry` on each node. Because `remove_entry` â†’ `update_ancestors_index_key` â†’ `calc_ancestors` relies on those link entries to find surviving ancestors, the pre-clearing causes `calc_ancestors` to return an empty set for every node in the subtree. Any pool entry that is an ancestor of the removed root but is **not** itself in the removed subtree never receives `sub_descendant_weight`, leaving its `inner.descendants_count`, `inner.descendants_fee`, `inner.descendants_cycles`, and `evict_key` permanently stale.

### Finding Description

The bug is in `remove_entry_and_descendants`:

```rust
// Step 1: pre-clear ALL links for the entire subtree
for id in &removed_ids {
    self.remove_entry_links(id);   // removes id from links map entirely
}

// Step 2: call remove_entry for each â€” but links are already gone
removed_ids.iter()
    .filter_map(|id| self.remove_entry(id))
    .collect()
```

Inside `remove_entry`, `update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`. Since `remove_entry_links` already called `self.links.remove(id)` for every node in the subtree, `calc_ancestors` traverses an empty/missing entry and returns `HashSet::new()`. The `for anc_id in &ancestors` loop body never executes, so no surviving ancestor's `evict_key` is updated. [5](#0-4) [6](#0-5) 

### Impact Explanation

Any pool entry that is a **parent** of the removed root (but not itself removed) retains inflated descendant weight fields. The `EvictKey` for those entries is computed from these fields: [7](#0-6) 

`next_evict_entry` uses `iter_by_evict_key()` to select which entry to evict when the pool is full: [8](#0-7) 

A surviving ancestor with inflated `descendants_fee` appears to have a higher-fee descendant package than it actually does, distorting its position in the eviction order. An attacker can exploit this to cause high-fee legitimate transactions to be evicted ahead of low-fee ones, directly harming miner revenue.

### Likelihood Explanation

The attack is fully reachable via normal transaction submission (P2P or RPC). The attacker:
1. Submits `grandparent_tx` (the surviving ancestor)
2. Submits `tx_root` spending `grandparent_tx`'s output, then N children, then M grandchildren
3. Submits a conflicting transaction spending the same input as `tx_root`, triggering `resolve_conflict` â†’ `remove_entry_and_descendants(tx_root)`
4. `grandparent_tx` now has stale, inflated descendant weights

No privileged access, leaked keys, or majority hashpower is required.

### Recommendation

Do not pre-clear links for the entire subtree before calling `remove_entry`. Instead, update surviving ancestors' weights **before** removing any links. One correct approach: for the root node only, call `update_ancestors_index_key` (with the aggregate descendant weight of the entire subtree being removed) before the pre-clearing loop, so surviving ancestors receive the correct `sub_descendant_weight` adjustment. Alternatively, remove the pre-clearing loop entirely and let `remove_entry`'s existing `remove_entry_links` call handle cleanup â€” the `update_descendants_index_key` call inside `remove_entry` will be a no-op for already-removed entries and is harmless.

### Proof of Concept

```
// Pool state: grandparent â†’ tx_root â†’ [child_1 .. child_100]
// Submit conflicting tx spending tx_root's input â†’ triggers remove_entry_and_descendants(tx_root)
// After removal:
//   grandparent.inner.descendants_count == 101  (should be 0)
//   grandparent.inner.descendants_fee   == sum_of_all_children_fees (should be 0)
//   grandparent.evict_key reflects a falsely high descendant fee-rate
// Pool is then filled; next_evict_entry picks grandparent ahead of actual low-fee entries
```

        relation: Relation,
