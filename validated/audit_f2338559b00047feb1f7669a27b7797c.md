Audit Report

## Title
Unbounded Relay Amplification via Duplicate RPC Transaction Submission — (`tx-pool/src/process.rs`)

## Summary
The `after_process` function in `tx-pool/src/process.rs` unconditionally enqueues a relay broadcast event when a local RPC submission returns `Reject::Duplicated`, with no rate-limiting or per-hash cooldown. The `send_bulk_of_tx_hashes` relay path for `original_peer = None` pushes the hash to every connected peer without a pre-push known-tx check, so `mark_as_known_tx` (called after the push) provides no protection against repeated submissions. An attacker with RPC access can submit the same pooled transaction N times and generate N × P relay broadcasts at negligible cost.

## Finding Description
**Root cause — `after_process`, lines 538–545 (`tx-pool/src/process.rs`):**

The `None` arm (local RPC, `remote = None`) matches `Err(Reject::Duplicated(_))` and unconditionally calls `send_result_to_relayer` with no guard, counter, or timestamp check: [1](#0-0) 

`send_result_to_relayer` is a non-blocking channel send that only logs on failure: [2](#0-1) 

**Relay amplification — `send_bulk_of_tx_hashes`, lines 662–669 (`sync/src/relayer/mod.rs`):**

In the `None` branch, the hash is pushed to `selected` for every connected peer before `mark_as_known_tx` is called. There is no pre-push `already_known_tx` check, so a previously-marked hash is re-broadcast on every subsequent duplicate submission: [3](#0-2) 

`mark_as_known_tx` inserts into `tx_filter` only after the push, providing no deduplication for the next event already queued in the channel: [4](#0-3) 

`take_relay_tx_verify_results` drains the channel with no back-pressure that would throttle the attacker: [5](#0-4) 

**Exploit flow:**
1. RPC `send_transaction` → `process_tx(tx, None)` → `_process_tx` → `check_txid_collision` returns `Err(Reject::Duplicated(...))`.
2. `after_process` is called with `remote = None`, hits the `Duplicated` branch, enqueues `TxVerificationResult::Ok { original_peer: None, tx_hash }` into `tx_relay_sender`.
3. On the next `TX_HASHES_TOKEN` tick, `send_bulk_of_tx_hashes` drains the channel (up to `MAX_RELAY_TXS_NUM_PER_BATCH` per tick) and sends `RelayTransactionHashes` to every full-relay peer.
4. Submitting the same tx N times enqueues N events; they are drained across ticks, each producing P relay messages (P = connected peers). The `tx_relay_receiver` is a `crossbeam_channel` receiver with no back-pressure.

## Impact Explanation
This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* Each duplicate RPC call costs only a cheap local round-trip — no PoW, no fee, no new transaction. Submitting the same tx 1000 times generates up to 1000 × P `RelayTransactionHashes` messages, congesting the P2P relay channel and degrading throughput for honest peers across the network.

## Likelihood Explanation
The RPC endpoint is the standard, documented interface for transaction submission. Any node operator who exposes the RPC (wallets, exchanges, public nodes) is vulnerable. The attacker needs no special privilege, no key material, and no new valid transaction — only one transaction already accepted into the pool. The exploit is trivially scriptable with a simple loop.

## Recommendation
1. In the `Err(Reject::Duplicated(_))` branch for `remote = None` in `after_process`, suppress the relay broadcast entirely, or gate it behind a per-`tx_hash` cooldown (e.g., only re-broadcast if the last broadcast for this hash was more than N seconds ago).
2. In `send_bulk_of_tx_hashes`, add a pre-push `already_known_tx` check in the `None` branch before adding the hash to `selected`, so that a hash already in the known-tx filter is not re-broadcast.

## Proof of Concept
```python
import requests

RPC = "http://127.0.0.1:8114"
tx = { ... }  # any valid, fee-paying transaction already accepted into the pool

# Step 1: confirm tx is in pool (first submission)
requests.post(RPC, json={"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[tx,"passthrough"]})

# Step 2: submit the same tx N times
for i in range(1000):
    r = requests.post(RPC, json={"id":i+2,"jsonrpc":"2.0","method":"send_transaction","params":[tx,"passthrough"]})
    # Each call returns PoolRejectedDuplicatedTransaction (-1107)
    # but internally enqueues TxVerificationResult::Ok into tx_relay_sender

# Expected: relay channel receives 1001 Ok events
# send_bulk_of_tx_hashes drains them across ticks → 1001 × P RelayTransactionHashes messages
# Invariant violated: relay broadcasts must only be triggered by newly accepted transactions
```
