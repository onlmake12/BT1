The vulnerability claim is **valid**. Here is the concrete finding:

---

### Title
Missing `#[account(mut)]` on `fee_collector` in `PostMessageUnreliable` bricks the instruction when `fee_lamports > 0` — (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_message/unreliable.rs`)

### Summary
`PostMessage` correctly marks `fee_collector` as `mut`, but `PostMessageUnreliable` omits the `mut` constraint. Both call the shared `handle_message_fee`, which issues a `system_program::transfer` CPI to `fee_collector` whenever `config.fee_lamports > 0`. Without `mut`, Anchor does not mark the account writable, so the Solana runtime rejects the CPI, permanently bricking `post_message_unreliable` under any non-zero fee configuration.

### Finding Description

**`PostMessage`** (reliable) — `fee_collector` is correctly declared with `mut`: [1](#0-0) 

```rust
#[account(
    mut,   // ← present
    seeds = [crate::constants::FEE_COLLECTOR_SEED_PREFIX],
    bump,
)]
fee_collector: Option<UncheckedAccount<'info>>,
```

**`PostMessageUnreliable`** — `fee_collector` is missing `mut`: [2](#0-1) 

```rust
#[account(
    // mut is absent
    seeds = [crate::constants::FEE_COLLECTOR_SEED_PREFIX],
    bump,
)]
fee_collector: Option<UncheckedAccount<'info>>,
```

Both paths converge on the same `handle_message_fee` function: [3](#0-2) 

When `config.fee_lamports > 0`, it calls `system_program::transfer` with `fee_collector` as the destination. The Solana runtime enforces that the destination of a `system_program::transfer` must be writable. Because Anchor did not set the writable flag (no `mut`), the runtime rejects the transaction. [4](#0-3) 

### Impact Explanation
Any call to `post_message_unreliable` fails with a writable-constraint error whenever `fee_lamports > 0`. The instruction path is entirely non-functional under that configuration. `post_message` (reliable) is unaffected because it has the correct `mut` annotation.

### Likelihood Explanation
Governance can set `fee_lamports` via `set_message_fee`. If it is ever set to a non-zero value (even 1 lamport), every caller of `post_message_unreliable` — including Pyth's own price-update emitters that use the unreliable path — will receive a hard transaction failure. No attacker action is required beyond the governance precondition; the bug is triggered by any ordinary call.

### Recommendation
Add `mut` to the `fee_collector` constraint in `PostMessageUnreliable`, matching the `PostMessage` definition:

```rust
#[account(
    mut,
    seeds = [crate::constants::FEE_COLLECTOR_SEED_PREFIX],
    bump,
)]
fee_collector: Option<UncheckedAccount<'info>>,
```

### Proof of Concept
1. Initialize the bridge with `fee_lamports = 1` via `set_message_fee`.
2. Call `post_message_unreliable` with a valid emitter and payload.
3. Observe the transaction fails with a writable-constraint / `AccountNotMutable` error on `fee_collector`.
4. Call `post_message` (reliable) under identical conditions — it succeeds.

The asymmetry between the two structs is the sole root cause and is directly verifiable in a local test environment without any privileged access beyond the governance precondition.
