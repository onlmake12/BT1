### Title
Missing `BLOCK_INVALID` Status Check Before Re-processing Headers Allows State Inconsistency - (`sync/src/synchronizer/headers_process.rs`)

### Summary

`HeaderAcceptor::accept()` in the sync subsystem explicitly acknowledges via a `FIXME` comment that it does not check for `BLOCK_INVALID` status before re-processing a header. An unprivileged peer can send a `SendHeaders` message containing a header already marked `BLOCK_INVALID`, causing the node to re-run all validation checks and potentially overwrite the `BLOCK_INVALID` status with `HEADER_VALID` via `insert_valid_header()`.

### Finding Description

In `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::accept()` begins by reading the block status:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer best_known_header and return early
    return result;
}
// Falls through to prev_block_check, non_contextual_check, version_check
// and ultimately to insert_valid_header() if all pass
```

The code guards against re-processing a header that is already `HEADER_VALID`, but there is **no guard for `BLOCK_INVALID`**. The `FIXME` comment is the developers' own acknowledgment of this gap.

`BLOCK_INVALID` (`1 << 12`) is a separate bit from `HEADER_VALID` (`1`). A header with status `BLOCK_INVALID` does not satisfy `status.contains(BlockStatus::HEADER_VALID)`, so execution falls through to the three sub-checks and, if all pass, to:

```rust
sync_shared.insert_valid_header(self.peer, self.header);
```

This call inserts the header into the `header_map` and updates the peer's `best_known_header`, effectively overwriting the `BLOCK_INVALID` status with `HEADER_VALID`.

The critical scenario: a block that failed **full contextual verification** (e.g., script execution failure in `chain/src/verify.rs`) is marked `BLOCK_INVALID`. Its header, however, may be perfectly well-formed and pass all three lighter checks in `HeaderAcceptor` (`prev_block_check`, `non_contextual_check`, `version_check`). When a peer re-sends that header, `accept()` re-validates it, all three checks pass, and `insert_valid_header()` is called â€” the node now treats the header of a known-bad block as `HEADER_VALID`.

The `BLOCK_INVALID` status is set in `chain/src/verify.rs` after full block verification failure:

```rust
if !is_internal_db_error(err) {
    self.shared
        .insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
}
```

And in `chain/src/chain_service.rs` after non-contextual block verification failure:

```rust
self.shared
    .insert_block_status(lonely_block.block().hash(), BlockStatus::BLOCK_INVALID);
```

By contrast, `compact_block_process.rs` **does** correctly check for `BLOCK_INVALID` before proceeding:

```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
```

The `SendHeaders` path is missing this guard.

### Impact Explanation

1. **State inconsistency / sync manipulation**: A peer can cause the node to re-classify a known-invalid block's header as `HEADER_VALID`. This updates `best_known_header` for that peer to point to an invalid block, corrupting the sync state machine's view of the peer's chain tip. The node may then request and attempt to process the full block again, wasting resources and potentially triggering repeated verification failures.

2. **CPU/resource exhaustion (DoS)**: Any unprivileged peer can repeatedly send `SendHeaders` messages containing headers already marked `BLOCK_INVALID`. Each message causes the node to re-run `HeaderVerifier::verify()` (non-trivial PoW and structural checks) for each header, with no rate-limiting protection from the cached `BLOCK_INVALID` status.

3. **Sync decision corruption**: If `best_known_header` is updated to point to an invalid block's header, the block fetcher (`block_fetcher.rs`) may select this peer as a sync target and request blocks from it, stalling or misdirecting IBD.

### Likelihood Explanation

The entry path is fully attacker-controlled: any peer connected to the node can send a `SendHeaders` P2P message. No special privileges are required. The `FIXME` comment confirms the developers are aware of the gap. The scenario where a block fails full verification but its header passes the lighter `HeaderAcceptor` checks is realistic (e.g., a block with a valid header but invalid scripts or DAO accounting).

### Recommendation

Add an early-return guard for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept()`, mirroring the pattern already used in `compact_block_process.rs`:

```rust
pub fn accept(&self) -> ValidationResult {
    let mut result = ValidationResult::default();
    let status = self.active_chain.get_block_status(&self.header.hash());

+   if status.contains(BlockStatus::BLOCK_INVALID) {
+       result.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
+       return result;
+   }

    if status.contains(BlockStatus::HEADER_VALID) {
        // ... existing early return
    }
    // ... rest of checks
}
```

### Proof of Concept

1. Node A receives and fully verifies a block B whose scripts fail. `BLOCK_INVALID` is set for B's hash in `block_status_map`.
2. Attacker peer P sends a `SendHeaders` message containing B's header.
3. `HeadersProcess::execute()` calls `accept_first(&headers[0])` â†’ `HeaderAcceptor::accept()`.
4. `get_block_status` returns `BLOCK_INVALID`. The `HEADER_VALID` check fails (different bit). No `BLOCK_INVALID` guard exists.
5. `prev_block_check` passes (B's parent is valid). `non_contextual_check` passes (B's header PoW and structure are valid). `version_check` passes.
6. `sync_shared.insert_valid_header(peer, &B_header)` is called â€” B's header is now in `header_map` as `HEADER_VALID` and P's `best_known_header` is updated to B.
7. The block fetcher may now request block B from P, triggering another failed verification cycle.

//! Provide BlockStatus
#![allow(missing_docs)]
#![allow(clippy::bad_bit_mask)]

use bitflags::bitflags;
bitflags! {
    #[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash)]
    pub struct BlockStatus: u32 {
        const UNKNOWN                 =     0;

        const HEADER_VALID            =     1;
        const BLOCK_RECEIVED          =     1 | (Self::HEADER_VALID.bits() << 1);
        const BLOCK_STORED            =     1 | (Self::BLOCK_RECEIVED.bits() << 1);
        const BLOCK_VALID             =     1 | (Self::BLOCK_STORED.bits() << 1);

        const BLOCK_INVALID           =     1 << 12;
    }
