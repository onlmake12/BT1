### Title
Inconsistent Block Header Version Validation Across Sync and Full Verification Paths — (`sync/src/synchronizer/headers_process.rs` vs `verification/src/header_verifier.rs`)

---

### Summary

CKB enforces a block header version check only in the P2P sync path (`HeaderAcceptor::version_check`), hardcoded against `0`. The full non-contextual block verification path (`BlockVerifier`, `HeaderVerifier`) contains **no block header version check at all**, despite a dedicated `BlockVersionError` type existing in the verification crate. This is the direct analog to "different pragma directives": version constraints are applied inconsistently across code paths, creating a gap where a block with a non-zero header version can bypass the version gate depending on how it enters the node.

---

### Finding Description

**Sync path** — `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::version_check`:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    if self.header.version() != 0 {
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
```

This check is hardcoded to reject any header whose `version != 0`. It is called inside `HeaderAcceptor::accept()` only when a header arrives via the P2P sync protocol. [1](#0-0) [2](#0-1) 

**Full verification path** — `verification/src/header_verifier.rs`, `HeaderVerifier::verify()`:

```rust
fn verify(&self, header: &Self::Target) -> Result<(), Error> {
    PowVerifier::new(header, self.consensus.pow_engine().as_ref()).verify()?;
    // ... number, epoch, timestamp checks ...
    Ok(())  // ← no version check
}
```

No block header version check is present. The verifier checks PoW, block number, epoch, and timestamp — but not version. [3](#0-2) 

**Non-contextual block verifier** — `verification/src/block_verifier.rs`, `BlockVerifier::verify()`:

Also contains no block header version check. [4](#0-3) 

**Contrast with transaction version** — `verification/src/transaction_verifier.rs`, `VersionVerifier::verify()`:

Transaction version IS checked in the full verification path, against `consensus.tx_version()`:

```rust
if self.transaction.version() != self.tx_version {
    return Err(TransactionError::MismatchedVersion { ... }.into());
}
``` [5](#0-4) 

The `NonContextualTransactionVerifier` calls `VersionVerifier` as part of block transaction verification: [6](#0-5) 

**`BlockVersionError` is defined but unreachable in the verification path:** [7](#0-6) 

---

### Impact Explanation

The inconsistency means:

- A block with `header.version != 0` submitted via the RPC `submit_block` endpoint (or any path that invokes `BlockVerifier`/`HeaderVerifier` directly, bypassing `HeaderAcceptor`) will **pass** all non-contextual and contextual verification checks.
- The same block, if later relayed to a peer via the P2P sync protocol, will be **rejected** by that peer's `HeaderAcceptor::version_check`.

This creates a consensus split vector: a node that accepted the block via RPC considers it valid; peers receiving it via sync reject it. The chain tip diverges silently.

Additionally, the hardcoded `!= 0` in the sync path is not derived from `consensus` (unlike `consensus.tx_version()` for transactions), meaning any future protocol upgrade that legitimately changes the block version would require coordinated changes in two separate, unlinked locations.

---

### Likelihood Explanation

An RPC caller (`submit_block`) or miner assembly caller can craft a block with `version = 1` (or any non-zero value). The block passes `BlockVerifier` and `HeaderVerifier` since neither checks version. The block is inserted into the local chain. When relayed to peers via sync, those peers' `HeaderAcceptor::version_check` rejects it. This is reachable by any unprivileged RPC caller or miner.

---

### Recommendation

1. Add a block header version check to `HeaderVerifier::verify()` (or `BlockVerifier::verify()`) using the consensus-derived expected version, consistent with how `VersionVerifier` handles transaction version:

```rust
// In HeaderVerifier::verify()
if header.version() != self.consensus.block_version() {
    return Err(BlockVersionError {
        expected: self.consensus.block_version(),
        actual: header.version(),
    }.into());
}
```

2. Remove the hardcoded `!= 0` in `HeaderAcceptor::version_check` and derive the expected version from consensus, or remove the redundant sync-path check once the verification path enforces it.

3. Ensure `BlockVersionError` (already defined in `verification/src/error.rs`) is actually raised in the canonical verification path.

---

### Proof of Concept

1. Craft a `BlockView` with `header.version = 1`.
2. Submit it via `submit_block` RPC. The block passes `BlockVerifier::verify()` and `HeaderVerifier::verify()` — no version check fires.
3. The block is accepted into the local chain.
4. A peer receives this block via the sync protocol. `HeaderAcceptor::accept()` calls `version_check`, which evaluates `1 != 0 → true`, marks the block `BLOCK_INVALID`, and rejects it.
5. The two nodes now have divergent chain tips for the same block hash.
