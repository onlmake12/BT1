### Title
`ScriptHashTypeVerifier` Skips Type Script Hash-Type Validation, Allowing Disallowed `ScriptHashType` Values to Bypass Non-Contextual Checks — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` only validates the `hash_type` field of each output's **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but never validates the `hash_type` of each output's **type** script. A transaction submitter can craft a transaction whose output carries a type script with a disallowed `ScriptHashType` (e.g., `Data3` = 6) and have it accepted by the non-contextual verifier and admitted to the tx-pool, bypassing the intended early-rejection gate.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is the consensus-level allowlist for script hash types:

```
{0 = Data, 1 = Type, 2 = Data1, 4 = Data2}
``` [1](#0-0) 

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and checks only `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { ... }
        } else { ... }
    }
    Ok(())
}
``` [2](#0-1) 

The type script (`output.type_()`) is never inspected here. The comment on `NonContextualTransactionVerifier` itself confirms the omission — it says "Check whether output **lock** hash type within enabled range": [3](#0-2) 

The low-level molecule `check_data()` does visit both lock and type scripts, but it only validates structural encoding (even value or 1) via `ScriptHashType::verify_value()`, not consensus-level enablement: [4](#0-3) 

`ScriptHashType::verify_value()` accepts any even byte or 1, so `Data3` (6), `Data4` (8), … all pass it: [5](#0-4) 

The two identifier spaces — lock-script hash type and type-script hash type — are structurally identical bytes but serve different roles. The verifier treats them as if only the lock-script namespace needs the consensus allowlist check, leaving the type-script namespace unchecked. This is the direct analog to the WormholeFacet bug: one identifier space (lock) is validated while the other (type) is silently ignored.

---

### Impact Explanation

1. **Tx-pool pollution / resource amplification.** A transaction with a type script carrying `Data3` (value 6) passes `NonContextualTransactionVerifier` and is admitted to the tx-pool. It is only rejected later during contextual script execution (`extract_script_and_dep_index` returns `ScriptError::InvalidScriptHashType`): [6](#0-5) 

This forces the node to perform expensive script-group resolution and VM setup before the transaction is discarded, wasting CPU and memory proportional to the number of such transactions submitted.

2. **Miner block-assembly waste.** A miner's block-assembly pipeline calls the same non-contextual verifier to pre-filter candidates. Invalid type-script-hash-type transactions pass that filter and are included in a candidate block, only to fail during full block verification, causing the miner to discard and reassemble the block.

3. **Inconsistent enforcement.** The `CellbaseVerifier` in `block_verifier.rs` also only checks lock scripts against `ENABLED_SCRIPT_HASH_TYPE` for cellbase outputs: [7](#0-6) 

The pattern of omitting type-script hash-type validation is consistent across both verifiers, confirming this is a systematic gap rather than an isolated oversight.

---

### Likelihood Explanation

Any unprivileged tx-pool submitter (RPC caller via `send_transaction`) can trigger this. Crafting a transaction with a type script using `Data3` requires no special privilege — only a valid fee and a syntactically correct molecule-encoded transaction. The minimum fee rate is the only barrier, making sustained tx-pool pollution economically bounded but practically reachable.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` for each output that carries one:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }).into());
            }
        }
    }
    Ok(())
}
```

Apply the same fix to `CellbaseVerifier` in `verification/src/block_verifier.rs`.

### Proof of Concept

1. Build a transaction with one output whose **lock** script uses `ScriptHashType::Type` (value 1, allowed) and whose **type** script uses `ScriptHashType::Data3` (value 6, disallowed).
2. Submit via `send_transaction` RPC.
3. Observe that `NonContextualTransactionVerifier::verify()` returns `Ok(())` — the transaction is admitted to the tx-pool.
4. Observe that subsequent contextual verification (script execution) returns `ScriptError::InvalidScriptHashType("The ScriptHashType/Data3 has not been activated…")` and the transaction is discarded.
5. Repeat at the minimum fee rate to sustain tx-pool pollution.

The existing unit test `test_not_enabled_hash_type_output_lock` confirms the lock-script path is tested, but no analogous test exists for the type-script path, confirming the gap: [8](#0-7)

    pub(crate) size: SizeVerifier<'a>,
    pub(crate) empty: EmptyVerifier<'a>,
    pub(crate) duplicate_deps: DuplicateDepsVerifier<'a>,

    /// Creates a new NonContextualTransactionVerifier
    pub fn new(tx: &'a TransactionView, consensus: &'a Consensus) -> Self {
        NonContextualTransactionVerifier {
            version: VersionVerifier::new(tx, consensus.tx_version()),
            size: SizeVerifier::new(tx, consensus.max_block_bytes()),
            empty: EmptyVerifier::new(tx),
            duplicate_deps: DuplicateDepsVerifier::new(tx),

            script_hash_type: ScriptHashTypeVerifier::new(tx),

        self.script_hash_type.verify()?;
        Ok(())
    }
