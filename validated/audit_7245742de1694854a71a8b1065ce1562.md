### Title
Unbounded `last_n_numbers` Collection via `difficulty_boundary=0` Bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

The guard in `GetLastStateProofProcess::execute` checks only the *message-supplied* `last_n_blocks` field against `GET_LAST_STATE_PROOF_LIMIT`, but the actual work vector `last_n_numbers` is sized by `last_block_number - difficulty_boundary_block_number`. When an attacker sends `difficulty_boundary=U256::zero()` and `difficulties=[]`, the binary-search short-circuits to `difficulty_boundary_block_number = start_block_number = 0`, and `last_n_numbers` collects every block from 0 to the chain tip — O(N) entries — before `complete_headers` iterates all of them.

### Finding Description

**Guard (lines 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

With `difficulties=[]` and `last_n_blocks=10` this evaluates to `0 + 20 = 20`, well under 1000. The guard passes. [1](#0-0) 

**Binary-search short-circuit (lines 29–36):**

`get_first_block_total_difficulty_is_not_less_than(0, N, &U256::zero())` checks `start_total_difficulty >= U256::zero()` on the very first call. Every block's total difficulty satisfies this, so it returns `Some((0, genesis_difficulty))` immediately, setting `difficulty_boundary_block_number = 0`. [2](#0-1) 

**Adjustment skipped (lines 313–316):**

```rust
if last_block_number - difficulty_boundary_block_number < last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
```

With `difficulty_boundary_block_number=0` and a 50,000-block chain: `50000 - 0 = 50000 < 10` is **false**, so the correction is never applied. [3](#0-2) 

**Unbounded collection (lines 318–319):**

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
```

This collects `(0..50000)` — 50,000 entries — into `last_n_numbers`. [4](#0-3) 

**O(N) work in `complete_headers` (lines 132–163):**

For each of the 50,000 entries, `complete_headers` calls `snapshot.get_ancestor(...)`, `snapshot.get_block(...)`, and `snapshot.chain_root_mmr(*number - 1).get_root()` — three storage/MMR operations per block. [5](#0-4) 

The constant is confirmed at 1000: [6](#0-5) 

### Impact Explanation

A single unauthenticated P2P peer can force the light-client protocol server to perform O(chain_length) MMR root computations and block lookups per `GetLastStateProof` message. On a mainnet node with hundreds of thousands of blocks, repeated requests cause sustained CPU and disk I/O exhaustion, effectively DoS-ing the light-client server without any PoW or privileged access.

### Likelihood Explanation

The exploit requires only a valid `last_hash` pointing to a real main-chain tip (observable by any peer) and a crafted message with `difficulty_boundary=0`, `difficulties=[]`, and a small `last_n_blocks`. No keys, no hashpower, no special role. Any peer connected to the light-client protocol port can trigger it.

### Recommendation

The guard must bound the *computed* output size, not the *requested* `last_n_blocks`. After computing `difficulty_boundary_block_number`, enforce:

```rust
if last_block_number - difficulty_boundary_block_number > constant::GET_LAST_STATE_PROOF_LIMIT as u64 {
    return StatusCode::MalformedProtocolMessage.with_context("too many last_n blocks");
}
```

Alternatively, cap `difficulty_boundary_block_number` from below so that `last_n_numbers` never exceeds `GET_LAST_STATE_PROOF_LIMIT` entries regardless of the supplied `difficulty_boundary`.

### Proof of Concept

1. Spin up a node with a MockChain of 50,000 blocks.
2. Connect as an unprivileged light-client peer.
3. Send:
   ```
   GetLastStateProof {
       last_hash:          <tip hash>,
       start_number:       0,
       start_hash:         <genesis hash>,
       last_n_blocks:      10,
       difficulty_boundary: U256::zero(),
       difficulties:       [],
   }
   ```
4. Observe that `complete_headers` is invoked with a 50,000-element slice and that wall-clock time grows linearly with chain length, while the guard at line 201 reports only `20` samples checked against the 1000 limit.
