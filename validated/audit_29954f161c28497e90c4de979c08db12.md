Audit Report

## Title
Cycle Accounting Undercount in `FullSuspendedState::size()` for `inherited_fd` — (File: script/src/types.rs)

## Summary

`FullSuspendedState::size()` at line 546 computes the byte cost of `inherited_fd` as `self.inherited_fd.len() * size_of::<Fd>()`, which counts only the number of outer `(VmId, Vec<Fd>)` tuples and omits both the inner `Fd` elements and the `VmId` per entry. This causes a systematic undercount of cycle charges during suspend/resume, allowing an attacker to obtain significantly more VM computation than they are billed for.

## Finding Description

The field is declared as:

```rust
pub inherited_fd: Vec<(VmId, Vec<Fd>)>,   // line 511
```

The `size()` implementation charges:

```rust
+ (self.inherited_fd.len() * (size_of::<Fd>())) as u64   // line 546
```

`self.inherited_fd.len()` returns the count of outer tuples, not the total number of `Fd` values across all inner `Vec<Fd>`s. Each inner `Vec<Fd>` can hold up to `MAX_FDS` entries (confirmed as a VM-enforced bound in `script/src/scheduler.rs`), and the `VmId` component of each tuple is not counted at all. With N spawned VMs each holding M inherited FDs, the formula charges `N × size_of::<Fd>()` bytes, while the actual data is `N × (size_of::<VmId>() + M × size_of::<Fd>())` bytes. The method's own doc comment at line 520–521 states it "should be used to derive cycles charged for suspending / resuming," confirming this is the authoritative cycle-metering path. `FullSuspendedState` is actively used in both `script/src/scheduler.rs` and `script/src/verify.rs`. ## Impact Explanation

This is a confirmed incorrect implementation of CKB-VM cycle metering, matching the allowed impact: **"Incorrect implementation or behavior of CKB-VM or system scripts" (High, 10001–15000 points)**. The cycle budget is the primary resource-exhaustion guard in CKB-VM. Undercounting the cost of `inherited_fd` allows an attacker to perform substantially more suspend/resume work than they are charged for, violating the invariant that all CKB-VM computation must be fully cycle-metered. This can be used to perform work that would otherwise exhaust the cycle budget, enabling network congestion at reduced cost.

## Likelihood Explanation

The exploit path is fully unprivileged: submit a `ScriptVersion::V2` transaction with a crafted lock script that spawns multiple child VMs with large inherited FD sets. No special keys, hashpower, or operator access are required. The maximum undercount is bounded by `MAX_VMS_COUNT × MAX_FDS` (VM-enforced constants), making the undercount consistently achievable at its maximum. The bug is in production code and is reachable through normal transaction submission.

## Recommendation

Replace the flat `len()` multiplication with a fold that sums the actual `Fd` count across all inner `Vec<Fd>`s and accounts for the `VmId` per entry:

```rust
+ self.inherited_fd.iter().fold(0usize, |acc, (_, fds)| {
    acc + size_of::<VmId>() + fds.len() * size_of::<Fd>()
}) as u64
``` [3](#0-2) 

## Proof of Concept

1. Construct a `ScriptVersion::V2` lock script that creates `MAX_FDS` pipes via `ckb_pipe` and spawns `MAX_VMS_COUNT` child VMs, passing all pipe FDs as inherited FDs to each via `ckb_spawn`.
2. Force a scheduler suspend by approaching the cycle limit within the script execution.
3. Resume execution and invoke `FullSuspendedState::size()`.
4. A unit test can directly construct a `FullSuspendedState` with 16 entries each containing 64 `Fd` values and assert that `size()` returns a value where the `inherited_fd` contribution equals `16 × size_of::<Fd>() = 128` bytes rather than the correct `16 × (size_of::<VmId>() + 64 × size_of::<Fd>())` bytes, confirming the undercount.
