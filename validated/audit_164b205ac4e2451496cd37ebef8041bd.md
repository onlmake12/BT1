Audit Report

## Title
Unbounded Cell Iteration in `get_cells_capacity()` Enables Local RPC Thread Exhaustion — (File: `util/indexer/src/service.rs`)

## Summary
`get_cells_capacity()` accepts no `limit` parameter and iterates every RocksDB cell matching a caller-supplied prefix, bounded only by a wall-clock `TimeoutIterator`. Unlike `get_cells()` and `get_transactions()`, which enforce a mandatory count-based `limit` capped at `self.request_limit`, `get_cells_capacity()` occupies an RPC handler thread for the full timeout window on any broad prefix query. Concurrent calls from an unprivileged caller can saturate the RPC thread pool, making the node's RPC interface unresponsive until each timeout expires.

## Finding Description
`get_cells()` (L193) and `get_transactions()` (L381) both accept a `limit: Uint32` parameter, validate it is non-zero and does not exceed `self.request_limit` at L212–221 and L388–397 respectively, and terminate iteration early with `.take(limit)` at L371.

`get_cells_capacity()` at L686–689 has no `limit` parameter in its signature, matching the RPC trait definition at `rpc/src/module/indexer.rs` L879–883. It constructs a `TimeoutIterator` at L720 and calls `.take_while(|(key, _value)| key.starts_with(&prefix))` at L728, iterating every matching cell with no count-based bound. For each cell it performs a secondary RocksDB lookup via `snapshot.get(Key::OutPoint(&out_point).into_vec())` at L749.

The `TimeoutIterator` at L52–60 checks elapsed time on each `next()` call; when the timeout fires it sets `timed_out = true` and returns `None`, after which the function returns an error at L838–839. The thread is blocked for the entire `timeout_limit` duration before being freed. There is no count-based bound, no pagination cursor, and no rejection of overly broad prefix queries.

## Impact Explanation
An unprivileged caller can send concurrent `get_cells_capacity` requests with a broad prefix, each occupying an RPC handler thread for the full `timeout_limit` duration. With enough concurrent requests the RPC thread pool is saturated and all other RPC calls time out or are queued indefinitely. This is a local RPC denial-of-service matching **Note (0–500 points): Any local RPC API crash** in the allowed impact scope. The node's consensus and P2P layers are unaffected; only the RPC interface becomes unresponsive.

## Likelihood Explanation
The `get_cells_capacity` RPC method requires no authentication. On mainnet, millions of cells already exist under common lock scripts; no attacker-created transactions are needed. The attacker only needs to call the endpoint repeatedly with a broad prefix. The attack is low-cost, repeatable, and requires no special privilege or key material.

## Recommendation
Apply the same limit-enforcement pattern used by `get_cells()` and `get_transactions()`:
1. Add a `limit: Uint32` parameter to `get_cells_capacity()` in both `rpc/src/module/indexer.rs` (L880–883) and `util/indexer/src/service.rs` (L686–689).
2. Reject requests where `limit == 0` or `limit > self.request_limit`, mirroring the checks at L212–221.
3. Apply `.take(limit)` on the iterator before `.sum()` and return a partial result with a cursor to allow pagination.

Alternatively, enforce a hard maximum on the number of cells scanned equal to `self.request_limit` and return an error if the result is truncated.

## Proof of Concept
1. On a synced CKB node with the indexer enabled, identify any script prefix that matches a large number of live cells (e.g., a common lock script with empty args on mainnet).
2. Send N concurrent JSON-RPC calls to `get_cells_capacity` with `search_key.script` set to that prefix and `script_search_mode: "prefix"`.
3. Observe that each call blocks its RPC handler thread for the full `timeout_limit` window before returning `"Indexer request timeout"` (L838–839).
4. While those calls are in flight, send any other RPC call (e.g., `get_tip_block_number`); observe it times out or is queued.
5. For contrast, send `get_cells` with the same search key and `limit: 1`; it returns immediately after one cell is found via `.take(limit)` at L371, confirming the asymmetry.
