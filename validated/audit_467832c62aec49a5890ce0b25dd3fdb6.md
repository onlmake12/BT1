### Title
Unbounded O(N·log H) RocksDB Reads per `GetLastStateProof` Request Enables Low-Cost DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

Any unprivileged remote peer can craft a single `GetLastStateProof` P2P message that forces the server to perform up to ~1000 independent `chain_root_mmr().get_root()` calls, each traversing O(log H) RocksDB nodes (where H is chain height). There is no caching between calls, no rate limiting, and the per-request DB read cost grows linearly with chain height. On a tall chain this creates a severe amplification DoS vector.

---

### Finding Description

**Entrypoint — limit check:**

The `execute()` handler enforces a single guard: [1](#0-0) [2](#0-1) 

With `last_n_blocks=499` and `difficulties.len()=2`: `2 + 499×2 = 1000`, which exactly meets the limit and passes.

**How `block_numbers` reaches ~1000 entries:**

`block_numbers` is the union of `reorg_last_n_numbers` (up to 499), `sampled_numbers` (up to 2), and `last_n_numbers` (up to 499): [3](#0-2) 

**The uncached MMR root computation loop:**

For every entry in `block_numbers`, `complete_headers` constructs a brand-new `ChainRootMMR` and calls `get_root()`: [4](#0-3) 

`chain_root_mmr(n)` creates a fresh `MMR` object with no shared state or cache: [5](#0-4) 

**Each `get_root()` reads O(log H) nodes directly from RocksDB:**

`MMRStore<HeaderDigest>` for `&Snapshot` calls `get_header_digest` on every access: [6](#0-5) 

`get_header_digest` issues a raw RocksDB `get_pinned` call — there is no application-level cache for `COLUMN_CHAIN_ROOT_MMR`: [7](#0-6) 

**Additional cost from `reply_proof`:**

After `complete_headers`, `reply_proof` calls `mmr.get_root()` and `mmr.gen_proof(items_positions)` on the full tip MMR, adding further O(log H + N·log H) reads: [8](#0-7) 

---

### Impact Explanation

On a chain of height H ≈ 12,000,000 (mainnet), log₂(H) ≈ 24. A single crafted request causes:

- ~1000 × 24 ≈ **24,000 RocksDB point-reads** in `complete_headers`
- Plus proof generation reads in `reply_proof`

The attacker sends a ~200-byte P2P message; the server performs tens of thousands of storage reads. With multiple concurrent connections (no per-peer rate limit exists), this saturates I/O and CPU, causing **CKB network congestion** — full nodes serving light clients become unresponsive to legitimate peers.

---

### Likelihood Explanation

- **No authentication required**: any peer that completes the P2P handshake can send `GetLastStateProof`.
- **No rate limiting**: no per-peer or global throttle on this message type was found.
- **Trivially crafted**: the attacker only needs to set `last_n_blocks=499`, two valid difficulty values, and a valid `last_hash` on the main chain.
- **Scales with chain age**: the attack becomes more effective as the chain grows.

---

### Recommendation

1. **Cache MMR roots within a single request**: compute `get_root()` once per unique MMR size and reuse the result across all entries in `block_numbers`.
2. **Rate-limit `GetLastStateProof` per peer**: enforce a per-peer request budget (e.g., N requests per second).
3. **Cap `block_numbers` size independently of the limit formula**: the current formula allows the full 1000-entry budget to be consumed by `last_n_numbers` alone.
4. **Bound DB reads per request**: enforce a hard cap on total `get_elem` calls across the entire request lifecycle.

---

### Proof of Concept

```
1. Connect to a CKB full node running the light-client protocol on a tall chain (H > 1,000,000).
2. Send GetLastStateProof with:
     last_hash   = <current tip hash>
     start_hash  = <genesis hash>
     start_number = 0
     last_n_blocks = 499
     difficulty_boundary = <tip total_difficulty>
     difficulties = [D1, D2]   # two valid, monotonically increasing values
3. Instrument chain_root_mmr().get_root() call count server-side.
4. Observe: call count == block_numbers.len() (~1000), each doing O(log H) RocksDB reads.
5. Repeat from 100 concurrent peers; measure node I/O saturation and response latency degradation.
```
