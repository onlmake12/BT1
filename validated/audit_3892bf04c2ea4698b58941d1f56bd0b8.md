### Title
Unimplemented Per-Peer Rate Limiting on GetHeaders Allows Unbounded Resource Exhaustion - (`sync/src/synchronizer/get_headers_process.rs`)

### Summary

A remote peer can flood a non-IBD CKB node with valid `GetHeaders` messages at arbitrary frequency. Each message triggers expensive DB lookups (`locate_latest_common_block`), a response fetch of up to `MAX_HEADERS_LEN=2000` headers (`get_locator_response`), and an unbounded async task spawn — with no per-peer rate limit enforced at any layer.

### Finding Description

The `GetHeadersProcess::execute` function processes incoming `GetHeaders` messages: [1](#0-0) 

The only structural guard is a locator size check: [2](#0-1) 

For a non-IBD node, every valid message unconditionally executes:
1. `locate_latest_common_block` — a DB scan over the locator hashes
2. `get_locator_response` — fetches up to `MAX_HEADERS_LEN=2000` headers from the DB
3. `async_handle().spawn(...)` — spawns a new async task to send the response [3](#0-2) 

The call to `getheaders_received` at line 77 is a documented no-op: [4](#0-3) 

The `TODO` comment confirms this was intended to implement tracking/rate-limiting but was never completed. `MAX_LOCATOR_SIZE` and `MAX_HEADERS_LEN` are defined as: [5](#0-4) [6](#0-5) 

The `received` handler in `Synchronizer` processes each message via `tokio::task::block_in_place`, meaning each GetHeaders floods a tokio blocking thread: [7](#0-6) 

The `BAD_MESSAGE_BAN_TIME` ban is only applied for structurally malformed messages, not for valid but excessive ones: [8](#0-7) 

A search of the network layer confirms rate limiting only exists for the hole-punching protocol, not for general sync messages. There is no token bucket, message counter, or frequency cap for `GetHeaders` at any layer.

### Impact Explanation

Each valid `GetHeaders` message (up to 100 locator hashes) causes:
- **CPU**: DB lookups for `locate_latest_common_block` + header fetches for `get_locator_response`
- **Async runtime**: One unbounded `spawn` per message, accumulating tasks faster than they drain
- **Memory**: Each spawned task holds a serialized response of up to 2000 headers (~384 KB at ~192 bytes/header)

A single attacker peer sending messages in a tight loop can exhaust the tokio thread pool (via `block_in_place` saturation), fill the async task queue, and cause OOM or severe latency for all other peers.

### Likelihood Explanation

The attack requires only a standard P2P connection — no special privileges, no PoW, no keys. The locator hashes can be the genesis block hash repeated, which is always a valid common ancestor. The attack is trivially scriptable and locally testable.

### Recommendation

Implement `getheaders_received` to enforce a per-peer rate limit (e.g., token bucket or sliding window counter). Disconnect or temporarily ban peers that exceed a threshold (e.g., >N `GetHeaders` per second). The `GET_HEADERS_TIMEOUT` constant already defined at line 50 of `sync/src/types/mod.rs` suggests the intent existed but was never wired up. [9](#0-8) 

### Proof of Concept

1. Connect a peer to a non-IBD CKB node.
2. In a tight loop, send `GetHeaders` messages with a single valid locator hash (e.g., the genesis block hash) and `hash_stop = 0x00...00`.
3. Each message passes the `locator_size <= MAX_LOCATOR_SIZE` check, is not in IBD, finds a common ancestor, and spawns an async send task.
4. Monitor CPU usage, tokio thread pool saturation, and async task queue depth.
5. Assert: no per-peer rate limit is enforced; resource consumption grows linearly with message rate; no ban is issued for valid messages.

    pub fn execute(self) -> Status {
        let active_chain = self.synchronizer.shared.active_chain();

        let block_locator_hashes = self
            .message
            .block_locator_hashes()
            .iter()
            .map(|x| x.to_entity())
            .collect::<Vec<Byte32>>();
        let hash_stop = self.message.hash_stop().to_entity();
        let locator_size = block_locator_hashes.len();
        if locator_size > MAX_LOCATOR_SIZE {
            return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                "Locator count({locator_size}) > MAX_LOCATOR_SIZE({MAX_LOCATOR_SIZE})"
            ));
        }

        if active_chain.is_initial_block_download() {
            info!(
                "Ignoring getheaders from peer={} because the node is in initial block download stage.",
                self.peer
            );
            self.send_in_ibd();
            let shared = self.synchronizer.shared();
            if let Some(flag) = shared.state().peers().get_flag(self.peer)
                && (flag.is_outbound || flag.is_whitelist || flag.is_protect)
            {
                shared.insert_peer_unknown_header_list(self.peer, block_locator_hashes);
            };
            return Status::ignored();
        }

        if let Some(block_number) =
            active_chain.locate_latest_common_block(&hash_stop, &block_locator_hashes[..])
        {
            debug!(
                "headers latest_common={} tip={} begin",
                block_number,
                active_chain.tip_header().number(),
            );

            self.synchronizer.peers().getheaders_received(self.peer);
            let headers: Vec<core::HeaderView> =
                active_chain.get_locator_response(block_number, &hash_stop);
            // response headers

            debug!("headers len={}", headers.len());

            let content = packed::SendHeaders::new_builder()
                .headers(headers.into_iter().map(|x| x.data()).collect::<Vec<_>>())
                .build();
            let message = packed::SyncMessage::new_builder().set(content).build();
            let nc = Arc::clone(self.nc);
            self.synchronizer
                .shared()
                .shared()
                .async_handle()
                .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
        } else {
            return StatusCode::GetHeadersMissCommonAncestors
                .with_context(format!("{block_locator_hashes:#x?}"));
        }
        Status::ok()
    }
