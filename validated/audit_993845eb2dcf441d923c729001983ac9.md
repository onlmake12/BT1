### Title
Cache Bypass via `refresh=1` Allows Unbounded `System::new_all()` Invocations — (`rpc/src/module/terminal.rs`)

### Summary

`get_overview(refresh=1)` unconditionally bypasses the 5-second system-info cache on every call with no minimum interval or cooldown, causing `System::new_all()` + `sys.refresh_all()` to be invoked at an unbounded rate. No per-method or per-connection rate limiting exists on the RPC server. Any caller with RPC access can saturate CPU and memory by looping this call.

---

### Finding Description

In `TerminalRpcImpl::get_sys_info`, the cache guard is:

```rust
if !refresh.contains(RefreshKind::SYSTEM_INFO)
    && let Some(cached) = self.cache.get_sys_info()
{
    return Ok(cached);
}
``` [1](#0-0) 

When `refresh` has the `SYSTEM_INFO` bit set (value `1`), the first operand is `false`, the short-circuit prevents the cache lookup entirely, and execution falls through unconditionally to:

```rust
let mut sys = System::new_all();
sys.refresh_all();
``` [2](#0-1) 

Additionally, `SysDisks::new_with_refreshed_list()` and `SysNetworks::new_with_refreshed_list()` are also called on every forced refresh, enumerating all disks and network interfaces. [3](#0-2) 

The result is written back to cache at line 568, but the next call with `refresh=1` ignores that cached value entirely — the TTL is rendered meaningless. [4](#0-3) 

The RPC server has **no per-method or per-connection rate limiting**. The only guards are a 30-second request timeout and an optional batch-size cap (disabled by default): [5](#0-4) [6](#0-5) 

---

### Impact Explanation

`sysinfo::System::new_all()` on Linux reads `/proc` for every running process, allocating per-process structs. `refresh_all()` repeats this scan. On a host with hundreds of processes, each call allocates several MB and consumes significant CPU. A tight loop of `get_overview(1)` calls drives RSS growth and CPU saturation in the `ckb` process, degrading or crashing consensus-critical services (chain, tx-pool, sync).

---

### Likelihood Explanation

The default RPC bind address is `127.0.0.1:8114`. [7](#0-6) 

However, the `Terminal` module is included in the default production `modules` list: [8](#0-7) 

Any local process (including unprivileged ones) on the node host can reach `127.0.0.1:8114` without credentials. Operators who expose the RPC port (a common pattern for remote monitoring) extend the attack surface to the network. The exploit requires no authentication, no special knowledge, and is trivially scriptable.

---

### Recommendation

1. **Enforce a minimum refresh interval**: Even when `SYSTEM_INFO` is set, check whether the cached entry is younger than the TTL and return it if so. The `refresh` flag should mean "refresh if stale" not "always bypass cache."
2. **Add per-method rate limiting** on the RPC server for expensive Terminal endpoints, similar to the `governor`-based rate limiters used in the P2P relayer and hole-punching protocols. [9](#0-8) 

3. **Reuse a long-lived `System` instance** rather than allocating a new one per call.

---

### Proof of Concept

```bash
# Measure RSS before
ps -o rss= -p $(pgrep ckb)

# Tight loop forcing sysinfo refresh on every call
while true; do
  curl -s -X POST http://127.0.0.1:8114 \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","method":"get_overview","params":[1],"id":1}' \
    > /dev/null
done &

# Observe RSS growth and CPU saturation
watch -n1 'ps -o %cpu,rss= -p $(pgrep ckb)'
```

Assert: RSS grows unboundedly or CPU of the `ckb` process saturates, degrading block processing and sync.
