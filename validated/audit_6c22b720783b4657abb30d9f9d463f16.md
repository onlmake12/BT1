Audit Report

## Title
Uncharged O(N) `header_deps` Linear Scan in `load_header` Syscall Breaks Cycle-to-CPU-Work Invariant — (`script/src/syscalls/load_header.rs`)

## Summary
The `load_header` syscall performs a full linear scan of `header_deps` on every invocation for `Source::Transaction(Input)` and `Source::Transaction(CellDep)`, but this scan is never charged VM cycles. When the scan finds no match, the syscall returns `ITEM_MISSING` via an early return that skips `add_cycles_no_checking` entirely. A malicious script can call the syscall in a tight loop, causing O(N × loop_count) host CPU work while consuming only O(loop_count) VM cycles, breaking the invariant that the cycle limit bounds node CPU work.

## Finding Description
In `load_header.rs`, the private `load_header` method performs a membership check by iterating the entire `header_deps` vector on every call:

```rust
if self
    .header_deps()
    .into_iter()
    .any(|hash| &hash == block_hash)
```

This is an O(N) scan over all `header_deps` entries, executed entirely in host Rust code outside the VM. In the `ecall` handler, when `fetch_header` returns any error (including `ITEM_MISSING` produced when the scan finds no match), the handler sets the error register and returns immediately:

```rust
let header = self.fetch_header(source, index as usize);
if let Err(err) = header {
    machine.set_register(A0, Mac::REG::from_u8(err));
    return Ok(true);
}
```

The `add_cycles_no_checking` call that charges for transferred bytes is only reached on the success path:

```rust
machine.add_cycles_no_checking(transferred_byte_cycles(len))?;
```

The cost model charges only for bytes written to VM memory (`transferred_byte_cycles`), not for host-side scan work. There is no `MAX_HEADER_DEPS` constant or independent count limit anywhere in the codebase. The `SizeVerifier` enforces only the total serialized byte size of the transaction. With `MAX_BLOCK_BYTES = 597,000` bytes and each `header_dep` being a 32-byte hash, a single transaction can carry approximately 18,000 `header_deps`. An attacker constructs a transaction with ~18,000 `header_deps` and one input whose `transaction_info.block_hash` is deliberately absent from `header_deps`, then loops calling `load_header(Source::Transaction(Input), 0)` until cycles are exhausted. Each call executes a small fixed number of charged RISC-V instructions in the VM, then triggers an 18,000-element hash scan in host code that is entirely invisible to the cycle counter.

## Impact Explanation
This breaks the fundamental invariant that the block cycle limit bounds the CPU time required to verify a block. With a 3,500,000,000-cycle block budget and ~5 cycles per syscall invocation, a script can make ~700 million calls, each triggering an 18,000-element scan in host code. Nodes with slower hardware will take disproportionately longer to verify such a block than the cycle limit implies, causing them to fall behind the chain tip. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially **Critical — Vulnerabilities which could easily cause consensus deviation** if nodes diverge on block acceptance timing.

## Likelihood Explanation
The attack requires no privileged access. Any user can submit a transaction via the standard `send_transaction` RPC or P2P relay, write a trivial RISC-V lock script that loops calling `load_header`, fill `header_deps` with valid on-chain block hashes (publicly available from any block explorer), and ensure the input's `block_hash` is not among the `header_deps` to guarantee the `ITEM_MISSING` path is always taken. All preconditions are attacker-controlled and reachable through normal transaction submission. The attack is repeatable across blocks.

## Recommendation
Charge cycles proportional to the number of `header_deps` scanned on every `load_header` invocation, regardless of whether the lookup succeeds — add a cycle charge of `transferred_byte_cycles(header_deps.len() as u64 * 32)` before the early return on the `ITEM_MISSING` path. Alternatively, replace the per-call linear scan with a `HashSet<Byte32>` built once per script group execution (charged upfront at O(N) cost), making each subsequent lookup O(1) with no additional uncharged work.

## Proof of Concept
1. Mine or obtain ~18,000 distinct valid block hashes `H[0..17999]` from the chain.
2. Construct a transaction with:
   - `header_deps = [H[0], ..., H[17999]]` (~576,000 bytes, within the 597,000-byte limit).
   - One input cell whose `transaction_info.block_hash` is a hash **not** in `header_deps`.
   - A lock script that executes: `loop { ckb_load_header(buf, &mut len, 0, 0, CKB_SOURCE_INPUT); }` until cycles are exhausted.
3. Submit the transaction and measure wall-clock verification time on a reference node.
4. Compare against a baseline transaction with 0 `header_deps` and the same cycle consumption.
5. Assert that wall-clock time for the attack transaction far exceeds the baseline — the ratio should approach O(N) ≈ 18,000×, demonstrating that the cycle limit does not bound actual CPU time.
