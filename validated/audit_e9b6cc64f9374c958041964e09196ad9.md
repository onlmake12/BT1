### Title
Permanent Subscription Deposit Cap Bypassed via Non-Cumulative Balance Check - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `MAX_DEPOSIT_LIMIT` cap for permanent subscriptions in `Scheduler.sol` is enforced only against the single-transaction deposit amount (`msg.value`) rather than the cumulative subscription balance (`status.balanceInWei + msg.value`). Any unprivileged user can bypass this cap by calling `addFunds()` repeatedly with amounts at or below `MAX_DEPOSIT_LIMIT`, permanently locking an unbounded amount of ETH in the contract.

### Finding Description
In `Scheduler.sol`, permanent subscriptions are subject to a `MAX_DEPOSIT_LIMIT` cap. This cap is checked in two places:

**`createSubscription()`** (line 48):
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

**`addFunds()`** (line 613):
```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

In both cases, the check compares only the current transaction's `msg.value` against `MAX_DEPOSIT_LIMIT`, not the resulting cumulative balance (`status.balanceInWei + msg.value`). Because `addFunds()` can be called an unlimited number of times, a user can deposit `MAX_DEPOSIT_LIMIT` wei per call and accumulate a balance far exceeding the intended cap.

The root cause is structurally identical to the external report: a non-cumulative (per-call) value is used for a cap check instead of the running total, allowing the cap to be bypassed through repeated calls.

Compounding the severity, permanent subscriptions explicitly block all withdrawals:
```solidity
// Prevent withdrawals from permanent subscriptions
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
```

This means any ETH deposited beyond `MAX_DEPOSIT_LIMIT` is permanently locked in the contract, irrecoverable by the depositor.

### Impact Explanation
- The `MAX_DEPOSIT_LIMIT` cap for permanent subscriptions is completely ineffective; any user can bypass it with multiple `addFunds()` calls.
- Unlimited ETH can be permanently locked in the Scheduler contract (permanent subscriptions cannot withdraw).
- The protocol's intended deposit ceiling is violated, undermining the risk management or regulatory purpose of the cap.
- **Impact: Medium** — funds are the depositor's own, but the cap is a protocol invariant that is fully bypassed, and the ETH is irrecoverable.

### Likelihood Explanation
- **Likelihood: High** — `addFunds()` is a public, permissionless function callable by any subscription manager. No special role or key is required. The bypass requires only repeated standard transactions.

### Recommendation
Change both cap checks to compare against the resulting cumulative balance rather than the single-transaction amount:

In `createSubscription()`:
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
```
→
```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) { // initial creation is fine
```
(The creation check is acceptable since `balanceInWei` starts at zero, but `addFunds` must be fixed.)

In `addFunds()`:
```solidity
// Before:
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;

// After:
status.balanceInWei += msg.value;
if (params.isPermanent && status.balanceInWei > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

### Proof of Concept
1. Alice creates a permanent subscription with `msg.value = MAX_DEPOSIT_LIMIT`. Check passes: `MAX_DEPOSIT_LIMIT > MAX_DEPOSIT_LIMIT` is false.
2. Alice calls `addFunds(subscriptionId)` with `msg.value = MAX_DEPOSIT_LIMIT`. Check passes again: `MAX_DEPOSIT_LIMIT > MAX_DEPOSIT_LIMIT` is false.
3. Alice repeats step 2 N times.
4. `status.balanceInWei` is now `(N+1) * MAX_DEPOSIT_LIMIT`, far exceeding the intended cap.
5. Alice cannot withdraw any of these funds (`CannotUpdatePermanentSubscription` reverts all withdrawals).
6. The ETH is permanently locked in the contract, and the cap is fully bypassed. [1](#0-0) [2](#0-1) [3](#0-2)
