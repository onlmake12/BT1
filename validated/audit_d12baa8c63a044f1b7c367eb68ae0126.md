Audit Report

## Title
`InnerPool::need_clean` samples only one HashMap child, allowing permanent suppression of orphan-pool cleanup — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::need_clean` evaluates expiry by inspecting exactly one arbitrarily-chosen child block via `map.iter().next()`. Because `non_contextual_verify` performs no PoW or epoch validation, any P2P peer can insert a "shield" block with a recent epoch alongside an expired-epoch sibling under the same fake parent hash. Whenever the shield block is iterated first, `need_clean` returns `false` and neither block is ever evicted, causing unbounded memory growth in the orphan pool and eventual OOM crash.

## Finding Description

`InnerPool::need_clean` at [1](#0-0)  calls `map.iter().next()` on a `HashMap<Byte32, LonelyBlockHash>`, returning one non-deterministic entry. The function's own comment confirms this: "get 1st block belongs to that parent and check if it's expired block." If that one entry is not expired, the entire parent group is retained regardless of whether other children are expired.

`clean_expired_blocks` iterates leaders and only removes a group when `need_clean` returns `true`: [2](#0-1) 

The precondition — inserting a block with an arbitrary epoch — is trivially reachable. `asynchronous_process_block` calls `non_contextual_verify` before inserting into the orphan pool: [3](#0-2) 

`non_contextual_verify` invokes only `BlockVerifier` and `NonContextualBlockTxsVerifier`: [4](#0-3) 

`BlockVerifier::verify` runs only `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier` — no PoW, no epoch check: [5](#0-4) 

The `epoch_number` stored in `LonelyBlockHash` is taken directly from the unvalidated block header field: [6](#0-5) 

The orphan pool has no hard eviction cap; `ORPHAN_BLOCK_SIZE` is used only as initial `HashMap` capacity: [7](#0-6) 

The cleanup timer fires every 60 seconds: [8](#0-7) 

**Exploit flow:**
1. Attacker sends two `SendBlock` messages with the same fake `parent_hash`: Block A with `epoch=1` (expired at `tip_epoch=20`) and Block B with `epoch=19` (recent).
2. Both pass `non_contextual_verify` (no PoW or epoch check).
3. Both are written to DB and inserted into the orphan pool.
4. When `clean_expired_blocks(20)` fires, `need_clean` samples one child via `map.iter().next()`.
5. If Block B is returned first (~50% probability), `need_clean` returns `false` and neither block is evicted.
6. Attacker repeats with many distinct fake parent hashes; ~50% of groups survive each 60-second cleanup pass permanently.

## Impact Explanation

The orphan block pool grows without bound. An attacker sending a steady stream of (expired, shield) block pairs under distinct fake parent hashes causes monotonically increasing RSS. With no hard cap and a broken cleanup path, the node eventually exhausts available memory and crashes. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

No PoW is required; any P2P peer can send arbitrary `SendBlock` messages with spoofed epoch fields. No privileged role is needed. The attack is cheap: two small crafted blocks per fake parent hash. The non-determinism favors the attacker: on average 50% of inserted groups escape cleanup permanently per pass. The 60-second cleanup interval means a modest send rate suffices to outpace cleanup.

## Recommendation

Replace the single-sample check with a check over all children, using `any` to detect whether at least one child is expired:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            map.values().any(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
        })
        .unwrap_or_default()
}
```

Additionally, add PoW verification (via `PowVerifier` from `HeaderVerifier`) to `non_contextual_verify` so that epoch-spoofed blocks are rejected at the network boundary before reaching the orphan pool. A hard eviction cap on the block orphan pool (analogous to `DEFAULT_MAX_ORPHAN_TRANSACTIONS` in the tx-orphan pool) should also be added as defense-in-depth.

## Proof of Concept

```rust
// Minimal unit test — insert two children under the same fake parent,
// one expired, one recent; assert the expired one is cleaned.
let pool = OrphanBlockPool::with_capacity(10);
let fake_parent = random_byte32();

// expired: epoch 1, tip will be 20 (1 + 6 < 20)
let expired = make_lonely_block_hash(fake_parent.clone(), epoch = 1);
// shield: epoch 19 (19 + 6 >= 20, not expired)
let shield  = make_lonely_block_hash(fake_parent.clone(), epoch = 19);

pool.insert(expired);
pool.insert(shield);

let removed = pool.clean_expired_blocks(20);

// BUG: if shield is iterated first by HashMap, removed.len() == 0
// The expired block is never cleaned.
// Run repeatedly — fails ~50% of the time.
assert_eq!(removed.len(), 1);
```

To confirm the insertion precondition, send a crafted `SendBlock` P2P message containing a block with `epoch=1` and an arbitrary (non-PoW-valid) nonce to a live node and verify it is accepted by `non_contextual_verify` and inserted into the orphan pool (observable via `pool.len()` metric or debug logs).
