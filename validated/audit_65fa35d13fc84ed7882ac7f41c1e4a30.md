### Title
Unconstrained `spill` Account in `UpgradeContract` Allows Any Relayer to Redirect Buffer Lamports — (`target_chains/solana/programs/core-bridge/src/legacy/processor/governance/upgrade_contract.rs`)

### Summary
The `spill` account in the `UpgradeContract` Anchor context carries only `#[account(mut)]` with no address constraint. Any unprivileged actor who relays a valid governance upgrade VAA can supply an attacker-controlled pubkey as `spill`, causing the BPF Loader Upgradeable program to transfer all excess lamports from the upgrade buffer to the attacker.

### Finding Description
In `UpgradeContract`, the `spill` field is declared as:

```rust
/// CHECK: This account receives any lamports after the result of the upgrade.
#[account(mut)]
spill: UncheckedAccount<'info>,
``` [1](#0-0) 

The `constraints()` function validates the VAA guardian set, emitter, and the `buffer` pubkey against the VAA payload, but performs no check on `spill`:

```rust
require_keys_eq!(
    Pubkey::from(decree.implementation()),
    ctx.accounts.buffer.key(),
    CoreBridgeError::ImplementationMismatch
);
``` [2](#0-1) 

The handler then passes `spill` directly to the BPF loader CPI:

```rust
&bpf_loader_upgradeable::upgrade(
    &crate::ID,
    &ctx.accounts.buffer.key(),
    &ctx.accounts.upgrade_authority.key(),
    &ctx.accounts.spill.key(),   // ← attacker-controlled
),
``` [3](#0-2) 

The `claim_vaa` replay guard is keyed on `[emitter_address, emitter_chain, sequence]` and only prevents the same VAA from being used twice — it does not restrict which account is `spill` or who submits the transaction first. [4](#0-3) 

### Impact Explanation
The BPF Loader Upgradeable `Upgrade` instruction transfers all lamports from the buffer account that exceed the rent-exempt minimum of the new `program_data` account to the `spill` address. For a large program buffer (e.g., 500KB–2MB), this can be several SOL. The attacker receives these lamports instead of the intended recipient (e.g., the protocol treasury or the legitimate relayer), constituting a direct financial loss from protocol-controlled funds.

### Likelihood Explanation
Once a governance upgrade VAA is posted on-chain, it is a public account readable by anyone. The attacker only needs to:
1. Monitor for posted governance upgrade VAAs.
2. Compute the deterministic `claim` PDA.
3. Submit `upgrade_contract` with their own `spill` before the legitimate relayer.

No privileged access, leaked keys, or guardian compromise is required. Front-running on Solana is practical via priority fees or MEV infrastructure.

### Recommendation
Add an `address` constraint to `spill` pinning it to a protocol-controlled account (e.g., the fee collector PDA or a hard-coded treasury pubkey), or encode the intended spill recipient in the governance VAA payload and enforce it in `constraints()`:

```rust
#[account(
    mut,
    address = <PROTOCOL_TREASURY_PUBKEY>
)]
spill: UncheckedAccount<'info>,
```
