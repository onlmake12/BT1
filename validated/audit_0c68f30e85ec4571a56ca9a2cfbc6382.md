### Title
`forward_rate_limiter` Bypass via Unauthenticated `from` PeerId Enables Multi-Hop Message Amplification — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

The `forward_rate_limiter` in the HolePunching protocol is keyed on `(from, to, msg_item_id)`, but the `from` field is taken directly from the message payload with no validation that it matches the actual sender's session identity. An unprivileged attacker can send up to 30 `ConnectionSync` messages/second (the per-session cap) each with a distinct fabricated `from` PeerId, creating 30 distinct rate-limiter keys and bypassing the intended 1-request/second-per-triple forwarding limit. With `MAX_HOPS = 6`, this produces up to 180 forwarding operations/second across the relay chain from a single attacker connection.

---

### Finding Description

**Two-layer rate limiting design:**

The `HolePunching` struct has two rate limiters:

- `rate_limiter`: keyed `(PeerIndex, u32)` = `(session_id, msg_item_id)`, quota 30/s — gates messages at the receiving node per session.
- `forward_rate_limiter`: keyed `(PeerId, PeerId, u32)` = `(from, to, msg_item_id)`, quota 1/s — intended to prevent the same `(from, to)` pair from being relayed more than once per second. [1](#0-0) 

**`msg_item_id` is a fixed enum discriminant per message type** — `ConnectionSync` always has `item_id() == 2`: [2](#0-1) 

So the effective `forward_rate_limiter` key for `ConnectionSync` is `(from, to, 2)`.

**`from` is parsed from the message payload without authenticating it against the sender's session:**

```rust
let from = PeerId::from_bytes(value.from().raw_data().to_vec()).map_err(|_| {
    StatusCode::InvalidFromPeerId.with_context("the from peer id is invalid")
})?;
``` [3](#0-2) 

The only check is that the bytes form a valid `PeerId`. There is no check that `from` equals the actual peer sending the message.

**The `forward_rate_limiter` check uses this unauthenticated `from`:** [4](#0-3) 

**`MAX_HOPS = 6` is the relay chain depth:** [5](#0-4) 

---

### Impact Explanation

An attacker connected to relay node A sends 30 `ConnectionSync` messages/second, each with a distinct fabricated `from_i` PeerId but the same `to` PeerId (any peer known to be connected to A):

- **At Node A:** `rate_limiter` allows all 30 (30/s per session). `forward_rate_limiter` sees 30 distinct `(from_i, to, 2)` keys — all first-time entries — all pass. All 30 are forwarded.
- **At Node B:** Same logic applies. Node B's `rate_limiter` is keyed on Node A's session ID (not the attacker's), so it allows 30/s. `forward_rate_limiter` again sees 30 distinct keys. All 30 forwarded.
- **Continues for up to 6 hops.**

Total forwarding operations = 30 × 6 = **180 message-send operations/second** across the network from a single attacker P2P connection. This violates the invariant that a single connection must not amplify network-wide message processing beyond a constant factor.

---

### Likelihood Explanation

- Requires only a standard P2P connection — no privilege, no key, no hashpower.
- The attacker only needs to know one `PeerId` connected to the first relay node (discoverable via the discovery protocol).
- `from` PeerIds are trivially fabricated: any 32-byte value that passes `PeerId::from_bytes` is valid.
- The `forward_rate_limiter` `HashMapStateStore` grows with distinct keys but stabilizes at ~30 entries/second (governor's `retain_recent` expires entries after the quota window), so there is no secondary memory exhaustion.

---

### Recommendation

Authenticate the `from` field against the actual sender's session identity before using it as a rate-limiter key. Concretely:

1. In `received()`, pass the session's `PeerId` (looked up from `peer_registry` by `session_id`) into each `*Process::new(...)` constructor.
2. In `ConnectionSyncProcess::execute`, assert `content.from == actual_sender_peer_id` before the `forward_rate_limiter` check, or replace the `forward_rate_limiter` key with `(actual_sender_peer_id, to, msg_item_id)` so the key cannot be varied by the attacker.

Alternatively, key the `forward_rate_limiter` on `(session_id, to, msg_item_id)` — the session is unforgeable — which caps forwarding to 1/s per `(session, to)` pair regardless of the `from` field content.

### Proof of Concept

```
Attacker → Node_A: 30 × ConnectionSync/s
  each message: from=random_PeerId_i, to=known_peer_connected_to_A, route=[Node_B, Node_C, ...]

Node_A.rate_limiter[(attacker_session, 2)]:       30/s budget → all 30 pass
Node_A.forward_rate_limiter[(from_i, to, 2)]:     30 distinct keys, each 1st use → all 30 pass
  → Node_A forwards 30 msgs/s to Node_B

Node_B.rate_limiter[(node_a_session, 2)]:         30/s budget → all 30 pass
Node_B.forward_rate_limiter[(from_i, to, 2)]:     30 distinct keys → all 30 pass
  → Node_B forwards 30 msgs/s to Node_C

... (repeat for MAX_HOPS=6 total hops)

Total forwarding ops/s across network = 30 × 6 = 180
Amplification factor per attacker connection = MAX_HOPS = 6
``` [6](#0-5) [7](#0-6)
