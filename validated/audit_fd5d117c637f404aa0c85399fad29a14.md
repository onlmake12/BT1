### Title
Missing Zero-Price Validation in `deriveCrossRate` Causes Panic/Revert When Quote Price Feed Returns Zero - (File: `target_chains/ethereum/sdk/solidity/PythUtils.sol`)

---

### Summary

`PythUtils.deriveCrossRate()` validates that `price2` is not negative but does not validate that `price2 != 0`. When a Pyth price feed legitimately returns a price of `0` for the quote asset, `price2` is passed directly as the denominator to `Math.mulDiv()`, which triggers either a Solidity division-by-zero revert or a custom `panic(DIVISION_BY_ZERO)` assembly revert. Any protocol using `deriveCrossRate` in a critical path (liquidations, oracle reads, cross-rate pricing) will be DoS'd for the duration that the quote feed holds a zero price.

---

### Finding Description

In `PythUtils.deriveCrossRate()`, the only price validation performed is a negativity check:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
```

No check for `price2 == 0` exists. After the exponent bounds check, `price2` is cast to `uint64` and used directly as the denominator in `Math.mulDiv`:

```solidity
if (deltaExponent > 0) {
    result = Math.mulDiv(
        uint64(price1),
        10 ** uint64(deltaExponent),
        uint64(price2)          // denominator — no zero guard
    );
} else {
    result = Math.mulDiv(
        uint64(price1),
        1,
        10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // denominator — no zero guard
    );
}
```

`Math.mulDiv` handles a zero denominator in two ways depending on the intermediate 512-bit product:

- When `high == 0`: Solidity's native `low / denominator` reverts with a standard division-by-zero error.
- When `high != 0` and `denominator <= high`: the custom `panic(DIVISION_BY_ZERO)` assembly path fires.

Both paths cause the transaction to revert with no descriptive error. The Pyth Scheduler contract itself already acknowledges that zero prices are a real runtime condition and explicitly guards against them:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
```

`deriveCrossRate` has no equivalent guard.

---

### Impact Explanation

Any on-chain protocol that calls `deriveCrossRate` with a quote feed (`price2`) that returns `0` will have every such call revert. If the function is used in a liquidation engine, a collateral pricing path, or a DEX oracle, the entire feature is DoS'd for as long as the quote feed holds a zero price. The Pyth SDK is the canonical integration path for EVM protocols; a silent revert with no descriptive error makes diagnosis difficult and recovery slow.

---

### Likelihood Explanation

Pyth price feeds can return `price = 0` in documented edge cases: newly listed assets before sufficient publisher coverage, market-halt conditions, or publisher aggregation producing a zero median. The Lazer parser test `test_parseUpdate_optionalMissing_priceZero` explicitly exercises this path, and the Scheduler's zero-price `continue` guard confirms the Pyth team treats zero prices as a realistic runtime value. An unprivileged price updater who submits a valid VAA containing a zero price for a quote asset (e.g., a stablecoin feed during an extreme event) can trigger this revert in every downstream protocol using `deriveCrossRate`.

---

### Recommendation

Add an explicit zero check for `price2` before the division, consistent with the pattern already used in `Scheduler.sol`:

```solidity
if (price2 == 0) {
    revert PythErrors.InvalidInputPrice(); // or a new ZeroPriceInput error
}
```

This should be placed immediately after the negativity check, before the exponent computation. The NatSpec should also be updated to document that `price2 == 0` is rejected.

---

### Proof of Concept

1. Deploy a contract that calls `PythUtils.deriveCrossRate(1e8, -8, 0, -8, -8)` — `price2 = 0`.
2. Observe the call reverts with either a Solidity division-by-zero panic or the custom `panic(DIVISION_BY_ZERO)` from `Math.mulDiv`.
3. No descriptive revert reason is surfaced to the caller.
