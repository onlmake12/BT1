Audit Report

## Title
Rate Limiter Counts Messages Not Hashes, Enabling ~983K Tx-Pool Lookups/Second Per Peer — (`sync/src/relayer/get_transactions_process.rs`, `sync/src/relayer/mod.rs`)

## Summary
The relay rate limiter in `Relayer::try_process` consumes exactly one token per `GetRelayTransactions` message regardless of payload size. Because `MAX_RELAY_TXS_NUM_PER_BATCH` is 32,767 and the cap is 30 messages/second, a single unprivileged peer can drive approximately 983,010 `fetch_txs_with_cycles` hash-lookups per second against the serialized tx-pool controller, potentially saturating it and delaying mining template generation, transaction admission, and proposal processing for all peers.

## Finding Description
`Relayer::new` creates a `governor::RateLimiter` keyed by `(PeerIndex, u32)` with a quota of 30 tokens/second (`mod.rs` L89–92). In `try_process`, `check_key(&(peer, message.item_id()))` deducts exactly one token per message with no awareness of payload size (`mod.rs` L116–123). `GetTransactionsProcess::execute` rejects messages only when `message_len > MAX_RELAY_TXS_NUM_PER_BATCH` (strict greater-than, `get_transactions_process.rs` L35–39), so a message carrying exactly 32,767 hashes passes. The intra-message duplicate guard (`L54–61`) converts the list to a `HashSet<ProposalShortId>` and rejects only if duplicates exist within the same message; it provides no cross-message protection. All hashes are then forwarded in a single awaited `fetch_txs_with_cycles(tx_hashes_set)` call (`L63`). The tx-pool controller is a serialized async actor, so each such call occupies it for the duration of 32,767 hash-map lookups before it can serve any other request.

## Impact Explanation
The tx-pool controller serializes all requests: transaction admission, mining template generation (`get_block_template`), and proposal processing all queue behind it. Sustained flooding at ~983K lookups/second from even one peer can stall the queue, delaying block production and degrading network throughput. This matches **High (10001–15000): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, since the attacker needs only a standard P2P connection and the cost is negligible (sending 30 small TCP messages per second).

## Likelihood Explanation
No privilege, proof-of-work, key, or stake is required — only a standard P2P connection to the relay protocol. The attacker constructs `GetRelayTransactions` messages with 32,767 distinct (non-existent) hashes and sends exactly 30 per second. Every message passes the rate limiter. The attack is trivially scriptable, repeatable indefinitely, and requires no victim interaction.

## Recommendation
Replace per-message token consumption with per-hash token consumption. After parsing `message_len`, call `check_key_n(&(peer, message.item_id()), NonZeroU32::new(message_len as u32).unwrap_or(ONE))` so that a 32,767-hash message costs 32,767 tokens. Alternatively, introduce a separate per-peer hash-per-second budget (e.g., 1,000 hashes/second) enforced before the `fetch_txs_with_cycles` call, independent of message count.
