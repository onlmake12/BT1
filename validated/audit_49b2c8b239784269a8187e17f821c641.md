### Title
Unbounded SQLite Bound-Parameter Query in `remove_batch_by_blobs` Causes Panic in `try_loop_sync` on Rollback of Large Blocks — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

`remove_batch_by_blobs` constructs a single `DELETE … WHERE id IN ($1, …, $N)` query with no chunking. When a block containing more than SQLite's `SQLITE_LIMIT_VARIABLE_NUMBER` (default 999 for SQLite < 3.32.0) transactions is rolled back, the resulting query exceeds the bound-parameter limit, SQLite returns an error, `rollback()` propagates `Err`, and `try_loop_sync` panics unconditionally via `.expect("rollback block should be OK")`, crashing the indexer service.

### Finding Description

**`remove_batch_by_blobs`** (no chunking): [1](#0-0) 

The helper `sqlx_param_placeholders` generates one `$i` placeholder per ID with no upper bound: [2](#0-1) 

`rollback_block` calls `remove_batch_by_blobs` with the full `tx_id_list` (one entry per transaction in the block), and also calls `query_outputs_by_tx_id_list` which uses the same unbounded placeholder pattern: [3](#0-2) [4](#0-3) 

The sync loop panics unconditionally if `rollback()` returns `Err`: [5](#0-4) 

By contrast, `append` errors are handled gracefully (logged and retried): [6](#0-5) 

`append` itself inserts transactions one-by-one, so it never hits the parameter limit regardless of block size: [7](#0-6) 

### Impact Explanation

When the panic fires, the `spawn_blocking` task running `try_loop_sync` terminates. The indexer stops tracking chain state. All RPC consumers receive stale data indefinitely until the process is restarted. On restart the same block may be re-appended and the same reorg re-triggered, causing a crash loop.

### Likelihood Explanation

**SQLite version dependency**: The 999-parameter limit is the default for SQLite < 3.32.0 (released May 2020). SQLite ≥ 3.32.0 raises the default to 32766. CKB's mainnet `max_block_bytes` (~597 000 bytes) with a minimum transaction size of ~100–200 bytes allows roughly 3 000–6 000 transactions per block — well above 999 but well below 32766. The window is therefore: **SQLite < 3.32.0 + block with > 999 transactions**.

**Attacker requirements**: Mining a single valid block does not require majority hashpower. An attacker with any non-zero hashpower can eventually produce such a block. The reorg can be natural (no attacker involvement needed after the block is on-chain) or attacker-triggered. The block itself is consensus-valid; no privileged access is required.

### Recommendation

1. Chunk all `IN (…)` queries in `remove_batch_by_blobs` and `query_outputs_by_tx_id_list` into batches of at most 999 (or a configurable safe limit).
2. Replace `.expect("rollback block should be OK")` with proper error handling (log + retry or graceful shutdown) so a DB error never panics the sync loop. [8](#0-7) 

### Proof of Concept

```
1. Configure node with SQLite backend (SQLite < 3.32.0).
2. Construct a block containing 1 000 minimal transactions (each ~100 bytes; fits within max_block_bytes).
3. Call indexer.append(&large_block) — succeeds (one-by-one insert).
4. Simulate a reorg: call indexer.rollback().
5. Observe: query_outputs_by_tx_id_list or remove_batch_by_blobs generates
   DELETE … WHERE id IN ($1, …, $1000) → SQLite error "too many SQL variables".
6. rollback() returns Err(_).
7. try_loop_sync panics: thread 'tokio-runtime-worker' panicked at
   'rollback block should be OK: DB("too many SQL variables")'.
8. Indexer service stops; all RPC calls return stale tip.
```

async fn remove_batch_by_blobs(
    table_name: &str,
    column_name: &str,
    ids: &[i64],
    tx: &mut Transaction<'_, Any>,
) -> Result<(), Error> {
    if ids.is_empty() {
        return Ok(());
    }

    // build query str
    let mut query_builder = SqlBuilder::delete_from(table_name);
    let sql = query_builder
        .and_where_in(column_name, &sqlx_param_placeholders(1..ids.len())?)
        .sql()
        .map_err(|err| Error::DB(err.to_string()))?;

    // bind
    let mut query: sqlx::query::Query<'_, Any, sqlx::any::AnyArguments<'_>> = sqlx::query(&sql);
    for hash in ids {
        query = query.bind(hash);
    }

    // execute
    query
        .execute(tx.as_mut())
        .await
        .map_err(|err| Error::DB(err.to_string()))?;

    Ok(())
}

fn sqlx_param_placeholders(range: std::ops::Range<usize>) -> Result<Vec<String>, Error> {
    if range.start == 0 {
        return Err(Error::Params("no valid parameter".to_owned()));
