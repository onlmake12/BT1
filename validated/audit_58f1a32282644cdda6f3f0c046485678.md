Audit Report

## Title
Stale `descendants_fee` Accounting After Subtree Removal Enables Pool Eviction Manipulation - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-removes all link entries for the entire subtree before calling `remove_entry` on each node. Because `update_ancestors_index_key` relies on `calc_ancestors` to find surviving ancestors, and those link entries are already gone, surviving ancestors outside the removed subtree never receive the `sub_descendant_weight` call. Their `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` remain permanently inflated, causing their `EvictKey` to be artificially elevated and making them immune to pool eviction.

## Finding Description

`remove_entry_and_descendants` collects the root and all its descendants, then calls `remove_entry_links` for every ID in that set before calling `remove_entry`: [1](#0-0) 

`remove_entry_links` removes the entry from `links.inner` entirely via `self.links.remove(id)`, and also severs it from its parents' children lists: [2](#0-1) 

When `remove_entry` subsequently calls `update_ancestors_index_key`, it calls `self.links.calc_ancestors(&child.proposal_short_id())`: [3](#0-2) 

`calc_ancestors` delegates to `calc_relative_ids`, which looks up the entry in `self.inner`. Since `remove_entry_links` already called `self.links.remove(id)`, the lookup returns `None`, `direct` is an empty set, and `calc_relation_ids` returns an empty set: [4](#0-3) 

The surviving ancestors of the removed subtree root — which are **not** in `removed_ids` — are therefore never passed to `sub_descendant_weight`. Their `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` remain permanently inflated.

The code comment in `remove_entry_and_descendants` ("so that we won't update_descendants_index_key in remove_entry") reveals the intent was only to suppress descendant-side updates for entries that are themselves being removed. The side effect of also suppressing ancestor-side updates for surviving entries outside the subtree is the defect.

`sub_descendant_weight` uses `saturating_sub`, so the stale inflation is silent — no panic, no error log, no correction path: [5](#0-4) 

The stale `descendants_fee` directly feeds `EvictKey` computation via `descendants_feerate.max(feerate)`: [6](#0-5) 

`resolve_conflict` — the RBF path — calls `remove_entry_and_descendants`, making this reachable by any tx-pool submitter: [7](#0-6) 

The pool's size-limit eviction loop also calls `remove_entry_and_descendants`, so the same stale accounting affects eviction under memory pressure: [8](#0-7) 

`total_tx_size` has a `recompute_total_stat` fallback on underflow, but `descendants_fee` has no equivalent correction path: [9](#0-8) 

## Impact Explanation

A surviving ancestor retains an inflated `descendants_feerate`, causing its `EvictKey` to rank it as highly valuable. The pool's eviction loop (`limit_size`) selects entries with the lowest `EvictKey` for removal; an entry with an artificially inflated `EvictKey` is never selected. An attacker can keep a near-zero-fee parent permanently in the pool and fill the pool with such entries, causing legitimate transactions to be rejected with `Reject::Full`. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

`send_transaction` is an unprivileged RPC endpoint. RBF is enabled when `min_rbf_rate > min_fee_rate` (a common production configuration). Each RBF replacement requires paying a higher fee for the child, but permanently inflates the parent's `descendants_fee` by the replaced child's fee. After N replacements the inflation is `sum(child_fees[0..N-1])`, growing without bound. No privileged access, key material, or majority hashpower is required. The cost per inflation step is bounded by the RBF fee bump requirement, making this economically feasible for a sustained attack.

## Recommendation

Before calling `remove_entry_links` for any entry in the subtree, collect the root's surviving ancestors and apply `sub_descendant_weight` to each of them. Concretely, in `remove_entry_and_descendants`, for the subtree root `id`, call `self.links.calc_ancestors(id)` **before** the link-removal loop, then iterate over those ancestor IDs and call `sub_descendant_weight` with the root entry's weight. For deeper subtrees, the same must be done for each node in `removed_ids` whose ancestors are not themselves in `removed_ids`. This mirrors the logic already present in `update_ancestors_index_key` but executed before links are torn down.

## Proof of Concept

```rust
// Setup: P is a low-fee parent, C1 is a high-fee child spending P's output.
// After adding both: P.descendants_fee = P.fee + C1.fee

// Step 1: Attacker submits C1' (higher fee, same input as C1).
// resolve_conflict calls remove_entry_and_descendants(C1_id):
//   remove_entry_links(C1_id)  → severs P→C1 link, removes C1 from links.inner
//   remove_entry(C1_id)
//     → update_ancestors_index_key(C1, Remove)
//         → calc_ancestors(C1_id) == {} (C1 already gone from links.inner)
//         → P.descendants_fee NOT decremented  ← BUG
// C1' added as new child of P → P.descendants_fee += C1'.fee

// After step 1:
//   P.descendants_fee = P.fee + C1.fee (stale, never removed) + C1'.fee

// After N replacements:
//   P.descendants_fee = P.fee + sum(C1..CN fees) + CN'.fee
//   P.EvictKey reflects this inflated value → P is never evicted.

// Attacker fills pool with such entries; legitimate txs receive Reject::Full.

// 1. Add P (low fee) and C1 (high fee, child of P) to pool.

// 3. Call pool_map.remove_entry_and_descendants(&C1.proposal_short_id()).
// 4. Assert P.descendants_fee == P.fee.  // FAILS: still equals P.fee + C1.fee
```

```rust
    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
```

```rust
    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
```

```rust
    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
```

```rust
    pub fn sub_descendant_weight(&mut self, entry: &TxEntry) {
        self.descendants_count = self.descendants_count.saturating_sub(1);
        self.descendants_size = self.descendants_size.saturating_sub(entry.size);
        self.descendants_cycles = self.descendants_cycles.saturating_sub(entry.cycles);
        self.descendants_fee = Capacity::shannons(
            self.descendants_fee

                .saturating_sub(entry.fee.as_u64()),
```
