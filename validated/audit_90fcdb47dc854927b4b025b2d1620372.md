Audit Report

## Title
Missing `compact_target` Epoch Difficulty Validation in Compact Block Relay Path Enables Zero-Cost HeaderMap and Disk Flooding — (`sync/src/relayer/compact_block_process.rs`, `verification/src/header_verifier.rs`)

## Summary

The compact block relay path's `HeaderVerifier` omits the epoch `compact_target` check that the contextual block verifier enforces. Because rate limiting is explicitly disabled for `CompactBlock` messages and `compact_target = 0x20ffffff` produces a near-maximum PoW target that any nonce satisfies with ~(1 − 1/2²⁴) probability, an unprivileged peer can flood the node's `HeaderMap` (up to 256 MB in memory, then unbounded on disk via sled), pollute `pending_compact_blocks` indefinitely, and trigger `GetBlockTransactions` amplification — all at zero real computational cost.

## Finding Description

**Rate limiting is explicitly bypassed for `CompactBlock`:**

`sync/src/relayer/mod.rs` L112–114 sets `should_check_rate = false` for all `CompactBlock` messages, with the comment *"CompactBlock will be verified by POW, it's OK to skip rate limit checking."* This assumption is the root of the problem.

**`compact_to_target(0x20ffffff)` produces a near-maximum target that passes PoW for essentially any nonce:**

From `util/types/src/utilities/difficulty.rs` L62–76: exponent = 32, mantissa = `0xffffff`, overflow = `!mantissa.is_zero() && (32 > 32)` = `false`. Target = `0xffffff << 232`. The test suite at `util/types/src/utilities/tests/difficulty.rs` L24–35 confirms `target_to_compact(U256::max_value()) == 0x20ffffff` and `compact_to_difficulty(0x20ffffff) == U256::one()`.

In `EaglesongPowEngine::verify` (`pow/src/eaglesong.rs` L16–26), `block_target.is_zero() || overflow` is `false`, so execution reaches the hash comparison. With `block_target = 0xffffff << 232`, only hashes whose top 24 bits equal `0xffffff` AND whose lower 232 bits are nonzero exceed the target — a ~1/2²⁴ failure probability. Virtually any nonce passes.

**The relay-path `HeaderVerifier` does not check `compact_target` against the epoch's expected difficulty:**

`contextual_check` in `compact_block_process.rs` L324–325 constructs a `HeaderVerifier` and calls `verify`. That verifier (`verification/src/header_verifier.rs` L32–50) runs only four sub-checks: `PowVerifier`, `NumberVerifier`, `EpochVerifier` (continuity only), and `TimestampVerifier`.

The relay-path `EpochVerifier::verify` (`verification/src/header_verifier.rs` L133–148) only checks `is_well_formed` and `is_successor_of` — it never validates the `compact_target` value against the epoch's expected target. An attacker sets any epoch field satisfying `is_successor_of` (trivially: increment index each block) and any `compact_target`.

**The real `compact_target` check exists only in the contextual block verifier**, which runs during full block acceptance — after `insert_valid_header` has already mutated state (`verification/contextual/src/contextual_block_verifier.rs` L500–507). The attacker never submits a full block, so this check is never reached.

**State mutation is unconditional once the relay-path check passes:**

`insert_valid_header` is called immediately after `contextual_check` returns `ok` (`compact_block_process.rs` L77–78), inserting the attacker's header into `HeaderMap` and updating `shared_best_header`. Each inserted header also builds a skip-list (`sync/src/types/mod.rs` L1114–1128), consuming CPU.

**`HeaderMap` memory is capped at 256 MB, but the sled backend is unbounded on disk:**

The 5-second `limit_memory` task (`shared/src/types/header_map/mod.rs` L57–76) spills excess entries to sled (`backend_sled.rs` L63–73), which has no size cap.

**`pending_compact_blocks` is also polluted without bound:**

When block reconstruction fails (all attacker blocks, since they carry no real transactions), the compact block is inserted into the unbounded `PendingCompactBlockMap` (`compact_block_process.rs` L354–361). The only cleanup path (`retain` by epoch number at L112–116 and `remove` at L107 on block acceptance) is never triggered because the attacker never submits a full block.

**Network amplification:** For each compact block with missing transactions, the node sends a `GetBlockTransactions` message back to the attacker (`compact_block_process.rs` L363–378).

## Impact Explanation

**High severity** — matches two allowed CKB bounty impacts:
- *"Vulnerabilities which could easily crash a CKB node"*: Unbounded sled disk growth causes disk exhaustion and node crash/unresponsiveness. `HeaderMap` consumes up to 256 MB of RAM from zero-cost headers.
- *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"*: Every attacker compact block elicits a `GetBlockTransactions` response, enabling traffic amplification. Thousands of such headers can be generated and relayed per second with no real hashpower.

## Likelihood Explanation

The attack requires no hashpower, no privileged access, and no special network position — only a standard P2P connection. Building a chain of headers with `compact_target = 0x20ffffff`, valid epoch succession (`is_successor_of` passes trivially by incrementing the index field), and monotonically increasing timestamps is trivial. With ~1/2²⁴ failure probability per nonce, an attacker finds a valid nonce in at most a few attempts on average. The attack is fully repeatable and requires no victim interaction.

## Recommendation

Add a `compact_target` validation step to `contextual_check` in `sync/src/relayer/compact_block_process.rs` (or to `HeaderVerifier::verify` in `verification/src/header_verifier.rs`) that computes the expected epoch target from the parent's epoch extension and rejects headers whose `compact_target` does not match. This mirrors the check already present in `verification/contextual/src/contextual_block_verifier.rs` `EpochVerifier::verify` at lines 500–507. The fix must execute **before** `insert_valid_header` is called.

## Proof of Concept

1. Connect to a mainnet/testnet node as a standard P2P peer.
2. Starting from genesis, build a chain of headers: for each block N, set `compact_target = 0x20ffffff`, `number = N`, `epoch = valid_successor(parent.epoch)` (satisfying `is_successor_of` by incrementing the index field), `timestamp = parent.timestamp + 1`, `parent_hash = hash(block_{N-1})`, and any nonce (retry on the ~1/2²⁴ failure probability).
3. Wrap each header as a `CompactBlock` with only a cellbase prefill (so reconstruction always fails with missing transactions, satisfying `CompactBlockVerifier`).
4. Send each `CompactBlock` over the relay protocol. No rate limit applies.
5. **Observe**: `HeaderMap` memory grows proportionally to messages sent; sled backend receives writes every 5 seconds and grows without bound; `pending_compact_blocks` accumulates entries; `GetBlockTransactions` messages are sent back to the attacker for each block.
6. The `compact_target` mismatch is only caught in the contextual block verifier — which is never reached because the attacker never submits a full block.

pub fn compact_to_target(compact: u32) -> (U256, bool) {
    let exponent = compact >> 24;
    let mut mantissa = U256::from(compact & 0x00ff_ffff);

    } else {
        ret = mantissa.clone();
        ret <<= 8 * (exponent - 3);
    }

    let overflow = !mantissa.is_zero() && (exponent > 32);
    (ret, overflow)

impl HeaderMap {
    pub fn new<P>(
        tmpdir: Option<P>,
        memory_limit: usize,
        async_handle: &Handle,
        ibd_finished: Arc<AtomicBool>,
    ) -> Self
    where
        P: AsRef<path::Path>,
    {
        if memory_limit < ITEM_BYTES_SIZE {
            panic!("The limit setting is too low");
        }
        if memory_limit < WARN_THRESHOLD {
            ckb_logger::warn!(
                "The low memory limit setting {} will result in inefficient synchronization",
                memory_limit
            );
        }
        let size_limit = memory_limit / ITEM_BYTES_SIZE;
        let inner = Arc::new(HeaderMapKernel::new(tmpdir, size_limit, ibd_finished));
        let map_weak = Arc::downgrade(&inner);

            let mut interval = tokio::time::interval(INTERVAL);

                    _ = interval.tick() => {
                        if let Some(map) = map_weak.upgrade() {

                            break;
                        }
                    }
                    _ = stop_rx.cancelled() => {
                        info!("HeaderMap limit_memory received exit signal, exit now");
                        break
                    },
                }
            }
        });
