### Title
Provider Accrued Fees Permanently Locked When Fee Manager Removed Without Prior Withdrawal — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, a registered provider can call `setFeeManager(address(0))` to remove their fee manager at any time, without any check that their accrued fees have been withdrawn first. Unlike `Entropy.sol`, `Echo.sol` provides **no direct `withdraw()` function for providers**. The only withdrawal path for provider fees is `withdrawAsFeeManager()`, which requires `msg.sender == feeManager`. Setting `feeManager` to `address(0)` makes this condition permanently unsatisfiable, locking all accrued provider fees in the contract with no recovery path.

---

### Finding Description

**Step 1 — Provider registers with `feeManager` defaulting to `address(0)`.**

`registerProvider()` does not initialize `feeManager`, so it defaults to `address(0)`:

```solidity
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    ...
    provider.isRegistered = true;
}
``` [1](#0-0) 

**Step 2 — Provider accrues fees from user requests.**

Each `requestPriceUpdatesWithCallback` call credits `accruedFeesInWei` to the provider's `ProviderInfo`.

**Step 3 — Provider calls `setFeeManager(address(0))` to "disable" the fee manager role.**

`setFeeManager` performs no check on `accruedFeesInWei` before overwriting the fee manager:

```solidity
function setFeeManager(address manager) external override {
    require(_state.providers[msg.sender].isRegistered, "Provider not registered");
    address oldFeeManager = _state.providers[msg.sender].feeManager;
    _state.providers[msg.sender].feeManager = manager;
    emit FeeManagerUpdated(msg.sender, oldFeeManager, manager);
}
``` [2](#0-1) 

**Step 4 — The only withdrawal path for provider fees is now permanently blocked.**

`withdrawAsFeeManager` enforces `msg.sender == feeManager`. With `feeManager == address(0)`, this check can never pass:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(msg.sender == _state.providers[provider].feeManager, "Only fee manager");
    ...
}
``` [3](#0-2) 

**Step 5 — No alternative withdrawal path exists for providers in Echo.**

Unlike `Entropy.sol`, which exposes a direct `withdraw()` function callable by the provider themselves, `Echo.sol`'s `IEcho` interface exposes only:
- `withdrawFees(uint128 amount)` — admin-only, drains `accruedPythFeesInWei` (protocol fees), not provider fees
- `withdrawAsFeeManager(address provider, uint128 amount)` — fee manager only [4](#0-3) 

There is no `withdraw()` for providers in Echo, unlike Entropy: [5](#0-4) 

The admin's `withdrawFees` only touches `_state.accruedPythFeesInWei`, not provider balances, so there is no admin rescue path either.

---

### Impact Explanation

A provider who sets `feeManager` to `address(0)` — whether intentionally (to "disable" the role) or accidentally — permanently forfeits all accrued fees stored in `_state.providers[provider].accruedFeesInWei`. The ETH is locked in the contract with no recovery mechanism. This is a direct, irreversible loss of provider funds.

---

### Likelihood Explanation

The scenario is realistic:
1. A provider may set `feeManager = address(0)` to revoke a previously trusted fee manager, not realizing this also blocks their own future withdrawals.
2. Providers who never set a fee manager (defaulting to `address(0)`) can never withdraw their fees at all — a silent trap for any provider who does not read the documentation carefully.
3. The entry path requires only a registered provider calling a public function — no privileged access needed.

---

### Recommendation

Add a guard in `setFeeManager` that prevents setting the fee manager to `address(0)` while `accruedFeesInWei > 0`:

```solidity
function setFeeManager(address manager) external override {
    require(_state.providers[msg.sender].isRegistered, "Provider not registered");
    require(
        manager != address(0) || _state.providers[msg.sender].accruedFeesInWei == 0,
        "Withdraw fees before removing fee manager"
    );
    ...
}
```

Alternatively (and more robustly), add a direct `withdraw()` function for providers in Echo, mirroring the pattern in `Entropy.sol`.

---

### Proof of Concept

1. Provider calls `registerProvider(...)` — `feeManager` defaults to `address(0)`.
2. Users call `requestPriceUpdatesWithCallback(...)` — `accruedFeesInWei` grows.
3. Provider calls `setFeeManager(address(0))` (no-op if already zero, or to revoke a previous manager).
4. Provider attempts to call `withdrawAsFeeManager(providerAddr, amount)` — reverts with `"Only fee manager"` because `msg.sender != address(0)`.
5. No other withdrawal function exists for the provider. Fees are permanently locked.
