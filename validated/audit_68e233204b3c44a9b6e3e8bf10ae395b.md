### Title
Port-Zero Peer Store Poisoning via Unauthenticated `GetNodes{listen_port: Some(0)}` — (`network/src/protocols/discovery/mod.rs`, `network/src/protocols/discovery/state.rs`)

---

### Summary

An inbound peer can send a `GetNodes` message with `listen_port = Some(0)`. The handler calls `update_port(0)` with no validation, producing a `RemoteAddress::Listen(/ip4/<peer_ip>/tcp/0/p2p/...)` that is unconditionally inserted into the peer store and subsequently broadcast to other nodes. No guard anywhere in the pipeline rejects port 0.

---

### Finding Description

**Step 1 — Attacker-controlled entry point.**
Any inbound TCP session that completes the discovery handshake can send one `GetNodes` message (a second one triggers `DuplicateGetNodes` and disconnects). The message carries an attacker-chosen `listen_port: Option<u16>`.

**Step 2 — `update_port` performs no validation.** [1](#0-0) 

`Protocol::Tcp(_) => Protocol::Tcp(port)` blindly substitutes whatever `u16` was supplied, including `0`. The result is immediately promoted to `RemoteAddress::Listen(addr)` with no rejection path.

**Step 3 — The Listen address is added to the peer store.** [2](#0-1) 

After `update_port(port)` the code checks `if let RemoteAddress::Listen(ref addr) = state.remote_addr` and calls `add_new_addr` with the poisoned address.

**Step 4 — `is_valid_addr` only checks IP reachability, not port.** [3](#0-2) 

`is_valid_addr` calls `multiaddr_to_socketaddr` and `is_reachable` on the IP. A public IP with port 0 passes this check. There is no `port != 0` guard anywhere in `add_new_addr` → `add_new_addrs` → `peer_store.add_addr`.

**Step 5 — `add_addr` stores it without port validation.** [4](#0-3) 

Only the ban list is consulted. `AddrInfo::new` calls `base_addr(&addr)` (which strips the peer-id component but preserves the TCP port) and stores the entry.

**Step 6 — The poisoned address is broadcast.** [5](#0-4) 

The `notify` loop calls `get_random` → `fetch_random_addrs`, which can return the port-0 entry, and pushes it into `announce_multiaddrs` for up to 3 peers per tick.

---

### Impact Explanation

- `/ip4/X/tcp/0/p2p/...` is stored in the peer store (capacity 16 384 slots).
- It is relayed to other nodes as a legitimate announce address.
- Every node that receives it will attempt to dial port 0, fail, increment `attempts_count`, and eventually mark the entry non-connectable — but only after `ADDR_MAX_RETRIES` failed dials.
- While alive in the store the entry occupies a slot and can trigger eviction of legitimate peers when the store is full.
- The address is also inserted into `addr_known` bloom filters, preventing the correct listen address from being re-announced to that session.

The realistic impact is **peer store pollution and wasted dial attempts** propagated across the gossip graph. The "network partitioning / consensus deviation" framing in the question is overstated; a single bad address per connection is not sufficient to partition the network. However, an attacker opening many inbound connections (each injecting one port-0 address) could meaningfully degrade peer discovery quality.

---

### Likelihood Explanation

The path requires only an inbound TCP connection — no authentication, no PoW, no special role. The `GetNodes` message is the first protocol message sent by a connecting peer. The bug is reachable on mainnet from any IP.

---

### Recommendation

Add a `port != 0` guard in `update_port` before accepting the caller-supplied value:

```rust
// state.rs – update_port
pub(crate) fn update_port(&mut self, port: u16) {
    if port == 0 {
        return;   // reject invalid listen port
    }
    // ... existing logic
}
```

Alternatively, add the check at the call site in `mod.rs` line 124:

```rust
if let Some(port) = listen_port.filter(|&p| p != 0) {
```

### Proof of Concept

```rust
// Unit test sketch (no network required)
use crate::protocols::discovery::state::{RemoteAddress, SessionState};
use p2p::multiaddr::Multiaddr;

let init_addr: Multiaddr = "/ip4/1.2.3.4/tcp/54321/p2p/Qm...".parse().unwrap();
let mut remote = RemoteAddress::Init(init_addr);
remote.update_port(0);
// Assert: must NOT become Listen with port 0
match &remote {
    RemoteAddress::Listen(addr) => {
        let port = addr.iter().find_map(|p| {
            if let p2p::multiaddr::Protocol::Tcp(port) = p { Some(port) } else { None }
        });
        assert_ne!(port, Some(0), "port 0 must be rejected");
    }
    _ => {}
}
```

The test currently **fails** (i.e., port 0 is accepted), confirming the missing guard.
