### Title
Insufficient Validation of Total Inputs Capacity in DAO Withdrawal Transactions Allows Capacity Inflation - (File: `verification/src/transaction_verifier.rs`)

### Summary

`CapacityVerifier::verify()` unconditionally skips the `inputs_sum >= outputs_sum` check for **any** transaction that contains at least one DAO-type-script input. The check is delegated entirely to the DAO type script, which only validates DAO-cell-specific withdrawal amounts and does not enforce total transaction capacity conservation. A miner can craft a mixed transaction (DAO + non-DAO inputs) where total outputs exceed total inputs plus DAO interest, creating CKB capacity from thin air.

### Finding Description

In `verification/src/transaction_verifier.rs`, `CapacityVerifier::verify()` contains the following guard:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    let inputs_sum = self.resolved_transaction.inputs_capacity()?;
    let outputs_sum = self.resolved_transaction.outputs_capacity()?;
    if inputs_sum < outputs_sum {
        return Err(...OutputsSumOverflow...);
    }
}
``` [1](#0-0) 

`valid_dao_withdraw_transaction()` returns `true` if **any** input uses the DAO type script:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
``` [2](#0-1) 

When this returns `true`, the entire `OutputsSumOverflow` check is bypassed for **all** inputs and outputs in the transaction — including non-DAO inputs. The code comment states: *"DAO withdraw transaction is verified via the type script of DAO cells."* [3](#0-2) 

The DAO type script (running in CKB-VM) only validates the withdrawal amount for each individual DAO cell. Per `DaoCalculator::calculate_maximum_withdraw()`, it computes the interest-bearing withdrawal for a single DAO output: [4](#0-3) 

It does not enforce that the sum of all outputs is bounded by the sum of all inputs (DAO + non-DAO). Non-DAO inputs in the same transaction are accounted for in `transaction_maximum_withdraw()` only in the tx-pool path (`DaoCalculator::transaction_fee()`): [5](#0-4) 

However, `DaoCalculator::transaction_fee()` is called only in `tx-pool/src/util.rs` (`check_tx_fee`), not in the consensus block verifier: [6](#0-5) 

The block verifier uses `ContextualTransactionVerifier`, which includes `CapacityVerifier` but does **not** call `DaoCalculator::transaction_fee()`. A miner bypassing the tx-pool (via `submit_block` RPC) faces only the `CapacityVerifier` check, which is skipped for DAO transactions.

### Impact Explanation

A miner can construct a transaction with:
- One DAO input (e.g., 100 CKB, earning 10 CKB interest → max withdrawal 110 CKB)
- One non-DAO input (e.g., 50 CKB)
- Outputs totaling 170 CKB (110 CKB DAO output + 60 CKB non-DAO output)

Total inputs: 150 CKB + 10 CKB interest = 160 CKB maximum. Total outputs: 170 CKB. The 10 CKB discrepancy is created from thin air.

The `CapacityVerifier` skips the check. The DAO type script validates the 110 CKB DAO withdrawal as correct. No consensus-level check catches the 10 CKB over-issuance. The block is accepted by the network, permanently inflating the CKB supply.

### Likelihood Explanation

Any miner — regardless of hashpower — can exploit this. The miner crafts the transaction locally, includes it in a self-assembled block, and submits it via the `submit_block` RPC. No majority hashpower is required; the miner only needs to find one valid PoW solution, which is guaranteed to happen eventually. The attack is fully deterministic once a valid block is found.

### Recommendation

Remove the blanket skip of `OutputsSumOverflow` for DAO transactions in `CapacityVerifier::verify()`. Instead, perform the capacity check for all transactions and separately allow the DAO type script to validate the DAO-specific interest component. Alternatively, explicitly validate that `outputs_sum <= inputs_sum_non_dao + dao_maximum_withdraw` at the consensus level, mirroring the logic already present in `DaoCalculator::transaction_fee()`.

### Proof of Concept

1. Obtain a DAO withdrawing cell (phase 2 input) with capacity 100 CKB and a non-DAO UTXO with 50 CKB.
2. Construct a transaction:
   - Inputs: [DAO withdrawing cell (100 CKB), non-DAO cell (50 CKB)]
   - Outputs: [DAO output (110 CKB, correct per DAO type script), non-DAO output (60 CKB)]
   - Total outputs: 170 CKB > 160 CKB maximum
3. Include this transaction in a self-mined block.
4. Submit via `submit_block` RPC.
5. `CapacityVerifier::verify()` calls `valid_dao_withdraw_transaction()` → returns `true` → skips `OutputsSumOverflow` check entirely.
6. DAO type script executes, validates 110 CKB withdrawal → passes.
7. Block is accepted. 10 CKB has been created from thin air.

    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {

                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )

    // Theoretically we cannot use size as weight directly to calculate fee_rate,

    // reject txs which fee lower than min fee rate

        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
