Audit Report

## Title
Integer Overflow in Guard Bypasses `GET_LAST_STATE_PROOF_LIMIT`, Enabling Unbounded DB Iteration — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary

The guard at line 201 uses `(last_n_blocks as usize) * 2`, which wraps to zero in Rust release mode when `last_n_blocks = 2^63`. This bypasses the `GET_LAST_STATE_PROOF_LIMIT = 1000` cap entirely. After bypass, both `reorg_last_n_numbers` and `last_n_numbers` can expand to cover the entire chain history, causing millions of sequential DB reads per single crafted request.

## Finding Description

**Root cause — overflow in the guard:** [1](#0-0) 

`last_n_blocks` is a `u64` decoded directly from the peer message with no prior validation. On a 64-bit target, `last_n_blocks as usize` is a no-op. In Rust release mode, `(2^63usize) * 2` wraps to `0`. With an empty `difficulties` list, the guard evaluates `0 + 0 > 1000` → **false** → bypassed.

**First unbounded vector — `reorg_last_n_numbers`:** [2](#0-1) 

With `last_n_blocks = 2^63` and `start_block_number = N`, `min(N, 2^63) = N`, so `min_block_number = 0` and the range becomes `(0..N)` — N entries. The `else` branch is triggered whenever the attacker sends a `start_hash` that does not match the actual ancestor at `start_block_number`, which requires no privilege.

**Second unbounded vector — `last_n_numbers`:** [3](#0-2) 

With `last_n_blocks = 2^63`, the condition `last_block_number - start_block_number <= last_n_blocks` is always true for any realistic chain. So `last_n_numbers = (start_block_number..last_block_number)` — all blocks from start to tip.

Combined, `block_numbers` covers `(0..last_block_number)` — the entire chain.

**No size check before the expensive loop:** [4](#0-3) 

`block_numbers` is passed directly into `complete_headers` with no size guard. Inside, each entry triggers `get_ancestor` + `get_block` + `chain_root_mmr.get_root()` — three DB reads per block. [5](#0-4) 

## Impact Explanation

On a mainnet node with ~14 million blocks, a single crafted `GetLastStateProof` message with `last_n_blocks = 2^63` and `start_block_number = 1` causes ~14 million sequential DB reads in a single request handler. An attacker sending this repeatedly, or from multiple peers simultaneously, causes sustained I/O and CPU exhaustion that degrades or crashes the node. This matches the allowed impact: **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**. [6](#0-5) 

## Likelihood Explanation

The attack requires only a P2P connection to the light-client protocol port — no authentication, no PoW, no stake. The crafted message is a single valid-length molecule-encoded `GetLastStateProof` packet. The overflow value (`2^63`) is a fixed constant. The mismatched `start_hash` (to trigger the `else` branch) requires sending any arbitrary `Byte32` value. Any attacker who can connect to the node's light-client port can trigger this repeatedly.

## Recommendation

1. Reject oversized `last_n_blocks` directly before any further use:
```rust
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("last_n_blocks too large");
}
```
2. Replace the unchecked multiplication with saturating arithmetic:
```rust
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```
3. Add a size check on `block_numbers` after construction and before calling `complete_headers`, as defense-in-depth.

## Proof of Concept

```rust
let content = packed::GetLastStateProof::new_builder()
    .last_hash(valid_main_chain_tip_hash)
    .start_hash(packed::Byte32::zero())       // mismatched → triggers else branch
    .start_number(1u64.pack())                // start near genesis → reorg covers 0..1
    .last_n_blocks(9223372036854775808u64.pack()) // 2^63
    .difficulty_boundary(some_valid_boundary)
    // difficulties: empty → len() == 0
    .build();
```

Guard: `0 + (9223372036854775808usize).wrapping_mul(2) = 0 > 1000` → **false** → passes.

`last_n_numbers = (1..last_block_number)` → ~14 million entries on mainnet → ~42 million DB reads in `complete_headers` per request.
