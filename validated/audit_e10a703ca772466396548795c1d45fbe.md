### Title
Inaccurate Multi-Denom Fee Sufficiency Check Assumes 1:1 Token Value Without Price Conversion — (`target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

---

### Summary

The Osmosis-specific `is_fee_sufficient` function in the Pyth CosmWasm contract accepts multiple token denominations as payment for price-feed update fees, but sums their raw amounts without converting them to a common value. This is the same root cause as the Alchemix `_melt` bug: an implicit 1:1 price ratio assumption between tokens of different market values. Any unprivileged caller can satisfy the fee check by sending a token worth less than `uosmo` in the required raw amount.

---

### Finding Description

`is_fee_sufficient` (Osmosis feature flag, `contract.rs` lines 163–198) iterates over all coins sent by the caller and accumulates their raw `u128` amounts into a single `total_amount`, regardless of denomination:

```rust
let mut total_amount = 0u128;
for coin in &info.funds {
    if coin.denom != state.fee.denom && !is_allowed_tx_fees_denom(deps, &coin.denom) {
        return Err(...)?;
    }
    total_amount = total_amount.checked_add(coin.amount.u128())...;
}
let base_denom_fee = get_update_fee(deps, data)?;
Ok(base_denom_fee.amount.u128() <= total_amount)
```

The required fee (`base_denom_fee`) is denominated in `uosmo`. The check passes if the raw sum of all sent coins — regardless of their denomination — meets or exceeds that amount. No spot-price conversion is applied. The companion function `get_update_fee_for_denom` (lines 721–735) has the same flaw: it returns the fee amount unchanged in the requested denom, treating 1 `uion` as equivalent to 1 `uosmo`.

The developers explicitly acknowledge this in two inline comments:

> *"convert each of them to the base token that is 'uosmo' … this logic will be updated to use spot price for different valid tokens in future"* (lines 169–172, 194–196)

Allowed non-base denoms include `uion` and IBC-wrapped tokens (e.g., `ibc/FF3065989E34457F342D4EFB8692406D49D4E2B5C70F725F127862E22CE6BDCD`), both of which trade at prices different from `uosmo`. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

A caller can pay the required fee using a token whose market value is lower than `uosmo` — for example, sending `N uion` when `N uion < N uosmo` in USD terms — and the contract will accept it as sufficient. This constitutes a fee-bypass: the protocol receives less economic value than it requires. If governance ever raises the per-VAA fee above its current negligible level (1 `uosmo`), the discrepancy between accepted token values becomes directly exploitable to drain fee revenue or to submit price updates at near-zero cost. The Osmosis mainnet deployment is confirmed active in `CosmWasmChains.json`. [3](#0-2) 

---

### Likelihood Explanation

The entry point is the public `update_price_feeds` execute message, callable by any unprivileged address on Osmosis. No special role or key is required. The attacker only needs to send an allowed non-base denom (e.g., `uion`) whose spot price is below `uosmo`. This is a persistent market condition. The exploit requires no sandwiching, no governance access, and no off-chain coordination. [4](#0-3) 

---

### Recommendation

Replace the raw-amount summation with a spot-price-converted accumulation before comparing against `base_denom_fee`. The Osmosis `TxfeesQuerier` already provides `denom_pool_id`; the corresponding pool can be queried for the current spot price of each non-base denom in terms of `uosmo`. The corrected logic should:

1. For each coin in `info.funds`, query the `uosmo`-equivalent value using the Osmosis TWAP or spot price.
2. Sum the converted values.
3. Compare the converted total against `base_denom_fee.amount`.

Until spot-price conversion is implemented, the safest mitigation is to reject all non-base denoms (i.e., require payment exclusively in `uosmo`), which is already the behavior of the non-Osmosis `is_fee_sufficient` path. [5](#0-4) 

---

### Proof of Concept

Assume the on-chain fee is configured as `{ denom: "uosmo", amount: 100 }` and `uion` trades at 0.5× the value of `uosmo`.

1. Caller constructs an `UpdatePriceFeeds` execute message with one VAA.
2. Caller attaches `{ denom: "uion", amount: 100 }` as `info.funds`.
3. `is_allowed_tx_fees_denom` returns `true` for `uion`.
4. `total_amount` accumulates to `100`.
5. `base_denom_fee.amount` is `100`.
6. `Ok(100 <= 100)` → `true` — fee accepted.
7. The caller paid 100 `uion` ≈ 50 `uosmo` worth of value, satisfying a 100 `uosmo` fee requirement at half cost.

The same path is confirmed by the existing test at lines 1010–1012, which asserts that `coins(100, "uion")` satisfies a fee of `100` without any value conversion: [6](#0-5)

/// Osmosis can support multiple tokens for transaction fees
/// This will return update fee for the given denom only if that denom is allowed in Osmosis's txFee module
/// Else it will throw error
pub fn get_update_fee_for_denom(deps: &Deps, vaas: &[Binary], denom: String) -> StdResult<Coin> {
    let config = config_read(deps.storage).load()?;

    // if the denom is not a base denom it should be an allowed one
    if denom != config.fee.denom && !is_allowed_tx_fees_denom(deps, &denom) {
        return Err(PythContractError::InvalidFeeDenom { denom })?;
    }

    // the base fee is set to -> denom = base denom of a chain, amount = 1
    // which is very minimal
    // for other valid denoms too we are using the base amount as 1
    // base amount is multiplied to number of vaas to get the total amount

    // this will be change later on to add custom logic using spot price for valid tokens
    Ok(coin(get_update_fee_amount(deps, vaas)?, denom))
}
