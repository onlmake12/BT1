### Title
Unnecessary `get_block_template` Call Inside `get_tx_pool_info` Enables Repeated Expensive Block Assembly via RPC — (File: `rpc/src/module/terminal.rs`)

---

### Summary

The `get_tx_pool_info` helper in the Terminal RPC module unconditionally calls `get_block_template` — a full, lock-acquiring block-template assembly operation — every time the cache is bypassed. The only value extracted from that call is `block_template.transactions.len()` (the "committing" count). An unprivileged RPC caller can force the cache to be bypassed on every request by setting the `TX_POOL_INFO` refresh bit flag, causing repeated expensive block-template assembly with no rate limit.

This is the direct CKB analog of the MakerDAO `pot.drip` issue: an expensive operation is invoked unconditionally inside a hot path when a simple guard (or a cheaper data source) could make it a no-op.

---

### Finding Description

In `rpc/src/module/terminal.rs`, the `get_tx_pool_info` function first checks the cache:

```rust
if !refresh.contains(RefreshKind::TX_POOL_INFO)
    && let Some(cached) = self.cache.get_tx_pool_info()
{
    return Ok(cached);
}
``` [1](#0-0) 

When the caller supplies `refresh = 0x4` (`TX_POOL_INFO`), the cache is skipped entirely and the function proceeds to call:

```rust
let block_template = self
    .shared
    .get_block_template(None, None, None)
    ...
``` [2](#0-1) 

The result is used only for:

```rust
committing: (block_template.transactions.len() as u64).into(),
``` [3](#0-2) 

`get_block_template` is not a lightweight read. It acquires the tx-pool lock, selects and orders transactions by fee rate, enforces block-size and cycle limits, and serializes the full template. This is the same work done by the mining `get_block_template` RPC — triggered here solely to count committing transactions.

The Terminal RPC module's own documentation acknowledges the cost and recommends using cached data for frequent calls:

> "For frequent monitoring calls, use cached data (refresh: null) to minimize system load. Force refresh only when real-time accuracy is critical." [4](#0-3) 

However, the API provides no server-side rate limit or guard against a caller that always sets the refresh flag.

---

### Impact Explanation

An unprivileged RPC caller (local or remote, depending on node configuration) can issue a tight loop of:

```
get_overview(refresh=4)   // TX_POOL_INFO = 0x4
```

Each call bypasses the 2-second TTL cache and triggers a full `get_block_template` execution. With a large mempool, this operation is CPU- and lock-intensive. Sustained calls can:

- Starve the tx-pool lock, delaying legitimate transaction submissions and block assembly for miners.
- Spike CPU usage on the node, degrading block propagation and sync performance.
- Cause the miner's own `get_block_template` RPC to queue behind repeated terminal-RPC-triggered assemblies.

The impact is **resource exhaustion / degraded node availability**, directly analogous to the gas waste in the MakerDAO finding but with broader consequences because the CKB node's tx-pool lock is shared across mining, relaying, and pool admission.

---

### Likelihood Explanation

- The `get_overview` RPC endpoint is publicly documented and reachable by any configured RPC caller.
- The `refresh` parameter is a simple integer; setting bit `0x4` requires no special knowledge.
- No authentication, rate limiting, or server-side guard prevents repeated forced refreshes.
- The Terminal RPC module is explicitly designed for monitoring dashboards, which may be called at high frequency by automated tools.

---

### Recommendation

1. **Avoid calling `get_block_template` inside `get_tx_pool_info`.** The "committing" count (transactions selected for the next block) can be obtained from a cheaper source — e.g., a dedicated atomic counter maintained by the block assembler, or by reading the block assembler's current cached template size without triggering a full rebuild.

2. **Apply server-side rate limiting** on forced-refresh calls to the Terminal RPC, independent of the client-supplied flag.

3. **Separate the "committing" metric** from the block-template assembly path. The block assembler already maintains a `current` template in memory; expose its transaction count directly rather than re-assembling.

---
