### Title
Unchecked External Callback Call in Legacy `revealWithCallback` Path Causes Permanent Request DoS - (File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol)

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains a legacy execution path (when `req.gasLimit10k == 0`) that makes a direct, unchecked external call to the requester's `_entropyCallback`. If that call reverts for any reason, the entire `revealWithCallback` transaction reverts, permanently blocking fulfillment of the request and locking the user's fee.

### Finding Description

`revealWithCallback` branches on whether `req.gasLimit10k != 0`:

- **New path** (lines 574–660): Uses `excessivelySafeCall`, which catches reverts and records a `CallbackFailed` event, leaving the request in a recoverable `CALLBACK_FAILED` state.
- **Legacy path** (lines 661–701): Clears the request first, then makes a **bare, unchecked** external call:

```solidity
// line 666 — request cleared before callback
clearRequest(provider, sequenceNumber);

// line 675–681 — direct call, no try/catch
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    );
}
``` [1](#0-0) 

Because `clearRequest` is called before the callback, if `_entropyCallback` reverts the entire transaction reverts (including `clearRequest`), leaving the request permanently active but unfulfillable.

The legacy path is triggered when `req.gasLimit10k == 0`, which happens when the provider's `defaultGasLimit` is `0`:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;  // opts out of new failure-state flow
}
``` [2](#0-1) 

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` confirms this revert behavior is reachable: [3](#0-2) 

### Impact Explanation

For any provider with `defaultGasLimit == 0`:

1. The request is permanently stuck — `revealWithCallback` always reverts.
2. The user's fee paid at request time is locked in the contract with no recovery path.
3. The provider's keeper wastes gas on every retry attempt, which always fails.

This is a direct analog to the reported Chainlink oracle DoS: an external call that can revert propagates the revert to the caller with no fallback, causing permanent denial of service for the affected request.

### Likelihood Explanation

Any unprivileged user can trigger this by:
1. Deploying a contract whose `_entropyCallback` always reverts (or conditionally reverts after receiving the random number).
2. Calling `requestWithCallback` against any provider with `defaultGasLimit == 0`.

The `requestWithCallback` function passes `gasLimit = 0`, which routes to the legacy path for any such provider: [4](#0-3) 

No privileged access, leaked keys, or external oracle manipulation is required.

### Recommendation

Wrap the legacy-path callback in a `try/catch` block, mirroring the new path's behavior:

```solidity
// Legacy path — replace bare call with guarded call
if (len != 0) {
    try IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    ) {
        // success — already emitted events below
    } catch {
        // emit a failure event; do NOT re-add the request
        emit CallbackFailed(provider, callAddress, sequenceNumber, ...);
    }
}
```

Alternatively, migrate all providers to set a non-zero `defaultGasLimit` so all requests use the safe `excessivelySafeCall` path, and deprecate the legacy branch.

### Proof of Concept

1. Deploy a malicious consumer:
```solidity
contract MaliciousConsumer is IEntropyConsumer {
    IEntropy entropy;
    constructor(address _entropy) { entropy = IEntropy(_entropy); }
    function request(address provider, bytes32 userRandom) external payable {
        entropy.requestWithCallback{value: msg.value}(provider, userRandom);
    }
    function _entropyCallback(uint64, address, bytes32) external override {
        revert("always fails");
    }
    function getEntropy() internal view override returns (address) { return address(entropy); }
}
```

2. Ensure the target provider has `defaultGasLimit == 0` (legacy mode).
3. Call `request(provider, userRandom)` with sufficient fee.
4. Provider's keeper calls `revealWithCallback` → always reverts.
5. The request is permanently stuck; the user's fee is locked in the Entropy contract. [5](#0-4)
