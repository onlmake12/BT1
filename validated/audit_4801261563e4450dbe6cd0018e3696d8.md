### Title
`DownloadScheduler::decrease` Uses Wrong Variable, Bypassing Timeout Accumulation Threshold — (`File: sync/src/types/mod.rs`)

### Summary

`DownloadScheduler::decrease` sets `self.timeout_count = self.task_count.saturating_add(num)` instead of `self.timeout_count = self.timeout_count.saturating_add(num)`. Because `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32` and `num` is 1 or 2, the expression always evaluates to ≥ 33, which is always `> 2`. The threshold guard is therefore always true, so `task_count` is decremented on **every** slow-response event instead of every third one, and `timeout_count` is immediately reset to 0 each time. The intended accumulation mechanism is completely bypassed.

### Finding Description

`DownloadScheduler` tracks how many concurrent block-download tasks a peer is allowed (`task_count`, initially 32, max 128). When a block arrives in the `NormalToUpper` or `UpperToMax` time quantile, `remove_by_block` calls `decrease(1)` or `decrease(2)` respectively. The design intent is to accumulate slow-response events in `timeout_count` and only penalize the peer (decrement `task_count`) after the count exceeds 2.

The actual code:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: should be self.timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

Because `task_count` is always ≥ 1 (peers with `task_count == 0` are already disconnected in `prune`), `task_count + num` is always ≥ 2 (and in practice ≥ 33 at initialization), so the `> 2` branch fires unconditionally. The correct code should read:

```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

This is structurally identical to the reported `min(X + Y, X)` class: a wrong variable is substituted in an arithmetic expression, making a conditional always evaluate the same way regardless of the intended accumulating state.

### Impact Explanation

When `task_count` reaches 0, `prune()` disconnects the peer:

```rust
if v.task_count == 0 {
    disconnect_list.insert(*k);
    false
}
```

With the bug, a peer is disconnected after **32 consecutive slow responses** (one decrement per event, starting from `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32`) instead of the intended **~96** (one decrement per 3 events). Peers that are legitimately slow due to transient network conditions are evicted 3× faster than the protocol intends. During IBD, this causes the node to shed sync peers prematurely, degrading block-download throughput and potentially stalling sync if few peers remain. An adversarial peer can also deliberately send slow responses to exhaust its own `task_count` and force a disconnect, but the primary impact is on honest peers under load.

### Likelihood Explanation

Any peer that responds to `GetBlocks` in the `NormalToUpper` or `UpperToMax` time band triggers `decrease`. This is a normal occurrence under real network conditions (variable latency, congested links, high-load peers). The code path is exercised continuously during sync on mainnet. No special attacker capability is required; the bug fires automatically for any slow peer.

### Recommendation

Change line 522 in `sync/src/types/mod.rs` from:

```rust
self.timeout_count = self.task_count.saturating_add(num);
```

to:

```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

This restores the intended accumulation: `task_count` is only decremented after `timeout_count` has accumulated past the threshold of 2.

### Proof of Concept

Given `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32`:

1. Peer P connects; `task_count = 32`, `timeout_count = 0`.
2. P returns a block in the `NormalToUpper` band → `remove_by_block` calls `decrease(1)`.
3. Inside `decrease`: `timeout_count = 32 + 1 = 33`; `33 > 2` → `task_count = 31`, `timeout_count = 0`.
4. Next slow response: `timeout_count = 31 + 1 = 32`; `32 > 2` → `task_count = 30`, `timeout_count = 0`.
5. After 32 such events: `task_count = 0` → peer disconnected in next `prune()` call.

With the correct code, step 3 would set `timeout_count = 0 + 1 = 1`; the guard `1 > 2` is false; `task_count` stays at 32. Only after 3 accumulated slow responses would `task_count` decrement once, requiring ~96 slow responses to reach 0. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

                                }
