### Title
OIS Slash Evasion via Undelegate Frontrun in Two-Step Slash Process - (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`, `governance/pyth_staking_sdk/src/types/integrity-pool.ts`)

### Summary

The Oracle Integrity Staking (OIS) slash mechanism uses a two-step process: a privileged `create_slash_event` instruction followed by a permissionless `slash` instruction that must be called separately per delegator. The window between these two steps allows a delegator to observe the `create_slash_event` transaction and call `undelegate` before `slash` is applied to their account, potentially escaping the penalty entirely.

### Finding Description

The OIS slash flow is:

1. `create_slash_event` — called by `rewardProgramAuthority`, creates a `SlashEvent` PDA containing `epoch`, `slash_ratio`, and `slash_custody`.
2. `slash` — permissionless, called per-delegator, takes `stake_account_positions` (writable) and `stake_account_custody` (writable) and transfers tokens to `slash_custody`. [1](#0-0) 

The `slash` instruction's `signer` account has no `relations` constraint tying it to the authority, confirming it is permissionless: [2](#0-1) 

The `DelegationRecord` tracks `nextSlashEventIndex` to enforce sequential slash application per delegator: [3](#0-2) 

Because `slash` must be called individually for each `stake_account_positions`, there is an unbounded window between `create_slash_event` and the moment `slash` is executed for any specific delegator. During this window, a delegator can call `undelegate` to move their positions from `LOCKED` to `PREUNLOCKING` (cooldown) state. [4](#0-3) 

The slashing rulebook explicitly states the investigation is public and concludes by end of `epoch_t+1` — a window of up to 14 days: [5](#0-4) 

Additionally, the `OutdatedDelegatorAccounting` error (code 6008) may require `advance_delegation_record` before `undelegate`, but if called before `create_slash_event` exists, no slash is applied — the delegator can then undelegate freely. [6](#0-5) 

### Impact Explanation

A delegator staked to a misbehaving publisher can escape the slash penalty by:
- Monitoring the public Pythian Council forum/on-chain mempool for `create_slash_event`
- Calling `undelegate` before `slash` is executed for their account

If the `slash` instruction only reduces LOCKED position amounts (not PREUNLOCKING/cooldown amounts), the delegator's slashable balance is zero or reduced. This directly undermines the security model of OIS: delegators bear no real risk for publisher misbehavior if they can exit before the slash lands. The pool's deterrence effect is broken.

**Note**: The exact exploitability depends on whether the Rust implementation of `slash` applies the penalty to cooldown positions. The Rust source for the integrity pool program was not available in the indexed codebase. The IDL architecture (writable `stake_account_positions` and `stake_account_custody` in `slash`) strongly suggests current-state application rather than retroactive epoch-based application.

### Likelihood Explanation

- The investigation timeline is intentionally public (forum posts, up to 14 days) — ample time to react.
- `undelegate` is a normal, permissionless user action with no delay before submission.
- The `slash` instruction must be called per-delegator, so no single atomic transaction covers all delegators.
- Any delegator with monitoring tooling (e.g., watching `create_slash_event` on-chain) can reliably execute this. [7](#0-6) 

### Recommendation

1. **Apply slashes retroactively based on epoch-snapshotted position data**, not current position state. The `SlashEvent.epoch` field already records the incident epoch — the `slash` instruction should compute the slashable amount from the position amounts that were LOCKED during that epoch, regardless of current state.
2. **Block `undelegate` when pending slash events exist** for the publisher. Check `nextSlashEventIndex < num_slash_events[publisher_index]` and revert if true.
3. **Alternatively**, apply all pending slashes atomically within `undelegate` before allowing the position state change, similar to how `advance_delegation_record` must be current before certain operations.

### Proof of Concept

1. Delegator D stakes 1,000,000 PYTH to publisher P in epoch_t.
2. Publisher P causes a price misprint in epoch_t.
3. Pythian Council announces investigation publicly (forum post).
4. In epoch_t+1, before `create_slash_event` is submitted, D calls `advance_delegation_record` to bring their record up to date (no slash event exists yet, so no slash is applied).
5. D calls `undelegate` — positions move to PREUNLOCKING (cooldown).
6. Authority calls `create_slash_event` with `slash_ratio = 500` (5%).
7. Anyone calls `slash` for D's `stake_account_positions` — but D's positions are now in cooldown state.
8. If `slash` only reduces LOCKED amounts, D's slashable amount is 0. D escapes the 50,000 PYTH slash.
9. D waits for cooldown to end and withdraws all 1,000,000 PYTH intact. [8](#0-7) [9](#0-8)
