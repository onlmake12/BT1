### Title
`FullSuspendedState::size()` Undercounts `inherited_fd` Cycle Cost by Factor of N — (`script/src/types.rs`)

### Summary

`FullSuspendedState::inherited_fd` is typed as `Vec<(VmId, Vec<Fd>)>` — a per-VM list of inherited file descriptors. The `size()` method at line 546 computes the cost as `self.inherited_fd.len() * size_of::<Fd>()`, which counts only the number of outer VM entries, completely ignoring the inner `Vec<Fd>` lengths. If each VM holds N inherited fds, the actual byte cost is N times larger than what `size()` reports.

### Finding Description

The field declaration: [1](#0-0) 

```rust
pub inherited_fd: Vec<(VmId, Vec<Fd>)>,
```

The size calculation: [2](#0-1) 

```rust
+ (self.inherited_fd.len() * (size_of::<Fd>())) as u64
```

This multiplies the number of outer `(VmId, Vec<Fd>)` entries by `size_of::<Fd>()`. It does **two things wrong**:
1. Ignores the `VmId` component of each tuple entirely.
2. Treats each entry as contributing exactly one `Fd` worth of bytes, regardless of how many `Fd` values are in the inner `Vec<Fd>`.

The correct computation would be:
```rust
self.inherited_fd.iter().fold(0, |acc, (_, fds)| {
    acc + size_of::<VmId>() + fds.len() * size_of::<Fd>()
}) as u64
```

The function's own documentation confirms its purpose is cycle metering: [3](#0-2) 

### Impact Explanation

If a script spawns multiple child VMs each receiving many inherited file descriptors, the `size()` value used to charge suspend/resume cycles will be proportionally smaller than the actual state size. With 16 VMs × 64 fds each, the charged size is `16 * 8 = 128` bytes instead of `16 * 64 * 8 = 8192` bytes — a 64× undercount. This violates the invariant that all computation must be fully cycle-metered.

### Likelihood Explanation

**Important caveat:** My search for call sites of `.size()` in `scheduler.rs` and `verify.rs` returned no matches. The function exists and is documented for cycle charging, but I was unable to confirm it is actually invoked in the production suspend/resume cycle-charging path within the available search iterations. If `size()` is not called in production code, the bug exists but has no runtime security impact.

### Recommendation

Fix line 546 to iterate over the inner `Vec<Fd>` lengths:

```rust
+ self.inherited_fd.iter().fold(0usize, |acc, (_, fds)| {
    acc + size_of::<VmId>() + fds.len() * size_of::<Fd>()
}) as u64
```

### Proof of Concept

The structural bug is directly readable from the source:

- Type: `Vec<(VmId, Vec<Fd>)>` — each entry holds a `VmId` and a variable-length list of `Fd`s. [1](#0-0) 
- Charge: `self.inherited_fd.len() * size_of::<Fd>()` — counts outer entries only, not inner fd counts. [2](#0-1) 

The discrepancy is unambiguous. Whether it constitutes an exploitable vulnerability depends on whether `size()` is actually invoked in the cycle-charging path — which I could not confirm from the available search results.

---

**Verdict:** The bug in `size()` is **real and structurally confirmed**. The full exploit path (cycle undercharge during suspend/resume) is plausible given the function's documented purpose, but the call site in the production charging path was not located in this analysis. A targeted audit of `scheduler.rs` and `verify.rs` for calls to `FullSuspendedState::size()` is needed to confirm exploitability.
