### Title
Entropy Provider Fee Takes Effect Immediately, Enabling Provider Front-Running DoS on User Requests — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Entropy contract allows any registered provider to change their `feeInWei` at any time via `setProviderFee` or `setProviderFeeAsFeeManager` with no delay or timelock. Because `requestHelper` re-reads the live fee at execution time, a provider can front-run a pending user `requestV2` transaction by spiking their fee, causing the user's transaction to revert with `InsufficientFee`. The provider then resets the fee to normal. This is a targeted, repeatable DoS against any user of that provider.

### Finding Description

`setProviderFee` writes the new fee directly to storage and returns immediately:

```solidity
// Entropy.sol L810-827
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;   // immediate, no delay
    emit ProviderFeeUpdated(...);
}
``` [1](#0-0) 

`setProviderFeeAsFeeManager` has the same immediate-effect pattern: [2](#0-1) 

At request time, `requestHelper` re-reads the live fee and reverts if `msg.value` is insufficient:

```solidity
// Entropy.sol L233-235
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [3](#0-2) 

`getFeeV2` reads `provider.feeInWei` live from storage: [4](#0-3) 

The same pattern exists in `EntropyGovernance.setPythFee` for the Pyth protocol fee component: [5](#0-4) 

The official SDK documentation and examples explicitly instruct users to call `getFeeV2()` before each `requestV2()` call, acknowledging that fees change over time: [6](#0-5) 

This creates a window between the user's fee query (off-chain or in a prior block) and the submission of `requestV2`, which a provider can exploit.

### Impact Explanation

A malicious provider can:
1. Monitor the mempool for pending `requestV2` transactions targeting their provider address.
2. Front-run with `setProviderFee(type(uint128).max)` (or any value above the user's `msg.value`).
3. The user's transaction executes after the fee spike and reverts with `InsufficientFee`.
4. The provider resets the fee to normal.

The user loses gas on every failed attempt. Consumer contracts that do not handle reverts gracefully may enter broken states. The attack is repeatable and can permanently deny service to targeted users or contracts. No user funds are lost (the revert returns ETH), but the service is rendered unusable for the targeted party.

### Likelihood Explanation

Any permissionlessly registered Entropy provider can execute this attack. Registration is open to anyone. The attack requires only two transactions (fee spike + reset) and is cheap relative to the gas cost imposed on the victim. On chains with a public mempool (all EVM chains where Entropy is deployed), pending `requestV2` transactions are visible before inclusion. The `IEntropyV2` interface itself warns that "the fee can change over time," confirming this is a known design gap without a mitigation.

### Recommendation

Implement a two-step fee update with a mandatory delay (e.g., 24–48 hours):
1. `proposeProviderFee(uint128 newFee)` — records the proposed fee and a `block.timestamp + delay` activation time.
2. `applyProviderFee()` — callable only after the delay has elapsed, commits the new fee.

This gives users advance notice and eliminates the front-running window. The same pattern should be applied to `setPythFee` in `EntropyGovernance`.

### Proof of Concept

```
Block N:
  User reads getFeeV2(provider, 0) = 1000 wei off-chain.
  User submits: requestV2{value: 1000}(provider, userRandom, 0)  [pending in mempool]

Block N (same block, earlier tx index):
  Provider calls: setProviderFee(type(uint128).max)
  → provider.feeInWei = 2^128 - 1

Block N (later tx index):
  User's requestV2 executes:
    requiredFee = getFeeV2(provider, 0) = (2^128 - 1) + pythFeeInWei  [overflows or is huge]
    msg.value (1000) < requiredFee → revert InsufficientFee

Block N+1:
  Provider calls: setProviderFee(1000)  [reset to normal]
```

The provider pays ~2× gas for two fee-update transactions; the user pays gas for a failed request. The cycle repeats for every user request. [7](#0-6) [3](#0-2)

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }
    }

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;
        providerInfo.feeInWei = newFeeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            provider,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
