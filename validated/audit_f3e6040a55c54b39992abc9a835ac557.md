### Title
`MAX_DEPOSIT_LIMIT` Not Enforced on Cumulative Balance for Permanent Subscriptions — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

`Scheduler.sol` enforces `MAX_DEPOSIT_LIMIT` (100 ether) only against the per-transaction `msg.value` in both `createSubscription` and `addFunds`, never against the cumulative `status.balanceInWei`. Because permanent subscriptions prohibit withdrawals, any ETH deposited is irrecoverably locked. An unprivileged caller can bypass the intended cap by calling `addFunds` repeatedly with amounts ≤ 100 ether, permanently locking an unbounded amount of ETH in the contract.

### Finding Description

`SchedulerConstants.sol` declares:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
``` [1](#0-0) 

Both deposit entry points check only `msg.value`:

```solidity
// createSubscription
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [2](#0-1) 

```solidity
// addFunds
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [3](#0-2) 

Neither check compares `status.balanceInWei + msg.value` against `MAX_DEPOSIT_LIMIT`. Withdrawals from permanent subscriptions are unconditionally blocked:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [4](#0-3) 

`addFunds` has no access control — any address may call it for any active subscription: [5](#0-4) 

### Impact Explanation

ETH deposited into a permanent subscription is irrecoverable. The `MAX_DEPOSIT_LIMIT` constant is publicly exposed and named as a hard cap, but it is trivially bypassed by splitting deposits across multiple `addFunds` calls (each ≤ 100 ether). A subscription manager — or any third party, since `addFunds` is permissionless — can accumulate an unbounded balance in a permanent subscription, permanently locking ETH far beyond the intended 100 ether ceiling. This is a direct loss-of-funds scenario for any caller who deposits under the mistaken belief that the limit is a total cap.

### Likelihood Explanation

The entry path is fully unprivileged: `addFunds(subscriptionId)` is callable by any EOA or contract with no role requirement. The bypass requires only repeated calls with `msg.value ≤ MAX_DEPOSIT_LIMIT`. Any subscription manager who wants to pre-fund a permanent subscription beyond 100 ether will naturally discover this path, and any third-party griever can force-lock ETH into someone else's permanent subscription at their own cost.

### Recommendation

Replace the per-transaction check with a cumulative check against the post-deposit balance:

```solidity
if (params.isPermanent) {
    uint256 newBalance = status.balanceInWei + msg.value;
    if (newBalance > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }
}
status.balanceInWei += msg.value;
```

Apply the same fix in `createSubscription` (where `status.balanceInWei` is initialized to `msg.value`, so the existing check is sufficient there, but consistency is recommended).

### Proof of Concept

1. Deploy `Scheduler` (or use the existing proxy).
2. Create a permanent subscription with `msg.value = MAX_DEPOSIT_LIMIT` (100 ether). Passes the check.
3. Call `addFunds{value: 100 ether}(subscriptionId)` — passes because `msg.value == MAX_DEPOSIT_LIMIT`.
4. Repeat step 3 N times. `status.balanceInWei` grows to `(N+1) × 100 ether`.
5. Attempt `withdrawFunds` — reverts with `CannotUpdatePermanentSubscription`.
6. All deposited ETH is permanently locked, far exceeding the advertised 100 ether limit.

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
