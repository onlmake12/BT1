### Title
Excess ETH sent to `requestPriceUpdatesWithCallback` is silently transferred to the provider instead of being refunded — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` only enforces `msg.value >= requiredFee` but never checks `msg.value == requiredFee`. Any ETH sent above `pythFeeInWei` is stored verbatim in `req.fee` and later paid out to the fulfilling provider via `executeCallback`. A user who accidentally overpays permanently loses the excess to the provider with no refund path.

---

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback` computes the required fee and validates only a lower bound:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [1](#0-0) 

The stored provider fee is then set to the full `msg.value` minus only the fixed Pyth protocol fee:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

Only `pythFeeInWei` is credited to the Pyth fee pool:

```solidity
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [3](#0-2) 

When `executeCallback` is later called, the provider is paid the full stored `req.fee`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [4](#0-3) 

If a user sends `msg.value = requiredFee + X` (overpays by `X`):

- `req.fee = (requiredFee + X) − pythFeeInWei = providerFee + X`
- Provider ultimately receives `providerFee + X + msg.value_callback − pythFee_for_updateData`
- The user's excess `X` is permanently transferred to the provider and is unrecoverable.

The `IEcho` interface NatSpec explicitly states the invariant that is never enforced:

> "The msg.value must be equal to getFee(callbackGasLimit)" [5](#0-4) 

---

### Impact Explanation

**Impact: High.** Any ETH overpaid by the user is permanently lost to the provider. There is no refund mechanism, no excess-ETH tracking, and no way for the user to recover the funds after the transaction. The `withdrawFunds` function is only available to the admin for Pyth-accrued fees, not to users. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: Low.** Exploitation requires a user to send more ETH than `getFee()` returns — a mistake in inputted data (e.g., a frontend bug, manual transaction construction error, or stale fee estimate). This mirrors the exact likelihood characterization in the reference RouterV2 report.

---

### Recommendation

Enforce exact equality between `msg.value` and the required fee, or refund the excess:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
// Add:
if (msg.value > requiredFee) {
    (bool ok, ) = msg.sender.call{value: msg.value - requiredFee}("");
    require(ok, "Refund failed");
}
```

Alternatively, enforce strict equality: `if (msg.value != requiredFee) revert InvalidFee();`

---

### Proof of Concept

1. Provider registers with `baseFeeInWei = 0.001 ether`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`.
2. `getFee(provider, callbackGasLimit, priceIds)` returns `pythFeeInWei + 0.001 ether`.
3. User calls `requestPriceUpdatesWithCallback{value: pythFeeInWei + 0.001 ether + 1 ether}(...)` — overpays by 1 ETH (e.g., due to a frontend bug).
4. `req.fee = (pythFeeInWei + 0.001 ether + 1 ether) − pythFeeInWei = 0.001 ether + 1 ether`.
5. `_state.accruedFeesInWei += pythFeeInWei` — only the Pyth fee is correctly accounted.
6. Provider calls `executeCallback(...)`.
7. Provider's `accruedFeesInWei` increases by `req.fee + msg.value_callback − pythFee_for_updateData`, which includes the user's extra 1 ETH.
8. User has permanently lost 1 ETH to the provider with no recourse. [7](#0-6)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
