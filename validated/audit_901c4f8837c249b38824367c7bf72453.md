### Title
Peer Ban Bypass via `ARGV_TOO_LONG_TEXT` in `should_ban()` Allows Unbounded Invalid Block Relay — (`sync/src/status.rs`)

---

### Summary

`should_ban()` in `sync/src/status.rs` contains an explicit string-match bypass: if the `Status` context contains `ARGV_TOO_LONG_TEXT`, it returns `None` even for 4xx (`BlockIsInvalid = 401`) status codes. An unprivileged peer can craft blocks whose lock/type scripts invoke the `exec` syscall with oversized argv, causing `VMError::Unexpected(ARGV_TOO_LONG_TEXT)` to propagate through the error chain into the `Status` context string, permanently suppressing the ban for that peer regardless of how many invalid blocks it relays.

---

### Finding Description

**Step 1 — Exec syscall emits the sentinel string**

In `script/src/syscalls/exec.rs` lines 168–170, when accumulated argv length exceeds `MAX_ARGV_LENGTH` (1 MiB), the syscall returns:

```rust
return Err(VMError::Unexpected(ARGV_TOO_LONG_TEXT.to_string()));
``` [1](#0-0) 

`ARGV_TOO_LONG_TEXT` is the sentinel `"@@@VM@@@UNEXPECTED@@@ARGV@@@TOOLONG@@@"`. [2](#0-1) 

**Step 2 — Error text is preserved through the full error chain**

`VMError::Unexpected` is not `ScriptError::Interrupts`, so `From<TransactionScriptError> for Error` routes it to `ErrorKind::Script.because(error)`, keeping the sentinel in the `Display` output. A dedicated test in the codebase explicitly asserts this:

```rust
assert!(format!("{}", error).contains(ARGV_TOO_LONG_TEXT));
``` [3](#0-2) [4](#0-3) 

**Step 3 — Block verification failure embeds the error string into Status context**

In `block_process.rs`, when block verification fails, `post_sync_process` is called with:

```rust
StatusCode::BlockIsInvalid.with_context(format!(
    "block {} is invalid, reason: {}",
    block_hash, err
))
```

The `err` Display includes `ARGV_TOO_LONG_TEXT`, so the `Status` context string contains the sentinel. [5](#0-4) 

The same pattern exists in `relayer/mod.rs` for the compact block relay path: [6](#0-5) 

**Step 4 — `should_ban()` explicitly suppresses the ban on sentinel match**

```rust
pub fn should_ban(&self) -> Option<Duration> {
    if !(400..500).contains(&(self.code as u16)) {
        return None;
    }
    if let Some(context) = &self.context {
        // TODO: it might be worthwhile to formalize all error texts
        // that won't be banned.
        if context.contains(ARGV_TOO_LONG_TEXT) {
            return None;   // ← ban suppressed for 401 BlockIsInvalid
        }
    }
    ...
}
``` [7](#0-6) 

**Step 5 — `post_sync_process` never bans the peer**

```rust
pub(crate) fn post_sync_process(..., status: Status) {
    if let Some(ban_time) = status.should_ban() {
        nc.ban_peer(peer, ban_time, status.to_string());
    } ...
}
```

With `should_ban()` returning `None`, `nc.ban_peer` is never called. [8](#0-7) 

---

### Impact Explanation

An unprivileged peer can repeatedly relay distinct invalid blocks (each with a unique hash, each containing a script that triggers the oversized-argv path) and will never be banned. The node continues to accept, verify, and reject each such block, consuming CPU (VM execution up to the argv check), memory, and I/O for each relay. The block is correctly rejected and marked `BLOCK_INVALID`, so **no consensus deviation occurs** — the actual impact is a **sustained, unthrottled DoS/resource-exhaustion** against the receiving node from a single peer. The claimed scope of "consensus deviation" is not met; the real severity is unbounded peer-level spam without penalty.

---

### Likelihood Explanation

The bypass is already present in production code with an explicit `TODO` comment acknowledging it is a workaround. Crafting a block with a lock script that calls exec with >1 MiB of argv is trivial. The sentinel string is a public constant. No privileged access, hashpower, or key material is required.

---

### Recommendation

1. Remove the string-match bypass from `should_ban()`. If the intent is to avoid banning peers for non-deterministic VM errors, use a typed error variant rather than a sentinel string in the context.
2. Introduce a typed `ScriptError::ArgvTooLong` (or similar) variant so the ban decision can be made on the error type, not on a fragile substring match.
3. Add a rate-limit or per-peer invalid-block counter that triggers a ban even when `should_ban()` returns `None`, to prevent unbounded relay abuse.

---
