### Title
Minimum Fee Rate Check Uses Serialized Size Instead of Transaction Weight, Allowing High-Cycles Transactions to Bypass the Minimum Fee Rate - (`tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the transaction's serialized byte size. However, the actual effective fee rate of a transaction — used for pool prioritization and eviction — is computed using `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. A transaction with high cycles but small serialized size passes the admission check with a fee that satisfies the size-based minimum, yet its true weight-based fee rate can be far below the configured minimum. This is a direct analog to the Panoptic bug where a fee is computed using a parameter that is always zero in certain call paths, causing the fee to be effectively zero.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee using only `tx_size`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(reject);
}
``` [1](#0-0) 

However, the actual fee rate of a `TxEntry` — used for pool scoring, eviction, and fee estimation — is computed via `get_transaction_weight`:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [2](#0-1) 

`get_transaction_weight` is defined as:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` and `max_tx_verify_cycles = 70_000_000`, the maximum cycles-equivalent weight is approximately **11,940 bytes**. A transaction with a small serialized size (e.g., ~200 bytes) but near-maximum cycles has:

- **Admission check** (`check_tx_fee`): `min_fee = 1000 × 200 / 1000 = 200 shannons` ✓ passes
- **Actual weight**: `max(200, 11940) = 11940` bytes
- **Effective fee rate**: `200 / 11940 × 1000 ≈ 16.7 shannons/KB` — roughly **60× below** the configured minimum of 1000 shannons/KB

The code comment in `check_tx_fee` acknowledges this discrepancy: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check"* — but no secondary weight-based check exists in the admission path. [4](#0-3) 

The eviction key also uses weight-based fee rate, meaning these transactions are evicted first — but they are still **admitted** to the pool, consuming pool resources and bypassing the spam-protection intent of `min_fee_rate`. [5](#0-4) 

---

### Impact Explanation

Any unprivileged RPC caller or transaction relayer can submit transactions whose effective fee rate is far below the configured `min_fee_rate`. The tx pool's spam-protection mechanism is bypassed for high-cycles transactions. An attacker can flood the tx pool with transactions that each pay only the size-based minimum fee (e.g., 200 shannons) while consuming weight-equivalent resources 60× larger. This degrades pool quality, displaces legitimate transactions, and undermines the fee market signal used by fee estimators. [6](#0-5) 

---

### Likelihood Explanation

Any transaction sender can craft a transaction with a script that consumes many cycles (e.g., a loop-heavy lock or type script) while keeping the serialized transaction small. This requires no special privilege — only the ability to submit a transaction via the `send_transaction` RPC or the P2P relay protocol. The `max_tx_verify_cycles` limit (70M) is the only bound on cycles, and it is reachable by design. [7](#0-6) 

---

### Recommendation

Replace the size-only minimum fee check in `check_tx_fee` with a weight-based check that mirrors how `TxEntry::fee_rate()` computes the effective fee rate:

```rust
// Instead of:
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);

// Use:
// cycles must be known at this point (from cache_entry or verification result)
let weight = get_transaction_weight(tx_size, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

This requires `cycles` to be available at the point of the fee check. If cycles are not yet known (pre-verification), the check should be deferred to after script verification, or a conservative upper-bound estimate should be used. [6](#0-5) 

---

### Proof of Concept

1. Craft a CKB transaction with a lock script that executes a tight computation loop consuming ~70,000,000 cycles. Keep the serialized transaction size small (~200 bytes).
2. Set the transaction fee to exactly `min_fee_rate × tx_size / 1000 = 1000 × 200 / 1000 = 200 shannons` (inputs capacity − outputs capacity = 200 shannons).
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 200 shannons`, fee = 200 shannons ≥ min_fee → **admitted**.
5. `TxEntry::fee_rate()` computes weight = max(200, 11940) = 11940, effective fee rate ≈ 16.7 shannons/KB — **60× below the minimum**.
6. Repeat to fill the tx pool with low-effective-fee-rate transactions, displacing legitimate higher-fee-rate transactions and degrading pool quality. [8](#0-7) [3](#0-2)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
