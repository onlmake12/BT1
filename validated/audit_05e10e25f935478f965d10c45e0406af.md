### Title
Hardcoded `DEFAULT_BYTES_PER_CYCLES` Constant Causes Incorrect Transaction Weight and Fee Rate Calculation When Consensus Parameters Differ — (`File: util/types/src/core/tx_pool.rs`, `util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

`get_transaction_weight` uses a hardcoded compile-time constant `DEFAULT_BYTES_PER_CYCLES` derived from the *default* `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES` ratio, rather than dynamically reading the actual runtime consensus values. This is the direct CKB analog of the ERC20 decimal-places assumption: just as the reported function hardcodes `1e10` (18 decimal places) instead of calling `decimals()`, CKB hardcodes the bytes-per-cycle ratio instead of reading it from the live `Consensus` object. When the actual consensus `max_block_bytes` or `max_block_cycles` differ from the defaults, every transaction weight, fee-rate ordering, eviction decision, and fee estimate produced by the tx-pool and fee estimator is wrong.

---

### Finding Description

In `util/types/src/core/tx_pool.rs`, the constant and function are defined as:

```rust
// Equal to MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES, see ckb-chain-spec.
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [1](#0-0) 

The constant `0.000_170_571_4` is computed from the *default* consensus constants:

- `MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT = 597 × 1000 = 597,000`
- `MAX_BLOCK_CYCLES = TWO_IN_TWO_OUT_CYCLES × TWO_IN_TWO_OUT_COUNT = 3,500,000 × 1000 = 3,500,000,000` [2](#0-1) 

However, the `Consensus` struct exposes `max_block_bytes` and `max_block_cycles` as independently configurable runtime fields, settable via `ConsensusBuilder`: [3](#0-2) [4](#0-3) 

`get_transaction_weight` is never passed the live `Consensus` object and always uses the baked-in ratio. The same pattern appears in the `weight_units_flow` fee estimator, which imports and uses the compile-time constant `MAX_BLOCK_BYTES` directly to compute how much weight is removed per block:

```rust
use ckb_chain_spec::consensus::MAX_BLOCK_BYTES;
// ...
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
``` [5](#0-4) [6](#0-5) 

Every consumer of `get_transaction_weight` inherits the stale ratio:

- `TxEntry::fee_rate()` — fee rate displayed and used for min-fee-rate admission check [7](#0-6) 
- `AncestorsScoreSortKey` — tx ordering in the pool [8](#0-7) 
- `EvictKey` — eviction decisions [9](#0-8) 
- `FeeRateCollector::statistics` — RPC `get_fee_rate_statistics` [10](#0-9) 
- `confirmation_fraction` fee estimator [11](#0-10) 

---

### Impact Explanation

If the runtime `max_block_cycles` is increased (e.g., via a hardfork or custom chain spec) while `DEFAULT_BYTES_PER_CYCLES` stays at the old value, the bytes-per-cycle ratio is *too high*. Cycle-heavy transactions are assigned inflated weights, making them appear more expensive than they are. This causes:

1. **Incorrect tx-pool ordering**: cycle-heavy transactions are ranked lower than their true fee rate warrants, starving them of block inclusion.
2. **Incorrect eviction**: legitimate high-fee cycle-heavy transactions are evicted in favour of cheaper byte-heavy ones.
3. **Incorrect fee estimates**: `weight_units_flow` uses the stale `MAX_BLOCK_BYTES` to compute `removed_weight`, so it systematically over- or under-estimates how quickly the mempool drains, producing wrong fee recommendations to RPC callers.

The inverse holds if `max_block_cycles` is decreased or `max_block_bytes` is increased.

---

### Likelihood Explanation

On the current mainnet the default constants happen to match the live consensus values, so the discrepancy is zero today. The bug activates the moment any hardfork or chain-spec change alters `max_block_bytes` or `max_block_cycles` — both of which are explicitly configurable fields in `Consensus` and `ConsensusBuilder`. CKB has already undergone hardforks (CKB2021, CKB2023) that adjusted VM and block parameters; a future hardfork adjusting block limits would silently corrupt all weight-based calculations without any code change being required. On custom devnets and testnets with non-default parameters the bug is already active. An unprivileged tx-pool submitter or RPC caller who is aware of the discrepancy can exploit the misordering to gain unfair priority or to cause others' transactions to be incorrectly evicted.

---

### Recommendation

Pass the live `Consensus` (or its `max_block_bytes` / `max_block_cycles` fields) into `get_transaction_weight` and the fee estimators instead of using the compile-time constant. For example:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64, bytes_per_cycle: f64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * bytes_per_cycle) as u64,
    )
}
```

where `bytes_per_cycle` is computed at call sites as `consensus.max_block_bytes() as f64 / consensus.max_block_cycles() as f64`. Similarly, replace the static `MAX_BLOCK_BYTES` import in `weight_units_flow.rs` with the runtime consensus value threaded through the estimator's API.

---

### Proof of Concept

1. Start a CKB node with a chain spec that sets `max_block_cycles` to `7_000_000_000` (double the default) while leaving `max_block_bytes` at `597_000`.
2. The correct bytes-per-cycle ratio is now `597_000 / 7_000_000_000 ≈ 0.0000853`, but `DEFAULT_BYTES_PER_CYCLES` remains `0.000_170_571_4` — exactly double the correct value.
3. Submit two transactions with equal byte size and equal fee: Tx A with 0 cycles (pure byte weight), Tx B with cycles equal to its byte size divided by the correct ratio (so both should have identical weight and fee rate).
4. Observe that Tx B is assigned a weight ~2× higher than Tx A due to the inflated constant, giving it an apparent fee rate ~2× lower, causing it to be deprioritized or evicted despite having the same true economic cost.
5. Confirm via `get_transaction_weight` return values and `TxEntry::fee_rate()` that the discrepancy matches the ratio `DEFAULT_BYTES_PER_CYCLES / actual_bytes_per_cycle`.
