Audit Report

## Title
Tx-Pool Admission Uses Size-Only Fee Check, Allowing Sub-`min_fee_rate` Transactions Into the Pool — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces `min_fee_rate` using only serialized transaction size, while the canonical weight used everywhere else is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For high-cycle transactions, cycles dominate the true weight by up to ~60×. A transaction can pass the size-only admission gate with a true fee rate ~62× below `min_fee_rate`, be admitted to the pool, and be relayed to all peers at negligible cost.

## Finding Description

`check_tx_fee` at `tx-pool/src/util.rs` L42–45 explicitly uses only `tx_size` for the fee gate, with a comment acknowledging the limitation:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The canonical weight function at `util/types/src/core/tx_pool.rs` L298–303 is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

The full pipeline in `_process_tx` (`tx-pool/src/process.rs` L705–753):

1. L715: `pre_check(&tx)` → calls `check_tx_fee(tx_size)` — size-only gate, cycles unknown
2. L724–732: `verify_rtx` — actual cycles determined here
3. L751: `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles
4. L753: `submit_entry` — admitted with no second fee-rate check

Confirmed by grep: there is no `min_fee_rate` reference anywhere in `process.rs`. `TxEntry::fee_rate()` at `tx-pool/src/component/entry.rs` L114–118 uses the true weight via `get_transaction_weight(self.size, self.cycles)` but is only consulted for eviction and sorting, never for admission gating.

**Exploit path:**
- `tx_size = 200` bytes, `cycles = 70,000,000` (default `max_tx_verify_cycles`)
- `fee = 201 shannons` (with default `min_fee_rate = 1000 shannons/KW`)
- Size-only check: `201 >= 1000 * 200 / 1000 = 200` ✓ — admitted
- True weight: `max(200, 70_000_000 × 0.000_170_571_4) = 11,940`
- True fee rate: `201 * 1000 / 11,940 ≈ 16 shannons/KW` — ~62× below `min_fee_rate`
- Transaction is admitted, relayed to all peers, and occupies pool space

## Impact Explanation

This matches the High impact class: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker can continuously submit high-cycle, tiny-fee transactions at ~201 shannons each. Each is relayed to all peers before eviction. The attacker can re-submit faster than `limit_size` eviction occurs, keeping the pool permanently polluted with unmineable entries and saturating relay bandwidth across the network at negligible cost per transaction.

## Likelihood Explanation

Any unprivileged user with RPC access to `send_transaction` can exploit this. The only prerequisite is deploying a script that consumes close to `max_tx_verify_cycles` (70,000,000), which requires only a standard on-chain deployment. No special keys, network position, or victim interaction is needed. The attack is repeatable and automatable.

## Recommendation

After `verify_rtx` returns the actual cycle count in `_process_tx`, perform a second fee-rate check using the true weight before calling `submit_entry`:

```rust
let true_weight = get_transaction_weight(tx_size, verified.cycles);
let true_min_fee = tx_pool_config.min_fee_rate.fee(true_weight);
if fee < true_min_fee {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, true_min_fee.as_u64(), fee.as_u64())), snapshot));
}
```

This mirrors how `TxEntry::fee_rate()` and `get_transaction_weight` already compute the canonical weight and closes the gap between the admission check and the true economic cost.

## Proof of Concept

1. Deploy a CKB script that loops to consume ~70,000,000 cycles.
2. Construct a transaction using that script as the lock, `tx_size ≈ 200` bytes.
3. Set fee to `201 shannons` (with default `min_fee_rate = 1000 shannons/KW`).
4. Submit via `send_transaction` RPC.
5. Observe: transaction accepted (size-only check: `201 >= 200` ✓).
6. Compute true fee rate: `201 * 1000 / max(200, 70_000_000 * 0.000_170_571_4) = 201_000 / 11_940 ≈ 16 shannons/KW`.
7. Observe: transaction is relayed to peers and occupies pool space at ~62× below `min_fee_rate`.
8. Repeat in a loop to continuously pollute the pool and saturate relay bandwidth.
