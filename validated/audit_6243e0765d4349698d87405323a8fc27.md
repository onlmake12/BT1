Audit Report

## Title
Off-by-One Epoch Gate Skips `DaoScriptSizeVerifier` for First Block of RFC0044 Activation Epoch — (`verification/contextual/src/contextual_block_verifier.rs`)

## Summary

`BlockTxsVerifier::verify` gates `DaoScriptSizeVerifier` using `self.parent.epoch().number()` instead of the current block's epoch number. Because `RFC0044_ACTIVE_EPOCH = 8651` and the gate uses `>=`, the first block of epoch 8651 (whose parent is in epoch 8650) evaluates `rfc0044_active(8650)` as `false`, causing `DaoScriptSizeVerifier` to be skipped for exactly that one block. A miner who wins the PoW race for that block can permanently commit a DAO phase-1 withdrawal with a mismatched lock script size to the canonical chain.

## Finding Description

`RFC0044_ACTIVE_EPOCH` is hardcoded to `8651`: [1](#0-0) 

`rfc0044_active` uses a `>=` comparison, so `rfc0044_active(8650)` returns `false`: [2](#0-1) 

`BlockTxsVerifier` holds both `self.header` (current block) and `self.parent` as distinct fields: [3](#0-2) 

The gate at line 445 passes `self.parent.epoch().number()` — the parent's epoch — not the current block's epoch: [4](#0-3) 

For the first block of epoch 8651: `self.parent.epoch().number()` = 8650, `rfc0044_active(8650)` = `8650 >= 8651` = `false`, so `DaoScriptSizeVerifier` is skipped. For every subsequent block in epoch 8651+, the parent is already in epoch 8651, so the verifier runs correctly. The bypass window is exactly one block.

The tx-pool path calls `DaoScriptSizeVerifier` unconditionally (no epoch gate), so a transaction submitted through the mempool would be rejected. However, a miner can construct a block directly, bypassing the tx-pool entirely: [5](#0-4) 

`DaoScriptSizeVerifier::verify` enforces that deposit and withdrawing cells use lock scripts of the same size — the check that is bypassed: [6](#0-5) 

All nodes replaying the chain use the same gate logic and would accept the block, permanently committing the malformed withdrawal.

## Impact Explanation

This is an incorrect implementation of system script (DAO) behavior. RFC0044 requires that a DAO phase-1 withdrawal cell's lock script size matches the deposit cell's lock script size. The one-block bypass allows a miner to violate this invariant permanently on the canonical chain. All full nodes accept the block because they all share the same off-by-one gate. This maps to **High (10001–15000 points): Incorrect implementation or behavior of CKB-VM or system scripts**. The claim that this could also qualify as Critical economic damage is not concretely proven — the `DaoScriptSizeVerifier` enforces lock script size parity, not capacity arithmetic; the DAO script itself still governs withdrawal amounts, so direct economic extraction from the size mismatch alone is not demonstrated.

## Likelihood Explanation

Exploiting this requires winning the PoW race for the first block of epoch 8651 — a standard miner capability, not a privileged role. The window is exactly one block, but any miner who mines that block can exploit it deterministically and locally without victim cooperation or external dependencies. The exploit is reproducible in a test environment by advancing a chain to the last block of epoch 8650.

## Recommendation

Replace `self.parent.epoch().number()` with the current block's epoch number in the gate condition:

```rust
// Use current block's epoch, not parent's
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
``` [7](#0-6) 

## Proof of Concept

1. Build a test chain advancing to the last block of epoch 8650 (parent is in epoch 8650, next block is the first block of epoch 8651).
2. Construct a DAO deposit cell committed at a block number ≥ `starting_block_limiting_dao_withdrawing_lock()`.
3. Construct a phase-1 withdrawal transaction where the output cell uses the DAO type script but has a lock script of a **different size** than the deposit cell's lock script (as demonstrated in the existing test `test_dao_disables_different_lock_script_size`). [8](#0-7) 
4. Mine the first block of epoch 8651 containing this transaction by constructing the block directly (bypassing the tx-pool).
5. Call `BlockTxsVerifier::verify` — it evaluates `rfc0044_active(8650)` = `false`, skips `DaoScriptSizeVerifier`, and returns `Ok`.
6. Assert the block is accepted by the chain and the malformed withdrawal is permanently committed.
7. Confirm that submitting the same transaction through the tx-pool is rejected (no epoch gate in `verify_rtx`).

**File:** verification/src/transaction_verifier.rs (L845-890)
```rust
    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
```
