### Title
Scheduler Subscription Price Feeds Stall During Network Congestion Due to Gas-Price-Dependent Keeper Fee Revert — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.sol` contract's `_processFeesAndPayKeeper` function calculates keeper reimbursement dynamically using `tx.gasprice`. During network congestion, gas prices spike, causing the required keeper fee to exceed a subscription's pre-funded balance. This causes `updatePriceFeeds` to revert with `InsufficientBalance`, leaving price feeds stale during exactly the periods when accurate prices are most critical for DeFi consumers.

---

### Finding Description

In `_processFeesAndPayKeeper`, the keeper fee is computed entirely at execution time:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;

if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [1](#0-0) 

The `gasCost` component is directly proportional to `tx.gasprice`, which is unbounded and volatile. Subscription managers fund their subscriptions based on expected gas prices, guided by `getMinimumBalance`. The `GAS_OVERHEAD` constant is fixed at 30,000 gas: [2](#0-1) 

During network congestion, gas prices can spike 10–100x. When this happens, `(startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice` can far exceed `status.balanceInWei`, causing the entire `updatePriceFeeds` call to revert. The price feed is not updated, and the keeper wastes gas on a failed transaction.

The `updatePriceFeeds` entry point is permissionless — any unprivileged keeper can call it: [3](#0-2) 

The revert happens at the very end of the function, after all validation and Pyth fee deduction have already occurred: [4](#0-3) 

This is structurally identical to the Umee oracle message prioritization issue: during congestion, the critical oracle update transaction cannot land, leaving the system operating on stale data.

---

### Impact Explanation

Scheduler subscriptions stop updating during network congestion. DeFi protocols that consume prices via `getPricesUnsafe` / `getEmaPricesUnsafe` on the Scheduler contract will read stale prices. An attacker can exploit this to:

- Avoid liquidation when their position is undercollateralized (stale collateral price too high)
- Trigger liquidations on healthy positions (stale debt price too low)
- Manipulate any protocol that uses Scheduler-pushed prices as a source of truth

The Scheduler README explicitly describes this as a push-style feed for protocols that "prefer to consume from a push-style feed rather than integrate the pull model": [5](#0-4) 

Protocols relying on this model have no fallback when the Scheduler stops updating.

---

### Likelihood Explanation

Network congestion and market volatility are strongly correlated — gas prices spike precisely when accurate prices are most critical (liquidation cascades, large price moves). Subscriptions funded at or near `getMinimumBalance` will fail to update during these events. The existing test `testUpdatePriceFeedsRevertsInsufficientBalanceForKeeperFee` in `PulseScheduler.t.sol` demonstrates that even funding at `mockPythFee + minKeeperFee` is insufficient at elevated gas prices: [6](#0-5) 

No attacker action is required to trigger the congestion — natural market events suffice. The attacker only needs to observe the stale price and exploit it.

---

### Recommendation

1. **Gas price cap in keeper fee calculation**: Bound `tx.gasprice` in `_processFeesAndPayKeeper` to a configurable maximum (e.g., admin-set `maxGasPriceForReimbursement`). Keepers can still submit at higher gas prices but absorb the excess themselves, ensuring the subscription balance is never the bottleneck.

2. **Deactivate rather than revert on insufficient balance**: Instead of reverting the entire `updatePriceFeeds` call when balance is insufficient for the keeper fee, emit an event and deactivate the subscription. This prevents keepers from wasting gas on doomed transactions and makes the failure mode explicit.

3. **Minimum balance buffer guidance**: Document that `getMinimumBalance` assumes a baseline gas price and that subscription managers should fund a multiple (e.g., 10×) of the minimum to handle gas price spikes.

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
