### Title
Publisher Stake Account Update Without Settling Self-Delegation State Causes Stale Accounting - (File: governance/pyth_staking_sdk/src/idl/integrity-pool.json)

### Summary
The `set_publisher_stake_account` instruction in the integrity pool program updates `pool_data.publisher_stake_accounts[publisher_index]` to a new stake account without requiring the publisher to first settle their self-delegation state via `advance_delegation_record`. If called while the publisher has active self-delegation positions in the old account, the `self_del_state` in `PoolData` becomes stale and incorrect, causing the publisher's existing positions to be reclassified as regular delegation rather than self-delegation, with downstream effects on reward accounting.

### Finding Description
The `PoolData` account tracks per-publisher self-delegation state in `self_del_state` (an array of `DelegationState` structs indexed by publisher) and the associated stake account in `publisher_stake_accounts`.

The `advance_delegation_record` instruction determines whether a delegation is self-delegation by comparing the caller-supplied `publisher_stake_account_positions` against `pool_data.publisher_stake_accounts[publisher_index]`. If they match, the delegation is treated as self-delegation and credited to `self_del_state`; otherwise it is treated as regular delegation and credited to `del_state`.

When `set_publisher_stake_account` is called to replace the publisher's stake account from account A to account B:
- `pool_data.publisher_stake_accounts[publisher_index]` is updated to B immediately.
- Any existing positions in account A that were previously self-delegation are **not settled** before the switch.
- All subsequent calls to `advance_delegation_record` for account A will now compare A against B, find a mismatch, and treat A's positions as regular delegation — permanently reclassifying them.
- The `self_del_state` for the publisher decreases incorrectly, while `del_state` increases.

The UI layer does enforce an "unstake first" requirement:

```typescript
// apps/staking/src/components/OracleIntegrityStaking/index.tsx
return hasAnyPositions(self) ? (
  <ModalDialog closeButtonText="Ok" title="You must unstake first" ...>
    You cannot designate another account while self-staked.
```

However, the on-chain IDL for `set_publisher_stake_account` imposes **no such constraint** — the instruction accepts `new_stake_account_positions_option` and `current_stake_account_positions_option` as optional accounts with no position-count validation:

```json
{
  "name": "set_publisher_stake_account",
  "accounts": [
    { "name": "signer", "signer": true },
    { "name": "publisher", ... },
    { "name": "pool_data", "writable": true, "relations": ["pool_config"] },
    { "name": "pool_config", ... },
    { "name": "new_stake_account_positions_option", "optional": true },
    { "name": "current_stake_account_positions_option", "optional": true }
  ],
  "args": []
}
```

A publisher can bypass the UI and call the instruction directly with active self-delegation positions still open in the old account.

### Impact Explanation
1. **Self-delegation accounting corruption**: `self_del_state[publisher_index].totalDelegation` drops to zero (or to whatever the new account holds), while the old account's positions are reclassified as regular delegation. This is a permanent accounting error until the old positions are closed.
2. **Reward miscalculation**: Self-delegation and regular delegation carry different reward ratios (`R^p_p` vs `R^d_p`). Reclassifying self-delegation as regular delegation changes the reward split between the publisher and their delegators for all future epochs.
3. **Pool cap distortion**: The pool cap formula `C_p = M * Σ 1/max(n_s, Z)` is unaffected, but the effective self-stake `S^p_p` used in `R^p_p = y * min(S^p_p, C_p)` is now understated, causing the publisher to receive fewer self-delegation rewards than they are entitled to.
4. **Delegator impact**: Because the delegation fee mechanism and reward split depend on the self-delegation ratio, delegators in the affected pool receive a different (potentially higher or lower) share of rewards than the protocol intends.

### Likelihood Explanation
The publisher controls their own keypair and can construct and submit the `set_publisher_stake_account` instruction directly without going through the staking dashboard UI. This requires no special privilege beyond being a registered publisher in `pool_data`. The scenario is realistic for any publisher who wants to migrate their self-staking account without waiting for the cooldown period to expire on existing positions.

### Recommendation
Add an on-chain guard in the `set_publisher_stake_account` handler that checks `self_del_state[publisher_index].total_delegation == 0` before allowing the stake account to be updated. Alternatively, require the publisher to call `advance_delegation_record` for the current stake account (settling all pending rewards) before the switch is permitted, mirroring the fix applied to the analogous `set_staker()` issue (which blocked the call when a staker was already set).

### Proof of Concept
1. Publisher P has stake account A with 1,000,000 PYTH self-delegated to their own pool. `pool_data.self_del_state[P_index].total_delegation = 1_000_000`.
2. Publisher P constructs `set_publisher_stake_account` directly (bypassing the UI), passing `current_stake_account_positions_option = A` and `new_stake_account_positions_option = B` (a new empty account).
3. On-chain: `pool_data.publisher_stake_accounts[P_index]` is updated to B. No position check is performed.
4. Next epoch: anyone calls `advance_delegation_record` for account A against publisher P. The instruction reads `pool_data.publisher_stake_accounts[P_index] = B`, compares with A, finds A ≠ B, and credits A's 1,000,000 PYTH to `del_state` (regular delegation) instead of `self_del_state`.
5. Result: `self_del_state[P_index].total_delegation` drops to 0. Publisher P loses all self-delegation reward entitlement for those positions.
