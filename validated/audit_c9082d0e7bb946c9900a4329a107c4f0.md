### Title
Lazer EVM `verifyUpdate` Lacks Message-Level Replay Protection and Expiry — (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

The `verifyUpdate` function in `PythLazer.sol` verifies that a Lazer price update was signed by a trusted signer, but performs **no replay protection** and **no message-level expiry check** on the signed payload. Any transaction sender or relayer can re-submit a previously valid signed price update at any future time, as long as the signer key has not yet expired. This is a direct structural analog to the reported vulnerability: nonce used as a salt (not a sequential counter) and no `expiredAt` argument in the signed inputs.

---

### Finding Description

`verifyUpdate` in `PythLazer.sol` performs the following checks:

1. EVM format magic bytes match
2. Signature recovers to a non-zero address
3. Recovered signer address passes `isValidSigner()` (i.e., `block.timestamp < trustedSignerToExpiresAtMapping[signer]`) [1](#0-0) 

The signed message is `keccak256(payload)`. The payload is returned raw to the caller. The contract performs **zero checks** on:

- A nonce or sequence number within the payload compared to any on-chain storage counter
- An expiry/deadline field within the payload compared to `block.timestamp` [2](#0-1) 

The signer-level `expiresAt` only controls whether the signing key is still authorized; it does **not** bound the validity window of any individual signed message. A signed update produced at time T remains replayable until the signer key itself expires, which can be months or years later. [3](#0-2) 

Contrast this with the Pyth governance path on EVM, which correctly enforces a monotonically increasing `lastExecutedGovernanceSequence`: [4](#0-3) 

And the Executor contract, which enforces the same ordering invariant: [5](#0-4) 

The Lazer contract has no equivalent protection.

---

### Impact Explanation

**Impact: High**

A malicious or front-running relayer who has retained a previously valid signed Lazer price update can re-submit it at any moment the stale price is financially advantageous. Because `verifyUpdate` is `external payable` and callable by any transaction sender, no privileged access is required.

Concrete harm to DeFi consumers:
- A relayer submits a stale price update showing a lower collateral value → triggers illegitimate liquidations
- A relayer submits a stale price update showing a higher asset price → allows over-borrowing against inflated collateral
- Two signed updates exist for the same feed; the relayer chooses which to submit based on which is more profitable (ordering attack)

The Lazer contract is explicitly designed for high-value, low-latency DeFi use cases, making price manipulation via replay directly financially exploitable.

---

### Likelihood Explanation

**Likelihood: Low**

Exploitation requires:
1. A relayer (or any party) who has retained a prior signed Lazer update
2. A consumer contract that does not independently validate the `publish_time` embedded in the returned payload
3. A price movement large enough to be profitable after gas costs

The first condition is trivially satisfied (any observer of the mempool or Lazer feed can archive updates). The second condition is a realistic consumer mistake, especially given that the Pyth contract itself does not enforce freshness. The third condition depends on market volatility.

---

### Recommendation

1. **Add a sequential global nonce to storage** (analogous to `lastExecutedGovernanceSequence`) and require the signed payload to include a nonce that equals the current storage nonce, incrementing it on each accepted call.

2. **Add an `expiredAt` field to the signed payload** and enforce `block.timestamp <= expiredAt` inside `verifyUpdate` before accepting the update. This bounds the replay window to the intended validity period of each signed message.

3. As a minimum mitigation, enforce that the `publish_time` embedded in the payload is strictly greater than the last accepted `publish_time` for each price feed, stored on-chain.

---

### Proof of Concept

```
1. At time T=0, Lazer signer signs payload P0 containing price=100 for feed F.
2. At time T=1, Lazer signer signs payload P1 containing price=80 for feed F.
3. Relayer submits P1 to the consumer contract; consumer records price=80.
4. At time T=2, actual price rises to 95.
5. Relayer re-submits P0 (price=100) by calling verifyUpdate(P0).
   - EVM magic: valid
   - keccak256(P0) signature: valid (signer key still active)
   - isValidSigner: passes
   - verifyUpdate returns (P0_payload, signer) — no revert
6. Consumer contract receives price=100 (stale), enabling over-borrowing or blocking legitimate liquidations.
```

The call `verifyUpdate(P0)` succeeds unconditionally at step 5 because `PythLazer.sol` contains no storage nonce check and no message-level expiry check. [1](#0-0)

        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,

            bytes32(update[4:36]),

        );

            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
