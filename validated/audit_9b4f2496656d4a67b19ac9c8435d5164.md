Audit Report

## Title
JSON-RPC Batch Deserialization Before Size Check with Unenforced Body Limit Enables Memory Exhaustion — (File: rpc/src/server.rs)

## Summary

`rpc/src/server.rs` fully deserializes the entire JSON-RPC payload via `serde_json::from_str::<Request>` before checking the batch element count. `JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is never initialized on default deployments because `rpc_batch_limit` is commented out in the shipped config. Separately, `RpcConfig::max_request_body_size` is declared and set to 10 MiB in the default config but is never passed into `start_server` and never applied to the axum router, so axum's own 2 MiB default is the only effective guard. An attacker with access to the RPC port can send many concurrent large batch payloads to amplify heap allocations and OOM-kill the node.

## Finding Description

**Root cause 1 — `max_request_body_size` never reaches the router.**

`start_server` accepts only `rpc`, `address`, `handler`, and `enable_websocket`; `RpcConfig` is not passed in: [1](#0-0) 

The router is built with no body-size layer — no `DefaultBodyLimit`, `RequestBodyLimitLayer`, or any `body_limit` variant appears anywhere in the repository (confirmed by exhaustive grep returning zero matches): [2](#0-1) 

The field is declared in the config struct: [3](#0-2) 

And set to 10 MiB in the shipped config: [4](#0-3) 

But it is never consumed in `start_server`. Axum 0.7's own 2 MiB default applies instead.

**Root cause 2 — Full deserialization before batch count check.**

`handle_jsonrpc` calls `serde_json::from_str::<Request>(req)` unconditionally, allocating the full `Vec<Call>` for every batch before any limit is consulted: [5](#0-4) 

Only after this allocation does the code branch on `Request::Batch` and consult `JSONRPC_BATCH_LIMIT`: [6](#0-5) 

**Root cause 3 — Batch limit is `None` in default deployments.**

`JSONRPC_BATCH_LIMIT` is initialized only when `config.rpc_batch_limit` is `Some`: [7](#0-6) 

In the shipped default config, `rpc_batch_limit` is commented out: [8](#0-7) 

So `JSONRPC_BATCH_LIMIT.get()` returns `None` on every default node, and the guard at line 275 is never evaluated.

**Exploit chain:**
1. Attacker sends POST bodies near axum's 2 MiB default limit, each containing a JSON array of hundreds of batch items.
2. `handle_jsonrpc` → `serde_json::from_str::<Request>` allocates the full `Vec<Call>` on the heap (6–8× wire-size amplification from JSON string/object overhead is realistic).
3. `JSONRPC_BATCH_LIMIT.get()` returns `None` (default config) → no rejection occurs.
4. Attacker opens many concurrent connections, each holding its allocation while the stream processes calls sequentially.
5. Heap grows proportionally to concurrent connections × amplified batch size; node is OOM-killed.

## Impact Explanation

An OOM kill terminates the entire CKB node process — not just the RPC handler — stopping block processing, transaction relay, and peer communication. This matches **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.** The two independent gaps (unenforced body limit, no default batch count limit) compound each other: even with axum's 2 MiB floor, many concurrent connections each amplifying to ~12–16 MiB can exhaust available memory on a typical node host.

## Likelihood Explanation

The RPC port binds to `127.0.0.1:8114` by default, restricting direct remote exploitation. However, any local process (scripts, CI runners, SSRF from a co-located service) qualifies. The optional TCP (`tcp_listen_address`) and WebSocket (`ws_listen_address`) endpoints may be bound to external interfaces, widening exposure. No authentication, chain knowledge, or special privilege is required — only the ability to POST to the RPC port. Both protections are disabled in the default shipped configuration, making the condition trivially reproducible on any default node.

## Recommendation

1. **Pass `RpcConfig` into `start_server`** and add `.layer(DefaultBodyLimit::max(config.max_request_body_size))` to the axum router so the operator-configured limit is enforced before the body reaches `handle_jsonrpc`.
2. **Check batch count before full deserialization.** Use a streaming or pre-scan approach (e.g., count top-level array elements in raw bytes) before calling `serde_json::from_str::<Request>`.
3. **Enable `rpc_batch_limit` by default.** Change the shipped config from the commented-out `# rpc_batch_limit = 2000` to an active conservative default (e.g., `rpc_batch_limit = 100`).
4. **Add per-connection concurrency limiting** on the HTTP handler to bound simultaneous large deserializations.

## Proof of Concept

```python
import socket, json, time

single = {"jsonrpc": "2.0", "method": "get_tip_block_number", "params": [], "id": 1}
# Fill toward axum's 2 MiB default with many batch items
batch = [single] * 500
payload = json.dumps(batch).encode()
print(f"Wire size: {len(payload)/1024:.1f} KB")

for _ in range(50):   # 50 concurrent connections
    s = socket.socket()
    s.connect(("127.0.0.1", 8114))
    http = (
        f"POST / HTTP/1.1\r\nHost: localhost\r\n"
        f"Content-Type: application/json\r\n"
        f"Content-Length: {len(payload)}\r\n\r\n"
    ).encode() + payload
    s.sendall(http)
    # Do not read response — keep connection open to hold allocation
    time.sleep(0.01)

print("Monitor node RSS for amplified heap growth.")
```

**Execution path:**
1. Each POST body is accepted (no enforced size limit beyond axum's 2 MiB floor, since `max_request_body_size` is never wired to the router).
2. `handle_jsonrpc` → `serde_json::from_str::<Request>` at line 238 allocates the full `Vec<Call>`.
3. `JSONRPC_BATCH_LIMIT.get()` returns `None` (default config) → no rejection.
4. 50 concurrent connections each hold their amplified allocation while processing sequentially.
5. Node RSS grows to `50 × ~12–16 MiB = ~600–800 MiB`; scaling connections further triggers OOM kill of the node process.
