### Title
Silent Callback Success for Non-Existent Requester in `revealWithCallback` New Gas-Limit Path — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`'s `revealWithCallback`, the new callback path (triggered when `req.gasLimit10k != 0`) invokes the requester's `_entropyCallback` via `excessivelySafeCall` **without first verifying that the requester address contains contract code**. Per EVM semantics, a low-level CALL to an EOA or a self-destructed/non-existent address always returns `(true, "")`. This causes the callback to be silently treated as successful: the request is cleared, `RevealedWithCallback` is emitted, and the user's fee is permanently consumed — even though the requester never received the random number and cannot retry.

---

### Finding Description

`revealWithCallback` contains two execution branches depending on `req.gasLimit10k`:

**New path (lines 574–660) — vulnerable:**

```solidity
(success, ret) = req.requester.excessivelySafeCall(
    uint256(req.gasLimit10k) * TEN_THOUSAND,
    256,
    abi.encodeWithSelector(
        IEntropyConsumer._entropyCallback.selector,
        sequenceNumber,
        provider,
        randomNumber
    )
);
```

`excessivelySafeCall` is a thin wrapper around the EVM `CALL` opcode. When `req.requester` is an EOA or a non-existent address, the EVM returns `(success=true, returndata="")`. The library propagates this unchanged. The contract then enters the `if (success)` branch:

```solidity
if (success) {
    emit RevealedWithCallback(...);
    clearRequest(provider, sequenceNumber);
}
```

The request is permanently deleted and `RevealedWithCallback` is emitted — indistinguishable from a genuine successful delivery.

**Legacy path (lines 661–702) — correctly guarded:**

```solidity
uint len;
assembly {
    len := extcodesize(callAddress)
}
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    );
}
```

The legacy path explicitly checks `extcodesize` before calling. The new path omits this check entirely, creating a regression relative to the existing defensive pattern already present in the same function. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

1. A user (EOA or a contract that later self-destructs) calls `requestV2(provider, userContribution, gasLimit)` with a non-zero `gasLimit`, paying the full fee.
2. The provider calls `revealWithCallback`.
3. `excessivelySafeCall` on the EOA/non-existent address returns `(true, "")`.
4. The contract clears the request and emits `RevealedWithCallback` as if delivery succeeded.
5. The user's random number is **never delivered**; the request **cannot be retried** (it has been cleared); the fee is **permanently consumed**.
6. Off-chain monitoring observing `RevealedWithCallback` has no way to distinguish this silent failure from a genuine successful callback.

The `CALLBACK_FAILED` state — which would allow recovery — is never set, so the user has no recourse path. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

- Any user who calls `requestV2` directly from an EOA wallet (e.g., via a script or front-end without a wrapper contract) with a non-zero `gasLimit` triggers this path.
- Any user whose requester contract self-destructs between request and fulfillment triggers this path.
- The `requestV2` entry points accept any `msg.sender` with no contract-existence check at request time, so the scenario is reachable by any unprivileged Entropy user. [6](#0-5) 

---

### Recommendation

Mirror the `extcodesize` guard already present in the legacy path. Before the `excessivelySafeCall` in the new path, add:

```solidity
uint len;
assembly { len := extcodesize(req.requester) }
if (len == 0) revert EntropyErrors.InvalidRequester(); // or set CALLBACK_FAILED
```

Alternatively, perform this check at request time in `requestHelper` so that EOA requesters are rejected upfront when a non-zero gas limit is specified. [3](#0-2) 

---

### Proof of Concept

1. Alice (EOA) calls `requestV2(provider, userContribution, 50000)` and pays the required fee. `req.requester = Alice`, `req.gasLimit10k = 5`.
2. Provider calls `revealWithCallback(provider, seqNum, userContribution, providerContribution)`.
3. `req.gasLimit10k != 0` → new path is taken.
4. `req.requester.excessivelySafeCall(500000, 256, calldata)` is executed against Alice's EOA.
5. EVM CALL to an EOA returns `(1, "")` → `success = true`, `ret = ""`.
6. `if (success)` branch: `RevealedWithCallback` is emitted, `clearRequest` deletes the request.
7. Alice's random number was never delivered. Alice cannot call `revealWithCallback` again (request cleared). Alice cannot call `reveal` (request was a callback-type). Alice's fee is gone.
