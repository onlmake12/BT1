### Title
Missing Zero-Price Validation in `deriveCrossRate` Causes Opaque Panic Revert Instead of Descriptive Error - (File: `target_chains/ethereum/sdk/solidity/PythUtils.sol`)

---

### Summary

`PythUtils.deriveCrossRate` validates that `price2` is non-negative but does not validate that `price2 != 0`. When `price2 == 0` is passed, the function forwards `0` as the denominator to `Math.mulDiv`, which triggers an opaque Solidity Panic (error code `0x12`) rather than a descriptive `PythErrors` custom error. This is the direct analog of the Bancor `mulDivF`/`mulDivC` finding: missing zero-denominator validation causing an opaque Panic instead of a meaningful revert.

---

### Finding Description

In `deriveCrossRate`, the only guard on `price2` is:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
```

This allows `price2 == 0` to pass through. The function then reaches one of two `Math.mulDiv` calls, both of which use `price2` (cast to `uint64`) as part of the denominator:

**Branch 1 (`deltaExponent > 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    10 ** uint64(deltaExponent),
    uint64(price2)          // denominator = 0 when price2 == 0
);
```

**Branch 2 (`deltaExponent <= 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    1,
    10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // denominator = 0 when price2 == 0
);
```

Inside `Math.mulDiv`, when `high == 0` (the common case for small inputs), the code executes `return low / denominator`, which in Solidity 0.8+ checked arithmetic reverts with Panic code `0x12` (division by zero). When `high != 0`, the explicit `panic(DIVISION_BY_ZERO)` path fires. Either way, the caller receives an opaque Panic rather than a named `PythErrors` error. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Any on-chain contract that calls `deriveCrossRate` with a zero `price2` — which can occur legitimately when a Pyth price feed returns `price == 0` for a newly initialized or temporarily zeroed feed — will receive an opaque Panic revert. The caller cannot distinguish this from other Panic conditions (overflow, out-of-bounds, etc.) without decoding the raw Panic code. This breaks integrators' error-handling logic, makes debugging difficult, and can cause silent failures in contracts that catch generic reverts.

The `PythStructs.Price.price` field is `int64` and Pyth does not guarantee it is always non-zero. The official documentation example passes `eurUsd.price` directly to `deriveCrossRate` without a zero-check, meaning any integrator following the documented pattern is exposed. [3](#0-2) 

---

### Likelihood Explanation

Pyth price feeds can return `price == 0` in edge cases (e.g., feed initialization, extreme market conditions, or a feed that has not yet received a valid update). The official `deriveCrossRate` documentation and example code do not warn callers to check for zero `price2` before calling the function. Any integrator following the documented pattern is directly exposed. The entry path is fully unprivileged: any transaction sender can trigger this by submitting a price update that results in a zero price for the denominator feed. [4](#0-3) 

---

### Recommendation

Add an explicit zero-price check for `price2` before the `mulDiv` calls, using a descriptive custom error:

```solidity
if (price2 == 0) {
    revert PythErrors.ZeroDenominatorPrice();
}
```

This should be placed immediately after the existing negativity check (line 92–94), before the `deltaExponent` computation. A corresponding `ZeroDenominatorPrice` error should be added to `PythErrors`.

        int64 price1,
        int32 expo1,
        int64 price2,
        int32 expo2,
        int32 targetExponent
    ) public pure returns (uint256 crossRate) {
        // Check if the input prices are negative
        if (price1 < 0 || price2 < 0) {
            revert PythErrors.NegativeInputPrice();
        }
        // Check if the input exponents are valid and not less than -255
        if (expo1 < -255 || expo2 < -255 || targetExponent < -255) {
            revert PythErrors.InvalidInputExpo();
        }

        // note: This value can be negative.
        int64 deltaExponent = int64(expo1 - (expo2 + targetExponent));

        // Bounds check: prevent overflow/underflow with base 10 exponentiation
        // Calculation: 10 ** n <= (2 ** 256 - 63) - 1
        //              n <= log10((2 ** 193) - 1)
        //              n <= 58.2
        if (deltaExponent > 58 || deltaExponent < -58)
            revert PythErrors.ExponentOverflow();

        uint256 result;
        if (deltaExponent > 0) {
            result = Math.mulDiv(
                uint64(price1),
                10 ** uint64(deltaExponent),
                uint64(price2)
            );

            );
        }
