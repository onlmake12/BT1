### Title
Non-IP Multiaddr Bypass in `is_valid_addr` Allows Peer Store Pollution and Address Propagation — (`network/src/protocols/discovery/mod.rs`)

### Summary

`DiscoveryAddressManager::is_valid_addr` unconditionally returns `true` for any multiaddr that `multiaddr_to_socketaddr` cannot parse (i.e., non-IP addresses such as `/dns4/...`, `/ws/...`). This means a remote peer can inject arbitrary DNS or other non-IP addresses into the local peer store via a crafted `Nodes` P2P message, and those addresses are subsequently re-propagated to other peers.

### Finding Description

The logic flaw is in `is_valid_addr`:

```rust
fn is_valid_addr(&self, addr: &Multiaddr) -> bool {
    if !self.discovery_local_address {
        match multiaddr_to_socketaddr(addr) {
            Some(socket_addr) => is_reachable(socket_addr.ip()),
            None => true,   // ← non-IP addresses skip all validation
        }
    } else {
        true
    }
}
``` [1](#0-0) 

When `multiaddr_to_socketaddr` returns `None` (which it does for `/dns4/...`, `/onion3/...`, `/ws/...`), the function returns `true` without any further check. The `is_reachable` guard is only applied to IP-based addresses.

`add_new_addrs` uses `is_valid_addr` as its sole filter before writing to the peer store:

```rust
for (addr, flags) in addrs.into_iter().filter(|addr| self.is_valid_addr(&addr.0)) {
    self.network_state.with_peer_store_mut(|peer_store| {
        peer_store.add_addr(addr.clone(), flags)
    });
}
``` [2](#0-1) 

`add_addr` in the peer store performs no address-type validation of its own — it only checks the ban list: [3](#0-2) 

`get_random` (used to respond to `GetNodes` requests from other peers) applies the same `is_valid_addr` filter, so stored non-IP addresses pass through and are forwarded to third-party peers: [4](#0-3) 

The `addr_manager`'s `fetch_random` also explicitly handles the `None` (non-IP) case and can return such addresses if they are marked connectable: [5](#0-4) 

### Impact Explanation

1. **Peer store pollution**: An attacker can fill the peer store with unresolvable or attacker-controlled DNS names, degrading peer discovery for the victim node and all peers it gossips to.
2. **Address propagation**: Because `get_random` uses the same `is_valid_addr` gate, injected DNS addresses are re-broadcast to other peers in `GetNodes` responses, spreading the pollution network-wide.
3. **DNS-based tracking**: If the node (or a downstream peer) attempts to connect to `/dns4/attacker.com/tcp/8114/p2p/...`, the attacker's authoritative DNS server observes the query, leaking the connecting node's IP address. This is a concrete privacy violation requiring no special privileges.

### Likelihood Explanation

Any peer that can complete a TCP connection and open the discovery protocol can send a `Nodes` message. No authentication, stake, or elevated privilege is required. The `discovery_local_address=false` condition is the production default. The exploit is a single crafted P2P message.

### Recommendation

Change the `None` branch of `is_valid_addr` to return `false` for address types that are not explicitly supported (e.g., only allow `Onion3` as a known non-IP exception), rather than silently passing all unrecognized address formats:

```rust
None => {
    // Only allow known non-IP types (e.g., Onion3); reject everything else
    addr.iter().any(|p| matches!(p, Protocol::Onion3(_)))
}
```

### Proof of Concept

1. Connect to a CKB node with `discovery_local_address=false` (default).
2. Send a `Nodes` message containing `Node { addresses: ["/dns4/attacker.com/tcp/8114/p2p/<peer_id>"], flags: ... }`.
3. Observe via peer store inspection that `/dns4/attacker.com/...` is now stored.
4. Send a `GetNodes` request from a third peer; observe the DNS address returned in the `Nodes` response.
5. On `attacker.com`'s DNS server, observe incoming A-record queries from the third peer's IP.
