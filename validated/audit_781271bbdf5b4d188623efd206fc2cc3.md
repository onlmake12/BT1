### Title
Malicious Requester Contract Can Grief Provider/Keeper Gas via Simulation-Tricking Callback in Legacy `revealWithCallback` Path â€” (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function has two execution paths. The new path (for requests with `gasLimit10k != 0`) uses `excessivelySafeCall` to catch callback reverts. The legacy path (for requests where `gasLimit10k == 0`, i.e., providers with `defaultGasLimit == 0`) calls `_entropyCallback` directly with no revert protection. A malicious requester contract can deploy a callback that passes off-chain simulation but reverts on-chain, causing the provider/keeper's `revealWithCallback` transaction to revert and wasting their gas with no reimbursement.

---

### Finding Description

In `requestHelper`, when a provider has `defaultGasLimit == 0`, the stored request gets `gasLimit10k = 0`: [1](#0-0) 

This causes `revealWithCallback` to take the legacy branch, which calls `_entropyCallback` directly â€” no try/catch, no `excessivelySafeCall`: [2](#0-1) 

Contrast this with the protected new path, which uses `excessivelySafeCall` and handles reverts gracefully without reverting the outer transaction: [3](#0-2) 

The `requestWithCallback` (legacy entry point) routes to `requestV2` with `gasLimit=0`, so any provider that has not called `setDefaultGasLimit` will produce requests that always land in the unprotected path: [4](#0-3) 

A malicious requester contract can implement `entropyCallback` to:
- Return success when called via `eth_call` (off-chain simulation, e.g., by checking `block.coinbase == address(0)`, `tx.origin`, or any state that differs between simulation and on-chain execution)
- Revert when called on-chain

When the provider/keeper simulates `revealWithCallback` off-chain, it succeeds. When submitted on-chain, the callback reverts, the entire `revealWithCallback` transaction reverts, and the provider loses gas with no reimbursement. The user's fee is already accrued to the provider in `requestHelper` (line 237), but the gas cost of the failed reveal transaction is unrecovered. [5](#0-4) 

---

### Impact Explanation

The provider/keeper wastes ETH on gas for every failed `revealWithCallback` transaction caused by a malicious callback. Since `revealWithCallback` is callable by anyone (not just the provider), the provider is the party economically harmed. A malicious user can repeatedly submit requests to a provider with `defaultGasLimit == 0`, each with a simulation-tricking callback, forcing the provider to waste gas on every reveal attempt. This is a direct financial loss to the provider with no on-chain recourse.

---

### Likelihood Explanation

Medium. Any unprivileged user can deploy a malicious `IEntropyConsumer` contract and call `requestWithCallback` targeting a provider with `defaultGasLimit == 0`. The `IEntropyConsumer` interface imposes no restrictions on the `entropyCallback` implementation. Providers that have not explicitly opted into the new gas-limit flow (by calling `setDefaultGasLimit`) are permanently exposed. The attack requires only a small upfront fee payment (the request fee), making it economically viable to grief providers repeatedly. [6](#0-5) 

---

### Recommendation

1. **Deprecate the legacy path**: Require all providers to set a non-zero `defaultGasLimit` before accepting `requestWithCallback` calls. Reject requests to providers with `defaultGasLimit == 0` in `requestHelper`, or migrate all legacy requests to the safe path.
2. **Apply `excessivelySafeCall` unconditionally**: In the `else` branch of `revealWithCallback` (the legacy path), replace the direct `IEntropyConsumer(callAddress)._entropyCallback(...)` call with an `excessivelySafeCall` using a reasonable gas cap, and emit a `CallbackFailed` event on revert rather than propagating the revert.
3. **Off-chain simulation guidance**: Document that providers should treat simulation success as non-binding for callbacks from unknown requester contracts, and consider an allowlist/reputation system for requester contracts.

// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

abstract contract IEntropyConsumer {
    // This method is called by Entropy to provide the random number to the consumer.
    // It asserts that the msg.sender is the Entropy contract. It is not meant to be
    // override by the consumer.
    function _entropyCallback(
        uint64 sequence,
        address provider,
        bytes32 randomNumber
    ) external {
        address entropy = getEntropy();
        require(entropy != address(0), "Entropy address not set");
        require(msg.sender == entropy, "Only Entropy can call this function");

        entropyCallback(sequence, provider, randomNumber);
    }

    // getEntropy returns Entropy contract address. The method is being used to check that the
    // callback is indeed from Entropy contract. The consumer is expected to implement this method.
    // Entropy address can be found here - https://docs.pyth.network/entropy/contract-addresses
    function getEntropy() internal view virtual returns (address);

    // This method is expected to be implemented by the consumer to handle the random number.
    // It will be called by _entropyCallback after _entropyCallback ensures that the call is
    // indeed from Entropy contract.
    function entropyCallback(
        uint64 sequence,
        address provider,
        bytes32 randomNumber
    ) internal virtual;
}
