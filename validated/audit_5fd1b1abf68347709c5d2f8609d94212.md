### Title
Peer Store Flooding via Off-by-One in `verify_nodes_message` Allows Discovery Partition — (`network/src/protocols/discovery/mod.rs`)

---

### Summary

`verify_nodes_message` uses strict `>` comparisons against both `MAX_ADDR_TO_SEND` (1000) and `MAX_ADDRS` (3), meaning a message with **exactly** 1000 items each carrying **exactly** 3 addresses passes all validation. A single session can inject 3000 addresses per `Nodes(announce=false)` message. Across ~6 attacker-controlled sessions, the 16384-entry peer store (`ADDR_COUNT_LIMIT`) is filled with crafted fake addresses from diverse network groups, causing `EvictionFailed` for all subsequent honest `add_addr` calls and poisoning peer discovery.

---

### Finding Description

**Off-by-one in `verify_nodes_message`:**

```
// Line 279
else if nodes.items.len() > MAX_ADDR_TO_SEND {   // 1000 > 1000 → false ✓ passes
// Line 292
if item.addresses.len() > MAX_ADDRS {             // 3 > 3 → false ✓ passes
```

Both checks use strict `>` instead of `>=`, so a message at exactly the stated limits is accepted as valid. [1](#0-0) 

**Address injection path:**

After `verify_nodes_message` returns `None`, the `received` handler flattens all node addresses and calls `add_new_addrs` with the full 3000-entry list: [2](#0-1) 

`DiscoveryAddressManager::add_new_addrs` iterates each address and calls `peer_store.add_addr`. Errors (including `EvictionFailed`) are only logged — execution continues: [3](#0-2) 

**Peer store eviction failure:**

`PeerStore::add_addr` calls `check_purge()` before inserting. When `addr_manager.count() >= ADDR_COUNT_LIMIT` (16384), it attempts two eviction strategies:
1. Evict non-connectable addresses — attacker's freshly-inserted addresses have `last_connected_at_ms = 0` and `attempts_count = 0`, so they are considered connectable and are not evicted.
2. Network-group eviction — only evicts from groups with **>4 peers**. If the attacker uses addresses from many distinct /24 subnets (≤4 per group), no group qualifies and this step also produces nothing.

When both steps yield no candidates, `check_purge` returns `Err(PeerStoreError::EvictionFailed)`: [4](#0-3) 

`ADDR_COUNT_LIMIT` is 16384: [5](#0-4) 

**Per-session replay prevention is insufficient:**

The `received_nodes` flag prevents a single session from sending a second `Nodes(announce=false)` message: [6](#0-5) 

But this is per-session. The attacker opens ~6 independent inbound connections, each sending one crafted message. 6 × 3000 = 18,000 unique addresses exceeds `ADDR_COUNT_LIMIT` of 16,384. `AddrManager.add` deduplicates by exact address, so the attacker simply uses 16,384+ distinct routable IP:port combinations: [7](#0-6) 

---

### Impact Explanation

Once the peer store is saturated with attacker-controlled fake addresses:
- All subsequent `add_addr` calls from honest peers return `EvictionFailed` (silently dropped in `add_new_addrs`).
- `fetch_addrs_to_feeler` returns attacker-controlled addresses, causing the victim to waste outbound connection attempts on non-existent peers.
- `fetch_random_addrs` (used for discovery gossip) filters for `last_connected_at_ms > 0`, so the fake addresses don't propagate further — but the victim's own discovery is still starved.
- The victim node cannot learn about honest peers, effectively partitioning it from the network.

---

### Likelihood Explanation

- Requires only ~6 inbound TCP connections — no authentication, no PoW, no privileged role.
- The attacker needs to craft valid routable IP:port multiaddrs (trivial; any public IP range works).
- The `is_valid_addr` filter only rejects non-routable IPs (`is_reachable`), which the attacker trivially avoids.
- The attack is repeatable after node restart (peer store is persisted to disk and reloaded).
- No rate limiting exists on the number of inbound discovery sessions or on the rate of `Nodes` messages.

---

### Recommendation

1. **Fix the off-by-one**: Change both comparisons in `verify_nodes_message` from `>` to `>=`:
   ```rust
   else if nodes.items.len() >= MAX_ADDR_TO_SEND { ... }
   // and
   if item.addresses.len() >= MAX_ADDRS { ... }
   ```
2. **Per-session address budget**: Track how many addresses each session has contributed and reject/disconnect once a per-session cap is reached.
3. **Rate-limit inbound discovery sessions**: Limit the number of concurrent inbound discovery protocol sessions per IP or per time window.
4. **Improve eviction**: Evict attacker-inserted addresses (those with `last_connected_at_ms == 0` and `attempts_count == 0`) preferentially before evicting addresses with connection history.

---

### Proof of Concept

```
1. Attacker opens 6 TCP connections to victim's P2P port.
2. On each connection, after discovery protocol handshake, attacker sends:
     Nodes {
       announce: false,
       items: [Node { addresses: [addr1, addr2, addr3], flags: 0 }; 1000]
     }
   where each addr is a unique routable IP:port (e.g., 1.0.0.1:8115 through 18.0.0.0:8115).
3. verify_nodes_message: 1000 > 1000 → false; 3 > 3 → false → returns None.
4. add_new_addrs inserts up to 3000 addresses per session → 18,000 total.
5. After addr_manager.count() reaches 16,384, check_purge finds no eviction candidates
   (all attacker addresses are "connectable", all in distinct /24 groups with ≤4 peers).
6. check_purge returns Err(EvictionFailed).
7. All subsequent honest add_addr calls fail silently.
8. Victim's fetch_addrs_to_feeler returns only fake addresses; honest peers cannot be discovered.
```

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {
                            Some(
                                addrs
                                    .iter()
                                    .choose_multiple(&mut rand::thread_rng(), 2)
                                    .into_iter()
                                    .map(|addr| addr.addr.clone())
                                    .collect::<Vec<Multiaddr>>(),
                            )
                        } else {
                            None
                        }
                    })
                    .flatten()
                    .collect()
            };

            for key in candidate_peers.iter() {
                self.addr_manager.remove(key);
            }

            if candidate_peers.is_empty() {
                return Err(PeerStoreError::EvictionFailed.into());
            }
        }
