### Title
Insufficient Price ID Verification Allows Malicious Provider to Deliver Wrong Price Data to Callback - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol` stores only the first 8 bytes (`bytes8`) of each requested price ID. During `executeCallback`, only these 8-byte prefixes are verified against the executor-supplied price IDs. A malicious provider (or any caller after the exclusivity period) can supply a different price feed whose first 8 bytes collide with the originally requested feed, causing the user's `_echoCallback` to receive price data for the wrong asset.

### Finding Description

In `requestPriceUpdatesWithCallback`, the contract stores only a truncated prefix of each price ID:

```solidity
req.priceIdPrefixes = new bytes8[](priceIds.length);
for (uint8 i = 0; i < priceIds.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    req.priceIdPrefixes[i] = prefix;
}
```

In `executeCallback`, the verification only checks these 8-byte prefixes against the executor-supplied `priceIds`:

```solidity
for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    if (prefix != req.priceIdPrefixes[i]) {
        revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
    }
}
```

After this prefix-only check passes, `parsePriceFeedUpdates` is called with the executor-supplied `priceIds` — not the original ones stored at request time:

```solidity
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(
    updateData,
    priceIds,                          // executor-controlled
    SafeCast.toUint64(req.publishTime),
    SafeCast.toUint64(req.publishTime)
);
```

The resulting `priceFeeds` (for the substitute asset) are then forwarded directly to the requester's callback:

```solidity
IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(sequenceNumber, priceFeeds)
```

No verification is ever performed that the full 32-byte price IDs returned by Pyth match what the user originally requested.

### Impact Explanation

A user's `_echoCallback` receives price data for a different asset than requested. For a DeFi application (e.g., a lending protocol, perpetuals exchange, or liquidation engine), receiving the wrong price can cause:
- Incorrect liquidation thresholds applied
- Wrong trade execution prices
- Financial loss proportional to the position size

The user paid fees expecting price data for asset A but receives data for asset B. This is a direct financial harm analogous to M-06: payment is made for a specific service, but a different (potentially worthless or adversarial) output is delivered.

### Likelihood Explanation

The attack requires two Pyth price feed IDs to share the same leading 8 bytes. Pyth currently publishes ~400+ price feeds. The 8-byte prefix space is 2^64, so natural collisions among existing feeds are unlikely today. However:

1. As Pyth expands to thousands of feeds, collision probability grows quadratically.
2. A malicious provider could attempt to register a new price feed (if the feed creation process allows attacker influence over the ID) with a crafted prefix.
3. After the exclusivity period expires, **any** address can call `executeCallback` — not just the assigned provider — widening the attacker surface to all unprivileged callers.

The permissionless nature of `executeCallback` after the exclusivity window means the attacker does not need to be the registered provider.

### Recommendation

Store and verify the full 32-byte price IDs, not just an 8-byte prefix. Replace `bytes8` storage with `bytes32`:

```diff
- req.priceIdPrefixes = new bytes8[](priceIds.length);
+ req.priceIds = new bytes32[](priceIds.length);
  for (uint8 i = 0; i < priceIds.length; i++) {
-     bytes32 priceId = priceIds[i];
-     bytes8 prefix;
-     assembly { prefix := priceId }
-     req.priceIdPrefixes[i] = prefix;
+     req.priceIds[i] = priceIds[i];
  }
```

And in `executeCallback`:

```diff
  for (uint8 i = 0; i < req.priceIds.length; i++) {
-     bytes32 priceId = priceIds[i];
-     bytes8 prefix;
-     assembly { prefix := priceId }
-     if (prefix != req.priceIdPrefixes[i]) {
+     if (priceIds[i] != req.priceIds[i]) {
          revert InvalidPriceIds(...);
      }
  }
```

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback(provider, t, [feedA], gasLimit)` paying the required fee. `feedA = 0xABCDEF1200000000...` (32 bytes). Echo stores prefix `0xABCDEF12`.

2. Attacker identifies `feedB = 0xABCDEF1299999999...` — a different Pyth price feed whose first 8 bytes equal `0xABCDEF12`.

3. After the exclusivity period, attacker calls `executeCallback(anyProvider, seqNum, updateDataForFeedB, [feedB])`.

4. Echo checks: `bytes8(feedB) == 0xABCDEF12 == stored prefix` → passes.

5. `pyth.parsePriceFeedUpdates(updateDataForFeedB, [feedB], ...)` returns price data for `feedB`.

6. `IEchoConsumer(req.requester)._echoCallback(seqNum, priceFeedsForFeedB)` is called.

7. The user's application processes the price of `feedB` believing it is the price of `feedA`, leading to incorrect financial logic.
