### Title
Unbounded Cell Scan in `get_cells_capacity` Enables RPC Resource Exhaustion DoS — (`util/indexer/src/service.rs`)

---

### Summary

The `get_cells_capacity` RPC endpoint in CKB's built-in RocksDB indexer iterates over every matching cell in the index without any count limit. Unlike the sibling endpoints `get_cells` and `get_transactions`, which both require a caller-supplied `limit: Uint32` parameter, `get_cells_capacity` accepts only a `search_key` and scans the entire matching key space. An unprivileged RPC caller can craft a broad prefix search key (e.g., empty `args`) to force the node to scan millions of cells, exhausting CPU and I/O resources for the duration of the scan. Concurrent such requests amplify the impact.

---

### Finding Description

`get_cells` and `get_transactions` both require a `limit` parameter that caps the number of records returned per call: [1](#0-0) 

`get_cells_capacity` has no such parameter — it accepts only a `search_key`: [2](#0-1) 

The implementation in `util/indexer/src/service.rs` builds a RocksDB iterator from the search key and folds over **every matching record** to accumulate a capacity sum: [3](#0-2) 

The only protection is a `TimeoutIterator` wrapping the RocksDB iterator: [4](#0-3) 

This is a time-based bound, not a count-based bound. It does not prevent:
- Exhausting CPU and I/O for the full duration of the timeout per request.
- Amplification via concurrent requests from multiple callers, each consuming resources for the full timeout window.
- Degradation of all other RPC handlers sharing the same thread pool.

The rich-indexer variant (`util/rich-indexer`) delegates to a SQL `SUM()` aggregate, which is also unbounded but handled by the database engine. The RocksDB-based indexer performs the aggregation in application code, making it the more directly exploitable path.

---

### Impact Explanation

An unprivileged RPC caller submits repeated `get_cells_capacity` requests with a broad prefix search key (e.g., a well-known `code_hash` with empty `args`). Each request causes the node to scan the entire matching cell set in RocksDB, consuming CPU and I/O for the full timeout duration. With concurrent requests, the node's RPC thread pool becomes saturated, causing all other RPC calls (including mining-related calls such as `get_block_template`) to time out or queue indefinitely. This constitutes a denial-of-service against the node's RPC service.

---

### Likelihood Explanation

The `get_cells_capacity` RPC is a standard, documented endpoint reachable by any RPC caller. On a mainnet node with the indexer enabled, the cell set for common lock scripts (e.g., secp256k1-blake160) contains millions of entries. No authentication or rate limiting is applied at the RPC layer by default. The attack requires only knowledge of a common `code_hash` and the ability to send JSON-RPC requests.

---

### Recommendation

Apply the same pattern used by `get_cells` and `get_transactions`: add an optional `limit` parameter to `get_cells_capacity` that caps the number of cells scanned per call. Alternatively, enforce a hard internal cap on the number of cells iterated before returning a partial result or an error. The `TimeoutIterator` should remain as a secondary safeguard but must not be the sole protection.

---

### Proof of Concept

Send the following JSON-RPC request repeatedly (or concurrently) to a node with the indexer enabled:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells_capacity",
  "params": [
    {
      "script": {
        "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
        "hash_type": "type",
        "args": "0x"
      },
      "script_type": "lock",
      "script_search_mode": "prefix"
    }
  ]
}
```

The empty `args` with `prefix` mode matches every cell locked by the secp256k1-blake160 script, forcing a full scan of the cell index. The node's own stress-test scripts confirm this pattern is used for load testing: [5](#0-4) 

On a mainnet-scale index, each such request will consume the full `timeout_limit` of CPU and I/O. Sending 30 concurrent requests (as the stress script does) saturates the node's RPC capacity, blocking all other callers.
