### Title
Fee Manager Replacement Race Condition Allows Draining Provider Accrued Fees Before Revocation - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

`setFeeManager` overwrites the authorized fee manager address unconditionally (no check on current value), mirroring the ERC20 `approve` race condition. A fee manager being replaced can observe the provider's `setFeeManager` transaction in the mempool and front-run it with `withdrawAsFeeManager` to drain all accrued fees before authorization is revoked. The same pattern exists in `Echo.sol`.

### Finding Description

`setFeeManager` in `Entropy.sol` sets `provider.feeManager = manager` atomically but without any guard on the current value: [1](#0-0) 

`withdrawAsFeeManager` allows the current fee manager to pull any amount up to `accruedFeesInWei`: [2](#0-1) 

Because `setFeeManager` is a plain SET with no transition guard, the window between the provider broadcasting the replacement transaction and it being mined is exploitable. The old fee manager can insert a `withdrawAsFeeManager` call with higher gas priority, draining the balance before the new manager is installed.

The identical pattern exists in `Echo.sol`: [3](#0-2) 

### Impact Explanation

A provider who has accrued fees and decides to rotate their fee manager (e.g., due to a key compromise or operational change) loses all accrued ETH fees. The new fee manager inherits an empty balance. The provider cannot recover the funds because the withdrawal already cleared `accruedFeesInWei` before the replacement took effect. This is a direct, irreversible financial loss to the provider.

### Likelihood Explanation

Any fee manager that is being replaced has a clear financial incentive to monitor the mempool for `setFeeManager` calls from their associated provider. On public EVM chains (Ethereum mainnet, Arbitrum, etc.) where Entropy is deployed, mempool visibility is standard. The attacker only needs to submit one `withdrawAsFeeManager` transaction with a higher gas price — no special tooling is required. The scenario is realistic whenever a provider rotates a compromised or untrusted fee manager.

### Recommendation

- Adopt a two-step rotation: require the provider to first set the fee manager to `address(0)` (clearing it), then set the new address. This forces the old manager's authorization to lapse before the new one is granted, eliminating the double-spend window.
- Alternatively, require `accruedFeesInWei == 0` before `setFeeManager` can be called, forcing the provider to withdraw or zero out fees first.
- At minimum, document the race condition prominently so providers know to withdraw fees before rotating the fee manager.

### Proof of Concept

1. Provider `P` has `accruedFeesInWei = 1 ETH` and fee manager `M1`.
2. `P` decides to rotate to `M2` and broadcasts `setFeeManager(M2)`.
3. `M1` observes the pending transaction and immediately submits `withdrawAsFeeManager(P, 1 ETH)` with higher gas.
4. `M1`'s transaction mines first: `P.accruedFeesInWei` drops to `0`, and `M1` receives `1 ETH`.
5. `P`'s `setFeeManager(M2)` mines next: `M2` is now the fee manager but `accruedFeesInWei == 0`.
6. `M2` calls `withdrawAsFeeManager(P, 1 ETH)` → reverts with `"Insufficient balance"`.
7. Net result: `M1` extracted `1 ETH` that `P` intended for `M2`; `P` suffers a permanent `1 ETH` loss.

The same exploit path applies to `Echo.sol` via its `setFeeManager` / `withdrawAsFeeManager` pair. [4](#0-3)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer
        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "withdrawal to msg.sender failed");

        emit EntropyEvents.Withdrawal(provider, msg.sender, amount);
        emit EntropyEventsV2.Withdrawal(

        );
    }

    function setFeeManager(address manager) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        address oldFeeManager = provider.feeManager;
        provider.feeManager = manager;
        emit ProviderFeeManagerUpdated(msg.sender, oldFeeManager, manager);
        emit EntropyEventsV2.ProviderFeeManagerUpdated(
            msg.sender,
            oldFeeManager,
            manager,
            bytes("")

    function setFeeManager(address manager) external override {
        require(
            _state.providers[msg.sender].isRegistered,
            "Provider not registered"
        );
        address oldFeeManager = _state.providers[msg.sender].feeManager;
        _state.providers[msg.sender].feeManager = manager;
        emit FeeManagerUpdated(msg.sender, oldFeeManager, manager);
    }

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
    }
