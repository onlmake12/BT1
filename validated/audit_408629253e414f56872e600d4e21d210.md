### Title
Unprivileged Users Can Exhaust `maxNumHashes` to Trigger Global `LastRevealedTooOld` DoS on Entropy Provider — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any unprivileged user can make `maxNumHashes` paid requests to the Entropy contract in rapid succession, causing every subsequent request from every user to revert with `LastRevealedTooOld` until the Fortuna keeper service advances the provider commitment. This is a repeatable, economically bounded denial-of-service attack that disrupts the Entropy randomness service globally for all users of the targeted provider.

---

### Finding Description

The `requestHelper` function in `Entropy.sol` computes `req.numHashes` as the gap between the newly assigned sequence number and the provider's last revealed commitment sequence number:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

`assignedSequenceNumber` is a global, monotonically increasing counter (`providerInfo.sequenceNumber`) that increments by 1 for every request, regardless of who makes it. `currentCommitmentSequenceNumber` only advances when the Fortuna keeper calls `advanceProviderCommitment` on-chain.

The Fortuna keeper (`apps/fortuna/src/keeper/commitment.rs`) checks this condition every **30 seconds** and only triggers when `outstanding_requests > 0.95 * maxNumHashes`:

```rust
const UPDATE_COMMITMENTS_INTERVAL: Duration = Duration::from_secs(30);
const UPDATE_COMMITMENTS_THRESHOLD_FACTOR: f64 = 0.95;
...
if outstanding_requests > threshold {
    // advance commitment
}
```

An attacker can submit exactly `maxNumHashes` requests in a single block (or across multiple blocks before the keeper responds). After those requests land, `numHashes` for the very next request equals `maxNumHashes + 1`, causing `LastRevealedTooOld` for **all** subsequent callers — not just the attacker — until the keeper advances the commitment (up to 30 seconds later). The attacker can repeat this cycle indefinitely.

`advanceProviderCommitment` is declared `public` with no access control, meaning anyone who knows the hash chain values (observable from on-chain reveals) can call it. However, the keeper is the only party that does so in practice, and its 30-second polling interval is the exploitable window.

---

### Impact Explanation

- All new entropy requests to the targeted provider revert with `LastRevealedTooOld` during the DoS window (up to ~30 seconds per cycle).
- Any on-chain application relying on Entropy for randomness (e.g., NFT mints, gaming contracts, lotteries) is blocked from requesting randomness during the attack.
- The attack is repeatable: the attacker can re-trigger the condition immediately after the keeper recovers, sustaining near-continuous service degradation.
- The default provider is the most widely used (`_state.defaultProvider`), so targeting it affects the broadest set of users.

---

### Likelihood Explanation

- **No privileged access required.** Any address with sufficient ETH to pay fees can execute this.
- **Cost is bounded by `maxNumHashes * fee`.** If `maxNumHashes` is set to a moderate value (e.g., 1,000) and the fee is low (e.g., 0.0001 ETH), the cost per 30-second DoS window is ~0.1 ETH — feasible for a motivated attacker.
- **The keeper's 30-second polling interval is a fixed, known window** that the attacker can exploit reliably.
- The Fortuna keeper itself logs a warning when the threshold is reached (`"possible outage or DDOS attack"`), confirming this attack vector is anticipated but not fully mitigated at the contract level.

---

### Recommendation

1. **Short term:** Reduce the Fortuna keeper's `UPDATE_COMMITMENTS_INTERVAL` from 30 seconds to a much shorter period (e.g., per-block or per-reveal). Advance the commitment proactively after every reveal, not just when the threshold is crossed.
2. **Short term:** Consider advancing `currentCommitmentSequenceNumber` automatically within `revealHelper` to keep the gap small without relying on a separate keeper loop.
3. **Long term:** Explore per-user request rate limiting at the contract level, or require a minimum delay between requests from the same address, to raise the economic cost of the attack.

---

### Proof of Concept

1. Read `providerInfo.maxNumHashes` and `providerInfo.currentCommitmentSequenceNumber` for the default provider.
2. Submit exactly `maxNumHashes` calls to `requestV2(provider, userRandomNumber, gasLimit)` paying the required fee each time. All succeed and increment `providerInfo.sequenceNumber` to `currentCommitmentSequenceNumber + maxNumHashes + 1`.
3. Submit one more `requestV2` call. It reverts with `LastRevealedTooOld` because `numHashes = maxNumHashes + 1 > maxNumHashes`.
4. Any other user attempting `requestV2` during this window also receives `LastRevealedTooOld`.
5. Wait up to 30 seconds for the Fortuna keeper to call `advanceProviderCommitment`, resetting `currentCommitmentSequenceNumber`.
6. Return to step 2 and repeat.
