### Title
Unauthenticated `ConnectionRequestDelivered` Drains `inflight_requests`, Suppressing Hole-Punching — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

---

### Summary

A directly connected peer can forge `ConnectionRequestDelivered` messages with `from` set to the victim's own peer ID and `to` set to any peer ID observed in gossip, causing the victim to unconditionally remove entries from `inflight_requests` with no verification that the actual sender is the claimed `to` peer. This drains all in-flight hole-punching state and prevents NAT traversal from completing.

---

### Finding Description

In `ConnectionRequestDeliveredProcess::execute`, when `content.route` is empty and `content.from == self_peer_id`, the code reaches the terminal branch and calls:

```rust
let request_start = self.protocol.inflight_requests.remove(&content.to);
``` [1](#0-0) 

The struct holds `self.peer` (the `PeerIndex` of the actual TCP session that sent the message): [2](#0-1) 

**`self.peer` is never compared against `content.to`.** There is no check that the actual sender of the message is the peer identified by `content.to`. The only guard is `content.from == self_peer_id`, which the attacker controls freely since the victim's peer ID is public.

The `inflight_requests` map is populated in `notify()` after broadcasting `ConnectionRequest` gossip messages. The `to` peer IDs are embedded in those gossip messages and are directly observable by any connected peer: [3](#0-2) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1 req/sec per unique tuple. For N distinct `to` values, N forged messages each with a different `to` all pass the rate limiter simultaneously: [4](#0-3) 

The session-level rate limiter allows 30 messages/sec per session, so up to 30 entries can be drained per second per attacker session: [5](#0-4) 

---

### Impact Explanation

When `inflight_requests.remove(&content.to)` returns `None` for a legitimate `ConnectionRequestDelivered` (because the attacker already removed the entry), the code returns `StatusCode::Ignore` and the NAT traversal is abandoned: [6](#0-5) 

As long as the attacker continuously drains entries faster than `notify()` repopulates them, the victim cannot complete any hole-punching handshake. The victim's outbound NAT traversal capability is fully suppressed for the duration of the attack.

---

### Likelihood Explanation

- **Attacker requirement**: Any peer with a direct P2P session to the victim — no privilege required.
- **Information needed**: Victim's peer ID (public) and `to` peer IDs (observable from gossip `ConnectionRequest` broadcasts the victim sends).
- **Effort**: Trivially automatable; send one forged message per observed `to` peer ID per second.
- **Persistence**: Must be maintained continuously, but `notify()` runs on a fixed interval so the attacker can time their drain to each cycle.

---

### Recommendation

In the terminal branch of `execute`, verify that the actual sender (`self.peer`) corresponds to the peer identified by `content.to` before removing from `inflight_requests`:

```rust
// Resolve the PeerIndex for content.to and assert it equals self.peer
let expected_sid = self.protocol.network_state
    .peer_registry.read()
    .get_key_by_peer_id(&content.to);
if expected_sid != Some(self.peer) {
    return StatusCode::Ignore.with_context("sender is not the claimed `to` peer");
}
let request_start = self.protocol.inflight_requests.remove(&content.to);
```

This ensures only the genuine `to` peer (the one whose session delivered the message) can remove an inflight entry.

---

### Proof of Concept

1. Connect attacker node A to victim node V (standard P2P connection).
2. Observe gossip `ConnectionRequest` messages from V; extract all `to` peer IDs → these are exactly the keys in V's `inflight_requests`.
3. For each observed `to_peer_id`, send a `ConnectionRequestDelivered` to V with:
   - `from` = V's own peer ID
   - `to` = `to_peer_id`
   - `route` = `[]` (empty)
   - `listen_addrs` = any valid TCP multiaddr
4. V processes each message: `content.from == self_peer_id` is true, `inflight_requests.remove(&content.to)` removes the entry.
5. When the legitimate `ConnectionRequestDelivered` from the real `to` peer arrives, `inflight_requests.remove` returns `None` → `StatusCode::Ignore` → NAT traversal aborted.
6. Repeat each `notify()` cycle to maintain suppression.
