Audit Report

## Title
Fee Admission Check Uses Serialized Size Only, Allowing High-Cycle Transactions to Bypass Effective `min_fee_rate` - (File: `tx-pool/src/util.rs`)

## Summary
`check_tx_fee` validates the minimum fee using only the transaction's serialized byte size, while pool ordering, block selection, and fee-rate statistics all use `get_transaction_weight`, which takes the maximum of byte size and cycle-equivalent weight. An attacker can craft a transaction with a small serialized size but near-maximum cycle consumption, pass the fee gate cheaply, and enter the pool with an effective fee rate far below `min_fee_rate`. The code's own comment explicitly acknowledges this is theoretically incorrect.

## Finding Description

**Root cause — `tx-pool/src/util.rs`, `check_tx_fee`, line 45:**

The admission check computes the minimum fee as `min_fee_rate × tx_size / 1000`, ignoring cycles entirely:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

**Actual weight used everywhere else — `util/types/src/core/tx_pool.rs`, `get_transaction_weight`:**

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` means at 70,000,000 cycles (consensus max), the cycle-equivalent weight is ~11,940 bytes regardless of actual serialized size. [3](#0-2) 

**Pool ordering uses full weight — `tx-pool/src/component/entry.rs`, `fee_rate`:** [4](#0-3) 

**The admission flow — `tx-pool/src/process.rs`, `pre_check`:**

`check_tx_fee` is called with `tx_size` before script verification (before cycles are known). After verification, `TxEntry::new` is created with the actual `verified.cycles`, creating a permanent mismatch between the fee that was checked and the weight that governs pool behavior: [5](#0-4) [6](#0-5) 

**Relay path has declared cycles available but does not use them for the fee check:** [7](#0-6) 

**Concrete numerical example:**

| Parameter | Value |
|---|---|
| `tx_size` | 300 bytes |
| `cycles` | 70,000,000 (max) |
| `cycle_weight` | `70,000,000 × 0.000_170_571_4 ≈ 11,940` bytes |
| `actual_weight` | `max(300, 11940) = 11,940` |
| Fee required by `check_tx_fee` | `1000 × 300 / 1000 = 300 shannons` |
| Effective fee rate in pool | `300 × 1000 / 11940 ≈ 25 shannons/KW` |
| Configured `min_fee_rate` | `1000 shannons/KW` |

The transaction passes admission at 1000 shannons/KW but occupies pool and block resources at an effective rate of ~25 shannons/KW — a **40× undercharge** for cycle-heavy scripts.

## Impact Explanation

This matches the allowed CKB bounty impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points).**

1. **Cheap pool flooding**: An unprivileged attacker can submit many small-size, high-cycle transactions paying only byte-size fees. Each transaction forces the node to execute up to 70M cycles of script verification, consuming significant CPU resources at a cost far below what `min_fee_rate` is intended to enforce.

2. **Fee estimation distortion**: Both `FeeRateCollector::statistics` (historical, from committed blocks) and `PoolMap::estimate_fee_rate` (pool-based fallback) use the full weight. Underpriced high-cycle transactions in the pool or committed blocks pull fee estimates downward, causing honest users to underpay and experience delayed confirmation. [8](#0-7) 

3. **Miner revenue loss**: `AncestorsScoreSortKey` uses full weight for ordering. A cycle-heavy transaction that paid only byte-size fees appears as a low-fee-rate entry and can displace legitimately priced transactions from blocks. [9](#0-8) 

## Likelihood Explanation

- **Entry path**: Reachable by any unprivileged actor via the `send_transaction` JSON-RPC or the P2P relay protocol. No special privilege is required.
- **Ease of exploitation**: A script author deploys a lock or type script that loops for many cycles stored in a cell dep; the transaction's serialized size remains ~300 bytes while consuming the full `max_tx_verify_cycles = 70,000,000` budget.
- **Detection**: The node logs nothing about this discrepancy. The comment in the code explicitly acknowledges the approximation as intentional for performance, meaning it will not be caught by routine monitoring.
- **Repeatability**: The attack can be repeated continuously at low cost, as each transaction pays only byte-size fees.

## Recommendation

For the **relay path**, use `declared_cycles` (already available and validated to match actual cycles at line 736–748 of `process.rs`) as the weight input for the fee check during `pre_check`, consistent with how `max_cycles` is set:

```rust
// In pre_check, when declared_cycles is available:
let weight = get_transaction_weight(tx_size, declared_cycles.unwrap_or(0));
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

For the **RPC path** (where cycles are unknown at `pre_check` time), add a post-verification weight-based fee check before pool insertion:

```rust
// After verify_rtx, before TxEntry::new:
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
``` [10](#0-9) 

## Proof of Concept

1. Deploy a lock script that executes a tight loop consuming ~70,000,000 cycles. Store the script binary in a cell dep so the transaction's serialized size remains ~300 bytes.
2. Submit via `send_transaction` RPC with `fee = 300 shannons` (satisfying `min_fee_rate × tx_size = 1000 × 300 / 1000 = 300`).
3. Observe the transaction is accepted into the pool (no `LowFeeRate` rejection).
4. Query `get_pool_tx_detail_info` — the `score_sortkey.weight` will reflect the cycle-dominated weight (~11,940), giving an effective fee rate of ~25 shannons/KW, far below the configured `min_fee_rate = 1000 shannons/KW`.
5. Repeat with many such transactions. Each forces the node to execute 70M cycles of script verification while paying only 300 shannons. The pool fills with entries whose effective fee rates are ~40× below `min_fee_rate`, distorting fee estimation and consuming block cycle budgets at below-minimum cost.
