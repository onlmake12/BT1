### Title
Peer Reputation Scoring System Documented as Active but Completely Disabled in Production — (`network/src/behaviour.rs`)

### Summary

CKB's network layer documents a peer reputation/scoring system that is supposed to track peer behavior, reduce scores for bad behavior, and eventually ban misbehaving peers. However, the `Behaviour` enum has zero production variants (all variants are `#[cfg(test)]` only), and `Behaviour::score()` unconditionally returns `0` in non-test builds. The entire graduated-response mechanism is inert in production, meaning any peer behavior that was supposed to trigger score degradation has no consequence whatsoever.

### Finding Description

In `network/src/behaviour.rs`, the doc comment explicitly states:

> "we maintain a score to each peer / report peer behaviour will affects peer's score"

But the very next line contradicts this:

> "Currently this feature is disabled, maybe someday we will add it back or totally remove it." [1](#0-0) 

The `Behaviour` enum has no production variants — `TestGood` and `TestBad` are both gated behind `#[cfg(test)]`. In non-test builds, the enum is empty and `score()` always returns `0`: [2](#0-1) 

The `PeerStore::report()` function is fully wired up with a `ban_score` threshold of `40` and a `default_score` of `100`: [3](#0-2) 

And `report()` does call `ban_addr` when score drops below `ban_score`: [4](#0-3) 

But since `behaviour.score()` always returns `0` in production, the score never changes, the ban threshold is never crossed through this path, and `report_peer()` is never called in any production code path (only in test stubs). The `report_session()` function in `network/src/network.rs` exists but can never be invoked with a real `Behaviour` value since the enum has no constructible production variants: [5](#0-4) 

The `CKBProtocolContext` interface exposes `report_peer()` as a first-class method alongside `ban_peer()`, implying callers are expected to use it for graduated responses: [6](#0-5) 

Yet in production, calling `report_peer()` is structurally impossible because there are no `Behaviour` values to pass.

### Impact Explanation

The graduated peer reputation system — the defense-in-depth layer between "no action" and "immediate ban" — is completely non-functional. Any peer behavior that was intended to trigger score degradation (repeated minor protocol violations, borderline-invalid messages, probing behavior) has zero consequence. The network's only protection against misbehaving peers is the direct `ban_peer()` path, which requires each individual violation to independently meet the threshold for immediate banning. Peers that exhibit repeated low-severity misbehavior can stay connected indefinitely without penalty, potentially enabling sustained resource exhaustion, network probing, or protocol abuse that falls below the immediate-ban threshold.

### Likelihood Explanation

Any unprivileged peer connecting to a CKB node benefits from this gap automatically — no special capability is required. The attacker simply connects and sends messages that would have triggered score reductions under the intended system. Since the scoring system is unconditionally disabled at compile time (not a runtime configuration), every production CKB node is affected.

### Recommendation

Either:
1. Define real production `Behaviour` variants (e.g., `UnsolicitedMessage`, `InvalidHeader`, `SlowResponse`) with meaningful negative scores, remove the `#[cfg(test)]` gates, and restore the `#[cfg(not(test))] 0` fallback to a proper match, or
2. Remove the dead infrastructure entirely (`Behaviour`, `PeerScoreConfig`, `PeerStore::report()`, `report_peer()` from the interface) and document that only direct banning is used.

The current state — documented as active, implemented as infrastructure, but silently returning 0 for all scores — is the exact pattern identified in M-03: a security mechanism stated to be in use that is not actually enforced.

### Proof of Concept

1. Compile CKB in release mode (non-test). Observe that `Behaviour` has no constructible variants — the enum is empty.
2. Inspect `Behaviour::score()`: the `#[cfg(not(test))]` arm unconditionally returns `0`.
3. Call `PeerStore::report(&addr, behaviour)` with any `Behaviour` value (only possible in test): `peer_addr.score` is unchanged because `saturating_add(0) == original_score`.
4. Confirm `ban_score` threshold (40) is never reached through this path regardless of how many times `report()` is called.
5. Confirm via `grep_search` that `report_peer` has zero call sites in production code — only in test stubs — because no `Behaviour` value can be constructed outside `#[cfg(test)]`. [7](#0-6) [8](#0-7)

/// Peers behaviours
/// we maintain a score to each peer
/// report peer behaviour will affects peer's score
///
/// Currently this feature is disabled, maybe someday we will add it back or totally remove it.
#[derive(Debug, Clone, Copy, Eq, PartialEq)]
pub enum Behaviour {
    /// Good behaviour
    #[cfg(test)]
    TestGood,
    /// Bad behaviour
    #[cfg(test)]
    TestBad,
}

impl Behaviour {
    /// Behaviour score
    pub fn score(self) -> Score {
        #[cfg(test)]
        match self {
            Behaviour::TestGood => 10,
            Behaviour::TestBad => -10,
        }
        #[cfg(not(test))]
        0
    }

    /// Report peer behaviours
    pub fn report(&mut self, addr: &Multiaddr, behaviour: Behaviour) -> ReportResult {
        if let Some(peer_addr) = self.addr_manager.get_mut(addr) {
            let score = peer_addr.score.saturating_add(behaviour.score());
            peer_addr.score = score;
            if score < self.score_config.ban_score {
                self.ban_addr(
                    addr,
                    self.score_config.ban_timeout_ms,
                    format!("report behaviour {behaviour:?}"),
                );
                return ReportResult::Banned;
            }
        }
        ReportResult::Ok
    }
