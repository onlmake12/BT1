### Title
Adversarial 2-Group Peer Store Fill Causes O(N) Amortized `add_addr` Cost via `take(len/2)` Integer Truncation — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

When the peer store is filled with exactly 2 network groups of connectable addresses, `check_purge`'s second-pass `take(len/2)` evaluates to `take(1)`, evicting only 2 peers per invocation. This causes `check_purge` — an O(N) scan — to be re-triggered every 2 insertions, degrading amortized `add_addr` cost from O(1) to O(N).

---

### Finding Description

`check_purge` is called at the top of `add_addr` whenever `addr_manager.count() >= ADDR_COUNT_LIMIT` (16384). [1](#0-0) 

When all stored addresses are `is_connectable`, the first pass collects nothing and the second pass runs: [2](#0-1) 

The critical line is:

```rust
let len = peers_by_network_group.len();   // number of distinct groups
// ...
peers.into_iter().take(len / 2)           // integer division
``` [3](#0-2) 

With exactly **2 groups**, `len / 2 = 1`. Only the single largest group is visited; from it, exactly 2 peers are randomly evicted: [4](#0-3) 

**Resulting steady-state cycle:**

| Step | Store size | Action |
|------|-----------|--------|
| `add_addr` call | 16384 | `check_purge` fires, evicts 2 → 16382 |
| addr added | 16383 | — |
| `add_addr` call | 16383 | 16383 < 16384 → no purge |
| addr added | 16384 | — |
| `add_addr` call | 16384 | `check_purge` fires again |

`check_purge` is invoked every **2 insertions**, and each invocation performs two full O(N) scans over all 16384 entries (first pass for non-connectable, second pass for grouping + sorting). [5](#0-4) 

---

### Impact Explanation

Each `check_purge` call iterates the entire `addr_manager` twice (O(16384) each), then sorts G groups. With G=2, the sort is O(1), but the two full scans dominate. Triggered every 2 insertions, the amortized cost per `add_addr` becomes **O(N)** instead of O(1). An attacker sustaining a stream of discovery addresses from 2 groups keeps the node in this degraded state indefinitely, consuming CPU proportional to the store size on every peer advertisement processed.

---

### Likelihood Explanation

The attacker path is the P2P discovery protocol, which calls `add_addr` for each advertised peer address: [6](#0-5) 

Addresses added via `add_addr` are untrusted and unverified (`last_connected_at_ms = 0`, `attempts_count = 0`). A freshly added address is `is_connectable` by default because `attempts_count` starts at 0, which is below both `ADDR_MAX_RETRIES` (3) and `ADDR_MAX_FAILURES` (10): [7](#0-6) 

An attacker controlling a single peer can advertise 16384 addresses from two /16 subnets (e.g., `10.0.x.x` and `172.16.x.x`) via discovery messages. No PoW, no stake, and no privileged role is required. [8](#0-7) 

---

### Recommendation

Replace `take(len / 2)` with a loop that continues evicting until the store drops below `ADDR_COUNT_LIMIT`, or compute the required number of evictions upfront and take enough groups to guarantee the store falls below the limit in a single pass. A minimal fix is to keep iterating over all groups (remove the `take`) and stop only once enough peers have been collected to bring the count below the limit.

---

### Proof of Concept

```rust
// Fill store with 16384 entries across exactly 2 /16 groups, all connectable
let mut peer_store = PeerStore::default();
for i in 0u32..8192 {
    let addr: Multiaddr = format!("/ip4/10.0.{}.{}/tcp/1234/p2p/{}", i/256, i%256,
        PeerId::random().to_base58()).parse().unwrap();
    peer_store.addr_manager_mut().add(AddrInfo::new(addr, 0, 100, Flags::COMPATIBILITY.bits()));
}
for i in 0u32..8192 {
    let addr: Multiaddr = format!("/ip4/172.16.{}.{}/tcp/1234/p2p/{}", i/256, i%256,
        PeerId::random().to_base58()).parse().unwrap();
    peer_store.addr_manager_mut().add(AddrInfo::new(addr, 0, 100, Flags::COMPATIBILITY.bits()));
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// Now measure check_purge invocations for K subsequent add_addr calls
let mut purge_count = 0;
for i in 0u32..100 {
    let addr: Multiaddr = format!("/ip4/10.0.{}.{}/tcp/9999/p2p/{}", i/256, i%256,
        PeerId::random().to_base58()).parse().unwrap();
    // Each add_addr triggers check_purge ~every 2 calls; each check_purge is O(16384)
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    purge_count += 1;
}
// Assert purge_count is ~K/2, not O(1) — demonstrating superlinear total work
```

The total work is `O(K * N / 2)` = `O(K * 8192)`, confirming quadratic behavior relative to the number of adversarial insertions.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
