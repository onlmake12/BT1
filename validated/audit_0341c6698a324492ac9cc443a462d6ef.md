### Title
Unthrottled `GetBlockFilterCheckPoints` Handler Enables I/O Exhaustion DoS — (`sync/src/filter/get_block_filter_check_points_process.rs`)

---

### Summary

The `GetBlockFilterCheckPoints` filter protocol handler performs up to 4,000 sequential RocksDB point lookups per request with no per-peer rate limiting. Any unprivileged P2P peer can flood this handler to saturate disk I/O and degrade block relay. The Relayer protocol has an explicit rate limiter; the Filter protocol does not.

---

### Finding Description

`GetBlockFilterCheckPointsProcess::execute` loops up to `BATCH_SIZE = 2000` iterations, and each iteration issues two RocksDB reads — `get_block_hash(block_number)` and `get_block_filter_hash(&block_hash)` — advancing `block_number` by `CHECK_POINT_INTERVAL = 2000` per step, covering up to 4,000,000 blocks. [1](#0-0) [2](#0-1) 

The only guard is `if latest >= start_number` — a trivially satisfied condition when `start_number=0` and any filter data exists. [3](#0-2) 

The `BlockFilter` protocol handler dispatches directly to `try_process` with no rate-limit check anywhere in the filter module: [4](#0-3) 

By contrast, the `Relayer` protocol explicitly installs a `governor`-based rate limiter keyed by `(peer, message_type)` at 30 req/s, and the developers' own comment acknowledges this is intentional protection: [5](#0-4) 

The filter protocol has **zero equivalent protection**.

---

### Impact Explanation

- **I/O saturation**: A single peer sending `GetBlockFilterCheckPoints(start_number=0)` in a tight loop triggers up to 4,000 RocksDB reads per message. Multiple peers amplify this linearly.
- **Degraded block relay**: RocksDB I/O contention directly competes with block validation and relay reads, increasing block processing latency.
- **Consensus deviation** (as claimed): This part is **overstated**. I/O saturation slows the node but does not directly cause consensus failure. The realistic impact is DoS-grade performance degradation, not consensus deviation.

---

### Likelihood Explanation

- Precondition (4M+ blocks) is met on CKB mainnet (running since 2019 at ~8s/block ≈ 27M+ blocks).
- No PoW or authentication is required to connect via the Filter protocol.
- The message is a fixed 8-byte struct (`start_number: Uint64`) — trivial to construct and send in a loop.
- The attack is local-testable on a private chain with sufficient blocks. [6](#0-5) 

---

### Recommendation

Add a per-peer rate limiter to `BlockFilter` mirroring the one in `Relayer`:

```rust
// In BlockFilter::new(), add:
let quota = governor::Quota::per_second(NonZeroU32::new(10).unwrap());
let rate_limiter = RateLimiter::hashmap(quota);
```

Check it in `try_process` before dispatching to any handler, returning `StatusCode::TooManyRequests` on violation. Additionally, consider caching checkpoint results (they are stable for a given chain tip) to amortize repeated identical requests.

---

### Proof of Concept

1. Run a CKB node with block filter enabled on a chain with ≥4,000,001 blocks.
2. Connect a custom P2P client supporting `SupportProtocols::Filter`.
3. Send `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop from N concurrent connections.
4. Observe: `iostat` shows disk read saturation; block relay latency increases; `process message=GetBlockFilterCheckPoints` log lines accumulate with no throttling. [7](#0-6)

impl CKBProtocolHandler for BlockFilter {
    async fn init(&mut self, _nc: Arc<dyn CKBProtocolContext + Sync>) {}

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer_index: PeerIndex,
        data: Bytes,
    ) {
        let msg = match packed::BlockFilterMessageReader::from_compatible_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                info_target!(
                    crate::LOG_TARGET_FILTER,
                    "Peer {} sends us a malformed message",
                    peer_index
                );
                nc.ban_peer(
                    peer_index,
                    BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        debug_target!(
            crate::LOG_TARGET_FILTER,
            "received msg {} from {}",
            msg.item_name(),
            peer_index
        );
        let start_time = Instant::now();
        self.process(nc, peer_index, msg).await;
        debug_target!(
            crate::LOG_TARGET_FILTER,
            "process message={}, peer={}, cost={:?}",
            msg.item_name(),
            peer_index,
            Instant::now().saturating_duration_since(start_time),
        );
    }
