### Title
Inconsistent Fee-Rate Calculation in `check_tx_fee` vs `TxEntry::fee_rate` Allows Below-Minimum-Fee-Rate Transactions into the Tx-Pool — (`File: tx-pool/src/util.rs`)

### Summary

The tx-pool admission function `check_tx_fee` computes the minimum required fee using only the transaction's serialized **byte size**, while the canonical fee-rate formula used everywhere else in the pool (`TxEntry::fee_rate`, fee estimators, eviction logic) uses **weight** = `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For a high-cycles, small-size transaction the two formulas diverge significantly. Because no weight-based re-check is performed after script execution reveals the actual cycle count, an unprivileged RPC caller or relay peer can submit transactions whose effective fee rate is far below `min_fee_rate`, bypassing the pool's spam-prevention gate.

### Finding Description

**Root cause — `tx-pool/src/util.rs` `check_tx_fee`:**

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
``` [1](#0-0) 

`FeeRate::fee(weight)` computes `fee_rate * weight / 1000`. Here `weight` is set to `tx_size`, so the threshold is `min_fee_rate × tx_size / 1000`.

**Canonical formula — `tx-pool/src/component/entry.rs` `TxEntry::fee_rate`:**

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [2](#0-1) 

`get_transaction_weight` returns `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. [3](#0-2) 

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`, so at the maximum cycle limit of 70 000 000 cycles the weight is `max(tx_size, 11 940)`. [4](#0-3) 

**The gap in the admission pipeline — `tx-pool/src/process.rs`:**

```rust
let (tip_hash, rtx, status, fee, tx_size) = try_or_return_with_snapshot!(ret, snapshot);
// ... verify_rtx runs here, actual cycles now known in `verified.cycles` ...
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
``` [5](#0-4) 

After `verify_rtx` returns the true cycle count there is **no second fee-rate check** using the actual weight. The entry is submitted directly.

The same pattern repeats in `readd_detached_tx`: [6](#0-5) 

### Impact Explanation

An attacker crafts a transaction with:
- Small serialized size (e.g., 200 bytes → `min_fee = 1000 × 200 / 1000 = 200 shannons`)
- High cycle consumption (e.g., 70 000 000 cycles → `weight = 11 940`)
- Fee just above the size-based threshold (e.g., 201 shannons)

Effective fee rate after admission: `201 × 1000 / 11 940 ≈ 16 shannons/KW`, far below the configured `min_fee_rate = 1000 shannons/KW`.

Such transactions enter the pool, consume pool space, and are propagated to peers (who apply the same flawed check). Miners will not prioritize them, so they linger and can be used to exhaust pool capacity (`max_tx_pool_size`), degrading service for legitimate users. The same inconsistency affects the eviction key and fee-estimator inputs, which use the correct weight-based formula, creating a mismatch between admission policy and pool management policy.

### Likelihood Explanation

Reachable by any unprivileged `send_transaction` RPC caller or relay peer. No special privilege, key, or majority hash power is required. Crafting a high-cycles script is straightforward (e.g., a loop in a CKB-VM script). The attack is cheap to mount repeatedly.

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee-rate check using the true weight:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let actual_fee_rate = FeeRate::calculate(fee, weight);
if actual_fee_rate < tx_pool.config.min_fee_rate {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Alternatively, refactor `check_tx_fee` to accept an optional `cycles` parameter so the same function can be called both before verification (size-only, as a cheap pre-filter) and after verification (weight-based, as the authoritative gate), consistent with how `TxEntry::fee_rate` and the fee estimators compute the rate.

### Proof of Concept

1. Configure a node with `min_fee_rate = 1000` (default).
2. Write a CKB-VM lock script that loops for ~70 000 000 cycles.
3. Deploy the script and create a cell locked by it.
4. Build a transaction spending that cell with serialized size ≈ 200 bytes and fee = 201 shannons.
5. Submit via `send_transaction` RPC.
6. Observe: the transaction is accepted into the pool (size-based check: `201 >= 1000 × 200 / 1000 = 200` ✓).
7. Query `get_pool_tx_detail_info` and compute `fee × 1000 / weight`: the effective fee rate is ≈ 16 shannons/KW, well below `min_fee_rate = 1000`.
8. Repeat to fill the pool with sub-minimum-fee-rate transactions, crowding out legitimate higher-fee transactions.
