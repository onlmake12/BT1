Audit Report

## Title
Missing Per-Protocol Decompressed Size Guard Enables Memory Amplification — (`network/src/compress.rs`)

## Summary
`LengthDelimitedCodecWithCompress::decode` enforces `max_frame_length` on the compressed wire frame but only checks the declared decompressed size against the global `MAX_UNCOMPRESSED_LEN` (8 MB). Because the per-protocol limit is never applied to the decompressed size, any peer can send a compressed frame within the wire limit whose snappy header declares a decompressed size up to 8 MB, causing the node to allocate a buffer up to 2× (RelayV3) or 4× (Sync) the intended per-protocol memory budget per message.

## Finding Description
`CKBProtocol::build()` constructs `LengthDelimitedCodecWithCompress` with an inner `LengthDelimitedCodec` configured to `max_frame_length` from `SupportProtocols::max_frame_length()` (e.g., 4 MB for RelayV3, 2 MB for Sync). In `decode`, `self.length_delimited.decode(src)?` enforces this limit on the compressed wire bytes. After that, the only guard before allocating the decompression buffer is:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN { // 8 MB global constant
    return Err(...);
}
let mut buf = BytesMut::zeroed(decompressed_bytes_len);
```

`self.length_delimited.max_frame_length()` is accessible in `decode` (it is the same field used by `process`), but is never consulted. The encoder-side `process` correctly rejects data exceeding `max_frame_length`, but the decoder has no equivalent guard. A peer crafts a snappy payload whose compressed form is just under the wire limit but whose snappy header declares a decompressed size just under 8 MB. Steps: (1) compressed frame ≤ 4 MB passes `length_delimited.decode`; (2) `decompress_len` returns ~8 MB; (3) `> MAX_UNCOMPRESSED_LEN` check passes; (4) `BytesMut::zeroed(~8 MB)` is allocated; (5) 8 MB buffer is returned to the relay handler. The relay handler's rate limiter fires only after this allocation.

## Impact Explanation
This matches **High: Vulnerabilities which could easily crash a CKB node**. With the RelayV3 rate limit of 30 req/s per peer and a typical node maintaining dozens of peers, an attacker with multiple connections can sustain allocation rates of several GB/s of heap pressure. The allocation occurs unconditionally in the codec layer before any application-level rate limiting or message validation, making OOM-induced node crashes achievable with standard P2P connections and no special privileges.

## Likelihood Explanation
Any peer with a standard P2P connection can trigger this — no credentials, proof-of-work, or privileged role required. Crafting a snappy payload that compresses at better than 2:1 is trivial (e.g., a large buffer of repeated bytes). The path is reachable on mainnet by any node that connects to a victim. The attack is repeatable and can be sustained across multiple connections simultaneously.

## Recommendation
In `decode`, after the `decompress_len` call, add a check against the per-protocol limit before allocating:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN
    || decompressed_bytes_len > self.length_delimited.max_frame_length()
{
    return Err(io::ErrorKind::InvalidData.into());
}
```

This mirrors the existing guard in `process` and closes the asymmetry between encoder and decoder.

## Proof of Concept
```rust
// Unit test: RelayV3 codec, max_frame_length = 4 MB
let max = 4 * 1024 * 1024;
let mut codec = LengthDelimitedCodecWithCompress::new(
    true,
    length_delimited::Builder::new().max_frame_length(max).new_codec(),
    SupportProtocols::RelayV3.protocol_id(),
);

// 7 MB of zeros compresses well under 4 MB with snappy
let raw = vec![0u8; 7 * 1024 * 1024];
let compressed = snap::raw::Encoder::new().compress_vec(&raw).unwrap();
assert!(compressed.len() < max); // fits on the wire

// Manually build: 4-byte length prefix + COMPRESS_FLAG + compressed payload
let mut frame = BytesMut::new();
frame.put_uint((compressed.len() + 1) as u64, 4);
frame.put_u8(0b1000_0000); // COMPRESS_FLAG
frame.extend_from_slice(&compressed);

// Should return Err because decompressed_len (7 MB) > max_frame_length (4 MB)
// Currently returns Ok(7 MB buffer) — confirming the missing guard
let result = codec.decode(&mut frame);
assert!(result.is_err(), "expected Err: decompressed size exceeds max_frame_length");
```
