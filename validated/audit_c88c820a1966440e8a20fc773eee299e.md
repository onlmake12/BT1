### Title
Incorrect Loop Termination Counter in `check_and_record_ancestors` Causes Excess Transaction Eviction from Tx-Pool — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `tx-pool/src/component/pool_map.rs`, the function `check_and_record_ancestors` contains an eviction loop that removes `cell_ref_parent` transactions to bring a new transaction's ancestor count within `max_ancestors_count`. The loop decrements `ancestors_count` by exactly `1` per iteration, but `remove_entry_and_descendants` can remove **multiple ancestors** in a single call when a `cell_ref_parent` has another `cell_ref_parent` as a descendant. This causes `ancestors_count` to overestimate the real ancestor count, driving the loop to run extra iterations and evict legitimate pool transactions unnecessarily.

---

### Finding Description

The vulnerable code is in `check_and_record_ancestors`: [1](#0-0) 

```rust
let mut iter = evict_candidates.iter();
while ancestors_count > self.max_ancestors_count {
    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id);
        ancestors_count = ancestors_count.saturating_sub(1);   // ← BUG
        parents.remove(next_id);
        evicted.extend(removed);
    } else {
        break;
    }
}
```

`ancestors_count` is initialized as `ancestors.len() + 1`, where `ancestors` is the full transitive ancestor set of the incoming transaction `T`. [2](#0-1) 

`cell_ref_parents` are transactions already in the pool that hold a **cell dep** on one of `T`'s input outpoints. They are added to `parents` and therefore appear in `ancestors`. [3](#0-2) 

Two `cell_ref_parents` A and B can be in a parent–child relationship: both reference the same outpoint as a cell dep, and B additionally spends an output of A. In that case, `remove_entry_and_descendants(A)` removes **both A and B** from the pool in a single call, reducing the real ancestor count by 2. However, `ancestors_count` is only decremented by 1.

**Concrete over-eviction scenario** (evict order: A, C, B; initial `ancestors_count = max_ancestors_count + 2`):

| Iteration | Action | `ancestors_count` | Real ancestor count |
|---|---|---|---|
| 1 | Evict A → removes A + B | `max + 1` | `max` |
| 2 | Evict C (unnecessary!) | `max` | `max - 1` |
| 3 | Try B (already gone, no-op) | `max - 1` | `max - 1` |

C is a legitimate, unrelated pool transaction that is evicted solely because `ancestors_count` was stale by 1 after iteration 1.

---

### Impact Explanation

An unprivileged tx-pool submitter can cause legitimate third-party transactions to be silently evicted from the mempool. The evicted transactions receive a `Reject::Invalidated` callback and are removed from the pool without being mined. This is a **targeted mempool griefing / denial-of-service** against other users' pending transactions. The victim transactions must be resubmitted and may miss time-sensitive proposal windows.

---

### Likelihood Explanation

The trigger requires:
1. Two pool transactions A and B that both hold a cell dep on the same outpoint, where B also spends an output of A (a realistic pattern for script-library transactions).
2. A new transaction T that spends that same outpoint as an input.
3. The evict-key ordering places an unrelated `cell_ref_parent` C between A and B in the sorted candidate list.

All three conditions are reachable by an unprivileged `send_transaction` RPC caller with no special privileges. The attacker controls the construction of A, B, and T.

---

### Recommendation

Decrement `ancestors_count` by the actual number of removed entries that were present in the `ancestors` set, not a fixed `1`. One approach:

```rust
while ancestors_count > self.max_ancestors_count {
    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id);
        // Decrement by the count of removed entries that were ancestors of T
        let removed_ancestor_count = removed
            .iter()
            .filter(|e| ancestors.contains(&e.proposal_short_id()))
            .count();
        ancestors_count =
            ancestors_count.saturating_sub(removed_ancestor_count.max(1));
        parents.remove(next_id);
        evicted.extend(removed);
    } else {
        break;
    }
}
```

Alternatively, recompute `ancestors_count` from the live pool state after each eviction, analogous to the ERC20Gauges fix of using `userFreed` (the variable that tracks what was actually freed) rather than `totalFreed` (the variable that undercounts).

### Proof of Concept

**Setup:**
1. Submit tx `A` to the pool: spends some unrelated UTXO, has a **cell dep** on outpoint `X`.
2. Submit tx `B` to the pool: spends an output of `A` (making B a child of A), also has a **cell dep** on outpoint `X`.
3. Submit tx `C` to the pool: unrelated to A/B, but also has a **cell dep** on outpoint `X` (so it is also a `cell_ref_parent` of T). Give C a fee rate between A and B so it sorts between them in the evict order.
4. Ensure the pool has enough other transactions so that `ancestors_count` for the next step equals `max_ancestors_count + 2`.
5. Submit tx `T`: spends outpoint `X` as an **input**. This makes A, B, and C all `cell_ref_parents` of T.

**Expected:** Only A and B are evicted (they are the conflicting cell-dep holders; B is removed as a descendant of A).

**Actual:** `check_and_record_ancestors` evicts A (removing B as a side effect), then evicts C (unnecessarily), because `ancestors_count` was not decremented to account for B's implicit removal. [4](#0-3)
