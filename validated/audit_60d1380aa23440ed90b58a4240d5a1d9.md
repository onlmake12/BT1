Audit Report

## Title
Unbounded Attacker-Controlled `OFFSET` in `get_tx_with_cell` SQL Query Enables Indexer RPC DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
The `get_transactions` RPC handler decodes a 4-byte offset from the caller-supplied `after` cursor and passes it verbatim as a SQL `OFFSET` with no upper bound. Because the rich-indexer has no query timeout, a single crafted request encoding `i32::MAX` as the offset forces the database to scan and discard up to 2 billion rows, rendering the indexer RPC unresponsive for the duration of the scan.

## Finding Description
**Cursor parsing — no offset validation:**
`get_transactions` (ungrouped path) splits the 12-byte `after` blob and decodes the last 4 bytes via `decode_i32`: [1](#0-0) 

`decode_i32` only validates that the slice is exactly 4 bytes; it accepts every value in `[i32::MIN, i32::MAX]` without any range check: [2](#0-1) 

**Offset injected directly into SQL:**
`get_tx_with_cell` applies the decoded offset to the query builder with no cap, producing `OFFSET 2147483647` in the emitted SQL: [3](#0-2) 

**No timeout on rich-indexer SQL queries:**
`AsyncRichIndexerHandle` carries only `store`, `pool`, and `request_limit` — there is no `timeout_limit` field: [4](#0-3) 

The `request_limit` guard (L27-32 of `get_transactions.rs`) bounds `limit`, not `offset`, so it provides no protection against this attack vector: [5](#0-4) 

## Impact Explanation
The impact is a complete unavailability of the rich-indexer RPC service for the duration of the attack. On SQLite (the default backend), a single query with `OFFSET 2147483647` holds the single writer connection for the full scan, blocking every concurrent indexer read and write. On PostgreSQL, a small number of concurrent crafted requests exhausts the connection pool. In both cases the indexer RPC becomes unresponsive — effectively crashed — with no server-side timeout to bound the damage. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation
The `after` parameter is a plain `JsonBytes` field accepted over the public JSON-RPC interface with no authentication. The crafted cursor is a fixed 12-byte value requiring no chain state, PoW, or privileged access. Any caller who can reach the RPC port can trigger this. The RPC port defaults to localhost but is commonly exposed in production deployments.

## Recommendation
1. **Clamp the offset at parse time.** After `decode_i32`, reject any value that exceeds a small practical bound (e.g., the configured `request_limit`):
   ```rust
   if offset < 0 || offset as usize > self.request_limit {
       return Err(Error::Params("invalid cursor offset".to_string()));
   }
   ```
2. **Add a query timeout to the rich-indexer.** Thread a `timeout_limit` through `AsyncRichIndexerHandle` and wrap SQL futures with `tokio::time::timeout`, mirroring the `TimeoutIterator` pattern used in the regular indexer.
3. **Consider keyset pagination instead of OFFSET.** The `last` (tx_id) field already provides a keyset anchor; bounding the offset to the maximum number of cells per transaction (a small constant) eliminates the attack surface entirely.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {

            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
