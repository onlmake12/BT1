### Title
Gas Forwarding via Try-Catch Silently Swallows Out-of-Gas Callback Failures, Permanently Losing Consumer Requests — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` function permanently clears a request and credits provider fees **before** invoking the consumer callback inside a bare `try/catch`. Due to EVM's 63/64 gas forwarding rule, a caller can supply just enough gas to complete the pre-callback state changes while the callback itself silently runs out of gas. The request is irrecoverably deleted, the provider is paid, and the consumer never receives their price update — with no retry path.

---

### Finding Description

In `executeCallback`, the following sequence occurs:

1. Provider fees are credited to storage.
2. The request is permanently cleared via `clearRequest`.
3. `firstUnfulfilledSeq` is advanced.
4. The consumer callback is invoked inside a `try/catch` with a fixed gas cap.

```solidity
// Step 1-3: irreversible state changes
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // line 161-162

clearRequest(sequenceNumber);                        // line 164

while (...) { _state.firstUnfulfilledSeq++; }       // lines 169-174

// Step 4: callback — but request is already gone
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(...);
} catch {
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
``` [1](#0-0) 

The `try/catch` forwards exactly `req.callbackGasLimit` gas to `_echoCallback`. Per EVM rules, at most 63/64 of the **current context's** remaining gas is forwarded. If the caller provides a transaction gas limit such that only slightly more than `callbackGasLimit` gas remains at the point of the call, the callback receives fewer than `callbackGasLimit` gas, reverts with out-of-gas, and the catch block fires — emitting only an event.

Critically, there is **no retry mechanism**. Unlike `Entropy.sol`, which uses `excessivelySafeCall`, checks `(startingGas * 31) / 32 > gasLimit` to detect insufficient-gas scenarios, reverts with `InsufficientGas`, and maintains a `CALLBACK_FAILED` state for re-execution, `Echo.sol` has none of these safeguards. [2](#0-1) 

The developers themselves noted the risk in a TODO comment directly above the fee-crediting line:

> `// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract.` [3](#0-2) 

However, the comment addresses only the case where `executeCallback` itself reverts — not the silent out-of-gas case inside the try-catch.

---

### Impact Explanation

- The consumer's request is **permanently deleted** from storage (`clearRequest` runs before the callback). There is no `CALLBACK_FAILED` state and no re-execution path.
- The provider is **credited fees** regardless of whether the callback succeeded.
- The consumer **paid fees** for a price update they never received.
- `firstUnfulfilledSeq` is advanced past the sequence number, so the request is also invisible to off-chain keepers scanning for pending work.

The net effect: consumer funds are locked in the protocol (credited to the provider), and the consumer's application state is never updated with the requested price data. [4](#0-3) 

---

### Likelihood Explanation

`executeCallback` is a permissionless `external payable` function — anyone can call it, not just the assigned provider. [5](#0-4) 

An attacker (or a malicious/griefing provider) can:
1. Observe a pending request on-chain.
2. Call `executeCallback` with a transaction gas limit crafted so that, at the point of the `try` call, fewer than `(64/63) * callbackGasLimit` gas units remain.
3. The callback receives less than `callbackGasLimit` gas, reverts silently, and the request is gone.

The gas arithmetic is straightforward to compute off-chain. The cost to the attacker is only the gas for the transaction itself. The `callbackGasLimit` is stored on-chain in the `Request` struct and is publicly readable. [6](#0-5) 

---

### Recommendation

Apply the same pattern used in `Entropy.sol`:

1. Use `excessivelySafeCall` (or equivalent) instead of a bare `try/catch` with `{gas: ...}`.
2. After the call, check whether the calling context had sufficient gas to forward the required amount: `require((startingGas * 63) / 64 >= callbackGasLimit, "InsufficientGas")` — or use the 31/32 margin Entropy uses for safety.
3. Do **not** clear the request or credit fees before confirming the callback succeeded (or introduce a `CALLBACK_FAILED` state that allows re-execution, as Entropy does). [7](#0-6) 

---

### Proof of Concept

```solidity
// Attacker calls executeCallback with a crafted gas limit.
// Assume callbackGasLimit = 500_000 for the target request.
// Gas consumed before the try-call ≈ G_pre (measurable via simulation).
// At the try-call, remaining gas ≈ callbackGasLimit + epsilon.
// EVM forwards at most (callbackGasLimit + epsilon) * 63/64 < callbackGasLimit.
// _echoCallback reverts with out-of-gas.
// catch block fires, emits PriceUpdateCallbackFailed.
// clearRequest already ran — request is gone.
// provider.accruedFeesInWei already incremented — provider is paid.
// Consumer's echoCallback was never executed.

echoContract.executeCallback{gas: G_pre + callbackGasLimit + 1000}(
    providerAddress,
    targetSequenceNumber,
    updateData,
    priceIds
);
// Result: PriceUpdateCallbackFailed emitted, request permanently deleted,
// consumer receives no price update, provider keeps fees.
```

The `callbackGasLimit` value is stored in the `Request` struct and readable via `getRequest(sequenceNumber)`, making the required gas value trivially computable by any attacker. [8](#0-7)
