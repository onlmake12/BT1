### Title
Cross-Chain Signature Replay in `PythLazer.verifyUpdate` — No Chain ID or Contract Address Binding in Signed Payload - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary

The `verifyUpdate` function in `PythLazer.sol` verifies a Lazer price-update signature by hashing only the raw payload bytes (`keccak256(payload)`) and checking the recovered signer against the trusted-signer list. The signed payload contains no EVM chain ID, no contract address, and no network-specific domain separator. Because the same `PythLazer` contract is deployed at the same address (`0xACeA761c27A909d4D3895128EBe6370FDE2dF481`) on dozens of EVM chains, a valid signed update obtained on one chain is cryptographically valid on every other chain where the same trusted signer key is registered.

### Finding Description

In `PythLazer.sol`, `verifyUpdate` performs the following signature check:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
```

The `hash` is derived solely from the payload bytes. The payload structure (parsed by `PythLazerLib.parsePayloadHeader`) contains: a 4-byte magic, an 8-byte timestamp, a 1-byte channel, a 1-byte feed count, and then feed data. None of these fields encode the EVM chain ID or the contract address.

The contract is deployed at the same proxy address on all supported EVM chains (Arbitrum, Base, BSC, Polygon, Monad, Berachain, HyperEVM, etc.) and uses the same set of trusted Pyth Lazer signers across all of them.

**Attack path:**

1. Attacker observes a valid signed Lazer update on Chain A (e.g., Arbitrum). The update bytes are public — they are submitted in a transaction's calldata.
2. The attacker takes the identical `update` bytes and calls `verifyUpdate` on Chain B (e.g., BSC or Monad) with the required fee.
3. `verifyUpdate` on Chain B recovers the same signer address, finds it in the trusted-signer list, and returns `(payload, signer)` as valid.
4. Any consumer contract on Chain B that calls `verifyUpdate` and then acts on the returned payload (e.g., updating an on-chain price state) will accept a price update that was signed for Chain A, potentially at a stale timestamp or with a price that diverges from the current Chain B market state.

The root cause is the absence of any chain-binding domain separator in the signed data, directly analogous to the `P2pLendingProxy.isValidSignature` issue in the reference report.

### Impact Explanation

A consumer contract on Chain B that does not independently enforce strict timestamp freshness (the Cardano docs explicitly warn this is not enforced by the protocol) can have its price state updated with a replayed, cross-chain payload. This can cause:

- **Stale price injection**: A price signed for Chain A at time T can be replayed on Chain B at time T+Δ, bypassing any freshness check that relies on the Lazer timestamp alone (since the timestamp is inside the signed payload and is not chain-specific).
- **Price manipulation**: If prices diverge between chains (e.g., due to different liquidity or oracle latency), an attacker can replay a more favorable price from one chain onto another to manipulate DeFi protocols (lending, perpetuals) that consume Lazer prices.
- **Fee drain**: Each `verifyUpdate` call requires a fee payment. An attacker replaying updates forces consumer contracts to pay fees for updates they did not request.

Severity: **Medium** — requires no privileged access; any unprivileged transaction sender can replay a captured update across chains.

### Likelihood Explanation

- The same contract address and same trusted signers are used across 30+ EVM chains.
- Signed update bytes are fully public in transaction calldata on any block explorer.
- No nonce, sequence number, or chain-specific field prevents reuse.
- The attack requires only a fee payment (1 wei by default) and the ability to submit a transaction on the target chain.

### Recommendation

Include the EVM chain ID (and optionally the contract address) in the data that is signed by the Pyth Lazer trusted signer. This can be done by:

1. Prepending `block.chainid` and `address(this)` to the hash before signature verification:
   ```solidity
   bytes32 hash = keccak256(abi.encodePacked(block.chainid, address(this), payload));
   ```
2. Or, more robustly, adopting an EIP-712 domain separator that binds the signature to the specific chain and contract instance.

This change must be coordinated with the Lazer signing infrastructure so that signed payloads include the chain ID in the signed message.

### Proof of Concept

1. On Arbitrum (chain ID 42161), submit a transaction calling `PythLazer.verifyUpdate{value: 1 wei}(update_bytes)`. Observe it succeeds and emits the payload.
2. Take the identical `update_bytes` from Arbitrum calldata.
3. On BSC (chain ID 56), call `PythLazer.verifyUpdate{value: 1 wei}(update_bytes)` with the same bytes.
4. The call succeeds: `keccak256(payload)` is identical, the same trusted signer key is registered on BSC, and `ECDSA.tryRecover` returns the same signer address.
5. Any BSC consumer contract that calls `verifyUpdate` and trusts the returned payload will accept the Arbitrum-originated price as valid on BSC.
