### Title
Single Shared `priceLastUpdatedAt` Timestamp Allows Stale Individual Feed Prices to Pass Freshness Check - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getPricesNoOlderThan()` and `getEmaPricesNoOlderThan()` validate freshness using a single shared `status.priceLastUpdatedAt` timestamp — set to the **maximum** `publishTime` across all feeds in the batch — rather than checking each individual feed's own `publishTime`. This means a subscription containing a closed-market feed (whose `publishTime` may be hours old) will pass the staleness check as long as any one feed in the batch is fresh, silently returning arbitrarily stale prices for the closed-market feed.

---

### Finding Description

In `_validateShouldUpdatePrices`, the contract deliberately selects the **most recent** `publishTime` across all feeds as the representative update timestamp: [1](#0-0) 

This value is then stored as the single subscription-level `priceLastUpdatedAt`: [2](#0-1) 

When a reader calls `getPricesNoOlderThan`, the freshness check compares `block.timestamp` against this single shared `priceLastUpdatedAt`, not against each individual feed's stored `publishTime`: [3](#0-2) 

The same pattern applies to `getEmaPricesNoOlderThan`: [4](#0-3) 

The design explicitly allows individual feeds to have timestamps far older than `priceLastUpdatedAt`. The comment in `updatePriceFeeds` acknowledges this: [5](#0-4) 

The constant `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` bounds how old an individual feed's `publishTime` can be relative to the most recent feed: [6](#0-5) 

The test `testUpdatePriceFeedsSucceedsWithStaleFeedIfLatestIsValid` explicitly confirms that a feed with `publishTime` 1 hour 5 minutes ago (outside `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`) is stored alongside a fresh feed, with `priceLastUpdatedAt` set to the fresh feed's timestamp: [7](#0-6) 

---

### Impact Explanation

A consumer calling `getPricesNoOlderThan(subscriptionId, priceIds, age)` receives a guarantee that all returned prices are within `age` seconds. This guarantee is **false**: the check passes based on the most recent feed's timestamp, but individual feeds (e.g., closed-market assets) can have `publishTime` values up to `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` (1 hour) older than `priceLastUpdatedAt`. A DeFi protocol relying on this function for multi-asset pricing (e.g., a lending protocol checking collateral and debt prices together) may consume a stale price for one asset while believing all prices are fresh, enabling mispriced liquidations or undercollateralized borrows.

---

### Likelihood Explanation

This is reachable by any unprivileged keeper calling `updatePriceFeeds()`. The Scheduler is designed to accept closed-market feeds with old timestamps in the same batch as fresh feeds — this is not an edge case but an explicitly supported and tested scenario. Any subscription containing a mix of 24/7 crypto feeds and traditional-market feeds (equities, FX) will routinely trigger this condition during market-closed hours.

---

### Recommendation

In `getPricesNoOlderThan` and `getEmaPricesNoOlderThan`, check each individual feed's own `publishTime` against `age_seconds`, not the shared `priceLastUpdatedAt`. Replace the single subscription-level check with a per-feed check inside `_getPricesInternal`, or store and validate each feed's individual `publishTime` separately.

---

### Proof of Concept

1. A subscription is created with two price feeds: `ETH/USD` (24/7, updates every minute) and `AAPL/USD` (equity, market closed — last `publishTime` = 55 minutes ago).
2. A keeper calls `updatePriceFeeds()` with a batch where `ETH/USD.publishTime = now` and `AAPL/USD.publishTime = now - 55 minutes`. Both share the same Pythnet slot.
3. `_validateShouldUpdatePrices` sets `updateTimestamp = max(now, now-55min) = now`. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` check passes (55 min < 1 hour). `priceLastUpdatedAt = now`.
4. A lending protocol calls `getPricesNoOlderThan(subscriptionId, [ETH_ID, AAPL_ID], 300)` (5-minute max age).
5. The check: `distance(block.timestamp, priceLastUpdatedAt) = distance(now, now) = 0 ≤ 300` → **passes**.
6. `getPricesUnsafe` returns both feeds. `AAPL/USD` has `publishTime = now - 55 minutes`, which is 54 minutes and 55 seconds outside the caller's stated 5-minute freshness requirement — but no revert occurs.
7. The lending protocol uses the 55-minute-old `AAPL` price for a collateral calculation, potentially enabling an exploit if the price has moved significantly. [8](#0-7) [1](#0-0) [6](#0-5)

    function getPricesNoOlderThan(
        uint256 subscriptionId,
        bytes32[] calldata priceIds,
        uint256 age_seconds
    )
        external
        view
        override
        onlyWhitelistedReader(subscriptionId)
        returns (PythStructs.Price[] memory prices)
    {
        SchedulerStructs.SubscriptionStatus memory status = _state
            .subscriptionStatuses[subscriptionId];

        // Use distance (absolute difference) since pythnet timestamps
        // may be slightly ahead of this chain.
        if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
            revert PythErrors.StalePrice();

        prices = this.getPricesUnsafe(subscriptionId, priceIds);
    }
