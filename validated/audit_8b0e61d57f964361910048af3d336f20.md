### Title
Unauthenticated `from` Field in `ConnectionRequest` Enables Targeted Hole-Punch DoS via `pending_delivered` Poisoning — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

An unprivileged peer connected to node B can send a spoofed `ConnectionRequest` with an arbitrary `from` peer ID (e.g., peer A's ID), causing node B to insert a poisoned entry into `pending_delivered[A]`. Any subsequent legitimate `ConnectionRequest` from A arriving within the 2-minute `HOLE_PUNCHING_INTERVAL` is silently dropped with `StatusCode::Ignore`, permanently blocking the A→B hole punch for as long as the attacker refreshes the poison.

---

### Finding Description

The `ConnectionRequest` message schema contains a `from` field (peer ID of the initiating node) that is parsed from the raw message bytes but **never authenticated against the actual sender's session identity**.

In `execute()`, the `from` field is accepted as-is: [1](#0-0) 

No check is performed to verify that `content.from` matches the peer ID of the session that sent the message. The only guard before `respond_delivered` is called is the `forward_rate_limiter`, keyed by `(content.from, content.to, msg_item_id)`: [2](#0-1) 

This limiter allows **1 request per second** per `(from, to, item_id)` tuple — it does not prevent a single spoofed request from succeeding.

When `self_peer_id == &content.to` (i.e., the message has reached its destination, node B), `respond_delivered` is called with the attacker-supplied `from_peer_id`: [3](#0-2) 

Inside `respond_delivered`, the deduplication guard checks `pending_delivered` keyed by `from_peer_id`: [4](#0-3) 

If no entry exists yet, the function proceeds to send a `ConnectionRequestDelivered` response back to the attacker's session (line 226–232), then **writes the attacker-controlled listen_addrs and current timestamp into `pending_delivered[A]`**: [5](#0-4) 

When A's legitimate `ConnectionRequest` subsequently arrives at B, `respond_delivered` finds the poisoned entry, evaluates `now - t < HOLE_PUNCHING_INTERVAL` as true, and returns `Ignore` — silently discarding the real request without banning or alerting A. [6](#0-5) 

---

### Impact Explanation

- The legitimate hole punch from A to B is blocked for the full 2-minute `HOLE_PUNCHING_INTERVAL` per poisoning event.
- The attacker can re-send the spoofed request every ~2 minutes to maintain the DoS indefinitely at negligible cost (one P2P message per 2 minutes per targeted pair).
- The `pending_delivered` entry is only cleaned up after 5 minutes of inactivity (`TIMEOUT`), but the attacker refreshes it before that.
- `StatusCode::Ignore` (code 501) triggers only a warning log on B — no ban is issued against A or the attacker. [7](#0-6) 

---

### Likelihood Explanation

- **Precondition 1**: Attacker must be connected to node B — trivially satisfied for any peer on the CKB P2P network.
- **Precondition 2**: Attacker must know A's peer ID — trivially satisfied because `ConnectionRequest` messages are broadcast via gossip (`filter_broadcast`) and the `from` field is in plaintext. [8](#0-7) 

- **Cost**: One valid P2P message per 2 minutes per targeted pair. No PoW, no stake, no privileged access required.

---

### Recommendation

Authenticate the `from` field against the actual sender. When a `ConnectionRequest` arrives directly (not forwarded), verify that `content.from` matches the peer ID of the session that sent it (i.e., `context.session`'s peer ID). For forwarded messages, the `route` field already records intermediate hops, but the originating `from` should be validated at the first hop before forwarding. Alternatively, key `pending_delivered` on the actual sending session ID rather than the unverified `from` field.

---

### Proof of Concept

```
1. Attacker connects to node B.
2. Attacker observes a gossip-broadcast ConnectionRequest from A (from=A, to=B).
3. Attacker sends ConnectionRequest { from=A, to=B, listen_addrs=[<valid TCP/IP addr>], max_hops=0, route=[] } to B.
4. B: forward_rate_limiter passes (first request for (A,B,0)).
5. B: self_peer_id == content.to → calls respond_delivered(A, B, [attacker_addr]).
6. B: pending_delivered.get(&A) → None → proceeds.
7. B: sends ConnectionRequestDelivered back to attacker's session.
8. B: pending_delivered.insert(A, ([attacker_addr], now)).
9. A's real ConnectionRequest arrives at B within 2 minutes.
10. B: respond_delivered(A, B, [A_real_addrs]) → pending_delivered.get(&A) = Some(_, t) → now-t < 120000ms → returns Ignore.
11. A's hole punch is silently dropped. Attacker repeats step 3 every ~119 seconds to maintain indefinite DoS.
```
