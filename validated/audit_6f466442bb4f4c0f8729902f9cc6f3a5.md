Audit Report

## Title
`ScriptHashTypeVerifier` Omits Type Script Hash Type Consensus Check, Enabling Zero-Cost DoS via Forced Contextual Verification — (`File: verification/src/transaction_verifier.rs`)

## Summary
`ScriptHashTypeVerifier::verify` enforces the `ENABLED_SCRIPT_HASH_TYPE` consensus allowlist only against `output.lock().hash_type()`, never against `output.type_().hash_type()`. A transaction with `lock.hash_type = 0` (allowed) and `type_.hash_type = 6` (`Data3`, disallowed) passes all non-contextual checks and is only rejected during contextual verification, after cell-dep resolution and script-group construction. Since rejection precedes fee deduction, the attacker bears zero economic cost and can repeat the attack at high rate.

## Finding Description
**Root cause — lock-only check:**

`ScriptHashTypeVerifier::verify` iterates over outputs and validates only the lock script hash type against `ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}`. The type script is never read.

```rust

        } else { ... }
    }
    Ok(())

``` [1](#0-0) 

**`check_data()` does not compensate:**

`check_data()` calls `ScriptHashType::verify_value(v)` which accepts any `v` where `v % 2 == 0 || v == 1`. Since `Data3 = 6` is even, it passes. The consensus allowlist is enforced exclusively by `ScriptHashTypeVerifier`, and only for the lock script. [2](#0-1) [3](#0-2) 

**`ScriptHashType::try_from(6)` succeeds:**

The enum is generated via `seq!` macro with `Data~N = N << 1`, so `Data3 = 6` is a valid enum variant. `try_from(6)` returns `Ok(ScriptHashType::Data3)` (confirmed by unit test at L97). This means the `TryInto` in `verify()` succeeds for `hash_type = 6`, the allowlist check fires — but only for the lock script. [4](#0-3) [5](#0-4) 

**Contextual rejection via `select_version()`:**

When the type script with `hash_type = 6` reaches `TransactionScriptsVerifier`, `select_version()` falls into the catch-all arm and returns `Err(ScriptError::InvalidScriptHashType(...))`. This happens after cell-dep resolution and script-group construction — work that should have been avoided by the non-contextual gate. [6](#0-5) 

**Error messages confirm lock-only intent:**

Both error variants in `error.rs` reference "lock Script hash_type", confirming the type script was never considered in this verifier. [7](#0-6) 

**Existing test coverage confirms the gap:**

The only test for `ScriptHashTypeVerifier` (`test_not_enabled_hash_type_output_lock`) tests a disallowed lock hash type. There is no corresponding test for a disallowed type script hash type. [8](#0-7) 

## Impact Explanation
The non-contextual gate exists to cheaply reject invalid transactions before expensive processing. By bypassing it, an attacker forces every relaying/validating full node to perform contextual verification — cell-dep resolution, script-group construction, and `select_version()` dispatch — for transactions that should have been dropped at the first gate. Since rejection precedes fee deduction, the attacker bears zero economic cost. A sustained high-rate stream of such transactions degrades node throughput and can cause network congestion. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
The attack requires only the ability to submit transactions via the public RPC (`send_transaction`) or the P2P relay protocol — both reachable by any unprivileged peer. Constructing a transaction with `type_.hash_type = 6` is trivial. `ScriptHashType::try_from(6)` succeeds, `verify_value(6)` returns `true`, and the lock script can carry any allowed hash type (e.g., `0`). The attack is fully repeatable and automatable with no key material, mining power, or special access required.

## Recommendation
Extend `ScriptHashTypeVerifier::verify` to also validate the type script hash type for each output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }

        // NEW: type-script check (if present)
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

A corresponding new error variant (e.g., `InvalidTypeScriptHashType`) should be added to `util/types/src/core/error.rs` to distinguish type-script rejections from lock-script rejections. [7](#0-6)

     capacity = sufficient

2. Submit via JSON-RPC:
     curl -X POST http://localhost:8114 \
       -H 'Content-Type: application/json' \
       -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<tx>,"passthrough"]}'

3. Observe:
   - ScriptHashTypeVerifier::verify passes (only lock hash_type=0 is checked)

   - No fee is deducted
