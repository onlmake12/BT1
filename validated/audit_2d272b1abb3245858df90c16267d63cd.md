### Title
Admin Role Perpetuation via Unrestricted `proposeAdmin()` in `EntropyGovernance` and `SchedulerGovernance` - (File: `target_chains/ethereum/contracts/contracts/entropy/EntropyGovernance.sol`, `target_chains/ethereum/contracts/contracts/pulse/SchedulerGovernance.sol`)

---

### Summary

Both `EntropyGovernance` and `SchedulerGovernance` implement a two-role system (`owner` and `admin`). The `proposeAdmin()` function is callable by **either** the owner or the current admin. Because `proposeAdmin()` unconditionally overwrites any pending proposal, a malicious admin can always re-nominate a controlled address after the owner attempts to replace them, perpetuating admin access indefinitely.

---

### Finding Description

In `EntropyGovernance.sol`, `proposeAdmin()` is guarded by `_authoriseAdminAction()`:

```solidity
function proposeAdmin(address newAdmin) public virtual {
    require(newAdmin != address(0), "newAdmin is zero address");
    _authoriseAdminAction();          // passes for owner OR admin
    _state.proposedAdmin = newAdmin;
    emit NewAdminProposed(_state.admin, newAdmin);
}
```

The authorization check in `EntropyUpgradable.sol` is:

```solidity
function _authoriseAdminAction() internal view override {
    if (msg.sender != owner() && msg.sender != _state.admin)
        revert EntropyErrors.Unauthorized();
}
```

The identical pattern exists in `SchedulerGovernance.sol` / `SchedulerUpgradeable.sol`:

```solidity
function _authorizeAdminAction() internal view override {
    if (msg.sender != owner() && msg.sender != _state.admin)
        revert SchedulerErrors.Unauthorized();
}
```

Because `proposeAdmin()` simply overwrites `_state.proposedAdmin` with no restriction on who the caller nominates, the current admin can:

1. Nominate themselves directly (`proposeAdmin(address(this_admin))`), then call `acceptAdmin()` in the same block.
2. Nominate a fresh address they control, then call `acceptAdmin()` from that address.
3. Overwrite any pending proposal the owner has already set.

The admin role is never actually revoked until `acceptAdmin()` is called by the owner's chosen successor. During the entire window between `proposeAdmin()` and `acceptAdmin()`, the current admin remains active and can overwrite the pending proposal.

---

### Impact Explanation

**Entropy contract:** A malicious admin retains the ability to call `setPythFee`, `setDefaultProvider`, and `withdrawFee`. The `withdrawFee` function transfers accrued protocol fees to an arbitrary address, representing a direct financial loss to the protocol.

**Scheduler (Pulse) contract:** A malicious admin retains the ability to call `setSingleUpdateKeeperFeeInWei` and `setMinimumBalancePerFeed`, disrupting keeper economics and subscription viability.

In both cases, the owner's intent to revoke admin access is defeated without any on-chain mechanism to prevent it.

**Impact: High** — admin perpetuation enables fee theft (Entropy) and protocol parameter manipulation (Scheduler).

---

### Likelihood Explanation

**Likelihood: Low** — requires the current admin key to be compromised or the admin to act maliciously. However, the admin is a lower-trust role than the owner (e.g., an ops key), making key compromise a realistic scenario. The attack requires no special conditions beyond the admin being active at the time the owner initiates a replacement.

---

### Recommendation

Restrict `proposeAdmin()` so that only the `owner` can call it. The admin role should not be able to nominate successors:

```solidity
function proposeAdmin(address newAdmin) public virtual onlyOwner {
    require(newAdmin != address(0), "newAdmin is zero address");
    _state.proposedAdmin = newAdmin;
    emit NewAdminProposed(_state.admin, newAdmin);
}
```

This mirrors the pattern used for `_authorizeUpgrade`, which is already correctly restricted to `onlyOwner` in both `EntropyUpgradable` and `SchedulerUpgradeable`.

---

### Proof of Concept

**Entropy scenario:**

1. Owner calls `proposeAdmin(trustedNewAdmin)` to replace a compromised admin.
2. Compromised admin observes the pending transaction and calls `proposeAdmin(attackerAddress)` — this overwrites `_state.proposedAdmin`.
3. `attackerAddress` calls `acceptAdmin()`, becoming the new admin.
4. Attacker calls `withdrawFee(attackerAddress, accruedFees)`, draining protocol fees.
5. Owner's `trustedNewAdmin` can no longer call `acceptAdmin()` (proposal was overwritten).

**Scheduler scenario:**

1. Owner calls `proposeAdmin(trustedNewAdmin)`.
2. Compromised admin calls `proposeAdmin(attackerAddress)`, overwriting the proposal.
3. `attackerAddress` calls `acceptAdmin()`.
4. Attacker calls `setMinimumBalancePerFeed(0)` or `setSingleUpdateKeeperFeeInWei(type(uint128).max)`, breaking keeper incentives for all active subscriptions.
