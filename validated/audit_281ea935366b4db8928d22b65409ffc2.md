### Title
`check_tx_fee` Uses `tx_size` Instead of `get_transaction_weight(tx_size, cycles)` for Minimum Fee Enforcement — (`File: tx-pool/src/util.rs`)

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the serialized transaction size as the weight denominator, explicitly ignoring the cycles dimension. The correct weight formula is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For compute-heavy transactions, this produces a minimum fee threshold that is orders of magnitude lower than intended, allowing any unprivileged RPC caller to inject high-cycle, low-fee transactions into the tx-pool.

### Finding Description

`FeeRate` in CKB is defined as shannons per kilo-**weight**, where weight accounts for both the serialized byte size and the VM cycles consumed:

```rust
// util/types/src/core/tx_pool.rs
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [1](#0-0) 

Every other fee-rate computation in the codebase — `TxEntry::fee_rate()`, `AncestorsScoreSortKey`, `EvictKey`, `FeeRateCollector::statistics()`, and the fee estimator — calls `get_transaction_weight(size, cycles)`:

```rust
// tx-pool/src/component/entry.rs
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [2](#0-1) 

However, the **admission gate** `check_tx_fee` uses only `tx_size` as the weight, with an explicit comment acknowledging the discrepancy:

```rust
// tx-pool/src/util.rs
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [3](#0-2) 

This is the **only** minimum-fee-rate enforcement point for tx-pool admission. There is no subsequent check after `verify_rtx` completes and the actual cycles are known.

### Impact Explanation

For a transaction with small serialized size `S` bytes and high cycles `C`:

- **Correct minimum fee**: `min_fee_rate × max(S, C × 0.000_170_571_4) / 1000`
- **Enforced minimum fee**: `min_fee_rate × S / 1000`

With the default `max_tx_verify_cycles = TWO_IN_TWO_OUT_CYCLES × 20 ≈ 3.5 × 10⁹` cycles and a compact transaction of ~200 bytes:

- Correct weight ≈ `max(200, 3.5×10⁹ × 0.000_170_571_4)` ≈ **597,000**
- Enforced weight = **200**

The enforced minimum fee is ~**2,985×** lower than the correct minimum. An attacker can flood the tx-pool with maximally compute-heavy transactions at a tiny fraction of the intended minimum fee, consuming disproportionate block-cycle budget relative to fee paid.

### Likelihood Explanation

Any unprivileged user with RPC access to `send_transaction` can exploit this. No special privileges, keys, or majority hashpower are required. The attacker only needs to craft a transaction with a large script that consumes many cycles but has a small serialized body — a standard pattern for CKB scripts. The default `min_fee_rate` of 1000 shannons/KW means the attacker pays ~0.2 shannons per transaction instead of ~597 shannons, making mass submission economically trivial. [4](#0-3) 

### Recommendation

Replace the size-only weight with the correct two-dimensional weight in `check_tx_fee`. Since cycles are not yet known at the pre-verification call site, the check should either:

1. Be deferred until after `verify_rtx` returns the actual cycles, then recheck using `get_transaction_weight(tx_size, cycles)`, or
2. Use a declared/estimated cycles value (already present in the tx submission path for cached entries) to compute the correct weight before admission.

The corrected check:

```rust
pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
    cycles: Cycle,          // pass actual cycles after verify_rtx
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(...).transaction_fee(rtx)?;
    let weight = get_transaction_weight(tx_size, cycles);
    let min_fee = tx_pool.config.min_fee_rate.fee(weight);
    if fee < min_fee {
        return Err(Reject::LowFeeRate(...));
    }
    Ok(fee)
}
```

### Proof of Concept

1. Craft a CKB transaction with a lock script that loops for ~3.5 billion cycles but whose serialized size is ~200 bytes.
2. Set the transaction fee to `min_fee_rate × 200 / 1000` shannons (e.g., 0 shannons if `min_fee_rate = 1000` and size < 1000 bytes, or a trivially small amount).
3. Submit via `send_transaction` RPC.
4. Observe the transaction is accepted into the tx-pool despite its effective fee rate (by weight) being ~2,985× below `min_fee_rate`.
5. Repeat to fill the tx-pool with high-cycle, near-zero-fee transactions, starving legitimate transactions and degrading miner revenue.
