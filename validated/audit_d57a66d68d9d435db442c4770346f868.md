### Title
Broken `request_limit` Guard Enables Unbounded RocksDB Iteration via `get_cells`/`get_transactions` RPC — (`util/indexer/src/service.rs`)

### Summary

When `request_limit` is not set in `ckb.toml`, `IndexerService::new` assigns it `usize::MAX`. The guard that is supposed to reject oversized `limit` values — `limit > self.request_limit` — becomes a tautologically false comparison for any `usize` value, so it never fires. Any caller can pass `limit = 0xFFFFFFFF` and the only remaining bound is the 10-second `TimeoutIterator`. Within that window the node iterates RocksDB, performs a secondary key lookup per cell, and accumulates all results in a heap-allocated `Vec` before serialising the response.

---

### Finding Description

**Root cause 1 — always-`usize::MAX` default:** [1](#0-0) 

`config.request_limit.unwrap_or(usize::MAX)` stores `usize::MAX` (18 446 744 073 709 551 615 on 64-bit) when the operator has not set the field.

**Root cause 2 — guard is always false:** [2](#0-1) 

`limit` is widened from `u32` to `usize` at line 212. The subsequent check `limit > self.request_limit` evaluates to `(u32_value as usize) > usize::MAX`, which is **always false** on any 64-bit platform. The guard never rejects any caller-supplied value. The identical dead guard exists in `get_transactions`: [3](#0-2) 

**Root cause 3 — `.take(limit)` with limit = 4 294 967 295:** [4](#0-3) [5](#0-4) 

The iterator chain calls `.take(4_294_967_295)`. For each item that passes `filter_map`, a secondary RocksDB point-lookup is issued (`snapshot.get(Key::OutPoint(...))`), and the resulting `IndexerCell` struct is pushed onto a heap `Vec`. All of this happens synchronously inside the RPC handler thread.

**The only real bound — `TimeoutIterator`:** [6](#0-5) 

`TimeoutIterator::next()` checks elapsed time before each item. With the default 10-second timeout, iteration stops after ~10 s. This prevents infinite execution but does **not** prevent large memory allocation: on a node with millions of indexed cells, a fast NVMe-backed RocksDB can yield hundreds of thousands of items in 10 seconds, each carrying serialised script, capacity, and output-data fields. The entire `Vec` is held in memory until the JSON response is written.

The `group_by_transaction` path in `get_transactions` has the same flaw — the `tx_with_cells.len() == limit` break condition at line 480 also never triggers when `limit = usize::MAX`: [7](#0-6) 

---

### Impact Explanation

A single unauthenticated RPC call with `limit=0xFFFFFFFF` against a well-indexed node (millions of cells under one script prefix) will:

1. Saturate one CPU core for up to 10 seconds doing RocksDB iteration + point-lookups.
2. Allocate a `Vec<IndexerCell>` whose size is bounded only by how many items RocksDB can yield in 10 seconds — potentially hundreds of MB to several GB on fast storage.
3. Serialize the entire result to JSON before returning, doubling peak memory.

Sending a modest number of concurrent such requests (e.g., 8–16) multiplies the memory pressure proportionally, making OOM-kill realistic on nodes with ≤16 GB RAM.

---

### Likelihood Explanation

The indexer RPC is enabled by default in many CKB node deployments and is publicly reachable. The attacker needs no credentials, no PoW, and no on-chain state — only a valid JSON-RPC call with a large `limit` field. The precondition (no `request_limit` in `ckb.toml`) is the default out-of-the-box configuration.

---

### Recommendation

1. **Change the default** from `usize::MAX` to a safe upper bound (e.g., `4096` or `10_000`):
   ```rust
   request_limit: config.request_limit.unwrap_or(4096),
   ```
2. **Fix the guard direction** — the check should be `limit > self.request_limit` where `request_limit` is a meaningful finite value, which the fix above provides.
3. Optionally add a hard compile-time cap (e.g., `min(limit, 10_000)`) so that even an operator who explicitly sets `request_limit = usize::MAX` cannot accidentally re-open the path.

---

### Proof of Concept

```bash
# Start a default CKB node (no request_limit in ckb.toml) with indexer enabled.
# Ensure the indexer has indexed at least ~100k cells (any active mainnet/testnet node).

curl -s -X POST http://127.0.0.1:8114/ \
  -H 'Content-Type: application/json' \
  -d '{
    "id": 1,
    "jsonrpc": "2.0",
    "method": "get_cells",
    "params": [
      {
        "script": {
          "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
          "hash_type": "type",
          "args": "0x"
        },
        "script_type": "lock"
      },
      "asc",
      "0xffffffff",
      null
    ]
  }'

# Expected (vulnerable): call runs for ~10 s, returns a massive JSON payload,
# and node RSS grows by hundreds of MB.
# Expected (patched): call is rejected immediately with
# "limit must be less than 4096".
```
