### Title
Unbounded `client_version` String in Identify Protocol Stored Per-Peer Without Length Validation — (`network/src/protocols/identify/mod.rs`)

---

### Summary

`Identify::verify` parses the remote peer's `client_version` field from a molecule-encoded Identify message and returns it as an owned `String` with no length check. `IdentifyCallback::received_identify` stores this string verbatim into `peer.identify_info.client_version`. An unprivileged remote attacker can connect as an inbound peer, send a crafted Identify message with an arbitrarily large `client_version`, and cause the victim node to allocate attacker-controlled memory proportional to `max_inbound × attacker_string_size`.

---

### Finding Description

**Entry point — `Identify::verify`:** [1](#0-0) 

At line 558, `reader.client_version().as_utf8().ok()?.to_owned()` converts the raw molecule bytes into an owned `String`. There is no length guard before or after this call. The molecule `IdentifyReader::from_slice` succeeds for any valid encoding regardless of field size.

**Storage — `IdentifyCallback::received_identify`:** [2](#0-1) 

The full, unbounded `client_version` string is passed directly to `registry_client_version`, which writes it into `peer.identify_info`.

**Storage struct — `PeerIdentifyInfo`:** [3](#0-2) 

`client_version` is a plain `String` with no capacity bound.

**Peer registry — `max_inbound` cap:** [4](#0-3) 

The registry enforces a `max_inbound` ceiling on the number of inbound peers, but places no constraint on per-peer memory. Each accepted inbound peer can independently contribute an arbitrarily large `client_version` allocation.

No application-level message size limit exists anywhere in the identify protocol path. No `MAX_CLIENT_VERSION` constant, no `.len()` check, and no truncation is present in the codebase.

---

### Impact Explanation

Each inbound peer connection can force the victim node to allocate an attacker-chosen amount of heap memory. With a typical `max_inbound` of 125 and a `client_version` of 1 MB per peer, total forced allocation is ~125 MB. With larger strings (bounded only by the transport frame limit of the underlying `tentacle`/`p2p` library, typically 8–24 MB), the total reaches 1–3 GB, sufficient to trigger OOM on a standard node. This degrades or crashes the node, causing network congestion and availability loss proportional to the number of attackers or connections.

---

### Likelihood Explanation

The attack requires only the ability to open a TCP connection to the victim's P2P port (default 8115), which is publicly reachable on any full node. No authentication, no PoW, no key material, and no privileged access is needed. The attacker sends a single Identify message per connection. The `check_duplicate` guard only prevents a second Identify on the same session; it does not limit string size. [5](#0-4) 

---

### Recommendation

Add a maximum length constant and enforce it in `Identify::verify` before allocating the owned `String`:

```rust
const MAX_CLIENT_VERSION_LEN: usize = 256;

fn verify(&self, data: &[u8]) -> Option<(Flags, String)> {
    let reader = packed::IdentifyReader::from_slice(data).ok()?;
    let name = reader.name().as_utf8().ok()?.to_owned();
    if self.name != name { return None; }
    let flag: u64 = reader.flag().into();
    if flag == 0 { return None; }
    let raw = reader.client_version().as_utf8().ok()?;
    if raw.len() > MAX_CLIENT_VERSION_LEN { return None; }  // <-- guard
    Some((Flags::from_bits_truncate(flag), raw.to_owned()))
}
```

Returning `None` here causes `received_identify` to ban and disconnect the peer, which is the correct response.

---

### Proof of Concept

1. Build a minimal peer that speaks the CKB identify protocol.
2. Construct a molecule-encoded `Identify` message with `name` matching the target network (e.g., `"ckb"`), a valid non-zero `flag`, and `client_version` = `"A".repeat(1_000_000)`.
3. Connect to the victim node's P2P port and send the message immediately after the session opens.
4. Observe via `/proc/<pid>/status` (`VmRSS`) or a heap profiler that the victim process has allocated ~1 MB for that peer's `identify_info.client_version`.
5. Repeat with 125 simultaneous connections; measure ~125 MB of forced heap growth with zero PoW cost.
