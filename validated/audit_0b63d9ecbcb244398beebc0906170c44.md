Audit Report

## Title
Compact Block Relay Accepts Zero-Cost PoW Blocks with Attacker-Chosen `compact_target`, Causing Unguarded State Mutations Before Contextual Rejection — (`sync/src/relayer/compact_block_process.rs`)

## Summary

`HeaderVerifier::verify()`, used during compact block relay, checks PoW only against the block's own `compact_target` field without verifying that `compact_target` matches the epoch's expected difficulty. The `compact_target` consistency check exists only in `ContextualBlockVerifier::EpochVerifier::verify()`, which runs during full block acceptance — after `insert_valid_header` (HeaderMap mutation) and `pending_compact_blocks` insertion have already occurred. CompactBlock messages explicitly bypass the per-peer rate limiter, making the attack unbounded.

## Finding Description

**Execution flow in `CompactBlockProcess::execute()`:**

1. `non_contextual_check()` checks uncle/proposal counts and block height. No `compact_target` check. [1](#0-0) 

2. `contextual_check()` calls `HeaderVerifier::verify()`, which runs `PowVerifier`, `NumberVerifier`, `EpochVerifier`, and `TimestampVerifier`. [2](#0-1) [3](#0-2) 

3. `EpochVerifier::verify()` in `header_verifier.rs` only checks `is_well_formed()` and `is_successor_of(parent_epoch)`. There is **no `compact_target` consistency check** against the epoch's expected difficulty. [4](#0-3) 

4. `PowVerifier` calls `EaglesongPowEngine::verify()`, which computes `compact_to_target(header.compact_target)` and only rejects if the result is zero or overflows U256. With `compact_target = 0x207fffff` (exponent 32, mantissa 0x7fffff), the target is `0x7fffff << 232` ≈ 2^255 — non-zero, no overflow. Roughly 50% of all eaglesong outputs satisfy this, meaning any nonce passes at ~2 hash attempts (effectively zero real cost). [5](#0-4) 

5. After `contextual_check()` passes, `insert_valid_header()` is called — HeaderMap mutation (in-memory LRU + sled backend). [6](#0-5) [7](#0-6) 

6. If block reconstruction has missing transactions, `missing_or_collided_post_process()` is called — `pending_compact_blocks` insertion + `GetBlockTransactions` sent to the attacker. [8](#0-7) [9](#0-8) 

7. Only when a full block is later accepted does `ContextualBlockVerifier::EpochVerifier::verify()` run the `compact_target` consistency check. At that point the block is marked `BLOCK_INVALID`, but all state mutations from steps 5–6 have already occurred and are not rolled back. [10](#0-9) 

8. `pending_compact_blocks` cleanup is triggered only by successful block reconstruction/acceptance (epoch-number-based `retain`). The `unix_time_as_millis()` timestamp stored at insertion is never used for timeout-based eviction. Entries for blocks whose transactions are never delivered persist until a legitimate block from a newer epoch is accepted (epochs last ~4 hours). [11](#0-10) 

9. Rate limiting is explicitly disabled for `CompactBlock` messages. [12](#0-11) 

The `BLOCK_INVALID` early-exit in `contextual_check()` does not mitigate the attack because the attacker generates a new block hash per iteration by varying the nonce or timestamp, bypassing the cached status check.

## Impact Explanation

An attacker can drive unbounded growth of the `HeaderMap` (memory + sled disk store) and `pending_compact_blocks` map, and generate amplified outbound `GetBlockTransactions` traffic — all at effectively zero PoW cost and with no rate limit. Sustained attack causes memory exhaustion and node crash (**High: Vulnerabilities which could easily crash a CKB node**) and/or saturates outbound bandwidth (**High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**).

## Likelihood Explanation

The attack requires only a P2P connection to the target node and knowledge of the current chain tip (publicly available via RPC or peer gossip). No hashpower, no privileged access, no key material. The PoW cost is ~2 eaglesong hash computations per block (trivial). The attack is locally testable, repeatable at maximum network throughput, and can be executed by any unprivileged external user.

## Recommendation

Move the `compact_target` epoch consistency check into the relay-time `HeaderVerifier`, or add it as an explicit check in `contextual_check()` before `insert_valid_header` is called. After verifying the parent exists, compute the expected `compact_target` for the block's epoch from the parent's `EpochExt` and reject the compact block if it does not match — mirroring the check already present in `ContextualBlockVerifier::EpochVerifier::verify()`. Additionally, apply a per-peer token bucket rate limit to `CompactBlock` messages, or at minimum ban peers that repeatedly send compact blocks with invalid headers. Add timeout-based eviction to `pending_compact_blocks` using the stored `unix_time_as_millis()` timestamp.

## Proof of Concept

```
1. Connect to a target CKB node via P2P (RelayV3 protocol).
2. Observe the current tip header to obtain: parent_hash, parent_epoch, block_number.
3. Construct a RawHeader with:
     - parent_hash    = tip.hash
     - number         = tip.number + 1
     - epoch          = valid successor of tip.epoch (correct epoch number, any fraction)
     - compact_target = 0x207fffff   ← attacker-chosen, minimum difficulty
     - timestamp      = tip.timestamp + 1 (within allowed window)
4. Compute calc_pow_hash over the RawHeader (includes compact_target).
5. Build pow_message = pow_hash(32 bytes) || nonce_le(16 bytes) with nonce = 0.
   eaglesong(pow_message) will be ≤ compact_to_target(0x207fffff) ≈ 2^255 with ~50%
   probability; try nonce = 0 and nonce = 1 — expected ~2 attempts.
6. Wrap in a CompactBlock with one short_id referencing a tx not in the target's tx-pool
   (ensures ReconstructionResult::Missing path).
7. Send the CompactBlock message. Never respond to the GetBlockTransactions reply.

Expected result per iteration:
  - Target node calls insert_valid_header → HeaderMap insertion (memory + sled).
  - Target node inserts into pending_compact_blocks (never evicted until next epoch boundary).
  - Target node sends GetBlockTransactions back to attacker (wasted outbound bandwidth).
  - Repeat from step 3 with a new nonce/timestamp; no rate limit applies.
  - Each iteration costs the attacker ~2 eaglesong hashes; each costs the target one
    HeaderMap entry, one pending_compact_blocks entry, and one outbound message.
  - Sustained attack exhausts memory and crashes the node.
```
