### Title
`DownloadScheduler::timeout_count` Accumulator Never Accumulates — Sync Download Capacity Degrades 3× Faster Than Intended - (File: sync/src/types/mod.rs)

### Summary

`DownloadScheduler::decrease()` contains a copy-paste bug: it assigns `self.task_count.saturating_add(num)` to `self.timeout_count` instead of `self.timeout_count.saturating_add(num)`. This means the timeout accumulator is never actually accumulated — it is reset to `task_count + num` on every call. As a result, every slow block response immediately decrements `task_count` (the peer's download concurrency budget), rather than only after 2+ accumulated timeouts as intended. A sync peer that deliberately delays block responses can exhaust a node's per-peer download capacity ~3× faster than the protocol intends.

### Finding Description

`DownloadScheduler` tracks how many concurrent block downloads a peer is allowed. `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32) and is adjusted based on block delivery latency. The `decrease` function is supposed to accumulate timeout events and only reduce `task_count` after more than 2 have been accumulated:

```rust
// sync/src/types/mod.rs lines 521-527
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num); // BUG: should be self.timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
``` [1](#0-0) 

Because `self.task_count` (not `self.timeout_count`) is used on the right-hand side, `timeout_count` is always set to `task_count + num` — a value that is always `> 2` as long as `task_count ≥ 2`. So the `if` branch fires on every single call, decrementing `task_count` by 1 and resetting `timeout_count` to 0 each time.

`decrease` is called from `remove_by_block` when a block arrives in the `NormalToUpper` or `UpperToMax` time quantile and `should_punish` is true (i.e., the node has more peers than `protect_num`):

```rust
// sync/src/types/mod.rs lines 799-811
TimeQuantile::NormalToUpper => {
    if should_punish {
        set.decrease(1)
    }
}
TimeQuantile::UpperToMax => {
    if should_punish {
        set.decrease(2)
    }
}
``` [2](#0-1) 

The `should_punish` flag is set when the number of connected peers exceeds `protect_num` (`MAX_OUTBOUND_PEERS_TO_PROTECT_FROM_DISCONNECT`): [3](#0-2) 

### Impact Explanation

**Intended behavior**: `timeout_count` accumulates across calls; `task_count` is decremented only after `timeout_count > 2` (i.e., roughly every 3 slow blocks).

**Actual behavior**: `task_count` is decremented on every single slow block response. Starting from 32, after 32 slow blocks the peer's download budget reaches 0 — the node stops fetching blocks from that peer entirely. Under the correct logic this would require ~96 slow blocks.

The `task_count` reaching 0 means `can_fetch()` returns 0:

```rust
fn can_fetch(&self) -> usize {
    self.task_count.saturating_sub(self.hashes.len())
}
``` [4](#0-3) 

This causes the block fetcher to skip that peer entirely during IBD or normal sync, degrading sync throughput. If a node has few peers and a malicious one deliberately delays responses, the node's effective sync bandwidth is reduced.

### Likelihood Explanation

The condition is met during normal operation whenever:
1. The node has more than `protect_num` connected peers (typical during IBD).
2. Any peer delivers blocks in the `NormalToUpper` or `UpperToMax` latency bucket.
3. `adjustment` is `true` (the default).

A malicious sync peer reachable over the P2P network can deliberately delay block responses to trigger this path repeatedly, exhausting its own `task_count` budget faster than intended. No special privileges are required — any peer that the node connects to can do this.

### Recommendation

Fix the assignment in `DownloadScheduler::decrease` to accumulate into `self.timeout_count` rather than overwrite it with `self.task_count`:

```rust
fn decrease(&mut self, num: usize) {
    // Fix: accumulate into timeout_count, not task_count
    self.timeout_count = self.timeout_count.saturating_add(num);
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

### Proof of Concept

Given `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32` (confirmed by the test asserting `32 >> 4` after `punish(4)`): [5](#0-4) 

**With the bug** — call `decrease(1)` 32 times:
- Call 1: `timeout_count = 32 + 1 = 33 > 2` → `task_count = 31`, `timeout_count = 0`
- Call 2: `timeout_count = 31 + 1 = 32 > 2` → `task_count = 30`, `timeout_count = 0`
- …
- Call 32: `timeout_count = 1 + 1 = 2`, not `> 2` → `task_count = 0` (after call 31 sets it to 1)

After ~32 slow blocks, `task_count = 0`. `peer_can_fetch_count` returns 0. The node stops requesting blocks from this peer.

**Without the bug** — call `decrease(1)` 32 times:
- Calls 1–2: `timeout_count` accumulates to 2, no decrement
- Call 3: `timeout_count = 3 > 2` → `task_count = 31`, `timeout_count = 0`
- …
- `task_count` reaches 0 only after ~96 slow blocks.

The attacker-controlled entry path: unprivileged sync peer → deliberately slow block responses → `remove_by_block` with high elapsed time → `decrease(1)` or `decrease(2)` → `task_count` drained 3× faster than intended → node stops fetching from that peer. [6](#0-5)
