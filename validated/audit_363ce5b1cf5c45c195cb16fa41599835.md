### Title
Unbounded `while` Loop in `executeCallback` Enables Gas-Exhaustion DOS - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past all consecutive inactive (fulfilled) requests. An attacker can engineer a sequence of N fulfilled requests immediately following a "stuck" active request, so that when the stuck request is finally fulfilled, the loop iterates N times — each iteration performing a cold storage read — exhausting the block gas limit and permanently preventing that request from ever being fulfilled.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs the following loop:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs up to two storage reads — one against the fixed `requests[shortKey]` array and one against the `requestsOverflow` mapping:

```solidity
req = _state.requests[shortKey];
if (req.sequenceNumber == sequenceNumber) {
    return req;
} else {
    req = _state.requestsOverflow[key];
}
``` [2](#0-1) 

The `requestsOverflow` mapping uses a unique `keccak256`-derived key per sequence number, so every lookup for a distinct fulfilled sequence number is a **cold SLOAD (2,100 gas)**. The developers themselves flagged this:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
``` [3](#0-2) 

The `State` struct stores `firstUnfulfilledSeq` as a `uint64` with no upper bound on how far it can lag behind `currentSequenceNumber`: [4](#0-3) 

**Attack construction:**

1. Attacker submits request at sequence number `k` (e.g., with a `publishTime` for which the provider cannot immediately source valid Pyth price data, or with a callback contract that always reverts — the callback failure is caught and does not prevent the request from being cleared, but the provider may deprioritize it).
2. Attacker submits N additional requests at sequences `k+1` through `k+N`.
3. Provider fulfills requests `k+1` through `k+N` (skipping `k`). Each fulfillment's `while` loop stops immediately at `firstUnfulfilledSeq = k` because request `k` is still active — so no gas accumulation yet.
4. Provider attempts to fulfill request `k`. `clearRequest(k)` executes, then the `while` loop starts at `k` and must scan through all N now-inactive entries `k+1 … k+N` before stopping at `currentSequenceNumber`. Each of the N iterations costs ≥ 2,100 gas (cold SLOAD into `requestsOverflow`).
5. At N ≈ 14,300 iterations: `14,300 × 2,100 ≈ 30,030,000 gas` — exceeding the Ethereum block gas limit. The transaction reverts (all state changes including `clearRequest` are rolled back). Request `k` remains active but can never be fulfilled.

The `requestPriceUpdatesWithCallback` function has no cap on the total number of outstanding requests, only a per-request `MAX_PRICE_IDS` limit: [5](#0-4) 

---

### Impact Explanation

- **Permanent fund lock:** The requester's fee for request `k` is locked in the contract forever, since `executeCallback` for sequence `k` will always revert out-of-gas once N is large enough.
- **Partial DOS:** `firstUnfulfilledSeq` is permanently stuck at `k`. While future requests can still be created and fulfilled (the loop stops at `k` immediately since it remains active), the stuck request and its locked funds are irrecoverable without an admin escape hatch — which does not exist in the current contract.
- **Provider griefing:** A malicious requester can force the provider's fulfillment transaction for a specific sequence number to always fail, denying the provider their earned fee for that request.

---

### Likelihood Explanation

The attack is reachable by any unprivileged user who can call `requestPriceUpdatesWithCallback`. The attacker pays fees for N requests but can recover most of them if they also act as the provider (providers can register permissionlessly via `registerProvider`). The net cost is primarily gas. The condition of having the provider skip request `k` is realistic when:

- The attacker sets `publishTime` to a timestamp for which the provider's off-chain Pyth data feed has a gap.
- The attacker's callback contract is designed to consume exactly `callbackGasLimit` gas, making the provider deprioritize it.
- The provider's keeper naturally processes requests by fee or recency rather than strict sequence order.

The `IEcho.sol` documentation explicitly acknowledges unbounded gas scaling for `getFirstActiveRequests`, confirming the developers are aware of the linear-scan pattern: [6](#0-5) 

---

### Recommendation

1. **Replace the `while` loop with a doubly-linked list** of active requests (as the TODO comment already suggests), so advancing `firstUnfulfilledSeq` is O(1).
2. **Alternatively, add an admin-callable `setFirstUnfulfilledSeq(uint64 seq)` function** (analogous to the Opyn fix) that allows the admin to skip past a stuck sequence number if the loop would otherwise exceed the gas limit.
3. **Cap the total number of outstanding requests** per provider or globally, so the gap between `firstUnfulfilledSeq` and `currentSequenceNumber` is bounded.

---

### Proof of Concept

```solidity
// Attacker registers as provider and requester
echo.registerProvider(minBase, minPerFeed, minPerGas);

// Step 1: Submit request at seq k=1 with a callback that always reverts
// (provider will deprioritize it)
uint64 seqK = echo.requestPriceUpdatesWithCallback{value: fee}(
    attackerProvider, block.timestamp, priceIds, callbackGasLimit
);

// Step 2: Submit N=15000 more requests
for (uint i = 0; i < 15000; i++) {
    echo.requestPriceUpdatesWithCallback{value: fee}(
        attackerProvider, block.timestamp, priceIds, callbackGasLimit
    );
}

// Step 3: Attacker (as provider) fulfills seqs 2..15001, skipping seq 1
for (uint64 s = 2; s <= 15001; s++) {
    echo.executeCallback(attackerProvider, s, updateData, priceIds);
    // Each call: while loop stops at firstUnfulfilledSeq=1 (still active) → O(1)
}

// Step 4: Provider attempts to fulfill seq 1
// clearRequest(1) executes, then while loop scans 15000 inactive entries
// ~15000 * 2100 gas = 31,500,000 gas > 30M block gas limit → REVERT
echo.executeCallback(attackerProvider, 1, updateData, priceIds); // OUT OF GAS
// Request 1 is permanently stuck; requester fee locked forever
```
