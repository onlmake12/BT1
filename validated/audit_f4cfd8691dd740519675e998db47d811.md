### Title
Attacker-Controlled Peer Can Skew Global `TimeAnalyzer` Thresholds, Causing Legitimate Peers to Be Penalized and Disconnected — (`sync/src/types/mod.rs`)

---

### Summary

The `InflightBlocks` block-download scheduler maintains a single, globally-shared `TimeAnalyzer` that dynamically computes three response-time thresholds (`fast_time`, `normal_time`, `low_time`) from a sliding window of recent block-download latencies. Because any connected peer can contribute samples to this global analyzer, a malicious peer that responds to block requests with near-zero latency can drive all three thresholds down to arbitrarily small values. Once the thresholds are skewed, legitimate peers whose responses fall above the artificially low `low_time` are repeatedly penalized (their `task_count` is halved via `punish`), and when `task_count` reaches zero they are disconnected. This is a direct analog of the Wise Lending heartbeat recalibration bug: a dynamic liveness/performance threshold is computed from recent history and can be poisoned by an attacker to cause a prolonged DoS against honest participants.

---

### Finding Description

**`TimeAnalyzer` — global, mutable, attacker-influenced**

`TimeAnalyzer` is a field of `InflightBlocks`, which is a single shared instance inside the synchronizer state. Every time any block is received from any peer, `InflightBlocks::remove_by_block` is called, which measures the elapsed download time and feeds it into `time_analyzer.push_time(elapsed)`. [1](#0-0) 

The `push_time` method accumulates `TIME_TRACE_SIZE` samples. When the buffer is full it sorts them and updates the three thresholds using an exponential moving average (EMA):

```rust
self.fast_time   = (self.fast_time.saturating_add(self.trace[FAST_INDEX]))   >> 1;
self.normal_time = (self.normal_time.saturating_add(self.trace[NORMAL_INDEX])) >> 1;
self.low_time    = (self.low_time.saturating_add(self.trace[LOW_INDEX]))    >> 1;
``` [2](#0-1) 

The EMA halves the current value and adds half the new sample, so sustained near-zero latency samples will drive all three thresholds toward zero.

**How the thresholds affect peer fate**

`push_time` returns a `TimeQuantile` that is used in `remove_by_block` to reward or penalize the responding peer's `task_count`. Responses above `low_time` are classified `UpperToMax` and trigger `decrease`, which reduces `task_count`. [3](#0-2) 

In `prune`, `low_time` is also used directly as a timeout limit for stuck in-flight blocks:

```rust
let timeout_limit = self.time_analyzer.low_time;
// ...
if now > timeout_limit + *time {
    d.punish(1);   // task_count >>= 1
``` [4](#0-3) 

`punish(1)` halves `task_count`. `punish(2)` (used for the fixed `BLOCK_DOWNLOAD_TIMEOUT` path) quarters it. When `task_count` reaches zero the peer is removed from `download_schedulers` and added to the disconnect list:

```rust
download_schedulers.retain(|k, v| {
    if v.task_count == 0 {
        disconnect_list.insert(*k);
        false
    } else { true }
});
``` [5](#0-4) 

**Attack scenario**

1. An attacker connects as a sync peer (unprivileged, no special role required).
2. The attacker pre-fetches or caches blocks and responds to every `GetBlocks` request with sub-millisecond latency, injecting `TIME_TRACE_SIZE` near-zero samples into the global `TimeAnalyzer`.
3. After one full buffer cycle the EMA drives `fast_time`, `normal_time`, and `low_time` toward zero.
4. Legitimate peers that respond at normal network latency (e.g., 200–500 ms) now fall into `UpperToMax` on every block delivery, causing their `task_count` to be decremented on each response.
5. Simultaneously, any in-flight block that has been waiting longer than the now-tiny `low_time` is pruned and its peer is punished with `punish(1)`.
6. Legitimate peers' `task_count` reaches zero and they are disconnected.

This is structurally identical to the Wise Lending issue: 50 fast Chainlink updates → heartbeat set to 1 minute → oracle considered dead after 1 minute of silence. Here: `TIME_TRACE_SIZE` fast block responses → `low_time` set to near-zero → legitimate peers considered "too slow" and disconnected.

---

### Impact Explanation

- **Peer isolation / network partitioning**: The victim node disconnects all honest peers whose latency exceeds the poisoned `low_time`, leaving it connected only to the attacker or to no one.
- **Sync stall / DoS**: With honest peers disconnected, block synchronization halts. The node cannot advance its chain tip, cannot validate new blocks, and cannot relay transactions.
- **Cascading effect**: Because `TimeAnalyzer` is global and persists in memory, the poisoned thresholds remain in effect until enough honest peers reconnect and contribute enough samples to raise the EMA back to realistic values — which is difficult if honest peers keep being disconnected.

---

### Likelihood Explanation

- Any unprivileged peer can connect to a CKB node; no key, no stake, no special role is required.
- Responding quickly to `GetBlocks` is trivially achievable: the attacker simply pre-downloads the chain and serves blocks from RAM.
- The attack is especially effective during IBD, when the node syncs primarily from one selected peer, giving the attacker near-total control over all `TimeAnalyzer` samples.
- The EMA formula means the attacker does not need to sustain the attack indefinitely; a single buffer-fill of fast samples is sufficient to shift the thresholds significantly.

---

### Recommendation

1. **Per-peer time tracking**: Maintain a separate `TimeAnalyzer` (or equivalent percentile tracker) per peer rather than a single global one. Peer-specific thresholds prevent one peer from poisoning the evaluation of others.
2. **Floor / clamp on thresholds**: Enforce a minimum value for `low_time` (e.g., `BLOCK_DOWNLOAD_TIMEOUT / 4`) so that even if all samples are near-zero the timeout cannot drop below a safe operational floor.
3. **Outlier rejection**: Before updating the EMA, discard samples that are more than N standard deviations below the current mean, or use a trimmed mean that ignores the bottom K% of samples.
4. **Rate-limit sample contribution per peer**: Cap how many samples a single peer can contribute to the global analyzer within a time window, preventing a single fast peer from dominating the distribution.

### Proof of Concept

```
Precondition: victim node is syncing; attacker controls one connected outbound peer.

1. Attacker peer pre-downloads the full chain locally.
2. For every GetBlocks request from the victim, attacker responds in < 1 ms.
3. After TIME_TRACE_SIZE responses, TimeAnalyzer::push_time triggers an EMA update:
       low_time_new = (low_time_old + trace[LOW_INDEX]) >> 1
   With trace[LOW_INDEX] ≈ 0, low_time converges toward 0 after repeated cycles.
4. Victim calls InflightBlocks::prune on the next timer tick.
       timeout_limit = self.time_analyzer.low_time  // now ≈ 0
       if now > timeout_limit + *time { d.punish(1); }
   Every honest peer's in-flight block (timestamp > 0 ms ago) immediately times out.
5. punish(1) halves task_count for each honest peer on each prune cycle.
6. After log2(INIT_BLOCKS_IN_TRANSIT_PER_PEER) = log2(32) = 5 prune cycles,
   task_count reaches 0 and the peer is added to disconnect_list.
7. All honest peers are disconnected; victim node is isolated.
``` [6](#0-5) [7](#0-6) [8](#0-7)

impl TimeAnalyzer {
    fn push_time(&mut self, time: u64) -> TimeQuantile {
        if self.index < TIME_TRACE_SIZE {
            self.trace[self.index] = time;
            self.index += 1;
        } else {
            self.trace.sort_unstable();
            self.fast_time = (self.fast_time.saturating_add(self.trace[FAST_INDEX])) >> 1;
            self.normal_time = (self.normal_time.saturating_add(self.trace[NORMAL_INDEX])) >> 1;
            self.low_time = (self.low_time.saturating_add(self.trace[LOW_INDEX])) >> 1;
            self.trace[0] = time;
            self.index = 1;
        }
