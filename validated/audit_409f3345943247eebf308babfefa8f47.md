### Title
`need_clean` Single-Sample Logic Flaw Allows Permanent Stale Orphan Block Retention — (`chain/src/utils/orphan_block_pool.rs`)

---

### Summary

`InnerPool::need_clean` samples only the first entry from a non-deterministic `HashMap` to decide whether an entire sibling group under a parent hash should be evicted. An unprivileged remote peer can insert one fresh-epoch block alongside many stale-epoch blocks under the same parent hash. When the sampler lands on the fresh block, `need_clean` returns `false` and the entire group — including all stale blocks — is permanently retained, violating the `EXPIRED_EPOCH` invariant and enabling unbounded orphan pool growth.

---

### Finding Description

**Root cause — `need_clean` samples one non-deterministic entry:** [1](#0-0) 

`self.blocks.get(parent_hash)` returns a `HashMap<packed::Byte32, LonelyBlockHash>`. Calling `.iter().next()` on a `HashMap` yields an entry in an unspecified, hash-randomized order. If the group contains both a stale block (epoch 0) and a fresh block (epoch ≥ `tip_epoch − EXPIRED_EPOCH`), the sampler may return the fresh block, causing `need_clean` to return `false` and `clean_expired_blocks` to skip the entire group: [2](#0-1) 

The stale sibling is never evicted on subsequent timer firings either, because the fresh block remains in the group and continues to "shield" it.

**Attack entry point — non-contextual block verification does not check PoW or epoch:**

The `ChainService::non_contextual_verify` calls only `BlockVerifier` and `NonContextualBlockTxsVerifier`: [3](#0-2) 

`BlockVerifier::verify` checks only proposals limit, block bytes, cellbase structure, duplicates, and Merkle root — **no PoW, no epoch validation**: [4](#0-3) 

Epoch continuity (`EpochVerifier::is_successor_of`) and PoW (`PowVerifier`) are only enforced in the contextual `HeaderVerifier`, which requires the parent header and is never invoked on the non-contextual path: [5](#0-4) 

Therefore, an attacker can craft blocks with **any epoch number** and **no valid PoW**, and they will pass non-contextual verification.

**Insertion path:**

After passing non-contextual verification, the block is stored to the DB and forwarded to `OrphanBroker::process_lonely_block`. If the parent is unknown, it is inserted into the orphan pool: [6](#0-5) 

The `epoch_number` stored in `LonelyBlockHash` is taken directly from the block header: [7](#0-6) 

**Timer path:**

Every 60 seconds, `clean_expired_orphans` fires, reads the tip epoch, and calls `clean_expired_blocks`: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

- Stale orphan blocks permanently occupy entries in `InnerPool::blocks` and `InnerPool::parents` (in-memory `HashMap`s with no hard size cap).
- Each block is also written to the DB via `insert_block` before orphan insertion; those DB entries are never cleaned because `delete_block` / `remove_block_status` are only called for blocks returned by `clean_expired_blocks`.
- The attacker can repeat the pattern across arbitrarily many distinct parent hashes, each shielded by one fresh block, causing unbounded memory and DB growth.
- The 60-second cleanup timer provides no relief because the shield block is never removed.

---

### Likelihood Explanation

- **No PoW required**: `BlockVerifier` does not check PoW; the attacker sends structurally valid blocks with arbitrary epoch fields and any `compact_target`.
- **No privileged access required**: the attack enters through the standard P2P `SendBlock` sync message path, available to any peer.
- **Deterministically triggerable**: the attacker controls the epoch numbers of both blocks. The only non-determinism is which block `map.iter().next()` returns, but since the attacker needs only a 50 %+ chance per cleanup cycle and the pool is never shrunk, the stale block survives with high probability over multiple cycles.
- **Low cost**: crafting and sending P2P block messages is essentially free; no mining is needed.

---

### Recommendation

Replace the single-sample check with an all-children check. `need_clean` should return `true` only if **all** children under the parent hash are expired (or alternatively, evict only the individually expired children rather than the whole group):

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            !map.is_empty()
                && map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
        })
        .unwrap_or_default()
}
```

Or, for finer-grained eviction, iterate all children and remove only those that are individually expired, updating `parents` accordingly.

### Proof of Concept

```rust
// Pseudocode — no PoW needed because BlockVerifier does not check it.
let parent_hash = some_unknown_hash(); // any hash not in the chain

// Stale block: epoch 0, tip_epoch = 20 → 0 + 6 < 20 → should be cleaned
let stale_block = BlockBuilder::default()
    .parent_hash(parent_hash.clone())
    .epoch(EpochNumberWithFraction::new(0, 0, 1000))
    .number(1u64)
    .build();

// Fresh block: epoch 15 → 15 + 6 >= 20 → should NOT be cleaned
let fresh_block = BlockBuilder::default()
    .parent_hash(parent_hash.clone())
    .epoch(EpochNumberWithFraction::new(15, 0, 1000))
    .number(1u64)
    .build();

// Both pass non_contextual_verify (no PoW, no epoch check in BlockVerifier).
// Both are inserted into the orphan pool under the same parent_hash.
pool.insert(stale_block.into());
pool.insert(fresh_block.into());

// Simulate cleanup at tip_epoch = 20.
let evicted = pool.clean_expired_blocks(20);

// Depending on HashMap iteration order, evicted may be empty (fresh block sampled first),
// leaving the stale block permanently in the pool.
// The stale block is never evicted on subsequent calls either.
assert!(pool.len() > 0); // stale block survives
```

The `need_clean` comment itself acknowledges the sampling: `"get 1st block belongs to that parent and check if it's expired block"` — but it does not account for mixed-epoch sibling groups. [10](#0-9)
