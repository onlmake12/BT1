### Title
TOCTOU Race in `HeaderMapKernel::limit_memory` Leaks Sled Entries and Produces Ghost `contains_key` Results — (`shared/src/types/header_map/kernel_lru.rs`)

### Summary
A time-of-check/time-of-use race between `limit_memory` and concurrent `remove` calls allows a header hash to be written into the sled backend after it has been removed from memory, with no subsequent cleanup. The leaked entry causes `contains_key` to return `true` for a removed header and can resurrect the header on a subsequent `get` call.

### Finding Description
`limit_memory` operates in three non-atomic steps:

1. **Snapshot** — `memory.front_n` acquires a read lock, copies the oldest N entries into an owned `Vec`, and releases the lock. [2](#0-1) 
2. **Persist** — `backend.insert_batch` writes the snapshot to sled. [4](#0-3) 
3. **Evict** — `memory.remove_batch` removes the snapshot keys from memory. [5](#0-4) 

Between steps 1 and 2, a concurrent `remove(hash_k)` can execute. It removes `hash_k` from memory, then checks `backend.is_empty()`. Because step 2 has not yet run, the backend is empty and `remove_no_return` is skipped. [9](#0-8) 

Step 2 then writes `hash_k` into sled. Step 3 silently skips `hash_k` (already absent from memory). `hash_k` is now permanently stranded in sled with no code path to clean it up unless `get(hash_k)` is called — which would incorrectly resurrect it. [7](#0-6) 

### Impact Explanation
- **Unbounded sled disk growth**: Each race occurrence leaks one entry. Under sustained header flooding, this accumulates without bound.
- **Stale `HEADER_VALID` ghost entries**: `contains_key` returns `true` for headers that were explicitly removed, corrupting sync state decisions that rely on header map membership.
- **Ghost resurrection**: A subsequent `get` call for a removed hash pulls it back from sled into memory, violating the invariant that `remove` permanently deletes the entry.

### Likelihood Explanation
The race window is the interval between `front_n` returning and `insert_batch` completing. An unprivileged peer can widen this window by flooding valid headers to keep memory near `memory_limit`, maximizing the frequency of `limit_memory` cycles (every 5 s). [10](#0-9)  Concurrent `remove` calls arise naturally from block verification and chain reorganization processing. No special privilege is required.

### Recommendation
Perform the backend removal **inside the same critical section** as the memory removal, or re-check backend membership after `insert_batch` and remove any key that is no longer in memory. A simpler fix: in `remove`, do not skip `backend.remove_no_return` based on `is_empty()` — always call it unconditionally (sled remove on a missing key is a no-op). This eliminates the TOCTOU window entirely. [9](#0-8) 

### Proof of Concept
```
Thread A (limit_memory):                Thread B (remove hash_k):
  front_n() → snapshot S (hash_k ∈ S)
  [read lock released]
                                          memory.remove(hash_k) → OK
                                          backend.is_empty() → true (insert_batch not yet called)
                                          → skips backend.remove_no_return
  backend.insert_batch(S)               ← hash_k written to sled
  memory.remove_batch(S)                ← hash_k already gone, no-op

Assert: backend.contains_key(hash_k) == true   ← INVARIANT VIOLATED
        memory.contains_key(hash_k)   == false
```
A concurrent invariant test that asserts `!backend.contains_key(hash)` for all hashes passed to `remove()`, run under concurrent `limit_memory` pressure with `memory_limit` set just below the number of inserted headers, will reproduce this failure deterministically.
