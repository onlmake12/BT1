Audit Report

## Title
O(N²) Ancestor BFS Re-traversal in `remove_by_detached_proposal` During Reorg — (`tx-pool/src/pool.rs`, `tx-pool/src/component/pool_map.rs`)

## Summary
`TxPool::remove_by_detached_proposal` removes a proposed transaction chain and re-inserts each entry into pending via `add_pending`. Each `add_pending` call triggers two full ancestor BFS traversals. For a linear chain of N transactions re-inserted in topological order, this produces O(N²) total BFS work while holding the tx-pool write lock, blocking all concurrent tx-pool operations for the duration of the reorg handler.

## Finding Description
**Root cause — `remove_by_detached_proposal` (`tx-pool/src/pool.rs` L333–356):**
The function removes the chain atomically via `remove_entry_and_descendants`, sorts entries by `ancestors_count` (root first), then calls `add_pending(entry)` for each of the N entries one at a time. Because entries are re-inserted in topological order, when inserting the k-th entry, exactly k−1 ancestors are already present in the pool.

**Traversal 1 — `check_and_record_ancestors` → `get_tx_ancenstors` → `calc_relation_ids` (`pool_map.rs` L588–600, L549–553; `links.rs` L52–72):**
`get_tx_ancenstors` calls `self.links.calc_relation_ids(parents.clone(), Relation::Parents)`, which is an iterative BFS over all ancestors currently in the pool. For the k-th insertion, this visits k−1 nodes.

**Traversal 2 — `record_entry_descendants` → `update_ancestors_index_key` → `calc_ancestors` (`pool_map.rs` L509–512, L432–444):**
`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`, which calls `calc_relative_ids` → `calc_relation_ids` — another full BFS over all ancestors. For the k-th insertion, this again visits k−1 nodes.

**Quadratic accumulation:**
```
Total BFS node visits = 2 × (0 + 1 + 2 + ... + (N-1)) = N(N-1) = O(N²)
```

**Lock scope — `_update_tx_pool_for_reorg` (`process.rs` L836–851, L1055–1056):**
The tx-pool write lock is acquired at L836 and held across the entire call to `_update_tx_pool_for_reorg`, which calls `remove_by_detached_proposal` at L1056. All concurrent tx-pool operations are blocked for the full duration.

## Impact Explanation
While the tx-pool write lock is held, all concurrent operations (transaction submission, block template generation, peer sync) are blocked. For a chain of N transactions at the configured `max_ancestors_count`, the O(N²) BFS work occurs in a single lock-holding call during reorg processing. Because natural reorgs affect all network participants simultaneously, a single attacker-submitted chain can cause coordinated stalls across many nodes. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
Requirements for the attacker:
1. Submit a chain of N transactions (N ≤ `max_ancestors_count`) via standard RPC — no privilege required.
2. Wait for miners to include the proposal short IDs in a block's proposal zone — miners do this automatically.
3. Wait for any natural 1-block reorg — these occur regularly on mainnet without attacker involvement.

Cost: transaction fees for N transactions. No hashpower, no privileged access, no victim mistakes required. The attack is repeatable after every reorg event.

## Recommendation
Replace per-insertion ancestor BFS with a single-pass bulk re-insertion:

1. After `remove_entry_and_descendants`, re-link all entries in topological order (already sorted by `ancestors_count`).
2. For each re-inserted entry, compute ancestors incrementally from the already-inserted parent set using O(1) parent lookups rather than a full BFS — since parents are guaranteed to be inserted before children in the sorted order.
3. Defer `update_ancestors_index_key` and `_record_ancestors` until the full chain is re-linked, then perform a single pass to update descendant weights, reducing total complexity from O(N²) to O(N·E) where E is the average out-degree (typically 1 for a chain).

## Proof of Concept
1. Build `tx0` (coinbase spend), `tx1` (spends `tx0` output), ..., `tx_{N-1}` (spends `tx_{N-2}` output) where N = `max_ancestors_count`.
2. Submit all N transactions to the node via RPC.
3. Wait for a block that includes all N proposal short IDs in its proposal zone.
4. Observe a natural 1-block reorg (or broadcast a competing block at the same height).
5. `_update_tx_pool_for_reorg` calls `remove_by_detached_proposal` with N IDs.
6. Benchmark `remove_by_detached_proposal` with chains of length 100, 200, 500, N — wall-clock time should scale quadratically (10× more transactions → ~100× longer execution).
7. During the stall, confirm that concurrent RPC calls for transaction submission and `get_block_template` are blocked.

                }
            }
        }
    }

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
    }
