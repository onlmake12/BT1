### Title
Unvalidated `providerToCredit` in `executeCallback` Allows Fee Theft After Exclusivity Period — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary
`Echo.executeCallback()` accepts a caller-supplied `providerToCredit` address and credits the full request fee to it without verifying the address is a registered provider. During the exclusivity period this is constrained to `req.provider`, but after the period expires any caller can pass an arbitrary registered provider address — including their own — and steal the fee that rightfully belongs to the original provider.

---

### Finding Description

`requestPriceUpdatesWithCallback` stores the fee paid by the user minus the Pyth protocol cut in `req.fee`, keyed to `req.provider`:

```solidity
req.provider = provider;
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```

`executeCallback` then credits the full accumulated fee to the caller-supplied `providerToCredit`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

The only guard on `providerToCredit` is the exclusivity-period check:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

Once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, this guard is skipped entirely. There is **no subsequent check** that `providerToCredit` equals `req.provider` or that `_state.providers[providerToCredit].isRegistered` is `true`. Any caller may pass any address — including their own registered provider address — and the fee is credited there unconditionally.

---

### Impact Explanation

**Direct financial loss to the legitimate provider.** The fee paid by the user at request time (`req.fee`) was computed using the legitimate provider's fee schedule (`getFee(provider, callbackGasLimit, priceIds)`). After the exclusivity window, an attacker who is a registered provider can front-run or simply call `executeCallback(attackerProvider, sequenceNumber, ...)`, redirect the entire fee to their own `accruedFeesInWei`, and withdraw it via `withdrawAsFeeManager`. The legitimate provider receives nothing for fulfilling the original commitment.

Additionally, if `providerToCredit` is an unregistered address (no `feeManager` set), the fee is permanently locked in the contract with no withdrawal path, constituting irreversible loss of user-paid funds.

---

### Likelihood Explanation

**High.** The exclusivity period is a configurable, finite window. After it expires, the vulnerability is unconditionally reachable by any externally-owned account that has previously called `registerProvider`. No privileged access, leaked key, or governance majority is required. The attacker only needs to monitor the mempool or block timestamps and submit a transaction after the exclusivity window closes.

---

### Recommendation

After the exclusivity period, restrict `providerToCredit` to registered providers and, optionally, enforce that it matches `req.provider` unless a penalty/redistribution mechanism is explicitly intended:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit must be a registered provider"
);
```

If the design intent is to allow any registered provider to fulfill after the exclusivity period (as suggested by the TODO comment), this check is the minimum required. If the intent is to penalise the original provider for lateness, an explicit penalty deduction from `req.provider` and transfer to `providerToCredit` should be implemented with clear bounds.

---

### Proof of Concept

1. **Setup**: `legitimateProvider` registers with fee schedule F. User calls `requestPriceUpdatesWithCallback(legitimateProvider, publishTime, priceIds, gasLimit)` paying fee F, stored as `req.fee = F - pythFee`, `req.provider = legitimateProvider`.

2. **Attacker registers**: Attacker calls `registerProvider(...)` to register `attackerProvider`. Attacker calls `setFeeManager(attackerProvider)` so they can later withdraw.

3. **Wait**: Attacker waits until `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`.

4. **Exploit**: Attacker calls:
   ```solidity
   echo.executeCallback(
       attackerProvider,   // <-- attacker's own provider, not req.provider
       sequenceNumber,
       updateData,
       priceIds
   );
   ```
   Inside `executeCallback`, the exclusivity check is skipped. The line:
   ```solidity
   _state.providers[providerToCredit].accruedFeesInWei += (req.fee + msg.value) - pythFee;
   ```
   credits the full fee to `attackerProvider`.

5. **Withdraw**: Attacker calls `withdrawAsFeeManager(attackerProvider, amount)` and receives the fee. `legitimateProvider.accruedFeesInWei` remains zero. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
