Audit Report

## Title
Unrestricted `ckb_wait` Syscall Allows Any Spawned VM to Consume Sibling Exit Codes, Causing Parent Script Failure - (File: `script/src/scheduler.rs`)

## Summary
The `Message::Wait` handler in `script/src/scheduler.rs` performs no check that the calling VM (`vm_id`) is the recorded parent of the target VM (`args.target_id`). Any VM can call `ckb_wait` with an arbitrary target ID, consume that target's exit code from `terminated_vms`, and permanently remove it via `retain`. A malicious spawned child can exploit this to steal a sibling's exit code before the legitimate parent retrieves it, causing the parent's subsequent `ckb_wait` to return `WAIT_FAILURE` and the lock script to fail.

## Finding Description

**No parent-child tracking exists in the scheduler.** The `Scheduler` struct contains no field mapping spawner to spawnee. A grep for `parent_id`, `child_vm`, or equivalent in `scheduler.rs` returns zero matches. The `Message::Spawn` handler at lines 539–562 records only fd ownership (`self.fds`, `self.inherited_fd`) — no parent relationship is stored.

**The `Message::Wait` handler has no ownership check.** At lines 564–592 of `script/src/scheduler.rs`, when `Message::Wait(vm_id, args)` is processed:

```rust
Message::Wait(vm_id, args) => {
    if let Some(exit_code) = self.terminated_vms.get(&args.target_id).copied() {
        // ... write exit code to vm_id's memory ...
        self.states.insert(vm_id, VmState::Runnable);
        self.terminated_vms.retain(|id, _| id != &args.target_id); // permanent removal
        continue;
    }
    if !self.states.contains_key(&args.target_id) {
        // ... set WAIT_FAILURE ...
        continue;
    }
    self.states.insert(vm_id, VmState::Wait { target_vm_id: args.target_id, ... });
}
```

There is no assertion that `vm_id` is the parent of `args.target_id`. The `terminated_vms.retain` call at line 575 permanently removes the exit code entry for `args.target_id` from the global map. Any VM — including a sibling — can reach this path.

**The `Wait` syscall passes the target ID directly from script-controlled register A0** (`script/src/syscalls/wait.rs`, lines 37–49) with no validation before posting `Message::Wait`.

**Largest-ID-first scheduling makes the attack deterministic.** Lines 336–348 of `scheduler.rs` show the scheduler always picks the runnable VM with the largest ID. A malicious child (always assigned a higher ID than the root) is guaranteed to execute before the root VM, giving it a reliable window to call `ckb_wait(sibling_id)` after the sibling terminates but before the root does.

**The `child_wait_dead_lock` test at `script/testdata/spawn_cases.c` lines 147–151 explicitly demonstrates that a child can call `ckb_wait(0)` (the parent's PID)**, confirming the complete absence of any ownership enforcement in the existing implementation.

**Contrast with fd ownership:** The `Message::Spawn` handler at line 525 checks `self.fds.get(fd) != Some(&vm_id)` before allowing fd transfer. The `Message::FdRead` handler at lines 619–626 checks `self.fds[&args.fd] == vm_id`. No equivalent guard exists for `Message::Wait`.

**Exploit flow:**
1. Root (pid=0) spawns Child A (pid=1, trusted) and Child B (pid=2, malicious cell dep).
2. Child B (pid=2) runs first (largest ID). It stays runnable.
3. Child A (pid=1) runs and terminates with exit code 0. Scheduler executes `terminated_vms.insert(1, 0)` (line 364). No VM is in `VmState::Wait { target_vm_id: 1 }`, so no immediate wakeup. Child A is removed from `states`, `instantiated`, `suspended`.
4. Child B (pid=2) runs again. It calls `ckb_wait(1)`. The handler finds `terminated_vms[1] = 0`, writes it to Child B's memory, marks Child B `Runnable`, and executes `terminated_vms.retain(|id, _| id != &1)` — permanently removing the entry.
5. Root (pid=0) calls `ckb_wait(1)`. `terminated_vms.get(&1)` → `None`. `states.contains_key(&1)` → `false`. Scheduler returns `WAIT_FAILURE` to root.
6. Root VM receives `WAIT_FAILURE`, fails, transaction is rejected.

## Impact Explanation

This is an **incorrect implementation of CKB-VM system script behavior** (High, 10001–15000 points). A malicious cell dep binary can deterministically cause a lock script using the spawn/wait pattern to fail, rejecting otherwise valid transactions. If the lock script uses the child's exit code for authorization decisions, the malicious child can also manipulate authorization outcomes by forcing incorrect return values. The `terminated_vms` map is shared mutable state with no per-entry access control, directly analogous to the class of isolation bugs where shared framework state is exploitable by any callee.

## Likelihood Explanation

The attack requires a lock script that spawns multiple children and allows at least one child's code to be supplied by the transaction author (e.g., a plugin or verifier cell dep pattern). This is a realistic design in modular lock scripts. The scheduler's deterministic largest-ID-first execution order eliminates any timing dependency: the malicious child is always guaranteed to run before the root VM. Sibling PIDs are sequential integers starting from 1 and are trivially predictable. The attack is fully reproducible without any race condition or external coordination.

## Recommendation

Track the parent-child relationship at spawn time. In `boot_vm` (called from the `Message::Spawn` handler at line 539), record `spawner_vm_id → spawned_vm_id` in a new `BTreeMap<VmId, VmId>` field (e.g., `vm_parents`). In the `Message::Wait` handler, before processing the wait, verify that `vm_parents.get(&args.target_id) == Some(&vm_id)`. If the check fails, return `WAIT_FAILURE` immediately without consuming or removing the `terminated_vms` entry. This mirrors the existing fd ownership check pattern used in `Message::Spawn` (line 525), `Message::FdRead` (line 619), and `Message::FdWrite`.

## Proof of Concept

**Minimal test plan:**

1. Write a root script (pid=0) that spawns two children and calls `ckb_wait` on pid=1, asserting `SUCCESS`.
2. Write Child A (pid=1) that immediately exits with code 0.
3. Write Child B (pid=2, malicious) that calls `ckb_wait(1)` before returning — consuming pid=1's exit code from `terminated_vms`.
4. Construct a CKB transaction with the root script as the lock, Child A from a trusted cell dep, Child B from an attacker-controlled cell dep.
5. Execute the transaction through the scheduler. Observe that root's `ckb_wait(1)` returns `WAIT_FAILURE` and the script fails.

The existing `child_wait_dead_lock` function in `script/testdata/spawn_cases.c` (lines 147–151) already demonstrates that a child can call `ckb_wait` on an arbitrary PID (the parent's pid=0) with no rejection, confirming the absence of any ownership enforcement and providing a direct template for the sibling-targeting variant.
