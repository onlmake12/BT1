### Title
`header.num_prices` Mutated Before Bounds Check in `buffer::update`, Corrupting Buffer State on Overflow - ([File: target_chains/solana/programs/pyth-price-store/src/accounts/buffer.rs])

---

### Summary

In `buffer::update`, `header.num_prices` is incremented **before** the slice bounds check that guards the actual write. If the bounds check fails, the function returns `Err(ExtendError::NotEnoughSpace)` but leaves `header.num_prices` permanently advanced for the current slot, making the buffer unwritable and unreadable for the remainder of that slot.

---

### Finding Description

The ordering in `buffer::update` is:

```
1. Validate new_prices length is a multiple of BufferedPrice size  (line 148)
2. Reset num_prices to 0 if slot changed                           (lines 153-156)
3. Compute start / end offsets                                     (lines 161-162)
4. *** Increment header.num_prices ***                             (lines 163-166)
5. Call prices.get_mut(start..end) — bounds check                  (lines 168-170)
6. If get_mut returns None → return Err(NotEnoughSpace)
``` [1](#0-0) 

Step 4 mutates persistent on-chain state. Step 6 is the error path that unwinds the Rust call stack — but Solana does **not** roll back account data on a `ProgramError` return from within the instruction handler; the borrow of `buffer.data` has already been mutated in place.

A publisher whose buffer holds `K` prices can submit a `prices_data` payload (length must be a multiple of `size_of::<BufferedPrice>()` = 20 bytes) large enough that `end > prices.len()`. After the failed call:

- `header.num_prices` = `K + num_new_prices` (no prices were actually written)
- Every subsequent call in the same slot computes `start` beyond the buffer capacity and also fails with `NotEnoughSpace`
- The validator's `buffer::read` checks `expected_len > prices_bytes.len()` and returns `InvalidNumPrices`, so it cannot read **any** prices from the buffer for that slot [2](#0-1) 

The corruption is bounded to one slot: the slot-change guard resets `num_prices = 0` at the start of the next slot. [3](#0-2) 

**Correction to the question's framing:** the claim of "exceeds `prices.len()` by exactly 1 byte" is impossible. The `InvalidLength` guard at line 148 requires `new_prices.len()` to be a multiple of 20 bytes, so overflow can only occur in 20-byte increments. The core bug is nonetheless real and reachable. [4](#0-3) 

---

### Impact Explanation

For the duration of slot S, the affected publisher's buffer is both unwritable (all further `submit_prices` calls fail) and unreadable by the validator (`InvalidNumPrices`). The publisher's price feed contributes nothing to the aggregate for that slot. The buffer self-heals at slot S+1.

---

### Likelihood Explanation

Only the publisher who owns the buffer can call `submit_prices` for it (signer check + PDA binding). [5](#0-4) 

This means the DoS is **self-inflicted**: a publisher cannot corrupt a rival's buffer. The trigger is accidental (miscalculating payload size) or intentional self-sabotage. It cannot be weaponized by a third party against an arbitrary publisher.

---

### Recommendation

Move the `header.num_prices` increment to **after** the successful `get_mut` call:

```rust
let destination = prices
    .get_mut(start..end)
    .ok_or(ExtendError::NotEnoughSpace)?;

// Only update header state after confirming the write will succeed
header.num_prices = header
    .num_prices
    .checked_add(num_new_prices)
    .expect("unexpected overflow");
``` [6](#0-5) 

---

### Proof of Concept

```rust
// Buffer sized for exactly 2 prices (2 * 20 = 40 bytes of price space)
let mut buf = vec![0u8; size(2)];
create(&mut buf, Default::default()).unwrap();
let (header, prices) = read_mut(&mut buf).unwrap();

// First call: write 1 price (20 bytes) — succeeds, num_prices = 1
update(header, prices, 1, &vec![1u8; 20]).unwrap();
assert_eq!(header.num_prices, 1);

// Second call: attempt to write 2 more prices (40 bytes) — overflows capacity
// start = 20, end = 60, prices.len() = 40 → NotEnoughSpace
// BUT header.num_prices is already incremented to 3 before the check fails
let result = update(header, prices, 1, &vec![1u8; 40]);
assert!(result.is_err()); // NotEnoughSpace
assert_eq!(header.num_prices, 3); // CORRUPTED: only 1 price was ever written

// Third call in same slot: start = 60, beyond buffer → also fails
let result2 = update(header, prices, 1, &vec![1u8; 20]);
assert!(result2.is_err()); // buffer locked for slot 1

// Validator read also fails
assert!(read(&buf).is_err()); // InvalidNumPrices
``` [7](#0-6)

```rust
pub fn update(
    header: &mut BufferHeader,
    prices: &mut [u8],
    current_slot: u64,
    new_prices: &[u8],
) -> Result<(), ExtendError> {
    if new_prices.len() % size_of::<BufferedPrice>() != 0 {
        return Err(ExtendError::InvalidLength);
    }
    // We expect the validator to apply the buffered prices at the end of each slot,
    // so when we observe a new slot we remove previously stored prices.
    if header.slot != current_slot {
        header.slot = current_slot;
        header.num_prices = 0;
    }
    let num_new_prices = (new_prices.len() / size_of::<BufferedPrice>())
        .try_into()
        .expect("unexpected overflow");
    let num_prices: usize = header.num_prices.try_into().unwrap();
    let start = size_of::<BufferedPrice>() * num_prices;
    let end = size_of::<BufferedPrice>() * num_prices + new_prices.len();
    header.num_prices = header
        .num_prices
        .checked_add(num_new_prices)
        .expect("unexpected overflow");

    let destination = prices
        .get_mut(start..end)
        .ok_or(ExtendError::NotEnoughSpace)?;

    // We don't validate the values of `new_prices` to make the publishing process
    // more efficient. They will be validated when applied in the validator.
    #[cfg(feature = "solana-program")]
    solana_program::program_memory::sol_memcpy(destination, new_prices, new_prices.len());
    #[cfg(not(feature = "solana-program"))]
    destination.copy_from_slice(new_prices);
    Ok(())
}
