Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Unique `(PeerId, PeerId)` Keys — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

The `forward_rate_limiter` in `HolePunching` uses a `governor::state::keyed::HashMapStateStore<(PeerId, PeerId, u32)>` whose entries are only evicted via `retain_recent()` inside `disconnected()`. An attacker maintaining a single persistent session can send `ConnectionRequest` messages at the per-session cap (30/sec) with unique attacker-controlled `(from, to)` PeerId pairs, inserting one new key per message into the store with no eviction until disconnect, causing unbounded memory growth and eventual OOM crash.

## Finding Description

`HolePunching` declares two rate limiters:

```rust
rate_limiter: RateLimiter<(PeerIndex, u32)>,
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,
``` [1](#0-0) 

`retain_recent()` is called on both only inside `disconnected()`:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
``` [2](#0-1) 

The `notify()` handler fires every 5 minutes and cleans `pending_delivered` and `inflight_requests`, but never calls `retain_recent()` on either rate limiter:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    ...
    self.pending_delivered.retain(|_, (_, t)| (now - *t) < TIMEOUT);
    self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);
``` [3](#0-2) 

In `received()`, the outer guard is `rate_limiter` keyed by `(session_id, msg_item_id)` at 30 req/sec: [4](#0-3) 

Each message that passes then hits `forward_rate_limiter.check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))`: [5](#0-4) 

`content.from` and `content.to` are fully attacker-controlled PeerIds parsed from the message body: [6](#0-5) 

`governor`'s `check_key` on a `HashMapStateStore` inserts a new entry for any previously unseen key. Since the attacker supplies unique `(from, to)` pairs, every message inserts a new entry. The `forward_rate_limiter` quota is 1 req/sec per key, so entries become stale after 1 second but are never evicted during an active session — `retain_recent()` is the only eviction mechanism and it is never called from `notify()`.

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Growth rate: 30 entries/sec × ~150 bytes/entry ≈ 4.3 KB/sec. On a 4 GB node this reaches OOM in approximately 11 days of continuous flooding. The crash is deterministic and requires no special privileges — only a single persistent P2P connection. The impact is a local node crash (OOM), which is in scope.

## Likelihood Explanation

- HolePunching must be enabled (opt-in via `support_protocols` config), which is a precondition but a supported production feature.
- Attacker needs exactly one persistent P2P connection — no elevated privileges.
- Generating unique PeerIds is trivial (random 32-byte values).
- The attack is slow (days to OOM) but fully deterministic and requires no victim interaction.
- The `notify()` timer at 5-minute intervals is the natural fix point but is unused for this cleanup, confirming the invariant is definitively broken.

## Recommendation

Add `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside the `notify()` handler alongside the existing cleanup:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    // ... existing pending_delivered / inflight_requests cleanup
``` [3](#0-2) 

This ensures stale entries are evicted every 5 minutes regardless of whether any peer disconnects.

## Proof of Concept

1. Enable HolePunching on a test node (`support_protocols` includes `HolePunching`).
2. Connect one session and send `ConnectionRequest` messages at 30/sec, each with a freshly generated random `from` and `to` PeerId (valid 32-byte multihash values).
3. After N seconds, inspect the `forward_rate_limiter` internal state — key count equals N × 30 with zero eviction.
4. Confirm `disconnected()` is never triggered (session stays alive); confirm `notify()` fires every 5 minutes but does not reduce key count.
5. Memory grows linearly without bound for the session lifetime, eventually causing OOM.
