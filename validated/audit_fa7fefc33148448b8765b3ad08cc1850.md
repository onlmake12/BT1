### Title
Echo Provider Fees Permanently Locked When No Fee Manager Is Set — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo` contract accumulates provider fees in `_state.providers[provider].accruedFeesInWei` but provides **no direct `withdraw()` function for providers**. The only withdrawal path is `withdrawAsFeeManager()`, which requires a fee manager to be configured. Providers who register without calling `setFeeManager()` have `feeManager = address(0)` by default, making their accrued fees permanently unclaimable.

---

### Finding Description

When a provider registers via `registerProvider()`, only fee-rate fields are initialized — `feeManager` is left as the Solidity zero-default (`address(0)`):

```solidity
// Echo.sol lines 381-393
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    provider.feePerFeedInWei = feePerFeedInWei;
    provider.feePerGasInWei = feePerGasInWei;
    provider.isRegistered = true;
    // feeManager is NOT set — defaults to address(0)
}
``` [1](#0-0) 

When `executeCallback()` runs, provider fees are credited:

```solidity
// Echo.sol line 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The **only** withdrawal path for provider fees is `withdrawAsFeeManager()`:

```solidity
// Echo.sol lines 360-379
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    ...
}
``` [3](#0-2) 

The `IEcho` interface confirms there is no `withdraw()` function for providers — only `withdrawFees()` (admin-only for Pyth protocol fees) and `withdrawAsFeeManager()`: [4](#0-3) 

If `feeManager == address(0)`, the `require(msg.sender == _state.providers[provider].feeManager)` check can never pass (no EOA or contract can send from `address(0)`), so the provider's accrued fees are permanently locked in the contract.

By contrast, the `Entropy` contract correctly provides a direct `withdraw(uint128 amount)` function callable by the provider itself: [5](#0-4) 

---

### Impact Explanation

Any Echo provider who registers but does not separately call `setFeeManager()` accumulates fees that are permanently unclaimable. Since `setFeeManager()` is a separate, optional transaction, this is the default state for newly registered providers. The fees are locked in the contract with no recovery path — an exact analog to the Aave incentives issue where rewards accumulate but cannot be claimed.

**Impact: High** — provider funds are permanently lost.

---

### Likelihood Explanation

**Likelihood: Medium** — Echo is a newer contract (still has `TODO` comments throughout). Any provider who registers and begins fulfilling requests without first calling `setFeeManager()` is immediately affected. The registration flow does not prompt or require fee manager setup, making this a realistic omission.

---

### Recommendation

Add a direct `withdraw(uint128 amount)` function to `Echo.sol` callable by the provider itself (analogous to `Entropy.withdraw()`):

```solidity
function withdraw(uint128 amount) external {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(provider.isRegistered, "Provider not registered");
    require(provider.accruedFeesInWei >= amount, "Insufficient balance");
    provider.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "Withdrawal failed");
    emit FeesWithdrawn(msg.sender, amount);
}
```

Alternatively, modify `withdrawAsFeeManager()` to also allow the provider itself to call it (i.e., `msg.sender == provider || msg.sender == feeManager`).

---

### Proof of Concept

1. Provider calls `Echo.registerProvider(baseFee, feedFee, gasFee)` — `feeManager` is `address(0)`.
2. Users call `requestPriceUpdatesWithCallback()`, paying fees.
3. Provider calls `executeCallback()` — `_state.providers[provider].accruedFeesInWei` grows.
4. Provider attempts to withdraw: there is no `withdraw()` function to call.
5. Provider attempts `withdrawAsFeeManager(providerAddress, amount)` — reverts with `"Only fee manager"` because `msg.sender != address(0)`.
6. Provider's accrued fees are permanently locked with no recovery path. [3](#0-2) [1](#0-0)
