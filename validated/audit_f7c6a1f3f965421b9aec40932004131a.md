### Title
`check_tx_fee` Passes Serialized Size Instead of Transaction Weight to `FeeRate::fee()`, Allowing Compute-Heavy Transactions to Bypass Minimum Fee Rate Enforcement - (`tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee()` in `tx-pool/src/util.rs` passes `tx_size` (serialized bytes) to `FeeRate::fee()`, which is documented to expect `weight` (= `max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`). For compute-heavy transactions where `cycles * DEFAULT_BYTES_PER_CYCLES > size`, weight is significantly larger than size, so the minimum fee check underestimates the required fee. This allows an unprivileged RPC caller to submit transactions that pass pool admission with an effective fee rate (shannons/kilo-weight) below `min_fee_rate`. The same unit mismatch exists in `calculate_min_replace_fee`, which underestimates the extra fee required for RBF replacement of compute-heavy transactions.

---

### Finding Description

`FeeRate` is explicitly typed as **shannons per kilo-weight**: [1](#0-0) 

`FeeRate::fee()` takes `weight` and computes `fee = fee_rate * weight / 1000`: [2](#0-1) 

Transaction weight is defined as `max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`: [3](#0-2) 

However, `check_tx_fee()` passes `tx_size` (raw serialized bytes) to `FeeRate::fee()` instead of weight. The code comment acknowledges the unit mismatch but dismisses it as a "cheap check": [4](#0-3) 

This is the **only** fee rate check before pool admission. `check_tx_fee` is called in `pre_check` before cycle verification: [5](#0-4) 

There is no second weight-based fee check after `verify_rtx` runs the scripts and determines actual cycles. The transaction enters the pool with whatever fee rate it has.

The same unit mismatch exists in `calculate_min_replace_fee`, which computes the extra RBF fee using `size` instead of weight: [6](#0-5) 

By contrast, all internal pool fee rate calculations correctly use `get_transaction_weight(size, cycles)`: [7](#0-6) 

---

### Impact Explanation

For a compute-heavy transaction where `cycles * DEFAULT_BYTES_PER_CYCLES > size`, weight can be substantially larger than size. With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`, weight exceeds size when `cycles > size / 0.000_170_571_4`. For a 1 KB transaction at the maximum allowed cycles (~70 million), weight ≈ 11,940 bytes — roughly **12× the serialized size**. [8](#0-7) 

The minimum fee check computes `min_fee = min_fee_rate * size / 1000` but the correct check should be `min_fee = min_fee_rate * weight / 1000`. A transaction that should require ~12,000 shannons minimum fee can pass with only ~1,000 shannons. Such transactions enter the pool with an effective fee rate (shannons/kilo-weight) far below `min_fee_rate`.

For RBF: `calculate_min_replace_fee` underestimates the extra fee required for a compute-heavy replacement transaction by the same factor, weakening the RBF economic disincentive.

---

### Likelihood Explanation

The attack path is directly reachable by any unprivileged user via the `send_transaction` RPC. The attacker crafts a transaction with a small serialized body but a script that consumes many cycles (e.g., a tight computation loop). No special privileges, keys, or majority hashpower are required. The `min_fee_rate` default is 1,000 shannons/kilo-weight, and the discrepancy scales with the cycle-to-size ratio, which is bounded only by `max_tx_verify_cycles`. [9](#0-8) 

---

### Recommendation

After `verify_rtx` completes and the actual cycles are known, perform a second fee rate check using the true weight:

```rust
// After verification, re-check fee rate against actual weight
let weight = get_transaction_weight(tx_size, completed.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

For `calculate_min_replace_fee`, replace `size` with `get_transaction_weight(size, cycles)` where `cycles` is the declared (or verified) cycle count of the replacement transaction.

---

### Proof of Concept

1. Craft a transaction with a small serialized size (e.g., 200 bytes) but a script that consumes ~50 million cycles.
2. Compute `weight = max(200, 50_000_000 * 0.000_170_571_4) ≈ 8,529`.
3. With `min_fee_rate = 1000` shannons/kilo-weight, the correct minimum fee is `1000 * 8529 / 1000 = 8,529` shannons.
4. But `check_tx_fee` computes `min_fee = 1000 * 200 / 1000 = 200` shannons.
5. Submit the transaction with a fee of 201 shannons via `send_transaction` RPC.
6. The transaction passes `check_tx_fee` and enters the pool, despite having an effective fee rate of `1000 * 201 / 8529 ≈ 23.6` shannons/kilo-weight — far below the configured `min_fee_rate` of 1,000. [10](#0-9) [11](#0-10)

use crate::core::Capacity;

/// shannons per kilo-weight
#[derive(Clone, Copy, Default, Debug, PartialEq, Eq, PartialOrd, Ord)]
pub struct FeeRate(pub u64);

const KW: u64 = 1000;

impl FeeRate {
    /// Calculates the fee rate from a total fee and weight.
    pub fn calculate(fee: Capacity, weight: u64) -> Self {
        if weight == 0 {
            return FeeRate::zero();
        }
        FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
    }

    /// Creates a fee rate from shannons per kilo-weight.
    pub const fn from_u64(fee_per_kw: u64) -> Self {
        FeeRate(fee_per_kw)
    }

    /// Returns the fee rate as shannons per kilo-weight.
    pub const fn as_u64(self) -> u64 {
        self.0
    }

    /// Creates a zero fee rate.
    pub const fn zero() -> Self {
        Self::from_u64(0)
    }

    /// Calculates the fee for a given weight.
    pub fn fee(self, weight: u64) -> Capacity {
        let fee = self.0.saturating_mul(weight) / KW;
        Capacity::shannons(fee)
    }

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
