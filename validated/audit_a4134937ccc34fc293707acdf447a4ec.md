### Title
Permissionless `initialize` Allows Permanent DoS on `InitializePublisher` — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary

The `initialize` instruction has no privileged access control. Any account can call it first and set an arbitrary, uncontrollable pubkey as `authority`. Because the config PDA is a singleton and there is no `update_authority` instruction, this permanently prevents `InitializePublisher` from ever succeeding.

---

### Finding Description

`initialize` only validates that the payer is a signer and writable: [1](#0-0) 

`validate_payer` imposes no identity constraint — any funded account qualifies: [2](#0-1) 

The `authority` field is raw instruction data with no validation that it corresponds to the payer or any privileged key: [3](#0-2) 

The config PDA is a global singleton derived only from `CONFIG_SEED`: [4](#0-3) 

`accounts::config::create` rejects re-initialization via the `format` magic check: [5](#0-4) 

The complete instruction set is exactly three entries — `Initialize`, `SubmitPrices`, `InitializePublisher` — with no `update_authority` or `reinitialize`: [6](#0-5) 

`InitializePublisher` enforces that the caller's key matches `config.authority` and is a signer: [7](#0-6) 

---

### Impact Explanation

If an attacker calls `initialize` before the legitimate operator — setting `authority` to a pubkey whose private key is unknown or discarded — then:

- `system_instruction::create_account` for the config PDA succeeds once and can never be repeated.
- Every subsequent `InitializePublisher` call fails at `validate_authority` because no one can produce a valid signature for the stored authority key.
- No publisher can ever be onboarded. All price-store functionality is permanently bricked for this program deployment.

---

### Likelihood Explanation

The attack requires no privileges. The attacker needs only enough SOL for rent and transaction fees. The vulnerability window is the gap between program deployment and the operator's first `initialize` call. On Solana, this window can also be exploited via priority-fee front-running of the operator's own `initialize` transaction, since all pending transactions are observable. The attack is one transaction, irreversible, and costs ~0.001 SOL.

---

### Recommendation

Add an access-control check in `initialize` that restricts who may call it. The standard pattern is to require the payer to be the program's upgrade authority (fetched from the `BPFLoaderUpgradeable` program data account), or to hardcode an expected initializer pubkey. At minimum, require that `args.authority` equals `payer.key`, so the caller must hold the authority key at initialization time and cannot set an uncontrollable pubkey.

---

### Proof of Concept

```
1. Deploy the pyth-price-store program.
2. Before the operator calls initialize, attacker submits:
     Instruction::Initialize {
         config_bump: <canonical bump>,
         authority: [0u8; 32],   // or any pubkey with no known private key
     }
   with any funded keypair as payer/signer.
3. Config PDA is created; authority = [0u8; 32].
4. Operator attempts initialize → fails (account already exists).
5. Operator attempts InitializePublisher → fails at validate_authority:
     authority.key.to_bytes() == config.authority  // [0u8;32] ≠ operator key
6. No recovery instruction exists. Program is permanently DoS'd.
``` [8](#0-7) [9](#0-8)

pub fn initialize(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    args: &InitializeArgs,
) -> ProgramResult {
    let mut accounts = accounts.iter();
    let payer = validate_payer(accounts.next())?;
    let config = validate_config(accounts.next(), args.config_bump, program_id, true)?;
    let system = validate_system(accounts.next())?;

    let lamports = (Rent::get()?).minimum_balance(accounts::config::SIZE);

    invoke_signed(
        &system_instruction::create_account(
            payer.key,
            config.key,
            lamports,
            accounts::config::SIZE
                .try_into()
                .expect("unexpected overflow"),
            program_id,
        ),
        &[payer.clone(), config.clone(), system.clone()],
        &[&[CONFIG_SEED.as_bytes(), &[args.config_bump]]],
    )?;

    accounts::config::create(*config.data.borrow_mut(), args.authority)?;

    Ok(())

pub fn validate_payer<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let payer = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, payer.is_signer);
    ensure!(ProgramError::InvalidArgument, payer.is_writable);
    Ok(payer)
}

pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)
