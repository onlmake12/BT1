### Title
TOCTOU in Tx-Pool Two-Phase Admission Allows CPU Exhaustion via Concurrent Conflicting Transactions — (`tx-pool/src/process.rs`)

---

### Summary

The CKB transaction pool (`TxPoolService`) separates transaction admission into two phases: a cheap `pre_check` (under a **read lock**) and an expensive `verify_rtx` (under **no lock**), followed by `submit_entry` (under a **write lock**). Because the pool's "cell consumed" state is only updated at `submit_entry`, multiple concurrent transactions spending the same live cell all pass `pre_check` simultaneously, all undergo full script verification, and only one ultimately succeeds. An unprivileged attacker can exploit this window to force the node to perform arbitrarily expensive, wasted script verification work.

---

### Finding Description

`TxPoolService::_process_tx` in `tx-pool/src/process.rs` follows this sequence:

1. **`pre_check`** — acquires a **read lock** on `tx_pool`, resolves the transaction against `pool_map`, checks that input cells are live, then **releases the lock**.
2. **`verify_rtx`** — runs full CKB-VM script verification **with no lock held** (this is the expensive step).
3. **`submit_entry`** — acquires a **write lock**, re-checks for conflicts, and only here marks the cell as consumed in `pool_map`. [1](#0-0) 

The `pre_check` function uses `with_tx_pool_read_lock`, which allows multiple concurrent readers: [2](#0-1) 

The `PoolCell` cell provider — used during `pre_check` to determine if a cell is live — only inspects `pool_map.edges` (cells already committed to the pool). It has **no visibility into the verify queue**: [3](#0-2) 

The async path (`resumeble_process_tx`) used for both remote P2P and local RPC submissions only deduplicates by txid, not by input cell conflict, before enqueuing: [4](#0-3) 

`VerifyMgr` spawns `max_tx_verify_workers` concurrent workers, each independently calling `_process_tx`: [5](#0-4) 

**Attack sequence:**

1. Attacker owns a live cell `C` with a computationally expensive lock script (e.g., near the cycle limit).
2. Attacker crafts `N` distinct transactions `T1…TN`, each spending `C` but with different outputs or witnesses (so they have different txids).
3. Attacker submits all `N` transactions via RPC (`send_transaction`) or P2P relay in rapid succession.
4. Each `Ti` passes `non_contextual_verify` and `verify_queue_contains` (different txids), and is enqueued in the verify queue.
5. Workers pick up all `Ti` concurrently. Each calls `pre_check` under a **read lock** — `C` appears live in `pool_map` for all of them simultaneously.
6. All `N` workers proceed to `verify_rtx` — full CKB-VM execution, no lock held — consuming up to `N × max_cycles` CPU cycles.
7. The first worker to reach `submit_entry` succeeds; all others are rejected with `Reject::Resolve(OutPointError::Dead(...))`.

The conflict check in `submit_entry` correctly prevents double-spending in the pool, but only **after** the expensive verification has already been performed for all `N` transactions: [6](#0-5) 

---

### Impact Explanation

An unprivileged attacker (RPC caller or P2P peer) can force the node to perform `N` full CKB-VM script verifications for transactions that will all but one be discarded. With a script consuming the maximum allowed cycles, this multiplies the CPU cost by `N` (bounded by the verify queue size and `max_tx_verify_workers`). Sustained attack degrades transaction processing throughput, delays block assembly, and can cause the node to fall behind in sync — a targeted availability impact.

---

### Likelihood Explanation

The attack requires only a single live cell and the ability to submit transactions via RPC or P2P. No privileged access, no key leakage, no majority hashpower, and no Sybil attack is needed. The verify queue accepts transactions without checking for input conflicts, making this straightforwardly reachable by any node operator or connected peer.

---

### Recommendation

Introduce an **input-conflict check at enqueue time** (before adding to the verify queue) that inspects both `pool_map` and the current verify queue contents. Specifically:

- In `resumeble_process_tx` / `enqueue_verify_queue`, before enqueuing, scan the verify queue for any existing entry that shares an input `OutPoint` with the incoming transaction and reject the duplicate with `Reject::Resolve(OutPointError::Dead(...))`.
- Alternatively, extend `PoolCell` (or a new `VerifyQueueCell` overlay) to treat cells currently referenced by verify-queue entries as `Dead`, so `pre_check` itself rejects conflicting transactions before they reach `verify_rtx`.

This mirrors the fix recommended in the external report: update state (mark the cell as "pending consumption") **before** performing the expensive operation, not after.

---

### Proof of Concept

```
# Attacker controls cell C with a script consuming ~10M cycles.
# Craft N=10 transactions T1..T10, each spending C, different outputs.

for i in 1..=10 {
    let tx_i = build_tx_spending(cell_C, output_i);  // distinct txid each time
    rpc.send_transaction(tx_i);  // all enqueued before any completes verify_rtx
}

# Result:
# - All 10 pass pre_check (read lock; C still live in pool_map for all)
# - All 10 run verify_rtx concurrently (~10 × 10M = 100M cycles consumed)
# - 9 are rejected at submit_entry (write lock; C now dead after first succeeds)
# - Node CPU saturated; legitimate tx processing stalled

    pub(crate) async fn _process_tx(
        &self,
        tx: TransactionView,
        declared_cycles: Option<Cycle>,
        command_rx: Option<&mut watch::Receiver<ChunkCommand>>,
    ) -> Option<(Result<Completed, Reject>, Arc<Snapshot>)> {
        let wtx_hash = tx.witness_hash();
        let instant = Instant::now();
        let is_sync_process = command_rx.is_none();

        let (ret, snapshot) = self.pre_check(&tx).await;

        let (tip_hash, rtx, status, fee, tx_size) = try_or_return_with_snapshot!(ret, snapshot);

        let verify_cache = self.fetch_tx_verify_cache(&tx).await;
        let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
        let tip_header = snapshot.tip_header();
        let tx_env = Arc::new(status.with_env(tip_header));

        let verified_ret = verify_rtx(
            Arc::clone(&snapshot),
            Arc::clone(&rtx),
            tx_env,
            &verify_cache,
            max_cycles,
            command_rx,
        )
        .await;

        let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

        if let Some(declared) = declared_cycles
            && declared != verified.cycles
        {
            info!(
                "process_tx declared cycles not match verified cycles, declared: {}, verified: {}, tx_hash: {}",
                declared,
                verified.cycles,
                tx.hash()
            );
            return Some((
                Err(Reject::DeclaredWrongCycles(declared, verified.cycles)),
                snapshot,
            ));
        }

        let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);

        let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
