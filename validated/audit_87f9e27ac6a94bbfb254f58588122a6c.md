### Title
Provider Accrued Fees Permanently Inaccessible When No Fee Manager Is Set - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.sol` accumulates provider fees in `ProviderInfo.accruedFeesInWei` but provides no direct `withdraw()` function for providers. The sole withdrawal path for provider fees is `withdrawAsFeeManager()`, which requires `msg.sender == _state.providers[provider].feeManager`. Because `feeManager` defaults to `address(0)` on registration, any provider who never calls `setFeeManager()` has their accrued fees permanently inaccessible — no EOA can satisfy `msg.sender == address(0)`.

### Finding Description
`Echo.sol::registerProvider()` initialises a `ProviderInfo` struct whose `feeManager` field defaults to `address(0)`. [1](#0-0) 

After registration, every fulfilled callback credits the provider's `accruedFeesInWei`: [2](#0-1) 

The only withdrawal path for those fees is `withdrawAsFeeManager()`: [3](#0-2) 

The guard `require(msg.sender == _state.providers[provider].feeManager, "Only fee manager")` is unsatisfiable when `feeManager == address(0)`, because `msg.sender` is never `address(0)` in a live transaction. Unlike `Entropy.sol`, which exposes a direct `withdraw(uint128 amount)` callable by the provider itself: [4](#0-3) 

`Echo.sol` has no equivalent. The `IEcho` interface confirms only two withdrawal selectors exist — `withdrawFees` (admin-only) and `withdrawAsFeeManager` (fee-manager-only): [5](#0-4) 

### Impact Explanation
Any provider who registers via `registerProvider()` and accrues fees through fulfilled callbacks, but never calls `setFeeManager()`, cannot withdraw their earned ETH. The funds sit in `_state.providers[provider].accruedFeesInWei` with no reachable code path to extract them. If the provider is a smart contract without a `setFeeManager` call path, the lock is permanent. Even for EOA providers, the funds are inaccessible until an additional governance-like step is taken — a step that is not documented or enforced at registration time.

### Likelihood Explanation
`registerProvider()` imposes no requirement to set a fee manager before or after registration: [6](#0-5) 

Any provider integrating Echo without reading the full contract will omit `setFeeManager()`. The default `feeManager = address(0)` is silent — no event, no revert, no warning. Providers porting from Entropy (which has a direct `withdraw()`) are especially likely to miss this requirement.

### Recommendation
Add a direct `withdraw(uint128 amount)` function callable by the provider themselves, mirroring `Entropy.sol`:

```solidity
function withdraw(uint128 amount) external {
    ProviderInfo storage info = _state.providers[msg.sender];
    require(info.isRegistered, "Provider not registered");
    require(info.accruedFeesInWei >= amount, "Insufficient balance");
    info.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "withdrawal failed");
    emit FeesWithdrawn(msg.sender, amount);
}
```

Alternatively, enforce that `feeManager` is set to a non-zero address at `registerProvider()` time.

### Proof of Concept
1. Provider EOA `0xPROV` calls `registerProvider(baseFee, feedFee, gasFee)`. `_state.providers[0xPROV].feeManager == address(0)`.
2. Users call `requestPriceUpdatesWithCallback{value: fee}(0xPROV, ...)` repeatedly.
3. Callbacks are executed; `_state.providers[0xPROV].accruedFeesInWei` grows to, say, 1 ETH.
4. `0xPROV` calls `withdrawAsFeeManager(0xPROV, 1 ether)` → reverts: `"Only fee manager"` because `address(0) != 0xPROV`.
5. No other function in `Echo.sol` allows `0xPROV` to retrieve the 1 ETH.
6. The 1 ETH remains locked in the contract until `0xPROV` separately calls `setFeeManager(0xPROV)` — a step with no on-chain enforcement or documentation at registration time. [3](#0-2) [7](#0-6)
