### Title
Active Subscription Not Deactivated When Balance Drops Below Minimum After Fee Deduction in `updatePriceFeeds` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

In `Scheduler.sol`, the `updatePriceFeeds()` function deducts both a Pyth protocol fee and a keeper fee from `status.balanceInWei`. After these deductions, there is no check to determine whether the remaining balance has fallen below the required minimum (`minimumBalancePerFeed * numFeeds`). The subscription is never automatically deactivated, leaving it in an inconsistent state: `params.isActive == true` while `status.balanceInWei < minimumBalance`. Any unprivileged keeper can repeatedly call `updatePriceFeeds()` to drain the balance further, and the subscription continues to appear in `getActiveSubscriptions()`.

### Finding Description

The Scheduler contract enforces a minimum balance invariant: an active subscription must hold at least `getMinimumBalance(numFeeds)` wei. This is enforced at creation time, at `updateSubscription()`, at `addFunds()`, and at `withdrawFunds()`. However, the fee-deduction path inside `updatePriceFeeds()` is exempt from this check.

`updatePriceFeeds()` deducts the Pyth fee directly: [1](#0-0) 

Then calls `_processFeesAndPayKeeper()` which deducts the keeper fee: [2](#0-1) 

Neither `updatePriceFeeds()` nor `_processFeesAndPayKeeper()` checks whether `status.balanceInWei` has fallen below `minimumBalance` after the deductions, and neither calls `_removeFromActiveSubscriptions()` or sets `params.isActive = false`. [3](#0-2) 

By contrast, `withdrawFunds()` correctly enforces the invariant before allowing a balance reduction: [4](#0-3) 

And `addFunds()` enforces it after adding funds: [5](#0-4) 

The `updatePriceFeeds()` path has no equivalent guard.

### Impact Explanation

After repeated keeper updates drain `status.balanceInWei` below `minimumBalance`, the subscription:

1. Remains `isActive = true` and stays in `_state.activeSubscriptionIds`, so `getActiveSubscriptions()` returns it as a valid active subscription.
2. Can continue to receive keeper calls to `updatePriceFeeds()` as long as the remaining balance covers the immediate per-call fee, draining it to zero.
3. Violates the protocol invariant that active subscriptions are adequately funded, corrupting the active subscription list used by keepers to discover work. [6](#0-5) 

### Likelihood Explanation

This is triggered by normal protocol operation. Any keeper (fully unprivileged, no special role required) calls `updatePriceFeeds()` on any active subscription. Every successful update deducts fees. Once the balance crosses below the minimum, the invariant is broken. No malicious intent is required — this happens in the ordinary course of keeper activity on any subscription that is not continuously topped up. [7](#0-6) 

### Recommendation

After `_processFeesAndPayKeeper()` returns in `updatePriceFeeds()`, check whether `status.balanceInWei` has fallen below `getMinimumBalance(params.priceIds.length)`. If so, set `params.isActive = false` and call `_removeFromActiveSubscriptions(subscriptionId)` to restore the invariant and emit a `SubscriptionDeactivated` event. This mirrors the pattern already used in `updateSubscription()` when a manager explicitly deactivates a subscription. [8](#0-7) 

### Proof of Concept

1. A subscription is created with `numFeeds = 2`, funded exactly at `minimumBalance = 2 * minimumBalancePerFeed`.
2. A keeper calls `updatePriceFeeds()` repeatedly. Each call deducts `pythFee + keeperFee` from `status.balanceInWei`.
3. After enough calls, `status.balanceInWei < minimumBalance`. The subscription is **not** deactivated.
4. `getActiveSubscriptions()` still returns this subscription as active.
5. The keeper continues calling `updatePriceFeeds()` until `status.balanceInWei` is fully exhausted (down to 0), at which point the call finally reverts with `InsufficientBalance`.
6. The subscription has been completely drained while remaining in the active list the entire time, with `params.isActive == true`.

This is confirmed by the existing test `testAddFundsEnforcesMinimumBalance` which explicitly demonstrates that repeated `updatePriceFeeds()` calls drain the balance below minimum while the subscription remains active: [9](#0-8)

        // Calculate how many results to return (bounded by maxResults and remaining items)

                startIndex + i
