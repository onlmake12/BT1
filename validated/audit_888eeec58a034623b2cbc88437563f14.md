Audit Report

## Title
Wrong Column Family in `migrate_transaction_info` Corrupts Transaction Info Storage — (File: `util/migrate/src/migrations/table_to_struct.rs`)

## Summary

`migrate_transaction_info` in `ChangeMoleculeTableToStruct` calls `db.traverse(COLUMN_UNCLES, ...)` at line 93 instead of `db.traverse(COLUMN_TRANSACTION_INFO, ...)`. This causes the migration to read 240-byte uncle `HeaderView` records and write truncated 224-byte versions of them into `COLUMN_TRANSACTION_INFO` keyed by uncle hashes, while the actual `COLUMN_TRANSACTION_INFO` records are never read or rewritten. Any CKB node upgrading from a pre-v0.35.0 database triggers this migration automatically on startup, permanently corrupting the transaction info column family.

## Finding Description

In `util/migrate/src/migrations/table_to_struct.rs`, `migrate_transaction_info` (line 77) is responsible for converting old molecule-table-encoded `TransactionInfo` records to the new struct encoding. The write closure at line 85 correctly targets `COLUMN_TRANSACTION_INFO`:

```rust
wb.put(COLUMN_TRANSACTION_INFO, key, &value[16..])?;
```

However, the `db.traverse(...)` call at line 93 reads from `COLUMN_UNCLES` instead of `COLUMN_TRANSACTION_INFO`:

```rust
let (_count, nk) =
    db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
```

`COLUMN_UNCLES` stores uncle block `HeaderView` records of 240 bytes each. The guard `if value.len() != TRANSACTION_INFO_SIZE` (52) evaluates to `true` for every 240-byte uncle record (240 ≠ 52), so every uncle entry passes the guard. The migration strips 16 bytes from the front and writes the remaining 224 bytes into `COLUMN_TRANSACTION_INFO` keyed by uncle hash — not transaction hash. The actual `COLUMN_TRANSACTION_INFO` records are never traversed, never rewritten, and remain in the old table-encoded format unreadable by post-migration code.

All three sibling functions are consistent between their traversal source and write target: `migrate_header` traverses and writes `COLUMN_BLOCK_HEADER` (line 40); `migrate_uncles` traverses and writes `COLUMN_UNCLES` (line 66); `migrate_epoch_ext` traverses and writes `COLUMN_EPOCH` (line 120). Only `migrate_transaction_info` has a mismatch.

The migration is registered unconditionally in `util/migrate/src/migrate.rs` at line 26 and runs automatically on node startup when the stored DB version is below `20200703124523`.

## Impact Explanation

**Medium (2001–10000 points) — Suboptimal implementation of CKB state storage mechanism.**

After `ChangeMoleculeTableToStruct` runs: (1) `COLUMN_TRANSACTION_INFO` is overwritten with truncated uncle `HeaderView` bytes keyed by uncle hashes; (2) old table-encoded `TransactionInfo` records remain in the old format, unreadable by post-migration deserialization code; (3) `get_transaction_info(tx_hash)` lookups return `None` or fail to deserialize, causing `get_transaction` RPC calls to return incorrect transaction status; (4) the tx-pool's duplicate-detection logic, which relies on `TransactionInfo` to check whether a transaction is already confirmed, operates on corrupted data.

## Likelihood Explanation

The migration is triggered automatically by the standard node upgrade procedure — no attacker involvement is required. Any node operator who ran a CKB node on a database created before v0.35.0 (released 2020-07-03) and then upgraded would have triggered this migration. The code path is deterministic and unconditional: if the stored DB version is below `20200703124523`, the migration runs on the next node startup. No special privileges, leaked keys, or external conditions are needed beyond performing a routine software upgrade.

## Recommendation

Change line 93 of `util/migrate/src/migrations/table_to_struct.rs` to traverse `COLUMN_TRANSACTION_INFO` instead of `COLUMN_UNCLES`:

```rust
let (_count, nk) =
    db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
```

Additionally, add regression tests that verify each migration sub-function reads from and writes to the correct column family.

## Proof of Concept

1. Create a RocksDB database at version `20191127135521` (pre-v0.35.0) with entries in `COLUMN_TRANSACTION_INFO` (column `"5"`) and entries in `COLUMN_UNCLES` (column `"11"`).
2. Run `Migrate::new(...).migrate(db, false)`, which calls `ChangeMoleculeTableToStruct::migrate`.
3. After migration, inspect `COLUMN_TRANSACTION_INFO`: it will contain entries keyed by uncle hashes with 224-byte values (truncated uncle `HeaderView` data), not the original transaction info records.
4. Inspect the original transaction info keys: they will either be absent or still in old table-encoded format, unreadable by post-migration deserialization code.
5. Call `get_transaction_info` for any previously stored transaction hash: it returns `None` or fails to deserialize.
