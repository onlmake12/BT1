### Title
Insufficient Per-Request DB Read Bound in GetLastStateProof Handler — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
The guard limiting `difficulties.len()` to `GET_LAST_STATE_PROOF_LIMIT` (1000) does not account for the O(log N) DB reads incurred per difficulty value during binary search and MMR root computation. A remote peer can send 1000 difficulty values with `last_n_blocks=0`, causing O(1000 × log N) DB reads per request.

### Impact Explanation
On a 10M-block chain: ~23,000 DB reads per request. A sustained flood from multiple peers could saturate DB I/O on nodes with the light client protocol enabled. Impact is **resource exhaustion / degraded availability**, not a crash in the strict sense.

### Likelihood Explanation
Moderate. The light client protocol server must be explicitly enabled. The amplification factor is logarithmic (not linear), and modern RocksDB with warm caches handles many reads efficiently. Peer banning does not apply to expensive-but-valid requests.

### Recommendation
Add a tighter bound on `difficulties.len()` independent of `last_n_blocks`, e.g., cap it at ~50–100. Alternatively, bound the total binary search steps across all difficulties (not just the count of difficulties) by tracking steps consumed and aborting early.
