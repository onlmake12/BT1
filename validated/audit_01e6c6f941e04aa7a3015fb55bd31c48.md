### Title
Unsafe ETH Refund via `address.transfer()` Causes DoS for Contract Callers - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` refunds excess ETH to the caller using `payable(msg.sender).transfer(...)`, which forwards only 2300 gas. Any contract caller whose `receive()` or `fallback()` function consumes more than 2300 gas will have the entire `verifyUpdate()` call revert, making the function permanently unusable for that caller.

### Finding Description
In `PythLazer.sol`, the `verifyUpdate()` function accepts a fee and refunds any excess ETH to `msg.sender`:

```solidity
// lazer/contracts/evm/src/PythLazer.sol, line 74-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
```

`address.transfer()` is hardcoded to forward exactly 2300 gas. Since EIP-1884 (Istanbul), many common operations (e.g., `SLOAD`, emitting events, updating mappings) cost more than 2300 gas. Any contract integrating Pyth Lazer that has a non-trivial `receive()` or `fallback()` — such as a multisig wallet, a proxy, or a DeFi protocol — will cause `verifyUpdate()` to revert whenever it sends `msg.value > verification_fee`.

The root cause is exclusively in Pyth code: the use of `.transfer()` instead of the recommended low-level `.call{value: ...}("")` pattern. [1](#0-0) 

### Impact Explanation
Any contract caller of `verifyUpdate()` that overpays the fee (e.g., to tolerate future fee increases, or due to imprecise fee estimation) will have the call permanently revert if its receive logic exceeds 2300 gas. This breaks Lazer price feed consumption for all such integrators, effectively a **DoS on the core Lazer verification path** for contract-based callers.

### Likelihood Explanation
DeFi protocols routinely call oracle update functions from smart contracts and commonly overpay fees as a safety margin. Multisigs (Gnosis Safe), proxy contracts, and any contract with a non-trivial `receive()` are all affected. The condition `msg.value > verification_fee` is easily triggered by any caller that does not compute the exact fee on-chain.

### Recommendation
Replace `address.transfer()` with a low-level `call` and check the return value:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Refund failed");
}
```

This forwards all available gas to the recipient and does not impose the 2300-gas restriction.

### Proof of Concept
1. Deploy a contract `Integrator` with a `receive()` function that writes to storage (costs >2300 gas).
2. Call `PythLazer.verifyUpdate{value: verification_fee + 1 wei}(update)` from `Integrator`.
3. The `payable(msg.sender).transfer(1 wei)` call at line 76 reverts due to out-of-gas in the recipient's `receive()`.
4. The entire `verifyUpdate()` transaction reverts — the Lazer price update is never processed. [2](#0-1)
