### Title
Unbounded `while` Loop in `Echo.executeCallback` Over Fulfilled Sequence Numbers Causes Permanent DoS — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past all cleared (fulfilled) requests. If a large number of requests accumulate between `firstUnfulfilledSeq` and the next active request, the loop performs proportionally many cold storage reads, potentially exceeding the block gas limit and permanently preventing fulfillment of the remaining request — locking the requester's funds.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs:

```solidity
// Echo.sol lines 169–174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs one or two storage reads:

```solidity
// Echo.sol lines 310–321
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];
    }
}
``` [2](#0-1) 

The developer explicitly acknowledged this in a code comment:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup
// for each sequence number.
``` [3](#0-2) 

The loop iterates from `firstUnfulfilledSeq` to the first still-active request. If that gap is large (e.g., N cleared requests), the loop performs N storage reads. At 2,100 gas per cold `SLOAD`, N ≈ 14,000 iterations exhausts the 30M block gas limit.

---

### Impact Explanation

When the gas cost of the `while` loop exceeds the block gas limit, `executeCallback` becomes permanently uncallable for the remaining active request. The requester's fee (stored in `req.fee`) is locked in the contract with no recovery path, since there is no timeout-based refund mechanism. This constitutes a permanent loss of user funds and a complete DoS of the Echo callback fulfillment path for that request.

---

### Likelihood Explanation

The attack is reachable by any unprivileged user:

1. An attacker (or organic traffic) creates N requests via `requestPriceUpdatesWithCallback`, paying the required fee per request.
2. Requests 2 through N are fulfilled via `executeCallback`. After each fulfillment, the `while` loop tries to advance `firstUnfulfilledSeq` from 1, but request 1 is still active, so the loop does not advance.
3. When request 1 is finally fulfilled, the `while` loop iterates from seq 1 through seq N — N iterations of storage reads.
4. If N is large enough, the transaction reverts with out-of-gas, making request 1 permanently unfulfillable.

The cost to the attacker is N × (minimum fee per request). With a low `pythFeeInWei` and low provider fees, this attack can be mounted cheaply. In a live deployment with organic traffic, the gap can grow naturally without any deliberate attack.

---

### Recommendation

Replace the unbounded `while` loop with a bounded scan (e.g., advance at most a fixed number of steps per call), or eliminate the `firstUnfulfilledSeq` pointer entirely and use a doubly-linked list of active requests as the developer's own TODO comment suggests:

```solidity
// TODO: a better solution would be a doubly-linked list of active requests.
``` [3](#0-2) 

Alternatively, remove the `firstUnfulfilledSeq` advancement from `executeCallback` and expose a separate, permissioned `advanceFirstUnfulfilledSeq(uint256 maxSteps)` function that callers invoke explicitly with a bounded step count.

---

### Proof of Concept

1. Deploy `Echo` with `pythFeeInWei = 1 wei` and a provider with minimal fees.
2. Call `requestPriceUpdatesWithCallback` N times (e.g., N = 15,000), creating requests with sequence numbers 1 through N.
3. Call `executeCallback` for requests 2 through N in any order. After each call, the `while` loop attempts to advance `firstUnfulfilledSeq` from 1 but stops immediately because request 1 is still active.
4. Now call `executeCallback` for request 1. The `while` loop starts at `firstUnfulfilledSeq = 1`, finds request 1 cleared, increments to 2, finds request 2 cleared, ..., increments to N+1. This performs N cold `SLOAD`s (≈ N × 2,100 gas). For N = 15,000, gas ≈ 31.5M > 30M block gas limit → transaction reverts.
5. Request 1 can never be fulfilled. The requester's fee is permanently locked.
