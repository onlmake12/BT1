### Title
Unfulfilled Echo Request Provider Fees Permanently Locked With No Refund Mechanism - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
In `Echo.sol`, when a user calls `requestPriceUpdatesWithCallback`, the provider fee portion of `msg.value` is stored in the request struct (`req.fee`) but is never credited to any withdrawable balance until the provider calls `executeCallback`. There is no cancellation, timeout, or refund function. If the assigned provider never fulfills the request, the user's provider fee ETH is permanently locked in the contract with no recovery path for the user, the admin, or anyone else.

### Finding Description
In `requestPriceUpdatesWithCallback`, the user pays `msg.value`. The Pyth protocol portion is immediately credited to `_state.accruedFeesInWei` (recoverable by admin via `withdrawFees`), while the provider portion is stored in the request:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
// ...
_state.accruedFeesInWei += _state.pythFeeInWei;
```

The `req.fee` ETH sits in the contract's raw balance — not in `_state.accruedFeesInWei`, not in any `providers[x].accruedFeesInWei`. It is only credited to a provider when `executeCallback` is called:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

There is no `cancelRequest`, `refundRequest`, or admin-recovery function anywhere in `Echo.sol` or `IEcho.sol`. The developers themselves acknowledge the gap with a TODO comment at the exact location:

```solidity
// TODO: there also needs to be some penalty mechanism in case the expected provider
// doesn't execute the callback.
```

The contract's accounting invariant is: `contract ETH balance = _state.accruedFeesInWei + Σ req.fee (all pending requests)`. The `_state.accruedFeesInWei` portion is recoverable; the `Σ req.fee` portion for unfulfilled requests is not.

### Impact Explanation
Any user who calls `requestPriceUpdatesWithCallback` and whose assigned provider subsequently goes offline, loses their key, or simply refuses to fulfill will permanently lose their provider fee. Over time — especially if a provider becomes inactive — a growing pool of ETH accumulates in the contract that no party (user, provider, admin) can recover. This is a direct loss of user funds with no mitigation path.

### Likelihood Explanation
Providers are external parties. A provider can go offline at any time due to infrastructure failure, key loss, or deliberate abandonment. The exclusivity period means only the assigned provider can fulfill during that window, and after it expires any provider can fulfill — but if no provider is incentivized to fulfill an old request (e.g., the fee is too low relative to current gas costs), the request remains permanently unfulfilled. This is a realistic, non-theoretical scenario.

### Recommendation
Add a cancellation/refund function that allows the original requester to reclaim `req.fee` after a configurable timeout period has elapsed without fulfillment. Alternatively, add an admin-controlled recovery function that can sweep stuck request fees to a treasury or return them to requesters.

### Proof of Concept
1. User calls `requestPriceUpdatesWithCallback{value: getFee(...)}(provider, ...)`.
   - `_state.accruedFeesInWei += pythFeeInWei` (Pyth portion credited).
   - `req.fee = msg.value - pythFeeInWei` stored in request struct.
   - Contract ETH balance increases by `msg.value`.
2. Provider goes offline and never calls `executeCallback`.
3. `req.fee` ETH remains in the contract's raw balance indefinitely.
4. User has no `cancelRequest` to call. Admin has no recovery function. `withdrawFees` only covers `_state.accruedFeesInWei`, not `req.fee`. `withdrawAsFeeManager` only covers `providers[x].accruedFeesInWei`, not `req.fee`.

        require(msg.sender == _state.admin, "Only admin can withdraw fees");
        require(_state.accruedFeesInWei >= amount, "Insufficient balance");

        (bool sent, ) = msg.sender.call{value: amount}("");
