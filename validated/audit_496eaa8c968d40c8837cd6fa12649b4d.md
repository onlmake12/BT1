### Title
`getPricesNoOlderThan` Staleness Check Uses Maximum Batch Timestamp Instead of Per-Feed Timestamp, Allowing Stale Individual Prices to Pass - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `getPricesNoOlderThan()` and `getEmaPricesNoOlderThan()` functions enforce a staleness check against `status.priceLastUpdatedAt`, which is the **maximum** publish time across all price feeds in the last submitted batch. However, the functions return individual per-feed prices that can be arbitrarily older than this maximum. An unprivileged keeper can submit a batch where one feed has a fresh timestamp (satisfying the `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` guard) and another feed has a very stale timestamp. The staleness check passes, but downstream readers receive stale prices they believe are fresh.

---

### Finding Description

In `_validateShouldUpdatePrices()`, the `updateTimestamp` is computed as the **maximum** publish time across all feeds in the batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [1](#0-0) 

This maximum timestamp is then stored as `status.priceLastUpdatedAt`:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` guard (1 hour) is also applied only against this maximum:

```solidity
uint256 minAllowedTimestamp = (block.timestamp > PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    : 0;
if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(...);
}
``` [3](#0-2) [4](#0-3) 

When a reader calls `getPricesNoOlderThan()`, the staleness check is:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();
prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [5](#0-4) 

The check passes if the **most recent** feed in the batch is within `age_seconds`, but `getPricesUnsafe()` returns the stored per-feed prices, which can have individually much older `publishTime` values. The same flaw exists in `getEmaPricesNoOlderThan()`. [6](#0-5) 

**Concrete attack scenario:**

A subscription tracks `[ETH/USD, AAPL/USD]`. A keeper (no access control on `updatePriceFeeds`) submits a batch where:
- `ETH/USD` publishTime = `now - 30s` (fresh)
- `AAPL/USD` publishTime = `now - 7 hours` (stale, e.g., after market close)

`updateTimestamp = max(now-30s, now-7h) = now-30s`. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` check passes (`30s < 1h`). `status.priceLastUpdatedAt = now-30s`.

A reader calls `getPricesNoOlderThan(subscriptionId, [ETH_USD, AAPL_USD], 60)`:
- `distance(now, now-30s) = 30s ≤ 60s` → check passes
- Returns ETH/USD (30s old, fresh) **and** AAPL/USD (7 hours old, stale)

The reader receives a 7-hour-old AAPL/USD price with no revert or warning, despite specifying a 60-second freshness requirement.

---

### Impact Explanation

Downstream protocols (e.g., lending, derivatives) integrating the Scheduler via `getPricesNoOlderThan()` rely on the function's staleness guarantee to make safe financial decisions. When the guarantee is silently violated, protocols can:

- Accept severely stale collateral valuations, enabling under-collateralized borrowing
- Execute trades or liquidations at prices that no longer reflect market conditions
- Suffer direct financial loss proportional to the price divergence over the stale period

The `Scheduler` is a production push-oracle contract intended for DeFi integrations. The impact is high because the function's name and signature explicitly promise a freshness bound that is not enforced per-feed.

---

### Likelihood Explanation

The entry path requires no privileged access. `updatePriceFeeds()` in `Scheduler.sol` has no access control — any address can act as a keeper. A malicious keeper can deliberately craft a batch with one fresh feed and one stale feed. Even without malicious intent, this occurs naturally whenever a subscription includes assets with different market hours (e.g., crypto + equities), which is a documented and expected use case per the code comments. [7](#0-6) 

---

### Recommendation

Replace the single `priceLastUpdatedAt` staleness check in `getPricesNoOlderThan()` and `getEmaPricesNoOlderThan()` with a per-feed check against each individual price's `publishTime`:

```solidity
function getPricesNoOlderThan(...) external view ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

This ensures every returned price individually satisfies the caller's freshness requirement, matching the semantic contract implied by the function name.

---

### Proof of Concept

1. Deploy `PulseSchedulerUpgradable` with a subscription containing `[ETH_USD, AAPL_USD]`.
2. As an unprivileged keeper, call `updatePriceFeeds()` with Pyth update data where `ETH_USD.publishTime = block.timestamp - 30` and `AAPL_USD.publishTime = block.timestamp - 25200` (7 hours).
3. The call succeeds: `_validateShouldUpdatePrices` computes `updateTimestamp = block.timestamp - 30`, which satisfies `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours`.
4. `status.priceLastUpdatedAt` is set to `block.timestamp - 30`.
5. A whitelisted reader calls `getPricesNoOlderThan(subscriptionId, [ETH_USD, AAPL_USD], 60)`.
6. The check `distance(block.timestamp, block.timestamp - 30) = 30 ≤ 60` passes.
7. The function returns AAPL/USD with `publishTime = block.timestamp - 25200` — 7 hours stale — without reverting.
