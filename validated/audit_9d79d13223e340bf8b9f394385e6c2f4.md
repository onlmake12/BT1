### Title
Future-Dated `priceLastUpdatedAt` Blocks Same-Block Updates and Delays Heartbeat Triggers in Scheduler - (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

In `Scheduler.sol`, the `updatePriceFeeds` function stores the maximum `publishTime` from the submitted price feeds as `status.priceLastUpdatedAt`. Because the Pyth contract accepts price feeds with timestamps up to `block.timestamp + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD` (10 seconds), a keeper who submits a price update with `publishTime = block.timestamp + 10` sets `priceLastUpdatedAt` to a value 10 seconds in the future. Any subsequent update in the same Ethereum block is then permanently blocked for that subscription, because the next update's `updateTimestamp` must be strictly greater than `priceLastUpdatedAt`, yet is bounded by the same ceiling of `block.timestamp + 10`.

### Finding Description

In `updatePriceFeeds`, the Pyth contract is called with a maximum timestamp of `curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD`:

```solidity
uint64 curTime = SafeCast.toUint64(block.timestamp);
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0,
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,  // = block.timestamp + 10
    ...
);
```

The resulting `updateTimestamp` (the maximum `publishTime` across all returned feeds) is then stored:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
```

In `_validateShouldUpdatePrices`, subsequent updates must satisfy:

```solidity
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(...);
}
```

If a keeper submits an update with `publishTime = T + 10` (where `T = block.timestamp`), then `priceLastUpdatedAt = T + 10`. Any subsequent call in the same block has `curTime = T`, so its maximum allowed `updateTimestamp = T + 10`. The constraint `updateTimestamp > T + 10` AND `updateTimestamp ≤ T + 10` is unsatisfiable — **no further update can succeed in the same block**.

Additionally, for heartbeat subscriptions, the next heartbeat trigger requires:

```solidity
updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds
```

With `lastUpdateTime = T + 10`, the next update needs `updateTimestamp ≥ T + 10 + heartbeatSeconds`, which requires `block.timestamp_next ≥ T + heartbeatSeconds`. This delays the heartbeat by up to 10 seconds compared to a normal update where `lastUpdateTime = T`.

### Impact Explanation

1. **Same-block update blocking**: Once a subscription's `priceLastUpdatedAt` is set to `block.timestamp + 10`, no second update can be processed for that subscription in the same Ethereum block. This is a persistent state — the subscription is locked for the remainder of the block.

2. **Heartbeat delay**: For heartbeat-based subscriptions, the effective heartbeat interval is extended by up to `FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD` (10 seconds), because the heartbeat timer starts from `priceLastUpdatedAt` (the publishTime), not from `block.timestamp`.

3. **Irreversibility within the block**: Once `priceLastUpdatedAt` is set to a future value, it cannot be lowered. Any legitimate keeper attempting to update the subscription in the same block will have their transaction revert with `TimestampOlderThanLastUpdate`.

### Likelihood Explanation

Any keeper (unprivileged updater) can call `updatePriceFeeds`. Pythnet publishes prices continuously at ~400ms intervals, and clock skew between Pythnet validators and Ethereum nodes can produce price updates with timestamps slightly in the future. A keeper can deliberately select the Pythnet price update with the highest available `publishTime` (closest to `block.timestamp + 10`) to maximize the stored future timestamp. This is a deliberate, low-cost action requiring no special privileges — only the ability to call `updatePriceFeeds` with valid Pythnet-signed price data.

### Recommendation

1. **Cap `priceLastUpdatedAt` at `block.timestamp`**: When storing the update timestamp, clamp it to `block.timestamp` to prevent future-dated values from being persisted:

```solidity
status.priceLastUpdatedAt = Math.min(latestPublishTime, block.timestamp);
```

2. **Alternatively, use `block.timestamp` directly** as the reference for heartbeat calculations, rather than the price feed's `publishTime`.

3. **Enforce the future timestamp check in `_validateShouldUpdatePrices`** in addition to the Pyth parsing call, to ensure the stored timestamp never exceeds `block.timestamp`.

### Proof of Concept

1. Keeper A calls `updatePriceFeeds(subscriptionId, updateData)` at Ethereum block time `T`, where `updateData` contains a valid Pythnet price update with `publishTime = T + 10` (within the 10-second future window).
2. `parsePriceFeedUpdatesWithConfig` accepts it (max timestamp = `T + 10`).
3. `_validateShouldUpdatePrices` returns `T + 10` as `latestPublishTime`.
4. `status.priceLastUpdatedAt = T + 10` is stored.
5. Keeper B (or Keeper A again) calls `updatePriceFeeds` in the same block (block.timestamp still = `T`).
6. The new `updateTimestamp` is at most `T + 10`.
7. `_validateShouldUpdatePrices` checks `updateTimestamp <= status.priceLastUpdatedAt` → `T + 10 <= T + 10` → **reverts with `TimestampOlderThanLastUpdate`**.
8. No further update is possible for this subscription until the next Ethereum block.
