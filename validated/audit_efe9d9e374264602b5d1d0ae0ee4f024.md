Based on my investigation of the CKB codebase, I found a direct analog to the missing access control / DOS vulnerability class.

---

### Title
Unauthenticated RPC Callers Can Wipe the Entire Transaction Pool or Disable All P2P Networking via `clear_tx_pool()` and `set_network_active()` — (`rpc/src/module/pool.rs`, `rpc/src/module/net.rs`)

### Summary

The CKB JSON-RPC server exposes `clear_tx_pool()` and `set_network_active(state: bool)` as callable methods with no per-method authorization check. Any client that can reach the RPC endpoint — including the explicitly in-scope "supported local CLI/RPC user" attacker profile — can call `clear_tx_pool()` to atomically evict every pending transaction from the mempool, or call `set_network_active(false)` to halt all P2P activity. Neither implementation performs any caller identity check before executing the destructive operation.

### Finding Description

**`clear_tx_pool()`** in `rpc/src/module/pool.rs`:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
``` [1](#0-0) 

There is no caller identity check, no role check, and no token/secret validation before `clear_pool(snapshot)` is invoked. The trait declaration is equally bare:

```rust
#[rpc(name = "clear_tx_pool")]
fn clear_tx_pool(&self) -> Result<()>;
``` [2](#0-1) 

**`set_network_active()`** in `rpc/src/module/net.rs`:

```rust
#[rpc(name = "set_network_active")]
fn set_network_active(&self, state: bool) -> Result<()>;
``` [3](#0-2) 

Passing `false` disables all P2P activity on the node. The RPC documentation confirms the semantics: "Disable/enable all p2p network activity." [4](#0-3) 

A companion method, `clear_tx_verify_queue()`, has the same pattern and wipes the pending verification queue:

```rust
fn clear_tx_verify_queue(&self) -> Result<()> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_verify_queue()
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
``` [5](#0-4) 

The RPC module list confirms all three methods are registered and exposed without any documented access restriction: [6](#0-5) 

**Structural parallel to the reported issue:** In RollerPeriphery, `approve()` was `public` with no `onlyTrusted` guard, so any caller could set the allowance to 0 and block deposits. Here, `clear_tx_pool()` and `set_network_active()` are registered RPC methods with no authorization guard, so any RPC caller can zero out the mempool or kill the network layer.

### Impact Explanation

- **`clear_tx_pool()`**: Atomically removes every pending and proposed transaction from the mempool. Users' submitted transactions are silently discarded; wallets and dApps relying on the node for transaction relay experience complete service loss until transactions are resubmitted (and the attacker can repeat the call immediately).
- **`set_network_active(false)`**: Halts all P2P connections — block sync, relay, and peer discovery stop. The node falls behind the chain tip and cannot propagate blocks or transactions until the operator manually re-enables networking.
- **`clear_tx_verify_queue()`**: Drops all transactions currently awaiting script verification, causing silent loss of in-flight submissions.

### Likelihood Explanation

The attacker profile is "supported local CLI/RPC user," which the prompt explicitly marks as in scope. The CKB RPC is also commonly exposed on non-loopback addresses by public infrastructure operators (exchanges, wallets, block explorers). No credentials, keys, or privileged access are required — a single HTTP POST with a valid JSON-RPC body is sufficient. The attack is trivially repeatable in a loop to maintain the DOS condition.

### Recommendation

Add a per-method authorization layer to all state-mutating administrative RPC methods. Concretely:

1. Introduce an optional `rpc_secret_token` in the node configuration (similar to Bitcoin Core's `rpcauth`).
2. Require the token to be present in the HTTP `Authorization` header for methods classified as administrative: `clear_tx_pool`, `clear_tx_verify_queue`, `set_network_active`, `remove_transaction`, `set_ban`, `clear_banned_addresses`.
3. Alternatively, split the RPC server into two listeners — a public read-only port and a privileged admin port bound exclusively to `127.0.0.1` — and register destructive methods only on the admin port.

### Proof of Concept

```bash
# Wipe the entire mempool — no credentials required
curl -s -X POST http://<node-rpc-host>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}'

# Disable all P2P networking
curl -s -X POST http://<node-rpc-host>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"set_network_active","params":[false]}'
```

Both calls return `{"result": null}` and take effect immediately. Repeating `clear_tx_pool` in a loop (e.g., every second) permanently prevents any transaction from accumulating in the pool, denying service to all users of that node.
