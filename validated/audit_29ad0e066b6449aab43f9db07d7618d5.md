Audit Report

## Title
`terminated_vms` Entry Prematurely Removed After First `ckb_wait` Causes Subsequent Waiters to Receive `WAIT_FAILURE` on a Legitimately Terminated VM — (`script/src/scheduler.rs`)

## Summary

In `Scheduler::process_message_box`, after the first VM successfully waits on an already-terminated target, line 575 unconditionally removes that target's entry from `terminated_vms`. Any subsequent VM calling `ckb_wait` on the same target finds it absent from both `terminated_vms` and `states`, and falls through to the `WAIT_FAILURE` branch at line 582, receiving an incorrect failure code despite the target having exited cleanly.

## Finding Description

When a non-root VM terminates, `iterate_process_results` stores its exit code in `terminated_vms` at line 364 and removes it from `states` at line 402. VMs already in `VmState::Wait` for that target are woken immediately at lines 375–397. VMs that call `ckb_wait` *after* the target has already terminated are handled in `process_message_box` at lines 564–593.

The first such waiter (VM B) hits the branch at line 565, receives `SUCCESS` and the correct exit code, is marked `Runnable`, and then line 575 executes:

```rust
self.terminated_vms.retain(|id, _| id != &args.target_id);
```

This removes the target's entry entirely from `terminated_vms`. When a second waiter (VM C) subsequently calls `ckb_wait` on the same target:
- Line 565's `get` on `terminated_vms` returns `None` (entry was removed at line 575)
- Line 578's `states.contains_key` returns `false` (entry was removed at line 402 on termination)
- The fallthrough at lines 579–583 fires, setting `A0` to `WAIT_FAILURE`

The `Wait::ecall` in `script/src/syscalls/wait.rs` at lines 40–50 pushes `Message::Wait` and returns `Err(VMError::Yield)`, so there is no way for the calling VM to avoid this path once it has issued the syscall. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation

This is a confirmed incorrect implementation of the `ckb_wait` system call behavior in CKB-VM. A V2 script where two or more child VMs call `ckb_wait` on the same already-exited target will observe incorrect return codes for all but the first waiter. This can cause a transaction that should be accepted (all child VMs exit cleanly) to be rejected (a child VM receives `WAIT_FAILURE` and returns non-zero), or cause incorrect branching within the script. This matches the allowed impact: **High (10001–15000 points) — Incorrect implementation or behavior of CKB-VM or system scripts**.

## Likelihood Explanation

No privileges, hash power, or key material are required. Any unprivileged user can craft a V2 transaction containing a script that spawns multiple child VMs and has more than one of them call `ckb_wait` on the same target after it has exited. This is a natural coordination pattern (e.g., a coordinator and a logger both waiting on a worker). The trigger is fully deterministic and reproducible on any node running this code.

## Recommendation

Remove line 575:
```rust
self.terminated_vms.retain(|id, _| id != &args.target_id);
```
Do not remove `terminated_vms` entries during `process_message_box`. Entries should be cleaned up only when they are provably unreachable — for example, when no live VM can still issue a `ckb_wait` on that target, or when the scheduler itself terminates. A conservative correct fix is to simply never remove from `terminated_vms` during `process_message_box` and defer cleanup to scheduler teardown.

## Proof of Concept

1. Root VM spawns three child VMs: A, B, C.
2. A immediately exits with code `42`.
3. B calls `ckb_wait(A_id, &exit_b)` — `process_message_box` finds A in `terminated_vms`, writes `42`, sets `SUCCESS`, then removes A's entry via `retain` at line 575.
4. C calls `ckb_wait(A_id, &exit_c)` — `process_message_box` finds A absent from `terminated_vms` (removed in step 3) and absent from `states` (removed at line 402 on termination); line 582 sets `WAIT_FAILURE`.
5. Assert both B and C received `SUCCESS` with `exit_code == 42`. The assertion for C fails, confirming the bug.
