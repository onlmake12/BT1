### Title
Collected Update Fees Are Permanently Locked in `PriceInfoObject` with No Withdrawal Mechanism — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every call to `update_single_price_feed` deposits SUI fee coins into a shared `PriceInfoObject` via `deposit_fee_coins`. No function in the entire contract suite ever removes those coins. The `fee_recipient_address` field in `State` is explicitly annotated as unused. All collected fees are permanently frozen.

### Finding Description

`update_single_price_feed` in `pyth.move` enforces a fee check and then immediately deposits the coins into the `PriceInfoObject`: [1](#0-0) 

`deposit_fee_coins` in `price_info.move` stores the coins as a dynamic object field under `FEE_STORAGE_KEY` using `dynamic_object_field::add` / `coin::join`: [2](#0-1) 

A `get_balance` accessor exists to read the accumulated balance, confirming the design intent to track it: [3](#0-2) 

However:
- `dynamic_object_field::remove` is called **nowhere** in the entire Move codebase.
- No `withdraw_fee_coins` function exists anywhere.
- `fee_recipient_address` is stored in `State` but is never read by any fee-transfer logic: [4](#0-3) 

The `set_fee_recipient` governance module explicitly documents this abandonment: [5](#0-4) 

The setter still writes to the field, but nothing ever reads it to route funds: [6](#0-5) 

### Impact Explanation

Every SUI coin paid as an update fee via `update_single_price_feed` is deposited into a shared `PriceInfoObject` and can never be recovered. There is no privileged withdrawal path, no governance action that drains the field, and no upgrade path that currently addresses this. The funds are not burned — they sit in an accessible object — but no code path can move them out. This matches the scope criterion of **permanent freezing of protocol funds**.

### Likelihood Explanation

This triggers on every successful `update_single_price_feed` call, which is the primary production entry point for price updates. Any deployed instance with a non-zero `base_update_fee` is actively accumulating permanently locked funds with each update.

### Recommendation

1. Add a `withdraw_fee_coins` function (gated by `LatestOnly` or a governance capability) that calls `dynamic_object_field::remove<vector<u8>, Coin<SUI>>(&mut price_info_object.id, FEE_STORAGE_KEY)` and transfers the result to `fee_recipient_address`.
2. Alternatively, transfer fees directly to `fee_recipient_address` inside `update_single_price_feed` using `transfer::public_transfer(fee, state::get_fee_recipient(pyth_state))` instead of depositing into the object.
3. Remove or repurpose the now-dead `fee_recipient_address` field and `set_fee_recipient` governance action to avoid further confusion.

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
