### Title
`payable(msg.sender).transfer()` in `verifyUpdate` DOSes Contract-Based Lazer Consumers — (`File: lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` uses the deprecated `.transfer()` primitive to refund excess ETH to `msg.sender`. Because `.transfer()` forwards only 2300 gas, any contract caller whose `receive()` function requires more than 2300 gas (or has no `receive()` at all) will have the entire `verifyUpdate()` call revert, permanently blocking that contract from consuming Lazer price updates.

### Finding Description
`verifyUpdate()` is the sole entry point for Lazer price-update consumers. When a caller sends more than `verification_fee`, the surplus is returned via:

```solidity
payable(msg.sender).transfer(msg.value - verification_fee);
``` [1](#0-0) 

`.transfer()` hard-caps the gas forwarded to the recipient at 2300. Any contract `msg.sender` that:
- has no `receive()` / `fallback()` function, or
- has a `receive()` that performs any non-trivial work (e.g., emits an event, writes a storage slot, calls another contract)

will cause the `.transfer()` to revert. Because the revert propagates up, the entire `verifyUpdate()` call fails — the price update is never delivered and the fee is not collected. [2](#0-1) 

### Impact Explanation
Any on-chain contract that integrates `PythLazer.verifyUpdate()` and sends `msg.value > verification_fee` is permanently DOSed if its `receive()` logic exceeds 2300 gas. The contract cannot receive Lazer price updates at all, breaking any protocol that depends on them. This is a complete loss of service for affected consumers.

### Likelihood Explanation
The `verifyUpdate()` function is the primary integration point for Lazer consumers. Contract-based integrators (e.g., DeFi protocols wrapping Lazer) are the expected majority of callers. Sending a small buffer above the exact fee is a common defensive pattern. The 2300-gas stipend is routinely exceeded by even minimal `receive()` implementations (e.g., a single `emit` costs ~375 gas plus topic/data costs). No privileged access is required; any unprivileged Lazer updater or consumer contract triggers this path.

### Recommendation
Replace `.transfer()` with a low-level `.call{value: ...}("")` and check the return value, consistent with how `Entropy.sol` and `Scheduler.sol` already handle ETH transfers:

```solidity
// Instead of:
payable(msg.sender).transfer(msg.value - verification_fee);

// Use:
(bool sent, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
require(sent, "Refund failed");
``` [3](#0-2) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.13;

import "lazer/contracts/evm/src/PythLazer.sol";

contract LazerConsumer {
    PythLazer public lazer;

    constructor(address _lazer) {
        lazer = PythLazer(_lazer);
    }

    // This contract has no receive() function.
    // Any attempt to call verifyUpdate with msg.value > verification_fee will revert.
    function consumeUpdate(bytes calldata update) external payable {
        // Sends 1 wei more than the fee — triggers the .transfer() refund path
        (bytes calldata payload, address signer) =
            lazer.verifyUpdate{value: msg.value}(update);
        // Never reached if msg.value > verification_fee
    }
}
```

Deploying `LazerConsumer` and calling `consumeUpdate` with `msg.value = verification_fee + 1` will always revert at the `.transfer()` line in `PythLazer.verifyUpdate()`, because `LazerConsumer` has no `receive()` function. [4](#0-3)
