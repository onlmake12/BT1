### Title
Stale Ancestor `descendants_*` Accounting in `remove_entry_and_descendants` Allows Unbounded `EvictKey` Inflation and Permanent Pool Griefing - (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`remove_entry_and_descendants` pre-removes all link entries before calling `remove_entry` on each removed transaction. Because `update_ancestors_index_key` resolves ancestors through the links structure, and those links are already gone, the surviving ancestor transactions never have their `descendants_count`, `descendants_size`, `descendants_cycles`, or `descendants_fee` fields decremented. An unprivileged RPC caller can exploit this by repeatedly submitting a child transaction and then a conflicting replacement, inflating an ancestor's `descendants_fee` toward `u64::MAX`. This makes the ancestor's `EvictKey` show a near-maximum fee rate, permanently shielding a low-fee transaction from pool eviction.

---

### Finding Description

**Root cause — `pool_map.rs` lines 252–265:**

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ← removes ALL links first
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
``` [1](#0-0) 

`remove_entry_links` removes the target node's own link entry (`self.links.remove(id)`) and unlinks it from its parents and children. [2](#0-1) 

When `remove_entry` is subsequently called for each removed ID, it calls `update_ancestors_index_key(&entry.inner, EntryOp::Remove)`:

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
``` [3](#0-2) 

Because the link for the removed entry was already erased, `calc_ancestors` returns an empty set. The surviving ancestor A is never reached, so `sub_descendant_weight` is never called on it. A's `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` remain permanently inflated. [4](#0-3) 

**`EvictKey` uses the stale fields directly:**

```rust
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_weight =
            get_transaction_weight(entry.descendants_size, entry.descendants_cycles);
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),
            descendants_count: entry.descendants_count,
        }
    }
}
``` [5](#0-4) 

The pool evicts the entry with the **lowest** `EvictKey`. An inflated `descendants_feerate` makes A appear more valuable than it is, shielding it from eviction. [6](#0-5) 

**The existing test does not catch this:**

```rust

// only checks tx2/tx3 are absent and tx1's link-set is empty
// never checks tx1.descendants_count, tx1.descendants_fee, tx1.evict_key
``` [7](#0-6)

### Impact Explanation

**Unbounded `EvictKey` inflation via repeated child replacement:**

1. Attacker submits low-fee transaction A (spends UTXO₀, produces output₀).
2. Attacker submits child B (spends output₀). Pool calls `add_entry` → A's `descendants_fee += B.fee`, `descendants_count += 1`, etc.
3. Attacker submits B′ (also spends output₀, conflicting with B). Pool calls `resolve_conflict` → `remove_entry_and_descendants(B)` → B removed, **A's `descendants_*` NOT decremented**. B′ is then added → A's `descendants_fee += B′.fee` again.
4. Repeat step 3 N times. After N iterations: `A.descendants_fee = A.fee + N × child_fee` (saturating at `u64::MAX`).

After enough iterations, A's `EvictKey.fee_rate` reaches the maximum representable value. `limit_size` will never select A for eviction regardless of how full the pool becomes. [8](#0-7) 

**Concrete consequences:**
- A low-fee transaction occupies a pool slot indefinitely, wasting pool capacity.
- If the pool is full, legitimate high-fee transactions may be rejected while the attacker's low-fee transaction survives.
- The attack requires no privileged access — only the ability to call `send_transaction` via RPC.
- The attacker pays no on-chain fees because neither B nor B′ are ever confirmed; they are only submitted to the mempool.

---

### Likelihood Explanation

Any unprivileged RPC caller can reach `resolve_conflict` → `remove_entry_and_descendants` by submitting two transactions that spend the same output. This is a standard double-spend submission pattern, requiring no special knowledge or resources beyond a funded UTXO. The inflation accumulates monotonically and is not reset unless the pool is cleared or A is confirmed. The attack is cheap to execute repeatedly.

---

### Recommendation

Before pre-removing links, collect the set of surviving ancestors (those NOT in `removed_ids`) and update their `descendants_*` fields explicitly. One correct approach:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();

    // For each entry being removed, update its surviving ancestors BEFORE links are torn down
    for rid in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(rid) {
            let entry_inner = entry.inner.clone();
            let ancestors = self.links.calc_ancestors(rid);
            for anc_id in ancestors {
                if !removed_set.contains(&anc_id) {
                    self.entries.modify_by_id(&anc_id, |e| {
                        e.inner.sub_descendant_weight(&entry_inner);
                        e.evict_key = e.inner.as_evict_key();
                    });
                }
            }
        }
    }

    for rid in &removed_ids {
        self.remove_entry_links(rid);
    }
    removed_ids.iter().filter_map(|rid| self.remove_entry(rid)).collect()
}
```

Alternatively, add a test asserting that after `remove_entry_and_descendants(tx2)`, `tx1.descendants_count == 1` and `tx1.descendants_fee == tx1.fee` to make the invariant explicit and catch regressions.

### Proof of Concept

**Setup:** Pool with chain A → B → C (A is parent, B is child of A, C is child of B).

**Step 1:** Add A, B, C. After insertion:
- `A.descendants_count = 3`, `A.descendants_fee = fee_A + fee_B + fee_C`

**Step 2:** Call `remove_entry_and_descendants(B)`. Expected: A.descendants_count = 1, A.descendants_fee = fee_A. Actual (due to bug): A.descendants_count = 3, A.descendants_fee = fee_A + fee_B + fee_C.

**Step 3:** Add B′ (same parent A, different tx). A.descendants_count = 4, A.descendants_fee = fee_A + fee_B + fee_C + fee_B′.

**Step 4:** Call `remove_entry_and_descendants(B′)`. A.descendants_count stays at 4. Repeat.

After N rounds, `A.descendants_fee` saturates at `u64::MAX`. `A.evict_key.fee_rate = u64::MAX / weight`, making A immune to `limit_size` eviction. [1](#0-0) [3](#0-2) [9](#0-8) [5](#0-4)
