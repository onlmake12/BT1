### Title
Staking Delegators Permanently Lose OIS Rewards Due to Fixed-Size 52-Epoch Circular Event Buffer - (File: `governance/pyth_staking_sdk/src/types/integrity-pool.ts`)

---

### Summary

The Oracle Integrity Staking (OIS) `PoolData` account stores per-epoch reward events in a fixed-size circular buffer of 52 entries. A delegator's rewards are computed by replaying events from `DelegationRecord.lastEpoch` forward. If a delegator does not call `advance_delegation_record` for more than 52 epochs (~1 year), the circular buffer wraps and overwrites old event entries, making those epochs' rewards permanently unrecoverable.

---

### Finding Description

The `PoolData` account contains an `events` field defined as a fixed-size array of exactly **52** `Event` structs:

```
name: "events"
type: { array: [{ defined: { name: "event" } }, 52] }
```

Each `Event` stores the epoch number, the yield rate `y`, and per-publisher reward ratios (`selfRewardRatio`, `otherRewardRatio`) for that epoch. The `advance` instruction appends a new event each epoch, writing into the buffer at index `num_events % 52`, overwriting the oldest entry once the buffer is full.

A delegator's per-publisher `DelegationRecord` stores only `lastEpoch` — the last epoch for which rewards were computed. When `advance_delegation_record` is called, the program iterates from `lastEpoch` forward through the events buffer to compute owed rewards.

If a delegator's `lastEpoch` is more than 52 epochs behind the current epoch, the events for those old epochs have been overwritten. The program can no longer reconstruct the reward amounts for those epochs. Those rewards are permanently lost to the delegator.

The SDK itself acknowledges this expiry window explicitly:

```typescript
expiry:
  instructions.lowestEpoch === undefined
    ? undefined
    : epochToDate(instructions.lowestEpoch + 53n),
```

This computes a deadline of `lowestEpoch + 53` epochs — after which rewards from `lowestEpoch` are gone. The `+53` (not `+52`) accounts for the current in-progress epoch not yet finalized.

---

### Impact Explanation

Any OIS delegator who does not call `advance_delegation_record` (or have it called on their behalf) within a 52-epoch window (~1 year, since epochs are weekly) permanently forfeits all earned rewards from epochs that have been overwritten in the circular buffer. There is no recovery path — the on-chain event data is gone. This is a direct, irreversible loss of earned PYTH token rewards for staking users.

**Impact: High** — permanent loss of earned staking rewards with no recourse.

---

### Likelihood Explanation

Epochs are weekly. A delegator must miss ~52 consecutive weeks without any interaction to trigger the loss. This is plausible for:
- Long-term "set and forget" delegators who stake and never return to the UI.
- Delegators whose wallet access is temporarily disrupted (lost key, migration, etc.) for an extended period.
- Delegators who are unaware of the expiry mechanic.

The SDK surfaces the expiry date in the UI, but there is no on-chain enforcement preventing the loss — it is silent and permanent.

**Likelihood: Low** — requires ~1 year of complete inactivity, but the consequence when triggered is total reward forfeiture.

---

### Recommendation

1. **Permissionless keeper**: Allow any third party to call `advance_delegation_record` for any stake account, so keepers can prevent expiry on behalf of inactive delegators. (The current design already allows the `payer` to be anyone — this should be actively used by a keeper service.)
2. **Larger buffer or unbounded accumulator**: Increase the `events` array beyond 52, or replace the circular buffer with a global cumulative reward-per-token accumulator (standard staking pattern) that does not expire.
3. **Explicit on-chain warning**: Emit an error or warning when `advance_delegation_record` is called and some epochs have already been lost, so the delegator is informed rather than silently receiving less than owed.

---

### Proof of Concept

1. Delegator stakes to publisher pool at epoch `E`.
2. `DelegationRecord.lastEpoch` is initialized to `E`.
3. No one calls `advance_delegation_record` for 53+ epochs.
4. The `events` circular buffer (size 52) has wrapped; entries for epochs `E` through `E+1` are overwritten.
5. Delegator calls `advance_delegation_record` at epoch `E+53`.
6. The program iterates events from `lastEpoch = E` but finds no event with `epoch == E` in the buffer (it was overwritten at index `E % 52`).
7. Rewards for epochs `E` through `E+1` are silently skipped — permanently lost.
8. The SDK's `getClaimableRewards` would have shown `expiry: epochToDate(E + 53n)` as a warning, but no on-chain enforcement prevented the loss.
