### Title
Arbitrary signer can front-run `initialize` to seize admin of the global message-buffer whitelist — (File: `pythnet/message_buffer/programs/message_buffer/src/lib.rs`)

---

### Summary
The `message_buffer` program's `initialize` instruction accepts any arbitrary signer as `admin` with no constraint verifying their authority. The whitelist PDA is derived from fixed, program-wide seeds, so there is exactly one global whitelist. An attacker who front-runs initialization permanently becomes the admin and can block all Pythnet price-data accumulation.

---

### Finding Description
The `Initialize` account struct imposes no constraint on who `admin` is:

```rust
// pythnet/message_buffer/programs/message_buffer/src/lib.rs  lines 142-158
#[derive(Accounts)]
pub struct Initialize<'info> {
    /// Admin that can update the whitelist and create/resize/delete buffers
    pub admin: Signer<'info>,          // ← any signer accepted, no constraint

    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(
        init,
        payer = payer,
        seeds = [MESSAGE.as_bytes(), WHITELIST.as_bytes()],  // ← fixed, global seeds
        bump,
        space = 8 + Whitelist::INIT_SPACE,
    )]
    pub whitelist: Account<'info, Whitelist>,
    pub system_program: Program<'info, System>,
}
```

The handler only rejects `Pubkey::default()`:

```rust
// lines 23-29
pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
    require_keys_neq!(ctx.accounts.admin.key(), Pubkey::default());
    let whitelist = &mut ctx.accounts.whitelist;
    whitelist.bump = *ctx.bumps.get("whitelist").unwrap();
    whitelist.admin = ctx.accounts.admin.key();
    Ok(())
}
```

Three properties combine to make this exploitable:

1. **No authority check** — `admin` is a plain `Signer<'info>` with zero constraints; any wallet qualifies.
2. **Single global PDA** — seeds are the constants `MESSAGE` + `WHITELIST`, so there is exactly one whitelist for the entire program. Whoever initializes it first owns it forever.
3. **One-shot init** — Anchor's `init` constraint means the account can never be re-initialized once it exists.

Every subsequent privileged operation (`set_allowed_programs`, `update_whitelist_admin`, `create_buffer`, `resize_buffer`, `delete_buffer`) enforces `has_one = admin` on the whitelist, so the legitimate Pyth team is permanently locked out if the attacker wins the race.

---

### Impact Explanation
An attacker who front-runs `initialize` with their own key as `admin`:

- Calls `set_allowed_programs([])` — empties the whitelist, so no program can pass the `is_allowed_program_auth` check required by `put_all` / `create_buffer`.
- Alternatively sets `allowed_programs` to attacker-controlled program IDs, injecting arbitrary messages into the accumulator.
- Refuses to call `update_whitelist_admin`, leaving the legitimate team with no recovery path.
- The entire Pythnet price-data pipeline (publisher → `message_buffer` → Wormhole VAA → Hermes → all target chains) is severed at the source.

**Impact: High** — complete denial of Pyth price-feed publication on Pythnet; all downstream target-chain receivers stop receiving valid updates.

---

### Likelihood Explanation
The `initialize` instruction is callable by any transaction sender on Pythnet. The deployment window (between program deployment and the first legitimate `initialize` call) is the attack surface. A bot watching the Pythnet transaction stream for the program ID can submit a competing `initialize` with a negligible transaction fee. No privileged access, leaked key, or social engineering is required.

**Likelihood: Medium** — requires timing the front-run during the deployment window, but the cost and skill barrier are low.

---

### Recommendation
Gate initialization behind a hardcoded or upgrade-authority-derived expected admin. The simplest fix is a program constant:

```rust
const EXPECTED_ADMIN: Pubkey = pubkey!("...known Pyth multisig pubkey...");

pub struct Initialize<'info> {
    #[account(
        constraint = admin.key() == EXPECTED_ADMIN
            @ MessageBufferError::UnauthorizedAdmin
    )]
    pub admin: Signer<'info>,
    // ...
}
```

Alternatively, require the admin to equal the program's upgrade authority, which is already a trusted key:

```rust
#[account(
    constraint = admin.key() == program_data.upgrade_authority_address.unwrap()
        @ MessageBufferError::UnauthorizedAdmin
)]
pub admin: Signer<'info>,
```

### Proof of Concept

1. Deploy (or observe the deployment of) the `message_buffer` program at `7Vbmv1jt4vyuqBZcpYPpnVhrqVe5e6ZPb6JxDcffRHUM`.
2. Before the Pyth team submits their `initialize` transaction, submit:
   ```
   initialize(
     admin    = <attacker_keypair>,   // any non-default pubkey
     payer    = <attacker_keypair>,
     whitelist = PDA([MESSAGE, WHITELIST], program_id),
   )
   ```
3. The whitelist PDA is created with `whitelist.admin = attacker`.
4. Call `set_allowed_programs(allowed_programs: [])` — passes `has_one = admin` because attacker is admin; empties the whitelist.
5. Any subsequent call to `put_all` or `create_buffer` by the legitimate Pyth oracle program fails with `CallerNotAllowed` because `is_allowed_program_auth` finds no matching entry.
6. The Pyth team's own `initialize` call fails with Anchor's account-already-initialized error — no recovery is possible without a program upgrade.
