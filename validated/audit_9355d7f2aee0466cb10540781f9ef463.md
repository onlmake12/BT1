Audit Report

## Title
`RecentReject::put` Never Persists Incremented Key Count — `count_limit` Eviction Permanently Disabled — (`tx-pool/src/component/recent_reject.rs`)

## Summary
In `RecentReject::put`, the incremented key count is computed into a local `let` binding but never written back to `self.total_keys_num`. Because the field stays frozen at its initial estimate (0 for a fresh DB), the `count_limit` guard is never satisfied, `shrink()` is never triggered, and the RocksDB backing store grows unchecked for the node's lifetime, bounded only by TTL expiry. An unprivileged attacker can exploit this by flooding the node with rejected transactions to exhaust disk space and crash the node.

## Finding Description
In `put()` (lines 62–69 of `tx-pool/src/component/recent_reject.rs`):

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // ← self.total_keys_num is NEVER written back
} else {
    self.shrink()?;
}
```

`total_keys_num` is a local binding. `self.total_keys_num` is only updated at line 111 inside `shrink()`:

```rust
let total_keys_num = self.estimate_total_keys_num()?;
self.total_keys_num = total_keys_num;
```

`shrink()` is only reached when `total_keys_num > self.count_limit`. Since `self.total_keys_num` is never incremented in the non-shrink branch, every call to `put()` evaluates `0 + 1 = 1` (for a fresh DB). With any `count_limit > 1` (the normal case), `1 > count_limit` is permanently false. `shrink()` is never called, and `self.total_keys_num` never advances past its initial estimate from `build()` (line 44). The DB grows without bound until RocksDB TTL expiry removes entries. `get_estimate_total_keys_num()` (line 80–82) returns the frozen stale value for the node's entire lifetime.

## Impact Explanation
**High — Vulnerability which could easily crash a CKB node.**

Rejected transactions are written to RocksDB at line 60 (`self.db.put(...)`) on every call to `put()`. Without `shrink()` firing, the only size bound is TTL expiry. With a long TTL (hours or days, operator-configured), an attacker can accumulate enough rejection records to exhaust available disk space. A full disk causes RocksDB writes to fail, which propagates as errors through the tx-pool, crashing or severely degrading the node. The `count_limit` mechanism — the sole intended defense against unbounded growth — is completely non-functional.

## Likelihood Explanation
No privilege is required. Any network peer can submit transactions that will be rejected (wrong fee rate, invalid script, duplicate nonce, etc.). Each rejection unconditionally calls `put()`. Rejected transactions require no on-chain fee, so the attacker's cost is only network bandwidth. The bug is triggered on the very first call to `put()` and persists for the node's lifetime. No race condition, special timing, or configuration is needed beyond `count_limit > 1`, which is the default.

## Recommendation
In the non-shrink branch of `put()`, write the incremented value back to `self.total_keys_num` before the guard check:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;   // ← add this line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```
