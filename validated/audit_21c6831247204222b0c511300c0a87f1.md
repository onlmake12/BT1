Audit Report

## Title
Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` via Unbounded `tx_age` Index into `confirm_blocks_to_failed_txs` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary
`remove_unconfirmed_tx` computes `tx_age` as `tip_height - entry_height` and uses `tx_age - 1` to index into `confirm_blocks_to_failed_txs`, which has a fixed length of `MAX_CONFIRM_BLOCKS = 1000`. No bounds check guards this access. When a tracked transaction is rejected after 1001 or more blocks, `tx_age - 1 >= 1000` produces a Rust index-out-of-bounds panic, crashing the node's tx-pool thread.

## Finding Description
`confirm_blocks_to_failed_txs` is allocated with exactly `max_confirm_blocks` (1000) rows, giving valid indices 0–999: [1](#0-0) 

In `remove_unconfirmed_tx`, `tx_age` is computed without an upper bound: [2](#0-1) 

The only guard present (line 208) protects the `block_unconfirmed_txs` branch exclusively: [3](#0-2) 

The `count_failure` branch that follows has no equivalent bounds check: [4](#0-3) 

`move_track_window` only zeroes out `block_unconfirmed_txs` slots and increments `old_unconfirmed_txs`; it never removes entries from `tracked_txs`, so a tx record persists indefinitely: [5](#0-4) 

`drop_tx` always passes `count_failure = true`, meaning any pool eviction of a tracked tx reaches the unguarded index: [6](#0-5) 

`track_tx` enforces `height == best_height` at entry, so `entry_height` is always the `best_height` at submission time. After 1001 blocks pass, `tx_age = best_height - entry_height >= 1001`, making `tx_age - 1 >= 1000` an out-of-bounds index.

## Impact Explanation
A Rust index-out-of-bounds access panics unconditionally. The fee estimator runs inside the tx-pool service thread. A panic there crashes or permanently disables the tx-pool, halting transaction and block processing on the affected node. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (10001–15000 points).

## Likelihood Explanation
No privilege is required. Any user can submit a valid low-fee transaction via the public RPC (`send_transaction`) or P2P relay. At ~10-second block times, 1001 blocks is approximately 2.8 hours — a realistic lifetime for a low-fee transaction that is never mined. Once the age threshold is crossed, any ordinary pool eviction event (capacity pressure, conflict, expiry) triggers the panic. The condition is deterministic and repeatable.

## Recommendation
Add a bounds check before accessing `confirm_blocks_to_failed_txs`, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure {
    if let Some(row) = self.confirm_blocks_to_failed_txs.get_mut(tx_age - 1) {
        row[bucket_index] += 1f64;
    }
}
```

Ages beyond `MAX_CONFIRM_BLOCKS` carry no useful statistical signal, so silently skipping them is correct behavior.

## Proof of Concept
1. Start a CKB node with the fee estimator enabled and IBD complete (`update_ibd_state(false)`).
2. Submit a valid transaction at block height H via `send_transaction`; this calls `accept_tx` → `track_tx`, recording `entry_height = H`.
3. Mine 1001 empty blocks (not including the transaction), advancing `best_height` to `H + 1001`. Each `commit_block` call invokes `move_track_window` and `decay` but leaves the tx in `tracked_txs`.
4. Trigger pool eviction of the transaction (e.g., via capacity pressure or a conflicting transaction). This calls `reject_tx` → `drop_tx` → `drop_tx_inner(tx_hash, true)` → `remove_unconfirmed_tx` with `tx_age = 1001`, `count_failure = true`.
5. The access `self.confirm_blocks_to_failed_txs[1000][bucket_index]` panics with index out of bounds on a 1000-element `Vec`, crashing the tx-pool thread.
