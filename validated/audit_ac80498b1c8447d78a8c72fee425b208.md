Audit Report

## Title
Missing Per-Peer Rate Limiting on Filter Protocol Allows I/O-Based Resource Exhaustion — (`sync/src/filter/mod.rs`)

## Summary
The `BlockFilter` protocol handler processes all three inbound request types (`GetBlockFilters`, `GetBlockFilterHashes`, `GetBlockFilterCheckPoints`) with no per-peer rate limiting. Each request triggers up to 4000 RocksDB reads. An unprivileged peer can flood the node with these requests at wire speed, saturating I/O and the async task queue, degrading or crashing the node.

## Finding Description
`BlockFilter` holds only a single `shared: Arc<SyncShared>` field — no `rate_limiter` is present. [1](#0-0) 

In `CKBProtocolHandler::received`, the handler parses the message and immediately calls `self.process(nc, peer_index, msg).await` with no quota check. [2](#0-1) 

`try_process` then dispatches to one of three process structs with no interposing guard. [3](#0-2) 

`GetBlockFilterCheckPointsProcess::execute` runs a `for _ in 0..BATCH_SIZE` loop (BATCH_SIZE = 2000), calling `active_chain.get_block_hash(block_number)` followed by `active_chain.get_block_filter_hash(&block_hash)` on every iteration — up to 4000 RocksDB reads per message. [4](#0-3) 

`GetBlockFilterHashesProcess::execute` has the identical pattern with the same BATCH_SIZE. [5](#0-4) 

By contrast, `Relayer` carries `rate_limiter: RateLimiter<(PeerIndex, u32)>` and `Relayer::try_process` checks `self.rate_limiter.check_key(&(peer, message.item_id()))` before any dispatch, returning `StatusCode::TooManyRequests` on excess. [6](#0-5) [7](#0-6) 

`BlockFilter` has no equivalent guard. There is no connection-level throttle, no message-count cap, and no ban triggered by high request volume from a single peer.

## Impact Explanation
A single attacker peer can sustain a stream of `GetBlockFilterCheckPoints { start_number: 0 }` messages at TCP line rate. Each message causes up to 4000 synchronous RocksDB reads plus response serialization and send. With no rate gate, this saturates the node's storage I/O and async executor, starving sync, relay, and RPC processing for all other peers. Under sufficient load this constitutes a practical single-peer node crash/hang. This matches the allowed **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"* and *"Vulnerabilities which could easily crash a CKB node"*.

## Likelihood Explanation
The attack requires only a valid P2P connection and knowledge of the publicly documented Filter protocol message format. `GetBlockFilterCheckPoints` is a 9-byte struct (`start_number: Uint64`). No keys, hashpower, or privileged access are needed. The attack is trivially repeatable and can be sustained indefinitely from a single peer.

## Recommendation
Add a `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`). In `BlockFilter::try_process`, check `self.rate_limiter.check_key(&(peer, message.item_id()))` before dispatching any of the three request handlers, returning `StatusCode::TooManyRequests` on excess. A quota of 30 requests/second per (peer, message-type) — matching the `Relayer` configuration — is a reasonable starting point. Also call `self.rate_limiter.retain_recent()` in `disconnected` to bound memory growth.

## Proof of Concept
1. Connect a peer to a full node with the Filter protocol enabled (filter feature active).
2. In a tight loop, send `GetBlockFilterCheckPoints { start_number: 0 }` as fast as the TCP connection allows.
3. Observe RocksDB read IOPS spike (via `rocksdb.block.cache.miss` metrics or OS iostat) and async task queue depth grow on the target node.
4. Confirm no ban or throttle is applied to the sender — the node continues processing every message.
5. Optionally repeat with `GetBlockFilterHashes { start_number: 0 }` to confirm the same path is unguarded.

        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,

                GetBlockFiltersProcess::new(msg, self, nc, peer)
                    .execute()
                    .await
            }

                    .await

                    .execute()
                    .await
            }

            | packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)

                    crate::LOG_TARGET_FILTER,

                );

        }
