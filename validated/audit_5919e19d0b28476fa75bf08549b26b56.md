Audit Report

## Title
Integer Division by 2 in `check_purge` Phase-2 Yields Zero Eviction Candidates When All Peers Share One Network Group, Permanently Blocking `add_addr` — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
When all 16384 peer store entries belong to a single `/16` network group, `peers_by_network_group.len()` equals 1, and `take(len / 2)` truncates to `take(0)` via integer division, producing no eviction candidates. Phase-1 also yields zero candidates because freshly added entries have `attempts_count = 0`, which does not satisfy the `>= ADDR_MAX_RETRIES` threshold for non-connectability. Every subsequent `add_addr` call returns `Err(EvictionFailed)`, permanently preventing new peer addresses from being stored until the node is restarted with a cleared peer store.

## Finding Description
**Phase-1 failure:** `add_addr` inserts entries with `last_connected_at_ms = 0` and `attempts_count = 0`. `is_connectable` returns `true` for these because the only non-connectable condition for never-connected addresses requires `attempts_count >= ADDR_MAX_RETRIES (3)` (line 95 of `types.rs`). All 16384 attacker-inserted entries pass as connectable, so phase-1 removes nothing.

**Phase-2 failure:** After grouping by network group, `len = peers_by_network_group.len() == 1`. The iterator `take(len / 2)` becomes `take(0)`, producing an empty iterator. The `flat_map` closure is never invoked. `candidate_peers` remains empty.

**Error return:** Lines 399–401 of `peer_store_impl.rs` return `Err(PeerStoreError::EvictionFailed)` when `candidate_peers` is empty, which propagates through `check_purge()?` in `add_addr`.

**No per-group cap:** `add_addr` performs no check on how many entries already exist for the incoming address's network group, allowing a single `/16` to monopolize all 16384 slots.

**Exploit path:**
1. Attacker establishes a standard P2P connection to the victim node.
2. Attacker sends discovery messages advertising 16384 distinct addresses from the same `/16` (e.g., `225.0.0.1`–`225.0.63.255`). A `/16` provides 65536 distinct IPs — more than sufficient.
3. Victim's peer store fills to `ADDR_COUNT_LIMIT` with a single `Group::IP4([225, 0])` key.
4. Every subsequent `add_addr` call: phase-1 finds zero non-connectable entries; phase-2 `take(0)` produces no candidates; `EvictionFailed` is returned.
5. No legitimate peer addresses can ever be added again until the node is restarted with a cleared peer store.

## Impact Explanation
The peer store is permanently locked with attacker-controlled, unreachable addresses. `add_addr` is called from discovery, identify, and DNS seeding code paths. Once locked, the node cannot learn about new peers. As existing connections drop naturally, the node exhausts its store of reachable peers and becomes fully isolated from the honest network — unable to sync blocks, propagate transactions, or maintain consensus participation. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**, as the node's networking layer is rendered permanently non-functional without a manual restart and store wipe.

## Likelihood Explanation
Any peer that can establish a standard P2P connection to the victim can trigger this. No special privileges, proof-of-work, cryptographic material, or victim mistakes are required. The attack is cheap (one connection, one batch of discovery messages), deterministic, and permanent. It can be repeated after each node restart. A single malicious node can target many victims simultaneously.

## Recommendation
Replace `take(len / 2)` with `take((len / 2).max(1))` to guarantee at least one group is always considered for eviction when the store is full:

```rust
peers
    .into_iter()
    .take((len / 2).max(1))
```

Additionally, enforce a per-group cap during `add_addr` (e.g., reject insertion if the incoming address's group already holds more than `ADDR_COUNT_LIMIT / expected_groups` entries) to prevent any single `/16` from monopolizing the store.

## Proof of Concept
```rust
// Fill store with 16384 addresses from 225.0.x.x (same /16 → Group::IP4([225, 0]))
for i in 0u32..16384 {
    let ip = Ipv4Addr::new(225, 0, (i / 256) as u8, (i % 256) as u8);
    let addr = format!("/ip4/{}/tcp/8115", ip).parse::<Multiaddr>().unwrap();
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
// Store is now full; all entries share one network group key.
// Any new add_addr must fail with EvictionFailed:
let new_addr = "/ip4/1.2.3.4/tcp/8115".parse::<Multiaddr>().unwrap();
assert!(matches!(
    peer_store.add_addr(new_addr, Flags::COMPATIBILITY),
    Err(e) if e.to_string().contains("EvictionFailed")
));
// Invariant: peers_by_network_group.len() == 1, take(1/2) == take(0), candidate_peers is empty.
```
