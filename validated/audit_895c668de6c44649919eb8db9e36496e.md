### Title
Silent Callback Failure via Dangling Storage Reference After `clearRequest` for Overflow Requests — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`'s `executeCallback`, a `storage` reference `req` is obtained from `findActiveRequest`, then `clearRequest(sequenceNumber)` is called **before** the external callback is made. For requests stored in the overflow map, `clearRequest` issues a `delete` on the overflow entry, zeroing the entire struct. The subsequent callback is then made to `address(0)` with 0 gas — silently succeeding at the EVM level — while the actual requester never receives their price feed data.

---

### Finding Description

`executeCallback` obtains a `storage` reference to the request:

```solidity
Request storage req = findActiveRequest(sequenceNumber);
``` [1](#0-0) 

The function then executes in this order:

**Step 1** — Credit provider fees (reads `req.fee` from storage):

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

**Step 2** — Clear the request:

```solidity
clearRequest(sequenceNumber);
``` [3](#0-2) 

`clearRequest` for an overflow request calls `delete _state.requestsOverflow[key]`, which **zeroes the entire struct** that `req` still points to:

```solidity
function clearRequest(uint64 sequenceNumber) internal {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    Request storage req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        req.sequenceNumber = 0;
    } else {
        delete _state.requestsOverflow[key];  // zeroes all fields
    }
}
``` [4](#0-3) 

**Step 3** — Make the external callback using the now-zeroed `req`:

```solidity
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
``` [5](#0-4) 

After `delete`, `req.requester == address(0)` and `req.callbackGasLimit == 0`. A call to `address(0)` with 0 gas returns `success = true` at the EVM level, so the `try` block succeeds silently. The actual requester's `_echoCallback` is never invoked.

A request lands in the overflow map whenever two sequence numbers hash to the same primary slot. `allocRequest` moves the existing occupant to overflow when a new request takes the primary slot:

```solidity
function allocRequest(uint64 sequenceNumber) internal returns (Request storage req) {
    (, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (isActive(req)) {
        (bytes32 reqKey, ) = requestKey(req.sequenceNumber);
        _state.requestsOverflow[reqKey] = req;
    }
}
``` [6](#0-5) 

The primary slot index is `uint8(hash[0] & NUM_REQUESTS_MASK)`, so with a small mask the collision rate is high under normal load.

---

### Impact Explanation

For any request that was displaced into the overflow map:

- The provider is credited the full fee.
- The request is permanently cleared from storage.
- The requester's `_echoCallback` is **never called**.
- The requester has no recourse — the request is gone and no refund mechanism exists.

This is a permanent, silent loss of service for the requester. Any DeFi protocol relying on Echo for price-triggered callbacks (liquidations, settlements, etc.) would silently miss the callback with no on-chain indication of failure.

---

### Likelihood Explanation

Overflow requests arise naturally whenever two active requests share the same primary slot index. With `shortHash = uint8(hash[0] & NUM_REQUESTS_MASK)` and a small mask, collisions occur in normal operation. Any unprivileged caller can trigger `executeCallback` for an overflow request, making this reachable without any special privileges.

---

### Recommendation

Save `req.requester` and `req.callbackGasLimit` to **memory** variables before calling `clearRequest`, mirroring the pattern already used in `Entropy.sol`'s else-branch:

```solidity
// Save to memory BEFORE clearRequest destroys the storage slot
address callbackTarget = req.requester;
uint32 callbackGas = req.callbackGasLimit;
uint96 reqFee = req.fee;

_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((reqFee + msg.value) - pythFee);

clearRequest(sequenceNumber);

// Now safe to use memory copies
try
    IEchoConsumer(callbackTarget)._echoCallback{gas: callbackGas}(sequenceNumber, priceFeeds)
```

This is exactly the pattern `Entropy.sol` uses in its legacy path:

```solidity
address callAddress = req.requester;
// ...
clearRequest(provider, sequenceNumber);
// WARNING: DO NOT USE req BELOW HERE AS ITS CONTENTS HAS BEEN CLEARED
// ...
IEntropyConsumer(callAddress)._entropyCallback(...)
``` [7](#0-6) 

---

### Proof of Concept

1. User A submits a request; it is assigned sequence number `X`, which hashes to primary slot `S`. The request is stored in `_state.requests[S]`.
2. User B submits a request; it is assigned sequence number `Y`, which also hashes to slot `S`. `allocRequest` moves User A's request to `_state.requestsOverflow[key_X]` and stores User B's request in `_state.requests[S]`.
3. A provider calls `executeCallback(provider, X, updateData, priceIds)`.
4. `findActiveRequest(X)` returns a `storage` reference pointing to `_state.requestsOverflow[key_X]` (User A's request, with `req.requester = userA`, `req.callbackGasLimit = gasA`).
5. Provider fees are credited correctly using `req.fee`.
6. `clearRequest(X)` calls `delete _state.requestsOverflow[key_X]`, zeroing the struct. Now `req.requester == address(0)`, `req.callbackGasLimit == 0`.
7. `IEchoConsumer(address(0))._echoCallback{gas: 0}(X, priceFeeds)` is called. The EVM call to `address(0)` returns `success = true`.
8. `emitPriceUpdate` is emitted, making the transaction appear successful on-chain.
9. User A's contract never receives `_echoCallback`. Any price-triggered logic (e.g., a liquidation) is permanently missed with no refund.
