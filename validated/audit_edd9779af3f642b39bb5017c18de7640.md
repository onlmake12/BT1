Audit Report

## Title
O(n) Linear Scan in `get_max_update_time` Over Unbounded Pool Entries Enables RPC Performance Degradation - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::get_max_update_time()` performs a full O(n) linear scan over every entry in the tx-pool to compute the maximum `timestamp`, while all other aggregate statistics are maintained as O(1) incrementally-updated cached fields. This function is called on every `tx_pool_info` RPC invocation, which is publicly accessible with no rate limiting. An attacker who fills the pool and repeatedly calls `tx_pool_info` forces repeated full-pool scans, consuming CPU proportional to pool size and call frequency.

## Finding Description
In `tx-pool/src/component/pool_map.rs` at lines 130–136, `get_max_update_time` iterates the entire `entries` collection:

```rust
pub(crate) fn get_max_update_time(&self) -> u64 {
    self.entries
        .iter()
        .map(|(_, entry)| entry.inner.timestamp)
        .max()
        .unwrap_or(0)
}
```

The `PoolMap` struct (lines 60–75) maintains `total_tx_size`, `total_tx_cycles`, `pending_count`, `gap_count`, and `proposed_count` as incrementally updated fields on every `add_entry`/`remove_entry`, but `last_txs_updated_at` is recomputed from scratch on every call. The `tx_pool_info` RPC handler at `rpc/src/module/pool.rs` lines 671–682 calls `tx_pool.get_tx_pool_info()` which invokes this scan on every request. No `rate_limit` or `RateLimit` constructs exist anywhere in `rpc/`. The default `DEFAULT_MAX_TX_POOL_SIZE` is confirmed at `util/app-config/src/legacy/tx_pool.rs` line 20 as `180_000_000` bytes (180 MB), with `DEFAULT_MIN_FEE_RATE` of 1000 shannons/KB at line 10.

## Impact Explanation
This matches **Low (501–2000 points): Any other important performance improvements for CKB**. The O(n) scan over up to ~900,000 entries (180 MB pool / ~200 bytes per minimal tx) imposes measurable CPU cost per RPC call. All other aggregate pool statistics are O(1) cached, making this inconsistency a clear and avoidable performance regression. The scan executes while holding the pool lock, which can delay concurrent pool write operations.

## Likelihood Explanation
The `tx_pool_info` RPC is publicly accessible with no authentication or rate limiting (confirmed: no `rate_limit`/`RateLimit` in `rpc/`). Filling the pool to 180 MB at the default minimum fee rate of 1000 shannons/KB costs approximately 1.8 CKB — a low and realistic cost. Once filled, an attacker can hammer `tx_pool_info` in a tight loop from any RPC client. The combination is straightforwardly reproducible on any public node.

## Recommendation
Maintain `last_txs_updated_at` as a cached `u64` field in `PoolMap`, mirroring the existing pattern for `total_tx_size` and `total_tx_cycles`:

- On **add**: `self.last_txs_updated_at = self.last_txs_updated_at.max(entry.timestamp);`
- On **remove**: update to current wall-clock time on any mutation (monotonic staleness indicator), or maintain a sorted structure if exact max-timestamp semantics are required.

Alternatively, redefine `last_txs_updated_at` as the wall-clock time of the last pool mutation (add or remove), which is trivially O(1) and semantically equivalent for its intended use as a staleness indicator.

## Proof of Concept
1. Fill the tx-pool to `max_tx_pool_size` (180 MB) with minimal transactions paying the minimum fee rate.
2. In a tight loop, call `tx_pool_info`:
   ```bash
   while true; do
     curl -s -X POST http://localhost:8114 \
       -H 'Content-Type: application/json' \
       -d '{"id":1,"jsonrpc":"2.0","method":"tx_pool_info","params":[]}'
   done
   ```
3. Profile CPU usage on the node; each call triggers a full scan of ~900,000 entries inside `get_max_update_time`. Compare CPU load against a node with an empty pool to confirm the O(n) relationship.
