### Title
Deviation Check in `_validateShouldUpdatePrices` Ignores Price Exponent, Producing Incorrect Deviation Calculations - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `_validateShouldUpdatePrices` function in `Scheduler.sol` computes price deviation by comparing raw `int64` price mantissa values without normalizing for the `expo` field. Pyth prices are represented as `price × 10^expo`; if the exponent differs between the stored price and the incoming price, the deviation calculation is silently wrong — either failing to trigger a required update or triggering a spurious one.

---

### Finding Description

In `_validateShouldUpdatePrices`, the deviation check reads:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;
int64 previousPrice = previousFeed.price.price;

if (previousPrice == 0 || currentPrice == 0) { continue; }

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
``` [1](#0-0) 

The `expo` field stored in `PythStructs.Price` is never read here. The actual price is `price × 10^expo`, so two feeds with identical mantissas but different exponents represent completely different real-world prices. The contract stores the full `PriceFeed` struct (including `expo`) in `_state.priceUpdates`, but the deviation logic discards it. [2](#0-1) 

The `UpdateCriteria` struct only stores a single `deviationThresholdBps` value with no exponent-awareness: [3](#0-2) 

---

### Impact Explanation

**False negative (missed update):** If the exponent decreases by 1 (e.g., `-8 → -9`), the same mantissa represents a 10× smaller actual price. A 10× actual price drop produces `0 bps` computed deviation — the update is never triggered even though the real price collapsed.

**False positive (spurious update):** If the exponent increases by 1 (e.g., `-8 → -7`), the same mantissa represents a 10× larger actual price. No real price change produces `90 000 bps` computed deviation — the update fires unconditionally, draining the subscription's balance and paying the keeper for a meaningless update.

Both outcomes corrupt the subscription's intended deviation-based trigger semantics. Downstream consumers reading prices via `_getPricesInternal` receive stale or incorrectly-timed data. [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged caller may invoke `updatePriceFeeds`. The Pyth network has historically changed exponents for price feeds during precision adjustments. When such a transition occurs, the first `updatePriceFeeds` call after the exponent change will compare the new mantissa against the old stored mantissa without normalization. The attacker does not need to forge data — they only need to submit a legitimately signed Pyth update that carries the new exponent. [5](#0-4) 

---

### Recommendation

Before computing deviation, normalize both prices to the same exponent:

```solidity
int32 currentExpo  = priceFeeds[i].price.expo;
int32 previousExpo = previousFeed.price.expo;

// Bring both to the same scale before differencing
// e.g. convert to a common fixed-point representation
```

Alternatively, reject updates where `currentExpo != previousExpo` and require a separate administrative reset when an exponent change is intended, similar to how `_clearRemovedPriceUpdates` resets `priceLastUpdatedAt` when price IDs change. [6](#0-5) 

---

### Proof of Concept

1. Subscription created with `updateOnDeviation = true`, `deviationThresholdBps = 100` (1%), `updateOnHeartbeat = false`.
2. First `updatePriceFeeds` stores: `price = 100_000_000, expo = -8` → actual price = **$1.00**.
3. Pyth network publishes an exponent change; next valid update carries: `price = 100_000_000, expo = -7` → actual price = **$10.00** (900% increase).
4. Unprivileged keeper calls `updatePriceFeeds` with the new data.
5. Deviation computed: `|100_000_000 − 100_000_000| / 100_000_000 × 10_000 = 0 bps`.
6. `0 bps < 100 bps` → `UpdateConditionsNotMet` revert. **The 900% actual price change is invisible to the contract.**

Conversely, with `price = 10_000_000, expo = -7` (actual = $1.00, unchanged):
- Deviation computed: `|10_000_000 − 100_000_000| / 100_000_000 × 10_000 = 9_000 bps`.
- `9_000 bps ≥ 100 bps` → update accepted and keeper paid, despite zero real price movement. [7](#0-6)

    function _validateShouldUpdatePrices(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams storage params,
        SchedulerStructs.SubscriptionStatus storage status,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal view returns (uint256) {
        // Use the most recent timestamp, as some asset markets may be closed.
        // Closed markets will have a publishTime from their last trading period.
        // Since we verify all updates share the same Pythnet slot, we still ensure
        // that all price feeds are synchronized from the same update cycle.
        uint256 updateTimestamp = 0;
        for (uint8 i = 0; i < priceFeeds.length; i++) {
            if (priceFeeds[i].price.publishTime > updateTimestamp) {
                updateTimestamp = priceFeeds[i].price.publishTime;
            }
        }

        // Calculate the minimum acceptable timestamp (clamped at 0)
        // The maximum acceptable timestamp is enforced by the parsePriceFeedUpdatesWithSlots call
        uint256 minAllowedTimestamp = (block.timestamp >
            PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            : 0;

        // Validate that the update timestamp is not too old
        if (updateTimestamp < minAllowedTimestamp) {
            revert SchedulerErrors.TimestampTooOld(
                updateTimestamp,
                block.timestamp
            );
        }

        // Reject updates if they're older than the latest stored ones
        if (
            status.priceLastUpdatedAt > 0 &&
            updateTimestamp <= status.priceLastUpdatedAt
        ) {
            revert SchedulerErrors.TimestampOlderThanLastUpdate(
                updateTimestamp,
                status.priceLastUpdatedAt
            );
        }

        // If updateOnHeartbeat is enabled and the heartbeat interval has passed, trigger update
        if (params.updateCriteria.updateOnHeartbeat) {
            uint256 lastUpdateTime = status.priceLastUpdatedAt;

            if (
                lastUpdateTime == 0 ||
                updateTimestamp >=
                lastUpdateTime + params.updateCriteria.heartbeatSeconds
            ) {
                return updateTimestamp;
            }
        }

        // If updateOnDeviation is enabled, check if any price has deviated enough
        if (params.updateCriteria.updateOnDeviation) {
            for (uint8 i = 0; i < priceFeeds.length; i++) {
                // Get the previous price feed for this price ID using subscriptionId
                PythStructs.PriceFeed storage previousFeed = _state
                    .priceUpdates[subscriptionId][priceFeeds[i].id];

                // If there's no previous price, this is the first update
                if (previousFeed.id == bytes32(0)) {
                    return updateTimestamp;
                }

                // Calculate the deviation percentage
                int64 currentPrice = priceFeeds[i].price.price;
                int64 previousPrice = previousFeed.price.price;

                // Skip if either price is zero to avoid division by zero
                if (previousPrice == 0 || currentPrice == 0) {
                    continue;
                }

                // Calculate absolute deviation basis points (scaled by 1e4)
                uint256 numerator = SignedMath.abs(
                    currentPrice - previousPrice
                );
                uint256 denominator = SignedMath.abs(previousPrice);
                uint256 deviationBps = Math.mulDiv(
                    numerator,
                    10_000,
                    denominator
                );

                // If deviation exceeds threshold, trigger update
                if (
                    deviationBps >= params.updateCriteria.deviationThresholdBps
                ) {
                    return updateTimestamp;
                }
            }
        }

        revert SchedulerErrors.UpdateConditionsNotMet();
    }
