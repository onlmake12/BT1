### Title
`GetTxStatus` and `GetTransactionWithStatus` Only Query `pool_map`, Ignoring `verify_queue` and `orphan` Pool — (`tx-pool/src/service.rs`)

---

### Summary

The `GetTxStatus` and `GetTransactionWithStatus` message handlers in `tx-pool/src/service.rs` only query `tx_pool.pool_map` (and `recent_reject_db` as a fallback). They never consult `verify_queue` or the `orphan` pool. As a result, any transaction that has been accepted into the node but not yet moved to `pool_map` — either because it is still awaiting script verification in `verify_queue`, or because it is waiting for a missing parent in the `orphan` pool — is reported as `Unknown` to any RPC caller. This is the direct CKB analog of the Swafe `recover_backups` bug: a lookup function queries the wrong (old) collection after items have been moved to a different one.

---

### Finding Description

The tx-pool service maintains three distinct collections for in-flight transactions:

| Collection | Purpose |
|---|---|
| `verify_queue` | Transactions awaiting background script verification |
| `orphan` | Transactions whose parent inputs are not yet known |
| `pool_map` | Fully verified, accepted transactions (Pending / Gap / Proposed) |

The lifecycle is: `verify_queue` → `pool_map` (or `orphan` → `verify_queue` → `pool_map`).

The `GetTxStatus` handler reads only from `pool_map`:

```rust
// tx-pool/src/service.rs  L867-L903
let tx_pool = service.tx_pool.read().await;
let ret = if let Some(PoolEntry { status, inner: entry, .. }) =
    tx_pool.pool_map.get_by_id(&id)          // ← only pool_map
{ … }
else if let Some(ref recent_reject_db) = tx_pool.recent_reject { … }
else { Ok((TxStatus::Unknown, None)) };      // ← verify_queue / orphan invisible
```

The `GetTransactionWithStatus` handler has the identical blind spot:

```rust
// tx-pool/src/service.rs  L904-L942
let tx_pool = service.tx_pool.read().await;
let ret = if let Some(PoolEntry { … }) =
    tx_pool.pool_map.get_by_id(&id)          // ← only pool_map
{ … }
else if let Some(ref recent_reject_db) = tx_pool.recent_reject { … }
else { Ok(TransactionWithStatus::with_unknown()) };  // ← verify_queue / orphan invisible
```

By contrast, `get_tx_for_compact_block` (used by `FetchTxs`) correctly searches all three collections:

```rust
// tx-pool/src/service.rs  L1178-L1207
verify_queue.get_tx_by_id(short_id)   // ✓
orphan.get(short_id)                  // ✓
tx_pool.get_tx_from_pool_or_store(…)  // ✓
```

And `exclude_existing_proposal` also correctly checks all three:

```rust
// tx-pool/src/service.rs  L1143-L1159
proposals.retain(|id| !verify_queue.contains_key(id));
proposals.retain(|id| !orphan.contains_key(id));
proposals.retain(|id| !tx_pool.contains_proposal_id(id));
```

The inconsistency is structural: the two RPC-facing status handlers were never updated to cover the `verify_queue` and `orphan` collections that were added later.

---

### Impact Explanation

**Incorrect `Unknown` status for in-flight transactions.** Any transaction that is:

1. A large-cycle transaction sitting in `verify_queue` awaiting background verification, or
2. An orphan transaction in the `orphan` pool waiting for its parent,

will be reported as `TxStatus::Unknown` by both `get_transaction` (verbosity 1 and 2) RPC calls. The transaction has been accepted by the node and is not lost, but the caller cannot distinguish this state from "never seen."

Concrete consequences for an unprivileged RPC caller or transaction sender:

- The caller submits a transaction, immediately polls `get_transaction`, and receives `Unknown`. Believing the submission failed, the caller resubmits. The resubmission is rejected with `Reject::Duplicated` (because `process_tx` does check `verify_queue_contains` and `orphan_contains`), leaving the caller in a confused state with no actionable status.
- An application that gates further actions on a `Pending` status (e.g., a wallet waiting to spend a child output) will stall indefinitely while the parent is in `verify_queue` or `orphan`, because the parent never appears as `Pending`.
- For orphan transactions, the window is not transient: an orphan can remain in the pool until its parent is confirmed on-chain, meaning the `Unknown` status persists for an extended, user-visible period.

---

### Likelihood Explanation

The trigger is straightforward and requires no special privilege:

1. **`verify_queue` path**: Submit any transaction whose declared cycle count exceeds `max_tx_verify_cycles` via a remote peer relay, or submit a local transaction that takes non-trivial time to verify. The transaction will sit in `verify_queue` during verification. Any concurrent `get_transaction` call during that window returns `Unknown`.

2. **`orphan` path**: Submit a child transaction before its parent is in the pool (a common pattern in multi-step transaction chains). The child lands in `orphan`. `get_transaction` on the child returns `Unknown` until the parent is confirmed.

Both paths are reachable by any unprivileged RPC caller or transaction sender and require no special network position.

---

### Recommendation

Update both `GetTxStatus` and `GetTransactionWithStatus` handlers to check `verify_queue` and `orphan` before falling through to `Unknown`, mirroring the pattern already used in `get_tx_for_compact_block` and `exclude_existing_proposal`:

```rust
// In GetTxStatus handler
let verify_queue = service.verify_queue.read().await;
if verify_queue.contains_key(&id) {
    return Ok((TxStatus::Pending, None));  // or a new "Verifying" variant
}
let orphan = service.orphan.read().await;
if orphan.contains_key(&id) {
    return Ok((TxStatus::Pending, None));
}
// then check pool_map as before
```

Apply the same pattern to `GetTransactionWithStatus`. Optionally introduce a distinct `TxStatus::Verifying` variant to give callers finer-grained visibility.

---

### Proof of Concept

1. Configure a node with a low `max_tx_verify_cycles` so that a normal transaction is treated as large-cycle and routed to `verify_queue`.
2. Submit a transaction via `send_transaction` RPC.
3. Immediately call `get_transaction` with the returned hash.
4. Observe that the response carries `status: "unknown"` even though the transaction is present in `verify_queue` (visible via `get_tx_pool_info`'s `verify_queue_size` counter).
5. Confirm that after verification completes and the transaction moves to `pool_map`, the same call returns `status: "pending"`.

For the orphan path: submit a child transaction spending an output that does not yet exist in the pool or chain. Call `get_transaction` on the child hash. Observe `status: "unknown"` despite the transaction being held in the `orphan` pool (visible via `get_tx_pool_info`'s `orphan` counter).
