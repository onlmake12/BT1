### Title
`HeaderAcceptor::accept()` Silently Re-validates Headers Previously Marked `BLOCK_INVALID`, Returning a Misleading Valid Result — (`sync/src/synchronizer/headers_process.rs`)

---

### Summary

The `HeaderAcceptor::accept()` function in the sync subsystem contains a developer-acknowledged `FIXME` where it fails to return early when a header's stored status is `BLOCK_INVALID`. Instead, the function falls through to the full non-contextual validation pipeline. If those checks pass, the header is inserted into the node's valid-header index via `insert_valid_header`, producing a `ValidationResult { state: Valid }` — a misleading success — for a header the node had already classified as invalid. This is the direct CKB analog to the `computeVote` pattern: a failed precondition is silently bypassed and the function returns a success-equivalent result.

---

### Finding Description

In `sync/src/synchronizer/headers_process.rs`, the `HeaderAcceptor::accept()` method begins by fetching the stored `BlockStatus` for the incoming header:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer best-known and return Valid
    return result;
}
```

The only early-exit guard is `BlockStatus::HEADER_VALID`. There is **no guard for `BlockStatus::BLOCK_INVALID`**. The `FIXME` comment at lines 301–302 explicitly acknowledges this omission and the developer's uncertainty about how to fix it.

When `status == BLOCK_INVALID` (and `HEADER_VALID` is absent), execution falls through to:

1. `prev_block_check` — marks invalid and returns if the *parent* is `BLOCK_INVALID`.
2. `non_contextual_check` — runs `HeaderVerifier` (PoW, timestamp, etc.).
3. `version_check` — checks header version field.

If all three pass, the function reaches:

```rust
sync_shared.insert_valid_header(self.peer, self.header);
result  // ValidationResult { state: Valid, error: None }
```

This overwrites the prior `BLOCK_INVALID` status with `HEADER_VALID` and updates the peer's best-known-header record — a silent, misleading success.

**When can a `BLOCK_INVALID` header pass all three checks?**

The `BLOCK_INVALID` status is written by `prev_block_check` when the *parent* was invalid at the time of first processing. If the parent's status later changes (e.g., the parent block arrives and is accepted, or a reorg occurs), the same header re-submitted by a peer will now pass `prev_block_check`, pass the non-contextual PoW/timestamp checks (which were always valid), and pass `version_check`. The function then silently promotes the header to `HEADER_VALID` without any indication that it was previously rejected.

The `execute()` caller in `HeadersProcess` then returns `Status::ok()` to the network layer, and no ban or warning is issued to the peer.

---

### Impact Explanation

- The node's in-memory sync state (`shared_best_header`, per-peer `best_known_header`) can be updated with a header that was previously classified as invalid, without any re-evaluation of the contextual reason for the original rejection.
- A peer can cause the local node to silently reclassify a previously-rejected header as valid, potentially shifting the node's view of the best chain tip used for IBD peer selection and `GetHeaders` targeting.
- Because `insert_valid_header` updates the block-status store, subsequent calls to `get_block_status` for this header will return `HEADER_VALID`, masking the prior `BLOCK_INVALID` classification for all future logic that depends on it.
- The misleading `Status::ok()` returned to the network layer means the peer is not penalized, and the event is not surfaced as a warning — identical to the `VoteResult(0,0,0,0)` silent-success pattern in the reference report.

---

### Likelihood Explanation

An unprivileged sync peer can trigger this path by:

1. Sending a header whose parent is not yet known to the local node (parent arrives later via normal sync), causing the header to be marked `BLOCK_INVALID` via `prev_block_check`.
2. Waiting for the parent to be accepted (normal IBD or relay).
3. Re-sending the same header. The `BLOCK_INVALID` guard is absent; the header passes all three checks and is silently promoted.

No mining, no privileged access, and no Sybil attack is required. The scenario arises naturally during normal out-of-order header delivery and is therefore reachable by any connected peer.

---

### Recommendation

Add an explicit `BLOCK_INVALID` guard immediately after fetching `status`, before the `HEADER_VALID` check, resolving the open `FIXME`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    state.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // existing logic
    return result;
}
```

A dedicated `ValidationError::PreviouslyRejected` variant would make the failure reason explicit and allow callers to distinguish this case from a fresh validation failure.

---

### Proof of Concept

**Entry path:** unprivileged sync peer → `SendHeaders` P2P message → `HeadersProcess::execute()` → `HeaderAcceptor::accept()`.

**Trigger sequence:**

1. Peer sends `SendHeaders` containing header `H` whose parent `P` is unknown. `prev_block_check` marks `H` as `BLOCK_INVALID` (parent not found → treated as invalid parent path).
2. Normal sync delivers block `P`; it is accepted and stored.
3. Peer re-sends `SendHeaders` containing `H`.
4. `accept()` fetches `status = BLOCK_INVALID`. The `HEADER_VALID` guard (line 304) is false. No `BLOCK_INVALID` guard exists (FIXME, lines 301–302). Execution falls through.
5. `prev_block_check`: parent `P` is now stored and not `BLOCK_INVALID` → passes.
6. `non_contextual_check`: `H` has valid PoW and timestamp → passes.
7. `version_check`: `H.version == 0` → passes.
8. `sync_shared.insert_valid_header(peer, H)` is called (line 356). `H` is now `HEADER_VALID`.
9. `execute()` returns `Status::ok()`. No ban, no warning, no error.

**Observable effect:** `shared.state().shared_best_header()` and the peer's `best_known_header` are updated to reflect `H`, which was previously classified as invalid, with no indication of the prior rejection. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
