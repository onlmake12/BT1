### Title
Division Before Multiplication in Fee Rate Estimator Causes Precision Loss, Returning Underestimated Fee Rates - (File: `util/fee-estimator/src/estimator/weight_units_flow.rs`)

### Summary

The `WeightUnitsFlow` fee estimator algorithm divides bucket weights by `historical_blocks` before multiplying by `target_blocks` when computing `added_weight`. This integer division-before-multiplication causes systematic precision loss, causing the estimator to underestimate how much weight will be added to the mempool during the target window, and therefore return a fee rate that is lower than the true required rate.

### Finding Description

In `do_estimate`, the flow speed per bucket is computed by dividing total accumulated weight by `historical_blocks`:

```rust
// util/fee-estimator/src/estimator/weight_units_flow.rs, line 268-271
buckets
    .into_iter()
    .map(|value| value / historical_blocks)   // <-- integer division first
    .collect::<Vec<_>>()
``` [1](#0-0) 

The resulting `flow_speed_buckets` values are then multiplied by `target_blocks` to compute `added_weight`:

```rust
// line 281
let added_weight = flow_speed_buckets[bucket_index] * target_blocks;
``` [2](#0-1) 

The correct order is to multiply first, then divide:

```
added_weight = (bucket_total_weight * target_blocks) / historical_blocks
```

Instead, the code computes:

```
added_weight = (bucket_total_weight / historical_blocks) * target_blocks
```

For any bucket where `bucket_total_weight < historical_blocks`, the integer division truncates to zero, making `added_weight = 0` regardless of `target_blocks`. For example, if a bucket accumulated `5` weight units over `historical_blocks = 10` blocks, the correct `added_weight` for `target_blocks = 3` is `5 * 3 / 10 = 1`, but the code computes `(5 / 10) * 3 = 0 * 3 = 0`.

The `FeeRate` type itself correctly avoids this pattern — `FeeRate::calculate` multiplies by `KW` before dividing by weight:

```rust
FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
``` [3](#0-2) 

But the flow speed computation in the estimator does not follow this pattern.

### Impact Explanation

The `added_weight` term represents the expected inflow of new transactions during the target window. When it is truncated to zero (or underestimated), the condition:

```rust
let passed = current_weight + added_weight <= removed_weight;
``` [4](#0-3) 

becomes easier to satisfy at lower fee rate buckets than it should be. The estimator therefore returns a fee rate that is lower than the true required rate. Users who rely on this estimate submit transactions with insufficient fees, causing those transactions to remain unconfirmed in the mempool longer than expected or to be evicted.

### Likelihood Explanation

This is triggered on every call to `estimate_fee_rate` via the `WeightUnitsFlow` algorithm whenever any bucket's accumulated weight over the historical window is less than `historical_blocks` (which equals `target_blocks * 2`). This is a common condition during low-traffic periods or for high-fee-rate buckets with sparse transactions. Any unprivileged RPC caller invoking `estimate_fee_rate` or `get_fee_rate_statistics` exercises this path. [5](#0-4) 

### Recommendation

Defer the division to after the multiplication. Replace:

```rust
buckets
    .into_iter()
    .map(|value| value / historical_blocks)
    .collect::<Vec<_>>()
```

with storing raw totals and computing:

```rust
let added_weight = (raw_bucket_total * target_blocks) / historical_blocks;
```

This matches the pattern already used correctly in `FeeRate::calculate` and in the DAO reward calculations throughout the codebase. [6](#0-5) 

### Proof of Concept

Suppose `historical_blocks = 20`, `target_blocks = 10`, and a bucket accumulated `15` weight units over the historical window.

**Current (buggy) code:**
- `flow_speed = 15 / 20 = 0` (integer truncation)
- `added_weight = 0 * 10 = 0`

**Correct computation:**
- `added_weight = (15 * 10) / 20 = 150 / 20 = 7`

With `added_weight = 0`, the estimator sees `current_weight + 0 <= removed_weight` and selects a lower fee rate bucket than warranted. A user following this estimate submits a transaction at too low a fee rate and finds it stuck in the mempool. [7](#0-6)

    pub fn estimate_fee_rate(
        &self,
        target_blocks: BlockNumber,

            return Err(Error::NotReady);

        let sorted_current_txs = {
            let mut current_txs: Vec<_> = all_entry_info
                .pending
                .into_values()
                .chain(all_entry_info.proposed.into_values())

            current_txs
        };

        self.do_estimate(target_blocks, &sorted_current_txs)
    }
