### Title
Unbounded O(ADDR_COUNT_LIMIT) Peer Store Scan Per Inbound GetNodes Message with Mutex Contention — (`network/src/protocols/discovery/mod.rs`)

---

### Summary

An unprivileged inbound peer can trigger an O(16384) Fisher-Yates shuffle over the entire peer store address list — while holding the global `peer_store` mutex — once per new TCP session, with no cross-session rate limiting. The per-session deduplication guard only prevents a second GetNodes from the *same* session; a new connection resets it entirely.

---

### Finding Description

**Entrypoint — `received`, line 120:**

`get_random(2500, required_flags)` is called unconditionally with a hardcoded `2500` before the `count` field from the attacker's message is ever consulted. [1](#0-0) 

The count cap is applied only *after* the expensive call returns: [2](#0-1) 

**Work done inside `get_random` — `fetch_random`, lines 53–95:**

`fetch_random` iterates over every entry in `random_ids` (up to `ADDR_COUNT_LIMIT = 16384`), performing a Fisher-Yates shuffle step (two `HashMap` lookups + a swap) and a filter check per entry. It only breaks early when `count` results pass the filter. [3](#0-2) 

`ADDR_COUNT_LIMIT` is confirmed at 16384: [4](#0-3) 

**Worst-case filter sparsity:**

`fetch_random_addrs` filters for addresses connected within 7 days (`ADDR_TIMEOUT_MS`). A peer store populated with 16384 discovered-but-never-connected addresses causes the loop to scan all 16384 entries before returning an empty result. [5](#0-4) 

**Mutex held for the entire scan:**

`get_random` acquires `peer_store: Mutex<PeerStore>` via `with_peer_store_mut` for the full duration of the O(16384) shuffle. All other subsystems that need the peer store (outbound peer service, identify, feeler, DNS seeding, dump service) are blocked during this time. [6](#0-5) [7](#0-6) 

**Per-session guard does not prevent cross-session attacks:**

`received_get_nodes` is set to `true` after the first GetNodes in a session. A second GetNodes from the same session triggers disconnect. However, a new TCP connection creates a fresh `SessionState` with `received_get_nodes = false`, so the attacker gets one full O(16384) scan per new connection. [8](#0-7) 

---

### Impact Explanation

Each inbound connection that sends one GetNodes message causes:
1. An O(16384) Fisher-Yates shuffle with HashMap lookups while holding the global `peer_store` mutex.
2. Serialization of all other peer store operations (connection management, outbound dialing, identify, feeler) behind this lock.

With `max_inbound` concurrent connections (default ~125), an attacker can sustain continuous mutex contention on the peer store, degrading the node's ability to manage connections and participate in block/transaction relay.

---

### Likelihood Explanation

The path requires only the ability to open TCP connections to the node's P2P port — no authentication, no PoW, no privileged role. The attack is repeatable: open connection → send GetNodes → disconnect → repeat. The only cost to the attacker is TCP connection establishment; the victim pays O(16384) mutex-held work per connection.

---

### Recommendation

1. Apply the `count` cap **before** calling `get_random`: compute `let n = min(MAX_ADDR_TO_SEND, count as usize)` and pass `n` instead of the hardcoded `2500`.
2. Add a per-IP or global rate limit on new discovery sessions to bound the connection-churn attack rate.
3. Consider releasing the peer store mutex before the shuffle by snapshotting `random_ids` into a local `Vec`, shuffling locally, then doing targeted lookups.

---

### Proof of Concept

```
1. Fill a test peer store with 16384 entries, none connected within 7 days.
2. Open an inbound TCP connection to the CKB node.
3. Send a valid GetNodes message (required_flags=0, count=u32::MAX).
4. Observe: get_random(2500, COMPATIBILITY) iterates all 16384 entries
   while holding peer_store mutex.
5. Disconnect. Repeat from step 2 with a new connection.
6. Benchmark: measure peer_store mutex hold time per GetNodes;
   simulate 100 concurrent connections each sending GetNodes and
   measure latency of concurrent peer store operations.
```

The existing benchmark at `network/src/benches/peer_store.rs` already measures `fetch_random_addrs` at sizes 10,000–20,000 and confirms the O(n) scaling. [9](#0-8)

    fn get_random(&mut self, n: usize, flags: Flags) -> Vec<(Multiaddr, Flags)> {
        let fetch_random_addrs = self
            .network_state
            .with_peer_store_mut(|peer_store| peer_store.fetch_random_addrs(n, flags));
        let addrs = fetch_random_addrs
            .into_iter()
            .filter_map(|paddr| {
                if !self.is_valid_addr(&paddr.addr) {
                    return None;
                }
                let f = Flags::from_bits_truncate(paddr.flags);
                Some((paddr.addr, f))
            })
            .collect();
        trace!("Discovered random addrs: {:?}", addrs);
        addrs
    }

    pub fn fetch_random_addrs(&mut self, count: usize, required_flags: Flags) -> Vec<AddrInfo> {
        // Get info:
        // 1. Connected within 7 days

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let addr_expired_ms = now_ms.saturating_sub(ADDR_TIMEOUT_MS);

        let filter = |peer_addr: &AddrInfo| {
            required_flags_filter(required_flags, Flags::from_bits_truncate(peer_addr.flags))
                && peer_addr.connected(|t| t > addr_expired_ms)
        };

        // get success connected addrs.
        self.addr_manager.fetch_random(count, filter)
    }
