### Title
Cross-Chain Replay of Lazer Price Updates Due to Missing Chain ID in Signature Binding - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate` computes the signed hash as `keccak256(payload)` with no chain ID binding. Because `PythLazer` is deployed on many EVM chains sharing the same trusted signer set, a valid signed Lazer update accepted on one chain is cryptographically identical to a valid update on every other chain. An unprivileged relayer/updater can replay a signed update from one EVM deployment on any other EVM deployment.

### Finding Description
In `verifyUpdate`, the signature is verified against a hash of the raw payload bytes only:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);          // no chain ID
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The payload itself contains only: a format magic, a timestamp, a channel byte, and feed price data. [2](#0-1) 

There is no `block.chainid`, no contract address, and no destination-chain identifier anywhere in the signed message. The `EVM_FORMAT_MAGIC` constant (`706910618`) is identical across all EVM deployments. [3](#0-2) 

`PythLazer` is explicitly deployed to many EVM chains (Ethereum, Arbitrum, BNB Chain, etc.) using the same trusted signer set managed via `updateTrustedSigner`. [4](#0-3) 

### Impact Explanation
A Lazer price update signed by a trusted Pyth signer for Chain A passes `verifyUpdate` on Chain B without modification. An attacker acting as a relayer/updater can:

1. **Replay stale prices**: Capture a signed update from Chain A at time T (when price was P), wait until the price on Chain B diverges, then submit the old signed update to Chain B's `PythLazer`. Consumer contracts on Chain B will receive stale price P as if it were current.
2. **Testnet-to-mainnet replay**: If the same trusted signer key is registered on both testnet and mainnet (a common operational pattern), a testnet-signed update with an attacker-influenced price can be replayed on mainnet.
3. **Fork replay**: In a chain-fork scenario, updates signed for one fork are valid on the other.

Consumer contracts that use `verifyUpdate` to gate financial decisions (e.g., liquidations, collateral pricing, derivatives settlement) would act on incorrect price data, leading to direct financial loss.

### Likelihood Explanation
`PythLazer` is already deployed across many EVM chains. The attack requires only the ability to call `verifyUpdate` with a previously observed signed update — no privileged access, no key compromise. Any party who has ever received a Lazer update (e.g., by subscribing to the Lazer WebSocket stream or observing on-chain transactions) possesses the necessary signed bytes. The attacker entry point is the public `verifyUpdate` function itself. [5](#0-4) 

### Recommendation
Bind the signed hash to the destination chain by including `block.chainid` (and optionally `address(this)`) in the hash:

```solidity
bytes32 hash = keccak256(abi.encodePacked(block.chainid, address(this), payload));
```

This requires a corresponding change in the Lazer server's signing logic to include the target chain ID when producing the EVM-format signature. Alternatively, adopt EIP-712 domain separation with `chainId` and `verifyingContract` fields.

### Proof of Concept

1. Deploy `PythLazer` on Chain A (e.g., Ethereum, `chainId=1`) and Chain B (e.g., Arbitrum, `chainId=42161`) with the same trusted signer.
2. On Chain A, call `verifyUpdate{value: fee}(update)` with a valid signed Lazer update. It succeeds and returns `payload`.
3. Take the identical `update` bytes and call `verifyUpdate{value: fee}(update)` on Chain B's `PythLazer` contract.
4. The call succeeds — `keccak256(payload)` is identical on both chains, the ECDSA signature recovers the same trusted signer address, and `isValidSigner` returns `true`.
5. A consumer contract on Chain B now processes a price update that was signed for Chain A, with no indication of the mismatch.

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
