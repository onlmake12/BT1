### Title
Permissionless `update_guardian_set_ttl` Overwrites Configurable TTL with Hardcoded 24-Hour Constant — (File: `target_chains/solana/programs/core-bridge/src/legacy/processor/update_guardian_set_ttl.rs`)

---

### Summary

The `update_guardian_set_ttl` instruction in the Solana Core Bridge program is permissionless and unconditionally overwrites `config.guardian_set_ttl` with a hardcoded constant of `86400` seconds (24 hours), ignoring any governance-set value. This is a direct analog to the reported issue: a fixed constant is used instead of a configurable stored parameter, and the function is callable by any unprivileged transaction sender.

---

### Finding Description

In `target_chains/solana/programs/core-bridge/src/legacy/processor/update_guardian_set_ttl.rs`, the constant `GUARDIAN_SET_TTL_SECONDS = 86400` is hardcoded and unconditionally written to `config.guardian_set_ttl`:

```rust
const GUARDIAN_SET_TTL_SECONDS: u32 = 86400;

fn update_guardian_set_ttl(ctx: Context<UpdateGuardianSetTtl>, _args: EmptyArgs) -> Result<()> {
    ctx.accounts.config.guardian_set_ttl = GUARDIAN_SET_TTL_SECONDS.into();
    Ok(())
}
```

The instruction is explicitly documented as permissionless — no signer check, no governance check, no access control of any kind. [1](#0-0) 

The `Config` struct stores `guardian_set_ttl` as a configurable field, initialized from `InitializeArgs.guardian_set_ttl_seconds` at deployment: [2](#0-1) 

This TTL is consumed in `guardian_set_update` to set the expiration time of the outgoing guardian set at the moment of a guardian set rotation: [3](#0-2) 

---

### Impact Explanation

The `guardian_set_ttl` controls how long a superseded guardian set remains valid for VAA verification after being replaced. If governance intentionally sets a short TTL (e.g., to rapidly invalidate a compromised guardian set after a rotation), any unprivileged attacker can call `update_guardian_set_ttl` immediately before the guardian set rotation executes, resetting the TTL to 24 hours. The old (potentially compromised) guardian set will then remain valid for VAA verification for 24 hours instead of the governance-intended shorter window.

Conversely, if governance sets a longer TTL to allow more time for VAA relay, an attacker can shorten it to 24 hours, causing premature expiry of the old guardian set and breaking in-flight VAA verification.

Since Pyth's price update and governance VAA verification on Solana depends on the Wormhole Core Bridge guardian set validity check, manipulation of the TTL directly affects the security window of Pyth's VAA-based price feed and governance message processing.

---

### Likelihood Explanation

The attack requires only a standard Solana transaction with no special privileges. The attacker simply calls `LegacyInstruction::UpdateGuardianSetTtl` at any time. The attack is most impactful in the narrow window between a governance decision to rotate guardian sets with a non-default TTL and the actual execution of the `guardian_set_update` instruction. This window is realistic and observable on-chain.

---

### Recommendation

Remove the permissionless `update_guardian_set_ttl` instruction entirely, or restrict it to a governance-authorized caller. If the intent is to allow resetting the TTL to a canonical value, the value should be passed as a governance-verified parameter rather than hardcoded, and the instruction should require a valid governance VAA — analogous to how `SetMessageFee` and `TransferFees` are gated. [4](#0-3) 

---

### Proof of Concept

1. Governance decides to rotate the guardian set and sets `config.guardian_set_ttl` to `3600` (1 hour) to quickly invalidate a compromised old guardian set.
2. Before the `GuardianSetUpdate` governance instruction is executed, an attacker submits a transaction calling `LegacyInstruction::UpdateGuardianSetTtl` (no arguments, no signer restriction).
3. `update_guardian_set_ttl` executes and writes `config.guardian_set_ttl = 86400`.
4. The `GuardianSetUpdate` instruction then executes and sets `curr_guardian_set.expiration_time = now + 86400` instead of `now + 3600`.
5. The old (compromised) guardian set remains valid for VAA verification for 24 hours instead of 1 hour, defeating the governance intent. [5](#0-4) [6](#0-5)
