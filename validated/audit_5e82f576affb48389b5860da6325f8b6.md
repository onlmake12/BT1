### Title
`total_tx_size` Over-Counted After Ancestor Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

### Summary

In `PoolMap::add_entry`, the updated pool-size totals are computed **before** `check_and_record_ancestors` runs. When that function evicts cell-dep-referencing ancestor transactions via `remove_entry_and_descendants`, those removals correctly decrement `self.total_tx_size`. However, the final assignment `self.total_tx_size = total_tx_size` then overwrites with the **pre-eviction** value, permanently inflating the tracked pool size by the byte-size of every evicted entry. The downstream `limit_size` call then evicts legitimate transactions to compensate for the phantom inflation, causing valid transactions to be rejected with `Reject::Full`.

---

### Finding Description

`PoolMap::add_entry` executes in this order:

```
1. updated_stat_for_add_tx(entry.size, entry.cycles)   // snapshot T_new = T_old + entry.size
2. check_and_record_ancestors(&mut entry)               // may call remove_entry_and_descendants
                                                        //   → self.total_tx_size -= S_evicted
3. insert_entry(&entry, status)
4. self.total_tx_size = total_tx_size                   // OVERWRITES with T_old + entry.size
``` [1](#0-0) 

`updated_stat_for_add_tx` only checks for arithmetic overflow; it does **not** enforce `max_tx_pool_size`. It captures the pre-eviction total: [2](#0-1) 

`check_and_record_ancestors` is the eviction site. When a new transaction's ancestor count exceeds `max_ancestors_count` but the excess is entirely composed of cell-dep-referencing parents (`cell_ref_parents`), it calls `remove_entry_and_descendants` in a loop, which correctly decrements `self.total_tx_size` for each removed entry: [3](#0-2) 

After `check_and_record_ancestors` returns, `self.total_tx_size` has been correctly reduced by `S_evicted`. But the final two lines of `add_entry` overwrite it:

```rust
self.total_tx_size = total_tx_size;   // T_old + entry.size  (ignores evictions)
self.total_tx_cycles = total_tx_cycles;
``` [4](#0-3) 

**Correct post-state**: `T_old − S_evicted + entry.size`
**Actual post-state**: `T_old + entry.size`
**Over-count**: `S_evicted` bytes

The `limit_size` function is called immediately after `_submit_entry` in `submit_entry`: [5](#0-4) 

`limit_size` loops while `total_tx_size > max_tx_pool_size` and evicts real entries to compensate: [6](#0-5) 

Because `total_tx_size` is inflated by `S_evicted`, `limit_size` evicts `S_evicted` bytes of legitimate, already-accepted transactions that should not have been removed.

---

### Impact Explanation

Every time the eviction branch of `check_and_record_ancestors` fires, the pool's size accounting is permanently inflated by the byte-size of the evicted entries. `limit_size` then compensates by evicting an equal volume of legitimate pending/proposed transactions, issuing `Reject::Full` for them. An attacker who repeatedly triggers this path can continuously drain the pool of honest transactions, degrading mempool availability for all users. The pool's `total_tx_size` counter drifts further from reality with each attack cycle, making the effect cumulative.

---

### Likelihood Explanation

The eviction branch requires:
- A new transaction whose ancestor count exceeds `max_ancestors_count` (default 25), **and**
- The excess ancestors are entirely `cell_ref_parents` (transactions that reference the same cell dep as the new transaction).

An unprivileged attacker can construct this scenario cheaply:
1. Submit ≥ 26 independent transactions each carrying a `cell_dep` pointing to the same live output (e.g., a widely-used lock script cell).
2. Submit a transaction that **consumes** that cell output as an input.

Step 2 triggers the eviction path. The attacker pays fees for ~26 small transactions per cycle. This is reachable via the public `send_transaction` RPC or via P2P transaction relay, requiring no privileged access. [7](#0-6) 

---

### Recommendation

Move `updated_stat_for_add_tx` to **after** `check_and_record_ancestors` so the computed totals reflect the post-eviction pool state before the new entry is counted:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;   // evictions happen first
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;  // now T_old is post-eviction
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, subtract the aggregate size of `evicts` from `total_tx_size` before the final assignment.

---

### Proof of Concept

```
Setup:
  max_ancestors_count = 25 (default)
  pool near-empty, max_tx_pool_size = 180 MB

Step 1: Submit 26 transactions T_ref[0..25], each with:
  - a unique input (independent)
  - cell_dep pointing to the same live cell C

Step 2: Submit transaction T_consume that:
  - spends cell C as an input

Execution inside add_entry for T_consume:
  - get_tx_ancenstors returns ancestors = {T_ref[0..25]}, cell_ref_parents = {T_ref[0..25]}
  - ancestors_count = 27 > max_ancestors_count (25)
  - 27 - 26 = 1 <= 25  → eviction branch taken
  - remove_entry_and_descendants(T_ref[0]) called:
      self.total_tx_size -= size(T_ref[0])   // correct decrement
  - remove_entry_and_descendants(T_ref[1]) called:
      self.total_tx_size -= size(T_ref[1])   // correct decrement
  - loop exits (ancestors_count now <= 25)
  - insert_entry(T_consume)
  - self.total_tx_size = total_tx_size       // OVERWRITE: restores T_old + size(T_consume)
                                             // ignores the two decrements above

Result:
  total_tx_size is inflated by size(T_ref[0]) + size(T_ref[1])
  limit_size fires and evicts size(T_ref[0]) + size(T_ref[1]) bytes of legitimate txs
  Those legitimate txs receive Reject::Full and are dropped from the pool
```

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
