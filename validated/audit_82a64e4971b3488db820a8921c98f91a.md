Audit Report

## Title
Rich-Indexer `insert_transaction` Unconditional `break` Leaves Subsequent Inputs Unspent When Cell Filter Is Active — (`util/rich-indexer/src/indexer/mod.rs`)

## Summary
In `insert_transaction`, the input-processing loop unconditionally `break`s when `spend_cell` returns `false` for any input. Because `spend_cell` returns `false` for any cell absent from the `output` table — the normal outcome for cells excluded by `cell_filter` — any multi-input transaction whose first input references a non-indexed cell will silently skip marking all subsequent inputs as spent, even if those inputs reference indexed cells. This produces persistent phantom live-cell state in the DB.

## Finding Description
In `util/rich-indexer/src/indexer/mod.rs` lines 228–248, the input loop is:

```rust
if tx_index != 0 {
    for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
        let out_point = input.previous_output();
        if !spend_cell(&out_point, tx).await? {
            break;   // line 232 — exits entire loop unconditionally
        }
        // build_input_rows, set is_tx_matched ...
    }
}
```

`spend_cell` (`util/rich-indexer/src/indexer/insert.rs` lines 403–427) issues `UPDATE output SET is_spent = 1 WHERE tx_id = ... AND output_index = ...` and returns `Ok(rows_affected > 0)`. It returns `false` whenever the referenced cell is not present in the `output` table — the expected outcome for any cell excluded by `cell_filter`. The `break` then terminates the loop entirely, so every input after the first non-indexed one is never visited. If any of those skipped inputs reference cells that ARE indexed, their `is_spent` column remains `0` permanently in the persistent DB.

The `get_cells` query (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs` line 107) filters with `.and_where("output.is_spent = 0")`, so any cell whose `is_spent` was never flipped will be returned as a live cell indefinitely, surviving node restarts.

The root cause is that `break` is used where `continue` is semantically correct: a non-indexed input should be silently skipped, not used as a signal to abort processing all remaining inputs.

## Impact Explanation
This is a persistent data-integrity corruption in the rich indexer's state storage. Cells that have been consumed on-chain remain visible as live cells in the indexer DB across node restarts. Wallets and dApps relying on the indexer compute incorrect balances and may attempt to re-spend already-consumed cells. This matches **Medium (2001–10000 points): Suboptimal/incorrect implementation of CKB state storage mechanism**, as the rich indexer is the designated CKB state storage and query layer, and the corruption is persistent and silent.

## Likelihood Explanation
The precondition — `cell_filter` active — is a supported, documented operator configuration. Once enabled, any valid block containing a multi-input transaction where the first input's cell is not indexed (e.g., created before the indexer started, or belonging to a different script type than the filter) triggers the bug. No special attacker capability is required; normal P2P block relay is sufficient. The condition is common in production deployments that use script-type filtering.

## Recommendation
Replace `break` with `continue` at line 232 of `util/rich-indexer/src/indexer/mod.rs`:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // cell not in indexer — skip, but process remaining inputs
}
```

This preserves existing semantics (non-indexed cells are silently skipped) while ensuring all inputs are visited so that indexed cells are correctly marked spent.

## Proof of Concept
1. Start a CKB node with the rich indexer and a `cell_filter` matching only cells with script `S`.
2. Index a block containing tx `T1` creating `output[0]` with script `S` (indexed, `is_spent=0`).
3. Construct tx `T2` with two inputs: `input[0]` spending a cell with script `X` (not in `output` table), `input[1]` spending `T1:output[0]` (indexed).
4. Submit a valid block containing `T2`.
5. After `append`, query: `SELECT is_spent FROM output WHERE ...` for `T1:output[0]`.
6. **Expected**: `is_spent = 1`. **Actual**: `is_spent = 0` — phantom live cell.
7. Call `get_cells` for script `S`; `T1:output[0]` is returned as live despite being consumed on-chain.

```rust
pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
}
```
