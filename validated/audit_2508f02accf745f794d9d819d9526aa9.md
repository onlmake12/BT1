Audit Report

## Title
Integer Overflow in Rate-Limit Guard Enables Unbounded O(chain-length) DB Reads — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary

In `GetLastStateProofProcess::execute`, the guard at line 201 computes `(last_n_blocks as usize) * 2` without overflow protection. In Rust release builds, this multiplication wraps on 64-bit targets when `last_n_blocks > usize::MAX / 2`, producing a tiny value that silently passes the `> 1000` check. The downstream comparison at line 291 then evaluates as always-true for any realistic chain, causing the server to collect and process every block from `start_block_number` to the chain tip — O(chain length) memory allocation and DB reads — per request from any unauthenticated P2P peer.

## Finding Description

**Guard check** (`get_last_state_proof.rs`, lines 199–205):
```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```
On a 64-bit host, `usize` is 64 bits; the cast `last_n_blocks as usize` is a no-op. Rust release builds use wrapping (two's-complement) arithmetic. With `last_n_blocks = 0x8000_0000_0000_0001`:

```
(0x8000_0000_0000_0001usize) * 2  →  0x0000_0000_0000_0002  (wraps)
0 + 2 > 1000  →  false  ← guard silently passes
```

**Downstream allocation** (`get_last_state_proof.rs`, lines 291–297):
```rust
let (sampled_numbers, last_n_numbers) = if last_block_number - start_block_number
    <= last_n_blocks
{
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
```
With `last_n_blocks = 0x8000_0000_0000_0001` (~9.2 × 10¹⁸), the condition is always true for any realistic chain. The `if` branch is always taken, and `last_n_numbers` collects every block number from `start_block_number` to `last_block_number`.

**DB reads** (`get_last_state_proof.rs`, lines 350–365):
```rust
let block_numbers = reorg_last_n_numbers
    .into_iter()
    .chain(sampled_numbers)
    .chain(last_n_numbers)
    .collect::<Vec<_>>();
// ...
sampler.complete_headers(&mut positions, &last_block_hash, &block_numbers)
```
`complete_headers` performs one DB read per entry in `block_numbers`. With the overflow, `block_numbers` contains every block on the chain.

**Why intermediate checks do not block the exploit:**
- `snapshot.is_main_chain(&last_block_hash)` (line 210): passes with any known tip hash.
- `start_block_number > last_block_number` (line 231): passes with `start_number = 0`.
- Difficulties checks (lines 252–288): with an empty `difficulties` vec, all `difficulties.first()` / `difficulties.last()` guards short-circuit via `unwrap_or(false)` and are skipped entirely.
- `reorg_last_n_numbers` (line 237): `start_block_number == 0` takes the `Vec::new()` branch.

No other guard exists between message receipt and the unbounded allocation.

## Impact Explanation

On CKB mainnet (~10 million blocks), each crafted request causes:
- Allocation of a `Vec<u64>` with ~10 million entries (~80 MB).
- ~10 million sequential DB reads in `complete_headers`.

Any number of peers can send this message concurrently. Repeated requests exhaust heap memory and saturate disk I/O, crashing the light-client server node. This matches the **High** bounty impact: *"Vulnerabilities which could easily crash a CKB node."*

## Likelihood Explanation

- No authentication or privilege is required; any P2P peer can send a `GetLastStateProof` message.
- The entry path `received → try_process → execute` is confirmed in `lib.rs` lines 55–92 and 108–112.
- The only prerequisite is a valid main-chain block hash for `last_hash`, trivially obtained from any block explorer or by syncing.
- Rust release builds use wrapping arithmetic by specification; this is not theoretical.
- The attack is repeatable and stateless — each request is independent.

## Recommendation

Replace the unchecked multiplication with a saturating variant, or add an early bound check before the combined expression:

```rust
// Option 1: saturating arithmetic
let combined = (last_n_blocks as usize)
    .saturating_mul(2)
    .saturating_add(self.message.difficulties().len());
if combined > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option 2: early individual bound check
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT / 2 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Either option closes the overflow before the combined expression is evaluated.

## Proof of Concept

```rust
// Attacker constructs:
let msg = packed::GetLastStateProof::new_builder()
    .last_hash(tip_hash)                        // any known main-chain tip
    .start_hash(genesis_hash)
    .start_number(0u64.pack())
    .last_n_blocks(0x8000_0000_0000_0001u64.pack())  // triggers overflow
    .difficulty_boundary(U256::one().pack())    // any value; never reached
    // difficulties: empty (default)
    .build();

// Guard: 0 + ((0x8000_0000_0000_0001usize).wrapping_mul(2)) = 2 > 1000 → false → passes
// Line 291: last_block_number - 0 <= 0x8000_0000_0000_0001 → true
// Line 296: (0..last_block_number).collect() → ~10M entries
// Line 359: complete_headers iterates all → ~10M DB reads
// Repeat from multiple peers → memory exhaustion + I/O saturation → node crash
```

A unit test can confirm the arithmetic: `assert_eq!((0x8000_0000_0000_0001usize).wrapping_mul(2), 2)`. A fuzz test targeting `execute()` with `last_n_blocks` values near `usize::MAX / 2` will reliably trigger the guard bypass. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
