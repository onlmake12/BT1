### Title
Entropy Provider Can Instantly Update Fee Without Timelock, Enabling Frontrunning of User Requests - (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

An Entropy provider can call `setProviderFee` at any time to instantly update their `feeInWei` to an arbitrarily high value with no cap, no timelock, and no grace period. A malicious provider can observe a user's pending `request` or `requestV2` transaction in the mempool, frontrun it with a fee increase, and cause the user's transaction to revert with `InsufficientFee`. Integrating contracts that query `getFeeV2()` off-chain and submit the fee in a separate transaction are especially vulnerable.

---

### Finding Description

`setProviderFee` in `Entropy.sol` performs an immediate, uncapped write to `provider.feeInWei`:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;   // instant, unbounded update
    ...
}
``` [1](#0-0) 

The same uncapped instant update is available via `setProviderFeeAsFeeManager` (callable by the provider's designated fee manager): [2](#0-1) 

In `requestHelper`, the fee is validated against the **current** `provider.feeInWei` at execution time:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [3](#0-2) 

`getFeeV2` reads the live `provider.feeInWei` at call time: [4](#0-3) 

There is no cap on `newFeeInWei`, no minimum notice period, and no mechanism for users to cancel pending requests or receive refunds if the fee changes between their off-chain fee query and their on-chain request submission.

The `ProviderInfo` struct confirms `feeInWei` is a plain `uint128` with no bounds: [5](#0-4) 

The interface also exposes `setProviderFee` and `setProviderFeeAsFeeManager` as public entry points with no restrictions beyond provider registration: [6](#0-5) 

---

### Impact Explanation

**Frontrunning / DoS of user requests**: A provider watching the mempool can call `setProviderFee(type(uint128).max)` immediately before a user's `request` or `requestV2` transaction lands. The user's transaction reverts with `InsufficientFee`. The user loses gas but not ETH (revert). However, integrating contracts (e.g., on-chain games, lotteries) that pre-fund themselves with ETH and call `getFeeV2()` off-chain before submitting a request can have their ETH effectively locked if the contract has no withdrawal path and the provider's fee is set to an unreachable value.

**Permanent denial of service**: Setting `feeInWei = type(uint128).max` causes `getFeeV2` to overflow when adding `pythFeeInWei`, making the provider's service permanently unusable until the fee is lowered. Any contract that hardcoded or cached the provider address is bricked.

**Selective griefing**: Because `setProviderFee` takes effect immediately, a provider can selectively deny service to specific users by timing fee spikes around their transactions.

---

### Likelihood Explanation

- Any registered Entropy provider (permissionless registration) can execute this attack.
- The default provider is the most impactful target: most integrating contracts use `getDefaultProvider()` and would be affected if the default provider is malicious or compromised.
- The Fortuna keeper service (`adjust_fee_if_necessary`) already automates fee updates based on gas prices, meaning fee changes are a normal operational event — a malicious provider can disguise an attack as routine fee adjustment. [7](#0-6) 

---

### Recommendation

1. **Enforce a timelock on fee increases**: Require that fee increases are announced at least N blocks/seconds in advance before taking effect. Fee decreases can remain immediate.
2. **Cap the maximum fee**: Introduce a protocol-level maximum `feeInWei` to prevent unbounded increases.
3. **Emit a pending-fee event**: Allow off-chain clients and integrating contracts to detect and react to upcoming fee changes before they take effect.

---

### Proof of Concept

1. Provider is registered with `feeInWei = 1000 wei`.
2. User calls `getFeeV2(provider)` off-chain → receives `1000 + pythFee`.
3. User submits `requestV2{value: 1000 + pythFee}(provider, ...)` to the mempool.
4. Provider observes the pending transaction and calls `setProviderFee(type(uint128).max)`.
5. Provider's `setProviderFee` transaction is mined first (higher gas price).
6. User's `requestV2` transaction executes: `getFeeV2` now returns an overflowed/huge value → `msg.value < requiredFee` → reverts with `InsufficientFee`.
7. User's ETH is returned (revert), but the request is never submitted. Any integrating contract that expected the request to succeed is now in an inconsistent state.

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }
    }

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;
        providerInfo.feeInWei = newFeeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            provider,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
