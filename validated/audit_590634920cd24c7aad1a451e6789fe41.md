### Title
Unbounded `while` Loop in `executeCallback` Can Run Out of Gas - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past cleared (fulfilled) requests. When many requests have been fulfilled out of order, this loop must scan through a large number of cleared sequence numbers — each requiring a storage read — and can exhaust the block gas limit, causing the entire transaction to revert.

---

### Finding Description

After crediting the provider fee and clearing the fulfilled request, `executeCallback` runs the following loop:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
// After successful callback, update firstUnfulfilledSeq if needed
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs a storage read into `_state.requests[shortKey]`: [2](#0-1) 

There is no cap on the number of iterations. The loop runs from `firstUnfulfilledSeq` up to the first still-active request. If requests are fulfilled out of order (e.g., sequence numbers 2, 3, …, N fulfilled before sequence number 1), `firstUnfulfilledSeq` remains stuck at 1. When sequence number 1 is finally fulfilled, the loop must scan through all N−1 previously cleared slots, performing N−1 storage reads.

The code itself acknowledges this with a TODO comment noting the gas concern and suggesting a doubly-linked list as a fix. The gas benchmark test also explicitly documents this worst-case: [3](#0-2) 

The `currentSequenceNumber` is a `uint64` with no practical upper bound, so the gap between `firstUnfulfilledSeq` and `currentSequenceNumber` can grow arbitrarily large over the contract's lifetime.

---

### Impact Explanation

If the `while` loop exhausts the block gas limit, the entire `executeCallback` transaction reverts. Because `clearRequest` and fee crediting occur **before** the loop: [4](#0-3) 

…a revert means:
- The request is **not** cleared (remains active, blocking that slot).
- The provider fee is **not** credited.
- The consumer's `_echoCallback` is **never** executed.

This permanently blocks callback execution for the affected sequence number, locking the requester's funds and breaking the Echo protocol's liveness guarantee.

---

### Likelihood Explanation

Out-of-order fulfillment is a normal operating condition in a decentralized keeper system — different keepers race to fulfill different requests. Each storage read in the loop costs ~100 gas (warm slot, since the 32-slot `requests` array is accessed repeatedly). With a 30M gas block limit and ~100 gas per warm read, the loop can iterate ~300,000 times before hitting the limit. However, with cold reads (~2,100 gas each), only ~14,000 iterations are possible. In practice, a sustained high-throughput usage pattern where requests are fulfilled out of order will cause `firstUnfulfilledSeq` to lag, and eventually a single `executeCallback` call will trigger a prohibitively expensive scan. The test suite explicitly benchmarks this scenario, confirming the developers are aware of the gas growth.

---

### Recommendation

Add a maximum iteration bound to the `while` loop, analogous to the fix described in the external report. For example:

```solidity
uint256 maxSteps = 100;
uint256 steps = 0;
while (
    steps < maxSteps &&
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
    steps++;
}
```

Alternatively, implement the doubly-linked list of active requests suggested in the existing TODO comment, which would eliminate the linear scan entirely.

---

### Proof of Concept

1. Deploy `Echo` with a registered provider.
2. Call `requestPriceUpdatesWithCallback` N times (e.g., N = 10,000), obtaining sequence numbers 1 through N.
3. Call `executeCallback` for sequence numbers 2 through N (in any order), fulfilling all but the first request. `firstUnfulfilledSeq` remains at 1.
4. Call `executeCallback` for sequence number 1.
5. The `while` loop now iterates from sequence number 1 through N−1, performing a `findRequest` storage read at each step.
6. With N large enough, the loop exhausts the block gas limit; the transaction reverts; the callback for sequence number 1 is never delivered and the request remains permanently stuck.

The existing benchmark test at `EchoGasBenchmark.t.sol:130` demonstrates the gas growth with only 10 requests in reverse order — confirming the pattern scales to a DoS at larger N. [5](#0-4)
