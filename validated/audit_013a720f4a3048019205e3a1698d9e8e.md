### Title
Unauthenticated RPC Callers Can Disable All P2P Networking or Wipe the Transaction Pool - (`rpc/src/module/net.rs`, `rpc/src/module/pool.rs`)

### Summary
The CKB JSON-RPC server exposes several destructive, node-wide administrative operations — `set_network_active`, `clear_tx_pool`, `clear_tx_verify_queue`, and `clear_banned_addresses` — with zero authentication or authorization enforcement. Any caller who can reach the RPC port can invoke these methods. The RPC server implements no token, credential, or caller-identity check at any layer; the only protection is the default `127.0.0.1` bind address, which is a network-level configuration advisory, not a code-level access control. This is a direct analog to the reported `destroyInstance()` pattern: privileged, destructive operations are callable by any unprivileged party.

### Finding Description

**Root cause — no authentication layer in the RPC server:**

`rpc/src/server.rs` builds the axum HTTP router with `CorsLayer::permissive()` and no authentication middleware:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())   // wide-open CORS, no auth
    ...
``` [1](#0-0) 

Every incoming JSON-RPC call is dispatched directly to `io.handle_call(call, T::default())` with a default (empty) metadata value — no caller identity is threaded through: [2](#0-1) 

**Destructive RPC methods with no access control:**

`set_network_active(false)` — disables all P2P message processing for the node. The implementation is a single atomic store with no caller check:

```rust
fn set_network_active(&self, state: bool) -> Result<()> {
    self.network_controller.set_active(state);
    Ok(())
}
``` [3](#0-2) 

`clear_tx_pool()` — atomically wipes every pending and proposed transaction from the mempool:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
``` [4](#0-3) 

`clear_tx_verify_queue()` — drops all transactions currently being verified: [5](#0-4) 

`clear_banned_addresses()` — erases the entire peer ban list, immediately re-admitting all previously banned malicious peers: [6](#0-5) 

All four are registered as first-class methods in the `Net` and `Pool` modules, which are enabled by default in the production configuration: [7](#0-6) 

**`set_network_active` is the highest-severity analog.** The underlying `NetworkController::set_active` writes a single atomic boolean that causes the entire P2P layer to discard every received message: [8](#0-7) 

### Impact Explanation

- **`set_network_active(false)`**: The node immediately stops processing all inbound P2P messages — block announcements, transaction relays, sync requests, light-client proofs. The node is silently isolated from the network. It continues to appear running but falls behind the chain tip indefinitely. Miners using this node receive stale block templates and waste hashpower. Recovery requires a privileged operator to call `set_network_active(true)` or restart the node.

- **`clear_tx_pool`**: All pending user transactions are silently dropped. Users whose transactions were in the pool must resubmit. Miners lose all queued fee revenue. Repeated calls prevent any transaction from ever being mined on this node.

- **`clear_banned_addresses`**: Removes all IP bans, immediately re-admitting peers that were banned for malicious behavior (e.g., sending invalid blocks or eclipse-attack probes).

- **`clear_tx_verify_queue`**: Drops all transactions currently under verification, causing silent loss of in-flight submissions.

### Likelihood Explanation

The RPC binds to `127.0.0.1:8114` by default, which limits exposure to processes on the same host. However:

1. The prompt explicitly lists "RPC caller" and "supported local CLI/RPC user" as valid unprivileged attacker profiles.
2. Any co-located process (another container, a compromised dependency, a malicious dApp running locally) can reach `127.0.0.1:8114` without any credential.
3. Operators frequently expose the RPC to non-localhost addresses (e.g., for remote miner access), as the configuration explicitly supports this. The README warning is advisory only — there is no code-level enforcement.
4. No token, API key, or HTTP Basic Auth is implemented anywhere in the server stack. The `parse_authorization` helper exists only in the *miner client* (outbound requests), not in the server. [9](#0-8) 

### Recommendation

Implement a mandatory authentication layer in the RPC server. Options include:

1. **Bearer token / HTTP Basic Auth**: Generate a random secret at node startup; require it in every request via the `Authorization` header. Validate it in an axum middleware before dispatching to any handler.
2. **Per-method privilege tiers**: Separate read-only methods (e.g., `get_block`) from destructive admin methods (e.g., `set_network_active`, `clear_tx_pool`) and require explicit opt-in or a separate authenticated endpoint for the latter.
3. **Allowlist by caller IP**: Enforce at the server layer (not just via OS firewall) that destructive methods are only callable from a configured set of trusted addresses.

### Proof of Concept

With a default CKB node running (RPC on `127.0.0.1:8114`, `Net` and `Pool` modules enabled):

```bash
# Disable all P2P networking — no credentials required
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":1}'
# Response: {"jsonrpc":"2.0","result":null,"id":1}
# Node is now isolated from the entire P2P network.

# Wipe the entire transaction pool — no credentials required
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":2}'
# Response: {"jsonrpc":"2.0","result":null,"id":2}
# All pending transactions are gone.
```

Both calls succeed immediately with no authentication, no confirmation, and no rate limiting. The attacker needs only TCP access to port 8114.
