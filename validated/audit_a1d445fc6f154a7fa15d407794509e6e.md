Audit Report

## Title
Fee Rate Minimum Check Uses Serialized Size Instead of Transaction Weight, Allowing High-Cycle Transactions to Bypass `min_fee_rate` Policy - (File: tx-pool/src/util.rs)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using only the raw serialized transaction size (`tx_size`) as the weight argument to `FeeRate::fee()`. Because `FeeRate` is defined as shannons per kilo-weight and the true weight is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`, any transaction where cycles dominate passes the fee check while paying far less than the configured minimum. No second weight-based fee check is performed after `verify_rtx` determines actual cycles, leaving the policy permanently unenforced for high-cycle transactions.

## Finding Description

`FeeRate` is defined as shannons per kilo-weight: [1](#0-0) 

`FeeRate::fee()` takes a `weight` argument: [2](#0-1) 

The correct transaction weight accounts for both size and cycles: [3](#0-2) 

But `check_tx_fee` passes only `tx_size` — not the actual weight — to `FeeRate::fee()`. The code comment explicitly acknowledges this is an approximation ("cheap check"): [4](#0-3) 

`_process_tx` calls `pre_check` (which calls `check_tx_fee`) before script execution, then calls `verify_rtx` to obtain actual cycles. After `verify_rtx` returns `verified.cycles` at line 734, the code at line 751 directly constructs a `TxEntry` and proceeds to `submit_entry` with no weight-based fee validation: [5](#0-4) 

A search across the codebase confirms `get_transaction_weight` is never called in `process.rs`, and `min_fee_rate` has no references in `process.rs` at all — the only fee rate enforcement is the pre-execution size-only check in `util.rs`. [6](#0-5) 

## Impact Explanation

This maps to **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

With default config (`min_fee_rate = 1000` shannons/KW, `max_tx_verify_cycles = 70,000,000`):

- A transaction with ~300 bytes serialized size and ~70,000,000 cycles has actual weight = `max(300, 70_000_000 × 0.000_170_571_4)` ≈ **11,940**.
- `check_tx_fee` requires only: `1000 × 300 / 1000 = 300 shannons`.
- Correct minimum should be: `1000 × 11,940 / 1000 = 11,940 shannons`.
- Attacker pays ~49× less than the effective minimum fee rate.

An attacker can flood the tx-pool with high-cycle, small-size transactions at a fraction of the intended cost, consuming CPU during script verification and pool memory. Repeated at scale, this degrades node performance and can cause network-wide congestion.

## Likelihood Explanation

Any unprivileged RPC caller with access to `send_transaction` can trigger this. Crafting a valid transaction with high cycles is straightforward: a lock script executing a tight RISC-V loop can consume up to `max_tx_verify_cycles` cycles while having a small serialized size. No privileged access, leaked keys, or majority hashpower is required. The attack is repeatable and cheap relative to the resources consumed.

## Recommendation

After `verify_rtx` returns actual cycles in `_process_tx`, perform a second fee rate check using the true transaction weight:

```rust
// After verify_rtx returns verified.cycles (process.rs ~line 734):
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

The pre-check in `check_tx_fee` can remain as a cheap early rejection using size only, but a definitive weight-based check must follow script execution once actual cycles are known. [4](#0-3) 

## Proof of Concept

1. Craft a CKB transaction with a lock script executing a tight loop consuming ~70,000,000 cycles. Serialized transaction size: ~300 bytes.
2. Set transaction fee to exactly `1000 × 300 / 1000 = 300 shannons`.
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` passes: `300 >= 300`. ✓ [7](#0-6) 
5. `verify_rtx` executes the script, consuming 70,000,000 cycles. [8](#0-7) 
6. No second fee check is performed; `TxEntry` is created and submitted. [6](#0-5) 
7. Actual weight = 11,940. Effective fee rate = `300 × 1000 / 11,940 ≈ 25 shannons/KW` — ~49× below the configured 1,000 shannons/KW minimum.
8. Repeat to flood the pool with high-cycle transactions at a fraction of the intended cost.
