### Title
`MAX_DEPOSIT_LIMIT` Bypass via Repeated `addFunds` Calls on Permanent Subscriptions — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract enforces a `MAX_DEPOSIT_LIMIT` (100 ETH) on permanent subscriptions to cap how much ETH can be permanently locked. However, the check in `addFunds` compares only the **per-transaction** `msg.value` against the limit, not the **cumulative** `status.balanceInWei`. An unprivileged user can call `addFunds` repeatedly with amounts ≤ `MAX_DEPOSIT_LIMIT`, accumulating a balance far exceeding the cap. Since permanent subscriptions prohibit withdrawals, this ETH is locked forever beyond the intended invariant.

---

### Finding Description

In `Scheduler.sol`, `MAX_DEPOSIT_LIMIT = 100 ether` is defined as the ceiling for permanent subscription balances:

```solidity
// SchedulerConstants.sol
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
```

Both `createSubscription` and `addFunds` enforce this limit with an identical check:

```solidity
// createSubscription (line 48)
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// addFunds (line 613)
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

The check is `msg.value > MAX_DEPOSIT_LIMIT` — it only validates the **incoming transaction value**, not the resulting cumulative balance. After the check, the balance is unconditionally incremented:

```solidity
status.balanceInWei += msg.value;  // line 617
```

There is no guard of the form `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`. An attacker can call `addFunds` N times with `msg.value = MAX_DEPOSIT_LIMIT` (or any amount ≤ it), and each call passes the check independently while the stored balance grows without bound.

This is structurally identical to the reference bug: the limit check operates on a **different scope** (per-call `msg.value`) than the value being limited (cumulative `balanceInWei`), so the invariant is trivially bypassed.

---

### Impact Explanation

- **Invariant broken:** The protocol explicitly intends permanent subscriptions to be capped at 100 ETH (`MAX_DEPOSIT_LIMIT`). This invariant is violated.
- **Permanent fund lock:** Permanent subscriptions block all withdrawals (`withdrawFunds` reverts with `CannotUpdatePermanentSubscription`). Any ETH deposited beyond the cap is irrecoverably locked in the contract.
- **No recovery path:** Because `isPermanent` cannot be unset and withdrawals are blocked, the excess ETH cannot be retrieved.

**Impact: Medium** — protocol invariant broken; ETH permanently locked beyond the intended cap.

---

### Likelihood Explanation

**Likelihood: High** — `addFunds` has no access control (anyone can call it for any active subscription). The bypass requires only repeated calls with standard amounts. No special privileges, leaked keys, or external oracle manipulation are needed.

---

### Recommendation

Change both the `createSubscription` and `addFunds` checks to compare the **post-deposit cumulative balance** against `MAX_DEPOSIT_LIMIT`:

```solidity
// In addFunds:
if (params.isPermanent && (status.balanceInWei + msg.value) > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

// In createSubscription:
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    // This one is correct at creation since balanceInWei starts at 0,
    // but for consistency use: msg.value > MAX_DEPOSIT_LIMIT (already correct here)
}
```

The `createSubscription` check is correct as-is (initial balance is `msg.value`). The critical fix is in `addFunds`.

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription at the limit
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);
// status.balanceInWei == 100 ether ✓ (at limit)

// 2. Call addFunds repeatedly — each call passes msg.value > MAX_DEPOSIT_LIMIT check
scheduler.addFunds{value: 100 ether}(subId);
// status.balanceInWei == 200 ether — limit bypassed

scheduler.addFunds{value: 100 ether}(subId);
// status.balanceInWei == 300 ether — limit bypassed further

// 3. Withdrawal is permanently blocked
scheduler.withdrawFunds(subId, 1 ether);
// reverts: CannotUpdatePermanentSubscription
// All 300 ETH is permanently locked
```
