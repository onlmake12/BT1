### Title
`RecentReject::put()` Never Increments `total_keys_num`, Permanently Disabling the Shrink Bound — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject::put()` computes a local `total_keys_num = self.total_keys_num + 1` but **never writes it back** to `self.total_keys_num`. Because the in-memory counter never advances, `shrink()` is never triggered on a fresh or sub-limit database. An unprivileged remote relay peer can write an unbounded number of entries into the RocksDB-backed recent-reject store, limited only by the TTL window, causing disk exhaustion and rendering the `keep_rejected_tx_hashes_count` configuration meaningless.

---

### Finding Description

**Root cause — missing counter update in `put()`** [1](#0-0) 

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    ...
    self.db.put(&shard, hash_slice, json_string)?;   // written to RocksDB

    if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
        if total_keys_num > self.count_limit {
            self.shrink()?;                          // only path that updates self.total_keys_num
        }
        // ← self.total_keys_num is NEVER assigned here
    } else {
        self.shrink()?;
    }
    Ok(())
}
```

The local binding `total_keys_num` is evaluated but never stored back. `self.total_keys_num` stays at whatever `estimate_total_keys_num()` returned at startup (0 for a fresh DB). Every subsequent call computes `0 + 1 = 1`, which is never `> count_limit` (default 10 000), so `shrink()` is never invoked. The only way `shrink()` fires is if the DB already held ≥ `count_limit` entries at node startup.

**`should_recorded()` is true for all non-Duplicated variants** [2](#0-1) 

**`TransactionError::Immature` is explicitly NOT malformed → no peer ban** [3](#0-2) 

So `Reject::Verification(TransactionError::Immature)` satisfies:
- `should_recorded()` → `true` → written to recent-reject DB
- `is_malformed_tx()` → `false` → peer is **not** banned
- `is_allowed_relay()` → `true` → peer may keep relaying

**Call chain from remote peer to `put()`** [4](#0-3) 

Remote relay path: `TransactionsProcess::execute()` → `tx_pool.submit_remote_tx()` → `after_process()` → `put_recent_reject()` → `RecentReject::put()`. [5](#0-4) 

**Relay-layer rate limiting exists but does not prevent the attack**

The relayer applies 30 req/s per peer per message type. [6](#0-5) 

Each `RelayTransactions` message may carry multiple transactions. With `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` hashes per batch and a 1-day TTL window, a single peer can inject tens of thousands of unique entries before any TTL expiry occurs. Multiple coordinated peers multiply the rate linearly.

**`shrink()` itself is correct but unreachable** [7](#0-6) 

`shrink()` drops a random shard and re-estimates `total_keys_num` from RocksDB. It is correct in isolation but is never reached because the guard condition `total_keys_num > count_limit` is evaluated against a counter that is permanently stale.

---

### Impact Explanation

- **Disk exhaustion**: The recent-reject RocksDB grows without bound within the TTL window. The operator-configured `keep_rejected_tx_hashes_count` limit is silently ignored.
- **Configuration invariant broken**: Operators who set `keep_rejected_tx_hashes_count = 10000` receive no protection; the actual DB size is bounded only by TTL × write rate.
- **Secondary (re-relay)**: Because `shrink()` never fires, previously recorded hashes are *not* lost early — so the re-relay concern described in the question does not materialise through this specific bug. The primary impact is disk exhaustion.

---

### Likelihood Explanation

- No special privileges required: any peer that can relay transactions can trigger this.
- No peer banning for non-malformed rejects (`Immature`, `Expiry`, `LowFeeRate`, `Resolve(Dead)`, `RBFRejected`, `Invalidated`).
- The relay rate limiter (30 req/s) slows but does not stop the attack; multiple peers or a sustained single peer over hours can exhaust disk.
- The bug is deterministic and reproducible locally with a single peer.

---

### Recommendation

In `RecentReject::put()`, assign the incremented value back before the shrink check:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;   // ← add this line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

`shrink()` already re-estimates `self.total_keys_num` from RocksDB after dropping a shard, so no double-update occurs.

### Proof of Concept

```rust
// tx-pool/src/component/tests/recent_reject.rs (extend existing test)
let shard_num = 2;
let limit = 100u64;
let ttl = -1; // no TTL expiry in test
let mut rr = RecentReject::build(tmp_dir.path(), shard_num, limit, ttl).unwrap();

// Insert 10× count_limit unique entries with a non-Duplicated reject
for i in 0..1000u64 {
    let key = Byte32::new(blake2b_256(i.to_le_bytes()));
    rr.put(&key, Reject::Expiry(i)).unwrap();
}

// Bug: total_keys_num is still 0 (or startup estimate), shrink never fired
assert_eq!(rr.total_keys_num, 0);  // demonstrates the stale counter

// Actual DB has 1000 entries, far above count_limit=100
let actual = rr.estimate_total_keys_num_pub(); // expose via test helper
assert!(actual > limit, "DB grew to {} entries, limit={}", actual, limit);
```
