### Title
Unbounded `while` Loop in `Echo::executeCallback` Enables Gas-Exhaustion DoS on Request Fulfillment — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol::executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past all cleared requests. An unprivileged attacker can create many cheap requests and fulfill them out of order, inflating the gap that the loop must scan when a victim's request is later fulfilled. If the gap is large enough, the loop exceeds the block gas limit, permanently preventing the victim's request from ever being fulfilled and locking their fee payment in the contract.

---

### Finding Description

After clearing the current request, `executeCallback` runs the following loop before invoking the consumer callback:

```solidity
// Echo.sol lines 169-174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs at least one cold `SLOAD` from `_state.requests[shortKey]` and, for overflow entries, a second cold `SLOAD` from `_state.requestsOverflow[key]`, plus a warm `SSTORE` to increment `firstUnfulfilledSeq`. [2](#0-1) 

The number of iterations equals the gap between `firstUnfulfilledSeq` and the next still-active request. An attacker controls this gap:

1. A legitimate requester creates request **N** (`firstUnfulfilledSeq = N`).
2. The attacker creates requests **N+1 … N+K** (each with `publishTime ≤ block.timestamp + 60`, paying the required fee).
3. The attacker fulfills requests **N+1 … N+K** with valid price data. Because request **N** is still active, `firstUnfulfilledSeq` never advances past **N**.
4. When the provider tries to fulfill request **N**, the `while` loop must iterate **K** times before finding the next active request (or reaching `currentSequenceNumber`).

The critical ordering is: `clearRequest(sequenceNumber)` executes at line 164 **before** the `while` loop at line 169, and the consumer callback executes **after** the loop at line 176. [3](#0-2) 

If the loop causes an out-of-gas revert, the entire transaction reverts — including the `clearRequest` — so request **N** remains permanently active and unfulfillable. The developers themselves flagged this in a TODO comment:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage
// lookup for each sequence number. a better solution would be a doubly-linked list of
// active requests.
``` [4](#0-3) 

There is no cap on the total number of outstanding requests; `currentSequenceNumber` is a `uint64` and `requestPriceUpdatesWithCallback` imposes no global limit. [5](#0-4) 

---

### Impact Explanation

- The victim's request is permanently unfulfillable once the loop gap exceeds the block gas limit (~30 M gas on Ethereum/Base/Optimism).
- At roughly 5 000–7 000 gas per loop iteration (cold `SLOAD` + warm `SSTORE`), a gap of ~4 000–6 000 cleared requests is sufficient to exhaust the block gas limit, even accounting for the other work done in `executeCallback` (price-feed parsing, fee accounting, etc.).
- The victim's fee (`req.fee`) is credited to the provider only inside `executeCallback`; if the call always reverts, those funds are locked in the contract with no recovery path.

    function findRequest(
        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];
        }
