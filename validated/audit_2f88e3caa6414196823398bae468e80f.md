Audit Report

## Title
Premature ~8MB Allocation Before Decompression Validation Enables Per-Frame Memory Amplification DoS — (`network/src/compress.rs`)

## Summary
In `LengthDelimitedCodecWithCompress::decode`, a `BytesMut::zeroed(decompressed_bytes_len)` allocation of up to ~8MB is made based solely on an attacker-controlled snappy varint header, before actual decompression is attempted. A crafted ~5-byte compressed frame causes an ~8MB allocation that is immediately freed when decompression fails on the garbage body. No ban or rate-limit is applied at the codec layer, so the attacker reconnects and repeats indefinitely.

## Finding Description
`MAX_UNCOMPRESSED_LEN` is defined as `1 << 23 = 8,388,608` at line 13 of `network/src/compress.rs`. The guard at line 235 rejects only if `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` (strictly greater than), so a varint-declared value of exactly 8,388,607 passes the check.

At line 242, `BytesMut::zeroed(decompressed_bytes_len)` allocates up to ~8MB of zeroed memory based solely on the attacker-supplied varint. If `SnapDecoder::new().decompress(...)` then fails on the garbage body (lines 245–248), the buffer is dropped and an `Err` is returned — but the allocation has already occurred.

The attacker's frame is tiny: 1 flag byte (`0x80`) + 4-byte snappy varint encoding 8,388,607 (`\xff\xff\xff\x03`) + arbitrary garbage body. This is far below every protocol's `max_frame_length` (Sync: 2MB, RelayV3: 4MB), so `length_delimited.decode` at line 226 passes the frame through without rejection.

The vulnerable codec is wired into every CKB protocol via `CKBProtocol::new_with_support_protocol` and `CKBProtocol::new` with `compress: true` by default (lines 219 and 243 of `network/src/protocols/mod.rs`), and is installed via `CKBProtocol::build` at lines 280–288.

Because the error is returned from the `Decoder` trait implementation (codec layer), it never reaches the application-layer `CKBHandler::received`, so `ban_peer` (lines 684–687 of `mod.rs`) is never invoked. The attacker's connection is closed, but no ban entry is recorded in the peer store. The attacker reconnects with a fresh identity and repeats.

## Impact Explanation
Each malicious frame causes an ~8MB allocation + immediate free (~1,600,000× amplification over the ~5-byte wire cost). On a remote network (50ms RTT), the tentacle noise-XX handshake overhead limits the attack to roughly 5 reconnects/second, yielding ~40 MB/s of large-object allocator churn. On a low-latency path (1ms RTT), this reaches ~250 reconnects/second and ~2 GB/s of churn. Sustained at these rates, the allocator fragmentation pressure degrades p99 message-processing latency across all active connections and, under sufficient load, can exhaust available memory and crash the node. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The snappy varint format is publicly documented. Any unprivileged peer can connect to a CKB node's P2P port (no authentication required) and send this frame after completing the tentacle handshake. No special tooling is needed — a raw TCP socket with a noise-XX implementation suffices. The default `max_peers = 125` does not prevent serial reconnection attacks; each closed connection frees the slot immediately.

## Recommendation
Move the allocation **after** a validation step, or avoid pre-allocating based on the header-declared size:

1. **Decompress into a fixed-size scratch buffer** (sized to `MAX_UNCOMPRESSED_LEN`) and reject if the output exceeds the limit, never trusting the varint for allocation sizing.
2. **Validate the snappy stream structure** (e.g., check chunk headers against the actual compressed payload length) before allocating, so corrupt frames are rejected with zero heap cost.

Additionally, apply a ban/score penalty at the codec error boundary by returning a sentinel error type that the session manager maps to a peer ban, so repeat offenders are blocked at the connection level before the application layer is reached.

## Proof of Concept

```python
# Craft the malicious frame
flag = b'\x80'                          # COMPRESS_FLAG
varint = b'\xff\xff\xff\x03'            # snappy varint for 8_388_607
garbage = b'\xde\xad\xbe\xef' * 4      # corrupt body — decompress will fail
payload = flag + varint + garbage

# Length-delimited framing (4-byte big-endian length prefix)
import struct
frame = struct.pack('>I', len(payload)) + payload

# After completing the tentacle noise-XX handshake with any CKB P2P port,
# send `frame` on any protocol substream (e.g., Sync or RelayV3).
# Observe: ~8MB allocation + free per frame, connection reset.
# Reconnect and repeat; measure allocator pressure and p99 latency degradation.
```
