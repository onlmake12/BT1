### Title
Missing `BLOCK_INVALID` Status Check in `HeaderAcceptor::accept()` Allows Repeated Re-processing of Rejected Headers — (File: `sync/src/synchronizer/headers_process.rs`)

### Summary

`HeaderAcceptor::accept()` checks whether a header's **parent** is `BLOCK_INVALID` but never checks whether the **header itself** is already `BLOCK_INVALID`. A developer-acknowledged `FIXME` comment at the exact location confirms this omission. An unprivileged P2P peer can exploit this by repeatedly re-submitting a previously-rejected header, causing the node to re-accept it as `HEADER_VALID`, re-request the corresponding block, re-verify it, mark it invalid again, and loop — a resource-exhaustion denial of service.

---

### Finding Description

`HeaderAcceptor::accept()` in `sync/src/synchronizer/headers_process.rs` performs the following status checks in order:

1. If the header's own status contains `HEADER_VALID` → return early (already accepted).
2. `prev_block_check` — if the **parent's** status is `BLOCK_INVALID` → mark this header invalid and return.
3. `non_contextual_check` — PoW, epoch, etc.
4. `version_check` — header version.

The critical gap is between steps 1 and 2. The code explicitly reads the header's own status but only acts on `HEADER_VALID`. The `BLOCK_INVALID` case is left unhandled, with a developer-written `FIXME` comment acknowledging the omission:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ...
    return result;
}
``` [1](#0-0) 

After a block fails full verification in `ConsumeUnverifiedBlockProcessor::consume_unverified_blocks`, the block's hash is marked `BLOCK_INVALID`:

```rust
self.shared.insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
``` [2](#0-1) 

At this point the header's status in the shared map is `BLOCK_INVALID`. If a peer now re-sends the same header via a `SendHeaders` message, `accept()` reads the status, sees it is not `HEADER_VALID`, skips the early-return, and falls through to `prev_block_check`. Because the header itself (not its parent) was the one marked invalid, `prev_block_check` passes. `non_contextual_check` and `version_check` also pass (the header bytes are structurally valid). The function then calls `sync_shared.insert_valid_header(self.peer, self.header)`, overwriting the `BLOCK_INVALID` status with `HEADER_VALID`. [3](#0-2) 

`BlockFetcher::fetch()` iterates candidate headers and adds them to the download queue if their status is neither `BLOCK_STORED` nor `BLOCK_RECEIVED`. It has **no check for `BLOCK_INVALID`**:

```rust
if status.contains(BlockStatus::BLOCK_STORED) {
    // ...
    break;
} else if status.contains(BlockStatus::BLOCK_RECEIVED) {
    // Do not download repeatedly
} else if ... && state.write_inflight_blocks().insert(...) {
    fetch.push(header)   // ← BLOCK_INVALID headers reach here
}
``` [4](#0-3) 

The `BlockStatus` flags confirm `BLOCK_INVALID` and `HEADER_VALID` are orthogonal bits, so neither check subsumes the other: [5](#0-4) 

---

### Impact Explanation

A malicious peer can drive the following loop indefinitely:

1. Peer sends a `SendHeaders` message containing header `H` whose block body will fail contextual verification (e.g., invalid script execution, capacity overflow, bad uncle).
2. Node accepts `H` as `HEADER_VALID` (header PoW is valid).
3. `BlockFetcher` schedules a `GetBlocks` request for `H`.
4. Peer delivers block `H`; `ConsumeUnverifiedBlockProcessor` runs full verification, which fails; `H` is marked `BLOCK_INVALID`.
5. Peer re-sends header `H` via `SendHeaders`.
6. `accept()` does not check `BLOCK_INVALID` → re-accepts `H` as `HEADER_VALID` (FIXME path).
7. Go to step 3.

Each iteration costs the victim node: one full block download (up to the consensus block size limit), one full script-execution verification pass (up to the cycle limit), and associated I/O. The attacker pays only the cost of sending a small header message to restart the cycle.

---

### Likelihood Explanation

- The entry path is a standard, unauthenticated P2P `SendHeaders` message — reachable by any peer.
- The attacker only needs to produce a block whose **header** passes PoW but whose **body** fails contextual verification. This is achievable without majority hash power (e.g., a block with a script that always aborts, or a capacity-overflow output).
- The FIXME comment confirms the developers are aware the check is absent; no special race condition or configuration is required.
- The `BlockFetcher`'s independent omission of a `BLOCK_INVALID` guard compounds the issue.

---

### Recommendation

**Short term:** At the top of `HeaderAcceptor::accept()`, add an explicit early-return for `BLOCK_INVALID`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) { ... }
```

**Short term (defense-in-depth):** Add a `BLOCK_INVALID` guard in `BlockFetcher::fetch()` alongside the existing `BLOCK_STORED` / `BLOCK_RECEIVED` guards so that even if the header is re-accepted, the block is not re-downloaded.

**Long term:** Resolve the `FIXME` by introducing a dedicated `ValidationError` variant (e.g., `PreviouslyRejected`) so the error can be surfaced to the caller and the peer can be penalised.

---

### Proof of Concept

1. Connect a malicious peer to a CKB node.
2. Craft block `B`: valid header (correct PoW, valid parent, version 0), but body contains a transaction whose lock script always exits with code 1.
3. Send `SendHeaders([header_of_B])` → node accepts, requests block.
4. Send `SendBlock(B)` → node verifies, script fails, marks `B` as `BLOCK_INVALID`.
5. Immediately send `SendHeaders([header_of_B])` again.
6. Observe (via debug logs or metrics) that the node re-accepts the header and issues another `GetBlocks` request.
7. Repeat steps 4–6 in a tight loop; monitor the victim's CPU and bandwidth consumption growing proportionally.

The root cause — the unguarded fall-through past `BLOCK_INVALID` — is confirmed by the `FIXME` comment at `sync/src/synchronizer/headers_process.rs` lines 301–302. [6](#0-5)

    pub fn accept(&self) -> ValidationResult {
        let mut result = ValidationResult::default();
        let sync_shared = self.active_chain.sync_shared();
        let state = self.active_chain.state();
        let shared = sync_shared.shared();

        // FIXME If status == BLOCK_INVALID then return early. But which error
        // type should we return?
        let status = self.active_chain.get_block_status(&self.header.hash());
        if status.contains(BlockStatus::HEADER_VALID) {
            let header_index = sync_shared
                .get_header_index_view(
                    &self.header.hash(),
                    status.contains(BlockStatus::BLOCK_STORED),
                )
                .unwrap_or_else(|| {
                    panic!(
                        "header {}-{} with HEADER_VALID should exist",
                        self.header.number(),
                        self.header.hash()
                    )
                })
                .as_header_index();
            state
                .peers()
                .may_set_best_known_header(self.peer, header_index);
            return result;
        }

        if self.prev_block_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-parent header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        if let Some(is_invalid) = self.non_contextual_check(&mut result).err() {
            debug!(
                "HeadersProcess rejected non-contextual header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            if is_invalid {
                shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            }
            return result;
        }

        if self.version_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-version header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        sync_shared.insert_valid_header(self.peer, self.header);
        result
    }

    pub struct BlockStatus: u32 {
        const UNKNOWN                 =     0;

        const HEADER_VALID            =     1;
        const BLOCK_RECEIVED          =     1 | (Self::HEADER_VALID.bits() << 1);
        const BLOCK_STORED            =     1 | (Self::BLOCK_RECEIVED.bits() << 1);
