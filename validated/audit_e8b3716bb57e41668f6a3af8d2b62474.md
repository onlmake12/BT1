### Title
Missing `MAX_RELAY_TXS_NUM_PER_BATCH` Count Guard in `TransactionsProcess::execute` Enables Per-Message CPU/Memory Amplification — (`sync/src/relayer/transactions_process.rs`)

### Summary

`TransactionsProcess::execute` deserializes every transaction in an incoming `RelayTransactions` P2P message into a full `TransactionView` object before applying any filter. Unlike the two sibling handlers that enforce `MAX_RELAY_TXS_NUM_PER_BATCH`, this handler has no count check, so a single oversized message forces unbounded deserialization work and holds two shared-state mutexes for the entire loop.

---

### Finding Description

**The missing guard.** Both peer-facing handlers that deal with transaction counts enforce the constant `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`:

- `TransactionHashesProcess::execute` rejects immediately if `tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH`. [1](#0-0) 

- `GetTransactionsProcess::execute` does the same. [2](#0-1) 

`TransactionsProcess::execute` has **no such check**. The entire body proceeds directly to deserialization: [3](#0-2) 

**Deserialization precedes filtering.** The Rust iterator chain calls `.map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))` on every element before `.filter(...)` is evaluated. `to_entity().into_view()` performs full molecule deserialization and hash computation for every transaction in the message, regardless of whether it will be discarded by the filter. Both `tx_filter` and `unknown_tx_hashes` mutexes are acquired and held for the entire iteration. [4](#0-3) 

**The only upstream gate is `check_data()`**, which validates molecule structural well-formedness but does not limit the count of items in a vector. [5](#0-4) 

**Partial mitigation — rate limiter.** A rate limiter keyed by `(PeerIndex, message_item_id)` caps at 30 messages per second per peer per message type. [6](#0-5) 

This limits the *number of messages* but not the *number of transactions per message*. An attacker can pack the maximum number of transactions into each of the 30 allowed messages per second.

**`MAX_RELAY_TXS_BYTES_PER_BATCH` is send-side only.** The 1 MiB byte budget is enforced only when the node *sends* `RelayTransactions`; it is never checked on receipt. [7](#0-6) 

---

### Impact Explanation

An attacker who is a connected peer can send `RelayTransactions` messages containing far more than 32,767 transactions (up to whatever the network transport's frame limit allows). For each message, the victim node:

1. Deserializes every transaction into a `TransactionView` (CPU + heap allocation per tx)
2. Holds `tx_filter` and `unknown_tx_hashes` mutexes for the full loop, blocking other relay threads that need those locks

At 30 messages/second (the rate-limiter ceiling), with each message carrying the maximum feasible transaction count, this produces sustained CPU and memory pressure disproportionate to the attacker's bandwidth cost. The impact is degraded transaction relay throughput on the victim node — matching the stated scope.

---

### Likelihood Explanation

The precondition is minimal: any connected peer can send `RelayTransactions`. The filter condition (requiring `requesting_peer == self.peer`) only determines whether transactions reach the tx pool; it does not prevent deserialization. The attacker does not need any prior handshake or privileged role. The asymmetry with sibling handlers is a straightforward oversight.

---

### Recommendation

Add the same count guard used by the sibling handlers at the top of `TransactionsProcess::execute`:

```rust
pub fn execute(self) -> Status {
    if self.message.transactions().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
        return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
            "Transactions count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
            self.message.transactions().len(),
            MAX_RELAY_TXS_NUM_PER_BATCH,
        ));
    }
    // ... rest of execute
}
```

Optionally, also add a total-bytes check against `MAX_RELAY_TXS_BYTES_PER_BATCH` on the incoming message, mirroring the send-side enforcement.

---

### Proof of Concept

1. Connect to a victim CKB node as a normal peer.
2. Craft a `RelayTransactions` molecule message containing N transactions (e.g., N = 65,535 minimal-size transactions with random hashes).
3. Send the message. No prior `RelayTransactionHashes` exchange is needed — the filter will discard all transactions, but deserialization of all N entries still occurs.
4. Repeat at 30 messages/second (the rate-limiter ceiling).
5. Measure victim node CPU usage and `tx_filter`/`unknown_tx_hashes` lock contention; assert linear growth with N and sustained elevation at 30 msg/s.
6. Compare with sending the same total transaction count split across `RelayTransactionHashes` messages — those are rejected immediately at the count guard, producing negligible CPU cost.

    pub fn execute(self) -> Status {
        let shared_state = self.relayer.shared().state();
        let txs: Vec<(TransactionView, Cycle)> = {
            // ignore the tx if it's already known or it has never been requested before
            let mut tx_filter = shared_state.tx_filter();
            tx_filter.remove_expired();
            let unknown_tx_hashes = shared_state.unknown_tx_hashes();

            self.message
                .transactions()
                .iter()
                .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))
                .filter(|(tx, _)| {
                    !tx_filter.contains(&tx.hash())
                        && unknown_tx_hashes
                            .get_priority(&tx.hash())
                            .map(|priority| priority.requesting_peer() == Some(self.peer))
                            .unwrap_or_default()
                })
                .collect()
        };
