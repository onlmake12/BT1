### Title
Unbounded `numHashes` Growth via Unfulfilled Entropy Requests Enables Provider DoS — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

An unprivileged attacker can flood an Entropy provider with paid `requestWithCallback` / `requestV2` calls and never trigger reveals. Because `numHashes` for each new request equals `assignedSequenceNumber − currentCommitmentSequenceNumber`, and because the guard on `maxNumHashes` is disabled when the field is zero (the default), the hash-chain walk in `constructProviderCommitment` grows linearly with the number of outstanding unfulfilled requests. Once the walk exceeds the block gas limit, the provider can no longer execute any `revealWithCallback` / `reveal` transaction, permanently halting randomness fulfillment.

---

### Finding Description

**Root cause — unbounded `numHashes` with no default cap**

In `requestHelper`, every new request records how far the provider must walk the hash chain to verify their contribution:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [1](#0-0) 

The guard fires only when `maxNumHashes != 0`. Because `maxNumHashes` is never set during `register`, it defaults to `0` in the `ProviderInfo` struct, so the branch is **never taken** for a freshly registered provider. [2](#0-1) 

**Expensive path — O(numHashes) keccak256 loop**

When the provider calls `revealWithCallback` / `reveal`, `revealHelper` calls:

```solidity
currentHash = revelation;
while (numHashes > 0) {
    currentHash = keccak256(bytes.concat(currentHash));
    numHashes -= 1;
}
``` [3](#0-2) 

Each `keccak256` costs ~30 gas. At 1 000 000 outstanding unfulfilled requests the loop consumes ~30 M gas, which equals or exceeds the Ethereum block gas limit, causing every reveal attempt to revert out-of-gas.

**Attacker-controlled entry path**

Any address can call `requestWithCallback` or `requestV2` by paying the provider's fee:

```solidity
function requestV2(
    address provider,
    bytes32 userContribution,
    uint32 gasLimit
) public payable override returns (uint64) {
``` [4](#0-3) 

The attacker pays the fee, receives a sequence number, and simply never calls `reveal`. Each such call increments `providerInfo.sequenceNumber` while `currentCommitmentSequenceNumber` stays fixed, widening the gap. [5](#0-4) 

---

### Impact Explanation

- **Entropy provider DoS**: Once `numHashes` is large enough, every `revealWithCallback` / `reveal` transaction for that provider reverts out-of-gas. No user of that provider can receive a random number.
- **Existing in-flight requests abandoned**: The provider's only recovery path is calling `advanceProviderCommitment` to reset `currentCommitmentSequenceNumber`, but this abandons all previously created requests, denying randomness to legitimate users who already paid.
- **Severity: High** — complete loss of service for the targeted provider and all its users; no privileged access required.

---

### Likelihood Explanation

- Any Ethereum address can call `requestWithCallback` permissionlessly.
- Providers with low fees (e.g., 1 wei) can be attacked for negligible cost; even at moderate fees the attacker only needs ~1 M requests to hit the block gas limit.
- `maxNumHashes` defaults to `0` (disabled) for every provider that does not explicitly call `setMaxNumHashes`, which is the common case.
- The attack is deterministic and requires no special knowledge beyond the provider's address.

---

### Recommendation

1. **Enforce a non-zero default for `maxNumHashes`** during `register`, or require providers to set it before accepting requests.
2. **Reject requests when `maxNumHashes == 0`** (treat zero as "not configured" and revert), forcing providers to opt in with an explicit limit.
3. Alternatively, cap `numHashes` at a protocol-level constant (e.g., 1 000) regardless of provider configuration.

---

### Proof of Concept

1. Deploy or target an Entropy provider with `maxNumHashes == 0` (default after `register`).
2. Call `requestV2(provider, randomBytes32, 0)` with the required fee, ~1 000 000 times, never calling `reveal`.
3. After step 2, `providerInfo.sequenceNumber` is ~1 000 000 ahead of `currentCommitmentSequenceNumber`.
4. Any legitimate user calls `requestV2` → their request gets `numHashes ≈ 1 000 000`.
5. The provider calls `revealWithCallback` for that request → the `constructProviderCommitment` loop executes ~1 000 000 keccak256 operations → transaction reverts out-of-gas.
6. The provider is permanently unable to fulfill any new request without abandoning all prior ones via `advanceProviderCommitment`. [6](#0-5) [1](#0-0)
