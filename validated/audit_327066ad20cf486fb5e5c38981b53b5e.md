Audit Report

## Title
Unbounded Cell Iteration in `get_cells_capacity` Enables RPC-Triggered CPU Exhaustion and Tx-Pool Lock Contention — (File: `util/indexer/src/service.rs`)

## Summary
`IndexerHandle::get_cells_capacity` performs an unbounded RocksDB scan over all matching live cells with no caller-supplied limit and no server-side `request_limit` check, unlike `get_cells` which enforces both. During the entire scan the tx-pool `RwLock` is held in read mode, blocking any concurrent writer for up to the full 10-second `TimeoutIterator` window. An unprivileged attacker sending a small number of concurrent broad-prefix requests can saturate the RPC thread pool and stall tx-pool writers at negligible cost.

## Finding Description
`get_cells` enforces both a caller-supplied limit and a server-side `request_limit` cap before any iteration begins: [1](#0-0) 

`get_cells_capacity` applies only the `Partial`-mode guard and then proceeds directly to an unbounded scan — no `limit` parameter exists, and `self.request_limit` is never consulted: [2](#0-1) 

The tx-pool read lock is acquired immediately after the iterator is created, before any cell is processed: [3](#0-2) 

The lock is then held across the entire `.take_while(...).filter_map(...).sum()` chain, which runs until the prefix is exhausted or the timeout fires: [4](#0-3) 

`TimeoutIterator` only checks wall-clock elapsed time on each `next()` call; it imposes no bound on the number of cells processed before the timeout fires: [5](#0-4) 

The defaults are `request_limit = usize::MAX` and `timeout_limit = 10 s`, meaning out-of-the-box deployments are fully exposed: [6](#0-5) 

The RPC entry point accepts only a `search_key` with no `limit` parameter and applies no authentication or rate-limiting: [7](#0-6) 

**Exploit flow:**
1. Attacker sends 10–20 concurrent `get_cells_capacity` requests with `code_hash` = secp256k1 lock and `args = "0x"` (prefix mode, matches all secp256k1 cells).
2. Each request acquires the tx-pool read lock and begins iterating all matching cells.
3. Each request holds the read lock for up to 10 s while performing per-cell RocksDB lookups.
4. Any concurrent tx-pool writer (block processing, `send_transaction`) that needs the write lock is blocked until all readers release.
5. The RPC thread pool is saturated; legitimate callers queue or time out.
6. Requests can be repeated indefinitely with no on-chain cost.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The attack requires only standard JSON-RPC HTTP requests with no keys or funds. A small number of concurrent requests can hold the tx-pool `RwLock` in read mode for up to 10 s per wave, blocking block-processing notifications and transaction submissions on the targeted node. Infrastructure nodes (wallets, explorers, dApps) that enable `--indexer` are the primary targets; degrading these nodes causes effective network congestion for their users at negligible attacker cost.

## Likelihood Explanation
The indexer RPC is publicly accessible on any node that enables `--indexer`, which is standard for infrastructure nodes. The secp256k1 lock `code_hash` is publicly known and has tens of millions of live cells on CKB mainnet. The attack requires only a standard HTTP client with no privileged access or on-chain funds. The default `timeout_limit` of 10 s and `request_limit` of `usize::MAX` are the out-of-the-box values; operators are unlikely to have tightened them. The attack is repeatable indefinitely with no cooldown or cost.

## Recommendation
1. **Add a server-side cell-count limit to `get_cells_capacity`**: mirror the guard in `get_cells` (lines 212–221) — after scanning `self.request_limit` matching cells, stop iteration and return an error.
2. **Release the tx-pool read lock before the scan**, or snapshot the consumed set into a local `HashSet` before iteration so the lock is not held across the full RocksDB scan.
3. **Consider a hard per-request cell-count cap** independent of wall-clock time, so that fast hardware cannot be forced to scan more cells than the operator intends within the timeout window.
4. **Consider rate-limiting** the `get_cells_capacity` endpoint at the RPC layer, or requiring callers to supply a narrower (exact-match) `search_key` when the result set is expected to be large.

## Proof of Concept
Send the following JSON-RPC request to a CKB node with the indexer enabled, repeated 10–20 times concurrently:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells_capacity",
  "params": [
    {
      "script": {
        "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
        "hash_type": "type",
        "args": "0x"
      },
      "script_type": "lock"
    }
  ]
}
```

**Expected observable effects:**
- Each request runs for ~10 s (until `TimeoutIterator` fires) and returns `Error: Indexer request timeout`.
- During those 10 s, the tx-pool read lock is held; any concurrent tx-pool write operation (e.g., `send_transaction`) blocks until the lock is released.
- Legitimate RPC calls queue behind the saturated thread pool.

**Minimal unit test plan:**
1. Populate a test indexer store with a large number of cells sharing the same lock script prefix.
2. Spawn 10 concurrent threads each calling `IndexerHandle::get_cells_capacity` with that prefix.
3. Concurrently attempt a tx-pool write operation and measure its latency — it should be blocked for up to `timeout_limit` seconds.
4. Verify that `get_cells` with the same prefix and `limit = 1` returns immediately, confirming the asymmetry.
