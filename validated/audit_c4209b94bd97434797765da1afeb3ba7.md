### Title
Unbounded JSON-RPC Batch Requests and Missing Response Size Cap Enable Unauthenticated Resource Exhaustion — (`rpc/src/server.rs`)

---

### Summary

CKB's JSON-RPC server has no default batch-size limit (`rpc_batch_limit = None`), no response-size cap, and no per-endpoint concurrency guard for large-output methods. An unauthenticated RPC caller can send arbitrarily large batch requests or single requests to high-amplification endpoints (`get_cells`, `get_transactions`, `get_block`), causing the server to perform unbounded sequential work and fully materialize large in-memory structures before streaming them, leading to sustained CPU/I/O exhaustion and potential OOM that denies service to legitimate users.

---

### Finding Description

**Root cause 1 — No default batch limit**

`rpc_batch_limit` is typed as `Option<usize>` and defaults to `None`: [1](#0-0) 

The production default config leaves it commented out: [2](#0-1) 

In `handle_jsonrpc`, the batch guard only fires when `JSONRPC_BATCH_LIMIT` has been set; when it is `None`, the check is skipped entirely and the batch is processed without any size gate: [3](#0-2) 

**Root cause 2 — No response size cap; single-request path fully materializes before sending**

For `Request::Single`, the server awaits the full handler result and then calls `serde_json::to_string(&response)` with no size check before returning: [4](#0-3) 

There is no `max_response_size` equivalent anywhere in the CKB RPC stack.

**Root cause 3 — High-amplification endpoints with attacker-controlled `limit`**

`get_cells` and `get_transactions` accept a `limit: Uint32` (up to 4,294,967,295). The RPC layer passes this value directly to the indexer handle with no server-side cap: [5](#0-4) [6](#0-5) 

The `IndexerHandle` does enforce a `request_limit` field, but this field is initialized from `IndexerConfig` and its production default is not bounded at the RPC layer. The `AsyncRichIndexerHandle` enforces the same field: [7](#0-6) 

If `request_limit` is set to a large value (e.g., `usize::MAX` as seen in tests), the entire result set is collected into a `Vec<IndexerCell>` in memory and then serialized in one pass by `serde_json::to_string`, with no size check at the RPC layer. [8](#0-7) 

---

### Impact Explanation

- **Unlimited batch**: With `rpc_batch_limit = None` (the default), an attacker sends a single HTTP POST containing a JSON array of N=100,000+ calls to `get_block` or `get_cells`. The server processes each sequentially, performing N database reads and N `serde_json::to_string` serializations. The node becomes unresponsive to legitimate users for the duration.
- **Single-request OOM**: A single `get_cells` call with `limit = 0xFFFFFFFF` (if `request_limit` is large) causes the server to collect millions of `IndexerCell` structs into a `Vec`, then serialize the entire structure with `serde_json::to_string` — duplicating memory — with no size gate. This can exhaust available RAM and trigger OOM termination.
- **No response size cap** means neither path has a safety valve; the size check that would reject oversized responses does not exist.

---

### Likelihood Explanation

- The RPC endpoint is reachable by any unauthenticated local or remote caller (depending on `listen_address`).
- The default config ships with `rpc_batch_limit` commented out (disabled) and no `max_response_size`.
- No authentication is required; a single crafted HTTP POST is sufficient.
- The Indexer and RichIndexer modules are listed as optional but are commonly enabled in production deployments.

---

### Recommendation

1. **Set a safe default for `rpc_batch_limit`** (e.g., 100–500) in `resource/ckb.toml` and enforce it unconditionally rather than only when the operator explicitly sets it.
2. **Add a response size cap** in `handle_jsonrpc` for `Request::Single`: check `json.len()` against a configurable `max_response_size` before returning, and return an error if exceeded.
3. **Cap `limit` at the RPC layer** for `get_cells` / `get_transactions` (e.g., 1000) independently of the indexer-internal `request_limit`, so the protection is always active regardless of indexer configuration.
4. For the batch path, consider adding a per-batch aggregate byte-budget check so that a batch of many moderately-sized responses cannot collectively exhaust resources.

---

### Proof of Concept

```
# Step 1: No batch limit — CPU/IO exhaustion
POST http://127.0.0.1:8114/
Content-Type: application/json

[
  {"id":1,"jsonrpc":"2.0","method":"get_block","params":["<tip_hash>","0x2"]},
  {"id":2,"jsonrpc":"2.0","method":"get_block","params":["<tip_hash>","0x2"]},
  ... (repeat 50,000 times)
]

# The server processes all 50,000 calls sequentially with no limit gate.
# Node becomes unresponsive for the duration.

# Step 2: Single-request amplification (if request_limit is large)
POST http://127.0.0.1:8114/
Content-Type: application/json

{
  "id":1,"jsonrpc":"2.0","method":"get_cells",
  "params":[
    {"script":{"code_hash":"0x00...00","hash_type":"data","args":"0x"},
     "script_type":"lock"},
    "asc",
    "0xFFFFFFFF",
    null
  ]
}

# Server attempts to collect up to 4B IndexerCell structs into a Vec,
# then calls serde_json::to_string on the entire structure.
# No size check is applied. Process is OOM-killed.
```
