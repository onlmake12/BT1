### Title
`Echo.executeCallback` Credits Fees to Caller-Controlled `providerToCredit` Without Validation After Exclusivity Period - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, the `executeCallback` function accepts a caller-controlled `providerToCredit` address and credits it with the requester's pre-paid fee. After the exclusivity period elapses, there is no validation that `providerToCredit` equals `msg.sender` or that it is the legitimate fulfilling provider. An unprivileged attacker who registers as a provider can call `executeCallback` after the exclusivity window and redirect the entire provider fee to their own address, stealing it from the original assigned provider.

### Finding Description
When a user calls `requestPriceUpdatesWithCallback`, the contract stores the provider fee:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [1](#0-0) 

When `executeCallback` is later called, the fee is credited to the caller-supplied `providerToCredit`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The only guard on `providerToCredit` is the exclusivity-period check:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(providerToCredit == req.provider, "Only assigned provider during exclusivity period");
}
``` [3](#0-2) 

Once the exclusivity period expires, **no constraint** is placed on `providerToCredit`. There is no requirement that it equals `msg.sender`, that it equals `req.provider`, or that it is even a registered provider. Any caller can pass an arbitrary address and have `req.fee` credited there.

Provider registration is permissionless:

```solidity
function registerProvider(uint96 baseFeeInWei, uint96 feePerFeedInWei, uint96 feePerGasInWei) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
``` [4](#0-3) 

Withdrawal requires a fee manager, which any registered provider can set:

```solidity
function setFeeManager(address manager) external override {
    require(_state.providers[msg.sender].isRegistered, "Provider not registered");
    _state.providers[msg.sender].feeManager = manager;
``` [5](#0-4) 

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(msg.sender == _state.providers[provider].feeManager, "Only fee manager");
    ...
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [6](#0-5) 

### Impact Explanation
An attacker who registers as a provider can steal the entire `req.fee` from the legitimate provider for every request whose exclusivity period has elapsed. The requester's callback is still executed (so the requester is not harmed), but the original provider receives zero compensation for the service they were assigned to provide. At scale, this drains provider revenue and disincentivizes legitimate providers from operating, degrading the liveness of the Echo service.

### Likelihood Explanation
The attack is straightforward and requires only:
1. Registering as a provider (permissionless, zero cost beyond gas).
2. Waiting for the exclusivity period to elapse on any pending request.
3. Fetching valid price update data from the public Hermes API.
4. Calling `executeCallback(attacker_address, sequenceNumber, updateData, priceIds)` with `msg.value = pythFee`.

The attacker profits whenever `req.fee > pythFee` (i.e., whenever the requester paid a meaningful provider fee, which is the normal case). Hermes price update data is publicly available, making step 3 trivial.

### Recommendation
Require `providerToCredit == msg.sender` in `executeCallback`, or at minimum require that `providerToCredit` is a registered provider at the time of the call. This ensures that only the actual transaction sender can claim the fee, preventing fee redirection attacks:

```solidity
// After exclusivity period, providerToCredit must be the caller
require(providerToCredit == msg.sender, "providerToCredit must be caller");
require(_state.providers[providerToCredit].isRegistered, "Provider not registered");
```

### Proof of Concept
1. Alice (requester) calls `requestPriceUpdatesWithCallback(legitimateProvider, ...)` paying `totalFee`. `req.fee = totalFee - pythFeeInWei` is stored; `req.provider = legitimateProvider`.
2. Attacker calls `registerProvider(...)` to register their address.
3. Attacker calls `setFeeManager(attacker_address)` to set themselves as fee manager.
4. After `exclusivityPeriodSeconds` elapses, attacker fetches valid `updateData` from Hermes.
5. Attacker calls `executeCallback(attacker_address, sequenceNumber, updateData, priceIds)` with `msg.value = pythFee`.
6. `_state.providers[attacker_address].accruedFeesInWei += req.fee + pythFee - pythFee = req.fee`.
7. Attacker calls `withdrawAsFeeManager(attacker_address, req.fee)` and receives `req.fee` in ETH.
8. `legitimateProvider` receives nothing despite being the assigned provider.
