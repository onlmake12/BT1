### Title
Incorrect Zero-Price Guard in Deviation Check Silently Skips Maximum-Deviation Updates - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

In `Scheduler.sol`, the `_validateShouldUpdatePrices` function contains a guard `if (previousPrice == 0 || currentPrice == 0) { continue; }` that silently skips the deviation check for any price feed whose current or previous price is zero. This is the direct analog of the Venus `Scores.sol` bug: an overly broad early-exit guard that does not account for the edge case where the guard condition itself represents the most extreme possible deviation (a price crashing to zero, or recovering from zero). For subscriptions configured with `updateOnDeviation: true` and `updateOnHeartbeat: false`, this causes the `updatePriceFeeds` call to revert with `UpdateConditionsNotMet` precisely when the price movement is most extreme and the update is most urgently needed.

### Finding Description

Inside `_validateShouldUpdatePrices`, after confirming a previous price feed exists, the code computes the deviation between the stored price and the incoming price:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;
int64 previousPrice = previousFeed.price.price;

// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;                          // ← silently skips this feed
}

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);

if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
    return updateTimestamp;            // trigger update
}
```

The guard is correct for preventing a division-by-zero when `previousPrice == 0`. However, `continue` is the wrong response: it treats the zero-price case as "no deviation detected" and moves on to the next feed. If all feeds in the subscription hit this branch, the loop exhausts without returning, and the function falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
```

The correct semantic is the opposite: a price moving **to** zero from any non-zero value represents a deviation of exactly 10,000 bps (100%), which exceeds every valid `deviationThresholdBps`. A price moving **from** zero to any non-zero value is an infinite deviation. Both cases should unconditionally trigger the update, not suppress it.

The Venus analog is structurally identical: `if (xvs == 0 || capital == 0) return 0` returned the wrong value for the edge case where the guard condition is mathematically valid (alpha = 1 makes capital irrelevant). Here, `continue` produces the wrong outcome for the edge case where the guard condition represents the maximum possible deviation.

### Impact Explanation

Subscriptions configured with **deviation-only** updates (`updateOnDeviation: true`, `updateOnHeartbeat: false`) will be unable to push any price update when a tracked asset's price reaches zero or recovers from zero. This is the worst possible time for a price feed to go stale: a price crashing to zero signals a token exploit, depeg, or market halt — exactly the moment downstream DeFi protocols (lending, derivatives, liquidation engines) need accurate data to protect users. The on-chain stored price remains at the last non-zero value, causing:

- Lending protocols to under-collateralize positions (no liquidations triggered).
- Derivatives protocols to misprice options/futures.
- Any protocol reading `getPricesNoOlderThan` with a loose staleness window to receive a dangerously incorrect price.

Keepers are also economically harmed: they spend gas submitting a valid update that reverts, receive no payment, and cannot retry until the heartbeat fires (if one is configured).

### Likelihood Explanation

- Pyth price feeds use `int64` for price values; a zero value is a valid on-chain state.
- Subscriptions with deviation-only criteria are explicitly supported and documented in the SDK (`updateOnHeartbeat: false`, `updateOnDeviation: true`).
- A price crashing to zero is a realistic market event (e.g., UST depeg, exploited tokens, synthetic assets).
- No privileged access is required; any keeper calling `updatePriceFeeds` triggers the revert.
- The bug is deterministic and reproducible with a simple Foundry test.

### Recommendation

Replace `continue` with a branch that treats a zero-price transition as maximum deviation:

```solidity
if (previousPrice == 0 || currentPrice == 0) {
    // A price moving to or from zero is an infinite/maximum deviation.
    // Treat it as exceeding any threshold and trigger the update.
    if (previousPrice != currentPrice) {
        return updateTimestamp;
    }
    continue; // both are zero: no change, skip
}
```

This mirrors the Venus fix: instead of a blanket early exit, handle the edge case correctly before falling through to the general formula.
