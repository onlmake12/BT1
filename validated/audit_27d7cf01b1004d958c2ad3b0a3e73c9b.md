### Title
Unchecked Return Values on Critical Tx-Pool Coordination Calls During Block Verification — (`chain/src/verify.rs`)

### Summary
In `chain/src/verify.rs`, the return values of `tx_pool_controller.suspend_chunk_process()` and `tx_pool_controller.continue_chunk_process()` are silently discarded using `let _ = ...`. These calls are the sole mechanism for coordinating between block verification and the tx-pool's concurrent chunk-processing loop. If either call fails, the failure is invisible to the caller and execution continues as if the coordination succeeded.

### Finding Description
Inside `ConsumeUnverifiedBlocks::start`, before and after every block is verified, the code calls:

```rust
let _ = self.tx_pool_controller.suspend_chunk_process();
// ... full contextual block verification + chain state mutation ...
let _ = self.tx_pool_controller.continue_chunk_process();
``` [1](#0-0) 

The same pattern appears in the `truncate` path:

```rust
let _ = self.tx_pool_controller.suspend_chunk_process();
let _ = responder.send(self.processor.truncate(&target_tip_hash));
let _ = self.tx_pool_controller.continue_chunk_process();
``` [2](#0-1) 

Both `suspend_chunk_process` and `continue_chunk_process` communicate with the tx-pool service over an internal channel and return a `Result`. If the tx-pool service is degraded, the channel is closed, or the send fails for any reason, the `Result::Err` is thrown away. Two distinct failure modes arise:

**Failure mode A — `suspend_chunk_process` silently fails:** Block verification and chain-state mutation (attach/detach blocks, update snapshot, reorg) proceed while the tx-pool's chunk-processing loop is still running. The tx-pool reads cell data and chain state concurrently with the verifier mutating it, producing a logical race that can leave the tx-pool holding transactions that reference cells that no longer exist or have been double-spent on the new chain tip.

**Failure mode B — `continue_chunk_process` silently fails:** After block verification completes, the tx-pool's chunk-processing loop is never resumed. The tx-pool stops verifying and admitting transactions permanently for the lifetime of the process. No further transactions can be included in mined blocks, constituting a node-level denial of service on transaction throughput.

### Impact Explanation
Failure mode B is the higher-severity outcome: a node whose tx-pool chunk-processing is permanently suspended will never include user transactions in blocks it mines or relays, silently degrading the node to a state where it participates in consensus but produces empty (or coinbase-only) blocks. Operators have no direct signal that this has occurred because the error is swallowed.

Failure mode A produces a logical race between the verifier and the tx-pool, potentially causing the tx-pool to retain double-spent or now-invalid transactions after a reorg, or to evict valid transactions based on stale cell state.

### Likelihood Explanation
The tx-pool controller communicates over a crossbeam or tokio channel. Channel send errors occur when the receiver has been dropped (service shutdown race) or, for bounded channels, when the channel is full. During IBD or under heavy block relay load, the tx-pool service can be under significant pressure. A peer that relays a burst of valid blocks in rapid succession can cause the block-verification loop to call `suspend_chunk_process` faster than the tx-pool service drains its inbox, making a send failure plausible without any privileged access. Any remote peer that can relay valid (or minimally valid) blocks is a sufficient attacker.

### Recommendation
Propagate the errors from both calls rather than discarding them. At minimum, log and halt block processing if `suspend_chunk_process` fails (since proceeding without the lock is unsafe), and log and attempt a retry or panic if `continue_chunk_process` fails (since leaving the tx-pool permanently suspended is worse than crashing):

```rust
self.tx_pool_controller.suspend_chunk_process()
    .map_err(|e| { error!("failed to suspend chunk process: {}", e); e })?;
// ... verification ...
if let Err(e) = self.tx_pool_controller.continue_chunk_process() {
    error!("failed to resume chunk process: {}", e);
    // retry or propagate
}
``` [3](#0-2) 

### Proof of Concept
1. Run a CKB node with an active tx-pool.
2. Arrange for the tx-pool service's inbox to be at capacity (e.g., by flooding the node with transactions that each require chunk-processing) at the moment a new block arrives from a peer.
3. The block-verification loop calls `suspend_chunk_process()`; the send to the tx-pool inbox fails because the channel is full; `let _ = ...` discards the error.
4. Block verification and chain-state mutation proceed concurrently with the tx-pool's chunk loop.
5. After verification, `continue_chunk_process()` similarly fails; the tx-pool's chunk loop is never resumed.
6. All subsequent transaction submissions are silently queued but never chunk-verified; the node mines empty blocks indefinitely. [4](#0-3)
