### Title
Unbounded `last_n_numbers` allocation via `difficulty_boundary=U256::zero()` bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
An unprivileged light-client peer can send a `GetLastStateProof` message with `difficulty_boundary=U256::zero()`, `last_n_blocks=499`, `start_number=0`, and a valid `last_hash`. The upfront limit check only bounds `difficulties.len() + last_n_blocks*2`, which passes (998 ≤ 1000). However, `difficulty_boundary=0` causes `get_first_block_total_difficulty_is_not_less_than` to immediately return block 0 as the boundary, setting `difficulty_boundary_block_number=0`. The subsequent adjustment guard does not fire (chain height >> 499), so `last_n_numbers = (0..last_block_number)` — a Vec proportional to the full chain height — is allocated and passed to `complete_headers` for N sequential DB lookups and MMR root computations.

### Finding Description
The root cause is that the limit check at line 201 only validates the *input* fields (`difficulties.len()` and `last_n_blocks`), but the *derived* quantity `last_n_numbers.len()` — which equals `last_block_number - difficulty_boundary_block_number` — is never bounded. When `difficulty_boundary=U256::zero()`, the binary-search function short-circuits at block 0 (since every block's total difficulty ≥ 0), collapsing the boundary to the chain start. The adjustment at line 313 only widens the boundary when there are *too few* blocks after it, not when there are too many. The result is that `last_n_numbers` spans the entire chain.

### Impact Explanation
For a chain of height N:
- `last_n_numbers` allocates a `Vec<u64>` of N entries (8N bytes).
- `complete_headers` performs N `get_ancestor` + `get_block` + `chain_root_mmr` calls — all synchronous DB/MMR operations on the server's snapshot.
- The response (`SendLastStateProof`) would contain N `VerifiableHeader` structs, potentially gigabytes of serialized data.

On a chain of 1M blocks this is ~8 MB for the Vec alone, but the DB I/O and response serialization dominate. The server thread handling this peer is blocked for the duration, and repeated requests from multiple peers can exhaust memory and I/O capacity, causing OOM or extreme latency on the light-client protocol server.

### Likelihood Explanation
The attack requires only a valid tip hash (publicly observable on the P2P network) and a single crafted `GetLastStateProof` message. No PoW, no key, no privileged role. Any peer connected to the light-client protocol port can trigger it.

### Recommendation
After computing `difficulty_boundary_block_number`, enforce that `last_block_number - difficulty_boundary_block_number <= last_n_blocks` (or `<= GET_LAST_STATE_PROOF_LIMIT`). Additionally, reject `difficulty_boundary=U256::zero()` explicitly, since it is semantically meaningless (every block satisfies it) and serves no legitimate client purpose.

### Proof of Concept
1. Mine a chain of 100,000 blocks on a local devnet with the light-client protocol enabled.
2. Obtain the tip hash `tip_hash`.
3. Send:
   ```
   GetLastStateProof {
     last_hash: tip_hash,
     start_hash: genesis_hash,
     start_number: 0,
     last_n_blocks: 499,
     difficulty_boundary: U256::zero(),
     difficulties: [],
   }
   ```
4. Observe that `last_n_numbers.len() == 100_000` (not ≤ 1000), the server performs 100,000 DB lookups, and the response is hundreds of MB.
5. Assert `last_n_numbers.len() <= 1000` — this assertion fails, confirming the invariant violation.
