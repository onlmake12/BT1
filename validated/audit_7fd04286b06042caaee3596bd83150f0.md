Audit Report

## Title
`get_fee_rate_statistics` RPC Panics on `txs_sizes: None` from Legacy `BlockExt` Deserialization — (File: `rpc/src/util/fee_rate.rs`)

## Summary
The `Unpack<core::BlockExt>` implementation for the legacy 5-field `packed::BlockExtReader` hardcodes `txs_sizes: None`. The `BlockExt2019ToZero` migration re-serializes all pre-hardfork blocks into the 7-field `BlockExtV1` format while preserving `txs_sizes: None`. Any call to `get_fee_rate_statistics` whose sliding window includes one of these migrated blocks hits an unconditional `.expect()` on `txs_sizes`, causing a panic in the RPC handler.

## Finding Description
`Unpack<core::BlockExt>` for `packed::BlockExtReader` (the legacy 5-field format) hardcodes `txs_sizes: None` at `util/types/src/conversion/storage.rs` lines 139–151. `ChainStore::get_block_ext` dispatches on `count_extra_fields()`: when the stored record has 0 extra fields it takes the legacy path and returns a `BlockExt` with `txs_sizes: None` (`store/src/store.rs`, lines 252–254). The `BlockExt2019ToZero` migration reads every pre-hardfork block ext, sets `cycles = None`, and re-inserts it via `insert_block_ext` (`util/migrate/src/migrations/set_2019_block_cycle_zero.rs`, lines 88–90). `insert_block_ext` serializes using `packed::BlockExtV1` (`store/src/transaction.rs`, line 246), so those blocks are now stored in 7-field format — causing `get_block_ext` to take the `count_extra_fields() == 2` path — but `txs_sizes` remains `None` because the `From<&core::BlockExt> for packed::BlockExtV1` conversion at `util/types/src/conversion/storage.rs` lines 189–200 faithfully encodes the `None` value. In `FeeRateCollector::statistics`, the closure passed to `collect` calls `txs_sizes.expect("expect txs_size's length >= 1")` (`rpc/src/util/fee_rate.rs`, line 93) with no prior `None` guard. When the sliding window (up to 101 blocks) includes any migrated block, this panics.

## Impact Explanation
The panic occurs inside the RPC handler closure. Depending on whether the JSON-RPC runtime uses `catch_unwind`, the result is either an internal error response or a thread/process abort. In either case `get_fee_rate_statistics` (and deprecated `get_fee_rate_statics`) becomes non-functional for any caller while migrated blocks remain in the window. This matches: **Note (0–500 points) — Any local RPC API crash.**

## Likelihood Explanation
The condition is met on any node that ran a pre-v0.106 binary and then upgraded. After upgrade, the migration re-writes all historical blocks with `txs_sizes: None`. The RPC is broken until 101 new post-upgrade blocks have been produced. During that window, any unauthenticated caller who sends `{"method":"get_fee_rate_statistics"}` triggers the panic. No special privilege, hashpower, or authentication is required.

## Recommendation
Replace the unconditional `.expect()` in `rpc/src/util/fee_rate.rs` line 93 with a graceful skip:
```rust
let Some(txs_sizes) = txs_sizes else { return fee_rates; };
```
This makes the closure skip blocks that lack `txs_sizes` rather than panicking, which is the correct behavior for migrated legacy blocks.

## Proof of Concept
1. Run a CKB node on a version prior to v0.106 until several blocks are stored in the legacy 5-field `BlockExt` format.
2. Upgrade the binary. The `BlockExt2019ToZero` migration re-writes those blocks in 7-field format with `txs_sizes: None`.
3. While the chain tip is fewer than 101 blocks past the last migrated block, send:
   ```json
   {"jsonrpc":"2.0","method":"get_fee_rate_statistics","params":[],"id":1}
   ```
4. `FeeRateCollector::collect` iterates the window, hits a block with `txs_sizes: None`, and the `.expect()` at `rpc/src/util/fee_rate.rs:93` panics, crashing or erroring the RPC handler.
