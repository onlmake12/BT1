### Title
Excess `msg.value` Is Not Refunded to Users in Entropy Request Functions - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

All public Entropy request functions (`request()`, `requestWithCallback()`, `requestV2()`) route through the internal `requestHelper()` function, which checks that `msg.value >= requiredFee` but silently absorbs any overpayment into Pyth's accrued fee pool (`_state.accruedPythFeesInWei`) rather than refunding the excess to the caller. Any user who overpays loses the excess ETH permanently.

---

### Finding Description

In `requestHelper()`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value` is split between `providerFee` and `accruedPythFeesInWei`. If `msg.value > requiredFee`, the surplus is credited to Pyth's fee pool — not returned to the caller. This is the root cause.

All three public entry points funnel into this path:

- `request()` [2](#0-1) 
- `requestWithCallback()` [3](#0-2) 
- `requestV2(address, bytes32, uint32)` [4](#0-3) 

The behavior is explicitly acknowledged in the interface documentation:

> "Note that excess value is *not* refunded to the caller." [5](#0-4) [6](#0-5) 

This confirms the behavior is present and known, but it is not mitigated. By contrast, `PythLazer.sol` correctly refunds excess:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [7](#0-6) 

This demonstrates Pyth is aware of the refund pattern and has applied it selectively — but not to Entropy.

---

### Impact Explanation

Any user who calls `request()`, `requestWithCallback()`, or any `requestV2()` variant with `msg.value` exceeding the required fee permanently loses the excess ETH. The surplus is credited to `_state.accruedPythFeesInWei`, which is only withdrawable by Pyth governance — there is no mechanism for users to reclaim overpayments. This is a direct, irreversible loss of user funds. [8](#0-7) 

---

### Likelihood Explanation

Overpayment is realistic in practice:
1. Fees can change between the time a user calls `getFee()`/`getFeeV2()` and the time the transaction is mined (provider re-registers with a new fee).
2. Frontend/SDK integrations may add a buffer to `msg.value` to avoid `InsufficientFee` reverts.
3. Users calling the contract directly may round up their payment.

Any of these common scenarios results in silent, permanent fund loss. The entry path requires no privilege — any EOA or contract can trigger it.

---

### Recommendation

In `requestHelper()`, after all state changes, calculate and refund any excess payment to `msg.sender`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee); // use requiredFee, not msg.value

// ... all state changes ...

// Refund excess at the end (CEI pattern)
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool success, ) = payable(msg.sender).call{value: excess}("");
    require(success, "Refund failed");
}
```

This mirrors the pattern already used in `PythLazer.sol`.

---

### Proof of Concept

1. Provider fee is 0.001 ETH, Pyth fee is 0.0001 ETH → `requiredFee = 0.0011 ETH`.
2. User calls `requestWithCallback{value: 0.01 ether}(provider, userRandomNumber)`.
3. `msg.value (0.01) >= requiredFee (0.0011)` → check passes.
4. `providerInfo.accruedFeesInWei += 0.001 ETH`.
5. `_state.accruedPythFeesInWei += (0.01 - 0.001) = 0.009 ETH` — the entire overpayment goes to Pyth.
6. User paid 0.01 ETH for a service costing 0.0011 ETH, losing 0.0089 ETH with no refund and no recourse. [9](#0-8)

            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.

            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
