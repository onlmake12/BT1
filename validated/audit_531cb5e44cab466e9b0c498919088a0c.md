Audit Report

## Title
`forward_rate_limiter` Bypassed via Attacker-Controlled `from`/`to` Fields — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`, where both `from` and `to` are deserialized directly from the attacker-supplied message body with no verification against the actual sending session. An attacker with a single standard P2P connection can send up to 30 `ConnectionRequest` messages per second (the outer per-session cap), each carrying a distinct fabricated `(from, to)` pair, causing every message to pass the inner limiter and be forwarded to sqrt(N) other peers — the `forward_rate_limiter` provides zero additional protection. Additionally, the `HashMapStateStore` backing `forward_rate_limiter` grows without bound for the duration of the attacker's session, as `retain_recent()` is never called in the periodic `notify()` callback.

## Finding Description

**Rate limiter setup** (`mod.rs` L251–257): Two rate limiters are created. The outer `rate_limiter` is keyed by `(PeerIndex, u32)` — the real, non-spoofable session index — and allows 30 req/s. The inner `forward_rate_limiter` is keyed by `(PeerId, PeerId, u32)` and allows 1 req/s per unique key. [1](#0-0) 

**Outer limiter check** (`mod.rs` L95–107): Correctly uses `(session_id, msg.item_id())` — the real session, not attacker-controlled. [2](#0-1) 

**Inner limiter check** (`connection_request.rs` L132–143): Uses `(content.from.clone(), content.to.clone(), self.msg_item_id)`. Both `content.from` and `content.to` are parsed from the message payload in `RequestContent::try_from()` (L35–82) with no assertion that `content.from` equals the peer ID of the sending session. The `peer: PeerIndex` field is present in `ConnectionRequestProcess` but is never used to validate `content.from`. [3](#0-2) 

The identical pattern appears in `connection_request_delivered.rs` L134–145: [4](#0-3) 

And in `connection_sync.rs` L85–95: [5](#0-4) 

**Forwarding path** (`connection_request.rs` L145–152, L273–305): When `content.to` is not the local peer and `max_hops > 0`, `forward_message()` is called. If the target is not directly connected, the message is broadcast to `sqrt(total_peers)` peers via `filter_broadcast`. An attacker sets `content.to` to any unknown peer ID to trigger this broadcast path. [6](#0-5) [7](#0-6) 

**No periodic pruning** (`mod.rs` L169–244): The `notify()` callback (fired every `CHECK_INTERVAL` = 5 minutes) prunes `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter. `retain_recent()` is only called in `disconnected()` (L67–68). A long-lived attacker session causes the `forward_rate_limiter`'s `HashMapStateStore` to accumulate one entry per unique `(from, to)` pair used, growing without bound for the duration of the connection. [8](#0-7) [9](#0-8) 

**Exploit flow:**
1. Attacker establishes a standard P2P connection to a CKB node on `SupportProtocols::HolePunching`.
2. Attacker sends 30 `ConnectionRequest` messages per second, each with a freshly generated `(from, to)` pair and `max_hops > 0`, `to` set to an unknown peer ID.
3. The outer limiter passes all 30 (budget not exceeded for the session).
4. Each unique `(from, to)` pair is a new key in `forward_rate_limiter`; each check passes (fresh bucket, 0 prior uses).
5. Each message triggers `forward_message()` → `filter_broadcast()` to sqrt(N) peers.
6. The `forward_rate_limiter` hashmap grows by 30 entries/second, never pruned until disconnect.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Each attacker connection (no privilege required) generates 30 forwarded broadcasts per second, each reaching sqrt(N) peers. With N=100 connected peers, that is 30 × 10 = 300 outbound forwarding sends per second per attacker connection. Multiple attacker connections multiply this linearly. This wastes outbound bandwidth and CPU on message serialization across the gossip graph, constituting network congestion achievable at minimal cost to the attacker. Additionally, the unbounded `HashMapStateStore` growth (≈30 entries/s/connection, each entry holding two `PeerId` values plus rate-limiter state) causes sustained memory growth for the lifetime of the attacker's session.

## Likelihood Explanation
Only a standard P2P connection is required — no keys, stake, or privileged role. Crafting molecule-encoded `ConnectionRequest` messages with varying `from`/`to` byte sequences is straightforward with any P2P client. The inner rate limiter is never triggered (no log warning emitted), so the attack is silent. It is repeatable indefinitely and scales linearly with the number of attacker connections.

## Recommendation
1. **Verify `content.from` against the actual session**: After parsing, assert `content.from == get_peer_id(self.peer)`. Reject (and ban) messages where `content.from` does not match the sending session's peer ID.
2. **Re-key `forward_rate_limiter` by session**: Use `(peer_index, msg_item_id)` — the same key as the outer limiter — so the forwarding budget is tied to the real sender, not attacker-supplied fields.
3. **Periodically prune the hashmap**: Call `forward_rate_limiter.retain_recent()` inside the `notify()` callback (every `CHECK_INTERVAL`) rather than only on disconnect, to bound memory growth during long-lived sessions.

## Proof of Concept
```
1. Connect to a CKB node on SupportProtocols::HolePunching.
2. In a loop at ≤30 iterations/second:
   a. Generate a fresh random PeerId for `from` (32 random bytes, valid multihash).
   b. Generate a random PeerId for `to` that is NOT the local node's peer ID.
   c. Build a valid ConnectionRequest molecule message:
        from=<random>, to=<random>, max_hops=6,
        listen_addrs=[/ip4/1.2.3.4/tcp/8115/p2p/<from>], route=[]
   d. Send it to the node.
3. Observe: the node forwards each message to sqrt(N) peers via filter_broadcast
   (no TooManyRequests returned for any message).
4. After T seconds: forward_rate_limiter contains 30*T entries;
   30*T*sqrt(N) forwarding sends have been issued to other peers.
5. Confirm memory growth by monitoring the node's RSS over time with the
   attacker connection held open (no disconnect to trigger retain_recent).
```
