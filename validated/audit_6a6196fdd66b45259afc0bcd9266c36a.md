Audit Report

## Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Eviction Values, Inflating Pool Size Accounting - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` snapshots `self.total_tx_size + entry.size` into local variables before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts entries via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, those calls immediately decrement `self.total_tx_size`/`self.total_tx_cycles`. Lines 218–219 then unconditionally overwrite `self` with the stale pre-eviction locals, erasing the decrements and inflating the pool's size accounting. The inflated total causes `limit_size` to evict legitimate pending/proposed transactions that would otherwise fit.

## Finding Description
**Root cause — stale snapshot overwrite:**

`add_entry` (lines 200–221) calls `updated_stat_for_add_tx` at line 210 before any eviction occurs:

```rust
// pool_map.rs:210-211
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
```

`updated_stat_for_add_tx` (lines 711–729) takes `&self` and returns new totals as local values without touching `self`:

```rust
// pool_map.rs:716
let total_tx_size = self.total_tx_size.checked_add(tx_size)…?;
```

`check_and_record_ancestors` (lines 588–640) enters the eviction branch when `ancestors_count > self.max_ancestors_count` and `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count`. Inside that branch it calls:

```rust
// pool_map.rs:618
let removed = self.remove_entry_and_descendants(next_id);
```

`remove_entry_and_descendants` → `remove_entry` (line 247) → `update_stat_for_remove_tx` immediately writes the decremented values back to `self.total_tx_size`/`self.total_tx_cycles` (lines 738–740).

After `check_and_record_ancestors` returns, lines 218–219 unconditionally overwrite those correctly-decremented fields with the stale locals:

```rust
// pool_map.rs:218-219
self.total_tx_size = total_tx_size;   // = old_total + new_size  (pre-eviction)
self.total_tx_cycles = total_tx_cycles;
```

**Correct post-eviction value** should be `old_total − Σ(evicted_sizes) + new_size`; the actual stored value is `old_total + new_size`. The error equals `Σ(evicted_sizes)` and accumulates with every eviction-triggering insertion.

**Why existing checks do not prevent this:** `update_stat_for_remove_tx` has an underflow guard that recomputes totals from scratch, but that guard is only reached when subtraction underflows — it does not protect against the subsequent overwrite at lines 218–219 which happens unconditionally after all removals have already succeeded.

## Impact Explanation
`limit_size` (pool.rs:298) loops while `self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts entries. With an inflated `total_tx_size`, the pool believes it is over capacity when it is not, causing it to evict legitimate pending/proposed transactions. The error compounds with each eviction-triggering insertion. A malicious actor who repeatedly submits transactions that trigger the eviction path can drive `total_tx_size` arbitrarily above the actual pool contents, causing the node to continuously reject or evict valid user transactions. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, since the attacker can degrade any targeted node's tx-pool throughput with low-cost crafted transactions, preventing legitimate transactions from being relayed or confirmed.

## Likelihood Explanation
The trigger requires: (a) a chain of transactions in the pool whose ancestor count exceeds `max_ancestors_count` (default 25), where some ancestors are `cell_ref_parents` (they reference the same cell dep as the incoming transaction), and (b) the new transaction consumes that cell dep. An unprivileged attacker can construct this scenario deliberately by first seeding the pool with a 25-deep chain that shares a cell dep, then submitting a transaction that consumes it. No special privilege, leaked key, or victim mistake is required. The attack is repeatable and cheap.

## Recommendation
Move the `updated_stat_for_add_tx` call to after `check_and_record_ancestors` completes, so eviction decrements are already reflected in `self.total_tx_size`/`self.total_tx_cycles` before the snapshot is taken:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
// Recompute AFTER evictions have updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, replace lines 218–219 with direct increments (`self.total_tx_size += entry.size; self.total_tx_cycles += entry.cycles;`) so eviction decrements are preserved.

## Proof of Concept
1. Create a `PoolMap` with `max_ancestors_count = 25` and `max_tx_pool_size = 1000`.
2. Build a chain of 25 transactions `tx_1 … tx_25`, each spending the previous output, all referencing `cell_dep_X`. Add all 25 to the pool. `total_tx_size = 25 * S`.
3. Submit `tx_new` (size = `S`) that also references `cell_dep_X` as a cell dep, making `tx_1 … tx_25` its `cell_ref_parents`. `ancestors_count = 26 > 25`; the eviction branch fires.
   - `updated_stat_for_add_tx` snapshots `total_tx_size_local = 25*S + S = 26*S`.
   - `check_and_record_ancestors` evicts one or more `cell_ref_parents` via `remove_entry_and_descendants` → `update_stat_for_remove_tx`, decrementing `self.total_tx_size` by `k*S`.
   - Lines 218–219 overwrite: `self.total_tx_size = 26*S` (stale), erasing the `k*S` decrement.
4. Assert `pool.total_tx_size == 26*S` while actual pool contains `(25 - k + 1)` entries of size `S` each, i.e., actual size `= (26-k)*S`. The discrepancy is `k*S`.
5. Repeat step 3 to compound the inflation until `limit_size` begins evicting entries that legitimately fit.
