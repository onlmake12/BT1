Audit Report

## Title
Unbounded `last_n_numbers` via Low `difficulty_boundary` Bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The upfront limit check in `GetLastStateProofProcess::execute` bounds only `difficulties.len() + last_n_blocks * 2`, but the actual work is determined by `last_n_numbers`, whose size is controlled by the attacker-supplied `difficulty_boundary`. By setting `difficulty_boundary` to the total difficulty of block 1, an attacker forces `last_n_numbers = (1..N)` — N−1 entries — causing O(N log N) synchronous DB reads per single crafted message, completely bypassing `GET_LAST_STATE_PROOF_LIMIT = 1000`.

## Finding Description
**Root cause:** `last_n_numbers` is never bounded against `GET_LAST_STATE_PROOF_LIMIT`.

**Step 1 — Limit check passes trivially.**
The guard at L201–205 evaluates `difficulties.len() + last_n_blocks * 2`. With `difficulties=[]` and `last_n_blocks=1` this is `0 + 2 = 2`, well under `GET_LAST_STATE_PROOF_LIMIT = 1000`. [1](#0-0) [2](#0-1) 

**Step 2 — Else-branch is taken.**
With `start_block_number=0`, `last_n_blocks=1`, and N >> 1: `N - 0 <= 1` is false, so the else-branch executes. [3](#0-2) 

**Step 3 — Binary search resolves boundary to block 1.**
`get_first_block_total_difficulty_is_not_less_than(0, N, total_difficulty[1])` binary-searches and returns block number 1, since block 0's total difficulty is strictly less than block 1's. [4](#0-3) 

**Step 4 — Adjustment guard is skipped.**
The guard `last_block_number - difficulty_boundary_block_number < last_n_blocks` evaluates to `N - 1 < 1`, which is false for any N > 2. `difficulty_boundary_block_number` remains 1. [5](#0-4) 

**Step 5 — Unbounded vector collected.**
`last_n_numbers = (1..N)` yields N−1 entries with no size check applied anywhere before or after. [6](#0-5) 

**Step 6 — O(N log N) synchronous DB reads.**
`complete_headers` iterates every entry in `block_numbers` (which includes all of `last_n_numbers`). For each block it calls `self.snapshot.chain_root_mmr(*number - 1).get_root()`, which is O(log N) DB reads per block. [7](#0-6) 

## Impact Explanation
A single malicious peer can send one crafted `GetLastStateProof` message that forces the server to allocate an N−1 entry vector and perform O(N log N) synchronous DB reads before returning. On mainnet with millions of blocks, this exhausts memory and saturates I/O on the light-client server thread, effectively crashing or hanging the node. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation
The attack requires only two pieces of public information: the current tip block hash (observable from the P2P network) and block 1's total difficulty (a fixed public constant on mainnet). No privileged role, no key material, no PoW, and no victim mistake is required. Any peer can craft and send this message. The condition is deterministic and reproducible on every mainnet node running the light-client server.

## Recommendation
After computing `last_n_numbers` and before calling `complete_headers`, add a hard bound:

```rust
if last_n_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage
        .with_context("last_n_numbers exceeds limit");
}
```

This must be applied in both the if-branch (L296) and the else-branch (L318–319). Alternatively, clamp `difficulty_boundary_block_number` from below before collecting `last_n_numbers`:

```rust
let min_boundary = last_block_number
    .saturating_sub((constant::GET_LAST_STATE_PROOF_LIMIT as u64)
    .saturating_sub(last_n_blocks));
if difficulty_boundary_block_number < min_boundary {
    difficulty_boundary_block_number = min_boundary;
}
```

The fix must be applied **after** the difficulty boundary is resolved and **before** `last_n_numbers` is collected. [8](#0-7) 

## Proof of Concept
1. Observe the current tip hash `H_tip` and block number `N` from the P2P network.
2. Obtain `total_difficulty[1]` (public constant for mainnet block 1).
3. Send:
   ```
   GetLastStateProof {
       last_hash:           H_tip,
       last_n_blocks:       1,
       difficulty_boundary: total_difficulty[1],
       difficulties:        [],
       start_hash:          genesis_hash,
       start_number:        0,
   }
   ```
4. Server resolves `difficulty_boundary_block_number = 1`.
5. Guard `N - 1 < 1` is false → no adjustment.
6. `last_n_numbers = (1..N)`, length = N−1.
7. `complete_headers` iterates N−1 blocks, each performing O(log N) MMR DB reads.
8. Differential test: measure response latency for `difficulty_boundary=total_difficulty[1]` vs `difficulty_boundary=total_difficulty[N-1]`. The former scales as O(N log N); the latter is O(log N).
