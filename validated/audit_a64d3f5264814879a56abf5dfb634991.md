### Title
ClickHouse SQL Injection via Unsanitized `resolution` Parameter Interpolated Directly into Query String — (File: `apps/insights/src/services/clickhouse.ts`)

---

### Summary

The `getHistoricalPrices` function in `apps/insights/src/services/clickhouse.ts` directly interpolates the caller-supplied `resolution` parameter into a ClickHouse SQL query string using a JavaScript template literal. All other user-supplied values (`symbol`, `publisher`, `from`, `to`) are correctly passed through ClickHouse's parameterized `{name: Type}` binding mechanism, but `resolution` bypasses this entirely and is concatenated raw into the query. The TypeScript type `Resolution = \`${number} ${ResolutionUnit}\`` is a compile-time constraint only; it provides zero runtime enforcement. Any code path that reads `resolution` from an HTTP request and passes it to this function without independent runtime validation exposes the ClickHouse database to SQL injection.

---

### Finding Description

In `apps/insights/src/services/clickhouse.ts` at line 387, the `resolution` value is embedded directly into the query string:

```typescript
query: `
    SELECT toUnixTimestamp(toStartOfInterval(publishTime, INTERVAL ${resolution})) AS timestamp,
           avg(price) AS price, avg(confidence) AS confidence, status
    FROM prices
    PREWHERE
      cluster = {cluster: String}
      AND publisher = {publisher: String}
      AND symbol = {symbol: String}
      AND version = 2
    WHERE
      publishTime >= toDateTime({from: UInt32})
      AND publishTime < toDateTime({to: UInt32})
    GROUP BY timestamp, status
    ORDER BY timestamp ASC
  `,
```

Every other dynamic value uses ClickHouse's safe parameterized syntax (`{symbol: String}`, `{publisher: String}`, `{from: UInt32}`, `{to: UInt32}`). The `resolution` value is the sole exception — it is raw-interpolated via `${resolution}` directly into the `INTERVAL` clause.

The declared type is:

```typescript
type ResolutionUnit = "SECOND" | "MINUTE" | "HOUR" | "DAY";
type Resolution = `${number} ${ResolutionUnit}`;
```

TypeScript template literal types are erased at compile time. At runtime, `resolution` is a plain `string`. If the value originates from an HTTP query parameter (as it does in `apps/insights/src/app/historical-prices/route.ts`, which calls `getHistoricalPrices`), an attacker can supply an arbitrary string that is injected verbatim into the SQL sent to ClickHouse.

---

### Impact Explanation

An attacker who can reach the `/historical-prices` API route (which is a public Next.js route handler) can inject arbitrary ClickHouse SQL into the `INTERVAL` clause. Concrete impacts:

- **Information disclosure**: ClickHouse supports subqueries and `SELECT` in many positions; an attacker can exfiltrate data from other tables in the same database (e.g., raw publisher price data, internal analytics, credentials stored in ClickHouse system tables).
- **Resource exhaustion / DoS**: Injecting computationally expensive expressions (e.g., deeply nested aggregations, cross-joins against large tables) can saturate the ClickHouse server, degrading or denying service to all users of the analytics backend.
- **Error-based information disclosure**: Malformed injections will produce ClickHouse error messages that may leak schema details (table names, column names, data types), providing a roadmap for further exploitation — directly analogous to the `MongoServerError` leakage described in the reference report.

---

### Likelihood Explanation

The `/historical-prices` route is a public, unauthenticated Next.js API endpoint. No authentication or authorization is required to call it. The `resolution` parameter is a standard query parameter that any browser or HTTP client can supply. The attacker needs only to send a crafted HTTP GET request; no special privileges, keys, or on-chain access are required.

---

### Recommendation

1. **Validate `resolution` at runtime** before it reaches `getHistoricalPrices`. Use an explicit allowlist check:
   ```typescript
   const ALLOWED_RESOLUTIONS = new Set([
     "1 SECOND", "1 MINUTE", "5 MINUTE", "1 HOUR", "1 DAY"
   ]);
   if (!ALLOWED_RESOLUTIONS.has(resolution)) {
     return new Response("Invalid resolution", { status: 400 });
   }
   ```
2. **Never interpolate user-supplied values into SQL strings.** Use ClickHouse's parameterized binding for all dynamic values, or map the validated resolution to a hardcoded SQL fragment server-side.
3. **Audit all other ClickHouse query builders** in `apps/insights/src/services/clickhouse.ts` for similar raw interpolation patterns.

---

### Proof of Concept

```
GET /historical-prices?symbol=Crypto.BTC%2FUSD&from=1700000000&to=1700086400&resolution=1%20MINUTE%20)%20AS%20ts%2C%20(SELECT%20name%20FROM%20system.tables%20LIMIT%201)%20AS%20leaked%20--%20
```

The injected `resolution` value closes the `INTERVAL` expression and appends an additional `SELECT` subquery reading from `system.tables`, leaking internal schema information.
