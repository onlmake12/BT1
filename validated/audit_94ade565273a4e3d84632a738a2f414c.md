### Title
Unauthenticated `pending_delivered` Poisoning via Spoofed `ConnectionRequest.from` Enables Attacker-Directed Outbound TCP Connections — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

---

### Summary

The hole-punching protocol's `ConnectionRequestProcess` and `ConnectionSyncProcess` handlers never verify that the `from` field in a received message matches the actual session peer ID. A single connected attacker can therefore (1) poison `pending_delivered` with an arbitrary peer ID mapped to attacker-controlled addresses, then (2) immediately trigger NAT traversal to those addresses — all within one P2P session, with no victim cooperation required.

---

### Finding Description

**Root cause — no `from` ↔ session binding in `respond_delivered`:**

In `ConnectionRequestProcess::execute()`, when `self_peer_id == content.to`, the handler calls `respond_delivered(content.from, ...)` where `content.from` is taken verbatim from the wire message: [1](#0-0) 

Inside `respond_delivered`, the attacker-supplied `from_peer_id` and attacker-supplied `remote_listens` are written directly into the shared map: [2](#0-1) 

There is no check anywhere in this path that `from_peer_id == session_peer_id` (i.e., the `PeerId` authenticated by the TLS/Noise handshake for `self.peer`).

**Root cause — no `from` ↔ session binding in `ConnectionSyncProcess`:**

`ConnectionSyncProcess::execute()` similarly reads `content.from` from the wire and looks it up in `pending_delivered` without verifying it matches the sending session: [3](#0-2) 

The retrieved addresses are passed directly to `try_nat_traversal`: [4](#0-3) 

**`try_nat_traversal` makes real outbound TCP connections:**

The function opens a TCP socket and repeatedly attempts `connect()` to the attacker-supplied `SocketAddr` for up to 30 seconds (~150 SYN packets at 200 ms intervals): [5](#0-4) 

If the TCP handshake succeeds, the resulting stream is handed to `raw_session` as an inbound P2P session: [6](#0-5) 

**Complete self-contained two-message attack (no victim cooperation):**

```
Attacker → Node X:
  msg1: ConnectionRequest { from: <any_peer_id>, to: X, listen_addrs: [attacker_ip:port] }
        → respond_delivered → pending_delivered[any_peer_id] = ([attacker_ip:port], now)

  msg2: ConnectionSync { from: <any_peer_id>, to: X, route: [] }
        → pending_delivered.get(any_peer_id) → try_nat_traversal(bind_addr, attacker_ip:port)
        → repeated outbound TCP SYN to attacker_ip:port for 30 s
        → on accept: raw_session(stream, attacker_addr, RawSessionInfo::inbound(...))
```

**Rate-limiter bypass:** The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`. [7](#0-6) 
The attacker can rotate `from` (any random bytes that parse as a valid `PeerId`) to issue a fresh pair of messages for each target IP:port, bypassing the per-`(from,to)` limit entirely.

---

### Impact Explanation

1. **Port scanning / internal SSRF**: The attacker directs the victim node to make outbound TCP connections to arbitrary IP:port pairs, including RFC-1918 addresses unreachable from the public internet. This leaks reachability information (open vs. closed ports) via timing side-channels.
2. **Amplification**: Each two-message exchange triggers ~150 TCP SYN packets over 30 seconds. With the per-session rate limit of 30 msg/s, a single attacker session can sustain continuous scanning.
3. **Session hijacking / injection**: If the attacker's server accepts the TCP connection, the stream is promoted to a full inbound `raw_session`. The attacker thereby obtains an authenticated-looking P2P session with the victim node without going through the normal peer-ID-authenticated handshake path, potentially enabling protocol-level attacks (e.g., feeding malicious blocks/headers) from a session the node treats as inbound.

---

### Likelihood Explanation

- Requires only a single outbound TCP connection to any CKB node that has the hole-punching protocol enabled.
- No cryptographic material, no privileged role, no hashpower.
- The two messages are structurally valid and pass all existing validation checks.
- Locally testable against a single node in minutes.

---

### Recommendation

In `respond_delivered`, assert that `from_peer_id` equals the authenticated peer ID of the sending session before writing to `pending_delivered`:

```rust
// In ConnectionRequestProcess::execute / respond_delivered:
let session_peer_id = self.protocol.network_state
    .peer_registry.read()
    .get_peer(self.peer)
    .and_then(|p| p.connected_addr.peer_id());  // or equivalent

if session_peer_id.as_ref() != Some(&from_peer_id) {
    return StatusCode::InvalidFromPeerId
        .with_context("from peer id does not match session peer id");
}
```

Apply the same check in `ConnectionSyncProcess::execute` — verify `content.from` matches the sending session's authenticated peer ID before consulting `pending_delivered`.

---

### Proof of Concept

```
State assertion: pending_delivered[victim_id] must only be populated
when the session that sent the ConnectionRequest has session_peer_id == victim_id.

Test:
  1. Start node X with hole-punching enabled.
  2. Connect attacker session A (peer_id = attacker_id) to X.
  3. Send ConnectionRequest { from: victim_id, to: X, listen_addrs: [attacker_ip:1234] }.
  4. Assert: pending_delivered[victim_id] == ([attacker_ip:1234], _).   ← FAILS invariant
  5. Send ConnectionSync { from: victim_id, to: X, route: [] }.
  6. Observe: outbound TCP SYN packets to attacker_ip:1234 from node X.
  7. Accept the TCP connection on attacker_ip:1234.
  8. Observe: raw_session promoted on node X with RawSessionInfo::inbound.
```
