Audit Report

## Title
Min-Fee-Rate Invariant Broken by Size-Only Weight Assumption in `check_tx_fee` — (File: tx-pool/src/util.rs)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using only the raw serialized byte size of a transaction as the weight, rather than the actual transaction weight (`max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`). Because no weight-based fee rate gate exists after cycles are determined by `verify_rtx`, an unprivileged RPC caller can submit cycle-heavy, byte-small transactions that are admitted to the pool at effective fee rates far below the configured `min_fee_rate`, bypassing the pool's anti-spam invariant and consuming verification resources cheaply.

## Finding Description

`get_transaction_weight` in `util/types/src/core/tx_pool.rs` defines the correct weight as `max(tx_size_bytes, cycles × DEFAULT_BYTES_PER_CYCLES)` where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`.

`check_tx_fee` in `tx-pool/src/util.rs` (L45) computes `min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64)`, using only `tx_size` as the weight. The code comment at L42–44 explicitly acknowledges this is theoretically incorrect, treating it as an acceptable "cheap check."

The admission flow in `_process_tx` (`tx-pool/src/process.rs` L705–777):
1. `pre_check` (L715) → calls `check_tx_fee` with size only (cycles unknown)
2. `verify_rtx` (L724–734) → actual cycles determined
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` (L751) → entry created with real cycles
4. `submit_entry` (L753) → **no subsequent weight-based fee rate check**

`TxEntry::fee_rate()` in `tx-pool/src/component/entry.rs` (L114–118) correctly uses `get_transaction_weight(self.size, self.cycles)`, but this method is only used for pool ordering and eviction after admission — never as an admission gate.

For a transaction with `cycles = 70,000,000` and serialized size of 100 bytes:
- Actual weight: `max(100, 70,000,000 × 0.000_170_571_4) ≈ 11,940 bytes`
- `check_tx_fee` computes `min_fee = 1000 × 100 / 1000 = 100 shannons`
- Actual fee rate if 100 shannons paid: `100 × 1000 / 11,940 ≈ 8 shannons/KW` — 125× below the configured minimum of 1000 shannons/KW

The transaction is admitted. Since eviction uses actual weight-based fee rate, these transactions are evicted first when the pool is full, but this does not prevent the admission bypass or the CPU consumption from `verify_rtx`.

## Impact Explanation

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. An attacker can continuously submit cycle-heavy, byte-small transactions via `send_transaction` RPC that are admitted at fee rates far below `min_fee_rate`. Each admitted transaction consumes up to 70M cycles of verification CPU resources while paying a fraction of the required fee, undermining the anti-spam fee floor and saturating the node's verification pipeline at negligible cost.

## Likelihood Explanation

- **Entry point**: `send_transaction` RPC, reachable by any unprivileged caller with no keys or special roles.
- **Craft cost**: attacker writes a CKB-VM script (e.g., a tight loop) consuming ~70M cycles, stored in a cell dep to keep the transaction body small (~100–200 bytes serialized).
- **No consensus violation**: the transaction is valid; it just pays less than the weight-based minimum.
- **Repeatability**: the attacker can submit many such transactions continuously, each consuming significant verification CPU before being evicted from the pool.

## Recommendation

After `verify_rtx` returns the actual cycle count, perform a second weight-based fee rate check before calling `submit_entry`:

```rust
// After verify_rtx returns `verified` in _process_tx:
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Some((Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64())), snapshot));
}
```

Alternatively, refactor `check_tx_fee` to accept an optional `cycles` parameter and use `get_transaction_weight(tx_size, cycles)` when cycles are available. The size-only pre-check can remain as a necessary approximation for the pre-execution path, but a mandatory post-execution weight-based check must be added.

## Proof of Concept

1. Write a CKB-VM lock script that executes a tight loop consuming ~70,000,000 cycles. Store it in a cell dep to keep the transaction body small (~100 bytes serialized).
2. Construct a transaction spending a cell locked by this script, with `outputs_capacity = inputs_capacity - 100` (fee = 100 shannons).
3. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000`.
4. `check_tx_fee` (L45 of `tx-pool/src/util.rs`) computes `min_fee = 1000 × 100 / 1000 = 100 shannons`. Fee equals min_fee → **admitted**.
5. `verify_rtx` executes the script, consuming ~70M cycles of CPU.
6. `_process_tx` (L751–753 of `tx-pool/src/process.rs`) creates the entry and calls `submit_entry` with no weight-based fee check.
7. Actual fee rate = `100 × 1000 / 11,940 ≈ 8 shannons/KW` — 125× below the configured minimum.
8. Repeat to continuously consume the node's verification CPU at negligible fee cost.
