### Title
Missing Underflow Guard in `ema_price_no_older_than` Causes Revert When `publish_time` Exceeds Current Time — (File: `target_chains/fuel/contracts/pyth-contract/src/main.sw`)

### Summary

The private function `ema_price_no_older_than` in the Fuel Pyth contract performs an unchecked unsigned subtraction `current_time - price.publish_time`. When a stored EMA price has a `publish_time` slightly ahead of the Fuel blockchain's `current_time` (a realistic condition due to clock skew between Pythnet and Fuel), this subtraction underflows and causes the transaction to revert. The sibling function `price_no_older_than` already contains an explicit comment and conditional guard for exactly this scenario, but `ema_price_no_older_than` was never updated to match.

### Finding Description

In `target_chains/fuel/contracts/pyth-contract/src/main.sw`, two analogous functions exist for staleness checking:

**`ema_price_no_older_than` (vulnerable):**
```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,  // ← unchecked u64 subtraction
        PythError::OutdatedPrice,
    );
    price
}
``` [1](#0-0) 

**`price_no_older_than` (safe — has the guard):**
```sway
fn price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
``` [2](#0-1) 

The comment `// Mimicking saturating subtraction to avoid underflow` in `price_no_older_than` explicitly acknowledges the underflow risk and applies a conditional guard. `ema_price_no_older_than` was never given the same treatment.

In Sway, arithmetic on unsigned integers reverts on underflow (analogous to Solidity ≥0.8 checked arithmetic). When `price.publish_time > current_time`, the expression `current_time - price.publish_time` underflows and the entire call reverts with a panic rather than the expected `PythError::OutdatedPrice`.

### Impact Explanation

Any user or contract calling the public `get_ema_price_no_older_than` entry point when the stored EMA price has a `publish_time` even one second ahead of the Fuel chain's `block.timestamp` will receive an unexpected revert. This:

- Blocks all consumers of EMA prices (e.g., DeFi protocols on Fuel using EMA for TWAP-style pricing) from reading prices during the affected window.
- Is a silent DoS: the caller receives a generic arithmetic panic rather than `PythError::OutdatedPrice`, making it harder to diagnose.
- Does not affect `get_price_no_older_than`, creating an asymmetric failure mode between spot and EMA price queries.

### Likelihood Explanation

Pythnet validators and Fuel validators run independent clocks. Price attestations are signed on Pythnet and relayed to Fuel; the `publish_time` embedded in the attestation reflects Pythnet's clock. A modest clock skew of even 1–2 seconds between Pythnet and Fuel is sufficient to trigger the underflow. This is a normal operating condition, not an adversarial one. The fact that the developers already added the guard to `price_no_older_than` confirms they are aware of the clock-skew scenario; the omission in `ema_price_no_older_than` is an oversight.

### Recommendation

Apply the same saturating-subtraction pattern already present in `price_no_older_than`:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow (same as price_no_older_than)
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

### Proof of Concept

```
Scenario:
  Pythnet clock:  T = 1_000_002  (publish_time stored in EMA price feed)
  Fuel block.timestamp: T = 1_000_000  (2 seconds behind Pythnet)
  time_period: 60 seconds

Call: get_ema_price_no_older_than(60, price_feed_id)

Execution:
  current_time      = 1_000_000  (u64)
  price.publish_time = 1_000_002  (u64)
  current_time - price.publish_time
    = 1_000_000 - 1_000_002
    = underflow → revert (Sway panics on u64 underflow)

Expected behavior (as in price_no_older_than):
  time_difference = 0  (clamped)
  0 <= 60  → price returned successfully

Contrast: get_price_no_older_than(60, price_feed_id) succeeds for the same inputs
          because it uses the conditional guard.
```
