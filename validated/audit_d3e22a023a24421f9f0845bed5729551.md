Audit Report

## Title
Insufficient RBF Fee Validation Before Expensive Script Execution Enables CPU Exhaustion - (File: `tx-pool/src/process.rs`)

## Summary
In `_process_tx`, the RBF branch of `pre_check` only validates `fee >= min_fee_rate` before returning `Ok`, allowing the transaction to proceed to full `ContextualTransactionVerifier` script execution. The actual RBF fee adequacy check (`min_replace_fee`) is deferred to `submit_entry` → `check_rbf` under write lock, which runs only after script verification completes. An attacker can repeatedly submit low-fee RBF candidates that pass `pre_check`, consume full CKB-VM cycles, and are then rejected by `check_rbf`, at negligible cost.

## Finding Description
In `_process_tx` (`process.rs`), the execution order is:
1. `pre_check` (read lock) — L715: validates only `fee >= min_fee_rate` for RBF path
2. `verify_rtx` (full script execution) — L724–732: runs `ContextualTransactionVerifier` including `ScriptVerifier`
3. `submit_entry` → `check_rbf` (write lock) — L753/L106: enforces actual RBF rules

In the `Err(Reject::Resolve(OutPointError::Dead(out)))` branch of `pre_check` (L292–309), after `find_conflict_outpoint` confirms a conflict exists, the function returns `Ok(...)` immediately. `check_tx_fee` (`util.rs` L45–52) only checks `fee >= min_fee_rate`. The `check_rbf` function (`pool.rs` L574–678) enforces three additional rules that are categorically stricter:
- **Rule #2** (L596–609): no new unconfirmed inputs
- **Rule #5** (L611–645): ≤100 replacement candidates
- **Rule #3/#4** (L664–676): `fee >= calculate_min_replace_fee(all_conflicted, tx_size)`

A transaction paying just above `min_fee_rate` (e.g., 363 shannons) but far below `min_replace_fee` (e.g., 10 CKB + extra) passes `pre_check`, triggers full VM execution bounded by `max_tx_verify_cycles`, and is then rejected by `check_rbf`. The node bears the full CPU cost; the attacker pays only the RPC/P2P submission cost.

## Impact Explanation
**High severity** — matches "Vulnerabilities or bad designs which could cause CKB network congestion with few costs." The attacker forces the node to execute CKB-VM scripts (up to `max_block_cycles` per transaction) for transactions guaranteed to be rejected. Sustained submission saturates the verification worker pool, degrading or halting legitimate transaction processing on the targeted node.

## Likelihood Explanation
- RBF must be enabled (`min_rbf_rate > min_fee_rate`), a node operator configuration choice, not attacker-controlled.
- The attacker only needs to observe a transaction in the pool, publicly visible via `get_raw_tx_pool` RPC or P2P relay.
- No privileged access is required — any RPC caller or P2P peer can submit transactions.
- The attack is repeatable: each iteration uses a fresh `tx_B` variant (different witness bytes) to change `witness_hash` and bypass `txs_verify_cache`.
- The verify queue size limit (`Reject::Full`) provides partial rate-limiting but does not prevent the attack — the attacker waits for the queue to drain and resubmits.

## Recommendation
Move the core RBF fee adequacy check (`min_replace_fee` comparison) and Rule #2 (no new unconfirmed inputs) into `pre_check` under the read lock, before the transaction is enqueued for script verification. In the `Err(Reject::Resolve(OutPointError::Dead(out)))` branch of `pre_check`, after `find_conflict_outpoint` confirms a conflict exists, compute the full set of conflicting entries and verify `fee >= calculate_min_replace_fee(conflicts, tx_size)` before returning `Ok`. These checks require only a consistent snapshot read and do not need write-lock atomicity. The final conflict resolution and pool mutation must remain in `submit_entry` under write lock.

## Proof of Concept
1. Submit `tx_A` spending `cell_X` with fee = 10 CKB. It enters the pending pool.
2. Observe `tx_A` via `get_raw_tx_pool` RPC.
3. Craft `tx_B` spending the same `cell_X`, with fee = `min_fee_rate * size` (e.g., 363 shannons — far below `min_replace_fee` of ~10 CKB + extra).
4. Submit `tx_B` via `send_transaction` RPC.
5. **`pre_check`**: `resolve_tx(..., rbf=true)` succeeds; `check_tx_fee` passes (363 ≥ min_fee); `find_conflict_outpoint` finds `tx_A` → returns `Ok`.
6. **`verify_rtx`**: Full `ContextualTransactionVerifier` runs CKB-VM script execution on `tx_B` (consuming up to `max_tx_verify_cycles`).
7. **`submit_entry` → `check_rbf`**: Rule #3/#4: `363 < 10_CKB + extra` → `Err(RBFRejected)`.
8. Node wasted full VM cycles. Repeat steps 3–8 with fresh `tx_B` variants (vary witness bytes to change `witness_hash` and bypass cache) until the verify queue is saturated, then wait and repeat.
