### Title
`Scheduler.getPricesNoOlderThan()` Validates Subscription-Level MAX Timestamp Instead of Per-Feed Timestamps, Enabling Stale-Price Consumption — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getPricesNoOlderThan()` and `getEmaPricesNoOlderThan()` are documented as the safe, freshness-checked read APIs for the Pyth Pulse (Scheduler) contract. However, the freshness check inside both functions validates `status.priceLastUpdatedAt` — a subscription-level field that stores the **maximum** `publishTime` across all feeds in the last update batch — rather than the `publishTime` of the individual feeds being requested. In subscriptions that mix active-market and closed-market feeds, the active feed's recent timestamp causes the check to pass while the returned closed-market feed price may be arbitrarily stale.

---

### Finding Description

`Scheduler.updatePriceFeeds()` computes `priceLastUpdatedAt` as the maximum `publishTime` across all feeds in the batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
// ...
status.priceLastUpdatedAt = latestPublishTime;
``` [1](#0-0) [2](#0-1) 

This design is intentional: the README explicitly states that closed-market feeds carry a `publishTime` from their last trading period, so the MAX is used to avoid rejecting valid batches. [3](#0-2) 

`getPricesNoOlderThan()` then uses this subscription-level MAX to gate the freshness check:

```solidity
function getPricesNoOlderThan(
    uint256 subscriptionId,
    bytes32[] calldata priceIds,
    uint256 age_seconds
) external view override onlyWhitelistedReader(subscriptionId)
    returns (PythStructs.Price[] memory prices)
{
    SchedulerStructs.SubscriptionStatus memory status = _state
        .subscriptionStatuses[subscriptionId];

    if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
        revert PythErrors.StalePrice();

    prices = this.getPricesUnsafe(subscriptionId, priceIds);
}
``` [4](#0-3) 

The check `distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds` passes as long as **any** feed in the subscription was recently updated. The subsequently returned prices for the **requested** `priceIds` are fetched via `getPricesUnsafe`, which performs no per-feed timestamp validation:

```solidity
function getPricesUnsafe(...) external view override ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        prices[i] = priceFeeds[i].price;
    }
    return prices;
}
``` [5](#0-4) 

`getEmaPricesNoOlderThan()` has the identical flaw, using the same `status.priceLastUpdatedAt` check before delegating to `getEmaPricesUnsafe`. [6](#0-5) 

The `IScheduler` interface documents `getPricesNoOlderThan` as the safe alternative to `getPricesUnsafe`, creating a false guarantee: [7](#0-6)

### Impact Explanation

A DeFi protocol (lending, derivatives, AMM) that subscribes to a mixed feed set — e.g., `[ETH/USD (active), XAU/USD (closed market)]` — and calls `getPricesNoOlderThan(subscriptionId, [XAU_USD_ID], 60)` will:

1. Pass the freshness check because `priceLastUpdatedAt` reflects ETH/USD's recent timestamp.
2. Receive a `XAU/USD` price that is hours or days old (from the last gold trading session).
3. Use this stale price in financial operations (collateral valuation, liquidation thresholds, swap rates).

An attacker who observes that the on-chain gold price lags the real market can time mint/borrow/swap operations against the stale rate, extracting value from the protocol — the exact stale-rate arbitrage described in the Mantle report.

---

### Likelihood Explanation

- Mixed active/closed-market subscriptions are explicitly supported and documented by the Scheduler design.
- The Scheduler README and `_validateShouldUpdatePrices` comments confirm closed-market feeds are a first-class use case.
- Any whitelisted reader (or any reader if `readerWhitelist` is empty) can call `getPricesNoOlderThan` — no privileged access required.
- The function is the recommended "safe" API, so integrators will use it without adding their own per-feed timestamp checks.

---

### Recommendation

Replace the subscription-level `priceLastUpdatedAt` check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with per-feed timestamp validation against the individual `publishTime` of each requested price feed:

```solidity
for (uint i = 0; i < priceFeeds.length; i++) {
    if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
        revert PythErrors.StalePrice();
}
```

Alternatively, document clearly that `getPricesNoOlderThan` guarantees freshness only at the subscription batch level (not per-feed), and rename or deprecate it to avoid misleading integrators.

---

### Proof of Concept

1. Create a subscription with two feeds: `ETH/USD` (active, updates every heartbeat) and `XAU/USD` (closed market, last `publishTime` = 8 hours ago).
2. A keeper calls `updatePriceFeeds` with a fresh ETH/USD price and a stale XAU/USD price (same Pythnet slot). `priceLastUpdatedAt` is set to ETH/USD's `publishTime` (now).
3. A reader calls `getPricesNoOlderThan(subscriptionId, [XAU_USD_ID], 60)`.
4. The check `distance(block.timestamp, priceLastUpdatedAt) > 60` evaluates to `distance(now, now) = 0 > 60` → **false** → no revert.
5. `getPricesUnsafe` returns XAU/USD with `publishTime` = 8 hours ago.
6. The caller receives a price it believes is ≤60 seconds old, but it is actually 8 hours stale.
7. An attacker exploits the delta between the stale on-chain gold price and the real market price to extract value from any DeFi protocol consuming this feed.
