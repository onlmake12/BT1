### Title
Selective Abort via Intentional Callback Revert in `revealWithCallback` Exposes Random Number Before Application State Update — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `CALLBACK_FAILED` recovery flow in `Entropy.sol`'s `revealWithCallback()` emits the final random number in on-chain events **before** the requester's application state is updated. A malicious requester contract can intentionally revert its `entropyCallback` upon receiving an unfavorable random number, observe the result from the emitted event, and selectively choose to only retry `revealWithCallback` when the outcome is favorable — enabling a selective-abort attack on any application built on Pyth Entropy.

---

### Finding Description

`revealWithCallback()` implements a two-phase callback failure recovery flow. When a request has `gasLimit10k != 0` and `callbackStatus == CALLBACK_NOT_STARTED`, the callback is invoked via `excessivelySafeCall`, which catches reverts without reverting the outer transaction. [1](#0-0) 

If the callback reverts and sufficient gas was provided, the random number is emitted in both the `CallbackFailed` event and the `EntropyEventsV2.Revealed` event, and the request transitions to `CALLBACK_FAILED` state — **without clearing the request**. [2](#0-1) 

The `CALLBACK_FAILED` state is explicitly allowed as a valid entry point for a subsequent `revealWithCallback` call: [3](#0-2) 

In the retry path (`CALLBACK_FAILED`), the code falls into the `else` branch (not the `excessivelySafeCall` branch), calling the callback directly with no revert protection: [4](#0-3) 

**Attack path:**

1. Malicious contract calls `requestV2(provider, userRandomNumber, gasLimit)` with a non-zero `gasLimit`.
2. Provider calls `revealWithCallback`. The malicious `entropyCallback` receives `randomNumber` as a parameter, checks if the outcome is favorable, and **intentionally reverts** if not.
3. The outer transaction succeeds (due to `excessivelySafeCall`). The `CallbackFailed` and `EntropyEventsV2.Revealed` events are emitted, publishing `randomNumber` on-chain.
4. The attacker observes the random number from the event.
5. **If favorable**: the attacker updates their contract to accept the callback and calls `revealWithCallback` again — the application state is now updated with a favorable outcome.
6. **If unfavorable**: the attacker never retries. The request remains in `CALLBACK_FAILED` state indefinitely. The application state is never updated.

The `EntropyStatusConstants` confirm the four states and that `CALLBACK_FAILED` is a persistent, non-expiring state: [5](#0-4) 

---

### Impact Explanation

Any on-chain application using Pyth Entropy's `requestV2`/`revealWithCallback` flow (e.g., lotteries, NFT mints, on-chain games) is vulnerable to selective abort. A malicious requester contract can:

- Observe the random number before committing to its use in application logic.
- Selectively accept only favorable outcomes (e.g., winning lottery draws, rare NFT traits).
- Abandon unfavorable outcomes at the cost of the request fee only.

The attacker does not recover the fee, but the ability to selectively abort unfavorable random outcomes breaks the fairness guarantee of any application relying on Entropy for unpredictable outcomes. **Impact: Medium** (breaks application-layer fairness; attacker pays fee per attempt but can farm favorable outcomes).

---

### Likelihood Explanation

The attack requires deploying a malicious requester contract with a conditional-revert `entropyCallback`. This is straightforward to implement and requires no privileged access — any unprivileged user can deploy such a contract and call `requestV2`. The attack is repeatable at the cost of the request fee per attempt. **Likelihood: Medium** (requires on-chain contract deployment and fee payment per attempt, but is fully permissionless and automatable).

---

### Recommendation

1. **Do not emit the random number in `CallbackFailed` events.** Emitting the random number before the application state is updated is the root enabler of the peek-and-abort pattern. Store the committed random number in request storage after the first reveal, and only emit it upon successful callback completion.
2. **Alternatively**, introduce a penalty or time-lock for `CALLBACK_FAILED` requests: e.g., require a waiting period before retry, or slash a portion of the fee on failure, raising the cost of selective abort.
3. **Document the risk** clearly in the Entropy SDK so application developers are aware that `entropyCallback` must not be used as the sole guard for application state transitions without additional commit-reveal protections.

### Proof of Concept

```solidity
contract MaliciousRequester is IEntropyConsumer {
    IEntropyV2 entropy;
    uint64 public pendingSeq;
    bytes32 public knownRandomNumber;
    bool public acceptCallback;

    constructor(address _entropy) { entropy = IEntropyV2(_entropy); }

    function requestEntropy(bytes32 userRandom, uint32 gasLimit) external payable {
        uint256 fee = entropy.getFeeV2(gasLimit);
        pendingSeq = entropy.requestV2{value: fee}(
            entropy.getDefaultProvider(), userRandom, gasLimit
        );
    }

    // Step 1: Provider calls revealWithCallback → this callback is invoked via excessivelySafeCall
    // The random number is passed as a parameter. We revert if unfavorable.
    function entropyCallback(
        uint64 /*seq*/, address /*provider*/, bytes32 randomNumber
    ) internal override {
        if (!acceptCallback) {
            // Peek at the random number, then revert.
            // The outer tx still succeeds; CallbackFailed event emits randomNumber.
            revert("unfavorable outcome, aborting");
        }
        // Step 3: If we decided to accept, process the favorable outcome.
        knownRandomNumber = randomNumber;
    }

    // Step 2: Attacker observes randomNumber from CallbackFailed event off-chain.
    // If favorable, call this to accept it.
    function acceptAndRetry(
        address provider, uint64 seq, bytes32 userContrib, bytes32 providerContrib
    ) external {
        acceptCallback = true;
        // revealWithCallback now calls entropyCallback directly (CALLBACK_FAILED path)
        entropy.revealWithCallback(provider, seq, userContrib, providerContrib);
    }

    function getEntropy() internal view override returns (address) { return address(entropy); }
}

library EntropyStatusConstants {
    // Status values for Request.status //
    // not a request with callback
    uint8 public constant CALLBACK_NOT_NECESSARY = 0;
    // A request with callback where the callback hasn't been invoked yet.
    uint8 public constant CALLBACK_NOT_STARTED = 1;
    // A request with callback where the callback is currently in flight (this state is a reentry guard).
    uint8 public constant CALLBACK_IN_PROGRESS = 2;
    // A request with callback where the callback has been invoked and failed.
    uint8 public constant CALLBACK_FAILED = 3;
}
