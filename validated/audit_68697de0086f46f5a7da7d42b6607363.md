Looking at the full execution path and comparing with other protocol handlers in the codebase.

### Title
Missing Per-Peer Rate Limit on `GetBlocksProof` Allows Unbounded I/O and CPU Amplification — (`util/light-client-protocol-server/src/lib.rs`, `src/components/get_blocks_proof.rs`)

---

### Summary

`LightClientProtocol` has no rate limiter. Any unprivileged peer can send back-to-back `GetBlocksProof` messages each carrying up to 1000 valid main-chain block hashes, forcing the server to execute up to 3000 RocksDB reads plus a 1000-leaf MMR proof generation per message, with no throttle of any kind.

---

### Finding Description

`LightClientProtocol` is defined as:

```rust
pub struct LightClientProtocol {
    pub shared: Shared,
}
``` [1](#0-0) 

There is no `rate_limiter` field. The `try_process` dispatcher routes `GetBlocksProof` directly to `GetBlocksProofProcess::execute()` with zero per-peer or per-message-type throttling: [2](#0-1) 

Inside `execute()`, the only size guard is a strict upper-bound check (`> 1000`), so exactly 1000 hashes are accepted: [3](#0-2) 

For every hash that resolves to a main-chain block, the handler issues three synchronous store reads: [4](#0-3) 

Then `reply_proof` calls `mmr.gen_proof(items_positions)` with up to 1000 positions — an O(N log N) MMR traversal over the full chain MMR: [5](#0-4) 

The constant confirming the limit is 1000: [6](#0-5) 

**Contrast with the `Relayer` protocol**, which carries an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` capped at 30 req/s per peer per message type, checked before any processing: [7](#0-6) 

The `LightClientProtocol` has no equivalent field or check anywhere in its source tree (confirmed: zero matches for `rate_limiter` / `RateLimiter` under `util/light-client-protocol-server/`).

---

### Impact Explanation

Per request a peer can force:
- 1000 × `is_main_chain()` index lookups (partition step)
- 1000 × `get_block_header()` DB reads
- 1000 × `get_block_uncles()` DB reads
- 1000 × `get_block_extension()` DB reads
- 1 × `mmr.gen_proof(1000 positions)` — O(N log N) over the full chain MMR

The message payload is ~32 KB (1000 × 32-byte hashes). The server-side work is orders of magnitude larger. A single persistent peer sending requests in a tight loop can saturate RocksDB I/O and CPU, degrading or halting block processing and sync for all other peers.

---

### Likelihood Explanation

The attacker needs only a standard P2P connection to a node that has the light-client protocol enabled. All 1000 block hashes are public information (any block explorer). No PoW, no key, no privileged role is required. The exploit is trivially scriptable.

---

### Recommendation

Add a per-peer, per-message-type rate limiter to `LightClientProtocol` mirroring the pattern already used in `Relayer`:

```rust
pub struct LightClientProtocol {
    pub shared: Shared,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

Check it at the top of `try_process` before dispatching to any handler. A limit of 1–5 `GetBlocksProof` requests per second per peer is sufficient for legitimate light-client use.

---

### Proof of Concept

1. Connect to a CKB full node with the light-client protocol enabled.
2. Collect 1000 distinct main-chain block hashes (e.g., blocks 1–1000).
3. Obtain any valid tip hash for `last_hash`.
4. In a tight loop, send `GetBlocksProof { last_hash, block_hashes: [h1..h1000] }`.
5. Observe: CPU and RocksDB I/O spike proportionally; block-processing latency increases; legitimate peers experience degraded sync.

    /// Init relay protocol handle

    /// This is a runtime relay protocol shared state, and any relay messages will be processed and forwarded by it

        // setup a rate limiter keyed by peer and message type that lets through 30 requests per second

        let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());

        Relayer {
            chain,
            shared,

        }
    }

    pub fn shared(&self) -> &Arc<SyncShared> {
