Audit Report

## Title
Unbounded Vec Allocation via Uncapped `max_bucket_index_by_fee_rate` in `do_estimate` Enables Node Crash - (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary

`WeightUnitsFlow::do_estimate` derives `max_bucket_index` directly from the highest fee rate in the live mempool snapshot and allocates two `Vec<u64>` of that size with no upper-bound guard. Because `check_tx_fee` enforces only a minimum fee rate and `max_bucket_index_by_fee_rate` has no ceiling, an attacker who places a single high-fee transaction in the pool and then calls the unauthenticated `estimate_fee_rate` RPC can force the node to attempt a multi-gigabyte heap allocation, crashing the process via OOM abort.

## Finding Description

`do_estimate` reads `max_fee_rate` from the first element of `sorted_current_txs` (the live mempool, sorted descending by fee rate) and passes it directly to `max_bucket_index_by_fee_rate` with no cap: [1](#0-0) 

`max_bucket_index_by_fee_rate` has no ceiling; for any fee rate above 2,000,000 shannons/KW it returns `(x + 11_500_000) / 100_000`, growing linearly without bound: [2](#0-1) 

Two `Vec<u64>` are then allocated with length `max_bucket_index + 1`, with no guard: [3](#0-2) [4](#0-3) 

For a transaction with fee rate of 10^15 shannons/KW: `max_bucket_index ≈ 10^10`, and each `vec![0u64; 10^10 + 1]` requires ~80 GB, totaling ~160 GB across both allocations.

`check_tx_fee` enforces only a minimum fee rate; there is no maximum: [5](#0-4) 

`TxPoolConfig` has no `max_fee_rate` field, confirming no admission-time ceiling exists: [6](#0-5) 

The `estimate_fee_rate` RPC is unauthenticated and publicly callable: [7](#0-6) 

Call chain: `estimate_fee_rate` RPC → `ExperimentRpcImpl::estimate_fee_rate` → `TxPoolService::estimate_fee_rate` → `FeeEstimator::estimate_fee_rate` → `WeightUnitsFlow::do_estimate` → `vec![0u64; max_bucket_index + 1]` → OOM abort.

## Impact Explanation

When the allocation fails, Rust's global allocator calls `handle_alloc_error`, which aborts the process. On Linux the OOM killer terminates the node before the allocation completes. Either path crashes the node process. This matches the allowed impact: **High — Vulnerability which could easily crash a CKB node.**

## Likelihood Explanation

- The `estimate_fee_rate` RPC requires no authentication. While the default config binds to `127.0.0.1:8114`, infrastructure operators (exchanges, explorers, public RPC providers) commonly expose this port externally, and any process on the same host can reach it regardless.
- The tx-pool admission gate has no upper fee-rate bound, so any valid UTXO with sufficient capacity can produce an arbitrarily high fee rate.
- The crash is repeatable: the attacker's UTXO is never spent (the transaction is never mined because the node crashes before mining), so the attacker resubmits the same transaction after each restart and calls the RPC again at zero marginal cost.
- Smaller allocations (e.g., 100,000 CKB fee) still produce indices exceeding 10^9, sufficient to exhaust memory on typical servers with 16–32 GB RAM.

## Recommendation

1. **Cap `max_bucket_index`** before any allocation in `do_estimate`:
   ```rust
   const MAX_BUCKET_INDEX: usize = 200; // matches the algorithm's designed range
   let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
       .min(MAX_BUCKET_INDEX);
   ```

2. **Add overflow safety** in `max_bucket_index_by_fee_rate` using `checked_add`, returning a capped value on overflow:
   ```rust
   x => x.checked_add(t * 11_500)
         .map(|v| v / (100 * t))
         .unwrap_or(MAX_BUCKET_INDEX as u64),
   ```

3. **Optionally clamp fee rate at pool admission** to a protocol-defined maximum to prevent other future misuse of unbounded fee rates.

## Proof of Concept

```bash
# 1. Attacker submits one transaction:
#    input capacity  = 1_000_000 CKB = 10^14 shannons (attacker's UTXO)
#    output capacity = 61 shannons (minimum cell)
#    fee             ≈ 10^14 shannons
#    serialized size ≈ 100 bytes
#    fee_rate        = 10^14 * 1000 / 100 = 10^15 shannons/KW
#    max_bucket_index ≈ 10^10
#    vec![0u64; 10^10 + 1] × 2 ≈ 160 GB → OOM / process killed

# 2. Attacker calls estimate_fee_rate:
curl -X POST http://localhost:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"estimate_fee_rate","params":[],"id":1}'
# Node crashes.

# 3. Node restarts; attacker's UTXO is still unspent (tx was never mined).
# 4. Repeat from step 1 indefinitely.
```

    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }
