### Title
Permissionless `execute_posted_vaa` with Non-Contiguous Sequence Check Allows Permanent Skipping of Governance VAAs — (`governance/remote_executor/programs/remote-executor/src/lib.rs`)

---

### Summary

The `execute_posted_vaa` instruction in the remote executor enforces only monotonic increase (`posted_vaa.sequence > claim_record.sequence`) rather than strict contiguity (`== claim_record.sequence + 1`). Because the instruction is fully permissionless (any account may be the payer/caller), an unprivileged attacker can relay a higher-sequence governance VAA first, permanently rendering all lower-sequence VAAs unexecutable.

---

### Finding Description

The sequence guard in `ExecutePostedVaa` is:

```
constraint = posted_vaa.sequence > claim_record.sequence
``` [1](#0-0) 

After passing the constraint, the record is updated unconditionally:

```rust
claim_record.sequence = posted_vaa.sequence;
``` [2](#0-1) 

`ClaimRecord` is initialized via `init_if_needed` with the Rust `Default` for `u64`, which is `0`. [3](#0-2) 

There is no caller restriction — the only required signer is an arbitrary `payer`: [4](#0-3) 

Wormhole-signed VAAs are publicly accessible via the Wormhole Guardian API. The legitimate relayer (`crank_pythnet_relayer`) fetches and executes them in ascending order: [5](#0-4) 

**Concrete attack sequence:**

| Step | Action | `ClaimRecord.sequence` |
|------|--------|----------------------|
| Initial state | — | 0 |
| Attacker submits VAA seq=12 | `12 > 0` → passes, executes | 12 |
| Legitimate relayer submits VAA seq=10 | `10 > 12` → `NonIncreasingSequence` | 12 (unchanged) |
| Legitimate relayer submits VAA seq=11 | `11 > 12` → `NonIncreasingSequence` | 12 (unchanged) |

Sequences 10 and 11 are permanently unexecutable. No on-chain mechanism exists to reset `ClaimRecord.sequence` downward.

---

### Impact Explanation

Governance actions encoded in the skipped VAAs (e.g., data source updates, fee changes, publisher set changes) are permanently blocked from execution on-chain, deviating from the outcome of the approved governance vote. The governance multisig would need to re-propose, re-vote, and re-issue new VAAs with new sequence numbers — a significant operational disruption and a direct manipulation of governance execution order.

---

### Likelihood Explanation

- The Wormhole Guardian API is public; any party can fetch signed VAAs for any emitter.
- `execute_posted_vaa` has no caller restriction — any funded Solana account can submit it.
- The attacker only needs to observe pending governance VAAs and front-run the legitimate relayer with a higher-sequence VAA. This is a straightforward race on a public RPC endpoint.
- No privileged access, leaked keys, or oracle manipulation is required.

---

### Recommendation

Replace the `>` check with a strict contiguity check:

```rust
// Instead of:
constraint = posted_vaa.sequence > claim_record.sequence

// Use:
constraint = posted_vaa.sequence == claim_record.sequence + 1
```

This ensures every sequence number must be executed in exact order, making it impossible to skip any governance VAA. Alternatively, use a per-sequence PDA bitmap (similar to the `Claim` account in `wormhole-solana/src/accounts/claim.rs`) so each VAA is independently claimable and out-of-order execution of one does not block others. [6](#0-5)

### Proof of Concept

```rust
// State test outline (locally executable with anchor test / banks-client):
// 1. Create three VAA accounts for the same emitter: seq=10, seq=11, seq=12
// 2. Execute seq=12 first (unprivileged payer, no special role)
//    → succeeds, ClaimRecord.sequence = 12
// 3. Attempt to execute seq=10
//    → fails with NonIncreasingSequence (10 > 12 is false)
// 4. Attempt to execute seq=11
//    → fails with NonIncreasingSequence (11 > 12 is false)
// 5. Assert the governance instructions in seq=10 and seq=11 were never applied
```

The existing test suite already demonstrates the `NonIncreasingSequence` error for replay and lower-sequence attempts: [7](#0-6) 

The same mechanism that blocks replays also permanently blocks legitimately skipped sequences — confirming the exploit path is locally testable with the existing simulator infrastructure.
