### Title
No Minimum Provider Fee Allows Zero-Fee Registration, Leaving Requests Permanently Unfulfilled — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo` contract (Pyth Pulse) allows providers to register with all-zero fees. When a provider's `baseFeeInWei`, `feePerFeedInWei`, and `feePerGasInWei` are all zero, the provider accrues nothing per fulfilled request but must pay gas and the Pyth oracle update fee to call `executeCallback`. This eliminates any economic incentive to fulfill requests. Because there is no cancel/refund path and no penalty mechanism for non-fulfillment, requests targeting zero-fee providers become permanently stuck. The contract's own TODO comment at line 157 explicitly acknowledges the missing penalty mechanism.

---

### Finding Description

`Echo.registerProvider` and `Echo.setProviderFee` accept all three fee parameters as `uint96` with no lower-bound check:

```solidity
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    provider.feePerFeedInWei = feePerFeedInWei;
    provider.feePerGasInWei = feePerGasInWei;
    provider.isRegistered = true;
    ...
}
``` [1](#0-0) 

`getFee` computes the total fee as the sum of the protocol fee plus all three provider fee components:

```solidity
feeAmount = baseFee + providerBaseFee + providerFeedFee + SafeCast.toUint96(gasFee);
``` [2](#0-1) 

When all provider fee components are zero, `getFee` returns only `pythFeeInWei`. The user pays only the protocol fee, and `req.fee` is stored as `msg.value - _state.pythFeeInWei = 0`: [3](#0-2) 

In `executeCallback`, the provider is credited `(req.fee + msg.value) - pythFee`. If `req.fee = 0` and the caller sends exactly the Pyth oracle update fee as `msg.value`, the provider nets zero while paying gas. No rational provider will call `executeCallback` under these conditions. [4](#0-3) 

The contract itself acknowledges the missing enforcement at line 157:

```solidity
// TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
// This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
// with time in order to ensure that the callback eventually gets executed.
``` [5](#0-4) 

During the exclusivity window, only the assigned provider may call `executeCallback`: [6](#0-5) 

After the exclusivity period expires, any address may call `executeCallback`, but with zero provider fee, no third party has economic incentive to do so either. There is no `cancelRequest` or refund function anywhere in the contract, so stuck requests are permanent.

Additionally, `getFirstActiveRequests` performs a linear scan from `firstUnfulfilledSeq` to `currentSequenceNumber`: [7](#0-6) 

As unfulfilled requests accumulate, this scan becomes increasingly expensive, eventually making the function unusable for off-chain keepers that rely on it to discover pending work.

---

### Impact Explanation

1. **Permanently stuck requests**: Users who submit requests to a zero-fee provider receive no callback. Their application logic (e.g., a DeFi protocol waiting for a price update to settle a trade) is permanently blocked with no recourse.
2. **Locked user funds**: The `pythFeeInWei` paid per request is credited to Pyth's accrued fees immediately and is non-refundable. Users lose these funds with no service delivered.
3. **Keeper DoS via `getFirstActiveRequests`**: An attacker paying only `pythFeeInWei` per request can flood the unfulfilled queue, making the linear scan prohibitively expensive for off-chain infrastructure.

---

### Likelihood Explanation

Registration is permissionless — any address can call `registerProvider(0, 0, 0)`. A malicious actor can register as a zero-fee provider and attract users (or self-submit requests) at minimal cost. Even without malicious intent, a legitimate provider who misconfigures fees to zero creates the same outcome. The protocol provides no guardrail.

---

### Recommendation

Enforce a minimum provider fee at registration and update time. For example:

```solidity
uint96 public constant MIN_PROVIDER_FEE = <governance-set value>;

function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    require(
        baseFeeInWei + feePerFeedInWei + feePerGasInWei >= MIN_PROVIDER_FEE,
        "Provider fee too low"
    );
    ...
}
```

The minimum should be set to a value that covers at least the expected gas cost of `executeCallback` plus the Pyth oracle update fee on the target chain. Additionally, implement the acknowledged penalty mechanism so that a provider who fails to fulfill within a deadline forfeits a bond to whoever eventually fulfills the request.

---

### Proof of Concept

1. Attacker calls `echo.registerProvider(0, 0, 0)` — succeeds with no revert.
2. User calls `echo.requestPriceUpdatesWithCallback(attackerProvider, publishTime, priceIds, gasLimit)` paying only `pythFeeInWei`.
3. `req.fee` is stored as `0`.
4. Attacker never calls `executeCallback` (gas cost > 0 reward).
5. After `exclusivityPeriodSeconds`, any address may call `executeCallback`, but since `req.fee = 0`, no one does.
6. User's callback never fires. No refund path exists. Request is permanently active in the queue.
7. Repeating steps 2–6 N times causes `getFirstActiveRequests` to scan N slots on every invocation, degrading keeper infrastructure.

            if (isActive(req)) {
                requests[actualCount] = req;
                actualCount++;

            currentSeq++;

            assembly {
                mstore(requests, actualCount)
            }
        }
    }
