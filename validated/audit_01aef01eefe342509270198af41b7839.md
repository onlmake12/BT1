Audit Report

## Title
Unbounded O(N) Pool Scan on Every Block in `_update_tx_pool_for_reorg` — (`File: tx-pool/src/process.rs`)

## Summary
`_update_tx_pool_for_reorg` performs two full linear scans over all Gap and Pending pool entries on every block when `mine_mode` is active, with no per-invocation element count guard. Because the pool is bounded only by byte size (default 180 MB), an attacker can fill it with ~900,000 minimum-size transactions, causing each block to trigger an O(N) scan that holds the tx-pool write lock and stalls all concurrent pool operations.

## Finding Description
The function `_update_tx_pool_for_reorg` in `tx-pool/src/process.rs` (L1039–1114) is called on every new block. When `mine_mode` is active it executes two unbounded loops:

```rust
// tx-pool/src/process.rs L1065-1080
for entry in tx_pool.pool_map.entries.get_by_status(&Status::Gap) { ... }
for entry in tx_pool.pool_map.entries.get_by_status(&Status::Pending) { ... }
```

Neither loop has an early-exit condition or a per-call element limit. The pool size is governed solely by `DEFAULT_MAX_TX_POOL_SIZE = 180_000_000` bytes (confirmed in `util/app-config/src/legacy/tx_pool.rs` L20). With a minimum CKB transaction size of ~200 bytes, the pool can hold on the order of 900,000 entries. Every block triggers both full scans.

A secondary surface exists in `tx-pool/src/component/links.rs` (L52–72): `calc_relation_ids` performs an unbounded BFS over the ancestor/descendant graph with no depth or count limit of its own. While `max_ancestors_count` (default 1,000, confirmed at L16 of the legacy config) guards the *depth* of a single chain at submission time via `check_and_record_ancestors`, it does not cap the *width* of the descendant tree traversed by `update_descendants_index_key` (pool_map.rs L447–460) or the total node count visited by `update_ancestors_index_key` (pool_map.rs L432–445) on every add/remove operation.

Existing guards are insufficient:
- `max_tx_pool_size` is a byte-size cap, not an entry-count cap.
- `max_ancestors_count` limits chain depth at submission, not traversal width or total nodes visited during reorg.
- No lock-time budget or iteration cap exists anywhere in the reorg path.

## Impact Explanation
The tx-pool service holds a write lock for the duration of `_update_tx_pool_for_reorg`. With a maximally filled pool (~900K entries), both scans dominate the block-processing critical path. All concurrent RPC calls and relay handlers that require the pool lock are blocked for the duration. Sustained block arrival (~10 s intervals on mainnet) with a full pool can keep a mining node perpetually behind tip, degrading block production and relay throughput. This maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, as mining nodes are directly impacted and their degradation propagates to network-wide block latency.

## Likelihood Explanation
The attack requires only the ability to submit transactions via the public `send_transaction` RPC or the P2P relay protocol — both are available to any unprivileged peer. The only precondition is that the target node has a block assembler configured (`mine_mode` active), which is true of all mining nodes. Filling 180 MB at the minimum fee rate (1,000 shannons/KB) is inexpensive on a low-fee network. No key material, special privilege, or majority hashpower is required. The attack is repeatable: the pool refills between blocks as evicted transactions are resubmitted.

## Recommendation
1. **Cap the per-block promotion scan**: introduce a configurable `MAX_ENTRIES_PER_REORG_SCAN` constant in `_update_tx_pool_for_reorg` and break out of the Gap/Pending loops once that limit is reached, deferring remaining entries to the next block.
2. **Enforce a maximum entry count** alongside the existing byte-size limit in `PoolMap` so that the number of iterations is predictable regardless of transaction size distribution.
3. **Bound `calc_relation_ids` traversal**: add an optional `max_count` parameter and return a truncated result or error when exceeded, mirroring the existing `max_ancestors_count` guard applied at submission time.

## Proof of Concept
1. Configure a CKB node with a block assembler (enabling `mine_mode`).
2. Submit ~900,000 independent minimum-size transactions via `send_transaction` until `total_tx_size` approaches 180 MB.
3. Trigger or observe block arrivals (~every 10 s on mainnet, or mine blocks locally in a test environment).
4. Instrument `_update_tx_pool_for_reorg` (e.g., via `Instant::now()` before/after the `if mine_mode` block) to confirm that each block causes both loops to iterate over all ~900,000 entries.
5. Confirm via RPC latency metrics that `get_transaction` and `send_transaction` calls spike on every block arrival, and that the node's reported tip lags behind the network tip.
