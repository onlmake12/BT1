## Title
TOCTOU Race in `_process_tx` Allows Concurrent Duplicate Script Verification â€” (File: `tx-pool/src/process.rs`)

## Summary
In `TxPoolService::_process_tx`, the duplicate-membership check (`check_txid_collision`) is performed under a short-lived read lock that is released before the expensive `verify_rtx` script-execution step. Because the service loop spawns an independent Tokio task per message, two or more concurrent `SubmitLocalTx` submissions of the same transaction can both pass all duplicate guards and both execute full RISC-V script verification before either one commits the transaction to the pool. With `DEFAULT_CHANNEL_SIZE = 512`, an attacker with local RPC access can trigger up to 512 concurrent full-cycle verifications of a single transaction, saturating CPU and temporarily rendering the node unresponsive.

## Finding Description
**Service loop spawns one task per message** â€” `service.rs:619-621`: each `SubmitLocalTx` message spawns an independent Tokio task, so concurrent submissions run fully in parallel.

**`process_tx` pre-guards each release their lock before `_process_tx` is entered** â€” `process.rs:237-245` and `process.rs:409-411`: `verify_queue_contains` acquires and immediately drops a read lock on `verify_queue`; `orphan_contains` acquires and immediately drops a read lock on `orphan`. Neither holds a lock when `_process_tx` is called at line 413.

**`_process_tx` execution order** â€” `process.rs:715-753`:
1. `pre_check` at line 715 acquires a read lock via `with_tx_pool_read_lock`, calls `check_txid_collision` inside the closure, then **drops the lock when the closure returns** at line 313-314.
2. `verify_rtx` at lines 724-732 runs with **no lock held**. For the local path (`command_rx = None`), this calls `block_in_place` at `util.rs:117`, blocking the current Tokio worker thread for the full duration of RISC-V script execution.
3. `submit_entry` at line 753 acquires a write lock and performs the first write that prevents a duplicate.

**Race window**: From the moment `pre_check` releases its read lock to the moment `submit_entry` acquires its write lock, the transaction is invisible to all duplicate guards. This window spans the entire duration of `verify_rtx`. All concurrent tasks observe an empty pool, all proceed to `verify_rtx`, and all race to `submit_entry`. Only the first insertion succeeds; all others are wasted.

**Contrast with the remote path** â€” `process.rs:860-867`: `enqueue_verify_queue` acquires a write lock on `verify_queue` before returning, preventing duplicate enqueuing. The local `process_tx` path has no equivalent guard.

## Impact Explanation
This is a **Note (0â€“500 points): local RPC API crash**. The attack requires local RPC access (default `127.0.0.1:8114`). With 512 concurrent `block_in_place` calls, each blocking a Tokio worker thread for the duration of RISC-V script execution, the Tokio thread pool is saturated and the node becomes temporarily unresponsive to all messages including block relay, sync, and mining template generation. The node recovers once all verifications complete â€” it does not terminate or permanently crash. The claimed "High: crash a CKB node" severity is not supported: the node is transiently unresponsive, not crashed, and the attack is bounded by the channel capacity and verification duration. The correct impact class is **Note**.

## Likelihood Explanation
Any process with access to the local RPC endpoint can issue 512 concurrent HTTP `send_transaction` requests with no special privileges, keys, or network position. The race window is wide â€” hundreds of milliseconds for a max-cycle transaction â€” making concurrent exploitation straightforward. However, the precondition of local machine access limits the realistic attacker population.

## Recommendation
Add an "in-verification" set (e.g., `Arc<RwLock<HashSet<Byte32>>>`) to `TxPoolService`. Before releasing the read lock in `pre_check` (or immediately after, under a write lock), insert the tx hash into this set and return `Reject::Duplicated` if it is already present. Remove the entry on completion (success or failure) of `verify_rtx` + `submit_entry`. This mirrors the protection already present in `enqueue_verify_queue`, which holds a write lock on `verify_queue` before returning.
