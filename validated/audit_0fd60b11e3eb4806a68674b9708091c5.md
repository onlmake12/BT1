### Title
Deviation-Only Subscriptions Silently Skip Update Trigger When Stored Price Is Zero — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

`Scheduler._validateShouldUpdatePrices()` silently skips the deviation check for any price feed whose stored or incoming price is zero, via an unconditional `continue`. When a price feed transitions from a stored value of zero to any non-zero value, the deviation is mathematically infinite, yet the contract treats it as "no deviation detected" and reverts with `UpdateConditionsNotMet`. This is the direct analog of the external report: a function that silently produces incorrect behavior for a mathematically invalid/edge-case input (zero as a denominator) instead of handling it correctly.

---

### Finding Description

In `_validateShouldUpdatePrices`, the deviation-check loop contains:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
``` [1](#0-0) 

The `continue` silently discards the deviation check for that feed and moves to the next one. If every feed in the subscription has a zero stored price (`previousPrice == 0`) or a zero incoming price (`currentPrice == 0`), the entire loop completes without returning, and the function falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [2](#0-1) 

The critical case is `previousPrice == 0` with `currentPrice != 0`. A price moving from zero to any non-zero value represents an infinite percentage deviation — the strongest possible trigger — yet the contract silently ignores it and rejects the update. The comment acknowledges the zero-skip is intentional ("to avoid division by zero"), but the chosen fix (skip entirely) is incorrect: the correct behavior is to treat a zero-to-nonzero transition as an unconditional trigger.

The entry point is the public `updatePriceFeeds()` function, callable by any unprivileged keeper:

```solidity
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {
``` [3](#0-2) 

The keeper's call passes through `_validateShouldUpdatePrices`, which silently skips the deviation check and reverts.

---

### Impact Explanation

A subscription configured with **deviation-only** update criteria (`updateOnHeartbeat = false`, `updateOnDeviation = true`) and a price feed whose on-chain stored price is zero will be permanently stuck: no keeper can ever trigger an update via the deviation path, regardless of how large the current price is. Consumers of the subscription (DeFi protocols reading prices via `getPricesUnsafe` / `getEmaPriceUnsafe`) will silently receive the stale zero price indefinitely.

Zero prices are valid in Pyth — they represent halted or suspended markets. When such a market reopens, the price transitions from zero to a real value. This is precisely the scenario where a deviation-only subscription fails silently.

The subscription manager has no recourse short of adding a heartbeat criterion or recreating the subscription, and may not even be aware of the failure because the keeper's revert (`UpdateConditionsNotMet`) looks identical to a legitimate "no deviation yet" rejection.

---

### Likelihood Explanation

- Pyth price feeds legitimately publish zero prices for halted markets (equities after hours, suspended tokens, etc.).
- Subscriptions configured with deviation-only criteria are a supported and documented configuration.
- Any permissionless keeper calling `updatePriceFeeds` triggers the path.
- No privileged access, key compromise, or governance action is required.
- The scenario (market halt → reopen) is a routine operational event.

---

### Recommendation

Replace the `continue` with an unconditional trigger when `previousPrice == 0` and `currentPrice != 0` (infinite deviation), and only skip when both are zero (no change):

```solidity
if (previousPrice == 0) {
    if (currentPrice != 0) {
        // Transition from zero to non-zero is infinite deviation — always trigger
        return updateTimestamp;
    }
    // Both zero: no change, skip
    continue;
}
if (currentPrice == 0) {
    // Transition to zero: treat as maximum deviation — trigger
    return updateTimestamp;
}
```

This mirrors the correct mathematical behavior: `ln(x)` is undefined at `x ≤ 0` and should revert; similarly, a deviation denominator of zero is undefined and should be treated as an unconditional trigger rather than silently skipped.

---

### Proof of Concept

1. Deploy `Scheduler` and create a subscription with `updateOnDeviation = true`, `updateOnHeartbeat = false`, `deviationThresholdBps = 100` (1%).
2. Push an initial update with `price = 0` for the subscribed feed. This stores `previousPrice = 0`.
3. Attempt to push a second update with `price = 100_000_000` (a large non-zero value).
4. `_validateShouldUpdatePrices` reaches the deviation loop, hits `previousPrice == 0`, executes `continue`, exhausts the loop, and reverts with `UpdateConditionsNotMet`.
5. The subscription is permanently stuck with a zero price despite the market having a real non-zero price.
