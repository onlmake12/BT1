### Title
Entropy Provider Has No On-Chain Throttle Mechanism — Unprivileged Users Can Exhaust the Provider's Hash Chain, Forcing Emergency Re-Registration and Service Disruption - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract provides no on-chain mechanism for a registered provider to pause, rate-limit, or reject incoming randomness requests. Any user paying the required fee can permissionlessly consume sequence numbers from a provider's committed hash chain. A malicious user can flood the provider with requests, exhausting `endSequenceNumber` and forcing the provider into an emergency re-registration cycle. During the re-registration window, all new requests revert with `OutOfRandomness`, causing a complete service disruption for all downstream consumers.

---

### Finding Description

When a provider registers via `register()`, it commits to a hash chain of finite length `chainLength`. The contract sets:

```
endSequenceNumber = provider.sequenceNumber + chainLength
```

Every call to `requestHelper` (invoked by `request`, `requestWithCallback`, `requestV2`) increments `providerInfo.sequenceNumber` by 1 and checks:

```solidity
if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
    revert EntropyErrors.OutOfRandomness();
providerInfo.sequenceNumber += 1;
```

There is no on-chain mechanism for the provider to:
- Pause incoming requests
- Reject requests from specific callers
- Set a per-block or per-user request cap

The only partial throttle is `maxNumHashes`, which limits the number of *concurrent in-flight* requests (by bounding `numHashes = assignedSequenceNumber - currentCommitmentSequenceNumber`). However, this does not prevent sequential flooding: once the provider fulfills or advances past old requests, new requests are accepted again. Furthermore, `maxNumHashes = 0` disables the check entirely.

An attacker paying the fee for each call can drain the entire remaining `endSequenceNumber - sequenceNumber` budget in a single block (limited only by block gas). Once exhausted, every new `request`/`requestWithCallback`/`requestV2` call reverts with `OutOfRandomness` until the provider calls `register()` again with a new commitment.

Additionally, `constructProviderCommitment` iterates `numHashes` times on-chain during `revealWithCallback`:

```solidity
while (numHashes > 0) {
    currentHash = keccak256(bytes.concat(currentHash));
    numHashes -= 1;
}
```

If `maxNumHashes` is unset (0), an attacker can create a large backlog of unresolved requests, making each subsequent `revealWithCallback` call progressively more expensive in gas, potentially exceeding the block gas limit and making fulfillment impossible.

---

### Impact Explanation

1. **Complete service disruption**: Once `sequenceNumber >= endSequenceNumber`, all new randomness requests revert with `OutOfRandomness`. Any downstream protocol (e.g., NFT minting, gaming, DeFi) that depends on Entropy is halted until the provider re-registers.
2. **Forced emergency re-registration**: The provider must prepare a new hash chain off-chain and submit a new `register()` transaction. The Fortuna provider daemon detects this condition and re-registers, but there is an unavoidable window of unavailability.
3. **Gas cost amplification**: Without `maxNumHashes` set, a large backlog of unresolved requests causes `revealWithCallback` to compute an unbounded number of hashes, potentially making fulfillment transactions revert due to out-of-gas.
4. **Cascading downstream failures**: Protocols that call `requestWithCallback` and wait for the callback will be permanently stuck for all requests made during the disruption window.

---

### Likelihood Explanation

- The attack is fully permissionless: any address paying the fee can call `requestWithCallback`.
- The attacker recovers the fees as `accruedFeesInWei` credited to the provider — but the attacker does not receive those fees; they are paid to the provider. The attacker's cost is the fees paid plus gas.
- On low-fee chains (e.g., BNB Chain, Polygon, Arbitrum), the fee per request is small, making bulk exhaustion economically viable.
- The default Fortuna provider uses a `chain_length` configured at deployment time. If this is set to a moderate value (e.g., 1,000,000), an attacker on a cheap chain can exhaust it in a single block with a batch contract.
- The attack is amplified if the attacker also withholds their `userRandomness`, preventing the provider from advancing `currentCommitmentSequenceNumber` and making each reveal more expensive.

---

### Recommendation

1. **Add a provider-controlled pause flag**: Allow providers to set a boolean `paused` field via a privileged setter. `requestHelper` should check this flag and revert if the provider is paused.
2. **Add a per-block or per-epoch request cap**: Allow providers to configure a maximum number of requests accepted per block or epoch.
3. **Enforce `maxNumHashes` as a mandatory non-zero value**: Require providers to set a non-zero `maxNumHashes` at registration time to bound both concurrent in-flight requests and per-reveal gas cost.
4. **Automatic re-registration buffer**: The Fortuna daemon already detects `endSequenceNumber <= sequenceNumber` and re-registers, but the on-chain contract should allow providers to pre-extend their chain before exhaustion via a dedicated `extendChain` function that does not reset in-flight requests.

### Proof of Concept

**Step 1**: Attacker deploys a batch contract:
```solidity
contract Attacker {
    IEntropy entropy;
    address provider;
    constructor(address _entropy, address _provider) payable {
        entropy = IEntropy(_entropy);
        provider = _provider;
    }
    function flood(uint256 count) external payable {
        uint128 fee = entropy.getFee(provider);
        for (uint256 i = 0; i < count; i++) {
            entropy.requestWithCallback{value: fee}(provider, bytes32(i));
        }
    }
}
```

**Step 2**: Attacker calls `flood(N)` where `N = endSequenceNumber - sequenceNumber` (the remaining chain length). Each call pays the required fee and increments `providerInfo.sequenceNumber`.

**Step 3**: After the flood, `providerInfo.sequenceNumber == providerInfo.endSequenceNumber`. Any subsequent call to `request`, `requestWithCallback`, or `requestV2` for this provider reverts:

```solidity
// Entropy.sol line 229-230
if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
    revert EntropyErrors.OutOfRandomness();
```

**Step 4**: The provider must call `register()` with a new commitment to resume service. During this window, all downstream consumers are blocked.

**Step 5 (gas amplification variant)**: Attacker makes `maxNumHashes` requests without ever calling `reveal`/`revealWithCallback`. The provider's next `revealWithCallback` must compute `maxNumHashes` iterations of `keccak256` in `constructProviderCommitment`, potentially exceeding the block gas limit if `maxNumHashes` is large or unset.

    function setMaxNumHashes(uint32 maxNumHashes) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
