### Title
Division by Zero in TWAP Calculation Due to Missing Slot Equality Check - (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary

`calculateTwap` in `Pyth.sol` divides by `slotDiff` (the difference between end and start `publishSlot`) without guarding against the case where both slots are equal. The upstream `validateTwapPriceInfo` only rejects the case where `start.publishSlot > end.publishSlot`, but silently allows `start.publishSlot == end.publishSlot`. When equal slots are submitted, `slotDiff` is zero and three consecutive Solidity divisions panic-revert the transaction.

### Finding Description

`validateTwapPriceInfo` enforces slot ordering with a strict-greater-than check: [1](#0-0) 

This allows `start.publishSlot == end.publishSlot` to pass validation. Control then flows to `calculateTwap`, which computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // panic: division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // panic: division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panic: division by zero
``` [2](#0-1) 

All three divisions are unguarded. In Solidity, integer division by zero triggers a `Panic(0x12)` revert rather than a clean, descriptive error.

The analogous Solana implementation avoids this entirely by using `checked_div`, which returns `ReceiverError::TwapCalculationOverflow` on a zero divisor: [3](#0-2) 

The Solana validator also uses `end_msg.publish_slot > start_msg.publish_slot` (strictly greater), so the same gap exists in validation logic there, but the arithmetic is safe. The EVM contract has no such safety net.

### Impact Explanation

Any caller of `parseTwapPriceFeedUpdates` who submits two Wormhole-verified TWAP VAAs sharing the same `publishSlot` (e.g., by reusing the same VAA for both the start and end arguments, or by obtaining two legitimately signed snapshots from the same slot) will cause the transaction to revert with an opaque panic. The function is `external payable`; the fee is consumed and no TWAP result is returned. Downstream integrators relying on this function for on-chain TWAP data are denied service for that call. [4](#0-3) 

### Likelihood Explanation

The entry path requires two Wormhole-guardian-signed TWAP VAAs. A user can trivially supply the same VAA for both `updateData[0]` and `updateData[1]`. The `validateTwapPriceInfo` checks for `prevPublishTime >= publishTime` and `expo` mismatch, but when the same VAA is reused, all fields are identical and all checks pass. The equal-slot condition is therefore reachable by any unprivileged transaction sender with no special access.

### Recommendation

Change the slot ordering check in `validateTwapPriceInfo` from strictly-greater-than to greater-than-or-equal, so that equal slots are explicitly rejected with a clean error before the arithmetic is reached:

```solidity
// Before (allows slotDiff == 0):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}

// After (rejects slotDiff == 0):
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

### Proof of Concept

1. Obtain any valid Wormhole-signed TWAP VAA from Hermes for a given price feed.
2. Call `parseTwapPriceFeedUpdates` with `updateData[0] = updateData[1] = <same VAA>` and `priceIds = [<that feed's id>]`, paying the required fee.
3. `validateTwapPriceInfo` passes: `prevPublishTime < publishTime` (valid), `expo` matches, `publishSlot` is not strictly greater (equal), `publishTime` is not strictly greater (equal).
4. `calculateTwap` executes `slotDiff = slot - slot = 0`, then `priceDiff / 0` triggers `Panic(0x12)`.
5. Transaction reverts; fee is consumed; no TWAP output is produced.
