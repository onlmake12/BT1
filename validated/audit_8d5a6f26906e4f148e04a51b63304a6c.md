Audit Report

## Title
Memory Amplification via Duplicate Indexes and Missing Byte-Size Cap in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` accepts up to 32,767 peer-supplied transaction indexes with no deduplication and no byte-size cap on the assembled response. An unprivileged peer can send all 32,767 indexes pointing at the same transaction (e.g., the coinbase), forcing the node to clone and buffer that transaction 32,767 times before sending. Both analogous handlers (`GetTransactionsProcess`, `GetBlockProposalProcess`) explicitly reject duplicate entries and enforce a 1 MiB `MAX_RELAY_TXS_BYTES_PER_BATCH` cap; this handler does neither.

## Finding Description

**Root cause — no deduplication (lines 61–71):**

```rust
let transactions = self
    .message
    .indexes()
    .iter()
    .filter_map(|i| {
        block
            .transactions()
            .get(Into::<u32>::into(i) as usize)
            .cloned()   // ← clones on every iteration
    })
    .collect::<Vec<_>>();
``` [1](#0-0) 

There is no `HashSet` deduplication step. Sending `indexes = [0u32; 32767]` produces a `Vec` of 32,767 clones of `block.transactions()[0]`.

By contrast, `GetTransactionsProcess` builds a `HashSet` and immediately returns `StatusCode::RequestDuplicate` if `message_len != set.len()`: [2](#0-1) 

And `GetBlockProposalProcess` does the same: [3](#0-2) 

**Root cause — no byte-size cap (lines 80–97):**

The entire `transactions` vec is packed into one message and sent unconditionally: [4](#0-3) 

Both `GetTransactionsProcess` and `GetBlockProposalProcess` accumulate `relay_bytes` and flush when `relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MiB): [5](#0-4) [6](#0-5) 

`MAX_RELAY_TXS_BYTES_PER_BATCH` is defined as 1 MiB: [7](#0-6) 

**Note on Defect 1 (off-by-one):** The `>` guard at line 37 is consistent with the identical guard in `GetTransactionsProcess` (line 35 of that file) and is not a meaningful independent defect — both handlers allow exactly 32,767 entries. The real amplification comes from defects 2 and 3. [8](#0-7) 

## Impact Explanation

**High — can easily crash a CKB node.**

Peak allocation per request = `32,767 × coinbase_tx_size`. CKB's consensus `max_block_bytes` is ~597 KB, so a legally mined block can have a coinbase approaching that size. Worst case: `32,767 × 597 KB ≈ 19.5 GB` allocated in a single async task before any send occurs — a guaranteed OOM crash on any normal server. Even with a typical mainnet coinbase of ~500 bytes the allocation is ~16 MB per request; at the rate-limiter ceiling of 30 req/s per peer across multiple simultaneous peers, memory is exhausted rapidly. The rate limiter does not bound per-request allocation. [9](#0-8) 

## Likelihood Explanation

The attack requires only a TCP connection to the relay protocol — no proof-of-work, no key, no privileged role. The attacker needs a block hash for a block stored on the target node, which is trivially obtained from any block explorer or by syncing headers. The message is structurally valid and passes all existing guards. The attack is repeatable and can be parallelized across many peers.

## Recommendation

1. **Deduplicate indexes before lookup**, matching the pattern in `GetTransactionsProcess` and `GetBlockProposalProcess`: collect indexes into a `HashSet`, return `StatusCode::RequestDuplicate` if `set.len() != message_len`, then iterate the deduplicated set.
2. **Add a `MAX_RELAY_TXS_BYTES_PER_BATCH` accumulator** over the assembled transactions and either truncate or split the response when the limit is exceeded, matching the batching loop in `GetTransactionsProcess::execute` and `GetBlockProposalProcess::execute`.
3. Optionally change the count guard to `>=` for consistency with the constant's semantic meaning, though this is a minor hardening step.

## Proof of Concept

```rust
// Minimal unit test sketch
let indexes = vec![0u32; 32767]; // all pointing at coinbase
let msg = GetBlockTransactions {
    block_hash: stored_block_hash, // any block stored on the target node
    indexes,
    uncle_indexes: vec![],
};
// Node executes GetBlockTransactionsProcess::execute():
//   - count guard passes (32767 > 32767 == false)
//   - no dedup: 32767 clones of block.transactions()[0] collected
//   - no byte cap: entire vec packed into one BlockTransactions message
// With coinbase_size = 1 KB  → ~32 MB allocated per request
// With coinbase_size = 597 KB → ~19.5 GB allocated per request → OOM crash
```

A fuzz test targeting `GetBlockTransactionsProcess::execute` with a stored block containing a large coinbase and `indexes` filled with `[0u32; 32767]` will reproduce the allocation spike deterministically.
