### Title
Uncapped `tx.gasprice` in Keeper Fee Calculation Allows Any Caller to Over-Drain Subscription Balance — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
`Scheduler._processFeesAndPayKeeper` reimburses the caller for gas using `tx.gasprice`, which the caller fully controls. Because no maximum gas price is enforced, any unprivileged keeper can set an arbitrarily high `tx.gasprice` and extract `GAS_OVERHEAD × tx.gasprice` extra wei from a subscription's balance on every `updatePriceFeeds` call — directly analogous to the reported pattern of an uncapped `amountOutMin` allowing a caller to extract unexpected value.

### Finding Description
`_processFeesAndPayKeeper` computes the total keeper payment as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
```

`GAS_OVERHEAD` is the constant `30_000` gas, added as a rough overhead estimate. [1](#0-0) [2](#0-1) 

The caller controls `tx.gasprice`. Their net profit per call is:

```
profit = GAS_OVERHEAD × tx.gasprice + keeperSpecificFee
       = 30_000 × tx.gasprice + keeperSpecificFee
```

This grows linearly and without bound as `tx.gasprice` increases. `updatePriceFeeds` has no access control — any address may call it: [3](#0-2) 

`SubscriptionParams` contains no `maxGasPrice` field, so the subscription owner has no on-chain mechanism to cap the gas price a keeper may use: [4](#0-3) 

### Impact Explanation
A malicious keeper sets `tx.gasprice` to an extreme value (e.g. 100 000 gwei). Per call:

- Keeper pays: `gasUsed × 100_000 gwei` (their own cost)
- Keeper receives: `(gasUsed + 30_000) × 100_000 gwei + keeperSpecificFee`
- **Net extraction from subscription balance: `30_000 × 100_000 gwei = 3 ETH` per call**

Repeated calls rapidly drain the subscription balance to zero, denying the subscription owner the price-feed service they paid for and destroying their deposited funds. The subscription owner cannot recover funds from a permanent subscription (`isPermanent = true`) at all. [5](#0-4) 

### Likelihood Explanation
`updatePriceFeeds` is permissionless — any EOA or contract may call it. Setting a custom `tx.gasprice` requires no special privilege, no leaked key, and no governance majority. The attacker only needs to supply valid Pyth update data (freely available from Hermes) and set a high gas price. The attack is repeatable every time the subscription's update criteria are met.

### Recommendation
Add a `maxGasPrice` field to `SubscriptionParams` (set by the subscription owner at creation/update time) and enforce it inside `_processFeesAndPayKeeper`:

```solidity
if (tx.gasprice > params.maxGasPrice) {
    revert SchedulerErrors.GasPriceTooHigh();
}
```

Alternatively, cap the reimbursed gas price at a protocol-wide governance-controlled maximum, analogous to how the original report recommends letting users set their own slippage tolerance.

### Proof of Concept
1. Subscription owner creates a subscription with 10 ETH balance, 5 price feeds, heartbeat = 60 s.
2. Attacker waits 60 s (heartbeat elapsed), fetches valid Pyth update data from Hermes.
3. Attacker calls `updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = 100_000 gwei`.
4. `_processFeesAndPayKeeper` computes:
   - `gasCost = (actualGasUsed + 30_000) × 100_000 gwei`
   - Suppose `actualGasUsed ≈ 300_000`: `gasCost = 330_000 × 100_000 gwei = 33 ETH`
5. Contract attempts to deduct 33 ETH; if balance < 33 ETH it reverts with `InsufficientBalance`. Attacker lowers `tx.gasprice` to just below the balance threshold and drains the full balance in one call.
6. Subscription balance reaches zero; subscription is effectively bricked. [6](#0-5)

```text
    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
```
