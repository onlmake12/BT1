### Title
Off-by-One Epoch Gate Skips `DaoScriptSizeVerifier` for First Block of RFC0044 Activation Epoch — (`verification/contextual/src/contextual_block_verifier.rs`)

### Summary

The `BlockTxsVerifier` in the contextual block verifier gates `DaoScriptSizeVerifier` using the **parent block's** epoch number rather than the current block's epoch number. Because `RFC0044_ACTIVE_EPOCH = 8651`, the very first block of epoch 8651 has a parent in epoch 8650, causing `rfc0044_active(8650)` to return `false` and the verifier to be skipped entirely for that block.

### Finding Description

`RFC0044_ACTIVE_EPOCH` is hardcoded to `8651`: [1](#0-0) 

`rfc0044_active` uses `>=` comparison against that constant: [2](#0-1) 

The gate in `BlockTxsVerifier::verify` passes `self.parent.epoch().number()` — the **parent's** epoch — not the current block's epoch: [3](#0-2) 

For the first block of epoch 8651:
- `self.parent.epoch().number()` = **8650**
- `rfc0044_active(8650)` = `8650 >= 8651` = **`false`**
- `DaoScriptSizeVerifier` is **skipped**

For every subsequent block in epoch 8651 and beyond, the parent is already in epoch 8651, so `rfc0044_active(8651)` = `true` and the verifier runs correctly. The window is exactly **one block**.

### Impact Explanation

A miner who wins the PoW race for the first block of epoch 8651 can include a DAO phase-1 withdrawal transaction where the withdrawing cell's lock script size differs from the deposit cell's lock script size. The block verifier accepts it. All other nodes replaying the chain use the same gate and also accept it. The malformed DAO withdrawal is permanently committed to the canonical chain, violating the RFC0044 invariant that lock-size enforcement must apply to all blocks at or after the activation epoch.

Note: the tx-pool path in `tx-pool/src/util.rs` calls `DaoScriptSizeVerifier` unconditionally (no epoch gate), so the transaction would be rejected if submitted through the mempool. However, a miner can construct a block directly, bypassing the tx-pool entirely. [4](#0-3) 

### Likelihood Explanation

Requires valid PoW for the first block of epoch 8651 — a standard miner capability, not a privileged role. The window is one block, but any miner who mines that block can exploit it. The exploit is deterministic and locally testable.

### Recommendation

Replace `self.parent.epoch().number()` with the current block's epoch number in the gate condition:

```rust
// current block's epoch number, not parent's
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
```

### Proof of Concept

1. Build a test chain advancing to the last block of epoch 8650 (parent is in epoch 8650).
2. Construct a DAO deposit cell committed at a block number ≥ `starting_block_limiting_dao_withdrawing_lock()`.
3. Construct a phase-1 withdrawal transaction where the output cell uses the DAO type script but has a lock script of a **different size** than the deposit cell's lock script.
4. Mine the first block of epoch 8651 containing this transaction.
5. Call `BlockTxsVerifier::verify` — it evaluates `rfc0044_active(8650)` = `false`, skips `DaoScriptSizeVerifier`, and returns `Ok`.
6. Assert the block is accepted by the chain. [3](#0-2) [5](#0-4) [1](#0-0)
