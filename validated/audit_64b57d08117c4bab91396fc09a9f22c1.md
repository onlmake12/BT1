Audit Report

## Title
Unauthenticated Outbound Raw Session to Attacker-Controlled Endpoint via Crafted `ConnectionRequestDelivered` — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary
`ConnectionRequestDeliveredProcess::execute()` reaches the NAT traversal branch when `content.route` is empty, `content.from` equals the local node's own `PeerId`, and `content.to` matches a key in `inflight_requests`. All three conditions are attacker-controllable. Once triggered, `try_nat_traversal` establishes a TCP connection to attacker-supplied addresses and calls `raw_session` with the Identify protocol, enabling peer-store poisoning with attacker-chosen addresses.

## Finding Description

**Precondition — getting `PeerId A` into `inflight_requests`:**

`inflight_requests` is populated in `notify()` via `fetch_nat_addrs()`. [1](#0-0) 

`fetch_nat_addrs` returns addresses that satisfy: (1) required flags, (2) not currently connected, (3) Ip4/Ip6 only, and (4) `last_connected_at_ms == 0` (never connected). [2](#0-1) 

The attacker uses **two PeerIds**: PeerId B (used for the live connection to the victim) and PeerId A (never connected, advertised via Discovery/Identify so it lands in the victim's peer store with `last_connected_at_ms == 0`). Because PeerId A has never connected, it passes all `fetch_nat_addrs` filters and is inserted into `inflight_requests` on the next 5-minute tick. [3](#0-2) 

**Triggering the NAT traversal branch:**

The attacker (connected as PeerId B) sends a `ConnectionRequestDelivered` message with:
- `from` = victim's own `PeerId` (public, obtained from Identify)
- `to` = PeerId A (in `inflight_requests`)
- `route` = `[]`
- `listen_addrs` = `[/ip4/attacker_ip/tcp/attacker_port/p2p/PeerId_A]`

With `route` empty and `self_peer_id == content.from`, the code removes PeerId A from `inflight_requests` and calls `try_nat_traversal`. [4](#0-3) 

**`listen_addrs` validation does not check IP ownership:**

The only check on `listen_addrs` is that any embedded `PeerId` must match `content.to`. Since `content.to` is PeerId A (attacker-controlled), the attacker freely embeds their own IP:port. [5](#0-4) 

**`try_nat_traversal` connects unconditionally:**

`try_nat_traversal` retries TCP connections for up to 30 seconds. On success, `raw_session` is called with `RawSessionInfo::outbound(Identify)` — no authentication of the endpoint occurs before this call. [6](#0-5) 

When `bind_addr` is `None` (all non-Linux platforms, and Linux without `reuse_port_on_linux`), `create_socket` still creates an unbound socket that connects to the attacker's endpoint. [7](#0-6) 

**Peer-store poisoning via Identify:**

The attacker's TCP listener accepts the connection and sends a crafted Identify response. `received_identify` calls `add_remote_listen_addrs`, which inserts attacker-chosen addresses into the victim's peer store. [8](#0-7) 

The attacker, already a connected CKB node, knows the network magic and can craft a valid Identify payload that passes `verify`. The victim's peer store is then poisoned with arbitrary attacker-controlled addresses.

## Impact Explanation

The victim node's peer store is poisoned with attacker-controlled addresses. The victim will subsequently attempt outbound connections to those addresses, enabling a progressive eclipse attack. A fully eclipsed node can be fed a false chain tip, causing it to deviate from consensus — mapping to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially escalating to **Critical** if the eclipse is completed and consensus deviation is achieved.

## Likelihood Explanation

The attack requires only a standard P2P connection and the ability to inject one address into the victim's peer store via Discovery (a normal protocol operation). The victim's `PeerId` is public. The `inflight_requests` window is 5 minutes, giving ample time to deliver the crafted message. The two-PeerId technique (PeerId A never connects; PeerId B sends the message) is straightforward. No special privileges, leaked keys, or majority hashpower are required. The attack is repeatable and platform-independent.

## Recommendation

1. **Authenticate the sender**: Verify that a `ConnectionRequestDelivered` message arrives from the session whose `PeerId` matches `content.to`. Reject messages where the sending session's `PeerId` does not equal `content.to`.
2. **Bind `inflight_requests` to the session**: Record the session ID alongside the `PeerId` in `inflight_requests` and reject `ConnectionRequestDelivered` messages that arrive from any other session.
3. **Cross-check `listen_addrs`**: Validate that the IP in `listen_addrs` matches the observed address of the connected peer that sent the message, rather than accepting arbitrary IP:port values.

## Proof of Concept

1. Attacker generates PeerId A (never connects with it) and PeerId B (used for the live connection).
2. Attacker (PeerId B) connects to victim V and advertises PeerId A's address (e.g., `/ip4/attacker_ip/tcp/attacker_port/p2p/PeerId_A`) via Discovery or Identify, causing V to add it to the peer store with `last_connected_at_ms == 0`.
3. Wait for V's `notify()` to fire (≤5 minutes); PeerId A is now in `V.inflight_requests`.
4. Attacker opens a TCP listener on `attacker_ip:attacker_port` (controlled by PeerId A's key).
5. Attacker (PeerId B) sends `HolePunchingMessage::ConnectionRequestDelivered` to V with `from` = V's `PeerId`, `to` = PeerId A, `route` = `[]`, `listen_addrs` = `[/ip4/attacker_ip/tcp/attacker_port/p2p/PeerId_A]`.
6. V processes the message, removes PeerId A from `inflight_requests`, and calls `try_nat_traversal`.
7. V connects to `attacker_ip:attacker_port`; attacker's listener accepts.
8. V calls `raw_session(stream, attacker_addr, RawSessionInfo::outbound(Identify))`.
9. Attacker's listener sends a crafted Identify response with arbitrary `listen_addrs`.
10. Assert: V's peer store now contains attacker-chosen addresses; V will attempt outbound connections to those addresses on the next dial cycle.

    pub fn fetch_nat_addrs(&mut self, count: usize, required_flags: Flags) -> Vec<AddrInfo> {
        // Get info:
        // 1. Never connected
        // 2. Not already connected
        // 3. Ip4 / Ip6 address only

        let peers = &self.connected_peers;

        let filter = |peer_addr: &AddrInfo| {
            required_flags_filter(required_flags, Flags::from_bits_truncate(peer_addr.flags))
                && extract_peer_id(&peer_addr.addr)
                    .map(|peer_id| !peers.contains_key(&peer_id))
                    .unwrap_or_default()
                && peer_addr.addr.iter().any(|p| {
                    matches!(
                        p,
                        p2p::multiaddr::Protocol::Ip4(_) | p2p::multiaddr::Protocol::Ip6(_)
                    )
                })
                && peer_addr.last_connected_at_ms == 0
        };

        self.addr_manager.fetch_random(count, filter)
    }

    fn add_remote_listen_addrs(&mut self, session: &SessionContext, addrs: Vec<Multiaddr>) {
        trace!(
            "IdentifyProtocol add remote listening addresses, session: {:?}, addresses : {:?}",
            session, addrs,
        );
        let flags = self.network_state.with_peer_registry_mut(|reg| {
            if let Some(peer) = reg.get_peer_mut(session.id) {
                peer.listened_addrs = addrs.clone();
                peer.identify_info
                    .as_ref()
                    .map(|a| a.flags)
                    .unwrap_or(Flags::COMPATIBILITY)
            } else {
                Flags::COMPATIBILITY
            }

            for addr in addrs {
                if let Err(err) = peer_store.add_addr(addr.clone(), flags) {
