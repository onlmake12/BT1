### Title
Unconditional Pre-Hardfork Enforcement of `MAX_DEP_EXPANSION_LIMIT` Due to Dead Hardfork Guard — (`util/types/src/core/cell.rs`)

### Summary
`resolve_transaction_deps_with_system_cell_cache` enforces `OverMaxDepExpansionLimit` unconditionally. The hardfork guard `is_disallow_over_max_dep_expansion_limit_enabled` (rfc_0038) is defined but never called. A transaction with dep expansion count > 2048 is rejected by upgraded nodes even before the rfc_0038 activation epoch, while pre-upgrade nodes accept it, causing a consensus split.

### Finding Description
`MAX_DEP_EXPANSION_LIMIT = 2048` is defined at line 33 of `cell.rs`. The function `resolve_transaction_deps_with_system_cell_cache` initializes `remaining_dep_slots = MAX_DEP_EXPANSION_LIMIT` and calls `checked_sub(...).ok_or(OutPointError::OverMaxDepExpansionLimit)?` on every dep resolution — both in the system-cell-cache path (lines 769–778) and in the fallback path via `resolve_transaction_dep` (lines 825–836). No epoch number, no consensus object, and no hardfork switch is passed into this function or its callers. The function `is_disallow_over_max_dep_expansion_limit_enabled` in `ckb2021.rs` is the intended gate for this behavior (rfc_0038), but it has zero call sites in production code.

The truncated comment at lines 757–761 documents only the post-hardfork rejection row, confirming the pre-hardfork "accept" row was intended but never implemented.

### Impact Explanation
Before the rfc_0038 activation epoch, a transaction with >2048 dep expansions is valid under the old consensus rules. An upgraded node running this code rejects it with `Reject::Resolve(OutPointError::OverMaxDepExpansionLimit)` — classified as a permanent malformed rejection. If a miner (running old software) includes such a transaction in a block before the hardfork epoch, upgraded nodes reject the block while non-upgraded nodes accept it. This is a direct consensus split between upgraded and non-upgraded nodes.

### Likelihood Explanation
Constructing a transaction with >2048 dep expansions requires crafting dep groups that expand to more than 2048 cells. This is technically feasible for any transaction submitter with access to the P2P network or RPC. The attack window is the period between node upgrade deployment and the rfc_0038 activation epoch. On mainnet, if nodes upgrade before the epoch boundary, this window exists.

### Recommendation
Pass the current epoch and the `CKB2021` hardfork switch into `resolve_transaction_deps_with_system_cell_cache` (and its callers `resolve_transaction` and `resolve_transaction_dep`). Gate the `OverMaxDepExpansionLimit` error behind `hardfork.is_disallow_over_max_dep_expansion_limit_enabled(current_epoch)`. Before the activation epoch, allow the transaction to proceed even if dep expansion exceeds 2048.

### Proof of Concept
Call `resolve_transaction` with a `TransactionView` whose cell deps expand (via dep groups) to 2049 cells, using an epoch number strictly less than the rfc_0038 activation epoch. The function returns `Err(OutPointError::OverMaxDepExpansionLimit)`. Under correct hardfork-gated behavior, it should return `Ok(ResolvedTransaction)`. [9](#0-8) [10](#0-9) [7](#0-6)

fn resolve_transaction_deps_with_system_cell_cache<
    F: FnMut(&OutPoint, bool) -> Result<CellMeta, OutPointError>,
>(
    transaction: &TransactionView,
    cell_resolver: &mut F,
    resolved_cell_deps: &mut Vec<CellMeta>,
    resolved_dep_groups: &mut Vec<CellMeta>,
) -> Result<(), OutPointError> {
    // - If the dep expansion count of the transaction is not over the `MAX_DEP_EXPANSION_LIMIT`,
    //   it will always be accepted.
    // - If the dep expansion count of the transaction is over the `MAX_DEP_EXPANSION_LIMIT`, the
    //   behavior is as follow:
    //   | ckb v2021 | yes |             reject the transaction              |
    let mut remaining_dep_slots = MAX_DEP_EXPANSION_LIMIT;
    if let Some(system_cell) = SYSTEM_CELL.get() {
        for cell_dep in transaction.cell_deps_iter() {
            if let Some(resolved_dep) = system_cell.get(&cell_dep) {
                match resolved_dep {
                    ResolvedDep::Cell(cell_meta) => {
                        resolved_cell_deps.push(cell_meta.clone());
                        remaining_dep_slots = remaining_dep_slots
                            .checked_sub(1)
                            .ok_or(OutPointError::OverMaxDepExpansionLimit)?;
                    }
                    ResolvedDep::Group(dep_group, cell_deps) => {
                        resolved_dep_groups.push(dep_group.clone());
                        resolved_cell_deps.extend(cell_deps.clone());
                        remaining_dep_slots = remaining_dep_slots
                            .checked_sub(cell_deps.len())
                            .ok_or(OutPointError::OverMaxDepExpansionLimit)?;
