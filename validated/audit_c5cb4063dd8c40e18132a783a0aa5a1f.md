### Title
Unprivileged Front-Running of `init_encoded_vaa` Allows Rent Theft via `write_authority` Hijack — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/init_encoded_vaa.rs`)

---

### Summary

The `init_encoded_vaa` instruction accepts any signer as `write_authority` and only requires the `encoded_vaa` account to be owned by `crate::ID` with zeroed data. There is no check binding `write_authority` to the account's rent-payer or creator. When `create_account` and `init_encoded_vaa` are executed in separate transactions — a pattern explicitly demonstrated in the test suite — an attacker can race between the two transactions, claim `write_authority` over the victim's account, and subsequently drain the rent lamports via `close_encoded_vaa`.

---

### Finding Description

**`InitEncodedVaa` account struct** only enforces:
- `write_authority`: is a `Signer` (any signer)
- `encoded_vaa`: is `mut` and `owner = crate::ID` [1](#0-0) 

The `constraints()` function checks only that the account is large enough and that its first `VAA_START` bytes are all zeros: [2](#0-1) 

The handler then writes the attacker-supplied `write_authority` key directly into the account header: [3](#0-2) 

`close_encoded_vaa` then transfers all lamports to whoever is stored as `write_authority`: [4](#0-3) [5](#0-4) 

The test suite explicitly demonstrates the two-transaction pattern that creates the race window: [6](#0-5) 

---

### Impact Explanation

After TX1 (`create_account`) is confirmed, the `encoded_vaa` account is owned by `crate::ID` with all-zero data. Any unprivileged attacker can submit `init_encoded_vaa(write_authority=attacker_key, encoded_vaa=victim_account)` before the legitimate relayer's TX2. The constraints pass because:
1. The account is owned by `crate::ID` ✓
2. The data is all zeros ✓
3. The attacker is a valid signer ✓

The victim's TX2 then fails with `AccountNotZeroed` because the discriminator and header are already written. The attacker, now holding `write_authority`, calls `close_encoded_vaa` to receive all lamports. The victim's rent is permanently lost.

---

### Likelihood Explanation

Solana validators see pending transactions before they are finalized. The attack requires observing TX1 and submitting a competing TX2 with higher priority fees — a standard front-running technique on Solana. The two-transaction pattern is explicitly present in the test suite and is a natural consequence of large VAAs that may not fit with `create_account` in a single transaction.

---

### Recommendation

Bind `write_authority` to the account's rent-payer at the protocol level. The simplest fix is to require that `write_authority` signs the `create_account` instruction by combining both instructions atomically in a single transaction (enforced at the instruction level, e.g., by requiring `encoded_vaa` to also be a `Signer`, which is only possible when the keypair is available in the same transaction). Alternatively, add a PDA-based initialization that derives the `encoded_vaa` address from the `write_authority` key, making hijacking impossible.

---

### Proof of Concept

```
1. Relayer generates `encoded_vaa_keypair`.
2. TX1 (relayer): system_instruction::create_account(
       payer=relayer, new_account=encoded_vaa_keypair.pubkey(),
       lamports=rent, space=N, owner=crate::ID)
   → confirmed; account is all zeros, owned by core bridge.

3. Attacker observes TX1 on-chain.
4. TX2 (attacker): init_encoded_vaa(
       write_authority=attacker_key,   ← attacker signs
       encoded_vaa=encoded_vaa_keypair.pubkey())
   → passes all constraints; attacker_key written as write_authority.

5. TX2 (relayer): init_encoded_vaa(write_authority=relayer_key, ...)
   → fails: CoreBridgeError::AccountNotZeroed (header already written).

6. TX3 (attacker): close_encoded_vaa(
       write_authority=attacker_key,
       encoded_vaa=encoded_vaa_keypair.pubkey())
   → write_authority check passes; all lamports transferred to attacker.
```
