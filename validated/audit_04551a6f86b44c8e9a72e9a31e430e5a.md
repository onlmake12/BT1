### Title
Stale Price Returned Without Staleness Check in `PythAggregatorV3.latestRoundData()` and Related Functions - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, the official Pyth-provided Chainlink `AggregatorV3Interface` adapter contract, implements `latestRoundData()`, `latestAnswer()`, `getRoundData()`, `latestTimestamp()`, and `decimals()` by calling `pyth.getPriceUnsafe(priceId)` in every case. `getPriceUnsafe` is explicitly documented to return a price from arbitrarily far in the past with no recency guarantee. No staleness check is applied anywhere in the adapter. Any DeFi protocol that migrates from Chainlink to Pyth using this officially recommended adapter inherits a stale-price vulnerability identical in class to the reported Chainlink issue.

---

### Finding Description

`PythAggregatorV3` is the official Pyth SDK contract for Chainlink compatibility, published as `@pythnetwork/pyth-sdk-solidity` and actively recommended in the official Chainlink migration guide.

Every price-reading function in the contract delegates to `pyth.getPriceUnsafe(priceId)`:

```solidity
// latestRoundData — line 110
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// latestAnswer — line 54
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// getRoundData — line 89
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// decimals — line 41
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
``` [1](#0-0) [2](#0-1) 

`getPriceUnsafe` is defined in `IPyth.sol` with the explicit warning:

> "This function returns the most recent price update in this contract without any recency checks. This function is unsafe as the returned price update may be arbitrarily far in the past." [3](#0-2) 

The safe alternative, `getPriceNoOlderThan(id, age)`, enforces a staleness threshold and reverts with `StalePrice` if the price is too old:

```solidity
if (diff(block.timestamp, price.publishTime) > age)
    revert PythErrors.StalePrice();
``` [4](#0-3) 

`PythAggregatorV3` never calls `getPriceNoOlderThan` and performs no equivalent check on `publishTime` before returning the price to callers.

---

### Impact Explanation

Any DeFi protocol (lending, borrowing, derivatives, liquidation engine) that uses `PythAggregatorV3` as a drop-in Chainlink replacement will receive stale prices from `latestRoundData()` / `latestAnswer()` without any on-chain indication that the price is outdated. Concrete impacts:

- **Undercollateralized borrowing**: A borrower can borrow against collateral valued at a stale (inflated) price, draining the lending pool.
- **Blocked or incorrect liquidations**: A stale price may prevent a liquidation that should occur, or trigger one that should not.
- **Derivative mispricing**: Perpetual or options protocols settle at an incorrect price, causing direct loss to counterparties.

The `roundId` returned is set to `uint80(price.publishTime)` — a timestamp cast — not a monotonically increasing round counter, so downstream contracts that check `answeredInRound >= roundId` as a staleness guard will also be silently bypassed. [5](#0-4) 

---

### Likelihood Explanation

- `PythAggregatorV3` is the officially published and recommended migration path from Chainlink to Pyth, documented in the Pyth developer hub.
- The contract is distributed via the `@pythnetwork/pyth-sdk-solidity` npm package and is actively used by protocols migrating from Chainlink.
- Pyth uses a pull-based update model: if no keeper calls `updateFeeds()`, the on-chain price can become arbitrarily stale. This is a normal operational condition, not an edge case.
- An unprivileged attacker only needs to observe that the on-chain price is stale (e.g., keeper is down or delayed) and then interact with the victim protocol — no special access is required. [6](#0-5) 

---

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`, where `maxAge` is a configurable constructor parameter (e.g., defaulting to `3600` seconds). Revert if the price is stale. Example fix for `latestRoundData`:

```solidity
uint256 public maxPriceAge;

function latestRoundData() external view returns (
    uint80 roundId, int256 answer,
    uint256 startedAt, uint256 updatedAt, uint80 answeredInRound
) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxPriceAge);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

Apply the same fix to `latestAnswer()`, `getRoundData()`, `latestTimestamp()`, and `decimals()`.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a price feed (e.g., BTC/USD).
2. Allow the on-chain price to become stale (do not call `updateFeeds()` for > 1 hour).
3. Call `latestRoundData()` — it returns the old price with no revert.
4. A lending protocol using this adapter accepts the stale (e.g., inflated) price as current.
5. Attacker borrows the maximum amount against collateral valued at the stale price, then the real price drops, leaving the protocol with bad debt.

The root cause is entirely within Pyth's own production SDK file: [1](#0-0) [7](#0-6) [8](#0-7)

    function getPriceNoOlderThan(
        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
        price = getPriceUnsafe(id);

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();

        return price;
    }
