### Title
`check_tx_fee` Uses Serialized Size Instead of Weight for `min_fee_rate` Gate, Allowing Sub-Threshold Fee-Rate Transactions Into the Pool - (`File: tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee by calling `min_fee_rate.fee(tx_size)`, treating raw serialized byte-size as the weight denominator. However, `FeeRate` is defined as **shannons per kilo-weight**, where `weight = max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For any transaction whose cycle cost dominates its byte size, the actual weight is larger than `tx_size`, so the actual fee rate is proportionally lower than what the gate checks. No second fee-rate check is performed after cycles are verified. This is a dimensional mismatch: the gate enforces `fee ≥ min_fee_rate × size / 1000` (shannons/KB semantics) while the pool's own fee-rate type and all downstream sorting use `fee × 1000 / weight` (shannons/KW semantics).

**The correct fee rate used everywhere else (sorting, eviction):** [5](#0-4) 

**Dimensional analysis:**

| Quantity | Formula | Unit |
|---|---|---|
| `min_fee_rate` | configured value | shannons / kilo-weight |
| `min_fee` (gate) | `min_fee_rate × tx_size / 1000` | shannons (uses size, not weight) |
| actual fee rate | `fee × 1000 / max(size, cycles × k)` | shannons / kilo-weight |

When `cycles × DEFAULT_BYTES_PER_CYCLES > tx_size`, the gate underestimates the required fee by a factor of `weight / tx_size`.

**Concrete numbers with default config (`min_fee_rate = 1000`, `max_tx_verify_cycles = 70_000_000`):**

- `tx_size = 242` bytes → gate requires `1000 × 242 / 1000 = 242` shannons
- `cycles = 70_000_000` → `weight = max(242, 70_000_000 × 0.000_170_571_4) ≈ 11_940`
- Actual fee rate of admitted tx: `242 × 1000 / 11_940 ≈ 20` shannons/KW — **50× below `min_fee_rate`** [6](#0-5) 

---

### Impact Explanation

An attacker can submit transactions that:
1. Pass the `check_tx_fee` gate (fee ≥ `min_fee_rate × size / 1000`)
2. Have actual fee rates far below `min_fee_rate` (because cycles dominate weight)
3. Are admitted to the tx-pool and relayed to peers

This bypasses the spam-prevention intent of `min_fee_rate`. The pool fills with low-fee-rate transactions that consume memory and relay bandwidth. Because `is_allowed_relay` returns `false` only for `LowFeeRate` rejections, and these transactions are **not** rejected with `LowFeeRate`, they are eligible for relay propagation across the network. [7](#0-6) 

---

### Likelihood Explanation

The attack is reachable by any unprivileged actor:
- **RPC caller** via `send_transaction` (local or remote if RPC is exposed)
- **P2P relay peer** via `RelayTransactions` message [8](#0-7) 

Crafting a high-cycle, small-size transaction is straightforward: a script with a tight loop in CKB-VM produces high cycles with minimal serialized bytes. The attacker only needs to pay `min_fee_rate × size / 1000` shannons, which can be orders of magnitude less than the correct threshold.

---

### Recommendation

After `verify_rtx` returns the actual cycle count, re-check the fee rate using the correct weight:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

This mirrors how `TxEntry::fee_rate()` already computes the fee rate for sorting and eviction. [5](#0-4) 

---

### Proof of Concept

1. Configure a node with default `min_fee_rate = 1000` shannons/KW and `max_tx_verify_cycles = 70_000_000`.
2. Craft a transaction with:
   - Serialized size ≈ 242 bytes (minimal transaction)
   - A script that consumes ~70,000,000 cycles (tight loop)
   - Fee = 242 shannons (exactly `1000 × 242 / 1000`)
3. Submit via `send_transaction` RPC.
4. **Expected (correct) behavior**: rejected with `LowFeeRate` because actual weight ≈ 11,940 → required fee = 11,940 shannons.
5. **Actual behavior**: accepted into the pool. The gate at `tx-pool/src/util.rs:45` computes `min_fee = 1000 × 242 / 1000 = 242` shannons and `fee (242) >= min_fee (242)` passes.
6. The admitted entry has `fee_rate() = 242 × 1000 / 11940 ≈ 20` shannons/KW — 50× below `min_fee_rate`. [9](#0-8) [2](#0-1)

/// Because there are two different limits, serialized size and consumed cycles,

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
