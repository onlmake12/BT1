Audit Report

## Title
`remove_expired` Uses Single-Entry Removal Instead of Descendant-Aware Removal, Leaving Child Transactions Permanently Stuck in Pool — (`tx-pool/src/pool.rs`)

## Summary
`TxPool::remove_expired` calls `pool_map.remove_entry` (single-entry removal) for each expired transaction, while every other bulk-eviction path (`limit_size`, `resolve_conflict`, `resolve_conflict_header_dep`, `remove_by_detached_proposal`, `remove_tx`) uses `pool_map.remove_entry_and_descendants`. When a parent transaction P expires before its children, the children remain in the pool with inputs referencing P's outputs — which are neither on-chain nor in the pool — making them permanently unmineable until their own expiry fires or `limit_size` non-deterministically evicts them. An attacker can exploit this to continuously inflate pool occupancy with stuck entries at low cost, causing legitimate transactions to be rejected.

## Finding Description

**Root cause — `remove_expired` (`pool.rs` L271–287):**

```rust
self.pool_map.remove_entry(&entry.proposal_short_id());
```

`remove_entry` (`pool_map.rs` L235–250) removes only the targeted entry. It calls `remove_entry_edges` which releases P's own inputs from `edges.inputs`, and `remove_entry_links` which removes P from each child's parent set in `links` and removes P's own link entry. It does **not** touch any child entry.

**What `remove_entry_links` does for child C (`pool_map.rs` L418–430):**
- Removes P from C's parent set in `links.inner[C].parents`
- Removes C from P's child set in `links.inner[P].children`
- Removes P's link entry entirely

**What is left behind after `remove_expired` removes parent P:**
1. P's entry is gone from `entries`; P's own inputs are released from `edges.inputs`.
2. C's input edge `P_output_outpoint → C_short_id` is **still present** in `edges.inputs` — `remove_entry_edges` only removes P's own inputs, not the edges of transactions that spend P's outputs.
3. C's `links` entry no longer lists P as a parent (cleaned by `remove_entry_links`), so C appears parentless in the links graph.
4. C retains its `Pending`/`Gap`/`Proposed` status, is counted in `pending_count`/`gap_count`/`proposed_count`, and occupies a pool slot.
5. C's input references P's output, which is neither on-chain (P was never mined) nor in the pool (P was just expired). The block assembler cannot resolve C's input.

**Contrast with `limit_size` (`pool.rs` L307) and `resolve_conflict` (`pool_map.rs` L310):**

```rust
let removed = self.pool_map.remove_entry_and_descendants(&id);
```

`remove_entry_and_descendants` (`pool_map.rs` L252–265) first collects the target plus the full transitive descendant set via `calc_descendants`, strips all links for all collected IDs, then removes every entry in that set — leaving no orphaned children.

**Trigger path:**
`_update_tx_pool_for_reorg` → `tx_pool.remove_expired(callbacks)` fires on every new block.

**Precondition for the asymmetry:** P submitted at `t0`, C submitted at `t0 + δ` (δ > 0). Both share the same `expiry` duration. At time `t0 + expiry + ε` (where `ε < δ`), P's expiry condition `expiry + t0 < now_ms` is true but C's `expiry + (t0 + δ) < now_ms` is false. `remove_expired` removes P but not C, leaving C stuck.

## Impact Explanation

**High — bad design causing CKB network congestion with few costs.**

An attacker constructs a fan-out chain: one parent P with N outputs, and N children C1…CN each spending one distinct output of P. After P's expiry fires, all N children are stuck in the pool occupying N pool slots, inflating `pending_count`/`gap_count`/`proposed_count`, and blocking legitimate transactions from being accepted once `max_tx_pool_size` is reached. The attacker can repeat this cycle continuously (each cycle requires only one new UTXO for a fresh P), keeping the pool partially or fully saturated with unresolvable entries. `limit_size` will eventually evict them by fee-rate, but this is non-deterministic and itself rejects legitimate low-fee transactions in the process. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

Triggerable by any unprivileged user via the `send_transaction` RPC. No special privileges, no victim interaction, and no external dependency required. The only precondition is that P and C have different submission timestamps, which is the normal case whenever a user submits a parent-child chain sequentially (even milliseconds apart suffice). `remove_expired` fires on every block, so the trigger is continuous and automatic.

## Recommendation

Replace `remove_entry` with `remove_entry_and_descendants` inside `remove_expired`, matching the behavior of every other bulk-removal path:

```rust
// current (broken)
self.pool_map.remove_entry(&entry.proposal_short_id());

// fix
let removed = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
for e in removed {
    let reject = Reject::Expiry(e.timestamp);
    callbacks.call_reject(self, &e, reject);
}
```

Because the outer loop pre-collects all expired entries before removal, calling `remove_entry_and_descendants` for a parent may also remove children that are themselves in the `removed` list. This is safe: `remove_entry` returns `Option<TxEntry>` and gracefully returns `None` for already-removed entries, and `remove_entry_links` on an already-removed ID is a no-op. For cleanliness, the safest approach is to collect only expired root entries (those whose parents are not also expired in the same batch), then call `remove_entry_and_descendants` once per root, deduplicating the callback invocations.

## Proof of Concept

1. Submit parent tx `P` (with ≥2 outputs) via `send_transaction`. Record `t0 = now`.
2. Immediately submit children `C1`, `C2`, …, `CN`, each spending one output of `P`. Their timestamps are `t0 + ε` (ε > 0, e.g., a few milliseconds).
3. Wait until `t0 + expiry + δ` elapses (P's expiry fires) but before `t0 + ε + expiry` (children not yet expired).
4. Mine a block. `_update_tx_pool_for_reorg` → `remove_expired` fires.
5. Assert: `P` is absent from the pool; `C1…CN` are still present with status `Pending` or `Proposed`.
6. Attempt block assembly: the assembler cannot resolve `C1…CN`'s inputs (P's outputs are gone from both chain and pool). No block including any `Ci` can be produced.
7. Query pool stats: `pending_count` is inflated by N.
8. Repeat steps 1–4 with fresh UTXOs. After K cycles, the pool holds up to K×N stuck entries, causing legitimate transactions to be rejected by `limit_size`.

A unit test can be written in `tx-pool/src/component/tests/pending.rs` following the pattern of existing tests: construct a parent entry, add children, call `remove_entry` for the parent (simulating `remove_expired`), and assert that children remain in the pool with their input edges still present in `edges.inputs` while their inputs reference a non-existent output.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }
