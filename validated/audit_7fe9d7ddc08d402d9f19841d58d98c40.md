### Title
Incorrect Variable Reference in `DownloadScheduler::decrease` Causes Overly Aggressive Sync Penalty — (`sync/src/types/mod.rs`)

---

### Summary

`DownloadScheduler::decrease` reads from `self.task_count` instead of `self.timeout_count` when updating the timeout counter. Because `task_count` is always much larger than the threshold of `2`, the intended "accumulate N slow responses before penalizing" logic is bypassed: every single slow block response immediately decrements `task_count`, reducing a peer's download parallelism slot far more aggressively than designed.

---

### Finding Description

In `sync/src/types/mod.rs`, `DownloadScheduler` tracks how many blocks can be in-flight to a peer simultaneously via `task_count`. The `decrease` function is intended to act as a soft penalty: accumulate a `timeout_count` counter, and only reduce `task_count` once that counter exceeds `2`. [1](#0-0) 

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: reads task_count, not timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

The correct line should be:
```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

Because `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (a value well above `2`), the expression `self.task_count.saturating_add(num)` always produces a value greater than `2`. This means `timeout_count > 2` is always `true` on every call, so `task_count` is decremented on **every** slow block response instead of only after accumulating `> 2` slow responses.

`decrease` is called from `remove_by_block` when a block response falls in the `NormalToUpper` or `UpperToMax` time quantile: [2](#0-1) 

---

### Impact Explanation

`task_count` controls how many blocks can be requested in parallel from a peer. Driving it toward zero means the local node stops requesting blocks from that peer, effectively removing it from the download pool. The intended design requires `> 2` consecutive slow responses before penalizing; the bug causes a penalty on every single slow response — a 3× more aggressive reduction than intended.

A malicious sync peer can deliberately send slow block responses to drive the local node's `task_count` for that peer to near-zero, degrading IBD (Initial Block Download) and normal sync throughput. Combined with the `punish` function (which right-shifts `task_count`), the peer's slot can be zeroed out entirely. [3](#0-2) 

---

### Likelihood Explanation

`decrease` is triggered by any block response that falls in the slow time quantile, which is a normal network condition. Any unprivileged sync peer that sends responses slightly above the `normal_time` threshold will trigger this path. A malicious peer can do so deliberately and repeatedly. The bug fires on every such event without requiring any special privilege. [4](#0-3) 

---

### Recommendation

Change line 522 from:
```rust
self.timeout_count = self.task_count.saturating_add(num);
```
to:
```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

This restores the intended accumulation behavior: `task_count` is only decremented after `timeout_count` has accumulated more than `2` slow-response events.

---

### Proof of Concept

Assume `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32`, `num = 1`:

**With the bug:**
- Call 1: `timeout_count = 32 + 1 = 33 > 2` → `task_count = 31`, `timeout_count = 0`
- Call 2: `timeout_count = 31 + 1 = 32 > 2` → `task_count = 30`, `timeout_count = 0`
- … every slow response decrements `task_count` by 1

**With the fix:**
- Call 1: `timeout_count = 0 + 1 = 1`, not `> 2` → no change to `task_count`
- Call 2: `timeout_count = 1 + 1 = 2`, not `> 2` → no change to `task_count`
- Call 3: `timeout_count = 2 + 1 = 3 > 2` → `task_count = 31`, `timeout_count = 0`

A remote sync peer sending slow block responses triggers `decrease` via `remove_by_block`. Due to the bug, each slow response immediately reduces the local node's download parallelism for that peer, degrading sync performance in a way that is directly controllable by an unprivileged peer. [5](#0-4)
