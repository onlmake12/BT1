### Title
Unbounded `last_n_numbers` Allocation in `GetLastStateProofProcess::execute` When `last_n_blocks=0` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

`GetLastStateProofProcess::execute` contains a guard at line 201 intended to bound the total work per request. When an attacker sends `last_n_blocks=0` and `difficulties=[]`, the guard evaluates to `0 + 0*2 = 0 ≤ 1000` and passes. The subsequent branching logic at line 291 then enters the `else` path and constructs `last_n_numbers` as a range `(difficulty_boundary_block_number..last_block_number)`, which can span the entire chain height H. This causes O(H) allocations, database reads, and MMR root computations per single unauthenticated P2P message.

### Finding Description

**Limit check (line 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

`GET_LAST_STATE_PROOF_LIMIT = 1000`. With `last_n_blocks=0` and `difficulties=[]`, the expression is `0 + 0 = 0`, which never exceeds 1000. The guard is completely bypassed. [1](#0-0) [2](#0-1)

With `last_n_blocks=0` and `start_block_number=0`, the condition is `H <= 0`, which is `false` for any non-empty chain. The `else` branch is always taken. [3](#0-2) 

**Unbounded `last_n_numbers` construction (lines 299–319):**

Inside the `else` branch, `difficulty_boundary_block_number` is found via binary search. With `difficulty_boundary=U256::one()`, this resolves to block 0 (genesis has total difficulty ≥ 1). The adjustment guard at line 313 is:

```rust
if last_block_number - difficulty_boundary_block_number < last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
```

With `last_n_blocks=0`, the condition is `H < 0` — impossible for `u64`, so the adjustment never fires. Then:

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
```

This collects `H` elements (e.g., 10,000 on a 10,000-block chain). [4](#0-3) 

**O(H) work in `complete_headers` (lines 124–180):**

For every element in `last_n_numbers`, `complete_headers` performs:
- `snapshot.get_ancestor(last_hash, number)` — DB read
- `snapshot.get_block(hash)` — DB read
- `ancestor_block.calc_uncles_hash()` — hash computation
- `snapshot.chain_root_mmr(number - 1).get_root()` — MMR root computation

All of this is done synchronously per message, with no cap. [5](#0-4) 

### Impact Explanation

A single unauthenticated P2P message forces the server to allocate a `Vec<u64>` of H elements, perform H×2 database reads, H uncle hash computations, and H MMR root computations. On mainnet CKB (chain height ~14 million blocks), this is catastrophic per-message CPU and I/O amplification. Multiple concurrent connections sending this message would exhaust server resources.

### Likelihood Explanation

The attacker only needs:
1. A valid `last_hash` on the main chain — publicly available from any peer's `SendLastState` broadcast.
2. A TCP connection to a light-client-protocol-server node.
3. No PoW, no authentication, no rate limiting in this code path.

The message is small (fixed-size fields, empty `difficulties` array). The attack is trivially repeatable.

### Recommendation

The limit check must account for the actual size of `last_n_numbers`, not just `last_n_blocks`. Two concrete fixes:

1. **Enforce `last_n_blocks > 0` or add a minimum**: Reject messages where `last_n_blocks == 0` unless `difficulties` is also non-empty and provides a meaningful bound.
2. **Cap `last_n_numbers` after construction**: After computing `difficulty_boundary_block_number`, assert `last_block_number - difficulty_boundary_block_number <= last_n_blocks` (or some absolute cap like `GET_LAST_STATE_PROOF_LIMIT`), and return `InvalidRequest` if exceeded.
3. **Fix the limit formula**: Change the guard to also bound the range `last_block_number - start_block_number` against the limit, independent of `last_n_blocks`.
