Audit Report

## Title
RBF `replace_count` Overcounts Shared Descendants, Causing False Rejection of Legitimate Replacements - (File: `tx-pool/src/pool.rs`)

## Summary
In `TxPool::check_rbf`, the Rule #5 guard accumulates raw per-conflict descendant-set sizes without deduplicating descendants shared across multiple conflicting transactions. When two or more conflicts share a common descendant sub-graph, `replace_count` exceeds the true unique eviction count, causing the node to incorrectly reject a legitimate RBF attempt that would only evict ≤ 100 unique transactions. This allows an unprivileged attacker to permanently pin a victim's transaction in the mempool at negligible cost.

## Finding Description
`check_rbf` at `tx-pool/src/pool.rs` lines 613–624 iterates over every direct conflict and adds `descendants.len() + 1` to `replace_count` without any cross-iteration deduplication:

```rust
let mut replace_count: usize = 0;
for conflict in conflicts.iter() {
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    replace_count += descendants.len() + 1;   // ← raw per-conflict, no dedup
    if replace_count > MAX_REPLACEMENT_CANDIDATES {
        return Err(Reject::RBFRejected(...));
    }
    ...
}
``` [1](#0-0) 

`calc_descendants` returns an independent `HashSet<ProposalShortId>` per conflict. If two conflicts A and D share a descendant sub-graph (e.g., B spends outputs of both A and D, and B has children C₁…C₄₉), every shared descendant is counted once per conflict that owns it:

- `calc_descendants(A)` = {B, C₁…C₄₉} → 50 entries → `replace_count = 51`
- `calc_descendants(D)` = {B, C₁…C₄₉} → 50 entries → `replace_count = 102`
- `102 > 100` → `Reject::RBFRejected`

Unique transactions that would actually be evicted: {A, D, B, C₁…C₄₉} = 52 — well within the limit.

The fee-calculation path `calculate_min_replace_fee` at lines 101–115 correctly deduplicates via a `HashMap` keyed on `id`, so only the Rule #5 guard is broken. [2](#0-1)  The constant `MAX_REPLACEMENT_CANDIDATES = 100` is defined at line 33. [3](#0-2) 

## Impact Explanation
This is a targeted transaction-pinning denial-of-service. An attacker can block any victim's RBF replacement indefinitely using ~52 low-fee transactions. The victim cannot unstick their transaction via RBF while the attacker's diamond graph persists in the mempool. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (High, 10001–15000 points). The attack is cheap, repeatable, and can be applied to any RBF user simultaneously.

## Likelihood Explanation
Any unprivileged mempool submitter can execute this attack. Required steps: (1) observe which inputs the victim's original transaction spends (public mempool data); (2) submit a diamond-shaped dependency graph rooted at two transactions each conflicting with one of the victim's inputs; (3) the attack persists until the attacker's transactions are mined or evicted. No privileged access, no key material, and no majority hashpower is required. The entry path is the standard `send_transaction` RPC or P2P relay.

## Recommendation
Replace the per-conflict accumulation with a single deduplicated set across all conflicts, then check its cardinality once after the loop:

```rust
let mut all_to_replace: HashSet<ProposalShortId> = HashSet::new();
for conflict in conflicts.iter() {
    all_to_replace.insert(conflict.id.clone());
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    // existing ancestor-disjoint and input checks remain here
    all_to_replace.extend(descendants);
}
if all_to_replace.len() > MAX_REPLACEMENT_CANDIDATES {
    return Err(Reject::RBFRejected(format!(
        "Tx conflict with too many txs, conflict txs count: {}, expect <= {}",
        all_to_replace.len(), MAX_REPLACEMENT_CANDIDATES,
    )));
}
```

This mirrors the deduplication already applied in `calculate_min_replace_fee`. [4](#0-3) 

## Proof of Concept
1. Enable RBF (`min_rbf_rate > min_fee_rate`). [5](#0-4) 
2. Attacker submits: **A** (spends confirmed UTXO I₁), **D** (spends confirmed UTXO I₂), **B** (spends output of A and output of D), **C₁…C₄₉** (each spends B's output).
3. Victim submits replacement **N** spending I₁ and I₂ with a higher fee.
4. `check_rbf` finds `conflicts = [A, D]`. [6](#0-5) 
   - Iteration 1 (A): `descendants = {B, C₁…C₄₉}` (50 items), `replace_count = 51`
   - Iteration 2 (D): `descendants = {B, C₁…C₄₉}` (50 items), `replace_count = 102`
   - `102 > 100` → `Reject::RBFRejected("Tx conflict with too many txs…")`
5. Unique transactions that would be evicted: {A, D, B, C₁…C₄₉} = 52 — within the limit.
6. Victim's replacement is permanently blocked.

A unit test in `tx-pool` can construct this exact pool state and assert that `check_rbf` incorrectly returns `Err` for a replacement that should succeed.
