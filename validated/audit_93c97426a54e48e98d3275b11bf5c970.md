### Title
Two-Phase Commit Bypass via 80-bit ProposalShortId Birthday Collision — (`util/gen-types/src/extension/shortcut.rs`)

### Summary

`ProposalShortId::from_tx_hash` truncates the 32-byte transaction hash to 10 bytes (80 bits). `TwoPhaseCommitVerifier::verify` enforces the two-phase commit invariant exclusively using these 10-byte short IDs. An attacker who finds two distinct valid transactions sharing the same 10-byte hash prefix can propose one and commit the other, bypassing the invariant entirely.

### Finding Description

`ProposalShortId::from_tx_hash` copies only the first 10 bytes of the transaction hash: [1](#0-0) 

`TwoPhaseCommitVerifier::verify` builds both the committed-ID set and the proposal-ID set as `HashSet<ProposalShortId>` and performs a pure short-ID set-difference check: [2](#0-1) 

No full 32-byte hash comparison is performed anywhere in this code path. The entire two-phase commit gate reduces to an 80-bit identifier match.

**Attack steps:**

1. Attacker iterates over crafted transaction bodies (varying a witness or output data field) computing `blake2b(tx)[0..10]` until two valid transactions T_propose and T_commit share the same 10-byte prefix. Expected cost: ~2^40 blake2b operations (birthday bound on 80-bit space), achievable on a GPU cluster.
2. Attacker relays T_propose to the network; a miner includes it in a block's proposal zone. T_propose's `ProposalShortId` is now recorded in `proposal_txs_ids`.
3. Attacker relays T_commit. The miner's block assembler checks the proposal window using `proposal_short_id()` — T_commit's short ID matches T_propose's entry — so the miner includes T_commit in the commitment zone.
4. `TwoPhaseCommitVerifier::verify` computes `committed_ids` from T_commit's `proposal_short_id()`, finds it present in `proposal_txs_ids`, and returns `Ok(())`.

The invariant "every committed non-cellbase transaction must have been explicitly proposed" is violated: T_commit was never proposed, yet it passes the verifier.

### Impact Explanation

The two-phase commit mechanism is a core consensus rule in CKB. Bypassing it allows an attacker to commit a transaction that was never surfaced in the proposal zone, defeating the mechanism's purpose (giving the network visibility into pending transactions before commitment). Because all nodes use the same short-ID logic, the malformed block is accepted network-wide — there is no honest-node rejection to limit the damage. The committed transaction still passes all other validity checks (scripts, capacity), so it is fully applied to chain state.

### Likelihood Explanation

A birthday attack on 80 bits requires ~2^40 hash evaluations. Blake2b throughput on commodity GPU hardware exceeds 10^9 hashes/second, placing the expected search time in the range of hours to days on a modest cluster. The attacker controls transaction content (witnesses, output data) freely, so the search space is unconstrained. This is well within the capability of a motivated, well-resourced but otherwise unprivileged actor. No miner collusion is required: the miner's own block assembler uses the same short-ID lookup and will naturally include T_commit once T_propose is in the proposal window.

### Recommendation

Replace the 10-byte truncation with the full 32-byte transaction hash for proposal tracking, or increase the short ID to at least 20 bytes (160 bits) to make birthday attacks computationally infeasible. If backward compatibility requires keeping the 10-byte wire format, add a secondary full-hash index that is checked during `TwoPhaseCommitVerifier::verify`.
