### Title
Improper Error Handling via `.expect()` Panic in `get_block_filter` RPC Handler — (File: `rpc/src/module/chain.rs`)

---

### Summary

The `get_block_filter` RPC handler uses `.expect()` instead of proper error propagation when retrieving the block filter hash from the database. If the database contains filter data for a block but no corresponding filter hash (an inconsistent state that can arise from a partial write, crash during indexing, or migration bug), any unprivileged RPC caller who queries that block hash will trigger a panic in the RPC handler thread, causing an unhandled exception instead of a graceful error response.

---

### Finding Description

In `rpc/src/module/chain.rs`, the `get_block_filter` implementation reads filter data and then unconditionally calls `.expect()` on the filter hash lookup:

```rust
Ok(store.get_block_filter(&block_hash).map(|data| {
    let hash = store
        .get_block_filter_hash(&block_hash)
        .expect("stored filter hash");   // <-- panics if hash is absent
    BlockFilter {
        data: data.into(),
        hash: hash.into(),
    }
}))
``` [1](#0-0) 

The assumption encoded in the `.expect()` message — "stored filter hash" — is that filter data and filter hash are always written atomically. However, this invariant is not enforced at the type level, and any scenario that leaves the two stores out of sync (crash between two separate RocksDB column-family writes, a partial migration, or a storage bug) will cause the closure to panic rather than return a `Result::Err`.

The function signature is `fn get_block_filter(&self, block_hash: H256) -> Result<Option<BlockFilter>>`, so the correct behavior is to propagate an error (e.g., `RPCError::DatabaseIsCorrupt`) rather than panic.

---

### Impact Explanation

A panic inside an async RPC handler propagates as an unhandled exception in the Tokio task. Depending on the JSON-RPC server's task-supervision strategy, this can:

- Terminate the serving task for that connection, causing the caller to receive no response or a connection reset.
- If the panic is not caught at the task boundary, it can propagate to the Tokio runtime and crash the entire node process.

The net effect is **availability loss**: the node becomes unresponsive to RPC clients for the affected connection or entirely, constituting a denial-of-service condition for any operator or application relying on the `get_block_filter` endpoint.

---

### Likelihood Explanation

**Likelihood: Low–Medium.**

The trigger requires the database to be in an inconsistent state (filter data present, filter hash absent). This can occur:

1. During a node crash between the two separate writes that store filter data and filter hash.
2. During a database migration that processes one column family but not the other.
3. Due to a latent storage bug in the block-filter indexing path.

Once such a block exists in the database, **any unprivileged RPC caller** who queries `get_block_filter` with that block's hash will trigger the panic. The attacker does not need to create the inconsistent state — they only need to discover a block hash for which it exists (e.g., by iterating over recent block hashes after a known node restart or crash event).

---

### Recommendation

Replace the `.expect()` with proper error propagation using `ok_or_else`:

```rust
Ok(store.get_block_filter(&block_hash).map(|data| {
    let hash = store
        .get_block_filter_hash(&block_hash)
        .ok_or_else(|| RPCError::custom(
            RPCError::DatabaseIsCorrupt,
            format!("block filter hash missing for block {:#x}", block_hash),
        ))?;
    Ok(BlockFilter {
        data: data.into(),
        hash: hash.into(),
    })
}).transpose()?)
```

This ensures that a missing filter hash is surfaced as a structured RPC error rather than an unhandled panic, preserving node availability and giving callers actionable diagnostic information.

---

### Proof of Concept

1. Identify a CKB node that has block-filter indexing enabled and has experienced a crash or partial migration, leaving at least one block with filter data but no filter hash in the store.
2. Send the following JSON-RPC request with the affected block's hash:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_block_filter",
  "params": ["<block_hash_with_inconsistent_filter_state>"]
}
```

3. The RPC handler executes `store.get_block_filter_hash(&block_hash).expect("stored filter hash")`, which returns `None` and triggers a panic.
4. The Tokio task handling the request terminates with an unhandled exception; the caller receives no valid response, and depending on the runtime configuration, the node process may crash. [2](#0-1)
