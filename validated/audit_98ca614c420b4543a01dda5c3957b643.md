### Title
Missing `BLOCK_INVALID` Status Check in `HeaderAcceptor::accept()` Allows Repeated Re-verification of Already-Rejected Headers — (`File: sync/src/synchronizer/headers_process.rs`)

---

### Summary

`HeaderAcceptor::accept()` in the P2P sync layer checks whether a header's status is `BLOCK_INVALID` before running expensive verification, but the check is absent. The code contains a developer-acknowledged `FIXME` comment confirming this gap. An unprivileged peer can repeatedly send `SendHeaders` messages containing headers already cached as `BLOCK_INVALID`, forcing the node to re-execute full header verification — including PoW proof-of-work validation — on every delivery, with no short-circuit.

---

### Finding Description

In `sync/src/synchronizer/headers_process.rs`, the `HeaderAcceptor::accept()` function manages the per-header acceptance decision during the sync protocol's `SendHeaders` message handler. [1](#0-0) 

The function reads the cached block status and immediately returns if the header is already `HEADER_VALID`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer state and return
    return result;
}
``` [2](#0-1) 

The developer-written `FIXME` comment on lines 301–302 explicitly acknowledges that the `BLOCK_INVALID` early-return is missing. When a header's cached status is `BLOCK_INVALID`, the function falls through to:

1. `prev_block_check` — checks if the parent is `BLOCK_INVALID` (a hash-map lookup)
2. `non_contextual_check` — calls `HeaderVerifier::verify()`, which runs **PoW verification first**, then number, epoch, and timestamp checks
3. `version_check` — checks the version field [3](#0-2) 

`HeaderVerifier::verify()` always runs `PowVerifier` as its first step: [4](#0-3) 

`BLOCK_INVALID` is a disjoint bit flag (`1 << 12`) that does not overlap with `HEADER_VALID` (`1`), so `status.contains(BlockStatus::HEADER_VALID)` is always `false` for an invalid header, and the early-return is never triggered. [5](#0-4) 

By contrast, the compact block handler in `sync/src/relayer/compact_block_process.rs` **does** perform the `BLOCK_INVALID` check correctly:

```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
``` [6](#0-5) 

The `SendHeaders` handler accepts up to `MAX_HEADERS_LEN` headers per message (imported from `ckb_constant::sync`): [7](#0-6) 

---

### Impact Explanation

An unprivileged peer can send repeated `SendHeaders` P2P messages, each containing up to `MAX_HEADERS_LEN` headers whose hashes are already cached as `BLOCK_INVALID`. For each such header, the node re-executes the full `HeaderVerifier::verify()` pipeline — including the Eaglesong PoW hash computation — instead of performing a single hash-map lookup and returning. This constitutes a **CPU-exhaustion DoS** against the sync subsystem. The attacker needs only a single valid peer connection and a set of previously-rejected header hashes (obtainable by observing prior sync traffic or constructing minimally-invalid headers once). There is no rate-limit or ban applied to peers that send already-invalid headers through this path, because the function returns `ValidationState::Invalid` without a peer-penalty that would disconnect the sender.

---

### Likelihood Explanation

The attack requires only a standard peer connection — no privileged access, no key material, no majority hashpower. The attacker needs to know the hash of at least one header previously marked `BLOCK_INVALID` on the target node, which is trivially obtained by sending a single malformed header first and observing the rejection. The FIXME comment in the production source confirms the developers are aware of the gap but have not resolved it. Any node reachable on the P2P port is exposed.

---

### Recommendation

Add an explicit `BLOCK_INVALID` early-return at the top of `HeaderAcceptor::accept()`, mirroring the pattern already used in `compact_block_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

// Short-circuit on already-invalid headers to avoid redundant PoW re-verification.
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing peer-state update
    return result;
}
```

This resolves the acknowledged FIXME and aligns the `SendHeaders` path with the already-correct compact block path.

---

### Proof of Concept

1. Attacker connects to a CKB node as a standard sync peer.
2. Attacker sends a `SendHeaders` message containing one header `H` with an invalid block number (e.g., number field set to a value inconsistent with its parent). The node runs `HeaderVerifier::verify()`, fails at `NumberVerifier`, marks `H` as `BLOCK_INVALID` in `block_status_map`, and returns.
3. Attacker repeatedly sends `SendHeaders` messages each containing `H` (and up to `MAX_HEADERS_LEN - 1` copies of other `BLOCK_INVALID` headers).
4. For each delivery, `HeaderAcceptor::accept()` reads `status = BLOCK_INVALID`, does **not** short-circuit (because `BLOCK_INVALID` does not contain the `HEADER_VALID` bit), and re-executes `PowVerifier` + `NumberVerifier` + `EpochVerifier` + `TimestampVerifier` for every header.
5. The node's sync thread burns CPU proportional to `(messages/sec) × MAX_HEADERS_LEN × verification_cost_per_header` with no bound from the cached invalid status. [8](#0-7) [4](#0-3)

    pub fn accept(&self) -> ValidationResult {
        let mut result = ValidationResult::default();
        let sync_shared = self.active_chain.sync_shared();
        let state = self.active_chain.state();
        let shared = sync_shared.shared();

        // FIXME If status == BLOCK_INVALID then return early. But which error
        // type should we return?
        let status = self.active_chain.get_block_status(&self.header.hash());
        if status.contains(BlockStatus::HEADER_VALID) {
            let header_index = sync_shared
                .get_header_index_view(
                    &self.header.hash(),
                    status.contains(BlockStatus::BLOCK_STORED),
                )
                .unwrap_or_else(|| {
                    panic!(
                        "header {}-{} with HEADER_VALID should exist",
                        self.header.number(),
                        self.header.hash()
                    )
                })
                .as_header_index();
            state
                .peers()
                .may_set_best_known_header(self.peer, header_index);
            return result;
        }

        if self.prev_block_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-parent header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        if let Some(is_invalid) = self.non_contextual_check(&mut result).err() {
            debug!(
                "HeadersProcess rejected non-contextual header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            if is_invalid {
                shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            }
            return result;
        }

        if self.version_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-version header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        sync_shared.insert_valid_header(self.peer, self.header);
        result
    }

use bitflags::bitflags;

    pub struct BlockStatus: u32 {
        const UNKNOWN                 =     0;

        const HEADER_VALID            =     1;
        const BLOCK_RECEIVED          =     1 | (Self::HEADER_VALID.bits() << 1);
        const BLOCK_STORED            =     1 | (Self::BLOCK_RECEIVED.bits() << 1);
        const BLOCK_VALID             =     1 | (Self::BLOCK_STORED.bits() << 1);

        const BLOCK_INVALID           =     1 << 12;
    }
