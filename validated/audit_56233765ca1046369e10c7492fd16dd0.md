### Title
Unbounded `forward_rate_limiter` HashMap Growth via Persistent Connection with Distinct Attacker-Controlled PeerId Pairs — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

The `HolePunching` protocol's `forward_rate_limiter` (`HashMapStateStore<(PeerId, PeerId, u32)>`) grows without bound when a remote peer maintains a persistent TCP connection and sends messages with distinct attacker-controlled `(from, to)` PeerId pairs. The only cleanup call, `retain_recent()`, is placed exclusively in the `disconnected()` handler and is never invoked by the periodic `notify` timer, violating the invariant that per-node rate-limiter memory must remain bounded regardless of connection lifetime.

---

### Finding Description

`HolePunching` declares `forward_rate_limiter` as a `governor::RateLimiter` backed by `HashMapStateStore<(PeerId, PeerId, u32)>`: [1](#0-0) 

Every call to `check_key` in `ConnectionSyncProcess::execute` (and the other two message processors) inserts a new entry into this HashMap for each unique `(from, to, msg_item_id)` triple: [2](#0-1) 

The `from` and `to` fields are parsed directly from attacker-controlled message bytes with no validation against actually-connected peers: [3](#0-2) 

The only place `retain_recent()` is called on `forward_rate_limiter` is in `disconnected()`: [4](#0-3) 

The periodic `notify` handler (fired every 5 minutes via `CHECK_INTERVAL`) cleans up `pending_delivered` and `inflight_requests` but **never** calls `retain_recent()` on either rate limiter: [5](#0-4) 

The first-layer `rate_limiter` (keyed by `(session_id, msg_item_id)`) limits throughput to 30 messages/second per session: [6](#0-5) 

This throttles insertion rate but does **not** bound the total number of distinct keys in `forward_rate_limiter`. Each of the 30 allowed messages per second can carry a fresh random `(from, to)` pair, creating a new HashMap entry every time.

---

### Impact Explanation

At 30 inserts/second with a persistent connection:

- Each governor HashMap entry holds two `PeerId` values (~39 bytes each) plus internal state: roughly 150–200 bytes per entry.
- After 1 hour: ~108,000 entries ≈ ~20 MB per attacker session.
- After 24 hours: ~2.6M entries ≈ ~500 MB per attacker session.
- Multiple concurrent attacker sessions multiply this linearly.

Since `retain_recent()` is never called while the connection is alive, all stale entries accumulate indefinitely. This leads to OOM on any CKB node acting as a relay for `ConnectionSync` messages, crashing the node and disrupting the network.

---

### Likelihood Explanation

The attack requires only a standard TCP connection to a CKB node with the HolePunching protocol enabled. No authentication, PoW, or privileged access is needed. The attacker simply sends well-formed `ConnectionSync` messages with random `from`/`to` PeerId bytes at the maximum allowed rate. The attack is slow but persistent and requires no special tooling beyond a basic P2P client.

---

### Recommendation

Call `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside the `notify` handler, which already fires every 5 minutes via `CHECK_INTERVAL`: [5](#0-4) 

This ensures stale entries are evicted periodically regardless of whether any peer disconnects, bounding memory to at most `30 req/s × 300 s × entry_size ≈ 1.8 MB` between cleanup cycles.

---

### Proof of Concept

```
1. Connect to a CKB node over TCP, open the HolePunching protocol session.
2. In a loop, send well-formed ConnectionSync messages at 30/second,
   each with a freshly generated random (from, to) PeerId pair.
3. Never disconnect.
4. After N seconds, the forward_rate_limiter HashMap contains N*30 entries.
5. Measure HolePunching struct heap size: it grows O(N) with no bound.
6. After ~24 hours, node RSS has grown by ~500 MB per attacker session;
   multiple concurrent sessions exhaust available memory and crash the node.
```
