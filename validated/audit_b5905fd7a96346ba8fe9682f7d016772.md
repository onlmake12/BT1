Audit Report

## Title
Unbounded `OFFSET` Injection via Crafted Pagination Cursor in `get_transactions` (Ungrouped) — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
The `get_transactions` RPC endpoint decodes the last 4 bytes of the `after` cursor as an `i32` and passes it verbatim as `OFFSET` to the underlying SQL query with no upper-bound validation. An unprivileged caller can supply `i32::MAX` (2,147,483,647) as the offset, forcing the database to scan and discard up to 2 billion rows before returning an empty result set. Repeated calls can exhaust the database connection pool (PostgreSQL) or serialize all DB operations (SQLite), rendering the indexer RPC unavailable.

## Finding Description
**Exact call chain:**

1. `AsyncRichIndexerHandle::get_transactions` accepts `after: Option<JsonBytes>` and validates only that its length is exactly 12 bytes (`get_transactions.rs` L45).
2. The last 4 bytes are decoded via `decode_i32` (`mod.rs` L286–294), which only validates byte length — no range check is performed on the resulting value.
3. The decoded `offset` is passed directly to `query_builder.offset(offset)` (`get_transactions.rs` L181) without any clamping or validation.
4. The existing `limit` guard (`get_transactions.rs` L23–32) only bounds the number of *returned* rows; it does not constrain the offset scan cost.

The generated SQL becomes `SELECT ... WHERE tx_id >= <last> ORDER BY tx_id ASC LIMIT <limit> OFFSET 2147483647`, forcing a full sequential scan of all matching rows before discarding them. Setting `last = 0` (first 8 bytes of the cursor) maximises the number of rows the DB must scan.

## Impact Explanation
This maps to **Note (0–500 points): Any local RPC API crash**. The rich-indexer RPC becomes unavailable to callers: under PostgreSQL, repeated concurrent crafted requests exhaust the `sqlx` connection pool, causing all new RPC requests to queue indefinitely or fail; under SQLite, the long-running read lock serialises all concurrent DB operations, stalling indexer writes and other RPC queries. The CKB node's core consensus and P2P layers are unaffected — this is scoped to the local indexer RPC service.

## Likelihood Explanation
The `after` cursor is a documented public pagination parameter. Any caller with network access to the RPC port can craft the 12-byte value with no authentication, proof-of-work, or privileged role. Indexer nodes are routinely exposed publicly. The attack is trivially repeatable in a tight loop.

## Recommendation
Add an upper-bound check on the decoded `offset` immediately after `decode_i32` in `get_transactions.rs`:

```rust
let offset = decode_i32(offset)?;
if offset < 0 {
    return Err(Error::Params("cursor offset must be non-negative".to_string()));
}
if offset as usize > self.request_limit {
    return Err(Error::Params(format!(
        "cursor offset must not exceed {}",
        self.request_limit
    )));
}
```

Alternatively, redesign the cursor to use a keyset-only scheme (store only the last `tx_id` and `io_index`/`io_type` as a tie-breaker) so that `OFFSET` is never needed and pagination is O(log N) regardless of cursor content.

## Proof of Concept

```python

    ]
```

```rust
fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
```
