### Title
`check_tx_fee` Uses Raw Serialized Size Instead of Transaction Weight for Minimum Fee Enforcement, Allowing High-Cycle Transactions to Bypass `min_fee_rate` - (File: `tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee by passing raw serialized byte size directly to `FeeRate::fee()`, which expects a **weight** value. Because CKB transaction weight is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`, a transaction with small size but high cycles has a weight far larger than its size. The fee check therefore underestimates the required minimum fee for such transactions, allowing them to enter the pool with an effective fee rate far below `min_fee_rate`.

---

### Finding Description

`FeeRate` is defined as **shannons per kilo-weight** (shannons/KW), where weight = `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. [1](#0-0) 

`get_transaction_weight` formalises this: [2](#0-1) 

`check_tx_fee`, however, passes raw `tx_size` (bytes) directly to `min_fee_rate.fee()` as if it were the weight: [3](#0-2) 

The comment even acknowledges the mismatch ("Theoretically we cannot use size as weight directly") but treats it as an acceptable approximation. It is not: for a transaction where cycles dominate, the weight can be orders of magnitude larger than the size, making the approximation arbitrarily wrong.

`FeeRate::fee(weight)` computes `fee_rate × weight / 1000`: [4](#0-3) 

So `min_fee = min_fee_rate × tx_size / 1000` instead of the correct `min_fee_rate × max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES) / 1000`.

---

### Impact Explanation

An unprivileged RPC caller submitting a transaction with:
- Small serialized size (e.g., 200 bytes)
- Near-maximum cycles (e.g., 70,000,000 — the default `max_tx_verify_cycles`)
- Fee just above `min_fee_rate × size / 1000`

passes `check_tx_fee` with a fee of ~200 shannons (for `min_fee_rate = 1000`), while the correct weight-based minimum fee would be:

```
weight = max(200, 70_000_000 × 0.000_170_571_4) ≈ 11,940
min_fee_correct = 1000 × 11940 / 1000 = 11,940 shannons
```

The transaction is admitted to the pool at ~1/60th of the required fee rate. This allows:
1. **Bypassing the `min_fee_rate` filter** for compute-heavy transactions.
2. **Filling the tx-pool** with artificially cheap, high-cycle transactions, consuming pool capacity and forcing eviction of legitimately-priced transactions.
3. **Degrading miner revenue** by polluting the pool with transactions that will not be mined at a fair fee rate. [2](#0-1) 

---

### Likelihood Explanation

The attack requires only a standard `send_transaction` RPC call — no privileged access, no key material, no majority hashpower. Any node with the RPC pool module enabled (the default) is reachable. The attacker needs only to craft a transaction with a large script that consumes many cycles (e.g., a loop in a lock script) while keeping the serialized transaction small. This is straightforward with CKB-VM scripting. [5](#0-4) 

---

### Recommendation

Replace the size-only approximation in `check_tx_fee` with the actual transaction weight:

```rust
// Use actual weight, not raw size
let weight = get_transaction_weight(tx_size, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

The `cycles` value is available from the `CacheEntry` / `Completed` result returned by the verifier. If cycles are not yet known at the point of the cheap check, the check should be deferred until after verification, or the size-only check should be documented as a pre-filter only and a second weight-based check added post-verification.

---

### Proof of Concept

1. Craft a CKB transaction whose lock script runs a tight loop consuming ~70,000,000 cycles. Keep the serialized transaction small (e.g., 200 bytes).
2. Compute `min_fee_rate` (default 1,000 shannons/KW). Set the transaction fee to 201 shannons (just above `1000 × 200 / 1000`).
3. Submit via `send_transaction` RPC.
4. Observe the transaction is accepted into the pool despite its effective fee rate being `201 / 11940 × 1000 ≈ 16.8 shannons/KW` — far below the 1,000 shannons/KW minimum.
5. Repeat to fill the pool with such transactions, evicting legitimately-priced transactions. [3](#0-2) [6](#0-5)

use crate::core::Capacity;

/// shannons per kilo-weight
#[derive(Clone, Copy, Default, Debug, PartialEq, Eq, PartialOrd, Ord)]
pub struct FeeRate(pub u64);

const KW: u64 = 1000;

impl FeeRate {
    /// Calculates the fee rate from a total fee and weight.
    pub fn calculate(fee: Capacity, weight: u64) -> Self {
        if weight == 0 {
            return FeeRate::zero();
        }
        FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
    }

pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {

    )

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
