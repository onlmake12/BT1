### Title
Entropy `revealWithCallback` Old Code Path Calls `_entropyCallback` Without Interface Check or Try/Catch, Permanently Locking Requests — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `revealWithCallback` function has two execution paths. The **new path** (when `req.gasLimit10k != 0`) uses `excessivelySafeCall` to safely invoke `_entropyCallback` and handles reverts gracefully. The **old path** (when `req.gasLimit10k == 0`, i.e., the provider has not set a `defaultGasLimit`) calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly with no try/catch and no check that the requester actually implements `IEntropyConsumer`. If the requester contract does not implement `_entropyCallback`, every `revealWithCallback` call for that request reverts, permanently locking the request in an unfulfillable state.

---

### Finding Description

In `Entropy.sol`, `revealWithCallback` branches on `req.gasLimit10k`:

```solidity
if (
    req.gasLimit10k != 0 &&
    req.callbackStatus == EntropyStatusConstants.CALLBACK_NOT_STARTED
) {
    // NEW PATH: uses excessivelySafeCall — catches reverts gracefully
    (success, ret) = req.requester.excessivelySafeCall(...);
    ...
} else {
    // OLD PATH: direct call, no try/catch
    address callAddress = req.requester;
    ...
    clearRequest(provider, sequenceNumber);   // ← cleared BEFORE callback
    ...
    if (len != 0) {
        IEntropyConsumer(callAddress)._entropyCallback(  // ← no interface check
            sequenceNumber,
            provider,
            randomNumber
        );
    }
}
``` [1](#0-0) 

The old path is entered whenever `req.gasLimit10k == 0`, which is set in `requestHelper` when `providerInfo.defaultGasLimit == 0`:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;
}
``` [2](#0-1) 

The `clearRequest` call precedes the callback. If `_entropyCallback` reverts (because the requester does not implement `IEntropyConsumer`), the entire transaction reverts — including the `clearRequest` — leaving the request permanently active. Every subsequent `revealWithCallback` attempt by the provider will also revert, making the request permanently unfulfillable.

By contrast, the new path uses `excessivelySafeCall` and transitions the request to `CALLBACK_FAILED` state on revert, allowing recovery: [3](#0-2) 

The `IEntropyConsumer` interface is an abstract contract, not an ERC-165-registered interface, so there is no on-chain mechanism to verify support before calling: [4](#0-3) 

---

### Impact Explanation

Any contract that calls `requestV2` (or `requestWithCallback`) against a provider with `defaultGasLimit == 0` but does not implement `IEntropyConsumer._entropyCallback` will have its request permanently stuck. The provider's `revealWithCallback` will always revert for that sequence number. The requester's fee (already credited to the provider and Pyth in `requestHelper`) is non-refundable, and the randomness is never delivered. There is no cancel/refund path in the contract. [5](#0-4) 

---

### Likelihood Explanation

The old code path is reachable today for any provider that has not called `setDefaultGasLimit`. A requester contract that calls `requestV2` without inheriting `IEntropyConsumer` (e.g., a contract that uses a raw low-level call or a proxy that strips the selector) will trigger this condition. The documentation recommends implementing `IEntropyConsumer` but the contract does not enforce it at request time.

---

### Recommendation

Apply the same `excessivelySafeCall` / try-catch pattern used in the new path to the old path as well, so that a missing or reverting `_entropyCallback` does not cause `revealWithCallback` to revert. Alternatively, validate at `requestV2` time that `msg.sender` exposes the `_entropyCallback` selector (e.g., via a static call probe or ERC-165 if `IEntropyConsumer` is extended to support it).

---

### Proof of Concept

1. Deploy a provider with `defaultGasLimit == 0` (never called `setDefaultGasLimit`).
2. Deploy a requester contract that calls `entropy.requestV2{value: fee}()` but does **not** inherit `IEntropyConsumer` (no `_entropyCallback` function).
3. The provider calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`.
4. Execution enters the old path (`req.gasLimit10k == 0`), calls `clearRequest`, then calls `IEntropyConsumer(requester)._entropyCallback(...)`.
5. The call reverts (no such function on requester). The entire transaction reverts, including `clearRequest`.
6. The request remains active. Every subsequent `revealWithCallback` attempt reverts identically.
7. The requester's fee is permanently lost; the randomness is never delivered.

// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

abstract contract IEntropyConsumer {
    // This method is called by Entropy to provide the random number to the consumer.
    // It asserts that the msg.sender is the Entropy contract. It is not meant to be
    // override by the consumer.
    function _entropyCallback(
        uint64 sequence,
        address provider,
        bytes32 randomNumber
    ) external {
        address entropy = getEntropy();
        require(entropy != address(0), "Entropy address not set");
        require(msg.sender == entropy, "Only Entropy can call this function");

        entropyCallback(sequence, provider, randomNumber);
    }

    // getEntropy returns Entropy contract address. The method is being used to check that the
    // callback is indeed from Entropy contract. The consumer is expected to implement this method.
    // Entropy address can be found here - https://docs.pyth.network/entropy/contract-addresses
    function getEntropy() internal view virtual returns (address);

    // This method is expected to be implemented by the consumer to handle the random number.
    // It will be called by _entropyCallback after _entropyCallback ensures that the call is
    // indeed from Entropy contract.
    function entropyCallback(
        uint64 sequence,
        address provider,
        bytes32 randomNumber
    ) internal virtual;
}
