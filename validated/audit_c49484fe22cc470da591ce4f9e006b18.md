Unbounded `while` Loop in `Echo.executeCallback` Enables Permanent Fund Lock via Sequence Number Exhaustion — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that linearly scans all fulfilled sequence numbers to advance `_state.firstUnfulfilledSeq`. An unprivileged attacker can create a large number of requests after a victim's request, fulfill them all (recovering their fees), and leave the victim's request in a state where every subsequent `executeCallback` attempt exhausts block gas inside the loop before the request is ever cleared. Because no cancel or refund path exists, the victim's locked ETH is permanently unrecoverable.

---

### Finding Description

**Root cause — `Echo.sol` lines 169–174:**

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing
// a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

`findRequest` performs two SLOADs per iteration: one into the fixed `Request[32]` array and, for any sequence number whose hash slot is occupied by a different request, a second cold SLOAD into `requestsOverflow` (a `mapping(bytes32 => Request)`). [1](#0-0) 

**Storage layout — `EchoState.sol`:**

The fixed array has only 32 slots (`NUM_REQUESTS = 32`). Any request whose `shortKey` collides with an already-occupied slot is evicted to `requestsOverflow`. After the first 32 distinct short-keys are touched in a single transaction, every subsequent `findRequest` call for an overflow entry costs a cold SLOAD (~2 100 gas). [2](#0-1) 

**Execution order inside `executeCallback`:**

```
1. credit provider fees          (line 161)
2. clearRequest(sequenceNumber)  (line 164)
3. while loop                    (lines 169-174)   ← can OOG here
4. try _echoCallback             (line 176)
```

If the `while` loop runs out of gas, the **entire transaction reverts**, undoing steps 1 and 2. The request is restored to active, but the loop condition is identical on every retry — the loop will always exhaust gas, so the request can never be cleared. [3](#0-2) 

The contract itself acknowledges the risk in a TODO comment at line 155:

> *"if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract."* [4](#0-3) 

**No cancel/refund path exists.** The only public state-mutating functions are `requestPriceUpdatesWithCallback`, `executeCallback`, `withdrawFees` (admin-only), `withdrawAsFeeManager`, `registerProvider`, and `setProviderFee`. There is no `cancelRequest` or user-facing refund function. [5](#0-4) 

---

### Impact Explanation

**Impact: High.**

A victim's ETH deposited via `requestPriceUpdatesWithCallback` is permanently locked with no recovery path. The `req.fee` field is stored in the `Request` struct and is only released inside `executeCallback` after `clearRequest` succeeds — which requires the `while` loop to complete first. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: Low.**

The attacker must create approximately **N ≈ 14 000–15 000** requests after the victim's request and fulfill them all. Each fulfilled request recovers its fee; the net attacker cost is only the gas for ~28 000 transactions (create + fulfill). At 30 M block gas limit and ~2 100 gas per cold SLOAD per loop iteration, the loop exhausts gas at roughly that threshold. This is expensive but achievable by a motivated, well-funded attacker targeting a high-value victim request. [7](#0-6) 

---

### Recommendation

1. **Remove the unbounded scan from `executeCallback`.** Replace `firstUnfulfilledSeq` maintenance with a doubly-linked list of active requests (as the TODO comment already suggests), so advancing the pointer is O(1).
2. **Add a `cancelRequest` function** that allows the original requester to reclaim their locked fee after a timeout, providing a safety valve regardless of the loop fix.
3. **Cap the loop** as an immediate short-term mitigation: add a `maxIterations` guard so the loop exits after a bounded number of steps, preventing OOG reverts even if the pointer is not fully advanced.

### Proof of Concept

```
1. Victim calls requestPriceUpdatesWithCallback(provider, t, priceIds, gasLimit)
   → seq = 1, victim's ETH locked in contract

2. Attacker calls requestPriceUpdatesWithCallback(...) × 15 000
   → seq = 2 … 15 001, attacker's ETH locked

3. Attacker calls executeCallback(seq=2), executeCallback(seq=3), …, executeCallback(seq=15001)
   → Each call: while loop stops immediately (seq=1 is still active), callback fires, attacker recovers fees
   → firstUnfulfilledSeq remains = 1

4. Provider calls executeCallback(seq=1):
   → clearRequest(1) executes (seq 1 now inactive)
   → while loop: findRequest(1) inactive → advance; findRequest(2) inactive → advance; …
   → After ~14 000 cold SLOADs × 2 100 gas ≈ 29.4 M gas → OOG
   → Entire tx reverts; seq 1 restored to active

5. Every subsequent executeCallback(seq=1) attempt repeats step 4 and reverts.
   Victim's ETH is permanently locked.
``` [8](#0-7)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);

        // TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
        // a better solution would be a doubly-linked list of active requests.
        // After successful callback, update firstUnfulfilledSeq if needed
        while (
            _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
            !isActive(findRequest(_state.firstUnfulfilledSeq))
        ) {
            _state.firstUnfulfilledSeq++;
        }

        try
            IEchoConsumer(req.requester)._echoCallback{
                gas: req.callbackGasLimit
            }(sequenceNumber, priceFeeds)
        {
            // Callback succeeded
            emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
        } catch Error(string memory reason) {
            // Explicit revert/require
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                reason
            );
        } catch {
            // Out of gas or other low-level errors
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                "low-level error (possibly out of gas)"
            );
        }
    }

interface IEcho is EchoEvents {

     * @notice Requests price updates with a callback
     * @dev The msg.value must be equal to getFee(callbackGasLimit)

     * @param publishTime The minimum publish time for price updates, it should be less than or equal to block.timestamp + 60
     * @param priceIds The price feed IDs to update. Maximum 10 price feeds per request.
     *        Requests requiring more feeds should be split into multiple calls.
     * @param callbackGasLimit The amount of gas allocated for the callback execution
     * @return sequenceNumber The sequence number assigned to this request
     * @dev Security note: The 60-second future limit on publishTime prevents a DoS vector where
     *      attackers could submit many low-fee requests for far-future updates when gas prices

     *      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
     *      the fee estimation unreliable.

        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit

     * @notice Executes the callback for a price update request
     * @dev Requires 1.5x the callback gas limit to account for cross-contract call overhead

     * @param providerToCredit The provider to credit for fulfilling the request. This may not be the provider that submitted the request (if the exclusivity period has elapsed).
     * @param sequenceNumber The sequence number of the request
     * @param updateData The raw price update data from Pyth
     * @param priceIds The price feed IDs to update, must match the request

     * @notice Gets the base fee charged by Pyth protocol

     * @return pythFeeInWei The base fee in wei that every request must pay

    function getPythFeeInWei() external view returns (uint96 pythFeeInWei);

     * @param provider The provider to fulfill the request
