### Title
Delegators Can Escape OIS Slashing by Unstaking Before Slash Execution — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The Oracle Integrity Staking (OIS) `slash` instruction applies slashing to a delegator's *current* on-chain positions at the time of execution. Because the Pythian Council's investigation and slash creation process takes up to two full epochs (~14 days), and because delegators can freely unstake at any time, a delegator who was staked during a misprint epoch can observe the public forum report, unstake before the slash is executed, and escape the penalty entirely — then re-stake afterward.

---

### Finding Description

The OIS slashing lifecycle has a mandatory multi-epoch delay:

1. A misprint occurs in epoch T.
2. The Pythian Council has until the end of epoch T+1 to investigate and call `create_slash_event`.
3. The `slash` instruction is then called per-delegator to apply the penalty.

The `SlashEvent` struct records the incident `epoch`, `slash_ratio`, and `slash_custody`: [1](#0-0) 

The `slash` instruction operates on the delegator's current `stake_account_positions` account at execution time: [2](#0-1) 

There is no snapshot or historical record of what a delegator held during the incident epoch. The `DelegationRecord` PDA only tracks `last_epoch` and `next_slash_event_index` — it does not record the historical staked amount at the time of the misprint: [3](#0-2) 

A delegator who calls `closePosition` (via the staking program) before the `slash` instruction is executed moves their positions to `PREUNLOCKING` or `UNLOCKING` state. The `getPositionState` logic confirms these are distinct from `LOCKED`: [4](#0-3) 

If the `slash` instruction only reduces currently `LOCKED` positions (the only positions that are "active" in the pool), a delegator in cooldown holds 0 active delegation — the slash either applies to 0 tokens or fails to find slashable balance, and the delegator escapes the penalty.

The slashing rulebook explicitly states the intended target is tokens staked *during the incident epoch*: [5](#0-4) 

But the on-chain `slash` instruction has no mechanism to enforce this retroactively against accounts that have since unstaked.

---

### Impact Explanation

- Delegators (and publishers) who monitor the public Pythian Council forum can front-run slash execution by unstaking immediately after a misprint report is posted.
- The slash is applied to 0 tokens for the escaping delegator, meaning the full slashing burden falls on remaining stakers who did not front-run.
- This directly undermines the security guarantee of OIS: stakers are supposed to have skin-in-the-game for the epoch they were staked. Sophisticated actors can participate in OIS for rewards while bearing no real slashing risk.
- The `poolData.num_slash_events` counter per publisher still increments, but the escaping delegator's `delegation_record.next_slash_event_index` advances past the slash event with 0 tokens slashed. [6](#0-5) 

---

### Likelihood Explanation

**High.** The Pythian Council is required to post publicly on the Pyth DAO forum before executing a slash. The investigation window is up to one full epoch (~7 days) after the incident epoch, giving delegators ample time to observe the report and unstake. The OIS cooldown is 1–2 epochs, meaning tokens can be fully withdrawn before the slash is executed. No on-chain mechanism prevents unstaking while a slash event is pending or being investigated. [7](#0-6) 

---

### Recommendation

1. **Snapshot-based slashing**: Record each delegator's staked amount at the start of each epoch (or at the time of `advance_delegation_record`). When a `SlashEvent` is created for epoch T, apply the slash ratio to the snapshotted amount from epoch T, not the current balance.

2. **Slash-pending lock**: When `create_slash_event` is called for a publisher, prevent delegators of that publisher from unstaking until all pending slash events have been applied to their account (i.e., require `delegation_record.next_slash_event_index == pool_data.num_slash_events[publisher_index]` before allowing `closePosition`).

3. **Apply slash to cooldown positions**: Treat `PREUNLOCKING` and `UNLOCKING` positions as still slashable for slash events whose `epoch` predates the position's `unlockingStart`.

---

### Proof of Concept

1. Alice delegates 1,000,000 PYTH to Publisher P during epoch T.
2. Publisher P contributes to a misprint in epoch T. The Pythian Council posts a forum report at the start of epoch T+1.
3. Alice reads the forum post and immediately calls `closePosition` on her delegation to Publisher P. Her position enters `PREUNLOCKING` state (cooldown).
4. The Pythian Council calls `create_slash_event` for Publisher P with `slash_ratio = 500` (5%) during epoch T+1. `pool_data.num_slash_events[P_index]` increments to 1.
5. Anyone calls `slash` on Alice's `stake_account_positions`. Alice's active delegation to P is now 0 (she is in cooldown). The slash applies to 0 tokens.
6. Alice's `delegation_record.next_slash_event_index` advances to 1. At the end of epoch T+2, Alice's cooldown ends, she withdraws her 1,000,000 PYTH intact, and re-stakes in epoch T+3.
7. Alice earned rewards during epoch T and paid 0 slashing penalty. The 50,000 PYTH that should have been slashed from Alice is instead borne by remaining delegators. [8](#0-7) [9](#0-8)

```typescript
export const getPositionState = (
  position: Position,
  currentEpoch: bigint,
): PositionState => {
  if (currentEpoch < position.activationEpoch) {
    return PositionState.LOCKING;
  }
  if (!position.unlockingStart) {
    return PositionState.LOCKED;
  }
  const hasActivated = position.activationEpoch <= currentEpoch;
  const unlockStarted = position.unlockingStart <= currentEpoch;
  const unlockEnded = position.unlockingStart + 1n <= currentEpoch;

  if (hasActivated && !unlockStarted) {
    return PositionState.PREUNLOCKING;
  } else if (unlockStarted && !unlockEnded) {
    return PositionState.UNLOCKING;
  } else {
    return PositionState.UNLOCKED;
  }
};
```
