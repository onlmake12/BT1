### Title
Entropy `admin` Role Has Excessive Permissions â€” Can Drain All Accrued Pyth Protocol Fees â€” (`File: target_chains/ethereum/contracts/contracts/entropy/EntropyUpgradable.sol`)

---

### Summary

`EntropyUpgradable` establishes two distinct roles: an `owner` with full power and an `admin` explicitly documented as able to "set some config parameters only." However, `_authoriseAdminAction()` grants the `admin` the same access as the `owner` to `withdrawFee`, allowing the admin to drain the entire `accruedPythFeesInWei` balance to any arbitrary address. This mirrors the external report's pattern exactly: a restricted role has excessive permissions that enable fund drainage.

---

### Finding Description

`EntropyUpgradable.sol` documents the role separation clearly:

> "The contract will have an owner and an admin. The owner will have all the power over it. The admin can set some config parameters only." [1](#0-0) 

Despite this stated intent, `_authoriseAdminAction()` is implemented as:

```solidity
function _authoriseAdminAction() internal view override {
    if (msg.sender != owner() && msg.sender != _state.admin)
        revert EntropyErrors.Unauthorized();
}
``` [2](#0-1) 

This check gates **all** governance functions equally â€” including `withdrawFee`:

```solidity
function withdrawFee(address targetAddress, uint128 amount) external {
    require(targetAddress != address(0), "targetAddress is zero address");
    _authoriseAdminAction();

    if (amount > _state.accruedPythFeesInWei)
        revert EntropyErrors.InsufficientFee();

    _state.accruedPythFeesInWei -= amount;
    (bool success, ) = targetAddress.call{value: amount}("");
    require(success, "Failed to withdraw fees");
    emit FeeWithdrawn(targetAddress, amount);
}
``` [3](#0-2) 

`accruedPythFeesInWei` accumulates from every randomness request paid by users:

```solidity
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [4](#0-3) 

The `EntropyState` comment reinforces the design intent: "Admin has the rights to update pyth configs" â€” fee withdrawal is not a config update. [5](#0-4) 

The admin can also call `setPythFee` to inflate the Pyth fee, accelerating fee accumulation before draining it, and `proposeAdmin` to rotate the admin key â€” all capabilities that exceed "config parameters only." [6](#0-5) 

---

### Impact Explanation

A compromised or malicious `admin` key can call `withdrawFee(attackerAddress, accruedPythFeesInWei)` in a single transaction, transferring the entire accumulated Pyth protocol fee balance (native ETH/gas token) to an attacker-controlled address. These fees accumulate continuously from all Entropy users across every deployed chain. The impact is **direct theft of protocol revenue** with no recovery path once executed.

---

### Likelihood Explanation

The `admin` role is a hot-wallet operational key (separate from the multisig `owner`), making it a higher-value target for compromise than the owner. The separation of roles implies the admin is not considered absolutely trustworthy â€” yet it holds fund-withdrawal capability. A single key compromise, insider threat, or phishing of the admin key is sufficient. The `testWithdrawFeeByAdmin` test in `EntropyAuthorized.t.sol` confirms this path is exercised and passes. [7](#0-6) 

---

### Recommendation

Restrict `withdrawFee` (and optionally `proposeAdmin`) to `onlyOwner`. The `admin` role should be limited to true config parameters: `setPythFee` and `setDefaultProvider`. This matches the documented design intent and eliminates the excessive permission.

```solidity
function withdrawFee(address targetAddress, uint128 amount) external {
    require(targetAddress != address(0), "targetAddress is zero address");
    // Only owner should be able to withdraw protocol fees
    if (msg.sender != owner()) revert EntropyErrors.Unauthorized();
    ...
}
```

---

### Proof of Concept

1. Deploy `EntropyUpgradable` with `owner = multisig`, `admin = hotWallet`.
2. Users call `request()`/`requestWithCallback()` over time; `accruedPythFeesInWei` grows.
3. Attacker compromises `hotWallet` (the `admin`).
4. Attacker calls:
   ```solidity
   EntropyUpgradable(proxy).withdrawFee(
       attackerEOA,
       EntropyUpgradable(proxy).getAccruedPythFees()
   );
   ```
5. `_authoriseAdminAction()` passes because `msg.sender == _state.admin`.
6. Entire `accruedPythFeesInWei` balance is transferred to `attackerEOA`.
7. `FeeWithdrawn` event emitted; funds are gone. [3](#0-2) [8](#0-7)
