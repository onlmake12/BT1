Audit Report

## Title
`Indexer::prune` Skips `Header`/`TxHash` Deletion When `ConsumedOutPoint` Keyspace Is Empty — (`util/indexer/src/indexer.rs`)

## Summary
`Indexer::prune` initialises `min_block_number` to `BlockNumber::MAX` and only updates it inside the `ConsumedOutPoint` deletion loop. When that keyspace is empty — the normal state for cellbase-only blocks — `min_block_number` remains `u64::MAX`, the RocksDB seek key for the `Header`/`TxHash` pass lands past all stored keys, and the deletion loop is silently skipped. Every `Header` and `TxHash` entry written since genesis is retained, causing unbounded storage growth.

## Finding Description
**Write path (`append`):**

`Key::ConsumedOutPoint` is written only inside the `if tx_index > 0` guard at line 339, so cellbase transactions (always `tx_index == 0`) never produce a `ConsumedOutPoint` entry. However, `tx_matched` is set to `true` at line 438 when any output — including a cellbase output — passes the cell filter, causing `Key::TxHash` to be written at lines 482–490. `Key::Header` is written unconditionally for every block passing the block filter at lines 496–509.

**Prune path (`prune`):**

`min_block_number` is initialised to `BlockNumber::MAX` at line 765 and is only updated at lines 777–779, inside the `ConsumedOutPoint` loop body. When the `ConsumedOutPoint` keyspace is empty (cellbase-only blocks), the loop body never executes and `min_block_number` stays at `u64::MAX`. The seek key for the `Header` pass is then constructed at line 784–785 as `[KeyPrefix::Header (0xE0)] ++ [0xFF×8]`, which positions the RocksDB iterator past all valid `Header` entries. The `take_while` predicate at lines 789–794 (`<= prune_to_block`) is correct in isolation but is never reached because the iterator is already exhausted. The entire `Header`/`TxHash` deletion loop (lines 795–805) becomes a no-op.

## Impact Explanation
`Header` and `TxHash` entries accumulate at a rate of at least one `Header` per block and one `TxHash` per cellbase transaction whose output passes the cell filter, with no upper bound. Over time, RocksDB storage grows without limit, degrading all indexer read paths and eventually exhausting disk on any node running the indexer. This is a concrete suboptimal implementation of the CKB state storage mechanism — **Medium (2001–10000 points)**.

## Likelihood Explanation
Cellbase-only blocks are the normal state of the CKB chain during low-activity periods; no special attacker capability is required. Any miner producing blocks without user transactions triggers this path. The condition is also reachable when a custom cell filter excludes all non-cellbase outputs, keeping `ConsumedOutPoint` empty even on a busy chain. The bug fires on every `prune` call under these conditions, making it continuous and self-compounding.

## Recommendation
Replace the `min_block_number`-based seek with an unconditional start from block 0 for the `Header`/`TxHash` pruning pass:

```rust
// Replace:
let mut key_prefix_header = vec![KeyPrefix::Header as u8];
key_prefix_header.extend_from_slice(&min_block_number.to_be_bytes());

// With:
let start_number = if min_block_number == BlockNumber::MAX { 0u64 } else { min_block_number };
let mut key_prefix_header = vec![KeyPrefix::Header as u8];
key_prefix_header.extend_from_slice(&start_number.to_be_bytes());
```

This ensures `Header`/`TxHash` pruning always runs up to `prune_to_block`, regardless of whether any `ConsumedOutPoint` entries exist.

## Proof of Concept
Using the existing `new_indexer` helper and block-building utilities in the test module:

1. Instantiate `Indexer` with `keep_num = 10`, `prune_interval = 1`.
2. Append `N = 10 * keep_num` blocks, each containing only a cellbase transaction with one output matching the default cell filter.
3. After each `append`, `prune` is called automatically (since `prune_interval = 1`).
4. Count `Key::Header` and `Key::TxHash` entries in RocksDB (pattern already used in existing tests at lines 1521–1529).
5. **Expected (correct):** entry count stabilises at `keep_num + 1`.
6. **Actual (buggy):** count equals `N` and never decreases, because `min_block_number == u64::MAX` causes the `Header`/`TxHash` deletion loop to be skipped on every `prune` call.
