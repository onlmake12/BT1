Audit Report

## Title
Tx-Pool Min-Fee-Rate Gate Uses Serialized Size Instead of Weight, Allowing Below-Minimum-Fee-Rate Transactions to Enter the Pool — (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using raw serialized byte size as the weight argument to `FeeRate::fee()`, despite `FeeRate` being defined in units of shannons per kilo-weight. Because weight is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`, a transaction with small size but high cycle consumption will have a true weight significantly larger than its byte size, allowing it to pass the admission gate with a fee far below the configured minimum. No weight-corrected rejection occurs after `verify_rtx` returns the actual cycle count, meaning the full CPU cost of script execution is irrecoverably spent before any eviction can take place.

## Finding Description
`FeeRate` is defined as shannons per kilo-weight in `util/types/src/core/fee_rate.rs` (L3–5), and `FeeRate::fee(weight)` computes `fee_rate * weight / 1000` (L34–36). The canonical weight function `get_transaction_weight` in `util/types/src/core/tx_pool.rs` (L298–303) returns `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`.

In `check_tx_fee` (`tx-pool/src/util.rs`, L45), the call is:
```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```
`tx_size` (raw bytes) is passed directly where weight is expected. The code comment at L42–44 explicitly acknowledges this: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check."*

In `_process_tx` (`tx-pool/src/process.rs`, L705–777), the execution order is:
1. `pre_check` → `check_tx_fee` (size-based gate, L715–717)
2. `verify_rtx` → full script execution, returns actual `cycles` (L724–734)
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` (L751)
4. `submit_entry` (L753)

There is no weight-based fee check between steps 2 and 4. The actual cycles are known at step 2, but no rejection is issued even when `get_transaction_weight(tx_size, verified.cycles)` would produce a minimum fee higher than what was paid. `TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs`, L115–117) correctly uses `get_transaction_weight`, but this is only used for post-admission eviction and prioritization, not for rejection at the admission gate.

## Impact Explanation
This maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** An attacker can repeatedly submit transactions with minimal serialized size and maximum cycle consumption, paying only the size-based minimum fee instead of the weight-based minimum. Each submission forces the node to execute expensive scripts via `verify_rtx` before any eviction occurs. The CPU cost is irrecoverable — it is spent during `verify_rtx` regardless of whether the transaction is later evicted by `limit_size`. At scale, this allows an attacker to consume node CPU at a fraction of the intended minimum cost, degrading tx-pool throughput and potentially causing network-wide congestion.

## Likelihood Explanation
The attack is reachable by any unprivileged caller via `send_raw_transaction`. The attacker controls both the serialized size (by choosing minimal inputs/outputs) and the cycle consumption (by embedding a looping script). The code comment confirms the gap is a known, intentional design shortcut. The undercount ratio is unbounded for small transactions: a 200-byte transaction consuming `MAX_BLOCK_CYCLES` cycles would have a weight orders of magnitude larger than its size, allowing the attacker to pay a fee rate many times below the configured minimum. The attack is repeatable with no state dependency.

## Recommendation
After `verify_rtx` returns the actual cycle count in `_process_tx`, perform a second weight-based fee check before calling `submit_entry`:
```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Weight-corrected fee check now that actual cycles are known
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```
This eliminates the gap between the cheap pre-check and the actual weight-based minimum without requiring cycles to be known at `pre_check` time.

## Proof of Concept
Given `min_fee_rate = 1_000 shannons/KW` (default):

| Parameter | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 10,000,000 |
| `weight` = max(200, 10,000,000 × 0.000_170_571_4) | **1,705** |
| `min_fee` (current size-based check, L45) | 1,000 × 200 / 1,000 = **200 shannons** |
| `correct_min_fee` (weight-based) | 1,000 × 1,705 / 1,000 = **1,705 shannons** |

A transaction paying 201 shannons passes `check_tx_fee` (L47: `fee < min_fee` is false), proceeds to `verify_rtx` which executes the expensive script, and enters the pool with an actual fee rate of `201 × 1,000 / 1,705 ≈ 117 shannons/KW` — 8.5× below the configured minimum. Submitting thousands of such transactions forces the node to spend CPU on script verification at a fraction of the intended cost.
