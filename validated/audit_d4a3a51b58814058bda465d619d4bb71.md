### Title
Subscription Manager Can Front-Run Keeper's `updatePriceFeeds()` to Cause Gas Loss Without Compensation — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, a subscription manager can front-run a keeper's `updatePriceFeeds()` transaction by calling `updateSubscription()` to deactivate the subscription, causing the keeper's transaction to revert and lose gas costs without receiving any compensation. This is the direct Pyth analog of the Perennial M-14 pattern: a user modifies a parameter (maxFee → 0 / isActive → false) after a keeper has committed to fulfilling a task, causing the keeper to lose their expected compensation.

---

### Finding Description

Keepers call `updatePriceFeeds()` to push price data and receive a fee paid from the subscription's balance. The fee is calculated dynamically from actual gas used plus a per-feed component:

```solidity
// Scheduler.sol line 279
uint256 startGas = gasleft();
...
if (!params.isActive) {
    revert SchedulerErrors.InactiveSubscription();  // line 287
}
``` [1](#0-0) 

The subscription manager can call `updateSubscription()` (restricted to `onlyManager`) at any time to set `isActive = false`:

```solidity
} else if (wasActive && !willBeActive) {
    currentParams.isActive = false;
    _removeFromActiveSubscriptions(subscriptionId);
    emit SubscriptionDeactivated(subscriptionId);
}
``` [2](#0-1) 

When a subscription is inactive, `withdrawFunds()` has **no minimum balance check**, allowing the manager to drain the entire balance:

```solidity
if (params.isActive) {
    uint256 minimumBalance = this.getMinimumBalance(...);
    if (status.balanceInWei - amount < minimumBalance) {
        revert SchedulerErrors.InsufficientBalance();
    }
}
// No check when inactive — full drain is possible
status.balanceInWei -= amount;
``` [3](#0-2) 

**Attack path:**

1. Keeper observes an active subscription with update conditions met and submits `updatePriceFeeds()`.
2. Manager front-runs with `updateSubscription(isActive=false)`.
3. Keeper's transaction reverts with `InactiveSubscription` — keeper loses all gas spent.

**More severe variant (balance drain):**

1. Manager deactivates subscription.
2. Manager calls `withdrawFunds(all)` — no minimum balance check when inactive.
3. Manager reactivates with exactly `minimumBalance`.
4. Keeper's transaction passes the `isActive` check and the initial balance check, executes the expensive `parsePriceFeedUpdatesWithConfig` call (paying the Pyth fee from balance), then fails at `_processFeesAndPayKeeper` because `status.balanceInWei < totalKeeperFee`.
5. The entire transaction reverts — keeper loses gas for the expensive Pyth parsing step without receiving any fee.

The keeper fee calculation in `_processFeesAndPayKeeper` includes dynamic gas cost:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;

if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [4](#0-3) 

At elevated gas prices, `minimumBalance` (a fixed value per feed) may be insufficient to cover `totalKeeperFee`, making the balance-drain variant viable without any reactivation step.

---

### Impact Explanation

A malicious subscription manager can repeatedly grief keepers by front-running their `updatePriceFeeds()` transactions. In the simple variant, keepers lose gas costs on reverted transactions. In the balance-drain variant, keepers lose gas for the expensive `parsePriceFeedUpdatesWithConfig` call (the dominant cost in `updatePriceFeeds`) without receiving any compensation. Repeated griefing can economically discourage keepers from servicing a subscription, effectively causing a liveness failure for that subscription's price feed consumers.

---

### Likelihood Explanation

The subscription manager is an unprivileged user (any address that created a subscription). On chains with a public mempool (Ethereum mainnet, most L2s), front-running is straightforward with standard MEV tooling. The manager has a direct economic incentive: they can deactivate and drain their subscription balance, then reactivate, effectively getting price feed updates for free while griefing keepers. The attack requires no special access, no leaked keys, and no third-party oracle misbehavior.

---

### Recommendation

1. **Time-lock on deactivation:** Require a delay (e.g., one heartbeat period) between a deactivation request and the subscription becoming inactive, giving in-flight keeper transactions time to land.
2. **Minimum balance lock:** Ensure the minimum balance enforced by `getMinimumBalance()` always covers at least one full keeper fee at a reasonable gas price ceiling, and enforce this check even when deactivating.
3. **Keeper fee escrow:** Alternatively, lock a per-update keeper fee in the contract at subscription creation/funding time, separate from the withdrawable balance, so it cannot be drained before a keeper's transaction lands.

    function updateSubscription(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams memory newParams
    ) external payable override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage currentStatus = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage currentParams = _state
            .subscriptionParams[subscriptionId];

        // Add incoming funds to balance
        currentStatus.balanceInWei += msg.value;

        // Updates to permanent subscriptions are not allowed
        if (currentParams.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        // If subscription is inactive and will remain inactive, no need to validate parameters
        bool wasActive = currentParams.isActive;
        bool willBeActive = newParams.isActive;
        if (!wasActive && !willBeActive) {
            // Update subscription parameters
            _state.subscriptionParams[subscriptionId] = newParams;
            emit SubscriptionUpdated(subscriptionId);
            return;
        }
        _validateSubscriptionParams(newParams);

        // Check minimum balance if subscription remains active
        if (willBeActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(newParams.priceIds.length)
            );
            if (currentStatus.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        // Handle activation/deactivation
        if (!wasActive && willBeActive) {
            // Reactivating a subscription - ensure minimum balance
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(newParams.priceIds.length)
            );

            // Check if balance meets minimum requirement
            if (currentStatus.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }

            currentParams.isActive = true;
            _addToActiveSubscriptions(subscriptionId);
            emit SubscriptionActivated(subscriptionId);
        } else if (wasActive && !willBeActive) {
            // Deactivating a subscription
            currentParams.isActive = false;
            _removeFromActiveSubscriptions(subscriptionId);
            emit SubscriptionDeactivated(subscriptionId);
        }

        // Clear price updates for removed price IDs before updating params
        bool newPriceIdsAdded = _clearRemovedPriceUpdates(
            subscriptionId,
            currentParams.priceIds,
            newParams.priceIds
        );

        // Reset priceLastUpdatedAt to 0 if new price IDs were added
        if (newPriceIdsAdded) {
            _state.subscriptionStatuses[subscriptionId].priceLastUpdatedAt = 0;
        }

        // Update subscription parameters
        _state.subscriptionParams[subscriptionId] = newParams;

        emit SubscriptionUpdated(subscriptionId);
    }
