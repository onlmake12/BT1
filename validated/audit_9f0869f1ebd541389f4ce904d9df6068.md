### Title
Copy-Paste Bug in `script_exists_in_output` Re-Checks `row_lock` Instead of `row_type` on PostgreSQL, Causing Premature Script Deletion During Block Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

`script_exists_in_output` contains a copy-paste error at line 252: the final `match` expression evaluates `row_lock` (the lock-script EXISTS result) instead of `row_type` (the type-script EXISTS result). On PostgreSQL, this causes the function to return `false` for any script that is referenced **only** as a `type_script_id` in surviving outputs, leading `rollback_block` to delete that script row from the `script` table even though live outputs still reference it.

---

### Finding Description

In `script_exists_in_output` (lines 204–257):

- **Lines 208–235**: Query `row_lock` — checks `WHERE lock_script_id = $1`. If true, return early.
- **Lines 237–249**: Query `row_type` — checks `WHERE type_script_id = $1`. Result stored in `row_type`.
- **Line 252 (bug)**: The final `match` re-evaluates `row_lock` instead of `row_type`:

```rust
// pg type is BOOLEAN
match row_lock.try_get::<bool, _>(0) {   // ← BUG: should be row_type
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

**PostgreSQL path** (`try_get::<bool, _>(0)` succeeds on `row_lock`): returns the lock-script check result, ignoring `row_type` entirely. If the script is only a type-script in surviving outputs, this returns `false` → script is deleted.

**SQLite path** (`try_get::<bool, _>(0)` fails on `row_lock` because it's `BIGINT`): falls through to `Err(_)` → correctly reads `row_type.get::<i64, _>(0)`. SQLite is **not** affected.

The caller in `rollback_block` removes outputs first, then calls `script_exists_in_output` to decide whether to delete the script row: [2](#0-1) 

Because outputs of the rolled-back block are already deleted before the check, a script shared between a rolled-back output and a surviving output will correctly show `lock_script_id` as absent — but the `type_script_id` check result is silently discarded on PostgreSQL.

---

### Impact Explanation

On a PostgreSQL-backed rich indexer deployment, any block rollback (reorg) involving a transaction whose output shares a `type_script_id` with a surviving output will permanently corrupt the `script` table: the shared script row is deleted while the surviving `output` row still holds a foreign-key reference to it. Subsequent `get_cells` / `get_transactions` RPC calls that JOIN on the `script` table will return `NULL` or missing script fields for those live cells. The corruption persists until the indexer is fully rebuilt.

The impact is **confined to the rich indexer** — CKB consensus, the VM, and fund custody are unaffected. However, wallets, dApps, and tooling that rely on the rich indexer for cell script data will receive silently incorrect responses.

---

### Likelihood Explanation

- Natural reorgs occur on any PoW chain; no adversarial hashpower is required.
- Type scripts (e.g., UDT, NFT, lock-proxy) are routinely shared across many outputs.
- PostgreSQL is a supported and documented backend for the rich indexer.
- The trigger condition (reorg + shared type script + PostgreSQL) is entirely ordinary.

---

### Recommendation

Change line 252 from `row_lock` to `row_type`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // ← fix
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [3](#0-2)

### Proof of Concept

1. Run a CKB node with the rich indexer on a **PostgreSQL** backend.
2. Submit a block containing two outputs both using the same type script `S` (same hash → same `type_script_id` row via deduplication).
3. Trigger a reorg that rolls back that block (e.g., via a longer competing chain tip).
4. `rollback_block` removes the outputs, then calls `script_exists_in_output(S_id)`.
5. `row_lock` query: `S` is not a `lock_script_id` in any remaining output → `false`.
6. Early-return check passes (false), so `row_type` query executes: `S` **is** a `type_script_id` in the surviving output → `true`.
7. Line 252 evaluates `row_lock` (not `row_type`) → returns `false` on PostgreSQL.
8. `S` is added to `script_id_list_to_remove` and deleted.
9. Assert: `SELECT * FROM script WHERE id = S_id` → empty. The surviving output's type script is gone.
10. `get_cells` RPC for the surviving output returns a cell with a `NULL` type script field.
