Audit Report

## Title
Non-Deterministic `CURRENT_CYCLES` Syscall Due to `iteration_cycles` Loss on Scheduler Resume — (`script/src/scheduler.rs`, `script/src/syscalls/current_cycles.rs`)

## Summary
`CURRENT_CYCLES` (syscall 2042) returns `total_cycles + machine.cycles()`, omitting `iteration_cycles`. After each `iterate_outer` call, `process_io()` re-populates `iteration_cycles` with VM suspend/resume overhead (`SPAWN_EXTRA_CYCLES_BASE`). When the scheduler is suspended at this point and later resumed, `Scheduler::resume()` unconditionally zeroes `iteration_cycles` at line 242, permanently discarding those charges. Nodes verifying the same transaction with different `step_cycles` configurations suspend at different boundaries, losing different cumulative amounts, causing `CURRENT_CYCLES` to return different values. A script branching on this syscall can produce different exit codes on different nodes, breaking consensus.

## Finding Description
**Root cause — `CURRENT_CYCLES` omits `iteration_cycles`:**

`CurrentCycles::ecall` computes `base.load(Ordering::Acquire) + machine.cycles()`, where `base` is `Arc::clone(&vm_context.base_cycles)`. [1](#0-0) 

`vm_context.base_cycles` is `Arc::clone(&self.total_cycles)` set in `create_dummy_vm`. [2](#0-1) 

`iteration_cycles` is never included.

**How `iteration_cycles` becomes non-zero after `iterate_outer`:**

`iterate_outer` runs: (1) `iterate_inner` → (2) `consume_cycles(iteration_cycles)` → (3) `iteration_cycles = 0` → (4) `process_io()`. [3](#0-2) 

Inside `process_io()`, `ensure_vms_instantiated` is called for read/write VM pairs, which invokes `resume_vm()` and `suspend_vm()`. Both add `SPAWN_EXTRA_CYCLES_BASE` directly to `self.iteration_cycles`. [4](#0-3) [5](#0-4) 

After `process_io()` returns, `iteration_cycles` is non-zero. The code comment and `FullSuspendedState` documentation both acknowledge this explicitly. [6](#0-5) 

**`suspend()` preserves the non-zero `iteration_cycles`:** [7](#0-6) 

**`resume()` unconditionally zeroes it, discarding the cycles:** [8](#0-7) 

In the non-suspended path those same cycles would have been moved to `total_cycles` at the next `iterate_outer`'s step 2. The divergence accumulates across every suspend/resume cycle that follows a `process_io()` call that triggered internal VM swaps.

**Divergence scenario:**
- Node A (`step_cycles` large, no suspension): IO-iteration cycles flow into `total_cycles` at the next `iterate_outer`. `CURRENT_CYCLES` reflects them.
- Node B (`step_cycles` small, suspends after `process_io()`): those same cycles are in `iteration_cycles` at suspension, zeroed on resume, permanently lost. `CURRENT_CYCLES` returns a smaller value by the sum of all discarded `SPAWN_EXTRA_CYCLES_BASE` charges.

The known-bug comment at lines 88–94 confirms this is the preserved Meepo hardfork behavior, but it does not make the behavior consensus-safe across nodes with differing `step_cycles`. [9](#0-8) 

## Impact Explanation
**Critical — consensus deviation.** A crafted V2 lock or type script that: (a) spawns a child VM connected via pipe, (b) transfers data (triggering `process_io()` with internal VM swaps), (c) calls `CURRENT_CYCLES`, and (d) branches on whether the result crosses a threshold calibrated to the expected discrepancy, will produce exit code 0 on Node A and exit code 1 on Node B for the identical transaction. This causes a permanent consensus split. Script deployment is permissionless on CKB, so no special privilege is required.

## Likelihood Explanation
Moderate. The attacker must deploy a specifically crafted script and submit a transaction using it. `step_cycles` is a node-level configuration parameter, not a consensus parameter, so legitimate nodes differ. The bug is explicitly documented in source, reducing the chance of an accidental fix, but also meaning developers are aware. The exploit requires no victim mistakes and is fully repeatable.

## Recommendation
Either: (a) include `iteration_cycles` in the value returned by `CURRENT_CYCLES` — changing the return to `total_cycles + iteration_cycles + machine.cycles()` — or (b) when resuming, add the saved `iteration_cycles` into `total_cycles` before zeroing it, so no cycles are silently discarded. The comment at line 242 should be updated to distinguish between scheduler-level suspend/resume overhead (legitimately uncharged) and IO-transfer VM-swap cycles (which must be preserved). Schedule the fix for the next hardfork superseding Meepo.

## Proof of Concept
Differential unit test:
1. Deploy a V2 lock script that: spawns a child VM connected via a pipe, transfers N bytes (ensuring `process_io()` triggers at least one internal VM swap via `ensure_vms_instantiated`), calls `CURRENT_CYCLES` (syscall 2042), exits 0 if result > `THRESHOLD`, else exits 1, where `THRESHOLD` is set between the two expected cycle counts.
2. Verify the same transaction twice using `TransactionScriptsVerifier` — once with `step_cycles = 1` (many scheduler suspensions) and once with `step_cycles = u64::MAX` (no suspension).
3. Assert both return the same exit code. The assertion will fail, demonstrating the consensus deviation.
