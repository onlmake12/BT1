### Title
Hardcoded `DEFAULT_BYTES_PER_CYCLES` Magic Ratio Ignores Actual Consensus Parameters, Causing Wrong Transaction Weight and Fee Estimates — (`util/types/src/core/tx_pool.rs`)

---

### Summary

`get_transaction_weight()` uses a hardcoded floating-point constant `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4_f64` to convert VM cycles into byte-equivalent weight. The comment states this equals `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES`, but both of those values are configurable per-chain-spec fields in the `Consensus` struct. The function never reads the actual live consensus parameters; it always uses the compile-time magic number. Any chain spec that deviates from the default block limits silently produces wrong transaction weights, wrong fee estimates, and wrong tx-pool ordering — directly analogous to the Oracle report's hardcoded `/ 100` divisor that assumed 8-decimal Chainlink feeds for all tokens.

---

### Finding Description

In `util/types/src/core/tx_pool.rs`:

```rust
/// Equal to MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES, see ckb-chain-spec.
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

The constant is derived from the default consensus values:
- `MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT = 597 × 1_000 = 597_000`
- `MAX_BLOCK_CYCLES = TWO_IN_TWO_OUT_CYCLES × TWO_IN_TWO_OUT_COUNT = 3_500_000 × 1_000 = 3_500_000_000_000`
- Ratio = `597_000 / 3_500_000_000 ≈ 0.000_170_571_4`

However, `max_block_bytes` and `max_block_cycles` are **runtime-configurable fields** of the `Consensus` struct, not compile-time invariants. They are set from the chain spec and can differ across networks or after a hard fork. `get_transaction_weight()` accepts no `Consensus` argument and always applies the hardcoded ratio regardless of what the live chain spec says.

The fee estimator compounds this by adding a second magic number:

```rust
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
```

This hardcodes 85% block utilization with only a comment referencing `MAX_BLOCK_PROPOSALS_LIMIT = 1500`, with no derivation or assertion that this assumption holds.

---

### Impact Explanation

`get_transaction_weight()` is consumed in two places:

1. **Fee estimator** (`util/fee-estimator/src/estimator/weight_units_flow.rs`, `TxStatus::new_from_entry_info`): computes per-transaction weight to populate fee-rate buckets. Wrong weights produce wrong bucket assignments and wrong `estimate_fee_rate` RPC responses.

2. **Tx-pool ordering** (via `AncestorsScoreSortKey`): transactions are ranked by fee / weight. If weight is systematically wrong, the pool orders transactions incorrectly, causing miners to assemble suboptimal blocks and users to receive misleading fee guidance.

On a chain spec where `max_block_cycles` is halved (e.g., a future hard fork or a custom devnet), `DEFAULT_BYTES_PER_CYCLES` would be **2× too small**, making cycle-heavy transactions appear lighter than they are. The fee estimator would underestimate required fees, and the tx-pool would deprioritize high-cycle transactions that should be prioritized. Conversely, doubling `max_block_cycles` makes the ratio 2× too large, causing the opposite distortion.

---

### Likelihood Explanation

- On mainnet Mirana with default parameters the hardcoded value is numerically correct, so no impact today.
- Any hard fork that adjusts `max_block_bytes` or `max_block_cycles` (both are consensus-configurable fields) immediately makes the constant stale without any compile-time or runtime warning.
- Custom devnets and testnets with non-default block limits are already affected.
- An RPC caller invoking `estimate_fee_rate` or submitting transactions to the pool is the unprivileged entry path; no special role is required.

---

### Recommendation

Replace the hardcoded constant with a computation derived from the live consensus parameters. `get_transaction_weight` should accept `max_block_bytes` and `max_block_cycles` as arguments (or a reference to `Consensus`), and compute the ratio at call time:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64,
    max_block_bytes: u64, max_block_cycles: u64) -> u64 {
    let bytes_per_cycle = max_block_bytes as f64 / max_block_cycles as f64;
    std::cmp::max(tx_size as u64, (cycles as f64 * bytes_per_cycle) as u64)
}
```

Similarly, the `85 / 100` utilization factor in the fee estimator should be named, documented with its derivation, and ideally derived from `MAX_BLOCK_PROPOSALS_LIMIT` and `TWO_IN_TWO_OUT_COUNT` rather than stated as a bare magic number.
