### Title
Hardcoded `CorsLayer::permissive()` Allows Any Web Origin to Call All RPC Methods — (`File: rpc/src/server.rs`)

### Summary
The CKB JSON-RPC HTTP/WebSocket server unconditionally applies `CorsLayer::permissive()` from `tower-http`, setting `Access-Control-Allow-Origin: *`, `Access-Control-Allow-Methods: *`, and `Access-Control-Allow-Headers: *` with no operator-configurable restriction. Because the RPC server has no authentication layer, any web page a user visits in a browser can issue cross-origin POST requests to `http://127.0.0.1:8114` and invoke every enabled RPC method — including state-mutating ones such as `send_transaction`, `clear_tx_pool`, `set_ban`, and `submit_block`.

### Finding Description
In `rpc/src/server.rs`, the `start_server` function builds the axum `Router` and unconditionally layers `CorsLayer::permissive()`:

```rust
// rpc/src/server.rs  line 119-129
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())          // ← hardcoded, no origin allowlist
    .layer(TimeoutLayer::with_status_code(...))
    .layer(Extension(stream_config));
```

`CorsLayer::permissive()` responds to every CORS preflight with:
```
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: *
Access-Control-Allow-Headers: *
```

The `RpcConfig` struct (`util/app-config/src/configs/rpc.rs`) exposes no field for configuring allowed CORS origins — there is no `cors_allowed_origins` or equivalent. The permissive policy is therefore not operator-overridable through the standard `ckb.toml` configuration.

The same `start_server` function is called for both the HTTP endpoint (`config.listen_address`) and the WebSocket endpoint (`config.ws_listen_address`), so both transports are affected.

### Impact Explanation
The CKB RPC server has no authentication mechanism. Combined with the wildcard CORS policy, any JavaScript running in the user's browser (from any origin) can:

- Call `Pool.send_transaction` to inject arbitrary transactions into the node's mempool.
- Call `Pool.clear_tx_pool` to wipe all pending transactions.
- Call `Net.set_ban` to ban legitimate peers, isolating the node.
- Call `Miner.submit_block` / `Miner.get_block_template` to interfere with block production.
- Call `Alert.send_alert` to broadcast network alerts.

All of these are state-mutating operations reachable by an unprivileged RPC caller. The default listen address is `127.0.0.1:8114`, which is accessible from the loopback interface and therefore reachable by any browser tab on the same machine.

**Impact: 3** — Unauthorized state mutation of a running CKB node (tx-pool manipulation, peer banning, mining interference) without any credential or key.

### Likelihood Explanation
The attack requires only that the node operator visit a malicious (or compromised) web page while the node is running. No special privileges, leaked keys, or network access beyond the local machine are needed. The default configuration (`127.0.0.1:8114`) is used by the vast majority of node operators.

**Likelihood: 3** — Realistic for any node operator who browses the web while running a node.

### Recommendation
1. Replace the hardcoded `CorsLayer::permissive()` with a restrictive default (e.g., `CorsLayer::new()` with no allowed origins, or `allow_origin(tower_http::cors::AllowOrigin::exact(...))`).
2. Add an optional `cors_allowed_origins: Vec<String>` field to `RpcConfig` so operators can explicitly whitelist trusted origins (e.g., a local wallet UI).
3. If broad CORS access is intentionally required for some deployments, document the risk prominently and require an explicit opt-in in `ckb.toml`.

### Proof of Concept
With a CKB node running at the default address, the following page — served from any origin — successfully clears the transaction pool:

```html
<script>
fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    id: 1, jsonrpc: "2.0",
    method: "clear_tx_pool", params: []
  })
}).then(r => r.json()).then(console.log);
</script>
```

Because `CorsLayer::permissive()` returns `Access-Control-Allow-Origin: *` on the preflight, the browser allows the request. No credentials or authentication tokens are required.

**Root cause:** `rpc/src/server.rs`, line 124 — `CorsLayer::permissive()` hardcoded with no configurable origin allowlist and no corresponding field in `util/app-config/src/configs/rpc.rs`. [1](#0-0) [2](#0-1)
