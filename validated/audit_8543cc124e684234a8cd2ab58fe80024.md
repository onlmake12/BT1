Audit Report

## Title
Unbounded `last_n_numbers` via `difficulty_boundary=U256::zero()` and empty `difficulties` bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
An unprivileged remote peer can send a `GetLastStateProof` message with `difficulties=[]`, `difficulty_boundary=U256::zero()`, and `last_n_blocks=0`. All four input guards are bypassed, `difficulty_boundary_block_number` collapses to `start_block_number` (block 0), and `last_n_numbers` spans the entire chain. The server then calls `complete_headers` over every block number, performing O(chain_length) database reads and MMR root computations for a single tiny P2P message, with no effective cap.

## Finding Description

**Size guard (lines 201–205):** checks `difficulties.len() + last_n_blocks * 2 <= 1000`. With `difficulties=[]` and `last_n_blocks=0`, this evaluates to `0`, passing trivially. [1](#0-0) 

**Monotonicity guard (line 254):** `[].windows(2).any(...)` is always `false` on an empty slice — passes. [2](#0-1) 

**Max-difficulty-vs-boundary guard (lines 259–266):** `[].last()` returns `None`, `.map(...).unwrap_or(false)` returns `false` — the boundary check is silently skipped entirely. [3](#0-2) 

**First-difficulty-vs-start guard (lines 268–288):** `[].first()` returns `None` — the entire block is skipped. [4](#0-3) 

**Branch selection (lines 291–292):** With `start_block_number=0` and `last_n_blocks=0`, the condition `last_block_number - 0 <= 0` is false for any non-genesis chain, so execution enters the `else` branch. [5](#0-4) 

**`get_first_block_total_difficulty_is_not_less_than(0, last_block_number, &U256::zero())` (lines 30–33):** Block 0's total difficulty is always `>= U256::zero()`, so the function immediately returns `Some((0, genesis_total_difficulty))`, setting `difficulty_boundary_block_number = 0`. [6](#0-5) 

**Adjustment check (line 313):** `last_block_number - 0 < 0` is false for any u64 — no adjustment. [7](#0-6) 

**Result (lines 318–347):** `last_n_numbers = (0..last_block_number)` — the entire chain. Since `difficulty_boundary_block_number == 0`, the `else` branch at line 345 fires, returning `(Vec::new(), last_n_numbers)`. [8](#0-7) 

**`complete_headers` (lines 356–366):** Called with all N block numbers. For each: one `get_ancestor` call, one `get_block` read, one `chain_root_mmr` computation. Total work is O(chain_length) per message. [9](#0-8) 

The root cause is that `GET_LAST_STATE_PROOF_LIMIT` is only checked against `difficulties.len() + last_n_blocks * 2`, which does not account for the `last_n_numbers` range when `difficulty_boundary_block_number` collapses to `start_block_number`. [10](#0-9) 

## Impact Explanation
A single attacker-controlled peer can repeatedly send a crafted `GetLastStateProof` message causing the server to perform O(chain_length) DB reads, block fetches, and MMR root computations. On CKB mainnet with millions of blocks, this exhausts server CPU, memory (collecting all block numbers into a `Vec`), and I/O. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**, and also **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, since any light-client-serving node is vulnerable and the attack is trivially repeatable.

## Likelihood Explanation
The attacker needs only a valid `last_hash` on the main chain, which is trivially obtained via `GetLastState`. No privilege, key, stake, or hashpower is required. The crafted message is a few bytes. The attack is fully reachable from any P2P peer and is repeatable at will.

## Recommendation
1. **Reject `difficulty_boundary == U256::zero()`** explicitly before the sampler is invoked, since zero is not a meaningful chain-anchored boundary.
2. **Fix the size guard** to bound the actual response size: after computing `difficulty_boundary_block_number`, check `(last_block_number - difficulty_boundary_block_number) + sampled_numbers.len() + reorg_last_n_numbers.len() <= GET_LAST_STATE_PROOF_LIMIT` before proceeding to `complete_headers`.
3. **Alternatively**, enforce that `difficulty_boundary` must be strictly greater than the total difficulty of `start_block_number`'s block, ensuring it is a meaningful chain-anchored value that cannot collapse to block 0.

## Proof of Concept
```rust
// On a chain mined to 10_000 blocks:
let content = packed::GetLastStateProof::new_builder()
    .last_hash(tip_hash)                          // valid tip on main chain
    .start_hash(genesis_hash)
    .start_number(0u64.pack())
    .last_n_blocks(0u64.pack())
    .difficulty_boundary(U256::zero().pack())     // zero boundary
    // difficulties left empty (default)
    .build();
// Server calls complete_headers with numbers [0..10_000]:
// 10_000 get_ancestor calls + 10_000 get_block reads + 10_000 MMR computations.
// Expected (with fix): server rejects or returns at most GET_LAST_STATE_PROOF_LIMIT headers.
// Actual (without fix): server processes all 10_000 blocks, violating the limit.
```
