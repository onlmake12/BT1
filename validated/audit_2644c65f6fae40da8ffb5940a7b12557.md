### Title
No `maxFee` Guard in Entropy Request Functions Allows Uncontrolled Fee Capture of Excess ETH — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract's `requestHelper()` function does not provide callers with any mechanism to cap the fee they are willing to pay. Any `msg.value` sent above the provider's fee is permanently captured as Pyth protocol fees (`accruedPythFeesInWei`) and is never refunded. Because providers can update their fee at any time via `setProviderFee()`, users have no on-chain protection against fee front-running or overpayment.

---

### Finding Description

In `requestHelper()`, the required fee is computed at execution time from the live on-chain provider fee:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

Any `msg.value` above `providerFee` is added to `accruedPythFeesInWei` and is never returned to the caller. The interface documentation explicitly acknowledges this: *"excess value is not refunded to the caller."* [2](#0-1) 

None of the `requestV2()` variants accept a `maxFee` parameter: [3](#0-2) 

Providers can change their fee at any time with no time-lock or delay:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
``` [4](#0-3) 

Furthermore, the Fortuna keeper service (`adjust_fee_if_necessary`) is explicitly designed to dynamically raise and lower provider fees in response to gas price changes, meaning fee changes are a routine, frequent occurrence in production: [5](#0-4) 

The same pattern exists in the Echo contract's `requestPriceUpdatesWithCallback()`: [6](#0-5) 

---

### Impact Explanation

Any user or integrating contract that sends `msg.value > requiredFee` (a common defensive pattern to avoid reverts from fee fluctuations) permanently loses the excess ETH to the Pyth fee pool. There is no refund path. For integrating contracts that hardcode a fee buffer, this results in a systematic, repeated fund loss on every request. The impact is direct loss of user funds with no recovery mechanism.

---

### Likelihood Explanation

The Fortuna keeper service is designed to adjust provider fees frequently based on gas prices. Users and integrating contracts are explicitly advised to call `getFeeV2()` before each request because fees change over time. This creates a natural race condition: a user reads the fee, submits a transaction with a buffer, and the buffer is captured. The likelihood is **medium-high** for any integrating contract that sends a fee buffer, and is a routine occurrence given the dynamic fee adjustment system.

---

### Recommendation

1. Add a `maxFee` parameter to `requestV2()` variants that reverts if `requiredFee > maxFee`, analogous to slippage protection in AMMs:
   ```solidity
   function requestV2(uint32 gasLimit, uint128 maxFee) external payable returns (uint64);
   ```
2. Alternatively, refund excess `msg.value` above `requiredFee` back to `msg.sender` at the end of `requestHelper()`.

---

### Proof of Concept

**Scenario A — Excess buffer captured:**
1. User calls `getFeeV2(provider, gasLimit)` → returns `F` wei.
2. User submits `requestV2{value: F + 5000}(gasLimit)` with a 5000 wei buffer to avoid reverts.
3. In `requestHelper()`: `providerFee = getProviderFee(...)` = `F - pythFeeInWei`.
4. `accruedPythFeesInWei += msg.value - providerFee` = `(F + 5000) - (F - pythFeeInWei)` = `pythFeeInWei + 5000`.
5. The 5000 wei buffer is permanently captured as Pyth fees; user receives no refund.

**Scenario B — Provider front-run:**
1. User reads fee `F`, submits `requestV2{value: F + buffer}()`.
2. Provider (or Fortuna keeper) front-runs with `setProviderFee(newFee)` where `newFee > F`.
3. If `F + buffer >= newFee + pythFeeInWei`: transaction succeeds, user pays the higher fee, remaining buffer is captured.
4. If `F + buffer < newFee + pythFeeInWei`: transaction reverts with `InsufficientFee`, user loses gas.
5. In neither case can the user enforce a maximum acceptable fee.

    // Set provider fee. It will revert if provider is not registered.
    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
