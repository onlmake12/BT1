### Title
TOCTOU Race in Concurrent Tx-Pool Submission Allows Duplicate Transaction Verification — (File: `tx-pool/src/process.rs`, `tx-pool/src/service.rs`)

---

### Summary
The tx-pool service spawns a new async task for every incoming message. The duplicate-detection logic in `resumeble_process_tx` (called by both `submit_remote_tx` and `notify_tx`) performs its existence checks under separate, short-lived read locks and then inserts under a separate write lock. Two concurrent tasks carrying the same transaction can both pass all duplicate checks and both enqueue the transaction into the verify queue, causing the same transaction to be script-verified multiple times.

---

### Finding Description

**Root cause — unbounded concurrent task spawning**

In `tx-pool/src/service.rs`, the main message loop spawns a fresh async task for every message it receives: [1](#0-0) 

```rust
Some(message) = receiver.recv() => {
    let service_clone = process_service.clone();
    handle_clone.spawn(process(service_clone, message));
},
```

There is no back-pressure or serialisation between tasks that carry the same transaction. Two `SubmitRemoteTx` (or `NotifyTxs`) messages for the same transaction from different peers produce two fully concurrent async tasks sharing the same `TxPoolService` state.

**Root cause — non-atomic check-then-act in `resumeble_process_tx`**

`resumeble_process_tx` (called by `submit_remote_tx` and `notify_tx`) performs three separate lock acquisitions: [2](#0-1) 

```rust
pub(crate) async fn resumeble_process_tx(...) -> Result<bool, Reject> {
    self.non_contextual_verify(&tx, remote).await?;

    if self.orphan_contains(&tx).await {          // ① read-lock acquired & released
        return Err(Reject::Duplicated(tx.hash()));
    }
    if self.verify_queue_contains(&tx).await {    // ② read-lock acquired & released
        return Err(Reject::Duplicated(tx.hash()));
    }
    self.enqueue_verify_queue(tx, ...).await       // ③ write-lock acquired
}
```

Each helper acquires and immediately drops its own lock: [3](#0-2) 

```rust
pub(crate) async fn verify_queue_contains(&self, tx: &TransactionView) -> bool {
    let queue = self.verify_queue.read().await;
    queue.contains_key(&tx.proposal_short_id())
}
pub(crate) async fn orphan_contains(&self, tx: &TransactionView) -> bool {
    let orphan = self.orphan.read().await;
    orphan.contains_key(&tx.proposal_short_id())
}
```

And `enqueue_verify_queue` acquires a separate write lock: [4](#0-3) 

```rust
async fn enqueue_verify_queue(...) -> Result<bool, Reject> {
    let mut queue = self.verify_queue.write().await;
    queue.add_tx(tx, is_proposal_tx, remote)
}
```

Between step ② and step ③ there is no lock held. A second concurrent task can observe the queue as empty (step ②), then both tasks proceed to step ③ and both successfully insert the same transaction.

**The same TOCTOU window exists in `_process_tx`** (used by the synchronous `process_tx` path): [5](#0-4) 

`pre_check` acquires a read lock, resolves the transaction, and releases the lock. The expensive `verify_rtx` then runs with no lock held. `submit_entry` later acquires a write lock. A second concurrent task can complete `pre_check` and begin `verify_rtx` for the same transaction before the first task reaches `submit_entry`.

---

### Impact Explanation

1. **CPU amplification / DoS**: Script verification (`verify_rtx`) is the most expensive operation in the tx-pool. An attacker who submits the same transaction from N simultaneous peer connections causes N independent script-verification runs for a single transaction. With `max_tx_verify_workers` workers and a large-cycle transaction, this saturates all verify workers with redundant work, starving legitimate transactions.

2. **Verify-queue pollution**: If `VerifyQueue::add_tx` does not deduplicate under its own write lock, the same transaction occupies N queue slots. Each slot is popped and processed independently by a worker, compounding the CPU waste.

3. **Incorrect rejection accounting**: The `Reject::Duplicated` path that should fire for the second submission is silently bypassed, so the relayer's filter state and recent-reject database are not updated correctly for the duplicate submissions.

---

### Likelihood Explanation

An unprivileged remote peer can open multiple simultaneous connections to a CKB node (the default peer limit is generous) and broadcast the same transaction on all of them at the same time. No special privilege, key, or majority hashpower is required. The race window is wide because `verify_rtx` can take hundreds of milliseconds for high-cycle transactions, giving ample time for a second task to pass the duplicate check before the first task completes insertion.

---

### Recommendation

Merge the duplicate check and the insertion into a single atomic operation under the write lock inside `enqueue_verify_queue`:

```rust
async fn enqueue_verify_queue(
    &self,
    tx: TransactionView,
    is_proposal_tx: bool,
    remote: Option<(Cycle, PeerIndex)>,
) -> Result<bool, Reject> {
    let mut queue = self.verify_queue.write().await;
    // Atomically check-and-insert under the same write lock
    if queue.contains_key(&tx.proposal_short_id()) {
        return Err(Reject::Duplicated(tx.hash()));
    }
    queue.add_tx(tx, is_proposal_tx, remote)
}
```

Remove the separate `verify_queue_contains` and `orphan_contains` pre-checks from `resumeble_process_tx`, or keep them only as a cheap fast-path hint while relying on the atomic write-lock check as the authoritative guard. Similarly, `submit_entry` should re-check for txid collision under its write lock before calling `_submit_entry`.

---

### Proof of Concept

1. Attacker opens N TCP connections to the CKB node (e.g., N = 8, matching `max_tx_verify_workers`).
2. Attacker crafts one high-cycle transaction T (near `max_tx_verify_cycles`).
3. Attacker simultaneously sends T as a `RelayTransactionV3` message on all N connections.
4. The relay handler calls `submit_remote_tx` → `resumeble_process_tx` for each peer concurrently.
5. All N tasks execute `verify_queue_contains` while the queue is still empty → all return `false`.
6. All N tasks call `enqueue_verify_queue` → all acquire the write lock in turn and insert T (or insert it once and silently drop the rest, depending on `add_tx` internals).
7. Up to N verify workers each independently run full script verification of T, consuming N × (verification time) of CPU for a single transaction submitted by one attacker. [6](#0-5) [7](#0-6) [4](#0-3)
