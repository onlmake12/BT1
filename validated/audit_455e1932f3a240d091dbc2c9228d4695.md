### Title
Stale Pre-Eviction Snapshot Overwrites Live `total_tx_size`/`total_tx_cycles` Counters in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the running pool-size counters `total_tx_size` and `total_tx_cycles` are computed into local variables **before** a conditional eviction step runs. If that eviction step removes transactions (correctly decrementing the live counters via `update_stat_for_remove_tx`), the stale pre-eviction snapshot is then unconditionally written back, silently cancelling every decrement. The counters become permanently inflated by the aggregate size/cycles of all evicted transactions.

---

### Finding Description

`PoolMap::add_entry` in `tx-pool/src/component/pool_map.rs` executes the following sequence:

```
// Step 1 – snapshot new totals BEFORE any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211

// Step 2 – may evict N transactions, each calling update_stat_for_remove_tx
//          which MUTATES self.total_tx_size / self.total_tx_cycles in-place
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

// Step 3 – unconditionally overwrite with the stale snapshot
self.total_tx_size  = total_tx_size;                            // line 218
self.total_tx_cycles = total_tx_cycles;                         // line 219
```

`updated_stat_for_add_tx` (lines 711–728) reads `self.total_tx_size` and returns `self.total_tx_size + new_size` as a local value — it does **not** write to `self`. `check_and_record_ancestors` (lines 588–640) may call `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` (lines 733–758), which **does** write to `self.total_tx_size` and `self.total_tx_cycles`. The final assignment at lines 218–219 then overwrites those correctly-decremented live values with the stale snapshot.

**Concrete arithmetic:**

| Step | `self.total_tx_size` | local `total_tx_size` |
|---|---|---|
| Before `add_entry` | 100 | — |
| After `updated_stat_for_add_tx` (new tx = 10) | 100 (unchanged) | 110 |
| After eviction of tx of size 20 | **80** (correctly decremented) | 110 (stale) |
| After line 218 assignment | **110** (wrong; should be 90) | — |

The evicted transaction's 20 bytes are never subtracted from the final counter. Each such `add_entry` call that triggers eviction inflates the counters by the total size/cycles of all evicted entries.

The eviction path inside `check_and_record_ancestors` is reached when:
- `ancestors_count > self.max_ancestors_count` (line 598), AND
- `ancestors_count - cell_ref_parents.len() <= self.max_ancestors_count` (line 603)

This means the new transaction has ancestors that include "cell-ref parents" — pool transactions that reference the same output as a cell dep. This is a reachable condition for any unprivileged transaction sender. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` (pool.rs line 298) to decide whether to evict further transactions:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee entries
}
```

An inflated `total_tx_size` causes `limit_size` to believe the pool is over capacity when it is not, triggering a cascade of spurious evictions of legitimate, fee-paying transactions. Repeated exploitation drives the counter arbitrarily high, causing the pool to continuously evict its own contents and reject all new submissions with `Reject::Full`, effectively disabling the mempool for honest users.

`total_tx_cycles` is similarly inflated, affecting the `tx-pool_info` RPC response and any future cycle-based admission logic. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The trigger condition — a new transaction whose ancestor set contains cell-ref parents that push the ancestor count just over `max_ancestors_count` — is fully under attacker control. An unprivileged transaction sender can:

1. Submit a base transaction `T0` that creates an output used as a cell dep by several other pool transactions (`T1…Tk`).
2. Submit a new transaction `Tnew` that spends an output of `T0` and has `T1…Tk` as cell-dep ancestors, pushing the ancestor count over the limit.
3. `check_and_record_ancestors` evicts `T1…Tk`; the stale snapshot is written back, inflating `total_tx_size` by `sum(size(T1…Tk))`.
4. Repeat with fresh transactions to accumulate unbounded inflation.

No privileged access, no majority hashpower, and no external dependency is required. The entry path is the standard `send_transaction` RPC / P2P relay path. [7](#0-6) 

---

### Recommendation

Compute the new totals **after** all evictions have completed, not before. Replace the pre-eviction snapshot pattern with a post-eviction addition:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Validate capacity BEFORE eviction (read-only check is fine here)
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Apply the addition AFTER evictions have already decremented the counters
    self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

This mirrors the fix recommended in the reference report: always perform the addition unconditionally after the removal, rather than computing a snapshot that is later invalidated by intermediate mutations. [1](#0-0)

### Proof of Concept

**Setup:**
- `max_ancestors_count = 25`, `max_tx_pool_size = 1_000_000` bytes.
- Submit 24 transactions `T1…T24`, each referencing output `O` of a base transaction `T0` as a cell dep. Each `Ti` is ~1 000 bytes. Pool now has 25 entries, `total_tx_size ≈ 25_000`.

**Trigger:**
- Submit `Tnew` (1 000 bytes) that spends an output of `T0`. Its ancestor set includes `T1…T24` (via cell-dep linkage), giving `ancestors_count = 25 + 1 = 26 > max_ancestors_count`.
- `cell_ref_parents = {T1…T24}`, so `26 - 24 = 2 ≤ 25` — the eviction branch is taken.
- `check_and_record_ancestors` evicts, say, 2 entries (`T1`, `T2`, each 1 000 bytes). `update_stat_for_remove_tx` correctly sets `self.total_tx_size = 23_000`.
- Line 218 then writes back the stale snapshot: `self.total_tx_size = 25_000 + 1_000 = 26_000` (should be `24_000`).
- Inflation per call: `+2_000` bytes (the evicted amount).

**Escalation:**
- Repeat the pattern ~490 times. `total_tx_size` reaches `≈ 1_000_000` (the pool limit) while the actual pool contains far fewer bytes.
- `limit_size` now fires on every subsequent `add_entry`, evicting legitimate transactions and returning `Reject::Full` to all new submissions.
- The mempool is effectively disabled for honest users without the attacker ever filling it with real data. [8](#0-7) [9](#0-8)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
