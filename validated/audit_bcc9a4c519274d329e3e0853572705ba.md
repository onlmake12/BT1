### Title
Stale `min_replace_fee` Snapshot from `get_transaction` RPC Causes Unexpected RBF Rejection — (`tx-pool/src/pool.rs`, `tx-pool/src/service.rs`)

### Summary

The `get_transaction` RPC exposes a `min_replace_fee` field computed from a point-in-time snapshot of the tx-pool. A user who reads this value and constructs a replacement transaction with exactly that fee can have their replacement silently rejected if the pool state changes between the read and the submission — because `check_rbf` recomputes the required fee at submission time against the live pool. An adversary who adds a high-fee descendant to the victim's pending transaction between those two moments can force the replacement to fail indefinitely.

---

### Finding Description

**Step 1 — Query path (snapshot read)**

When a caller invokes `get_transaction` with `verbosity=2`, the tx-pool service handler reads the pool under a **read lock** and calls `tx_pool.min_replace_fee(entry)`: [1](#0-0) 

`min_replace_fee` is defined as:

```
min_replace_fee = sum(fee of target tx + all its current descendants) + extra_rbf_fee
``` [2](#0-1) 

This value is returned to the caller as a field in `TransactionWithStatus`: [3](#0-2) 

**Step 2 — Submission path (live recomputation)**

When the caller submits a replacement transaction via `send_transaction`, `submit_entry` acquires a **write lock** and calls `check_rbf`, which recomputes `min_replace_fee` from the current pool state at that instant: [4](#0-3)

**The gap**: between the read-lock snapshot (Step 1) and the write-lock check (Step 2), any other actor can add a new descendant transaction to the target tx. Because `min_replace_fee` includes the fees of all descendants, the required fee at submission time is now higher than what was returned to the caller. The replacement is rejected with:

```
"Tx's current fee is X, expect it to >= Y to replace old txs"

### Impact Explanation

**Unexpected rejection for honest users.** A user who faithfully reads `min_replace_fee` from `get_transaction` and constructs a replacement with exactly that fee will have their replacement rejected if any concurrent pool activity adds a descendant to the target transaction. This is directly analogous to the bonding-curve `require(msg.value >= totalEthValue)` revert: the user computed the exact required value from a stale snapshot.

**Adversarial griefing.** An attacker who monitors the mempool can:
1. Observe a victim's pending transaction `tx_A`.
2. Wait for the victim to query `min_replace_fee` (or simply act preemptively).
3. Submit a descendant `tx_B` of `tx_A` with a large fee, increasing the required replacement fee.
4. The victim's replacement `tx_C` (built against the stale `min_replace_fee`) is rejected.
5. The attacker can repeat this each time the victim retries, keeping `tx_A` stuck in the pool.

The attacker's cost is the fee for `tx_B`, but this can be structured as a low-cost descendant that the attacker is willing to have committed anyway. The victim's cost is repeated failed submissions and inability to urgently replace a time-sensitive transaction.

**Impact: Medium** — No direct fund loss, but legitimate RBF replacements fail unexpectedly and can be blocked by a low-cost adversary.

---

### Likelihood Explanation

**Likelihood: Medium.** On a busy network, concurrent pool mutations are routine. Even without a deliberate attacker, two users independently adding descendants to the same transaction chain can trigger this for an innocent third party attempting an RBF replacement. With a deliberate attacker, the window is easy to exploit because the attacker only needs to submit one descendant transaction before the victim's replacement arrives.

---

### Recommendation

1. **Document the staleness**: The `get_transaction` RPC documentation should explicitly state that `min_replace_fee` is a best-effort snapshot and may be stale by the time the replacement is submitted.

2. **Add a tolerance/slippage parameter to `send_transaction`**: Allow callers to specify a `max_fee` cap or a `fee_tolerance` so that the node can accept a replacement whose fee is within a user-defined margin above the stale `min_replace_fee`, rather than requiring an exact or higher match against a potentially shifted target.

3. **Return a versioned pool tip hash alongside `min_replace_fee`**: The caller can detect whether the pool state changed between query and submission and retry proactively, rather than discovering the staleness only after rejection.

---

### Proof of Concept

```
1. Alice has tx_A (fee = 100 shannons) pending in the pool.
2. Alice calls get_transaction(tx_A.hash) → min_replace_fee = 200 shannons.
3. Attacker submits tx_B (descendant of tx_A, fee = 500 shannons).
   Now actual min_replace_fee = 100 + 500 + extra_rbf_fee = 700+ shannons.
4. Alice submits tx_C (replacement for tx_A, fee = 200 shannons).
5. check_rbf recomputes: min_replace_fee = 700 shannons.
6. 200 < 700 → Reject: "Tx's current fee is 200, expect it to >= 700 to replace old txs"
7. Alice's replacement fails. Attacker repeats step 3 on each retry.
```

    /// min_replace_fee = sum(replaced_txs.fee) + extra_rbf_fee
    fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
        let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
        // don't account for duplicate txs
        let replaced_fees: HashMap<_, _> = conflicts
            .iter()
            .map(|c| (c.id.clone(), c.inner.fee))
            .collect();
        let replaced_sum_fee = replaced_fees
            .values()
            .try_fold(Capacity::zero(), |acc, x| acc.safe_add(*x));
        let res = replaced_sum_fee.map_or(Err(CapacityError::Overflow), |sum| {
            sum.safe_add(extra_rbf_fee)
        });
        if let Ok(res) = res {
            Some(res)
        } else {
            let fees = conflicts.iter().map(|c| c.inner.fee).collect::<Vec<_>>();
            error!(
                "conflicts: {:?} replaced_sum_fee {:?} overflow by add {}",
                conflicts.iter().map(|e| e.id.clone()).collect::<Vec<_>>(),
                fees,
                extra_rbf_fee
            );
            None
        }
    }
