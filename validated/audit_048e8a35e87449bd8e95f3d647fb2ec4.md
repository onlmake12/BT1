### Title
Unbounded `tx.gasprice` in `_processFeesAndPayKeeper` Allows Any Caller to Drain Subscription Balance in a Single Transaction - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler._processFeesAndPayKeeper()` computes the keeper gas reimbursement as `(gasUsed + GAS_OVERHEAD) * tx.gasprice` and immediately transfers the full amount to `msg.sender`. Because `updatePriceFeeds()` has no access control and `tx.gasprice` is fully attacker-controlled, any address can call `updatePriceFeeds()` with a pathologically high priority fee and drain a subscription's entire ETH balance in one transaction.

---

### Finding Description

`_processFeesAndPayKeeper()` calculates the keeper fee using the raw `tx.gasprice` opcode:

```solidity
// Scheduler.sol line 846
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
...
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [1](#0-0) 

Post-EIP-1559, `tx.gasprice` equals `baseFee + priorityFee`. A caller can set `maxPriorityFeePerGas` to an arbitrarily large value (e.g., 100,000 gwei), making `tx.gasprice` enormous. There is no cap on `tx.gasprice` anywhere in the contract or in `SchedulerConstants.sol`. [2](#0-1) 

The entry point `updatePriceFeeds()` carries **no access control** — it is `external override` with no keeper whitelist, no `onlyManager` guard, and no role check:

```solidity
// Scheduler.sol line 275
function updatePriceFeeds(
    uint256 subscriptionId,
    bytes[] calldata updateData
) external override {
``` [3](#0-2) 

The only precondition is that the subscription is active and the update conditions (heartbeat or deviation) are satisfied — both of which are observable on-chain and satisfiable by any caller with valid Pyth update data.

---

### Impact Explanation

A malicious caller drains `status.balanceInWei` — the ETH deposited by the subscription owner — in a single transaction. The entire drained amount is transferred to `msg.sender` (the attacker). Permanent subscriptions can hold up to `MAX_DEPOSIT_LIMIT = 100 ether`. [4](#0-3) 

The subscription owner loses their full deposited balance. The protocol's price-feed service is also disrupted because the subscription is left with zero balance and cannot fund future updates.

---

### Likelihood Explanation

- `updatePriceFeeds()` is permissionless; any EOA or contract can call it.
- Valid Pyth update data is publicly available from the Pyth price service.
- The attacker only needs to wait for the heartbeat interval (or observe a sufficient price deviation) to satisfy `_validateShouldUpdatePrices`.
- Setting a high `maxPriorityFeePerGas` is a standard EIP-1559 transaction parameter requiring no special access.
- The profit motive is direct: the attacker receives `totalKeeperFee` immediately.

---

### Recommendation

Cap `tx.gasprice` to a governance-configurable maximum (e.g., `maxGasPriceInWei`) before computing `gasCost`:

```solidity
uint256 effectiveGasPrice = tx.gasprice < _state.maxGasPriceInWei
    ? tx.gasprice
    : _state.maxGasPriceInWei;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, use a TWAP gas oracle (e.g., EIP-1559 base fee from a recent block) as the reference price instead of the live `tx.gasprice`.

---

### Proof of Concept

1. Subscription owner creates a subscription with `balanceInWei = 10 ether` and a heartbeat of 60 seconds.
2. Attacker waits 60 seconds (heartbeat elapsed).
3. Attacker fetches valid Pyth update data for the subscribed price IDs.
4. Attacker calls:
   ```solidity
   scheduler.updatePriceFeeds{maxPriorityFeePerGas: 100_000 gwei}(subscriptionId, updateData);
   ```
5. Inside `_processFeesAndPayKeeper`:
   - `gasUsed ≈ 200_000` (typical for the function)
   - `GAS_OVERHEAD = 30_000`
   - `tx.gasprice ≈ 100_000 gwei` (attacker-set priority fee dominates)
   - `gasCost = 230_000 * 100_000e9 = 23 ETH`
6. `totalKeeperFee = 23 ETH` exceeds `balanceInWei = 10 ETH` → reverts with `InsufficientBalance`.
7. Attacker adjusts to `tx.gasprice` that yields `gasCost` just below `balanceInWei`:
   - Target: `gasCost = 9.9 ETH` → `tx.gasprice = 9.9e18 / 230_000 ≈ 43_000 gwei`
8. The subscription's entire 9.9 ETH balance is transferred to the attacker in one transaction. [5](#0-4)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
```
