### Title
Unauthenticated Block Template Injection via Miner Notify HTTP Endpoint — (`miner/src/client.rs`)

### Summary

When the CKB miner is configured with `listen` mode (`config.listen = Some(addr)`), it binds an HTTP server that accepts `BlockTemplate` payloads from any TCP client with no authentication, no source IP check, and no shared secret. Any unprivileged local process (or remote process if the address is `0.0.0.0`) can POST a crafted `BlockTemplate` JSON and cause the miner to immediately abandon its legitimate work and mine on the attacker-supplied template.

---

### Finding Description

`listen_block_template_notify` binds a `TcpListener` to the configured address and dispatches every accepted connection to the `handle` free function: [1](#0-0) 

The peer address is silently discarded (`_`). The `handle` function performs no authentication of any kind: [2](#0-1) 

Any body that successfully deserializes as `BlockTemplate` is forwarded directly to `update_block_template`, which sends it to all mining workers: [3](#0-2) 

The `work_id` deduplication guard (`id != work_id || id == 0`) is trivially bypassed by setting `work_id = 0` in the injected template, since the condition `id == 0` always evaluates to `true` when the injected value is zero: [4](#0-3) 

The `listen` field is a standard `Option<SocketAddr>` in the production `ClientConfig`: [5](#0-4) 

---

### Impact Explanation

- **Hashpower theft / waste**: Workers immediately switch to the injected template. If the attacker supplies a stale `parent_hash`, all CPU/ASIC cycles spent on that template are wasted.
- **Invalid block submission**: When a worker finds a nonce for the injected template, `submit_nonce` → `submit_block` is called against the CKB node RPC. The node will reject the block, but the miner has already consumed hashpower and generated an RPC round-trip.
- **Persistent disruption**: Because the attacker can re-inject at any time (the endpoint is always open while `listen` mode is active), the miner can be kept perpetually on attacker-controlled work.

The claim of network-wide compact block broadcasts is **overstated**: the CKB node validates submitted blocks before relaying them, so an invalid block is rejected locally and never propagated.

---

### Likelihood Explanation

- The `listen` mode is a documented, supported production feature explicitly described in the miner configuration.
- The listen address defaults to whatever the operator sets; `127.0.0.1` restricts to local processes, but `0.0.0.0` exposes it to the network.
- No privilege escalation is required — any process that can open a TCP connection to the address can exploit this.
- The exploit is a single HTTP POST with a valid-schema JSON body.

---

### Recommendation

Add origin authentication to the notify endpoint. The simplest approach is a configurable shared secret (bearer token or HMAC) that the CKB node includes in the `Authorization` header and the miner verifies in `handle` before calling `update_block_template`. Alternatively, restrict accepted connections to a configurable allowlist of source IPs checked immediately after `listener.accept()`.
