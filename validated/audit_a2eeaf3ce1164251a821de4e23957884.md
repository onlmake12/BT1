Audit Report

## Title
Unbounded RocksDB I/O Amplification via Unauthenticated `GetBlockFilterHashes` Flood — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_hashes_process.rs`)

## Summary
`BlockFilter`, the handler for `SupportProtocols::Filter`, contains no per-peer rate limiter. A single well-formed `GetBlockFilterHashes` message with `start_number=0` triggers up to 4,000 synchronous RocksDB reads (up to 2,000 × `get_block_hash` + 2,000 × `get_block_filter_hash`). Because the Filter protocol is enabled by default and no request throttling exists, any unauthenticated remote peer can drive unbounded DB I/O amplification, degrading or crashing the node.

## Finding Description
**Root cause:** `BlockFilter` holds only `shared: Arc<SyncShared>` with no rate-limiting field. [1](#0-0) 

`BlockFilter::received` dispatches directly to `self.process(...)` with zero throttling: [2](#0-1) 

`try_process` routes `GetBlockFilterHashes` to `GetBlockFilterHashesProcess::execute` with no guard: [3](#0-2) 

Inside `execute`, the loop runs up to `BATCH_SIZE = 2000` iterations: [4](#0-3) 

Each iteration performs two RocksDB reads (`get_block_hash` + `get_block_filter_hash`): [5](#0-4) 

The same unbounded-loop pattern applies to `GetBlockFilters` (BATCH_SIZE=1000, 2 reads/iter, additionally bounded by a 1.8 MB response cap): [6](#0-5) 

**Contrast with `Relayer`**, which has an explicit `governor`-based per-peer rate limiter declared as a struct field: [7](#0-6) 

Checked before any dispatch at 30 req/s per `(PeerIndex, message_type)`: [8](#0-7) 

`BlockFilter` has no equivalent check anywhere in its dispatch path. Filter is enabled by default: [9](#0-8) 

Default peer limits allow up to ~117 inbound peers (`max_peers=125`, `max_outbound_peers=8`): [10](#0-9) 

## Impact Explanation
Each 9-byte `GetBlockFilterHashes(start_number=0)` message triggers up to 4,000 synchronous RocksDB reads with no throttle. With ~117 inbound peers each flooding at line rate, the victim node's RocksDB I/O and async task queue saturate, starving the Sync and Relay protocols of processing time. This causes the node to become unresponsive to block/transaction relay, effectively removing it from the network. This matches the **High** impact class: *"Vulnerabilities which could easily crash a CKB node"* and *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The attack requires only: (1) opening a TCP connection to any CKB full node (no authentication, no PoW, no stake), (2) sending a valid molecule-encoded `GetBlockFilterHashes` message (~9 bytes), and (3) repeating at line rate from multiple IPs. No privileged access, no key material, and no majority hashpower is needed. Since Filter is on by default, all full nodes are equally exposed. The exploit is locally testable with a basic P2P client.

## Recommendation
Add a `governor`-based per-peer rate limiter to `BlockFilter`, mirroring the pattern already used in `Relayer` (`sync/src/relayer/mod.rs` lines 63–98). Specifically:
1. Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `BlockFilter`.
2. Initialize it in `BlockFilter::new` with a quota of 1–2 requests/second per peer per message type.
3. In `try_process`, check the limiter before dispatching any `GetBlock*` variant, returning `StatusCode::TooManyRequests` on excess.
4. Call `rate_limiter.retain_recent()` in the `disconnected` handler to prevent unbounded memory growth.

## Proof of Concept
```python
# Connect N peers, flood GetBlockFilterHashes(start_number=0)
# Molecule-encoded: full_size=9 (LE), union_tag=0 (LE), start_number=0 (8 bytes LE)
MSG = bytes.fromhex("09000000" + "00000000" + "0000000000000000")

peers = [connect_ckb_filter_protocol("victim_ip", 8115) for _ in range(117)]
while True:
    for p in peers:
        p.send(MSG)  # 9 bytes → triggers up to 4000 RocksDB reads on victim
```
Monitor victim RocksDB read IOPS and async task queue depth; assert node stops responding to Sync/Relay messages within seconds. The same test applies to `GetBlockFilters` (BATCH_SIZE=1000, additionally bounded by 1.8 MB response cap).
