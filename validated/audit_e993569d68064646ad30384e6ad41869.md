### Title
Unprivileged Caller Can Steal Relayer's Rent Deposit via `close_signature_set` — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

### Summary
`close_signature_set` accepts any signer as `sol_destination` and transfers the `signature_set` account's full lamport balance to that signer. There is no check that `sol_destination` is the original creator/payer of the `signature_set`. Any unprivileged attacker can drain the rent deposit paid by the original relayer.

### Finding Description
The `CloseSignatureSet` account context is:

```rust
#[derive(Accounts)]
pub struct CloseSignatureSet<'info> {
    #[account(mut)]
    sol_destination: Signer<'info>,          // ← any signer, unconstrained

    #[account(
        seeds = [...],
        bump,
        has_one = signature_set               // ← only checks posted_vaa.signature_set == signature_set.key()
    )]
    posted_vaa: Account<'info, LegacyAnchorized<PostedVaaV1>>,

    #[account(
        mut,
        close = sol_destination               // ← lamports go to whoever sol_destination is
    )]
    signature_set: Account<'info, AccountVariant<SignatureSet>>,
}
``` [1](#0-0) 

The only constraint on `sol_destination` is that it is a `Signer`. The `has_one = signature_set` constraint only verifies that `posted_vaa.info.signature_set == signature_set.key()` — it says nothing about who funded the account. [2](#0-1) 

The `SignatureSet` state stores only `sig_verify_successes`, `message_hash`, and `guardian_set_index` — no creator/payer pubkey is recorded. [3](#0-2) 

Similarly, `PostedVaaV1Info` stores no payer field, so there is no on-chain record of who funded the accounts. [4](#0-3) 

### Impact Explanation
An attacker who observes any on-chain `posted_vaa` + `signature_set` pair (created by a legitimate relayer who paid rent) can call `close_signature_set` with `sol_destination = attacker_pubkey`. Anchor's `close = sol_destination` macro unconditionally transfers all lamports from `signature_set` to `sol_destination`. The original relayer permanently loses their rent deposit. The `signature_set` account is also destroyed, which may break downstream consumers that still reference it.

### Likelihood Explanation
The attack requires no privilege, no leaked key, and no governance action. All inputs are public on-chain. Any transaction submitted to a Solana RPC node with a valid `(posted_vaa, signature_set)` pair and an attacker-controlled `sol_destination` will succeed. The attack is trivially scriptable and can be front-run or executed at any time after a relayer posts a VAA.

### Recommendation
Record the original payer/creator in the `SignatureSet` account at creation time and add a constraint in `CloseSignatureSet` requiring `sol_destination.key() == signature_set.creator`. For example:

1. Add a `creator: Pubkey` field to `SignatureSet`.
2. Populate it in the `verify_signatures` legacy instruction from the `payer` signer.
3. In `CloseSignatureSet`, add `constraint = signature_set.creator == sol_destination.key()`.

### Proof of Concept
1. Keypair A (relayer): calls the legacy `verify_signatures` + `post_vaa` flow, paying rent for `signature_set` and `posted_vaa`.
2. Keypair B (attacker): constructs a transaction with:
   - `sol_destination = B`
   - `posted_vaa = A's posted_vaa PDA`
   - `signature_set = A's signature_set`
3. Calls `close_signature_set`.
4. Anchor's `has_one = signature_set` passes (the pubkey stored in `posted_vaa.info.signature_set` matches the provided account).
5. Anchor's `close = sol_destination` transfers all lamports from `signature_set` to B.
6. Assert: B's lamport balance increased by the full rent of the `signature_set` account; A's rent is permanently lost.
