### Title
Missing Upper-Bound Validation of `msg.value` in `Echo::requestPriceUpdatesWithCallback` Silently Redirects Excess ETH to Provider — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo::requestPriceUpdatesWithCallback` is a payable function that only enforces a **lower bound** on `msg.value` (`msg.value >= requiredFee`). It does not enforce an upper bound and provides no mechanism to refund excess ETH to the caller. Any ETH above `requiredFee` is silently credited to the provider's `accruedFeesInWei` balance rather than returned to the user.

---

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback` computes a `requiredFee` and checks only that `msg.value` is at least that amount:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [1](#0-0) 

Immediately after, the entire `msg.value` minus only the fixed `pythFeeInWei` is stored as the provider's fee for this request:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

And only `pythFeeInWei` is credited to Pyth's accrued fees:

```solidity
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [3](#0-2) 

When `executeCallback` is later called, the provider is credited `(req.fee + msg.value) - pythFee`, meaning the excess from the original request flows entirely into the provider's withdrawable balance:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [4](#0-3) 

The `IEcho` interface explicitly documents that `msg.value` **must equal** `getFee(callbackGasLimit)`, but this is never enforced on the upper side:

```solidity
/// @dev The msg.value must be equal to getFee(callbackGasLimit)
``` [5](#0-4) 

There is no refund path anywhere in the function. The excess is neither returned to `msg.sender` nor tracked separately — it is silently absorbed into the provider's accrued balance.

---

### Impact Explanation

Any user who calls `requestPriceUpdatesWithCallback` with `msg.value > getFee(provider, callbackGasLimit, priceIds)` permanently loses the excess ETH to the provider. The provider can withdraw it at any time via `withdrawAsFeeManager`. The user has no recourse. This constitutes a direct, unrecoverable loss of user funds triggered by a single unprivileged transaction.

---

### Likelihood Explanation

Likelihood is low-to-medium. Users who compute the fee on-chain via `getFee()` and pass it exactly are unaffected. However, users who add a buffer (e.g., to account for fee changes between estimation and submission), use a frontend that over-estimates, or make a simple arithmetic mistake will silently lose the excess. The pattern is realistic enough to occur in production integrations.

---

### Recommendation

Add an upper-bound check requiring `msg.value` to equal exactly `requiredFee`, or refund the excess to the caller:

```solidity
// Option A: strict equality
if (msg.value != requiredFee) revert InvalidFee();

// Option B: refund excess
if (msg.value > requiredFee) {
    (bool ok, ) = msg.sender.call{value: msg.value - requiredFee}("");
    require(ok, "refund failed");
}
```

---

### Proof of Concept

1. Provider registers with `baseFeeInWei = 1000`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`. `pythFeeInWei = 100`.
2. `getFee(provider, 0, priceIds)` returns `1100` (pythFee + baseFee).
3. User calls `requestPriceUpdatesWithCallback{value: 2000}(...)` — sending `900` excess.
4. `req.fee = 2000 - 100 = 1900` (provider fee, includes the 900 excess).
5. `_state.accruedFeesInWei += 100` (Pyth gets only its share).
6. Provider calls `executeCallback`. Provider is credited `1900 + 0 - pythFee_for_update`. The 900 excess is now in the provider's withdrawable balance.
7. User has no way to recover the 900 excess. [6](#0-5)
