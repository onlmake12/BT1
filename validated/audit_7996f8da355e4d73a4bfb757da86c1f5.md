### Title
Unbounded `uri` and `commitmentMetadata` in Entropy Provider Registration Enables Gas Exhaustion / DOS of `getProviderInfoV2` — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract stores provider-supplied `bytes uri` and `bytes commitmentMetadata` in the `ProviderInfo` struct without any length limit. Any address can register as a provider (permissionless) and set arbitrarily large byte fields. Reading back the full struct via `getProviderInfoV2()` or `getProviderInfo()` then consumes gas proportional to the stored data size. On low-cost chains (Polygon, BNB Chain, etc.) this can be used to DOS these view functions or impose large gas costs on callers.

---

### Finding Description

`EntropyStructsV2.ProviderInfo` contains two unbounded dynamic `bytes` fields: [1](#0-0) 

Both `commitmentMetadata` and `uri` are stored as raw `bytes` with no length cap.

The `register()` function writes both fields directly from calldata with no length validation: [2](#0-1) 

The `setProviderUri()` function similarly stores an unbounded `newUri` and additionally reads the **entire old URI from storage into memory** before emitting it in an event — making repeated calls progressively more expensive: [3](#0-2) 

Both `getProviderInfo()` and `getProviderInfoV2()` return the full struct, loading all stored bytes from storage: [4](#0-3) 

---

### Impact Explanation

- `getProviderInfoV2()` and `getProviderInfo()` are public view functions. Any on-chain contract or off-chain caller that reads provider info will pay gas proportional to the size of `uri` + `commitmentMetadata`. With a sufficiently large payload, these calls can exceed the block gas limit on low-cost chains, permanently DOSing provider info queries for that provider.
- `setProviderUri()` reads `oldUri` into memory on every call. If the stored URI is already large, each subsequent update is itself expensive, compounding the cost.
- Fortuna (the off-chain provider service) and any integrating contract that calls `getProviderInfoV2()` to discover provider endpoints would be affected.

---

### Likelihood Explanation

- `register()` is fully permissionless — any address can call it to become a provider.
- The attacker only needs to register once, then incrementally grow the URI via repeated `setProviderUri()` calls. Each individual call can stay within the block gas limit while cumulatively building a URI large enough to DOS reads.
- This is viable on Polygon, BNB Chain, Arbitrum, and other low-cost EVM chains where Pyth Entropy is deployed, where block gas limits are 30M+ and gas prices are fractions of a cent.

---

### Recommendation

Add a maximum length check in both `register()` and `setProviderUri()` for the `uri` and `commitmentMetadata` fields, analogous to the fix applied in Hats Protocol (PR #114). A reasonable cap (e.g., 2048 bytes for `uri`, 512 bytes for `commitmentMetadata`) should be enforced:

```solidity
uint256 public constant MAX_URI_LENGTH = 2048;
uint256 public constant MAX_COMMITMENT_METADATA_LENGTH = 512;

// In register() and setProviderUri():
if (uri.length > MAX_URI_LENGTH) revert EntropyErrors.UriTooLong();
if (commitmentMetadata.length > MAX_COMMITMENT_METADATA_LENGTH) revert EntropyErrors.CommitmentMetadataTooLong();
```

---

### Proof of Concept

1. Attacker calls `register(feeInWei, commitment, largeMetadata, chainLength, largeUri)` with `largeUri` and `largeMetadata` each set to the maximum bytes that fit within the block gas limit for a single write (~30M gas on Polygon ≈ ~100KB of new storage).
2. Attacker calls `setProviderUri(newLargerUri)` repeatedly, each time incrementally growing the stored URI by the maximum amount that fits in one transaction.
3. After enough iterations, `getProviderInfoV2(attackerAddress)` requires reading all stored bytes from storage, consuming gas exceeding the block gas limit.
4. Any contract or off-chain service calling `getProviderInfoV2()` for this provider is now permanently DOSed or faces prohibitive gas costs.

    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }
```

    // Set provider uri. It will revert if provider is not registered.
    function setProviderUri(bytes calldata newUri) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        bytes memory oldUri = provider.uri;
        provider.uri = newUri;
        emit ProviderUriUpdated(msg.sender, oldUri, newUri);
        emit EntropyEventsV2.ProviderUriUpdated(
            msg.sender,
            oldUri,
            newUri,
            bytes("")
        );
    }
```
