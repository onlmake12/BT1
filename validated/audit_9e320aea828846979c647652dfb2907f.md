Based on my thorough investigation of the Sui contracts, here is my assessment:

---

### Title
Permanently Locked Fee Accumulation in PriceInfoObject with No Protocol Withdrawal Path — (`target_chains/sui/contracts/sources/price_info.move`, `pyth.move`)

### Summary
Every call to `update_single_price_feed` deposits `Coin<SUI>` fees unconditionally into the `PriceInfoObject` via a dynamic object field. No function exists anywhere in the production contract to withdraw or transfer those coins. The `fee_recipient_address` stored in `State` is explicitly documented as unused. All collected fees are permanently locked and irrecoverable.

### Finding Description

**Step 1 — Fee deposit is unconditional and precedes the price update:**

In `pyth.move`, `update_single_price_feed` deposits the fee before any freshness check:

```
assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);
// store fee coins within price info object
price_info::deposit_fee_coins(price_info_object, fee);   // line 277
// ... then update_cache (which checks is_fresh_update)
``` [1](#0-0) 

Even if the price data is stale (not a fresh update), the fee is already deposited and cannot be refunded.

**Step 2 — Fees are stored as a dynamic object field on `PriceInfoObject`:**

`deposit_fee_coins` in `price_info.move` stores the `Coin<SUI>` under `FEE_STORAGE_KEY = b"fee_storage"` as a dynamic object field on the `PriceInfoObject`. Each subsequent call joins the new coins into the existing balance. [2](#0-1) 

**Step 3 — No withdrawal function exists anywhere in the contract:**

A complete search of all `.move` files under `target_chains/sui/contracts/sources/` for `withdraw`, `transfer.*fee`, `fee.*transfer` returns **zero matches**. The only fee-related functions on `PriceInfoObject` are:
- `get_balance` (read-only)
- `deposit_fee_coins` (deposit-only, `public`) [3](#0-2) 

**Step 4 — The `fee_recipient_address` in `State` is explicitly vestigial:**

The `set_fee_recipient.move` module opens with:

> "The previous version of the contract sent the fees to a recipient address but **this state is not used anymore**. This module is kept for backward compatibility." [4](#0-3) 

The `State` struct stores `fee_recipient_address` and the governance action `SET_FEE_RECIPIENT` can update it, but neither field nor action has any connection to the coins accumulating in `PriceInfoObject`. [5](#0-4) 

**Step 5 — No governance sweep function exists:**

The six governance actions are: `CONTRACT_UPGRADE`, `SET_GOVERNANCE_DATA_SOURCE`, `SET_DATA_SOURCES`, `SET_UPDATE_FEE`, `SET_STALE_PRICE_THRESHOLD`, `SET_FEE_RECIPIENT`. None involves moving coins out of any `PriceInfoObject`. [6](#0-5) 

**Step 6 — Tests confirm the accumulation and implicitly confirm no withdrawal:**

The test suite explicitly asserts that fees accumulate in `PriceInfoObject` after each update call, and no test exercises any withdrawal path (because none exists). [7](#0-6) 

### Impact Explanation

Every `update_single_price_feed` call by any relayer permanently locks `Coin<SUI>` inside the target `PriceInfoObject`. Over time, across all price feeds and all updates, the total locked SUI grows without bound. The protocol can never collect this revenue. The `fee_recipient_address` governance parameter is a no-op. The fee model is economically broken: fees are charged and collected but are permanently inaccessible to the protocol.

### Likelihood Explanation

This is not a conditional exploit — it is a systemic property of every price update call. Any unprivileged relayer triggers it simply by calling `update_single_price_feed` with a valid VAA and sufficient fee. No special preconditions, no privileged access, no timing dependency.

### Recommendation

1. Add a `withdraw_fee_coins` function gated by `LatestOnly` (or a governance-controlled capability) that extracts the `Coin<SUI>` from the `FEE_STORAGE_KEY` dynamic field and transfers it to `state::get_fee_recipient(pyth_state)`.
2. Either remove `fee_recipient_address` / `SET_FEE_RECIPIENT` governance action or wire them to the actual withdrawal path.
3. Consider whether fees should be refunded when `is_fresh_update` returns false (stale update still charges the full fee).

### Proof of Concept

Invariant test (pseudocode):
```move
// After N calls to update_single_price_feed:
let total_fees_paid = N * DEFAULT_BASE_UPDATE_FEE;
assert!(price_info::get_balance(&price_info_object) == total_fees_paid);

// Attempt to find any function that reduces the balance — none exists.
// grep for "withdraw" in all .move sources → 0 results.
// grep for "FEE_STORAGE_KEY" → only deposit and read paths.
// Invariant: balance is monotonically increasing and never decreases.
``` [8](#0-7)

    public fun get_balance(price_info_object: &PriceInfoObject): u64 {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            return 0
        };
        let fee = dynamic_object_field::borrow<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY);
        coin::value(fee)
    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
```
