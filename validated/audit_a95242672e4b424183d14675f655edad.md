### Title
Unbounded `callbackGasLimit` and Missing `gasleft()` Pre-Check in `Echo.executeCallback` Enable Permanent Fund Locking - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` accepts a user-controlled `callbackGasLimit` (`uint32`) with no upper-bound validation. `Echo.executeCallback` then forwards that gas to an external callback inside a plain `try...catch` block with no `gasleft()` pre-check. Unlike `Entropy.revealWithCallback`, which uses `excessivelySafeCall` and an explicit `(startingGas * 31) / 32 > gasLimit` guard, Echo has no such protection. Combined with the absence of any cancel/refund mechanism, a user can create a request whose gas limit exceeds the chain's block gas limit, making it permanently unfulfillable and locking their deposited funds.

---

### Finding Description

**Root cause 1 — No maximum `callbackGasLimit` validation:**

`requestPriceUpdatesWithCallback` stores the caller-supplied `callbackGasLimit` directly with no upper-bound check: [1](#0-0) 

The field is a `uint32`, so the maximum accepted value is `4,294,967,295` gas — far above the block gas limit of any production EVM chain. Compare this to `Entropy.revealWithCallback`, which enforces `MAX_GAS_LIMIT = type(uint16).max * 10,000 = 655,350,000` via `roundTo10kGas`: [2](#0-1) 

Echo has no equivalent constant or check.

**Root cause 2 — No `gasleft()` pre-check before the `try...catch`:**

`executeCallback` clears the request and credits the provider *before* the callback, then calls the consumer inside a plain `try...catch`: [3](#0-2) 

There is no check that `gasleft() >= callbackGasLimit * 64/63 + catch_overhead` before the `try` statement. If the outer context retains only `gasleft() / 64` gas after forwarding `callbackGasLimit` to the callback, and that residual is insufficient to execute the `catch` block (which emits `PriceUpdateCallbackFailed`), the entire transaction reverts — including the `clearRequest` and fee-credit effects. The own test confirms this: [4](#0-3) 

`Entropy.revealWithCallback` solves this with `excessivelySafeCall` and an explicit gas-sufficiency guard that reverts with `InsufficientGas()` when the outer context cannot safely forward the required gas: [5](#0-4) 

**Root cause 3 — No cancel/refund mechanism:**

Echo has no `cancelRequest` or refund path. The only way to clear a request is a successful `executeCallback`. If a request is permanently unfulfillable, the deposited fee is locked forever. [6](#0-5) 

The developer-acknowledged TODO comment in the contract itself flags this risk: [7](#0-6) 

---

### Impact Explanation

**Permanent fund locking:** A user who sets `callbackGasLimit` to any value exceeding the chain's block gas limit (e.g., `type(uint32).max = 4,294,967,295`) creates a request that no provider can ever fulfill — no block can supply that much gas. Because there is no cancel/refund path, the `req.fee` deposited at request time is permanently locked in the contract.

**Transaction reversion griefing:** Even for reasonable gas limits, a provider who calls `executeCallback` with insufficient outer-context gas will have the entire transaction revert (including the `clearRequest` and fee-credit effects). Unlike Entropy, which reverts early with a clear `InsufficientGas()` error, Echo gives no on-chain signal about the minimum gas required, making it difficult for providers to size their transactions correctly.

---

### Likelihood Explanation

- If `feePerGasInWei > 0`, the fee for `callbackGasLimit = type(uint32).max` is enormous, making accidental self-locking unlikely but not impossible for users who copy-paste or miscalculate.
- If a provider sets `feePerGasInWei = 0`, any user can create an unfulfillable request at minimal cost.
- The missing `gasleft()` guard is a latent operational hazard for every provider integrating with Echo, since the only documentation hint is a comment in `IEcho.sol` ("Requires 1.5x the callback gas limit") with no on-chain enforcement. [8](#0-7) 

---

### Recommendation

1. **Add a maximum `callbackGasLimit` constant** in `EchoState` (e.g., matching Entropy's `655,350,000`) and enforce it in `requestPriceUpdatesWithCallback`.
2. **Add a `gasleft()` pre-check** in `executeCallback` before the `try` block, analogous to Entropy's `(startingGas * 31) / 32 > gasLimit` guard, reverting with a clear error (e.g., `InsufficientGas()`) when the outer context cannot safely forward the required gas.
3. **Add a cancel/refund function** so users can recover funds from requests that cannot be fulfilled.

---

### Proof of Concept

```solidity
// 1. Provider registers with feePerGasInWei = 0
echo.registerProvider(baseFee, feePerFeed, 0);

// 2. Attacker/user creates a request with callbackGasLimit = type(uint32).max
//    Fee = baseFee + feePerFeed * numFeeds (gas component = 0)

uint96 fee = echo.getFee(provider, impossibleGasLimit, priceIds);

    provider, publishTime, priceIds, impossibleGasLimit
);

// 3. No block on any EVM chain can supply 4.29B gas.
//    executeCallback can never be called successfully.
//    No cancelRequest() exists.
//    => req.fee is permanently locked in Echo contract.
```

    function roundTo10kGas(uint32 gas) internal pure returns (uint16) {
        if (gas > MAX_GAS_LIMIT) {
            revert EntropyErrors.MaxGasLimitExceeded();
        }

        uint32 gas10k = gas / TEN_THOUSAND;
        if (gas10k * TEN_THOUSAND < gas) {
            gas10k += 1;
        }
        // Note: safe cast here should never revert due to the if statement above.
        return SafeCast.toUint16(gas10k);
    }
