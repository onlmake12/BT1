### Title
Unbounded `last_n_numbers` Vec allocation via tiny `difficulty_boundary` in `GetLastStateProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The input-size guard in `GetLastStateProofProcess::execute` only bounds `difficulties.len() + last_n_blocks * 2`, but the actual `last_n_numbers` Vec allocated at line 319 is bounded by `last_block_number - difficulty_boundary_block_number`. When an attacker sends `difficulty_boundary = U256::one()`, the binary-search helper immediately returns block 0 as the boundary, causing `last_n_numbers = (0..last_block_number)` — a Vec whose size equals the entire chain height. No guard prevents this.

---

### Finding Description

**Guard that is supposed to protect the server** (lines 201–205):

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // = 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) 

With attacker payload `difficulties=[], last_n_blocks=1`, the check evaluates to `0 + 2 = 2 ≤ 1000` — it passes unconditionally.

**The vulnerable allocation** (lines 318–319):

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
``` [2](#0-1) 

`difficulty_boundary_block_number` is set by `get_first_block_total_difficulty_is_not_less_than`. When `min_total_difficulty = U256::one()`, the very first check (line 31) fires:

```rust
if start_total_difficulty >= *min_total_difficulty {
    return Some((start_block_number, start_total_difficulty));
}
``` [3](#0-2) 

The genesis block's total difficulty on CKB mainnet is `0x20000` (far above 1), so `difficulty_boundary_block_number` is set to `start_block_number = 0`.

The adjustment guard at lines 313–316 only *increases* `difficulty_boundary_block_number` when there are too few blocks after it, but `N - 0 ≥ last_n_blocks=1` is always true for a synced node, so it does not fire:

```rust
if last_block_number - difficulty_boundary_block_number < last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
``` [4](#0-3) 

Result: `last_n_numbers = (0..N).collect()` — a Vec of N `u64` values, where N is the current chain height.

The Vec is then passed into `complete_headers`, which performs O(N) database lookups (`get_ancestor`, `get_block`, `chain_root_mmr`) for every element: [5](#0-4) 

---

### Impact Explanation

For a mainnet node at block ~14,000,000:
- `last_n_numbers` Vec: 14M × 8 bytes ≈ **112 MB** per request
- `complete_headers` then allocates `Vec<packed::VerifiableHeader>` of the same length, each header being ~200+ bytes → **~2.8 GB** per request
- Each iteration calls `get_ancestor` (O(log N) DB reads), `get_block`, and `chain_root_mmr` — **O(N log N) disk I/O**

A single attacker peer can trigger this repeatedly, causing OOM or sustained CPU/IO exhaustion. The node process will be killed by the OS OOM killer or become unresponsive.

---

### Likelihood Explanation

- Any unauthenticated P2P peer can send a `GetLastStateProof` message to a light-client-protocol-enabled CKB node.
- The message is structurally valid and passes all input validation checks.
- The only required knowledge is the tip block hash (publicly available via `GetLastState`/`SendLastState`).
- No PoW, no key, no privileged role required.

---

### Recommendation

After computing `difficulty_boundary_block_number`, enforce that `last_n_numbers` length is bounded by `GET_LAST_STATE_PROOF_LIMIT`:

```rust
let last_n_count = last_block_number.saturating_sub(difficulty_boundary_block_number);
if last_n_count as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many last_n blocks");
}
let last_n_numbers = (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
```

Alternatively, rewrite the upfront guard to bound the *effective* `last_n_numbers` size rather than just the raw `last_n_blocks` field, since `difficulty_boundary_block_number` can be pushed to 0 regardless of `last_n_blocks`.

---

### Proof of Concept

```
peer → GetLastStateProof {
    last_hash:           <current tip hash from SendLastState>,
    start_hash:          <genesis hash>,
    start_number:        0,
    last_n_blocks:       1,
    difficulty_boundary: 0x0000...0001,   // U256::one()
    difficulties:        [],
}
```

Execution trace:
1. Guard: `0 + 1*2 = 2 ≤ 1000` → passes. [6](#0-5) 
2. `last_block_number - 0 > 1` → enters else branch. [7](#0-6) 
3. `get_first_block_total_difficulty_is_not_less_than(0, N, &1)` → genesis difficulty `≥ 1` → returns `(0, genesis_diff)`. [3](#0-2) 
4. `N - 0 ≥ 1` → adjustment guard skipped. [4](#0-3) 
5. `(0..N).collect::<Vec<_>>()` → N-element Vec allocated. [2](#0-1) 
6. `complete_headers` iterates all N entries with DB lookups → OOM / node crash. [8](#0-7)
