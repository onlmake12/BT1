Audit Report

## Title
Missing Ancestor-Bound Check in `GetTransactionsProofProcess::execute` Enables Unbounded Resource Exhaustion Without Peer Ban — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary
`GetTransactionsProofProcess::execute` partitions `tx_hashes` by `is_main_chain` only, with no check that each transaction's block number is at or before `last_block`. An unprivileged peer can supply up to 1000 `tx_hashes` from main-chain blocks mined *after* `last_hash`, causing the server to perform full block DB reads and CBMT proof construction for every such block before `reply_proof` fails with `InternalError` (500). Because `should_ban()` only fires for 4xx codes, no ban is ever issued and the attack is freely repeatable.

## Finding Description

**Step 1 — Entry guards pass.**
`last_hash` is verified to be on the main chain at lines 43–49, and `tx_hashes` is capped at `GET_TRANSACTIONS_PROOF_LIMIT = 1000` at lines 37–39. [1](#0-0) [2](#0-1) [3](#0-2) 

**Step 2 — Partition checks only `is_main_chain`, not block height.**
Lines 54–64 classify a tx as `found` if and only if its block is on the main chain. There is no `tx_info.block_number <= last_block.number()` guard. Transactions in blocks 101–200 when `last_hash` is block 100 all pass this filter. [4](#0-3) 

**Step 3 — Expensive work is performed for every out-of-range block.**
For each block in `txs_in_blocks` the server performs: a full block DB read (`get_block`), a CBMT Merkle proof construction over all transactions in the block, an uncle hash fetch, and an extension fetch. All of this happens before any height validation. [5](#0-4) 

**Step 4 — Out-of-range positions are pushed without a bound check.**
`leaf_index_to_pos(block.number())` is pushed for each block at line 116 with no comparison against `last_block.number()`. [6](#0-5) 

**Step 5 — `reply_proof` builds an MMR bounded at `last_block.number() - 1` then calls `gen_proof` with the out-of-range positions.**
`chain_root_mmr(last_block.number() - 1)` creates an MMR of size `leaf_index_to_mmr_size(last_block.number() - 1)`, covering only blocks 0..last_block. Positions for blocks beyond that range cause `gen_proof` to return an error, which is mapped to `StatusCode::InternalError`. [7](#0-6) [8](#0-7) 

**Step 6 — `InternalError` (500) never bans the peer.**
`should_ban()` returns `Some` only for codes in `400..500`. Code 500 falls outside that range, so `nc.ban_peer` is never called. The `received` handler only logs a warning. [9](#0-8) [10](#0-9) 

## Impact Explanation
Each malicious request triggers up to 1000 full-block DB reads plus CBMT proof construction (O(n log n) in per-block transaction count) before returning an error. Because no ban is issued, a single peer can repeat this in a tight loop, monotonically exhausting disk I/O and CPU on any node serving the light-client protocol. This matches the allowed CKB bounty impact: **High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation
The light-client protocol is a standard P2P sub-protocol. Any peer that has observed the chain can trivially craft the attack: pick any confirmed `last_hash` at block N, then supply `tx_hashes` from blocks N+1 and beyond. No privilege, key, or hashpower is required. The attack is fully repeatable with zero rate limiting or backoff enforced by the server.

## Recommendation
In the partition at lines 54–64, additionally require `tx_info.block_number <= last_block.number()`:

```rust
let (found, missing): (Vec<_>, Vec<_>) = self
    .message
    .tx_hashes()
    .to_entity()
    .into_iter()
    .partition(|tx_hash| {
        snapshot
            .get_transaction_info(tx_hash)
            .map(|tx_info| {
                snapshot.is_main_chain(&tx_info.block_hash)
                    && tx_info.block_number <= last_block.number()  // ADD THIS
            })
            .unwrap_or_default()
    });
```

This moves the rejection to before any expensive DB work. Transactions in blocks after `last_hash` are treated as missing, which is semantically correct and consistent with the protocol's intent.

Note: `GetBlocksProofProcess::execute` in `get_blocks_proof.rs` has the same structural pattern (partition by `is_main_chain` only, no height bound) and should be audited for the same class of issue. [11](#0-10) 

## Proof of Concept
1. Mine 200 blocks; ensure blocks 101–200 each contain at least one non-coinbase transaction.
2. Send `GetTransactionsProof { last_hash: block[100].hash, tx_hashes: [tx from block 101, …, tx from block 200] }` (up to 1000 hashes).
3. Observe: server performs up to 1000 full block fetches + CBMT proofs, then `reply_proof` returns `InternalError` when `gen_proof` is called with positions outside the MMR bounded at block 99.
4. Observe: peer is **not** banned — `nc.not_banned(peer_index)` passes, consistent with the existing test structure at line 100 of the test file. [12](#0-11) 
5. Repeat in a tight loop; measure monotonically increasing DB read and CPU time with no rate limiting applied.
