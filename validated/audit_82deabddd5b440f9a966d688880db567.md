Audit Report

## Title
Peer Store Permanently Blocked by Single-Network-Group Flood via Discovery Gossip — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

`check_purge` contains an integer-division truncation bug: when all `ADDR_COUNT_LIMIT` (16,384) entries belong to a single `/16` network group, `take(len / 2)` with `len == 1` evaluates to `take(0)`, producing zero eviction candidates and returning `Err(PeerStoreError::EvictionFailed)`. An unprivileged remote peer can reach this state by flooding discovery `Nodes` messages whose addresses all share the same IPv4 `/16` prefix, permanently preventing the victim node from recording any new peer addresses via `add_addr`.

## Finding Description

**Root cause — `check_purge` integer-division edge case**

The second eviction pass groups all stored addresses by network group, then takes the top `len / 2` groups to evict from: [1](#0-0) 

When `len == 1`, `1 / 2 == 0` in integer arithmetic, so `take(0)` immediately exhausts the iterator. The subsequent check finds `candidate_peers.is_empty() == true` and returns the error: [2](#0-1) 

**First eviction pass is bypassed**

`add_addr` stores entries with `last_connected_at_ms = 0` and `attempts_count = 0`: [3](#0-2) [4](#0-3) 

`is_connectable` returns `true` for these entries because the only non-connectable condition for `last_connected_at_ms == 0` requires `attempts_count >= ADDR_MAX_RETRIES (3)`, which is not satisfied at `attempts_count == 0`: [5](#0-4) 

So the first eviction pass removes nothing, and execution falls through to the broken network-group path.

**Network group computation confirms /16 granularity**

All IPv4 addresses sharing the same first two octets map to the identical `Group::IP4([bits[0], bits[1]])` key: [6](#0-5) 

So 16,384 addresses from a single `/16` block produce exactly `len == 1`.

**Attacker entry point — discovery gossip**

`add_new_addrs` iterates every received address and calls `peer_store.add_addr` with no per-session or per-source-IP quota: [7](#0-6) 

**Constants confirmed** [8](#0-7) 

## Impact Explanation

Once the peer store reaches 16,384 entries all in one `/16` group, every subsequent call to `add_addr` returns `Err(EvictionFailed)`. The node can no longer record any newly discovered peer addresses. `fetch_addrs_to_attempt` returns nothing useful (all injected entries have `last_connected_at_ms = 0`, failing the "connected within 3 days" filter), and `fetch_addrs_to_feeler` returns only attacker-controlled addresses. The victim is effectively isolated from the honest peer graph — a direct prerequisite for an eclipse attack that can cause consensus deviation.

This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*, and is a credible precursor to the **Critical** impact class: *Vulnerabilities which could easily cause consensus deviation*.

## Likelihood Explanation

- No privilege required: any peer that completes a P2P handshake can send `Nodes` messages.
- `MAX_ADDR_TO_SEND = 1000` items and `MAX_ADDRS = 3` addresses per `Node` item allows up to 3,000 addresses per message; filling 16,384 slots requires approximately 6 messages across as few as 1–6 sessions.
- `is_valid_addr` filters RFC-1918/loopback addresses but passes any globally routable `/16` block (e.g., `1.2.0.0/16`), of which many exist.
- `AddrManager.add` deduplicates by exact multiaddr, so the attacker needs 16,384 distinct addresses within the `/16` — trivially achievable with varying ports.
- The attack is permanent: once the store is full, no self-healing mechanism exists to recover.

## Recommendation

Replace `take(len / 2)` with a guard that always selects at least one group when `len >= 1`:

```rust
let take_count = std::cmp::max(1, len / 2);
peers.into_iter().take(take_count)...
```

Additionally:
- Enforce a per-network-group cap (e.g., no more than `ADDR_COUNT_LIMIT / 32` entries per `/16` group) in `add_addr`.
- Add a per-session or per-source-IP rate limit in `add_new_addrs`.

## Proof of Concept

```rust
// Fill peer store with 16384 addresses all in 1.2.x.x/16
let mut peer_store: PeerStore = Default::default();
for i in 0u32..16384 {
    let ip = format!("1.2.{}.{}", (i >> 8) & 0xff, i & 0xff);
    let addr: Multiaddr = format!("/ip4/{}/tcp/8114/p2p/QmFakeId{}", ip, i).parse().unwrap();
    peer_store.add_outbound_addr(addr, Flags::COMPATIBILITY);
}
// All 16384 entries are in Group::IP4([1, 2]) — len == 1
// Attempt to add one more address via the discovery path
let new_addr: Multiaddr = "/ip4/5.6.7.8/tcp/8114/p2p/QmHonestPeer".parse().unwrap();
let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
// check_purge: first pass removes nothing (all connectable),
// second pass: len=1, take(1/2)=take(0), candidate_peers empty -> EvictionFailed
assert!(matches!(result, Err(e) if e.to_string().contains("EvictionFailed")));
```

The same state is reachable via discovery gossip by sending `Nodes` messages containing addresses in `1.2.0.0/16` — no special privileges required beyond completing a P2P handshake.
