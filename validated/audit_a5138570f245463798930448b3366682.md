### Title
Unbounded Descendant Iteration in `PoolMap::remove_entry` Enables O(D) CPU Spike During Block Commit Processing - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

When a transaction is committed in a block, `remove_committed_tx` calls `pool_map.remove_entry` directly. Inside `remove_entry`, `update_descendants_index_key` is invoked **before** the entry's links are removed, causing it to traverse and update every descendant of the committed transaction. The number of descendants is bounded only by `max_tx_pool_size` (bytes), not by `max_ancestors_count`. An attacker who submits a wide fan-out transaction tree and gets the root committed can force O(D) CPU work on the node during block processing, where D is the total descendant count.

---

### Finding Description

`remove_entry` in `tx-pool/src/component/pool_map.rs` calls both `update_ancestors_index_key` and `update_descendants_index_key` **before** calling `remove_entry_links`: [1](#0-0) 

```rust
pub(crate) fn remove_entry(&mut self, id: &ProposalShortId) -> Option<TxEntry> {
    self.entries.remove_by_id(id).map(|entry| {
        self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
        self.update_descendants_index_key(&entry.inner, EntryOp::Remove);
        self.remove_entry_edges(&entry.inner);
        self.remove_entry_links(id);   // ← links removed AFTER the traversals
        ...
    })
}
```

`update_descendants_index_key` calls `calc_descendants`, which walks the full descendant sub-tree while links are still intact: [2](#0-1) 

The safe bulk-removal path (`remove_entry_and_descendants`) avoids this by pre-removing all links first, so each subsequent `remove_entry` call sees an empty descendant set: [3](#0-2) 

However, `remove_committed_tx` in `tx-pool/src/pool.rs` calls `remove_entry` **directly**, not `remove_entry_and_descendants`: [4](#0-3) 

The same direct call pattern appears in `remove_expired`: [5](#0-4) 

`max_ancestors_count` limits the **depth** of a dependency chain but places **no bound on fan-out (width)**. A transaction with 100 outputs can have 100 children, each with 100 grandchildren, etc. The total descendant count D is bounded only by `max_tx_pool_size` divided by the minimum transaction size. [6](#0-5) 

---

### Impact Explanation

Every time a block is committed, `remove_committed_tx` is called for each committed transaction. If the committed transaction is the root of a large fan-out tree in the pool, `update_descendants_index_key` iterates over all D descendants to update their ancestor-weight bookkeeping. With a default pool of 72 MB and a minimum transaction size of ~100 bytes, D can reach hundreds of thousands. This O(D) work occurs on the critical block-processing path, potentially delaying the node's tip advancement and causing it to fall behind peers. The same issue applies to `remove_expired`, which is called periodically.

---

### Likelihood Explanation

An unprivileged tx-pool submitter (RPC `send_transaction`) can craft this scenario:

1. Submit root transaction T0 with many outputs (bounded by tx size limit).
2. Submit many children T1…Tn each spending a distinct output of T0.
3. Repeat for grandchildren, etc., filling the pool with descendants of T0.
4. Pay a fee rate high enough for T0 to be proposed and committed.

When T0 is committed, the node performs O(D) descendant updates. The attacker must pay fees for all pool transactions, which limits the attack's cost-effectiveness, but the pool size is large enough that the CPU cost is non-trivial. The attack is realistic for a well-funded adversary targeting a specific node.

---

### Recommendation

1. **Add a `max_descendants_count` limit** analogous to `max_ancestors_count`. Reject or evict transactions that would cause any pool entry's descendant count to exceed this limit. This mirrors the existing ancestor-count guard in `check_and_record_ancestors`. [7](#0-6) 

2. **In `remove_committed_tx`**, after removing the committed entry, update descendants' ancestor weights incrementally using the already-maintained `links` structure rather than a full `calc_descendants` traversal, or pre-remove links before calling `remove_entry` (as `remove_entry_and_descendants` does).

3. **In `remove_expired`**, use `remove_entry_and_descendants` instead of `remove_entry` to also evict now-orphaned descendants and avoid the full descendant traversal.

---

### Proof of Concept

```
1. Attacker calls send_transaction(T0) where T0 has 100 outputs.
2. Attacker calls send_transaction(T1..T100), each spending one output of T0,
   each with 100 outputs.
3. Attacker calls send_transaction(T101..T10100), each spending one output of
   T1..T100.
4. Pool now contains T0 + 100 + 10,000 = 10,101 descendants of T0.
5. Attacker pays a high fee rate for T0 so it is proposed and committed.
6. On block commit, remove_committed_tx(T0) → pool_map.remove_entry(&T0_id)
   → update_descendants_index_key iterates all 10,100 descendants.
7. Each iteration performs a hash-map modify on a live pool entry.
8. Scaled to pool capacity (~720,000 min-size txs), this O(D) traversal
   occurs synchronously on the block-processing critical path.
```

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn update_descendants_index_key(&mut self, parent: &TxEntry, op: EntryOp) {
        let descendants: HashSet<ProposalShortId> =
            self.links.calc_descendants(&parent.proposal_short_id());
        for desc_id in &descendants {
            // update child score
            self.entries.modify_by_id(desc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_ancestor_weight(parent),
                    EntryOp::Add => e.inner.add_ancestor_weight(parent),
                };
                e.score = e.inner.as_score_key();
            });
        }
    }

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }
