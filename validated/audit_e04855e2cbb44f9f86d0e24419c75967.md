Audit Report

## Title
Stale Descendant-Weight Accounting in `remove_entry_and_descendants` Leaves Ancestor Entries with Inflated Eviction Score — (`tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::remove_entry_and_descendants` strips all link records for every entry it is about to delete **before** calling `remove_entry` on each one. Because `update_ancestors_index_key` resolves ancestors through those same link records, it finds an empty ancestor set and never calls `sub_descendant_weight` on ancestor transactions that remain in the pool. Those ancestors permanently retain inflated `descendants_fee / descendants_size / descendants_cycles / descendants_count` fields, corrupting the `EvictKey` used to rank transactions for eviction when the pool is full.

## Finding Description
`remove_entry_and_descendants` collects the target and all its descendants, then calls `remove_entry_links` on every one of them before iterating to call `remove_entry`: [1](#0-0) 

Inside `remove_entry`, `update_ancestors_index_key` is called: [2](#0-1) 

`update_ancestors_index_key` resolves the ancestor set via `self.links.calc_ancestors(...)`: [3](#0-2) 

`calc_ancestors` traverses `self.links.inner` (the `parents` map): [4](#0-3) 

Because `remove_entry_links` already removed every entry from `self.links.inner` in the pre-pass, `calc_ancestors` returns an empty set for every entry being processed. `sub_descendant_weight` is never called on any ancestor that remains in the pool: [5](#0-4) 

The existing test for `remove_entry_and_descendants` only verifies that removed entries are absent and that `calc_descendants` no longer lists them — it does **not** assert that the remaining ancestor's `descendants_count / descendants_fee` are decremented: [6](#0-5) 

By contrast, the direct `remove_entry` path works correctly because links are still intact at call time: [7](#0-6) 

`resolve_conflict` calls `remove_entry_and_descendants` for every conflicting input and dep: [8](#0-7) 

## Impact Explanation
`EvictKey` is computed directly from the stale fields: [9](#0-8) 

An ancestor whose removed descendants had a higher fee rate than itself retains an inflated `descendants_feerate`, making its `fee_rate` appear higher than it truly is. When the pool is full and `next_evict_entry` selects the lowest-`EvictKey` entry to drop, that ancestor is ranked as more valuable than it deserves, so other legitimate transactions are evicted in its place. This constitutes **damage to CKB economy**: an unprivileged attacker can cheaply and repeatedly cause legitimate users' transactions to be evicted from the mempool, preventing them from being confirmed. This matches the allowed impact: **Vulnerabilities which could easily damage CKB economy** (Critical, 15001–25000 points).

## Likelihood Explanation
The trigger requires no special privileges. An attacker submits a low-fee root transaction X, then high-fee descendants A → B spending X's output, then a conflicting transaction A′ spending the same input as A. `resolve_conflict` calls `remove_entry_and_descendants(A)`, leaving X with permanently inflated descendant stats. The only cost is the fee for A′. The attack is repeatable: each round inflates a different ancestor. No majority hash power, no social engineering, and no victim mistakes are required.

## Recommendation
Collect and apply ancestor updates **before** tearing down links. Specifically, for each entry being removed, resolve its ancestors while links are still intact and call `sub_descendant_weight` only on ancestors that are **not** themselves in the removal set. Only after all ancestor updates are applied should `remove_entry_links` be called. This mirrors the logic already used by the single-entry `remove_entry` path and ensures `EvictKey` values remain accurate for entries that stay in the pool.

## Proof of Concept
**Setup**: pool contains chain `X → A → B`.

After insertion, X's tracked state:
- `descendants_fee = 600`, `descendants_size = 500`, `descendants_cycles = 500`, `descendants_count = 3`

**Trigger**: submit tx A′ spending the same input as A. `resolve_conflict` calls `remove_entry_and_descendants(A)`.

**Expected state of X**:
- `descendants_fee = 100`, `descendants_size = 100`, `descendants_cycles = 100`, `descendants_count = 1`

**Actual state of X (bug)**:
- `descendants_fee = 600`, `descendants_size = 500`, `descendants_cycles = 500`, `descendants_count = 3`

A regression test can be added to `tx-pool/src/component/tests/score_key.rs` following the pattern of `test_remove_entry_and_descendants`: after calling `map.remove_entry_and_descendants(&tx2_id)`, assert that `map.get(&tx1_id).unwrap().descendants_count == 1` and `map.get(&tx1_id).unwrap().descendants_fee == Capacity::shannons(100)`. Under the current code, both assertions fail.

```rust
    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
```

```rust
    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
```

```rust
    pub fn sub_descendant_weight(&mut self, entry: &TxEntry) {
        self.descendants_count = self.descendants_count.saturating_sub(1);
        self.descendants_size = self.descendants_size.saturating_sub(entry.size);
        self.descendants_cycles = self.descendants_cycles.saturating_sub(entry.cycles);
        self.descendants_fee = Capacity::shannons(
            self.descendants_fee
                .as_u64()
                .saturating_sub(entry.fee.as_u64()),
        );
    }
```
