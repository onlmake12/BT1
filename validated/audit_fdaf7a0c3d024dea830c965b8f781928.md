### Title
Division-by-Zero Panic in `deriveCrossRate` When `price2 == 0` — (`File: target_chains/ethereum/sdk/solidity/PythUtils.sol`)

---

### Summary

`PythUtils.deriveCrossRate` validates that `price2` is not negative but does not check for `price2 == 0`. When `price2 == 0` is passed, the denominator in the `Math.mulDiv` call becomes zero, causing an unhandled Solidity panic revert (division-by-zero) instead of a meaningful `PythErrors` revert. Any on-chain protocol that calls `deriveCrossRate` with a zero-valued quote price will have its transaction revert unexpectedly, causing a DoS of that protocol's price-dependent logic.

---

### Finding Description

`deriveCrossRate` in `PythUtils.sol` computes a cross-rate `price1 / price2` scaled to a target exponent. The function guards against negative prices:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
``` [1](#0-0) 

However, `price2 == 0` passes this check. The function then proceeds to use `price2` as part of the denominator in two `Math.mulDiv` calls:

**Branch 1 (`deltaExponent > 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    10 ** uint64(deltaExponent),
    uint64(price2)          // denominator = 0 → panic
);
``` [2](#0-1) 

**Branch 2 (`deltaExponent <= 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    1,
    10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // 10^n * 0 = 0 → panic
);
``` [3](#0-2) 

`Math.mulDiv` explicitly panics on a zero denominator:

```solidity
if (high == 0) {
    return low / denominator;  // Solidity reverts if denominator == 0
}
if (denominator <= high) {
    panic(ternary(denominator == 0, DIVISION_BY_ZERO, UNDER_OVERFLOW));
}
``` [4](#0-3) 

The result is an unhandled panic (opcode `0x4e487b71` with code `0x12`) rather than a `PythErrors.NegativeInputPrice` or a new `PythErrors.ZeroInputPrice` revert. Integrators cannot distinguish this from other panics and cannot catch it with a specific selector.

---

### Impact Explanation

Any on-chain DeFi protocol that uses `deriveCrossRate` to compute synthetic prices (e.g., ETH/EUR from ETH/USD and EUR/USD) will have its price-dependent transactions revert with an unhandled panic whenever the quote feed (`price2`) reports a price of zero. This causes a complete DoS of that protocol's price-dependent functions for the duration that the feed reports zero. The official developer documentation actively promotes `deriveCrossRate` for cross-rate use cases. [5](#0-4) 

---

### Likelihood Explanation

Pyth's `PythStructs.Price.price` is an `int64` with no protocol-level enforcement of non-zero values. A price feed can legitimately report `0` during market closure, for newly listed assets, or in edge-case aggregation scenarios. The Lazer payload reference notes "Non-zero when present" only as an invariant for Lazer feeds; the core Pyth feed format does not enforce this. Any integrator following the documented `deriveCrossRate` example without adding their own zero-price guard is exposed.

---

### Recommendation

Add an explicit zero-price check for `price2` before the division:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
if (price2 == 0) {
    revert PythErrors.ZeroInputPrice(); // new error, or reuse NegativeInputPrice
}
``` [6](#0-5) 

This mirrors the existing pattern for `price1 < 0` and ensures callers receive a meaningful, catchable revert selector rather than an opaque panic.

        vm.expectRevert(); // catches the division-by-zero panic

            0,            // price2: EUR/USD = 0 (zero price, e.g. market closed)
            0,            // expo2
            -8            // targetExponent
        );
        // The revert is an unhandled panic (0x4e487b71 / 0x12),
        // NOT PythErrors.NegativeInputPrice or any catchable PythErrors selector.
    }
}

    function deriveCrossRate(
        int64 price1,
        int32 expo1,
        int64 price2,
        int32 expo2,
        int32 targetExponent
    ) public pure returns (uint256 crossRate) {
        // Check if the input prices are negative
        if (price1 < 0 || price2 < 0) {
            revert PythErrors.NegativeInputPrice();
        }
        // Check if the input exponents are valid and not less than -255
        if (expo1 < -255 || expo2 < -255 || targetExponent < -255) {
            revert PythErrors.InvalidInputExpo();
        }

        // note: This value can be negative.
        int64 deltaExponent = int64(expo1 - (expo2 + targetExponent));

        // Bounds check: prevent overflow/underflow with base 10 exponentiation
        // Calculation: 10 ** n <= (2 ** 256 - 63) - 1
        //              n <= log10((2 ** 193) - 1)
        //              n <= 58.2
        if (deltaExponent > 58 || deltaExponent < -58)
            revert PythErrors.ExponentOverflow();

        uint256 result;
        if (deltaExponent > 0) {
            result = Math.mulDiv(
                uint64(price1),
                10 ** uint64(deltaExponent),
                uint64(price2)
            );
        } else {
            result = Math.mulDiv(
                uint64(price1),
                1,
                10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)
            );
        }

        return result;
    }
}
