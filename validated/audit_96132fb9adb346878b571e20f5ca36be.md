### Title
`total_tx_size` / `total_tx_cycles` Overwritten With Pre-Eviction Value in `add_entry`, Inflating Pool Size Accounting - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the pool's aggregate size and cycle counters (`total_tx_size`, `total_tx_cycles`) are computed **before** a potential in-path eviction occurs, then written back **after** that eviction has already decremented the same counters. The result is that the counters are inflated by the size of the evicted entries, causing the pool to believe it is larger than it actually is. This mirrors the `collectedEther` bug exactly: an accumulator is incremented by the full gross amount rather than the net amount after a side-effect reduces the actual quantity.

---

### Finding Description

`PoolMap::add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```rust
// Step 1: snapshot new totals BEFORE any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//  ^ total_tx_size = self.total_tx_size + entry.size  (stale snapshot)

// Step 2: may evict existing entries, which calls remove_entry →
//         update_stat_for_remove_tx → DECREMENTS self.total_tx_size
evicts = self.check_and_record_ancestors(&mut entry)?;

// Step 3: overwrite self.total_tx_size with the PRE-eviction snapshot
self.total_tx_size = total_tx_size;    // ← stale value, ignores evictions
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

The eviction path inside `check_and_record_ancestors` is triggered when a new transaction's ancestor count exceeds `max_ancestors_count` and some of those ancestors are `cell_ref_parents` (pool transactions that reference the same cell dep as the new transaction's inputs). In that case, `remove_entry_and_descendants` is called, which calls `remove_entry`, which calls `update_stat_for_remove_tx` and correctly decrements `self.total_tx_size`. [2](#0-1) [3](#0-2) [4](#0-3) 

But immediately after, lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot, erasing the correct decrement. The final value is:

```
self.total_tx_size = old_total + new_entry.size
```

whereas the correct value should be:

```
self.total_tx_size = old_total - evicted_sizes + new_entry.size
```

The inflation equals the total serialized size of all evicted entries.

---

### Impact Explanation

`total_tx_size` is the sole counter used by `limit_size` to decide whether to evict further transactions:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee-rate entries
}
``` [5](#0-4) 

An inflated `total_tx_size` causes `limit_size` to evict additional legitimate transactions that would otherwise remain in the pool. The attacker can:

1. **Targeted mempool eviction (DoS):** By repeatedly triggering the bug, inflate `total_tx_size` enough that `limit_size` evicts specific high-value or time-sensitive transactions submitted by other users.
2. **Pool admission denial:** The inflated counter persists across subsequent `add_entry` calls. `updated_stat_for_add_tx` uses `self.total_tx_size` to check for overflow, so future legitimate transactions may be rejected with `Reject::Full` even though the pool has physical capacity. [6](#0-5) 

---

### Likelihood Explanation

The trigger condition requires:

1. The attacker submits a transaction whose ancestor count in the pool exceeds `max_ancestors_count` (default 125).
2. Some of those ancestors are `cell_ref_parents` — pool transactions that reference the same cell dep out-point as the attacker's transaction inputs.

An attacker who observes the mempool (via `get_raw_tx_pool` RPC) can identify existing transactions and craft a new transaction that shares a cell dep with them, then build a chain of 125+ ancestors to exceed the limit. This is achievable by any unprivileged RPC caller or P2P transaction relayer with no special privileges. [7](#0-6) 

---

### Recommendation

Move the computation of `total_tx_size` / `total_tx_cycles` to **after** `check_and_record_ancestors` completes, so it reflects the post-eviction state:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, Default::default()));
    }
    // Validate that adding this entry won't overflow (pre-check only)
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    let evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Compute net totals AFTER evictions have already updated self.total_tx_size
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, subtract the evicted sizes from the pre-computed snapshot before writing it back.

---

### Proof of Concept

**Setup:** Pool with `max_tx_pool_size = 10_000` bytes, currently holding transactions totaling `9_500` bytes. `max_ancestors_count = 125`.

1. Attacker observes pool transaction `P` that uses cell dep `D`.
2. Attacker builds a chain of 125 transactions `A1 → A2 → … → A125`, each spending the previous output, all using cell dep `D`. Total chain size: `1_000` bytes.
3. Attacker submits `A125` (the tip of the chain) via `send_transaction` RPC.
4. Inside `add_entry`:
   - `updated_stat_for_add_tx` snapshots `total_tx_size = 9_500 + size(A125)`.
   - `check_and_record_ancestors` finds 124 ancestors + 1 = 125 > `max_ancestors_count`. `P` is a `cell_ref_parent`. `P` (say 200 bytes) is evicted via `remove_entry_and_descendants`, decrementing `self.total_tx_size` to `9_300 + size(A125)`.
   - Lines 218–219 overwrite `self.total_tx_size` back to `9_500 + size(A125)`.
5. `limit_size` now sees `total_tx_size > 10_000` and evicts additional legitimate transactions from the pool, even though the actual pool size is well under the limit. [8](#0-7)

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
