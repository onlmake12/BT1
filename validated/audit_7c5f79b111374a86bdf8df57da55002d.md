Audit Report

## Title
Missing Per-Peer Rate Limit in Filter Protocol Handler Enables I/O Exhaustion — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

## Summary
The `BlockFilter` protocol handler holds no rate limiter field and performs no rate check before dispatching incoming messages. Any peer connected via the Filter protocol can send `GetBlockFilters` messages in a tight loop, each triggering up to 1000 RocksDB reads and accumulating up to ~1.8 MB of molecule-encoded response data per request. The `Relayer` and `HolePunching` handlers both carry explicit `governor`-based per-peer rate limiters; `BlockFilter` has none.

## Finding Description
`BlockFilter` holds only `shared: Arc<SyncShared>` with no rate limiter field: [1](#0-0) 

`try_process` dispatches directly to `GetBlockFiltersProcess::execute()` (and the other two sub-handlers) with no rate check at any point in the call chain: [2](#0-1) 

`GetBlockFiltersProcess::execute()` iterates up to `BATCH_SIZE = 1000` blocks, performing two DB reads per block (`get_block_hash` + `get_block_filter`) and accumulating up to 1.8 MB of response data per request before breaking: [3](#0-2) [4](#0-3) 

A grep across the entire `sync/src/filter/` directory confirms zero occurrences of `rate_limiter`, `governor`, or `RateLimiter`. By contrast, `Relayer` defines a `governor`-based `RateLimiter<(PeerIndex, u32)>` type alias and installs it at construction: [5](#0-4) [6](#0-5) [7](#0-6) 

`Relayer.try_process` checks the rate limiter before every dispatch and returns `StatusCode::TooManyRequests` on violation: [8](#0-7) 

`HolePunching` applies the same guard at its `received` entry point: [9](#0-8) 

`BlockFilter` has no equivalent guard anywhere in `sync/src/filter/`.

## Impact Explanation
A single attacker peer connecting via the Filter protocol can saturate the node's RocksDB read throughput and async task queue. Each `GetBlockFilters` request costs up to 2000 DB reads and ~1.8 MB of encoding and send work with no throttle applied. At even modest send rates this monopolizes the I/O path shared with consensus-critical Relay and Sync protocol messages, causing processing delays or drops for those messages. This maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, since the attacker's cost is negligible (a 9-byte message in a loop) while the per-request server cost is substantial and unbounded by any guard.

## Likelihood Explanation
The Filter protocol is optional but is enabled on nodes that serve light clients. The attack requires only a standard P2P connection — no credentials, no PoW, no stake. The attacker sends a fixed 9-byte `GetBlockFilters(start_number=0)` message in a tight loop. The gap is directly analogous to the one the `Relayer` rate limiter was added to close, and the exploit is immediately reproducible on any node with block filter service enabled and ≥1 block filter built.

## Recommendation
Add a `governor`-based `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter`, mirroring the pattern in `Relayer`: [5](#0-4) [6](#0-5) 

Check it at the top of `try_process` before dispatching any `GetBlockFilters`, `GetBlockFilterHashes`, or `GetBlockFilterCheckPoints` message, returning `StatusCode::TooManyRequests` on violation. Call `rate_limiter.retain_recent()` in the `disconnected` handler to bound memory growth, as `Relayer` does: [10](#0-9) 

## Proof of Concept
1. Enable block filter service on a CKB node; mine ≥1000 blocks so filters are built.
2. Connect a test peer using `SupportProtocols::Filter`.
3. Send `GetBlockFilters { start_number: 0 }` in a tight loop with no sleep.
4. Observe: each message triggers up to 2000 RocksDB reads and a ~1.8 MB `async_send_message_to` call with no throttle.
5. Measure latency of concurrent Relay/Sync messages — assert they are delayed proportionally to the flood rate.
6. Assert no `TooManyRequests` status is ever returned by the Filter handler — confirmed by code: no such check exists anywhere in `sync/src/filter/`. [2](#0-1)
