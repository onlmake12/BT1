Audit Report

## Title
Quadratic Work in Indexer `append` Due to Unbounded Linear Scan Over Block Transactions for Within-Block Cell Dependencies — (`File: util/indexer/src/indexer.rs`)

## Summary
The `Indexer::append` function performs an O(T) linear scan over all block transactions for every input whose referenced cell was produced within the same block, because `self.store.get` reads only from the committed store and misses cells written to the uncommitted batch. This creates O(T² × I) total work per block. An unprivileged user can trigger worst-case behavior by submitting a chain of N transactions spending each other's outputs in a single block, causing the indexer to fall behind the chain tip.

## Finding Description
In `util/indexer/src/indexer.rs`, the `append` method iterates over all transactions (outer loop, O(T)) and all inputs per transaction (inner loop, O(I)). For each input, it first attempts a committed-store lookup:

```rust
if let Some(stored_live_cell) = self.store.get(&key_vec)?.or_else(|| {
    transactions
        .iter()
        .enumerate()
        .find(|(_i, tx)| tx.hash() == out_point.tx_hash())
        ...
})
```

The batch being built in this same `append` call has not yet been committed at the time `self.store.get` is called (line 318: `let mut batch = self.store.batch()?`; commit happens only at the end). Therefore, any cell created by an earlier transaction **in the same block** is invisible to `self.store.get`, which returns `None`, causing the `or_else` closure to execute a full O(T) `find` scan over all block transactions. The total complexity for a block with T transactions each having I within-block inputs is O(T² × I). There is no cap, hash-map cache, or early-exit beyond the block size limit.

## Impact Explanation
This maps to **Low (501–2000 points): Any other important performance improvements for CKB**. The indexer is a local service; the CKB node itself continues operating normally. The degradation is confined to the indexer falling behind the chain tip, making it unavailable to wallets, dApps, and tooling that depend on it. It does not crash the CKB node, cause network congestion, or affect consensus.

## Likelihood Explanation
Any unprivileged transaction sender can trigger this by constructing a chain of N transactions (each spending the previous output) and having them included in a single block. This is a standard, economically rational pattern (UTXO consolidation, batch payments) indistinguishable from legitimate use. The attacker pays O(N) in fees while imposing O(N²) indexer work. No special privileges or majority hashpower are required.

## Recommendation
Build an in-memory `HashMap<OutPoint, Vec<u8>>` from the current block's outputs **before** the main input-processing loop. Populate it once in O(T × outputs_per_tx) and consult it in O(1) per input lookup, replacing the `or_else` linear scan:

```rust
let mut block_cell_map: HashMap<Vec<u8>, Vec<u8>> = HashMap::new();
for (i, tx) in transactions.iter().enumerate() {
    for (idx, (output, data)) in tx.outputs_with_data_iter().enumerate() {
        let out_point = OutPoint::new(tx.hash(), idx as u32);
        let key = Key::OutPoint(&out_point).into_vec();
        let val: Vec<u8> = Value::Cell(block_number, i as u32, &output, &data).into();
        block_cell_map.insert(key, val);
    }
}
// In the input loop:
if let Some(stored_live_cell) = self.store.get(&key_vec)?
    .or_else(|| block_cell_map.get(&key_vec).cloned())
{ ... }
```

This reduces total complexity to O(T × I).

## Proof of Concept
1. Construct a chain of N = 3,000 transactions: `tx[0]` creates one output; `tx[k]` spends `tx[k-1]`'s output.
2. Submit all 3,000 transactions to the mempool and have them included in a single block (valid by CKB consensus).
3. When the indexer calls `append` on this block, for each `tx[k]` (k = 1…2999): `store.get` returns `None` (cell not yet committed), triggering the `or_else` scan over all 3,000 transactions → 3,000 hash comparisons per input.
4. Total: ~9,000,000 hash comparisons for one block instead of ~3,000.
5. Measure wall-clock time of `append` with and without the chained pattern to confirm the quadratic growth.
