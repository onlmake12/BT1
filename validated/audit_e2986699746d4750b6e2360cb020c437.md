### Title
Entropy Keeper Underpaid When Callback Fails and Retry Is Required — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
When `revealWithCallback` is called and the consumer's `entropyCallback` fails (transitioning the request to `CALLBACK_FAILED`), the keeper must submit a second `revealWithCallback` transaction to retry. The fee paid by the user at request time is credited to the provider immediately and is sized to cover only **one** reveal transaction. The retry transaction's gas cost is entirely uncompensated, causing the keeper/provider to lose money on every callback failure.

### Finding Description
In `Entropy.sol`, the user pays a fee at request time inside `requestHelper`. The fee is immediately credited to the provider:

```solidity
providerInfo.accruedFeesInWei += providerFee;
``` [1](#0-0) 

The fee is sized by `getProviderFee`, which scales linearly with the callback gas limit for a **single** reveal execution:

```solidity
return provider.feeInWei + additionalFee; // proportional to one gasLimit
``` [2](#0-1) 

When the keeper calls `revealWithCallback` and the callback reverts, the contract sets `callbackStatus = CALLBACK_FAILED` and the request remains on-chain:

```solidity
req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED;
``` [3](#0-2) 

The keeper must then submit a **second** `revealWithCallback` transaction. This second call takes the `else` branch (the recovery path), which invokes the callback directly without error catching:

```solidity
IEntropyConsumer(callAddress)._entropyCallback(
    sequenceNumber,
    provider,
    randomNumber
);
``` [4](#0-3) 

The keeper's off-chain fee adjustment logic (`adjust_fee_if_necessary`) calculates the target fee as a function of `default_gas_limit` × current gas price — covering exactly **one** reveal transaction:

```rust
let gas_limit: u128 = u128::from(provider_info.default_gas_limit);
let max_callback_cost: u128 = estimate_tx_cost(middleware, legacy_tx, gas_limit).await...;
``` [5](#0-4) 

Neither the on-chain fee formula nor the off-chain fee adjustment accounts for the cost of a retry transaction when the callback fails.

### Impact Explanation
The keeper/provider bears the full gas cost of the retry `revealWithCallback` transaction with no compensation. On chains with high gas prices, a single callback failure can erase the profit from many successful reveals. If a consumer contract is deployed that reliably fails on the first callback attempt (e.g., due to a logic bug or deliberate design), every request from that consumer forces the keeper to pay for two reveal transactions while receiving only one fee payment.

### Likelihood Explanation
Callback failures are a documented, expected scenario — the contract explicitly handles `CALLBACK_FAILED` state and the documentation warns that callbacks must not revert. Any consumer contract with a bug in `entropyCallback`, or any user who deliberately deploys a consumer that fails transiently, triggers this condition. The likelihood is **medium**: failures are not the common case, but they are a reachable and realistic scenario for any production deployment.

### Recommendation
The fee calculation should account for the possibility of callback failure and retry:

1. **Increase the minimum fee** to cover the cost of at least one retry reveal transaction (e.g., add `feeInWei` as a buffer for the retry overhead).
2. **Update `adjust_fee_if_necessary`** in `apps/fortuna/src/keeper/fee.rs` to factor in a retry multiplier when computing `max_callback_cost`, analogous to the GMX recommendation to update gas estimation configuration to account for the failure case.
3. Alternatively, track failed callbacks and charge an additional fee from the provider's `accruedFeesInWei` when a retry is needed, so the cost is borne by the fee pool rather than the keeper's wallet.

### Proof of Concept

1. User calls `requestV2` with a consumer contract whose `entropyCallback` always reverts on the first invocation. Fee is credited to provider immediately.
2. Keeper calls `revealWithCallback` (Transaction 1). The `excessivelySafeCall` catches the revert, emits `CallbackFailed`, sets `callbackStatus = CALLBACK_FAILED`. Keeper pays ~`defaultGasLimit × gasPrice` in gas with no additional compensation.
3. Keeper calls `revealWithCallback` again (Transaction 2, recovery path). The callback now succeeds. Keeper pays another ~`defaultGasLimit × gasPrice` in gas.
4. Total keeper gas cost ≈ 2× the cost of one reveal. Total fee received = 1× `feeInWei`. The keeper is net-negative on every such request.

The `CALLBACK_FAILED` state and the two-path flow in `revealWithCallback` are the necessary on-chain steps that make this loss concrete and repeatable. [6](#0-5) [7](#0-6)

        ) {
            revert EntropyErrors.InvalidRevealCall();
        }

        bytes32 randomNumber;
        (randomNumber, ) = revealHelper(
            req,
            userContribution,
            providerContribution
        );

        // If the request has an explicit gas limit, then run the new callback failure state flow.
        //
        // Requests that haven't been invoked yet will be invoked safely (catching reverts), and
        // any reverts will be reported as an event. Any failing requests move to a failure state
        // at which point they can be recovered. The recovery flow invokes the callback directly
        // (no catching errors) which allows callers to easily see the revert reason.
        if (
            req.gasLimit10k != 0 &&
            req.callbackStatus == EntropyStatusConstants.CALLBACK_NOT_STARTED
        ) {
            req.callbackStatus = EntropyStatusConstants.CALLBACK_IN_PROGRESS;
            bool success;
            bytes memory ret;
            uint256 startingGas = gasleft();
            (success, ret) = req.requester.excessivelySafeCall(
                // Warning: the provided gas limit below is only an *upper bound* on the gas provided to the call.
                // At most 63/64ths of the current context's gas will be provided to a call, which may be less
                // than the indicated gas limit. (See CALL opcode docs here https://www.evm.codes/?fork=cancun#f1)
                // Consequently, out-of-gas reverts need to be handled carefully to ensure that the callback
                // was truly provided with a sufficient amount of gas.
                uint256(req.gasLimit10k) * TEN_THOUSAND,
                256, // copy at most 256 bytes of the return value into ret.
                abi.encodeWithSelector(
                    IEntropyConsumer._entropyCallback.selector,
                    sequenceNumber,
                    provider,
                    randomNumber
                )
            );
            uint32 gasUsed = SafeCast.toUint32(startingGas - gasleft());
            // Reset status to not started here in case the transaction reverts.
            req.callbackStatus = EntropyStatusConstants.CALLBACK_NOT_STARTED;

            if (success) {
                emit RevealedWithCallback(
                    EntropyStructConverter.toV1Request(req),
                    userContribution,
                    providerContribution,
                    randomNumber
                );
                emit EntropyEventsV2.Revealed(
                    provider,
                    req.requester,
                    req.sequenceNumber,
                    randomNumber,
                    userContribution,
                    providerContribution,
                    false,
                    ret,
                    SafeCast.toUint32(gasUsed),
                    bytes("")
                );
                clearRequest(provider, sequenceNumber);
            } else if (
                (startingGas * 31) / 32 >
                uint256(req.gasLimit10k) * TEN_THOUSAND
            ) {
                // The callback reverted for some reason.
                // We don't use ret to condition the behavior here (out-of-gas or other revert), as we have found that some user contracts
                // catch out-of-gas errors and revert with a different error.
                // In this case, ensure that the callback was provided with sufficient gas. Technically, 63/64ths of the startingGas is forwarded,
                // but we're using 31/32 to introduce a margin of safety.
                emit CallbackFailed(
                    provider,
                    req.requester,
                    sequenceNumber,
                    userContribution,
                    providerContribution,
                    randomNumber,
                    ret
                );
                emit EntropyEventsV2.Revealed(
                    provider,
                    req.requester,
                    sequenceNumber,
                    randomNumber,
                    userContribution,
                    providerContribution,
                    true,
                    ret,
                    SafeCast.toUint32(gasUsed),
                    bytes("")
                );
                req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED;
            } else {
                // Callback reverted by (potentially) running out of gas, but the calling context did not have enough gas
                // to run the callback. This is a corner case that can happen due to the nuances of gas passing
                // in calls (see the comment on the call above).
                //
                // (Note that reverting here plays nicely with the estimateGas RPC method, which binary searches for
                // the smallest gas value that causes the transaction to *succeed*. See https://github.com/ethereum/go-ethereum/pull/3587 )
                revert EntropyErrors.InsufficientGas();
            }

pub async fn adjust_fee_if_necessary(
    contract: Arc<InstrumentedSignablePythContract>,
    chain_id: ChainId,
    provider_address: Address,
    legacy_tx: bool,
    min_profit_pct: u64,
    target_profit_pct: u64,
    max_profit_pct: u64,
    min_fee_wei: u128,
    max_fee_wei: Option<u128>,
    high_water_pnl: &mut Option<U256>,
    sequence_number_of_last_fee_update: &mut Option<u64>,
    metrics: Arc<KeeperMetrics>,
) -> Result<()> {
    let provider_info = contract
        .get_provider_info_v2(provider_address)
        .call()
        .await
        .map_err(|e| anyhow!("Error while getting provider info. error: {:?}", e))?;

    if provider_info.fee_manager != contract.wallet().address() {
        return Err(anyhow!("Fee manager for provider {:?} is not the keeper wallet. Fee manager: {:?} Keeper: {:?}", contract.provider(), provider_info.fee_manager, contract.wallet().address()));
    }

    // Calculate target window for the on-chain fee.
    let middleware = contract.client();
    let gas_limit: u128 = u128::from(provider_info.default_gas_limit);
    let max_callback_cost: u128 = estimate_tx_cost(middleware, legacy_tx, gas_limit)
        .await
        .map_err(|e| anyhow!("Could not estimate transaction cost. error {:?}", e))?;
