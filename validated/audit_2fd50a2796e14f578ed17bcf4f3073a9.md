### Title
Echo Protocol Fee Double-Counted in `accruedFeesInWei` — Admin Fee Withdrawal Permanently Blocked - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the protocol fee (`_state.pythFeeInWei`) is accrued to `_state.accruedFeesInWei` at request time, but the same ETH is also credited in full to the provider's `accruedFeesInWei` at callback time. This double-counting means the contract's actual ETH balance is always less than `_state.accruedFeesInWei` by `pythFeeInWei × (number of fulfilled requests)`. The admin's `withdrawFees` call will revert with `"Failed to send fees"` once providers have withdrawn their share, permanently blocking Echo protocol fee collection.

---

### Finding Description

**Step 1 — Request time:** `requestPriceUpdatesWithCallback` charges the user `req.fee = _state.pythFeeInWei + providerFees` and immediately accrues the protocol cut:

```solidity
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [1](#0-0) 

**Step 2 — Callback time:** `executeCallback` credits the provider with the *entire* `req.fee` (which already contains `_state.pythFeeInWei`) minus only the underlying Pyth oracle fee:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

Because `req.fee` already embeds `_state.pythFeeInWei`, the provider's credit includes the Echo protocol fee a second time. The accounting invariant breaks:

| Variable | Value |
|---|---|
| `_state.accruedFeesInWei` | `pythFeeInWei` |
| `providers[p].accruedFeesInWei` | `req.fee + msg.value − pythFee` (includes `pythFeeInWei`) |
| **Total accounting** | `2 × pythFeeInWei + providerFees + msg.value − pythFee` |
| **Actual contract ETH** | `pythFeeInWei + providerFees + msg.value − pythFee` |

The accounting overstates the real ETH balance by exactly `pythFeeInWei` per fulfilled request.

**Step 3 — Withdrawal:** `withdrawFees` passes the balance check (`_state.accruedFeesInWei >= amount`) but the low-level ETH transfer fails because the ETH has already been credited to the provider:

```solidity
require(_state.accruedFeesInWei >= amount, "Insufficient balance");
_state.accruedFeesInWei -= amount;
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "Failed to send fees");
``` [3](#0-2) 

The `require(sent, ...)` reverts the whole transaction, so `_state.accruedFeesInWei` is not permanently corrupted, but the withdrawal is permanently blocked as long as providers have withdrawn their share.

---

### Impact Explanation

- **Echo protocol fees are unwithdrawable** once any provider has withdrawn their `accruedFeesInWei`. The admin's `withdrawFees` always reverts with `"Failed to send fees"`.
- **Echo protocol fees are effectively transferred to providers** — providers receive `req.fee + msg.value − pythFee`, which includes the Echo protocol cut that was supposed to go to the admin.
- This is a permanent, passive loss of protocol revenue on every fulfilled request, not requiring any special timing or front-running.

---

### Likelihood Explanation

This triggers on **every fulfilled request** with a non-zero `_state.pythFeeInWei`. No adversarial action is required — any legitimate provider fulfilling a callback causes the accounting divergence. Registration as a provider is permissionless (`registerProvider` has no access control), so any user can become a provider and withdraw the double-counted fees. [4](#0-3) 

---

### Recommendation

Deduct the Echo protocol fee from the provider's credit at callback time so the two accounting variables do not overlap:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee - _state.pythFeeInWei);
```

Alternatively, do not accrue `_state.accruedFeesInWei` at request time; instead deduct it from the provider credit at callback time and add it to `_state.accruedFeesInWei` then, ensuring the two accounting buckets are always mutually exclusive.

---
