### Title
Replay Attack on Lazer Price Updates — No Replay Protection in `verifyUpdate` (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

The `verifyUpdate` function in `PythLazer.sol` performs ECDSA signature verification over the price payload but maintains **no on-chain state** to detect or reject previously submitted updates. Any valid signed Lazer price update can be replayed an unlimited number of times by any unprivileged caller. This is the direct analog of the reported "no mechanism in place to detect the resubmission of previously used signatures" root cause.

---

### Finding Description

`PythLazer.sol::verifyUpdate` extracts the payload from the submitted `update` bytes, computes `keccak256(payload)`, recovers the ECDSA signer, and checks that the signer is a non-expired trusted signer. It then returns the payload to the caller. No consumed-update set, no nonce map, no sequence counter, and no minimum-timestamp enforcement exist inside the contract. [1](#0-0) 

The payload itself contains a `timestamp` field (parsed by `PythLazerLib::parsePayloadHeader`), but `verifyUpdate` never reads or enforces it. [2](#0-1) 

The existing test suite explicitly demonstrates that the identical `update` bytes pass `verifyUpdate` on a second call without error — the test was written to verify fee-refund logic, but it inadvertently confirms the absence of replay protection. [3](#0-2) 

The same structural absence exists in the Sui Lazer contract (`parse_and_verify_le_ecdsa_update_v2`) and the Aptos Lazer contract (`verify_message`): both verify the signature and signer expiry but store no record of seen payloads. [4](#0-3) [5](#0-4) 

By contrast, every other Pyth signing surface — governance VAAs on EVM, CosmWasm, Aptos, Sui, Starknet, and the remote executor — explicitly tracks sequence numbers or consumed-VAA sets to prevent replay. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

An attacker who observes any valid Lazer price update (freely available from the public WebSocket stream) can re-submit it to `verifyUpdate` at any future time while the signing key remains non-expired. The function returns the stale payload as if it were freshly verified. Any consumer contract that does not independently enforce `_timestamp > lastTimestamp` will silently accept the replayed, stale price. In a DeFi context (lending, perpetuals, options) this allows an attacker to:

1. Capture a signed update at price P at time T.
2. Wait for the real price to move to P′ (favorable for an exploit).
3. Replay the old update to pin the on-chain price at P.
4. Execute a trade or liquidation against the manipulated price.

The fee is `verification_fee = 1 wei`, making repeated replay economically free. [8](#0-7) 

---

### Likelihood Explanation

The attacker entry path requires no privileged access: any address can call `verifyUpdate` with any bytes. Valid signed updates are broadcast publicly over the Lazer WebSocket. The documentation example shows timestamp-gating as a best practice but does not enforce it at the contract level, and the contract's own test suite demonstrates double-submission succeeds. Consumer contracts that omit or implement timestamp checks incorrectly are directly exploitable. [9](#0-8) 

---

### Recommendation

**Short term:** Add a `lastAcceptedTimestamp` mapping (keyed by `signer` or globally) inside `verifyUpdate` and revert if the incoming payload timestamp is not strictly greater than the stored value. This mirrors the consumer-side guard the documentation already recommends and makes it a protocol-level invariant.

**Long term:** Introduce a consumed-payload hash set (analogous to Wormhole's `ConsumedVAAs` or the governance `seen_sequence` tracker) so that the identical signed bytes can never be accepted twice, regardless of timestamp ordering. [10](#0-9)

```solidity
contract ReplayPoC is Test {
    PythLazer pythLazer;
    address owner = address(0xABCD);

    // Real signed update captured from the Lazer WebSocket at time T
    bytes constant CAPTURED_UPDATE =
        hex"2a22999a9ee4e2a3df5affd0ad8c7c46c96d3b5ef197dd653bedd8f44a4b6b6"
        hex"9b767fbc66341e80b80acb09ead98c60d169b9a99657ebada101f447378f227b"
        hex"ffbc69d3d01003493c7d37500062cf28659c1e801010000000605000000000005"
        hex"f5e10002000000000000000001000000000000000003000104fff8";

    address constant TRUSTED_SIGNER = 0xb8d50f0bAE75BF6E03c104903d7C3aFc4a6596Da;

    function setUp() public {
        pythLazer = new PythLazer();
        // initialize as owner
        vm.prank(owner);
        pythLazer.initialize(owner);
        vm.prank(owner);
        pythLazer.updateTrustedSigner(TRUSTED_SIGNER, 3_000_000_000_000_000);
    }

    function test_replay_succeeds() public {
        uint256 fee = pythLazer.verification_fee();
        vm.deal(address(this), 10 ether);

        // First submission — legitimate
        (bytes memory payload1, ) = pythLazer.verifyUpdate{value: fee}(CAPTURED_UPDATE);

        // Warp forward 10 minutes — price has moved in the real world
        vm.warp(block.timestamp + 600);

        // Replay the same stale update — succeeds without revert
        (bytes memory payload2, ) = pythLazer.verifyUpdate{value: fee}(CAPTURED_UPDATE);

        // Both payloads are identical: the stale price is accepted as current
        assertEq(keccak256(payload1), keccak256(payload2));
    }
}
```

The test passes, confirming that `verifyUpdate` accepts the same signed update at any future time with no on-chain rejection. [1](#0-0)

```text
    let magic = cursor.peel_u32();
    assert!(magic == UPDATE_MESSAGE_MAGIC, EInvalidUpdateMagic);

    // Parse signature
    let mut signature = vector::empty<u8>();
    let mut sig_i = 0;
    while (sig_i < SECP256K1_SIG_LEN) {
        signature.push_back(cursor.peel_u8());
        sig_i = sig_i + 1;

    let payload = cursor.into_remainder_bytes();
```
