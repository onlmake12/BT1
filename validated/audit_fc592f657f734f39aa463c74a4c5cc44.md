### Title
Stale Price Returned Without Staleness Validation in `PythAggregatorV3.latestRoundData()` — (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink `AggregatorV3Interface` adapter, distributed as part of `@pythnetwork/pyth-sdk-solidity` and recommended for Chainlink-to-Pyth migrations. Every price-returning function in this contract — including `latestRoundData()`, `latestAnswer()`, `getRoundData()`, `decimals()`, and `latestTimestamp()` — calls `pyth.getPriceUnsafe(priceId)` with no staleness check whatsoever. This is a direct structural analog to the reported vulnerability: a price feed interface that returns arbitrarily old prices without validation.

---

### Finding Description

All price-returning functions in `PythAggregatorV3` use `getPriceUnsafe`:

```solidity
// latestRoundData — lines 110–118
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
roundId = uint80(price.publishTime);
return (
    roundId,
    int256(price.price),
    price.publishTime,
    price.publishTime,  // updatedAt = publishTime, no freshness check
    roundId
);
```

`getPriceUnsafe` is explicitly documented in `IPyth.sol` as returning "the most recent price update in this contract without any recency checks" and that "the returned price update may be arbitrarily far in the past." No call to `getPriceNoOlderThan` or any equivalent staleness guard is present in any of the five affected functions.

The same pattern applies to:
- `latestAnswer()` (line 54)
- `getRoundData()` (line 89)
- `decimals()` (line 41)
- `latestTimestamp()` (line 59)

Additionally, `roundId` is derived as `uint80(price.publishTime)` — a timestamp cast to `uint80`. This is semantically incompatible with Chainlink's round-based model and can silently overflow for large timestamps, producing a misleading `roundId` value that downstream consumers may use for round-validity checks.

---

### Impact Explanation

`PythAggregatorV3` is the official Pyth-provided drop-in replacement for Chainlink feeds, actively recommended in Pyth's migration documentation. Any DeFi protocol (lending, derivatives, fee registries) that integrates this adapter and calls `latestRoundData()` will:

1. Receive a price from arbitrarily far in the past with no revert or warning.
2. Receive an `updatedAt` value equal to a potentially very old `publishTime`, which downstream staleness checks (e.g., `require(updatedAt != 0)`, `require(block.timestamp - updatedAt < threshold)`) may or may not catch depending on their implementation.
3. Receive a `roundId` that is a truncated timestamp, not a real round counter — protocols checking `answeredInRound >= roundId` for Chainlink-style round validity will get a tautologically true result that provides no actual safety.

The concrete impact is that a downstream protocol using this adapter can be exploited via stale price arbitrage: an attacker waits for the on-chain Pyth price to drift from the true market price (no one is obligated to push updates), then interacts with the downstream protocol at the stale price.

---

### Likelihood Explanation

Pyth is a pull oracle — price updates are not automatic. If no keeper or user calls `updateFeeds()`, the on-chain price can become arbitrarily stale. An unprivileged attacker can:
1. Monitor the on-chain `publishTime` of the target feed.
2. Wait for a favorable price divergence between the stale on-chain price and the true market price.
3. Interact with the downstream protocol (e.g., borrow against overvalued collateral, liquidate at a stale price) without ever needing to update the feed.

This is a realistic, low-effort attack path requiring no privileged access.

---

### Recommendation

Replace `getPriceUnsafe` with `getPriceNoOlderThan` in all price-returning functions, using a configurable `maxAge` parameter set at construction time:

```solidity
uint256 public maxAge;

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    // ...
}
```

This causes `latestRoundData()` to revert with `StalePrice()` if the feed has not been updated within the acceptable window, matching the behavior downstream Chainlink-compatible protocols expect.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a Pyth contract and a price feed ID.
2. Do not call `updateFeeds()` for an extended period (e.g., 1 hour).
3. Call `latestRoundData()` — it returns the price from 1 hour ago with no revert.
4. The returned `updatedAt` is `price.publishTime` from 1 hour ago; no on-chain check prevents consumption of this stale value.
5. A downstream lending protocol using this adapter would accept the stale collateral price, enabling under-collateralized borrowing or incorrect liquidations.
