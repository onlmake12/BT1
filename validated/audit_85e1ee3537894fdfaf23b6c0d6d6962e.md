Audit Report

## Title
`HeaderAcceptor::accept()` Allows Re-insertion of `BLOCK_INVALID` Headers as `HEADER_VALID`, Corrupting Sync State - (File: `sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` noting that a `BLOCK_INVALID` header should trigger an early return, but no such guard exists. Because `BLOCK_INVALID` (`1 << 12 = 4096`) shares no bits with `HEADER_VALID` (`1`), the existing early-return branch is never taken for invalid-block headers. A header whose corresponding block failed contextual verification will pass all three remaining checks and be re-inserted via `insert_valid_header`, corrupting the node's `shared_best_header` and per-peer `best_known_header` sync state.

## Finding Description
In `accept()` at lines 301–304 of `sync/src/synchronizer/headers_process.rs`, the block status is fetched and checked only for `HEADER_VALID`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) { ... return result; }
```

`BlockStatus` is defined in `shared/src/block_status.rs`:
- `HEADER_VALID = 1`
- `BLOCK_INVALID = 1 << 12 = 4096`

These share no bits, so `status.contains(BlockStatus::HEADER_VALID)` is `false` when status is `BLOCK_INVALID`. Execution falls through to all three checks:

1. **`prev_block_check`** (line 244): checks whether the *parent* is `BLOCK_INVALID`. For a block that failed contextual verification (invalid script, double-spend), the parent is valid — passes.
2. **`non_contextual_check`** (line 255): runs `HeaderVerifier::verify()` — PoW, number, epoch, timestamp. A block with valid PoW but invalid transactions passes all of these.
3. **`version_check`** (line 286): checks `version == 0` — passes for any normal block.

All three pass, so `insert_valid_header` is called at line 356, which (per `sync/src/types/mod.rs` lines 1094–1141):
- Inserts the header into `header_map`
- Calls `may_set_best_known_header` for the sending peer
- Calls `may_set_shared_best_header`, potentially updating the global fork-choice view

`get_block_status` in `shared/src/shared.rs` (lines 425–445) confirms the status path: if the hash is in `block_status_map` (set by `chain/src/verify.rs` line 177 on contextual failure), it returns `BLOCK_INVALID` directly. If not in the map but in `header_map`, it returns `HEADER_VALID`. After `insert_valid_header` runs, the header is in `header_map` but `block_status_map` still holds `BLOCK_INVALID`, so subsequent `get_block_status` calls still return `BLOCK_INVALID` — but the sync state (`shared_best_header`, `best_known_header`) has already been corrupted.

The inconsistency is confirmed by contrast with `compact_block_process.rs` lines 259–260, which correctly guards:
```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
```
The headers path has no equivalent guard.

## Impact Explanation
The concrete impact is corruption of the node's sync state machine: `shared_best_header` may be updated to point to a header backed by a known-invalid block. The node's actual chain state is not altered (the block remains `BLOCK_INVALID` in the store), but the fork-choice view is corrupted. The `block_fetcher` uses `best_known_header` to determine what blocks to fetch; if `shared_best_header` is corrupted to a high-difficulty invalid header, the node may stall syncing from honest peers until a higher-difficulty honest header arrives to displace it, and may waste bandwidth issuing inflight requests for blocks it cannot process. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage/sync mechanism**, as the sync state is demonstrably corrupted by an unprivileged peer action.

## Likelihood Explanation
An attacker needs: (1) a block with valid Eaglesong PoW but failing contextual verification (invalid script, double-spend, invalid DAO data), and (2) the ability to send a `SendHeaders` P2P message — available to any connected peer. The PoW cost is a one-time investment; the header can be replayed to any number of target nodes indefinitely. The `FIXME` comment confirms the developers are aware of the gap, making this a known-unmitigated issue rather than a speculative one.

## Recommendation
Add an explicit `BLOCK_INVALID` guard immediately after the status fetch, before the `HEADER_VALID` branch, mirroring the pattern already used in `compact_block_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a new KnownInvalid variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing path
}
```

A dedicated `ValidationError::KnownInvalid` variant would be cleaner than reusing `InvalidParent`, but either resolves the vulnerability.

## Proof of Concept
1. Mine block `B` with valid Eaglesong PoW but containing an invalid transaction (e.g., a double-spend or a script that fails execution).
2. Relay `B` to the target node via normal block propagation. `B` passes `non_contextual_verify`, is stored (`BLOCK_STORED`), then `ConsumeUnverifiedBlockProcessor` fails contextual verification. `chain/src/verify.rs` line 177 calls `insert_block_status(B.hash, BLOCK_INVALID)`.
3. Connect to the target node as a P2P peer and send a `SendHeaders` message containing only `B`'s header.
4. `HeadersProcess::execute()` calls `accept_first` or the loop body for `B`'s header, invoking `HeaderAcceptor::accept()`.
5. `get_block_status(&B.hash())` returns `BLOCK_INVALID` (from `block_status_map`). The `HEADER_VALID` branch is not taken (no early return — the FIXME).
6. `prev_block_check`: `B`'s parent is valid → passes.
7. `non_contextual_check` via `HeaderVerifier::verify`: PoW valid, number/epoch/timestamp valid → passes.
8. `version_check`: version == 0 → passes.
9. `insert_valid_header` is called: `B`'s header enters `header_map`; `may_set_best_known_header` and `may_set_shared_best_header` are updated to `B`'s header index.
10. **Observable result**: `shared_best_header` now points to `B`'s header (an invalid block). The node's sync decisions for all peers are based on this corrupted view until an honest peer provides a higher-difficulty header.
