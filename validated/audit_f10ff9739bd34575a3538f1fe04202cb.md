Audit Report

## Title
Missing Per-Peer Rate Limiter in `BlockFilter` Enables Head-of-Line Blocking via `GetBlockFilterCheckPoints` Flood — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_check_points_process.rs`)

## Summary
`BlockFilter` contains no rate limiter, allowing any peer to flood `GetBlockFilterCheckPoints` messages. Each message triggers a synchronous loop of up to 2,000 iterations performing two RocksDB reads each (4,000 reads total) with no async yield points. Because `CKBHandler::received` fully awaits the handler before returning, a single flooding peer monopolizes the Filter protocol handler and starves all other light-client peers of responses.

## Finding Description

**Sequential dispatch confirmed:** `CKBHandler::received` in `network/src/protocols/mod.rs` calls `self.handler.received(Arc::new(nc), peer_index, data).await` and returns only after the future completes. No concurrent dispatch occurs. [1](#0-0) 

**No rate limiter in `BlockFilter`:** The struct contains only `shared: Arc<SyncShared>` — no `rate_limiter` field. `try_process` dispatches directly to `GetBlockFilterCheckPointsProcess::execute` with zero quota check. [2](#0-1) [3](#0-2) 

**Per-message cost:** `GetBlockFilterCheckPointsProcess::execute` loops up to `BATCH_SIZE = 2000` iterations. Each iteration calls `active_chain.get_block_hash(block_number)` followed by `active_chain.get_block_filter_hash(&block_hash)` — both synchronous RocksDB reads. The loop body contains no `.await` point; the entire 4,000-read sequence runs without yielding to the async executor. [4](#0-3) [5](#0-4) 

**Contrast with protected handlers:** `Relayer` defines `rate_limiter: RateLimiter<(PeerIndex, u32)>` and enforces it at the top of `try_process`. `HolePunching` likewise carries `rate_limiter: RateLimiter<(PeerIndex, u32)>`. `BlockFilter` has neither. [6](#0-5) [7](#0-6) 

**Exploit path:**
1. Attacker opens one TCP connection on `SupportProtocols::Filter`.
2. Attacker sends `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop.
3. Each message occupies the handler for the full 4,000 synchronous RocksDB reads before `CKBHandler::received` returns.
4. Legitimate light-client peers sending `GetBlockFilterHashes` or `GetBlockFilters` are queued behind the flood and receive no responses until the flood stops.

## Impact Explanation
A single attacker with one TCP connection and negligible bandwidth can completely deny the Filter protocol service to all light clients on the targeted node. The Filter protocol is the sole mechanism by which light clients obtain compact block filter data for transaction monitoring. Complete starvation of this handler maps to **High: bad designs which could cause CKB network congestion with few costs** — the attacker's cost is one connection and small messages while the victim node's Filter handler is fully monopolized. [8](#0-7) 

## Likelihood Explanation
The attack is trivially reproducible: any peer connecting on the Filter protocol can send arbitrary `GetBlockFilterCheckPoints` messages. No authentication, special state, or hashpower is required. The absence of a rate limiter is a concrete, verifiable code gap directly observable in the struct definition. The attack is locally reproducible with two mock peers using the existing test infrastructure in `test/src/specs/sync/block_filter.rs`. [9](#0-8) 

## Recommendation
Add a per-peer, per-message-type `RateLimiter` to `BlockFilter`, mirroring the pattern in `Relayer`:

```rust
pub struct BlockFilter {
    shared: Arc<SyncShared>,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

Check the limiter at the top of `try_process` before dispatching to any `*Process::execute`. A quota of 5–10 `GetBlockFilterCheckPoints` requests per second per peer is sufficient for legitimate light-client use while preventing flooding. [10](#0-9) 

## Proof of Concept
1. Start a CKB node with the Filter protocol enabled and a synced chain of ≥4,000,000 blocks.
2. Connect mock peer A and mock peer B using `SupportProtocols::Filter` (as done in `test/src/specs/sync/block_filter.rs`).
3. Have peer A send `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop.
4. Have peer B send `GetBlockFilterHashes { start_number: 0 }` at a steady rate of 1 req/sec.
5. Measure peer B's response latency with and without peer A's flood.
6. Assert that peer B's latency increases proportionally to peer A's flood rate, confirming head-of-line blocking caused by the missing rate limiter in `BlockFilter::try_process`. [11](#0-10)

```rust
    pub async fn execute(self) -> Status {
        let active_chain = self.filter.shared.active_chain();
        let start_number: BlockNumber = self.message.to_entity().start_number().into();
        let latest: BlockNumber = active_chain.get_latest_built_filter_block_number();

        let mut block_filter_hashes = Vec::new();

        if latest >= start_number {
            let mut block_number = start_number;
            for _ in 0..BATCH_SIZE {
                if let Some(block_filter_hash) = active_chain
                    .get_block_hash(block_number)
                    .and_then(|block_hash| active_chain.get_block_filter_hash(&block_hash))
                {
                    block_filter_hashes.push(block_filter_hash);
                } else {
                    break;
                }
                let Some(next_block_number) = block_number.checked_add(CHECK_POINT_INTERVAL) else {
                    break;
                };
                block_number = next_block_number;
            }
            let content = packed::BlockFilterCheckPoints::new_builder()
                .start_number(start_number)
                .block_filter_hashes(block_filter_hashes)
                .build();

            let message = packed::BlockFilterMessage::new_builder()
                .set(content)
                .build();
            async_send_message_to(&self.nc, self.peer, &message).await
        } else {
            Status::ignored()
        }
    }
```
