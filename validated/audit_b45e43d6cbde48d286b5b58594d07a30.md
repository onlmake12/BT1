### Title
Admin/Owner Cannot Revoke a Compromised Fee Manager in Echo Contract - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

---

### Summary

In `Echo.sol`, `setFeeManager` is exclusively callable by the registered provider (`msg.sender`). No admin or owner override exists to revoke or replace a fee manager. If a provider's key is compromised, an attacker can install a malicious fee manager that drains all accrued provider fees via `withdrawAsFeeManager`, and neither the admin nor the owner can intervene.

---

### Finding Description

`Echo.setFeeManager` enforces that only the registered provider can set their own fee manager:

```solidity
function setFeeManager(address manager) external override {
    require(
        _state.providers[msg.sender].isRegistered,
        "Provider not registered"
    );
    address oldFeeManager = _state.providers[msg.sender].feeManager;
    _state.providers[msg.sender].feeManager = manager;
    emit FeeManagerUpdated(msg.sender, oldFeeManager, manager);
}
``` [1](#0-0) 

The fee manager holds two significant powers:

1. **Drain provider fees** — `withdrawAsFeeManager` transfers the provider's entire accrued ETH balance to the fee manager's address:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(msg.sender == _state.providers[provider].feeManager, "Only fee manager");
    ...
    _state.providers[provider].accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [2](#0-1) 

2. **Manipulate provider fees** — `setProviderFee` accepts calls from either the provider or the fee manager:

```solidity
require(
    msg.sender == provider ||
        msg.sender == _state.providers[provider].feeManager,
    "Only provider or fee manager can invoke this method"
);
``` [3](#0-2) 

The admin's governance functions (`withdrawFees`, `setDefaultProvider`, `setExclusivityPeriod`) provide no mechanism to revoke or override a provider's fee manager: [4](#0-3) [5](#0-4) 

The same structural gap exists in `Entropy.sol`: [6](#0-5) 

---

### Impact Explanation

**Impact: High**

If a provider's private key is compromised (leaked or seized), the attacker can:

1. Call `setFeeManager(attackerAddress)` to install a malicious fee manager.
2. Call `withdrawAsFeeManager(provider, fullBalance)` to drain all ETH accrued by that provider.
3. Call `setProviderFee(provider, 0, 0, 0)` to set fees to zero, disrupting the provider's economics.

The admin and owner have no function to revoke the malicious fee manager or freeze the provider. The only party who can fix this is the provider themselves — but their key is compromised, making self-remediation impossible.

---

### Likelihood Explanation

**Likelihood: Low**

This requires a provider's private key to be compromised (leaked or seized). This is an uncommon but realistic operational risk for any long-running on-chain service provider, consistent with the external report's stated scenario.

---

### Recommendation

Add an admin/owner emergency override function that can revoke or replace a provider's fee manager:

```solidity
function revokeProviderFeeManager(address provider) external {
    _authorizeAdminAction(); // only admin or owner
    address oldFeeManager = _state.providers[provider].feeManager;
    _state.providers[provider].feeManager = address(0);
    emit FeeManagerUpdated(provider, oldFeeManager, address(0));
}
```

This mirrors the pattern already used in `EntropyGovernance.sol` where both admin and owner can act on governance parameters: [7](#0-6) 

---

### Proof of Concept

1. Provider `P` registers via `registerProvider(...)` and accrues fees over time.
2. Provider `P`'s private key is leaked.
3. Attacker calls `echo.setFeeManager(attackerAddress)` from `P`'s compromised key.
4. Attacker calls `echo.withdrawAsFeeManager(P, fullBalance)` — all accrued ETH is transferred to `attackerAddress`.
5. Admin attempts to revoke the fee manager — **no such function exists**. The admin can only call `withdrawFees` (for Pyth protocol fees), `setDefaultProvider`, or `setExclusivityPeriod`. None of these affect the provider's fee manager.
6. Attacker additionally calls `echo.setProviderFee(P, 0, 0, 0)` to zero out the provider's fees, preventing future revenue.
