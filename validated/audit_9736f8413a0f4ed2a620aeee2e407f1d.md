### Title
Indefinite Outbound Connection Exhaustion via Feeler-to-Attempt Promotion Loop — (`network/src/services/outbound_peer.rs`, `network/src/peer_store/peer_store_impl.rs`)

---

### Summary

An unprivileged attacker operating listening nodes can cause a victim CKB node to repeatedly issue outbound `dial_identify` attempts to attacker-controlled addresses every 15 seconds, indefinitely. The root cause is a combination of: (1) `Feeler::connected` promoting an address into the attempt pool with `last_connected_at_ms=now`, (2) `fetch_addrs_to_attempt` lacking the `tried_in_last_minute` guard that `fetch_addrs_to_feeler` has, and (3) `IdentifyCallback::unregister` refreshing `last_connected_at_ms` on every outbound disconnect, keeping the address perpetually eligible. With enough attacker IPs (equal to the victim's `max_outbound`), all outbound slots are monopolized.

---

### Finding Description

**Step 1 — Feeler promotes address into attempt pool**

`Feeler::connected` calls `add_outbound_addr` with `last_connected_at_ms = unix_time_as_millis()`: [1](#0-0) 

`add_outbound_addr` stores the address with `last_connected_at_ms = now`: [2](#0-1) 

**Step 2 — `fetch_addrs_to_attempt` selects it after 15 seconds**

The filter in `fetch_addrs_to_attempt` requires `last_connected_at_ms` to be within 3 days and at least `DIAL_INTERVAL` (15s) ago. Critically, it has **no `tried_in_last_minute` guard**: [3](#0-2) 

Compare with `fetch_addrs_to_feeler`, which explicitly blocks re-attempt within 60 seconds: [4](#0-3) 

Constants confirming the values: [5](#0-4) 

**Step 3 — `mark_tried` is called, `dial_identify` issued**

`try_dial_peers` calls `mark_tried` (incrementing `attempts_count`, setting `last_tried_at_ms`) then issues `dial_identify`: [6](#0-5) 

**Step 4 — Attacker drops connection; `unregister` refreshes `last_connected_at_ms`**

When the attacker drops the identify session, `IdentifyCallback::unregister` fires and calls `update_outbound_addr_last_connected_ms`, setting `last_connected_at_ms = now` again: [7](#0-6) 

`update_outbound_addr_last_connected_ms` updates only `last_connected_at_ms`, **not** `attempts_count`: [8](#0-7) 

**Step 5 — `is_connectable` never returns false**

`fetch_random` in `AddrManager` calls `is_connectable`: [9](#0-8) 

`is_connectable` only returns false when `now - last_connected_at_ms > ADDR_TIMEOUT_MS (7 days) && attempts_count >= 10`. Because `update_outbound_addr_last_connected_ms` keeps refreshing `last_connected_at_ms` to `now` on every disconnect, `now - last_connected_at_ms` is always near zero, so this condition never triggers: [10](#0-9) 

**Step 6 — Cycle repeats every 15 seconds**

The `OutboundPeerService` ticks every `try_connect_interval` (15s by default), calling `try_dial_peers` each time: [11](#0-10)

### Impact Explanation

Each attacker IP consumes one outbound slot per 15-second tick (IP deduplication in `fetch_random` limits one address per IP per call). With `N` attacker IPs where `N = max_outbound`, all outbound connection attempts are directed at attacker nodes. The victim node never successfully connects to legitimate peers, isolating it from the honest network. The attacker's cost is minimal: accept TCP connections and immediately drop them.

---

### Likelihood Explanation

The attacker needs only to:
1. Operate `N` listening nodes (one per victim outbound slot, typically 8–10)
2. Get their addresses into the victim's peer store (via any gossip/addr relay path, or by being a bootnode neighbor)
3. Accept feeler connections and drop identify sessions

No PoW, no keys, no privileged access required. The cycle is self-sustaining once started.

---

### Recommendation

1. **Add `tried_in_last_minute` guard to `fetch_addrs_to_attempt`**, mirroring the guard already present in `fetch_addrs_to_feeler`. This prevents re-selecting the same address within 60 seconds regardless of `last_connected_at_ms`.
2. **Do not call `update_outbound_addr_last_connected_ms` on failed/short-lived identify sessions.** Only call it when the identify handshake completes successfully (i.e., after `received_identify` returns `Continue`). Currently it is called unconditionally in `unregister`, which fires even on immediate disconnects.
3. **Respect `attempts_count` in `fetch_addrs_to_attempt`**: add a cap on how many times an address can be retried within a short window, independent of `last_connected_at_ms`.

### Proof of Concept

```
T=0:    victim.dial_feeler(attacker_addr)
        → Feeler::connected → add_outbound_addr(last_connected_at_ms=T0)
        → feeler disconnects

T=15s:  OutboundPeerService tick
        → fetch_addrs_to_attempt: T0 > now-3days ✓, T0 <= now-15s ✓
        → mark_tried(attempts_count=1), dial_identify(attacker_addr)
        → attacker drops TCP → IdentifyCallback::unregister
        → update_outbound_addr_last_connected_ms → last_connected_at_ms=T1

T=30s:  OutboundPeerService tick
        → fetch_addrs_to_attempt: T1 > now-3days ✓, T1 <= now-15s ✓
        → mark_tried(attempts_count=2), dial_identify(attacker_addr)
        → attacker drops TCP → last_connected_at_ms=T2

... repeats every 15s indefinitely, attempts_count grows but
    is_connectable never returns false because last_connected_at_ms ≈ now
```

With `max_outbound=8` attacker IPs, all 8 outbound slots are consumed each tick, and the victim connects to zero legitimate peers.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
