### Title
Unbounded `while` Loop in `executeCallback()` Can Run Out of Gas, Permanently Blocking Request Fulfillment — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` contains an unbounded `while` loop that iterates over every sequence number between `_state.firstUnfulfilledSeq` and `_state.currentSequenceNumber`, performing cold storage reads on each iteration. An unprivileged attacker can create many requests and fulfill them out of order, causing the loop to exhaust the block gas limit when the oldest unfulfilled request is finally fulfilled, permanently preventing that request from being completed.

---

### Finding Description

After clearing a fulfilled request, `executeCallback()` advances the `firstUnfulfilledSeq` pointer with the following loop:

```solidity
// After successful callback, update firstUnfulfilledSeq if needed
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest()`, which performs at least one cold `SLOAD` (2,100 gas) against `_state.requests[shortKey]`, and potentially a second cold `SLOAD` against `_state.requestsOverflow[key]`:

```solidity
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];
    }
}
``` [2](#0-1) 

The developers themselves flagged this risk in a `TODO` comment immediately above the loop:

> "I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number." [3](#0-2) 

There is no cap on the total number of open requests. `requestPriceUpdatesWithCallback()` only limits `priceIds.length` per request, not the total number of requests:

```solidity
if (priceIds.length > MAX_PRICE_IDS) {
    revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
}
``` [4](#0-3) 

**Attack steps:**

1. Attacker (or colluding users) calls `requestPriceUpdatesWithCallback()` N+1 times, creating requests with sequence numbers 1 through N+1.
2. Attacker fulfills requests 2 through N+1 (skipping request 1). Each `executeCallback()` call runs the `while` loop but stops immediately at sequence 1 (still active), so gas cost is negligible.
3. `firstUnfulfilledSeq` remains pinned at 1.
4. When anyone attempts to fulfill request 1, the `while` loop starts at sequence 1 and scans all the way to N+1 (all now cleared), performing N+1 storage reads. At ~2,100 gas per cold SLOAD, N ≈ 14,000 requests exhausts a 30M gas block limit.
5. The entire `executeCallback()` transaction reverts (including `clearRequest()`), so request 1 is never cleared and can never be fulfilled.

---

### Impact Explanation

- **Request 1 is permanently unfulfillable.** Every attempt to call `executeCallback()` for sequence 1 will re-trigger the same unbounded loop and revert.
- **Consumer funds are locked.** The fee paid by the requester at sequence 1 is held in the contract and cannot be recovered because there is no refund path for a request that cannot be cleared.
- **Cascading DoS.** Because `firstUnfulfilledSeq` can never advance past sequence 1, any future logic that depends on `firstUnfulfilledSeq` (e.g., `getFirstActiveRequests`) is also degraded.

---

### Likelihood Explanation

The attack requires paying fees for N requests. However, the fee per request is set by the provider and can be low. On chains with high block gas limits (e.g., Arbitrum, BNB Chain), the required N is larger, but the attack remains economically feasible. The vulnerability is reachable by any unprivileged user who can call `requestPriceUpdatesWithCallback()` — no privileged role is required.

---

### Recommendation

Replace the unbounded `while` loop with a bounded scan (e.g., advance at most `MAX_SCAN_STEPS` per call), or use a doubly-linked list of active requests as the developers themselves suggested in the `TODO` comment. A simple fix is:

```solidity
uint256 MAX_SCAN = 100;
uint256 scanned = 0;
while (
    scanned < MAX_SCAN &&
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
    scanned++;
}
```

### Proof of Concept

```solidity
// 1. Create N+1 requests (attacker pays fees)
for (uint i = 0; i < N + 1; i++) {
    echo.requestPriceUpdatesWithCallback{value: fee}(
        provider, publishTime, priceIds, gasLimit
    );
}
// Sequence numbers: 1 .. N+1, firstUnfulfilledSeq = 1

// 2. Fulfill requests 2 .. N+1 (skip seq 1)
for (uint i = 2; i <= N + 1; i++) {
    echo.executeCallback(provider, i, updateData, priceIds);
    // while loop stops at seq 1 (still active) — cheap
}
// firstUnfulfilledSeq still = 1

// 3. Attempt to fulfill seq 1 — while loop scans 1..N+1, OOG revert
echo.executeCallback(provider, 1, updateData, priceIds); // REVERTS
```

The `while` loop at lines 169–174 of `Echo.sol` iterates N times, each performing at least one cold SLOAD (2,100 gas). For N = 14,000 on a 30M gas chain, the loop alone consumes ~29.4M gas, exhausting the block limit before the callback is ever executed. [5](#0-4)
