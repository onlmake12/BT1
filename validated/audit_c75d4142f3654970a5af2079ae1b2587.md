### Title
Raw Block Timestamp Used as Base for Relative `since` Timestamp Verification Instead of Median Time — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

After the CKB2021 hardfork, `SinceVerifier::verify_relative_lock()` uses the **raw block timestamp** of the cell's inclusion block as the base for relative timestamp `since` verification, instead of the **median time** over 37 blocks. The raw block timestamp is a single, miner-settable data point — directly analogous to Uniswap's `slot0` — while the median time is the time-averaged equivalent of TWAP. A miner can set the block timestamp anywhere in the window `(median_of_previous_37_blocks, now + ALLOWED_FUTURE_BLOCKTIME]`, shifting the base of any relative time lock on cells they include.

---

### Finding Description

In `verify_relative_lock` for `SinceMetric::Timestamp`, the post-CKB2021 branch reads:

```rust
let base_timestamp = if hardfork_switch
    .ckb2021
    .is_block_ts_as_relative_since_start_enabled(epoch_number)
{
    self.data_loader
        .get_header_fields(&info.block_hash)
        .expect("header exist")
        .timestamp          // ← single raw block timestamp; miner-controlled
} else {
    self.parent_median_time(&info.block_hash)   // ← median of 37 blocks; robust
};
let current_median_time = self.block_median_time(&parent_hash);
let required_timestamp = base_timestamp.checked_add(timestamp)...;
if current_median_time < required_timestamp {
    return Err((TransactionError::Immature { index }).into());
}
``` [1](#0-0) 

The `TimestampVerifier` enforces only two bounds on a block's timestamp:

- **Lower bound**: strictly greater than the median of the previous 37 blocks.
- **Upper bound**: `unix_time_as_millis() + ALLOWED_FUTURE_BLOCKTIME` (2 hours). [2](#0-1) 

This gives a miner a manipulation window of up to `ALLOWED_FUTURE_BLOCKTIME` (2 hours) above the current wall-clock time, and a downward window equal to the gap between the current time and the median of the previous 37 blocks.

The median time computation — used correctly for the *current* side of the check — averages 37 ancestor timestamps: [3](#0-2) 

Before CKB2021, `parent_median_time` was used for the *base* side as well, making both sides of the inequality equally manipulation-resistant. The hardfork replaced only the base side with the raw single-point timestamp.

---

### Impact Explanation

The relative `since` timestamp check is: `current_median_time ≥ base_timestamp + relative_lock`.

**Early-expiry attack**: A miner sets the block timestamp to just above the median of the previous 37 blocks when including a target cell. This depresses `base_timestamp`, so `base_timestamp + relative_lock` is reached sooner. The relative time lock expires earlier than the wall-clock duration the cell owner intended.

**Late-expiry attack**: A miner sets the block timestamp up to 2 hours in the future. This inflates `base_timestamp`, delaying when `current_median_time` satisfies the condition. A relative time lock intended to expire in, say, 1 hour can be stretched to ~3 hours.

Any protocol that relies on relative timestamp `since` for security guarantees — payment channels, escrow contracts, atomic swaps, or any script using `SinceMetric::Timestamp` in relative mode — is affected. The miner does not need majority hashpower; a single block inclusion suffices.

---

### Likelihood Explanation

The attacker role is **miner/block-template caller**, explicitly within the CKB bounty scope. No special privilege beyond producing a valid block is required. The manipulation window for the late-expiry direction is a hard 2-hour ceiling (`ALLOWED_FUTURE_BLOCKTIME`), which is large enough to be practically meaningful. The early-expiry window depends on how far the median lags real time (bounded but non-zero in all conditions). The attack is targeted: the miner must include the specific cell whose relative lock they wish to manipulate, which is feasible for a miner colluding with one party in a two-party protocol.

---

### Recommendation

Use the **median time of the cell's inclusion block** (i.e., `self.block_median_time(&info.block_hash)`) as `base_timestamp` instead of the raw `header_fields.timestamp`. This mirrors the treatment of the current-side value (`current_median_time`) and restores the pre-CKB2021 manipulation resistance on both sides of the inequality, while still satisfying the RFC intent of anchoring relative locks to the block that included the cell.

```rust
// Recommended: use median time on both sides
let base_timestamp = self.block_median_time(&info.block_hash);
let current_median_time = self.block_median_time(&parent_hash);
```

### Proof of Concept

1. Alice creates a cell protected by a relative timestamp `since` of 3,600,000 ms (1 hour).
2. Miner M (colluding with Bob, who wants early access) includes Alice's cell in block B. M sets `B.timestamp = median_of_B's_37_ancestors + 1` — the minimum valid value, which may be, e.g., 90 seconds below the actual wall-clock time.
3. `base_timestamp` is set to this artificially low value.
4. The lock condition becomes: `current_median_time ≥ (median_of_B's_37_ancestors + 1) + 3_600_000`.
5. Because `base_timestamp` is ~90 seconds below the real inclusion time, the lock is satisfiable ~90 seconds earlier than Alice intended.
6. For the late-expiry direction: M sets `B.timestamp = unix_time_as_millis() + 7_200_000` (2 hours ahead). Alice's 1-hour relative lock now requires `current_median_time ≥ real_inclusion_time + 2h + 1h = 3h` of real elapsed time, tripling the intended lock duration and potentially locking Alice's funds for an unintended period.     pub fn verify(&self) -> Result<(), Error> {
        // skip genesis block
        if self.header.is_genesis() {
            return Ok(());
        }

        let min = self.data_loader.block_median_time(
            &self.header.data().raw().parent_hash(),
            self.median_block_count,
        );
        if self.header.timestamp() <= min {
            return Err(TimestampError::BlockTimeTooOld {
                min,
                actual: self.header.timestamp(),
            }
            .into());
        }
        let max = self.now + ALLOWED_FUTURE_BLOCKTIME;
        if self.header.timestamp() > max {
            return Err(TimestampError::BlockTimeTooNew {
                max,
                actual: self.header.timestamp(),
            }
            .into());
        }
        Ok(())
    }

    fn block_median_time(&self, block_hash: &Byte32, median_block_count: usize) -> u64 {
        let mut timestamps: Vec<u64> = Vec::with_capacity(median_block_count);
        let mut block_hash = block_hash.clone();
        for _ in 0..median_block_count {
            let header_fields = self
                .get_header_fields(&block_hash)
                .expect("parent header exist");
            timestamps.push(header_fields.timestamp);
            block_hash = header_fields.parent_hash;

            if header_fields.number == 0 {
                break;
            }
        }

        // return greater one if count is even.
        timestamps.sort_unstable();
        timestamps[timestamps.len() >> 1]
    }
