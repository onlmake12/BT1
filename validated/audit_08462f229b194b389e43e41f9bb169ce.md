### Title
`CyclesEstimator` Treats Historically-Existing Dead Cells as Live, Enabling Free Repeated Script Execution via `estimate_cycles` RPC — (`rpc/src/module/chain.rs`)

---

### Summary

The `CyclesEstimator` struct used by the `estimate_cycles` RPC resolves input cells by calling `snapshot.get_cell()`, which returns cell metadata for **any cell that has ever existed** in the chain — including already-spent (dead) cells. The result is unconditionally wrapped in `CellStatus::live_cell(...)`. This is the direct analog of the external report's `require(subDir[author][ref] != 0)` pattern: it checks historical existence rather than current active status. Any unprivileged RPC caller can submit a transaction referencing dead cells, have it fully resolved and script-verified, and receive a cycle count — bypassing the liveness check that every other code path enforces.

---

### Finding Description

In `rpc/src/module/chain.rs`, the `CyclesEstimator` implements `CellProvider` as follows:

```rust
impl<'a> CellProvider for CyclesEstimator<'a> {
    fn cell(&self, out_point: &packed::OutPoint, eager_load: bool) -> CellStatus {
        let snapshot = self.shared.snapshot();
        snapshot
            .get_cell(out_point)
            .map(|mut cell_meta| {
                // ...
                CellStatus::live_cell(cell_meta)
            })  // treat as live cell, regardless of live or dead
            .unwrap_or(CellStatus::Unknown)
    }
}
``` [1](#0-0) 

The comment is self-documenting: `// treat as live cell, regardless of live or dead`. The function `snapshot.get_cell()` returns `Option<CellMeta>` for any cell that exists in the database — live **or** dead. The `CellStatus::live_cell(cell_meta)` wrapping then presents a spent cell to `resolve_transaction` as if it were unspent.

This contrasts with every other `CellProvider` implementation in the codebase. For example, `PoolCell` correctly returns `CellStatus::Dead` for spent cells: [2](#0-1) 

And `resolve_transaction` itself rejects dead cells with `OutPointError::Dead`: [3](#0-2) 

The `CyclesEstimator::run()` then passes the successfully-resolved transaction (containing dead-cell metadata) directly into `ScriptVerifier::verify(max_cycles)`: [4](#0-3) 

---

### Impact Explanation

**Misleading cycle estimates**: Applications that call `estimate_cycles` to pre-validate a transaction before submission will receive a successful cycle count for a transaction that would be immediately rejected by the tx-pool with `OutPointError::Dead`. This breaks the contract that `estimate_cycles` approximates what `send_transaction` would do.

**Free repeated script execution (CPU DoS)**: An attacker can:
1. Deploy a cell whose lock/type script consumes close to `max_block_cycles` cycles.
2. Spend that cell on-chain (making it dead).
3. Repeatedly call `estimate_cycles` with a transaction referencing the dead cell.

Each call triggers a full `ScriptVerifier` run up to `max_block_cycles` at zero cost to the attacker (no fee, no on-chain action). The `estimate_cycles` RPC has no rate limiting. This allows sustained CPU exhaustion on any node that exposes this RPC endpoint.

---

### Likelihood Explanation

- The `estimate_cycles` RPC is reachable by any unprivileged RPC caller.
- Dead cells are abundant on any live chain; an attacker can also manufacture them cheaply.
- No authentication, rate limiting, or fee is required to call `estimate_cycles`.
- The root cause is a single-line design choice (`get_cell` instead of `cell`) that is explicitly acknowledged in a comment, making it stable and unlikely to be accidentally fixed.

---

### Recommendation

Replace `snapshot.get_cell(out_point)` with `snapshot.cell(out_point, eager_load)` in `CyclesEstimator::cell()`. The `cell()` method returns `CellStatus::Dead` for spent cells, which causes `resolve_transaction` to return `OutPointError::Dead` and the RPC to return an error — consistent with what `send_transaction` would do.

```rust
impl<'a> CellProvider for CyclesEstimator<'a> {
    fn cell(&self, out_point: &packed::OutPoint, eager_load: bool) -> CellStatus {
        let snapshot = self.shared.snapshot();
        // Use snapshot.cell() to respect liveness, not get_cell() which
        // returns metadata for dead cells too.
        snapshot.cell(out_point, eager_load)
    }
}
```

---

### Proof of Concept

1. On a running CKB node with `estimate_cycles` exposed, commit a transaction `tx_A` that creates an output with an expensive script (e.g., a loop consuming ~`max_block_cycles` cycles).
2. Commit a second transaction `tx_B` that spends `tx_A`'s output, making it dead.
3. Construct `tx_C` that references `tx_A`'s output as an input (the now-dead cell).
4. Call `estimate_cycles` with `tx_C` in a loop:

```bash
while true; do
  ckb-cli rpc estimate_cycles --transaction <tx_C_hex>
done
```

Expected (correct) behavior: `TransactionFailedToResolve: OutPoint(Dead(...))`.
Actual behavior: Returns a cycle count after running the expensive script, consuming full CPU per call.

    pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();
                let tx_env = TxVerifyEnv::new_submit(tip_header);
                match ScriptVerifier::new(
                    Arc::new(resolved),
                    snapshot.as_data_loader(),
                    consensus,
                    Arc::new(tx_env),
                )
                .verify(max_cycles)
                {
                    Ok(cycles) => Ok(EstimateCycles {
                        cycles: cycles.into(),
                    }),
                    Err(err) => Err(RPCError::custom_with_error(
                        RPCError::TransactionFailedToVerify,
                        err,
                    )),
                }
            }
            Err(err) => Err(RPCError::custom_with_error(
                RPCError::TransactionFailedToResolve,
                err,
            )),
        }
    }
