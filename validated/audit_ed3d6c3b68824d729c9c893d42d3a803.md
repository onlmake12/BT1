### Title
Wrong `ancestors_count` Decrement in Cell-Ref Eviction Loop Causes Over-Eviction of Pool Transactions - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `check_and_record_ancestors`, when the ancestor count of a new transaction exceeds `max_ancestors_count` and cell-dep-linked ("cell ref") parents are evicted to make room, the loop decrements `ancestors_count` by exactly 1 per call to `remove_entry_and_descendants`. However, `remove_entry_and_descendants` removes the target entry **and all its descendants**, some of which may themselves be ancestors of the incoming transaction. The tracked `ancestors_count` therefore remains artificially inflated after each eviction step, causing the loop to continue evicting more transactions than are actually necessary to satisfy the limit.

---

### Finding Description

In `tx-pool/src/component/pool_map.rs`, `check_and_record_ancestors` computes the ancestor set of a new transaction and checks it against `max_ancestors_count`:

```rust
let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);
let mut ancestors_count = ancestors.len() + 1;
``` [1](#0-0) 

When `ancestors_count` exceeds the limit but can be reduced by evicting cell-dep-linked parents, the following loop runs:

```rust
while ancestors_count > self.max_ancestors_count {
    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id);
        ancestors_count = ancestors_count.saturating_sub(1);   // ← always -1
        parents.remove(next_id);
        evicted.extend(removed);
    } else {
        break;
    }
}
``` [2](#0-1) 

`remove_entry_and_descendants(next_id)` removes `next_id` **plus every descendant** of `next_id` from the pool. If any of those descendants are also ancestors of the incoming transaction (i.e., they appear in the `ancestors` set), the actual remaining ancestor count drops by more than 1. But `ancestors_count` is only decremented by 1 regardless of how many ancestor-set members were removed. The variable therefore stays inflated, the `while` condition remains true longer than it should, and the loop evicts additional cell-ref parents that were not needed to satisfy the limit.

The `get_tx_ancenstors` function shows that `cell_ref_parents` are parents reachable via cell-dep edges, while `ancestors` is the full transitive closure over both input-spending and cell-dep edges:

```rust
for input in entry.inputs() {
    let input_pt = input.previous_output();
    if let Some(deps) = self.edges.deps.get(&input_pt) {
        cell_ref_parents.extend(deps.iter().cloned());
        parents.extend(deps.iter().cloned());
    }
    let id = ProposalShortId::from_tx_hash(&input_pt.tx_hash());
    if self.links.inner.contains_key(&id) {
        parents.insert(id);
    }
}
``` [3](#0-2) 

A cell-ref parent can have descendants that are also input-spending parents of the new transaction. Removing the cell-ref parent cascades to those descendants, but `ancestors_count` only decrements by 1.

The final `assert` at line 636 checks the **recalculated** `ancestors.len()` (not `ancestors_count`), so it does not catch the over-eviction:

```rust
assert!(ancestors.len() < self.max_ancestors_count);
``` [4](#0-3)

---

### Impact Explanation

An unprivileged tx-pool submitter can craft a transaction whose dependency graph triggers the eviction loop. Because `ancestors_count` is not decremented by the true number of ancestor-set members removed per iteration, the loop runs extra iterations and evicts more legitimate pending transactions than the limit requires. This silently purges valid, fee-paying transactions from the pool without cause, degrading pool liveness and enabling a targeted DoS against specific transactions already in the pool.

---

### Likelihood Explanation

Any tx-pool submitter (no privilege required) can reach `check_and_record_ancestors` via `send_transaction` / `test_tx_pool_accept` RPC. The triggering condition — a new transaction whose ancestor count exceeds `max_ancestors_count` but is reducible by evicting cell-ref parents — is explicitly supported and tested by the codebase (see `TxPoolLimitAncestorCount` integration test). Constructing the required dependency graph requires only normal transaction submission capability.

---

### Recommendation

Replace the fixed `-1` decrement with the count of removed entries that actually belong to the current `ancestors` set:

```rust
let removed = self.remove_entry_and_descendants(next_id);
let removed_ancestor_count = removed
    .iter()
    .filter(|e| ancestors.contains(&e.proposal_short_id()))
    .count();
ancestors_count = ancestors_count
    .saturating_sub(removed_ancestor_count.max(1));
```

Alternatively, after the eviction loop, recompute `ancestors_count` from the recalculated `ancestors` set (as is already done for `ancestors` itself at lines 631–633) rather than relying on the incrementally tracked variable.

---

### Proof of Concept

1. Submit tx_A to the pool (it will be a cell-dep provider).
2. Submit tx_B spending an output of tx_A (tx_B is an input-spending child of tx_A, and thus an ancestor of the final tx).
3. Submit 999 transactions each referencing tx_A as a cell-dep (`cell_ref_parents`). Pool now has 1001 ancestors for the next tx.
4. Submit a new tx spending tx_B's output and referencing tx_A as a cell-dep. `ancestors_count` = 1002 (> 1000). The eviction branch fires.
5. The loop calls `remove_entry_and_descendants` on the first cell-ref parent. That call also removes tx_B (a descendant of tx_A that is an input-spending ancestor of the new tx). `ancestors_count` decrements by 1 (to 1001), not by 2. The loop runs one extra iteration and evicts a second cell-ref parent unnecessarily.
6. Observe that two cell-ref parents are evicted even though evicting one (plus its cascade of tx_B) was sufficient to bring the ancestor count to 1000. [6](#0-5)
