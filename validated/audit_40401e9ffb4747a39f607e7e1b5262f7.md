### Title
Silent u128→u64 Truncation in DAO Maximum Withdraw Calculation Corrupts DAO State Field - (File: util/dao/src/lib.rs)

### Summary
`calculate_maximum_withdraw` in `util/dao/src/lib.rs` casts a `u128` intermediate result to `u64` using the Rust `as u64` operator (line 156), which silently truncates the high bits on overflow. Every other analogous `u128`→`u64` narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, making this the sole unguarded cast. When the truncation fires, the computed maximum-withdraw capacity is wrong, which propagates into the per-block DAO field (`current_s`) written into every subsequent block header, permanently corrupting the on-chain NervosDAO accounting state.

### Finding Description

`calculate_maximum_withdraw` computes the interest-bearing withdrawal capacity as:

```rust
// util/dao/src/lib.rs  lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a **truncating** cast in Rust: if `withdraw_counted_capacity > u64::MAX`, the upper 64 bits are silently discarded and the function returns a wrong (smaller, or arbitrarily wrapped) value with no error.

Every other `u128`→`u64` narrowing in the same file is guarded:

If `withdraw_counted_capacity` overflows `u64`, the truncated value is smaller than the true value, so `withdrawed_interests` is under-counted, `current_s` is inflated, and the DAO savings field written into the block header is permanently wrong.

### Impact Explanation

**Consensus-level DAO state corruption.** The DAO field (`C`, `S`, `U`, `AR`) is embedded in every block header and is the authoritative source for all future NervosDAO interest calculations. An inflated `S` (savings) field means:

1. Future depositors can withdraw more interest than was actually accrued — effectively minting CKB out of thin air.
2. The `current_s.safe_sub(withdrawed_interests)` path in subsequent blocks may underflow and return `DaoError::Overflow`, halting block production for any block that includes a DAO withdrawal.

The `transaction_fee` path also uses the wrong value, so fee accounting for DAO withdrawal transactions is incorrect.

### Likelihood Explanation

**Overflow condition:** `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`

- `counted_capacity` is bounded by the total CKB supply (~3.36 × 10¹⁸ shannons, well below `u64::MAX ≈ 1.84 × 10¹⁹`).
- Initial `ar₀ = 10¹⁶` (confirmed in `test/src/specs/dao/dao_verifier.rs` line 65).
- Overflow fires when `withdrawing_ar / deposit_ar > u64::MAX / counted_capacity ≈ 1.84×10¹⁹ / 3.36×10¹⁸ ≈ 5.5`. [7](#0-6) 

So `ar` must grow to ~5.5× its initial value. The `ar` accumulation rate is driven by secondary issuance (~1.344 billion CKB/year) relative to total capacity. Over a multi-decade horizon this threshold is reachable. A single large depositor holding close to the total supply maximises the per-deposit `counted_capacity` and minimises the required `ar` ratio. The condition is not immediately exploitable but is a deterministic future failure embedded in the consensus code.

### Recommendation

Replace the silent cast on line 156 with the same checked conversion used everywhere else in the file:

```rust
// util/dao/src/lib.rs  line 155-156  (fix)
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
```

This is consistent with lines 204, 244–245, and 258 in the same file and ensures that an overflow is surfaced as a `DaoError::Overflow` rather than silently corrupting the result.

### Proof of Concept

Construct a scenario where `withdraw_counted_capacity` exceeds `u64::MAX`:

```
counted_capacity  = 3_360_000_000_000_000_000  (3.36 × 10¹⁸ shannons, ~max realistic deposit)
deposit_ar        = 10_000_000_000_000_000      (10¹⁶, genesis value)
withdrawing_ar    = 55_000_000_000_000_000      (5.5 × 10¹⁶, ~5.5× growth)

withdraw_counted_capacity (u128)
  = 3_360_000_000_000_000_000 × 55_000_000_000_000_000
    / 10_000_000_000_000_000
  = 18_480_000_000_000_000_000   (> u64::MAX = 18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 18_480_000_000_000_000_000 mod 2^64
  = 33_255_926_290_448_385       ← wrong, ~1.8% of the true value
```

With the current code, `withdraw_capacity` is computed as ~33 quadrillion shannons instead of ~18.48 quintillion shannons. `withdrawed_interests` is therefore massively under-counted, `current_s` is inflated by the difference, and the DAO field written into the block header is permanently corrupted from that block onward.     /// Calculates the new dao field with specified [`EpochExt`].
    pub fn dao_field_with_current_epoch(
        &self,
        rtxs: impl Iterator<Item = &'a ResolvedTransaction> + Clone,
        parent: &HeaderView,
        current_block_epoch: &EpochExt,
    ) -> Result<Byte32, DaoError> {
        // Freed occupied capacities from consumed inputs
        let freed_occupied_capacities =
            rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
                self.input_occupied_capacities(rtx)
                    .and_then(|c| capacities.safe_add(c))
            })?;
        let added_occupied_capacities = self.added_occupied_capacities(rtxs.clone())?;
        let withdrawed_interests = self.withdrawed_interests(rtxs)?;

        let (parent_ar, parent_c, parent_s, parent_u) = extract_dao_data(parent.dao());

        // g contains both primary issuance and secondary issuance,
        // g2 is the secondary issuance for the block, which consists of
        // issuance for the miner, NervosDAO and treasury.
        // When calculating issuance in NervosDAO, we use the real
        // issuance for each block(which will only be issued on chain
        // after the finalization delay), not the capacities generated
        // in the cellbase of current block.
        let current_block_number = parent.number() + 1;
        let current_g2 = current_block_epoch.secondary_block_issuance(
            current_block_number,
            self.consensus.secondary_epoch_reward(),
        )?;
        let current_g = current_block_epoch
            .block_reward(current_block_number)
            .and_then(|c| c.safe_add(current_g2))?;

        let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
            / u128::from(parent_c.as_u64());
        let miner_issuance =
            Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
        let nervosdao_issuance = current_g2.safe_sub(miner_issuance)?;

        let current_c = parent_c.safe_add(current_g)?;
        let current_u = parent_u
            .safe_add(added_occupied_capacities)
            .and_then(|u| u.safe_sub(freed_occupied_capacities))?;
        let current_s = parent_s
            .safe_add(nervosdao_issuance)
            .and_then(|s| s.safe_sub(withdrawed_interests))?;

        let ar_increase128 =
            u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
        let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
        let current_ar = parent_ar
            .checked_add(ar_increase)
            .ok_or(DaoError::Overflow)?;

        Ok(pack_dao_data(current_ar, current_c, current_s, current_u))
    }
