### Title
Reverting Consumer Callback in Legacy `revealWithCallback` Path Permanently Locks Consumer Fees and Occupies Request Slots — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In the legacy code path of `revealWithCallback` (triggered when `req.gasLimit10k == 0`), the consumer's `_entropyCallback` is invoked directly without any error handling. If the callback reverts, the entire transaction reverts — including the preceding `clearRequest` call — permanently preventing the request from being fulfilled. Because the consumer's fee is already accrued to the provider at request time, the consumer loses their fee with no recourse, and the request slot is permanently occupied.

---

### Finding Description

`Entropy.sol`'s `revealWithCallback` function contains two distinct execution paths:

**New path** (`req.gasLimit10k != 0`): Uses `excessivelySafeCall` to invoke the consumer callback, catching reverts and emitting a `CallbackFailed` event. The request transitions to `CALLBACK_FAILED` state and can be retried.

**Legacy path** (`req.gasLimit10k == 0`): Calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly with no try/catch. Critically, `clearRequest` is called *before* the callback:

```solidity
clearRequest(provider, sequenceNumber);
// ...
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    );
}
```

If `_entropyCallback` reverts, the EVM unwinds the entire transaction, including the `clearRequest` call. The request is never cleared and remains permanently active. Since `requestHelper` already accrued the consumer's fee to the provider at request time:

```solidity
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

the consumer's ETH is permanently locked with no refund path.

The legacy path is reachable for any provider that has not called `setDefaultGasLimit`. When `providerInfo.defaultGasLimit == 0`, `requestHelper` sets `req.gasLimit10k = 0`, routing all fulfillments for that provider through the unprotected legacy path.

---

### Impact Explanation

- **Consumer fee loss**: The consumer pays the fee at request time. If their callback always reverts (buggy or malicious contract), the provider can never fulfill the request, and the consumer's ETH is permanently locked in the contract.
- **Request slot exhaustion (DoS)**: An attacker can deploy multiple consumer contracts that revert in `_entropyCallback`, request randomness from a legacy provider (one without `defaultGasLimit` set), and fill all `NUM_REQUESTS` slots with permanently unfulfillable requests. This prevents any new legitimate requests from being made to that provider.
- **Provider griefing**: The provider's `revealWithCallback` calls always revert for these requests, wasting gas and blocking slot reuse.

---

### Likelihood Explanation

- Any provider that registered without calling `setDefaultGasLimit` uses the legacy path. This includes providers registered before the new callback failure flow was introduced.
- Any unprivileged user can deploy a consumer contract with a reverting `_entropyCallback` and submit a request.
- No privileged access, leaked keys, or external oracle manipulation is required.
- The attack is cheap: the attacker only pays the request fee per slot consumed.

---

### Recommendation

1. Wrap the direct callback in the legacy path with a try/catch block, mirroring the new path's behavior:
   ```solidity
   try IEntropyConsumer(callAddress)._entropyCallback(sequenceNumber, provider, randomNumber) {
       // success
   } catch {
       // emit failure event, mark request as failed
   }
   ```
2. Alternatively, deprecate the legacy path entirely and require all providers to set a non-zero `defaultGasLimit` before accepting new requests.
3. Consider adding a refund mechanism for consumers whose requests can never be fulfilled.

---

### Proof of Concept

1. Provider registers via `register(...)` without ever calling `setDefaultGasLimit`. `providerInfo.defaultGasLimit == 0`.
2. Attacker deploys `MaliciousConsumer` whose `_entropyCallback` always reverts with `revert("griefed")`.
3. `MaliciousConsumer` calls `requestV2{value: fee}(provider, userContribution, 0)`. In `requestHelper`, because `providerInfo.defaultGasLimit == 0`, `req.gasLimit10k` is set to `0`. Fee is accrued: `providerInfo.accruedFeesInWei += providerFee`.
4. Provider calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`.
5. Since `req.gasLimit10k == 0`, the legacy branch executes. `clearRequest` is called, then `_entropyCallback` is called on `MaliciousConsumer`, which reverts.
6. The entire transaction reverts. `clearRequest` is undone. The request remains active.
7. Every subsequent `revealWithCallback` attempt by the provider reverts identically.
8. The consumer's fee is permanently locked in the contract. The request slot is permanently occupied.
