### Title
`ScriptHashTypeVerifier` Omits Type Script `hash_type` Check, Allowing Future-Version Type Scripts On-Chain Before Hardfork Activation — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates only `output.lock().hash_type()` against `ENABLED_SCRIPT_HASH_TYPE`. It never inspects `output.type_().hash_type()`. An unprivileged submitter can craft a transaction whose output carries a structurally valid but not-yet-consensus-activated type script hash type (e.g., `Data3 = 6`), submit it via RPC or P2P relay, have it admitted to the tx-pool and committed on-chain, and later trigger a consensus split when that cell is spent on nodes with different hardfork activation states.

---

### Finding Description

`ScriptHashTypeVerifier::verify()` is defined as:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...ScriptHashTypeNotPermitted...);
            }
        } else {
            return Err(...InvalidScriptHashType...);
        }
    }
    Ok(())
}
``` [1](#0-0) 

It only reads `output.lock().hash_type()`. The `output.type_()` field is never touched. The verifier's own doc comment confirms the oversight: "Check whether output **lock** hash type within enabled range." [2](#0-1) 

`ENABLED_SCRIPT_HASH_TYPE` is `{0, 1, 2, 4}` — the four currently activated hash types. [3](#0-2) 

The `ScriptHashType` enum includes future variants `Data3 = 6`, `Data4 = 8`, … via the `seq!` macro, and these are also present in the JSON RPC type, making them serializable/deserializable. [4](#0-3) 

A separate structural check, `check_data` / `verify_value`, exists in the P2P relay path and checks both lock and type scripts — but it only validates structural form (`v % 2 == 0 || v == 1`), not consensus activation. So `Data3 = 6` passes `check_data` (6 is even) and also passes `ScriptHashTypeVerifier` (type script is never checked). [5](#0-4) [6](#0-5) 

`NonContextualTransactionVerifier` — the gate used by the tx-pool for both RPC and P2P submissions — calls `ScriptHashTypeVerifier` as its only hash-type guard. [7](#0-6) 

The RPC `send_transaction` path converts JSON → packed → `TransactionView` and calls `submit_local_tx`, which runs `non_contextual_verify` → `NonContextualTransactionVerifier`. No `check_data` call exists in this path. [8](#0-7) 

---

### Impact Explanation

A cell output with `type_.hash_type = Data3 (6)` passes all admission checks and is committed on-chain. When that cell is later **spent**, `select_version` and `extract_script_and_dep_index` in the script engine evaluate the type script's hash type: [9](#0-8) [10](#0-9) 

On a node where the hardfork activating `Data3` has not yet occurred, this returns `InvalidScriptHashType` and the spending transaction is rejected. On a node where the hardfork has activated, it succeeds. The two nodes reach different conclusions about the same block → **consensus split / chain fork**.

---

### Likelihood Explanation

The attack requires only:
1. Constructing a transaction with a type script using `hash_type = 6` (a valid JSON enum value `"data3"`).
2. Submitting it via `send_transaction` RPC with `outputs_validator = "passthrough"`.
3. Having a miner include it in a block.

No privileged access, no key material, no majority hashpower is needed. The RPC is publicly accessible on any node that exposes it. The specific example values `3` and `5` from the question are structurally invalid (odd, not 1) and would be caught by `check_data` in the P2P path and by JSON deserialization in the RPC path. The correct exploit values are even numbers not in `{0, 2, 4}` — i.e., `6, 8, 10, …` — which are valid `ScriptHashType` variants.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also check `output.type_()` when present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // check lock
        check_hash_type(output.lock().hash_type())?;
        // check type_ if present
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the existing `TryInto::<ScriptHashType>` + `ENABLED_SCRIPT_HASH_TYPE.contains` logic.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
