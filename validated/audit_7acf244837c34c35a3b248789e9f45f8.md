## Title

## Summary
The `GET_LAST_STATE_PROOF_LIMIT` guard bounds only the *count* of difficulty samples, not the computational cost of resolving them. Each of up to 999 difficulty entries independently triggers a full binary search over the entire block range, each step issuing two DB reads. An unprivileged peer can craft a valid `GetLastStateProof` message that causes approximately 46,000 DB reads per request, with no rate limiting and no ban triggered for valid requests.

## Impact Explanation
A single attacker peer can continuously send crafted `GetLastStateProof` requests, each causing ~46,000 synchronous RocksDB reads, saturating the node's I/O path and degrading performance for all other operations (block sync, tx relay, RPC). This matches the allowed CKB bounty impact: **High ŌĆö Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001ŌĆō15000 points). The attacker sustains the attack indefinitely at zero cost beyond maintaining a P2P connection, with no ban or backpressure mechanism to stop them.

## Likelihood Explanation
The attacker requires only a valid P2P connection to a node running the light client server. The total difficulty values for the first ~999 blocks are trivially obtained by syncing the chain or querying a public explorer. No special privileges, keys, or hashpower are required. The crafted message is indistinguishable from a legitimate request and returns `OK`, so the attacker is never banned and can repeat the attack at maximum throughput.

## Recommendation
Replace the flat count limit with a cost-aware limit. The most targeted fix is to cap the total estimated binary search steps before processing: compute `difficulties.len() ├Ś ŌīłlogŌéé(end_block_number - start_block_number)Ōīē` and reject requests exceeding a threshold. Alternatively, replace the per-difficulty binary search with a single linear scan over the block range that emits block numbers as each difficulty threshold is crossed ŌĆö O(H) total but with a single pass and no repeated range traversal. A complementary defense is per-peer request rate limiting (e.g., N `GetLastStateProof` requests per peer per time window) enforced before any DB access.
