### Title
Unbounded Linear Storage Scan in `executeCallback` Causes OOG DoS, Locking User Fees — (`Echo.sol`)

---

### Summary

`Echo.sol::executeCallback` contains an unbounded `while` loop that linearly scans through all sequence numbers from `_state.firstUnfulfilledSeq` to `_state.currentSequenceNumber`, performing a cold storage lookup per iteration. When a sufficient number of requests have been created and fulfilled while an earlier request remains pending, fulfilling that earlier request triggers a scan over the entire gap, causing an out-of-gas (OOG) revert. The user's fee is permanently locked in the contract.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` advances `_state.firstUnfulfilledSeq` with this loop: [1](#0-0) 

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

Each iteration calls `findRequest`, which performs **two storage reads** per sequence number: [2](#0-1) 

1. `_state.requests[shortKey]` — a fixed-size array slot (warm after first access, 100 gas).
2. `_state.requestsOverflow[key]` — a mapping keyed by `keccak256(sequenceNumber)`, meaning **every sequence number maps to a unique, cold storage slot** (2,100 gas each).

`_state.currentSequenceNumber` is a `uint64` and increments with every call to `requestPriceUpdatesWithCallback`. There is no cap on how large the gap between `firstUnfulfilledSeq` and `currentSequenceNumber` can grow. [3](#0-2) 

---

### Impact Explanation

When `executeCallback` OOGs, the entire transaction reverts. The request is never cleared, and the fee stored in `req.fee` remains locked in the contract with no withdrawal path for the user. The provider also cannot earn their fee. This is a permanent, irrecoverable loss of funds for the requester.

---

### Likelihood Explanation

The scenario requires:

1. Request seq=1 is created but not fulfilled for an extended period (e.g., provider ignores it during the exclusivity window, or the request is for a future `publishTime`).
2. Many subsequent requests (seq=2 … N) are created by any users and fulfilled normally. Each fulfilled request leaves `firstUnfulfilledSeq` pinned at 1.
3. When seq=1 is eventually fulfilled (by anyone after the exclusivity period), the loop scans all N−1 cleared slots.

With Ethereum's 30M gas block limit and ~2,200 gas per cold-SLOAD iteration (overflow mapping), approximately **~13,600 cleared requests** ahead of the pending one are sufficient to cause OOG. `currentSequenceNumber` is `uint64` and grows with normal protocol usage. A single attacker can also deliberately manufacture this condition by:

- Submitting request seq=1 with a minimal fee (making it unattractive for the provider to fulfill during exclusivity).
- Submitting and having fulfilled a large batch of subsequent requests.
- Triggering `executeCallback` for seq=1 after the exclusivity period, causing OOG.

The code itself contains a developer `TODO` acknowledging this exact problem: [4](#0-3) 

---

### Recommendation

Replace the linear scan with a data structure that supports O(1) advancement of `firstUnfulfilledSeq`. The developer comment already suggests the correct fix:

> *"a better solution would be a doubly-linked list of active requests."*

Alternatively, remove `firstUnfulfilledSeq` entirely and rely on off-chain indexing (events) to enumerate active requests, since `getFirstActiveRequests` is a view function used only off-chain.

---

### Proof of Concept

```solidity
// Scenario: seq=1 is pending; N requests are created and fulfilled; seq=1 is fulfilled last.
// Each iteration of the while loop in executeCallback does:
//   findRequest(seq) => SLOAD requests[shortKey] (warm ~100 gas)
//                    => SLOAD requestsOverflow[keccak256(seq)] (cold ~2100 gas)
// Total per iteration: ~2200 gas
// Ethereum block gas limit: 30,000,000 gas
// Iterations to OOG: 30,000,000 / 2,200 ≈ 13,636

// Step 1: Create request seq=1 (attacker or any user)
echo.requestPriceUpdatesWithCallback{value: fee}(provider, publishTime, priceIds, gasLimit);
// seq=1 is now pending; firstUnfulfilledSeq=1

// Step 2: Create and fulfill ~14,000 more requests (seq=2..14001)
for (uint i = 0; i < 14000; i++) {
    uint64 seq = echo.requestPriceUpdatesWithCallback{value: fee}(...);
    echo.executeCallback(provider, seq, updateData, priceIds);
    // firstUnfulfilledSeq stays at 1 because seq=1 is still active
}
// currentSequenceNumber = 14002, firstUnfulfilledSeq = 1

// Step 3: Fulfill seq=1 — the while loop must scan 14,000 cleared slots → OOG
echo.executeCallback(provider, 1, updateData, priceIds); // REVERTS with OOG
// req.fee for seq=1 is permanently locked in the contract
```

The `EchoGasBenchmark.t.sol` test `testMultipleRequestsOutOfOrderFulfillment` already demonstrates the gas growth pattern at small scale (10 requests), confirming the linear relationship: [5](#0-4)

    function findRequest(
        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];
        }
    }
