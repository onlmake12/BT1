### Title
`block.number` / `blockhash` Mismatch Permanently Locks Entropy Requests on Arbitrum ŌĆö (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Pyth Entropy's `request()` function stores `block.number` as the request's block number and later calls `blockhash(req.blockNumber)` during `reveal()`. On Arbitrum, `block.number` returns the L1 (Ethereum) block number, while `blockhash(n)` returns the L2 block hash for L2 block `n`. Because L2 block numbers are orders of magnitude larger than L1 block numbers, the stored L1 block number is always outside the 256-block window for L2 blockhashes, causing `blockhash` to return `bytes32(0)` and the reveal to permanently revert with `BlockhashUnavailable`. Any user who calls `request()` with `useBlockHash = true` on Arbitrum loses their fee and their request is permanently unresolvable.

---

### Finding Description

In `requestHelper`, the current block number is stored into the request struct:

```solidity
req.blockNumber = SafeCast.toUint64(block.number);  // line 262
req.useBlockhash = useBlockhash;
``` [1](#0-0) 

During `revealHelper`, if `useBlockhash` is true, the contract calls:

```solidity
bytes32 _blockHash = blockhash(req.blockNumber);
if (_blockHash == bytes32(uint256(0)))
    revert EntropyErrors.BlockhashUnavailable();
``` [2](#0-1) 

On Arbitrum:
- `block.number` returns the **L1 Ethereum block number** (e.g., ~22,000,000 as of mid-2025).
- `blockhash(n)` returns the **L2 Arbitrum block hash** for L2 block `n`.
- The current L2 block number on Arbitrum is ~300,000,000+.

Therefore, `blockhash(22_000_000)` attempts to retrieve the L2 blockhash for L2 block 22,000,000, which is approximately 278,000,000 L2 blocks in the past ŌĆö far outside the EVM's 256-block window. The opcode returns `bytes32(0)`, triggering the `BlockhashUnavailable` revert unconditionally.

The legacy `request()` function is the entry point that allows `useBlockHash = true`:

```solidity
function request(
    address provider,
    bytes32 userCommitment,
    bool useBlockHash
) public payable override returns (uint64 assignedSequenceNumber) {
``` [3](#0-2) 

The `Request` struct in `EntropyStructsV2` stores `blockNumber` as a `uint64` field: [4](#0-3) 

---

### Impact Explanation

Any user who calls `request(provider, userCommitment, true)` on Arbitrum pays the required fee (provider fee + Pyth fee), but the subsequent `reveal()` call will **always revert** with `BlockhashUnavailable`. The request is permanently stuck: the fee is non-refundable, the sequence number is consumed, and the random number can never be delivered. This constitutes a **permanent loss of user funds** for every such request on Arbitrum.

---

### Likelihood Explanation

Pyth Entropy is **actively deployed on Arbitrum mainnet** (`0x7698E925FfC29655576D0b361D75Af579e20AdAc`) and Arbitrum Sepolia (`0x549Ebba8036Ab746611B4fFA1423eb0A4Df61440`): [5](#0-4) 

The `request()` function with `useBlockHash = true` is part of the public ABI. Any user or integrating contract that calls it on Arbitrum ŌĆö following the documented interface ŌĆö will trigger the bug. The `IEntropy` interface exposes this function, so third-party integrators may use it.

---

### Recommendation

1. In `requestHelper`, replace `block.number` with `block.timestamp` for the stored value used in blockhash-based randomness, or use a chain-specific L2 block number accessor (e.g., `ArbSys(address(100)).arbBlockNumber()` on Arbitrum).
2. Alternatively, disable `useBlockhash = true` on chains where `block.number` does not correspond to the same numbering space as `blockhash(n)`, and document this clearly.
3. Add a chain-awareness check or NatSpec warning on `request()` noting that `useBlockHash = true` is unsafe on L2s where `block.number` returns the L1 block number.

---

### Proof of Concept

1. Deploy or use the existing Entropy contract on Arbitrum One (`0x7698E925FfC29655576D0b361D75Af579e20AdAc`).
2. Call `request(provider, keccak256(abi.encode(secret)), true)` with sufficient fee. This stores `req.blockNumber = block.number` (the current L1 block, e.g., 22,000,000) and `req.useBlockhash = true`.
3. Wait at least 1 block (so the request block is not the current block).
4. Call `reveal(provider, sequenceNumber, secret, providerContribution)`.
5. Inside `revealHelper`, `blockhash(22_000_000)` is evaluated. The current Arbitrum L2 block is ~300,000,000, so L2 block 22,000,000 is outside the 256-block window. `blockhash` returns `bytes32(0)`.
6. The transaction reverts with `BlockhashUnavailable`. The user's fee is permanently lost.
