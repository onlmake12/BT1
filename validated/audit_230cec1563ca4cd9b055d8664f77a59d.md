### Title
Missing Domain Separator in `verifyUpdate()` Enables Cross-Chain Signature Replay — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` computes the signature hash as `keccak256(payload)` with no chain ID or contract address bound into the digest. Because the same Pyth Lazer trusted-signer keys are used across every EVM deployment, a signed update captured on one chain is cryptographically valid on every other EVM chain where `PythLazer` is deployed.

---

### Finding Description

In `PythLazer.verifyUpdate()`, the hash that is passed to `ECDSA.tryRecover` is constructed as:

```solidity
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` is a raw byte sequence whose header (parsed by `parsePayloadHeader`) contains only: a fixed `FORMAT_MAGIC` (2479346549), a `timestamp`, a `channel` byte, and feed count — none of which are chain-specific. [2](#0-1) 

The outer `EVM_FORMAT_MAGIC` (706910618) checked before the hash is also a hardcoded constant identical on every EVM chain. [3](#0-2) 

There is no `block.chainid`, no `address(this)`, and no EIP-712 `domainSeparator` mixed into the digest. The `PythLazerStructs.Update` struct confirms the payload carries only `timestamp`, `channel`, and `feeds[]` — no chain-binding field. [4](#0-3) 

---

### Impact Explanation

Pyth Lazer is deployed on multiple EVM chains (Ethereum, Arbitrum, BNB Chain, etc.) with the same set of trusted signers registered via `updateTrustedSigner`. Because the signed digest contains no chain ID or contract address, a valid `(r, s, v, payload)` tuple accepted on chain A is byte-for-byte accepted on chain B.

Concrete consequences:

1. **Stale-price injection across chains.** An attacker observes a valid signed update on a low-activity chain (e.g., a testnet or a chain with slower block times), then replays the same bytes on a high-value chain. The timestamp in the payload may still be within any freshness window enforced by the consuming protocol, causing it to accept a price that was signed for a different deployment context.

2. **Signer-revocation bypass.** If a trusted signer is compromised and removed from the `PythLazer` contract on one chain, any signatures that signer produced while still valid on another chain can be replayed on the chain where they were revoked — because `isValidSigner` checks the recovered address against the current mapping, but the mapping state differs per chain. [5](#0-4) 

---

### Likelihood Explanation

- `verifyUpdate` is a `payable` function callable by any unprivileged address with `msg.value >= verification_fee` (1 wei).
- The attacker only needs to observe a valid update on any one EVM chain (trivially done by monitoring the mempool or on-chain events) and submit the same calldata on another chain.
- No privileged access, no key compromise, and no off-chain collusion is required.
- Pyth Lazer is already deployed on multiple EVM chains, making the cross-chain surface immediately reachable.

---

### Recommendation

Bind the signed digest to the specific deployment by including `block.chainid` and `address(this)` in the hashed message, following EIP-712:

```solidity
bytes32 domainSeparator = keccak256(abi.encode(
    keccak256("EIP712Domain(string name,uint256 chainId,address verifyingContract)"),
    keccak256("PythLazer"),
    block.chainid,
    address(this)
));
bytes32 hash = keccak256(abi.encodePacked(
    "\x19\x01",
    domainSeparator,
    keccak256(payload)
));
```

Alternatively, at minimum prepend `abi.encodePacked(block.chainid, address(this))` to the payload before hashing, mirroring the fix applied in the referenced `a356632` commit for `hashForKYC()`.

---

### Proof of Concept

1. Deploy `PythLazer` on chain A (e.g., Arbitrum, chainId=42161) and chain B (e.g., BNB Chain, chainId=56) with the same trusted signer.
2. On chain A, call `verifyUpdate{value: 1 wei}(update)` with a freshly signed update. Record the `update` bytes from the transaction calldata.
3. On chain B, submit an identical transaction with the same `update` bytes.
4. `verifyUpdate` on chain B computes `keccak256(payload)` — identical to chain A — recovers the same signer address, passes `isValidSigner`, and returns successfully.
5. Any protocol on chain B consuming the returned `payload` now processes a price update that was signed for chain A, with no cryptographic evidence it was intended for chain B. [6](#0-5)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );

            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
