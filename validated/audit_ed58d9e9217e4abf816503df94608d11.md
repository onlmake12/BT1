### Title
Stakers Can Front-Run `createSlashEvent` to Escape Slashing via `undelegate` — Unfair Slash Loss Distribution - (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`, `governance/pyth_staking_sdk/src/types/integrity-pool.ts`)

---

### Summary

The OIS integrity pool applies slashing lazily, per-staker, via a separate `slash` instruction that must be called individually for each delegator's account. The `undelegate` instruction does not require the caller's `DelegationRecord` to be up-to-date (i.e., all pending `SlashEvent`s applied) before allowing undelegation. A staker who monitors the chain can observe a `createSlashEvent` transaction and immediately call `undelegate` before the `slash` instruction is applied to their account, escaping the slash entirely. Remaining stakers bear the full proportional slash, breaking the fairness guarantee of the slashing mechanism.

---

### Finding Description

The integrity pool uses a two-step slashing design:

1. **`createSlashEvent`** — called by `rewardProgramAuthority`, creates a `SlashEvent` PDA at a sequential `index` for a given publisher, storing `epoch`, `slash_ratio`, and `slash_custody`. It increments `pool_data.num_slash_events[publisher_index]`.

2. **`slash`** — a **permissionless** instruction that applies a specific `SlashEvent` to a single staker's `stakeAccountPositions`. It reads `delegationRecord.next_slash_event_index` to know which slash events have already been applied to that staker, then advances it.

The `DelegationRecord` struct tracks per-staker state:
- `last_epoch` — last epoch rewards were settled
- `next_slash_event_index` — the next slash event index that must be applied to this staker [1](#0-0) 

The `slash` instruction requires `delegationRecord` as a writable account: [2](#0-1) 

However, the **`undelegate` instruction does not include `delegationRecord` in its accounts at all**: [3](#0-2) 

Because `undelegate` has no access to `delegationRecord`, it cannot enforce that `delegationRecord.next_slash_event_index == pool_data.num_slash_events[publisher_index]` before allowing the staker to exit. A staker can undelegate with pending, unapplied slash events on their account.

This is confirmed by the SDK's `unstakeFromPublisher` implementation, which calls `undelegate` directly with no pre-check or forced `advanceDelegationRecord` call: [4](#0-3) 

---

### Impact Explanation

A staker who front-runs `createSlashEvent` by calling `undelegate` before `slash` is applied to their account:

- Exits with their full stake intact, paying zero slash penalty
- The slash ratio is applied only to stakers who remain and have `slash` called on them
- Since the slash ratio is fixed (e.g., 5% of the pool), the remaining stakers absorb the same absolute token loss across a smaller remaining pool, effectively increasing their proportional loss

This directly mirrors the SUSD vault pattern: early "withdrawers" (undelegators) escape the loss; late stakers bear a disproportionate share. The slashing mechanism's core security guarantee — that all stakers in a pool at the time of a misprint are penalized proportionally — is broken.

---

### Likelihood Explanation

- `createSlashEvent` is an on-chain transaction, publicly visible in the mempool and block history
- The `slash` instruction must be called **per-staker** (one transaction per `stakeAccountPositions`), creating a window between event creation and individual application
- Any staker running a monitoring bot (or even manually watching the chain) can detect `createSlashEvent` and submit `undelegate` in the same or next block
- The `undelegate` call is self-signed by the staker — no third-party cooperation needed
- This is a rational, self-interested action with direct financial benefit (avoiding up to 5% loss)

Likelihood: **High** for sophisticated stakers; **Medium** overall.

---

### Recommendation

Enforce that a staker's `DelegationRecord` is fully up-to-date before allowing `undelegate`. Specifically:

1. Add `delegationRecord` as a required account in the `undelegate` instruction.
2. In the `undelegate` handler, assert:
   ```
   delegation_record.next_slash_event_index == pool_data.num_slash_events[publisher_index]
   ```
   returning `OutdatedDelegatorAccounting` (error 6008, which already exists) if not satisfied.
3. Require the staker to call `slash` for all pending slash events (or `advance_delegation_record`) before undelegating. [5](#0-4) 

---

### Proof of Concept

1. Staker Alice delegates 1,000,000 PYTH to publisher P. Pool total = 2,000,000 PYTH (Alice + Bob each 1M).
2. A price misprint occurs. `rewardProgramAuthority` calls `createSlashEvent(index=0, slash_ratio=50000)` (5% slash). `pool_data.num_slash_events[P] = 1`.
3. Alice's bot detects the `createSlashEvent` transaction on-chain.
4. Alice immediately calls `undelegate(position_index, 1_000_000)`. This succeeds because `undelegate` does not check `delegation_record.next_slash_event_index`.
5. Alice withdraws her full 1,000,000 PYTH — zero slash applied.
6. `slash(index=0)` is now called on Bob's account. Bob's 1,000,000 PYTH is slashed by 5% = 50,000 PYTH.
7. Bob loses 50,000 PYTH. Alice loses 0. The intended outcome was each losing 25,000 PYTH. [6](#0-5) [1](#0-0)

  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }
