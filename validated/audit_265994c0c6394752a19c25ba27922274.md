### Title
Unprotected `initialize` Function Allows Anyone to Seize `governance_authority` - (File: `target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

The `initialize` instruction of the `pyth-solana-receiver` Solana program imposes no constraint on who the caller (`payer`) must be. Any unprivileged transaction sender can call `initialize` before the legitimate deployer and supply an arbitrary `initial_config`, setting themselves as `governance_authority` and configuring malicious data sources, a fake Wormhole address, and a minimal signature threshold.

---

### Finding Description

The `Initialize` accounts struct contains only a generic `payer: Signer<'info>` with no `constraint` tying it to the program upgrade authority or any other privileged key:

```rust
#[derive(Accounts)]
#[instruction(initial_config : Config)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(init, space = Config::LEN, payer=payer,
              seeds = [CONFIG_SEED.as_ref()], bump)]
    pub config: Account<'info, Config>,
    pub system_program: Program<'info, System>,
}
``` [1](#0-0) 

The handler itself only validates `minimum_signatures > 0` and then blindly writes the caller-supplied `initial_config` into the PDA:

```rust
pub fn initialize(ctx: Context<Initialize>, initial_config: Config) -> Result<()> {
    require!(
        initial_config.minimum_signatures > 0,
        ReceiverError::ZeroMinimumSignatures
    );
    let config = &mut ctx.accounts.config;
    **config = initial_config;
    Ok(())
}
``` [2](#0-1) 

The `Config` struct written here includes `governance_authority`, `valid_data_sources`, `wormhole`, `single_update_fee_in_lamports`, and `minimum_signatures`. All governance-gated instructions (`set_data_sources`, `set_fee`, `set_wormhole_address`, `set_minimum_signatures`, `request_governance_authority_transfer`) require the signer to match `config.governance_authority`:

```rust
pub struct Governance<'info> {
    #[account(constraint =
        payer.key() == config.governance_authority @
        ReceiverError::GovernanceAuthorityMismatch
    )]
    pub payer: Signer<'info>,
    ...
}
``` [3](#0-2) 

Whoever wins the initialization race owns all of these capabilities permanently (until a governance transfer is accepted by the legitimate party — which is impossible if the attacker initialized first).

An identical pattern exists in `pythnet/message_buffer`: the `Initialize` struct accepts any `admin: Signer` with no deployer constraint, allowing anyone to become the whitelist admin who controls which programs may write to the accumulator. [4](#0-3) 

---

### Impact Explanation

An attacker who calls `initialize` first can:

1. **Set themselves as `governance_authority`** — gaining permanent control over all governance-gated parameters.
2. **Supply malicious `valid_data_sources`** — causing the receiver to accept price updates from attacker-controlled Wormhole emitters, injecting arbitrary prices into every downstream DeFi protocol consuming Pyth feeds on Solana.
3. **Set a fake `wormhole` address** — bypassing guardian-set verification entirely.
4. **Set `minimum_signatures = 1`** — reducing the guardian quorum to a single signature, trivially forgeable.

The config PDA is initialized with `init` (not `init_if_needed`), so once the attacker initializes it, the legitimate deployer cannot overwrite it. The impact is complete compromise of the Pyth Solana receiver's trust model.

---

### Likelihood Explanation

On Solana, program deployment and initialization are separate transactions. An attacker monitoring on-chain activity for a newly deployed `pyth-solana-receiver` program ID can immediately submit an `initialize` transaction. Solana's ~400 ms block time creates a realistic front-running window. No privileged access, leaked key, or oracle collusion is required — only the ability to submit a transaction.

---

### Recommendation

Add a constraint in the `Initialize` accounts struct that restricts the caller to the program's upgrade authority (the BPF Upgradeable Loader's upgrade authority PDA for the program):

```rust
#[derive(Accounts)]
#[instruction(initial_config: Config)]
pub struct Initialize<'info> {
    #[account(mut,
        constraint = payer.key() == program_data.upgrade_authority_address
            .ok_or(ReceiverError::Unauthorized)?
            @ ReceiverError::Unauthorized
    )]
    pub payer: Signer<'info>,
    #[account(
        seeds = [crate::ID.as_ref()],
        seeds::program = bpf_loader_upgradeable::ID,
        bump
    )]
    pub program_data: Account<'info, ProgramData>,
    #[account(init, space = Config::LEN, payer = payer,
              seeds = [CONFIG_SEED.as_ref()], bump)]
    pub config: Account<'info, Config>,
    pub system_program: Program<'info, System>,
}
```

Apply the same fix to `message_buffer`'s `Initialize` struct.

---

### Proof of Concept

1. Deploy `pyth-solana-receiver` to devnet. Note the program ID.
2. Before the Pyth team calls `initialize`, submit a transaction calling `initialize` with:
   - `payer` = attacker keypair (any funded wallet)
   - `initial_config.governance_authority` = attacker pubkey
   - `initial_config.valid_data_sources` = attacker-controlled emitter
   - `initial_config.wormhole` = attacker-controlled fake wormhole program
   - `initial_config.minimum_signatures` = 1
3. The config PDA is now owned by the attacker. The legitimate deployer's subsequent `initialize` call reverts with Anchor's account-already-initialized error.
4. Attacker calls `set_data_sources` (signed by attacker as `governance_authority`) to point the receiver at a fake accumulator, then posts price updates signed by a single attacker-controlled guardian key — accepted by the receiver as valid.
