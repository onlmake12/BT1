### Title
Deviation-Only Subscriptions Permanently DOSed When All Price Feeds Have Zero Prices — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

In `Scheduler.sol`, the `_validateShouldUpdatePrices` function silently skips every price feed when either the stored previous price or the incoming current price is zero. When a subscription is configured with **only** deviation-based update criteria (`updateOnDeviation = true`, `updateOnHeartbeat = false`) and all subscribed feeds carry a zero price, the loop exhausts without ever returning a valid timestamp, and the function falls through to an unconditional `revert UpdateConditionsNotMet()`. Keepers are permanently blocked from updating the subscription, and for `isPermanent` subscriptions the state is irreversible.

### Finding Description

`_validateShouldUpdatePrices` in `Scheduler.sol` handles the deviation check as follows:

```solidity
if (params.updateCriteria.updateOnDeviation) {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        ...
        int64 currentPrice  = priceFeeds[i].price.price;
        int64 previousPrice = previousFeed.price.price;

        // Skip if either price is zero to avoid division by zero
        if (previousPrice == 0 || currentPrice == 0) {
            continue;          // ← silently skips the feed
        }
        ...
    }
}
revert SchedulerErrors.UpdateConditionsNotMet();   // ← always reached when all feeds are skipped
```

The guard correctly avoids a division-by-zero in `Math.mulDiv`, but it does not handle the case where **every** feed is skipped. After the loop, execution falls unconditionally to the revert at the bottom of the function.

**Trigger conditions:**

1. Subscription has `updateOnDeviation = true` and `updateOnHeartbeat = false`.
2. At least one of the following is true for every feed in the subscription:
   - The previously stored price is `0` (e.g., an asset that was listed at zero or whose first stored update was zero).
   - The incoming current price is `0` (e.g., an asset that has collapsed to zero — a rug pull, de-peg, or token death).

When condition 2 holds for all feeds, every iteration of the loop hits `continue`, the loop exits without returning, and `UpdateConditionsNotMet` is thrown. The entire `updatePriceFeeds` call reverts, no state is written, and the keeper receives no payment.

For subscriptions marked `isPermanent = true`, the subscription owner cannot call `updateSubscription` to add a heartbeat criterion as a workaround, making the DOS permanent.

### Impact Explanation

- **Keeper DOS:** Any keeper calling `updatePriceFeeds` for an affected subscription will have their transaction revert. They receive no fee and cannot update the subscription.
- **Stale prices served to consumers:** The subscription's stored prices remain at the last non-zero value. Consumers calling `getPricesNoOlderThan` or `getPricesUnsafe` will receive the stale pre-zero price, which is precisely the moment accurate price data is most critical (e.g., a token collapse).
- **Permanent lock for `isPermanent` subscriptions:** Because `updateSubscription` is blocked for permanent subscriptions, there is no on-chain recovery path. The subscription is frozen indefinitely.
- **Subscription balance is not drained** (the revert undoes all state changes), but the subscription is rendered permanently non-functional.

### Likelihood Explanation

- Assets going to zero is a realistic and recurring event in DeFi (rug pulls, de-pegs, protocol exploits).
- Subscriptions configured with deviation-only criteria (no heartbeat) are a valid and documented configuration.
- The `isPermanent` flag is explicitly supported and intended for long-lived, immutable subscriptions — exactly the use case most harmed by this bug.
- No privileged access or external oracle manipulation is required; the trigger is a legitimate on-chain price value of `0`.

### Recommendation

When all feeds are skipped due to zero prices, do not silently fall through to `UpdateConditionsNotMet`. Instead, treat a price going to zero as an unconditional trigger for an update (a 100% deviation is always above any threshold):

```solidity
if (previousPrice == 0 || currentPrice == 0) {
    // A zero price is always a significant event; trigger the update.
    return updateTimestamp;
}
```

Alternatively, track whether any feed was actually evaluated, and if none were (all skipped), return `updateTimestamp` to allow the update.

### Proof of Concept

1. Deploy `Scheduler` and create a subscription with:
   - Two price feed IDs.
   - `updateOnHeartbeat = false`, `updateOnDeviation = true`, `deviationThresholdBps = 100`.
   - `isPermanent = true`.
2. Call `updatePriceFeeds` with both feeds having `price = 1000`. This succeeds and stores the prices.
3. Call `updatePriceFeeds` again with both feeds having `price = 0` (asset collapsed).
4. Observe: the call reverts with `UpdateConditionsNotMet`.
5. Attempt to call `updateSubscription` to add a heartbeat — observe it reverts with `CannotUpdatePermanentSubscription`.
6. The subscription is now permanently frozen, serving the stale price of `1000` to all consumers.

        // If updateOnDeviation is enabled, check if any price has deviated enough
        if (params.updateCriteria.updateOnDeviation) {
            for (uint8 i = 0; i < priceFeeds.length; i++) {
                // Get the previous price feed for this price ID using subscriptionId
                PythStructs.PriceFeed storage previousFeed = _state
                    .priceUpdates[subscriptionId][priceFeeds[i].id];

                // If there's no previous price, this is the first update
                if (previousFeed.id == bytes32(0)) {
                    return updateTimestamp;
                }

                // Calculate the deviation percentage
                int64 currentPrice = priceFeeds[i].price.price;
                int64 previousPrice = previousFeed.price.price;

                // Skip if either price is zero to avoid division by zero
                if (previousPrice == 0 || currentPrice == 0) {
                    continue;
                }

                // Calculate absolute deviation basis points (scaled by 1e4)
                uint256 numerator = SignedMath.abs(
                    currentPrice - previousPrice
                );
                uint256 denominator = SignedMath.abs(previousPrice);
                uint256 deviationBps = Math.mulDiv(
                    numerator,
                    10_000,
                    denominator
                );

                // If deviation exceeds threshold, trigger update
                if (
                    deviationBps >= params.updateCriteria.deviationThresholdBps
                ) {
                    return updateTimestamp;
                }
            }
        }

        revert SchedulerErrors.UpdateConditionsNotMet();
