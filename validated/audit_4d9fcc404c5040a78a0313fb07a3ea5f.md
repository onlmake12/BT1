### Title
DoS in DAO Withdrawal via Mismatched `header_dep_index` Width Between Rust `DaoCalculator` and On-Chain C DAO Script — (`util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator` in `util/dao/src/lib.rs` reads the `header_dep_index` stored in the witness `input_type` field as a full `u64` (8 bytes, little-endian), while the on-chain C DAO script (`dao.c`) reads only the **lowest byte** of that same field. When a DAO withdrawal transaction encodes an index whose full `u64` value exceeds 255 but whose lowest byte is a valid deposit-block position, the two implementations resolve different headers. The Rust node rejects the transaction at the tx-pool and contextual-verification layers, even though the C script would accept it. This is a direct interface mismatch between the host verifier and the on-chain script, causing a DoS for any DAO withdrawal that uses a `header_dep_index > 255`.

---

### Finding Description

In `DaoCalculator::transaction_maximum_withdraw`, the Rust code extracts the deposit-block index from the witness:

```rust
// util/dao/src/lib.rs, lines 83–96
let header_deps_index_data: Option<Bytes> =
    witness.input_type().to_opt().map(|witness| witness.into());
// ...
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))  // full u64
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // indexes with full u64
        // ...
})
```

The on-chain C DAO script (`dao.c`) is documented in the codebase to resolve the same field using only the **lowest byte** of the 8-byte little-endian value. The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly encodes this discrepancy:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
// input_type = 257, lowest byte = 1
```

With `header_dep_index = 257` (LE bytes: `[0x01, 0x01, 0x00, …]`):
- **C script** reads lowest byte → index `1` → deposit block (block 100) → `100 == 100` → **PASS**
- **Rust** reads full u64 → index `257` → withdraw block (block 200) → `200 != 100` → **`DaoError::InvalidOutPoint`**

The Rust node rejects the transaction at two enforcement points:
1. **Tx-pool admission** via `check_tx_fee` in `tx-pool/src/util.rs` (line 34–41), which calls `DaoCalculator::transaction_fee`.
2. **Contextual block verification** via `FeeCalculator::transaction_fee` in `verification/src/transaction_verifier.rs` (line 265–273), called from `ContextualTransactionVerifier::verify` (line 170).

---

### Impact Explanation

A DAO withdrawal transaction that the on-chain C DAO script considers valid is permanently rejected by the Rust node. The transaction can never enter the tx-pool or be included in a block. Any user who constructs a DAO withdrawal with more than 255 `header_deps` and places the deposit-block hash at a position whose index exceeds 255 (but whose lowest byte is the correct position) will have their withdrawal permanently blocked. This renders the DAO withdrawal feature unusable for that class of transactions — a direct DoS on DAO withdrawals.

---

### Likelihood Explanation

The scenario requires a DAO withdrawal transaction with more than 255 `header_deps`. While uncommon in typical usage (most withdrawals have 2 header_deps), the CKB protocol does not enforce a hard cap of 255 header_deps at the consensus layer, making the condition reachable by any unprivileged transaction sender. The discrepancy is already documented in a production test, confirming the root cause is real and not hypothetical.

---

### Recommendation

Align the Rust `DaoCalculator` with the C DAO script's index interpretation. If the C script reads only the lowest byte, the Rust code should do the same:

```rust
// util/dao/src/lib.rs, line 91
// Change:
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// To (if C script uses lowest byte):
Ok(u64::from(header_deps_index_data.unwrap()[0]))
```

Alternatively, if the C script is the one that should be updated to use the full u64, update `dao.c` accordingly and add a consensus rule capping `header_deps` length to 255 to prevent the ambiguity. Either way, both implementations must agree on the same interface for the witness index field.
