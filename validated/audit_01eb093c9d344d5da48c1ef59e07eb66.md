### Title
Unprivileged Tx-Pool Submitter Permanently Evicts Other Users' Pending Cell-Dep Transactions via Cell-Consuming Submission - (File: tx-pool/src/component/pool_map.rs)

### Summary
When a transaction that **consumes** a cell (as an input) is submitted to the tx-pool, the ancestor-count enforcement logic in `check_and_record_ancestors` silently and permanently evicts other users' pending transactions that reference the same cell as a **cell-dep**. These evictions are irreversible: evicted entries receive `Reject::Invalidated` and are not placed into any recovery queue. An unprivileged tx-pool submitter can exploit this to permanently remove other users' pending transactions from the pool, even if the consuming transaction itself is subsequently evicted.

### Finding Description
In `tx-pool/src/component/pool_map.rs`, `check_and_record_ancestors` (lines 582–640) handles the case where a new cell-consuming transaction would exceed `max_ancestors_count` because many pending transactions reference the consumed cell as a cell-dep (tracked as `cell_ref_parents`). When `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, the function evicts the lowest-fee-rate cell-ref transactions one by one until the ancestor count is within limits:

```rust
while ancestors_count > self.max_ancestors_count {
    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id);
        ancestors_count = ancestors_count.saturating_sub(1);
        parents.remove(next_id);
        evicted.extend(removed);
    } else { break; }
}
``` [1](#0-0) 

These evicted entries bubble up through `add_entry` → `_submit_entry` → `submit_entry`, where they are permanently rejected:

```rust
for evict in evicted {
    let reject = Reject::Invalidated(format!(
        "invalidated by tx {}", evict.transaction().hash()
    ));
    self.callbacks.call_reject(tx_pool, &evict, reject);
}
``` [2](#0-1) 

Unlike RBF-replaced transactions, which are placed into `conflicts_cache` for potential recovery via `record_conflict`, these cell-ref evictions are **not** placed into any recovery structure. [3](#0-2) 

The root flaw: the eviction is triggered by a **pending, unconfirmed** cell-consuming transaction. If that consuming transaction is itself later evicted (e.g., by `limit_size` due to pool pressure), the victims' cell-dep transactions are permanently gone even though the cell remains live on-chain. [4](#0-3) 

### Impact Explanation
An unprivileged attacker (any `send_transaction` RPC caller or relay peer) can:
1. Observe via RPC that many pending transactions reference cell X as a cell-dep.
2. Submit a transaction consuming cell X as an input.
3. Cause permanent, silent eviction of up to `ancestors_count − max_ancestors_count` of those cell-dep transactions.
4. If the consuming transaction is subsequently evicted from the pool (e.g., pool full, low fee rate), the victims' transactions are permanently gone even though cell X is still live.

Victims must resubmit their transactions. If those transactions carried time-sensitive `since` constraints, they may miss their valid commitment window entirely. There is no recovery path.

### Likelihood Explanation
- Cell-dep usage is publicly observable via `get_transaction` and `get_live_cell` RPC.
- The default `max_ancestors_count` is 25; only 26+ pending cell-dep transactions on the same cell are needed to trigger eviction.
- No privileged access, leaked keys, or majority hashpower is required — any unprivileged tx-pool submitter qualifies.
- The behavior was intentionally introduced as an improvement (`#4363: Evict possible cell ref txs before submitting cell consuming transaction`) without a recovery mechanism for the evicted transactions. [5](#0-4) 

### Recommendation
1. **Add recovery**: Place evicted cell-ref transactions into `conflicts_cache` (mirroring the RBF path) so they can be re-queued if the consuming transaction is later evicted.
2. **Reject instead of evict**: Reject the consuming transaction with `ExceededMaximumAncestorsCount` rather than silently evicting other users' transactions.
3. **Document the risk**: At minimum, document that submitting a cell-consuming transaction can permanently evict other pending cell-dep transactions from the pool.

### Proof of Concept
The integration test `TxPoolLimitAncestorCount` in `test/src/specs/tx_pool/limit.rs` (lines 70–157) directly demonstrates the mechanism: [6](#0-5) 

1. Alice submits 2000 transactions T1–T2000, each using cell X (output of committed `tx_a`) as a cell-dep.
2. Bob submits a transaction consuming cell X as an input (with ≤25 non-cell-ref ancestors).
3. `check_and_record_ancestors` evicts 1002 of Alice's transactions (lowest fee-rate first) with `Reject::Invalidated`.
4. Bob's consuming transaction enters the pool.
5. Bob's transaction is then evicted by `limit_size` (pool full, low fee rate).
6. Alice's 1002 transactions are permanently gone; cell X is still live on-chain. Alice must resubmit, potentially missing any `since`-gated windows.

    /// Check ancestors and record for entry
    // FIXME: In the scenario that a transaction passed all RBF rules, and then removed the conflicted
    // transaction in txpool, then failed with max ancestor limits, we now need to rollback the removing.
    // this is not an issue currently, because RBF have a rule that not allow any unknown inputs except
    // the conflicted inputs, so the new transaction can not be in a long transaction chain.
    // but it's still safer to report an error before any writing kind of operation.
    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
