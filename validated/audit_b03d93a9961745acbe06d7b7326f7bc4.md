### Title
Gas Griefing via Unbounded Callback in `revealWithCallback()` Legacy Path — (File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol)

---

### Summary

In `Entropy.sol`'s `revealWithCallback()` function, when a provider has not configured a default gas limit (`req.gasLimit10k == 0`), the callback to the requester's contract is executed as a direct, unbounded call. A malicious requester can deploy a contract whose `_entropyCallback` consumes all available gas before reverting. This causes the provider's entire `revealWithCallback` transaction to revert — including the `clearRequest` state change — leaving the request permanently stuck and forcing the provider to bear repeated gas losses with no ability to clear the request.

---

### Finding Description

`revealWithCallback()` has two distinct execution paths:

**New path** (`req.gasLimit10k != 0`, lines 574–660): Uses `excessivelySafeCall` with an explicit gas cap and a callback-failure state machine. Reverts in the callback are caught and recorded; the request is marked `CALLBACK_FAILED` and can be retried or recovered.

**Legacy path** (`req.gasLimit10k == 0`, lines 661–702): Falls into the `else` branch. The code:

1. Clears the request from storage (`clearRequest(provider, sequenceNumber)` — line 666)
2. Checks if the requester is a contract (`extcodesize` — lines 670–673)
3. Calls the callback **without any gas limit** (line 676):

```solidity
IEntropyConsumer(callAddress)._entropyCallback(
    sequenceNumber,
    provider,
    randomNumber
);
```

Because this is a direct (non-low-level) call with no gas stipend, any revert inside `_entropyCallback` — including an out-of-gas revert from an unbounded loop — propagates upward and reverts the **entire** `revealWithCallback` transaction. This rolls back `clearRequest`, leaving the request active. The provider's gas is consumed with no compensation and no progress.

The legacy path is active for any provider whose `defaultGasLimit` is `0` (i.e., providers that have not opted into the new callback-failure flow by calling `setDefaultGasLimit`). [1](#0-0) 

The new path's protection via `excessivelySafeCall` is explicitly absent in the legacy branch: [2](#0-1) 

The fee is credited to the provider at **request time** inside `requestHelper`, not at reveal time: [3](#0-2) 

This means the provider already holds their fee when they call `revealWithCallback`. However, the gas cost of each failed `revealWithCallback` attempt is borne entirely by the provider, and since the request is never cleared, an automated provider service will retry indefinitely.

---

### Impact Explanation

- The provider's `revealWithCallback` transaction reverts on every attempt against a malicious requester.
- `clearRequest` is rolled back each time, so the request is never cleared.
- The provider's automated fulfillment service keeps retrying, spending gas on each attempt.
- Cumulative gas losses across many such requests (each costing the attacker only the request fee) can far exceed the fees collected.
- The stuck request also occupies a slot in the provider's active request set, potentially degrading service for legitimate users.

---

### Likelihood Explanation

The attack requires only:
1. Deploying a contract with a gas-exhausting `_entropyCallback`.
2. Calling `requestWithCallback` (or `requestV2`) — a permissionless, unprivileged action — paying the provider's fee.
3. Waiting for the provider's automated service to attempt fulfillment.

The attacker pays the request fee once per griefed request. The provider pays gas on every failed `revealWithCallback` attempt. The ratio of attacker cost to provider damage is highly favorable to the attacker, especially at scale. Any provider that has not called `setDefaultGasLimit` (keeping `defaultGasLimit == 0`) is exposed. [4](#0-3) 

---

### Recommendation

- **Short term:** In the legacy path (`req.gasLimit10k == 0`), replace the direct call with a low-level `call` that caps gas (e.g., using `excessivelySafeCall` with a reasonable default), or catch reverts so the transaction does not roll back `clearRequest`.
- **Long term:** Require all providers to set a non-zero `defaultGasLimit` before accepting new requests, effectively retiring the legacy path. This forces all requests through the safe callback-failure state machine already implemented in the new path.

---

### Proof of Concept

```solidity
// Malicious requester contract
contract MaliciousRequester is IEntropyConsumer {
    IEntropy entropy;

    constructor(address _entropy) { entropy = IEntropy(_entropy); }

    function getEntropy() internal view override returns (address) {
        return address(entropy);
    }

    // Consumes all forwarded gas, then reverts
    function _entropyCallback(
        uint64, address, bytes32
    ) external override {
        uint256 i = 0;
        while (true) { i++; } // unbounded loop
    }

    function attack(address provider, bytes32 userRandom) external payable {
        entropy.requestWithCallback{value: msg.value}(provider, userRandom);
    }
}
```

**Steps:**
1. Deploy `MaliciousRequester`.
2. Call `attack(provider, userRandom)` with sufficient fee — this calls `requestWithCallback`, crediting the provider's fee immediately.
3. Provider's automated service calls `revealWithCallback(provider, seqNum, userContrib, providerContrib)`.
4. Execution reaches line 676; `_entropyCallback` runs an unbounded loop, exhausting all gas.
5. The call reverts; the revert propagates, rolling back `clearRequest` at line 666.
6. Provider's transaction is reverted; provider loses all gas spent.
7. The request remains active. The provider's service retries — step 3–6 repeats indefinitely. [5](#0-4)
