### Title
Provider Fee Decrease Frontrun Causes Irrecoverable User Overpayment to Pyth Accrued Fees — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract allows providers (and their fee managers) to update `feeInWei` at any time with no delay or lock. The `requestHelper` function never refunds excess `msg.value` to the caller: all value above `providerFee` is permanently credited to `_state.accruedPythFeesInWei`. When a provider's fee decreases between a user's `getFeeV2()` query and their `requestV2()` execution — a scenario that occurs routinely via the Fortuna keeper's automated `adjust_fee_wrapper` — the user overpays and the difference is irrecoverably absorbed by the protocol.

---

### Finding Description

In `requestHelper`, the fee split is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);   // providerFee + pythFee
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

The critical accounting error is on line 238–239: Pyth receives `msg.value − providerFee`, **not** `msg.value − requiredFee`. Any value above `providerFee` — including the entire `pythFee` component and any excess from a fee decrease — flows into `accruedPythFeesInWei` with no refund path.

`setProviderFee` and `setProviderFeeAsFeeManager` impose no time-lock, no minimum notice period, and no guard against in-flight requests:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;   // immediate, unconditional
}
```

The Fortuna keeper (`apps/fortuna/src/keeper/fee.rs`) runs `adjust_fee_wrapper` on a polling loop and calls `set_provider_fee_as_fee_manager` whenever the on-chain fee falls outside the target window:

```rust
let contract_call = contract.set_provider_fee_as_fee_manager(provider_address, target_fee);
send_and_confirm(contract_call).await?;
```

This automated adjustment races with user transactions in the mempool on every fee change.

**Concrete overpayment scenario (fee decrease):**

| Step | Action |
|------|--------|
| 1 | User calls `getFeeV2(provider, 0)` → `F_old = providerFee_old + pythFee` |
| 2 | User submits `requestV2{value: F_old}(provider, ...)` |
| 3 | Keeper (or provider) frontruns: `setProviderFee(providerFee_new)` where `providerFee_new < providerFee_old` |
| 4 | User tx executes: `msg.value = F_old ≥ providerFee_new + pythFee` → passes `InsufficientFee` check |
| 5 | Provider accrues `providerFee_new` |
| 6 | Pyth accrues `F_old − providerFee_new = pythFee + (providerFee_old − providerFee_new)` |
| 7 | User loses `providerFee_old − providerFee_new` with no refund |

The user followed the documented best practice (querying `getFeeV2` immediately before the call) and still loses funds.

---

### Impact Explanation

Users permanently lose native token (ETH/native gas token) equal to the fee delta whenever a provider fee decreases between their query and their transaction's execution. The lost funds are credited to `_state.accruedPythFeesInWei` and are not recoverable by the user. On high-traffic chains or during periods of gas price volatility (when the keeper adjusts fees frequently), multiple users can be affected per fee update cycle.

---

### Likelihood Explanation

**Medium.** The Fortuna keeper's `adjust_fee_wrapper` runs continuously and adjusts fees whenever the on-chain fee falls outside `[target_fee_min, target_fee_max]`. On any chain where gas prices fluctuate, fee decreases are routine. Any user whose `requestV2` transaction sits in the mempool during a downward fee adjustment will overpay. This is not a hypothetical: it is a structural race condition between the keeper's fee management loop and user request transactions, requiring no malicious intent.

---

### Recommendation

Replace the current accounting with an exact split that refunds excess to the caller:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += _state.pythFeeInWei;          // exact Pyth share
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

Alternatively, add a `maxFee` slippage parameter to `requestV2` so users can bound their exposure.

---

### Proof of Concept

1. Provider registers with `feeInWei = 1 ether`; `pythFeeInWei = 0.01 ether`.
2. User calls `getFeeV2(provider, 0)` → `1.01 ether`.
3. User broadcasts `requestV2{value: 1.01 ether}(provider, userRandom, 0)`.
4. Keeper observes gas price drop; calls `setProviderFeeAsFeeManager(provider, 0.5 ether)` — this lands first.
5. User tx executes: `msg.value = 1.01 ether ≥ 0.51 ether` (new `requiredFee`) → no revert.
6. `providerInfo.accruedFeesInWei += 0.5 ether`.
7. `_state.accruedPythFeesInWei += 1.01 ether − 0.5 ether = 0.51 ether` (instead of `0.01 ether`).

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );

        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
