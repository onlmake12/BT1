### Title
Stale `descendants_*` Accumulator Fields After `remove_entry_and_descendants` Enables Tx-Pool Eviction Manipulation — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link entries before calling `remove_entry` on each member of the subtree. Because `update_ancestors_index_key` relies on the live link graph to find which entries to update, removing the links first silently prevents the ancestors of the removed subtree from having their `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` fields decremented. Those fields remain permanently inflated until the pool is cleared or the node restarts. The inflated values feed directly into each ancestor's `EvictKey`, causing the eviction subsystem to treat those ancestors as more valuable than they actually are and defer their eviction indefinitely.

---

### Finding Description

**Root cause — `remove_entry_and_descendants`** [1](#0-0) 

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ← ALL links torn down first
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

The comment explains the intent: by removing links first, `update_descendants_index_key` inside `remove_entry` becomes a no-op (the descendant set is empty), avoiding redundant work on entries that are themselves being removed. However, `update_ancestors_index_key` is equally neutralised: [2](#0-1) 

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());  // ← empty; link already gone
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
```

Because `remove_entry_links(root_id)` has already been called, `calc_ancestors(root_id)` returns an empty set. The loop body never executes. Every entry that is an ancestor of the removed subtree but is **not** itself in `removed_ids` retains stale, inflated `descendants_count / descendants_size / descendants_cycles / descendants_fee`.

**Where the stale values are consumed**

`EvictKey` is recomputed from `descendants_fee` and `descendants_size/cycles`: [3](#0-2) 

```rust
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_weight =
            get_transaction_weight(entry.descendants_size, entry.descendants_cycles);
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        let feerate = FeeRate::calculate(entry.fee, weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),
            ...
        }
    }
}
```

An ancestor whose `descendants_fee` is inflated will have an artificially high `descendants_feerate`, making `fee_rate` appear larger than the entry's own fee rate. The eviction iterator picks the entry with the **lowest** `EvictKey`, so the inflated ancestor is systematically skipped.

The block-template selector also reads `ancestors_size` / `ancestors_cycles` from the cached fields for a pre-check: [4](#0-3) 

```rust

**Trigger path — `resolve_conflict`**

`resolve_conflict` is called every time a new transaction is submitted whose inputs conflict with pool entries: [5](#0-4) 

Any unprivileged RPC caller (`send_transaction`) or P2P relay peer can reach this path.

---

### Impact Explanation

An ancestor entry whose `descendants_*` fields are inflated will:

1. **Evade eviction** — `next_evict_entry` iterates by `EvictKey`; the inflated ancestor is never the minimum, so it is never selected for eviction even when the pool is over its size limit.
2. **Corrupt subsequent eviction decisions** — every future call to `limit_size` skips the ancestor, potentially evicting legitimate higher-fee transactions instead.
3. **Persist until pool clear or restart** — the stale fields are never corrected by any background task; only `clear()` or a full reorg that triggers `clear_pool` resets them.

---

### Likelihood Explanation

The trigger is straightforward and cheap:

1. Submit parent tx **P** (low fee, just above `min_fee_rate`).
2. Submit high-fee children **C₁, C₂, …** spending P's outputs. P's `descendants_*` fields are updated upward.
3. Submit conflicting transactions **C₁′, C₂′, …** (RBF-eligible, slightly higher fee) that spend the same inputs as C₁, C₂, …. `resolve_conflict` → `remove_entry_and_descendants(C₁)` → P's `descendants_*` fields are **not** decremented.
4. P now appears to have high-fee descendants it no longer has. Repeat steps 2–3 to refresh the illusion.

The attacker pays only the marginal RBF fee bump on each cycle. No privileged access, no majority hash power, and no Sybil attack is required — a single RPC connection suffices.

---

### Recommendation

Before tearing down the link graph in `remove_entry_and_descendants`, propagate the removal to the ancestors of the root entry. Concretely, call `update_ancestors_index_key(root_entry, EntryOp::Remove)` **before** the `remove_entry_links` loop, while the link graph is still intact and `calc_ancestors` can return the correct ancestor set. Only then proceed to strip links and remove entries.

---

### Proof of Concept

```
State before:
  A (low fee, pending) → B (high fee, pending) → C (high fee, pending)
  A.descendants_fee = fee_B + fee_C + fee_A
  A.descendants_count = 3

Attacker submits B' (conflicts with B, fee slightly > fee_B via RBF):
  resolve_conflict(B) → remove_entry_and_descendants(B)
    removed_ids = [B, C]
    remove_entry_links(B)   ← B's link entry deleted
    remove_entry_links(C)   ← C's link entry deleted
    remove_entry(B):
      update_ancestors_index_key(B, Remove)
        calc_ancestors(B) → {} (link gone)
        → A.sub_descendant_weight(B) NEVER CALLED
    remove_entry(C): similarly no-op

State after:
  A still in pool
  A.descendants_fee  = fee_B + fee_C + fee_A  ← STALE (should be fee_A only)
  A.descendants_count = 3                      ← STALE (should be 1)
  A.EvictKey.fee_rate = descendants_feerate(inflated) > A.own_feerate
  → A is never selected by next_evict_entry even when pool is over limit
