Audit Report

## Title
Unbounded `GetBlocks` Flood Without Rate Limiting or Peer Ban — (`sync/src/synchronizer/get_blocks_process.rs`)

## Summary
`GetBlocksProcess::execute` silently `continue`s and returns `Status::ok()` for any block hash that fails `contains_block_status(BLOCK_VALID)`, performing up to 32 synchronous DB lookups per message inside `tokio::task::block_in_place`. The `Synchronizer` struct has no rate limiter, and `post_sync_process` only bans peers on 4xx status codes. An unprivileged remote peer can flood the victim at wire speed with fabricated hashes, exhausting tokio worker threads and crashing the node's ability to participate in the network.

## Finding Description

**Root cause:** In `GetBlocksProcess::execute`, when `active_chain.contains_block_status(&block_hash, BlockStatus::BLOCK_VALID)` returns `false` for an unknown hash, the code `continue`s the loop and ultimately returns `Status::ok()` (code 100). [1](#0-0) 

The three early-exit ban paths (`block_hashes.len() > MAX_HEADERS_LEN` → 400, genesis hash → 417, duplicate hash → 418) are all bypassed by a message containing 1–32 distinct, non-genesis, random hashes. For all such messages, the function always returns `Status::ok()`. [2](#0-1) 

`post_sync_process` calls `nc.ban_peer` only when `status.should_ban()` returns `Some`, which requires the status code to be in `400..500`. A 100-series code never triggers a ban. [3](#0-2) [4](#0-3) 

The `Synchronizer` struct has no `rate_limiter` field: [5](#0-4) 

By contrast, `Relayer` initializes a `RateLimiter` at 30 req/sec and enforces it before dispatching any message: [6](#0-5) 

`StatusCode::TooManyRequests = 110` exists but is only used in the relayer path — never in the synchronizer: [7](#0-6) 

Each `GetBlocks` message iterates up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32) hashes, calling `contains_block_status` (a synchronous DB lookup) for each, all inside `tokio::task::block_in_place`, which blocks a tokio worker thread for the duration: [8](#0-7) [9](#0-8) 

## Impact Explanation

An attacker sending `GetBlocks` messages at wire speed causes 32 synchronous DB lookups per message, each blocking a tokio worker thread via `block_in_place`. With enough concurrent inbound connections or a sufficiently fast sender, all tokio worker threads can be saturated, preventing the node from processing any other protocol messages (block announcements, header sync, etc.). This matches the allowed **High** impact: **"Vulnerabilities which could easily crash a CKB node."**

## Likelihood Explanation

The attack requires only a standard inbound P2P connection — no PoW, no key material, no special privileges. The attacker constructs `GetBlocks` messages with 32 random 32-byte hashes (all unknown to the victim), which are guaranteed to pass all three ban-triggering checks and return `Status::ok()`. The attack is trivially reproducible with a minimal P2P client implementing the CKB sync protocol wire format. The asymmetry with `Relayer` (which has rate limiting) confirms this is an oversight, not a design choice.

## Recommendation

Add a per-peer rate limiter to `Synchronizer`, mirroring the pattern in `Relayer`:

1. Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to the `Synchronizer` struct.
2. In `Synchronizer::try_process`, before dispatching to `GetBlocksProcess`, check the rate limiter and return `StatusCode::TooManyRequests` (or a 4xx code to ban persistent abusers) when the limit is exceeded.
3. Consider applying the rate limiter to all `Synchronizer` message types, not just `GetBlocks`.

## Proof of Concept

**Invariant (provable from code):** For any `GetBlocks` message where `0 < block_hashes.len() <= MAX_HEADERS_LEN`, no hash equals genesis, and no duplicate hashes exist, `GetBlocksProcess::execute` always returns `Status::ok()` regardless of whether the hashes correspond to real, unknown, or fabricated blocks. [10](#0-9) 

**Attack loop (pseudocode):**
```python
while True:
    hashes = [random_bytes(32) for _ in range(32)]  # all unknown to victim
    send GetBlocks(hashes) to victim via SyncProtocol
    # victim performs 32 block_in_place DB lookups, returns ok(), never bans
```

The existing unit tests for `get_blocks_process` confirm the only ban-triggering cases are genesis and duplicate hashes — unknown hashes are never tested for ban behavior, consistent with the code returning `Status::ok()` for them.

    pub fn execute(self) -> Status {
        let block_hashes = self.message.block_hashes();
        // use MAX_HEADERS_LEN as limit, we may increase the value of INIT_BLOCKS_IN_TRANSIT_PER_PEER in the future
        if block_hashes.len() > MAX_HEADERS_LEN {
            return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                "BlockHashes count({}) > MAX_HEADERS_LEN({})",
                block_hashes.len(),
                MAX_HEADERS_LEN,
            ));
        }
        let active_chain = self.synchronizer.shared.active_chain();

        let iter = block_hashes.iter().take(INIT_BLOCKS_IN_TRANSIT_PER_PEER);

        let mut dedup = HashSet::new();
        for block_hash in iter {
            debug!("get_blocks {} from peer {:?}", block_hash, self.peer);
            let block_hash = block_hash.to_entity();

            if block_hash == self.synchronizer.shared().consensus().genesis_hash() {
                return StatusCode::RequestGenesis.with_context("Request genesis block");
            }

            if !dedup.insert(block_hash.clone()) {
                return StatusCode::RequestDuplicate.with_context("Request duplicate block");
            }

            if !active_chain.contains_block_status(&block_hash, BlockStatus::BLOCK_VALID) {
                debug!(
                    "Ignoring get_block {} request from peer={} as it is not verified.",
                    block_hash, self.peer
                );
                continue;
            }

            if let Some(block) = active_chain.get_block(&block_hash) {
                debug!(
                    "respond_block {} {} to peer {:?}",
                    block.number(),
                    block.hash(),
                    self.peer,
                );
                let content = packed::SendBlock::new_builder().block(block.data()).build();
                let message = packed::SyncMessage::new_builder().set(content).build();

                let nc = Arc::clone(self.nc);
                self.synchronizer
                    .shared()
                    .shared()
                    .async_handle()
                    .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
            } else {
                // TODO response not found
                // TODO add timeout check in synchronizer

                // We expect that `block_hashes` is sorted descending by height.
                // So if we cannot find the current one from local, we cannot find
                // the next either.
                debug!("Stopping getblocks, since {} is not found", block_hash);
                break;
            }
        }

        Status::ok()
    }
