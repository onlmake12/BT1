### Title
Delegated Stake Permanently Locked When Publisher Is Removed from `pool_data` - (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

In Pyth's Oracle Integrity Staking (OIS) program, users delegate PYTH tokens to publisher pools. The `undelegate` instruction explicitly validates the publisher against `pool_data`. If a publisher is removed from `pool_data` (their slot set to `PublicKey.default`), the check fails with `PublisherNotFound` (error 6000), permanently locking any delegated stake that targeted that publisher.

---

### Finding Description

The `pool_data` account stores an array of up to 1024 publisher pubkeys. Publishers are considered "active" when their slot is non-default; removal is represented by setting the slot to `PublicKey.default`. The off-chain SDK's `extractPublisherData` confirms this by filtering out default-key entries:

```ts
poolData.publishers.filter((publisher) => !publisher.equals(PublicKey.default))
``` [1](#0-0) 

The `undelegate` instruction's IDL carries the comment:

> `"CHECK : The publisher will be checked against data in the pool_data"` [2](#0-1) 

This is the standard Anchor pattern for a manual runtime check. The program iterates `pool_data.publishers` to find the publisher's index; if the publisher is absent (removed), it returns error 6000 `PublisherNotFound`: [3](#0-2) 

The same publisher-existence check applies to `advance_delegation_record`, which is required before `undelegate` can be called (to settle pending rewards). Both paths are blocked once the publisher is removed: [4](#0-3) 

The `pool_data` structure confirms publishers are stored in a fixed 1024-slot array alongside per-publisher `del_state`, `self_del_state`, `num_slash_events`, and `delegation_fees` arrays â€” all indexed by publisher position: [5](#0-4) 

When a publisher is removed (slot zeroed), the user's `stakeAccountPositions` still contains a `Position` record with `targetWithParameters.integrityPool.publisher` pointing to the removed key. There is no instruction in the IDL that allows closing such a position without passing through the `undelegate` path that requires the publisher to exist in `pool_data`.

The `unstakeFromAllPublishers` SDK helper iterates all positions and calls `undelegate` for each â€” it would also fail for any position targeting a removed publisher: [6](#0-5) 

---

### Impact Explanation

A delegator who staked PYTH tokens to a publisher that is subsequently removed from `pool_data` loses the ability to:

1. Call `undelegate` (fails `PublisherNotFound`).
2. Call `advance_delegation_record` (same check, same failure).
3. Claim any pending rewards for that publisher.
4. Withdraw those tokens from the custody account.

The affected PYTH tokens remain locked in the staking custody account indefinitely. Because OIS positions occupy slots in the fixed-size `stakeAccountPositions` array, the locked positions also consume position slots, potentially preventing the user from creating new delegations.

---

### Likelihood Explanation

Publishers can be removed from the active set through governance or through the `advance` instruction when `publisher_caps` no longer includes them. Publisher removal is a normal operational event (e.g., a publisher stops operating, is slashed out, or is replaced). Any delegator who did not manually undelegate before the removal epoch boundary would be affected. The `advance` instruction is permissionless and is called regularly to update pool state, meaning the window to undelegate before removal takes effect may be as short as one epoch (one week).

---

### Recommendation

1. **Allow `undelegate` for removed publishers**: Remove or relax the publisher-existence check in `undelegate` when the position's publisher is no longer in `pool_data`. The position's stored publisher key is sufficient to identify and close the position without requiring the publisher to still be active.

2. **Emit a removal event**: When a publisher is removed from `pool_data` during `advance`, emit an on-chain event so delegators can react within the same epoch.

3. **Provide a force-close path**: Add a dedicated instruction (e.g., `force_undelegate_removed_publisher`) that allows closing positions whose publisher key is no longer present in `pool_data`, bypassing the existence check.

---
