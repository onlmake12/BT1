Audit Report

## Title
Silent `u64` Truncation in `calculate_maximum_withdraw` Causes Incorrect NervosDAO Withdrawal Amounts - (File: util/dao/src/lib.rs)

## Summary

In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `u128` result `withdraw_counted_capacity` is cast to `u64` via `as u64` at line 156, silently truncating when the value exceeds `u64::MAX`. Every other analogous `u128`→`u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When truncation occurs, the function returns a drastically wrong (much smaller) withdrawal amount without signaling an error, and any block containing a withdrawal transaction built from that amount is subsequently rejected at the `dao_field_with_current_epoch` step due to underflow in `withdrawed_interests`.

## Finding Description

**Root cause — confirmed in source:**

`util/dao/src/lib.rs` lines 152–156:
```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
```

The `as u64` cast at line 156 is a silent truncation. Every other `u128`→`u64` conversion in the same file uses checked conversion:
- Line 204: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- Line 245: `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?`
- Line 258: `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?`

**Overflow condition:**
```
counted_capacity * withdrawing_ar / deposit_ar > u64::MAX
```
`deposit_ar` starts at `10^10`. `counted_capacity` can be up to ~3.36 × 10^18 shannons (total CKB supply). Overflow requires `withdrawing_ar > ~5.36 × 10^10`, i.e., AR must grow ~5.36× from deposit time.

**Exploit flow:**
1. `calculate_maximum_withdraw` returns a truncated (much smaller) value instead of an error.
2. The public RPC `calculate_dao_maximum_withdraw` (`rpc/src/module/experiment.rs`) calls this function and returns the wrong value to the user.
3. The user constructs a withdrawal transaction using the truncated amount.
4. When a miner includes this transaction in a block, `dao_field_with_current_epoch` calls `withdrawed_interests` (lines 312–333), which computes `maximum_withdraws.safe_sub(input_capacities)`. Because `maximum_withdraws` is the truncated value — smaller than `input_capacities` — `safe_sub` returns `DaoError::Overflow`, and the block is rejected.

**Why existing checks fail:**
The existing test `check_withdraw_calculation_overflows` (lines 296–350) only exercises the `safe_add` overflow path (where `withdraw_counted_capacity` is just below `u64::MAX` and adding `occupied_capacity` overflows). It does not cover the silent truncation path where the truncated value is small enough that `safe_add` succeeds with a wrong result.

## Impact Explanation

This matches **"Vulnerabilities which could easily damage CKB economy"**: a NervosDAO depositor who deposited a large amount of CKB and holds until AR has grown sufficiently will find the RPC reports a drastically wrong withdrawal amount. Any withdrawal transaction constructed from that amount causes the containing block to be rejected, permanently preventing the depositor from withdrawing their funds via the normal path. A miner who includes such a transaction loses their block reward.

## Likelihood Explanation

On mainnet, the secondary epoch reward is ~1.344 billion CKB/year against a total capacity of ~33.6 billion CKB, giving an AR growth rate of ~4%/year. A 5.36× increase requires approximately 42 years of chain operation — a long-term scenario on mainnet. However, on testnets or devnets with higher secondary reward ratios or lower total capacity, the condition is reachable much sooner. The bug is structurally present and inconsistent with the rest of the file today, and any future parameter change that increases secondary issuance or reduces total capacity accelerates the timeline.

## Recommendation

Replace the silent `as u64` cast with the same checked conversion pattern used everywhere else in the file:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

## Proof of Concept

Construct a unit test with:
- `counted_capacity = 3_360_000_000_000_000_000` shannons (~33.6 billion CKB)
- `deposit_ar = 10_000_000_000`
- `withdrawing_ar = 60_000_000_000` (AR grown 6×)

Computation:
```
withdraw_counted_capacity = 3_360_000_000_000_000_000 * 60_000_000_000 / 10_000_000_000
                          = 20_160_000_000_000_000_000
```
`u64::MAX = 18_446_744_073_709_551_615`

`20_160_000_000_000_000_000 > u64::MAX`, so:
```
withdraw_counted_capacity as u64
  = 20_160_000_000_000_000_000 - 18_446_744_073_709_551_616
  = 1_713_255_926_290_448_384
```

The user receives ~1.71 × 10^18 shannons instead of ~2.016 × 10^19 shannons — roughly 11.8× less than entitled. Subsequently, in `withdrawed_interests`:
```
maximum_withdraws (truncated) < input_capacities → safe_sub underflows → DaoError::Overflow → block rejected
```

A test mirroring `check_withdraw_calculation_overflows` but using the above parameters would demonstrate that `calculate_maximum_withdraw` currently returns `Ok` with a wrong value instead of `Err(DaoError::Overflow)`, and that after the fix it correctly returns `Err`.
