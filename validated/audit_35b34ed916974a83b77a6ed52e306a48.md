Audit Report

## Title
Stale `descendants_*` Fields on Surviving Ancestors After `remove_entry_and_descendants` Corrupts `EvictKey` — (`tx-pool/src/component/pool_map.rs`)

## Summary

`remove_entry_and_descendants` severs all parent/child links via `remove_entry_links` for every transaction in the removal batch before calling `remove_entry` on each. Because `update_ancestors_index_key` inside `remove_entry` relies on `links.calc_ancestors` to find surviving pool entries that need their `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` decremented, and those links are already gone at call time, surviving ancestors of the removed subtree are never updated. Their descendant-weight fields remain permanently inflated, corrupting the `EvictKey` used to decide which transactions to evict when the pool is full.

## Finding Description

**Root cause — link severance before ancestor update:**

`remove_entry_and_descendants` collects the target and all its descendants, then calls `remove_entry_links` for every ID in the batch before any `remove_entry` call: [1](#0-0) 

`remove_entry_links` removes the entry from each parent's `children` set, removes the entry from each child's `parents` set, and removes the entry's own record from `self.links.inner`: [2](#0-1) 

After this, when `remove_entry` is called for each removed ID, it invokes `update_ancestors_index_key` at line 242: [3](#0-2) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`: [4](#0-3) 

`calc_ancestors` delegates to `calc_relative_ids`, which does `self.inner.get(short_id)`. Since `remove_entry_links` already called `self.links.remove(id)`, this returns `None`, and `unwrap_or_default()` yields an empty `HashSet`: [5](#0-4) 

The loop over ancestors is never entered. `sub_descendant_weight` is never called on any surviving ancestor: [6](#0-5) 

**Contrast with single-entry path:** In `remove_entry` called directly (not via `remove_entry_and_descendants`), `update_ancestors_index_key` is called at line 242 and `remove_entry_links` is called at line 245 — links are still intact when ancestors are looked up, so the single-entry path is correct.

The developer comment at line 256 (`// update links state for remove, so that we won't update_descendants_index_key in remove_entry`) confirms the intent was only to suppress descendant updates, but the same link severance also silences the ancestor update path.

**Stale fields corrupt `EvictKey`:**

The `EvictKey` for each surviving ancestor is computed from the now-stale `descendants_fee` and `descendants_size`: [7](#0-6) 

`EvictKey` is ordered ascending by `fee_rate`: [8](#0-7) 

`next_evict_entry` takes the first (lowest `EvictKey`) entry, so a transaction with an inflated `fee_rate` is ranked later and is less likely to be evicted: [9](#0-8) 

## Impact Explanation

When the pool exceeds `max_tx_pool_size`, `limit_size` calls `next_evict_entry` to select which transaction to drop: [10](#0-9) 

A surviving ancestor with stale (inflated) `descendants_fee` appears to have a higher fee rate than it actually does. It is ranked below legitimate high-fee transactions in the eviction order and is not dropped. Legitimate high-fee transactions are evicted in its place. An attacker can keep a near-zero-fee transaction in the pool indefinitely at the cost of a single RBF replacement fee, and can repeat this to fill the pool with low-fee transactions that resist eviction. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

`remove_entry_and_descendants` is called from four reachable paths, all triggerable by an unprivileged peer or RPC caller:

1. `resolve_conflict` — any conflicting/RBF `send_transaction` RPC call [11](#0-10) 
2. `resolve_conflict_header_dep` — any block reorg invalidating a header dep [12](#0-11) 
3. `limit_size` — automatic when pool exceeds `max_tx_pool_size` [13](#0-12) 
4. `check_and_record_ancestors` — when ancestor count exceeds `max_ancestors_count` [14](#0-13) 

The simplest exploit requires only two `send_transaction` calls followed by one conflicting replacement. No special privileges, no victim mistakes, no external dependencies.

## Recommendation

Before severing links in `remove_entry_and_descendants`, collect the set of external ancestors (ancestors not in `removed_ids`) and update their `descendants_*` fields first:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();

    // Update surviving ancestors BEFORE severing links
    for removed_id in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(removed_id) {
            let entry_inner = entry.inner.clone();
            let ancestors = self.links.calc_ancestors(removed_id);
            for anc_id in ancestors.iter().filter(|a| !removed_set.contains(*a)) {
                self.entries.modify_by_id(anc_id, |e| {
                    e.inner.sub_descendant_weight(&entry_inner);
                    e.evict_key = e.inner.as_evict_key();
                });
            }
        }
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }
    removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
}
```

Alternatively, restructure `remove_entry` so that `update_ancestors_index_key` is called before `remove_entry_links`, matching the invariant already upheld by the single-entry removal path.

## Proof of Concept

**Setup:**
- Submit `tx0` (low fee, e.g. 100 shannons) to the pool. After insertion: `tx0.descendants_fee = 100`, `tx0.descendants_count = 1`.
- Submit `tx1` (high fee, e.g. 10,000 shannons, spending `tx0`'s output). After insertion: `tx0.descendants_fee = 10,100`, `tx0.descendants_count = 2`.

**Trigger:**
- Submit `tx1'` conflicting with `tx1` (fee > `tx1` fee + RBF minimum). `resolve_conflict` calls `remove_entry_and_descendants(&tx1_id)`.
  - `remove_entry_links(tx1_id)` is called: severs the `tx0 → tx1` link.
  - `remove_entry(tx1_id)` is called: `update_ancestors_index_key` calls `calc_ancestors(tx1_id)` → returns `{}` (links already gone) → `sub_descendant_weight` never called on `tx0`.

**Result:**
- `tx1` is removed. `tx0.descendants_fee` remains `10,100` (should be `100`). `tx0.evict_key.fee_rate` is computed from the inflated `descendants_fee`, making `tx0` appear as a high-fee transaction.

**Effect:**
- Fill the pool to capacity with medium-fee transactions. `limit_size` calls `next_evict_entry`, which iterates `iter_by_evict_key()` ascending. `tx0`'s inflated `EvictKey` ranks it above medium-fee transactions, so medium-fee transactions are evicted before `tx0`. The attacker keeps a near-zero-fee transaction in the pool indefinitely. Repeating this pattern with multiple parent transactions fills the pool with low-fee transactions that resist eviction, causing legitimate high-fee transactions to be rejected with `Reject::Full`.

    // Remove transactions from the pool until total size <= size_limit.
    // Return a `Reject` for current inserting entry if it's removed
    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
