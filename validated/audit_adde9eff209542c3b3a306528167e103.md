### Title
TOCTOU Race Condition in `resumeble_process_tx` Allows Silent Duplicate Enqueue and Verify-Queue Position Reset — (File: `tx-pool/src/process.rs`)

---

### Summary

A time-of-check-time-of-use (TOCTOU) race condition exists in `TxPoolService::resumeble_process_tx`. The duplicate-presence check (`verify_queue_contains`) and the actual insertion (`enqueue_verify_queue`) are performed under **separate, non-overlapping locks**. Because the tx-pool service spawns each incoming message as an independent Tokio task, two concurrent submissions of the same transaction can both pass the duplicate check before either acquires the write lock. When the second task then calls `add_tx`, the `VerifyQueue` silently returns `Ok(false)` for non-proposal transactions (analogous to the Solidity `when_edit_allowed` modifier's silent `return`) or, for proposal transactions, **removes the existing entry and re-inserts it**, resetting its queue position. Neither outcome is surfaced as a `Reject::Duplicated` error to the caller.

---

### Finding Description

**Root cause — split-lock check-then-act pattern:**

`resumeble_process_tx` performs three sequential async operations, each acquiring and releasing its own lock:

```
1. orphan_contains(&tx)        → read-locks orphan pool, releases
2. verify_queue_contains(&tx)  → read-locks verify_queue, releases
3. enqueue_verify_queue(tx)    → write-locks verify_queue, inserts
``` [1](#0-0) 

Between steps 2 and 3 the write lock is not held. The tx-pool service loop spawns every incoming message as a separate Tokio task: [2](#0-1) 

Two concurrent tasks carrying the same transaction can both observe the queue as empty at step 2, then both proceed to step 3.

**Silent failure in `VerifyQueue::add_tx`:**

When the second task reaches `add_tx` under the write lock, the queue already contains the transaction. For `is_proposal_tx = false` (the `submit_remote_tx` / P2P relay path), `add_tx` returns `Ok(false)` — a silent success — instead of `Err(Reject::Duplicated(...))`: [3](#0-2) 

For `is_proposal_tx = true` (the `notify_tx` / P2P notification path), `add_tx` **removes the existing entry and re-inserts it**, resetting `added_time` to the current timestamp: [4](#0-3) 

`notify_tx` always passes `is_proposal_tx = true`: [5](#0-4) 

**Propagation — no error reaches the caller:**

`resumeble_process_tx_and_notify_full_reject` only inspects the result for `Err(Reject::Full(_))`; an `Ok(false)` from the race passes through silently: [6](#0-5) 

---

### Impact Explanation

**Scenario A — `submit_remote_tx` path (`is_proposal_tx = false`):**
The second concurrent task receives `Ok(false)` instead of `Err(Reject::Duplicated(...))`. The relayer handler treats this as success and does **not** penalize the originating peer. A peer that floods the node with the same transaction repeatedly can avoid any ban/score penalty that would normally follow a `Duplicated` rejection.

**Scenario B — `notify_tx` path (`is_proposal_tx = true`):**
The second concurrent task causes the existing `VerifyEntry` to be removed and re-inserted with a fresh `added_time`. Because the `VerifyQueue` sorts non-large-cycle transactions by `added_time` (FIFO), the transaction is moved to the **back of the queue**. An attacker peer that repeatedly sends the same transaction via `SendTransactions` P2P messages can keep resetting the transaction's position, indefinitely delaying its verification — a targeted, low-cost DoS against specific pending transactions. [7](#0-6) 

---

### Likelihood Explanation

- **Attacker entry point**: Any unprivileged P2P peer can send `SendTransactions` messages, which trigger `submit_remote_tx` and `notify_tx` paths.
- **Race window**: The window between releasing the `verify_queue` read lock and acquiring the write lock spans at least one `await` yield point, making concurrent interleaving realistic under normal Tokio scheduling.
- **No special privileges required**: No key, operator access, or majority hashpower is needed.
- **Repeatability**: The attacker can send the same transaction in a tight loop; each pair of concurrent messages has a chance to win the race.

Likelihood: **Medium** — the race window is narrow but the attack is repeatable and requires no special access.

---

### Recommendation

1. **Combine the check and the insert under a single write lock.** Move the `orphan_contains` and `verify_queue_contains` checks inside `enqueue_verify_queue` so they execute atomically with the insertion:

   ```rust
   async fn enqueue_verify_queue(...) -> Result<bool, Reject> {
       let mut queue = self.verify_queue.write().await;
       // check orphan pool here too, or accept that orphan is a separate lock
       if queue.contains_key(&tx.proposal_short_id()) {
           return Err(Reject::Duplicated(tx.hash()));
       }
       queue.add_tx(tx, is_proposal_tx, remote)
   }
   ```

2. **Make `VerifyQueue::add_tx` return `Err(Reject::Duplicated(...))` for all duplicate non-proposal insertions** rather than `Ok(false)`, so callers cannot silently ignore the condition. [8](#0-7) 

3. **Propagate `Reject::Duplicated` to the peer-scoring layer** so that peers sending duplicate transactions are penalized consistently regardless of whether the duplicate is detected before or after the lock is acquired.

---

### Proof of Concept

```
1. Attacker peer connects and sends the same transaction T in two rapid
   back-to-back SendTransactions P2P messages.

2. The tx-pool service spawns two Tokio tasks, Task-A and Task-B,
   both executing resumeble_process_tx(T, true, None).

3. Task-A: verify_queue_contains(T) → false  (read lock released)
   Task-B: verify_queue_contains(T) → false  (read lock released, T still absent)

4. Task-A: enqueue_verify_queue acquires write lock → inserts T with
   added_time = t0 → releases lock → returns Ok(true).

5. Task-B: enqueue_verify_queue acquires write lock → add_tx finds T,
   removes it (added_time t0 gone), re-inserts with added_time = t1 > t0
   → releases lock → returns Ok(true).

6. T is now at the back of the FIFO verify queue.
   Attacker repeats steps 1-5 continuously.
   T's verification is indefinitely deferred.
```

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }
