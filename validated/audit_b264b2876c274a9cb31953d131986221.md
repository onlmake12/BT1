### Title
Non-Manager Depositor Permanently Loses ETH Deposited via `addFunds` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.addFunds()` accepts ETH from **any caller** with no access control, but `Scheduler.withdrawFunds()` is gated by `onlyManager`. There is no mechanism for a non-manager depositor to recover their ETH. Any address that calls `addFunds` on a subscription it does not manage permanently loses those funds.

---

### Finding Description

`addFunds` is a fully open payable function:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    ...
    status.balanceInWei += msg.value;
    ...
}
``` [1](#0-0) 

The subscription manager is set once at creation and is never transferable:

```solidity
_state.subscriptionManager[subscriptionId] = msg.sender;
``` [2](#0-1) 

Withdrawal is exclusively gated to the manager:

```solidity
function withdrawFunds(
    uint256 subscriptionId,
    uint256 amount
) external override onlyManager(subscriptionId) {
``` [3](#0-2) 

```solidity
modifier onlyManager(uint256 subscriptionId) {
    if (_state.subscriptionManager[subscriptionId] != msg.sender) {
        revert SchedulerErrors.Unauthorized();
    }
``` [4](#0-3) 

There is no `transferSubscription`, `changeManager`, or any other function that would allow a non-manager depositor to recover their ETH. The test suite itself acknowledges the asymmetry with the comment `"Anyone can add funds (not just manager)"` while `withdrawFunds` remains manager-only. [5](#0-4) 

The permanent-subscription variant is even more severe: `addFunds` permits deposits into permanent subscriptions (up to `MAX_DEPOSIT_LIMIT` per call), yet `withdrawFunds` unconditionally reverts for permanent subscriptions, meaning **no one** — not even the manager — can ever recover ETH deposited by a third party into a permanent subscription. [6](#0-5) 

---

### Impact Explanation

Any ETH sent to `addFunds` by an address that is not the subscription manager is permanently inaccessible to the sender. For permanent subscriptions, the funds are irrecoverable by anyone. This is a direct loss of user funds with no protocol-level recovery path.

---

### Likelihood Explanation

Realistic triggering scenarios:

1. A DeFi protocol that consumes price feeds from a subscription it does not own calls `addFunds` to keep the subscription alive — its ETH is permanently controlled by the subscription manager.
2. A user mistypes a subscription ID and funds the wrong subscription.
3. A subscription manager is a smart contract that lacks a `withdrawFunds` call path; a third-party funder's ETH is then permanently stranded.
4. A subscription is later made permanent (via `updateSubscription` setting `isPermanent = true`); any ETH previously added by a non-manager, or added afterward, is permanently locked.

The open, permissionless nature of `addFunds` makes accidental or intentional third-party deposits a realistic occurrence.

---

### Recommendation

Restrict `addFunds` to `onlyManager`, **or** track the depositing address per deposit and allow each depositor to withdraw their own contribution independently of the manager. At minimum, document clearly that only the manager can ever recover funds, so callers are warned before sending ETH they do not control.

---

### Proof of Concept

```
1. Alice calls createSubscription{value: minBalance}(...) → subscriptionId = 1, manager = Alice.
2. Bob calls addFunds{value: 5 ether}(1).
   → status.balanceInWei increases by 5 ether.
3. Bob calls withdrawFunds(1, 5 ether).
   → Reverts: Unauthorized (onlyManager check, msg.sender != Alice).
4. Bob has no other function to call. His 5 ether is permanently held by the contract,
   withdrawable only by Alice.

Permanent-subscription variant:
5. Alice calls updateSubscription(1, params{isPermanent: true}).
6. Bob calls addFunds{value: 1 ether}(1).
   → Succeeds (isPermanent check only enforces MAX_DEPOSIT_LIMIT, not caller identity).
7. Alice calls withdrawFunds(1, 1 ether).
   → Reverts: CannotUpdatePermanentSubscription.
8. Bob's 1 ether is now irrecoverable by anyone.
```

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
```
