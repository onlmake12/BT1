### Title
Provider Fee Front-Running Causes Irrecoverable User Overpayment with No Slippage Protection - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The Entropy contract's `requestV2` / `requestHelper` flow reads the provider's current `feeInWei` at execution time with no user-supplied maximum fee cap. Because `setProviderFee` takes effect immediately and excess `msg.value` is explicitly never refunded, a registered provider can front-run a user's request transaction to drain the user's overpayment buffer into the protocol's fee pool.

### Finding Description

When a user wants to request randomness they follow the documented pattern:

1. Call `getFeeV2(provider, gasLimit)` off-chain to read the current fee `F`.
2. Submit `requestV2{value: F}(provider, userRandom, gasLimit)`.

Inside `requestHelper`, the fee is re-read from live storage at execution time:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);   // line 234
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee(); // line 235
uint128 providerFee = getProviderFee(provider, callbackGasLimit);    // line 236
providerInfo.accruedFeesInWei += providerFee;                        // line 237
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) -
    providerFee);                                                     // lines 238-239
``` [1](#0-0) 

The provider can call `setProviderFee` at any time with no rate limit, no timelock, and no cap:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;   // immediate, unbounded
    ...
}
``` [2](#0-1) 

The interface explicitly documents that excess value is never returned:

> "Further note that excess value is *not* refunded to the caller." [3](#0-2) 

None of the `requestV2` overloads accept a `maxFee` parameter: [4](#0-3) 

### Impact Explanation

**Attack path A – DoS / forced revert:**
User sends `msg.value = F` (exact fee). Provider front-runs with `setProviderFee(F * 100)`. `requestHelper` reverts with `InsufficientFee`. User's ETH is returned but the request fails; the user must retry at the new, higher fee.

**Attack path B – Irrecoverable fund loss (higher severity):**
Users commonly send `msg.value = F + buffer` to avoid reverts from minor fee fluctuations. The provider front-runs to set `feeInWei` to exactly consume the buffer. The transaction succeeds, but:
- `providerFee` is computed at the new (higher) rate.
- The remainder `msg.value − providerFee` is credited to `_state.accruedPythFeesInWei` — permanently, with no refund path for the user. [5](#0-4) 

The user loses the difference between what they expected to pay and what was actually charged, with no recourse.

### Likelihood Explanation

- Any address can permissionlessly register as a provider via `register()` and immediately call `setProviderFee` to any value.
- The attack requires only two transactions in the same block (fee update + user request), which is trivially achievable by a provider watching the mempool.
- The Fortuna keeper's `adjust_fee_wrapper` already adjusts fees automatically and frequently, meaning fee changes between `getFeeV2()` and `requestV2()` are a normal operational occurrence — not just a malicious scenario. [6](#0-5) 

### Recommendation

1. Add a `maxFeeInWei` parameter to `requestV2` variants. Inside `requestHelper`, revert if `requiredFee > maxFeeInWei`.
2. Refund excess `msg.value` to the caller (`msg.value − requiredFee`) instead of silently crediting it to `accruedPythFeesInWei`.
3. Optionally add a `deadline` parameter and revert if `block.timestamp > deadline`.

### Proof of Concept

```solidity
// 1. Attacker registers as a provider with a low fee
entropy.register(1 wei, commitment, metadata, chainLength, uri);

// 2. Victim observes fee = 1 wei, prepares tx with msg.value = 1000 wei (buffer)
// (tx sits in mempool)

// 3. Attacker front-runs: set fee to 999 wei (just below victim's msg.value)
entropy.setProviderFee(999 wei);

// 4. Victim's tx executes:
//    requiredFee = 999 wei <= msg.value = 1000 wei → succeeds
//    providerFee = 999 wei → accruedFeesInWei[attacker] += 999 wei
//    accruedPythFeesInWei += 1 wei
//    Victim paid 1000 wei, expected to pay 1 wei → lost 999 wei with no refund
entropy.requestV2{value: 1000 wei}(attackerProvider, userRandom, gasLimit);
``` [7](#0-6)

    function requestHelper(
        address provider,
        bytes32 userCommitment,
        bool useBlockhash,
        bool isRequestWithCallback,
        uint32 callbackGasLimit
    ) internal returns (EntropyStructsV2.Request storage req) {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];
        if (_state.providers[provider].sequenceNumber == 0)
            revert EntropyErrors.NoSuchProvider();

        // Assign a sequence number to the request
        uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
        if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
            revert EntropyErrors.OutOfRandomness();
        providerInfo.sequenceNumber += 1;

        // Check that fees were paid and increment the pyth / provider balances.
        uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
        if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
        uint128 providerFee = getProviderFee(provider, callbackGasLimit);
        providerInfo.accruedFeesInWei += providerFee;
        _state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) -
            providerFee);

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
