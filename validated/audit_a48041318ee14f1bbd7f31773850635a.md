### Title
No-Deadline Fee Mismatch in `requestPriceUpdatesWithCallback` Allows Provider to Capture Excess ETH or Force Revert — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.sol`'s `requestPriceUpdatesWithCallback` reads the required fee from mutable provider state at execution time and has no deadline parameter. A provider can call `setProviderFee` to raise fees between when a user estimates the fee off-chain and when the user's transaction executes. This causes either a forced revert (wasted gas) or silent capture of excess ETH the user sent as a buffer, with no refund mechanism.

### Finding Description
In `Echo.sol`, `requestPriceUpdatesWithCallback` computes the required fee at execution time by reading live provider state:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
...
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [1](#0-0) 

`getFee` aggregates three mutable provider fields — `baseFeeInWei`, `feePerFeedInWei`, and `feePerGasInWei` — all of which can be changed at any time by the provider or its fee manager via `setProviderFee`:

```solidity
function setProviderFee(
    address provider,
    uint96 newBaseFeeInWei,
    uint96 newFeePerFeedInWei,
    uint96 newFeePerGasInWei
) external override {
``` [2](#0-1) 

The `getFee` view function reads these fields directly:

```solidity
uint96 providerBaseFee = _state.providers[provider].baseFeeInWei;
uint96 providerFeedFee = SafeCast.toUint96(
    priceIds.length * _state.providers[provider].feePerFeedInWei
);
uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei;
``` [3](#0-2) 

Critically, the entire `msg.value - pythFeeInWei` is stored as `req.fee` and credited to the provider at callback time — there is **no refund** of any excess ETH the user sent:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [4](#0-3) 

There is no `deadline` or `maxFee` parameter in `requestPriceUpdatesWithCallback`: [5](#0-4) 

The same pattern exists in `Entropy.sol`'s `requestHelper`, where `getFeeV2` reads the mutable `provider.feeInWei` at execution time and `setProviderFee` / `setProviderFeeAsFeeManager` can change it at any time: [6](#0-5) 

### Impact Explanation
Two concrete impacts:

1. **Forced revert / wasted gas**: A user estimates `getFee()` off-chain, submits `requestPriceUpdatesWithCallback{value: estimatedFee}`. If the provider raises fees before the tx executes (e.g., due to gas price adjustment by the automated keeper in `apps/fortuna/src/keeper/fee.rs`), the tx reverts with `InsufficientFee`, wasting the user's gas with no service rendered. [7](#0-6) 

2. **Silent ETH capture**: A user who sends `msg.value = estimatedFee + buffer` to avoid reverts has the entire buffer silently credited to the provider as `req.fee`. There is no refund path. If the provider (or its automated fee manager) raises fees to match the user's `msg.value` (visible in the mempool), the provider captures the buffer with no recourse for the user. [8](#0-7) 

### Likelihood Explanation
**Medium.** Providers are permissionless participants (anyone can call `registerProvider`) and the Fortuna keeper actively and automatically adjusts fees based on gas prices via `adjust_fee_if_necessary`. A fee update racing with a user's pending transaction is a realistic, non-adversarial scenario. The mempool visibility of `msg.value` also enables deliberate front-running by a malicious provider. [9](#0-8) 

### Recommendation
Add a `maxFee` (or `deadline`) parameter to `requestPriceUpdatesWithCallback` and revert if `requiredFee > maxFee`. Additionally, refund any excess `msg.value` above `requiredFee` to `msg.sender` rather than silently crediting it to the provider.

### Proof of Concept
1. Provider registers with `baseFeeInWei = 1000 wei`.
2. User calls `getFee(provider, gasLimit, priceIds)` off-chain → returns `1000 + pythFee`.
3. User submits `requestPriceUpdatesWithCallback{value: 1000 + pythFee + 500}` (500 wei buffer) — tx enters mempool.
4. Provider (or Fortuna keeper) calls `setProviderFee(provider, 1500, ...)` — tx mines first.
5. User's tx executes: `getFee` now returns `1500 + pythFee`. Since `msg.value = 1000 + pythFee + 500 = 1500 + pythFee`, the check `msg.value < requiredFee` passes.
6. `req.fee = msg.value - pythFeeInWei = 1500` — the provider is credited 1500 wei instead of 1000 wei. The user paid 500 wei more than the fee they agreed to, with no refund. [10](#0-9)
