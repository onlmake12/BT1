### Title
Conflicting Transactions Permanently Removed from Pool When RBF Replacement Fails - (File: tx-pool/src/process.rs)

---

### Summary

In `TxPoolService::submit_entry`, when RBF is enabled, `process_rbf` permanently removes conflicting transactions from the active pool **before** `_submit_entry` and `limit_size` can fail. If either subsequent step fails, the conflicting transactions are irreversibly lost from the pool without the new transaction being added, causing pool accounting inconsistency and silent transaction loss.

---

### Finding Description

In `submit_entry` (`tx-pool/src/process.rs`, lines 136–152), the execution order is:

```
1. process_rbf(...)          // removes conflicting txs — irreversible state change
2. _submit_entry(...)?       // adds new tx — can fail with early return
3. limit_size(...)?          // enforces pool size — can fail with early return
4. spawn(may_recovered_txs)  // re-queues recovered txs — only reached if 2 and 3 succeed
```

`process_rbf` (lines 203–231) calls `tx_pool.pool_map.remove_entry_and_descendants(id)` for every conflicting transaction, then calls `tx_pool.record_conflict(old.transaction().clone())` and `self.callbacks.call_reject(...)`. These operations are **not rolled back** anywhere. The removed transactions are moved into the bounded `conflicts_cache` LRU and their reject callbacks are fired — they are permanently evicted from the active pool.

If `_submit_entry` (line 137) returns `Err`, the `?` causes an early return before the new transaction is added. If `limit_size` (lines 150–152) returns `Some(Reject::Full(...))` — which happens when the newly inserted transaction itself is the lowest-fee entry and gets evicted — `map_or(Ok(()), Err)?` also causes an early return. In both cases:

- The conflicting transactions are permanently gone from the active pool.
- The new transaction is not present in the pool (either never added, or added then evicted).
- The `tokio::spawn` that re-queues `may_recovered_txs` is **never reached**, so those transactions are also not recovered.
- `total_tx_size` and `total_tx_cycles` in `PoolMap` are understated because the removed transactions' sizes/cycles were subtracted (via `update_stat_for_remove_tx` inside `remove_entry_and_descendants`) but the new transaction's were never added.

The `limit_size` failure path is the most realistic trigger. After removing conflicting transactions and inserting the new transaction, if the pool is still over `max_tx_pool_size` and the new transaction has the lowest fee rate among all remaining entries, `limit_size` evicts it and returns `Some(Reject::Full(...))`.

The `_submit_entry` failure path is also possible: `add_entry` in `pool_map.rs` (lines 213–214) calls `check_and_record_ancestors` and `record_entry_edges`, both of which can return `Reject` errors after `process_rbf` has already mutated pool state.

The existing FIXME comment at `pool_map.rs` line 583 acknowledges a related rollback problem inside `check_and_record_ancestors` itself, confirming the developers are aware of the general class of issue but have not addressed the outer `submit_entry` layer.

---

### Impact Explanation

- **Transaction loss**: Valid, fee-paying transactions (`tx_old` and its descendants) are permanently removed from the active pool without being replaced. They will not be mined.
- **Pool accounting corruption**: `total_tx_size` and `total_tx_cycles` are understated. The pool believes it has more capacity than it actually does, which can cause it to accept more transactions than its configured `max_tx_pool_size` limit.
- **Fee revenue loss for miners**: The removed transactions' fees are lost. The miner's block assembler will not include them.
- **Recovered transactions not re-queued**: `may_recovered_txs` (transactions from the conflicts cache whose inputs are now free) are silently dropped because the `tokio::spawn` is never reached on the early-return paths.

---

### Likelihood Explanation

- RBF must be enabled (`min_rbf_rate > min_fee_rate`), which is a supported and documented node configuration.
- The `limit_size` failure path requires the pool to be near `max_tx_pool_size` and the new transaction to have a lower fee rate than other pool entries (but still higher than the conflicting transaction, satisfying RBF rules). This is a realistic steady-state condition on a busy node.
- The entry point is the `send_transaction` JSON-RPC method, reachable by any unprivileged RPC caller.
- No special privileges, keys, or majority hashpower are required.

---

### Recommendation

Move `process_rbf` to execute **after** both `_submit_entry` and `limit_size` succeed, so conflicting transactions are only removed once the replacement is confirmed to be in the pool. Alternatively, implement a rollback path that re-inserts the removed conflicting transactions if `_submit_entry` or `limit_size` fails. At minimum, the `may_recovered_txs` spawn should be executed unconditionally (not gated behind the `limit_size` success check) so that recoverable transactions are not silently dropped.

---

### Proof of Concept

1. Configure a node with RBF enabled (`min_rbf_rate > min_fee_rate`).
2. Fill the pool to near `max_tx_pool_size` with high-fee-rate transactions.
3. Submit `tx_old` (moderate fee rate) that spends output `O`.
4. Submit `tx_new` via `send_transaction` RPC: spends the same output `O`, fee higher than `tx_old` (passes RBF rule #3/#4), but lower fee rate than the high-fee transactions already in the pool.
5. Inside `submit_entry`:
   - `check_rbf` passes (line 106).
   - `process_rbf` removes `tx_old` from the pool and fires its reject callback (line 136). `tx_old` is now in `conflicts_cache`.
   - `_submit_entry` adds `tx_new` to the pool (line 137). Pool size now exceeds `max_tx_pool_size`.
   - `limit_size` finds `tx_new` is the lowest-fee entry, evicts it, returns `Some(Reject::Full(...))` (lines 150–152).
   - `map_or(Ok(()), Err)?` causes early return with `Err`.
6. **Result**: `tx_old` is permanently gone from the active pool. `tx_new` is also gone. `total_tx_size` and `total_tx_cycles` are understated by `tx_old`'s size and cycles. The `send_transaction` RPC returns an error to the caller, who may retry — potentially repeating the drain.

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }

    pub(crate) fn add_entry(
        &mut self,
        mut entry: TxEntry,
        status: Status,
    ) -> Result<(bool, HashSet<TxEntry>), Reject> {
        let tx_short_id = entry.proposal_short_id();
        let mut evicts = Default::default();
        if self.entries.get_by_id(&tx_short_id).is_some() {
            return Ok((false, evicts));
        }
        let (total_tx_size, total_tx_cycles) =
            self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
        trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
        evicts = self.check_and_record_ancestors(&mut entry)?;
        self.record_entry_edges(&entry)?;
        self.insert_entry(&entry, status);
        self.record_entry_descendants(&entry);
        self.track_entry_statics(None, Some(status));
        self.total_tx_size = total_tx_size;
        self.total_tx_cycles = total_tx_cycles;
        Ok((true, evicts))
    }
