### Title
Unconditional `expect` on `txs_sizes` in `FeeRateCollector::statistics` Panics RPC Thread on Nodes Upgraded from Pre-v0.106 — (`rpc/src/util/fee_rate.rs`)

---

### Summary

`FeeRateCollector::statistics` calls `.expect("expect txs_size's length >= 1")` unconditionally on `block_ext.txs_sizes`, which is `None` for any `BlockExt` stored in the legacy 5-field molecule layout (`BlockExt`, not `BlockExtV1`). Any node upgraded from a pre-v0.106 binary retains those old-format rows in `COLUMN_BLOCK_EXT`. While those rows remain in the tip window (up to 101 blocks after upgrade), an unprivileged caller invoking `get_fee_rate_statistics` triggers the panic.

---

### Finding Description

**The panic site** — `rpc/src/util/fee_rate.rs` line 93:

```rust
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
``` [1](#0-0) 

This line is reached for every `BlockExt` returned by `collect()`, with no prior guard on `txs_sizes`.

**How `None` enters** — `store/src/store.rs` `get_block_ext`, lines 247–263:

```rust
match reader.count_extra_fields() {
    0 => reader.into(),   // ← old 5-field BlockExt: txs_sizes = None
    2 => packed::BlockExtV1Reader::from_slice_should_be_ok(slice.as_ref()).into(),
    _ => { panic!(...) }
}
``` [2](#0-1) 

When `count_extra_fields() == 0` the old `BlockExtReader` is used, whose `Unpack` impl hard-codes `txs_sizes: None`: [3](#0-2) 

**No migration converts old rows** — `util/migrate/src/migrate.rs` lists every registered migration; none rewrites old `BlockExt` rows to `BlockExtV1`. `BlockExt2019ToZero` only zeroes the `cycles` field while leaving the 5-field molecule layout intact: [4](#0-3) [5](#0-4) 

**The molecule schemas confirm the two layouts:** [6](#0-5) 

**The `collect` iterator does not filter `None` txs_sizes** — `filter_map` only drops blocks absent from the DB; old-format blocks are present and return `Some(BlockExt { txs_sizes: None, … })`: [7](#0-6) 

**RPC entry point** — no authentication or privilege check: [8](#0-7) 

---

### Impact Explanation

A panic in the RPC handler thread crashes that thread. Depending on the jsonrpc thread-pool implementation, repeated calls can exhaust the pool or propagate the panic to the server process, causing a node crash or sustained RPC unavailability. The scope target is "Any local RPC API crash," which this satisfies directly.

---

### Likelihood Explanation

Any mainnet or testnet node that ran a pre-v0.106 binary and upgraded retains old-format `BlockExt` rows permanently (no migration rewrites them). During the window of up to 101 blocks after upgrade, the tip window contains those rows. The RPC is unauthenticated by default. The trigger is a single JSON-RPC call with no parameters.

---

### Recommendation

Replace the unconditional `expect` with a graceful skip:

```rust
let txs_sizes = match txs_sizes {
    Some(s) => s,
    None => return fee_rates, // skip blocks without txs_sizes
};
```

This matches the intent of the subsequent guard `if txs_sizes.len() > 1 && !txs_fees.is_empty()` and eliminates the panic for both old-format rows and legitimately empty `BlockExtV1` entries.

---

### Proof of Concept

1. Open a `ChainDB` and write genesis + N blocks whose `COLUMN_BLOCK_EXT` entries are serialized as the 5-field `BlockExt` molecule layout (no `cycles`/`txs_sizes` fields).
2. Set the tip header to block N.
3. Construct a `Snapshot` over that DB and call `FeeRateCollector::new(snapshot.as_ref()).statistics(None)`.
4. Observe the thread panics with `"expect txs_size's length >= 1"`.

The existing test in `store/src/tests/db.rs` already stores a `BlockExt` with `txs_sizes: None` and round-trips it successfully — confirming the DB layer preserves `None` — while the fee-rate collector unconditionally unwraps it. [9](#0-8)
