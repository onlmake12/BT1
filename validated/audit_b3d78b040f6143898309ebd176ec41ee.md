### Title
Snappy Decompression Allocation Bypasses `max_frame_length` Bound in `LengthDelimitedCodecWithCompress::decode()` — (`network/src/compress.rs`)

### Summary
`LengthDelimitedCodecWithCompress::decode()` enforces `max_frame_length` only on the compressed wire bytes, but allocates a buffer sized by the snappy-declared decompressed length (up to `MAX_UNCOMPRESSED_LEN` = 8 MB), which is independent of and larger than `max_frame_length`. Any unprivileged remote peer can trigger up to 4× memory amplification per frame on the Sync protocol.

### Finding Description
In `network/src/compress.rs`, `LengthDelimitedCodecWithCompress::decode()` first calls `self.length_delimited.decode(src)` which enforces `max_frame_length` on the wire frame (2 MB for Sync, 4 MB for RelayV3). [1](#0-0)  However, when the compress flag is set, the code reads the snappy header's declared output size via `decompress_len()` and only rejects if it exceeds `MAX_UNCOMPRESSED_LEN` (8 MB). [3](#0-2)  It then unconditionally allocates `BytesMut::zeroed(decompressed_bytes_len)` — up to 8 MB — before any actual decompression. [4](#0-3)  The `MAX_UNCOMPRESSED_LEN` constant is hardcoded at `1 << 23` (8 MB) and is not tied to `max_frame_length` in any way. [5](#0-4) 

### Impact Explanation
With N simultaneous peers each sending max-size crafted compressed frames on Sync, the node allocates N × 8 MB instead of the expected N × 2 MB. This is a 4× memory amplification for Sync and 2× for RelayV3. Under a large peer count, this degrades node throughput through memory pressure and increased GC/allocator overhead. The `max_frame_length` invariant — that it bounds per-peer memory consumption — is violated.

### Likelihood Explanation
Any unprivileged peer that can establish a P2P connection can trigger this. No PoW, no key, no special role required. The Sync and RelayV3 protocols are open to all peers by default. The attack is sustained (continuous stream of crafted frames) and requires no coordination.

### Recommendation
In `decode()`, after reading `decompressed_bytes_len`, add a second guard: reject if `decompressed_bytes_len > self.length_delimited.max_frame_length()`. This ensures the decompressed allocation is also bounded by `max_frame_length`, restoring the intended invariant. Alternatively, set `MAX_UNCOMPRESSED_LEN` dynamically from `max_frame_length` rather than using a single global constant. [6](#0-5)
