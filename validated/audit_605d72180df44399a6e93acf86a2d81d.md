### Title
Unbounded `recent_reject` Store Growth Due to Missing Counter Increment in `RecentReject::put` — (`tx-pool/src/component/recent_reject.rs`)

### Summary

`RecentReject::put` never increments `self.total_keys_num` in the normal (non-shrink) path. The eviction guard computes a local variable but never writes it back to the struct field, so `shrink()` is never triggered. Any unprivileged caller who can submit transactions that fail with a non-`Duplicated` reject can grow the RocksDB store without bound.

### Finding Description

In `RecentReject::put`, after writing the entry to RocksDB, the code computes a candidate count but never persists it:

```rust
// tx-pool/src/component/recent_reject.rs  lines 62-69
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

`total_keys_num` here is a **local binding** — the result of `checked_add(1)`. `self.total_keys_num` is never assigned the new value. On a fresh database `self.total_keys_num` starts at `0`; every subsequent call evaluates `0 + 1 > count_limit`, which is `false` for any `count_limit ≥ 1`. `shrink()` is therefore never called. [1](#0-0) 

`shrink()` is the **only** place that updates `self.total_keys_num` (line 111), so the in-memory counter is permanently stale and the guard is permanently disabled. [2](#0-1) 

`should_recorded()` returns `true` for every `Reject` variant except `Reject::Duplicated`: [3](#0-2) 

Both the callback path in `shared/src/shared_builder.rs` and the async path in `tx-pool/src/process.rs` call `put` whenever `should_recorded()` is true: [4](#0-3) [5](#0-4) 

### Impact Explanation

An attacker submits N distinct transactions (unique tx hashes) that each fail with a recordable reject — for example `Reject::Resolve(OutPointError::Unknown)` (referencing a non-existent cell dep, which is **not** classified as malformed, so no peer ban occurs). Each rejection is written to the RocksDB store. Because `shrink()` is never triggered, the store grows without bound, leading to:

- Unbounded disk consumption on the node
- Degraded `get_tx_status` / `get_transaction` RPC query performance as the store grows
- Eviction of legitimate entries never occurs, violating the documented `keep_rejected_tx_hashes_count` invariant

The existing unit test `test_basic` passes trivially because it asserts `total_keys_num < 100` after 160 inserts — but `total_keys_num` is still `0` (never incremented), so the assertion is vacuously true and does not validate eviction: [6](#0-5) 

### Likelihood Explanation

- Reachable via RPC `send_transaction` (no peer banning, no rate limit by default) and via P2P relay using `Reject::Resolve(OutPointError::Unknown)` rejects (not malformed, no ban).
- No special privileges required.
- Cost is low: each transaction only needs to be structurally valid and have a unique hash; fees can be zero if `min_fee_rate = 0`.
- The TTL mechanism (`keep_rejected_tx_hashes_days`, default 1 day) provides only passive mitigation — RocksDB TTL removes entries only during compaction, not on write. [7](#0-6) 

### Recommendation

In `RecentReject::put`, assign the incremented value back to `self.total_keys_num` in the non-shrink branch:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;   // ← missing assignment
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

### Proof of Concept

1. Start a CKB node with default config (`keep_rejected_tx_hashes_count = 1000`, `min_fee_rate = 0`).
2. Submit 10 000 transactions via RPC, each referencing a distinct non-existent cell dep (unique `out_point` → `Reject::Resolve(OutPointError::Unknown)`, `should_recorded() = true`).
3. Observe that `RecentReject::total_keys_num` remains `0` throughout and `shrink()` is never called.
4. Measure the RocksDB directory size — it grows proportionally to N with no eviction.
5. Assert that `recent_reject.get_estimate_total_keys_num()` returns `0` even after 10 000 inserts, confirming the counter is broken. [8](#0-7)
