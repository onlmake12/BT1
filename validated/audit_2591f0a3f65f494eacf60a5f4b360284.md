Audit Report

## Title
Out-of-Bounds Index Panic in `TxConfirmStat::remove_unconfirmed_tx` When `tx_age > 1000` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary
In `TxConfirmStat::remove_unconfirmed_tx`, a bounds guard protects `block_unconfirmed_txs` from out-of-bounds access when `tx_age >= 1000`, but no equivalent guard exists before the `confirm_blocks_to_failed_txs[tx_age - 1]` access. When a tracked transaction has been in the mempool for more than 1000 blocks and is then evicted via `reject_tx`, Rust panics with an index-out-of-bounds error, crashing the node process.

## Finding Description
`confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` are both initialized with exactly `MAX_CONFIRM_BLOCKS = 1000` entries (lines 139–141). In `remove_unconfirmed_tx`, the guard at line 208 correctly routes transactions with `tx_age >= self.block_unconfirmed_txs.len()` (i.e., `>= 1000`) to the `old_unconfirmed_txs` counter. However, the `count_failure` block at lines 214–216 has no such guard: `self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64`. When `tx_age == 1000`, index `999` is valid. When `tx_age >= 1001`, index `tx_age - 1 >= 1000` exceeds the vector length of 1000, causing a Rust index-out-of-bounds panic. The reachable call chain is: `FeeEstimator::reject_tx` (mod.rs:87) → `Algorithm::reject_tx` (line 475) → `drop_tx` (line 428) → `drop_tx_inner(tx_hash, true)` (line 416) → `remove_unconfirmed_tx(..., count_failure=true)` (line 418). This path is only active for the `ConfirmationFraction` variant. The existing guard for `block_unconfirmed_txs` is entirely absent for `confirm_blocks_to_failed_txs`, making the asymmetry the direct root cause.

## Impact Explanation
A Rust index-out-of-bounds panic in `remove_unconfirmed_tx` will unwind the thread. Because `reject_tx` is called while holding a write lock (`algo.write().reject_tx(tx_hash)`), the panic poisons the `RwLock`, making the fee estimator permanently inaccessible for the lifetime of the node process. This constitutes a node crash matching the allowed impact: **High (10001–15000 points) — Vulnerability which could easily crash a CKB node**.

## Likelihood Explanation
The preconditions are fully attainable by an unprivileged attacker: (1) submit a low-fee transaction via RPC or P2P relay — no special privilege required; (2) the transaction is tracked by the estimator because `track_tx` records txs where `height == best_height`, the normal case for any freshly submitted tx; (3) the chain advances more than 1000 blocks without confirming the tx (deliberate fee underpricing ensures this); (4) pool eviction (`limit_size` or RBF replacement) triggers `reject_tx` with `count_failure=true`. The attack is repeatable and requires no privileged access. The only constraint is that the `ConfirmationFraction` algorithm must be configured.

## Recommendation
Add a bounds check before the `confirm_blocks_to_failed_txs` access, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This is consistent with the existing design: transactions older than `MAX_CONFIRM_BLOCKS` are already classified as "old" via `old_unconfirmed_txs`, and their failure should be silently dropped rather than recorded in the fixed-size array.

## Proof of Concept

```rust
#[test]
fn test_remove_unconfirmed_tx_oob_panic() {
    let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
    let mut stat = TxConfirmStat::new(buckets, 1000, 0.993);

    // Track a tx at height 0
    let bucket_index = stat.add_unconfirmed_tx(0, FeeRate::from_u64(1000)).unwrap();

    // tx_age = 1001 - 0 = 1001 > 1000 = confirm_blocks_to_failed_txs.len()
    // confirm_blocks_to_failed_txs[1001 - 1] = confirm_blocks_to_failed_txs[1000] → PANIC
    stat.remove_unconfirmed_tx(0, 1001, bucket_index, true);
}
```

Expected output against unpatched code:
```
thread 'test' panicked at 'index out of bounds: the len is 1000 but the index is 1000'
```
