Audit Report

## Title
Unauthenticated `pending_delivered` Poisoning via Spoofed `ConnectionRequest.from` Enables Unbounded Background Task Spawning and Attacker-Directed Outbound TCP Connections — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary

`ConnectionRequestProcess::respond_delivered` accepts `content.from` verbatim from the wire and writes it into `pending_delivered` without verifying it matches the authenticated session peer ID. A single attacker session can rotate the spoofed `from` value to bypass both the `forward_rate_limiter` and the `HOLE_PUNCHING_INTERVAL` deduplication guard, then trigger `ConnectionSync` to spawn unbounded background tasks each making ~150 outbound TCP connections over 30 seconds, exhausting file descriptors and crashing the node.

## Finding Description

**Root cause — no `from` ↔ session binding in `respond_delivered`:**

`ConnectionRequestProcess::execute()` passes `content.from` directly from the wire to `respond_delivered`: [1](#0-0) 

Inside `respond_delivered`, the attacker-supplied `from_peer_id` and `remote_listens` are written directly into `pending_delivered` with no check that `from_peer_id` equals the authenticated peer ID of the sending session: [2](#0-1) 

The `ConnectionRequestProcess` struct holds `peer: PeerIndex` (the authenticated session index) but it is never consulted to validate `content.from`: [3](#0-2) 

**Root cause — no `from` ↔ session binding in `ConnectionSyncProcess`:**

`ConnectionSyncProcess` does not even hold a `peer` field — the session ID is not passed to its constructor: [4](#0-3) 

It reads `content.from` from the wire and looks it up in `pending_delivered` without any session verification: [5](#0-4) 

The retrieved addresses are passed directly to `try_nat_traversal` inside a spawned background task: [6](#0-5) [7](#0-6) 

**`try_nat_traversal` makes real outbound TCP connections:**

The function creates a new TCP socket and calls `connect()` every ~200 ms for up to 30 seconds (~150 attempts per invocation): [8](#0-7) 

**Rate-limiter bypass:**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` — all attacker-controlled wire values: [9](#0-8) [10](#0-9) 

Rotating `from` to a fresh random `PeerId` on each pair yields a fresh bucket, bypassing the 1 req/s per `(from, to)` limit entirely. The only binding constraint is the per-session `rate_limiter` at 30 msg/s per message type: [11](#0-10) [12](#0-11) 

The `HOLE_PUNCHING_INTERVAL` deduplication check in `respond_delivered` is also keyed by `from_peer_id` and is equally bypassed by rotating `from`: [13](#0-12) 

**Unbounded task accumulation:**

At 30 msg/s per message type, the attacker sends 30 `(ConnectionRequest, ConnectionSync)` pairs per second. Each pair spawns one background task lasting 30 seconds. After 30 seconds of sustained attack: 30 tasks/s × 30 s = 900 concurrent background tasks, each creating a new TCP socket every ~200 ms. This exhausts the process's file descriptor limit and crashes the node.

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

A single attacker session sustains 30 new background tasks per second indefinitely. Each task holds open a TCP socket for up to 30 seconds and creates a new one every ~200 ms. The resulting file descriptor exhaustion causes the node process to fail on any subsequent socket allocation (including normal P2P connections), crashing the node. Additionally, the victim node is directed to make outbound TCP connections to arbitrary IP:port pairs supplied by the attacker, constituting attacker-directed port scanning and potential internal network probing.

## Likelihood Explanation

- Requires only a single outbound TCP connection to any CKB node with hole-punching enabled.
- No cryptographic material, no privileged role, no hashpower required.
- Both messages are structurally valid and pass all existing validation checks.
- The rate-limiter bypass via `from` rotation is trivial: generate random bytes of the correct length for a valid `PeerId`.
- Locally reproducible against a single node.

## Recommendation

In `respond_delivered`, resolve the authenticated `PeerId` for `self.peer` from the peer registry and assert it equals `from_peer_id` before writing to `pending_delivered`:

```rust
// In ConnectionRequestProcess::respond_delivered:
let session_peer_id = self.protocol.network_state
    .peer_registry
    .read()
    .get_key_by_peer_id_from_session(self.peer);

if session_peer_id.as_ref() != Some(&from_peer_id) {
    return StatusCode::InvalidFromPeerId
        .with_context("from peer id does not match session peer id");
}
```

Pass the session `PeerIndex` into `ConnectionSyncProcess` and apply the same check in `execute()` — verify `content.from` matches the sending session's authenticated peer ID before consulting `pending_delivered`. This eliminates the spoofing vector entirely and makes the rate limiter effective.

## Proof of Concept

```
1. Start node X with hole-punching enabled and at least one listen address configured.
2. Connect attacker session A (peer_id = attacker_id) to X.
3. In a loop (up to 30 req/s per message type):
   a. Generate a fresh random victim_id (valid PeerId bytes).
   b. Send: ConnectionRequest { from: victim_id, to: X_peer_id, listen_addrs: [attacker_ip:1234] }
      → forward_rate_limiter passes (fresh (victim_id, X_peer_id, item_id) bucket)
      → HOLE_PUNCHING_INTERVAL check passes (victim_id not in pending_delivered)
      → respond_delivered writes pending_delivered[victim_id] = ([attacker_ip:1234], now)
   c. Send: ConnectionSync { from: victim_id, to: X_peer_id, route: [] }
      → forward_rate_limiter passes (fresh bucket)
      → pending_delivered.get(victim_id) → Some([attacker_ip:1234])
      → runtime::spawn(try_nat_traversal(...)) — background task created
      → background task makes TCP SYN to attacker_ip:1234 every ~200ms for 30s
4. After ~30 seconds: ~900 concurrent background tasks each holding/creating TCP sockets.
5. Observe: node X fails to open new sockets → crashes or becomes unresponsive.

Invariant violated:
  pending_delivered[victim_id] must only be populated when the session
  that sent the ConnectionRequest has session_peer_id == victim_id.
  Step 3b populates it with attacker_id session but victim_id key — invariant broken.
```
