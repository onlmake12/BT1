### Title
Missing `is_verified` Check in `advance` Instruction Allows Unverified Publisher Caps to Corrupt Pool Reward Distribution - (File: integrity-pool program / publisher-caps program)

### Summary

The `advance` instruction in the integrity pool program consumes a `publisher_caps` account to update pool reward state. The `PublisherCaps` struct contains an `is_verified: u8` flag that is only set to `1` after a successful Wormhole VAA proof via `verify_publisher_caps`. However, the `advance` instruction does not check `is_verified == 1` before reading and applying the publisher caps data. An unprivileged attacker can create a `PublisherCaps` account, write arbitrary data into it, and call `advance` with the unverified account, corrupting the pool's reward distribution state.

This is a direct structural analog to the reported airdrop vulnerability: just as `airdrop_claim_handler` was missing a `finalized == 1` check, `advance` is missing an `is_verified == 1` check.

---

### Finding Description

The `PublisherCaps` account is a two-phase construct:

1. **Phase 1 (unverified):** `init_publisher_caps` creates the account with `is_verified = 0`. `write_publisher_caps` allows the `write_authority` (stored in the account itself, set by the creator) to write arbitrary bytes into `publisher_caps_message_buffer`.
2. **Phase 2 (verified):** `verify_publisher_caps` validates the buffer contents against a Wormhole-attested VAA and sets `is_verified = 1`.

The `advance` instruction takes `publisher_caps` as a read-only account with no PDA constraint and no `relations` binding to a trusted source. The integrity pool IDL error list contains `outdatedPublisherCaps` (code 6007), which checks staleness, but there is no corresponding `unverifiedPublisherCaps` error ã confirming the `is_verified` flag is never checked.

The `write_publisher_caps` instruction only requires the `write_authority` signer, and `write_authority` is set by whoever calls `init_publisher_caps`. This means any unprivileged user can:

1. Call `init_publisher_caps` ã creates a `PublisherCaps` account with themselves as `write_authority` and `is_verified = 0`.
2. Call `write_publisher_caps` ã writes attacker-controlled bytes into `publisher_caps_message_buffer`.
3. Call `advance` ã passes the unverified account; no `is_verified` check blocks execution.

The pool data (`last_updated_epoch`, `del_state`, `self_del_state`, `claimable_rewards`, `delegation_fees`) is then updated using attacker-controlled publisher cap values.

### Impact Explanation

An attacker can inject arbitrary publisher cap values into the pool's epoch advance step. This corrupts `claimable_rewards`, `del_state`, `self_del_state`, and `delegation_fees` for all publishers in the pool. Consequences include:

- Inflating or zeroing out reward allocations for specific publishers.
- Disrupting the integrity of the OIS reward epoch, causing legitimate stakers and publishers to receive incorrect rewards or none at all.
- Potentially draining the `pool_reward_custody` token account by inflating `claimable_rewards` to the maximum, then claiming via `advance_delegation_record`.

This is a direct loss-of-funds / reward manipulation impact on the on-chain staking protocol.

---

### Likelihood Explanation

The attack requires no privileged access. `init_publisher_caps` and `write_publisher_caps` are permissionless for any signer. The `advance` instruction is also permissionless (payer is any signer). The only constraint is that the attacker must craft a `publisher_caps_message_buffer` that passes the `outdatedPublisherCaps` staleness check (i.e., contains a recent-enough epoch marker), which is fully attacker-controlled since the buffer is written by the attacker. Likelihood is **high**.

---

### Recommendation

In the `advance` instruction handler, add an explicit check before consuming `publisher_caps`:

```rust
require!(
    publisher_caps.is_verified == 1,
    IntegrityPoolError::PublisherCapsNotVerified
);
```

This mirrors the remediation applied to the airdrop vulnerability:

```rust
if target.finalized != 1 {
    return Err(ErrorCode::AirdropNotFinal);
}
```

Additionally, consider binding the `publisher_caps` account in the `advance` instruction via a PDA or a `relations` constraint to a trusted registry, preventing substitution of arbitrary accounts entirely.

---

### Proof of Concept

1. Attacker calls `init_publisher_caps` ã creates `PublisherCaps` with `write_authority = attacker`, `is_verified = 0`.
2. Attacker calls `write_publisher_caps(index=0, data=<crafted_bytes>)` ã writes a malicious publisher caps message buffer with inflated cap values for a target publisher and a recent epoch timestamp to pass the staleness check.
3. Attacker calls `advance(publisher_caps = attacker_caps_account)` ã the instruction reads the unverified buffer, parses attacker-controlled cap values, and updates `pool_data.claimable_rewards` and `del_state` accordingly.
4. Attacker (or colluding publisher) calls `advance_delegation_record` to claim the inflated rewards from `pool_reward_custody`.

The critical missing guard is the absence of `require!(publisher_caps.is_verified == 1, ...)` in the `advance` handler, directly analogous to the missing `finalized != 1` guard in `airdrop_claim_handler`.
