### Title
Incorrect Cycle Accumulation in `resume_from_state` Uses `consumed_cycles` Instead of `used_cycles` for Subsequent Script Groups — (`script/src/verify.rs`)

---

### Summary

In `TransactionScriptsVerifier::resume_from_state`, the loop that processes script groups *after* the resumed group accumulates `consumed_cycles` into the `cycles` return value instead of `used_cycles`. The parallel function `resumable_verify` correctly accumulates `used_cycles` into `cycles` and `consumed_cycles` into the limit-tracking counter. The discrepancy means that any transaction whose verification is suspended and then resumed will return a different total cycle count than a single-pass verification of the same transaction, breaking the cycle-accounting invariant used for tx-pool admission and block assembly.

---

### Finding Description

`resumable_verify` and `resume_from_state` both call `verify_group_with_chunk`, which returns `ChunkState::Completed(used_cycles, consumed_cycles)` — two distinct values.

In `resumable_verify` the accounting is:

```rust
// script/src/verify.rs  lines 242-245
Ok(ChunkState::Completed(used_cycles, consumed_cycles)) => {
    current_consumed_cycles =
        wrapping_cycles_add(current_consumed_cycles, consumed_cycles, group)?;
    cycles = wrapping_cycles_add(cycles, used_cycles, group)?;   // ← used_cycles
}
```

`cycles` (the value ultimately returned and stored in the tx-pool) accumulates `used_cycles`. `current_consumed_cycles` (the limit-checking counter) accumulates `consumed_cycles`.

In `resume_from_state`, the resumed group is handled correctly:

```rust
// script/src/verify.rs  lines 299-302
Ok(ChunkState::Completed(used_cycles, consumed_cycles)) => {
    current_used = wrapping_cycles_add(current_used, consumed_cycles, current_group)?;
    cycles = wrapping_cycles_add(cycles, used_cycles, current_group)?;  // ← used_cycles ✓
}
```

But the loop over *subsequent* groups (line 314 onward) discards `used_cycles` and uses `consumed_cycles` for both counters:

```rust
// script/src/verify.rs  lines 322-325
Ok(ChunkState::Completed(_, consumed_cycles)) => {          // ← used_cycles discarded
    current_used = wrapping_cycles_add(current_used, consumed_cycles, group)?;
    cycles = wrapping_cycles_add(cycles, consumed_cycles, group)?;  // ← WRONG: should be used_cycles
}
```

The `_` silently drops `used_cycles`, and `consumed_cycles` is used for `cycles` instead. This is structurally identical to the Yeti Finance bug: the wrong variable (total/consumed) is used where the incremental/actual variable (`used_cycles`) is required. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

The `cycles` value returned by `resume_from_state` is stored in the tx-pool entry and used for:

1. **Tx-pool admission** — `total_tx_cycles` is updated with the entry's cycles; the pool enforces `max_tx_pool_size` and cycle limits based on this value.
2. **Block assembly** — `TxSelector::txs_to_commit` filters entries by `ancestors_cycles <= cycles_limit`.
3. **Block validation** — `BlockTxsVerifier` checks that the block's total cycles do not exceed `max_block_cycles`. [4](#0-3) [5](#0-4) 

If `consumed_cycles < used_cycles` for any subsequent group, the resumed path **under-counts** cycles. A transaction whose true cycle cost exceeds `max_block_cycles` could be admitted to the pool and assembled into a block template with a falsely low cycle count. Nodes that re-verify the block from scratch (using `resumable_verify`) will compute the correct, higher cycle total and reject the block, causing a **consensus split**: nodes that accepted the block (having used the resumed path) diverge from nodes that rejected it.

If `consumed_cycles > used_cycles`, the resumed path **over-counts** cycles, causing valid transactions to be incorrectly evicted or rejected from the pool.

---

### Likelihood Explanation

The chunked/resumable verification path is triggered whenever a transaction's script execution cannot complete within a single `limit_cycles` budget. This is a normal operational path for large or complex scripts. Any transaction with **two or more script groups** where the first group causes a suspension will exercise the buggy loop in `resume_from_state`. An unprivileged tx-pool submitter can craft such a transaction and submit it via the standard `send_transaction` RPC or the P2P relay protocol. No privileged access is required. [6](#0-5) [7](#0-6) 

---

### Recommendation

In the subsequent-groups loop inside `resume_from_state`, bind `used_cycles` instead of discarding it, and accumulate it into `cycles`:

```rust
// Before (buggy):
Ok(ChunkState::Completed(_, consumed_cycles)) => {
    current_used = wrapping_cycles_add(current_used, consumed_cycles, group)?;
    cycles = wrapping_cycles_add(cycles, consumed_cycles, group)?;
}

// After (correct, matching resumable_verify):
Ok(ChunkState::Completed(used_cycles, consumed_cycles)) => {
    current_used = wrapping_cycles_add(current_used, consumed_cycles, group)?;
    cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
}
```

Add a test that verifies `resume_from_state` returns the same cycle count as `resumable_verify` for a transaction with multiple script groups that triggers suspension in the first group.

---

### Proof of Concept

1. Construct a transaction with two lock script groups: group A (large, causes suspension at `step_cycles`) and group B (small).
2. Submit the transaction to a node configured with `max_tx_verify_cycles = step_cycles`.
3. The node calls `resumable_verify(step_cycles)` → suspended after group A.
4. The node calls `resume_from_state(&state, full_limit)`.
   - Group A (resumed): `cycles += used_cycles_A` ✓
   - Group B (subsequent loop): `cycles += consumed_cycles_B` ✗ (should be `used_cycles_B`)
5. If `consumed_cycles_B ≠ used_cycles_B`, the stored entry cycles differ from the true cost.
6. A miner assembles a block using the under-counted entry; the block passes the miner's local cycle check but fails re-verification on other nodes, causing a fork. [8](#0-7) [9](#0-8)

        let mut cycles = 0;
        let mut current_consumed_cycles = 0;

        let groups: Vec<_> = self.groups().collect();
        for (idx, (_hash, group)) in groups.iter().enumerate() {
            // vm should early return invalid cycles
            let remain_cycles = limit_cycles
                .checked_sub(current_consumed_cycles)
                .ok_or_else(|| {
                    ScriptError::Other(format!("expect invalid cycles {limit_cycles} {cycles}"))
                        .source(group)
                })?;

            match self.verify_group_with_chunk(group, remain_cycles, &None) {
                Ok(ChunkState::Completed(used_cycles, consumed_cycles)) => {
                    current_consumed_cycles =
                        wrapping_cycles_add(current_consumed_cycles, consumed_cycles, group)?;
                    cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
                }
                Ok(ChunkState::Suspended(state)) => {
                    let current = idx;
                    let state = TransactionState::new(state, current, cycles, remain_cycles);

                    #[cfg(feature = "logging")]
                    logging::on_script_error(_hash, &self.hash(), &e);
                    return Err(e.source(group).into());
                }
            }
        }

        Ok(VerifyResult::Completed(cycles))
    }

        &self,
        state: &TransactionState,
        limit_cycles: Cycle,
    ) -> Result<VerifyResult, Error> {
        let TransactionState {
            current,
            state,
            current_cycles,
            ..
        } = state;

        let mut current_used = 0;
        let mut cycles = *current_cycles;

        let (_hash, current_group) = self.groups().nth(*current).ok_or_else(|| {
            ScriptError::Other(format!("snapshot group missing {current:?}")).unknown_source()
        })?;

        let resumed_script_result =
            self.verify_group_with_chunk(current_group, limit_cycles, state);

        match resumed_script_result {
            Ok(ChunkState::Completed(used_cycles, consumed_cycles)) => {
                current_used = wrapping_cycles_add(current_used, consumed_cycles, current_group)?;
                cycles = wrapping_cycles_add(cycles, used_cycles, current_group)?;
            }
            Ok(ChunkState::Suspended(state)) => {
                let state = TransactionState::new(state, *current, cycles, limit_cycles);
                return Ok(VerifyResult::Suspended(state));
            }
            Err(e) => {
                #[cfg(feature = "logging")]
                logging::on_script_error(_hash, &self.hash(), &e);
                return Err(e.source(current_group).into());
            }
        }

        for (idx, (_hash, group)) in self.groups().enumerate().skip(current + 1) {
            let remain_cycles = limit_cycles.checked_sub(current_used).ok_or_else(|| {
                ScriptError::Other(format!(
                    "expect invalid cycles {limit_cycles} {current_used} {cycles}"
                ))
                .source(group)
            })?;

            match self.verify_group_with_chunk(group, remain_cycles, &None) {
                Ok(ChunkState::Completed(_, consumed_cycles)) => {
                    current_used = wrapping_cycles_add(current_used, consumed_cycles, group)?;
                    cycles = wrapping_cycles_add(cycles, consumed_cycles, group)?;
                }
                Ok(ChunkState::Suspended(state)) => {
                    let current = idx;
                    let state = TransactionState::new(state, current, cycles, remain_cycles);
                    return Ok(VerifyResult::Suspended(state));
                }
                Err(e) => {
                    #[cfg(feature = "logging")]
                    logging::on_script_error(_hash, &self.hash(), &e);
                    return Err(e.source(group).into());
                }
            }
        }

        Ok(VerifyResult::Completed(cycles))
    }
