### Title
`undelegate` Blocked by Insufficient `pool_reward_custody` Balance When `advance_delegation_record` Is Required — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `undelegate` instruction in the Pyth integrity pool program enforces that a delegator's accounting is current (error `OutdatedDelegatorAccounting`, code 6008) before allowing principal withdrawal. Bringing the accounting current requires calling `advance_delegation_record`, which transfers accrued rewards from the on-chain `pool_reward_custody` PDA to the user's `stake_account_custody`. If `pool_reward_custody` holds fewer tokens than the computed reward payout, `advance_delegation_record` reverts with `InsufficientRewards` (code 6013), making it impossible to satisfy the precondition for `undelegate`. Users are therefore unable to withdraw their staked principal even though it is held in their own `stake_account_custody`.

---

### Finding Description

The integrity pool program exposes two coupled instructions:

**`advance_delegation_record`** — permissionless, callable by any payer. It reads `pool_data.claimable_rewards`, computes the delegator's share of rewards since `delegation_record.last_epoch`, and transfers that amount from `pool_reward_custody` (a PDA-controlled ATA) to `stake_account_custody`. The instruction requires `pool_reward_custody` as a writable account and will revert with `InsufficientRewards` if the custody balance is less than the computed payout. [1](#0-0) [2](#0-1) 

**`undelegate`** — the principal withdrawal instruction. It includes `delegation_record` as a writable account and enforces that the delegator's accounting is up to date, reverting with `OutdatedDelegatorAccounting` (code 6008) if `delegation_record.last_epoch` is behind the current epoch. [3](#0-2) [4](#0-3) 

The `pool_reward_custody` is funded externally (by the `reward_program_authority`) and is not automatically replenished. The `PoolData.claimable_rewards` field accumulates as the `advance` instruction runs each epoch. If the authority fails to top up `pool_reward_custody` in proportion to `claimable_rewards`, the balance can fall below what `advance_delegation_record` needs to pay out. [5](#0-4) 

The TypeScript SDK confirms the mandatory sequencing: `advanceDelegationRecord` is always called before `unstakeFromPublisher` / `undelegate`. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any staker who has accumulated unclaimed rewards and whose `delegation_record.last_epoch` is behind the current epoch cannot call `undelegate` until `advance_delegation_record` succeeds. If `pool_reward_custody` is underfunded, `advance_delegation_record` reverts with `InsufficientRewards`, and the staker's principal — held in their own `stake_account_custody` — is effectively locked with no alternative withdrawal path. The staker cannot bypass the check because `undelegate` enforces `OutdatedDelegatorAccounting` at the program level. [8](#0-7) 

---

### Likelihood Explanation

The reward rate `y` is currently set to 0 per OP-PIP-103, meaning no rewards accrue and `advance_delegation_record` transfers 0 tokens, so the bug is dormant today. However, `y` is a governance-updatable parameter. Once `y > 0`, `claimable_rewards` will grow each epoch. If the `reward_program_authority` does not fund `pool_reward_custody` proportionally — whether due to operational oversight, key loss, or deliberate action — the custody balance will eventually fall short of the computed payout, triggering `InsufficientRewards` and blocking all undelegations for affected stakers. The longer `y > 0` runs without adequate funding, the larger the gap becomes. [9](#0-8) [10](#0-9) 

---

### Recommendation

Decouple the `OutdatedDelegatorAccounting` enforcement from reward payment. Specifically:

1. Allow `undelegate` to proceed even when `delegation_record.last_epoch < current_epoch`, recording the outstanding reward debt without requiring immediate payment; **or**
2. Allow `advance_delegation_record` to succeed with a partial or zero payout when `pool_reward_custody` is underfunded, updating `delegation_record.last_epoch` to the current epoch and recording the unpaid amount as a debt to be settled later; **or**
3. Provide a separate `undelegate_without_claim` instruction that closes the position and updates `delegation_record.last_epoch` without transferring rewards, preserving the reward claim for later settlement.

---

### Proof of Concept

1. Governance sets `y > 0`. Over several epochs, `pool_data.claimable_rewards` grows.
2. `reward_program_authority` fails to fund `pool_reward_custody` with sufficient PYTH tokens.
3. Staker Alice calls `advance_delegation_record` for publisher P. The instruction computes Alice's reward share as `R > pool_reward_custody.amount`. The token transfer fails → instruction reverts with `InsufficientRewards` (code 6013).
4. Alice's `delegation_record.last_epoch` remains behind the current epoch.
5. Alice calls `undelegate(position_index, amount)`. The program checks `delegation_record.last_epoch == current_epoch` → fails with `OutdatedDelegatorAccounting` (code 6008).
6. Alice cannot recover her staked principal. There is no alternative instruction path that bypasses step 5. [8](#0-7) [2](#0-1) [4](#0-3)

  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }
