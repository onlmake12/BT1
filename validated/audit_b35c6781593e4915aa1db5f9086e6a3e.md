### Title
Improper `MAX_DEPOSIT_LIMIT` Enforcement in `addFunds()` Allows Permanent Subscription Balance to Exceed Cap - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `addFunds()` function in `Scheduler.sol` checks only whether the *incoming deposit alone* (`msg.value`) exceeds `MAX_DEPOSIT_LIMIT`, rather than whether the *resulting total balance* (`status.balanceInWei + msg.value`) would exceed the cap. Any unprivileged caller can therefore push a permanent subscription's balance arbitrarily above the 100 ETH ceiling by making multiple deposits each individually below the limit.

---

### Finding Description

`MAX_DEPOSIT_LIMIT` (100 ETH) is defined in `SchedulerConstants.sol` as the maximum balance a permanent subscription may hold. Permanent subscriptions are irreversible — `withdrawFunds()` unconditionally reverts for them — so the cap is the only mechanism preventing unbounded ETH from being permanently locked.

In `createSubscription()` the guard is correct because the initial balance equals `msg.value`:

```solidity
// line 48 – correct for creation: balance starts at msg.value
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

But in `addFunds()` the same pattern is reused without accounting for the pre-existing balance:

```solidity
// line 613 – WRONG: checks only the new deposit, not the running total
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;   // line 617 – balance now exceeds cap
```

The correct guard should be:
```solidity
if (params.isPermanent &&
    status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Permanent subscriptions cannot be withdrawn from:

```solidity
// withdrawFunds – line 640
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [3](#0-2) 

Any ETH deposited into a permanent subscription is locked forever in the contract. The `MAX_DEPOSIT_LIMIT` is the sole safeguard against excessive permanent ETH lock-up. By bypassing it, an attacker (or any user) can:

1. Lock an arbitrary amount of ETH permanently in the contract beyond the intended 100 ETH ceiling.
2. Cause the contract to hold far more ETH than governance/design intended, which is a direct violation of the protocol's deposit-cap invariant.

---

### Likelihood Explanation

The entry path requires no privilege — `addFunds()` has no access control and is callable by anyone:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
``` [4](#0-3) 

The exploit is trivially reproducible: create a permanent subscription with exactly `MAX_DEPOSIT_LIMIT` (100 ETH), then call `addFunds` with any amount ≤ 100 ETH. Each such call passes the flawed guard while incrementing `balanceInWei` beyond the cap. No special knowledge, leaked keys, or external oracle manipulation is required.

---

### Recommendation

Replace the per-deposit check in `addFunds()` with a post-addition total check:

```solidity
if (params.isPermanent &&
    status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [1](#0-0) 

---

### Proof of Concept

```solidity
// 1. Create a permanent subscription funded at exactly the cap
SchedulerStructs.SubscriptionParams memory params = ...;
params.isPermanent = true;
uint256 subId = scheduler.createSubscription{value: 100 ether}(params);
// balanceInWei == 100 ether  ✓  (at cap)

// 2. Add any amount ≤ MAX_DEPOSIT_LIMIT — check passes, balance exceeds cap
scheduler.addFunds{value: 50 ether}(subId);
// msg.value (50 ETH) > MAX_DEPOSIT_LIMIT (100 ETH)? → false → no revert
// balanceInWei == 150 ether  ✗  (50 ETH above cap, permanently locked)

// 3. Repeat to lock arbitrarily large amounts
scheduler.addFunds{value: 100 ether}(subId);
// balanceInWei == 250 ether  ✗
```

The `MAX_DEPOSIT_LIMIT` invariant is fully bypassed through repeated deposits each individually below the threshold, directly mirroring the EtherCollateral pattern where the pre-operation state was checked instead of the post-operation total. [5](#0-4) [2](#0-1)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {

                uint8(params.priceIds.length)
            );

                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
