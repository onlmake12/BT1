### Title
Unauthenticated `from` Field in `ConnectionRequest` Enables `pending_delivered` Poisoning and Sustained Hole-Punching DoS — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

Any connected peer can send a `ConnectionRequest` with an arbitrary `from` peer ID. The victim node stores the attacker-supplied `listen_addrs` in `pending_delivered` under the spoofed peer ID. Because the only re-insertion guard is a 2-minute cooldown keyed on `from_peer_id`, the attacker can refresh the poisoned entry every 2 minutes indefinitely. This blocks legitimate hole-punching for the targeted peer ID and, when a `ConnectionSync` is eventually processed, causes the victim to attempt NAT traversal to attacker-controlled TCP endpoints.

---

### Finding Description

**Root cause — no authentication of `from`:**

The `from` field is parsed from raw bytes with no cryptographic proof that the sender owns that peer ID: [1](#0-0) 

**Attacker-controlled addresses stored unconditionally:**

`respond_delivered` is reached whenever `self_peer_id == &content.to`. The only guard before writing to `pending_delivered` is a timestamp check: [2](#0-1) 

If no entry exists, or the existing entry is older than `HOLE_PUNCHING_INTERVAL` (2 minutes), the attacker's `remote_listens` are written: [3](#0-2) 

**`ConnectionSync` consumes the poisoned entry for NAT traversal:**

When the victim later receives a `ConnectionSync` with `from=legitimate_peer_id`, it reads `pending_delivered[from]` and dials those addresses: [4](#0-3) [5](#0-4) 

**`HOLE_PUNCHING_INTERVAL` and `TIMEOUT` constants:** [6](#0-5) 

---

### Impact Explanation

**Scenario A — Pre-emptive DoS (no race needed):**
The attacker sends a spoofed `ConnectionRequest` with `from=A`, `to=victim` before peer A does. The victim stores the poisoned entry. When A's legitimate `ConnectionRequest` arrives within 2 minutes, the `now - t < HOLE_PUNCHING_INTERVAL` check fires and returns `Ignore` — A never receives `ConnectionRequestDelivered`, never sends `ConnectionSync`, and hole-punching fails entirely.

**Scenario B — Sustained address poisoning:**
The attacker waits until the legitimate entry is ≥2 minutes old, then overwrites it. A's retry `ConnectionRequest` (which arrives shortly after) is blocked for another 2 minutes by the freshly-written attacker entry. The attacker repeats every 2 minutes, maintaining the poisoned state indefinitely. Any `ConnectionSync` that does reach the victim triggers `try_nat_traversal` against attacker-controlled TCP endpoints for up to 30 seconds per attempt: [7](#0-6) 

The P2P handshake will reject the connection (peer ID mismatch), but the victim has already made outbound TCP connections to attacker IPs, disclosing its source IP and wasting socket/thread resources.

---

### Likelihood Explanation

- Requires only a standard P2P connection — no elevated privilege.
- The target peer ID is publicly observable from the P2P network.
- The `forward_rate_limiter` (1 req/sec per `(from, to, item_id)` tuple) does not prevent the attack; the attacker only needs one successful insertion per 2-minute window.
- The per-session `rate_limiter` (30 req/sec) is also not a meaningful barrier.
- The attack is fully automatable and can target any peer ID the attacker wishes to block from hole-punching through the victim.

---

### Recommendation

1. **Verify sender identity**: Only accept a `ConnectionRequest` as originating from `from_peer_id` if the message arrives on a session whose authenticated peer ID matches `from`. If the session peer ID does not match `from`, treat the message as a forwarded relay and do **not** store `remote_listens` in `pending_delivered` under `from_peer_id` — or require a separate authenticated advertisement.
2. **Scope `pending_delivered` to the sending session**: Key the map on `(session_id, from_peer_id)` or only accept direct (non-forwarded) `ConnectionRequest` messages for the `respond_delivered` path.
3. **Authenticate listen addresses**: Require that each address in `listen_addrs` carries a signature from the private key corresponding to `from_peer_id`.

---

### Proof of Concept

```
1. Attacker (peer E) connects to victim node B.
2. E sends ConnectionRequest { from=A, to=B, listen_addrs=[attacker_ip:port] }.
3. B: pending_delivered[A] = ([attacker_ip:port], T0). B sends ConnectionRequestDelivered to E.
4. Legitimate peer A sends ConnectionRequest { from=A, to=B, listen_addrs=[A_real_ip:port] }.
5. B: pending_delivered[A] exists, now - T0 < HOLE_PUNCHING_INTERVAL → returns Ignore.
   A never receives ConnectionRequestDelivered. Hole-punching fails.
6. At T0 + 2min + 1ms, E sends another ConnectionRequest { from=A, to=B, listen_addrs=[attacker_ip:port] }.
7. B: pending_delivered[A] = ([attacker_ip:port], T0+2min). Entry refreshed.
8. A retries → blocked again. Repeat from step 6 indefinitely.
9. If a ConnectionSync { from=A, to=B } ever reaches B (e.g., from a prior exchange),
   B dials attacker_ip:port for up to 30 seconds, disclosing B's source IP to E.
```
